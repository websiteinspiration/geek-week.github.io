Spring Boot + Apache Kafka et SSL dans le conteneur Docker

salut les amis Dans l'article, je vais vous expliquer comment créer une solution pour démarrer rapidement votre développement à partir d'une application Java autonome, d'un courtier et de Kookka Zookeeper, élevée dans des conteneurs Docker séparés et configurée avec une authentification en texte brut pour l'accès à la fois à la machine locale et au circuit externe.

Le projet terminé sur github peut être récupéré à la fin de l'article



Choisir une image Docker Apache Kafka + Zookeeper


Vous devez d'abord choisir le fournisseur d'images Apache Kafka pour notre conteneur. Les exigences de chaque projet spécifique varient selon le niveau nécessaire de sécurité et de fiabilité de la solution, dans certains cas, bien sûr, vous devrez assembler votre propre image, mais pour la plupart des projets, il sera sage d'en choisir une disponible sur le docker hub. Pour 2020, il existe trois principales images de fournisseurs populaires kafka + zookeper:

  • images confluentes [subjectivement, un gros plus - elles sont fournies par les développeurs d'Apache Kafka lui-même et sont les plus fiables en termes de sécurité de l'information]
  • images de bitnami [plus, en comparaison avec confluent, que les images ont besoin d'une configuration minimale et "commencent par la boîte"]
  • wurstmeister [ , github hub.docker, ]

Docker-compose


Pour lancer et configurer des images, Docker-compose est utilisé , ce qui facilite l'exécution d'applications multi-conteneurs. Vous ne pouvez pas vous en passer, car la configuration minimale de kafka prévoit un tas de gardiens et au moins un courtier.

ompose est fourni avec mac / windows, la distribution Docker, donc l'installation de l'environnement nécessaire se résume au téléchargement et à l'installation de la distribution depuis docs.docker.com/engine/install .

Mais les utilisateurs de Linux doivent l'installer séparément. Pour un guide d'installation de docker + compose, il est préférable de consulter la documentation officielle docs.docker.com/compose/install .

Écrire un fichier yaml pour composer


Le schéma général de notre configuration pour tous les clients et conteneurs sera le suivant:



Un courtier avec les ports ouverts 9093 sera connecté à un coordinateur - pour les connexions à l'hôte local et 9092 pour les connexions à partir d'autres machines.

Pour accéder à partir de réseaux externes, activez l'autorisation via un mot de passe de connexion sans cryptage SSL. Pour travailler avec SSL, vous devez émettre vos propres clés et signer des certificats, l'ensemble du processus est bien décrit dans la documentation officielle .

En conséquence, le fichier ressemblera à ceci (l'adresse doit être remplacée par votre IP publique pour l'accès à partir d'autres réseaux):

version : ' 2 '
services :
gardien :
image : wurstmeister / gardien de zoo
expose:
- "2181"
kafka:
image: wurstmeister/kafka
ports:
- "9092:9092"
expose:
- "9093"
environment:
KAFKA_AUTO_CREATE_TOPICS_ENABLE: 'true'
KAFKA_ADVERTISED_LISTENERS: INSIDE://kafka:9093,OUTSIDE://<your public IP here>:9092
KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: INSIDE:PLAINTEXT,OUTSIDE:SASL_PLAINTEXT
KAFKA_LISTENERS: INSIDE://0.0.0.0:9093,OUTSIDE://0.0.0.0:9092
KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
KAFKA_INTER_BROKER_LISTENER_NAME: INSIDE
KAFKA_OPTS: "-Djava.security.auth.login.config=/etc/kafka/kafka_server_jaas.conf"
KAFKA_SASL_ENABLED_MECHANISMS: PLAIN
KAFKA_SASL_MECHANISM_INTER_BROKER_PROTOCOL: PLAIN
depends_on:
- zookeeper
volumes:
- ./:/etc/kafka

Pour activer l'autorisation, vous devez transférer le fichier kafka_server_jaas.conf vers kafka. Dans le dossier contenant le fichier compose yaml, créez kafka_server_jaas.conf et ajoutez un utilisateur administrateur avec le mot de passe secret administrateur dedans:

KafkaServer {
org.apache.kafka.common.security.plain.PlainLoginModule requis
username = "admin"
mot de passe = "admin-secret"
user_admin = "admin-secret";
};
Client {};

Ce fichier sera transféré vers l'image avec kafka broker via:

  1. spécifiant le chemin d'accès au fichier dans la configuration de kafka directement
    KAFKA_OPTS: "-Djava.security.auth.login.config=/etc/kafka/kafka_server_jaas.conf"
  2. mappage du répertoire local avec le fichier dans / etc / kafka le répertoire image
    volumes:
    - ./:/etc/kafka

Lancement de conteneurs


Par conséquent, nous devrions obtenir 2 fichiers qui se trouvent dans le même répertoire:

  1. docker-compose.yml
  2. kafka_server_jaas.conf

Dans ce répertoire, vous devez appeler la commande:

$ docker-compose up -d

l'indicateur -d vous permet de démarrer en mode Détaché et de fermer la console si nécessaire sans désactiver les images.

Client Java sur Spring Boot


Nous allons écrire un client minimaliste avec la possibilité de publier et de lire des messages de kafka. Pour ce faire, créez un nouveau projet via Spring Initializr ou tout autre outil que vous connaissez. Il suffit de connecter une dépendance:



je préfère configurer les applications Spring Boot autant que possible via des fichiers de configuration yaml avec une configuration minimale en code Java. En raison de la structuration intégrée au format Yaml, ces fichiers sont beaucoup plus pratiques pour la perception humaine que les fichiers .properties et vous permettent de regrouper les paramètres par modules. Le seul inconvénient de la configuration .yml est qu'elle n'est pas prise en charge dans les tests, mais elle est facilement contournée par les outils de démarrage intégrés.
Je recommande fortement d'essayer de voir comment votre perception de l'ancien fichier .properties (en particulier pour les grands projets) change si vous le réécrivez dans yaml. Cela peut être fait en quelques clics sur www.toyaml.com/index.html .
