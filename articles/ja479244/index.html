ビットマップインデックス付きのプレフィックスツリー

少し前に、他の人々によってすでに実装されている機能を実装する必要がありましたが、いくつかの特性には適していませんでした。この場合、文字列キーまたはキーのいくつかの最初の文字で検索する機能を備えた、ある種のデータ構造が必要でした。キー-いずれの場合もラテン文字の文字列、スペースと数字。

他の実装は、まず第一に、消費されたメモリに適合しませんでした。そしていずれの場合でも、プレフィックスツリーの別の実装は不要ではありません。

ノード内の後続ノードへのリンクのインデックス付けにビットマップインデックスを使用することが決定されたため、この アイデアはおそらくこの記事から発想を得たものです。

明らかに、1つのノード内のリンクの数は63以下にする必要があります（10桁、各場合に26文字とスペース）。したがって、ビットマップインデックスの場合、64ビットの数値が各ノードで使用されます。任意の位置のセットビットは、対応するシンボルとノード（サブツリー）の存在を意味します。

ビットマップインデックスの分布は次のとおりです。0〜9の数字は0〜9ビット、a〜zの文字は10〜35ビット、a〜zの文字は36〜62ビット、スペースは63ビットを占めます。ビット数を取得するために、ASCII文字コードが使用され、特定の数だけ減らされます。文字の範囲ごとに、数値は異なります：数値48（ASCIIコードの数字の範囲は48で始まります）、文字az 87（文字azの範囲は、10桁で既に占められているマイナス10ビットで始まります）など。

対応する表を以下に示します。

キャラクター分布
シンボル 0 1 ... 9 a b ... z あ B ... Z スペース
ビット数 0 1 ... 9 10 十一 ... 35 36 37 ... 61 62

ノードは以下のコードで表されます：

type Trie struct {
    rw sync.RWMutex
    characters uint64
    subTries []*Trie
    data interface{}
}

したがって、各ノードには、次の文字のビットマップインデックス、サブツリーへのリンクを含むスライス、およびデータ自体が格納されます。検索、挿入、削除のメカニズムについて簡単に説明します。

入力を入力すると、キーはルートノードから始まり、ツリーを象徴的に通過します。ASCIIコードの各文字についてビット数が計算され、それに基づいて、サブツリーへのリンク（subTries）のスライスでインデックスが計算されます。インデックス値は、目的のビットの前のビット数です。たとえば、次のようなインデックスがあります：00 ... 101。これは、ビットマップインデックスに数値0と2があることを意味します。すると、数値0のインデックスはゼロになり、数値2のインデックスは1になります。つまり、subTriesはサブツリーへの2つのリンクを格納します。subTries[0]とsubTries [1] 。

文字のビット番号を取得するコード：

func calcBitNum(char rune) (uint64, bool) {
	// Characters a-z use bit positions 10-35
	if char > 96 && char < 123 {
		return uint64(char - 87), true
	}

	// Characters A-Z use bit positions 36-61
	if char > 64 && char < 91 {
		return uint64(char - 29), true
	}

	// digits 0-9 use bit positions 0-9
	if char > 47 && char < 58 {
		return uint64(char - 48), true
	}

	// space uses 62 bit position
	if char == 32 {
		return 62, true
	}

	return 0, false
}

キャラクターのノードインデックスを取得するコード：

func (trie *Trie) getSubTreeIndex(bitNum uint64) int {
	shifted := trie.characters << (64 - bitNum)
	return bits.OnesCount64(shifted)
}

指定されたキャラクターのノードを取得するコード：

func (trie *Trie) getSubTrie(char rune) (uint64, int, *Trie) {
	bitNum, _ := calcBitNum(char)
	subTrieId := trie.getSubTreeIndex(bitNum)

	// There is no subTrie under given character since the corresponding bit is zero
	if trie.characters&(1<<bitNum) == 0 {
		return bitNum, subTrieId, nil
	}

	return bitNum, subTrieId, trie.subTries[subTrieId]
}

インサート


要素を挿入するためにツリーをたどります。一部の文字（およびそれ以降のすべての文字）が欠落している場合は、ビットマップインデックスに挿入されます。サブツリーが作成され、スライスsubTriesの適切な場所にリンクが挿入されます（適切な場所は、目的のビット数によって決定されます）。キーの最後の文字に達すると、データが最後のサブツリーに挿入されます。

探す


検索するには、キーが象徴的にツリーを「実行」します。終了するとすぐに結果が返されます。キーのデータはないが、サブツリーがさらにある場合、結果は、データを持つ後続のすべてのサブツリーから選択されます。返されるデータの数に制限を設定できます。したがって、キー全体の一致によって1つのノードを検索したり、いくつかの最初の文字の一致によってノードのセットを検索したりすることができます。

削除する


除去手順は、残りの部分よりも（予想どおり）複雑になりました。キーを削除するには、ツリーをたどって、その間にデータのないノードが別のスライスに記録されます。データのあるノードが見つかった場合、スライスはゼロにリセットされ、ツリーの通過はさらに続行されます。つまり、削除するノードのスライスには、データがない一連のノードが含まれている必要があります。最後にスライスが空でない場合、スライスは逆の順序で進み、対応するビットがビットマップインデックスから削除され、ノードが削除されます。

合計


その結果、速度とメモリ消費量が他の類似物よりも優れたプレフィックスツリーになります（平均で、メモリ量が30％少なく、はるかに高速です）。また、ツリーを使用したマルチスレッド操作を可能にするほど面倒ではなかったため、特に検索操作で生産性が大幅に向上しました。もちろん、このソリューションは非常に特殊化されています（重要なコンポーネントとしてのラテン文字、数字、スペースの制限）。

接頭辞ツリーの目的に基づいて、このソリューションを使用して辞書、検索インデックスを作成できます。
