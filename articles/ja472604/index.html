その結果、理解できないデータセットを含むカスタムメーターも追加されることがわかります。

当然のことながら、そのようなデータをリポジトリに保存したくありません。どうやって自分自身を守るのですか？最も簡単なオプションは、たとえば、すべての文字（文字）がない場合にエラーを返すことです。これを行うには、ifを使用してFirstNameの条件を設定し、条件が満たされない場合は、returnを使用して関数を停止し、コンソールに「名前にエラーがあります」というメッセージを表示します。 LastNameについても同様です。年齢については、たとえば、ここで数値の範囲を確認します。

if (customer.Age < 14 || customer.Age > 180)

次に、購入者のフィールドを追加する必要があると仮定しましょうたとえば、電話。入力された値は「+79」で始まり、数字のみを含む必要があるという条件を使用して、電話を検証します。これ自体はかなり面倒な設計になりますが、電子メールを追加したい場合はどうでしょうか。

とにかく、上記の操作を実行した後、一連のifsと大きなコードシートを取得します。外部の開発者がそのようなコードを理解することは容易ではありません。何をすべきか？

流暢な検証の接続


LINQPadには、Fluent Validationライブラリを接続する機能があります。さらに、バリデータとなる別のクラスCustomerValidatorを作成します。したがって、必要なルールはすべて規定しています。追加の調整を行い、不要になった多数のifを削除します。

結果として、最終的なコードは次のようになります。

void Main()
{
 var customer = new Customer
 {
  FirstName = "Alex2",
  LastName = "Petrov1",
  Age = 10,
  Phone = "+791234567893",
  Email = "adsf@fadsf3.com"
 };
 
 var manager = new CustomerManager();
 manager.Add(customer);
}
 
class Customer
{
 public string FirstName { get; set; }
 public string LastName { get; set; }
 public int Age { get; set; }
 public string Phone { get; set; }
 public string Email { get; set; }
}
 
class CustomerManager
{
 CustomerRepository _repository;
 CustomerValidator _validator;
 
 public CustomerManager()
 {
  _repository = new CustomerRepository();
  _validator = new CustomerValidator();
 }
 
 public void Add(Customer customer)
 {
  if (!ValidateCustomer(customer))
  {
   return;
  }
 
  _repository.Add(customer);
  Console.WriteLine($" {customer.FirstName} {customer.LastName}  .");
 }
 
 private bool ValidateCustomer(Customer customer)
 {
  var result = _validator.Validate(customer);
  if (result.IsValid)
  {
   return true;
  }
 
  foreach(var error in result.Errors)
  {
   Console.WriteLine(error.ErrorMessage);
  }
  return false;
 }
}
 
class CustomerValidator : AbstractValidator<Customer>
{
 public CustomerValidator()
 {
  var msg = "   {PropertyName}:  {PropertyValue}";
 
  RuleFor(c => c.FirstName)
  .Must(c => c.All(Char.IsLetter)).WithMessage(msg);
 
  RuleFor(c => c.LastName)
  .Must(c => c.All(Char.IsLetter)).WithMessage(msg);
 
 RuleFor(c => c.Age)
  .GreaterThan(14).WithMessage(msg)
  .LessThan(180).WithMessage(msg);

  RuleFor(c => c.Phone)
  .Must(IsPhoneValid).WithMessage(msg)
  .Length(12).WithMessage("    {MinLength}  {MaxLength}.  : {TotalLength}");
 
  RuleFor(c => c.Email)
  .NotNull().WithMessage(msg)
  .EmailAddress();
 }
 
 private bool IsPhoneValid(string phone)
 {
  return !(!phone.StartsWith("+79")
  || !phone.Substring(1).All(c => Char.IsDigit(c)));
 }
}
 
class CustomerRepository
{
 Random _random;
 
 public CustomerRepository()
 {
  _random = new Random();
 }
 
 public void Add(Customer customer)
 {
  var sleepInSeconds = _random.Next(2, 7);
  Thread.Sleep(1000 * sleepInSeconds);
 }
}

そしてもう少し理論


Fluent Validationについてもう少し説明します。このツールは、「流体」インターフェースのためにそのように呼ばれています。繰り返しになりますが、Wikipediaによると、流動的なインターフェースは、プログラムのソースコードの読みやすさを向上させることを目的とした、オブジェクト指向のAPIを実装する方法であるということです。私たちが見るように、定義には多くの美しく長い単語が含まれていますが、必ずしも明確ではありません。しかし、別の言い方もできます。
「流体インターフェイスは、メソッドが呼び出されたのと同じインターフェイスをメソッドが返すオブジェクト指向APIを実装する方法です。」
