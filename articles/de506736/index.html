Wie viele Methoden sollte eine Klasse haben?

Vielleicht ist dies eines der wichtigsten Probleme, mit denen Entwickler konfrontiert sind, die Klassen verwenden. Wir haben die Prinzipien von SOLID, wir haben unsere eigene Vision von dem Schönen und einer Vielzahl von Diskrepanzen. Ich hoffe nicht, eine erschöpfende Antwort zu geben: "Wie viele Methoden sollten in Ihrer Klasse sein?". Aber ich werde meine Gedanken darüber teilen, warum es im Unterricht weniger Methoden geben sollte, viel weniger. Bis zu 1.

In größerem Umfang betrifft meine Argumentation Dienstleistungen (staatenlose Objekte), die meiner Meinung nach in realen Projekten Vorrang vor allen anderen Arten von Objekten haben. Eine Klasse sollte nicht nur wegen ihrer ähnlichen Implementierung eine Vielzahl von Methoden in sich speichern!

Nachdem Anfänger die Grundlagen von OOP kennengelernt haben, verbinden sie Klassen und Objekte mit Katzen und Hunden. Das heißt, unter dem ersten Eindruck werden Objekte als Projektionen der realen Welt oder als einige Modelle wahrgenommen, die eine Reihe notwendiger Methoden zur Lösung des Problems enthalten. Und dies ist eine großartige Meinung über OOP, genauer als es sein sollte.

Aber nach ein paar kommerziellen Projekten bricht die Ansicht zusammen. Unternehmen werden von Diensten und anämischen Modellen beherrscht. Ich weiß nicht, ob es gut oder schlecht ist, aber es ist Realität. Daher werden in der Praxis häufig dicke Klassen erhalten, die keine Analogien zur realen Welt aufweisen und einer eindeutigen Verallgemeinerung nicht zugänglich sind. Oft werden sie Dienste genannt, oft bilden sie Schichten. Und das Verständnis des Objekts fällt auf die Ebene - eine Gruppe von Methoden mit einem ähnlichen Zweck. Oder Methoden, die mit ähnlichen Daten arbeiten. Oder etwas ähnliches.

Jedes Mal, wenn ich anfange, einen Dienst zu schreiben, denke ich lange über den Namen nach ... Sinnlos AnotherEntityService immer wieder. Aber was ist, wenn es kein Substantiv geben sollte? Wie spiegelt der Name des Dienstes die Vielzahl der darin gesammelten Anfragen wider? Wer wendet diese Methoden in der Regel sofort an?

Das Refactoring-Beispiel von Onkel Bob lässt mich nicht lange los [Clean Code, S. 177]:

class Sql {
    constructor(private table: string, private columns: Column[]) {}

    public create(): string;
    public insert(fields: object[]): string;
    public selectAll(): string;
    public select(criteria: Criteria): string;

Die SQL-Klasse ändert sich, wenn ein neuer Befehlstyp hinzugefügt wird. Darüber hinaus ändert es sich, wenn die Details der Implementierung eines vorhandenen Befehlstyps geändert werden ... Zwei Gründe für die Änderung bedeuten, dass die SQL-Klasse gegen das Prinzip der einheitlichen Verantwortung verstößt.

Ich dachte, weil das Teilen so viele Vorteile hat, warum herrschen in Wirklichkeit aufgeblähte Klassen vor? Weil wir Klassen auf die Ebene der Funktionen aufteilen und es schwieriger wird, das Gesamtbild zu verstehen? (Aber ist FP-Schnicks diesbezüglich komplex?) Oder weil die Verkapselung aus unseren fetten Serviceleistungen austritt? Es scheint mir, dass es eine dunkle Seite gibt, die das Geheimnis großer Klassen bewahrt.

Was gibt die Klasse in Bezug auf Programmiersprachenkonstrukte, wenn wir die Romantik verwerfen? Die Klasse hat eine gute Seite - öffentliche Methoden, die automatisch die Schnittstelle des Objekts bilden. Die Schnittstelle bildet eine Abstraktion, die das Programm vereinfachen soll. Das ist Romantik.

Wenn Sie jedoch pragmatisch aussehen, sammelt die Klasse nur die erforderlichen Abhängigkeiten über den Konstruktor, speichert private Mitglieder, die nirgendwo anders verwendet werden, und bietet Clients eine Reihe von Sticking-Funktionen (öffentlich).

Und das ist die dunkle Seite - private Felder, private Methoden und Abhängigkeiten. Dies ist, was Entwickler am häufigsten durch das Kombinieren von Methoden im Service geleitet werden. Manchmal scheint es, dass wenn die Klasse alle notwendigen Mitglieder enthält, dies der beste Ort für eine neue Methode ist! Und der Service wird zu einem Verwalter von Abhängigkeiten und Methoden, die diese Abhängigkeiten benötigen.

Seit wann sollten gemeinsame Abhängigkeiten etwas vereinen? Die Ähnlichkeit der internen Implementierung ist nicht der Grund für die Erstellung der Abstraktion. Mit anderen Worten, wenn wir dem Dienst eine neue Methode hinzufügen, weil sie in ihrer Implementierung „anderen Methoden sehr ähnlich“ ist, verletzen wir die Kapselung, SRP, OCP und ISP. Aber ich stoße ständig darauf.

In der Praxis bildet die Dienstarchitektur also keine Abstraktionen auf hoher Ebene, sondern stellt lediglich eine Analogie des Prozeduralismus dar. Darüber hinaus ist die schlechteste Analogie, da Prozeduren keine überflüssigen Entitäten in Form von pseudo-generalisierenden Dienstnamen haben und keine nicht verwendeten Abhängigkeiten ziehen. (Irgendwo düsteres FPShniki).

@Injectable({ providedIn: 'root' })
export class HeroService {
 
 private heroesUrl = 'api/heroes';  // URL to web api
 
 httpOptions = {
   headers: new HttpHeaders({ 'Content-Type': 'application/json' })
 };
 
 constructor(
   private http: HttpClient,
   private messageService: MessageService) { }
 
 /** GET heroes from the server */
 getHeroes(): Observable<Hero[]> {
   return this.http.get<Hero[]>(this.heroesUrl)
     .pipe(
       tap(_ => this.log('fetched heroes')),
       catchError(this.handleError<Hero[]>('getHeroes', []))
     );
 }
 
 /** GET hero by id. Will 404 if id not found */
 getHero(id: number): Observable<Hero> {
   const url = `${this.heroesUrl}/${id}`;
   return this.http.get<Hero>(url).pipe(
     tap(_ => this.log(`fetched hero id=${id}`)),
     catchError(this.handleError<Hero>(`getHero id=${id}`))
   );
 }
 
 /* GET heroes whose name contains search term */
 searchHeroes(term: string): Observable<Hero[]> {
   if (!term.trim()) {
     // if not search term, return empty hero array.
     return of([]);
   }
   return this.http.get<Hero[]>(`${this.heroesUrl}/?name=${term}`).pipe(
     tap(x => x.length ?
        this.log(`found heroes matching "${term}"`) :
        this.log(`no heroes matching "${term}"`)),
     catchError(this.handleError<Hero[]>('searchHeroes', []))
   );
 }
 
 //////// Save methods //////////
 
 /** POST: add a new hero to the server */
 addHero(hero: Hero): Observable<Hero> {
   return this.http.post<Hero>(this.heroesUrl, hero, this.httpOptions).pipe(
     tap((newHero: Hero) => this.log(`added hero w/ id=${newHero.id}`)),
     catchError(this.handleError<Hero>('addHero'))
   );
 }
 
 /** DELETE: delete the hero from the server */
 deleteHero(hero: Hero | number): Observable<Hero> {
   const id = typeof hero === 'number' ? hero : hero.id;
   const url = `${this.heroesUrl}/${id}`;
 
   return this.http.delete<Hero>(url, this.httpOptions).pipe(
     tap(_ => this.log(`deleted hero id=${id}`)),
     catchError(this.handleError<Hero>('deleteHero'))
   );
 }
 
 /** PUT: update the hero on the server */
 updateHero(hero: Hero): Observable<any> {
   return this.http.put(this.heroesUrl, hero, this.httpOptions).pipe(
     tap(_ => this.log(`updated hero id=${hero.id}`)),
     catchError(this.handleError<any>('updateHero'))
   );
 }
 
 /**
  * Handle Http operation that failed.
  * Let the app continue.
  * @param operation - name of the operation that failed
  * @param result - optional value to return as the observable result
  */
 private handleError<T>(operation = 'operation', result?: T) {
   return (error: any): Observable<T> => {
 
     // TODO: send the error to remote logging infrastructure
     console.error(error); // log to console instead
 
     // TODO: better job of transforming error for user consumption
     this.log(`${operation} failed: ${error.message}`);
 
     // Let the app keep running by returning an empty result.
     return of(result as T);
   };
 }
 
 /** Log a HeroService message with the MessageService */
 private log(message: string) {
   this.messageService.add(`HeroService: ${message}`);
 }
}


Dies ist ein typischer Service. (Ich würde sogar "Best Practices" sagen). Und auf den ersten Blick ist alles in Ordnung mit ihm. HeroService funktioniert mit Hero-Entitäten, sonst nichts. Viel überflüssig! Kein Client verwendet die Schnittstelle auf konsistente Weise - nur in Teilen.

Was tun, wenn eine neue API-Methode für Hero angezeigt wird? Wann müssen Sie aufhören, alles zu einem zusammenzufalten und die Klasse aufzubrechen? Ich bin sicher, dass jeder diese Fragen gestellt hat. Das Vorhandensein dieses Dienstes im Projekt verwirrt die Entwickler jedoch nur und zwingt sie, Code durch eine Änderung und nicht durch eine Erweiterung zu schreiben.

Mal sehen, was die in HeroService integrierten Methoden „ähnlich“ macht und wie sie zusammenbrechen.

  • Alle Methoden verwenden eine gemeinsame URL - sie wird gelöst, indem BASE_HERO_API_URL in die Konstante eingefügt wird.
  • Einige Methoden verwenden dieselben http-Header - dies wird entschieden, indem sie in einen Interceptor eingefügt werden. Oder Sie können einen speziellen HttpClient auswählen, der die Methoden get, post, put und delete mit den erforderlichen Optionen umschließt.
  • handleError - wird gelöst, indem es in eine separate Klasse eingeordnet wird, die perfekt als die Person dient, die für die Behandlung von Fehlern verantwortlich ist.

Es stellte sich heraus, dass alle verbindenden Faktoren optionale Details der Implementierung sind, die anders hätten sein können, wenn nicht der Wille zufälliger Umstände gewesen wäre. Es gab keine externen Gründe für die Fusion. Wir schreiben das Web, und es ist seltsam, wenn eine Schaltfläche zuerst addHero aufruft und dann die andere HeroService-Funktionalität verwendet.

@Injectable({ providedIn: 'root' })
export class ErrorHandler {
 constructor(private messageService: MessageService) {}
 
 handleError<T>(operation: string, result?: T) {
   return (error: any): Observable<T> => {
     // TODO: send the error to remote logging infrastructure
     console.error(error); // log to console instead
 
     // TODO: better job of transforming error for user consumption
     this.messageService.add(`${operation} failed: ${error.message}`);
 
     // Let the app keep running by returning an empty result.
     return of(result as T);
   };
 }

@Injectable({ providedIn: 'root' })
export class HeroHttpClient {
 private httpOptions = { headers: new HttpHeaders({ 'Content-Type': 'application/json' }) };
 
 constructor(@Inject(BASE_HERO_API_URL) private baseUrl: string, private http: HttpClient) {}
 
 get<T>(endpoint = ''): Observable<T> {
   return this.http.get<T>(this.getUrl(endpoint));
 }
 
 private getUrl(endpoint: string): string {
   return Location.joinWithSlash(this.baseUrl, endpoint);
 }
 
 post<T>(endpoint = '', data: unknown): Observable<T> {
   return this.http.post<T>(this.getUrl(endpoint), data, this.httpOptions);
 }
 
 put<T>(endpoint = '', data: unknown): Observable<T> {
   return this.http.put<T>(this.getUrl(endpoint), data, this.httpOptions);
 }
 
 delete<T>(endpoint = ''): Observable<T> {
   return this.http.delete<T>(this.getUrl(endpoint), this.httpOptions);
 }
}


Ich versuche nicht (und möchte nicht) Aussagen zu machen, wie zum Beispiel: "Eine Klasse kann nicht mehr als die 1. Methode haben!". Wenn ein Objekt einen Zustand hat, treten natürlich möglicherweise andere Regeln in Kraft. Ich bin auch nicht dagegen, Methoden aufzunehmen, die konsistent in der Klasse verwendet werden sollten: isExecutable, validateParams, execute.

Was ich sage, ist mehr über Monsterdienste. Da Dienste meistens keine vernünftige Abstraktion darstellen. Und es wäre besser, wenn sich der Schwerpunkt auf ihre Abnahme verlagern würde. Für mich selbst habe ich das folgende Prinzip formuliert: Eine

Klasse sollte nicht nur wegen ihrer ähnlichen Implementierung eine Reihe von Methoden enthalten.

Nach dieser Regel können Sie verlorene Systemelemente erkennen, die in privaten Methoden verborgen sind. Dieser Ansatz wird dazu beitragen, OCP und ISP buchstäblich einzuhalten. Wenn Ihre Funktionen das Programm wirklich erweitern und keine Methoden in 10 Klassen in 10 Ebenen hinzufügen.
