JavaScriptで変数のlet宣言と結果のクロージャーの機能を使用する

私は、Habréの記事を読んでこのメモを書くように勧められました。「Var、let or const？変数とES6の範囲の問題とそのコメント、およびZakas N.の著書「ECMAScript 6の理解」の対応する部分。私が読んだことに基づいて、varまたはletの使用を評価する際にすべてがそれほど単純ではないという結論に達しました。著者とコメンテーターは、古いバージョンのブラウザーをサポートする必要がない場合は、デフォルトでvarの使用を完全に放棄し、古いものの代わりにいくつかの簡略化された構造を使用するのが理にかなっていると信じる傾向があります。

上記の資料を含め、これらの広告の範囲については十分に述べられているため、いくつかの自明でない点のみに焦点を当てたいと思います。

まず、ループ内ですぐに呼び出される関数（Immediately Invoked Function Expression、IIFE）の式を検討します。
let func1 = [];

for (var i = 0; i < 3; i++) {
    func1.push(function(i) { return function() { console.log(i); } }(i));
}

func1.forEach(function(func) { func(); });

Zakas N.は、同じ結果を与える同様の例はどちらもまったく同じように機能すると主張しています。
「このループは、varとIIFEを使用したループとまったく同じように機能しますが、間違いなくよりクリーンです。」
しかし、彼自身は、さらに、間接的に反論します。

実際のところ、letを使用すると、ループの各反復で個別のローカル変数iが作成され、同時に、配列に送信される関数内のバインディングも、各反復から個別の変数になります。

この特定のケースでは、結果はまったく同じですが、コードを少し複雑にするとどうなるでしょうか。
let func1 = [];

for (var i = 0; i < 3; i++) {
    func1.push(function(i) { return function() { console.log(i); } }(i));
    ++i;
}

func1.forEach(function(func) { func(); });

/*
  

0 newECMA6add.js:4:59
2 newECMA6add.js:4:59
*/

ここで++ iを追加すると、ループ自体が渡された場合でも呼び出し時に関連したi値で関数を呼び出したため、結果はかなり予測可能であることがわかりました。したがって、後続の操作++ iは、配列内の関数に渡された値に影響を与えませんでした。で閉鎖された機能（I）の特定の値とI。

今と比較let-なしバージョン生命維持

結果は明らかに変更されました。この変更の性質として、値を指定して関数をすぐに呼び出さなかったのですが、関数は、サイクルの特定の反復でクロージャーで使用可能な値を取得しました。

何が起こっているかの本質をよりよく理解するために、2つの配列を持つ例を考えてみましょう。まず、IIFEなしでvarを取り上げます。

両方の例で、次のことが起こります

。1.サイクルの最後の反復の開始時にi == 2、それから1（++ i）だけ増加し、最後にi ++からさらに1が追加されます。その結果、サイクル全体の終わりにi == 4になります。

そしてここでは、よく知られているlet inサイクルの原理を考えると、ループ変数への短絡を形成しました。1つではなく2つであり、別々ではありませんが、論理的です。

結果として、最初のクロージャーでは、配列内の関数を呼び出す前の値はi == 1で、2番目のクロージャーではi == 3です。これらは、変数という値であり、私が受け取った前に私++およびループ反復が、ループブロック内の命令のすべての後、彼らはそれぞれ、特定の反復のため閉鎖されています。

次に、配列func1にある関数が呼び出され、両方のクロージャー内の対応する変数をインクリメントします。その結果、最初のi == 2と2番目のi == 4になります。。

その後のfunc2の呼び出しはさらにインクリメントし、それぞれi == 3および5を取得します。func2とfunc1

合計は何ですか


1.すぐに呼び出される関数の式を呼び出すことは、ループ内の関数で反復可能なlet変数を使用することと同じではなく、場合によっては、異なる結果を導きます。

2. イテレータにlet宣言を使用すると、反復ごとに個別のローカル変数が作成されるため、ガベージコレクタによる不要なデータの破棄について問題が生じます。この時点で、私は告白、私は使用してテスト配列をソートする場合、それぞれ、大型で多数の変数を作成し、ループが、しかし、コンパイラを遅くすることを疑って、最初は焦点の注意にたかってみましょう変数の宣言は、100,000セルの配列に対して実行時間のほぼ2倍の向上を示しました。
var付きのオプション：
const start = Date.now();
var arr = [],
    func1 = [],
    func2 = [];

for (var i = 0; i < 100000; i++) {
    arr.push(Math.random());
}

for (var i = 0; i < 99999; i++) {
    var min,
        minind = i;

    for (var j = i + 1; j < 100000; j++) {
        if (arr[minind] > arr[j])
	    minind = j;
    }
  
    min = arr[minind];
    arr[minind] = arr[i];
    arr[i] = min;
    func1.push(function(i) { return function() { return i; } }(arr[i]));
}
func1.push(function(i) { return function() { return i; } }(arr[99999]));

for (var i = 0; i < 100000; i++) {
    func2.push(func1[i]());
}

const end = Date.now();

console.log((end - start)/1000); // 9.847
