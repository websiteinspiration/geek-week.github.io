This article is devoted to the approach defined in the IEC 61508 standard, which consists in using secure subsets of the C and C ++ programming languages ‚Äã‚Äãfor the development of critical software.



IEC 61508 has seven parts and an official translation into Russian, which has the status of a national standard of the Russian Federation. For the topic under discussion, we are primarily interested in part 3, ‚ÄúSoftware requirements‚Äù [2], which contains the necessary basic principles, and part 7, ‚ÄúMethods and tools‚Äù [3], which offers specific recommendations. Further we will consider the standards ‚ÄúMISRA C: 2012‚Äù [4] and ‚ÄúMISRA C ++: 2008‚Äù [5], which have the most immediate practical value for programmers.



Programming as a risk management tool



IEC 61508 provides structured and systematic recommendations on the application of various risk reduction tools and techniques at various stages of the development life cycle. Writing code, i.e. coding is one of the stages. Other things, I dare say, it is physically impossible to exclude even with the most non-deterministic software development process.



Of course, measures to reduce risk have their price, expressed in the cost of working time for their implementation. And the working time of each employee has a very specific monetary expression.

At a conference on avionics software, I heard a story about some management system developed by a large international company. The product had to meet the requirements of regulators in 66 countries, therefore, only in 2015, audits of the developing company and the product were carried out by more than a hundred organizations. Over the year, the developer spent 3,500 man-days to ensure the conduct of these audits.



The higher the requirements to reduce risk, the higher the cost of implementing the relevant activities. Therefore, IEC 61508 divides systems that ensure functional safety into four hierarchical categories, called in Russian safety integrity levels (UPB). The highest level is UPB4, the lowest is UPB1. The English original uses the term safety integrity level (SIL).



Criteria for choosing a programming language



Appendix A of the seventh section of IEC 61508 [3] contains a very informative guide to the selection of methods and tools, and Appendix C, in turn, provides an overview of the methods and tools with a brief description and links to sources of information about them.



Guidelines for selection are formulated very simply: for each level and each means a designation is given. These designations are called ‚Äúrecommendations‚Äù. The most stringent recommendations are HR (highly recommended) and NR (strongly not recommended).



Of course, for the development of software for critical systems, there are a number of specialized programming languages, such as, for example, Ada, D, FORTRAN 77 or RUST, which provide strict typing of data and solve other important issues related to coding. All of them have their own niches and are very interesting from an engineering point of view. But, since we, dear reader, are realists, for this article the choice fell on the language in which programs for critical systems are actually most often written - in C. And since object-oriented technologies such as Qt and Mesa 3D, are gaining more and more popularity among developers of real-time systems, you can not ignore the issues of using C ++.



What does IEC 61508 say about these languages?



For high levels of security completeness - UPB3 and UPB4 - for the C, C ++ languages ‚Äã‚Äãand, just in case, I note Java, the NR recommendation is given. However, for C and C ++, as stated in the standard, ‚Äúwith a subset and coding standard, as well as the use of static analysis tools,‚Äù HR recommendation is given. At the same time, for example, for Java, even for a subset with disabled or deterministic garbage collection, the recommendation NR is indicated for the levels of UPB3 and UPB4.



What are ‚Äúsubsets of the language‚Äù for? The answer is simple: to eliminate potentially dangerous constructions, to ensure readability of the code and its suitability for processing by static analysis programs.



Let‚Äôs figure it out with you which subsets and coding standards we can apply for C and C ++.



Language C



The most authoritative document defining a subset of the C language is the standard known as ‚ÄúMISRA C‚Äù [4]. Based on it, or at least under its influence, standards for secure coding of various organizations are developed, for example, the JPL coding standard [6].



In the introductory part, ‚ÄúMISRA C‚Äù gives reasons for the popularity of C. We will name some of them:



  • Programs written in C can be compiled into efficient machine code;
  • There are ready-made C compilers for a wide variety of hardware architectures;
  • C is supported by a significant number of tools for static analysis and testing;
  • There is an international standard for the language C [7], [8];
  • Great experience has been gained in using C in the development of critical systems.


At the same time, the developers of ‚ÄúMISRA C‚Äù note the drawbacks of the C language. Among them are inaccurate definitions that allow compiler developers to implement some functionality in different ways, as well as the possibility of introducing errors that are not such from the point of view of the compiler - a classic problem of replacing the assignment sign on an equal sign or vice versa. Also, as a flaw, which is the flip side of effective code compilation, the absence of runtime error control is indicated, which requires the programmer to explicitly control errors in the program.



‚ÄúMISRA C‚Äù allows the use of both C99 and C90. There are no recommendations regarding the C11 / C18 standard only because at the time of finalizing the text ‚ÄúMISRA C‚Äù, the ISO / IEC working group had not yet had time to approve the new version of C, and referring to an unapproved standard would be a bad idea. Therefore, of course, C11 / C18 may well be used.



The standard attaches great importance to the selection of tools directly: compilers and static analyzers. At the same time - not by accident - the IEC 61508, ISO 26262 [9] and DO-178C [10] standards are mentioned. If we add to this list the standards EN 50128 [11] and IEC 62304 [12], which are also referred to in ‚ÄúMISRA C,‚Äù and which relate to the software development life cycle, then, in general, the circle will close. All other standards related to the functional safety of software contain clarifications or specialized versions of those principles and approaches that are described in the listed documents. However, I do not want to say at all that there is no need to study other documents - of course, standards such as, for example,IEC 60880 [13] is very important from the point of view of applying certain tools and methods to solving a certain class of problems.



There are two types of ‚ÄúMISRA C‚Äù directions: rules and directives.



Rules refer to such indications, compliance with which can be checked by analyzing the source text without any other sources of information. Static analyzers should be able to strictly check the implementation of the rules. Directives include instructions that cannot be formulated as strictly as rules. To analyze code compliance with the directive, code alone is not enough: you may need to study the design documentation.



Static analyzers can help during the checks, but the results of their work must be correctly interpreted.



All directions - rules or directives - are categorized according to one of three degrees of importance: mandatory, required and recommended.



If a code written in C is declared as complying with the ‚ÄúMISRA C‚Äù standard, then it must, without any reservations or exceptions, satisfy all the mandatory instructions of the standard.



Similarly, a C code declared as conforming to ‚ÄúMISRA C‚Äù should satisfy all the ‚Äúrequired‚Äù guidelines of the standard. The difference is that deviations are allowed for indications of this degree, each of which must be clearly documented. An example of documentation of a deviation is given in one of the appendices of the standard. For some projects, the ‚Äúrequired‚Äù directions by decision of the customer or developer may be equated to ‚Äúmandatory‚Äù.



And finally, the ‚Äúrecommended‚Äù directions can be violated without documented explanations. However, it should be remembered that these instructions were developed on the basis of tremendous experience, so they should not be ignored in any case. The standard advises to draw up a document explaining the reason for non-compliance with recommendations. Not only that, for some projects, ‚Äúrecommended‚Äù directions by decision of the customer or developer can be equated to ‚Äúrequired‚Äù or even ‚Äúmandatory‚Äù.



Rules, regardless of their importance, may or may not be decidable. Solvable rules include those for which the static analyzer can tell you exactly if the code matches the rule or not. But there are constructions for which, in order to draw such a conclusion, it is necessary to have data that can be known only during the execution of the code. In this case, the rule is considered unsolvable, and the opinion of the static analyzer about the compliance or non-compliance of the code with one or another unsolvable rule is not reliable. Therefore, to determine compliance, a dynamic analyzer is needed.



And finally, to completely confuse you, the last classification: according to the scale of analysis, the rules are divided into modular and system. Modular rules are those rules, for which it is enough to check the compilation unit, i.e. separate file. As it‚Äôs not hard to guess, to verify compliance with the system scale rule, it is necessary to analyze two or more compilation units, and most often, the entire source text of the project.

The rules on their scope are divided into twenty-two groups, and the directives into four groups. There are instructions that relate to data types, to working with pointers, to control statements, to preprocessor directives, etc.



An example of a ‚Äúrequired‚Äù rule, included in the ‚ÄúStandard Libraries‚Äù group, of a decidable compilation unit scale: the memory allocation and deallocation functions defined in the header file <stdlib.h> of the standard C library should not be used. The purpose of this rule is quite obvious: to reduce such common programming errors as memory leak, freeing unallocated memory, access to allocated memory before initialization, etc. Suppose you still need to use the heap. Since the rule is ‚Äúrequired‚Äù and not ‚Äúmandatory‚Äù, it can be ‚Äúviolated‚Äù by documenting the rationale for why this is done. But in this case, the binding rules from the ‚ÄúResources‚Äù group, unsolvable, of the scale of the system come into force:



  • all resources allocated dynamically using the standard C library must be explicitly released;
  • a memory area should only be freed if it is allocated using the functions of the standard C library.


As you can see, the second of these rules to some extent narrows the use of the free () function in comparison, for example, with the malloc () function.



In conclusion of this brief review of the ‚ÄúMISRA C‚Äù standard, I would like to emphasize that the study of this document, in my opinion, is mandatory for any programmer who uses the C language in developing critical applications. Moreover, regardless of whether this standard is implemented in your organization or not. At a minimum, it will allow you to deepen your knowledge of C and understand many of the nuances of this language. Since you need a personal or corporate license to read this document, you can pay attention to the JPL coding rules developed under his influence [6], mentioned above, and a brief but effective and fairly well-known set of ten rules by Gerard J. C ++ language



When I drew up the plan for this article, there was no section on C ++. My considerations were simple: there was no need to encourage the use of such a non-deterministic technology by mentioning it in an article on functional safety. But, in the course of work on the material, I still decided that my opinion was my personal opinion, and it would be wrong to leave such a widespread language outside the brackets. Without information about the limitations of C ++, the article would be clearly incomplete.



The standard GOST R IEC 61508-7-2012 [3] in Appendix G briefly defines (HR recommendation for UPB3 and UPB4) the basic principles of writing object-oriented code regardless of the programming language, for example:



  • Inheritance is allowed only to clarify the base class;
  • The depth of inheritance should be limited;

The standard strongly recommends (HR) the use of proven design patterns (templates) and software frameworks (frameworks). Indeed, wireframes and automatically generated code are probably the main ways to use C ++ in critical systems. Since the late 1990s, the popularity of frameworks with the generation of code from UML models, which are convenient for using algorithms based on state machines, has grown. Then, when the power of embedded systems made it possible to use rich UX, advanced communication and other technologies, the Qt framework also became widespread. Patterns for different design stages are dedicated, for example, to Bruce Paul Douglass's ‚ÄúReal-Time Design Patterns‚Äù [15].Examples of frameworks include OXF from IBM Rational and RXF from Willert Software Tools.



Be that as it may, The Motor Industry Software Reliability Association did not remain aloof from the process and contributed to the development of approaches to safe coding in C ++ by releasing the MISRA C ++: 2008 standard [5]. Despite the fact that several years ago the C ++ 11 standard was adopted [16], MISRA C ++ is still based on C ++ 03 [17], since at the time of approval it was the current edition. MISRA C ++ expressly states that the developers of the document do not intend to promote the C ++ language. Their goal is the recommended IEC 61508 definition of the C ++ subset required for various SILs in the development of critical software.



Once again, C ++ is especially relevant when using frameworks in conjunction with automatic code generation from models of one type or another. The main goals of model-oriented development is to increase the level of abstraction (and therefore productivity) of developing complex applications, as well as reducing errors by automating code generation and analysis.



Conclusion



And finally, a few words need to be said about the static analysis of code written in C and C ++. As a matter of fact, one of the tasks of the relevant MISRA standards was to provide such rules that would be suitable for static analysis in automatic mode. The choice of suitable analyzers, from the point of view of MISRA standards, is practically not separable from the choice of compilers.



There are a considerable number of both freely distributed and commercial static analyzers. Moreover, some compilers have built-in capabilities to analyze the implementation of restrictive rules.



Among non-commercial products, I would single out the popular cppcheck utility, the use of which can find a lot of useful information in the public domain, for example, an article on the Habrahabr resource. As for the analysis of code for compliance with MISRA C and MISRA C ++, a list of possible tools can be found in the corresponding Wikipedia articles.
