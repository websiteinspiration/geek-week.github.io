iOSストーリーボード：長所と短所の分析、ベストプラクティス



Appleは、開発者がiOSアプリケーションの画面とそれらの間の関係を視覚化できるように、ストーリーボードを作成しました。誰もがこのツールを気に入ったわけではなく、正当な理由があります。ストーリーボードを批判する多くの記事に出会いましたが、ベストプラクティスを考慮して、すべての長所と短所の詳細で公平な分析を見つけていません。結局自分で書くことにした。

ストーリーボードを使用することの欠点と利点を詳しく分析してみます。それらを比較検討した後、プロジェクトでそれらが必要かどうかに関係なく、有意義な決定を下すことができます。この決定は過激である必要はありません。一部の状況でストーリーボードが問題を引き起こす場合、他の状況ではそれらの使用が正当化されます。これは、タスクを効果的に解決し、シンプルで簡単に保守できるコードを書くのに役立ちます。

欠点から始めて、それらすべてがまだ関連性があるかどうかを分析しましょう。

欠点


1.ストーリーボードでは、変更をマージするときに競合を管理するのが難しい


ストーリーボードはXMLファイルです。コードほど読みにくいので、コード内の競合を解決するのはより困難です。ただし、この複雑さは、ストーリーボードの操作方法にも依存します。以下のルールに従うと、タスクを大幅に簡略化できます。

  • UI全体を1つのストーリーボードに配置するのではなく、複数の小さなストーリーボードに分割してください。これにより、競合のリスクなしに、また必然的に発生した場合に、ストーリーボードの作業を開発者間で配布できるようになり、解決のタスクが簡略化されます。
  • 複数の場所で同じビューを使用する必要がある場合は、独自のXibファイルを使用して別のサブクラスで選択します。
  • 小さな部分で行われる変更を処理する方がはるかに簡単なので、コミットをより頻繁に行う。

1つではなく複数のストーリーボードを使用すると、アプリケーションのマップ全体を1つのファイルで見ることができなくなります。しかし、多くの場合、これは必要ありません。現在取り組んでいる特定の部分で十分です。

2.ストーリーボードはコードの再利用を妨げます


プロジェクトでXibなしでストーリーボードのみを使用することについて話している場合、問題は確実に発生します。ただし、私の意見では、Xibはストーリーボードを操作するときに必要な要素です。それらのおかげで、再利用可能なビューを簡単に作成でき、コードでの作業にも便利です。

まず、ストーリーボードのXibで作成されたXibViewレンダリングを担当する基本クラスを作成しますUIView。

@IBDesignable
class XibView: UIView {
	
    var contentView: UIView?
}

XibViewロードするUIViewにXIBからcontentView、そのサブビューとして追加します。メソッドでそれをしましょうsetup()：

private func setup() {
    guard let view = loadViewFromNib() else { return }

    view.frame = bounds
    view.autoresizingMask = [.flexibleWidth, .flexibleHeight]
    addSubview(view)
    contentView = view
}

メソッドloadViewFromNib()は次のようになります。

private func loadViewFromNib() -> UIView? {
    let nibName = String(describing: type(of: self))
    let nib = UINib(nibName: nibName, bundle: Bundle(for: XibView.self))
    return nib.instantiate(withOwner: self, options: nil).first as? UIView
}

メソッドsetup()はイニシャライザで呼び出す必要があります。

override init(frame: CGRect) {
    super.init(frame: frame)
    setup()
}

required init?(coder aDecoder: NSCoder) {
    super.init(coder: aDecoder)
    setup()
}

3. UIViewControllersストーリーボードで作成されたイニシャライザは使用できません


Storyboardの場合、イニシャライザで依存関係を渡すことはできませんUIViewControllers。通常は次のようになります。

override func prepare(for segue: UIStoryboardSegue, sender: Any?) {
    guard segue.identifier == "detail", let detailVC = segue.destination as? DetailViewController else {
        return
    }
    let object = Object()
    detailVC.object = object
}

このコードは、SwiftGenやR.swiftのような識別子やツールを表す何らかの定数を使用するか、場合によってはPerformを使用することでより適切に実行できます。しかし、この方法では、文字列リテラルを削除して構文糖を追加するだけであり、発生する問題を解決しません。

  • 上記の例で構成する方法を知るにはどうすればよいDetailViewControllerですか？プロジェクトに不慣れで、この知識がない場合は、このコントローラーの説明を含むファイルを開いて、調査する必要があります。
  • プロパティDetailViewControllerは初期化後に設定されます。つまり、オプションにする必要があります。プロパティが等しい場合に対処する必要がnilあります。そうでない場合、アプリケーションは最も不都合な瞬間にクラッシュする可能性があります。プロパティを暗黙的に展開されたオプション（var object: Object!）としてマークできますが、本質は変わりません。
  • プロパティはvar、ではなくとしてマークする必要がありますlet。したがって、外部の誰かがそれらを変更したい状況が発生する可能性があります。DetailViewControllerそのような状況を処理する必要があります。

4.ストーリーボードが大きくなると、ナビゲーションが難しくなります


前に述べたように、すべてを1つのストーリーボードに入れる必要はありません。複数の小さなストーリーボードに分割することをお勧めします。ストーリーボードリファレンスの登場により、非常にシンプルになりました。
オブジェクトライブラリからストーリーボードにストーリーボード参照を追加し

5.ストーリーボードの読み込み時にXcodeが遅くなる


Storyboardに多数の制約のある多数の画面が含まれている場合、ロードには実際に時間がかかります。ただし、ここでも、大きなストーリーボードを小さなものに分割することをお勧めします。個々に、それらははるかに速くロードされ、それらと一緒に作業することがより便利になります。

6.ストーリーボードは壊れやすく、バグが原因で実行時にアプリケーションがクラッシュする可能性があります


主な弱点：

  • 識別子UITableViewCellとのエラーUICollectionViewCell。
  • セグエ識別子のエラー。
  • UIView存在しないサブクラスを使用する。
  • 同期IBActionsとIBOutletsコード。

このすべての問題および他のいくつかの問題により、実行時にアプリケーションがクラッシュする可能性があります。つまり、このようなエラーがリリースビルドに含まれる可能性があります。たとえば、ストーリーボードでセル識別子またはセグエを設定する場合は、使用する場所にコードにコピーする必要があります。 1か所で識別子を変更することにより、残りのすべてで識別子を変更する必要があります。あなたは単にそれを忘れるか、タイプミスをする可能性がありますが、アプリケーションの実行中にのみエラーについて学びます。

コード内の文字列リテラルを取り除くことにより、エラーの可能性を減らすことができます。これを行うには、識別子にセルクラス自体の名前を割り当てることUITableViewCellもUICollectionViewCellできます。たとえば、識別子ItemTableViewCellは文字列「ItemTableViewCell」です。コードでは、次のようなセルを取得します。

let cell = tableView.dequeueReusableCell(withIdentifier: String(describing: ItemTableViewCell.self)) as! ItemTableViewCell

UITableView適切なジェネリック関数に 追加できます：

extension UITableView {
	
    open func dequeueReusableCell<T>() -> T where T: UITableViewCell {
        return dequeueReusableCell(withIdentifier: String(describing: T.self)) as! T
    }
}

そして、セルを取得するのが簡単になります：

let cell: ItemTableViewCell = tableView.dequeueReusableCell()

ストーリーボードでセル識別子の値を指定するのを突然忘れた場合、Xcodeは警告を表示するので、それらを無視しないでください。

セグエ識別子については、列挙型を使用できます。特別なプロトコルを作成しましょう：

protocol SegueHandler {
	
    associatedtype SegueIdentifier: RawRepresentable
}

UIViewControllerこのプロトコルをサポートする場合は、同じ名前のネストされた型を定義する必要があります。これは、これUIViewControllerが処理できるすべてのセグエ識別子をリストします。

extension StartViewController: SegueHandler {
	
    enum SegueIdentifier: String {
        case signIn, signUp
    }
}

さらに、プロトコル拡張でSegueHandlerは、2つの関数を定義します。1つはUIStoryboardSegueそれSegueIdentifierに対応する値を受け入れて返します。もう1つは単にを呼び出しperformSegueて入力を受け入れますSegueIdentifier。

extension SegueHandler where Self: UIViewController, SegueIdentifier.RawValue == String {

    func performSegue(withIdentifier segueIdentifier: SegueIdentifier, sender: AnyObject?) {
        performSegue(withIdentifier: segueIdentifier.rawValue, sender: sender)
    }

    func segueIdentifier(for segue: UIStoryboardSegue) -> SegueIdentifier {
        guard let identifier = segue.identifier, let identifierCase = SegueIdentifier(rawValue: identifier) else {
            fatalError("Invalid segue identifier \(String(describing: segue.identifier)).")
        }
        return identifierCase
    }
}

そして今でUIViewControllerは、新しいプロトコルをサポートしているためprepare(for:sender:)、次のように作業できます。

extension StartViewController: SegueHandler {
	
    enum SegueIdentifier: String {
        case signIn, signUp
    }
	
    override func prepare(for segue: UIStoryboardSegue, sender: Any?) {
        switch segueIdentifier(for: segue) {
        case .signIn:
            print("signIn")
        case .signUp:
            print("signUp")
        }
    }
}

そして、次のようにセグエを実行します：

performSegue(withIdentifier: .signIn, sender: nil)

新しい識別子をに追加するとSegueIdentifier、Xcodeは間違いなくそれをで処理しswitch/caseます。

識別子セグエなどの文字列リテラルを取り除く別のオプションは、R.swiftなどのコード生成ツールを使用することです。

10.洗練されたコードレビュー


コードレビューは実際にはバグ検索ではないことに注意してください。はい、それらはコードを表示する過程で見つかりますが、主な目的は、長期的に問題を引き起こす可能性のある弱点を特定することです。ストーリーボードの場合、これは主に自動レイアウトの作業です。あいまいで見当違いがあるべきではありません。それらを見つけるには、ストーリーボードXMLで「ambiguous =“ YES”」と「misplaced =“ YES”」の行を検索するか、または単にStory BuilderをInterface Builderで開いて、赤と黄色のドットを探します。

ただし、これでは不十分な場合があります。アプリケーションの実行中に、制約間の競合も検出できます。同様の状況が発生した場合、これに関する情報がコンソールに表示されます。このようなケースは珍しくないので、彼らの調査も真剣に受け止められるべきです。

その他はすべて、要素の位置とサイズとデザインの対応、正しいバインディングでIBOutletsありIBActions、コードのレビュー用ではありません。

さらに、コミットをより頻繁に行うことが重要です。そうすれば、レビュー担当者が変更を細かく分割して表示しやすくなります。彼は何も見落とすことなく詳細をよりよく理解することができます。これは、コードレビューの品質に良い影響を及ぼします。

5.サイズクラス


毎年、ユーザーインターフェイスを調整する必要がある新しいデバイスが表示されます。特性のバリエーション、特に画面のサイズと向きに応じてUIを作成できるサイズクラスの概念は、これに役立ちます。

サイズクラスは、デバイス画面の高さ（h）と幅（w）をコンパクトで標準的なもの（CおよびR）で分類します。たとえば、iPhone 8にはサイズクラス（wC hR）が縦向き、（wC hC）が横向きで、iPhone 8 Plusにはそれぞれ（wC hR）と（wR hC）があります。残りのデバイスはここにあります。

サイズクラスごとに1つのストーリーボードまたはXibで、独自のデータセットを保存できます。アプリケーションは、実行時にデバイスと画面の向きに応じて適切なデータセットを使用し、現在のサイズクラスを識別します。一部のレイアウトパラメータがすべてのサイズクラスで同じである場合、デフォルトですでに選択されている「Any」カテゴリで構成できます。

たとえば、サイズクラスに応じてフォントサイズを設定します。ストーリーボードで縦向きに表示するためにiPhone 8 Plusデバイスを選択し、新しい条件を追加しますfont。幅が標準の場合（残りを「すべて」に設定）、フォントサイズは37にする必要があります。

ここで、画面の向きを変更すると、フォントサイズが大きくなります。iPhone8 Plusには横向きのサイズクラス（wR hC）があるため、新しい条件が機能します。ストーリーボードでは、サイズクラスに応じて、ビューの非表示、制約の有効化/無効化、値の変更constantなどを行うことができます。これをすべて行う方法については、こちらをご覧ください。

上のスクリーンショットでは、レイアウトを表示するためのデバイスの選択について下部パネルに注目する価値があります。これにより、任意のデバイスのUIの適応性と任意の画面方向をすばやく確認でき、現在の構成のサイズクラス（デバイス名の横）も表示されます。とりわけ、右側にボタンがあります« Vary for Traits」その目的は、幅、高さ、または幅と高さの特定のカテゴリに対してのみ、同時に特性のバリエーションを有効にすることです。たとえば、サイズクラス（wR hR）の iPadを選択し、[Vary for Traits]をクリックして、幅と高さの横にあるチェックボックスをオンにします。これで、以降のすべてのレイアウト変更は、（wR hR）のデバイスにのみ適用され、[ Done Varying ]をクリックします。
