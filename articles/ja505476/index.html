理論的紹介




戦車にいる人のために、たとえば上記の記事のいずれかを読んでいない場合は、Denoが何であるかを教えます。 Denoは、JavaScriptとTypeScriptで実行されるランタイムランタイムです（TypeScriptはそのまま使用できます。TypeScriptコンパイラバージョンはDenoバージョンにハードワイヤードされています。Denoバージョンを変更した場合にのみ変更できます）。これは、V8エンジンとRustプログラミング言語に基づいています。 Denoは、Node.jsの作成者であるRyan Dahlによって作成されました。Denoの主な特徴は、パフォーマンスとセキュリティです。 Denoは2018年に発表され、多くの情報源が言及し忘れていることは、Golangの影響を大きく受けていた（そして実際、元々はGolangで書かれていたが、後でRustで書き直す必要があった）。

標準ライブラリは標準Goライブラリのモデルに基づいて作成され、多くのツールまたはそれらの実装はGolangからDenoに切り替わりました。たとえば、npmのようなエコシステムの欠如と、初期アセンブリの段階でWebリソースから直接ライブラリをポンプする（これにより、Nodeでなんらかの混乱が発生します）。当然のことながら、Goを使用した同様のシステムに慣れていない開発者のjs）。それで、今、デノは誰のためですか？Golangのアイデアやツールなど、型付き言語が好きだが、もっと単純なものに裏付けをしたい場合は、実際にDenoを試してみるときです。ただし、プラットフォームが「熟成」するまで待機したい場合は、さらに多くのライブラリとStackOverflowへの回答が表示されるので、少し待機する必要があります。

作成を開始


Denoの若さにもかかわらず、その周りにエコシステムは形成されていません（もちろん、まだNode.jsエコシステムと比較することはできません）。これにより、独自のAPIを作成するための基本的なニーズをカバーできます。作成するものを決めましょう：バックエンドでDenoとTypeScriptを使用してシンプルなCRUD Apiを作成します（TypeScriptサポートがあるので、なぜそれを使用しないのか）、データベースとしてのPostgreSQL（一般に、MySQLを使用したかったのですが、誤ってアクセス権を壊しました） localhostで修正できませんでした。興味深い事実は、外部IPで、組み込みのPHP 7.4サーバーがMySQLデータベースに接続でき、Denoがパケットを順不同で表示したことです。リクエストは次のようになります。

方法 ルート 実行結果
取得する / api / todos / get すべてのタスクを取得する
役職 / api / todos / post 新規事業の創出
パッチ / api / todos /：id 特定のケースの更新
削除 / api / todos /：id IDでケースを削除


さて、クエリスキームを決定し、PostgreSQLサーバーを起動し（ここからダウンロードできる場合は、非常に優れたPostico GUIクライアントです）、crud_apiデータベースを作成しました。
次は何ですか？もちろん、SQLクエリまたはクライアントを使用して手でテーブルを作成できますが、Denoの移行でどのように機能するかを確認したいと思います。この場合、nessiaライブラリーが役立ちます。これにより、Laravelでの移行と同様の移行を作成できます。しかし、移行に入る前に、開発環境の編成について話しましょう。

先日、JetBrains Denoの公式開発プラグインがリリースされました。ここで見つけることができます。。Visual Studioでは、Denoプラグインをインストールする必要があり、プロジェクトフォルダーに.vscode設定フォルダーを作成します。ここで、settings.jsonでdenoサポートを有効にします（ただし、それがなくても機能するようです）。

    "deno.enable":true

これで準備段階は完了です。移行に戻ることができます。最初に、ライブラリのドキュメントから初期化コマンドを入力する必要があります。

deno run --allow-net --allow-read --allow-write https://deno.land/x/nessie/cli.ts init

ライブラリの構成ファイルを作成します。これを使用して、接続するDBMSおよびデータベースを決定できます。
最初に、ファイル内にPostgreSQL、MySQL、Sqliteの3つのテンプレートが作成されます（デフォルトでは、PostegreSQLの設定がエクスポートされます）。

その結果、nessie.config.tsファイルは次のようになります。

さて、移行ファイルを作成して、作成するテーブルとデータを指定して編集できるようになりました。これを行うには、次のコマンドを入力します。

deno run --allow-net --allow-read --allow-write https://deno.land/x/nessie/cli.ts make create_todo

おそらく、経験の浅い読者は、なぜそれほど多くのフラグがあるのか​​についてすでに疑問を抱いているでしょう。これはDenoの安全です。コマンドを実行するには、コマンドに与えるアクセスレベルを指定する必要があります（これは、私たちが待ち望んでいた独創的なセキュリティシステムです）。これらのフラグは、多くの場合非常に鋭い焼き付けを引き起こします-それらは印刷する必要がありますか？常に完全な権限で起動するbashスクリプトを追加しますか？このシステムはいつか変わると思いますが、今のところはそうです。このように.migrations、移行ファイルは私たちのフォルダーに表示され、移動中に編集できます。次のように編集しました。

export const up = (): string => {
  return "CREATE TABLE todo (id serial, text text, done boolean)";
};

export const down = (): string => {
  return "DROP TABLE todo"
};

その後、次のコマンドを使用してデータベースに移行できます。

deno run --allow-net --allow-read https://deno.land/x/nessie/cli.ts migrate

nessiaにはより高度な移行オプションがありますが、少し簡単にすることにしました。さらに、独自のモデルの作成に進むことができます。

私たちはモデルと協力します


modelsリクエスト自体を 入れるフォルダーを作成します。最初config.tsに、データベースに接続するための設定を追加するファイルを作成する必要があります（はい、すでにそれらをnessie.config.tsで説明しましたが、そのファイルは移行に使用されました）。ファイル内の接続にはconfig.ts、deno-posgresライブラリを使用します。その結果、構成ファイルは次のようになります。

import { Client } from "https://deno.land/x/postgres/mod.ts";

const client = new Client({
  user: "isakura313",
  database: "deno_crud",
  hostname: "localhost",
  port: 5432,
});

export default client;

いいね！次に、モデルに直接渡します。SQLクエリを作成するために、DenoのKnexライブラリのポートであるDexライブラリを使用します。これにより、実行するSQLクエリをプログラムで決定できます。Dexが今回使用する必要がある方言を定義し、todoのインターフェースを定義することから始めましょう。

const dex = Dex({ client: "postgres" });

interface Todo {
  id?: number;   //? -     TypeScript
  text: string;
  done: boolean;
}

さて、これでパルプ自体に進むことができます。最初に、すべてのタスクを受け取るgetリクエストを定義します。非同期関数の実行を使用して、awaitを介してデータベースに接続し、クエリ結果を取得します。

async function getAllTodo() {
  await client.connect();
  const getQuery = dex.queryBuilder().select("*").from("todo").toString();
  const result = await client.query(getQuery);
  return result;
}

TypeScriptとc async / awaitに少しでも精通している場合は、ここで質問する必要はありません。しかし、さらに-データベースにケースを追加するpost-requestを記述する必要があります。

async function addTodo(todo: Todo) {
  await client.connect();
  const insertQuery = dex.queryBuilder().insert([todo]).into("todo").toString();
  return client.query(insertQuery).then(async () => {
    const getQuery = dex.queryBuilder().select("*").from("todo").where(
      { text: todo.text },
    ).toString();
    const result = await client.query(getQuery);
    const result_data = result.rows ? result.rows[0] : {};
    return result_data;
  });
}

上記のコードは短くすることができますが、行数を犠牲にしても、できるだけ単純で表現力のあるものにしようとしました。一般的に、次のことがそこで発生します-関数内で、データベースへの接続を作成し、挿入を組み込みます-要求、要求が発生し、その中で新しい要求が作成され、新しく作成されたケースまたは空のオブジェクトが返されます。

その他の機能-編集と削除、一緒にあげます。編集では、ケースを追加する場合と同じことが起こりますが、editTodoでは更新があります。削除では、ケースを削除して何も返しません。

async function editTodo(id: number, todo: Todo) {
  await client.connect();
  const editQuery = dex.queryBuilder().from("todo").update(todo).where({ id })
    .toString();
  return client.query(editQuery).then(async () => {
    const getQuery = dex.queryBuilder().select("*").from("todo").where(
      { text: todo.text },
    ).toString();
    const result = await client.query(getQuery);
    const result_data = result.rows ? result.rows[0] : {};
    return result_data;
  });
}

async function deleteTodo(id: number) {
  await client.connect();
  const deleteQuery = dex.queryBuilder().from("todo").delete().where({ id })
    .toString();
  return client.query(deleteQuery);
}

ワオ。それは私たちの機能をエクスポートするためだけに残っています、そしてあなたはサーバーとルーティングを構成し始めることができます：

export {
  addTodo,
  getAllTodo,
  editTodo,
  deleteTodo,
};

ルーティングとサーバー操作


ルートフォルダを作成し、そこにroutes.tsを置きます。ルートを操作するために、expressJSに触発されたライブラリーであるtraintrainを使用します。このライブラリーは、URLリクエストの操作、ルーティングの設定、その他多くの便利な機能を許可されています。関数とルーターをインポートします。

import { Router } from "https://deno.land/x/denotrain@v0.5.0/mod.ts";
import { addTodo, getAllTodo, editTodo, deleteTodo } from "../models/models.ts";

const api = new Router();

getおよびpostメソッドを追加します。

api.get("/", (ctx) => {
  return getAllTodo().then((result) => {
    return result.rows;
    // 
  });
});

api.post("/", (ctx) => {
  const body = {
  //  
    text: ctx.req.body.text,
    done: ctx.req.body.done,
  };

  return addTodo(body).then((newTodo) => {
    ctx.res.setStatus(201); //   "Created"
    return newTodo;
  });
});

Ctxは、応答コンテキストを担当する変数です。コメントに何を追加すればいいのかわかりません。私の意見では、すべてがすでに明白になっています。パッチを追加してメソッドを削除し、APIをエクスポートするだけです。

api.patch("/:id", (ctx) => {
  const todo = {
    text: ctx.req.body.text,
    done: ctx.req.body.done,
  };

  return editTodo(ctx.req.params.id as number, todo).then((result) => {
    return result;
  });
});

api.delete("/:id", (ctx) => {
  return deleteTodo(ctx.req.params.id as number).then(() => {
    ctx.res.setStatus(204);
    return true;
  });
});

export default api;

私たちのサーバーを上げるだけです。ルートフォルダーに、server.tsdenotrainからアプリケーションをインポートするファイルを作成し、サーバーを起動します。

import {Application} from "https://deno.land/x/denotrain@v0.5.0/mod.ts";
import api from "./routes/routes.ts";

const app = new Application({port: 1337}) //      1337

app.use("/api/todos", api) //  ,      
app.run() //

アプリケーションで問題が発生する可能性があります。コンパイルしたサーバーを毎回手動で再起動したくないので、nodemon- Denomonの類似物を使用します。これにより、コードが忠実に再コンパイルされ、サーバーが再起動します。この場合、次のコマンドが適しています。

denomon --allow net,read server.ts

私の感想

Denoでの執筆は本当に楽しかったです。 TypeScriptと十分に構築された開発環境の助けを借りて、彼の方向性にはいくらか懐疑的ですが、できるだけ予測可能に機能する信頼性の高いサービスを作成することで、素晴らしい結果を得ることができます。まだ成長の余地があり（たとえば、プロジェクトの再コンパイルの速度が遅いなど）、真のプロダクションにDenoを使用するのは時期尚早ですが、恐竜がNode.jsから最高のものを採用し、アーキテクチャの奇妙さを少し変えて、その開発を継続できることを願っています。
