Zabbix - Erweiterung der Makrogrenzen

Als ich eine Lösung für den Kunden entwickelte, gab es zwei Aufgaben, die ich wunderschön und mit regulären Zabbix-Funktionen lösen wollte.

Aufgabe 1. Verfolgen der aktuellen Firmware-Version auf Mikrotik-Routern.

Die Aufgabe ist einfach zu lösen - indem Sie der HTTP-Vorlage einen Agenten hinzufügen. Der Agent erhält die aktuelle Version von der Mikrotik-Website, und der Trigger vergleicht die aktuelle Version mit der aktuellen und gibt im Falle einer Diskrepanz eine Warnung aus.

Wenn Sie 10 Router haben, ist dieser Algorithmus nicht kritisch, aber was tun mit 3000 Routern? 3000 Anfragen an den Server senden? Natürlich wird ein solches Schema funktionieren, aber die Idee von 3000 Abfragen passte nicht zu mir, ich wollte eine andere Lösung finden. Darüber hinaus hatte ein solcher Algorithmus immer noch einen Nachteil: Die andere Seite kann so viele Anforderungen von einer IP für einen DoS-Angriff berechnen, dass sie ihn einfach verbieten kann.

Aufgabe 2Verwenden einer Autorisierungssitzung in verschiedenen HTTP-Agenten.

Wenn Sie über einen HTTP-Agenten Informationen von "geschlossenen" Seiten erhalten müssen, benötigen Sie ein Autorisierungscookie. Zu diesem Zweck gibt es normalerweise ein Standardautorisierungsformular mit einem Login / Passwort-Paar und der Einstellung der Sitzungs-ID im Cookie.

Es gibt jedoch ein Problem: Sie können nicht von einem Element des HTTP-Agenten auf Daten eines anderen Elements zugreifen, um diesen Wert im Header zu ersetzen.

Es gibt auch ein „Web-Szenario“, das eine weitere Einschränkung aufweist und keinen Empfang von Inhalten zur Analyse und weiteren Speicherung zulässt. Sie können nur das Vorhandensein der erforderlichen Variablen auf den Seiten überprüfen oder die zuvor erhaltenen Variablen zwischen den Schritten des Web-Skripts übergeben.

Nach einigem Nachdenken über diese Aufgaben habe ich mich für die Verwendung von Makros entschieden, die in jedem Teil des Überwachungssystems perfekt sichtbar sind: in Vorlagen, Hosts, Triggern oder Elementen. Makros können über die Webinterface-API aktualisiert werden.

Zabbix verfügt über eine gute und detaillierte API-Dokumentation. Für den Datenaustausch über die API wird das Json-Datenformat verwendet. Details finden Sie in der offiziellen Dokumentation .

Die Abfolge der Aktionen zum Abrufen der benötigten Daten und zum Aufzeichnen in einem Makro ist in der folgenden Abbildung dargestellt.



Schritt 1


Der allererste Schritt kann aus einer Aktion oder mehreren Aktionen bestehen. In den ersten Schritten wird die gesamte Grundlogik festgelegt, und die Hauptschritte sind die letzten drei Schritte.

In meinem Beispiel bestand der erste Schritt darin, die Autorisierungscookies auf der TK-Anlage für die erste Aufgabe zu erhalten. Für die zweite Aufgabe habe ich die Nummer der aktuellen Mikrotik-Firmware-Version erhalten.

URL der aktuellen Mikrotik-Firmware-Versionen
  • upgrade.mikrotik.com/routeros/LATEST.6 - URL der aktuellen Stable-Version
  • upgrade.mikrotik.com/routeros/LATEST.6fix — URL LTS

Der erste Schritt ist für jeden Fall völlig individuell und die Logik seiner Arbeit kann unterschiedlich sein. Es hängt alles von Ihrer Aufgabe ab.
Behalten Sie bei der Arbeit mit Web-Skripten im Auge, welche Methode Sie benötigen, um die Antwort zu erhalten. HTTP - Response - Header oder die Antwort Körper selbst ohne Header?
Wenn Autorisierungscookies benötigt werden, legen Sie die Antwortmethode für Header wie im Fall von Asterisk fest.

Wenn Sie Daten benötigen, wie es der Fall mit der Antwort des mikrotik Server ist, legen Sie die Antwortkörper ohne Header.

Schritt 2


Wir gehen zum zweiten Schritt über. Erhalten einer Autorisierungssitzung:

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc": "2.0",
    "method": "user.login",
    "params": {
        "user": "Admin"
        "password": "zabbix"
    },
    "id": 1,
    "auth": null
}

jsonrpc ist die Version des verwendeten JSON-RPC-Protokolls.
Zabbix implementiert JSON-RPC Version 2.0;

  • Methode - die aufgerufene Methode;
  • params - Parameter, die von der Methode übergeben werden;
  • id - eine beliebige Anforderungskennung;
  • auth - Benutzerauthentifizierungsschlüssel; Als hätten wir es noch nicht, setzen wir es auf null.

Um mit der API zu arbeiten, habe ich ein separates Konto mit eingeschränkten Rechten erstellt. Erstens müssen Sie keinen Zugriff darauf gewähren, wo Sie nicht müssen. Und zweitens konnte vor Version 5.0 das durch das Makro angegebene Passwort gelesen werden. Wenn Sie das Zabbix-Administratorkennwort verwenden, ist das Administratorkonto dementsprechend leicht zu stehlen.

Dies gilt insbesondere dann, wenn Sie mit der API über Skripts von Drittanbietern arbeiten und nebenan Anmeldeinformationen speichern.

Seit Version 5.0 wird die Option zum Ausblenden des im Makro gespeicherten Kennworts angezeigt.



Überprüfen Sie beim Erstellen eines separaten Kontos zum Aktualisieren von Daten über die API, ob auf die benötigten Daten über die Weboberfläche zugegriffen werden kann und ob eine Aktualisierung möglich ist. Ich habe nicht überprüft, und dann konnte ich lange Zeit nicht verstehen, warum das benötigte Makro für die API nicht sichtbar war.



Nachdem Sie die Autorisierung in der API erhalten haben, erhalten Sie eine Liste der Makros.

Schritt 3


Die API erlaubt es nicht, das Host-Makro nach Namen zu aktualisieren. Dazu müssen Sie zuerst die Makro-ID abrufen. Um eine Liste der Makros für einen bestimmten Host zu erhalten, müssen Sie außerdem die ID dieses Hosts ermitteln. Dies ist eine zusätzliche Anforderung. Sie können das reguläre Makro {HOST.ID} in der Anforderung nicht verwenden. Die Einschränkung entschied sich, dies zu umgehen:



Ich habe ein lokales Makro mit der ID dieses Hosts erstellt. Es ist sehr einfach, die Host-ID über die Weboberfläche herauszufinden.

Die Antwort mit einer Liste aller Makros eines bestimmten Hosts kann durch eine Vorlage gefiltert werden:

regex:{"hostmacroid":"([0-9]+)"[A-z0-9,":]+"{\$MIKROTIK_VERSION}"



Auf diese Weise erhalten wir die ID des benötigten Makros, wobei MIKROTIK_VERSION der Name des gesuchten Makros ist. In meinem Fall wird nach dem Makro MIKROTIK_VERSION gesucht , das dem Host zugewiesen wurde.

Die Anfrage selbst sieht folgendermaßen aus:

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.get",
    "params":{
        "output":"extend",
        "hostids":"{$HOST_ID}"
    },
    "auth":"{sid}",
    "id":1
}

Die Variable {sid} wurde im zweiten Schritt erhalten und wird ständig verwendet, wenn mit der API-Schnittstelle gearbeitet werden muss.

Final 4 STEP - Makro-Update


Jetzt kennen wir die Makro-ID, die aktualisiert werden muss, das Autorisierungscookie oder die Firmware-Version des Routers. Sie können das Makro selbst aktualisieren.

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.update",
    "params":{
        "hostmacroid":"{hostmacroid}",
        "value":"{mikrotik_version}"
    },
    "auth":"{sid}",
    "id":1
}

{mikrotik_version} - der im ersten Schritt erhaltene Wert. In meinem Beispiel - der Version der aktuellen Firmware mikrotik
{hostmacroid} - wurde der Wert im dritten Schritt ermittelt - der Makro-ID, die wir aktualisieren.

Schlussfolgerungen


Der Ansatz zur Lösung des Problems mit regulären Funktionen ist um ein Vielfaches komplizierter und länger. Vor allem, wenn Sie sich mit Programmierung auskennen und schnell die notwendige Logik in das Skript werfen können.

Der offensichtliche Vorteil dieses Ansatzes ist die „Portabilität“ der Lösung zwischen verschiedenen Servern.

Persönlich ist es für mich seltsam, dass der HTTP-Agent nicht auf die Daten eines anderen Elements zugreifen kann und dass er im Anforderungshauptteil oder in den Headern ersetzt wird [ ZBXNEXT-5993 ].
