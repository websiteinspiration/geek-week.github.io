サーバーには何がありますか


各スレッドは、TCP接続を確立し、データを送信し、応答を待機し、接続を閉じ、再度開き、データを送信します。一見すると、すべてのデータが同時に送信されますが、トランスポート層の性質、安全な接続（HTTPS）を確立してDNS（burpの場合はない）と多くの層を解決する必要があるため、HTTPリクエスト自体は同期的に到着せず、一貫性がありませんネットワークデバイスに送信される前にデータを渡す抽象化。ミリ秒になると、これは重要な役割を果たすことができます。

サーバーには何がありますか


Webサーバーは要求を順次受信し（キーワード）、優先順位に従って応答を処理します。この機能は、いくつかのステップで攻撃するために（最小時間で2つのアクションを連続して実行する必要がある場合）、または攻撃の成功率を高めるために、最初のリクエストでサーバーの速度を落とすなどに使用できます。
トリック-特にINSERT / UPDATEが使用されている場合は、DBMSをロードすることでサーバーがリクエストを処理しないようにすることができます。リクエストが多ければ多いほど負荷が「遅くなる」可能性があるため、このレースで勝つ可能性が高くなります。

HTTPリクエストを2つの部分に分割する


まず、HTTPリクエストがどのように生成されるかを覚えておいてください。さて、ご存じのとおり、最初の行はプロトコルのメソッド、パス、バージョンです。

GET / HTTP/1.1

次は改行前のヘッダーです。 しかし、WebサーバーはHTTPリクエストが終了したことをどのようにして知るのでしょうか。 例を見てみましょう。「nc google.com 80」と入力すると、Enterキーを押しても何も起こりません。もう一度クリックすると、答えが表示されます。 つまり、WebサーバーがHTTP要求を受け入れるためには、2つの改行が必要です。正しいリクエストは次のようになります 。POSTメソッドの場合（Content-Lengthを忘れないでください）、正しいHTTPリクエストは次のようになります。 または 、コマンドラインから同様のリクエストを送信してみてください。

Host: google.com
Cookie: a=1




GET / HTTP/1.1
Host: google.com



GET / HTTP/1.1\r\nHost: google.com\r\n\r\n



POST / HTTP/1.1
Host: google.com
Content-Length: 3

a=1



POST / HTTP/1.1\r\nHost: google.com\r\nContent-Length: 3\r\n\r\na=1



echo -ne "GET / HTTP/1.1\r\nHost: google.com\r\n\r\n" | nc google.com 80

その結果、HTTPリクエストが完了したので、レスポンスを受け取ります。ただし、最後の\ n文字を削除すると、回答が得られません。

実際、多くのウェブサーバーは\ nを転送として使用する必要があるだけなので、\ rと\ nを入れ替えないことが重要です。そうしないと、それ以上のトリックが機能しない可能性があります。

それは何を与えるのですか？リソースへの多くの接続を同時に開いて、HTTPリクエストの99％を送信し、最後のバイトを未送信のままにすることができます。サーバーは、改行の最後の文字に到達するまで待機します。データの主要部分が送信されたことが明確になった後、最後のバイト（またはいくつか）を送信します。

これは、ファイルのアップロードが必要な場合など、大規模なPOSTリクエストに関しては特に重要です。しかし、わずかなリクエストでも、数キロバイトの情報を同時に配信するよりも数バイトの配信の方がはるかに高速であるため、これは理にかなっています。

リクエストの2番目の部分を送信するまでの遅延


Vlad Roskov の調査によると、リクエストを分割するだけでなく、データの本体部分と最後の部分を送信する間に数秒の遅延を設けることも理にかなっています。また、Webサーバーが完全にリクエストを受信する前でも、リクエストの解析を開始するためです。



サーバーには何がありますか


たとえば、HTTPリクエストヘッダーを受信すると、nginxはそれらの解析を開始し、欠陥のあるリクエストをキャッシュします。最後のバイトが到着すると、Webサーバーは部分的に処理された要求を受け取り、それをアプリケーションに直接送信するため、要求の処理時間が短縮され、攻撃の可能性が高まります。

整然としたトランザクション（シリアライズ可能）-トランザクションが厳密に順次実行されるようにしますが、これはパフォーマンスに影響を与える可能性があります。

競合状態のセッション機能


セッションの特徴の1つは、それ自体がレースの利用を妨げることです。たとえば、PHPでは、session_start（）の後でセッションファイルがロックされ、そのロック解除はスクリプトの最後でのみ発生します（session_write_closeへの呼び出しがない場合）。この時点でセッションを使用する別のスクリプトが呼び出されると、待機します。

この機能を回避するには、簡単なトリックを使用して、必要な回数だけ認証を行います。Webアプリケーションで1人のユーザーに対して多くのセッションを作成できる場合は、すべてのPHPSESSIDを収集して、各リクエストに独自の識別子を作成します。
