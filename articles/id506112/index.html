Cara memvisualisasikan pengeluaran harian untuk solusi cloud GCP


Pelanggan tidak suka membayar lebih dari yang direncanakan - justifikasi terperinci tentang biaya merupakan bagian integral dan penting dari implementasi teknologi cloud.

Google Cloud Platform menyediakan berbagai paket harga untuk sumber daya yang digunakan. Misalnya, biaya GCE tergantung pada konfigurasi komputer ( CPU , memori, modul jaringan, hard drive). Biaya Google Kubernetes Engine (GKE) dan Google Cloud Dataproc didasarkan pada semua node yang berjalan di Google Compute Engine (GCE). Biaya yang tersisa dapat dihitung dengan formula yang rumit dan rumit. Merencanakan anggaran menjadi semakin sulit, terutama jika Anda menggunakan beberapa teknologi cloud. Pemantauan dan pelaporan tepat waktu menjadi semakin berharga karena pengeluaran untuk infrastruktur meningkat.

Kemungkinan pengecekan harian atas laporan pengeluaran juga akan memungkinkan penyesuaian kapasitas yang didistribusikan secara tepat waktu, dan tagihan akhir di akhir bulan tidak akan menimbulkan kejutan.

Latar Belakang


Proyek seringkali membutuhkan pelaporan yang luas untuk meningkatkan efisiensi dan melacak biaya. Meskipun GCP memungkinkan Anda untuk mengekspor data pengeluaran ke BigQuery untuk analisis komprehensif lebih lanjut, pemrosesan manual BigQuery seringkali tidak cukup. Idealnya, kita harus dapat memvisualisasikan data, memfilternya dan menumpuknya sesuai dengan parameter tambahan.

Filter umum:

  • Jenis sumber daya;
  • Suatu jangka waktu;
  • Departemen dan tim;
  • Perbandingan dengan periode sebelumnya.

Solusi pertama, yang didasarkan pada rekomendasi Google, adalah menggunakan Google Data Studio . Implementasinya sangat sederhana - Anda hanya perlu mengkonfigurasi sumber data. Laporan sampel dan dasbor sudah diinstal sebelumnya, tetapi solusinya sendiri tidak cukup fleksibel. Saat membuat bagan dan grafik di Google Data Studio , Anda tidak bisa memasukkan formula, Anda harus memilih semua parameter secara manual.

Grafana adalah alat yang sederhana dan mudah dipahami untuk memantau dan menganalisis data. Dia mapan. Dasbor seperti itu akan bagus untuk memvisualisasikan data pembayaran. Pertanyaannya tetap bagaimana menghubungkan antarmuka web ke BigQuery (BQ) . Plugin BQ, open source, terlalu buggy. Akibatnya, banyak BQ-Grafana tidak cukup stabil. Selain itu, ada titik lain yang tidak nyaman - permintaan BQ mengembalikan data terlalu lama. Mengunggah

data ke PostgreSQL melalui CloudSQL ternyata merupakan solusi yang baik . DBMS ini memiliki plugin PostgreSQL resmi untuk Grafana . Dan hasil tes selanjutnya menunjukkan bahwa metode yang dipilih memiliki keunggulan jelas dalam kecepatan.

Alur kerja dapat secara skematis dijelaskan sebagai berikut. Alat tugas yang tertunda di kluster Kubernetes menjalankan tugas di Cloud Dataflow setiap 4 jam . Script untuk memuat data


Diperlukan skrip untuk mengunduh data dari BigQuery ke CloudSQL . Cloud DataFlow mendukung Python dan Javascript . Anda juga akan memerlukan perpustakaan Apache Beam , file json (diatur dalam variabel lingkungan GOOGLE_APPLICATION_CREDENTIALS dengan hak istimewa akun yang telah dikonfigurasi sebelumnya), dan file requirement.txt yang berisi daftar paket untuk instalasi (dalam kasus kami, kami hanya memerlukan satu paket nugget balok ). Skrip Python disajikan di bawah ini:

Teks tersembunyi
bq-to-sql :

args = parser.parse_args()
project = args.project
job_name = args.job_name + str(uuid.uuid4())
bigquery_source = args.bigquery_source
postgresql_user = args.postgresql_user
postgresql_password = args.postgresql_password
postgresql_host = args.postgresql_host
postgresql_port = args.postgresql_port
postgresql_db = args.postgresql_db
postgresql_table = args.postgresql_table
staging_location = args.staging_location
temp_location = args.temp_location
subnetwork = args.subnetwork
 
options = PipelineOptions(
        	flags=["--requirements_file", "/opt/python/requirements.txt"])
# For Cloud execution, set the Cloud Platform project, job_name,
# staging location, temp_location and specify DataflowRunner.
 
google_cloud_options = options.view_as(GoogleCloudOptions)
google_cloud_options.project = project
google_cloud_options.job_name = job_name
google_cloud_options.staging_location = staging_location
google_cloud_options.temp_location = temp_location
google_cloud_options.region = "us-west1"
worker_options = options.view_as(WorkerOptions)
worker_options.zone = "us-west1-a"
worker_options.subnetwork = subnetwork
worker_options.max_num_workers = 20
 
options.view_as(StandardOptions).runner = 'DataflowRunner'
 
start_date = define_start_date()
with beam.Pipeline(options=options) as p:
	rows = p | 'QueryTableStdSQL' >> beam.io.Read(beam.io.BigQuerySource(
                    	query='SELECT \
                        	billing_account_id, \
                        	service.id as service_id, \
                        	service.description as service_description, \
                        	sku.id as sku_id, \
                        	sku.description as sku_description, \
                        	usage_start_time, \
                        	usage_end_time, \
                        	project.id as project_id, \
                        	project.name as project_description, \
                        	TO_JSON_STRING(project.labels) \
                            	as project_labels, \
                        	project.ancestry_numbers \
                            	as project_ancestry_numbers, \
                        	TO_JSON_STRING(labels) as labels, \
                        	TO_JSON_STRING(system_labels) as system_labels, \
                        	location.location as location_location, \
                        	location.country as location_country, \
                        	location.region as location_region, \
                        	location.zone as location_zone, \
                        	export_time, \
                        	cost, \
                        	currency, \
                        	currency_conversion_rate, \
                        	usage.amount as usage_amount, \
                        	usage.unit as usage_unit, \
                        	usage.amount_in_pricing_units as \
                         	usage_amount_in_pricing_units, \
                        	usage.pricing_unit as usage_pricing_unit, \
                        	TO_JSON_STRING(credits) as credits, \
                        	invoice.month as invoice_month, \
                   	     cost_type \
FROM `' + project + '.' + bigquery_source + '` \
                        	WHERE export_time >= "' + start_date + '"',
                    	use_standard_sql=True))
	source_config = relational_db.SourceConfiguration(
          	              drivername='postgresql+pg8000',
                        	host=postgresql_host,
                        	port=postgresql_port,
                        	username=postgresql_user,
                        	password=postgresql_password,
                        	database=postgresql_db,
                        	create_if_missing=True,
                        	)
	table_config = relational_db.TableConfiguration(
                        	name=postgresql_table,
            	            create_if_missing=True
                        	)
	rows | 'Writing to DB' >> relational_db.Write(
    	source_config=source_config,
    	table_config=table_config
	)


Untuk konsistensi data, Anda perlu menentukan export_time maksimum , waktu ekspor ke PostgreSQL , dan kemudian memuat catatan dari BigQuery yang akan dimulai dengan cut-off waktu ini.

Setelah data dimuat, Anda perlu memperbarui tampilan terwujud. Karena proses ini membutuhkan waktu, ini akan memungkinkan Anda untuk membuat tampilan terwujud baru dengan struktur dan indeks yang sama, menghapus yang lama dan mengganti nama yang baru.

Membuat file JSON dengan izin SA


Akun yang Anda buat sebelumnya untuk alur kerja Cloud Dataflow juga digunakan dalam tugas Cron . Anda perlu menentukan perintah yang akan membuat kata sandi kunci pribadi untuk akun tersebut, yang kemudian diunggah ke kluster Kubernetes , di mana ia akan bertindak sebagai kata sandi tersembunyi untuk mengakses tugas Cron .

gcloud iam service-accounts keys create ./cloud-sa.json \
--iam-account "bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--project ${project}

Menyebarkan kata sandi rahasia di kluster K8s:

kubectl create secret generic bq-to-sql-creds --from-file=./cloud-sa.json
