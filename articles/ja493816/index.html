Gitガイド パーツ番号1：.gitディレクトリについて知っておく必要があるすべて



Gitの使用を開始することは、言語がわからない新しい国を訪れるようなものです。あなたがどこにいるのか、どこに行くのかは明らかですが、すべてうまくいきますが、もしあなたが道に迷ったら、大きな問題が起こります。

インターネットに投稿されたGitコマンドのチュートリアルはたくさんありますが、この記事では、Gitの作業は単にコマンドを学ぶだけではありません。

これは、Mail.ru Cloud SolutionsによるPierre de WulfブログのGitガイドの最初の部分です。 

新規ユーザーは、Gitに慣れるのが難しいと感じています。これは強力なツールですが、残念ながら、簡単に習得することはできません。多くの新しい概念、さまざまなアクションを実行するコマンド、ファイルがパラメーターとして渡されるかどうか、不明確なフィードバック...

これらの問題をすべて克服する唯一の方法は、Gitの仕組みを理解するために、単にgit commit / push以外のことを学ぶことです。

.Gitフォルダー


git initコマンドで新しいリポジトリを作成すると、Gitはマジックフォルダー.gitを作成します。これには、Gitが機能するために必要なものがすべて含まれています。Gitをプロジェクトから削除したいが、プロジェクトファイルをディスクに残したい場合は、.gitフォルダーを削除してください。誰がこれを必要とするかもしれませんが？

    ├── HEAD
    ├── branches
    ├── config
    ├── description
    ├── hooks
    │ ├── pre-commit.sample
    │ ├── pre-push.sample
    │ └── ...
    ├── info
    │ └── exclude
    ├── objects
    │ ├── info
    │ └── pack
    └── refs
     ├── heads
     └── tags


最初のコミット前の典型的な.gitフォルダーの内容は次のとおりです。

  1. HEAD-後で検討します。
  2. config — , , , url , , email . git config, .
  3. description — gitweb .
  4. hooks — , Git. , , / commit/rebase/pull… . push .
  5. info-exclude-リポジトリーに含めたくないファイルについて、ここで説明します。このファイルの機能は、リポジトリに転送されないことを除いて、.gitignoreファイルと同じです。実際には、通常、すべてのタスクには.gitignoreで十分です。

コミットの内容は何ですか？


ファイルを作成して変更をコミットするたびに、Gitはファイルをアーカイブしてデータ構造に保存します。アーカイブされたオブジェクトは一意の名前で作成され、オブジェクトフォルダーに保存されます。

オブジェクトフォルダーを調べる前に、コミットとは何かを明確にしましょう。コミットは、作業フォルダー内のファイルの現在の状態のナゲットですが、それだけではありません。

実際、変更をコミットすると、Gitは次の2つのことを行います。

  1. 作業フォルダー内のファイルが変更されていない場合は、スナップショットに圧縮ファイル（ハッシュ）の名前を追加するだけです。
  2. 作業フォルダー内のファイルが変更されている場合は、ファイルを圧縮し、オブジェクトフォルダーに配置して、圧縮ファイル（ハッシュ）の名前をスナップショットに追加します。

もちろん、ここではすべてがいくぶん単純化された方法で説明されていますが、これは進行中のプロセスを理解するには十分です。

スナップショットが作成されるとすぐに、それもアーカイブされ、ハッシュで名前が付けられてから、オブジェクトフォルダーに配置されます。

├── 4c
│ └── f44f1e3fe4fb7f8aa42138c324f63f5ac85828 // hash
├── 86
│ └── 550c31847e518e1927f95991c949fc14efc711 // hash
├── e6
│ └── 9de29bb2d1d6434b8b29ae775ad8c2e48c5391 // hash
├── info // let's ignore that
└── pack // let's ignore that too


これは、file_1.txtを作成してコミットした後のオブジェクトフォルダーです。ファイルのハッシュが「4cf44f1e ...」で始まる場合、Gitは「4c」という名前のサブフォルダーに「f44f1e ...」という名前で保存することに注意してください。したがって、ファイルは256のサブフォルダーに配置され、各ファイルにあまり多くのファイルが含まれることはありません。

ご覧のとおり、3つのハッシュがあります。1つはfile_1.txt用で、もう1つはコミット時に取得されるスナップショット用です。3番目の目的は何ですか？コミットもオブジェクトであるため、3番目のハッシュが作成されます。これもアーカイブされ、オブジェクトフォルダーに配置されます。

コミットは4つの要素で構成されることを覚えておく必要があります。

  1. 作業ディレクトリのスナップショットの名前（ハッシュ）。
  2. コメント。
  3. 誰がコミットを実行したかに関する情報。
  4. 親コミットのハッシュ。

コミットファイルを解凍するとどうなるか自分で確かめてください：

git cat-file -p 4cf44f1e3fe4fb7f8aa42138c324f63f5ac85828

そしてこれがあなたが見るものです：

tree 86550c31847e518e1927f95991c949fc14efc711
author Pierre De Wulf 
<test[@gmail.com](mailto:pierredewulf31@gmail.com)> 1455775173 -0500
committer Pierre De Wulf 
<[test@gmail.com](mailto:pierredewulf31@gmail.com)> 1455775173 -0500
commit A

予想どおり、スナップショットのハッシュ、作成者、およびコミットのコメントが表示されます。ここでは2つのことが重要です。

  1. スナップショット「86550 ...」のハッシュもオブジェクトであり、オブジェクトフォルダーで確認できます。
  2. これは最初のコミットなので、親コミットはありません。

写真には本当に何がありますか？

git cat-file -p 86550c31847e518e1927f95991c949fc14efc711
100644 blob e69de29bb2d1d6434b8b29ae775ad8c2e48c5391 file_1.txt

ここでは、オブジェクトのストレージにあるオブジェクトが表示されています。これは、画像内で唯一のオブジェクトです。

ブランチ、タグ、ヘッドは同じです


これで、Gitのすべてが正しいハッシュを通じて取得できることがわかりました。HEADを見てみましょう。だから何がありますか？

cat HEAD
ref: refs/heads/master

HEADは作業しているブランチのトップへのポインタであるため、ハッシュはありません。ファイルrefs / heads / masterを見ると、次のことがわかります。

cat refs/heads/master
4cf44f1e3fe4fb7f8aa42138c324f63f5ac85828
 

見覚えがある？当然、これは最初のコミットのハッシュです！これは、タグとブランチがコミットへの単なるポインタであることを示しています。これを理解すると、必要なすべてのタグ、必要なすべてのブランチを削除でき、それらが指しているコミットはそのまま残ります。唯一のことは、それにアクセスすることがより困難になることです。これについてもっと知りたい場合は、git bookをチェックしてください。

最後のコメント


それを読んだ後、Gitが行うことは作業フォルダーをアーカイブし、いくつかの追加情報と共にそれをオブジェクトフォルダーに配置することだけであることが明らかになります。Gitについて十分に理解している場合は、コミットに含めるファイルと含めないファイルを完全に制御できます。

コミットは作業フォルダのスナップショットではなく、コミットしたいファイルのスナップショットだと思います。そして、Gitはコミットしたいファイルのリストをどこに保存しますか？このリストをインデックスファイルに保存します。この問題について深く掘り下げることはしませんが、興味があれば、ここで詳細を確認できます。Mail.ru Cloud Solutionsの
