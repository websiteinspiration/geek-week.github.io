標準への注釈


このPEPは、Pythonの関数に任意の注釈（メタデータ）を追加するための構文を導入します。

正当化


Python 2.xの関数には、パラメーターに注釈を付けて値を返す組み込みの方法がありませんでした。このギャップに対処するために、多くのツールとライブラリが登場しています。PEP 318で説明されているデコレーターを使用するものもあれば、docstring関数を分析してそこで情報を探すものもあります。

このPEPは、メカニズムと構文のさまざまなバリエーションによって引き起こされる、これまでに存在していた混乱を減らすために、関数に注釈を付ける単一の標準的な方法を提供することを目的としています。

機能アノテーションの基本


Python 3.0の機能アノテーションについて説明する前に、まずそれらの機能について一般的な用語で説明しましょう。

  1. パラメータの注釈と関数の戻り値は完全にオプションです。
  2. 注釈は、コンパイル時に任意の式を関数のさまざまな部分に関連付ける方法にすぎません。

    Python自体は注釈に注意を払いません。唯一のことは、それらにアクセスできることです。これについては、以下の「機能アノテーションへのアクセス」セクションで説明します。

    注釈は、サードパーティのライブラリによって解釈された場合にのみ意味があります。関数アノテーションを使用すると、好きなことができます。たとえば、1つのライブラリが文字列注釈を使用して、改善された参照データを提供する場合があります。次に例を示します。

    def compile(source: "something compilable",
            filename: "where the compilable thing comes from",
            mode: "is this a single statement or a suite?"):
    ...

    別のライブラリでは、Python関数とメソッドアノテーションを使用して型の一致をチェックする場合があります。このライブラリは、注釈を使用して、予想される入力データのタイプと返されるデータのタイプを示すことができます。多分それはこのようなものになるでしょう：

    def haul(item: Haulable, *vargs: PackAnimal) -> Distance:
    ...

    もう一度繰り返します。どの例のアノテーションもそれ自体には意味がありません。彼らはサードパーティのライブラリとの組み合わせでのみ本当の意味を獲得します。
  3. 段落2からわかるように、このPEPは、組み込み型であっても、注釈を処理するための標準的なメカニズムを導入しようとはしません。この作業はすべてサードパーティのライブラリに割り当てられています。

構文


パラメーター


パラメータアノテーションは、パラメータ自体の名前に続くオプションの式です。

def foo (a: , b:  = 5):
    ...

疑似文法では、パラメーターは次のようになります：パラメーター[：式] [=式]。つまり、デフォルト値と同様に、注釈はオプションであり、常に後者の前にあります。等号を使用してデフォルト値を示すように、コロンは注釈に使用されます。デフォルト値のようなすべての注釈式は、関数定義が実行されるときに評価されます。「追加の」引数（つまり、* argsと** kwargs）を取得する場合も同じ構文を使用します。

def foo(*args: expression, **kwargs: expression):
    ...

ネストされたパラメーターの注釈は、最後のブラケットではなく、常にパラメーター名の後に続きます。ネストされたパラメータの各「名前」の注釈は不要です。

def foo((x1, y1: expression),
        (x2: expression, y2: expression)=(None, None)):
    ...

戻り値


これまでのところ、関数で戻り値の型に注釈を付ける方法の例は提供していません。これは次のように行われます：

def sum() -> expression:
    ...

つまり、リテラル->およびなんらかの式がパラメーターのリストに従うことができます。パラメータアノテーションと同様に、この式は関数定義の実行時に評価されます。

関数宣言の完全な文法は次のとおりです。

decorator: '@' dotted_name [ '(' [arglist] ')' ] NEWLINE
decorators: decorator+
funcdef: [decorators] 'def' NAME parameters ['->' test] ':' suite
parameters: '(' [typedargslist] ')'
typedargslist: ((tfpdef ['=' test] ',')*
                ('*' [tname] (',' tname ['=' test])* [',' '**' tname]
                 | '**' tname)
                | tfpdef ['=' test] (',' tfpdef ['=' test])* [','])
tname: NAME [':' test]
tfpdef: tname | '(' tfplist ')'
tfplist: tfpdef (',' tfpdef)* [',']

ラムダス


Lambda関数はアノテーションをサポートしていません。もちろん、構文を修正するには、引数を角括弧で「ラップ」する機能を追加しますが、次の理由により、このような変更は行わないことにしました。

  1. これは下位互換性に違反します。
  2. ラムダは定義により中立で匿名です
  3. ラムダは常に関数として書き直すことができます


機能の注釈にアクセスする


コンパイル後、__ annotations__属性を介して関数の注釈を使用できるようになります。この属性は、変数の名前とそれらに示されている注釈の値を比較する変更可能な辞書です。

__annotations__辞書には、特別な「return」キーがあります。このキーは、関数の戻り値にアノテーションも定義されている場合にのみ存在します。たとえば、次の注釈：

def foo (a: 'x', b: 5 + 6, c: list) -> max (2, 9):
    ...

__annotations__属性を介して次のように返されます：

{'a': 'x',
 'b': 11,
 'c': list,
 'return': 9}

キー名「return」が選択されたのは、パラメーター名と競合できないためです（パラメーター名としてreturnを使用しようとすると、SyntaxError例外が発生します）。

関数に注釈がない場合、または関数がラムダ式を介して作成された場合、__ annotations__属性は空になります。

ユースケース


注釈の議論中に、それらの使用のためのいくつかのオプションを検討しました。それらのいくつかはここに提示され、送信される情報の「クラス」に従ってグループ化されます。アノテーションを使用できる既存の製品とパッケージの例もここに含まれています。

  • タイプ情報の提供
    • タイプチェック
    • 期待される引数型と返される引数型に関するIDEのヒント
    • 関数のオーバーロード/ジェネリック関数[約 関数のオーバーロードは他の言語で人気があり、同じ名前のいくつかの関数が存在しますが、それらによって受け入れられるパラメーターの数は異なります]
    • 異なるプログラミング言語間の橋渡し
    • 適応
    • 述語論理関数
    • データベースクエリのマッピング
    • RPCパラメータマーシャリング[約 マーシャリングとは、RAMに格納された情報を、格納または送信に適した形式に変換するプロセスです。RPC-リモートプロシージャコール]
  • その他の情報の提供
    • パラメータと戻り値の文書化
