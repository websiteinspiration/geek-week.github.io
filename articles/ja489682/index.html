Chrome拡張機能でのクロスオリジン読み取りブロック（CORB）


Chromeの拡張機能を開発したことがある場合、その機能が停止している可能性があります。

その理由は、昨年から、コンテンツスクリプトからのクロスオリジンリクエストがChromeブラウザでブロックされるためです。つまり、拡張機能がコンテンツスクリプトからサードパーティのAPIに直接アクセスする場合、クロスオリジン読み取りブロック（CORB）のおかげで、そのようなリクエストはブロックされます。

ブロッキングの問題を解決するための推奨される方法は、コンテンツスクリプトからバックグラウンドスクリプトにリクエストを転送することです。ドキュメントの例：

古いコンテンツスクリプト：

var itemId = 12345;
var url = "https://another-site.com/price-query?itemId=" +
         encodeURIComponent(request.itemId);
fetch(url)
  .then(response => response.text())
  .then(text => parsePrice(text))
  .then(price => ...)
  .catch(error => ...)

新しいコンテンツスクリプト：

chrome.runtime.sendMessage(
    {contentScriptQuery: "queryPrice", itemId: 12345},
    price => ...);

新しいバックグラウンドスクリプト：

chrome.runtime.onMessage.addListener(
  function(request, sender, sendResponse) {
    if (request.contentScriptQuery == "queryPrice") {
      var url = "https://another-site.com/price-query?itemId=" +
          encodeURIComponent(request.itemId);
      fetch(url)
          .then(response => response.text())
          .then(text => parsePrice(text))
          .then(price => sendResponse(price))
          .catch(error => ...)
      return true;  // Will respond asynchronously.
    }
  });

また、バックグラウンドスクリプトからプロキシを作成してはならないことにも注意してください。データを受信するURLのコンテンツスクリプトからバックグラウンドスクリプトにメッセージを送信しないでください。ただし、メッセージ内のデータに基づいてバックグラウンドスクリプト内のURLを決定する必要があります。したがって、上記の例では、リクエストのURLはcontentScriptQueryおよびitemIdパラメータから決定されます。ドキュメントからの「悪い」メッセージの例：

{
  contentScriptQuery: "fetchUrl",
  url: "https://example.com/any/path/or/site/allowed/here"
}

PS：私

はFirefoxとChromeの拡張機能を開発していますが、メインのブラウザーはFirefoxであり、拡張機能の新しいバージョンをダウンロードするときにChromeのパフォーマンスを確認します。先日、CORBのおかげで、私の拡張機能の1つが機能しなくなったことがわかりました。これがあなたにとって驚きにならないように、拡張機能を確認してください。
