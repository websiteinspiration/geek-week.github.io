静的テストまたはプライベートライアンを保存

リリースはしばしば気付かれずに忍び寄ります。そして、彼の前で突然発見された間違いは、締め切りの変更、修正プログラム、朝までの仕事、そして神経質なもので私たちを脅かしています。このような急増が体系的に発生し始めたとき、私たちはあなたがもはやそのように生きることができないことに気づきました。リリース前に午後9時、または午後10時、または11時に帰宅した通常のRyan開発者Artyom を救うために包括的な検証システムを開発することが決定されました。変更はまだリポジトリに到達しておらず、彼自身がタスクのコンテキストを失っていた間に、開発者がエラーについて知ることが目的でした。


今日、行われた変更は、まずローカルで、次にアセンブリファームでの一連の統合テストで注意深くチェックされます。この記事では、検証の最初の段階、つまりリソー​​スの正確さを監視してコードを分析する静的テストについて説明します。これはチェーンの最初のサブシステムであり、見つかったエラーの大部分を占めます。

すべての始まり


リリース前にゲームをチェックする手動プロセスは、リリースの1週間半前にQAで始まりました。当然、この段階にあるバグはできるだけ早く修正する必要があります。

良い解決策の時間が足りないため、一時的な「松葉杖」が追加されます。これは長い間定着し、他のあまり人気のない解決策に囲まれています。

まず第一に、私たちは明白な間違いの発見を自動化することに決めました：クラッシュ、ゲームの主要なアクションセットを完了できない（ストアを開く、購入する、レベルをプレイする）。これを行うために、ゲームは特別な自動再生モードで実行され、問題が発生した場合は、ファームでのテストに合格した直後にそのことがわかります。

しかし、テスターと自動煙テストの両方で検出されたエラーのほとんどは、リソースの不足または異なるシステムの不適切な設定でした。したがって、次のステップは静的テストでした。アプリケーションを起動せずに、リソースの可用性、リソースの関係、設定を確認しました。このシステムは、アセンブリファームの追加の手順によって開始され、エラーの発見と修正を大幅に簡略化しました。しかし、問題のコードをコミットしてリポジトリに取り込む前でもエラーを検出できるのに、なぜアセンブリファームのリソースを無駄にするのでしょうか。これは、コミットが作成されてリポジトリに送信される前に開始されるプリコミットフックを使用して行うことができます。

確かに、コミット前とアセンブリファームでの静的テストが1つのコードで実行されるので、サポートが大幅に簡略化されます。

私たちの取り組みは、次の3つの領域に分けることができます。

  • アセンブリファームの作成-コミットされたすべてのものが収集およびチェックされるまさにその場所。
  • 静的テストの開発-リソースの正確性、それらの関係のチェック、コードアナライザーの起動;
  • ランタイムテストの開発-自動再生モードでアプリケーションを起動します。

別のタスクは、開発者によるマシンでのテストの起動を整理することでした。ローカルで実行時間を最小限に抑え（開発者が1行をコミットするのに10分待つ必要がない）、変更を導入する各システムにシステムがインストールされていることを確認する必要がありました。

多くの要件-1つのシステム


開発中、便利なアセンブリのセットがすべてあります。チート、ベータまたはアルファ、iOSまたはAndroidがあってもなくてもかまいません。いずれの場合も、異なるリソース、設定、または異なるコードが必要になる場合があります。可能なすべてのアセンブリの静的テスト用のスクリプトを作成すると、多くのパラメーターを持つ複雑なシステムになります。維持と変更が難しいことに加えて、各プロジェクトには独自の松葉杖のセットもあります。

試行錯誤の結果、1つのシステムに到達しました。各テストでは、起動コンテキストを考慮に入れ、それを実行するかどうか、正確に、そして確認する方法を決定できます。テストの開始時に、3つの主要なプロパティを識別しました。

  • アセンブリのタイプ：リリースおよびデバッグリソースの場合、チェックの重要度、適用範囲の完全性、および識別子の設定と使用可能な機能の検証が異なります。
  • プラットフォーム：Androidに有効なものはiOSでは正しくない可能性があります。リソースの収集方法も異なり、AndroidバージョンのすべてのリソースがiOSにあるわけではなく、その逆も同様です。
  • 起動場所：どこで起動するか-利用可能なすべてのテストが必要なビルドエージェント上、またはスタートアップのリストを最小化する必要があるユーザーのコンピューター上。


静的試験システム


システムのコアと静的テストの基本セットは、Pythonで実装されています。基本はほんの少数のエンティティです：


テストコンテキストは広範な概念です。上記で説明したビルドパラメーターと起動パラメーターの両方と、テストで満たされて使用されるメタ情報が格納されます。

まず、実行するテストを理解する必要があります。このため、メタ情報には、今回のリリースで特に関心のあるリソースのタイプが含まれています。リソースタイプは、システムに登録されているテストによって決定されます。テストは1つまたは複数のタイプに「関連付ける」ことができます。コミット時に、このテストがチェックするファイルが変更されていることが判明した場合、関連するリソースが変更されています。これは便利に私たちのイデオロギーに適合します-可能な限り少数のチェックをローカルで実行するため：テストの原因となっているファイルが変更されていない場合、実行する必要はありません。

例えば、魚の説明には3Dモデルとテクスチャーが記載されており、説明ファイルが変更されていれば、そこに記載されているモデルとテクスチャーが存在するか確認します。その他の場合は、フィッシュチェックを実行する必要はありません。

一方、リソースを変更すると、それに応じて変更とエンティティが必要になる場合があります。XMLファイルに保存されているテクスチャのセットが変更された場合、モデルに必要なテクスチャが削除されていることが判明する可能性があるため、追加の3Dモデルを確認する必要があります。上記の最適化は、コミット時にユーザーのマシンでローカルにのみ適用され、アセンブリファームで起動すると、すべてのファイルが変更されていると想定され、すべてのテストを実行します。

次の問題は、いくつかのテストが他のテストに依存していることです。すべてのテクスチャとモデルを見つける前に魚をチェックすることは不可能です。したがって、すべての実行を2つの段階に分けました。

  • コンテキスト準備
  • チェックする

最初の段階では、コンテキストには、見つかったリソースに関する情報（魚の場合は、パターンとテクスチャーの識別子）が表示されます。2番目のステージでは、保存された情報を使用して、目的のリソースが存在するかどうかを確認します。簡略化したコンテキストを以下に示します。

class VerificationContext(object):
   def __init__(self, app_path, build_type, platform, changed_files=None):
       self.__app_path = app_path
       self.__build_type = build_type
       self.__platform = platform
       #   
       self.__modified_resources = set()
       self.__expected_resources = set()
       #      ,       
       self.__changed_files = changed_files
       # -  ,   
       self.__resources = {}

def expect_resources(self, resources):
   self.__expected_resources.update(resources)

def is_resource_expected(self, resource):
   return resource in self.__expected_resources

def register_resource(self, resource_type, resource_id, resource_data=None):
   self.__resources.setdefault(resource_type, {})[resource_id] = resource_data

def get_resource(self, resource_type, resource_id):
   if resource_type not in self.__resources or resource_id not in self.__resources[resource_type]:
       return None, None
   return resource_id, self.__resources[resource_type][resource_id]

テストの起動に影響を与えるすべてのパラメーターを決定したので、基本クラス内のすべてのロジックを隠すことができました。特定のテストでは、テスト自体とパラメーターに必要な値のみを書き込む必要があります。

class TestCase(object):
   def __init__(self, name, context, build_types=None, platforms=None, predicate=None,
                expected_resources=None, modified_resources=None):
       self.__name = name
       self.__context = context
       self.__build_types = build_types
       self.__platforms = platforms
       self.__predicate = predicate
       self.__expected_resources = expected_resources
       self.__modified_resources = modified_resources

       #         
       #   ,    
       self.__need_run = self.__check_run()
       self.__need_resource_run = False

   @property
   def context(self):
       return self.__context

   def fail(self, message):
       print('Fail: {}'.format(message))

   def __check_run(self):
       build_success = self.__build_types is None or self.__context.build_type in self.__build_types
       platform_success = self.__platforms is None or self.__context.platform in self.__platforms
       hook_success = build_success
       if build_success and self.__context.is_build('hook') and self.__predicate:
           hook_success = any(self.__predicate(changed_file) for changed_file in self.__context.changed_files)
       return build_success and platform_success and hook_success

   def __set_context_resources(self):
       if not self.__need_run:
           return
       if self.__modified_resources:
           self.__context.modify_resources(self.__modified_resources)
       if self.__expected_resources:
           self.__context.expect_resources(self.__expected_resources)

    def init(self):
       """
         ,         
            ,    
       """
       self.__need_resource_run = self.__modified_resources and any(self.__context.is_resource_expected(resource) for resource in self.__modified_resources)

   def _prepare_impl(self):
       pass

   def prepare(self):
       if not self.__need_run and not self.__need_resource_run:
           return
       self._prepare_impl()

   def _run_impl(self):
       pass

   def run(self):
       if self.__need_run:
           self._run_impl()

魚の例に戻ると、2つのテストが必要です。1つはテクスチャを見つけてコンテキストに登録し、もう1つは見つかったモデルのテクスチャを検索します。

class VerifyTexture(TestCase):
   def __init__(self, context):
       super(VerifyTexture, self).__init__('VerifyTexture', context,
                                           build_types=['production', 'hook'],
                                           platforms=['windows', 'ios'],
                                           expected_resources=None,
                                           modified_resources=['Texture'],
                                           predicate=lambda file_path: os.path.splitext(file_path)[1] == '.png')

   def _prepare_impl(self):
       texture_dir = os.path.join(self.context.app_path, 'resources', 'textures')
       for root, dirs, files in os.walk(texture_dir):
           for tex_file in files:
               self.context.register_resource('Texture', tex_file)


class VerifyModels(TestCase):
   def __init__(self, context):
       super(VerifyModels, self).__init__('VerifyModels', context,
                                          expected_resources=['Texture'],
                                          predicate=lambda file_path: os.path.splitext(file_path)[1] == '.obj')

   def _run_impl(self):
       models_descriptions = etree.parse(os.path.join(self.context.app_path, 'resources', 'models.xml'))
       for model_xml in models_descriptions.findall('.//Model'):
           texture_id = model_xml.get('texture')
           texture = self.context.get_resource('Texture', texture_id)
           if texture is None:
               self.fail('Texture for model {} was not found: {}'.format(model_xml.get('id'), texture_id))

プロジェクトスプレッド


Playrixでのゲーム開発は独自のエンジンで行われるため、すべてのプロジェクトは同じルールを使用して同様のファイル構造とコードを持っています。したがって、一度記述されて一般的なコードに含まれる多くの一般的なテストがあります。プロジェクトがテストシステムのバージョンを更新し、新しいテストをプロジェクトに接続するだけで十分です。

統合を簡素化するために、構成ファイルと設計テスト（後でそれらについて）を受け取るランナーを作成しました。構成ファイルには、上記で記述した基本的な情報（アセンブリのタイプ、プラットフォーム、プロジェクトへのパス）が含まれています。

class Runner(object):
   def __init__(self, config_str, setup_function):
       self.__tests = []

       config_parser = RawConfigParser()
       config_parser.read_string(config_str)

       app_path = config_parser.get('main', 'app_path')
       build_type = config_parser.get('main', 'build_type')
       platform = config_parser.get('main', 'platform')

       '''
       get_changed_files         CVS 
       '''
       changed_files = None if build_type != 'hook' else get_changed_files()
       self.__context = VerificationContext(app_path, build_type, platform, changed_files)
       setup_function(self)

   @property
   def context(self):
       return self.__context

   def add_test(self, test):
       self.__tests.append(test)

   def run(self):
       for test in self.__tests:
           test.init()

       for test in self.__tests:
           test.prepare()

       for test in self.__tests:
           test.run()

構成ファイルの優れている点は、自動モードのさまざまなアセンブリに対してアセンブリファームで生成できることです。ただし、このファイルを介してすべてのテストの設定を渡すことは、あまり便利ではない場合があります。これを行うために、プロジェクトリポジトリに格納される特別な構成xmlがあり、無視されたファイルのリスト、コードで検索するためのマスクなどが書き込まれます。

構成ファイルの例
[main]
app_path = {app_path}
build_type = production
platform = ios

XMLの調整の例
<root>
	<VerifySourceCodepage allow_utf8="true" allow_utf8Bom="false" autofix_path="ci/autofix">
		<IgnoreFiles>*android/tmp/*</IgnoreFiles>
	</VerifySourceCodepage>
	<VerifyCodeStructures>
		<Checker name="NsStringConversion" />
		<Checker name="LogConstructions" />
	</VerifyCodeStructures>
</root>

一般的な部分に加えて、プロジェクトには独自の特徴と相違点があるため、ランナーの構成を介してシステムに接続されるプロジェクトテストのセットがあります。例のコードの場合、実行するには数行で十分です。

def setup(runner):
   runner.add_test(VerifyTexture(runner.context))
   runner.add_test(VerifyModels(runner.context))


def run():
   raw_config = '''
   [main]
   app_path = {app_path}
   build_type = production
   platform = ios
   '''
   runner = Runner(raw_config, setup)
   runner.run()

集められた熊手


python自体はクロスプラットフォームですが、ユーザーが独自の環境を持っているという問題が定期的にありました。この環境には、間違ったバージョン、予期したバージョン、いくつかのバージョン、またはインタープリターがまったくない場合があります。その結果、期待どおりに機能しないか、まったく機能しません。この問題を解決するためにいくつかの反復がありました：

  1. Pythonとすべてのパッケージはユーザーによってインストールされます。ただし、2つの「欠点」があります。すべてのユーザーがプログラマーであるとは限らず、デザイナーにとってはpip installを介したインストールであり、プログラマーにとっても問題になる可能性があります。
  2. 必要なすべてのパッケージをインストールするスクリプトがあります。これはすでに優れていますが、ユーザーが間違ったpythonをインストールしている場合、作業中に競合が発生する可能性があります。
  3. 正しいバージョンのインタープリターと依存関係を成果物ストレージ（Nexus）から提供し、仮想環境でテストを実行します。

別の問題はパフォーマンスです。テストが多いほど、ユーザーのコンピューターで変更がチェックされる時間が長くなります。数か月ごとに、ボトルネックのプロファイリングと最適化があります。そのため、コンテキストが改善され、テキストファイルのキャッシュが表示され、述語メカニズムが改善されました（このファイルがテストに興味深いと判断）。

そして、それはすべてのプロジェクトにシステムを実装する方法の問題を解決し、すべての開発者に中古のフックを含めるように強制することだけに残っていますが、これは完全に異なる話です...
