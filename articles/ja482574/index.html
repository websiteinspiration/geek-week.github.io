Androidプラットフォーム用の標準ベースの公開鍵暗号ワークステーション

公開鍵標準に基づく暗号化ワークステーションcryptoarmpkcsがモバイルプラットフォームの1つ、つまりAndroidでどのように機能するかを示す時が来ました。

cryptoarmpkcsユーティリティの開発中に定められた概念は、ユーザーが電子署名を作成および検証するときに最小限の不便を経験する必要があるということです。そのため、PKCS＃11暗号化トークンやPKCS＃12セキュアコンテナーを個人証明書のキーメディアとして使用することをお勧めします。たとえば、同じPKCS＃11トークンが特定のプラットフォームをサポートしていない場合など、多くの場合、PKCS＃12の使用がほとんど唯一の可能性であると私は言わなければなりません。そこで、PKCS＃12セキュアコンテナーをサポートすることで、cryptoarmpkcsユーティリティのAndroidプラットフォームへの移植を開始することにしました。

すぐに、プロジェクトはCおよびTcl / Tkプラットフォームで行われたため、移植によって基本的な問題が発生することはありませんでした。これは、Androwishテクノロジーのおかげで可能になりました。環境のデプロイとセットアップに問題はありませんでした（私の作業環境はLinux-Mageia-7.0です）。 Androidコマンドラインツール（sdk-tools-linux-xxxx.zip）をデプロイするためのスクリプトを1つだけ紹介します。

proc scaleImage {im xfactor {yfactor 0}} {
   set mode -subsample
   if {$xfactor>=0 && $yfactor>=0} {
       set mode -zoom
   } else {
	set xfactor [expr $xfactor * -1]
   }
   if {$yfactor == 0} {set yfactor $xfactor}
   set t [image create photo]
   $t copy $im
   $im blank
   $im copy $t -shrink $mode $xfactor $yfactor
   image delete $t
}

proc createtile {w  backg} {
    image create photo tiled

proc create_rectangle  {canv img x1 y1 x2 y2 color alfa {wbd 0} {colorline black} } {
    image create photo $img -format "default -colorformat  rgb"
    set rgb1 [winfo rgb $canv $color]
    set cr  [lindex $rgb1 0]
    set cg  [lindex $rgb1 1]
    set cb  [lindex $rgb1 2]
    set fill [format "#%04x%04x%04x" $cr $cg $cb ]
#  
    $img put $fill -to 0 0 [expr {$x2 - $x1}] [expr {$y2 -$y1}]
# 
    set dimg [$img data -format png]
# image   alpha 
    image create photo $img -data $dimg -format "png -alpha $alfa"
#    $img put [list $rgb1] -to 0 0 [expr {$x2 - $x1}] [expr {$y2 -$y1}]
#  
    set imgr [$canv create image $x1 $y1 -image $img -anchor nw] 
    set cc [subst {butImg $img}]
    $canv bind $imgr <ButtonPress-1> $cc
#   
    if {$wbd > 0 } {
	set item [$canv create rect $x1 $y1 $x2 $y2 -outline $colorline -width $wbd ]
	$canv bind $item <ButtonPress-1> $cc
    }
   return $imgr
}

空白には、クラシックウィジェットとテーマウィジェット（ラベルフレーム、ボタンなど）が入力されます。これらの塗りつぶされたキャンバスの1つは、右側の最初のスクリーンショットで見ることができます。最初の段階ではPKCS＃12コンテナーでの作業に焦点を当てたため、cryptoarmpkcs-Aのコードを変更せずに実際に使用しました。この段階で、次の機能が実装されます。

  • 文書に署名します（Cades-BES、CAdes-T、CAdes-XLT1）;
  • 署名者の追加を含め、ES（PKCS7）で作業します。
  • 証明書/証明書要求を表示します。

  • PKCS12 / PFXを使用しています。
  • ユーティリティ/ディストリビューションについて：


その他の機能は、PKCS＃11トークンにより関連しています。彼らの移植は新年に延期されます。ソフトウェアトークンを接続し、トークンのクラウドに接続する予定です。

機能の観点から見ると、ほとんどすべてがcryptoarmpkcsユーティリティに似ています。しかし、いくつかの違いがあります。たとえば、ドキュメントに署名した後、ユーティリティは、国家サービスのWebサイトで署名を確認するかどうかを尋ねます。


[はい]をクリックすると、ドキュメントと証明書の署名を確認するためのページがブラウザに読み込まれます。このページはスマートフォンの画面にあまり対応していないので、すぐに予約してください。これは、署名を選択するときに顕著になり、署名が切断されている場合は、ドキュメントを含むファイルになります。しかし、すべてが順調であれば、肯定的な結果が得られます。



国家サービスのウェブサイトで署名と証明書の両方を検証することは、証明書が認定された認定センター（CA）で受け取られた場合にのみ意味があることに注意してください。そうしないと、署名は常に無効になります。
