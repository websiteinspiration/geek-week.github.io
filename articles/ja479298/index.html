PostgreSQLアンチパターン：CTE x CTE

活動の種類によって、1は、開発者が要求を書き込み、と思ったときに「状況に対処するために持っているデータベースは、スマートで、彼女はすべてを処理することができます！」

「フランケンシュタイン」の出現に、いくつかのケースでは（一部はデータベース機能の無知から、一部は時期尚早の最適化から）このアプローチのリードを。

まず、そのようなクエリの例を示します。

--        
WITH RECURSIVE cte_bind AS (
  SELECT DISTINCT ON (key_a, key_b)
    key_a a
  , key_b b
  , fld1 bind_fld1
  , fld2 bind_fld2
  FROM
    tbl
)
--  min/max     
, cte_max AS (
  SELECT
    a
  , max(bind_fld1) bind_fld1
  , min(bind_fld2) bind_fld2
  FROM
    cte_bind
  GROUP BY
    a
)
--        min/max-
, cte_a_bind AS (
  SELECT
    cte_bind.a
  , cte_bind.b
  , cte_max.bind_fld1
  , cte_max.bind_fld2
  FROM
    cte_bind
  INNER JOIN
    cte_max
      ON cte_max.a = cte_bind.a
)
SELECT * FROM cte_a_bind;

リクエストの品質を客観的に評価するために、任意のデータセットを作成します。

CREATE TABLE tbl AS
SELECT
  (random() * 1000)::integer key_a
, (random() * 1000)::integer key_b
, (random() * 10000)::integer fld1
, (random() * 10000)::integer fld2
FROM
  generate_series(1, 10000);
CREATE INDEX ON tbl(key_a, key_b);

データ 自体の読み取りにかかった時間は、クエリ実行時間全体の4分の1未満であることが

わかります。

骨で分解


私たちはリクエストを注意深く見て、困惑します：

  1. 再帰的なCTEがない場合、WITH RECURSIVEがここにあるのはなぜですか？
  2. とにかくそれらが元のサンプルにまだアタッチされているのに、なぜ最小/最大値を別のCTEにグループ化するのですか？
    +時間の25％
  3. なぜ最後に無条件の「SELECT * FROM」を介して以前のCTEからの再読み取りを使用するのですか？
    +時間の14％

この場合、Nested LoopではなくHash Joinが接続に選択されたのは非常に幸運でした。CTEスキャンパスが1つではなく、10Kだったからです。

CTEスキャンについて少し
, CTE Scan Seq Scan — , , 10K x 0.3ms = 3000ms cte_max 1K x 1.5ms = 1500ms cte_bind!

実際、あなたは結果として何を得たいと思ったのですか？ええ、「3階建て」リクエストの分析の5分のどこかに彼が訪れるのは、通常、この種の質問です。

一意のキーペアごとに、key_aによるグループからmin / maxを削除する必要がありました。
したがって、これにはウィンドウ関数を使用します。

SELECT DISTINCT ON(key_a, key_b)
	key_a a
,	key_b b
,	max(fld1) OVER(w) bind_fld1
,	min(fld2) OVER(w) bind_fld2
FROM
	tbl
WINDOW
	w AS (PARTITION BY key_a);


[explain.tensor.ruを見てください]

両方のバージョンでのデータの読み取りには約4〜5ミリ秒かかるため、このような要求が十分頻繁に実行された場合、-32％の時間の増加はすべて、最も純粋な形でベースCPUから削除されます。

一般に、ベースを「ラウンド-ウェア、スクエア-ロール」にすることはできません。
