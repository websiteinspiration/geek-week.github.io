「新しい」かそうでないか...

また会いましたね。Android開発の基本コースと上級コースの開始を予想して、興味深い翻訳をご用意しました。




依存性注入では、新しい演算子とアプリケーションロジックを分離する必要があります。この分離により、アプリケーションのリンクを担当するコードでファクトリを使用することが奨励されます。ただし、ファクトリを作成するのではなく、GUICEなどの自動依存性注入を使用してバインディングを引き継ぎます。しかし、依存性注入は本当にすべての新しいオペレーターから私たちを救うことができますか？
2つの極端を見てみましょう。 AudioDeviceを取得する必要があるMusicPlayerクラスがあるとします。ここでは、依存関係注入を使用して、MusicPlayerコンストラクターでAudioDeviceを要求します。これにより、テストに適したAudioDeviceを追加できます。これを使用して、MusicPlayerから正しいサウンドが出ていると主張できます。 new演算子を使用してBuiltInSpeakerAudioDeviceのインスタンスを作成すると、テストが困難になります。それでは、AudioDeviceやMusicPlayerなどのオブジェクトを「Injectable」と呼びましょう。 Injectableは、コンストラクターで要求するオブジェクトであり、依存関係を実装するためのフレームワークがそれらを提供することを期待しています。

もう一方の極端に。あなたがint intプリミティブを持っているが、それをIntegerに自動パックしたい場合、最も簡単なのはnew Integer（5）を呼び出すことで、これで終わりです。しかし、依存性注入が新しい「新しい」ものである場合、なぜ新しいインラインと呼ぶのでしょうか。テストに悪影響はありますか？依存関係を注入するためのフレームワークは、必要な整数を提供できないことがわかります。なぜなら、それらは、それがどのような整数であるかを理解していないからです。これはちょっとしたおもちゃの例なので、もっと複雑なものを見てみましょう。

ユーザーがログインフィールドに電子メールアドレスを入力し、新しい電子メール（«a@b.com»）このままにすることはできますか、それともコンストラクタでメールをリクエストする必要がありますか？繰り返しになりますが、依存関係注入フレームワークでは、電子メールを提供できません。最初に、電子メールが配置されている文字列を取得する必要があるためです。そして、選択できるStringはたくさんあります。ご覧のとおり、依存関係注入フレームワークでは提供できないオブジェクトが多数あります。手動でnewを呼び出さなければならないので、「Newable」と呼びましょう。

まず、いくつかの基本的なルールを設定しましょう。 Injectableクラスは、そのコンストラクタで他のInjectableをクエリできます。 （Injectableをサービスオブジェクトと呼ぶこともありますが、この用語はオーバーロードされています。）Injectableは、テストに便利な実装に置き換える必要がある可能性があるため、インターフェイスを持つ傾向があります。ただし、Injectableは、コンストラクターで非Injectable（Newable）をクエリすることはできません。これは、依存性注入フレームワークがNewableの作成方法を認識していないためです。依存関係注入フレームワークに期待するクラスの例をいくつか示します。CreditCardProcessor、MusicPlayer、MailSender、OfflineQueueです。同様に、Newableは他のNewableからコンストラクターで要求できますが、Injectableは要求できません（Newableを値オブジェクトとして呼び出すこともありますが、この用語は過負荷です）。 Newableの例：電子メール、MailMessage、ユーザー、CreditCard、Song。これらの区別に従うと、コードのテストと操作が簡単になります。これらのルールに違反すると、コードのテストが困難になります。

MusicPlayerとSongの例を見てみましょう

class Song {
  Song(String name, byte[] content);
}
class MusicPlayer {
  @Injectable
  MusicPlayer(AudioDevice device);
  play(Song song);
}

SongはNewableであるオブジェクトのみをクエリすることに注意してください。これにより、テストでソングを簡単にインスタンス化できます。MusicPlayerは、AudioDevice引数と同様に完全に注入可能であるため、依存関係注入のフレームワークから取得できます。

次に、MusicPlayerがルールに違反し、コンストラクターでNewableを要求するとどうなるかを見てみましょう。

class Song {
  String name;
  byte[] content;
  Song(String name, byte[] content);
}
class MusicPlayer {
  AudioDevice device;
  Song song;
  @Injectable
  MusicPlayer(AudioDevice device, Song song);
  play();
}

ここでも、SongはまだNewableであり、テストやコードで簡単に作成できます。 MusicPlayerはすでに問題です。依存関係の注入をフレームワークからMusicPlayerに依頼すると、フレームワークはどのSongであるかを認識できないため、クラッシュします。依存関係注入フレームワークを初めて使用するほとんどの人は、気づきやすいので、この間違いをすることはほとんどありません。コードが機能しないためです。

次に、Songがルールに違反し、コンストラクターでInjectableを要求した場合にどうなるかを見てみましょう。

class MusicPlayer {
  AudioDevice device;
  @Injectable
  MusicPlayer(AudioDevice device);
}
class Song {
  String name;
  byte[] content;
  MusicPlayer palyer;
  Song(String name, byte[] content, MusicPlayer player);
  play();
}
class SongReader {
  MusicPlayer player
  @Injectable
  SongReader(MusicPlayer player) {
    this.player = player;
  }
  Song read(File file) {
    return new Song(file.getName(),
                    readBytes(file),
                    player);
  }
}

一見、すべてが順調です。しかし、曲がどのように作成されるかを考えてください。おそらく、曲はディスクに保存されているため、SongReaderが必要です。 SongReaderは、MusicPlayerを要求する必要があります。これにより、Songのnewを呼び出すときに、SongのMusicPlayerへの依存性を満たすことができます。ここで何かお気づきですか？ SongReaderはMusicPlayerについてどのような恐怖を知っている必要がありますか？これはデメテルの法則の違反です。 SongReaderはMusicPlayerについて知らないはずです。少なくとも、SongReaderがMusicPlayerでメソッドを呼び出さないためです。 SongがNewable / Injectableの分離に違反しているため、彼はMusicPlayerについてしか知りません。 SongReaderは、Songのエラーを支払います。エラーが発生する場所と結果が現れる場所は同じではないため、このエラーは非常に微妙であり、診断が困難です。また、多くの人がこの間違いを犯す可能性が高いことも意味しています。

テストに関しては、これは本当の苦痛です。 SongWriterがあり、それがSongをディスクに正しくシリアライズすることを確認したいとします。 MockMusicPlayerを作成してSongに渡して、SongWritterに渡せるようにする必要があります。なぜここでMusicPlayerに出くわすのですか？別の方法で見てみましょう。 Songはシリアライズしたいものであり、これを行う最も簡単な方法はJavaシリアライゼーションを使用することです。したがって、Songだけでなく、MusicPlayerおよびAudioDeviceもシリアル化します。 MusicPlayerもAudioDeviceもシリアル化しないでください。ご覧のとおり、小さな変更はテスト容易性を大幅に向上させます。

ご覧のとおり、これらの2種類のオブジェクトを分離すると、コードの操作が簡単になります。それらを混ぜると、コードのテストが難しくなります。 Newableは、アプリケーションのオブジェクトグラフの最後にあるオブジェクトです。 Newableは他のNewableに依存している可能性があります。たとえば、CreditCardがCityに依存しているAddressにどのように依存しているかなどは、アプリケーショングラフのシートです。それらはシートであり、外部サービス（外部サービスはInjectable）と通信しないため、スタブを実行する必要はありません。文字列そのもの以外は文字列のようには見えません。新しいユーザーを呼び出すだけの場合、ユーザーのスタブを作成する必要があるのはなぜですか。メール、メールメッセージ、ユーザー、クレジットカード、曲のいずれかでスタブを作成するのはなぜですか。newを呼び出して終了します。

ここで、非常に微妙なことに注意を払いましょう。NewableがInjectableについて知っているのは正常です。通常ではないことは、NewableがInjectableをフィールドとして参照していることです。つまり、SongはMusicPlayerについて知ることができます。たとえば、Injectable MusicPlayerがスタックを介してNewable Songに渡されるのは正常です。スタックを通過することは、依存性注入のフレームワークから独立しているためです。この例のように：

class Song {
  Song(String name, byte[] content);
  boolean isPlayable(MusicPlayer player);
}

この問題は、SongにMusicPlayerへのリンクフィールドがある場合に発生します。リンクフィールドは、コンストラクターを介して設定されます。これにより、呼び出し元のデメテルの法則に違反し、テストが困難になります。
