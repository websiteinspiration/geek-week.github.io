言語C



C言語のサブセットを定義する最も信頼できるドキュメントは、「MISRA C」として知られる標準です[4]。これに基づいて、または少なくともその影響下で、さまざまな組織の安全なコーディングのための標準、たとえばJPLコーディング標準[6]が開発されています。



導入部では、「MISRA C」がCの人気の理由を示しています。そのうちのいくつかを挙げます。



  • Cで書かれたプログラムは、効率的なマシンコードにコンパイルできます。
  • さまざまなハードウェアアーキテクチャ用の既製のCコンパイラがあります。
  • Cは、静的分析とテストのための多数のツールによってサポートされています。
  • 言語Cには国際標準があります[7]、[8]。
  • 重要なシステムの開発でCを使用することで素晴らしい経験が得られました。


同時に、「MISRA C」の開発者は、C言語の欠点に注意します。その中には、コンパイラー開発者がいくつかの機能をさまざまな方法で実装できる不正確な定義や、コンパイラーの観点からはそうでないエラーが発生する可能性-割り当て記号を置き換える典型的な問題があります等号またはその逆。また、効果的なコードコンパイルの裏側である欠点として、ランタイムエラー制御が存在しないことが示されます。これにより、プログラマーはプログラム内のエラーを明示的に制御する必要があります。



「MISRA C」では、C99とC90の両方を使用できます。 C11 / C18標準に関する推奨事項はありません。「MISRA C」というテキストを確定する時点では、ISO / IECワーキンググループはまだ新しいバージョンのCを承認する時間がないため、未承認の標準を参照することは悪い考えです。したがって、もちろんC11 / C18を使用することもできます。

「MISRA C」の方向には、ルールとディレクティブの2つのタイプがあります。



ルールはそのような表示を参照し、他の情報源なしでソーステキストを分析することにより、その表示を確認できます。静的アナライザーは、ルールの実装を厳密にチェックできる必要があります。ディレクティブには、ルールほど厳密には策定できない指示が含まれています。指令へのコードの準拠を分析するには、コードだけでは十分ではありません。設計ドキュメントの調査が必要になる場合があります。



静的アナライザーはチェック中に役立ちますが、それらの作業の結果は正しく解釈されなければなりません。



すべての指示-規則または指令-は、必須、必須、推奨の3つの重要度のいずれかに従って分類されます。

この規格では、実績のある設計パターン（テンプレート）とソフトウェアフレームワーク（フレームワーク）の使用を（HR）に強く推奨しています。実際、重要なシステムでC ++を使用する主な方法は、ワイヤーフレームと自動的に生成されたコードであろう。 1990年代後半以降、ステートマシンに基づくアルゴリズムの実装に使用するのに便利なUMLモデルからコードを生成するフレームワークの人気が高まっています。その後、組み込みシステムの能力により、豊富なUX、高度な通信、その他のテクノロジーを使用できるようになったとき、Qtフレームワークも広く普及しました。さまざまな設計段階のパターンは、たとえば、ブルースポールダグラスの「リアルタイム設計パターン」[15]専用です。フレームワークの例には、IBM RationalのOXFやWillert Software ToolsのRXFが含まれます。

ここでも、C ++は、フレームワークをあるタイプのモデルからの自動コード生成と組み合わせて使用​​する場合に特に関係があります。モデル指向開発の主な目的は、複雑なアプリケーションを開発する際の抽象化のレベル（したがって生産性）を高め、コードの生成と分析を自動化してエラーを減らすことです。
