MQTTプロトコル：概念的な没入

メッセージキューテレメトリトランスポート（MQTT）プロトコルは長年使用されてきましたが、IoTの爆発的な成長により、特に関連性が高くなっています。コンシューマデバイスと産業用デバイスの両方が分散ネットワークとエッジコンピューティングを実装し、継続的なデータ伝送を行うデバイスが日常の一部になっています。人生の。

これは、軽量でオープンで手頃な価格のプロトコルが今後さらに重要になることを意味します。この記事では、MQTTの概念の没入について説明します。それがどのように機能し、現在どのように使用され、将来どのように使用されるかを示します。

小さな紹介


MQTTは、パブリッシャーサブスクライバーベースのメッセージングプロトコル（pub / sub）です。 1999年の最初のバージョンは、IBMのAndy Stanford-ClarkおよびCirrus LinkのArlene Nipper によって公開されました。彼らは、MQTTを、帯域幅が制限されているネットワークや予測できない通信を行うネットワーク内のマシン間の通信を維持する方法と見なしていました。その使用の最初のオプションの1つは、パイプラインのフラグメントが互いに、衛星経由で中央リンクと確実に接触するようにすることでした。

過酷な操作条件を考えると、プロトコルは小さくて軽くなります。低電力デバイスやバッテリ寿命が限られている場合に最適です。現在では、ユビキタススマートフォンや、増え続けるセンサーや接続デバイスが増えています。

したがって、MQTTは、限られたCPUパワーやバッテリ寿命でデバイス間でデータをストリーミングするためのプロトコルになり、高額または低帯域幅、予測不可能な安定性、または高遅延のネットワークにも対応しています。そのため、MQTTはIoTの理想的な手段として知られています。これはTCP / IPプロトコルに基づいて構築されていますが、Bluetooth、UDP、ZigBee、およびTCP / IP以外の他のIoTネットワークを介して作業するためのMQTT-SNブランチがあります。

MQTTは、この種のリアルタイムパブ/サブメッセージングプロトコルだけではありませんが、マシン間通信に依存するさまざまな環境ですでに広く採用されています。彼のピアには、Webアプリケーションメッセージングプロトコル、ストリーミングテキスト指向メッセージングプロトコル、代替メッセージキュープロトコル。

MQTTは、信頼性の高い機能と、ブラウザー、スマートフォン、IoTデバイスなどのインターネットに接続されたデバイスやアプリケーションとの幅広い互換性を備えたアプリケーションを作成したい開発者にとって論理的な選択です。

MQTTの仕組み：基本


MQTT上に構築された通信システムは、パブリッシャーサーバー、ブローカーサーバー、および1つ以上のクライアントで構成されます。パブリッシャーは、メッセージを受信するサブスクライバーの数または場所の設定を必要としません。さらに、サブスクライバーは特定のパブリッシャーにチューニングする必要はありません。システムに複数のメッセージブローカーが存在する可能性があります。



MQTTは、リーフを持つ一種のブランチである通信チャネルの階層を作成する方法を提供します。パブリッシャーが顧客に配布するための新しいデータを持っている場合は常に、メッセージに配信制御に関するメモが添付されます。上位レベルのクライアントは各メッセージを受信できますが、下位レベルのクライアントは、階層の最下部にある「ブランチ」である1つまたは2つの基本チャネルのみに関連するメッセージを受信できます。これにより、サイズが2バイトから256メガバイトに及ぶ情報の交換が容易になります。MQTTブローカーを

介して接続するようにクライアントを構成する方法の例：

var options = {
  keepalive: 60,
  username: 'FIRST_HALF_OF_API_KEY',
  password: 'SECOND_HALF_OF_API_KEY',
  port: 8883
};
var client = mqtt.connect('mqtts:mqtt.ably.io', options);

MQTTはバイナリプロトコルであるため、MQTTブローカーによって発行または受信されたデータはすべてバイナリ形式でエンコードされます。つまり、元のコンテンツを取得するには、メッセージを解釈する必要があります。以下は、AblyとJavaScriptでの表示です。

var ably = new Ably.Realtime('REPLACE_WITH_YOUR_API_KEY');
var decoder = new TextDecoder();
var channel = ably.channels.get('input');
channel.subscribe(function(message) {
  var command = decoder.decode(message.data);
});

MQTTブローカーは、現在のサブスクライバーを持たないチャネルに関連するメッセージを蓄積することがあります。この場合、制御メッセージの指示に応じて、メッセージは破棄または保存されます。これは、次の送信を待つのではなく、新しいサブスクライバーが最近記録されたデータポイントを必要とする場合に役立ちます。

MQTTがセキュリティ資格情報をクリアテキストで送信することは注目に値します。それ以外の場合、認証またはセキュリティ機能はサポートされません。これがSSLフレームワークの出番であり、送信された情報が傍受または改ざんされないように保護するのに役立ちます。

さらに、MQTTでは、APIキーを実際のMQTTクライアントにまったく公開したくない場合は、トークンでAbly認証を使用できます（SSLなしのMQTTの場合、APIテキストがクリアテキストで転送されないようにするためにトークンが必要です）。トークン認証の例：

var options = {
  keepalive: 60,
  username: INSERT_TOKEN_HERE,
  password: '',
  port: 8883
};
var client = mqtt.connect('mqtts:mqtt.ably.io', options);
client.subscribe("[mqtt]tokenevents", {
  /* Create a new token called 'NEW_TOKEN' */
  client.end();
  options.username = NEW_TOKEN;
  client = mqtt.connect('mqtts:mqtt.ably.io', options);
});

MQTT機能：より深い没入感


IBMによると、MQTTには次の特性があります。

  • メッセージの内容に中立
  • 1対多の分散通信および切断されたアプリケーションに最適
  • クライアントの異常切断を当事者に通知するLWT機能（Last Will and Testament、「Last Will and Testament」）を装備
  • 基本的な通信タスクをTCP / IPに依存
  • 「最大1回」、「少なくとも1回」、「正確に1回」のテンプレートを使用してメッセージを配信するように設計されています

MQTTシステムのメンバーは、パブリッシャー、コンシューマー、またはその両方の役割を一度に引き受けることができます。

MQTTの際立った特徴の1つは、チャネルに対する独自の理解です。たとえば、チャネルはそれぞれ、ファイルパスとして扱われます。

channel = "user/path/channel"

チャネルは、すべての顧客が彼宛てのメッセージを確実に受信できるようにします。MQTTは、チャネルをファイルパスとして扱うことにより、クライアントがファイルパスにサブスクライブする場所（レベルまたはブランチ）に基づいてメッセージをフィルタリングするなど、あらゆる種類の有用な通信機能を実行します。

MQTTメッセージ形式


各MQTTメッセージを構成する2つのコンポーネントを見てください。

  • バイト1：メッセージのタイプ（クライアントの接続要求、サブスクリプション確認、ping要求など）、複製フラグ、メッセージの保存手順、およびサービス品質（QoS）に関する情報が含まれます。
  • バイト2：ペイロードおよびオプションの変数のヘッダー内のデータを含む、残りのメッセージ長に関する情報が含まれます。

バイト1のQoSフラグは、MQTTがサポートする可変機能の基礎となるため、特別な注意が必要です。QoSフラグには、メッセージの意図と緊急性に基づいて、次の値が含まれています。

  • 0 = 1回のみ：サーバーがトリガーされ、忘れます。メッセージが失われたり重複したりする可能性があります。
  • 1 =少なくとも1回：受信者が配信を確認します。メッセージは複製できますが、配信は保証されます
  • 2 =正確に1回：サーバーが配信を提供します。メッセージは、損失や重複なしに1回だけ到着します

IoTデバイスやその他のアプリケーションでさまざまなQoSレベルを使用する方法を見てみましょう。

MQTTはどこで使用できますか？


現在、IoTアプリケーションが大規模に実装されているため、MQTTは、消費者市場と産業市場の両方で、より幅広いユーザーベースに分散コンピューティングとIoT機能を展開するためのオープンでシンプルかつスケーラブルな方法として注目されています。

上記のように、MQTTは信頼性のないネットワークとデバイス用に構築された軽量のメッセージングプロトコルであり、電源とCPUに制限があります。ただし、これは、パケット損失の可能性のある接続が唯一のアプリケーションであることを意味するものではありません。 MQTTは、データサンプリングの繰り返しから産業用機械の管理まで、さまざまなタイプのIoTインフラストラクチャにさまざまなレベルのサービスを提供します。

MQTTを使用しない場合


開発者は、MQTT、HTTP、CoAP、WebSockets（CPU /バッテリーが許可する場合）などを含む、双方向IoT通信チャネルを設計および展開するための幅広いプロトコルを選択できます。MQTTが最良の選択であるかどうかは、ハードウェアとアプリケーションタスクによって異なります。

帯域幅が非常に狭い環境向けに設計されたMQTTプロトコルは、すべてのバイトを節約するという点で柔軟性に欠ける場合があります。たとえば、仕様では、サーバーが接続を拒否できる5つのエラーメッセージのみが定義されています（たとえば、無効なユーザー名/パスワード、またはプロトコルの許容できないバージョン）。サーバーが他のエラーを指摘したい場合、それは運が悪いです。さらに悪いことに、接続の開始後にエラーが発生した場合、エラーを報告するメカニズムがまったくありません。サーバーは肩をすくめて、TCP接続を突然中断することができるだけであり、クライアントがドロップした理由の手がかりはありません（意図的な切断と一時的なネットワークの問題を区別する方法もありません）。人々のために、より柔軟でデバッグしやすい（帯域幅のコスト効率は低いですが）pub / subプロトコルに慣れているため、このようなSpartanアプローチは少し原始的に見えるかもしれません。

MQTTはHTTPと共に参照されることが多いため、Googleは、応答時間、トラフィック量、および開発者にとって重要なその他の属性についてそれらを比較する調査を実施しました。 MQTTはGoogleのテストで1位になりましたが、接続を再利用して複数のペイロードを送信できる場合のみです。

HTTPおよびMQTTは、比較的少量のトラフィック、低バッテリーおよびメモリ要件のため、IoTアプリケーションに適しています。

CoAPは、IoTシステムを開発するためにMQTTと比較されることが多い別のプロトコルです。それらは似ていますが、顕著な違いがあります。MQTTは多対多のプロトコルですが、CoAPは基本的にサーバーとクライアント間の通信用の1対1のプロトコルです。同時に、CoAPはMQTTにはないメタデータ、検出、およびコンテンツネゴシエーション機能を提供します。

クライアントがデータのみを受信する必要がある場合は、サーバー送信イベント  も適切なオプションです。

MQTTを素早くセットアップする方法


GitHubのMQTTリポジトリには、さまざまな言語のオープンソースMQTTライブラリの広範なリストがあります。以下は、オープンソースのMQTTブローカー、JavaScriptライブラリ、.NETライブラリを使用したカスタマイズの2つの例です。

Eclipse Mosquitto-オープンソースのMQTTブローカー


Eclipse Mosquitto  は、MQTTプロトコルバージョン5.0、3.1.1、および3.1を実装するオープンソースのメッセージングブローカー（EPL / EDLライセンス）です。Mosquittoは軽量で、低電力のシングルボードコンピューターから本格的なサーバーまで、すべてのデバイスでの使用に適しています。

MQTT.js


MQTT.js  は、MQTTプロトコルのクライアントライブラリであり、Node.jsとブラウザ用にJavaScriptで記述されています。MQTT.jsを使用してメッセージを送信する例を次に示します。

var mqtt = require('mqtt')
var client  = mqtt.connect('mqtt://test.mosquitto.org')
client.on('connect', function () {
  client.subscribe('presence', function (err) {
    if (!err) {
      client.publish('presence', 'Hello mqtt')
    }
  })
})
client.on('message', function (topic, message) {
  // message is Buffer
  console.log(message.toString())
  client.end()
})

MQTTnet


MQTTnet  は、クライアントとMQTTサーバー（ブローカー）の両方を提供する高性能.NETライブラリです。

MQTTクライアントのインストール：

// Create a new MQTT client.
var factory = new MqttFactory();
var mqttClient = factory.CreateMqttClient();

MQTTクライアント設定を構成した後、接続を確立できます。次のコードは、サーバーに接続する方法を示しています。

// Use WebSocket connection.
var options = new MqttClientOptionsBuilder()
    .WithWebSocketServer("broker.hivemq.com:8000/mqtt")
    .Build();

await client.ConnectAsync(options);

着信メッセージを受信します。

client.UseApplicationMessageReceivedHandler(e =>

スケーリングについてはどうですか？


MQTTのスケーリングに関しては、考慮すべき2つの考慮事項があります。1）これが正しいプロトコルであるかどうか。 2）プロトコルの選択に関係なく、MQTTを使用するデバイス間の増加したトラフィックを処理するために必要なインフラストラクチャとネットワーク機能。

軽量Machine-to-Machine（LWM2M）は、エンタープライズレベルでMQTTと共に使用できる別のプロトコルです。 MQTTと比較して、長期間のIoTシステムに適している場合があります。。 MQTTは簡単にIoTを試してみるのに理想的ですが、LWM2Mは長期的な用途の広いインフラストラクチャに機能を提供します。 LWM2Mは、接続監視、ファームウェアの更新、リモートデバイスアクションなどの優れたデバイス管理ツールも提供します。大量のデータを中央プラットフォームに送信する、管理されていないデバイスが多数ある企業には、LWM2Mが最適です。ただし、IoTの大規模な導入について話しているため、通常、MQTTは適切なオプション以上のものです。さらに、MQTTがより一般的で、より幅広いサポートがあります。

次に、インフラストラクチャの可能性について説明します。サーバーの読み込みに関しては、同時接続数がボトルネックになることはほとんどありません。ほとんどの優れたMQTTサーバー/ブローカーは数千の同時接続をサポートしていますが、MQTTサーバーが実際のデータを受信した後にメッセージを処理して応答するために必要なワークロードはどれくらいですか？原則として、データベースの読み取りと書き込み、サーバーとの統合、各クライアントのリソースの分散と管理など、あらゆる種類の潜在的な問題があります。1台のマシンが負荷に対応できなくなったらすぐに、サーバーを追加する必要があります。つまり、負荷分散、異なるサーバーに接続されているクライアント間のメッセージ同期、接続時間やクライアントが接続されている特定のサーバーに関係なく、クライアントの状態への一般化されたアクセス-リストは何度も続きます。

このような問題は別の記事に値します。多くの情報は、ブログのエンジニアリングセクションにあります。特に、大規模なリアルタイムメッセージングインフラストラクチャのサービスの複雑さに関する記事をご覧ください。
