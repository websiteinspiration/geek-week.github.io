Tipo de dados abstratos. Parte 2: Gerenciamento de Dados

Na primeira parte, descrevi como ser guiado na determinação do tipo de dados (doravante denominado TD). Nesta parte, quero lhe dizer o que é o Tipo de Dados Abstratos (doravante denominado ADT), como ajuda a resolver tarefas logicamente complexas (o que se entende por tarefas logicamente complexas, consulte o primeiro artigo) e quais vantagens temos.


Em vez de um prefácio:

Um Tipo de Dados Abstrato é uma coleção que inclui dados e as operações executadas nele (S. McConnell, “Código Perfeito”, Capítulo 6.1. Fundamentos das Classes: Tipos de Dados Abstratos). Nesta série de artigos, consideramos exatamente essa visão do ADT, estendida pelo tipo de dados.

Torvalds Linus - “Os programadores ruins se preocupam com o código. Bons programadores se preocupam com estruturas de dados e seus relacionamentos. ”
Cálculo é a aquisição de novos conhecimentos a partir de dados de entrada.


ADT - é considerado como uma classe contendo estruturas de dados com as quais trabalhamos no futuro e nesta classe. Os dados se referem a conjuntos de pontos de acesso que foram discutidos na primeira parte .
Estruturas ADT para armazenamento de dados - meios de armazenamento de dados, ou seja, pode ser uma pilha, uma fila e qualquer estrutura de dados clássica / não clássica que seja conveniente para trabalhar com eles, depende do contexto da tarefa e da sua imaginação.


Por que essa abordagem permite organizar uma lógica complexa:

  1. Trabalhamos apenas com dados (TD) sem hesitar em obtê-los. Colocamos a lógica da aquisição de dados em "provedores de dados" (veja abaixo). Em essência, trata-se de uma luta com a complexidade - estamos pensando separadamente em como obter dados que sempre têm um certo tipo e separadamente em como trabalhar com dados desse tipo.
  2. Podemos organizar o ADT apenas com os conjuntos de dados necessários, realizando a decomposição do ADT, se necessário (também um método de lidar com a complexidade).

Os requisitos para ADT são semelhantes aos requisitos para TD, com uma pequena adição:

  1. O ADT deve poder trabalhar apenas com seus dados.
  2. ADTs não devem depender do estado externo do aplicativo.
  3. Para receber dados, use “provedores de dados” e transmita seus resultados ao ADT; não use os métodos ADT para obtê-los.

Provedor de dados - isso implica uma classe que recebe os dados de que precisamos das fontes disponíveis, de várias maneiras, e retorna APs ou conjuntos de APs. Como organizar provedores de dados é um tópico separado, dependendo do contexto da tarefa e da imaginação do desenvolvedor do sistema.

  4. Temos pessoas de altura e peso diferentes. Dependendo do crescimento, uma pessoa se encaixa em um tanque específico ou não. A massa do tanque também depende do peso, que por sua vez afeta o consumo e a velocidade do combustível.

Concordo, já existem muitos parâmetros e alguns parâmetros podem afetar os parâmetros de outras entidades. E então lembramos que um tipo de tanque pode ter tipos diferentes de motor, para os quais apenas um determinado tipo de combustível é adequado. O que fazer, reescrever todo o código?

A solução completa usando a abordagem descrita é muito longa, então tentarei descrever brevemente o próprio conceito:

  2. No ADT, gerenciamos dados. Por exemplo, carregamos no ATD a área em que a batalha ocorre e tanques de vários tipos com um conjunto diferente de dados. Em seguida, simplesmente gerenciamos os dados chamando certos métodos, por exemplo, movemos o tanque 1 para o ponto B em nossa área ou disparamos do tanque 2 para o ponto C. Conseqüentemente, conhecendo a velocidade do tanque, a velocidade do projétil, o tipo de dano do projétil etc. podemos fazer cálculos no ATD, onde, no momento do impacto do projétil lançado pelo tanque 2, o tanque 1 estará localizado e se a carcaça do tanque 2 entrar no tanque 1, que dano será causado a ele.

Não pretendo que essa abordagem seja a melhor solução, apenas decidi compartilhar meus pensamentos, talvez eles ajudem alguém a resolver seus problemas.

Total:

  1. O ADT armazena dados agregados de uma forma conveniente para nós
  2. O ADT deve poder trabalhar apenas com seus dados.
  3. O ADT não deve depender do estado externo do aplicativo.
  4. Não use métodos ADT para obter dados; para isso, use “Data Providers” que retornam pontos de acesso ou conjuntos de pontos de acesso. Em seguida, transferimos os dados recebidos para o ADT para cálculos sobre eles (anexamos / aplicamos a ação ao AT).
