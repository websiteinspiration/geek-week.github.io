Umpan balik pemberian memori dalam SQL Server 2019

Halo! Untuk mengantisipasi dimulainya kursus "Pengembang MS SQL Server" , kami telah menyiapkan terjemahan lain yang menarik untuk Anda.





Jika pengoptimal tidak benar menghitung jumlah memori yang diperlukan untuk menjalankan query, itu akan menjadi pemborosan memori yang dapat digunakan oleh proses lain, atau akan ada data yang tumpah ke disk (disk spill). Untuk mengatasi masalah ini, Microsoft telah menambahkan Memory Grant Feedback. Dalam artikel ini, Greg Larsen menjelaskan cara kerjanya.


Memory Grant Feedback dalam versi SQL Server sebelumnya (sebelum SQL Server 2019 atau 15.x) hanya diterapkan untuk kueri yang berjalan dalam mode batch. Kueri dalam mode batch memindai dan menghitung hingga 900 baris sekaligus, berbeda dengan kueri dalam mode baris, saat hanya satu baris yang diproses pada satu waktu. Dalam versi 15.x, umpan balik hibah memori telah diperluas untuk mendukung permintaan dalam mode string.



Apa umpan balik dari pemberian memori? Ini adalah proses menyesuaikan perhitungan memori yang diperlukan untuk permintaan, dengan mempertimbangkan berapa banyak memori yang digunakan dalam eksekusi sebelumnya. Ini berarti bahwa jika permintaan cache menggunakan terlalu banyak memori pada menjalankan terakhir, maka SQL Server akan mengurangi alokasi memori pada menjalankan berikutnya. Atau jika SQL Server mendeteksi permintaan yang menggunakan disk karena fakta bahwa terakhir kali dialokasikan memori tidak mencukupi, itu akan menambah memori untuk permintaan. Tujuan umpan balik pada hibah memori adalah untuk menyesuaikan persyaratan memori setiap kali kueri dieksekusi sampai kueri menggunakan jumlah memori yang sesuai dengan jumlah baris yang diproses.



Fungsi Memory Grant Feedback (dalam mode batch dan string) adalah bagian dari keluarga fungsi Intelligent Query Processing (IQP, Intelligent Query Processing ). Gambar 1 menunjukkan diagram yang menunjukkan semua fitur IQP yang hadir dalam Azure SQL Database dan SQL Server 2019, serta fitur yang awalnya merupakan bagian dari Pemrosesan Adaptif Adaptif yang disertakan dalam versi sebelumnya Azure SQL Database dan SQL Server 2017.

Daftar 1. Uji kueri



Sebelum menjalankan kueri ini, izinkan saya memberi tahu Anda mengapa perkiraan memori yang diperlukan mungkin salah, dan masih membuat beberapa konfigurasi awal.



Mengapa perkiraan memori yang dibutuhkan mungkin salah



Ada banyak faktor yang memengaruhi jumlah memori yang dibutuhkan untuk suatu permintaan. Saat menghitung total memori yang dibutuhkan, optimizer kueri harus mempertimbangkan persyaratan memori untuk setiap operasi dalam hal eksekusi. Berikut adalah beberapa parameter yang dipertimbangkan pengoptimal ketika menghitung memori yang diperlukan untuk menjalankan kueri:



  • Kardinalitas tabel yang terlibat dalam kueri
  • Pengeras suara yang dapat dipilih
  • Diperkirakan Ukuran Baris
  • Apakah perlu menyortir dan / atau menggabungkan data
  • Apakah kueri berjalan secara paralel


Untuk benar menghitung memori yang diperlukan, SQL Server harus memperhitungkan banyak aspek di mana memori mungkin diperlukan saat menjalankan kueri. Jika terlalu banyak memori dialokasikan, maka SQL Server membuang memori ketika bisa digunakan oleh proses lain. Atau jika tidak cukup memori yang dialokasikan, maka permintaan akan menggabungkan data ke disk, yang merupakan operasi yang mahal.



SQL Server tidak selalu baik dalam menghitung memori pertama kali. Statistik yang kedaluwarsa atau tidak akurat adalah alasan paling umum SQL Server menghitung jumlah memori yang salah. Tetapi bahkan jika statistiknya benar, SQL Server dapat melebih-lebihkan atau meremehkan jumlah memori yang diperlukan untuk beberapa alasan lain. Salah satu alasan ini adalah bahwa SQL Server tidak benar mengevaluasi Ukuran Baris Diperkirakan untuk kolom yang dikembalikan. Dalam contoh berikut, kita akan melihat bagaimana memperbesar baris menyebabkan kueri menerima peringatan hibah berlebihan karena tiga kolom NVARCHAR (MAX), yang selalu NULL.



Tapi sekarang, jika SQL Server salah menghitung memori yang diperlukan pada eksekusi pertama permintaan, maka untuk eksekusi selanjutnya, umpan balik pada hibah memori digunakan. Setiap kali kueri dieksekusi, informasi tentang memori yang dialokasikan untuk eksekusi sebelumnya dari kueri ini dianalisis. Jika pada awal sebelumnya alokasi memori salah, maka Mesin Database mengoreksi alokasi memori.



Setup untuk jalankan pertama



Untuk menunjukkan bagaimana, seiring waktu, alokasi memori menyesuaikan dengan permintaan, Anda perlu melakukan beberapa pekerjaan persiapan. Yang pertama adalah mengubah tingkat kompatibilitas basis data WideWorldImporters menjadi 150. Anda perlu melakukan ini karena Memory Grant Feedback (Mode Baris) hanya tersedia untuk basis data dengan tingkat kompatibilitas 150. Kode dalam Daftar 2 mengubah tingkat kompatibilitas basis data di 150. 



ALTER DATABASE WideWorldImporters SET COMPATIBILITY_LEVEL = 150;
GO

Ketika jumlah memori yang dibutuhkan tidak stabil



Ada permintaan yang setiap eksekusi membutuhkan jumlah memori yang berbeda. Jika ini terjadi, maka SQL Server berhenti mencoba menyesuaikan ukuran memori yang dialokasikan dan menonaktifkan umpan balik untuk permintaan tersebut. Salah satu situasi ketika ini terjadi adalah peluncuran prosedur tersimpan, di mana, pada setiap eksekusi, nilai parameter yang berbeda dilewatkan.



Untuk mereproduksi situasi ini ketika SQL Server menonaktifkan umpan balik pada hibah memori, kami menjalankan prosedur tersimpan dengan parameter yang berbeda beberapa kali. Dengan demikian, pada setiap eksekusi, permintaan dalam prosedur akan memerlukan jumlah memori yang berbeda karena jumlah catatan yang dikembalikan berbeda, tergantung pada parameter. Kode prosedur yang disimpan ditunjukkan pada Listing 4. 



USE WideWorldImporters;
GO
CREATE OR ALTER PROC TestProc (@KeyValue int)
AS
SELECT * FROM Sales.Orders
WHERE OrderID > 1 and OrderID < @KeyValue
ORDER BY OrderDate

DECLARE @I INT = 1;

Seperti yang bisa Anda tebak, setiap eksekusi akan membutuhkan jumlah memori yang berbeda, karena jumlah baris sangat berbeda.

Menonaktifkan Umpan Balik Memori



Jika karena alasan apa pun Anda ingin menonaktifkan umpan balik pemberian memori dalam mode baris (Umpan Balik Hibah Memori - Mode Baris), ada beberapa cara untuk melakukan ini. Yang pertama adalah mengatur tingkat kompatibilitas untuk database menjadi kurang dari 150. Hanya ada satu masalah dengan metode ini: semua fungsi lain yang muncul di versi 15.x juga akan dinonaktifkan. Atau, Anda dapat menonaktifkan umpan balik tingkat basis data melalui opsi ROW_MODE_MEMORY_GRANT_FEEDBACK



. 



USE WideWorldImporters;
GO
ALTER DATABASE SCOPED CONFIGURATION SET ROW_MODE_MEMORY_GRANT_FEEDBACK = OFF;
GO


Daftar 7. Menonaktifkan umpan balik alokasi memori untuk database



Di sini, umpan balik pada hibah memori di tingkat database dinonaktifkan. Sekarang permintaan untuk WideWorldImporters tidak akan menggunakan umpan balik string. Tetapi ini bukan satu-satunya cara untuk menonaktifkan umpan balik. Anda juga dapat menonaktifkan umpan balik untuk permintaan tertentu menggunakan PETUNJUK, seperti yang ditunjukkan pada Listing 8. 



USE WideWorldImporters;
GO
SELECT * FROM Sales.Orders
ORDER By  OrderDate
OPTION
(USE HINT ('DISABLE_ROW_MODE_MEMORY_GRANT_FEEDBACK'));


Listing 8. Menggunakan HINT untuk menonaktifkan umpan balik



Listing 8 menggunakan petunjuk DISABLE_ROW_MODE_MEMORY_GRANT_FEEDBACK



yang menonaktifkan umpan balik hibah memori untuk satu permintaan.



Optimalisasi perhitungan memori yang diperlukan untuk permintaan



Umpan balik tentang hibah memori dalam mode batch telah ada selama beberapa waktu. Tetapi hanya dalam versi 15.x apakah umpan balik ini tersedia untuk kueri dalam mode baris. Semua yang diperlukan untuk mengaktifkan umpan balik untuk kueri dalam mode string adalah mengatur tingkat kompatibilitas database ke 150. Ini, tentu saja, akan berfungsi jika Anda menggunakan versi Azure SQL Database atau SQL Server 2019 saat ini. Fungsi umpan balik hibah memori diaktifkan. dalam mode string, SQL Server menggunakan informasi alokasi memori dari eksekusi sebelumnya dari permintaan cache untuk menyesuaikan alokasi memori untuk eksekusi saat ini. Mengoreksi memori yang diperlukan dengan umpan balik setiap kali kueri dieksekusi cepat atau lambat akan mengarah pada perhitungan jumlah memori yang benar.
