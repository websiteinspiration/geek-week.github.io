Reactの制御および非制御コンポーネントは複雑である必要はありません

こんにちは、ハブル！Gosha Arinichによる「Reactでの制御および非制御のフォーム入力は複雑である必要はない」という記事の翻訳を紹介します。

「refsは悪い」とドキュメントが主張している一方で、「あなたは「setState」を使用してはいけない」という多くの記事を見たことがあるかもしれません。これはすべて非常に矛盾しています。時々、物事を正しく行う方法を理解するのが難しい、これらの方法を選択するための基準は何

ですか？では、どのようにフォームを作成しますか？結局のところ、フォームは多くのWebアプリケーションの中心です。それでも、Reactでのフォーム処理は土台ですよね？

しかし、それはそうではありませんとても難しいこれらのアプローチの違いと、それぞれのアプローチをいつ使用すべきかを説明しましょう。

管理されていないコンポーネント


アンマネージコンポーネントは、通常のHTMLフォームに似ています。

class Form extends Component {
  render() {
    return (
      <div>
        <input type="text" />
      </div>
    );
  }
}

彼らはあなたがタイプしたすべてを覚えています。次に、refを使用してそれらの値を取得できます。たとえば、onClickハンドラでは次のようになります。


class Form extends Component {
  handleSubmitClick = () => {
    const name = this._name.value;
    // do something with `name`
  }
  render() {
    return (
      <div>
        <input type="text" ref={input => this._name = input} />
        <button onClick={this.handleSubmitClick}>Sign up</button>
      </div>
    );
  }
}

つまり、必要なときにフィールドから値を「引き出す」必要があります。これは、フォームを送信するときに実行できます。

これは、フォームを実装する最も簡単な方法です。もちろん、それを使用する十分な理由があるはずです。つまり、Reactの研究中の最も単純なフォームです。
ただし、この方法はそれほど柔軟ではないので、マネージコンポーネントについてさらに詳しく見ていきましょう。

管理コンポーネント：


マネージコンポーネントは、現在の値をプロパティとして受け入れ、この値を変更するためのコールバックも受け入れます。これはコンポーネントを制御するためのより「反応的な」方法であると言えますが、これは常にこの方法を使用する必要があることを意味するものではありません。

<input value={someValue} onChange={handleChange} />

これはすべて非常に優れていますが、入力フォームの値は何らかの状態で存在している必要があります。通常、入力フォーム（つまりフォーム）をレンダリングするコンポーネントは、それらをその状態で保存します。


class Form extends Component {
  constructor() {
    super();
    this.state = {
      name: '',
    };
  }

  handleNameChange = (event) => {
    this.setState({ name: event.target.value });
  };

  render() {
    return (
      <div>
        <input
          type="text"
          value={this.state.name}
          onChange={this.handleNameChange}
        />
      </div>
    );
  }
}

（もちろん、別のコンポーネントの状態、または別の状態ストア（Reduxなど）でもかまいません）。

新しい文字を入力するたびに、handleNameChangeが呼び出されます。入力フォームの新しい値を受け取り、それを状態に書き込みます。



  • それはすべて空の行で始まります-'';
  • 文字「a」を入力すると、handleNameChangeがそれを取得し、setStateを呼び出します。次に、入力フォームが値「a」で再度レンダリングされます。
  • 文字「b」を入力すると、handleNameChangeが値「ab」を取得して、stateに設定します。Opet入力フォームがレンダリングされますが、値は 'ab'になっています。

このストリームは変更をフォームに「プッシュ」するようです。そのため、コンポーネントは、明示的な更新リクエストを必要とせずに、常に入力データの現在の値を持っています。

つまり、データ（状態）とユーザーインターフェイス（入力フォーム）は常に同期しています。状態はフォームに値を与え、フォームは現在の状態の値を変更します。

また、フォームは変更に即座に応答できることを意味します。これは、次のことを行うために必要です。

  • 検証などの迅速なフィードバック。
  • すべてのフォームフィールドが有効になるまで、特定のボタンを無効にします。
  • クレジットカード番号などの特定の入力フィールド形式の処理を有効にします。

しかし、それが必要なく、管理されていないコンポーネントの方がはるかに簡単だと思われる場合は、それらを使用してください。

コンポーネントを「管理可能」にするものは何ですか？


もちろん、チェックボックス、ラジオ、テキストエリア、選択などの他のフォーム要素があります。
値の使用の小道具を設定すると、コンポーネントが管理可能になります。それで全部です。

ただし、フォーム要素ごとに値を設定する方法が異なるため、以下に一般的な理解のための小さな表を示します。

<input type="checkbox" />
 チェック= {ブール} onChange event.target.checked 
<input type="radio" />

 値= "string" onChange event.target.value

調査結果


マネージコンポーネントとアンマネージコンポーネントの両方に長所と短所があります。特定の状況を評価し、アプローチを選択します。これで問題が解決した場合は、それを利用してみませんか？

ユーザーインターフェイスとの対話という点でフォームが非常に単純な場合は、アンマネージコンポーネントが最適です。いろいろな記事が悪いと言うことを聞く必要はありません。



さらに、コンポーネントの種類を選択することは、一度にすべての決定を下すことではありません。管理されていないコンポーネントを常に管理されているコンポーネントに置き換えることができます。あるものから別のものへの移行はそれほど難しくありません。
