.NETの優先ロック

プログラムで複数のスレッドを使用する各プログラマーは、同期プリミティブに遭遇しました。.NETのコンテキストでは、それらはたくさんあります。ここではリストしませんが、MSDNはすでにこれを行っています。

私はこれらのプリミティブの多くを使用する必要があり、それらはタスクに完全に対処するのに役立ちました。しかし、この記事では、デスクトップアプリケーションの通常のロックと、PriorityLockと呼ばれる新しい（少なくとも私にとっては）プリミティブがどのように表示されるかについてお話します。

問題


高負荷のマルチスレッドアプリケーションを開発する場合、無数のスレッドを処理するマネージャーがどこかに表示されます。それで私と一緒でした。そして、このマネージャーは機能し、何百ものスレッドからの大量のリクエストを処理しました。そして、すべては彼と一緒に大丈夫だったが、通常のロックの中で働いた。

そして、ある日、ユーザー（たとえば、私）がアプリケーションインターフェイスのボタンを押すと、ストリームはマネージャー（もちろん、UIストリームではない）に飛んで、非常にフレンドリーなレセプションが表示されることを期待しています。誰があなたを導きましたか 私はあなたのような950をもっと持っています。行ってそれらに入る。どのようにそれを理解するかは気にしません。」これは、.NETでロックが機能する方法です。そして、すべてが問題ないようで、すべてが正しく実行されますが、ユーザーは自分のアクションへの応答を数秒待つことを明らかに計画していませんでした。

ここで悲惨な話が終わり、技術的な問題の解決策が始まります。

決定


標準のプリミティブを調べたところ、適切なオプションが見つかりませんでした。したがって、私は自分のロックを作成することにしました。これには、標準の優先度の高いエントリが含まれます。ちなみに、筆記後、nugetについても調べましたが、検索が不十分だったかもしれませんが、そのようなものは見つかりませんでした。

このようなプリミティブ（またはプリミティブではない）を作成するには、SemaphoreSlim、SpinWait、およびInterlockedの操作が必要でした。スポイラーでは、PriorityLockの最初のバージョン（同期コードのみですが、それが最も重要です）とその説明を引用しました。

public class LockMgr
{
    internal int HighCount;
    internal int LowCount;
    internal Thread CurThread;
    internal int RecursionCount;

    internal readonly SemaphoreSlim Low = new SemaphoreSlim(1);
    internal readonly SemaphoreSlim High = new SemaphoreSlim(1);
    internal SpinWait LowSpin = new SpinWait();
    internal SpinWait HighSpin = new SpinWait();

    public Locker HighLock()
    {
        return new HighLocker(this);
    }
    public Locker Lock(bool high = false)
    {
        return new Locker(this, high);

public class Locker : IDisposable
{
    private readonly bool _isHigh;
    private LockMgr _mgr;

    public Locker(LockMgr mgr, bool isHigh = false)
    {
        _isHigh = isHigh;
        _mgr = mgr;
        if (mgr.CurThread == Thread.CurrentThread)
        {
            mgr.RecursionCount++;
            return;
        }
        if (_isHigh)
        {
            Interlocked.Increment(ref mgr.HighCount);
            mgr.High.Wait();
            while (Interlocked.CompareExchange(ref mgr.LowCount, 0, 0) != 0)
                mgr.HighSpin.SpinOnce();
        }
        else
        {
            mgr.Low.Wait();
            while (Interlocked.CompareExchange(ref mgr.HighCount, 0, 0) != 0)
                mgr.LowSpin.SpinOnce();
            try
            {
                mgr.High.Wait();
                Interlocked.Increment(ref mgr.LowCount);
            }
            finally
            {
                mgr.High.Release();
            }
        }
        mgr.CurThread = Thread.CurrentThread;
    }

    public void Dispose()
    {
        if (_mgr.RecursionCount > 0)
        {
            _mgr.RecursionCount--;
            _mgr = null;
            return;
        }
        _mgr.RecursionCount = 0;
        _mgr.CurThread = null;
        if (_isHigh)
        {
            _mgr.High.Release();
            Interlocked.Decrement(ref _mgr.HighCount);
        }
        else
        {
            _mgr.Low.Release();
            Interlocked.Decrement(ref _mgr.LowCount);
        }
        _mgr = null;
    }
}

public class HighLocker : Locker
{
    public HighLocker(LockMgr mgr) : base(mgr, true)
    { }
}


LockMgrクラスオブジェクトの使用は非常に簡潔でした。この例は、重要なセクション内で_lockMgrを再利用する可能性を明確に示していますが、優先順位はもはや重要ではありません。

だから私は私の問題を解決しました。ユーザーアクションの処理は高い優先度で実行されるようになり、誰も傷つかず、誰もが勝ちました。

非同期性


SemaphoreSlimクラスのオブジェクトは非同期待機をサポートしているので、この機会も自分に追加しました。コードの違いは最小限であり、記事の最後でソースコードへのリンクを提供します。

ここで、タスクがスレッドにアタッチされていないため、ロックの非同期再利用を同様の方法で実装できないことに注意してください。さらに、MSDNで説明されているTask.CurrentIdプロパティは何も保証しません。これは私のオプションが終わったところです。

解決策を探して、私はNeoSmart.AsyncLockプロジェクトに出くわしました。非同期ロックの再利用のサポートが示されている説明で。技術的には、作品を再利用します。しかし、残念ながらロック自体はロックではありません。このパッケージを使用する場合は注意してください。正しく機能しないことに注意してください！

結論


結果は、再利用を伴う同期操作と、再利用を伴わない非同期操作をサポートするクラスです。非同期操作と同期操作は並べて使用できますが、一緒に使用することはできません！非同期オプションの再利用がサポートされていないためです。

私はそのような問題で一人ではなく、私の解決策が誰かに役立つことを願っています。ライブラリをgithubとnugetに投稿しました。

リポジトリには、PriorityLockの正常性を示すテストがあります。このテストの非同期部分で、NeoSmart.AsyncLockがテストされ、テストは失敗しました。
