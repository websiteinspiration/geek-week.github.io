まず、ちょっとした理論


ほとんどのAndroidアプリはランタイム（ART）上で実行されます。これは、廃止されたDalvik仮想マシンに置き換わるものです。 ARTとDalvikは、Java仮想マシン（JVM）に似ており、類似の設計原則を共有しています。アプリケーションデータを格納するために、スタックとヒープの2つの別々のスペースを使用します。

スタックメモリ

Javaのスタックメモリは、ローカル変数（プリミティブ型とオブジェクト参照）を格納するために使用されます。各Javaスレッドには、独自の個別のスタックがあります。スタックメモリは、ヒープメモリと比較して比較的小さいです。 DalvikのJavaスタックサイズは、Javaコードの場合は通常32 KB、ネイティブコード（C ++ / JNI）の場合は1 MBです。 JavaとC ++の統合スタックがARTに登場しました。そのサイズは約1 MBです。

アプリケーションがスタックメモリ全体を限界まで選択すると、エラーがスローされStackOverflowErrorます。スタック制限に達する可能性が最も高い理由は、無限再帰または過度に深いメソッド呼び出しのいずれかです。スタックメモリへの参照は、常にLIFOの順序で行われます（最後に来る-先着順）。メソッドが呼び出されるたびに、このメソッドのローカル変数を使用して新しいフレームがスタックにプッシュされます。メソッドが完了すると、そのフレームがスタックからポップされ、結果の値がスタックに返されます。したがって、最初の問題（無限再帰）は修正が簡単なバグですが、2番目の問題は、再帰的なメソッド呼び出しをデプロイしてループに変換するというリファクタリングが必要です。

ヒープメモリ

Javaのヒープメモリは、オブジェクトを割り当てるために仮想マシンによって使用されます。オブジェクトが作成されるときはいつでも、それはヒープ上で発生します。JVMやARTなどの仮想マシンは、定期的にガベージを収集し、参照されなくなったすべてのオブジェクトを削除して、メモリを解放して新しいオブジェクトを割り当てます。
使いやすさを確保するために、Androidは実行中の各アプリケーションのヒープサイズを厳しく制限しています。ヒープサイズの制限はデバイスによって異なり、そのデバイスに搭載されているRAMの量によって異なります。アプリケーションが最大ヒープサイズに達し、より多くのメモリを割り当てようとするOutOfMemoryErrorと、エラーが生成され、アプリケーションが終了します。この状況を回避するのに役立ついくつかの例を見てみましょう。

イベントに応じてメモリを解放する


Androidでは、より重要なタスクを実行するためにメモリを解放する必要があるときに、アプリケーションにメモリを割り当てるか、アプリケーションを強制的に終了させる必要があります。これが発生する前に、システムは必要のないすべてのメモリを提供できるようにします。アクティビティでは、インターフェースを実装する必要がありますComponentCallbacks2。この場合、システムでメモリが不足すると、メソッドが呼び出され、メモリonTrimMemory()を解放したり、メモリ不足の状況では機能しない機能を無効にしたりできます。

したがって、そのようなコールバックはPSPDFKitアプリケーションで処理されます。PSPDFKitアプリケーションは、キャッシュのためのメモリのアクティブな使用量の計算を使用して設計されているため、アプリケーションはできるだけスムーズに実行されます。最初は、デバイスで使用可能なメモリの量が不明であるため、PSPDFKitは状況に適応し、十分なメモリがないという通知を受け取ったときにメモリの使用を制限します。したがって、PSPDFKitと統合されたアプリケーションはローテクデバイスでも機能しますが、キャッシュが無効になっているためにパフォーマンスが低下します。

大きな山


高いメモリ要件に対処するためのフロントエンドソリューションの1つは、アプリケーションに大量のDalvikを要求することです。これを行うにandroid:largeHeap="true"は、ファイルの<application>タグに追加できますAndroidManifest.xml。

プロパティlargeHeapがvalue trueに設定されている場合、Androidは大きなヒープを持つアプリケーションのすべてのプロセスを作成します。この設定は、本質的にそれなしでは機能できない、つまり、同時にメモリに収まる必要のある膨大なリソースを使用するアプリケーションのみを対象としています。

可能なメモリ使用量の上限のみを上げたい場合は、大きなヒープを使用しないことを強くお勧めします。メモリーが少ない弱いデバイスで作業する場合、アプリケーションの大きな山でも十分ではない場合があるため、メモリー使用量は常に最適化する必要があります。

アプリケーションが使用できるメモリ量を確認する


アプリケーションのヒープの大きさを確認したり、コードや使用可能な機能をこれらのメモリ制限に動的に適合させたりしても問題はありません。メソッドを使用して実行時に最大ヒープサイズを直接確認できます（getMemoryClass()またはgetLargeMemoryClass()大きなヒープが有効になっている場合）。

Androidは、512 MBのRAMしか搭載していないデバイスもサポートしています。ローテクデバイスを無視しないようにしてください。メソッドの使用isLowRamDevice()アプリケーションが、十分な使用可能メモリがないデバイスで実行されているかどうかを確認できます。このメソッドの正確な動作はデバイスに依存しますが、RAMが1 GB未満のデバイスでは通常trueを返します。これらのデバイスでアプリケーションが正しく動作することを確認し、デバイスで大量のメモリを使用するすべての機能を無効にする必要があります。

最適化されたデータ構造を使用する


多くの場合、アプリケーションは、最適なデータ構造を使用しないという単純な理由で、メモリを使いすぎます。

Javaコレクションは効率的なプリミティブ型を格納できず、キーと値をパックする必要があります。たとえば、HashMap整数キーは最適化されたものに置き換える必要がありますSparseArray。最終的には、コレクションの代わりに常にraw配列を使用できます。これは、コレクションのサイズを変更できない場合に最適です。

メモリ使用量の点で非効率的な他のデータ構造には、さまざまなシリアル化が含まれます。はい、確かに、XMLまたはJSON形式を使用すると便利です。たとえば、プロトコルバッファーなど、より効率的なバイナリ形式を使用すると、メモリ使用量を削減できます。。

メモリを節約するためにデータ構造が最適化されたこれらの例はすべて、単なるヒントにすぎません。リファクタリングと同様に、まず問題の原因を見つけてから、そのようなパフォーマンスの最適化に進む必要があります。

メモリのシャッフルを防ぐ


Java / Android仮想マシンは、オブジェクトを非常に迅速に割り当てます。ガベージコレクションも非常に高速です。ただし、短期間に多数のオブジェクトを割り当てると、「メモリチャーン」と呼ばれる問題が発生する場合があります。この場合、仮想マシンはこのペースでオブジェクトを割り当てる時間がないため、ガベージコレクターがオブジェクトを破棄し、アプリケーションの速度が低下し、極端な場合はすべてのメモリを使い果たします。

この場合のAndroidテリトリーの主な問題は、ガベージコレクションが発生するタイミングを制御できないことです。これにより、問題が発生する可能性があります。たとえば、ガベージコレクターは、アニメーションが画面上で展開されたときに正確に機能し、しきい値の16ミリ秒を超えると、フレームがスムーズに表示されます。したがって、コードでのメモリの過剰割り当てを防ぐことが重要です。

メモリシャッフルにつながる状況の例は、大きなオブジェクトの割り当てです（たとえば、onDraw()プレゼンテーションメソッド内のペイント）。この場合、多くのオブジェクトがすばやく作成され、ガベージコレクションが開始される可能性があり、このビューのパフォーマンスに悪影響を与える可能性があります。上記のように、このような状況を回避するために、常にメモリ使用量を監視する必要があります。

結論


モバイルデバイスのランダムアクセスメモリ（RAM）は、非常に限られたリソースになる可能性があります。アプリケーションがラスターグラフィック（PDFビューア、Webブラウザ、フォトエディタ）や大きなメディアファイル（オーディオエディタまたはビデオエディタ）などの比較的大きなオブジェクトで動作する場合、アプリケーションでメモリを効率的に使用することが特に重要です。これらのヒントに従って、最もパワフルなデバイス上でも、許容可能なレベルで機能する高品質のアプリケーションを作成する方法を学びます。
