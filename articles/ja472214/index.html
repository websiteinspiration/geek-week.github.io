誤って分岐が予測されると、プログラムの実行時間が大幅に増加する可能性があります


最新のプロセッサはスーパースカラです。つまり、複数の命令を同時に実行できます。たとえば、一部のプロセッサは、サイクルごとに4〜6個の命令を処理できます。さらに、そのようなプロセッサの多くは、命令を順不同で開始することができます。これらのプロセッサは、コード内にあるコマンドの処理をずっと後で開始できます。

同時に、コードには多くの場合ブランチ（演算子if–then）が含まれています。このような分岐は、「遷移」として実装されることが多く、プロセッサはコードの下の命令の実行に進むか、現在のパスを続行します。

コマンドのスーパースカラー実行が順不同であると、分岐は困難です。このため、プロセッサには高度な分岐予測ブロックがあります。つまり、プロセッサは未来を予測しようとしています。トランジションを意味するブランチを見つけたとき、彼はプログラムがどの方向に進むかを推測しようとします。

多くの場合、これは非常にうまく機能します。たとえば、ほとんどのループはブランチとして実装されます。ループの各反復の終わりに、プロセッサは次の反復が実行されるかどうかを予測する必要があります。多くの場合、サイクルが継続することを（永遠に）プロセッサが予測する方が安全です。この場合、プロセッサはサイクルごとに1つの分岐のみを誤って予測します。

他の一般的な例があります。配列のコンテンツにアクセスする場合、多くのプログラミング言語は「バインドチェック」を追加します。これは、配列の値にアクセスする前に、インデックスの正確さの非表示チェックです。インデックスが正しくない場合、エラーが生成されます。それ以外の場合、コードは通常の方法で実行され続けます。通常の状況ではすべてのアクセス操作が正しいはずなので、境界チェックは予測可能です。したがって、ほとんどのプロセッサは結果をほぼ完全に予測するはずです。

分岐が予測しにくい場合はどうなりますか？


プロセッサ内では、実行されたが誤って予測された分岐にあるすべての命令をキャンセルし、計算を新たに開始する必要があります。分岐予測エラーごとに、10サイクル以上を支払うことが予想されます。このため、プログラムの実行時間が大幅に増加する可能性があります。

ランダムな整数を出力配列に書き込む単純なコードを見てみましょう。

while (howmany != 0) {
    out[index] =  random();
    index += 1;
    howmany--;
}

平均して適切な乱数を3サイクル生成できます。つまり、乱数ジェネレータの合計遅延は10サイクルに等しくなる可能性があります。しかし、プロセッサーはスーパースカラーです。つまり、複数の乱数計算を同時に実行できます。したがって、約3サイクルごとに新しい乱数を生成できます。

関数を少し変更して、奇数のみが配列に書き込まれるようにします。

while (howmany != 0) {
    val = random();
    if( val is an odd integer ) {
      out[index] =  val;
      index += 1;
    }
    howmany--;
}

この新機能の方が速いかもしれないと単純に思うかもしれません。実際には、平均して2つの整数のうちの1つだけを記録する必要があるためです。コードには分岐がありますが、整数のパリティをチェックするには、1ビットをチェックするだけです。

SkylakeプロセッサのC ++でこれら2つの関数をベンチマークしました。

すべての乱数を記録する 整数の3.3サイクル
奇数の乱数のみを書き込む 整数あたり15サイクル

2番目の機能は約5倍長く機能します！

ここで何か修正できますか？はい、分岐をなくすことができます。奇数の整数は、1の値が1と等しいビット単位の論理ANDになるように特徴付けることができます。秘訣は、ランダムな値が奇数である場合にのみ、配列のインデックスを1つ増やすことです。

while (howmany != 0) {
    val = random();
    out[index] = val;
    index += (val bitand 1);
    howmany--;
}

この新しいバージョンでは、必要でない場合でも、出力配列に常にランダムな値を書き込みます。一見すると、これはリソースの無駄です。しかし、それは誤って予測されたブランチから私たちを救います。実際には、パフォーマンスは元のコードとほぼ同じで、ブランチを備えたバージョンよりもはるかに優れています。

すべての乱数を記録する 整数の3.3サイクル
奇数の乱数のみを書き込む 整数あたり15サイクル
ブランチを削除して 整数あたり3.8サイクル

コンパイラはこの問題を自分で解決できますか？一般的に、答えはノーです。ソースコードに演算子が含まれている場合でも、コンパイラには分岐を完全に排除するオプションがある場合がありますif-then。たとえば、分岐は「条件付き移動」または他の算術トリックに置き換えることができます。ただし、そのようなトリックはコンパイラでの使用には安全ではありません。

重要な結論：誤って予測された分岐は重要な問題ではなく、大きな影響を与えます。
