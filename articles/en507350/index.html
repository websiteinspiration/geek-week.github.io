Besides the fact that we made presentations and told a lot of interesting things at our stand, we also learned a lot from communicating with conference participants and from speakers. So, at the autumn conference Joker 2019, a report by Dalia Abo Sheasha ‚ÄúMigrating beyond Java 8‚Äù inspired us to implement a new diagnostic rule that allows you to identify incompatibilities in the Java SE API between different versions of Java. This will be discussed.



Relevance of Java SE Compatibility Issues



Java SE 14 has already been released. Despite this, many companies continue to use previous versions of Java (Java SE 6, 7, 8, ...). Due to the fact that time is running and Java is being updated all the time, the compatibility problem between different versions of the Java SE API is becoming more and more relevant every year.



When new versions of Java SE are released, they are usually backward compatible with earlier versions, that is, for example, an application developed on the basis of Java SE 8 should run without problems on version 11 of Java. However, in practice, some incompatibility may arise in a number of classes and methods. This incompatibility lies in the fact that some APIs undergo changes: they are deleted, changed in behavior, marked as obsolete and much more.



This problem will only worsen when you start thinking about migrating your project to more recent Java SE. Or when the technical support of your application will receive more and more letters that the application behaves incorrectly or cannot start at all.



I think this is enough to draw attention!



Existing Toolkit



There is no one-size-fits-all solution for switching from one version of Java to another. And if your application is the result of many years of continuous development using a variety of non-trivial solutions, then when you are puzzled by the transition to the latest version of Java, you are likely to encounter a rather time-consuming process.



This process consists of identifying a problematic or potentially problematic API that needs to be reviewed and replaced with an alternative solution. This can seriously affect the business logic, which can take a lot of time to fix and test. If you need not only to switch to the new version of Java SE, but also to ensure compatibility with a number of versions, then this task will become much more complicated.



Having rummaged in the Internet on the topic of identifying API incompatibilities between different Java SEs, I have only come across tools that come with the JDK: javac, jdeps, jdeprscan.



There was no third-party tool in this business, except for the one I had the honor to meet at the Joker 2019 report - Migration Toolkit for Application Binaries .



javac



When developing an application, do not forget about compiler warnings. Everyone has a different attitude to warnings:



  • Someone strictly watches them and a little something immediately sounds the alarm. Perhaps a little paranoid, but it may well reduce technical risks in the future;
  • Someone collects them in a separate file with the thought that they will necessarily review them, how they will be freed;
  • Someone generally ignores warnings.


The first option is good because part of the problems can be solved without delay, for example, just not using a method or class that is marked as deprecated. Using such an API is not a blocking problem, but you should pay attention to it, since it is already likely that when using the newly released version of Java your application will behave differently or even crash.



jdeps



If you are already puzzled by the issue of migrating your application to a more recent version of Java SE, then jdeps is in a hurry to help you.



jdeps is a command line tool that performs static analysis of the dependencies of your application and libraries, accepting * .class files or * .jar as input . Starting with Java SE 8, it comes bundled with the JDK.



And here we are interested in the fact that if you run this tool with the --jdk-internals option , it will tell you which internal JDK API your every class depends on. This is a very important point due to the fact that the internal API does not guarantee that it will not change in future versions of Java.



Let's look at an example. Suppose you have been developing your Java 8 application for a long time and you have never been puzzled by the compatibility of the Java SE API used with newer versions. There was a question about migrating your application to, for example, Java 11. In this case, you can go along the path of minimal resistance and immediately start the application using Java 11. But something went wrong, and suppose your application started to crash. In this case, you can run jdeps from Java 11 by setting your application's * .jar files .



The result will be approximately as follows:













The output shows that the dependency sun.misc.BASE64Encoder in Java 11 will be removed. And the crash of your application when it was first launched in Java 11 is most likely due to the java.lang.NoClassDefFoundError error . In addition to informing you that it can neither please, jdeps can offer you an alternative dependency that can be used instead of the current one. In this case, he suggested replacing the remote dependency with java.util.Base64 .



The second warning of the tool signals that we are still using another internal dependency in the code, but so far it is correct. Will there be any changes to this API in future versions of Java, but this must be taken into account.



Of course, jdeps can do more than just test the use of internal JDK components. This is not included in the scope of this article, but you can familiarize yourself with its capabilities on the official page yourself.



jdeprscan



The goals of jdeprscan are exactly the same as those of jdeps, namely, help in finding an unwanted and problematic API.



jdeprescan is a static analysis tool that scans a * .jar file (or some other collection of * .class files) for using outdated API elements. Using an outdated API is not a blocking problem, but you should pay attention to it. Starting with Java SE 9, it comes bundled with the JDK.



Also suppose that there is a question of migrating the application to Java 11. In this case, by running the command 



jdeprscan --release 8 app.jar


You‚Äôll get a list of APIs that have become deprecated for Java 8, that is, the API that may be removed in future versions of Java. After correcting all the warnings, you can run 

jdeprscan --release 11 app.jar


which will produce a list of APIs that are already deprecated for Java 11. In this way, you can find and fix (if necessary) all the non-recommended APIs.



Migration Toolkit for Application Binaries



This tool is designed to help you quickly evaluate your user application for potential problems before deploying it on different servers (JBOSS, WebShere, Tomcat, WebLogic, ...). In addition to all the functionality, the tool also allows you to identify differences in the Java SE API of different versions.



Let's take a quick look at what this tool is all about.



A quick launch of the tool looks like this: 



java -jar binaryAppScanner.jar yourApp.jar --analyzeJavaSE 
--sourceJava=oracle8 --targetJava=java11 ....


The analyzeJavaSE option implies the use of various parameters, which you can learn about by calling help.



After starting the analysis, you will soon see a report in a web browser:

In the report, you can see the triggering of rules with 3 levels of severity:



  • Rules with a high level of severity (remote APIs, a change in behavior that makes the application inoperative and needs to be fixed);
  • Warning rules (changing API behavior that can lead to problems and they are worth exploring);
  • Information rules (using outdated APIs, API behavior that may slightly change behavior, but does not affect the program).


Each operation can be expanded and see a description of what and how. There is a recommendation in informational messages, they say run jdeps additionally besides us. They say we are focused on the migration of the application, and jdeps will additionally help to detect the problem in the internal JDK packages (in addition to what they find).



Also at the bottom of the report you can find a list of rules according to which the analysis was performed.

Implementation in PVS-Studio



After studying the considered tools, we came to the conclusion that the search for potential compatibility problems of different versions of the Java SE API is a worthy task for static analysis.



The tools discussed will really make it easier to migrate the application or search for an API that can break your application‚Äôs performance on more recent versions of Java SE (if you do not want to migrate the application). But thinking that these tools all the time you need to run on the command line separately from the development process to identify problems, we came to the decision that this is not very convenient. And starting from the fact that static analysis is necessary to detect problematic or potentially problematic code at the very early stages of development, we implemented diagnostic rule V6078, which will signal you about the "problem" API.



Rule V6078 warns you in advance that your code is dependent on certain Java SE API functions and classes, which may cause you difficulty in future versions of Java. And you at the very start of the implementation of a particular feature will not be tied to this API, thereby reducing technical risks in the future.



The diagnostic rule issues warnings in the following cases:



  • If the method / class / package is removed in the target version of Java;
  • If the method / class / package is marked deprecated in the target version of Java;
  • If the method has changed the signature.


The rule currently allows you to analyze the compatibility of Oracle Java SE from versions 8 to 14. For a rule to become active, it must be configured.



IntelliJ IDEA



In the IntelliJ IDEA plugin, you need to enable the rule and specify the parameters in the Settings> PVS-Studio> API Compatibility Issue Detection tab, namely:



  • Source Java SE - the version of Java on which your application is developed;
  • Target Java SE - the version of Java with which you want to check the compatibility of the used API in your application (Source Java SE);
  • Exclude packages - packages that you want to exclude from compatibility analysis (packages are listed with a comma).












Plugin for gradle



Using the gradle plugin, you need to configure the analyzer settings in build.gardle: 



apply plugin: com.pvsstudio.PvsStudioGradlePlugin
pvsstudio {
    ....
    compatibility = true
    sourceJava = /*version*/  
    targetJava = /*version*/
    excludePackages = [/*pack1, pack2, ...*/]
}


Plugin for Maven



Using the maven plugin, you need to configure the analyzer settings in pom.xml: 



<build>
  <plugins>
    <plugin>
      <groupId>com.pvsstudio</groupId>
      <artifactId>pvsstudio-maven-plugin</artifactId>
      ....
      <configuration>
        <analyzer>
          ....
          <compatibility>true</compatibility>
          <sourceJava>/*version*/</sourceJava>
          <targetJava>/*version*/</targetJava>
          <excludePackages>/*pack1, pack2, ...*/</excludePackages>        
        </analyzer>
      </configuration>
    </plugin>
  </plugins>
</build>


Using the kernel directly



If you use the analyzer directly through the command line, then to activate the compatibility analysis of the selected Java SE APIs, you must use the following parameters: 



java -jar pvs-studio.jar /*other options*/ --compatibility 
--source-java /*version*/ --target-java /*version*/ 
--exclude-packages /*pack1 pack2 ... */


Analyzer operation



Let's assume that we are developing an application based on Java SE 8, and we have a class with the following contents: 



/* imports */
import java.util.jar.Pack200;
public class SomeClass
{
  /* code */
  public static void someFunction(Pack200.Packer packer, ...)
  {
    /* code */
    packer.addPropertyChangeListener(evt -> {/* code */});
    /* code */
  }
}


Running a static analysis with various settings for the diagnostic rule, we will observe the following picture:



  • Source Java SE - 8, Target Java SE - 9
    • The 'addPropertyChangeListener' method will be removed.


  • Source Java SE - 8, Target Java SE - 11
    • The 'addPropertyChangeListener' method will be removed.
    • The 'Pack200' class will be marked as deprecated.


  • Source Java SE - 8, Target Java SE - 14
    • The 'Pack200' class will be removed.




First, in Java SE 9, the 'addPropertyChangeListener' method in the 'Pack200.Packer' class was removed. In version 11, the fact that the class 'Pack200' was marked as deprecated added to this. And in version 14, this class was completely deleted.



Therefore, starting the application in Java 11, you will get 'java.lang.NoSuchMethodError', and if you run in Java 14 - 'java.lang.NoClassDefFoundError'.



Knowing this information, when developing your application, you will consider alternative solutions to the task.



Ideas for further development



During the implementation of the diagnostic rule, ideas arose about expanding:



  • Due to the fact that jdeprscan and jdeps cannot warn about using reflection to access the encapsulated API, it makes sense to modify the rule so that it tries to figure out what kind of API they are trying to use. The result may not be perfect, but why not ?!
  • There is a wide variety of JDK implementations (from Oracle, IBM, Red Hat, ...). As a rule, they are compatible with each other. But how much does the internal JDK API differ? After all, developers can be tied to it, which can lead to potential problems when migrating an application from one JDK to another.


These are all research questions, and what will turn out in the end, time will tell =) If you know interesting scenarios for this diagnosis and would like to see them in PVS-Studio, then write to us .
