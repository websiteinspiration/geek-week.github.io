PostgreSQLアンチパターン：ウサギの穴の深さはどれくらいですか？階層を通過する

複雑なERPシステムでは、祖先と子孫の関係ツリーに均質なオブジェクトが並ぶと、多くのエンティティが階層的な性質を持ちます。これは、企業の組織構造（これらのすべてのブランチ、部門、ワークグループ）、製品カタログ、作業領域、および地理です。 POS、... 実際には、少なくとも一部の階層が結果にならないビジネス自動化の単一の領域はありません。しかし、「ビジネス」に従事していなくても、階層関係を簡単に見つけることができます。トリテ、あなたの家系図やショッピングセンターの敷地のフロアスキームも同じ構造です。 そのようなツリーをDBMSに格納する方法はたくさんありますが、今日は1つのオプションのみに焦点を当てます。

索引による一括校正


また、ノードごとに、個別のインデックス要求を行う必要がありますか？私たちは、インデックスから読み取ることはできません-それは何のことが判明し、一度に複数のキー上で1つのコール= ANY(array)。

そして、そのような識別子の各グループでは、前のステップで見つかったすべてのIDを「ノード」で取得できます。つまり、次の各ステップで、特定のレベルのすべての子孫を一度にすぐに検索します。

しかし、残念ですが、再帰的な選択では、サブクエリで自分自身を参照することはできませんが、前のレベルで見つかったものを正確に選択する必要があるだけです...サブクエリをサンプル全体ではなく、その特定のフィールドに対して行うことができます-できる。そして、このフィールドは配列にすることもできます-これは私たちが使用する必要があるものですANY。

少しワイルドに聞こえますが、図では、すべてが単純です。



WITH RECURSIVE T AS (
  SELECT
    ARRAY[id] id$
  FROM
    hier
  WHERE
    id = 5300
UNION ALL
  SELECT
    ARRAY(
      SELECT
        id
      FROM
        hier
      WHERE
        pid = ANY(T.id$)
    ) id$
  FROM
    T
  WHERE
    coalesce(id$, '{}') <> '{}' --     -  
)
SELECT
  unnest(id$) id
FROM
  T;

そして、ここで最も重要なことは、時間内に1.5回も勝つことではありませんが、30の代わりにインデックスへの呼び出しが5つしかないため、減算するバッファーが少なくなります。

ノードタグ


生産性の向上に役立つ次の考慮事項は、「葉」には子供を作れないことです。つまり、子供は何も見下す必要はありません。私たちのタスクの定式化では、これは、部門のチェーンに沿って進み、従業員に到達した場合、このブランチをさらに検索する必要がないことを意味します。

テーブルに追加booleanフィールドを導入してみましょう。これにより、ツリー内のこの特定のエントリが「ノード」であるかどうか、つまり、子を持つことができるかどうかがすぐにわかります。

ALTER TABLE hier
  ADD COLUMN branch boolean;

UPDATE
  hier T
SET
  branch = TRUE
WHERE
  EXISTS(
    SELECT
      NULL
    FROM
      hier
    WHERE
      pid = T.id
    LIMIT 1
);
--   : 3033    42 .

いいね！すべてのツリー要素の30％強だけが子孫を持つことがわかります。

ここLATERALで、少し異なるメカニズムを適用します-を介して再帰部分に接続します。これにより、再帰的な「テーブル」のフィールドにすぐにアクセスでき、ノードに基づくフィルタリング条件で集約関数を使用して、キーのセットを削減します。



WITH RECURSIVE T AS (
  SELECT
    array_agg(id) id$
  , array_agg(id) FILTER(WHERE branch) ns$
  FROM
    hier
  WHERE
    id = 5300
UNION ALL
  SELECT
    X.*
  FROM
    T
  JOIN LATERAL (
    SELECT
      array_agg(id) id$
    , array_agg(id) FILTER(WHERE branch) ns$
    FROM
      hier
    WHERE
      pid = ANY(T.ns$)
  ) X
    ON coalesce(T.ns$, '{}') <> '{}'
)
SELECT
  unnest(id$) id
FROM
  T;

＃2 基礎に戻る


このアルゴリズムは、「ツリーの上の」すべての要素のレコードを収集する必要がある場合に役立ちます。たとえば、どのソースシート（およびどのインジケーター）がサンプルに該当するかについての情報を維持します。たとえば、ノードへの集計を含むサマリーレポートを生成します。


要求は非常に扱いにくいので、さらに、概念実証としてのみ使用する必要があります。しかし、それがデータベースを支配している場合は、そのような手法の使用を検討する価値があります。

いくつかの簡単なステートメントから始めましょう：

  • データベースから同じレコードを1回だけ読み取ることをお勧めします。
  • データベースからのエントリは、個別よりも「バンドル」で効率的に読み取られます。

次に、必要なクエリを設計してみましょう。

ステップ1


明らかに、再帰の初期化（それがない場合！）では、ソース識別子のセットからリーフ自体のレコードを減算する必要があります。

WITH RECURSIVE tree AS (
  SELECT
    rec --    
  , id::text chld --  ""    
  FROM
    hier rec
  WHERE
    id = ANY('{1,2,4,8,16,32,64,128,256,512,1024,2048,4096,8192}'::integer[])
UNION ALL
  ...

「セット」が配列ではなく文字列に格納されているのが誰かにとって奇妙に思えた場合は、簡単な説明があります。文字列の場合、組み込みの「接着」機能string_aggが組み込まれていますが、配列の場合はできません。自分で実装することは難しくありませんが。

ステップ2


これで、セクションIDのセットが取得され、さらに差し引く必要があります。ほとんどの場合、それらはソースセットの異なるレコードに複製されます。ソースリーフに関する情報を維持しながら、それらをグループ化します。

しかし、ここでは3つの問題が待ち受けています。

  1. クエリの「部分再帰」部分には、集計関数cを含めることはできませんGROUP BY。
  2. 再帰的な「テーブル」への呼び出しは、ネストされたサブクエリに含めることはできません。
  3. 再帰部分のクエリにCTEを含めることはできません。

幸い、これらの問題はすべて簡単に回避できます。最後から始めましょう。

再帰部分のCTE


これはそれがどのように機能するかではありません：

WITH RECURSIVE tree AS (
  ...
UNION ALL
  WITH T (...)
  SELECT ...
)

そして、それが機能するので、括弧が決定します！

WITH RECURSIVE tree AS (
  ...
UNION ALL
  (
    WITH T (...)
    SELECT ...
  )
)

再帰的な「テーブル」のネストされたクエリ


うーん...再帰的なCTEの呼び出しはサブクエリに含めることはできません。ただし、CTEの内部にある場合があります。サブリクエストはすでにこのCTEを参照できます！

再帰内のGROUP BY


不愉快ですが、... DISTINCT ONウィンドウ関数を使用してGROUP BYをシミュレートする簡単な方法もあります。

SELECT
  (rec).pid id
, string_agg(chld::text, ',') chld
FROM
  tree
WHERE
  (rec).pid IS NOT NULL
GROUP BY 1 --  !

そしてそれはうまくいきます！

SELECT DISTINCT ON((rec).pid)
  (rec).pid id
, string_agg(chld::text, ',') OVER(PARTITION BY (rec).pid) chld
FROM
  tree
WHERE
  (rec).pid IS NOT NULL

ここで、数値IDがテキストに変換された理由を確認します。これにより、数値IDをコンマで接着できます。

ステップ3


最後に、何も残っていません。

  • グループ化されたIDのセットで「セクション」のレコードを読み取ります
  • 差し引かれたセクションをソースシートの「セット」と一致させる
  • を使用してセット文字列を「拡張」します unnest(string_to_array(chld, ',')::integer[])

WITH RECURSIVE tree AS (
  SELECT
    rec
  , id::text chld
  FROM
    hier rec
  WHERE
    id = ANY('{1,2,4,8,16,32,64,128,256,512,1024,2048,4096,8192}'::integer[])
UNION ALL
  (
    WITH prnt AS (
      SELECT DISTINCT ON((rec).pid)
        (rec).pid id
      , string_agg(chld::text, ',') OVER(PARTITION BY (rec).pid) chld
      FROM
        tree
      WHERE
        (rec).pid IS NOT NULL
    )
    , nodes AS (
      SELECT
        rec
      FROM
        hier rec
      WHERE
        id = ANY(ARRAY(
          SELECT
            id
          FROM
            prnt
        ))
    )
    SELECT
      nodes.rec
    , prnt.chld
    FROM
      prnt
    JOIN
      nodes
        ON (nodes.rec).id = prnt.id
  )
)
SELECT
  unnest(string_to_array(chld, ',')::integer[]) leaf
, (rec).*
FROM
  tree;
