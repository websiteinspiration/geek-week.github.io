話すのは難しいです。非プログラマーとのコミュニケーションに関するエッセイ

プログラマーは難しい問題についてさまざまなことわざを持っています。おそらく私のお気に入りのオプション：「コンピュータサイエンスには2つの難しい問題があります。キャッシュの無効化、ネーミング、およびユニットあたりのエラーです。」

私はこれらの問題のそれぞれに直面するのに十分な長さのソフトウェアを書いてきました。同時に、私はビジネスの側から話をして、顧客に私たちの製品の技術的な部分を説明しているので、プログラマーではない人に常に出会います。最も重要なことは、他の人が理解できるような方法でアイデアやコンセプトを正しく伝えることです。

そして、これは非常に困難です。

すべての専門分野と同様に、プログラマーは概念をすばやく転送するための独自の専門用語を開発しました。

テクノレペット


専門用語は非常に重要です。私と同僚が特定の問題を解決するためのアルゴリズムを選択した場合、「最初O(n^2)からのオーバーヘッドが最小限の最初のオプション、2番目のオプションはに減額されO(n log n)ますが、インストールと構成に費用がかかります」と言えます。

この単一の文は、アルゴリズムが使用されるべきシナリオとそれらが内部でどのように実装されるかについて多くを語っています：

  • オプションAは、入力が少ないシステムに最適です。
  • オプションBは、非常に大量の入力データを処理するのに最適です。この場合、システムのインストールと構成の高コストは、プロセスのパフォーマンスの向上によって相殺されます。
  • アルゴリズムをループで使用する場合は、最初のオプションを選択して、高価なインストールおよび構成コードを回避する必要があります。
  • おそらく、キャッシュを使用してオプションBをインストールするコストを削減できます。
  • おそらく、オプションAでは、入力データの各要素が他のすべての要素と比較されます（たとえば、for x in inputs { for y in inputs { if some_condition(x, y) { ... }}}）。
  • おそらく、オプションBは最初にツリーを作成し、次に線形検索を実行してから、このツリーで検索を実行します。

ご覧のとおり、正しい用語を使用するだけで、複数の段落の情報を1つの文で伝えることができました。

この場合、衝突、つまり2つ以上のオブジェクト間の交差を検出するアルゴリズムを意味しました。1つ目は入力データのすべての可能な組み合わせをチェックし、2つ目は象限ツリーまたはバイナリスペースパーティションを使用して近い要素のみをチェックします。

しかし、プログラミングの世界の外にいる人々（たとえば、機械または電気のエンジニア、マネージャー、マーケティング担当者）とのミーティングがある場合、この技術用語は人々を侮辱し、混乱させるだけで、ほとんど貴重な情報を送信しません。

これは私がテクノレスと呼ぶもので、プログラマーのつぶやきです。

率直に言って、通常、これらの詳細と用語は他の領域の人々には必要とされず、本当に気にしません。

たとえば、AからBへの最短経路を見つけ、ナビゲーショングリッドを生成し、A *検索を適用する驚くべき新しい関数を説明しているとします。

前の文のように、新しい関数を説明する必要はありません。代わりに、次のように言います。「私たちは可能な経路の束を計算してから、ゲーム業界で使用されているスマートアルゴリズムを適用して最適なルートを見つけます。」

ここでA *検索を使用していることは問題ではありません（ダイクストラアルゴリズムや幅検索とは異なります）。ゲームでは、*を使用するのはキャラクターを移動するだけではありません。対談者は、AからBへの適切な方法を見つけることができ、他の領域ですでにテスト済みの信頼性の高いツールを使用していることを知っていれば十分です。


あなたが何を意味するのかを示すのに害はありません...

経験豊富なプログラマーがtech-jargonを使用して卓越性を示し、信じられないほどの知性を示し、優位性を確立する様子を偶然観察しました。これは本当に迷惑です。

誤解しないでください。十分な能力を持つ対談者が「これは非常に難しいのですが、なぜXを作成しないのですか？」 -そして、あなたはあなたがここで専門家であることを示す必要がありますが、彼は彼が何について話しているのかわかりません...しかし、あなたはまださまざまな方法でそれを行うことができます。

テクノレペットを使ってエゴを満たそうとしないでください。テクノレペットを使用すると、職業が人を汚し、職業に不名誉を与えます。

尊敬


ですから、次のポイントに徐々に進みます...尊重を忘れないでください。

私が一緒に働く多くの人々は本当に賢く、その分野の専門家ですが、必ずしも情報システムを作成するのに十分な知識と経験があるわけではありません。

複雑な問題を説明するときは、多くの場合、概念を単純化する方が良いでしょう。しかし、軽蔑しないようにしてください。

私がしばしば言及したソフトウェアエンジニアは、「普通の」人がこれまたはその機能がどのように機能するかを尋ねたとき、「それは...複雑です」と答えました。 20年間のプログラミング経験がなければ理解できないほどコードが洗練されているかのようです。

このようにしないでください。

最良の答え：「少し単純化すると、最初にXを実行し、次にYを実行し、最後にZを実行しますが、12のボーダー状況（おそらく1つまたは2つを説明する）を覚えておく必要がありますが、本質はこれです。」あなたの同僚は賢く、テクノ専門用語がなければ、あなたを十分に理解できます。

別の本当に便利なトリックは、ラバーダックのように、プログラマーではない人をデッキとして使用することです。複雑な問題に取り組んでいる場合は、解決策を簡単な言葉で彼に説明し、より良いオプションを考え出せるかどうか尋ねます。

私たちの会社はCNCの分野で働いています。つまり、それは現実の世界と関係があります（たとえば、カットの幅の補正を実装しようとした場合）。エンジニアは、物理学と幾何学に関連する問題の分析が本当に得意です。

擬人化、誇張、類推


人々は社会的な存在です。私たちは、新しい概念を既存の概念に関連付けるために、関係を理解し​​、アナロジーを愛するようにプログラムされています。誰かと話しているときに、これらのトリックを使用して、複雑なトピックを理解することができます。

あなたの会社がオンライン調達システムを持っているとしましょう。マーケティング担当者は、顧客の質問により的確に答えるために、オンライン注文から小包の配達までのチェーン全体を理解したいと考えています。私はこのようなことを言うことができます：

これらすべては少しコミカルに聞こえますが、この例はかなりまとまりがありませんが、これは、「Webサーバー」、「イベントアーキテクチャ」、および「Apache Kafka」という用語を含む大規模なネットワーク図よりもわかりやすい説明であることを保証できます（テクノレプテのセクションを参照））

もう一つの例。パス検索アルゴリズムがどのように機能するかを部外者に説明しようとする場合、「訪問したパスにはより多くの重みが与えられる」とは言わず、「アルゴリズムは実際にすでに進んだパスに沿って進むことを望んでいない」と言います。

これは微妙な違いですが、アルゴリズムの擬人化を行っています。彼は「本当にXを作りたい」または「Yを避けようと非常に努力する」と言います。多くの場合、これは人々が理解するのに役立ちます。

$ FEATUREを10倍に加速しました


いくつかのパフォーマンス調整を行い、特定のコード部分がはるかに速く動作するとします。これをプログラマーではない人やソースコードを見ない人に説明するにはどうすればいいですか？

一方では、先週あなたが何をしたかという質問に対して、その人に真実を伝えることができます。「lookup_something ()一般的な検索を高速化し、アルゴリズムをdetect_collisions()c からに変換するためにキャッシングを追加しましO(n^2)たO(n log n)」が、彼らは何も理解していない可能性が高い（technoleptを参照） 。「生産性を最適化した」

と言うこともできますが、言い訳のように聞こえ、マネージャーはなぜあなたを雇ったのか考えます。 多くの場合、開発者は「$ FEATUREを10倍高速化した」と述べています。

（または他の任意の数）とこれは制限されています。そのようなフレーズは、町民からのお祝いの束を楽しむことを可能にしますが、しばしばそれは少し...誤解を招くものです。

ほとんどの場合、実際にはいくつかの関数のパフォーマンスを改善しましたが、これらの関数がパフォーマンスのボトルネックでなかった場合、高い確率で誰も違いを見ることはありません。

注意。アムダールの法則に精通している場合、ボトルネックではない関数のパフォーマンスを向上させることは、本質的に一貫性のあるタスクにより多くのCPUコアを使用することに似ています。

質問はまた頼みます：生産性を10倍にするために何をしましたか？これは驚くほど具体的な数ですか？ソースコードは1つの作業を数回繰り返しましたか？または、データを要求した（たとえば、ハードウェアまたは一部のサードパーティのサイトから）必要な時間よりも10倍遅くなりましたか？これでも、初期のポーリング速度を（不十分に）推測したか、より効率的なイベントアーキテクチャの代わりにポーリングを使用してパフォーマンスを犠牲にしているため、自分の能力に疑問を感じます。

正確さとわかりやすさの中間点を見つけた方がいいようです。コードフラグメントのアルゴリズムの複雑さを改善する変更を加えた場合（たとえば、関数detect_collisions()がからO(n^2)に切り替えられたO(n log n)場合）、次のように言うことができます。「関数Xは、妥当な時間内に大量の入力にスケーリングできるようになりましたが、以前はできませんでした。」

これは、同僚や上司の尊敬を失う素晴らしい方法です。

一方、「わかりませんが、5〜10分で答えられます」という言葉は、一度にいくつかのことを示しています。

  • あなたは彼の質問に答えるためにあなたの時間の一部を費やすのに十分な人を尊重します。
  • あなたはいいやつだ:)
