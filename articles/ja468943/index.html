これとEcmaScriptのScopeChain

こんにちは、ハブル！

では前の記事、 ECMAScriptのとJSと古典言語のOOPとの違いについて、初心者の開発者の人気の誤謬に適用されるように、私たちは、OOPの一般的な理論を検討しました。

我々は他の二つの同様に重要なのECMAScriptの概念について話しましょう今日、つまり、実行コンテキストとエンティティの関係（これは、この接続である）と発電状況（とエンティティの関係ScopeChain）。

それでは始めましょう！

実際には、状況EcmaScript言語のこの中心的な概念では、状況は少し複雑になります。順番に考えてみましょう。

変数がグローバルに宣言されているJavaScriptプログラムがあるとします。グローバル関数;ローカル関数（他の関数内で宣言）、関数から返される関数。

実行可能コードに制御を移すと、実行コンテキストにエントリが作成されます。実行可能コード-これは、特定の時点で実行する任意のコードであり、グローバルコードまたは任意の関数のコードにすることができます。

実行コンテキストは、コードを分類して区切る抽象化です。この抽象化の観点から、コードはグローバル（接続されたスクリプト、インラインスクリプト）と関数コード（ネストされた関数のコードは親関数のコンテキストに属していません）に分けられます。

3番目のタイプ-EvalCodeがあります。この記事では、それを無視します。

論理的には、実行コンテキストのコレクションはスタックです、後入れ先出し（lifo）の原則に取り組んでいます。スタックの一番下は常にグローバルコンテキストで、一番上が現在の実行可能ファイルです。関数が呼び出されるたびに、そのコンテキストにエントリが作成されます。関数が完了すると、そのコンテキストは終了します。使用済みコンテキストは、スタックから順番に逆の順序で削除されます。

上記のコードを見てください。グローバルコードでfooBar関数を呼び出しています。fooBar関数では、すぐに呼び出す無名関数を返します。スタックでは、次の変更が発生します。グローバルコンテキストがそれに入る-fooBarが呼び出されると、そのコンテキストがスタックに乗る-fooBarコンテキストが終了し、無名関数が返されるスタックから削除されます- 匿名関数が呼び出され、そのコンテキストがスタックにプッシュされます- 匿名関数が処理され、値が返され、そのコンテキストがスタックから削除されます-スクリプトの最後に、グローバルコンテキストがスタックから削除されます。

実行コンテキストは、条件付きでオブジェクトとして表すことができます。このオブジェクトのプロパティの1つは、字句環境（LO）です。

字句環境には次のものが含まれます。

  • すべてのコンテキスト変数宣言
  • すべての関数宣言
  • 関数のすべての仮パラメーター（関数のコンテキストについて話している場合）

実行コンテキストに入ると、インタープリターはコンテキストをスキャンします。すべての変数宣言と関数宣言は、コンテキストの最初まで続きます。変数はundefinedと同じように作成され、関数は完全に使用できる状態です。

これはまた、いくつかの初心者の面接の答えとして実行コンテキストのプロパティではなく、コンテキスト自体は、！これは、コンテキストに入るときに定義され、コンテキストの有効期間が終了するまで（コンテキストがスタックから削除されるまで）変更されません。

グローバル実行コンテキストでは、これはストリクトモードによって決定されます：strictモードがオフの場合、これにはグローバルオブジェクトがあり（ブラウザーではウィンドウオブジェクトの最上位にプロキシされます）、 'use strict'ではこれは定義されていません。

これは関数のコンテキストで-質問ははるかに興味深いです！
関数内のこれは呼び出し元によって決定され、呼び出しの構文に依存します。たとえば、ご存じのとおり、これを呼び出しにしっかりと固定できるメソッド（call、apply）と、「fixed this」（bind）でラッパーを作成できるメソッドがあります。これらの状況では、これを明示的に指定し、その定義に疑いの余地はありません。

通常の関数呼び出しでは、状況はさらに複雑になります。

EcmaScript組み込み型の1つであるReferenceTypeは、これが関数にどのように配置されるかを理解するのに役立ちます。。これは、実装レベルで使用できる内部タイプの1つです。論理的には、これは2つの基本プロパティ（ReferenceTypeが返される特定の基本オブジェクトへの参照）、propertyName（ReferenceTypeが返されるオブジェクト識別子の文字列表現）を持つオブジェクトです。

ReferenceTypeは、すべての変数宣言、関数宣言、およびプロパティアクセスに対して返されます（これは、これを理解する上で私たちが関心を持つケースです）。

決定するためのルール、これを通常の方法で呼び出される機能のため：
存在する場合にするReferenceType機能起動ブラケットの左側には、その後のベースこのthis ReferenceType。他のタイプのブラケットの左側にある場合、thisグローバルオブジェクトがに固定されるかundefined（実際に貼り付けnullが、ヌルは、ECMAスクリプトの観点から特定の値を持っていないので、それは同じであってもよい参照先のグローバルオブジェクトにキャストされundefined応じ厳格モードから）。

例を見てみましょう：

const x = 0;

const obj = {
  x: 10,
  foo: function() {
    return this.x;
  }
}

obj.foo();//  10 ..    ReferenceType  base     obj

const test = obj.foo;//      

test();//  0 ..  test()   .test(),..  base    ,       0.

定義の仕方がわかりやすいと思います。ここで、あまり明白でないいくつかのケースを考えます。

関数式


ReferenceTypeに少し戻りましょう。この型には、ReferenceTypeを通じて取得したオブジェクトの真の型を返す組み込みのGetValueメソッドがあります。式ゾーンでは、GetValueが常に発生します。

例：

(function (){
  return this;// this     undefined    strict mode
})()

これは、GetValueが常に式ゾーンで機能するためです。 GetValueは関数タイプを返しますが、アクティブ化ブラケットの左側はReferenceTypeではありません。決定するための私たちのルールを思い出して、これを：他のタイプは、ブラケットの左側にある場合、thisグローバルオブジェクトが置かれているにかundefined（実際に置くnullが、ヌルは、それがグローバルオブジェクトへのリンクに変換され、ECMAScriptのの観点から一定の値を持っていないので、 strictモードによっては未定義になる場合があります）。

式ゾーンは、割り当て（=）、演算子||です。または他の論理演算子、三項演算子、配列初期化子、コンマ区切りリスト。

const x = 0;

スコープチェーン


スコープチェーンも、このような実行コンテキストのプロパティです。これは、現在のコンテキストとすべての生成コンテキストの字句環境のオブジェクトのリストです。識別子名を解決するときに変数の検索が行われるのは、このチェーン内です。

注：これは、関数を実行コンテキストに関連付け、ScopeChainを子コンテキストに関連付けます。

仕様には、ScopeChainが配列であると記載されています。

  SC = [LO, LO1, LO2,..., LOglobal];

ただし、JSなどの一部の実装では、スコープチェーンはリンクリストを介して実装されます。

ScopeChainをよりよく理解するために、関数のライフサイクルについて説明します。作成フェーズと実行フェーズに分かれています。

関数が作成されると、内部[[SCOPE]]プロパティが割り当てられます。
で[SCOPE] 、より高い（発生）コンテキストの字句環境のオブジェクトの階層チェーンが記録されます。このプロパティは、ガベージコレクターによって関数が破棄されるまで変更されません。

注意！[[SCOPE]] ScopeChainとは異なり、関数自体のプロパティではなく、そのコンテキストではありません。

関数が呼び出されると、その実行コンテキストが初期化され、入力されます。ScopeChain = LO（関数自体の）+ [[SCOPE]]（コンテキストに影響するLOの階層チェーン）がコンテキストに付加されます。

識別子名の解決-ScopeChainチェーン内のLOオブジェクトの左から右への順次ポーリング。出力はReferenceTypeで、その基本プロパティは識別子が見つかったLOオブジェクトを指し、PropertyNameは識別子名の文字列表現になります。 これは、フードの下にクロージャーが配置されている方法です！クロージャは、本質的には、ScopeChainで、関数に識別子が存在するすべての変数を検索した結果です。



const x = 10;

function foo () {
  return x;
}

(function (){
  const x = 20;
  foo();// 10 ..     <b><i>[[SCOPE]]</i></b> foo         
})()

次の例は、ライフサイクル[[SCOPE]]を示しています。

function foo () {
  const x = 10;
  const y = 20;

  return function () {
    return [x,y];
  }
}

const x = 30;

const bar = foo();//   ,   foo   
bar();// [10,20] .. [[SCOPE]]    foo

重要な例外は、コンストラクター関数です。このタイプの関数の場合、[[SCOPE]]は常にグローバルオブジェクトを指します。

また、ScopeChainチェーン内のリンクの1つにプロトタイプがある場合、プロトタイプ内でも検索が実行されることを忘れないでください。

結論


主要なアイデアを理論的に発表します。

  • これは、エンティティと実行コンテキストの関係です
  • ScopeChainは、エンティティとすべての発生コンテキストとの関係です
  • thisとScopeChainは実行コンテキストのプロパティです
  • この関数は呼び出し元によって決定され、呼び出しの構文に依存します
  • ScopeChainは、現在のコンテキスト+ [[Scope]]の字句環境です。
  • [[Scope]]-これは関数自体のプロパティであり、コンテキストを生成する字句環境の階層チェーンを含みます
