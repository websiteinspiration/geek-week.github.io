優れたプログラマーがコピーし、優れたプログラマーが盗む

コードをコピーするだけでは危険です。ただし、ソフトウェア開発における他の多くの側面については、それらが不注意に扱われるという条件で言えます。この投稿では、次のような問題について説明します。

  • ソフトウェア開発におけるコードのコピーとはどういう意味ですか？
  • 正しいコードの盗難とはどういう意味ですか？
  • 不適切なコピーの落とし穴は何ですか？

ここでスタックオーバーフローに関する質問への回答として公開されているサンプルコードがプログラムの最終バージョンになることがよくあることは、プログラマにとって秘密ではありません。たぶんあなたが質問をすると、彼らはあなたの答えに理想的なサイクル図を送ってくれましたfor。多分あなたasync awaitはあなたのアプリケーションに適したCコードの断片があったという素晴らしい答えを見つけました。

コードを1回記述して、100万回動作させる


Stack Overflowからのコードのコピーは、盗作の一種です。つまり、1つ以上のプロジェクトからのコードの複製とその再利用です。あなたが尋ねるプログラマーを見ると、彼の仕事では5から10パーセント、さらには7から23パーセントのコードが他のソースからコピーされていることがわかるかもしれません。借用が正しく行われたかどうかは、すでに議論の余地がある問題です。

個々のプロジェクトの実際の借入率に関係なく、コードの重複は非常に一般的であると言えます。ボイラープレートコードは、プロジェクト全体で定期的に繰り返されるコードです。プログラマーが自分で毎回タイプしないことがよくあります。ロンボクのような図書館 ボイラープレートを書く必要性を減らすことを可能にしますが、事実は残っています：

  • コードの一部は常に存在し、プロジェクト全体で何度も表示される必要があります。

そして

  • これらのパーツには小さな変更が必要になる可能性が高いため、これらのコードフラグメントを個別の関数または依存関係に削除することはできません。

依存関係について言えば、ライブラリと外部の依存関係について言及できます。これらは、コードの機能をコピーせずに再利用する効果的な方法です。コードのコピーとほとんど同じように機能しますが、唯一の違いは、この場合、コードのコピーされた部分の操作を維持する責任がないことです。地獄、今、ほとんどのインターネットは、開発を簡素化するさまざまなプラットフォームとプラグインライブラリで実行されています。ライブラリでコードを再利用することは、各ライブラリが1つのことを実行し、それでもうまくいくため、非常に効果的です。そして、科学コミュニティで作品を書くときにすべてのソースを示す必要がある方法とは異なり、これらのライブラリの多くは、示すためにあなたの側で何も必要としません他人のコードを使用してプログラムすること、または他人のコードをベースとして使用すること。

npm JavaScriptパッケージマネージャーは、これを極限まで引き出します。コマンドラインを使用して、小さな単一機能のライブラリー（コード行のサイズもある）をプロジェクトにインストールできます。 100万以上の無料のオープンソースパッケージを入手して、アプリケーションに機能を組み込むことができます。

もちろん、他のアプローチと同様に、このメソッドにもマイナスがあります。パッケージをインストールすると、コードに対する制御が失われます。一部の攻撃者は本当に便利なパッケージを作成し、まともなダウンロードレベルになるまで待ってから、ビットコインウォレットを盗むようにコードを微調整しました。私たちはnpmスタッフがそのような攻撃を迅速に撃退することができたので、彼らに敬意を表する必要があります。ただし、外部の依存関係が増えるほど、想像できる攻撃領域が増えます。

Stack Overflowの回答でさえ、コードの重複から保護されていません。独立した研究者が、コードが他のソースからコピーされたスタックオーバーフローに対するいくつかの応答を発見しました。1つのJavaコードが40以上の応答で見つかりました。

良いアーティストはコピーし、素晴らしいアーティストは盗む


アートの世界では、盗難も素晴らしい作品の作成の一部です。アムステルダムのゴッホ美術館に行って、彼の初期の作品をそこで見たのを覚えています。私は衝撃を受けた。しかし、実際には、それらの絵画が日本画や木版画（木版画）に触発されたものであることを知りませんでした。彼の時代に特有のスタイルだと思っていたのは、実際には開発の歴史の一部でしたが、私が考えていたスタイルではありませんでした。デンマークのアーティストがすでに取った開発の道筋をたどるのではなく、ゴッホはパリで見つけた日本の版画のアイデアを使用し（彼は建築構成とブラシストロークのスタイルのアイデアを使用しました）、それらを作品に組み合わせました。

ピカソの格言は、「優れたアーティストはコピーし、優れたアーティストは盗む」というものです。ピカソ自身がアフリカやポリネシアの芸術から多くのアイデアを引き出し、スケッチと組み合わせました。盗難自体の考えは間違っています。確かに、他人の作品を彼らのものと呼ぶことは盗作です。借りた物はまだ他人のものです。あなたはスタイルをコピーし、それはまだ所有者に属しています。ただし、盗むには、このアイデアを自分のものにする必要があります。誰かのアイデアの作成者を自分で宣言するには、借用することです。しかし、ピカソは、アイデアを「盗む」ためには、そのアイデアを理解して彼の作品に織り込む必要があると考えていました。スティーブジョブズはこの引用を愛し、アップルは彼のリーダーシップの下で成功しました。彼らがアイデアを盗み、組み合わせ、そして改善したからです。

コードをコピーすると、単にコードを借用するリスクが生じます。借用されたコードは、十分に埋め込まれていてエラーが発生しない場合は完全にプロジェクトに配置されますが、バグが含まれていたり、サイバー犯罪者によって作成されたりする可能性があり、その目的でそれらを使用する可能性があります。誤ってコピーされた（または修正されてコピーされた）コードに関連するリスクの数は膨大です。実際、重複コードの苦情は盗作者を追跡することができます。コードを理解していないと、本来は単に概念のデモであると想定されていたものの保護に穴が残る可能性があります。Stack Overflowで最も多くコピーされたコードでさえ、バグを奪われていません。

一方、コードを盗む場合は、コードの機能を確実に理解しています。そのコアとその機能は、独自のコードに適合します。メモリから再度書き込むことができる場合、これはコードが正しく盗まれたことを示しています。つまり、コードの手直しにより、単なるコピー以上の効果が得られます。彼女はあなたに何か新しくてオリジナルを与えました。
はい、コードを盗みます。それを理解し、プロジェクトに適用してください。それをあなたのものにします。あなたは経験を積むことができ、プロジェクトを改善することができます、そしておそらくあなたの履歴書（あなたのctrl + C ctrl + V）さえも。しかし、新しく見つかったコードとその機能を完全に理解せずにコピーすると、独自のコードを悪化させるリスクがあります。
