Flywayを使用したデータベースの移行

この記事の翻訳は、Java開発者コースの開始前に準備されました。


1.はじめに


この記事では、Flywayの主要な概念と、このフレームワークを使用して、フライウェイMavenプラグインを使用する例として、インメモリH2データベースを使用してデータベーススキーマを継続的に変更する例について説明します。

Flywayは、移行を使用してデータベースのバージョンを更新します。移行は、SQL（特定のDBMSに固有の構文を使用）またはJavaで記述できます。

移行は、バージョン付けすることも、定期的に行うこともできます。前者には固有のバージョンがあり、1回だけ適用されます。 2番目のバージョンにはバージョン番号がなく、チェックサムが変更されたときに適用されます。

1回の実行で繰り返される移行は、バージョン対応の移行が完了した後に常に適用されます。繰り返される移行は、記述されている順序で適用されます。1回の移行では、すべての操作が単一のデータベーストランザクション内で実行されます。

この記事では、データベースの移行にMavenプラグインを使用することに焦点を当てます。

2. Flyway Mavenプラグイン


flyway mavenプラグインをに追加pom.xml：

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
</plugin>

プラグインの現在のバージョンは、Maven Centralで表示できます。
プラグインパラメータのリストはドキュメントにあります。プラグインパラメータは、4つの異なる方法で設定できます。

2.1。<configuration>プラグインセクション
パラメータは、のプラグインセクションのタグで直接指定できますpom.xml。

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
    <configuration>
        <user>databaseUser</user>
        <password>databasePassword</password>
        <schemas>
            <schema>schemaName</schema>
        </schemas>
        ...
    </configuration>
</plugin>

2.2。Mavenプロパティ


でパラメータを指定してプラグインを設定することもできます<properties>：

<project>
    ...
    <properties>
        <flyway.user>databaseUser</flyway.user>
        <flyway.password>databasePassword</flyway.password>
        <flyway.schemas>schemaName</flyway.schemas>
        ...
    </properties>
    ...
</project>

2.3。外部設定ファイル


または、別の.propertiesファイルに構成を記述します。

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=schemaName
...

デフォルト名は構成ファイルですflyway.properties。このファイルは、ファイルと同じディレクトリにある必要がありますpom.xml。エンコードはパラメーターで設定されますflyway.encoding（デフォルトはUTF-8）。

ファイルに別の名前（たとえばcustomConfig.properties）を使用する場合は、mavenを呼び出すときにそれを明示的に指定する必要があります。

$ mvn -Dflyway.configFile=customConfig.properties

2.4。システムプロパティ


最後に、コマンドラインからmavenを呼び出すときに、パラメーターをシステムプロパティとして指定できます。

$ mvn -Dflyway.user=databaseUser -Dflyway.password=databasePassword
  -Dflyway.schemas=schemaName

構成がいくつかの方法で指定されている場合、優先順位は次のようになります。

  1. システムプロパティ
  2. 外部設定ファイル
  3. セクションの<プロパティ>
  4. プラグイン<構成セクション>

3.移行の例


このセクションでは、Mavenプラグインを使用した例として、インメモリH2データベースを使用してデータベーススキーマを移行するために必要な手順を検討します。Flywayを構成するには、外部ファイルを使用します。

3.1。POMの変更


まず、H2への依存関係を追加します。

<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>1.4.196</version>
</dependency>

ここでは、Maven Centralで利用可能な最新のドライバーのバージョンも確認できます。前述のように、Flywayのプラグインを追加します。

3.2。外部ファイルでFlywayを構成する


次の内容の$PROJECT_ROOTファイルmyFlywayConfig.propertiesを 作成します。

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=app-db
flyway.url=jdbc:h2:mem:DATABASE
flyway.locations=filesystem:db/migration

上記の設定は、移行スクリプトがディレクトリdb/migrationにあり、H2 databaseUserとを使用してデータベースに接続することを示していますdatabasePassword。

アプリケーションのデータベーススキーマapp-db。

もちろん、パラメータにflyway.user、flyway.passwordそしてflyway.urlあなたは、データベースの名前、パスワードおよびURLを指定する必要があります。

3.3。最初の移行


Flywayの規則では、移行スクリプトの名前は次の形式にする必要があります。

<Prefix<code>><</code>Version>__<code><</code>Description>.sql

どこ：

  • <接頭辞>-接頭辞。バージョン管理された移行の場合、デフォルトは「V」です。接頭辞はflyway.sqlMigrationPrefixプロパティを通じて構成されます。
  • <バージョン>-移行バージョン番号。メジャーバージョンとマイナーバージョンは、下線で区切ることができます。バージョンは常に1から始める必要があります。
  • <Description>-移行のテキストによる説明。説明は、バージョン番号と2つのアンダースコアで区切る必要があります。

例：V1_1_0__my_first_migration.sql

だから、ディレクトリに作成してみましょうdb/migrationでの$PROJECT_ROOTスクリプトの移行V1_0__create_employee_schema.sqlテーブルを作成すると、SQLをemployee：

CREATE TABLE IF NOT EXISTS `employee` (
 
    `id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name` varchar(20),
    `email` varchar(50),
    `date_of_birth` timestamp
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;

3.4。マイグレーションを実施


次に、$PROJECT_ROOT次のmavenコマンドを実行して、データベースの移行を適用します。

$ mvn clean flyway:migrate -Dflyway.configFile=myFlywayConfig.properties

最初の移行を完了する必要があります。

これで、データベーススキーマは次のようになります
。

+----+------+-------+---------------+
| id | name | email | date_of_birth |
+----+------+-------+---------------+

前の手順を繰り返して、他の移行を実行できます。

3.5。2回目の移行


2回目の移行でV2_0_create_department_schema.sqlは、次の2つのリクエストを含む名前のファイルを作成します。

CREATE TABLE IF NOT EXISTS `department` (
 
`id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
`name` varchar(20)
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;
 
ALTER TABLE `employee` ADD `dept_id` int AFTER `email`;

最初の移行と同じように、移行を実行してみましょう。

これで、データベースのスキーマが変更されました。employee新しい列が追加され、新しいテーブルが作成されましたdepartment。

employee:
+----+------+-------+---------+---------------+
| id | name | email | dept_id | date_of_birth |
+----+------+-------+---------+---------------+

department:

+----+------+
| id | name |
+----+------+

両方の移行が成功したことを確認するには、次のmavenコマンドを実行します。

$ mvn flyway:info -Dflyway.configFile=myFlywayConfig.properties

4. Spring BootでFlywayを無効にする


Flywayの移行を無効にする必要がある場合があります。

これは、データベーススキーマがエンティティに基づいて生成されるテスト中に必要になる場合があります。この場合、テストプロファイルのFlywayを無効にすることができます。

Spring Bootでは、これは非常に簡単です。

4.1。スプリングブート1.x


flyway.enabledファイルに プロパティを設定するだけですapplication-test.properties。

flyway.enabled=false


4.2。スプリングブート2.x


Spring Bootの以降のバージョンでは、このプロパティは 

spring.flyway.enabled:
spring.flyway.enabled=false

4.3空のFlywayMigrationStrategy


起動時にFlywayの自動移行のみを無効にしたいが、手動で移行を開始したい場合、上記のプロパティを使用しても機能しません。

これは、Spring BootがFlyway binを自動的に構成しないため、それらを自分で構成する必要があるため、あまり便利ではありません。

この場合、Flywayを有効のままにして、空のFlywayMigrationStrategyを実装できます。

@Configuration
public class EmptyMigrationStrategyConfig {
 
    @Bean
    public FlywayMigrationStrategy flywayMigrationStrategy() {
        return flyway -> {
            // do nothing 
        };
    }
}

実際、これにより、アプリケーションの起動時にFlywayの移行が無効になります。

ただし、引き続き手動で移行を開始できます。

@RunWith(SpringRunner.class)
@SpringBootTest
public class ManualFlywayMigrationIntegrationTest {
 
    @Autowired
    private Flyway flyway;
 
    @Test
    public void skipAutomaticAndTriggerManualFlywayMigration() {
        flyway.migrate();
    }
}

5. Flywayの仕組み


いつ、誰が、どのマイグレーションが適用されたかを追跡するために、メタデータを含む特別なテーブルがデータベーススキーマに追加されます。このテーブルには、移行のチェックサムと、移行が成功したかどうかに関する情報も格納されます。

フレームワークは次のように機能します。

  1. データベーススキーマでメタデータテーブルをチェックします（デフォルトSCHEMA_VERSION）。メタデータテーブルが存在しない場合は作成します。
  2. クラスパスをスキャンして、使用可能なマイグレーションを探します。
  3. 移行をメタデータテーブルと比較します。バージョン番号が最新としてマークされたバージョン以下である場合、それは無視されます。
  4. 残りのすべての移行を保留中としてマークします。次に、それらをバージョン番号の昇順にソートし、指定された順序で実行します。
  5. 移行が適用されると、メタデータテーブルを更新します。

6.チーム


Flywayには、次の主要な移行管理チームがあります。

  • 情報。データベーススキーマの現在のステータス/バージョンを表示します。予想される移行、適用された移行、完了した移行のステータス、およびそれらの完了日に関する情報。
  • 移行します。データベーススキーマを現在のバージョンに更新する。クラスパスをスキャンして利用可能な移行を見つけ、保留中の移行を適用します。
  • ベースライン。データベーススキーマバージョンをインストールし、baselineVersionまでの移行を無視します。ベースラインは、既存のデータベースでFlywayを使用するのに役立ちます。新しい移行は通常どおり適用されます。
  • 検証。現在のデータベーススキーマを利用可能な移行と照合します。
  • 修理。メタデータテーブルを復元します。
