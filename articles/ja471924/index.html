Sassモジュールの紹介

こんにちは、ハブル！Miriam Suzanneによる記事「Introducing Sass Modules」の翻訳を紹介します。

最近、他の言語でおなじみの機能であるモジュラーシステムがSassに登場しました。これは、@importSassで最も一般的に使用される機能の1つにとって大きな前進です。既存のディレクティブを@import使用すると、サードパーティのパッケージを接続し、スタイルをサポートされている要素に分割できますが、いくつかの制限があります：

  • @import CSSもあり、それらの動作の違いは混乱を招く可能性があります。
  • その場合は@import、一つのファイルに対して数回、これは、コンパイルを遅くする原因の再定義の競合、そしてあなたは、出力の重複コードを取得することができます。
  • サードパーティのパッケージを含め、すべてがグローバルスコープにあります。これは、私の関数colorが既存の関数をオーバーライドする方法、colorまたはその逆の方法です。
  • たとえば、関数を使用する場合、関数colorが定義されている場所を正確に知ることはできません。どちらが@import接続しましたか？

Sassパッケージの作者（私のような）は、変数と関数の接頭辞を手動で設定することで名前空間の問題を回避しようとしましたが、Sassモジュールははるかに強力なソリューションです。つまり、@importより明示的なルール@useとに置き換えられます@forward。今後数年間@import、Sassは非推奨と見なされ、削除されます。あなたはまだ使用できるCSS Importのが、彼らはサスでコンパイルされることはありません。でも心配はいりません。アップグレードに役立つ移行ツールがあります。

を使用してファイルをインポートする @use


@use 'buttons';

新しいもの@useはに似て@importいますが、いくつかの顕著な違いがあります。

  • @useプロジェクトで何回使用しても、ファイルは1回インポートされます。
  • アンダースコア（_）またはハイフン（-）で始まる変数、ミックスイン、および関数（Sassでは「メンバー」と呼ばれます）はプライベートと見なされ、インポートされません。
  • 接続されている@useファイルのメンバー（この場合buttons.scss）はローカルでのみアクセス可能で、後続のインポートには転送されません。
  • 同様に、@extendsチェーン全体にのみ適用されます。つまり、拡張子はインポートされるスタイルにのみ適用され、インポートされるスタイルには適用されません。
  • インポートされたすべてのメンバーには、デフォルトで独自の名前空間があります。

を介してファイルを接続すると@use、Sassはファイル名に基づいて名前空間を自動的に生成します。

@use 'buttons'; /*    `buttons`*/
@use 'forms'; /*    `forms`*/

これで、ファイルbuttons.scssとファイルの両方のメンバーにforms.scssアクセスできますが、このアクセスはインポート間で転送されforms.scssません。それでも、で定義された変数にはアクセスできませんbuttons.scss。インポートされたエンティティには名前空間があるため、それらにアクセスするには新しいドット区切りの構文を使用する必要があります。

/* : <namespace>.$variable */
$btn-color: buttons.$color;
$form-border: forms.$input-border;

/* : <namespace>.function() */
$btn-background: buttons.background();
$form-border: forms.border();

/* : @include <namespace>.mixin() */
@include buttons.submit();
@include forms.input();

importに追加することで、デフォルトの名前空間を変更または削除できますas <name>。

@use 'buttons' as *; /*      */
@use 'forms' as 'f';

$btn-color: $color; /* buttons.$color    */
$form-border: f.$input-border; /* forms.$input-border    */

を使用as *すると、モジュールがルート名前空間に追加されるため、プレフィックスは不要ですが、そのメンバーは現在のドキュメントによってローカルに制限されます。

Sass組み込みモジュールのインポート


Sassの内部機能もモジュラーシステムに移行しているため、グローバルネームスペースを完全に制御できます。-いくつかの内蔵モジュールがありmath、color、string、list、map、selectorとmeta-明示的に使用する前に、ファイルにインポートします。

@use 'sass:math';

$half: math.percentage(1/2);

組み込みモジュールをグローバルスペースにインポートすることもできます。

@use 'sass:math' as *;

$half: percentage(1/2);

map-getor などのプレフィックス名がすでに含まれている組み込み関数は、str-indexこのプレフィックスを複製せずに使用できます。

@use 'sass:map';
@use 'sass:string';

$map-get: map.get(('key': 'value'), 'key');
$str-index: string.index('string', 'i');

組み込みモジュール、関数、および名前の変更の完全なリストは、Sassモジュール仕様にあります。

新機能と変更されたコア機能


追加の利点として、これはSassが名前の競合を引き起こさずに新しい内部ミックスインと関数を安全に追加できることを意味します。最も驚くべき例はload-css、モジュールのミックスインですsass:meta。これはと同様に機能し@useますが、生成されたCSSのみを返し、コード内のどこでも動的に機能します。

@use 'sass:meta';
$theme-name: 'dark';

[data-theme='#{$theme-name}'] {
  @include meta.load-css($theme-name);
}

最初の引数はモジュールのURLです（など@use）が、変数を使用して動的に変更できますtheme-#{$name}。たとえば、補間を使用することもできます。2番目の（オプションの）引数は、map構成構造を取ります。

/*   $base-color  'theme/dark'   */
@include meta.load-css(
  'theme/dark', 
  $with: ('base-color': rebeccapurple)
);

引数を$with使用すると、構造体を使用mapして、読み込まれたモジュール内の任意の変数を構成できますが、この変数は次の条件を満たす必要があります。

  • _またはで始まるプライベート変数ではありません-
  • ディレクティブでマーク!default

/* theme/_dark.scss */
$base-color: black !default; /*    */
$_private: true !default; /*        */
$config: false; /*    ,      !default */

キー'base-color'が変数を設定することに注意してください$base-color。

モジュールにはいくつかの新機能がありますsass:meta：module-variables()およびmodule-functions()。それらのそれぞれはmap、すでにインポートされたモジュールから名前と値の構造を返します。それらは、モジュール名前空間に対応する1つの引数を取ります。

@use 'forms';

$form-vars: module-variables('forms');
/*
 (
   button-color: blue,
   input-border: thin,
 )
*/

$form-functions: module-functions('forms');
/*
 (
   background: get-function('background'),
   border: get-function('border'),
 )
*/

他のいくつかの機能sass:meta- 、、global-variable-exists() および-追加の引数受け取り、私たちは、明示的に各名前空間をチェックすることができます。function-exists()mixin-exists()get-function()$module

色の調整とスケーリング


このモジュールsass:colorには、私たちの古い問題のいくつかを解決することについての興味深い予約もあります。これらのレガシー関数の多くは、明示的な関数と同様に使用することが推奨されるlighten()かadjust-hue()、推奨されなくなりました。color.adjust()color.scale()

/*  lighten(red, 20%) */
$light-red: color.adjust(red, $lightness: 20%);

/*  adjust-hue(red, 180deg) */
$complement: color.adjust(red, $hue: 180deg);


これらのレガシー機能の一部（などadjust-hue）は冗長で不要です。その他-のようなlighten、darken、saturate、など -内部ロジックを改善するには、再実装が必要です。元の関数はadjust()、線形演算を使用するに基づいていました。上記の例で20%は、現在の明度redに色を追加しています。ほとんどの場合、scale()現在の値に対して特定の割合で色を変更（）します。

/*        20,   0.2,     */
$light-red: color.scale(red, $lightness: 20%);

これらの機能は、完全に非推奨にされて削除された後、最終的sass:colorにcolor.scale()はではなく、に基づく新しい動作で再表示されcolor.adjust()ます。これは、突然の下位互換性の問題を回避するために徐々に行われます。それまでの間は、コードを手動でチェックして、どこにcolor.scale()役立つかを確認することをお勧めします。

インポートされたライブラリを構成する


多くの場合、サードパーティまたは再利用可能なライブラリには、オーバーライドできるいくつかのデフォルト値を持つ変数が付属しています。インポートする前に変数でこれを行いました：

/* _buttons.scss */
$color: blue !default;

/* old.scss */
$color: red;
@import 'buttons';

モジュールを使用すると、ローカル変数にアクセスできなくなるため、値を設定する新しい方法が必要です。これを行うには、設定をmapに渡します@use。

@use 'buttons' with (
  $color: red,
  $style: 'flat',
);

これはの引数$withに似ていますがload-css()、変数名をキーとして使用する代わりに、変数自体を記号とともに使用します$。

設定がいかに明示的であるかは気に入っていますが、何度か困惑しているルールが1つあります。モジュールは、初めて使用するときに1回しか構成できません。接続の順序は、Sassにとって常に重要@importでしたが、これらの問題は気付かれることはありませんでした。これで明確なエラーが発生しました。これは良いことであり、少し予想外のことです。を介してライブラリを接続@useし、それらをファイルエントリポイント（他のすべてのファイルをインポートする中央のドキュメント）で構成して、これらの設定がを介した他のライブラリ接続の前にコンパイルされるようにします@use。

（現時点では）構成を「リンク」して編集可能に保つことはできませんが、構成したモジュールをラップして新しいモジュールとして転送することはできます。

を使用したファイル転送 @forward


必ずしもファイルを使用してそのメンバーを参照する必要はありません。時々、それを後続のインポートに渡したいだけです。フォームに関連付けられた複数のファイルがあり、それらをすべて1つの名前空間として接続したいとします。これは@forward次の方法で行うことができます。

/* forms/_index.scss */
@forward 'input';
@forward 'textarea';
@forward 'select';
@forward 'buttons';

そのような転送されたファイルのメンバーは現在のドキュメントでは使用できず、名前空間は作成されませんが、これらの変数、関数、およびミックスインは、別のファイルがそれらを接続する@useか、コレクション全体をを介して転送するときに使用できます@forward。送信された個々のファイルに実際のCSSが含まれている場合、パッケージ自体が使用されるまで、直接生成せずに送信されます。この段階では、これらすべてが1つの名前空間を持つ1つのモジュールと見なされます。

/* styles.scss */
@use 'forms'; /*        `forms` */

ノートの場合：フォルダを接続するためにサスを依頼した場合、それはそれでファイルを検索しますindexか_index。

デフォルトでは、すべてのパブリックメンバーがモジュールとともに転送されます。ただし、追加または除外したい特定のメンバーの条件showとhide表示を利用して、より選択的にすることができます。

/*    `border()`   `$border-color`   `input` */
@forward 'input' show border, $border-color;

/*     `buttons`    `gradient()` */
@forward 'buttons' hide gradient;

注：関数とミックスインに共通の名前がある場合、それらも一緒に追加および非表示になります。

ソースを明確にするか、転送されたモジュールの名前の競合を回避するために、次を使用して接続ファイルのメンバーにプレフィックスを追加できますas。

/* forms/_index.scss */
/* @forward "<url>" as <prefix>-*; */
/* ,      `background()` */
@forward 'input' as input-*;
@forward 'buttons' as btn-*;

/* style.scss */
@use 'forms';
@include forms.input-background();
@include forms.btn-background();

また、必要に応じて、常に同じモジュール@useを使用して転送し@forward、両方のルールを追加できます。

@forward 'forms';
@use 'forms';

これは、他のファイルに転送する前にライブラリを事前設定したり、追加のツールを追加したりする場合に特に便利です。これにより、接続パスを簡略化できます。

/* _tools.scss */
/*        */
@use 'accoutrement/sass/tools' with (
  $font-path: '../fonts/',
);
/*    */
@forward 'accoutrement/sass/tools';

/* -  ... */


/* _anywhere-else.scss */
/*      */
@use 'tools';

また@use、@forwardドキュメントのルート（ネストされていない）とファイルの先頭で宣言する必要があります。@charsetインポートディレクティブの前に表示できるのは、単純な変数定義のみです。

モジュラーシステムへの移行


新しい構文をテストするために、新しいSassオープンソースライブラリ（Cascading Color Systems）とグループ用の新しいサイトを作成しました-どちらもまだ開発中です。ライブラリの作成者の視点とサイト開発者の視点からモジュールを理解する必要がありました。モジュールの構文を使用してサイトスタイルを作成する「エンドユーザー」の経験から始めましょう...

サポートと書き込みスタイル


サイトでモジュールを使用することは楽しかったです。新しい構文は、すでに使用しているコードアーキテクチャをサポートしています。グローバル設定とツールのすべてのインポートは、config必要なすべてのものを転送するインデックスファイルと同じディレクトリ（私はと呼びます）にあります。

/* config/_index.scss */
@forward 'tools';
@forward 'fonts';
@forward 'scale';
@forward 'colors';

サイトの他の部分を開発することにより、これらのツールと構成を必要な場所にインポートできます。

/* layout/_banner.scss */
@use '../config';

.page-title {
  @include config.font-family('header');
}

古い構文を使用しているAccoutrementやHerman などの既存のライブラリと連携して動作します@import。ルール@importがすべての場所で一度に置き換えられることはないため、Sassの開発者は移行のために時間を割いています。モジュールは現在利用可能ですが、@import1〜2年で期限切れにならず、その1年後に言語から削除されます。同時に、2つのシステムは任意の方法で連携します。

  • @import新しい構文を含むファイルに対して実行すると、@use/@forwardパブリックメンバーのみが名前空間なしでインポートされます。
  • 古い構文を含むファイルに対して@useいずれかを実行する@forwardと、@importネストされたすべてのインポートに単一の名前空間としてアクセスできます。

これは、お気に入りのライブラリの新しいバージョンのリリースを待たずに、新しいモジュール構文の使用をすぐに開始できることを意味します。すべてのライブラリを更新するのに少し時間をかけることができます！

移行ツール


Jennifer Thakarが作成した移行ツールを使用すれば、アップグレードに時間がかかりません。NPM、ChocolateyまたはHomebrewを使用してインストールできます。

npm install -g sass-migrator
choco install sass-migrator
brew install sass/sass/migrator

これは、モジュールに移行するための1回限りのツールではありません。 Sassがアクティブな開発に戻ったので（以下を参照）、移行ツールは定期的に更新を受け取り、各新機能の移植に役立ちます。このツールをグローバルにインストールし、将来使用するために保存することをお勧めします。

migratorはコマンドラインから起動でき、できればCodeKitやScoutなどのサードパーティアプリケーションに追加される予定です。たとえば、1つのSassファイルを彼にポイントし、style.scssどのマイグレーションを適用するかを彼に伝えます。現時点では、移行と呼ばれるのは1つだけmoduleです。

# sass-migrator <migration> <entrypoint.scss...>
sass-migrator module style.scss

デフォルトでは、移行で一つだけのファイルを更新しますが、ほとんどの場合、私たちは、メインファイルとそのすべての依存関係を更新する：介して接続された任意の要素@import、@forwardまたは@use。これを行うには、各ファイルを個別に指定するか、単にフラグを追加します--migrate-deps。

sass-migrator --migrate-deps module style.scss

試用の場合は、ソースファイルを変更せずに、追加--dry-run --verbose（または省略形-nv）して結果を確認できます。移行の設定に使用できるオプションは他にもたくさんあります。ライブラリの作成者が手動で作成した古い名前空間を削除できるように特別に設計されたオプションも1つありますが、ここではそれらすべてについては説明しません。移行ツールは、Sass Webサイトに完全に文書化されています。
