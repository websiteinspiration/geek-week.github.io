Sysbenchと確率変数の分布

記事の翻訳は、特に「データベース」コースの学生のために準備されました。



sysbenchの乱数生成について知らなかった可能性

があるSysbenchは、人気のあるパフォーマンステストツールです。もともとは2000年代初頭にPeter Zaitsevによって書かれ、テストとベンチマークの事実上の標準になりました。現在、アレクセイコピトフでサポートされており、Githubのに投稿されています。

しかし、その幅広い分布にもかかわらず、sysbenchの多くには不慣れな瞬間があることに気づきました。たとえば、Luaを使用してMySQLテストを簡単に変更したり、組み込みの乱数ジェネレーターのパラメーターを構成したりする機能。

この記事については何ですか？


この記事を書いて、sysbenchを要件に合わせて簡単にカスタマイズできることを示しました。sysbenchの機能を拡張するには多くの方法があり、そのうちの1つはランダム識別子（ID）の生成を構成することです。

デフォルトでは、sysbenchには、乱数を生成するための5つの異なるオプションが用意されています。しかし、ほとんどの場合（実際にはほとんどありません）、明示的に示されているものはなく、生成パラメーターを確認できる頻度は少なくなっています（使用可能なオプションの場合）。

質問がある場合：「なぜこれに興味があるのですか？結局のところ、デフォルト値は非常に適しています。」この記事は、なぜこれが常に当てはまるとは限らないかを理解するのに役立つように設計されています。

はじめましょう


sysbenchで乱数を生成する方法は何ですか？以下は現在実装されています（--helpオプションで簡単に確認できます）：

  • スペシャル（特別配布）
  • ガウス（ガウス分布）
  • パレート（パレート分布）
  • Zipfian（Zipf分布）
  • 均一（均一分布）

デフォルトでは、Specialは次のパラメーターで使用されます。

  • rand-spec-iter = 12 -特別な分布の反復回数
  • rand-spec-pct = 1 -「特別な」値が特別な分布で分類される範囲全体のパーセンテージ
  • rand-spec-res = 75 -特別なディストリビューションで使用する「特別な」値の割合

私はシンプルで再現しやすいテストとスクリプトが好きなので、後続のすべてのデータは次のsysbenchコマンドを使用して収集されます。

  • sysbench ./src/lua/oltp_read.lua -mysql_storage_engine = innodb –db-driver = mysql –tables = 10 –table_size = 100 prepare
  • sysbench ./src/lua/oltp_read_write.lua –db-driver=mysql –tables=10 –table_size=100 –skip_trx=off –report-interval=1 –mysql-ignore-errors=all –mysql_storage_engine=innodb –auto_inc=on –histogram –stats_format=csv –db-ps-mode=disable –threads=10 –time=60 –rand-type=XXX run

自由に実験してみてください。スクリプトの説明とデータはここにあります。

sysbenchが乱数ジェネレータを使用するのはなぜですか？目的の1つは、クエリで使用されるIDを生成することです。したがって、この例では、それぞれ100行の10個のテーブルの作成を考慮して、1〜100の数値が生成されます。

上記のようにsysbenchを実行し、-rand-typeのみを変更するとどうなりますか？

それぞれの分布を詳しく見てみましょう。

特別


デフォルトではスペシャルが使用されるため、rand-typeを指定しない場合、sysbenchはスペシャルを使用します。Specialは非常に限られた数のID値を使用します。この例では、50-51の値が主に使用されており、残りの44-56の値は非常にまれですが、他の値は実際には使用されていません。選択した値は、1から100の使用可能な範囲の中央にあることに注意してください。

この場合、ピークはサンプルの2％を表す約2つのIDです。レコード数を100万個に増やすと、ピークは残りますが、サンプルの0.74％である7493になります。これはより制限的なものになるため、ページ数は複数になる可能性があります。

均一（均一分布）


名前が示すように、均一を使用する場合、IDにはすべての値が使用され、分布は...均一になります。

Zipfian（Zipf分布）


Zip分布は、ゼータ分布とも呼ばれ、言語学、保険、およびレアイベントモデリングで一般的に使用される離散分布です。この場合、sysbenchは最小（1）から始まる数値を使用し、非常に迅速に使用頻度を減らし、大きい数値に移動します。

パレート（パレート）


パレートは「80-20」ルールを適用します。この場合、生成されたIDは不鮮明にならず、小さなセグメントに集中します。この例では、すべてのIDの52％の値が1で、値の73％が最初の10個の数値に含まれていました。

ガウス（ガウス分布）


ガウス分布（正規分布）はよく知られており、よく知られています。これは主に、統計と中心的要因の予測に使用されます。この場合、使用されるIDは、平均値から始まり、徐々にエッジに向かって減少する、ベル型の曲線に沿って分散されます。

これの意味は何ですか？


上記の各オプションには独自の用途があり、目的別にグループ化できます。ホットスポットに焦点を当てたパレートおよびスペシャル。この場合、アプリケーションは同じページ/データを何度も使用します。これは私たちが必要とすることかもしれませんが、私たちは自分たちが何をしているのかを理解しなければならず、ここで間違いを犯してはなりません。

たとえば、読み取り中にInnoDBページ圧縮のパフォーマンスをテストする場合、デフォルト値のSpecialまたはParetoを使用しないでください。 1 TBのデータセットと30 GBのバッファープールがあり、同じページを何度も要求した場合、このページは既にディスクから読み込まれ、メモリに圧縮されていない状態で使用できます。

つまり、このようなテストは時間と労力の無駄です。

録音のパフォーマンスを確認する必要がある場合も同じです。同じページを何度も書くことは最善の方法ではありません。

パフォーマンステストはどうですか？

繰り返しますが、パフォーマンスをテストしたいと思いますが、どのような場合ですか？乱数の生成方法はテスト結果に大きく影響することを理解することが重要です。そして、あなたの「十分なデフォルト」は誤った結論につながる可能性があります。

調査結果


この時点で、sysbenchでの乱数生成の設定がどれほど簡単か、そしてこれがどれほど便利かを理解しているはずです。上記は、たとえば、次を使用する場合のすべての呼び出しに適用されることに注意してくださいsysbench.rand.default。

local function get_id()
   return sysbench.rand.default(1, sysbench.opt.table_size)
End

これを踏まえて、他の人の記事からコードを不注意にコピーしないでください。必要なものと、これを達成する方法を考えて掘り下げてください。
