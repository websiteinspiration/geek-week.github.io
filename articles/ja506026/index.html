廃止されたルート証明書の問題。Next in line Let's EncryptとスマートTV



ブラウザがWebサイトを認証するために、ブラウザは有効な証明書チェーンとして表示されます。上記の典型的なチェーンは、複数の中間証明書を持つことができます。有効なチェーン内の証明書の最小数は3です。

ルート証明書は認証局の中心です。それは文字通りあなたのOSまたはブラウザに埋め込まれ、それはあなたのデバイス上に物理的に存在します。サーバー側から変更することはできません。デバイスのOSまたはファームウェアを強制的に更新する必要があります。

セキュリティの専門家であるスコットヘルメ氏は、Let's Encrypt認証局で主な問題が発生すると書いています。これは、インターネットで最も人気のあるCAであり、ルート証明書がすぐに機能しなくなるためです。ルートを変更してみましょう2020年7月8日予定。

証明機関（CA）の最終証明書と中間証明書はサーバーからクライアントに配信され、クライアントには既にルート証明書があるため、この証明書コレクションを使用してチェーンを構築し、Webサイトを認証できます。

問題は、各証明書に有効期間があり、その後に置き換える必要があることです。たとえば、2020年9月1日から、SafariはサーバーのTLS証明書の有効期間に最大398日間の制限を導入する予定です。

これは、私たち全員が少なくとも12か月ごとにサーバー証明書を置き換える必要があることを意味します。この制限はサーバー証明書にのみ適用され、ルートCA証明書には適用されません。

CA証明書はさまざまなルールのセットによって管理されているため、有効期限が異なります。有効期間が5年の中間証明書と、耐用年数が25年のルート証明書は非常に一般的です。

中間証明書はサーバーによってクライアントに配信されるため、通常は問題はありません。サーバー自体が証明書を変更することが多いため、この手順で中間証明書を置き換えるだけです。ルートCA証明書とは異なり、サーバー証明書で置き換えるのはかなり簡単です。

すでに述べたように、ルートCAはクライアントデバイス自体、OS、ブラウザー、またはその他のソフトウェアに直接埋め込まれています。ルートCA Webサイトの変更は制御できません。これには、OSとソフトウェアのどちらを更新する場合でも、クライアントでの更新が必要です。

一部のルートCAは非常に長い間存在しており、約20〜25年の話です。すぐに、最も古いルートCAのいくつかは、自然の寿命の終わりに近づき、時間はほぼ終了します。私たちのほとんどにとって、CAは新しいルート証明書を作成し、それらは長年にわたってOSおよびブラウザーの更新で世界中に配布されているため、これはまったく問題になりません。しかし、誰かがOSまたはブラウザを非常に長い間更新していない場合、これは一種の問題です。

この状況は、2020年5月30日10:48:38 GMTに発生しました。これは、Comodo Certificate Authority（Sectigo）からのAddTrustルート証明書が腐っている正確な時刻です。

リポジトリに新しいUSERTrustルート証明書がないレガシーデバイスとの互換性を確保するために、クロス署名に使用されました。

残念ながら、問題は古いブラウザーだけでなく、OpenSSL 1.0.x、LibreSSL、およびGnuTLSに基づく非ブラウザークライアントでも発生しました。たとえば、Rokuテレビセットトップボックス、Herokuサービス、フォーティネット、Chargifyアプリケーション、Linuxおよびその他の多くの .NET Core 2.0プラットフォーム。

最近のブラウザーは2番目のUSERTRustルート証明書を使用できるため、問題はレガシーシステム（Android 2.3、W​​indows XP、Mac OS X 10.11、iOS 9など）にのみ影響すると想定されていました。しかし実際には、無料のOpenSSL 1.0.xおよびGnuTLSライブラリを使用した何百ものWebサービスでクラッシュが発生しました。証明書の陳腐化エラーにより、安全な接続が確立されなくなりました。

次へ-暗号化しましょう


今後のルートCAの変更のもう1つの良い例は、Let's Encrypt証明機関です。戻る2019年4月で、彼らは自分のISRGルートのチェーンにアイデントラストチェーンから切り替えるために計画されたが、これは実現しませんでした。



「AndroidデバイスでのISRGルートの配布が不十分であるという懸念のため、2019年7月8日から2020年7月8日まで、独自のルートへの移行日を延期することを決定しました」とLet's Encryptは公式声明で述べています。

「ルートの伝播」と呼ばれる問題、またはルートCAがすべてのクライアントにあまり広がっていない場合のルートの伝播の欠如のために、日付を延期する必要がありました。

Let's Encryptは、IdenTrust DST Root CA X3のルートへのチェーンを持つクロス署名された中間証明書を使用します。このルート証明書は2000年9月に発行され、2021年9月30日に有効期限が切れます。それまでの間、Let's Encryptは独自の自己署名ルートISRGルートX1に切り替える予定です。



ISRGルートは2015年6月4日にリリースされました。その後、2018年8月6日に終了した認証センターとしての承認手続きが始まりました。これ以降、オペレーティングシステムまたはソフトウェアの更新により、すべてのクライアントがルートCAを使用できるようになりました。必要なことは、更新をインストールすることだけでした。

しかし、それが問題です。

携帯電話、テレビ、またはその他のデバイスが2年間更新されていない場合、新しいISRGルートX1ルート証明書をどのようにして知ることができますか？システムにインストールされていない場合、Let's Encryptが新しいルートに切り替わるとすぐに、すべてのLet's Encryptサーバー証明書がデバイスによって無効になります。また、Androidエコシステムには、長期間更新されていない古いデバイスがたくさんあります。

これが、Let's Encryptが独自のISRGルートへの移行を遅らせ、IdenTrustルートに至る中間リンクをまだ使用している理由です。しかし、いずれにしても移行は行われる必要があります。ルート変更の日付は2020年7月8日です。

ISRG X1ルートがデバイス（TV、セットトップボックス、またはその他のクライアント）にインストールされていることを確認するには、テストサイトhttps://valid-isrgrootx1.letsencrypt.org/を開きます。セキュリティ警告が表示されない場合、通常はすべて順調です。

新しいルートに切り替える問題を解決しなければならないのは、Let's Encryptだけではありません。インターネット上の暗号化は20年ほど前に使用され始めたので、ちょうど今、多くのルート証明書が期限切れになる時が来ています。

スマートTVソフトウェアを何年も更新していないスマートTVの所有者は、この問題に直面する可能性があります。たとえば、新しいGlobalSign R5ルートルート2012年にリリースされましたが、いくつかの古いスマートTVは、このルートCAを持たないため、チェーンを構築できません。特に、これらのクライアントは、bbc.co.ukへの安全な接続を確立できませんでした。問題を解決するために、BBC管理者はだまさなければなりませんでした。R3ルートとR1ルートの古いルートを使用して、追加の中間証明書を通じてこれらのクライアントの代替チェーンを構築しました。

www.bbc.co.uk（リーフ）
GlobalSign ECC OV SSL CA 2018（中級）
GlobalSign Root CA-R5（中級）
GlobalSign Root CA-R3（中級）

これは一時的な解決策です。クライアントソフトウェアを更新しなければ、問題はどこにも起こりません。スマートTVは、本質的に機能が制限されたLinuxベースのコンピューターです。そして、更新がないと、そのルート証明書は必然的に腐敗します。

これは、テレビだけでなく、すべてのデバイスに適用されます。インターネットに接続されていて、「スマート」デバイスとしてアドバタイズされたデバイスがある場合、不正な証明書の問題はほぼ間違いなくそれに関係しています。デバイスが更新されない場合、CAルートリポジトリは時間の経過とともに廃止され、最終的に問題が表面化します。問題が発生するまでの時間は、ルートストアが最後に更新された時間によって異なります。これは、デバイスが実際にリリースされる日付の数年前になる場合があります。

ちなみに、これは、いくつかの大規模メディアプラットフォームがLet's Encryptなどの最新の自動化された証明機関を使用できない理由であるとScott Helmeは書いています。これらはスマートTVには適しておらず、ルートの数が少なすぎるため、レガシーデバイスでの証明書のサポートを保証できません。そうしないと、テレビが最新のストリーミングサービスを起動できなくなります。

最新のAddTrustインシデントは、大規模なIT企業でさえルート証明書の有効期限の準備ができていないことを示しました。

この問題に対する解決策は1つしかありません-更新です。スマートデバイスの開発者は、ソフトウェアとルート証明書を事前に更新するメカニズムを提供する必要があります。一方、保証期間の終了時にデバイスの動作を保証することは、メーカーにとって不採算です。
