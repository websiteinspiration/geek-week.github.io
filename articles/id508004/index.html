Kapel



Chapel menggunakan forall loop untuk memparalelkan utas. C-pointer untuk setiap elemen juga digunakan, dan bukan referensi standar untuk array, dan iterasi yang dipandu oleh indeks diterapkan : 



proc calculateKernelMatrix(K, data: [?D] ?T)
{
  var n = D.dim(0).last;
  var p = D.dim(1).last;
  var E: domain(2) = {D.dim(0), D.dim(0)};
  var mat: [E] T;
  var rowPointers: [1..n] c_ptr(T) =
    forall i in 1..n do c_ptrTo(data[i, 1]);

  forall j in guided(1..n by -1) {
    for i in j..n {
      mat[i, j] = K.kernel(rowPointers[i], rowPointers[j], p);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


Kode kapel adalah yang paling sulit untuk mengoptimalkan kinerja dan membutuhkan perubahan kode terbanyak.



D



Untuk memparalelkan kode D, digunakan stream taskPool dari paket std.parallel . Kode D telah mengalami paling sedikit perubahan untuk mengoptimalkan kinerja - sangat bermanfaat untuk menggunakan kompiler tertentu dan kunci kompilasi yang dipilih (dibahas nanti). Implementasi Matriks saya memungkinkan saya untuk memilih kolom dengan referensi menggunakan refColumnSelect . 



auto calculateKernelMatrix(alias K, T)(K!(T) kernel, Matrix!(T) data)
{
  long n = data.ncol;
  auto mat = Matrix!(T)(n, n);

  foreach(j; taskPool.parallel(iota(n)))
  {
    auto arrj = data.refColumnSelect(j).array;
    foreach(long i; j..n)
    {
      mat[i, j] = kernel(data.refColumnSelect(i).array, arrj);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


Julia



Kode Julia menggunakan makro utas untuk memparalelkan kode dan makro dilihatuntuk referensi ke array. Satu-satunya hal yang membingungkan dengan array di Julia adalah status referensi mereka. Terkadang, seperti dalam kasus ini, array akan berperilaku sebagai objek nilai, dan mereka harus dirujuk menggunakan makrodilihatkalau tidak mereka akan menghasilkan salinan. Dalam kasus lain, mereka berperilaku seperti objek referensi, misalnya, ketika meneruskannya ke suatu fungsi. Ini mungkin sedikit rumit untuk ditangani, karena Anda tidak selalu tahu set operasi yang akan dihasilkan salinannya, tetapi di mana itu terjadidilihatmemberikan solusi yang baik.



Tipe Symmetric menyimpan sedikit pekerjaan tambahan yang diperlukan untuk mencerminkan dalam matriks. 



function calculateKernelMatrix(Kernel::K, data::Array{T}) where {K <: AbstractKernel,T <: AbstractFloat}
  n = size(data)[2]
  mat = zeros(T, n, n)
  @threads for j in 1:n
      @views for i in j:n
          mat[i,j] = kernel(Kernel, data[:, i], data[:, j])
      end
  end
  return Symmetric(mat, :L)
end


The @ batas dan @ macro SIMD dalam fungsi utama yang digunakan untuk menonaktifkan pemeriksaan perbatasan dan menerapkan SIMD optimasi perhitungan: 



struct DotProduct <: AbstractKernel end
@inline function kernel(K::DotProduct, x::AbstractArray{T, N}, y::AbstractArray{T, N}) where {T,N}
  ret = zero(T)
  m = length(x)
  @inbounds @simd for k in 1:m
      ret += x[k] * y[k]
  end
  return ret
end


Optimalisasi ini memberikan peningkatan yang cukup nyata, tetapi sangat mudah digunakan.

Optimalisasi kinerja



Proses mengoptimalkan kinerja dalam ketiga bahasa sangat berbeda, dan ketiga komunitas sangat berguna dalam proses ini. Tetapi ada beberapa poin umum.



  • Penjadwalan statis fungsi kernel alih-alih menggunakan polimorfisme. Ini berarti bahwa ketika mentransfer fungsi kernel, polimorfisme parametrik (waktu kompilasi) digunakan, dan bukan polimorfisme dinamis (runtime), di mana penjadwalan dengan fungsi virtual memerlukan overhead.
  • Menggunakan tampilan / tautan alih-alih menyalin data dalam mode multi-utas sangat penting.
  • Paralelisasi perhitungan sangat penting.
  • Mengetahui bahwa array adalah dasar untuk baris / kolom, dan menggunakan ini dalam perhitungan adalah yang paling penting.
  • Pemeriksaan batas dan optimisasi kompiler membuat perbedaan besar, terutama di Chapel dan D.
  • Dimasukkannya SIMD dalam D dan Julia telah berkontribusi pada kinerja. Di D, ini dilakukan menggunakan flag -mcpu = asli, dan di Julia, ini dilakukan menggunakan makro @ simd .


Dari sudut pandang spesifik bahasa, yang paling sulit adalah transisi ke kode produktif di Kapel, dan kode di Kapel paling banyak berubah: dari operasi yang mudah dibaca dengan array hingga penggunaan pointer dan pengulangan yang dikendalikan. Namun di sisi kompiler, relatif mudah untuk menambahkan - cepat dan mendapatkan peningkatan kinerja yang besar.



Kode D telah berubah sangat sedikit, dan sebagian besar kinerja telah diperoleh dengan memilih kompiler dan flag pengoptimalannya. Kompiler LDC kaya akan fitur optimasi kinerja. Ini memiliki tingkat optimasi 8-O, tetapi beberapa dari mereka diulang. Misalnya, -O, -O3 dan -O5 identik, dan ada banyak flag lain yang memengaruhi kinerja. Dalam kasus ini, flag -O5 --boundscheck = off -fast-math digunakan, yang merupakan optimisasi kompiler yang agresif, pengecekan batas, dan fast-math LLVM, dan -mcpu = asli untuk mengaktifkan instruksi vektorisasi.



Di Julia, makro dalam perubahan yang diulas sebelumnya secara nyata meningkatkan kinerja, tetapi mereka tidak terlalu membingungkan. Saya mencoba mengubah level optimisasi -O, tetapi itu tidak meningkatkan kinerja.

D memiliki perpustakaan standar paling lengkap, tetapi Julia adalah yang terbaik, kemudian Chapel, tetapi semuanya tidak pernah sesederhana itu. Sebagai contoh, pengguna yang ingin menulis biner I / O mungkin menemukan Julia yang paling mudah untuk pemula; ia memiliki antarmuka dan dokumentasi paling sederhana, paling intuitif, diikuti oleh Chapel, dan kemudian D. Meskipun dalam implementasi program saya untuk membaca file BEI, I / O D adalah yang tercepat, tetapi kode Julia mudah ditulis untuk kasus-kasus yang tidak tersedia di dua bahasa lainnya.



Manajer dan ekosistem paket



Dalam hal dokumentasi, penggunaan dan fitur, manajer paket D-Dub adalah yang paling komprehensif. D juga memiliki ekosistem paket yang kaya di situs web Dub , tetapi manajer paket Julia terintegrasi dengan GitHub dan merupakan sistem paket yang baik dengan dokumentasi yang baik. Chapel memiliki manajer paket, tetapi tidak ada ekosistem yang sangat maju.



Integrasi C



C-interoperabilitas mudah digunakan dalam ketiga bahasa; Chapel memiliki dokumentasi yang baik, tetapi tidak sepopuler yang lain. Dokumentasi D lebih baik, dan dokumentasi Julia adalah yang paling komprehensif. Namun, anehnya, tidak satu dokumentasi bahasa memiliki perintah yang diperlukan untuk mengkompilasi kode C Anda sendiri dan mengintegrasikannya dengan bahasa, yang merupakan kekhilafan, terutama ketika menyangkut pemula. Namun, dalam D dan Julia mudah untuk mencari dan menemukan contoh proses kompilasi.



Masyarakat



Ada tempat yang nyaman untuk ketiga bahasa di mana pengguna dapat mengajukan pertanyaan. Untuk Chapel, tempat termudah adalah Gitter, untuk Julia itu Wacana (meskipun ada juga Julia Gitter), dan untuk D ini adalah situs forum resmi. Komunitas Julia adalah yang paling aktif, lalu D, lalu Kapel. Saya menemukan bahwa Anda mendapatkan jawaban yang bagus dari ketiga komunitas, tetapi Anda mungkin mendapatkan jawaban yang lebih cepat untuk D dan Julia.

Ringkasan



Jika Anda seorang programmer pemula yang menulis algoritma numerik dan melakukan perhitungan ilmiah, dan menginginkan bahasa yang cepat dan mudah digunakan, Julia adalah pilihan terbaik Anda. Jika Anda seorang programmer berpengalaman yang bekerja di bidang yang sama, Julia masih merupakan pilihan bagus. Jika Anda ingin bahasa yang lebih tradisional, "industri", dikompilasi secara statis, berkinerja tinggi dengan semua "whistle-blower", tetapi menginginkan sesuatu yang lebih produktif, aman dan tidak terlalu menyakitkan daripada C ++, maka D adalah pilihan terbaik Anda. Anda dapat menulis "apa saja" dalam D dan mendapatkan kinerja luar biasa berkat kompilernya. Jika Anda perlu menghitung array pada cluster, maka Chapel mungkin solusi yang paling nyaman.



Dalam hal kinerja kotor dalam tugas ini, D adalah pemenang, jelas menunjukkan kinerja terbaik dalam 5 dari 9 tugas referensi. Studi ini menunjukkan bahwa label Julia sebagai bahasa kinerja tinggi lebih dari sekadar hype - ia memiliki kelebihan sendiri dibandingkan dengan bahasa yang sangat kompetitif. Lebih sulit dari yang diharapkan untuk mendapatkan kinerja kompetitif dari Chapel - tim Chapel perlu banyak riset untuk menghasilkan solusi saat ini. Namun, ketika bahasa Chapel mulai matang, kita akan dapat melihat peningkatan lebih lanjut.
