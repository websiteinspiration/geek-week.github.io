Otro calendario de producción en MS SQL. PO-> SHA-> GO-> IN

Se me solicitó que escribiera mi decisión en una publicación sobre Habré sobre un tema similar en el que esta tarea se resolvió "de frente", mediante una simple lista de días y una bandera de trabajo / día para toda la vida útil del sistema en el que se utiliza este calendario. En una situación similar, decidí hacer un poco más complicado, lo que finalmente resulta ser mucho más fácil de soportar. Si está interesado en cómo se hizo esto, bienvenido a cat:



Además de la simplicidad, el método "frontal" tiene dos ventajas: 1) la velocidad logarítmica de búsqueda / selección, porque la fecha en sí misma es una PC natural, y elegir el atributo de día libre por un conjunto de fechas es una operación en el campo del índice (cluster), y 2) solo hay una ligera disminución en el rendimiento cuando se verifica un conjunto aleatorio (en el rango de almacenamiento, por supuesto) con respecto al sub-rango de fecha continua.



Como resultado, si a) la verificación de la fecha puede estar en un amplio rango (incluso antes de la Gran Revolución de Octubre), b) en el predicado de la verificación, el alcance de las fechas verificadas no se mide durante una década, yc) no desea molestarse con miles de registros en la tabla, que en el 99% de los casos no llevar información útil, se puede hacer de manera diferente, a saber ...



... no almacene todas las fechas posibles (para que no haya ningún objeto al que se deba aplicar la búsqueda de índice;), sintetice el eje de fechas directamente sobre la marcha, en un rango determinado, obtenga el atributo de trabajo / día por el número del día de la semana (lunes ... viernes: trabajo, sábado ... Su: día libre), y almacene excepciones a esta regla en una tabla compacta.



Bueno,



comencemos : cree un generador de eje de fecha en un rango determinado (deje que sea una función en línea con valores de tabla, para la posibilidad de reutilización en diferentes aplicaciones): 



CREATE FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq
	FROM DR
)


Verificamos: 



SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')

Ya es más cálido: sin







embargo, hay un pequeño problema: el número de día devuelto por la función DATEPART incorporada depende del valor del parámetro DATEFIRST, y su valor predeterminado, a su vez, depende de la configuración regional del servidor. Por lo tanto, con la esperanza de que tenemos un sistema americano, y la semana comienza el domingo, es generalmente poco patriotas poco fiable.



¿Qué se puede hacer para evitar esto? Por ejemplo, lea el valor actual de DATEFIRST a través de la variable @@ DATEFIRST (que devuelve el número del primer día relativo a la presentación predeterminada para los Estados Unidos, en el que el día 1 es domingo (!)) Y luego deduzca los números de día en la notación actual en función del valor actual @@ DATEFIRST correspondiente a sábado y domingo. Tómese un momento e intente hacerlo en su mente.



Sucedió? Yo tampoco. Decidí no tomar ni un minuto más de la ayuda del club, y lo hice de manera diferente: recibí los días de la semana en la notación actual de las fechas para las que sé a priori que es sábado y domingo, por ejemplo, 1 y 2 de enero de 2000. Y luego simplemente comparó el número de días emitidos con estos a priori conocidos en el contexto actual: 



ALTER FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq,
		CONVERT(BIT, CASE WHEN DATEPART(WEEKDAY, DR.DateSeq) IN (Q.DOW_Sat, Q.DOW_Sun) THEN 1 ELSE 0 END) AS IsHoliday
	FROM DR
	CROSS JOIN (
		SELECT TOP (1)
			DATEPART(WEEKDAY, '1/1/2000')	AS DOW_Sat,	--  
			DATEPART(WEEKDAY, '1/2/2000')	AS DOW_Sun	--  
	) Q

)


Uno puede preguntarse, ¿por qué todo este CROSS JOIN / SELECT TOP (1) ... se mezcla con la solicitud? Pero para qué: dado que mi TVF está en línea y el cuerpo de la función es esencialmente un SELECT, no puede ingresar ninguna variable; todo en lo que podemos confiar principalmente en SELECT puede ser hardcode (y esto es lo que queremos evitar) ), o salida a subconsulta.



Por lo tanto, usamos CROSS JOIN (puedes INTERIAR ... EN 1 = 1, como quieras).



TOP (1) aquí necesitamos más motor SQL que nosotros mismos (después de todo, ¡entendemos que en subconsulta siempre hay una línea!), Pero dicenque para construcciones similares TOP (1) le dice al compilador de consultas que la subconsulta debe ejecutarse una vez antes que la principal, y no como CROSS APPLY para cada fila de la selección principal. No estoy 100% seguro de esto, especialmente si el cuerpo es subconsulta no determinista (lo comprobaré de alguna manera), pero que así sea por ahora.



Verificamos: 



SET DATEFIRST 5	--          
SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')


¡Bingo! ¡Incluso en la Sunny City, el fin de semana es sábado y domingo!







Ahora permanece la trivialidad: crear una tabla para almacenar excepciones y fusionarla correctamente. Haremos todo bajo el supuesto de que automatizaremos el negocio global, y unas vacaciones para un país pueden ser un día de trabajo bastante gris en otro: 



CREATE TABLE CountryCalendarOverride (
	CountryID INT NOT NULL,
	CalendarDate DATE NOT NULL,
	IsHoliday BIT NOT NULL,
       CONSTRAINT PK_CountryCalendarOverride PRIMARY KEY CLUSTERED 
       (
	  CountryID,
	  CalendarDate
       )
)


Pero, ¿qué pasa si queremos dejar la etiqueta para las vacaciones, para no olvidar que estamos celebrando? ¿O incluso no redefinir la señal de trabajo / día libre, sino simplemente dejar un comentario al contrario de alguna fecha? Muy simple: 



ALTER TABLE CountryCalendarOverride ALTER COLUMN IsHoliday BIT NULL
GO
ALTER TABLE CountryCalendarOverride ADD DateComment NVARCHAR(256) NULL


¿Por qué necesitamos IsHoliday anulable ahora? Entonces, que algún día queremos dar un comentario simplemente para una fecha determinada, sin saber / no pensar si es un día libre o un día hábil, y no vamos a redefinir esta bandera en particular, sino simplemente para decir que esta fecha necesita atención.



Ahora todo está listo para el brote final: 



SET DATEFIRST 5	--           
SELECT 
	DX.DateSeq,
	ISNULL(CCO.IsHoliday, DX.IsHoliday) AS IsHoliday,
	CCO.DateComment
FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01') DX
LEFT JOIN CountryCalendarOverride CCO ON DX.DateSeq = CCO.CalendarDate 
                     AND CCO.CountryID = 1	-- ,     
ORDER BY DX.DateSeq
OPTION (MAXRECURSION 0)
