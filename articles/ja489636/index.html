フロントエンドの何が問題になっていますか？


すべての人がフロントエンドで問題ないことに同意しますか？



テストはほとんどありませんか？遅いアセンブリ？生態系？良い。

フロントエンドに関して、私はフロントエンド開発者の一人が彼の本で言った引用が好きです：



Javascriptには型システムはありません。次に、作業中に発生した問題に名前を付け、解決方法を説明します。

型システムをJavasriptの型システムと呼ぶのは一般に困難です-オブジェクトの型を示す線がありますが、実際には型とは関係ありません。この問題は、TypeScript（Javasriptのアドオン）およびFlow（Javascriptの静的チェッカー型）で解決されています。実際、フロントエンドは、Javascriptの型システムの問題の解決策にすでに達しています。



そのため、ブラウザには標準ライブラリはありません。ブラウザには、いくつかの組み込みオブジェクトと「マジック」関数があります。しかし、私はそのようなJavaScriptの標準ライブラリを持っていません。この問題は、かつてjQueryによって解決されました（誰もが、動作に必要なすべてのプロトタイプ、ヘルパー、関数でjQueryを使用していました）。現在、誰もがLodash：



Callback hell を使用しています。誰もが約5年前にJavascriptコードを見たと思います。それは、コールバックの信じられないほどの複雑さから「ヌードル」のように見えました。現在、この問題は（ES-15またはES-16のリリースで）解決され、Javascriptに約束が追加され、しばらくの間誰もが呼吸しやすくなりました。



Promiceの地獄が来るまで...フロントエンド業界がどのように管理しているかはわかりませんが、彼らは常にいくつかの奇妙な荒野に追い込まれています。彼らはまた、「約束」をなんとかしました。次に、新しいプリミティブを追加することでこの問題を解決しました-async / await：



問題は非同期で解決されます。Async / awaitは、さまざまな言語でかなり人気のあるプリミティブです。「Python」などでは、そのようなアプローチがあります。それで十分です。問題は解決されました。

解決されない問題は何ですか？指数関数的に成長するフレームワークの複雑さ、エコシステムとプログラム自体の複雑さ。



  • Javasriptの構文は少し奇妙です。配列やオブジェクトの追加、その他のジョークの問題は誰もが知っています。
  • JavaScriptはマルチパラダイムです。現在、生態系が非常に大きい場合、これは特に緊急のシステムです。

    • 誰もが異なるスタイルで書き込みます-誰かが構造的に書き込み、誰かが機能的に書き込み、異なる開発者が異なる方法で書き込みます。
    • 異なるパッケージから、異なるパッケージを使用するときの異なるパラダイム。
    • Javasriptの関数型プログラミングには多くの「楽しみ」があります-ラムダライブラリが登場し、このライブラリで作成されたプログラムをだれも読むことができなくなりました。

  • これはすべてエコシステムに大きな影響を与え、信じられないほど成長しました。パッケージは互いに互換性がありません：約束の誰か、非同期/待機の誰か、コールバックの誰か。彼らはまた、異なるパラダイムで書いています！
  • これにより、プロジェクトの維持が困難になります。コードを読み取れない場合、バグを見つけるのは困難です。

Web Assemlyとは何ですか？


Mozilla Foundationや他のいくつかの企業の勇敢な連中が、Web Assemblyのようなものを思いついた。それは何ですか？



  • これは、バイナリ形式をサポートするブラウザに組み込まれた仮想マシンです。
  • バイナリプログラムはそこに到達し、ほぼネイティブに実行されます。つまり、ブラウザーは毎回JavaScriptコードのすべての「ヌードル」を解析する必要はありません。
  • すべてのブラウザがサポートを宣言しています。
  • これはバイトコードなので、任意の言語のコンパイラを作成できます。
  • 4つの主要なブラウザには、Webアセンブリサポートがすでに付属しています。
  • Goでのネイティブサポートをまもなく待ちます。そのような新しいアーキテクチャはすでに追加されています：GOARCH = wasm GOOS = js（まもなく）。これまでのところ、私が理解しているように、それは機能的ではありませんが、それは間違いなくGoに含まれるという記述があります。

私は何を提案しますか？


現在、フロントエンド業界は、これより前にJavascriptから叫んだすべての言語がそこに殺到するような状態にあります。これで全員がWebアセンブリにコンパイルされます。 「ゴーファー」としてそこにふさわしい場所を取るために何が必要ですか？



Goは伝統的にシステムプログラミング言語であると考えてきましたが、UIを操作するためのライブラリは実際にはありません。何かがそこにありますが、それは半分放棄され、半分は機能していません。

そして、GopherJSで実行されるGoでUIライブラリを作成する良い機会です！ついに独自のフレームワークを書くことができます！あなたがフレームワークを書くことができる時が来ました、そして彼は初期の適応を受ける最初の一人の一人となり、あなたはスターになります（それが良いフレームワークであれば）。

Goエコシステムにすでに含まれている多くの異なるパッケージを、ブラウザーの詳細（例えば、テンプレートエンジン）に適合させることができます。これらは既に機能しているため、便利なバインディングを作成して、ブラウザーで直接コンテンツを簡単に直接レンダリングできます。さらに、たとえば、同じコードを使用して、サーバーとフロントエンドで同じものをレンダリングできるサービスを、すべてフロントエンド開発者が好きなように（現在はGoでのみ）作成できます。
