複雑すぎる

プログラマーになるためには怠惰である必要のある表現を聞きました。しかし、プログラミングの怠惰は、ひどい技術的負債につながることがあります。私のSRPに関する記事で、この原則に違反すると、複雑さが増したり、さらには倍増したりする可能性があることを述べました。同僚の1人が興味深い例を作成し、それがどのように見えるかを示すことにしました。



この過度の複雑さを判断しましょう。しかし、最初に、その逆、要件の複雑さについて話しましょう。たとえば、時給と労働時間から従業員の給与を計算する必要があります。また、従業員が5年以上会社にいる場合は、ボーナスが発生します。これは「要件」に由来するものであり、回避することはできません。何らかの形で、それはアプリケーションコードの複雑さの要素になり、おそらく条件演算子「if」の形になります。ただし、複雑さが要件から先に進むのではなく、問題を解決するための開発者のアプローチに従う場合もあります。

「if」演算子、「戦略」、多態性メソッドなどのパターンは、この過度の複雑さを含む可能性のあるプログラミング手法の完全なリストではありません。個人的には、ちなみに私は、開発者が特定の問題を解決できるという理由だけでパターンを使用することに反対しています。

これは簡単な例です。架空のように見えるかもしれませんが、そうではありません。単純化すらされておらず、数年前のコードレビューで彼に会ったのはこの形でした。コードの2つの場所で、同じ関数を呼び出しましたが、ブール値パラメーターが異なりました。

// first place
doSomething(true);

// second place
doSomething(false);

同様のデザインは常に疑わしく見え、この機能は私を失望させませんでした。このパラメーターは、この関数内でチェックされる唯一の目的で渡されました。

doSomething(flag: boolean): void {
  if(flag) {
    // do first thing
  } else {
    // do second thing
  }
}

このチェックは、「場所Aから呼び出された場合は1つの処理を行い、それ以外の場合は場所Bから呼び出された場合は別の処理を行う」と説明できます。このフラグ、この「if」は、このメモ全体に関するものです。複雑さはビジネス要件に起因するものではありません。もちろん、次のようにコードを変更することをお勧めします。

// first place
doFirstThing();

// second place
doSecondThing();

//method is split into 2 parts each having their own responsibility
doFirstThing(): void {
  // do first thing
}

doSecondThing(): void {
  // do second thing
}

以上で、複雑さはなくなりました。これは、開発者が怠惰であってはならず、別の関数シグネチャを記述する必要がある場所です。

ここで、「しかし、これは1つの 'if'にすぎません」、または「この違反は明白です。誰がそのようなコードを書いているのですか？」と宣言できます。そして、これが2番目の例です。これは、違反を確認することが著しく困難になる可能性があること、およびこの違反のコストが1つの「if」だけではない場合があることを示しています。最初の例と同様に、関数は2つの場所で使用されます。

// first place
checkValidity(obj);

// second place
checkValidity(arrayOfObjs);

メソッドは、その名前から次のように、オブジェクトの有効性をチェックします。ただし、オブジェクトの配列の有効性もチェックできることは明らかではありませんでした。この違反を強調するために、変数名を調整しました。メソッドは次のようになります。

checkValidity(parm: MyType | MyType[]): void {
  if(Array.isArray(parm)) {
    parm.forEach(p => checkValidity(p));
  } else {
    // here the object gets checked 
    // and conditional exception is thrown
  }
}

ここにあります。たくさんのイフになるなら一つ。配列に100個のオブジェクトが含まれている場合、この「if」は101回実行されます。実際のデータでは、3万個のオブジェクトが存在する可能性があり、これはすでに目覚しいパフォーマンスの低下です。

明らかに、唯一の責任の原則に従って、このメソッドは2つのメソッドが取得されるようにリファクタリングする必要があります。

checkItemsValidity(parms: MyType[]): void {  
  parms.forEach(p => checkItemValidity(p));
}

checkItemValidity(parm: MyType): void {
    // here the object gets checked 
    // and conditional exception is thrown  
}

それに応じて、呼び出しポイントも調整する必要があります。

SRPについての記事で私が挙げた例がSLOCの増加につながったのは興味深いことです。同じ例が、逆に、コードの品質の予想される改善とともに、SLOCがわずかに減少しました。

それで全部です。優れたコードの最も重要な原則を示すための簡単な例を2、3だけ示します。
