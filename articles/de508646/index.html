Die Hauptidee der Turniersortierung besteht darin, einen relativ kleinen (im Vergleich zum Hauptarray) Hilfsheap zu verwenden, der als Prioritätswarteschlange fungiert. In diesem Heap werden die Elemente auf den unteren Ebenen verglichen, wodurch die kleineren Elemente (in diesem Fall der MIN-HEAP-Baum) nach oben gehen und das aktuelle Minimum des Teils der Array-Elemente, der in diesen Heap fällt, im Stammverzeichnis angezeigt wird. Das Minimum wird auf ein zusätzliches Array von "Gewinnern" übertragen, wodurch der Heap die verbleibenden Elemente vergleicht / verschiebt - und jetzt gibt es ein neues Minimum in der Wurzel des Baums. Beachten Sie, dass bei einem solchen System das nächste Minimum einen höheren Wert hat als das vorherige - dann kann ein neu geordnetes Array von „Gewinnern“ einfach zusammengestellt werden, wobei neue Minima einfach am Ende des zusätzlichen Arrays hinzugefügt werden.



Das Übertragen der Minima auf ein zusätzliches Array führt dazu, dass freie Stellen für die folgenden Elemente des Hauptarrays im Heap angezeigt werden. Infolgedessen werden alle Elemente in der Reihenfolge der Warteschlange verarbeitet.



Die wichtigste Subtilität ist, dass es neben den „Gewinnern“ auch „Verlierer“ gibt. Wenn einige Elemente bereits in das Array "Gewinner" übertragen wurden und die unverarbeiteten Elemente kleiner als diese "Gewinner" sind, macht es zu diesem Zeitpunkt keinen Sinn, sie durch den Turnierbaum zu sichten. Das Einfügen dieser Elemente in das Array "Gewinner" ist zu teuer (in Sie können ihnen nicht bereits ein Ende setzen, aber um sie am Anfang zu setzen, müssen Sie die bereits eingefügten Tiefs verschieben. Solche Elemente, die nicht in die Reihe der „Gewinner“ passen, werden an die Reihe der „Verlierer“ gesendet. Sie werden in einer der folgenden Iterationen durch den Turnierbaum geführt, wenn alle Aktionen für die verbleibenden Verlierer wiederholt werden.



Es ist nicht einfach, eine Implementierung dieses Algorithmus im Internet zu finden, aber auf YouTube wurde eine klare und hübsche Implementierung in Ruby gefunden. Im Abschnitt "Links" wird Java-Code noch erwähnt, sieht dort jedoch nicht so lesbar aus.

Dies ist eine naive Implementierung, bei der nach jeder Unterteilung in "Verlierer" und "Gewinner" diese Arrays kombiniert werden und dann für das kombinierte Array alle Aktionen erneut wiederholt werden. Wenn die "verlierenden" Elemente zuerst im kombinierten Array angezeigt werden, werden sie beim Durchsuchen des Turnierstapels zusammen mit den "Gewinnern" sortiert.









Diese Implementierung ist recht einfach und offensichtlich, aber Sie werden sie nicht als effektiv bezeichnen. Sortierte "Gewinner" gehen mehr als einmal durch den Turnierbaum, was offensichtlich unnötig erscheint. Ich gehe davon aus, dass die zeitliche Komplexität hier log-quadratisch ist, O ( n log 2 n ) .



Multipath Merge Option



Der Algorithmus wird spürbar beschleunigt, wenn die geordneten Elemente, die das Sieb passieren, nicht wiederholt durch Turnierrennen geleitet werden.



Nachdem das ungeordnete Array in sortierte „Gewinner“ und unsortierte „Verlierer“ unterteilt wurde, wird der gesamte Vorgang nur für „Verlierer“ erneut wiederholt. Bei jeder neuen Iteration sammelt sich eine Reihe von Arrays mit „Gewinnern“ an und wird auf diese Weise fortgesetzt, bis in keinem Schritt mehr „Verlierer“ vorhanden sind. Danach müssen nur noch alle Arrays von "Gewinnern" zusammengeführt werden. Da alle diese Arrays sortiert sind, erfolgt diese Zusammenführung zu minimalen Kosten.









In dieser Form findet der Algorithmus möglicherweise bereits eine nützliche Anwendung. Wenn Sie beispielsweise mit Big Data arbeiten müssen, werden während der Verwendung des Turnierhaufens Pakete mit geordneten Daten freigegeben, mit denen Sie bereits etwas tun können, während alles andere verarbeitet wird.



Jedes der n Elemente des Arrays durchläuft den Turnierbaum nur einmal, was zeitlich O (log n ) kostet . In einer solchen Implementierung hat der Algorithmus die schlechteste / durchschnittliche Zeitkomplexität O ( n log n ) .
