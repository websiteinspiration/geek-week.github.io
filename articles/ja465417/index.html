Kubernetes Storage Volume Plugins：Flexvolume to CSI



Kubernetesがまだv1.0.0であった時代には、ボリュームプラグインが存在していました。これらは、永続的な（永続的な）コンテナーデータを格納するためにKubernetesシステムに接続するために必要でした。その数は少なく、GCE PD、Ceph、AWS EBSなどのストレージプロバイダーが最初でした。

プラグインはKubernetesと一緒に配信され、その名前がツリー内に付けられました。しかし、そのようなプラグインの既存のセットの多くは十分ではありませんでした。職人はパッチを使用してKubernetesのコアにシンプルなプラグインを追加し、その後、独自のKubernetesを組み立ててサーバーに配置しました。しかし、時間が経つにつれて、Kubernetesの開発者たちは、その魚を解決できないことに気づきました。人々は釣り竿を必要としています。そして、Kubernetes v1.2.0リリースでは、それが登場しました...

Flexvolumeプラグイン：最小限の釣り竿


Kubernetesの開発者はFlexVolumeプラグインを作成しました。これは、サードパーティのFlexvolumeドライバーを操作するための変数とメソッドの論理的なバインディングです。

停止して、FlexVolumeドライバーが何であるかを詳しく見てみましょう。これは特定の実行可能ファイル（バイナリファイル、Pythonスクリプト、Bashスクリプトなど）であり、実行されると、コマンドライン引数を取り、以前から既知のフィールドを持つメッセージをJSON形式で返します。慣例により、最初のコマンドライン引数は常にメソッドであり、残りの引数はそのパラメーターです。

最小限のメソッドセットは次のようになります。

flexvolume_driver mount #      pod'
#   :
{
  "status": "Success"/"Failure"/"Not supported",
  "message": "       ",
}

flexvolume_driver unmount #      pod'
#   :
{
  "status": "Success"/"Failure"/"Not supported",
  "message": "       ",
}

#   :
{
  "status": "Success"/"Failure"/"Not supported",
  "message": "       ",
  // ,     attach/deatach
  "capabilities":{"attach": True/False}
}

メソッドattachを使用detachして、将来ドライバーを呼び出すときにkubeletを実行するシナリオを定義します。そこ特殊な方法もあるexpandvolumeし、expandfsそれが動的にボリュームのサイズを変更する責任があります。

メソッドが追加する変更の例として、expandvolumeリアルタイムでボリュームのサイズを変更できる機能として、Rook Cephオペレーターでプルリクエストを確認できます。

以下は、NFSを操作するためのFlexvolumeドライバーの実装例です。

usage() {
    err "Invalid usage. Usage: "
    err "\t$0 init"
    err "\t$0 mount <mount dir> <json params>"
    err "\t$0 unmount <mount dir>"
    exit 1
}

err() {
    echo -ne $* 1>&2
}

log() {
    echo -ne $* >&1
}

ismounted() {
    MOUNT=`findmnt -n ${MNTPATH} 2>/dev/null | cut -d' ' -f1`
    if [ "${MOUNT}" == "${MNTPATH}" ]; then
        echo "1"
    else
        echo "0"
    fi
}

domount() {
    MNTPATH=$1

    NFS_SERVER=$(echo $2 | jq -r '.server')
    SHARE=$(echo $2 | jq -r '.share')

    if [ $(ismounted) -eq 1 ] ; then
        log '{"status": "Success"}'
        exit 0
    fi

    mkdir -p ${MNTPATH} &> /dev/null

    mount -t nfs ${NFS_SERVER}:/${SHARE} ${MNTPATH} &> /dev/null
    if [ $? -ne 0 ]; then
        err "{ \"status\": \"Failure\", \"message\": \"Failed to mount ${NFS_SERVER}:${SHARE} at ${MNTPATH}\"}"
        exit 1
    fi
    log '{"status": "Success"}'
    exit 0
}

unmount() {
    MNTPATH=$1
    if [ $(ismounted) -eq 0 ] ; then
        log '{"status": "Success"}'
        exit 0
    fi

    umount ${MNTPATH} &> /dev/null
    if [ $? -ne 0 ]; then
        err "{ \"status\": \"Failed\", \"message\": \"Failed to unmount volume at ${MNTPATH}\"}"
        exit 1
    fi

    log '{"status": "Success"}'
    exit 0
}

op=$1

if [ "$op" = "init" ]; then
    log '{"status": "Success", "capabilities": {"attach": false}}'
    exit 0
fi

if [ $# -lt 2 ]; then
    usage
fi

shift

case "$op" in
    mount)
        domount $*
        ;;
    unmount)
        unmount $*
        ;;
    *)
        log '{"status": "Not supported"}'
        exit 0
esac

exit 1

そのため、実際の実行可能ファイルを準備したら、ドライバーをKubernetesクラスターに配置する必要があります。ドライバーは、事前定義されたパスに従って、クラスターの各ノードに配置する必要があります。デフォルトで選択されています：

/usr/libexec/kubernetes/kubelet-plugins/volume/exec/__~_/

...しかし、さまざまなKubernetesディストリビューション（OpenShift、Rancher ...）を使用する場合、パスは異なる場合があります。

フレックスボリュームの問題：釣り竿を投げる方法は？


フレックスボリュームドライバーをクラスターノードに配置するのは簡単な作業ではありません。操作を手動で1回実行すると、新しいノードの追加、自動水平スケーリング、またはさらに悪いことに、誤動作によるノードの交換が原因で、クラスターに新しいノードが表示される状況が発生しやすくなります。この場合、同じ方法でFlexvolumeドライバーを手動でノードに追加するまで、これらのノードのストレージを操作することはできません。

この問題の解決策はKubernetesプリミティブの1つでした-DaemonSet。クラスターに新しいノードが表示されると、ノードは自動的にDaemonSetからポッドを取得します。ポッドには、Flexvolumeドライバーを見つけるためにローカルボリュームが接続されています。作成が成功すると、ポッドはドライバーがディスクで動作するために必要なファイルをコピーします。

FlexvolumeプラグインをレイアウトするためのこのようなDaemonSetの例を次に示します。

apiVersion: extensions/v1beta1
kind: DaemonSet
metadata:
  name: flex-set
spec:
  template:
    metadata:
      name: flex-deploy
      labels:
        app: flex-deploy
    spec:
      containers:
        - image: <deployment_image>
          name: flex-deploy
          securityContext:
              privileged: true
          volumeMounts:
            - mountPath: /flexmnt
              name: flexvolume-mount
      volumes:
        - name: flexvolume-mount
          hostPath:
            path: <host_driver_directory>

...そしてFlexvolumeドライバーをレイアウトするためのBashスクリプトの例：

#!/bin/sh

set -o errexit
set -o pipefail

VENDOR=k8s.io
DRIVER=nfs

driver_dir=$VENDOR${VENDOR:+"~"}${DRIVER}
if [ ! -d "/flexmnt/$driver_dir" ]; then
  mkdir "/flexmnt/$driver_dir"
fi

cp "/$DRIVER" "/flexmnt/$driver_dir/.$DRIVER"
mv -f "/flexmnt/$driver_dir/.$DRIVER" "/flexmnt/$driver_dir/$DRIVER"

while : ; do
  sleep 3600
done

コピー操作はアトミックではないことを忘れないでください。 kubeletは、準備のプロセスが完了する前にドライバーの使用を開始する可能性が非常に高いため、システムでエラーが発生します。正しいアプローチは、最初に別の名前でドライバーファイルをコピーしてから、アトミックな名前変更操作を使用することです。


RookオペレーターでCephを操作するスキーム：図のFlexvolumeドライバーはRookエージェント内

にありますFlexvolumeドライバーを使用する際の次の問題は、クラスターノードのほとんどのストレージに必要なソフトウェアをインストールする必要があることです（たとえば、Cephのceph-commonパッケージ）。 。当初、Flexvolumeプラグインは、このような複雑なシステムを実装するようには設計されていませんでした。

この問題の元の解決策は、RookオペレーターのFlexvolumeドライバーの実装で確認

できます。ドライバー自体は、RPCクライアントとして設計されています。通信用のIPCソケットは、ドライバ自体と同じディレクトリにあります。ドライバーファイルをコピーするには、ディレクトリとドライバーをボリュームとして接続するDaemonSetを使用するのが良いでしょう。必要なルークドライバーファイルをコピーした後、このポッドは停止しませんが、付属のボリュームを介して本格的なRPCサーバーとしてIPCソケットに接続します。ceph-commonパッケージは、ポッドコンテナー内にすでにインストールされています。IPCソケットは、kubeletが同じノードにある特定のポッドと通信するという信頼を与えます。独創的なものはすべてシンプルです！..

さようなら、私たちの愛情深い...ツリー内プラグイン！


Kubernetes開発者は、カーネル内のストレージ用のプラグインの数が20であることを発見しました。そして、それぞれの変更は、なんとかして完全なKubernetesリリースサイクルを通過します。

ストレージに新しいバージョンのプラグインを使用するには、クラスター全体を更新する必要があることがわかりました。これに加えて、Kubernetesの新しいバージョンが使用中のLinuxカーネルと突然互換性を失うことに驚くかもしれません...したがって、涙を拭き取り、歯を磨き、当局とユーザーとLinuxカーネルとKubernetesクラスターを更新する時間を調整します。サービスの提供にダウンタイムが発生する可能性があります。

状況はコミカルではありませんか？このアプローチが機能しないことがコミュニティ全体に明らかになりました。Kubernetesの開発者は強い意志をもって決定し、新しいストレージプラグインがカーネルに受け入れられなくなることを発表しました。Flexvolumeプラグインの実装では、すでに知っているように、他のすべての問題が明らかになり

ました。永続的なデータストレージの問題を解決するために、Kubernetesのボリュームに最後に追加されたCSIと呼ばれるプラグインが呼び出されました。完全にOut-of-Tree CSIボリュームプラグインと呼ばれるそのアルファ版は、Kubernetes 1.9で発表されました。

コンテナーストレージインターフェイス、またはCSI 3000回転！


まず、CSIは単なるボリュームプラグインではなく、データウェアハウスを操作するためのカスタムコンポーネントを作成するための実際の標準であることを指摘しておきます。KubernetesやMesosなどのコンテナーオーケストレーションシステムは、この標準に従って実装されたコンポーネントの操作方法を「学習」する必要があると想定されていました。そして今、Kubernetesはすでに学びました。

KubernetesのCSIプラグインのデバイスは何ですか？CSIプラグインは、サードパーティの開発者が作成した特別なドライバー（CSIドライバー）で動作します。KubernetesのCSIドライバーは、少なくとも2つのコンポーネント（ポッド）で構成する必要があります。

  • コントローラー -外部永続ストアを管理します。プリミティブが使用されるgRPCサーバーとしてリリースされますStatefulSet。
  • ノード -永続ストアをクラスターノードにマウントします。また、gRPCサーバーとして実装されますが、プリミティブが使用されますDaemonSet。
