Game of Thronesの例でES6 +でデザインパターンを生成する



設計パターン-ソフトウェア開発で最も一般的な問題を解決する方法。この記事では、ゲーム・オブ・スローンズを参照して生成パターンを見ていきます。ここで

構造パターンについてお読みください。

生成パターンは、この状況で最も適切な方法でオブジェクトを作成するために、オブジェクトを構築するメカニズムと連携するように設計されています。

最も一般的な生成パターンは次のとおりです。

  • 工場（ファブリック）
  • 抽象化（抽象）
  • コンストラクター（ビルダー）
  • プロトタイプ（プロトタイプ）
  • シングルトン（シングルトン）

工場


ファクトリーは、いわゆるファクトリー・メソッドを使用してオブジェクトを作成するパターンであり、作成されたオブジェクトのクラスを判別する必要はありません。これは何を意味するのでしょうか？

兵士を作れるようになりたいと想像してみてください。これらの兵士は、ターガリエンの家かラニスターの家のどちらかから来ます。

これには、次のものが必要です。

  • 兵士を定義するためのインターフェース
  • すべての家に力を与える兵士の各タイプのクラス
  • 彼が所属する家に関係なく、新しい兵士の作成を要求するクラス

class Soldier {
    constructor(name) {
        this.name = name
        this.attack = this.attack.bind(this)
    }

    attack() {}
}

class LannisterSoldier extends Soldier {
    attack() {
        return 'Lannister always pays his debts'
    }
}

class TargaryenSoldier extends Soldier {
    attack() {
        return 'Fire and blond'
    }
}

class Spawn {
    constructor(type, name) {
        if (type === 'lannister') return new LannisterSoldier(name)
        else return new TargaryenSoldier(name)
    }
}

(() => {
    const lannister = new Spawn('lannister', 'soldier1')
    const targaryen = new Spawn('targaryen', 'soldier2')
    console.log(lannister.attack())
    console.log(targaryen.attack())
})()

いつ使用されますか？


  • インターフェースまたは抽象クラスの実装が間もなく変更されることが予想される場合
  • 既存の実装が変更にすばやく適応できない場合
  • 初期化プロセスが単純で、コンストラクターが少数のパラメーターを受け入れる場合

抽象化


免責事項：オブジェクトに基づく抽象テンプレート。関数型プログラミングで使用するのは非常に困難です。

このテンプレートを使用すると、特定のクラスを定義する必要なく、同様のタスクを実行する個々のファクトリのグループをカプセル化できます。標準的な使用では、クライアントソフトウェアは抽象ファクトリの特定の実装を作成し、次に一般的なファクトリインターフェイスを使用して、単一システムの一部として特定のオブジェクトを作成します。これには共通のインターフェースが使用されるため、クライアントは各内部ファクトリーから受け取る特定のオブジェクトを知りません（または重要ではありません）。

デザイナーの目的は、複雑なオブジェクトをその表現から分離することです。オブジェクトの複雑さにより、このテンプレートを使用すると、新しいオブジェクトを作成するプロセスを別のオブジェクト（コンストラクター）で分離できます。

すべての家に艦隊を作成したいと想像してください。各家族には一定数の船と戦士がいます。作成を容易にするために、必要なすべての設定を自動的に作成するmakeNavyメソッドを呼び出すことができるはずです。

これには、次のものが必要です。

  • フリートを作成するためのクラス
  • コンストラクター定義のインターフェース
  • 軍のオブジェクトを作成するクラス
  • 兵士と船の作成を担当するクラス

class Lannister {
    constructor() {
        this.soldiers = []
        this.ships = []
        this.makeNavy = this.makeNavy.bind(this)
    }
    makeNavy(soldiers, ships) {
        const Build = new ConcreteBuilder()
        for (let i = 0; i < soldiers; i++) {
            this.soldiers.push(Build.createSoldier())
        }
        for (let i = 0; i < ships; i++) {
            this.ships.push(Build.createShip())
        }
    }
}

class Builder {
    createSoldier() {}
    createShip() {}
}

class ConcreteBuilder extends Builder {
    createSoldier() {
        const soldier = new Soldier()
        return soldier
    }
    createShip() {
        const ship = new Ship()
        return ship
    }
}

class Soldier {
    constructor() {
        console.log('soldier created')
    }
}

class Ship {
    constructor() {
        console.log('ship created')
    }
}

(() => {
    const lannister = new Lannister()
    lannister.makeNavy(100, 10)
})()

いつ使用されますか？


  • オブジェクトの作成プロセスが非常に複雑な場合、多数の必須およびオプションのパラメーターが含まれます
  • コンストラクターパラメーターの数の増加がコンストラクターの数の増加につながる場合
  • クライアントが、構築されたオブジェクトに対して異なるビューを期待している場合

プロトタイプ


プロトタイプを使用すると、プロトタイプ継承によって作成されたオブジェクトのタイプを判別し、既存のオブジェクトのスキームを使用して新しいオブジェクトを作成できます。これにより、パフォーマンスが向上し、メモリの損失が最小限に抑えられます。

白い歩行者の軍隊を作りたいと想像してみてください。この軍隊に特別な要件はありません。それらの多くが必要であり、それらは同じ特性を持ち、メモリ領域をあまり消費しません。

これには、次のものが必要です。

  • ホワイトウォーカーに関する情報を格納するクラス
  • 同じメソッドを返すインスタンスを複製するためのメソッド

class WhiteWalker {
    constructor(force, weight) {
        this.force = force
        this.weight = weight
    }

    clone() {
        console.log('cloned')
        return new WhiteWalker(this.name, this.weight)
    }
}

(()=>{
    const firstWalker = new WhiteWalker()
    const walkers = [firstWalker]
    for(let i=0;i<100;i++){
        walkers.push(firstWalker.clone())
    }
})()

プロトタイプの長所と短所


長所：

  • 新しい演算子を使用して新しいオブジェクトを作成する必要がなくなるため、コスト、時間、生産性を節約できます
  • オブジェクトの初期化の複雑さを軽減します。各クラスは独自の複製方法を使用します
  • 抽象テンプレートを使用する場合のように、オブジェクトを初期化するために多くのサブクラスを分類および作成する必要はありません
  • コピーされたオブジェクトの一部のプロパティを変更して新しいオブジェクトを作成することにより、システムの柔軟性を向上させます

マイナス：

  • 循環参照を持つ複雑なオブジェクトのクローン作成は簡単な作業ではありません

シングルトン


シングルトンを使用すると、作成したオブジェクトが特定のクラスの唯一のインスタンスであることを確認できます。このようなオブジェクトは通常、システム内のいくつかの操作を制御するために使用されます。

ドラゴンマザーが1人になることを確認したいとします。

これには、次のものが必要です。

  • ドラゴンの母に関する情報を保存するクラス
  • ドラゴンマザーの保存されたインスタンス

let INSTANCE = null

class MotherOfDragons {
    constructor(name, dragons) {
        if(!!INSTANCE) return INSTANCE
        this.dragons = dragons
        this.name = name
        this.getMyDragons = this.getMyDragons.bind(this)
        INSTANCE = this
    }

    getMyDragons(){
        console.log(`I'm ${this.name} and my dragons are`, this.dragons)
    }
}

(()=>{
    const dragonMother = new MotherOfDragons('Daenerys Targaryen', [
        'Drogon',
        'Rhaegal',
        'Viserion'
    ])
    const dragonMother2 = new MotherOfDragons('Cercei Targaryen', [
        'Tirion',
        'Jennifer',
        'Goblin'
    ])
    dragonMother.getMyDragons()
    dragonMother2.getMyDragons()
    console.log(dragonMother instanceof MotherOfDragons)
    console.log(dragonMother2 instanceof MotherOfDragons)
    console.log(dragonMother2 === dragonMother)
})()

いつ使用されますか？


  • オブジェクトの作成に使用されるリソースが制限されている場合（データベース接続オブジェクトなど）
  • シングルトンで承認を構築してパフォーマンスを向上させることをお勧めします。
  • アプリケーションを構成するクラスを作成する必要がある場合
  • リソース割り当て用のクラスを作成する必要がある場合
