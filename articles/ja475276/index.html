アサートを正しく書く方法

それでは、アサーションについて話しましょう。

アサーションは当たり前のことの1つなので、ここにあるものはすべて専門家のようです。誰かが組み込みのJavaまたはJunitを使用し、誰かが高度なライブラリを使用し、誰かが自分のライブラリを構築します。

しかし、もっと真剣にこれに取り組みましょう。実際、どうですか？

まず、アサーションは仕様を複製します。仕様shouldReturnValidOrderが指定されている場合は、assertNotNullではなく、これをチェックする必要があります。

次に、アサーションは読みやすくする必要があります。つまり、仕様と比較して瞬時に。

アサーションが多すぎる場合（私の極端な趣味では、それは複数ですが、少なくとも5から7の制限を設けます）、テストは理解できなくなります。良いことに、それぞれのテストが失敗する理由は1つだけです。

アセットは、プリミティブフィールド、オブジェクト、コレクション、まれに機能またはサンクに向けることができます。

オブジェクトが複雑になるほど、アサーションはより複雑で多様になり、それらを読み取ってエラーを見つけるのが難しくなります。

肯定的なステートメントは、二重の否定的なステートメントよりも読みやすくなります。

標準メソッドは、カスタムヘルパーよりも適切に読み取られます。（誰が、どこでヘルパーのライブラリをテストしたかを知っている人）。また、アサーションはテストメソッドの本体にある必要があり、テストヘルパーの後ろのどこかにあるべきではありません（ソナーはアサーションの欠如を正しく誓います）。
フィールドに関するアセットは、オブジェクト、特にコレクションに関するアサーションよりも理解しやすくなっています。

ネストされたフィールドの場合、完全にテストすることは理にかなっています。

ない

assertNotNull(order.getCustomer().getName)

そして

assertNotNull(order)
assertNotNull(order.getCustomer())
assertNotNull(order.getCustomer().getName())

アサーは、メソッドから返されたものを愚かにチェックするだけでなく、それに影響を与えます。そして、私たちがそれを変更できるなら、それを変更する必要があります。

コレクションについて主張することは比較的困難です。ゼロが含まれているか、ソートされているか、正確にソートされているか、要素が等しいかどうかがどのようにチェックされているかなど。-これはすべて、特にコレクションの要素に関連するメソッドにロジックがある場合、アサーションにとってコレクションを困難なオブジェクトにします。

したがって、次のような方法：

List<Order> getOrderList(List<OrderDao> orderDaoList){
        return orderDaoList.stream().map(orderDao=>order.name(orderDao.getName).build()).collect(Collectors.toList())
}

簡単に2つに分割して、1つのトランスフォーマーorderDao => orderを個別にテストし、2つ目は、抽象マッパーでのコレクションのマッピングをテストし、スタブでテストできます。

List<Order> getOrderList(List<OrderDao> orderDaoList, OrderDaoToOrderMapper mapper){
        return orderDaoList.stream().map(mapper::map).collect(Collectors.toList())
}

一方、コレクションは、類型化と拡張に適しています。テストされたすべてのプロパティ、共分散および反変などを備えた独自のタイプのコレクションを比較的簡単に作成できます。したがって、ジェネリックリストの代わりに、独自のOrderList、またはOrderSet、またはOrderSortedSetを作成することができます。そして、テストは簡単になります。

関数に関するステートメントの作成は、オブジェクトに関するものよりもはるかに複雑ではなく、一部の言語では型付けが適切であるため、より優れた型付けを推奨できます。Func <Order、Order>の代わりに、いくつかのOrderComparatorを返します。
