Skalierungskonzept bedeutet die Möglichkeit, die Anzahl der Replikate durch Erhöhen oder Verringern der Anzahl der Anwendungsinstanzen zu steuern.



Beispiel: Wenn Sie das System auf 6 Replikate skalieren möchten, verwenden Sie den Befehl docker service scale web = 6.

Neben dem Replicated Service-Konzept in Docker gibt es das Konzept des Shared Services Global Service. Angenommen, ich möchte auf jedem Knoten des Clusters eine Instanz desselben Containers ausführen. In diesem Fall handelt es sich um einen Container der Prometheus-Webüberwachungsanwendung. Diese Anwendung wird verwendet, wenn Sie Metriken zum Betrieb von Hosts erfassen müssen. In diesem Fall verwenden Sie den Unterbefehl - - mode = global - - name = prom prom / Prometheus.



Infolgedessen wird die Prometheus-Anwendung ausnahmslos auf allen Knoten des Clusters gestartet. Wenn dem Cluster neue Knoten hinzugefügt werden, wird sie automatisch im Container und in diesen Knoten gestartet. Ich hoffe, Sie verstehen den Unterschied zwischen Replicated Service und Global Service. Normalerweise starten Sie den Replizierten Dienst.



Wir haben also die Grundkonzepte oder Grundentitäten von Docker untersucht und werden nun die Entitäten von Kubernetes betrachten. Kubernetes ist auch eine Art Planer, eine Plattform für die Container-Orchestrierung. Es muss beachtet werden, dass das Hauptkonzept des Schedulers darin besteht, zu wissen, wie Container auf verschiedenen Hosts geplant werden. Wenn Sie auf eine höhere Ebene wechseln, können Sie sagen, dass Orchestrierung die Erweiterung Ihrer Funktionen zum Verwalten von Clustern, zum Abrufen von Zertifikaten usw. bedeutet. In diesem Sinne sind sowohl Docker als auch Kubernetes Orchestrierungsplattformen, die beide über einen integrierten Scheduler verfügen.

Orchestrierung ist eine automatisierte Verwaltung verwandter Entitäten - Cluster virtueller Maschinen oder Container. Kubernetes ist eine Sammlung von Diensten, die einen Containercluster und seine Orchestrierung implementieren. Es ersetzt Docker nicht, erweitert jedoch seine Funktionen erheblich und vereinfacht die Verwaltung der Bereitstellung, des Netzwerkroutings, des Ressourcenverbrauchs, des Lastausgleichs und der Fehlertoleranz laufender Anwendungen.
Im Vergleich zu Kubernetes konzentriert sich Docker auf die Arbeit mit Containern und die Erstellung ihrer Bilder mithilfe einer Docker-Datei. Wenn wir die Docker- und Kubernetes-Objekte vergleichen, können wir sagen, dass Docker die Container verwaltet, während Kubernetes den Docker selbst verwaltet.
Wie viele von Ihnen haben sich mit Raketencontainern befasst? Verwendet jemand Rocket in der Produktion? Nur eine Person hob die Hand in der Halle, dies ist ein typisches Bild. Dies ist eine Alternative zu Docker, das in der Entwicklergemeinde noch nicht Fuß gefasst hat.

Das Kernstück von Kubernetes ist also Pod. Es handelt sich um eine verwandte Gruppe von Containern, die einen gemeinsamen Namespace, einen gemeinsamen Speicher und eine gemeinsame IP-Adresse verwenden. Alle Container im Herd kommunizieren über den lokalen Host miteinander. Dies bedeutet, dass Sie die Anwendung und die Datenbank nicht im selben Herd platzieren können. Sie müssen in verschiedenen Pods platziert werden, da sie unterschiedliche Skalierungsanforderungen haben.



So können Sie in einem Pod beispielsweise einen WildFly-Container, einen Anmeldecontainer, einen Proxy-Container oder einen Cache-Container platzieren, und Sie müssen sich verantwortungsbewusst der Zusammensetzung der Containerkomponenten nähern, die Sie skalieren möchten.

Normalerweise wickeln Sie Ihren Container in das Replikatset ein, da Sie eine bestimmte Anzahl von Instanzen im Herd ausführen möchten. Das Replikatset weist Sie an, so viele Replikate zu starten, wie der Docker-Skalierungsdienst erfordert, und gibt an, wann und wie dies zu tun ist.

Pods ähneln Containern in dem Sinne, dass ein Pod, der auf einem Host ausfällt, auf einem anderen Pod mit einer anderen IP-Adresse neu gestartet wird. Als Java-Entwickler wissen Sie, dass Sie sich beim Erstellen einer Java-Anwendung, die mit der Datenbank kommuniziert, nicht auf eine dynamische IP-Adresse verlassen können. In diesem Fall verwendet Kubernetes Service. Diese Komponente veröffentlicht die Anwendung als Netzwerkdienst, erstellt einen statischen permanenten Netzwerknamen für eine Reihe von Herden und verteilt gleichzeitig die Last zwischen den Herden. Es kann gesagt werden, dass dies der Dienstname der Datenbank ist und die Java-Anwendung nicht auf der IP-Adresse basiert, sondern nur mit dem Namen der Datenbankkonstante interagiert.

Dies wird durch die Tatsache erreicht, dass jeder Pod mit einem bestimmten Etikett geliefert wird, das im verteilten Speicher usw. gespeichert ist, und der Service diese Etiketten überwacht und eine Verbindung zwischen den Komponenten herstellt. Das heißt, Pods und Services interagieren mithilfe dieser Etiketten stabil miteinander.

Schauen wir uns nun an, wie Sie einen Kubernetes-Cluster erstellen. Dafür benötigen wir wie in Docker einen Masterknoten und einen Arbeitsknoten. Ein Knoten in einem Cluster wird normalerweise durch eine physische oder virtuelle Maschine dargestellt. Hier wie in Docker ist der Assistent eine zentrale Steuerungsstruktur, mit der Sie den gesamten Cluster über den Scheduler und den Controller-Manager steuern können. Standardmäßig ist ein Masterknoten im Singular vorhanden, es gibt jedoch viele neue Tools, mit denen Sie mehrere Masterknoten erstellen können.



Der Master-Knoten bietet Benutzerinteraktion über den API-Server und enthält den verteilten Speicher usw., der die Konfiguration des Clusters, den Status seiner Objekte und Metadaten enthält.

Worker-Node-Work-Nodes sind ausschließlich zum Starten von Containern vorgesehen. Dazu sind zwei Kubernetes-Dienste installiert - ein Proxy-Service-Netzwerkrouter und ein Kubelet-Scheduler-Agent. Während diese Knoten ausgeführt werden, überwacht Docker sie mit systemd (CentOS) oder monit (Debian), je nachdem, welches Betriebssystem Sie verwenden.

Betrachten Sie die Kubernetes-Architektur allgemeiner. Wir haben einen Master, der einen API-Server (Pods, Dienste usw.) enthält, der mit der CLI kubectl verwaltet wird. Mit Kubectl können Sie Kubernetes-Ressourcen erstellen. Es sendet Befehle an den API-Server, z. B. "Erstellen unter", "Dienst erstellen", "Replikate erstellen".

Weiter ist hier der Scheduler, der Controller Manager und das etcd-Repository. Nachdem der Controller-Manager die Anweisungen des API-Servers erhalten hat, ordnet er Replikatetiketten Herdetiketten zu, um eine stabile Interaktion zwischen Komponenten sicherzustellen. Der Scheduler, der die Aufgabe zum Erstellen unter erhalten hat, scannt die Arbeitsknoten und erstellt sie dort, wo sie bereitgestellt werden. Natürlich erhält er diese Informationen von etcd.

Als nächstes haben wir mehrere Arbeitsknoten, und der API-Server kommuniziert mit den darin enthaltenen Kubelet-Agenten und teilt ihnen mit, wie die Pods erstellt werden sollen. Hier ist der Proxy, mit dem Sie auf eine Anwendung zugreifen können, die diese Pods verwendet. Mein Client wird rechts auf der Folie angezeigt. Dies ist eine Internetanforderung, die an den Load Balancer gesendet wird. Er wendet sich an den Proxy, der die Anforderung entsprechend der Tonhöhe verteilt und die Antwort zurücksendet.



Sie sehen die letzte Folie, die den Kubernetes-Cluster und die Funktionsweise aller seiner Komponenten zeigt.

Lassen Sie uns mehr über Service Discovery und den Docker Load Balancer sprechen. Wenn Sie Ihre Java-Anwendung starten, geschieht dies normalerweise in mehreren Containern auf mehreren Hosts. Docker Compose enthält eine Komponente, mit der Anwendungen mit mehreren Containern einfach ausgeführt werden können. Es beschreibt Multicontainer-Anwendungen und startet sie mit einer oder mehreren Yaml-Konfigurationsdateien.



Standardmäßig sind dies die Dateien docker-compose.yaml und docker-compose.override.yaml, wobei mehrere Dateien mit - f angegeben werden. In der ersten Datei schreiben Sie den Dienst, Bilder, Replikate, Tags usw. Die zweite Datei wird zum Überschreiben der Konfiguration verwendet. Nach dem Erstellen von docker-compose.yaml wird es auf dem von Docker Swarm zuvor erstellten Multi-Host-Cluster bereitgestellt. Sie können eine grundlegende Konfigurationsdatei docker-compose.yaml erstellen, in der Sie bestimmte Konfigurationsdateien für verschiedene Aufgaben mit bestimmten Ports, Images usw. hinzufügen. Wir werden später darauf eingehen.

Auf dieser Folie sehen Sie ein einfaches Beispiel für eine Service Discovery-Datei. Die erste Zeile gibt die Version an, und Zeile 2 gibt an, dass es sich um die Datenbank- und Webdienste handelt.



Ich möchte, dass mein Web-Service nach dem „Heben“ mit dem DB-Service kommuniziert. Dies sind einfache Java-Anwendungen, die in WildFly-Containern bereitgestellt werden. In Zeile 11 schreibe ich die Umgebung couchbase_URI = db. Dies bedeutet, dass mein Datenbankdienst diese Datenbank verwendet. In Zeile 4 wird das Couchbase-Bild und in den Zeilen 5-9 bzw. 15-16 die Ports angezeigt, die für den Betrieb meiner Dienste erforderlich sind.

Der Schlüssel zum Verständnis des Service Discovery-Prozesses besteht darin, dass Sie eine Art Abhängigkeit erstellen. Sie geben an, dass der Webcontainer vor dem Datenbankcontainer gestartet werden soll, dies jedoch nur auf Containerebene. Wie Ihre Anwendung reagiert, wie sie startet - das sind ganz andere Dinge. Beispielsweise "steigt" der Container normalerweise in 3-4 Sekunden, der Start des Datenbankcontainers dauert jedoch viel länger. Daher sollte die Logik zum Starten Ihrer Anwendung in Ihrer Java-Anwendung "gebacken" werden. Das heißt, die Anwendung muss die Datenbank anpingen, um sicherzustellen, dass sie bereit ist. Da es sich bei der Couchbase-Datenbank um eine REST-API handelt, sollten Sie diese API aufrufen und fragen: „Hey, sind Sie bereit? Wenn ja, dann bin ich bereit, Ihnen Anfragen zu senden! “

Daher werden Abhängigkeiten auf Containerebene mithilfe des Docker-Compose-Dienstes ermittelt. Auf Anwendungsebene werden Abhängigkeiten und Lebensfähigkeit jedoch anhand von Verantwortlichkeitsumfragen ermittelt. Anschließend nehmen Sie die Datei docker-compose.yaml und stellen sie im Docker für mehrere Hosts mit dem Befehl docker stack deploy und dem Unterbefehl - - compose-file = docker-compose.yaml webapp bereit. Sie haben also einen großen Stapel, in dem es mehrere Dienste gibt, die mehrere Probleme lösen. Grundsätzlich sind dies die Aufgaben beim Starten von Containern.



Mal sehen, wie der Load Balancer funktioniert. Im obigen Beispiel habe ich mit dem Befehl docker service create einen Dienst erstellt - den WildFly-Container, der die Portnummer in der Form 8080: 8080 angibt. Dies bedeutet, dass Port 8080 auf dem Host - dem lokalen Computer - Port 8080 im Container zugeordnet wird, sodass Sie über localhost : 8080 auf die Anwendung zugreifen können . Dies ist der Zugangsport zu allen Arbeitsknoten.



Denken Sie daran, dass der Load Balancer hostorientiert und nicht containerorientiert ist. Es verwendet die Ports 8080 jedes Hosts, unabhängig davon, ob die Container auf dem Host ausgeführt werden oder nicht, da der Container jetzt auf einem Host ausgeführt wird und nach Abschluss der Aufgabe auf einen anderen Host übertragen werden kann.

Client-Anforderungen werden also vom Load Balancer empfangen, er leitet sie an einen der Hosts weiter. Wenn er mithilfe der IP-Adresstabelle mit einem nicht ausgeführten Container zum Host gelangt, leitet er die Anforderung automatisch an den Host weiter, auf dem der Container ausgeführt wird.

Ein einzelner Hop ist nicht teuer, aber in Bezug auf die Skalierung Ihrer Dienste völlig „nahtlos“. Dank dessen können Sie sicher sein, dass Ihre Anfrage genau an den Host geht, auf dem der Container ausgeführt wird.
