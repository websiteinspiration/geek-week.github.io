Proxmoxクラスターの操作：インストール、ネットワークセットアップ、ZFS、一般的な問題の解決

過去数年間、私はProxmoxクラスターと密接に連携してきました。多くのクライアントは、プロジェクトを開発できる独自のインフラストラクチャを必要としています。そのため、あなたが遭遇する可能性のある最も一般的な間違いや問題についてお話しすることができます。さらに、3つのノードのクラスターを最初から構成します。


Proxmoxクラスターは、2つ以上のサーバーで構成できます。クラスター内のノードの最大数は32個です。独自のクラスターは、マルチキャスト上の3つのノードで構成されます（記事では、一意性でクラスターを上げる方法についても説明します。たとえば、クラスターインフラストラクチャをHetznerまたはOVHに基づいている場合に重要です）。つまり、マルチキャストを使用すると、複数のノードに同時にデータを転送できます。マルチキャストでは、クラスター内のノード数について考えることはできません（上記の制限に焦点を当てています）。

クラスター自体は内部ネットワーク上に構築されます（IPアドレスが同じサブネット上にあることが重要です）。同じHetznerとOVHは、仮想スイッチ（Hetzner）とvRack（OVH）テクノロジーを使用して異なるデータセンターのノードをクラスター化できます-仮想スイッチについて記事でも話します。ホスティングプロバイダーで同様のテクノロジーが機能していない場合は、ProxmoxでネイティブにサポートされているOVS（Open Virtual Switch）を使用するか、VPNを使用できます。ただし、この場合は、少数のノードでユニキャストを使用することをお勧めします。このようなネットワークインフラストラクチャに基づいてクラスターが単に「バラバラ」になり、復元する必要がある場合がよくあります。したがって、私は仕事でOVHとHetznerを正確に使用するようにしています。そのようなインシデントはほとんど見られませんが、まず、ホスティングプロバイダーを調べます。ホストするもの：代替テクノロジーがあるかどうか、提供するソリューションがあるかどうか、マルチキャストをサポートするかどうかなど。

Proxmoxをインストールする


Proxmoxは、2つの方法でインストールできます。ISOインストーラーとシェル経由のインストールです。2番目の方法を選択するため、サーバーにDebianをインストールします。

各サーバーへのProxmoxのインストールに直接進みます。インストールは非常に簡単で、ここの公式ドキュメントで説明されています。

Proxmoxリポジトリとこのリポジトリのキーを追加します。

echo "deb http://download.proxmox.com/debian/pve stretch pve-no-subscription" > /etc/apt/sources.list.d/pve-install-repo.list
wget http://download.proxmox.com/debian/proxmox-ve-release-5.x.gpg -O /etc/apt/trusted.gpg.d/proxmox-ve-release-5.x.gpg
chmod +r /etc/apt/trusted.gpg.d/proxmox-ve-release-5.x.gpg  # optional, if you have a changed default umask

リポジトリとシステム自体の更新：

apt update && apt dist-upgrade

更新が成功したら、必要なProxmoxパッケージをインストールします。

apt install proxmox-ve postfix open-iscsi

注：Postfixとgrubはインストール中に構成されます-それらの1つが失敗する可能性があります。おそらく、これは、ホスト名が名前で解決されないために発生します。hostsエントリを編集してapt-get updateを実行するこれ

Nginxをインストールして証明書を暗号化しましょう


証明書とIPアドレスの状況が気に入らないので、NginxをインストールしてLet's Encrypt証明書を設定することをお勧めします。Nginxのインストールについては説明しません。Let's暗号化証明書を機能させるための重要なファイルのみを残します。

/etc/nginx/snippets/letsencrypt.conf
location ^~ /.well-known/acme-challenge/ {
  allow all;
  root /var/lib/letsencrypt/;
  default_type "text/plain";
  try_files $uri =404;
}



SSL証明書を発行するコマンド：

certbot certonly --agree-tos --email sos@livelinux.info --webroot -w /var/lib/letsencrypt/ -d proxmox1.domain.name

NGINXでのサイト構成
upstream proxmox1.domain.name  {
      server 127.0.0.1:8006;
}

server {
    listen 80;
    server_name proxmox1.domain.name;

    include snippets/letsencrypt.conf;
    return 301 https://$host$request_uri;
}


server {
    listen 443 ssl;

    server_name  proxmox1.domain.name;

    access_log  /var/log/nginx/proxmox1.domain.name.access.log;
    error_log  /var/log/nginx/proxmox1.domain.name.error.log;

    include snippets/letsencrypt.conf;

    ssl_certificate /etc/letsencrypt/live/proxmox1.domain.name/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/proxmox1.domain.name/privkey.pem;

    location / {
        proxy_pass  https://proxmox1.domain.name;
        proxy_next_upstream error timeout invalid_header http_500 http_502 http_503                                                                              http_504;
        proxy_redirect off;
        proxy_buffering off;
        proxy_set_header        Host            $host;
        proxy_set_header        X-Real-IP       $remote_addr;
        proxy_set_header        X-Forwarded-For $proxy_add_x_forwarded_for;
   
}


SSL証明書をインストールした後、cron経由で自動更新するように設定することを忘れないでください：

0 */12 * * * /usr/bin/certbot -a \! -d /run/systemd/system && perl -e 'sleep int(rand(3600))' && certbot -q renew --renew-hook "systemctl reload nginx"

いいね！これで、HTTPS経由でドメインにアクセスできます。

ネットワーク設定

クラスターに接続する前に、ハイパーバイザーでネットワークインターフェイスを構成します。残りのノードの構成は、IPアドレスとサーバー名を除いて同じであるため、それらの設定を複製しないことに注意してください。

仮想マシン（私のバージョンでは便宜上LXCコンテナーが存在します）が最初にハイパーバイザーの内部ネットワークに接続され、相互に対話できるように、内部ネットワーク用のネットワークブリッジを作成します。次に、少し後で、外部ネットワーク用のブリッジを追加して、仮想マシンが独自の外部IPアドレスを持つようにします。したがって、現時点ではコンテナはNAT'omの背後にあります。

Proxmoxのネットワーク構成を操作するには、Webインターフェースを使用する方法と、構成ファイル/ etc / network / interfacesを使用する方法の2つがあります。最初のオプションでは、サーバーを再起動する必要があります（または、interfaces.newファイルの名前をinterfacesに変更し、systemdを介してネットワークサービスを再起動することもできます）。構成を開始したばかりで、仮想マシンまたはLXCコンテナーがまだない場合は、変更後にハイパーバイザーを再始動することをお勧めします。

ZFSをインストールして構成する


ZFSは、Proxmoxで使用できるファイルシステムです。これにより、データを別のハイパーバイザーに複製したり、仮想マシン/ LXCコンテナーを移行したり、ホストシステムからLXCコンテナーにアクセスしたりすることができます。インストールは非常に簡単です。分析に進みましょう。私のサーバーでは3つのSSDが利用可能です。これらを組み合わせてRAIDアレイにします。

リポジトリを追加します。

nano /etc/apt/sources.list.d/stretch-backports.list

deb http://deb.debian.org/debian stretch-backports main contrib
deb-src http://deb.debian.org/debian stretch-backports main contrib

nano /etc/apt/preferences.d/90_zfs

Package: libnvpair1linux libuutil1linux libzfs2linux libzpool2linux spl-dkms zfs-dkms zfs-test zfsutils-linux zfsutils-linux-dev zfs-zed
Pin: release n=stretch-backports
Pin-Priority: 990

パッケージのリストを更新する：

apt update

必要な依存関係を設定します。 

 apt install --yes dpkg-dev linux-headers-$(uname -r) linux-image-amd64

ZFS自体をインストールします。 

apt-get install zfs-dkms zfsutils-linux

将来的にエラーfusermountが発生する場合：ヒューズデバイスが見つかりません。まず 'modprobe fuse'を試してから、次のコマンドを実行します。

modprobe fuse

他のドライブでも同様のアクションを実行する必要があります。すべてのディスクの準備ができたら、次のステップに進みます

。zpool create -f -o ashift = 12 rpool / dev / sda4 / dev / sdb4 / dev / sdc4

パフォーマンス上の理由から、ashift = 12を選択します-これはzfsonlinux自体の推奨事項です。詳細あなたは彼らのwikiでそれについて読むことができます：github.com/zfsonlinux/zfs/wiki/faq#performance-considerations

ZFSのいくつかの設定を適用しましょう：

zfs set atime=off rpool
zfs set compression=lz4 rpool
zfs set dedup=off rpool
zfs set snapdir=visible rpool
zfs set primarycache=all rpool
zfs set aclinherit=passthrough rpool
zfs inherit acltype rpool
zfs get -r acltype rpool
zfs get all rpool | grep compressratio

次に、zfs_arc_maxを計算するためにいくつかの変数を計算する必要があります。これは次のように行います。

mem =`free --giga | grep Mem | awk '{print $2}'`
partofmem=$(($mem/10))
echo $setzfscache > /sys/module/zfs/parameters/zfs_arc_max
grep c_max /proc/spl/kstat/zfs/arcstats

zfs create rpool/data
cat > /etc/modprobe.d/zfs.conf << EOL
options zfs zfs_arc_max=$setzfscache
EOL

echo $setzfscache > /sys/module/zfs/parameters/zfs_arc_max
grep c_max /proc/spl/kstat/zfs/arcstats

現時点では、プールは正常に作成されており、データサブプールも作成されています。zpool statusコマンドを使用して、プールのステータスを確認できます。このアクションは、すべてのハイパーバイザーで実行してから、次のステップに進む必要があります。

次に、ZFSをProxmoxに追加します。[ストレージ]セクションでデータセンターの設定（個別のノードではなく、データセンター）に移動し、[追加]ボタンをクリックして[ZFS]オプションを選択すると、次のパラメーターが表示されます

。ID：ストアの名前。local-zfs
ZFS Pool という名前を付けました。rpool/ dataを作成し、ここに追加します。
ノード：使用可能なすべてのノードを指定します

このコマンドは、選択したドライブで新しいプールを作成します。各ハイパーバイザーで、local-zfsと呼ばれる新しいストレージが表示されます。その後、仮想マシンをローカルストレージからZFSに移行できます。

インスタンスを隣接するハイパーバイザーに複製する


Proxmoxクラスターには、ハイパーバイザー間でデータを複製する機能があります。このオプションを使用すると、インスタンスをサーバー間で切り替えることができます。データは、最後の同期時に関連します。その時間は、レプリケーションの作成時に設定できます（15分が標準として設定されています）。インスタンスを別のProxmoxノードに移行するには、手動と自動の2つの方法があります。最初に手動オプションを見てみましょう。最後に、ハイパーバイザーの1つが利用できない場合に、アクセス可能なハイパーバイザー上に仮想マシンを作成できるようにするPythonスクリプトを紹介します。

「ホストキーの検証に失敗しました」エラー

クラスターを構成するときに、同様の問題が発生することがあります。これにより、マシンが移行および複製を作成できなくなり、クラスターソリューションの利点が失われます。このエラーを修正するには、known_hostsファイルを削除し、SSH経由で競合するノードに接続します。

/usr/bin/ssh -o 'HostKeyAlias=proxmox2' root@172.30.0.16

ホストキーを受け入れ、次のコマンドを入力してみてください。サーバーに接続します。

/usr/bin/ssh -o 'BatchMode=yes' -o 'HostKeyAlias=proxmox2' root@172.30.0.16

Hetznerのネットワーク設定の機能


ロボットパネルに移動し、[仮想スイッチ]ボタンをクリックします。次のページに、仮想スイッチインターフェースを作成および管理するためのパネルが表示されます。最初に作成し、次に専用サーバーをそれに接続する必要があります。検索で、接続に必要なサーバーを追加します。サーバーを再起動する必要はありません。仮想スイッチへの接続がアクティブになるまで10〜15分待つ必要があります。

Webパネルを介してサーバーを仮想スイッチに追加した後、サーバーに接続し、ネットワークインターフェイスの構成ファイルを開きます。ここで、新しいネットワークインターフェイスを作成します。

auto enp4s0.4000
iface enp4s0.4000 inet static
        address  10.1.0.11/24
        mtu 1400
        vlan-raw-device enp4s0

それが何であるかを詳しく見てみましょう。その中心には、これはenp4s0と呼ばれる単一の物理インターフェースに接続するVLANであり、VLAN番号が異なります-これは、Hetzner Robot Webパネルで作成した仮想スイッチ番号です。ローカルである限り、任意のアドレスを指定できます。

通常どおりにenp4s0を構成する必要があることに注意してください。実際には、物理​​サーバーに発行された外部IPアドレスが含まれているはずです。他のハイパーバイザーでこれらの手順を繰り返し、それらのネットワークサービスを再起動し、仮想スイッチのIPアドレスを使用して隣接ノードにpingします。 pingが成功した場合、仮想スイッチを使用してサーバー間の接続が正常に確立されています。

sysctl.conf構成ファイルも添付します。転送パッケージやその他のネットワークパラメーターに問題がある場合は、次のようにしてください。

net.ipv6.conf.all.disable_ipv6=0
net.ipv6.conf.default.disable_ipv6 = 0
net.ipv6.conf.all.forwarding=1
net.ipv4.conf.all.rp_filter=1
net.ipv4.tcp_syncookies=1
net.ipv4.ip_forward=1
net.ipv4.conf.all.send_redirects=0

ヘッツナーへのIPv4サブネットの追加

作業を開始する前に、ヘッツナーでサブネットを注文する必要があります。これは、ロボットパネルから行うことができます。

このサブネットからのアドレスを持つネットワークブリッジを作成します。構成例：

auto vmbr2
iface vmbr2 inet static
        address ip-address
        netmask 29
        bridge-ports none
        bridge-stp off
        bridge-fd 0

次に、Proxmoxの仮想マシンの設定に移動し、vmbr2ブリッジに接続される新しいネットワークインターフェイスを作成します。私はLXCコンテナーを使用しています。その構成はProxmoxですぐに変更できます。Debianの最終構成：

auto eth0
iface eth0 inet static
        address ip-address
        netmask 26
        gateway bridge-address

注：29ではなく26のマスクを指定しました。これは、ネットワークが仮想マシンで機能するために必要です。

HetznerへのIPv4アドレスの追加

単一のIPアドレスの状況は異なります。通常、Hetznerはサーバーサブネットから追加のアドレスを提供します。つまり、vmbr2の代わりにvmbr0を使用する必要がありますが、現時点ではそれがありません。つまり、vmbr0にはIronサーバーのIPアドレスが含まれている必要があります（つまり、物理ネットワークインターフェイスenp2s0を使用したアドレスを使用します）。アドレスはvmbr0に移動する必要があります。これには次の構成が適しています（KVMを注文することをお勧めします。この場合、ネットワーク操作を再開します）。

auto enp2s0
iface enp2s0 inet manual

auto vmbr0
iface vmbr0 inet static
        address  ip-address
        netmask  255.255.255.192
        gateway  ip-gateway
        bridge-ports enp2s0
        bridge-stp off
        bridge-fd 0

可能であればサーバーを再起動し（そうでない場合は、ネットワークサービスを再起動します）、IP aを介してネットワークインターフェイスを確認します。 

2: enp2s0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc pfifo_fast master vmbr0 state UP group default qlen 1000
    link/ether 44:8a:5b:2c:30:c2 brd ff:ff:ff:ff:ff:ff

ここを見るとわかるように、enp2s0はvmbr0に再割り当てされているため、vmbr0に接続され、IPアドレスがありません。

次に、仮想マシンの設定で、vmbr0に接続されるネットワークインターフェイスを追加します。ゲートウェイには、vmbr0に接続されているアドレスを指定します。
