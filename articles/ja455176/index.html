Reduxがなくても可能ですか？

今日、あなたは反応/還元が必要とされる多くのポジションを見つけることができます。反応は素晴らしく、質問はありません。 Reduxの問題は、それなしでそれが可能かどうかです。少しグーグルすると、Habréに関する確かな記事があり、著者が同じ質問をします。簡単な例（todoList）を使用する記事では、this.updateViews（）メソッドが頻繁に（7〜8回）呼び出されており、実行する方が簡単なようです。

ここでの主なアイデアは観察可能なモデルであり、reactは観察可能なものに対して責任があります。残っているのはモデルの作成だけです。

モデルを作成する前に、クライアントの設計（アーキテクチャ）について一言：

インデックス-生データ
履歴-配列[モデル]
オブザーバー-モデル
ビュー-エラー、フォーカス、フラグ

index.jsx-ユーザー画面のプログラムのエントリポイント。インデックスは、デフォルトのデータですべてのコンポーネントをレンダリングし、非同期クエリを作成し、新しいデータでコンポーネントを再描画します。

// index.jsx

<History>
  <Observer>
     <HeaderView />
     <MainView />
  </Observer>
</History>

Observer.jsxは、複数のビューのモデルの同期を担当します。たとえば、Petyaはオファーフォームに記入し、ページヘッダーにリアルタイムのプレビューを表示します。オブザーバーはモデルオブジェクトを格納し、子コンポーネントにapi：onModelChange（フィールド、値）を提供します。

History.jsxは、モデルオブジェクトのスタックで、api：コミットとロールバックを行います。

Model.jsは、ユーザーがペンで入力できるもの、つまり最も価値のあるものです。モデル内の他のデータを保存する必要はありません。モデルは、reactコンポーネントではなく、通常のjsクラスです。

class Model {
  constructor(other = {}) {} // copy constructor (and default too)
  isEqual(other) {} // operator ==
  less(other) {} // operator< 

  swap(other) {}
  hash() {}
  fieldNameConstrains() {} //see below please 
}

コピーコンストラクタは、少なくともHistoryに必要です。isEqualメソッドは、ポップアップの未保存の変更用です（状態のフラグよりはるかに便利です）。fieldNameConstrainsメソッドは依存フィールド用です。

大まかに言えば、依存フィールドがある場合は、それらを連続して変更する必要があります。

class Model { 
   // constrains
   // distance  <== velocity * time

   velocityConstrains(newVelocity) {
     this.velocity = newVelocity;
     this.distance = this.velocity * this.time; 
   }
   timeConstrains(newTime) { … } 

   distanceConstrains(newDistance) {
     this.distance = newDistance;
     this.time = this.distance / this.velocity;   
   }
}

個人的な経験から、model.field.onchangeのようなものは機能しません。これは、コピーコンストラクターを呼び出す必要があり、onchangeイベントがまったく必要ない場合があるためです。

検証 モデルで検証を行う必要はありません。ビューで実行する必要があります（ビューはユーザーの画面であり、モデル全体を画面に表示できるわけではないことを忘れないでください）。バリデーターは述語のセットです。検証には2つのアルゴリズムしかありません。1）フォーム内のすべてのエラーを見つけるか、2）最初のエラーを見つけます。例えば、

アクセス権。ここでの主なことは、継承を使用せずにとどまることです。アイデアは、モデルにすべてのフィールドが含まれていて、ロールのためにフィールドをトリミングすることです。つまり、これはホワイトリストであり、モデルの残りのフィールドはデフォルトで残ります。検証のために1つのステップが追加されます-検証オブジェクトのプロジェクションを作成します（これもマッパーです。上記を参照）。つまり、必要なフィールドのみを検証します。

生産について。このアプローチは1年前から運用されています。これは、バナーを含む広告キャンペーンを作成するためのインターフェースです。さまざまな複雑さのフォーム-1つのモデルから画面まで、さまざまなタイプの多くのモデルで終わります。ここで、バックエンドがネストされた構造を送信することを好むことを追加できます。内気である必要はなく、ビューにフラット構造のみを保存する必要があります。

isEqualモデルメソッドについて。 utils.jsのどこかにisEqualメソッドとisEqualArrayメソッドがあります。

  function isEqual(left, right) {
     return left.isEqual(right); 
  }
  isEqualArray(v1, v2) {
    if (v1.length !== v2.length) { return false }
    for (var k = 0; k != v1.length; k++) {
      if (!isEqual(v1[k], v2[k])) { return false; }
    }
    return true;   
  }

モデルをネストしないようにする必要があります。モデルはデータ構造ではなく、ユーザーデータであることを忘れないでください。
