Typescript Compiler API: Take control of the compiler





Typescript application development always has a project build phase. Usually, build systems and workflow automation are used for this, such as webpack or gulp, hung with enough plug-ins, or the build process is spread in package.json commands and shell scripts using native tsc or the CLI commands used in the framework of the project. All these solutions have their pros and cons. Often during the assembly process, something non-standard needs to be done, and it turns out that the assembly system used does not provide the necessary functionality from the box, and the existing plug-ins do not quite what is needed. At such moments, work on the project gets up, and frantic picking in the configs begins and the search for a suitable plug-in. At some point you realize that in the time spent searching for a suitable crutch,You could write your decision.



In many cases, critical processes in a project can be automated with javascript scripts, the expressiveness and functionality of which makes it possible to describe the desired workflow and choose from a wide variety of libraries without bothering with the presence of plug-ins for them for a specific build system. An important advantage of this approach is complete control over the processes and maximum flexibility. For projects that use Typescript as the main development language, the question arises of how to embed the process of compiling it into your workflow. The Typescript Compiler API comes to the rescue here. In this article, we will see how it can be used to compile a project implemented on Typescript, interacting with the compiler at different stages of its work, and write a script for the hot-reloading of the REST server,developed by Nest.js.



I have a project on a node that I work on in my free time. Initially, it was intended for a small business that never took off, but now I use it as a training ground for my experiments. The project, which was originally built on Nuxt.js in conjunction with Fastify, has gone through many transformations. Here and the transition to typescript and conversion to a mono-repository, and replacing Fastify with Nest.js. Since the project was not supposed to have a small and serious load, all this was supposed to be spinning on the same server, so I used a single Express instance for both the REST server and the front end via Nuxt.js. For this, Express, which runs under the hood of Nest.js, gets the render function of Nuxt.js.



This solution works well in production, but since I was working on a project alone, it was convenient to write API functions and make an interface in parallel. For this, restarting Nest.js when changing files should not lead to a restart of Nuxt.js operating in the HMR mode, otherwise it would start the build process anew, which would make no sense. But here a problem arose, the fact is that although there is a watch mode in CLI Nest.js that can do hot-reload, it loads it in a separate process that restarts when the project is recompiled. Because of this, each restart of the server in this mode will lead to a loss of context for Nuxt and then its full assembly will start.



After all this became clear, I decided to make my own hot-reload in the project. Initially, I tried to implement the solution using webpack, as suggested in the Nest.js documentation , but it turned out that the start-server-webpack-plugin, which is used there to implement the restart, used the same strategy as the CLI, i.e. ran it in a separate process and restarted as needed. I rewrote this plugin so that it works as I need, but using Webpack the solution turned out to be rather difficult and not without problems that surfaced here and there, making it difficult to focus on the development of the project.



Then I decided that I needed to change the approach, I wanted to directly use the mechanisms laid down in the tsc compiler, which always worked like clockwork quickly and stably. I ended up findingA good article that showed examples of using the TypeScript Compiler API. It turned out that the tools built into the Typescript library allow me to implement the functions I need and are quite convenient to use. Unfortunately, I came across that there is very little information on the Typescript Compiler API, so I decided to share what I managed to find out. I also note that this API at the time of writing is under development and some points may change over time.



Simple compilation



To understand the principles of the TypeScript Compiler API, for starters, let's just try to compile the project with it. The project on which I will experiment has a standard structure for nest.js: the sources are located in the src folder, the entry point is the ./src/main.ts file.



In order for the compiler to successfully compile the project, it needs to pass compilation parameters to it. The easiest way is to parse it right in the script. Create the tsconfig.js file: 



const {ModuleResolutionKind, ModuleKind, ScriptTarget} = require("typescript")

module.exports =  {
    moduleResolution: ModuleResolutionKind.NodeJs,
    module: ModuleKind.CommonJS,
    target: ScriptTarget.ES2019,
    declaration: true,
    removeComments: true,
    emitDecoratorMetadata: true,
    experimentalDecorators: true,
    sourceMap: true,
    outDir: "./dist",
    baseUrl: "./",
    allowJs: true,
    skipLibCheck: true,
  }


Please note that some parameters, such as moduleResolution, module, target, are defined through enumerations, so it will not work directly to pick up parameters from tsconfig.json, where they are written as strings, so I won‚Äôt use it to implement the simplest example. How to correctly pick up the parameters from tsconfig.json we will understand a little later.



Now we have the compilation options and you can go to the next step. To do this, create a ts-compiler.js file with the following contents: 



const ts = require('typescript');

function compile() {
  const compilerOptions = require('./tsconfig');
  const program = ts.createProgram(['./src/main.ts'], compilerOptions);
  const emitResult = program.emit();
}

compile();


If you now run this script for execution, the compiled program files should appear in the ./dist directory. Compilation takes place in two stages: first, the createProgram command analyzes the source files and creates a list of files for compilation, after which we compile and save the compiled files using the emit function. Pretty simple.



To improve usability, we now need to implement the loading of compilation options from the tsconfig.json file. Add the following function to the script: 




const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function getTSConfig() {
  const configPath = ts.findConfigFile('./', ts.sys.fileExists, 'tsconfig.json');
  const readConfigFileResult = ts.readConfigFile(configPath, ts.sys.readFile);
  if (readConfigFileResult.error) {
    throw new Error(ts.formatDiagnostic(readConfigFileResult.error, formatHost));
  }
  const jsonConfig = readConfigFileResult.config;
  const convertResult = ts.convertCompilerOptionsFromJson(jsonConfig.compilerOptions, './');
  if (convertResult.errors && convertResult.errors.length > 0) {
    throw new Error(ts.formatDiagnostics(convertResult.errors, formatHost));
  }
  const compilerOptions = convertResult.options;
  return compilerOptions;
}


To work with configuration files, the API provides a number of functions that allow you to find, load and convert the configuration to the form that the compiler understands. Here, the findConfigFile function is used to determine the path to the configuration file, then it is loaded as json using readConfigFile, and then, if there are no errors, using convertCompilerOptionsFromJson we obtain compilation parameters.



If something went wrong while loading the config, our function throws exceptions using the error message transfer mechanisms embedded in the API. The library returns all error messages and warnings as objects of the Diagnostic class. In order to form a message on their basis for output to the console, you can use the standard library formatting functions, in this case formatDiagnostic for one message and formatDiagnostics for an array. For the formatting functions to work correctly, a formatHost object is required, which contains parameters important for formatting the message.



Now, to get the compiler options, we can call the getTSConfig function in our script.

At this stage, if the compiler finds errors in the code, we will not know about it, let's fix it. All errors in the code that were detected at the compilation stage, the emit function returns in the diagnostcs field of the EmitResult object. To beautifully display errors in the code, you can use the formatDiagnosticsWithColorAndContext function. It also makes sense to check for errors detected during the program creation phase. To do this, use the getPreEmitDiagnostics function. We supplement our script as follows: 



function compile() {
  const compilerOptions = getTSConfig();
  const program = ts.createProgram(['./src/main.ts'], compilerOptions);
  console.log(
    ts.formatDiagnosticsWithColorAndContext(
      ts.getPreEmitDiagnostics(program),
      formatHost,
    ),
  );
  const emitResult = program.emit();
  console.log(
    ts.formatDiagnosticsWithColorAndContext(emitResult.diagnostics, formatHost),
  );
  return emitResult.diagnostics.length === 0;
}

So, we got a script that will compile our sources and display errors in the console if they occur. As you can see, this approach makes it possible to implement the compilation process very flexibly, since we have access to all stages and results of compilation. Additional flexibility can be achieved by customizing CompilerHost, which contains a set of functions necessary for the compiler. Suppose we wanted to display in the console the paths to the files that the compiler reads in the process. This is easy to do by passing the custom CompilerHost to the createProgram function. Let's see how it works in practice.



I don‚Äôt want to clog the output with a footcloth from hundreds of lines of file paths, therefore I will write a function that will output the paths in one line: 



function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
  };
  return displayFunction;
}


Now let's add the compile () function code. 



function compile() {
  const compilerOptions = getTSConfig();
  const compilerHost = ts.createCompilerHost(compilerOptions);
  compilerHost.readFile = displayFilename(compilerHost.readFile, 'Reading');
  compilerHost.readFile.enableDisplay();
  const program = ts.createProgram(
    ['./src/main.ts'],
    compilerOptions,
    compilerHost,
  );
  compilerHost.readFile.disableDisplay();

  console.log(
    ts.formatDiagnosticsWithColorAndContext(
      ts.getPreEmitDiagnostics(program),
      formatHost,
    ),
  );

  compilerHost.writeFile = displayFilename(compilerHost.writeFile, 'Emitting');
  compilerHost.writeFile.enableDisplay()
  const emitResult = program.emit();
  compilerHost.writeFile.disableDisplay();
  console.log(
    ts.formatDiagnosticsWithColorAndContext(emitResult.diagnostics, formatHost),
  );
  return emitResult.diagnostics.length === 0;
}


So, now we create the host using the createCompilerHost function and replace with our implementation the readFile and writeFile functions. The compile () function also returns true if no errors are found after the end of the operation. Now, if the compilation went smoothly, you can immediately start the compiled server. DIY Incremental program watcher



In the previous section, we looked at how the TypeScript Compiler API can be used to customize the Typescript compilation process. Now it's time to do something more urgent and useful, which can really increase the speed and ease of development. During compilation of the program, the compiler needs to resolve all dependencies in the program.



To do this, he sequentially reads and analyzes project files and library files. Even to compile a relatively small project like mine at the preparation stage, the compiler needs to read, parse and analyze more than 2000 files. This leads to a significant slowdown in the compilation process, for example, on my laptop compilation takes about 20 seconds. At the same time, during development, usually only a few files change between compiler launches, so it‚Äôs logical that starting from version 2.7, Typescript has the option of incremental compilation, which, together with the watch mode, which allows you to monitor changes to files on the disk, significantly increased the speed of recompilation.



These functions are also used by CLI Nest.js, and you can start working in this mode using the command 



nest start --watch


Another opportunity to take advantage of this development mode is to run tsc with the --watch switch. But, as I already wrote in the introduction, if the capabilities of standard commands at some point cease to be enough, it's time to look under the hood of Typescript and find out how it works there.



To begin with, as in the previous case, create ts-wather.js with the following contents: 



const ts = require('typescript');

const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

async function watchMain() {
  const configPath = ts.findConfigFile(
    './',
    ts.sys.fileExists,
    'tsconfig.json',
  );

  const host = ts.createWatchCompilerHost(
    configPath,
    {},
    ts.sys,
    ts.createEmitAndSemanticDiagnosticsBuilderProgram,
    diagnostic =>
      console.log(
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
    diagnostic =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );
  ts.createWatchProgram(host);
}
watchMain();

And our profit is that we can now do different things with the help of hooks, which will greatly facilitate our lives.



In order to make it convenient for us to hook different things in the host, let's write a small helper: 



function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function() {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}


It will allow us to perform the necessary actions before and after the called function in the host, while receiving the arguments that are passed to it and the result of the execution. Now this helper can be used to conveniently interfere with the compiler at different stages. For example, let's add messages to the console in preparation for compilation:

Let's figure out what's going on here. Firstly, we need the app variable, in which we write the result of the bootstrap function, in order to correctly close the server after changing the code and starting recompilation. It will contain a handler for calling the server shutdown command. Secondly, the server starts up only if no errors were found in the code during the compilation process. To do this, we check with getSemanticDiagnostics that the number of diagnostics is zero. And thirdly - since the modular system caches loaded modules, after making changes to the code, we need to clean the cache from dependencies. To do this, I wrote a small helper function clearCache, which we previously call. 



function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      );
    });
}


Here the cache is cleared of dependencies that may prevent us from correctly restarting the server. In our case, we must remove all modules of our project from the dist directory from the cache. I also delete typeorm modules, because the features of their implementation prevent the server from restarting (why this happens is another story). Using the timeout, we wait until the server is closed and the app variable is cleared. const ts = require('typescript');
const { join, dirname } = require('path');
const { exit } = require('process');

const formatHost = {
  getCanonicalFileName: (path) => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function () {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}

function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      process.stdout.clearLine(); // clear current text
      process.stdout.cursorTo(0); // move cursor to beginning of line
      process.stdout.write(
        `Clearing cache ${Math.floor((index * 100) / arr.length + 1)}%`,
      );
    });
  process.stdout.write(' finished.\n');
}

function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  let counter = 0;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    counter++;
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    counter = 0;
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
    console.log(`${counter} times function was called`);
  };
  return displayFunction;
}
