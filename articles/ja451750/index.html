Get best of the week

本「Elasticsearch、Kibana、Logstash、および次世代検索エンジン」

こんにちは、habrozhiteli！私たちは、Elastic Stackに関する本を出版しました。これは、大量のデータを処理し、あらゆるソースからあらゆる形式で確実にそれらを抽出し、データをリアルタイムで検索、分析、視覚化したいプロフェッショナル向けに設計されています。この本は、分散コンピューティングとリアルタイムデータ処理の分野におけるElastic Stackの作業の基本的な理解が必要な場合に役立ちます。

以下に、集約に関する一節についての本の構造を示します。

本の構造


第1章「Elastic Stackの概要」では、Elastic Stackの主要コンポーネントを紹介し、全体的な構造におけるそれらの役割を説明し、各コンポーネントの目的を説明します。この章では、Elasticsearchを使用して実現される分散型のスケーラブルな検索と分析の必要性についても説明します。最後に、ElasticsearchとKibanaをダウンロードしてインストールするためのガイドがあるので、これらのツールを使い始めることができます。

第2章「Elasticsearchの使用開始」では、Elastic Stackの基盤であるElasticsearch検索エンジンの主要な原則を紹介します。インデックス、タイプ、ノード、クラスターなどの概念に慣れます。また、REST APIを使用して基本的な操作を実行する方法についても学習します。

第3章「検索は重要」では、Elasticsearchが提供する検索方法に焦点を当てます。テキスト分析の基本、トークナイザー、アナライザー、関連検索の機能について学びます。この章では、関連する検索の実際的な例も示します。

第4章「Elasticsearchを使用したデータの分析」では、さまざまなタイプの集計について説明します。データ分析の原則をよりよく理解するのに役立つ例が含まれています。単純なものから複雑なものまで、さまざまなタイプの集計の使用方法を学び、巨大なデータセットをナビゲートします。この章を読むと、いつどの集約オプションを使用するのが適切かがわかります。

第5章「ログ分析」では、Logstashを使用する必要性、そのアーキテクチャ、インストール、および構成について説明します。 Elastic 5は、Logstash構成を置き換えることができる取り込みノードツールを提供します。この章を読むことで、Elastic Ingestノードを使用してコンテナーを作成する方法を学びます。

第6章「Logstashを使用したコンテナの設計」では、Logstashの基本について説明します。これにより、さまざまなソースからのデータを動的に識別し、選択したフィルターを使用してデータを正規化できます。幅広いフィルターを使用することで、コードを記述せずに、他のリアルタイムおよびほぼリアルタイムのストリーミング処理フレームワークと同等のLogstashを実現する方法を学びます。また、リモートマシンからログファイル（ログファイル）を転送するために使用されるBeatsプラットフォームとFileBeatコンポーネントについても学びます。

第7章「Kibanaでのデータの視覚化」では、Kibanaを使用してデータを視覚化し、印象的に提示する方法を示します。例として単純なデータセットを使用して、数回のクリックで視覚化を作成する方法について説明します。

第8章「Elastic X-Pack」では、Elasticsearch拡張について説明します。このときまでに、Elasticsearchとそのデータコンテナーを作成するための主要コンポーネントを学習し、拡張機能を接続して特定の問題を解決できるようになります。この章では、Elastic StackにX-Packコンポーネントをインストールして構成する方法を読み、セキュリティとモニタリングの基本を学び、さまざまな通知を追加する方法を学びます。

第9章「Elastic Stackを稼働させる」では、Elastic Stack Complexを商用運用にするための推奨事項を示します。アプリケーションの実装方法に関する推奨事項を受け取り、操作の要件に応じてデフォルト設定を変更します。また、Elastic Cloudクラウドサービスの使用方法についても学習します。

第10章「センサーからのデータを分析するためのアプリケーションの作成」では、さまざまなソースからのデータを分析および処理するためのアプリケーションの作成について説明します。 Elasticsearchでデータをモデル化し、データコンテナーを作成してKibanaで視覚化する方法を学習します。また、X-Packコンポーネントを効果的に使用してコンテナーのセキュリティと監視を確保し、さまざまなイベントの通知を受信する方法についても学びます。

第11章「サーバーインフラストラクチャの監視」では、Elastic Stackを使用して、完全にElastic Stackで作成されたサーバーとアプリケーションのリアルタイム監視を構成する可能性を示します。 Beatsプラットフォームの別のコンポーネントであるサーバー/アプリケーションの監視に使用されるMetricbeatについて理解します。

合計値、平均値、最大値、最小値の集計


フィールドの合計、最小値または最大値、または平均を見つけることは、かなり一般的な操作です。SQLでは、合計を計算するクエリは次のとおりです。

SELECT sum(downloadTotal) FROM usageReport;

したがって、テーブル内のすべてのエントリのdownloadTotalフィールドの合計が計算されます。これを行うには、テーブル内のすべてのレコードまたは選択したコンテキスト内のすべてのレコードを調べ、選択したフィールドの値を追加します。

Elasticsearchでは、合計集計を使用して同様のクエリを記述できます。

金額集計


金額の単純な集計を記述する方法は次のとおりです。

GET bigginsight/_search
{
    "aggregations": {                        1
       "download_sum": {                     2
           "sum": {                          3
              "field": "downloadTotal"       4
           }
       }
    },
    "size": 0                                5
}

  1. トップレベルの集計または集計は、集計のラッパーとして機能する必要があります。
  2. 集計に名前を付けます。この場合、downloadTotalフィールドで金額を集計し、適切な名前download_sumを選択します。あなたは彼女を好きなように呼ぶことができます。このフィールドは、応答結果でこの特定の集計を見つける必要がある場合に役立ちます。
  3. 合計の集計を行うため、合計要素が適用されます。
  4. downloadTotalフィールドに用語を集計します。
  5. 生の結果が応答に入らないようにするには、size = 0を指定します。検索結果ではなく、集計結果のみが必要です。高レベルのクエリ要素を指定しなかったため、クエリはすべてのドキュメントで機能します。応答に未処理のドキュメント（または検索結果）は必要ありません。

答えは次のようになります。

{
   "took": 92,
   ...
   "hits": {
      "total": 242836,                 1
      "max_score": 0,
      "hits": []
   },
   "aggregations": {                   2
      "download_sum": {                3
         "value": 2197438700           4
      }
   }
}

答えの主なパラメータを理解します。

  1. hits.total要素は、リクエストコンテキストに一致するドキュメントの数を示します。追加のクエリまたはフィルターが指定されていない場合、タイプまたはインデックス内のすべてのドキュメントが含まれます。
  2. リクエストと同様に、この回答は集約内に配置され、このフォームで表示されます。
  3. リクエストした集計の応答はdownload_sumと呼ばれるため、同じ名前の要素内の金額の集計から回答が得られます。
  4. 実際の値は、集計金額を適用した後に表示されます。

平均値、最大値、最小値の集計は非常に似ています。それらについて簡単に検討します。

平均集計


平均の集計は、リクエストのコンテキストですべてのドキュメントの平均値を見つけます。

GET bigginsight/_search
{
    "aggregations": {
       "download_average": { 1
          "avg": { 2
             "field": "downloadTotal"
          }
       }
    },
    "size": 0
}

金額の集計との顕著な違いは以下の通りです。

  1. この集計が平均値を計算するように設計されていることは明らかであるため、download_averageという別の名前を選択しました。
  2. 実行される集約のタイプは、前の例のように、合計ではなく平均です。

応答の構造は前のサブセクションの回答と同じですが、値フィールドには要求されたフィールドの平均値が表示されます。

最小値と最大値の集計は似ています。

最小集計


インデックス/タイプ全体で、downloadTotalフィールドの最小値を見つけます。

GET bigginsight/_search
{
    "aggregations": {
       "download_min": {
          "min": {
              "field": "downloadTotal"
          }
       }
    },
    "size": 0
}

最大値の集計


インデックス/タイプ全体で、downloadTotalフィールドの最大値を見つけます。

GET bigginsight/_search
{
     "aggregations": {
        "download_max": {
           "max": {
               "field": "downloadTotal"
           }
        }
     },
     "size": 0
}

これらは非常に単純な集計です。次に、より複雑な統計の集計と高度な統計を見てみましょう。

統計と高度な統計の集計


これらの集計は、単一のクエリ内で、追加のクエリを実行せずに、いくつかの一般的な統計値を計算します。統計は一度に計算され、何度も要求されないため、Elasticsearchリソースは節約されます。このようなデータのいくつかのタイプに関心がある場合は、クライアントコードも簡単になります。統計の集計の例を見てください。

統計集約


統計の集計は、一度に合計、平均、最大、最小値、およびドキュメントの総数を計算します。

GET bigginsight/_search
{
    "aggregations": {
       "download_stats": {
          "stats": {
             "field": "downloadTotal"
         }
       }
    },
    "size": 0
}

構造に関する統計のリクエストは、すでに知っている他のメトリック集計と同様です。ここでは特別なことは何も起こりません。

答えは次のようになります。

{
   "took": 4,
   ...,
   "hits": {
      "total": 242836,
      "max_score": 0,
      "hits": []
   },
   "aggregations": {
      "download_stats": {
         "count": 242835,
         "min": 0,
         "max": 241213,
         "avg": 9049.102065188297,
         "sum": 2197438700
      }
   }
}

ご覧のとおり、download_stats要素を含む回答には、合計、最小、最大、平均、合計が含まれています。この結論は、リクエストの数を減らし、クライアントコードを簡素化するため、非常に便利です。

高度な統計の集計を見てください。

高度な統計の集計


拡張統計の集約により、前のオプションに加えて、もう少し統計が返されます。

GET bigginsight/_search
{
    "aggregations": {
       "download_estats": {
          "extended_stats": {
             "field": "downloadTotal"
          }
       }
    },
    "size": 0
}

答えは次のようになります。

{
   "took": 15,
   "timed_out": false,
   ...,
   "hits": {
      "total": 242836,
      "max_score": 0,
      "hits": []
   },
   "aggregations": {
      "download_estats": {
         "count": 242835,
         "min": 0,
         "max": 241213,
         "avg": 9049.102065188297,
         "sum": 2197438700,
         "sum_of_squares": 133545882701698,
         "variance": 468058704.9782911,
         "std_deviation": 21634.664429528162,
         "std_deviation_bounds": {
            "upper": 52318.43092424462,
            "lower": -34220.22679386803
         }
      }
   }
}

答えでは、平方和、不一致、標準偏差とその境界も取得します。

電力集約


一意のアイテムのカウントは、電力集約を使用して行うことができます。これは、次に示すように、クエリ結果の検索に似ています。

select count(*) from (select distinct username from usageReport) u;

特定のフィールドの一意の値のパワーまたは数を決定することは、かなり一般的なタスクです。たとえば、サイトへのさまざまな訪問者からのクリックストリーム（クリックストリーム）がある場合、選択した日、週、または月にサイトに滞在しているユニークユーザーの数を調べることができます。

利用可能なネットワークトラフィックデータを使用してユニークユーザーの数を見つける方法を考えてみましょう。

GET bigginsight/_search
{
     "aggregations": {
        "unique_visitors": {
            "cardinality": {
               "field": "username"
            }
        }
     },
     "size": 0
}

電力集約応答は、他のメトリック集約と同じように見えます。

{
   "took": 110,
   ...,
   "hits": {
      "total": 242836,
      "max_score": 0,
      "hits": []
   },
   "aggregations": {
      "unique_visitors": {
         "value": 79
      }
   }
}

最も単純なタイプの集約を理解したので、いくつかのセグメント集約を検討できます。
