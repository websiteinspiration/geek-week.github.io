Zabbix-マクロ枠の拡大

クライアントのソリューションを作成する際、美しく、通常のZabbix機能で解決したい2つのタスクがありました。

タスク1. Mikrotikルーターの現在のファームウェアバージョンを追跡する。

HTTPテンプレートにエージェントを追加することで、タスクは簡単に解決されます。エージェントはMikrotik Webサイトから現在のバージョンを受信し、トリガーは現在のバージョンを現在のバージョンと比較し、不一致の場合はアラートを発行します。

10台のルーターがある場合、そのようなアルゴリズムは重要ではありませんが、3000台のルーターをどうするか？サーバーに3000リクエストを送信しますか？もちろん、そのようなスキームは機能しますが、3000クエリのアイデアは私には合いませんでした。別の解決策を見つけたかったのです。さらに、このようなアルゴリズムには欠点がありました。相手側は、DoS攻撃に対して1つのIPからそのような数の要求を計算でき、単純に禁止できます。

タスク2さまざまなHTTPエージェントでの承認セッションの使用。

HTTPエージェントを介して「閉じた」ページから情報を受信する必要がある場合は、認証Cookieが必要です。このため、通常はログイン/パスワードのペアとCookieにセッションIDを設定する標準の承認フォームがあります。

しかし、問題があります。HTTPエージェントの1つのアイテムから別のアイテムのデータにアクセスして、ヘッダーのこの値を置き換えることはできません。

「Webスクリプト」もあります。これには別の制限があり、分析用のコンテンツを取得してさらに保存することはできません。ページで必要な変数の可用性を確認するか、Webスクリプトのステップ間で以前に取得した変数を渡すことができます。

これらのタスクについて少し検討した後、監視システムのすべての部分（テンプレート、ホスト、トリガー、アイテム）で完全に表示されるマクロを使用することにしました。また、マクロはWebインターフェイスAPIを介して更新できます。

Zabbixには、優れた詳細なAPIドキュメントがあります。API経由でデータを交換するには、Jsonデータ形式を使用します。詳細は公式ドキュメントに記載されています。

必要なデータを取得してマクロに記録するための一連のアクションを、次の図に示します。

私の例では、最初のステップは、最初のタスクのPBXで認証Cookieを取得することでした。2番目のタスクでは、現在のMikrotikファームウェアバージョンの番号を取得しました。

ステップ2


次のステップに進みます。承認セッションを取得する：

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc": "2.0",
    "method": "user.login",
    "params": {
        "user": "Admin"
        "password": "zabbix"
    },
    "id": 1,
    "auth": null
}

jsonrpcは、使用されるJSON-RPCプロトコルのバージョンです。
ZabbixはJSON-RPCバージョン2.0を実装しています。

  • method-呼び出されるメソッド。
  • params-メソッドによって渡されるパラメーター。
  • id-任意のリクエスト識別子。
  • auth-ユーザー認証キー。まだ持っていないように、nullに設定します。

APIを使用するために、権限が制限された別のアカウントを作成しました。まず、不要な場所へのアクセスを許可する必要はありません。次に、バージョン5.0より前のバージョンでは、マクロで指定されたパスワードを読み取ることができました。したがって、Zabbixの管理者パスワードを使用すると、管理者アカウントが盗みやすくなります。

これは、サードパーティのスクリプトを介してAPIを操作し、資格情報を保存する場合に特に当てはまります。

バージョン5.0以降、マクロに保存されたパスワードを非表示にするオプションが登場しました。



APIを介してデータを更新するための別のアカウントを作成するときは、必要なデータがWebインターフェースを介して利用できるかどうか、およびそれを更新できるかどうかを確認してください。私はチェックしませんでしたが、長い間、必要なマクロがAPIに表示されない理由を理解できませんでした。



APIで承認を受けた後、マクロのリストを取得します。

ステップ3


このAPIでは、ホストマクロを名前で更新することはできません。これには、まずマクロIDを取得する必要があります。さらに、特定のホストのマクロのリストを取得するには、このホストのIDを調べる必要があり、これは追加の要求です。通常のマクロ{HOST.ID}をリクエストで使用することはできません。制限により、これを回避することにし



ました。このホストのIDを使用してローカルマクロを作成しました。WebインターフェイスからホストIDを見つけるのは非常に簡単です。

特定のホストのすべてのマクロのリストを含む応答は、テンプレートでフィルタリングできます。

regex:{"hostmacroid":"([0-9]+)"[A-z0-9,":]+"{\$MIKROTIK_VERSION}"



したがって、必要なマクロのIDを取得します。MIKROTIK_VERSIONは、探しているマクロの名前です。私の場合、ホストに割り当てられているMIKROTIK_VERSIONマクロを検索します。

リクエスト自体は次のようになります。

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.get",
    "params":{
        "output":"extend",
        "hostids":"{$HOST_ID}"
    },
    "auth":"{sid}",
    "id":1
}

変数{sid}は 2番目のステップで取得され、APIインターフェースを操作する必要がある場合に常に使用されます。

最終4ステップ-マクロの更新


これで、更新が必要なマクロID、承認Cookie、またはルーターのファームウェアバージョンがわかりました。マクロ自体を更新できます。

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.update",
    "params":{
        "hostmacroid":"{hostmacroid}",
        "value":"{mikrotik_version}"
    },
    "auth":"{sid}",
    "id":1
}

{mikrotik_version} -最初のステップで取得した値。私の例では-現在のファームウェアmikrotik
{hostmacroid}のバージョン-値は3番目のステップで取得されました-更新するマクロID。
