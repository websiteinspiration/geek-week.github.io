ロードバランサは、Webアーキテクチャで重要な役割を果たします。負荷を複数のバックエンドに分散できるため、スケーラビリティが向上します。また、いくつかのバックエンドが構成されているため、1つのサーバーで障害が発生した場合、バランサーは別の稼働中のサーバーを選択できるため、サービスの可用性が高くなります。

NGINXのようなプロのバランサーを使って、シンプルなバランサーを作って楽しみました。私はGoでそれを書いた、それは完全な並列処理をサポートする現代の言語です。Goの標準ライブラリには多くの機能があり、少ないコードで高性能アプリケーションを作成できます。さらに、配布を容易にするために、静的にリンクされた単一のバイナリを生成します。

バランサーのしくみ


さまざまなアルゴリズムを使用して、バックエンド間で負荷を分散します。例えば：

  • 最小接続-負荷は、アクティブな接続の数が最も少ないサーバーに分散されます。

バランサーでは、最も単純なアルゴリズムであるラウンドロビンを実装します。

サーバー選択プロセス


次のサーバー選択時に、横になっているサーバーをスキップする必要があります。しかし、カウントを整理する必要があります。

多数のクライアントがバランサーに接続し、各クライアントが次のノードにトラフィックの転送を要求すると、競合状態が発生する可能性があります。これを防ぐには、でブロックServerPoolしmutexます。ただし、ブロックしたくないというだけでなく、冗長になりますServerPool。カウンターを1つ増やすだけです。

これらの要件を満たす最良のソリューションは、アトミックインクリメントです。 Goはパッケージでそれをサポートしていますatomic。

func (s *ServerPool) NextIndex() int {
  return int(atomic.AddUint64(&s.current, uint64(1)) % uint64(len(s.backends)))
}

現在の値をアトミックに1増やし、配列の長さを変更することでインデックスを返します。つまり、値は常に0から配列の長さの範囲内にある必要があります。最後に、カウンター全体ではなく、特定のインデックスに関心があります。

ライブサーバーの選択


リクエストがすべてのサーバーで循環的に循環することはすでにわかっています。そして、アイドルをスキップするだけです。

GetNext()常に0から配列の長さの範囲の値を返します。いつでも次のノードを取得できます。ノードが非アクティブの場合は、ループの一部として配列をさらに検索する必要があります。


配列をループします。

図に示すように、次のノードからリストの最後に移動します。これはで行うことができますnext + length。ただし、インデックスを選択するには、配列の長さに制限する必要があります。これは、変更操作を使用して簡単に行うことができます。

検索中に動作中のサーバーを見つけたら、それを最新としてマークする必要があります。

// GetNextPeer returns next active peer to take a connection
func (s *ServerPool) GetNextPeer() *Backend {
  // loop entire backends to find out an Alive backend
  next := s.NextIndex()
  l := len(s.backends) + next // start from next and move a full cycle
  for i := next; i < l; i++ {
    idx := i % len(s.backends) // take an index by modding with length
    // if we have an alive backend, use it and store if its not the original one
    if s.backends[idx].IsAlive() {
      if i != next {
        atomic.StoreUint64(&s.current, uint64(idx)) // mark the current one
      }
      return s.backends[idx]
    }
  }
  return nil
}

バックエンド構造での競合状態の回避


ここでは、重要な問題を覚えておく必要があります。構造にBackendは、いくつかのゴルーチンが同時に変更または要求できる変数が含まれています。

goroutinesは変数に書き込むよりも変数を読み取ることを知っています。そのため、アクセスのシリアル化のためにAliveを選択しましたRWMutex。

// SetAlive for this backend
func (b *Backend) SetAlive(alive bool) {
  b.mux.Lock()
  b.Alive = alive
  b.mux.Unlock()
}

// IsAlive returns true when backend is alive
func (b *Backend) IsAlive() (alive bool) {
  b.mux.RLock()
  alive = b.Alive
  b.mux.RUnlock()
  return
}

リクエストの分散


これで、リクエストのバランスをとるための簡単な方法を策定できます。すべてのサーバーが落ちた場合にのみ失敗します。

// lb load balances the incoming request
func lb(w http.ResponseWriter, r *http.Request) {
  peer := serverPool.GetNextPeer()
  if peer != nil {
    peer.ReverseProxy.ServeHTTP(w, r)
    return
  }
  http.Error(w, "Service not available", http.StatusServiceUnavailable)
}

このメソッドは、単純な形式でHTTPサーバーに渡すことができますHandlerFunc。

server := http.Server{

// lb load balances the incoming request
func lb(w http.ResponseWriter, r *http.Request) {
  attempts := GetAttemptsFromContext(r)
  if attempts > 3 {
    log.Printf("%s(%s) Max attempts reached, terminating\n", r.RemoteAddr, r.URL.Path)
    http.Error(w, "Service not available", http.StatusServiceUnavailable)
    return
  }

  peer := serverPool.GetNextPeer()
  if peer != nil {
    peer.ReverseProxy.ServeHTTP(w, r)
    return
  }
  http.Error(w, "Service not available", http.StatusServiceUnavailable)
}

まず、コンテキストのキーを設定する必要があります。文字列ではなく、一意の数値を使用することをお勧めします。Goにはiota、定数を段階的に実装するためのキーワードがあり、それぞれに一意の値が含まれています。これは、数値キーを定義するための優れたソリューションです。

// GetAttemptsFromContext returns the attempts for request
func GetRetryFromContext(r *http.Request) int {
  if retry, ok := r.Context().Value(Retry).(int); ok {
    return retry
  }
  return 0
}

パッシブサーバーの検証


パッシブチェックは、落ちたサーバーを特定して回復します。一定の間隔でpingを送信して、ステータスを確認します。

pingするには、TCP接続を確立してみます。サーバーが応答した場合、サーバーは機能しているとマークします。このメソッドは、などの特定のエンドポイントを呼び出すように調整できます/status。サーバーへの追加の負荷を軽減するために、作成後に接続を必ず閉じてください。それ以外の場合は、この接続を維持しようとし、最終的にリソースを使い果たしてしまいます。

// isAlive checks whether a backend is Alive by establishing a TCP connection
func isBackendAlive(u *url.URL) bool {
  timeout := 2 * time.Second
  conn, err := net.DialTimeout("tcp", u.Host, timeout)
  if err != nil {
    log.Println("Site unreachable, error: ", err)
    return false
  }
  _ = conn.Close() // close it, we dont need to maintain this connection
  return true
}

これで、サーバーを反復してステータスをマークできます。

// HealthCheck pings the backends and update the status
func (s *ServerPool) HealthCheck() {
  for _, b := range s.backends {
    status := "up"
    alive := isBackendAlive(b.URL)
    b.SetAlive(alive)
    if !alive {
      status = "down"
    }
    log.Printf("%s [%s]\n", b.URL, status)
  }
}

このコードを定期的に実行するには、Goでタイマーを実行します。チャンネルのイベントを聞くことができます。

// healthCheck runs a routine for check status of the backends every 2 mins
func healthCheck() {

  for {
    select {
    case <-t.C:
      log.Println("Starting health check...")
      serverPool.HealthCheck()
      log.Println("Health check completed")
    }
  }
}
