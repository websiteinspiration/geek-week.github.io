Liquibaseを使用してSpring Bootアプリケーションのデータベース構造を管理する。パート2

で前の部分、我々は基本的な機能を考え出しLiquiBaseをするとの基本的な例を書いた春のブートデータベースを初期化するためにLiquiBaseをを使用するアプリケーションを。ベースアプリケーションの完全なコードは、GitHubで確認できます。この記事では、liquibase-maven-pluginと、データベース構造のバージョニングに使用できる追加機能について説明します。まず、比較関数を使用してスクリプトを自動的に作成する方法から始めましょう。

データベースの構造を変更する必要があるとします。たとえば、メールがnullにならないようにしたい。もちろん、このような小さな変更の場合、コードとスクリプトを手動で修正できますが、さらに変更がある場合はどうなりますか？この場合、Liquibaseに組み込まれたデータベースを比較する機能が役立ちます。これの興味深い特徴は、2つのデータベースだけでなく、アプリケーション内の一連のJPAエンティティを含むデータベースも比較できることです。それがまさに私たちが今やることです！

liquibase-diffを使用して変更を加えたスクリプトを作成する


pom.xmlファイルのplugins セクションに、このかなり複雑なデザインを追加します。これはliquibase-maven-pluginであり、Hibernateエンティティの分析とYAMLファイルの操作のために依存関係が接続されています。プラグインは、2つのデータベースの構造を比較することによって、またはデータベースのデータ構造とアプリケーション内のhiberanteエンティティのセットを比較することによって、liquibaseスクリプトを自動的に生成するのに役立ちます（これは、まさにliquibase-hibernate5が追加されたものです）。

<plugin>
    <groupId>org.liquibase</groupId>
    <artifactId>liquibase-maven-plugin</artifactId>
    <version>3.8.1</version>
    <configuration>
        <propertyFile>${project.build.outputDirectory}/liquibase-maven-plugin.properties</propertyFile>
        <systemProperties>
            <user.name>your_liquibase_username</user.name>
        </systemProperties>
        <logging>info</logging>
    </configuration>
    <dependencies>
        <dependency>
            <groupId>org.liquibase.ext</groupId>
            <artifactId>liquibase-hibernate5</artifactId>
            <version>3.6</version>
         </dependency>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-data-jpa</artifactId>
            <version>2.2.0.RELEASE</version>
        </dependency>
    </dependencies>
</plugin>

user.name 設定に注意してください。これはオプションですが、これがない場合、作成されたスクリプトで、プラグインが起動されている現在のOSユーザーの名前がLiquibaseによって示されます。

プラグインの設定は、pom.xmlに直接書き込むか、mavenを呼び出すときにコマンドラインパラメーターとして渡すことができますが、個別のliquibase-maven-plugin.propertiesファイルを使用するオプションをお勧めします。その内容は次のようになります。

changeLogFile= @project.basedir@/src/main/resources/db/changelog/db.changelog-master.yaml
url= jdbc:mysql://localhost:3306/geek_db?createDatabaseIfNotExist=true&allowPublicKeyRetrieval=true&useSSL=false&useUnicode=true&characterEncoding=UTF-8&serverTimezone=UTC
username= dbusername
password= dbpassword
driver= com.mysql.cj.jdbc.Driver
referenceUrl= hibernate:spring:ru.usharik.liquibase.demo.persist.model?dialect=org.hibernate.dialect.MySQLDialect
diffChangeLogFile= @project.basedir@/src/main/resources/db/changelog/db.changelog-@timestamp@.yaml
ignoreClasspathPrefix= true
ここでは、urlパラメータとreferenceUrlパラメータに注意を払う価値があります。比較後にリキベースを作成するスクリプトは、ベースリンクreferenceUrlとベースリンクurlの違いになります。その後、リンクURL Noに基づいてスクリプトを実行すると、リンクreferenceUrlと同じになります。 referenceUrlリンクには特に注意が必要です。ご覧のとおり、これはデータベースではなく、エンティティークラスが配置されているアプリケーションのパッケージを指します。これにより、コードに加えられた変更をデータベースに追加するスクリプトを見つけることができます。

次に、maven-resource-pluginを設定して@project.basedir@、およびなどの設定ファイル内のプレースホルダーを置き換える必要があります@timestamp@。これを行うには、次のフォームのリソースセクションをビルドセクションに追加します。

<resources>
    <resource>
        <directory>src/main/resources</directory>
        <filtering>true</filtering>
        <includes>
            <include>*.properties</include>
        </includes>
    </resource>
    <resource>
        <directory>src/main/resources</directory>
        <filtering>false</filtering>
        <includes>
            <include>**/*.*</include>
        </includes>
    </resource>
</resources>

ちなみに、Springブートは、maven-resource-pluginが入力されたプレースホルダーの標準形式をからに変更${smth}し@smth@ます。実際のところ@smth@、Spring Bootのビュープレースホルダーは、Spring Bootアプリケーション自体の環境変数とパラメーターを置き換えるために使用されます。

また、我々はわずかに変化のプロパティのセクションをでのpom.xmlに値を代入するタイムスタンプ変数我々が必要とする形式です。残念ながら、標準形式には、一部のオペレーティングシステムのファイル名で禁止されている文字が含まれている場合があります。

<properties>
    <java.version>1.8</java.version>
    <timestamp>${maven.build.timestamp}</timestamp>
    <maven.build.timestamp.format>yyyyMMdd-HHmmssSSS</maven.build.timestamp.format>
</properties>

    private String email;

最後に、比較のためにliquibase-maven-pluginを使用してビルドコマンドを実行します。

mvn clean install liquibase:diff -DskipTests=true

この場合、プラグイン（liquibase：diff）はコンパイルされたソースではなく、分析のためにターゲットフォルダーからコンパイルされたエンティティクラスファイルを使用するため、プロジェクトを完全に再構築する必要があります。

すべてが正しく行われている場合、resources / db / changelogフォルダーでコマンドが正常に実行された後、db.changelog-20190723-100748666.yamlという名前のファイルが表示されます。ファイル名に現在の日付と時刻を使用しているため、起動するたびに新しいファイルが作成されるため、非常に便利です。前のレッスンに対応するテーブル構造を持つデータベースをすでに作成している場合、ファイルの内容は次のようになります。

databaseChangeLog:
- changeSet:
    id: 1563876485764-1
    author: your_liquibase_username (generated)
    changes:
    - addNotNullConstraint:
        columnDataType: varchar(255)
        columnName: email
        tableName: users
ご覧のとおり、このスクリプトはコードで行われた変更を正確に行います。演習として、このスクリプトを空のデータベースに対して実行し、結果を確認することをお勧めします。

次に、このファイルからdb.changelog-master.yamlに changeSetをコピーするか、このファイルを命令で接続します。

  - include:
      file: db.changelog-20190723-100748666.yaml
      relativeToChangelogFile: true

また、このファイルlogicalFilePath: db/changelog/db.changelog-20190723-100748666.yamlでは、db.changelog-master.yamlでの方法と同様に指定する必要があります。

変更をロールバックする


ここで、行った変更をロールバックする方法を見てみましょう。何らかの理由で、changeSetsで指定した識別子を使用してそれらにロールバックすることはできません。3つのオプションがあり、ロールバックポイントを指定します

  • 現在から数えたchangeSetsの数を通して
  • 変更日後
  • タグ経由（特別な種類のchangeSetを使用して設定）

タグは以下のように設定されています。

  - changeSet:
      id: some_uniqui_id
      author: liquibase_user_name
      changes:
        - tagDatabase:
            tag: db_tag

さて、ロールバックを行う3つのリストされた方法のコマンド

mvn liquibase:rollback -Dliquibase.rollbackTag=db_tag
mvn liquibase:rollback -Dliquibase.rollbackCount=1
mvn liquibase:rollback "-Dliquibase.rollbackDate=Jun 03, 2017"
