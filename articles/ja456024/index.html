PHP 2019：考えるよりも良い



かつて人気のあった投稿「PHP：A Bad Fractal of Bad Design」を覚えていますか？私が最初に読んだとき、私は古いPHPプロジェクトがたくさんある愚かな場所で働いていました。彼女は、私が去って、プログラミングとはまったく違う何かをするべきかどうか疑問に思いました。

幸い、私はすぐに仕事を変えることができ、さらに重要なことに、PHPは5. *バージョンから「わずかに」開発することができました。そして今日、この記事を通して、もはやPHPでプログラミングをしていないか、レガシープロジェクトで立ち往生している人々にアピールしたいと思います。

スポイラー：今日、PHPは、ほとんどすべてのプログラミング言語と同様に、欠陥に満ちています。主な機能の多くは依然として一貫性のないシグネチャを持ち、構成設定は混乱しています。現在のプロジェクトでは動作が異なるため、または他の方法では実行できないため、多くの開発者が下手なコードを書いています。

今日、私は明るい面を見てみたいと思います。何が変わったか、そしてクリーンでメンテナンスされたPHPコードをどのように書くかに焦点を当てましょう。数分間偏見をなくすようお願いします。

その後、PHPは以前とまったく同じように考えることができます。ほとんどの場合、過去数年間にPHPで行われたいくつかの改善に驚くでしょう。

PHPリリースサイクルの概要。現在のバージョンはPHP 7.3で、バージョン7.4は 2019年末にリリースされる予定です。 PHP 8.0は7.4以降の次のバージョンになります。 PHP 5 *の後期以来、メンテナのチームは毎年のリリースサイクルを維持しようと努めており、過去4年間非常によく機能しています。

一般に、新しい各リリースは2年間サポートされ、さらに1年間「セキュリティ修正」を受けます。目標は、バージョンを最新に保つようにPHP開発者を動機付けることです。毎年小さな更新を適用することは、たとえば5.4から7.0に切り替えるよりもはるかに簡単です。

フレームワークとエコシステム


フレームワークと言えば、PHPはもはやWordPressに限定されていません。私はプロのPHP開発者としてあなたに言うことができます：WordPressは決して現代のエコシステムの代表ではありません。

一般的に、SymfonyとLaravelの 2つの主要なWebアプリケーションフレームワークがあります。そして、いくつかの小さなもの：Zend、Yii、Cake、CodeIgniterなど。しかし、最新のPHP開発がどのように見えるかを知りたい場合は、2つの最大の1つを知ることをお勧めします。

両方のフレームワーク（SymfonyとLaravel）には、パッケージと製品の大規模なエコシステムがあります。管理パネルとCRMからスタンドアロンパッケージ（元の「スタンドアロンパッケージ」）、CIからプロファイラー、Webソケットサーバー、キューマネージャー、支払い統合などの多数のサービス。率直に言って、すべてをリストするには多すぎます。

ただし、これらのフレームワークは直接開発を目的としています。コンテンツ管理だけが必要な場合、WordPressやCraftCMSのようなプラットフォームはどんどん良くなっています。

PHPエコシステムの現在の状態を測定する1つの方法は、PHPのメインパッケージリポジトリであるpackagistを調べることです。それは指数関数的に成長しています。 1日あたり約2,500万ダウンロードで、PHPエコシステムは以前の弱点ではなくなったと言えます。

経時的なパッケージとバージョンの数を示すこのグラフを見てください。また、packagistのWebサイトにもあります。



アプリケーションフレームワークとCMSに加えて、近年では非同期フレームワークも増加しています。

PHPまたは他の言語で記述されたこれらのフレームワークとサーバーにより、完全に非同期のPHPを使用できます。たとえば、Swoole、Amp、ReactPHPなどです。

非同期の世界に入って以来、多数のI / O操作を伴うWebソケットやアプリケーションのようなものがPHPの世界で本当に重要になっています。

主要な開発者が言語開発について話し合う内部のメーリングリストでも、カーネルにlibuvを追加することについて話しました。libuvを知らない人のために、これはNode.jsがすべての非同期性を保証するために使用するのと同じライブラリです。

言語機能について話している間、言語が今日どのように発達しているかというトピックにも触れましょう。コミュニティーがRFCを提供することもできる一方で、言語を前進させるボランティアのメンテナーのチームがあります。

これらのRFCは、インターネットでも読むことができる内部メーリングリストで議論されます。新しい言語機能を追加する前に、投票する必要があります。コアに含めることができるのは、投票の過半数、少なくとも全投票者の⅔のRFCのみです。

投票には約100人が参加し、各RFCに投票する必要はありません。もちろん、投票する機会を持つ人々のグループには、コードベースを維持する必要があるため、メンテナが含まれます。それらに加えて、メンバーがPHPコミュニティから個別に選ばれたグループがあります。このグループは、PHPドキュメントのメンテナ、主要なPHPプロジェクトの作者、およびPHPコミュニティの著名な開発者で構成されています。

ほとんどの開発は自発的に行われていますが、PHPコアの開発者の1人であるNikita Popovは最近、JetBrainsに雇われ、フルタイムで言語に取り組んでいます。もう1つの例は、最近Zend Frameworkへの投資を決定したLinux Foundationです。このような作業と買収により、PHPとその開発の将来の安定性が保証されます。

ツール


コア自体に加えて、過去数年にわたって、その周りのツールが増加しています。詩篇のように静的アナライザVimeoの、によって作成されたものとしてPhanさん、そしてPHPStanは気にしています。

これらのツールは、PHPコードを静的に分析し、一般的なエラー、起こり得るバグなどを報告します。ある意味では、これらのツールが提供する機能はTypeScriptと比較できます（翻訳者注：静的アナライザーは、エラー/欠陥を検索する言語の機能を拡張します）言語を改善すると、TSは条件付きでJSに加えて同じことを行いますが、現時点ではPHP言語はトランスパイルされていないため、カスタム構文は許可されません。

まだドックブロックとタイピングに依存する必要がありますが、PHPの作成者であるRasmus Lerdorfは、静的分析メカニズムをカーネルに追加するアイデアに言及しました。この仕事には多くの可能性が含まれていますが、人件費の面で巨大です。

転置といえば、PHP構文をカーネルレベルではなく、JavaScriptで実装されているユーザーライブラリのレベルで拡張する試みがあったことは注目に値します。Preと呼ばれるプロジェクトは、まさにそれを行います。これにより、通常のPHPコードにラップする新しいPHP構文を使用できます。

このアプローチはJavaScriptの世界で実証されていますが、IDEと静的分析に適切なサポートが提供されている場合にのみ、PHPで利益を得ることができます。これは非常に興味深いアイデアですが、メインストリームと呼ばれるまでには長い道のりがあります。

これにより、信頼性が高く、メンテナンスの行き届いた高品質のソフトウェアを作成できます。私が働いているクライアントは、私と同じように、最終結果に満足しています。

PHPを使用する場合でも非常に悪いコードを書くことができるという事実にもかかわらず、正しく使用すれば、これはWeb開発に最適な選択肢だと思います。
