SSDの信頼性テスト：3dnews対JEDEC対常識。真実はどこですか、兄弟？



3dnewsの 伝説的なSSD信頼性テスト（2018.01に公開）は誰もが知っています。その結果、一部の予算ドライブは、メーカーが宣言したリソースを10倍超えました。

この調査の後、陰謀の神学的理論が現れ、製造業者は予算のSSDのリソースを過小評価し、ほとんどすべてのSSDが質的にウェアレベリングを実行および実行できるという信念が広まりました。

3dnews.ruからの調査は、Techreport.comのテストに基づいて行われました（2013.08.20からの公開）。

耐摩耗性を測定する方法論も同様に使用されました。

1.このテクニックの何が問題になっていますか？


問題は、ディスクが順番にいっぱいになることです。これは、JEDEC（半導体工学標準化委員会、またはソリッドステートテクノロジーアソシエーションと呼ばれる、主要なフラッシュメモリメーカーすべてを含む）が推奨する実際の使用例または耐摩耗性手順のいずれにも対応していません。

OSとコントローラー（DRAMキャッシュが小さい場合）の両方が、特定のデバイスに固有の大きなネイティブブロックの記録と書き込みを行う連続ブロックをグループ化します。同時に、実質的に記録ゲインはなく、ウェアレベリングアルゴリズムは必要ありません。

実際の状況では、乗算係数とウェアレベリングアルゴリズムの高品質な実装の両方が、ストレージリソースに大きな影響を与えます。

問題1. WAFファイルを連続して書き込む場合≈1


レコード乗算係数（WAF）が一致する傾向にあるように構築されたテストは、リソースに対して過大に見積もられた結果を予測可能に与えます。3dnews（WAFを抽出できる）で分析されたほとんどのディスクの記録ゲインは≈1-1.11でした。3dnewsの専門家は、これを効率的なコントローラーアルゴリズムで説明しました。文字通りWAFの1つのインスタンスだけが、無効なコントローラーによって説明される3を持っていました。

ただし、私の意見では、全体のポイントはテスト方法論にあり、SSDドライブのリソースの過大評価をもたらす順次レコード（WAF→1）を生成しました。
次に何回評価します。

問題2.ウェアレベリングアルゴリズムの品質がテストされていない


ディスクの連続的な充填とその後の消去により、ウェアレベリングメカニズムのテストは不十分です。製造元がディスクの最初の数GB（ディスクのビットマップ、FATおよびその他のメタデータが配置されている場所）をSLCモードで動作させる（またはRAMバッファーにキャッシュする）と推測した場合、ウェアレベリングアルゴリズムがファームウェアに完全になく、シーケンシャルレコーディングですべて同じことが実現されます。優れたリソースインジケータ。

問題3.リソース枯渇後のデータ保持がテストされていない


旅行に数か月かかる場合、パスポートリソースを作成したディスクを信頼できるかどうかは不明です。

問題4.ディスクを順番に埋めてからほぼ完全に消去するのは、実際のユーザー定義の動作パターンではない


SSDはまだかなり高価なリソースであるため、通常、SSDは可能な限り完全に使用し、最小限の空き領域を残します。

私の意見では、理想的なテストは、テスト中にSSDが80〜90％詰まっている一方で、古いファイルをランダムに削除して新しいファイルを追加することです。

2.アニメーション要素の記録


2.1。ファイルシステムの断片化


Windows SSDではデフラグが無効になっており、デフォルトのNTFSクラスターサイズは4Kであるため、実際にはディスクは非常に断片化されています。この場合、シーケンシャル録音でもほぼランダムな速度になります。

変更された1つのクラスターを書き留めるには、コントローラーは最初にNANDハードウェアページ全体（サイズが数百キロバイトに達する可能性があります）を読み取り、4Kを変更してから、すべてを書き留める必要があります。NANDページの有効容量のサイズが64Kの場合、記録ゲインは16倍になります。

2.2。摩耗調整アルゴリズム


このようなアルゴリズムは、コントローラー内の別のプロセスとして実装できます。それはこのような何かを動作します：

高い摩耗と地域に静的データを移動するには、データが解放されたTRIMブロックがあることを提供する、移動中のデータのサイズに等しい記録する必要があります。また、空きブロックの数が少ない場合、データをスワップするには2つのエントリを作成する必要があります。

RAM = a
a = b
b = RAM

DRAMバッファが小さいか欠落している場合、3つのエントリが必要です。

temp = a
a = b
b = temp

摩耗の少ないページを使用するために静的データのみを移動することは理にかなっているため、理想的なウェアレベリングアルゴリズムでのこれらの操作は非常にまれであり、レコードアニメーションに対するウェアレベリングアルゴリズムの影響は無視します。もちろん、実際のアルゴリズムが理想的なアルゴリズムを使用するという保証はありません。

2.3。通常のSSD書き込みはランダムで、4〜8KBブロック


SSD記録の通常の性質は、基本的にランダムな8KBレコードです。断片化がない場合でも、通常の記録ブロックのサイズはNANDページのサイズよりも小さくなり、記録が倍増します。

2.4。ガベージコレクションアルゴリズム


ここに最大の落とし穴があります。 NANDメモリのブロックサイズは数メガバイトに達します。ブロックは複数のページで構成されています。

ページは個別に読み書きでき、ブロックは完全に消去することしかできません。

時間の経過とともに、ブロック内の多くのページが無効としてマークされます。これは、ページ内で変更され、別の場所に書き込まれたデータ、またはTRIM呼び出しが原因です。そして遅かれ早かれ、ガベージコレクターは、いくつかの部分的に満たされたブロックを取り、完全に書き込まれたブロックと解放されたブロックをまとめる必要があります。

空き領域がある限り、ほとんどの場合は開始しません。したがって、時間の経過とともに、ブロックに多数の穴が形成されます。

ディスク容量が小さいほど、WAF（書き込み増幅係数）の増加が速くなります。

興味深い事実：

ご存知のように、NANDメモリはブロックに結合されたページに分割されます。書き込みと読み取りはページごとに行われ、消去はブロック単位でのみ行われます。

何かを書き換える必要がある場合、理想的なファームウェアは次のように機能します。変更されたデータを含むページを、まだ書き込まれていないページがあるブロックに書き込み、FTL（Flash Translation Layer）で新しい対応LBA→ページを確立し、古いページを無効としてマークします。そのようなブロックがない場合は、ガベージコレクターが起動し、半分埋められたブロックのデータを、ブロックが解放された埋められたブロックにコンパイルします。

これは理想的なオプションです。すべてのカスタムSSDに適切なウェアレベリングアルゴリズムがあるわけではありません。 3dnewsの劇的に異なるテスト結果からわかるように。

一般的なページサイズは8KB以上で、ブロックサイズは2MB以上です。JEDECのテストでは、ドライブが38％を超えていっぱいになることはなかったため、疑わしいことに、そこには常に空きブロックがあり、SSDリソースを消費するガベージコレクターのアクティブな作業はありませんでした。しかし、データが不完全なページで書き込まれることがあり、1つのレコードがNANDページの境界に沿って渡されることがあったため、WAF（書き込み乗算係数）がありました。

JEDECのテストで、ページサイズに応じてWAFを計算するスクリプトを書きました。これは、NANDページサイズに応じたWAFです。
1K 2K 4K 8K 16K 32K 64K 128K
WAF 1.0 1.0 1.01 1.11 1.31 2.06 3.54 6.5

3dnewsのテストでは、ファイルは順次書き込まれ、Windowsには書き込みキャッシュがあり、その間にセクターが規則正しく書き込まれるため、WAFは約1です。

典型的なシナリオでは、ページが8K（Samsung 840 EVO）の場合、WAFはわずか1.11（3dnewsデータからの11％エラー）であり、これは許されるようです。しかし、ガベージコレクションアルゴリズムを導入するWAFを考慮すると、許すことができません。

6. 3dnews SSD


これは悪い考えだと言わざるを得ません。しかし、多くはそれを行います。そのため、ユーザーディスクのリソースをサーバーのリソースとして決定するための修正係数を計算します。 3dnewsテストから耐摩耗性を取得し、それを必要な係数で除算して、企業で使用するために期待されるリソース（JEDECメソッド）を取得します。
サーバーは、カスタムSSDのように、切断された状態でこのような長期間のストレージを必要としません。これが対応する表です。



負荷がかかった状態での典型的な50℃の動作では、ドライブは25℃のオフ状態で58週間≈1年のデータストレージを提供する必要があります。

カスタムアプリケーション（オフ状態で2年間データの安全性が要求される）では、リソースを2分の1に削減しました。企業で使用する場合、このような長い有効期間は必要ないため、たとえば1.3のように、より少ない数を使用します。

その後、企業の負荷に一般的なWAFを乗算し、ガベージコレクターの作業を考慮に入れて、次の係数テーブルを取得します。3dnewsで得られた結果は、この数値で割る必要があります。

この規格は、企業ドライブのバックアップ領域または空き領域のサイズを記述していないという事実に問題があります。したがって、JEDEC企業テストの WAFを正確に評価することはできないため、JEDECカスタムSSDテストからこの数値（1.55）を取得します。

本当に長い旅行に出かけることがあり、現在ドライブが使用されていない場合は、パスポートリソース内に留まることをお勧めします。その後、ドライブを交換する必要があります。

NANDのページサイズが16Kを超え、ディスクが非常に密集しているまれなケースでは、ドライブの実際のリソースを計算するために、数十倍、場合によっては数百倍削減する必要があります。

また、サーバーに予算を投入する場合は、レイドとバックアップの両方に注意してください。安定した応答時間と速度、電源保護、その他の企業向けドライブの優れた機能はありませんが、記事の対応する表から修正除算器を使用してリソースを計算できます。一般的なケースでは、11で割ります。
