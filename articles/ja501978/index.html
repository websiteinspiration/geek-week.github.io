Prometheusヒストグラムはどのように機能しますか？

記事の翻訳は、コース「モニタリングとロギング：Zabbix、Prometheus、ELK」の開始前に準備されました。



以前は、カウンター、ゲージ、および概要を確認しました。次に、Prometheusでのヒストグラムの動作について説明します。
ヒストグラムは、要約といくつかの類似点があります。ヒストグラムは、さまざまなカウンターの組み合わせです。サマリーメトリックと同様に、ヒストグラムメトリックは、メソッドを使用して、イベントのディメンションインジケーター、多くの場合はそれらの期間を追跡するために使用されますobserve。計時ツールは通常、掲示と同じです。それらが異なるのは、変位値の処理です。

以下は、露出用の Prometheusラベル形式の例ですhandler。

# TYPE prometheus_http_request_duration_seconds histogram
prometheus_http_request_duration_seconds_bucket{handler="/",le="0.1"} 25547
prometheus_http_request_duration_seconds_bucket{handler="/",le="0.2"} 26688
prometheus_http_request_duration_seconds_bucket{handler="/",le="0.4"} 27760
prometheus_http_request_duration_seconds_bucket{handler="/",le="1"} 28641

prometheus_http_request_duration_seconds_sum{handler="/"} 1863.80491025699
prometheus_http_request_duration_seconds_count{handler="/"} 28860

_sumこれら_countは、サマリーとまったく同じように機能し、過去5分間の平均実行時間を取得するために使用できます。

 rate(prometheus_http_request_duration_seconds_sum[5m]

ただし、_sumMySQLdエクスポーターからの一部のメトリックなどでは、これが存在しない非常にまれなケースがあります。

ヒストグラムの注目すべき部分は時系列です_bucket。これは実際には指標のヒストグラムの部分です。具体的には、これらは累積ヒストグラムを形成するカウンターです。leより小さいか等しいことを示します。したがって、26688リクエストは200ミリ秒以下、27760リクエストは400ミリ秒以下かかり、合計で28860リクエストがありました。バケツの値は単調に減少せず、+Infバケツが最大の値になります。+Infバケットは常に存在し、値と一致する必要があります_count。

たとえば、0.9（90パーセンタイル）の分位数を計算するには、次を使用する必要があります。

histogram_quantile(0.9, 
  rate(prometheus_http_request_duration_seconds_bucket[5m])
)

要約と比較したヒストグラムの大きな利点の1つは、変位値を計算する前にバケットを集約できることですle。ラベルを失わないように注意してください。

histogram_quantile(0.9, 
  sum without (handler)(
    rate(prometheus_http_request_duration_seconds_bucket[5m])
  )
)

集計に加えて、ヒストグラムは、カウンターがすばやくインクリメントされるため、クライアントの方が安価です。では、なぜ常にヒストグラムを使用しないのですか？長い答えがありますですが、短いバージョンでは、ヒストグラムでは最初にバケットを選択する必要があり、バケット要素の数により、コストはクライアントからPrometheus自体に転送されます。デフォルトで10バケットは、ミリ秒から1秒の範囲の遅延を持つ一般的なWebサービスをカバーします。場合によっては、これを変更する必要があります。ここでは、たとえば、デフォルトで2分のタイムアウトを持つPromQLクエリをより適切に追跡するために再定義されました。 1ダース以上のバケットを使用すると、より正確な結果が得られますが、多くの時系列を生成することもできます。特に他のタグと組み合わせて。

Prometheusなどのリアルタイムモニタリングシステムを使用する場合、目標は、それに基づいてエンジニアリングの決定を行うのに十分な分析値を提供することです。たとえば、90パーセンタイルの遅延が50ミリ秒増加したことを知ることは、現在562ミリ秒または563ミリ秒であるかどうかを知ることよりも重要であり、通常は10バケットで十分です。正確な答えが必要な場合は、いつでもログから後で計算できます。バケットが多すぎる場合、前に示したように、データ処理中にそれらが破棄される可能性があります。もっと極端なケースでは、あなたは完全に一連の無視することができ_bucket秒の平均に依存している_sumと_count。

結論として、要素の数を考慮に入れる必要がありますが、ヒストグラムを使用すると、総変位値を計算できます。ヒストグラムが必要ない場合は、変位値なしの要約の方が安価なオプションです。
