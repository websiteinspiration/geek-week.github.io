メモ帳用のゲームの作り方



インディーズの開発者からの珍しいソリューションについて読んでいると、金に出会いました。これは、テキストエディタでのゲームに関する記事です。アート、アニメーション、プロット-すべてが本来あるべき姿です。

私はゲームを作成しましたが、それでも痛いです（おそらく、作者はそれが痛いと言いたかったのですが、このオプションを意図的に使用することができました-およそ）。


プロジェクトはここからダウンロードでき、コードはGithubで表示できます。

すべては2017年に「メモ帳でゲームを作成することは可能ですか？」という質問から始まりました。それから私はニヤニヤしました。 3年が経ちました。すべてがどのように機能するかを考え、それが現実であることを確認した後、私はこのゲームを作ることにしました。

通常、ボタンを押すと、ゲームで何かが起こります。 Aを押すと、マリオがジャンプします。すべては情報とフィードバックを受け取ることに結びついています。ゲームは入力を受け取り、それ自体を表示します。

メモ帳のゲームでは、ユーザーがファイルに加えた入力と出力で、ゲームがファイルに加えた変更があります。これを行うために、アプリケーションはファイルが最後に保存された時間を追跡します。変更されている場合、ゲームはファイルの内容を読み取り、新しいデータをファイルに導入します。

問題があります。Microsoftのメモ帳では、ファイルが変更されているかどうかはチェックされません。ファイルを保存して閉じ、もう一度開く必要があります。そのようなゲームを作成することは可能ですが、あまり面白くありません。私は代替案を探す必要がありました。

結果としてゲームが通常のメモ帳で作成されなかったため、あなたの失望を理解することができます。私のタイトルはその中で実行できます-少し混乱したプロセスです。ゲームをもっと楽しくするために、プロジェクトのクールさを犠牲にすることにしました。



オルタナティブ


別のテキストエディターを探す必要がありました。唯一の要件は、自動ファイル更新でした。少し後で、別の機能を使用したことがわかります。

最初に、メモ帳++とSublime Textが思い浮かびました。ただし、外観はメモ帳のようには見えません。プロジェクトの魅力は完全に失われていました。さらに、プレーヤーにファイルを更新するかどうかを尋ねます。これはファイルを閉じて開くよりもはるかに優れていますが、それでもゲームプレイの邪魔になります。ファイルを自動的に更新したかったのですが。それからNotepad2が私の目を引きました。彼はほとんど完璧だった。

エディターは、MS Notepadのように構成できます。最も重要なのは、ファイルに加えられた変更をチェックすることです。しかし、Notepad ++やSublime Textと同様に、Notepad2はファイルを変更する必要があるかどうかをプレーヤーに尋ねます。幸い、エディターにはオープンソースコードがあり、完璧に仕上げることができました。



Notepad2はCで書かれています。たとえ専門家とは言えなくても、私はこの言語に少し慣れています。経験豊富なJavaScriptプログラマーであれば、コードの本質を読んで理解することができますが、Notepad2のソースコードを理解して必要な変更を加えることはそれほど容易ではありませんでした。

まず、ダイアログボックスからテキストを検索することにしました。「ファイルは外部プログラムによって変更されました。ファイルを再ロードしますか？ "これは、ダイアログボックス関数の引数として使用される変数の値です。そして私は彼女を見つけました。

if ((iFileWatchingMode == 2 && !bModified && iEncoding == iOriginalEncoding) ||
        MsgBox(MBYESNO,IDS_FILECHANGENOTIFY) == IDYES) {

このコードは、ファイルの内容が変更されたかどうかを確認します。変更されている場合、ウィンドウが開き、プログラムはユーザーが「はい」を選択したかどうかを確認します。ピースを交換するだけでよかった

MsgBox(MBYESNO,IDS_FILECHANGENOTIFY) == IDYES

TRUEにすると、プログラムは自動的にファイルを更新し始めました。そこで、ASCIIベースのレンダリングを作成しました。適切なエンジンを作成するために残っています。

お絵かき


ゲームは愛を込めて作られています。LÖVEはLuaで書かれた2Dゲームのオープンソースフレームワークです。私はこのプラットフォームを長年使用し、チュートリアルも作成しました。このプロジェクトでは、必要なすべての機能を提供するため、ファイルシステムのLÖVEモジュールが主に使用されました。通常、LÖVEはイメージを作成し、それを画面に表示します。

ほぼ同じものが必要でした。テキストファイルでのASCIIアートの出力です。私は家と鳥から始めました、そして鳥はやすりを通って飛ぶはずでした。彼はASCIIアートで見つけたアートを採用しましたが、ゲームではオリジナルの作品のみが使用されます（フォントを除く）。



そして：



アートのダウンロードは単にファイルを読んでいるだけです。

house = love.filesystem.read("art_house.txt")
bird = love.filesystem.read("art_bird.txt")

家は背景になっているので、「画面」に描くところから始めました。この場合の画面はhome.txtです。

love.filesystem.write("home.txt", house)

私は鳥がこのように働くことを望みました：

local x, y = 20, 40
drawArt(bird, x, y)

xは列番号、yは行番号です。したがって、彼は画面と鳥を線のリストに分けました。

-- Get the current canvas
screen = love.filesystem.read("home.txt")

-- Create a table. A table is like an array.
screen_lines = {}

-- The lua pattern (.-)\n says capture everything until the \n (newline).
-- We add an \n to the end of the file so that it captures the last line as well.
for line in (screen .. "\n"):gmatch("(.-)\n") do
    table.insert(screen_lines, line)
end

鳥と同じことをしました。今、鳥を説明するコードは家のコードと重複するはずでした。ここに私が必要なものがあります：

  1. 鳥を描く線を見つけます。
  2. 行全体をxに出力します。
  3. x +鳥のいるアートの長さで始まる行の残りを印刷します。
  4. 最初のパート、鳥、残りのパートで新しいラインを作成します。
  5. 他のすべての行に対して同じことを繰り返します。

コードでは、次のようになります。

function drawArt(art, x, y)
    art_lines = getLines(art)

    -- In Lua, you can get the length of a table and string with #
    for i=1, #screen_lines do
        if i == y then
            for j=1 ,#art_lines do
                -- With string:sub(start, end) we can get part of a string
                local first_part = screen_lines[i]:sub(1, x - 1)
                local second_part = screen_lines[i]
                                    :sub(x + #art_lines[j], #screen_lines[i])
                screen_lines[i] = first_part .. art_lines[i] .. second_part
            end
        end
    end
end

何が起こったの



でしょう：おそらく、あなたは鳥が長方形であることに気づきました-スペースはその芸術で使われています 状況を改善するために、各行の先頭にあるスペースの数を計算し、この数を座標に追加して、アートのみが描画されるようにしました。

-- (%s) gets all the whitespace characters.
local spaces = art_lines[j]:match("(%s*)")
-- Remove the spaces from the line.
art_lines[i] = art_lines[i]:sub(#spaces + 1, #art_lines[i])
local first_part = screen_lines[i]:sub(1, x + #spaces - 1)
local second_part = screen_lines[i]:sub(x + #spaces + #art_lines[j], #screen_lines[i])
screen_lines[i] = first_part .. art_lines[i] .. second_part

アニメーション


アニメーションなどのチップを追加し始めました。



すべてのフレームは1つのファイルに配置され、{{F}}タグで区切られています。タグは読み取り中に決定され、フレームのシーケンスを設定できます。これのおかげで、私たちは古典的なアニメーションを取得します。タイマーを作成し、それに従ってフレームを描画します。

また、印刷されたテキストの出力を実装し、ソリューションを入力するための画面、インベントリ、およびウィンドウを個別に表示しました。問題が1つありました。ゲームはファイルが開かれたことをどのようにして知るのですか？これは、先ほどお話しした2番目の機能です。

Notepad2のソースコードでは、ファイルを開いた直後に保存するように規定しました。その後、ゲームは最後の保存時間が変更されたかどうかを確認します。そのため、彼女はファイルが開いていることを知り、それを変更できます。

// At the end of the FileLoad function.
// If the file is not new, and the file has not been reloaded, save it.
if (!bNew && !bReload) {
    FileSave(TRUE,FALSE,FALSE,FALSE);
}

その結果、私はあなたが働くことができるフレームワークを得ました。ゲームを作成する時が来ました。

開発の9日間（gifファイルが作成された日付から判断して）、私はこれ



を行いました。ゲームを実行すると、印刷可能なテキストまたはアニメーションがないことがわかります。これにはいくつかの理由がありました：

  • , HDD/SSD . . , .
  • . , , . , .
  • — , . . . , , -, . . , , . , , , — .


ユーザーがゲームを開始するためにファイルをNotepad2ウィンドウにドラッグしなければならないことに私は激怒しました。ファイルをダブルクリックすると、メモ帳または.txtを読み取るための別のデフォルトプログラムが開かれました。そのようなファイルのアプリケーションを変更するコマンドをNotepad2に登録することは可能でしたが、ある種のゲームが私のコンピューターでそのようなフェイントを実行した場合、私は個人的にはそれを望みません。

ゲームを閉じると元の設定に戻るのでは？可能ですが、ゲームがクラッシュしたり予期せず終了したりすると問題が発生します。

通常の.txtファイルの代わりに別の「x」を使用できることに気づくまで、すべての決定は十分に実証されていないように見えました。正確には、Unicode文字はU + 0445（キリル文字Ha）です。混乱しないように、ファイルには* .tXtという名前を付けました。その結果、すべてのゲームファイルは* .tXtの解像度で、デフォルトでNotepad2で開かれました。

assoc .tXt=tXt
ftype tXt=[directory]/.notepad/Notepad2.exe "%1"

デフォルトのプログラムは、管理者としてのみ割り当てることができます。別のアカウントでゲームを開くと、txtファイルが使用されます。通常のメモ帳でファイルを開くと、開いているメモ帳ウィンドウにファイルをドラッグする必要があることがゲームから通知されます。または、ダブルクリックして開くように、管理者として実行します。

動機


実際、すべてが3年前に行われました。残りの時間は何をしましたか？動機の欠如の典型的な例。

当初、計画は今より少し長くなりました。ドラゴンはあなたの両親を殺しました、あなたは鍛冶屋フェルダンに行かなければならないので、彼は剣を鍛造しました。剣は集める必要のある3つの素材でできていると考えられていました。これにより、ゲームのボリュームが大幅に増加し、開発が終了しました。ゲームはあまり面白くなかったので、2か月後にプロジェクトを中止しました。

でもずっと頭の中に入れていました。メモ帳でゲームを作成できるフレームワーク全体をデバッグしましたが、プロジェクトは地面から離れませんでした。仕上げる必要がありました。 2019年、私はほとんどすべてのプロジェクトを完了しませんでした。がっかりしたので、2020年に未完を終えることを決意しました。

そして、彼女はここにいます。私は計画を短くし、すべてに1か月を費やし（1週間長くなることが判明）、戦闘に突入しました。まだA MAZEに応募しました。賞はそれぞれ2月2日に予定されていました。やる気がありました。

結論


ゲームをクリアしてよかった。プロジェクトがデジタルダストを収集していた期間は驚くべきことですが、最終的には1か月で十分でした。ゲームは、最初は私が望んだほどのボリュームにすべきではありませんでした。このような非標準のプロジェクトは、実装可能な機能のみを表示する必要があります。

次は何ですか？ペイントゲーム？電卓のゲーム？私はそれらを作る可能性は低いです。しかし、私は従来とは異なるプラットフォームを使用するゲームについて考えるのが好きです。
