﻿ほぼ高品質の電子芸術図書館

GitHubのElectronic Artsリポジトリに注意が向けられました。これは非常に小さく、23のプロジェクトのうち、EASTL、EAStdC、EABase、EAThread、EATest、EAMain、およびEAAssertのいくつかのC ++ライブラリにのみ興味がありました。プロジェクトも非常に小さい（約10ファイル）ため、エラーは20ファイルの「最大」にしか見つかりませんでした：D興味深いファイルも見つかりました！記事の執筆中に、同僚と私はEAのゲームとその戦略についても活発に議論しました。D



前書き


Electronic Arts（EA）は、ビデオゲームを配信するアメリカの企業です。GitHubのWebサイトには、小さなリポジトリといくつかのC ++プロジェクトがあります。これらはC ++ライブラリです：EASTL、EAStdC、EABase、EAThread、EATest、EAMain、EAAssert。それらは非常に小さく、PVS-Studioアナライザーを使用して、それらの「最大」のEAStdC（20ファイル）でのみ興味深いエラーが見つかりました。このようなボリュームでは、コード全体の品質について話すのは困難です。5つの警告を評価して、自分で決めてください。

警告1


V524「>>」関数の本体が「<<」関数の本体と完全に同等であるのは奇妙です。EAFixedPoint.h 287

template <class  T,
  int  upShiftInt, int downShiftInt,
  int  upMulInt,   int downDivInt>

struct FPTemplate
{
  ....
  FPTemplate operator<<(int numBits) const { return value << numBits; }
  FPTemplate operator>>(int numBits) const { return value << numBits; }

  FPTemplate& operator<<=(int numBits) { value <<= numBits; return *this;}
  FPTemplate& operator>>=(int numBits) { value >>= numBits; return *this;}
  ....
}

シフト演算子をオーバーロードするとき、プログラマーはそれらの1つにタイプミスを作り、<<および>>演算子を混乱させました。ほとんどの場合、これはコピーと貼り付けのプログラミングの結果です。

警告5


V728過剰チェックを簡略化できます。'（A &&！B）|| （！A && B） '式は、' bool（A）！= Bool（B） '式と同等です。int128.h 1242

inline void int128_t::Modulus(....) const
{
  ....
  bool bDividendNegative = false;
  bool bDivisorNegative = false;
  ....
  if(    (bDividendNegative && !bDivisorNegative)
     || (!bDividendNegative &&  bDivisorNegative))
  {
    quotient.Negate();
  }
  ....
}

このコードフラグメントは便宜上フォーマットされていますが、元の状態では非常に長く、読みにくい状態です。もう1つは、アナライザーがアドバイスするように、条件式を単純化する場合です。

if( bDividendNegative != bDivisorNegative)
{
  quotient.Negate();
}

コードが大幅に短くなり、条件式のロジックの理解が大幅に簡素化されました。

結論


お気づきかもしれませんが、不審な警告のほとんどは3つの場所で同じファイル内に複製されています。コードの複製は非常に悪い開発慣行です。コードのサポートが複雑になります。そして、そのようなコードでエラーが発生すると、コード全体にエラーが拡散するため、プログラムの安定性が急激に低下します。

他に興味深いものが公開され、再びこのリポジトリに戻ることを期待しましょう:)。とりあえず、PVS-Studioをダウンロードして自分のプロジェクトをチェックしたい人におすすめです。
