Manage multiple JDKs on Mac OS, Linux, and Windows WSL2

Hello again. In anticipation of the start of the Java Developer course , we have prepared for you a translation of an interesting article.





If you, like me, have to work simultaneously with several projects (both legacy and modern), and perhaps you are developing some kind of library or utility for users with different versions of Java, then you will understand the problem of switching between different JDK.







Nowadays, installing, managing, and switching between JDKs during development becomes a daunting task because a large number of developers still work with Java 8, although many are already migrating to Java 11. This problem can be solved in many ways. In this article we will consider some of them.



Manual way



You can download the distributions from the JDK vendor‚Äôs site and install them all manually (or just unzip them into some kind of folder like that $user/jdks



) and that‚Äôs all. But then you have to check and update each time JAVA_HOME



so that the JDK you need is there.



This can be done using bash scripts, bash functions, etc. What is the problem then? You can simply forget to call the script / function and realize that something went wrong only when you see it UnsupportedClassVersionError



.



If you prefer to do everything manually, you can dig through the Internet to find solutions. And this can be a good option for advanced users. But I use a couple of utilities for this. So let's look at them.



The best way



In my opinion, the ideal solution would be to combine two open source tools that have existed for quite some time, and perhaps you are familiar with at least one of them. But what you do not know is that they can be used together.



  • SDKMAN!
  • jEnv


Let's look at their sharing.



Install multiple JDKs





After installing SDKMAN! type: 



$ sdk list java


Find the desired vendor, version and enter: 



$ sdk install java <id>


Let's install four JDKs: the last two versions of Oracle OpenJDK and the two LTS versions of AdoptOpenJDK.



When installing, do not select the default JDK.



  1. AdoptOpenJDK build OpenJDK 8u252 - LTS
  2. AdoptOpenJDK build OpenJDK 11.0.7 - LTS
  3. Oracle OpenJDK 14 - Latest GA
  4. Oracle OpenJDK 15 - Early Access Build


These four versions should meet your needs for legacy Java 8 projects, for modern Java 11 projects, and for experimenting with innovations in Java 14 and Java 15.



Great. You have four JDKs installed locally using the SDKMAN! I think this utility is really convenient for installing JDK and, I hope, you liked it too.



All installed JDKs can be found in the following directory: 



$ cd /Users/bruno/.sdkman/candidates/java
$ ls
11.0.7.hs-adpt  14.0.1-open     15.ea.20-open   8.0.252.hs-adpt


If you don‚Äôt need to switch between different JDKs often, then this will be enough. You can use SDKMAN! to select JDK: 



$ sdk current java
Using java version 15.ea.19-open
$ sdk default java 15.ea.20-open
Default java version set to 15.ea.20-open


You can stop at this and use only SDKMAN !, but sooner or later, you will forget to switch the JDK in the same way as with a manual approach. Therefore, I prefer JDK auto-switching.



Manage multiple JDKs with jEnv







Despite the fact that SDKMAN! It‚Äôs convenient to install the JDK, but it does not help when switching automatically between different JDKs when switching from one project to another. Here jEnv will help us .



After you install several JDKs using SDKMAN! or in any other ways, you will need to add them to jEnv: 



$ cd ~/.sdkman/candidates/java
$ ls -1
11.0.7.hs-adpt 
14.0.1-open    
15.ea.20-open
8.0.252.hs-adpt
$ jenv add 15.ea.20-open
openjdk64-15-ea added
15-ea added
15-ea added
...
$ jenv versions
 system
 1.8
 1.8.0.252
 11.0
 11.0.7
 14.0
 14.0.1
* 15-ea (set by /Users/bruno/.jenv/version)
 openjdk64-1.8.0.252
 openjdk64-11.0.7
 openjdk64-14.0.1
 openjdk64-15-ea


Repeat the command jenv add



for the remaining three versions of the JDK.



jEnv is especially convenient for those who constantly use the command line. He will let you:



  1. Install the Java version globally for your system.
  2. Install the Java version for the current directory / project in which you are located.
  3. Install the Java version for the current shell.


jEnv uses shim and automatically manages the JAVA_HOME environment variable. Thus, after you configure your system, jEnv will switch the JDK version to the most suitable one, based on the above priorities and the current directory. Cool! It has other useful features - see the website for more information .



Auto switch between JDK



You now have Java 8, 11, 14, and 15-EA installed. So how do you switch between them? Let's customize the versions and test the switch.



First you need to specify the default global JDK in the system. For this, I usually use Early Access. Whenever I start working with a new project, I automatically use the upcoming release, which helps me identify potential problems that I can report to the OpenJDK project. You do it too ... 



$ jenv global 15


Now for any folder in which you are located, upon execution java -version



you will receive OpenJDK 15 EA.



If you have a project that should use Java 8, then go to the folder and run: 



$ jenv local 1.8




As a result, a file is created under the name .java-version



with the following contents: 



$ cat .java-version
1.8


This file tells jEnv which JDK to use when you are in this folder.



Finally, if you want to temporarily switch to another version of the JDK, then use jenv shell



.



There is a demo here that can explain all of the above and how auto-switching works.



After the article was published on Reddit, there were several interesting comments that are worth mentioning here:



  • SDKMAN! from version 5.8.1+ it supports sdk env and automatic switching is also available there.
  • direnv is very good and not tied to java. It does the same as jEnv, except it doesn't help you switch JDK easily. You must edit the files manually.
  • autoenv is another direnv alternative with the same limitations.
  • jabba is the most promising alternative to the combination of SDKMAN + jEnv, since it supports all their functionality and works in Windows (PowerShell), which is not supported by the other tools mentioned above.
