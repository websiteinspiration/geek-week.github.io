Schnauben oder Suricata. Teil 3: Schützen Sie das Büronetzwerk

In einem früheren Artikel haben wir darüber gesprochen, wie die stabile Version von Suricata unter Ubuntu 18.04 LTS ausgeführt wird. Das Konfigurieren von IDS auf einem Knoten und das Verbinden freier Regelsätze ist recht einfach. Heute werden wir uns ansehen, wie Sie das Unternehmensnetzwerk mit Suricata schützen können, das auf einem virtuellen Server installiert ist. Dies ist die häufigste Art von Angriff. Dafür benötigen wir VDS unter Linux mit zwei Kernen. Die Größe des Arbeitsspeichers hängt von der Auslastung ab: 2 GB reichen für jemanden aus, und für schwerwiegendere Aufgaben sind möglicherweise 4 oder sogar 6 erforderlich. Außerdem eine virtuelle Maschine mit der Möglichkeit zum Experimentieren: Sie können mit einer minimalen Konfiguration beginnen und die Ressourcen nach Bedarf erhöhen.

Das Entfernen von IDS zu einer virtuellen Maschine kann in erster Linie zum Testen erforderlich sein. Wenn Sie sich noch nie mit solchen Lösungen befasst haben, sollten Sie sich nicht beeilen, physische Hardware zu bestellen und die Netzwerkarchitektur zu ändern. Es ist besser, das System sicher und ohne zusätzliche Kosten zu betreiben, um die Computeranforderungen zu ermitteln. Es ist wichtig zu verstehen, dass der gesamte Unternehmensverkehr über einen einzigen externen Knoten geleitet werden muss: Um ein lokales Netzwerk (oder mehrere Netzwerke) mit installiertem IDS Suricata mit VDS zu verbinden, können Sie SoftEther verwenden , einen einfach zu konfigurierenden plattformübergreifenden VPN-Server, der eine starke Verschlüsselung bietet. Eine Office-Internetverbindung verfügt möglicherweise nicht über eine echte IP-Adresse. Daher ist es besser, sie auf VPS zu aktualisieren. Es gibt keine vorgefertigten Pakete im Ubuntu-Repository, Sie müssen die Software auch mit herunterladenProjektwebsite oder aus einem externen Repository im Launchpad- Dienst (wenn Sie ihm vertrauen): 



sudo add-apt-repository ppa:paskal-07/softethervpn
sudo apt-get update


Sie können die Liste der verfügbaren Pakete mit dem folgenden Befehl anzeigen: 



apt-cache search softether






Wir benötigen softether-vpnserver (der Server in der Testkonfiguration läuft auf VDS) sowie softether-vpncmd - Befehlszeilenprogramme, um ihn zu konfigurieren. 



sudo apt-get install softether-vpnserver softether-vpncmd


Zum Konfigurieren des Servers wird ein spezielles Befehlszeilenprogramm verwendet: 



sudo vpncmd






Wir werden nicht im Detail auf die Einstellung eingehen: Das Verfahren ist recht einfach, es ist in zahlreichen Veröffentlichungen gut beschrieben und gilt nicht direkt für das Thema des Artikels. Kurz gesagt, nach dem Start von vpncmd müssen Sie Element 1 auswählen, um zur Serververwaltungskonsole zu gelangen. Geben Sie dazu den Namen localhost ein und drücken Sie die Eingabetaste, anstatt den Namen des Hubs einzugeben. Das Administratorkennwort wird in der Konsole mit dem Befehl serverpasswordset festgelegt, der virtuelle DEFAULT-Hub (Befehl hubdelete) wird gelöscht und ein neues mit dem Namen Suricata_VPN erstellt. Das Kennwort wird ebenfalls festgelegt (Befehl hubcreate). Wechseln Sie als Nächstes mit dem Befehl hub Suricata_VPN zur Verwaltungskonsole des neuen Hubs, um eine Gruppe und einen Benutzer mit den Befehlen groupcreate und usercreate zu erstellen. Das Benutzerkennwort wird mit dem Benutzerkennwort festgelegt.



SoftEther unterstützt zwei Arten der Verkehrsübertragung: SecureNAT und Local Bridge. Die erste ist eine proprietäre Technologie zum Aufbau eines virtuellen privaten Netzwerks mit eigenem NAT und DHCP. SecureNAT erfordert weder TUN / TAP noch das Einrichten von Netfilter oder einer anderen Firewall. Das Routing wirkt sich nicht auf den Kern des Systems aus. Alle Prozesse sind virtualisiert und funktionieren auf jedem VPS / VDS, unabhängig vom verwendeten Hypervisor. Dies führt zu einer höheren Prozessorlast und einer geringeren Geschwindigkeit im Vergleich zum Local Bridge-Modus, der den virtuellen SoftEther-Hub mit einem physischen Netzwerkadapter oder einem TAP-Gerät verbindet.



Die Einrichtung ist in diesem Fall kompliziert, da das Routing auf Kernelebene mit Netfilter erfolgt. Unsere VDSs basieren auf Hyper-V. Im letzten Schritt erstellen wir eine lokale Bridge und aktivieren das TAP-Gerät mit dem Befehl bridgecreate Suricate_VPN -device: suricate_vpn -tap: yes. Nach dem Verlassen der Hub-Verwaltungskonsole wird im System eine neue Netzwerkschnittstelle angezeigt, der noch keine IP zugewiesen wurde: 



ifconfig






Als Nächstes müssen Sie das Paketrouting zwischen Schnittstellen aktivieren (IP-Weiterleitung), wenn es inaktiv ist: 



sudo nano /etc/sysctl.conf


Kommentieren Sie die folgende Zeile aus: 



net.ipv4.ip_forward = 1


Wir speichern die Änderungen in der Datei, beenden den Editor und wenden sie mit dem folgenden Befehl an: 



sudo sysctl -p


Als Nächstes müssen wir ein Subnetz mit Dummy-IPs für das virtuelle Netzwerk definieren (z. B. 10.0.10.0/24) und der Schnittstelle eine Adresse zuweisen: 



sudo ifconfig tap_suricata_vp 10.0.10.1/24


Dann müssen Sie die Netfilter-Regeln formulieren.



1. Zulassen, dass eingehende Pakete bei Bedarf an empfangenden Ports gesendet werden (proprietäres Protokoll SoftEther verwendet HTTPS und Port 443). 



sudo iptables -A INPUT -p tcp -m tcp --dport 443 -j ACCEPT
sudo iptables -A INPUT -p tcp -m tcp --dport 992 -j ACCEPT
sudo iptables -A INPUT -p tcp -m tcp --dport 1194 -j ACCEPT
sudo iptables -A INPUT -p udp -m udp --dport 1194 -j ACCEPT
sudo iptables -A INPUT -p tcp -m tcp --dport 5555 -j ACCEPT


2. Konfigurieren Sie NAT vom 10.0.10.0/24 Subnetz zum Haupt-IP-Server 



sudo iptables -t nat -A POSTROUTING -s 10.0.10.0/24 -j SNAT --to-source 45.132.17.140


3. Wir erlauben das Weiterleiten von Paketen aus dem Subnetz 10.0.10.0/24 



sudo iptables -A FORWARD -s 10.0.10.0/24 -j ACCEPT


4. Wir erlauben das Weiterleiten von Paketen für bereits hergestellte Verbindungen 



sudo iptables -A FORWARD -p all -m state --state ESTABLISHED,RELATED -j ACCEPT


Die Automatisierung des Prozesses beim Neustart des Systems mithilfe von Initialisierungsskripten wird den Lesern als Hausaufgabe überlassen.



Wenn Sie IP-Clients automatisch ausgeben möchten, müssen Sie auch einen DHCP-Dienst für die lokale Bridge installieren. Damit ist die Servereinrichtung abgeschlossen und Sie können zu den Clients gehen. SoftEther unterstützt viele Protokolle, deren Verwendung von den Fähigkeiten der LAN-Geräte abhängt. 



netstat -ap |grep vpnserver






Da unser Testrouter auch unter Ubuntu funktioniert, werden wir die Pakete softether-vpnclient und softether-vpncmd von einem externen Repository darauf installieren, um das proprietäre Protokoll zu verwenden. Sie müssen den Client starten: 



sudo vpnclient start


Zum Konfigurieren verwenden wir das Dienstprogramm vpncmd und wählen localhost als Computer aus, auf dem vpnclient ausgeführt wird. Alle Befehle werden in der Konsole ausgeführt: Sie müssen eine virtuelle Schnittstelle (NicCreate) und ein Konto (AccountCreate) erstellen.



In einigen Fällen müssen Sie die Authentifizierungsmethode mit den Befehlen AccountAnonymousSet, AccountPasswordSet, AccountCertSet und AccountSecureCertSet angeben. Da wir kein DHCP verwenden, wird die Adresse für den virtuellen Adapter manuell festgelegt.



Außerdem müssen wir ip forward (den Parameter net.ipv4.ip_forward = 1 in der Datei /etc/sysctl.conf) aktivieren und statische Routen konfigurieren. Bei Bedarf können Sie auf VDS mit Suricata die Portweiterleitung so konfigurieren, dass die im lokalen Netzwerk installierten Dienste verwendet werden. In dieser Hinsicht kann die Vereinigung von Netzwerken als vollständig angesehen werden.

Im vorherigen Artikel haben wir über zwei IDS-Betriebsmodi gesprochen: über die NFQUEUE-Warteschlange (NFQ-Modus) und über die Nullkopie (AF_PACKET-Modus). Die zweite erfordert zwei Schnittstellen, unterscheidet sich jedoch in der höheren Geschwindigkeit - wir werden sie verwenden. Der Parameter wird standardmäßig in / etc / default / suricata festgelegt. Wir müssen auch den Abschnitt vars in /etc/suricata/suricata.yaml bearbeiten und das virtuelle Subnetz dort als Ihr Heim-Subnetz festlegen.







Verwenden Sie den Befehl, um IDS neu zu starten: 



systemctl restart suricata


Die Lösung ist fertig. Jetzt müssen Sie sie möglicherweise auf Widerstand gegen die Aktionen von Angreifern überprüfen.



Wir simulieren Angriffe



Es kann verschiedene Szenarien für die Kampfnutzung eines externen IDS-Dienstes geben:



Schutz vor DDoS-Angriffen (Hauptzweck)



Es ist schwierig, eine solche Option im Unternehmensnetzwerk zu implementieren, da die Analysepakete über die Internetschnittstelle an das System gesendet werden müssen. Selbst wenn IDS sie blockiert, kann unechter Verkehr einen Datenkanal setzen. Um dies zu vermeiden, müssen Sie einen VPS mit einer ziemlich effizienten Internetverbindung bestellen, die den gesamten Datenverkehr im lokalen Netzwerk und den gesamten externen Datenverkehr überspringen kann. Dies ist oft einfacher und billiger als die Erweiterung des Bürokanals. Alternativ sind spezielle Dienste zum Schutz vor DDoS zu erwähnen. Die Kosten für ihre Dienste sind vergleichbar mit den Kosten für einen virtuellen Server, erfordern jedoch keine aufwändige Einrichtung, es gibt jedoch Nachteile: Für das Geld erhält der Client nur Schutz vor DDoS, während sein eigenes IDS nach Ihren Wünschen konfiguriert werden kann.



Schutz vor externen Angriffen anderer Art



Suricata ist in der Lage, Versuche zu bewältigen, verschiedene Schwachstellen in Unternehmensnetzwerkdiensten auszunutzen, auf die über das Internet zugegriffen werden kann (Mailserver, Webserver und Webanwendungen usw.). In der Regel werden dazu IDS nach den Grenzgeräten im LAN installiert, der Take-out hat jedoch das Existenzrecht.



Schutz vor internen Eindringlingen



Trotz aller Bemühungen des Systemadministrators können Computer im Unternehmensnetzwerk mit Malware infiziert sein. Außerdem erscheinen manchmal Hooligans im LAN und versuchen, bestimmte illegale Operationen durchzuführen. Suricata kann dabei helfen, solche Versuche zu blockieren. Es ist jedoch besser, es im Umkreis zu installieren, um das interne Netzwerk zu schützen, und es zusammen mit einem verwalteten Switch zu verwenden, der den Datenverkehr auf einen Port spiegeln kann. Das externe IDS ist auch in diesem Fall nicht nutzlos - zumindest kann es Versuche von im LAN lebender Malware abfangen, einen externen Server zu kontaktieren.



Zuerst erstellen wir einen weiteren Test, der VPS angreift, und auf dem lokalen Netzwerkrouter wird Apache mit der Standardkonfiguration ausgelöst. Anschließend leiten wir den 80. Port vom IDS-Server an ihn weiter. Als nächstes simulieren wir einen DDoS-Angriff vom angreifenden Knoten. Laden Sie dazu von GitHub herunter, kompilieren Sie ein kleines xerxes-Programm und führen Sie es auf dem angreifenden Knoten aus (möglicherweise muss das gcc-Paket installiert werden): 



git clone https://github.com/Soldie/xerxes-DDos-zanyarjamal-C.git
cd xerxes-DDos-zanyarjamal-C/
gcc xerxes.c -o xerxes 
./xerxes 45.132.17.140 80
