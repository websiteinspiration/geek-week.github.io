1.デフォルト値で列を追加しないでください


ゴールデンPostgreSQLルール：本番環境でテーブルに列を追加するときは、デフォルト値を指定しないでください。

列を追加するには、読み取りと書き込みの両方をブロックする非常に強力なテーブルロックが必要です。デフォルト値で列を追加すると、PostgreSQLはテーブル全体を上書きして各行のデフォルト値を入力します。大きなテーブルでは数時間かかる場合があります。同時に、すべてのリクエストがブロックされるため、データベースは利用できなくなります。

こんなことしないで：

--     ,       (?)
ALTER TABLE items ADD COLUMN last_update timestamptz DEFAULT now();

次のようにしてください：

-- select, update, insert  delete ,      ()
ALTER TABLE items ADD COLUMN last_update timestamptz;
-- select  insert ,  update  delete ,   
UPDATE items SET last_update = now();

または、さらに良いことに、少量を更新することにより、閉塞updateをdelete長期間にわたって回避します。次に例を示します。

do {
  numRowsUpdated = executeUpdate(
    "UPDATE items SET last_update = ? " +
    "WHERE ctid IN (SELECT ctid FROM items WHERE last_update IS NULL LIMIT 5000)",
    now);
} while (numRowsUpdate > 0);

この方法では、ユーザーへの干渉を最小限に抑えて、新しい列を追加してデータを設定できます。

3.非ブロッキングインデックスの作成を使用する


PostgreSQLのもう1つの黄金律：常に非ブロッキングインデックスの作成を使用します。
大規模なデータセットのインデックスの作成には数時間または数日かかる場合があり、通常のコマンドはコマンドの実行中CREATE INDEXすべてのレコードをロックします。SELECTそれはsをブロックしないという事実にもかかわらず、それはまだかなり悪いです、そしてより良い方法があります：CREATE INDEX CONCURRENTLY。

こんなことしないで：

--   
CREATE INDEX items_value_idx ON items USING GIN (value jsonb_path_ops);

代わりに、次のようにします。

--    DDL
CREATE INDEX CONCURRENTLY items_value_idx ON items USING GIN (value jsonb_path_ops);

非ブロッキングインデックスの作成には欠点があります。何か問題が発生しても、ロールバックは行われず、不完全な（「無効な」）インデックスが残ります。これが発生しても、心配しないで、実行してください
DROP INDEX CONCURRENTLY items_value_idx
もう一度作成してみてください。

4.可能な限り遅く積極的なロックを使用する


積極的なテーブルロックを受け取るコマンドを実行する必要がある場合は、クエリをできるだけ長く継続できるように、トランザクションでできるだけ遅くこれを実行してください。

たとえば、テーブルの内容を完全に置き換えたい場合などです。こんなことしないで：

BEGIN;
--     :
TRUNCATE items;
-  :
\COPY items FROM 'newdata.csv' WITH CSV 
COMMIT;

代わりに、データを新しいテーブルにロードしてから、古いテーブルを置き換えます。

BEGIN;
CREATE TABLE items_new (LIKE items INCLUDING ALL);
--  :
\COPY items_new FROM 'newdata.csv' WITH CSV
--     :
DROP TABLE items;
ALTER TABLE items_new RENAME TO items;
COMMIT;

問題が1つあります。最初からレコードをブロックしなかったため、リセットした時点で古い要素のテーブルが変更されている可能性があります。これを防ぐために、テーブルを書き込み用に明示的にロックできますが、読み取り用にはロックできません。

BEGIN;
LOCK items IN EXCLUSIVE MODE;
...

ブロッキングを自分の手で行う方が良い場合もあります。

5.最小限のブロックで主キーを追加する


多くの場合、テーブルに主キーを追加することをお勧めします。たとえば、論理レプリケーションを使用する場合、またはCitus Warpを使用してデータベースを移行する場合。

Postgresではを使用して主キーを非常に簡単に作成できますが、主キーALTER TABLEのインデックスを作成すると、テーブルが大きい場合に時間がかかり、すべてのリクエストがブロックされます。

ALTER TABLE items ADD PRIMARY KEY (id); --

幸いなことに、最初にCREATE UNIQUE INDEX CONCURRENTLY一意のインデックスを主キーとして使用することで、すべての困難な作業を最初に行うことができます。これは迅速な操作です。

CREATE UNIQUE INDEX CONCURRENTLY items_pk ON items (id); --   ,    
ALTER TABLE items ADD CONSTRAINT items_pk PRIMARY KEY USING INDEX items_pk;  --  ,

主キーの作成を2つの段階に分割しても、実際にはユーザーに影響はありません。

7.コマンドを配置してデッドロックを回避する


PostgreSQLをしばらく使用している場合は、次のようなエラーが発生している可能性があります。

ERROR:  deadlock detected
DETAIL:  Process 13661 waits for ShareLock on transaction 45942; blocked by process 13483.
Process 13483 waits for ShareLock on transaction 45937; blocked by process 13661.

これは、同時トランザクションが同じロックを異なる順序で取得する場合に発生します。たとえば、1つのトランザクションが次のコマンドを実行します。

BEGIN;
UPDATE items SET counter = counter + 1 WHERE key = 'hello'; --    hello
UPDATE items SET counter = counter + 1 WHERE key = 'world'; --    world
END;

同時に、別のトランザクションが同じコマンドを発行することがありますが、順序は異なります。

BEGIN
UPDATE items SET counter = counter + 1 WHERE key = 'world'; --    world
UPDATE items SET counter = counter + 1 WHERE key = 'hello';  --    hello
END;

トランザクションのこれらのブロックが同時に実行されると、それらが互いに待機してスタックし、終了することはありません。 Postgresは約1秒でこの状況を認識し、もう一方を完了するために一方のトランザクションをキャンセルします。これが発生した場合は、アプリケーションを調べて、トランザクションが常に同じ順序で実行されることを確認できるかどうかを確認する必要があります。両方のトランザクションが最初に変更されたhello場合world、最初のトランザクションはhello他のロックを取得する前に2番目のトランザクションをロックします。
