Zimbra Collaboration Suite Open-Source EditionでのSSLセキュリティ設定の改善

暗号化の信頼性は、ビジネスに情報システムを使用する際の最も重要な指標の1つです。なぜなら、彼らは毎日、大量の機密情報の転送に関与しているからです。 SSL接続の品質を評価するための一般的なツールは、Qualys SSL Labsの独立したテストです。このテストは誰でも実行できるため、SaaSプロバイダーにとって、このテストのスコアが最大であることが特に重要です。 SaaSプロバイダーだけでなく、通常の企業もSSL接続の品質を管理します。彼らにとって、このテストは潜在的な脆弱性を特定し、サイバー犯罪者のすべての抜け穴を事前に塞ぐ絶好の機会です。





Zimbra OSEでは、2種類のSSL証明書が許可されています。 1つ目は自己署名証明書で、インストール時に自動的に追加されます。この証明書は無料で、使用期間に制限はありません。つまり、Zimbra OSEのテストや、内部ネットワーク内でのその使用のみに最適です。ただし、Webクライアントに入ると、ユーザーはブラウザからこの証明書が信頼できないという警告が表示され、Qualys SSL Labsによるテストは確実にサーバーで失敗します。



2つ目は、証明機関によって署名された商用SSL証明書です。このような証明書はブラウザで簡単に認識され、Zimbra OSEの商用運用で一般的に使用されます。商用証明書が正しくインストールされた直後、Zimbra OSE 8.8.15は、Qualys SSL LabsのテストでグレードAを示しています。これは素晴らしい結果ですが、私たちの目標はA +の結果を達成することです。











Zimbra Collaboration Suite Open-Source Editionを使用しているときにQualys SSL Labsのテストで最大のマークを達成するには、いくつかの手順を実行する必要があります



。1. Diffie-Hellmanプロトコルのパラメーターを増やす



デフォルトでは、OpenSSLを使用するすべてのZimbra OSE 8.8.15コンポーネントでは、Diffie-Hellmanプロトコル設定は2048ビットです。原則として、これはQualys SSL LabsのテストでA +スコアを取得するのに十分です。ただし、古いバージョンからアップグレードする場合は、パラメーターの値が低くなる可能性があります。したがって、更新が完了した後、コマンドzmdhparam set -new 2048を実行して、Diffie-Hellmanプロトコルパラメータを許容可能な2048ビットに増やし、同じコマンドを使用して、パラメータ値を3072または4096ビットに増やすことができます。一方、生成時間は、メールサーバーのセキュリティレベルにプラスの影響を与えます。



2.使用されている暗号の推奨リストを含める



デフォルトでは、Zimbra Collaborataion Suite Open-Source Editionは、安全な接続を通過するデータを暗号化する強力な暗号と弱い暗号を幅広くサポートしています。ただし、弱い暗号の使用は、SSL接続のセキュリティをチェックするときに重大なマイナスになります。これを回避するには、使用する暗号のリストを構成する必要があります。



これを行うには、次のコマンドを使用します。zmprov mcf zimbraReverseProxySSLCiphers 'ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-AES256-GCM-SHA384:DHE-RSA-AES128-GCM-SHA256:DHE-DSS-AES128-GCM-SHA256:kEDH+AESGCM:ECDHE-RSA-AES128-SHA256:ECDHE-ECDSA-AES128-SHA256:ECDHE-RSA-AES128-SHA:ECDHE-ECDSA-AES128-SHA:ECDHE-RSA-AES256-SHA384:ECDHE-ECDSA-AES256-SHA384:ECDHE-RSA-AES256-SHA:ECDHE-ECDSA-AES256-SHA:DHE-RSA-AES128-SHA256:DHE-RSA-AES128-SHA:DHE-DSS-AES128-SHA256:DHE-RSA-AES256-SHA256:DHE-DSS-AES256-SHA:DHE-RSA-AES256-SHA:AES128-GCM-SHA256:AES256-GCM-SHA384:AES128:AES256:HIGH:!aNULL:!eNULL:!EXPORT:!DES:!MD5:!PSK:!RC4'







推奨される暗号のこのセットは、このコマンドにすぐに含まれます。そのおかげで、コマンドは信頼できる暗号をリストにすぐに含め、信頼できない暗号を除外できます。あとは、zmproxyctl restartコマンドを使用してリバースプロキシでノードを再起動するだけです。再起動後、変更が有効になります。



何らかの理由でこのリストが適さない場合は、コマンドを使用して、リストからいくつかの弱い暗号を削除できますzmprov mcf +zimbraSSLExcludeCipherSuites



。したがって、たとえば、zmprov mcf +zimbraSSLExcludeCipherSuites TLS_RSA_WITH_RC4_128_MD5 +zimbraSSLExcludeCipherSuites TLS_RSA_WITH_RC4_128_SHA +zimbraSSLExcludeCipherSuites SSL_RSA_WITH_RC4_128_MD5 +zimbraSSLExcludeCipherSuites SSL_RSA_WITH_RC4_128_SHA +zimbraSSLExcludeCipherSuites TLS_ECDHE_RSA_WITH_RC4_128_SHA



RC4暗号の使用を完全に排除するチームです。 AESおよび3DES暗号でも同じことができます。



3. HSTSの有効化



接続の暗号化とTLSセッションの回復を強制する有効なメカニズムは、Qualys SSL Labsからテストで最高のスコアを取得するための前提条件でもあります。それらを有効にするには、コマンドを入力する必要がありますzmprov mcf +zimbraResponseHeader "Strict-Transport-Security: max-age=31536000"



。このコマンドは必要なヘッダーを構成に追加します。新しい設定を有効にするには、zmcontrol restartコマンドを使用してZimbra OSEを再起動する必要があります。



すでにこの段階で、Qualys SSL LabsのテストはA +スコアを示しますが、サーバーのセキュリティをさらに改善したい場合は、いくつかの方法をとることができます。







たとえば、プロセス間接続の強制暗号化を有効にしたり、Zimbra OSEサービスに接続するときに強制暗号化を有効にしたりできます。プロセス間接続を確認するには、次のコマンドを入力します。 



zmlocalconfig -e ldap_starttls_supported=1
zmlocalconfig -e zimbra_require_interprocess_security=1
zmlocalconfig -e ldap_starttls_required=true


強制暗号化を有効にするには、次のように入力する必要があります。 



zmprov gs `zmhostname` zimbraReverseProxyMailMode
zmprov ms `zmhostname` zimbraReverseProxyMailMode https

zmprov gs `zmhostname` zimbraMailMode
zmprov ms `zmhostname` zimbraMailMode https

zmprov gs `zmhostname` zimbraReverseProxySSLToUpstreamEnabled
zmprov ms `zmhostname` zimbraReverseProxySSLToUpstreamEnabled TRUE




これらのコマンドのおかげで、プロキシサーバーとメールサーバーへのすべての接続が暗号化され、これらすべての接続がプロキシされます。







したがって、推奨事項に従って、SSL接続のセキュリティのテストで最高点を達成できるだけでなく、Zimbra OSEインフラストラクチャ全体のセキュリティを大幅に向上させることもできます。
