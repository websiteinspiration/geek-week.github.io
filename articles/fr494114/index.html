Comment réduire le nombre et augmenter la lisibilité du code dans react-redux, redux-saga

Dans cet article, je voudrais partager mon expérience de l'utilisation du bundle react-redux et redux-saga , ou plutôt du «vélo» que j'utilise pour réduire la quantité du même type de code et simplifier sa perception.

Qu'est-ce qui ne me convenait pas


Les bibliothèques react-redux et redux-saga sont simples, flexibles et pratiques, mais ont une redondance de code. Les principaux éléments sont:

  1. Usines événementielles


    const actionCreatorFactory = type => payload => ({ type, payload });

export const INITIALIZE = 'INITIALIZE';

export const ON_FIELD_CHANGE = 'ON_FIELD_CHANGE';
export const onFieldChange = actionCreatorFactory(ON_FIELD_CHANGE);

export const HANDLE_FIELD = 'HANDLE_FIELD';
export const handleField = actionCreatorFactory(HANDLE_FIELD);

export const GO_TO_NEXT_STEP = 'GO_TO_NEXT_STEP';
export const goToNextStep = actionCreatorFactory(GO_TO_NEXT_STEP);

    À ce titre, plusieurs choses me confondent:

    - une description des types d'événements. Dans cet exemple, vous pouvez certainement vous passer de constante, mais vous devez toujours passer son type en usine, qui sera identique au nom des événements créés dans la notification de chameau ( camelCase ).

    - si vous avez oublié la structure de la charge utile , pour vous en souvenir, vous devez vous rendre dans reducer / saga , où cet événement est utilisé, et voir ce qui doit y être transféré

  2. Réducteurs


    import getInitialState, {
  formDataInitialState as initialState,
} from '../helpers/initialState';
import { HANDLE_FIELD_DONE, ON_FIELD_CHANGE, RESET } from '../actionCreators';

export default (state = initialState, { type, payload }) => {
  switch (type) {
    case RESET: {
      return getInitialState().formDataInitialState;
    }

    case ON_FIELD_CHANGE: {
      const { name } = payload;
      return {
        ...state,
        [name]: '',
      }
    }

    case HANDLE_FIELD_DONE: {
      const { name, value } = payload;
      return {
        ...state,
        [name]: value,
      }
    }
  }

  return state;
};

  3. Sagas


    import { all, put, select, fork, takeEvery } from 'redux-saga/effects';
import { runServerSideValidation } from '../actionCreators';
import { HANDLE_FIELD } from '../actionCreators';

function* takeHandleFieldAction() {
  yield takeEvery(HANDLE_FIELD, function*({ payload }) {
    const { validation, formData } = yield select(
      ({ validation, formData }) => ({
        validation: validation[payload.name],
        formData,
      })
    );

    const valueFromState = formData[payload.name];

    if (payload.value !== valueFromState) {
      const { name, value } = payload;
      const { validator } = validation.serverValidator;
      yield put(
        runServerSideValidation({
          name,
          value,
          validator,
          formData,
        })
      );
    }
  });
}

export default function* rootSaga() {
  yield all([fork(takeHandleFieldAction())]);
}

    Dans les sagas, le plus gros problème est la difficulté de lecture. Juste un coup d'œil rapide pour comprendre ce qui rend la saga difficile. L'une des raisons à cela est la nidification profonde. Vous pouvez bien sûr retirer les ouvriers pour réduire la profondeur, mais alors la redondance augmente.

    import { all, put, select, fork, takeEvery } from 'redux-saga/effects';
import { runServerSideValidation } from '../actionCreators';
import { HANDLE_FIELD } from '../actionCreators';

function* takeHandleFieldWorker({ payload }) {
    const { validation, formData } = yield select(
      ({ validation, formData }) => ({
        validation: validation[payload.name],
        formData,
      })
    );

    const valueFromState = formData[payload.name];

    if (payload.value !== valueFromState) {
      const { name, value } = payload;
      const { validator } = validation.serverValidator;
      yield put(
        runServerSideValidation({
          name,
          value,
          validator,
          formData,
        })
      );
  }
}

function* takeHandleFieldWatcher() {
  yield takeEvery(HANDLE_FIELD, takeHandleFieldWorker);
}

export default function* rootSaga() {
  yield all([fork(takeHandleFieldWatcher())]);
}

    return state;

  3. import { all, put, select, } from 'redux-saga/effects';
import { runServerSideValidation } from '../actionCreators';
import { HANDLE_FIELD } from '../actionCreators';
import { sagas, takeEvery, filterActions } from 'sweet-redux-saga';

@sagas()
class MySagas {
  @takeEvery([HANDLE_FIELD])
  @filterActions(
    ({state, payload }) => state.formData[payload.name] === payload.value
  )
  * takeHandleFieldAction({ payload }) {
    const { validation, formData } = yield select(({ validation, formData }) => ({
      validation: validation[payload.name],
      formData,
    }));

    const { name, value } = payload;
    const { validator } = validation.serverValidator;
    yield put(
      runServerSideValidation({
        name,
        value,
        validator,
        formData,
      })
    );
  }
}

export default function* rootSaga() {
  yield all([
    new MySagas(),
  ]);
}

    créer une classe avec l'annotation sagas (). Lors de la création d'une instance de la classe, nous obtenons un générateur de fonctions qui appelle tous les champs de la classe marqués avec des annotations takeEvery ([... [actionTypes]]) ou takeLatest ([... [actionTypes]]) dans un thread séparé:

    function* mySagas() {
  yield all([fork(mySagas.takeHandleFieldAction())]);
}

    L'annotation filterActions ({state, type, payload}) peut également être utilisée avec des champs , dans ce cas la saga ne sera appelée que si la fonction retourne true .

Conclusion


En général, en utilisant ces annotations simples, les sagas sont devenues plus claires et plus propres, et la quantité de code nécessaire pour ajouter une nouvelle logique a été réduite. La navigation dans les projets est devenue plus facile.

J'ai fait ces annotations dans le paquet sweet-redux-saga . S'il existe d'autres solutions, je serai heureux de partager avec moi.
