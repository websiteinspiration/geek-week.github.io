HTTP over UDP-利点のあるQUICプロトコルを使用



QUIC（Quick UDP Internet Connections）は、TCP、TLS、HTTP / 2のすべての機能をサポートし、それらの問題のほとんどを解決するUDP上のプロトコルです。新しいまたは「実験的な」プロトコルと呼ばれることが多いですが、実験の段階を長く生き延びてきました。開発は7年以上続いています。この間、プロトコルは標準になることができませんでしたが、それでもまだ広く使用されていました。たとえば、GoogleやFacebookなどの巨人は、QUICを使用してトラフィックを高速化し、モバイルネットワークのレイテンシを削減します。IETFは、HTTP / 3標準の基礎としてプロトコルの分岐を発表しました（HTTP / 2は44.8％のサイトしか使用しないという事実にもかかわらず）。

概念


QUICは、元々損失率が低い有線ネットワーク用に設計されたレガシーTCPの代替として開発されました。 TCPはパケットを順番に配信するため、1つのパケットを失うと、キュー全体（行頭ブロッキング）が上昇し、接続の品質と安定性に悪影響を及ぼします。大規模な損失を回避するために、セルラーネットワークは大きなバッファーの使用に頼っています。これにより、冗長性と偽陰性のプロトコル反応（bufferbloat）が発生します。さらに、TCPは接続の確立に多くの時間を費やしています。SYN/ ACKとTLSの要求は別々に送信されるため、QUICのように1回ではなく3回のラウンドトリップが必要です。



QUICはTCPの置き換えとTLS 1.3の実装を組み合わせているため、すべての接続は常に暗号化され、そのようなトラフィックの解読はHTTPS経由の場合より簡単ではありません。さらに、QUICはアプリケーションレベルで実装されています。これは、TCPスタックを完全に置き換えるのに永遠にかかるためです。

HTTP / 2の多重化サポートにも関わらず、パケットを順番に配信する必要があるため、行頭ブロッキングの問題が残っていました。QUICはUDPの上に実装されているため、原則としてブロッキングはありません。また、パケットが永続的に失われることはなく、番号が付けられ、「ネイバー」の一部を含むことができるため、冗長性が提供されます。さらに、QUICは、モノリシックキューを、同じ接続内の異なるタイプのリクエスト用のいくつかのスレッドに分割します。したがって、パケットが失われた場合、問題は1つのキューでのみ発生する可能性があります（たとえば、特定のファイルを転送する場合）。



使用する


QUICはもともとGoogleが内部で開発したもので、主に内部で使用するために調整されました。2013年に彼はIETFに標準化のために転送され（現在も進行中）、今では誰もが欠けているものを提案することでプロトコルの開発に参加できます。IETFワーキンググループは毎年会議を開催し、その中で新しい標準が承認され、イノベーションについて議論されます。このQUIC実装は主要な実装と見なされており、HTTP / 3標準が認定されていることを基本としています。

これまでのところ、HTTP / 3をメインプロトコルとして有効にすることについては触れていません。これは、まだ完了しておらず、ほとんどサポート



されていないためです。ただし、QUICは、アプリケーションとサーバー間のトランスポートとして実装できます。

当然のことながら、Google CloudはGoogleによって開発されたプロトコルでうまく機能しますが、代替手段は何ですか？

Nginx


少し前まで、CloudFlare は nginx（デフォルトではHTTP / 3を認識していない）をQuicheツールとクロスしようとしました。実装は、インストールチュートリアルが含まれている単一の.patchファイルとして利用できます。

curl -O https://nginx.org/download/nginx-1.16.1.tar.gz
tar xvzf nginx-1.16.1.tar.gz
git clone --recursive https://github.com/cloudflare/quiche
cd nginx-1.16.1
patch -p01 < ../quiche/extras/nginx/nginx-1.16.patch

ここでは、必要に応じてモジュールを接続できます。

./configure                          	\
   	--prefix=$PWD                       	\
   	--with-http_ssl_module              	\
   	--with-http_v2_module               	\
   	--with-http_v3_module               	\
   	--with-openssl=../quiche/deps/boringssl \
   	--with-quiche=../quiche
 make

あとは、HTTP / 3サポートを有効にするだけです。

events {
    worker_connections  1024;
}

http {
    server {
        # Enable QUIC and HTTP/3.
        listen 443 quic reuseport;

        # Enable HTTP/2 (optional).
        listen 443 ssl http2;

        ssl_certificate      cert.crt;
        ssl_certificate_key  cert.key;

        # Enable all TLS versions (TLSv1.3 is required for QUIC).
        ssl_protocols TLSv1 TLSv1.1 TLSv1.2 TLSv1.3;

        # Request buffering in not currently supported for HTTP/3.
        proxy_request_buffering off;

        # Add Alt-Svc header to negotiate HTTP/3.
        add_header alt-svc 'h3-27=":443"; ma=86400';
    }
}

通常のブラウザでは、まだHTTP / 3経由で接続することはできませんが、Chrome Canaryを使用してフラグを付けて実行--enable-quicし、サーバーにアクセスするか、たとえばquic.rocksウェブサイトにアクセスして、開発者ツールで接続タイプを確認できます：

HTTP / 3の代わりに、と書かれてhttp2+quic/99いますが、それは本質的に同じものです。

結論




QUICへの関心は不安定ですが、それを標準化する作業が進んでいます。新しいプロトコルの実装はほぼ毎月登場し、毎年ますます多くの開発者が未来はQUICにあると確信しています。TCPスタックの将来のバージョンにプロトコルを含めることも可能です。つまり、遅かれ早かれ、インターネット全体がより安定したより高速な接続に移行します。

すでに、インフラストラクチャにQUICインタラクションを設定するか、ブラウザに提供することもできます。これらはすべて、プロトコルサポートを追加することを計画しており、カヌースを使用した悲しい統計がさらに楽しくなります。
