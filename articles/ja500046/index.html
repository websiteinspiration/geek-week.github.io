JetBrains Rider-アンリアルエンジンに対応

こんにちはHabr！

先週、バージョン2020.1のリリース後、すべてのデスクトップ製品で別の大きなイベントが発生しました。Riderfor Unreal Engineの試用版への公開アクセスを開始しました。現時点では、これは別の製品であり、Riderの開発環境のバージョンですが、C ++とUnreal Engineをサポートしています。 やめて！C ++開発環境。もう一つ！順番に整理しましょう。





小さな話から始めましょう。数年前、JetBrainsでC ++ツールを作るすべての人を集めて、ユーザーがさまざまなオファーをナビゲートするのがいかに難しいかについて話し合いました。結局のところ、CLion、ReSharper C ++、およびAppCodeでのC ++のサポートがあります。数時間の議論の後、次の結論に達しました。

  • ユニバーサルツールは魅力的であるように見えますが、ターゲティングの拡大、統合されたサードパーティツールの数の増加、サポートされているユースケースでは、利便性、インターフェースの直感性、および豊富な機会のバランスを見つけることが非常に難しくなります。
  • ソフトウェア開発の世界には、長い間、開発言語に明確な境界がありませんでした。多くのプロジェクトは多言語対応です。さらに重要なのは、そこで使用されるソフトウェアとテクノロジーの範囲です。

その結果、C ++のツールでユーザーエクスペリエンスを統合することを決定しましたが、ソフトウェア開発のさまざまな領域に異なるツールを（共通のカーネルまたは共通のプラットフォームで）作成するアプローチが適切であることに同意しました。実際には、次のようになります。

  • ReSharper C++ — , Windows, , Visual Studio. C++ C#. Visual Studio, ReSharper, C++.

それで、ライダーはそれと何をしているのでしょうか？ C ++に関連する.NETのクロスプラットフォーム開発環境とは何ですか？実際には、Riderユーザーの約30％がゲームスタジオとゲーム開発会社です。 Riderは正確にUnityの開発環境として広く普及しているため、理解できます。ライダーチームがこのゲームエンジン専用に楽器をカスタマイズしたのは、今年が初めてではありません。その結果は、Unityプログラマーだけでなく、テクノロジー自体の観点から長く実りあるやり取りを行ってきたUnity Technologies自体にも好まれています。

ゲーム開発の分野で多くのユーザーに話を伺ったところ、多くのスタジオには明確な分離がないことに気付きました。Unity上でのみ、またはUnreal Engine上でのみゲームを作成してください。今日、彼らは1つのテクノロジーでプレーし、明日は別のテクノロジーでプレーします。または、あるチームがアンリアルエンジンを使用し、他のUnity、または独自のカスタムエンジンを使用します。開発者と企業は一般に、異なる開発環境間で「ジャンプ」することを本当に嫌うことは明らかです。そしてここで、RiderをUnityで成功させることができれば、さらに進んでゲーム開発の世界で一般的に成功させることができると判断しました。したがって、Riderのアイデアは、Game Devの普遍的な開発環境として生まれました。

Rider for Unreal Engineの構成要素


イベントのさらなる展開は、Riderが構築されているテクノロジーに精通している人にとっては非常に明白です。Riderは、IntelliJプラットフォームに基づくフロントエンドパーツとReSharperに基づくバックエンドパーツで構成されています。すべての言語サポートはバックエンドで機能します。したがって、同じテクノロジーを使用して、ReSharper C ++の既存のC ++およびUnreal EngineサポートをRiderに接続しました。具体的には、私はさらに実装する必要がありました：

  • エンジンバージョン4.22〜4.24のUnreal Editor（Rider Source Code Access）で開発環境としてRiderを選択するためのプラグイン。バージョン4.25以降、この機能はすでにUEエンジン自体に含まれています。
  • Rider 用の UnrealLink プラグインおよびRiderLink for Unreal Editorは、Riderのブループリントとの統合を提供し、一般にRiderとUnreal Editor間の通信を担当します。
  • デバッガ

Rider for Unreal EngineのプレビューがWindowsでのみ利用可能になり、.slnファイルを開き、Microsoftのコンパイラーによるビルドに依存していることは注目に値します。今後、リリースまでに3つのプラットフォームすべてに拡張し、.uprojectのサポートをメインの設計モデルとして実装する予定です。

C ++サポート


繰り返しますが、ReSharper C ++のすべての言語サポート機能がRider for Unreal Engine Previewで利用可能になりました。以下が含まれます：

  • 約170の異なるコード検査と50以上のクイック修正。
  • シンボルのすべてのコンテキスト使用を考慮に入れたリファクタリング（名前の変更、メソッドシグネチャの変更、メソッドの削除、フィールド/変数/ typedefの追加、変数またはtypedefの置換（インライン）など）。
  • コード生成（ゲッター/セッターメソッド、コンストラクター、比較演算子、コードド​​キュメントテンプレート）。
  • コードの記述、ナビゲーション、フォーマット、ネーミングスタイル、ソート#includeディレクティブのサポートに役立つアクション。



HLSL、C＃、uproject / uplugin方言のサポート


今年、ReSharper C ++でHLSLシェーダーを作成するための言語サポートに取り組み始め、すぐにRider for Unreal Engineの初期プレビューに参加しました。サポートには、構文の強調表示、ドキュメントのツールチップ、パラメーターの名前とタイプのツールチップ、ナビゲーションアクション、オートコンプリート、仮想ファイルパスのサポート、さらにはリファクタリングが含まれます。

Rider for Unreal Engineは.NETの開発環境に基づいているため、言語サポートはBuild.csファイルとTarget.csファイルの両方で機能します。

そして最後に、言語の方言をサポートすることになると、Rider for Unreal Engineは.uproject / .upluginファイルの方言を理解し、オートコンプリートおよびドキュメントツールのオプションを提供します。



ブループリントの統合


ブループリントファイルはバイナリデータであり、通常はUnreal Editor内のビジュアルエディターで編集されます。これらのファイルのオブジェクトはC ++のクラスから継承され、ゲームのC ++部分のプロパティを再定義します。ここでRider for Unreal Engineは、必要なすべてのBlueprintsファイルを読み取り、C ++コードエディターでこれらのリンクを表示する独自の開発環境



です。たとえば、Unreal Engineエディターでプロパティの値を変更してアセットを保存すると、ここに値が表示されますまた、Riderでも自動的に更新されます（ウォッチャーがハングしてアセットファイルを変更します）。



ファイルを保存しなくても、これがすぐに機能することを期待しています（UnrealLinkプラグインでこの準備が行われています）。

Find usages呼び出しには、C ++コードでの使用だけでなく、Blueprintsファイルも含まれます。このような用途をダブルクリックすると、Unrealエディタが開きます。

UE4のリフレクションメカニズムを理解する


アンリアルエンジンでのリフレクションは、特別なマクロ（UCLASS、UFUNCTION、UPROPERTYなど）を使用して実装されます。ライダーは、そのようなマクロのパラメーターが単なるテキストではないことを知っています。 ReSharper C ++およびRiderのC ++パーサーは、Unrealヘッダービルドツールを起動しなくても（つまり、.generated.hファイルの内容が実際に生成される前に）、これらのマクロの意味を本当に「理解」できます。

ちなみに、.generated.hファイルについて言えば、ReSharper C ++とRiderは、自動的に追加された欠落している#includeディレクティブを.generated.hファイルに接続する前に厳密に挿入する必要があることを知っています。また、リファクタリングの名前を変更する際に、これらの生成されたファイルを考慮に入れてください。

リフレクションメカニズムに戻ると、リフレクション指定子はReSharper C ++とRiderの単なるテキストではないことは言うまでもありません。オートコンプリートとドキュメントのヒントがあります。



ドキュメントのヒントは、リフレクションマクロ自体にも使用できます。

そして、コードアナライザーはリフレクションマクロの使用をチェックし、これに関連するエラーを示します。例えば：

  • UCLASSクラスは、UObjectまたはその子孫クラスからパブリックに継承する必要があります。さらに、そのような親は正確に1つでなければなりません。
  • UCLASS指定子はクラスに使用し、USTRUCTは構造に使用する必要があります。
  • UE4クラスを他のクラス内で宣言することはできません。
  • UPROPERTYとマークされていないフィールドに格納されたオブジェクトは、ガベージコレクターによっていつでも収集できます。

ナビゲーションおよびコード生成アクションのリモートプロシージャコール（RPC）サポート


通常のC ++パーサーの観点からリモートプロシージャの呼び出しを検討すると、異なる名前（たとえば、サフィックス_Validateと_Implementation）を持ついくつかの関数が関数宣言に対応している可能性があるというパーサーの無知にすぐに気づくでしょう。ReSharperのC ++ /ライダーでは、我々は、指定子を超えるRPCを識別するためのパーサを訓練してきたClient、ServerとNetMulticastマクロの設定でUFUNCTION。それによって：

  • ナビゲーションは、機能を実装するために必要なすべてのオプションを提供します。
  • コード生成では、サフィックス_Validateを持つ関数、_Implementationまたは欠落している関数のみを作成できます。
  • コードのリファクタリング[名前の変更と署名の変更]は、必要なすべての機能を更新し、コードを正しい状態に保ちます。

アンリアルエンジン4の命名規則に関する知識


ReSharper C ++とRiderは、Unreal Engineコードの公式の命名規則を認識しています。これらのルールは、ゲッターとセッターの生成や変数のリファクタリング（変数の導入）など、コードを操作するすべてのアクションで開発環境によって使用されます。そして最も重要なのは、一貫性のないUE4命名検査コードチェックとそれに対応するクイックフィックスがあり、名前の変更がリファクタリングされ、ルールに準拠しない名前のすべての使用が名前変更されることです。

エディターのパフォーマンス


ReSharper C ++でアンリアルエンジンの開発サポートをかなり長い間行ってきましたが、もちろん、エディターのパフォーマンスが主な不満であることがわかります。そのアーキテクチャーにより、RiderはReSharperに存在する多くのパフォーマンスの問題から解放されます（ReSharperが動作するだけでなく、ReSharperが動作する32ビットプロセスに関するスタジオの制限が原因である場合もあります）。

さらに、アンリアルエンジンのパフォーマンスを向上させるためにIDEを特別に調整します。まず、ユーザーゲームのコードにインデックスを付け、ユーザーコードに対するエディターのすべての巧妙なアクションを即座に含めます。その後、バックグラウンドでエンジンコード自体のインデックス作成が行われます。インデックス作成を管理するための追加オプションがいくつかあります。



その結果、Rider for Unreal Engineの使用を既に開始しているユーザーは、エディターのパフォーマンスについて非常に好意的に反応します。そして、私たちはさらに良いことができると確信しています。
