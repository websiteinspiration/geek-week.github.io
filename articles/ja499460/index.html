1.シリアルからパラレルへ


最も単純な算術方程式のプログラミングを検討してください

。C2= A + B + A1 + B1; （1）

単純な算術演算を実装するブロックがあるとします。この場合、ブロックの合計で十分です。ブロックの数、それらの構造、およびそれらの間の関係についての明確で正確な考えが米を与えます。1.そして図2で VKP（a）媒体の構成は、方程式（1）を解くために与えられます。

2.問題としての並行性


特定の並列ソリューションを支持して多くの引数が呼び出されても驚かないでください。ただし、通常の逐次プログラミングには存在しない可能性のある問題については黙っています。並列処理の正しい実装の問題の同様の解釈の主な理由。彼らは彼女について控えめに言っている。もし彼らが言うなら。データへの並列アクセスに関する部分で触れます。

プロセスは、連続した分割不可能な多数のステップとして表すことができます。多くのプロセスでは、そのような各ステップで、すべてのプロセスに属するアクションが同時に実行されます。そして、ここでは、次の基本的な例で明らかになる問題に遭遇する可能性があります。

次の連立方程式に対応する2つの並列プロセスがあるとします。

c = a + b; （3）
a = b + c;

変数a、b、cに初期値1、1、0が割り当てられていると仮定します。5つのステップの計算プロトコルは次のようになると予想できます。

a              b               c
1.000       1.000       0.000      
1.000       1.000       2.000       
3.000       1.000       2.000       
3.000       1.000       4.000       
5.000       1.000       4.000        
5.000       1.000       6.000

それを形成するとき、我々は演算子が並行して（同時に）そして1つの個別の測定（ステップ）内で実行されるという事実から進んだ。ループ文の場合、それはループの反復になります。また、計算の過程で、変数には離散メジャーの最初に固定された値があり、それらの変化は最後に発生すると仮定することもできます。これは、操作の完了に時間がかかる実際の状況と非常によく一致しています。多くの場合、特定のブロックに固有の遅延に関連しています。

しかし、ほとんどの場合、このプロトコルのようなものが得られます：

   a              b               c
1.000       1.000       0.000      
3.000       1.000       2.000       
5.000       1.000       4.000       
7.000       1.000       6.000       
9.000       1.000       8.000       
11.000     1.000     10.000

これは、サイクル内で2つの連続したステートメントを実行するプロセスの作業と同等です

。c = a + b; a = b + c; （4）

しかし、ステートメントの実行がまったく逆になり、プロトコルは次のようになる場合があります。

   a              b               c
1.000       1.000       0.000      
1.000       1.000       2.000       
3.000       1.000       4.000       
5.000       1.000       6.000       
7.000       1.000       8.000       
9.000       1.000     10.000

マルチスレッドプログラミングでは、状況はさらに悪化します。プロセスの同期がない場合、オペレーターの起動順序を予測することは難しいだけでなく、オペレーターの作業もどこでも中断されます。これはすべて、オペレーターの共同作業の結果に影響を与えることはできません。

APテクノロジのフレームワーク内では、一般的なプロセス変数を使用した作業が簡単かつ正確に許可されます。ここでは、ほとんどの場合、プロセスを同期してデータを操作するために特別な作業は必要ありません。ただし、条件付きで瞬間的で不可分と見なされるアクションを選び出し、自動プロセスモデルを作成する必要があります。私たちの場合、アクションは加算演算子であり、循環遷移を伴うオートマトンがそれらの起動を担当します。
リスト1は、合計演算を実装するプロセスのコードを示しています。そのモデルは、1つの状態と無条件ループ遷移を備えた有限状態マシン（図3を参照）であり、2つの変数を合計する演算を実行したアクションy1のみが結果を3番目に入れます。

3.そしてコルーチン？...


Kotlin言語で表されるコルーチンがタスクにどのように対処するかを知っておくとよいでしょう。[1]の説明で検討したプログラムをソリューションテンプレートとして考えてみましょう。それは、必要な外観に簡単に縮小できる構造を持っています。これを行うには、その中の論理変数を数値型に置き換えて別の変数を追加します。論理演算の代わりに、合計演算を使用します。対応するコードをリスト2に示します。

リスト2. Kotlin並列合計プログラム 
import kotlinx.coroutines.*

suspend fun main() =
    // Structured concurrency: if any child coroutine fails,
    // everything else will be cancelled
    coroutineScope {
        var a = 1
        var b = 1
        var c = 0;
        // Use default thread pool 
        withContext(Dispatchers.Default) {
            for (i in 0..4) {
               var res = listOf(async { a+b }, async{ b+c }).map { it.await() }
			c = res[0]
			a = res[1]
               println("$a, $b, $c")
            }
        }
    }


このプログラムの結果について質問はありません。上記のプロトコルの最初のものと完全に一致します。

ただし、ソースコードの変換は、見かけほど簡単ではありませんでした。次のコードを使用するのは自然なことです。

listOf(async {  = a+b }, async{  = b+c })

テストで示されているように（これはKotlin Webサイト（kotlinlang.org/#try-kotlinでオンラインで実行できます））、使用すると完全に予測できない結果が生じ、起動ごとに変化します。そして、ソースプログラムをより注意深く分析するだけで、正しいコードが導き出されました。

プログラムの機能の観点からはエラーが含まれているが、言語の観点からは正当であるコードは、プログラムの信頼性を恐れさせます。この意見は、おそらく、Kotlinの専門家が異議を唱える可能性があります。それにもかかわらず、「コルーチンプログラミング」の理解の欠如だけでは説明できない間違いを犯しやすいことは、それにもかかわらず、そのような結論を根強く押し付けています。

4. Qtのイベントマシン


以前に、イベントオートマトンはその古典的な定義ではオートマトンではないことを確立しました。彼は良い人も悪い人もいますが、ある程度までは、イベントマシンは依然としてクラシックマシンの親戚です。遠いですか、近くですか。誤解がないように、直接話し合う必要があります。 [2]でこれについて話し始めましたが、すべて続行しました。ここで、Qtでのイベントマシンの使用の他の例を調べて、これを行います。

もちろん、イベントオートマトンは、イベントに関連付けられた不定および/または可変のサイクル期間を持つ古典的なオートマトンの退化したケースと見なすことができます。そのような解釈の可能性は、以前の記事で1つだけ、さらにはかなり具体的な例を解決したときに示されました（詳細[2]を参照）。次に、このギャップをなくそうとします。

Qtライブラリは、マシン遷移をイベントに関連付けるだけであり、これは重大な制限です。たとえば、同じUML言語では、遷移は開始イベントと呼ばれるイベントだけでなく、保護条件（イベントの受信後に計算される論理式）にも関連付けられます[3]。 MATLABでは、状況はさらに軽減され、次のように聞こえます。「イベントの名前が指定されていない場合、イベントが発生すると遷移が発生します」[4]。しかし、あちこちで、遷移の根本的な原因はイベント/秒です。しかし、イベントがない場合はどうなりますか？

イベントがない場合、...イベントを作成してみることができます。リスト3と図 4は、VKPa環境のオートマトンクラスLFsaApplの子孫をイベントQtクラスの「ラッパー」として使用して、これを行う方法を示しています。ここで、オートマトン空間の離散時間の周期性を持つアクションy2は、Qtオートマトンの遷移の開始を開始する信号を送信します。後者は、s0Exitedメソッドを使用して、合計演算を実装するアクションy1を開始します。イベントマシンは、LFsaApplクラスのローカル変数の初期化をチェックした後に厳密にアクションy3によって作成されることに注意してください。

5.二次関数と...蝶？


入力データを外部並列プロセスとして表すと便利です。この上には、AND-NOT要素のモデルを管理するためのダイアログがありました。さらに、データの変化率は結果に大きな影響を与える可能性があります。これを確認するために、2次関数y =ax²+ bx + cの計算を検討します。これは、並列機能および相互作用ブロックのセットの形で実装します。

二次関数のグラフは放物線の形をしていることが知られています。しかし、数学者が描いた放物線と、たとえばオシロスコープが表示する放物線は、厳密には一致しません。この理由は、数学者は、入力量の変化がすぐに関数の計算につながると信じて、瞬間的なカテゴリーを考えることが多いためです。しかし、実際にはこれはまったく当てはまりません。関数グラフの外観は、計算機の速度、入力データの変化率などに依存します。等はい、速度のプログラム自体は互いに異なる場合があります。これらの要因は関数の形に影響を与え、特定の状況では放物線を推測することが困難な形になります。私たちはこれをさらに確信します。

したがって、乗算、除算、および累乗のブロックを合計ブロックに追加します。そのようなセットがあれば、どんな複雑な数式も「収集」できます。しかし、より複雑な機能を実装する「キューブ」を使用することもできます。図5。二次関数の実装は2つのバージョンで示されています-マルチブロック（ラベル付きの矢印-1と図6も参照）と1つのブロックからのバリアント（図5のラベル2付きの矢印）です。

7.結論


CPSU（a）のフレームワーク内で、計算モデルはプログラムに、実際の「生きている」オブジェクトに特徴的な特性を与えます。したがって、「生きている数学」との比喩的な関連性。実践が示すように、私たちのモデルでは、実際の生活では無視できないことを考慮する必要があります。並列計算では、これは主に計算の時間と有限性です。もちろん、[自動]数学モデルが1つまたは別の「生きている」オブジェクトに適切であることを忘れないでください。

負けられないものと戦うことは不可能です。そろそろ時間だ。これはおとぎ話でのみ可能です。ただし、それを考慮に入れたり、独自の目的で使用したりすることは理にかなっています。現代の並列プログラミングでは、時間を無視すると、シグナルレース、デッドロックプロセス、同期の問題など、制御および検出が困難な多くの問題が発生します。等VKP（a）テクノロジは、リアルタイムモデルを含み、計算プロセスの有限性を考慮しているため、このような問題からほとんど解放されています。これには、ほとんどの類似物が単に無視されているものが含まれています。

結論として。蝶は蝶ですが、たとえば、2次関数と1次関数の方程式系を考えることができます。これを行うには、既に作成されているモデルに線形関数モデルとそれらの一致を制御するプロセスを追加するだけで十分です。したがって、ソリューションはモデリングによって見つかります。ほとんどの場合、分析ほど正確ではありませんが、より簡単かつ迅速に取得されます。そして、多くの場合、これで十分です。そして、分析的な解決策を見つけることは、原則として、しばしば未解決の問題です。
