First thoughts on Deno

A translation of the article was prepared ahead of the launch of the Node.js Developer course .


At Codegram, we love experimenting with new technologies, and Deno came out just a week ago! Read the article if you want to know about this new JavaScript runtime and what Codegram thinks about it.

You may have already heard about Deno, or maybe not, but now the Internet is full of information about it and there is a reason for that! A new JavaScript runtime has appeared with a cute dinosaur as a logo (some people think that the logo has a sad sock, but no ...). We even tweeted about him a week before the first release, but to tell you the truth, we first heard about Deno two years ago after watching this amazing performance by Ryan Dahl .

In this article, we will go over Deno superficially and share our thoughts about it. Let's get started!

Hello Deno!


Deno is a JavaScript runtime like Node. To use it, you must first install the deno CLI. To do this, follow the instructions for your operating system on deno.land (our domain name compliments!). Deno is not a fork of Node, it was created from scratch on Rust. I wrote an article about Rust for JS programmers , you can read it if you want to know more about the language!

After installing the CLI, you can run the classic program ‚ÄúHello, World!‚Äù in the following way:

deno run https://deno.land/std/examples/welcome.ts

Are we running TypeScript out of the box? Yes! TS is a welcome guest at Deno, so you don‚Äôt need anything else to run the code on TS. You can run JS code as well, but I'm a big fan of TS, so I was the first to talk about it.

Keep in mind that Node code is not compatible with Deno. However, right now they are working on ensuring compatibility , so that in the future you will be able to use your favorite NPM packages.

There is one more thing worth mentioning that Deno is a sandbox by default. That is, initially you do not have access to the network or file system. If your program needs access, you need to explicitly allow it (i.e. use --allow-net to grant network privileges).

No more package.jsonornode_modules
Who has not seen this joke on the Internet?



Despite a slight exaggeration, there is still some truth in this joke. As your application grows, the list of dependencies grows, and you end up with a huge dependency tree. The problem is that every application has a copy of the package @ version that you are using.
One of Deno‚Äôs main goals is to mirror the browser. When you want to import something into a browser, you write a tag <script>and use the URL. Deno works just as follows:

import { assertEquals } from "https://deno.land/std/testing/asserts.ts";
assertEquals("hello", "hello");
assertEquals("world", "world");
console.log("Asserted! ");

I agree, it looks a little strange. Since we do not have any package.json, there is no way to specify external dependencies, so we import them when they are needed. At the first start, Deno will download the file and save it in the cache. You can specify the --reload flag if you want to download the package again.

You probably have a lot of questions on this subject (I had them too), so I invite you to visit this FAQ page in the Deno manual to get more information on the topic.

Browser API compatible runtime


Another important goal of Deno was browser compatibility, so it has a set of features available from the global scope, such as fetch or addEventListener . You can also use the global window object (however, for now, I recommend using globalThis). You can learn more about global functions in the documentation .

The runtime also has a global Deno for APIs that are not a web standard. You can use it for low-level operations such as reading a file, opening a TCP socket, etc.

Rich standard library


Deno developers have created a collection of standard ready-to-use modules. The library is also hosted on the deno.land domain , so you can import any module into your application using the following URL:

import { v4 } from "https://deno.land/std/uuid/mod.ts";

console.log(v4.generate());

The standard library includes many useful modules, but there are many third-party ones, and this list is updated every day. There is one more good news: you can import any module hosted on a public URL, which means that you can just as easily import modules from GitHub. Deno has a URL rewriting service that you can use to make your modules available. You can learn more about this mechanic on this page .

Built-in tools


Deno CLI also includes built-in tools for performing some common tasks, such as running tests, for example, or even creating a package! Does this mean that Deno can replace jest, prettier and webpack? Probably not. Deno is a fairly new solution and does not yet have such an extensive ecosystem, but the presence of all this tools, access to which can be obtained by installing one binary, is still impressive.

Conclusion


Jump on this hype wave! I'm really impressed with Deno. After two years of development and a long road to success, the new JavaScript runtime looks great and I‚Äôm eager to start experimenting with it. Node appeared about 11 years ago and became a new milestone in the history of JavaScript. He made the language better, and I think we have reached a new milestone, it's time Deno!
