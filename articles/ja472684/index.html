fsync（）PostgreSQLへのサプライズ

必要に応じて、DBMS開発者は、データが安全に永続的なストレージに格納されることを懸念しています。したがって、PostgreSQLコミュニティが、カーネルがI / Oエラーを処理する方法が、ユーザースペースにエラーが報告されずにデータ損失につながる可能性があることを発見したとき、多くの不満が生じました。 PostgreSQLがバッファリングされたI / Oを実行するという事実によって悪化する問題はLinuxに固有のものではなく、そこでさえ解決するのは容易ではありません。

Craig Ringer は、この問題を3月下旬にpgsql-hackersメーリングリストに最初に報告しました。つまり、PostgreSQLは呼び出しが成功したと見なしますfsync()最後に成功した呼び出し以降に記録されたすべてのデータが永続ストレージに安全に転送されたことを示します。ハードウェアエラーが原因でバッファリングされたI / Oレコードが失敗した場合、ファイルシステムの反応は異なりますが、この動作には通常、対応するページのデータの削除とクリーンとしてのマーク付けが含まれます。したがって、書き込まれたばかりのブロックを読み取ると、別の何かが返されますが、記録されたデータは返されません。

エラー報告はどうですか？ 1年前、Linuxファイルシステム、ストレージ、およびメモリ管理サミット（LSFMM）サミットには、すべて「混乱」と呼ばれるエラー報告セッションが含まれていました。エラーは簡単に失われる可能性があるため、アプリケーションでエラーが表示されることはありません。4.13に含まれる一部のパッチは、開発サイクル中に状況をある程度改善しました（さらに、4.16ではさらに改善するためにいくつかの変更がありました）。ただし、以下に説明するように、エラー通知を失う方法があります。これがPostgreSQLサーバーで発生すると、データベースが自動的に破損する可能性があります。

PostgreSQL開発者は不満を抱いていました。トムレーンはこれを「核の脳損傷「ロバート・ハースはそれを「100％バカ」と呼びました。状態（以降の試行の場合）、およびPostgreSQLサーバーが問題をスキップできないように、対応するファイル記述子を一定のエラー状態にする必要があります。

どこで問題が発生しましたか


しかし、カーネルコミュニティが議論に入る前でさえ、状況は見かけほど単純ではないことが明らかになりました。 Thomas Munro は、Linuxはこの振る舞いにおいてユニークではないと述べました。 OpenBSDとNetBSDは、ユーザー空間での書き込みエラーも報告しない場合があります。そして、結局のところ、PostgreSQLがバッファI / Oを処理する方法は、非常に複雑です。

このメカニズムは、ハースによって詳細に説明されています。 PostgreSQLサーバーは一連のプロセスとして機能し、その多くはデータベースファイルに対してI / Oを実行できます。ジョブを呼び出すfsync()ただし、単一のチェックポインタープロセス（「チェックポインター」プロセス）で処理され、障害から回復するためにディスクストレージを一貫した状態に維持します。チェックポインタは通常、関連するすべてのファイルを開いたままにしないため、呼び出しの前にファイルを開かなければならないことがよくありますfsync()。ここで問題が発生します。カーネル4.13以降のバージョンでも、ファイルが開かれる前に発生したエラーはチェックポインターで確認できません。open()チェックポインタが呼び出される前に何か問題が発生した場合、次の呼び出しfsync()は成功を返します。呼び出し以外でI / Oエラーを発生させる方法はいくつかありますfsync()。たとえば、カーネルはバックグラウンドの書き戻しを実行中にそれらの1つに遭遇する可能性があります。誰かが電話するsync()、I / Oエラーが発生し、受信したエラー状態を「吸収」する場合もあります。

Haasは、この動作をPostgreSQLの期待に応えることができないと説明しました。
あなた（または誰か）が持っているのは、基本的
に、特定のエラーに関連している可能性のあるファイル記述子についての実証されていない仮定ですが、PostgreSQLがそれに一致することはありませんでした。あなたは問題が私たちの推測にあると言い続けることができますが、私たちがこれまでにそれらを行った唯一のプログラムであると仮定するのは私には間違っているようです。

その結果、Joshua Drake は、対話をカーネル開発コミュニティの一部を含むext4の開発リストに移動しました。 Dave Chinnerはすぐにこの動作を「各OSプラットフォームの動作が異なり、期待されたものとほとんど一致しないクロスプラットフォームコードでの災害のレシピ」と説明しました。代わりに、Ted Tso は説明しました影響を受けるページがI / Oエラーの後にクリーンとしてマークされる理由;つまり、I / Oエラーの最も一般的な原因は、ユーザーが間違ったタイミングでUSBドライブをイジェクトした場合です。プロセスがこのディスクに大量のデータをコピーした場合、結果としてダーティページがメモリに蓄積され、システムに他のタスク用の十分なメモリがない可能性があります。したがって、これらのページは保存できず、ユーザーがそのようなイベントの後にシステムを使用可能なままにしたい場合は消去されます。

ChinnerとTsoは共に、PostgreSQLには適切なソリューションがあり、ダイレクトI / O（DIO）に切り替えると述べました。DIOを使用すると、ライトバックとI / Oをより一般的に制御できます。これには、失敗したI / O操作に関する情報へのアクセスが含まれます。Andres Freundは、他のいくつかのPostgreSQL開発者と同様に、DIOが最良の長期的なソリューションであることを認めました。しかし彼はまた、開発者がこのタスクの実装に深く突入することを期待すべきではないと述べました。その間、彼はまたこの振る舞いを起こしやすい他のプログラム（彼はdpkgについて言及した）があると言った。

短期的な解決に向けて


ディスカッション中に、書き込みの失敗は影響を受けるページがダーティな状態でメモリに格納されるという事実につながるはずであるという考えにかなりの注意が払われました。しかし、PostgreSQL開発者はすぐにこのアイデアから離れ、それを要求しませんでした。彼らが本当に必要としているのは、何かがうまくいかなかったかどうかを突き止める信頼できる方法です。これを念頭に置いて、従来のPostgreSQLエラー処理メカニズムでこれを処理できます。しかし、彼の不在ではほとんど何もできません。

ディスカッションのある時点で、Tso はGoogleには独自のI / Oエラー処理メカニズムがあります。カーネルは、netlinkソケットを介してI / Oエラーを報告するように指示されました。専用プロセスはこれらの通知を受け取り、それに応じて応答します。しかし、このメカニズムは入力時にこれを実行したことはありません。 Freind は、このメカニズムはPostgreSQLにとって「理想的」であるため、近い将来、パブリックドメインに登場する可能性があると述べています。

その間、Jeff Leightonは別のアイデアについて考えていました。I/ Oエラーが発生したときにファイルシステムのスーパーブロックにフラグを設定することです。syncfs()その後、呼び出しはこのフラグをクリアし、設定されている場合はエラーを返します。 PostgreSQLチェックポインタは定期的に呼び出しsyncfs()て、データベースを含むファイルシステムのエラーをポーリングできます。フロイントこれは問題の実行可能な解決策になる可能性があることに同意しました。

もちろん、そのようなメカニズムは新しいカーネルでのみ表示されます。それまでの間、PostgreSQLのインストールは通常、エンタープライズディストリビューションでサポートされている古いカーネルで実行されます。これらのカーネルでは、4.13に含まれていた改善でさえ欠けているようです。これらのシステムでは、PostgreSQLがI / Oエラーを検出するのに役立つことはほとんどありません。システムログをスキャンしてそこでI / Oエラーメッセージを探すデーモンを起動するだけで十分な場合があります。最も洗練された解決策ではなく、ブロックドライバーやファイルシステムが異なると、原則としてさまざまな方法でエラーが報告されるため、複雑になりますが、これが最善の方法かもしれません。

次のステップは、おそらく4月23日に開催されるLSFMM 2018イベントでの議論になるでしょう。運が良ければ、利害関係者に役立つ何らかの解決策があるでしょう。ただし、変更されないことの1つは、エラー処理を正しく行うことが難しいという単純な事実です。
