RSA


RSAでは、関数の反転の複雑さは、因数分解（数値の因数分解）が乗算よりもはるかに長くかかるという事実に基づいています。より正確には、多項式時間で大きな数を因数分解する方法は、従来のコンピューターでは見つからなかったと主張できますが、このようなアルゴリズムは存在できないことが証明されています。

RSAでは、公開鍵を使用する他の暗号化システムと同様に、公開鍵と秘密鍵の2つの鍵があります。 RSAアルゴリズムは、ビット文字列で表される入力メッセージを受け取り、それにランダムなものと区別できない結果を取得するために、数学演算を適用します（大きな数を法として累乗します）。復号化の場合、結果が取得され、同様の操作が実行されて元のメッセージが受信されます。非対称暗号化では、暗号化は公開鍵で実行され、復号化は秘密鍵で実行されます。

これはどのようにして可能ですか？使用される値は有限の巡回グループ（モジュラー演算で乗算を行う整数のセット）に属しているためです。コンピュータは、任意に大きな数ではうまく機能しませんが、循環グループのプロパティは「ラップアラウンド」です。許可された最大値より大きい数は、許可されたセットの値に「ラップ」されます。これにより、特定のビット数「以下」の長さのキーを使用できます。楕円曲線に基づく暗号化では、循環（乗算）グループも使用されますが、後で説明するように、構成が少し異なります。

非常に原始的なレベルでは、RSAが行うことはすべて、2つの大きな素数を取り、それらを乗算していわゆるモジュールを取得することです。操作が実行される他のすべての数値は、ゼロとモジュール値の間でなければなりません。モジュール自体が公開鍵の一部になります-モジュールの長さが鍵の長さを決定します。公開鍵の2番目の部分は、モジュールからゼロとオイラー関数の値（最新のRSA実装ではオイラーの代わりにカーマイケル関数を使用）から選択された数値ですが、いくつかの制限があります。最後に、モジュラー方程式を解くことで秘密鍵を計算できます。メッセージを暗号化するには、メッセージを表す数値を取り、それを単に公開鍵の値に等しい累乗し、復号化します—元のメッセージを受信し、累乗します。秘密鍵の値と同じです。グループの循環的な性質により、同じ意味になります。

プロトコルの最新バージョンでは、キー配布メカニズムは（EC）DH（E）に制限されています。x25519は、共通キーを取得するための最も一般的な機能です。これは、ほとんどのブラウザーおよびサーバーのTLSライブラリでサポートされています。暗号スイートは、TLS_AES_128_GCM_SHA256、TLS_AES_256_GCM_SHA384、TLS_CHACHA20_POLY1305_SHA256の3つのエントリのみで構成されています。以前のバージョンのプロトコルであるTLS 1.2での暗号スイートの命名方法に精通している方は、TLS 1.3への移行中に、鍵交換メカニズムの表示が暗号スイートから「分離」されたことは明らかです。また、静的なRSAおよびDH鍵交換メソッドは仕様から外されました。 PSKと前のセッションのチケットを利用したTLS 1.3でのセッションの回復でさえ、最後のE-暦が特に重要であるECDHEプロトコルに従って発生します。また、TLS 1.3では、Diffie-Hellmanメカニズムに独自の値を設定することはできません。定義済みのセットのみがプロトコル仕様に残されており、このセットのセキュリティに関してコンセンサスがあります。

特に興味深いのは、現代の非対称暗号化メカニズムの動作が異なるという事実です。 ECCでは（特にECDSA証明書を使用する場合）、許容できるレベルのセキュリティを確保するために、RSAと比較して比較的短い長さのキーを使用します。これにより、スマートカードなどの必要な操作を行うのに十分な電力が不足していると思われるデバイスでも、強力な非対称暗号化とキー交換スキームを使用できます。

まず、TLS 1.3の説明に適用される「ハイブリッド暗号システム」という用語の意味を明確にする必要があります。ハイブリッド暗号化システムは、非対称（公開キーを使用）暗号化スキームを使用して共有シークレットを確立します。共有シークレットは、対称ブロックまたはストリーム暗号化アルゴリズムのキーとして使用されます。

次に、公開キー基盤（PKI）と証明書（CA）基盤があります。興味深いのは、2004年にマーティンヘルマンが「名も無き英雄」の1つであるローレンケンフェルダーについて言及したことです。MITの学士号の卒業証書を擁護するための論文は、ツリー構造を作成する可能性でした。今日、PKIとして知られています。しかし、証明書に戻りましょう。

サーバーが実際に秘密鍵を所有しているという事実は、その署名によって検証されます。署名は公開鍵を使用して検証できます。また、サーバー上の証明書ファイルは、特定の公開鍵が特定のサーバーに属していることを証明します。これは、サイバー詐欺ではなく、詐欺師ではなく、目的の対話者との安全な接続を確立していることを意味します。

TLS 1.3では、以前のバージョンのプロトコルと比較して、ネゴシエーション方式が大幅に改善されています。これで、サーバーは、そのような署名の作成時にハンドシェイクで受信したすべての情報に署名します。クライアントのhelloとそのサーバーのhello、および使用するために選択された一連の暗号。RFC 8446の対応する相互作用の説明セクションを見てみましょう。

TLS 1.3では、クライアントは（必要なパラメーターとともに）鍵の一部と、最初のメッセージ（Client Hello）で使用可能な署名のリストを送信します。この段階で必要なキーは、オンザフライでクライアントによって作成されます。ユーザーはこれに気づくことすらありません。次に、クライアントとサーバーがこの情報を交換して、共通の秘密を作成します。これは、サーバーがクライアントにメッセージ（Server Hello）で応答した後に受信したプレマスターペアから作成された（より正確には、派生した）セッションキーです。
「Server Hello」側では、クライアントへの証明書の転送（Certificate *）に対応するレコードと、CertificateVerify *チェックによって、パーティが対応する公開キーレコードに関連する秘密キーを実際に持っていることを確認し、セッション（対称）キーペアを作成します。成功した場合。つまり、データ要求側（クライアント）は、共有シークレットの使用に切り替えることにより、応答側（サーバー）の信頼性を正常に検証しました。

この転送では、署名の作成と署名の検証という2つの主要な暗号操作が保護されています。ほとんどの場合、クライアントはサーバーの信頼性を確認するため、これらの操作は通常、接続の異なる端で実行されます。署名は基本的に、公開鍵に対応する秘密鍵の所有権の確認です。つまり、署名者からデータを受信し、転送中にメッセージが変更されていないことを確認します。

RSAアルゴリズムを使用する場合は、後で数値を示しますが、署名を作成する操作が最も負荷がかかります。 2048ビットまたは3072ビットの鍵で署名するため、このような操作はサーバーに大きな負荷をかけます-（署名）検証中にクライアントよりもはるかに強力です。

ECDSAでは、比較的短いキーで操作します（ベンチマークの例では、NIST P-256曲線（またはsecp256v1）を使用してECDSAを調べますが、これらのキーはより複雑な操作に関与します。その結果、ECCを使用すると、「反転」RSAとして表すことができます。パフォーマンスの観点：クライアントは署名検証操作よりもはるかに重いですが、サーバーは署名作成操作を簡単に処理します。これは、ベンチマークセクションで提供する測定によって確認されます。

この効果はインターネットのスケーリングに役立ちます-最近のクライアントはサーバーとほぼ同じくらい強力であるため（プロセッサコアのクロック速度のみを考慮した場合）、問題なく効率的に計算困難な操作を実行できます。次に、サーバーは解放された計算能力を使用して、より多くの署名を作成し、その結果、より多くのセッションを確立できます。

Let's Encryptでの証明書の署名


Let's Encryptが公開署名されており、トラストチェーンの証明書に証明書として含まれているECDSA鍵ペアを使用してTLSセッションを確立できるサーバーを独自に作成できる、シンプルで理解可能な指示を読者に提供します。
鍵の作成、Let's Encryptの証明書署名要求（CSR）の作成、および必要なチェーンとECDSA証明書自体を返すcertbotユーティリティを使用して署名に転送することから、完全なパスを示すことにしました。

まず、鍵ペアを作成する必要があります。これにはOpenSSLライブラリを使用します。 OpenSSLユーザーガイドによると、楕円曲線に基づくアルゴリズムのキーの作成は、楕円曲線に基づくアルゴリズムのファミリー専用の特別なコマンドを使用して行われるとのことです。

openssl ecparam -genkey -name -secp256v1 -out privatekey.pem

チームが正しく機能していることを確認するために、秘密鍵を含む新しく作成されたファイルへのパスを示すecコマンドを実行します。

openssl ec -in privatekey.pem -noout -text

出力には、キーの生成に使用された曲線が表示されます。

次のステップは、CSRを作成する際に非常に重要です。証明書に署名するために必要なアンケートに毎回記入しないようにするには、構成ファイルが必要です。幸い、私たちのほとんどすべての作業はMozillaによって行われ、「SSL構成ジェネレーター」が作成されました。その中で、TLS接続を確立できるサーバーモードのさまざまなオプションから選択できます。クリーンなOpenSSL構成は、何らかの理由でMozillaジェネレーターから欠落しているため、次のようになります。

[ req ]
prompt = no
encrypt_key = no
default_md = sha256
distinguished_name = dname
req_extensions = reqext

[ dname ]
CN = example.com
emailAddress = admin@example.com

[ reqext ]
subjectAltName = DNS:example.com, DNS:*.example.com

注：構成ファイルがなくてもかまいません。ファイルがない場合は、コマンドラインのすべてのフィールドに入力するよう求められます。次のコマンドで指定するパスである構成ファイルを使用すると、プロセスが簡略化されます。

次のステップは、証明書署名要求（CSR）自体を作成することです。このために、便利なOpenSSLチームがあります。

openssl req -new -config -pathtoconfigfile.cnf  -key privatekey.pem -out csr.pem

また、新しく作成したCSRの正当性を確認することもできます。

openssl req -in csr.pem -noout -text -verify

最後に、最終段階に到達します。ACMEクライアントであるcertbotを使用して、Let's Encrypt組織証明書への署名リクエストを転送します。

Certbotは証明書の取得を支援するだけでなく、他にも多くの優れたオプションを備えています。公開鍵を使用する暗号化に不慣れで、現時点で存在する公開鍵インフラストラクチャを実際に理解していない--dry-run場合は、ドメインの実際の証明書を取得する前に、このオプションを使用することをお勧めします。

certbot certonly --dry-run --dns-somednsprovider --domain “example.com” --domain “*.example.com” --csr csr.pem

この場合、certbotクライアントは、コマンドラインで要求された必須ドメインのリストが証明書署名要求（CSR）のリストと一致することを確認します。オプションの中では、--dns-somednsproviderLet's Encryptがインターネットトラフィックの特定の部分を所有していることを確認する多くの方法があるため、少し嘘をつきました。ただし、DigitalOcean、AWS、Azure、Hetznerなどのクラウドプロバイダーを使用している場合は、プロバイダーが統合ツールを処理しているため、認証に必要な情報を提供する簡単な方法がすでにあるはずです。

その後、certbotを使用してLet's Encryptを使用してCSRに渡されたパラメーターが正しいことが確実な場合は--dry-run、コマンドからパラメーターを削除して続行します。

成功した場合、クライアントはいくつかの証明書を返します。署名付き証明書自体、中間証明書とルート証明書、および証明書のチェーンの形での後者の組み合わせで、すべて.pem形式です。

OpenSSLには、証明書の内部を調べるために使用するコマンドがあります。

openssl x509 -in chainfilepath.pem -noout -text

Let's EncryptがSHA256ハッシュアルゴリズムを使用して証明書に署名したことは明らかです。これに加えて、Let's Encryptはルートと中間の署名をECDSAに追加することのみを計画しているため、現時点では中間のRSA署名を含むコンテンツのままです。しかし、とにかくECDSA公開鍵を使用するので、これは恐ろしいことではありません。

このセクションの最後に、さまざまなアルゴリズムのキー長の比較に関する情報を追加します。情報セキュリティでは、通常、セキュリティレベルは2 ^ xであると言います。xはビット長です（RSAは例外です。RSAは指数よりも少し遅くなるためです）。さまざまなアルゴリズムのキーが互いに比較される方法を概算するために、OpenSSL wikiページを参照します：

既知の問題とNSA



著者：xkcd

おそらく、楕円曲線に基づく暗号化を使用する際の中心的な問題は、特定のセキュリティレベルのキーを作成するために必要な、非常にきちんと実装された乱数ジェネレータの必要性でした。

巨大なスキャンダルがDual_EC_DRBGアルゴリズムに関連付けられていました -解決に何年もかかりました。 ECCに関する特許基盤については不確実性があります。多くの特許は、Blackberryが買収したCerticomに属していることが知られています。いくつかの既知のBlackberry ECC認定ケースもあります。最後に、NIST規格に対する不信感があり、NSAまたは他の米国の諜報機関の影響を受ける可能性があります。

暗号規格の実装におけるエラーは、完全に直交するトピックです。 2010年に、PlayStation 3は、ECDSAアルゴリズムの誤った実装が原因でソニーの秘密鍵の漏洩に苦しみました-ソニーはRNGに対応できず、同じ乱数を提供しました。これにより、秘密の入力で関数を解くことが可能になりました。 OpenSSLは翌年苦しみましたが、タイムアタックを使用して秘密鍵を取得することを可能にする脆弱性をすぐに修正しました。詳細は、元の研究出版物に記載されています。

2013年のRSAカンファレンスでは、研究者グループが「ランダムに失敗しました！ ” SecureRandom Javaクラスの脆弱性専用です。半年後、ケースが展開で暗号的に安全でない疑似乱数ジェネレータを使用して作成されたビットコインウォレット。

2013年8月に、いくつかの深刻な脆弱性が短期間で発見されたという事実のため、IETFはRFC 6979をリリースしました。これは、キーを作成する際のkの決定論的生成について説明しています。この方法で問題が解決されたと書くこともできますが、プロトコル仕様からの不必要な逸脱が原因で、いつでも研究者が実装に新しいエラーを見つける可能性があります。

そして、NSAについて一言。 Dual_EC_DRBGの履歴を認識していない場合は、少し時間をかけて関連資料を読んでください。詳細を理解したことを後悔することはありません。エドワードスノーデンはこのストーリーの一部になりました。存在していたすべての疑いが確認されたのは、2013年に彼のリークがあったためです。これにより、ECDSAで機能する楕円曲線とアルゴリズムの多くを作成および記述したのはこの組織であったため、多くの著名な暗号技術者がNISTへの信頼を失いました。

Daniel Burnshiteによって作成された曲線25519とキー配布用のDiffie-Hellman関数は、上記の質問の多くに対する答えです。そして、すでに書いたように、EdDSAへの着実な動きがあります。NIST曲線の場合、それらの脆弱性の確認はまだ発見されていません。乱数の経験に関しては、それは非常に苦痛であり、急速な同化に貢献しました。

この部分を締めくくるために、ジョンフォンノイマンを引用します。「乱数を取得するための算術手法が苦手な人は誰もが間違いなく罪を犯していません。」
