Angular（zone.jsなし）：最高のパフォーマンス

Angular開発者はzone.jsに多くを負っています。たとえば、彼女はAngularでの作業をほぼ魔法のように簡単にするのに役立ちます。実際、ほとんどの場合、プロパティを変更する必要があるだけで、何も考えずに変更すると、Angularは対応するコンポーネントを再レンダリングします。その結果、ユーザーが見るものには常に最新の情報が含まれています。これは素晴らしいです。

ここでは、新しいIvyコンパイラー（Angular 9で登場）を使用して、zone.jsの使用を拒否できるようにする方法のいくつかの側面を調べたいと思います。



このライブラリを破棄することで、高負荷で実行されているAngularアプリケーションのパフォーマンスを大幅に向上させることができました。同時に、TypeScriptデコレーターを使用して必要なメカニズムを実装できたため、システムリソースの追加コストが非常に小さくなりました。

この記事で紹介したAngularアプリケーションを最適化するアプローチは、Angular IvyとAOTがデフォルトで有効になっているためにのみ可能であることに注意してください。この記事は教育目的で書かれたものであり、Angularプロジェクトの開発へのアプローチを促進することを目的としていません。

なぜzone.jsなしでAngularを使用する必要があるのでしょうか？


先に進む前に、1つの重要な質問をしてみましょう。「このライブラリを使用すると、簡単にテンプレートを再レンダリングできるので、zone.jsを削除する価値はありますか？」もちろん、このライブラリは非常に便利です。しかし、いつものように、あなたはすべてを支払う必要があります。

アプリケーションに特定のパフォーマンス要件がある場合、zone.jsを無効にすると、それらの要件を満たすことができます。パフォーマンスが重要なアプリケーションの例は、インターフェースが頻繁に更新されるプロジェクトです。私の場合、そのようなプロジェクトはリアルタイムの取引アプリケーションであることがわかりました。クライアント部分は常にWebSocketプロトコルを介してメッセージを受信します。これらのメッセージのデータは、できるだけ早く表示する必要があります。

Angular Ivy：detectChangesとɵmarkDirtyを使用して自分で変更を検出する


TypeScriptデコレーターの作成を開始する前に、コンポーネントの変更を検出し、それをダーティにして、zone.jsとDIをバイパスするプロセスをIvyで呼び出す方法を学ぶ必要があります。

からエクスポートされた2つの追加機能が利用可能になりました@angular/core。これɵdetectChangesとɵmarkDirtyです。これらの2つの関数は、依然として内部使用を目的としており、不安定ɵです。名前の最初はシンボルです。

これらの機能の使い方を見てみましょう。

ɵɵmarkDirty関数


この機能を使用すると、コンポーネントをマークして、「ダーティ」、つまり再レンダリングが必要なコンポーネントにすることができます。彼女は、コンポーネントが呼び出される前に「ダーティ」とマークされていなかった場合、変更検出プロセスを開始する予定です。

import { ɵmarkDirty as markDirty } from '@angular/core';
@Component({...})
class MyComponent {
  setTitle(title: string) {
    this.title = title;
    markDirty(this);
  }
}

ChangeɵdetectChanges関数


Angular内部ドキュメントには、パフォーマンス上の理由から、この関数を使用しɵdetectChangesないでくださいと記載されています。代わりに、関数を使用することをお勧めしますɵmarkDirty。この関数ɵdetectChangesは、コンポーネントとそのサブコンポーネントの変更を検出するプロセスを同期的に呼び出します。

import { ɵdetectChanges as detectChanges } from '@angular/core';
@Component({...})
class MyComponent {
  setTitle(title: string) {
    this.title = title;
    detectChanges(this);
  }
}

TypeScriptデコレーターを使用して変更を自動的に検出する


Angularが提供する機能は、DIを実行できるようにすることで開発の使いやすさを向上させますが、変更検出プロセスを開始するために自分でこれらの関数をインポートして呼び出す必要があるという事実にプログラマーは不満を感じることがあります。

変更検出の自動開始を簡素化するために、この問題を個別に解決するTypeScriptデコレーターを作成できます。もちろん、以下で説明するいくつかの制限がありますが、私の場合、このアプローチがまさに私が必要とするものであることが判明しました。

the @ observedデコレーターの紹介


変更を検出するために、できるだけ少ない労力で、3つの方法で適用できるデコレータを作成します。つまり、次のエンティティに適用できます。

  • 同期メソッドに。
  • 観測可能なオブジェクト。
  • 通常のオブジェクトに。

いくつかの小さな例を考えてみましょう。次のコードでは@observed、オブジェクトstateとメソッドにデコレータを適用していますchangeTitle。

export class Component {
    title = '';

    @observed() state = {
        name: ''
    };

    @observed()
    changeTitle(title: string) {
        this.title = title;
    }

    changeName(name: string) {
        this.state.name = name;
    }
}

  • オブジェクトへの変更を確認するには、オブジェクトへの変更をstateインターセプトし、変更検出プロシージャを呼び出すプロキシオブジェクトを使用します。
  • changeTitle最初にこのメソッドを呼び出し、次に変更検出プロセスを開始する関数を適用して、メソッドを再定義します。

そしてここに例がありBehaviorSubjectます：

export class AppComponent {
    @observed() show$ = new BehaviorSubject(true);

    toggle() {
        this.show$.next(!this.show$.value);
    }
}

Observableオブジェクトの場合、デコレータの使用は少し複雑に見えます。つまり、監視対象のオブジェクトをサブスクライブし、サブスクリプションでコンポーネントを「ダーティ」としてマークする必要がありますが、サブスクリプションをクリアする必要もあります。これを行うためにngOnInit、ngOnDestroyサブスクリプションとその後のクリーニングの両方を再割り当てします。

  • メソッドを次のように再定義します。元の関数を呼び出してmarkDirty(this)から、変更を検出するプロセスを開始するためにそれを呼び出します。これは次のようになります。

次に、処理しているプロパティの種類を確認する必要があります。これは、Observableオブジェクトまたは通常のオブジェクトにすることができます。ここでは、別の内部Angular APIを使用します。私は、それは通常のアプリケーションでの使用を意図していません（申し訳ありません！）。

const cmp = getCmp(target.constructor);
const onInit = cmp.onInit || noop;

監視が編成されるプロパティをそのようにマークするために、ReflectMetadataその値を使用してに設定しますtrue。その結果、コンポーネントが初期化されるときにプロパティを監視する必要があることがわかります。

Reflect.set(target, propertyKey, true);

次にonInit、コンポーネントをインスタンス化するときに、フックをオーバーライドしてプロパティを確認します。

cmp.onInit = function() {
  checkComponentProperties(this);
  onInit.call(this);
};

checkComponentPropertiesコンポーネントのプロパティをバイパス する関数を定義し、先に設定した値に従ってフィルタリングしますReflect.set。

const checkComponentProperties = (ctx) => {
  const props = Object.getOwnPropertyNames(ctx);

  props.map((prop) => {
    return Reflect.get(target, prop);
  }).filter(Boolean).forEach(() => {
    checkProperty.call(ctx, propertyKey);
  });
};

関数checkPropertyは、個々のプロパティの装飾を担当します。まず、プロパティがObservableか通常のオブジェクトかを確認します。これがObservableオブジェクトである場合は、それをサブスクライブし、その内部ニーズのためにコンポーネントに格納されているサブスクリプションのリストにサブスクリプションを追加します。

const checkProperty = function(name: string) {
  const ctx = this;

  if (ctx[name] instanceof Observable) {
    const subscriptions = getSubscriptions(ctx);
    subscriptions.add(ctx[name].subscribe(() => {
      markDirty(ctx);
    }));
  } else {
    //  
  }
};

プロパティが通常のオブジェクトである場合、それをProxyオブジェクトに変換しmarkDirty、その関数で呼び出しますhandler。

const handler = {
  set(obj, prop, value) {
    obj[prop] = value;
    ɵmarkDirty(ctx);
    return true;
  }
};

ctx[name] = new Proxy(ctx, handler);

最後に、コンポーネントを破棄した後、サブスクリプションをクリアする必要があります。

cmp.onDestroy = function() {
  const ctx = this;
  if (ctx[subscriptionsSymbol]) {
    ctx[subscriptionsSymbol].unsubscribe();
  }
  onDestroy.call(ctx);
};

このデコレータの可能性は包括的とは言えません。大規模なアプリケーションに表示される可能性のある、可能なアプリケーションのすべてのオプションについては説明していません。たとえば、これらはObservableオブジェクトを返すテンプレート関数の呼び出しです。しかし、私はそれに取り組んでいます。

それにもかかわらず、上記のデコレータは私の小さなプロジェクトには十分です。完全なコードは資料の最後にあります。

上記で実施したパフォーマンス分析は、特に科学的とは言えません。さまざまなフレームワークのパフォーマンスをより真剣に調査するには、このベンチマークを使用することをお勧めします。調査のために、Angularはこのフレームワークの通常のバージョンとzone.jsなしのバージョンを選択する必要があります。

このベンチマークのアイデアに触発されて、重い計算を実行するプロジェクトを作成しました。Angular Benchpressでパフォーマンスをテストしました。

テストしたコンポーネントのコードは次のとおりです。

@Component({...})
export class AppComponent {
  public data = [];
  @observed()
  run(length: number) {
    this.clear();
    this.buildData(length);
  }
  @observed()
  append(length: number) {
    this.buildData(length);
  }
  @observed()
  removeAll() {
    this.clear();
  }
  @observed()
  remove(item) {
    for (let i = 0, l = this.data.length; i < l; i++) {
      if (this.data[i].id === item.id) {
        this.data.splice(i, 1);
        break;
      }
    }
  }

  trackById(item) {
    return item.id;
  }

  private clear() {
    this.data = [];
  }

  private buildData(length: number) {
    const start = this.data.length;
    const end = start + length;

    for (let n = start; n <= end; n++) {
      this.data.push({
        id: n,
        label: Math.random()
      });
    }
  }
}

import { Observable, Subscription } from 'rxjs';
import { Type, ɵComponentType as ComponentType, ɵmarkDirty as markDirty } from '@angular/core';

interface ComponentDefinition {
  onInit(): void;
  onDestroy(): void;
}

const noop = () => {
};

const getCmp = <T>(type: Function) => (type as any).ɵcmp as ComponentDefinition;
const subscriptionsSymbol = Symbol('__ng__subscriptions');

export function observed() {
  return function(
    target: object,
    propertyKey: string,
    descriptor?: PropertyDescriptor
  ) {
    if (descriptor) {
      const original = descriptor.value;
      descriptor.value = function(...args: any[]) {
        original.apply(this, args);
        markDirty(this);
      };
    } else {
      const cmp = getCmp(target.constructor);

      if (!cmp) {
        throw new Error(`Property ɵcmp is undefined`);
      }

      const onInit = cmp.onInit || noop;
      const onDestroy = cmp.onDestroy || noop;

      const getSubscriptions = (ctx) => {
        if (ctx[subscriptionsSymbol]) {
          return ctx[subscriptionsSymbol];
        }

        ctx[subscriptionsSymbol] = new Subscription();
        return ctx[subscriptionsSymbol];
      };

      const checkProperty = function(name: string) {
        const ctx = this;

        if (ctx[name] instanceof Observable) {
          const subscriptions = getSubscriptions(ctx);
          subscriptions.add(ctx[name].subscribe(() => markDirty(ctx)));
        } else {
          const handler = {
            set(obj: object, prop: string, value: unknown) {
              obj[prop] = value;
              markDirty(ctx);
              return true;
            }
          };

          ctx[name] = new Proxy(ctx, handler);
        }
      };

      const checkComponentProperties = (ctx) => {
        const props = Object.getOwnPropertyNames(ctx);

        props.map((prop) => {
          return Reflect.get(target, prop);
        }).filter(Boolean).forEach(() => {
          checkProperty.call(ctx, propertyKey);
        });
      };

      cmp.onInit = function() {
        const ctx = this;

        onInit.call(ctx);
        checkComponentProperties(ctx);
      };

      cmp.onDestroy = function() {
        const ctx = this;

        onDestroy.call(ctx);

        if (ctx[subscriptionsSymbol]) {
          ctx[subscriptionsSymbol].unsubscribe();
        }
      };

      Reflect.set(target, propertyKey, true);
    }
  };
}

概要


Angularプロジェクトのパフォーマンスの最適化についての私の話を気に入っていただければ幸いですが、急いでプロジェクトからzone.jsを削除しないようにしてください。ここで説明する戦略は、Angularアプリケーションのパフォーマンスを向上させるために使用できる最後の手段でなければなりません。

最初に、OnPush変更検出戦略の使用、trackByコンポーネントの適用、無効化、zone.jsの外部でのコードの実行、zone.jsイベントのブラックリスト化などのアプローチを試す必要があります（この最適化方法のリストは継続できます）。ここに示したアプローチは非常に費用がかかります。パフォーマンスにこのような高い代償を払う気があるかどうかはわかりません。

実際、zone.jsを使用しない開発は、最も魅力的なものではないかもしれません。おそらく、これは彼の完全な制御下にあるプロジェクトに関与している人のためだけではありません。つまり、依存関係の所有者であり、すべてを適切な形式にする能力と時間があります。

すべてを試してみて、zone.jsがプロジェクトのボトルネックであると思われる場合は、変更を自分で検出してAngularを高速化する必要があるかもしれません。
