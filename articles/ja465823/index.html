Tarantool kubernetesオペレーター



Kubernetesは、ステートレスアプリケーションを実行するための事実上の標準になりつつあります。これは主に、新機能を提供するための製品化までの時間を大幅に短縮できるためです。ステートフルアプリケーション（データベース、ステートフルマイクロサービス）の立ち上げは依然として困難な作業ですが、競争に耐え、高い配信率を維持する必要があるため、企業はこの分野で実験を行い、そのようなソリューションに対する需要を生み出しています。

ステートフルクラスタを起動するためのソリューションTarantool Cartridge：Tarantool Kubernetes Operatorを紹介します。

Tarantoolは、オープンソースのデータベースとアプリケーションサーバーを1つのパッケージにまとめたものです。データベースとしては、鉄の利用効率の高さ、柔軟なデータスキーム、インメモリとディスクストレージの両方のサポート、Lua言語の使用による拡張の可能性など、多くのユニークな特徴があります。アプリケーションサーバーとして、最小限の応答時間と最大のスループットを実現しながら、アプリケーションコードをデータにできるだけ近づけることができます。さらに、Tarantoolには広範なエコシステムがあり、アプリケーションの問題を解決するための既製のモジュールを提供しています。シャーディング、キュー、開発を容易にするモジュール（カートリッジ、luatest）、運用ソリューション（メトリック）、ansible）-そして、これらはほんの一例です。

Tarantoolの1つのインスタンスの機能は無制限ではありません。テラバイトのデータを保存し、数百万のリクエストを処理するには、数十、数百のインスタンスを生成する必要があります。これは、固有の問題をすべて備えた分散システムです。それらを解決するために、分散アプリケーションの作成におけるあらゆる種類の困難を隠し、開発者がアプリケーションのビジネス価値に集中できるようにすることを主なタスクとするフレームワークであるTarantool Cartridgeがあります。カートリッジは、自動クラスターオーケストレーション、自動データ分散、操作用のWebUI、開発者ツール用の強力なコンポーネントセットを提供します。

Tarantoolはテクノロジーだけでなく、ターンキーエンタープライズシステム、ボックスソリューション、オープンソースコンポーネントのサポートの開発に従事するエンジニアのチームでもあります。

グローバルに、私たちのタスクは2つの領域に分けることができます：新しいシステムの開発と既存のソリューションの拡張です。たとえば、有名なベンダーの大きな拠点があります。読み取り用にスケーリングするために、最終的に整合性のあるキャッシュを背後にあるTarantoolに配置しました。またはその逆：レコードをスケーリングするために、Tarantoolをホット/コールド構成に置き、データが「クールダウン」すると、データがコールドストレージにダンプされ、分析キューと並行してダンプされます。または、既存のシステムをバックアップするために、このシステムのライトバージョン（機能予備）が書き込まれ、メインシステムからのデータ複製でメインの「ホット」を予約します。T + 2019のレポートから詳細を学ぶことができます。

これらすべてのシステムには、共通点が1つあります。操作が非常に難しいことです。冗長性を備えた100以上のインスタンスのクラスターを3つのデータセンターにすばやく展開し、ダウンタイムやメンテナンスの負担なしにデータを保存するアプリケーションを更新し、災害や人為的なエラーが発生した場合にバックアップを復元し、コンポーネントの目立たないフェイルオーバーを確認し、構成管理を整理します...一般的に、多くの面白い。

そして、これがまだ開発されているのと同じくらい簡単に操作されていたら素晴らしいと思います。Kubernetesを使用すると、目的の結果を得ることができますが、専門のオペレーターを使用すると、作業がさらに簡単になります。

Tarantool OperatorはKubernetes APIをそのリソース定義で拡張し、作成します。

$ kubectl create -f https://raw.githubusercontent.com/tarantool/tarantool-operator/0.0.1/deploy/crds/tarantool_v1alpha1_role_crd.yaml

オペレーターは何をしますか


Tarantool Cartridgeクラスターの起動と操作は、特定の瞬間に特定のアクションを特定の順序で実行するストーリーです。

クラスター自体は、主に管理API（GraphQL over HTTP）を介して管理されます。もちろん、レベルを下げてコマンドをコンソールに直接入力することもできますが、これはほとんど起こりません。たとえば、これはクラスタの起動方法です。

  1. たとえば、systemdの下で必要なTarantoolインスタンスの数を増やします。
  2. インスタンスをメンバーシップにマージします。

    mutation {
    probe_instance: probe_server(uri: "storage:3301")
}

  3. インスタンスにロールを割り当て、インスタンスとレプリカセットの識別子を指定します。これにはGraphQL APIも使用されます。

    mutation {
     join_server(
        uri:"storage:3301",
        instance_uuid: "cccccccc-cccc-4000-b000-000000000001",
        replicaset_uuid: "cccccccc-0000-4000-b000-000000000000",
        roles: ["storage"],
        timeout: 5
    )
}

  4. シャーディングを担当するコンポーネントのブートストラップを実行します。また、APIを介して：

    mutation {
    bootstrap_vshard

    cluster {
        failover(enabled:true)
    }
}


簡単ですよね？

クラスタの拡張に関しては、すべてがより興味深いものになります。例からのルーターの役割は単純に拡張されます。インスタンスをさらに増やし、それらを既存のクラスターにフックします-これで完了です！ストレージの役割はやや複雑です。ストレージはシャード可能であるため、インスタンスを追加/削除するときは、データをリバランスして、新しいインスタンスに移動したり、削除されたインスタンスから移動したりする必要があります。これが行われなかった場合、1つのケースではインスタンスが過負荷になり、2番目のインスタンスではデータが失われます。そして、1つだけでなく、トポロジが異なる12個のクラスターが稼働している場合はどうでしょうか。

一般に、Tarantoolオペレーターはこれらすべてに忙しい。ユーザーはTarantool Cartridgeクラスターの望ましい状態を説明し、オペレーターはこれをk8sリソースに対する一連のアクションとTarantoolクラスター管理APIへの特定の呼び出しに変換します-特定の順序で特定の瞬間に、一般的にユーザーからすべてのニュアンスを隠そうとします。

更新は、2フェーズコミットメカニズムを使用して協調して行われます。アップグレードを成功させるには、100％のクォーラムが必要です。各インスタンスが応答する必要があります。運用上、これはどういう意味ですか？クラスターの状態を変更する管理APIへのすべてのリクエストは、1つのインスタンス、リーダーに送信するのが最も信頼できます。そうしないと、インスタンスごとに異なる設定が取得されるリスクがあります。 Tarantool Cartridgeはリーダー選挙の方法を知りません（しかし、方法はわかりません）が、Tarantool Operatorは知っています-オペレーターはすべてを台無しにするので、これは面白い事実としてしか知ることができません。

オペレーターの仕組み


オペレーターのタスクは、システムをユーザーが設定した状態にし、新しい指示を受け取るまでシステムをこの状態に維持することです。オペレーターが自分の作業を実行できるようにするには、次のものが必要です。

  1. システムステータスの説明。
  2. システムをこの状態にするコード。
  3. このコードをk8sに統合するためのメカニズム（たとえば、状態変化の通知を受け取るため）。

Tarantool Cartridgeクラスターは、カスタムリソース定義（CRD）を介してk8sの観点から説明されています。オペレーターには、tarantool.io / v1alphaグループの下で統合された3つのカスタムリソースが必要です。

  • クラスターは、1つのTarantool Cartridgeクラスターに対応するトップレベルのリソースです。
  • 役割-Tarantool Cartridgeに関しては、これはユーザーの役割です。
  • ReplicasetTemplate-StatefulSetが作成されるテンプレート（ステートフルな理由-後で少し説明します。k8sReplicaSetと混同しないでください）。

これらのリソースはすべて、Tarantool Cartridgeクラスター記述モデルを直接反映しています。共通のディクショナリがあるため、オペレーターは開発者とやり取りして、製品に何を表示したいかを理解するのが簡単になります。

システムを特定の状態にするコード-k8sに関しては、これはコントローラーです。Tarantoolオペレーターの場合、いくつかのコントローラーがあります。

  • ClusterController-Tarantool Cartridgeクラスターとの対話を担当し、インスタンスをクラスターに接続し、インスタンスをクラスターから切断します。
  • RoleController-ユーザーロールコントローラー。テンプレートからStatefulSetをデプロイし、その数を指定された数に維持します。

コントローラはどのようなものですか？徐々にあなたの周りの世界を整えます。ClusterControllerは、次のように概略的に表すことができます



。エントリポイントは、イベントが発生したクラスタリソースが存在するかどうかを確認するチェックです。存在しません？出発します。存在しますか？次のブロックに進みます。ユーザーロールの所有権を取得します。1つをキャプチャ-左、2番目の円で2つ目をキャプチャします。など、すべてをキャプチャするまで。すべての役割が取得されていますか？次の操作ブロックに進んでください。そして、最後に到達するまで。次に、制御対象システムが特定の状態にあると想定できます。

一般的に、すべてが簡単です。各ステージを通過するための成功基準を決定することが重要です。たとえば、条件付き成功= trueを返したときではなく、「すでに参加しています」のようなエラーを返したときに、クラスターに参加する操作が成功したと見なします。

このメカニズムの最後の部分は、コントローラーとk8sの統合です。航空写真では、k8全体は、イベントを生成してそれらに応答する一連のコントローラーで構成されています。イベントは、サブスクライブできるキューを通過します。概略的には、次のように表すことができ



ます。kubectl create -f tarantool_cluster.yaml、対応するクラスターリソースが作成されます。ClusterControllerは、クラスターリソースの作成を通知されます。そして、彼が最初にしようとしていることは、このクラスターの一部であるはずのすべてのロールリソースを見つけることです。見つかった場合は、クラスターを役割の所有者として割り当て、役割リソースを更新します。RoleControllerはロール更新通知を受け取り、リソースに所有者がいることを確認して、StatefulSetの作成を開始します。サークル内のように：2番目の最初のストライガー、3番目の2番目のストライガー-誰かが止まるまで続きます。また、たとえば5秒ごとにトリガーすることもできます。

これがオペレーター全体です。カスタムリソースを作成し、リソースのイベントに応答するコードを記述します。

オペレーターが展開するもの


オペレーターのアクションは最終的にポッドとコンテナーを作成するk8につながります。 k8sにデプロイされたTarantool Cartridgeクラスターでは、すべてのポッドがStatefulSetにマージされます。

なぜStatefulSetなのか？以前に書いたように、Tarantool Clusterの各インスタンスは、クラスターのトポロジーと構成のコピーを保持し、多くの場合、アプリサーバー上でno、noを保持します。たとえば、順番に、またはデータを参照するなど、何らかの種類のスペースを使用しますが、これはすでに完全な状態です。また、StatefulSetはIDポッドの保持も保証します。これは、クラスター内のインスタンスをクラスター化するときに重要です。インスタンスのIDを修正する必要があります。そうしないと、再起動時にクォーラムが失われるおそれがあります。

すべてのクラスターリソースが作成され、目的の状態になると、次の階層が形成されます。



矢印は、リソース間の所有者と依存関係を示しています。クラスターの削除などの場合、ガベージコレクターがクリーンアップする必要があります。

ステートフルセットに加えて、Tarantoolオペレーターはリーダーレス選挙に必要なヘッドレスサービスを作成し、それを介してインスタンスが相互に通信します。

Tarantoolオペレーターの内部にはオペレーターフレームワークがあり、オペレーターコード自体はgolangにあります。ここで特別なことはありません。
