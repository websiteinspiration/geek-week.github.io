Windows Server 2008 R2-王は死んでいる、王は長生きする



こんにちは、ハブル！2020年1月14日が近づいており、この日付はWindows Server 2008 R2の拡張サポートの終了を示しています。WindowsServer 2008 R2を引き続き使用している場合は、より最新のプラットフォームへの移行を検討する必要があります。

カットの下には、新旧のWindows Serverプラットフォームで消費されたリソースの小さな比較があります-RDSHの役割。まず、VMware Vsphere 5.5を実行している既存のHP G6 / G7サーバーを引き続き使用する機会に興味がありました。

Windows Server Lifecycleに関する情報は、このSearch Product Lifecycleリンクから入手できます。

2つのテストベンチがあります。


  1. HP DL120 G7、1 * CPU Intel Xeon E3-1240
  2. HP ML350 G6、2 * CPU Intel Xeon E5620

サーバーはVMware ESXi 5.5、イメージVMware-ESXi-5.5.0-Update1-1746018-HP-5.75.4-Dec2014を実行します。 VMware-ESXi-5.5.0-Update3-3116895-HP-550.9.4.26-Nov2015イメージも使用してみましたが、これは結果には影響しません。

ただし、VMware-ESXi-5.5.0-Update2-2403361-HP-550.9.2.40.2-Sep2015のイメージは、サーバーが表示する統計が実際の統計と2倍異なるため、何らかの理由で結果が歪んでいます。この画像では、サーバーが表示する最大値はプロセッサ負荷の50％であり、明らかに何らかのバグです。最初は最初の結果が出て嬉しかったのですが、何かがおかしいことに気づき......

また、VMware Vsphere 5.5のサポートも永遠ではなく、プラットフォームはもはや関係がないため、更新を検討する必要があることにも言及する必要があります。VMwareライフサイクルに関する情報は、このVMwareライフサイクル製品マトリックスで入手できます。



また、VMware互換性ガイドもお忘れなく。HP DL120 G7サーバーについて話す場合、このサーバーは互換性マトリックスに存在していませんが、ESXi 5.5 U3はHP ML350 G6サーバーへのインストールを正式に許可されています。

ゲストOS


ゲストOSとのESXiの互換性に関する情報は、 『VMware互換性ガイド』にも記載されています。

ESXi 5.5 U3では、ゲストOSとしてWindows Server 2016をインストールできます。バージョン5.5でも十分です。ただし、Windows Server 2019をインストールするには、ESXiバージョン6以降が必要ですが、これによって計画のテストが妨げられることはありません。

次のテストVMを用意します：WS2008R2SP1、WS2012R2、WS2016、およびWS2019。

VMware Tools 5.5.0-10.2.5.8068406エージェントがすべてのVMにインストールされています。 2019年5月のMicrosoftからのすべてのアップデート、さらには2019年6月のアップデートもインストールされた後、アップデートサービスが無効になりました。

私のテストでは、最低限の調整を行いました。すべてのものが最も標準的なパラメーターで機能するようにしたいと思います。これを行うには、ローカルグループポリシーを使用します。これについては、最後に少し説明します。

  1. 「デスクトップ構成」を無効化
  2. オーディオとビデオのリダイレクトを許可
  3. WS2008R2SP1およびWS2008R2SP1クライアントで有効なRemoteFX
  4. 無効化されたフォントスムージング
  5. カラーパレットのビット深度を32に制限
  6. モニターの数を1の値に制限

RemoteFXコーデックがWS2008R2SP1で動作を開始するには、RDSHロールをインストールする必要があります。新しいバージョンでは、これを省略できます。RDSHをインストールせずにWS2008R2SP1でRemoteFXを使用している可能性がありますが、このコーデックのアクティブ化に関するログにイベントはありません。

最後の部分-チューニング


私はすべてのテストVMをHP DL120G7サーバーに戻し、Windows 10を搭載したラップトップと、解像度が1366 * 768のネイティブ画面をクライアントとして使用しました。

残念ながら、プロセッサーの消費を減らす試みを成功とは言いませんが、結果もないということは不可能です。以下は、チューニングに使用できるすべての利用可能なローカルグループポリシー設定のリストです。


Windows Server 2008R2では、[RemoteFX使用時のビジュアルエクスペリエンスの最適化]オプションを使用して、画質とフレーム数を制御でき、これにより顕著な結果が得られます。フレーム数を減らすと、プロセッサとネットワークの負荷が軽減されます。 Windows Server 2019の場合、これらのオプションは機能しません。


Windows Server 2019の画像の品質は、「RemoteFXアダプティブグラフィックスの画質を構成する」パラメーターで制御できますが、フレーム数を変更するものがないか、そのようなパラメーターが見つかりませんでした。

パラメータのさまざまな組み合わせを試しましたが、わずかなケースで結果に顕著な変化が見られました。そして、最も興味深いのは、トラフィックの変化に気付くことができることですが、同時に、プロセッサの負荷はほとんど変化しません。

-H.264 / AVC 444の優先順位付けにより、プロセッサの負荷が増加しましたが、同時にトラフィックが2倍から1メガビットに大幅に減少しました。
このポリシー設定は、RemoteFX以外のvGPUシナリオのH.264 / AVC 444グラフィックモードを優先します。RDPサーバーでこの設定を使用すると、サーバーは、クライアントとサーバーの両方がH.264 / AVC 444を使用できるRDP 10接続のコーデックとしてH.264 / AVC 444を使用します。

-RDP圧縮を無効にしてもプロセッサの負荷は減少しませんでしたが、同時にトラフィックが2メガビットから7メガビットに増加しました。

-画質のLoslesパラメータを含めると、プロセッサの負荷を変更することなく、トラフィックの消費が不十分になります。どのオプションについて、このオプションが私に追加されたのかは完全に明確ではありません。
このポリシー設定を有効にして品質をロスレスに設定すると、RemoteFXアダプティブグラフィックスはロスレスエンコーディングを使用します。このモードでは、グラフィックスデータの色の整合性は影響を受けません。ただし、この設定により、ネットワーク帯域幅の消費量が大幅に増加します。これは非常に特殊な場合にのみ設定することをお勧めします。




概要


以前は、10人のアクティブユーザーの作業では、8台の核サーバーで十分であり、まだ空き容量がありましたが、現在、このようなサーバーで十分なのは5人のユーザーだけです。

HP G6およびG7世代のサーバーを交換する準備が整ったときだと思います。100 GBを超えるRAMがインストールされているサーバーを手放すのは残念です。

古いプロセッサにハードウェア命令がないことを除外するつもりはないので、近い将来、テスト用に最新のサーバーを入手する予定です。正直なところ、私の予測は悲観的であり、ブルートフォース、プロセッサコアの数の増加によってのみ望ましい結果を得ることが可能になると思います。これにより、必要なライセンスの数とコストが増加します。
