Docker Composeでビルドするときに秘密を保護する

この記事の翻訳は、Python Web Developerコースの開始前に準備されました。



Dockerイメージをビルドするときに、プライベートパッケージリポジトリのパスワードなどのシークレットが必要になる場合があります。このシークレットが最終的にイメージに含まれるのは望ましくありません。イメージにアクセスした人はだれでもプライベートリポジトリにアクセスできるからです。
注：「環境変数を使用しないのはなぜですか？」と考える場合、イメージの作成時にランタイムのシークレットに使用されます。この記事では、Dockerファイルを使用してイメージを作成するときに使用されるビルドシークレットに焦点を当てます。
新しいバージョンのDockerは、実験的なBuildKitサービスを使用してシークレットを維持します。DockerCompose 1.25以降では、BuildKitを使用してすでにイメージを作成できます。残念ながら、2020年3月の時点で、Composeのシークレットを安全に処理する機能はまだ開発中です。

それで、今何をすべきか？

本日の記事では、同じDockerfileを使用して、Docker Composeを使用した迅速な開発の利点を失うことなく、シークレットイメージを安全に作成する方法を紹介します。

dockerfileを使用するための2つのオプション


本番用とDocker Composeを使用したローカル開発用に同じDockerfileを使用すると非常に便利です。通常、DockerfileはComposeのビルド関数と共に使用します。

version: "3.7"
services:
  yourapp:
    build:
      context: "."

次に、次のことができます。

$ docker-compose up

このコマンドを使用すると、イメージを（再）コンパイルして実行できます。
本番環境で使用するには、画像を収集してpushで送信します。

$ docker build -t myimage .
$ docker push myimage

そして、すべてが順調に進んでいる間。しかし、秘密のビルドが必要な場合はどうでしょうか？

最初の試み（危険）


たとえば、プライベートDevPIリポジトリからPythonパッケージをダウンロードするために、シークレットビルドが必要なスクリプトがあるとします。簡単にするために、秘密があることuse-secret.shを示すために、その秘密を導き出します。

#!/bin/bash
set -euo pipefail

echo "Secret is: $THEPASSWORD"

Docker Composeを含むすべての場所でサポートされているため、Dockerビルド引数を使用してシークレットを渡すだけです。
注：ディスカッションの範囲を超えて、この記事でDockerファイルを使用することはベストプラクティスではありませんが、複雑すぎると記事の主な意味を伝えるのに支障をきたす可能性があります。
したがって、Dockerを使用して本番環境でPythonアプリケーションを実行する場合は、次の2つの方法でこれを行うことができます。

  • DIYが必要な場合：例とリンクを含む詳細なリスト
  • できるだけ早く機能するバージョンが必要な場合：ベストプラクティスを考慮したテンプレート

FROM python:3.8-slim-buster
# Using ARG for build secrets is INSECURE!
ARG THEPASSWORD
COPY use_secret.sh .
RUN ./use_secret.sh

docker-compose.yml秘密裏に送信されるを 書くことができます：

version: "3.7"
services:
  yourapp:
    build:
      context: "."
      args:
        THEPASSWORD: "s3kr!t"

ローカル作業の場合、Composeを使用してイメージを実行またはビルドできます。

$ docker-compose build | grep Secret
Secret is: s3kr!t

そして、すべてが良いです。

また、イメージをイメージレジストリに移動する準備として、Dockerを使用してイメージをアセンブルすることもできます。

$ docker build -t myimage --build-arg THEPASSWORD=s3krit . | grep Secret
Secret is: s3krit

これを行うのは危険です。絶対に行わないでください。画像のレイヤーを確認すると、その秘密がそこにあるのがわかります。

$ docker history myimage
IMAGE               CREATED              CREATED BY                                      SIZE
c224231ec30b        47 seconds ago       |1 THEPASSWORD=s3krit /bin/sh -c ./use_secre…   0B
6aef62acf0db        48 seconds ago       /bin/sh -c #(nop) COPY file:7aa28bbe6595e0d5…   62B
f88b19ca8e65        About a minute ago   /bin/sh -c #(nop)  ARG THEPASSWORD              0B
...

この画像にアクセスした人は誰でもあなたのパスワードを認識します。次に何ができるでしょうか？

BuildKitシークレット（部分的なソリューション）


BuildKitは、Dockerイメージを作成するための新しい（まだ実験的な）ソリューションであり、特に、アセンブリ中にシークレットを安全に使用するためのサポートを追加します。Docker Composeはv1.25以降、BuildKitをサポートしています。

ただし、1つの問題があります。DockerComposeはまだBuildKitシークレット機能をサポートしていません。現在、この作業は進行中ですが、2020年3月現在、安定版リリースは言うまでもなく、既製のソリューションはありません。

したがって、これらの2つのアプローチを組み合わせます。

  • Docker Composeは引き続きシークレットを渡すためにビルド引数を使用します。
  • docker buildを使用してビルドされた本番イメージの場合、BuildKitを使用してシークレットを渡します。

このようにして、同じDockerfileを使用してローカルおよび本番環境で作業できます。
BuildKitはシークレットを次のように処理し/var/secrets/thepasswordます。たとえば、でRUNコマンドが実行されている間、シークレットを含むファイルは一時ディレクトリにマウントされます。RUNコマンドの実行中にマウントされるため、最終的なイメージには追加されません。

ファイルuse_secret.shを変更して、そのような一時ファイルが存在するかどうかを確認します。存在する場合は、環境変数の設定を使用します$THEPASSWORD。ファイルが存在しない場合は、環境変数に戻ります。つまり、$THEPASSWORDBuildKitまたはビルド引数を使用してインストールできます。

#!/bin/bash
set -euo pipefail
if [ -f /run/secrets/thepassword ]; then
   export THEPASSWORD=$(cat /run/secrets/thepassword)
fi

echo "Secret is: $THEPASSWORD"

次に、Dockerfileを変更してBuildKitを追加し、シークレットをマウントします。

# syntax = docker/dockerfile:1.0-experimental
FROM python:3.8-slim-buster
# Only use the build arg for local development:
ARG THEPASSWORD
COPY use_secret.sh .
# Mount the secret to /run/secrets:
RUN --mount=type=secret,id=thepassword ./use_secret.sh

ファイルdocker-compose.ymlは変更しません。

version: "3.7"
services:
  yourapp:
    build:
      context: "."
      args:
        THEPASSWORD: "s3kr!t"

次に、2つの環境変数を定義する必要があります。1つはBuildKitを使用する必要があることをDockerに通知し、2つ目はComposeがDockerのCLIバージョンを使用する必要があること、したがってBuildKitを使用することを通知します。シークレットもファイルに書き込みます。

$ export DOCKER_BUILDKIT=1
$ export COMPOSE_DOCKER_CLI_BUILD=1
$ echo 's3krit' > /tmp/mypassword

Composeでは、ビルド引数を使用します。

$ docker-compose build --progress=plain \
    --no-cache 2>&1 | grep Secret
#12 0.347 Secret is: s3kr!t

--no-cache上記のすべてを実行すると、イメージが本当に再構築されることを理解する必要 があることに注意してください。実際には、このパラメーターは省略できます。正しい操作のために2>&1リダイレクトstderrしstdoutますgrep。

本番環境でビルドする準備ができたら、BuildKitのシークレット機能を備えたdocker buildを使用します。

$ docker build --no-cache -t myimage \
    --secret id=thepassword,src=/tmp/mypassword \
    --progress=plain . 2>&1 | grep Secret
#12 0.359 Secret is: s3krit

安全ですか？


秘密が見えないようにしましょう：

$ docker history myimage
IMAGE               CREATED             CREATED BY                                      SIZE
a77f3c32b723        25 seconds ago      RUN |1 THEPASSWORD= /bin/sh -c ./use_secret.…   0B
<missing>           25 seconds ago      COPY use_secret.sh . # buildkit                 160B
...

やったー！Composeとを使用して同じDockerfileにシークレットを渡しましたdocker buildが、後者の場合、アセンブリからシークレットは明らかにされませんでした。
