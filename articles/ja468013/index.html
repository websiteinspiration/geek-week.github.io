Helmでカナリアデプロイメントを自動化する簡単で安全な方法



カナリアデプロイメントは、ユーザーのサブセットで新しいコードをテストする非常に効果的な方法です。特定のサブグループ内でのみ発生するため、展開プロセス中に問題を引き起こす可能性があるトラフィック負荷を大幅に軽減します。この記事では、Kubernetesとデプロイの自動化を使用して同様のデプロイを整理する方法について説明します。HelmとKubernetesのリソースについてある程度知っていることを前提としています。



Kubernetesのシンプルなカナリアデプロイには、サービス自体とデプロイツールという2つの主要なリソースが含まれています。カナリアデプロイメントは、更新トラフィックを処理する2つの異なるリソースと相互作用する1つのサービスを通じて機能します。これらのリソースの1つは「カナリア」バージョンで動作し、2番目のリソースは安定版で動作します。この状況では、メンテナンスに必要なトラフィック量を減らすために、カナリアバージョンの数を調整できます。たとえば、Yamlを使用する場合、Kubernetesで次のように表示されます。

kind: Deployment
metadata:
  name: app-canary
  labels:
    app: app
spec:
  replicas: 1
  ...
    image: myapp:canary
---
kind: Deployment
metadata:
  name: app
  labels:
    app: app
spec:
  replicas: 5
  ...
    image: myapp:stable
---
kind: Service
selector:
  app: app # Selector will route traffic to both deployments.

kubectlでこのオプションを想像するのはさらに簡単です。Kubernetesのドキュメントには、このシナリオに関する完全なチュートリアルもあります。しかし、この投稿の主な問題は、Helmを使用してこのプロセスを自動化する方法です。

カナリアデプロイメントの自動化


まず、Helmチャートマップが必要です。このマップには、上記で説明したリソースが既に追加されています。次のようになります。

~/charts/app
├── Chart.yaml
├── README.md
├── templates
│   ├── NOTES.txt
│   ├── _helpers.tpl
│   ├── deployment.yaml
│   └── service.yaml
└── values.yaml

Helmの中核は、マルチリリースのリリース管理です。安定版は、プロジェクトコードの主要な安定版ブランチです。しかし、Helmでは、実験的なコードでカナリアリリースをデプロイできます。主なことは、安定版とカナリアリリースの間のトラフィック交換を維持することです。これらすべてを、特別なセレクターを使用して管理します。

selector:
  app.kubernetes.io/name: myapp

「カナリア」と安定した展開リソースの両方で、モジュールにこのラベルが表示されます。すべてが正しく設定されている場合、Helmチャートのカナリアバージョンのデプロイ中に、トラフィックが新しくデプロイされたモジュールに送られることがわかります。このコマンドの安定版は次のようになります。

helm upgrade
  --install myapp \
  --namespace default \
  --set app.name=myapp \      # Goes into app.kubernetes.io/name
  --set app.version=v1 \      # Goes into app.kubernetes.io/version
  --set image.tag=stable \
  --set replicaCount=5

では、カナリアリリースを見てみましょう。カナリアバージョンをデプロイするには、2つのことを覚えておく必要があります。現在の安定したバージョンで更新をロールアップしないように、リリース名は異なる必要があります。異なるコードをデプロイし、リソースラベルによって違いを識別できるように、バージョンとタグも異なる必要があります。

helm upgrade
  --install myapp-canary \
  --namespace default \
  --set app.name=myapp \      # Goes into app.kubernetes.io/name
  --set app.version=v2 \      # Goes into app.kubernetes.io/version
  --set image.tag=canary \
  --set replicaCount=1

それが実際にすべてです！サービスにpingを実行すると、カナリア更新が一部の時間のみトラフィックをルーティングしていることがわかります。


説明したロジックを含むデプロイメント自動化ツールを探している場合は、GitHubでDeliverybotとHelm自動化ツールを確認してください。上記の方法を実装するために使用されるHelmチャートは、ここにあるGithubにあります。一般的に、これは実際のカナリアバージョンのデプロイを実装する方法の理論的な概要であり、特定の概念と例が含まれています。
