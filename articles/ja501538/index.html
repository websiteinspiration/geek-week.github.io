Pythonの場合、そのような操作はできないようです。

Pythonのインデント問題を解決する


Pythonのインデントは、次のコマンドによって形成されます。

- class
- def

- for
- while

- if

- try

- with

そして、インデントへの依存を取り除くために、これらのコマンドのそれぞれについて、コマンドのブロック（インデント）を確実に閉じる「最終」コマンドを使用することをルールにすることにしました。

クラスおよびdefコマンド


class / defコマンドの場合、通常、インデントブロックを完了するのに問題はありません。新しいclass / defコマンドでブロックが閉じられます。

唯一のケースは、別のサブプログラム/メソッド内で宣言された1つ以上のサブプログラムの存在です。

    def ppg_2():
        ...  ppg_2 ...
    def ppg_3():
        ...  ppg_3 ...
        ...  ppg_3 ...
        ...  ppg_3 ...
    ...  ppg_1 ...

次に、最後の内部サブプログラムの命令のブロックを誤ってシフトすると、この内部サブプログラムが宣言されているサブプログラム/メソッドのコマンドとマージされます。

    def ppg_2():
        ...  ppg_2 ...
    def ppg_3():
        ...  ppg_3 ...
    ...  ppg_3 ...
    ...  ppg_3 ...
    ...  ppg_1 ...

この場合、この内部ルーチン/メソッドの最後に「リターン」を置くだけで、コマンドのブロックの終わりを明確に示すことができます。

def ppg_1():
    def ppg_2():
        ...  ppg_2 ...
    def ppg_3():
        ...  ppg_3 ...
        ...  ppg_3 ...
        ...  ppg_3 ...
        return

コマンドを試す


「if」と完全に類似しています。

try:                   #  
    ...  ....
except <...>:
    ...  ....
except 1:              #  "else"
    ...  ....
except 0: pass         #

唯一の問題は、finally：コマンドです。その後、現在のtryブロックのコマンドを置くことができなくなります。

したがって、それを使用する必要がある場合、自動フォーマットの可能性を維持し、インデントを誤って削除しないようにするには、ローカルサブプログラムの「finally：」の後にコマンドブロック全体を削除する必要があります（つまり、現在のサブプログラム内のサブプログラムとして宣言します）。

それら。「finally：」を含むテキストは次のようになります。

    def my_ppg():
        ...
        return

    ...

    finally:
        my_ppg()

    ...

この場合、オートフォーマットを安全に適用して、
インデントを誤って削除することを恐れないようにすることもできます。

「with」コマンド


ブロックの終わりとして機能するPythonの「with」用の追加コマンドはありません。したがって、withの状況は、finallyの状況と似ています。
それら。 「with」ステートメントブロックで実行されたすべてのコマンドをローカルサブルーチンに転送するか、または...しかし、私はひどく冒とく的なことを言います。「...または、それを使用する必要がないだけです。」

事実、一方で「with」は既存のPythonコマンドの「ラッパー」にすぎない（つまり、いつでも同様のコマンドセットで置き換えることができる）一方で、実践ではジュニアプログラマー向けにこのコンテキストが示されていますマネージャーは完全な開発が難しい。したがって、後のプログラマーが実装されたプロジェクトに落ち着いて同行するようにしたい場合は、そのプロジェクトの作業を妨げるチームを使用する必要はありません。
