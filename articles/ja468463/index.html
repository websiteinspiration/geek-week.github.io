パーティショニングとインデックス作成によるZabbix + PostgreSQLのパフォーマンスの向上

約1年前、私と同僚は、人気のあるネットワークインフラストラクチャ監視システムであるZabbixを使用して問題を解決する必要がありました。ドキュメントを検討した後、すぐに負荷テストに進みました。Zabbixが顕著なパフォーマンスの低下なしに機能できるパラメーターの数を評価したかったのです。 PostgreSQLのみがDBMSとして使用されました。

テスト中に、データベースレイアウトの一部のアーキテクチャ機能と監視システム自体の動作が特定されました。これにより、デフォルトでは、監視システムが最大電力に到達できません。その結果、主にデータベースのチューニングに関して、いくつかの最適化対策が開発、実施、およびテストされました。

この記事で行った作業の結果を共有したいと思います。この記事は、ZabbixとPostgreSQLの両方のDBA管理者、および一般的なPosgreSQL DBMSをよりよく理解して理解したいすべての人に役立ちます。

小さなスポイラー：毎分20万パラメーターの負荷がかかる弱いマシンで、CPU iowaitを20％から2％に削減し、プライマリテーブルへの部分的な記録時間を250倍に、集約データテーブルを32倍に削減し、インデックスのサイズを削減しました。 5〜10倍、場合によっては18倍までの履歴サンプルの取得を高速化します。

ストレステスト


負荷テストは、1つのZabbixサーバー、1つのアクティブなZabbixプロキシ、2つのエージェントというスキームに従って実行されました。各エージェントは、毎分50トンの整数と50トンの文字列パラメーターを提供するように構成されました（合計200のエージェント、毎分200トンのパラメーター、または毎秒3333パラメーター）。エージェントパラメータを生成するには、Zabbixのプラグインを使用しました。エージェントが生成できるパラメータの数を確認するには、同じプラグイン作成者zabbix_module_stressの特別なスクリプトを使用する必要があります。Zabbix web-adminは大きなテンプレートの登録に問題があるため、パラメーターを5トンのパラメーター（2500の数値と2500の文字列）を含む20のテンプレートに分割しました。

これに加えて、1時間ごとの統計が計算されると同時に、状況が悪化します-同時に、ディスクからのインデックスページのアクティブな読み取りと書き込み、履歴からの古いデータの削除が実行され、同じ結果につながります-データベースのパフォーマンスの低下と実行時間の増加リクエスト（制限では、最大5分間続くリクエストが記録されました！）

最適化活動


PostgreSQLデータベースのパフォーマンスを向上させるために、さまざまな最適化手段が実行されました。その主なものは、インデックスのパーティション化と変更です。ただし、PostgreSQLデータベース管理システム下の任意のデータベースの動作を高速化できるいくつかの重要で有用な対策について、一言言及する価値があります。

重要なお知らせ。記事の資料の収集時には、Zabbixバージョン4.0を使用しましたが、バージョン4.2はすでにリリースされており、バージョン4.4のリリースが準備されています。これについて言及することがなぜ重要なのですか？バージョン4.2から、ZabbixはTimescaleDB時系列を操作するための特別な強力な拡張機能をサポートし始めましたが、これまでの実験モードでは、この拡張機能を使用することのすべての利点について、一部のリクエストの動作が遅くなり、未解決のパフォーマンスの問題が残っていると考えられます（バージョン4.4で解決）- この記事を読んでください。次の記事では、このソリューションのケースと比較して、TimescaleDB拡張機能をすでに使用している負荷テストの結果について説明する予定です。PostgreSQLのバージョン10が使用されましたが、指定されたすべての情報は11および12バージョンに関連しています（待機中！​​）。

pgtuneユーティリティを使用した構成ファイルの構成


実際、PostgreSQLはかなり軽量のDBMSです。そのデフォルトの構成ファイルは、私の同僚が言うように、「コーヒーマシンで作業する」ように構成されています。非常に控えめなアイアンで。したがって、メモリの量、プロセッサの数、データベースの使用目的の種類、ディスクの種類（HDDまたはSSD）、および接続数を考慮して、サーバー構成用にPostgreSQLを構成する必要があります。

悲しいかな、すべてのDBMSをチューニングするための単一の公式はありませんが、ほとんどの構成に適した特定のルールとパターンがあります（より正確なチューニングはすでに専門家の仕事です）。DBAの生活を簡素化するために、pgtuneユーティリティが作成されました。これは、ユーザーによってWebバージョンによって補足されました。le0pard-PostgreSQLの管理に関する興味深く有用な本の著者。

「データウェアハウス」アプリケーションタイプの100接続（Zabbixには厳しいWeb管理者がいる）を使用してコンソールでユーティリティを実行する例：

pgtune -i postgresql.conf -o new_postgresql.conf -T DW -c 100

pg_pathmanを使用した履歴テーブルのパーティション分割


Zabbixのストレステスト中に発生した問題の1つ-PostgreSQLは、古いデータをデータベースから削除できません。パーティション化の助けを借りて、テーブルをその構成部分に分割し、それによってインデックス全体のサイズとスーパーテーブルの構成部分を減らすことができ、データベース全体の速度にプラスの影響を与えます。

解く一度に2つの問題仕切る：

テーブル全体を削除することによって、古いデータの削除を促進1.

2分割インデックスを各複合テーブルに対して

：PostgreSQLのに区画する4つの機構が存在する

。1.標準constraint_exclusionの

2 pg_partman拡張子（pg_pathmanとしない混同行う）

3. pg_pathmanの拡張は

4.自分でパーティションを手動で作成して保守するパーティション分割

の最も便利で信頼性が高く、最適化されたソリューションは、pg_pathman拡張です。このパーティション分割方法を使用すると、クエリプランナーは、どのパーティションでデータを検索するかを柔軟に決定します。噂によると、PostgreSQLの12番目のバージョンでは、すぐに使える優れたパーティションがすでに存在します。

したがって、私たちはスーパーテーブルとは別の継承されたテーブルに毎日の監視データを書き込み始め、古いパラメーター値の削除は、すべての古いテーブルを一度に削除することによって行われ始めました。これは、DBMSにとって人件費ではるかに簡単です。削除は、データベースのユーザー定義関数をZabbixサーバーの監視パラメーターとして午前2時に呼び出し、統計ストレージの許容範囲を示して行われました。

CREATE OR REPLACE FUNCTION public.delete_old_partitions(history_days integer, trends_days integer, str_days integer)
 RETURNS text
 LANGUAGE plpgsql
AS $function$
/*
   , ,    :
history_days -   history_x, history_uint_x
trends_days  -   trends_x, trends_uint_x
str_days     -   history_str_x, history_text_x, history_log_x
*/
declare clock_today_start         int;
declare clock_delete_less_history int  = 0;
declare clock_delete_less_trends  int  = 0;
declare clock_delete_less_strings int  = 0;
clock_delete_less                 int  = 0;
declare iterator                  int  = 0;
declare result_str                text = '';
declare buf_table_size            text;
declare buf_table_len             text;
declare partition_name            text;
declare clock_max                 text;
declare err_detail                text;
declare t_start                   timestamp = clock_timestamp();
declare t_end                     timestamp;
begin
    if $1 <= 0 then return 'ups, something wrong: history_days argument must be positive integer value'; end if;
    if $2 <= 0 then return 'ups, something wrong: trends_days argument must be positive integer value'; end if;
    if $3 <= 0 then return 'ups, something wrong: str_days argument must be positive integer value'; end if;
    clock_today_start = extract(epoch from date_trunc('day', now()))::int;
    clock_delete_less_history = extract(epoch from date_trunc('day', now()) - ($1::text || ' days')::interval)::int;
    clock_delete_less_trends  = extract(epoch from date_trunc('day', now()) - ($2::text || ' days')::interval)::int;
    clock_delete_less_strings = extract(epoch from date_trunc('day', now()) - ($3::text || ' days')::interval)::int;
    clock_delete_less = least(clock_delete_less_history, clock_delete_less_trends, clock_delete_less_strings);
    --raise notice 'clock_today_start % (%)', to_timestamp(clock_today_start), clock_today_start;
    --raise notice 'clock_delete_less_history % (%) % days', to_timestamp(clock_delete_less_history), clock_delete_less_history, $1;
    --raise notice 'clock_delete_less_trends  % (%) % days', to_timestamp(clock_delete_less_trends),  clock_delete_less_trends,  $2;
    --raise notice 'clock_delete_less_strings % (%) % days', to_timestamp(clock_delete_less_strings), clock_delete_less_strings, $3;
    for partition_name, clock_max in select partition, range_max from pathman_partition_list where 
    range_max::int <= greatest(clock_delete_less_history, clock_delete_less_trends, clock_delete_less_strings) and 
    (partition::text like 'history%' or partition::text like 'trends%') order by partition asc
    loop
        if (partition_name ~ 'history_uint_\d'  and clock_max::int <= clock_delete_less_history)
        or (partition_name ~ 'history_\d'       and clock_max::int <= clock_delete_less_history)
        or (partition_name ~ 'trends_\d'        and clock_max::int <= clock_delete_less_trends)
        or (partition_name ~ 'history_log_\d'   and clock_max::int <= clock_delete_less_strings)
        or (partition_name ~ 'history_str_\d'   and clock_max::int <= clock_delete_less_strings)
        or (partition_name ~ 'history_text_\d'  and clock_max::int <= clock_delete_less_strings)
        then 
            iterator = iterator + 1;
            raise notice '%', format('!!! delete %s %s', partition_name, clock_max);
            select max(reltuples::int), pg_size_pretty(sum(pg_relation_size(pg_class.oid))) as "size" from pg_class where relname like partition_name || '%' into strict buf_table_len, buf_table_size;
            if result_str != '' then result_str = result_str || ', '; end if;
            result_str = result_str || format('%s (dt < %s, len %s, %s)', partition_name, to_char(to_timestamp(clock_max::int), 'YYYY-MM-DD'), buf_table_len, buf_table_size);
            execute format('drop table if exists %s', partition_name);
        end if;
    end loop;
    if iterator = 0 then  result_str = format('there is no partitions to delete older, then %s date', to_char(to_timestamp(clock_delete_less), 'YYYY-MM-DD')); 
    else                  result_str = format('deleted %s partitions in %s seconds: ', iterator, trunc(extract(seconds from (clock_timestamp() - t_start))::numeric, 3)) || result_str;
    end if;
    --raise notice '%', result_str;
    return result_str;

履歴テーブルのインデックスタイプをbrin（クロック）およびbtree-gin（itemid）に変更


ユーザーに感謝 エロゴフ以下のためのPostgreSQLのインデックスのレビュー記事の偉大なシリーズ。そして実際、PostgresPROチーム全体。これらの記事に感銘を受けて、監視データテーブルのさまざまな種類のインデックスをいじり、どのフィールドのどの種類のインデックスが最大のパフォーマンス向上をもたらすかという結論に達しました。

複合インデックスbtree（itemid、クロック）は、モニタリングデータのすべてのテーブルにデフォルトで作成されることがわかっています。特に単調に並べられた値の検索は高速ですが、大量のデータ（1,000万を超える）がある場合、ディスク上で「膨らみ」ます。

通常、一意のインデックスは1時間ごとに集計された統計テーブルに作成されますが、これらのテーブルはデータストレージと一意性のためにここでアプリケーションサーバーレベルで提供され、一意のインデックスはデータ挿入を遅くするだけです。

さまざまなインデックスのテスト中に、最も成功したインデックスの組み合わせが明らかになりました。監視データのすべてのテーブルの時計フィールドのbrinインデックスとitemidフィールドのbtree-ginインデックスです。

ブリンインデックスは、イベントの事実のタイムスタンプなど、単調に増加するデータに最適です。時系列。また、btree-ginインデックスは基本的に標準データ型に対するginインデックスであり、一般的には従来のbtreeインデックスよりもはるかに高速です。 ginインデックスは、新しい値の追加時に再構築されませんが、それらによって補足されるだけです。btree-ginインデックスは、PostgreSQLの拡張として配置されます。

クエリ実行統計の収集と分析pg_stat_statements


Pg_stat_statementsは、サーバー全体のクエリパフォーマンスに関する統計を収集するための拡張機能です。この拡張機能の利点は、PostgreSQLログを収集して解析する必要がないことです。

pg_stat_statements拡張機能の使用
psql:

CREATE EXTENSION pg_stat_statements;

postgresql.conf:

shared_preload_libraries = 'pg_stat_statements'
pg_stat_statements.max = 10000 #   sql ,     (     );
pg_stat_statements.track = all # all -   (    ), top -   /, none -  
pg_stat_statements.save = true #

:

SELECT pg_stat_statements_reset();

:
select substring(query from '[^(]*') as query_sub, 
sum(calls) as calls, avg(mean_time) as mean_time
from pg_stat_statements
where
query ~ 'insert into' or query ~ 'update trends'
group by substring(query from '[^(]*')
order by calls desc
