ActiveXサーバーを介したLabVIEW VI PHPアプリケーションへの簡単なWebアクセス

長年、LabVIEWは、組み込みのActiveXサーバーのみを使用して、LabVIEWからの複雑な公開やサーバー設定なしで、WebをVIデバイスに「結び付ける」ことができました。LabVIEW 2020 Communityエディションも例外ではありません。

現在のLabVIEWでは、仮想デバイスをWebで公開する方法がいくつかあり、さまざまなレベルの知識が必要で、さまざまな機能を提供しています。この記事ではそれらについて説明するつもりはありませんが、LabVIEWに組み込まれたActiveX / COMサーバーの非標準的な使用法を紹介して、VIへのWebアクセスを整理し、LabVIEW環境自体を管理します。 ActiveX / COMはすでにWindowsで使用されている古いテクノロジーですが、Webを含めてLabVIEWおよびVIデバイスを簡単に管理できる統合されたActiveXサーバーを介しています。

最初に行うことは、これと同じActiveXサーバーをLabVIEWに含めることです。これは、環境設定で行われます。ツール->オプション-> VIサーバー、ActiveXチェックボックス。


VBScriptの簡単なスクリプトを使用して、サーバーがオンになっていて、サーバーにアクセスできることを確認できます。デスクトップにテキストファイルlabview_test.vbsを作成し、次の内容を入力する必要があります。

Dim obj
Set obj = CreateObject("LabVIEW.Application")
'Dim vi
'Set vi = obj.GetVIReference("C:\Users\Dell\Desktop\LabVIEW Web ActiveX\ActiveX Server Executable _LV2012_NI Verified\Executable as ActiveX Server\ActiveX Server.vi")
WScript.Echo(obj.AppName & " ver: " & obj.Version)
'WScript.Echo(vi.GetControlValue("Count"))
'Set vi = Nothing
Set obj = Nothing

スクリプトを実行する前に、LabVIEW環境を実行してください。ただし、最初に環境を起動しなくても機能します。スクリプトの実行中、LabVIEWインスタンスはActiveX / COMサーバーとして起動され、スクリプトが完了するとインスタンスが閉じられるため、このすべての「ロードおよびアンロード」が完了するまで待機する必要があります。labview_test.vbsの出力は、ルートアプリケーションの名前とそのバージョンになります。


次に、シンプルなVIデバイス「ActiveX Server.vi」を作成しました。いくつかのコントロールと補助機能が含まれています。このVIをロードして管理します。


LabVIEWからは何も必要ありません。これで、Webレイヤーから始めることができます。

GetVIReference（）メソッドは、VIをメモリにロードし、それとの接続を確立します。メインパラメータ：選択したVIへの絶対パス。

ブラウザへのスクリプト出力：

123

確かに、現在のWindowsユーザー用に構成したIISアプリケーションプールと匿名ユーザーの検証の設定を少し調整する必要がありました。

私はASPに深く入り込まないことに決め、PHPに切り替えました。IISの場合、PHP FastCGIデーモンを構成しました。設定は提供していません。この記事の主要部分では重要ではありません。PHPは、タイプ別にLabVIEW COMオブジェクトにアクセスすることもできました。

$obj = new COM('LabVIEW.Application');

どちらの場合も、LabVIEWが実行されていて、VIが開かれている場合（ActiveX Server.vi） PHP（ASP）スクリプトをリクエストすると、LabVIEW.exeの新しいインスタンスが並行して（そしてかなりの時間）起動され、次にGetVIReference（）メソッドを使用して、ActiveX Server.viの独自のインスタンスをロードしました。 PHPスクリプトが完了すると、LabVIEWインスタンスが閉じられました。それら。 LabVIEW環境のすでに実行中のインスタンスとの交差はありませんでした。よく知られているProcess Explorerユーティリティを使用すると、これはよく観察されます。 IISアプリケーションプールの設定を使用した「ゲーム」でも、特別な結果は得られませんでした。私自身、IISはシステムに代わってシステムデーモンとして機能するため、システムコンテキストにバインドされたLabVIEW.exeの別のインスタンスが作成され、すでに開いているインスタンスをWindowsユーザーとして再利用することはできません。

次に、現在のユーザーに代わって、シンプルなアプリケーションモードで実行されているサードパーティのWebサーバーを試すというアイデアが生まれました。その選択はNGINXに落ちました、さらに、私はすでにそれをLabVIEW Webサービスのリバースプロキシとして使用しました。

Nginx


nginx for Windowsの利用可能な現在のバージョンを採用しています。現在nginx-1.17.10。PHPをNGINXに接続するには、次の説明を使用しました。

シンプルな最小限のNGINXセットアップ。私はファイルを持っています：c：\ nginx-1.17.10 \ conf \ nginx.conf
ブラウザにルートディレクトリリストを追加：

nginx.conf：

location / {
	root   html;
	index  index.html index.htm;
	autoindex on;
}

サーバールートでFastCGIを介してPHPを有効にする：

# pass the PHP scripts to FastCGI server listening on 127.0.0.1:9000
#
location ~ \.php$ {
	root           html;
	fastcgi_pass   127.0.0.1:9000;
	fastcgi_index  index.php;
	fastcgi_param  SCRIPT_FILENAME  $document_root$fastcgi_script_name;
	include        fastcgi_params;
}

PHP


現在のバージョンのPHP for Windowsを採用しています。私はphp-7.4.5-nts-Win32-vc15-x64.zipを使用しました。c：\ php-7.4.5-nts-Win32-vc15-x64に

配置します。php.iniの名前を変更して構成します（phpです。アーカイブからのini-development）。次の変更を行います：

php.ini：

short_open_tag = On
html_errors = On
error_reporting = E_ALL & ~E_NOTICE
extension_dir = "ext"

ここでは、画像を操作するためのGDライブラリが接続され（LabVIEWから画像を取得する必要がある場合）、PHPでActiveX / COMオブジェクトを操作するためのphp_com_dotnet.dllモジュールが接続されています。

PHPでCOMを操作しているときに、本文に0x0文字を含む文字列（VT_BSTR）を操作するときに不愉快なバグが発見されました。 php_com_dotnet.dllを修正して再コンパイルしたものに置き換えることで解決します。バグの説明と修正されたパッチはここにあります。残念ながら、それはまだPHPで公式に修正されていません。私はphp_com_dotnet.dll（php-7.4.5-nts-Win32-vc15-x64の場合）を再構築しました。修正されたphp_com_dotnet.dllはリンクにあります。セルフビルドのPHPと拡張機能のガイドは、こちらにあります。

デフォルトでは、NGINXはTCPポート80、PHP FastCGIデーモンはポート9000で実行されます。これらのポートを使用している他の動作しているアプリケーションがないことを確認してください。

NGINXおよびPHP FastCGIデーモンの開始と停止は、さまざまな方法で構成できます。これらのcmdスクリプトは、デバッグのニーズに合わせて具体化しました。start-restart-all.cmdは、バックグラウンドで（デーモンウィンドウを開かずに）起動/再起動し、NGINXディレクトリに配置したkill-all.cmdを停止します。説明から取られたRun Hidden Consoleユーティリティを使用。

start-restart-all.cmd：

rem @echo off
set PHP_FCGI_MAX_REQUESTS=0
@echo Shutting down servers...
taskkill /f /IM nginx.exe
taskkill /f /IM php-cgi.exe
@timeout 1
@echo Starting servers...
@rem start /b /D "C:\php-7.4.5-nts-Win32-vc15-x64" php-cgi.exe -b 127.0.0.1:9000
RunHiddenConsole.exe "C:\php-7.4.5-nts-Win32-vc15-x64\php-cgi.exe" -b 127.0.0.1:9000
start /b /D "c:\nginx-1.17.10\" nginx.exe
@timeout 3

kill-all.cmd：

taskkill /f /IM nginx.exe
taskkill /f /IM php-cgi.exe
pause

環境変数PHP_FCGI_MAX_REQUESTSに注目したいと思います。デフォルトでは500です。500リクエスト後、PHP FastCGIデーモンは作業を完了するため、デバッグのためにこのカウンターを無効にしました。ドキュメントからのリフレクションの引用は次のとおりです。

このPHPの動作は、PHP_FCGI_MAX_REQUESTSを0に設定することで無効にできますが、PHPアプリケーションがリソースをリークすると問題になる可能性があります。あるいは、PHP_FCGI_MAX_REQUESTSをデフォルトよりはるかに高い値に設定して、この問題の頻度を減らすことができます。

付録。NGINXをHTTP基本アクセス認証でリバースプロキシとして構成し、WebServices LabVIEWと連携する


少し前に、WebServices LabVIEWを少し試してみました（LabVIEWのかなり古いバージョンで実を言うため）。次に、ページ（WebServicesリソース）に単純なアクセス制御がないことを発見しました。アプリケーションサーバーでユーザーを構成し、「デッド」なMicrosoft Silverlightを使用することが提案されました。また、HTTP Basicアクセス認証のパスワード検証など、いくつかの簡単なオプションが必要でした。

NGINXを使用して、auth_basicを有効にしたリバースWebプロキシとして構成しました。以下の設定を使用して、アドレスhttp：// server_name：5500にパスワードを入力した後にアクセスすると、ユーザーはhttp://127.0.0.1:8001/webservice1/で実行されているWebServiceアプリケーションにアクセスできます。
webservice1アプリケーションのすべてのリソースが保護されます。

nginx.conf：

server {
    listen       5500;
    server_name  localhost;
    location / {
        auth_basic "Unauthorized";
        auth_basic_user_file htpasswd;
        root html;
        #autoindex on;
        #index index.html index.htm;
        proxy_pass http://127.0.0.1:8001/webservice1/;
    }
    error_page   500 502 503 504  /50x.html;
    location = /50x.html {
        root   html;
    }
}

htpasswdファイルとユーザーパスワード：

admin:{PLAIN}1

このアイデアをさらに発展させると、HTTPSを介したNGINXプロキシへのアクセスを有効にして、HTTPをNGINXからLabVIEWに残すことができます。
