C ++ 20の準備ができました！ISOからの印を付けることは残っていますが、これは問題がないはずの純粋に正式なステップです。

この素晴らしいイベントでおめでとうございます！概念、コルーチン、モジュール、範囲、std ::形式、constexpr newおよびconstexprアルゴリズム+ベクトル+文字列、datetime、jthread、span、bit_cast、およびその他の多くの小規模および大規模なイノベーション。

彼らが最後の瞬間に何とか追加して修正したもの、彼らが破ることを提案したもの、そして誰もがC ++ 23で見たいもの-カットの下のすべてについて。

CからC ++ 20へのトリック


最近、C ++の未定義の動作（UB）で初心者の開発者を怖がらせる伝統がありました。それを変更する時が来ました！

ここでは、たとえば、そのようなコードはCに対して完全に有効です。

struct X { int a, b; };

X *make_x() {
  X *p = (X*)malloc(sizeof(struct X));
  p->a = 1;
  p->b = 2;
  return p;
}

しかし、C ++には大きな問題がありました。Cはバイトで動作し、C ++はオブジェクトで動作します。しかし、オブジェクトには寿命があり、C ++ 20より前のバージョンでは、オブジェクトの寿命の始まりはnewの呼び出しから考慮されていました。

委員会は、バイトと単純な構造を使用した低レベルの作業に真剣に取り組んでいます。彼らは、特定の関数セット（memcpy、memmove、malloc、aligned_alloc、calloc、realloc、bit_cast）がオブジェクトの存続期間を開始すると言う改善を採用しました。現在、低レベルのCトリックのほとんどは、C ++での動作が保証されています。

C ++ 20ではboolの信頼性が向上しました


エラーが何であるかを推測してください：

template <typename T, size_t N>
auto count_unique(const std::array<T, N>& v) {
    return std::unordered_set<T>{v.begin(), v.end()}.size();
}

回答
T bool v.begin() v.end() (, libc++ libstdc++) — count_unique 1.

- , std::unordered_set<bool>{v.begin(), v.end()} bool std::unordered_set<bool>{true, true}.

boolの変換は、ナローイング変換と見なされます。これにより、多くのコードベースで問題を見つけることができました（P1957の文自体の例が増える）。

C ++ 20の概念を高速化


最近まで、次のような概念を書くことができます。

template <class T>
concept Reservable = requires(T v) {
    v.reserve(int{});
};

そして、それは異なる結果を返すことに驚かれます：

struct Test;

static_assert(!Reservable<Test>);

struct Test {
    void reserve(int);
};

static_assert(Reservable<Test>);

前回、国からコメントを送信しました。「概念に不完全なタイプを使用しないでください。そうしないと、複数のODR違反が発生するためです。」私たちのコメントは拒否されましたが、提案P2104により、部分的に望ましい結果が得られました。

おまけとして、コンパイラーは型に概念を適用した結果をキャッシュする資格があるため、コンパイルが高速になります。

そして、C ++ 23またはC ++ 26のすべてを壊しましょう。


長期にわたる議論により、アプリケーションバイナリインターフェース（ABI、APIと混同しないでください）が提案されました。興味深い質問が提起されました：

1. C ++ 23でABIを完全に変更し、パフォーマンスを5〜10％向上させることができます。


さらに、古いC ++ライブラリはすべて再構築する必要があり、新しいABIのライブラリにリンクすることはできません。以前のバージョンのC ++でビルドされたライブラリをC ++ 23プロジェクトで使用することはできません。

もちろん、もちろん、誰も再構築できない古い商用ソフトウェアは常に存在しますが、それはその標準ライブラリを引きずります（そうです、ビデオゲーム、私はあなたについて話しているのです！）。

ABIは、わずかな投票の余地があるため、C ++ 23で中断しないことを決定しました。

2.ユーザーに、ABIを壊したり変更したりしないことを保証するようにしましょう。


そして、彼らは保証を与えないことに決めました。ベンダーごとにプラットフォームの計画が異なり、ユーザーに害を及ぼすことなく、ABIを破ることができる場合もあります。

そして、どこにも追加しませんか？


歴史的に、前提条件を備えた標準関数では、例外がスローされない場合でも、noexceptとしてマークされていません。ここでは、たとえば、operator-> have std :: optional：

constexpr const T* operator->() const;
constexpr T* operator->();
    Requires: *this contains a value.
    Returns: val.
    Throws: Nothing.

彼は何もスローしませんが、noexceptの代わりに「スロー：なし」と言います。これは、「*これには値が含まれている」という前提条件があるためです。

noexceptを持つユーザーはより明確になります。P1656の良いアイデア！



番号！

全体のサブグループSG21：契約があります。契約には、契約（事前条件と事後条件）をチェックするための共通のメカニズムがあります。コントラクトハンドラーは例外をスローでき、noexcept関数から例外がスローされると、std ::終了し、アプリケーションがクラッシュします。契約の例外がnoexcept関数から飛び出すことができる特別な松葉杖を挿入した場合...とにかくすべてが故障し、型の特性はnoexceptの存在によって導かれ、それらはあなたに嘘をつき始め、noexceptでマークされた関数は前提条件で例外をスローします。

しかし、これは最大の問題ではありません。標準ライブラリのフォークがあり、すでに一部のケースでは、事前条件チェックが明示的に挿入されています。たとえば、可用性を最大化する必要がある重要なプロジェクトがありますか。同様のフォークを使用していて、誰かが突然空のベクターに対してstd :: vector :: back（）を呼び出した場合、例外がスローされ、コードの上位で処理され、フォールバックが使用され始めます。P1656からの編集により、そのようなライブラリは標準と見なされなくなりました。

そしてこれがすべての問題ではありません！..標準ライブラリを除く追加のnoexがバイナリファイルのサイズの減少またはより多くのパフォーマンスの形でプラスの効果をもたらすだけでなく、変更は少なくとも2つの会社のコードを壊すだけでなく、それはいずれかの方法を破壊します契約の使用...したがって、この提案は2つのサブグループで承認されました。

いつものように、私たちはさまざまなサブグループで作業し、実装経験を共有し、提案を発表しましたが、その著者は参加できませんでした。

私たちが提示して幸運だった優れたアイデアの1つは、Anton Zhilin P2025名前付き戻りオブジェクトのコピー省略を保証するというアイデアでした。その実装により、コピーおよび移動コンストラクターなしでオブジェクトのファクトリー関数を作成できます。実際、これは破壊的な動きであり、90年代半ば以降、標準にひそかに存在しており、個々の言語規則によって明確に禁止されていました。

著者自身によるアイデアの優れた精巧さのおかげで、EWG-IおよびEWGインスタンスを介してアイデアをうまく​​ドラッグできました。 CWGの段階は残り、2、3回のミーティングの後、標準で必要な単語、およびコンパイラーでの最初の実装を確認する機会がすべてあります。

このアイデアに加えて、P1990R0のアイデアをドラッグしました： LEWG-Iを介してstd :: initializer_listに演算子[]を追加し、P1944R0：constexpr <cstring>および<cwchar>に関する有用なフィードバックを得ました。 Daniil Goncharovの両方のアイデアには、C ++ 23に入る可能性がすべてあります。

std ::ハッシュの分野では、予期しない失敗が待ち受けていました。議論p1406r1：詳細はstd ::追加ハッシュ専門は突然縮退境界例の議論と遠くのC ++ 2 *の可能性になりました。その結果、委員会は何も変更しないことを決定しました。

SG6と番号で一緒に成長しませんでした。 SG6のメインディスカッションはABIのディスカッションと交差したため、SG6のクォーラムは蓄積されませんでした。このp1889のため：C ++ Numerics Work In Progress、P2010：iostream演算子をP1889から削除し、P1890：C ++ Numerics Work In Progress問題は議論されませんでした。

C ++ 23プラン


C ++ 20の開発が始まって以来、委員会は計画どおりに行動し始めました。つまり、次の標準のためのいくつかの主要な興味深いアイデアを特定し、その後の会議では、他のトピックに関する提案を検討しないようにします。

C ++ 23の場合、そのような計画はプラハで承認されました。C ++ 23の主な優先事項：

  1. 標準ライブラリでのCorutinサポート
  2. 標準ライブラリをモジュールに変換
  3. 執行者
  4. ネットワーキング

最初の段落では、誰もがCppCoroライブラリーに案内されます。したがって、すでにC ++ 20コルーチンを使用したい場合は、このライブラリを使用して開始する必要があります。

2つ目のポイントであるモジュールを使用すると、座って実行するだけで、特別な問題は発生しません。

しかし、死刑執行人は問題です。彼らのデザインは明白ではなく、すべてのユーザーケースがカバーされているわけではなく、現在の形では誰も使用していませんでしたし、デザインはまだ承認されていません。

委員会はまた、次の分野での提案に優先順位を付けることにも同意しました。
