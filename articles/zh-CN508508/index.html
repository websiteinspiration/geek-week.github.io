依赖注入是一种可独立使用的方法。另一方面，这种方法既可以用于接口也可以用于依赖项注入/管理倒置（DI / IoC）的容器。在这种情况下应用依赖注入时，您可能会遇到一些混乱，这是我最初遇到的。



在我的整个职业生涯中（我专门从事Net / C＃开发），我曾经以最纯粹的形式使用依赖注入。同时，我实施DI时完全不求助于容器或控制反转。最近，当我接到一项必不可少的任务，即使用容器时，一切都发生了变化。然后我强烈怀疑我以前知道的一切。



以这种风格工作了几周后，我意识到容器和接口不会使依赖关系的实现复杂化，但是相反，扩展了此范式的功能。


（这里需要注意的重要一点是：接口和容器仅在依赖注入的情况下使用。依赖注入可以在没有接口/容器的情况下实现，但是从本质上讲，接口或容器的唯一目的是促进依赖的实现）。



本文将展示如何在有接口和无接口的情况下进行依赖项注入。我希望阅读后能清楚地理解依赖注入的原理，并能够在实现依赖时何时何地使用接口和容器做出明智的决定。

using System;
using System.Linq;

namespace OfferZenDiTutorial
{
    class Program
    {
        static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            var calc = new Calculator();
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }

        private static float GetNumber(string message)
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write(message);
                var input = Console.ReadLine();
                isValid = float.TryParse(input, out var number);
                if (isValid)
                    return number;

                Console.WriteLine("Please enter a valid number. Press ^C to quit.");
            }

            return -1;
        }

        private static char GetOperator()
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write("Please type the operator (/*+-) > ");
                var input = Console.ReadKey();
                Console.WriteLine();
                var operation = input.KeyChar;
                if ("/*+-".Contains(operation))
                {
                    isValid = true;
                    return operation;
                }

                Console.WriteLine("Please enter a valid operator (/, *, +, or -). " +
                                  "Press ^C to quit.");
            }

            return ' ';
        }

        private static float GetResult(Calculator calc, float number1, float number2, 
            char operation)
        {
            switch (operation)
            {
                case '/': return calc.Divide(number1, number2);
                case '*': return calc.Multiply(number1, number2);
                case '+': return calc.Add(number1, number2);
                case '-': return calc.Subtract(number1, number2);
                default:
                    //    ,        
                    throw new InvalidOperationException("Invalid operation passed: " + 
                                                        operation);
            }
        }
    }
}

关于接口的小免责声明



如您所见，接口是一个非常强大的工具，但是它们为应用程序带来了更高的抽象级别，因此也带来了额外的复杂性。容器可以帮助您克服复杂性，但是，正如下一节将显示的那样，您必须注册所有接口，并且在使用特定类型时，这不是必需的。



所谓的“借助抽象进行混淆”实际上就是为了使所有这些不同的层次归结为项目的复杂化。如果需要，可以在没有所有这些接口的情况下进行操作，除非出于特殊原因需要它们。通常，不应创建仅具有一个实现的接口。



依赖注入容器



本文中得出的示例非常简单：我们正在处理一个只有一个依赖项的类。现在假设我们有很多依赖项，并且每个依赖项都与其他依赖项相关联。即使在进行中度复杂的项目时，您也可能不得不处理此类情况，不仅要记住您需要创建所有这些类，还要记住哪一类取决于哪一类-特别是如果您决定使用接口。



熟悉依赖项注入容器。它简化了您的生活，但是这种容器的操作原理似乎非常混乱，尤其是当您刚开始掌握它时。乍一看，这个机会可以带来一些魔力。
