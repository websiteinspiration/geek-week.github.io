GRASP-Vorlagen: Ersteller

Hallo Habr. Ich heiße Vladislav Rodin. Derzeit bin ich Leiter des High Load Architect-Kurses bei OTUS und unterrichte auch Kurse zur Softwarearchitektur.

Speziell für den Start eines neuen Sets für den Kurs "Architektur- und Designmuster" habe ich ein kleines Material geschrieben, das ich gerne mit Ihnen teile.


Einführung


GRASP-Muster werden in Craig Larmans Buch Applying UML and Patterns, 3. Auflage, beschrieben und sind eine Verallgemeinerung von GoF-Mustern sowie eine direkte Folge der OOP-Prinzipien. Sie ergänzen den fehlenden Schritt in der logischen Leiter, mit dem Sie GoF-Muster aus den Prinzipien von OOP abrufen können. GRASP-Vorlagen sind eher keine Entwurfsmuster (wie GoFs), sondern grundlegende Prinzipien für die Verteilung der Verantwortung zwischen Klassen. Die Praxis zeigt, dass sie nicht sehr beliebt sind. Die Analyse entworfener Klassen unter Verwendung des vollständigen Satzes von GRASP-Mustern ist jedoch eine Voraussetzung für das Schreiben von gutem Code.

Eine vollständige Liste der GRASP-Vorlagen besteht aus 9 Elementen:

  • Informationsexperte
  • Schöpfer
  • Regler
  • Niedrige Kopplung
  • Hoher Zusammenhalt
  • Polymorphismus
  • Reine Herstellung
  • Indirektion
  • Geschützte Variationen

Das letzte Mal haben wir das Prinzip des Informationsexperten besprochen . Jetzt schlage ich vor, einen ähnlichen Schöpfer in Betracht zu ziehen.

Schöpfer


Wortlaut


Dieses Muster löst das gleiche typische Problem wie sein Vorgänger: Die Klasse, die sie benötigt, muss Instanzen der Klasse erstellen .

Beispiel für Verstöße


Betrachten Sie alle das gleiche Problem mit Bestellungen und Waren. Angenommen, der von uns geschriebene Code entspricht dem Informationsexperten:

@Setter
@Getter
@AllArgsConstructor
public class Order {
    private List<OrderItem> orderItems;
    private String destinationAddress;
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }
}

@Setter
@Getter
@AllArgsConstructor
public class OrderItem {
    private Good good;
    private int amount;

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

Trotz der scheinbar Trivialität des Prinzips, das wir untersuchen, finden Sie in einigen Client-Codes Folgendes:

public class Client {
    public void doSmth() {
        Good good = new Good("name", 2);
        OrderItem orderItem = new OrderItem(good, amount);
        List<OrderItem> orderItems = new ArrayList<>();
        orderItems.add(orderItem);
        Order order = new Order(orderItems, "abc");
        // client code 
    }
}

Wenn Sie ein UML-Klassendiagramm erstellen, können Sie feststellen, dass die Client-Klasse jetzt von der Order-Klasse und all ihren Interna abhängt: OrderItem und Good. Daher können wir die Client-Klasse nicht ohne die oben genannten Klassen wiederverwenden, die der Client nicht benötigt. Wir haben das Ergebnis aller Bemühungen zur Einhaltung des Informationsexperten tatsächlich für ungültig erklärt, da die Client-Klasse alle Objekte erstellt hat.

In älteren Projekten können Sie häufig sehen, wie eine Klasse ein Objekt einer anderen erstellt und es als Parameter in der Methode über 5-6 Klassen weiterleitet, in denen dieses Objekt nicht verwendet wird. Dies ist nichts weiter als ein paar Abhängigkeiten von Grund auf neu hinzuzufügen.

Anwendungsbeispiel


Passen wir die Verteilung der Verantwortung zwischen den Klassen so an, dass die Verteilung nicht nur den Informationsexperten, sondern auch den Schöpfer zufriedenstellt:

@Setter
@Getter
public class Order {
    private List<OrderItem> orderItems = new ArrayList<>();
    private String destinationAddress;
    
    public Order(String destinationAddress) {
        this.destinationAddress = destinationAddress;
    }
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }

    public void addOrderItem(int amount, String name, int price) {
       orderItems.add(new OrderItem(amount, name, price));
   }
}

@Setter
@Getter
public class OrderItem {
    private Good good;
    private int amount;

    public OrderItem(int amount, String name, int price) {
        this.amount = amount;
        this.good = new Good(name, price);
    }

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

Jetzt ist die Anzahl der Abhängigkeiten zwischen Klassen minimal. Der Client-Code ist etwas vereinfacht und sieht möglicherweise folgendermaßen aus:

public class Client {
    public void doSmth() {
        Order order = new Order("address");
        order.addOrderItem(amount, name, price);
        // client code 
    }
}

Fazit


Creator, den Entwickler häufig vergessen, kann als Sonderfall von Information Expert betrachtet werden, da ein Konstruktoraufruf mit einem Methodenaufruf identisch ist. Die Einhaltung zusammen mit Information Expert ermöglicht es, eine minimale Anzahl von Verbindungen zwischen Klassen und eine größere Wiederverwendung zu erreichen.
