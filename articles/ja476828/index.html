操作「移行」：データラインクラウドへの移行方法

約7年前、最初のプロジェクトは単純かつ気取らずにクラウドに移行しました。仮想マシンのイメージがFTPサーバーにアップロードされたか、ハードドライブに配置されました。次に、特別なインポートサーバーを介して、VMがクラウドにアップロードされました。

クライアントが1日か2日仮想マシンをオフにしても問題ない場合（または他のオプションではない場合）、それは可能です。ただし、ダウンタイムが最大で1時間の場合、この方法は機能しません。今日は、最小限のダウンタイムでクラウドに移行するのに役立つツールと、移行プロセスがどのように機能するかについて説明します。



Veeam Backup and Replicationを使用した移行


Veeam Backup and Replicationは、バックアップとレプリカを作成するためのツールとして誰もが知っています。サイト間の移行や、プライベート仮想化を使用したクライアントをクラウドに転送するために使用します。クライアント仮想マシンはvCenterに複製され、その後エンジニアがそれらをvCloud Directorに追加します。

プライマリレプリケーションは、有効な仮想マシンで実行されます。合意された時間に、クライアント側のマシンはオフになります。レプリケーションが再び開始され、最初のレプリケーション以降に発生した変更が転送されます。その後、仮想マシンはすでにクラウドで起動しています。



通常、クライアントのインフラストラクチャでマシンがオフにされた瞬間から、オンにされる瞬間まで、30分以内、15〜20分でクラウドを通過します。

同時に、元の仮想マシンはクライアントのサイトに残ります。突然問題が発生した場合は、いつでもロールバックして電源を入れることができます。クライアントにとって、この方法は、Veeamを必要としないという点でも便利です。

ケース1
クライアントには、VMwareに基づく独自の仮想インフラストラクチャがあり、30 Tbの容量を持つ40のVMがありました。クラスターがデプロイされた機器はすでに古くなっており、クライアントは新しい機器を購入することなく、パブリッククラウドに移行することにしました。重要なシステムのダウンタイムの要件は、わずか1時間でした。ツールとしてVeeam Replicationを選択しました。さらに、クライアントのインターネットプロバイダーがデータセンターに存在していて、優れたチャネルを編成することができました。移行には約1か月かかりましたが、切り替えは簡単でしたが、仮想マシンの1つのグループに最大30分かかりました。

Veeam Cloud Connectで移行する


Veeam Cloud Connectは、仮想マシンのレプリケーションを構成し、サービスプロバイダーのクラウドでレプリカを実行するのに役立つツールです。2019年の更新後、仮想マシンをvCloud Directorに直接複製できるようになりました。唯一の条件は、クライアント側で、Veeam Backup and Replicationを少なくともバージョン9にデプロイする必要があることです。つまり、詳細なバージョンはこちらですが、プロセス全体は次のようになります。

vCloud Directorは、必要なリソースとネットワークを備えた組織を作成します。 Veeam Cloud Connectでは、アカウントを作成し、クライアントがVeeam B＆Rからアカウントに接続し、DataLineプロバイダーと組織を選択し、レプリケーション用のタスクを設定します。そのような移行中、15〜20分以内で簡単になるという事実に加えて、クライアントはプロバイダーのテクニカルサポートに依存せず、プロセス全体を個別に管理します。レプリケーションタスクを作成し、レプリケーション自体を実行し、マシンをオフにして、新しいサイトでの開始を開始します。

VMware vCloud可用性による移行


ヴイエムウェアは今年3月にvCloud Availability 3.0をリリースしました。これにより、仮想マシンを異なるクラウド間（vCloud Director-vCloud Director）からプライベートクライアント仮想化スタンドからクラウド（vCenter-vCloud Director）に移行できます。主な利便性は、vCloud Directorインターフェイスとの統合です。これにより、レプリケーション管理が大幅に簡素化され、切り替え時のダウンタイムが最小限に抑えられます。

このツールを使用して、クライアントの1つをモスクワのクラウドからサンクトペテルブルクのクラウドに移行しました。合計容量が14 TBの18台の仮想マシンを転送する必要がありました。クライアントのために、組織がサンクトペテルブルククラウドに作成され、必要なネットワークが編成されました。次に、クライアントはvCloud DirectorインターフェイスからvCloud可用性設定に切り替え、レプリケーションタスクを作成し、都合のよいときにサンクトペテルブルクサイトに切り替えました。切り替え中のダウンタイムは12分でした。


サンクトペテルブルクとモスクワのDataLineクラウド間の移行スキーム。

VCloud可用性には、VMをクライアントサイトからクラウドに移行するためのメカニズムがあります。これを行うには、vCenterクライアントに特別なvCloud Availabilityアプリケーションをデプロイします。簡単なセットアップの後、クラウドに接続され、移行タスクが構成されます。また、クライアントは独立してプロセス全体を管理し、移行時間を最小限に抑えます。

移行の準備


ツールを選択して実際に移行を進めるには、次の点を決定する必要があり

ます。プライベートソリューションから移行する場合は、ツールを自由に選択できます。プロバイダーの外に出ると、さらに難しくなります。ほとんどの場合、セキュリティ上の理由により、2つのプロバイダーのインフラストラクチャをリンクし、VMをドラッグするだけでは失敗します。時には、クライアントが拒否しようとしているプロバイダーが完全に有害になり始め、時間がかかります。 VMをディスクにアップロードしてFTPを実行するか、アプリケーションレベルで移行するという、昔ながらの方法でプロバイダーを残すことができます。後者の名前は任意で、次のようになります。

事例3
ヨーロッパのプロバイダーからクライアントのSAPシステムを移行する必要がありました：54 Tbのボリュームを持つ34 VM。リソースは、クラウド内のクライアントに割り当てられました。私たちとヨーロッパのプロバイダーのインフラストラクチャの間で、ネットワーク接続が編成されました。アプリケーションサーバーが再展開され、必要な構成がまとめられました。大規模なデータベースは、バックアップをクラウドにアップロードすることで移行されました。次に、サイト上のデータベースとソースサイト間のレプリケーションを構成しました。合意された時間に、クラウド内のデータベースに切り替えました。

データの量とインターネットチャネル。通常、クライアントに、メモリ、CPU、ディスクのパラメータを持つシステムでのアンロードを提供するよう依頼します。チャネルが仮想マシンのレプリカまたはバックアップを直接送信するのに十分かどうかを推定します。

有効な単純な。さまざまなシステム、つまり仮想マシンについては、ビジネスにとっての重要度によって異なります。通常、クライアントには移行中のダウンタイムに関する既成の要件があり、これに基づいて適切なツールと移行計画を選択します。クライアントのエンドユーザーがわずかなダウンタイムでも気付かないように、最終的な切り替えを夜間または週末に計画しています。
