クレイジーな自家製エンジニア、Chris Fenton



Meet ZedRipper のサイトからの記事。これは、Z80プロセッサーをベースに、83 MHzで実行される16コアの獣であり、非現実的であるほど移植性があります。これは、楽しみのためにコンピュータを組み立てようとする私の最近の試みであり、一度にいくつかの願望を満たしました。
  • 最後に、私と一緒に寝転んでいた巨大なFPGAを使用しました。
  • コンピューターを作成する別の歴史を演じ、鉄の観点からマルチタスクの問題に取り組みます。
  • 電車の途中で面白いショートプログラムを書けるコンピューターを組み立てます。
  • コンピュータアーキテクチャを使用した比較的簡単な実験を実行できるプラットフォームを構築します。

それで、この獣は何ですか？




ZedRipperは、CP / M 2.2で最もクールなコンピューターを構築しようとした結果です。

  • 83.33 MHzの周波数で動作する16個のZ80プロセッサ。
  • 各Z80に64 KBの専用メモリ。
  • 16出力のANSI互換ターミナルアクセラレータ。
  • すべてのプロセッサとデバイスは、83 MHzで動作する完全同期単方向リングネットワークによって接続されます。
  • SDカード上の128 MBドライブ（CP / Mでは16 x 8 MBドライブ）。
  • CP / M 2.2で起動し、CP / NETファイルサーバー（コンピューター自体のTurbo Pascal 3で作成された！）を実行する「サーバー」カーネル。SDカードへの共有アクセスを提供します。
  • ROMからCP / NOSを実行する15個の「クライアント」コア。各クライアントは共通のリポジトリにアクセスでき、他のカーネルとリソースを奪い合うことなく、誰でも任意のCP / M 2.2プログラムを実行できます。

別の道



Turbo Pascalエディターから私をそらすのはチェスとプラネットフォールですか？ゲームをKayproに移植する

冒険の後40年前にこのプリミティブなオペレーティングシステムを驚くほど温かく感じ、開発することを決めました。ストーリーが逆方向になり、PCがすぐに多くのCPUを備えた開発パスに沿った場合はどうなりますか？ 1980年代でさえ、プロセッサ自体（およびすぐにRAM）は比較的安価でしたが、PCマルチタスクは、1つの大きなリソース（RAMまたはCPU）が競合するプログラム間で分割されたときの時間のカットのみに基づいていました。 Ironはこれに対処できず（DOSなどのOSでプログラムを正常に動作させることは非常に困難でした）、メモリが4 MBを超える386とコンピュータの時代に移行するまでは。

コンピュータで私の歴史的な趣味を研究する過程で、私は非常に興味深い何かに出会いました：OSの開発の初期段階CP / Mは、CP / NETと呼ばれる「ネットワーク」バージョンをサポートしていました。ほとんどの人はまだ彼女のアイデアを今日でも知っています-大容量のドライブとプリンターを備えた1つまたは2つの「実際の」マシンをオフィスに置くということです。各ユーザーは、大きなディスクとプリンターにアクセスできるCP / Mを実行している自分のマシンがあるかのように作業します。

前述したように、CPUとRAM（通常、Z80には64 KBのDRAMが搭載されていました）はそれほど高価ではありませんでしたが、便利なコンピューター（ドライブ、プリンター、モニターなど）を作成するために必要なすべての外部プリブラッドは、合計コストに加算されました。当時、コンピュータに複数のCPU / RAMを追加することは、1人のユーザーに複数のCPUとRAMを提供するためのやや退廃的なアプローチのようでした。 CP / Mでさえ、MP / M OSの期間を分割する方法を採用しました。

私はExidyがこれに最も近いことがわかりました-1981年に彼らはマルチNET 80マシンをリリースしました。これにより、それぞれがZ80とRAMを備えた最大16枚のカードを追加できました。ただし、最大16人の個人ユーザー向けに設計されており、16のプログラムを同時に起動した1人のユーザー向けではありません。


とても近い…

40年早送り-トランジスタは実際に価格が下がっています。研究所が閉鎖された後、私はいくつかの巨大なFPGA（Stratix IV 530GX）を継承しましたが、そのうちの1つを使用するのはとても興味深いと思いました。ある時点で、Grant Searle Multi-Compによる非常に興味深いプロジェクトに出会いました。CP/ Mと1つのCPUを備えた作業用マシンを組み立てるのは非常に簡単でした。しかし、私はもっと必要でした。私は、CP / Mで実際のマルチタスク処理を備えたマルチコアマシンを作成できるかどうかを確認することにしました。

ソフトウェアを構成して起動します


このプロジェクトでは、主にハードウェアに集中し、アセンブラーで1行のコードも記述しませんでした。CPU 0は、Grantから取得したROMから直接ロードされ、残りのノードは、Atariシミュレーターから取得した4KB CP / NOS ROMからロードされます。

両方のROMは、標準インターフェイスを介してシリアルターミナルへの接続を待機しますが、CP / NOSクライアントは、サーバーに接続された別のシリアルポートを期待します。このような大規模なFPGAでは、独自のロジックを設計するのは簡単です。独自のアドレスデコードロジックを開発しました。これにより、必要に応じて、各CPUのZリングがアドレスマッピングスキームに表示されます。

内面




ZedRipperの心臓部は、これらの巨大なStratix IV 530GX FPGAの1つです。HSMCカードは、表示、キーボードコントローラーからのデータの受信、SDカードへの接続に使用されます。ファームウェアのダウンロードにはイーサネットが使用されるため、ケースの側面にSDカードアダプターと外部シリアルポート用のスロット（まだ使用されていません）とともに、このようなポートがあります。

キーボードとコントローラー



キーボードとフォアグラウンドの穴で、位置決めデバイスが後でインストールされます。

コンパクトなPS / 2キーボード（ラップトップを使用した古いプロジェクトの1つからのもの）があり、FPGAの2.5 V I / Oに接続したいと思っていました。私は簡単な方法で、Teensy 2.0マイクロコントローラーをバンドルに追加することにしました。


キーボード下部のホットグルーコントローラー

これにより、PS / 2をASCIIに変換できるだけでなく、追加のキー（F1〜F12）の一部をターミナルコマンドの「マジック」シーケンスに簡単にマーク付けできるため、利便性が向上しました。次に、コントローラーは9600ボーでUARTにZ80バイトを提供します（FPGAの場合、5 Vから2.5 Vに変化する単純な分圧器を使用）。このプロジェクトは、私のワークショップにあるさまざまなゴミから組み立てられたものであることを考えると、実際にうまく機能している便利なソリューションでした。

表示



読み込み画面、左上隅にサーバーが実行されており、異なるコアには3つの異なるユーザープログラムがあります。

表示機能：1280×800 10.1″、VGAを理解します。FPGAは抵抗器の単純なネットワークを使用して、最大64色（R2G2B2）を提供します。ディスプレイには83.33 MHz（1280×800 @ 60Hz）のタイマーが必要なので、簡単にするために回路全体がこの周波数で動作します。

GrantのプロジェクトMulticompには、単純なANSI互換端末用のVHDLコードが含まれていました。私はそのロジックをVerilogで書き直し、1つのZリングノードを介して接続された16個の独立した端子をサポートするビデオコントローラーを開発しました。 1280×800のディスプレイは、160x50の文字ディスプレイ（8x16フォントを使用）と見なされ、各端末は画面上の任意の場所に移動できる80x25の「スプライト」として機能します（端末のレンダリングシーケンスを調整する優先順位のリストを使用）。各端末は他の端末とは独立して動作するため、独自のステートマシンがあり、2 KBの文字用RAMと2 KBの「属性」（色情報の格納用）を備えています。各文字は4ビットの背景と文字色をサポートしています。すべての端末は同じインデント文字を使用する必要があり、8x16の「セル」に含めることができるのは1文字のみです。すべての端末は、フォントを含む同じ2 KB ROMを使用できます。一般に、表示ロジックは約66 KBのブロックRAMを使用します。

一般的に、CP / M端末用の非常にシンプルなウィンドウマネージャーを取得します。これは、ハードウェアが原因でほぼ完全に機能します。これは最も豊かな研究分野の1つです。これまでのところ、サーバーのCPUだけが端末を再配置できますが、マウスなどの位置決めデバイスを追加して、ウィンドウをドラッグしてウィンドウの優先順位を変更できるようにする遠大な計画があります。

ターミナルコントローラーはZリングノードの1つにすぎないため（このインターフェイスをZ80のいずれかにリダイレクトするのは非常に簡単です）、将来の計画では、「フルスクリーン」ターミナル160x50（おそらく「バックグラウンド」として）と実際のディスプレイ1280x800x64を追加することが可能ですボード上の高速外部SRAMの色。

Zリング


Z80をまとめる方法は？私の仕事では、ネットワークを開発することは難しいという1つのことをしっかりと学びました。このネットワークの全体的な目的は次のとおりです。

  • シンプルな実装。
  • シンプルなインターフェース
  • 任意の拡張性。
  • 適切なパフォーマンス。

すでに述べたように、私のZ80はシリアルポートに接続することを期待しているので、インターフェースは非常にシンプルでした-シリアルポートに変装する必要がありました！基本的に、Zリングは、フローを制御するために「クレジット」を使用する同期単方向リングネットワークです。各ノードには、ネットワーク内の他のノードごとに1バイトの着信バッファーがあります。リセット後、各ノードには残りのネットワークノードごとに1つの「クレジット」があります。スキームはパラメーター化されているため、非常に少量のロジックを追加するだけで数百のノードに簡単に拡張できますが、現在のZ-Ringは最大32のノードをサポートしています（したがって、各ノードには32バイトのバッファーが必要です）。

「バス」自体は、有効なビット、「ソース」ID、「ターゲット」ID、および1バイトのペイロード（19ビット）で構成されます。 TTLロジックを使用して実装するのは非常に簡単だと思います（1981年に人が失敗してFPGAが見つからなかった場合）。各「ノード」には、バストリガー用の2つのパイプライン（0ステージと1ステージ）があり、メッセージを入力すると、0番目のステージが空になるまで待機してから、1番目のステージにマージします。メッセージはソースノードに入力され、ターゲットに到達するまでリング内を移動します。その後、メッセージは適切なバッファで検出され、データ準備フラグを更新します。受信ノードはバッファを読み取ると、元のメッセージに再び入り、再びソースに到達するまでリング内を移動し続け、「クレジット」を返します。存在しない住所に荷物を送ると、全周後にローンが自動的に返却されます。

リングの各ストップは2つのコンベヤステージで構成され、バックプレッシャがないため、各メッセージの配信サイクルは2 *（ノード数）以下です。現在の実装には17ノード（16 CPU +ディスプレイ/キーボードコントローラー）があり、12 nsのタイマーで動作するため、メッセージの配信とローンの返却には約400 nsかかります。ディスプレイコントローラーは到着率でトラフィックを送信できるため、各CPUはその端末に2〜2.5 Mb / sの帯域幅を持っています（バスは16個のCPUすべてを提供するのに十分です）。

現在の構成では、すべてが正常に機能しますが、いくつかの明らかな改善を行うことができます。

  • ノードのスループットを向上させる受信バッファーを深めるために-FPGAには、1 KBのRAMの多くの空きブロックがあり、32クレジットで32ノードをサポートするため、各CPUは理論的にバスを飽和させることができます。
  • アドレスモードのサポートを追加します。16ビット（またはそれ以上）のアドレスを追加すると、メモリ（DMA）に直接アクセスできます（各ノードにDMAを追加するのは簡単です）。FPGAには膨大な量の追加ハードウェア（数メガバイトのスタティックRAMと約1ギガバイトのDDR3）があります。
  • ノード間にフロー制御（およびバッファリング）を追加します。

しかし、これはすべてより良い時まで待つことができます。

栄養！


FPGAを搭載したデバッグボードには12〜20 Vの入力電力が必要で、ディスプレイには12 Vが必要です。キーボードとコントローラーには5 Vが必要です。FPGAには3.3、5、12 Vのコントローラーがあり、接続が非常に簡単です。 FPGAは、14.4 Vの電圧で5000 mAhのリチウムポリマーバッテリーから直接電力を受け取り、他のすべてのデバイスに電力を分配します。困難な点の1つは、充電するたびにラップトップを分解したくなかったのですが、バッテリーには通常の+ /-電源コネクターと、個々のセルに接続された「バランス」コネクターがありました。私の不完全な解決策は、電源ボタンがFPGAの電源とスライドカバーで閉じられたくぼみにある充電コネクタ間のバッテリー接続を切り替えることです。あまり便利ではありませんが、蓋をスライドするだけで、そこからコネクタを引き出し、六角キーを使用せずに充電器に接続します。


充電が変に見える

バッテリーを完全にテストしていませんが、少なくとも3時間は持続します（これは私の電車の乗り物をカバーするのに十分な長さです）。ほとんどの場合、消費を最適化せずに約6時間続きます。充電と同時に使用することはできませんが、ラップトップは十分に長いバッテリ電源で動作するため、これは問題になりません。

ハウジング


標準的な「ハッカー」デザインのケースは、3 mmレーザー切断合板と3Dプリンターで印刷されたプラスチックの組み合わせです。私は画面のヒンジをスプリング式に装填したので、実際にはそれは普通のラップトップのように感じられます。1980年代風にしたかったので、画面の上隅が少しクレイのようになっていて、手首の下にフェイクレザーのスタンドが作られています。レーザーカットされた合板の端は手にとって非常に不快なので、このスタンドは驚くほど機能的でした。

速度


CP / M専用のベンチマークを1つも試したことはありません（たぶんそうしていると思いますが、特に探していませんでした）。このマシンはTurbo Pascalでプログラムを作成するために作られたので、速度についていくつかのマイクロテストを試しました。1秒あたり15〜35 Kの浮動小数点演算（TPの48ビット実数型を使用）、および1秒あたり約100万の整数演算（16ビット整数型）が判明しました。8ビットCPUと便利な十分なプログラミング環境には悪くありません。

将来のための興味深いプロジェクトは、浮動小数点演算のためのアクセラレータの開発かもしれません。
