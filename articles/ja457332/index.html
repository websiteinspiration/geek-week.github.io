CおよびC ++コードのPVS-Studioアナライザーによって生成された興味深い警告をすばやく確認するにはどうすればよいですか？


時々、PVS-Studioコードアナライザーに慣れ始めたプログラマーは、「エラーを正確に示す警告のリストはありますか？」と質問します。あるプロジェクトと別のプロジェクトで興味のない（誤った）警告が非常に重要で有用であることが判明する理由のため、そのようなリストはありません。ただし、アナライザーとの関係を最も興味深い警告で開始することはかなり可能です。このトピックを詳しく見てみましょう。

問題は、最初の起動時に、プログラマーが原則として「溺れる」という膨大な数の警告を受け取ることです。当然のことながら、彼はこの時間すべてを費やす必要があるかどうかを理解するために、最も興味深い警告について知ることを望んでいます。すばらしいです。ここに、最も興味深いポジティブを知るための3つの簡単なステップを示します。

ステップ1





一般（GA）を除くすべてのタイプのアラートを無効にします。よくある間違い：あらゆる種類の警告を有効にします。経験の浅いユーザーにとっては、それをオンにすればするほど良いと思われます。本当じゃない！64ビットチェックやMISRAルールなどの診断のセットがあり、それが何であり、どのように操作するかを明確にした場合にのみ使用する必要があります。たとえば、通常のアプリケーションプログラムのMISRA診断をオンにすると、次のような数十、数千、または数十万のメッセージが発生します。

  • V2506。ミスラ。関数には、最後に単一の出口点が必要です。
  • V2507。ミスラ。ループ\条件ステートメントの本体は中括弧で囲む必要があります。
  • V2523。ミスラ。符号なしタイプのすべての整数定数には、「u」または「U」の接尾辞が必要です。

ほとんどのMISRA警告はエラーを示すものではなく、コードのにおいがします。当然、人は質問をし始めます。これらすべての警告の中で面白いものを見つけるにはどうすればよいですか？彼は何の番号で診断すべきですか？これらは間違った質問です。 MISRAキットを無効にする必要があるだけです。これは、組み込みデバイス用の高品質コードを記述するための標準です。標準の本質：コードを非常にシンプルでわかりやすくすること。それが不適切であるところでそれを適用しようとしないでください。

注意。はい、実際のエラーを検出することを目的としたMISRA標準にはルールがあります。例：V2538-初期化されていない変数の値は使用しないでください。ただし、MISRA標準を無効にすることを恐れないでください。何も失うことはありません。真のエラーは、汎用診断（GA）でも見つかります。たとえば、V614診断のおかげで、初期化されていない変数が見つかります。

ステップ2





静的アナライザーは最初の起動時に誤検知を生成し、特定の構成が必要です。これでは何もできませんが、見た目ほど怖くはありません。簡単なクイックセットアップでも、誤った警告のほとんどを取り除くことができ、すでに十分に適切なレポートに慣れることができます。これについては、この記事で何度も書いたため、詳しくは触れません。たとえば、「EFLコアライブラリの例を使用したPVS-Studioアナライザーの特性、誤検知の10〜15％です。」

少し時間をかけて、明示的に無関係なアラートを無効にし、マクロによる誤検知に対処します。失敗したマクロが使用される場合は常に警告が表示されるため、マクロは一般に誤検知の主な原因です。マクロの警告を抑制するために、特別な種類のコメントを宣言の隣に置くことができます。コメントを書き込むための形式については、ドキュメントで詳しく説明しています。

はい、初期設定には少し時間がかかりますが、気を散らすノイズを排除することにより、レポートの認識を劇的に改善します。これには少し時間がかかります。問題や質問がある場合は、いつでもアナライザーを構成するための最善の方法を支援および提案する準備ができています。自由に書いてください 質問してください。

では、なぜまだリストがないのですか？





ただし、最も有用な警告のリストを含むアイデアは、とにかく妥当に思えるかもしれません。診断の有用性は相対的であり、プロジェクトに依存することを実際的な例で示します。

警告V550を検討してください。警告は、演算子==または！=を使用して浮動小数点数を比較しているために発生する可能性のあるエラーを示しています。

私が話をしたほとんどの開発者は、この診断が役に立たないと考えて、それを無効にします。なぜなら、彼らのプロジェクトのすべての肯定的な結果は誤りです。このため、この診断の信頼性は低く、3番目のレベルにあります。

実際、ほとんどのアプリケーションでは、float / doubleタイプは非常に単純なアルゴリズムで使用されます。また、定数との比較は、デフォルト値が設定されているかどうか、またはデフォルト値が変更されたかどうかを確認するためにのみ使用されることがよくあります。この場合、正確なチェックが非常に適切です。これを疑似コードで説明します。

float value = 1.0f;
if (IsUserInputNewValue())
  value = GetUserValue();
if (value == 1.0f)
  DefaultBehavior();
else
  Foo(value);

ここでの比較（値== 1.0f）は正確で安全です。

これは、V550が面白くない診断であることを意味しますか？番号。それはすべてプロジェクトに依存します。ユーザーの1人が書いた記事「血管内手術のトレーニングシミュレーターのプロジェクトで静的分析をどのようにテストしたか」の一部を引用します。

したがって、静的アナライザーは何に注意を向けますか？

V550奇妙な正確な比較：t！=0。定義された精度の比較を使用する方が良いでしょう：fabs（A-B）> Epsilon。 objectextractpart.cpp 3401

D3DXVECTOR3 N = VectorMultiplication(
                  VectorMultiplication(V-VP, VN), VN);
float t = Qsqrt(Scalar(N, N));
if (t!=0)
{
  N/=t;
  V = V - N * DistPointToSurface(V, VP, N);
}

このライブラリでは、同様のエラーが頻繁に繰り返されます。これが私にとって驚きだったとは言いません。以前に、このプロジェクトで浮動小数点数を使用した誤った作業が発生しました。ただし、これについてソースを体系的にチェックするリソースはありませんでした。検証の結果によると、浮動小数点数の扱いに関して開発者の視野を広げるために、開発者に何かを読む必要があることが明らかになりました。私は彼にいくつかの良い記事へのリンクを投げました。結果を見てみましょう。このエラーがプログラムに実際の誤動作を引き起こすかどうかを明確に言うことは困難です。現在の解決策は、動脈の初期の多角形ネットワークにいくつかの要件を設定し、それに沿って放射線不透過性物質の拡散がシミュレーションされます。要件が満たされていない場合、プログラムがクラッシュするか、明らかに誤った操作が行われる可能性があります。これらの要件のいくつかは、分析的に、そして部分的に-経験的に得られます。要件のこの2番目の部分は、浮動小数点数を使用した誤った処理が原因で大きくなる可能性があります。浮動小数点数の正確な比較を使用して見つかったすべてのケースがエラーであったわけではないことに注意してください。

ご覧のように、いくつかのプロジェクトで面白くないものは他のプロジェクトに興味があります。これは、「最も興味深い」のリストを作成することを不可能にします。

注意。設定を使用して警告レベルを設定することが可能です。たとえば、V550の診断に注意が必要だと思われる場合は、3番目のレベルから1番目のレベルに移動できます。このタイプの設定は、ドキュメントで説明されています（「特定の診断のレベルを設定する方法」を参照）。
