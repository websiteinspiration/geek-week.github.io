UARTによるARM Cortex-Mマイクロコントローラーのデバッグ

この記事では、ARM Cortex-Mコア上に構築されたマイクロコントローラーでデバッグレジスタとブレークポイントレジスタを使用する方法について説明します

前書き


マイクロコントローラーのARM Cortex-Mコアが自分でデバッグできることをご存知ですか？
彼らはできることがわかりました。コア

のテクニカルリファレンスマニュアルを読んで、Cortex M3はDebugMon割り込みがあることを発見しました。次に、それに関連付けられているすべてのレジスタを分析しました。その結果、PCレジスタとFP_COMPレジスタの1つが等しい場合、MKがこの割り込みに陥ることがあることがわかりました。
これは、デバッグされたファームウェアにブレークポイントを設定できることを意味します。DEMCRレジスタのMON_PENDビットを1に設定して、DebugMon割り込みを強制することもできます。

理論チェック


これらのレジスタはARM Cortex-Mコアのファミリ全体に存在するため、最初に出会ったデバッグボードを採用します。それは私にとってstm32f723e-discoであることがわかりました。周辺機器の初期化コードの記述に時間を無駄にしないために、周辺機器
からCubeMXを使用します。ボード上のST-LinkとLEDに接続されているUART6のみが必要です。

CubeMXの設定


プロジェクトを生成し、すぐにそのIDEを開きます。

デバッグに関連するレジスタで混乱しないように、コードでの定義をすぐに紹介します。

レジスタ定義
#define DHCSR (*(uint32_t*)0xE000EDF0)
#define DCRSR (*(uint32_t*)0xE000EDF4)
#define DCRDR (*(uint32_t*)0xE000EDF8)
#define DEMCR (*(uint32_t*)0xE000EDFC)

#define NUMOFBKPTS 8

typedef struct {
  uint32_t FP_CTRL;
  uint32_t FP_REMAP;
  uint32_t FP_COMP[NUMOFBKPTS];
} fp_t;

#define FP ((fp_t*)0xE0002000)


プロジェクトが機能することを確認するには、原則として、LEDで点滅するコードを記述します。

LED点滅
  while (1)
  {
    for (int i=0;i<1000000;i++)
      asm("nop");
    
    HAL_GPIO_TogglePin(GPIOA,GPIO_PIN_7);
  }


LEDは正常に点滅しました。これは、プロジェクトが正常にコンパイルおよび起動されたことを意味します。

このループに行を追加します。

asm("BKPT #0");

このアセンブラー挿入はソフトウェアブレークポイントです。これで、ST-Linkを介してデバッグする場合、プログラムは常にこの行で停止します。IDEでデバッグを終了すると、以前と同様にLEDが点滅し始めます。

しかし、それは私たちに何を与えましたか？

そして、DebugMonを中断することでブレークポイントをキャッチできるという事実。

そのハンドラを書きます：

DebugMonハンドラー
void DebugMon_Handler(void)
{
  while (1)
  {
    if ((USART6->ISR & USART_ISR_TXE) != 0U)
    {
      USART6->TDR = '!';
    }
  }
}


コンパイル、フラッシュ、何も変更されていません。DebugMon割り込みを有効にするには、DEMCRレジスタのMON_ENビットを上げる必要があります。

これで、ST-Linkを介してデバッグすると、プログラムは以前と同じようにこの行で停止します。デバッグモードを終了すると、感嘆符がターミナルに表示されます。

端子出力


次に、FP_COMPレジスタで同じことを確認します。

IDEを使用してメイン関数ループ内の任意の命令のアドレスを見つけ、ブレークポイントをアクティブにします。

FP->FP_COMP[0] = 0x080017CC | 1; //     = 0x080017CC

ハードウェアデバッガーが無効になると、マイクロコントローラーもDebugMon割り込みに入ります。

それの使い方


上記のレジスターとDebugMon割り込みを使用すると、SWD / JTAGデバッガーなしでマイクロコントローラーをデバッグできます。接続されているデバイスのファームウェアのデバッグも可能になりますが、SWDピンへのアクセスは困難です。
