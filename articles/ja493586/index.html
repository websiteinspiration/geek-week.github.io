.NET 5向けのJITコンパイラの最適化

少し前、私はJITコンパイラーの世界へのすばらしい旅を始めました。手を入れて何かをスピードアップできる場所を見つけるために、主な作業の過程で、LLVMとその最適化に関する少量の知識が蓄積されました。この記事では、JIT（.NETでは一部のドラゴンやアニメにちなんでRyuJITと呼ばれています-わかりませんでした）の改善点のリストを共有したいと思います。そのほとんどは既にマスターに達しており、.NET（Core）5で利用できるようになります。私の最適化は、JITのさまざまなフェーズに影響を与えます。



これは、次のように非常に概略的に示すことができます。図からわかるように、JITは、狭いJit-Interfaceに関連する個別のモジュールであり、JITがいくつかのことについて相談します。あるクラスを別のクラスにキャストします。JITが後でメソッドをTier1にコンパイルするほど、ランタイムが提供できる情報が多くなります。たとえば、static readonlyフィールドを定数に置き換えることができるという理由により、クラスはすでに静的に初期化されています。

それでは、リストから始めましょう。

PR ＃1817：パターンマッチングにおけるボックス化/ボックス化解除の最適化


フェーズ：インポーター
多くの新しいC＃機能は、ボックス/ボックス解除 CILオペコードを挿入することによってしばしば罪を犯します。これは非常に負荷の高い操作であり、本質的にはヒープ上の新しいオブジェクトの割り当て、スタックからの値のコピー、そして最後にGCの読み込みも行われます。この場合、JITにはすでにいくつかの最適化がありますが、C＃8でパターンマッチングが見つからないことがわかりました。次に例を示します。

public static int Case1<T>(T o)
{
    if (o is int x)
        return x;
    return 0;
}

public static int Case2<T>(T o) => o is int n ? n : 42;

public static int Case3<T>(T o)
{
    return o switch
    {
        int n => n,
        string str => str.Length,
        _ => 0
    };
}

PR ＃1157 typeof(T1).IsAssignableFrom(typeof(T2)) ⇨ true/false


フェーズ：インポーター
ほぼ同じ-最適化されていないことを心配することなく、ジェネリックメソッドの階層を確認できます。次に例を示します。

PR ＃1644：バウンドチェックの最適化。


フェーズ：境界チェックの排除
多くの場合、インデックスで配列にアクセスするたびに、JITは配列がそれを超えないことのチェックを挿入し、これが発生した場合に例外をスローします-誤ったロジックの場合、ランダムメモリを読み取り、値を取得して続行します。

int Foo(int[] array, int index)
{
    // if ((uint) array.Length <= (uint) index)
    //     throw new IndexOutOfRangeException();
    return array[index];
}

このようなチェックは便利ですが、パフォーマンスに大きな影響を与える可能性があります。1つ目は比較操作を追加してコードを分岐させないこと、2つ目はメソッドに例外呼び出しコードを追加してすべての結果をもたらします。ただし、多くの場合、JITは、インデックスがそれを超えないこと、またはすでに他のチェックがあり、もう1つ追加する必要がないことを証明できる場合、これらのチェックを削除できます-境界（範囲）チェックの削除。私は彼がそれらを対処および修正できないいくつかのケースを発見しました（そして将来的にはこのフェーズのいくつかのさらなる改善を計画しています）。

var item = array[index & mask];

このコードでは、& mask上からのインデックスを本質的に値に制限することをJITに伝えていますmask。mask配列の値と長さがJITに認識されている場合、バインドされたチェックを挿入できません。同じことは、％、（＆x >> y）操作にも当てはまります。この最適化をaspnetcoreで使用する例。
また、たとえば、配列に256個以上の要素があることがわかっている場合、不明なインデクサーがバイト型であると、どんなに難しい試みを行っても、境界外に出ることができなくなります。PR：github.com/dotnet/coreclr/pull/25912

PR ＃32716：ブランチレスコードでの単純な比較の最適化


フェーズ：フロー分析
これは、1つのブロック内の式ではなく、基本ブロックで動作する別のクラスの最適化です。ここで、JITは少し保守的で、可能な場合はcmoveインサートなどの改善の余地があります。この場合の簡単な最適化から始めました。

x = condition ? A : B;

たとえば、AとBが定数で、両者の差が1の場合condition ? 1 : 2、比較演算自体が0または1を返すことがわかっているため、jumpをaddに置き換えることができます。RyuJITの観点からは次のようになります。PR 自体の説明



を確認することをお勧めします。すべてが明確に説明されていることを願っています。

すべての最適化が同じように役立つわけではありません。


最適化にはかなり高い料金が必要です：
*増加=サポートと読み取りのための既存のコードの複雑さ
*潜在的なバグ：コンパイラーの最適化のテストは非常に難しく、何かを見逃しやすく、ユーザーから何らかのセグメンテーション違反が発生します。
*コンパイルが遅い
* JITバイナリのサイズを大きく

するすでに理解したように、最適化のすべてのアイデアとプロトタイプが受け入れられるわけではなく、それらが生きる権利を持っていることを証明する必要があります。 .NETでこれを証明するために受け入れられている方法の1つは、jit-utilsユーティリティを実行することです。このユーティリティは、一連のライブラリ（すべてのBCLとcorelib）をAOTコンパイルし、最適化の前後にすべてのメソッドのアセンブラーコードを比較します。これは、このレポートが最適化を探す方法です。"str".Length。レポートに加えて、一目で有用性を評価し、経験の高さからすべてをハックし、.NET のどの場所がボトルネックになる可能性があるか、どの場所が不可能かを理解できる特定の人々（jkotasなど）のサークルがまだあります。さらにもう1つ、「誰も書き込みをしない」、「Roslynで警告を表示するほうがいい」という引数で最適化を判断しないでください。JITが可能なすべてをインライン化し、定数を埋めた後のコードの見え方はわかりません。
