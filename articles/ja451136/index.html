.NET 5で導入

5月6日、.NET Core 3.0以降の次のリリースが.NET 5になることが発表されました。これは、.NETファミリーの次の大きなリリースになるでしょう。

将来的には、1つの.NETのみが存在し、Windows、Linux、macOS、iOS、Android、tvOS、watchOS、WebAssemblyおよびその他のプラットフォームでの開発に使用できます。

.NET 5の一部として、新しい.NET API、ランタイム機能、および言語機能を紹介します。



.NET Coreプロジェクトの立ち上げ以降、約5万のAPI .NET Frameworkをプラットフォームに追加しました。その機能により、.NETのコア3.0は密接の.NET Framework 4.8に接近し、そのWindowsフォームのおかげで、WPFとEntity Frameworkの6が利用可能になりました。.NET 5はバトンを引き継いだ、それが基づいていた.NETのコアからのすべての最高のMonoプロジェクトの結果として、これは、すべての最新の.NETコードに使用できる単一のプラットフォームです。

.NET 5は2020年11月にリリースする予定で、最初のプレビューバージョンは2020の前半に利用可能になります。このプラットフォームは、Visual Studio 2019、Visual Studio for Mac、およびVisual Studio Codeの将来のアップデートで利用可能になります。

.NET 5 = .NET Core vNext


.NET 5は.NET Coreの次のステップです。このプロジェクトは、いくつかの重要な側面で.NETを改善することを目的としています。

  • どこでも使用できる単一のランタイムとフレームワークを作成し、同じランタイム動作と開発経験を持ちます。
  • .NET Core、.NET Framework、Xamarin、Monoの最高の機能で.NETを拡張します。
  • 開発者（Microsoftとコミュニティの）が協力して拡張できる単一のコードベースから製品を構築します。これにより、考えられるすべてのシナリオが改善されます。

この新しいプロジェクトと方向性は、.NETの状況を完全に変えます。.NET 5のおかげで、作成しているアプリケーションのタイプに関係なく、コードとプロジェクトファイルは均一に見えます。各アプリケーションから、ほぼ毎日corefxに実装される新しいパフォーマンスの改善を含む、同じランタイム、同じAPIおよび言語機能にアクセスできます。

.NET Coreの好きなものはすべて保持されています。

  • オープンソースとコミュニティはGitHubに焦点を当てています。
  • クロスプラットフォームの実装。
  • WindowsフォームやWPF for Windowsなどの特定のプラットフォーム固有の機能、およびXamarinの各ネイティブプラットフォームのネイティブバインディングの使用のサポート。
  • ハイパフォーマンス。
  • 並列インストール。
  • 小さなプロジェクトファイルサイズ（SDKスタイル）。
  • 強力なコマンドラインインターフェイス（CLI）。
  • Visual Studio、Visual Studio for Mac、およびVisual Studio Codeとの統合。

イノベーション：

  • ランタイム機能が増えます（これについては以下で詳しく説明します）。
  • .NET 5からJavaコードを呼び出す機能は、すべてのプラットフォームで利用できます。
  • .NET 5からのObjective-CおよびSwiftコードの呼び出しは、いくつかのオペレーティングシステムでサポートされます。
  • CoreFXは、静的な.NET（Ahead-of-Time-AOT）コンパイルをサポートするように拡張され、フットプリントを削減し、より多くのオペレーティングシステムをサポートします。

.NET Core 3.0は今年9月に、.NET 5-2020年11月に利用できるようになります。その後、毎年11月、毎年11月に.NETのメインバージョンをリリースします



。バージョン4.xで長くリリースされている.NET Frameworkとユーザーが混乱する可能性があるため、4番目のバージョンはスキップします。さらに、.NET 5が.NETプラットフォームの未来であることを明確にしたかったのです。

また、この機会を利用して、名前の順序を簡略化することにしました。.NETが1つしか開発されない場合は、「コア」という説明用語は必要ないと考えています。短い名前は単純です。つまり、.NET 5の機能と動作が統一されています。必要に応じて、.NET Coreという名前を引き続き使用できます。

ランタイム環境


Monoは、.NETのオリジナルのクロスプラットフォーム実装です。 .NET Frameworkに代わるオープンソースの手段として始まり、その後、iOSおよびAndroidデバイスの人気が高まるにつれ、モバイルセグメントに方向転換しました。 Monoは、Xamarinの一部として使用されるランタイム環境です。

CoreCLRは、.NET Coreの一部として使用されるランタイム環境です。当初はマイクロソフトの最大のサービスを含むクラウドアプリケーションのサポートに重点を置いていましたが、現在はデスクトップWindowsアプリケーション、IoT、機械学習にも使用されています。

.NET CoreとMonoランタイムには多くの共通点がありますが（ただし、どちらも.NETランタイムです）、それぞれに独自の機能があります。したがって、必要な使用体験を選択する機会を与えることは理にかなっています。現在、CoreCLRとMonoプラグインを相互に置き換えられるよう取り組んでいます。プロセスは、アセンブリを切り替えて異なるランタイムオプションを選択するのと同じくらい簡単です。

次の章では、.NET 5の主要な計画について説明します。これらの計画は、2つのランタイムを同時に、同時に別々に開発する方法を理解するのに役立ちます。

高い生産性と生産性


当初から、.NETはJITコンパイラに依存して、中間言語コードを最適化されたマシンコードに変換していました。業界をリードする非常に高いパフォーマンスのJITランタイムを作成し、開発者がコードをすばやく簡単に作成できるようにしました。

JITコンパイラーは、長時間実行されるクラウドやクライアントスクリプトに適しています。特定のプロセッサ命令を含む、ハードウェア構成の機能を考慮したコードを生成できます。 JITは実行時にメソッドを再度生成することもできます。この手法により、高速でコンパイルすると同時に、いくつかのメソッドが頻繁に使用される場合にコードの微調整バージョンを作成できます。

TechEmpower ベンチマークの結果に見られるように、ASP.NET Coreのパフォーマンスを加速する取り組みは、JIT機能の良い例であり、CoreCLRへの貢献です。コンテナーを使用するために.NET Coreを準備しようとしましたが、これはランタイム環境が限られた環境に動的に適応する能力を示しています。

開発者ツールは、JITが実証したもう1つの領域です。たとえば、ドットネットウォッチや「編集して続行」モードなどです。多くの場合、ツールは再起動せずに同じプロセスで何度もコードをコンパイルしてダウンロードする必要があり、これを非常に迅速に行う必要があります。

.NET Coreまたは.NET Frameworkを使用する開発者は、主にJITに依存しています。だから彼らは彼らに馴染みがあるように見えるはずです。

ほとんどの.NET 5ワークロードの標準的なアプローチは、JITでCoreCLRランタイムを使用することです。2つの重要な例外は、iOSとクライアントBlazor（WebAssembly）です。これらには、ネイティブの事前（事前）コンパイルが必要です。

高速起動、低フットプリント、メモリフットプリントの削減


Monoプロジェクトの一環として、ほとんどの取り組みはモバイルセグメントとゲームコンソールに向けられていました。このプロジェクトの主な機会と成果は、LLVMコンパイラーに基づいて開発された.NET用のAOTコンパイラーです。Mono AOTコンパイラーを使用すると、.NETコードを、C ++コードのように、任意のマシンで実行できる単一のネイティブ実行可能コードにコンパイルできます。プリコンパイル（AOT）アプリケーションは、限られたリソース（小さな場所）で効率的に実行でき、必要に応じて、それらの起動のパフォーマンスを犠牲にします。Blazor

プロジェクトはすでにMono AOTを使用しており、.NET 5に切り替える最初のプロジェクトの1つです。計画を立証する方法の1つとしてそれを使用しています。 AOTソリューションには2つのタイプがあります。



  • 完全なAOTコンパイルが必要です。
  • ソリューション。ほとんどのコードはAOTでコンパイルされていますが、AOTとは関係のないコードパターン（ジェネリックなど）に対してJITまたはインタープリターを使用できます。

Mono AOTは両方のタイプをサポートします。最初のタイプのAOTはiOSおよび一部のゲームコンソールに必要ですが、これは主にセキュリティ要件が原因です。2番目のタイプのソリューションは、AOTの利点をすべて備えているため、その欠点はありません。

.NET Nativeは、Windows UWPアプリケーションに使用するAOTコンパイラです。それはAOTソリューションの最初のタイプに属します。この特定の実装では、.NET APIと使用可能なオプションが制限されています。これは、AOTソリューションがすべての.NET APIとパターンをカバーする必要があることを理解するのに役立ちました。

iOS、WebAssembly、および一部のゲームコンソールでは、AOTコンパイルが引き続き必要です。アプライアンスに組み込まれている（アプライアンスのような）アプリケーションの場合は、クイックスタートや低CPU消費が必要な場合は、オプションにします。

基本と関連要件


起動、パフォーマンス、メモリ消費、信頼性、および診断のためのコントロールを備えたプラットフォームとして進化し続けることは、私たちにとって重要です。同時に、私たちの努力に集中することをお勧めします。 CoreCLRのパフォーマンスと信頼性を改善し、Mono AOTコンパイラの起動とファイルサイズを改善するために、さらに努力します。私たちには良い組み合わせのようです。パフォーマンスと信頼性は密接に関連しており、ファイルサイズを小さくすることで起動速度も向上します。

いくつかの特性を改善するために異なるリソースを投資することをお勧めしますが、他の特性を改善することはお勧めできません。

診断機能は、.NET 5全体で同じである必要があります。これは、機能とパフォーマンスの両方に適用されます。同じプロセッサとOSをサポートすることも重要です（iOSとWebAssemblyを除く）。

これが理にかなっているあらゆる種類のワークロードとスクリプトに対して.NET 5を最適化し続けます。特に、異なる負荷が同様の要件を課す場合は、最適化が最も重視されます。

すべての.NET 5アプリケーションはCoreFXフレームワークを使用します。 CoreFXが現在使用されていない場所、主にXamarinクライアントのBlazorタスクで正常に機能することを確認します。

すべての.NET 5アプリケーションは.NET CLIを使用して構築できますすべてのプロジェクトで、コマンドラインに基づいた単一のツールキットを使用できるようにします。

C＃は.NET 5とともに進化します。.NET5アプリケーションを作成する開発者は、C＃の最新バージョンとそのプロパティにアクセスできます。

プロジェクトの誕生


技術チームとして、2018年12月にボストンに集まり、このプロジェクトを開始しました。.NETチーム（Mono / Xamarinおよび.NET Core）の主要なアーキテクトとUnityは、さまざまな技術的機能とアーキテクチャの開発の方向性について話しました。

現在、チームとしてプロジェクトを移動しています。12月以来、私たちはいくつかのプロジェクトで大きな進歩を遂げました：

  • 99％以上のCoreFXを共通のコードにするために、ランタイムとマネージコードレイヤーの間の相互作用を定義する最小レベルを決定しました。
  • MonoVMは、CoreFXとそのクラスライブラリを使用できるようになりました。
  • その実装を使用して、MonoVMですべてのCoreFXテストを実行しました。
  • MonoVMでASP.NET Core 3.0アプリケーションをリリース。
  • CoreCLRでMonoDevelopおよびVisual Studio for Macをリリース。

単一の.NET実装を追求すると、重要な問題が生じます。最終的なフレームワークは何ですか？NuGetパッケージの互換性ルールは残りますか？.NET 5 SDKはボックスからどのような負荷をサポートしますか？特定のアーキテクチャ用のコードを書く方法は？.NET Standardが必要ですか？現在、このすべてに取り組んでおり、すぐにプロジェクトのドキュメントを共有して、ドキュメントを読んでフィードバックを提供できるようになります。

結論


.NET 5プロジェクトは、.NETにとって重要で刺激的な新しい方向性です。.NETがより簡単になることがわかりますが、同時により広く使用されるようになり、より多くの機会が得られます。C＃の新しいバージョンを含むすべての新しい開発機能は、.NET 5の一部になります。

幅広いアプリケーション、オペレーティングシステム、プロセッサアーキテクチャで同じ.NET APIと言語を使用できる明るい未来が待っています。Visual Studio、Visual Studio for Mac、Visual Studio Code、Azure DevOps、またはコマンドラインから、好きなようにアプリケーションをアセンブルすることで、ビルド構成を簡単に変更できます。
