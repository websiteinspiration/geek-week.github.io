node.jsでAPIサービスを改善する方法。パート1

node.jsでAPIを記述する際に、適切な量の資料が利用可能になりました。それらのほとんどは、ドキュメントのチュートリアルとデモの形式です。これは、すぐに独自のものを見つけて書くのに十分です。しかし、なぜこれがそのように行われるのかについての詳細はめったに見つかりません。また、いくつかのポイントは、簡潔さと簡潔さのために完全に省略されています。

この記事は、発生する可能性があるギャップを埋め、最終的にはnode.jsでのサービスを改善することを目的としています。

PS私は自分自身を専門家とは考えていません。成長する余地はあります。しかし同時に、共有する必要があるものもあります。

プロジェクトファイルの構造


ファイル構造は基本的ですが、非常に重要です。それを作成するときは、サービスをスケーリングする可能性を考慮する価値があります。このため、1つのディレクトリのフラット構造にファイルを配置する価値はありません。階層が必要で、モジュール性が必要です。

ディレクトリに名前を付けるときは、確立された標準に従う必要があります。これは、同僚や数か月後に、どこに何があるかをすばやく見つけるのに役立ちます。

ファイル構造の例

src/
    controllers/
         users/
             index.js
         index.js
    db/
         mongo/
             index.js
         index.js
    helpers/
    middlewares/
         auth.js
    models/
         users.js
    routes/
        users/
             index.js
        index.js
    index.js

.eslintrc
.gitignore
README.md
...
package.json

ノート：

  1. すべての実行可能ファイルは./srcに配置する必要があります。
    これにより、。/ distまたは./buildに配置されるコンパイル済みのtypescriptまたはbabelからソースファイルを分離できます。
    また、ターゲットファイルが別のディレクトリにあるため、リンターやその他の構成可能なツールの構成が容易になります。
  2. すべてのコントローラーとルートエンティティを別々のディレクトリに配置します。
    これにより、。/ routesまたは./controllerが開き、サブディレクトリエンティティをインポートおよびエクスポートする1​​つのindex.jsファイルのみが表示されます。
    これにより、コントローラーやルーターの実装について詳しく説明することなく、インターフェースのみを説明するため、既存の機能をすばやくナビゲートできます。

JavaScriptコンパイルについて


もちろん、JavaScriptの変換にbabelやtypescriptを使用することはできません。原則として、新しいECMAScript標準はすぐにnode.jsになり、ブラウザーベースのJavaScriptを除いて、サポートを制御できます。
しかし、私はそれらが使用する価値があると確信しています。そしてそれが理由です：

  1. Node.jsの最新バージョンは、本番環境で常に利用できるとは限りません。
  2. Node.js（v。13）はECMAScriptモジュールのサポートにまだ問題がありますが、表示されてフラグを残しましたが、まだ実験段階です。早く売りに。

プロジェクトにbabelを追加することを検討してください。

依存関係をインストールします。

npm install @babel/core @babel/node @babel/preset-env --save-dev

.babelrc構成ファイルをプロジェクトルートに追加します。
{
    "presets": [
      "@babel/preset-env"
    ]
}

スクリプト実行の例
"start": "nodemon --exec babel-node src/index.js",

ノート：

  1. 依存関係をインストールするときは、開発段階でのみ--save-dev（-D）スイッチを指定することを忘れないでください。少なくともセマンティクスには必要です。

リンターの必要性について


明らかな事実から始めましょう。リンターが必要です。私たちはコード、たくさんのコードを書きます。その多くは、彼ら自身がその均一性を制御することができません。このアイテムは、チーム開発で信じられないほど強化されています。

そして、コードの読みやすさの鍵は均一性です。

リンターの厳格さについての問題は残っています。どちらを選択するかは、チームの規模、チームの専門レベルに基づく必要があり、重要な基準はプロジェクト自体です。

プロジェクトに簡単なリンターを追加する

依存関係をインストールしましょう：

npm install eslint --save-dev

.eslintrc構成ファイルをプロジェクトのルートに追加します。

{
    "env": {
        "node": true,
        "es6": true,
    },
    "extends": "eslint:recommended"
}

原則として、このような設定では不十分です。ここで自分でルールを拡張するか、用意されているより厳密なオプションを詳しく見てみましょう。

 ,  
...
    "extends": "eslint:recommended",
    "rules": {
        "quotes": ["error", "single"]
    }
}

より厳密なリンターの使用

同じスタイルガイドに基づいた一般的な構成がいくつかあります。

  • グーグル

     npm install --save-dev eslint-config-google
  • Airbnb

     npm install --save-dev eslint-config-airbnb-base eslint-plugin-import
  • 慣用句
     npm install --save-dev eslint-config-idiomatic

したがって、.eslintrcを修正する必要があります。

{
    "env": {
        "node": true,
        "es6": true,
    },
    "extends": "google" | "airbnb-base" | "idiomatic"
}

リンターの起動を別のスクリプトに追加することも必要です。

"lint": "eslint ./src --cache && echo \"eslint: no lint errors\"",
"lint:fix": "eslint ./src --fix && echo \"eslint: no lint errors\""

アプリケーションの起動について


プロジェクトリポジトリへのアクセス権を持つユーザーは、誰でもそれを実行できるはずです。彼がそれを実行できるかどうか、そして彼がこれを実行するのにどのくらいの時間がかかるかは、プロジェクトの品質基準の1つです。

README.mdファイルにサービスを段階的に開始する方法を事前に文書化し、主なアクションのコマンドを規定することが重要です。

"start": "npm run dev",
"dev": "nodemon --exec babel-node ./src/index.js",
"build": "babel ./src --out-dir ./build",
"prod": "NODE_ENV=production node ./build/index.js",
"lint": "eslint ./src --cache && echo \"eslint: no lint errors\"",

上記のコマンドを使用するには、次のパッケージをインストールする必要がある場合があります。

npm install --save-dev @babel/cli nodemon babel-node

注意：

  1. 基本的なコマンドをpackage.jsonに追加した場合でも、README.mdに起動プロセスを記述します。
