使用客户旅程图进行客观的质量得分测试

您好，哈布罗夫斯克市民！

我是专业分析师，我从事系统设计和编写要求已有10年了。我建议讨论测试UX / UI的方法。

在隔离之前，我遇到了一个情况：我不得不买机票。大约30分钟，我进入了无空气的空间：我填写了所有字段（尽管服务中有配置文件），然后我再次仔细检查了所有内容，然后仔细删除了所有付费选项。
恕我直言，这是不人道的。

为什么不记得我的数据并替代它呢？为什么要施加其他选择-降低忠诚度？

我认为我不是唯一对不舒服和无法理解的UX / UI感到恼火的人。问题：“如何治疗？”



到目前为止，我得出的结论是，解决方案可能是系统质量指标的系统且独立的测试。在研究所和教科书上谈论的人：人体工程学，便利性，易懂性，任务执行速度。

通常，质量测试可以归结为以下几点：

  • 团队竭尽全力按时完成功能，没有人考虑过UX / UI
  • 仅检查UX / UI是否符合布局
  • UX / UI测试是根据测试人员的品味和个人评估进行的。是的，这可以通过一般规则来证明是正确的（例如，来自Cooper的“ On Interfaces”一书），但是在定制和B2B系统中，通常会有不平凡的解决方案。团队可能对美丽的品味和想法有所不同。结果，由于意见分歧而导致非建设性的纠纷。他们中最强的胜利。

然后我认为在收集产品需求时也会发生类似的故事。碰巧的是，创作者会为自己的痛苦做出决定，并认为他们会满足最终用户的需求。为了绘制现实，有一种客户旅程地图的方法。

该方法的本质是在解决特定问题时，从多个用户的统计样本中获取经验。因此，您可以找出其他人是否面临问题，以及这种情况是少数人所独有的还是全部问题。

为什么不检查这个原理和UX / UI？

测试算法：

1）确定测试策略。

  • 用于测试的用户脚本集
  • 经过测试的质量指标：便利性，人体工程学，易懂性等

2）查找要测试的受访者

3至5名受访者。它可以是最终用户，也可以是另一个团队的成员。最主要的是独立于开发团队。

3）进行研究。对每个响应者重复此

循环
>将获取用户卡模板。
>向受访者介绍任务和正在运行的系统/原型/布局
>受访者的步骤，他的反馈，执行时间-所有这些都记录在模板
末尾



4）汇总数据。

由于一种地图模板是一种意见，因此您需要找到受访者的相交之处。对于每种情况，都会按照CJM的最佳传统编译一个差距图矩阵。



  • 矩阵列-脚本步骤。
  • 线条表示不满。
  • 在十字路口-来自受访者地图的结果。
  • 花费大量时间或负面情绪，误解的步骤-这是系统质量的痛苦。

5）列出发现的问题。

我将其命名为GrowthPointsList。



数据汇总在问题清单+受访者的期望中，系统将如何运行。您可以指定对脚本通过的影响级别：

  1. 无法完成脚本
  2. 负数但脚本已通过
  3. 一些受访者表示否定，但他们设法通过了脚本。

因此，即使在布局阶段进行开发之前，也可以以较低的成本测试布局并消除问题。测试结果合理，不会引起参与者和客户之间的争议。

我认为应用程序的世界因此可以改变为最终用户带来方便和简单。
