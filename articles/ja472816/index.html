最新のJavaScriptのエレガントなパターン（Bill Sourourコンパイルサイクルの記事）

こんにちは、ハブル！当時、かなり有名なJavaScriptの教師であるBill Sourourが、JSの最新パターンに関するいくつかの記事を書いています。この記事の一部として、私たちは彼が共有したアイデアをレビューしてみます。それがいくつかのユニークなパターンであったというわけではありませんが、記事がその読者を見つけることを望みます。この記事は、Habrのポリシーの観点からの「翻訳」ではありません。ビルの記事が私に指摘した私の考えを説明します。

ルーロ


略語は、オブジェクトの受信、オブジェクトの返却-オブジェクトの取得、オブジェクトの返却を意味します。元の記事へのリンク

は次のとおりです。リンクビルは、ほとんどの関数が1つのパラメーター（関数の引数を持つオブジェクト）のみをとる関数の記述方法に到達したと書いています。結果のオブジェクトも返します。ビルは、このアイデア（ES6機能の1つ）の再構築に触発されました。

解体について知らない人のために、ストーリーの中で必要な説明をします。

データオブジェクトに表示されるアプリケーションの特定のセクションに対する権限を含むユーザーデータがあるとします。このデータに基づいて特定の情報を表示する必要があります。これを行うには、次の実装を提供できます。

//  
const user = {
  name: 'John Doe',
  login: 'john_doe',
  password: 12345,
  active: true,
  rules: {
    finance: true,
    analitics: true,
    hr: false
  }
};

//  
const users = [user];

//,    
function findUsersByRule (  rule,   withContactInfo,   includeInactive) {
  //        active 
  const filtredUsers= users.filter(item => includeInactive ? item.rules[rule] : item.active && item.rules[rule]);
//  ()   ( )     withContactInfo
  return withContactInfo ?
    filtredUsers.reduce((acc, curr) => {
      acc[curr.id] = curr; 
      return acc;
    }, {})
    : filtredUsers.map(item => item.id)
}

//    
findUsersByRule(  'finance',   true,   true)

上記のコードを使用すると、望ましい結果が得られます。ただし、この方法でコードを記述する場合、いくつかの落とし穴があります。

まず、関数呼び出しはfindUsersByRule非常に疑わしいです。最後の2つのパラメーターがどれほどあいまいであるかに注意してください。アプリケーションが連絡先情報（withContactInfo）をほとんど必要とせず、ほとんど常に非アクティブなユーザー（includeInactive）を必要とする場合はどうなりますか？常に論理値を渡す必要があります。ここで、関数宣言がその呼び出しの隣にありますが、これはそれほど恐ろしいことではありませんが、このような呼び出しが別のモジュールのどこかで行われたことを想像してください。純粋な形式の2つの論理値がモジュールに転送される理由を理解するために、関数宣言のあるモジュールを探す必要があります。

次に、一部のパラメータを必須にしたい場合は、次のように記述する必要があります。

function findUsersByRule (  role,   withContactInfo,   includeInactive) {  
   if (!role) {      
      throw Error(...) ;
  }
//... 
}

この場合、私たちの機能は、その検索責任に加えて、検証も実行します。特定のパラメーターでユーザーを検索したかっただけです。もちろん、検索関数は検証関数を取ることができますが、入力パラメーターのリストは拡張されます。これもそのようなコーディングパターンのマイナスです。

非構造が単純な部品に複雑な構造を破壊含まれます。 JavaScriptでは、このような複雑な構造は通常、オブジェクトまたは配列です。分解構文を使用すると、配列またはオブジェクトから小さなフラグメントを抽出できます。この構文は、変数またはその目的を宣言するために使用できます。ネストされた構造化の構文を使用して、ネストされた構造を管理することもできます。

分解を使用すると、前の例の関数は次のようになります。

function findUsersByRule ({  rule,  withContactInfo,   includeInactive}) {
//   
}

findUsersByRule({  rule: 'finance',   withContactInfo: true,   includeInactive: true})

パラメータを括弧で囲んでいることを除いて、関数の外観はほとんど同じであることに注意してください。代わりに、私達の機能の三つの異なるパラメータを受信する今、次のプロパティを持つ単一のオブジェクトを期待：rule、withContactInfoとincludeInactive。

これは、あいまいさがはるかに少なく、読みやすく、理解しやすくなっています。さらに、パラメーターのスキップや別の順序は、オブジェクトの名前付きプロパティになっているため、問題ではなくなりました。関数宣言に安全に新しいパラメーターを追加することもできます。さらに、解体は渡された値をコピーするので、関数での変更は元の値に影響を与えません。

必要なパラメーターの問題も、よりエレガントな方法で解決できます。

function requiredParam (param) { 
 const requiredParamError = new Error( `Required parameter, "${param}" is missing.`  )
}

function findUsersByRule ({  rule = requiredParam('rule'),  withContactInfo,   includeInactive} = {}) {...}

ルールの値を渡さない場合、デフォルトで渡される関数が機能し、例外がスローされます。

JSの関数は1つの値のみを返すことができるため、オブジェクトを使用してより多くの情報を転送できます。もちろん、多くの情報を返す関数が常に必要なわけではありません。たとえば、プリミティブの戻りで満足できる場合もあります。たとえば、findUserId1つの識別子がなんらかの条件で自然に戻る場合があります。

また、このアプローチにより、機能の構成が簡素化されます。実際、合成では、関数は1つのパラメーターのみを取る必要があります。 ROROパターンは同じ契約に従います。

ビル・サワー：「他のテンプレートと同様に、ROROはツールキットの別のツールと見なす必要があります。メリットのある場所で使用し、パラメーターリストをより理解しやすく柔軟にし、戻り値をより表現力のあるものにします。」

製氷工場


このリンク で元の記事を見つけることができます。

著者によると、このテンプレートは凍結されたオブジェクトを作成して返す関数です。

ビルは考えています。状況によっては、このパターンで通常のES6クラスを置き換えることができます。たとえば、製品を追加/削除できる特定のフードバスケットがあります。

ES6クラス：

// ShoppingCart.js
class ShoppingCart {
  constructor({db}) {
    this.db = db
  }
  
  addProduct (product) {
    this.db.push(product)
  }
  
  empty () {
    this.db = []
  }
  get products () {
    return Object
      .freeze([...this.db])
  }
  removeProduct (id) {
    // remove a product 
  }
  // other methods
}
// someOtherModule.js
const db = [] 
const cart = new ShoppingCart({db})
cart.addProduct({ 
  name: 'foo', 
  price: 9.99
})

キーワードを使用して作成されたオブジェクトnew は変更可能です。クラスインスタンスメソッドをオーバーライドできます。

const db = []
const cart = new ShoppingCart({db})
cart.addProduct = () => 'nope!' //   JS 
 
cart.addProduct({ 
  name: 'foo', 
  price: 9.99
}) // output: "nope!"

また、JSのクラスはプロトタイプ委譲で実装されているため、クラスのプロトタイプでメソッドの実装を変更でき、これらの変更は既存のすべてのインスタンスに影響します（これについては、OOPに関する記事で詳しく説明しました）。

const cart = new ShoppingCart({db: []})
const other = new ShoppingCart({db: []})
ShoppingCart.prototype
  .addProduct = () => ‘nope!’
//     JS

cart.addProduct({ 
  name: 'foo', 
  price: 9.99
}) // output: "nope!"

other.addProduct({ 
  name: 'bar', 
  price: 8.88
}) // output: "nope!"

そうですね、そのような機能は私たちに多くの問題を引き起こす可能性があります。

もう1つの一般的な問題は、インスタンスメソッドをイベントハンドラーに割り当てることです。

document
  .querySelector('#empty')
  .addEventListener(
    'click', 
    cart.empty
  )

ボタンをクリックしてもバスケットは空になりません。このメソッドは、ボタンに新しいプロパティをdbという名前で割り当て、カートオブジェクトのdbに影響を与える代わりに、このプロパティを[]に設定します。ただし、コンソールにはエラーはなく、常識ではコードは機能するはずですが、機能しません。

このコードを機能させるには、矢印関数を記述する必要があります。

document
  .querySelector("#empty")
  .addEventListener(
    "click", 
    () => cart.empty()
  )

または、バインドでコンテキストをバインドします。

document
  .querySelector("#empty")
  .addEventListener(
    "click", 
    cart.empty.bind(cart)
  )

製氷工場 はこれらのわなを避けるのに役立ちます。

function makeShoppingCart({
  db
}) {
  return Object.freeze({
    addProduct,
    empty,
    getProducts,
    removeProduct,
    // others
  })
  function addProduct (product) {
    db.push(product)
  }
  
  function empty () {
    db = []
  }
  function getProducts () {
    return Object
      .freeze([...db])
  }
  function removeProduct (id) {
    // remove a product
  }
  // other functions
}
// someOtherModule.js
const db = []
const cart = makeShoppingCart({ db })
cart.addProduct({ 
  name: 'foo', 
  price: 9.99
})

このパターンの特徴：

  • 新しいキーワードを使用する必要はありません
  • これをバインドする必要はありません
  • カートは完全に動かない
  • 外部から見えないローカル変数を宣言できます

function makeThing(spec) {
  const secret = 'shhh!'
  return Object.freeze({
    doStuff
  })
  function doStuff () {
    //    secret

}
// secret   
const thing = makeThing()
thing.secret // undefined
