測量作業


2つの典型的な測地タスクがあります：直接と逆です。

たとえば、現在の座標（緯度と経度）がわかっていて、厳密に北東、井戸、または北まで1000 キロ歩いたとします。私は今どのような座標を持っていますか？ -私がどのような座標を持つかを知ることは、直接測地問題を解決することです。

つまり、直接測地線タスクは、既知の距離と方向角によってポイントの座標を見つけることです。

逆の問題では、すべてが完全に明確です。たとえば、座標を決定してから、しばらく直線で歩き、再び座標を決定しました。私がどれだけ行ったかを見つけることは、逆測地線問題を解決することを意味します。

つまり：逆測地線問題は、既知の地理座標を持つ2つのポイント間の距離を見つけることです。

これらの問題は、必要な精度とそれに費やす時間に応じて、いくつかの方法で解決できます。

最も簡単な方法は、地球が平らであると想像することです。これは球体です。やってみよう。
直接的な問題を解決するための式（source）は次のとおりです。

ϕ2=arcsin(sinϕ1cosδ+cosϕ1sinδcosθ)

λ2=λ1+atan2(sinθsinδcosϕ1,cosδ−sinϕ1sinϕ2)


ここに 、ϕ1 -出発点の緯度と経度、λ1は方向角で、北方向から時計回りに測定されます（上から見た場合）。θδは角距離d / Rです。dは測定された（移動された）距離、Rは地球の半径です。ϕ2、λ2所望の点（私たちが到着したもの）の緯度と経度です。

逆問題を解決するために、別の方法があります（それほど単純な公式ではありません）。

a=sin2(Δϕ/2)+cosϕ1cosϕ2sin2(Δλ/2)

d=R∗atan2(a,1−a)


どこ ϕ1、λ1とϕ2、λ2は点の座標、Rは地球の半径です。

説明されている式は、Haversine方程式と呼ばれます。

  • C＃実装では、対応する関数はHaversineDirectおよびHaversineInverseと呼ばれ、Algorithms.csにあります。
  • で錆の実装で、これらはhaversine_directとhaversine_inverse機能。
  • , Matlab :
    HaversineDirect HaversineInverse

C＃の場合は、関数の名前とそれらが配置されているファイルへのリンクを提供します。 Rustの場合-（ライブラリ全体が1つのファイルにあるため）関数の名前のみ。Matlabの場合-対応するスクリプトファイルへのリンク。Matlabには1つの関数-1つのスクリプトがあるためです。

明らかに、ある種の問題があります。地球は球体ではなく平面であり、これが何らかの形でこれらの数式の適用性やソリューションの精度に影響を与えているはずです。

そして本当に。しかし、これを決定するには、何かと比較する必要があります。
1975年に戻って、Thaddeus Vincenty は回転楕円体の表面の測地線の問題を直接および逆算するための計算上効果的なソリューションを発表しました（革命の楕円体、同志というニックネームでよく知られています！回転のエリソイド）、ほぼ標準になりました。

メソッドデバイスの説明は別の記事に描かれているので、送信するのはVincentiのオリジナルの作品と、アルゴリズムの説明が記載されたオンライン計算機のみに限定します。

UCNLNavライブラリでは、Vincenti式を使用した直接および逆測地線問題の解決策は次の関数にあります。

  • C＃：VincentyDirectとVincentyInverseでAlgorithms.cs。
  • Rust：vincenty_inverseおよびvincenty_direct
  • Matlab：Nav_vincenty_directおよびNav_vincenty_inverse。

なぜならVincentiの解法は反復的であり、反復の最大数（it_limit）はパラメーターリストにあり、実際の反復数は結果リストにあります。停止条件（イプシロン）を指定するしきい値もあります。ほとんどの場合、10回を超える反復は必要ありませんが、ほぼ正反対の点（北極と南極など）の場合、メソッドの収束は不十分であり、最大2000回の反復が必要になる場合があります。

最も重要な違いは、これらの式は回転楕円体の解を満たし、そのパラメーターを関数に転送する必要があることです。これを説明する単純な構造があります。

すべての実装において、1つの標準楕円体を1行で取得できます。 （非常に頻繁にWGS84 [https://en.wikipedia.org/wiki/World_Geodetic_System]が使用され、例としてそれを示します）：

  • Cの場合：Algorithms.csには、静的フィールドAlgorithms.WGS84Ellipsoidがあります-メソッドに渡すことができます。
  • Rustについて：
    let el: Ellipsoid = Ellipsoid::from_descriptor(&WGS84_ELLIPSOID_DESCRIPTOR);
  • Matlabの場合：
    el = Nav_build_standard_ellipsoid(‘WGS84’);

残りのパラメーターの名前は非常に明白であり、あいまいさを引き起こしてはなりません。

ナビゲーションの問題を解決する


現時点では、ライブラリは以下を決定できます。

  • 2Dおよび3Dで既知の座標を持つポイントまでの距離によるオブジェクトの場所。これがTOA-Time Of Arrival（正確には、TOF-Time Of Flight）と呼ばれるものです。
  • 2Dと3Dの到着時間の違いによるオブジェクトの場所。これがTDOA（到着時差）と呼ばれるものです。

実際には、信号の到着の範囲または時間（したがって、それらの差）を、エラーやノイズとともに常に測定します。したがって、ほとんどの場合のナビゲーションの問題の解決策は、エラーの最小化です。最小二乗法とそれだけです。

最小化する必要があるものは、残差関数と呼ばれます。

TOAタスクの場合、次のようになります。

argminϵ(x,y,z)=∑i=1N[(x−xi)2+(y−yi)2+(z−zi)2)−ri]2


どこ ϵ(x,y,z) -座標を持つ特定のポイントの残差関数の値 (x,y,z); Nは座標を持つ制御点の数です(xi,yi,zi)、 ri-基準点から配置されたオブジェクトまでの測定距離。

そして、このようなTDOAタスクの場合：

argminϵ(x,y,z)=∑i=1,j=2,i≠jN[(x−xi)2+(y−yi)2+(z−zi)2)−(x−xj)2+(y−yj)2+(z−zj)2)−ν(tAi−tAj)]2



ここではすべてが同じで、参照ポイントと対応する到着時間の異なるペアのみが考慮されます。 tAi そして tAj、および ν-信号伝搬速度。

そして、これらの関数はコードを調べます。

Cで：
/// <summary>
/// TOA problem residual function
/// </summary>
/// <param name="basePoints">base points with known locations and distances to them</param>
/// <param name="x">current x coordinate</param>
/// <param name="y">current y coordinate</param>
/// <param name="z">current z coordinate</param>
/// <returns>value of residual function in specified location</returns>
public static double Eps_TOA3D(TOABasePoint[] basePoints, double x, double y, double z)
{
     double result = 0;
     double eps = 0;
     for (int i = 0; i < basePoints.Length; i++)
    {
         eps =  Math.Sqrt((basePoints[i].X - x) * (basePoints[i].X - x) +
                                 (basePoints[i].Y - y) * (basePoints[i].Y - y) +
                                 (basePoints[i].Z - z) * (basePoints[i].Z - z)) - basePoints[i].D;
         result += eps * eps;
    }
    return result;
}

/// <summary>
/// TDOA problem residual function
/// </summary>
/// <param name="baseLines">base lines, each represented by two base points with known locations and times of arrival</param>
/// <param name="x">current x coordinate</param>
/// <param name="y">current y coordinate</param>
/// <param name="z">current z coordinate</param>
/// <returns>value of residual function in specified location</returns>
public static double Eps_TDOA3D(TDOABaseline[] baseLines, double x, double y, double z)
{
     double result = 0;
     double eps;
     for (int i = 0; i < baseLines.Length; i++)
     {
          eps = Math.Sqrt((baseLines[i].X1 - x) * (baseLines[i].X1 - x) +
                                 (baseLines[i].Y1 - y) * (baseLines[i].Y1 - y) +
                                 (baseLines[i].Z1 - z) * (baseLines[i].Z1 - z)) -
                   Math.Sqrt((baseLines[i].X2 - x) * (baseLines[i].X2 - x) +
                                 (baseLines[i].Y2 - y) * (baseLines[i].Y2 - y) +
                                 (baseLines[i].Z2 - z) * (baseLines[i].Z2 - z)) - baseLines[i].PRD;
          result += eps * eps;
      }
      return result;
}


Rustについて：

pub fn eps_toa3d(base_points: &Vec<(f64, f64, f64, f64)>, x: f64, y: f64, z: f64) -> f64 {
    
    let mut result: f64 = 0.0;
    
    for base_point in base_points {
        result += (((base_point.0 - x).powi(2) +
                    (base_point.1 - y).powi(2) +
                    (base_point.2 - z).powi(2)).sqrt() - base_point.3).powi(2);
    }

    result
}
 
pub fn eps_tdoa3d(base_lines: &Vec<(f64, f64, f64, f64, f64, f64, f64)>, x: f64, y: f64, z: f64) -> f64 {

    let mut result: f64 = 0.0;
    
    for base_line in base_lines {
        result += (((base_line.0 - x).powi(2) +
                    (base_line.1 - y).powi(2) +
                    (base_line.2 - z).powi(2)).sqrt() -                  
                   ((base_line.3 - x).powi(2) +
                    (base_line.4 - y).powi(2) +
                    (base_line.5 - z).powi(2)).sqrt() - base_line.6).powi(2);        
    }

    result
}

ライブラリの内部構造について詳しく説明します


この段階では、ライブラリーは2Dおよび3Dタスクで動作し、ソルバー自身は最小化された機能がどのように見えるかを知りませんし、知りたくありません。これは次の方法で実現されます。

ソルバーには2つの側面があります。ネルダーミード法、またはシンプレックス法とも呼ばれる2Dおよび3Dソルバーです。

この方法は導関数の計算を必要としないため（いわゆる導関数なしの最小化）、理想的には、ライブラリーのユーザーは必要に応じて独自の残差関数を使用できます。さらに、理論的には、問題の解決に使用される制御点の数に上限はありません。

C＃およびRustでは、2Dおよび3Dソルバーはジェネリックメソッドです。

public static void NLM2D_Solve<T>(Func<T[], double, double, double, double> eps, 
   T[] baseElements,...

//       :
fxi[0] = eps(baseElements, xix[0], xiy[0], z);

    [ x_best, y_best, rerr, it_cnt ] = Nav_nlm_2d_solve(@Nav_eps_toa3d, base_points, x_prev, y_prev, z,...
        max_iterations, precision_threshold, simplex_size);
end

#[test]
fn test_tdoa_locate_3d() {
   let el: Ellipsoid = Ellipsoid::from_descriptor(&WGS84_ELLIPSOID_DESCRIPTOR);

   let base_number = 4; // 4  
   let start_base_z_m: f64 = 1.5; //  Z    
   let base_z_step_m = 5.0; //        5 

   let actual_target_lat_deg: f64 = 48.513724 // singed degrees
   let actual_target_lon_deg: f64 = 44.553248; // signed degrees
   let actual_target_z_m: f64 = 25.0;          // meters - ,    !

   // generate base points via Vincenty equations
   let mut base_points = Vec::new();

   let start_dst_projection_m = 500.0;          //      500 
   let dst_inc_step_m = 50.0;                      //   -  50  
                      
   // azimuth step
   let azimuth_step_rad = PI2 / base_number as f64; //    

   let actual_target_lat_rad = actual_target_lat_deg.to_radians();
   let actual_target_lon_rad = actual_target_lon_deg.to_radians();

   // signal propagation speed
   let velocity_mps = 1450.0; // m/s,        
   //    
   for base_idx in 0..base_number {
       //       
       let dst_projection_m = start_dst_projection_m + dst_inc_step_m * base_idx as f64;
       //      
       let azimuth_rad = azimuth_step_rad * base_idx as f64;
       //        Vincenty
       let vd_result = vincenty_direct(actual_target_lat_rad, actual_target_lon_rad, 
           azimuth_rad, dst_projection_m, 
           &el, 
           VNC_DEF_EPSILON, VNC_DEF_IT_LIMIT);
                
       //   Z
       let base_z_m = start_base_z_m + base_z_step_m * base_idx as f64;
       //       
       let dz_m = actual_target_z_m - base_z_m;
       //     
       let slant_range_m = (dst_projection_m * dst_projection_m + dz_m * dz_m).sqrt();
       
       //   .               Rust   !
       base_points.push((vd_result.0.to_degrees(), vd_result.1.to_degrees(), base_z_m, slant_range_m / velocity_mps));
   }
        
    //     -   NAN-
    let lat_prev_deg = f64::NAN;
    let lon_prev_deg = f64::NAN;
    let prev_z_m = f64::NAN;

    //  
    let tdoa_3d_result = tdoa_locate_3d(&base_points,
    lat_prev_deg, lon_prev_deg, prev_z_m,
       NLM_DEF_IT_LIMIT, NLM_DEF_PREC_THRLD, 10.0, &el, velocity_mps);

    //         
    let vi_result = vincenty_inverse(actual_target_lat_rad, actual_target_lon_rad, 
       tdoa_3d_result.0.to_radians(), tdoa_3d_result.1.to_radians(),
       &el, VNC_DEF_EPSILON, VNC_DEF_IT_LIMIT);        

    assert!(vi_result.0 < start_dst_projection_m * 0.01, "Estimated location is farer than limit (1%): {}", vi_result.0);
    assert_approx_eq!(tdoa_3d_result.2, actual_target_z_m, start_dst_projection_m * 0.05);
        
    assert!(tdoa_3d_result.3 < start_dst_projection_m * 0.01, "Residual function greater than limit (1%): {}", tdoa_3d_result.3);
    assert!(tdoa_3d_result.4 < NLM_DEF_IT_LIMIT, "Method did not converge: iterations limit exeeded {}", tdoa_3d_result.4);
}
