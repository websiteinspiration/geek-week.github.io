Apache Arrowを使用した列データのストリーミング

記事の翻訳は、データエンジニアコースの学生向けに特別に準備されました。



過去数週間にわたって、Nong Liと私は既存のランダムアクセス/ IPCファイルフォーマットを補完するバイナリストリーミングフォーマットをApache Arrowに追加しました。JavaとC ++およびPythonバインディングの実装があります。この記事では、フォーマットがどのように機能するかを説明し、DataFrame pandasで非常に高いデータスループットを実現する方法を示します。

列データのストリーミング


Arrowのユーザーからよく寄せられる質問は、表形式のデータの大きなセットを行指向または行指向の形式から列形式に移動するための高コストの問題です。マルチギガバイトのデータセットの場合、メモリまたはディスクでの転置は、非常に困難な作業です。

ストリーミングデータの場合、ソースデータが文字列であるか列であるかに関係なく、1つのオプションは行の小さなパケットを送信することです。

Apache Arrowでは、テーブルチャンクを表すメモリ内の列配列のコレクションをレコードバッチと呼びます。論理テーブルの単一のデータ構造を表すために、いくつかのレコードパッケージを収集できます。

既存の「ランダムアクセス」ファイル形式では、テーブルレイアウトとファイルの最後にあるブロックの場所を含むメタデータを記録します。これにより、データセットからレコードパッケージまたは列を非常に安価に選択できます。ストリーミング形式では、一連のメッセージを送信します。スキームと、1つ以上のレコードのパケットです。

PyArrowストリーミング：アプリケーション


これがどのように機能するかを示すために、単一のストリームチャンクを表すサンプルデータセットを作成します。

import time
import numpy as np
import pandas as pd
import pyarrow as pa

def generate_data(total_size, ncols):
    nrows = int(total_size / ncols / np.dtype('float64').itemsize)
    return pd.DataFrame({
        'c' + str(i): np.random.randn(nrows)
        for i in range(ncols)
    })

ここで、それぞれ1 MBのチャンクで構成される1 GBのデータを記録し、合計で1024のチャンクを記録するとします。まず、16列の最初の1 MBデータフレームを作成します。

KILOBYTE = 1 << 10
MEGABYTE = KILOBYTE * KILOBYTE
DATA_SIZE = 1024 * MEGABYTE
NCOLS = 16

df = generate_data(MEGABYTE, NCOLS)

次に、それらをpyarrow.RecordBatch次のように変換します。

batch = pa.RecordBatch.from_pandas(df)

次に、RAMに書き込み、作成する出力ストリームを作成しますStreamWriter。

sink = pa.InMemoryOutputStream()
stream_writer = pa.StreamWriter(sink, batch.schema)

次に、1024のチャンクを書き込み、最終的に1GBのデータセットを構成します。

for i in range(DATA_SIZE // MEGABYTE):
    stream_writer.write_batch(batch)

RAMに書き込んだので、1つのバッファーでストリーム全体を取得できます。

In [13]: source = sink.get_result()

ストリーミングパフォーマンス


ストリーミングチャンクのサイズが小さくなると、キャッシュアクセススキームが非効率になるため、パンダで連続したDataFrame列を再構築するコストが増加します。また、C ++データ構造と配列、およびそれらのメモリバッファを操作することによるオーバーヘッドもあります。

上記のように1 MBの場合、私のラップトップ（クアッドコアXeon E3-1505M）では、次のようになります。

パフォーマンスが256Kから64Kチャンクに大幅に低下します。1 MBのチャンクが16 MBよりも速く処理されたことに驚きました。より綿密な調査を行い、これが正規分布であるのか、それとも何か他のものがそれに影響を与えるのかを理解することは価値があります。

現在の形式の実装では、データは原則的に圧縮されないため、メモリとワイヤのサイズはほぼ同じです。将来的には、圧縮が追加オプションになる可能性があります。

合計


列データのストリーミングは、小さなチャンクを使用して、大きなデータセットをパンダなどの列分析ツールに転送する効果的な方法です。ライン指向のストレージを使用するデータサービスは、プロセッサーのL2およびL3キャッシュにとってより便利な小さなデータチャンクを転送および転置できます。

完全なコード

import time
import numpy as np
import pandas as pd
import pyarrow as pa

def generate_data(total_size, ncols):
    nrows = total_size / ncols / np.dtype('float64').itemsize
    return pd.DataFrame({
        'c' + str(i): np.random.randn(nrows)
        for i in range(ncols)
    })

KILOBYTE = 1 << 10
MEGABYTE = KILOBYTE * KILOBYTE
DATA_SIZE = 1024 * MEGABYTE
NCOLS = 16

def get_timing(f, niter):
    start = time.clock_gettime(time.CLOCK_REALTIME)
    for i in range(niter):
        f()
    return (time.clock_gettime(time.CLOCK_REALTIME) - start) / NITER

def read_as_dataframe(klass, source):
    reader = klass(source)
    table = reader.read_all()
    return table.to_pandas()
NITER = 5
results = []

CHUNKSIZES = [16 * KILOBYTE, 64 * KILOBYTE, 256 * KILOBYTE, MEGABYTE, 16 * MEGABYTE]

for chunksize in CHUNKSIZES:
    nchunks = DATA_SIZE // chunksize
    batch = pa.RecordBatch.from_pandas(generate_data(chunksize, NCOLS))

    sink = pa.InMemoryOutputStream()
    stream_writer = pa.StreamWriter(sink, batch.schema)

    for i in range(nchunks):
        stream_writer.write_batch(batch)

    source = sink.get_result()

    elapsed = get_timing(lambda: read_as_dataframe(pa.StreamReader, source), NITER)

    result = (chunksize, elapsed)
    print(result)
    results.append(result)
