すべてのReact開発者が知っておくべき3つの一般的なセキュリティミス

本日の翻訳を公開している記事の著者は、インタラクティブインターフェイスを作成するためのお気に入りのライブラリはReactであると述べています。Reactは使いやすく、十分に保護されています。ただし、これは、Reactアプリケーションが完全に脆弱であるという意味ではありません。プロジェクトがReactを使用しているため、XSS攻撃について心配する必要がないと判断したため、不合理な落ち着きに陥るのは非常に簡単です。

Reactの脆弱性は、ほとんどの場合、開発者がこのライブラリの保護メカニズムを使用していると考えたときに発生しますが、実際にはそうではありません。したがって、Reactの機能を正しく評価し、プログラマーが自分で解決する必要があるタスクを知ることが重要です。



今日は、典型的なReactの脆弱性、コードレビュー中にそれらを見つける方法、およびそれらを防御する方法について説明します。

クロスサイトスクリプティング（XSS）は、深刻な問題につながる可能性があるクライアントの脆弱性です。 XSS攻撃は、攻撃者がWebサイトをだましてユーザーのブラウザーで任意のJavaScriptコードを実行させることができる場合に発生します。

反映されたXSS攻撃は、ブラウザによってコードの形式で処理されるテキスト情報を含むリンクによって実行されます。たとえば、これはクライアント側で特別な要求テキストが入力されるフォームフィールドです。

ストアドXSS攻撃は、攻撃者がサーバーにアクセスできる状況であり、サーバーで実行されたコードがクライアントのWebページに到達するものを生成する場合です。このような攻撃の典型的な方法は、コメントと画像をサーバーにアップロードすることです。

脆弱なWebサイトは、ユーザーをパスワードや個人データの盗難にさらす可能性があります。そして、これは他の脆弱性を利用する通常の方法です。悪意のあるスクリプトは、スパムを送信したり、ユーザーを詐欺サイトにリダイレクトしたりするために最もよく使用されます。これにより、攻撃に成功したサイトの評判とSEOパフォーマンスが損なわれる可能性があります。

脆弱性No. 1：サーバーのレンダリング中に使用されるページの初期状態の制御


場合によっては、ページの初期状態を形成するときに、危険なため、JSON文字列に基づいてドキュメントを作成することがあります。このコードの脆弱性は次のようになります。

<script>window.__STATE__ = ${JSON.stringify({ data })}</script>

これは、メソッドがJSON.stringify()何も「考える」ことなく、ページに表示される文字列形式で有効なJSONデータである限り、それに提供されたデータを変換するという事実により危険です。あなたがいる場合{ data }フィールド、ユーザ名やユーザ情報として信頼できないユーザ缶編集、それを持っている、あなたは、これらのフィールドに次のようなものを埋め込むことができます。

{ username: "pwned", bio: "</script><script>alert('XSS Vulnerability!')</script>" }

このパターンは、Reduxを使用するReactアプリケーションのサーバー側レンダリングでよく使用されます。彼はReduxの公式ドキュメントに参加し、その結果、GitHubにある多くのチュートリアルとサンプルアプリケーションテンプレートが引き続き使用されています。

脆弱性№2：陰湿なリンク


タグには、サイトの別のページ、別のサイト、現在のページのある場所へのリンクを含む<a>属性hrefを含めることができます。リンクには、次のようなスクリプトを含めることができますjavascript: stuff()。このHTML機能について知らなかった場合は、次のコードをブラウザーの行にコピーして、今すぐ試してください。

data:text/html, <a href="javascript: alert('hello from javascript!')" >click me</a>

Web開発者にとってこれは、ユーザーが入力したデータに基づいてリンクのコンテンツが設定されている場合、攻撃者はで始まる悪意のあるコードを追加できることを意味しjavascript:ます。次に、ユーザーが不正なリンクをクリックすると、攻撃者スクリプトがブラウザで起動されます。

この脆弱性は間違いなくReactの特徴であるだけでなく、対応する値が自動的に正しくエスケープされることを期待するときにReact開発者がしばしば直面する問題の1つです。Reactの将来のバージョンでは、この問題はそれほど深刻ではなくなることに注意してください。

codeコードレビュー中の脆弱性の特定


プロジェクトユーザーは、他のユーザーがクリックできるページへのリンクを追加できますか？その場合は、次のようにページに「リンク」を追加してみてください。

リンクをクリックしたときに対応するメッセージボックスが表示された場合、これはプロジェクトがXSS攻撃に対して脆弱であることを意味します。リンクを追加できる場所ならどこでもこれを試してください。このような場所のすべてが脆弱であるとは限りません。
