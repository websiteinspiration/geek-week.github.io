開発者向けのローカリゼーションルール



最初にそうしますが、次に考えます-またはその逆？  Alconost  は、ゲーム開発段階で検討するローカリゼーション機能について説明します。

この記事は、開発者から繰り返し寄せられる質問に対する回答として作成しました。「私のゲームの何が問題になっていますか？翻訳が十分ではないのはなぜですか？どうすれば修正できますか？」

ゲームの開発を開始すると、多くの人がすべてのリソースを節約しようとします。これは、時間の経過とともに頭脳を開発する予定がない場合でも効果的です。

しかし、ほとんどの開発者は、ローカル市場でゲームをリリースした直後に、他の国からより多くのゲーマーを引き付ける方法について考え始めます。そして、ローカリゼーションの必要があります。

ここで、ローカライゼーションは、インターフェイスの適応された翻訳を意味しますが、その要素の変更を意味しないことを強調しておく必要があります。

たとえば、ドイツ語に翻訳されたゲームに英語の民間伝承の登場人物に関するジョークがある場合、ローカライズすると、そのジョークはドイツ語で理解されるものに置き換えられます。しかし、ドイツ語のテキストがインターフェースに適合しない場合、この問題の解決策はローカリゼーションの範囲を超えます。

ローカリゼーションのもう1つの例は、数字を含むテキストの翻訳です。 en-US（アメリカ英語）などの一部の言語標準では、数字ではなく、単語で記述する必要がある場合があります。他の言語では、数字は特定の形の単数形および複数形の名詞に対応している必要があります。たとえば、ロシア語には1つよりもはるかに多くのオプションがあり、英語に値する価値があります。そして、日本語と中国語では、複数形はまったくありません。まあ、ゲーム内の数字とテキストが静止画像として修正されている場合、テキストを翻訳するだけでは十分ではありません。

ユーモアと数字を含むこれらの2つの例は、氷山の一角にすぎません。翻訳中に発生する可能性のあるローカリゼーション以外の問題が他にもたくさんあります。これらは、疑似ローカリゼーションまたは国際化エラーと呼ばれることがあります。

一般に、ローカリゼーションに先行する技術的なタスクは、国際化に関連しています（つまり、「i18n」）。これらは、原則として、回避できたはずの技術的な問題であり、その修正には開発者の努力が必要になる場合があります。

このようなエラーを予測するために、Alconostはルールのリストを作成することを決定しました。開発の最初からそれらをフォローする場合は、ローカリゼーションのためにゲームを準備するために、不必要な努力をせずに静かに行います。

1.事前にローカライズ言語を選択する


実際にゲームを起動する前に、可能なすべての翻訳言語を予測することは不可能であると主張することができます。これは部分的に当てはまります。最初にローカライズされていないバージョンでオーディエンスをテストし、次にローカライズによってリーチを拡大します。しかし、このルールは普遍的ではありません。

まず第一に、ゲームには非常に多くの文化的および地域的なタブーが存在する可能性があり、ローカリゼーションなしでは、ゲームがどんなに刺激的であっても、そのような地域の観客を引き付けることは不可能です。

2.可能な言語用のインターフェースを準備する


通常、「小さなマージンで」インターフェース要素を作成して、そのサイズがソーステキストに必要なものより少なくとも30％大きくなるようにすることをお勧めします。これは、メニュー項目などの短い行に特に当てはまります。

しかし、私たちには別のアイデアがあります。ルール1から始めて、すでに言語のリストをコンパイルしている場合は、最も複雑な（ローカライズに最もボリュームのある）言語標準のインターフェースを開発します。

理解のために、ドイツ語のテキストは英語、ロシア語、アラビア語のテキストより平均で30％長くなります-約10％長くなります。しかし、繁体字中国語の文字は、英語のテキストより30％少ないスペースを占めます。

3.コードにテキスト文字列を埋め込まない


コードから取得できない行はローカライズできません。ローカライズされた文字列は、コードを編集せずに編集する必要があることに注意してください。

もう1つの重要なヒント：単一の単語またはフラグメントからテキストを作成しないようにしてください。そのようなエラーの良い例は、Google開発者によってStackExchangeプラットフォームで共有されていました：

String currency = Locale :: getCurrencyString（）+ money.toString（）; //「$ 123」を与える

この例では、「$ 123」の形式の文字列を取得しますが、他のいくつかの言語では、通貨記号は数字の後にあるはずです。

したがって、ローカライズ可能なフォーマット文字列を使用することをお勧めします。例：

String format = Locale :: get（ "currency format"）; //英語で「$ {0}」を返します

文字列通貨=文字列::フォーマット（format、money.toString（））;

後者のオプションを使用すると、翻訳者は文字列内の単語を並べ替えることができます。

4.時間、日付、単位、数値もローカライズする必要があることに注意してください。


数値情報も翻訳に利用できる必要があることを前のルールに追加する必要があります-コードに残すことはできません。

インターフェイスで数値の形式を変更する準備をしてください。たとえば、ゲームの時間をカウントする時計も、おそらくローカライズする必要があります。ここでのポイントは、西側諸国の文化は基本的に単時性である、つまりそこにいる人々は時間を時間軸として、そしてアジア諸国では円として表現しているということです。

また、日付の形式と測定単位が言語によって異なることは言うまでもありません。

したがって、ローカライズする前に、これらのニュアンスを準備して考慮することをお勧めします。

5.変数とフォーマット設定ツールを使用して、それらにアクセスできるようにする


テキストをローカライズおよび編集するとき、変数を使用することは、ハードコーディングに代わる優れた方法のように見えることがあります。しかし、翻訳者に自由裁量で変数を使用する機会を与えない場合、これはあなたに反するかもしれません。

問題は、異なる言語の語句の順序が異なることです。したがって、ここで推奨するのは、変数をフレーズの一部にして、適切なコンテキストに挿入できるようにすることです。正しい例と正しくない例を次に示します。

間違った例：「ママが食べた」+％数+「りんご」。
正しい例：「ママはリンゴを％食べました。」

変数を簡単に説明すると便利です。これにより、変数と前のテキストのフラグメントとの混乱や誤った相関（または非相関）が回避されます。

6.テキストを画像に縫い付けないでください


ゲームに画像がある場合は、ローカライズする準備をしてください。特に、テキストが多い場合は、グラフィックを最初からやり直す必要があります。

多くの場合、グラフィックリソースの編集は不必要ではないため、その色と記号は選択された領域の標準に対応しています。しかし、翻訳されたテキストを挿入するために画像がやり直されると、これは時間と労力の無駄になります。

7.正しいフォントとエンコーディングを使用する


文字列クラスに適合しない「特殊」文字を使用すると、エンコードの問題が発生します。また、ローカライズ後にターゲット製品のエンコードが正しくない場合、文字を削除するのに多くの時間と労力がかかるようです。

フォントについても同様です。一部のゲームフォントには、すべての言語のグリフがありません。その結果、特定の言語バージョンでは、他のフォントを選択する必要があります。忘れないでください。そうしないと、字幕テキストの代わりに□□□のような文字がたくさん表示される危険があります。

ASCIIではなくUnicodeを使用することをお勧めします。UTF-8が最も一般的で最適なエンコーディングです。

一般に、ソースファイルのエンコーディングに注意してください。ここでは詳細には触れません。このトピックの包括的なガイドは、「krakozyabry」の狩猟に関する英語の記事。

8.ローカリゼーションの準備はできていますか？疑似翻訳を試す


そのため、技術的なニュアンスが考慮されます。テストのローカライズの時間です。テキストの長さの変更、エンコーディングのチェック、ハードコードされた文字列の検索など、外国語のインターフェースを模倣する多くの優れた疑似ローカリゼーションツールがあります。

基本的に、ターゲット言語をシミュレートしてビルドを生成するスクリプトを実行します。ビルドは、テスト中に、ローカライズされていない元のビルドとしてチェックされます。

このような予備テストは万能薬ではありませんが、役立ちます。また、テキストの代わりに「abracadabra」を使用して新しいインターフェースをテストするのはとても楽しいです。

9.事前に用語集の編集を開始します。


用語集は、同じ方法で翻訳する必要があるゲーム内の用語と概念のコレクションです。これらは主にオブジェクト、状態の名前、ステータス、アーティファクト、キャラクター名です。

用語集を使用して、ゲーム内で一貫した用語を維持することが重要です。あるアイテムが1つの場所で「ポーション」として、別の場所で「エリクサー」として翻訳されたと想像してください。

10.前後関係を示す準備をする


翻訳プロセスでは、ローカリゼーションチームにコンテキストを与えることが重要です。Alconostでの経験から判断すると、翻訳者、ローカリゼーションマネージャー、開発者の間で確立されたコミュニケーションは、このコンテキストを決定するのに役立ちます。

すべての開発者が24時間体制で連絡を取ることは不可能であり、必要もありません。そのため、翻訳管理プラットフォームを介してリアルタイムで翻訳者と連絡を取る担当者を選択することをお勧めします。

通常、このようなプラットフォームはお客様が選択します。したがって、ローカリゼーションチームとの仮想コミュニケーションは便利で生産的です。まあ、それは正当化されます。なぜなら、心で行われた仕事は確かに報われるからです。
