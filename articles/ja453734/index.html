ヘルムの歴史


誕生


Helm 1は、Deisが作成したオープンソースプロジェクトとして始まりました。私たちは、2017年の春にマイクロソフトに吸収された小さなスタートアップでした。Deisとも呼ばれる他のオープンソースプロジェクトにdeisctlは、艦隊クラスターに Deisプラットフォームをインストールして操作するために（とりわけ）使用されるツールがありました。フリートは、当時の最初のコンテナオーケストレーションプラットフォームの1つでした。

2015年半ばに、コースを変更することを決定し、Deis（当時の名前はDeis Workflowに変更）をFleetからKubernetesに転送しました。最初の1つは、再設計されたインストールツールですdeisctl。これを使用して、艦隊クラスターにDeisワークフローをインストールおよび管理しました。

Helm 1は、Homebrew、apt、yumなどの有名なパッケージマネージャーのイメージで作成されました。その主なタスクは、Kubernetesでのアプリケーションのパッケージ化やインストールなどのタスクを簡素化することでした。 Helmは、2015年にサンフランシスコのKubeConカンファレンスで正式に紹介されました。

Helmでの最初の試みは成功しましたが、深刻な制限がありました。彼は一連のKubernetesマニフェストを取り、ジェネレーターによって前付けのYAMLブロック* としてフレーバーし、結果をKubernetesにアップロードしました。

* 注意ペレフ。：Helmの最初のバージョンから、Kubernetesリソースを記述するためにYAML構文が選択され、構成を記述する際にJinjaテンプレートとPythonスクリプトがサポートされました。これとHelmの最初のバージョンのデバイスについては、この資料の「簡単なHelmの歴史」の章で詳しく説明しました。

たとえば、YAMLファイルのフィールドを置き換えるには、マニフェストに次の構成を追加する必要がありました。

#helm:generate sed -i -e s|ubuntu-debootstrap|fluffy-bunny| my/pod.yaml

今日テンプレートエンジンがあるのは素晴らしいですね。

多くの理由により、この初期のKubernetesインストーラーはハードコードされたマニフェストファイルのリストを必要とし、小さな固定された一連のイベントのみを実行しました。Deis Workflow R＆Dチームが製品をこのプラットフォームに転送しようとするときに苦労するほど使用するのは大変でしたが、アイデアの種はすでに播種されていました。私たちの最初の試みは素晴らしい学習の機会でした。私たちは、ユーザーの日常的な問題を解決する実用的なツールを作成することに真摯に取り組んでいることに気付きました。

過去の失敗の経験に基づいて、Helm 2の開発を開始しました。

クリエーションヘルム2


2015年の終わりに、Googleチームから連絡がありました。彼らはKubernetesの同様のツールに取り組みました。Kubernetes用のDeployment Managerは、Google Cloud Platformに使用されていた既存のツールの移植です。「私たちは、類似点と相違点について話し合うために数日を費やすだろうか」と尋ねました。

2016年1月、HelmチームとDeployment Managerチームがシアトルで会合し、アイデアを交換しました。DEISとGoogleと一緒ヘルム2を作成するために、両方のプロジェクトを組み合わせて、より男：交渉は野心的な計画で終わっSkippBox - （約TRANSL Bitnamiの今一部。。）開発チームに参加し、我々はヘルム2に仕事を始めた

これらの目標を達成するために、2番目の要素がHelmエコシステムに追加されました。このクラスター内コンポーネントはTillerと呼ばれ、Helmチャートのインストールとその管理に関与していました。

2016年にHelm 2がリリースされて以来、Kubernetesはいくつかの大きな革新を遂げてきました。役割ベースのアクセス制御（RBAC）が導入され、最終的には属性ベースのアクセス制御（ABAC）が置き換えられました。新しいタイプのリソースが導入されました（その時点でのデプロイメントはまだベータステータスのままでした）。カスタムリソース定義（元々はサードパーティリソースまたはTPRと呼ばれていました）が発明されました。そして最も重要なのは、一連のベストプラクティスが現れたことです。

これらすべての変更の中で、HelmはKubernetesユーザーに忠実にサービスを提供し続けました。3年と多くの新しい追加を行った後、コードベースに大幅な変更を加えて、Helmが開発中のエコシステムの増大するニーズに対応し続けることができることが明らかになりました。

ティラーへの穏やかな別れ


Helm 2の開発中に、GoogleのDeployment Managerとの統合の一部としてTillerを導入しました。Tillerは、共通のクラスター内で作業するチームにとって重要な役割を果たしました。これにより、インフラストラクチャを運用するさまざまな専門家が同じリリースのセットとやり取りできるようになりました。

役割ベースのアクセス制御（RBAC）がKubernetes 1.6でデフォルトで有効になっていたため、本番環境でTillerを操作することがより困難になりました。考えられるセキュリティポリシーの数が非常に多いため、デフォルトではアクセス許可を提案するという立場にあります。これにより、初心者は最初にセキュリティ設定に飛び込む必要なしにHelmとKubernetesを試すことができました。残念ながら、この寛容な構成は、ユーザーに必要のない過度に広い範囲の権限を与える可能性があります。 DevOpsとSREのエンジニアは、マルチテナントクラスターにTillerをインストールして、追加の運用手順を学ぶ必要がありました。

特定の状況でコミュニティメンバーがどのようにHelmを使用するかを学習した後、Tillerのリリース管理システムは状態を維持したりリリース情報を備えた中央ハブとして機能するためにクラスター内コンポーネントに依存する必要がないことに気付きました。代わりに、Kubernetes APIサーバーから情報を取得し、クライアント側のグラフを生成して、インストールレコードをKubernetesに保存できます。

Tillerの主なタスクはTillerなしで実行できるため、Helm 3に関する最初の決定の1つはTillerの完全な拒否でした。

Tillerの辞任により、Helmセキュリティモデルが大幅に簡素化されました。 Helm 3は、今日のKubernetesの最新のセキュリティ、識別、および認証方法をすべてサポートするようになりました。Helmの権限は、ファイルkubeconfig。クラスタ管理者は、あらゆるレベルの細分度でユーザー権限を制限できます。リリースは引き続きクラスター内に保存され、残りのHelm機能は保持されます。

リリース管理


Helm 3では、アプリケーションの状態は2つのオブジェクトによってクラスター内で監視されます。

  • releaseオブジェクト-アプリケーションのインスタンスを表します。
  • release version secret — (, ).

呼び出しhelm installにより、リリースオブジェクトとリリースバージョンシークレットが作成されます。呼び出しにhelm upgradeはリリースオブジェクト（変更可能）が必要であり、新しい値と準備されたマニフェストを含む新しいリリースバージョンシークレットを作成します。

リリースオブジェクトにはリリース情報が含まれています。リリースは、名前付きチャートと値の特定のインストールです。このオブジェクトは、リリースに関するトップレベルのメタデータを記述します。リリースオブジェクトは、アプリケーションのライフサイクル全体を通じて保持され、Helmチャートによって直接作成されたすべてのオブジェクトと同様に、すべてのリリースバージョンシークレットの所有者として機能します。

リリースバージョンシークレットは、リリースを一連のリビジョン（インストール、更新、ロールバック、アンインストール）に関連付けます。

Helm 2では、改訂は非常に一貫していた。コールhelm install作成されたv1、その後の更新（アップグレード）-v2など。リリースとリリースバージョンのシークレットは、リビジョンと呼ばれる単一のエンティティにまとめられました。リビジョンはTillerと同じ名前空間に格納されていました。つまり、各リリースは名前空間に関して「グローバル」でした。その結果、名前のインスタンスは1つしか使用できませんでした。

Helm 3では、各リリースは1つ以上のリリースバージョンのシークレットに関連付けられています。リリースオブジェクトは常に、Kubernetesにデプロイされている現在のリリースを示します。各リリースバージョンシークレットは、このリリースの1つのバージョンのみを示します。たとえば、アップグレードでは新しいリリースバージョンのシークレットが作成され、リリースオブジェクトがこの新しいバージョンを指すように変更されます。ロールバックが発生した場合は、以前のリリースバージョンのシークレットを使用して、リリースを以前の状態にロールバックできます。

Tillerを放棄した後、Helm 3はリリースと共に単一の名前空間にリリースデータを格納します。このような変更により、同じ名前のチャートを別の名前空間にインストールでき、データはクラスターの更新/ etcdの再起動の間に保存されます。たとえば、Wordpressをネームスペース「foo」にインストールしてからネームスペース「bar」にインストールすると、両方のリリースを「wordpress」と呼ぶことができます。

グラフの依存関係の変更

ライブラリチャートの紹介


Helm 3は、ライブラリチャートと呼ばれるチャートクラスをサポートしています。このチャートは他のチャートで使用されますが、それ自体ではリリースアーティファクトを作成しません。ライブラリチャートテンプレートはアイテムのみを宣言できますdefine。他のコンテンツは単に無視されます。これにより、ユーザーは多くのグラフで使用できるコードフラグメントを再利用および共有できるため、重複を避け、DRYの原則を守ることができます。

ライブラリチャートはdependencies、ファイルのセクションで宣言されますChart.yaml。それらのインストールと管理は、他のチャートと同じです。

dependencies:
  - name: mylib
    version: 1.x.x
    repository: quay.io

このコンポーネントがチャート開発者に提供するユースケースと、ライブラリチャートから生じる可能性のあるベストプラクティスを楽しみにしています。

次は何ですか？


Helm 3.0.0-alpha.1-Helmの新しいバージョンを作成するための基礎。この記事では、Helm 3のいくつかの興味深い機能について説明しました。それらの多くはまだ開発の初期段階にあり、これは正常です。アルファ版リリースの本質は、アイデアをテストし、最初のユーザーからフィードバックを収集し、想定を確認することです。

アルファバージョンがリリースされるとすぐに（これは既に発生していることを思い出してください-約Transl。）、コミュニティからHelm 3のパッチの受信を開始します。新しい機能の開発と採用を可能にする強固な基盤を作成する必要があり、ユーザーはプロセスに関与していることを感じ、チケットを開き、修正を行うことができます。

この記事では、Helm 3に現れるいくつかの深刻な改善点を強調してみましたが、このリストは決して完全なものではありません。Helm 3の本格的な計画には、更新戦略の改善、OCIレジストリとのより深い統合、チャート値をチェックするためのJSONスキームの使用などの革新が含まれます。また、コードベースをクリアして、過去3年間無視されてきたコード部分を更新する予定です。
