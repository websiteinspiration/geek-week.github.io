Android上のWebRTC：複数のデバイスでハードウェアエンコーディングを有効にする方法

Badooでのビデオ通話には、WebRTC標準とH.264コーデックを使用します。ドキュメントが信頼できる場合、このコーデックはAndroid 5.0以降のすべてのAndroidデバイスで問題なく動作するはずです。しかし実際には、すべてがそうではないことがわかりました。この記事では、WebRTCでH.264コーデックにハードウェアエンコーディングを実装する機能と、それをより多くのデバイスで動作させる方法について説明します。



H.264を選ぶ理由


WebRTC経由で接続すると、セッションに参加しているすべてのデバイスが、ビデオコーデックやオーディオコーデックなどのさまざまな通信パラメーターを送信します。デバイスが複数のコーデック（たとえば、VP8とH.264）をサポートしている場合、プラットフォームの優先コーデックが最初に表示されます。このデータはWebRTC の調整段階で使用され、その後、すべてのデバイスでサポートされているコーデックのみが残ります。復号化を使用したこのようなデータの例は、このドキュメントに記載されています。

ビデオコールの場合、一方のデバイスがH.264コーデックをサポートしていない場合、両方のデバイスは、たとえば、デバイスのハードウェア実装に依存しないVP8コーデックに切り替えることができます。しかし、私たちのアプリケーションは、前世代のスマートフォンを含むさまざまなガジェットで利用できます。したがって、ビデオ通信では、可能な限りハードウェアエンコーディングを使用したいと考えました。これにより、プロセッサの負荷が軽減され、バッテリーをあまり消費しなくなります。これは、古いガジェットにとって重要です。H.264ハードウェアエンコーディングのサポートは、同じVP8とは異なり、多数のデバイスに実装されています。

AndroidでのH.264サポート


マルチメディア形式 のサポートの説明が信頼できる場合、H.264ベースラインプロファイルのデコードは、Android 3.0以降のすべてのAndroidデバイスとエンコードで機能するはずです。 Badooでは、Android 5.0以降のデバイスをサポートしているため、問題は発生しません。しかし、それほど単純ではありませんでした。5番目のバージョンのガジェットであっても、多数の機能が見つかりました。

接続できるものは何ですか？

ご存知のように、Androidで新しいデバイスを開発する場合、どのメーカーも互換性テストスイートに合格する必要があります。デバイスに接続されたPCで実行され、その結果をGoogleに送信して、デバイスが指定されたバージョンのAndroid OSの要件を満たしていることを確認する必要があります。その後、ガジェットを市場にリリースできます。

WebRTCでのH.264サポート


HardwareVideoEncoderFactory から始めましょう。

話名isHardwareSupportedInCurrentSdkH264を持つメソッドがあります。

private boolean isHardwareSupportedInCurrentSdkH264(MediaCodecInfo info) {
  // First, H264 hardware might perform poorly on this model.
  if (H264_HW_EXCEPTION_MODELS.contains(Build.MODEL)) {
    return false;
  }
  String name = info.getName();
  // QCOM H264 encoder is supported in KITKAT or later.
  return (name.startsWith(QCOM_PREFIX) && Build.VERSION.SDK_INT >= Build.VERSION_CODES.KITKAT)
      // Exynos H264 encoder is supported in LOLLIPOP or later.
      || (name.startsWith(EXYNOS_PREFIX)
             && Build.VERSION.SDK_INT >= Build.VERSION_CODES.LOLLIPOP);
}

ご覧のとおり、Androidでのハードウェアエンコーディングのサポートは、QualcommおよびExynosチップセットに対してのみ実装されています。標準のWebRTC実装で他のチップセットがサポートされていないのはなぜですか？おそらく、これはハードウェアコーデックメーカーの実装の特殊性によるものです。また、特定のデバイスを常に見つけることができるとは限らないため、本番環境でのみこれらの機能を特定できる場合がよくあります。

デバイス上のすべてのコーデックの説明は、media_codecs.xmlファイルに保存されます。たとえば、ここでは、Pixel XLとHUAWEI P8 lite用のこのファイルです。 MediaCodecListオブジェクトのgetCodecInfos（）メソッドを使用してコーデックのリストを取得すると、このファイルが解析され、そこに格納されているコーデックが返されます。この操作と、製造元によるこのファイルの正しい入力は、CTSテストでカバーされています。MediaCodecListTestもAndroid 4.3の160行からAndroid 10の740行に増加し

ました。Badooでは、isHardwareSupportedInCurrentSdkH264メソッドコードを変更し、「ホワイト」コーデックリストを拒否して、WebRTCにリストされているプログラムコーデックプレフィックスの「ブラック」リストに置き換えました。

static final String[] SOFTWARE_IMPLEMENTATION_PREFIXES = {"OMX.google.", "OMX.SEC."};

しかし、それを利用して、製造元の特殊性に注意を払わずにすべてのコーデックのサポートを実装することはできません。Discussion-webrtcグループの Androidでのハードウェアコーディングに関するトピックの名前から、この場合は間違いなくエラーが発生することがわかります。基本的に、それらはコーデックの設定段階で表示されます。

コーデック構成オプション


エンコーディング用のコーデックの初期化は次のとおりです。

MediaCodec mediaCodec = createByCodecName(codecName);
MediaFormat format = MediaFormat.createVideoFormat(mime, width, height);
format.setInteger(MediaFormat.KEY_BIT_RATE, targetBitrateBps);
format.setInteger(MediaFormat.KEY_BITRATE_MODE, VIDEO_ControlRateConstant);
format.setInteger(MediaFormat.KEY_COLOR_FORMAT, colorFormat);
format.setInteger(MediaFormat.KEY_FRAME_RATE, targetFps);
format.setInteger(MediaFormat.KEY_I_FRAME_INTERVAL, keyFrameIntervalSec);
mediaCodec.configure(format, null, null, MediaCodec.CONFIGURE_FLAG_ENCODE);

これらのパラメーターのいくつかでミスを犯しやすいのは、コーデックを構成するときに例外をスローし、アプリケーションを混乱させることです。また、コーデックを使用する場合、コーデック自体が間違っているため、さまざまな要因に応じてビットレートを調整する必要がある場合があります。このため、WebRTCでは、BaseBitrateAdjusterクラスが責任を持ち、2つの子孫があります。

  • DynamicBitrateAdjuster-データの量に応じてビットレートを調整します
  • FramerateBitrateAdjuster-フレームレートに応じてビットレートを調整します。

したがって、コーデックごとに独自のビットレート調整メカニズムを選択する必要があります。ハードウェアコーデックの初期化パラメーターを設定する機能をさらに詳しく検討してみましょう。

ストリーム解像度


コーデックのMediaCodecInfoオブジェクトを受け取った後、CodecCapabilitiesクラスでその機能を取得することにより、コーデックをより詳細に調べることができます。それらから、コーデックが選択した解像度とフレームレートをサポートしているかどうかを確認できます。これらのパラメーターをサポートしている場合は、安全に設定できます。

ただし、このルールが機能しない場合があります。接頭辞が「OMX.MARVELL」のコーデックという事実に直面しています。ストリームの解像度が4：3と異なる場合、画面の端に緑のストライプが表示され、正しくエンコードされません。同時に、コーデック自体は、選択された解像度とフレームレートがサポートされていると主張しました。

ビットレートモード


すべてのビデオコーデックの標準モードは、固定ビットレートです。ただし、可変ビットレートを使用する必要があった場合：

format.setInteger(MediaFormat.KEY_BITRATE_MODE, VIDEO_ControlRateVariable);

これは、プレフィックスが「OMX.sprd。」で始まるSpreadtrumチップセット（現在はUnisoc）を搭載したLenovo A1000デバイスで発生しました。インターネットでの検索により、この問題とその解決方法を説明する6年前のFirefox OSに関する投稿がありました。

カラーフォーマット


コーデックをバイトバッファモードで使用する場合は、正しい形式を選択する必要があります。これは通常、次の形式の関数を使用して行われます。

@Nullable
static Integer selectColorFormat(int[] supportedColorFormats, CodecCapabilities capabilities) {
  for (int supportedColorFormat : supportedColorFormats) {
    for (int codecColorFormat : capabilities.colorFormats) {
      if (codecColorFormat == supportedColorFormat) {
        return codecColorFormat;
      }
    }
  }
  return null;
}

大まかに言えば、サポートされているカラーフォーマットの最初のものが常に選択されます。

ただし、プレフィックス「OMX.IMG.TOPAZ。」、「OMX.hisi」で始まるHUAWEIコーデックの場合。と「OMX.k3。」は機能しませんでした。長い検索の結果、解決策が見つかりました。これらのコーデックが返す形式に関係なく、COLOR_FormatYUV420SemiPlanar形式を使用する必要があります。このスレッドは、1つの中国のフォーラムでこれを理解するのに役立ちました。

ビットレート調整


標準のWebRTCコードには以下が含まれます。

private BitrateAdjuster createBitrateAdjuster(VideoCodecMimeType type, String codecName) {
  if (codecName.startsWith(EXYNOS_PREFIX)) {
    if (type == VideoCodecMimeType.VP8) {
        // Exynos VP8 encoders need dynamic bitrate adjustment.
        return new DynamicBitrateAdjuster();
      } else {
        // Exynos VP9 and H264 encoders need framerate-based bitrate adjustment.
        return new FramerateBitrateAdjuster();
    }
  }
  // Other codecs don't need bitrate adjustment.
  return new BaseBitrateAdjuster();
}

このコードからわかるように、Exynosを除くすべてのチップセットでは、ビットレート調整がオフになっています。ただし、ExynosとQualcommのみが標準コードでサポートされているため、これはQualcommにのみ適用されます。この設定のさまざまな値を試し、インターネットも検索したところ、「OMX.MTK」というプレフィックスが付いたコーデックが見つかりました。それもオンにする必要があります。プレフィックス「OMX.IMG.TOPAZ。」、「OMX.hisi」で始まるHUAWEIコーデックについても、これを行う必要があります。または「OMX.k3」。これは、コーデックを構成するときにすべてのフレームに同じ周波数セットが設定されていると想定して、これらのコーデックがビットレートを調整するためにフレームのタイムスタンプを使用しないためです。

結論として、Android 5.0および5.1のデバイス用に受け取ったコーデックのリストを示します。新しいバージョンのAndroidでは状況が改善し、非標準のコーデックが小さくなっているので、私たちは主にそれらに興味を持っていました。
