GRASPテンプレート：作成者

こんにちは、ハブル。私の名前はウラジスラフロダンです。現在、私はOTUSのHigh Load Architectコースの責任者であり、ソフトウェアアーキテクチャのコースも教えています。

特に「建築とデザインパターン」コースの新しいセットの開始のために、私はあなたに喜んで共有する小さな資料を書きました。


前書き


クレイグ・ラーマンの著書 『UMLとパターンの適用、第3版』に記載されているGRASPパターンは、GoFパターンの一般化であり、OOP原則の直接的な結果です。これらは、論理ラダーで欠落しているステップを補完します。これにより、OOPの原理からGoFパターンを取得できます。GRASPテンプレートは、（GoFのような）デザインパターンではなく、クラス間で責任を分散するための基本原則です。実際にはそれほど人気が​​ないことが示されていますが、GRASPパターンのフルセットを使用して設計されたクラスを分析することは、優れたコードを記述するための前提条件です。

GRASPテンプレートの完全なリストは、9つの要素で構成されています。

  • 情報専門家
  • クリエーター
  • コントローラ
  • 低カップリング
  • 高い凝集力
  • ポリモーフィズム
  • 純粋な製造
  • 間接
  • 保護されたバリエーション

前回、情報エキスパートの原則について説明しました。今、私はそれに似たクリエイターを考えることを提案します。

クリエーター


言い回し


このパターンは、以前のパターンと同じ典型的な問題を解決します。それらを必要とするクラスは、クラスのインスタンスを作成する必要があります。

違反の例


注文と商品について同じ問題を考えます。記述したコードがInformation Expertに準拠していると仮定します。

@Setter
@Getter
@AllArgsConstructor
public class Order {
    private List<OrderItem> orderItems;
    private String destinationAddress;
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }
}

@Setter
@Getter
@AllArgsConstructor
public class OrderItem {
    private Good good;
    private int amount;

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

私たちが研究している原則は一見平凡なものですが、一部のクライアントコードでは、これを見つけることができます。

public class Client {
    public void doSmth() {
        Good good = new Good("name", 2);
        OrderItem orderItem = new OrderItem(good, amount);
        List<OrderItem> orderItems = new ArrayList<>();
        orderItems.add(orderItem);
        Order order = new Order(orderItems, "abc");
        // client code 
    }
}

UMLクラス図を作成すると、ClientクラスがOrderクラスとそのすべての内部（OrderItemとGood）に依存していることがわかります。したがって、クライアントが必要としない上記のクラスがなければ、クライアントクラスを再利用することはできません。Clientクラスがすべてのオブジェクトを作成したため、情報エキスパートに準拠するためのあらゆる努力の結果を実際に無効にしました。

レガシープロジェクトでは、あるクラスが別のクラスのオブジェクトを作成し、このオブジェクトが使用されていない5〜6個のクラスを介してメソッドのパラメーターとして転送する方法をよく見ることができます。これは、最初からいくつかの依存関係を追加することに他なりません。

応用例


クラス間の責任の分布を調整して、分布が情報エキスパートだけでなく作成者も満たすようにします。

@Setter
@Getter
public class Order {
    private List<OrderItem> orderItems = new ArrayList<>();
    private String destinationAddress;
    
    public Order(String destinationAddress) {
        this.destinationAddress = destinationAddress;
    }
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }

    public void addOrderItem(int amount, String name, int price) {
       orderItems.add(new OrderItem(amount, name, price));
   }
}

@Setter
@Getter
public class OrderItem {
    private Good good;
    private int amount;

    public OrderItem(int amount, String name, int price) {
        this.amount = amount;
        this.good = new Good(name, price);
    }

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

これで、クラス間の依存関係の数は最小限になります。クライアントのコードは多少簡略化されており、次のようになります。

public class Client {
    public void doSmth() {
        Order order = new Order("address");
        order.addOrderItem(amount, name, price);
        // client code 
    }
}
