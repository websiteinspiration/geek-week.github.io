C / C ++での簡単なレポート

C / C ++で病院、ショップ、ジャイロスクーターのレンタルサービス、携帯電話の修理サービスのオペレーター（管理者）向けのソフトウェアを開発している場合、レポートを作成してプリンターで印刷する、または少なくともPDFでレポートを作成する作業に直面することでしょう。 Embarcadero RAD Studioには、これを可能にする多くのサードパーティパッケージがあります。 FastReport、QuickReport、Crystal Reportsなど。しかし、私の意見では、これらのパッケージはすべて、そのメカニズムとインターフェースを深く理解する必要があります。それらを研究するにはかなりの時間がかかります。最も重要なのは、最も単純なレポートでさえ生成するコードが膨大な数の行で構成されることです。 C / C ++でレポートを生成するタスクに最初に出会ったとき、私はFastReportから始めて、このツールがまったく好きではないことに気付きました。

その瞬間、すばらしい考えが浮かびました。単純なEXCELファイルを取得し、静的な情報を追加して、ニーズに合わせてフォーマットするのがいかに素晴らしいかです。プログラムはこのファイルのみを開き、動的なデータを入力して、保存するか印刷して送信します。これは、Embarcadero RAD Studioで開発されたプログラムからMS Officeファイルを操作するためのOLEメカニズムを研究するための出発点となりました。

私の活動の過程で、このトピックを深く掘り下げることができましたが、今では、MS OfficeとOLEが提供するすべてのツールが私のすべてのレポートニーズをカバーしていると自信を持って言えます。そうだとすれば、おそらく他の開発者にとっては、それは自給自足のシンプルで便利なツールでしょう。したがって、DLLを作成し、Excelドキュメントの作成中に遭遇する一般的に使用されるすべてのMS Excel関数のリストをそのDLLにパックすることが決定されました。レポートを作成するために、一連のマニュアルやOLEを学習する必要がないため、これは非常に便利です。必要なのは、DLLとその関数をロードすることだけです。まあ、多くはDLLを扱うことができます。

こちらがプロジェクトのサイトです。

プロジェクトサイトでは、DLLを操作するためのメカニズムが十分に詳しく説明されており、すべての関数の例と説明があります。

DLLには、ヘッダーファイルと静的LIBライブラリがありません。したがって、DLLはLoadLibrary関数を使用して動的に接続する必要があります。ライブラリを含むZIPアーカイブには、次のファイルが含まれています。

  • " light_report.dll " -DLLライブラリ自体。
  • " DLLTest.cpp "-ライブラリの使用例。
  • " LPDLL.h "-ライブラリの使用例のヘッダーファイル（これはヘッダーDLLファイルではありません）。
  • 「Report.xlsx」-ライブラリの使用例については、MS EXCELドキュメント。

LPDLL.h ライブラリの使用例のヘッダーファイルには、次の宣言が含まれています。

  • DLL関数への引数として使用される列挙型。
  • DLL関数のタイプ。
  • DLL関数インスタンス
  • DLLダウンロード記述子（ " HINSTANCE DLL_Handle; "）;
  • 機能LoadLightReportDLL動的にDLLとそのすべての機能をロードし、。
  • DLLをアンロードするFreeLightReportDLL関数。

「LPDLL.h」 ファイルには、DLLとその機能を操作するために必要なすべてのものがすでに含まれています。しかし、それを編集したり、プロジェクトに最も必要なものだけを取ることもできます。

ヘッダーファイル「LPDLL.h」をプロジェクトに接続したところを想像してみてください。

次に、プロジェクト自体で次のことを行うだけです

。1）レポートの変数ハンドルを宣言します。

Variant report;

2）DLLとそのすべての機能を動的にロードします。

if(!LoadLightReportDLL("C:\\LightReport\\light_report.dll"))return;

3）次に、try-catch構成で、準備されたレポートテンプレート（MS Excelファイル）を開きます。

report=OpenReport("C:\\LightReport\\Report.xlsx",0);

4）レポートにデータを追加します。

WriteCell(report, "Sheet1", 9, 1, "Hello world!");

...
5）ファイルを保存または印刷します。

Save(report);
SaveAs(report, "C:\\LightReport\\Report copy.xlsx");
ExportToPDF(report, "C:\\LightReport\\Report.pdf", false);
PrintOut(report);

6）ファイルを閉じます。

CloseReport(report);

7）DLLライブラリをアンロードします。

FreeLightReportDLL();

以上です！絶対に何もない！注意点は1つだけですが、それでもデバッグ中にのみ表示されます。実行中のアプリケーションでは、すべてがうまく機能します。レポートを操作するときに何らかのエラーが発生する状況を意味します。事実、ドキュメントを操作するためのOLEメカニズムには、エラーが発生した場合にのみ例外をスローすることが含まれます。そのため、DLL関数を使用する場合は、try-catch構文を使用する必要があります。 catchブロックでは、保存せずにレポートを閉じる必要があります。

catch(...){CloseReport(report);}
。
デバッグ時にエラーが発生した場合は、プログラムを停止できます。同時に、MS Excelプロセスは実行されたままになり、タスクマネージャーを通じてのみ閉じることができます。したがって、アプリケーションを複数回デバッグする場合、エラーが発生したときにプログラムを一時停止する頻度に応じて、コードがキャッチで実行されるのを待たずに、MS Excelプロセスの複数のインスタンスが起動されることがあります。これは監視する必要があります。

このリリースでは、エラーが発生した場合、catchブロックで指定されたコードは確実に機能し、レポートが閉じ、MS Excelプロセスが完了します。システムでハングしているMS Excelプロセスは観察されません。しかし、それでも彼らはエラーのないコードを書こうとしているので、実行中のアプリケーションでこの状況が発生しないことを願っています。
