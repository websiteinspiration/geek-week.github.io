ネクラソフ氏は、「頭の中で気まぐれになると、彼女を打ち負かすことはできません」と語り、「ロシアに住むことができる人」と語った。

ですから、その考えが頭に浮かんだので、しばらくして、実装のための最初のアイデアが浮かびました。すべてが単純に見えました-meet.company.com/user/confidリンクへの呼び出しを傍受する必要があります-Webアプリケーションのローカルプロセスを車の127.0.0.1に配置し、localhostを指す会議にログインするための/ etc / hostsを介して、会社のドメインの静的エントリを追加します。さらに、このWebサーバーは、そこに到達したリンクを処理し、どういうわけかそれをピジンに転送する必要があります（この段階では、どのようにして彼に渡すかまったくわからなかったとすぐに言います）。もちろん、解決策は松葉杖のようなにおいがしますが、私たちはプログラマーです。松葉杖は私たちを怖がらせません（きしむ）。

その後、偶然にも私はどういうわけかGoogle Chromeへの招待へのリンクを開きました（通常、私は常にMozilla Firefoxを使用しています）。そして驚いたことに、Webページは完全に異なって見えました-ユーザーデータ入力フォームがなく、ページに入った直後にxdg-openを介して何かを開く要求がありました。興味があるため、「はい」をクリックするとエラーメッセージが表示されます-リンクlync15：confjoin？Url = https：//meet.company.com/user/confidを開くことができません。うーんこれはどのようなxdg-openであり、そのようなリンクを開くには何が必要ですか？ドキュメントを読んだ剖検は、これが、uriスキームのプロトコルまたは特定のタイプのファイルのいずれかを使用して、関連するアプリケーションを起動するのに役立つグラフィカルシェルハンドラーであることを示しました。関連付けは、MIMEタイプの関連付けを通じて構成されます。したがって、lync15という名前のuriスキームの一致するアプリケーションの検索を開始していることがわかります。リンクはxdg-openに渡されます。これはさらに考えて、このタイプのリンクを担当するアプリケーションにリンクを転送する必要があります。もちろん、システムにはありません。そうでない場合、彼らはオープンソースの世界でどのように行動しますか？そうです、私たちはそれを自分で書きます。

Linuxの世界、特にグラフィカルシェル（デスクトップ環境、DE）がどのように機能するかの研究において、Linux MintでこのXfceを使用していることから、アプリケーションとそれに関連付けられているMIMEタイプは通常、拡張子が付いたショートカットファイルに直接書き込まれることがわかりました。 。デスクトップ。さて、なぜそうしないのか、単純なアプリケーションショートカットを作成します。これは、bashスクリプトを起動し、それに渡された引数をコンソールに出力するだけです。ショートカットファイル自体のみを指定します。

xdg-openをブラウザーから起動し、ブラウザーからのリンクと同じリンクを使用して...残念です。再び彼は彼がリンクを処理することができないと言います。

結局のところ、関連するMIMEタイプのディレクトリをアプリケーションで更新していません。これは簡単なコマンドで行われます：

xdg-mime default lync.desktop x-scheme-handler/lync15

〜/ .config / mimeapps.listファイルを 編集するだけです。

xdg-openを呼び出して2番目の試行を試みてください-そして再び失敗します。何も、困難が私たちを怖がらせることはありませんが、関心を煽るだけです。そして、bashのすべての能力（つまり、トレース）を武器に、デバッグに真剣に取り組みます。ここで、xdg-openは単なるシェルスクリプトであることに注意してください。

bash -x xdg-open $url

トレース後に出力を分析すると、さらに制御がexo-openに移されていることが少しわかります。そして、これは既にバイナリファイルであり、引数にリンクを渡すときに失敗した戻りコードを返す理由を理解することは、すでにより困難です。

xdg-openの内部を実行したところ、さまざまな環境パラメーターを分析し、特定のDEに固有のファイル/リンクを開くためのいくつかのツールに制御を渡すか、open_generic関数の形式でフォールバックしていることがわかりました

open_xfce()
{
if exo-open --help 2>/dev/null 1>&2; then
exo-open "$1"
elif gio help open 2>/dev/null 1>&2; then
gio open "$1"
elif gvfs-open --help 2>/dev/null 1>&2; then
gvfs-open "$1"
else
open_generic "$1"
fi

if [ $? -eq 0 ]; then
exit_success
else
exit_failure_operation_failed
fi
}

渡された引数の分析を使用して、ここで小さなハックをすばやく切り取り、特定のサブストリングlync15がそこにある場合は、すぐにopen_generic関数に制御を渡します。

3番を試してみて、それがうまくいったと思いますか？ええ、今どのように。しかし、エラーメッセージは既に変更されており、これはすでに進行中です。ファイルが見つからないこと、そして引数として渡されたリンクそのものをファイルの形式で書いたことを彼は私に伝えました。

今回は、is_file_url_or_path関数であることが判明しました、これはファイルfileへのリンクを分析します：//ファイルへのパスまたは何か他の入力のいずれかに送信されます。また、プレフィックス（URLスキーム）に数字が含まれているため、チェックは正しく機能しませんでした。正規表現は、アルファ：ドットとダッシュで構成される文字セットに対してのみチェックされます。統一リソース識別子の rfc3986標準を参照した後今回はマイクロソフトが何も壊さなかったことが明らかになりました（私はそのようなバージョンを持っていましたが）。単なる文字クラス：alpha：ラテンアルファベットの文字のみが含まれます。定期チェックをすぐに英数字に変更します。完了です。あなたは楽しいです。すべてがようやく起動し、すべてのチェックがスクリプトアプリケーションに渡された後で制御されます。リンクがコンソールに表示されます。その後、exo-openのすべての問題は、回線内の番号によるリンクフォーマットの検証が原因であると疑い始めます。仮説をテストするために、アプリケーションのMIMEタイプ登録をlyncスキームと出来上がりだけに変更します。open_xfce関数を再定義しなくても、すべてが機能します。しかし、会議に参加するためのWebページがlync15とのリンクを正確に作成するため、これは私たちには何の助けにもなりません。

これで、パスの最初の部分が完成しました。リンク呼び出しをインターセプトしてから、どういうわけかそれを処理してPidgin内に渡す必要があります。「会議への参加」メニューのリンクを使用してデータを入力するときに内部でどのように機能するかを理解するために、Sipeプロジェクトリポジトリのクローンを作成し、コードに再び飛び込む準備をしました。しかし、ここで、幸いにも、私はcontrib / dbus /ディレクトリのスクリプトに惹かれました。

ステップ2.自動結合のハンドラーを実装する


真珠には既製の例があるので、私はsipe-join-conference-with-uri.plを使用して、自分用に少し変更することにしました。パールで書けるので特に問題はありませんでした。

スクリプトを個別にテストした後、その呼び出しをlync.desktopファイルに入力しました。そしてそれは勝利でした！会議に参加するページに入り、xdg-openの起動を許可すると、Pidginからの会議のポップアップウィンドウが自動的に開きました。私が喜んだ方法。
この成功に勇気づけられて、私はメインブラウザーのMozilla Firefoxでも同じことをすることにしました。キツネを介して入ると、承認用のページが開き、一番下にオフィスのコミュニケーターを使用した参加ボタンがあります。彼女は私の注意を引いた。ブラウザでそれをクリックすると、リンクは次の場所に移動します。

conf:sip:{user};gruu;opaque=app:conf:focus:id:{conf-id}%3Frequired-media=audio

彼はそれを開く方法がわからない、そしておそらく、そのようなプロトコルに関連するアプリケーションを持っていないと彼に親切に教えてくれました。さて、私たちはすでに合格しました。

スクリプトアプリケーションをconf uri-schemeにもすばやく登録しても、何も起こりません。ブラウザは私のリンクを処理するアプリケーションがないと不平を言い続けます。同時に、パラメーターを指定したxdg-openコンソールからの呼び出しは正常に機能します。

「Firefoxでカスタムプロトコルハンドラーを設定する」-この質問で私はオンラインに行きました。stackoverflowに関するいくつかの議論の後（そしてそれがない場合）、答えが見つかったようです。about：configに特別なパラメータを作成する必要があります（もちろんfooをconfに置き換えます）：

network.protocol-handler.expose.foo = false

リンクを作成して開き、そこにありませんでした。ブラウザーは、何も起こらなかったかのように、アプリケーションを認識していないと言います。

Mozillaへのプロトコルの登録に関する公式ドキュメントを読みました。関連付けをgnomeデスクトップ自体に登録するオプションがあります（もちろんfooをconfに置き換えます）。

gconftool-2 -s /desktop/gnome/url-handlers/foo/command '/path/to/app %s' --type String
gconftool-2 -s /desktop/gnome/url-handlers/foo/enabled --type Boolean true

登録してブラウザを開くと、あごひげが再び。
