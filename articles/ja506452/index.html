ESNIで公開サイトを保護する方法

こんにちはHabr、私の名前はIlyaです。私はExnessプラットフォームチームで働いています。製品開発チームを使用する基本的なインフラストラクチャコンポーネントを開発して実装します。

この記事では、公開Webサイトのインフラストラクチャに暗号化SNI（ESNI）テクノロジを実装した経験を紹介します。



このテクノロジーを使用すると、公開Webサイトで作業するときのセキュリティレベルが向上し、会社が採用した内部セキュリティ基準に準拠します。

まず、テクノロジーが標準化されておらずドラフト段階にあることに注意したいと思いますが、CloudFlareとMozillaはすでにそれをサポートしています（draft01）。これは私たちにそのような実験への動機を与えました。

理論のビット


ESNIはTLS 1.3プロトコルの拡張であり、「Client Hello」TLSハンドシェイクメッセージでSNIを暗号化できます。ESNIサポート



 を使用したClient Helloは次のようになります（通常のSNIの代わりに、ESNIが表示されます）。ESNIを使用するには、3つのものが必要です。

  • DNS 
  • 顧客サポート
  • サーバー側のサポート。

DNS


AとTXTの 2つのDNSレコードを追加する必要があります（TXTレコードには、クライアントがSNIを暗号化できる公開キーが含まれています）。以下を参照してください。さらに、利用可能なクライアント（下記参照）はDoHなしではESNIサポートをアクティブ化しないため、DoHサポート（DNS over HTTPS）が必要です。ESNIはアクセスしているリソースの名前の暗号化を意味するため、これは論理的です。つまり、UDPを介してDNSにアクセスしても意味がありません。さらに、DNSSECを使用すると、このシナリオでのキャッシュポイズニング攻撃から保護されます。

現在、いくつかのDoHプロバイダーが利用可能です。

  • クラウドフレア
  • Opendns
  • グーグル

CloudFlare は、サーバーがESNIをサポートするようになったことを確認します（ブラウザーを確認→暗号化されたSNI→詳細）。つまり、DNSのCloudFlareサーバーの場合、少なくとも2つのレコード（AとTXT）があります。以下の例では、我々は、GoogleのDNS（オーバーHTTPS）を要求：

ESNIはTLS 1.3の拡張であるため、もちろん、TLS 1.3を使用してESNIをサポートする必要があります。

ESNIサポートを使用してバックエンドをテストするために、クライアントをオンに実装しましたが、後で詳しく説明します。

サーバー側のサポート


ESNIはOpenSSL / BoringSSLを介してTLSで動作するため、ESNIは公式にはサポートされていないため、現在nginx / apacheなどのWebサーバーではサポートされていません。

したがって、ESNIを使用したTLS 1.3の終了をサポートし、ESNIをサポートしないアップストリームへのHTTP（S）トラフィックをプロキシする独自のフロントエンドコンポーネント（ESNIリバースプロキシ）を作成することにしました。これにより、基本コンポーネントを変更することなく、既存のインフラストラクチャにテクノロジーを適用できます。つまり、ESNIをサポートしていない現在のWebサーバーを使用できます。

私は、ESNIなしでクライアントをサポートするために、ESNIなしでTLS接続を終了する機能を備えたプロキシが考案されたことに注意します。また、アップストリームと通信するためのプロトコルは、1.3より前のTLSバージョンのHTTPまたはHTTPSのいずれかです（アップストリームが1.3をサポートしていない場合）。この設計は最大の柔軟性を提供します。私たちは、借りた

上ESNIサポートの実装を、外出先からのCloudFlare。すぐに、標準の暗号化/ tlsライブラリの変更を意味し、したがってアセンブリの前にGOROOTの「パッチ」が必要になるため、実装自体は非常に重要です。

ESNIキーを生成するために、esnitool（これもCloudFlareの発案によるもの）を使用しました。これらのキーは、SNIの暗号化/復号化に使用されます。
Linux（Debian、Alpine）とMacOSでgo 1.13を使用してビルドをテストしました。 

運用機能について一言


ESNIリバースプロキシは、RPS、アップストリームレイテンシと応答コード、失敗/成功したTLSハンドシェイク、TLSハンドシェイクの継続時間など、Prometheus形式のメトリックを提供します。一見すると、これはプロキシがトラフィックを処理する方法を評価するには十分であるように見えました。
