スマートホーム用センサー

ここで家の制御盤を作り、ここでコントローラーを開発しました。

ライトは電話からオン/オフすることができ、漏水に関する通知、散水が含まれていることが電報になっても神はまだ何を知っています。

しかし、このシステム全体はスマートハウスとはほど遠く、腕と脚の延長にすぎず、唯一の助けは、ベッドから出て電気を消す必要がないことです。

この記事はあなたの家をスマートに近づけることができるものについてです、そしてあなたはそれを正しく推測しました-これはセンサーの開発です。これらはあなたの家のための目、耳、そしてもう少しです。センサーからの情報に基づいて、家を活性化し、静かにあなたを助け、あなたの人生をより快適にする自動スクリプトを書くことはすでに可能です。

現在、市場には、著名な製造元からの、または出所が不明な、そのようなさまざまなデバイスがあります。そして、それほど気にしない場合は、それらを使用することができますが、1つだけ、またはいくつかでもあります。まず、それらのほとんどは独自のプロトコルを使用し、ハブでのみ動作します。また、これらのハブのAPIはいつでも閉じられる可能性があり、制御は製造元の適用を通じてのみ可能になるため（Googleのネストの話を思い出してください）、すべての自動化が一時的にクラッシュします。

次に、すべてのメーカーが例外なくこれから何らかの形で利益を得ようとしています。これは大規模な現象ではないため、外に出る必要があるため、各センサーは独自のバッテリーを備えた独自のケースで作られています。数えてみましょう。各部屋が5部屋あります。温度、動き、光、COセンサーが必要です。複雑なマットではありません。計算すると、天井に20個のボックスができます。まず、これは美しくありません。次に、1〜2年で、はしごとドライバーを使用してこれらのセンサーのバッテリーを半日で交換する必要があります。最も可能性が高いのは、いくつかの留め具を壊し、ドライバーをラミネートに数回落とすことです。はい、そして3番目に、2、3年でいずれかのセンサーが故障すると、同じセンサーを購入できなくなります。ほとんどの場合、古いハブは新しいセンサーをサポートせず（利益は重要です）、すべてのセンサーを変更してシステムに再統合する必要があります。どちらの方法をとるかは、オープンソースから何かを見るか、自分で行うかです。少なくとも、コードは常にあなたと一緒にあります。したがって、説明に照らして、センサーを備えたボックスの開発は、それほどやりすぎのタスクにはなりません。

だから私のセンサーを開発するとき、私は上記の「しかし」を避けようとしました：

  1. すべてのセンサーを1つのハウジングに集約
  2. 有線電源を使用します（ゼロから構築）
  3. 標準プロトコルに従ってインターフェイスを作成します

次に、そのようなデバイスがどのように機能するかについて説明します。

2つの可能なオプションがあり、それらは、デバイスを使用する環境、集中管理されたシステム、分散システム、または一般に独立したデバイスとしての環境に依存します。

最初のオプションはより簡単ですセンサーからの生データのみが必要であり、次にデバイスに問い合わせを行う中央コントローラーがそれらをどうするかを決定します。そのようなデバイスの例はオフィスの煙センサーです。たとえば、センサーからのデータが特定のしきい値を上回っていますが、中央コントローラーは武装モードがオンになっていることを認識しているため、アラームを発生させる必要があります。

2番目のオプションは、デバイス自体が現在のモードを認識し、センサーからの生データを変換し、ユーザーへの通知について決定を下す必要があることを前提としています。もちろん、このようなデバイスの開発には、プログラムの作成と使いやすさの両方の観点から、より多くの努力が必要です。

私は最初に2番目のオプションに従ってデバイスを作成することを計画し、mqttとmodbusの2つの通信チャネルを敷設しました。無線LANを介したMQTTなので、RS-485を介してmodbusで遊んだり、ホームコントロールシステムに接続したりできます。

ESP8266 luaライブラリーのおかげで、WIFI経由のMqttは難しくありません。アルゴリズムはかなり標準的です。

  1. 最初の起動時、またはWIFIに接続できない場合は、ESP8266からアクセスポイントを作成する必要があります
  2. WIFI mqtt ( )
  3. ESP8266
  4. mqtt WIFI

しかし、デバイスをmodbus経由でホームシステムに統合するという質問が出てきたとき、センサーからの生データを見逃し始め、最終的にすべての生データもmodbusテーブルに入れました。たとえば、武装解除モードのモーションセンサーからの不必要に見えるイベントは、あらゆる種類のスマートホーム機能にも使用できます。

modbusテーブルは次のようになりました：
Nタイプの説明
0。-r-センサータグの状態（1-アイドルモード/ 2-保護からアイドル/ 3-保護モード）

6. -r-明るい値（ADCからの生の値）
7. -r- これのアラームイベントの数ライフサイクル
8. -r-センサータグの再起動回数
9. -r- 未使用
10 -wr-アラームのアクティブ化（0-アイドルに切り替える/ 1-アラームに切り替える）
11. -wr-モーション感度（0-100％、は0を意味します-最も敏感です）
12. -wr-ガス濃度限界

さらに、いつものように、それはボードとソフトウェアの開発に帰着します。この場合、ボードの開発は私にとってより困難でした。アナログ部分（パワー、モーションセンサーとCOのバインディング）がありますが、私はあまり友好的ではありません。ソフトウェアについて書くことはほとんどありません。すべてが本当にシンプルだからです。それら。普通のだけでなく、ADCから読み取り値を読み取り、変換し、（すべてのチャネルにスライディングウィンドウ方式を使用して）処理し、テーブルに入れました。 PIRセンサーからの信号の温度補償を行うことは悪いことではありません。夏には、室温が+30度を超えると、センサーの応答が大幅に低下します。したがって、この記事では、ボード自体の開発について説明し、使いやすさに少し注意を払います。

マイクロコントローラーの選択は難しくありません。devboardsを使用してボックスにアクセスし、1つのボードを取得するだけです。しかし真剣に、私はマイクロコントローラーの多くのファミリの知識を誇ることができないので、私はSTM32、ATSAMおよびAVRに精通しています。私が後者を選択したのは、Arduino Leonardoが2、3枚横になっていて、MK2が昔からなくなっていたからです。 AVRは水晶の周波数で足を引っ張ることに言及しません。これは私にとって有用であり、周辺機器の範囲が広く、ADCがかなり高速であること、TQFPパッケージがあり、顕微鏡なしでタンバリンと一緒に踊ることができることは私にとって非常に重要です。マイクロコントローラーの選択が10分の問題である場合、センサーの選択は簡単な問題ではありません。まず、それらを接続する方法、信号を処理する方法、およびそれらが互いにどのように影響するかについて考える必要があります。Arduinoセットから一部のセンサーを取り出すことができ、それらが1ペニーの費用がかかる場合、他のセンサーを使用すれば、よりまれに、より長くいじってより多くの費用をかけることができます。さらに、私の選択基準も私の追加基準によって制限されていました。

  1. センサーは好きなように接続できますが、I2Cを介しては接続できません。このようにして、マイクロコントローラーにI2Cを確実に実装することができませんでした。このバスのサポートは、ハードウェアとソフトウェアの半分で実装されます（AVRのように）。たまたま機能し、1か月間は機能し、その後中断し、回復シーケンスは役に立たないため、ADCに接続されているアナログセンサーをできるだけ多く使用し、SPIを温度のみに使用することにしました（ちなみに、友情は強く、信頼性があります）。
  2. それらは十分に信頼でき、耐用年数中に特性を変更しませんでした（私は10年と定義しました）
  3. モーゼルなどからの注文を台無しにしないように、私たちの市場で入手可能でした。

すべての調査の結果、次のセットが得られました。

  1. PIRセンサー-D203S
  2. 光センサー-GL5516
  3. 温度センサー-LM95071
  4. COセンサー-TGS5141

そして、最初の3つのポジションがあれば、誰も質問をしないと思うのですが、TGS5141がすぐに必要な理由は明確ではありません。あなたはおそらく言うでしょう-「MQ-7タイプのarduinoセットからのペニーCOセンサーがあるので、なぜそんなに高価なものか」確かに、私は彼らと実験し、その結果、いくつかの理由でそれらを拒否しました。第1に、これは触媒センサーであり、温度が上昇するため、デバイスのケースの温度に大きく影響し、PIRセンサーからの信号の温度補償ができません。第2に、MQ-7の感度は、10〜1000ppmmと記載されているスペックでさえ、まだ十分とは言えませんが、実際にはもっと悲しく、センサーごとの安定性はありません。第三に、センサー自体の寿命は明確ではありません。触媒であるため、その特性は何年にもわたって大きく変化すると思います。そして今、電気化学TGS5141の利点について-それは熱くなりません、感度は0〜5000ppm（線形特性で）、10年の保証された耐用年数、小さいサイズ、およびその他多くのプラスです。軽薄なクラフトではなく、本当にやりがいのあることをしたいのであれば、選択は明白だと思います。

マイクロコントローラー、センサーを選択しました。次に、ユーザーに意図したことを伝える方法を説明します。前に書いたように、これには2つの通信チャネル、rs-485 modbusとWIFI経由のmqttがあります。私のsensorortegを個別のデバイスとして使用できるようにするので、重要なのは、デバイス自体、たとえば、LEDの状態の表示です。

コンセプトを決定したら、要素ベースの残りの部分に少し注意を払うことができます。私はADM2587E 485番目のドライバーを使用します。チップには、ドライバー自体だけでなく、RX、TX、およびDEのトランスガルバニック絶縁も含まれています。ここでは、WIFIを介したmqtt通信にESP-07を使用します。ここでは、コメントは不要です。怠惰なだけがESP8266で何かをしませんでした。ビジュアルディスプレイをさらにいじくり回す必要があったため、sk6812 RGBW LEDを使用することにしました。 AVRで判明したように、それらの管理はそれほど単純ではなく、タイミングはかなり厳しいため、元々計画されていなかったアセンブラコードの一部を追加する必要がありました。しかし、その結果はそれ自体正当であり、特にプレキシガラスリングの端にあるこれらのLEDで輝いている場合、それは有益で美しく判明しました。この兆候は、これですでに犬を食べていた従業員から私に提案されました（Sashaに感謝）。



明るさを少し試してみると、明るさの1つのレベルは私の決断ではないことが明らかになりました。日中は見えにくく、夜は非常に明るくなります。そのため、室内の照明に応じて、これらのLEDのグローの明るさを変更する必要がありました。さて、それだけです、あなたは指示を理解しました、とあなたは思いましたが、いいえ、私にとって最も難しいことは、それらのグローパターンとタイミングを発明することでした。

私はこのオプションで停止しました：

  • 青色が遅い-アイドル、Wi-Fi経由の通信のみ（modbusなし）
  • 青色の高速-アイドルモードからアラームモードに切り替え、Wi-Fiのみでの通信（Modbusなし）
  • 緑が遅い-アイドル（modbus通信はOK）
  • 緑の高速-アイドルからアラームモードに切り替えます（modbus通信はOK）
  • 赤い遅い-アラームモード
  • 赤く速く-モーションアラーム
  • 黄色の高速-ガス警報



その兆候を理解しました。次のセクションはパワーです。

先に述べたように、私は家の中で24Vの低電流を持っているので、モジュールの電力はそれだけです。今後については、最初のボードを作成した後、多くの栄養をやり直さなければならなかったと言います。次に、ステップダウンコンバーターの仕様、特に配線のニュアンスが記述されている場所を注意深く読む必要があると確信しました。調査に基づいて、私はステップダウンコンバーターMCP16311に立ち寄り、仕様に書かれていることを正確に行った後にのみ、非常にうるさいことがわかりました-うまくいきました。しかし、後で判明したように、これは最大の問題ではありませんでした。LEDの明るさを変更すると、電力低下が小さくなり、D203Sセンサーからの信号を損なうのに十分です。センサーのオペアンプ回路とステップダウンコンバーター回路の両方にコンデンサーを追加することで、この問題を克服することがわかりました。 （ESP8266を介して）mqttを使用してすべてをデバッグした後、私はrs-485を接続しましたが、ここでも障害が待っていて、D203Sセンサーチャネルのオペアンプの出力で再びゴミを見つけました。このゴミはrs-485の交換に明らかに対応していました。一般に、アナログ部分全体に追加のLDOコンバーターを使用してこれを克服することが判明し、幸福がもたらされました。

彼らは、基板をはんだ付けし、ケースを印刷し、すべてを組み立て、天井に吊り下げると、データがシステムに入ります。したがって、当然のことながら、このデータをスマートホームシステムに適用する方法を想像してみましょう。もちろん、アラームを電報に送信することもできます。最初の点を除いて、まだ何もしていませんが、次のスクリプトを投げました。

  • 階段に光が含まれている場合、この光がオンになっていない場合、部屋に動きがあった場合、さらに現在は暗く、夜間
  • 1階のシャッターが閉じている場合、すでに朝であり、1日で初めてリビングの動きが検出された場合は、1階のシャッターが自動的に開きます
  • COの濃度がいずれかのレベルを超えた場合は換気フラップを開き、COのレベルが低下した場合は換気フラップを閉じます（実際には暖炉のある部屋）。
  • 温度が一定の制限を超えた場合は換気フラップを開き、それ以外の場合はそれらを閉じます
  • シャッターが開いていて、夜に路上でモーションセンサーが機能していて、家が夜警の場合、1階のシャッターは自動的にシャットダウンします。
