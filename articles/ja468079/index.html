何度も節約してくれたGoogle Analyticsのカスタムディメンション

Google Analytics（GA）を介して分析を設定する場合、ユーザーパラメータまたはCD（カスタムディメンション）はしばしば無視されます。これは、その範囲を理解するのが困難なためです。CDがデータの収集と分析にどのように役立つか、およびCDがどのように構成されているかは正確には不明です。この記事では、多くのプロジェクトでAdventumで使用する主なCDとセットアッププロセスについて説明します。

ユーザーオプションとは


Googleアナリティクスでレポートを作成するために、「ソース」、「市」、「ブラウザ」、「オペレーティングシステム」など、200を超えるさまざまなパラメータと指標を利用できます。ただし、標準的なパラメータでは必ずしも分析に十分ではないため、Googleはユーザーが独自のパラメータを作成できるようにしています。CDの価値は、ユーザーのあらゆるヒットで転送できます。たとえば、カスタムパラメータを使用して、アダプティブデータのコンテキストでサイトのパフォーマンスを分析できます。

合計4つのCDアクションエリアがあります。

  • ヒット
  • セッション
  • ユーザー
  • 製品

例えば、ユーザー許可パラメーターが保管されているCDがあります（0-許可されていない、1-許可されています）。セッションの開始時には、すべてのユーザーは許可されていません。CDの範囲に応じて、このパラメーターが入力後に変更されるとどうなるかを見てみましょう。



  • Hitスコープを持つCDの場合、ユニットは、認証が発生したヒットの認証パラメーターに割り当てられます。
  • セッションのスコープを持つCDの場合、許可パラメーターは、イベントが発生したセッションのすべてのヒットで1になります（このセッションのCDの古い値を上書きします）。
  • スコープがUserのCDの場合、認証パラメータは、ログインが行われたセッションから開始して、すべてのユーザーセッションで1になります。

重量など、Googleアナリティクスの基本的なパラメータでサポートが規定されていない製品の特性を分析する必要がある場合があります。複数の製品に関する情報を1回のヒットで送信できるため、より詳細なスコープを追加する必要があります。「製品」スコープが使用されるのは、この細分性のためです。



CDセットアップ


ユーザーパラメータの設定は、GAでパラメータを作成することと、サイトからデータを送信することの2つのステップで構成されています。次に、CDのセットアップ手順について説明します。

Google AnalyticsでCDを作成する


  1. 「管理者」セクションに移動します
  2. [ユーザーパラメータ]セクションの[リソース設定]で、[特別なパラメータ]サブセクションを選択します。
  3. 「+特殊パラメータ」ボタンをクリックします
  4. フォームが開きます。

    パラメーターのスコープを選択します。

    • ヒット
    • セッション
    • ユーザー
    • 製品

    データの収集を開始し、CDがレポートに反映されるように、チェックマークを「アクティブ」の反対側のままにします。
CDの名前に関する推奨事項
ユーザーパラメータに名前を付けるための独自のシステムを開発しました。
各CDの最初に、スコープの記号を追加します。ヒットの場合はh_（たとえば、h_hitId）、セッションの場合はs_（たとえば、s_sessionId）です。これにより、カウンター設定を確認することなく、ユーザーパラメーター内を移動し、常にそのスコープを知ることができます。同じ名前でスコープが異なる2つのパラメーターを作成することが必要になる場合があります。たとえば、A / Bテストを実施し、それが正しく構成されているかどうかを確認したい場合、セッション中にテストバリアントを変更します。これを行うには、同じ値を持つ2つのパラメーターを作成します。名前+テストバージョンです。ユーザーはその中にアクセスしましたが、そのうちの1つ、2番目のセッションがヒットしています。この場合、提案された命名は非常に便利です。

Googleタグマネージャーの設定


サイトからの送信パラメータを設定するには2つの方法があります。

  • サイトのHTMLコードを介して直接。
  • Googleタグマネージャー（GTM）経由。

どちらの方法でも、Google AnalyticsにCDインデックスが必要です。これは、「特別な設定」セクションで確認できます。



サイトのHTMLコードを介して送信

するGoogle Analyticsトラッキングコードのリソースの設定をそこにcustom_mapを追加して更新します（コメントで両側で強調表示されているものを追加する必要があります）：

gtag('config', 'GA_MEASUREMENT_ID' ,
// 
{
  'custom_map': {'dimension<Index>': 'dimension_name'}
}
// 
);

ここで、GA_MEASUREMENT_IDはアナリティクスのカウンター識別子です。<Index>をGAのパラメーターのインデックスに置き換え、dimension_nameの代わりにその名前を指定します。

複数のユーザーパラメータがある場合は、'dimension <Index>'：'dimension_name'をそれぞれにコンマで指定する必要があります。

例えば：

gtag('config', 'UA-XXXXXXXXX', {
  'custom_map': {'dimension1': 'projectName',
                 'dimension2': 'hitId',
                 'dimension3': 'sessionId',
                 'dimension4': 'screenWindow'}
});

CDをページビューと共に送信するには、上記のコードの前に、これらのパラメーターを目的の値に割り当てる必要があります。

gtag('set', { 'dimension_name': 'dimension_value'});

ここで、dimension_nameは'custom_map'で指定したものと一致し、dimension_valueは CDに転送する値にする必要があります。

例えば：

gtag('set', {'projectName': 'shop',
             'hitId': '1121243'});

イベントと共にCD値を送信する場合は、コードを送信する必要があります。
gtag('event', 'action_value', {'event_category': 'category_value',
                               'event_label': 'label_value', 
                               'value': 'event_value',
                               'dimension_name': 'dimension_value'});

代わりに：

action_value -イベントアクション
category_value - イベントのカテゴリ
label_value -イベントラベル
event_value -イベント値（整数非負）
DIMENSION_NAME -パラメータ名、私たちは前のコードで指定されたことを
dimension_value -値CD

例：

gtag('event', 'clickButton', {'event_category': 'conversion',
                              'event_label': 'registration', 
                              'projectName': 'shop',
                              'hitId': '1121243'});

プロジェクトでは、2番目のオプションであるGTMによるチューニングを使用します。より詳しく考えてみましょう。

CDをヒットと共にGTMを介してGAに

送信するCDをヒット（ページビュー、イベント）とともにGAに送信することは、「Google Analytics設定」タイプの変数で構成されます。この変数をすでに構成している場合は、それを開きます。そうでない場合は、作成する必要があります。 [詳細設定]セクションの[特別な設定]サブセクションに移動します。ここでは、GAで以前に作成したすべてのユーザーパラメータを追加



する必要があります。追跡識別子で、GAカウンタ番号を指定する必要があります。 「インデックス」フィールドに、データを転送するGAからのCDの識別子を書き込みます。

「パラメーター値」フィールドでは、指定されたインデックスでCDに転送される値を指定する必要があります。この値は、次の2つの方法で指定できます。

  • すべてのイベントで同じになる場合は手動で。
  • このパラメーターの値が格納されている変数を使用します（その名前は二重中括弧で書かれています）。たとえば、{{SCREEN_WINDOW}}-「ブラウザウィンドウサイズ」。



このユーザーパラメータの設定では、次の点を考慮することが重要です。

  1. タグが最初に処理されるとき、GoogleユーザーID（clientID）を持つCookieはまだ作成されません。このパラメーターのCDへの転送を構成する場合、最初のページの表示でこれを行うことはできません。
  2. 異なるユーザーパラメータにデータを送信するときに同じ変数を使用する場合、この変数はユーザーパラメータごとに再計算されます。たとえば、タグ処理時間タグを2つの異なるユーザーパラメータに送信すると、数ミリ秒の値の違いに気付く場合があります。

customTaskを使用すると、これらの問題を回避できます。

customTaskの使用


customTaskは、優先度が最も高い関数であり、自分で定義するコードとロジックです。少し前に、GoogleはGTMでこの機能を使用する機会を与えました。

customTaskは、モデルオブジェクト（追跡カウンターのすべてのフィールドを含む）、Cookieなどからのデータを使用します。この機能の優先度は他のすべての優先度よりも高く、タグに働きかけます。これにより、ユーザーが最初にページを表示したときからのユーザーIDの送信を構成できます。

customTaskを使用してCDを構成するには、JS_customTaskという名前で「ネイティブJavaScriptコード」というタイプのGTM変数を作成します。



次に、Googleアナリティクスの設定変数の[設定するフィールド]セクションに作成したJS_customTaskを追加します。

  • 「フィールド名」にcustomTaskと書き込みます。
  • 「値」に、上で作成した変数の名前を書き込みます。すべての変数は二重中括弧で記述する必要があります。



GTMタグで定義できるcustomTaskは1つだけです。したがって、このメソッドで判別できるすべてのCDは、前に作成したJS_customTask変数に書き込まれます。

よく使われるCD


ユーザーID


仕事。カウンターは、セッション数の大幅な増加を記録しました。同時に、ユーザー数はわずかに増加しています。一部のユーザーからのボットトラフィックに関する仮説がありました。デフォルトでは、GAで個々のユーザーの特性を便利な形式で表示することはできません。

決定。 Google Analyticsは、サイトの各訪問者に一意の識別子を割り当てます-クライアントID。 Googleアナリティクスのインターフェースでは、「ユーザー統計」（ユーザーエクスプローラー）という1つのレポートでのみ確認できます。 1つの画面で1人のユーザーのみのアクションのシーケンスを確認でき、レポートに追加のパラメーターを追加できないため、これを使用するのは不便です。

理想的なソリューションは、標準またはユーザーレポートの個別のパラメーターとしてclientIDを表示する機能です。そして、これはGTMを使用して行うことができます。

テクニック設定。 ClientIdとして、GAのユーザーIDの数字のシーケンスを使用します（これは、_gaという名前のCookieに保存されます。たとえば、_ga = GA1.2.1111111111.111111111、ClientId = 111111111.111111111）。この値を表示するには、サイトページを右クリックして[コードの表示]を選択します。



次に、[アプリケーション]タブの左側のメニューで、[Cookie]とサイト名を選択し、表で_ga変数を探します。



Cookieに加えて、GAユーザーIDはモデルオブジェクトに保存され、.get（ 'clientId'）メソッドを使用して取得できます。 customTaskを使用してこの値を取得し、GAに渡します。 JS_customTaskという名前の変数に次のコードを追加します。

function() {
  var customDimensionIndex_client = 5;
  return function(model) {
    model.set('dimension' + customDimensionIndex_client, model.get('clientId'));
  }
}

変数customDimensionIndex_clientの等号の後に、GAのClientIdに対応するCDインデックスを指定する必要があります。このようにして、カウンターからクライアントIDを取得し、それをユーザーパラメーターに送信しました。

結果。個々のユーザーに関するレポートを作成したところ、一部のユーザーが実際に1,000セッションを行っていることがわかりました。すべてのセッションは直帰率が100％で、明らかに価値がありませんでした。これらのユーザーを除外するセグメントを設定し、不要な訪問なしにトラフィックのダイナミクスを分析することができました。

ヒットID


仕事。ユーザーがサイトでどのように行動するか、どのようなアクションをどのような順序で実行するかを確認する必要があります。これらのデータに基づいて、ビジネスを発展させるための仮説を立てることができます。

決定。各セッションでのユーザーのヒット数を使用して、独自のCDを作成します。ヒット数の多い順にソートすることで、ユーザーのあり方を見ることができます。

テクニック設定。便宜上、HitIdとして、ClientId +ヒット時間をタイムスタンプ形式で渡します。したがって、イベント、ページビュー、トランザクションごとに値があります。このパラメーターは、（ClientIdと同様に）customTaskを使用して構成できます。

1つのGTMタグで定義できるcustomTaskは1つだけです。したがって、前に作成した変数「Custom JavaScript」に、コメントで両側が強調表示されたコードを追加します。GAのh_HitIdという名前のCDインデックスは、変数customDimensionIndex_hitに格納されます。

function() {
  var customDimensionIndex_client = 5;

//   
  var customDimensionIndex_hit = 2;
  var now = new Date();
//   

  return function(model) {
    model.set('dimension' + customDimensionIndex_client, model.get('clientId'));

//    
    model.set('dimension' + customDimensionIndex_hit, model.get('clientId').concat('_', now.getTime()));
//   

  }
}

パラメータタイプがヒットしました。タイムスタンプ形式の時間も含まれているため、追加のパラメーターを使用せずに、すべてのユーザー/セッションヒットを実行順にソートできます。

まったく同じ設定を行う場合、GAでのみパラメーターのスコープを選択します-セッションの場合、ユーザーのセッションIDを格納するCDを取得します。

結果。2つのレポートを作成しました。

  • ユーザーID、ヒットID、イベントカテゴリ、イベントアクション、イベントラベル、追加。パラメーター。
  • ユーザーID、ヒットID、追加。パラメーター。

次に、このデータをGAからアンロードし、Excel（またはデータベース）に結合しました。行をClientId、次にHitIdで並べ替えると、サイトの各ユーザーのパスが取得されます。これでピボットテーブルを作成して視覚化できます。

A / Bテスト


仕事。同社はABテストを実施することを決めた。設定は、Googleオプティマイズを使用せずに開発者によって行われました。疑問が生じました。実験の効果を分析するために、テストオプションをGoogleアナリティクスに転送する方法。

決定。この場合、CDはGAでのテスト結果の比較に役立ちます。テストの識別子とユーザーが使用しているオプションで値を取るユーザーパラメーターを設定するだけで十分です。そのため、テストオプションに従ってオーディエンスをグループに分け、結果を比較できます。

テクニック設定。前に構成したものと同様に、GAでカスタムパラメータを作成します。 「スコープ」-ヒットとして選択するので、テストに入る前と後で、イベントをユーザーがコミットするイベントに分割できます。たとえば、h_AB-Testと呼びます。一度に複数のテストでデータをCDに転送することが可能です。テストの名前とバージョンにない記号（ "|"など）でそれらを区切ることができます。

自分でテストを行い、それに関するデータをCookieに保存する場合、「ネイティブJavaScriptコード」のような変数を抽出してそれらを抽出する必要があります。

function(){
  var cookie_name ='{{test_variant}}';
  var matches = document.cookie.match(new RegExp(
     "(?:^|; )" + cookie_name.replace(/([\.$?*|{}\(\)\[\]\\\/\+^])/g, '\\$1') + "=([^;]*)"
     ));
  return matches ? decodeURIComponent(matches[1]) : '(not set)';
}

ここで、{{test_variant}}の代わりに、テストの名前とバリアントが格納されるCookieの名前を指定する必要があります。GA設定変数に追加することを忘れないでください。

レポートにh_AB-Testパラメータを追加すると、セッションが属するテストバリアントがGoogleアナリティクスに表示されます。このパラメーターの値に応じて、ユーザーをセグメント化できます。

結果。条件h_AB-Testのテストケースと同じ数のセグメントを作成します。「name + test case」の値と一致する必要があります。レポートでは、関心のあるイベントを確認します。

完全なページアドレス


仕事。コンテキスト広告マネージャーがYandex.Directで広告キャンペーンを開始しました。数日後、彼はGAソースレポートを調べ、彼が作成した広告キャンペーンから発生したユーザーセッションの数は、Yandex.Directでの広告のクリックよりもはるかに少ないことを発見しました。

決定。 Google Analyticsには、広告を追跡するための必須パラメーター-utm_sourceがあります。そうである場合、その値は標準パラメーター「キャンペーンソース」に追加されます。同様に、Googleは他のUTMタグを使用して、対応するパラメーターに書き込みます。

すべてのUTMタグはサイトのページのレポートから自動的に削除され、そこで分析することはできません。この場合、utm_sourceラベルが設定されていなければ、Googleは見つかったすべてのUTMタグをページのレポートから削除するだけで、対応する組み込みパラメーターにその内容を記録しません。

広告が正しく構成されていない場合でも、すべてのタグを表示するために、パラメーターをページ全体のアドレスで構成します。それをh_fullPageと呼びましょう。

テクニック設定。「グローバル変数名」セクションで「JavaScript変数」タイプの変数を作成して、document.URLを指定します



。変数のスコープを設定する-ヒット。

結果。次のUTMタグのシーケンスが広告キャンペーン全体で使用されました：utm_medium、utm_source、utm_campaign。ページの完全なアドレスを調べたところ、広告に関して、アンパサンドがutm_mediumとutm_sourceの間でスキップされていることがわかりました。utm_sourceはすべての広告キャンペーンに必須のutmタグであり、Google Analyticsは特殊文字（？、＆）の後にそれを検出しなかったため、この広告のクリックはノーリファラーで記録されました。h_fullPageパラメータのおかげで、問題を検出し、広告パネルの設定で修正することができました。
