在React应用程序中实现TypeScript的12个技巧

TypeScript是现代前端开发的标准。根据JavaScript研究状况，TS显然是程序员感兴趣的。根据2019年的调查，近60％的受访者尝试过TS并继续使用它，而22％的人没有尝试并想学习。

本文是有关如何实现和改进TypeScript使用的提示的集合。技巧的前半部分是关于方法和基础结构的一般性提示。第二个是一些特别有用的语言功能。



我坚持使用React应用程序的上下文，但是一些技巧将与其他生态系统相关。

提示1：容错


正确的类型是对书面代码的信任和对重构的促进。为了让您的打字工作更有效，您需要严格地进行类型检查：

  1. 检查开发人员模式以立即查看错误。
  2. 为每个池请求将验证添加到CI。这将确保主分支始终是正确的类型。

编译ts文件的主要方法有两种：原始TSC和@ babel /预置类型脚本。babel方法很有趣：从.ts文件中剪切类型信息，这使程序集变得快速。这是介绍文章。

如果使用babel，则使用fork-ts-checker-webpack-plugin额外添加类型检查。在开发模式下，检查应异步进行，以免阻塞汇编程序且不干扰编写代码。相反，在池请求中，如果发现错误，则类型检查应停止构建。

秘诀2：后端合约


通过API本身与后端的通信意味着数据交换合同。将这些实体分成一个单独的层，创建单独的接口是很有用的。我们称它们为DTO- 数据传输对象。

为了实现这一层，需要一些协议。这是您可以在团队中实施的规则的示例：

  • 接口名称的后缀Dto：FooBarDto。
  • 位于一个特殊的文件夹中。
  • 它只能随API更改而更改。
  • 该接口的数据仅用于读取，禁止编辑。

这种方法的优点：界面上的数据源立即清晰可见，查找错误和调试代码，验证数据更加容易。这些类型很容易记录下来，提供了描述和示例-其他开发人员将帮助您。

如果您使用Swagger之类的工具来描述API ，请参阅如何自动创建这些类型。

提示3：任何别名


您和您的团队可以编写出色的代码-没问题，没有错误，而且准时。如果是这样，请转到下一点；）

实际上，存在省略键入的诱惑，尤其是在现有项目中实现TS时。我想显式或隐式使用任何。有充分的理由：复杂的代码，紧迫的任务，轻巧-付出的努力并没有付出任何代价。

而不是忽略问题，请尝试将其合法化：

export type TODO_ANY = any;

优点：

  • 您不会作弊，也不会写拐杖。
  • 您有意识地表明，将来值得完成代码。
  • 通过搜索项目很容易找到此类型。
  • 可以在类型无关紧要的地方显式使用Any，例如：

const getArrayLength = (array: any[]) => array.length

局限性：

  • 过于频繁地使用该技术不太可能是有益的。
  • 需要有意识和责任感，才能在将来消除此类困扰。

您可能还会发现它有用： 

export type TODO_UNKNOWN = unknown;

在任何情况下，编译器将允许几乎任何动作。相反，在未知的情况下，几乎任何使用都会导致错误。他们设置TODO_UNKNOWN直到清除对象的类型→将来使用该对象→收到错误→更正了该类型→令人惊奇。

提示4：库的类型


如果您正在使用流行的js库，请在DefinitelyTyped中查找类型。它们很可能已经写好了-我们连接并享受生活。

但是，如果您的内部库中肯定没有类型怎么办？尝试自动生成它们。对于使用jsDoc的用户来说，这是个好消息：在TypeScript 3.7中，tsc编译器已学会从.ts和.js文件生成 .d.ts类型。 

另一个好消息：有js-> ts转换器。例如，TypeStat帮助我处理典型的React组件。

坏消息是，好的类型系统不太可能自动运行。依靠“骨骼”，随后将需要建立“肉”。

提示5：类型之上的类型


...和类型追逐！

碰巧项目的外部依赖项类型不合适-它们过于笼统（您想更具体），它们包含错误，版本不适合-或类型丢失。

通常，此类问题发生在内部库（也经常更新）中，而在外部库中则很少见。如果出现问题，请尝试手动定义/重新定义类型。

我们描述示例模块的类型。我强调只重新定义类型；实现不会改变：

declare module "example" {
    const foo: { bar: string };
    export default foo;
}

最好将此类替代移动到单独的目录和文件，例如，键入/ example.d.ts。将所有类型的输入内容连接到tsconfig中的editorOptions.typeRoots字段：

{
    "compilerOptions": {
        "typeRoots": ["node_modules/@types", "./typings"]
    }
}

瞧-例如，类型取自重新定义！

提示8：文字与枚举类型


TypeScript引入了一种方便的语法结构- 枚举。枚举用于最终枚举。掌握了枚举之后，起初我想经常编写它。但是，枚举对编译后生成的代码有相当的批评。

另一方面，还有文字类型-最终枚举的一种更简单，更方便的替代方法。在某些情况下，使用此类类型更为合适。

一个生活的例子就是简单的组件道具：

//   enum
enum ColorEnum {
    yellow = "yellow",
    transparent = "transparent"
}

const ExampleEnum = (props: { color: ColorEnum }) => <div>{props.color}</div>;

//    
type ColorLiteral = "yellow" | "transparent";

const ExampleLiteral = (props: { color: ColorLiteral }) => <div>{props.color}</div>;

//   
const Example = () => (<>
        <ExampleEnum color={ColorEnum.yellow} />
        <ExampleLiteral color="yellow" />
 </>);

在此示例中，两个组件都检查道具颜色的类型。带有enum的变体更加冗余-这是编译期间的附加代码，源代码中的附加导入以及更麻烦的编写结构。带有字符串文字的变量更加简洁，同时还包含类型检查。使枚举代码复杂化没有任何意义。

对于简单的类型检查，建议使用字符串文字。使用枚举来描述更复杂的逻辑。最好优先使用const枚举（但是，babel不支持此结构，您必须添加一个plugin）。

提示9：const断言


较少使用枚举的另一个原因是const断言。

as const告诉编译器该对象不会更改。因此，可以更详细地推导类型。

const loadAction = {
    start: "LOAD_START",
    done: "LOADED",
    error: "LOAD_ERROR"
} as const;

type LoadActionType = typeof loadAction;
type LoadActionKey = keyof LoadActionType; // 'start' | 'done' | 'error'
type LoadActionValue = LoadActionType[LoadActionKey]; // 'LOAD_START' | 'LOADED' | 'LOAD_ERROR'
//  "as const" LoadActionValue   string

您也可以将const应用于数组：

const DIRECTION_ASC = 1;
const DIRECTION_DESC = -1;
const sortDirectionTypes = [DIRECTION_ASC, DIRECTION_DESC] as const;
type SortDirection = typeof sortDirectionTypes[number]; // -> 1 | -1

使用转移很方便。例如，检查字典是否描述了所有选项：

const sortDirectionDescription: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " ",
    [DIRECTION_DESC]: " "
};
const sortDirectionDescription2: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " "
};
// Error: TS2741: Property '[-1]' is missing // in type '{ [DIRECTION_ASC]: string; }'

PS：您可以在本文中阅读有关const断言的更多信息。

秘诀10：品牌类型


品牌是创建唯一（“品牌”）类型的小工具。通常，这些类型是根据原始类型（字符串，数字）创建的。

例如，在使用数据ID时经常使用。有时将一种类型的ID与另一种类型的ID（例如，用户的ID与商店中他的订单的ID）区分开来是非常重要的，尽管两者都是字符串。使用品牌，您可以得到两种不同的类型：

type Brand<T, U> = T & { __brand: U };

type UserId = Brand<string, "userId">; // "5435436" –  id 
type OrderId = Brand<string, "orderId">; // "42da-fet3-cdsl" –  id 

declare const userID = "1111" as UserId; // - 
declare function getOrderById(id: OrderId): any; // - 

getOrderById(userID);
// Error: TS2345
// Type '"userId"' is not assignable to type '"orderId"'.

关于此主题的更多理论-有关标称和结构类型-可以在Habré以及此处和此处找到。

秘诀11：区分工会类型


区分工会 这个复杂的术语在实践中很容易理解：它创建具有共同部分（区分）的几种类型的工会。使用此类联合时，ts编译器可以过滤掉选项并指定类型。

使用区分联合的经典示例非常适合reducer模板。由于公共类型字段，有效负载的类型将自动显示：

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {
        case 'SET_ONE':
            return { ...state, one: action.payload}; // `payload` is string
        case 'SET_TWO':
            return { ...state, two: action.payload}; // `payload` is number
        default:
            return state;
    }
}

秘诀12：从不


从未有过针对原始案件的良好保险。取自上一段，仅不带“ SET_TWO”，并附加了typeguard：从不；

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {

        case 'SET_ONE':
            return { ...state,
                        one: action.payload}; // `payload` is string

        default:
            const typeguard: never = action;
            return state;
            // Error: TS2322: Type '{ type: "SET_TWO"; payload: number; }'
            // is not assignable to type 'never'.
    }
}

编译器推断出typeguard的类型并产生矛盾。一方面-永远不会。另一方面，输出类型为“ SET_TWO”。类型不收敛，我们得到一个错误。

在React中使用never的另一个有趣（但很少见）的情况是禁止嵌套或传递道具：

type IconProps = {
    size: 's' | 'm';
    children?: never;
    foo?: never;
};

const Icon: React.FC<IconProps> = () => <div>Whatewer</div>;
    
const Example = () => (<>
    <Icon size="s"> <div /> </Icon> {/*  –  children */}
    <Icon size="s" foo="123"/> {/*  -  foo */} 
</>);

同样，永远不会完全用于条件类型（conditional-types）。如果之前的所有示例对您来说都太简单了，那么这是一篇很酷的文章。
