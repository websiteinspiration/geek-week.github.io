OOPが存在する場合、1Cのレジスタはどのように見えるか


1Cでは、レジスターはシステムの重要な要素の1つです。この用語には英語で対応するものがあります-元帳。もともとは会計実務に登場しましたが、やがてそのロジックは他の分野でも使用されるようになりました。

レジスターが組み込みタイプの1つである1Cとは異なり、lsFusionプラットフォーム自体にはそのような概念はありません。しかし、その中には、継承、ポリモーフィズム、および集約があります。これにより、特に、同様のレジスタロジックを実装できます。この記事では、例を挙げて具体的に説明します。

レジスターは一連のレコードであり、各レコードは特定の一連の被験者（または次元）の状態変化を反映します。

1Cには4種類のレジスタがあります。

  1. 会計台帳
  2. 課金レジスター
  3. 貯蓄登録
  4. 情報登録

最初の2つは高度に専門化されており、会計と給与計算にのみ使用されます。lsFusionはビジネスアプリケーションを開発するためのユニバーサルプラットフォームであるため、実装は非常に簡単ですが、私はそれらを考慮しません。最後の2種類のレジスタについてのみ説明します。

貯蓄レジスター


レジスター内のエントリーは、何らかの抽象クラスのオブジェクトと見なすことができます。在庫のある製品の残高を計算する単純なレジスターを実装するとします。

これを行うには、抽象SkuLedgerクラスを宣言します。
CLASS ABSTRACT SkuLedger '   ';

正式には、その1つのインスタンスは、特定の製品と特定の倉庫（プラスまたはマイナス）の特定の倉庫の残高の単一の変更を反映します。

彼の測定値を、それぞれSku（製品）およびStock（倉庫）タイプの抽象的なプロパティとして設定しました。これらは、registerクラスから特定のクラスを継承するときに実装する必要があります。
sku 'SKU' = ABSTRACT Sku (SkuLedger);
stock '' = ABSTRACT Stock (SkuLedger);

残りが変更された時間を示すプロパティを追加します。
dateTime '/' = ABSTRACT DATETIME (SkuLedger);

最後に、いわゆる1C リソースを追加します。つまり、このレジスタエントリによって余りが変更される量。
quantity '-' = ABSTRACT NUMERIC[14,2] (SkuLedger);

次に、レジスターのエントリーに基づいて現在のバランスを計算します。
balance (Stock st, Sku sk) '' = GROUP SUM quantity(SkuLedger l) IF stock(l) = st AND sku(l) = sk;

デフォルトでは、現在の残高はデータベースに保存されません。これにアクセスすると、レジスタの現在の状態に基づいてそれを計算する要求が生成されます。レジスタが変更されたときに常に自動的に更新されるようにするには、宣言の最後にキーワードMATERIALIZEDを追加する必要があります。このフラグの存在により、実際には、1C用語でのバランスまたはターンの累積のこのレジスターが決定されます。

このようにして構築された残基に、常に正になるように制限を追加することができます。
CONSTRAINT balance(Stock st, Sku sk) < 0 
    MESSAGE '       ';

プラットフォームは、レジスタに書き込むときに残りがゼロより大きくなるように自動的に制御します。条件に違反している場合、変更はデータベースに書き込まれず、対応するメッセージがユーザーに発行されます。ちなみに、この制限が1C UTでどのように実装されているかを比較して、1Cプログラマーが経験する本当の痛みを評価することができます。

必要に応じて、製品の売上高をすばやく計算します。たとえば、次のプロパティを作成できます。
balance (Stock st, Sku sk, INTEGER year) = 
    GROUP SUM quantity(SkuLedger l) IF stock(l) = st AND sku(l) = sk AND extractYear(dateTime(l)) = year MATERIALIZED;

このプロパティの値へのアクセスは瞬時に行われ、レジスタデータにはまったく影響しません。同様に、レジスターに基づいて計算された式は常に保存され、プラットフォーム自体がそれらを更新する責任を負います。

同様に、一定時間の残りを決定するプロパティを構築できます。
balance '  ' (Stock st, Sku sk, DATETIME dt) = 
    GROUP SUM quantity(SkuLedger l) IF stock(l) = st AND sku(l) = sk AND dateTime(l) <= dt;

この値を計算するために、最初から指定された日付までレジスタを完全に読み取るように要求が行われます。しかし実際には、参照は通常、最新の日付になります。したがって、現在の剰余が格納されている場合（MATERIALIZED）、現在の剰余とは逆の方向でこのプロパティを計算することが論理的です。
balance '  ' (Stock st, Sku sk, DATETIME dt) =
   currentBalance(sk, st) (-) (GROUP SUM quantity(SkuLedger l) IF stock(l) = st AND sku(l) = sk AND dateTime(l) > dt);

当然、これは、インデックスがdateTimeプロパティに基づいて構築されている場合にのみ有効です。
INDEX dateTime(SkuLedger l);

そのようなレジスタを1つ作成するためのすべてのコードを別のモジュールに配置し、必要に応じて接続するのが論理的です。

ここで、登録書類で実行する方法を示します。倉庫での商品の受け取りのドキュメントを発表したとします。
CLASS Receipt '  ';
dateTime '/' = DATA DATETIME (Receipt);
stock '' = DATA Stock (Receipt);

CLASS ReceiptDetail '   ';
receipt '' = DATA Receipt (ReceiptDetail) NONULL DELETE;

sku 'SKU' = DATA SKU (ReceiptDetail);

quantity '-' = DATA NUMERIC[14,2] (ReceiptDetail);
price '' = DATA NUMERIC[14,2] (ReceiptDetail);

作成したレジスターに従って実行されることを確認する必要があります。これを行うには、領収書行ReceiptDetailを取得し、抽象SkuLedgerクラスから継承します。
EXTEND CLASS ReceiptDetail : SkuLedger;

その後、レジスターのすべての抽象プロパティを実装する必要があります。これは、値が取得されるレシートラインの特定のプロパティを示します。
dateTime(ReceiptDetail d) += dateTime(receipt(d));

stock(ReceiptDetail d) += stock(receipt(d));

sku(ReceiptDetail d) += sku(d);
quantity(ReceiptDetail d) += quantity(d);

数量と商品はラインから直接、時間と倉庫はドキュメントから直接置き換えます。

倉庫から倉庫に移動するためのドキュメントが宣言されている、より複雑なケースを考えてみましょう。

在庫から在庫への転送
CLASS Transfer '    ';
posted '' = DATA BOOLEAN (Transfer);
dateTime '/' = DATA DATETIME (Transfer);

fromStock ' ()' = DATA Stock (Transfer);
toStock ' ()' = DATA Stock (Transfer);

CLASS TransferDetail '   ';
transfer '' = DATA Transfer (TransferDetail) NONULL DELETE;

sku 'SKU' = DATA SKU (TransferDetail);

quantity '-' = DATA NUMERIC[14,2] (TransferDetail);
price '' = DATA NUMERIC[14,2] (TransferDetail);

これもこのレジスタで実行する必要がありますが、すでに2回実行されています。彼は倉庫から（どこから）商品の転送された数量を償却する必要があり、2回目にそれを宛先倉庫に追加します。商品を帳消しにするために、継承の同じロジックを使用します。
流動費
EXTEND CLASS TransferDetail : SkuLedger;

dateTime(TransferDetail d) += dateTime(transfer(d));

stock(TransferDetail d) += fromStock(transfer(d));

sku(TransferDetail d) += sku(d);
quantity(TransferDetail d) += -quantity(d);

これは消耗品の操作であるため、数量をマイナスにして、送信者の倉庫を倉庫に置き換えます。

あるクラスを別のクラスから2回継承することはできないため、再登録するために、新しいTransferSkuLedgerクラスの集約オブジェクトを作成し、それをSkuLedgerから継承します。
CLASS TransferSkuLedger '   ()' : SkuLedger;
transferSkuLedger = AGGR TransferSkuLedger WHERE stockTo(transfer(TransferDetail transferDetail));

AGGR コンストラクトは、宛先倉庫が設定されている各移動ラインについて、ソースラインへのリンクを持つ新しいTransferSkuLedgerクラスオブジェクトを作成する必要があることをプラットフォームに通知します。このような行を削除すると、集約されたオブジェクトも自動的に削除されます。
新しいクラスの抽象プロパティを実装するだけです。
dateTime(TransferSkuLedger d) += dateTime(transfer(transferDetail(d)));

stock(TransferSkuLedger d) += toStock(transfer(transferDetail(d)));

sku(TransferSkuLedger d) += sku(transferDetail(d));
quantity(TransferSkuLedger d) += quantity(transferDetail(d));

新しく作成されたクラスには、それを生成した元のクラスへのリンクがあるという事実を利用します。

ちなみに、1Cではドキュメント行が1つのレジスタエントリしか生成できないため、これには特定の問題があります。
システムは、累積レジスタに格納されたレコードの一意性を制御します。このため、累積レジスタには、同じドキュメントの同じ行に関連する2つのエントリはありません。

もう1つの重要な違いは、すべてのロジックが、文書の保持において宣言的に設定され、強制的に設定されないことです。これにより、ドキュメントの変更でレジスターをより効率的に更新でき、ドキュメントの完全な配布と保持を必要としません。ドキュメント内の1つのエントリのみが変更された場合、レジスタの更新は1つのエントリのみに影響します。

情報登録


累積レジスターとは異なり、情報レジスターは指標の量を計算しませんが、特定の時間に有効な最後の値を計算します。

このようなレジスタの宣言は、累積レジスタのロジックと完全に同じです。たとえば、レシートの価格を変更するためのレジスターを作成しましょう：
領収書の価格の変更の登録
CLASS ABSTRACT PriceLedger '   ';

dateTime '/' = ABSTRACT DATETIME (PriceLedger);

sku 'SKU' = ABSTRACT SKU (PriceLedger);
stock '' = ABSTRACT Stock (PriceLedger);

price '' = ABSTRACT NUMERIC[14,2] (PriceLedger);

領収書の最終価格を計算するには、次の構成を使用します。
price '' (Stock st, Sku sk, DATETIME dt) = 
    GROUP LAST price(PriceLedger l)
          ORDER dateTime(l), l
          WHERE dateTime(l) <= dt
          BY stock(l), sku(l);

残念ながら、このプロパティの値は無限の時点で定義されるため、具体化することはできません。ただし、目的のインデックスを作成すると、呼び出しは対数時間で比較的速く発生します。
INDEX stock(PriceLedger l), sku(l), dateTime(l), l;

1Cとは異なり、同じ時刻のレコードはlsFusionに存在できるため、レジスター自体がインデックスと順序に追加されます。この場合、追加の式として内部レジスタエントリコードが使用されます。

レジスターに関する情報は、累算レジスターと同じです。
EXTEND CLASS ReceiptDetail : PriceLedger;

dateTime(ReceiptDetail d) += dateTime(receipt(d));

stock(ReceiptDetail d) += stock(receipt(d));

sku(ReceiptDetail d) += sku(d);
price(ReceiptDetail d) += price(d);

結論


1Cのレジスタースキームを使用すると、継承と構成を使用して通常のプログラミングで実装されるものを実行できます（レジスターは単なるインターフェースです）。したがって、彼らは特定のケースで、1Cプラットフォームにこれらのメカニズムが存在しないという問題を解決しようとしています。実際には、レジスタは、ドキュメント自体の行、またはそれに基づいて作成された何らかの集約オブジェクト（構成）のいずれかを実装できるインターフェースです。

lsFusionでは、継承とポリモーフィズムが一般的なケースで実装されているため、より普遍的でより多くのケースに適用できます。実際、レジスタのロジックは、これらのレジスタからのデータに基づいて計算されたプロパティを計算する方法のみが異なるため、レジスタ内に実装できます。
