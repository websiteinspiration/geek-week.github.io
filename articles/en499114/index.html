Exporting the Federal Communications Agency Numbering Plan to a Relational Database

The Federal Communications Agency regularly updates the publicly available numbering plan . If you use this plan to determine the region or subscriber provider in your dialplan, then you are most likely interested in the relevance of this information. At first glance, there is nothing difficult in writing an application that downloads, processes and sends data to the database, however, when you start implementation, you will inevitably come across pitfalls, which I will talk about now.

The dial plan consists of four tabular files in csv format:

rossvyaz.ru/data/ABC-3xx.csv
rossvyaz.ru/data/ABC-4xx.csv
rossvyaz.ru/data/ABC-8xx.csv
rossvyaz.ru/data/ DEF-9xx.csv

URLs sometimes change.

The structure of all tables is the same:

/ DEF;;;;;

In the Region field , some providers have extra semicolons:

 ; | ;

To avoid an error while reading the table by your application, you need to disable checking for the expected number of fields or replace the incorrect record with the correct one.

We are going to write data to a table with the following structure:

first BIGINT PRIMARY KEY NOT NULL  //  
last BIGINT UNIQUE NOT NULL        //  
provider TEXT                      // 
source_region TEXT                 //        
region INT NOT NULL                //

Most regions in the tables can be unambiguously identified by the occurrence of a substring. For example, all entries for the Saratov region contain the substring Sarat .

For those regions that cannot be unambiguously determined, additional logic is needed, about it below.

Example 1:

  • Yamal-Nenets Autonomous Okrug
  • Nenets Autonomous Okrug

The Yamalo-Nenets Autonomous Okrug is part of the Tyumen Region, and the Nenets Autonomous Okrug is part of the Arkhangelsk. The problem of determining the numerical code of the region is that the substring Nenets AO is fully included in the Yamalo-Nenets AO . This means that two regions will correspond to entries containing the substring of the Nenets Autonomous Area.

To solve this problem, you need to add a check for the absence of a substring (excluding substring). In other words, the Yamalo-Nenets Autonomous Okrug will be determined by the entry of Yamalo , and the Nenets Autonomous Okrug by the entry of Nenets and the absence of Yamalo .

Example 2:

  • g.o. Krasnogorsk
  • Lyantor
  • Partza

An example of unidentified regions. There are many such records in the tables of Rossvyaz. Finder helps to find out that GO Krasnogorsk - Moscow region, Lyantor - the Khanty-Mansi Autonomous Okrug, and the settlement of Partz - the Republic of Mordovia. The solution is simple, the desired substring is converted to an array of strings, and we will check the occurrence in a loop. The exclusive substring is also converted to an array.

Example 3:

  • Siberian Federal District, Far Eastern Federal District
  • Krasnoyarsk Territory, Republic of Khakassia, Moscow, St. Petersburg
  • the Russian Federation

In this example, it is not possible to uniquely identify a region. You can choose the most populated of the listed, or assign a special code to such records. In fact, all the records that correspond to several regions are numbers 8-80 [0-9], federal numbers for which calls are not charged. I assigned codes to such records in the range from 200 to 210. I do not think that they will ever be useful to me.

The data_source field contains the paths to the Rossvyaz table files. If the path starts with http , the application will load the table using the built-in web client, otherwise it will try to find the table in the file system.

Pay attention to the commented out lines. These are the paths to the Rossvyaz table files stored in the repository. If you do not have the fastest Internet connection, each application launch will download these files again. For a faster application, uncomment these lines and comment out the hyperlinks. You may need to run the application more often than you think, later I will explain why using an example.

Fields exceptions and regionscontain paths to files with corrections and a description of the regions. About them a little later.

The db section describes the parameters for connecting to the MySQL DBMS. To use another DBMS, you need to replace the driver in the code and the connection string template in the constructor of the Service structure

db, err := dbr.Open("mysql",
		fmt.Sprintf("%s:%s@tcp(%s)/%s", c.DB.User, c.DB.Password, c.DB.Host, c.DB.Name),
		nil)

However, there is nothing complicated in adding a DBMS type field to the config and adding support for these types to the application, pull-requests are welcome again.

The exceptions.yml file contains a list of fixes. All matches of substrings in exceptions.yml before the colon, found in the table files of Rossvyaz, will be replaced by the substrings following the colon.

The regions.yml file contains a list of numeric codes of regions with arrays of occurrences and excluding substrings used to determine a specific region. If no record is found for any record or there are two or more matches, then this record will go to the database, but the region field will contain the value 0 (region not defined).

As a cherry on the cake, I offer the Asterisk dialplan macro to determine the region by subscriber number:

[macro-get-region]
        exten => s,1,MYSQL(Connect conn localhost user password dbname)
        exten => s,n,MYSQL(Query result ${conn} SELECT region FROM codes WHERE ${ARG1} BETWEEN first AND last LIMIT 1)
        exten => s,n,MYSQL(Fetch region ${result} region_num)
        exten => s,n,MYSQL(Clear ${result})
        exten => s,n,MYSQL(Disconnect ${conn})

For the same purposes, you can use the Lua script:

row = cur:fetch ({}, "a")
if row ~= nil then
