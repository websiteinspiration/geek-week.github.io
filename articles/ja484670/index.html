PostgreSQLアンチパターン：珍しいレコードがJOINの真ん中に到達する

実装する必要のあるアルゴリズムを分析せずにSQLクエリを作成する場合、これは通常、パフォーマンスの点で優れたものにはなりません。

このような要求は、プロセッサ時間を「消費」し、ほとんど突然にデータをアクティブに読み取ることを好みます。さらに、これは必ずしもある種の複雑なクエリであるとは限りません。逆に、単純に記述すればするほど、問題が発生する可能性が高くなります。そして、JOINオペレーターが出たら...


テーブルを結合すること自体は、有害であり有用ではありません。これは単なるツールですが、使用できる必要があります。

監督のグループ化


最初に、非常に単純な例を取り上げます。

100エントリの「辞書」があります（たとえば、これらはロシア連邦の地域です）。

CREATE TABLE tbl_dict AS
SELECT
  generate_series(0, 100) k;
ALTER TABLE tbl_dict ADD PRIMARY KEY(k);

...そして添付されているのは、10万エントリごとの関連する「事実」の表です。

CREATE TABLE tbl_fact AS
SELECT
  (random() * 100)::integer k
, (random() * 1000)::integer v
FROM
  generate_series(1, 100000);
CREATE INDEX ON tbl_fact(k);

次に、各「リージョン」の値の合計を計算してみましょう。

聞かれるように書いてあります


SELECT
  d.k
, sum(f.v)
FROM
  tbl_fact f
NATURAL JOIN
  tbl_dict d
GROUP BY
  1;

：自体は時間の18％を取ったデータを読み込み、残りは処理していた


[explain.tensor.ru見]

そして、すべてのハッシュ結合とハッシュ集計があるために私たちの願いで100Kレコードのそれぞれを処理しなければならなかったので、リンクされたテーブルのフィールドをグループ。

工夫を凝らしている


ただし、このフィールドの値は、集計テーブルのフィールドの値と同じです！つまり、最初に「事実」をグループ化することを誰も気にかけず、次につながりを作るだけです。

SELECT
  d.k
, f.sum
FROM
  (
    SELECT
      k
    , sum(v)
    FROM
      tbl_fact
    GROUP BY
      1
  ) f
NATURAL JOIN
  tbl_dict d;

もちろん、この方法は普遍的ではありませんが、「通常のJOIN」の場合、リクエストの最小限の変更で時間の増加は2倍になりました。わずか100。

不平等な条件


ここで、タスクを複雑にしてみましょう。1つの識別子で接続された3つのテーブルがあります。メインと2つの補助データといくつかのアプリケーションデータがあり、これによってフィルタリングします。

小さいが非常に重要な発言：ターゲットタスクの「適用された」知識に基づいていても、最初のテーブルで条件が満たされることはすでにわかっています-ほとんどの場合（明確性-3：4）、2つ目-非常にまれ（1：8 ）

メインテーブルと最初の補助テーブルから、すべてのテーブルの条件が満たされているID値が偶数のIDで最初の100レコードを選択する必要があります。テーブル内のすべてのレコード、再び100Kにしましょう。

スクリプトジェネレーター
CREATE TABLE base(
  id
    integer
      PRIMARY KEY
, val
    integer
);

INSERT INTO base
SELECT
  id
, (random() * 1000)::integer
FROM
  generate_series(1, 100000) id;

CREATE TABLE ext1(
  id
    integer
      PRIMARY KEY
, conda
    boolean
);

INSERT INTO ext1
SELECT
  id
, (random() * 4)::integer <> 0 -- 3:4
FROM
  generate_series(1, 100000) id;

CREATE TABLE ext2(
  id
    integer
      PRIMARY KEY
, condb
    boolean
);

INSERT INTO ext2
SELECT
  id
, (random() * 8)::integer = 0 -- 1:8
FROM
  generate_series(1, 100000) id;

聞かれるように書いてあります


SELECT
  base.*
, ext1.*
FROM
  base
NATURAL JOIN
  ext1
NATURAL JOIN
  ext2
WHERE
  id % 2 = 0 AND
  conda AND
  condb
ORDER BY
  base.id
LIMIT 100;

工夫を凝らしている


以下のアプローチを使用して、加速を実現します。

  1. まず、リンクテーブルのすべての条件をチェックすることは、メインテーブルの条件（偶数IDの場合も）が満たされた場合にのみチェックすることが理にかなっていることを理解しています。
  2. 出力はbase.idでソートする必要があります。これには、このテーブルの主キーが最適です。
  3. ext2データは必要なく、状態を確認するためにのみ使用されます。これは、このテーブルを使用するすべての作業をJOINからWHERE部分に安全に削除できることを意味します。そして、チェックにEXISTSを使用します。そうでない場合、そのようなレコードがまったくない場合はどうなりますか？
  4. baseとext2の残りのチェックが成功した場合にのみ、ext1から少なくとも一部のデータを取得する必要があります。つまり、ext1との接続はbase / ext2を使用したすべてのアクションの後に実行する必要があります。これはLATERALを使用して実現できます。
  5. クエリプランナーがext2のsub2チェックをJOINに変えようとしないように、サブリクエストは"hide under CASE"です。

SELECT
  base.*
, ext1.*
FROM
  base
, LATERAL( --       base
    SELECT
      *
    FROM
      ext1
    WHERE
      id = base.id AND
      conda --  
    LIMIT 1
  ) ext1
WHERE
  CASE
    WHEN base.id % 2 = 0 THEN
      EXISTS( --       
        SELECT
          NULL
        FROM
          ext2
        WHERE
          id = base.id AND
          condb --  
        LIMIT 1
      )
  END
ORDER BY
  base.id --     PK,      
LIMIT 100;
