VPN / Mikrotik two-factor authentication - simple and scalable

Hello!

For the writing of this article, I was prompted to read the similar content of a user articlenkusnetsov. The number of views shows that the community is interested in this topic.

Therefore, I decided to share with you my own solution, which was previously implemented by me and in its basic form has:

  • Low level of entry and simplicity of code (for understanding / debugging by another employee)
  • Simple ROS scripts do not create any load and even work on hAP Lite
  • Scalability - the ability to connect a large number of VPN gateways to reduce load or geographical distribution
  • Ability to use Mikrotik CHR as a VPN server
  • ‚Äú1xN‚Äù - 1 SMS gateway for an unlimited number of routers with the possibility of expansion with increasing load
  • Ability to bind a separate router to a "specific" modem (for what? - more on that later)
  • Using just one php script on a remote server
  • It doesn‚Äôt matter which device initiated the VPN connection, authorization by the link from SMS
  • Ability to log employee authorizations on the server
  • Increase fault tolerance and reduce system load by sending randomly SMS from multiple modems


UPD 1.0 - php code updated. Now it includes logging all requests to the script and randomly sending messages through several gateways if more than one is specified.


Statement of the problem and solution


Task 1


In the process of developing this solution, the task was set to minimize the number of usb modems - reducing the cost of ownership, simplifying administration, localizing modems in one place and thereby improving the maintainability of the system.

It was decided to use one router installed on the ‚Äúreliable channel‚Äù as a basic SMS gateway. The only thing I did not know was the load on the system, the modems would cope and whether the operator would block it. Therefore, I made it possible to connect several modems and further divide the system into groups. Moreover, additional modems can be connected to the base gateway through a simple usb-hub. But for full functioning, with minimal functionality, you need one modem, regardless of the number of routers.

Task 2


To provide for the possibility of using local SIM cards in the installation area of ‚Äã‚Äãthe router.
Example: a wide branch network with several stores in Kazakhstan. Sending an SMS message from the Russian Federation will be quite expensive. This solution allows employees from Kazakhstan to receive sms from a local number.

But in the process of reflection, it turned out that the solution had already been found and implemented in Task 1.

Task 3


Tunnel authorization from a mobile device without having to physically be on the authorized device.

The goal is the ability to authorize not only user tunnels, but also any VPN connections: Mikrotik-> Mikrotik, Server-> Mikrotik, etc. In this case, the user responsible for these tunnels just needs to follow the link from the SMS message, in which also which tunnel wants to log in is displayed.

For the possibility of authorization in this way, it became necessary to transfer the script to an external resource - a server accessible from anywhere and also use the authorization code for marking the address list in the firewall and subsequent unlocking.

The solution to this problem already unambiguously implied the use of the RouterOS API (or SSH). API won as the easiest option.

Logics


  1. User logs in with pre-added username and password
  2. The VPN gateway puts its ip in the address waiting list and sends a POST request to the server
  3. The server receives data, checks, generates an authorization code and sends an SMS to the contact number of the form: ‚ÄúTo autorize user 79001112233 connection open - http_: //synome.ru/? Ruid = vrG7yYMbZ6 & auth = YU6zc‚Äù
  4. The user follows the link from any device
  5. The server makes a request to the router, checks for the presence in the address list of the entry with the authorization code in the comment. If a record is found, deletes it, and the user displays a successful login page

In all other cases, if something does not pass the test, the user will receive a Request error.

Setup and code


Now let's move on from the conceptual part to setting up Mikrotik, the code and their description.
Explanation: these scripts and php code are functional for any type of VPN server. Choosing a server type (PPP / L2TP / SSTP), setting up a profile and a tunnel - you choose "to your taste", I will give a general example of quick configuration.

Add to Mikrotik:


Firewall


Be sure to create a list of allowed resources, among which: own MT address, any public DNS, server address on which authorization will take place 
/ip firewall address-list
add address=10.10.0.1 list=Allow-list
add address=8.8.8.8 list=Allow-list
add address=synome.ru list=Allow-list


Add a rule blocking the traffic of VPN users from the waiting list to any host other than allowed 
/ip firewall raw
add action=drop chain=prerouting dst-address-list=!Allow-list src-address-list=VPN-blocked disabled=no


PPP Profile


Create a profile in which we set the idle connection timeout. The time should be less than that specified in the following On Up script, otherwise, if authorization has not been done, then after removing the list from the address-list, the user will get access for a time equal to the difference (idle timeout minus address-list timeout)

Add profile 
/ppp profile
add dns-server=10.10.0.1 idle-timeout=59m local-address=10.10.1.1 name=2F-VPN use-compression=no use-encryption=no use-mpls=no


Next, on the last tab of the Script add:

On up 
:global ruid "#ROUTERLOGIN#";
:global pass "#ROUTER_PASSWORD#";
:local userip [/ppp active get [find name=$user] address];
# if phone number stored in comment
#:local userphone [/ppp secret get [find name=$user] comment];
# if phone number = username
:local userphone $user;

:local authkey [/tool fetch http-method=post http-data="ruid=$ruid&pass=$pass&tel=$userphone" url="https://yourwebsite.ru/" mode=https as-value output=user];

/ip firewall address-list remove [find address=$userip];
/ip firewall address-list add address=$userip list=VPN-blocked timeout=1h comment=($authkey->"data");

:log info message="User connect:";
:log info message=$userphone;
:log info message=$userip;
:log info message=($authkey->"data");


On down 
:local userip [/ppp secret get [find name=$user] remote-address];
/ip firewall address-list remove [find address=$userip];
:log info message="User disconnect:";
:log info message=$user;
:log info message=$userip;
