Jetbrains KTorおよびR2DBCの例に関する反応データへの命令的アプローチ

corutinからデータベースへのリアクティブアクセスの使用に関する記事。 Springはすべてを簡素化しますが、アプリケーションの実際のプロセスの理解には影響しません。デモンストレーションでは、KTorフレームワークが選択されました（単に私がJetBrainsが何をするのかを見たいからです）。これは、コルーチンを集中的に使用するため、リアクティブストリームとこれらの同じコルーチンを組み合わせるタスクに関心が追加されます。その過程で、何が起こっているのかが、理解できないジェットストリームがわかりやすい命令型プログラミングに変換され、その上で犬が食べられたことの明確な例であることが明らかになりました。私は反応チェーンが好きですが、軍の秩序が好きな人を喜ばせてみませんか？


Jetアプリケーションは多くの開発者の心をつかんでおり、これらのセットは著しく交差しています。仕様の作成者から消化可能なライブラリに純粋な心の流れを適応させるコミュニティの努力がなければ、それらはさらに植え付けられます。これは、R2DBC仕様とSpring（Boot）フレームワークで発生しました。開発者は、使い慣れた反応データ型を備えた使い慣れたSpring Data APIをすでに確認できます。ただし、Springを使用しない理由はいくつかあります。Springは必要なく、何か新しいものが必要です。さて、レガシーコードはまだ残っていますが、この場合、リアクティブなデータアクセスはほとんどありません。


この場合、ニスを塗っていないR2DBCを調べる必要があります。また、JDBCがSpring Data JPAと異なるように、完成したフレームワークで提供されるものとは大きく異なることが予想されます。さらに反応性。そして、Reactive Streams仕様に従った反応性。そして、私たちはコルーチンを聞きます。どちらが未来のようであり、それでもそれらを書き換えます。


メインメソッドからコルーチンを手動で開始することもできますが、並列処理と競合が本当に必要であることを想像してみましょう。つまり、高負荷（たとえば、1時間に1リクエストです！）であり、Springを使用せずに非常に真剣に書くことにしました。しかし、ここで判明したように、Kotlinで完全に記述された軽量のアナログがあり、言語自体の作成者から、さらには私たちが夢見るコルーチンまでです。

プール統計を提供します


美しさのために（メトリックの標準モジュールを接続しなかったため）、プール統計を表示するためのポイントを追加します。私たちの言語ツールで次のことができるため、テンプレートエンジンは不必要です。


        get("/pool") {
            showPool@
            //      
            call.respondText {
                (pool.metrics.map {
                    """
                Max allocated size:                 ${it.maxAllocatedSize}
                Max pending size  :                 ${it.maxPendingAcquireSize}
                Acquired size     :                 ${it.acquiredSize()}
                Pending acquire size:               ${it.pendingAcquireSize()}
                Allocated size    :                 ${it.allocatedSize()}
                Idle size         :                 ${it.idleSize()}\n
            """.trimIndent()
                }.orElse("NO METRICS"))
            }
        }

もちろん、必要に応じて、HTML DSLを介してこれを提供できます。
