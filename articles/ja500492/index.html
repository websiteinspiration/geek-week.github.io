ただし、カプセル化のためにビュースタックに直接アクセスすることはできません。ナビゲーションは前方にのみプログラムされ、戻ることができるのは1レベルだけ後、「戻る」ボタンのカプセル化されたコードを介してのみです。

同様の要素のリストの画面の場合、すべてがコンパクトに見えます。

 NavigationView{

私の意見では、NavigationLinkの最も深刻な問題は、指定されたすべてのビューリンクが遅延ではないことです。それらはリンクがトリガーされた瞬間ではなく、作成時に作成されます。多くの要素のリストまたは多くの異なるViewヘビーコンテンツへの遷移がある場合、これはアプリケーションのパフォーマンスに最良の方法で影響を与えません。ロジック付きのこれらのビューにViewModelがまだアタッチされており、その実装でライフサイクルビューが考慮されないか、正しく考慮されない場合、状況は非常に困難になります。

たとえば、同じタイプの要素を含むニュースリストがあります。一度に1つのニュース項目の1つの画面に行ったことは一度もありません。モデルはすでに記憶に残っています

まず、Viewは孤立して存在するのではなく、UIHostingControllerにレンダリングされることを思い出してください。 

open class UIHostingController<Content> : UIViewController where Content : View {

    public init(rootView: Content)

    public var rootView: Content
//...
}

そして、これはUIViewControllerです。したがって、次のことができます。新しいUIHostingController内の次のビューへの遷移のすべての責任を現在のビューのコントローラーに転送します。ビューから呼び出すナビゲーションおよび構成モジュールを作成しましょう。

UIViewControllerと連携するナビゲーターは次のようになります。

class Navigator {
    private init(){}
    
    static let shared = Navigator()
    
    private weak var view: UIViewController?
    
    internal weak var nc: UINavigationController?
        
   func setup(view: UIViewController) {
        self.view = view
    }
     
  internal func open<Content:View>(screen: Content.Type, _ data: Any? = nil) {
     if let vc = ModuleConfig.shared.config(screen: screen)?
        .createScreen(data) {
        self.nc?.pushViewController(vc, animated: true)
        }
   }

同じ原則で、特定のモジュールのコンフィギュレーターの実装を提供するコンフィギュレーターのファクトリーを作成します。

protocol IConfugator: class {
    func createScreen(_ data: Any?)->UIViewController
}

class ModuleConfig{
    private init(){}
    static let shared = ModuleConfig()
    
    func config<Content:View>(screen: Content.Type)->IConfugator? {
        if screen == NewsListView.self {
            return NewsListConfigurator.shared
        }
      // -
        return nil
    }
}

ナビゲーターは、画面のタイプごとに、特定のモジュールのコンフィギュレーターを要求し、必要なすべての情報をそれに転送します。

class NewsListConfigurator: IConfugator {
    static let shared = NewsListConfigurator()
    
    func createScreen(_ data: Any?) -> UIViewController {
         var view = NewsListView()
        let presenter = NewsListPresenter()
         let interactor = NewsListInteractor()
        
        interactor.output = presenter
        presenter.output = view
        view.output = interactor
        
        let vc = UIHostingController<ContainerView<NewsListView>>
                    (rootView: ContainerView(content: view))
        return vc
    }
}

コンフィギュレーターはナビゲーターであるUIViewControllerを提供し、UINavigationControllerを共有スタックにプッシュします。



コードのNavigationLinkをNavigatorの呼び出しに置き換えます。トリガーとして、リストアイテムをクリックするイベントがあります。

  List(model.data) { item in
            NewsItemRow(data: item)
                .onTapGesture {
                Navigator.shared.open(screen: NewsItemView.self, item)
            }
        }

同じ方法で、ViewメソッドでNavigatorを呼び出すことを妨げるものはありません。体内だけではありません。

コードが著しくクリーンになっただけでなく、メモリもアンロードしました。結局のところ、このアプローチでは、ビューは呼び出されたときにのみ作成されます。
