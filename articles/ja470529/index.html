レガシープロジェクトをGraphQLに変換する方法

こんにちは、ハブル。私の名前はアントンポタポフです。FINCHの iOS開発者です。今日、モバイルプロジェクトをGraphQLに変換する方法について詳しく話し、このアプローチの長所と短所を説明します。始めましょう。

簡単な紹介


「レガシー」私は誰もがこの恐ろしい言葉を聞いたと思います、そして大部分は彼と向かい合って会いました。したがって、新しい優れた機能をプロジェクトに統合することがどれほど難しいかを伝える必要はありません。



私たちのプロジェクトの1つは、宝くじの大企業への申し込みです（NDA以来、名前を付けることはできません）。最近、気を失うことなくそれをGraphQLに変換する必要がありました。

プロジェクトは非常に大規模です。一見すると、少なくとも200〜300時間を費やす必要がありましたが、これは考えられるすべての2週間のスプリントを超えています。また、GraphQLよりも重要な副次的な機能もあったため、スプリント全体を1つのタスクに割り当てることもできませんでした。
私たちは長い間何をすべきかを考え、プロジェクトを段階的に、モデルごとに翻訳することにしました。このアプローチでは、移行がスムーズになり、200〜300時間が複数のスプリントに分散されます。

テクニカルポイント


プロジェクトに取り組むために、私はアポロライブラリを使用しました。Habréには、それを操作することのすべての微妙な点を説明する記事が既にあるので、ここでは繰り返しません。事前に警告します。コード生成モデルでの作業はあまり便利ではなく、「ネットワーク」として維持することをお勧めします。各エンティティには、__ typename：文字列フィールドが含まれ、奇妙なことに、型名を返します。

タスク分解


まず最初に、どのレガシーモデルをGraphQLに変換するかを決定します。私の場合、巨大なGameInfoモデルの1つを翻訳し、それにDrawInfoを埋め込むのは論理的でした。

  • GameInfo-宝くじゲームと抽選に関する情報が含まれています。
  • DrawInfo-配布に関するデータが含まれています

選択を終えて、私は古いレガシーモデルをGraphQLで取得した新しいモデルで初期化することにしました。このアプローチでは、古いモデルが使用されているアプリケーションの部分を置き換える必要はありません。以前のモデルを完全に完了することは危険でした-プロジェクトが以前のように機能し、時間がかかりすぎるという事実ではありませんでした。

合計すると、GraphQL実装の3つの段階を区別できます。

  • 顧客の創造;
  • レガシーモデルの初期化子を作成します。
  • APIリクエストをGraphQLで置き換える。

GraphQLClient


他のクライアントと同様に、GraphQLClientにはフェッチメソッドが必要です。その後、必要なデータがロードされます。

私の意見では、GraphQLClientのfetchメソッドは、対応する要求が実行される列挙に基づいて受け入れる必要があります。このアプローチにより、目的のエンティティまたは画面のデータを簡単に受信できます。リクエストがパラメータを受け入れる場合、それらは関連する値として渡されます。このアプローチを使用すると、GraphQLを使用して柔軟なクエリを作成するのが簡単になります。

enum GraphQLRequest {
    case image(ImageId?)
}

カスタマイズされたGraphQLクライアントには、既存の機能用にカスタマイズされたApolloClientと、上記のフェッチロードメソッドが含まれている必要があります。

func fetch<Response>(requestType: GraphQLRequest, completion: @escaping (QLRequestResult<Response>) -> Void)

QLRequestResultとは何ですか？それは普通です

 typealias QLRequestResult<Response> = Result<Response, APIError>

fetchメソッドを使用すると、プロトコルを介してクライアントにアクセスし、requestTypeで切り替えを実行できます。requestTypeに応じて、対応するプライベートロードメソッドを使用できます。このメソッドでは、結果のモデルを古いモデルに変換できます。例えば：

    private func fetchGameInfo<Response>(gameId: String? = "", completion: @escaping (QLRequestResult<Response>) -> Void) {
        
	//   Apollo 
        let query = GetLotteriesQuery(input: gameId)
        
	//      
        apolloClient.fetch(query: query, cachePolicy: .returnCacheDataAndFetch, queue: .global()) { result in
            
            switch result {
                
            case .success(let response):
                guard let gamesQL = response.data?.games,
                    let info = gamesQL.info else {
                    completion(.failure(.decodingError))
                    return
                }
                
		//   
                let infos: [Games.Info] = info.compactMap({ gameInfo -> Games.Info? in
                    
                    guard let gameIdRaw = gameInfo?.gameId,
                        let gameId = GameId(rawValue: gameIdRaw),
                        let bonusMultiplier = gameInfo?.bonusMultiplier,
                        let maxTicketCost = gameInfo?.maxTicketCost,
                        let currentDraws = gameInfo?.currentDraws else { return nil }
                    
                    let currentDrawsInfo = Games.Info.CurrentDrawsInfo(currntDraw: currentDraws)
                    let gameInfo = Games.Info(gameId: gameId, bonusMultiplier: bonusMultiplier, maxTicketCost: maxTicketCost, currentDraws: currentDrawsInfo)
                    
                    return gameInfo
                })
                
		//   
                let games = Games(info: infos)
                
                guard let response = games as? Response else {
                    completion(.failure(.decodingError))
                    return
                }
                
                completion(.success(response))
                
            case .failure(let error):
                
                …
            }
        }
    }

その結果、GraphQLから入​​手できる既製の古いモデルが得られます。

スカラー型


GraphQLのドキュメントでは、スカラータイプは次のように説明されています。「スカラータイプは、オブジェクトの再選択やキャッシュのキーとしてよく使用される一意の識別子です。」Swiftの場合、スカラー型はタイプエイリアスに簡単に関連付けることができます。

クライアントを作成しているときに、私のスキームにスカラー型のLongがあるという問題に遭遇しました。

typealias Long = Int64

すべてが問題ないでしょうが、Apolloは自動的にすべてのユーザースカラーを文字列にキャストするため、クラッシュが発生します。私はこのように問題を解決しました：

  • codegenスクリプトに追加した-passthroughCustomScalars
  • タイプエイリアス用に別のファイルを作成しました
  • ファイルに追加されました 
    public typealias Long = Int64

将来的には、各スカラー型に対して、新しい型エイリアスを追加する必要があります。Apolloバージョン14.0では、「Intカスタムスカラーのサポート」が追加されました。このアプローチとcodegenのフラグの使用を避けたい場合は、Apollo gitでこの問題の解決策を確認してください。

長所と短所


なぜこのアプローチが良いのですか？すべての古いモデルを削除するアプローチに関して、GraphQLの使用へのかなり迅速な移行。

今後、任意の画面/モデルのデータを取得する必要がある場合は、GraphQLClientを使用して必要なデータを取得できます。

マイナスの：

  • クライアントでモデルのタイプを変換すると、別のエンティティに転送できます データの操作はクライアントの責任ではありません。
  • 広大なGraphQLClient。新しいクエリを追加すると、クラスはますます大きくなります。解決策の1つは、読み込みメソッドが記述される拡張機能です。これについては、GraphQLClientの章で説明しました。

調査結果


一般に、適切に作成されたクライアントを使用すれば、GraphQLへの切り替えはすばやく簡単に行えます。約3日= 24時間かかりました。この間、クライアントの作成、モデルの翻訳、GameInfoモデルの再作成に成功しました。説明されているアプローチは万能薬ではなく、短時間で実装された純粋に私の決定です。
