トランザクションの分離を確実にする1つの方法としてのMVCC

こんにちは、ハブル。私の名前はウラジスラフロダンです。現在、私はOTUSのHigh Load Architectコースの責任者を務めており、ソフトウェアアーキテクチャのコースも教えています。

特に「高負荷の建築家」コースの新しいセットの開始のために、私はあなたに喜んで共有する少しの資料を書きました。



前書き


前回は、データベースのトランザクション分離を弱めることの結果についてお話しました。今日は、この分離を確実にし、考慮された異常を回避する方法の1つについて、より詳細に説明します。お気づきかもしれませんが、前回の記事では2つのアプローチがしばしば区別されました。1つはレコードにいくつかのバージョンがあるという事実に基づいており、もう1つは何らかの方法で記録をブロックするという事実に基づいていました。したがって、データベースの2つのクラスが区別されます：バージョンとブロッカー。本日は、バージョンコントローラーとは何かについてお話します。次回はブロッカーの検討から離れましょう。

バージョン管理者


言ったように、アプローチの1つはバージョン管理に基づいています。これは、楽観的アプローチまたはMVCC（マルチバージョン同時実行制御）とも呼ばれます。実際、アクティブなトランザクションのバージョンストレージが発生します。

バージョンはどこに保存されますか？


メカニズムの実装はデータベースに依存します。それらのいくつかの例を挙げます。

PostgreSQL


各トランザクションは、いくつかのid-shnikによって特徴付けられます。 Id-shnikトランザクションは単調に増加します。どの行にも、メカニズムを提供するためのメタ情報を表す2つの属性があります。updated_by_id-このレコードを最後に更新したトランザクションのidとdeleted_by_id-このレコードを削除したトランザクションのid。新しいトランザクションが到着すると、データベースは現在実行されているトランザクションのIDニックネームを判別します。これらのトランザクションによって行われた変更は、着信トランザクション内では無視されます。着信トランザクションは、データのバージョンと同じように機能することがわかります。古いバージョンのデータは、現在のバージョンと同じ場所に保存されます。更新が到着すると、別の行が追加され、このレコードがアクティブになります。このスキームを正しく動作させるには、「ガベージコレクター」が必要であることも明らかです。一部のレコードがdeleted_by_id = 100で、現在のトランザクション間の最小id-shnikが150の場合、このレコードを削除する必要があります。

MySQL（InnoDBエンジン）


現在のバージョンのみがMySQLデータベースに保存されます。更新が到着すると、テーブルファイル内のデータが修正されます。データを調整した後、以前のバージョンはロールバックログにあります。MySQLにはいくつかのロールバックログがあります（これらは、insert'ovとupdate'ovでは異なります）。トランザクションが前のバージョンの行を必要とする場合、システムはログを元に戻し、必要なバージョンを復元します。バージョンはトランザクションIDによっても決まります。

オラクル


スキームはMySQLに似ています。現在のバージョンはデータファイルに保存され、古いバージョンは元に戻すログのおかげで復元されます。Oracleの取り消しログは周期的に書き直されます。したがって、トランザクションが非常に古いバージョンのレコードを必要とする状況が発生する可能性がありますが、元に戻すログには存在しません。この状況が発生すると、トランザクションは失敗します。

MS SQL


MS SQLでは、バージョン管理モードとブロッカーモードの両方を含めることができます。データベース設定でバージョン対応モードを有効にするには、スナップショットを有効にする必要があります。 MS SQLには、一時テーブルを格納するように設計されたシステムテーブル（tempdb）があります。テーブルには現在のバージョンが含まれていますが、古いバージョンを格納するために使用されるのはtempdbです。システムプロセスは、tempdbに誰も参照していないバージョンが存在することを監視しているため、削除できます。トランザクションが長期にわたる場合は、バージョンが保存されます。 Tempdbが大きくなり、最大サイズに達すると、MS SQLは少しのディスク領域を割り当てます。終了した場合、スナップショット分離を伴うトランザクションは実行できません。この操作モードを使用する場合、長いトランザクションを監視する必要があります。そのようなトランザクションのロールバックのロールバックは、それが機能するのと同じくらいのコストがかかる場合があり、おそらくもう少しかかるためです。

紛争


このアプローチは楽観的と呼ばれます。これは、並列のデータ変更トランザクションが実行されても競合が発生しないことを期待しているためです。紛争が発生した場合はどうなりますか？トランザクションT1が10,000レコードを変更し、トランザクションT2が並行して1レコードを変更するとします。この1つのレコードがこれらの10,000に含まれている場合は、トランザクションの1つがロールバックされます。T1が最初に実行された場合、T2がロールバックされ、それ以外の場合は逆になります。

ロールバックメカニズム


バージョン付きバージョンでのトランザクションロールバックのメカニズムは、実装によって異なります。たとえば、PostgreSQLでは、トランザクションは退避済みとしてマークされ、真空によりディスク領域が解放されます。このようなメカニズムは十分に高速です。Oracleでは、ロールバックの場合、データは取り消しログから復元されます。この場合、稼働時間は長くなりますが、ロッカーよりもはるかに高速です。MySQLはOracleと同じように動作します。

結論


ライターはリーダーをブロックせず、リーダーは自分のバージョンのデータを読み取るだけなので、楽観的なアプローチは優れています。したがって、バージョン管理は、主な負担が書き込みではなく読み取りにある場合に役立ちます（ブログ、レポート、および頻繁に大量に読む必要があるその他の場合）。
