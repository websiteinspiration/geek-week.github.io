MySQL（Percona Server）を5.7から8.0にアップグレードする



進歩は止まらないので、MySQLの最新バージョンにアップグレードする理由はますます重要になってきています。少し前まで、私たちのプロジェクトの1つで、居心地の良いPercona Server 5.7クラスターをバージョン8にアップグレードするときがきました。これはすべて、Ubuntu Linux 16.04プラットフォームで発生しました。最小限のダウンタイムで同様の操作を実行する方法と、アップグレード中に発生した問題-この記事をお読みください。

トレーニング


データベースサーバーの更新は、データベースの移行に関連している可能性があります。システムリソースの制限の要件の変更と、廃止されたディレクティブをクリアする必要があるデータベース構成の修正です。

更新する前に、必ず公式ドキュメントを参照します。

  • MySQL 8リリースノート。
  • MySQLからのアップグレードガイド。
  • Perconaからのアップグレードガイド。
  • レプリカとウィザードの更新に関するMySQLチュートリアル。

そして行動計画を立てます：

  1. 廃止されたディレクティブを削除して、構成ファイルを修正します。
  2. ユーティリティとの互換性を確認してください。
  3. パッケージをインストールして、スレーブデータベースを更新しますpercona-server-server。
  4. 同じパッケージを配置してウィザードを更新します。

計画の各項目を分析し、何が問題になるかを確認します。

重要！GaleraベースのMySQLクラスターのアップグレード手順には、記事に記載されていない独自の微妙な点があります。この場合、この命令を使用しないでください。

パート1：設定の確認


バージョン8では、MySQLが削除されましたquery_cache。実際、バージョン5.7で廃止されたと宣言されていましたが、現在は完全に削除されています。したがって、関連するディレクティブを削除する必要があります。また、クエリをキャッシュするために、ProxySQLなどの外部ツールを使用できるようになりました。

また、古いproディレクティブがconfigに見つかりましたinnodb_file_format。MySQL 5.7でInnoDB形式を選択できた場合、8番目のバージョンはすでにバラクーダ形式でのみ機能します。

その結果、次のディレクティブが削除されました。

  • query_cache_type、query_cache_limitおよびquery_cache_size;
  • innodb_file_formatとinnodb_file_format_max。

検証には、Percona ServerのDockerイメージを使用します。サーバー設定をディレクトリmysql_config_testに配置し、次にデータとログ用のディレクトリを作成します。percona-server構成テストの例：

mkdir -p {mysql_config_test,mysql_data,mysql_logs}
cp -r /etc/mysql/conf.d/* mysql_config_test/
docker run  --name some-percona -v $(pwd)/mysql_config_test:/etc/my.cnf.d/  -v $(pwd)/mysql_data/:/var/lib/mysql/ -v $(pwd)/mysql_logs/:/var/log/mysql/ -e MYSQL_ROOT_PASSWORD=${MYSQL_PASSWORD} -d percona:8-centos

結果：Dockerログ、またはログのあるディレクトリのいずれかに-構成に応じて-問題のディレクティブが記述されるファイルが表示されます。

これが私たちが持っていたものです：

2020-04-03T12:44:19.670831Z 0 [Warning] [MY-011068] [Server] The syntax 'expire-logs-days' is deprecated and will be removed in a future release. Please use binlog_expire_logs_seconds instead.
2020-04-03T12:44:19.671678Z 0 [Warning] [MY-013242] [Server] --character-set-server: 'utf8' is currently an alias for the character set UTF8MB3, but will be an alias for UTF8MB4 in a future release. Please consider using UTF8MB4 in order to be unambiguous.

したがって、エンコーディングを処理し、廃止されたディレクティブを置き換える必要がありましたexpire-logs-days。

パート2：実行中のインストールの確認


データベースの互換性をチェックするための更新ドキュメントには、2つのユーティリティがあります。それらの使用は、管理者が既存のデータ構造の互換性を確認するのに役立ちます。

古典的なmysqlcheckユーティリティから始めましょう。単に実行する：

mysqlcheck -u root -p --all-databases --check-upgrade

問題が検出されない場合、ユーティリティはコード0で終了します。



さらに、mysql-shellユーティリティは、MySQLの最新バージョンで使用できます（Perconaの場合、これはパッケージですpercona-mysql-shell）。これは、古典的なmysqlクライアントの代替であり、クライアント、SQLエディター、MySQL管理ツールの機能を組み合わせたものです。更新する前にサーバーを確認するには、サーバーから次のコマンドを実行します。

mysqlsh -- util check-for-server-upgrade { --user=root --host=1.1.1.1 --port=3306 } --config-path=/etc/mysql/my.cnf

エンコーディングの問題を示す上記の警告に関するメモ。事実は、最近になるまでのMySQLでそのUTF-8である「本物」UTF-8ではない、それはわずか3を保存するので、バイトの代わりに、MySQLの8 4.、彼らは最終的にそれを修正することを決めた：別名はutf8すぐにエンコードにつながるutf8mb4、と古いものテーブルの列はになりutf8mb3ます。将来、エンコーディングutf8mb3は削除されますが、このリリースでは削除されません。そのため、DBMSを更新した後、稼働中のDBMSのエンコーディングを修正することにしました。

パート3：サーバーの更新


そのようなシックな計画がある場合、何が問題になるのでしょうか？..ニュアンスは常に発生することをよく認識しているため、MySQL開発クラスターで最初の実験を行いました。

すでに述べたように、公式ドキュメントはレプリカでMySQLサーバーを更新する問題を強調しています。つまり、MySQL 8はバージョン5.7のウィザードから複製できるため、最初はすべてのレプリカ（スレーブ）を更新する価値があります。いくつかの困難は、リモートマスターが読み取り専用モードのときにマスター<->マスターモードを使用するという事実にあります。つまり、実際には、戦闘トラフィックは1つのデータセンターに入り、2番目はバックアップです。 トポロジは次のとおりです。 アップグレードは、mysqlレプリカdc 2レプリカから開始する必要があります。





、mysql master dc 2およびmysql replica dc 1、およびmysql master dc 1. serverで終了します。信頼性を高めるために、仮想マシンを停止し、スナップショットを作成し、更新の直前にコマンドを使用してレプリケーションを停止しましたSTOP SLAVE。残りの更新は次のようになります。

  1. 各レプリカは、設定オプション3を追加し、再起動しますskip-networking、skip-slave-start、skip-log-bin。実際のところ、データベースを更新すると、システムテーブルを更新したバイナリログが生成されます。これらのディレクティブは、データベース内のアプリケーションデータが変更されないこと、およびシステムテーブルの更新に関する情報がバイナリログに記録されないことを保証します。これにより、レプリケーションを再開するときの問題を回避できます。
  2. パッケージをインストールしますpercona-server-server。MySQL 8では、サーバーの更新後にコマンドを実行する必要がないことに注意してくださいmysqlupgrade。
  3. 正常に起動したら、サーバーを再起動します。最初の段落で追加されたパラメーターはすでにありません。
  4. レプリケーションが正常に機能することを確認しSHOW SLAVE STATUSます。アプリケーションデータベース内のカウンターを持つテーブルが更新されていることを確認します。

これらはすべて非常にシンプルに見えます。開発アップデートは成功しました。OK、プロダクションの夜間アップグレードを安全に計画できます。

悲しみはありませんでした-製品版を更新しました


しかし、成功した開発経験を本番環境に移植することには、驚きがありました。

幸い、更新プロセス自体はレプリカから始まるため、問題が発生したため、作業を停止し、スナップショットからレプリカを復元しました。問題の調査は翌朝再スケジュールされました。次のエントリがログに表示されました。

2020-01-14T21:43:21.500563Z 2 [ERROR] [MY-012069] [InnoDB] table: t1 has 19 columns but InnoDB dictionary has 20 columns
2020-01-14T21:43:21.500722Z 2 [ERROR] [MY-010767] [Server] Error in fixing SE data for db1.t1
2020-01-14T21:43:24.208365Z 0 [ERROR] [MY-010022] [Server] Failed to Populate DD tables.
2020-01-14T21:43:24.208658Z 0 [ERROR] [MY-010119] [Server] Aborting

Googleのさまざまなメーリングリストのアーカイブを調査すると、このような問題はMySQLのバグが原因で発生することがわかった。むしろもののそれもユーティリティのバグであるmysqlcheckとmysqlsh。

MySQLが10進数フィールド（int、tinyintなど）のデータの表示方法を変更したことが判明したため、mysql-server内でそれらを格納する別の方法が使用されます。データベースが元々バージョン5.5または5.1であり、その後5.7にアップグレードした場合、OPTIMIZEいくつかのテーブルを作成する必要がある場合があります。その後、MySQLはデータファイルを更新し、現在のストレージ形式に転送します。

ユーティリティでこれを確認することもできますmysqlfrm：

mysqlfrm --diagnostic -vv /var/lib/mysql/db/table.frm
...
 'field_length': 8,
  'field_type': 246, #  
  'field_type_name': 'decimal',
  'flags': 3,
  'flags_extra': 67,
  'interval_nr': 0,
 'name': 'you_deciaml_column',
...

field_type0の 場合、テーブルでは古いタイプが使用されます-実行する必要がありますOPTIMIZE。ただし、値が246の場合は、すでに新しいタイプがあります。タイプの詳細については、コードを参照してください。

さらに、このバグは、システムテーブルにInnoDBテーブルが存在しない2番目の考えられる理由を考慮しています。テーブルINNODB_SYS_TABLESPACESがバージョン5.1で作成されている場合、それらのテーブルがシステムテーブルにないためです。アップグレード中の問題を回避するために、添付のSQLスクリプトを使用できます。

開発者にそのような問題がなかったのはなぜですか？ベースは定期的にプロダクションからコピーされます-したがって、テーブルが再作成されます。

残念ながら、実際に機能する大規模なデータベースでは、ユビキタスなデータベースを取得して実行するだけでは機能しませんOPTIMIZE。Percona-toolkitが役立ちます。pt-online-schema-changeユーティリティは、オンラインのOPTIMIZE操作に最適です。

更新された計画は次のとおりです。

  1. すべてのテーブルを最適化します。
  2. データベースのアップグレードを実行します。

これを確認し、同時に更新時間を確認するために、レプリカの1つを無効にし、すべてのテーブルで次のコマンドを実行しました。

pt-online-schema-change --critical-load Threads_running=150 --alter "ENGINE=InnoDB" --execute --chunk-size 100 --quiet --alter-foreign-keys-method auto h=127.0.0.1,u=root,p=${MYSQL_PASSWORD},D=db1,t=t1

ユーティリティはメインテーブルからデータをコピーする新しい一時テーブルを作成するため、テーブルは長いロックなしで更新されます。両方のテーブルが同じになると、元のテーブルはロックされ、新しいテーブルに置き換えられます。今回のケースでは、テストを実行したところ、すべてのテーブルの更新に約1日かかることを示しましたが、データをコピーすると、ディスクに過大な負荷がかかりました。

これを回避するため、本番環境--sleepではコマンドに値10の引数を追加しました。このパラメーターは、データのパケットを新しいテーブルに転送した後の待機の長さを制御します。これにより、実際に実行中のアプリケーションが応答時間を要求する場合に、負荷を軽減できます。

最適化を実行した後、更新は成功しました。

...しかし完全ではない！


更新後30分で、クライアントに問題が発生しました。ベースは非常に奇妙に動作しました：定期的に、接続のドロップが始まりました。ここでは、監視中のように見えたものです：



ノコギリグラフが原因MySQLサーバスレッドの一部を定期的にエラーで落ちたという事実のために、スクリーンショットに表示されます。アプリケーションにエラーが表示されました：

[PDOException] SQLSTATE[HY000] [2002] Connection refused

ログを簡単に調べたところ、mysqldデーモンがオペレーティングシステムから必要なリソースを取得できなかったことがわかりました。エラーを処理しているときに、システムの「孤立した」apparmorポリシーファイルが見つかりました。

# dpkg -S /etc/apparmor.d/cache/usr.sbin.mysqld
dpkg-query: no path found matching pattern /etc/apparmor.d/cache/usr.sbin.mysqld
# dpkg -S /etc/apparmor.d/local/usr.sbin.mysqld
dpkg-query: no path found matching pattern /etc/apparmor.d/local/usr.sbin.mysqld
# dpkg -S /etc/apparmor.d/usr.sbin.mysqld
mysql-server-5.7: /etc/apparmor.d/usr.sbin.mysqld
# dpkg -l mysql-server-5.7
rc  mysql-server-5.7 5.7.23-0ubuntu0.16.04.1      amd64

これらのファイルは、数年前のMySQL 5.7へのアップグレード中に形成され、リモートパッケージに属しています。ファイルを削除してapparmorサービスを再起動すると問題が解決しました：

systemctl stop apparmor
rm /etc/apparmor.d/cache/usr.sbin.mysqld
rm /etc/apparmor.d/local/usr.sbin.mysqld
rm /etc/apparmor.d/usr.sbin.mysqld
systemctl start apparmor

最終的に


最も単純な操作であっても、予期しない問題が発生する可能性があります。また、十分に検討された計画があっても、期待される結果が常に保証されるわけではありません。現在、どの更新計画にも、最近のアクションの結果として表示される可能性のある追加ファイルの必須のクリーニングが含まれています。

そして、このあまりプロではないグラフィック作品で、素晴らしい製品を提供してくれたPerconaに感謝します。
