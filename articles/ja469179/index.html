Kustomizeの簡単な紹介

注意ペレフ。：この記事は、印刷された7冊の書籍（主にVMware vSphere用）を執筆/共同執筆した上級ITエンジニアのスコットロウが執筆しました。彼は現在、クラウドコンピューティングとKubernetesを専門とするVMware子会社のHeptio（2016年に買収）で働いています。テキスト自体が、K8に最近含まれているKustomizeテクノロジーを使用したKubernetesの構成管理の容量が多く理解しやすい導入部として機能します。



Kustomizeは、ユーザーが「元のYAMLをそのまま使用可能なままにして、さまざまな目的のためにシンプルでテンプレートフリーのYAMLファイルをカスタマイズする」ことができるツールです（説明は GitHubのkustomizeリポジトリから直接借用します） Kustomizeは直接起動することも、Kubernetes 1.14以降でkubectl -kその機能にアクセスするために使用することもできます（Kubernetes 1.15以降、個別のバイナリはkubectlに組み込まれている機能よりも新しくなっています）。（注：また、最近のKubernetes 1.16リリースでは、kustomizeが kubeadmユーティリティでもサポートされています。）この投稿では、読者にkustomizeの基本を紹介します。

最も単純な形式/アプリケーションでは、kustomizeは単にリソースのコレクション（Kubernetesオブジェクトを定義するYAMLファイル：デプロイメント、サービスなど）に加えて、これらのリソースに対して行う必要のある変更に関する指示のリストです。makeがに含まれる命令セットを使用しMakefile、Dockerがからの命令に基づいてコンテナーをDockerfile構築するのと同様に、kustomizeはそれkustomization.yamlを使用して、ユーザーがリソースセットに加えたい変更についての命令を格納します。

以下はファイルの例ですkustomization.yaml。

resources:
- deployment.yaml
- service.yaml
namePrefix: dev-
namespace: development
commonLabels:
  environment: development

ファイル内のすべての可能なフィールドについて話そうとはしませんがkustomization.yaml（これはここによく書かれています）、特定の例について簡単に説明します。

  • resources , ( ) kustomize. deployment.yaml service.yaml ( ).
  • namePrefix kustomize' ( — dev-) name , resources. , Deployment' name nginx-deployment, kustomize dev-nginx-deployment.
  • namespace kustomize' . , Deployment Service development.
  • 最後に、フィールドにcommonLabelsはすべてのリソースに追加されるラベルのセットが含まれています。この例では、kustomizeは名前environmentと値を含むラベルをリソースに割り当てdevelopmentます。

ユーザーがkustomize build .ファイルkustomization.yamlと必要なリソース（つまり、ファイルdeployment.yamlとservice.yaml）を含むディレクトリで実行すると、出力で、で記述された変更を含むテキストを受け取りますkustomization.yaml。


注意 ペレフ。：kustomizeの「単純な」使用に関するプロジェクトドキュメント

の図変更をコミットする必要がある場合は、出力をリダイレクトできます。

kustomize build . > custom-config.yaml

出力は確定的であるため（同じ入力を使用すると、同じ出力が取得されます）、結果をファイルに保存できません。代わりに、すぐに別のコマンドに渡すことができます。

kustomize build . | kubectl apply -f -

kustomize関数へのアクセスは、kubectl -k（バージョン1.14 Kubernetes以降）からも取得できます。ただし、個別のkustomizeパッケージは、kubectlに統合されたパッケージよりも速く更新されることに注意してください（少なくとも、Kubernetes 1.15のリリースではこれが当てはまります）。

読者は、「ファイルを直接編集できるのに、なぜこれらすべての困難が必要なのか」と尋ねるかもしれません。すばらしい質問です。この例では、本当にできるのファイルを変更deployment.yamlし、service.yaml直接、彼らは誰かの他のプロジェクトのフォークは何ですか？ファイルを直接変更すると、ソース/ソースに変更が加えられたときにフォークをリベースすることが（まったく不可能ではないにしても）困難になります。 kustomizeを使用すると、これらの変更をファイルに集中化できます。kustomization.yaml元のファイルをそのまま残し、必要に応じてソースファイルのリベースを容易にします。

kustomizeの利点は、より複雑なユースケースで明らかになります。上記の例kustomization.yamlでは、リソースは同じディレクトリにあります。ただし、kustomizeは、基本構成とそのバリアント（オーバーレイとも呼ばれる）の多くがある場合の使用例をサポートします。たとえば、ユーザーは、私が例として使用したnginxの展開とサービスを利用して、それらのファイルの開発バージョン、ステージングバージョン、および製品バージョン（またはバリアント）を作成したいと考えていました。これを行うには、前述のオーバーレイと、実際には基本的なリソース自体が必要です。

オーバーレイとベースリソースの概念を説明する、ディレクトリが次のような構造になっているとします。

- base
  - deployment.yaml
  - service.yaml
  - kustomization.yaml
- overlays
  - dev
    - kustomization.yaml
  - staging
    - kustomization.yaml
  - prod
    - kustomization.yaml

ファイルでは、base/kustomization.yamlユーザーはフィールドresourcesを使用してkustomizeが有効にするリソースを宣言します。

各ファイルで、overlays/{dev,staging,prod}/kustomization.yamlユーザーはフィールドの基本構成を参照し、この環境にresources固有の変更を示します。たとえば、ファイルoverlays/dev/kustomization.yamlは前に示した例のようになります。

resources:
- ../../base
namePrefix: dev-
namespace: development
commonLabels:
  environment: development

この場合、ファイルoverlays/prod/kustomization.yamlは完全に異なる可能性があります。

resources:
- ../../base
namePrefix: prod-
namespace: production
commonLabels:
  environment: production
  sre-team: blue

ユーザーがkustomize build .ディレクトリoverlays/devで起動すると、kustomizeは開発オプションを生成します。あなたが実行した場合kustomize build .のディレクトリにoverlays/prod、本番のオプションを取得します。そして、これらすべて-元の（ベース）ファイルに変更を加えることなく、これらすべて-宣言的かつ決定的な方法で。基本構成とオーバーレイディレクトリをバージョン管理システムに直接コミットできます。これらのファイルに基づいて、いつでも必要な構成を再現できることがわかります。
