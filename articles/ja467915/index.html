Openshift内のpostgresの監視

ハブルの良い日住民！

今日私は、OpenShiftクラスター内のpostgresといくつかのエンティティをどのように監視したいのか、そしてどのようにそれを実行したのかをお話ししたいと思います。

入り口には、次のものがありました。

  • オープンシフト
  • 兜
  • プロメテウス


Javaアプリケーションを操作するために、すべてが非常にシンプルで透過的でしたが、より正確に言うと、次のようになります

。1）build.gradleに追加する

 implementation "io.micrometer:micrometer-registry-prometheus"

2）構成を使用してprometheusを起動します 

 - job_name: 'job-name'
    metrics_path: '/actuator/prometheus'
    scrape_interval: 5s
    kubernetes_sd_configs:
    - role: pod
      namespaces:
        names: 
          - 'name'

3）Grafanaへの表示の追加

名前空間で私たちの近くにあるベースを監視する時が来るまで、すべてが非常にシンプルで平凡でした（そうです、これは悪いことですが、誰もそれをしませんが、状況は異なります）。

どのように機能しますか？


postgresとprometheus自体を備えた炉に加えて、別のエンティティ-輸出業者が必要です。

抽象エクスポーターは、アプリケーションまたはサーバーからメトリックを収集するエージェントです。postgresの場合、エクスポーターはGoで記述されており、内部でsqlに基づいてスクリプトを実行するという原則に基づいて機能し、prometheusが結果を取得します。また、独自のメトリックを追加して、収集したメトリックを拡張することもできます。

これを次のように展開します（deployment.yamlの例、何にもバインドしない）：


---
apiVersion: extensions/v1beta1
kind: Deployment
metadata:
  name: postgres-exporter
  labels:
    app: {{ .Values.name }}
    monitoring: prometheus
spec:
  serviceName: {{ .Values.name }}
  replicas: 1
  revisionHistoryLimit: 5
  template:
    metadata:
      labels:
        app: postgres-exporter
        monitoring: prometheus
    spec:
      containers:
      - env:
        - name: DATA_SOURCE_URI
          value: postgresdb:5432/pstgr?sslmode=disable
        - name: DATA_SOURCE_USER
          value: postgres
        - name: DATA_SOURCE_PASS
          value: postgres
        resources:
          limits:
            cpu: 100m
            memory: 50Mi
          requests:
            cpu: 100m
            memory: 50Mi
        livenessProbe:
          tcpSocket:
            port: metrics
          initialDelaySeconds: 30
          periodSeconds: 30
        readinessProbe:
          tcpSocket:
            port: metrics
          initialDelaySeconds: 10
          periodSeconds: 30
        image: exporter
        name: postgres-exporter
        ports:
        - containerPort: 9187
          name: metrics

彼はまた、サービスとイメージストリームを必要としていました

。

次の部分をprometheus構成に追加します。

  - job_name: 'postgres_exporter'
    metrics_path: '/metrics'
    scrape_interval: 5s
    dns_sd_configs:
    - names:
      - 'postgres-exporter'
      type: 'A'
      port: 9187

そして、それはすべてうまくいきました、グラファンにこのすべての良さを加えて、結果を楽しむことは残ります。

独自のクエリを追加する機能に加えて、prometheusでは設定を変更して、より的を絞った必要なメトリックを収集できます。

同様の方法が行われました：

  • カフカ
  • Elasticsearch
  • モンゴ

PS名前、ポート、その他のすべてのデータは天井から取得され、情報は含まれません。
