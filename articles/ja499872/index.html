Java 9のモジュール性

Java 9の主な革新は、モジュール性の導入でした。この機能について多くの話があり、すべてを適切に完了するためにリリース日が数回延期されました。この投稿では、モジュールのメカニズムを提供するもの、および一般的に有用なJava 9によってもたらされるものについて説明します。ポストの基礎は私の同僚、セルゲイ・マルケビッチの報告でした。




このバージョンのJavaでモジュールを実装するために、いくつかのJEPとJSRを含むプロジェクト全体（プロジェクトジグソー）が割り当てられました。



公式ドキュメントのファンは、各JEPの詳細をこちらで読むことができます。

Project Jigsawの詳細


モジュール性を実装するプロジェクトジグソーは、2005年に開発され始めました。最初のJSR 277がリリースされ、すでに2008年にプロジェクトの直接作業が始まりました。リリースは2017年にのみ行われました。つまり、Javaでモジュールをねじ込むために、ほぼ10年かかりました。実際、これは、作業の完全なスケールと、モジュール性の実装中に行われた変更を強調しています。

開発者が設定した目標は何ですか？

  • 大規模なアプリケーションとライブラリの開発を促進します。
  • 一般にJava SE、特にJDKのセキュリティを向上させます。
  • アプリケーションのパフォーマンスを向上させます。
  • メモリーを使いすぎないように、JREのサイズを小さくして小さなデバイスで実行する機能を作成します。
  • JAR HELL（詳細は後ほど）。

Java 9がもたらした有用なもの


バージョン9より前は、JDKとJREはモノリシックでした。サイズはリリースごとに大きくなりました。Java 8はすでに数百メガバイトを占めていたため、Javaアプリケーションを実行できるようにするために、開発者は毎回このすべてを「実行する」必要がありました。のみだけでrt.jarの 60メガビット程度の重さを。さて、ここではスロースタートと高メモリ消費も追加しています。Javaの9は、救助に来ました。

ではJDK 9モジュール分離が導入されました。つまり、JDKは73個のモジュールに分割されました。そして、新しいバージョンごとに、これらのモジュールの数は増加しています。バージョン11では、この数は100に近くなっています。この分離により、開発者はJLINKユーティリティを作成できました。JLINKを使用すると、アプリケーションが実際に必要とする「必要な」モジュールのみを含むカスタムJREセットを作成できます。したがって、単純なアプリケーションと、最小（または小さい）モジュールのセットを含む一部のcustomJREは、最終的に20 MBに収まる可能性があります。これは朗報です。

モジュール


実は。モジュールとは？モジュールは、パッケージとリソースの新しいレベルの集約です（元の「関連するパッケージの一意に名前が付けられた、再利用可能なグループ、およびリソースとモジュール記述子」）。

モジュールは、パッケージとモジュール記述子
module-info.javaを含む JARファイルで提供されます。module-info.javaファイルには、モジュールの説明（
名前、依存関係、エクスポートされたパッケージ、消費されて提供されたサービス、リフレクションアクセスの権限）が含まれています。

モジュール記述子記述の例：

module java.sql {
    requires transitive java.logging;
    requires transitive java.transaction.xa;
    requires transitive java.xml;

    exports java.sql;
    exports javax.sql;

    uses java.sql.Driver;
}

module jdk.javadoc {
   requires java.xml;
   
   requires transitive java.compiler;
   requires transitive jdk.compiler;
   
   exports jdk.javadoc.doclet;
   
   provides java.util.spi.ToolProvider with
       jdk.javadoc.internal.tool.JavadocToolProvider;
   
   provides javax.tools.DocumentationTool with
       jdk.javadoc.internal.api.JavadocTool;
   
   provides javax.tools.Tool with
      jdk.javadoc.internal.api.JavadocTool;   
}

サービスについて少し：

抽象サービスMyServiceを実装するいくつかのモジュールが接続されているとしましょう。アプリケーションを構築するとき、必要なサービス実装モジュールを--module-pathに「ドラッグ」することにより、使用するサービス実装を決定する機会があります。

Iterable<MyService> services = 
        ServiceLoader.load(MyService.class);

したがって、返されるIteratorには、MyServiceインターフェースの実装のリストが含まれています。実際には、--module-pathにあるモジュールにあるすべての実装が含まれます。

原則として、なぜサービスが導入されたのですか？これらは、コードの使用方法を示すために必要です。つまり、意味的な役割があります。また、モジュール化はカプセル化とセキュリティに関するものです。実装をプライベートにして、リフレクションによる不正アクセスの可能性を排除できるからです。

また、サービスを使用するためのオプションの1つは、プラグインのかなり単純な実装です。アプリケーションのプラグインインターフェイスを実装し、モジュールを接続してそれらを操作できます。

モジュールを記述するための構文に戻りましょう：

9kiまでは、リフレクションを通じてほとんどすべてにアクセスでき、好きなことを好きなように行うことができました。また、すでに述べたように、9番目のバージョンでは、「違法な」リフレクションアクセスから身を守ることができます。

openを宣言することで、リフレクションアクセス用のモジュールを完全に開くことができます。

open module my.module {
}

または、opensを宣言することにより、リフレクションアクセスの任意のパッケージを指定できます。

module my.module {
    opens com.my.coolpackage;
}

ここでは、opens com.my.coolpackage to ...を使用することもできます。これにより、後で示すパッケージからcom.my.coolpackageパッケージへのリフレクションアクセスが提供されます。

Class-path vs module-path


モジュールの出現により、新しい概念-module-pathが登場しました。本質的に、これは同じclass-pathですが、モジュール用です。

モジュラーアプリケーションの起動は次のとおりです



。通常の起動モードでは、オプションとメインクラスへのフルパスを指定します。モジュールを操作する場合は、オプションと-mまたは-moduleパラメーターも指定します。これは、モジュールを実行することを示すだけです。つまり、アプリケーションを自動的にモジュールモードに変換します。次に、モジュールの名前と、モジュールからメインクラスへのパスを示します。

また、通常モードの場合、-cpおよび--class-pathオプションを使用することに慣れています。、モジュールモードでは、新しいパラメーター-pおよび--module-pathを規定します。これは、アプリケーションで使用されるモジュールへのパスを示します。

多くの場合、開発者はモジュールを使用する必要があると考えているため、バージョン9以降に切り替えないという事実に出会います。実際には、パラメーターを記述したりモジュールを使用したりせずに、他の新しいチップのみを使用して、アプリケーションを古いモードで実行できます。

瓶地獄


また、Jar Hellの問題について斜めに説明したいと思います。



一言で言えば、Jar Hellとは何ですか？たとえば、ある種のアプリケーションがあり、X ライブラリとYライブラリに依存しています。同時に、これらのライブラリはどちらもZライブラリに依存していますが、バージョンは異なります。Xはバージョン1に、Yはバージョン2に依存しています。まあ、バージョン2がバージョン1と下位互換性がある場合は問題ありません。そうでない場合は、バージョンの競合が発生していることは明らかです。つまり、同じライブラリを同じクラスローダーでメモリにロードできません。

この状況からどうやって抜け出すのですか？最初のJava以来​​、開発者が使用した標準的な方法があります。たとえば、exclude、誰かがMavenのプラグインを使用して、ライブラリのルートパッケージの名前を変更します。または、互換性のあるオプションを見つけるために、Xライブラリの異なるバージョンを探しています。

なぜ私なのか：最初のジグソープロトタイプは、モジュールにバージョンがあり、さまざまなClassLoadersを介して複数のバージョンをロードできることを暗示していましたが、後でそれは放棄されました。その結果、多くの人が待ち望んでいた「銀の弾丸」はうまくいかなかった。

しかし、箱から出してすぐに、私たちはそのような問題から少し安全でした。 Java 9は分割パッケージを無効にします-複数のモジュールに分割されたパッケージ。つまり、1つのモジュールにcom.my.coolpackageパッケージがある場合、同じアプリケーション内の別のモジュールでそれを使用することはできません。同じパッケージを含むモジュールでアプリケーションを起動すると、クラッシュします。この小さな改善により、Splitパッケージのダウンロードに関連する予期しない動作の可能性が排除されます。

また、モジュール自体に加えて、Jar Hell問題への対処に役立つレイヤーメカニズムまたはジグソーレイヤーもあります。

ジグソーレイヤーは、ローカルのモジュラーシステムとして定義できます。ここで、上記の分割パッケージは1つのジグソーレイヤーのフレームワーク内でのみ禁止されていることに注意してください。同じパッケージのモジュールには場所がありますが、それらは異なるレイヤーに属している必要があります。

それは次のようになります。



アプリケーションが起動するが、ときにブート層が作成され、ブートストラップによってロードされたプラットフォーム・モジュールを含む、追加のプラットフォーム・モジュールは、アプリケーションローダによってロードされたプラットフォーム・ローダおよび当社のアプリケーションモジュールによってロードされます。

いつでも、独自のレイヤーを作成し、そこに異なるバージョンのモジュールを「置く」ことができます。
