SQL Server 2019中的内存授予反馈

你好！预期“ MS SQL Server Developer”课程的开始，我们为您准备了另一种有趣的翻译。





如果优化器未正确计算执行查询所需的内存量，则可能是浪费内存，而该内存可能被另一个进程使用，或者将数据浪费到磁盘上（磁盘溢出）。为了解决此问题，Microsoft添加了“内存授予反馈”。在本文中，格雷格·拉森（Greg Larsen）解释了它是如何工作的。


SQL Server早期版本（SQL Server 2019或15.x之前）中的内存授予反馈仅针对以批处理模式运行的查询实现。批处理模式下的查询一次可以扫描并计算多达900行，而行模式下的查询一次只能处理一行。在15.x版中，内存授权反馈已扩展为支持字符串模式下的查询。



什么是内存授予反馈？这是调整查询所需的内存计算的过程，要考虑到查询在先前执行期间使用了多少内存。这意味着，如果缓存的查询在上次运行时使用了过多的内存，则SQL Server将在其下一次运行时减少内存分配。或者，如果SQL Server由于上次分配的内存不足而检测到使用磁盘的查询，则它将增加查询的内存。关于内存授予的反馈的目的是每次执行查询时都要调整内存需求，直到查询使用与处理的行数相对应的内存量为止。



内存授予反馈 功能（在批处理和字符串模式下）是智能查询处理功能系列（IQP，智能查询处理）的一部分。图1显示了一个图表，该图表显示了Azure SQL数据库和SQL Server 2019中存在的所有IQP功能以及Azure SQL数据库和SQL Server 2017的早期版本中包含的自适应查询处理的原始功能。

为什么对所需内存的估计可能是错误的



有许多因素会影响请求所需的内存量。在计算所需的总内存时，查询优化器必须在执行方面考虑每个操作的内存需求。这是优化器在计算执行查询所需的内存时要考虑的一些参数：



  • 查询中涉及的表的基数
  • 可选音箱
  • 估计的行大小
  • 是否需要排序和/或联接数据
  • 查询是否并行运行


为了正确地计算所需的内存，SQL Server必须考虑执行查询时可能需要内存的许多方面。如果分配了过多的内存，则SQL Server在其他进程可以使用它时会浪费内存。否则，如果分配的内存不足，则请求会将数据合并到磁盘，这是一项昂贵的操作。



SQL Server并不总是擅长于第一次计算内存。统计信息过时或不准确是SQL Server计算错误内存量的最常见原因。但是，即使统计信息正确，SQL Server也会由于其他一些原因高估或低估所需的内存量。这些原因之一是SQL Server无法正确评估返回列的估计行大小。在下面的示例中，我们将看到由于三个NVARCHAR（MAX）列（始终为NULL）而导致行的尺寸过大如何导致查询接收到过多的授予警告。



但是现在，如果SQL Server在第一次执行查询时错误地计算了所需的内存，则对于后续执行，将使用内存授予反馈。每次执行查询时，都会分析有关该查询先前执行的已分配内存的信息。如果在前一次启动时内存分配不正确，则数据库引擎会更正内存分配。

当所需的内存量不稳定时



有些请求的每次执行都需要不同的内存量。如果发生这种情况，则SQL Server将停止尝试调整分配的内存的大小，并禁用对此类请求的反馈。发生这种情况的情况之一是启动存储过程，每次执行时都会向其中传递不同的参数值。



为了在SQL Server禁用对内存授予的反馈时重现这种情况，我们多次执行具有不同参数的存储过程。因此，在每次执行时，过程内的查询将需要不同的内存量，这取决于返回的记录数（取决于参数）。清单4显示了存储过程的代码。 



USE WideWorldImporters;
GO
CREATE OR ALTER PROC TestProc (@KeyValue int)
AS
SELECT * FROM Sales.Orders
WHERE OrderID > 1 and OrderID < @KeyValue
ORDER BY OrderDate


清单4.测试存储过程



存储过程接受带有name的参数@KeyValue



。清单5中显示了用于测试存储过程的代码。 



DECLARE @I INT = 1;
DECLARE @TestKeyValue INT;
WHILE @I < 35
BEGIN
	IF @I % 2 = 0
	 EXEC TestProc @KeyValue = 20000;
	ELSE
	 EXEC TestProc @KeyValue = 100;
	SET @I = @I + 1;
END

禁用内存授予反馈



如果出于任何原因要在行模式（“内存授予反馈-行模式”）中禁用内存授予反馈，则有几种方法可以执行此操作。第一个是将数据库的兼容性级别设置为小于150。此方法只有一个问题：15.x版中出现的所有其他功能也将被禁用。或者，您可以通过option禁用数据库级别的反馈ROW_MODE_MEMORY_GRANT_FEEDBACK



。 



USE WideWorldImporters;
GO
ALTER DATABASE SCOPED CONFIGURATION SET ROW_MODE_MEMORY_GRANT_FEEDBACK = OFF;
GO

优化请求所需内存的计算



批处理模式下有关内存授予的反馈已有很长时间了。但是只有在15.x版中，此反馈才可用于行模式下的查询。启用以字符串方式进行查询反馈的所有操作是将数据库兼容性级别设置为150。如果您使用当前版本的Azure SQL数据库或SQL Server 2019，这当然将起作用。启用内存授予反馈功能时在字符串模式下，SQL Server使用来自缓存查询的先前执行的内存分配信息来调整当前执行的内存分配。每次执行查询时，通过反馈更正所需的内存，迟早会自动导致计算出正确的内存量。
