Chapel



Chapel uses the forall loop to parallelize threads. C-pointers to each element are also used, and not a standard reference to arrays, and guided iteration is used according to indexes: 



proc calculateKernelMatrix(K, data: [?D] ?T)
{
  var n = D.dim(0).last;
  var p = D.dim(1).last;
  var E: domain(2) = {D.dim(0), D.dim(0)};
  var mat: [E] T;
  var rowPointers: [1..n] c_ptr(T) =
    forall i in 1..n do c_ptrTo(data[i, 1]);

  forall j in guided(1..n by -1) {
    for i in j..n {
      mat[i, j] = K.kernel(rowPointers[i], rowPointers[j], p);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


Chapel code was the most difficult to optimize for performance and required the most code changes.



D



To parallelize the D code, taskPool streams from the std.parallel package are used . D code has undergone the least number of changes to optimize performance - it is of great benefit to use a specific compiler and selected compilation keys (discussed later). My Matrix implementation allows me to select columns by reference using refColumnSelect . 



auto calculateKernelMatrix(alias K, T)(K!(T) kernel, Matrix!(T) data)
{
  long n = data.ncol;
  auto mat = Matrix!(T)(n, n);

  foreach(j; taskPool.parallel(iota(n)))
  {
    auto arrj = data.refColumnSelect(j).array;
    foreach(long i; j..n)
    {
      mat[i, j] = kernel(data.refColumnSelect(i).array, arrj);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


Julia



Julia code uses a macro threads for parallelizing code and macros viewsfor references to arrays. The only thing that is confusing with arrays in Julia is their reference status. Sometimes, as in this case, arrays will behave as value objects, and they must be referenced using a macroviewsotherwise they will generate copies. In other cases, they behave like reference objects, for example, when passing them to a function. It can be a little tricky to deal with, because you don‚Äôt always know which set of operations the copy will generate, but where it happensviewsprovides a good solution.



The Symmetric type saves a little extra work needed to reflect in the matrix. 



function calculateKernelMatrix(Kernel::K, data::Array{T}) where {K <: AbstractKernel,T <: AbstractFloat}
  n = size(data)[2]
  mat = zeros(T, n, n)
  @threads for j in 1:n
      @views for i in j:n
          mat[i,j] = kernel(Kernel, data[:, i], data[:, j])
      end
  end
  return Symmetric(mat, :L)
end


The @ bounds and @ simd macros in the main functions were used to disable border checking and apply SIMD optimization to calculations: 



struct DotProduct <: AbstractKernel end
@inline function kernel(K::DotProduct, x::AbstractArray{T, N}, y::AbstractArray{T, N}) where {T,N}
  ret = zero(T)
  m = length(x)
  @inbounds @simd for k in 1:m
      ret += x[k] * y[k]
  end
  return ret
end


These optimizations give quite a noticeable increase, but are very easy to use.

Performance optimization



The process of optimizing performance in all three languages ‚Äã‚Äãwas very different, and all three communities were very useful in this process. But there were general points.



  • Static scheduling of kernel functions instead of using polymorphism. This means that when transferring the kernel function, parametric (compilation time) polymorphism is used, and not dynamic (runtime) polymorphism, in which scheduling with virtual functions entails overhead.
  • Using views / links instead of copying data in multi-threaded mode is of great importance.
  • Parallelization of calculations is of great importance.
  • Knowing that an array is basic to a row / column, and using it in calculations is of utmost importance.
  • Boundary checks and compiler optimizations make a huge difference, especially in Chapel and D.
  • The inclusion of SIMD in D and Julia has contributed to performance. In D, this was done using the -mcpu = native flag, and in Julia, this was done using the @ simd macro .


From the point of view of language specifics, the most difficult was the transition to a productive code in Chapel, and the code in Chapel changed the most: from simple to read operations with arrays to the use of pointers and controlled iterations. But on the compiler side, it was relatively easy to add --fast and get a big performance boost.



The D code has changed very little, and most of the performance has been obtained by choosing the compiler and its optimization flags. The LDC compiler is rich in performance optimization features. It has 8-O optimization levels, but some of them are repeated. For example, -O, -O3 and -O5 are identical, and there are countless other flags that affect performance. In this case, the flags -O5 --boundscheck = off -ffast-math were used, which are aggressive compiler optimizations, bounds checking, and LLVM's fast-math, and -mcpu = native to enable vectorization instructions.



In Julia, macros in the previously reviewed changes markedly improved performance, but they were not too confusing. I tried changing the optimization level -O, but that did not improve performance.

Interactivity



Programmers need a fast coding / compilation / result cycle during development to quickly observe the results and conclusions in order to move forward or make the necessary changes. The Julia interpreter is the best for this and offers a smooth and multi-functional design, and D is close to that. This coding / compilation / result loop can be slow even when compiling a little code. There are three compilers in D: the standard DMD compiler, the LLVM LDC compiler, and the GCC GDC compiler. DMD and LDC compilers were used in this development process. DMD compiles very quickly, which is very convenient for development. And LDC does a great job of creating fast code. Documentation and Examples



One way to compare documentation in different languages ‚Äã‚Äãis to compare all of them with the official Python documentation, which is the gold standard for programming languages. It combines examples with formal definitions and instructions in a simple and user-friendly way. Since many programmers are familiar with Python documentation, this approach gives an idea of ‚Äã‚Äãhow they compare.



The Julia documentation is closest in quality to the Python documentation and gives the user a very smooth, detailed and relatively painless transition to the language. It also has a rich ecosystem of blogs, and topics on many aspects of the language are easily accessible. The official D documentation is not so good and can be complicated and disappointing, however there is a very good free book ‚ÄúProgramming in D‚Äù, which is an excellent introduction to the language, but not a single book can cover the entire programming language and not so many source code examples for advanced topics. The Chapel documentation is good enough to do something, although the examples presented vary in availability and quality. Often a programmer needs to know where to look. A good topic for comparison is the file I / O libraries in Chapel, D, and Julia. The Chapel I / O library contains too few examples, but is relatively clear and simple; D input / output is distributed across several modules, and documentation is more difficult to follow; Julia's I / O documentation contains many examples and is clear and easy to understand.



Perhaps one of the factors influencing the adoption of Chapel is the lack of examples - since arrays have a non-standard interface, the user has to work hard to find out. Despite the fact that the documentation in D may not be so good in places, the language has many similarities with C and C ++, therefore it gets away with more scarce documentation.

‚ÄúArrays‚Äù here do not refer to arrays in the style of C and C ++ available in D, but to mathematical arrays. Julia and Chapel come with array support, but D does not, but it does have a Mir library that contains multidimensional arrays (ndslice). In the implementation of calculating the matrix core, I wrote my matrix object in D, which is easy if you understand the principle, but this is not what the user wants to do. However, D has a linear Lubeck algebra library, which has impressive performance features and interfaces with all common BLAS implementations. Arrays Julia, of course, the most simple and familiar. Chapel arrays are more complex for entry-level than Julia arrays, but they are designed to run on single-core, multi-core systems and computer clusters using a single or very similar code, which is a good unique point of attraction.



Tongue power



Since Julia is a dynamic programming language, some may say, ‚ÄúWell, Julia is a dynamic language that is much more permissive than static programming languages, so the debate is over,‚Äù but it‚Äôs much more complicated. Static type systems have their power. Julia has a type system that is similar in nature to type systems from static languages, so you can write code as if you were using a static language, but you can do things reserved only for dynamic languages. It has highly developed syntax for general and meta-programming, as well as powerful macros. It also has a very flexible object system and multiple dispatch. This combination of features makes Julia the most powerful language of the three.



D was conceived as a replacement for C ++ and took a lot from C ++ (and also borrowed from Java), but makes template programming and compile-time calculations (CTFE) much more user-friendly than in C ++. This is a language with single dispatch (although there is a package with multimethods). Instead of macros, D has a mixin for strings and patterns that serve a similar purpose.



Chapel has generic support and nascent support for OOP with single dispatch, it lacks macro support, and in these matters it is not as mature as D or Julia.

In the Chapel concurrency model, much more attention is paid to data concurrency, but there are tools for task concurrency and cluster-based concurrency support is provided.



Julia has good support for both concurrency and concurrency.



D has industrially strong support for concurrency and concurrency, although its support for multithreading is much less well documented by examples.

Summary



If you are a beginner programmer who writes numerical algorithms and performs scientific calculations, and want a fast and easy to use language, Julia is your best choice. If you are an experienced programmer working in the same field, Julia is still a great option. If you want a more traditional, ‚Äúindustrial‚Äù, statically compiled, high-performance language with all the ‚Äúwhistle-blowers‚Äù, but want something more productive, safe and less painful than C ++, then D is your best option. You can write ‚Äúanything‚Äù in D and get great performance thanks to its compilers. If you need to calculate arrays on clusters, then Chapel is probably the most convenient solution.



In terms of gross performance in this task, D was the winner, clearly demonstrating better performance in 5 out of 9 reference tasks. The study showed that the label of Julia as a high-performance language is more than just hype - it has its own advantages in comparison with highly competitive languages. It was harder than expected to get competitive performance from Chapel - the Chapel team needed a lot of research to come up with the current solution. However, as the Chapel language matures, we can see further improvement.
