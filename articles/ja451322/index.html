この本「Java Concurrency in Practice」は、Java 8と11の時代に関連していますか？

こんにちは！

本日の翻訳は、ブライアン・ゲッツ著の時代を超えた著書「Java Concurrency in実際」の重要性を改めて思い起こさせます。


オリジナルのこの記事へのコメントの日付は、著者が初めてそれを更新して再発行していないことを示しています。したがって、マニングからJava 8 in Actionのタイトルで発行された、ラウル=ガブリエルウルム、マリオフスコ、およびアランマイクロフトの記事に記載されている本へのリンクも更新することができました。 Modern Javaと呼ばれる新版の翻訳を準備しています。しかし、今のところ、古典について話しましょう。あなたは猫に招待されています。


ニックネームのShobhitを持つ読者が、中級レベルのプログラマー向けの12の高度なJavaの本についての記事へのコメントで私にこの質問をしました-パート1。質問は本当に良いです。私たちの時代に誰かが「Java Concurrency in Practice」を読むことを勧めたとき、多くのJavaプログラマーは同様の疑問を抱いたと思います。この本が最初にリリースされたとき、2006年、Javaの世界全体はJava 1.5で行われた競争力の革新をまだ理解できていませんでした。私は、マルチスレッド化と競争力に対するJavaの組み込みサポートを改善するために最初の真剣な試みがなされたと思います。当時、多くのプログラマーはAPIに登場した新しいツール、例えばCountDownLatch、CyclicBarrier、ConcurrentHashMapその他多数。この本は、これらのツールを操作するためのデバッグされた紹介としてそれらを提供し、高性能で競争力のあるJavaアプリケーションを作成するためにそれらをどのように使用できるかを説明しました。

これはまさにこの本の一般的な考えであり、この脈で多くの開発者は、「Java Concurrency in Practice」と尋ねた場合に説明しますか？ただし、この本の解釈は少し異なります。そのため、Javaに精通しているすべての初心者、または競争に関連する概念を習得したい中級レベルの開発者にもお勧めします。

この本で紹介する最も重要な資料は、競合プログラミングの明確な概念と基礎、特に、可視性、順序付け、スレッドの安全性、不変性、並行性など

彼女はまた、ほとんどの場合、競合するJavaアプリケーションのスペルが間違っている理由、およびJavaプログラマーが競合状態、デッドロック、アクティブロック、メモリ干渉、および単に誤った計算など、マルチスレッドの問題につながる一般的な間違いを犯している理由についても説明します。

この本では、問題の正しい高品質な解決策を読者に紹介する前に、「間違った方法」の説明に添えられている絵文字を使用しています。そのため、この本は、多くのJava開発者が抱えている誤解に対処するだけでなく、マルチスレッド化と競合の扱い方に関する正しい情報をJavaコミュニティに浸透させるのにも役立ちます。

マルチスレッド化と競争が複雑であることは間違いありません。それらをコードに正しく実装することは容易ではなく、理解して説明することも同様に困難です。さまざまなスレッドが同じコード要素とどのように相互作用し、さまざまなデータを操作するのかを明確に表現できないプログラマーがたくさんいます。

古典的な再帰の例と同様に、一部のプログラマーにとってマルチスレッド化は非常に簡単ですが、アプリケーションシナリオでマルチスレッド化を理解して適用することは困難です。

Javaの世界の発展に対するJavaの本の並行性の最大の貢献は、競争を単純化することではなく、欠けていた真実で正確な情報を提供することです。それ以来、私は多くのインタビューを実施してきましたが、プログラマーが常にフローデバイスとフローデバイスがどのように機能するかを常に正確に想像しているわけではないことを知っています。

4〜5年間Javaで作業できたプログラマーの多くは、揮発性変数がどのように配置されているかを理解していません。彼らが知っているのは、可変変数を使用する場合、メインメモリにどのような値があるかを各比較で確認する必要があることです。これは真実ですが、真実の一部にすぎません。

変数変数がコードの順序とそのコアでの計算命令の実行にどのように影響するかについては、Javaメモリモデルについては説明されていません。これは、最適化に動的コンパイル（JIT）とJava仮想マシン（JVM）を使用することです。このような最適化は、微妙な論理エラーを引き起こす可能性があります。

他のスレッドから変数にアクセスする前に、1つのスレッドで何が行われたかを可変変数がどのように確認できるかなどはわかりません。メモリバリアとは何か、それが可視性にどのように影響するかは誰もが知っているわけではありません。

本のJava Concurrency in Practiceから、多くのJavaプログラマーがこれらすべての概念を学びました。私自身、それを読むまでは、マルチスレッド化と競争力の多くの重要な問題、特に最終的な変数のシーケンス、可視性と暗黙の影響、および安全な公開について、大いに誤解されていたことを認めなければなりません。この本は私がこれを理解するのに役立ちました。

ちなみに、本の一部のセクションがあなたにとって少しわかりにくいと思われる場合は、あなただけではなく、私を信じてください。ここでは、Java同時実行バンドルのコースで本の資料を簡略化された形で提示したハインツカブツ博士に感謝します。



この資料が複雑すぎると感じたとしても、ハインツにはマスタースレッドというコースがありますこれは、平均的なJavaプログラマーのマルチスレッドを理解するのに役立ちます。

次に、Java 8についてお話ししましょう。つまり、Java 1.5からJava 8に変更された点です。JDKには、競争力を導入し、より競争力のあるJavaアプリケーションを設計するための多くの新しいツールが登場しました。fork-joinプールは JDK 7 、Java 8-で登場しましたCompleteableFutures。 Java 8以降、ラムダ式によって提供される新しいより機能的なプログラミングスタイルが修正されたことは、さらに重要です。

ストリームと並列ストリームもあり、開発者はプログラミングせずに競争力を活用できます。アプリケーション開発者から競争の実装を取り、それをAPIの開発者に委任するという全体的なアイデアは、Javaでの競争の状況をわずかに簡素化し、その実装のリスクを減らします。

これは、Javaでは、スレッド、同期されたキーワード、または待機と通知のメソッドに関連する1行のコードを記述することなく、いくつかのメソッドを使用して複数のスレッドで一括操作を実行できることも意味します。

間違いなく、どのJava開発者もテクノロジーの開発に追いつくためにこれらの新しいツールを学ぶ必要があります-もちろん、これは「モダンJavaインアクション。 "これは、Java言語のすべての革新を紹介するだけでなく、日常のタスクでそれらを使用する方法を学び、これらの革新の動機を理解し、最新のJava言語の一般的な考え方を理解するのにも役立ちます。



現在の形式のJava Concurrency in Practiceは、これらの重要な概念とツールのすべてを網羅しているわけではありませんが、スレッド、並行処理、およびマルチスレッドに関連するJava言語の主要機能を探索するのに非常に貴重です。

ゲッツの本は、アプリケーション開発におけるJavaの最大の長所であるマルチスレッド化と競争力を学び、習得したいJava開発者にとって、まだ必読です。
