BazelでC ++をビルドする

紹介と動機


最近、cbrakeとc ++が友だちであるという投稿がHabrに掲載されました。ヘッダーのみのライブラリを収集する方法の例だけでなく、少なくともいくつかの新しいビルドシステム（bazel、buck、gnなど）の概要もありません。私と同じように、2k20でC ++で書く場合は、c ++プロジェクトのビルドシステムとしてbazelについて理解することをお勧めします。

cmakeや他の既存のシステムが何に悪いのかという問題は残し、bazel自体に何ができるかに集中します。特にあなたに最適なものを決定するために、私は特にあなたに任せます。

定義と動機から始めましょう。Bazelは、c ++プロジェクトをビルドできる多言語のGoogleビルドシステムです。なぜ別のビルドシステムを検討する必要があるのでしょうか。まず、Tensorflow、Kubernetes、Gtestなどのいくつかの大規模プロジェクトが既に彼女に提供されているため、それらと統合するには、bazelを使用できる必要があります。第二に、グーグルのほかに、bazelはまだspaceX、nvidia、および他の企業を使用しており、bazelconでのパフォーマンスで判断しています。最後に、bazelはgithubでかなり人気のあるオープンソースプロジェクトであるため、一見の価値があり、試してみる価値があります。

例1.ささいなこと


main.ccがあり、それをコンパイルする必要があります：

main.cc

#include <iostream>

int main() {
   std::cout << "Hello, habr" << std::endl;
   return 0;
}

すべてはワークスペース宣言から始まります。 bazelワークスペースに関しては、これはすべてのソースファイルが配置されているディレクトリです。このワークスペースを指定するには、必要なディレクトリにWORKSPACEという名前の空のファイルを作成する必要があります。通常、これはsrcディレクトリです。

コードをbazelで編成するための最小単位はパッケージです。パッケージは、ソースディレクトリと、これらのソースのアセンブル方法を記述する特別なBUILDファイルによって定義されます。

プロジェクトにメインパッケージを追加し



ます。BUILDファイルで、メインからビルドするものを記述する必要があります。当然、実行可能バイナリをコンパイルしたいので、cc_binaryルールを使用します。 Bazelはすでに標準でC ++をサポートしているため、C ++の目標を構築するための特定のルールセットがすでにあります。残りの部分については後で説明します。

cc_binaryルールをBUILDファイルに追加します。このルールには、実行可能ファイルとコンパイラーに渡されるソースの配列を持つ名前が付けられています。これはすべて、切り捨てられたpythonであるstarlarkで記述されています。

cc_binary(
  name = "hello_world",
  srcs = "main.cc"
)

cmakeとは異なり、Bazelはコマンドに基づいていませんが、ルールを通じて依存関係を宣言的に記述できます。基本的に、ルールは複数のアーティファクトを特定の操作に関連付けます。それらを使用して、bazelはコマンドグラフを作成し、それをキャッシュして実行します。この例では、ソースファイルmain.ccはコンパイル操作に関連付けられており、その結果はアーティファクトhello_world（バイナリ実行可能ファイル）になります。

実行ファイルを取得するには、workspaceのあるディレクトリに移動して、次のように入力する必要があります。

bazel build //main:hello_world

ビルドシステムは、プロジェクトのルートから開始して、ビルドコマンドと目標へのパスを受け入れます。

結果のバイナリはbazel-bin / main / hello_worldにあります。

例2.ライブラリを使用してビルドする


幸い、そのような単純なプロジェクトは誰も必要としないので、プロジェクトに機能を追加する方法を見てみましょう。個別にビルドされ、メインにリンクされるライブラリを追加します。

それをSquare、明確な二乗関数を提供するライブラリとしましょう。新しいライブラリを追加することは、新しいパッケージを追加することを意味します。これを正方形と呼びましょう。



square.h

#ifndef SQUQRE_SQUARE_H_
#define SQUQRE_SQUARE_H_

int Square(int x);

#endif // SQUQRE_SQUARE_H_

square.cc
#include "square/square.h"

int Square(int x) {
  return x * x;
}

ヘッダーファイルの接続に注意してください。ファイルが同じディレクトリにある場合でも、ワークスペースからのパスを介して行います。このアプローチは、Google C ++スタイルガイドから継承されたChromeコードスタイルガイドで採用されています。この方法により、ヘッダーファイルの接続元をすぐに理解できます。心配しないでください。ファイルがあり、bazelはヘッダーファイルを検索するためのパスを追加しますが、このルールに従わない場合、bazelのビルド中にヘッダーファイルが見つからない可能性があります。

ライブラリのBUILDファイルに、cc_libraryライブラリを構築するためのルールを記述します。

cc_library(
  name = "square",
  srcs = ["square.cc"],
  hdrs = ["square.h"],
  visibility = ["//visibility:public"]
)

ここでは、ソースファイルとヘッダーファイルを個別に一覧表示し、公開するかどうかを指定します。後者は、プロジェクトのどこからでもライブラリに依存できるようにするために必要です。

main.ccでは、ライブラリを使用します。

#include <iostream>
#include "square/square.h"

int main() {
  std::cout << "Hello, bazel!" << std::endl;
  std::cout << Square(2) << std::endl;
  return 0;
}

ここでも、ワークスペースからのパスを介してライブラリのヘッダーファイルをインクルードしていることに注目します。これはすでに絶対に必要です。bazelはフードの下でLinuxコンテナーを使用してアセンブリの最小レベルの気密性を確保しているため、ワークスペースからのパスを介して配置されるように正方形のライブラリヘッダーファイルをマウントします。

また、正方形ライブラリのmainのアセンブリルールの依存関係について説明します。

cc_binary(
  name = "hello_world",
  srcs = ["main.cc"],
  deps = ["//square:square"]
)

私たちのプログラム全体は、ライブラリを使用しない場合と同じ方法でアセンブルされます。bazel自体が依存関係を理解し​​、グラフを構築し、結果をキャッシュし、再構築する必要があるものだけを再構築します。

bazel build //main:hello_world

例3.テストを接続する


テストなしで生きるには？ありえない！ちなみにすでにbazelでのアセンブリをサポートしているbazel GTestに接続するには、外部依存関係を追加する必要があります。これはWORKSPACEファイルで行われます。

load("@bazel_tools//tools/build_defs/repo:git.bzl", "git_repository")

git_repository(
  name = "googletest",
  remote = "https://github.com/google/googletest",
  tag = "release-1.8.1"
)

流行に敏感な人のように、彼らはgit_repositoryルールを接続し、ダウンロードするバージョンをbazelに伝えました。

次に、テストテスト用の個別のパッケージを作成し、テストをライブラリに追加します

。square_unittest.cc

#include "gtest/gtest.h"
#include "square/square.h"

TEST(SquareTests, Basics) {
    EXPECT_EQ(Square(-1), 1);
    EXPECT_EQ(Square(1), 1);
    EXPECT_EQ(Square(2), 4);
}

次に、テストのルールを定義します。

cc_test(
  name = "unittests",
  srcs = ["square_unittest.cc"],
  deps = [
   "//square:square",
   "@googletest//:gtest_main"
  ]
)

ライブラリとgtest_mainへの依存関係を追加して、gtestライブラリ自体がランチャー実装を提供するようにしました。

テストは次のコマンドで実行されます。

bazel test //test:unittests

BazelはGTest自体をダウンロードしてビルドし、テストに必要なすべてのものをリンクして、テスト自体を実行します。

bazelはコードカバレッジの方法も知っていることを述べました。

bazel coverage //test:unittests

テストをデバッグする必要がある場合は、デバッグモードで次のような文字を使用してすべてをコンパイルできます。

bazel build //test:unittests --compilation_mode=dbg -s

例4.バゼルの方法がわからない他のライブラリを接続する


もちろん、世界はbazelだけで構築されているわけではないため、他のライブラリも接続できる必要があります。最近、私のプロジェクトでは、コマンドライン引数を解析するためのライブラリが必要でした。まあ、あなた自身のそのようなライブラリーを2k20で私に書いて、主な仕事から気を取られないように。getopsのようなハーフメジャーを使用したり、プロジェクトにブーストをドラッグしたりしたくありません。

広告ではなく、CLI11ライブラリを接続します。これは、stl標準C ++ 11のみを使用し、多かれ少なかれ便利なインターフェイスを提供します。

ライブラリはヘッダーのみなので、接続が特に簡単です。

WORKSPACEで外部依存関係を接続します。

load("@bazel_tools//tools/build_defs/repo:http.bzl", "http_file")

http_file(
  name = "CLI11",
  downloaded_file_path = "CLI11.hpp",
  urls = ["https://github.com/CLIUtils/CLI11/releases/download/v1.9.0/CLI11.hpp"],
  sha256 = "6f0a1d8846ed7fa4c2b66da3eb252aa03d27170258df...",
)

このライブラリへの依存関係を構築するために、サードパーティのディレクトリとCLI11パッケージを追加します。

cc_library(
  name = "CLI11",
  hdrs = ["@CLI11//file"],
  strip_include_prefix = "/external/CLI11/file",
  include_prefix = "CLI11",
  linkstatic = True,
  visibility = ["//visibility:public"],
)

Bazelはデフォルトでパス/ external / CLI11に沿ってライブラリファイルを検索するため、CLI11 /を介して接続するようにパスを少し変更します。

main.cc

#include <iostream>

#include "CLI11/CLI11.hpp"
#include "square/square.h"

int main() {
  std::cout << "Hello, bazel!" << std::endl;
  std::cout << Square(2) << std::endl;
  return 0;
}

メインに応じて、「// third_party / CLI11：CLI11」を追加すると、すべてが機能し始めます。
私はあなたのことは知りませんが、なじみのないライブラリを接続して、この形式のC ++プロジェクトで使用することは私を喜ばせます。

はい、ヘッダーのみのライブラリを使用すると、すべてが単純であると言えますが、ヘッダーのみではないライブラリを使用すると、まだbazelでビルドされていないため、すべてが同じくらい簡単です。単にhttp_archiveまたはgit_repositoryを介してそれをダウンロードし、ライブラリの構築方法を記述したサードパーティディレクトリの外部BUILDファイルを追加します。Bazelは、cmake_externalルールを介して、任意のcmdの呼び出し、さらにはcmakeの呼び出しをサポートしています。

例5.スクリプトと自動化


自動化用のスクリプトなしで2k20のベアC ++のプロジェクトが必要なのは誰ですか？通常、このようなスクリプトは、パフォーマンステストを実行したり、成果物をCIのどこかにデプロイしたりするために必要です。まあ、通常それらはpythonで書かれています。

このため、bazelも適しています。これは、ほとんどすべての一般的な言語で使用でき、実際のプロジェクトで頻繁に見られるさまざまなプログラミング言語からそのようなsolyankaを収集するように設計されているためです。

メインを実行するPythonスクリプトを接続してみましょう。

perf-testsパッケージを追加します。

py_binary(
  name = "perf_tests",
  srcs = ["perf_tests.py"],
  data = [
    "//main:hello_world",
  ],
)

データ依存関係として、hello_worldバイナリ依存関係を追加します。

perf_tests.py

import subprocess
import time

start_time = time.time()
process = subprocess.run(['main/hello_world, 'param1', ],
                         stdout=subprocess.PIPE,
                         universal_newlines=True)
end_time = time.time()
print("--- %s seconds ---" % (end_time - start_time))

テストを実行するには、次のように記述します。
bazel run //perf-tests:perf_tests
