Laravelでの単体テスト

コミュニティでの議論の中で、Laravelでの単体テストは間違っていて複雑であり、テスト自体は長く、何のメリットもないという意見をよく耳にします。そのため、これらのテストを記述して機能テストのみに限定する人はほとんどいません。また、単体テストの使用は0
になりがちです。また、一度考えてみましたが、一度考えて自分自身に問いかけたところ、調理方法がわからないのではないでしょうか。


しばらくの間、私は理解し、出口でユニットテストについて新たに理解しました。そして、テストは明確で、フレンドリーで、速くなり、私を助け始めました。
私の理解をコミュニティと共有し、このトピックをよりよく理解して、テストをさらに改善したいと思います。


ちょっとした哲学と限界


Laravelは、場所によっては一種のフレームワークです。特にファサードと雄弁の面で。これらのポイントの議論や非難には触れませんが、ユニットテストとどのように組み合わせるかを示します。
メインコードを記述した後（または同時に）テストを記述します。おそらく、私のアプローチはTDDアプローチと互換性がないか、部分的な調整が必要になります。


テストを書く前に私が自問する最も重要な質問は、「正確に何をテストしたいか」です。これは重要な問題です。ユニットテストとプロジェクトコード自体の記述に関する私の考えを再考することができたのは、このアイデアでした。


テストは安定しており、環境への依存度は最小限である必要があります。変異を加えたときにテストが失敗した場合、おそらくそれらは良好です。逆に、彼らが落ちなければ、彼らはおそらくあまり良くありません。


Laravelはそのままの状態で3種類のテストをサポートしています。

  • 特徴
  • 単位

主に単体テストについてお話します。


単体テストですべてのコードをテストすることはしません（たぶんこれは正しくありません）。一部のコードはまったくテストしていません（これについては以下で詳しく説明します）。


テストでモクを使用する場合は、tearDownでMockery :: close（）を実行することを忘れないでください。

class TitlecaseMiddleware
{
    public function handle($request, Closure $next)
    {
        if ($request->title) {
            $request->merge([
                'title' => title_case($request->title)
            ]);
        }

        return $next($request);
    }
}

        $middleware->handle($request, function ($req) {
            $this->assertEquals('Title  Is In Mixed Case', $req->title);

public function testSuccessfulSendNetworkService()
{
    $mockHandler = new MockHandler([
        new Response(200),
   ]);

    $handler = HandlerStack::create($mockHandler);
    $client = new Client(['handler' => $handler]);

    app()->instance(\GuzzleHttp\Client::class, $client);

    $networkService = resolve(NetworkService::class);

    $response = $networkService->sendRequestToSite('GET', '/');

}
