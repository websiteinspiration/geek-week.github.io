GiSTのカバーインデックス

カバリングインデックスは、便利な機能の1つにすぎません。これは純粋に実用的です。それらがなければ、インデックスのみのスキャンは成功しないかもしれません。さまざまな状況でのカバリングインデックスはさまざまな方法で効果的ですが。

これは完全にインデックスをカバーすることに関するものではありません：厳密に言えば、いわゆる包括的インデックスはPostgresに登場しました。しかし、順番に：カバリングインデックスは、クエリで必要なすべての列の値を含むインデックスです。ただし、テーブル自体へのアクセスは不要になりました。ほぼ。あなたは、することができます読ん「ほとんど」と他のニュアンスについての記事でYegor Rogov 10（！）部品の彼のインデックスシリーズに含まれ、。包括的なインデックスこれは、典型的なクエリでの検索用に特別に作成されます。検索できないフィールド値が検索インデックスに追加されます。これらは、テーブルを再度参照しないようにするためにのみ必要です。このような索引は、キーワードINCLUDEで形成されます。

Anastasia Lubennikova（Postgres Professional）は、追加の列をインデックスに含めることができるようにbtreeメソッドを完成させました。このパッチはPostgreSQL 11に含まれていましたが、GiST / SP-GiSTアクセス方法のパッチには、このバージョンのリリース前に成熟する時間がありませんでした。 12日までにGiSTは熟しました。

GiSTの包括的インデックスを作成するという建設的な欲望はずっと前に生じました。AndreyBorodinによるテストパッチが2018年4月中旬にコミュニティに提供されました。彼はすべての基本的で非常に難しい仕事をしました。

2019年8月上旬、アレクサンダーコロトコフは化粧品の改善を追加し、パッチをコミットしました。

デモといくつかの研究のために、300万個の長方形のセットを生成します。同時に、ボックスタイプについての簡単な説明です。ボックスタイプに関するすべての操作が直感的であるとは限らないためです。

ボックスのタイプ-つまり、長方形-は、Postgresで長い間使用されてきました。2つのポイント（ポイントの幾何学的タイプ）-長方形の反対の頂点（つまり、長方形は斜めにできず、横に散らばっています）によって定義されます。ではドキュメント 「ボックス型の値は、次のいずれかの形式で記述されています。

( ( x1 , y1 ) , ( x2 , y2 ) )
  ( x1 , y1 ) , ( x2 , y2 )
    x1 , y1   ,   x2 , y2

実際には、次のように書く必要があります。

SELECT box('1,2', '3,4');
     box     
-------------
 (3,4),(1,2)
(1 row)

まず、Postgresは右上の頂点を表示し、次に左下の頂点を表示します。このように書くと

SELECT box('5,2', '3,4');
     box     
-------------
 (5,4),(3,2)
(1 row)

次に、Postgresがピークを与えなかったことを確認します。彼は私たちの左上と右下から右上と左下を計算しました。これは、たとえばランダム生成の場合など、頂点の位置が事前にわからない場合に便利なプロパティです。「1,2」、「3,4」という表記は、ポイント（1,2）、ポイント（3,4）と同等です。このフォームの方が便利な場合があります。


ビジネスの場合：300万の長方形で検索


CREATE TABLE boxes(id serial, thebox box, name text);

300万個のランダムな長方形を生成します。正規分布が必要ですが、tablefunc拡張を使用しないために、「貧しい」アプローチを使用します。ランダム（）-ランダム（）を使用します。これにより、四角形の素敵な画像（図を参照）も得られます。大きいほど、中心に近づきます。それらの重心もランダムです。このような分布は、いくつかのタイプの実際の都市データに特徴的です。また、統計の法則を調べたり、思い出をリフレッシュしたりしたい場合は、たとえば、ここで確率変数の違いについて読むことができます。




INSERT INTO boxes(thebox, name)
       SELECT box(
                         point(
                              random()-random(),
                              random()-random()
                         ),
                         point(
                              random()-random(),
                              random()-random()
                         )
                     ), 'box no.' || x
             FROM generate_series(1,3000000) AS g(x);


表示されるテーブルのサイズ\dt+は242MBです。これで検索を開始できます。

選択性とゲイン


包括的インデックスのパフォーマンスは、クエリの条件の選択性に大きく依存します。この依存関係を少し調査するために、逆の問題を解決します。ポイントタイプのインデックスを持つラベルを生成し、指定されたボックスに入るポイントの数を探します。ドットを四角に均等に広げます。

CREATE TABLE test_covergist(id serial, tochka point, name text);

INSERT INTO test_covergist(tochka, name) SELECT point(trunc(1000000*random()), trunc(1000000*random())), 'point no.' || g.x FROM generate_series(1,3000000) AS g(x);

テーブルのサイズは211 MBです。

CREATE INDEX on test_covergist USING gist(tochka);

サイズ213 MB。

銃のようなバッファ


説明は、まだ発砲していないが、壁にぶら下がっていたショットガン、つまり読み取ったブロックの数で確認できます。包括的インデックスの場合、インデックス自体のブロックのみが読み取られます（ヒープフェッチ：0）。 3つのケースでは、これらは番号40492、3735、および52でした。ただし、通常のインデックスを使用する場合、読み取られるブロックは、ビットマップヒープスキャンインデックスで読み取られたビット（300万レコードの54248）とヒープから読み取らなければならなかったビット（27223）の合計で構成されます。 、名前フィールドは通常のインデックスから抽出できないため。 54248 + 27223 = 81471。その他の2つの場合：29534 + 2510 = 31044および2655 + 31 = 2686。従来のインデックスの場合、いずれにせよより多くのブロックが読み取られますが、選択性の向上により、ヒープから必要なブロックの数がインデックスブロックの読み取りよりもゆっくりと減少するため、読み取られるブロックの数が2倍ではなく数桁異なります。

ブロックの読み取り（通常/包括的） 81471/40492 31044/3735 2686/52
