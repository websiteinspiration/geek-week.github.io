Corda-ビジネス向けのオープンソースのブロックチェーン

Cordaは、さまざまな金融機関間の金融義務を保存、管理、同期するための分散型台帳です。

Cordaには、ビデオ講義を含むかなり優れたドキュメントがあります。Cordaが内部でどのように構築されているかを簡単に説明します。

Cordaの主な機能と他のブロックチェーン間のその独自性を検討してください：

  • Cordaには独自の暗号通貨はありません。
  • Cordaは、マイニングの概念とProof-of-Workシステムを使用しません。
  • データ転送は、トランザクション/契約の参加者間でのみ発生します。すべてのネットワークノードへのグローバルブロードキャストはありません。
  • すべてのトランザクションを管理する中央コントローラーはありません。
  • Cordaはさまざまな合意メカニズムをサポートしています。
  • システム全体のレベルではなく、個別の合意/契約のレベルで参加者間でコンセンサスが得られます。
  • 取引は、関係する参加者のみが確認します。
  • Cordaは、正式な人間の法的言語とスマートコントラクトコード間の直接リンクを提供します。

元帳


Cordaの元帳の概念は主観的です。単一の中央データウェアハウスはありません。代わりに、各ノードは既知の事実の個別のデータベースを維持します。

たとえば、円がノードに知られている事実である5つのノードのネットワークを想像してください。



見てわかるように、エド、カール、デミは事実3を知っており、アリスとボブは彼を疑っていません。Cordaは、各ノードのデータベースに一般的な事実が保持されることを保証し、データは同一になります。

州


状態は不変オブジェクトであり、特定の時点で1つ以上のネットワークノードに認識されている事実です。

状態には、株式、債券、ローン、識別情報などの任意のデータを保存できます。

たとえば、次の状態はIOU-アリスがボブにXの量を負っているという合意です。


時間の経過に伴うファクトのライフサイクルは、一連の状態で表されます。現在の状態を更新する必要がある場合は、新しい状態を作成し、現在の状態を履歴としてマークします。



取引


トランザクションは、レジャーAを更新するための提案です。それらはすべての元帳参加者にブロードキャストされるわけではなく、それらを表示および管理する法的権利を持つネットワークメンバーのみが利用できます。

次の場合、トランザクションが元帳に追加されます。

  • 有効な契約
  • 必要な参加者全員が署名
  • 二重の廃棄物を含まない（2倍の費用）

Cordaは、各元帳の状態が不変であるUTXO（未使用のトランザクション出力）モデルを使用します。

トランザクションが作成されると、前のトランザクションの入力状態（ハッシュとインデックスによる）が入力に送信されます。


トランザクションのライフサイクル：

  • 作成（現時点では、トランザクションは元帳Aの更新の提案にすぎません）
  • 署名の収集（トランザクションに必要な参加者は、トランザクションに署名を追加することにより、更新提案を承認します）
  • 元帳でコミットを送信

元帳にトランザクションを追加すると、入力状態は履歴としてマークされ、将来のトランザクションで使用できなくなります。


トランザクションには、入力と出力の状態に加えて、次のものが含まれます。

  • コマンド（トランザクションの目的を示すトランザクションパラメータ）
  • 添付ファイル（休日カレンダー、通貨換算）
  • 時間枠（有効期間）
  • 公証人（公証人、特別なネットワークメンバー、取引の検証）


契約


トランザクションの有効性について話すときは、必要な署名の可用性だけでなく、契約の有効性も意味します。各トランザクションは、それを受け入れ、入力と出力の状態を検証するコントラクトに関連付けられています。トランザクションは、その状態がすべて有効である場合にのみ有効と見なされます。

Cordaの契約は、任意のJVM言語（Java、Kotlinなど）で記述されています。

class CommercialPaper : Contract {
    override fun verify(tx: LedgerTransaction) {
        TODO()
    }
}

Contract クラスを継承し、verifyメソッドをオーバーライドする必要があります。検証が無効な場合、例外がスローされます。

トランザクションの検証は確定的でなければなりません。契約は常にトランザクションを受け入れるか拒否する必要があります。これに関連して、トランザクションの有効性は時間、乱数、ホストファイルなどに依存することはできません

。Cordaでは、契約はいわゆるサンドボックスで実行されます-契約の確定的な実行を保証するわずかに変更されたJVM。

ストリーム


ホストの通信を自動化するために、スレッドが追加されました。

ストリームは、特定の元帳更新を実行する方法をノードに指示する一連のステップであり、その時点でトランザクションに署名して検証する必要があります。



取引がすべての関係者によって署名され、元帳になるまでに数時間、数日かかる場合があります。トランザクションに関係するノードを切断するとどうなりますか？ストリームには、ストリームの状態がノードデータベースに書き込まれるブレークポイントがあります。ネットワーク内のノードを復元すると、停止した場所から続行されます。

コンセンサス


元帳に入るには、トランザクションは2つのコンセンサスに達する必要があります。有効性と一意性です。

取引の有効性に関する決定は、直接関係する当事者のみが行います。

公証ノードはトランザクションの一意性を検証し、二重の支出を防ぎます。

同じ入力状態を使用して、ボブが100ドル持っていて、80ドルをチャーリーに、70ドルをダンに転送したいとします。



そのようなトリックCordaはクランクすることはできません。トランザクションは検証チェックに合格しますが、一意性チェックは失敗します。

結論


R3ブロックチェーンコンソーシアムによって開発されたCordaプラットフォームは、ブロックチェーンテクノロジーの使用の純粋な例ではありません。Cordaは、金融機関向けの高度に専門化されたツールです。
