使用Flyway进行数据库迁移

在Java Developer课程开始之前，准备了本文的翻译。


1.简介


本文介绍了Flyway的关键概念，并举例说明了使用内存中的H2数据库使用此框架连续更改数据库架构的示例，以及使用flyway maven插件的示例。

Flyway使用迁移更新数据库版本。迁移可以用SQL（特定于特定DBMS的语法）或Java编写。

迁移可以进行版本控制或重复进行。前者有一个独特的版本，只适用一次。第二个没有版本号，它们在校验和更改时适用。

在版本化迁移完成之后，总是会在单次运行中应用重复的迁移。重复的迁移按照描述的顺序进行应用。在一次迁移中，所有操作都在单个数据库事务中执行。

在本文中，我们将重点介绍如何使用maven插件进行数据库迁移。

2. Flyway Maven插件


将flyway maven插件添加到pom.xml：

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
</plugin>

可以在Maven Central中查看该插件的当前版本。
插件参数列表可以在文档中找到。插件参数可以通过四种不同的方式进行配置。

2.1。<configuration>插件部分
可以在插件部分的标签中直接指定参数pom.xml：

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
    <configuration>
        <user>databaseUser</user>
        <password>databasePassword</password>
        <schemas>
            <schema>schemaName</schema>
        </schemas>
        ...
    </configuration>
</plugin>

2.2。Maven属性


您还可以通过在中指定参数来配置插件<properties>：

<project>
    ...
    <properties>
        <flyway.user>databaseUser</flyway.user>
        <flyway.password>databasePassword</flyway.password>
        <flyway.schemas>schemaName</flyway.schemas>
        ...
    </properties>
    ...
</project>

2.3。外部配置文件


或在单独的.properties文件中描述配置：

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=schemaName
...

默认名称是配置文件flyway.properties。该文件必须与该文件位于同一目录中pom.xml。编码在参数中设置flyway.encoding（默认为UTF-8）。

如果为文件使用其他名称（例如customConfig.properties），则在调用maven时需要明确指定它：

$ mvn -Dflyway.configFile=customConfig.properties

2.4。系统属性


最后，当从命令行调用maven时，可以将参数指定为系统属性：

$ mvn -Dflyway.user=databaseUser -Dflyway.password=databasePassword
  -Dflyway.schemas=schemaName

如果以多种方式指定配置，则优先级如下：

  1. 系统属性
  2. 外部配置文件
  3. 部分<属性>
  4. 插件<配置部分>

3.迁移示例


在本节中，我们将考虑使用内存中H2数据库迁移数据库架构的必要步骤，并使用maven插件作为示例。要配置Flyway，我们将使用一个外部文件。

3.1。POM变更


首先，添加对H2的依赖：

<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>1.4.196</version>
</dependency>

在这里，我们还可以检查Maven Central中最新的可用驱动程序版本。如前所述，为Flyway添加一个插件。

3.2。在外部文件中配置Flyway


创建一个具有以下内容的$PROJECT_ROOT文件myFlywayConfig.properties：

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=app-db
flyway.url=jdbc:h2:mem:DATABASE
flyway.locations=filesystem:db/migration

上面的配置表明迁移脚本位于目录中db/migration，并使用H2 databaseUser和.NET连接到数据库databasePassword。

应用程序的数据库模式app-db。

当然，在参数flyway.user，flyway.password并且flyway.url必须指定数据库的名称，密码和网址。

3.3。第一次迁移


在Flyway约定下，迁移脚本的名称必须采用以下格式：

<Prefix<code>><</code>Version>__<code><</code>Description>.sql

哪里：

  • <前缀>-前缀。对于版本化的迁移，默认值为“ V”。前缀是通过flyway.sqlMigrationPrefix属性配置的。
  • <版本>-迁移版本号。主版本和次版本可以通过下划线分隔。版本应始终以1开头。
  • <描述>-迁移的文本描述。说明必须与版本号之间用两个下划线分隔。

示例：V1_1_0__my_first_migration.sql

因此，让我们db/migration在$PROJECT_ROOT脚本迁移V1_0__create_employee_schema.sql和SQL中创建目录以创建表employee：

CREATE TABLE IF NOT EXISTS `employee` (
 
    `id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name` varchar(20),
    `email` varchar(50),
    `date_of_birth` timestamp
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;

3.4。我们进行迁移


接下来，$PROJECT_ROOT运行以下maven命令以应用数据库迁移：

$ mvn clean flyway:migrate -Dflyway.configFile=myFlywayConfig.properties

我们的第一次迁移必须完成。

现在数据库架构如下所示：
employee：

+----+------+-------+---------------+
| id | name | email | date_of_birth |
+----+------+-------+---------------+

我们可以重复前面的步骤来执行其他迁移。

3.5。第二次迁移


对于第二次迁移，创建一个名称V2_0_create_department_schema.sql包含以下两个请求的文件：

CREATE TABLE IF NOT EXISTS `department` (
 
`id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
`name` varchar(20)
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;
 
ALTER TABLE `employee` ADD `dept_id` int AFTER `email`;

让我们执行迁移，就像我们第一次迁移一样。

现在，我们数据库的架构已更改：employee添加了新列并创建了新表department：

employee:
+----+------+-------+---------+---------------+
| id | name | email | dept_id | date_of_birth |
+----+------+-------+---------+---------------+

department:

+----+------+
| id | name |
+----+------+

要验证两个迁移均成功，请运行以下maven命令：

$ mvn flyway:info -Dflyway.configFile=myFlywayConfig.properties

4.在Spring Boot中禁用Flyway


有时您可能需要禁用Flyway迁移。

在基于实体生成数据库架构的测试过程中可能需要这样做。在这种情况下，您可以为测试配置文件禁用Flyway。

在Spring Boot中，这非常简单。

4.1。春季启动1.x


我们需要做的就是flyway.enabled在文件中设置属性application-test.properties：

flyway.enabled=false


4.2。Spring Boot 2.x


在更高版本的Spring Boot中，此属性已更改为 

spring.flyway.enabled:
spring.flyway.enabled=false

4.3空的飞行路线迁移策略


如果我们只想在启动时禁用自动Flyway迁移，而想手动启动迁移，那么使用上述属性将对我们不起作用。

这是由于Spring Boot不会自动配置Flyway Bin，因此我们必须自己配置它们，这不是很方便。

在这种情况下，我们可以启用Flyway并实现一个空的FlywayMigrationStrategy：

@Configuration
public class EmptyMigrationStrategyConfig {
 
    @Bean
    public FlywayMigrationStrategy flywayMigrationStrategy() {
        return flyway -> {
            // do nothing 
        };
    }
}

实际上，这将在应用程序启动时禁用Flyway迁移。

但是我们仍然可以手动开始迁移：

@RunWith(SpringRunner.class)
@SpringBootTest
public class ManualFlywayMigrationIntegrationTest {
 
    @Autowired
    private Flyway flyway;
 
    @Test
    public void skipAutomaticAndTriggerManualFlywayMigration() {
        flyway.migrate();
    }
}

5.飞道的运作方式


为了跟踪何时，由谁进行迁移以及应用了什么迁移，将带有元数据的特殊表添加到数据库架构中。该表还存储迁移的校验和以及有关迁移是否成功的信息。

该框架的工作方式如下：

  1. 检查数据库架构中的元数据表（默认值SCHEMA_VERSION）。如果元数据表不存在，它将创建它。
  2. 扫描类路径以获取可用的迁移。
  3. 将迁移与元数据表进行比较。如果版本号小于或等于标记为当前的版本，则将其忽略。
  4. 将所有剩余的迁移标记为未决。然后，将它们以升序版本号排序并以指定顺序执行它们。
  5. 在进行迁移时，更新元数据表。

6.团队


Flyway具有以下主要的迁移管理团队：

  • 资讯。显示数据库模式的当前状态/版本。有关预期进行哪些迁移，已应用哪些迁移，已完成迁移的状态以及完成日期的信息。
  • 迁移。将数据库架构更新到当前版本。扫描类路径以查找可用的迁移并应用挂起的迁移。
  • 基线。安装数据库架构版本，而忽略直到及包括基线版本的迁移。基准可帮助您在现有数据库上使用Flyway。照常应用新迁移。
  • 验证。根据可用的迁移检查当前数据库架构。
  • 维修。恢复元数据表。
