CSSグリッド：レスポンシブな20行の雑誌レイアウトの入力


最近、私はブログロール（外部の便利で興味深いブログのリスト）の最新の実装に取り​​組みました。アイデアは、サイドバーのリンクの乾燥したリストではなく、雑誌のレイアウトにパッケージ化されたこれらのブログの最新の投稿の選択を読者に提供することでした。

タスクの最も簡単な部分は、お気に入りのRSSフィードから投稿とその抜粋（抜粋-カットの前の紹介テキスト）のリストを取得することです。これを行うために、Feedzy lite WordPressプラグインを使用しました。このプラグインは、複数のフィードを1つのリストに集約し、時間順に並べ替えることができます。この場合は理想的なソリューションです。難しいのは、すべてを美しくすることです。

プラグインの標準リストインターフェースはおそらく味がないので、大小さまざまな「選択された」ブロックが混在する新聞や雑誌のウェブサイトとしてスタイルを設定したかったのです。

CSSグリッドを使用する絶好の機会のようですね。さまざまなレイアウト、たとえば1つの5列と1つの3列のグリッドレイアウトを作成し、異なる画面サイズでメディアクエリを使用してそれらを切り替えます。正しい？しかし、グリッドパラメータを使用するだけでauto-fitアダプティブグリッドが作成される場合、これらのメディアクエリとコントロールポイントの定義に関するこのすべての問題が本当に必要ですか？

このアイデアは私には魅力的であるように見えましたが、グリッドのいくつかの列（スパン）にまたがる要素を追加し始めたとき）、グリッドは狭い画面でページから這い出し始めました。メディアクエリが唯一のソリューションのように思われました。しかし、私はもっと良いものを見つけました！

CSSグリッドに関する多くの記事を調べたところ、それらは主に2つのタイプに分かれていることがわかりました。

  1. スパンを使用して、指定された列数で興味深いレイアウトを作成する方法。
  2. グリッド上でアダプティブレイアウトを作成する方法。ただし、列の幅は同じ（つまり、スパンなし）。

グリッドにこれとそれの両方を実装して欲しい：適応的にサイズ変更された複数列要素を使用した完全適応レイアウト。

アダプティブグリッドの制限を理解し始めるとすぐに、スパンが適応性を損なう理由とタイミングがわかると、数十行のコードと1つのメディアクエリを含む雑誌のレイアウトを簡単に作成できます（または、さまざまなスパンを制限したい場合は、それらを使用しないことも簡単です）。

.archive {

行の高さが、行のコンテンツの最も高いブロックに自動的に調整されることに注意してください。上のリンクからデモの幅を変更すると、要素が自動的に増減し、列数がそれぞれ1から5に変化することがわかります。

ここで実際に動作しているのは、CSSグリッドマジックと呼ばれるものauto-fitです。このキーワードは、にminmax()適用される機能と連動して機能しgrid-template-columnsます。

使い方


5列のレイアウト自体は、次のようにして取得できます。

.archive {
  display: grid;
  grid-template-columns: repeat(5, 1fr);
}

ただし、これにより、さまざまな画面幅で伸縮する5列のレイアウトが作成されますが、常に5列のままなので、小さな画面では非常に狭い列になります。最初に頭に浮かぶのは、大量のメディアクエリを記述し、列数が異なるグリッドを再定義することです。それはうまくいきますが、キーワードauto-fitはすべて自動的に行います。必要に応じて関数

をauto-fit使用しますminmax()。したがって、列をどれだけ圧縮できるか、どれだけ伸張できるかをブラウザに伝えます。境界の1つに達すると、列の数はそれぞれ増加または減少します。

.archive {
  grid-template-columns: repeat (auto-fit, minmax(180px, 1fr));
}

この例では、ブラウザは幅が180ピクセルの列と同じ数の列に対応しようとします。余分なスペースがある場合、すべての列が拡張され、列間で均等に分割されます。これが意味を規定するものです。1fr列サイズを利用可能な幅の分数（frアクション）に等しくするためです。

ブラウザーウィンドウを拡大すると、空き領域が増えると、すべての列が等しく増加します。新しく見つかったスペースが180ピクセルに達するとすぐに、その場所に新しい列が表示されます。また、ブラウザーウィンドウを縮小すると、すべてが逆に起こり、グリッドが1列のレイアウトになるまで完全にフィットします。マジック！

「オートフロー：密」でスパンを作成する


したがって、現時点ではアダプティブグリッドがすでにあります。これは、すべての要素（同じ幅）です。新聞や雑誌のレイアウトは、選択されたブロックの存在を意味します。このコンテキストでは、2つ、3つ、またはすべての利用可能な列をカバーするブロックが存在します。

複数列のスパンを作成するには、grid-column: spanより多くのスペースを占有するはずの要素でプロパティを使用できます。3番目のリスト項目を2列幅にしたいとします。

.article:nth-child(3) {
  grid-column: span 2;
}

ただし、スパンを追加すると、多くの問題が発生する場合があります。まず、ワイド要素がラインに適合しない場合にグリッドに穴が形成され、次の要素にauto-fit転送されます。


これはgrid-auto-flow: dense、グリッドにプロパティを追加することで簡単に修正できます。このプロパティにより、ブラウザは穴を他の要素で埋める必要があることを理解します。これにより、幅の広い要素の周りの流れが狭くなります。


注意：要素の順序が壊れているため、4番目の要素が3番目の要素の前にあります。私の知る限り、これを回避することはできません。これは、CSSグリッドの受け入れられる制限の1つです。

スパンを特定する方法


アイテムが占める列の数を指定する方法はいくつかあります。grid-columns: span [n]要素の1つに適用するのが最も簡単です。ここで、nは、要素が占める列の数です。レイアウトの3番目の要素にはプロパティregistered grid-column: span 2があり、その幅が他の要素の2倍の大きさである理由を説明しています。

他の方法を使用するには、グリッド線を指定する必要があります。グリッド線には次のように番号が付けられています。


グリッド線は、左から右に正の数（1、2、3など）を使用して、または右から左に負の数（-1、-2、-3）を使用して示すことができます。次のgrid-columnように、プロパティを使用してグリッドに要素を配置するために使用できます。

.grid-item {
  grid-column: ( ) / ( );
}

したがって、グリッド線はスパンを定義する機能を拡張します。柔軟性は、初期値または最終値をキーワードに置き換える機能によって追加されspanます。たとえば、上記の例の3列の青いブロックは、次のプロパティを8番目のグリッド要素に適用することで作成できます。

  • grid-column: 3 / 6
  • grid-column: -4 / -1
  • grid-column: 3 / span 3
  • grid-column: -4 / span 3
  • grid-column: span 3 / -1
  • 等

非アダプティブグリッド（つまり、列数が固定）では、これらの各プロパティは同じ結果になります（上記の青いブロックの例と同様）。しかし、グリッドがアダプティブであり、列の数が変化する場合、その違いは非常に顕著になります。一部のスパンは、自動折り返しがオンになっているとレイアウトを壊し、これら2つのソリューションは互換性がないように見えます。幸いなことに、いくつかのトリックにより、それらを安全に組み合わせることができます。

水平スクロールの問題


グリッドライン方式（クリック可能）を使用して作成されたいくつかの「注目のアイテム」は次のとおりです。


幅全体（5列）では、すべてが適切に見えますが、画面を2列にする必要があるサイズに縮小すると、レイアウトは次のように壊れます。


ご覧のとおり、グリッドはその適応性を失っており、コンテナーは縮小していますが、グリッドは5つの列すべてをサポートしようとしています。これを行うために、彼女は同じ列幅を試し続け、最終的にはコンテナーの境界を越えて右側に行きます。これから、横スクロールが現れます。

なぜそれが起こるのですか？問題は、ブラウザがグリッドラインの正確な方向に準拠しようとしていることです。この幅では、auto-fitグリッドには2列しか表示されませんが、スタック行の番号付けシステムはこれと矛盾し、特に5行目を参照しています。この矛盾は無秩序につながります。暗黙の2列グリッドを正しく表示するには、次のように、1、2、3および-3、-2、-1のみを使用できます。


しかし、グリッドの要素の1つにgrid-columnこれらの制限外の命令（4、5、または-6など）が含まれている場合、ブラウザーはあいまいな命令を受け取ります。一方では、フレキシブルな列を自動的に作成するようにお願いします（これは-暗黙的に-この画面幅で2のままである必要があります）。一方、2列形式では存在できないグリッド線を明示的に参照しました。暗黙的な（自動）列と明示的に定義された数の間に矛盾がある場合、グリッドは常に明示的な定義を優先します。これは、不要な列と水平オーバーフローが表示される方法です（CSSデータ損失と呼ばれています）。）グリッド線番号のようなスパンは、明示的な列定義を作成できます。grid-column：span 3（デモの8番目のグリッド要素）は、2つの暗黙的な列が必要であるにもかかわらず、グリッドに明示的に少なくとも3つの列を強制します。

メディアクエリを使用grid-columnして目的の幅で値を変更することが唯一のオプションのように思えるかもしれませんが、急いではいけません。私も最初はそう思っていました。しかし、少し考えて、さまざまな設定を試してみたところ、この問題を回避する方法がいくつかあることがわかりました。おかげで、最も狭いデバイスに対するメディア要求はまだ1つしかありません。

使用法：動的な幅を繰り返すためのnth-child（）


コードを20行に減らすために、別のトリックを適用しました。セレクターを:nth-child(n)使用すると、多数の要素を一度にスタイルできます。スパンに関する私の全体的なアイデアは、選択したブロックが定期的にページに表示されるように、フィード内の多くの投稿に適用することでした。まず、要素番号を明確に定義したセレクターのコンマ区切りのリストを書きました。

.article:nth-child(2),
.article:nth-child(18),
.article:nth-child(34),
.article:nth-child(50)  {
  background-color: rgba(128,0,64,0.8);
  grid-column: -3 / -1;
}

特に、記事内で変更する必要がある各子要素（見出し、リンクなど）の条件のリスト全体をコピーする必要がある場合、これは非常に時間のかかるプロセスであることを理解しました。プロトタイピング中に、スパンの位置を変更するたびに、これらの各リストの数値を手動で変更する必要がありました。退屈でエラーが発生しやすいプロセス。

そのとき、疑似セレクターの優れた機能を利用できることに気づきまし:nth-childた。整数値の代わりに、たとえば:nth-child(2n+ 2)、式を入力します。これは、2番目ごとの子要素を意味します。

これは:nth-child([])、ページの上部に表示され、リストの途中に表示されるグリッドに全幅の青いブロックを作成するために使用した方法です。

.article:nth-child(31n + 1) {
  grid-column: 1 / -1;
  background: rgba(11, 111, 222, 0.5);
}

大括弧（31n + 1）で囲まれたコードは、1番目、32番目、63番目などを選択する役割を果たします。子要素。ブラウザは、n = 0（31 * 0 + 1 = 1）、次にn=1（31 * 1 + 1 = 32）、最後にn=2（31 * 2 + 1 = 63）でループを開始します。後者の場合、ブラウザは63番目の子要素がないことを理解し、ルールを無視してサイクルを停止し、1番目と32番目の要素にルールを適用します。

ページ全体の左側または右側に表示される紫色のブロックについても、同様のことを行っています。

.article:nth-child(16n + 2) {
  grid-column: -3 / -1;
  background: rgba(128, 0, 64, 0.8);
}

.article:nth-child(16n + 10) {
  grid-column: 1 / -2;
  background: rgba(128, 0, 64, 0.8);
}

最初のセレクターは、左の紫色のブロック用です。式16n + 2は、2番目から始めて、16番目のグリッド要素ごとにスタイルを適用します。

2番目のセレクターは、右の紫色のブロック用です。間隔は同じですが（16n）、シフトは異なります（10）。その結果、これらのブロックは、グリッドの右側に、10、26、42などの番号が付けられた要素で定期的に見つかります。

これらの要素の視覚スタイルについては、コードの繰り返しを回避するために別のトリックを使用しました。紫色のブロックの一般的なスタイル（background-colorたとえば、）の場合、1つのセレクターを使用できます。

.article:nth-child(8n + 2) {
  background: rgba(128, 0, 64, 0.8);
  /* Other shared syling */
}

このセレクターは、アイテム2、10、18、26、34、42、50などを選択します。つまり、左と右の両方のブロックを選択します。

これが機能するのは、8nこれがちょうど半分16nであり、2つのセレクターのシフト差も8だからです。

最後の言葉


CSSグリッドを使用して、最小限のコードで柔軟で応答性の高いグリッドを作成できるようになりました。ただし、レトログレードメディアクエリの使用を回避する場合、グリッド内の要素の配置に大きな制限があります。

小さな画面で水平スクロールやオーバーフローを引き起こさないスパンを作成できるとしたら、とてもクールです。これでブラウザに「アダプティブグリッドを作ってください」と伝えることができます。しかし、追加する必要があるのは、「ああ、グリッドのこの要素は4列に拡大してください」であり、彼はハンドルを狭い画面に振り、アダプティブグリッドではなく4列のスパンの要求を優先します。たとえば次のように、グリッドを反対にすることができます。

.article {
  grid-column: span 3, autofit;
}

レスポンシブグリッドのもう1つの問題は最後の行です。画面の幅を変更すると、空白になることがよくあります。グリッドの最後の要素を残りの列に引き延ばす（それぞれ行を埋める）方法を長い間探しましたが、それは不可能のようです。少なくとも今のところは。要素の初期位置を、auto「左端から開始して最後まで行を埋める」のように、のようにキーワードで設定するとよいでしょう。このようなもの：

.article {
  grid-column: auto, -1;
}

...グリッドの左端のスパンを線の終わりまで伸ばします。
