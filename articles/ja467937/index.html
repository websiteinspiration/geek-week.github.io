それでも、私は自分の自転車を発明することにしました。部分的には、スキルを向上させるために、部分的には、リソースを消費せず、十分に高速なeコマースプラグインを作成しようとします。最近、公式リポジトリに投稿しました。したがって、私はそれをテストするようにみんなを招待します。この記事では、可能性の概要については説明しませんが、興味深い技術的ソリューションについてのみ説明します。



パーマリンクの問題


一般に、WordPressのパーマリンクは、接続と依存関係の多くの問題を解決する必要があるため、最も難しいサイトの1つです。 WpStoreには、パーマリンクを管理する機能があります。たとえばproduct、製品のURLからスラッグを削除したり、独自のスラッグに変更したり、カテゴリスラッグを追加したり、ネストしたカテゴリを追加したりできます。製品へのリンクは次のようになります./-/---/---/-/。悪くない。

しかし、この時点でパフォーマンスの問題が発生しました。サイトは、特にいくつかの製品の出力ページで、動作が遅くなり始めました。たとえば、16個の製品しか出力されないカテゴリページでは、データベースに対して約90のクエリが行われ、サーバーの応答時間は約25〜30％劇的に増加しました。the_permalink

関数を呼び出すと、 WordPressは多くの操作を実行することがわかりました。CNCのタイプを取得し、このタイプに応じて投稿データを収集します。 1つのリンクを表示するために、WordPressはデータベースに対していくつかの非キャッシュ要求を生成します。さらに、分類と商品の階層を取得するプロセスは最速ではありません。

常に生成されるリンクは適していないため、どこかに保存し、そこからプルするのが論理的です。特別なフィールドを使うことになりましたguidテーブルwp_posts。WP開発者はこれを変更することを推奨していませんが、いくつかの理由でこの警告を無視することができます。

  • RSSの生成にはguidが使用されており、それを使用する人はほとんどいません。
  • RSSにはエントリのみが表示され、投稿タイプは製品です。

データベース内のリンクを正しく維持するためにsave_post、GUIDフィールドを更新するイベントで関数をハングさせましょう。

add_action( 'save_post', 'wpsl_guid_rewrite', 100 );
function wpsl_guid_rewrite( $id ) {
	
	if( !is_admin() ) return;
	
	if( defined( 'DOING_AUTOSAVE' ) && DOING_AUTOSAVE ) return false;

	if ( strpos( get_option( 'product_permalink' ), '%category%' ) !== false || strpos( get_option( 'product_permalink' ), '%categories%' ) !== false ) {
		if( $id && get_post_type( (int)$id ) == 'product' ){
			global $wpdb;
			$wpdb->update( $wpdb->posts, [ 'guid' => ( get_permalink( $id ) ) ], [ 'ID' => intval( $id ) ] );
		}

		clean_post_cache( $id );
	}
}

フックのリンク出力をインターセプトしてpost_type_link、生成されたリンクを出力することは残ります。

add_filter( 'post_type_link', 'wpsl_get_permalink_change', 10, 4 );
function wpsl_get_permalink_change( $post_link, $post, $leavename, $sample ){
	if ( isset( $post->guid ) && $post->guid && $post->post_type == 'product' && ( strpos( get_option( 'product_permalink' ), '%category%' ) !== false || strpos( get_option( 'product_permalink' ), '%categories%' ) !== false ) ) {
		return $post->guid;
	}
	return $post_link;
}

ここでは、ガイドフィールド、ポストタイプ、CNCタイプの完全性を確認します。3つのパラメーターの値が適切な場合は、以前に保存したリンクを表示します。

今、最良の部分は次のとおりです。結果を見てください。データベースへのクエリ数は、約90から44に、約2倍に減少しました。また、テストホスティングでのサーバーの応答時間は、許容できる0.24秒に減少しました。

GUIDフィールドの更新は、商品の作成時と編集時だけでなく、プラグインには、csvを介して商品をインポートするための組み込みシステムがあるため、インポートが完了したときにも更新が行われます。
