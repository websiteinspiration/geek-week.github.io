私たちは理想的な評価の灯台で最もアクセスしにくいサイトを作成します

組み込みのGoogle Lighthouseテストツールは、サイトのアクセシビリティを0〜100のスケールで評価します。コンテンツの可読性を最大限に高めるよう努めることは称賛に値しますが、100の評価は、サイトが理想的にアクセス可能であることを意味しません。証拠として、小さな実験を行いました。

TwitterでLighthouseの評価について自慢する人々を見るのはいつも素晴らしいことです。



灯台は、100の数字が付いた緑色の円で最高のサイトに報酬を与えます。これは、顧客や友人に誇らしげに見せます。

コードの品質を評価することは重要ですが、さらに重要なことは、コードを正しく解釈することです。自動ツールがサイトに100％アクセス可能であると言った場合、これは必ずしもそうではありません。これは、手動テストの基礎を築いたことを意味するだけです。テストでは、Lighthouseは独自のルールセットを備えたax-coreライブラリを使用します。彼女はいくつかの悪い習慣を明らかにしましたが、すべてではありません。他の慣行はそれ自体は悪いことではありませんが、乱用すると害を及ぼす可能性があります。

1つの自動テストでは、高品質は提供されません。これを証明するために、完璧な灯台の評価で最もアクセスしにくいサイトを作成しました。

基本として、このシンプルでアクセスしやすいページを採用しています。

必要なCSS


簡単なものから始めましょう。CSSがないと完璧なWebサイトが機能しないことを確認したいと思います。これを行うhiddenには、要素に属性を追加しますbody。これはdisplay: none;CSS のHTMLと同等です（アクセシビリティを維持しながらコンテンツを非表示にする方法については、Scott O'Hara の包括的非表示を参照してください）。

HTML


<body hidden>
  ...
</body>


完全にクリーンなページ

hiddenは、コンテンツを視覚的に隠し、アクセシビリティツリーから削除します。これだけで、すべての訪問者を完全に排除し、Lighthouseテストに合格するのに十分ですが、簡単な方法を探しているわけではありません。完全にアクセスできず、技術的にコンテンツが表示されているサイトを作成したいと考えています。したがって、CSSを追加してコンテンツを元に戻します。

HTML


<head>
  <link rel="stylesheet" href="style.css">
</head>
<body class="loaded" hidden>
  ...
</body>

CSS


.loaded {
  display: block;
}

以前の状態に戻りましたが、CSSが読み込まれるはずです。

CodePen：100％の可用性ステップ1

必須JS


別の依存関係を追加します。現在追加しているコンテンツを表示するためのクラスは、HTMLではなく、JSを介しています。

HTML


<head>
  <link rel="stylesheet" href="style.css">
  <script src="script.js"></script>
</head>

<body hidden>
  ...
</body>

Js


  document.querySelector('body').classList.add('loaded');

うわー！サイトはまだ変更されていませんが、少なくとも一部のコンテンツを表示するには、CSSおよびJSファイルがロードされ、正しく機能する必要があります。

スクリーンリーダーを利用するには


スクリーンリーダーを妨害する方法はたくさんあります。最も簡単で最も効果的なのは、属性を使用することですaria-hidden="true"。この強力な属性は、ユーザー補助ツリーからアイテムを削除するため、注意して使用する必要があります。これは通常、冗長なコンテンツや無関係なコンテンツを削除して、スクリーンリーダーを使用するユーザーを支援するために使用されます。私たちのサイトでは、要素にこの属性を規定していますbody。

HTML


<body hidden aria-hidden="true">
  ...
</body>

現在、スクリーンリーダーを使用している人は、サイトにアクセスできなかったときに、「まれな」瞬間の 1つを体験しています。

CodePen：100％の可用性ステップ3

キーボードでユーザーを遮断


キーボードを使用するユーザーは、Tabキーを押してインタラクティブ要素から別の要素に移動することで、ページを移動できます。ブラウザーは、フォーカスされている要素の周りのアウトラインを表示します。



これを取り除こう。

CSS


*:focus {
  outline: none !important;
}

3行のCSSで、私たちはサイトから人々のグループ全体を切り取りました。技術的には、それらは引き続きページと対話できます。Tabキーを押すと、対話型要素間に遷移があり、フォーカスインジケーターだけが表示されなくなります。私たちの実験では、人の能力を完全に制限する必要があるので、キーボードを完全にオフにしましょう。

Js


document.addEventListener('keydown', function(e) {
  e.preventDefault();
})

今回のアプリケーションでは、すべてのキーのデフォルト機能を削除します。

ハイコントラストモードの悪用


Windowsでは、視覚障害者は、いわゆるハイコントラストモードをオンにすることでコントラストを改善できます。



次に、オペレーティングシステム全体が、ブラウザーを含むすべてのアプリケーションに対してコントラストスキームを使用し始めます。特にこれらのユーザーを

対象にプロパティをターゲティングできmediaます。

CSS


@media screen and (-ms-high-contrast: active) {
  /* High contrast styling rules */
  * {
    color: #000000;
  }
}

これらのルールは、Windowsでハイコントラストモードが有効になっている場合にのみ適用されます。残念ながら、テーマに使用されている色が明るいか暗いかはわかりません。#000000ユーザーの設定によっては、すべての要素の色を設定しても機能しない場合があります。この50％の確率は私には合いませんが、幸運でした。Windowsの対照的な色がCSSシステムカラーのキーワードにマッピングされています。これらのキーワードを使用して、ハイコントラストモードでテキストが常に背景の色と一致するようにしてください。OSの背景色windowsはCSS の色と一致します。したがって、ページ上のすべてのテキストにこの色を指定します。

CSS


@media screen and (-ms-high-contrast: active) {
  * {
    color: window !important;
  }
}



何てことだ。これは意地悪です。今、私はFacebookとUberからの仕事のオファーのシャワーを取得します。

CodePen：100％の可用性ステップ5

マウスユーザーと地獄に


これは非常に簡単です。画面からカーソルを削除するだけです。

CSS


*,
*:hover {
  cursor: none;
}

マウスを使用するユーザーのプロパティcursor: none;は、キーボードを使用するユーザーのプロパティと同じoutline: none;です。カーソルがなければ、最初はナビゲートするのは難しいですが、要素はまだクリック可能です。アプリケーションの品質を向上させましょう。ここでも、アクセス可能なインターフェースでユーザーをトリミングします。

CSS


body {
  pointer-events: none;
}

さて、別のもの。指摘した後、pointer-events: none;ユーザーはアイテムをクリックできなくなります。このプロパティは十分にサポートされていますが、関数ができるだけ多くのブラウザーで機能するようにするには、漸進的低下の原則を適用することをお勧めします。

Js


function removeA11y() {
  if ("pointerEvents" in document.body.style) {
    console.log('pointer-events supported')
    return;
  }

  document.addEventListener('click', function(e) {
    e.preventDefault();
  })
}

removeA11y();

このフォールバックスクリプトは、ブラウザがプロパティをサポートしていない場合に実行され、すべての要素からクリックイベントを削除しますpointer-events。

CodePen：100％の可用性、ステップ6
