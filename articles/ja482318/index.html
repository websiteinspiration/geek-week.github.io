Rustが関数型プログラミング言語になる理由

こんにちは、ハブル！

Scalaの研究を始めてすぐ、私は最も単純なクイックソートアルゴリズムの機能実装が、同様の命令型実装よりも根本的に遅く動作し、かなり多くのメモリを消費するという事実に出くわしました。同時に、関数型コードの方が簡潔で、表現力があり、エラー耐性があることに異議を唱える人はいません。両方の例をRustに書き直して、共有したいいくつかの重要なことを発見しました。カットの詳細と、ここでは簡単な結論のみを示します。

  1. 必須の実装の場合、Rustからの利益はわずか20％であることが判明しました。これは、JVMがネイティブパフォーマンスに近づいており、改善の余地がないことを意味します。
  2. 機能的な実装では、Rustは4.5倍高速であり、メモリ消費量は5.5倍減少し、ガベージコレクタがないため、プログラムがより安定しました（インジケーターの変動が少ない）。これは、高速な関数型プログラムを作成したい人にとって興味深いものです。
  3. Rustで採用されているデータの唯一の所有者（および唯一の変更可能な参照）の概念は、不変性の概念に非常に似ています。その結果、不変性、再帰、およびコピーに基づく機能アルゴリズムは、書き換えなしで実際にRustに簡単に当てはまり、命令型アルゴリズムはコードの再設計を強制します。リンクの可変性、存続期間などを検討する

結論 -RustはFP用に特別に作成されたようですが、その構文の可能性はまだScalaに到達していません。

それでは、Scalaから始めて、2つのスタイルでクイックソートを実装してみましょう。

命令的に -同じ配列を使用し、再帰的な手順を使用して要素を再配列します。潜在的にタイプミスに対して脆弱である冗長なコードが表示されますが、メモリを最適に使用し、可能な限り高速です。

def sort_imp(arr: Array[Double]) {
  def swap(i: Int, j: Int) {
    val t = arr(i)
    arr(i) = arr(j)
    arr(j) = t
  }

  def sort1(l: Int, r: Int) {
    val pivot = arr((l + r) / 2)
    var i = l
    var j = r
    while (i <= j) {
      while (arr(i) < pivot) i += 1
      while (arr(j) > pivot) j -= 1
      if (i <= j) {
        swap(i, j)
        i += 1
        j -= 1
      }
    }
    if (l < j) sort1(l, j)
    if (i < r) sort1(i, r)
  }

  if (arr.length > 1)
    sort1(0, arr.length - 1)
}

機能的に-3つの新しい配列を作成し、それらの結合を返す再帰関数を使用します。非常にコンパクトで美しいコードであり、間違いを犯す場所はほとんどありませんが、非常に遅く、大食い（貧弱、GC不良）です。

Imperativeで同じことを書き換えます-アルゴリズムは変更されていません。唯一の問題は、内部関数のクロージャーが機能せず、再帰時に特別なクロージャー構文を使用すると問題が発生するため、関数内の元の配列への可変リンクを明示的に渡す必要があったことです。アルゴリズムがより複雑な場合、ボローチェッカーの制限に遭遇する可能性があります。

fn sort_imp(arr: &mut Vec<f64>) {
  fn swap(arr: &mut Vec<f64>, i: usize, j: usize) {
    let t = arr[i];
    arr[i] = arr[j];
    arr[j] = t;
  };

  fn sort1(arr: &mut Vec<f64>, l: usize, r: usize) {
    let pivot = arr[(l + r) / 2];
    let mut i = l;
    let mut j = r;
    while i <= j {
      while arr[i] < pivot { i += 1; }
      while arr[j] > pivot { j -= 1; }
      if i <= j {
        swap(arr, i, j);
        i += 1;
        j -= 1;
      }
    }
    if l < j { sort1(arr, l, j); }
    if i < r { sort1(arr, i, r); }
  };

  if arr.len() > 1 {
    sort1(arr, 0, arr.len() - 1);
  }
}

機能的に -アルゴリズムは同じであることがわかりますが、Rust構文は機能にあまり適しておらず、中間ベクトルを宣言する必要がありました。iter（）とfilter（）を配列の要素に

順次適用すると、クロージャーパラメーターxがタイプ&& f64を取得するため、二重参照**を逆参照する必要があります。フィルタリングされた値を明示的に複製する必要があります。append（）関数のシグネチャには、ベクター自体ではなく、可変リンクが必要です。これにより、文字数も増加します。

fn sort_fun(arr: Vec<f64>) -> Vec<f64> {
  if arr.len() <= 1 {
    arr
  } else {
    let pivot = arr[arr.len() / 2];
    let mut a = Vec::<f64>::with_capacity(arr.len());
    a.append(&mut sort_fun(arr.iter().filter(|x| pivot > **x).cloned().collect()));
    a.append(&mut arr.iter().filter(|x| pivot == **x).cloned().collect());
    a.append(&mut sort_fun(arr.iter().filter(|x| pivot < **x).cloned().collect()));
    a
  }
}

UPD
より美しいオプション：

fn sort_fun(arr: Vec<f64>) -> Vec<f64> {
    if arr.len() <= 1 {
        arr
    } else {
        let pivot = arr[arr.len() / 2];
        [
            sort_fun(arr.iter().filter(|&&x| pivot > x).cloned().collect()),
            arr.iter().filter(|&&x| pivot == x).cloned().collect(),
            sort_fun(arr.iter().filter(|&&x| pivot < x).cloned().collect()),
        ]
        .concat()
    }
}

配列を組み合わせる代わりにiter（）。Filter（...）chain（...）反復子を適用することで、コードをさらに美しくすることができます。これは完全にゼロコストですが、イテレータでは要素の数を決定できないため、半分に分割できないため、アルゴリズムを実装します。しかし、他の場合では、遅延イテレータは美しく、経済的で、高速です。

配列への可変参照を命令型アルゴリズムに渡し、関数自体には配列自体を渡します。これは、関数が終了すると自動的に破棄されます（再帰の各ステップのすべての一時配列も破棄されます）。

：

共通の変更可能な状態を使用して命令的に記述する場合、パフォーマンスの向上はプログラミングの複雑さとソーステキストの冗長性を補わないため、アプリケーションプログラマーが制御された言語からコンパイルされた言語に切り替えることは意味がありません。ただし、機能的に記述している場合-おそらくエンタープライズアプリケーションでのRustの使用を検討します。ソースコードの文字数を少し増やすだけで、大幅な高速化とメモリの節約につながるため、ガベージコレクタがないと、Rustの機能プログラムがより安定して安全になりますメモリを使用すると、脚を撃つことはできません（はい、それもnullです）。

世界には多くの高速で表現力のあるOOP言語がありますが、本当に高速のゼロコスト関数型言語はそれほどではありません。そして、もしRustがこの方向に進むとしたら、FPアプローチはより多くの支持者を見つけるでしょう。さらに、2019年の結果によると、ScalaとRustの両方が、主流の言語の減少の中で人気が大幅に増加しました。
