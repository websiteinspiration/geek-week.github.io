面白い考古学：虫眼鏡の下のRスタイルガイド

ご存知のように、コードは書かれたよりもはるかに頻繁に読み取られます。少なくとも著者以外の誰かがそれを読むことができるように、そしてスタイルガイドがあります。Rの場合、これは、たとえば、Hadleyのマニュアルになります。

スタイルガイドは、開発者間の暗黙の合意ではありません。多くのルールには、興味深い背景があります。なぜ矢印が<-等号よりも優れているの=か、Rの古参者が下線を好まないのか、推奨される行の長さがパンチカードにどのように関連しているかなど、さらに多くのこと。

1.代入演算子：<-vs=


利用可能なすべてのガイドは、他の現代の言語に慣れている<-、等号ではなく、非標準の演算子の使用を推奨してい=ます。三の他の演算子（<<-、->、->>（以前のバージョンに存在していた）にも言及されていません:=）。どうしてこの非標準の矢が必要なのでしょうか？

歴史は私たちにカードを明らかにします：Rでは、矢はSから来ました、そしてそれは次にAPLからそれを継承しました。 APLでは、割り当てと平等を区別できるようになりました。 Rでは、等価演算子は標準なので、違いは異なります。矢印が最初は代入演算子だった場合、等号は名前付きパラメーターにのみ値を割り当てました。 2001年には、等号が代入演算子になりましたが、矢印と同義になることはありませんでした。

=矢印の完全な交換 を検討できるのは何ですか？まず=、代入演算子がトップレベルでのみ機能する方法。たとえば、関数内では、すべてが以前と同様に機能します。

mean(x = 1:5)
# [1] 3
x 
# Error: object 'x' not found

mean (x <- 1:5)
# [1] 3
x
# [1] 1 2 3 4 5

ここで=は、関数パラメーターを設定するだけで<-、値を変数xに割り当てます。括弧内に代入演算を置くことで同じ効果を得ることができます（いいえ、これはまだLispではありません）。

mean ((x = 1:5))
# [1] 3
x
# [1] 1 2 3 4 5

...または中括弧で：

mean ({x = 1:5})
# [1] 3
x
# [1] 1 2 3 4 5

さらに、矢印は等号よりも優先されます。

x <- y <- 1
# OK

x = y = 2
# OK

x = y <- 3
# OK

x <- y = 4
# Error in x <- y = 4 : could not find function "<-<-"

最後の式は同等なので失敗し(x <- y) = 4、パーサーはそれを次のように解釈します

`<-<-`(x, y = 4, value = 4)

つまり、不適切な操作を実行しようとしています。最初にx yの値を割り当て、次にxとyの値を4に割り当てようとします。式は、操作の角括弧の優先度を変更した場合にのみエラーなしで処理されますx <- (y = 4)。

2.間隔


このガイドでは、演算子の間にスペースを置くことをお勧めします（もちろん、角括弧、：、：、::::を除く）。明らかに、これはGNUコーディング標準の一部です。ただし、この句は<-、代入演算子としての使用と密接に関連しています。例えば、

x <-1

それは何ですか？Xは-1未満ですか？または、xを1に設定しますか？

ただし、余分なスペースは、不足しているスペースと同じです。次に例を示します。

x <- 0
ifelse(x <-1, T, F)
# [1] TRUE

x <- 0
ifelse(x < -1, T, F)
# [1] FALSE

最初のケースでは、何の間にスペースがない<と-代入演算子を作成し、。

3.関数と変数の名前


名前の問題では、スタイルガイドは異なります。Hadleyガイドはすべての名前にアンダースコアを推奨しています。 Googleガイド-変数とキャメルスタイルをドットで区切り、関数を最初の小文字にします。バイオコンダクターは、関数と変数の両方にlowerCamelを推奨しています。この問題に関してRコミュニティには統一性がなく、すべての可能なスタイルを見つけることができます。

lowerCamel
period.separation
lower_case_with_underscores
allowercase
UpperCamel

ベースRの名前に対しても統一されたスタイルはありません（たとえば、rownamesとrow.namesは異なる関数です！）。判読不能なアローケースを考慮しない場合（Matlabユーザーのみがそれを気に入るはずです）、3つの最も人気のあるスタイルがあります。lowerCamel、小文字の_付き、小文字のドット付き10進数です。



関数名とパラメーターのさまざまなスタイルの人気（1つの名前がさまざまなスタイルに対応する場合があります）。出典：RasmusBååthのuseR！2017でのパフォーマンス。

ポイントツーポイントの分離は、オブジェクト指向プログラミングにおけるメソッドの使用を不吉に思い起こさせますが、歴史的に一般的です。この特定のスタイルは本当にR'vskyと見なすことができるほど一般的です。たとえば、ほとんどの基本的な関数は、それを具体的に使用します（そして、誰もがdata.tableとas.factorに会っただけです）。

しかし、分離_は、最も人気のないスタイルの1つです（ここでは、ハドレーは多数派に反対しています）。多くのRユーザーにとって、アンダースコアは煩わしいでしょう。人気のあるEmacs Speaks Statistics拡張機能では、デフォルトで代入演算子に置き換えられています<-。そしてもちろん、デフォルトの設定はほとんど誰も変更しません。

ただし、Emacs ESSの影響は、「尾を振る犬」のカテゴリからの説明です。より古くからの理由があります。Rの以前のバージョンでは、下線はarrowと同義<-でした。たとえば、2000年には次のようになります。

#      R

[1] 3
c_mean <- mean(c)

[1] 3

ここでは、変数を作成する代わりに、c_meanRは値3を最初に変数meanに割り当て、次に変数cに割り当てました。現代のRでは、そのような変態はもちろん発生しません。

人気がないため、このスタイルの_関数は基本的な関数にはほとんどありません。

最後に、長い名前を使用すると、lowerCamelスタイルは読みにくくなります。

4.中括弧


ガイドによると、新しい行は開始中括弧に続く必要があり、終了行は別の行にある必要があります（他に続く場合を除きます）。それら。そのように：

if (x >= 0) {
  log(x)
} else {
  message("Not applicable!")
}

ここでのすべてはそれほど興味深いものではありません。これは、K＆Rの標準的なインデントスタイルで、C言語とカーニガンとリッチーによる有名な本「Cプログラミング言語」（または著者の名前によるK＆R）にさかのぼります。

このスタイルの起源も明らかです。読みやすさを維持しながら、行を保存できます。初期のコンピューターでは、垂直方向のスペースは贅沢すぎました。たとえば、CはPDP-11で開発され、ターミナルは24行しかありませんでした。K＆Rブックを印刷するとき、このスタイルは紙を節約しました！

5. 80文字の文字列


ガイドによる推奨の行の長さは80文字です。マジックナンバー80は、Rだけでなく、他の多数の言語（Java、Perl、PHPなど）にも見られます。言語だけでなく、Windowsのコマンドラインも80文字で構成されています。

プログラミングで初めて、この数値は、データ用に正確に80列があった標準のIBMパンチカードではなく、1928年に登場しました。さらに興味深い質問は、なぜそのような規格が選ばれたのかということです。結局、長さが異なる（24列または45列の）パンチカードが以前は使用されていました。



最も一般的な答えは、パンチカードの長さをタイプライターの行の長さに関連付けます。最初のマシンはアメリカの標準用紙8½x 11インチ用に設計され、マージンのサイズに応じて72文字から90文字まで印刷できました。したがって、1行あたり80文字のバージョンは、最後の手段では当てはまりませんが、かなりありそうです。人間工学の観点からすると、80文字がちょうど中間点である可能性があります。

6.行のインデント：スペースとタブ


ガイドが推奨するスタイルは、タブではなく2つのスペースです。集計の拒否は非常に理解できます。TABの長さはテキストエディターによって異なります（2から8スペースまで）。それらを拒否すると、2つの利点が同時に得られます。最初に、コードは入力したとおりに完全に同じになります。第二に、推奨される文字列長の偶発的な違反はありません。同時に、もちろん、ファイルサイズを大きくします（2k19でそのようなマイクロ最適化に従事したいですか？）

紛争スペースとタブは長い歴史があり、宗教的なもの（Win vs Linux、Android vs iOSなど）と同等と見なすことができます）ただし、誰が勝利したかはすでにわかっています。調査によるとスペースを使用する開発者であるスタックオーバーフローは、タブを使用する開発者よりも多くの収入を得ます。スタイルガイドのルールよりも強力な議論ですね。

結論の代わりに：スタイルガイドのルールは奇妙で非論理的に見えるかもしれません。確かに、<-標準の演算子がある場合、なぜ矢印が使われるの=でしょうか。しかし、さらに深く掘り下げると、各ルールの背後にいくつかのロジックがあり、多くの場合すでに忘れられています。
