ヒューリスティックの代わりに理論：フロントエンド開発者としてより良くなる



翻訳ヒューリスティックの代わりにファンダメンタルズを使用して、より優れたフロントエンド開発者になる

私たちの経験から、非技術的な開発者や独学の人々は、理論的な原理ではなくヒューリスティックな方法に依存していることがよくあります。

ヒューリスティック-開発者が実践から学んだパターンと実績のあるルール。彼らは不完全にまたは限られた範囲で働くことができますが、十分な程度であり、真剣な思考を必要としません。以下にヒューリスティックの例をいくつか示します。

  • “ $(document).ready(function(){})jQueryサイトでコードを初期化するために使用する”
  • 「var self = thisコールバック関数でメソッドを呼び出すには構造が必要です」
  • 「矢印関数には演算子がありませんreturn」

同時に、他の問題の解決策を見つけるために理論的な原理を使用できます。彼は常に忠実であり、しばしば1つまたは別の要素のまさにそのデバイスを決定します。理論上の原則には、たとえば次のものが含まれます。

  • アロー関数の公式構文
  • Lodash APIドキュメント
  • 使用規則と機能垂直配置

注意：ヒューリスティックスの例を引用符で囲んでいます-理論的根拠の厳密​​さと比較してヒューリスティックスの職人的性質を強調するためです。どのヒューリスティックな例もすべてのケースに共通するわけではありませんが、十分な数の状況で機能するため、それらを使用する開発者は、その動作を完全に理解せずに機能するコードを受け取ります。

理論的アプローチの議論


技術教育を受けていない開発者は、理論的な原理を使用して問題を解決する傾向がないという事実にしばしば遭遇しました。原則として、これは彼らのキャリアの初めには彼らを学ぶ機会がなかったという事実によって説明され、ヒューリスティック手法は十分に機能するので、彼らはそれらを使い続けます。

ただし、見かけは複雑ですが、理論を学ぶことは非常に役立ちます。何のために？次に、その理論により、他の誰かのソリューションを探すことなく、ソリューションが機能することを確信し、新しい質問への回答を個別に表示できます。短期的には、ヒューリスティックアルゴリズムは単純で迅速な解決策のように見えるかもしれませんが、たとえあったとしても、それらはしばしば不完全な解決策につながります。

さらに、ヒューリスティック手法に依存しているため、実際に問題を解決する方法を学ぶことは決してありません。多くの場合、有効な解決策を見つけることができるかもしれませんが、遅かれ早かれ、停止状態になり、そこからの道は見えなくなります。C＆Pプログラマーは、作業においてヒューリスティックに依存しています。

開発者のスキルレベル基準


フロントエンド開発者にインタビューするとき、私たちは彼らのためにプログラミングタスクを設定し、彼らがGoogleであれスタックオーバーフローであれ、あらゆるソースを自由に使用できると言います。このようにして、開発者がヒューリスティックスまたは理論の支持者であるかどうかを簡単に判断できます。

前者は、例外なく、スタックオーバーフローを使用して、多かれ少なかれ適切な例からコードをコピーします。コードが計画どおりに機能しない場合にのみ、コードの微調整を開始しました。しばしば失敗します。

後者は、APIドキュメントで回答を探す傾向があります。そこで、特定の関数がとるパラメータの数、または目的のCSSプロパティの拡張形式の特定の構文に関する情報を見つけます。

面接の最初の5分間で、候補者がどのタイプのプログラマーに属しているかを正確に判別できます。

例


たとえば、ビルを見てみましょう。彼はいくつかのトレーニングコースを修了し、いくつかのJavaScriptタスクを解き、余暇にWebサイトを作成しましたが、JavaScriptを実際には学習しませんでした。

ビルがこのようなオブジェクトに出くわしたら：

const usersById = {
    "5": { "id": "5", "name": "Adam", "registered": true },
    "27": { "id": "27", "name": "Bobby", "registered": true },
    "32": { "id": "32", "name": "Clarence", "registered": false },
    "39": { "id": "39", "name": "Danielle", "registered": true },
    "42": { "id": "42", "name": "Ekaterina", "registered": false }
}

このようなオブジェクトは、ユーザーのリストと、ユーザーが特定のイベントに登録したかどうかを表示できます。

Billが登録済みユーザーのリストを取得する必要があるとします。つまり、それらを除外します。彼は、メソッドを.filter()使用してリストをフィルタリングするコードに出くわしました。だから彼は次のようなことを試みます：

const attendees = usersById.filter(user => user.registered);

そして、ここに彼が得るものがあります：

TypeError: usersById.filter is not a function

「なんらかのナンセンスなことです」とビルは考え.filter()ています。彼がフィルターとして働いているコードを見たからです。

問題は、ビルがヒューリスティック手法に依存していたことです。彼はそれを理解していませんfilter-配列で定義されたメソッドに対してusersById-メソッドを持たない通常のオブジェクトfilter。

混乱したビルは「javascriptフィルター」をググっています。彼は配列への多くの参照を見つけusersById、配列に変換するために何が必要かを理解しています。次に、「javascriptがオブジェクトを配列に変換する」というリクエストで、スタックオーバーフローを使用した例を見つけObject.keys()ます。その後、彼は試みます：

const attendees = Object.keys(usersById).filter(user => user.registered);

今回はエラーは表示されませんが、Billの驚いたことに、フィールドattendeesは空のままです。

実際にはObject.keys()、オブジェクトのキーを返しますが、その値は返しません。実際、変数の名前はuserオブジェクトではなくuser識別子、つまり文字列であるため、誤解を招きやすくなっています。registered行の属性が定義されていないため、filter各レコードはfalseと見なされ、配列は空のままになります。

ビルはスタックオーバーフローの回答を詳しく調べ、次の変更を加えます。

const attendees = Object.keys(usersById).filter(id => usersById[id].registered);

今回は結果が良くなり["5", "27", "39"]ます。しかし、ビルはIDではなく訪問者のオブジェクトを取得したいと考えていました。

ビジターをフィルタリングする方法を理解するために、イライラしたビルは「javascriptオブジェクトフィルター」を検索し、スタックオーバーフローの検索結果を調べて、次のコードでこの回答を見つけます。

Object.filter = (obj, predicate) => 
    Object.keys(obj)
          .filter( key => predicate(obj[key]) )
          .reduce( (res, key) => (res[key] = obj[key], res), {} );

ビルはこれらの行をコピーして試みます：

const attendees = Object.filter(usersById, user => user.registered);

すべてが機能します-理由は明らかではありません。ビルは、何が必要reduceでどのように使用されるのか理解していません。さらに、ビルは、グローバルオブジェクトに対してObject新しい非標準メソッドを定義したことを理解していません。

しかし、ビルは気にしません-それは機能します！結果は彼にとってまだ興味の対象ではありません。

基本に移りましょう


ビルが理論的アプローチの支持者であった場合、プロセスは次のようになります。

  1. 指定された入力データを識別し、目的の出力を決定するには、それらのプロパティの意味で：「IDを表す文字列をキーとするオブジェクトがあり、値はユーザーを表すオブジェクトです。値がユーザーオブジェクトになる配列を取得したいのですが、登録ユーザーのオブジェクトのみです」
  2. オブジェクト内を検索する方法を理解するには：「を呼び出すことで、オブジェクト内のキーの配列を取得できることを知っていますObject.keys()。配列は列挙型をサポートしているので、配列を取得したい。」
  3. このメソッドがキーを取得するのに役立ち、キーを値に変換する必要があることを理解するにmapは、別の配列の値を変換することによって新しい配列を作成する明白な方法を覚えておいてください：

    Object.keys(usersById).map(id => usersById[id])
  4. フィルタリングできるユーザーオブジェクトの配列があり、フィルタリングする実際の値が含まれていることを確認するには：

    Object.keys(usersById).map(id => usersById[id]).filter(user => user.registered)

このようにビルを行けば、彼は私たちのために働くことができました。

なぜ人々はより頻繁に理論に訴えないのですか？


時々彼らは彼女を知らないだけです。多くの場合、彼らは忙しすぎて、問題を解決するためのこの方法を学ぶ時間を見つけることができません。彼らは、このアプローチを自分のスキルの発達の障害となる習慣に変える危険を冒しています。

このようなエラーを回避するには、常に理論から始めます。プロセスの各段階で、どのような種類のデータを扱っているかを考えてください。いつも慣れ親しんだパターンに依存する代わりに、配列、オブジェクト、文字列などのプリミティブデータタイプを検討してください。関数またはメソッドを使用する場合は、ドキュメントを参照して、それらをサポートするデータ型、それらが取る引数、および結果を正確に把握してください。

このアプローチでは、最初の試行で実用的なソリューションを見つけることができます。特定の入力と必要な出力に基づいてアクションを特別に選択したので、その正確性を確認できます。ファジービジネスの定式化（「登録ユーザー」など）ではなく、各操作の基本（データ型と戻り値）を深く理解してください。
