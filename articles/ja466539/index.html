動的インポート


実際のインポート式を使用してコードとロードモジュールを分離する利点の1つは、モジュールをサポートしていないブラウザーで作業するタスクが開発者にあることです。

幸い、この問題は、サイズが小さく（サイズが約400バイト）、非常に高速なポリフィルの解決に役立ちます。

このポリフィルをWebプロジェクトに追加するのは非常に簡単です。それをインポートして、アプリケーションへのメインエントリポイントで初期化するだけです（コード内のコマンドを呼び出す前にimport()）。

そして、このスキームが機能するために行う必要がある最後のことimport()は、選択した名前を使用して（output.dynamicImportFunctionオプションを介して）コードに表示される動的コマンドの名前を変更する必要があることをRollupに伝えることです。動的インポートの可能性を実装するポリフィルは、デフォルト__import__で名前を使用しますが、カスタマイズすることもできます。

式の名前を変更する必要import()があるのimportは、JavaScriptではキーワードであるためです。これは、ポリフィル手段を使用して、標準チームimport()を同じ名前のチームに置き換えることを編成することは不可能であることを意味します。これを実行しようとすると、構文エラーが発生します。

ただし、Rollupがプロジェクトのビルド中にコマンドの名前を変更することは非常に良いことです。これは、標準のコマンドをソースコードで使用できることを意味します。さらに、将来的には、ポリフィルが不要になったときに、プロジェクトのソースコードを書き直す必要がなくなり、import以前は何とか別の名前が付けられていたものに変更されます。

プリロードするモジュールのリストを作成する


Rollup バンドルオブジェクトの入力フラグメントには、静的依存関係ツリーのインポートの完全なリストが含まれています。その結果、RollupのgenerateBundleフックで、プリロードする必要があるファイルのリストを簡単に取得できます。

npmにはmodulepreloadリストを生成するためのプラグインがありますが、依存関係ツリーの各入力ポイントに同様のリストを作成するには、数行のコードしか必要ありません。したがって、このコードのようなものを使用して、そのようなリストを手動で作成することを好みます。

{
  generateBundle(options, bundle) {
    //         .

    const modulepreloadMap = {};

    for (const [fileName, chunkInfo] of Object.entries(bundle)) {
      if (chunkInfo.isEntry || chunkInfo.isDynamicEntry) {
        modulepreloadMap[chunkInfo.name] = [fileName, ...chunkInfo.imports];
      }
    }

    //  -   ...

    console.log(modulepreloadMap);
  }
}
