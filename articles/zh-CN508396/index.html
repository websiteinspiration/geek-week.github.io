分子动力学转移至CUDA。第一部分：基础知识

本文的目的是提出使用CUDA技术使用分子动力学（MD）方法并行化数值模拟程序代码的问题。为什么根本没有必要这样做，因为已经有用于MD的软件包，包括可在CUDA上运行的软件包。事实是我正在开发自己的“非恒定力场”概念，而现有的MD程序并未实现这一概念。



为满足这些需求而重新编写他人的代码是一件相当不费力的任务，因此我决定转移已经编写的顺序代码，同时分享一些想法。此外，这是经常在这里闪烁像CUDA文章评论的回应此。



那么什么是分子动力学？在哈布雷（Habré）上已经有一些关于此主题的帖子，例如here或here。简而言之，MD是一种方法，它允许您模拟许多粒子（包括原子，离子，分子）的运动并根据该运动来计算系统的集合特性。这个怎么运作？假设对于一组N粒子，给出了一些初始坐标，速度，质量和（最重要的！）它们之间的相互作用定律。我们根据速度更改坐标。根据相互作用定律，我们计算粒子之间作用的力。因为我们知道力量和质量，所以我们知道加速度。我们在考虑加速度的情况下调整速度。再一次，我们继续更改坐标。所以我们重复一千遍直到你累 我们将无法获得足够的统计数据。







在实际应用中，基于一系列的粒子坐标函数的展开，“坐标-速度-力”的计算顺序有些复杂。最常见的是速度Verlet算法。基于Verlet积分器的MD程序的伪代码如下： 



init_md;
int iStep = 0;
while (iStep < nStep)
{
    reset_quantities;
    clear_cell_list;
    verlet_1stage;
    forces_calculation;
    verlet_2stage;
    temp_scale;
    termostat;
    some_output;
    iStep++;
}
final_md;


主程序循环执行用户指定的次数（nStep）。此循环的迭代称为时间步，或简称为时间步。将不考虑函数INIT_MD，reset_quantites，some_output和final_md，它们读取输入信息，清零值，输出数据，分配和释放内存等。从信息输出的角度来看，分子动力学是在视频卡上进行计算的理想选择：大多数计算不需要卸载。



函数verlet_1stage，verlet_2stage，temp_scale和termostat与运动和温度方程的积分器有关。forces_calculation函数计算作用在粒子之间的力，clear_cell_list函数引用加速计算的技术（稍后会对此进行详细介绍）。

函数可以使飞出模拟区域（框）边界的粒子出现在另一侧（周期性边界条件）： 



__device__ void put_periodic(float3& xyz, float3 vel, float mass, int type, cudaMD *md)
{
   if (xyz.x < 0)
   {
      xyz.x += md->leng.x;

      atomicAdd(&md->specAcBoxNeg[type].x, 1); 
      atomicAdd(&md->negMom.x, mass * (-vel.x)); 
   }
   else
     if (xyz.x >= md->leng.x)
     {
        xyz.x -= md->leng.x;

        atomicAdd(&md->specAcBoxPos[type].x, 1);
        atomicAdd(&md->posMom.x, mass * vel.x);
     }
}


该功能适用​​于具有矩形框几何形状的框。该代码用于测量x，其余的测量完全相似。该楞和revLeng字段是在相应尺寸的箱体尺寸和它的逆值。通常，要将粒子返回拳击，可以不使用条件运算符： 



xyz.x -= floor(xyz.x * md->revLeng.x) * md->leng.x;

“温度”函数temp_scale和termostat。分子系统与机械系统的不同之处在于它们具有温度的概念。 MD从分子动力学理论继承了温度的定义，其中温度是与颗粒平均动能成比例的值（有理由认为这是谬论）。要以这种方式保持所需的温度，您可以简单地缩放粒子速度，就像temp_scale函数那样： 



__global__ void temp_scale(int atPerBlock, int atPerThread, cudaMD *md)
{
   float k;
   int i;
   int id0 = blockIdx.x * atPerBlock + threadIdx.x * atPerThread;
   int N = min(id0 + atPerThread, md->nAt);

   k = sqrt(md->teKin / md->engKin);
   for (i = 0; i < N; i++)
   {
      md->vls[i].x *= k;
      md->vls[i].y *= k;
      md->vls[i].z *= k;
   }

   if (!blockIdx.x)
    if (!threadIdx.x)
     {
        md->engKin = md->teKin;
     }
}


teKin 字段包含目标动能，它与用户设置的温度成正比。该函数将速度矢量的所有分量简单地乘以k，然后一个流将动能设置为等于给定值。这里，变量k是在每个线程中定义的，尽管可以在一个线程中定义它（因为它是相同的），以便不做不必要的工作，特别是因为涉及平方根。这种原始的温度标度通常在模拟的初步阶段（平衡期）中使用，频率为几步。更复杂的温度维持算法（恒温器）可模拟与给定温度下的某些储层的粒子碰撞，其中最受欢迎的是Nose-Hoover恒温器：

2力的计算和单元列表



在分子系统中，作用在粒子上的力的主要来源（除了已经提到的外部场）是其他粒子。这些力的计算是MD最复杂，最耗时和最模糊的部分。最简单的近似之一是系统中的所有相互作用都是每个粒子对之间相互作用的总和（成对近似）。为了这些目的，引入了一对电势-两个粒子相互作用的势能对它们之间距离的函数依赖性。最有名的对势：伦纳德·琼斯（或称“ 6-12”），白金汉和博恩·梅耶·哈金斯的势。例如，我将给出前两个的解析表达式：





U=4ϵ(σ6r6−σ12r12)





和





U=Aexp⁡(−rρ)−Cr6,







其中U是一对粒子的势能，r是粒子之间的距离，所有其他字母都是仅取决于相互作用粒子类型的参数，并在初始化阶段设置。但是，除了能量以外，我们还对这些粒子之间的相互作用力感兴趣。通过对势的导数，这些力是微不足道的，为此，也很容易获得解析表达式并将其缝制到程序中。并且，由于作用力等于反应的强度，我们立即获得作用在两个粒子上的力，即伪代码： 



U = 0;
for (i = 0; i < N; i++)
	for (j = i + 1; j < N-1; j++)
	{
		dx = x[i] – x[j];
		dy = y[i] – y[j];
		dz = z[i] – z[j];
		r = sqrt(dx*dx + dy*dy + dz*dz);
		U += pair_pot(r, types[i], types[j]);
		F = force(r, types[i], types[j])
		fx[i] = F * dx;
		fx[j] = -F * dx;
		fy[i] = F * dy;
		fy[j] = -F * dy;
		fz[i] = F * dz;
		fz[j] = -F * dz;

	}


代码很简单：我们找到粒子之间的距离，将其代入势对函数和力函数（均取决于相互作用粒子的类型），然后计算力的投影。循环执行N（N-1）/ 2次，其中N是粒子数。为了至少以某种方式减少计算量，我们可以利用以下事实：这些电位的值随距离而迅速减小。在大于6-8埃的距离处，无法计算电位，它将几乎为零。添加用于检查距离（或更确切地说是其平方）的条件，我们将减少计算次数，但不会减少循环的迭代次数。对于大型系统，大多数检查将给出错误。下列技术可以通过丢弃成对原子间距较大的部分对来减少检查次数。



有两种主要方法可减少对粒子对的搜索次数：邻居列表（又称Verlet列表）和单元格列表（单元格列表）。首先，对于每个粒子，我们记住其在势能半径内的邻居。需要以一些以前未知的周期性来更新邻居列表，这是该方法的缺点。因此，让我们关注单元格列表，其本质是将模拟的体积（盒）划分为（子）单元格（单元）。最初，选择像元大小大于或等于截止半径。这样就足以将每个单元格内以及相邻单元格之间的粒子对挑选出来。像元大小可能会有所不同，主要是确定您需要检查哪些像元对。实际上，这是尝试以立方体描述球体的尝试。传统上，为了实现顺序代码中的单元格列表，引入了两个数组，头和列表。第一个的大小等于细胞的数量，第二个的大小等于粒子的数量。第一个存储给定单元格中一个粒子的索引，第二个索引对应于粒子的索引，内容对应于该单元格中下一个粒子的索引（如果该单元格中没有更多的粒子，则为-1），下面给出插图。

{
   int iC, jC;	//  
   int i, j;	//

   //   
   for (iC = id0; iC < N; iC++)
   {
      i = md->chead[iC];

      {

我认为，这种实现方式无法完全显示视频卡的潜力，因为单个流可以处理一个单元，通常该单元明显少于粒子数。其次，在处理一对粒子的功能内，仅通过原子操作即可工作。让我们尝试更强地并行化任务，让每个线程处理一对相邻的单元格（我们相信已经创建了此类对的列表）： 



__global__ void cell_list1(int pairPerBlock, int pairPerThread, cudaMD *md)
{
   int iP, iC, jC;	//     
   int i, j;		//  

   int id0 = blockIdx.x * pairPerBlock + threadId.x * pairPerThread;
   int N = min(id0 + cellPerThread, md->nCell);

   {
      iC = md->cellPairs[iP].x;
      jC = md->cellPairs[iP].y;
      i = md->chead[iC];

      {

                i-j;
             j = md->clist[j];
         }
         i = md->clist[i];
      }

}

用于计算两个不同单元之间的交互的函数： 



__device__ void pair_between_cell1(float3 xyzA, float3* fxyzA, int typeA, float3 xyzB, float3* fxyzB, int typeB, float3 shift, cudaMD* md, float& engVdw, float& engCoul)
{
    float r, r2, f;
    float dx, dy, dz;
    cudaVdW* vdw;

    dx = xyzA.x - xyzB.x - shift.x;
    dy = xyzA.y - xyzB.y - shift.y;
    dz = xyzA.z - xyzB.z - shift.z;
    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)
    {
        r = 0.0;
        f = 0.0;

        f += real_ewald(r2, r, md->chProd[typeA][typeB], md->alpha, engCoul); 

        vdw = md->vdws[typeA][typeB];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw);


        fxyzA->x += f * dx;
        atomicAdd(&(fxyzB->x), -f * dx);
        fxyzA->y += f * dy;
        atomicAdd(&(fxyzB->y), -f * dy);
        fxyzA->z += f * dz;
        atomicAdd(&(fxyzB->z), -f * dz);
    }
}

__device__ float cu_fe_lj(float r2, cudaVdW* vdw, float& eng)
// U = 4e[(s/r)^12 - (s/r)^6]
{
    float r2i = 1.0 / r2;
    float sr2 = vdw->p1 * r2i;
    float sr6 = sr2 * sr2 * sr2;

    eng += vdw->p0 * sr6 * (sr6 - 1.0);
    return vdw->p2 * r2i * sr6 * (2.0 * sr6 - 1.0);
}

__device__ float cu_fer_buck(float r2, float& r, cudaVdW* vdw, float& eng)
// U = A exp(-r/ro) - C/r^6
{
    float r2i = 1.0 / r2;
    float r4i = r2i * r2i;
    if (r == 0.0)    //  r , 
        r = sqrt(r2);

    eng += vdw->p0 * exp(-r / vdw->p1) - vdw->p2 * r4i * r2i;
    return vdw->p0 * exp(-r / vdw->p1) / r / vdw->p1 - 6.0 * vdw->p2 * r4i * r4i;
}


函数返回力（必须乘以距离矢量的投影），并将能量添加到变量eng。指向这些功能之一的指针存储在cudaVdW结构中。一般来说，要获得运动轨迹，就无法计算出能量，这需要进行不同的统计。如果我们从这些函数中删除能量表达式，则计算可能会加速。您可以执行以下操作：只需几个步骤即可调用完整功能，而在其余时间中，只需调用简化的模拟量即可。库仑相互作用的一部分由real_ewald函数计算： 



__device__ float real_ewald(float r2, float& r, float chprd, float alpha, float& eng)
{
    float ar; 
    float erfcar;
    float kqq = chprd * d_Fcoul_scale; // q[i]*q[j]*1/4pie0;

    r = sqrt(r2);
    ar = alpha * r;
    erfcar = erfc(ar);

    eng += kqq * erfcar / r;
    return kqq / r / r2 * (erfcar + 2 * ar / d_sqrtpi * exp(-ar * ar));
}


这与熟悉的库仑定律不同，但在另一篇文章中对此有更多论述。

__global__ void all_pair(cudaMD* md)
{
    int i, j;
    int step = blockDim.x * gridDim.x;
    int ex = 0;  // exit flag
    int nAt = md->nAt;

    i = 0;
    j = blockIdx.x * blockDim.x + threadIdx.x + 1;
    while (1)
    {
        while (j >= nAt)
        {
            i++;
            if (i >= nAt - 1)
            {
                ex = 1;
                break;
            }
            j = i + 1 + j - nAt;
        }
        if (ex) break;
        global_pair(i, j, iStep, md);
        j = j + step;
    }
} 

__device__ void global_pair(int i, int j, cudaMD* md)
{
    float r, r2, f;
    float engCoul = 0;
    float engVdw = 0;
    cudaVdW* vdw;
    int ti, tj;

    float dx = md->xyz[i].x - md->xyz[j].x;
    float dy = md->xyz[i].y - md->xyz[j].y;
    float dz = md->xyz[i].z - md->xyz[j].z;

    //   
    if (dx > md->halfLeng.x)
        dx -= md->leng.x;
    else
        if (dx < -md->halfLeng.x)
            dx += md->leng.x;

    if (dy > md->halfLeng.y)
        dy -= md->leng.y;
    else
        if (dy < -md->halfLeng.y)
            dy += md->leng.y;

    if (dz > md->halfLeng.z)
        dz -= md->leng.z;
    else
        if (dz < -md->halfLeng.z)
            dz += md->leng.z;

    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)    
    {
        r = 0.0;
        f = 0.0;

        ti = md->types[i];
        tj = md->types[j];

        f += real_ewald(r2, r, md->chProd[ti][tj], md->alpha, engCoul);

        vdw = md->vdws[ti][tj];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw); 

        dx *= f;
        dy *= f;
        dz *= f;

        atomicAdd(&(md->frs[i].x), dx);
        atomicAdd(&(md->frs[j].x), -dx);
        atomicAdd(&(md->frs[i].y), dy);
        atomicAdd(&(md->frs[j].y), -dy);
        atomicAdd(&(md->frs[i].z), dz);
        atomicAdd(&(md->frs[j].z), -dz);

        atomicAdd(&(md->engCoul1), engCoul);
        atomicAdd(&(md->engVdW), engVdw);

    }
}
