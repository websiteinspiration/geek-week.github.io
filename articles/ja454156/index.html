産業オートメーションのタイヤとプロトコル：すべての仕組み



確かに多くの人があなたを知っているか、たとえば原子力発電所や多くの技術ラインを持つプラントなどの大きな自動オブジェクトがどのように制御されているかを見たことさえあります。主なアクションは、多くの画面、ライト、リモートのある広い部屋で行われることがよくあります。このコントロールコンプレックスは通常、メインコントロールルームと呼ばれ、生産施設を監視するためのメインコントロールパネルです。

確かに、ハードウェアとソフトウェアの観点からこれがどのように機能するのか、そこでどのようなデータ転送プロトコルが使用されているのか疑問に思っていました。この記事では、メインコントロールパネルにさまざまなデータがどのように送信されるか、機器のコマンドがどのように送信されるか、コンプレッサーステーション、プロパン製造装置、自動車の組立ライン、さらには下水ポンプシステムを制御するために何が必要かを理解します。

下位レベルまたはフィールドバスは、すべての出発点です。


この不明瞭な単語のセットは、制御デバイスと下位の機器（入出力モジュールや測定デバイスなど）との通信手段を記述する必要がある場合に使用されます。

制御デバイスとは、PLCを意味します。プログラマブルロジックコントローラー（英語PLC）、またはPKA、つまり プログラマブルオートメーションコントローラー（英語：PAC）。PLCとPKAの間にはいくつかの違いがありますが、この記事の枠組み内ではそれらは重要ではないため、簡略化のために「コントローラー」という一般用語を使用します。

asushniksのロシア語圏コミュニティでは、コントローラーと他のデバイス間の通信チャネルは、「フィールド」からのデータ転送を担当するため、通常「フィールドバス」と呼ばれます。

「フィールド」は、コントローラーが相互作用する特定の機器（センサーやアクチュエーターなど）が、遠く離れた場所の路上、フィールド、夜のカバーの下にあるという事実を表す専門用語です。また、センサーをコントローラーから0.5 m離れた場所に配置して、たとえば自動化キャビネット内の温度を測定できることは問題ではありませんが、「フィールド内」にあると見なされます。ほとんどの場合、I / Oモジュールに到達するセンサーからの信号は、数十から数百メートル（場合によってはそれ以上）の距離をカバーし、リモートサイトまたは機器から情報を収集します。したがって、実際には、コントローラがこれらの同じセンサーから値を受信する交換バスは、通常、フィールドバスと呼ばれますが、下位バスまたは産業用バスと呼ばれることはあまりありません。

ここで、ヨーロッパとアメリカでは、「フィールド内」にあるデバイスのみがデータ伝送媒体ではなく、フィールドレベルと見なされます。ロシアの現実では、「フィールドバス」または「下位バス」という用語はおそらく少し曖昧であり、I / Oモジュールからコントローラーへ、またはその逆にデータを転送する方法を示します。


産業施設の自動化の一般的なスキーム

そのため、センサーからの電気信号は、いくつかのセンサーが接続されているケーブルライン（ほとんどの場合、特定の数のコアを備えた通常の銅ケーブル）に沿って特定の距離を通過します。次に、信号は処理モジュール（I / Oモジュール）に入り、そこでコントローラーが理解できるデジタル言語に変換されます。さらに、この信号はフィールドバスを介して直接コントローラーに送信され、すでに完全に処理されています。これらの信号に基づいて、コントローラー自体のロジックが構築されます。リターンパスがあります。コントローラーから、フィールドバスを介した制御コマンドが出力モジュールに入り、デジタルからアナログに変換され、ケーブルラインを介してアクチュエーターやさまざまなデバイス（上の図には示されていません）に送信されます。

上位レベル：ガーランドからワークステーション全体まで


上のレベルは、プロセスを制御する通常の人間のオペレーターが触れることができるすべてのものを指します。最も単純なケースでは、上のレベルは電球とボタンのセットです。電球は、システムで発生したいくつかのイベントについてオペレーターに信号を送り、ボタンはコントローラーにコマンドを与えるために役立ちます。このようなシステムは、非常によく似ているため（記事の冒頭の写真からわかるように）、「ガーランド」または「クリスマスツリー」と呼ばれます。

オペレーターの方が幸運であれば、オペレーターパネルをトップレベルとして取得します。一種のフラットパネルコンピューターであり、コントローラーから表示するデータを何らかの方法で受け取り、画面に表示します。このようなパネルは通常、自動化キャビネット自体に取り付けられているため、原則として、立った状態で操作する必要があります。これにより、不都合が発生するだけでなく、画像の品質とサイズ（小さなフォーマットのパネルの場合）には、多くのことが望まれます。



そして最後に、前例のない寛大さの魅力-通常のパーソナルコンピューターであるワークステーション（またはいくつかの複製）です。

ワークステーションやフラットパネルコンピュータで情報を視覚的に表示するには、SCADAシステムという専用のソフトウェアを使用します。 SCADAは、監視制御とデータ収集のシステムとして人間の言語に翻訳されます。技術プロセスを視覚化するヒューマンマシンインターフェース、これらのプロセスの制御システム、パラメーターアーカイブシステムとイベントログ、アラーム管理システムなど、多くのコンポーネントが含まれています。これらすべてにより、オペレーターは生産で発生するプロセスの全体像を把握できるだけでなく、プロセスを管理し、技術プロセスからの逸脱に迅速に対応することができます。

トップレベルの機器は、何らかの方法でコントローラーと相互作用する必要があります（そうでなければ、なぜそれが必要なのですか？）。この対話には、上位レベルのプロトコルと、イーサネットやUARTなどのいくつかの伝送技術が使用されます。もちろん、クリスマスツリーの場合、そのような高度な設定は必要ありません。電球は通常の物理的なラインを使用して照明され、洗練されたインターフェイスやプロトコルはありません。

一般に、この上位レベルはフィールドバスほど面白くないので、この上位レベルはまったくない場合があります（シリーズからオペレーターが見るものは何もないため、コントローラーは何をどのように行うかを理解します）。

古代のデータプロトコル：ModbusとHART


知っている人はほとんどいませんが、世界の創造の7日目に、神は休まずにModbusを創造しました。 HARTプロトコルに加えて、Modbusはおそらく最も古い産業用データ転送プロトコルであり、1979年にすでに登場しています。

最初はシリアルインターフェイスが伝送媒体として使用されていましたが、TCP / IPを介してModbusが実装されました。これは、マスター/スレーブ方式（マスター/スレーブ）による同期プロトコルであり、要求/応答の原理が使用されます。プロトコルはかなり重くて遅いです。交換レートは受信機と送信機の特性に依存しますが、通常、特にシリアルインターフェイスを介した実装では、数百ミリ秒近くです。

さらに、Modbusデータ転送レジスタは16ビットであり、実数型と倍精度型の転送に即座に制限を課します。それらは部分的に送信されるか、精度が失われます。 Modbusは、高い交換レートが必要なく、送信されたデータの損失が重要でない場合に、依然として広く使用されています。さまざまなデバイスの多くのメーカーは、Modbusプロトコルを独自の方法で拡張し、非標準機能を追加することを望んでいます。したがって、このプロトコルには、多くの変異や標準からの逸脱がありますが、現代でも成功しています。
HARTプロトコルは80年代から存在し、2線式電流ループラインを介して交換するための工業用プロトコルであり、4-20 mAセンサーとその他のデバイスをHARTプロトコルのサポートで直接接続します。

HART回線の切り替えには、特別な装置、いわゆるHARTモデムが使用されます。出力で、たとえばModbusプロトコルをユーザーに提供するコンバータもあります。

4-20 mAセンサーのアナログ信号に加えて、プロトコル自体のデジタル信号も回路で送信されるため、HARTはおそらく注目に値します。これにより、デジタル部分とアナログ部分を1本のケーブルで接続できます。最新のHARTモデムは、コントローラーのUSBポートに接続したり、Bluetoothを介して接続したり、従来の方法でシリアルポートを介して接続したりできます。10年前、Wi-Fiとの類推により、ISMバンドで動作するWirelessHART標準が登場しました。

第2世代プロトコルまたはそれほど業界ではないISA、PCI（e）、およびVMEバス


ModbusおよびHARTプロトコルは、ISA（MicroPC、PC / 104）またはPCI / PCIe（CompactPCI、CompactPCIシリアル、StacPC）、VMEなどの完全ではない産業用バスに置き換えられました。

さまざまなボード（モジュール）を接続して特定の統一された信号を処理できる汎用データ転送バスを自由に使える計算機の時代が到来しました。原則として、この場合、プロセッサモジュール（計算機）は、他のデバイスとのバス通信を提供する、いわゆるフレームに挿入されます。フレーム、またはトリットオートメーションエンジニアがそれを「クレート」と呼んでいるように、必要なI / Oカード（アナログ、ディスクリート、インターフェースなど）が追加されているか、またはこれらすべてがフレームなしのサンドイッチの形で互いに貼り付けられています。その後、バス上のこのマニホールド（ISA、PCIなど）は、プロセッサモジュールとデータを交換します。プロセッサモジュールは、センサーから情報を受け取り、何らかのロジックを実装します。


PCIバス上のPXIシャーシ内のコントローラおよびI / Oモジュール。出典：National Instruments Corporation

これらのISA、PCI（e）、およびVMEバスでは、特にこれらの場合はすべて問題ありません。為替レートは混乱せず、システムコンポーネントは単一のフレームに配置されているため、コンパクトで便利なホットスワップ可能なI / Oカードを使用できます。そして、そうではありませんが、それでも本当にしたくはありません。

しかし、軟膏にはハエがあり、ハエはありません。分散システムをこのような構成で構築するのは非常に困難です。ローカルバスは交換可能です。他の下位ノードまたはピアノードとデータを交換するために何かを発明する必要があります。TCP/ IPまたは他のプロトコルの上にある同じModbus。さて、2つ目はあまり楽しいものではありません。I/ Oボードは通常、いくつかの統一された信号が入力されるのを待ち、フィールド機器とのガルバニック絶縁がないため、さまざまな変換モジュールや中間回路から庭をフェンスで囲む必要があり、要素のベースが非常に複雑になります。


ガルバニック絶縁を備えた中間信号変換モジュール。出典：DataForth Corporation

「そして、産業用バスでの交換のためのプロトコルはどうですか？」- あなたが尋ねる。しかし、何も。そのような実装はありません。ケーブルラインを介して、信号はセンサーから信号コンバーターに渡され、コンバーターはディスクリートまたはアナログI / Oボードに電圧を提供し、ボードからのデータはOSツールを使用してI / Oポートを介して既に読み取られています。そして、特別なプロトコルはありません。

最新の工業用タイヤとプロトコルの仕組み


今何？今日まで、自動システムを構築する古典的なイデオロギーは少し変わっています。自動化も便利であるという事実から始まり、ノードが互いに離れている分散自動化システムへのトレンドで終わる多くの要因が役割を果たしました。

おそらく、今日、ビルディングオートメーションシステムには、ローカライズされた自動システムと分散された自動システムという2つの基本的な概念があると言えます。

データの収集と制御が1つの特定の場所に集中しているローカライズされたシステムの場合、独自の通信プロトコルを備えたコントローラーを含む、共通の高速バスによって相互接続されたI / Oモジュールの特定のセットの概念が必要です。さらに、原則として、入出力モジュールには信号コンバータとガルバニック絶縁の両方が含まれます（もちろん、常にではありません）。つまり、最終消費者は自動化システムに存在するセンサーとメカニズムのタイプを理解し、さまざまなタイプの信号に必要な入出力モジュールの数を数え、それらをコントローラーとの1つの共通ラインに接続するだけで十分です。この場合、原則として、各メーカーは入出力モジュールとコントローラーの間でやり取りする好きなプロトコルを使用しており、多くのオプションが存在する可能性があります。

分散システムの場合、ローカライズされたシステムについて述べられていることはすべて当てはまります。さらに、個々のコンポーネント（たとえば、入出力モジュールのセットと情報を収集して送信するためのデバイス）が、ブースのどこかにある非常にスマートなコントローラーではないことが重要です油をブロックするクレーンの隣のフィールドは、同じノードと、遠く離れたメインコントローラーと効果的な為替レートで相互作用する可能性があります。

開発者はどのようにしてプロジェクトのプロトコルを選択しますか？すべての最新の交換プロトコルはかなり高いパフォーマンスを提供するため、多くの場合、製造元の選択は、この同じ産業用バスの交換レートによるものではありません。プロトコル自体の実装はそれほど重要ではありません。これは、システム開発者の観点からは、何らかの内部交換構造を提供するブラックボックスであり、外部の干渉用に設計されていないためです。ほとんどの場合、彼らは実用的な特性に注意を払います：電卓のパフォーマンス、タスクにメーカーのコンセプトを適用する便利さ、必要なタイプの入力/出力モジュールの存在、バスを壊すことなくモジュールをホットスワップする機能など。

人気のある機器サプライヤーは、産業用プロトコルの独自の実装を提供しています。たとえば、有名なシーメンス企業は、独自の一連のProfinetおよびProfibusプロトコル、B＆R-Powerlinkプロトコル、Rockwell Automation-EtherNet / IPプロトコルを開発しています。この例のリストの国内ソリューション：ロシアの会社FastwelのFBUSプロトコルバージョン。

EtherCATやCANなどの特定の製造元に関連付けられていない、より一般的なソリューションがあります。記事の続きでこれらのプロトコルを詳細に分析し、特定のアプリケーション（自動車および航空宇宙産業、電子機器、位置決めシステム、ロボット工学）に最適なプロトコルを見つけます。連絡を取り合う！
