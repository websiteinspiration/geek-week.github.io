CassandraからKubernetesへの移行：機能とソリューション



私たちは定期的にApache Cassandraデータベースに遭遇し、Kubernetesベースのインフラストラクチャのフレームワーク内でそれを操作する必要があります。この記事では、CassandraをK8sに移行するために必要な手順、基準、および既存のソリューション（オペレーターの概要を含む）についてのビジョンを共有します。

「女性を管理できる人は国家に対処します」


カサンドラとは これは、大量のデータを管理し、単一障害点のない高可用性を提供するように設計された分散ストレージシステムです。プロジェクトに長い紹介はほとんど必要ないので、特定の記事のコンテキストに関連するCassandraの主な機能のみを示します。

  • CassandraはJavaで書かれています。
  • Cassandraトポロジには、いくつかのレベルがあります。
    • ノード-Cassandraの1つのデプロイ済みインスタンス。
    • ラック-1つのデータセンターにある、任意の属性によって結合されたCassandraインスタンスのグループ。
    • データセンター-1つのデータセンターにあるCassandraインスタンスのすべてのグループの合計。
    • クラスター-すべてのデータセンターのコレクション。
  • CassandraはIPアドレスを使用してホストを識別します。
  • 読み取りおよび書き込み操作の速度のために、Cassandraはデータの一部をRAMに格納します。

次に、Kubernetesへの実際の移行について説明します。

移行のチェックリスト


CassandraからKubernetesへの移行について言えば、移動に伴って管理することがより便利になることを願っています。これには何が必要で、これには何が役立ちますか？

1.データストレージ


既に指定したように、CassandaがRAMに保存するデータの一部（Memtable内）。しかし、ディスクに保存されるデータには、SSTableの形式で別の部分があります。このデータには、エンティティログログ（すべてのトランザクションのレコードもディスクに保存される）が追加されます。


Cassandra書き込みトランザクションスキーム

Kubernetesでは、PersistentVolumeを使用してデータを保存できます。よく開発されたメカニズムのおかげで、Kubernetesでのデータの操作は毎年簡単になります。


Cassandraを使用する各ポッドに、PersistentVolumeを割り当てます

Cassandra自体がデータレプリケーションを意味し、このための組み込みメカニズムを提供していることに注意することが重要です。したがって、多数のノードからCassandraクラスターを構築している場合は、CephやGlusterFSなどの分散システムを使用してデータを保存する必要はありません。この場合、ローカル永続ディスクまたはマウントを使用してホストディスクにデータを保存するのが合理的hostPathです。

別の質問は、機能ブランチごとに個別の開発環境を作成するかどうかです。この場合、正しいアプローチは、1つのCassandraノードを発生させ、データを分散ストレージに格納することです。前述のCephとGlusterFSはオプションです。その後、開発者は、Kuberntesクラスターのノードの1つが失われた場合でも、テストデータが失われないようにします。

2.モニタリング


Kubernetesでモニタリングを実装するための実質的に非代替的な選択肢は、Prometheusです（これについては、対応するレポートで詳しく説明しました）。Cassandraは、Prometheusのメトリックエクスポーターをどのように使用していますか？そして、いくつかの点でさらに重要なのは、Grafanaに適したダッシュボードです。


Cassandra

エクスポーター用のGrafanaでのグラフの外観の例は、jmx_exporterとcassandra_exporterの 2つだけです。

次の理由により、私たちは自分用に最初のものを選択しました。

  1. JMXエクスポーターは成長と発展を続けていますが、Cassandraエクスポーターは適切なコミュニティサポートを受けることができませんでした。Cassandraエクスポーターは、Cassandraのほとんどのバージョンをまだサポートしていません。
  2. フラグを追加することで、javaagentとして実行できます-javaagent:<plugin-dir-name>/cassandra-exporter.jar=--listen=:9180。
  3. 彼にとっては、Cassandra Exporterと互換性のない適切なダッシュボードがあります。

上記のCassandraクラスターの構造に従って、そこで説明されているすべてのものをKubernetesの用語に翻訳してみましょう。

  • Cassandraノード→ポッド
  • カサンドララック→ステートフルセット
  • Cassandraデータセンター→StatefulSetのプール
  • Cassandra Cluster→???

Cassandraクラスター全体を一度に管理するために、いくつかの追加エンティティーが欠落していることがわかりました。しかし、何かがない場合は作成できます！Kubernetesでは、これは独自のリソース（カスタムリソース定義）を決定するためのメカニズムです。

1つのCassandraノードのデータのフルバージョンをスケジュールに従って保持したいと考えています。 KubernetesはCronJobを使用して便利な機会を提供しますが、ここではCassandraがスティックをホイールに挿入します。

Cassandraデータの一部がメモリに保存されていることを思い出させてください。完全バックアップを作成するには、メモリ（Memtables）からディスク（SSTables）にデータを転送する必要があります。この時点で、Cassandraノードは接続の受け入れを停止し、クラスターから完全にシャットダウンします。

その後、バックアップ（スナップショット）が削除され、スキーム（キースペース）が保存されます。そして、バックアップだけでは何も得られないことがわかります。Cassandraノードが担当していたデータ識別子を保存する必要があります。これらは特別なトークンです。


Cassandraノードが担当するデータを特定するためのトークンの配布

。KubernetesでGoogleのCassandraバックアップを削除するためのスクリプトの例は、このリンクにあります。スクリプトが考慮しない唯一のポイントは、スナップショットを削除する前にノードにデータをダンプすることです。つまり、現在の状態ではなく、少し前の状態のバックアップが行われます。しかし、これはノードが動作しなくなるのを防ぐのに役立ちます。これは非常に論理的に思えます。

set -eu

if [[ -z "$1" ]]; then
  info "Please provide a keyspace"
  exit 1
fi

KEYSPACE="$1"

result=$(nodetool snapshot "${KEYSPACE}")

if [[ $? -ne 0 ]]; then
  echo "Error while making snapshot"
  exit 1
fi

timestamp=$(echo "$result" | awk '/Snapshot directory: / { print $3 }')

mkdir -p /tmp/backup

for path in $(find "/var/lib/cassandra/data/${KEYSPACE}" -name $timestamp); do
  table=$(echo "${path}" | awk -F "[/-]" '{print $7}')
  mkdir /tmp/backup/$table
  mv $path /tmp/backup/$table
done


tar -zcf /tmp/backup.tar.gz -C /tmp/backup .

nodetool clearsnapshot "${KEYSPACE}"

KubernetesのCassandra向けの既製ソリューション


現在CassandraをKubernetesにデプロイするために何を使用していますか。これらのうちどれが所定の要件に最も適していますか？

1. StatefulSetまたはHelmチャートソリューション


基本的なStatefulSetを使用してCassandraクラスターを開始することは、適切なオプションです。HelmチャートとGoテンプレートを使用して、Cassandraをデプロイするための柔軟なインターフェースをユーザーに提供できます。

通常、これは正常に機能します...予期しないことが発生するまで-たとえば、ノードがダウンするまで 標準のKubernetesツールでは、上記の機能をすべて考慮に入れることはできません。さらに、このアプローチは、ノードの交換、バックアップ、復元、監視など、より複雑な使用のために拡張できる範囲が非常に制限されています。

代表者：

  • メインのHelmリポジトリからのチャート。
  • bitnamiからのグラフ。

どちらのグラフも同等に優れていますが、上記の問題が発生しやすくなっています。

このようなオプションは、広範なクラスター管理機能を提供するため、より興味深いものです。他のデータベースと同様に、Cassandraオペレーターを設計するための適切なパターンは、Sidecar <-> Controller <-> CRDのようになります。CRD：

既存のオペレーターを検討してください。

1. instaclustrによるCassandra-operator


  • Github
  • 意欲：アルファ
  • ライセンス：Apache 2.0
  • 実装：Java

これは確かに、Cassandraが管理するデプロイメントを提供する会社からの非常に有望で急速に発展しているプロジェクトです。上記のように、HTTP経由でコマンドを受け入れるサイドカーコンテナーを使用します。Javaで記述されているため、より高度なclient-goライブラリ機能が欠けている場合があります。また、オペレーターは1つのデータセンターに対して異なるラックをサポートしていません。

しかし、オペレーターには、監視サポート、CRDを使用した高レベルのクラスター管理、バックアップの削除に関するドキュメントなどの利点があります。

2. Jetstackによるナビゲーター


  • Github
  • 意欲：アルファ
  • ライセンス：Apache 2.0
  • 実装：Golang

DB-as-a-Serviceをデプロイするためのオペレーター。現在、ElasticsearchとCassandraの2つのデータベースをサポートしています。これには、RBACを介したデータベースへのアクセス制御などの興味深いソリューションがあります（このために、独自の個別のnavigator-apiserverが発生します）。注目に値する興味深いプロジェクトですが、最後のコミットは1年半前に行われたため、その可能性が明らかに低下しています。

4. RookのCassandra-operator


  • Github
  • 意欲：アルファ
  • ライセンス：Apache 2.0
  • 実装：Golang

開発が思ったほど速くないオペレーター。これは、クラスターを管理するための十分に検討されたCRD構造を備えており、ClusterIPを使用したサービス（同じ「ハック」）を使用してノードを識別するという問題を解決します。現時点では、監視とバックアップはありません（ちなみに、私たちは自分自身で監視を開始しました）。興味深い点は、この演算子を使用してScyllaDBをデプロイできることです。

注意：プロジェクトの1つでこのオペレーターを少し変更して使用しました。全運転中（約4か月の運転中）、オペレーターの作業に問題はありませんでした。

  • 意欲：アルファ

リストの最年少オペレーター：最初のコミットは2019年5月23日に行われました。すでに、彼は彼の兵器庫に私たちのリストから多数の機能を持っています。それらの詳細はプロジェクトリポジトリで見つけることができます。オペレーターは、一般的なオペレーターSDKに基づいています。すぐに使用できる監視をサポートします。他の演算子との主な違いは、Pythonで実装され、Cassandraノード間の通信に使用されるCassKopプラグインの使用です。
