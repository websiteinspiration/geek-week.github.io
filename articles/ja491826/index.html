みなさん、こんにちは。

Bitrixでのロギング（イベントの監査）に関する情報を探していたところ、Habrには何もありませんでした。残りは何かを調べましたが、誰がそこに見つけますか？

ナレッジベースを補充するために、この記事を書くことにしました。私の経験を共有し、レーキの可能性について警告します。

問題の定式化


私の仕事は、広告構造の最も単純な会計システムを開発することでした。国家契約の条件の下で、システムはBitrix（バージョン15）に基づいて機能する必要があります。

Bitrixの側ですべてを自転車に乗ることは可能でしたが、私はそれが顧客に対して不誠実すぎると判断し、Bitrixの機能が最大限に使用されました。

  • ユーザ認証
  • データストレージシステム（EAV）
  • データエディター
  • 監査イベントハンドラー
  • ユーザーアクションを承認するためのロールモデル
  • ユーザー管理
  • ディレクトリを操作する

この記事は主に監査イベントについてですが、承認と情報ブロックレコードカードへのカスタムブックマークの追加についても少しお話します。

監査イベント


開発されたシステムでは、データの操作はAPIを介して実行され（新しいオブジェクトをマップに追加し、その座標を変更します）、APIはすべてのデータ変更要求を処理しないため、オブジェクトのその他のパラメーターはBitrixエディターを介して編集され、APIのみの監査は完了しません。 Bitrix側の監査が必要です。

監査とは何かを知らなかったというわけではありませんが、頻繁に記述する必要はありません。通常、これは古いバージョンの行（古いデータ）を別のテーブルに書き込むことで解決され、実装は完全にDBMS側で実行されます。また、変化前の状態と変化後の状態が同時に存在します。

DBMSトリガーに関する監査を作成する場合、またはBitrixイベントに関する監査を作成する場合、違いは大きくありません。
処理するイベントはスクリプトに登録されています "bitrix / php_interface / init.php "、ファイルがない場合は、作成する必要があります。

use Topliner\Scheme\Logger;
//  
require_once($_SERVER['DOCUMENT_ROOT'] . '/vendor/autoload.php');
//        
if (!defined('IBLOCK_MODULE')) {
    define('IBLOCK_MODULE', 'iblock');
}
//    
AddEventHandler(IBLOCK_MODULE, 'OnIBlockElementAdd',
    Array(Logger::class, 'OnAdd'));
//     ,  -    ,              ,

これは1つのハンドラーの例です。私の実装ではいくつかあります。コードはリポジトリーで表示でき、リンクは記事の最後にあります。

各イベントのイベントを処理するメソッドのシグネチャには独自のものがあり、Bitrixのドキュメントまたはデビューでソースを確認します（より明確に、すべてのイベントと呼び出しハンドラのすべての瞬間は、デバッグ中にソースでも確認できます）。

ここで、DBMSを使用する場合、現在の（古い）データと記録されるデータ（新しい）の両方があり、Bitrixを使用する場合、古いデータと新しいデータのどちらかがあり、古いものと新しいものは同時に存在しません。

もう1つの問題は、データを変更する前にいくつかのイベントがあり、それらの違いを理解できなかったことです。同じストーリーは、データを変更した後のいくつかのイベントのトリガーに関するもので、目は選択の豊富さから見つめていますが、実際は単なる選択の錯覚です。

グローバル状態の使用


イベントハンドラー間で状態を保存して渡すには、グローバル変数が必要です。グローバル変数はリファクタリングできないものなので、クラスの静的プロパティを使用します。

class Logger
{
    const CHANGE = 'change';
    const REMOVE = 'remove';
    const CREATE = 'create';
    const UNDEFINED = 'undefined';

    public static $operation = self::UNDEFINED;

    const NO_VALUE = [];

    private static $fields = self::NO_VALUE;
    private static $properties = self::NO_VALUE;
    private static $names = self::NO_VALUE;
}

ここでは、「フィールド」と「プロパティ」の使用について説明する必要があります。フィールドは、各情報ブロックエントリが持つパラメーター（識別子、名前、情報ブロックの「親」）です。プロパティは、特定のタイプの情報ブロックのレコードに固有のパラメーターです。EAVモデルでは、これらは属性とその値です。

「名前」は属性の名前であり、1つのイベントには属性と名前の識別子があり、別のイベントには識別子しかありません。ジャーナルの美しいレコードでは、名前を保存する必要があります（もちろん、識別子を差し引くことができますが、何らかの理由で望まない場合があります）。

「操作」は現在の操作であり、「フィールド」を操作する場合、Bitrixは特定のイベントを提供します。「プロパティ」（関数「SetPropertyValues」と「SetPropertyValuesEx」）を操作すると、操作のタイプにイベントはありません。呼び出しの前後にイベントがありますが、そうではありません実行される操作（追加/更新/削除）がわかっているため、操作もハンドラーからハンドラーに転送する必要があります。

たぶん私はそれを理解していなかったし、Bitrixで値とその状態を同時に確認できます。あるいは、前の状態と後の状態を別々に記録する必要があるかもしれませんが、何らかの理由でログエントリは「プロパティ％name％は、変更前の％値でした％=>変更後の％値になりました」ので、グローバル状態の維持に多くの問題があります。

変更が必要かどうかの判断


奇妙なことに、情報ブロックのレコードを変更するためにハンドラーが呼び出されます。
したがって、ハンドラーでは、パラメーターで受け取った情報ブロックのレコードを理解する必要があります。
レコードの情報ブロックは2つの値によって特徴付けられます：情報ブロック自体（ 'IBLOCK_ID'）とそのセクション（ 'IBLOCK_SECTION_ID'）、およびセクション識別子は、インデックス 'IBLOCK_SECTION_ID'と時々 'IBLOCK_SECTION'を持つ値の配列にあるため、セクション識別子を「IBLOCK_SECTION」を優先する両方のキー。

さらに、状況によってはセクション識別子を原則的に決定できない場合があるため、監査ログにエントリを追加する必要があるかどうかは、情報ブロック識別子に基づいてのみ確認する必要があります。

情報ブロックとセクションを決定したら、イベントをジャーナルに登録する必要があるかどうかを決定します。

変更前の状態の保存


各情報ブロックエントリは2つの部分で構成され、1つは「フィールド」であり、これらのパラメータはメソッドによって変更されます。

  • CIBlockElement ::追加（）
  • CIBlockElement ::更新（）
  • CIBlockElement ::削除（）

もう1つの部分は「プロパティ」メソッドです。
  • CIBlockElement :: SetPropertyValues（）
  • CIBlockElement :: SetPropertyValuesEx（）

理由は覚えていませんが、コードでCIBlockElement :: SetPropertyValuesEx（）を使用しないことをお勧めします。

イベントハンドラーのデータの各部分は互いに分離されているため、[保存]ボタンをクリックすると、監査ログに2つのエントリを作成できます。

「フィールド」と「プロパティ」のイベントのシグネチャは異なります。「フィールド」を処理する部分では、フィールドの現在の状態を保存し、値を操作に設定します。「プロパティ」を処理する部分では、プロパティとその名前を保存します。

もちろん、これは「before」ハンドラで行います。

情報ブロックカードにカスタムタブを追加する（Bitrix分類器内）


カードを追加するには、監査と同様のアプローチが使用されます。ハンドラーを追加します。

AddEventHandler('main', 'OnAdminIBlockElementEdit',
    Array(PermitTab::class, 'OnInit'));

'OnInit'メソッドでは、Bitrixエディターのタブを操作するための他のメソッドを定義します。

class PermitTab
{
    const LINKS = 'links';

    function OnInit($arArgs)
    {
        $permits = BitrixScheme::getPermits();
        $pubPermits = BitrixScheme::getPublishedPermits();
        $blockId = (int)$arArgs['IBLOCK']['ID'];
        $letShow = $blockId === $permits->getBlock()
            || $blockId === $pubPermits->getBlock();
        $result = false;
        if ($letShow) {
            $result = [
                'TABSET' => 'LinksToConstructs',
                'GetTabs' => [static::class, 'GetTabs'],
                'ShowTab' => [static::class, 'ShowTab'],
                'Action' => [static::class, 'Action'],
                'Check' => [static::class, 'Check'],
            ];
        }

        return $result;
    }
}

最初に、もちろん、この情報ブロックでユーザータブを表示する必要があることを確認します。必要に応じて、メソッドを設定します。

ここで特別なことは何もありません。src/ Topliner / Scheme / PermitTab.phpリポジトリのソースを参照し、ドキュメントを読んでください。

ユーザーアクションを承認するためのロールモデル


Bitrixの承認方法は、権利の線形スケールで機能します。つまり、1つは1番目と2番目、3番目は不可、もう1つは2番目と3番目になる可能性がありますが、1つ目は不可で、Bitrixを直接実装することはできません。

Bitrixでのこのようなトリッキーなシナリオでは、いくつかの権限のチェックがあり、操作を実行するときのコードで、ユーザーに権限があるかどうかを確認し、1つの役割の1番目と2番目の役割と、役割の2番目と3番目の役割を許可します。

        $constSec = BitrixScheme::getConstructs();
        $isAllow = CIBlockSectionRights::UserHasRightTo(
            $constSec->getBlock(), $constSec->getSection(),
            BitrixPermission::ELEMENT_ADD, false);
        if (!$isAllow) {
            $output['message'] = 'Forbidden, not enough permission;';
        }

  1. $ constSec-> getBlock（）-情報ブロック識別子
  2. $ constSec-> getSection（）-セクション識別子
  3. BitrixPermission :: ELEMENT_ADD-許可コード

必要な情報ブロックとセクションの分類子（情報ブロックディレクトリ）で、適切な権限を役割に割り当てます。役割をユーザーに配布すると、すべてがあなたの管理下に置かれます。

ディレクトリを操作する


Bitrixの「新しい」（2019年11月現在）ディレクトリは「HighloadBlock」と呼ばれ、それらの使用は少し標準的ではありません。

各Hyloadブロックは個別のテーブルに保存できるため、読み取るテーブルの名前を決定する必要があるたびに、ディレクトリにアクセスできます。これを常に行わないようにするために、Bitrixではデータにアクセスするためのクラスのインスタンスを作成する必要があります。インスタンスは2行で作成されます。

CModule::IncludeModule('highloadblock');
$entity = HighloadBlockTable::compileEntity('ConstructionTypes');
$reference = $entity->getDataClass();

ここで、「ConstructionTypes」は参照コードです。

これらの2行を常に記述しないようにするために、参照インスタンスを作成するクラス（src / Topliner / Bitrix / BitrixReference.php）を記述できます。

次に、通常のBitrix ORMクラス（D7）と同様にインスタンスを操作します。

/* @var $reference DataManager */
$data = $reference::getList(array(
    'select' => array('UF_NAME', 'UF_XML_ID'),
    'filter' => array('UF_TYPE_ID' => 0)
    ));
