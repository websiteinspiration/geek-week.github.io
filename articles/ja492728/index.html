Telegram / Slack / E-mailでアスタリスクと送信を見逃した

コールセンターがあります。キューが構成されたアスタリスク/ FreePBXがあります。コールに対応する必要のあるエージェントがいます。しかし、潜在的な顧客は非常に多く、エージェントが少ないため、最初のエージェントが2番目のエージェントに到達できず、1分間並んでハングアップした後、電源をオフにします。

しかし、何らかの理由で彼らは電話をかけました！多分彼らは会社にお金をもたらしたいですか？FreePBXの例を使用して、顧客とお金の両方を返却してみましょう。

キュー設定では、フェールオーバー先を指定できます-キューがいっぱいになった場合、タイムアウト期間が経過した場合などに、コールをどこに送信するかなど。ただし、発信者がフェイルオーバー先にリダイレクトされる前に接続が切断されることがよくあります。そのような場合のための既製の解決策はありません。したがって、私たちはカットの下に行き、あなた自身のものを書いてください-テレグラム/スラック/電子メール/どこかにアラートを送ることで。

そして、私たちが最初に理解する必要があるのは、FreePBXがキューログを書き込む方法です。最も単純なケースでは、これはテキストファイル/ var / log /アスタリスク/ queue_logです。

したがって、定期的な通話は、302221XXXXを呼び出したサブスクライバー8964467XXXXXからログに書き込まれます。302221XXXXは、603番の最初の位置にあり、エージェントガールが8秒後に応答しました。彼らは133秒話し、加入者は最初に電話を切りました-誰もがそのような丁寧なマネージャーを持っているでしょう！

それは彼についてです、私たちはメッセンジャーに書き込みます。

これを行うには、ABANDONイベントの発生をログファイルでキャッチする必要があります。私はこれがどのようにしてできるのか長い間戸惑っていましたが、偶然にも私は素晴らしいツール-MONITに出くわしました。この例では、キューログを監視し、「ABANDON」という行が表示されるとすぐに、このイベントを処理するためのスクリプトを呼び出します。

これを行うには、パッケージをインストールした後：

yum install monit

設定します-ファイル/etc/monit.d/queue_logを作成し、文字通り3行を書き込みます。

check file queue_log with path /var/log/asterisk/queue_log
    if content = "ABANDON" then
        exec "/var/lib/asterisk/agi-bin/qlogevent.sh $EVENT"

したがって、キューログに「ABANDON」という単語が表示され、サブスクライバーの接続を途中で切断するように指示された場合、$ EVENTパラメーターを指定した/var/lib/asterisk/agi-bin/qlogevent.shスクリプトが呼び出されます。この単語が出現した行。
それは簡単です。

そして今は、チェスではありません。考える必要があります。

スクリプトに渡された行には、発信者の番号に関する情報が含まれていません。

1581728720|1581728690.59367|210|NONE|ABANDON|1|1|10

タイムスタンプ（1581728720）、一意のコールID（1581728690.59367）、キュー番号（210）、イベント自体、およびその他のいくつかの重要でないパラメーターのみが含まれます。それら。加入者番号を自分で検索する必要があります。そして、どのように？また、呼び出しのUIDがあるため、非常に簡単です。

このコールのログ、具体的には、サブスクライバーがキューに入る行をもう一度見てみましょう。

1581728710|1581728690.59367|210|NONE|ENTERQUEUE||8996453|1

「ENTERQUEUE」イベントが発生した場合、パラメーターの1つは必要なサブスクライバー番号です。この行は、queue_logファイルでのみ見つかります。これはまさにbashスクリプトが行っていることであり、MONITが注意深く呼び出しました。

#!/bin/bash
#MONIT_DESCRITION - ,      .
# timestamp    - ( №1).
timedate=$(echo $MONIT_DESCRIPTION | cut -d\| -f1 | cut -d\: -f2 | gawk '{print strftime("%d.%m.%Y %H:%M:%S",$0);}')
# UID ( №2)
call_id=$(echo $MONIT_DESCRIPTION | cut -d\| -f2)
#   ( №3)
qnum=$(echo $MONIT_DESCRIPTION | cut -d\| -f3)

#      
case $qnum in
    210)
        qname=" 1"
        ;;
    220)
        qname=" 2"
        ;;
esac

#      UID,     ENTERQUEUE
str=$(grep "|$call_id|$qnum|.*|ENTERQUEUE" /var/log/asterisk/queue_log)
#      ( №7)
caller_id=$(echo $str | cut -d\| -f7)
#     ( №1)
time_enter=$(echo $str | cut -d\| -f1)
#     ( №1   ABANDON)
time_abandon=$(echo $MONIT_DESCRIPTION | cut -d\| -f1 | cut -d\: -f2)
# 
data=" $caller_id $timedate   $qname,       $(($time_abandon-$time_enter)) ."
#    /.     Slack.
/usr/bin/curl -X POST -H 'Content-type: application/json' --data '{"text":"'"$data"'"}' "https://hooks.slack.com/services/XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"

実際、それだけです！

確かに、1つのインシデントがあります-MONITはすべてを監視するように設計されています。それ自体が監視を必要とするため、定期的に落ちる。
