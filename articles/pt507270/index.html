Selecionando dependências para um projeto

Saudação, Khabrovsk. Antecipando o início do curso "Fullstack JavaScript Developer" , preparamos outra tradução útil para você.


Todo desenvolvedor da Web encontrou esse quebra-cabeça: qual dependência escolher? Por que escolhemos jQuery, não Prototype, ou Prototype, e não Mootools, ou preferimos Vue em vez de React, ou preferimos Angular em vez de Ember, ou Lodash, em vez de Sublinhado, e obtemos um número ilimitado de combinações.

Eu próprio tomo uma decisão tão rapidamente que pode parecer inconsciente. No entanto, não é assim, e decidi que era hora de descrever alguns dos meus processos de trabalho, pelo menos para ajustá-los ou revisá-los ao longo do tempo.

Tipos de Dependências


Quando seleciono novas dependências, corro entre dois incêndios. Preciso de um projeto em nível de estrutura ou de uma solução pronta (plug-in, biblioteca ou algo semelhante). O processo de seleção também é influenciado por fatores específicos.

Abaixo está uma lista de perguntas de acordo com as quais priorizo ​​ao considerar quaisquer dependências para um projeto.

1. O projeto está sendo desenvolvido ativamente?


A resposta a esta pergunta é especialmente importante para projetos grandes, pois geralmente mais de uma pessoa trabalha no código neles. Uma visita a um repositório inativo está longe de ser um motivo para abandonar a tecnologia, mas preciso entender o potencial técnico da nova dependência.

Aqui estão dois aspectos aos quais presto muita atenção:

  • Atividade recente de confirmação - uma rápida olhada na página principal do projeto no github geralmente ajuda aqui.
  • A idade de qualquer solicitação de pull aberto - solicitações de pull longo suspenso pode indicar que ninguém está envolvido no projeto há um longo tempo (eu mesmo fiz isso com o Nodemon)


2. O projeto é usado?


A popularidade não é um fator decisivo para mim, mas quanto mais software for usado, mais erros serão encontrados nele.

Nesta questão, presto atenção a:

  • O número total de problemas - um grande número de problemas abertos - nem sempre é ruim;
  • Número de emissão encerrada;
  • A taxa na qual a emissão é encerrada.


A velocidade com que o problema se encerra não é um fator particularmente importante para mim, especialmente em um projeto em que existem muitos deles. Para mim, ter a discussão deles é mais importante.

3. Profundidade da Dependência


Há uma piada na comunidade Node e npm de que o objeto mais pesado do universo não é o Sol, nem uma estrela de nêutrons, nem um buraco negro node_modules.

Nós, como desenvolvedores da Web, assumimos um grande risco e agradecemos a Deus por ter aparecido recentemente Deno, que está tentando resolver esse problema com a ajuda stdlibque impede o crescimento do gráfico de dependência.

Portanto, é importante não o número de dependências (que você pode ver em sites como o npmjs), mas a profundidade. É estranho que eu tenha visto isso apenas com o Snyk.io , ou seja, dependências de nodemon .

4. Disponibilidade de exemplos


Quanto às bibliotecas do lado do cliente, preciso de exemplos interativos de código de trabalho e código suficiente em geral para poder recriar a demonstração.

Então, imediatamente, você pode obter a resposta para a pergunta: "Este projeto fará o que eu quero dele".

5. Documentação e complexidade


O segundo mais importante, além do código de trabalho, é a documentação para mim. Para projetos pequenos, espero ver exemplos de código no README. Em projetos maiores, espero algumas páginas de documentação.

Quando olho para a documentação, procuro exemplos com os quais começar e me pergunto: a montagem deste projeto requer ferramentas adicionais?

Muitos dos projetos que encontrei na documentação também contêm suposições sobre a presença de qualquer conhecimento, e o número de tais suposições aumenta o grau de complexidade, especialmente se eu não estiver familiarizado com os sistemas em questão no projeto.

Por exemplo, por experiência, quase todo projeto Python envolve saber como instalar uma dependência, e eu não me enquadro nessa categoria.

6. Números da versão


Se o projeto não usar sempre, isso já é uma bandeira vermelha, pois uma quantidade muito grande de informações pode ser encontrada no número da versão.

Lembre-se de que em todas as versões principais são indicadores de alterações críticas, versões secundárias são alterações nas funções e versões de patches são correções de erros.

Há uma certa nebulosa em torno da documentação e refatoração (quando a API externa funciona da mesma maneira, e a implementação interna é diferente, e você libera, mas isso não corresponde a nenhuma alteração no semver).

No entanto, um grande número de versão principal implica uma alta frequência de alterações críticas, o que pode indicar possíveis problemas com o suporte do meu projeto.

Um grande número menor (por exemplo, a ferramenta de linha de comando Snyk, agora versão 1.323.0) envolve muitas funções e, possivelmente, uma nova versão para cada função.

Parece-me que o grande número da versão do patch implica estabilidade e suporte constante ao projeto.

7. idade


Na vida real, isso seria envelhecer, mas no mundo dos programadores, a idade de um projeto pode dizer muito.

A “velhice” do projeto fala de sua maturidade ou pode servir como um sinal peculiar de abandono. Mas o que é a velhice? Se o Node completou 11 anos em 2020, provavelmente todos os projetos cuja idade excedeu 10 anos podem ser considerados antigos.

Normalmente evito projetos muito jovens (com alguns meses).

Muitas dessas dicas realmente importam: as dependências devem ser criadas e estar no projeto em que estou trabalhando?
Eu sempre me faço essa pergunta e, se o problema específico que estou tentando resolver for suficientemente claro, posso até retirar o código, copiar a licença e citar a fonte para que eu tenha um vício completamente congelado.
