コンストラクターとセッターにパラメーターを値渡しするための教育プログラム（最新のC ++、例）

次の投稿のhabr.com/en/post/460831/#comment_20416435のコメントとそこで展開されたディスカッションから判断すると、コンストラクターまたはセッターに引数を正しく渡す方法に関する記事は、Habréの妨げにはなりません。StackOverflowには同様の資料がたくさんありますが、ここでは覚えていません。

その記事の例は完全に正しいので、記事の著者は完全に正しいからです。次に例を示します。

// .
struct person {
  person(std::string first_name, std::string last_name)
    : first_name{std::move(first_name)} // 
    , last_name{std::move(last_name)} // std::move  !
  {}
private:
  std::string first_name;
  std::string last_name;
};

そのようなコードを使用すると、クラスを使用するためのすべての（まあ、ほぼすべての）可能なオプションをカバーできます。

std::string first{"abc"}, last{"def"};
person p1{first, last};  // (1)   
person p2{std::move(first), last}; // !!!   
person p2{std::move(first), std::move(last)}; // (3)  
person p3{"x", "y"}; //

組み合わせの爆発を恐れていない場合は、&&にチャンスを与えることができます（しかし、なぜですか？速度が実際に向上することはなく、オプティマイザは居眠りしません）。

std ::文字列がなく、独自に作成した大きなクラスのオブジェクトがあり、人々にそれを移動させる（コピーしない）ようにしたい場合でも、この場合、コンストラクターがこの大きなクラスからコピーするのを禁止して、どこにでも渡すのではなく、沿って ＆＆。これはより信頼性が高く、コードは短くなります。

  //   ,      shared_ptr:
  //  ,

なぜこれが起こっているのですか？基本原理は何ですか？それは簡単です。オブジェクトは、原則として、そのプロパティを所有する必要があります。

オブジェクトが何かを所有したくない場合は、この「何か」のshared_ptrを所有できます。ちなみに、この場合、shared_ptrも定数リンクではなく値で渡す必要があります。記事の最初の最初の例と違いはありません。

//  (  ).
struct person {
  person(std::shared_ptr<portfolio> pf) 
    : pf{std::move(pf)} // std::move    
  {}
private:
  std::shared_ptr<portfolio> pf;
};

auto psh = std::make_shared<portfolio>("xxx", "yyy", "zzz");
...
person p1{psh};
person p2{std::move(psh)}; // (X)  ,  psh

注意：std :: shared_ptrの移動は完全に合法であり、shared_ptrリンクカウンターをメモリにロックするオーバーヘッド（数百のCPUサイクル）とその増分を排除します。オブジェクトおよびそのオブジェクトへの他のリンクの存続期間には影響しません。ただし、（X）はもちろん、以下のコードのpshリンクが不要になった場合にのみ実行できます。

モラル：const＆を一般的に使用しないでください。状況に応じて見てください。

PS
コンストラクタパラメータを渡すときは、（）ではなく{}を使用してください。ファッショナブルでモダンな若者。

PPS
結論として、もう1つ：std :: move（）は実際には何も移動せず、それ自体がアセンブラー命令をゼロに変換します。 std :: move（）が行うことは、リンクに特別な「スティッキーラベル」を付けて、&&-右辺値参照に変換することだけです。さらに、このラベルを使用すると、関数パラメーターのタイプを個別に「一致」させることができます（たとえば、&&-パラメーター用の個別の関数オーバーロードと＆-パラメーター用の個別の関数オーバーロードを持つことができます）。 &&-ショートカットの意味は、呼び出しコードが呼び出されたことを通知できるようにすることです。「必要に応じて、このリンクから値を食べることができます。もう必要ありません。しかし、あなたが食べて骨を残した場合のみ、私はまだ残りのスケルトンのデストラクタを呼び出す必要があります。」同じ成功で、通常の＆リンクを送信することも可能です（それらの「食べる」オブジェクトを使用することもできます）が、&&より良いセマンティクスが判明します。混乱することはありません。どこで食べることができ、どこで匂いを嗅ぐことができるかです。

この点で、名前std :: move（）は非常に失敗したと認識されるべきです。std :: eat_me_if_you_want（）またはstd :: bon_appetit（）と呼ぶのが正しいでしょう。ただし、std :: move（）の方が短いです。
