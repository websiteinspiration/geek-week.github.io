コード分​​析デモSecond Reality


2013年7月23日、Second Realityデモ（1993）のソースコードが公開されました。多くの人と同じように、私は何年にもわたって私たちに多大なインスピレーションを与えてきたデモの内部を見たいと切望していました。

アセンブラーからモノリシックな混乱が見られると思っていましたが、その代わりに、驚いたことに、いくつかの言語をエレガントに組み合わせた複雑なアーキテクチャーを発見しました。私はこれまでこのようなコードを見たことがなく、デモの開発に不可欠な2つの側面を完全に表しています。

  • チームワーク。
  • 難読化。

いつものように、私はノート用の記事を作成しました。これにより、誰かが数時間節約でき、他の人がより多くのソースコードを読んで、より経験豊富なエンジニアになるように促したいと思います。

パート1：はじめに


デモ


コードに着手する前に、伝説のデモをHDビデオ（Michael Hut）でキャプチャするためのリンクを示します。今日、これはグラフィカルな不具合なしにデモを完全に評価する唯一の方法です（DOSBoxでも正しく起動できません）。


コードとの最初の接触

デモを起動


理解するのは難しいですが、リリースされたデモはDosBoxで起動できます。名前を変更してU2.EXE、適切な場所から実行する必要があります。

ゲーム開発における私の経験に基づいて、私は常に全体像が次のようになることを期待しています：

  • SECOND.EXE：実行可能ファイルにすべての効果を持つエンジン。
  • REALITY.FC：WADDoom 独自の暗号化形式のアセット（音楽、効果音、画像）。

しかし、それを読んだ後MAIN/PACK.C、私は間違っていることに気づきました。SecondRealityエンジンは単なるローダーであり、割り込みサーバー（DISと呼ばれます）です。各シーン（「PART」とも呼ばれます）のデモは、完全に機能するDOS実行可能ファイルです。各パーツはローダーローダーによってロードされ、次々に起動されます。パーツは最後に暗号化された形式で保存されSECOND.EXEます。


  • REALITY.FC デモ中に演奏される2つの楽曲が含まれます（難読化を埋めるために、最初にフィリングとマーカーを追加しました）。
  • SECOND.EXE ブートローダーとデモ割り込みサーバー（DIS）が含まれています。
  • 終了後、SECOND.EXEデモの32の部分（パート）が実行可能なDOSファイル（暗号化）として追加されます。

このようなアーキテクチャには多くの利点があります。

  • : PART , _start (450 ).
  • EXE SECOND.EXE -.
  • : Loader DIS 20 . DOS .
  • : PART PART .
  • / : , PART ( ), : EXE , .
  • PARTプログラミングには任意の言語を使用できます。コードにはC、Assembly ...、Pascalがあります。

推奨読書


Second Realityのソースコードを理解するための3つの柱は、VGA、アセンブラー、およびPCアーキテクチャー（PICおよびPITプログラミング）です。ここにいくつかの信じられないほど便利なリンクがあります：

  • VGAアーキテクチャ：FreeVGAプロジェクト
  • Art of Assembly（特に割り込みとISRに関する第17章）。
  • PIT 8254のプログラミング
  • ソースコードには「FAR」と「NEAR」のポインターがたくさんあるので、ポインターデータ型に関するIntelのBasic Architecture Documentation、特に第4.3章を読むと役立ちます。
  • グラフィックプログラミングブラックブックの Michael Abrashによる包括的な記事：第3章：（ZenタイマーとPIT）、第24章：VGAツアー、第47章、48章、および49章：（モードX）。

パート2：リアリティエンジン


第1部で説明したように、Second Realityの基礎は次の要素で構成されています。

  • DOS実行ファイルとしてのブートローダー。
  • メモリマネージャー（単純なスタックプール）
  • デモ割り込みサーバー（DIS）。

このパートでは、エンジンとブートローダーを読みたいプログラマに推奨事項を示します（DISについては次のパートで説明します）。

エンジンコード


エンジンコードは100％ASMですが、非常によく記述され、かなりよく文書化されています。

  • MAIN/U2.ASM （芯）。
  • MAIN/U2A.ASM （ユーティリティ）。
  • MAIN/VMODE.ASM （VGA手順）。

擬似コードでは次のように書くことができます：

    exemus  db 'STARTMUS.EXE',0
    exe0    db 'START.EXE',0
    ...
    exe23   db 'ENDSCRL.EXE',0

    start:
       cli                         ; Disable all interrupts
       mov     ah,4ah              ; Deallocate all memory
       call checkall               ; Check for 570,000 bytes of mem, 386 CPU and VGA
       call file_getexepath        
       call dis_setint             ; Install Demo Interrupt Server on Interrupt 0fch
       call file_initpacking       ; Check exe signature (no tempering) !
       call file_setint            ; Replace DOS routines (only OPENFILE, SEEK and READ) on Interrupt 021h
       call flushkbd               ; Flush the keyboard buffer
       
       call  checkcmdline          ; check/process commandline

       ;======== Here we go! ========
       call vmode_init             ; Init VGA (not necessarly Mode13h or ModeX), each PARTs had its own resolution

       mov si,OFFSET exe0
       call executehigh            ; loaded to high in memory. Used for loading music loaders and stuff.
   
       call  _zinit ; Start music
       call  restartmus

       mov   si,OFFSET exe1     ;Parameter for partexecute: Offset to exec name
       call  partexecute
       ; Execute all parts until exe23

       call fademusic
       ;======== And Done! (or fatal exit) ========

    fatalexit:
       mov cs:notextmode,0
       call vmode_deinit

すべての手順は非常に読みやすいです。

  1. DIS割り込みサーバーを割り込みに設定します0fch。
  2. DOSシステムコールを割り込みで置き換える021h（詳細については、「開発モードと製品モード」のセクションを参照してください）。
  3. EMSメモリを介してサウンドカードに音楽をダウンロードします。
  4. 音楽を実行しています。
  5. デモの各部分を実行します。
  6. できた！

手順の詳細execute：

 execute:
      cld
      call  openfile ; Open the DOS executable for this PART
      call  loadexe  ; loads the specified exe file to memory, does relocations and creates psp
      call  closefile
      call  runexe   ;runs the exe file loaded previously with loadexe.
                     ; returns after exe executed, and frees the memory
                     ; it uses.

メモリマネージャー


セカンドリアリティがMMUを介して複雑なメモリマネージャーを使用しているという多くの伝説があり、エンジンには痕跡がありませんでした。メモリ管理は実際にはDOSに転送されます。エンジンはすべてのRAMを解放し、要求に応じてそれを配布することから始まります。唯一のトリッキーなトリックは、ヒープの最後からRAMを割り当てる機能です。RAMが多すぎる場合は、malloc DOS戻り値を使用して行われます。

パート3：DIS


デモ割り込みサーバー（DIS）は、異なるPART間のデータ交換からVGAとの同期まで、各PARTに幅広いサービスを提供します。

DISサービス


実行時には、PART、DISサーバーがサービスを提供します。機能のリストはにありますDIS/DIS.H。

最も重要なサービス：

  • 異なるPART間の交換（dis_msgarea）：DISは64バイトの3つのバッファーを提供するため、PARTは前のPARTのローダーからパラメーターを受け取ることができます。
  • Emulation Copper（dis_setcopper）：VGAの状態によって切り替えられる操作を実行できるAmiga Copperシミュレーター。
  • Dev / Prod（dis_indemo）モード：PARTがDEVモードで実行されている（つまり、ビデオを初期化する必要がある）か、またはPRODモードのブートローダーから起動されていることをPARTに認識させます。
  • VGAフレーム数（_dis_getmframe）
  • VGAバックストップを待機していdis_waitbます（）。

デモ割り込みサーバーコード


DISソースコードも100％ASM ...であり、かなりよくコメントされています。

  • DIS/DIS.ASM（割り込みハンドラはintに設定されています0fch）。
  • DIS/DISINT.ASM （DIS手順自体）。
  • Second RealityもCで部分的に記述されているため、コードにはC：DIS/DIS.HおよびのインターフェイスがありますDIS/DISC.ASM。

使い方


DISは、プログラムによるintの割り込みハンドラーとして設定され0fchます。SECOND.EXEデモの実行中に内部で実行できること、またはDevモードで常駐プログラム（TSR）として実行できることの素晴らしい点です。この柔軟性により、開発中にさまざまなPARTデモを個別にテストできます。

銅



「Copper」は、Amigaのデモの開発者が愛したコプロセッサーです。これはオリジナルのチップセットの一部であり、ビデオ機器と同期したプログラム可能なコマンドストリームを実行できます。PCにはそのようなコプロセッサーはなく、Future CrewはDIS内で実行されるCopperシミュレーターを作成する必要がありました。

FCチームは8254-PITおよび8259-PIC PCハードウェアチップセットを使用して銅線をシミュレーションしました。彼女は、垂直後方ビームの3か所で手順を開始できる、VGA周波数と同期したシステムを作成しました。

  • 配置0：ディスプレイをオンにした後（約スキャンライン25）
  • 配置1：逆ビームスイープの直後（ITを回避できる可能性があります）
  • 配置2：スキャンビームの逆ビーム

これがどのように行われるかを読むことができますMAIN/COPPER.ASM（そして下の図を見てください）：

  1. 8254チップのタイマーは、目的の周波数でIRQ0をトリガーするように構成されています。
  2. 8h割り込みハンドラー（IRQ0の受信後に8259 PICによって呼び出されます）は、ここで手順に置き換えられますintti8。

注：DISフレームカウントサービスは、実際にはカッパーシミュレーターによって提供されます。

パート4：開発モードと製品モード


セカンドリアリティのソースコードを読むと、チームがDEVからPRODへのシームレスな切り替えにどれだけ注意を払ったかに最も驚かされます。

開発モード



開発モードでは、デモの各コンポーネントは個別の実行可能ファイルでした。

  • DISは常駐TSRにロードされ、割り込みを通じてアクセスされました0cfh。
  • ブートローダーにより、DOS割り込みにより21hファイルのオープン、読み取り、検索、クローズが行われました。

このDEV構成には、次の利点があります。

  • 各コーダーとアーティストは、チームの他のメンバーに影響を与えることなく、実行可能ファイルを操作して個別にテストできます。
  • いつでも完全なデモをテストできますSECOND.EXE（最後にすべてのEXEを追加せずに）。各PARTの実行可能ファイルは、DOS割り込みを使用021hして個別のファイルからロードされました。

プロダクション（デモモード）



プロダクションモードでは、小さいSECOND.EXE（ブートローダーを含む）、DIS、および個別のEXEとしてのデモの一部が1つの厚いものに結合されましたSECOND.EXE。

  • DISへのアクセスはまだ割り込みを介して行われました0fch。
  • DOS 21h割り込みAPIは、大きなファイルの最後からファイルを開く独自のFuture Crewルーチンによってパッチが適用されましたSECOND.EXE。

このPROD構成には、ロード時間とリバースエンジニアリングに対する保護の点で利点がありますが、最も重要なのは、プログラミングまたはPARTのロードの観点から、DEVからPRODに切り替えても変更がないことです。

パート5：別のパート


セカンドリアリティの各視覚効果は、完全に機能するDOS実行可能ファイルです。それらはPARTと呼ばれ、そのすべてが23です。このようなアーキテクチャソリューションにより、迅速なプロトタイピング、並行開発が可能になり（FCにはバージョン管理ツールがない可能性が高いため）、言語を自由に選択できます（ASM、C、さらにPascalもソースに含まれています）。

資産


画像アセット（.LBM）は、90年代に非常に人気のあるビットマップエディタであるDeluxe Paintを使用して生成されます。興味深いことに、それらはバイトの配列に変換され、PART内でコンパイルされます。この結果、exeファイルはすべてのアセットもダウンロードします。さらに、これはリバースエンジニアリングを複雑にします。

クールなアセットセットには、最新の3Dシーンの有名なCITYとSHIPがあります。



内部ユニットPART


それらはすべてDOS実行可能ファイルにコンパイルされているため、PARTでは、任意の言語を使用できます。

  • ASM。
  • C.
  • パスカル：ブラスナックルモンスター（MNTSCRL.EXE）、ドットトンネル（TUNNELI.EXE）、レイトレーシングソード（RAYSCRL.EXE）。
  • 一部のパーツはさらに進んで、Cを使用してASMコードを生成します：GLENZ / DOLOOP.C（ダウンロード可能な多面体）と水の効果（3DSINFLD.EXE）。

メモリ使用量については、ウィキペディアや他のWebサイトでMMUについてよく読んでいますが、実際には、実行後にメモリから完全にアンロードされるため、各部分は何でも使用できます。

VGAで作業する場合、各パーツは独自の一連のトリックを使用し、その解像度で作業しました。それらのすべてで、Mode 13hとModeXではなく、独自の解像度を持つ変更されたモード13hモードが使用されました。SCRIPTファイルでは、320x200と320x400がよく言及されています。

残念ながら、PARTを分析する場合、ソースコードの読み取りは困難な作業になります。コードとコメントの品質が劇的に低下します。おそらく、これは急いで発生したか、各PARTが独自の開発者で作業したために発生した（つまり、コメントやコードの理解に対する「実際の」必要性はなかった）が、結果は完全に混乱したものになった。

  • パニック
  • グレンツ

高度なアルゴリズムだけではなく、変数の名前すらを理解することは困難です（a、b、co[]...）。開発者がリリースノートにヒントを残しておけば、コードははるかに読みやすくなります。その結果、私は各部分を勉強するのに多くの時間を費やしませんでした。例外は、U2A.EXEおよびU2E.EXEを担当する3Dエンジンでした。

3DエンジンSecond Reality




とにかく、私は2つの部分で使用される3Dエンジンを詳細に検討することにしました：U2A.EXEとU2E.EXE。

ソースコードはCで、アセンブラーに最適化されたプロシージャ（特にGouroの塗りつぶしと陰影）を備えています。

  • CITY.C（メインコード）。
  • VISU.C（ライブラリvisu.lib）。
  • AVID.ASM（最適化されたアセンブラービデオ（クリーニング、画面コピーなど））。
  • ADRAW.ASM（オブジェクトの描画と切り捨て）。
  • ACALC.ASM（行列および高速sin / cos計算）。


これらのコンポーネントのアーキテクチャは非常に注目に値します。ライブラリVISUは、3DSオブジェクト、マテリアル、フロー（カメラと船の動き）などのアセットのロードなど、すべての複雑なタスクを実行します。

エンジンは、描画する必要があるオブジェクトを並べ替え、アーティストのアルゴリズムを使用してレンダリングします。これにより大量の再描画が行われますが、VGAラッチを使用すると4ピクセルを同時に記録できるため、それほど悪くはありません。

興味深い事実：エンジンは「古い」方法で変換を実行します。一般的な同種の4x4行列を使用する代わりに、3 * 3回転行列と変位ベクトルを使用します。
