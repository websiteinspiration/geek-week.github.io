プログラマーコミュニティでのグローバル変数の使用に関するコンセンサスはありません。これは個人の好みやスタイルの問題であるため、このセクションのタイトルにある質問に正解はありません。それにもかかわらず、多くのプログラマーは、多くのプログラミング教師が提唱するグローバル変数の完全な禁止は、不必要に厳しいものであると信じています。このセクションでは、R。構造のコンテキストでグローバル変数の考えられる利点を調べます。「グローバル変数」という用語は、対象のコードのレベルより上の環境階層にあるすべての変数を意味します。

Rでのグローバル変数の使用は、予想よりも一般的です。驚いたことに、Rはその内部実装（CコードとR関数の両方）で非常に広くグローバル変数を使用しています。したがって、スーパー代入演算子<<-は、Rの多くのライブラリ関数で使用されます（ただし、通常は、変数の階層で1レベルだけ高い変数に書き込むために使用されます）。高速プログラムの作成に使用されるマルチスレッドコードとGPUコード（第16章を参照）は、通常、並列実行プログラム間の相互作用の主要なメカニズムを提供するグローバル変数を使用します。

ここで、具体的に説明するために、セクション7.7の以前の例に戻りましょう。

f <- function(lxxyy) { # lxxyy — ,  x  y
    ...
    lxxyy$x <- ...
    lxxyy$y <- ...
    return(lxxyy)
}
#  x  y
lxy$x <- ...
lxy$y <- ...
lxy <- f(lxy)
#   x  y
... <- lxy$x
... <- lxy$y

前述のように、特にxとy自体がリストである場合、このコードは扱いにくくなる可能性があります。

一方、グローバル変数を使用した別のスキームを見てみましょう：

f <- function() {
     ...
     x <<- ...
     y <<- ...
}

#  x  y
x <-...
y <-...
f() #  x  y 
#   x  y
... <- x
... <- y

おそらく、2番目のバージョンはよりクリーンでかさばらず、リストの操作を必要としません。明確なコードは、通常、書き込み、デバッグ、およびメンテナンスで発生する問題が少なくなります。

これらの理由から-コードを単純化してかさばらないようにするために-前に示したDESコードでリストを返す代わりに、グローバル変数を使用することにしました。この例をさらに詳しく検討します。

2つのグローバル変数が使用されました（両方とも異なる情報を含むリストです）。sim変数はライブラリコードに関連付けられ、mm1glbls変数は特定のアプリケーションコードM / M / 1に関連付けられています。シムから始めましょう。

グローバル変数について予約されているプログラマーでさえ、そのような変数の使用は、それらがプログラムで広く使用されているという意味で、本当にグローバルである場合に正当化できることに同意します。これはすべて、DESの例のsim変数に関連しています。これは、ライブラリコード（schedevnt（）、getnextevnt（）およびdosim（））とM / M / 1コード（mm1reactevnt（））の両方で使用されます。この特定の例では、後続のsimの呼び出しは読み取りに限定されていますが、状況によっては記録が可能です。この種の典型的な例は、イベントキャンセルの可能な実装です。たとえば、「2つのうちの早いもの」の原則をモデル化すると、このような状況が発生する可能性があります。

したがって、グローバル変数としてsimを使用することは正当化されるようです。ただし、グローバル変数の使用を断固として拒否した場合、simはdosim（）内のローカル変数に配置できます。この関数は、前の段落で述べたすべての関数（schedevnt（）、getnextevnt（）など）の引数にsimを渡し、これらの各関数は変更されたsim変数を返します。

（たとえば）appvarsという名前のローカル変数をdosim（）に含めることで、mm1glblsで同様のことができます。しかし、これが2つの変数で行われる場合、上記のf（）関数の例のように、両方の変数が関数から返されるように、それらをリストに追加する必要があります。そして、上記のリスト内のリスト、またはリスト内のリスト内のリストの扱いにくい構造があります。

一方、グローバル変数を使用する反対者は、コードの単純さは無駄ではないことに気づきます。変更はプログラムのどこでも発生する可能性があるため、デバッグプロセス中にグローバル変数が値を変更する場所を見つけるのが困難であると彼らは心配しています。変数のすべての出現箇所を見つけるのに役立つ最新のテキストエディターと統合開発ツールの世界では、問題は途中で起こっているようです（グローバル変数の使用をやめるように促す元の記事は1970年に公開されました！）。それにもかかわらず、この要素を考慮に入れる必要があります。

批評家が言及している別の問題は、異なる値を持つプログラムのいくつかの無関係な部分から関数を呼び出すときに発生します。たとえば、関数f（）がプログラムのさまざまな部分から呼び出され、それぞれの呼び出しが1つの値ではなく独自のx値とy値を受け取ることを想像してください。この問題は、プログラム内のf（）の各インスタンスが個別の要素を持つx値とy値のベクトルを作成することで解決できます。ただし、これにより、グローバル変数の使用の単純さが失われます。

これらの問題は、Rだけでなく、より一般的な状況でも発生します。ただし、Rでは、ユーザーが通常このレベルで多くの変数を持っているため、上位レベルでグローバル変数を使用すると、さらに問題が発生します。グローバル変数を使用するコードが、完全に無関係な変数を同じ名前で誤って置き換える可能性があるという危険があります。

もちろん、問題は簡単に解決されます。特定のアプリケーションに関連付けられているグローバル変数の長い名前を選択するだけで十分です。ただし、環境も、DESの例の次の状況のように、妥当な妥協点を提供します。

7.8.5。短絡


Rのクロージャは、呼び出し時の環境に関連した引数と関数の本体で構成されていることを思い出してください。環境が含まれているという事実は、クロージャとも呼ばれる概念を使用するプログラミングパラダイムに関係しています（ここでは、用語のオーバーロードがいくつかあります）。

クロージャーは、ローカル変数を作成してから、この変数にアクセスする別の関数を作成する関数です。説明が抽象的すぎるので、例を挙げたほうがよいでしょう。

1 > counter
2 function () {
3      ctr <- 0
4      f <- function() {
5          ctr <<- ctr + 1
6          cat("this count currently has value",ctr,"\n")
7      }
8      return(f)
9 }

実装の詳細に入る前に、このコードがどのように機能するかを確認しましょう。

> c1 <- counter()
> c2 <- counter()
> c1
function() {
        ctr <<- ctr + 1
        cat("this count currently has value",ctr,"\n")
     }
<environment: 0x8d445c0>
> c2
function() {
        ctr <<- ctr + 1
        cat("this count currently has value",ctr,"\n")
     }
<environment: 0x8d447d4>
> c1()
this count currently has value 1
> c1()
this count currently has value 2
> c2()
this count currently has value 1
> c2()
this count currently has value 2
> c2()
this count currently has value 3
> c1()
this count currently has value 3

ここでは、counter（）関数が2回呼び出され、結果にc1とc2が割り当てられます。予想通り、これら2つの変数は関数、つまりf（）のコピーで構成されています。ただし、f（）は、スーパー割り当て演算子を介してctr変数にアクセスします。この変数は、環境階層のパスの最初になるため、counter（）にローカルで指定された名前の変数になります。これは環境f（）の一部であり、counter（）の呼び出し側に戻るものにパッケージ化されています。重要な点は、counter（）への呼び出しが異なると、ctr変数が異なる環境にあることです（サンプル環境では、アドレス0x8d445c0および0x8d447d4のメモリに格納されていました）。つまり、counter（）への異なる呼び出しは、ctrの物理的に異なるインスタンスを作成します。

その結果、関数c1（）およびc2（）は、完全に独立したカウンターとして機能します。これは、各関数が複数回呼び出される例からわかります。
