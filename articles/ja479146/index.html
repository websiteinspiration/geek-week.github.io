バイパスブロックツールの比較\ VPN

ネットワーク上のさまざまなリソースへのアクセスをますます積極的にブロックするにつれて、ロックのバイパスの問題はますます関連性が高まっています。つまり、「ロックをバイパスする方が速いのはなぜですか」という質問です。

DPI \ホワイトリスト\ブラックリストを回避するという観点から、効率性のトピックを残して、一般的なロックバイパスツールのパフォーマンスを比較します。

注意：ネタバレの下の記事にはたくさんの写真があります。

免責事項：この記事では、「理想的な」条件に近い条件で、人気のあるVPN /プロキシソリューションのパフォーマンスを比較します。ここで取得して説明した結果は、必ずしもフィールドでの結果と一致しません。多くの場合、速度テストの数値は、バイパスツールの生産性ではなく、プロバイダーが\ QoSit \を調整する方法に依存するためです。

方法論


クラウドプロバイダー（DO）が世界のさまざまな国で3 VPSを購入しました。オランダでは2つ、ドイツでは1つです。最も生産性の高いIPN（コアの数による）は、クーポンローンを提供するアカウントで利用可能なIPNから選択されました。

プライベートiperf3サーバーが最初のオランダ語サーバーに展開されます。

2番目のオランダのサーバーでは、バイパスツールをロックするためのさまざまなサーバーが順番に展開されます。

ドイツのUPUでは、VNCと仮想デスクトップを備えたデスクトップLinux（xubuntu）のイメージを展開しました。このVPSは条件付きクライアントであり、さまざまなプロキシ\ VPNクライアントが1つずつ配置/起動されます。

速度測定は3回行われ、平均に焦点を合わせ、3つのツールを使用しています。 fast.comを介してクロムで;コンソールからiperf3経由でプロキシ4を介して（iperf3トラフィックをプロキシに入れる必要がある場合）。

直接クライアント/サーバー接続iperf3は、iperf3では2 Gbpsの速度を示し、高速/速度テストでは少し遅くなります。

好奇心旺盛な読者は、「なぜspeedtest-cliを選ばなかったのですか？」そして正しいでしょう。

スピードテストテストは、スループットを測定するための信頼できない/不十分な方法であることがわかりました。3回連続して測定すると、3つのまったく異なる結果が得られる可能性があります。たとえば、スループットがIPSのポート速度よりはるかに高い場合があります。たぶん問題は私の目が細いのかもしれませんが、そのような装置で研究を行うことは不可能に思えました。

3つの測定方法（速度テスト\高速\ iperf）の結果に関して、私はiperfインジケーターが最も正確で信頼性の高い\信頼性の高いものであり、高速\速度テスト-参照であると考えています。しかし、一部のバイパスツールでは、iperf3を介して3つの測定を完了することができませんでした。そのような場合は、speedtest / fastに集中できます。

合計で、24の異なる回避策ツールまたはその組み合わせ、あるいはその両方をテストしました。それぞれについて、簡単な説明と、それらを使用した印象を説明します。しかし、本質的に、目標はシャドウソック（およびさまざまな難読化ツールのヒープ）の速度をopenVPNとワイヤーガードで比較することでした。

v2rayプラグイン


V2rayプラグインは、ss libの主要な「公式」難読化ツールとして単純なobfsを置き換えました。単純なobfsとは異なり、それはまだリポジトリにありません。ビルド済みのバイナリをダウンロードするか、自分でコンパイルする必要があります。

3つの動作モードをサポートしています。デフォルトのhttp-webソケット（宛先ホストのヘッダーのスプーフィングをサポート）。tls-webソケット（c-obfsとは異なり、これは本格的なtlsトラフィックであり、任意のWebサーバー\リバースプロキシによって認識され、たとえば、cloudfleurサーバーまたはnginxでtls終了を構成できます）; quic-udpで機能しますが、残念ながらv2のQuickのパフォーマンスは非常に低くなっています。

単純なobfsと比較した利点：v2rayプラグインは、トラフィックのあるhttps-webソケットモードのCFを介して問題なく機能します。tls-modeでは、本格的なtls-trafficを表し、証明書を必要とします（たとえば、暗号化または自己署名から）。

shadowsocks \ v2ray-plugin-http：ダウンロード-1404 mbit \ s; 1938 mbit \ sをアップロードします。


shadowsocks \ v2ray-plugin-tls：ダウンロード-1214 mbit \ s; 1898 mbit \ sをアップロードします。


shadowsocks \ v2ray-plugin-quic：ダウンロード-183 mbit \ s; 384メガビット\ sをアップロードします。


すでに述べたように、v2reiはヘッダーを設定できるため、リバースプロキシ\ CDN（たとえばcloudfleur）を使用してヘッダーを操作できます。一方では、これによりトンネルの検出が複雑になります。他方では、遅延がわずかに増加する（場合によっては減少する）可能性があります。すべて、ユーザーとサーバーの場所によって異なります。現在、CFはquicで作業をテストしていますが、これまでのところ、このモードは利用できません（少なくとも無料のアカウントの場合）。

shadowsocks \ v2ray-plugin-http + CF：ダウンロード-1284 mbit \ s; 1785 mbitをアップロードします。


shadowsocks \ v2ray-plugin-tls + CF：ダウンロード-1261 mbit \ s; 1881 mbit \ sをアップロードします。

開発者によると、プラグインの最初のバージョンでは、tls 1.2再開セッションメカニズムを使用して、tlsの宛先アドレスを偽装していました。新しいバージョン（clok-2）のリリース後、このメカニズムを説明するgithub上のすべてのwikiページが削除されました;暗号化/難読化の現在の説明ではこれについての言及はありません。著者の説明によると、「暗号の重大な脆弱性」が存在するため、シュレッドの最初のバージョンは使用されていません。テストの時点では、セスプールの最初のバージョンしかありませんでした。そのバイナリはまだgithubにあり、とりわけ、重大な脆弱性はそれほど重要ではありません。shadousoksは、シュレッドなしの場合と同じ方法でトラフィックを暗号化し、crypto shadowsoksはcrypto shadowsoksaに影響を与えません。
