oEmbedを使用してプログラムで一般的なコードを検索する

全てにおいて良い日！Drew McLellanによる記事「 " プログラムによる共有コードのoEmbedによる発見 "の翻訳を共有したいと思います。

前書き


多くのサイトは、いくつかのHTML埋め込みコードを使用して別のプラットフォームで共有できるマルチメディアとコンテンツをホストしています。アイテムのURLのみがあり、人が介入することなくマルチメディアの埋め込みバージョンを見つける必要がある場合はどうなりますか？ここでoEmbedが役に立ちます。

ネットワークには、ビデオ、画像、音楽、ポッドキャスト、マップ、グラフィックスなどのさまざまなコンテンツや、あらゆる種類のエンターテインメントを含むサービスが満載です。ほとんどの場合、コンテンツをサイトに追加すると、このコンテンツをWebページのどこかに埋め込む方法が提供されます。

YouTubeなどのサイトには独自の埋め込みプレーヤーがあり、ブログ投稿や製品ページでよく使用されています。 Soundcloudには、音楽プレーヤーをバンドのWebサイトに埋め込むためのコードがあります。慈善団体の募金活動者は、Stravaのようなサイトに大きなレースルートをアップロードし、それを資金調達サイトで共有してスポンサーに表示することができます。

これはすべて、ホスティングサイトで「共有」オプションを見つけて、通常HTMLとJavaScriptの混合であるコードをコピーすることで実行できます。このコードは通常、ランディングページに貼り付けることができ、ホスティングサイトは、すべての友達、顧客、連絡先に表示するコンテンツの豊富なプレゼンテーションを提供します。

これまでのところ、非常に優れており、コンテンツを手動で埋め込むのに適しています。ただし、結果は同じですが、そのパスが完全に異なる別のユースケースがあります。

プログラム交換


ユーザーからコンテンツを受け取るアプリケーションまたはWebサイトを作成しているとしましょう。たとえば、従業員が同僚とニュースを共有するための基本的なWebページや、人々が登録してメッセージの投稿を開始できるソーシャルネットワーク全体など、非常にシンプルなものにすることができます。

どちらの場合も、ユーザーがこのコンテンツの一部としてURLを追加した場合の対処方法を決定する必要があります。あなたはシナリオを想像することができます：
このビデオをチェックしてください！youtu.be/jw7bRnFbwAI

現時点では、出版システムとして、何をすべきかを理解する必要があります。最初のオプションは、何もせず、URLをプレーンテキストのままにすることです。ユーザーは通常URLをクリックしたいので、これはあまりよくありません。プレーンテキストでは、相手側のページに移動できません。

2番目のオプションは、リンクに変換することです。ユーザーはリンクをたどってコンテンツにアクセスできるため、これは良い選択です。しかし、同時に彼らはあなたのサイトを去り、戻ってこないかもしれません。

最高のユーザーエクスペリエンスは、URLだけでなく、このコンテンツのプレーヤーを抽出してそこに直接貼り付ける機能である可能性があります。これにより、ユーザーはFacebookなどのサイトでコンテンツを直接表示できます。

これは問題を引き起こします。 URLを指定して、ほとんどのユーザーがページに表示するために必要なHTML / JavaScript埋め込みコードにどのように変換できますか？

YouTubeのような有名なサイトであれば、YouTube APIを使用してビデオ情報を抽出し、この方法で埋め込むコードを取得または作成するコードを書くことができます。 VimeoやVIVOなどの他のビデオサービスでも同じことができます。 FlickrとInstagramのURLを認識するコードを記述し、それらのAPIを使用して、埋め込み可能な優れたバージョンの写真を取得できます。ツイッターやツイートについても同様です。しかし、それは多くの時間と労力がかかります！

理想的には、コンテンツのURLを埋め込みコードブロックに渡して、そのコンテンツをページに表示する標準化された方法が必要です。注意を払えば、oEmbedがこれに適していることがわかります。

Origin OEmbed


これがまさに問題だったリア・カルバーは Pownceの（TwitterのVHSのためのベータマックスだった、真に革新的なソーシャル・ネットワーキング・サイト）での作業中に直面しました。Pownceは、ユーザー更新のストリームにリッチコンテンツ表現を埋め込むことを望んでいましたが、統合用のコードを具体的に記述したサービスだけにサポートを限定したくありませんでした。同僚のMike Maloneと、（当時、このようなコンテンツの最大のプロバイダーの1つであるFlickrでエンジニアリングを率いていた）Cal HendersonとRichard Crowleyとの夕食で、埋め込みコードを取得するためのオープンスタンダードのアイデアを考え出しました。URL ヘンダーソンは立ち去り、議論に基づいて何かを開発し、oEmbedが誕生しました。

OEmbedの使用


これがどのように機能するかです。

コンテンツの単一の要素を指すURLで始まります。これは、YouTubeのビデオ、画像、またはその他の可能性があります。これは通常、サイトまたはアプリケーションのユーザーが、公開したいコンテンツの一部として提供されます。最初のステップは、HTMLページであるこのURLのページコンテンツを取得することです。

コンテンツをホストしているサイトがoEmbedをサポートしている場合、このページのセクションにoembedコンテンツタイプのlink要素があるはずです。

<link rel="alternate" type="application/json+oembed"
  href="http://www.youtube.com/oembed?url=https%3A%2F%2Fyoutu.be%2Fjw7bRnFbwAI&format=json"
  title="Inclusive Components with Heydon Pickering" />

XMLの注記：oEmbedは、XML形式とJSON形式の両方の応答をサポートしています。この記事では野蛮人ではないので、私はJSONのみを考慮します。ただし、XMLを使用する必要がある場合は、oEmbed仕様でサポートされていることに注意してください。ただし、一部のプロバイダーはJSON応答しか提供しない場合があります。

このリンクタグはrel属性としてalternateに設定され、typeはapplication / json + oembedまたはtext / xml + oembedのいずれかに設定されます。この属性は、href URLで指定されたものが実際にはコンテンツに関する情報を取得するためのoEmbed APIエンドポイントであることを伝えます。

このURLには通常、urlとformatの2つのパラメーターがあります。


初期コンシューマ要求のための一般的なURLパラメータ

のフル仕様ここではさらに詳細に説明します（独自の実装を作成する場合は参照する必要があります）が、これらの2つのパラメーターは最もよく表示されます。

したがって、URLがあり、ページを選択し、APIエンドポイントの異なるURLを持つoEmbedリンクタグを見つけました。次に、この新しいURLを要求すると、このコンテンツについてサービスが提供する必要のあるすべての情報が返されます。

そして今、話しましょう！答えは私たちにコンテンツに関する多くの情報を提供します。近い将来、1.0になるはずです。これは、oEmbed仕様の現在のバージョンです。返されるその他の情報は、タイプのコストに大きく依存します。

回答の種類


応答のキーのタイプ値は、どのタイプのメディアを埋め込むかを示しています。

専用のビデオコンテンツに加えて、実際に目にすることができる最も一般的なタイプはリッチです。Flickr自体も、写真の応答を送信しながら、画像の埋め込み可能なクールな「プレーヤー」をhtmlに提供します。

ほとんどの場合、サイトにコンテンツを埋め込むことは、HTML値として提供されるコードを使用する場合にすぎません。

安全上の注意


1つの点に注意する必要があります。HTML応答を取得して、投稿するページにプログラムで埋め込む場合、貼り付けたコードを再確認する人がいないと、このコードが悪意のある可能性が常にあります。したがって、リスクを軽減するための適切な対策を講じる必要があります。

これには、スキームとドメインが期待どおりであることを確認するためのURLフィルタリング、およびCookieのない別のドメインのiframe内のサンドボックス化されたコードが含まれる場合があります。コードを使用している状況にアクセスし、過度のリスクにさらされていないことを確認する必要があります。

はじめましょう


oEmbedを使用するときのプロセスを理解することも同様に重要ですが、実際には、ほとんどの一般的な言語には、プロセスを抽象化して比較的単純にするライブラリーがあります。

たとえば、NPMパッケージoembedは、コンテンツURLに基​​づいて要求を作成し、oEmbedへの応答を受信するための非常にシンプルなインターフェースを提供します。

まず、プロジェクトにパッケージをインストールします。

npm i oembed

次に、URLを要求します。ここでは、NotistのプレゼンテーションURLを使用しています。

const oembed = require('oembed');
const url = 'https://noti.st/drewm/ZOFFfI';

oembed.fetch(url, { maxwidth: 1920 }, function(error, result) {
    if (error)
        console.error(error);
    else
        console.log("oEmbed result", result);
});

そして答えは：

{ 
        type: 'rich',
        version: '1.0',
        title: 'Understanding oEmbed',
          author_name: 'Drew McLellan',
        author_url: 'https://noti.st/drewm',
        provider_name: 'Notist',
        provider_url: 'https://noti.st',
        cache_age: 604800,
        thumbnail_url: 'https://on.notist.cloud/slides/deck4179/large-0.png',
        thumbnail_width: 1600,
        thumbnail_height: 900,
        html:
           '<p data-notist="drewm/ZOFFfI">View <a href="https://noti.st/drewm/ZOFFfI">Understanding oEmbed</a> on Notist.</p><script async src="https://on.notist.cloud/embed/002.js"></script>',
        width: 960,
        height: 540 
}

PHPで同じことをしたい場合は、embos / embedという便利なパッケージをComposerからインストールできます。

composer require embed/embed

そして、あなたのPHPプロジェクトで：

use Embed\Embed;

$info = Embed::create('https://noti.st/drewm/ZOFFfI');

$info->title; // "Understanding oEmbed"
$info->authorName; // "Drew McLellan
$info->code; // "<p data-notist="drewm/ZOFFfI"> ... </script>"

ご覧のとおり、ライブラリを使用すると、プロセスが非常に簡単になり、URLから埋め込みコードにすばやく切り替えて、ユーザーコンテンツの豊富な表現をすぐに表示できます。

結論


oEmbedは、特定の問題に対する非常に洗練されたソリューションです。大規模なソーシャルネットワークで作業するエンジニアは数人しかいないと思われるかもしれませんが、実際には、ユーザーがURLを入力できる公開システムは驚くほど一般的です。ある時点で、ある種のCMSを作成する必要がない内部エンジニアを1人見つけてください。同じ条件で考えることはできませんが、ユーザー入力を受け入れる場合は、この入力にURLが含まれている場合の対処方法を考える必要があります。

oEmbed（申し訳ありません）については理解できましたが、将来のプロジェクトでURLを処理する方法に真剣に注意を払わない言い訳はありません。
