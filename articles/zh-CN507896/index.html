造型Android应用程序和设计系统：如何做到并与彼此结交朋友









各位读者好！  



在某个时刻，任何大型应用程序都会增长，因此很难在任何地方维护相同的设计，并且很难动态响应设计和UX需求的任何变化和趋势。  



因此，我们决定在我们的应用程序中引入一个设计系统，并增加对几个主题的支持。 



研究了各种方法后，我们开发了解决此类问题的方法。我想使设计系统和样式支持在其他项目中可重用。根据这一想法，开发了组件和主题。



设计系统组件



设计系统及其组件旨在在整个应用程序中统一设计和样式统一。



在我们的案例中，我们将称呼设计系统的自定义视图组件具有适应多种应用程序样式的能力。组件可以应用于应用程序中的任何位置（按钮，列表项，标题等）。



设计，调试和完善设计系统的组件



设计系统组件由设计师订购。在第一阶段，我们与他们协调对元素（重用评估）及其功能的需求。 



协调之后，应该清楚应该在自定义视图属性中放置哪些选项（文本颜色，文本，图标，图标色调颜色等），以及将哪些选项隐藏在外部更改之外（这可以防止开发人员对元素的不当使用）。



接下来，设计人员在其环境中绘制组件并放弃以进行开发。 



在实现组件时，您需要添加对主题（浅色或深色主题等）的支持，下面我将描述该组件如何支持多个主题。



最佳实践



  • 创建包含设计系统组件的模块。从积极的方面来看：可以在其他应用程序中使用单独的模块，并且模块化允许您快速导航。
  • 创建具有设计系统组件的测试应用程序。这样可以加快开发和调试速度。


在应用程序中嵌入主题的方法



我知道两种在Android中支持样式的方法：



  • 软件（软件重绘）。
  • Android中的标准样式机制。


软件方式



   我们在运行时重新绘制整个视图层次结构。我们递归地研究它，并根据从一个主题到另一个主题的某些过渡规则，重新绘制组件。那些不应该重新粉刷的标记为android:tag



或android:contentDescription



。解析屏幕层次结构时不考虑这些组件。



    在显示屏幕之前（例如，在onStart()



y中Activity



）以及使用它时，都可以重新绘制。      



缺点



  • 需要更多资源，降低性能。在初始化所有组件之后应用样式化。
  • 您需要注意从一个主题转移到另一个主题的规则。重新绘制需要很多规则，您可能会忘记一些东西。结果是switch — case



    （Java）或when



    （Kotlin）的一长篇论文。在附件中，您需要考虑不需要使用上述标记绘制的元素。
  • 您不能根据主题部分重画。任何规则都有例外，并非应用程序中的所有内容总是根据设计系统完成。如果需要对某些元素进行部分重新粉刷，则不清楚如何进行。



样式的应用归结为特定元素更改的描述： 



if (view is TextView) {
    (view as TextView).setTextColor(
        if (darkMode) R.color.blue else R.color.black
    )
} else if (view is TabLayout) {
  (view as TabLayout).doAnything()
}


优点



不需要重新创建Activity（这很重要！更改线程时不闪烁）。

我在每个人都知道的产品中实现了这种方法（请参见屏幕截图）。使用相同类型的简单布局，它可以非常快速地工作（在这种情况下，它很简单）。
