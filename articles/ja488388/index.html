再ログインとHTTP基本認証

Web開発者は長い間、HTTP Basic Authorizationで保護されたサイトへのログインとログインの問題を認識してきました。この問題の影響を受けない認証方法は他にもありますが、基本認証は依然として多くの場合最良の選択です。ネットワークには、さまざまな一般的および特定のソリューションを説明する十分な資料があります。しかし、私が見つけたそれらすべては、残念ながら、あるブラウザーでは機能し、別のブラウザーでは機能しない一部のソリューションのみを説明しています。猫の下で、私はこの問題に関する私の研究の一般化された最終結果を与えます

私はフロントエンドの開発者ではなく、さまざまな「ブランド」やブラウザのバージョンを詳しく調べなかったので、すぐに予約します。執筆時点で関連するバージョンを持つ主流のみ。ほとんどのWebタスクではこれで十分です。ブラウザの「動物園」全体をサポートする必要がある開発者にとって、この記事は出発点として

最適であり、問​​題の本質は、HTTP Basic Authorization標準がロギングを提供していないことです。一般的に。基本認証で保護されたページにアクセスすると、ブラウザ自体がログイン/パスワードのリクエストを表示する独自のウィンドウを表示し、ログインが成功すると、それらをその奥のどこかに保存します。次に、このサイトの他の保護されたページへの後続のすべてのリクエストは、Authorizationヘッダーのこれらの値を自動的に使用します。
承認：基本的なbm9uZTpub25l
ここで、bm9uZTpub25lは、base64でエンコードされたログイン/パスワードのリンクです。なし：なし（ユーザー名とパスワードは異なる場合があります）

ログインするには、ブラウザの奥深くで古いユーザー名/パスワードをリセットする必要があります新しいものを記録する場所。ここで驚きが待っています。このアクションを規制する標準はないため、各ブラウザは独自の理解に基づいてこれを行います。

さらに悪いことに、ログイン/パスワード要求ウィンドウは、すでにHTTPページヘッダーのみに応答する独自のブラウザーウィンドウであるため、このウィンドウは、ページ本文の処理の前にポップアップします。つまり、ステータスが401のサーバー応答を受信したときに一部のJavaScriptを実行しても機能しません。ユーザーの前に、このウィンドウは何度も何度もポップアップし、ログイン/パスワードのリクエストが繰り返されます。

実際、この点は重要です。ほとんどすべてのブラウザーでログアウトする唯一の方法は、明らかに間違ったログイン/パスワードを使用してサーバーにリクエストを送信し、ステータス401のサーバー応答を受信して​​から、ブラウザーのログイン/パスワードキャッシュをリセットすることです。ただし、同時にブラウザが応答401の処理を許可せず、新しいユーザーのログインプロセスを既に続行している場合、HTTPヘッダーを読み取り、入力した認証のフォームをスローする段階でも制御を行うと、これはあなたの顔では機能しません。すでに問題があります。これが参照による通常のリクエストであるか、XMLHttpRequestまたはフェッチによるものであるかは問題ではありません。ヘッダーを解析する段階で、ブラウザーは引き続き制御を取得します。

Google Chrome（79.0.3945.88）


Chromeの場合、fetchメソッドは適切に機能します。しかし、XMLHttpRequestは機能しません（（（キャッシュはリセットされず、ロギングは発生しません。さらに、openメソッドのパラメーターとして、およびヘッダーの設定として、ログイン/パスワードを設定しようとしました。

else if (/chrome/i.test(window.navigator.userAgent)) {
    fetch(url401, {
      credentials: 'include',
      headers: {
        'Authorization': 'Basic ' + btoa('none:none'),
        'X-Requested-With': 'XMLHttpRequest'
      }
    }).then(() => {
      window.location.assign(url);
    }).catch(err => console.error(err));
}

明らかに正しくないユーザー名/パスワードを使用してページ401へのフェッチ要求を行い、サーバーから401応答を受け取り、ブラウザがキャッシュをフラッシュします。

重要！サーバーはWWW-Authenticateヘッダーを返すべきではありません。それ以外の場合は、ブラウザが制御を取得し、ページ401からのリダイレクトは発生しません。慣例として、リクエストでX-Requested-With：XMLHttpRequestが指定されている場合、サーバーはこのヘッダーを返しません。そのため、X-Requested-Withヘッダーがリクエストに追加されました。
