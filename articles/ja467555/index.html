CSSをどれだけ知っていますか？（+ミニテスト）



CSSの使用の成功と、CSSに対処するための苦痛な試みの違いは、多くの場合、細部に依存します。実際、CSSには多くのニュアンスがあります。私がそのような苦労にしばしば気づく最も一般的な領域の1つは、レイアウトのスタイリングです。個人的には、CSSパターンを学ぶのが好きです。レイアウトに関するほとんどの問題を解決するために、それらを少数使用する傾向があることに気付きました。この記事では、レイアウトの問題を克服するために使用するCSSパターンについて説明します。SMACSS、BEM、またはホットなCSS-in-JSトピックなど、使用されるCSS手法に関係なく、状況は考慮されます。これらはすべて、アーキテクチャ、組織、または戦略ではなく、CSSプロパティ自体に焦点を当てているためです。

質問1：ブロックモデル


ブロックモデルの学習は、フロントエンド開発者のリストの優先事項です。おそらく「CSSボックスモデル」という記事は少し古く、最新のCSSに対するその価値と関連性を過小評価しないでください。ブロックモデルは、CSSレイアウトに関連するほとんどすべての問題の基礎となります。

この特定の質問では、ブロックモデルの原則に基づいて幅を見つける方法を確認します。ブロックにはプロパティによって明示的な幅width: 100pxがありますが、ブロックモデルのルールは、デフォルトでwidthは、このブロックの「コンテンツ」レイヤーにプロパティを適用することがわかります。結果の幅（ブロックがページに描画される幅）は、コンテンツ、パディング、およびボーダーレイヤーの合計です。このため、答えは112pxです。

.box {
  width: 100px; /* Take this */
  height: 50px;
  padding: 5px; /* Plus this x2 for left and right */
  border: 1px solid red; /* Plus this x2 for left and right */
  background-color: red;
  /* = 112px of computed width */
}

すべての人に示しwidth: 20%;、親要素の100％に収まると確信しているにもかかわらず、インターフェイスの最後の列またはタブが次の行に移動する状況に直面している場合、これはおそらく問題になりました。幅20％の5つの列は100％に配置されますが、それらにパディングやボーダーが追加されると、それぞれの幅が増加し、現在の行に最後の列のための十分なスペースがなくなります。

CSS3が導入されたとき、と呼ばれる新しいツールbox-sizing。これにより、プロパティを適用するブロックモデルのレイヤーを制御できますwidth。たとえば、次のように指定できますbox-sizing: border-box。つまり、「コンテンツ」レイヤーではなく、外側の「境界」レイヤーに幅ルールを適用する必要があります。テストの質問では、プロパティが適用されている場合、box-sizing: border-boxブロックの合計幅は100pxでした。

一部の人にとって、これはよく知られている事実ですが、それでも専門家と初心者の両方にとって良い思い出となるでしょう。

ブロックモデルと、box-sizingプロパティをリセットとして使用してプロジェクト全体にすぐに適用する方法については、数多くの記事があります。Box SizingとInheriting box-sizingおそらくわずかに優れたベストプラクティスは、CSSトリックに関する2つの優れた記事です。

3: — absolute fixed


それぞれのタイプをいつ使用するか、視覚表現でどのように異なるかを理解することに加えて、上、右、下、または左のプロパティを使用して、各配置方法が親要素にどのようにバインドされるかのルールを知ることも非常に重要です。

まず、包含ブロックを見てみましょう。簡単な定義は、包含ブロックがほとんどの場合、問題の要素の親であることです。ただし、包含ブロックのルールは、絶対位置と固定位置の要素では異なります。

1）絶対要素の場合。包含ブロックは、位置がでない最も近い祖先ですstatic。たとえば、要素が絶対的に配置され、プロパティが含まれているtop場合、right、bottomそれともleft彼らは配置が設定されている任意の親に対して配置されるabsolute、relative、fixedまたはsticky。
2）固定要素の場合。以外の位置にある親要素があるにもかかわらず、包含ブロックはビューポートですstatic。さらに、スクロール動作は絶対配置された要素とは異なります。固定要素は表示領域で「固定」されたままなので、名前が付けられます。

多くの開発者は、絶対配置された要素は相対配置で最も近い親要素のみを探すと考えていposition: relativeます。この誤解は、それがposition: relative最も頻繁にposition: absolute包含ブロックを作成します。これがよく使用される理由は、親ブロックの相対的な配置がそれをストリームに残すためです。絶対配置要素の親自体が絶対配置される場合があります。これは完全に正常です。すべての親が静的である場合、絶対的に配置された要素がビューポートにアタッチされますが、ビューポートとともにスクロールします。


上記のルールにはあま​​り知られていない追加があります。親要素がtransform （特に）以外の値を持つプロパティを持っている場合none、それは絶対的かつ固定的に配置された要素の包含ブロックになります。これの確認は、CodePenで確認できます。ここでは、「Notice！」というテキストを含む要素 固定要素であり、親には変換プロパティがありますが、マウスカーソルがその上にあるときのみ（状態:hover）


質問4：親要素と子要素の折りたたみマージン


これは、CSSがどのように機能するかわからない場合に多くの問題を引き起こす可能性があるCSSの小さなものの1つです。折りたたみマージンと呼ばれる概念があり、多くの人は、隣接する兄弟の折りたたみマージンと呼ばれる、この現象の兆候に精通しています。ただし、あまり知られていない、親と最初/最後の子の折りたたみマージンと呼ばれる別のフォームがあります。以下は、両方のケースのデモです。


各段落タグには、1emの上下のマージンがあり、ブラウザーによってスタイルが設定されています。これは、これまでのところ最も簡単な状況です。しかし、なぜ段落間の間隔が2em（上と下の合計）ではないのですか？これは、隣接する兄弟のマージンの折りたたみと呼ばれます。マージンは、2つのマージンのうち大きい方がギャップのサイズになるようにオーバーラップするため、この場合、ギャップは1emになります。

しかし、何か他の奇妙なことが起こっています。最初の段落の上マージンは、それと青いコンテナーdivの間にギャップを作成しないことに気づきましたか？分割するのではなく、divに上部マージンがあるかのように、親divにマージンを埋め込むように見えます。これは、親と最初/最後の子の折りたたみマージンと呼ばれます。このタイプのマージンの崩壊は、以下が親の特性である場合、特定の状況では発生しません。

  • ゼロより大きい上/下パディングがあります
  • ゼロより大きい上/下の境界があります
  • overflow: hiddenまたはを使用して作成できるブロックフォーマットコンテキストが定義されていますoverflow: auto
  • 表示：フロールートプロパティセット（貧弱なブラウザーサポート）

この小さなCSSの詳細を人々に説明し、パディングまたは境界線でそれを解決できると喜んでいるとき、答えはほとんど常に「パディングまたは境界線が0の場合はどうなりますか？」です。まあ、値は正の整数でなければならないので、これは機能しません。

前の例では、1pxのパディングにより、折りたたみの使用と親と子の折りたたみマージンの切り替えを切り替えることができます。最初/最後の段落と親の間に表示されるギャップは1pxのパディングですが、パディングレイヤーがマージンの崩壊を防ぐバリアを作成するため、マージンはコンテナの内側と見なされます。

質問については、このインターフェイスで問題が何であるかを確認できるはずです。


クラスのある最初の要素.comment（クラスなし.moderator）はマージンを縮小します。コードを見なくても、クラスを.moderator持つ要素にはフレームがあり、このクラスがない要素にはフレームがないことがわかります。この質問には実際に正しいと考えられる3つの回答がありましたが、実際にはそれぞれが既にCodePenソースに適用されており、コメント化されています。

このタイプの崩壊マージンが他のタイプほど広く知られていない理由の1つは、これを誤って回避できる状況が数多くあるためです。フレックスボックスとグリッドの要素は、ブロックのフォーマットコンテキストを作成するため、これらを使用すると、このタイプの折りたたみマージンは発生しません。コメントのインターフェースが実際のプロジェクトである場合、四隅すべてに余白を残すためにパディングが設定されている可能性が高く、これによりマージンの崩壊が修正されました。

質問5：何の割合ですか？


パーセンテージの単位が使用される場合、パーセンテージはそれを含むブロック（通常は親）の幅または高さに基づきます。前に述べたように、プロパティtransform を持つ要素は包含ブロックになるため、要素が変換を使用する場合、単位はパーセントで（の場合のみtransform） 、親のサイズではなく、要素自体のサイズになります。

以下の例では、50％がコンテキストに応じて2つの異なる値をとることがわかります。最初の赤いブロックにはプロパティmargin-left: 50%があり、2番目の赤いブロックはを使用しtransform: translateX(50%)ます。


質問6：ブロックモデルが再びストライキします...


ブロックモデルについて話し終えたと思ったのはあなただけです...


問題は、width: 100%フッターを使用すると同時にパディングを追加するためです。コンテナの幅は500pxです。つまり、このサイズにパディングが追加される前のフッターコンテンツレイヤー（100％）も500pxです。

この問題は、次の2つの一般的な方法のいずれかで修正できます。

  1. フッターのボックスサイズプロパティを直接使用するか、前述のリセットを使用します
  2. width left: 0 right: 0. left right. , width auto, padding borders, left: 0 right:0.

注： 1つのオプションは、フッターからパディングを削除することでした。技術的には、コンテンツレイヤーが100％であり、コンテナーの幅を超えて拡張するためのパディングまたは境界線がないため、これで問題が解決します。しかし、ブロックモデルの問題に適応するためにインターフェースを変更する必要がないため、このソリューションは間違ったアプローチであると思います。それがなくても簡単に修正できます。

私にとっての現実box-sizing: border-boxは、私の一般的なリセットの一部として常にプロパティセットを持っていることです。同じことを行うと、この問題が発生することはほとんどありません。しかし、私はまだプロパティを設定すると技術のようにleft:0とright:0時間が示されているように、100％の幅が原因で発生するブロックモデルの問題を解決するよりも（いずれにせよ、私の経験から判断すると）信頼性が高いためです。

これで、上記のすべての要素を、絶対要素と固定要素の中央揃えに組み合わせ始めました。


このテストの質問ではほとんどの資料をカバーしているので、水平および垂直のセンタリングは、従来の方法を使用して負のマージンを介して、または新しいトランスフォームプロパティを使用して実行できることを示します。また、CSS-Tricks要素の優れたセンタリングガイドも紹介します。

Flexboxは、複雑なレイアウトの問題を解決するための多くの素晴らしいツールをもたらしました。リリース前は、垂直方向の中央揃えはCSSに実装された最も複雑な機能の1つであると報告されていました。Flexboxの登場により、垂直方向の中央揃えが日常的になりました。

.parent { display: flex; }
.child { margin: auto; }

質問9：異なる単位での計算


calc（）関数の使用は、2単位では計算できないため、または分数をわかりやすくする必要がある場合に便利です。このテスト用の質問ではcalc(100% + 1em)、div要素の幅が100pxであるという事実から始めて、式の結果がどうなるかを知ることができます。実際、div要素の幅が100pxであることは重要ではないため、これは少し混乱するかもしれません。パーセンテージは常に親の幅から始まるため、正解は「包含（親）ブロックの100％プラス1em」でした。

私が定期的に使用するいくつかの状況がありますcalc()。まず、何かを100％シフトしたい場合は常に、一定量のスペースを追加します。ドロップダウンメニューは、この良い例です。


ここでの特徴は、さまざまなサイズの呼び出し要素（この場合は2つの異なるサイズのボタン）と一緒に使用できるドロップダウンメニューを作成することです。このメニューを呼び出す要素の高さはわかりませんtop: 100%が、ドロップダウンメニューの上端が呼び出しボタンの下端に配置されることはわかっています。各メニューが対応するボタンの下端にプラス0.5emである場合、これはで実現できますcalc(100% + 0.5em)。もちろん、も使用できますtop: 110%が、これらの追加の10％は、呼び出しボタンとコンテナの高さに依存します。

質問10：負のマージン


兄弟をはじく正のマージンとは異なり、負のマージンは兄弟をシフトせずに互いに近づけます。この最後のテストの質問は2つの解決策を提供します。どちらもボタングループの二重境界線を技術的に排除しますが、フレームを削除するとマウスオーバー効果などの特定の技法の実行が難しくなるため、負のマージン技法を強く推奨します。


結果として生じる効果は、ボタンの間に表示される「共通の境界線」です。実際、ボタンに共通の境界線を設定することはできないため、1つのフレームが別のフレームと重なるように、負のマージン手法を使用する必要があります。次にz-index、マウスホバーの状態に基づいて、別のフレームの上に配置するフレームを設定するために使用します。z-indexここでは絶対配置を使わなくても便利ですが、お願いしましたposition: relative。2番目のボタンの左フレームを削除する手法を使用した場合、この効果の実装ははるかに困難になります。

それはすべて理にかなっています。


前に説明した多くのトリックを使用する最後のデモをもう1つ紹介します。タスクは、インデントを考慮して、コンテナの左端と右端に拡張するタイルを作成することです。タイルとは、幅に十分なスペースがない場合に次の行に折り返すブロックのリストを作成する機能を意味します。タイルにカーソルを合わせると結果が表示されます。


このタスクを実行する上での障害はインデントです。パディングがなければ、コンテナーの左端と右端に隣接するタイルを簡単に取得できます。問題は、インデントがマージンを使用して作成されることであり、タイルのすべての側面にマージンを追加すると、2つの問題が発生します。

  1. 33.33% margin . box-sizing padding borders .tile 33.33%, margins — , 100%, .
  2. .

最初の問題はで解決できますcalc((100% / 3) - 1em)。つまり、33.33％から各タイルの左マージンと右マージンを差し引いたものです。隣接する姉妹要素の崩壊は、上部と下部のマージンでのみ可能であるため、ここでは発生しません。その結果、2つのタイル間の水平距離は2つのマージンの合計（1em）になります。この場合、最初と最後のタイルは技術的に隣接していないため、たとえ一方が他方の下にあるとしても、折りたたみは上部マージンと下部マージンには適用されません。

とってcalc()3つのタイルは一列に収まりますが、それでもコンテナーの端には触れません。これを行うには、各タイルの左マージンと右マージンに等しいサイズの負のマージンを使用できます。例の緑の点線は、周囲のコンテンツの端に対応するタイルを描画するために負のマージンを適用するコンテナーです。それらが親要素のパディング領域にどのように適合するかを確認できます。負のマージンは周囲の隣接する要素をはじかないため、これは正常です。

その結果、タイルの外側に隣接する段落タグと整列するように、端から端まで広がる十分なスペースがあるタイルが得られます。

タイルを作成するには多くの方法があります（通常、タイルには長所と短所があります）。たとえば、Haydon Pickeringが話したかなりエレガントなCSSグリッドソリューションがあります。このソリューションは、コンテナーの要求をシミュレートする手法を使用して実装されます（ただし、グリッドマジックを使用します）。最終的に、グリッドを使用した彼のソリューションはflexboxよりも優れています。
