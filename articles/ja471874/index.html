レビューガイドラインを回避して電話でサーバーを起動した方法

こんにちは、ハブル。私の名前はアントン・ログインノフです。FINCHのiOS開発者です。

最近、ギャンブルにWebベースのインターフェイスを使用するという問題に直面しました。AppStoreレビューガイドラインの次の更新では、クパチーノの同僚が再びルールを強化しました。より具体的には、Webベースのインターフェイスのいずれかがリアルマネーギャンブルとして分類される場合、Appleはアプリケーションをリダイレクトできます。

私たちのアプリケーションの1つは90％のギャンブルであり、残りの10％はこれらのゲームの宣伝に使用されます。それらのいくつかはwebViewを介して機能するため、何らかの方法でリダイレクトから身を守る必要がありました。

何ができるか：

  1. これらのゲームをメインアプリケーションの外に持っていきます。
    つまり、避けられないことを延期するだけです。
  2. 簡単に更新できるゲームには特定のコンテナを使用します。
    それは良さそうに聞こえる→浸透しようとした→ReactとReact Nativeを研究するために数日間殺された→「スキーは行かない」→あまりよく聞こえないことに気づいた

    時間がほとんどなく、ゲームをゼロから書き直さなければならないため、これは非常に高価なソリューションです。それはすべて内部ルーティングに関するものです-それは完全にurlPathComponentsに関連付けられています
  3. ゲームをネイティブで実現します。
    長く、高価で、そしてまた長い。将来的には継続的にサポートする必要がありますが、そのような機会はありませんでした。
  4. ローカルで横になっているサイトにゲームを提供するサーバーの動作をシミュレートします。
    奇妙に聞こえますが、これは私が選択したオプションです。ゲームのレガシーへの最小限の変更が必要なため、これは高速です。
    推定されるデメリットには、ローカルにあるサイトのためにアセンブリのサイズが大きくなる、サーバーを起動してデバイスの負荷を増やす、などがあります。

電話でサーバーを起動する方法を説明しているHabréに関する記事は見つかりませんでした。このケースは非常にまれで興味深いものであると判断したので、ここでハブレについて話すことにしました。

トレーニング


私たちの勇敢なフロントマンがゲームのサイズを16倍に減らし（80 Mb-> 5 Mb）、内部パスを相対パスに変更しようとしたときに、GCDWebServerを選択してライブラリを決定しました。これは、数行のコードでHTTPサーバーを起動できる軽量のフレームワークです。

ライブラリを選択してから、サーバーが内部でどのように機能するか、どの時点で何が起こるか、システムリソースを無駄にしないようにサーバーを構成する方法については、何時間も勉強して理解しました。私たちのサーバーはトランジションをキャッチして処理することを学び、私はバリケードの反対側にあるサーバーを操作することを学びました。

開始


実際、サーバーを起動して実行するためのパラメーターを指定します。

do {
     try webServer.start(options: [GCDWebServerOption_BindToLocalhost: true,
                                                GCDWebServerOption_Port: 8080])
} catch {
     assertionFailure(error.localizedDescription)
     webServer.start(withPort: 8080, bonjourName: "PROJECT_NAME Web Server")
}

private func handle(request: GCDWebServerRequest) -> GCDWebServerResponse? {
        
        // 1)  
        if request.url.pathComponents.contains(Endpoint.game.rawValue) {
            
            guard let indexURL = bundle.url(forResource: "index", withExtension: "html") else {
                return sendError(.noHTML(nil))
            }
            
            do {
                let data = try Data(contentsOf: indexURL)
                let htmlString = String(data: data, encoding: .utf8) ?? ""
                
                return GCDWebServerDataResponse(html: htmlString)
            
            } catch {
                return sendError(.noHTML(error))
            }
            
        // 2)   (js etc)   
        } else if request.url.pathComponents.contains(Endpoint.nstatic.rawValue) {
            
            guard let resoursePath = bundle.resourcePath else {
                return sendError(.noJS(nil))
            }
          
            let relativePath = request.url.pathComponents.joined(separator: "/")
            let absolutePath = resoursePath + relativePath.dropFirst()
            let staticURL = URL(fileURLWithPath: absolutePath)
            
            do {
                let data = try Data(contentsOf: staticURL)
                return GCDWebServerDataResponse(data: data, contentType: ContentType.js.description)
           
            } catch {
                return sendError(.noJS(error))
            }

結論


面白くて緊張しました。まず、私はこれまでこのようなことをしたことがありませんでした。第二に、私たちは最近まで、私たちのアイデアが親アプリケーションにどのように影響するかを理解していませんでした。

実装に約32時間かかりました。サイトのサイズを最適化するために8時間、この機能を設計および作成するために24時間かかりました。

記事を書いているときに、このテクノロジーを使用するより一般的な方法は、バックエンドの準備が整うのを待たずにネイティブを開発することであるという結論に達しました。

さて、選択したアプローチの利点を要約すると：

  • データモデルを与えることでバックエンド時間を節約する
  • サーバーの動作をテストする機能
  • Mockから実際のAPIに切り替えるには、特定のハンドラーをオフにするだけです
