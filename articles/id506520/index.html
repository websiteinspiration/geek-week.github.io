Konferensi DEVOXX UK. Pilih kerangka kerja: Docker Swarm, Kubernetes atau Mesos. Bagian 1

Docker Swarm, Kubernetes dan Mesos adalah kerangka kerja orkestrasi kontainer yang paling populer. Dalam pidatonya, Arun Gupta membandingkan aspek-aspek berikut dari karya Docker, Swarm, dan Kubernetes:

  • Pengembangan lokal.
  • Fitur Penempatan
  • Aplikasi multicontainer.
  • Layanan penemuan layanan penemuan.
  • Layanan penskalaan.
  • Tugas yang dijalankan sekali saja.
  • Integrasi dengan Maven.
  • Pembaruan "bergulir".
  • Membuat cluster basis data Couchbase.

Sebagai hasilnya, Anda akan mendapatkan ide yang jelas tentang apa yang masing-masing instrumen orkestrasi tawarkan, dan belajar bagaimana menggunakan platform ini secara efektif.

Arun Gupta adalah ahli teknologi produk open-source utama Amazon Web Services yang telah mengembangkan komunitas pengembang Sun, Oracle, Red Hat, dan Couchbase selama lebih dari 10 tahun. Dia memiliki pengalaman luas dalam memimpin tim lintas fungsi yang terlibat dalam pengembangan dan implementasi kampanye dan program pemasaran. Dia memimpin tim teknik Sun, adalah salah satu pendiri tim Java EE dan pencipta Devoxx4Kids cabang Amerika. Arun Gupta adalah penulis lebih dari 2 ribu posting di blog TI dan telah membuat presentasi di lebih dari 40 negara.



Selamat siang, saya tahu ini adalah penampilan terakhir sebelum makan malam, tetapi saya harap Anda memiliki energi lebih banyak dari sarapan untuk mendengarkan saya. Nama saya Arun Gupta, saya bekerja untuk AWS, yang akan kita bicarakan sekarang, serta Docker dan Kubernetes, sehingga Anda dapat memilih kerangka kerja yang paling cocok untuk Anda sendiri.

Saya adalah kapten Docker, yang sangat saya banggakan, hanya ada 70 orang di seluruh dunia yang diakui oleh Docker dalam status ini. Kami mulai dengan versi paling awal dari sistem 0.3, saya menulis buku tentang Docker, jadi saya tahu sesuatu tentang teknologi ini. Selain itu, saya telah menjadi bintang rock JavaOne selama 4 tahun berturut-turut, jadi saya mengerti topiknya. Saya selalu belajar dari audiens saya dan bertukar tips yang berguna dengannya, jadi terima kasih telah berpartisipasi.

Kami mulai mempertimbangkan masalah ini dari tingkat tertinggi. Ini bukan Docker 1.01, jadi jangan harap saya mengajarkan Anda dasar-dasarnya. Angkat tangan Anda, siapa yang menggunakan Docker dalam bentuk apa pun hari ini?



Luar biasa, hampir 90% hadir. Jadi, secara global, alur kerja Docker terdiri dari 3 komponen utama: bagian klien, host Docker, dan registri Docker. Misalkan Anda ingin menjalankan wadah untuk gambar tertentu. Anda memberi tahu klien: "Unduh gambar ini dan luncurkan wadah!". Untuk berkomunikasi dengan tuan rumah, klien menggunakan buruh pelabuhan membangun, buruh pelabuhan menarik, dan menjalankan perintah buruh pelabuhan. Dia memberikan perintah kepada tuan rumah, dia berkata: "Ya, saya tidak punya gambar ini!", Kemudian secara default dia mengakses registri hub Docker di hubdocker.com, mengunduh gambar yang diinginkan, menyimpannya dan menjalankannya dalam wadah. Dalam hal ini, klien sama sekali tidak memiliki status di luar lokasi kerja, ia hanya mengirim perintah seperti CLI melalui kawat, ia pergi ke host sebagai REST API, dan host, memahami persyaratan API,mengunduh gambar dari hub atau registri otomatis wadah ECR Docker Amazon, dan kemudian meluncurkan wadah berdasarkan gambar itu. Ini adalah keadaan yang sepenuhnya statis, dikelola hanya pada host. Dalam hal ini, kami memiliki satu host, tetapi cukup mudah untuk menggunakan solusi multi-host.

Inilah kerangka kerja orkestrasi - Anda tidak ingin membuat sistem aplikasi terdistribusi pada satu host, karena ini adalah titik terlemah, yang, jika terjadi kegagalan, akan menjatuhkan seluruh sistem. Anda ingin mendistribusikan tugas di beberapa host untuk menghindari kegagalan global. Misalnya, Anda ingin menjalankan aplikasi Java yang menggunakan server aplikasi, server database, caching, banyak komponen berbeda. Selain itu, sering memerlukan operasi simultan dari beberapa contoh aplikasi ini, karena aplikasi Java yang khas adalah aplikasi multi-kontainer yang menggunakan kontainer pada beberapa host. Oleh karena itu, Anda memerlukan kerangka kerja orkestrasi, tidak hanya untuk perencanaan kontainerisasi, tetapi juga untuk menciptakan kemampuan untuk mengelola cluster Anda,siklus hidup dan operasi lain yang diperlukan.

Dalam presentasi ini, saya akan memberi tahu Anda bagaimana dua alat orkestrasi aplikasi utama berbeda - Docker dan Kubernetes, kemampuan mereka, pro dan kontra menggunakannya. Saya akan mempertimbangkan masalah-masalah ini dari sudut pandang pengembang yang mulai membuat aplikasi apa pun dengan konsep kunci dan mendasar. Konsep ini mencakup komponen seperti kluster, wadah tunggal, banyak wadah, interaksinya satu sama lain, penemuan layanan (atau katalogisasi layanan yang memungkinkan mereka untuk saling mengenal), penyeimbangan muatan, penyimpanan persisten yang ditujukan untuk kontainer stateful , serta pengetahuan tentang bagaimana pembangunan lokal akan dilakukan.



Dari perspektif Dev, kita akan beralih ke perspektif Ops, ke sisi arsitektur sistem: cara melipatgandakan simpul master, bagaimana algoritma penjadwal harus bekerja ketika bekerja pada banyak host, apa aturan dan batasan yang dapat saya jalankan wadah pada host tertentu, bagaimana saya bisa melakukan pemantauan dan bergulir pembaruan, cara menggunakan dukungan cloud. Hari ini kita akan membicarakan semua ini.

Mari kita mulai dengan Docker Swarm - ini adalah salah satu fitur dari Docker Engine, dimulai dengan versi 1.2, alat pengelompokan standar. Ini mengubah satu set host Docker menjadi satu cluster Swarm berurutan tunggal. Jika Anda masuk ke docker.com, Anda dapat mengunduh satu dari dua edisi: Edisi Komunitas atau Edisi Perusahaan (hari ini versi ini disebut Stable dan Edge). Build pertama adalah apa yang saya gunakan, itu stabil dan diperbarui setiap 3 bulan, tidak seperti build Edge terbaru, yang diperbarui setiap bulan. Saya menggunakan versi Docker CE untuk Mac. Jadi, ketika saya mengunduh Docker CE ke komputer saya, selama instalasi saya menandai Swarm yang terkandung di dalamnya, dan setelah itu saya dapat segera mulai membuat cluster, mengatur dan merencanakan wadah.

Harap dicatat bahwa modul Docker Swarm membutuhkan komunikasi dengan host, jadi ketika menggunakan kartu jaringan dengan sistem multi-host, Anda perlu mendengarkan alamat host Docker tertentu menggunakan unit swarm docker - - dengarkan-addr: 2377 perintah.
Ikon kuning menunjukkan satu master node secara default. Wizard memberi saya token kontrol pekerja-token, saya mengambilnya dan menggunakannya untuk melampirkan simpul kerja baru, mesin fisik atau virtual, tidak masalah.



Token ini menunjukkan bahwa simpul ini harus bergabung dengan master asli. Menggunakan token pekerja, saya bisa membentuk cluster Docker saya sendiri. Dengan demikian, menggunakan Docker, sangat mudah untuk membuat sekelompok beberapa node. Slide berikutnya menunjukkan sekelompok satu simpul utama dan 5 simpul yang bekerja.



Dengan kemudahan yang sama, Anda dapat membuat cluster dengan beberapa node master - node master primer dan node master sekunder, yang menghubungkannya dengan node kerja.



Di sisi klien, ketika Anda mengirim permintaan ke master sekunder, itu sebagai proxy mengalihkannya ke master utama, dan yang sudah menjalankan permintaan, mendistribusikan beban antara node kerja. Secara default, kontainer Anda juga bisa bekerja di node manajer, tetapi saat membuat cluster, kontainer harus ditempatkan hanya di node kerja, dan manajer hanya akan mengelola cluster.

Konsep dasar berikut, sebuah cluster, berarti arsitektur dari beberapa node master dan beberapa node yang berfungsi di mana kontainer berjalan. Ini adalah skema yang sangat konsisten, direplikasi berdasarkan algoritma konsensus Raft, dengan kecepatan tertinggi karena membaca dari RAM.



Saya sangat merekomendasikan menggunakan templat dengan protokol Raft, karena jika Anda membagi jaringan, sistem tidak harus memikirkan siapa yang sekarang memainkan peran sebagai penyihir, karena dengan Raft, pemilihan akan dilakukan secara otomatis. Di tingkat bawah adalah simpul kerja Swarm yang berkomunikasi satu sama lain melalui protokol Gossip. Anda melihat bahwa kontainer merah bekerja pada 3 node, dan container hijau pada dua node - ini menunjukkan bahwa Swarm Worker memungkinkan Anda untuk memulai dan melakukan beberapa tugas berbeda.

Bagaimana konsep Layanan Replikasi Docker bekerja? Karena saya seorang pengembang Java, dalam hal ini saya menggunakan wadah WildFly.



Untuk melakukan ini, melalui baris perintah Docker, saya memasukkan layanan docker perintah membuat - - replika 3 - nama web jboss / wildfly, menunjukkan jumlah replika yang saya butuhkan, nama layanannya adalah web, dan menambahkan gambar wildfly. Gambar ini akan diunggah ke host 3 Docker dan meluncurkan wadah.

Konsep penting lainnya adalah kegagalan simpul Kegagalan Node. Apa yang terjadi?



Docker melihat bahwa keadaan saat ini dari sistem tidak sesuai dengan yang diinginkan, satu node jatuh dari cluster, yang berarti bahwa perlu untuk mentransfer kontainer dari node ini ke host lain. Artinya, jika Anda menyatakan perlunya tiga replika bekerja, Docker akan memastikan bahwa 3 replika ini ada dalam kondisi apa pun. Suatu situasi dapat terjadi ketika sebuah wadah gagal di simpul yang berfungsi. Mirip dengan kasus sebelumnya, Docker melacak jumlah kontainer yang ditentukan, dan jika menurun, scheduler diaktifkan yang segera mengembalikan wadah, membawa keadaan cluster saat ini ke keadaan yang diinginkan. Artinya, jika Anda mengumumkan 3 replika, semua replika ini harus bekerja pada waktu tertentu.
