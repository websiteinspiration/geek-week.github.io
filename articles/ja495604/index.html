Symfony 4 + Twigでの一時的なローカリゼーション

製品が一時的にローカライズされる必要性は、製品がさまざまなタイムゾーンでの作業を必要とする規模に成長したときに発生します（証拠）。このケースを解決するための簡単なアイデアの変形について説明します。

背景は次のとおりです。私たちはニッチなCRM / ERPシステムを開発しましたが、明日はウラジオストクからカリーニングラードまでのフランチャイズでこのシステムを使用することになると言われました。残念ながら、そのようなシナリオは当初考えられていなかったため、最小限のコストと最大の利便性でこれを行う方法を学び始めました。

合計すると、3つのタスクが大きくなったことがわかりました。データを出力する方法と入力する方法、およびそれらの間で、すべてを保存する方法です。ご存じのように、時間は比較的文字通りであり、比喩的であるため、以前と同様にモスクワUTC + 3に時間を格納することを決定しましたが、入力と出力で処理します（参照ポイントはUTC + 3であることに常に注意してください）。もちろん、私たちはこの方向や他の方向に他の解決策があることを理解しました。既存のすべてのエントリをUTC + -0に変換したり、DBMSでタイムゾーンを格納する特殊なタイプを使用したり、データベースがそのような機能を完全にサポートしなくなった場合は、このカスタムタイプを自分で作成できます。しかし、単純さの原則に基づいて、提案されたパスに沿って進みましたが、一目で、彼は残りの部分に多くを失うことはありませんでした。そして、希望のタイムゾーンを決定するためのロジックは非常に簡単でした。

モスクワが基準点になった後、各ユーザーにカスタムタイムゾーンパラメーターを追加し、いくつかの関連エンティティ（組織、都市、アプリケーション、トランザクションなど）を追加しました。次に、ユーザーまたはエンティティがどのタイムゾーンで作業しているかを明確に確認することができました。そこにあるロジックは標準であり、プロジェクトに固有のものであることがよくあります。このロジックをサービスにラップし、必要なタイムゾーンを取得しました

$localizationService->getTimezone();

テンプレートで日付をローカライズする決定は次のとおりです。Twig拡張機能を初期化するときに、タイムゾーンが目的のタイムゾーンに変更されました。

function __construct(Environment $twig, LocalizationService $localizationService) {
    $twig->getExtension('Twig_Extension_Core')->setTimezone($localizationService->getTimezone());
}

日付と時刻が表示された後、サブスクリプション「01.01.2020 12:30（モスクワ）」を作成する必要があるため、たとえば、タイムゾーンにリンクされている条件付きオーダー/タスク/トランザクションでは、時間に関する情報が表示されるため、状況はさらに複雑になりました。ベルト。実用的な理由から、これは、単一のコールセンターがタスク/アプリケーション/トランザクションのフレームワーク内でさまざまなタイムゾーンを快適に使用できるようにするために必要です。

タイムゾーンの優先度を決定するためのすべてのロジックは、前述のgetTimezoneに組み込まれました。

日時の入力の問題を解決するために残っています。1つの解決策：

private function getOffsetHours()
{
    if (!$this->isInit)
        $this->init();

    $local = new \DateTime('now', new \DateTimeZone($this->getTimezone()));
    $user = new \DateTime('now');

    $localOffset = $local->getOffset() / 3600;
    $globalOffset = $user->getOffset() / 3600;

    $diff = $globalOffset - $localOffset;
    return $diff;
}

public function toGlobalTime(\DateTimeInterface $dateTime): \DateTimeInterface 
{
    if (!$this->isInit)
        $this->init();

    $offsetHours = $this->getOffsetHours();

    if ($offsetHours > 0) {
        return $dateTime->modify('+ '.$offsetHours.' hours');
    } else  if($offsetHours < 0){
        return $dateTime->modify($offsetHours.' hours');
    }

    return $dateTime;
}

次に、リスナーなどで日時を保存する前に呼び出します。このオプションは、基本的にプロジェクトの時刻と日付が特定のイベントに対して固定されており、手動で入力されなかったために思い付きました。別の極端なケースでは、フォームを通じて常に時間が導入されるため、ソリューションが最適でない場合があります。

ボーナスとして。プロジェクトはOmines datatables-bundleを使用してテーブルを出力します。そこでの解決策はさらに簡単でした。ローカリゼーション用のDateTimeColumnの代わりに、以下が使用されました：

class CustomDateTimeColumn extends DateTimeColumn
{
    
    private $localizationService;
    private $timeZone;
    
    public function __construct(LocalizationService $localizationService)
    {
        $this->localizationService = $localizationService;
        $this->timeZone = $localizationService->getTimezoneObject();
    }
    
   
    public function normalize($value)
    {
        $value->setTimeZone($this->timeZone);
        return parent::normalize($value);
    }
}
