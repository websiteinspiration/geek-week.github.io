Pemrograman Fungsional Praktis







Teks artikel diambil dari presentasi yang saya perlihatkan di LinkedIn pada tahun 2016. Presentasi berusaha menjelaskan pemrograman fungsional tanpa menggunakan konsep-konsep seperti "monad," "kekekalan," atau "efek samping." Alih-alih, dia berfokus pada bagaimana cara berpikir tentang komposisi dapat membuat Anda menjadi programmer yang lebih baik, apa pun bahasa yang Anda gunakan.



40 tahun yang lalu, pada 17 Oktober 1977, Hadiah Turing diberikan kepada John Bakus untuk kontribusinya pada pengembangan sistem pemrograman tingkat tinggi, terutama bahasa pemrograman Fortran. Semua pemenang Turing Prize diberi kesempatan untuk memberikan kuliah tentang topik pilihan mereka selama tahun di mana mereka menerima hadiah. Sebagai pencipta bahasa pemrograman Fortran, orang akan berharap Bakus memberikan ceramah tentang manfaat Fortran dan perkembangan masa depan dalam bahasa ini. Sebagai gantinya, ia memberikan kuliah berjudul "Apakah mungkin untuk membebaskan pemrograman dari gaya von Neumann"? di mana ia mengkritik beberapa bahasa utama saat itu, termasuk Fortran, karena kekurangan mereka. Dia juga mengusulkan alternatif: gaya pemrograman fungsional.



Kuliah ini kontras dengan program tradisional dan "ketidakmampuan mereka untuk secara efektif menggunakan bentuk penggabungan yang kuat" dengan program fungsional yang "didasarkan pada penggunaan bentuk gabungan". Dalam beberapa tahun terakhir, pemrograman fungsional telah memicu minat baru karena pertumbuhan komputasi scalable dan paralel. Tetapi keuntungan utama pemrograman fungsional adalah apakah program Anda diparalelkan atau tidak: pemrograman fungsional memiliki komposisi yang lebih baik.



Komposisi adalah kemampuan untuk membentuk perilaku kompleks dengan menggabungkan bagian-bagian sederhana. Dalam pelajaran ilmu komputer, banyak perhatian diberikan kepada abstraksi: mengambil masalah besar dan membaginya menjadi beberapa bagian. Penekanan kurang ditempatkan pada sebaliknya: begitu Anda menyadari bagian-bagian kecil, bagaimana menyatukannya. Tampaknya beberapa fungsi dan sistem mudah disatukan, sementara yang lain jauh lebih rumit. Tetapi kita perlu mengambil langkah mundur dan bertanya: sifat apa dari fungsi dan sistem ini yang memfasilitasi tata letaknya? Properti apa yang membuat tata letak mereka sulit? Setelah membaca kode yang cukup, pola mulai muncul, dan pola ini adalah kunci untuk memahami pemrograman fungsional.



Mari kita mulai dengan melihat fungsi yang terhubung dengan sangat baik: 



String addFooter(String message) {
  return message.concat(" - Sent from LinkedIn");
}


Kami dapat dengan mudah menautkannya dengan fungsi lain tanpa harus membuat perubahan pada kode sumber kami: 



boolean validMessage(String message) {
  return characterCount(addFooter(message)) <= 140;
}


Itu hebat, kami mengambil sebagian kecil fungsionalitas dan menggabungkannya untuk melakukan sesuatu yang lebih. Pengguna fungsi validMessage



bahkan tidak perlu menyadari fakta bahwa fungsi ini dibangun dari yang lebih kecil; itu disarikan sebagai detail implementasi.



Sekarang, mari kita lihat fungsi yang tidak diatur dengan baik: 



String firstWord(String message) {
  String[] words = message.split(' ');
  if (words.length > 0) {
    return words[0];
  } else {
    return null;
  }
}


Dan kemudian coba buat dengan fungsi lain: 



// “Hello world” -> “HelloHello”
duplicate(firstWord(message));


Meskipun kesederhanaan pada pandangan pertama, jika kita menjalankan kode di atas dengan pesan kosong, kita mendapatkan pengecualian yang mengerikan NullPointerException



. Salah satu opsi adalah memodifikasi fungsi dublicate untuk menangani fakta bahwa inputnya terkadang null



: 



String duplicateBad(String word) {
  if (word == null) {
    return null;
  } else {
    return word.concat(word);
  }
}


Sekarang kita dapat menggunakan fungsi ini dengan fungsi firstWord



dari contoh sebelumnya dan hanya memberikan nilai nol null



. Tetapi ini bertentangan dengan komposisi dan abstraksi. Jika Anda terus-menerus harus masuk dan memodifikasi komponen setiap kali Anda ingin melakukan sesuatu yang lebih, maka ini tidak sesuai dengan tata letak. Idealnya, Anda ingin fungsi terlihat seperti kotak hitam, di mana detail implementasi yang tepat tidak masalah.



Nol objek tidak dikomposisi dengan baik.


Mari kita lihat implementasi alternatif yang menggunakan tipe Java 8 Optional



(juga disebut Option



atau Maybe



dalam bahasa lain): 



Optional<String> firstWord(String message) {
  String[] words = message.split(' ');
  if (words.length > 0) {
    return Optional.of(words[0]);
  } else {
    return Optional.empty();
  }
}


Sekarang kita akan mencoba untuk membuatnya dengan fungsi yang tidak berubah dublicate



: 



// "Hello World" -> Optional.of("HelloHello")
firstWord(input).map(this::duplicate)


Berhasil! Jenis opsional memperhatikan bahwa firstWord terkadang tidak mengembalikan nilai. Jika Optional.empty()



dikembalikan dari firstWord



, maka fungsi tersebut .map



hanya akan melewatkan awal fungsi dublicate



. Kami dapat dengan mudah menggabungkan fungsi tanpa perlu memodifikasi dublicate



. Bandingkan ini dengan null



kasing ketika kami perlu membuat fungsi duplicateBad



. Dengan kata lain: objek nol dikompilasi dengan buruk, dan pilihannya bagus.



Pemrogram fungsional terobsesi untuk membuat sesuatu menjadi komposit. Sebagai hasilnya, mereka menciptakan seperangkat alat besar yang diisi dengan struktur yang membuat kode non-komposable menjadi dapat digabungkan. Salah satu alat ini adalah tipe opsional untuk bekerja dengan fungsi yang mengembalikan output yang valid hanya pada waktu tertentu. Mari kita lihat beberapa alat lain yang dibuat.



Kode asinkron terkenal sulit untuk dirakit. Fungsi asinkron biasanya menerima "panggilan balik" yang dipicu ketika bagian asinkron panggilan selesai. Misalnya saja fungsinyagetData



dapat membuat panggilan HTTP ke layanan web dan kemudian menjalankan fungsi pada data yang dikembalikan. Tetapi bagaimana jika Anda ingin membuat panggilan HTTP lain tepat setelah itu? Dan lagi? Melakukan ini dengan cepat membawa Anda ke dalam suatu situasi yang dikenal sebagai panggilan balik neraka. 



getData(function(a) {  
    getMoreData(a, function(b) {
        getMoreData(b, function(c) { 
            getMoreData(c, function(d) { 
                getMoreData(d, function(e) { 
                    // ...
                });
            });
        });
    });
});


Misalnya, dalam aplikasi web yang lebih besar, ini menghasilkan kode spageti yang sangat bersarang. Bayangkan mencoba mengisolasi salah satu fungsi getMoreData



dalam metodenya sendiri. Atau bayangkan Anda mencoba menambahkan penanganan kesalahan ke fungsi bersarang ini. Alasan tidak dapat ditautkan adalah karena ada banyak persyaratan kontekstual untuk setiap blok kode: blok bagian dalam memerlukan akses ke hasil dari a, b, c, dll., Dll.



Nilai lebih mudah disatukan daripada fungsi


Mari kita lihat toolkit programmer fungsional untuk menemukan alternatif: Promise



(kadang-kadang disebut Future



dalam bahasa lain). Ini kodenya sekarang: 



getData()
  .then(getMoreData)
  .then(getMoreData)
  .then(getMoreData)
  .catch(errorHandler)


Fungsi getData



sekarang mengembalikan nilai Promise



alih-alih menerima fungsi panggilan balik. Nilai lebih mudah untuk disatukan daripada fungsi karena mereka tidak memiliki prasyarat yang sama dengan panggilan balik. Sekarang mudah untuk menambahkan penanganan kesalahan ke seluruh blok berkat fungsi yang disediakan objek kepada kami Promise



.



Contoh lain dari kode non-komposer yang kurang banyak dibicarakan daripada kode asinkron adalah loop atau, lebih umum, fungsi yang mengembalikan beberapa nilai, seperti daftar. Mari kita lihat sebuah contoh: 



// ["hello", "world"] -> ["hello!", "world!"]
List<String> addExcitement(List<String> words) {
  List<String> output = new LinkedList<>();
  for (int i = 0; i < words.size(); i++) {
    output.add(words.get(i) + “!”);
  }
  return output;
}

// ["hello", "world"] -> ["hello!!", "world!!"]
List<String> addMoreExcitement(List<String> words) {
  return addExcitement(addExcitement(words));
}


Kami telah menyusun fungsi yang menambahkan satu tanda seru ke fungsi yang menambahkan dua karakter. Ini berfungsi, tetapi tidak efisien karena melewati loop dua kali, dan tidak hanya sekali. Kita bisa kembali dan mengubah fungsi aslinya, tetapi, seperti sebelumnya, ini melanggar abstraksi.



Ini adalah contoh yang sedikit dibuat-buat, tetapi jika Anda membayangkan kode tersebar di basis kode yang lebih besar, itu menggambarkan poin penting: pada sistem besar, ketika Anda mencoba memecah hal-hal menjadi modul, operasi pada satu bagian data tidak akan hidup bersama. Anda harus memilih antara modularitas atau kinerja.



Dengan pemrograman imperatif, Anda hanya bisa mendapatkan modularitas atau kinerja yang adil. Dengan pemrograman fungsional, Anda dapat memiliki keduanya.


Jawaban programmer fungsional (setidaknya di Java 8) adalah ini Stream



. Stream



standarnya adalah lazy, yang berarti bahwa data hanya pergi jika diperlukan. Dengan kata lain, fungsi "malas": mulai berfungsi hanya ketika ditanya tentang hasilnya (bahasa pemrograman fungsional Haskell dibangun berdasarkan konsep kemalasan). Mari kita menulis ulang contoh di atas dengan menggunakan Stream



: 



String addExcitement(String word) {
  return word + "!";
}

list.toStream()
  .map(this::addExcitement)
  .map(this::addExcitement)
  .collect(Collectors.toList())


Dengan demikian, perulangan melalui daftar hanya akan sekali dan akan memanggil fungsi addExcitement



dua kali untuk setiap elemen. Sekali lagi, kita perlu membayangkan bahwa kode kita bekerja dengan potongan data yang sama di beberapa bagian aplikasi. Tanpa struktur malas seperti itu, sebagai Stream



upaya untuk meningkatkan produktivitas dengan menggabungkan semua perayapan daftar di satu tempat akan berarti penghancuran fungsi yang ada. Dengan objek yang malas, Anda dapat mencapai modularitas dan kinerja, karena bypass tertunda hingga akhir.



Sekarang kita telah melihat beberapa contoh, mari kembali ke tugas untuk mencari tahu properti mana yang membuatnya lebih mudah untuk membuat beberapa fungsi dibandingkan dengan yang lain. Kami telah melihat bahwa hal-hal seperti objek null, callback, dan loop tidak dapat dikomposisikan dengan baik. Di sisi lain, tipe opsional, janji, dan stream benar-benar menumpuk dengan baik. Kenapa begitu?



Jawabannya adalah bahwa contoh-contoh gabungan memiliki perbedaan yang jelas antara apa yang ingin Anda lakukan dan bagaimana Anda benar-benar melakukannya.



Semua contoh sebelumnya memiliki satu kesamaan. Cara fungsional dalam melakukan berbagai hal berfokus pada apa yang Anda inginkan hasilnya. Mode tindakan yang berulang berfokus pada bagaimana Anda benar-benar sampai di sana, pada detail implementasi. Ternyata menulis instruksi berulang tentang cara melakukan sesuatu tidak terlihat sebagus deskripsi tingkat tinggi tentang apa yang perlu dilakukan.



Misalnya, dalam hal Janji: yang dalam hal ini membuat satu panggilan HTTP, diikuti oleh yang lain. Pertanyaannya tidak relevan dan abstrak: mungkin ia menggunakan kumpulan utas, kunci mutex, dll., Tetapi itu tidak masalah.



Pemrograman fungsional memisahkan apa yang Anda inginkan dari hasil dari bagaimana hasil itu dicapai.


Itulah definisi praktis saya tentang pemrograman fungsional. Kami ingin memiliki pemisahan masalah yang jelas dalam program kami. Bagian "apa yang Anda inginkan" adalah baik dan komposisional dan membuatnya mudah untuk membuat hal-hal besar dari yang lebih kecil.Pada titik tertentu, bagian "bagaimana Anda melakukannya" diperlukan, tetapi dengan memisahkannya, kami menghapus bahan yang tidak begitu komposit dari bahan yang lebih komposit,



kita bisa melihatnya dalam contoh nyata:



  • Apache Spark API untuk melakukan perhitungan pada dataset besar mengabstraksi detail dari mesin apa ia akan berjalan dan di mana data disimpan.
  • React.js menjelaskan presentasi dan menjadikan transformasi DOM sebagai algoritma yang efisien


Bahkan jika Anda tidak menggunakan bahasa pemrograman fungsional, memisahkan apa dan bagaimana dari program Anda akan membuatnya lebih mudah dikomposisikan.
