なぜ階層型アーキテクチャなのですか？


システム全体がドメインモデル（下の画像の中央の円）を中心に構築されているため、このアプローチはDDDに適しています。内層は外層に直接依存しておらず、インジェクションで使用します。さらに、優れた柔軟性が得られるため、各レイヤーには独自の責任範囲があり、着信データはニーズに応じて各レイヤーによって検証されます。したがって、内層は常に外層から有効なデータを受け取ります。テストは言うまでもありません。インターフェイスに基づく依存モックを使用すると、ユニットテストが簡単になり、データベースなどのシステムの外部部分から抽象化できます。

ドメイン


ドメイン層は、定義により、ビジネスルールの生息地です。外部レイヤーについて何も認識せず、依存関係がなく、簡単にテストできます。アプリケーション全体を変更した場合でも、ドメインには変更が加えられません。ドメインには、このコードを読むすべての人の目的を理解するために使用できるビジネスルールのみが含まれているためです。この層は、テストで可能な限りカバーする必要があります。

したがって、すべてのドメインクラスに共通の特定のロジックを含むことができるエンティティ基本クラスがあります。この例では、すべての一般的なロジックは、水平スケーリングに適した衝突耐性のある識別子を生成することであり、すべてのオブジェクトがこのアプローチを使用します。

export abstract class Entity<T> {
  protected readonly _id: string
  protected props: T

  constructor(props: T, id?: string) {
    this._id = id ? id : UniqueEntityID()
    this.props = props
  }

  // other common methods here...
}

Entityクラスを記述したら、Entity抽象クラスを拡張する中央ドメインクラスの作成を開始できます。

このクラスにはそれほど複雑なものはありませんが、注意すべきいくつかの興味深い点があります。

まず、コンストラクタはプライベートです。つまり、新しいCart（）を実行するとエラーが発生します。これが必要です。 DDDでは、ドメインオブジェクトを常に有効な状態に保つことをお勧めします。空のCartオブジェクトを直接作成する代わりに、Factoryパターンを使用しますCartクラスの完成したインスタンスを返します。作成手順が必要なすべての属性を受け取ったことを確認するために、それらを検証できます。同様に、ゲッターとセッターはドメインとのやり取りに使用されます。このため、クラスの内部状態はプライベートプロップオブジェクトに格納されます。ゲッターは、公開する必要がある属性への読み取りアクセスを提供します。同様に、パブリックセッターやその他のメソッドを使用すると、オブジェクトの状態の有効性を常に保証しながらドメインを変更できます。

まとめると、重要なポイントを強調できます。ドメインロジックは、プロパティではなく動作に焦点を当てるべきです。

export class Cart extends Entity<ICartProps> {
  private constructor({ id, ...data }: ICartProps) {
    super(data, id)
  }

  public static create(props: ICartProps): Cart {
    const instance = new Cart(props)
    return instance
  }

  public unmarshal(): UnmarshalledCart {
    return {
      id: this.id,
      products: this.products.map(product => ({
        item: product.item.unmarshal(),
        quantity: product.quantity
      })),
      totalPrice: this.totalPrice
    }
  }

  private static validQuantity(quantity: number) {
    return quantity >= 1 && quantity <= 1000
  }

  private setProducts(products: CartItem[]) {
    this.props.products = products
  }

  get id(): string {
    return this._id
  }

  get products(): CartItem[] {
    return this.props.products
  }

  get totalPrice(): number {
    const cartSum = (acc: number, product: CartItem) => {
      return acc + product.item.price * product.quantity
    }

    return this.products.reduce(cartSum, 0)
  }

  public add(item: Item, quantity: number) {
    if (!Cart.validQuantity(quantity)) {
      throw new ValidationError(
        'SKU needs to have a quantity between 1 and 1000'
      )
    }

    const index = this.products.findIndex(
      product => product.item.sku === item.sku
    )

    if (index > -1) {
      const product = {
        ...this.products[index],
        quantity: this.products[index].quantity + quantity
      }

      if (!Cart.validQuantity(product.quantity)) {
        throw new ValidationError('SKU exceeded allowed quantity')
      }

      const products = [
        ...this.products.slice(0, index),
        product,
        ...this.products.slice(index + 1)
      ]

      return this.setProducts(products)
    }

    const products = [...this.products, { item, quantity }]
    this.setProducts(products)
  }

  public remove(itemId: string) {
    const products = this.products.filter(product => product.item.id !== itemId)
    this.setProducts(products)
    this.emitCartMutation()
  }

  public empty() {
    this.setProducts([])
  }
}

Cartオブジェクトは、ドメインメソッドを使用するアプリケーションで使用できますが、場合によっては、クリーンなオブジェクトに「デプロイ」する必要がある場合があります。たとえば、データベースに保存するか、JSONオブジェクトとしてクライアントに送信します。これは、unmarshal（）メソッドを使用して実装できます。

アーキテクチャの柔軟性を高めるために、ドメインレイヤーはドメインイベントのソースになることもできます。ここでは、ドメインエンティティが変更されたときにイベントを作成して、イベントソーシングアプローチを適用できます。

ユーザースクリプト


ここでは、インフラストラクチャレベルから実装されたドメインメソッドとオブジェクトを使用してデータを格納します。inversify

ライブラリを使用して、インフラストラクチャレイヤーからこのシナリオにリポジトリを挿入するInversion of Managementアプローチを実装し、ドメインメソッドを使用してバスケットを操作し、その後データベースに変更を保存する機能を提供します。

import { inject, injectable } from 'inversify'

@injectable()
export class CartService {
  @inject(TYPES.CartRepository) private repository: CartRepository

  private async _getCart(id: string): Promise<Cart> {
    try {
      const cart = await this.repository.getById(id)
      return cart
    } catch (e) {
      const emptyCart = Cart.create({ id, products: [] })
      return this.repository.create(emptyCart)
    }
  }

  public getById(id: string): Promise<Cart> {
    return this.repository.getById(id)
  }

  public async add(cartId: string, item: Item, sku: number): Promise<Cart> {
    const cart = await this._getCart(cartId)
    cart.add(item, sku)

    return this.repository.update(cart)
  }

  public async remove(cartId: string, itemId: string): Promise<Cart> {
    const cart = await this._getCart(cartId)
    cart.remove(itemId)

    return this.repository.update(cart)
  }
}

この層は、アプリケーションの操作を担当します。このレイヤーへのコード変更は、ドメインエンティティやデータベースなどの外部依存関係には影響しません。

インフラ


すべての自明性にもかかわらず、インフラストラクチャ層は、データベースなどの外部システムと対話します。

データをデータベースに保存するには、データマッパーとリポジトリのアプローチを使用します。

マッパーはデータベースからソースデータを取得し、対応するドメインオブジェクトに変換できます。

import { Cart, CartItem } from 'src/domain/cart'

const getProducts = (products: CartItem[]) => {
  return products.map(product => ({
    item: product.item,
    quantity: product.quantity
  }))
}

export class CartMapper {
  public static toDomain(raw: any): Cart {
    return Cart.create({
      id: raw.id,
      couponCode: raw.couponCode,
      products: getProducts(raw.products || [])
    })
  }
}

リポジトリ自体は、特定のデータベースのクライアントライブラリに依存する場合があります。たとえば、RAMのストレージから、次のメソッドを使用してデータを管理します。

import { injectable, inject } from 'inversify'
import { Cart } from 'src/domain/cart'
import { CartMapper } from '../mappers/cart'

interface ICartRepository {
  getById(id: string): Promise<Cart>
  create(cart: Cart): Promise<Cart>
  update(cart: Cart): Promise<Cart>
}

@injectable()
export class CartRepository implements ICartRepository {
  @inject(TYPES.Database) private _database: MemoryData

  async getById(id: string): Promise<Cart> {
    const cart = await this._database.cart.getById(id)
    if (!cart) {
      throw new ResourceNotFound('Cart', { id })
    }
    return CartMapper.toDomain(cart)
  }

  async create(cart: Cart): Promise<Cart> {
    const dtoCart = cart.unmarshal()
    const inserted = await this._database.cart.insert(dtoCart)
    return CartMapper.toDomain(inserted)
  }

  async update(cart: Cart): Promise<Cart> {
    const dtoCart = cart.unmarshal()
    const updated = await this._database.cart.update(cart.id, dtoCart)
    return CartMapper.toDomain(updated)
  }
}
