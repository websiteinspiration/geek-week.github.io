PCRE2がApache 2.4をサポートする方法

PHP 7でもPCRE2ライブラリを長くサポートしてきましたが、オープンソースのApache Software Foundationはまだないので、Apache 2.4をPCRE2に変換した経験を共有したいと思います。
もちろん、おそらく次のリリースではPCRE2のサポートがすでに可能であるというApache geoからのソースを使用しているため、おそらくPCRE2をサポートするApacheのリリースよりも先に進んでいます。方法の1つを共有するリリース。

この記事では、ソースから必要なすべてのソフトウェアを収集していること、および執筆時点でのソフトウェアとバージョンのリストを想定しています

。PCRE2-10.33
APR 1.7.0
APR-util 1.6.1
Apache httpd 2.4.41

ステップ1：PCRE2をビルドしてコンパイルする


ソースからソースをダウンロードするタイミングはわかりにくいので省略します。アーカイブを解凍し、PCRE2ソースのあるフォルダーに移動して、次のコマンドを実行してUTFをサポートします。

./configure --prefix=/etc/webserver/pcre2-1033 --enable-pcre2-8 --enable-pcre2-16 --enable-pcre2-32 --enable-unicode

ライブラリのインストールに標準の場所を使用しない場合は、プレフィックスにパスを指定します。

--prefix=///

それ以外の場合は、接頭辞なしで収集します。

残りのコマンドは、サポートする8ビット、16ビット、および32ビットのコードブロックPCREを含めることを示します。この実施形態では、アセンブリはそれらで実行されました。

そしてもちろん、コマンドの順次実行を使用してこのケースをコンパイルします。

make
make install

すべてのルールとコンパイルが問題なく完了したら、次のステップに進みます。

ステップ2：PCRE2ライブラリをAPRに接続する


ApacheはAPRを使用してソースをコンパイルするため、APR自体のライブラリを接続する必要があります。そうしないと、新しいPCRE2関数を使用するため、Apacheソースの不明な関数に関するエラーが発生する可能性があります。

ソースからソースをダウンロードすることは明らかであるため省略します。そのため、アーカイブを解凍してAPRを構成しました。

./configure --prefix=/etc/webserver/apr-170

当然、ライブラリのインストールに標準の場所を使用しない場合、または指定しない場合は、プレフィックスにパスを指定します。

--prefix=///

構成が完了したら、次のディレクトリに移動し

ます

。

EXTRA_LIBS=-lrt -lcrypt  -lpthread -ldl

ライブラリ接続：

-lpcre2-8 -L///  pcre2/lib

保存して、APRソースのルートディレクトリに移動します。

変更したAPRをコンパイルします。

make
make install

すべてのルールとコンパイルが問題なく完了したら、次のステップに進みます。

ステップ3：ソースからApache用のAPR-utilをビルドする


このライブラリをソースのオフィスからダウンロードし、APR-utilで解凍したアーカイブのルートフォルダーに移動して、次のコマンドを順番に入力します。

./configure --prefix=/etc/webserver/apr-util-161 --with-apr=///  apr
make
make install

当然、ライブラリのインストールに標準の場所を使用しない場合、または指定しない場合は、プレフィックスにパスを指定します。

--prefix=///

また、ここでは、APRを接続します。

--with-apr=///  apr

ステップ4：PCRE2をサポートするためにApache Gitからソースをダウンロードする


重要：最新バージョンのgitaからソースをダウンロードしてください。

ap_regex.hとutil_pcre.cなどの2つのソース、以下のリンクをダウンロードする必要があります
。ap_regex.h
util_pcre.c

次に、Apache httpdソースディレクトリに移動し、次のコマンドでApacheを収集します。

./configure --prefix=/etc/webserver/apache-2441 --with-apr=///  apr --with-apr-util=///  apr-util --with-pcre=///  pcre2/bin/pcre2-config

当然、ライブラリのインストールに標準の場所を使用しない場合、または指定しない場合は、プレフィックスにパスを指定します。

--prefix=/// Apache httpd

また、Apacheを自由に構築するための追加のコマンドを指定します。つまり、モジュールとライブラリを無効にするコマンドです。

次に、Apache httpdソースディレクトリに移動します。

/etc/webserver/srcsrv/httpd-2.4.41

あなたは自然にあなたのディレクトリに行き、ディレクトリで置き換えます：

/etc/webserver/srcsrv/httpd-2.4.41/include

Apache gitからダウンロードしたap_regex.hファイル。

また、ディレクトリに移動します。

/etc/webserver/srcsrv/httpd-2.4.41/server

util_pcre.cファイルを、Apache

からダウンロードしたApacheで置き換えます。次に、Apache自体にPCRE2接続を追加する必要があります。次のディレクトリにあるap_config_auto.hファイルを見つける必要があります。

/etc/webserver/srcsrv/httpd-2.4.41/include

このファイルの最初に、次の行を挿入します。

/* Load PCRE2 */
#define HAVE_PCRE2 1

さて、これで、PCRE2サポートを使用してApache httpdをコンパイルする真の瞬間の準備が整いました。
Apache httpdソースディレクトリに移動し、コマンドの順次実行を使用してこのケースをコンパイルします。

make
make install

これで、すべてがうまくいき、エラーがなければ、PCRE2をサポートするApache httpdをアセンブルおよびコンパイルしたことになります。これは、PCRE正規表現を使用するApacheモジュールのポジティブな変更であり、その1つがモジュールの書き換えです。

結論として、この方法では、Apache Software Foundationからのリリース前にPCRE2を使用できます。PCRE2をサポートするバージョンがすぐにリリースされることを願っています。

また、標準の.htaccessのテスト中に、コメントにエラーがあった場合でもエラーは発生しませんでした。
