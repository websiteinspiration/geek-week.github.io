Xiaomi Gateway 2ははんだ付けできません



こんにちはHabr！前のシリーズでは、私は：

  • 私はスマートホーム用にXiaomiからデバイスを購入し、はんだごてを通して、ホームアシスタントを介してネイティブサーバーなしでデバイスをエキサイティングな方法で機能させました（リンクへのリンク）
  • 通知、メニュー、ドットバーなどのサポートを使用して、ホームアシスタントのウェブインターフェースをelectron（ラップへのリンク）でラップしました（コードはこちら）。

しかし、ますます多くのデバイスが登場するにつれて、奇妙なことが開かれ始めました...

たとえば、Aqaraワイヤレスリレー2ch（LLKZMK11LM）がxiaomi_aqara統合をサポートしていなかったため、GitHubで問題が発生したときに、コードの所有者は、デバイスファームウェアを更新しないとこのリレーをサポートすることは不可能であると答えました。「ここで何かがおかしい」と私は考えて、コードがどのように発生したかを理解しました。そして、私は最初にxiaomi_miio（link）を介してリレーを操作するために統合を洗い流し、次にこの統合のためのキューブとXiaomiボタンのサポートを実装しました（link）。今、その理由と方法を説明します。

開発者API


むかしむかし、Xiaomiがゲートウェイ（lumi.gateway.v3）をリリースすると、その開発者APIとそのドキュメントさえも装備されました。開発者APIは、UDPを介してポート4321および9898で通信するものです。コミュニティはこの動きを非常に気に入っており、可能な限りすべてのデバイスをデバイスに統合する多数のリポジトリで対応しました。この開発者APIを使用して、ホームアシスタントxiaomi_aqara統合（リンク）の基礎を形成し、これまでのところすべてが素晴らしかったです...

Xiaomiはオーバーライドせず、開発者のAPIサポートを利用していませんが、新しいデバイスの識別子は表示されなくなりました。目的のリレーは、ゲートウェイを介してzigbeeに接続されているデバイスのリストにありますが、リレーは空のモデル識別子で表示され、それがないと何もできません。その後、最近、Xiaomiはインストール後にゲートウェイファームウェアの更新を公開しました。これは、はんだごてなしで開発者APIをオンにすることが不可能でした（更新前にオンになっていない場合）、リンク。

このサンタバーバラの中心に着くと、私は突然涙を流し、AmazonのドイツのZigBee ConBee2ドングルにゲートウェイなしでzig​​beeデバイスを操作するように注文しました。しかし、黙示録の状態にあるドイツ人が私の命令を送る準備をしている間、私は待つのに疲れて、より深く掘り始めました...

よりネイティブなMIIOプロトコル


Miioは、wi-fi経由でネットワークに接続されたXiaomiデバイスを操作するための新しいプロトコルです。ネイティブのMi Homeアプリケーションは、このプロトコルを介して、wi-fiソケット、IRリモート、掃除機、および他のすべての人と、さらにはxiaomiゲートウェイv2と通信します。これもUDPプロトコルですが、今回はポート4321ではなく、ポート54321（ポートを選択するための優れた方法を検討しています）です。 「グーフィー」と私は思いました。「彼女は別のプロトコルも知っていますが、この新しいプロトコルを使用してリレーを接続する必要がありますか？」

miioプロトコルはすべてのメッセージをトークンで暗号化しますが、幸いなことに、トークンの抽出と暗号化/復号化のメソッドはすでに開かれており、それらを使用する作業はいくつかのライブラリに実装されています（1回リンク、2つリンク））miioプロトコルを使用して新しいデバイスを統合するのは私が最初ではないため、パスはわかっていて、それに従いました。

方法はこれです：

  • セキュリティトークンデバイスを取得するには、ログ（リンク）または変更されたアプリケーション（リンク）からデバイスを抽出します
  • モバイルアプリケーションとデバイス間のトラフィックを削除します
  • 最初に受信したセキュリティトークンでUDPパケットを復号化します
  • 同様のパッケージを作成するための学習コード

私はwi-fiルーターから直接トラフィックを取得しましたが、幸いなことに、カスタムのasuswrt-merlinファームウェアがあり、tcpdumpはEntwareを使用してインストールできます。ユーティリティは問題ありませんが、tcpdumpからのダンプは認識しませんが、Wiresharkによって生成されたjson-dumpは認識します。バイナリダンプをtcpdumpからjsonに上書きするには、tsharkユーティリティを使用しました。その後、mi homeアプリケーションがリレーを制御するコマンドを確認し、リレーサポート（リンク）を追加してPRを生成しました。

ソフトウェアの方が速くて楽しいので、ConBee2の注文をキャンセルしました...

なぜ鍛冶屋が必要なのですか？


そして、それは私がダッシュベアですべてがうまくいくように見える状況にいることを意味しますが、今私のホームアシスタントでは私のXiamo Gateway 2は2つのエンティティで表され、1つは開発者APIプロトコル（xiaomi_aqara統合）を使用し、もう1つはmiioパンクチャ（xiaomi_miio統合）を使用しています。そして、私ははんだ付けするのが好きではありません、私は私の人生と私自身を簡素化したかったです。xiaomi_miioにデバイスの欠落部分を実装する必要がありました。

ほとんどの場合、ソフトウェアのストラップを作成した後、これは機械的な仕事です。

  1. 有効なtcpdump
  2. 追跡したいMi Homeアプリケーションに自動化アクションを作成しました
  3. この自動化を開始しました
  4. ダンプを解読した
  5. 復号化からコードへのコピーアンドペースト
  6. コンソールからコマンドを送信する
  7. ファイルの輝きの仕上げ

しかし、かゆみを手に入れ、冒険は呼びかけます...

偽のmiioデバイス


剖検の結果、miioプロトコルにはコールバックメカニズムがないことが判明しました。センサーは20秒ごとに1回引くことができますが、ボタンやキューブを正しく操作するために、1秒あたり10回デバイスに問い合わせることは悪い考えでした。このパラドックスでは、興味深いアクティビティを見ましたが、キャプチャされたパケットの機械的なエンコードを実際に処理したくありませんでした。

miioプロトコル（リンク）で動作する別のライブラリーの研究は、コアに驚きました。著者たちは、miioプロトコルを介してデバイスと通信するハーベスターを組み立てましたが、キューブとボタンからのイベントをインターセプトするためにdeveloper-apiを使用しています。このアプローチは私を完全に悩ませました、それは機能の一部がすぐに機能し始めるという事実につながります、しかし一部はデバイスをはんだ付けする必要がある...

tcpdumpで再び武装し、1回の実行ですべてのデバイスの復号化機能を使用してデコーダーを変更したので、ネットワーク経由でUDPポート54321に飛ぶすべてを聞き始めました。そして、かなり興味深い画像が表示されます。 「ジグビーキューブをオンにした場合は、wi-fiソケットをオンにする」のようなオートメーションを作成すると、jsonが文字列にエンコードされているデータフィールド内のmiioプロトコルを使用して、アプリケーションからゲートウェイにjsonが送信されます。 「DSL」-何らかの理由で、太古の昔から、Stirlitzはこのように梱包してきました...詳細はこちらです。

この行の内容を見ると、アクションへの反応がどこかでクラウドに送信されていないことがわかりました。このjsonプログラムでは、wi-fiソケットのIPアドレスと暗号化トークンが直接表示されました（この段落は、最初のシリーズの段落と矛盾しています。 、何のパブ/サブプロトコルはダンプで見つかりませんでしたされてはい）雲の助けを借りずに、スクリプト自体はホームアシスタントから自動化に似ている、トリガパラメータがある。ここで、反応を送信するには、ipがある-彼優しいデバイス自身と通信します。。。

しかし、アイデアは、すべてのものから生まれましたボタンとキューブからすべての必要なイベントをキャッチするには：

  1. アプリケーションでmiioデバイスをエミュレートし、PINGアプリケーションとゲートウェイからのリクエストに応答します
  2. ゲートウェイごとにスクリプトジェネレーターを作成して、ペアごとに1つを生成します（デバイス、イベント）。
  3. イベントが発生したときにmiioリクエストがエミュレーターに送信されるように、スクリプトを送信します

これを実現し、それを神の形にするのに数日かかりました。

現時点では、python-miioライブラリはリレー、キューブ、およびボタンlinkで動作します。

このアップデートをホームアシスタントにバインドすることは残り、はんだ付けや開発者APIなしでXiaomiゲートウェイ2を使用することが可能になります。
