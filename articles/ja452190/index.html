この記事では、JavaScriptをWebAssemblyに置き換えて、ブラウザーアプリケーションを高速化するケースについて説明します。

WebAssembly-それは何ですか？


つまり、これはスタックされた仮想マシン用のバイナリ命令フォーマットです。 Wasm（短縮名）はしばしばプログラミング言語と呼ばれますが、そうではありません。命令フォーマットはJavaScriptとともにブラウザで実行されます。

C / C ++、Rust、Goなどの言語でソースをコンパイルしてWebAssemblyを取得できることが重要です。それは静的型付けといわゆるフラットメモリモデルを使用します。上記のように、コードはコンパクトなバイナリ形式で保存されるため、コマンドラインを使用してアプリケーションを起動した場合とほぼ同じ速度で実行されます。これらの機能により、WebAssemblyの人気が高まりました。

私たちはあなたを思い出させる： 「Habr」のすべての読者のために-万ルーブルの割引を「Habr」プロモーションコードを使用して、もちろんすべてのSkillbox登録時。

Wasmは現在、Doom 3などのゲームから、AutocadやFigmaなどのWebアプリケーションへの移植まで、多くのアプリケーションで使用されています。Wasmは、サーバーレスコンピューティングなどの分野でも使用されます。

この記事では、Wasmを使用して分析Webサービスを高速化する例を示します。明確にするために、WebAssemblyにコンパイルされた、Cで記述された実用的なアプリケーションを使用しました。結果は、JSの低生産性セクションを置き換えるために使用されます。

この例では、遺伝学者向けのブラウザサービスfastq.bioを使用します。このツールを使用すると、DNAシーケンス（デコード）の品質を評価できます。

必要なインジケーターを計算した後、それらはPlotly.jsを使用して視覚化され、サービスは新しいデータ選択で作業を開始します。断片化は、UXの品質を向上させるために行われます。すべてのデータを一度に処理する場合、シーケンス結果を含むファイルは数百ギガバイトのファイル領域を占めるため、プロセスはしばらくフリーズします。このサービスは、サイズが0.5〜1 MBのデータを取得し、それらと段階的に連携してグラフィックデータを構築します。

WebAssemblyのテスト


Wasmを使用する可能性を評価するために、プロジェクトチームはfastqファイルに基づいてQCメトリック（QC-品質管理）を作成するための既製のソリューションの検索を開始しました。C、C ++、またはRustで記述されたツール間で検索が実行されたため、コードをWebAssemblyに移植できました。さらに、ツールは「未加工」であってはならず、科学者によって既に検証されたサービスが必要でした。

その結果、選択はseqtkを優先して行われました。アプリケーションは非常に人気があり、オープンソースで、ソース言語はC

です。Wasmに変換する前に、デスクトップ用にseqtkをコンパイルする原理を確認する必要があります。Makefileによると、ここに必要なものがあります：

# Compile to binary
$ gcc seqtk.c \
   -o seqtk \
   -O2 \
   -lm \
   -lz

基本的に、Emscriptenを使用してseqtkをコンパイルできます。ない場合は、Dockerイメージを回避してください。

$ docker pull robertaboukhalil/emsdk:1.38.26
$ docker run -dt --name wasm-seqtk robertaboukhalil/emsdk:1.38.26

ご希望の場合はご自分で回収できますが、時間がかかります。

コンテナー内では、gccの代わりにemccを簡単に使用できます。

# Compile to WebAssembly
$ emcc seqtk.c \
    -o seqtk.js \
    -O2 \
    -lm \
    -s USE_ZLIB=1 \
    -s FORCE_FILESYSTEM=1

最小限の変更：

Emscriptenバイナリファイルに出力する代わりに、.wasmと.jsを使用してファイルを生成します。これは、WebAssembyモジュールの実行に使用されます。

zlibライブラリをサポートするために、USE_ZLIBフラグが使用されます。ライブラリは配布されてWebAssemblyに移植され、Emscriptenはそれをプロジェクトに含めます。

Emscrippten仮想ファイルシステムがアクティブ化されます。これは、ブラウザ内のRAMで動作するPOSIXのようなFSです。ページが更新されると、メモリはクリアされます。

仮想ファイルシステムが必要な理由を理解するには、コマンドラインからseqtkを起動する方法と、コンパイルされたWebAssemblyモジュールを実行する方法を比較する価値があります。

# On the command line
$ ./seqtk fqchk data.fastq
 
# In the browser console
> Module.callMain(["fqchk", "data.fastq"])

ファイル入力ではなく文字列でseqtkを上書きしないように、仮想ファイルシステムにアクセスする必要があります。この場合、データフラグメントは、main（）seqtkが呼び出された仮想FSのdata.fastqファイルとして表示されます。

これが新しいアーキテクチャです。この



図は、ブラウザのメインストリームがコンピューティングの代わりにWebWorkersを使用することを示しています。このメソッドを使用すると、ブラウザーの応答性を損なうことなく、バックグラウンドスレッドで計算を実行できます。まあ、WebWorkerコントローラーはワーカーを起動し、メインスレッドとの相互作用を制御します。

seqtkコマンドは、マウントされたファイルでWorkerを使用して起動されます。作業が完了すると、Workerは結果をPromiseの形式で表示します。メインスレッドがメッセージを受信すると、その結果を使用してスケジュールが更新されます。そして、数回の反復で。

WebAssemblyのパフォーマンスはどうですか？


パフォーマンスの変化を評価するために、プロジェクトチームは1秒あたりの読み取り操作数のパラメーターを使用しました。どちらの実装もJavaScriptを使用するため、インタラクティブなチャート作成時間は考慮されません。

すぐに使えるソリューションを使用すると、生産性は9倍向上しました。



これは素晴らしい結果ですが、結局のところ、それを最適化する機会があります。実際のところ、seqtkは多数のQC分析結果を使用しないため、それらを削除できます。これを行うと、JSに比べて結果が13倍向上します。

しかし、それだけではありません。実際のところ、この段階でfastq.bioはCのさまざまな関数を呼び出して分析結果を受け取ります。それらのいずれも独自の特性セットを計算するため、ファイルの各フラグメントが2回読み取られます。

この問題を回避するために、2つの機能を1つに統合することが決定されました。その結果、生産性が20倍に向上しました。



このような優れた結果が常に達成できるとは限らないことは注目に値します。場合によっては、パフォーマンスが低下するため、各ケースを評価する価値があります。
