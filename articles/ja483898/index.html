Confluenceに対する、またはConfluenceと連携したコードとしてのドキュメント？リポジトリからConfluenceに公開するいくつかの方法の概要

多くの人は、ドキュメントをコードとして保存および公開するモデルを長い間または積極的に使用しているか、またはそのモデルに目を向けてきました。つまり、プログラムコードと同じルール、ツール、および手順をドキュメントに適用します。たとえば、リポジトリへの保存、テストの実行、コンパイルおよびリリースCI / CDで。このアプローチにより、使い慣れた開発ツールを使用して、コード、バージョン、および変更を追跡してドキュメントを最新に保つことができます。

ただし、同時に、多くの企業がWikiを何年も使用していて、他のチームや従業員（プロジェクトマネージャーなど）がドキュメントにアクセスできます。ストレージとパブリケーションを単一のビューに、つまりConfluenceのHTMLパブリッシュドックと一緒に利用したい場合はどうでしょうか。この記事では、リポジトリからConfluenceにドキュメントを発行するタスクのソリューションの概要を説明します。

私はインターフェース開発チームで1つのソリューション（RST-Sphinx + sphinxcontribbuilderバンドル）を長い間積極的に使用してきましたが、他のソリューションを代替案として紹介します。実際に試したことがないので、すぐに構成を調べました。

Sphinxドキュメント+ sphinxcontribbuilder



Sphinx（同じ名前の検索インデックスと混同しないでください）は、Pythonで記述され、コミュニティで積極的に使用されているドキュメントジェネレーターです。他の環境でも同様に機能します。

私はそれを詳細に設定することはしませんが、私は箱から出して静的なHTML、man、pdf、および他のいくつかのフォーマットを生成できることを予約します。リポジトリでの正しいアセンブリとパブリケーションには、index.rstファイル（メインページのマークアップ）が必要です。 conf.py（構成ファイル）とMakefile（フォーマットの生成プロセスを記述したファイル。ここでは、それをdockerに縫い込み、そこでsphinx-buildコマンドを実行することが可能です）。

Sphinxはそのままの状態で* .rst形式（RestructuredText）の軽量マークアップからドックを生成できますが、より快適な開発者向けにMarkdown（CommonMarkフレーバー）で書き込む機能を追加しました（MDをRSTに変換するm2r拡張機能がこれを助けてくれました） 。

Sphinxの環境全体をすでにセットアップしており、ドキュメントアセンブリはJenkins パイプラインの別のステージに配線されているため、先に進んで、Confluenceのネイティブ形式でドックを収集して公開できるsphinxcontrib.confluencebuilder拡張機能を使用しました。この場合のConfluenceは、HTMLとともに、ドキュメント出力形式の1つです。



これを機能させるには、conf.pyの拡張機能を接続する必要があります。以下は構成の一部です。

extensions = [
    'sphinxcontrib.confluencebuilder',
    'm2r'
]

templates_path = ['_templates']

source_suffix = ['.rst', '.md']

master_doc = 'index'

exclude_patterns = [
    u'docs/warning-plate.rst',
    u'FEATURE.md',
    u'CHANGELOG.md',
    u'builder/README.md'
]

次に、拡張機能を構成します。これには一連の設定があります。

confluence_publish = True
#      Confluence
confluence_space_name = 'YOURSPACEKEY'
# ,        
confluence_parent_page = 'Raw Documentation'
#  ,     
confluence_server_url/confluence_cloud_url = 'https://yourconfluence.site.net/'
#   Confluence
confluence_publish_prefix = 'WIP-'
#    ,    
confluence_publish_postfix = '-postfix'
#     
confluence_header/footer_file
#   ,    /  (  include   , ,  ,      ,     )
confluence_page_hierarchy = True
#   -  ,                  
confluence_purge
#  ,      ,          ,     -        ,   
confluence_remove_title
#   title
confluence_publish_subset
#     
confluence_max_doc_depth
#    ,        Confluence
confluence_prev_next_buttons_location = 'top'
#     , 

confluence_server_user = os.getenv("CONFLUENCE_USERNAME", "confluence-bot")
confluence_server_pass = os.getenv("CONFLUENCE_PASSWORD", "")

target = os.getenv("TARGET", "")

if target == "CONFLUENCE":
    confluence_publish_prefix = ''
    confluence_parent_page = 'Your Space'
#            ,    WIP-,       ,     ""

重要な点は、ページ（.rstのソース）がtocで指定されておらず、exclude_patternsに追加されていない場合でも、公開されますが、階層の外にあるということです。
Confluenceのページの名前は、ページの最初のタイトルに対応します。たとえば、example.rstファイルの例の見出しに等号の下線が付いている場合は、Confluenceのページの名前になります。

非常に明白な衛生規則ですが、ドキュメントを発行する認証データを使用してボットを作成し、パイプラインで使用されるdocker composeの環境変数の形式で転送できます。

もちろん、落とし穴もあります。まず、すべてのRST構文がConfluenceでの公開をサポートしているわけではありません（╯°□°）╯︵┻━┻）。これは、1つのソースからHTMLとConfluenceを収集する場合に不便です。コンテナー、hlistディレクティブはサポートされていません。たとえば、ブロックコードの行の強調表示、目次の番号付け、listtableの配置と幅など、ほとんどすべてのディレクティブ属性はサポートされていません。サポートされているもののリストはかなり良いです。

快適なインクルードがサポートされています。これにより、異なるドキュメント間でコンテンツのフラグメントを再利用したり、コードからドキュメントをアセンブルするためのautodoc、数式用の数学、jiraからチケットやフィルターを描画したりすることができます（このため、構成にJiraサーバーを登録する必要があります）、番号付きヘッダーなどもう1つは、文字通り1月3日に大きなアップデートを投げたということです。

ちなみに、Jiraのサポートは、バージョン2.7.3以降、Pandocマルチコンバーターに登場しました。Pandocは、対応するConfluence Wikiマークアップをサポートしました。

パブリケーションをすぐに「製品」にするのではなく、ローカルリポジトリからテストページに構成する必要があるのはなぜですか。実際のところ、公開時にはすべてのページが毎回再公開され、手動での変更やコメントが崩れます。そのため、ドキュメントが進行中の場合は、公開モードをレビューに追加してコメントを収集するために、ドキュメントを開発モードなどの別のページで公開することにしました。

CIでは、パブリケーションはJenkinsパイプラインの個別のステージとして実装され、このステージ内で、Dockerイメージがリモートレジストリで起動され、目的の構成でsphinx-buildが実装されます。このステップはすぐにスキップすることをお勧めします。



pipeline {
    agent {
        label "${AGENT_LABEL}"
    }
        stage("Documentation") {
            steps { ansiColor('xterm') {
                withCredentials([usernamePassword(
                    credentialsId: "${DOCUMENTATION_BOT}",
                    usernameVariable: 'CONFLUENCE_USERNAME',
                    passwordVariable: 'CONFLUENCE_PASSWORD'
                )]) {
                    sh "docker-compose -p $COMPOSE_ID run sphinx-doc confluence"
                }
            }}
        }

ステージ内で、docker-compose -p release-branch-name run sphinx-doc confluenceが実際に起動されます。次に、Jenkinsfileは、依存関係と、ステップが実行される環境、ターゲット内の情報を組み立てて更新するプロセスについて説明します。これまでのテストから、doc8とmarkdownlinterを使用した.mdと.rstの構文チェックのみが行われました。

別のニュアンス：ページsbsetを公開するたびに、Sphinxは各ページのツリー全体を更新します。つまり、コンテンツが変更されていない場合でも変更が作成され、チャネルで通知が構成されている場合は、通知が大量に詰まります。

さらにいくつかの方法


バックエンドとしてConfluenceを持つFoliant


Mkdocsと多数のプリプロセッサーを備えたFoliantドキュメンテーションツールで、Confluenceの形で内部とバックエンドの下にあります。詳細はこちらで読むことができますが、簡単に言うと、pandocを使用してmdをHTMLに変換し、Confluenceで公開します。バックエンドを構成し、環境に依存関係としてpandocをインストールするだけで済みます。

最初のソリューションとの好ましい違い：インラインコメントは、ページが公開される前と同じ場所に復元でき、構成でそれらを設定してページを作成し、名前を編集し、コンテンツを既存のページに挿入することもできます。これには手動で設定する必要があります合流状態のページ上の葉状アンカー。

メトロ


GoogleドキュメントからSalesforce Quipまで、およびMarkdownでも、Confluenceでさまざまなソース形式を公開するマルチツール。

公開するには、manifest.jsonファイルを.mdファイルが置かれているフォルダーに配置し、その中のフォルダー、公開するファイルを指定する必要があります。各ファイルについて、confluenceページIDを指定します。ページタイトルは、ファイル（＃）の最初のタイトルになります。このツールには、Markdownマークアップを使用したいくつかの誤解があるため、ドックを確認してください。添付ファイルと画像は同じフォルダに配置する必要があり、このツールを使用すると、目次の使用を設定で直接指定することもできます。

Gem md2conf


Ruby gem md2confは、MarkdownをConfluence XHTMLのネイティブに変換します。次に、Rakeタスクを作成し、Gitlab CI / Jenkinsを介して呼び出してマスターにプッシュし、Confluence APIをプルしてページを公開できます。Ruby環境を使用しないようにするには、このgemの依存関係をコンテナーにラップします。
