モバイル開発でクロスプラットフォームソリューションを拒否した理由


何か共有させてください。クロスプラットフォーム開発のアイデアが好きです。私のすべてのタスクに1つのツールセットを使用する機能は夢です。1つのツールだけを使用してタスクを正常に完了したくないのは誰ですか？一度書いて、どこでも走りますか？したい！


ユートピア社会の考え方も好きです。誰もが他者の信念を尊重し、有意義/論理的/合理的な議論を持ち、彼らが昨日よりも良くなるように課題を設定する社会。しかし、現実世界がめったにないので、完璧です。

認めておきたいのですが、ネイティブ、クロスプラットフォーム、およびハイブリッドアプリケーション開発の間の論争について長い間話し合いたかったのです。動機の欠如のためではなく、ある日、組み合わされたツールのセットが表示され、モバイルアプリケーションを作成するための最良の価値を提供するという希望または欲求のためです。私はまだこの日を待っています。この記事は、私のネイティブ開発への愛着を説明するのに役立ちます。

ネイティブ、クロスプラットフォーム、ハイブリッド？何？


各タイプの開発の違いについて詳しく説明することはできますが、それらの間の比較は非常に長い間行うことができるため、この部分を短くするように努めます。

ネイティブ開発


ネイティブ開発では、iOSまたはAndroid用のアプリケーションを作成するための専用ツールを使用します。つまり、プラットフォームごとに1つのコードベースがあります。ツールキットには通常、Swift / Objective-C + Xcode / AppCode for iOSおよびKotlin / Java + Android Studio for Androidが含まれます。

クロスプラットフォーム開発


ネイティブコンポーネントであるかのようにコードを変換、コンパイル、実行するプラットフォーム/ツールを使用します。プロセスは、使用するツールによって異なりますが、最終的には、同じコードが実行され、AndroidプラットフォームとiOSプラットフォームでほぼ同じように機能します。

ツールの例：Xamarin、React Native、Flutter

ハイブリッド開発


ハイブリッド開発では、WebViewを起動するプラットフォーム/ツールを使用して、アプリケーションをホストします。基本的に、これは、アプリケーションが独自のアプリケーションから実行されるWebサイトであることを意味します。プラットフォーム/ツールは通常、プラグインを介して、カメラなどの独自の機能へのアクセスも提供します。

楽器の例：CordovaおよびIonic

主要機能と必須機能


私の選択を正しく理解するには、2つの異なるタイプのツール特性を説明する必要があります。

長年にわたり、あるツールが他のツールよりも優れている理由については議論が交わされてきました。ほとんどの記事では、コードベースのサイズ、開発時間、アプリケーションを開発できるプラットフォームの数など、ツールの明らかな機能について説明しています。これらの比較は、各ツールボックスの「主な特徴」と呼んでいます。このツールの主な特徴は、他のツールと比較するのに最も直接的です。通常、経験的に違い（つまり、開発速度、ビルド時間など）を確認できるためです。これらの特性は通常、ほとんどの人にとって関心事ですが、本質を明らかにするものではありません。

ツールを選択した結果を完全に理解するには、「固有の特性」を考慮する必要があります。これらの特性は一般的にはより微妙であり、その影響は主要な特性よりも評価が困難です。残念ながら、これらの特性の影響についての理解は、通常、ツールが選択され、大きな資金がそれに投資された後にのみ発生します。これらの特性は、ドキュメントの完全性と明確性、便利なツールの可用性などです。
開発ツールを適切に決定するには、その固有の特性を理解する必要があります。


ネイティブ開発を選択する理由


長年にわたるソフトウェア開発の中で、さまざまなプロジェクトでいくつかのツールを使用する機会がありました。通常、開発ツールの選択は、管理エンジニアまたは技術エンジニアによって行われました。フリーランスに切り替えた途端、技術スタックを決めることにしました。

最後に、私の最終的な決定は、ツールの要因だけでなく、固有の特性がさまざまな状況でプロジェクトにどのように影響するかを検討することでした。これが私の結論です。

1.一定の人気


2013年に初めてプログラミングを始めたときは、誇大広告ボーカルのCordovaとAppceleratorがたくさんありました。数年後、HypeはXamarinに切り替えました。数年後、それはすでにReact Nativeでした。現在？フラッターは人気を得ています。

このツールの人気は、ツールが受けるサポートにも表れていることを発見しました。ツールが民間企業のものである場合、人気は販売と同等であり、サポートコストの増加につながります。ツールがオープンソースである場合、人気はコードベースを改善しているアクティブな開発者の数に依存します。人気は行き来し、このツールに関連する開発エコシステム全体に影響を与える可能性があります。
さまざまなクロスプラットフォームツールを時系列で示したこのGoogleトレンドグラフをご覧ください。


Googleトレンドのグラフには、さまざまなクロスプラットフォームモバイルテクノロジーの検索クエリ数の結果が大まかに表示されます。
このグラフは人気の大まかな見積もりですが（検索数の結果のみを表しているため）、開発者として経験したことに対する洞察を提供します。企業や個人は新しいものを使用する傾向があります。これは必ずしもツールが良いか悪いかを意味するわけではありません（多くの記事がリリース日に関係なく賛成/反対するため）ので、人々は常に新しいものに切り替えます。

ツールがリリースされると、通常は開発コミュニティによって宣伝され、「未来」と見なされます。これにより、古いツールのサポートが少なくなり、時間の経過とともにゆっくりと萎縮する可能性があります。

これは、サポートを受けることが保証されているネイティブ開発とは対照的です。 AppleはiOSをサポートし、GoogleはAndroidをサポートしていますが、ネイティブ開発が常に関係します。これが、ネイティブ開発に開発者の最大のエコシステムがある理由です。それはもっと長く、その人気は一貫していた。

必要なもう1つの考慮事項は、プロジェクトの移管です。次の開発者がネイティブ開発の分野で少なくともある程度の経験を持つ可能性は非常に高いです。彼らが主にネイティブツールで動作しない場合でも、ほとんどの開発者は少なくとも何らかの形でこれらのツールに関係しており、クロスプラットフォームソリューションではほとんどありません。

2.ビジネスバリュー/ビジョン


開発ツールを比較するときに、多くの人がビジネス価値について話し合うことはほとんどありません。私はこれについてさまざまな開発者や技術ビジネスの所有者にさえ尋ねましたが、ほとんどの場合、私が受け取ったのは空っぽな顔つきでした。ビジネス価値が特定のツールとどのように関連付けられているか、そしてどのようにしてその価値を提供できるかを理解することが重要です。

たとえば、Xamarinを取り上げます。 2016年2月、マイクロソフトはXamarinを約4億〜5億ドルで購入しました。当時のマイクロソフトのビジョンは、主にモバイルアプリケーションの開発に関連していた。彼らはこのツールで大幅な進歩を遂げ、Visual StudioおよびVisual Studio for Macの追加設定なしのアドインとして含めました。数年前に進み、マイクロソフトのビジョンは人工知能にシフトしました。

このビジョンの変化を通して、私はXamarinとのプロジェクトに取り組んできました。この間、私は、安定したツールセットから徐々に機能が低下するツールに切り替えなければならないプラットフォームに慣れました。ツールの各アップデートは、少なくとも何も壊さないという試みのように見えました。最良のケースでは、すべてが正常に機能しました。最悪のシナリオでは、更新を数時間または1日か2日ロールバックして、環境を正常な状態に戻そうとしました。

このゆっくりとした低下はXamarinホームページ（理解可能）には記載されていませんが、フォーラムで情報を探すと、時間の経過とともに何かが変わったように感じることがあります。従来はそのままで機能していたものに関するメッセージが機能しなくなった、またはこのツールを使用した開発者のエクスペリエンスがどのように変わったかについてのメッセージ。これはすべて、生のクロックのためだけでなく、コードの品質のためにも、開発コストに直接影響します。

新しいツールへの移行中のラップは、間接費の増加と利益の損失につながります


ここでネイティブ開発にも問題があることを認めます。たとえば、Appleは通常、開発者に特定の方法（たとえば、StorybordsまたはMVCテンプレートを使用して）での開発を試みます。ツールキットも必ずしも完璧ではないので、実際にはほとんどのiOS開発にAppCodeを使用しています。それにもかかわらず、AppleとGoogleは、開発者がネイティブソリューションを快適に開発できるようにするために膨大な時間とお金を費やしています。Android Jetpack、SwiftUI、またはKotlinとSwiftに切り替えるだけです。

結局のところ、GoogleやAppleは世界で最高のビジョンを持っていませんが、それは一貫しています。企業が一晩でツールのサポートを拒否する可能性はごくわずかです。

3.時間の節約


ああ、時間を節約できます。発達効果の聖杯。あらゆる開発ツールに関するこの激論されているトピックは、ビジネスを創出または破壊する可能性があります。このトピックは、コストに直接影響するため、あらゆるプロジェクトの最前線にあります。しかし、一歩下がって、自分自身に質問してみましょう。

時間を節約するために何ができますか？


説明します。ビジネスソフトウェアと同様に、問題を解決するにはいくつかの方法があります。決定は主に、特定の個人または企業の状況に基づいて行われます。つまり、「1つのソリューションですべてに対応できる」というアプローチは危険です。それはまた、「私たちはいつもやっていた」という考え方に非常に近くなります。残念ながら、モバイル開発に関しては、時間を節約するための一般的なアプローチは、クロスプラットフォームツールを選択することです。しかし、それはとても良いですか？たとえば、問題を解決する別の方法を考え出すことはできますか？ビジネスでは、「人はプロセスより上、ツールより上にある」という言い回しがあります。
プロセスの上の人、ツールの上の人


これはまさに決定を下すときに考慮すべきことです。ツールの選択からプロセス/手順の作成、または開発への異なるアプローチの作成までの開発時間はどのようになっていますか？

ツールは開発中に悪い考え方を解決できますか？

ツールは開発プロセスを合理化しますか？

わずかに変更されたプロセスまたは考え方が、より良い時間節約の結果につながりますか？

インフラストラクチャを使用して問題を解決できますか？

これらはすべて正しい質問であり、ソリューションの複雑さも示しています。ネイティブ開発ではなく、次のクロスプラットフォームフレームワークを選択するほど簡単ではありません。あなたやあなたのビジネスに何が適していますか？あなただけがこの選択をすることができます。

ネイティブ開発ではデフォルトで高品質の製品が生成され、ほぼ無期限にサポートされるため、このツールセットを選択します。私の個人的な価値観がこの選択を決定します（卓越性のために努力します）。独自の開発には時間がかかります（どれだけ変化するか）。時間を節約する方法はありますか？時間を節約すべきでしょうか？予算が問題にならないプロジェクトを検索するだけですか？

時間節約の問題を解決する1つの方法は、再利用可能なコードを作成することです。高品質で再利用可能なコードを作成し、複数のプロジェクトにすばやく統合して展開できます。私は耐久性のためにツールキットを選択したので、私の再利用可能なコードがしばらくの間使用可能であると確信できます。

4.その他の推論


最終的に、ネイティブ開発を選択する前に、他の多くの結論を出しました。その多くは、キャリア全体でさまざまなツール、プロセス、アプローチを試してきました。これらすべての説明は、この記事を信じられないほど膨大なものにします。したがって、簡潔にするために、それらのいくつかの論文を示します。

2.サポートインフラストラクチャ

8.プラグインに対するプラットフォームの依存性
9.ツールを使用するときの開発者の幸福
10.設計コスト：両方のプラットフォームをサポートするような方法で設計する必要性
11.カスタムプロジェクト構造
12.ナレッジリソース-必要なナレッジエリアの数（たとえば、Xamarinに必要なもの：+ Xamarin Apiプラットフォーム、C＃、特定のAndroid機能、特定のiOS機能、各プラットフォームの機能に関する知識、およびXamarinを介した相互作用の方法。）
13.ライセンス+コスト
14.モバイルオペレーティングシステムの状況。たとえば、主要なプラットフォームが2つだけ続くのはどれくらいですか？
