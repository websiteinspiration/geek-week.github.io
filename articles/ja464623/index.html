ORの代わりにUnionを使用する

クエリを少し変更することで、クエリが遅い場合は修正できる場合があります。このような例の1つは、ORまたはIN演算子を使用してWHERE句で複数の値を比較する場合に説明できます。多くの場合、ORはインデックスまたはテーブルのスキャンを引き起こす可能性があり、I / O消費または全体的なクエリ速度の観点からは、好ましい実行計画ではない場合があります。

クエリオプティマイザが実行プランを作成するとき、多くの変数が関係します。これらの変数には、多くのハードウェア特性、インスタンス設定、データベース設定、統計（テーブル、インデックス、自動生成）、およびクエリを書き込む方法が含まれます。ここで、リクエストの記述方法を変更します。2つの異なるクエリが同じ結果を返す可能性がある場合でも、これがどのように予期せず見えても、クエリの形式によっては、それらがたどるパスが完全に異なる場合があります。

UNION vs OR


SQL Serverに関する私の経験のほとんどについて、ORは通常UNIONよりも効率が良くありません。ORで通常発生するのは、スキャンを引き起こすことが多いことです。これは場合によっては最良の方法である場合があり、別の記事として残しますが、一般に、多数のエントリが影響を受ける場合、これが速度低下の主な理由であることがわかりました。それでは、比較を始めましょう。

以下はORステートメントです。

SELECT SalesOrderID, *
FROM sales.SalesOrderDetail
WHERE ProductID = 750 OR ProductID = 953



この実行計画から、121,000行をスキャンしていることがわかります。（行数は表示されませんが、表示されます）。

同じリクエストを実行しますが、ORではなくUNIONを使用して記述します。

SELECT [SalesOrderID], *
FROM sales.SalesOrderDetail
WHERE ProductID = 750
UNION
SELECT [SalesOrderID], *
FROM sales.SalesOrderDetail
WHERE ProductID = 953



ここでは、2つのオペレーションブランチを示しています。1つのブランチは358ラインに影響し、他の346ラインは影響します。両方のブランチは、両方の結果セットを組み合わせる連結演算を実行することがわかります。2つの個別の検索がありますが、必要なSELECTリストを取得するためのキー検索もあります。スキャン操作のすべてのラインに影響を与えたため、これはスキャン操作に必要ではなかったため、データは後ではなくスキャン中に取得されました。これは、UNIONやORではなく、必要なインデックスと行によるものです。ただし、selectはsearch vs scanを選択する際の要素でもあると言いますが、この記事ではこれを無視します。

説明


UNIONがスキャンの代わりにより多くの検索を行う理由。各操作は、検索の対象となるために特定の選択性要件を満たす必要があるためです。 （選択性は、特定のフィルターされた列の一意性です）。 ORは1回の操作で発生するため、各列の選択性が組み合わされて特定のパーセンテージを超えると、スキャンがより効率的になります。

UNIONはデフォルトで演算子ごとに個別の操作を実行するため、各列の選択性は結合されず、検索を実行する可能性が高くなります。 UNIONは2つの演算を実行するため、上記の連結演算を使用して結果セットを一致させる必要があります。これは通常、高価な操作ではありません。

OR句はINステートメントと同じように機能することにも注意してください。
