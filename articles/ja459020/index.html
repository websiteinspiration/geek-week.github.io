DFSRが一部のファイルを複製しない理由とその対処方法


多くの人が知っているように、レプリケートフォルダーのプロパティでファイルマスクの形式で例外を構成できます。その場合、サービスは指定されたマスクに一致するファイルをレプリケートしません。ただし、ファイルに「一時的な」属性があることを誰もが知っているわけではなく、DFSRはそのようなファイルを設計上処理しません。これが考慮されていない場合、DFSRフォルダーの内容が同期しなくなる可能性がありますが、サービスログはすべてクリーンで美しく、最も不都合なときに表示される可能性があります。問題自体とその解決策はすでにインターネットで複数回分析されています。この記事の目的は、以前に作成された解決策を完成させ、柔軟性と利便性を追加することです。それが関係する人-私は猫の下で尋ねます。

このトピックは、公式ブログ、ここ、Habr自体など、インターネットで2回以上取り上げられています。したがって、私は繰り返しませんが、要点に直行します。元のソリューションは、指定されたフォルダー内のすべてのファイルの「一時」属性をリセットするPowerShellスクリプトです。私はそれをベースにして、ターゲットサーバー上のレプリケートフォルダーを見つけて各フォルダーを実行し、問題の属性をリセットして、メールで見つかったファイルに関するレポートを送信する独自​​のバージョンを作成しました。長いパスのサポートも追加されました（PowerShellの5番目のバージョンが必要です）。

スクリプトを機能させるには、DFSサービス管理ツールが必要です（デフォルトでは、DFSレプリケーションの役割が追加されてインストールされます）。移行先サーバーで利用できない場合は、次のコマンドレットを実行します。

Install-WindowsFeature RSAT-DFS-Mgmt-Con

そしてここにスクリプト自体があります：

TempAttrFixer.ps1
Param(
    #  
    [parameter(Mandatory=$false)][String]$OutDir
)

$SMTPServer = "mail.mydomain.com"
$MailFrom = "sender@mydomain.com"
$MailTo = "recipient@mydomain.com"

Function ConvertTo-LiteralPath {
    #      LiteralPath
    Param(
        [parameter(Mandatory=$true)][String]$Path
    )
    #    UNC
    If ($Path.Substring(0,2) -eq "\\") {
        Return ("\\?\UNC" + $Path.Remove(0,1))
    }
    Else {
        Return "\\?\$Path"
    }
}

$StartTime = Get-Date

$Error.Clear()

If (!$OutDir) {
    $OutDir = (Get-WmiObject Win32_OperatingSystem).SystemDrive +"\TempAttrFixer_Report"
}

#   ,   
$FoldersToScan = @(Get-DfsrMembership -ComputerName $env:COMPUTERNAME | Sort-Object GroupName, FolderName).ContentPath

$LogFileName = "$env:COMPUTERNAME" + "_TempFiles_" + (Get-Date -Format "yyyy-MM-dd-HH-mm-ss") + ".csv"
$LogFilePath = "$OutDir\$LogFileName"
$Delimiter = "`t"
$FilesCount = 0
If (!(Test-Path $OutDir -PathType Container)) {
    New-Item -ItemType Directory -Force -Path $OutDir | Out-Null
}

ForEach ($Folder in $FoldersToScan) {
    # PowerShell 5.1,      LireralPath
    Write-Output "Scanning `"$Folder`"..."
    Get-ChildItem -LiteralPath (ConvertTo-LiteralPath $Folder) -Recurse | ForEach-Object -Process {
        if (($_.Attributes -band 0x100) -eq 0x100) {
            $FilesCount += 1
            $Entry = $_.FullName + $Delimiter + $_.GetAccessControl().Owner + "`r`n"
            $Entry
            $Entry | Out-File -FilePath $LogFilePath -Encoding unicode -Append -NoNewline
            $_.Attributes = ($_.Attributes -band 0xFEFF)
        }
    }
}

$FinishTime = Get-Date
$TimeSpan = $FinishTime - $StartTime
Write-Output ("Done, errors: " + $Error.Count)

$Encoding = [System.Text.Encoding]::Unicode

If ($FilesCount -gt 0) {
    $MessageBody = $null
    $Subject = "     "
    $MessageBody += " : $env:COMPUTERNAME`r`n"
    $MessageBody +=  "  : " + $TimeSpan.ToString() + "`r`n"
    $MessageBody += "   `"$LogFilePath`"`r`n"
    Send-MailMessage -SmtpServer $SMTPServer -From $MailFrom -Subject $Subject -Encoding $Encoding -To $MailTo -Body $MessageBody -Attachments $LogFilePath
}
Else {
    $MessageBody = $null
    $Subject = "      "
    $MessageBody += " : $env:COMPUTERNAME`r`n"
    $MessageBody +=  "  : " + $TimeSpan.ToString() + "`r`n"
    Send-MailMessage -SmtpServer $SMTPServer -From $MailFrom -Subject $Subject -Encoding $Encoding -To $MailTo -Body $MessageBody
}

#,  -   
If ($Error.Count -gt 0) {
    $MessageBody = $null
    $Subject = "     "
    $MessageBody += " : $env:COMPUTERNAME`r`n"
    $MessageBody += " : $env:UserName`r`n"
    $MessageBody += " :`r`n"
    $MessageBody += $Error[0]
    Send-MailMessage -SmtpServer $SMTPServer -From $MailFrom -Subject $Subject -Encoding $Encoding -To $MailTo -Body $MessageBody
}


メールアラートを担当する変数の値を修正し、スクリプトをスケジューラに追加します。

powershell.exe -NoLogo -ExecutionPolicy Bypass -NoProfile -File "<  >\TempAttrFixer.ps1"

SYSTEM内からタスクを実行するのが最も簡単ですが、パラノイアのレベルが高く、情報セキュリティに真剣に取り組んでいる場合は、個別のアカウントを使用して、バッチジョブとしてログインする権限と、レプリケートフォルダー内のファイルを変更する権限を付与できます。

電子メールレポートの代わりに（またはそれらに加えて）、zabbixまたは別の監視システムと対話するようにスクリプトを構成できます。また、プロシージャはリソースを大量に消費するため、あまり頻繁に実行しないでください。 10TBのデータを処理するのに約1時間かかり、スクリプトは1日に1回、夜に実行されます。

私の同僚と私は、一時的な属性を持つファイルがどこから来るのかという問題に興味を持っていました。したがって、各レポートでは、ファイルの完全な名前とともに、そのNTFS所有者が表示されます。収集されたデータに基づいて、RDPを介して転送されたローカルディスクからデータをコピーすると、この属性がファイルに追加される場合があることがわかりました（リモートデスクトップテクノロジーを積極的に使用しています）。しかし、ファイルに最初は「欠陥がある」可能性があります。これまでのところ、これ以上詳しく知ることはできません。

簡単なアンケートに合格して、コメントで経験を共有していただければ幸いです。
