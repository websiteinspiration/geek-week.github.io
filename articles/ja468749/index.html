Android LiveDataイベント

LiveDataは、データとオブジェクトの状態をライフサイクル（LifecycleOwner、通常はフラグメントまたはアクティビティ）にリンクするための優れたツールです。

通常、LiveDataはViewModelに配置され、UIの状態を更新するために使用されます。多くの場合、ViewModelはLifecycleOwnerを存続し、LiveData状態を維持できます。このようなメカニズムは、構成の変更後など、データを保存してしばらくしてから復元する必要がある場合に適しています。

しかし、状態ではなくイベントのメカニズムを使用したい場合はどうでしょうか。また、ブラウザーのライフサイクル（LifecycleOwner）のコンテキストで必要です。たとえば、LifecycleOwnerがまだアクティブで、アクティブなブラウザーがあり、UIを更新する準備ができている場合、非同期操作の後にメッセージを表示する必要があります。LiveDataを使用する場合、構成を変更するたびに、または新しいサブスクライバーごとに同じメッセージを受け取ります。それ自体を示唆するソリューションの1つは、一部のブラウザーでデータを処理した後、LiveDataでデータを処理することです。

たとえば、次のようなコード：

Observer {
	handle(it)
	yourViewModel.liveData.value = null
}

しかし、このアプローチにはいくつかの欠点があり、必要なすべての要件を満たしていません。

次のようなイベントメカニズムが必要です。

  1. アクティブなサブスクライバーのみに通知します
  2. サブスクリプション時に以前のデータについて通知しません、
  3. イベントの処理を中断するために、handledフラグをtrueに設定する機能があります。

上記のプロパティをすべて備え、通常のLiveDataのように機能するMutableLiveEventクラスを実装しました。

使い方：

//   EventArgs       
class MyIntEventArgs(data: Int) : EventArgs<Int>(data)

//  viewModel
class MainViewModel : ViewModel() {

    private val myEventMutable = MutableLiveEvent<MyIntEventArgs>()
    val myEvent = myEventMutable as LiveData<MyIntEventArgs>

    fun sendEvent(data: Int) {
        myEventMutable.value = MyIntEventArgs(data)
    }
}

val vm = ViewModelProviders.of(this).get(MainViewModel::class.java)

vm.myEvent.observe(this, Observer {
   //  
   /*
   *   ,    , 
   *      ,   handled = true
   */
   it.handled = true
})

すべてのコードはGitHubで入手できます。以下では、実装について少し説明します。

class MutableLiveEvent<T : EventArgs<Any>> : MutableLiveData<T>() {

    internal val observers = ArraySet<PendingObserver<in T>>()

    @MainThread
    override fun observe(owner: LifecycleOwner, observer: Observer<in T>) {
        val wrapper = PendingObserver(observer)
        observers.add(wrapper)

        super.observe(owner, wrapper)
    }

    override fun observeForever(observer: Observer<in T>) {
        val wrapper = PendingObserver(observer)
        observers.add(wrapper)

        super.observeForever(observer)
    }

    @MainThread
    override fun removeObserver(observer: Observer<in T>) {

        when (observer) {
            is PendingObserver -> {
                observers.remove(observer)
                super.removeObserver(observer)
            }
            else -> {
                val pendingObserver = observers.firstOrNull { it.wrappedObserver == observer }
                if (pendingObserver != null) {
                    observers.remove(pendingObserver)
                    super.removeObserver(pendingObserver)
                }
            }
        }
    }

    @MainThread
    override fun setValue(event: T?) {
        observers.forEach { it.awaitValue() }
        super.setValue(event)
    }
}

アイデアは、MutableLiveEventクラス内のobserveメソッドとobserveForeverメソッドで、ブラウザーを特別な内部クラスPendingObserverでラップするというものです。

internal class PendingObserver<T : EventArgs<Any>>(val wrappedObserver: Observer<in T>) : Observer<T> {

    private var pending = false

    override fun onChanged(event: T?) {
        if (pending && event?.handled != true) {
            pending = false
            wrappedObserver.onChanged(event)
        }
    }

    fun awaitValue() {
        pending = true
    }
}

PendingObserverでは、保留フラグはデフォルトでfalseに設定されています。これは、リストの項目2（古いデータを通知しない）を解決します。

そして、MutableLiveEventのコード

override fun setValue(event: T?) {
        observers.forEach { it.awaitValue() }
        super.setValue(event)
}

最初にpendingをtrueに設定し、その後、それ自体のデータを更新します。これにより、クレーム1の実装が保証されます。（アクティブなサブスクライバーのみをアラートします）。

私がまだ話していない最後のポイントは、EventArgsです。このクラスは、イベントのさらなる処理を中断するための処理済みフラグがある一般化です（条項3）。

open class EventArgs<out T>(private val content: T?) {

    var handled: Boolean = false

    val data: T?
        get() {
            return if (handled) {
                null
            } else {
                content
            }
        }
}
