車輪を再発明することが有用である理由



先日、先輩と名乗るJavaScript開発者にインタビューをしました。面接にも同席した同僚が候補者にHTTPリクエストを生成する関数を作成するよう依頼し、失敗した場合は数回再試行しました。

彼はボードにすぐにコードを書いたので、おおよそのことを描写するのに十分でしょう。彼が問題の本質が何であるかをよく理解していることを単に示しただけで、私たちは完全に満足するでしょう。しかし、残念ながら、彼は成功する解決策を見つけることができませんでした。次に、これを興奮として書き留め、タスクを少し楽にすることを決め、コールバックを持つ関数からのプロミスに基づいて関数を作成するように依頼しました。

しかし悲しいかな。はい、彼が以前に同様のコードに出くわしたことは明らかでした。彼はすべてがそこでどのように機能するかを一般的な用語で知っていました。概念の理解を示すソリューションのスケッチで十分です。しかし、候補者が理事会に書いたコードはまったくナンセンスでした。彼はJavaScriptの約束が何であるかについて非常に曖昧な考えを持っていました、そして彼はそれらがなぜ必要なのかを本当に説明することができませんでした。ジュニアにとってはこれは言い訳になりませんでしたが、シニアの立場はもはや引き分けられませんでした。この開発者は、約束のある複雑なチェーンのバグをどのように排除し、他の人に彼が正確に何をしたのかをどのように説明しますか？

開発者は既製のコードを自明だと考えています


開発の過程で、私たちは常に再現可能な素材に出会います。コードフラグメントを転送するので、繰り返し書き込む必要はありません。したがって、重要な部分にすべての注意を集中させ、私たちが作業する完成したコードを自明なものとして見ていきます-私たちはすべてがその中で正常に機能することを前提としています。

そして、通常それは実際に機能しますが、困難が生じた場合、そのメカニズムの理解は実を結ぶだけではありません。

したがって、上級開発者のポジションの候補者は、promiseオブジェクトは自明であると考えました。彼はおそらく彼らが他の誰かのコードのどこかで会ったときにそれらに対処する方法を想像しましたが、彼は一般原則を理解しておらず、インタビューでそれを繰り返すことはできませんでした。おそらく彼はその断片を暗記していました。それほど難しくはありません。

return new Promise((resolve, reject) => {
  functionWithCallback((err, result) => {
   return err ? reject(err) : resolve(result);
  });
});

私もそれをやった-はい、私たちはおそらくいつかやった。彼らは単にコードの一部を記憶しただけなので、後でそれを仕事で使用できるようになりますが、一般的な用語ではすべてがそこでどのように機能するかを想像するだけです。ただし、開発者がコンセプトを本当に理解していれば、何も覚える必要はありません。その方法を知っているだけで、コードに必要なすべてを簡単に再現できます。

ルーツへ


フロントエンドフレームワークの優位性がまだ確立されていない2012年、jQueryは世界を支配し、私はjQueryの作成者であるJohn Rezigが執筆した 『Secrets of the JavaScript Ninja』という本を読みました。

この本は読者が独自のjQueryをゼロから作成する方法を教え、ライブラリーの作成につながった一連の思考に参加するユニークな機会を提供します。近年、jQueryは以前の人気を失いましたが、それでもこの本を強くお勧めします。彼女について私を最も驚かせたのは、私はこれらすべてを自分自身で考えることができる永続的な感情でした。筆者が描いた手順は非常に論理的でわかりやすかったので、ビジネスに取り掛かったとしても、jQueryを簡単に作成できると真剣に思えました。

もちろん、実際には、私はこのようなものを習得しなかったでしょう-私はそれが非常に困難であると決めたでしょう。私自身の決定は、私には単純すぎて働きがたいように思え、あきらめるでしょう。私はjQueryを自明なものに帰します。正しい操作は盲目的に信じる必要があるだけです。その後、私はこのライブラリのメカニズムを掘り下げる時間を無駄にすることはほとんどありませんでしたが、単に一種のブラックボックスとして使用しました。

しかし、この本を知ることで私は別の人になりました。私はソースコードを読み始めたところ、多くのソリューションの実装が実際には非常に透過的であり、明白であることもわかりました。もちろん、そのようなことを自分で考えることはできません。これは別のオペラからのものです。しかし、他人のコードの研究と既存のソリューションの再現により、私たち自身の何かを思い付くことができます。

あなたが描くインスピレーションと気づき始めるパターンは、開発者としてあなたを変えます。常に使用していて、魔法のアーティファクトとして考えるのに慣れている美しいライブラリは、魔法ではまったく機能せず、簡潔で機知に富んだ方法で問題を解決するだけです。

場合によっては、コードを段階的に分解してコードを細かく分解する必要がありますが、このように、小さな連続したステップで移動して、ソリューションへの作成者のパスを繰り返すことができます。これにより、コードの作成プロセスをより深く掘り下げて、独自のソリューションを検索する際の信頼性を高めることができます。

私が最初にプロミスを使い始めたとき、これは純粋な魔法のように思えました。その後、同じコールバックに基づいていることがわかり、プログラミングの世界がひっくり返った。つまり、コールバックから私たちを救うことを目的とするパターン自体は、コールバックを使用して実装されます！

これは、別の目で問題を見て、私の前にはいくつかの厄介なコードがないこと、私の人生では決して理解できない超越的な複雑さがないことを理解するのに役立ちました。これらは、好奇心と深い没入感で問題なく理解できるパターンにすぎません。これは、人々が開発者としてプログラミングおよび成長する方法を学ぶ方法です。

このホイールを再発明


ですから、自由に車輪を作り直してください。自分でデータバインディングのコードを記述したり、独自の約束を作成したり、自分の手で条件を管理することを決定したりしてください。
誰もがこれをすべて使用することは決して問題ではありませんが、今やあなたはそれを行うことができます。そして、後で自分のプロジェクトでそのような開発を使用する機会があれば、これは一般的に素晴らしいことです。あなたはそれらを開発して何か他のものを学ぶことができます。

ここでのポイントは、コードを本番環境に送ることではなく、何か新しいことを学ぶことです。既存のソリューションの実装を自分で処方することは、最高のプログラマーから学び、それによってスキルを磨くための優れた方法です。
