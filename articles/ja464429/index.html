Springプラットフォームで作業する際の最も一般的な10の間違い。パート1

みなさん、こんにちは。今日は、記事の最初の部分を共有します。その翻訳は、特にコース「Developer on the Spring Framework」の学生のために準備されました。始めましょう！




Springはおそらく最も人気のあるJava開発プラットフォームの1つです。これは強力ですが、習得が難しいツールです。基本的な概念は理解と学習がかなり簡単ですが、経験豊富なSpring開発者になるには時間と労力が必要です。

この記事では、Springでの作業中に発生する最も一般的な間違い、特にWebアプリケーションの開発とSpring Bootプラットフォームの使用に関連するいくつかの間違いについて説明します。Spring Bootのウェブサイトに記載されているように、Spring Bootは標準化されたすぐに使用できるアプリケーションを作成するためのアプローチ。この記事では、このアプローチを順守します。Spring Bootに基づく標準のWebアプリケーションの開発に効果的に使用できるいくつかの推奨事項を示します。

Spring Bootプラットフォームに精通していないが、この記事の例を試してみたい場合は、この記事の追加資料を含むGitHubリポジトリを作成しました。この記事を読んでいるときに、少し混乱している場合は、このリポジトリのクローンを作成し、コンピューターでコードを試してみることをお勧めします。

よくある間違い番号1.低すぎるプログラミングへの切り替え


「他人の開発の拒否のシンドローム」はプログラミング環境にとって非常に典型的であるため、 私たちはこの一般的な間違いに簡単に屈します。その症状の1つは、一般的に使用されるコードの一部を絶えず書き換えることであり、これは多くのプログラマーに見られる症状です。
特定のライブラリの内部構造と実装機能を研究することは、多くの場合、有用で必要かつ興味深いプロセスですが、関数の低レベルの実装を実行しながら同じタイプのコードを常に記述していると、ソフトウェア開発者にとって有害になる場合があります。これが、Springなどの抽象化とプラットフォームが存在し、使用されている理由です。これらを使用すると、手作業を繰り返す必要がなくなり、サブジェクトエリアのオブジェクトとプログラムコードロジックに高いレベルで集中することができます。

したがって、抽象化は避けてはなりません。次に特定の問題を解決する必要がある場合は、最初に簡単な検索を行って、この問題を解決するライブラリが既にSpringに組み込まれているかどうかを確認します。適切なターンキーソリューションが見つかる可能性があります。このような便利なライブラリの1つは、この記事の例で使用する注釈であるProject Lombokです。Lombokはテンプレートコードジェネレーターとして使用されるので、私たち一人一人に住んでいる怠惰な開発者は、このライブラリーに親しむことが非常にうれしいでしょう。たとえば、標準のJavaBeanコンポーネントがLombokでどのように表示されるかを確認してください。

@Getter
@Setter
@NoArgsConstructor
public class Bean implements Serializable {
    int firstBeanProperty;
    String secondBeanProperty;
}

すでにご存じかもしれませんが、上記のコードは次の形式に変換されます。

public class Bean implements Serializable {
    private int firstBeanProperty;
    private String secondBeanProperty;

    public int getFirstBeanProperty() {
        return this.firstBeanProperty;
    }

    public String getSecondBeanProperty() {
        return this.secondBeanProperty;
    }

    public void setFirstBeanProperty(int firstBeanProperty) {
        this.firstBeanProperty = firstBeanProperty;
    }

    public void setSecondBeanProperty(String secondBeanProperty) {
        this.secondBeanProperty = secondBeanProperty;
    }

    public Bean() {
    }
}

統合開発環境でLombokを使用する場合は、適切なプラグインをインストールする必要がある可能性が高いことに注意してください。このプラグインのIntelliJ IDEAバージョンは、こちらにあります。

よくある間違い番号2.内部構造の「リーク」


内部構造へのアクセスを許可することは、サービスモデルの柔軟性を損ない、その結果、悪いプログラミングスタイルの形成に寄与するため、決して良い考えではありません。内部構造の「リーク」は、データベース構造が特定のAPIエンドポイントからアクセス可能になるという事実に現れます。たとえば、次の「古き良きJavaオブジェクト」（POJO）がデータベース内のテーブルを表すとします。

@Entity
@NoArgsConstructor
@Getter
public class TopTalentEntity {

    @Id
    @GeneratedValue
    private Integer id;

    @Column
    private String name;

    public TopTalentEntity(String name) {
        this.name = name;
    }

}

オブジェクトデータにアクセスする必要があるエンドポイントがあると仮定しますTopTalentEntity。インスタンスを返すのTopTalentEntityは魅力的なアイデアのようですが、より柔軟な解決策は、APIエンドポイントのTopTalentEntityデータを表す新しいクラスを作成することです。

@AllArgsConstructor
@NoArgsConstructor
@Getter
public class TopTalentData {
    private String name;
}

したがって、データベースの内部コンポーネントを変更しても、サービスレベルで追加の変更を行う必要はありません。TopTalentEntityユーザーのパスワードハッシュをデータベースに保存するためにクラスにパスワードフィールドが追加された場合にどうなるかを見てみましょう。などのコネクタがなくTopTalentData、開発者がサービスのインターフェース部分の変更を忘れた場合、秘密情報の非常に望ましくない開示につながる可能性があります。

よくある間違い番号3.コードで分散したほうがよい関数を組み合わせる


アプリケーションコードの成長に合わせて整理することは、ますます重要なタスクになります。奇妙なことに、効果的なプログラミングの原則のほとんどは、特にアプリケーションアーキテクチャが十分に検討されていない場合、特定の規模の開発に到達すると機能しなくなります。そして最も頻繁に犯される間違いの1つは、個別に実装する方が合理的なコード内の関数の組み合わせです。責任分担の

原則に違反した理由通常は、既存のクラスに新しい機能を追加します。おそらくこれは良い一時的な解決策です（特に、必要なコードの量が少なくなります）が、将来的には、コードのテストと保守の段階やそれらの間など、問題になることは避けられません。TopTalentDataリポジトリから戻る次のコントローラーを考えてみます。

@RestController
public class TopTalentController {

    private final TopTalentRepository topTalentRepository;

    @RequestMapping("/toptal/get")
    public List<TopTalentData> getTopTalent() {
        return topTalentRepository.findAll()
                .stream()
                .map(this::entityToData)
                .collect(Collectors.toList());
    }

    private TopTalentData entityToData(TopTalentEntity topTalentEntity) {
        return new TopTalentData(topTalentEntity.getName());
    }

}

一見すると、このコードフラグメントには明らかなエラーはないようです。TopTalentDataクラスインスタンスから取得したオブジェクトのリストを提供しますTopTalentEntity。しかし、コードをさらに詳しく見ると、実際にTopTalentControllerいくつかのアクションを実行していることがわかります。つまり、リクエストを特定のエンドポイントと関連付け、リポジトリからデータを抽出し、から受け取ったオブジェクトをTopTalentRepository別の形式に変換します。よりクリーンなソリューションでは、これらの関数を個別のクラスに分離する必要があります。たとえば、次のようになります。

@RestController
@RequestMapping("/toptal")
@AllArgsConstructor
public class TopTalentController {

    private final TopTalentService topTalentService;

    @RequestMapping("/get")
    public List<TopTalentData> getTopTalent() {
        return topTalentService.getTopTalent();
    }
}

@AllArgsConstructor
@Service
public class TopTalentService {

    private final TopTalentRepository topTalentRepository;
    private final TopTalentEntityConverter topTalentEntityConverter;

    public List<TopTalentData> getTopTalent() {
        return topTalentRepository.findAll()
                .stream()
                .map(topTalentEntityConverter::toResponse)
                .collect(Collectors.toList());
    }
}

@Component
public class TopTalentEntityConverter {
    public TopTalentData toResponse(TopTalentEntity topTalentEntity) {
        return new TopTalentData(topTalentEntity.getName());
    }
}

この階層のもう1つの利点は、クラス名を見るだけで、関数がどこにあるかを理解できることです。その後、テスト中に、必要に応じて、これらのクラスのコードを代理コードに簡単に置き換えることができます。

よくある間違い番号4.均一なコードと不十分なエラー処理


コードの均一性のトピックは、Spring（または一般にJava）に固有のものではありませんが、それでも、Springでプロジェクトを操作する際に考慮しなければならない重要な側面です。特定のプログラミングスタイルの選択は議論の的となるかもしれません（通常、開発チーム内または会社全体で一貫しています）が、いずれの場合でも、コードを記述するための共通標準の存在は作業効率の向上に役立ちます。これは、複数の人がコードで作業する場合に特に当てはまります。統一されたコードは、それを維持したり、これらまたはこれらのクラスが必要な理由についての長い説明に多くの労力を費やすことなく、開発者から開発者に渡すことができます。

さまざまな構成ファイル、サービス、コントローラーがあるSpringプロジェクトがあると想像してください。名前のセマンティックの均一性を維持しながら、簡単に検索でき、開発者がコードを簡単に理解できる構造を作成します。たとえば、構成サフィックスを構成クラスの名前に追加し、サービスサフィックスをサービス名に追加し、コントローラーサフィックスをコントローラー名に追加できます。

サーバー側のエラー処理はコードの均一性と密接に関連しており、特別な注意が必要です。作成が不十分なAPIからの例外を処理したことがある場合は、これらの例外の意味を正しく理解することがどれほど難しいか、実際に発生した理由を特定することがさらに難しいことをおそらく知っています。
API開発者として、理想的には、ユーザーが使用しているすべてのエンドポイントをカバーし、それらが単一のエラーメッセージ形式を使用するようにしたいと考えています。通常、これは、標準のエラーコードとその説明を使用し、ユーザーに「500内部サーバーエラー」メッセージやスタックトレース結果を提供するなどの疑わしい決定を放棄する必要があることを意味します（ちなみに、後者のオプションは、内部データを公開しているため、絶対に避ける必要があります。さらに、そのような結果はクライアント側で処理するのが困難です）。
たとえば、エラーメッセージの一般的な形式は次のようになります。

@Value
public class ErrorResponse {

    private Integer errorCode;
    private String errorMessage;

}

これに似た形式は、最も人気のあるAPIによく見られ、簡単に体系的に文書化できるため、原則として問題なく機能します。@ExceptionHandlerメソッドに注釈を追加することで、例外をこの形式に変換できます（注釈の例については、「一般的な間違い＃6」セクションを参照してください）。

よくある間違い番号5.マルチスレッドでの正しくない作業


マルチスレッドの実装は、デスクトップアプリケーションで使用されているか、Webアプリケーションで使用されているか、Springプロジェクトで使用されているか、他のプラットフォームのプロジェクトで使用されているかに関係なく、難しい作業です。プログラムの並列実行によって引き起こされる問題を追跡することは困難であり、デバッガーを使用してそれらを処理することは、しばしば非常に困難です。並列実行エラーを処理していることを理解している場合は、エラーの根本的な原因が見つかるまで、デバッガーを放棄し、コードを手動で調べる必要があります。残念ながら、そのような問題を解決する標準的な方法はありません。いずれの場合も、状況を評価し、特定の条件で最適と思われる方法で問題を「攻撃」する必要があります。

もちろん、マルチスレッドに関連するエラーを完全に回避することが理想的です。ここには普遍的なレシピはありませんが、私はまだいくつかの実用的な推奨事項を提供できます。

グローバルステートの使用を避ける


まず、「グローバルな状態」の問題を常に思い出してください。マルチスレッドアプリケーションを開発している場合は、グローバルに変更されたすべての変数を注意深く監視し、可能であればそれらを完全に取り除く必要があります。グローバル変数が変更可能である必要がある理由がまだある場合は、同期を適切に実装し、アプリケーションのパフォーマンスを監視します。追加の待機期間が原因で速度が低下しないことを確認する必要があります。

変更可能なオブジェクトを避ける


このアイデアは、関数型プログラミングの原則に直接由来しており、OOPの原則に適合しているため、変更可能なクラスや変更可能な状態は避ける必要があると述べています。つまり、これは、メソッド（セッター）の設定を控え、すべてのモデルクラスでfinal修飾子を持つプライベートフィールドを持つ必要があることを意味します。それらの値が変更されるのは、オブジェクトが作成されたときだけです。この場合、リソースの競合に関連する問題がないことを確認でき、オブジェクトのプロパティにアクセスすると、常に正しい値が取得されます。

重要なデータを記録する


アプリケーションで問題が発生する可能性のある場所を評価し、すべての重要なデータのロギングを事前に設定します。エラーが発生した場合は、受け取った要求に関する情報を入手でき、アプリケーションの誤動作の理由をよりよく理解できます。ただし、ロギングには追加のファイルI / Oが含まれ、アプリケーションのパフォーマンスに悪影響を及ぼす可能性があるため、悪用しないでください。

既製のスレッド実装を使用する


スレッドを生成する必要がある場合（たとえば、さまざまなサービスへの非同期要求を作成する場合）、独自のスレッドを作成する代わりに、既成の安全なスレッド実装を使用します。ほとんどの場合、ExecutorService抽象化と壮大な機能抽象化CompletableFuture for Java 8 を使用してスレッドを作成できます。Springでは、DeferredResultクラスを使用して非同期リクエストを処理することもできます。
