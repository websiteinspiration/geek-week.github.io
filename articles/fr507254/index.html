Zabbix - élargir les frontières des macros

Faire une solution pour le client, il y avait 2 tâches que je voulais résoudre magnifiquement et avec la fonctionnalité Zabbix régulière.

Tâche 1. Suivi de la version actuelle du micrologiciel sur les routeurs Mikrotik.

La tâche est résolue facilement - en ajoutant un agent au modèle HTTP. L'agent reçoit la version actuelle du site Web Mikrotik et le déclencheur compare la version actuelle avec la version actuelle et, en cas de divergence, émet une alerte.

Lorsque vous avez 10 routeurs, cet algorithme n'est pas critique, mais que faire avec 3000 routeurs? Envoyer 3000 requêtes au serveur? Bien sûr, un tel schéma fonctionnera, mais l'idée de 3000 requêtes ne me convenait pas, je voulais trouver une autre solution. De plus, il y avait encore un inconvénient dans un tel algorithme: l'autre côté peut calculer autant de requêtes d'une IP pour une attaque DoS, ils peuvent simplement l'interdire.

Tâche 2Utilisation d'une session d'autorisation dans différents agents HTTP.

Lorsque, via un agent HTTP, vous devez recevoir des informations de pages "fermées", vous avez besoin d'un cookie d'autorisation. Pour cela, il existe généralement un formulaire d'autorisation standard avec une paire identifiant / mot de passe et définissant l'ID de session dans le cookie.

Mais il y a un problème, vous ne pouvez pas accéder aux données d'un autre élément à partir d'un élément de l'agent HTTP pour remplacer cette valeur dans l'en-tête.

Il existe également un «scénario Web», il a une autre limitation, il ne permet pas de recevoir du contenu pour analyse et stockage ultérieur. Vous pouvez uniquement vérifier la présence des variables nécessaires sur les pages ou passer les variables précédemment obtenues entre les étapes du script Web.

Après une petite réflexion sur ces tâches, j'ai décidé d'utiliser des macros parfaitement visibles dans n'importe quelle partie du système de surveillance: dans les modèles, les hôtes, les déclencheurs ou les éléments. Et les macros peuvent être mises à jour via l'API de l'interface Web.

Zabbix a une bonne documentation API détaillée. Pour échanger des données via l'API, le format de données Json est utilisé. Les détails peuvent être trouvés dans la documentation officielle .

La séquence d'actions pour obtenir les données dont nous avons besoin et les enregistrer dans une macro est présentée dans le diagramme ci-dessous.



Étape 1


La toute première étape peut consister en une ou plusieurs actions. Dans les premières étapes, toute la logique de base est posée, et les principales sont les 3 dernières étapes.

Dans mon exemple, la première étape a été d'obtenir les cookies d'autorisation sur le PBX pour la première tâche. Pour la deuxième tâche, j'ai obtenu le numéro de la version actuelle du firmware Mikrotik.

  • upgrade.mikrotik.com/routeros/LATEST.6fix — URL LTS

La première étape est complètement individuelle pour chaque cas et la logique de son travail peut être différente. Tout dépend de votre tâche.
Lorsque vous travaillez avec des scripts Web, gardez une trace de la méthode dont vous avez besoin pour obtenir la réponse. En- têtes de réponse HTTP ou le corps de réponse lui-même sans en-têtes?
Si des cookies d'autorisation sont nécessaires, définissez la méthode de réponse des en- têtes comme dans le cas d'Asterisk.

Si vous avez besoin de données, comme c'est le cas avec la réponse du serveur mikrotik, mettez le corps de réponse sans en-têtes.

Étape 2


Nous passons à la deuxième étape. Obtenir une session d'autorisation:

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc": "2.0",
    "method": "user.login",
    "params": {
        "user": "Admin"
        "password": "zabbix"
    },
    "id": 1,
    "auth": null
}

jsonrpc est la version du protocole JSON-RPC utilisée;
Zabbix implémente JSON-RPC version 2.0;

  • méthode - la méthode qui est appelée;
  • params - paramètres transmis par la méthode;
  • id - un identifiant de demande arbitraire;
  • auth - clé d'authentification de l'utilisateur; comme si nous ne l'avons pas encore, nous le définissons sur null.

Pour travailler avec l'API, j'ai créé un compte distinct avec des droits limités. Premièrement, vous n'avez pas besoin de donner accès là où vous n'en avez pas besoin. Et d'autre part, avant la version 5.0, le mot de passe spécifié via la macro pouvait être lu. Par conséquent, si vous utilisez le mot de passe administrateur Zabbix, le compte administrateur est facile à voler.

Cela sera particulièrement vrai lorsque vous travaillez avec l'API via des scripts tiers et stockez les informations d'identification sur le côté.

Depuis la version 5.0, l'option de masquer le mot de passe enregistré dans la macro est apparue.



Lorsque vous créez un compte distinct pour la mise à jour des données via l'API, assurez-vous de vérifier si les données dont vous avez besoin sont disponibles via l'interface Web et s'il est possible de les mettre à jour. Je n'ai pas vérifié, puis pendant longtemps je n'ai pas pu comprendre pourquoi la macro dont j'avais besoin n'était pas visible par l'API.



Après avoir reçu l'autorisation dans l'API, nous procédons à l'obtention d'une liste de macros.

Étape 3


L'API ne permet pas de mettre à jour la macro hôte par son nom, pour cela, vous devez d'abord obtenir l'ID de la macro. De plus, pour obtenir une liste de macros pour un hôte spécifique, vous devez trouver l'ID de cet hôte, et c'est une demande supplémentaire. Vous ne pouvez pas utiliser la macro régulière {HOST.ID} dans la demande. La limitation a décidé de contourner ce problème:



j'ai créé une macro locale avec l'ID de cet hôte. Il est très facile de trouver l'ID d'hôte à partir de l'interface Web.

La réponse avec une liste de toutes les macros d'un hôte donné peut être filtrée par un modèle:

regex:{"hostmacroid":"([0-9]+)"[A-z0-9,":]+"{\$MIKROTIK_VERSION}"



Ainsi, nous obtenons l'ID de la macro dont nous avons besoin, où MIKROTIK_VERSION est le nom de la macro que nous recherchons. Dans mon cas, il recherche la macro MIKROTIK_VERSION qui a été affectée à l'hôte.

La demande elle-même ressemble à ceci:

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.get",
    "params":{
        "output":"extend",
        "hostids":"{$HOST_ID}"
    },
    "auth":"{sid}",
    "id":1
}

La variable {sid} a été obtenue dans la deuxième étape et sera utilisée en permanence, là où il est nécessaire de travailler avec l'interface API.

Final 4 STEP - mise à jour des macros


Nous connaissons maintenant l'ID de macro à mettre à jour, le cookie d'autorisation ou la version du firmware du routeur. Vous pouvez mettre à jour la macro elle-même.

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.update",
    "params":{
        "hostmacroid":"{hostmacroid}",
        "value":"{mikrotik_version}"
    },
    "auth":"{sid}",
    "id":1
}

{mikrotik_version} - la valeur obtenue à la première étape. Dans mon exemple - la version du micrologiciel actuel mikrotik
{hostmacroid} - la valeur a été obtenue dans la troisième étape - l'ID de macro, que nous mettons à jour.

résultats


L'approche pour résoudre le problème avec des fonctionnalités régulières est beaucoup plus compliquée et plus longue. Surtout si vous connaissez la programmation et pouvez rapidement jeter la logique nécessaire dans le script.

L'avantage évident de cette approche est la «portabilité» de la solution entre différents serveurs.

Personnellement, c'est étrange pour moi le manque de possibilité d'accéder aux données d'un autre élément dans l'agent HTTP et sa substitution dans le corps ou les en-têtes de la demande [ ZBXNEXT-5993 ].
