DNS GeoIPの選択


クライアントがサイトをノックしている場合、どのサーバーがデータを彼に送信する必要があるかをどのように理解するのですか？もちろんDNSを使用します。つまり、DNSのアドレスを指定したIPアドレスに応じて、関連する回答が提供されます。

独自のDNS（Maxmindのプラグインを含むBIND）またはターンキーソリューション（Route53）を使用できます。Maxmind GeoIP-databasesのサブスクリプションは月額25ドル、Route53はドメインゾーンごとに0.50ドルかかります。さらに、DDoS攻撃を強化する別のポイントを作成することは最後です。そのため、私の選択はRoute53にありました。

この記事はヨーロッパのCDNについてです。Route53は都市ではなく国ごとのルーティングを提供するため、ロシアのCDNが必要な場合は、独自のDNSを選択するのは明らかです。

1. IISをインストールする


1.1。IISのインストールIIS

の基本コンポーネント、一元化された証明書のサポートコンポーネント、およびメインサーバーにリモート管理の追加サポートをインストールします。このコンポーネントは1つのサーバーにのみ必要です。リモート構成が構成されている場合でも、共有構成を有効にすると他のサーバーを管理できなくなります。



Powershell経由： 

Install-WindowsFeature Web-Server, Web-CertProvider

メインサーバーで、さらにインストールする必要があります： 

Install-WindowsFeature Web-Mgmt-Service

1.1.1リモート管理を

有効にするIISサーバーをリモートで管理するには、IISリモート管理サービスを上げる必要があります。メインサーバーでサービスを開始します。

start-service WMSVC
set-service -Name WMSVC -StartupType Automatic

そして今、私たちはレジストリを介してそのように管理する機能を含みます。

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\WebManagement\Server

もちろん、レジストリを管理する最も簡単な方法は、Windows管理センターを使用することです。 



しかし、これはPowershellを介して行うこともできます：

Set-Itemproperty -path "HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\WebManagement\Server" -Name "EnableRemoteManagement" -value "1"

Windows Server 2012および2016では、ファイアウォールルール自体が上昇しないため、ファイアウォールを編集する必要があります。 



Set-NetFirewallRule -Name IIS-WebServerRole-WMSVC-In-TCP -Enabled True

上昇しているかどうかを確認します。



Test-NetConnection 8.8.8.8 -port 8172

3つの異なる場所で3つの編集を行うと、デスクトップを備えた別のサーバーを使用して、IISマネージャーを介してサーバーに接続できます。これは、その後の管理の便宜のためだけに必要でした。

1.2構成の集中化を有効にする

IISマネージャーには集中化された構成と証明書をインストールするためのボタンはなく、ローカルサーバーマネージャーにのみあるため、Server Coreの場合はPowershellを介してすべてを実行する必要があります。

構成と証明書の集中化は、SMBを通じて実行されます。したがって、私は2人の権利を剥奪されたユーザー（構成用と証明書用に別々に）を作成し、フォルダーへの読み取りアクセスのみを持ち、certadminとconfigadminという名前を付けました。

ユーザーがメインサーバーのローカルにいることをお勧めします。ドメインコントローラーが停止すると、ドメインユーザーに代わってSMBを介して構成と証明書に固執するアプリケーションが停止します。ローカルユーザーを使用すると、このような状況が解消されます。

1.2.1共有フォルダの作成

サーバーの1つに、構成と証明書を取得する2つの共有フォルダを保存する必要があります。共有フォルダーへのパスとして、IIS構成への既定のパスを使用します。

New-SmbShare -ReadAccess configadmin@**.**wtf -Path C:\windows\System32\Inetsrv\Config -Name sharedconfigs

また、証明書には任意のものを選択できます。IISのあるフォルダに入れました。

New-SmbShare -Path C:\inetpub\centralizedcerts -Name sharedconfigs -ReadAccess configadmin

1.2.3 Brotli

各サーバーで、IIS圧縮ファイルをダウンロードして実行します

Start-Process .\iiscompression_amd64.msi -ArgumentList /quiet

すでに構成を共有しているため、何かを1か所で構成するだけで済みます。IISマネージャーを使用して、メインサーバーの構成エディターに移動します。



system.webServer/httpCompression

そして、StaticCompressionLevelをBrotliの場合は11に、Gzipの場合は9に設定します。これは、ユーザーが実行できる最大値です。

4. DFSをインストールする


DFSを介して証明書と構成を同期しないでください。特殊なツールが理由のために発明されました。私が試したところ、いくつかの不明確な理由により、構成を取得したサーバーの構成が変更の読み取りを停止しましたが、初めてすべてが機能しましたが、それはうまくいきませんでした。失敗の原因を特定できなかったため、別の方法で行うことにしました。

詳細な構成はRSATを介して排他的に行われます。ドキュメントで指定されているコマンドレットは、GUIを備えたWindowsサーバーでのみ機能します。 Server Coreのみを使用してDFSレプリケーションを展開することはできません。 GUIを備えた別のサーバー、またはドメインに参加しているRSATがインストールされたWindows 10 Proを実行しているコンピューターが必要です。
