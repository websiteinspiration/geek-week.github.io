1年半エンジンとゲームをどのように作成したか。二部。インフラ

最初に、前の記事のトレースに関するいくつかのコメント。私たちは、dava.frameworkまたはdava.engineというエンジンを開発したWargamingで実際に働いていました。そのため、現在も良好な関係にある多くの年長の同僚が積極的に議論に参加しています。 多くの人が疑問を持っています。これは同じテクノロジーですか、それとも別のテクノロジーですか？回答：これはゼロから書かれた新しいテクノロジーです。 わずか1年でどのように管理できましたか？私たちのチームには膨大な経験があります。多くは、15年以上にわたってエンジンとゲームを開発してきました。





オープンソースにもある古いエンジンを採用できるのであれば、なぜ最初から？彼は約10歳で、コードのほとんどは古くなっています。私たちが誇りに思っているエンジンの最高の部分でさえ、コードの一部と5、7、時には10年前のいくつかの原始が含まれていました。多くの建築ソリューションは当時のデバイス向けに設計されました-3G iPhoneから始まります。今、私たちは少なくともiPad Air 1と同様のパワーAndroidデバイスに焦点を当てています。したがって、アプローチは多少変更されました。

そして最も一般的な質問：なぜ自分のエンジン？前回の記事では、説得力の程度が異なるいくつかの議論がありました。私は主なことに集中したいと思います。私たちのテクノロジーだけがあなたが鉄を最大限に活用することを可能にし、あなたのゲームプレイ、視覚的なスタイルのために特に最適化の最大数を作ります。私たちは、ゲーム開発者だけでなく、テクノロジー企業としても位置づけています。私たちはエンジニアのレベルと経験により、ハイテクモバイル製品の市場で真剣に競争できると信じています。

そして今、要点として、このかなり野心的なタスクを短時間で達成するのに役立つツールとテクニックは何ですか？

インフラ


アトラシアンのBitbucket Server + Jenkinsを選択しました。Bitbucketには、Jenkinsが接続されているメインリポジトリ（マスター）があります。各開発者は自分のフォークを持っています。フォーク内の新しいフォークは、タスクごとに作成され、プルリクエストを通じて統合されます。一般に、このスキームはかなり標準的です。各プルリクエストは、必須のレビューと自動テストを受けます。そして、成功すると、自動的にマスターにマージされます。

ジェンキンス


Jenkinsにはいくつかの欠点があります。彼は古代であり、それほど高速ではなく、大食いで、90年代のインターネットポータルのように見えます。ただし、その柔軟性、膨大な数のモジュール、および無料で、2019年でも優れた選択肢となっています。モジュールとカスタマイズを試してみると、わかりやすい外観、パイプの宣言的な記述（リポジトリーにある）を実現できます。ちなみに、現在、約40のパイプラインがあります。すべてのプラットフォーム用のテスト、エディター、ゲーム。サーバーインフラストラクチャとメタゲームを操作します。20個すべてのbuildagentovを収集します。

もちろん将来的には、GitLabやセルフホストのTravisCIなど、最新の流行に敏感なソリューションを試してみたいと思います。リポジトリ、アセット、ビルド時間とアーティファクトのサイズが大きいため、クラウドソリューション（Nevercode、Bitrise、CircleCIなど）は完全には考慮していません。

ビルドシステム


主なシステム要件は次のとおりです。1つのスクリプトでのiOS、MacOS、Android、Windows、Linuxのプロジェクト生成。Premake、SCons、Bazel、CMakeを試してみました。さまざまな理由で、私たちは時間をかけてテストされたCMakeで停止しました。

近年、CMakeはC ++ライブラリのほぼ標準になりました。abseilからSDLまでのほとんどすべてを、ほんの数行でCMakeプロジェクトに接続できます。もちろん、OpenSSLやV8のような例外はありますが、これらには少し汗を流さなければなりませんでした。裸のZmeikの上に小さなフレームワーク（合計約3,000行）を開発しました。主な機能：
モジュール性。エンジンの個々のパーツはモジュールとして設計されています。たとえば、サウンド、UI、物理学、ネットワークなど。各モジュールは独自のアセット（シェーダーなど）を持つことができ、他のモジュールに依存する場合があります。

エンジン上の最終的なアプリケーション（ゲーム、エディター、ユーティリティ）は、必要なモジュールのみを接続します。少し離れているのはコアモジュールで、これは他のほとんどのモジュールに依存しています。コアは、エントリポイント、メインアプリケーションサイクル、オペレーティングシステムとの相互作用、その他の基本エンティティを実装します。
サードパーティのモジュール。私たちのフレームワークでは、gitリポジトリをダウンロードしたり、数行でアーカイブしたり、ライブラリやソースをアンパック、コンパイル、コピーしたりできます。今日、このようなサードパーティモジュールが66個あります。分析、サードパーティのファイル形式、物理学のようなミドルウェア、サウンドライブラリなどです。

開発工程


以前の経験に基づいて、エンジンとゲームの両方を1つのリポジトリに追加することにしました。これにより、簡単にエンジンAPIを変更し、ゲームを同期的に適応させることができます。その結果、その長所と短所を持ついわゆる単一リポジトリーができました。しかし、私たちはすぐに非常に高いペースで開発を続けることを計画していたため、エンジンとゲームの同期リファクタリングの可能性は、このソリューションの他のすべての欠点を上回っていました。

平均して、1日あたり20を超えるプルリクエストが追加されます。これは、マスターが1日に20回壊れる可能性があることを意味します。幸いにも、1991年に、彼らは継続的インテグレーション技術を思いつきました。私たちは何に来ましたか？

テスト用のほとんどのプロジェクトは、「警告をエラーとして扱う」フラグをオンにして収集され、さらにclang AddressSanitizerがオンになっているMacOSプラットフォームで収集されるため、プルリクエストの準備段階でさらに多くのエラーをキャッチできます。

継続的デリバリー


私たちは自動毎日（またはむしろ毎日）リリースの概念を使用しています。これはどのように正確に起こりますか？

  1. gitタグが作成され、
  2. すべてのテストのフルバージョンを実行し、
  3. 成功すると、成果物が収集されます。

  • MacOSおよびWindows用のエディター。したがって、毎朝、誰もが新しいバージョンのツールを持っています。また、自動テストのおかげで、一定の品質と安定性に自信を持っています。
  • すべてのプラットフォームのゲームクライアントとサーバー。iOSのクライアントは、TestFlight、Android、Google Play、その他のプラットフォーム、JFrog Artifactory、ゲームサーバー、その他のサービスに、クラウドにアップロードされます。つまり、毎朝新しいバージョンのゲームがあり、テストとプレイテストの準備ができています。

もちろん、毎晩のリリースが正常に終了するわけではありません。一部のテストが失敗するか、アプリケーションの起動時に重大なエラーが発生します。この場合、発見された問題は、当日の勤務中の開発者によって修復され、リリースプロセスが再開されます。
毎日いくつかの義務があります：

  1. 第1レベルのアテンダント。メインリポジトリ内のテストの安定性を監視します。
  2. ゲームの2次レベルアテンダント。ゲームのバグを修復します。
  3. 編集者の2次レベルアテンダント。編集上のバグを修正し、ユーザー（アーティスト、レベルデザイナー、ゲームデザイナー）にアドバイスします。

また、当日は、技術的な負債を負うこともできます。古い機能の不足しているテストを追加したり、ドキュメントを補足したり、通常のリリース計画に時間をかけないリファクタリングを行ったりできます。
