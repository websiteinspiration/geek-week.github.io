VLANはスイッチの概念であるため、各スイッチは仮想ネットワークのデータベースの作成と保存をサポートしています。これは、特定のVLANに対応するポートを示すテーブルです。したがって、スイッチがPC1のトラフィックを受信すると、このトラフィックがVLAN10の一部であるかどうかを確認し、コンピューターにルーティングします。 PC1からのトラフィックがPC4に対するものである場合、スイッチはSW1-SW2トランクを介してトラフィックをルーティングします。トラフィックが最初のスイッチのトランクポートに入るとすぐに、VLAN IDを含むVLAN TAGヘッダーがフレームに提供されます。この場合、10です。このトラフィックを受信した後、2番目のスイッチはフレーム情報を読み取り、VLAN10トラフィックであることを確認してルーティングします。 PC4の青いポートに。

ネイティブVLANの概念について引き続き説明します。すでに述べたように、各スイッチポートは特定のVLANに関連付けられています。



たとえば、最初のスイッチの2つのポートはVLAN10に接続され、3番目のアクセスポートはVLAN20を使用し、4番目のポートはトランクポートです。同様に、SW2はVLAN10ポートを介してPC4に、VLAN20アクセスポートを介してPC5に、トランクポートを介してハブに接続されています。ただし、1つの問題があります。スイッチが高価であるため、2つのスイッチをハブを介して相互に接続する方式がよく使用されます。 2つのスイッチがトランクを使用してハブに接続されていますが、ハブ自体はVLANの概念について何も認識せず、信号をコピーするだけです。すでに述べたように、VLANトラフィックが直接コンピュータに送信されると、それが何であるかを理解していないため、トラフィックはドロップされます。 PC4コンピューターとの接続を確立しようとしている場合、ハブに直接接続されているPC6コンピューターをどのように処理しますか？

PC6は、スイッチSW2に向かうトラフィックを送信します。このトラフィックを受信したスイッチは、フレームにVLANタグがないことを確認し、VLAN10またはVLAN20のどのネットワークに送信するかを認識していません。この場合、シスコはネイティブVLANと呼ばれるテクノロジーを作成しました。デフォルトでは、VLAN1はネイティブVLANです。

別のコンピュータがあり、スイッチSW2を介してそれを描画するとします。このPCはポートVLAN1を介してスイッチに接続されています。同じコンピュータがSW1の上にあり、VLAN1を介して接続されています。右のスイッチの下に別のコンピュータを描きます。



VLAN1を介してスイッチSW2に接続されている2台のコンピューターは互いに通信できますが、他のコンピューターとは通信できません。スイッチは、トランクを介してタグなしトラフィックを受信すると、このトラフィックがVLAN1またはネイティブVLAN宛てであると見なし、VLAN1ポートに接続されているコンピューターに転送します。同様に、タグなしのPC6トラフィックを受信すると、スイッチはそのVLAN1ネットワークに対応します。

PC5とスイッチSW2に接続されているVLAN20の赤い線にCisco IP Phoneがある場合はどうなりますか？これは、オフィスのネットワーク機器の典型的なレイアウトです。この場合、ネイティブVLANの概念も使用されます。私が言ったように、コンピュータはVLANが何であるかを知らず、電話は知っています。問題は、同じVLANを介してデータと音声を送信できるかどうかです。これは非常に危険な状況です。コンピュータがIP電話と同じ通信回線上にある場合、ハッカーはそのような通信チャネルに簡単に接続し、Wiresharkを使用して音声パケットを傍受できるためです。次に、これらの音声パケットをオーディオファイルに変換し、電話での会話を盗聴できます。したがって、実際には、音声トラフィックとデータトラフィックが同じVLANで送信されることはありません。これはどのように設定できますか？

IP電話が接続されているポートをトランクポートに変更し、このポートを通過するトラフィックはすべてVLAN30ネットワークの音声トラフィックであると見なします。すべてのCisco IP Phoneは、一般に.1QまたはDot 1Qと呼ばれる802.1qカプセル化プロトコルを使用します。したがって、電話機からのトラフィックが対応するポートに到達すると、スイッチはこれがVLAN30音声トラフィックであることを理解します。 VLAN30の一部でもあるSWスイッチに接続されている別の電話が必要です。

この場合、アクセスポートを介してスイッチに接続されているPC4コンピュータで何が起こりますか？結局のところ、このコンピューターがスイッチと交換するすべてのトラフィックは、青いVLAN10に属しています。ただし、PC5はトランクを介してスイッチに接続されており、トランクについてはVLANを構成しません。この場合、ポートはアクセスモードではなくトランクモードで動作するため、switchport access VLAN＃コマンドを使用できません。これは、PC6の場合と同じ概念を使用しています。スイッチがタグなしトラフィックを受信すると、ネイティブVLANポートにルーティングされます。デフォルトでは、VLAN1です。

問題は、ネイティブVLANを変更できるかどうかです。答えは「はい」です。たとえば、これを行うことができます。たとえば、赤い線の場合、ネイティブVLANをVLAN20に変更すると、スイッチはトラフィックをPC5からVLAN20ネットワークにリダイレクトします。両方のスイッチがトランクで接続されているため、VLAN20トラフィックを受信したSW2スイッチは、それをネイティブVLANトラフィックと見なし、タグなしとしてSW1スイッチを送信します。

このトラフィックを受信したSW1スイッチは、タグなしのネイティブトラフィックとして認識し、ネイティブVLANはVLAN1であるため、このトラフィックをこのネットワークに送信します。ネイティブVLANを変更する場合は、すべてのスイッチのすべてのネイティブVLANが正しく変更されていることを確認するように注意して行う必要があります。変更しないと、多くの問題が発生する可能性があります。

これはネイティブVLANの簡単な概要でした。次に、独自のVTPトランキングプロトコル（VLAN）に移ります。まず、VTPはその名前にもかかわらず、トランキングプロトコルではないことを覚えておいてください。



前のレッスンから、トランキングプロトコルは2つしかないことがわかっています。ISLと呼ばれる独自のシスコプロトコルと、一般的に受け入れられている802.1qプロトコルです。

VTPもシスコ独自のプロトコルですが、トランク接続を作成するという意味ではトランキングしません。コンピューターが接続されている最初のスイッチのポートにVLAN10を作成したとします。次に、トランクSW1-SW2とトランクSW2-SW3があります。トランクポートSW1がコンピュータトラフィックを受信すると、それはVLAN10トラフィックであることを認識し、2番目のスイッチに転送します。ただし、2番目のスイッチは、トランク以外は何も接続されていないため、VLAN10が何であるかを認識していません。したがって、このトラフィックを受信して​​さらに送信するために、ポートにVLAN10を作成します。スイッチ3も同じように動作します。トランクでトラフィックを受信すると、VLAN10が作成されます。

SW3に2つのアクセスポートを作成でき、どちらもVLAN10になります。 3つすべてのスイッチで、別のネットワークVLAN20を作成するとします。これは、VLAN20のポートが作成された後にのみ可能になります。ネットワークに追加するデバイス、コンピューター、およびスイッチの数が多いほど、新しいVLANの作成が難しくなります。そのため、シスコはVTPを作成してこのプロセスを自動化しました。

新しいVLANを作成する場合、いずれかのスイッチでVLAN30と呼び、トランクで接続された他のすべてのスイッチで、同じVLAN30ネットワークが自動的に作成されます。



拡張された更新済みのVLANデータベースは、すべてのスイッチに送信されるだけです。その後、コンピューターのアクセスポートを作成するだけです。このプロトコルがないと、すべてのスイッチを手動で再構成する必要があります。 VTPの欠点は、VLANデータベースを変更すると、リビジョン番号（リビジョン番号）が変更されることです。通常、ボックスから直接スイッチを使用すると、すべての設定のリビジョン番号はゼロになります。 10番目などの新しいVLANを追加すると、データベースSW1はリビジョン番号＃1を取得します。この場合、2番目のスイッチは次のように述べています。「わかりました。リビジョン1があり、リビジョン0があるので、リビジョン番号を1に変更して、すべてのデータをVLANテーブルからテーブルにコピーする必要があります。」 3番目のスイッチも同じです。

ここで、スイッチ2がVLAN20を追加し、リビジョン番号を2に変更するとします。最初のスイッチと3番目のスイッチも同じようにする必要があります。リビジョン番号を変更するたびに、プロトコルはこの番号が大きい方をチェックし、他のすべてのリビジョン番号をこの番号に変更しながら、VLANテーブルを更新します。さらに、VTPは、最も高いリビジョン番号のスイッチを無条件で信頼します。

この状況を想像してみてください。新しい従業員が会社に来て、角のどこかでスタッフのトレーニングに使用されるスイッチを発見します。彼はこれについて何も知らず、このスイッチの方が新しいように見え、共有ネットワークに接続することにしました。彼はこのスイッチを構成し、たとえばSW2スイッチに接続してトランクを作成します。そして、それがオンになるとすぐに、ネットワーク全体が落ちます！コンピュータとスイッチ間の接続が完全になくなるため、すべてが機能しなくなります。



なぜこれが起こったのですか？スイッチの会社のVLANは5、10、20、30、40、50しかないため、スイッチの会社の最大リビジョン番号は50です。新しいスイッチがトレーニングに使用され、より多くのネットワークがスイッチに接続され、設定に多くの変更が加えられた結果、リビジョン番号が100に増加しました。同時に、VLANデータベースには番号105のネットワークが1つしかありません。

SW Trainingがトランクを介してSW2に接続された後、2番目のスイッチは、初心者がより高いリビジョン番号を持っていることを確認し、番号をより高いものに変更することを決定しました。同時に、彼は新しいスイッチのVLANテーブルを自分にコピーし、既存のVLAN10、20、30ネットワークをすべて自動的に削除し、既存のネットワークに存在しなかった1つのVLAN105に置き換えます。最初のスイッチと3番目のスイッチは同じことを行い、リビジョン番号を50から100に変更し、古いネットワークをデータベースから削除しました。これは、SWトレーニングスイッチのVLANテーブルに含まれていないためです。
スイッチSW1はVLAN10ネットワークのアクセスポートを作成しましたが、リビジョンを更新した後、このネットワークは表示されなくなりました。スイッチは、アクセスポートがVLANデータベースにないネットワークで動作するように構成されている場合、このポートがプログラムで無効になるように設計されています。 VLAN20およびVLAN30ネットワークでも同じことが起こりました。スイッチは仮想ネットワークの更新されたデータベースでそれらを検出せず、対応するアクセスポートを単に無効にしました。その後、会社の既存のローカルネットワークに障害が発生しました。



これは実際に頻繁に発生することを保証します。個人的に、誰かが新しいスイッチを接続したという事実のためにネットワークが機能しなくなったときのイベントを2回目撃しました。 VTPは非常に強力なものであるため、注意してください。シスコでは、同様の問題が発生する可能性があるため、VTPを使用しない方がよいと考えています。

VTP使用エラーによって引き起こされるネットワーク障害を防止するメカニズムがあります。これはVTPドメインのメカニズムであり、このように機能します。ネットワーク上のスイッチの1つのドメインがVTPプロトコルを実行している他のスイッチのドメインと異なる場合、このスイッチはVLANデータベースに複製されません。ただし、このメカニズムにもかかわらず、シスコは特別な必要なしにこのプロトコルを使用することを推奨しません。

ただし、ネットワークを作成するときにVTPが役立つこと、およびスイッチの構成に責任を持ってアプローチできることが確実な場合は、それを使用してみることができます。 VTPには、サーバー、クライアント、トランスペアレントの3つのモードがあります。

VTPサーバーモードでは、ネットワークに変更を加えることができます。つまり、スイッチのコマンドラインからVLANを作成、削除、変更できます。デフォルトでは、このモードはすべてのCiscoスイッチで設定されています。
私は3つのスイッチを描きました。最初のスイッチはサーバーモードで、他の2つはクライアントモードです。最初のスイッチでのみ新しいVLANを作成できます。その後、データベースは2番目と3番目のスイッチで複製されます。 2番目のスイッチでこれを実行しようとすると、「サーバーではないので、私の設定でこのような変更を行うことはできません。」という答えが返されます。これは変更を防ぐメカニズムです。したがって、サーバーによってスイッチの1つを選択し、その設定に変更を加えると、スイッチ-クライアントで繰り返されます。ただし、VTPを使用しない場合はどうなりますか？

このプロトコルの使用を完全に拒否するには、スイッチをトランスペアレントモードにする必要があります。同時に、VTPモードを無効にせず、スイッチがVTPアドバタイズを生成せず、VLANデータベースを更新せず、常に設定リビジョン番号0

を使用します。2番目のスイッチに透過モードを使用するとします。 VTP情報を受信すると、このプロトコルが適用されないことがわかり、この情報をクライアントモードにある次のスイッチに渡すだけで、自分の設定は何も更新されません。したがって、トランスペアレントモードは、特定のスイッチでVTPを使用することを拒否することを意味します。



したがって、サーバーモードでは変更を加えることができ、クライアントモードではこれらの変更を受け取ることができ、透過モードではVTPプロトコルを介して変更が適用されないようにし、ネットワークを介してさらに変更を転送することに注意してください。

次に、VTPプルーニングと呼ばれる概念について説明します。スイッチSW1に2つのVLAN30ネットワーク、赤いVLAN20ネットワークと2つの青いVLAN10ネットワークがあると仮定します。



スイッチSW2にはVLAN30用のポートがありません。ただし、デフォルトでは、SW1はタグ付きのVLAN10、20、および30トラフィックをトランク経由で送信します。ネットワーク管理者は、SW2スイッチにVLAN30がないことを知っていますが、トラフィックが正しく送信されることを確認する必要があります。これを行うには、VTPプルーニングを使用してSW1から発信されたトラフィックの追加情報を使用します。最初のスイッチは、トランクを介してVLAN30ネットワークのトラフィックを送信する可能性を除いて、トランクを介してVLAN10およびVLAN20ネットワークのトラフィックのみを送信できるように構成します。これがVTPプルーニングの概念のすべてです。次のビデオチュートリアルでは、今日お話しした設定を行う方法について説明します。
