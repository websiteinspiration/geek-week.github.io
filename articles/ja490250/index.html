ホラーSet.removeAll

JDKの標準コレクションが非常にうまく作られ、直感的に動作するという事実に慣れています。しかし、本当にそうでしょうか？昨日のローマのエリザロフエリザロフTwitterに新しい興味深い枠についてのニュースを投稿しました。

しばらくお待ちください：Set.removeAll(list)場合によってはO（N²）でも機能します。どうして？



Romanは、驚くべき理由で動作が遅すぎるコードをデバッグしたときに問題を発見しました。彼はIntelliJ IDEAに組み込まれたプロファイラーでそれを起動し、AbstractSet.removeAll呼び出すためにメソッド内で常に無駄にしていたことをフレームグラフで即座に確認しましたlist.contains（コード内の正確な場所へのリンク）。

何が起こっているのかを理解するために、少し異なる例を考えてみましょう。以前、Jon Skeetは「私の抽象化には穴があります、Lear、dear Liza」という投稿を書きました。彼は次の興味深いケースを検討しています。

何かを削除するHashSetがあるとします。あるコレクションAから別のコレクションBから要素を削除するとします。Bからの多くの要素が単にAに存在しないことがよくあります。AとBがまったく交差しない、つまり何もする必要がないという特別な場合を分析します。

コレクションのサイズをコマンドラインから設定する簡単なプログラムを作成します。これらのセットが完全に交差しないように、一方を正の数値のみで埋め、もう一方を負の数値のみで埋めます。次に、最初のコレクションから2番目のコレクションのすべての要素を削除し、次を使用して経過時間を測定しますSystem.currentTimeMillis()。これは期間を計算するための世界で最良の方法ではありませんが、コードをHabrからIdeaに直接コピーして貼り付けることが可能になり、JMHの新しいMavenプロジェクトを設定する必要がなくなります。

import java.util.*;
public class Test {
    public static void main(String[] args) {
       int sourceSize = Integer.parseInt(args[0]);
       int removalsSize = Integer.parseInt(args[1]);

       Set<Integer> source = new HashSet<Integer>();
       Collection<Integer> removals = new ArrayList<Integer>();

       for (int i = 0; i < sourceSize; i++) {
           source.add(i);
       }
       for (int i = 1; i <= removalsSize; i++) {
           removals.add(-i);
       }

       long start = System.currentTimeMillis();
       source.removeAll(removals); 
       long end = System.currentTimeMillis();
       System.out.println("Time taken: " + (end - start) + "ms");
    }
}

意識を取り戻すことなく書くことができる非常にシンプルなコード。次に、さまざまなパラメーターを指定して実行します。最初に、100をスローする必要がある100要素のセットを試します。

$ java Test 100 100
Time taken: 1ms

これまでのところ、すべてが十分に高速ですが、数値を増やすとどうなりますか？

java Test 1000000 300000
Time taken: 38ms


$java Test 300000 300000
Time taken: 178131ms

どのようにあなたはこれを好きですか：今、私たちは3分待ちます。小さなコレクション（2番目のケースでは300,000アイテム）の処理時間は、大きなコレクション（最初のケースでは100万アイテム）を処理するときよりも短いはずです。すぐにその逆になります。人生はこれに備えていませんでしたよね？

今フォーカスの秘密。

実際、対応するJavaDocにクリアテキストで記述されています。
実装コードは、それが少ないかどうかを決定します：セットまたはコレクション。これはsize、それぞれのメソッドを使用して行われます。セット内の要素の数が少ない場合、セットで反復が実行され、反復ごとに現在の要素がコレクション内にあるかどうかが確認されます。はいの場合、要素はremoveイテレータメソッドを使用してセットから削除されます。コレクションの要素数が少ないことが判明した場合は、コレクションをバイパスしてremove、セットからメソッドを使用してそのような要素をすべてセットから削除する必要があります。
実際には、これは呼び出されたときに次のことを意味しsource.removeAll(removals)ます。

  • コレクションがremovalsより小さい場合source、メソッドはで呼び出さremoveれHashSet、かなり高速です。
  • コレクションのremovalsサイズがより大きいか等しい場合、の動作が非常に遅いsourceメソッドが呼び出さremovals.containsれArrayListます。

JDK 8では、次のようなコードがこのコード部分を担当します。

public boolean removeAll(Collection<?> c) {
    Objects.requireNonNull(c);
    boolean modified = false;

    if (size() > c.size()) {
        for (Iterator<?> i = c.iterator(); i.hasNext(); )
            modified |= remove(i.next());
    } else {
        for (Iterator<?> i = iterator(); i.hasNext(); ) {
            if (c.contains(i.next())) {
                i.remove();
                modified = true;
            }
        }
    }
    return modified;
}

JDKはタスクに対処するためにかなり貧弱な方法を選択したようですが、それはすでにJavaDocに記述されているため、後戻りはありません。
