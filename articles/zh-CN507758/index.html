JavaScript内存-无泄漏









创建和访问数据的方式可能会影响应用程序的性能。让我们看看如何。



介绍



JavaScript是一种非常高级的语言。因此，大多数开发人员不会考虑数据如何存储在内存中。在本文中，我们将研究数据如何存储在内存中，如何影响处理器和内存，以及如何将数据分发到JS并访问它如何影响性能。



三角恋



当计算机需要执行某些计算时，处理器（CPU）需要处理数据。因此，根据任务，他通过总线向存储器发送数据采样请求。



看起来像这样：







这是我们的浪漫三角-处理器->总线->内存



维持三人的健康关系很困难



处理器比内存快得多。因此，此过程处理器->总线->内存->总线->处理器在计算上“花费”时间。在扫描内存时，处理器处于空闲状态。



为防止停机，已将缓存添加到系统中。我们不会详细介绍什么是高速缓存或什么类型的高速缓存，但是可以说高速缓存是处理器的内部存储器就足够了。



当CPU收到要执行的命令时，它首先在高速缓存中查找数据，如果那里没有数据，它将通过总线发送请求。



总线依次传输所请求的数据以及存储器的条件部分，并将其存储在高速缓存中以供快速参考。



因此，CPU将对内存的速度有较少的不满，因此，CPU的停机时间将减少。



争吵是任何关系的一部分。



可能出现的问题（尤其是在处理大量数据时）是一种称为高速缓存未命中的现象。



高速缓存未命中意味着在计算期间，CPU会检测到高速缓存中没有必要的数据，并且它需要通过常规通道请求此数据-您还记得内存很慢。







现金单插图。处理了来自阵列的数据，但是还请求超出缓存限制的数据进行计算-这会导致“缓存未命中”。



好的，但是我是JavaScript开发人员，为什么我应该关心？



好问题。在大多数情况下，您无需考虑。但是，如今，随着越来越多的数据通过用Node.js编写的服务器传递，在对大型数据集进行排序时，您更有可能遇到缓存未命中的问题。

这是一个叫做Boom的类。 



class Boom {
  constructor(id) {
    this.id = id;
  }
  setPosition(x, y) {
    this.x = x;
    this.y = y;
  }
}


此类（Boom）仅具有3个属性-id，x和y。



现在，让我们创建一个填充x和y的方法。



让我们设置数据： 



const ROWS = 1000;
const COLS = 1000;
const repeats = 100;
const arr = new Array(ROWS * COLS).fill(0).map((a, i) => new Boom(i));


现在，我们将在方法中使用此数据： 



function localAccess() {
  for (let i = 0; i < ROWS; i++) {
    for (let j = 0; j < COLS; j++) {
      arr[i * ROWS + j].x = 0;
    }
  }
}


localAccess的作用是线性遍历数组并将x设置为0。



如果我们重复执行此函数100次（请查看重复常数），我们可以测量执行所需的时间： 



function repeat(cb, type) {
  console.log(`%c Started data ${type}`, 'color: red');
  const start = performance.now();
  for (let i = 0; i < repeats; i++) {
    cb();
  }
  const end = performance.now();
  console.log('Finished data locality test run in ', ((end - start) / 1000).toFixed(4), ' seconds');
  return end - start;
}
repeat(localAccess, 'Local');

缓存未命中的价格



现在，根据我们上面学到的，如果我们处理迭代过程中距离较远的数据，则可以考虑缓存未命中问题。删除的数据是不在相邻索引中的数据。这是此功能： 



function farAccess() {
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      arr[j * ROWS + i].x = 0;
    }
  }
}


这里发生的是，在每次迭代中，我们都转到索引，该索引位于距上次迭代的ROWS距离处。因此，如果ROWS为1000（在我们的情况下），我们将得到以下迭代：[0，1000，2000，...，1，1001，2001，...]。



让我们将其添加到速度测试中： 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local');
}, 2000);


这是最终结果：







非本地迭代的速度几乎慢了4倍。这种差异将随着数据的增加而增加。这是因为处理器由于高速缓存未命中而处于空闲状态。



那么您要付出什么代价呢？这完全取决于您的数据大小。

有多种解决方案可以解决此问题，但是现在您知道性能下降的原因，您可能可以自己提出解决方案。您只需要在内存中存储尽可能接近处理的数据。

这种技术称为数据局部性设计模式。
让我们继续我们的例子。假设在我们的应用程序中，最常见的过程是使用farAccess函数中显示的逻辑进行数据处理。我们希望对数据进行优化，以使其在最常见的for循环中快速运行。



我们整理以下数据： 



const diffArr = new Array(ROWS * COLS).fill(0);
for (let col = 0; col < COLS; col++) {
  for (let row = 0; row < ROWS; row++) {
    diffArr[row * ROWS + col] = arr[col * COLS + row];
  }
}


因此，现在，在diffArr中，原始数组中索引为[0,1,2，...]的对象现在设置如下[0,1000,2000，...，1，1,1001，2001，...，2，1002 ，2002，...]。数字表示对象的索引。这模拟了数组的排序，这是实现数据局部性设计模式的一种方法。



为了轻松地检查这一点，我们将稍微修改farAccess函数以获得一个自定义数组： 



function farAccess(array) {
  let data = arr;
  if (array) {
    data = array;
  }
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      data[j * ROWS + i].x = 0;
    }
  }
}


现在将脚本添加到我们的测试中： 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local')
  setTimeout(() => {
    repeat(() => farAccess(diffArr), 'Non Local Sorted')
  }, 2000);
}, 2000);
