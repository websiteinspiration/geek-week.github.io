Linuxには多くの面があります。どのディストリビューションで作業するか



あらゆるディストリビューションで実行されるバックアップアプリケーションを作成することは簡単な作業ではありません。Veeam Agent for LinuxがRHEL 6およびDebian 6からopenSUSE Leap 15.1およびUbuntu 19.04までのディストリビューションで確実に実行されるようにするには、特にカーネルモジュールがソフトウェア製品の一部であると考える場合、さまざまな問題を解決する必要があります。

この記事は、LinuxPiter 2019カンファレンスでのプレゼンテーションに基づいています。

Linuxは、最も人気のあるOSの1つだけではありません。実際、これは、独自の何か、独自の何かを実行できる基礎となるプラットフォームです。このおかげで、Linuxにはソフトウェアコンポーネントのセットが異なる多くのディストリビューションがあります。ここで問題が発生します。ソフトウェア製品がどのディストリビューションでも機能するためには、それぞれの特性を考慮する必要があります。

パッケージマネージャー。.debと.rpm


さまざまなディストリビューションに製品を配布するという明らかな問題から始めましょう。
ソフトウェア製品を配布する最も一般的な方法は、システムに組み込まれたパッケージマネージャーがそこからパッケージをインストールできるように、パッケージをリポジトリに置くことです。
ただし、rpmとdebの 2つの一般的なパッケージ形式があります。だから、誰もがサポートする必要があります。

debパッケージの世界では、互換性レベルは驚くべきものです。同じパッケージが等しくインストールされ、Debian 6とUbuntu 19.04の両方で動作します。古いDebianディストリビューションに定められた、パッケージの構築とそれらを使用するプロセスの標準は、新しいLinux Mintと基本OSに引き続き関連しています。したがって、Veeam Agent for Linuxの場合、ハードウェアプラットフォームごとに1つのdebパッケージで十分です。

しかし、rpmパッケージの世界では、大きな違いがあります。まず、Red HatとSUSEの2つの完全に独立したディストリビューターが存在するため、互換性は絶対に必要ありません。第二に、これらのディストリビューターはそれらからのディストリビューションを持っています。サポートと実験。それらの間で、互換性も必要ありません。el6、el7、およびel8では独自のパッケージであることが判明しました。Fedoraの個別のパッケージ。SLES11および12用のパッケージで、openSUSE用のパッケージです。主な問題は、依存関係とパッケージ名です。

パッケージマネージャーの代替


依存関係の問題は古く、長い間明らかにされてきました。依存関係の地獄を思い出してください。
さまざまなライブラリとアプリケーションを組み合わせて、すべてが安定して動作し、競合しないようにします。実際、どのLinuxディストリビューターもこの問題を解決しようとしています。CanonicalのSnappy

パッケージマネージャーは、この問題をまったく異なる方法で解決しようとしています。主なアイデア：アプリケーションは、メインシステムから分離および保護されたサンドボックスで実行されます。アプリケーションにライブラリが必要な場合は、ライブラリ自体がアプリケーション自体に付属しています。Flatpakでは、Linuxコンテナーを使用してサンドボックスでアプリケーションを実行することもできます。AppImageもあり、プログラムのポータブルイメージを作成できます。



これらのソリューションを使用すると、任意のディストリビューション用に1つのパッケージを作成できます。FlatpakとAppImageの場合、管理者の知識がなくてもアプリケーションのインストールと起動が可能です。

主な問題は、すべてのアプリケーションがサンドボックス内でroot権限なしで実行できるわけではないことです。一部はプラットフォームに直接アクセスする必要があります。カーネルモジュールは、カーネルに大きく依存していて、サンドボックスの概念に適合していません。

2番目の問題は、企業環境でのRed HatおよびSUSEからの人気のあるディストリビューションがSnappyおよびFlatpakをまだサポートしていないことです。

この点では、Linux用のVeeam社のエージェントは、上でもないsnapcraft.ioも上flathub.org。

パッケージマネージャーに関する質問の最後に、バイナリファイルとそれらを1つのパッケージにインストールするためのスクリプトを組み合わせることにより、パッケージマネージャーを完全に破棄するオプションがあることに注意します。

このようなバンドルにより、さまざまなディストリビューションおよびプラットフォーム用の1つの共通パッケージを作成し、インタラクティブなインストールプロセスを実行して、必要なカスタマイズを実行できます。Linux向けのこのようなパッケージをVMwareからしか知りませんでした。

問題を更新



すべての依存関係の問題が解決されたとしても、プログラムは同じディストリビューションでは異なる動作をする場合があります。重要なのはアップデートです。

3つのアップグレード戦略があります。

  • 最も簡単なのは更新することです。サーバーを構成し、忘れてしまった。すべてが機能するのになぜ更新するのですか？問題は、サポートに初めて連絡したときに始まります。ディストリビューションの作成者は、更新されたリリースのみをサポートしています。
  • ディストリビューターを信頼して、自動更新をセットアップできます。この場合、更新が失敗した直後にサポートに連絡する可能性があります。
  • テストインフラストラクチャで実行した後でのみ手動で更新するオプションは、最も忠実ですが、費用と時間がかかります。誰もがそれを買う余裕はありません。

ユーザーごとに異なる更新方法を使用するため、最新リリースと以前にリリースされたすべてのリリースの両方をサポートする必要があります。これは、開発プロセスとテストプロセスを複雑にし、サポートサービスに頭痛の種を追加します。

静的および/または動的リンク



そして、ここに「ライブラリにリンクする方法-動的または静的に？」という質問があります。議論する価値があります。

原則として、Linux用のC / C ++アプリケーションは動的リンクを使用します。アプリケーションが特定のディストリビューション用に特別に構築されている場合、これはうまく機能します。

タスクが1つのバイナリファイルでさまざまなディストリビューションをカバーすることである場合、サポートされている最も古いディストリビューションに焦点を合わせる必要があります。私たちにとって、これはRed Hat 6です。gcc4.4が含まれていますが、C ++ 11標準でも完全にはサポートされていません。

C ++ 14を完全にサポートするgcc 6.3を使用してプロジェクトを構築しています。当然、この場合、Red Hat 6では、libstdc ++をドラッグして、ライブラリをブーストする必要があります。それらにリンクする最も簡単な方法は静的です。

しかし残念ながら、すべてのライブラリを静的にリンクできるわけではありません。

まず、libfuse、libblkidなどのシステムライブラリを動的にリンクして、カーネルとそのモジュールとの互換性を確保する必要があります。

第二に、ライセンスには微妙な点があります。

GPLライセンスでは、基本的にオープンソースコードでのみライブラリをリンクできます。MITとBSDは静的リンクを有効にし、ライブラリをプロジェクトに含めることができます。しかし、LGPLは静的リンクと矛盾していないようですが、リンクに必要なファイルを共有する必要があります。

一般に、動的リンクを使用すると、何かを提供する必要がなくなります。

C / C ++アプリケーションの構築


さまざまなプラットフォームやディストリビューション向けのC / C ++アプリケーションを構築するには、適切なバージョンのgccを取得またはアセンブルし、特定のアーキテクチャ用のクロスコンパイラを使用して、ライブラリのセット全体を収集するだけで十分です。この作業はかなり実行可能ですが、かなり面倒です。また、選択したコンパイラとライブラリが実行可能なオプションを提供するという保証はありません。

明らかなプラス：アセンブリプロセス全体を1台のマシンで実行できるため、インフラストラクチャが大幅に簡素化されます。さらに、1つのアーキテクチャ用のバイナリファイルのセットを1つ収集するだけで十分であり、それらを異なるディストリビューション用のパッケージにパックできます。これは、Veeam Agent for Linuxのveeamパッケージが構築される方法です。

このオプションとは対照的に、ビルドファーム、つまりアセンブリ用の複数のマシンを簡単に準備できます。このような各マシンは、特定のディストリビューションおよび特定のアーキテクチャー向けのアプリケーションのコンパイルとパッケージのアセンブリを提供します。この場合、ディストリビューターが用意した方法で編集を行います。つまり、コンパイラーの準備段階とライブラリーの選択が不要になります。さらに、組み立てプロセスを簡単に並列化できます。

ただし、このアプローチにはマイナスがあります。同じアーキテクチャ内のディストリビューションごとに、独自のバイナリファイルセットを収集する必要があります。また、大量のディスク領域とRAMを割り当てるために、多くのマシンを保守する必要があることもマイナスです。

したがって、Red Hatディストリビューション用のveeamsnapカーネルモジュールのKMODパッケージが収集されます。

オープンビルドサービス


SUSEの同僚は、アプリケーションのコンパイルとパッケージのビルドのための特別なサービスであるopenbuildserviceとして、中間点を実装しようとしました。

実際、仮想マシンを作成し、そこに必要なすべてのパッケージをインストールし、アプリケーションをコンパイルし、この分離された環境でパッケージをコンパイルするのはハイパーバイザーであり、その後、そのような仮想マシンは解放されます。



OpenBuildServiceに実装されたスケジューラーは、最適なパッケージビルド速度で実行できる仮想マシンの数を決定します。組み込み署名メカニズム自体がパッケージに署名し、それらを組み込みリポジトリに配置します。組み込みのバージョン管理システムは、変更とアセンブリの履歴を保存します。このシステムにソースコードを追加するだけです。サーバー自体もレイズする必要はありませんが、オープンなサーバーを使用できます。

ただし、ここでは問題があります。そのような組み合わせは、既存のインフラストラクチャに適合させることが困難です。たとえば、バージョン管理は必要ありません。ソースにはすでに独自のコントロールがあります。署名メカニズムは異なります。特別なサーバーが使用されます。リポジトリも必要ありません。

さらに、他のディストリビューション（たとえば、Red Hat）のサポートは、十分に実装されておらず、理解できます。

このサービスの利点は、SUSEディストリビューションの次のバージョンの迅速なサポートです。公式リリースの発表の前に、アセンブリに必要なパッケージがパブリックリポジトリにアップロードされます。OpenBuildServiceの利用可能なディストリビューションのリストに新しいものが表示されます。チェックを入れて、組立計画に追加します。したがって、ディストリビューションの新しいバージョンの追加は、ほぼワンクリックで実行されます。

私たちのインフラストラクチャでは、OpenBuildServiceを使用して、SUSEディストリビューション用のveeamsnapカーネルモジュールのさまざまなKMPパッケージを収集しています。

さらに、カーネルモジュールに固有の問題について詳しく説明します。

カーネルABI


Linuxカーネルモジュールは、これまでソース形式で配布されてきました。実際のところ、カーネルの作成者は、カーネルモジュールの安定したAPIを維持するために自分自身に負担をかけることはなく、さらにバイナリレベル、さらにはkABIに負担をかけません。

バニラカーネル用のモジュールを構築するには、この特定のカーネルのヘッ​​ダーが必要であり、このコアでのみ機能します。

DKMSを使用すると、カーネルの更新時にモジュールのアセンブリプロセスを自動化できます。その結果、Debianリポジトリのユーザー（およびその多くの親戚）は、ディストリビューターリポジトリから、またはDKMSを使用してソースからアセンブルされたカーネルモジュールを使用します。

ただし、この状況はエンタープライズセグメントでは特に快適ではありません。独自のコードディストリビュータは、コンパイルされたバイナリの形式で製品を配布したいと考えています。

管理者は、セキュリティ上の理由から、本番サーバー上に開発ツールを保持したくないと考えています。 Red HatやSUSEなどのエンタープライズLinuxディストリビューターは、ユーザーのために安定したkABIを維持できると判断しました。その結果、Red Hat用のKMODパッケージとSUSE用のKMPパッケージが登場しました。

このソリューションの本質は非常に簡単です。ディストリビューションの特定のバージョンでは、カーネルAPIはフリーズしています。ディストリビューターは、彼がカーネル（たとえば3.10）を使用することを宣言し、カーネルインターフェースに影響を与えない修正と改善のみを行います。最初のカーネル用にアセンブルされたモジュールは、再コンパイルせずに後続のすべてのモジュールに使用できます。

Red Hatは、ライフサイクル全体にわたるディストリビューションのkABI互換性を発表しました。つまり、rhel 6.0（2010年11月リリース）用にアセンブルされたモジュールは、バージョン6.10（2018年6月リリース）でも動作するはずです。そして、これはほぼ8年です。当然、タスクはかなり複雑です。
kABIの互換性の問題により、veeamsnapモジュールが機能しなくなったいくつかのケースを記録しました。

RHEL 7.0用にコンパイルされたveeamsnapモジュールはRHEL 7.5のカーネルと互換性がないことが判明しましたが、サーバーをロードしてドロップすることが保証されていたため、RHEL 7のkABI互換性の使用を一般的に拒否しました。

現在、RHEL 7のKMODパッケージには、各リリースバージョンのアセンブリと、モジュールのロードを提供するスクリプトが含まれています。

SUSEはより慎重にkABI互換性タスクに取り組みました。これらは、1つのサービスパックでのみkABI互換性を提供します。

たとえば、SLES 12のリリースは2014年9月に行われました。また、SLES 12 SP1は2015年12月にすでにリリースされています。つまり、1年以上経過しています。どちらのリリースも3.12カーネルを使用しますが、kABIとの互換性はありません。明らかに、kABI互換性をわずか1年間維持することははるかに簡単です。コアモジュールの年次更新サイクルは、モジュールの作成者に問題を引き起こすべきではありません。

このSUSEポリシーの結果、veeamsnapモジュールのkABI互換性に関する問題は修正されませんでした。確かに、SUSEのパッケージ数はほぼ1桁多いです。

パッチとバックポート


ディストリビューターはkABIの互換性とカーネルの安定性を確保しようとしているという事実にもかかわらず、パフォーマンスを改善し、この安定したカーネルの欠陥を排除しようとしています。

同時に、エンタープライズLinuxカーネルの開発者は、独自の「エラーへの取り組み」に加えて、バニラカーネルの変更を追跡し、それらを「安定した」カーネルに転送します。

時々これは新しいエラーにつながります。

Red Hat 6の最新リリースでは、マイナーアップデートの1つに誤りがありました。これにより、スナップショットがリリースされたときに、veeamsnapモジュールがシステムをクラッシュさせることが保証されていました。更新前後のカーネルソースを比較することで、バックポートが原因であることがわかりました。同様の修正がバニラカーネルバージョン4.19で行われました。しかし、バニラコアでのみ、この修正は正常に機能し、「安定した」2.6.32に転送すると、スピンロックに問題がありました。

もちろん、誰もが常にエラーを抱えていますが、コードを4.19から2.6.32にドラッグする価値があり、安定性にリスクがありましたか？

最悪の場合、綱引きの「安定性」<->「近代化」マーケティングが関連している場合。マーケティング部門は、更新されたディストリビューションの中核を安定させると同時に、パフォーマンスを向上させ、新しい機能を搭載する必要があります。これは奇妙な妥協につながります。

SLES 12 SP3から4.4カーネルでモジュールをビルドしようとしたとき、その中にバニラ4.8の機能が含まれていることに驚きました。私の意見では、SLES 12 SP3のカーネル4.4のブロックI / Oの実装は、SLES12 SP2の安定した4.4カーネルの以前のリリースよりも4.8カーネルに似ています。 4.8カーネルからSLES 4.4 SP3に転送されたコードの割合を判断することはできませんが、カーネルを同じ安定版4.4と呼ぶ機会はまだありません。

これについて最も不愉快なことは、異なるコアで同様にうまく機能するモジュールを書くとき、あなたはもはやカーネルバージョンに依存することができないということです。また、分布も考慮する必要があります。新しい機能とともに表示される定義に参加できることは良いことですが、この機能が常に表示されるとは限りません。

その結果、コードは条件付きコンパイルのためのファンシーなディレクティブに囲まれています。

文書化されたカーネルAPIを変更するパッチもあります。KDEネオン 5.16
配布キットに出くわし、このカーネルバージョンでのlookup_bdev呼び出しが入力パラメーターのリストを変更したことを確認して、非常に驚​​きました。

まとめるには、lookup_bdev関数にマスクパラメーターがあるかどうかを確認するスクリプトをmakefileに追加する必要がありました。

カーネルモジュールの署名


しかし、パッケージ配布の問題に戻ります。

安定したkABIの利点の1つは、カーネルモジュールをバイナリファイルとして署名できることです。この場合、開発者は、モジュールが誤って損傷したり、意図的に変更されていないことを確認できます。これはmodinfoコマンドで確認できます。

Red HatおよびSUSEディストリビューションでは、適切な証明書がシステムに登録されている場合にのみ、モジュールの署名を検証してダウンロードできます。証明書は、モジュールの署名に使用される公開鍵です。別パッケージで配布しています。

ここでの問題は、証明書がカーネルに組み込まれる（ディストリビューターによって使用される）か、mokutilユーティリティを使用して不揮発性EFIメモリに書き込まれる必要があることです。Mokutilユーティリティ証明書をインストールするときは、システムを再起動する必要があり、オペレーティングシステムのカーネルをロードする前でも、新しい証明書のダウンロードを許可するように管理者に指示します。

したがって、証明書を追加するには、管理者がシステムに物理的にアクセスする必要があります。マシンがクラウドのどこかにあるか、リモートサーバールームにあり、ネットワーク経由（sshなど）でのみアクセスできる場合、証明書を追加することはできません。

仮想マシン上のEFI


EFIがマザーボードのほとんどすべての作成者によってサポートされているという事実にもかかわらず、システムをインストールするとき、管理者はEFIの必要性について考えず、無効にすることができます。

すべてのハイパーバイザーがEFIをサポートしているわけではありません。VMWare vSphereは、バージョン5
以降のEFIをサポートしています。MicrosoftHyper-Vは、Windows Server 2012R2用のHyper-V以降のEFIサポートも受けています。

ただし、デフォルト構成では、この機能はLinuxマシンでは無効になっています。つまり、証明書をインストールできません。

vSphere 6.5では、Flash経由で機能する古いバージョンのWebインターフェイスでのみセキュアブートオプションを設定できます。HTML-5のWeb UIははるかに遅れています。

実験的分布


そして最後に、実験的ディストリビューションと公式サポートのないディストリビューションの問題を検討してください。一方では、そのようなディストリビューションが真面目な組織のサーバーで見つかることはほとんどありません。そのようなディストリビューションに対する公式のサポートはありません。したがって、それらを提供します。そのようなディストリビューションでの製品サポートは不可能です。

ただし、このようなディストリビューションは、新しい実験的ソリューションをテストするための便利なプラットフォームになります。たとえば、Fedora、OpenSUSE Tumbleweed、または不安定版のDebian。彼らはかなり安定しています。彼らは常に新しいバージョンのプログラムと常に新しいカーネルを持っています。 1年後、この実験的な機能は更新されたRHEL、SLES、またはUbuntuに含まれる可能性があります。

したがって、実験的な配布キットで何かが機能しない場合、これは問題を整理して解決する機会です。この機能が間もなくユーザーの本番サーバーに表示されるようになる準備をする必要があります。
