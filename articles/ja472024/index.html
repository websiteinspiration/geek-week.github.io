バージョン3.0：改善

ビジネスは常にIT製品を開発しています。ここに「瞬間を止める」ことはありません。さもなければ、最高のプログラムでさえも必然的に時代遅れになるでしょう。ヨーロッパ向けの新しいバージョンの医療アプリケーションを作成した方法と、同時に解決された問題について説明します。



病院アプリ


私たちはヨーロッパの病院向けの医療アプリケーションに取り組んでいます。書類業務を減らすことで、医師は患者との時間をより多く費やすことができます。医師は患者と検査に関する情報を指示し、アプリケーションは音声録音をテキスト形式に変換し、テンプレートに記入し、医療従事者と患者のためのドキュメントを生成します。作業の各段階で、独自のビジネスロジックといくつかの外部システムとの統合が確立されます。

クライアントは私たちに、投資家向けのデモ用のアプリケーションの新しいバージョンを開発するタスクを設定しました。



プロジェクトの詳細


オーディオ


バージョン2.0でオーディオを録音する方法：

  1. アプリケーションを開きました。
  2. レコードの追加ボタンをクリックしました。
  3. 表示されたウィンドウで、必要なバージョンの録音デバイスを選択しました。
  4. ディクテーションされたオーディオ録音。
  5. 患者番号、訪問日、訪問番号（訪問=医師の予約）が入力されました。
  6. 彼らは[完了]ボタンをクリックして、音声と訪問データをサーバーにアップロードしました。

現在、バージョン3.0では、記述に必要な労力が少なくなっています。

  1. 医師がアプリケーションを開きます。
  2. リストから予約（日付、時刻、番号、または患者名）を選択し、1回クリックして訪問カードを開きます。
  3. オーディオを録音します。
  4. Completeボタンを押して、オーディオをサーバーに送信します。

バージョン3.0では、医師の作業は可能な限り自動化されています。プログラム自体が患者に関するデータを追加する一方で、手術の回数は減りました。

文字で作業する


文字の操作も簡単になりました。バージョン2.0では、処理のために困難なキューがありました。医師はすぐに手紙の完全なリストを受け取ることができませんでした-部分的に、特定の順序でのみ。開始するには、以下を行う必要がありました。

  • クリックして手紙のリストを取得します。
  • 別のウィンドウに移動します。
  • 文字をダブルクリックして開きます。
  • リストからの手紙を処理した後、もう一度クリックすると、リストに次の手紙が表示されます。

バージョン3.0では、医師は処理可能なすべての文字を確認できます。彼は2つの方法でドキュメントを選択できます-リストで手動で、またはフィルターとソートを使用して（それらを保存して、ワンクリックでさらにレターを開くことができます）。手紙を開くには、一度クリックするだけです。

インターフェース


一般に、バージョン2.0のインターフェースは扱いにくく、不便でした。アプリケーションの最初のタスクは、紙のワークフローを減らし、オーディオ録音を操作することにより、医療専門家の時間を節約することでした。実際には、アプリケーションはこのタスクを適切に処理できませんでした。記録を作成するために、医師は長いドロップダウンリストで多くの設定を選択する必要がありました。

私たちの専門家はインターフェースを操作し、ユーザーが最少のクリック数でタスクを完了することができるようにオーディオボタンを前面に表示しました。

次に、新しいバージョンの開発方法について詳しく説明します。

新しいニーズ


クライアントがバージョン2.0のアップグレードを要求したとき、アプリケーションは約3年間動作していました。これはユーザーに馴染みがあり、特定の利点がありました。

  • 複雑なビジネスロジック、サードパーティシステムとの統合を実行するための一連の関数がありました。
  • クライアントはプログラムに慣れており、プログラムを放棄したくありません。
  • テクニカルサポートスタッフはアプリケーションを知っていて、ユーザーをすばやく支援しました。
  • ビジネス上の要望に応じてシステムを構成するための柔軟な設定がありました。
  • システムのサーバー部分で起こりうる問題の追跡が確立され、それらを解決するための回避策がわかっていました。

ただし、アプリケーションの動作を分析したところ、次のような問題が見つかりました。

  • 新機能の開発とテストには、ますます時間がかかりました。
  • 新しい機能を追加すると、システムにエラーが表示されました。
  • その結果、複雑な改善の最大30％が既製になり、アプリケーションを古くする恐れがありました。たとえば、病院ではますます複雑なテンプレートが表示され、ワー​​クフローに新しいロールを追加する必要がありました。
  • アプリケーションアーキテクチャは新しいソリューションをサポートできませんでした。
  • 開発時間の40％はサポートに費やされました。
  • デスクトップ製品の新しいバージョンとアップデートをインストールするときに問題がありました。
  • 2000年代の扱いにくいインターフェースは、新しい顧客を怖がらせていました。
  • 設定の不便なシステムにより、システムが迅速に起動できなくなり、エラーのリスクも高まりました。

強みと課題を評価する際、アプリケーションをよりモバイル（デスクトップではなくWebバージョン）にし、便利で競争力があり、ビジネスタスクに簡単に適応できるようにする必要があるという結論に達しました。

バージョン要件


アプリケーションの機能を分析し、製品をビジネスにとって価値のあるものにする最も重要なものを特定しました。このデータに基づいて、新しいバージョンでのプログラムを決定しました。

  • 医師やその他のユーザーにとって直感的なアプリケーションの主要な機能が必要です。
  • ユーザー（ヘルスケアプロバイダーおよびサポートスタッフ）がプログラムの使用方法を簡単に習得できるようにする必要があります。
  • 極端な条件（停電、不安定なインターネットアクセスなど）でもデータの損失をなくします。
  • システムによって生成されたドキュメントは、法律の基準と要件に準拠している必要があります。
  • システムの更新中は、ユーザーおよびサポートサービスの不便を最小限に抑える必要があります。
  • 分散された疎結合コンポーネントに基づいてサービス指向のモジュール式アーキテクチャを作成する必要があります。これにより、複雑なソフトウェアシステムの構築にそれらを使用できるようになります。
  • Active Directoryを使用して、作業環境を均一に構成します。

さらに、新しいバージョンを古いバージョンの「複雑なクローン」にすることは不可能でした。したがって、アプリケーションをオーバーロードすることなく、主要な機能を保存する必要がありました。冗長な複雑な設定を容赦なく放棄しました。

バージョン2.0を使用するビジネスアナリストは、変更をすぐには受け入れませんでした。参照に関して、「バージョン2.0のようにする必要がある」、「バージョン2.0で作業スキーマを使用する」というフレーズがよく見つかりました。この段階で最も困難だったのは、以前のバージョンのように、すべてを取りたいという誘惑への抵抗でした。

プロジェクトチーム


原則として、各プロジェクトの開始時に、私たちSimbirSoftは、アナリスト、品質保証スペシャリスト（QA）など、さまざまなプロファイルのエキスパートのチームに接続します。医療アプリケーションに取り組むとき、私たちは次のチームをまとめました：

  • コードを記述し、バージョン2.0の機能を適合させた開発者。
  • 品質保証スペシャリスト（QA）。彼らは、開発者とともに、レガシーコード、アーキテクチャソリューション、バージョン2.0のエラーに精通しており、新しい機能もテストしました。
  • テストオートメーションエンジニア（SDET）。デスクトップバージョンとWebバージョンのテストケースの自動検証を設定します。
  • ビジネス・インテリジェンス。彼らは顧客や医師と話をし、ビジネスプロセスがどのように構築されたか、アプリケーションの要件と希望は何かを発見しました。その後、チーム全体が理解できるビジネスプロセス図を作成しました。
  • デザイナーおよびユーザビリティの専門家。彼らはユーザーフレンドリーなインターフェースを開発する責任がありました。
  • 時間の管理とスケジュールに関与したプロジェクトマネージャー。

このプロセスでは、申し立てられているリスクの表を新しいバージョンで常に維持しています。それらを防ぐために、アプリケーション設定の柔軟なシステムを考え、テストを自動化してユーザーをエラーから保護しました。特に、SDETスペシャリストは、コード内のすべての変更をすばやく確認し、回帰テストに費やす時間を短縮できるフレームワークを作成しました。

課題と解決策


アプリケーションをアップグレードする際に、いくつかの困難な段階に遭遇しました。これらについて以下で説明します。

アプリのデザイン


以前のバージョンは扱いにくいインターフェースだったので、アプリケーション設計を再設計しました。5つの予備オプションを提案し、アプリケーションのユーザーの中からフォーカスグループに提示し、ユーザビリティテストを実施しました。その結果、医療専門家は1つのオプションを選択しましたが、彼らの意見では、最も便利で魅力的で使いやすいことがわかりました。

さまざまなブラウザー用のプラグイン開発


アプリケーションに関連するさまざまな技術的リスクを提供しています。たとえば、実装用に選択されたプラグインは、一部のインターネットブラウザには適さない場合があります。このリスクを軽減するために、私たちはまず、ほとんどのパートナー医療機関で使用されているソフトウェアのプラグインを開発しました。将来的には、他のブラウザ用のプラグインを静かに変更しました。

無効なビジネス仮説


私たちの仕事は、投資家に提示するための製品の限定バージョンを開発することでした。このため、最も必要な機能のみをアプリケーションに実装するようにしました。私たちは顧客のいくつかの仮説を分析し、それらを実装することを拒否しました。

たとえば、顧客は医療専門家の仕事を計画するためのカレンダーを作成することを提案しました。仮説によると、カレンダーは医師の仕事をより便利にすることができます。ただし、実際にはこの機能は成功しなかったため、最終的には無効にされました。その後、カレンダーは別のユーザーグループのニーズに合わせて調整されました。つまり、医療従事者ではなく患者です。無効な仮説─これはビジネスの不可欠な部分であり、それらに備える必要があります。

外部システムとの統合


レターを送信および保存するためにアプリケーションを外部システムと統合する手順についてパートナーと合意するのに長い時間がかかりました。

アプリケーション─医療施設─のユーザーは、バージョン3.0の古い使い慣れた統合システムを使用したいと考えていましたが、変更できませんでした。私たちのパートナーは、この場合、統合は高速になると提案しました。ただし、実際には、多くのタスクが統合に関連付けられていました。

  • メール送信システムの仕組みに関する一般的な情報を収集します。
  • 2.0にあった重大なバグのリストを作成します。
  • 分析や開発の段階でこれらのケースを検討し、それらを考慮に入れて、同じレーキを踏まないようにしてください。
  • バージョン2.0の機能がバージョン3.0のプロセスに適しているかどうか、または何かを変更する必要があるかどうかを分析します。変更の場合は、特定の機能が必要な理由を毎回指定し、アナリストだけでなく、お客様の技術スペシャリストと通信してください。

お客様との交渉の過程で、統合作業に時間がかかることを証明できました。私たちのパートナーは私たちに同意しました：あるバージョンから別のバージョンにコードを取得して転送することはできません。

テストと分析


プロジェクトの以前のバージョンは、アナリストの参加なしで作成されました。開発者とQAスペシャリストは、アプリケーションの作成プロセスで要件を指定しました。3番目のバージョンはすでにアナリストの要件に基づいていましたが、テストには依然として問題がありました。

  • さまざまなチームがプロジェクトに取り組みました。
  • 大量の並列タスクがありました。
  • スプリント中に、要件とタスクはしばしば変更されました。
  • 新しい機能と承認済みの機能との相互作用を考慮する必要がありました。

新しいバージョンの製品で作業するには、たとえば次のように、個々のワークフローを変換する必要がありました。

  • 開発側とQAからの分析を強化し、さらに1時間追加しました。
  • レビューの要件でテストされている機能の目的を示すのはルールでした。これにより、アナリストのタスクに対する理解が向上し、最適なソリューションを提案する機会が提供されました。

お客様との定期的なオンライン集会は、彼のビジネスプロセスの理解を深めるのに役立ちました。会話の中で、私たちのパートナーは医師の働きについて詳細を話し、ビジネス目標を説明しました。次に、技術的なニュアンスを説明し、さまざまな自明でないケースを示しました。これにより、医療機関のニーズをカバーし、実装コストの点で最適な製品要件を策定することができました。

概要


作業プロセスの柔軟性とビジネスニーズへの配慮により、開発プロセス中に発生したすべての困難を克服することができました。欧州の医療機関向けのアプリケーションの新しいバージョンを作成し、立ち上げました。

現在、製品の開発と新機能の導入を続けています。実際のユーザーがシステムをどのように使用しているかを調べ、原因不明のケースとユーザーストーリーを収集し、新しいビジネス価値を特定します。

製品の新しいバージョンを作成するとき、開発者は通常、私たちと同じ問題に直面します。たとえば、次のようになります。

  • ビジネス側からの誤った仮説が可能です。
  • ユーザーの欲求に矛盾があります（すべてをそのままにするか、新しい方法で作り直します）。
  • より良い結果を達成するために、チームの作業を再構成する必要がある場合があります。

主なことは、IT製品の新しいバージョンのリリースは、以前のバージョンの「Ctrl + C」、「Ctrl + V」コードのコピーではなく、ゼロからの本格的な開発であることです。これは、ビジネスプロセス、ユーザー要件、およびITソリューションの市場の状況も変化しているためです。
