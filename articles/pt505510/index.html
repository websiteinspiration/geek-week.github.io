Simplificando a API do Check Point com o Python SDK

Todo o poder de interagir com a API é revelado quando usado em conjunto com o código do programa, quando é possível gerar dinamicamente solicitações e ferramentas da API para analisar respostas da API. No entanto, o Python Software Development Kit (doravante referido como Python SDK) para a API de gerenciamento de ponto de verificação , mas em vão , permanece invisível . Simplifica significativamente a vida dos desenvolvedores e amantes da automação. Ultimamente, o Python ganhou imensa popularidade e eu decidi preencher a lacuna e fornecer uma visão geral dos principais recursos do Kit de Desenvolvimento Python da Check Point API . Este artigo é um excelente complemento para outro artigo sobre a API Habré Check Point R80.10. Gerenciamento através de CLI, scripts e muito mais. Consideraremos como escrever scripts usando o Python SDK e detalharemos mais sobre a nova funcionalidade da API de gerenciamento na versão 1.6 (suportada desde R80.40). Para entender o artigo, você precisará de conhecimentos básicos sobre como trabalhar com APIs e Python.

A Check Point está desenvolvendo ativamente a API e, no momento, existem:

  • API de gerenciamento de ponto de verificação (versão atual 1.6) - trabalhe com o servidor de gerenciamento por meio da API (e a capacidade de executar scripts em gateways sob o controle do servidor de gerenciamento)
  • API GAIA do Check Point (versão atual 1.4) - trabalhe com gateways de segurança
  • API de prevenção de ameaças 1.0 - Sandboxing na nuvem de ponto de verificação
  • API do Identity Awareness - trabalhando com o blade Identity Awareness em gateways
  • API do Security Management Portal - trabalhando com o portal de gerenciamento de gateway SMB ( Saiba mais sobre gateways SMB )
  • API IoT - interação com controladores IoT
  • API do CloudGuard Connect - trabalhando com o CloudGuard Connect (solução de segurança SD-WAN)
  • API do Dome9 - trabalhando com o Dome9

Atualmente, o SDK do Python suporta apenas a interação com a API de gerenciamento e a API do Gaia . Abordaremos as classes, métodos e variáveis ​​mais importantes deste módulo.



Instalação do módulo


O módulo cpapi é instalado de maneira rápida e fácil a partir do repositório oficial do Check Point no github usando pip . Instruções detalhadas de instalação estão em README.md . Este módulo está adaptado para trabalhar com versões do Python 2.7 e 3.7. Neste artigo, exemplos serão fornecidos usando o Python 3.7. No entanto, o SDK do Python pode ser executado diretamente do servidor de gerenciamento Check Point (Smart Management), mas eles suportam apenas o Python 2.7, portanto, a última seção fornecerá o código para a versão 2.7. Imediatamente após a instalação do módulo, eu recomendo a olhar para os exemplos nas examples_python2 e examples_python3 diretórios .

Início do trabalho


Para podermos trabalhar com os componentes do módulo cpapi , pelo menos duas classes necessárias devem ser importadas do módulo

cpapi : APIClient e APIClientArgs

from cpapi import APIClient, APIClientArgs

A classe APIClientArgs é responsável por conectar-se ao servidor da API e a classe APIClient é responsável por interagir com a API.

Determinamos os parâmetros de conexão


Para definir vários parâmetros para conectar-se à API, você precisa criar uma instância da classe APIClientArgs . Em princípio, seus parâmetros são predefinidos e você pode omiti-los ao executar o script no servidor de gerenciamento.

client_args = APIClientArgs()

Porém, ao iniciar em um host de terceiros, você deve especificar pelo menos o endereço IP ou o nome do host do servidor da API (também é um servidor de gerenciamento). No exemplo abaixo, definimos o parâmetro de conexão do servidor e atribuímos a ele o endereço IP do servidor de gerenciamento como uma sequência.

client_args = APIClientArgs(server='192.168.47.241')

Vejamos todos os parâmetros e seus valores padrão que podem ser usados ​​ao se conectar ao servidor da API:

Argumentos para o método __init__ da classe APIClientArgs 
class APIClientArgs:
    """
    This class provides arguments for APIClient configuration.
    All the arguments are configured with their default values.
    """

    # port is set to None by default, but it gets replaced with 443 if not specified
    # context possible values - web_api (default) or gaia_api
    def __init__(self, port=None, fingerprint=None, sid=None, server="127.0.0.1", http_debug_level=0,
                 api_calls=None, debug_file="", proxy_host=None, proxy_port=8080,
                 api_version=None, unsafe=False, unsafe_auto_accept=False, context="web_api"):
        self.port = port
        # management server fingerprint
        self.fingerprint = fingerprint
        # session-id.
        self.sid = sid
        # management server name or IP-address
        self.server = server
        # debug level
        self.http_debug_level = http_debug_level
        # an array with all the api calls (for debug purposes)
        self.api_calls = api_calls if api_calls else []
        # name of debug file. If left empty, debug data will not be saved to disk.
        self.debug_file = debug_file
        # HTTP proxy server address (without "http://")
        self.proxy_host = proxy_host
        # HTTP proxy port
        self.proxy_port = proxy_port
        # Management server's API version
        self.api_version = api_version
        # Indicates that the client should not check the server's certificate
        self.unsafe = unsafe
        # Indicates that the client should automatically accept and save the server's certificate
        self.unsafe_auto_accept = unsafe_auto_accept
        # The context of using the client - defaults to web_api
        self.context = context


Acredito que os argumentos que podem ser usados ​​nas instâncias da classe APIClientArgs sejam intuitivos para os administradores do Check Point e não precisem de comentários adicionais.

Conecte-se via APIClient e gerenciador de contexto


A classe APIClient é mais convenientemente usada através do gerenciador de contexto. Tudo o que você precisa passar para uma instância da classe APIClient são os parâmetros de conexão que foram definidos na última etapa.

with APIClient(client_args) as client:

O gerenciador de contexto não fará automaticamente uma chamada de login para o servidor da API, mas fará uma chamada de logout quando sair. Se, por algum motivo, um logout não for necessário no final do trabalho com as chamadas da API, você precisará iniciar o trabalho sem usar o gerenciador de contexto:

client = APIClient(clieng_args)


Verificação de conexão


A maneira mais fácil de verificar se uma conexão passa pelos parâmetros fornecidos é usando o método check_fingerprint . Se a verificação do hash sha1 para impressão digital do certificado da API do servidor não passou (o método retornou False ), isso geralmente é causado por problemas de conexão e podemos parar o programa (ou permitir que o usuário corrija os dados da conexão):

    if client.check_fingerprint() is False:
        print("Could not get the server's fingerprint - Check connectivity with the server.")
        exit(1)

Note-se que no futuro, o APIClient classe irá verificar com cada chamada API ( api_call e api_query métodos , vamos falar sobre eles um pouco mais) certificado SHA1 impressão digital no servidor de API. Mas se um erro for detectado ao verificar a impressão digital sha1 do certificado da API do servidor (o certificado é desconhecido ou foi alterado), o método check_fingerprint fornecerá a capacidade de adicionar / alterar informações sobre ele na máquina local no modo automático. Essa verificação pode ser desativada completamente (mas isso só pode ser recomendada se os scripts forem executados no próprio servidor da API, quando conectado ao 127.0.0.1) usando o argumento APIClientArgs - unsafe_auto_accept (veja detalhes sobre APIClientArgs anteriormente em "Definindo parâmetros de conexão").

client_args = APIClientArgs(unsafe_auto_accept=True)

Efetue login no servidor da API


No APIClient, existem até 3 métodos de login no servidor da API e cada um deles valor zaponimaet sid (o ID da sessão), usado automaticamente em cada chamada subsequente da API no cabeçalho (o nome do título nessa configuração - o X-CHKP-sid ), portanto que você não precisa processar mais esse parâmetro.

Método de login


A opção usando o nome de usuário e a senha (no exemplo, o nome de usuário admin e a senha 1q2w3e são passados ​​como argumentos posicionais):

     login = client.login('admin', '1q2w3e')

Os parâmetros opcionais opcionais também estão disponíveis no método de logon.Listo seus nomes e valores padrão: 

continue_last_session=False, domain=None, read_only=False, payload=None

Método Login_with_api_key


Uma variante que usa a chave api (suportada desde a versão de gerenciamento R80.40 / Management API v1.6, "3TsbPJ8ZKjaJGvFyoFqHFA ==" é o valor da chave API para um dos usuários no servidor de gerenciamento com o método de autorização de chave API):

     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')

No método login_with_api_key , os mesmos parâmetros opcionais estão disponíveis como no método de login .

Método Login_as_root


A opção de login na máquina local com o servidor da API:

     login = client.login_as_root()

Apenas dois parâmetros opcionais estão disponíveis para este método:

domain=None, payload=None

api_query


Imediatamente, faça uma reserva de que esse método é aplicável apenas a chamadas cuja saída envolve deslocamento (shift). Essa conclusão ocorre quando contém ou pode conter uma grande quantidade de informações. Por exemplo, isso pode ser uma solicitação de uma lista de todos os objetos criados do tipo host no servidor de gerenciamento. Para essas solicitações, a API retorna uma lista de 50 objetos por padrão (você pode aumentar o limite para 500 objetos na resposta). E, para não extrair as informações várias vezes, alterando o parâmetro offset na solicitação da API, existe um método api_query que faz isso funcionar automaticamente. Exemplos de chamadas em que esse método é necessário: show-sessions, show-hosts, show-networks, show-wildcards, show-groups, show-address-range, show-simple-gateways, show-simple-clusters, show-access- funções, show-trust-clients, show-packages. De fato, no nome dessas chamadas de API, vemos palavras plurais, portanto, essas chamadas serão mais fáceis de manipular através de api_query

show_hosts = client.api_query('show-hosts')

Processando resultados de chamadas da API


Depois disso, você pode usar as variáveis ​​e os métodos da classe APIResponse (dentro e fora do gerenciador de contexto). A classe APIResponse possui 4 métodos e 5 variáveis ​​predefinidas; abordaremos os mais importantes com mais detalhes.



sucesso


Para começar, seria bom verificar se a chamada da API foi bem-sucedida e retornou o resultado. Existe um método de sucesso para isso :

In [49]: api_versions.success                                                   
Out[49]: True

Retorna True se a chamada da API foi bem-sucedida (o código de resposta é 200) e False se não for bem-sucedida (qualquer outro código de resposta). É conveniente usar imediatamente após a chamada da API, para que, dependendo do código de resposta, você possa exibir informações diferentes.

if api_ver.success: 
    print(api_versions.data) 
else: 
    print(api_versions.err_message)

statuscode


Retorna o código de resposta após fazer uma chamada de API.

In [62]: api_versions.status_code                                               
Out[62]: 400

Códigos de resposta possíveis: 200.400.401.403.404.409.500.501 .

set_success_status


Pode ser necessário alterar o valor do status de sucesso. Tecnicamente, você pode colocar qualquer coisa lá, mesmo uma string comum. Mas um exemplo real seria redefinir esse parâmetro para False sob certas condições concomitantes. Abaixo, preste atenção a um exemplo quando houver tarefas executadas no servidor de gerenciamento, mas consideraremos esta solicitação sem êxito (defina a variável success como False , apesar do fato de a chamada da API ter sido bem-sucedida e ter retornado o código 200).

for task in task_result.data["tasks"]:
    if task["status"] == "failed" or task["status"] == "partially succeeded":
        task_result.set_success_status(False)
        break

Exemplos úteis


A seguir, exemplos que usam chamadas de API que foram adicionadas à Management API 1.6.

Para começar, considere a operação de chamadas add-host e add-address-range . Suponha que precisamos criar todos os endereços IP da sub-rede 192.168.0.0/24 como objetos do tipo host, cujo último octeto seja 5, e gravar todos os outros endereços IP como objetos do intervalo de endereços. Nesse caso, o endereço de sub-rede e o endereço de broadcast devem ser excluídos.

Portanto, abaixo está um script que resolve esse problema e cria 50 objetos do tipo host e 51 objetos do tipo de endereços. Para resolver o problema, são necessárias 101 chamadas de API (sem contar a chamada de publicação final). Além disso, usando o módulo timeit, calculamos o tempo necessário para executar o script até que as alterações sejam postadas.

Script usando add-host e add-address-range 
import timeit
from cpapi import APIClient, APIClientArgs

start = timeit.default_timer()

first_ip = 1
last_ip = 4

client_args = APIClientArgs(server="192.168.47.240")

with APIClient(client_args) as client: 
     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')
     for ip in range(5,255,5):
         add_host = client.api_call("add-host", {"name" : f"h_192.168.0.{ip}", "ip-address": f'192.168.0.{ip}'})
     while last_ip < 255:
         add_range = client.api_call("add-address-range", {"name": f"r_192.168.0.{first_ip}-{last_ip}", "ip-address-first": f"192.168.0.{first_ip}", "ip-address-last": f"192.168.0.{last_ip}"})
         first_ip+=5
         last_ip+=5
     stop = timeit.default_timer() 
     publish = client.api_call("publish")
     
print(f'Time to execute batch request: {stop - start} seconds')


No meu ambiente de laboratório, leva 30 a 50 segundos para executar esse script, dependendo da carga no servidor de gerenciamento.

Agora vamos ver como resolver o mesmo problema usando a API de chamada add-objects-batch , cujo suporte foi adicionado à API versão 1.6. Essa chamada permite criar muitos objetos ao mesmo tempo em uma solicitação de API. Além disso, eles podem ser objetos de diferentes tipos (por exemplo, hosts, sub-redes e intervalos de endereços). Assim, nossa tarefa pode ser resolvida na estrutura de uma única chamada de API.

Script usando add-objects-batch 
import timeit
from cpapi import APIClient, APIClientArgs

start = timeit.default_timer()

client_args = APIClientArgs(server="192.168.47.240")

objects_list_ip = []
objects_list_range = []

for ip in range(5,255,5):
    data = {"name": f'h_192.168.0.{ip}', "ip-address": f'192.168.0.{ip}'}
    objects_list_ip.append(data)
    
first_ip = 1
last_ip = 4


while last_ip < 255:
    data = {"name": f"r_192.168.0.{first_ip}-{last_ip}", "ip-address-first": f"192.168.0.{first_ip}", "ip-address-last": f"192.168.0.{last_ip}"}
    objects_list_range.append(data)
    first_ip+=5
    last_ip+=5

data_for_batch = {
  "objects" : [ {
    "type" : "host",
    "list" : objects_list_ip
}, {
    "type" : "address-range",
    "list" : objects_list_range
  }]
}


with APIClient(client_args) as client: 
     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')
     add_objects_batch = client.api_call("add-objects-batch", data_for_batch)
     stop = timeit.default_timer() 
     publish = client.api_call("publish")
     
print(f'Time to execute batch request: {stop - start} seconds')


E leva de 3 a 7 segundos para executar esse script no meu ambiente de laboratório, dependendo da carga no servidor de gerenciamento. Ou seja, em média, em um objeto 101 API, uma chamada de tipo de lote é 10 vezes mais rápida. Em mais objetos, a diferença será ainda mais impressionante.

Agora vamos ver como trabalhar com set-objects-batch . Usando esta chamada de API, podemos alterar massivamente qualquer parâmetro. Vamos definir a primeira metade dos endereços do exemplo anterior (para 0,124 hosts e intervalos também) a cor sienna e a segunda metade dos endereços que atribuiremos à cor khaki.
