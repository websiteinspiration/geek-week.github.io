Complex code or story of one interview

I would like to share my story of failure, if we mean by this a negative feedback from the company after the review of the technical task. Of course, each such experience is always a great opportunity to revise the strategy, to draw certain conclusions from one's own or others' mistakes. And in this case, I have a great reason to reflect on the experience of the interview, even though it did not end with the offer. I want to express my gratitude in advance for the time you spend reading this article with the code and the effort put into understanding them. I want to express my double gratitude if the reader shares his opinion.



All source code is available in the public repository on GitHub at this link . He has not undergone any changes since the publication, despite the fact that there are a number of points in it that I would bring to mind, but intentionally did not. There is always some kind of agreement with ourselves regarding the time we spend on the task. We are trying to prioritize based on personal preferences, unless otherwise specified in the requirements for the assignment: someone will prefer to write documentation on their APIs, and someone will decide to increase test coverage. In my case, as you can see, there are no e2e tests and unit tests of UI components, CSS is also whipped up and no methodology follows. It is likely that you can continue this list with other shortcomings.



Impressed by the profile of one company, backed up by the recall of a former colleague who worked there once, I decided to send my resume there and begin the interview process. After a week or two, the recruiter contacted me and offered to talk with the leader of one of the teams. The conversation with the leader came down more to general questions: what do you like, what technologies do you use, what are you proud of, what problems did you solve, and there was also room for my questions - a completely ordinary conversation, quite typical for an IT company. Some time later that day, the task was dropped. Below I will write the requirements of this assignment. The text of the task is saved in the original:

You are invited to write a service for playing tic-tac-toe together.

The game contains several playing fields, each player can either join the game on an existing field or create a new field.

The initial screen contains a list of playing fields and their brief description (how many players are playing now, when was the last move). Data is updated in real time.

The playing field has a fixed size of 10 x 10. Several players can play on each field. Each player can make a move at any time, but cannot make several moves in a row.

The game server can store the state of the game process in memory.

You can implement the solution on any technologies that are convenient for you, but it will be great if you make them on [project technologies are designated]

Of the features of the server code, I note that to protect against race conditions, the ReentrantLock map is used - for the game session, its ReentrantLock. I decided not to do the implementation of synchronization with client time, but simply generate the running time on the server.



After the first feedback about UX with a proposal to fix and add some things, I also decided to rewrite the frontend on functional components with hooks. I have long wanted to test this approach and I liked it. The authors of reactjs in the documentation emphasize that this approach minimizes the number of errors made by programmers related to the component‚Äôs life cycle when using a style based on classes and callbacks - and I think this is an important point. Of course, I would like to have a conceptual discussion, to be able to defend certain decisions. I accept a comment regarding CSS, but I totally don't understand the rest. Is it over-engineering? Or am I faced with the technical immaturity of the interviewers? In principle, here any assessment will be very subjective and in its own way takes place on existence.



Thinking about the meaning of the question - What is complex code? It seems to me that the answer to this question is very simple: it is code that is difficult to read and difficult to change. I, like many, prefer to think that we are writing code for people, not for machines. And I try to leave room for the thought process of composing the code before sitting down to write it. Does the composition I have made really complicate the reading of this code and pretend to be stigmatized - overengineering? I would like to see an opinion on this issue in your comments.
