DBA：VACUUMがパスしたとき-テーブルを手動でクリーニングします

VACUUMはPostgreSQLのテーブルから「クリーンアップ」できるのは、誰にも見えないものだけです。つまり、これらのレコードが変更される前に開始された単一のアクティブなクエリはありません。

そして、そのような不快なタイプがある場合（OLTPベースの長期OLAP負荷）長いクエリに囲まれてアクティブに変化するテーブルをクリーンアップして、レーキを踏まない方法は？



すくいを広げる


まず、解決したい問題とその発生方法を特定します。

通常、この状況は比較的小さなテーブルで発生しますが、多くの変更が発生します。通常、これらは、UPDATEが頻繁に実行されるさまざまなカウンター/集計/レーティング、または常にINSERT / DELETEであるレコードである、常に実行されているイベントのストリームを処理するためのバッファーキューです。

評価付きのオプションを再現してみましょう。

CREATE TABLE tbl(k text PRIMARY KEY, v integer);
CREATE INDEX ON tbl(v DESC); --      

INSERT INTO
  tbl
SELECT
  chr(ascii('a'::text) + i) k
, 0 v
FROM
  generate_series(0, 25) i;

並行して、別の接続では、長い長いクエリが開始され、いくつかの複雑な統計が収集されますが、テーブルには影響しません。

SELECT pg_sleep(10000);

ここで、カウンターの1つの値を何度も更新します。実験の純粋さのために、これは実際に発生するため、dblinkを使用して個別のトランザクションでこれを行います。

DO $$
DECLARE
  i integer;
  tsb timestamp;
  tse timestamp;
  d double precision;
BEGIN
  PERFORM dblink_connect('dbname=' || current_database() || ' port=' || current_setting('port'));
  FOR i IN 1..10000 LOOP
    tsb = clock_timestamp();
    PERFORM dblink($e$UPDATE tbl SET v = v + 1 WHERE k = 'a';$e$);
    tse = clock_timestamp();
    IF i % 1000 = 0 THEN
      d = (extract('epoch' from tse) - extract('epoch' from tsb)) * 1000;
      RAISE NOTICE 'i = %, exectime = %', lpad(i::text, 5), lpad(d::text, 5);
    END IF;
  END LOOP;
  PERFORM dblink_disconnect();
END;
$$ LANGUAGE plpgsql;

NOTICE:  i =  2000, exectime = 0.739
NOTICE:  i =  3000, exectime = 1.188
NOTICE:  i =  4000, exectime = 2.508
NOTICE:  i =  5000, exectime = 1.791
NOTICE:  i =  6000, exectime = 2.658

NOTICE:  i =  8000, exectime = 2.572

どうした？単一のレコードの最も単純なUPDATEであっても、ランタイムが0.524ミリ秒から3.808ミリ秒に7倍低下したのはなぜですか。そして、私たちの評価はますますゆっくりと構築されています。

MVCCのせいです


それはすべて、MVCCメカニズムに関するものであり、レコードの以前のバージョンをすべて表示するよう要求を強制します。それでは、「死んだ」バージョンからテーブルをきれいにしましょう：

VACUUM VERBOSE tbl;

INFO:  vacuuming "public.tbl"
INFO:  "tbl": found 0 removable, 10026 nonremovable row versions in 45 out of 45 pages
DETAIL:  10000 dead row versions cannot be removed yet, oldest xmin: 597439602

おお、掃除するものは何もない！並列クエリは私たちを悩ませています -結局のところ、ある日、彼はこれらのバージョンを参照したいと思うかもしれません（もしどうしたら？）、そしてそれらは彼に利用可能であるべきです。したがって、VACUUM FULLでさえ私たちを助けません。

テーブルの「固定」


しかし、テーブルにはクエリが不要であることは確かです。したがって、システムのパフォーマンスを適切なフレームワークに戻して、VACUUMが合格したため、少なくとも「手動で」、テーブルから不要なものをすべて破棄してみましょう。

わかりやすくするために、バッファテーブルの例を考えてみましょう。つまり、大きなINSERT / DELETEストリームがあり、テーブルが完全に空になる場合があります。しかし、そこが空でなければ、現在の内容を保存する必要があります。

＃0：状況を評価する


各操作の後でさえもテーブルを使って何かを試すことができるのは明らかですが、あまり意味がありません-メンテナンスのオーバーヘッドは、対象となるクエリのスループットより明らかに多くなります。

次の場合、「行動する時がきた」という基準を策定します。


  • VACUUM が長時間実行されているため、大きな負荷が予想されるため、最後の[auto] VACUUMから60秒待ちます。
  • テーブルの物理サイズがターゲットよりも大きい潜在的な空のテーブル
    の最小サイズ- ヒープあたり1 blk +各インデックスごとに1 blk-の2倍のページ数（8 KBブロック）として定義します。特定の量のデータが常に「正常に」バッファーに残ると予想される場合は、この式を厳しくするのが妥当です。

確認リクエスト
SELECT
  relpages
, ((
    SELECT
      count(*)
    FROM
      pg_index
    WHERE
      indrelid = cl.oid
  ) + 1) << 13 size_norm --    * current_setting('block_size')::bigint,     ?..
, pg_total_relation_size(oid) size
, coalesce(extract('epoch' from (now() - greatest(
    pg_stat_get_last_vacuum_time(oid)
  , pg_stat_get_last_autovacuum_time(oid)
  ))), 1 << 30) vaclag
FROM
  pg_class cl
WHERE
  oid = $1::regclass -- tbl
LIMIT 1;

relpages | size_norm | size    | vaclag
-------------------------------------------
       0 |     24576 | 1105920 | 3392.484835

＃1：とにかく真空


並列クエリが本当に私たちを妨げているのかどうか、つまり、最初から「古くなっている」レコードの正確な数を事前に知ることはできません。したがって、それでも何らかの方法でテーブルを処理することを決定した場合、いずれの場合でも、最初にVACUUMを実行する必要があります。これは、VACUUM FULLとは異なり、読み取り/書き込みデータを使用した並列プロセスに干渉しません。

同時に、彼は私たちが削除したいもののほとんどをすぐに片付けることができます。はい。このテーブルに対する以降のリクエストは、「ホットキャッシュ」で送信されます。これにより、その期間が短縮されます。したがって、サービストランザクションで他のユーザーをブロックする合計時間が短縮されます。

＃2：誰かが家にいますか？


チェックしてみましょう-テーブルに何かありますか：

TABLE tbl LIMIT 1;

単一のレコードが残っていない場合は、TRUNCATEを実行するだけで、処理を大幅に節約できます。

各テーブルの無条件DELETEコマンドと同じように動作しますが、実際にはテーブルをスキャンしないため、はるかに高速です。また、すぐにディスク領域が解放されるため、その後にVACUUM操作を実行する必要はありません。
テーブルシーケンスのカウンター（RESTART IDENTITY）を同時にリセットする必要があるかどうか-自分で決定します。

＃3：すべて-順番に！


私たちは非常に競争の激しい環境で作業しているため、テーブルにエントリがないことをここで確認している間、誰かがすでに何かをそこに書き込む可能性があります。この情報を失うべきではないので、何ですか？そうです、誰も確実に記録できないようにする必要があります。

これを行うには、トランザクションのSERIALIZABLE分離を有効にし（はい、ここでトランザクションを開始します）、テーブルを「しっかり」ロックする必要があります。

BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
LOCK TABLE tbl IN ACCESS EXCLUSIVE MODE;

このレベルのブロッキングは、そのブロックに対して実行する操作が原因です。

＃4：利益相反


私たちはここに来て、タブレットを「ロック」したいと思います。たとえば、その瞬間に誰かがアクティブだった場合は、それを読みますか？このロックの解放を見越して「ハング」しますが、読み取りを希望する他の人は私たちの中に残ります...

これが起こらないようにするために、「自分自身を犠牲にします」。それでも、特定の（許容可能な短い時間）内にロックを取得できなかった場合は、例外ベースからですが、少なくとも私たちは残りの部分にあまり干渉しません。

これを行うには、セッション変数lock_timeout（バージョン9.3以降の場合）または/およびstatement_timeoutを設定します。覚えておくべき主なことは、statement_timeoutの値は次のステートメントからのみ適用されるということです。つまり、このような接着では、機能しません。

SET statement_timeout = ...;LOCK TABLE ...;

変数の「古い」値の復元に対処しないために、設定の範囲を現在のトランザクションに制限するSET LOCALフォームを使用します。

statement_timeoutは後続のすべてのリクエストに適用されるため、テーブルに大量のデータがある場合にトランザクションが許容できない値に拡大しないようにしてください。

＃5：データをコピーする


テーブルが完全に空でないことが判明した場合、データは補助的な一時ラベルを介して再保存する必要があります。

CREATE TEMPORARY TABLE _tmp_swap ON COMMIT DROP AS TABLE tbl;

ON COMMIT DROP シグネチャは、トランザクションが終了すると一時テーブルが存在しなくなり、接続のコンテキストで手動で削除する必要がないことを意味します。

「ライブ」データは多くないと想定しているため、この操作は十分迅速に実行されます。

まあ、それだけです！必要に応じて、トランザクションの完了後にANALYZEを実行して、テーブルの統計を正規化することを忘れないでください。

最終的な脚本を集めます


私たちはそのような「疑似python」を使用します。

#      VACUUM  
if stat.size > 2 * stat.size_norm and stat.vaclag is None or stat.vaclag > 60:
  -> VACUUM %table;
