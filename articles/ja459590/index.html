Wavesブロックチェーン上の分散型オープンソースアフィリエイトプログラム

BettexチームによるWaves Labsの助成金の一部として実装された、Wavesブロックチェーン上の分散型アフィリエイトプログラム。

投稿は広告ではありません

dAppがアフィリエイトプログラムに導入したのは、機能の一部としてアフィリエイトを含むプロジェクトのテンプレートです。コードは、コピーのテンプレート、ライブラリ、または技術的な実装のアイデアのセットとして使用できます。

機能面では、これは、紹介者への登録、紹介に対する報酬の複数レベルの獲得、およびシステムへの登録の動機（キャッシュバック）を実装する通常のアフィリエイトシステムです。システムは「純粋な」dAppです。つまり、Webアプリケーションは、バックエンドやデータベースなどを使用せずにブロックチェーンと直接やり取りします。

他の多くのプロジェクトでも役立つテクニックが使用されています。

  • スマートアカウントをクレジットで即時に返済します（呼び出しの時点では、アカウントには呼び出しの支払いに必要なトークンはありませんが、呼び出しの結果としてそこに表示されます）。
  • PoW-captcha-スマートアカウント機能の高頻度の自動呼び出しに対する保護-captchaに類似していますが、コンピューティングリソースの使用の証明によるものです。
  • テンプレートによるデータキーのリクエスト。

アプリケーションはで構成されています：

  • ride4dapps言語のスマートアカウントコード（設計上、アフィリエイト機能を実装する必要があるメインスマートアカウントにマージされます）;
  • WAVES NODE REST APIを介して抽象化レイヤーを実装するjsラッパー。
  • ライブラリとRIDEコードの使用例であるvuejsフレームワークのコード。

これらの機能のすべてについて説明します。

クレジットでスマートアカウントを即時に返済するには、

InvokeScriptの呼び出しで、トランザクションを開始するアカウントから手数料を支払う必要があります。アカウントにいくつかのWAVESトークンを持っているブロックチェーンマニア向けのプロジェクトを実行している場合、これは問題になりませんが、製品が広範な使用を目的としている場合、これは深刻な問題になります。結局のところ、ユーザーはWAVESトークン（またはトランザクションの支払いに使用できる別の適切な資産）を購入する必要があるため、プロジェクトに参加するためのすでにかなりのしきい値が高くなります。私たちはユーザーに資産を配布できます。これにより、トランザクションの支払いが可能になり、自動システムが作成されてシステムから液体資産を汲み上げるときに、ユーザーが誤用されるリスクに直面します。

「受信者を犠牲にして」（スクリプトがインストールされているスマートアカウント）にInvokeScript呼び出しを行うことができれば、非常に便利です。そのような可能性は、明白な方法ではありませんが存在します。

InvokeScript内でScriptTransferを呼び出し元のアドレスに作成し、これが料金に費やされたトークンを補う場合、呼び出し時に呼び出し元のアカウントにアセットがなかったとしても、そのような呼び出しは成功します。これは、トランザクションが呼び出される前ではなく、呼び出された後に十分な数のトークンがチェックされるため、クレジットでのトランザクションを即時に返済できるため、可能です。

ScriptTransfer（i.caller、i.fee、unit）

以下のコードは、スマートアカウントの費用で費やされた料金を払い戻します。この機能の誤用を防ぐために、発信者が適切な資産に適切な制限内で料金を支払っていることを確認する必要があります。

func checkFee(i:Invocation) = {
if i.fee > maxFee then throw(“unreasonable large fee”) else
if i.feeAssetId != unit then throw(“fee must be in WAVES”) else true
}

また、悪意のある無意味な資金の浪費に対する保護には、自動呼び出し（PoWキャプチャ）に対する保護が必要です。PoWキャプチャ 作業証明

キャプチャの概念自体

は新しいものではなく、WAVESに基づくプロジェクトを含め、さまざまなプロジェクトですでに実装されていますこのアイデアは、プロジェクトのリソースを浪費するアクションでは、呼び出し元が自分のリソースを費やす必要があるため、リソースの枯渇に対する攻撃が非常に高くつくということです。トランザクションの送信者がPoW問題を解決したという非常に簡単で低コストの検証のために、トランザクションIDのチェックがあります：

if take（toBase58String（i.transactionId）、3）！=“ 123” then throw（“ proof of work failed”）else

トランザクションを実行するために、呼び出し元は、base58コード（id）が123から始まるようにそのようなパラメーターを選択する必要があります。これは、平均で数十秒のプロセッサー時間に相当し、通常はタスクにとって妥当です。より単純な、またはより複雑なPoWが必要な場合、タスクは明白な方法で簡単に調整できます。

テンプレートによるデータキーのクエリ

ブロックチェーンをデータベースとして使用するには、テンプレートによってデータベースをkey-valとしてクエリするためのAPIツールが不可欠です。そのようなツールキットは、2019年7月の初めに？ REST APIリクエスト/アドレス/データのパラメーターと一致しますか？一致= regexp。ここで、一度に1つのキーではなくすべてのキーではなく、一部のグループのみをWebアプリケーションから取得する必要がある場合は、キー名で選択できます。たとえば、このプロジェクトでは、出金トランザクションは次のようにエンコードされます

withdraw_${userAddress}_${txid}

これにより、テンプレートに従って任意の住所の資金を引き出すためのトランザクションのリストを取得できます。

?matches=withdraw_${userAddress}_.*

レジスタ、資金、および引き出し機能で構成されます。

登録機能はユーザシステムに登録します。 2つのパラメーターがあります：リファラー（リファラーアドレス）と、トランザクションコード（PoWキャプチャタスク）を選択するために必要な関数コードで使用されないソルトパラメーター。

関数（このプロジェクトの残りの関数と同様）は借用手法を使用します。関数の結果として、この関数を呼び出すための料金が融資されます。この決定のおかげで、ウォレットを作成したばかりのユーザーはすぐにシステムを操作でき、取引手数料を支払うことができる資産の購入や受け取りの問題を気にする必要がありません。

登録関数の結果は2つのエントリです。

${owner)_referer = referer
${referer}_referral_${owner} = owner

これにより、直接検索と逆検索（このユーザーの参照元とこのユーザーのすべての参照）が可能になります。

ファンド機能は、この機能を開発するためのテンプレートである可能性が高いです。提示された形式で、彼女はトランザクションによって転送されたすべての資金を受け取り、リファラー1、2、3のアカウント、キャッシュバックアカウント、および変更アカウントに分配します（以前のアカウントへの分配時に残っているものはすべてここに含まれます）。

キャッシュバックは、紹介システムに参加するようにエンドユーザーを刺激する手段です。ユーザーは、紹介料と同じように、システムが支払ったコミッションの一部を「キャッシュバック」の形で引き出すことができます。

紹介システムを使用するときは、システムが機能するスマートアカウントのメインロジックに埋め込まれたファンド機能を変更する必要があります。たとえば、ベットに紹介料が支払われた場合、ベットが行われる（または手数料が支払われる別のターゲットアクションが実行される）ロジックにファンド機能を組み込む必要があります。この関数は、3レベルの紹介報酬をエンコードします。レベルを増減したい場合は、これもコードで修正されます。報酬のパーセンテージはlevel1-level3定数によって設定されます。コードでは、金額* level / 1000と見なされます。つまり、値1は0.1％に対応します（これはコードで変更することもできます）。
