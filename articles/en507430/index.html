DIY production calendar in Postgresql







Hello, my name is Victor and I am a developer at Gems Development. I want to tell how we implemented the creation and completion of the production calendar in Postgresql.



The workflow depends on official national and international holidays. In corporate applications, the task of calculating the date often takes into account working days and weekends. For example, services such as ‚ÄúIssuing a construction permit‚Äù, ‚ÄúIssuing a commissioning permit‚Äù should be provided on time.



After analyzing the task, we came to the conclusion that it is enough to store weekends and holidays in the calendar, because it is they that are difficult for calculations and can change in accordance with the production calendar in each year.



Rules for working with the calendar:



  • The definition of a working day depends on whether the date is on the list of weekends and holidays or not.
  • If the working day in connection with the transfer becomes non-working, then it is also recorded in the table.
  • If the day off is a working day, then, accordingly, the record from this table is deleted.


For demonstration, we will create a separate scheme and set a variable in the session so that each time we do not specify the scheme in the requests: 



create schema calendar;
set search_path=calendar;


To support such calculations, a table was created in the database: 



CREATE TABLE d_holidays
(
  holiday_date date,
  comment character varying(1000),
  CONSTRAINT d_holidays_pk PRIMARY KEY (holiday_date)
);


As well as an index that allows you to speed up the search for the desired date in the table: 

CREATE UNIQUE INDEX d_holidays_pk_idx
  ON d_holidays
  USING btree
  (holiday_date);


To create data, we use a standard function from Postgresql that returns a set. We will check the operation of the index by pre-filling the table with a sufficiently large set of generated data: all days from 01/01/2017 to 12/31/2025. 



insert into d_holidays(holiday_date)
select generate_series(to_date('01.01.2017','dd.mm.yyyy'),
            to_date('31.12.2025','dd.mm.yyyy'),
            '1 day');


We got 3287 records, which corresponds to the number of days from the specified dates. It is important to ensure that the parameter ‚Äúautovacuum = on‚Äù is enabled in the Postgresql configuration.



Let's see the query plan: 



explain(analyze)
select comment from d_holidays where holiday_date='09.09.2020';

The calculation of the holidays will be carried out within the year. After checking all the dates from January 1 of the specified year to December 31, select the days off and write them in the table. To implement the algorithm, we use the procedural language plpgsql and the construction of an anonymous block.



We will prepare a design that forms the date for January 1 and December 31. 



do
$$
declare
    calc_year integer:=2020; -- 
    begin_date date; --    
    end_date date; --    
    dow integer; --  . -1,...,-6,-0
begin
    --   
    if calc_year between 1988 and 2099 then
        begin_date :=to_date('01.01.'||calc_year, 'dd.mm.yyyy');
        end_date :=to_date('31.12.'||calc_year, 'dd.mm.yyyy');
    end if;
end$$;


To calculate the day of the week, we use the extract function, which determines the number of the day of the week (taking into account that Sunday is 0). To determine the name of the day of the week, we use the simple case construct.



We get the following script: 



do
$$
declare
    calc_year integer:=2020; -- 
    begin_date date; --    
    end_date date; --    
    dow_value integer; --  . -1,...,-6,-0
begin
    --   
    if calc_year between 1988 and 2099 then
        begin_date :=to_date('01.01.'||calc_year, 'dd.mm.yyyy');
        end_date :=to_date('31.12.'||calc_year, 'dd.mm.yyyy');
        
        while begin_date<=end_date
        loop
            raise info '%',to_char(begin_date,'dd.mm.yyyy');
            begin_date:=begin_date+interval '1 day';
            dow_value:=extract(dow from begin_date);
            if dow_value in(0,6) then
                insert into d_holidays(holiday_date,comment)
                        values(begin_date,
                            case when dow_value=6 then '' else '' end);
            end if;
        end loop;
    end if;
end$$;


Let's analyze the algorithm for adding holidays: 



insert into d_holidays(holiday_date,comment) 
    values(to_date('01.01.2020','dd.mm.yyyy'),' ') 
        on conflict(holiday_date) do update set comment=EXCLUDED.comment;

There may be a situation when the day off becomes a working day. In this case, the algorithm is as follows (for example, 2018 was chosen, since there are no working days off in 2020): 



delete from d_holidays where holiday_date=to_date('29.12.2018','dd.mm.yyyy'); 
delete from d_holidays where holiday_date=to_date('09.06.2018','dd.mm.yyyy');


Pay attention to the design: 



on conflict(holiday_date) do update set comment=EXCLUDED.comment;


This innovation appeared in Postgresql version 9.5 and allows you to effectively handle conflicts when inserting records. In our example, if the holiday is already a holiday, then in order to avoid conflict over the primary key when adding a record with an existing date, we update the date description without creating a duplicate.
