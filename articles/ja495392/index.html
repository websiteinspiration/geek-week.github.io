フロントエンドでバグを検索する方法：4つのメインステージ


この記事では、フロントエンド（ブラウザーまたはWebビュー）で発生するエラーを特定して処理する問題について検討します。

フロントエンドでは、JSコードがブラウザーで実行されます。JavaScriptはコンパイルされた言語ではないため、プログラムを直接使用すると、常にパフォーマンスエラーが発生する可能性があります。実行エラーは、エラーの場所の後にあるコードをブロックし、プログラムのユーザーは、再ロードまたは閉じることしかできない機能しないアプリケーション画面を残すリスクを負います。しかし、そのような状況を回避するために、エラーとその安全な付随を検出する方法があります。

1. JavaScriptツール


ブロックを試す/キャッチ


失敗する可能性のある関数は、try / catchブロックでラップされている場合があります。まず、プログラムはtryブロックのコードを実行しようとします。なんらかの理由でコードの実行が失敗した場合、プログラムはcatchブロックに入り、3つのパラメーターが使用可能になります。

  • name-エラーの標準化された名前。
  • message-エラーの詳細に関するメッセージ。
  • stack-エラーが発生した現在の呼び出しスタック。

つまり：

try {

   callFunc();

} catch (e) {

   console.log(e.name) // ReferenceError

   console.log(e.message) // callFunc is not defined

   console.log(e.stack) // ReferenceError: callFunc is not defined at window.onload

}

開発者にとって主なことは、プログラムがcatchブロックの後も実行を継続できることです。したがって、ユーザーの操作は中断されません。

try / catchブロックを使用すると、たとえばデータをチェックするときに、独自のエラーを発生させることもできます。

const user = {

  name : «Mike»

};

try {

   if (!user.age) throw new SyntaxError(«User age is absent!»);

} catch (e) {

   console.log(e.name) // SyntaxError

   console.log(e.message) // User age is absent!

   console.log(e.stack) // SyntaxError: User age is absent! at window.onload

}

最後に、このステートメントをもう1つのブロックで拡張できます。最後に、常に実行されます。試行でエラーが発生しなかった場合と、コントロールがcatchブロックに渡された場合の両方です。

try {

   callFunc();

} catch (e) {

   console.log(e)

} finally {

   ...

}

特定のエラーを処理せずにコードを使い続けることができるように、（キャッチなしで）try / finallyステートメントを使用する場合があります。

Window.onerrorイベント


プログラムが壊れてユーザーセッションが正常に終了しなかった場合でも、ページ上のスクリプトが壊れていることを知ることはしばしば役に立ちます。通常、この情報はエラーログシステムで使用されます。

グローバルウィンドウオブジェクトにはonerrorイベントがあります（慎重に使用してください：実装はブラウザーによって異なる場合があります！）：

window.onerror = function(message, url, line, col, error) {

   console.log(`${message}\n  ${line}:${col}  ${url}`);

};

このコードをスクリプトの最初に配置するか、最初に別のスクリプトにロードすると、開発者の下のエラーについて、開発者はその詳細情報を入手できます。

ただし、完全な情報は、同じドメインからダウンロードされたスクリプトでのみ利用できます。壊れたスクリプトが別のドメインからロードされた場合、window.onerrorは機能しますが、エラーの詳細はありません。

フレームワークコンポーネント


一部のJSフレームワーク（React、Vue）は、独自のエラー処理ソリューションを提供しています。たとえば、Reactはエラーが発生したブロックの場所に特別なレイアウトを描画できます。

class ErrorBoundary extends React.Component {

   constructor(props) {

       super(props);

       this.state = { hasError: false };

   }

   static getDerivedStateFromError(error) {

       //    ,      UI.

       return { hasError: true };

   }

   componentDidCatch(error, errorInfo) {

       //           

       logErrorToMyService(error, errorInfo);

   }

   render() {

       if (this.state.hasError) {

           //    UI  

           return <h1>-   .</h1>;

       }

       return this.props.children;

   }

}

<ErrorBoundary>

   <MyWidget />

</ErrorBoundary>

実際、Reactコンポーネントは、エラーを処理する特別なコンポーネントにラップされています。これは、try / catch構文を使用して関数をラップするのと似ています。

2.プロジェクトアセンブリツール


最近のJSスクリプトは通常、変換されます。つまり、最新のES標準を使用して開発が行われます。次に、プロジェクトビルダー（Webpackなど）を使用する開発者のコ​​ードが、選択した数のブラウザーでの動作が保証されるコードに変換されます。

ビルド段階で、コードの構文が正しいかどうかがチェックされます。開いたブラケットまたはクラスフィールドの誤った指定は、アセンブリ中にすぐにエラーを引き起こし、バンドルは単にアセンブリされません。開発者が作業を続けるには、このようなエラーをすぐに修正する必要があります。

また、コレクターは、プログラムの実行時に一部のコードが使用されないことを示唆する場合があります。おそらくこれにより、開発者はコードのより深い調査について考えるようになり、間接的に新しいエラーの識別に影響を与える可能性があります。

3.テスト


コードのエラーを防ぐ別の方法は、コードをテストすることです。単体テストを効果的に使用するためのツールがフロントエンドにあります。通常、Jest、Karma、Mocha、Jasmineなどのフレームワークが使用されます。多くの場合、テストフレームワークと一緒に、Enzyme、React Testing Library、Sinonなどの拡張機能を使用します。これにより、モカ、スパイ機能、その他のツールを使用してテストを強化できます。

エラーを検索する場合、テストは主に、実行エラーにつながる可能性のあるさまざまなデータをロードするのに役立ちます。したがって、次のコードは構文検証に合格し、期待どおりに機能します。

const func = (data) => {

   return JSON.parse(data)

}

func('{«a»:1}')

ただし、間違った値を指定すると壊れます。

func() // Uncaught SyntaxError: Unexpected token u in JSON at position 0.


このコードは、アセンブリ中に検証も渡します。

const obj = {

   outer : {

       last : 9

   }

}

if (obj.outer.inner.last) {

   console.log(«SUCCESS»)

}

ただし、実行中に壊れることもあります。テスト後、開発者はおそらく追加のチェックを行います。

if (obj.outer?.inner?.last) {

   console.log(«SUCCESS»)

}

多くの場合、このようなエラーは、サーバーからデータを受信すると（AJAXリクエストなどで）後続の分析で発生します。コードをテストすることで、クライアントのブラウザーでの実行中にコードが壊れる可能性があるケースを事前に特定して排除できます。

4.ロギング


プロジェクトの開発およびアセンブリ中のエラーを防止するために、可能なすべての対策を講じたとします。ただし、エラーは依然として生産的なコードに侵入する可能性があります。私たちは何とかして彼らの存在を知り、即座に是正措置をとる必要があります。ユーザーにブラウザコンソールを開いてスクリーンショットを撮るよう依頼することは、最善の方法ではありません。したがって、エラーログをプロジェクトに接続することをお勧めします。

意味は単純です。window.onerrorイベントごと、またはコード実行のcatchブロックへの遷移ごとに、特別に割り当てられたサーバーアドレスに対して単純なAJAX要求が行われ、その中にエラー情報が配置されます。次に、新しいエラーの存在をテクニカルサポートと開発者にすばやく通知し、それらを効果的に使用できるツールが必要になります。これらのフロントエンドツールの中で最も人気のあるのは、Sentryです。

Sentryロギングシステムを使用すると、リアルタイムでエラーを収集、グループ化、表示できます。JavaScriptを含むさまざまな言語のアセンブリがあります。プロジェクトはビジネス向けの高度な機能を備えた有料アクセスを提供しますが、無料のテストアカウントでその主な機能を試すことができます。

Sentryは、HTMLファイルに直接接続することも、一般的なフレームワーク（React、Vue、Angular、Emberなど）で実行されるコンポーネントに接続することもできます。

セクションのブラウザでログ機能を直接接続するには、スクリプトをロードします。

<script

   src=«https://browser.sentry-cdn.com/5.13.0/bundle.min.js»

   integrity=«sha384-ePH2Cp6F+/PJbfhDWeQuXujAbpil3zowccx6grtsxOals4qYqJzCjeIa7W2UqunJ»

   crossorigin="anonymous"></script>


次に、JSコードで、以下を初期化します。

Sentry.init({

   dsn: 'https://<your account key here>@sentry.io/<your project id here>'

});

すべて。この行の下のコードでエラーが発生した場合、Sentryはそれをログに記録します。他のドメインからのスクリプトの障害が原因でエラーが発生した場合でも、ログは記録されます。



Sentryには、一連のエラーメッセージを分析して通知を設定する十分な機会があります。製品のリリースごとにエラーログをグループ化することもできます。

Sentry.init({

   dsn: 'https://<your account key here>@sentry.io/<your project id here>',

   release: '2020.03.06.1'

});

Sentryを使用すると、統計を使用して、エラーのコンテキスト（顧客情報、指紋、エラーレベル（致命的、エラー、警告、情報、デバッグ）、タグなど）を転送できます。

ユーザーイベントを統計に記録することが可能です。たとえば、トラッキングを設定して、ブラウザウィンドウのサイズを変更したり、AJAXリクエストを作成したりできます。Sentryには、エラー発生時にユーザーに表示できるフィードバックウィンドウを備えた独自のウィジェットもあります。これにより、エラーの状況を調査するための追加情報が提供されます。

フレームワークとともにSentryをデプロイするには、パッケージをインストールして接続するだけです。

# Using yarn

yarn add @sentry/browser

# Using npm

npm install @sentry/browser


プロジェクトのメインスクリプトで初期化を行います（ReactおよびAngular用）。

import * as Sentry from «@sentry/browser»;

Sentry.init({ dsn: 'https://<your account key here>@sentry.io/<your project id here>' });


VueとEmberの場合、必要な別の構成行を渡します。

# Vue

Sentry.init({

   dsn: '<your account key here>@sentry.io/<your project id here>',

   integrations: [new Integrations.Vue({Vue, attachProps: true})],

});

# Ember

Sentry.init({

   dsn: '<your account key here>@sentry.io/<your project id here>',

   integrations: [new Integrations.Ember()]

});

統合パッケージは個別にインストールされます。

# Using yarn

yarn add @sentry/integrations

# Using npm

npm install @sentry/integrations

1つのプロジェクトで複数のスクリプトを接続するときの情報の競合と重複を防ぐために、Sentryではログの初期化ごとに個別のクライアントを作成できます。

import { BrowserClient } from «@sentry/browser»;

const client = new BrowserClient({

   dsn: '<your account key here>@sentry.io/<your project id here>',

});

client.captureException(new Error('example'));

プロジェクトのウェブサイトには、使用例を含む詳細なドキュメントがあります：https : //docs.sentry.io。
