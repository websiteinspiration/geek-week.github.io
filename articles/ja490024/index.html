PhpStormプラグインでルーチンを取り除く

こんにちは、ハブル！私はAntispamチームで働いており、ほとんどのBadooバックエンド開発者と同様に、ほとんどの時間はPHPコードでの作業に費やしています。

この作業に関連する多くの特定の問題とタスクがあり、解決または簡略化できます。 IDEでできることを手動で行うことにうんざりしているとき、私は社内で最も一般的なIDE-PhpStorm-を完成させ、日常的なタスクを解決するプラグインを作成して、自分と同僚の生活を簡素化することにしました。

そこで、PhpStorm用のBadooプラグインが登場しました。これは、今日私たちが積極的に使用しています。長年にわたり、その機能は大幅に拡張されており、開発中です。この記事では、事例を使用してIDEを自分のタスクやツールに適合させる方法を説明し、見た目ほど難しくないことを証明します。



JetBrainsが商用製品を開発していることは誰もが知っていますが、当時IDEAがオープンソースプロジェクトであることは私にとって発見でした。実際、これはIDEを構築するためのプラットフォームであり、すべてのJetBrainsは、特定のプログラミング言語の問題の解決に特化したプラグインのセットです。ここから、PhpStormが実装するすべての機能を自分のニーズに合わせて調整できることがわかりました。ちなみに、Eclipseは同様のアプローチを使用しています。

したがって、PhpStormで実行できるすべてのトリックを使用して、より専門的な環境を作成し、生活をさらに簡単にすることができます。 Badooでは、独自のツールの多くを使用し、開発をできるだけ快適にするように努めているため、独自のプラグインを作成するというアイデアは、少なくとも実験に値するものであるように思われました。

プラグインの作成方法には重点を置きません。これについてはHabréに関する記事がすでにありました。サイトにはQuickStartがあり、一般に、テンプレートをここでダウンロードするだけで十分です（この場合はSimpleで十分です）。トピックが完全に開示されていないと思われる場合は、コメントで教えてください-私はそれについてより詳細な記事を書きます。

それでは、PhpStormをどのような場合にどのように変更したかについて例を見てみましょう。

カスタムコード検証


PHPには、printfのような関数があり、テンプレートが最初の引数として渡された後、このテンプレートに代入される引数が渡されます。ただし、標準のPhpStormインスペクションは、PHP組み込み関数でのみ有効性を検証します。そのため、ロギング機能に同様のチェックを追加しました。

たとえば、infof、errorfなどのメソッドを持つ独自のロガーがあります。引数の数がテンプレートの置換の数に対応することを検証するプラグインを作成しています。

IntelliJに関しては、このような機能を実装するためにアノテーターまたは検査を使用できます。詳細には触れませんが、アノテーターは検査のより単純なバージョンであると想定します。そして、これを検査の例だけで実装します。

継承クラスLocalInspectionToolを作成します。

class LoggerFormatInspection : LocalInspectionTool() {
   override fun buildVisitor(holder: ProblemsHolder, isOnTheFly: Boolean, session: LocalInspectionToolSession): PsiElementVisitor {
       if (session.file !is PhpFile) {
           //    -PHP-
           return super.buildVisitor(holder, isOnTheFly, session)
       }
       return LoggerFormatVisitor(holder, isOnTheFly)
   }
}


主な作業はLoggerFormatVisitorクラスで行われます。実際、ソースコードのすべての要素（PsiElement、PSI-プログラム構造インターフェース）を実行し、それらのハンドラーを呼び出します。buildVisitorメソッドは、プロジェクト内の各ファイルに対して実行されます。

class LoggerFormatVisitor(private val holder: ProblemsHolder, onTheFly: Boolean) : PsiElementVisitor() {
   override fun visitElement(element: PsiElement) {
       if (!isLoggerFormatFunction(element)) {
           return //  ,      
       }

       checkMethodReference(element as MethodReference) //
   }
}

最初のメソッド（isLoggerFormatFunction（））は、現在の要素が関数呼び出しであることを確認し、メソッドシグネチャを確認します。ロガー変数は\ Logger \ Loggerクラスの子孫である必要があり、メソッド名は関数のprinfのようなパラメーターをサポートするものの1つである必要があります（debugf、infof、noticef、warningf、errorf、infofの6つがあります）。

2番目の方法は、直接作業を行います。パラメーターをチェックし、エラーを強調表示します。

分解してみましょう。

private fun checkMethodReference(element : MethodReference) {
   //  printf — :
   //  ()  — -;   ,        ,       (   )
   //      , -      
   val printfArgument = 0
   val firstPossibleArgument = printfArgument + 1
   if (element.parameters.size <= printfArgument) {
       return
   }

   val formatLine = element.parameters[printfArgument] as? StringLiteralExpression //     , 
           ?: return

   val expectingParameters = getExpectingParameters(formatLine)
   val arguments = element.parameters.slice(IntRange(firstPossibleArgument, element.parameters.size - 1))

最初の引数-フォーマット文字列を見つけて解析しました（解析コードは省略します。最終的なソースにあります）。expectingParametersに、後続の引数で予期されるパラメーターの説明が含まれるようになりました。そして引数には、チェックする引数があります。

確認コードを書くだけです！

最初に、フォーマット要素を指定するのを忘れているかもしれませんが、適切な引数を渡します。

if (expectingParameters.isEmpty() && arguments.isNotEmpty()) {
   holder.registerProblem(arguments.first(), "No format item found in first parameter but call has more than one argument", ProblemHighlightType.WARNING)
   ++ problems
}

クリティカル警告を伴う最初の説明されていない引数は、エディターで強調表示されます。
期待される引数の数は、渡された数（両方向）と一致しない場合があります。

if (expectingParameters.isNotEmpty()) {
   var expectingIndex = 0
   for (i in arguments.indices) {
       if (expectingParameters.size <= expectingIndex) {
           //   ,   -,   
           holder.registerProblem(
                   arguments[i],
                   "Format line expecting only ${expectingParameters.size} parameters",
                   ProblemHighlightType.WARNING
           )
           ++ problems
           continue
       }
       ++ expectingIndex
   }

   if (arguments.size < expectingParameters.size) {
       for (i in arguments.size until expectingParameters.size) {
           val item = expectingParameters[i]
           //    ,    warning
           holder.registerProblem(
                   formatLine,
                   "Unused format item",
                   ProblemHighlightType.WARNING,
                   TextRange(item.rangeStart + 1, item.rangeEnd + 2)
           )
           ++ problems
       }
   }
}

holder.registerProblem（）の最初の引数はPSI要素で、これは誤りと見なされます。パラメータは明白に見えるので、ここでは詳しく説明しません。

注意深い読者は、2番目のケースでは、数値による奇妙な操作を伴うTextRangeがまだあることに気付くでしょう。アイテム構造には、プレースホルダーが見つかったフォーマット文字列のテキスト内の場所が含まれています。ただし、formatLine要素には、文字列の内容に加えて、それを囲む引用符が含まれています。プレースホルダーの場所を見つけるために、プレースホルダーを追加してオフセットを調整します。すでに理解したように、TextRange引数は、フォーマット文字列全体ではなく、プレースホルダーが配置されているその部分のみを強調表示するために必要です。

そして最後に、エラーがまだ検出されている場合は、関数名を強調表示して、エラーをより明確に表示します。

if (problems > 0) {
   val elementStart = element.textOffset
   val nameNodeTextRange = element.nameNode!!.textRange

   val nameTextRange = TextRange(nameNodeTextRange.startOffset - elementStart, nameNodeTextRange.endOffset - elementStart)
   holder.registerProblem(element, nameTextRange, "Invalid format function usage")
}

また、オフセットによる操作。それらの本質は、MethodReference要素内のオフセットを指定する必要があることです（メソッドの呼び出し）。次に、オブジェクト、矢印演算子、呼び出されたメソッドの名前、および引数で構成されます。より正確にするために、関数の名前のみをマークします。エラーの重大度はここには示されていません-検査設定から取得されます（PhpStorm->エディター->検査設定）。
以上で機能の準備は完了です。

データベースを操作する


歴史的な理由から、データベースを操作するための独自のラッパーがあり、SQLクエリを含むテンプレートと、名前付き置換を含む配列を入力として受け取ります。これはインスペクションを使用するもう1つの例です。パラメーター付きの配列に必要なすべての置換が含まれ、不要な置換がないことを検証します。



ご覧のとおり、リクエストには標準のSQL構文に加えて、開発者が置換を指定する必要のあるプレースホルダーが含まれています。コードの記述中に、よく知られたプレースホルダー（これらは配列のキー）をオートコンプリートに追加し、それらがすべて示されているかどうかを確認しました。

したがって、私たちのフレームワークに固有のチェックのかなりの部分は、コードを書く段階で発生します。



ところで、リクエストがクラス定数で指定されていることに気づきましたか？これにもかかわらず、インスペクションは機能します：sqlパラメーターのすべての要素を調べ、引数の値を計算し、必要に応じて連結を実行します。したがって、リクエストがどこに書き込まれるかは問題ではありません-すべてのパラメーターが正しく入力されます。

さらに、100を超えるMySQLサーバーがあります。すべてのマシンのデータベースとテーブルのデータを収集し、リクエストで正しい接続が使用されているかどうかを確認できます（この機能は現在開発中です）。

ボイラープレートコードの生成


多くのフレームワークには、ある種のテンプレートに従って記述された機能があり、プロジェクトの多くの場所で、ほぼ同じコードを記述する必要があります。例はYiiのモデルです。

これは私たちのフレームワークにも存在します。しかし、私たちは皆、同じことを書くのはつまらないことを知っています。したがって、PhpStormにこれを行うことを教えました。

たとえば、モジュールファサードで関数を定義すると、PhpStormは対応するクラスを作成するためのクイックフィックスを提供します。



関数の本体を埋め、必要な要求ハンドラクラスを作成し、システムに登録するために必要なアクションを実行します。

public function onAcceptRequest(\Framework\Request $request, \User $user): void
{
   $command = new Commands\OnAcceptRequest($request, $user);
   return $command->run();
}

SoftMocksのサポート


フレームワークのもう1つの特徴は、テストでのSoftMocksのアクティブな使用です。例えば：

\Badoo\SoftMocks::redefineMethod(\User\ProfilePhoto::class, ‘getAll’, $arguments, $code)

このコードフラグメントでは、\ User \ ProfilePhoto :: getAll（）メソッドの呼び出しをインターセプトして、テストからデータベースに移動しないようにしています。これは言語構文の観点からのメソッド呼び出しではありませんが、フレームワークのこの機能を知っているため、この行を対応するメソッドへの呼び出しとして認識するように環境を教えました（ナビゲーション、Find Usagesの表示、2番目のパラメーターのオートコンプリートを含む）。

テストデバッグ（リモートおよびSoftMocksを使用！）


テストを書くことに加えて、それらをデバッグすることはしばしば開発者の責任です。時にはそれはリモートサーバーで行われます。SoftMocksがソーステキストを書き直し、PHPインタープリターがIDEに表示されるコードを実行しない（同等ではあります）ため、タスクは複雑です。また、実行可能ファイルへのパスは動的に決定されるため、PhpStormの標準のパスマッピングは無力であり、リモートサーバー上のパスを開いているプロジェクト内のパスと関連付けることができません。

標準ツールを使用すると、このタスクは次のように実行できます。

  1. Xdebugプロキシをインストールします。
  2. テストフレームワークPhpStormを設定して、リモートマシンで実行します（詳細については、こちらをご覧ください）。
  3. プロジェクトのパスマッピングを構成します。
  4. すべてがエラーなしで行われることを祈ります。

インタラクティブなデバッグでスコアを付けたり、printを使用してデバッグしたりすることもできます（非常に効果的な方法）。

この手順の関連性を考慮して、パスの動的マッピングのサポート、SSH経由の接続、テストの実行、および実際のデバッグをサポートするために、IDEにすべてを独自に行うように教えました。

統計を厳しくしています


コードのパフォーマンスを注意深く監視しています。そのため、Liveprofはサーバーで積極的に使用されています（これについては以前に説明しました）。コードパフォーマンスの変化を追跡し、問題が発生した瞬間を追跡できます。

また、収集した統計をIDEにインポートして、すばやくアクセスできるようにします。この機能は、個別のプラグインとして利用できます。IDEで直接、前日のメソッド呼び出しの統計が、それを使用したアプリケーションごとの内訳とともに表示されます。
