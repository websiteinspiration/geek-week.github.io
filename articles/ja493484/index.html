C＃はすでに「機能」しています。

こんにちは、ハブル！Igal Tabachnikによる元の記事「Your C＃はすでに機能していますが、許可した場合のみ」の翻訳を提示します。

数日前、C＃8.0の新機能のいくつかを使用して、FizzBu​​zzを実装するC＃コードスニペットをツイートしました。 「バイラルになった」ツイートは、その簡潔さと機能性に感心した人もいましたが、他の人から、なぜ私がF＃で記述しなかったのかと聞かれました。

前回C＃で書いてから4年以上が経過し、通常関数型プログラミングを使用しているという事実は、今日のコードの記述方法に明らかに影響を与えました。私が書いたスニペットはとてもきちんとしていて自然なようですが、一部の人々はそれがC＃コードのように見えないという懸念を表明しています。
「機能的すぎるようです。」彼らは私に手紙を書いた。
質問者によって、「関数型プログラミング」の意味は人によって異なります。しかし、セマンティクスについて説明する代わりに、このFizzBu​​zzの実装が機能しているように見える理由を説明したいと思います。

まず、このコードの機能を見てみましょう。

public static void Main(string[] args)
{
      string FizzBuzz(int x) => 
            (x % 3 == 0, x % 5 == 0) switch
            {  
                  (true, true) => "FizzBuzz", 
                  (true, _) => "Fizz", 
                  (_, true) => "Buzz", 
                  _ => x.ToString()
            };
    
      Enumerable.Range(1, 100 ) 
            .Select(FizzBuzz).ToList() 
            .ForEach(Console.WriteLine); 
}

ここでは、ラムダ式で表されるローカルメソッドを作成します。その結果はタプルを使用して計算されます。

ここでの目新しさは、2つの式（x％3 = = 0およびx％5 = = 0）を一緒に計算した結果を処理するためのタプル（ペア）の使用です。これにより、パターンマッチングを使用して最終結果を決定できます。どのオプションも一致しない場合、デフォルトでは、数値の文字列表現が返されます。

ただし、このコードで使用されている多くの「機能的な」「機能」（LINQスタイルのforeachループを含む）のいずれも、このコード自体を機能させるものではありません。これを機能させるのは、結果をコンソールに出力することを除いて、このプログラムで使用されるすべてのメソッドが式であることです。

簡単に言えば、表現は常に答えがある質問です。プログラミングに関して、式は定数、変数、演算子、および値を計算（「返す」）するためにランタイムによって計算される関数の組み合わせです。ステートメントとの違いを説明するために、C＃プログラマー向けのFizzBu​​zzのより馴染みのあるバージョンを書きましょう。

public static void Main(string[] args) 
{
      foreach( int x in Enumerable.Range(1, 100)) 
      {
             FizzBuzz(x);
      }
} 

public static void FizzBuzz( int x) 
{
      if (x % 3 == 0  && x % 5 == 0) 
             Console.WriteLine("FizzBuzz"); 
      else if (x % 3 == 0 ) 
             Console.WriteLine("Fizz"); 
      else if (x % 5 == 0 ) 
             Console.WriteLine("Buzz"); 
      else
             Console.WriteLine(x);
}

当然、これは「クリーン」なコードとはほど遠く、改善することはできますが、これが通常のC＃コードであることに同意せざるを得ません。 FizzBu​​zzメソッドを詳しく調べると、その単純さを考慮しても、いくつかの設計上の問題が明らかに示されています。

まず、このプログラムは、SOLIDの最初の原則、つまり単一責任の原則に違反しています。数値に基づいて出力値を計算し、この値をコンソールに出力するロジックを組み合わせます。その結果、依存関係の逆転の原則（SOLIDの最後）に違反し、結果の出力とコンソールに密接に接続します。最後に、このようなプログラムの実装は、コードの再利用やサンドボックス化を困難にします。もちろん、このような単純なプログラムの場合、デザインの複雑さに入るのは意味がありません。そうでなければ、このようなことが判明する可能性があります。

上記の問題はすべて、値の受け取りとコンソールへの出力を分離することで解決できます。派手な言語構造を使用しなくても、結果の値を呼び出しコードに返すだけで、この値を使用する責任がなくなります。

public static string FizzBuzz(int x)
{
      if (x % 3 == 0 && x % 5 == 0)
            return "FizzBuzz";
      else if (x % 3 == 0)
            return "Fizz";
      else if (x % 5 == 0)
            return "Buzz";
      else
            return x.ToString();
}

もちろん、これらは根本的な変更ではありませんが、これですでに十分です。

  1. FizzBu​​zzメソッドは、数値を取り、文字列を返す式になりました。
  2. それは他の責任や隠された影響を持たず、それが純粋な機能に変わります。
  3. 追加の依存関係や設定なしで、独立して使用およびテストできます。
  4. この関数を呼び出すコードは、結果に対して何でも自由に実行できます。現在、これは私たちの責任ではありません。

そして、これが関数型プログラミングの本質です-プログラムは任意の値をもたらす式で構成され、これらの値は呼び出し元のコードに返されます。これらの式は、原則として完全に独立しており、呼び出しの結果は入力データにのみ依存します。最上部のエントリポイント（または「エンドオブザワールド」と呼ばれることもあります）では、関数によって返される値が収集され、残りの世界と相互作用します。オブジェクト指向の専門用語では、これは「オニオンアーキテクチャ」（または「ポートとアダプタ」）と呼ばれることもあります。これは、ビジネスロジックと、外界との対話を担当する命令型の外部シェルで構成されるクリーンなコアです。

ステートメントの代わりに式を使用することは、すべてのプログラミング言語である程度使用されています。 C＃は時間の経過とともに進化し、LINQ、式ベースのメソッド、パターンマッチングなど、式を使いやすくする関数が導入されました。これらの「機能」は、F＃やHaskellのような言語では存在するため、「関数型」と呼ばれることが多く、式以外の存在はほとんど不可能です。

実際、このプログラミングスタイルはC＃チームによって推奨されています。 NDC Londonでの最近のスピーチで、Bill Wagnerは、開発者に（必須の）習慣を変更し、最新の方法を採用するよう促しています。


C＃（およびJavaなどの他の命令型言語）は機能的に使用できますが、多大な労力を必要とします。これらの言語は、機能スタイルを例外ではなく、標準にします。関数型プログラミング言語を学び、一流のスペシャリストになることをお勧めします。
