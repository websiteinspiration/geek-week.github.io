Azure DevOps拡張機能を作成する

カスタマイズされたCI / CDプロセスのプロジェクトがあります。開発者がタスクを完了し、彼の変更を開発\ qaに注入すると、ビルドが自動的に起動し、アプリケーションの新しいバージョンがテスト環境に配置されます。理想的な世界では、テスターは完了したタスクと、それらが展開されている環境について自動的に学習します。この場合、ワークフローは継続的で中断されなくなり、必要な通信が少なくなるため、集中的な作業の妨げになります。実際には、すべてがそれほどバラ色ではありません。

そしてある朝、チームリーダーは私に尋ねました：「ビルドに添付されているタグがこのビルドを通過した後、指定されたタグをハングするようにTFSのためにそのようなことを行うことができますか？」

タスクのビルド\リリースタスクを実装することにしました。さらに、すべてのビルドタスクのソースコードはgithubにあります、およびすべての情報が利用可能です。

私たちのタスクは、完了したがテストされていないタスクの色をマークします。これにより、開発者は目的のタグを付けるのを忘れた場合にすぐに気づき、QAは何をチェックする必要があるかをすぐに確認できます。これにより、タスクのステータスが視覚化され、プロジェクトの作業がスピードアップします。

この記事では、必要なロジックの実装、拡張パッケージ化について説明します。したがって、そのようなプラグインの作成方法に興味がある場合は、catへようこそ。

最もいらいらする人のために：githubと市場での準備が整った拡張機能。



Azure DevOpsには、ボード上のマスクを異なる色で着色できるフィルターを作成する機能があります。




次のようなタスクに関心があります。

  • 完成した
  • テスト環境に注がれた
  • まだ確認されていないQA

タグはポイントbとcに最適ですが、手動で設定するのは面倒で、誰もが忘れてしまいます。そのため、拡張機能を作成します。これは、展開後に自動的に付加されます。

したがって、ヒューマンファクターを減らし（開発者がタグを付けるのを忘れた）、QAを支援するために、カスタムビルドとリリースのステップが必要です（何をチェックする必要があるかすぐにわかります）。

前提条件


拡張機能を開発するには、次のものが必要です。

  1. お気に入りのIDE
  2. インストールされたTypeScript + node.js + npm（バージョン3.5.1 \ 12.4 \ 6.9.0がインストールされました）
  3. tfx-cli-拡張機能をパッケージ化するためのライブラリ（npm i -g tfx-cli）。

-gフラグの存在に注意して

ください。Microsoftには、それらが最上位にあり、なんらかの種類の拡張機能の作成方法が記載されている優れたドキュメントがあります。さらに、同じ方法で、ビルド\リリースタスクの作成に関するドックがあります。

どちらの記事も、細かい点や説明の点で欠点があると思いますので、あまり気づかなかったと思われる点を中心に説明します。

一般的に言えば、かなり多くの言語でビルド\リリースステップを記述できます。TypeScriptの例を示します。

TypeScriptを選ぶ理由


ビルドステップ 'aの最初のバージョンはPowerShell'eで作成されましたが、それを知っていたのは私たちのチームと数人だけでした。すぐに、Dockerビルドエージェントで実行されるビルドにタスクを追加しようとすると、PowerShellがなくなり、タスクが機能しなくなるという事実に直面しました。また、PowerShellの不気味が原因で、さまざまな種類のエラーが発生する場合がありました。したがって、結論-ソリューションはクロスプラットフォームである必要があります。

REST APIにアクセスするには、accessTokenを取得する必要があります

const accessToken = tl.getEndpointAuthorization('SystemVssConnection', true).parameters.AccessToken;

次に、ヘッダー認証を値「Bearer $ {accessToken}」で設定する必要があります

。ビルドにバインドされた作業項目の受信に戻ります。

URL Azure DevOpsサーバーとTeamProjectの名前は、次のように環境変数から取得できます。

const collectionUrl = process.env["SYSTEM_TEAMFOUNDATIONCOLLECTIONURI"];
const teamProject = process.env["SYSTEM_TEAMPROJECT"];


async function getWorkItemsFromBuild() {
   const buildId = process.env["BUILD_BUILDID"];
   const uri = `${collectionUrl}/${teamProject}/_apis/build/builds/${buildId}/workitems`;
   const options = createGetRequestOptions(uri);
   const result = await request.get(options);
   return result.value;
}


function createGetRequestOptions(uri: string): any {
    let options = {
        uri: uri,
        headers: {
            "authorization": `Bearer ${accessToken}`,
            "content-type": "application/json"
        },
        json: true
    };
    return options;
}

作業項目apiにパッチリクエストを送信します。

const patchOptions = getPatchRequestOptions(uri, newTags);
await request.patch(patchOptions)

function getPatchRequestOptions(uri: string, newTags: string): any {
    const options = {
        uri: uri,
        headers: {
            "authorization": `Bearer ${accessToken}`,
            "content-type": "application/json-patch+json"
        },
        body: [{
            "op": "add",
            "path": "/fields/System.Tags",
            "value": newTags
        }],
        json: true
    };
    return options
}

アセンブリおよびパッケージ拡張 'a


発生するすべての美しさのために、tsconfig.jsonをcompilerOptionsに追加することを提案します 
"outDir": "dist"
。ここでtsc 、TaskFolder内でコマンドを実行すると、distフォルダーが取得されます。これは、最終的なパッケージに移動するindex.jsになります。

なぜなら index.jsはdistフォルダーにあり、それを最終パッケージにもコピーします。task.jsonを少し修正する必要があります。

"execution": {
        "Node": {
            "target": "dist/index.js"
        }
    }

ファイルセクションのvss-extension.jsonで、最終的なパッケージに何をコピーするかを明示的に宣言する必要があります。 

"files": [
    {
      "path": "TaskFolder/dist",
      "packagePath": "TaskFolder/dist"
    },
    {
      "path": "TaskFolder/node_modules",
      "packagePath": "TaskFolder/node_modules"
    },
    {
      "path": "TaskFolder/icon.png",
      "packagePath": "TaskFolder/icon.png"
    },
    {
      "path": "TaskFolder/task.json",
      "packagePath": "TaskFolder/task.json"
    }
  ]

最後のステップ-拡張機能をパックする必要があります。

これを行うには、次のコマンドを実行します。

 tfx extension create --manifest-globs ./vss-extension.json

実行後、* .vsixファイルを取得します。このファイルはTFSにインストールされます。

PS * .vsixファイルは基本的に通常のアーカイブです。たとえば、7-zipで簡単に開くことができ、必要なものがすべて内部にあることがわかります。

vss-extension.jsonにセクションを追加できます。

"icons": {

  }
