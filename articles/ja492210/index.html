Infinidatストレージアーキテクチャの技術概要

InfiniBoxは、すぐに魔法の広場の右側に落ちる最新のストレージシステムです。その独自性は何ですか？

簡単な背景

InfiniBoxとは何ですか？これはInfinidatストレージシステムです。 Infinidatとは何ですか？これは、理想的なエンタープライズレベルのストレージプロジェクトを実装するために、Moshe Yanay（SymmetrixおよびXIVの作成者）が作成した会社です。

同社はソフトウェア開発者として作成され、実績のある機器、つまりSDSを搭載していますが、単一のモノリシックセットとして提供されています。

前書き

この記事では、InfiniBoxストレージシステム、そのアーキテクチャ、動作方法、および比較的低価格で高い信頼性（99.99999％）がどのように達成されるか、パフォーマンス、容量について説明します。ストレージシステムの基礎はそのソフトウェアであり、特にこのシステムでは、ソフトウェアに重点が置かれるため、美しい写真腺はありません。

市場に別のストレージシステムが必要なのはなぜですか？

非常に大きな容量が必要なタスクは数多くありますが、信頼性とパフォーマンスも重要です。たとえば、クラウドシステム、大企業の標準的なタスク、モノのインターネット、遺伝子研究、大規模構造のセキュリティシステムなどです。このようなタスクに最適なストレージシステムを見つけることは、特に価格を見ると非常に困難です。このようなタスクを念頭に置いて、InfiniBoxのソフトウェアアーキテクチャが構築されました。

アドレス指定

無制限の量のデータをどのように保存できますか？無制限のアドレス空間を提供することにより。これを行うために、InfiniBoxはVUA-仮想ユーザーアドレス空間を使用します。ユーザーがアクセスできるInfiniBoxで作成されたすべてのオブジェクト（ボリューム、スナップショット、ファイルシステム）がこのVUAに含まれ、これらのオブジェクトの合計サイズはVUAの現在のサイズです。アドレス指定はシンであり、ディスクに関連付けられていません。つまり、VUAサイズは使用可能なディスク容量よりもはるかに大きくなる可能性があり、実際には無制限です。



次に、このスペースを提供するパーツに分割する必要があります。そうすることで、それらのスペースを使いやすくなります。



アドレス空間全体は、VU（仮想ユニット-仮想ピース）の6つの部分に分かれています。ボリュームなどの各オブジェクトのアドレス空間は、これらの部分に均等に分散されます。記録は64kBのブロックで行われ、その過程で、このボリュームアドレスがどのVUに属しているかを信じられないほど簡単かつ迅速に理解できます（モジュロ機能であるLBA / 64kB除算の残りは、最小CPUサイクル数に対して非常に迅速に実行されます）。

小さなスペースでの作業を簡素化することに加えて、VUは物理ディスクからの最初の抽象化レベルであり、コントローラーレベルでのシステムフォールトトレランスの基礎となります。たとえば、3つのコントローラー（物理サーバー）があり、それぞれが操作を担当し、2つのVUのメインであり、他の2つのVUのバックアップです。コントローラーは、予約されているVUにサービスを提供しませんが、同時にこのVUの他のコントローラーからメタデータと書き込み操作を受信するため、メインコントローラーに障害が発生した場合、このVUでのすべての作業がすぐに取得されます。



1つのコントローラーに障害が発生した場合、他の2つがそのタスクを引き継ぎます-そのVUで作業します。



アドレス指定、スナップショット、キャッシング



VUAは、ユーザーが使用できる仮想アドレス空間であり、事実上無制限です。 VDA（仮想ディスクアドレス）は、仮想内部ストレージスペースです。そのサイズは、ハードドライブの数と容量（パリティ、メタデータ、および故障したドライブを交換するためのスペースを差し引いたもの）によって事前に厳密に決定されます。 VUAとVDA間の接続は、トライ（プレフィックスツリー）を介して構成されます。ツリーの各エントリは、ユーザー空間（VUA）から内部空間（VDA）へのポインターです。プレフィックスツリーの形式でアドレス指定すると、任意のサイズのブロックをアドレス指定できます。つまり、任意のサイズの要素（ファイル、シリアルデータストリーム、オブジェクト）をディスクに追加すると、ツリー内の1つのレコードでアドレス指定でき、ツリーはコンパクトなままです。

ただし、ツリーの最も重要な機能は、ツリーを検索して展開するときの高いパフォーマンスです。検索は、ディスク上で特定のアドレスを持つブロックを見つける必要がある読み取り操作中に実行されます。ツリーの拡張は、書き込み操作中に発生します。新しいデータをディスクに追加し、それを見つけることができるアドレスをツリーに追加する必要があります。大きな構造で作業する場合、パフォーマンスは非常に重要です。たとえば、ディスクのボリュームが増加した場合など、プレフィックスツリーを使用すると、将来のために大きなマージンを確保できます。

VUAとVDAの関係について言えること：

  1. VUAサイズはVDAサイズをはるかに超える可能性があります
  2. VUAは、そこにシンプロビジョニングが書き込まれるまで、VDAとは関係ありません。
  3. 複数のVUAが1つのVDAを参照できます（スナップショット/クローン）

したがって、VUAとVDAの構成、それらの間の接続、およびこれらの接続のアドレス指定により、非常に高速なスナップショットとシンプロビジョニングを実装できます。スナップショットの作成はメモリ内のメタデータを更新するだけであり、作業中は常に発生する操作であるため、実際にはこの操作には時間がかかりません。通常、スナップショットを作成すると、クラシックストレージシステムはトランザクションの整合性を保証するために、メタデータやI / Oの更新を停止します。これにより、I / Oレイテンシが不均一になります。問題のシステムの動作は異なります。何も停止せず、ブロックメタデータのタイムスタンプ（64 + 4KB）を使用して、操作がスナップショット内にあるかどうかを判断します。したがって、システムは作業を遅くせずに数十万ショットを撮影できます。数百のスナップショットがあるボリュームとスナップショットがないボリュームのパフォーマンスに違いはありません。すべてがメモリ内で実行され、これらは通常のプロセスであるため、ボリュームグループで1秒あたり数十のスナップショットを取得できます。これにより、スナップショットに非同期レプリケーションを実装することができます。重要なのは、パフォーマンスに影響を与えることなく、コピーの違いを数秒以内に抑えることです。

システム全体を見て、データがどのようになるかを見てみましょう。操作は、3つすべてのコントローラー（サーバー）のポートを介して受け取られます。ポートドライバーはユーザースペースで機能します。これにより、ポート上のイベントの組み合わせによってドライバーが中断された場合に、それらを簡単に再起動できます。従来の実装では、これはカーネルで行われ、問題はコントローラー全体を再起動することで解決されます。

次に、ストリームは64KB + 4KBのセクションに分割されます。これらの追加の4KBは何ですか？これはサイレントエラーに対する保護であり、チェックサム、時間、および操作に関する追加情報が含まれています。これは、この操作の分類に使用され、キャッシュの最適化と先読みに使用されます。



レコーディングキャッシングは、読み取りとは言えない非常に単純なものです。何を読むべきかがわかっていれば、読み取りキャッシュはうまく機能します。従来のシステムは、順次アクセス先読みアルゴリズムを使用しています。しかし、恣意的に何をすべきか？実際のアプリケーションの作業では完全にランダムなアクセスは非常にまれであり、正しくエミュレートすることは非常に困難です。実際の乱数のジェネレーターを作成することは、かなり興味深いタスクです。しかし、古典的なシステムで行われているように、各I / O操作を個別に検討すると、完全に連続するものを除いて、それらのすべてが恣意的で予測不可能です。ただし、I / Oストリーム全体をしばらく見ると、さまざまな操作を組み合わせたパターンを確認できます。

システムキャッシュは基本的に、ボリューム、ファイル、VUAで構築された論理構造については何も知りません。キャッシュはセクションとそのメタデータのみを調べ、その動作と属性に基づいてキャッシングを行うため、実際に接続されているさまざまなアプリケーション間の依存関係を見つけることができます。入出力操作では、アクティビティー・ベクトルが作成されます。



システムは統計を蓄積し、これらの活動ベクトルを構築し、現在のI / Oを識別して既知のベクトルにバインドするか、新しいものを構築しようとします。識別後、ベクトルに従って予測読み取りが行われます。つまり、アプリケーションの動作が予測され、予測読み取りは一見任意の負荷に対して行われます。

ディスクに書き込む

14のセクションは、ディスク用にストライプ状に組み立てられています。これは、そのようなストリップのセクションを選択する特別なプロセスによって行われます。



次に、2つのパリティセクションが考慮されます-ストライプはディスクに書き込む準備ができています。パリティは、リードソロモンコードに基づくよりも2倍高速なXORに基づくいくつかの演算を通じてカウントされます。さらにストリップ（14 + 2セクション）がRAIDグループ（RG）に割り当てられます。 RAIDグループは、いくつかのストリップを格納するためのオブジェクトにすぎません。ストリップは次のように上下にグループ化され、垂直の列はRAIDグループのメンバーと呼ばれます。 VDS（仮想ディスクスペース）はユーザーデータに使用可能なディスクスペースであり、VDAはその中のアドレスです。



RAIDグループの列またはメンバーは、1つのシェルフ（ディスクユニット）の1つのディスク（PD-物理ドライブ）に書き込まれます。 RAIDグループのメンバーが書き込まれる場所は、ディスクパーティション（DP-ドライブパーティション）と呼ばれます。ディスク上のDPの数は一定で、264に等しく、サイズはディスクのサイズによって異なります。この設計により、すべてのディスクを均等にロードできます。同時に、このアルゴリズムは、1つのRAIDグループの列を互いに異なるディスクとシェルフに可能な限り分散します。これにより、2つのディスクに障害が発生した場合、それらの共通のストリップの数は同時に最小になり、システムは数分で保護状態NからN + 1に切り替わり、2つの列がすぐになくなったストリップを再構築します（信頼性は7ナインです）。 ）

物理的な実装

システムは、アレイ内の電力チャネルやデータチャネルを含め、すべてのコンポーネントがN + 2または2Nスキームに従って保護されるように作成されています。これは電源の実装図です。

このスキームを使用すると、電源障害や1つの電源回路の一時的なシャットダウンなどの複雑なイベント中に、コントローラーとメモリの整合性を保護できます。 UPSは管理可能です。これにより、正確な充電情報を取得し、キャッシュサイズを柔軟に変更できるため、コントローラーは常にリセットする時間を確保できます。つまり、バッテリーが完全に充電されている場合にのみキャッシュがオンになると、従来のスキームとは異なり、システムはキャッシュをより早くアクティブに使用し始めます。

これは、システム内のデータチャネルの図です。



コントローラーはInfinibandを介して相互接続され、SASを介してドライブに接続されます。各コントローラーは、システム内のすべてのディスクにアクセスできます。さらに、コントローラーとディスク間の接続が機能しない場合、コントローラーは、Infinibandを介してプロキシとして機能する別のコントローラーを介してデータを要求できます。シェルフには、ディスクへの同時アクセス用のSASスイッチが含まれています。各シェルフには60個のディスクが含まれ、シェルフは2つ、4つ、または8つで、合計480個のディスクで3、4、6、8、または12 TBになります。ユーザーが利用できる総容量は圧縮なしで4.1PB以上です。圧縮といえば、パフォーマンスを損なうことなく圧縮を実装するために、メモリは圧縮されません。その結果、システムは圧縮をオンにした状態でさらに高速に動作することがあります-読み取り時には、読み取る必要が少なく、プロセッサの電力は十分に余裕があります。記録時には、答えはメモリから直接送られ、圧縮はディスクへの直接記録中に非同期で行われます。

コントローラーには、2つのディスクグループが含まれています。1つはシステムで、RAMをフラッシュし、2つ目はSSDで読み取り操作をキャッシュします（システムあたり最大368TB）。このように大きなキャッシュを使用すると、大きなチャンクを予測的に読み取ることができます。ストライプ内のデータはほぼ同じアクセス頻度で選択されるため、このような大きなチャンクは物理ディスクの負荷を軽減するだけでなく、近い将来需要が高まる可能性があります。
