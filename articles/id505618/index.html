Templat GRASP: Pembuat

Halo Habr. Nama saya Vladislav Rodin. Saat ini saya adalah kepala kursus Arsitek Beban Tinggi di OTUS, dan saya juga mengajar kursus arsitektur perangkat lunak.

Khusus untuk permulaan set baru untuk kursus "Arsitektur dan Pola Desain" Saya menulis sedikit materi, yang dengan senang hati saya bagikan kepada Anda.


pengantar


Dijelaskan dalam buku Craig Larman Menerapkan UML dan pola, edisi ke-3, pola GRASP merupakan generalisasi dari pola GoF, serta konsekuensi langsung dari prinsip-prinsip OOP. Mereka melengkapi langkah yang hilang di tangga logis, yang memungkinkan Anda untuk mendapatkan pola GoF dari prinsip-prinsip OOP. Templat GRASP lebih mungkin bukan pola desain (seperti GoF), tetapi prinsip dasar untuk distribusi tanggung jawab antar kelas. Praktek menunjukkan bahwa mereka tidak terlalu populer, namun, analisis kelas yang dirancang menggunakan set lengkap pola GRASP adalah prasyarat untuk menulis kode yang baik.

Daftar lengkap template GRASP terdiri dari 9 elemen:

  • Pakar informasi
  • Pencipta
  • Pengendali
  • Kopling rendah
  • Kohesi tinggi
  • Polimorfisme
  • Fabrikasi murni
  • Tipuan
  • Variasi yang dilindungi

Terakhir kali kami membahas prinsip Pakar Informasi . Sekarang saya mengusulkan untuk mempertimbangkan pencipta yang mirip dengannya.

Pencipta


Susunan kata


Pola ini memecahkan masalah khas yang sama dengan pendahulunya: kelas yang membutuhkannya harus membuat instance kelas .

Contoh pelanggaran


Pertimbangkan masalah yang sama dengan pesanan dan barang. Anggap kode yang kami tulis sesuai dengan Pakar Informasi:

@Setter
@Getter
@AllArgsConstructor
public class Order {
    private List<OrderItem> orderItems;
    private String destinationAddress;
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }
}

@Setter
@Getter
@AllArgsConstructor
public class OrderItem {
    private Good good;
    private int amount;

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

Meskipun tampaknya sepele prinsip yang kita pelajari, sekali lagi, dalam beberapa kode klien, Anda dapat menemukan ini:

public class Client {
    public void doSmth() {
        Good good = new Good("name", 2);
        OrderItem orderItem = new OrderItem(good, amount);
        List<OrderItem> orderItems = new ArrayList<>();
        orderItems.add(orderItem);
        Order order = new Order(orderItems, "abc");
        // client code 
    }
}

Jika Anda membuat diagram kelas UML, Anda dapat menemukan bahwa kelas Klien sekarang bergantung pada kelas Order dan semua internalnya: OrderItem dan Good. Dengan demikian, kami tidak dapat menggunakan kembali kelas Klien tanpa kelas-kelas di atas, yang tidak dibutuhkan oleh Klien. Kami sebenarnya membatalkan hasil dari semua upaya untuk mematuhi Pakar Informasi, karena kelas Klien membuat semua objek.

Dalam proyek lawas, Anda sering dapat melihat bagaimana satu kelas membuat objek dari yang lain dan meneruskannya sebagai parameter dalam metode melalui 5-6 kelas, di mana objek ini tidak digunakan. Ini tidak lebih dari menambahkan beberapa dependensi dari awal.

Contoh aplikasi


Mari kita sesuaikan distribusi tanggung jawab antar kelas sehingga distribusi tidak hanya memuaskan Ahli Informasi, tetapi juga Pencipta:

@Setter
@Getter
public class Order {
    private List<OrderItem> orderItems = new ArrayList<>();
    private String destinationAddress;
    
    public Order(String destinationAddress) {
        this.destinationAddress = destinationAddress;
    }
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }

    public void addOrderItem(int amount, String name, int price) {
       orderItems.add(new OrderItem(amount, name, price));
   }
}

@Setter
@Getter
public class OrderItem {
    private Good good;
    private int amount;

    public OrderItem(int amount, String name, int price) {
        this.amount = amount;
        this.good = new Good(name, price);
    }

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

Sekarang jumlah dependensi antar kelas akan minimal. Kode klien agak disederhanakan dan mungkin terlihat seperti ini:

public class Client {
    public void doSmth() {
        Order order = new Order("address");
        order.addOrderItem(amount, name, price);
        // client code 
    }
}

Kesimpulan


Pembuat, yang sering dilupakan oleh pengembang, dapat dianggap sebagai kasus khusus Ahli Informasi, karena panggilan konstruktor sama dengan panggilan metode. Ketaatannya bersama dengan Pakar Informasi memungkinkan untuk mencapai jumlah koneksi minimum antara kelas dan penggunaan kembali yang lebih besar.
