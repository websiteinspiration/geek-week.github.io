SwiftとiOSユニバースのファクトリーメソッドと抽象ファクトリー

「ファクトリー」という言葉は、プログラマーが自分の（または他の人の）プログラムについて話し合うときに最も頻繁に使用する言葉の1つです。しかし、その中に埋め込まれた意味は大きく異なる可能性があります。オブジェクトを生成するクラスである可能性があります（ポリモーフィックかどうか）。そして、あるタイプのインスタンス（静的かそうでないか）を作成するメソッド。それだけでなく、任意の生成メソッド（コンストラクターを含む）でも発生します。

もちろん、何かのインスタンスを生成するすべてのものを「工場」という言葉で呼ぶことができるわけではありません。さらに、この言葉の下では、ギャングオブフォー兵器からの2つの異なる生成パターンを隠すことができます。「ファクトリーメソッド」と「抽象ファクトリー」です。これらの詳細については、それらの古典的な理解と実装に特に注意を払いながら、少し掘り下げたいと思います。

抽象工場


Kerivsky氏の著書では、このテンプレートの使用が役立つ2つのケースが示されています。

1つは、共通のインターフェースによって接続された特定のクラスに関する知識のカプセル化です。この場合、ファクトリであるタイプのみがこの知識を持ちます。ファクトリのパブリックAPIは、共通のインターフェースタイプのインスタンスを返す一連のメソッド（静的または非静的）で構成され、（特定の目的でどのメソッドを呼び出す必要があるかを理解するために）いくつかの「トーキング」名を持ちます。

2番目の例は最初の例と非常に似ています（一般に、パターンを使用するためのすべてのシナリオは、だいたい互いに似ています）。これは、同じグループの1つ以上のタイプのインスタンスがプログラムの別の場所に作成される場合です。この場合、ファクトリーはインスタンスを作成するコードに関する知識を再びカプセル化しますが、モチベーションは少し異なります。たとえば、これらのタイプのインスタンスを作成するプロセスが複雑で、コンストラクターの呼び出しに限定されていない場合、これは特に当てはまります。「iOS」

での開発のトピックに近づくには、サブクラス化の練習が便利です。実際、これは間違いなく「iOS」開発で最も一般的なタイプの1つであり、ほとんどの場合、使用前に「継承」されます。特定のサブクラスは、クライアントコードにとって重要ではないこともよくあります。UIViewController
私は、コード例をGang of Fourの古典的な実装にできるだけ近づけるように努めますが、実際には、コードは多くの場合、何らかの方法で単純化されています。そして、テンプレートを十分に理解するだけで、より自由に使用できるようになります。

詳細な例


アプリケーションで車両を取引していて、マッピングが特定の車両のタイプに依存しているとしますUIViewController。異なる車両には異なるサブクラスを使用します。さらに、すべての車両の状態は異なります（新品と中古）。

enum VehicleCondition{
    case new
    case used
}

final class BicycleViewController: UIViewController {
    
    private let condition: VehicleCondition
    
    init(condition: VehicleCondition) {
        self.condition = condition
        super.init(nibName: nil, bundle: nil)
    }
    required init?(coder aDecoder: NSCoder) {
        fatalError("BicycleViewController: init(coder:) has not been implemented.")
    }
    
}

final class ScooterViewController: UIViewController {
    
    private let condition: VehicleCondition
    
    init(condition: VehicleCondition) {
        self.condition = condition
        super.init(nibName: nil, bundle: nil)
    }
    required init?(coder aDecoder: NSCoder) {
        fatalError("ScooterViewController: init(coder:) has not been implemented.")
    }
    
}

したがって、同じグループのオブジェクトのファミリーがあり、そのタイプのインスタンスは、ある条件に応じて同じ場所に作成されます（たとえば、ユーザーがリスト内の製品をクリックしたか、それがスクーターか自転車かによって、適切なコントローラを作成します）。コントローラーコンストラクターには、毎回設定する必要があるいくつかのパラメーターがあります。これらの2つの議論は、適切なコントローラを作成するためのロジックについての知識のみを持つ「ファクトリ」を作成することに賛成ですか？

もちろん、この例は非常に単純であり、実際のプロジェクトで同様の場合、「工場」の導入は明示的な「オーバーエンジニアリング」になります。。それでも、2種類の車両がなく、デザイナーが複数のパラメーターを持っていると想像すると、「工場」の利点はより明白になります。

それでは、「抽象ファクトリ」の役割を果たすインターフェースを宣言しましょう。

protocol VehicleViewControllerFactory {
    func makeBicycleViewController() -> UIViewController
    func makeScooterViewController() -> UIViewController
}

（Swift言語での「API」の設計に関するかなり短い「ガイドライン」では、 makeという単語で始まるファクトリメソッドを呼び出すことをお勧めします。） （ギャングフォーの本の例はC ++で提供され、継承と仮想関数に基づいています）もちろん、Swiftを使用すると、プロトコル指向プログラミングのパラダイムに近づきます。） 抽象ファクトリーインターフェイスには、自転車とスクーターを販売するためのコントローラーを作成するための2つのメソッドしか含まれていません。メソッドは、特定のサブクラスではなく、共通の基本クラスのインスタンスを返します。したがって、特定のタイプに関する知識の範囲は、それが本当に必要な領域に限定されます。





「コンクリートファクトリ」として、抽象ファクトリインターフェースの2つの実装を使用します。

struct NewVehicleViewControllerFactory: VehicleViewControllerFactory {
    
    func makeBicycleViewController() -> UIViewController {
        return BicycleViewController(condition: .new)
    }
    func makeScooterViewController() -> UIViewController {
        return ScooterViewController(condition: .new)
    }
    
}

struct UsedVehicleViewControllerFactory: VehicleViewControllerFactory {
    
    func makeBicycleViewController() -> UIViewController {
        return BicycleViewController(condition: .used)
    }
    func makeScooterViewController() -> UIViewController {
        return ScooterViewController(condition: .used)
    }
    
}

この場合、コードからわかるように、特定の工場がさまざまな条件の車両（新品と中古）を担当しています。

適切なコントローラを作成すると、次のようになります。

let factory: VehicleViewControllerFactory = NewVehicleViewControllerFactory()
let vc = factory.makeBicycleViewController()

ファクトリのカプセル化クラス


さて、Kerivskyが彼の本で提供しているユースケースについて簡単に説明します。

最初のケースは、特定のクラスのカプセル化に関連しています。たとえば、車両に関するデータを表示するために同じコントローラを使用します。

final class BicycleViewController: UIViewController { }
final class ScooterViewController: UIViewController { }

プラグインライブラリなどの別のモジュールを扱っているとします。この場合、上記で宣言されたクラスは（デフォルトで）残りinternal、ファクトリーはライブラリーのパブリック「API」として機能し、そのメソッドでコントローラーの基本クラスを返すため、ライブラリー内の特定のサブクラスに関する知識が残ります。

public struct VehicleViewControllerFactory {

ファクトリー内でのオブジェクトの作成に関する知識の移動


2番目の「ケース」は、オブジェクトの複雑な初期化について説明しています。Kerivskyは、コードを簡略化してカプセル化の原則を保護する方法の1つとして、初期化プロセスに関する知識の工場外への拡散を制限することを示唆しています。

同時に車を販売したいとします。そして、これは間違いなく、より多くの特性を備えた、より複雑な手法です。たとえば、使用する燃料の種類、トランスミッションの種類、リムのサイズに制限します。

enum Condition {
    case new
    case used
}

enum EngineType {
    case diesel
    case gas
}

struct Engine {
    let type: EngineType    
}

enum TransmissionType {
    case automatic
    case manual
}

final class CarViewController: UIViewController {
    
    private let condition: Condition
    private let engine: Engine
    private let transmission: TransmissionType
    private let wheelDiameter: Int
    
    init(engine: Engine,
         transmission: TransmissionType,
         wheelDiameter: Int = 16,
         condition: Condition = .new) {
        self.engine = engine
        self.transmission = transmission
        self.wheelDiameter = wheelDiameter
        self.condition = condition
        
        super.init(nibName: nil, bundle: nil)
    }
    required init?(coder aDecoder: NSCoder) {
        fatalError("CarViewController: init(coder:) has not been implemented.")
    }
    
}

対応するコントローラーの初期化の例：

let engineType = EngineType.diesel
let engine = Engine(type: engineType)

let transmission = TransmissionType.automatic

let wheelDiameter = 18

let vc = CarViewController(engine: engine,
                           transmission: transmission,
                           wheelDiameter: wheelDiameter)

私たちは、専門工場の「肩」にこれらすべての「小さなこと」の責任を負うことができます。

struct UsedCarViewControllerFactory {
    
    let engineType: EngineType
    let transmissionType: TransmissionType
    let wheelDiameter: Int
    
    func makeCarViewController() -> UIViewController {
        let engine = Engine(type: engineType)
        return CarViewController(engine: engine,
                                 transmission: transmissionType,
                                 wheelDiameter: wheelDiameter,
                                 condition: .used)
    }
    
}

そして、この方法でコントローラを作成します：

let factory = UsedCarViewControllerFactory(engineType: .gas,
                                           transmissionType: .manual,
                                           wheelDiameter: 17)
let vc = factory.makeCarViewController()

工場法


2番目の「単一ルート」テンプレートは、特定の生成された型に関する知識もカプセル化しますが、この知識を特殊なクラス内に隠すのではなく、ポリモーフィズムによってカプセル化します。彼の本のKerivskyはJavaでの例を示し、抽象クラスの使用を提案していますが、Swiftユニバースの住民はこの概念に精通していません。ここには独自の雰囲気とプロトコルがあります。
本「Gangs of Four」は、テンプレートが「仮想コンストラクタ」としても知られていると報告していますが、これは無駄ではありません。「C ++」では、派生クラスで再定義される仮想関数が呼び出されます。言語はデザイナーに仮想を宣言する機会を与えません、そしてそれはこのパターンの発明につながった望ましい行動を模倣する試みであった可能性があります。

ポリモーフィックオブジェクトの作成


テンプレートの有用性の典型的な例として、階層内で、このメソッドで作成および使用されるオブジェクトを除いて、異なるタイプが1つのメソッドの同じ実装を持っている場合を考えます。解決策として、このオブジェクトを個別のメソッドで作成して個別に実装し、一般的なメソッドを階層の上位に上げることが提案されています。したがって、異なるタイプはメソッドの一般的な実装を使用し、このメソッドに必要なオブジェクトは多態的に作成されます。

たとえば、車両を表示するためのコントローラーに戻りましょう。

final class BicycleViewController: UIViewController { }
final class ScooterViewController: UIViewController { }

また、特定のエンティティを使用してそれらを表示するとします。たとえば、これらのコントローラーを別のコントローラーからモーダルに表すcoordinatorなどです。

protocol Coordinator {
    var presentingViewController: UIViewController? { get set }
    func start()
}

この場合、start()異なるコントローラーを作成することを除いて、メソッドは常に同じ方法で使用されます。

final class BicycleCoordinator: Coordinator {
    
    weak var presentingViewController: UIViewController?
    
    func start() {
        let vc = BicycleViewController()
        presentingViewController?.present(vc, animated: true)
    }
    
}

final class ScooterCoordinator: Coordinator {
    
    weak var presentingViewController: UIViewController?
    
    func start() {
        let vc = ScooterViewController()
        presentingViewController?.present(vc, animated: true)
    }
    
}

提案された解決策は、別のメソッドで使用されるオブジェクトの作成を行うことです：

protocol Coordinator {
    
    var presentingViewController: UIViewController? { get set }
    
    func start()
    func makeViewController() -> UIViewController
    
}

そして主な方法は基本的な実装を提供することです：

extension Coordinator {
    
    func start() {
        let vc = makeViewController()
        presentingViewController?.present(vc, animated: true)
    }
    
}

この場合の特定のタイプは次の形式になります。

final class BicycleCoordinator: Coordinator {
    
    weak var presentingViewController: UIViewController?
    
    func makeViewController() -> UIViewController {
        return BicycleViewController()
    }
    
}

final class ScooterCoordinator: Coordinator {
    
    weak var presentingViewController: UIViewController?
    
    func makeViewController() -> UIViewController {
        return ScooterViewController()
    }
    
}
