クエリのカウント：基本的なDjangoパフォーマンステスト

みなさん、こんにちは。昨日開始された「Web-developer in Python」コースの受講生向けに、別の有用な資料の翻訳を用意しました。




TDDなどのテスト方法や、アプリケーションのビジネスロジックをテストする方法についてよく耳にします。ただし、アプリケーションのパフォーマンスのテストは、まったく別のタスクです。さまざまな方法がありますが、最も一般的なアプローチは、アプリケーションにDDoS攻撃を実行してその動作を観察できる環境を作成することです。これは非常に興味深いトピックですが、今日はお話ししたくありません。今日は、デフォルトのDjangoユニットテストを使用して実行できる、より簡単なテスト、つまり、アプリケーションがデータベースにアクセスする回数をテストします。

これをテストすることは非常に簡単で、これはまさに初期段階でアプリケーションのパフォーマンスを損なう可能性がある側面です。この側面は、何かがゆっくりと動作し始めたときにテストされる最初のものです。良いニュースは、この種のテストを作成するために知っておくべきことが1つだけあることです。assertNumQueriesメソッドであり、使い方は非常に簡単です。次に例を示します。

from django.test import TestCase, Client
from django.urls import reverse
from trucks.models import Truck

class TrucksTestCase(TestCase):
    def test_list_trucks_view_performance(self):
        client = Client()

        Truck.objects.create(...)

        with self.assertNumQueries(6):
            response = client.get(reverse("trucks:list_trucks"))

        self.assertEqual(response.context["trucks_list"], 1)

上記のコードは、ビューの間、"trucks:list_trucks"アプリケーションがデータベースにアクセスするのは6回だけであると主張しています。ただし、他にもあります。開始する前に、まず新しいオブジェクトを作成し、Truckその後、ビューのコンテキストデータにtrucks_list少なくとも1つのオブジェクトがあると言います。この種のテストでは、空のデータセットでテストしないことを保証する必要があるため、これは重要です。クラスをインスタンス化するだけでTruckは十分ではないことを理解することが重要です。コンテキストに含まれているかどうかを確認する必要があります。おそらく、トラックのリストをフィルタリングしているため、インスタンスTruckが結果に含まれない可能性があります。

上記のすべてを実行して、すでに大きな進歩を遂げましたが、忘れがちなもう1つの重要なステップがあります。ビューをスケーリングする場合は、返されるアイテムの数が増えるときにパフォーマンスが低下しないようにする必要があります。結局、データベースに6回ではなく1つの要素を取得するとパフォーマンスの問題が残りますが、要素が100の場合は106です。一定数のデータベース呼び出しが必要です。これは、返されるアイテムの数に依存しません。幸い、この問題も非常に簡単に解決できます。データベースに1つ（または複数）の要素を追加して、ヒット数を再度数える必要があります。これは、テストが最終バージョンでどのように見えるかです。

from django.test import TestCase, Client
from django.urls import reverse
from trucks.models import Truck

class TrucksTestCase(TestCase):
    def test_list_trucks_view_performance(self):
        client = Client()

        Truck.objects.create(...)

        with self.assertNumQueries(6):
            response = client.get(reverse("trucks:list_trucks"))

        self.assertEqual(response.context["trucks_list"], 1)

        Truck.objects.create(...)

        with self.assertNumQueries(6):
            response = client.get(reverse("trucks:list_trucks"))

        self.assertEqual(response.context["trucks_list"], 2)

コンテキストで返されたアイテムの数を再度確認しますが、2回目の実行では2台のトラックが予想されることに注意してください（Truck）。この動作の理由は、最初のケースと同様です。

新しいデータを追加するときにデータベースへの呼び出しの数を一定にすることは、一般に呼び出しの数を減らすよりも優先されます。

最後に行うことは、データができる限り水和されていることを確認することです。つまり、ビューの処理中に使用される関連データを作成する必要があります。そうしないと、アプリケーションがテストよりも本番環境でより頻繁にデータベースにアクセスするリスクがあります（ただし、成功する可能性はあります）。この例ではTruckDriver、私たちの会社を作成する必要がありましたTruck。

from trucks.models import Truck, TruckDriver
...
        truck = Truck.objects.create(...)
        TruckDriver.objects.create(name="Alex", truck=truck)

上記の手順を実行した後、データベース呼び出しの数が一定ではなくなった場合は、select_related メソッドとprefetch_relatedメソッドの詳細を調べてください。

今日はこれですべてです。この瞬間から、プロジェクトの最初にデータベースへのアプリケーションのクエリ数を確認し始めてください。これにはそれほど時間はかかりませんが、アプリケーションのユーザー数の増加に伴って発生する可能性のある問題を防ぐことができます。

ちなみにコースはまだ受けられます。またね。
