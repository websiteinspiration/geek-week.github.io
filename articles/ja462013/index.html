Dockerストレージ（docker root）移行の問題履歴

数日前までに、サーバーの1つで、Dockerストレージ（Dockerがコンテナー、イメージのすべてのファイルを格納するディレクトリ）を、
より容量の大きい別のパーティションに移動することが決定されました。その仕事は、どうやら、取るに足らないものであり、トラブルの前兆ではありませんでした...

はじめに：

1.アプリケーションのすべてのコンテナを停止して終了します。

docker-compose down

多数のコンテナーがあり、それらが異なる構成である場合、これを行うことができます。 

docker rm -f $(docker ps -q)

2. Dockerデーモンを停止します。 

systemctl stop docker

3.ディレクトリを目的の場所に移動します。 

cp -r /var/lib/docker /docker/data/storage

4. Dockerにデーモンに通知して、新しいディレクトリを探すようにします。いくつかのオプションがあります。-gフラグを使用してデーモンに新しいパスを指定するか、使用したsystemd configsを使用します。まあ、またはシンボリックリンク。あまり描きません。インターネットには、Dockerルートを新しい場所に移植するためのマニュアルがたくさんあります。

5. dockerデーモンを起動し、それがどこにあるべきかを確認します。

systemctl status docker

出力行の1つに、次のように表示されます。
├─19493 /usr/bin/dockerd --data-root=/docker/data/storage

オプションがデーモンに渡されたことを確認しました。次に、オプションが適用されたかどうかを確認します（ありがとう inkvizitor68sl）！
docker info | awk '/Root Dir/ {print $NF}'

6.アプリケーションを開始します。 
docker-compose up -d

7.ここで確認し

ます。ここで最も興味深いのは、DBMS、MQなどです。ベースはそのままで、nginx以外はすべて機能します。 Kerberos と遊女を持つ独自のnginxアセンブリがあります。また、コンテナのログを表示すると、彼が/ var / tmpに書き込むことができなかったことが示され、アクセスが拒否されました。ウイスキーで指を伸ばして状況を分析しようとしています…どうですか？ Dockerイメージは変更されませんでした。ディレクトリを移動しました。それは常に機能し、ここにそれがあなたのためです...実験のために、私はペンでコンテナに入り、このディレクトリへの権限を変更しました。ルート、ルート755、与えられたルート、ルート777がありました。そしてすべてが始まった...頭の中で思いが響いた-ある種のナンセンス...私は何かを考慮に入れていなかったと思う...

転送中のファイルへのアクセス権が気に入ったと判断しました。彼らはアプリケーション、dockerデーモンを停止し、新しいディレクトリを削除し、すでに使用している/ var / lib / dockerディレクトリのコピーを作成しましたrsync -a。

私は今、すべてが順調であると思います、私たちはドッカー、アプリケーションを上げています。

III ...問題が残っています...目がぴくぴく動いた。さまざまなテストを実行している仮想マシンのコンソールに急いで行き、このnginxイメージを持っていて、コンテナー内に登りました。ここで、ルート、ルート777の権限は/ var / tmpディレクトリにあります。つまり、私が手で設定したものと同じものです。しかし、画像は同じです！

どこでもFS xfsを使用しました。

コマンドで比較した

docker inspect my-nginx:12345

すべてのハッシュは同一で、すべて1対1です。サーバーと仮想マシンの両方で。ローカルのnginxイメージを削除し、レジストリを使用して再度スプールしました。これは、いくつかの理由で同じマシン上にあります。そして問題は同じです...今私の第二の目がぴくぴくと動いています。

「AAAAAAA」などの悲鳴以外に、頭の中にどんな思いがあったか覚えていません。午前4時の路上で、画像レイヤーのハッシュの原理を理解するためにDockerソースが使用されました。彼はエネルギーの3番目の缶を開きました。そして最後に、それはしかし、ハッシュは、アカウントにのみファイル、その内容を取ることを私に夜が明けない権利アクセスの！つまり、不可解な方法で、権利が無効にされ、selinuxが無効になり、aclが使用されず、スティッキービットが存在しなくなります。

ローカルイメージを削除し、Dockerレジストリからもイメージを削除して、再起動しました。そしてそれはうまくいった。転送中に、ローカルイメージ内とレジストリにあるイメージ内の両方で権利が侵害されたことが判明しました。私が言ったように、いくつかの理由で、それは同じ手押し車にありました。その結果、同じディレクトリ/ var / lib / dockerにあります。

そして、彼らが港湾労働者の視線を古いカタログに戻そうとしたかどうかという疑問を予想しました-いいえ、悲しいかな、状況は許しませんでした。はい、本当に理解したかったのです。

この記事を書いた後、問題の解決策は私には明らかであるように見えますが、分析の時点ではそのようではなかったようです。私は正直にググって、似たような状況を見つけませんでした。
