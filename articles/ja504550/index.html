Amazon EKSマルチテナントクラスターを設定する


私たちは常にクラウドネイティブのテクノロジーを使用し、Kubernetesプラットフォームのコンテナでシステムを起動しています。このテクノロジーは、その柔軟性と仮想マシンをベアメタルに直接インストールできるため、コンテナのロードオーケストレーションに最適です。Kubernetesがステートレスな単純なワークロードにのみ適していたとしたら、データベースの保存、自動車のトレーニング、複雑なアプリケーションのデプロイが可能になりました。

2018年にAmazon EKSが無料になって 以来、彼はAWSでワークロードKubernetesを実行することを選択しました。独自のリソースにKubernetesをポストすることは高価で困難であり、一般に、ビジネスの観点からは疑わしいです。Amazon EKSなどのワークロードとサービスをより適切に管理します。

Amazon EKSは、オーケストレーションKubernetesのサポートレイヤーを介した「マルチテナントnogo」（マルチテナント）サービスに最適です。同じサーバーでさまざまなワークロードを実行できます。これにより、Amazon EC2インスタンスの密度が向上します。。SaaSアプリケーションの起動時に、ログインセキュリティとマルチテナントデータの実装で問題が発生する可能性があります。Amazon EKS for SaaSを使用するすべてのユーザーは、いくつかの便利なトリックを見つけることができます。

ResourceQuotaによるソフト分離、またはテナントがリソースを自身に描画しないようにする方法は？


名前空間は、分離のためだけに使用されるのではありません。また、CPU、メモリ、ストレージなどのリソースの機能を正直に分散するため。特定の名前空間で特定のワークロードのリソースを独占しないようにするために、ResourceQuotaとソフト分離方法が役立ちます。ResourceQuotaオブジェクトは、名前空間での総リソース消費を制限するために使用されます。これは、1：1のマッピング比率- 名前空間/テナントのテナントでも機能します。コンテナがリソースを独占しないことを確認するには、LimitRangeオブジェクトを作成することをお勧めします。グローバルに定義されたPriorityClassを通じて利用可能なリソースを割り当てる-優先ワークロードに基づくオブジェクト。

ネットワーク設定による通信の管理


ネットワーク設定は、複数の基準に基づいてインバウンドとアウトバウンドのアクセス許可を制御します。以下のためにマルチテナントEKS、テナントがされた名前空間にマッピングされました。同じネームスペース内のネームスペースとポッド間のデータ交換を制限するには、namespaceSelectorとpodSelectorの値を使用します。これは、テナントAの名前空間がすでに構成され、ラベルに関連付けられていることを意味します。

kubectl label namespace/tenant-a tenant=a

以下のコードは、ネットワーク設定により、唯一のテナント共有名前空間トラフィックがラベルapp：apiでどのように処理されるかを示しています。

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: same-namespace-only
  namespace: tenant-a
spec:
  podSelector:
	matchLabels:
  	app: api
  policyTypes:
  - ingress
  - egress
  ingress:
  - from:
	- namespaceSelector:
    	matchLabels:
      	tenant: a
  egress:
  - to:
	- namespaceSelector:
    	matchLabels:
      	tenant: a
