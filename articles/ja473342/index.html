コードやデータセットを含むすべての記事の資料は、GitHubから無料で入手できます。

GitHubのデータはcsv形式で表示され、44のエントリが含まれています。原則として、これらは例の分析だけでなく、使用することもできます（使用する必要があります）。

データ列の意味は次のとおりです。

  • num-レコードインデックス
  • 年-記録年
  • 月-記録の月
  • total_appeals-月あたりの合計ヒット数
  • eals_to_mayor-市長への上訴の総数
  • res_positive-肯定的な決定の数
  • res_explained-明確化のための呼び出しの数
  • res_negative-否定的な決定を持つ呼び出しの数
  • El_form_to_mayor-電子形式での市長への上訴の数
  • Pap_form_to_mayor — to_10K_total_VAO…to_10K_total_YUZAO – 10000
  • to_10K_mayor_VAO… to_10K_mayor_YUZAO– 10000

データ収集プロセスを自動化して手動で収集する方法が見つからなかったため、少し誤解することがありました。そうでなければ、データの信頼性は著者に任されます。

現時点では、モスクワ政府のウェブサイト全体で、データは2016年1月から2019年8月まで表示されています（9月には、一部のデータが欠落しています）。したがって、44レコードになります。もちろん少しですが、これで十分です。

モデルをトレーニングするには 、Trainメソッドが必要であることを確認しましょう。

public static ITransformer Train(MLContext mlContext, string dataPath)
        {
            IDataView dataView = mlContext.Data.LoadFromTextFile<MayorAppel>(dataPath, hasHeader: true, separatorChar: ',');
            var pipeline = mlContext.Transforms.CopyColumns(outputColumnName: "Label", inputColumnName: "ResPositive")
                .Append(mlContext.Transforms.Categorical.OneHotEncoding(outputColumnName: "MonthEncoded", inputColumnName: "Month"))
                .Append(mlContext.Transforms.Concatenate("Features", "Year", "MonthEncoded", "TotalAppeals", "AppealsToMayor",
                "ResExplained", "ResNegative", "ElFormToMayor", "PapFormToMayor", "To10KTotalVAO", "To10KMayorVAO",
                "To10KTotalZAO", "To10KMayorZAO", "To10KTotalZelAO", "To10KMayorZelAO", "To10KTotalSAO", "To10KMayorSAO"
                , "To10KTotalSVAO", "To10KMayorSVAO", "To10KTotalSZAO", "To10KMayorSZAO", "To10KTotalTiNAO", "To10KMayorTiNAO"
                , "To10KTotalCAO", "To10KMayorCAO", "To10KTotalYUAO", "To10KMayorYUAO", "To10KTotalYUVAO", "To10KMayorYUVAO"
                , "To10KTotalYUZAO", "To10KMayorYUZAO")).Append(mlContext.Regression.Trainers.FastTree());
            var model = pipeline.Fit(dataView);
            return model;
        }


最初に、トレーニングサンプルからデータを読み取ります。次に、チェーン内で、予測するパラメーター（ラベル）を決定します。

私たちの場合、これは1か月あたりの市民の訴えに関して問題が解決された問題の数です。
この場合、回帰に基づいて決定木をブースティングするモデルが使用されるため、すべての符号を数値にする必要があります。
Accord.NETの場合とは異なり、既製のOneHotEncodingソリューションがすぐにドキュメントに表示されます。

前述のように、列を形成するために残った後、それらはすべて同じデータ型（この場合はfloat）である必要があります。

結論として、完成したモデルを形成して返します。

次に、モデルによる予測の質を評価します。

 private static void Evaluate(MLContext mlContext, ITransformer model)
        {
            
            IDataView dataView = mlContext.Data.LoadFromTextFile<MayorAppel>(_testDataPath, hasHeader: true, separatorChar: ',');
            var predictions = model.Transform(dataView);
            var metrics = mlContext.Regression.Evaluate(predictions, "Label", "Score");

            Console.WriteLine();
            Console.WriteLine($"*************************************************");
            Console.WriteLine($"*       Model quality metrics evaluation         ");
            Console.WriteLine($"*------------------------------------------------");
            Console.WriteLine($"*       RSquared Score:      {metrics.RSquared:0.##}");
            Console.WriteLine($"*       Root Mean Squared Error:      {metrics.RootMeanSquaredError:#.##}");

        }

テストサンプル（セットからの最後の4か月）を読み込み、Transform（）メソッドを使用して、トレーニング済みモデルのテストデータの予測を取得します。次に、メトリックを計算して出力します。この場合、決定係数と標準偏差です。前者は理想的には1になり、後者は本質的にゼロになる傾向があります。

原則として、予測を行うためにこの方法は必要ありませんでしたが、モデルがどれほどひどく予測しているかを理解するのは良いことです。

この例では、トレーニング済みモデルとテストデータセットに基づいて単一の予測を取得できるPredictionEngineクラスを使用しました。

予測用のデータで3つの「プローブ」を作成します。
1つ目はデータの最小セット（1か月と1年のみ）、2つ目は平均、3つ目は完全な特性セットです。

3つの異なる予測を取得して印刷します。

スクリーンショット（Windows 10 x64）でわかるように、地区の住民10,000人あたりの呼び出し数に関するデータを追加すると、この場合はすべてが台無しになりますが、残りのデータを追加すると、予測の精度がわずかに向上します。

結論として、約束どおり、ML.NETとAccord.NETおよびPython機械学習ライブラリとの主観的な比較分析を少し行います。

1.開発者が機械学習の分野のトレンドに対応しようとしているように感じられます。もちろん、Anacondaに多数のライブラリがインストールされているPythonでは、このタスクをよりコンパクトに解決でき、開発に費やす時間を短縮できます。しかし、全体的に見て、ML.NETの問題解決アプローチは、Pythonを使用して機械学習の問題を解決することに慣れている人にとってはフレンドリーだと思われます。

2. Accord.NET Frameworkとの比較-ML.NETは、Pythonで機械学習を試した人にとってより便利で有望に見えます。2年前にAccord.NETで何かを書こうとしたとき、いくつかのクラスとメソッドの説明と例がひどく欠けていたのを覚えています。この点で、Ml.NETは、フレームワークがAccord.NETよりもはるかに新しいという事実にもかかわらず、ドキュメントに関して少し良い状況にあります。もう1つの重要な要素は、GitHubでのアクティビティによって判断されるML.NETは、Accord.NETよりもはるかに集中的に開発されており、ロシア語のトレーニング資料が多いことです。

その結果、一見すると、ML.NETはPythonやRを使用できない場合（たとえば、.NETで実行されるCAD APIを操作する場合）に武器を補完する便利なツールのように見えます。
