Angular：バックエンドでコードを読みやすくします。おまけ：APIのスワップとクエリのキャッシュ

多くの場合、プロジェクトでは、フロントエンドの開発ペースがバックエンドの開発ペースよりも進んでいます。この状況では、2つのことが発生します。

  1. バックエンドなしで、または個別のエンドポイントなしでフロントを起動する機能。
  2. 必要なエンドポイント、要求の形式、応答などをバックエンドに説明します。

APIリクエストを担当するコードを整理する方法を共有したいと思います。これにより、これら2つの問題が完全に解決され、リクエストをキャッシュすることもできます。

プロジェクトルートにapi.config.json構成ファイルを作成します。

{
  "apiUrl": "https://api.example.com",
  "useFakeApiByDefault": false,
  "fakeEndPoints": ["Sample"]
}

ここでは、APIのベースURLを設定します。useFakeApiByDefault= trueパラメータの場合、アプリケーションはすべてのリクエストではなくスタブのみを使用します。falseの場合、スタブはfakeEndPoints配列からのリクエストにのみ使用されます。

JSONをコードにインポートするために、tsconfig.jsonファイルのCompilerOptionsセクションに次の2行を追加します。

    "resolveJsonModule": true,
    "esModuleInterop": true,

基本クラスBaseEndpointを作成します。

/src/app/api/_base.endpoint.ts：

import {ApiMethod, ApiService} from '../services/api/api.service';
import * as ApiConfig from '../../../api.config.json';
import {ErrorService} from '../services/error/error.service';
import {NgModule} from '@angular/core';

@NgModule({providers: [ApiService, ErrorService]})

export abstract class BaseEndpoint<Request, ResponseModel> {
  protected url: string;
  protected name: string;
  protected method: ApiMethod = ApiMethod.Post;
  protected sampleResponse: ResponseModel;
  protected cache: boolean = false;
  protected responseCache: ResponseModel = null;

  constructor(private api: ApiService, private error: ErrorService) {
  }

  public execute(request: Request): Promise<ResponseModel> {
    if (this.cache && this.responseCache !== null) {
      return new Promise<ResponseModel>((resolve, reject) => {
        resolve(this.responseCache);
      });
    } else {
      if (ApiConfig.useFakeApiByDefault || ApiConfig.fakeEndPoints.includes(this.name)) {
        console.log('Fake Api Request:: ', this.name, request);
        console.log('Fake Api Response:: ', this.sampleResponse);
        return new Promise<ResponseModel>((resolve) => resolve(this.sampleResponse));
      } else {
        return new Promise<ResponseModel>((resolve, reject) => {
          this.api.execute(this.url, this.method, request).subscribe(
            (response: Response<ResponseModel>) => {
              if (response.status === 200) {
                if (this.cache) { this.responseCache = response.payload; }
                resolve(this.payloadMap(response.payload));
              } else {
                this.error.emit(response.error);
                reject(response.error);
              }
            }, response => {
              this.error.emit('    ')
              reject('    '));
           }
        });
      }
    }
  }

  protected payloadMap(payload: ResponseModel): ResponseModel { return payload; }
}

abstract class Response<T> {
  public status: number;
  public error: string;
  public payload: T;
}

<Request、ResponseModel>-リクエストとレスポンスのインスタンスのタイプ。それに応じて、ペイロードはすでに引き出されています。

ApiServiceクラスとErrorServiceクラスは投稿しません。投稿が膨らまないようにするため、特別なものはありません。 ApiServiceはhttpリクエストを送信し、ErrorServiceを使用してエラーをサブスクライブできます。エラーを表示するには、実際にエラーを表示するコンポーネントでErrorServiceにサブスクライブする必要があります（モーダルまたはツールチップを作成するメインレイアウトに署名します）。

このクラスから継承すると魔法が始まります。

派生クラスは次のようになります。/src/app/api/get-users.endpoint.ts

import {BaseEndpoint} from './_base.endpoint';
import {Injectable} from '@angular/core';
import {UserModel} from '../models/user.model';

@Injectable()
export class GetUsersEndpoint extends BaseEndpoint<GetUsersRequest, UserModel[]> {
  protected name = 'GetUsers';
  protected url= '/getUsers';
  protected sampleResponse = [{
      id: 0,
      name: 'Ivan',
      age: 18
  },
  {
      id: 1,
      name: 'Igor',
      age: 25
   }

  protected payloadMap(payload) {
    return payload.map(user => new UserModel(user));
  }
}

export class GetUsersRequest {
   page: number;
   limit: number;
}

さらに、ファイルの内容から、どのURL、どの要求形式（GetUserRequest）、どの応答形式かがすぐにわかります。

そのようなファイルが別のフォルダー/ apiにある場合、各ファイルは独自のエンドポイントに対応します。このフォルダーをバックエンドに表示できます。理論的には、apiに関するドキュメントを書くのが面倒な場合は、それなしで行うことができます。コントローラーに応じて、フォルダー/ api内のファイルをフォルダーに分散させることもできます。

「fakeEndPoints」設定配列に「GetUsers」を追加すると、リクエストはなくなり、応答はsampleResponseからのスタブデータに置き換えられます。

偽のリクエストをデバッグするために（もちろん、[ネットワーク]タブには何も表示されません）、コンソールの出力を2行出力してコンソールクラスに出力しました。

console.log('Fake Api Request:: ', this.name, request);
console.log('Fake Api Response:: ', this.sampleResponse);

クラスプロパティcache = trueをオーバーライドすると、リクエストがキャッシュされます（最初にAPIへのリクエストが行われたときに、最初のリクエストの結果が常に返されます）。ここでの真実はファイナライズすることです。キャッシュを機能させる必要があるのは、要求パラメーター（UserRequestクラスのインスタンスのコンテンツ）の場合のみです。

サーバーから受信したデータの変換が必要な場合は、payloadMapメソッドを再定義します。メソッドが再定義されない場合、ペイロードからのデータは返されたPromiseにあります。

次に、コンポーネントのAPIからデータを取得します。

import {Component, OnInit, ViewChild} from '@angular/core';
import {UserModel} from '../../models/user.model';
import {GetUsersEndpoint, GetUsersRequest} from '../../api/get_users.endpoint';

@Component({
  selector: 'app-users',
  templateUrl: './users.component.html',
  styleUrls: ['./users.component.css'],
  providers: [GetUsersEndpoint]
})
export class UsersComponent implements OnInit {

  public users: UserModel[];
  public request: GetUsersRequest = {page: 1, limit: 20};

  constructor(private getUsersEndpoint: GetUsersEndpoint) {    
  }

  private load() {
    this.getUsersEndpoint.execute(this.request).then(data => {
      this.users = data;
    });
  }

  ngOnInit(): void {
    this.load();
  }
}

そのような実装では、バックウィッシュがまだ完了していない場合でも、ウィッシュリストの完了結果を顧客に示すことができます。必要なエンドポイントをスタブに配置してください。そうすれば、すでに機能を「操作」してフィードバックを得ることができます。

コメントでは、機能を構築するときにどのような問題が発生する可能性があるか、他の落とし穴が生じる可能性があるという建設的な批評を楽しみにしています。

将来的には、このソリューションをpromiseからasync / awaitに書き換えたいと思います。コードはさらにエレガントになると思います。

在庫がありますが、Angularにはさらにいくつかのアーキテクチャソリューションがあります。近い将来、共有する予定です。
