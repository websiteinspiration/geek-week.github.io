编译器使用多少个处理器指令？

一个月前，我试图计算现代处理器支持多少种不同的指令，并在冰湖计算了945个。评论员提出了一个有趣的问题：所有这些多样性的哪一部分真正被编译器使用？例如，某位Pepijn de Vos在2016年计算了他的/ usr / bin中的二进制文件涉及多少个不同的指令，并计算为411-即当时存在的所有x86_64指令中，大约有三分之一未在其OS中的任何标准程序中使用。他的另一个奇怪发现是x86_64的代码包含第三条指令mov。 （一般地，这是已知的是单独的指令是mov足够写任何程序。）

我决定使用LLVM / Clang编译器作为“参考代码”来进行de Vos研究。与未命名操作系统的未命名版本的/ usr / bin的内容相比，它具有几个优点：

  1. 使用起来很方便：这是一个巨大的二进制文件，大小可与普通Linux的/ usr / bin的所有内容相提并论；
  2. 它使您可以比较不同的ISA：x86，ARM，SPARC，MIPS和PowerPC的正式版本可在releases.llvm.org/download.html上获得；
  3. 它使您可以跟踪历史趋势：自2003年以来，所有发行版均提供官方二进制文件；
  4. 最后，在研究编译器时，将编译器用作实验对象是合​​理的:-)

这是根据指令数量的分布：


出乎意料的是，SPARC的11％代码由nops填充分支延迟时隙组成。对于i386，在最常见的指令中，我们看到这既int3填补了函数之间的空白，又nop使循环与缓存行对齐。mov在两个x86变体上都证实了de Vos观察到的代码是三分之一的组成。但是即使在加载存储体系结构上mov，如果不是最频繁的指令，那么结果也会是加载后的第二条指令。

以及所使用的指令集如何随时间变化？

每个版本具有官方二进制文件的唯一ISA是i386：


右轴上的灰线是编译器中使用的不同指令的数量。如我们所见，前一段时间，编译器的编译方式更加多样化。int3它仅在2018年被用来填补空白;以前使用的功能与nop函数内的对齐方式相同。在这里，您可以看到自2013年以来一直在使用函数内部对齐；在那之前nop更少了。有趣的是，直到2016年，它们mov几乎占编译器的一半。

在clang出现之前，LLVM的第一个版本也随SPARC的二进制文件一起发布。然后，对SPARC的支持就失去了意义，仅在14年后才出现-数量增加了一个数量级nop：


从历史上看，发布LLVM二进制文件的下一个ISA是PowerPC：首先用于Mac OS X，然后在中断十年后，用于RHEL。正如你可以从图中看到，这个突破到ISA的64位版本后的过渡伴随着更换了广大lwz用ld，并且除了各种指令加倍：


在二进制文件64位x86和ARM，使用不同的指令的频率几乎没有变化：




当计数ARM指令，我切断状态的后缀 -在一起使用它们，可以获得大约一千种不同的指令，但是即使没有它们，ARM在生成的各种指令中也远远领先于其他ISA。因此层b此图上也包括所有条件分支。对于ISA的其余部分，其中只有条件转换和其他一些指令才是有条件的，因此在计算过程中不会截断条件的后缀。
