マイクロコントローラーでデスクトップソフトウェアを起動する


私のレポートは、デスクトップソフトウェア（Qt、OpenCV、pjsip）の起動に関するいくつかの作業の結果をまとめたもののようなものでした。このレポートは一般市民を対象としています。habrの場合は、多くの詳細を減らすようにし、必要な場合はリンクまたは検索用のキーワードを指定します。

まず、アジェンダのようなものを紹介します。実際には、3つの部分があります。最初に、デスクトップソフトウェアをマイクロコントローラーに転送する難しさを説明し、その過程でマイクロコントローラーとマイクロプロセッサーの違いを説明します。主な質問nafig山羊ボタンアコーディオンにも答えます、なぜマイクロコントローラでデスクトップソフトウェアを実行するのか、そして最後に、この「デスクトップ」ソフトウェアをマイクロコントローラに転送するコストを大幅に削減する方法を説明します。

マイクロコントローラーとマイクロプロセッサーの違いは何ですか


したがって、クエストに関する質問の1つが聞こえ、おそらくこれの影響下で、典型的なプログラマーに、それらの違いを説明する記事が表示されました。私は記事のすべてに同意しますが、私たちは私たち自身の言葉で説明し、他のいくつかのアクセントを付けようとします。

マイクロコントローラー（マイクロコントローラーユニット、MCU）とは？Wikipediaを見ると、マイクロコントローラーはチップ上のシステムであることがわかります。つまり、プロセッサーと周辺機器は1つのチップに配置されています。

システムオンチップ（System-on-a-Chip、SoC）という用語マイクロコントローラよりも広い概念です。これは、マイクロプロセッサ（CPU）と周辺機器を含む1つのチップに統合された電子回路です。どの携帯電話にも、いくつかのプロセッサコアに加えて周辺機器がたくさんあるチップがあります。x86プロセッサでさえ、今やブリッジやgpuの機能さえ含んでいます。しかし、明確な分類ではなく、特徴に焦点を当てたいと思います。誰もがそのチップ上のシステムを知っています：
  • 少ない消費
  • 安いです
  • 生産性が低い

接点の数を減らすことで信頼性が向上し、温度範囲が広がることを誰もが知っているわけではありません。

マイクロコントローラーはさらに優れた統合性を備えており、周辺機器とも呼ばれるメモリー（RAMおよびROM）も備えています。したがって、特性は維持され、消費はさらに少なく、コストはさらに低く、生産性は低く、信頼性はさらに高くなります。

「生産性の低下」についてもっとお話ししたいと思います。実際のところ、それは事実であり、現代の汎用CPUまたはSoCを使用して、いわゆるDMIPSで大幅に​​優れたパフォーマンスを実現します。。しかし、なぜそのようなパフォーマンスを考えたら？例として、QuakeとDoomのゲーム（John Romeroがフェスティバルに参加したクリエイター）を紹介します。彼らはPentium MMXでうまく機能しました。しかし、このCPUベースのシステムのパフォーマンスは、トップエンドのマイクロコントローラーによってすでにブロックされています。私は意味STM32F7、あるいはSTM32H7を。つまり、これらのMCUに基づいて、DoomおよびQuakeレベルのアプリケーションを実行することは非常に可能です。

MCUの別の機能である内部メモリについて話しましょう。私が言ったように、彼女はチップ上に直接配置されているため、非常に高速です。しかし、もちろん、彼女は十分ではありません。しかし、どれだけですか？ STM32F769マイクロコントローラーには、2 mbフラッシュ（ROM）と512 + 16 + 4 kB SRAM（RAM）が含まれています。さらに、たとえば、STM32F769i-discoボード上また、直接アドレス指定される外部メモリ（128MビットSDRAMおよび512MビットQuad-SPIフラッシュメモリ）もあります。私は、SDカードやUSBを接続する機能についてさえ話していません。さて、「640 kbで十分なはずです」という有名なフレーズを思い出すと、合理的な疑問が生じます。なぜ現代のデスクトップソフトウェアはそんなに大食いなのでしょうか。たとえば、Windows 10のアイドルモードの電卓は、16 MBものメモリを消費します。

MCUの次の機能であるROMの存在に移りましょう。実際、CPUを備えたシステムにはROMも含まれています。ROMから実行が開始され、BIOSが起動しますが、残りのソフトウェアはRAMにロードされ、そこから実行されます。組み込みシステムでは、eXecute-In-Place（XIP）と呼ばれるコードを直接実行するのが通例ですが、Linuxはこのモードもサポートしていますが、主なものはすべてをメインメモリにロードして実行することです。 ROMから直接実行できることは2つの理由で重要です。1つはハーバードアーキテクチャで、コマンドバスとデータバスを分離することでプログラムの実行速度を上げることができます。 2番目-少なくともコードセグメントを他の場所にコピーする必要がないため、当然、無駄なメモリが少なくなります。

MCUのもう1つの重要な機能は、次のようなモジュールがないことです。MMU、つまりマイクロコントローラには、仮想メモリを管理するためのハードウェアサポートがありません。MCUでは、すべてにMPUがあるわけではなく、コードは同じアドレス空間で実行されるため、完全なフォーク（）を構成することは困難です。ただし、記事「fork（）vs. vfork（）はかなり難しいシステムコールであり、機能の点では、多くの場合、vfork（）またはposix_spawn（）で置き換えることができます。

要約すると、マイクロコントローラーとマイクロプロセッサーの違いは何ですか？
ハードウェアの観点から：
  • マイクロコントローラには、メモリを含むすべての必要な周辺機器があります
  • マイクロコントローラーのメモリは大幅に少ない
  • マイクロコントローラーにはハードウェアMMUがありません
  • マイクロコントローラはクロック速度が低く、プロセッサアーキテクチャ（ギフトアーキテクチャ）が異なる場合があります。


消費者特性に関して：
  • マイクロコントローラーはより少ないエネルギーを消費し、バッテリーで動作できます
  • マイクロコントローラーは安価です
  • マイクロコントローラーはより信頼性が高い
  • マイクロコントローラーは生産性が低い


マイクロコントローラーでデスクトップソフトウェアを実行する理由


そのため、MCUは生産性が低く、もともと制御タスク用に意図されていました（意図されています）。では、なぜ別のクラスのシステム用に設計されたソフトウェアを実行するのでしょうか。まあ、それらの力はそのようなタスクのためにすでに十分であるとしても、これらのタスクはマイクロコントローラーの伝統的な方法で解決することができます。つまり、さまざまな小さなRTOSを使用したり、コードを最初から作成したりする、いわゆるベアメタルです。明らかな答えがあります。最初は管理タスクは非常に単純でしたが、今では機能の要件が急速に高まっています。彼らはインターネットを介して制御を期待するいくつかの種の電球からでさえ、私はやかんについて話していません、彼らはおそらく所有者の気分に応じてお茶の組成を選択し、環境状況を改善するために最適な量の水を最適な温度に加熱するでしょう。 :)

したがって、マイクロコントローラですでに開発されたソフトウェアを使用する試みが絶えず行われていることは当然のことです。たとえば、小規模なプラットフォームでLinuxを実行する最初かつ広く知られている試みの1つはucLinuxですが、現在は主にLinuxのNOMMUモードになっています。MMUのないハードウェアプラットフォームでLinuxを実行するという考えでしたが、これがCPUとMCUの主な違いの1つであることがわかりました。

簡単に言えば、既製のソフトウェアを使用する利点は次のとおりです。
  • 開発コストを削減
  • 市場投入までの時間の短縮
  • コード内のエラーの削減。既製のソフトウェアはより信頼性があります。


Emboxでの方法


ここでは理解できますが、長い間話すことができますが、再度、論文の要点をわかりやすく説明します。

最初の1つ。 Emboxは、モジュールとシステムを記述するために独自の言語でMybuildビルドシステムを使用します。これについては、記事「Mybuild-モジュラーアプリケーション用のビルドシステム」で書きました。このアセンブリシステムでは、画像に余分なものを含めないか、必要なものだけを含めることができます。システムの特性を非常に細かく設定できます。たとえば、STM32F7でPJSIPを実行するには、システムの構成ファイルに行を書き込むだけで、イーサネットパケットの数を16に制限しました。
include embox.net.skbuff(amount_skb=16)

Mybuildは依存関係を分析し、ヘッダーファイル、リンカースクリプト、Makefileなどのさまざまなアーティファクトを生成します。これにより、必要なメモリの量を大幅に削減し、まったく使用されていない部分を含めることがなくなります。

二つ目。当然、EmboxはPOSIXをサポートする必要があり、サポートされています。さらに、システムの構成に応じて再構築する必要があるため、独自の標準ライブラリを実装しました。

第3。静的リンク。単一のアドレス空間の問題を回避し、画像サイズを縮小して必要な部分のみを含めるために、静的リンクを使用します。さまざまな段階でシステムのさまざまな部分が組み立てられますが、最後にすべて（カーネル、すべてのサブシステム、ライブラリ、およびアプリケーション）が1つのイメージにリンクされます。画像内のすべてのキャラクターに関するすべての情報が含まれています。一見すると、標準のint main（..）エントリポイントを持つ多くのアプリケーションがありますが、結果のイメージでは、これらはコマンドラインからシステムで使用できる異なる文字になります。

第4。多くの一般的なアプリケーションとライブラリは利点を利用しているため、C ++のランタイムを実装する必要がありました。

第五。ビルドシステムを使用する機能を追加しました
 ./configure; make; make install

どこかからソースをダウンロードし、必要なパッチを適用した後。

結論


今日、以前はマイクロコントローラーで作成されたシステムの機能要件が大幅に増大しています。制御システムは、ユニバーサルシステムとの互換性が期待されます。マイクロコントローラの特性も大幅に向上しているため、対応する機能を備えたマイクロコントローラ上でソフトウェアを実行できます。歴史的に、最適化を目的としたマイクロコントローラー用のソフトウェアは特定のタスク用に作成されましたが、機能要件の増加に伴い、ソフトウェアの信頼性を大幅に向上させることができるユニバーサルシステムの世界からのアプローチ、つまりモジュールの再利用、分散開発を適用する必要があります。組み込みソフトウェアの機能は考慮されていませんが、非常に優れた機能性を備えたユニバーサルソフトウェア。したがって、マイクロコントローラーでデスクトップソフトウェアを使用すると利益が得られます。結局のところ、一方では、より安価で信頼性が高く、消費電力の少ないハードウェアプラットフォームを使用できます。その一方で、そのようなシステムの開発コストは大幅に削減されます。
