Pythonメモリ管理：メモリの断片化について少し

この記事に関するいくつかの考え。

先日、64ビットUbuntuの Python3のCPythonでPythonメモリ管理がどのように機能するかに興味を持ちました。

理論のビット


glibcシステムライブラリにはmallocアロケーターがあります。各プロセスには、ヒープと呼ばれるメモリ領域があります。 malloc関数を呼び出してメモリを動的に割り当てることにより、このプロセスのヒープからチャンクを取得します。要求されたメモリのサイズが小さい場合（128KB以下）、メモリは空きチャンクのリストから取得できます。これが不可能な場合、メモリはmmap （sbrk、brk）システムコールを使用して割り当てられます。 mmapシステムコールは、仮想メモリを物理メモリにマップします。メモリは4KBページで表示されます。大きなチャンク（128KB以上）は、常にmmapシステムコールを通じて割り当てられます。メモリを解放するときに、フリーズされていない小さなチャンクがフリーズされていないメモリの領域に接している場合、メモリの一部がオペレーティングシステムに戻ることがあります。大きなチャンクはすぐにオペレーティングシステムに戻ります。

Cのアロケータによる講義 から取得した情報。

CPythonには、「プライベートヒープ」用の独自のアロケーター（PyMalloc）と、最初の「上」で機能するオブジェクトのタイプごとのアロケーターがあります。 PyMallocは、アレナと呼ばれるオペレーティングシステムライブラリのmallocを介して256KBのメモリチャンクをリクエストします。次に、4KBごとにプールに分割されます。各プールは固定サイズのチャンクに分割され、各プールは64サイズのチャンクに分割できます。

各タイプのアロケータは、すでに割り当てられているチャンクがあればそれを使用します。ない場合、PyMallocは最初のアリーナから新しいプールを発行します。そこには、新しいプールの場所があります（アリーナは占有率の降順で「ソート」されています）。これが機能しない場合、PyMallocはOSに新しいArenaを要求します。要求されたメモリサイズが512Bを超える場合を除き、メモリはシステムライブラリからmallocを介して直接割り当てられます。

オブジェクトを削除すると、メモリはOSに返されませんが、チャンクは対応するプールに、プールはアリーナに戻ります。すべてのチャンクが解放されると、アリーナはオペレーティングシステムに戻ります。結局のところ、比較的少数のチャンクがアリーナで使用される場合、すべて同じようにアリーナのすべてのメモリがPVMによって使用されます。ただし、128KBを超えるチャンクはmmapを介して割り当てられるため、無料のArenaはすぐにオペレーティングシステムに戻ります。

私は2つの点に焦点を当てたいと思います：

  1. PyMallocは、新しいArenaを作成するときに256KBの物理メモリを割り当てることがわかりました。
  2. 無料のアリーナのみがオペレーティングシステムに返されます。

例


次の例を考えてみます。

iterations = 2000000

l = []
for i in range(iterations):
    l.append(None)
        
for i in range(iterations):
    l[i] = {}

s = []  # [1]
# s = l[::2]  # [2]
# s = l[2000000 // 2::]  # [3]
# s = l[::100]  # [4]

for i in range(iterations):
    l[i] = None        
        
for i in range(iterations):
    l[i] = {}

この例では、200万個の要素のリストlが作成され、そのすべてが1つのNoneオブジェクトを指しています。次のサイクルでは、要素ごとにオブジェクトが作成されます-空の辞書です。次に、2番目のリストsが作成されます。その要素は、最初のリストのいくつかの要素によって参照されるいくつかのオブジェクトを指します。次のクロールの後、リストlの要素は再びNoneオブジェクトをポイントし始めます。そして最後のサイクルでは、最初のリストから各要素に対して辞書が再び作成されます。

Sリストオプション：

  1. s = []
  2. s = l[::2]
  3. s = l[200000 // 2::]
  4. s = l[::100]

それぞれの場合のメモリ消費量に関心があります。

PyMallocロギングを有効にしてこのスクリプトを実行します。

export PYTHONMALLOCSTATS="True" && python3 source.py 2>result.txt

結果の説明


写真では、それぞれの場合のメモリ消費量を確認できます。横軸には、そのような消費が行われた時間と値の相関関係はありません。ログの各値がそのシリアル番号に関連付けられているだけです。



「要素なし」


最初のケースでは、リストsは空です。2番目のサイクルでオブジェクトを作成した後、約500MBのメモリが消費されます。そして、これらのオブジェクトはすべて3番目のサイクルで削除され、使用されたメモリはオペレーティングシステムに返されます。最後のサイクルで、オブジェクトのメモリが再度割り当てられ、同じ500MBが消費されます。

"毎秒"


リストlの2番目の要素ごとにリストを作成する場合、メモリがオペレーティングシステムに返されないことがわかります。つまり、この場合、約250MBの辞書が削除されますが、各プールには削除されていないチャンクが存在するため、対応するアリーナが解放されません。ただし、2回目の辞書作成時には、これらのプールの空きチャンクが再利用されるため、約250MBの新しいメモリのみが割り当てられます。

"後半"


リストlの要素の後半からリストを作成する場合、前半は別々のアリーナにあります。これにより、約250MBのメモリがオペレーティングシステムに返されます。その後、約500MBが新しい辞書に再割り当てされます。これが、750MBの領域での合計消費量の理由です。

この場合、2番目とは異なり、メモリは部分的にオペレーティングシステムに返されます。これは、一方で、他のプロセスがこのメモリを使用できるようにする一方で、システムメモリを解放して再割り当てするためのシステムコールを必要とします。

「100分の1」


最後のケースが最も興味深いようです。そこで、最初のリストの100番目ごとの要素から2番目のリストを作成します。これには約5MBが必要です。しかし、各アリーナには一定数の占有されたチャンクが残っているため、このメモリは解放されず、消費量は500MBのレベルのままです。2回目に辞書を作成するとき、新しいメモリはほとんど割り当てられず、最初に割り当てられたチャンクが再利用されます。

この状況では、メモリの断片化により、必要以上に100倍使用します。ただし、このメモリが繰り返し必要になる場合は、システムコールを行って割り当てる必要はありません。

合計


多くのアロケータを使用すると、メモリの断片化が発生する可能性があることに注意してください。それでも、検索ツリーなどのツリー構造を持つデータ構造など、一部のデータ構造を慎重に使用する必要があります。追加と削除の任意の操作が上記の状況につながる可能性があるため、メモリ消費の観点からこれらの構造を使用することの妥当性は疑わしいためです。

def parse_result(filename):
    ms = []
    with open(filename, "r") as f:
        for line in f:
            if line.startswith("Total"):
                m = float(line.split()[-1].replace(",", "")) / 1024 / 1024
                ms.append(m)
    return ms
