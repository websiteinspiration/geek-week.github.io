10分で現在のコレクション


Robert V. Ruggieroによる写真

トピックは新しいものではありません。しかし、「並行コレクションとは何か、いつ使用するのか」という質問をします。インタビューやコードレビューでは、ほとんどの場合、「競合状態から私たちを完全に保護してくれます」という1つの文からなる答えを受け取ります（これは理論的にも不可能です）。または：「通常のコレクションのようですが、内部はすべてロックされています」。これも現実とは完全には一致しません。

この記事の目的は、10分でトピックを理解することです。これは、いくつかの微妙なことをすぐに理解するのに役立ちます。または、インタビューの前に思い出をリフレッシュします。

まず、System.Collections.Concurrent名前空間の内容を簡単に見ていきます。次に、コンカレントコレクションとクラシックコレクションの主な違いについて説明します。いくつかの明らかでない点に注意してください。結論として、起こりうる落とし穴と、どのタイプのコレクションを使用する価値があるかについて説明します。

コンカレントコレクションとクラシックコレクションの違い


内部状態の保護


クラシックコレクションは最大のパフォーマンスを考慮して設計されているため、それらのインスタンスメソッドはスレッドの安全性を保証しません。

たとえば、Dictionary.Addメソッドのソースコードを見てみましょう。
次の行が表示されます（コードは読みやすくするために簡略化されています）。

if (this._buckets == null)
{
    int prime = HashHelpers.GetPrime(capacity);
    this._buckets = new int[prime];
    this._entries = new Dictionary<TKey, TValue>.Entry[prime];
}

ご覧のとおり、辞書の内部状態は保護されていません。複数のスレッドからアイテムを追加する場合、次のシナリオが考えられます。

  1. スレッド1がAddを呼び出し、if条件に入った直後に実行が停止した
  2. スレッド2がAddを呼び出し、コレクションを初期化し、アイテムを追加しました
  3. ストリーム1は動作に戻り、コレクションを再初期化して、ストリーム2によって追加されたデータを破棄しました。

つまり、クラシックコレクションは、複数のストリームからのレコーディングには適していません。

APIは、コレクションの現在の状態を許容します。


ご存知のように、重複するキーをDictionaryに追加することはできません。同じキーでAddを 2回呼び出すと、2回目の呼び出しでArgumentExceptionがスローされます。

この保護は、シングルスレッドのシナリオで役立ちます。しかし、マルチスレッドでは、コレクションの現在の状態を確認できません。当然、次のようなチェックは、常にロック状態にある場合にのみ私たちを救います。

if (!dictionary.ContainsKey(key))
{
    dictionary.Add(key, “Hello”);
}

例外ベースのAPIは不適切なオプションであり、マルチスレッドシナリオでの安定した予測可能な動作を許可しません。代わりに、コレクションの現在の状態を想定せず、例外をスローせず、特定の状態の有効性に関する決定を呼び出し元に任せるAPIが必要です。

並行コレクションでは、APIはTryXXXパターンに基づいて構築されます。通常のAdd、Get、およびRemoveの代わりに、TryAdd、TryGetValueおよびTryRemoveメソッドを使用します。そして、これらのメソッドがfalseを返す場合、これが例外的な状況であるかどうかを判断します。

クラシックコレクションにも状態許容メソッドが含まれるようになったことは注目に値します。しかし、クラシックコレクションでは、そのようなAPIはすばらしい追加であり、並行コレクションでは必須です。

競合状態を最小限に抑えるAPI


最も単純な要素の更新操作を考えます。

dictionary[key] += 1;

コードは非常に単純ですが、3つのアクションを実行します。コレクションから値を取得し、1を追加して、新しい値を書き込みます。マルチスレッド実行では、コードが値を取得してインクリメントを実行し、インクリメントの実行中に別のスレッドによって書き込まれた値を安全に消去する可能性があります。

このような問題を解決するために、並行コレクションAPIには多数のヘルパーメソッドが含まれています。たとえば、TryUpdateメソッドは、キー、新しい値、予想される現在の値の3つのパラメーターを取ります。コレクションの値が予期されたものと一致しない場合、更新は実行されず、メソッドはfalseを返します。

別の例を考えてみましょう。文字通り、次のコードのすべての行（Console.WriteLineを含む））マルチスレッド実行で問題が発生する可能性があります。

if (dictionary.ContainsKey(key))
{
    dictionary[key] += 1;
}
else
{
    dictionary.Add(key, 1);
}
Console.WriteLine(dictionary[key]);

値を追加または更新し、その結果を使用して操作を実行することは、かなり典型的なタスクです。したがって、コンカレントディクショナリにはAddOrUpdateメソッドがあり、1回の呼び出しで一連のアクションを実行し、スレッドセーフです。

var result = dictionary.AddOrUpdate(key, 1, (itemKey, itemValue) => itemValue + 1);
Console.WriteLine(result);

知っておくべきことは1つあります。

実装AddOrUpdateのメソッド呼び出しTryUpdateの方法は、上記とコレクションから現在の値を渡します。更新が失敗した場合（隣接するスレッドが既に値を変更している場合）、試行が繰り返され、送信された更新デリゲートが更新された現在の値で再度呼び出されます。つまり、更新デリゲートは複数回呼び出すことができるため、副作用が含まれていてはなりません。

並行コレクションが提供しないもの


上記の例から明らかなように、並行コレクションは競合状態に対する完全な保護を提供しないため、それに応じてコードを設計する必要があります。しかし、それだけではありません。知っておくべきポイントがいくつかあります。

クラシックコレクションの多態性


並行コレクションは、クラシックと同様に、インターフェイスIDictionary、ICollection、およびIEnumerableを実装します。ただし、これらのインターフェイスのAPIの一部は、定義上、スレッドセーフにすることはできません。たとえば、上記で説明したAddメソッドです。

並行コレクションは、スレッドセーフなしでこのようなコントラクトを実装します。安全でないAPIを「隠す」ために、インターフェースの明示的な実装を使用します。これは、ICollectionなど、入力を受け取るメソッドに同時コレクションを渡すときに覚えておく価値があります。

また、並行コレクションは、古典的なコレクションに関するLiskov置換の原則に準拠していません。

例えば、古典的なコレクションの内容は中に変更することはできません反復 ;次のコードは、スローされますと、InvalidOperationExceptionをするためにリストクラス：

foreach (var element in list)
{
    list.Remove(element);
}

並行コレクションについて話す場合、列挙時の変更は例外につながりません。そのため、異なるストリームから同時に読み取りと書き込みを実行できます。

さらに、並行コレクションは、列挙中の変更の可能性を異なる方法で実装します。ConcurrentDictionaryは単にチェックを実行せず、反復の結果を保証せず、ConcurrentStack / Queue / Bagはロックして現在の状態のコピーを作成し、反復します。

考えられるパフォーマンスの問題


前述のとおり、ConcurrentBagは隣接するスレッドから要素を「盗む」ことができます。これにより、異なるスレッドからConcurrentBagに読み書きすると、パフォーマンスの問題が発生する可能性があります。

また、並行コレクションは、コレクション全体の状態（Count、IsEmpty、GetEnumerator、ToArrayなど）を照会するときに完全ロックを課すため、従来の同等のものよりも大幅に遅くなります。

結論：この選択は「無料」ではないため、並行コレクションを使用することは、それらが本当に必要な場合にのみ価値があります。
