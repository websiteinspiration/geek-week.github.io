SSR（サーバー側レンダリング）を使用したNuxtサーバー側メモリリーク

こんにちは、ハブル！この記事は、Vue SSR、特にNuxtを使用するすべての人にとって必読の記事です。これはaxios使用時のメモリリークに関するものです。

バックグラウンド


半年前、私はVueJS + Nuxtスタックを使用したプロジェクトに乗りました。その特異性は、Nodサーバー（Nuxt）が常に製品で停止し、新しいサーバーが代わりに使用されていたことです。グラフとログから、ノードプロセスオペレータが100％に達し、メモリ不足エラーで落ちたことは明らかでした。このとき、新しいプロセスが終了したプロセスの場所まで上昇しました。これには約30秒かかりました。これは、ユーザーが502エラーを受け取るのに十分な時間でした。明らかに、コードのどこかに、見つける必要のあるメモリリークがありました。

この記事の一部だけを読んでも、すべての質問に答えられるとは限らないため、すぐに重要なポイントを強調したいと思います。

1. Axiosインターセプター


私たちは問題を探して私の苦しみをすばやく調べ、axios.interceptorsがすべての責任を負うことになっているという事実にすぐに進みます（申し訳ありません、Habr、責任者を探してくれました）。

axiosが次のように作成されたことをすぐに予約します。

import baseAxios from 'axios';

const axios = baseAxios.create({
  timeout: 10000,
});


export default axios;

そして、次のようにアプリケーションコンテキストにアタッチされます。 

import axios from './index';

export default function(context) {

  if(!context.axios) {
    context.axios = axios;
  }
}

  • リークを長時間検索した後、すべてのaxios.interceptorsを無効にすると、メモリのクリーンアップが始まります。
  • 何か問題でもありますか？
  • インターセプターは、すべての応答または要求をインターセプトし、すべての要求に対してグローバルに要求を送信する前に、応答を含むコードを実行したり（たとえば、エラーを処理するため）何かを追加したり、1か所で便利なプロキシです。これがどのように見えるかの例です（ファイル 'plugins / axios / interceptor.js'）

export default function({ axios }) {

  const interceptor = axios.interceptors.response.use( (response) => {
    return response;
  }, function (error) {
    //-   ,  
    return Promise.reject(error);
  });

}

そして、ここから楽しみが始まります。nuxt.config.jsにプラグイン経由でインターセプターを追加する機能を追加します

  plugins: [
    { src: '~/plugins/axios/bindContext' },
    { src: '~/plugins/axios/interceptor' },
  ]

新しいリクエストごとにnuxtが自動的にすべてのプラグイン機能を実行してから、nuxtServerInitを実行すると、すべてが通常どおり行われます。つまり、最初のユーザーの場合、サーバー側のインターセプターをasyncDataまたはフェッチのコンポーネントのどこかにリクエストを作成し、インターセプターが正常に機能すると、2番目のユーザーが2番目のインターセプターを作成し、関数内のコードが2回機能します。

私の言葉をよりよく理解するために、関数が呼び出されるたびに増分するカウンターを描画し、インデックスを5回ノックします



。15 回の呼び出しが発生していることがわかります。これは1 + 2 + 3 + 4 + 5であり、次のインターセプターを作成するためにさらに時間をかけて確認しました以前に作成されたものの課題があること。

学校から、算数の進行の式をよく覚えています。1からnまでの合計は、n *（n + 1）/ 2と書くことができます。1000番目のユーザーが入ると、関数が1000回呼び出され、これはすでに50万回の呼び出しであるため、負荷が中または高の場合でも、サーバーがクラッシュしても驚かないでください。

問題の解決策


UPD。解決策＃0-コメントは、この問題の適切な解決策を説明しています。

解決策1-axios.interceptorsを使用しないでください。

ソリューションNo. 2-すべてが非常に簡単です。axiosのドキュメントに従って、インターセプターを自分でクリーニングする必要があります。

export default function({ axios }) {

  const interceptor = axios.interceptors.response.use( (response) => {
    
    if(process.server) {
      axios.interceptors.response.eject(interceptor);
    }
    
    return response;
  }, function (error) {
    if(process.server) {
      axios.interceptors.response.eject(interceptor);
    }
    
    return Promise.reject(error);
  });

}

これはサーバー側でのみ実行する必要があります。クライアント側では、最初のリクエストが正常に実行された後、このインターセプターは実行を停止するためです。まだサーバー上にいて次のユーザーのリクエストを処理している間、いくつかのリクエストがありますが、このインターセプターの排出により、最初のリクエストを除くすべてのリクエストが通過しないという事実には、もう1つのニュアンスがあります。イジェクトを実行する必要がある瞬間を独立して考えるには、これを行う最も簡単な方法は、たとえば10秒後にsetTimeoutを使用することです。サーバー側では、現在のユーザーのすべてのリクエストを完了し、この時間中にすべてのリクエストが実行されると想定できます。インターセプターはまだアクティブです。

runInNewContext


これは非常に面白いオプションです。そのため、このバグはローカルで再生できませんが、ビルドで非常に簡単に再生できます。それについてはこちらをお読みください。この記事を書く準備をしているときに、この問題を再現するためにスターターテンプレートのnuxtプロジェクトを作成しました。また、通常のユーザーごとに、インターセプターがnではなく1回実行されたことに驚きました。問題は、npm run devを作成するとき-このオプションはデフォルトでtrueであり、サーバー側のプラグインから機能を実行するたびに、コンテキストは毎回（明らかにフラグ名から）新しく、ビルドで自動的に行われます製品のパフォーマンスを向上させるにはfalseなので、nuxt.config.jsでこのオプションを無効にする必要がありました


render: {
    bundleRenderer: {
      runInNewContext: false,
    },
  },
