Mengapa lebih baik mengkompilasi TS di JS terlebih dahulu

Suatu hari muncul pertanyaan: "Gunakan pelari yang akan mengkompilasi TypeScript on-the-fly dalam JavaScript (misalnya, node-ts), atau kompilasi sendiri di muka (misalnya, melalui` tsc`) dan sudah menjalankan kode JavaScript?" - Googling tidak memberikan jawaban yang jelas, jadi saya merumuskannya dengan cara yang sangat penting:
TypeScript harus dikompilasi terlebih dahulu, melalui kompiler khusus (misalnya, tsc) dan menjalankan kode JS sudah.

Beberapa alasan topikal:

Lintas-platform


Anda tidak pernah tahu di mana Anda akan menyebarkan kode Anda.

Hari ini adalah server yang Anda kontrol di mana Anda dapat menggunakan node-ts, besok itu adalah AWS Lambda atau Google App Engine, di mana Anda perlu menggunakan kode JS.

Ketika beralih dari node-tspra-kompilasi, kesalahan dan keributan yang tidak terduga dapat terjadi dengan perubahan dalam konfigurasi CI / CD dan sejenisnya, oleh karena itu lebih dapat diandalkan untuk segera memulai dengan pra-kompilasi.

Tangkap jika Anda bisa!


Kesalahan menjijikkan, tidak terlihat ...

Saya menemukan salah satunya baru-baru ini: Saya menulis aplikasi TypeScript untuk Node.js, mengkompilasinya dan memasangnya di Google App Engine.

Semuanya berfungsi pada Mac pribadi, dan kesalahan muncul di server bahwa tidak ada file di sepanjang jalansome/folder/path/FileName.ts

Error: Cannot find module 'some/folder/path/FileName.ts' at ...

Orang akan berpikir bahwa kode itu tidak dapat dikompilasi, atau saya mengunduh versi lama, atau crash NODE_PATHdan sejenisnya, tetapi semuanya ternyata jauh lebih buruk:

Karena saya mengkompilasi terlebih dahulu, saya bisa melihat kode js seperti apa yang dikirim ke server. Dan kemudian menjadi jelas bahwa hanya SATU file ini tidak dikompilasi dengan nama asli: FileName.ts- tetapi dengan huruf kecil: fileName.ts...



Berguna untuk mencari tahu apa masalahnya dan ternyata:

(1) TS kompiler menamai file yang dikompilasi tidak sesuai dengan nama aslinya, tetapi menurut dengan impor dalam kode.

Artinya, jika Anda menulis dengan huruf kecil import * from "some/folder/path/fileName", dan aslinya ditulis dengan huruf besar ( FileName.ts), TS akan mengkompilasinya dalam file dengan huruf kecil. Bahkan jika dalam sisa kode ada impor di mana-mana dengan huruf kapital:import * from "some/folder/path/FileName"

(2) Pada poppy Node.js, aplikasi ini tidak peka huruf besar kecil, jadi semuanya bekerja dengan tenang

(3) Di Google App Engine, huruf besar-kecilnya huruf ...

Saya memiliki opsi untuk mengkompilasi file ts dengan cepat dan mengirimkannya ke Google App Engine dan kemudian saya akan menghabiskan terlalu banyak waktu untuk mencari tahu masalah yang sama. Tapi karena saya sedang melalui pra-kompilasi, saya bisa dengan aman masuk ke file JS yang dikirim dan mengamankannya.

Total


Jika Anda menggunakan TS, maka kompilasi dulu dengan alat yang sesuai (tsc, webpack, dll.), Dan baru kemudian jalankan / deploy ke platform yang diinginkan.

Jadi ketika TS memiliki VM sendiri, maka ... kami akan mengkompilasi selama beberapa tahun sampai muncul di ekosistem yang berbeda seperti Google, Amazon, .etc, dan sekarang kita akan hidup dengan cara.

Apa yang Anda pikirkan tentang ini: apakah ada manfaat untuk menggunakannya node-ts? Pernahkah Anda menemui masalah yang tidak jelas tanpa kompilasi?

Semoga sukses untuk semua orang, jangan biarkan debugging membunuh kebahagiaan pengkodean!
