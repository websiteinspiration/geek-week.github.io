あなたがそれについて何も知らなくて、あなたが決めるとき、その仕事は単純に思えます。

「素晴らしい」ある日、私は多数のフォーラムに参加することに興味がないことに気付きましたが、自分のチャンネルを作成して「知恵」を共有したいと思います。

PILライブラリを使用して処理した後（透かしの直後）の画像で、もう1つのわかりにくいことが起こりました。ボットに送信します。

 with open(photo_path, 'rb') as fi:
    bot.send_photo(message.chat.id, fi)

すべてが素晴らしいです、私は絵が好きです。次に、画像にコメントを追加して、すべてが一緒に美しく見えるかどうかを確認する必要がありますか？私たちは書く：

bot.send_photo(message.chat.id, message.photo[-1].file_id, caption=' ')

何らかの理由で、元の未処理の画像がボットに入ります。まあ、だましてみましょう：getupdates.offset -1意味がありません。テレグラムはこれが同じ写真であることを確信しています。わかりました、これを行います：

 with open(photo_path, 'rb') as fi:
    info = bot.send_photo(message.chat.id, fi)

リライト：

bot.send_photo(message.chat.id, info.photo[-1].file_id, caption=' ')

結果は同じです-元の画像が表示されます。そして、最初の引数のメッセージをinfoに置き換えるだけで、望ましい結果が得られました。

Roskomnadzorとの戦い方について一言。 APIへのアクセスがブロックされているため、ボットをローカルで開発するには、何らかの形で強固な防御側にトラフィックを渡す必要があります。これは、VPN経由またはプロキシ経由の2つの方法で実行できます。最も単純で最速の「通常の」方法はsshトンネルです。クライアントとプロキシサーバー間の接続を確立します。ローカルホストにポートがあり、トラフィックは私たちの側から送信でき、反対側（既にドイツのどこか）から送信されます。便宜上、この接続の自動化の一部を追加することができます-必要に応じて起動するスクリプトとデスクトップ上のショートカット。 「通常の」方法とは、ここで私たちが自分で入口と出口を制御する状況を意味します。左側がラップトップ、右側がドイツのVPSです。途中で、トラフィックは必要に応じて傍受でき、暗号化されています。出口では、他の州の管轄内にいることがわかり、ロシア連邦の法律は適用されません。さらに、私たちが自分で変更するまで、プロキシサーバーが変更またはオフにならないという追加の保証が得られます。それら。叔父のVPNとは異なり、私たちは安心と将来の自信を持っています。

サーバーとの接続を確立し、ポートが開いたら、何らかの方法でそこにトラフィックを転送する必要があります。最小の抵抗のパスに沿って歩き、IDE \ docker \ pythonでプロキシを構成する方法を考えないようにするために、1つの設定を行うことができます。proxychainsはそのような設定になります。このユーティリティを使用してソフトウェアを実行すると、configに登録されているプロキシチェーンを介してトラフィックがリダイレクトされます。

/etc/proxychains.conf

私たちの場合、これは1つのチェーンであり、書くことは難しくありません。

socks5 127.0.0.1 1080

その後、フィールドを入力します。その後、IDEの視点からの起動は次の形式になります：proxychains python3 app.py-これがまさに私たちが必要とするものです。したがって、プロジェクトを開始する場合は、デスクトップでスクリプトを実行し、IDEの[再生]ボタンをクリックしてボットを起動するだけです。残りのすべてのマジックは、proxychainsとsshによって実行されます。
