React Nativeでレガシープロジェクトをリファクタリングするときのナビゲーションの操作

この資料は、モバイル開発におけるナビゲーションのリファクタリングに取り組むためのものです。
この記事では、考えられる問題の例と、React Nativeでのモバイル開発におけるナビゲーションの作業に関連する困難な状況から抜け出す方法を提供します。

モバイルレガシープロジェクトの主な機能と問題を検討し、タスクの設定、ボトルネックの特定、既存の問題の解決策の発見、および望ましい結果の取得からなる一連のアクションを順に説明します。



ワークスペース


React Nativeは、Facebookが作成した比較的若いオープンソースのモバイルアプリ開発プラットフォームです。Android、iOS、UWP用のアプリケーションを作成するために使用され、開発者が独自のプラットフォーム機能と一緒にReactを使用できるようにします。

ナビゲーションは、モバイルアプリケーションでほぼ最も重要な要素です。アプリケーションのアーキテクチャを構築するときは、このトピックに特に注意する必要があります。

ナビゲーションを構築するための最も一般的なライブラリ：

  • React Navigation これは最も一般的なライブラリであり、公式ドキュメントでよく言及されています。したがって、彼女はGitHubの貯金箱に多くのスターを抱えています。
  • React-Native-Navigation。このプラットフォームは、iOSおよびAndroid用の基本的なネイティブAPIを使用します。これは、React-Navigationの一般的な代替手段です。これは、プラットフォームの合意の遵守を高く評価し、カスタマイズについてあまり気にしない人に適しています。

あまり一般的でないライブラリ：

  • React-Router-Native。これは不完全なライブラリです。しかし、すでにReact Router APIに精通していて、アプリケーションにかなり単純な要件を作成している場合、このプラットフォームが役立ちます。
  • React-Native-Router-Flux。このライブラリはReact-Navigationに基づいていますが、それと対話するための別のAPIを提供します。

私の経験


開発者としての私の経験について簡単に説明します。私は2年以上にわたってReact Nativeを使用しています。アプリストアで公開されたいくつかの完成したプロジェクトがあります。この間、React Nativeの環境で多数のツールに精通することができました。ナビゲーションは、多くの場合、モバイル開発プロジェクトの障害です。私が取り組んでいたレガシープロジェクトも例外ではありませんでした。これについては後で説明します。

仕事


ロールを2つのタイプ（ユーザーと管理者）に分割する代わりに、アプリケーションをリファクタリングするプロセスでコードが複製されました。残念ながら、ナビゲーションにも同じ問題がありました。
アプリケーションのアーキテクチャ上の機能は考慮されていません。

すべての画面は、認証画面、管理者画面、ユーザー画面の3つのタイプに分けられました。

ルートスタックと役割切り替えによるナビゲータータブ間の論理的な接続の代わりに、ユーザーナビゲーションの役割スタックへの非論理的な分割がありました。



さらに、ユーザーごとのナビゲーションの構築が繰り返されました。



次の問題を解決する必要がありました。

  • アプリケーションの一般的な速度低下。
  • メイン画面の「プリレンダラー」の欠如。これはユーザーの快適性に最良の方法で影響を与えませんでした。
  • ナビゲーションはネイティブに見えませんでした。トランジションアニメーションは目立ち、作業には不便でした。

タブナビゲーターとして機能したモーダルウィンドウコンポーネントのスクリーンショットを追加します。



  • ナビゲーションスタックの「固定」画面、つまり 排出（浄化）はなかった。開いた画面はスタックに保存されました。したがって、メモリリークとそれに続くアプリケーションのクラッシュはほとんど避けられませんでした。

決定


作業の結果：

  • ナビゲーションの最適化、プリレンダラーの欠如、および一般的な速度低下に関連する問題は、ナビゲーションライブラリを更新し、タブナビゲーターを使用し、セルフスタイルのモーダルウィンドウをナビゲーターとして使用することを拒否することで解決しました。
  • リファクタリング後、ナビゲーションはより自然になりました。トランジションアニメーションの作業、およびタブのメイン画面の事前レンダリングを追跡することで、ユーザーの快適性が向上しました。
  • ナビゲーションは、ナビゲーションタブにスタックされています。トランジションの使用がより適切に制御されました。上位レベルへの遷移が使用されました。

次のメソッドは、React-Navigationのバージョンで使用されました。

  • popToTop-スタックのメイン画面に移動します
  • replace-現在のルートを新しいルートに置き換えます
  • リセット-ナビゲーターの状態をクリアし、いくつかのアクションの結果で置き換えます

結果


重複したナビゲーション画面のほとんどを取り除き、より論理的な構造を構築することができました。

リファクタリングプロセスでは、すべてのナビゲーションが再設計されました。ナビゲーションのリファクタリングの一部には、デュアルスクリーンを取り除き、統一されたナビゲーション構造を作成することも含まれていました。タブの外側にあるはずの画面はルートナビゲーターにあり、ナビゲータータブ自体の画面はスクリーンショットで見ることができます。



調査結果


まとめると、ナビゲーションライブラリの重要性に再度触れ、それらを操作するための最も効果的な戦略の使用と技術文書の詳細な調査に注意を払います。それ以外の場合は、既存のプロジェクトとそのコードベースのリファクタリングと改良に多くの時間と労力を費やします。

ナビゲーションライブラリの選択は、アプリケーション開発の重要なステップです。十分な検討と注意なしに行われた決定は、スケーラビリティを損なったり、他の問題を引き起こす可能性があります。
