Vue.js：3つのアンチパターン

Vue.jsはおそらく最も優れたJavaScriptフレームワークの1つです。直感的なAPIがあり、高速で柔軟性があり、使いやすいです。ただし、Vue.jsの柔軟性には特定の危険が伴います。このフレームワークを使用する一部の開発者は、小さな見落としがちです。これは、アプリケーションのパフォーマンス、または長期的にはそれらをサポートする機能に悪影響を与える可能性があります。 今日私たちが公開している資料の著者は、Vue.jsでアプリケーションを開発する人が犯したいくつかの一般的な間違いを解析することを提案しています。





計算されたプロパティ内の副作用


計算されたプロパティは非常に便利なVue.jsメカニズムであり、他の状態フラグメントに依存する状態フラグメントを使用して作業を整理できます。計算されたプロパティは、その状態の他のデータに依存する状態に格納されているデータを表示するためにのみ使用する必要があります。計算されたプロパティ内でいくつかのメソッドを呼び出すか、他の状態変数にいくつかの値を書き込むことが判明した場合、これは何か間違っていることを意味している可能性があります。例を考えてみましょう。

export default {

特定の顧客の注文のコストの誤った計算

同様の状況では、次のことが起こります。計算されたプロパティは、無限ループで常に「再カウント」されます。割引を変更すると、計算されたプロパティがこれに反応し、注文の合計コストを再計算して、イベントを生成します。このイベントを処理すると、割引が再び増加し、これにより計算されたプロパティの再計算などが発生し、無限大になります。

このような間違いは実際のアプリケーションではできないように思われるかもしれません。しかし、本当にそうでしょうか？スクリプト（このようなことがこのアプリケーションで発生した場合）は、デバッグが非常に困難になります。このような間違いを追跡することは非常に困難です。実際には、このエラーが発生するためには、注文が特別な購入者によって行われる必要があり、そのような注文の1つに1000の通常の注文がある場合があります。

ネストされたプロパティを変更する


時々、開発者propsは、オブジェクトまたは配列であるプロパティの何かを編集したくなるかもしれません。そのような欲求は、それを行うのが非常に「簡単」であるという事実によって決定づけられます。しかし、それは価値がありますか？例を考えてみましょう。

<template>
  <div class="hello">
    <div>Name: {{product.name}}</div>
    <div>Price: {{product.price}}</div>
    <div>Stock: {{product.stock}}</div>

    <button @click="addToCart" :disabled="product.stock <= 0">Add to card</button>
  </div>
</template>
export default {
  name: "HelloWorld",
  props: {
    product: {
      type: Object,
      default: () => ({})
    }
  },
  methods: {
    addToCart() {
      if (this.product.stock > 0) {
        this.$emit("add-to-cart");
        this.product.stock--;
      }
    }
  }
};

ここにはProduct.vue、製品の名前、その値、および商品の数量を表示するコンポーネントがあります。このコンポーネントは、バイヤーが商品をバスケットに入れることを可能にするボタンも表示します。product.stockボタンをクリックした後でプロパティの値を減らすのは非常に簡単で便利なように思えるかもしれません。これを行うには、真実は簡単です。しかし、それだけを行うと、いくつかの問題が発生する可能性があります。

  • プロパティの変更（変更）を実行し、親エンティティには何も報告しません。
  • これにより、予期しないシステム動作が発生したり、さらに悪いことに、奇妙なエラーが発生したりする可能性があります。
  • productいくつかのロジックをコンポーネントに導入しますが、これはおそらくそこに存在すべきではありません。

user指令で 物事を進めるのは簡単v-modelです。Vue.jsはこれを可能にします。それだけではどうですか？それについて考えてください：

問題を「修正」する簡単な方法userは、プロパティとして送信する前にオブジェクトを複製することです。

<user-form :user="{...user}">

これはうまくいくかもしれませんが、問題を回避するだけで解決はしません。コンポーネントにUserFormは独自のローカル状態が必要です。これが私たちにできることです。

<template>
  <div>
   <input placeholder="Email" type="email" v-model="form.email"/>
   <input placeholder="Name" v-model="form.name"/>
   <button @click="onSave">Save</button>
   <button @click="onCancel">Save</button>
  </div>
</template>
export default {
  props: {
    user: {
     type: Object,
     default: () => ({})
    }
  },
  data() {
   return {
    form: {}
   }
  },
  methods: {
   onSave() {
    this.$emit('submit', this.form)
   },
   onCancel() {
    this.form = {...this.user}
    this.$emit('cancel')
   }
  }
  watch: {
    user: {
     immediate: true,
     handler: function(userFromProps){
      if(userFromProps){
        this.form = {
          ...this.form,
          ...userFromProps
        }
      }
     }
    }
  }
 }

このコードは確かにかなり複雑に見えますが、以前のバージョンよりも優れています。それはあなたが上記の問題を取り除くことができます。（watch）プロパティの変更を期待userし、それを内部データにコピーしますform。その結果、フォームは独自の状態になり、次の機能が得られます。

  • 変更をキャンセルすると、フォームを再度割り当てることができますthis.form = {...this.user}。
  • フォームには孤立した状態があります。
  • 親コンポーネントが必要ない場合でも、アクションは親コンポーネントに影響を与えません。
  • 変更を保存しようとするとどうなるかを制御します。

親コンポーネントへの直接アクセス


コンポーネントが別のコンポーネントを参照し、そのコンポーネントに対して何らかのアクションを実行すると、不整合やエラーが発生する可能性があります。これにより、アプリケーションの奇妙な動作や、関連するコンポーネントの外観が発生する可能性があります。

非常に単純な例、ドロップダウンメニューを実装するコンポーネントについて考えてみましょう。コンポーネントdropdown（親）とコンポーネントdropdown-menu（子）があるとします。ユーザーが特定のメニュー項目をクリックすると、を閉じる必要がありdropdown-menuます。このコンポーネントは、親コンポーネントによって非表示にされ、表示されますdropdown。例を見てみましょう。

// Dropdown.vue ( )

<template>
  <div>
    <button @click="showMenu = !showMenu">Click me</button>
    <dropdown-menu v-if="showMenu" :items="items"></dropdown-menu>
  </div>
<template>
export default {
  props: {
   items: Array
  },
  data() {
   return {
     selectedOption: null,
     showMenu: false
   }
  }
 }
// DropdownMenu.vue ( )
<template>
  <ul>
    <li v-for="item in items" @click="selectOption(item)">{{item.name}}</li>
  </ul>
<template>
export default {
  props: {
   items: Array
  },
  methods: {
    selectOption(item) {
     this.$parent.selectedOption = item
     this.$parent.showMenu = false
    }
  }
}

メソッドに注意してくださいselectOption。これが発生することは非常にまれですが、誰かが直接に連絡したい場合があり$parentます。この欲求は、実行が非常に簡単であるという事実によって説明できます。

一見すると、そのようなコードは正しく動作しているように見えるかもしれません。しかし、ここにはいくつかの問題があります。

  • プロパティを変更した場合はどうなりますshowMenuかselectedOption？ドロップダウンメニューを閉じることができず、その項目は選択されません。
  • dropdown-menu何らかのトランジションを使用してアニメーション化する必要がある場合はどうなりますか？

このコードも、変更のため$parent、機能しません。コンポーネントはdropdown親ではなくなりましたdropdown-menu。これで、親dropdown-menuがコンポーネントになりますtransition。

プロパティはコンポーネント階層の下に渡され、イベントは上に渡されます。これらの単語には、問題を解決するための正しいアプローチの意味が含まれています。以下は、イベント用に変更された例です。

// Dropdown.vue ( )
<template>
  <div>
    <button @click="showMenu = !showMenu">Click me</button>
    <dropdown-menu v-if="showMenu" :items="items" @select-option="onOptionSelected"></dropdown-menu>
  </div>
<template>
export default {
  props: {
   items: Array
  },
  data() {
   return {
     selectedOption: null,
     showMenu: false
   }
  },
  methods: {
    onOptionSelected(option) {
      this.selectedOption = option
      this.showMenu = true
    }
  }
 }
// DropdownMenu.vue ( )
<template>
  <ul>
    <li v-for="item in items" @click="selectOption(item)">{{item.name}}</li>
  </ul>
</template>
 export default {
  props: {
   items: Array
  },
  methods: {
    selectOption(item) {
     this.$emit('select-option', item)
    }
  }
 }

イベントを使用することで、子コンポーネントは親コンポーネントにバインドされなくなりました。親コンポーネントのデータを使用してプロパティを自由に変更し、アニメーション化された遷移を使用できます。ただし、コードが親コンポーネントにどのように影響するかについては考えていません。発生したことをこのコンポーネントに通知するだけです。この場合、コンポーネントdropdown自体が、ユーザーによるメニュー項目の選択とメニューを閉じる操作の処理方法を決定します。
