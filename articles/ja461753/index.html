InterSystems IRISグローバルトランザクション

InterSystems IRIS DBMSは、興味深いデータストレージ構造-グローバルをサポートしています。実際、これらはマルチレベルのキーであり、トランザクションの形でさまざまな追加の利点があり、データツリーをトラバースするためのクイック関数、ロック、および独自のObjectScript言語があります。

あなたは、リレーショナルデータベースの理論から知っているように、優れたトランザクションの実装では、の要件を満たす必要がありますACID：

C-一貫性。トランザクションが完了した後、データベースの論理状態は内部的に一貫している必要があります。この要件は多くの点でプログラマーに適用されますが、SQLデータベースの場合は外部キーにも適用されます。

グローバルは非リレーショナルデータ構造です。これらは非常に限られたハードウェアでの超高速作業用に作成されました。公式のIRIS Dockerイメージを使用して、グローバルでのトランザクションの実装を理解しましょう。

IRISでトランザクションをサポートするには、次のコマンドを使用します：TSTART、TCOMMIT、TROLLBACK。

1.原子性


原子性をチェックする最も簡単な方法。データベースコンソールから確認します。

Kill ^a
TSTART
Set ^a(1) = 1
Set ^a(2) = 2
Set ^a(3) = 3
TCOMMIT

次に結論を下します。

Write ^a(1), “ ”, ^a(2), “ ”, ^a(3)

我々が得る：

1 2 3

すべて順調。観察された原子性：すべての変更が記録されます。

タスクが複雑になり、エラーが発生し、トランザクションが部分的にまたはまったく保存されない方法がわかります。

原子性をもう一度確認してみましょう。

Kill ^A
TSTART
Set ^a(1) = 1
Set ^a(2) = 2
Set ^a(3) = 3

次に、コンテナを強制的に停止し、起動して確認します。 

docker kill my-iris

このコマンドは、SIGKILLプロセスを直ちに停止する信号を送信するため、強制的に電源をオフにするのとほぼ同じです。

トランザクションが部分的に保存されたのでしょうか？

WRITE ^a(1), ^a(2), ^a(3)
^
<UNDEFINED> ^a(1)

-いいえ、保存されません。

ロールバックコマンドをテストします。

Kill ^A
TSTART
Set ^a(1) = 1
Set ^a(2) = 2
Set ^a(3) = 3
TROLLBACK

WRITE ^a(1), ^a(2), ^a(3)
^
<UNDEFINED> ^a(1)

何も保存されませんでした。

2.一貫性


グローバル上のデータベースでは、キーもグローバルで作成されるため（グローバルはリレーショナルテーブルよりもデータを格納するための低レベルの構造であることを思い出します）、整合性の要件を満たすために、グローバルの変更と同じトランザクションにキーの変更を含める必要があります。

たとえば、パーソナリティを保存するグローバルな^人がいて、TINをキーとして使用しています。

^person(1234567, ‘firstname’) = ‘Sergey’
^person(1234567, ‘lastname’) = ‘Kamenev’
^person(1234567, ‘phone’) = ‘+74995555555
...

姓と名ですばやく検索できるように、キー^インデックスを作成しました。

^index(‘Kamenev’, ‘Sergey’, 1234567) = 1

ベースを合意するには、次のようなパーソナリティを追加する必要があります。

TSTART
^person(1234567, ‘firstname’) = ‘Sergey’
^person(1234567, ‘lastname’) = ‘Kamenev’
^person(1234567, ‘phone’) = ‘+74995555555
^index(‘Kamenev’, ‘Sergey’, 1234567) = 1
TCOMMIT

したがって、削除するときは、トランザクションも使用する必要があります。

TSTART
Kill ^person(1234567)
ZKill ^index(‘Kamenev’, ‘Sergey’, 1234567)
TCOMMIT

言い換えれば、一貫性の要件を満たすことは完全にプログラマーにかかっています。しかし、グローバルに関しては、その低レベルの性質により、これは正常です。

3.分離


これが荒野の始まりです。多くのユーザーが同じデータベースで同時に作業し、同じデータを変更します。

この状況は、多くのユーザーがコードを使用して同じリポジトリで同時に作業し、その中で多数のファイルへの変更を一度にコミットしようとする状況に匹敵します。

データベースはこれをリアルタイムで解決する必要があります。真面目な企業では、バージョン管理（ブランチのマージ、競合の解決など）を担当する特別な人がいることを考えると、データベースはこれらすべてをリアルタイムで実行する必要があり、タスクの複雑さとデータベースの正しい設計とそれを提供するコード。

データベースは、ユーザーが同じデータで作業する場合の競合を防ぐためにユーザーが実行するアクションの意味を理解できません。別のトランザクションとは逆に、1つのトランザクションのみをキャンセルするか、それらを順次実行することができます。

もう1つの問題は、トランザクションの実行中（コミット前）にデータベースの状態が不整合になる可能性があるため、他のトランザクションがデータベースの不整合状態にアクセスしないことが望ましいことです。これは、リレーショナルデータベースで、スナップショットの作成、マルチバージョン行の作成、および等

トランザクションの並列実行では、トランザクションが互いに干渉しないことが重要です。これは分離の特性です。

SQLは4つの分離レベルを定義します。

  • コミットされていない読み取り
  • コミット読み取り
  • 反復可能な読み取り
  • シリアライズ可能

各レベルを個別に検討してみましょう。各レベルを実装するコストは、ほぼ指数関数的に増加しています。

READ UNCOMMITTEDは最低レベルの分離ですが、最も高速です。トランザクションは、相互に加えられた変更を読み取ることができます。

READ COMMITTEDは次のレベルの分離であり、妥協です。トランザクションは、コミット前に相互に加えられた変更を読み取ることはできませんが、コミット後に行われた変更を読み取ることはできます。

長いトランザクションT1があり、その間にトランザクションT2、T3 ... Tnでコミットがあり、T1と同じデータを処理した場合、T1でデータを要求すると、毎回異なる結果が得られます。この現象は、非反復読み取りと呼ばれます。

反復可能な読み取り-この分離レベルでは、データを読み取るリクエストごとに結果データのスナップショットが作成され、同じトランザクションで再利用されるときにスナップショットのデータが使用されるため、繰り返し不可の読み取りという現象はありません。ただし、この分離レベルでは、ファントムデータを読み取ることができます。これは、同時にコミットされたトランザクションによって追加された新しい行を読み取ることを指します。

SERIALIZABLE-最高レベルの分離。これは、トランザクション（読み取りまたは変更）で何らかの方法で使用されたデータが、最初のトランザクションの完了後にのみ他のトランザクションで使用可能になるという特徴があります。

最初に、トランザクション内の操作がメインスレッドから分離されているかどうかを確認します。 2つのターミナルウィンドウを開きます。
Kill ^t

Write ^t(1)
2
 

TSTART
Set ^t(1)=2

孤立はありません。 1つのスレッドは、トランザクションを開いた2番目のスレッドが何を行うかを確認します。

異なるフローのトランザクションが内部で何が起こっているのかを確認してみましょう。

2つのターミナルウィンドウを開き、2つのトランザクションを同時に開きます。
kill ^t
TSTART
Write ^t(1)
3
 

TSTART
Set ^t(1)=3

並行トランザクションは、互いのデータを参照します。したがって、最もシンプルでありながら最速の分離レベルであるREAD UNCOMMITEDを取得しました。

原則として、これは常にスピードが最優先されているグローバルに期待できます。

しかし、グローバルオペレーションでより高いレベルの分離が必要な場合はどうでしょうか。

ここでは、断熱レベルが必要な理由とその機能について考える必要があります。

SERIALIZEの最高の分離レベルは、並行して実行されたトランザクションの結果がそれらの順次実行と同等であることを意味し、衝突の不在を保証します。

これを行うには、ObjectScriptの適切なロックを使用します。これには、さまざまな適用方法があります。LOCKコマンドを使用して、定期的、増分的、複数のロックを実行できます。

より低い分離レベルは、データベースの速度を上げるように設計されたトレードオフです。

ロックを使用してさまざまなレベルの分離を実現する方法を見てみましょう。

この演算子を使用すると、データの変更に必要な排他ロックだけでなく、読み取り中に他のプロセスによって変更されてはならないデータを読み取る必要があるときに、一度に複数のスレッドをとることができる、いわゆる共有ロックも取得できます。

難しさは、トランザクション中にデータベースの状態は不整合になる可能性がありますが、不整合なデータが他のプロセスから見えるという事実にあります。これを回避するには？

ロックを使用して、データベースの状態が合意されるような可視性のウィンドウを作成します。そして、合意された状態の可視性のそのようなウィンドウへのすべての呼び出しは、ロックによって制御されます。

同じデータの共有ロックは再利用可能です-複数のプロセスがそれらを取得できます。これらのロックにより、他のプロセスがデータを変更できなくなります。それらは、一貫したデータベース状態のウィンドウを形成するために使用されます。

排他ロックは、データを変更するために使用されます。そのようなロックを取得できるのは1つのプロセスだけです。排他的ブロッキングには、次のものがかかります。

  1. ,
  2. , shared- .



可視性ウィンドウが狭くなるほど、他のプロセスがそれを待つ時間が長くなりますが、その中のデータベースの状態の一貫性が高まります。

READ_COMMITED-このレベルの本質は、ロックされている他のストリームからのデータのみを表示することです。別のトランザクションのデータがまだコミットされていない場合は、古いバージョンが表示されます。

これにより、ロックが解放されるのを待つのではなく、作業を並列化できます。

特別なトリックがないと、IRISで古いバージョンのデータを見ることができないため、ロックを使用する必要があります。

したがって、一貫した瞬間にのみデータを読み取ることができるように、共有ロックを使用する必要があります。

相互に送金するユーザーベース^人がいるとします。

人123から人242に転送される瞬間：

LOCK +^person(123), +^person(242)
Set ^person(123, amount) = ^person(123, amount) - amount
Set ^person(242, amount) = ^person(242, amount) + amount
LOCK -^person(123), -^person(242)

引き落としの前に123人に金額を要求する瞬間は、排他ロックを伴う必要があります（デフォルト）：

LOCK +^person(123)
Write ^person(123)

また、アカウントのステータスをアカウントに表示する必要がある場合は、共有ロックを使用することも、まったく使用しないこともできます。

LOCK +^person(123)#”S”
Write ^person(123)

ただし、データベース操作がほぼ瞬時に実行されると仮定すると（グローバルはリレーショナルテーブルよりもはるかに低いレベルの構造であることを思い出します）、このレベルの必要性は低くなります。

REPEATABLE READ-この分離レベルでは、並行トランザクションによって変更できるデータの複数の読み取りが可能であると想定されています。

したがって、変更するデータの読み取りには共有ロックをかけ、変更するデータには排他ロックをかけなければなりません。

幸い、LOCK演算子を使用すると、1人のオペレーターが必要なすべてのロックを詳細にリストできます。

LOCK +^person(123, amount)#”S”
 ^person(123, amount)

その他の操作（現時点では、並列スレッドは^ person（123、amount）を変更しようとしますが、変更できません）

LOCK +^person(123, amount)
 ^person(123, amount)
LOCK -^person(123, amount)

 ^person(123, amount)
LOCK -^person(123, amount)#”S”

コンマで区切られたロックをリストする場合、それらは順番に取られます。

LOCK +(^person(123),^person(242))

その後、それらは原子的に一度に取得されます。

SERIALIZE-最終的に共通のデータを持つすべてのトランザクションが順次実行されるように、ロックを設定する必要があります。このアプローチでは、ほとんどのロックは排他的で、パフォーマンスのためにグローバルの最小領域に適用されます。

グローバル^パーソンでの償却について語る場合、彼はSERIALIZE分離レベルのみを受け入れることができます。厳密に連続してお金を使う必要があるためです。それ以外の場合は、同じ金額を数回使うことができます。

4.耐久性


コンテナを徹底的に切り落としてテストを行いました

docker kill my-iris

基地は彼らによく耐えました。問題は確認されませんでした。

結論


グローバルの場合、InterSystems IRISはトランザクションをサポートしています。それらは本当にアトミックで信頼性があります。グローバルでデータベースの一貫性を確保するには、外部キーなどの複雑な組み込み構造がないため、プログラマーの努力とトランザクションの使用が必要です。

ロックを使用しないグローバルの分離レベルはREAD UNCOMMITEDであり、ロックを使用する場合は、SERIALIZEレベルまで確保できます。

グローバルでのトランザクションの正確性と速度は、プログラマーのスキルに大きく依存します。読み取り時に広く使用される共有ロックが多いほど、分離レベルが高くなり、排他ロックが多く取られるほど、速度は速くなります。
