TypeScriptをReactアプリケーションに実装するための12のヒント

TypeScriptは、最新のフロントエンド開発の標準です。JavaScriptの調査によると、TSはプログラマにとって明らかに興味深いものです。2019年の調査によると、回答者の60％近くがTSを試し、それを使い続け、22％はTSを試みずに勉強したいと考えました。

この記事は、TypeScriptの使用を実装および改善する方法に関するヒントをまとめたものです。ヒントの前半は、アプローチとインフラストラクチャに関する一般的なものです。2番目は、いくつかの特に有用な言語機能です。



私はReactアプリケーションのコンテキストに固執しますが、いくつかのヒントは他のエコシステムに関連します。

ヒント1：エラー耐性


正しいタイプは、書かれたコードへの信頼とリファクタリングの促進です。タイピングが機能するためには、タイプチェックに厳密に適用する必要があります。

  1. devモードをチェックして、すぐにエラーを確認します。
  2. 各プールリクエストのCIに検証を追加します。これにより、メインブランチが常に正しいタイプであることが保証されます。

tsファイルをコンパイルするには、主に2つの方法があります。元のTSCと@ babel / preset-typescriptです。babelメソッドは興味深いものです。型情報は.tsファイルから切り取られるため、アセンブリが高速です。こちらが紹介記事です。

babelの場合のタイプチェックは、fork-ts-checker-webpack-pluginを使用して追加されます。開発モードでは、アセンブリをブロックしたりコードの記述を妨げたりしないように、チェックは非同期で行われる必要があります。逆に、プールリクエストでは、タイプチェックはエラーが見つかった場合にビルドを停止します。

ヒント2：バックエンドコントラクト


API自体を介したバックエンドとの通信は、データ交換契約を意味します。これらのエンティティを個別のレイヤーに分離し、個別のインターフェースを作成すると便利です。それらをDTO- データ転送オブジェクトと呼びます。

このような層を実装するには、いくつかの合意が必要です。以下は、チームに実装できるルールの例です。

  • インターフェース名のサフィックスはDto：FooBarDtoです。
  • 特別なフォルダにあります。
  • APIの変更によってのみ変更できます。
  • このインターフェイスのデータは読み取り専用に使用され、編集は禁止されています。

このアプローチの利点：データソースはインターフェイス上ですぐに明確になり、エラーを探してコードをデバッグし、データを検証するのが簡単になります。これらのタイプは簡単に文書化でき、説明と例を提供します-仲間の開発者が手元に運びます。Swagger

などのツールを使用してAPIを説明する場合は、これらのタイプを自動的に作成する方法をご覧ください。

ヒント3：任意のエイリアス


あなたとあなたのチームは見事に型付けされたコードを書きます-問題なく、エラーもなく、時間通りに。その場合は、次のポイントに進みます;）

実際には、特に既存のプロジェクトにTSを実装する場合は、タイピングを省略したいという誘惑があります。明示的または暗黙的にanyを使用したい。十分な理由があります：複雑なコード、緊急のタスク、不便さ-努力は結果にお金を払いません。

問題を無視する代わりに、それを合法化してみてください：

export type TODO_ANY = any;

長所：

  • あなたはカンニングや松葉杖を書かないでください。
  • あなたは意識的に、将来、コードは確定する価値があることを示します。
  • このタイプは、プロジェクトを検索することで簡単に見つけることができます。
  • 次のように、タイプが重要でない場合は、どれでも明示的に使用できます。

const getArrayLength = (array: any[]) => array.length

制限：

  • この手法を頻繁に使用しても、あまり効果がありません。
  • 将来そのようなギャグを取り除くには、意識と責任が必要です。

また、便利な場合もあります。 

export type TODO_UNKNOWN = unknown;

いずれ の場合でも、コンパイラはほとんどすべてのアクションを許可します。逆に、unknownの場合、ほとんどすべての使用はエラーになります。彼らはオブジェクトのタイプが明確になるまでTODO_UNKNOWNを設定します→将来、オブジェクトを使用しました→エラーを受け取りました→タイプを修正しました→あなたは素晴らしいです。

ヒント4：ライブラリのタイプ


人気のあるjsライブラリを使用している場合は、DefinitelyTypedで型を探します。ほとんどの場合、それらはすでに書かれています-私たちはつながり、人生を楽しんでいます。

しかし、タイプがまったくない内部ライブラリがある場合はどうでしょうか。自動的に生成してみてください。jsDocを使用している方に朗報です。TypeScript3.7では、tscコンパイラが.tsファイルと.jsファイルの両方から.d.tsタイプを生成する方法を学びました。 

別の良いニュース：js-> tsコンバーターがあります。たとえば、TypeStatは典型的なReactコンポーネントの処理に役立ちました。

悪いニュースは、優れた型システムが自動的に機能する可能性が低いことです。「肉」を構築するために後で必要となる「骨格」を当てにしてください。

ヒント5：タイプの上にあるタイプ


...そして追跡するタイプ！

プロジェクトの外部依存関係のタイプが適切ではない-それらは一般的すぎる（そして具体的にしたい）、エラーが含まれている、バージョンが適合しない-またはタイプが単に存在しない場合があります。

通常、このような問題は内部ライブラリ（頻繁に更新される）で発生しますが、外部ライブラリではまれです。問題が発生した場合は、タイプを手動で定義/再定義してみてください。

サンプルモジュールのタイプについて説明します。タイプのみが再定義されることを強調します。実装は変更されません。

declare module "example" {
    const foo: { bar: string };
    export default foo;
}

このようなオーバーライドは、たとえば、typings / example.d.tsのように、別のディレクトリとファイルに移動することをお勧めします。入力からすべての内容をtsconfigのcompilerOptions.typeRootsフィールドに接続します。

{
    "compilerOptions": {
        "typeRoots": ["node_modules/@types", "./typings"]
    }
}

出来上がり-たとえば、型は再定義から取得されます！

ヒント6：スタイルガイド


タイプまたはインターフェース？constまたはenum？機能コンポーネントで型を記述する方法は？とメモに？

TSがチームに導入されたばかりのとき、そのような質問が常に発生します。プロジェクトの初心者にも同様の一連の質問が表示されます。嵐は数週間続き、その後通常のコーディングスタイルが解決されます。

ルールを形式化することは、プール要求での紛争やホリバーの時間を節約するための優れた方法です。チームを集めて、スタイルガイドを作成します。

  • 基本として、人気のスタイルガイドを使用してください。私はこれらが好きです：react-typescript-cheatsheet、react-redux-typescript-guide、modern-typescript-with-examples-cheat-sheet。
  • リンターとフォーマッターの作業-マシンに汚れた作業を任せます。
  • プロジェクトの詳細（一般的なテンプレート、ベストプラクティス、ワーストプラクティス）を追加します。
  • これらの決定をコミットし、コードレビューで参照してください。
  • これらの決定について話し合い、変更する機会をチームに提供します。

よく構築されたスタイルガイドは、CPDD（コピーペースト駆動開発）の不思議を明らかにします。開発者は同じスタイルで記述し、既製のボイラープレートのおかげで時間を節約できます。壮大な戦闘のタブとスペースの配置（申し訳ありませんが、タイプとインターフェイス）は配置されていません。プロジェクトへの初心者のエントリーが簡略化されています。

PS：もう1つのインスピレーションの源は、tsユーティリティライブラリです。役立つヘルパーだけでなく、ソースのプロからのコードも見つかります。たとえば、utility-types、SimplyTypedです。

ヒント8：リテラル型と列挙型


TypeScriptでは、便利な構文構造- 列挙型が導入されています。 Enumは最終的な列挙に使用されます。列挙型をマスターしたので、最初は非常に頻繁に書きたいと思います。ただし、列挙型は、コンパイル後に生成されたコードに関してかなりの批判を持っています。

一方、リテラル型があります-最終的な列挙のより簡単で便利な代替です。場合によっては、そのようなタイプの使用がより適切です。

生活の例は単純なコンポーネントの小道具です：

//   enum
enum ColorEnum {
    yellow = "yellow",
    transparent = "transparent"
}

const ExampleEnum = (props: { color: ColorEnum }) => <div>{props.color}</div>;

//    
type ColorLiteral = "yellow" | "transparent";

const ExampleLiteral = (props: { color: ColorLiteral }) => <div>{props.color}</div>;

//   
const Example = () => (<>
        <ExampleEnum color={ColorEnum.yellow} />
        <ExampleLiteral color="yellow" />
 </>);

この例では、両方のコンポーネントが小道具の色のタイプをチェックします。enumを持つバリアントは、より冗長であることが判明しました-これは、コンパイル中の追加のコード、ソースコードの追加のインポート、およびより面倒な書き込み構造です。文字列リテラルを使用したバリアントはより簡潔ですが、型チェックも含まれています。列挙型コードを複雑にするのは意味がありません。

単純な型チェックには文字列リテラルを優先します。enumを使用して、より複雑なロジックを記述します。この場合、const enumを優先することをお勧めします（ただし、この構造はbabelではサポートされていないため、プラグインを追加する必要があります）。

ヒント9：constアサーション


enumをあまり使用しないもう1つの理由は、constアサーションです。

constは、このオブジェクトが変更されないことをコンパイラに伝えます。したがって、タイプをより詳細に推定できます。

const loadAction = {
    start: "LOAD_START",
    done: "LOADED",
    error: "LOAD_ERROR"
} as const;

type LoadActionType = typeof loadAction;
type LoadActionKey = keyof LoadActionType; // 'start' | 'done' | 'error'
type LoadActionValue = LoadActionType[LoadActionKey]; // 'LOAD_START' | 'LOADED' | 'LOAD_ERROR'
//  "as const" LoadActionValue   string

constとして配列に適用することもできます。

const DIRECTION_ASC = 1;
const DIRECTION_DESC = -1;
const sortDirectionTypes = [DIRECTION_ASC, DIRECTION_DESC] as const;
type SortDirection = typeof sortDirectionTypes[number]; // -> 1 | -1

乗り換えに便利です。たとえば、辞書にすべてのオプションが記述されていることを確認します。

const sortDirectionDescription: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " ",
    [DIRECTION_DESC]: " "
};
const sortDirectionDescription2: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " "
};
// Error: TS2741: Property '[-1]' is missing // in type '{ [DIRECTION_ASC]: string; }'

PS：constアサーションの詳細については、この記事をご覧ください。

ヒント10：ブランドタイプ


Brandは、ユニークな（「ブランド」）タイプを作成するための小さなユーティリティです。通常、これらのタイプはプリミティブ（文字列、数値）に基づいて作成されます。

たとえば、データIDを使用する場合は常に使用されます。どちらも文字列ですが、あるタイプのIDを別のタイプ（たとえば、ユーザーIDとストア内の注文のID）を区別することが非常に重要な場合があります。ブランドを使用すると、2つの異なるタイプを取得できます。

type Brand<T, U> = T & { __brand: U };

type UserId = Brand<string, "userId">; // "5435436" –  id 
type OrderId = Brand<string, "orderId">; // "42da-fet3-cdsl" –  id 

declare const userID = "1111" as UserId; // - 
declare function getOrderById(id: OrderId): any; // - 

getOrderById(userID);
// Error: TS2345
// Type '"userId"' is not assignable to type '"orderId"'.

ヒント11：差別化されたユニオンタイプ


複雑な用語の識別された共用体は、実際に理解するのは簡単です。それは、共通の部分（判別式）を持ついくつかのタイプの共用体を作成します。このような共用体を使用する場合、tsコンパイラーはオプションをフィルターで除外し、タイプを指定できます。

差別的なユニオンを使用する古典的な例は、レデューサーテンプレートにうまく適合します。共通タイプフィールドのおかげで、ペイロードのタイプが自動的に表示されます。

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {
        case 'SET_ONE':
            return { ...state, one: action.payload}; // `payload` is string
        case 'SET_TWO':
            return { ...state, two: action.payload}; // `payload` is number
        default:
            return state;
    }
}

ヒント12：しない


未加工のケースに対する優れた保険は決してありません。前の段落から、「SET_TWO」がなく、typeguardが追加されている場合のみ：決して;

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {

        case 'SET_ONE':
            return { ...state,
                        one: action.payload}; // `payload` is string

        default:
            const typeguard: never = action;
            return state;
            // Error: TS2322: Type '{ type: "SET_TWO"; payload: number; }'
            // is not assignable to type 'never'.
    }
}

コンパイラはタイプガードのタイプを推測し、矛盾します。一方では、そのタイプは決してありません。一方、出力タイプは「SET_TWO」です。タイプは収束せず、エラーが発生します。

Reactでneverを使用するもう1つの興味深い（ただしまれな）ケースは、プロップのネストまたはパスを禁止することです。

type IconProps = {
    size: 's' | 'm';
    children?: never;
    foo?: never;
};

const Icon: React.FC<IconProps> = () => <div>Whatewer</div>;
    
const Example = () => (<>
    <Icon size="s"> <div /> </Icon> {/*  –  children */}
    <Icon size="s" foo="123"/> {/*  -  foo */} 
</>);

また、条件付きタイプ（conditional-types）に完全に使用されることはありません。ここだクールな記事のすべての例は前に、あなたのためにあまりにも簡単だった場合には、この主題には。

結論


プロジェクトでTypeScriptを使用すると、コードの記述方法が変わります。コードの学習、リファクタリングの実行が容易になります。コードでコントラクトを使用すると便利になります。使用する関数の実装の詳細やオブジェクトの形状を覚えておく必要はありません。インターフェースに記述してください。 

プロジェクトはレベルアップを受け取り、新しい利点と共に新しいタスクが来ます-正しい型システムをサポートします。これらのヒントがシステムの改善に役立つことを願っています。
