新しいプロジェクトを始めるとき、組立システムの変更の可能性を考えました。これは、Gradleの代替手段を探す良い機会であるように思われました。さらに、既存のプロジェクトを翻訳するよりも、最初にこれを行う方が簡単です。以下のGradleの欠陥により、このステップに進みました。

  • 彼は間違いなくインクリメンタルアセンブリに問題があります。
  • 彼は非常に大規模なモノリシックプロジェクトでは不十分です。
  • 悪魔が非常に長い間始まるということが起こります。
  • それが実行されているマシンに要求します。

APK


まず最初に、Androidアプリケーションの構成を覚えておいてください：コンパイルされたコード、リソース、AndroidManifest.xml。



ソースは、Android仮想マシンで使用できる特別なdex形式のclasses.dexファイル（アプリケーションのサイズによっては、いくつかのファイルがある場合があります）にあります。今日、それは古いデバイスでのARTです-Dalvik。さらに、ネイティブソースがサブフォルダーに配置されているlibフォルダーを見つけることができます。これらは、x86、armなどのターゲットプロセッサアーキテクチャに応じて名前が付けられます。 exoplayerを使用している場合は、おそらくlibがあります。そして、プロセス間通信インターフェースを含むaidlフォルダー。別のプロセスで実行されているサービスにアクセスする必要がある場合に便利です。このようなインターフェースは、Android自体とGooglePlayServicesの両方で使用されます。

画像などのさまざまなコンパイルされていないリソースは、resフォルダーにあります。スタイル、ラインなどのすべてのコンパイル済みリソースは、resource.arscファイルにマージされます。アセットフォルダーには、原則として、カスタムフォントなど、リソースに収まらないすべてのものを入れます。

これに加えて、APKにはAndroidManifest.xmlが含まれています。その中で、アクティビティ、サービス、さまざまな権限など、アプリケーションのさまざまなコンポーネントについて説明します。それはバイナリ形式であり、内部を見るには、まず人間が読めるファイルに変換する必要があります。

コンソール


アプリケーションの構成がわかったので、Hello、world！Android SDKが提供するツールを使用してコンソールから。ビルドシステムはすべてこれらのユーティリティにある程度依存しているため、ビルドシステムの動作を理解する上でこれは非常に重要なステップです。プロジェクトはKotlinで記述されているため、コマンドライン用のコンパイラーが必要です。個別にダウンロードするのは簡単です。

アプリケーションのアセンブリは、次の手順に分けることができます。

  • プロジェクトが依存するすべてのライブラリをダウンロードして解凍します。私の場合、これはappcompat下位互換性ライブラリであり、これは今度はappcompat-coreに依存するため、同様にそれを汲み上げます。
  • R.javaを生成します。この素晴らしいクラスには、アプリケーション内のすべてのリソースの識別子が含まれており、コードでそれらにアクセスするために使用されます。
  • Android仮想マシンは通常のバイトコードを処理する方法を知らないため、ソースをバイトコードにコンパイルしてDexに変換します。
  • すべてをAPKにパックしますが、最初に画像などのすべての非圧縮リソースをファイルの先頭を基準に配置します。これにより、APKのサイズを完全に取るに足らない増加の代償として、その作業を大幅に加速できます。したがって、システムはmmap（）関数を使用して、リソースをRAMに直接マップできます。
  • アプリケーションに署名します。この手順により、APKの整合性が保護され、作成者が確認されます。そしてこれのおかげで、たとえば、Playマーケットはアプリケーションがあなたによって構築されたことを確認できます。

ビルドスクリプト
function preparedir() {
	rm -r -f $1
	mkdir $1
}

PROJ="src/main"

LIBS="libs"
LIBS_OUT_DIR="$LIBS/out"

BUILD_TOOLS="$ANDROID_HOME/build-tools/28.0.3"
ANDROID_JAR="$ANDROID_HOME/platforms/android-28/android.jar"
DEBUG_KEYSTORE="$(echo ~)/.android/debug.keystore"

GEN_DIR="build/generated"
KOTLIN_OUT_DIR="$GEN_DIR/kotlin"
DEX_OUT_DIR="$GEN_DIR/dex"

OUT_DIR="out"

libs_res=""
libs_classes=""


preparedir $LIBS_OUT_DIR
aars=$(ls -p $LIBS | grep -v /)
for filename in $aars;
do
	DESTINATION=$LIBS_OUT_DIR/${filename%.*}
	echo "unpacking $filename into $DESTINATION"

	unzip -o -q $LIBS/$filename -d $DESTINATION
	libs_res="$libs_res -S $DESTINATION/res"
	libs_classes="$libs_classes:$DESTINATION/classes.jar"
done

preparedir $GEN_DIR
$BUILD_TOOLS/aapt package -f -m \
	-J $GEN_DIR \
	-M $PROJ/AndroidManifest.xml \
	-S $PROJ/res \
	$libs_res \
	-I $ANDROID_JAR --auto-add-overlay


preparedir $KOTLIN_OUT_DIR
compiledKotlin=$KOTLIN_OUT_DIR/compiled.jar
kotlinc $PROJ/java $GEN_DIR -include-runtime \
	-cp "$ANDROID_JAR$libs_classes"\
	-d $compiledKotlin


preparedir $DEX_OUT_DIR
dex=$DEX_OUT_DIR/classes.dex
$BUILD_TOOLS/dx --dex --output=$dex $compiledKotlin


preparedir $OUT_DIR
unaligned_apk=$OUT_DIR/unaligned.apk
$BUILD_TOOLS/aapt package -f -m \
	-F $unaligned_apk \
	-M $PROJ/AndroidManifest.xml \
	-S $PROJ/res \
	$libs_res \
	-I $ANDROID_JAR --auto-add-overlay


cp $dex .
$BUILD_TOOLS/aapt add $unaligned_apk classes.dex
rm classes.dex


aligned_apk=$OUT_DIR/aligned.apk
$BUILD_TOOLS/zipalign -f 4 $unaligned_apk $aligned_apk

$BUILD_TOOLS/apksigner sign --ks $DEBUG_KEYSTORE $aligned_apk


数値によると、何もキャッシュせず、毎回すべてを再構築しないため、クリーンアセンブリには7秒かかり、インクリメンタルアセンブリはそれに遅れをとらないことがわかります。

メイベン


これは、Javaプロジェクトを構築するためにApache Software Foundationのメンバーによって開発されました。そのためのビルド構成はXMLで記述されています。Mavenの初期のリビジョンはAntによって収集され、現在は最新の安定版リリースに切り替えられています。

Mavenの長所：

  • アセンブリアーティファクトのキャッシュをサポートします。インクリメンタルビルドはクリーンよりも高速でなければなりません。
  • サードパーティの依存関係を解決できます。それら。MavenまたはGradleの設定でサードパーティライブラリへの依存関係を指定する場合、依存関係について心配する必要はありません。
  • かなり前から市場に出回っているので、詳細なドキュメントがたくさんあります。
  • また、バックエンドからAndroid開発の世界に最近来た場合は、おなじみのビルドメカニズムになる可能性があります。

短所Maven：

  • アセンブリが行われるマシンにインストールされているJavaのバージョンによって異なります。
  • Androidプラグインがサードパーティの開発者によってサポートされるようになりました。個人的には、これを非常に重大な欠点と考えています。
  • XMLは冗長性と扱いやすさのため、ビルド構成の記述にはあまり適していません。
  • 後で説明するように、少なくともテストプロジェクトでは、Gradleよりも実行速度が遅くなります。

バゼル


グーグルの内臓のエンジニアは彼らのプロジェクトを構築するためにバゼルを発明し、比較的最近それをオープンソースに移しました。build-configsの説明では、pythonのようなSkylarkまたはStarlarkが使用されていますが、どちらの名前にも意味があります。独自の最新の安定版リリースを使用してアセンブルされます。

バゼルの長所：

  • さまざまなプログラミング言語のサポート。ドキュメントを信じれば、彼はiO、Android、またはバックエンドのプロジェクトを収集する方法を知っています。
  • 以前に収集したアーティファクトをキャッシュできます
  • Mavenの依存関係を処理できます。
  • Bazelは、私の意見では、分散プロジェクトを非常に優れた方法でサポートしています。彼はgitリポジトリの特定のリビジョンを依存関係として指定でき、ビルドプロセス中にそれらをアンロードしてキャッシュします。たとえば、スケーラビリティをサポートするために、Bazelはクラウドベースのビルドサーバーにさまざまなターゲットを配布できます。これにより、かさばるプロジェクトをすばやくビルドできます。

バック


それはグーグルからフェイスブックへの脱北者によって発明されました。彼はPythonを使用して構成を記述した後、今日言及したSkylarkに移行しました。彼は突然、Antシステムを使用します。

バックプロ：

  • さまざまなプログラミング言語をサポートし、AndriodとiOSの両方を構築できます。
  • 以前に収集したアーティファクトをキャッシュできます
  • Buckは独自のdex実装を作成しました。これは標準の実装よりも高速に動作し、システムデーモンでハングします。したがって、dexの初期化にかかる時間を節約できます。エンジニアは本当に多くのことを最適化しています。たとえば、ライブラリの内部を変更するときにインターフェースが変更されていない場合、Buckはライブラリに依存するコードを収集しません。同様にリソースの場合：識別子が変更されていない場合、リソースを変更してもコードは再構成されません。
  • BuckをGredlovsky構成の背後に隠すことができるプラグインがあります。それら。実際にBuckを通じてビルドされた通常のGradleプロジェクトのようなものが得られます。

短所：

  • Bazelと同じように維持するのは難しいです。それら。ここでは、組み立てプロセスを明確に説明する低レベルのルールを説明することも必要です。
  • 特に、Buckはそれ自体ではMavenの依存関係を解決できません。

調査結果


各ビルドシステムについて説明したので、Hello、world！の例を使用して、それらを互いに比較できます。コンソールアセンブリは、その安定性に満足しています。ここでは、スクリプトを解析するためのサードパーティのコストが最小限であるため、ターミナルからのスクリプトの実行時間は、クリーンビルドのアセンブリの参照と見なすことができます。この場合、インクリメンタルアセンブリの非常にわずかな増加のために、Mavenを明白な部外者と呼びます。 Bazelは非常に長い間、構成を解析して初期化します。インクリメンタルビルドはクリーンよりもはるかに高速に実行されるため、初期化の結果をキャッシュするという考えがあります。バックはこのコレクションの誰もが認めるリーダーです。クリーンアセンブリとインクリメンタルアセンブリの両方が非常に高速です。



ここで長所と短所を比較します。MavenはGradleに負けており、市場ではほとんど使用されていないため、比較には含めません。BuckとBazelの利点と欠点はほぼ同じなので、私はそれらを結合します。

したがって、Gradleについて：

  • 最初に、そして私の意見では、最も重要なことは、それが単純であることです。とてもシンプルです。
  • 箱から出して、依存関係をアンロードおよびアンロードします。
  • 彼にとっては、さまざまなトレーニングとドキュメントがたくさんあります。
  • Googleとコミュニティの両方で積極的にサポートされています。現在の主力開発ツールであるAndroid Studioとの優れた統合。そして、Androidアプリケーションを構築するためのすべての新機能が最初にGradleに登場します。

バック/バゼルについて：

  • Gradleに比べて間違いなく非常に高速です。これは非常に大規模なプロジェクトで特に顕著だと思います
  • iOSとAndroidの両方のソースコードが含まれる1つのプロジェクトを保持し、それらを1つのビルドシステムでアセンブルできます。これにより、アプリケーションの一部がプラットフォーム間を移動できます。たとえば、これはChromiumの使用方法です。
  • 依存関係を詳細に説明することを余儀なくされたため、開発者は文字通りマルチモジュールになりました。
