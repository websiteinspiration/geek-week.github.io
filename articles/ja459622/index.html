セガサターンのゲームは1995年に書かれたので

これは、Neversoftスタジオの最初のゲームであるSkeleton Warriorsに取り組んでいたときに1995年に書いたドキュメントです。これは、68Kアセンブリ言語を使用しなかった最初のゲームでした。

当時の写真。開発キット（「Small Box」とICE）は私の右側にあります。


ゲームの状態


以下のドキュメントでは、セガサターンのスケルトンウォリアーズコードのステータスについて簡単に説明し、実行する必要がある多くの側面のいくつかについても説明しています。

このドキュメントは、Dan、Ken、Jamesが既成のコードに慣れるまでの時間を短縮し、各モジュールの目的とそれらの間の相互作用を説明するために必要でした。彼はまた、私にこのコードの悲しい状態を評価することを許可しました。

また、プログラムへのデータ（.GOVファイルと.GOBファイル）の埋め込み、および今後の予定についても少し説明します。

開発機器


ターゲットプラットフォームは、2つのSH2 Riscマイクロプロセッサと1つの68000を備えたSega Saturnです。マスターSH2メインプロセッサのみを使用している限り、これを行う方法を理解するときに補助SH2スレーブが使用されます。 68000はサウンドチップの制御に使用されます。セガが提供するサウンドライブラリを使用するため、コードを記述する必要はありませんでした。

プログラムはほぼ完全に純粋なCで書かれています。SH2出力アセンブラーを取得するには、GNU SH2コンパイラーを使用します。コードにはいくつかのSH2モジュールがあり、主にデータのみが含まれています。これまでのところ、SH2について意味のあるものは何も書いていません。

開発システムとして、PsyQを使用しています。これは標準のSega開発システムではありませんが、これを使用したすべての人が最高のシステムだと考えています。セガが所有するCross Productsによって作成されたSNASMは、これに代わるものです。 Segaが提供するコードサンプルのほとんどは、SNASM開発システムで動作するはずですが、PsyQに簡単に変換できます。

PsyQシステムは、PCに取り付けられたSCSIインターフェイスカード、土星に挿入されたケーブルを接続するカートリッジで構成されます。ソースはPCでコンパイルされ、プログラムが起動する土星にダウンロードされます。 PCからコードをデバッグできます。


PsyQ

通信開発システムは、マシン間の通信を処理する常駐プログラム（PSYBIOS）によって制御されます。これにより、Saturnコンソールは、CDからファイルをダウンロードするのとほぼ同じ方法でPCからファイルをダウンロードできます。この機能を使用して、各レベルのファイルをダウンロードします。

私の部屋には大きくて大きな引き出しがいくつかあり、さらに2台のPCがあります。 2つのボックスのうち小さい方は、組み込みSH2エミュレーターであるE7000PCです。 PsyQデバッガーがプログラムを停止しない場合、プログラムがどこでクラッシュするかを理解するのに役立ちます。また、メモリ書き込みの追跡にも役立ちますが、これまでのところ、この機能はほとんど使用していません。

2番目の大きな引き出しは、小さな箱と呼ばれるものです（最初の大きな箱は、小型冷蔵庫のサイズとほぼ同じでした）。本質的には、E7000とCDエミュレーター用の追加インターフェースを備えた土星です。フロントパネルには、国IDスイッチと、PALとNTSC間のスイッチがあります。

2台目のコンピューターの内部には、CDエミュレーターがあります。これは、コンピューターのハードドライブがCDドライブのふりをしている大きなボードです。CDイメージを収集し、リアルタイムでエミュレートして、実際のCDに到達したときのゲームの外観を確認できます。エミュレータは多少は機能しますが、現在Segaで取り組んでいるいくつかの問題があります。


セガの開発キット自体

コンパイルとリンク


完成したプログラムの一般的なアセンブリは、1つのメイクファイルMAKEFILE.MAKによって制御されます。 .GOBファイルと.GOVファイルのコンパイルを含む、プロジェクト全体の依存関係とターゲットが含まれています。

個々のCソースコードモジュール（.Cファイル）は、CCSHによってSH2オブジェクトモジュール（.OBJ）にコンパイルされます。最初にCPPSH（C：\ GNUSH2 \ BINにあります）と呼ばれるGNU Cプリプロセッサーを呼び出し、次に出力用にCC1SHを呼び出してSH2アセンブラーコードを作成し、最後にASSH（C：\ PSYQ内）を呼び出してそれをビルドします完成したオブジェクト形式に。

巨大なオブジェクトファイルを作成すると言われたため、C ++は使用しません。しかし、私は彼と一緒に仕事をしませんでした、あなたは実験することができます。

SH2アセンブラー言語のいくつかのファイル（拡張子.S）は、ASSMHを使用して.OBJファイルに直接収集されます（これはASSHと同じではありませんが、より複雑なマクロアセンブラーです）。現在、これらはデータの埋め込みにのみ使用され、マシン依存のコードは含まれていません。

コードをロードできる土星RAMは、2つの1MBブロックに分割されます。 1つは06,000,000ドルから始まり、もう1つは002,000,000ドルからです。 00,200,000ドルのブロックは、メインキャラクターのグラフィックスを格納するためにのみ使用されます。プログラムコードは$ 06010000のアドレスに書き込まれます（最初の$ 10,000バイトはシステムスペース、スタックなどに使用されます）

。コードは位置に依存し、この特定のアドレス（$ 06010000）で実行されるようにコンパイルされます。

.OBJファイルはPSYLINKプログラムを使用してリンクされ、MAIN.CPEファイルを作成します。これは、RUNコマンドでSaturnにダウンロードできる小さなヘッダーを持つ実行可能プログラムです。PSYLINKはTEST.LNKファイルを使用して、どの.OBJファイルを含め、どこに置くかを指定します。

データ


ゲームはいくつかのレベルに分かれており、多くのレベルは同じデータを使用しますが、基本的にはレベルごとに異なります。各レベルのすべてのデータは、2つの巨大な.GOVファイルと.GOBファイルに収集されます。 （地雷の場合、MINE.GOVとMINE.GOBです）。 GOVファイルには短いヘッダーが含まれており、ビデオメモリにあるすべてのデータが含まれています。 .GOBファイルには、RAM内にある必要のあるすべてのデータが含まれています。

レベルは、以下に示すデータファイルの一部で構成されています。

.SSQ-スプライトシーケンサーファイル
.SBM-ビット背景に使用されるビットマップファイル
.MAP-シンボルで満たされた背景の両方のカード。
.TIL-シンボルで満たされた背景のタイルセットとパレット。
.PTH-道路ポイントとトリガーのデータ。
.TEX-道路のテクスチャ。

.SSQファイルと.SBMファイルは、ますます不快になってきているSEQシーケンサーによって作成されました。.MAP、.TIL、.PTH、および.TEXファイルは、ますます素晴らしいTULEマップエディターであるDanによって作成されました。

これらのファイルは、ASSMHアセンブラを使用して、対応する.GOVおよび.GOBファイルにアセンブルされます。これがどのように行われるかを確認するには、LEVEL.SおよびLEVEL1.Sファイルを参照してください。.GOVファイルには、特定のレベルのデータも含まれます。

モジュール


TEST.S-特別なことは何もせず、ラベルをいくつか設定します。

MAIN.Cはプログラムのトップレベルです。これには、機器の初期化、レベル設定、レベルを渡すためのコード、および実際により適切なモジュールに配置する必要があるその他のさまざまな小さな要素が含まれます。クイックテストのために、このモジュールに新しいものを追加するのが最も簡単なため、大量のゴミが含まれています。 PCのCDまたはファイルサーバーからのダウンロードコードが含まれています。 TIMINGカラーバーを有効または無効にするフラグが含まれています。

Gfxlib.c-機器にアクセスし、さまざまなグラフィック機能を実行するためのさまざまな手順。それらのほとんどすべてはDanによってゼロから書かれており、しばしば非常に非効率的です。ここから手順を頻繁に使用する場合、それが何をするかを見て、コードでより高速なバージョンを書くとよいでしょう。

ただし、すべての機能が機能し、大まかな実装とテストのための優れたフレームワークを提供します。ダンに感謝します、彼なしではそれは不可能だったでしょう。

SMP_PAD.C-土星のジョイスティックから読み取るためのさまざまな手順。装置に大きく依存します。

GLOBALS.C-すべてのグローバル変数といくつかの一般的な関数。グローバル変数の使用は、許容できるプログラミング方法です。ただし、さまざまな理由により、SH2でのグローバル変数の実装はかなり遅いため、必要に応じて、時間の経過とともにパーツをグローバル構造に変換する可能性があります。MANおよびPATHの状態を説明する変数が含まれています。

MAN.C-人の動きと表示を処理します（プリンスライトスター、タリン、ガーディアン、グリムスカル-プレイヤーが操作するキャラクター）。これまでのところ、これは主に移動と道路との衝突のロジックです。さらに、各アクションに適切なアニメーションを提供します。やるべきことはまだたくさんあります。

OB.C-ゲーム内のオブジェクト、特にスケルトンウォリアーや小さなエイリアンなどの敵のオブジェクトの動きと表示を処理します。ゲームプレイの主要部分はここでプログラムされます：敵AI、基本的な動き、トリガーのトリガー。データ構造はまだ準備ができていません。特に、衝突とアニメーションの問題は完全には解決されていません。やるべきことはまだたくさんあります。

DATA.S-さまざまなテーブル、現在は主にプレイヤーのメインキャラクターのアニメーション。

LAYER.C-視差付きの背景のスクロール。シンボルの背景とスクロールビットマップを更新します。フォグレイヤーのライン（波効果）もスクロールします。これまでのところ、シンボルマップレイヤーのテーブルは圧縮せずに格納されています。これらは、Genesisバージョンで使用したRLE形式に圧縮する必要があります。土星の開発システムをソニーよりも早く入手できれば、この仕事はケンに任せることができます。

PAL.C-パレット。 2048色からお選びいただけます。画面上のピクセルは、これらの色のいずれかになります。パレットを256色の8つのパレットに論理的に分割しました。 PAL.Cには、初期化、準備、および周期的変更のコードが含まれています。また、調光やより複雑な循環シフト、明るさの点滅なども必要になります。

BUL.C-シェルを処理するためのプリミティブシステム（剣を投げる、手をパンチする、腕から撃たれるロケットなど）を個別のオブジェクトとして。シェルをより複雑に使用するには、依然としてかなりの作業が必要です。正しい衝突とアニメーションのコードも必要です。

PAD.Cは、ジョイスティックの状態をより便利な形式で保存するためのシンプルなモジュールです。ボタンが最近押されたかどうか、および現在押されているかどうかを記憶します。

START.C-バッチファイルで簡単に変更できるように、最初のレベルを示す1行。

PANEL.C-筋の筋を撤回するための簡単な手順。

PATH.C-道路を描画し、道路との衝突を処理するための巨大な手順。

MATH.C-角度上の点の単純な正弦、余弦、回転。

[更新] MAN.Cからのサンプルコードは次のとおりです。すべてが厳格にコードで記述され、グローバルデータ構造Manを参照します。コードに書かれた数字の束。

/**************************************************************/
/* Trigger jumping if needed, also variable height jump logic */

Man_JumpTrigger()
{
  if ( Man.JumpFudge )
  {
    Man.JumpFudge--;
  }

  if ( Man.Mode != M_Crouch || Man_StandingRoom() )    // ok if not crouched, or there is headroom
  {
    if (Pad_Jump->Pressed)               /* jump button pressed */
    {
      if ((Man.Contact || (Man.Mode == M_Hang) || Man.JumpFudge) && Pad_Jump->Triggered && !Man.Blocking) /* and not already jumping */
      {
        if (Man.Mode == M_Hang && Pad1.Down.Pressed)
        {
          Man.Contact=0;
          Man.Mode=M_Jump;
          Man.AnimBase = LS_Jumping;    /* Change base anim to jumping */
          Man_TriggerSeq(LS_Jump);    /* start the jumping start anim */
          Man.YV.f = 0x10000;           /* and have no YV */
          Man.Y.i += 4;           /* and have no YV */
        }
        else
        {
          Pad_Jump->Triggered = 0;
          if ( !JetPacCheat )
            Man.YV.f = -0x00080000;     /* Initial jump speed */
          else
            Man.YV.f = -0x00008000;     // Initial speed in Jetpac mode
          Man.Contact = 0;          /* not on the ground any more */
          Man.JumpTime = 0;         /* just started jumping */
          Man.AnimBase = LS_Jumping;    /* Change base anim to jumping */
          Man_TriggerSeq(LS_Jump);    /* start the jumping start anim */
          Man.XV.f+=Man.FlyVel;

          if (Man.HangEnd && Man.Mode == M_Hang)  // if hanging
          {                   // and on the end of a path
            Man.HangEnd = 0;
            Man.X.i += 12*Man.Facing; // the move past end of path
            Man.JumpTime = -3;      // bit more fixed v jump time
          }
          Man.Mode = M_Jump;    /* change mode to jumping */

        }
      }
      else                        /* Already jumping */
      {
        if (Man.JumpTime++ < MaxJumpTime) /* Still in initial jump period */
          Man.YV.f -= 0x0005000;        /* So can maintain jump YV */
      }
    }
    else                      /* jump button not pressed */
    {
      Man.JumpTime = MaxJumpTime+1;     /* so can't alter YV again until landed */
    }

  }

}
