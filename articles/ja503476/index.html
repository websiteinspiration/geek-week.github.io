ヘッドレスChromeで動作するビデオUI自動テストを記録する

みなさん、こんにちは！

この記事では、ヘッドレス Chromeで動作するビデオの自動テストを記録するタスクがどのように解決されたかについて説明したいと思います（非ヘッドレスモードでの記録には問題はないはずです）。いくつかの異なるアプローチが検討され、それらを解決するための困難と方法が通知されます。

問題の定式化

  1. Windowsで実行されるテスト
  2. テストではSelenium Web Driver + Headless Chromeを使用します
  3. テストは複数のスレッドで実行されます

落ちたテストでは、ビデオを保存する必要がありますが、

  1. ランタイムは10％を超えて増加しないはずです。
  2. 現在の実装では最小限の変更で行う

解決策に興味がある場合は、猫へようこそ。

素朴なアプローチ。スクリーンショット


テストフレームワークには、Seleniumの低レベルのラッパーがあります。したがって、最初の実装は非常に単純で非常に単純でした。ページを変更するすべての場所（クリック、テキストボックスの設定、ナビゲートなど）にコードが追加され、Selenium Web Driverを介してスクリーンショットが保存されました。

Driver.TakeScreenshot().SaveAsFile(screenshotPath);

このアプローチでのテスト実行時間は大幅に増加しました。理由：スクリーンショットを保存する操作が、0.5秒から数秒まで、まったく迅速に機能しません。

別のストリームのスクリーンショット


ページを変更するすべての場所にスクリーンショットを保存するコード（クリック、テキストボックスの設定、ナビゲート）の代わりに、スクリーンショットを常に別のストリームに保存するコードが追加されました

テストの実行時間はまだ非常に長かった。遅れの理由がわかりませんでした。おそらく、スクリーンショットが保存されている間、Seleniumは何かを拒否します。おそらく、別のSeleniumのインスタンスが役立ち、同じセッションで終了します。

Puppeteerの別のスレッドのスクリーンショット


セレニウムの2つのインスタンスを実行することはあまり面白くありませんでした。私は長い間、操り人形師の鋭い試練を試してみたかったのですが、ここで適切な理由を見つけました。Seleniumの側では、Seleniumを介して既に作成されているChromeに単純に接続されたPuppeteerが作成されました

マイナス

  1. Puppeteerを介してスクリーンショットを保存する時間は瞬時ではなく、一部のフレームが失われ、解析が興味深いものになる場合があります。
  2. Seleniumがタブを切り替える場合は、Puppeteerに通知する必要があります。そうでない場合は、上記のコードの最初のタブのスクリーンショットのみが表示されます（アクティブなタブを見つける方法がある可能性があるため、確認する必要があります）。

最初のマイナスは私にとってブロッキングであることが判明したので、次の解決策に進みます。

スクリーンキャスト


Chromeには興味深い機能があります-Page.startScreencast。説明によると-彼女は必要なことをするだけです-彼女は変更したフレームをキャストするので、希望する人はそれを傍受して面白いことをすることができます。

SeleniumとPuppeteerの両方で、Page.startScreencastを開始できますが、どちらか一方にハンドラーを追加することはできません。ホテル経営者はすでに発言しています-私たちは実装を待っています。ChromeDevTools

ライブラリを使って友達を作ってみました。残念ながら、私は彼女とすぐに良い関係を築くことができませんでした。さらに検索した後、mafredri / cdpの ScreenCastのソリューションが見つかりました。元の例から不要なナビゲーションが削除され、必要な入力パラメーターが追加されました。

隠しテキスト 
package main

import (
    "os"
    "context"
    "fmt"
    "io/ioutil"
    "log"
    "time"
    "flag"

    "github.com/mafredri/cdp"
    "github.com/mafredri/cdp/devtool"
    "github.com/mafredri/cdp/protocol/page"
    "github.com/mafredri/cdp/rpcc"
)

func main() {

    folderPtr := flag.String("folder", "", "folder path for screenshots: example c:\\temp\\screens\\")
    chromePtr := flag.String("chrome", "http://localhost:9222", "chrome connection - example: http://localhost:9222")
    
    widthPtr := flag.Int("width", 1280, "screencast width")
    heightPtr := flag.Int("height", 720, "screencast height")
    qualityPtr := flag.Int("quality", 100, "screencast quality")
    
    flag.Parse()

    if err := run(*folderPtr, *chromePtr, *widthPtr, *heightPtr, *qualityPtr); err != nil {
        panic(err)
    }
}

func run(folder string, chromeConnection string, width int, height int, quality int) error {
    ctx, cancel := context.WithCancel(context.TODO())
    defer cancel()
    
    chromePath := chromeConnection
    folderPath := folder

    devt := devtool.New(chromePath)

    pageTarget, err := devt.Get(ctx, devtool.Page)
    if err != nil {
        return err
    }

    conn, err := rpcc.DialContext(ctx, pageTarget.WebSocketDebuggerURL)
    if err != nil {
        return err
    }
    defer conn.Close()

    c := cdp.NewClient(conn)

    err = c.Page.Enable(ctx)
    if err != nil {
        return err
    }

    // Start listening to ScreencastFrame events.
    screencastFrame, err := c.Page.ScreencastFrame(ctx)
    if err != nil {
        return err
    }

    go func() {
        defer screencastFrame.Close()

        for {
            ev, err := screencastFrame.Recv()
            if err != nil {
                log.Printf("Failed to receive ScreencastFrame: %v", err)
                os.Exit(0)
            }
            log.Printf("Got frame with sessionID: %d: %+v", ev.SessionID, ev.Metadata)

            err = c.Page.ScreencastFrameAck(ctx, page.NewScreencastFrameAckArgs(ev.SessionID))
            if err != nil {
                log.Printf("Failed to ack ScreencastFrame: %v", err)
                os.Exit(0)
            }

            // Write to screencast_frame-[timestamp].png.
            name := fmt.Sprintf("screencast_frame-%d.png", ev.Metadata.Timestamp.Time().Unix())
            
            filePath := folderPath + name

            // Write the frame to file (without blocking).
            go func() {
                err = ioutil.WriteFile(filePath, ev.Data, 0644)
                if err != nil {
                    log.Printf("Failed to write ScreencastFrame to %q: %v", name, err)
                }
            }()
        }
    }()

    screencastArgs := page.NewStartScreencastArgs().
        SetQuality(quality).
        SetMaxWidth(width).
        SetMaxHeight(height).
        SetEveryNthFrame(1).
        SetFormat("png")
    err = c.Page.StartScreencast(ctx, screencastArgs)
    if err != nil {
        return err
    }

    // Random delay for our screencast.
    time.Sleep(600 * time.Second)

    err = c.Page.StopScreencast(ctx)
    if err != nil {
        return err
    }

    return nil
}

スクリーンショットを記録するための別のストリームが不要として破棄されました。作業アルゴリズムは次のとおりです。

  1. Selenium経由でChromeを起動する
  2. 組み立てられたバイナリからScreencastを開始します-Chromeに接続し、フレームのストリームを指定したフォルダーに保存し始めます
  3. テストの最後にChromeを閉じます-バイナリは自動的に閉じます
  4. テストが失敗した場合-ビデオを作成します
  5. フレームでフォルダーを掃除します

このアプローチは、実行時間の点で最良の結果をもたらしました（実際には遅延はありません）。さらに、彼はテストに関する最大の情報を提供しました（実際に失われたフレームはありません）。

短所

1.スクリーンキャストの解像度が低い。いくつかのストリームでテストを実行し、Chromeの解像度を2560 * 1440に設定すると、データ転送に割り当てられたバッファがオーバーフローします。

2.解像度が高くなると、CPUの負荷が高くなります。

その結果、スクリーンキャストでは1024 * 576の解像度を選択しました。この解像度では、テストは6スレッドで正常に動作し、プロセッサーは快適モード（6コアi7-5820）で動作しました。

動画を集めています


ビデオのフレームを収集するために残ります。これには、SharpAviライブラリを使用しました

隠しテキスト 
private void GenerateVideoFromScreens(string videoPath)
{
    try
    {
        var videoWriter = new AviWriter(videoPath) { FramesPerSecond = 1, EmitIndex1 = true };

        var videoStream = videoWriter.AddMotionJpegVideoStream(1024, 576);

        var screens = new DirectoryInfo(_framesFolderPath).GetFiles().OrderBy(f => f.CreationTimeUtc.Ticks).ToList();
        foreach (var screen in screens)
        {
            try
            {
                using (var bmp = new Bitmap(screen.FullName))
                {
                    var bits = bmp.LockBits(new Rectangle(0, 0, videoStream.Width, videoStream.Height), ImageLockMode.ReadOnly, PixelFormat.Format32bppRgb);
                    var videoFrame = new byte[videoStream.Width * videoStream.Height * 4];
                    Marshal.Copy(bits.Scan0, videoFrame, 0, videoFrame.Length);
                    bmp.UnlockBits(bits);

                    videoStream.WriteFrameAsync(
                        true,
                        videoFrame,
                        0,
                        videoFrame.Length).GetAwaiter().GetResult();
                }
            }
            catch(Exception ex)
            {
                // ignore all video related errors per frame
            }
        }

        videoWriter.Close();
    }
    catch
    {
        // ignore all video related errors per streams
    }
}


高級な写真


スクリーンキャストの解像度は1024 * 576と非常に小さいため、Chrome自体に小さな解像度を設定する必要があります。そうしないと、小さなテキストで問題が発生します。

Chrome 2560 * 1440-> 1024 * 576でのスクリーンキャスト=小さなテキストが実質的に読めない
Chrome 1920 * 1080-> 1024 * 576でのスクリーンキャスト*小さなテキストが読みづらい
Chrome 1408 * 792-> 1024 * 576 =小さなテキストが問題なく読み取られる

結果のビデオ1024 * 576は改善できます-PhotoSauceライブラリを使用してフレームを1920 * 1080に縮小した場合

隠しテキスト 
public Bitmap ResizeImage(Bitmap bitmap, int width)
{
	using (var inStream = new MemoryStream())
	{
		bitmap.Save(inStream, ImageFormat.Png);
		inStream.Position = 0;
		using (MemoryStream outStream = new MemoryStream())
		{
			var settings = new ProcessImageSettings { Width = width };
			MagicImageProcessor.ProcessImage(inStream, outStream, settings);
			return new Bitmap(outStream);
		}
	}
}
