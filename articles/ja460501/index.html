1.なぜBuildBotですか？


habr-eの前半で、BuildBot を使用した継続的インテグレーションの実装に関する記事に会いました。たとえば、これは私にとって最も有益であるように見えました。別の例- より単純です。これらの記事はで味付けすることができ、マニュアルからの例、およびこれは英語で、後に。クーペは良い出発点です。これらの記事を読んだら、BuildBotですぐに何かをしたいと思うでしょう。

やめる！誰かが自分のプロジェクトでそれを使ったことはありますか？はい、結局のところ、多くの人が自分のタスクにそれを適用しています。BuildBot の使用例を見つけることができますとGoogleコードアーカイブで。

では、ビルドボットを使用している人々のロジックは何ですか？結局のところ、他のツールがあります：CruiseControlとJenkinsです。私はこのように答えます。ほとんどのタスクでは、Jenkinsで十分です。次に、BuildBotはより適応性が高く、そこでのタスクはJenkinsと同じくらい簡単に解決されます。あなたを選びます。しかし、ターゲットプロジェクトを開発するためのツールを探しているので、簡単な手順から始めて、対話性と独自のインターフェイスを持つビルドシステムを取得できるツールを選択してみませんか。

ターゲットプロジェクトがpythonで記述されている場合は、「プロジェクトで使用されている言語に関して明確なインターフェースを持つ統合システムを選択しないのはなぜですか？」という疑問が生じます。次に、BuildBotの利点を紹介します。
それで、私たちの「楽器カルテット」。私自身、BuildBotの4つの機能を定義しました。
  1. これは、GPLに基づくオープンソースフレームワークです。
  2. これは、Pythonを構成ツールとして使用し、必要なアクションを説明しています。
  3. これは、組み立てが行われる機械から応答を受け取る機会です。
  4. これらは最終的にホストの最小要件です。デプロイにはpythonおよびtwistedが必要で、仮想マシンやJavaマシンは必要ありません。

2. BuildMasterが主導するコンセプト




タスク分散アーキテクチャの中心はBuildMasterです。次のようなサービスです。

  • プロジェクトソースツリーの変更を追跡する
  • Workerサービスがプロジェクトをビルドしてテストするために実行するコマンドを送信します
  • 実行されたアクションの結果をユーザーに通知します

BuildMasterは、master.cfgファイルを通じて構成されます。このファイルはBuildMasterのルートにあります。後で、このルートがどのように作成されるかを示します。master.cfgファイル自体には、Python（BuildBot呼び出しを使用するスクリプト）が含まれています。

次に重要なBuildBotはWorkerと呼ばれます。このサービスは、異なるOSの別のホスト、またはBuildMasterがある場所で実行できます。また、独自のパッケージと変数を使用して、特別に準備された仮想環境に存在することもできます。これらの仮想環境は、vertualenv、venvなどのPythonユーティリティを使用して準備できます。

ビルドマスターコマンドを各Workerにブロードキャストし、次にWorkerがそれらを実行します。つまり、プロジェクトのビルドとテストのプロセスは、Windowsを実行しているワーカーとLinuxを実行している別のワーカーに行くことができることがわかりました。プロジェクトのソースコードの

チェックアウトは、すべてのワーカーで行われます。

4.最初のステップ


BuildMaster を作成する時間です。それを/ home / habr / masterフォルダーに入れます。
mkdir master
buildbot create-master master #

次のステップ。ワーカーを作成します。それを/ home / habr / workerフォルダーに入れます。
mkdir worker
buildbot-worker create-worker --umask=0o22 --keepalive=60 worker localhost:4000 yourWorkerName password

Worker を起動すると、デフォルトで/ home / habr / workerに、master.cfgで指定されているプロジェクトの名前でフォルダーが作成されます。そして、プロジェクトの名前が付いたフォルダーに、ビルドディレクトリを作成し、そのディレクトリでチェックアウトを行います。Workerの作業ディレクトリは、/ home / habr / yourProject / buildディレクトリになります。

黄金の鍵
そして今、私が前の段落を書いたもののために、スクリプトが実行する権限を持っていないので、マスターがワーカーがこのディレクトリでリモートで行うことを要求するスクリプトは実行されません。状況を修正するには、キーが必要です--umask = 0o22。このディレクトリへの書き込みを禁止しますが、起動権は残します。そして必要なのはこれだけです。

BuildMasterとWorkerは、それらの間の接続を確立します。中断して、WorkerがBuildMasterからの応答をしばらく待機する場合があります。応答がない場合、接続が再開されます。キー--keepalive = 60は、接続が再起動するまでの時間を示すために必要なものです。

5.3 change_source


c['change_source'] = []
c['change_source'].append(changes.GitPoller(
                            'git://github.com/buildbot/hello-world.git',
                             workdir='gitpoller-workdir', branch='master',
                             pollInterval=300))

cディクショナリのchange_source キーを使用して、プロジェクトのソースコードでリポジトリをポーリングするオブジェクトを配置するリストにアクセスします。この例では、一定の周期でポーリングされるGitリポジトリを使用しています。

最初の引数はリポジトリへのパスです。

workdirは、Worker側のパスへのパスであり、/ home / habr / worker / yourProject / buildからの相対パスで、 gitはリポジトリのローカルバージョンを格納します。

ブランチには、監視する必要があるリポジトリ内の特定のブランチが含まれています。

pollIntervalには、BuildMasterがリポジトリの変更をポーリングするまでの秒数が含まれています。

プロジェクトリポジトリの変更を追跡する方法はいくつかあります。

最も簡単な方法はポーリングです。これは、BuildMasterがリポジトリを使用してサーバーを定期的にポーリングすることを意味します。コミットがリポジトリ内の変更を反映した場合、BuildMasterは少し遅れて内部Changeオブジェクトを作成し、それをSchedulerイベントハンドラーに送信します。これにより、Workerでプロジェクトをビルドしてテストする手順が開始されます。これらのステップの中で、リポジトリの更新が示されます。それは労働者にあります-eは、リポジトリのローカルコピーを作成します。このプロセスの詳細については、次の2つのセクション（5.4および5.5）で説明します。

リポジトリ内の変更を追跡するさらに洗練された方法は、プロジェクトのソースコードの変更についてメッセージが配置されているサーバーからBuildMasterに直接メッセージを送信することです。この場合、開発者がcommitを行うとすぐに、プロジェクトリポジトリを持つサーバーがBuildMasterにメッセージを送信します。そして、これはPBChangeSourceオブジェクトを作成することによってインターセプトされます。さらに、このオブジェクトはスケジューラに転送されます、プロジェクトをビルドしてテストする手順をアクティブにします。このメソッドの重要な部分は、リポジトリ内のサーバーフックスクリプトの操作です。でフックスクリプト中にアクションを処理するための責任がコミット、あなたが呼び出す必要がありsendchangeのユーティリティをとのネットワークアドレスを指定BuildMasterを。PBChangeSourceをリッスンするネットワークポートを指定する必要があります。ちなみに、PBChangeSourceはBuildMasterの一部です。この方法では、プロジェクトリポジトリが配置されているサーバーでadmin -a 特権が必要です。まず、バックアップリポジトリを作成する必要があります。

6.独自の構成の例


BuildBot
 を使用して実装することを提案するサンプルプロジェクトのアーキテクチャを次に示します。

バージョン管理システムとしてsvnを使用します。リポジトリ自体は特定のクラウドに配置されます。このクラウドのアドレスはsvn.host/svn/yourProject/trunkです。svnの下のクラウドには、ユーザー名：user、passwd：password accountがあります。build- a ステップであるスクリプトは、svnブランチの別のbuildbot / worker_linuxフォルダーにもあります。これらのスクリプトは、実行可能プロパティが格納されたリポジトリにあります。

BuildMasterおよびWorker1つのproject.hostホストで作業します。BuildMasterはそのファイルを/ home / habr / masterフォルダーに保存します。ワーカーは次のパス/ home / habr / workerを保存します。間の通信BuildMasterとワーカー処理がされている実行使用4000ポートを介してBuildbotのの -aのプロトコル、すなわち、「PB」プロトコル。

ターゲットプロジェクトはすべてPythonで記述されています。タスクは、変更の追跡、実行可能ファイルの作成、ドキュメントの生成、テストの実施です。失敗した場合、すべての開発者は、失敗したアクションがあることを示すメッセージをメールに送信する必要があります。

WebマッピングBuildBotproject.hostのポート80に接続します。 Apatchはオプションです。ツイストライブラリには既にWebサーバーがあり、BuildBotがそれを使用します。BuildBotの

それは多くの方法でメッセージを送ることができます。

MailNotifierはメールを使用して通知を送信します。

template_htmlは、ニュースレターのテキストテンプレートを設定します。マークアップを作成するには、htmlを使用します。jinja2エンジンによって変更されます（djangoと比較できます）。BuildBotには、メッセージのテキストを形成するプロセスでテンプレートに値が代入される一連の変数があります。これらの変数は、{{二重中括弧}}で表記されています。したがって、たとえば、概要には、完了した操作のステータス、つまり成功または失敗が表示されます。そしてプロジェクトはyourProjectを出力します。したがって、jinja2で制御コマンドを使用する、BuildBot変数、Python文字列フォーマッターを使用すると、非常に有益なメッセージを作成できます。

MailNotifierには、次の引数が含まれています。

fromaddr-誰もがニュースレターを受け取るアドレス。

sendToInterestedUsers = Trueは、コミットを行った所有者とユーザーにメッセージを送信します。

lookupは、ニュースレターを受け取るユーザーの名前に追加されるサフィックスです。したがって、ユーザーとしての管理者は、admin @ your.domainでニュースレターを受信します。

relayhostは、ホストの名前を設定しますSMTPがあるオープンし、そしてsmptPortは、ポート番号を設定するSMTPリッスンへサーバ。

mode =“ warning”は、失敗または警告ステータスで終了したビルドステップが少なくとも1つある場合にのみメールを送信する必要があることを示します。成功した場合、郵送は必要ありません。

extraRecipientsには、所有者とコミットを行った人に加えて、メールを送信する必要がある個人のリストが含まれています。

messageFormatterは、メッセージのフォーマット、そのテンプレート、およびjinja2から利用可能な変数のセットを定義するオブジェクトです。wantProperties = TrueやwantSteps = Trueなどのパラメーターは、この使用可能な変数のセットを指定します。

['services'] = [sendMessageToAll]サービスのリストを提供します。その中にはレポーターも含まれます。
