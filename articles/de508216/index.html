Was ist CI (Continuous Integration)

CI (Continuous Integration) - wörtlich übersetzt als "Continuous Integration". Dies bezieht sich auf die Integration einzelner Teile des Anwendungscodes untereinander. Je öfter wir den Code zusammen sammeln und prüfen:



  • Wird er gehen?
  • Bestehen Autotests?


Umso besser! Mit CI können Sie solche Überprüfungen automatisch durchführen. Es wird in fortgeschrittenen Entwicklungsteams verwendet, die nicht nur Code, sondern auch Autotests schreiben. Er wird bei Interviews gefragt - zumindest um zu verstehen, was es ist. Ja, auch mit Testern.



Deshalb werde ich in einem Artikel darüber erzählen, was es ist. Wie CI funktioniert und wie es für Ihr Projekt nützlich sein wird.

Was ist CI?



CI ist das Zusammenstellen, Bereitstellen und Testen einer Anwendung ohne menschliches Eingreifen. Jetzt werde ich mit einem Beispiel erklären.



Nehmen wir an, wir haben zwei Entwickler - Masha und Vanya. Und Tester Katya.



Mascha schreibt den Code. Fügt es dem Versionskontrollsystem hinzu (vom englischen Versionskontrollsystem VCS ). Dies ist so etwas wie eine Dropbox für Code - ein Speicherort, in dem alle Änderungen gespeichert werden und Sie jederzeit sehen können, wer was wann geändert hat.



Dann beendet Wanja seine Funktionalität. Und speichert auch den Code in VCS.







Dies ist jedoch nur der Quellcode - eine Reihe von Dateien mit der Erweiterung .java oder einer anderen. Damit Katya die Änderungen testen kann, die Sie benötigen:



  1. Erstellen Sie einen Build aus dem Quellcode
  2. Führen Sie es auf einem Testcomputer aus


Das Erstellen eines Builds erfolgt, wenn eine ausführbare Datei aus einer Reihe von Quellcodedateien erstellt wird: Sie







können einen Build manuell erstellen. Dies ist jedoch eine unnötige Hämorrhoide. Denken Sie daran, dass Sie in welcher Reihenfolge, welche Dateien voneinander abhängen, keinen Fehler im Befehl machen ... Normalerweise verwenden sie einen speziellen das Programm. Für Java ist dies Ant, Maven oder Gradle. Mit Hilfe des Collectors richten Sie den Erstellungsprozess einmal ein und führen ihn dann mit einem Befehl aus. Startbeispiel für Maven: 



mvn clean install


Dies ist eine Halbautomatisierung - Sie benötigen noch eine Person, die in ein Team eintritt und den Build mit „Stiften“ zusammenbaut. Nehmen wir an, Entwickler tun dies. Wenn Katya den Build zum Testen auffordert, aktualisiert Vanya die Version aus dem Repository und sammelt den Build.







Aber bauen Sie einen Build - holen Sie sich eine App zum Testen. Es muss noch gestartet werden! Dies erfolgt durch den Anwendungsserver. Server sind unterschiedlich: Wildfly, Apache, Jetty ...



Wenn dies Wildfly ist, dann brauchen Sie:



  1. Fügen Sie den Build dem Standalone- / Bereitstellungsverzeichnis hinzu
  2. Starten Sie den Server (nachdem Sie den Dienst einmal eingerichtet haben).


Und das ist wieder Halbautomatisierung. Weil der Entwickler das resultierende Archiv nach der Montage auf den Prüfstand kopieren und den Service aktivieren muss. Ja, dies wird von ein paar Teams durchgeführt, aber immer noch von einer Person.







Aber wenn wir eine Person aus diesem Schema entfernen, erhalten wir CI!







CI ist eine Anwendung, mit der Sie den gesamten Prozess automatisieren können. Änderungen werden mit Code aus dem Repository abgerufen. Von selbst! Es gibt zwei Konfigurationsoptionen:



  • CI fragt das Repository ab "Hey Kuckuck, hast du irgendwelche Änderungen?" alle N Stunden / Minuten, wie zu konfigurieren.



  • Das Repository winkt CI mit der Hand, wenn es festlegt: „Hey, hallo! Und ich habe hier ein Update! “ (Ist es ein Git-Hook oder ein Äquivalent in Ihrem VCS?)







Wenn das CI die Änderungen empfängt, beginnt es mit dem Erstellen des Builds und den Autotests.



Wenn die Montage fehlgeschlagen ist (die Tests fielen oder das Projekt nicht zusammengebaut werden konnte), schreibt das System eine E-Mail an alle interessierten Parteien:



  • Projektmanager (um zu wissen, was getan wird!)
  • Der Entwickler, der die Änderungen vorgenommen hat
  • Alle anderen - wenn Sie es einrichten, wird es so sein.








Wenn der Build erfolgreich ist, stellt CI die Anwendung auf dem Testcomputer bereit. Und am Ende kann Katka die neue Baugruppe testen!







Ja, natürlich, wenn Sie alles eingerichtet haben - teilen Sie dem CI-Server mit, woher die Änderungen stammen, welche Autotests ausgeführt werden sollen, wie das Projekt erstellt werden soll, wo es später erstellt werden soll ... Aber dann habe ich es einmal eingerichtet - und dann!



Autotests müssen auch von uns selbst geschrieben werden, aber gut gemacht =)







Wenn es an Ihren Fingern liegt, ist das CI- System ( Continuous Integration ) eine Art Programm, das Ihre Quellcodeverwaltung überwacht. Wenn Änderungen dort angezeigt werden, werden sie automatisch zusammengeführt, erstellt und automatisch getestet (natürlich, wenn sie geschrieben wurden).



Im Falle eines Misserfolgs wird dies allen Interessenten in erster Linie bis zum letzten Committer mitgeteilt. (c) habr.com/de/post/352282






CI-Programme



Am beliebtesten sind Jenkins und TeamCity.



Es gibt jedoch eine Reihe anderer Varianten: CruiseControl, CruiseControl.Net, Atlassian Bamboo, Hudson und Microsoft Team Foundation Server.





Wie sieht es aus



Mal sehen, wie es aus Sicht des Benutzers aussieht. Ich werde mit einem Beispiel des TeamCity-Systems zeigen.



Wenn ich mich anmelde, sehe ich alle Aufgaben. Die Aufgaben sind unterschiedlich:



  • Build Build
  • Auto-Tests fahren
  • Stellen Sie die Anwendung auf dem Prüfstand bereit
  • Führen Sie an diesem Stand GUI-Tests (oder Postman-Tests) durch
  • Benachrichtigen Sie alle Interessierten per E-Mail über die Ergebnisse der Montage und Prüfung


Aufgaben können gruppiert werden. Hier haben wir zum Beispiel ein CDI-Projekt. Wenn ich hineingehe, sehe ich die Aufgaben für dieses spezielle Projekt:







  • CDI Archetype und CDI Core sind Builds. Sie überprüfen, ob die Anwendung im Allgemeinen erstellt wird. Trainieren Sie in ein paar Minuten und führen Sie jede Codeänderung aus.
  • CDI Core mit Tests - Zusammenstellung des Projekts mit allen Autotests, die, wie auf dem Bildschirm zu sehen ist, über 4000 Teile umfassen. Tests dauern eine halbe Stunde, werden aber auch für jedes Commit ausgeführt.

Wie CI funktioniert



Wie und wo sammelt CI Builds und führt automatische Tests durch? Ich werde TeamCity als Beispiel verwenden, aber andere Systeme funktionieren ähnlich.



TeamCity selbst sammelt nichts. Das Zusammenstellen und Ausführen von Autotests erfolgt auf anderen Computern, die als "Agenten" bezeichnet werden:







"Agent" ist ein einfacher Computer. Ein Stück Eisen oder eine virtuelle Maschine, nicht der Punkt. Aber wie versteht dieser Computer, was er tun muss?



TeamCity hat einen Server und einen Client. Der Server ist genau die Anwendung, in der Sie dann Tasten drücken und ein schönes Bild "wie alles gut gelaufen ist" sehen. Es ist auf einem Computer installiert.







Und die "Client" -Anwendung ist auf den "Agent" -Maschinen installiert. Und wenn wir auf dem Server auf die Schaltfläche "Ausführen" klicken:







Der Server wählt einen freien Client aus und gibt ihm alle Anweisungen: Was genau muss getan werden? Der Client sammelt den Build, führt Autotests durch, sammelt das Ergebnis und kehrt zum Server zurück: "On, hold, draw".







Der Server zeigt dem Benutzer das Ergebnis an und sendet eine E-Mail an alle Interessenten.



Gleichzeitig sehen wir immer, auf welchem ​​bestimmten Agenten die Assembly stattgefunden hat:







Und Sie können selbst auswählen, wo die automatischen Tests ausgeführt werden sollen. Weil es vorkommt, dass Autotests nur auf einen Build-Agenten fallen. Dies bedeutet, dass er etwas mit der Konfiguration nicht stimmt.

Wir können auch für jede Assembly eine Liste von Agenten konfigurieren, auf denen sie ausgeführt werden kann. Wozu? Zum Beispiel haben wir die Hälfte der Linux-Agenten und die Hälfte der Windows-Agenten. Und wir haben die Montage nur für ein System durchgeführt. Oder unter Windows kriecht eine Art schwebender Fehler heraus, aber das Beheben ist lang und teuer, und der gesamte Client unter Linux - nun, warum dann?



Und es kommt auch vor, dass Agenten zwischen Projekten aufgeteilt werden, so dass es keinen Kampf gibt - dieses Projekt verwendet Burana und Apollo und Chip und Dale. Die Sache ist, dass nur eine Aufgabe auf einem Agenten ausgeführt werden kann. Daher macht es keinen Sinn, ein cooles Auto für einen Agenten zu kaufen. Sofort können Sie dort ohnehin keine Tests durchführen.



Am Ende, wie es funktioniert: Zuerst kauft der Administrator Computer unter den "Agenten" und installiert die TeamCity-Clientanwendung auf diesen. Sie sollten nicht zu cool sein, da sie nicht viele Aufgaben gleichzeitig erledigen.







In diesem Fall zahlen Sie in TeamCity für die Anzahl der Lizenzen für Build-Agenten. Je weniger sie sind, desto besser.



Auf einem separaten Computer installiert der Administrator den TeamCity-Server. Und konfiguriert es - richtet Baugruppen ein, gibt an, welche Baugruppen auf welchen Maschinen angesteuert werden können usw. Der Server benötigt einen Speicherort für Artefakte - die Ergebnisse der Assembly.



Wir haben zwei Projekte - Single Client und Factor, die miteinander interagieren. Der Einzelclient-Tester erfasst den Faktor möglicherweise nicht lokal. Er startet den Build in TeamCity und lädt den fertigen Build von Artefakten herunter!


Außerdem wählt der Entwickler aus, welche Assembly er ausführen möchte, und klickt auf "Ausführen". Was passiert in diesem Moment:



1. Der TeamCity-Server überprüft die Liste, auf welchen Agenten diese Assembly ausgeführt werden kann. Dann prüft er, welcher dieser Agenten derzeit frei ist:







Frei gefunden? Ich habe ihm das Rätsel gegeben!







Wenn alle Agenten beschäftigt sind, wird die Aufgabe in die Warteschlange gestellt. Die Warteschlange arbeitet nach dem FIFO-Prinzip - first in, first out. Wer stand zuerst auf - das und Hausschuhe.







Die Warteschlange kann manuell angepasst werden. Wenn ich also sehe, dass die Warteschlange voll von Assemblys ist, die vom Versionskontrollsystem gestartet wurden, werde ich meine ganz nach oben heben. Wenn ich sehe, dass Leute Versammlungen gestartet haben, bedeutet das, dass sie auch wichtig sind. Sie müssen warten.







Dies ist üblich, wenn die Agentenkapazitäten nicht durchgehend ausreichen und eine Warteschlange erstellt wird. Sie sehen, wer es gestartet hat:



  • Roboter? Dies ist also nur eine Routineprüfung, um festzustellen, ob nichts mehr kaputt ist. Dies kann 5-10-30 Minuten warten, das ist okay
  • Kollege? Diese Versammlung ist ihm wichtig, da er nicht auf den geplanten Start gewartet hat. Wir stehen in der Schlange, Sie sollten nicht vorwärts gehen.


Manchmal können Sie den Build sogar vom Versionskontrollsystem abbrechen, wenn es sehr heiß ist und alle Agenten mit stündlichen Tests beschäftigt sind. In diesem Fall können Sie:



  • Erhöhen Sie Ihren Zug ganz nach oben, damit er beim ersten freien Agenten beginnt
  • Melden Sie sich beim Agenten an und brechen Sie den aktuellen Build ab
  • starte sie neu! Obwohl es ganz unten in der Zeile steht, ist es nur hässlich, die Baugruppe abzubrechen


2. Der Agent führt die Aufgabe aus und gibt das Ergebnis an den Server zurück.







3. Der Server zeichnet das Ergebnis in der grafischen Oberfläche und speichert die Artefakte. So kann ich in TeamCity die vollständigen Protokolle der vergangenen Autotests in den Artefakten anzeigen oder die Projektassembly herunterladen, um sie lokal bereitzustellen.



Es wird dringend empfohlen, die Anzahl der von CI gespeicherten Assemblys vorkonfigurieren. Wenn die Artefakte Builds mit mehr als 200 MB enthalten und viele davon vorhanden sind,







schlägt der nächste Start der Assembly mit dem Fehler " Nicht genügend Speicherplatz" fehl: 4. Der Server führt das E-Mail-Versenden durch - so richten Sie es ein. Er kann einen positiven Newsletter verfassen, "die Montage war erfolgreich", und er kann nur im Falle eines Fehlers "Oh, sie, etwas ist schief gelaufen!" E-Mails senden.





Integration mit VCS



Ich habe über verschiedene Optionen zum Konfigurieren der CI-VCS-Integration gesprochen:



  • CI fragt das Repository ab "Hey Kuckuck, hast du irgendwelche Änderungen?" alle N Stunden / Minuten, wie zu konfigurieren.



  • Das Repository winkt CI mit der Hand, wenn es festlegt: „Hey, hallo! Und ich habe hier ein Update! “ (Ist es ein Git-Hook oder ein Äquivalent in Ihrem VCS?)







Aber wann wird welche verwendet?



Das Beste ist natürlich, dass ein Versionskontrollsystem den CI-Server benachrichtigt. Führen Sie für jede Änderung den gesamten Zyklus aus: Zusammenbauen, Testen, Bereitstellen. Jede Änderung des Codes wird sofort in die Testumgebung übernommen, die so relevant wie möglich ist.







Außerdem werden bei jeder Änderung Autotests ausgeführt. Und wenn die Tests fehlschlagen, ist sofort klar, wessen Commit sie gebrochen hat. Immerhin hat es funktioniert und nach Vasyas Änderungen ist es plötzlich kaputt gegangen - was bedeutet, dass sein Engagement zu einem Sturz geführt hat. Mit seltenen Ausnahmen, wenn der Herbst schwimmt.







Im wirklichen Leben ist ein solches Schema jedoch selten anwendbar. Denken Sie nur - Sie können viele Projekte haben, viele Entwickler. Jeder begeht mindestens einmal pro halbe Stunde etwas. Und wenn Sie für jedes Commit eine halbe Stunde lang 10 Assemblys ausführen, werden die Warteschlangen in TeamCity niemals geharkt!



Eines unserer Produkte verfügt über ein Kernmodul und es gibt mehr als 15 Kunden. Jeder hat seine eigenen Autotests. Kundenmontage ist Kern + Kundenmerkmale. Das heißt, eine Änderung im Stammprojekt kann 15 verschiedene Assemblys betreffen. Sie müssen also alle ausgeführt werden, wenn Sie sich im Kern festlegen.



Als wir 4 Build-Agenten hatten, wurden am Dienstagabend alle Builds und Tests für diese Kunden gestartet. Und um 10 Uhr waren noch ein paar Stunden in TeamCity.


Eine andere Möglichkeit ist der Kauf vieler Agenten. Dies ist jedoch der Preis für das Auto selbst + für die Lizenz in TeamCity, die bereits viel teurer ist und sogar jeden Monat bezahlt wird.



Daher gehen sie normalerweise wie



folgt vor : 1. Sehr schnelle und wichtige Baugruppen können jedem Commit überlassen werden. Wenn es 1-2 Minuten dauert, lassen Sie sie jagen.







2. Der Rest der Builds prüft, ob Änderungen am VCS vorgenommen wurden - beispielsweise alle 15 Minuten. Wenn ja, dann starten wir.







3. Lange Tests (z. B. Leistungstests) - einmal alle paar Tage in der Nacht.











CI beim Testen



Wenn wir über die Entwicklung unserer Anwendung sprechen, sind Tests im Standardzyklus enthalten. Sie oder Ihre Entwickler schreiben Autotests, die dann von CI gesteuert werden. Dies können Unit-, API-, GUI- oder Stresstests sein.







Aber was ist, wenn Sie eine Black Box testen? Es gibt eine Anwendung, keinen Quellcode. Dies sind die harten Realitäten von Integrationstestern - der Lieferant gibt Ihnen eine neue Anwendungsversion, die Sie überprüfen müssen, bevor Sie sie in Produktion nehmen.



Angenommen, Sie haben API-Tests in Postman. Oder GUI-Tests in Selen. Kann ich die CI-Schleife für sie anpassen?







Natürlich kannst du!



CI legt keinen strengen Rahmen fest, z. B. "Ich arbeite nur in Projekten mit automatischen Tests" oder "Ich arbeite nur, wenn ich Zugriff auf den Quellcode habe". Es kann in das Versionskontrollsystem schauen, oder es kann nicht schauen. Dies ist eine optionale Bedingung!



Autotests geschrieben? Sagen Sie dem CI-Server, wie er sie häufig ausführen soll - und genießen Sie das Ergebnis =)









Gesamt



CI - kontinuierliche Integration. Dies ist der Zeitpunkt, an dem Ihre Anwendung ständig getestet wird: Ist alles gut damit? Bestehen die Tests? Geht die Versammlung? Darüber hinaus werden alle Überprüfungen automatisch und ohne menschliches Eingreifen durchgeführt.



Dies ist besonders wichtig für Teams, in denen mehrere Entwickler am Code einer Anwendung arbeiten. Wie kommt es dazu? Separat funktionieren Teile des Programms, aber nicht mehr zusammen. Mit CI können Sie solche Probleme sehr schnell erkennen. Und je schneller Sie es finden, desto billiger ist es, es zu reparieren.







Daher der Name - eine ständige Überprüfung der Integration von Codeteilen untereinander.

Typische CI-Aufgaben:



  • Überprüfen Sie, ob der Code aktualisiert wurde
  • Build Build
  • Auto-Tests fahren
  • Stellen Sie die Anwendung auf dem Prüfstand bereit
  • Führen Sie an diesem Stand GUI-Tests (oder Postman-Tests) durch
  • Benachrichtigen Sie alle Interessierten per E-Mail über die Ergebnisse der Montage und Prüfung








Und das alles - automatisch, ohne menschliches Eingreifen! Das heißt, sie haben es einmal eingerichtet und dann selbst.



Wenn CI im Projekt konfiguriert ist, verfügen Sie über ständig aktuelle Prüfstände. Und wenn etwas im Code kaputt geht, wissen Sie sofort Bescheid, der CI-Server sendet einen Brief. Und Sie können in die grafische Oberfläche gehen und sehen, ob alle Builds erfolgreich sind und die Tests grün sind? Bewerten Sie das Bild des Projekts in einer Minute.
