Visual Basicの興亡

自白する必要があります。 C＃やJava（そして私たちがJavaScriptとして知っている猛烈な混乱）のような現代の言語を扱う立派な開発者になる前、私は非常に人気があり不器用なVisual Basicの熱狂的なファンでした。

私の愛は誠実でした。子供の頃、BASICでプログラミングすることを学びました。しかし、BASICだけではありません。私は、古代のDOSオペレーティングシステムで、QuickBASICと呼ばれるすばらしいMicrosoft環境から始めました。私はまだ彼女の元気な青い背景に白いテキストを入力したことを覚えています。


1988年のプログラミングにとって、QuickBASICは魔法のようなものでした。面倒な行番号なしでコードを記述し、入力中に構文エラーをキャッチし、コマンドラインに煩わされることなく開発環境から直接プログラムを実行することが可能でした。プログラムを終了したら、次の人たちの助けを借りて、プログラムを友達と共有できます。


現在、QuickBASICは、はるか昔のギミックです。そうですか？QB64と呼ばれる革新的なプロジェクトにより、最先端のQuickBASICレプリカが作成されました。Windows、MacOS、Linuxで動作し、エミュレーターは必要ありません。プログラムがQB64で起動すると、環境はトリッキートリックを実行します。最初にBASICコードをC ++に変換し、次にそれをコンパイルします。

クラシックVBとビジュアルエイジ


QuickBASICは、テキストベースのコンピュータシステムの遠い日々にとって印象的な媒体でした。しかし、MicrosoftがWindowsをリリースし、PCユーザーをボタンとクリックのグラフィックの世界に押し込んだとき、すべてが変わりました。Microsoft Windows 3.0（真の成功を収めた最初のバージョン）のリリースと共に、Visual Basic 1.0がリリースされました。

まったく新しいものでした。ユーザーはプログラムのボタンを作成し、アーティストのキャンバスのようにウィンドウの表面にそれらを描画できます。ボタンに何らかのアクションを実行させるには、開発環境でボタンをダブルクリックしてコードを記述するだけで十分です。そして、コードは、クラスのヒープ、複雑なメモリ管理、および不明瞭なWindows API呼び出しを伴う、暗号化されたC ++で書かれていませんでした。あなたは文明人として、親しみやすいVBにコードを書きました。


これらのグラフィカルガジェットはすべて印象的でしたが、VBの成功の真の秘密はその実用性にありました。開発者が完全なインターフェイスをスケッチし、VBのように高速にコーディングを開始できるツールは他にありませんでした。そして歴史家はVisual Basicの視覚的な部分について推測するのが大好きですが、彼の比類のないスキルはグラフィックウィジェットとは何の関係もありませんでした。 VBは、開発者がプロ​​グラムを実行し、エラーを見つけて修正し、新しいコードで作業を続けることができる伝説的なエディットコンティニュー機能のおかげで有名になりました。これは、人類に知られている他のほとんどすべてのプログラミング環境とは非常に異なり、開発者は作業を再度コンパイルして、変更のたびにやり直す必要がありました。

元のVisual Basicは約12年間栄えていました。初心者にとってフレンドリーな環境として登場し、本格的なプログラマーにとって十分な機能を備えたツールになりました。クラシックVisual Basicの最新バージョンであるVB 6がリリースされるまでに、世界中の過酷なC ++の約10倍のVBエンコーダーが存在していました。また、おもちゃのアプリのモックアップを作成しただけではありません。 Visual Basicは、会社のオフィス、さらにはASP（Active Server Pages）を使用したインターネットにも普及しました。そのおかげで、VBコンポーネントと通信したり、データベースと通信したり、HTMLをオンザフライで書き込んだりできるWebページを作成できました。

あなたがそれを自分で作成することを決めない限り、これは事実上構造なしで起こりました。これはVisual Basicでの不文律でした-後悔する自由を与えてくれました。

クラシックVBの問題


多くのサークルで、クラシックVBの評判は悪かった。 OOPマニアは、VBに継承のサポートが欠けていると不満を言うことがよくあります。 （これは奇妙な不満でした。継承は、経験の浅い開発者が足で撃つための優れた方法であることが多いためです-これはVBに必要なものではありません。）実際、クラシックVBはオブジェクト指向に傾倒していませんでした。彼の人生の終わりに向かって、彼はすでにインターフェイス、ポリモーフィズム、およびクラスライブラリをサポートしていました。これはすべて、COMから借用されたもので、コンポーネントテクノロジーの基本部分であり、Windowsの各バージョンで「フラッシュ」されています。

古典的なVBでの本当の問題は、それがあったということでした、あまりにも成功した。彼は効果的に新しいプログラマーの障壁を減らし、ほとんど誰もがそこで働くことができました。のんきな初心者、疲れた会社の従業員、学生は、他のプラットフォームでははるかに複雑な問題を解決し、どこにでもスパゲッティコードをぶら下げました。

つまり、VBは使いやすいので、プログラマーとして勉強しなくても、プログラマーとしての知識がない人でも利用されていました。そしてVBは彼らの悪い習慣の根づきを防ぐために何もしませんでした。

VB.Fredと.NETプラットフォーム


2002年に進みましょう。マイクロソフトは、マイクロソフトが最善を尽くすことに取り組んでいます。これは、まったく新しい作業方法を導入することで、開発者の独自のエコシステムを破壊します。

今回、このプロジェクトは、Windows（および間接的にはクラシックVB）をサポートするコンポーネントテクノロジであるCOMのアーキテクチャの大きな変化でした。これは、そのような危険な変更をすでに何度か行った巨石企業にとってのオールインベットでした。


Microsoftは、当時のインターネット技術が新鮮で刺激的なものであったこと、および.NETにMicrosoftのWebサービス機能が含まれていたことなど、多くの疑わしい理由により、これを.NETプラットフォームを「再起動」と呼んでいましたしっかりと宣伝することにしました。この製品はJavaの強力な影響下で作成され、データベースとの通信、Webサイトの作成、マルチスレッドプログラムの作成、ソケットを介した接続の確立など、ビジネス開発者が想像できるほとんどすべてのアプリケーションのためのツールがすべて揃っていました。

唯一の問題は、これらすべての機能を実装するために、MicrosoftがほとんどすべてのクラシックVBを削除しなければならないことでした。


目を細めると、VB.NETと呼ばれる新しいバージョンはほとんど同じに見えました。しかし、そこには多くの大小の革命的変化が見られました。大規模：VBの古いプログラムは古いプログラムの世界に閉じ込められたままで、驚くほど弱い移行ウィザードではほとんど修正できませんでした。小：古典的なVBプログラマーは、配列要素をカウントする新しい方法に慣れる必要がありました。彼らはもはや普通の人々のように1から始めることができませんでした。今、彼らは公式のプログラマーのように、0から始めなければなりませんでした。

もう1つの大きな変更：編集および継続機能がなくなりました。

開発者は文句を言うのが大好きで、VB開発者はこれで特に知られています。ほとんどすぐに、影響力のあるVB開発者のグループがMicrosoftの新しいプログラミング環境VB.Fredに吹き替え、この新しい言語はVisual Basicではないという事実を強調しました。

Visual Basicが死ぬ運命にあるもの


.NETの変更により、VBファウンデーションに亀裂が発生し、避けられない低下につながったと考えられます。しかし、すべてが異なっていました。実際、VB.NETは新しい方向に進み、重要な変更を加えて、クラシックVBの「孤立した」絶対に優れたコードにしたにもかかわらず、非常に人気を博しています。これは、VB.NETが従来のVBでは開発者がこれまで達成できなかったものを提供したという事実が原因で発生しました。

.NETの世界では、VB言語とC＃言語の重みは同じです。 VBの各コード行は、C＃の同等のコード行に変換でき、その逆も可能です。両方の言語は同じ機能を持ち、同じコンポーネントを使用し、まったく同じ方法で（中間言語と呼ばれるものに）コンパイルします）VBはついに醜いアヒルの子症候群から解放されました。

しかし、他の立派なプログラミング言語に変えることは問題を引き起こしました。VBをこのような大勢の人々の好きな言語にした熱意は消え去りました。VBが変更されたためではなく、C＃が変更されたためです。

VBがC＃と同じパワーを獲得するとすぐに、C＃もVisual Basicの便利さを採用しました。たとえば、タイプセーフティ機能と.NETメモリ管理の存在により、C＃開発者はVB開発者のようにメモリリークを心配する必要がなくなりました。

言い換えれば、C＃は、アマチュア、学生、初心者を保護し、その力を弱めることのない「セーフティネット」を備えています。突然、VBは特別なものではなくなりました。これは、一連の熟練したプログラマーからの単なる別のツールになりました。

Visual Basicの現在の状態


今日、Visual Basicは奇妙な状態にあります。プロの開発者の間では、約0％の注意が必要です。VB は、プロの開発者の調査やGitHubリポジトリにも表示されません。しかし、彼はまだ生きています-彼はOfficeマクロ、古いAccessデータベース、古代のASP Webページを管理しており、また.NETに新しい人を惹きつけています。検索結果によって言語の人気度を測定するTIOBEインデックスは、VBを最も議論されている5つの言語のリストに入れています。

しかし、最近、言語開発の慣性がなくなったようです。2017年、マイクロソフトはこれにより、Visual Basicでは表示されない可能性のある新しい言語機能がC＃に追加されます。この変更はVBに醜いアヒルの子のステータスを返しませんが、.NETの一部としてのステータスに悪影響を与えます。

率直に言って、VBを傍観する傾向は数年前に始まりました。真面目な開発者は、.NETの主要部分がC＃で記述されていることを知っています。彼らは、C＃がプレゼンテーション、本、コース、ワークショップに適した言語であることを知っています。 VBを話したい場合、これは作成したアプリケーションに害を及ぼすことはありませんが、他の開発者と通信する機能を制限する可能性があります。

Visual Basicが自然に感じるニッチの1つは、教育市場です。しかし、そこにも黒い羊が残っています。 C＃やPythonなどの現代の言語はシンプルで安全であり、最初に学習する言語として選択されます。子供向けにもっとシンプルなものが必要な場合は、Scratchなどのグラフィカルプログラミングツールで市場がいっぱいです。中かっこで囲まれた言語も魅力的です：C＃、C、JavaScript、Java。それらは同様の構文を持ち、それらの1つを学んだ人は他のものに自信を感じるでしょう。

マイクロソフトはWeb開発の分野で、機会を逃していないことを模索しています。 VB 6と同じくらい使いやすく、JavaScriptでコンパイルしてHTMLフォームデザイナーと組み合わせることができるバージョンのVBに魅了されないのは誰ですか？はい、そのようなツールで新しいGoogleマップを作成することはできませんが、面倒なJavaScriptを使用せずにビジネス開発者、学生、アマチュアが簡単なオンラインアプリケーションを作成できるようにすることで、Visual Basicの魅力を取り戻すことができます。代わりに、Microsoftは、LightSwitchと呼ばれるVBベースの製品を作成しましたが、これはSilverlightブラウザープラグインの終了とともに姿を消しました。 Microsoftがさらに別の「大衆向けのコーディング」クラス製品の作成を開始した場合、それはおそらく次のような少量のコードを含むテンプレート駆動型ツールになるでしょうPowerApps。

Visual Basicは以前、さまざまな危険に直面していました。しかし今回は、すべてが違うようです。太陽が実際に世界で最も人気のあるプログラミング言語の1つに沈んでいるようです。trueの場合、Visual Basicは数十年にわたって衰退しません。それは別のレガシー製品となり、ユーザーや将来への愛情のない、過小評価されたツールになります。私たちが特別な何かを失ったのか、苦痛のあるテクノロジーを苦痛から救ったのかは、あなた次第です。
