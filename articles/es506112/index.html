Cómo visualizar el gasto diario en soluciones en la nube de GCP


A los clientes no les gusta pagar más de lo planeado : una justificación detallada de los costos es una parte integral e importante de la implementación de tecnologías en la nube.

Google Cloud Platform proporciona varios planes de precios para los recursos utilizados. Por ejemplo, el costo de GCE depende de la configuración de la computadora ( CPU , memoria, módulos de red, discos duros). Los costos de Google Kubernetes Engine (GKE) y Google Cloud Dataproc se basan en todos los nodos que se ejecutan en Google Compute Engine (GCE). Los costos restantes se pueden calcular mediante una fórmula compleja e intrincada. La planificación de un presupuesto es cada vez más difícil, especialmente si utiliza varias tecnologías en la nube. El monitoreo y los informes oportunos se vuelven aún más valiosos a medida que aumenta el gasto en infraestructura.

La posibilidad de verificar diariamente los informes de gastos también permitirá un ajuste oportuno de las capacidades distribuidas, y la factura final a fin de mes no causará sorpresa.

Antecedentes


Los proyectos a menudo necesitan informes extensos para mejorar la eficiencia y rastrear los costos. Aunque GCP le permite exportar datos de gastos a BigQuery para un análisis más exhaustivo, el procesamiento manual de BigQuery a menudo no es suficiente. Idealmente, deberíamos poder visualizar datos, filtrarlos y acumularlos según parámetros adicionales.

Filtros comunes:

  • Tipo de recurso;
  • Un período de tiempo;
  • Departamento y equipo;
  • Comparación con períodos anteriores.

La primera solución, que se basa en la recomendación de Google, es utilizar Google Data Studio . La implementación es muy simple: solo necesita configurar la fuente de datos. Los informes de muestra y un panel de control están preinstalados, pero la solución en sí no es lo suficientemente flexible. Al crear cuadros y gráficos en Google Data Studio , no puede ingresar una fórmula, debe seleccionar todos los parámetros manualmente.

Grafana es una herramienta simple y comprensible para monitorear y analizar datos. El esta bien establecido. Tal tablero sería ideal para visualizar datos de pago. La pregunta sigue siendo cómo conectar la interfaz web a BigQuery (BQ) . BQ plug-in, código abierto, con errores. Como resultado, el grupo de BQ-Grafana no es lo suficientemente estable. Además, hay otro punto inconveniente: las consultas BQ devuelven datos durante demasiado tiempo. La carga de

datos a PostgreSQL a través de CloudSQL resultó ser una buena solución . Este DBMS tiene el complemento oficial de PostgreSQL para Grafana . Y los resultados de las pruebas posteriores mostraron que el método seleccionado tiene claras ventajas en velocidad.

Resumen de la solución


El flujo de trabajo se puede describir esquemáticamente de la siguiente manera. La herramienta de tarea pendiente en el clúster de Kubernetes ejecuta una tarea en Cloud Dataflow cada 4 horas . Esta tarea, a su vez, inicia el proceso de carga de datos de BigQuery a PostgreSQL . La descarga se realiza solo de los datos que se encontraron después de la última exportación de BQ . Después de eso, los últimos datos se pueden ver en Grafana , conectado a PostgreSQL .

Flujo de datos


Creamos una cuenta con acceso a DataFlow y BigQuery . Esto es para garantizar que las tareas de DataFlow puedan recibir datos de BigQuery .

export  project=myproject
gcloud iam service-accounts create "bq-to-sql-dataflow" --project ${project}
 
gcloud projects add-iam-policy-binding ${project} \
--member serviceAccount:"bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--role roles/dataflow.admin
 
gcloud projects add-iam-policy-binding ${project} \
--member serviceAccount:"bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--role roles/bigquery.dataViewer


Para las tareas de DataFlow , debe crear dos contenedores: uno para temporal y otro para salida.

gsutil mb gs://some-bucket-staging
gsutil mb gs://some-bucket-temp

Script para cargar datos


Se requiere un script para descargar datos de BigQuery a CloudSQL . Cloud DataFlow es compatible con Python y Javascript . También necesitará la biblioteca Apache Beam , un archivo json (establecido en la variable de entorno GOOGLE_APPLICATION_CREDENTIALS con privilegios de cuenta preconfigurados) y un archivo require.txt que contiene una lista de paquetes para la instalación (en nuestro caso, solo se necesita un paquete beam-nuggets ). El script de Python se presenta a continuación:

Texto oculto
bq-to-sql :

args = parser.parse_args()
project = args.project
job_name = args.job_name + str(uuid.uuid4())
bigquery_source = args.bigquery_source
postgresql_user = args.postgresql_user
postgresql_password = args.postgresql_password
postgresql_host = args.postgresql_host
postgresql_port = args.postgresql_port
postgresql_db = args.postgresql_db
postgresql_table = args.postgresql_table
staging_location = args.staging_location
temp_location = args.temp_location
subnetwork = args.subnetwork
 
options = PipelineOptions(
        	flags=["--requirements_file", "/opt/python/requirements.txt"])
# For Cloud execution, set the Cloud Platform project, job_name,
# staging location, temp_location and specify DataflowRunner.
 
google_cloud_options = options.view_as(GoogleCloudOptions)
google_cloud_options.project = project
google_cloud_options.job_name = job_name
google_cloud_options.staging_location = staging_location
google_cloud_options.temp_location = temp_location
google_cloud_options.region = "us-west1"
worker_options = options.view_as(WorkerOptions)
worker_options.zone = "us-west1-a"
worker_options.subnetwork = subnetwork
worker_options.max_num_workers = 20
 
options.view_as(StandardOptions).runner = 'DataflowRunner'
 
start_date = define_start_date()
with beam.Pipeline(options=options) as p:
	rows = p | 'QueryTableStdSQL' >> beam.io.Read(beam.io.BigQuerySource(
                    	query='SELECT \
                        	billing_account_id, \
                        	service.id as service_id, \
                        	service.description as service_description, \
                        	sku.id as sku_id, \
                        	sku.description as sku_description, \
                        	usage_start_time, \
                        	usage_end_time, \
                        	project.id as project_id, \
                        	project.name as project_description, \
                        	TO_JSON_STRING(project.labels) \
                            	as project_labels, \
                        	project.ancestry_numbers \
                            	as project_ancestry_numbers, \
                        	TO_JSON_STRING(labels) as labels, \
                        	TO_JSON_STRING(system_labels) as system_labels, \
                        	location.location as location_location, \
                        	location.country as location_country, \
                        	location.region as location_region, \
                        	location.zone as location_zone, \
                        	export_time, \
                        	cost, \
                        	currency, \
                        	currency_conversion_rate, \
                        	usage.amount as usage_amount, \
                        	usage.unit as usage_unit, \
                        	usage.amount_in_pricing_units as \
                         	usage_amount_in_pricing_units, \
                        	usage.pricing_unit as usage_pricing_unit, \
                        	TO_JSON_STRING(credits) as credits, \
                        	invoice.month as invoice_month, \
                   	     cost_type \
FROM `' + project + '.' + bigquery_source + '` \
                        	WHERE export_time >= "' + start_date + '"',
                    	use_standard_sql=True))
	source_config = relational_db.SourceConfiguration(
          	              drivername='postgresql+pg8000',
                        	host=postgresql_host,
                        	port=postgresql_port,
                        	username=postgresql_user,
                        	password=postgresql_password,
                        	database=postgresql_db,
                        	create_if_missing=True,
                        	)
	table_config = relational_db.TableConfiguration(
                        	name=postgresql_table,
            	            create_if_missing=True
                        	)
	rows | 'Writing to DB' >> relational_db.Write(
    	source_config=source_config,
    	table_config=table_config
	)


Para mantener la coherencia de los datos, debe determinar el tiempo máximo de exportación , el tiempo de exportación a PostgreSQL y luego cargar los registros de BigQuery que comenzarían con este límite de tiempo.

Después de cargar los datos, debe actualizar la vista materializada. Dado que este proceso lleva algo de tiempo, le permitirá crear una nueva vista materializada con estructura e índices idénticos, eliminar la anterior y cambiar el nombre de la nueva.

Crear un archivo JSON con permisos SA


La cuenta que creó anteriormente para el flujo de trabajo Cloud Dataflow también se usa en tareas de Cron . Debe especificar un comando que creará una contraseña-clave privada para la cuenta, que luego se cargará en el clúster de Kubernetes , donde actuará como una contraseña oculta para acceder a la tarea Cron .

gcloud iam service-accounts keys create ./cloud-sa.json \
--iam-account "bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--project ${project}

Implementar una contraseña secreta en un clúster K8s:

kubectl create secret generic bq-to-sql-creds --from-file=./cloud-sa.json
