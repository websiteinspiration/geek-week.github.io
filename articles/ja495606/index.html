みなさん、こんにちは。この記事では、すべてが終わったときに、上記の2つの主要な段階についてお話ししたいと思います。ここで

はどのようなアプリケーションを読むことができます。
そして、言い訳と修正のためにここに送ることについて。

カットの下で、アプリケーションの分析の最初に見たものを見て、コードから興味深い点をすべて調べます。リークの多いアプリケーションを楽しんで、「実行時間の確認」を受け入れません。

バックグラウンド


それはすべて、テレグラムチャネル「IT and COPM」で始まりました。私はすぐにアプリケーションへのリンクとポストを見ていないプレイ市場とのapkです以下。約30分後、このチャンネルの投稿数から読んでみることにしました。当時、彼らはすでにアプリケーションを逆コンパイルしていて、データがどこに送信されたかを積極的に監視していました。彼らはいくつかのエンドポイント、ホスト、トークンを引き出し、コードの一部のスクリーンショットを投稿し始めました。APKを適用した後、逆コンパイルされたコードをGitHubに公開するように依頼しました-1 時間半後に投稿されました。次に接続しました...

主要部分


バッキング


最初に、アプリケーション内にIdentix.one顔認識サービスAPIを操作するためのトークンがあったことを言いたいと思います。これらの担当者は、最初にユーザーデータベースに写真とデータをロードする必要があるように機能します。その後、認識のために写真を送信できます。

トークンはアプリケーション内にあり、バックエンドではなくクライアントを介してこのサービスのAPIを操作していました。したがって、トークンを引き裂いたユーザーは、ロシア連邦の市民の写真をアップロードし始め、それらに関する情報を受け取ることができます。素晴らしいですね。

このサービスの料金を見て、私はこれが一般的にCCTVカメラに当てはまることに気づきました。カメラごとにかなりの金額を支払う必要があり、アプリケーション開発者がその金額を支払う必要があることに驚きました。

API ドキュメントのセクションに入ると、ロードされたユーザーベースやその他のものを取得することは私には興味深いように思えました。だから私はリクエストを満たし、トークンをチェックすることにしました。幸いなことに、または残念なことに、トークンは無効になりました。それをすぐにTwitter経由でITおよびCOPMチャネルの所有者に通知しました。 そこで何か面白いことを見つけようと計画していたので、私は背中から勉強を始めました（たとえば、sql inj）。ホストと管理パネルにアクセスするためのリンクはすでにありました。管理エリアに行ってきました。クエリをすぐに見ると、これが一種のAtmosphere Frameworkであることが明らかになりました

。リクエストでワイルドなお尻を見つけたとき、どのボタンが押されたか、フォームのどのフラグが設定されているかを示すと、私は落ち込みました。私はこれについて掘り下げたくはなく、できる限りのことを行いました。「Atmosphere Framework CVE」をGoogleに導入しました。見出しをざっと見ただけで、XSS以外に興味深いものは何も見当たりませんでした。そして、私たちは私たちが知っている他のエンドポイントに行きました。

これらのエンドポイントは、作業を開始および終了するためのペンでした。これは、仕事に費やされる時間の範囲を特徴付けるでしょう。コードはすでにありました。私はプロジェクトを開いて、適切な場所にこすりつけ、パラメーターの研究を始めました。それらの中には、deviceEUI、idxid、緯度と経度がありました。最後の2つでそれがすぐに明らかであるならば、最初のもので-いいえ。リクエストが正常に完了するまでにN分を費やしたので、Access deniedで解決しました。

そこで、勉強の目的を理解するために、自分でアプリをダウンロードすることにしました。ダウンロードしましたが、もうログインできませんでした...その時までにバックアップを取っていました（まだ知りませんでした）。

プロジェクトコードをGitHubではなく、完全にアイデアで開いたので、新しいエンドポイントを探し始めました。見つかりました！適切な名前：/ api / covid / Device ...

私はこれらのペンですでに研究を始めました。長い間、私は承認がどのように行われるかを理解していませんでした。トークンなどのヘッダーはどこにありますか。そして、それは私に夜明けしました...彼女は消えました！これを確認した後、リクエストの処理と生成を分析した後、バックエンド部分で結論を出し、再度Twitterに送信しました。

バックエンド部分の結論：登録は、デバイス（電話）に関する情報を送信することによって行われます。これには、住所、電話番号、名前などが含まれます。登録直後、彼らはあなたからの写真を必要とし、あなたのプロフィールにアップロードします。実際、これが誰の写真であるかを示すために、リクエストに追加のパラメータ-IMEIが示されます。デバイスモデルのdeviceIdフィールドにあります登録の最初のステップで送信したもの。IMEIがあなたのものであることの検証はありません。他の人のメールを送信する価値はあります。別のデバイスからリクエストを正常に完了します。注目に値するのは、エンドポイント間に/ api / covid / Device / passwordがあることですが、応答の処理は次のとおりです。



デバイスを登録した後、APIはauthCodeを返しますが、残念ながら、このコードはオブジェクトフィールドに格納され、そこにあり、ダストを収集しています。他のリクエストの形成には関与していません。

したがって、認証がなければ、IMEIを知っているすべてのユーザーが人（またはそれを拾って/整理）は、誤った地理位置情報を送信したり、写真をアップロードしたり、営業日の開始に関するステータスを送信したりすることができます。

私はそのような楽しい実装で遊んでみたいですが、悲しいかな、すべてがオフになりました。

その時以来、私は完全にコードに遭遇し、自分のテレグラムチャネルに表示される内容を説明し始めました。パラグラフ間に何の関係もない完全なナンセンスであることが判明しましたが、この投稿の後、私はそれを書き直して読みやすくする強さ（すでに2〜3泊でした）を見つけました。

アプリケーションの解析


この時までにアプリケーションが機能しなくなったことはすぐに注目に値します。そのため、視覚的なものになるとは思いもしませんでした。彼はすべてを盲目的にそしてコードで勉強しました。私たちは皆、逆コンパイルが何であるかを知っていますが、起こったことは特定の結論を導き出すのに十分でした。

アプリケーションは、デバイス、名前、電話番号、住所に関する情報を保存します。コメントなどのあいまいなフィールドがあります。そして興味深いものがあります-医者。

顔の識別には、identixサービスが使用されます（後ろの部分と同じパラメーター）。アップロードした写真から年齢、性別、その他の情報を返すことができます。アプリケーション内で使用される個人の一意の識別子があります。

ナビゲーションにはBluetooth、WiFiの2種類のデバイスが使用されます。どちらにも名前とRSSI（信号品質、ポイントまでの距離について話す）があります。

あなたの地理位置情報は、GoogleとGPSモジュールに感謝します。

保存されているさまざまなタイプの情報の中に場所があります。驚いたことに、それは膨大な数のフィールドを持っています。バッテリーレベル、ステップ数（このモデルは別ですが）などを含みます。

設定では、テレメトリー間隔を設定できます。集荷と発送の両方。これらのパラメーターを変更する機能については不明です。

一意の識別子には3種類あります。

  • IMEIデバイス。
  • デバイスID（調査されていません）。
  • WiFiのMACアドレス。

したがって、IMEIの転送と選択について上記で説明したのは、理解のための簡略化です。これは実際には3つの識別子の1つです。

あなたのIDは、どのバージョンに依存SDKあなたが持っています。簡単に言えば、何らかの一意の識別子を取得することは可能ですか？アプリケーション開発者自身が混乱していて、1つの名前に集中することはできません。

記事にコードを配置した唯一の場所。

if (Build.VERSION.SDK_INT >= 29) {
    return getMac(context);
}
return getImei(context);

そしてgetImeiの中では、ここにifがあります：

if (Build.VERSION.SDK_INT >= 26) {
    str = telephonyManager.getImei();
} else {
    str = telephonyManager.getDeviceId();
}

アプリケーションはSIMカード番号を取得することもできますが、注意が必要な場合のみです。機会があります-それはするでしょう。

QRコードは、3種類のテキストIDで生成されます。このトピックは、顔認識サービスとともに、誰もが議論しています。私は焦点を合わせていません。

アプリケーションの興味深い機能は、デバイス上のあなたへのプッシュ通知です。開発者はリモートでコマンドを送信でき、それらはあなたによって実行されます。チームの1つはOPEN_CAMERAと呼ばれます。もう1つ-SERVER_ADDRESS_KEY-ここでは、ほとんどの場合、Telegramに類似しています。ブロックされると、すべてのユーザーに新しいIPが送信されましたアドレスと私たちはメッセンジャーを使用し続けました。さらに、テレメトリの送信と収集の間隔をリモートで設定できます。可能なプッシュ通知については、こちらをご覧ください。

もちろん、アバター、回転の方法、ローカルでの保存方法、正規化の方法について話すこともできます。アプリケーションの自動更新を実行したいという要望について話すことはできますが、時間でダイヤラにアクセスする機能、連絡先のリストを維持する機能（最も基本的なアドレス帳は名前と番号です）、周りのアクティブなファイルポイントの数と対話するという事実について話すことができましたしかし、それはすべて退屈です。

結論
