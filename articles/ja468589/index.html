共通コンポーネントのライブラリでの作業を整理する方法

あなたの会社が同じスタイルで複数の製品を作っている場合、ある日、共通のコードでライブラリを作成するというアイデアが思い浮かぶでしょう。たとえば、UIコンポーネント、承認サービスを使用したり、サードパーティのAPIを操作したりします。あなたは疑問に思うかもしれません：誰がこのコードをサポートするべきですか？ユーザーに変更を伝える方法は？結局のところ、どうやって彼らにあなたのライブラリーを使わせるのですか？

2015年以降、私はビジネスサービス部門のTinkoffで働いています。この間、私たちのチームは3人から60人以上の開発者に成長し、Tinkoff Businessエコシステムは3つから50個のWebアプリケーションに成長しました。開発のさまざまな段階で、一般的なコードをさまざまな方法で処理するようにアプローチしましたが、この記事でこれについてお話ししたいと思います。



財団：安くて明るい


2015年に早送りします。キャッシュ管理サービス、給与計算プロジェクト、コントロールパネルの3つのWebアプリケーションしかありません。そして多くの開発者。

アプリケーションインターフェイスは同じスタイルで作成され、一般的なコードは別のリポジトリのFoundationライブラリに移動されます。ライブラリはコンパイルされません。厳密に言えば、コンパイルするものはありません。すべてのES5コードはnpmで公開されていませんが、package.jsonのブランチの名前で接続されています。ステータスを修正するために、製品リリース用に別個のFoundationリリースブランチが作成されました。 Foundationブランチを修正するのを忘れた場合、修正プログラムを適用すると、Foundationが変更され、修正プログラムのバージョンが変更されないことがわかりました。

これは、package.jsonのFoundation接続は次のようになります。

"dependencies": {
    ...
    "sme-foundation": "git+https://stash_url/sme-foundation.git#develop"
    ...
},

現時点でのライブラリの主な原則は、一般的なコードの所有権です。

製品の機能がFoundationの改訂を必要とする場合、機能開発者は自分でそれを行いました。また、以前のバージョンと互換性がない場合は、すべてのプロジェクトでこのコードを使用することも規定しています。大規模なリファクタリングについても同じことが言えます。コンポーネントをリファクタリングしてそのAPIを変更したい場合は、同時に、すべての使用方法を試してください。

あるプロジェクトですでに別のプロジェクトにあるものを再利用したい場合-問題でもないので、ライブラリに持って行ってください。

幸い、コードは多くありませんでした。このアプローチは、3つ、4つ、5つのプロジェクトに最適で、多くの利点がありました。

  • 共通コードは1か所にあり、プロジェクトで再利用されました。
  • プロジェクトの開発が速くなりました。
  • 図書館は成長していた。
  • 誰もが共通のコードとプロジェクトの両方の知識を持っていたため、アーキテクチャの決定のレビューと採用がより効率的になりました。
  • 一般的なコードを改良する必要は機能の開発を妨げませんでした。

この時点で、最低限のドキュメントがありました。いくつかのJSDocと単体テストです。UIコンポーネントについては、視覚的なストアフロントが十分ではなかったため、安価で高速なソリューションであるデモUIを見ました。実際、それは、コンポーネント自体と対応するマークアップが挿入されたページに、角張ったアプリケーションでした。



2019年には、同じことをする可能性は低いです。しかし、これはこの物語から学べることです：

  • 一般的なコードを共有することは、小さなチームに最適です。
  • 3人しかいない場合でも、繰り返しコードを作成するのを怠らないでください。これは、あらゆる規模のチームが利用できます。
  • コンポーネントのリストを含む通常のページでさえ、あなたの人生を大幅に簡略化できます。

時が経つにつれ、Tinkoff Businessエコシステムと開発チームは成長しました。 12を超えるプロジェクトがあった場合、このアプローチは機能しなくなりました。

共通コンポーネントの変更は高額になりました。 1つのプロジェクトの開発者は、他のすべてのプロジェクトに慣れなくなったため、コンポーネントの使用と変更を行うための検索がより複雑になりました。財団に影響を与える新機能を実装する時間が増えました。

新しい開発者は、何が行われ、どのように使用されたのかを完全に把握していませんでした。コンポーネントのAPIを変更することは恐ろしいことでした。これが原因で、フランケンシュタインコンポーネントは多数の入力パラメーターとともに表示されました。

時々これはUXの矛盾につながりました。キーボードの操作、フォーカスの操作などの詳細は、コンポーネントごとに異なる方法で実装されていました。そして、どこかで、開発者がビジネス機能に集中していたので、それらはまったく実装されませんでした。

他のAngularプロジェクトが社内に登場し、私たちのライブラリも使用することを提案しました。最初は彼らも同意しました...しかし、改善が必要になるとすぐに、私たちは困難な状況に陥りました。すべての開発者はプロジェクトで忙しく、同僚は他の誰かのライブラリを扱う意欲がありません。誰もが財団の責任を負い、特に誰も責任を負いませんでした。これは同僚には適していませんでした。

UIキットと作業組織への新しいアプローチ


再設計と新しいUIキットについて2017年になると、別の方法で新しいコンポーネントの開発を開始しました。まず、専用のチームがあります。

チーム


私たちは製品チームから3人を選び、次のように述べました。「今、これらの人たちは他のプロジェクトのUIコンポーネントを作成しています。」

専任チームは何を提供しましたか？

  • まず、製品チームの基本コンポーネントを短時間で準備しました。開発の開始から2週間後、同僚は最も必要なものを受け取りました。その後、お客様の優先事項に重点を置いてライブラリを開発しました。
  • 専任チームは特にコンポーネントとUXに重点を置いていました。私たちの仕事は、インターフェイスのエンドユーザーの観点（正しいインデント、コントラスト、キーボードの操作-これはクジラチームにとってささいなことではありません）の観点から、および製品チームの開発者の観点（一貫したAPI、便利な接続、拡張性）の観点から、定性的なコンポーネントを作成することでした。
  • 専任チームが責任を負います。コンポーネントでコンポーネントが検出された場合、製品開発者は彼と一緒に放置されることはありません。重大なバグは高い優先度で修正され、ホットフィックスが準備されますが、重要度の低いバグは優先度順に修正されます。ここで注目すべきは、強調表示されたチームが現れる前に、製品チームにとって重要ではなかった欠陥（たとえば、入力と選択でのプレースホルダーの色がわずかに異なる）は、長い間バックログにあり、ビジネス機能に道を譲る可能性があることです。しかし、クジラチームにとっては、コンポーネントの外観が最優先事項です。

コンポーネントの専門知識が1つのチームに集中している場合、他のユーザーにこれらのコンポーネントの使用を教えるにはどうすればよいでしょうか。このため、優れたドキュメントを作成しました。

ドキュメンテーション


私たちのデモスタンドを再設計するという考えは長い間空中にあり、新しいライブラリの開発はそれを可能にしました。

その後、AngularのStorybookはまだリリースされていないので、独自の方法で進めました。それは最高です。私たち自身の実装は私たちの想像力を制限するものではなく、私たちが望むすべてを完全に行うことができました。

そしてこれが私たちがしたことです：

  1. ライブラリ全体に関する追加情報：ライブラリの接続方法の段階的な説明、およびサポートされているブラウザのリスト。
  2. 各コンポーネントの詳細な説明を用意しました。なぜそれが必要なのか、どのように接続するのか、どの入出力パラメータをサポートするのか（それらをポークする機能、ストーリーブック付き）、一般的な使用例、同様のコンポーネントのリストです。



  3. このコンポーネントが使用されるプロジェクトのリストを追加しました。

  5. : UI Kit.


もちろん、これはすぐには現れませんでしたが、進化しました。

私たちの部署では、専用のチームとドキュメントで十分です。同僚は一般的なコードの使用に慣れており、UIキットの開発者を知っており、一般的に非常に忠実です。

しかし、UIキットの設計自体は、会社のすべての製品に共通するものとして位置付けられていました。つまり、社内のHRプロジェクトから、数万人のリモート従業員のための作業システムであるWebOfficeまで、何十ものチームが同じコンポーネントを必要としていました。したがって、クジラチームはより広いタスクに直面しました。すべてのAngularチームが使用する高品質の内部製品を作成することです。

概して、他の部門の同僚は好意的でしたが、それでも彼らはいくつかの疑問を持っていました：必要な機能が十分に迅速に開発されるか、彼らが高品質になるか...誰かがすでにコンポーネントを自分で作り始めていました。

これらの疑問を解決するために、私たちは作品をできるだけ透明性のあるものにまとめました。

透明性


この記事から1つだけ考えてみたら、次のアイデアを考えてください。ライブラリを成功させるには、優れたコンポーネントを作成するだけでは不十分です。できるだけ透明性があり、顧客指向である必要があります。この場合、顧客は最終製品の開発者です。

可能な限りの手段で、何をしているのか、なぜ、どのように使用するのかを同僚に伝えなければなりません。

どのツールを使用しましたか？

定期的なリリースとデモ。最初のデモは、開発の開始から2週間後に行われ、次のリリースの日に毎週開催されました。時には多くの新機能があり、時にはバグ修正のみでした。何があってもデモを行いました。

今後は、主な作業が完了し、APIが安定し、チームはリリースの代替に切り替えました-1つは機能付きリリース、もう1つは編集と改善付き-デモは新機能付きリリースでのみ実行されます。

変更ログ。従来のコミットとそれらからの変更ログの生成を導入し、チームの新しいバージョンへの移行を大幅に簡素化しました。



「レスポンシブ」チーム。すべてのチームと同様に、Slackには独自のチャネルがありますが、これは新しいことではありません。実際、私たちは2つのチャネルを持っています。1つはチーム内でのコミュニケーション用、もう1つはユーザー用です。質問への回答、発表、調査、デモの実施、その他の活動です。

入ってくる質問が本当に解決されることが重要です。いくつかの質問はケースで複雑で、いくつかはドキュメントのギャップを指摘しました（または、誰もがそれを読んでいるわけではないことをお知らせください）。そして時々、Angularの初心者は彼らの難しさについて書きました。同等の準備ができているクジラチームは、問題がチャットで解決されない場合に電話をするのを怠ったり、他人のコードを自分にダウンロードしたりすることなく、すべての人を助けました。当然、そのようなコミュニケーションは開発者の時間を浪費しますが、これはライブラリの作業の一部です。

現在、クジラチャネルには200人を超えるユーザーがおり、クジラの開発者の参加なしで多くの質問が解決されています。同僚が経験を共有し、お互いの質問に答えます。

リリース後の変更点のリストを含むニュースレター。彼らの対象読者は、開発者、マネージャー、製品デザイナーです。ニュースレターでは、変更は簡単かつ簡単に説明されていました。これは、常にChangelogで可能であるとは限りません。これは素晴らしいツールだと私は確信していますが、毎週高品質で明確なダイジェストを準備することは困難でした。しばらくして送信を停止しましたが、おそらくこの方法に戻ります。



ユーザー調査側面から見て、弱点を特定できるようにします。たとえば、次の調査の結果によると、UIキットの開発者が支援する準備ができていることにほとんどの同僚が完全に同意していることがわかりました。つまり、この方向ですべてを正しく行っているということです。しかし、「新しいコンポーネントが迅速に開発されている」という発言は、少数の回答者に同意しました。そのような結果をどうするか？一方では、すべてのチームメンバーの注意を引き、弱点に取り組みます。一方、ユーザーと協力してください。速度が重要な場合は、そのような用語の出所の説明に注意してください。



投票には、「何を改善できるか」のような自由回答式の質問が含まれていました。たとえば、サンプルコードの横に[コピー]ボタンを作成したり、クラスの日付をunixtimeで使用できるようにしたりするなど、同僚から貴重なヒントが得られたと言わざるを得ません。

チームの役割


上記で、私はすでに「応答性の高いチーム」について書いています。これは、会社全体のライブラリの成功のために多くを決定します。この考えをもう一度強調し、2つの関連するプラクティスについてお話ししたいと思います。

義務。ある時点で、同僚とのコミュニケーションは非常に多くなり、義務の導入は避けられないように思われました。これがTinkoffのサービスチームの数です。 1日1回、2日、1週間に1人のチームメンバーが勤務します。彼はチャンネルを監視し、質問に答え、ほとんどのコミュニケーションを担当しますが、チームメイトはサポートに気を取られません。

私たちはこの時点に達しておらず、やがて必要性はなくなりました。しかし、当直の他のチームが助けます。

監査。クジラの構成要素については、それらを開発しているチームよりも誰も知りません。また、2年間の開発で、Angularに関する優れた専門知識も蓄積してきました。「クジラチームからの監査」の実践を紹介します。彼らは製品プロジェクトのレビューに接続し、最終製品とそのコードを表示します。彼らの目標は、誤用を特定し、ドキュメントを改善し、良い習慣と悪い習慣のリストを作成することです。次回は何が起こるかについてお話します。

課題と妥協


多数の利害関係者とのプロセスと同様に、共通ライブラリの開発は妥協を模索しなければなりません。

コードの品質とAPIの安定性のバランスを見つけるのは簡単ではありませんでした。一方で、設計は作業の最初に変更されました。新しいコンポーネントまたは古いコンポーネントの新しい状態が現れました。逆に、何かが時代遅れになり、切り取られました。一方、正しいコンポーネントAPIに対する開発者のビジョンは変化していました。これらすべては必然的に重大な変更につながりました。しかし、ライブラリーのユーザー数は増加し、私たちの重大な変更はそれらに大きな不便を引き起こしました。

私たちは妥協案を発見しました。5つおきのリリースよりも頻繁に、つまり5週間に1回、重大な変更を加えることです。たとえば、そのような変更では、リリース0.50、0.55、0.60を呼び出すことができます。しかし、0.50から0.51への移行には何の努力も必要ありませんでした。これは、安定バージョン1.0.0のリリースまで続きました。現在、APIは安定しており、すべての大規模な変更はいつまでも2.0.0に延期されます。

何度か、新しいバージョンに切り替えるには、プレフィックスの名前の変更、アイコンのインポートの変更など、多くの日常的な作業が必要でした。そのような場合のために、移行スクリプトを実装しました。
