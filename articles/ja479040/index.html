視覚回帰テスト。リブート

前回の記事では、ビジュアルテスト、つまりビジュアル回帰テストを開発するためにGeminiエンジンを使用した経験について話しました。このようなテストでは、現在のスクリーンショットを以前に修正された参照スクリーンショットと比較することにより、次の変更後にUIで何かが「移動」したかどうかを確認します。それ以来、使用されているエンジンを含め、ビジュアルテストを作成するためのアプローチに多くの変更が加えられました。ここではHermioneを使用しますが、この記事では、Hermioneだけでなく、それ以降に蓄積された問題とその解決方法についても説明します。これにより、新しいエンジンへの移行にもつながりました。

まず、テストは機能しましたが、かなり成功しましたが、テストの対象と対象外について明確な理解がありませんでした。もちろん、カバレッジの程度についてはある程度のアイデアはありましたが、定量的には測定していませんでした。第二に、テストの構成は時間とともに増加し、さまざまなテストが同じものをテストすることがよくありました。異なるスクリーンショットでは、一部が同じ部分と一致していますが、異なるスクリーンショットです。その結果、CSSの小さな変更でも、一度に多くのテストを圧倒し、多数の標準の更新を必要とする場合があります。3つ目は、ダークテーマが製品に表示され、テストで何らかの形でカバーするために、一部のテストがダークテーマを使用するように選択的に切り替えられたため、カバレッジの程度を決定する際の問題が明確になりませんでした。

パフォーマンスの最適化


奇妙なことに、パフォーマンスの最適化から始めました。その理由を説明します。私たちの視覚テストはストーリーブックに基づいています。。ストーリーブックの各ストーリーは単一のコンポーネントではなく、全体の「ブロック」です（たとえば、エンティティのリスト、エンティティカード、ダイアログ、またはアプリケーション全体のグリッド）。このブロックを表示するには、ユーザーに表示されるデータだけでなく、ブロック内で使用されるコンポーネントの状態も含めて、ストーリーをデータで「ポンプ」する必要があります。この情報は、アプリケーションの状態のシリアル化された表現（reduxストア）を含むjsonファイルの形式でソースコードと共に保存されます。はい、このデータは控えめに言っても冗長ですが、テストの作成を大幅に簡略化します。新しいテストを作成するには、アプリケーションで目的のカード、リスト、またはダイアログを開き、アプリケーションの現在の状態のスナップショットを取り、それをファイルにシリアル化します。次に、新しいストーリーと、このストーリーのスクリーンショットを撮るテスト（すべて数行のコード）を追加します。

このアプローチでは、必然的にバンドルのサイズが大きくなります。その中のデータの重複の程度は、「ロールオーバー」するだけです。テストを実行すると、geminiエンジンは各テストスイートを個別のブラウザーセッションで実行します。各セッションはバンドルを新たにロードし、そのようなスキームのバンドルのサイズは最後の値とはかけ離れています。

テストの実行時間を短縮するために、テストスイートのテスト数を増やしてテストスイートの数を減らしました。したがって、1つのテストスイートが複数のストーリーに同時に影響を与える可能性があります。このスキームでは、ジェミニではテストスイート全体のスクリーンショット領域のみを設定できるため、スクリーンショットの特定の領域のみを「スクリーニング」する機能は実質的に失われました（APIでは各スクリーンショットの前にこれを行うことができますが、実際には機能しません）。

テストでスクリーンショットの領域を制限できないため、参照画像に視覚情報が重複しています。多くのテストはありませんでしたが、この問題は重大なものではありませんでした。はい、UIはそれほど頻繁に変更されませんでした。しかし、これは永遠に続くことはできませんでした-再設計は地平線に迫りました。

今後については、ハーマイオニーではショットごとにスクリーンショット領域を設定できるため、一見すると、新しいエンジンに切り替えることですべての問題を解決できると思います。しかし、それでも大規模なテストスイートを「破棄」する必要があります。実際のところ、視覚テストは本質的に安定しておらず（これは、ネットワークラグ、アニメーションの使用、または「火星の天気」などのさまざまな理由が原因である可能性があります）、自動再試行なしで実行するのは非常に困難です。 GeminiとHermioneの両方が全体としてテストスイートの再試行を実行します。テストスイートが「厚い」ほど、再試行中に正常に完了する可能性が低くなります。次の実行では、以前に正常に完了したテストが失敗する可能性があります。シックテストスイートの場合、Geminiエンジンに組み込まれた代替の再試行スキームを実装する必要があり、新しいエンジンに切り替えるときに、これを再度実行したくありませんでした。

したがって、テストスイートの読み込みを高速化するために、モノリシックバンドルを複数の部分に分割し、アプリケーションの状態の各スナップショットを、ストーリーごとに個別に読み込まれた「オンデマンド」で個別の「ピース」にハイライト表示します。ストーリー作成コードは次のようになります。

視覚情報の重複の排除


上で述べたように、Geminiではテストスイート全体のスクリーンショット領域セレクターを設定できます。つまり、スクリーンショットの視覚情報の重複の問題を完全に解決するには、スクリーンショットごとに独自のテストスイートを作成する必要があります。ストーリーの読み込みの最適化を考慮に入れても、速度の点では楽観的になりすぎず、テストエンジンの変更を検討しました。

実際、なぜハーマイオニーなのか？現時点では、Geminiリポジトリは非推奨としてマークされており、遅かれ早かれ、どこかに「移動」する必要がありました。 Hermione構成ファイルの構造はGemini構成ファイルの構造と同じであり、この構成を再利用することができました。 GeminiおよびHermioneプラグインも一般的です。さらに、テストインフラストラクチャ（仮想マシンと展開されたセレングリッド）を再利用することができました。

Geminiとは異なり、Hermioneはレイアウトの回帰テスト専用のツールとして位置付けられていません。そのブラウザ操作機能ははるかに広く、Webdriver IOの機能によってのみ制限されます。モカと組み合わせるこのエンジンは、レイアウトテストよりも機能テスト（ユーザーアクションのシミュレーション）に多く使用すると便利です。レイアウトの回帰テストでは、HermioneはassertView（）メソッドのみを提供します。これは、ブラウザーページのスクリーンショットを参照と比較します。スクリーンショットは、CSSセレクターを使用して指定された領域に制限できます。

私たちの場合、個々のストーリーのテストは次のようになります。

     return this.browser
       //  story

       // ,      story
       .waitForVisible('.loading-stub', true)
       //

waitForVisible（）メソッドは、その名前にもかかわらず、2番目のパラメーターをtrueに設定すると、外観だけでなく要素の非表示も期待できるようになります。ここでは、マーカー要素が非表示になるのを待機するためにそれを使用します。これは、データスナップショットがまだ読み込まれておらず、ストーリーがまだスクリーンショットの準備ができていないことを示しています。

HermioneのドキュメントでwaitForVisible（）メソッドを見つけようとしても、何も見つかりません。問題は、waitForVisible（）メソッドがWebdriver IO APIメソッドであることです。。 url（）メソッドもそれぞれ。 url（）メソッドでは、ストーリーブック全体ではなく、特定のストーリーのフレームアドレスを渡します。まず、これはストーリーリストがブラウザウィンドウに表示されないようにするために必要です。テストする必要はありません。次に、必要に応じて、フレーム内のDOM要素にアクセスできます（webdriverIOメソッドを使用すると、ブラウザーコンテキストでJavaScriptコードを実行できます）。

テストの記述を簡略化するために、モカテストのラッパーを作成しました。事実は、回帰テストのテストケースの詳細な説明に特別な意味はないということです。すべてのテストケースは同じです-「エタロンに等しいはずです」。まあ、各テストでデータの読み込みを待つコードも複製したくありません。したがって、すべての「サル」テストの同じ作業はラッパー関数に委任され、テスト自体は宣言的に（まあ、ほぼ）記述されています。この関数のテキストは次のとおりです。

create-test-suite.js
const themes = [ 'default', 'dark' ];
const rootClassName = '.explorer';
const loadingStubClassName = '.loading-stub';
const timeout = 2000;

function createTestSuite(testSuite) {
  const { name, storyName, browsers, testCases, selector } = testSuite;
  //  ,      
  browsers && hermione.only.in(browsers);
  //     
  themes.forEach(theme => {
    describe(`${name}_${theme}`, () => it('should equal to etalon', function() {
      let browser = this.browser
        //   story
        .url(`${storybookUrl}/iframe.html?selectedKind=visual-regression&selectedStory=${storyName}-${theme}`)
        //    
        .waitForVisible(loadingStubClassName, timeout, true)
        .waitForVisible(rootClassName);

      //    (  )
      if (testCases && testCases.length > 0) {
        testCases.forEach(testCase => {
          if (testCase.before)
            browser = testCase.before(browser);
          browser = browser.assertView(`${name}__${testCase.name}_${theme}`, 
            testCase.selector || selector || rootClassName, testCase.options);
        });
        return browser;
      }

      //    ,   
      return browser.assertView(`${name}_${theme}`, selector || rootClassName);
    }));
  });
}

カバレッジの決定


そのため、速度と冗長性を把握しましたが、テストの有効性、つまりテストでのコードのカバレッジの程度を判断することはここでも残っています（ここでコードとはCSSスタイルシートを意味します）。

テストストーリーでは、1つのスクリーンショットでできるだけ多くのスタイルをカバーするために、カード、リスト、およびその他の要素を埋めるのが最も困難なものを経験的に選択しました。たとえば、エンティティカードをテストするために、さまざまな種類のコントロール（テキスト、数値、転送、日付、グリッドなど）が多数あるカードが選択されました。さまざまな種類のエンティティのカードには固有の仕様があります。たとえば、ドキュメントカードからドキュメントバージョンのリストを含むパネルを表示でき、タスクカードにはこのタスクの対応が表示されます。したがって、エンティティのタイプごとに、独自のストーリーと、このタイプに固有の一連のテストなどが作成されました。最終的に、すべてがテストでカバーされているように見えましたが、「いいね」よりも少し自信をつけたかったのです。

Chrome DevToolsでカバレッジを評価するために、この場合に非常に適したCoverageという名前のツールがあります。



カバレッジを使用すると、ブラウザーページの操作時に使用されたスタイルまたはjsコードを特定できます。緑のストライプの使用に関するレポートは、使用されているコードを示し、赤-使用されていません。 「hello、world」レベルのアプリケーションがあればすべて問題ありませんが、何千行ものコードがある場合はどうすればよいでしょうか。カバレッジ開発者はこれをよく理解し、レポートをファイルにエクスポートする機能を提供しました。このファイルは、プログラムで既に作成できます。

今のところ、カバー率を自動的に収集する方法は見つかっていません。理論的には、これはpupeteer headless browserを使用して行うことができます。、ただしpupeteerはセレンの制御下では機能しません。つまり、テストのコードを再利用することはできません。したがって、ここでは、この非常に興味深いトピックをスキップして、ペンで作業します。

テストを手動モードで実行すると、カバレッジレポート（jsonファイル）が取得されます。各CSS、JS、TSなどのレポートで ファイルはそのテキスト（1行）とこのテキストで使用されるコードの間隔（この行の文字インデックスの形式）を示します。以下はレポートの一部です。

一見すると、未使用のCSSセレクターを見つけることは難しいことではありません。しかし、この情報をどうすればよいでしょうか。実際、最終的な分析では、特定のセレクターではなく、テストでカバーするのを忘れたコンポーネントを見つける必要があります。 1つのコンポーネントのスタイルは、12を超えるセレクターで設定できます。その結果、レポートの分析結果に基づいて、何百もの未使用のセレクターを取得します。それぞれを処理すると、多くの時間を費やす可能性があります。

ここでは、正規表現が役立ちます。もちろん、CSSクラスの命名規則が満たされている場合にのみ機能します（このコードでは、CSSクラスはBEM方法論に従ってブロック名が付けられています-block_name_name_name_modifier）。正規表現を使用して、ブロック名の一意の値を計算します。これにより、コンポーネントに関連付けるのが難しくなりません。もちろん、要素と修飾子にも興味がありますが、そもそも、より大きな「魚」を扱う必要があります。以下は、カバレッジレポートを処理するためのスクリプトです

coverage.js
const modules = require('./coverage.json').filter(e => e.url.endsWith('.css'));

function processRange(module, rangeStart, rangeEnd, isUsed) {
  const rules = module.text.slice(rangeStart, rangeEnd);
  if (rules) {
    const regex = /^\.([^\d{:,)_ ]+-?)+/gm;
    const classNames = rules.match(regex);
    classNames && classNames.forEach(name => selectors[name] = selectors[name] || isUsed);
  }
}

let previousEnd, selectors = {};
modules.forEach(module => {
  previousEnd = 0;
  for (const range of module.ranges) {
    processRange(module, previousEnd, range.start, false);
    processRange(module, range.start, range.end, true);
    previousEnd = range.end;
  }
  processRange(module, previousEnd, module.length, false);
});

console.log('className;isUsed');
Object.keys(selectors).sort().forEach(s => {
  console.log(`${s};${selectors[s]}`);
});

履歴書の代わりに


ストーリーブックを視覚的テストの基礎として使用することで、十分な成果が得られました。比較的少数のストーリーと新しいストーリーを作成するためのコストが最小限のテストでcssコードを十分にカバーしています。

新しいエンジンへの移行により、スクリーンショットの視覚情報の重複を排除でき、既存のテストのサポートが大幅に簡素化されました。
