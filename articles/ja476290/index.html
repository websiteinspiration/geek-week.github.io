Expressでのエラー処理

最初にExpressで作業を開始し、エラーの処理方法を理解しようとしたとき、私は苦労しました。必要なものについて誰も書いていないようだった。結果として、自分の質問に対する答えを自分で探す必要がありました。今日は、Expressアプリケーションでのエラー処理について知っていることをすべて伝えたいと思います。同期エラーから始めましょう。



同期エラー処理


同期エラーを処理する必要がある場合は、最初に、命令を使用throwして、Expressリクエストハンドラーでそのようなエラーを発行できます。リクエストハンドラーは「コントローラー」とも呼ばれますが、私には「リクエストハンドラー」という用語を使用した方がわかりやすいので注意してください。

これは次のようになります。

app.post('/testing', (req, res) => {
  throw new Error('Something broke! ')
})

このようなエラーは、Expressエラーハンドラーを使用してキャッチできます。独自のエラーハンドラーを作成しなかった場合（これについては以下で詳しく説明します）、Expressはデフォルトハンドラーを使用してエラーを処理します。

標準のExpressエラーハンドラは次のとおりです。

  1. HTTP応答ステータスコードを500に設定します。
  2. リクエストを実行するエンティティにテキスト応答を送信します。
  3. テキスト応答をコンソールに記録します。

非同期エラー処理


非同期エラーを処理するには、引数を介してExpressエラーハンドラにエラーを送信する必要がありますnext。

app.post('/testing', async (req, res, next) => {
  return next(new Error('Something broke again! '))
})

これは、このエラーをログに記録するときにコンソールに到達するものです。


コンソールへのエラーメッセージ出力

Expressアプリケーションでasync / await構文を使用する場合は、 express-async-handlerのようなラッパー関数を使用する必要があります。これにより、 try / catchブロックなしで非同期コードを記述できます。Expressでの非同期/待機の詳細については、こちらをご覧ください。

const asyncHandler = require('express-async-handler')

app.post('/testing', asyncHandler(async (req, res, next) => {
  //  -
}))

リクエストハンドラーがにラップされた後、express-async-handler上記のように、命令を使用してエラーをスローすることができますthrow。このエラーはExpressエラーハンドラーに送られます。

app.post('/testing', asyncHandler(async (req, res, next) => {
  throw new Error('Something broke yet again! ')
}))

独自のエラーハンドラーを書く


エクスプレスエラーハンドラーは4つの引数を取ります。

  1. エラー
  2. 必要
  3. 解像度
  4. 次

中間ハンドラーとルートの後に配置する必要があります。

app.use(/*...*/)
app.get(/*...*/)
app.post(/*...*/)
app.put(/*...*/)
app.delete(/*...*/)

//          
app.use((error, req, res, next) => { /* ... */ })

独自のエラーハンドラを作成すると、Expressは標準エラーハンドラの使用を停止します。エラーを処理するには、エラーが発生したエンドポイントに対処したフロントエンドアプリケーションの応答を生成する必要があります。つまり、次のことを行う必要があります。

  1. 適切な応答ステータスコードを生成して送信します。
  2. 適切な回答を作成して送信します。

特定のケースで適切なステータスコードは、何が起こったかによって異なります。処理する準備ができている必要がある一般的なエラーのリストを次に示します。

  1. 400 Bad Request. . - — , ( — ). - — , ( — ).
  2. 401 Unauthorized. , ( , ).
  3. 403 Forbidden. , .
  4. 404 Not Found. , .
  5. エラー500 Internal Server Error。フロントエンドから送信されたリクエストは正しく生成されたが、バックエンドでエラーが発生した場合に適用されます。

適切な応答ステータスコードを定義したら、次を使用して設定する必要がありますres.status。

app.use((error, req, res, next) => {
  // ,       
  res.status(400)
  res.json(/* ... */)
})

応答ステータスコードは、エラーメッセージに対応している必要があります。これを行うには、エラーとともにステータスコードを送信します。

これを行う最も簡単な方法は、http-errorsパッケージを使用することです。エラーのある3つの情報を送信できます。

  1. 応答ステータスコード。
  2. エラーに関連するメッセージ。
  3. 送信する必要のあるデータ（これはオプションです）。

パッケージのインストール方法はhttp-errors次のとおりです。

npm install http-errors --save

このパッケージの使用方法は次のとおりです。

const createError = require('http-errors')

//  
throw createError(status, message, properties)

このすべてを正しく理解できる例を考えてみましょう。

私たちは彼のメールアドレスでユーザーを見つけようとしていると想像してください。しかし、このユーザーは見つかりません。その結果、対応するリクエストへの応答としてUser not found、ユーザーが見つからなかったことを発信者に通知するエラーを送信することにしました。

エラーを作成するときに必要なことは次のとおりです。

  1. 応答ステータスコードを400 Bad Request（結局のところ、ユーザーが誤ったデータを入力した）に設定します。これが最初のパラメーターになります。
  2. のようなメッセージを発信者に送信しますUser not found。これは2番目のパラメーターになります。

app.put('/testing', asyncHandler(async (req, res) => {
  const { email } = req.body
  const user = await User.findOne({ email })

  //     -  
  if (!user) throw createError(400, `User '${email}' not found`)
}))

デザインを使用してステータスコードを取得し、次を使用しerror.statusてエラーメッセージを取得できますerror.message。

//  
app.use((error, req, res, next) => {
  console.log('Error status: ', error.status)
  console.log('Message: ', error.message)
})


エラーをコンソールに記録した結果、

を使用して応答の状態が設定されres.status、メッセージが次の場所に書き込まれres.jsonます。

app.use((error, req, res, next) => {
  //    
  res.status(error.status)

  //  
  res.json({ message: error.message })
})

個人的には、ステータスコード、メッセージ、スタックトレース結果をそのような応答で送信することを好みます。これにより、デバッグが容易になります。

app.use((error, req, res, next) => {
  //    
  res.status(error.status)

  //  
  res.json({
    status: error.status,
    message: error.message,
    stack: error.stack
  })
})

▍デフォルトの応答ステータスコード


エラーの原因がそうcreateErrorでない場合、プロパティはありませんstatus。以下は、存在しないファイルを読み取る試みが行われた例ですfs.readFile。

const fs = require('fs')
const util = require('util')

//  readFile  ,  ,  async/await-.
//     : https://zellwk.com/blog/callbacks-to-promises
const readFilePromise = util.promisify(fs.readFile)

app.get('/testing', asyncHandler(async (req, res, next) => {
  const data = await readFilePromise('some-file')
})

このようなエラーオブジェクトにはプロパティがありませんstatus。

app.use((error, req, res, next) => {
  console.log('Error status: ', error.status)
  console.log('Message: ', error.message)
})


コンソールでのエラーロギングの結果

このような場合、デフォルトのエラーコードを設定できます。つまり、エラーについて話しているの500 Internal Server Errorです。

app.use((error, req, res, next) => {
  res.status(error.status || 500)
  res.json({
    status: error.status,
    message: error.message,
    stack: error.stack
  })
})

errorエラーステータスコードの変更


ユーザーから提供されたデータを使用して特定のファイルを読み取るとします。そのようなファイルが存在しない場合、エラーが発生する必要があることを意味します400 Bad Request。結局のところ、ファイルが見つからないため、サーバーが見つかりません。

この場合、try / catch構文を使用して元のエラーをキャッチする必要があります。次に、エラーオブジェクトをcreateError次のように再作成する必要があります。

app.get('/testing', asyncHandler(async (req, res, next) => {
  try {
    const { file } = req.body
    const contents = await readFilePromise(path.join(__dirname, file))
  } catch (error) {
    throw createError(400, `File ${file} does not exist`)
  }
})

▍404エラー処理


リクエストがすべての中間ハンドラーとルートを通過したが処理されなかった場合、これはそのようなリクエストに対応するエンドポイントが見つからなかったことを意味します。

エラーを処理404 Not Foundするには、ルートとエラーハンドラーの間にハンドラーを追加する必要があります。404エラーオブジェクトの作成は次のようになります。

//  ...
// ...

app.use((req, res, next) => {
  next(createError(404))
})

//  ...


エラーの詳細

▍ERR_HTTP_HEADERS_SENTエラーノート


エラーメッセージが表示されても、慌てないでくださいERR_HTTP_HEADERS_SENT: Cannot set headers after they are sent to the client。これは、応答ヘッダーを設定するメソッドが同じハンドラーで繰り返し呼び出されるために発生します。以下は、応答ヘッダーを自動的に設定するために呼び出すメソッドです。

  1. res.send
  2. res.json
  3. res.render
  4. res.sendFile
  5. res.sendStatus
  6. res.end
  7. res.redirect

あなたがメソッドを呼び出す場合たとえば、res.renderとres.json同じハンドラでは、答えは、エラーが発生しますERR_HTTP_HEADERS_SENT。

app.get('/testing', (req, res) => {
  res.render('new-page')
  res.json({ message: '¯\_(ツ)_/¯' })
})

その結果、このエラーが発生した場合は、応答ハンドラーコードを注意深く確認し、上記のメソッドのいくつかが呼び出される状況がないことを確認してください。

▍エラー処理とデータストリーミング


フロントエンドへの応答のストリーミング中に問題が発生した場合、同じエラーが発生する可能性がありますERR_HTTP_HEADERS_SENT。

この場合、エラー処理を標準ハンドラーに渡す必要があります。このようなハンドラーはエラーを送信し、接続を自動的に閉じます。

app.use((error, req, res, next) => {
  //       ,      
  if (res.headersSent) {
    return next(error)
  }

  //    
})
