リモコン


PowerShellを使用すると、1台以上のコンピューターをリモートで管理できます。これを行うには、対応する関数コマンドレット、コマンドのリモート開始、または対話型リモートセッションをサポートする.NET、CIM、WMIオブジェクトを使用できます。接続はPSリモーティング/ WinRM over HTTP（ポート5985）またはHTTPS（ポート5986、SSL証明書が必要）を介して行われます。デフォルトでは、接続はHTTPを介して確立されますが、データは引き続きAES-256対称鍵を使用して暗号化されます。NTLMとKerberos（デフォルト）の2つの認証方法がサポートされています。2番目の方法は、リモートコンピューターがADドメインの一部であり、ドメイン名の代わりにワークグループまたはIPが指定されている場合に使用でき、NTLMが使用され、追加の構成が必要です。接続先のコンピューターで、WinRMサービスが実行されている必要があります。



サービスが実行されていない場合は、Enable-PSRemotingコマンドレットを使用できます。Windows Server Coreを備えたRuVDS仮想マシンは、すぐに構成できるリモートアクセスで構成されており、追加のアクションは必要ありません。HTTPを使用するには、コマンドを起動するコンピューターのTrustedHostに接続する必要があるホストも追加する必要があります。マシンがグローバルネットワークで実行されている場合は、ブランドマウアーの構成が必要になる場合があります。PSリモート処理/ WinRMを使用すると、リモートセッション（New-PSSession、Get-PSSession、Enter-PSSession）を作成し、1つまたは複数のコンピューターでコマンドとスクリプト（Invoke-Command）を実行し、-ComputerNameパラメーターを指定してコマンドレットを使用できます。後者は、CIM / WMIコマンドレットにも当てはまります。詳細については、ウェブサイトをご覧ください。 マイクロソフト

コマンドレットとCIM / WMI


一部のコマンドレットは-ComputerNameパラメーターをサポートしていますが、ネットワーク上のホストを操作するための使用は制限されています。セッションを作成し、PowerShellコマンドをリモートで実行するだけでは必ずしも十分ではありません。WMIまたはCIMを操作するためのコマンドレットが役立ちます（NETオブジェクトで実行できる場合もあります）。WindowsでのWMIの開発とサポートが中止されたため、CIMを使用することをお勧めします。どちらの場合も、Common Information Model（CIM）と対応するクラスのインスタンスが使用されます。必要に応じて、結果のオブジェクトの構造を表示し、フィルタリング、並べ替え、コレクションからの選択、およびその他のアクションを実行できます結果以上。インタラクティブモードで作業する場合、テキスト出力は簡単にフォーマットでき、スクリプトでは通常、オブジェクトのプロパティとメソッドを参照します。WMIコマンドレットとCIMコマンドレットは似ていますが、最初のケースでは、特定のオブジェクト（クラスのインスタンス）への参照を取得し、Invoke-WmiMethodを使用せずにそのメソッドの1つを直接呼び出すのは簡単です。クラスメソッドの結果のCIMオブジェクトには含まれないため、制御のためにInvoke-CimMethodコマンドレットを呼び出す必要があります。

CIM / WMIによるプロセス管理


以前の記事では、システムで実行されているSystem.Diagnostics.Processタイプのオブジェクトを返すGet-Processをよく使用しました。このコマンドレットを使用すると、ローカルコンピューターに関する情報のみを取得できます。リモートマシンで作業するには、CIM / WMIコマンドレットとWin32_Processクラスが再び必要です。それを操作するために使用されるコマンドの例を示します。

Get-WmiObject Win32_Process

または

Get-CimInstance Win32_Process



前の例は、ローカルマシンで実行されているプロセスに関する情報の取得を示しています。-ComputerNameパラメーターは、リモートコンピューターでの作業に使用されます。

Get-WmiObject Win32_Process -ComputerName IP_ADDRESS
Get-WmiObject Win32_Process -ComputerName NAME

または

Get-CimInstance Win32_Process -ComputerName IP_ADDRESS
Get-CimInstance Win32_Process -ComputerName NAME

CIM / WMIによるサービス管理


現在の実装のGet-Serviceコマンドレットは-ComputerNameパラメーターをサポートしますが、リモートマシン上のサービスのステータスに関する情報を取得することしかできません。結果のオブジェクトのプロパティとメソッドを介してサービスを管理することはできません。分散環境でのより適切なアプローチは、CIMおよびWMIコマンドレットを使用することです。プロセスと同様に、CIMオブジェクトモデルには、サービス用の特別なWin32_Serviceクラスがあります。

Get-WmiObject Win32_Service

または

Get-CimInstance Win32_service



リモートコンピューターを操作するには、-ComputerNameパラメーターを使用する必要があります。

Get-WmiObject Win32_Service -ComputerName IP_ADDRESS
Get-WmiObject Win32_Service -ComputerName NAME

または

Get-CimInstance Win32_service -ComputerName IP_ADDRESS
Get-CimInstance Win32_service -ComputerName NAME

クエリを修飾するには、-Filterオプションを使用します。これにより、サービスが自動的に開始するのか、手動で開始する必要があるのか​​を簡単に見つけることができます。

Get-Wmiobject Win32_service -Filter "StartMode <>'disabled'" | sort StartMode | format-table -GroupBy StartMode -Property Name,State,PathName -AutoSize

同様に、アカウントごとにサービスをグループ化できます。

Get-Wmiobject Win32_service | group startname

次のコマンドは、管理者として起動されたすべてのサービスを返します。

Get-Wmiobject Win32_service -Filter "startname like '%administrator%'" | Select Name,startmode,state,startname,systemname

CIMコマンドレットの場合、フィルターは同様に機能します。

Get-CimInstance Win32_service -filter "startmode='auto' AND state<>'Running'" | Select Name,State,Systemname

実際の作業では、コマンドレットのテキスト出力は、バイナリオブジェクト自体ほど重要ではありません。彼らの助けを借りて、対応するサービスを開始および停止したり、サービスの開始モードを変更したり、プロパティを設定したりできます。WMIの場合、オブジェクトメソッドは直接またはInvoke-WmiMethodを介して呼び出されます。CIMオブジェクトの場合、使用できるのはInvoke-CimMethodのみです。

レジストリを操作する


Windowsレジストリは、コンピュータのハードウェアおよびソフトウェア構成に関する情報を含むツリーベースの階層データベースです。各ノードは、他のノード（データベースレコード）または何らかの値のキーを含むセクションです。セクションにはネストされたサブセクションが含まれる場合があります。レジストリの構造はファイルシステムに似ており、PowerShellにはそれを操作する仮想「ディスク」があります。 Get-PSDriveコマンドレットを使用してそれらを表示し、Set-Location（エイリアスcd）を使用して対応するセクションを参照できます。



レジストリの「ディスク」には、「ディレクトリ」（セクション）、サブディレクトリ「サブセクション」、「ファイル」（パラメータ）が含まれています。サブセクションを表示するには、Get-ChildItemコマンドレット（エイリアスdir）を使用し、パラメーターGet-ItemPropertyを表示します。レジストリキーは.NETクラスMicrosoft.Win32.RegistryKeyのインスタンスであり、パラメーターはSystem.Management.Automation.PSCustomObjectです。リモートマシンのレジストリにアクセスするには、.NET、WMI、またはCIMオブジェクトを直接使用する必要があります。

イベントログを操作する


Windowsのイベントログは管理者にとって重要なツールであり、システムとアプリケーションのステータスを監視したり、発生した問題をすばやく分析したりできます。通常、セキュリティログ、システムログ、およびアプリケーションログが機能する必要があります。インストールされているコンポーネントと構成されている役割に応じて、サーバー上にディレクトリサービスログ、ディレクトリサービスログ、またはファイルレプリケーションサービスログなどが存在する場合があります。これらをローカルマシンで表示するには、MMC管理コンソールを使用し、PowerShellではGet-EventLogコマンドレットを使用します。リモートホストを操作するには、特定のログ（Win32_NTEventlogFileクラスのインスタンス）に対応するCIMおよびWMIオブジェクトを使用できます。

$Logs = Get-WmiObject Win32_NTEventlogFile 
$Logs | Format-List

または

$Logs = Get-CimInstance Win32_NTEventlogFile 
$Logs | Format-List



この例は、ローカルログの操作を示しています。リモートコンピューターに接続するには、-ComputerNameパラメーターが必要です。

$Logs = Get-WmiObject Win32_NTEventlogFile -ComputerName NAME
$Logs | Format-List

または

$Logs = Get-CimInstance Win32_NTEventlogFile -ComputerName NAME
$Logs | Format-List

OSおよびハードウェア情報


CIMおよびWMIコマンドレットを使用して、ネットワーク上のコンピューターからさまざまな情報を取得および管理できます。CIM / WMIコマンドレットと対応するクラスは、これらの目的で使用されます。実際の例での使用を検討してください。

Win32_OperatingSystemクラスは、さまざまなタスクを解決します。彼のWin32Shutdownメソッドを使用して、ユーザーセッションが終了します。

(Get-WMIObject Win32_OperatingSystem).Reboot()

BIOS情報に必要なWin32_BIOSクラス

Get-WmiObject Win32_BIOS | Select-Object -Property * -ExcludeProperty __*

または

Get-CimInstance Win32_BIOS | Select-Object -Property * -ExcludeProperty __*



オペレーティングシステムのプロパティの出力：

Get-WmiObject Win32_OperatingSystem | Select-Object –Property * -ExcludeProperty __*

または

Get-CimInstance Win32_OperatingSystem | Select-Object –Property * -ExcludeProperty __*



Win32_QuickFixEngineeringクラスは、インストールされている更新のリストを取得するように設計されています。

Get-WmiObject Win32_QuickFixEngineering

または

Get-CimInstance Win32_QuickFixEngineering


公開サイクルの最後に、PowerShellは単なるコマンドシェルではないことを思い出してください。むしろ、.NETオブジェクトとCIMオブジェクトモデルを使用する機能を備えたBashとSSH / SSHDのハイブリッドに似ています。これにより、面倒な構文構造が必要になることもありますが、分散IT環境の管理の複雑なタスクを解決できます。今日、PowerShellの知識がなければ、一人の有能なWindowsシステム管理者はできません。
