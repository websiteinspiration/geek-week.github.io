大規模なプロジェクトでより美しく：セットアップに20分を費やし、フォーマットを永遠に忘れましょう

開発者はコードのフォーマットに同意できないことがよくあり、多くの人にとって典型的な作業日は次のようになります。コーヒー、コーディング、すべてが平和で良いです。括弧が間違っているか、何かを新しい行に転送しませんでした。



1年前、Skyengのチームの1つは、ほとんどすべてのレビューでそのようなホリバーに遭遇しました。しかし、最も苦痛のある人は言った、「今、私たちはプレティエに住んでいます、同意しますか？」次の数か月間、彼らはフォーマットの問題を提起することはありませんでした、そして今、これは全体のフロントエンドのモノリポジトリにあります-そしてそれはそこで呼び出すすべてのチームによって使用されます。

どうやって


Prettierの目標は、開発者がフォーマットについて考えないようにすることです。これには最小限の設定があります。アーサーはチームのチャットでのプロジェクトへのリンクを投げたとき、これはパベルを買収：

-きれいに身入れ、
-事前にコミットフック（置くより）、
-コードスタイルがあればそこに（例えば、論理演算であることをチャットでのコメントのカップルを残しました「ああは行頭ではなく行末に置かれる）

それだけなので、チームはプレティエに切り替えました。Prettierの前後の業界コードの例を以下に示します。このツールの機能を示すために少し再フォーマットされています。

前：

  public listenDndForFocusEvents(channel: string): Observable<boolean> {
    return this.drag
      .pipe(
        filter(
          event => event.channel === channel
        ), 
        filter(
          event => event instanceof DndDragStartEvent || event instanceof DndDragEndEvent
        ), 
        map(
          event => event instanceof DndDragStartEvent
        )
      )
  }

後：

  public listenDndForFocusEvents(channel: string): Observable<boolean> {
    return this.drag.pipe(
      filter(event => event.channel === channel),
      filter(event => event instanceof DndDragStartEvent || event instanceof DndDragEndEvent),
      map(event => event instanceof DndDragStartEvent),
    );
  }

1行で記述されていても、次のように再フォーマットされます。
