WAL-G: Backups e recuperação de banco de dados PostgreSQL

Há muito se sabe que fazer backups em dumps SQL (usando pg_dump ou pg_dumpall ) não é uma boa ideia. Para fazer backup do PostgreSQL, é melhor usar o comando pg_basebackup , que faz uma cópia binária dos logs do WAL. Mas quando você começar a estudar todo o processo de criação de um backup e restauração, entenderá que precisa escrever pelo menos alguns triciclos para que tudo isso funcione e não cause dores tanto por cima quanto por baixo. A fim de aliviar o sofrimento, o WAL-G foi desenvolvido.

O WAL-G é uma ferramenta escrita em Go para fazer backup e restaurar bancos de dados PostgreSQL ( e, mais recentemente, MySQL / MariaDB, MongoDB e FoundationDB) Ele suporta o trabalho com armazenamentos do Amazon S3 (e análogos, por exemplo, Yandex Object Storage), bem como Google Cloud Storage, Azure Storage, Swift Object Storage e apenas com o sistema de arquivos. Toda a configuração é reduzida a etapas simples, mas devido ao fato de os artigos estarem espalhados na Internet, não há um manual de instruções completo que inclua todas as etapas de e para (existem várias postagens em Habré, mas faltam muitos pontos).



Este artigo foi escrito principalmente para sistematizar seu conhecimento. Eu não sou um DBA e, em algum lugar, posso expressá-lo em uma linguagem de desenvolvimento filisteu, para que quaisquer correções sejam bem-vindas!

Separadamente, observo que tudo o que se segue é relevante e verificado para o PostgreSQL 12.3 no Ubuntu 18.04, todos os comandos devem ser executados como um usuário privilegiado.

Instalação


No momento da redação deste artigo, a versão estável do WAL-G é a v0.2.15 (março de 2020) . Nós o usaremos ( mas se você deseja montá-lo a partir do branch master, o repositório do github tem todas as instruções para isso ). Para baixar e instalar, você precisa:

#!/bin/bash

curl -L "https://github.com/wal-g/wal-g/releases/download/v0.2.15/wal-g.linux-amd64.tar.gz" -o "wal-g.linux-amd64.tar.gz"
tar -xzf wal-g.linux-amd64.tar.gz
mv wal-g /usr/local/bin/

Depois disso, você deve primeiro configurar o WAL-G e depois o próprio PostgreSQL.

Configuração WAL-G


Para um exemplo de armazenamento de backup, o Amazon S3 será usado ( porque é mais próximo dos meus servidores e seu uso é muito barato ). Para trabalhar com isso, você precisa de um “balde s3” e teclas de acesso.

Todos os artigos anteriores sobre o WAL-G usavam a configuração usando variáveis ​​de ambiente, mas a partir desta versão as configurações podem ser localizadas no arquivo .walg.json no diretório inicial do usuário do postgres. Para criá-lo, execute o seguinte script bash:

#!/bin/bash

cat > /var/lib/postgresql/.walg.json << EOF
{
    "WALG_S3_PREFIX": "s3://your_bucket/path",
    "AWS_ACCESS_KEY_ID": "key_id",
    "AWS_SECRET_ACCESS_KEY": "secret_key",
    "WALG_COMPRESSION_METHOD": "brotli",
    "WALG_DELTA_MAX_STEPS": "5",
    "PGDATA": "/var/lib/postgresql/12/main",
    "PGHOST": "/var/run/postgresql/.s.PGSQL.5432"
}
EOF
#    :
chown postgres: /var/lib/postgresql/.walg.json

Vou explicar um pouco em todos os aspectos:

  • WALG_S3_PREFIX - o caminho para o seu S3-bucket onde os backups serão executados (você pode tanto na raiz quanto na pasta);
  • AWS_ACCESS_KEY_ID – S3 ( – ReadOnly Policy! );
  • AWS_SECRET_ACCESS_KEY – S3;
  • WALG_COMPRESSION_METHOD – , Brotli ( /);
  • WALG_DELTA_MAX_STEPS – «» ( , , );
  • PGDATA - o caminho para o diretório com os dados do seu banco de dados ( você pode descobrir executando o comando pg_lsclusters );
  • PGHOST - conexão ao banco de dados, com um backup local é melhor fazê-lo através do soquete unix, como neste exemplo.

Outros parâmetros podem ser encontrados na documentação: https://github.com/wal-g/wal-g/blob/v0.2.15/PostgreSQL.md#configuration .

Configuração do PostgreSQL


Para que o arquivador dentro do banco de dados faça upload de logs WAL na nuvem e recupere-os (se necessário), é necessário definir vários parâmetros no arquivo de configuração /etc/postgresql/12/main/postgresql.conf . Apenas para iniciantes, você precisa garantir que nenhuma das configurações abaixo esteja definida com outros valores, para que, ao reiniciar a configuração, o DBMS não caia. Você pode adicionar esses parâmetros usando:

#!/bin/bash

echo "wal_level=replica" >> /etc/postgresql/12/main/postgresql.conf
echo "archive_mode=on" >> /etc/postgresql/12/main/postgresql.conf
echo "archive_command='/usr/local/bin/wal-g wal-push \"%p\" >> /var/log/postgresql/archive_command.log 2>&1' " >> /etc/postgresql/12/main/postgresql.conf
echo “archive_timeout=60” >> /etc/postgresql/12/main/postgresql.conf
echo "restore_command='/usr/local/bin/wal-g wal-fetch \"%f\" \"%p\" >> /var/log/postgresql/restore_command.log 2>&1' " >> /etc/postgresql/12/main/postgresql.conf

#     SIGHUP    
killall -s HUP postgres

Descrição dos parâmetros a serem definidos:

  • wal_level - quanta informação escrever nas revistas WAL, "réplica" - para escrever tudo;
  • archive_mode - ativa o download de logs WAL usando o comando do parâmetro archive_command ;
  • archive_command - comando para arquivar o log WAL completo;
  • archive_timeout - o arquivamento de logs é feito apenas quando concluído, mas se o servidor alterar / adicionar dados ao banco de dados um pouco, faz sentido definir um limite em segundos aqui, após o qual o comando de arquivamento será forçado ( eu tenho uma gravação intensa no banco de dados a cada segundo, portanto, Recusei-me a definir este parâmetro na produção );
  • restore_command - um comando para restaurar um log WAL de um backup, será usado se as últimas alterações do banco de dados estiverem ausentes no "backup completo" (backup básico).

Você pode ler mais sobre todos esses parâmetros na tradução da documentação oficial: https://postgrespro.ru/docs/postgresql/12/runtime-config-wal .

Configurando um agendamento de backup


Goste ou não, mas o cron é a maneira mais conveniente de começar. É isso que vamos configurar para criar backups. Vamos começar com o comando para criar um backup completo: no wal-g, este é o argumento para executar backup-push . Mas, primeiro, é melhor executar este comando manualmente a partir do usuário do postgres para garantir que tudo esteja bem (e que não haja erros de acesso):

#!/bin/bash

su - postgres -c '/usr/local/bin/wal-g backup-push /var/lib/postgresql/12/main'

Os argumentos de início indicam o caminho para o diretório de dados - lembro que você pode reconhecê-lo executando pg_lsclusters .

Se tudo correu bem e os dados foram carregados no repositório S3, você pode configurar o lançamento periódico no crontab:
#!/bin/bash

echo "15 4 * * *    /usr/local/bin/wal-g backup-push /var/lib/postgresql/12/main >> /var/log/postgresql/walg_backup.log 2>&1" >> /var/spool/cron/crontabs/postgres
#       
chown postgres: /var/spool/cron/crontabs/postgres
chmod 600 /var/spool/cron/crontabs/postgres

Neste exemplo, o processo de backup inicia todos os dias às 4:15 da manhã.

Excluir backups antigos


Provavelmente, você não precisa armazenar absolutamente todos os backups da era mesozóica, portanto, será útil "limpar" periodicamente seu armazenamento ("backups completos" e logs do WAL). Faremos tudo isso também através da tarefa cron:

#!/bin/bash

echo "30 6 * * *    /usr/local/bin/wal-g delete before FIND_FULL $(date -d '-10 days' '+%FT%TZ') --confirm >> /var/log/postgresql/walg_delete.log 2>&1" >> /var/spool/cron/crontabs/postgres
#          (        )
chown postgres: /var/spool/cron/crontabs/postgres
chmod 600 /var/spool/cron/crontabs/postgres

O Cron executará essa tarefa todos os dias às 6:30 da manhã, excluindo tudo (backups completos, deltas e WALs), exceto cópias nos últimos 10 dias, mas deixará pelo menos um backup antes da data especificada, para que qualquer ponto após a data caia no PITR .

Restaurar do backup


Não é segredo que a garantia de uma base saudável é a recuperação periódica e a verificação da integridade dos dados internos. Como recuperar usando o WAL-G - contarei nesta seção e falaremos sobre verificações posteriormente.

Separadamente, é importante notar que, para recuperação em um ambiente de teste (tudo o que não é produção) - é necessário usar a conta Somente leitura no S3, para não sobrescrever acidentalmente os backups. No caso do WAL-G, você precisa definir ao usuário S3 os seguintes direitos na Diretiva de Grupo ( Efeito: Permitir ): s3: GetObject , s3: ListBucket , s3: GetBucketLocation . E, é claro, não se esqueça de definir archive_mode = off no arquivo de configurações postgresql.conf antespara que sua base de teste não queira fazer backup silenciosamente.

A recuperação é realizada com um leve movimento da mão, com a exclusão de todos os dados do PostgreSQL (incluindo usuários), portanto, seja extremamente cuidadoso ao executar os seguintes comandos.

#!/bin/bash

#     (, pgbouncer),    ,       
service pgbouncer stop
#   ,     (, monit),        (   pgsql12)
monit stop pgsql12
#    
service monit stop
#    
service postgresql stop
#       (!!!);     ,      
rm -rf /var/lib/postgresql/12/main
#      
su - postgres -c '/usr/local/bin/wal-g backup-fetch /var/lib/postgresql/12/main LATEST'
#      -   (. https://postgrespro.ru/docs/postgresql/12/runtime-config-wal#RUNTIME-CONFIG-WAL-ARCHIVE-RECOVERY ),        postgres
su - postgres -c 'touch /var/lib/postgresql/12/main/recovery.signal'
#   ,     
service postgresql start

Para aqueles que desejam verificar o processo de recuperação - um pequeno pedaço de bash-magic é preparado abaixo, para que, em caso de problemas na recuperação - o script trava com um código de saída diferente de zero. Neste exemplo, 120 verificações são feitas com um tempo limite de 5 segundos (apenas 10 minutos para recuperar) para descobrir se o arquivo de sinal foi excluído (isso significa que a recuperação foi bem-sucedida):

#!/bin/bash

CHECK_RECOVERY_SIGNAL_ITER=0
while [ ${CHECK_RECOVERY_SIGNAL_ITER} -le 120 ]
do
    if [ ! -f "/var/lib/postgresql/12/main/recovery.signal" ]
    then
        echo "recovery.signal removed"
        break
    fi
    sleep 5
    ((CHECK_RECOVERY_SIGNAL_ITER+1))
done

#        ,    
if [ -f "/var/lib/postgresql/12/main/recovery.signal" ]
then
    echo "recovery.signal still exists!"
    exit 17
fi

Após uma recuperação bem-sucedida, não se esqueça de reiniciar todos os processos (pgbouncer / monit, etc.).

Verificação de dados após recuperação


Certifique-se de verificar a integridade do banco de dados após a recuperação, para que não haja situação com um backup quebrado / torto. E é melhor fazer isso com cada arquivo criado, mas onde e como depende de sua imaginação (você pode aumentar servidores individuais a cada hora ou executar um teste no IC). Mas pelo menos - você precisa verificar os dados e índices no banco de dados.

Para verificar os dados, basta executá-los no dump, mas é melhor que, ao criar o banco de dados, você tenha as somas de verificação ativadas ( somas de verificação de dados ):

#!/bin/bash

if ! su - postgres -c 'pg_dumpall > /dev/null'
then
    echo 'pg_dumpall failed'
    exit 125
fi

Para verificar índices - existe um módulo amcheck , pegaremos a consulta sql nos testes WAL-G e construiremos um pouco de lógica em torno de:

#!/bin/bash

#  sql-       
cat > /tmp/amcheck.sql << EOF
CREATE EXTENSION IF NOT EXISTS amcheck;
SELECT bt_index_check(c.oid), c.relname, c.relpages
FROM pg_index i
JOIN pg_opclass op ON i.indclass[0] = op.oid
JOIN pg_am am ON op.opcmethod = am.oid
JOIN pg_class c ON i.indexrelid = c.oid
JOIN pg_namespace n ON c.relnamespace = n.oid
WHERE am.amname = 'btree'
AND c.relpersistence != 't'
AND i.indisready AND i.indisvalid;
EOF
chown postgres: /tmp/amcheck.sql

#          
# (       – )
cat > /tmp/run_amcheck.sh << EOF
for DBNAME in \$(su - postgres -c 'psql -q -A -t -c "SELECT datname FROM pg_database WHERE datistemplate = false;" ')
do
    echo "Database: \${DBNAME}"
    su - postgres -c "psql -f /tmp/amcheck.sql -v 'ON_ERROR_STOP=1' \${DBNAME}" && EXIT_STATUS=\$? || EXIT_STATUS=\$?
    if [ "\${EXIT_STATUS}" -ne 0 ]
    then
        echo "amcheck failed on DB: \${DBNAME}"
        exit 125
    fi
done
EOF
chmod +x /tmp/run_amcheck.sh

#  
/tmp/run_amcheck.sh > /tmp/amcheck.log

#         exit code  grep’ 
if grep 'amcheck failed' "/tmp/amcheck.log"
then
    echo 'amcheck failed: '
    cat /tmp/amcheck.log
    exit 125
fi
