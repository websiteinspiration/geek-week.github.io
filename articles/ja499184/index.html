C＃8の新しいインターフェイス機能



まず、Andrew TroelsenとPhilip Jepixによるインターフェイスの定義を見てみましょう。抽象メソッドは標準の実装を提供しないため、純粋なプロトコルです。インターフェースによって定義される特定のメンバーは、それがモデル化する動作に正確に依存します。言い換えると、インターフェイスは、特定のクラスまたは構造がサポートすることを選択できる動作を表現します。さらに、テキスト内で次のようにします。インターフェイスは純粋なプロトコルであるため、実装が提供されることはありません。」

まあ、すべては明らかなようです、インターフェイスのほぼ同じ解釈が言語の他の本で説明されますが、C＃8ではインターフェイスがわずかに改訂され、矛盾するタイプの機能が一目で追加されました。C＃8のインターフェースをさらに詳しく見てみましょう。

デフォルトの実装またはデフォルトのメンバーの実装


定義により、インターフェースから継承する場合、インターフェースの各メンバーの動作を実装する必要があります。C＃8では、デフォルトのインターフェースメンバーを実装する機能が導入されました。

interface ICommand
    {
        void exec();
     
        // default implementations
        public void sendNotification(string mes)
        {
            Console.WriteLine(mes);
        }
    }

「exec」メソッドは、慣れ親しんでいるインターフェースの通常の抽象メソッドですが、インターフェースの「sendNotification」メソッドは奇妙に見え、デフォルトの実装を提供します。

それがどのように機能するか見てみましょう：

public class AppFirstCommand : ICommand
    {
        public void exec()
        {
            // some action
            Console.WriteLine("Action complete!");
        }
    }

インターフェースを実装する場合、抽象メソッド「exec」を実装する必要があり、メソッド「sendNotification」の実装を省略できます。便利ですか？

class Program
    {
        static void Main(string[] args)
        {
            ICommand appFirstCommand = new AppFirstCommand();

            appFirstCommand.exec();
            appFirstCommand.sendNotification("Default implementations");
        }
    }

実行結果：

アクションが完了しました！
デフォルトの実装デフォルトの実装で

インターフェースメソッドを参照するには、参照インターフェースタイプの変数を宣言する必要があります。インターフェースリンク：

ICommand appFirstCommand = new AppFirstCommand();

それ以外の場合、AppFirstCommandクラスにはこのメソッドの実装がないため、インターフェイスのsendNotificationメソッドにアクセスできません。これは機能しません：

AppFirstCommand appFirstCommand = new AppFirstCommand();

はい、sendNotificationメソッドの動作をオーバーライドすることもできます。

public class AppSecondCommand : ICommand
    {
        public void exec()
        {
            // some action
            Console.WriteLine("Action complete!");
        }

        // override sendNotification
        public void sendNotification()
        {
            Console.WriteLine("Override method from AppSecondCommand");
        }
    }

class Program
    {
        static void Main(string[] args)
        {
            AppSecondCommand appSecondCommand = new AppSecondCommand();

	    appSecondCommand.exec();
            appSecondCommand.sendNotification();
        }
    }

実行結果：

アクションが完了しました！

AppSecondCommandからのオーバーライドメソッド

この場合、このクラスではsendNotificationメソッドを明示的に実装したため、AppSecondCommandクラスのインスタンスを直接参照します。

「デフォルトの実装」によるアプローチはどのように便利ですか？


デフォルトでは、インターフェースメンバーを使用すると、そのインターフェースの既存の実装とのソースコードの互換性を損なうことなく、将来のバージョンで新しいメンバーをインターフェースに追加できます。便利です。

すべてのC＃アクセス修飾子は、デフォルトで実装されているインターフェースメンバーで使用できることに注意してください。

インターフェースには静的メンバーが含まれる場合があります


interface IStaticMembers
    {
        private static string commandName = "Default command name";

        private void build(string name)
        {
            commandName = name;
        }

        public void setCommandName(string name)
        {
            build(name);
        }
    }

インターフェイスには仮想メンバーが含まれる場合があります


ただし、インターフェイスを継承するクラスは、仮想メンバーをオーバーライドできません。
インターフェースの仮想メンバーのオーバーライドは、インターフェースのみが可能です。

interface IVirtualMembers
    {
        public virtual void sendNotification()
        {
            Console.WriteLine("IVirtualMembers's  sendNotification");
        }
    }

    interface IOverrideVirtualMembers : IVirtualMembers
    {
        void IVirtualMembers.sendNotification()
        {
            Console.WriteLine("IOverrideVirtualMembers's  sendNotification");
        }
    }

また、インターフェイスの抽象的なメンバーを定義することも可能ですが、これは意味がありません。

したがって、C＃8の新しいインターフェース機能の短いリスト：

  • デフォルトの実装またはデフォルトのメンバーの実装。
  • インターフェイスには静的メンバーが含まれる場合があります。
  • インターフェイスには仮想メンバーが含まれる場合があります。
