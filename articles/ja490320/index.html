POSTGRESQLカウントの高速化（*）



PostgreSQLでのカウント（*）が非常に遅いという不満がよくあります。

この記事では、できるだけ早く結果を得ることができるようにオプションを探っていきたいと思います。

カウント（*）がなぜ遅いのですか？


ほとんどの人は、次のリクエストがゆっくり実行されることを問題なく理解しています：

SELECT count(*)
FROM /*   */;

結局のところ、これは複雑なクエリであり、PostgreSQLはそれが含む行数を知る前に結果を計算する必要があります。

しかし、次のクエリが遅いことがわかったとき、多くの人はショックを受けます。

SELECT count(*) FROM large_table;

ただし、もう一度考えてみると、上記のすべてが当てはまります。PostgreSQLは、結果セットをカウントする前に計算する必要があります。「マジックローカウンター」はテーブルに保存されないため（MyISAM MySQLのように）、ローをカウントする唯一の方法はそれらを調べることです。

したがって、カウント（*）は通常、順次テーブルスキャンを実行しますが、非常にコストがかかる可能性があります。

カウント（*）の「*」は問題ですか？


SELECT * FROM ...の "*"はすべての列に適用されます。したがって、多くの人は、count（*）の使用は非効率的であると考え、代わりにcount（id）またはcount（1）を使用します。

ただし、カウント（*）の「*」は完全に異なります。これは単に「文字列」を意味し、まったく展開しません（実際には、「null引数を持つ集約」です）。count（1）またはcount（id）という表記は、実際にはcount（*）よりも低速です。これは、引数がNULLかどうかをチェックする必要があるためです（countは、ほとんどの集計と同様に、引数NULLを無視します）。

したがって、「*」を回避しても何も達成されません。

インデックスのみのスキャンの使用


行数をカウントするために、テーブル全体ではなく小さなインデックスをスキャンするのは魅力的です。ただし、マルチバージョンの同時実行管理戦略のため、これはPostgreSQLではそれほど単純ではありません。行の各バージョン（「タプル」）には、その行が表示できるデータベーススナップショットに関する情報が含まれています。ただし、この（冗長な）情報はインデックスに格納されません。したがって、PostgreSQLはテーブルエントリ（「ヒープタプル」）を調べてインデックスエントリが表示されていることを確認する必要があるため、通常はインデックス内のエントリをカウントするだけでは不十分です。

この問題を軽減するために、PostgreSQLは可視性マップを実装しました。これは、テーブルブロック内のすべてのタプルが全員に表示されるかどうかに関する情報を格納するデータ構造です。
テーブル内のほとんどのブロックが完全に表示されている場合、インデックススキャンでは、可視性を判断するために多数のタプルに頻繁にアクセスする必要はありません。このようなインデックススキャンは「インデックスのみのスキャン」と呼ばれ、多くの場合、インデックスをスキャンして行をカウントする方が高速です。

現在、可視性マップをサポートしているのはVACUUMです。したがって、小さなインデックスを使用してカウント（*）を高速化する場合は、自動バキュームが頻繁に実行されるようにしてください。

ピボットテーブルの使用


上記で、PostgreSQLはテーブルに行数を格納しないと書きました。

このイベントはすべてのデータ変更で発生し、効果が得られないため、このような行数を維持することは大きなオーバーヘッドです。それは悪い取引でしょう。さらに、リクエストごとに異なるバージョンの文字列を参照できるため、カウンターもバージョン管理する必要があります。

しかし、そのようなラインカウンターを自分で実装することを妨げるものは何もありません。
mytableの行数を追跡するとします。あなたは次のようにそれを行うことができます：

START TRANSACTION;
 
CREATE TABLE mytable_count(c bigint);
 
CREATE FUNCTION mytable_count() RETURNS trigger
   LANGUAGE plpgsql AS
$$BEGIN
   IF TG_OP = 'INSERT' THEN
      UPDATE mytable_count SET c = c + 1;
 
      RETURN NEW;
   ELSIF TG_OP = 'DELETE' THEN
      UPDATE mytable_count SET c = c - 1;
 
      RETURN OLD;
   ELSE
      UPDATE mytable_count SET c = 0;
 
      RETURN NULL;
   END IF;
END;$$;
 
CREATE CONSTRAINT TRIGGER mytable_count_mod
   AFTER INSERT OR DELETE ON mytable
   DEFERRABLE INITIALLY DEFERRED
   FOR EACH ROW EXECUTE PROCEDURE mytable_count();
 
-- TRUNCATE triggers must be FOR EACH STATEMENT
CREATE TRIGGER mytable_count_trunc AFTER TRUNCATE ON mytable
   FOR EACH STATEMENT EXECUTE PROCEDURE mytable_count();
 
-- initialize the counter table
INSERT INTO mytable_count
   SELECT count(*) FROM mytable;
 
COMMIT;

すべての処理を1つのトランザクションで実行するため、リング状態が原因で並行トランザクションのデータ変更が「失われる」ことはありません。
これは、すべての同時変更を防止するSHARE ROW EXCLUSIVEモードでテーブルをロックするCREATE TRIGGERコマンドによって保証されます。
欠点は、すべての並列データ変更がSELECTカウント（*）が実行されるまで待機する必要があることです。

これにより、カウント（*）の非常に迅速な代替手段が得られますが、テーブル内のすべてのデータ変更の速度が低下します。遅延制約トリガーを使用すると、同時実行性を向上させるために、mytable_countの行ロックができるだけ短くなります。

このカウンターテーブルは多くの更新を受け取ることができるという事実にもかかわらず、危険はありませんこれらはすべて「ホット」更新（HOT更新）になるため、「テーブルの肥大化」はありません。

本当に数が必要です（*）


最善の解決策は、代替案を探すことです。

多くの場合、近似は十分であり、正確な量は必要ありません。この場合、PostgreSQLがクエリのスケジュールに使用するスコアを使用できます。

SELECT reltuples::bigint
FROM pg_catalog.pg_class
WHERE relname = 'mytable';

この値は自動バキュームと自動分析の両方で更新されるため、10％を超えてはなりません。このテーブルのautovacuum_analyze_scale_factorを減らして、自動分析がより頻繁に実行されるようにすることができます。

クエリ結果の数の見積もり


これまで、テーブル行のカウントを高速化する方法を調査してきました。

ただし、SELECTステートメントが実際にクエリを実行せずに返す行数を知る必要がある場合があります。

明らかに、この質問に対する正確な回答を得る唯一の方法は、リクエストを完了することです。ただし、グレードが十分であれば、PostgreSQLオプティマイザを使用して取得できます。

次の単純な関数は、動的SQLとEXPLAINを使用して、クエリ実行プランを引数として渡し、行数の推定値を返します。

CREATE FUNCTION row_estimator(query text) RETURNS bigint
   LANGUAGE plpgsql AS
$$DECLARE
   plan jsonb;
BEGIN
   EXECUTE 'EXPLAIN (FORMAT JSON) ' || query INTO plan;
 
   RETURN (plan->0->'Plan'->>'Plan Rows')::bigint;
END;$$;

この関数は本質的にSQLインジェクションに対して脆弱であるため、信頼されていないSQLステートメントの処理には使用しないでください。
