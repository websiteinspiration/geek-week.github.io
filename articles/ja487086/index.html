S805-Bプロセッサソフトウェア保護（セキュアブート）

プロセッサ自体に実装されているソフトウェアを保護する方法について説明します。実験では、Comigo Quattroマルチメディアコンソールを選択しました。目標は、Linuxカーネルを起動することです。

短いレビュー


一見すると、コンソールソフトウェアはAndroidのクローンです。ssh経由のアクセスは閉じられます。ディストリビューターからの有効なサブスクリプションでのみ機能します。ストックファームウェアは完全に暗号化されています。戦車のように、すべてが鈍い。

トレーニング


まず、プロセッサーの技術的な説明を詳しく知りました。プロセッサがUARTインターフェースを備えていて、構成レジスタを使用して外部から構成できるさまざまなソース（USB、SDなど）からのシステムのロードをサポートしていました。コンソールにはmSDカード用の場所もありましたが、コネクタははんだ付けされていませんでした。コンフィギュレーションレジスタとUARTピンのスポットを見つけるために、プロセッサをはんだ付けしてトラックを鳴らす必要がありました。すべての連絡先が見つかったら、UARTアダプターを接続してコンソールを起動しました。画面でU-Bootが点滅した。キーを押すとシステムブートが中断され、U-Bootは入力モードに入りましたが、Enterキー以外のキーには反応しませんでした。そしてこの抜け穴もふさがれた。

この段階では、元のU-Bootでは何もできなかったため、mSDカードからの起動を確認することにしました。これを行うには、コネクタをはんだ付けし、構成レジスタの一対の接点を質量に接続して、プロセッサがカードからシステムにロードするようにしました。新しいU-Bootは、同じプロセッサをベースとした実験用ボードODROID-Cからのものです。





これらすべてを収集した後、私はSDカードを挿入し、コンソールをオンにしました...私はこれを見ました：

SERIAL:4;STS:0;BOOT:1;INIT:0;READ:0;CHECK:FFFFBF00;USB:3;

そして、このテキストは毎秒何度も繰り返されました。プロセッサがU-Bootをまったくロードしなかったため、気分が悪くなりました。

理由を探す


プロセッサでセキュアブートがアクティブになっていて、起動する前にコードの信頼性をチェックしていることがわかりました。CHECK：FFFFBF00パラメータは、mSDカードからダウンロードされたコードが何らかの理由でプロセッサに完全に適合しないことを示していました。比較のために、オリジナルのダウンロードは次のように始まりました：

QA5:A;SVN:B72;POC:17F;STS:0;BOOT:0;INIT:0;READ:0;CHECK:0;PASS:0;

プロセッサは何をチェックしましたか？AMLogicのU-Bootコースでこの答えを見つけたいと思っていました。少し検索したところ、U-BootのSPL部分がこのような構造になってしまうことがわかりました。

   typedef struct {
      unsigned int   nSizeH;         ///4@0
      struct st_secure{
         unsigned int   nORGFileLen;  ///4@4

      unsigned int   nVer;           ///4@248

   }st_aml_chk_blk; //256

少し余談です。2段階システムブートは、組み込みハードウェアでよく使用されます。それら。プロセッサROMは、最初にコードの一部（いわゆるSPL）をロードし、制御をコードに転送します。次に、メイン設定の後に、2番目のほとんど（TPL）をロードし、制御をそれに転送します。さて、最終的な設定の後、TPLはカーネルをロードして起動します。

今後は、SPL自体が32 KBで構成されていると言います。その最後には、st_aml_chk_blk構造があり、これは復号化された形式では次のようになります。


したがって、プロセッサはこのデータブロックを使用してSPLをチェックします。

解決策を探しています


プロセッサーにU-Bootをロードさせるために、多くのオプションを試しました。しかし、それらすべては望ましい結果をもたらしませんでした-プロセッサが頑固にCHECK：FFFFBF00を発行しました。私は徐々に保護が100％行われ、穴はないという結論に達しました。

別の試みが失敗した後、私はプレフィックスを残して、JTAGの接続を考えて（もちろんお茶のために）キッチンに行き、戻ったとき、私はこの結果を待っていました。

分析


まず、プロセッサとヒューズのROMをダンプするプログラムを作成しました（これは、1回しかフラッシュできないプロセッサ内のメモリの一部です。ここには、AES、公開RSAキー、およびいくつかのプロセッサ構成が格納されています）。このようにしてAESキーを取得したら、元のU-Bootを復号化してROMを分析しました。 SPLをロードする原理は次のとおりです。ROMは最初にヒューズから構成を読み取り、それを変数として保存します。次に、フラッシュドライブから32 KBをメモリにロードし、署名と暗号化の検証について構成が示されていることを確認しました。私たちのケースでは、ROMが指数eとモジュールNをヒューズから読み取り、PublicKey（e、N）を収集してst_aml_chk_blkを復号化し、st_aml_chk_blkなしでSPLからsha2を解決して、st_aml_chk_blkのキーで何が起こったかを比較しました。そして、不一致が発生した場合、彼はそれ以上の措置を講じませんでした（新しいRSA公開鍵をfuseで登録することはできず、もちろん私たちもプライベートディストリビューターを持っていないため、新しい署名を作成することはできません）。それで、プロセッサが私のSPLを実行したのはなぜですか？署名がまったく一致しなかった...
RSAをサポートするために、PolarSSLライブラリがROMに含まれています（高い確率で）（ROMを調べたところ、このライブラリのリソースとの類似点が見つかりました）。しかし、私の記憶が適切に機能するのであれば、彼はハードウェアレベルでAESアルゴリズムをサポートしました。 ROMは独自のメカニズム（mallocやmemfreeなど）を使用して空きメモリを制御しました。 PolarSSLライブラリから関数を呼び出す前に、ROMはそれに特定のメモリを割り当てましたが、場合によってはそれを解放しない場合がありました。たとえば、この試みでSPLで使用した無効なブロックを解読できないためにRSA機能が落ちた場合。どうした？ ROMがメモリを割り当て、偽の署名を解読しようとし、メモリを解放するのを忘れ、エラーCHECK：FFFFBF00をスローし、サイクルが繰り返されました。ある時点で、別の新しいメモリが変数をヒューズから消去し、どのROMが最初に保存したか。次のサイクルでは、この変数の値はすでに0でした。つまり、持っているすべてのものをロードして実行するだけです。
また、原則としておもしろいものはありませんでした。元のU-Bootをやり直し、TPLチェック、bootmコマンドカーネルチェックを削除し、Hushインタープリターをアクティブにして、コマンドに応答するようにしました。カーネルを復号化し、BusyBoxで最小値を構築した後、やり直したU-BootからLinuxを起動し、40秒でロードを開始しました。
