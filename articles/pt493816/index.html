Guia Git Número da peça 1: tudo o que você precisa saber sobre o diretório .git



Começar a usar o Git é como visitar um novo país cujo idioma você não conhece. Embora esteja claro onde você está e para onde ir, tudo está bem, mas se você se perder, grandes problemas começam.

Existem vários tutoriais sobre comandos do Git publicados na Internet, mas neste artigo, o trabalho do Git é examinado mais profundamente do que apenas aprender os comandos.

Esta é a primeira parte do guia Git do blog Pierre de Wulf traduzido pela equipe do Mail.ru Cloud Solutions . 

Novos usuários acham difícil se sentir confortável com o Git. Essa é uma ferramenta poderosa, mas, infelizmente, não é muito fácil de aprender. Muitos conceitos novos, comandos que executam ações diferentes, se o arquivo é passado como parâmetro ou não, feedback pouco claro ...

Provavelmente, a única maneira de superar todas essas dificuldades é aprender um pouco mais do que apenas git commit / push, para entender como o Git funciona.

Pasta .Git


Quando você cria um novo repositório com o comando git init, o Git cria uma pasta mágica, .git. Ele contém tudo o que você precisa para o Git funcionar. Se você deseja remover o Git do seu projeto, mas deixar os arquivos do projeto em disco, basta excluir a pasta .git. Embora quem possa precisar disso?

    ├── HEAD
    ├── branches
    ├── config
    ├── description
    ├── hooks
    │ ├── pre-commit.sample
    │ ├── pre-push.sample
    │ └── ...
    ├── info
    │ └── exclude
    ├── objects
    │ ├── info
    │ └── pack
    └── refs
     ├── heads
     └── tags


Aqui está o conteúdo de uma pasta .git típica antes do seu primeiro commit:

  1. CABEÇA - consideraremos isso mais tarde.
  2. config — , , , url , , email . git config, .
  3. description — gitweb .
  4. hooks — , Git. , , / commit/rebase/pull… . push .
  5. Os arquivos info - exclude - que você não deseja incluir no repositório são descritos aqui. A funcionalidade desse arquivo é a mesma do arquivo .gitignore, exceto que ele não é transferido para o repositório. Na prática, geralmente .gitignore é suficiente para todas as tarefas.

O que há dentro do commit?


Cada vez que você cria um arquivo e confirma as alterações, o Git arquiva o arquivo e o armazena em sua estrutura de dados. Um objeto arquivado é criado com um nome exclusivo e armazenado na pasta de objetos.

Antes de examinar a pasta do objeto, vamos esclarecer o que é a confirmação. Uma confirmação é uma pepita do estado atual dos arquivos em uma pasta de trabalho, mas não apenas isso.

De fato, quando você confirma alterações, o Git faz apenas duas coisas:

  1. Se o arquivo na pasta de trabalho não foi alterado, ele simplesmente adiciona o nome do arquivo compactado (hash) ao instantâneo.
  2. Se o arquivo na pasta de trabalho foi alterado, ele o compacta, coloca na pasta de objetos e adiciona o nome do arquivo compactado (hash) ao instantâneo.

Obviamente, aqui tudo é descrito de uma maneira um pouco simplificada, no entanto, isso é suficiente para entender os processos em andamento.

Assim que um instantâneo é tirado, ele também é arquivado e nomeado com um hash e, em seguida, colocado na pasta de objetos.

├── 4c
│ └── f44f1e3fe4fb7f8aa42138c324f63f5ac85828 // hash
├── 86
│ └── 550c31847e518e1927f95991c949fc14efc711 // hash
├── e6
│ └── 9de29bb2d1d6434b8b29ae775ad8c2e48c5391 // hash
├── info // let's ignore that
└── pack // let's ignore that too


É assim que a pasta de objetos se parece depois que eu criei o arquivo_1.txt e o confirmei. Observe que, se o hash do seu arquivo começar com "4cf44f1e ...", o Git o salvará com o nome "f44f1e ..." em uma subpasta chamada "4c". Assim, os arquivos serão dispostos em 256 subpastas e cada um não terá muitos arquivos.

Como você pode ver, temos três hashes. Um para o arquivo_1.txt, o segundo para uma captura instantânea feita no commit. Para que serve o terceiro? O terceiro hash é criado porque o commit também é um objeto, também é arquivado e colocado na pasta de objetos.

Você precisa se lembrar que o commit consiste em quatro coisas:

  1. O nome (hash) da captura instantânea do diretório de trabalho.
  2. Comente.
  3. Informações sobre quem executou o commit.
  4. O hash do commit pai.

Veja você mesmo o que acontece se você descompactar o arquivo de confirmação:

git cat-file -p 4cf44f1e3fe4fb7f8aa42138c324f63f5ac85828

E aqui está o que você verá:

tree 86550c31847e518e1927f95991c949fc14efc711
author Pierre De Wulf 
<test[@gmail.com](mailto:pierredewulf31@gmail.com)> 1455775173 -0500
committer Pierre De Wulf 
<[test@gmail.com](mailto:pierredewulf31@gmail.com)> 1455775173 -0500
commit A

Você vê, como esperado, o hash do instantâneo, o autor e o comentário do commit. Duas coisas são importantes aqui:

  1. O hash do instantâneo "86550 ..." também é um objeto e pode ser visto na pasta de objetos.
  2. Como esse é o primeiro commit, ele não possui um commit pai.

O que realmente está na foto?

git cat-file -p 86550c31847e518e1927f95991c949fc14efc711
100644 blob e69de29bb2d1d6434b8b29ae775ad8c2e48c5391 file_1.txt

Aqui vemos o objeto que estava em nosso armazenamento de objetos, o único objeto em nossa imagem.

Ramo, tags, HEAD são a mesma coisa


Agora você entende que tudo no Git pode ser obtido através do hash correto. Vamos dar uma olhada no HEAD agora. Então o que há lá?

cat HEAD
ref: refs/heads/master

Não há hash, e isso faz sentido, pois HEAD é um ponteiro para o topo do ramo com o qual você está trabalhando. Se você olhar o arquivo refs / heads / master, verá:

cat refs/heads/master
4cf44f1e3fe4fb7f8aa42138c324f63f5ac85828
 

Parece familiar? Naturalmente, é um hash do primeiro commit! Isso mostra que tags e ramificações são apenas ponteiros para uma confirmação. Entendendo isso, é possível excluir todas as tags que você deseja, todos os ramos que deseja e o commit que eles apontaram permanecerá no lugar. A única coisa, será mais difícil acessá-lo. Se você quiser saber mais sobre isso, consulte o livro git .

Último comentário


Depois de ler, deve ficar óbvio para você que tudo o que o Git faz é arquivar sua pasta de trabalho e colocá-la na pasta de objetos com algumas informações adicionais. Se você estiver familiarizado o suficiente com o Git, terá controle total sobre quais arquivos serão incluídos no commit e quais não.

Acredito que a confirmação não é um instantâneo da pasta de trabalho, mas um instantâneo dos arquivos que você deseja confirmar. E onde o Git armazena a lista de arquivos que você deseja confirmar? Ele salva esta lista em um arquivo de índice. Não vamos nos aprofundar nessa questão; se você estiver interessado, mais detalhes podem ser encontrados aqui .
