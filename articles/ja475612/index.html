与えられたとき、主張と実装の信頼性

でアサーションに関する最後のテキスト、重要な何かが欠けていたと合意されていませんでした。

これはアサーションにどのように関係していますか？

これの根底にある考え方は単純です-アサーションの数を制限したいのです。

小さな例を考えてみましょう。

メソッドに有効なオブジェクトを渡して、追加のサービスを呼び出し、そのプロパティを充実させてみましょう。

テストの前にコードを書くべきではないので、これは全体像を伝えるためだけのものであると想定します。

public User enrichUser(User validUser){
      user.setDetails(enrichmentService.getUserDetails(validUser.getId()));
      return user;
}


validUserバリエーションには興味がありません。 nullではなく、常にIDがあり、有効です。これは前提条件です。与えられた。

実際、enrichmentServiceの失敗と成功という2つの条件を考慮する必要があります。これは条件です。いつ。

1つを他から区別する方法？-が必要な場合、他のケースの検証は必要ありません。それら。 whenValidUserはwhenInvalidUserを必要とし、givenValidUserは必要としません。

そしてenrichmentservice == null？依存関係を挿入した場合は、構成のこの部分を検討し、テストでは考慮しないことができます。もちろんです。記載しても意味がないという前提条件があります。

メソッドがユーザーのみを受け入れる場合、チェックするスクリプトの数が増加します。いつになるかを与えられます。

別の質問-しかし、厳密に言えば、偶然または故意に、メソッドはユーザーを台無しにする可能性がありますか？調べてみませんか？

私たちが答えるなら-はい、そうしなければなりません。これは大きな負担になる可能性があることを認めなければなりません。ユーザーは50のプロパティを持つことができ、各メソッドでそれらをチェックするとコストが高くなる可能性があります。ライブラリを使用した一括チェックがありますが、テストの重要な機能が失われます。つまり、必要な機能を規定し、実装曲線の背後にある汚れを拾わないようにします。

これは空港のセキュリティと比較できます。誰かがチケット、誰かの荷物、誰かのポケットをチェックし、チェックがすべてのステップで複製されるわけではありません。それら。アセットは、実装が何をするかを信頼し、規定した​​ものだけをチェックします。

したがって、適切に構築された前提条件は、スクリプトとアサーションの数を減らします。条件付きでのみアサーションを作成します。

残念ながら、Cucumberのようなフレームワークは大きな違いはありません。前提条件自体は、Given、that Whenが同じ、純粋に説明的な用語であることをチェックしません。

より複雑なケースでは、臭いのあるものを区別することが難しい場合があります。たとえば、多くのテストが続けて同じことをチェックしたり、テストの一部の条件が忘れられたり、不適切として無視されたりします。

これが、仕様を再構築して再設計する正当な理由です。
