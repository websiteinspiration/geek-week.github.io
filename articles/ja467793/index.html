一人のインターネット会社の退屈な技術スタック


ListenNotes.comでの検索結果

Listen Notes  は検索エンジンとポッドキャストデータベースです。技術は実際には非常に退屈です。 AI、ディープラーニング、ブロックチェーンはありません。「AIの実装を発表する必要がある場合、実際のAIは使用していません」 :)

この記事を読んだ後、私のプロジェクトを繰り返すか、簡単にそのようなことを行うことができます。多くの開発者を雇う必要はありません。 Instagramが5750万ドルを調達し、 10億ドルでFacebookに行ったとき、 13人の従業員しかいなかったことを思い出してください。これらは開発者だけではありません。 Instagramの購入は2012年の初めに行われました。今が2019年であり、今日では、小さなエンジニアリングチームで有意義なものを簡単に作成できるようになりました。

Listen Notesを見たことがない場合は、今すぐ試してください。



概観


プロジェクトの要件または機能から始めましょう。

Listen Notesには2つの機能があります。

  • ポッドキャストリスナーのウェブサイト。検索エンジン、ポッドキャストのデータベース、後で聴くための再生リスト、ポッドキャストから切り取られた保存済みクリップ、および指定されたキーワードがインターネット上のいくつかの新しいポッドキャストで言及されたときの通知があります。
  • 開発者向けのポッドキャストとAPIを検索します。APIの使用状況を追跡し、加入者からお金を受け取り、カスタマーサポートを提供する必要があります。

バックエンド


バックエンド全体はDjango / Python3で書かれています。オペレーティングシステムはUbuntuです。UWSGIは、

Webトラフィックを処理するために使用されます。 uWSGIプロセスの前にnginxをインストールしました。ロードバランサーとしても機能します。 メインのデータウェアハウスはPostgreSQLです。これには、長年にわたって開発と運用の広範な経験があります。実績のあるテクノロジーにより、夜は安らかに眠れます。Redisはさまざまな目的で使用されます（キャッシュ、統計など）。Elasticsearchがどこかで使用されていることは容易に想像できます。はい、ポッドキャストのインデックス作成と検索クエリの提供に使用しています。これは、ほとんどの退屈な会社と同じです。



Celeryはオフライン処理に使用され、Celery Beatは Cronタスクに似ているが少し良いタスクをスケジュールすることを目的としています。将来的にListen Notesが人気になり、CeleryとBeatがスケーリングの問題を引き起こす場合、おそらく前の雇用者のために行った2つのプロジェクトに切り替えるでしょう：ndkaleとndscheduler。

Supervisordは、各サーバーのプロセスを管理します。

ちょっと待って、Docker、Kubernetes、サーバーレスアーキテクチャはどうですか？何もない。経験を積むと、あまりやらないことを学びます。実際、私は2014年に前職でDockerと少し仕事をしました。10億ドル相当の平均的な新興企業にとって何が良かったのかは、小さな1人の会社には必要ないようです。

フロントエンド


Webインターフェイスは、主にReact  + Redux  + Webpack  + ESを使用して構築されています。最近かなり標準的です。本番環境にデプロイすると、JSパッケージがAmazon S3にダウンロードされ、CloudFrontを通じて発行されます。

ListenNotes.comでは、ほとんどのWebページはサーバー側（Djangoテンプレート）で半分レンダリングされ、クライアント側（React）で半分レンダリングされます。Webページテンプレートはサーバーから取得され、クライアント側ではインタラクティブなWebアプリケーションが主にレンダリングされます。しかし、私の怠惰といくつかの潜在的なSEOの利点のために、いくつかのWebページがサーバー上で完全に準備されています。

マシンの準備とコードの配置


Ansible はプロビジョニングに使用されます。基本的に、各タイプのサーバーに必要な構成ファイルとソフトウェアを示すために、一連のyamlファイルを作成しました。ボタンをクリックするだけで、すべての正しい構成ファイルとすべてのインストール済みソフトウェアを備えたサーバーをデプロイできます。これらのAnsible yamlファイルのディレクトリ構造は次のとおりです


。ディレクトリに名前を付けた方がいいでしょう。ただし、これでも十分です

。Ansibleは、コードを本番環境にデプロイするのにも役立ちます。基本的に、私はmacOSで動作するdeploy.shラッパースクリプトを持っています。

./deploy.sh Production HEAD Web

このスクリプトは3つの引数を取ります。

  • 環境：生産またはステージング。
  • listennotesのバージョン：HEAD リポジトリは、「最新バージョンをデプロイするだけ」を意味します。SHAコミットが指定されている場合、特定のバージョンのコードがデプロイされます。これは、不適切なデプロイメントの後にロールバックする必要がある場合に特に役立ちます。
  • サーバーのタイプ：ウェブ、ワーカー、API、またはすべて。一度にすべてのサーバーにデプロイする必要はありません。ときどきJavaScriptコードに変更を加えた後、APIやワーカーに触れずに、それをWebにのみ展開する必要があります。

デプロイプロセスは主にAnsible yamlファイルで構成されており、もちろん非常に簡単です。

  • 私のMacbook Proでは、それがWebサーバーのデプロイの場合、JavaScriptパッケージが作成され、S3にアップロードされます。
  • ターゲットサーバーでは、 gitはlistennotesリポジトリをタイムスタンプと呼ばれるフォルダーに複製し、特定のバージョンをチェックし、新しいPython依存関係があればインストールします。
  • ターゲットサーバー上で、シンボリックリンクは前述のタイムスタンプと呼ばれるフォルダーを指し、その後、supervisorctlを使用してサーバーを再起動します。

ご覧のとおり、私はこれらの豪華なCIツールを使用していません。実際に機能する最も単純で信頼性の高いツールのみ。
