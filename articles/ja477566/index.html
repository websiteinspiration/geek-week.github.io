Huawei Cloud：PVS-Studioの曇り


今世紀、誰もがクラウドサービスについて聞いたことがあります。多くの企業がこの市場セグメントをマスターし、さまざまな方向でクラウドサービスを作成しています。私たちのチームはまた、PVS-Studioコードアナライザーとの統合という点で、これらのサービスに最近関心を持っています。私たちの常連の読者は、今回チェックするプロジェクトの種類をすでに推測していると思います。Huaweiが提供するクラウドサービスのコードがその選択につながりました。

前書き


PVS-Studioチームをフォローしている場合は、最近クラウドテクノロジーに非常に興味を持っていることに気づいたでしょう。このトピックに関連する記事はすでにいくつか公開されています。

  • PVS-Studioがクラウドに移行：Azure DevOps
  • PVS-Studioがクラウドに移行：Travis CI
  • PVS-Studioがクラウドに移行：CircleCI
  • PVS-Studioがクラウドに移行：GitLab CI / CD

それで、次の記事のために別の興味深いプロジェクトをピックアップしていたとき、Huaweiからメールで求人を受け取りました。この会社に関する情報を収集したところ、彼らは独自のクラウドサービスを持っていることがわかりましたが、主なことは、これらのサービスのソースコードがGitHubで無料で入手できることです。これが、この記事でこの会社を選択した主な理由になりました。ある中国人の賢者が言ったように：「事故は偶然ではありません。」

次に、アナライザーについて少し説明します。PVS-Studioは、C、C ++、C＃、およびJavaで記述されたプログラムのソースコードのエラーと脆弱性を識別する静的コードアナライザーです。アナライザーは、Windows、Linux、およびmacOSプラットフォームで動作します。Visual StudioやIntelliJ IDEAなどの従来の開発環境のプラグインに加えて、アナライザーはSonarQubeおよびJenkinsと統合することができます。

  • PVS-Studio分析結果のSonarQubeへの統合
  • JenkinsでのPVS-Studioの実行

プロジェクト分析


記事の情報を検索する過程で、Huaweiには、クラウドサービスの情報、ガイド、およびソースコードを入手できるデベロッパーセンターがあることがわかりました。これらのサービスを作成するためにさまざまなプログラミング言語が使用されましたが、Go、Java、Pythonなどの言語が最も人気があることが判明しました。

私はJavaを専門としているため、プロジェクトはそれに応じて選択されました。この記事で分析したプロジェクトのソースは、GitHubのhuaweicloudリポジトリから入手できます。

プロジェクトを分析するには、いくつかのアクションを実行するだけで済みました。

  • リポジトリからプロジェクトを取得します。
  • Javaアナライザーを開始するための手順を使用して、各プロジェクトで分析を開始します。

プロジェクトを分析した後、私が注目したい3つだけを選び出すことができました。これは、残りのJavaプロジェクトのサイズが小さすぎるためです。

プロジェクト分析結果（警告の数とファイルの数）：

  • huaweicloud-sdk-java：31-高、2-中、16-低、2700以上のファイル。
  • huaweicloud-dis-agent：7-高、6-中、6-低、100以上のファイル。
  • huaweicloud-sdk-java-dis：15-高、6-中、16-低、270以上のファイル。

警告はほとんどないので、一般的にはコードの品質が良いと言えます。さらに、すべての操作が有効なエラーであるとは限りません。これは、正しいコードと正しくないコードを区別するのに十分な情報がアナライザーにない場合があるためです。したがって、アナライザーの診断は、ユーザーから受け取った情報の助けを借りて毎日改善されています。「静的アナライザーが誤検知と戦う方法と理由」の記事も参照してください。

プロジェクトを分析する過程で、最も興味深い警告を選択しました。これについては、この記事で説明します。

UntrustedSSL クラスのコンストラクターで例外が発生した場合、この例外に関する情報は、LOGロガーを使用してキャッチブロックに記録されます。ただし、静的フィールドの初期化順序のため、INSTANCEフィールドの初期化時のLOGはまだ初期化されていません。したがって、コンストラクターで例外情報をログに記録すると、NullPointerExceptionがスローされます。この例外は、静的フィールドの初期化中に例外が発生した場合にスローされる別のExceptionInInitializerError例外の原因です。上記の問題を解決するために必要なのは、LOG初期化を配置することだけです。INSTANCEを初期化する前。

public static String buildCanonicalHost(URL url) 
{

  }

}

このコードを記述したとき、型Stringのオブジェクトの不変性のため、concatメソッドを呼び出してもホスト文字列が変更されないことは考慮されていませんでした。メソッドを正しく動作させるには、結果メソッドの連結変数ホストをブロックifに割り当てる必要があります。正しく：

if (port > -1) {
  host = host.concat(":" + Integer.toString(port));
}

結論
