ジョブNeoQUEST-2020の例に関するリバースクライアントサーバーAPKのガイド


今日、豊富なプログラムがあります（一度に非常に多くのサイバーセキュリティ領域があります！）：Androidアプリケーションの逆コンパイル、URLを取得するためのトラフィックのインターセプト、ソースコードなしのapkの再構築、暗号解読者との連携などを検討してください。

伝説のNeoQUEST-2020によると、主人公はキーを取得するために使用する必要がある古いロボット部品を見つけました。はじめましょうか！

1.リバーシムAPK


だから、私たちの前に、少し分解されたロボットからなんとかして抽出することができた少しです- どういうわけかキーを取得するのに役立つはずのapkアプリケーション。最も明白なことをしましょう：apkを実行し、その機能を確認します。ミニマル以上のアプリケーションインターフェイスは間違いなく、これはリモートサーバーからファイルをダウンロードできるカスタムFileDroidファイルクライアントです。さて、それは簡単に見えます。電話をインターネットに接続し、ダウンロードを試行します（すぐにkey.txt-ええと、どうなったのでしょうか？）-失敗しましたが、サーバーにファイルがありません。



複雑さの観点から次のイベントに進みます-JADXを使用してapkを逆コンパイルしますそして、幸いなことに、まったく難読化されていないアプリケーションのソースコードを分析します。私たちの現在のタスクは、リモートサーバーがダウンロード用に提供するファイルを理解し、それらからキーを持つファイルを選択することです。

まず、com.ctf.filedroid.MainActivityクラスから始めます。このクラスには、「ダウンロード」ボタンのクリックが処理される、最も興味深いonClick（）メソッドが含まれています。このメソッド内では、ConnectionHandlerクラスが2回呼び出されます。最初に、ConnectionHandler.getToken（）メソッドが呼び出され、その後、ユーザーによって要求されたファイルの名前を渡すConnectionHandler.getEncryptedFile（）が呼び出されます。



ええ、つまり、最初にトークンが必要です！入手の過程でもう少し検討していきます。
ConnectionHandler.getToken（）メソッドは2行の入力を受け取り、GETリクエストを送信して、これらの行を「crc」および「sign」パラメータとして渡します。応答として、サーバーはデータをJSON形式で送信します。アプリケーションからアクセストークンを抽出し、それを使用してファイルをダウンロードします。もちろん、これはすべて問題ありませんが、「crc」および「sign」とは何ですか。



これを理解するために、Checksクラスにさらに移動し、badHash（）およびbadSign（）メソッドを提供してください。 1つ目は、classes.dexとresources.arscからチェックサムを計算し、これらの2つの値を連結して、Base64でラップします（フラグ10 = NO_WRAP | URL_SAFEに注意してください。これは便利になります）。そして、2番目の方法はどうですか？また、アプリケーション署名のSHA-256フィンガープリントについても同様です。えっと、FileDroidは再構築したくはないようです:(



OK、トークンを受け取ったとしましょう。次は何ですか？これをConnectionHandler.getEncryptedFile（）メソッドの入力に渡します。このメソッドは、要求されたファイルの名前をトークンに追加し、別のGET要求を生成します。今回は「token」および「file」パラメーターを使用します。サーバーは（メソッドの名前で判断して）応答として、/ sdcard /に保存されている暗号化されたファイルを送信します。

したがって、少し小計をまとめると、2つのニュースがあり、...どちらも悪いことです。第一に、FileDroidはapkを変更するための熱意を実際にサポートしておらず（チェックサムと署名がチェックされています）、第二に、サーバーから受信したファイルは暗号化されることを約束します。

問題が解決したら、問題を解決します。主な問題は、ダウンロードする必要があるファイルがまだ分からないことです。ただし、ConnectionHandlerクラスを調べていると、getToken（）メソッドとgetEncryptedFile（）メソッドの間に、FileDroid開発者がgetListing（）という別の非常に魅力的なメソッドを忘れていることに気づかざるを得ませんでした。したがって、サーバーはこのような機能をサポートしています...これが必要なもののようです！



リストを取得するには、よく知られている「crc」と「sign」が必要です。問題ではなく、どこから来たのかはすでにわかっています。値を読み取り、GETリクエストを送信します。GETリクエストはどこに送信しますか？最初にリモートサーバーのURLを取得すると便利です。ええと、MainActivity.onClick（）に戻り、netPath引数が生成されてgetToken（）メソッドとgetEncryptedFile（）メソッドを呼び出す方法を確認します。

3. Fridaでapkを再構築する


この記事の一部として、Fridaフレームワークを使用した、より簡単な（そして私たちの意見では、より美しい）ソリューションメソッドを検討します。これにより、ランタイムに必要な引数で任意のapkアプリケーションメソッドを実行できます。これを行うには、ルート権限のある電話またはエミュレータが必要です。次の行動計画を想定しています。

  1. PCとテスト用電話へのFridaコンポーネントのインストール。
  2. トークンまたはリスティングリクエストに一致するURLを回復し、ファイルをダウンロードします（Fridaを使用）。
  3. 元のアプリケーションのチェックサムと署名の値を取得します。
  4. サーバーに保存されているファイルのリストを取得し、目的のファイルを識別します。
  5. ファイルをダウンロードして復号化します。

4.サーバー上のファイルのリストを取得する


最後に、リモートサーバーが少し身近になりました。これで、サーバーが次の方法でリクエストをサポートすることがわかりました。

  1. filedroid.neoquest.ru/api/verifyme?crc= {crc}＆sign = {sign}
  2. filedroid.neoquest.ru/api/list_post_apocalyptic_collection?crc= {crc}＆sign = {sign}
  3. filedroid.neoquest.ru/api/file?file= {file}＆トークン= {token}

利用可能なファイルのリストを取得するために、元のアプリケーションのチェックサムと署名の値を計算し、それらをbase64でエンコードする必要があります。

python3のこのようなスクリプトを使用すると、これを実行できます。

ネタバレ 
import hashlib
import binascii
import base64
from asn1crypto import cms, x509
from zipfile import ZipFile


def get_info(apk):
    with ZipFile(apk, 'r') as zipObj:
        classes = zipObj.read("classes.dex")
        resources = zipObj.read("resources.arsc")
        cert = zipObj.read("META-INF/CERT.RSA")
        crc = "%s%s" % (get_crc(classes), get_crc(resources))
        return get_full_crc(classes, resources).decode("utf-8"), get_sign(cert).decode("utf-8")


def get_crc(file):
    crc = binascii.crc32(file) & 0xffffffff
    return crc


def get_full_crc(classes, resources):
    crc = "%s%s" % (get_crc(classes), get_crc(resources))
    return base64.urlsafe_b64encode(bytes(crc, "utf-8"))


def get_sign(file):
    pkcs7 = cms.ContentInfo.load(file)
    data = pkcs7['content']['certificates'][0].chosen.dump()
    sha256 = hashlib.sha256()
    sha256.update(data)
    return base64.urlsafe_b64encode(sha256.digest())  

get_info('filedroid.apk')

アプリケーションに署名するための同様の手順を実行するには、JADXウィンドウで「APK署名」セクションに移動し、「SHA-256フィンガープリント」の値をコピーして、base64でバイト配列としてエンコードします。



重要：元のapkでは、base64エンコードはURL_SAFEフラグを使用して実行されます。文字「+」と「/」の代わりに、「-」と「_」がそれぞれ使用されます。これがセルフコーディングでも確認されるようにする必要があります。これを行うには、オンラインでエンコードするときに、「ABCDEFGHIJKLMNOPQRSTUVWXYZabcde fghijklmnopqrstuvwxyz0123456789 + /」で使用されているアルファベットを「ABCDEFGHIJKLMNOPQRSTUVWXYZabc script64を使用して64個のpythonスクリプトコードスクリプトで数文字だけに置き換えます。

5.ピンチ暗号を追加...


ええと、早すぎてFileDroidを延期しました。 JADXに戻り、ファイルクライアントの開発者が私たちに役立つものを残していないか確認します。はい、これはコードのクリーンアップが明らかに人気がない場合に当てはまります。未使用のencryptFile（）メソッドは、ConnectionHandlerクラスでの注意を静かに待っています。何があるの？

暗号化AESモードCBC、sinhroposylkaは最初の16バイトを占有します...怠惰-進歩のエンジン。Fridaを再度使用して、私たちの0p3n1fuw4nt2esk4p3.jpgを簡単に解読することをお勧めします。しかし、暗号化キーとして渡すだけですか？多くのオプションはありませんが、別の「忘れられた」savePlainFile（文字列ファイル、文字列トークン）メソッドが存在する場合、その選択は明白です。
