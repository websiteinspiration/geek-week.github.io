宇宙事故は開発者として私に何を教えましたか


原文：悪魔火星人のブログ「Martian Chronicles」のAndrey Sitnikによる記事「宇宙での事故から開発者として学んだこと」は、PostCSSとAutoPrefixerの著者であるAndrey Sitnik がソビエト連邦による宇宙探査に関連するいくつかのストーリーを取り上げました。アンドレイがオープンソース運動の開発者および参加者として成長するために、Andreiが彼らから学んだ教訓を学びます。ドッキングの失敗、大気への劇的な進入、宇宙船間のレールに沿ったユニークな移行-これはすべて、現代のWeb開発とどのように関係していますか？このすべてについて投稿で読んでください！



宇宙探査は私が覚えているのと同じくらい私に興味を持っていました。個人的に私を知っている人々は、彼らが望むよりも宇宙についてもっと多くを聞いた。邪悪な火星人に加わる前に、私はロシア語版のウィキペディアを管理しており、私の好きな趣味の1つは宇宙関連の記事の編集でした。バイコヌールとケープカナベラルでの打ち上げを見学しましたが、宇宙探査の取り組みについて学ぶほど、この知識は開発者としての私に影響を与えました。 

プログラムを書くことは、ロケットを構築すること（ほとんどの場合）ほど難しいことではありませんが、ソフトウェアエンジニアである私たちは、複雑なシステムを作成する大規模なチームで作業することがよくあります。そして宇宙探検家として、時々私たちは複雑さとの戦いに負けます。

「1回の緊急打上げで、何十もの成功したシステムよりも、システムを学び、改善することができます」

- ロケットおよび宇宙ロケット複合体の自律制御システムの分野の専門家、デザイナー、専門家であるニコライピリギン。

エラーはすべての宇宙プログラムで発生しますが、すべてが悲劇的なわけではありません。この記事では、宇宙探査のソビエトとロシアの歴史の例について話します。これらは、熱狂的なコミュニティの外ではほとんど知られていません（私たちは西洋のコミュニティについて話します-約Transl。）。これらの物語はうまく終わりました。

最初のレッスン：ユーザーを責めないでください*


* ...そして、それぞれのアピールに変更を加えます。

1960年代後半、アメリカとソ連は新世代の宇宙船の研究を完了しました。アポロと北軍は実験的な水星/ジェミニと東/サンライズの後の大きな前進でした。新しい船は地球の軌道だけでなく、月への飛行とその逆の飛行のためにも機能するように設計されました。

1968年10月、ソ連はソユーズ1号の災害から回復し、新たな試みを行う準備ができていました。軌道上で2隻の船舶の手動ドッキングを行うことが初めて計画されました。自動ソユーズ-2は最初に宇宙に行くことでした。その後、ソユーズ3号のGeorgy Beregovoiが同じ軌道に入り、船をドッキングしました。


ソユーズ4、ソユーズ8、ソユーズ10ウラジミールシャタロフのパイロットは、2隻の船のドッキングを示しています。

打ち上げはうまくいきました。わずか1.5時間で、ソユーズ-3はドッキング距離にありました。操縦は、地球の影で「夜」の時間のために計画されました。ただし、手動ドッキングの試みはすべて失敗しました。

そして、両方の船が日側にあったときだけ、Beregovoiはエラーを発見しました：ソユーズ2は縦軸に沿って180度回転しました。

その時までにすべての燃料供給がドッキングの試みのために使い果たされていたので、Beregovoiは任務を完了して4日後に地球に戻るように命じられました。新聞は飛行が成功したと呼び、Beregovoyはソビエト連邦の英雄の称号を授与され、すぐに昇進しました。ただし、このストーリーから正しい教訓が得られ、その後のすべての結合に新しいルールが導入されました。

  • サニーサイドにのみドッキングします。
  • パイロットが軌道に慣れる時間があるように、打ち上げで1日でドッキングする予定はありません。


映画「フォー・イン・スペース」のソユーズ4号をドッキングします。

USB Type-Aプラグを反対側にもう一度挿入するときは、この話を思い出してください。

悪いユーザーはいません-悪いインターフェースがあります。

ユーザーのせいにするのは簡単です。しかし、人々は常に間違っているでしょう。開発者も例外ではありません。オープンソースムーブメントの参加者として、私は開発者の間違いを非難しても、それでも将来のエラーの防止には役立たないことに気付きました。

オープンソースリポジトリの問題に関する問題を失礼なRTFMスタイルの回答（またはさらに悪いことに、まったく回答なし）でクローズすると、同じ問題の出現から身を守ることはできません。同じメッセージを何度も受信します。

PostCSSとAuto Prefixerの作成者として、多くの問題レポートを受け取ります。長期的には時間を無駄にしないために、私はルールに従います。すべての問題は、コードやドキュメントの変更につながるはずです。

ユーザーエクスペリエンスUX（オープンソースライブラリの観点からはAPI）を改善して、将来同じミスをすることが不可能になるようにすることをお勧めします。最悪の場合、警告を追加できます。

たとえば、多くのPostCSSユーザーは、パーサーオプションを忘れて、対応するパーサーなしで.lessおよび.scssファイルを処理しようとします。ユーザーを非難する代わりに、小さな警告を追加しました。

if (error.name === 'CssSyntaxError' && opts.from.endsWith(‘.scss’)) {
  error.message += '\nYou tried to parse SCSS with ' +
                   'the standard CSS parser; ' +
                   'try again with the postcss-scss parser'
}

2番目のレッスン：必ず、問題について開発者に通知してください


ソユーズ-2とソユーズ-3の飛行から3か月後、ソ連は手動によるドッキングの新たな試みの準備が整い、さらに野心的でした。

宇宙飛行士が1名いるソユーズ4と3名がいるソユーズ5の2つの有人宇宙船を打ち上げる予定でした。ドッキング後、ソユーズ5号のパイロットエンジニアと研究エンジニアは、オープンスペースを通過してソユーズ4に向かい、地球に戻ることになっています。つまり、彼らはある船で離陸し、宇宙を散歩し、別の船に着陸しました。


ソユーズ-5の宇宙空間からソユーズ-4への移行。ソユーズ5号パイロットの

ボリス・ボリノフは独りのままで、1日後に地球に戻ることになっていた。

今回はドッキングが成功し、ソユーズ-4は問題なく任務を完了しました。そして1969年1月18日、家に帰ってヴォリノフに行く時がきた。ユニオンは3つの取り外し可能なモジュールで構成されており、中央の1つだけが大気圏に入るように設計されています。


ユニオンスキーム。出典：NASA

Volynovの帰還中、計測モジュールは正常に分離しませんでした。着陸を中断するには遅すぎた。宇宙船は、誤った方向を向いた状態で大気に入りました。前方の機首です。入ってくる流れは薄い入口のハッチによってのみ反対され、熱シールドは分割されていないモジュールの間の後ろにありました。

ハッチの周りのシールが燃え始め、着陸カプセルは有毒な煙で満たされ始めました。重力と大気が彼らの仕事をしました;無防備なVolynovは椅子に固定されました。


ユニオン4の雰囲気への入り口の芸術的なイメージ。

しかし、Volynovを殺す可能性のある熱が彼を予期せずに救いました。計器モジュールの燃料タンクが爆発し、その結果、分離しました。着陸カプセルが正しい位置になり、遮熱板が前方に。

この間ずっと、ボリーノフは最後の数分間生きていたと確信し、彼が見聞きするすべてのものを声に出して録音しました。

問題はそこで終わりませんでした。パラシュートのスリングが絡まりましたが、ソフトランディングエンジンは機能しませんでした。カプセルは計画されたエリアから遠く離れて高速で地面に衝突しました。負傷したVolynovは、救急隊員が到着するまで、-38°Cの寒さの中で生き残る必要がありました。

宇宙飛行士は、すべての音と振動について大声でコメントし、フライトレコーダーがクラッシュに耐え、エンジニアが録音を聞いて新しい災害を防ぐことを望んでいました。

ハリウッド映画の適応に値するこの物語は私に何を教えましたか？

ライブラリや開発ツールに関連する問題に遭遇するたびに、ボリスヴォリノフを思い出します。彼が死の危機に瀕していても問題を報告できたなら、私は間違いなく開発者に報告を送るのに数分かかるでしょう。

単純なレポートでも、プロジェクトに大きく貢献できます。

私たちの業界に蔓延している詐欺師症候群のために、私たちはあまりにも頻繁に自分たちが発生する問題の罪を犯していることに気づきます。ドキュメントで何かが見落とされたか、単に十分に賢くなかった。しかし、最初のストーリーからの教訓を忘れないでください。悪いユーザーはいない、悪いインターフェースしかない。そして悪い開発者はいません。悪い開発ツールしかありません。

フレームワークを使用するときに間違いを犯した場合、あなたはおそらく最初と最後ではありません。さらに、毎日コードを見る開発者には見逃されていた視点から製品を見ました。見た目が一新されているため、ドキュメントや一般的な開発エクスペリエンスの問題に気付きやすくなっています。

あなたはタイプミスをし、理解できないエラーメッセージを受け取り、その結果、1時間のデバッグに費やしましたか？これは、新しい問題またはPRでエラーメッセージを改善する絶好の機会です。長い間、図書館の周りの道を探していますか？ドキュメントのどの情報が役立つかを考えてください。

3番目のレッスン：信頼の自動化


この話は1997年に、現在は消滅したミール宇宙ステーションで起こりました。 

ソビエトとアメリカの宇宙計画の間には1つの重要な違いがありました。ソビエト連邦では、船はロケットを作ったのと同じ人たちによって作られました。彼らは自動化を広範囲に利用し、パイロットを実質的に負担と見なしていました。そしてアメリカでは、船（特にスペースシャトル）は飛行機を設計した人々によって作成され、彼らはコンピューターをパイロットの補助的なツールとしてのみ考えていました。

1967年以来、ソ連は全自動ドッキングシステムを使用していました。最初はイグルー、次にコースです。これらの開発により、独自のコンピュータ、パワーシステム、およびエンジンを備えた独立した宇宙船として機能する自動モジュールからミールステーションを構築することが可能になりました。


ミール駅のモジュール。

しかし、ソビエトのドッキングシステムはウクライナで製造され、ソ連崩壊後のモスクワとキエフは価格に同意しませんでした。 Roscosmosは、外部コンポーネントへの依存度を減らすために、代替手段であるTORUを開発しました。これにより、宇宙ステーションのオペレーターは2つのジョイスティックを使用して船を遠隔操作できます。

TORUは宇宙でのテストに成功しました。 1997年、プログレスM-34は、ステーション用の貨物を積んで、世界へのドッキングに成功しました。その後、予期せず、ドッキングを解除する指示があり、地球に戻る代わりに手動でステーションにドッキングして、ドッキングシステムを再度確認しました。


ヴァシリー・ツィブリエフはミール駅から船を操縦しています。

乗組員は過負荷状態のプログレス（ステーションから廃棄物を運び、地球に戻さなければならなかった）の制御を維持しようとしましたが、船から送信されたビデオで世界を検出することは困難でした。宇宙飛行士が窓から接近する船をようやく見たとき、遅すぎるには遅すぎました。船はスペクトラムモジュールのソーラーパネルを損傷し、ステーションの電力の40％を供給し、船体に穴を開けました。

宇宙飛行士は笛を聞き、耳がふさがれました。駅は空気を失っていた。

乗組員は、Spectrumからのケーブルを外し、モジュールを閉じて、減圧したままにしなければなりませんでした。そして3年後、駅は海に浸水した。

その結果、コースはドッキングシステムから放棄されたわけではなく、現在はロシアで製造されています。コースとTORUはロシアの船で使用され、TORUはバックアップシステムです。

今日に至るまで、最初の、そしてこれまでのところ唯一の宇宙での衝突の理由は完全には明らかではありません。過負荷のために船の重心がずれ、宇宙飛行士は疲れていてTOPUの所有が不十分でした。テスト自体は急いで組織されました。その時、アメリカの宇宙飛行士は世界に乗っていました、そして、ステーションが減圧されるまで、彼もNASAもテストについて知りませんでした。


1997年のプログレスM-34との衝突後の世界へのダメージ。

この話は、Linuxシステムを破壊するビデオカード用のドライバーをインストールするための有名なスクリプトを思い出させました。夜働いた疲れたライブラリ開発者は、たった1つのギャップを逃しました。

 # GIANT BUG... causing /usr to be deleted... so sorry....
- rm -rf /usr /lib/nvidia-current/xorg/xorg
+ rm -rf /usr/lib/nvidia-current/xorg/xorg

人々は間違っています。自動化を優先します。車は苦しむ必要があります！

私はルールに従います。ソースコードに同じエラーが2回表示される場合は、今後このエラーを防ぐことができる自動ツールを作成することをお勧めします。

（のようなソースコードリンターESLint JavaScriptとのためStylelint CSSのためには）非常によくあなたの時間とたくさんのお金を要することができるバグを見つけます。あなた自身のリンタープラグインを書くのに数時間を費やしますが、長期的にはそれは報われます。

ドキュメントのタイプミスを避けたいですか？yaspellerを試してください。 CSSのプロパティを特定の順序で整理したいですか？開発ツールにstylelint-orderを追加します。
