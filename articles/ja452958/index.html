IMAPを満足させなかったもの


プロトコルIMAPは1986年に導入されました。標準で説明されている多くのことは、今日ではもはや関係ありません。たとえば、プロトコルはメッセージ行とMD5チェックサムの数を返すことができます -この機能は、最近のメールクライアントでは実際には使用されていません。

もう1つの問題は、トラフィックの消費です。 IMAPの場合、メールはサーバーに保存され、定期的にローカルクライアントと同期されます。何らかの理由でユーザーのデバイス上のコピーが「破損」している場合は、すべてのメールを再同期する必要があります。現代の世界では、何千ものモバイルデバイスをサーバーに接続できる場合、このアプローチにより、トラフィックとコンピューティングリソースの消費が増加します。

プロトコル自体だけでなく、プロトコルを使用する電子メールクライアントでも問題が発生します。IMAPはその誕生以来、何度もさまざまな改訂が行われてきました。現在のバージョンはIMAP4です。同時に、彼には多くのオプションの拡張機能があり、追加された90のRFC がネットワーク上で公開されています。最新の1つは2019年に導入されたRFC8514です。

同時に、多くの企業がそれをIMAPでの作業を簡素化したり、交換する必要があり、自分の独自のソリューションを提供します：Gmailの、見通し、Nylas。その結果、既存の電子メールクライアントは、使用可能なすべての機能の一部しかサポートしていません。このような多様性は、市場の細分化につながります。

指定された問題を解決するためにJMAPが呼び出されます。Internet Engineering Council（IETF）の指導の下、FastMailの専門家によって開発されました。プロトコルはHTTPSの上で実行され、JSONを使用します（このため、電子メッセージを交換するだけでなく、クラウド内の多数のタスクを解決するのにも適しています）。モバイルシステムでメールを使用する作業の編成を簡素化します。レターの処理に加えて、JMAPは連絡先やカレンダープランナーを操作するための拡張機能を接続する機能も提供します。

新しいプロトコルの特徴


JMAPはステートレスプロトコルであり、メールサーバーへの永続的な接続を必要としません。この機能により、不安定なモバイルネットワークでの作業が簡素化され、バッテリー電源デバイスを節約できます。

JMAPの電子メールは、JSON構造形式で表示されます。これには、RFC5322メッセージ（インターネットメッセージ形式）からのすべての情報が含まれています。これは、メールアプリケーションで必要になる場合があります。開発者によると、サーバーは潜在的な問題（MIME、ヘッダーの読み取り、エンコードに関連）を解決する責任があるため、このアプローチはクライアントの作成を簡素化するはずです。

クライアントはAPIを使用してサーバーにアクセスします。これを行うには、認証されたPOSTリクエストを生成します。そのプロパティはJMAPセッションオブジェクトに記述されています。リクエストはapplication / json形式であり、単一のJSONリクエストオブジェクトで構成されます。サーバーは1つの応答オブジェクトも生成します。仕様（第3項）、著者らは、要求に以下の例を与えます。



{
  "using": [ "urn:ietf:params:jmap:core", "urn:ietf:params:jmap:mail" ],
  "methodCalls": [
    [ "method1", {
      "arg1": "arg1data",
      "arg2": "arg2data"
    }, "c1" ],
    [ "method2", {
      "arg1": "arg1data"
    }, "c2" ],
    [ "method3", {}, "c3" ]
  ]
}

以下は、サーバーが形成する応答の例です。

{
  "methodResponses": [
    [ "method1", {
      "arg1": 3,
      "arg2": "foo"
    }, "c1" ],
    [ "method2", {
      "isBlah": true
    }, "c2" ],
    [ "anotherResponseFromMethod2", {
      "data": 10,
      "yetmoredata": "Hello"
    }, "c2"],
    [ "error", {
      "type":"unknownMethod"
    }, "c3" ]
  ],
  "sessionState": "75128aab4b1b"
}

実装例を含む完全なJMAP仕様は、プロジェクトの公式Webサイトにあります。同じ場所に、著者はJMAP連絡先とJMAPカレンダーの仕様の説明を投稿しました-それらはカレンダーと連絡先リストでの作業を目的としています。著者によると、連絡先とカレンダーは、「コア」に関係なくさらに開発および標準化できるように、別々のドキュメントに分割されていました。JMAPのソースコードはGitHubのリポジトリにあります。

展望


規格に関する作業は公式にはまだ完了していないという事実にもかかわらず、すでに本番環境で実装されています。たとえば、Cyrus IMAPオープンメールサーバーの作成者は、JMAPバージョンを実装しました。 FastMail開発者は新しいPerlプロトコル用のサーバーフレームワークをリリースし、 JMAP作成者はプロキシサーバーを導入しました。

今後、JMAPベースのプロジェクトが増えることが予想されます。たとえば、Open-Xchangeの開発者が新しいプロトコルを作成し、Linuxシステム用のIMAPサーバーを作成する可能性があります。 IMAPを拒否することは、会社のツールを中心に結成されたコミュニティのメンバーから強く求められます。

IETFおよびFastMailの開発者は、メッセージングのための新しいオープンスタンダードの必要性を認識するユーザーがますます増えていると述べています。JMAPの作成者は、将来、より多くの企業がこのプロトコルを実装し始めることを期待しています。
