JavaScriptが厳密モードを必要とするのはなぜですか？

Strictモードは、最新のJavaScriptの重要な部分です。開発者が標準よりも制限された構文を使用できるのはこのモードです。

ストリクトモードのセマンティクスは、「スロッピーモード」と呼ばれることもある従来の非ストリクトモードとは異なります。このモードでは、言語の構文規則はそれほど厳密ではなく、いくつかのエラーが発生した場合、システムはそれらについてユーザーに通知しません。つまり、エラーを無視して、エラーが発生したコードをさらに実行することができます。これにより、予期しないコード実行結果が生じる可能性があります。 厳密モードでは、JavaScriptセマンティクスにいくつかの変更が導入されています。これは、適切な例外をスローすることにより、システムがエラーに目をつぶるのを防ぎます。これにより、プログラムの実行が停止します。





さらに、厳格モードは、JSエンジンがコードを最適化するのを妨げる欠点がないプログラムの作成に役立ちます。さらに、このモードでは、言語の将来のバージョンで特別な意味を持つ構文要素を使用することは禁止されています。

ストリクトモードを使用する機能


厳格モードは、個々の関数またはスクリプト全体に適用できます。個々の命令または中括弧で囲まれたコードのブロックにのみ適用することはできません。スクリプト全体のレベルでstrictモードを使用するには、ファイルの最初の部分で、他のコマンドの前に、"use strict"または構文を配置する必要があります'use strict'。

プロジェクトに、strictモードを使用しないスクリプトと、このモードを使用するスクリプトがある場合、これらのスクリプトがマージされる可能性があります。

これにより、厳密モードでの実行を意図していないコードが、システムが厳密モードで実行しようとすると、そのような状態になるという事実につながります。逆も可能です-厳格モード用に書かれたコードは非厳格モードに分類されます。したがって、「厳格な」スクリプトと「厳格でない」スクリプトを混在させないことをお勧めします。

すでに述べたように、strictモードは個々の関数に適用できます。これを行うには、他のコマンドの前に、構造体"use strict"または'use strict'関数の本体の上部に配置する必要があります。このアプローチのストリクトモードは、ネストされた関数を含む、関数の本体に配置されるすべてに適用されます。

例えば：

const strictFunction = ()=>{
  'use strict';
  const nestedFunction = ()=>{
    //      
  }
}

ES2015標準に登場したJavaScriptモジュールでは、strictモードがデフォルトで有効になっています。したがって、それらを使用する場合、明示的に含める必要はありません。

ストリクトモードによってJSコードに加えられた変更


ストリクトモードは、コードの構文とプログラム実行中のコードの動作の両方に影響します。コード内のエラーは例外に変換されます。クワイエットモードでは、ストリクトモードで静かにクラッシュするため、エラーメッセージが表示されます。これは、システムが緩いモードで構文エラーに応答する方法に似ています。厳密モードでは、変数の操作が簡素化され、関数evalとオブジェクトの使用が厳しく規制argumentsされ、言語の将来のバージョンで実装できる構造の操作が合理化されます。

silentサイレントエラーを例外に変換する


サイレントエラーは、厳密モードで例外に変換されます。緩やかなモードでは、システムはそのようなエラーに明示的に応答しません。厳密モードでは、そのようなエラーが存在すると、コードが操作不能になります。

したがって、これにより、strictモードの変数と定数はvar、ディレクティブ、letまたはを使用しないと宣言できないため、誤ってグローバル変数を宣言するというミスを犯すことは困難constです。その結果、これらのディレクティブなしで変数を作成すると、プログラムが動作しなくなります。たとえば、次のコードを実行しようとすると、例外がスローされますReferenceError。

'use strict';
badVariable = 1;

このようなコードは厳密モードでは実行できません。厳密モードがオフになっていると、グローバル変数が作成されるためですbadVariable。厳格モードは、プログラマーが誤ってグローバル変数を作成するのを防ぎます。

通常モードでは機能しないコードを実行しようとすると、例外がスローされるようになりました。エラーは、laxモードでは単に無視された不正な構文構成であると見なされます。

したがって、たとえば、厳密モードではarguments、NaNやなどの読み取り専用エンティティに対して値の割り当て操作を実行することはできませんeval。

ストリクトモードでは、たとえば、次の場合に例外がスローされます。 

  • ある種の書き換え可能なグローバルプロパティなど、読み取り専用プロパティに値を割り当てようとする試み。
  • ゲッターのみを持つプロパティに値を書き込もうとした。
  • 拡張不可能なオブジェクトのプロパティに何かを書き込もうとする試み。

厳格モードの例外につながる構文構成の例を次に示します。

'use strict';

let undefined = 5;

let obj = {};
Object.defineProperty(obj, 'foo', { value: 1, writable: false });

let obj2 = { get foo() { return 17; } };
obj2.foo = 2

let fixedObj = {};
Object.preventExtensions(fixedObj);
fixed.bar= 1;

このようなコードフラグメントをストリクトモードで実行しようとすると、例外がスローされますTypeError。だから、例えば、undefinedそしてInfinityグローバルなその値が上書きできないエンティティとの財産ですfooオブジェクトがobj書き換えをサポートしていません。fooオブジェクトのプロパティにはobj2ゲッターのみがあります。fixedObjメソッドはメソッドを使用して、オブジェクトを拡張不可能にしますObject.preventExtensions。削除できないプロパティを削除しようとすると、次のような

厳格モードでは、同じ名前のプロパティをオブジェクトに割り当てることが禁止されています。その結果、次のコードを実行しようとすると、構文エラーが発生します。

'use strict';
let o = { a: 1, a: 2 };

厳密モードでは、関数パラメーター名が一意である必要があります。非厳密モードでは、たとえば、2つの関数パラメーターが同じ名前oneである場合、引数関数を渡すと、パラメーターの値は最後に宣言された引数に該当するものになります。

厳密モードでは、同じ名前の関数のパラメーターは禁止されています。その結果、次のコードを実行しようとすると、構文エラーが発生します。

'use strict';
const multiply = (x, x, y) => x*x*y;

厳密モードでは、数値の前にゼロを付けた8進数表記は使用できません。これは仕様にはありませんが、この機能はブラウザでサポートされています。

このような状況は開発者を混乱させ、数字の前の0は単に無視され、あまり意味がないと信じ込ませます。厳密モードでは、最初に0である数値を使用しようとすると、構文エラーが発生します。

厳格モードでは、最適化を妨げる構造の使用も禁止されています。コードの最適化を実行する前に、インタープリターは変数が正確にどこに格納されているかを知る必要があります。厳密モードでは、最適化を妨害するものは禁止されています。

そのような禁止の一例は指示に関するwith。この命令を使用すると、同じ名前のエンティティが命令ブロックの外側と内側の両方に存在する可能性があるため、JSインタープリターが参照している変数またはプロパティを見つけられなくなりますwith。

次のようなコードがあるとします。

let x = 1;
with (obj) {
  x;
}

インタープリターはx、ブロック内の変数がwith外部変数を参照しているxか、obj.xオブジェクトのプロパティを参照しているかを確認できませんobj。

その結果、値がメモリ内のどこに配置されるかは正確にはわかりませんx。このようなあいまいさを取り除くために、厳密モードでは、命令の使用はwith禁止されています。ストリクトモードで次のコードを実行しようとするとどうなるか見てみましょう。

'use strict';
let x = 1;
with (obj) {
  x;
}

この試行の結果は構文エラーになります。

strictモードでも、methodに渡されるコードで変数を宣言することは禁止されていますeval。

たとえば、通常モードでは、ビューコマンドeval('let x')は変数宣言になりxます。これにより、プログラマは文字列内の変数宣言を非表示にでき、外部にある同じ変数の定義を上書きできますeval。

これを防ぐために、厳密モードでは、文字列としてメソッドに渡されるコードで変数を宣言することは禁止されていますeval。

厳格モードでは、通常の変数の削除も禁止されています。その結果、次のコードを実行しようとすると、構文エラーが発生します。

'use strict';
let x;
delete x;

厳密モードでは、このプロパティはサポートされていませんarguments.callee。通常モードでは、calleeオブジェクトプロパティargumentsを調査している関数の親関数の名前を返します。

このプロパティarguments.calleeにアクセスすると、埋め込み関数以外の関数へのリンクが必要になるため、このプロパティのサポートは、インライン関数などの最適化を妨げます。 strictモードでは、使用arguments.calleeすると例外が発生しますTypeError。

厳密モードでは、キーワードがthis常にオブジェクトである必要はありません。通常の状況下では、場合this関数は、次のようにバインドされているcall、applyまたはbindプリミティブ型の値などの対象ではない何かにundefined、null、numberまたはboolean、同様の値はオブジェクトでなければなりません。

コンテキストthisがオブジェクトではないものに変わると、グローバルオブジェクトが代わりに使用されます。たとえば- window。これは、関数をthisオブジェクトではない特定の値に設定して呼び出すthisと、グローバルオブジェクトへのリンクが代わりにこの値に含まれることを意味します。

  return this;

▍セキュリティ関連の変更


ストリクトモードではcaller、およびのプロパティをpublicにすることはできませんarguments。実際にはcaller、たとえば、プロパティcallerにアクセスしている関数を呼び出した関数へのアクセスを与えることができます。

オブジェクトargumentsは、呼び出されたときに関数に渡された引数を格納します。たとえば、関数がある場合fn、これは、fn.callerその関数を呼び出した関数にアクセスでき、ヘルプfn.argumentsを使用fnして、呼び出し中に渡された引数を確認できることを意味します。

これらの機能は、潜在的なセキュリティリスクをもたらします。その結果、これらのプロパティへのアクセスは、strictモードでは禁止されています。

function secretFunction() {
  'use strict';
  secretFunction.caller;
  secretFunction.arguments;
}
function restrictedRunner() {
  return secretFunction();
}
restrictedRunner();

前の例では、我々は、strictモードで、に向けることができないsecretFunction.callerとsecretFunction.arguments。実際、これらのプロパティを使用して、関数呼び出しのスタックを取得できます。このコードを実行しようとすると、例外がスローされTypeErrorます。

厳密モードでは、JavaScriptの将来のバージョンで使用される可能性のある識別子を使用して、変数やオブジェクトのプロパティに名前を付けることはできません。：それは、例えば、以下の識別子であるimplements、interface、let、package、private、protected、public、staticとyield。

ES2015以降のバージョンの標準では、これらの識別子は予約語になりました。また、厳密モードでは、変数またはプロパティの名前付けに使用できません。

概要


Strictモードは、長年存在している標準です。それは非常に幅広いブラウザのサポートを楽しんでいます。ストリクトモードコードの問題は、Internet Explorerなどの古いブラウザでのみ発生します。

最新のブラウザーは、厳密なJavaScriptモードで問題が発生しないはずです。その結果、このモードは「サイレント」エラーを防止し、アプリケーションのセキュリティを向上させるために使用する必要があると言えます。サイレントエラーは、プログラムの実行を妨げる例外に変換されます。セキュリティを向上させるという観点から、たとえば、eval関数呼び出しスタックへのアクセスを防止します。さらに、strictモードを使用すると、JSエンジンによるコードの最適化が容易になり、プログラマーはJavaScriptの将来のバージョンでアプリケーションを見つける可能性がある予約語を慎重に処理する必要があります。
