Go：誤った入力によるJSON逆シリアル化、またはAPI開発者のエラーを回避する方法



最近、たまたまエンコード形式としてjsonを使用したREST APIを提供するサービスのために、Goでhttpクライアントを開発しました。標準的な作業ですが、作業中に私は非標準的な問題に直面する必要がありました。ポイントは何ですか。

ご存知のように、json形式にはデータ型があります。4つのプリミティブ：文字列、数値、ブール値、null。2つの構造タイプ：オブジェクトと配列。この場合、プリミティブ型に関心があります。以下は、異なるタイプの4つのフィールドを持つjsonコードの例です。

{
	"name":"qwerty",
	"price":258.25,
	"active":true,
	"description":null,
}

例に示すように、文字列値は引用符で囲まれています。数値-引用符はありません。ブール型は、trueまたはfalse（引用符なし）の2つの値のいずれか1つのみを持つことができます。そして、null型はそれに応じてnullになります（引用符なし）。

そして今、問題自体。ある時点で、サードパーティのサービスから受け取ったjsonコードを詳細に調べたところ、フィールドの1つ（価格と呼びましょう）には、数値に加えて定期的に文字列値（引用符で囲まれた数）があることがわかりました。つまり、パラメータが異なる同じクエリは、数値を数値として返すことも、同じ数値を文字列として返すこともできます。このような結果を返すコードがどのように編成されているかは想像できませんが、これは明らかに、サービス自体がアグリゲーターであり、さまざまなソースからデータをプルしているためであり、開発者はサーバー応答jsonを単一の形式に変換していません。それにもかかわらず、何であるかで作業する必要があります。

しかし、それから私はさらに驚きました。論理フィールド（アクティブとしましょう）は、trueとfalseに加えて、文字列値「true」、「false」、さらに数値1と0（それぞれtrueとfalse）を返しました。

弱い型付けのPHPでjsonの発言を処理する場合、データ型に関するこのすべての混乱は重要ではありませんが、Goには強い型付けがあり、逆シリアル化フィールドの型を明確に示す必要があります。その結果、逆シリアル化プロセス中にアクティブフィールドのすべての値を論理型に変換し、priceフィールドの任意の値を数値に変換できるメカニズムを実装する必要がありました。

価格フィールドから始めましょう。

次のようなjsonコードがあるとします。

[
	{"id":1,"price":2.58},
	{"id":2,"price":7.15}
]

つまり、jsonには、数値型の2つのフィールドを持つオブジェクトの配列が含まれています。Goのこのjsonの標準の逆シリアル化コードは次のようになります。

type Target struct {
	Id    int     `json:"id"`
	Price float64 `json:"price"`
}

func main() {
	jsonString := `[{"id":1,"price":2.58},
					{"id":4,"price":7.15}]`

	targets := []Target{}

	err := json.Unmarshal([]byte(jsonString), &targets)
	if err != nil {
		fmt.Println(err)
		return
	}

	for _, t := range targets {
		fmt.Println(t.Id, "-", t.Price)
	}
}

このコードでは、idフィールドをintに、priceフィールドをfloat64に逆シリアル化します。ここで、jsonコードが次のようになっているとします。

[
	{"id":1,"price":2.58},
	{"id":2,"price":"2.58"},
	{"id":3,"price":7.15},
	{"id":4,"price":"7.15"}
]

つまり、価格フィールドには、数値タイプと文字列の両方の値が含まれています。この場合、価格フィールドの数値のみがfloat64型にデコードできますが、文字列値は型の非互換性に関するエラーを引き起こします。つまり、float64も他のプリミティブ型もこのフィールドの逆シリアル化には適しておらず、独自の逆シリアル化ロジックを持つ独自のカスタム型が必要です。

そのような型として、float64型の単一のFloat64フィールドを持つCustomFloat64構造体を宣言します。

type CustomFloat64 struct{
	Float64 float64
}

ターゲット構造の価格フィールドにこのタイプをすぐに示します。

type Target struct {
	Id    int           `json:"id"`
	Price CustomFloat64 `json:"price"`
}

次に、タイプCustomFloat64のフィールドをデコードするための独自のロジックを記述する必要があります。

encoding / jsonパッケージは、MarshalJSONおよびUnmarshalJSONという 2つの特別なメソッドを提供します。これらは、特定のユーザーデータタイプのエンコードおよびデコードロジックをカスタマイズするように設計されています。これらのメソッドをオーバーライドして、独自の実装を記述するだけで十分です。

任意のタイプCustomFloat64のUnmarshalJSONメソッドをオーバーライドします。この場合、メソッドのシグネチャに厳密に従う必要があります。そうしないと、メソッドが機能しなくなり、最も重要なのはエラーが発生しなくなります。

func (cf *CustomFloat64) UnmarshalJSON(data []byte) error {

入力では、このメソッドはバイト（データ）のスライスを取得します。これには、デコードされたjsonの特定のフィールドの値が含まれます。このバイトシーケンスを文字列に変換すると、フィールドの値がjsonで記述された形式で正確に表示されます。つまり、文字列型の場合、二重引用符（ "258"）を含む文字列が正確に表示され、数値型の場合、引用符（258）を含まない文字列が表示されます。

数値と文字列値を区別するには、最初の文字が引用符であるかどうかを確認する必要があります。 UNICODEテーブルの二重引用符文字は1バイトを占めるため、データスライスの最初のバイトをUNICODEの文字番号と比較して確認するだけです。これは34番です。一般に、1バイトは2バイトを超える可能性があるため、1文字は1バイトに相当しないことに注意してください。 Goのシンボルは、rune（ルーン）に相当します。私たちの場合、この条件で十分です：

if data[0] == 34 {

条件が満たされている場合、値は文字列型であり、引用符の間の文字列、つまり最初のバイトと最後のバイトの間のスライスバイトを取得する必要があります。このスライスには、プリミティブ型float64にデコードできる数値が含まれています。これは、結果をCustomFloat64構造体のFloat64フィールドに保存しながら、json.Unmarshalメソッドを適用できることを意味します。

err := json.Unmarshal(data[1:len(data)-1], &cf.Float64)

データスライスが引用符で始まっていない場合は、すでに数値データ型が含まれているため、json.Unmarshalメソッドをデータスライス全体に直接適用できます。

err := json.Unmarshal(data, &cf.Float64)

UnmarshalJSONメソッドの完全なコードは次のとおりです。

func (cf *CustomFloat64) UnmarshalJSON(data []byte) error {
	if data[0] == 34 {
		err := json.Unmarshal(data[1:len(data)-1], &cf.Float64)
		if err != nil {
			return errors.New("CustomFloat64: UnmarshalJSON: " + err.Error())
		}
	} else {
		err := json.Unmarshal(data, &cf.Float64)
		if err != nil {
			return errors.New("CustomFloat64: UnmarshalJSON: " + err.Error())
		}
	}
	return nil
}

その結果、jsonコードにjson.Unmarshalメソッドを使用すると、priceフィールドのすべての値が透過的にプリミティブ型float64に変換され、結果がCustomFloat64構造体のFloat64フィールドに書き込まれます。

ここで、Target構造をjsonに戻す必要があるかもしれません。ただし、json.MarshalメソッドをCustomFloat64型に直接適用する場合は、この構造をオブジェクトとしてシリアル化します。価格フィールドを数値にエンコードする必要があります。カスタムタイプCustomFloat64のコーディングロジックをカスタマイズするには、メソッドのシグネチャを厳密に監視しながら、MarshalJSONメソッドを実装します。

func (cf CustomFloat64) MarshalJSON() ([]byte, error) {
	json, err := json.Marshal(cf.Float64)
	return json, err
}

このメソッドで行う必要があるのは、json.Marshalメソッドを使用することだけですが、CustomFloat64構造体ではなく、Float64フィールドにすでに適用しています。メソッドから、受信したバイトスライスとエラーを返します。

シリアライゼーションとデシリアライゼーションの結果を表示する完全なコードは次のとおりです（簡潔にするためにエラーチェックは省略されています。二重引用符記号の付いたバイト数は一定です）。

package main

import (
	"encoding/json"
	"errors"
	"fmt"
)

type CustomFloat64 struct {
	Float64 float64
}

const QUOTES_BYTE = 34

func (cf *CustomFloat64) UnmarshalJSON(data []byte) error {
	if data[0] == QUOTES_BYTE {
		err := json.Unmarshal(data[1:len(data)-1], &cf.Float64)
		if err != nil {
			return errors.New("CustomFloat64: UnmarshalJSON: " + err.Error())
		}
	} else {
		err := json.Unmarshal(data, &cf.Float64)
		if err != nil {
			return errors.New("CustomFloat64: UnmarshalJSON: " + err.Error())
		}
	}
	return nil
}

func (cf CustomFloat64) MarshalJSON() ([]byte, error) {
	json, err := json.Marshal(cf.Float64)
	return json, err
}

type Target struct {
	Id    int           `json:"id"`
	Price CustomFloat64 `json:"price"`
}

func main() {
	jsonString := `[{"id":1,"price":2.58},
					{"id":2,"price":"2.58"},
					{"id":3,"price":7.15},
					{"id":4,"price":"7.15"}]`

	targets := []Target{}

	_ := json.Unmarshal([]byte(jsonString), &targets)

	for _, t := range targets {
		fmt.Println(t.Id, "-", t.Price.Float64)
	}

	jsonStringNew, _ := json.Marshal(targets)
	fmt.Println(string(jsonStringNew))
}

コード実行結果：

1 - 2.58
2 - 2.58
3 - 7.15
4 - 7.15
[{"id":1,"price":2.58},{"id":2,"price":2.58},{"id":3,"price":7.15},{"id":4,"price":7.15}]

2番目の部分に進み、論理フィールドの値に一貫性のないjson逆シリアル化の同じコードを実装してみましょう。

次のようなjsonコードがあるとします。

[
	{"id":1,"active":true},
	{"id":2,"active":"true"},
	{"id":3,"active":"1"},
	{"id":4,"active":1},
	{"id":5,"active":false},
	{"id":6,"active":"false"},
	{"id":7,"active":"0"},
	{"id":8,"active":0},
	{"id":9,"active":""}
]

この場合、アクティブフィールドは論理型を意味し、trueとfalseの2つの値のいずれか1つしか存在しません。ブール値以外の値は、逆シリアル化中にブール値に変換する必要があります。

現在の例では、次の一致を認めています。真の値は、真（論理）、真（文字列）、1（文字列）、1（数値）に対応します。false値は、false（論理）、false（文字列）、0（文字列）、0（数値）、 ""（空の文字列）に対応します。

最初に、逆シリアル化のターゲット構造を宣言します。Activeフィールドのタイプとして、カスタムタイプCustomBoolをすぐに指定します。

type Target struct {
	Id     int        `json:"id"`
	Active CustomBool `json:"active"`
}

CustomBoolは、bool型の単一のboolフィールドを持つ構造体です。

type CustomBool struct {
	Bool bool
}

この構造にUnmarshalJSONメソッドを実装します。コードをすぐにお渡しします。

func (cb *CustomBool) UnmarshalJSON(data []byte) error {
	switch string(data) {
	case `"true"`, `true`, `"1"`, `1`:
		cb.Bool = true
		return nil
	case `"false"`, `false`, `"0"`, `0`, `""`:
		cb.Bool = false
		return nil
	default:
		return errors.New("CustomBool: parsing \"" + string(data) + "\": unknown value")
	}
}

今回のケースのアクティブフィールドの値の数は限られているため、スイッチケースの構成を使用して、CustomBool構造のBoolフィールドの値をどのようにするかを決定できます。チェックするには、2つのケースブロックのみが必要です。最初のブロックでは、trueの値をチェックし、2番目のブロックではfalseをチェックします。

可能な値を書き込むときは、砂利の役割に注意を払う必要があります（これは、英語のレイアウトで文字Eが付いたキーの引用符です）。この文字を使用すると、文字列内の二重引用符をエスケープできます。わかりやすくするために、引用符付きと引用符なしの値をこの記号で囲みました。したがって、 `false`は文字列false（引用符なし、jsonでboolと入力）に対応し、` false 'は文字列“ false”（引用符付き、jsonでstringと入力）に対応します。値が「1」と「1」の場合も同じです。1つ目は数値1（jsonで引用符なしで記述）、2つ目は文字列「1」（jsonで引用符付き）です。このエントリ `` "`は空の文字列です。つまり、json形式では次のようになります： ""。

対応する値（trueまたはfalse）は、CustomBool構造のBoolフィールドに直接書き込まれます。

cb.Bool = true

デフォルトブロックでは、フィールドに不明な値があることを示すエラーを返します。

return errors.New("CustomBool: parsing \"" + string(data) + "\": unknown value")

これで、json.Unmarshalメソッドをjsonコードに適用でき、アクティブフィールドの値がプリミティブ型boolに変換されます。

MarshalJSONメソッドをCustomBool構造に実装します。

func (cb CustomBool) MarshalJSON() ([]byte, error) {
	json, err := json.Marshal(cb.Bool)
	return json, err
}

ここには何も新しいものはありません。このメソッドは、CustomBool構造のBoolフィールドをシリアル化します。

シリアライズとデシリアライズの結果を表示する完全なコードは次のとおりです（簡潔にするためにエラーチェックは省略しています）。

package main

import (
	"encoding/json"
	"errors"
	"fmt"
)

type CustomBool struct {
	Bool bool
}

func (cb *CustomBool) UnmarshalJSON(data []byte) error {
	switch string(data) {
	case `"true"`, `true`, `"1"`, `1`:
		cb.Bool = true
		return nil
	case `"false"`, `false`, `"0"`, `0`, `""`:
		cb.Bool = false
		return nil
	default:
		return errors.New("CustomBool: parsing \"" + string(data) + "\": unknown value")
	}
}

func (cb CustomBool) MarshalJSON() ([]byte, error) {
	json, err := json.Marshal(cb.Bool)
	return json, err
}

type Target struct {
	Id     int        `json:"id"`
	Active CustomBool `json:"active"`
}

func main() {
	jsonString := `[{"id":1,"active":true},
					{"id":2,"active":"true"},
					{"id":3,"active":"1"},
					{"id":4,"active":1},
					{"id":5,"active":false},
					{"id":6,"active":"false"},
					{"id":7,"active":"0"},
					{"id":8,"active":0},
					{"id":9,"active":""}]`

	targets := []Target{}

	_ = json.Unmarshal([]byte(jsonString), &targets)

	for _, t := range targets {
		fmt.Println(t.Id, "-", t.Active.Bool)
	}

	jsonStringNew, _ := json.Marshal(targets)
	fmt.Println(string(jsonStringNew))
}

調査結果


第一に。任意のデータ型のMarshalJSONおよびUnmarshalJSONメソッドをオーバーライドすると、特定のjsonコードフィールドのシリアル化およびデシリアライズをカスタマイズできます。示された使用例に加えて、これらの関数はnull許容フィールドを処理するために使用されます。

第二に。jsonテキストエンコード形式は、情報を交換するために広く使用されているツールであり、他の形式に対するその利点の1つは、データ型の可用性です。これらのタイプへの準拠は厳密に監視する必要があります。
