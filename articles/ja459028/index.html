マイクロコントローラーでの静的メモリ割り当て

ホームズ：親愛なる、私たちがどこにいるか教えてくれませんか？
羊飼い：あなたは気球の中にいます!!!
ホームズ：プログラマーである必要があります。
羊飼い：はい、しかし、あなたはどう思いましたか？
ホームズ：プログラマーだけがそのような正確であり
ながら役に立たない答えを出すことができました。

...有名なジョークからの抜粋


マイクロコントローラー用にプログラミングしたことがある場合は、Arduino IDEを使用したり、AVR、ARM、またはESPのコンパイラーを直接操作したりすることは関係ありません。おそらく、次のようなビルド完了レポートが表示されます。

Sketch uses 1,090 bytes (3%) of program storage space. Maximum is 30,720 bytes.
Global variables use 21 bytes (1%) of dynamic memory, leaving 2,027 bytes for local variables. Maximum is 2,048 bytes.

または

 text data   bss   dec  hex filename
52136 1148 12076 65360 ff50 MyProject

そのようなレポートは本当に完全に正確です...それは単に不完全であり、したがってそれほど有用ではありません。問題は、静的に配信されたデータのみがここで考慮されることです。ただし、newまたはmallocを介して割り当てられたものはすべて統計には含まれません。その結果、メモリが突然切れてファームウェアが正しく動作しなくなった瞬間を追跡するのがはるかに難しくなります。ただし、マイクロコントローラのメモリは通常それほど多くないため、このパラメータは注意深く監視する必要があります。

vkidkuでは、動的メモリ割り当ての使用が実際に行われるジュニアおよびミディアムマイクロコントローラーの1つの例を思い出しませんでした。正当化された。原則として、これはバッファの割り当て、またはファームウェアの最初の部分でのいくつかのオブジェクトの作成です。その後、これらのオブジェクトは、次のリセットまでメモリに残ります。そして、これはそのようなメモリを静的に割り当てる機会です-今日はそれを行います。

この記事は初心者を対象としています（ただし、基本的なことはお伝えしませんが、読者は少なくともC ++に関する本を読んだことがあるはずです）。行け。

静的変数分布


さまざまなデータセグメントの詳細と説明を詳しく説明しない場合、マイクロコントローラーのRAMは次の3つのタイプに分類できます。

  • 静的に分散されたメモリ-プロジェクトの構築段階でコンパイラーが知っているすべてのもの：グローバル変数とオブジェクト、関数とオブジェクトの静的変数
  • ヒープ（ヒープ）-システム（メモリアロケータ）が必要に応じて誰にでも断片を切り取る、メモリの広い領域。mallocやnew演算子などの関数は、ここからメモリを取得します。
  • スタックは、プロセッサがレジスタを格納し、関数からアドレスを返す場所ですが、関数のローカル変数もスタックに配置されます。

これを実際に見てみましょう。コンパイラーとビルドシステムで問題が発生しないように、ターゲットのArduino NanoプラットフォームにArduino IDEを使用します。

上から（ヘッダーから）レポートをもう一度見てみましょう。すでに述べたように、コンパイル段階で配布されたオブジェクトのみが統計に含まれます。

Global variables use 21 bytes (1%) of dynamic memory, leaving 2,027 bytes for local variables. Maximum is 2,048 bytes.

空きメモリのメッセージから判断すると、単純にヒープです。しかし、コードを見てみましょう

int * buf;
void setup()
{
  buf = new int[3000];
}
void loop() {}

ファームウェアは、マイクロコントローラー自体にあるよりも多くのメモリを割り当てようとしています。もちろん、そのようなコードは機能しません。しかし、私の意見では。ここでの最大の問題は、コードが正常にコンパイルされ、コンパイラが時間どおりに手を振らず、メモリが不足する可能性があることを警告しなかったことです。そして、明確なエラーハンドラがあるとよいでしょう。そしてそうでなければ？

もちろん、あなたがたは今、彼らが邪悪なピノキオを自分に言うと言うでしょう！レポートには2kbしかないことが示され、それぞれ2バイトの3000エレメントを割り当てようとしていることがわかります。しかし、実際のプロジェクトでどのように見えるかを考えてみましょう。ほとんどの場合、その部分はすでにいくつかの変数で占められており、スタックは何らかの方法で使用され、一部のメモリはすでに動的に割り当てられています。そして、突然、いくつかのまれなイベントセットが別のメモリを必要とするようになりました...おお、皆さん、私たちは到着しました。

静的割り当てとはどういう意味ですか？また、一般的に何を達成したいですか？動的メモリ割り当てに反対することはありません。しかし、あらかじめ決められたサイズのオブジェクト、バッファ、配列に動的割り当てを使用する理由がよくわかりません。なぜそれをグローバル変数として宣言しないのですか？

一時的に配列のサイズをより適切なサイズに縮小しました。

int buf[300];
 
void setup()
{
  buf[0] = 1; //Avoid optimizing this array
}
 
void loop()
{
}

これで、コンパイラーとリンカーは、300 * 2 = 600バイトのサイズの特定の配列を事前に知っており、RAMに配置する必要があります。さらに、リンカはこの配列に固定アドレスを割り当てることができます。これは、必要に応じて、objdumpユーティリティで表示できます（もちろん、Arduino IDEがバイナリを配置する場所を見つけない限り）。

cd C:\Users\GrafAlex\AppData\Local\Temp\arduino_build_55567
"C:\Program Files (x86)\Arduino\hardware\tools\avr\bin\avr-objdump.exe" -x -D -S -s StaticMem.ino.elf
…
00800100 l 	O .bss      	00000258 buf
…

ここで、0x00800100はリンカがバッファに割り当てたアドレス、0x258はそのサイズ（600バイト）です。

次に、3000要素の不適切なサイズを返して、何が起こるかを見てみましょう。アセンブリシステムから自然に「fe」を取得します

Sketch uses 456 bytes (1%) of program storage space. Maximum is 30,720 bytes.
Global variables use 6,009 bytes (293%) of dynamic memory, leaving -3,961 bytes for local variables. Maximum is 2,048 bytes.
...
Not enough memory; see http://www.arduino.cc/en/Guide/Troubleshooting#size for tips on reducing your footprint.

実際、これが演習の目標でした。コンパイラとリンカーは、アセンブリ段階で握手をし、そのような大きなバッファーを割り当てることは悪い考えであると述べました。

さて、優れたコーディング手法の専門家が来て、グローバル変数は悪だと言います。私はこの点について論争を画面外に残すことを提案します。グローバル変数が混乱する場合、いくつかの選択肢を提供できます。実際、以下に説明するすべてのアプローチは同じ目的を果たします-コンパイル/リンクの段階で特定のオブジェクトをメモリに割り当てることができることをコンパイラに通知し、あらかじめ決められたアドレスを与えてRAMに配置します。おまけとして、メモリの断片化の欠如もあり、動的割り当て（このようなオブジェクトの場合）のオーバーヘッドもありません。

静的グローバル変数


staticという単語を使用して、配列/オブジェクト/変数を宣言できます。これにより、変数のスコープが制限されます。このcppファイルからのみアクセスできます。

static int buf[300];
 
void setup()
{
  buf[0] = 1; //Avoid optimizing this array
}

ここで、別のcppファイルを作成して、たとえばexternを通じて配列にアクセスしようとしても、何も機能しません。

extern int buf[300];

void foo()
{
    buf[0]++;
}

リンカはbuf変数が見つからないことを誓います。

C:\Program Files (x86)\Arduino\hardware\arduino\avr\cores\arduino/main.cpp:47: undefined reference to `buf'

両方のファイルでbuf変数が静的に宣言されている場合、各cppファイルには独自の変数があります。

static int buf[300];

void foo()
{
    buf[0]++;
}

この場合のobjdumpの出力は次のようになります。

0080036a l     O .bss	00000258 _ZL3buf.lto_priv.12
00800112 l     O .bss	00000258 _ZL3buf.lto_priv.13

名前のない名前空間


名前のない名前空間にグローバル変数を配置して、前の段落と同じバンを取得できます。これは基本的にスコープが制限されたグローバル変数です。

namespace
{
int buf[300];
}

静的クラスのメンバー


静的クラスメンバーもコンパイル時に割り当てられます。

class A
{
    static int buf[300];
public:
    int * getBuf()
    {
        return buf;
    }
};

int A::buf[300];
 
void setup() {}

void loop()
{
    A a;
    a.getBuf()[0] += 1;
}

クラスAのインスタンスが多数存在する可能性がある（たとえば、loop（）関数に入るたびに1つのインスタンスを作成する）にもかかわらず、バッファーはクラスの静的メンバーとして宣言されます。これは、単一のコピーに存在し、リンク段階でメモリに分散されることを意味します。

00800100 l     O .bss	00000258 A::buf

関数の静的変数


特定のオブジェクトを静的に分散する必要があるが、1つの関数だけがそれを認識している必要がある場合は、ローカルの静的変数を使用すると非常に便利です。

int getCounter()
{
    static int counter = 0;
    return ++counter;
}

void setup() 
{
    Serial.begin(9600);
}

void loop()
{
    Serial.println(getCounter());
}

カウンター変数は静的に分散されます。getCounter（）関数が呼び出されるたびに作成されるわけではありませんが、メモリ内に固定アドレスがあります

00800116 l     O .bss	00000002 getCounter()::counter

この分布の副作用（同時に便利な機能）は、関数の呼び出し間で変数の値が「存続」することです。getCounter（）を呼び出すたびに、変数の値が1つ増加します。

それにもかかわらず、この方法にはいくつかのニュアンスがあります。たとえば、静的な配列の分布は正常に機能します

int * getBuf()
{
  static int buf[300];
  return buf;
}

C ++言語標準では、getBuf（）が最初に呼び出されたときにbuf変数が初期化されると記載されていますが、初期化のために何もする必要がないため、この配列は.bssセクション（ファームウェアの開始時にゼロで埋められるメモリ領域）のどこかに単に割り当てられます）

int * getBuf()
{
  static int buf[300] = {1, 2, 3};
  return buf;
}

このバージョンでは、コンパイラーもこの変数をRAMに静的に分散しますが、プログラムのメモリーに600バイトが追加され、この配列の初期値が格納されます。

class Buf
{
public:
  int buf[300];
 
  Buf(int v)
  {
	buf[1] = v;
  }
};
 
int * getBuf()
{
  static Buf buf(1234);
  return buf.buf;
}

この場合、最初にgetBuf（）関数が呼び出されたときに何かを行う重要なコンストラクターを取得します。AVRプラットフォーム（ATMega）の場合、コンパイラーは、コンストラクターを開始する必要があるかどうか、またはコンストラクターを以前に実行したことがあるかどうかを規制する特別なフラグを生成します。これは心に留めておく必要があります。このフラグはRAMを少し消費します。また、getBuf（）を呼び出すたびにフラグの暗黙的なチェックが行われ、パフォーマンスに影響する可能性があります。

しかし、ARMプラットフォーム（STM32など）では、非常に予期しない仕掛けが発生します。重要なデザイナーが現れるとすぐに、ファームウェアはすぐに約60kb大きくなり、マイクロコントローラーに収まらなくなる可能性があります。これは、ARMプラットフォームのコンパイラがC ++標準に厳密に準拠し、静的変数のストリームセーフな初期化を実装しているためです（複数のスレッドが突然getBuf（）関数に突然入った場合）。

さらに、このコードは、変数の初期化中にこの関数が再帰的に呼び出されるかどうかもチェックしますか？そしてこれは標準では未定義の動作ですが、g ++名の実装はrecursive_init_error例外をスローします。そして、例外があるので、つまり、これらの例外を処理するコードです。大きなプロセッサの標準ではそれほど多くありません（同じ60kbです）が、マイクロコントローラの場合、これは非常に不適切です。

解決策は、コンパイラキー-fno-threadsafe-staticsを追加することです。これは、マルチスレッド化がなく、このコードが1つのスレッドから厳密に呼び出されると確信している場合、このすべてのがらくたを無効にするように設計されています。

シングルトン


最後に、シングルトンを使用できます。これは、メモリ内に1つのコピーとして存在するオブジェクトです。一般的なケースでは、オブジェクトを動的に作成できますが、マイクロコントローラーの場合、コンパイル段階でオブジェクトを配布することは理にかなっています。

class Singleton
{
    int buf[300];

public:
    static Singleton & getInstance()
    {
        static Singleton instance;
        return instance;
    }
    
    int * getBuf()
    {
        return buf;    
    }
};

void setup() 
{
    Serial.begin(9600);
    Singleton::getInstance().getBuf()[42] = 10;
}

void loop()
{
    Serial.println(Singleton::getInstance().getBuf()[42]);
}

技術的には、これは上記のバリエーションと組み合わせです。ここでは、クラスの静的メンバーと静的ローカル変数を使用します。それでも、このオプションは、1つのエンティティにカプセル化されたアクセス/処理/変更のメソッド/機能をデータロードにアタッチできるという点でも優れています。この場合、データ自体は静的に分散されます。

00800116 l     O .bss	00000258 Singleton::getInstance()::instance

buf配列自体は、クラスデータの通常のメンバーとして宣言されていることに注意してください。配列内に含まれるインスタンスインスタンスは静的です。残念ながら、objdumpはオブジェクト内のデータを詳細に記述していません。シングルトンクラスの内部に、bufのほかに他のフィールドがある場合でも、objdumpはそれらをまとめてブラインドし、1行で表示します。

筋金入りでテンプレートの魔法が欲しい人は、この記事を読むことをお勧めします。昨日出たばかりで、マイクロコントローラーとの関係でシングルトンのテーマをより完全に明らかにします。

結論


小さなマイクロコントローラーのRAMは、原則としてそれほど多くありません。運が良ければ、ARM（STM32、nRF、NXP）またはESP8266を使用すると、数十キロバイト（マイクロコントローラーの「厚さ」によって異なります）まで自由に使用できます。 （Arduinoの一部として、または単独で）AVRを使用している場合、これは1〜4kbです。運が悪く、MCS51アーキテクチャを使用している場合は、数百バイトしか使用できません。

また、標準ライブラリでは動的メモリ割り当ての関数（新規、malloc）を使用でき、これらの関数は多くのライブラリ（たとえば、STMキューブの一部であるか、Arduinoの世界で利用可能）で使用されていますが、マイクロコントローラーでの動的メモリ割り当てにより、さらに多くのライブラリが作成されるようです利益よりも問題。

この記事では、コンパイルの段階で、メモリに長く存在するオブジェクトを静的に分散することを提案します。これにより、メモリの分布をより詳細に監視し、メモリが突然不足した場合のリスクを最小限に抑えることができます。

プログラム内のすべてのオブジェクトを急いで再実行してグローバル変数にすることは決してしないでください。説明されている方法は、ファームウェアの最初に作成され、プログラムが実行されている間ずっとライブしているオブジェクトとバッファに主に適しています。または、オブジェクトのプールを編成し、そこから特別なマネージャーが必要な人にオブジェクトを配布します。

存続期間の短いオブジェクトにこのメソッドを使用する価値はありません。スタックに配置する方がはるかに適切です。また、オブジェクトを作成するためにコンパイル段階で利用できない情報が必要な場合、このメソッドは機能しません。
