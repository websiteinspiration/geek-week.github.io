依存関係の逆転の原則


依存関係の逆転の原則（DIP）は2つの部分で構成されています。

  • 上位レベルのモジュールは、下位レベルのモジュールに依存するべきではありません。それらも他のものも抽象化に依存するべきです。
  • 抽象化は詳細に依存すべきではありません。詳細は抽象化に依存する必要があります。

部屋の一部を照らすためにランプをオンにしたかった最後の時間にリフレッシュします。壁に穴を開け、配線を探して掘り、露出して、ランプを家の配線に直接はんだ付けする必要がありましたか？もちろん、そうではありません（少なくとも私は望んでいません！）。コンセントは、このための標準インターフェースを提供します。ほとんどの先進国では、従来のランプが建物の電気配線に直接はんだ付けされるとは予想していません。さらに、コンセントにランプを1つだけ挿入できるとは誰も予想していません。ランプ、コンピュータ、テレビ、掃除機、その他の機器の接続に慣れています。標準の120ボルト、60ヘルツのコンセントは、米国のユビキタスな社会の一部となっています。

同じ原理がソフトウェア開発にも当てはまります。密接に相互接続されている一連のクラスを操作する代わりに、標準インターフェースを操作したいとします。さらに、LSP（Liskov Substitution Principle-Liskov Substitution Principle）によれば、このインターフェースの期待に違反することなく実装を置き換えることができることを保証したいと思います。したがって、インターフェイスを使用していて、それを置き換えることができるようにしたい場合は、特定の実装ではなく、インターフェイスのみを操作する必要があります。つまり、インターフェースに依存するコードは、そのインターフェースに関する詳細のみを知っている必要があります。彼は、このインターフェースを実装する特定のクラスを認識してはなりません。

ポリシー、詳細、および抽象化の関係


逆転の原理を少し言い換えると、ポリシー（高レベル）は詳細（実装）に依存すべきではなく、詳細はポリシーに依存すべきであると言えます。より高いレベルのポリシーは、それが参照する抽象化を定義する必要があります。そこでは、一部の実装が要求されたアクションを実行します。この視点は、依存関係の抽象化の原則だけでなく、依存関係の逆転の原則である理由を説明するのに役立ちます。

パーツのポリシーへの依存関係が依存関係の逆である理由の例として、FormatReaderServiceで記述したコードを検討してください。フォーマットされたファイルリーダーサービスはポリシーです。これは、IFileFormatReaderインターフェイスが行うべきこと、つまりこれらのメソッドの予想される動作を定義します。これにより、詳細の実装（個々の形式のリーダー）を考慮せずに、ポリシー自体に焦点を当て、リーディングサービスの動作を決定できます。したがって、リーダーはリーダーサービスによって提供される抽象化に依存しています。結局のところ、サービスと個々のリーダーはどちらも、フォーマットリーダーインターフェイスの抽象化に依存しています。

依存関係を逆にすることで接続を減らす


依存関係がない（接続性がない）ことは、クラスにとって非現実的であることを知っています。接続性がゼロの場合、使用可能なクラスのセットはありません。ただし、可能であれば、直接的なつながりを取り除くこともできます。システムを分割して、これらの部分以外は変更せずに個々の部分を変更できるようにしたいとします。依存関係の逆転の原則は、この概念の鍵です。インターフェイスや基本クラスなどの抽象化のみに依存することで、システムのさまざまな部分間の接続を減らすことができます。これにより、さまざまな実装でシステムを再構築できます。

必要な機能の一部を取得できるように、明確な階層でインスタンス化する必要がある多くのクラスを検討してください。最も簡単な方法は、最高レベルのクラス（呼び出したいクラス）を使用して1レベル下のクラスのインスタンスを作成し、それを使用してクラスをさらにインスタンス化するなどです。図14は、上位レベル（ポリシー）のオブジェクトが依存し、下位レベル（詳細）のオブジェクトに直接接続されているオブジェクトのグラフを示しています。

図15：抽象化を使用した分離。

このシナリオでは、インターフェースを導入することで、Barの実装をFooでの使用から分離できます。ただし、実装を分離するには、インターフェースを選択する必要があります。まだ依存構造を反転しておらず、このスキーマのすべての接続問題を修正していません。

このシナリオでバーを変更したい場合はどうなりますか？加えたい変更に応じて、IBarインターフェイスの変更を強制する連鎖反応を引き起こすことができます。 FooはIBarインターフェイスに依存しているため、Fooの実装も変更する必要があります。 Barの実装を分離した可能性がありますが、FooはBarへの変更に依存しました。つまり、ポリシーは依然として詳細に依存しています。

依存関係の構造を反転させて、パーツがポリシーに依存するようにする場合は、まず、視点を変更する必要があります。このシステムを使用する開発者は、インターフェースから実装を抽象化しただけではないことを理解する必要があります。はい、この分離は必要ですが、十分ではありません。抽象化が誰に属するかを理解する必要があります-政治または詳細。

依存関係の逆転の原則では、詳細はポリシーに依存する必要があるとされています。これは、パーツが実装する抽象化を定義および所有するポリシーが必要であることを意味します。スクリプトFoo-> IBar-> Barでは、IBarを単なるBarのラッパーとしてではなく、Fooの一部と見なす必要があります。構造的には何も変更されていませんが、図16に示すように、所有権の観点が変更されています。


図16：政治は抽象を所有しています。詳細はポリシーによって異なります。

FooにIBar抽象化がある場合、これらの2つの構成をBarとは独立したパッケージに入れることができます。独自の名前空間、独自のアセンブリなどに配置できます。これにより、どのクラスまたはモジュールがどのクラスまたはモジュールに依存しているかの可視性が大幅に向上します。 AssemblyAにFooとIBarが含まれ、AssemblyBがIBarの実装を提供していることがわかった場合、Barの細分性がFooによって定義されたポリシーに依存していることに気付く方が簡単です。

依存関係の構造を正しく反転させると、ポリシーや詳細の変更に対する連鎖反応も正しくなります。Barの実装を変更すると、変更の上向きのチェーンが観察されなくなります。これは、BarがFooによって提供される抽象化に準拠する必要があるためです-詳細はもはやポリシーの変更を指示しません。次に、Fooのニーズを変更すると、IBarインターフェイスが変更され、変更が構造全体に広がります。バー（詳細）はポリシーの変更に応じて強制的に変更されます。

メール送信システムの依存関係の分離と反転


コードベースを見ると、IFileFormatReaderが既に依存関係の逆転の例であることがわかります。 FormatReaderServiceクラスは、フォーマットリーダーインターフェイス定義を所有しています。フォーマットされたファイルリーダーサービスのニーズが変化すると、フォーマットリーダーに向かって下降する一連の変更が表示されます。ただし、特定のファイル形式のリーダーが変更された場合、これらの変更がフォーマットされたファイルの読み取りサービスに影響を与えることはほとんどありません。これにより、他にどこでシステムの依存関係を反転させることができるのか不思議に思います。

最初に行うことは、ログメッセージを受信し、それを電子メールで送信するロジックを送信フォームから分離することです。 2つの電子メールの読み取りおよび送信サービスへのリンクは気にしませんが、いつ、何を引き起こすかという疑問は疑問を投げかけます。プロセスが実際にフォームで複製されていることに気付きました。1回はファイルから送信するためで、もう1回はデータベースから送信するためです。次に、これを使用している他のすべての部門を思い出して、複製プロセスの実際の範囲について疑問を持ち始めます。さらに、あなたの友人の何人かは最近「ユニットテスト」について話しました。テストしているプロセスの実際のロジックが、外部システムへの参照を持たないオブジェクトにカプセル化されていることを確認する必要があると彼らは言っています。

そのことを念頭に置いて、ProcessingServiceというオブジェクトを作成することにしました。数分コードを前後に入れ替えてプロセスを統合しようとした後、処理サービスをデータベースリーダーまたはファイル読み取りサービスに直接接続したくないことに気づきました。少し考えてみると、GetMessageBodyメソッドという2つのパターンの間に気づくでしょう。このメソッドを基礎として使用して、新しいIMessageInfoRetrieverインターフェイスを作成し、データベースリーダーとファイルリーダーの両方として実装します。

public interface IMessageInfoRetriever
{ 
  string GetMessageBody(); 
}

public class FileReaderService: IMessageInfoRetriever
{
  public string GetMessageBody() {/* ... */}
}

public class DatabaseReaderService: IMessageInfoRetriever
{
  public string GetMessageBody() {/* ... */}
}
