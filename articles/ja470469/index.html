PVS-Studioがクラウドに移行：CircleCI


クラウドCIシステムでのPVS-Studio静的アナライザーの使用に関する一連の記事を続けます。今日、私たちは別のサービス-CircleCIを検討しています。今回は、Kodiメディアプレーヤーが分析のためのプロジェクトとして機能し、そのソースコードで興味深い場所を見つけようとします。

注意。PVS-StudioをクラウドCIシステムに統合することに関するその他の記事は、次の場所にあります。

  • PVS-Studioがクラウドに移行：Azure DevOps
  • PVS-Studioがクラウドに移行-Travis CIで分析を開始

アナライザー警告の設定と分析に直接進む前に、使用および分析されるソフトウェアについて少し説明します。

CircleCIは、ソフトウェアの組み立て、テスト、公開を自動化するためのクラウドベースのCIサービスです。コンテナーと、Windows、Linux、およびmacOSを実行する仮想マシンの両方でのプロジェクトのアセンブリをサポートします。

Kodiは無料のオープンソースクロスプラットフォームメディアプレーヤーです。ホームコンピュータとローカルネットワークまたはインターネットの両方にあるオーディオファイルとビデオファイルを再生できます。プラグインをインストールすることにより、テーマと機能をサポートします。 Windows、Linux、macOS、Androidで利用できます。

PVS-Studio-C、C ++、C＃、Javaで記述されたコードのエラーや潜在的な脆弱性を検索するための静的コードアナライザー。Windows、Linux、およびmacOSで動作します。

カスタマイズ


CircleCI ホームページにアクセスし、「サインアップ」ボタンをクリックします


次のページでは、GitHubまたはBitbucketアカウントで認証するように求められます。GitHubを選択し、CircleCIアプリケーション認証ページに移動します。


アプリケーションを承認し（緑色のボタン[circleciを承認]）、ウェルカムページ「Welcome to CircleCI！」にリダイレクトします。


このページでは、CircleCIでアセンブルするプロジェクトをすぐに構成できます。リポジトリにマークを付け、「フォロー」をクリックします。

リポジトリを追加すると、CircleCIは自動的にビルドを開始しますが、リポジトリにはまだ設定ファイルがありません-ビルドタスクは失敗します。


構成ファイルを追加する前に、アナライザーのライセンスデータを含むプロジェクト変数を追加します。これを行うには、左側のペインで[設定]をクリックし、[組織]グループで[プロジェクト]を選択して、必要なプロジェクトの右側にある歯車をクリックします。設定ウィンドウが開きます。


「環境変数」のセクションに興味があります。その中に入って、アナライザーのユーザー名とライセンスキーを含むPVS_USERNAMEおよびPVS_KEY変数を作成します。


プロジェクトのビルドを開始すると、CircleCIはリポジトリ内のファイルからパス.circleci / config.ymlに沿ってタスク構成を読み取ります。追加してください。

PVS-Studioリポジトリを追加し、アナライザーをインストールします。

- run: wget -q -O - https://files.viva64.com/etc/pubkey.txt
             | sudo apt-key add - 
       && sudo wget -O /etc/apt/sources.list.d/viva64.list
             https://files.viva64.com/etc/viva64.list
- run: sudo -- sh -c "apt-get update 
             && apt-get install pvs-studio -y"

プロジェクトの依存関係を収集しましょう：

- run: sudo make -C tools/depends/target/flatbuffers PREFIX=/usr/local

アセンブリディレクトリにMakefileを生成します。

- run: mkdir build && cd build && cmake -DCMAKE_BUILD_TYPE=Debug ..

次のステップは、プロジェクトの静的分析を構成して実行することです。

まず、アナライザーライセンスを含むファイルを作成します。2番目のコマンドは、プロジェクトアセンブリトレースのコンパイルを開始します。

トレース後、静的解析を直接開始します。トライアルライセンスを使用する場合、アナライザーをパラメーター

--disableLicenseExpirationCheckで開始する必要があります。

最後のコマンドは、アナライザーの結果を含むファイルをHTMLレポートに変換します。

- run: pvs-studio-analyzer credentials -o PVS.lic ${PVS_USER} ${PVS_KEY}
- run: pvs-studio-analyzer trace -- make -j2 -C build/
- run: pvs-studio-analyzer analyze -j2 -l PVS.lic 
          -o PVS-Studio.log --disableLicenseExpirationCheck
- run: plog-converter -t html -o PVS-Studio.html PVS-Studio.log

テストが完了したら、アナライザーレポートを保存します。

- run: mkdir PVS_Result && cp PVS-Studio.* ./PVS_Result/
- store_artifacts:
          path: ./PVS_Result

コードのフォーマットから判断すると、次の実行ロジックが想定されています。

  • 場合arttypesが NULLポインタである、方法を完了する。
  • もしarttypes - null以外のポインタ、ベクトルクリアartworkMapを、そしてより多くのいくつかのアクションを実行します。

ただし、欠落している文字「;」調整した; その結果、実行ロジックはフォーマットにまったく対応していません。その結果、次のようになります。

  • 場合arttypesがヌルポインタである、artworkMapのベクトルであり、クリア及び方法は終了。
  • 場合arttypesが非ゼロのポインタであり、更なるアクションが実行されるが、artworkMapのベクトルがされていないクリーンアップ。

一般に、間違いがないことはほとんどありません。そして、誰かがreturnArtworkMap.clear（）の精神で表現を書くことはありそうにありません。:)。
