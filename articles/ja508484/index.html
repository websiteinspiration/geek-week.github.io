Typescript Compiler API：コンパイラを制御する





Typescriptアプリケーション開発には、常にプロジェクトのビルド段階があります。通常、これにはビルドシステムとワークフロー自動化が使用されます。たとえば、十分なプラグインでハングアップしたwebpackやgulp、またはビルドプロセスは、プロジェクトのフレームワークで使用されるネイティブのtscまたはCLIコマンドを使用して、package.jsonコマンドとシェルスクリプトに分散されます。これらすべてのソリューションには長所と短所があります。多くの場合、アセンブリプロセス中に非標準的な処理を行う必要があり、使用されているアセンブリシステムはボックスから必要な機能を提供せず、既存のプラグインは必要な機能を十分に備えていません。そのような瞬間に、プロジェクトでの作業が始まり、構成での必死な選択が始まり、適切なプラグインの検索が始まります。ある時点で、適切な松葉杖を探すのに費やした時間に、あなたはあなたの決定を書くことができます。



多くの場合、プロジェクトの重要なプロセスはJavaScriptスクリプトで自動化できます。その表現力と機能により、特定のビルドシステム用のプラグインの存在を気にすることなく、目的のワークフローを記述し、さまざまなライブラリから選択することができます。このアプローチの重要な利点は、プロセスの完全な制御と最大の柔軟性です。 Typescriptを主要な開発言語として使用するプロジェクトの場合、それをコンパイルするプロセスをワークフローに組み込む方法について疑問が生じます。 Typescript Compiler APIがここで役に立ちます。この記事では、Typescriptに実装されたプロジェクトをコンパイルし、作業のさまざまな段階でコンパイラーと対話し、RESTサーバーのホットリロード用のスクリプトを作成するためにそれを使用する方法を説明します。Nest.jsによって開発されました。



空き時間に作業しているノードにプロジェクトがあります。当初は離陸しない中小企業を対象としていましたが、今では実験のトレーニングの場として使用しています。このプロジェクトは、もともとNuxt.jsとFastifyを組み合わせて構築されたもので、多くの変換を経てきました。ここで、typescriptへの移行とモノリポジトリへの変換、そしてFastifyをNest.jsに置き換えます。プロジェクトの負荷は小さく、深刻なものではないため、これらすべてが1つのサーバーで回転することが想定されていたため、RESTサーバーとNuxt.jsを介したフロントエンド配信の両方に単一のExpressインスタンスを使用しました。このため、Nest.jsの内部で実行されるExpressがNuxt.jsのレンダリング機能を取得します。



このソリューションは本番環境ではうまく機能しますが、私はプロジェクトだけで作業していたので、API関数を記述してインターフェースを並行して作成すると便利でした。このため、ファイルの変更時にNest.jsを再起動しても、HMRモードで動作するNuxt.jsが再起動することはありません。そうしないと、ビルドプロセスが新たに開始され、意味がありません。しかし、ここで問題が発生しました。実際、CLI Nest.jsにはホットリロードを実行できる監視モードがありますが、プロジェクトが再コンパイルされたときに再起動する別のプロセスでそれをロードします。このため、このモードでサーバーを再起動するたびにNuxtのコンテキストが失われ、完全なアセンブリが開始されます。



これらすべてが明らかになった後、私はプロジェクトで独自のホットリロードを作成することにしました。最初は、Nest.jsドキュメントで提案されているように、webpackを使用してソリューションを実装しようとしましたが、再起動を実装するためにそこで使用されるstart-server-webpack-pluginがCLIと同じ戦略を使用していることがわかりました。別のプロセスで実行し、必要に応じて再起動しました。必要に応じて機能するようにこのプラグインを書き直しましたが、Webpackを使用することで解決策はかなり難しくなり、あちこちに表面化した問題がないわけではなく、プロジェクトの開発に集中することが困難になりました。



次に、アプローチを変更する必要があると判断し、常に時計仕掛けのように迅速かつ安定して機能するtscコンパイラーに規定されたメカニズムを直接使用したいと思いました。見つけたTypeScriptコンパイラAPIの使用例を示した優れた記事。Typescriptライブラリに組み込まれたツールを使用すると、必要な機能を実装でき、非常に便利です。残念ながら、TypescriptコンパイラAPIに関する情報がほとんどないことに出くわしたので、見つけた情報を共有することにしました。また、執筆時点でのこのAPIは開発中であり、いくつかの点は時間とともに変更される可能性があることにも注意してください。



簡単なコンパイル



TypeScriptコンパイラAPIの原理を理解するために、まず、それを使ってプロジェクトをコンパイルしてみましょう。私が実験するプロジェクトは、nest.jsの標準構造を持っています。ソースはsrcフォルダーにあり、エントリポイントは./src/main.tsファイルです。



コンパイラーがプロジェクトを正常にコンパイルするためには、コンパイルパラメーターをそれに渡す必要があります。最も簡単な方法は、スクリプトで正しく解析することです。 tsconfig.jsファイルを作成します。 



const {ModuleResolutionKind, ModuleKind, ScriptTarget} = require("typescript")

module.exports =  {
    moduleResolution: ModuleResolutionKind.NodeJs,
    module: ModuleKind.CommonJS,
    target: ScriptTarget.ES2019,
    declaration: true,
    removeComments: true,
    emitDecoratorMetadata: true,
    experimentalDecorators: true,
    sourceMap: true,
    outDir: "./dist",
    baseUrl: "./",
    allowJs: true,
    skipLibCheck: true,

moduleResolution、module、targetなどの一部のパラメーターは列挙によって定義されるため、tsconfig.jsonから直接パラメーターを取得することはできません。パラメーターが文字列として書き込まれているため、最も簡単な例の実装には使用しません。 tsconfig.jsonからパラメーターを正しく取得する方法については、少し後で説明します。



これでコンパイルオプションが用意され、次のステップに進むことができます。これを行うには、次の内容でts-compiler.jsファイルを作成します。 



const ts = require('typescript');

function compile() {
  const compilerOptions = require('./tsconfig');
  const program = ts.createProgram(['./src/main.ts'], compilerOptions);
  const emitResult = program.emit();
}

compile();


このスクリプトを実行する場合、コンパイルされたプログラムファイルは./distディレクトリに表示されます。コンパイルは2つの段階で行われます。最初に、createProgramコマンドはソースファイルを分析し、コンパイルするファイルのリストを作成します。その後、emit関数を使用して、コンパイルしたファイルをコンパイルして保存します。ものすごく単純。



使いやすさを向上させるには、tsconfig.jsonファイルからのコンパイルオプションのロードを実装する必要があります。スクリプトに次の関数を追加します。 




const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function getTSConfig() {
  const configPath = ts.findConfigFile('./', ts.sys.fileExists, 'tsconfig.json');
  const readConfigFileResult = ts.readConfigFile(configPath, ts.sys.readFile);
  if (readConfigFileResult.error) {
    throw new Error(ts.formatDiagnostic(readConfigFileResult.error, formatHost));
  }
  const jsonConfig = readConfigFileResult.config;
  const convertResult = ts.convertCompilerOptionsFromJson(jsonConfig.compilerOptions, './');
  if (convertResult.errors && convertResult.errors.length > 0) {
    throw new Error(ts.formatDiagnostics(convertResult.errors, formatHost));
  }
  const compilerOptions = convertResult.options;
  return compilerOptions;
}


構成ファイルを操作するために、APIは、構成を検索、ロード、およびコンパイラーが理解できる形式に変換できるようにする多数の関数を提供します。ここでは、findConfigFile関数を使用して構成ファイルへのパスを決定し、readConfigFileを使用してjsonとしてロードし、エラーがない場合は、convertCompilerOptionsFromJsonを使用してコンパイルパラメーターを取得します。



設定の読み込み中に問題が発生した場合、APIに組み込まれたエラーメッセージ転送メカニズムを使用して、関数が例外をスローします。ライブラリは、すべてのエラーメッセージと警告をDiagnosticクラスのオブジェクトとして返します。コンソールに出力するためのメッセージを作成するには、標準ライブラリのフォーマット関数を使用できます。この場合、1つのメッセージにはformatDiagnostic、配列にはformatDiagnosticsを使用できます。フォーマット機能が正しく機能するためには、メッセージのフォーマットに重要なパラメーターを含むformatHostオブジェクトが必要です。

この段階で、コンパイラがコードにエラーを見つけた場合、それについてはわかりません。修正しましょう。コンパイル段階で検出されたコード内のすべてのエラー、emit関数は、EmitResultオブジェクトのdiagnostcsフィールドに戻ります。コードのエラーを美しく表示するには、formatDiagnosticsWithColorAndContext関数を使用できます。プログラム作成フェーズで検出されたエラーをチェックすることも意味があります。これを行うには、getPreEmitDiagnostics関数を使用します。スクリプトを次のように補足します。 



function compile() {
  const compilerOptions = getTSConfig();
  const program = ts.createProgram(['./src/main.ts'], compilerOptions);
  console.log(
    ts.formatDiagnosticsWithColorAndContext(
      ts.getPreEmitDiagnostics(program),
      formatHost,
    ),
  );
  const emitResult = program.emit();
  console.log(
    ts.formatDiagnosticsWithColorAndContext(emitResult.diagnostics, formatHost),
  );
  return emitResult.diagnostics.length === 0;

つまり、ソースをコンパイルし、エラーが発生した場合にコンソールにエラーを表示するスクリプトを取得しました。ご覧のとおり、コンパイルのすべてのステージと結果にアクセスできるため、このアプローチにより、コンパイルプロセスを非常に柔軟に実装できます。コンパイラに必要な一連の関数を含むCompilerHostをカスタマイズすることで、柔軟性をさらに高めることができます。コンパイラがプロセスで読み取るファイルへのパスをコンソールに表示したいとします。これは、カスタムCompilerHostをcreateProgram関数に渡すことで簡単に行えます。実際にどのように機能するか見てみましょう。



ファイルパスの数百行の足布で出力を詰まらせたくないので、パスを1行で出力する関数を記述します。 



function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
  };
  return displayFunction;
}


次に、compile（）関数コードを追加します。 



function compile() {
  const compilerOptions = getTSConfig();
  const compilerHost = ts.createCompilerHost(compilerOptions);
  compilerHost.readFile = displayFilename(compilerHost.readFile, 'Reading');
  compilerHost.readFile.enableDisplay();
  const program = ts.createProgram(
    ['./src/main.ts'],
    compilerOptions,
    compilerHost,
  );
  compilerHost.readFile.disableDisplay();

  console.log(
    ts.formatDiagnosticsWithColorAndContext(
      ts.getPreEmitDiagnostics(program),
      formatHost,
    ),
  );

  compilerHost.writeFile = displayFilename(compilerHost.writeFile, 'Emitting');
  compilerHost.writeFile.enableDisplay()

  compilerHost.writeFile.disableDisplay();
  console.log(

DIYインクリメンタルプログラムウォッチャー



前のセクションでは、TypeScriptコンパイラAPIを使用してTypescriptコンパイルプロセスをカスタマイズする方法を説明しました。さて、開発のスピードと使いやすさを本当に向上させることができる、もっと緊急で役立つ何かをする時が来ました。プログラムのコンパイル中に、コンパイラーはプログラム内のすべての依存関係を解決する必要があります。



これを行うには、プロジェクトファイルとライブラリファイルを順番に読み取り、分析します。私のような比較的小さなプロジェクトをコンパイルする場合でも、準備の段階で、コンパイラーは2000以上のファイルを読み取り、解析し、分析する必要があります。これにより、コンパイルプロセスが大幅に遅くなります。たとえば、私のラップトップでのコンパイルには約20秒かかります。同時に、開発中は通常、コンパイラの起動間で変更されるファイルはほんの数ファイルなので、Typescriptにはバージョン2.7以降、インクリメンタルコンパイルのオプションがあり、ウォッチモードとともにディスク上のファイルへの変更を監視できるため、再コンパイルの速度が大幅に向上します。



これらの関数はCLI Nest.jsでも使用され、次のコマンドを使用してこのモードで作業を開始できます 



nest start --watch


この開発モードを利用するもう1つの機会は、-watchスイッチを指定してtscを実行することです。しかし、冒頭ですでに述べたように、ある時点で標準コマンドの機能が十分でなくなった場合は、Typescriptの内部を調べて、そこでどのように機能するかを調べます。



まず、前のケースと同様に、次の内容でts-wather.jsを作成します。 



const ts = require('typescript');

const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

async function watchMain() {
  const configPath = ts.findConfigFile(
    './',
    ts.sys.fileExists,
    'tsconfig.json',
  );

  const host = ts.createWatchCompilerHost(
    configPath,
    {},
    ts.sys,
    ts.createEmitAndSemanticDiagnosticsBuilderProgram,
    diagnostic =>
      console.log(
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
    diagnostic =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );
  ts.createWatchProgram(host);
}
watchMain();

ホストでさまざまなものをフックするのを便利にするために、小さなヘルパーを書きましょう： 



function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function() {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}


これにより、ホストで呼び出された関数の前後に必要なアクションを実行でき、それに渡される引数と実行結果を受け取ります。これで、このヘルパーを使用して、さまざまな段階でコンパイラーを簡単に妨害できます。たとえば、コンパイルの準備としてコンソールにメッセージを追加しましょう。

。それでは、それが何であるかをやってみましょう-コンパイルされたプログラムを再起動します。これを行うには、コンパイルされたファイルがディスクに書き込まれた後に起動するフックを作成する必要があります。 



let currentProgram;
  on(
    host,
    'afterProgramCreate',
    (program) => {
      console.log('** We finished making the program! Emitting... **');
      currentProgram = program;
    },
    () => {
      console.log('** Emit complete! **');
      const onAppClosed = () => {
        if (app) {
          setTimeout(onAppClosed, 100);
        } else {
          clearCache();
          require('./dist/bootstrap')
            .bootstrap()
            .then((res) => {
              app = res;
            });
        }
      };
      if (currentProgram && currentProgram.getSemanticDiagnostics().length === 0) {
        onAppClosed();
      }
    },

ここで何が起こっているのかを理解しましょう。まず、コードを変更して再コンパイルを開始した後でサーバーを正しく閉じるために、ブートストラップ関数の結果を書き込むapp変数が必要です。サーバーのシャットダウンコマンドを呼び出すためのハンドラーが含まれます。次に、コンパイルプロセス中にコードでエラーが見つからなかった場合にのみ、サーバーが起動します。これを行うには、診断の数がゼロであることをgetSemanticDiagnosticsで確認します。そして3番目に、モジュール式システムはロードされたモジュールをキャッシュするため、コードを変更した後、依存関係からキャッシュをクリーンアップする必要があります。これを行うために、以前に呼び出した小さなヘルパー関数clearCacheを書きました。 



function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      );
    });
}


ここでは、サーバーの正常な再起動を妨げる可能性のある依存関係がキャッシュからクリアされています。この場合、プロジェクトのすべてのモジュールをキャッシュのdistディレクトリから削除する必要があります。typeormモジュールも削除します。これは、それらの実装の機能がサーバーの再起動を妨げるためです（これが起こる理由は別の話です）。タイムアウトを使用して、サーバーが閉じ、アプリ変数がクリアされるまで待機します。コンパイラがファイルの変更を検出し、変更されたファイルでプログラムの作成を開始した時点でサーバーを閉じます。これを行うには、createProgram関数にフックを追加します。

最後の仕上げは、process.exit（）の置き換えです。

一般に、これを行うべきではありませんが、Nest.jsは起動プロセス中に問題が発生した場合にそれを引き起こすという悪い癖があるため、exit関数をエラーメッセージに置き換えます。この場合、スクリプトはprocess.exit（）を呼び出した後も引き続き機能します。これは、watchMain関数を呼び出す前に行う必要があります。



そこで、スクリプトを実行して、何が起こったかを確認します。







ここで、サーバーコードでエラーを発生させます。その後、サーバーが起動せず、エラーメッセージが表示され、エラーが修正された後、サーバーは正常に再起動します。以下は、結果のスクリプトです。



ts-compiler.js 
const ts = require('typescript');
const { join, dirname } = require('path');
const { exit } = require('process');

const formatHost = {
  getCanonicalFileName: (path) => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function () {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}

function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      process.stdout.clearLine(); // clear current text
      process.stdout.cursorTo(0); // move cursor to beginning of line
      process.stdout.write(
        `Clearing cache ${Math.floor((index * 100) / arr.length + 1)}%`,
      );
    });
  process.stdout.write(' finished.\n');
}

function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  let counter = 0;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    counter++;
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    counter = 0;
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
    console.log(`${counter} times function was called`);
  };
  return displayFunction;
}

async function watchMain() {
  const configPath = ts.findConfigFile(
    './',
    ts.sys.fileExists,
    'tsconfig.json',
  );

  const host = ts.createWatchCompilerHost(
    configPath,
    {},
    ts.sys,
    ts.createEmitAndSemanticDiagnosticsBuilderProgram,
    (diagnostic) =>
      console.log(
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
    (diagnostic) =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );

  host.readFile = displayFilename(host.readFile, 'Reading');
  let app;
  on(
    host,
    'createProgram',
    () => {
      console.log("** We're about to create the program! **");
      app && app.close().then(() => (app = undefined));
      host.readFile.enableDisplay();
    },
    () => {
      host.readFile.disableDisplay();
      console.log('** Program created **');
    },
  );
  let currentProgram;
  on(
    host,
    'afterProgramCreate',
    (program) => {
      console.log('** We finished making the program! Emitting... **');
      currentProgram = program;
    },
    () => {
      console.log('** Emit complete! **');
      const onAppClosed = () => {
        if (app) {
          setTimeout(onAppClosed, 100);
        } else {
          clearCache();
          require('./dist/bootstrap')
            .bootstrap()
            .then((res) => {
              app = res;
            });
        }
      };
      if (currentProgram && currentProgram.getSemanticDiagnostics().length === 0) {
        onAppClosed();
      }
    },
  );

  ts.createWatchProgram(host);
}
process.exit = (code) => {
  console.log('!!!!!!!!!!!!!!!!!!!!!!! WARNING!!!!!!!!!!!!!!!!!!!!!!!!!');
  console.trace(`Process try to exit with code ${code}.`);
};
watchMain();
