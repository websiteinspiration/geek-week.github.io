動作原理


Mockerは、どこかにホストされている小さなWebサービスで、特定のポートでトラフィックをリッスンし、事前に準備されたデータで特定のネットワーク要求に応答できます。

シーケンスは次のとおりです

。1.クライアントがリクエストを送信します。
2. Mockerがリクエストを受け取ります。
3. Mockerが目的のモックを見つけます。
4. Mockerは目的のモックを返します。

ポイント1、2、および4ですべてが明確である場合、3は問題を提起します。

サービスがクライアントに必要なモックを見つける方法を理解するために、まずモック自体の構造を検討します。

モックは次の形式のJSONファイルです。

{
    "url": "string",
    "method": "string",
    "statusCode": "number",
    "response": "object",
    "request": "object"
}

各フィールドを個別に分析しましょう。

url


このパラメーターは、クライアントがアクセスする要求のURLを指定するために使用されます。

たとえば、モバイルアプリケーションがurlをリクエストする場合host.dom/path/to/endpoint、フィールドurlに書き込む必要があります/path/to/endpoint。
つまり、このフィールドには、エンドポイントへの相対パスが格納されます。

このフィールドはurl-template形式でフォーマットする必要があります。つまり、次の形式を使用できます。

  1. /path/to/endpoint-通常のURLアドレス。リクエストが受信されると、サービスは行を文字ごとに比較します。
  2. /path/to/endpoint/{number}-パスパターンのあるURL。このようなURLのモックは、このパターンに一致するすべてのリクエストに応答します。
  3. /path/to/endpoint/data?param={value}-パラメータパターンのあるURL。このようなURLのモックは、指定されたパラメーターを含むリクエストをトリガーします。さらに、パラメータの1つがリクエストにない場合、テンプレートと一致しません。

したがって、URLパラメータを制御することで、特定のモックが特定のURLに戻ることを明確に判断できます。

方法


これは予想されるhttpメソッドです。たとえばPOSTまたはGET。
文字列には大文字のみを含める必要があります。

statusCode


これは、応答のhttpステータスコードです。つまり、このモックを要求すると、クライアントはstatusCodeフィールドにステータスが記録された応答を受け取ります。

応答


このフィールドには、要求への応答の本文でクライアントに送信されるJSONオブジェクトが含まれています。

リクエスト


ここには、クライアントからの受信が予想されるリクエストの本文が記録されており、リクエストの本文に応じて、目的の応答を返すために使用されます。たとえば、リクエストパラメータに応じて回答を変更する場合などです。

キャッシングプロキシ


Mockerはキャッシングプロキシモードで動作します。つまり、サービスがクライアントからリクエストを受信すると、実サーバーとスキームが配置されているホストのアドレスを取り出します（プロトコルを決定するため）。次に、結果のクエリを取得し（そのすべてのHederを使用して、メソッドが認証を必要とする場合、心配しないで、Authorization: Bearer ...転送されます）、彼のサービス情報を切り出し（同じhost、およびscheme）、リクエストを実サーバーに送信します。

200番目のコードを含む応答を受信した後、Mockerは応答をMockファイルに保存し（そう、それをコピーまたは変更できます）、実サーバーから受信したものをクライアントに返します。さらに、彼はファイルをランダムな場所に保存するだけでなく、ファイルを整理して、手本となるようにし、Mockerは次のURLにリクエストを送信しますhostname.dom/main/products/loans/info。それから彼は、フォルダを作成しますhostname.dom、それの内側に、彼は、フォルダを作成し、mainその中に、フォルダをproducts...

だから、モキが重複しないこと、名前がに基づいて形成されたHTTPメソッド（GET、PUT ...）と実サーバのレスポンスボディからハッシュ。この場合、特定の回答にすでにモックがある場合、それは単に上書きされます。

この機能は、リクエストごとに個別にアクティブ化できます。これを行うには、リクエストに3つのヘッダーを追加します。

X-Mocker-Redirect-Is-On: "true",
X-Mocker-Redirect-Host: "hostaname.ex:1234",
X-Mocker-Redirect-Scheme: "http"

配備


Mockerをデプロイする最も簡単な方法は、Dockerを使用することです。さらに、Dockerからサービスをデプロイすると、Webベースのインターフェースが自動的にデプロイされ、Mokaでの作業がより便利になります。Dockerを介したデプロイに必要なファイルはリポジトリにあります。

ただし、このオプションが適切でない場合は、ソースからサービスを個別にアセンブルできます。これで十分です：

git clone https://github.com/LastSprint/mocker.git
cd mocker
go build .

次に、構成ファイル（例）を記述してサービスを開始する必要があります。

mocker config.json

  • 新しいファイルを作成するたびにcurl mockerhost.dom/update_models、サービスがファイルを再度読み取るようにする必要があります。それ以外の場合は迅速かつエレガントな方法で更新できませんでした
  • CloudCommanderにバグがある（または私が何か間違ったことをした）ことがあり、Webインターフェイスを介して作成されたmokiを編集できない場合があります。ブラウザのキャッシュを消去することで処理されます。
  • application/json. form-url-encoding.


Mockerは、何らかの理由でサーバーの準備ができていない場合にクライアント/サーバーアプリケーションの開発の問題を解決するWebサービスです。

このサービスを使用すると、同じURLにさまざまなモックを作成したり、URLでパラメーターを明示的に指定したり、予期されるリクエスト本文を直接設定したりして、リクエストとレスポンスを相互に接続できます。このサービスには、ユーザーの生活を大幅に簡素化するWebベースのインターフェースがあります。

サービスの各ユーザーは、必要なエンドポイントと必要な要求を個別に追加できます。この場合、クライアントで実サーバーに切り替えるには、定数をホストアドレスに置き換えるだけで十分です。
