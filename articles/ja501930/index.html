Springはその動作をカスタマイズするためにどのような機会を提供しますか？

みなさん、こんにちは。ウラジスラフ・ロダンと接触。現在、私はOTUSのHigh Load Architectコースの責任者を務めており、ソフトウェアアーキテクチャのコースも教えています。

教えることに加えて、HaberでOTUSブログの著作権資料も書いています。現在募集中のコース「Developer on the Spring Framework」を開始するための今日の記事と一致させたいです。



前書き


読者の視点から見ると、Springを使用するアプリケーションコードは非常にシンプルに見えます。いくつかのBeanが宣言され、クラスに注釈が付けられ、必要に応じてBeanが挿入され、すべてが正常に機能します。しかし、好奇心旺盛な読者には質問があります。何が起こっていますか？"。この記事では、この質問に答えようとしますが、アイドルの好奇心を満たすためではありません。

Springフレームワークは十分な柔軟性があることで知られており、フレームワークの動作をカスタマイズするためのオプションを提供します。Springには、特定のアノテーション（トランザクションなど）を適用するためのかなり興味深いルールも多数あります。これらのルールの意味を理解し、それらを導き出し、さらにSpringで何をどのように構成するかを理解するには、Springの内部のいくつかの原則を理解する必要があります。ご存知のように、いくつかの原則の知識は多くの事実の知識から免除されます。確かに知らない場合は、以下の原則に慣れることをお勧めします。

設定の読み取り


最初に、アプリケーションの構成を解析する必要があります。構成にはいくつかのタイプ（xml-、groovy-、java-configurations、アノテーションに基づく構成）があるため、それらを読み取るためにさまざまなメソッドが使用されます。いずれにせよ、Map <String、BeanDefinition>という形式のマップが収集され、Beanの名前がそのBean定義に割り当てられます。BeanDefinitionクラスのオブジェクトは、Beanに関するメタ情報であり、BeanのID-ID、名前、クラス、destroy-およびinit-methodsが含まれています。

このプロセスに含まれるクラスの例：GroovyBeanDefinitionReader、XmlBeanDefinitionReader、AnnotatedBeanDefinitionReader、BeanDefinitionReaderインターフェースを実装。

Beandefinitionsのセットアップ


ですから、私たちは豆について説明していますが、豆自体はなく、まだ作成されていません。Beanを作成する前に、Springは結果のBeanDefinitionsをカスタマイズする機能を提供します。これらの目的のために、BeanFactoryPostProcessorインターフェースが使用されます。次のようになります。

public interface BeanFactoryPostProcessor {
    void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) throws BeansException;
}

このインタフェースのメソッドパラメータを使用すると、独自のgetBeanDefinitionNamesメソッドを使用して、マップからBeanDefinitionsを取得して編集できる名前を取得できます。

なぜこれが必要なのでしょうか？一部のBeanが、データベースなどの外部システムに接続するために詳細を必要とするとします。Beanはすでに詳細で作成されている必要がありますが、詳細自体はproperty-fileに格納されています。標準のBeanFactoryPostProcessors-PropertySourcesPlaceholderConfigurerの1つを適用できます。これにより、BeanDefinitionのnameプロパティが、プロパティファイルに格納されている実際の値に置き換えられます。つまり、BeanDefinionのValue（ "user"）をValue（ "root"）に実際に置き換えます。これを機能させるには、もちろん、PropertySourcesPlaceholderConfigurerを接続する必要があります。ただし、これに限定されず、BeanFactoryPostProcessorを登録して、BeanDefinitionsの処理に必要なロジックを実装できます。

Beanの作成


この段階で、キーによって特定されたBeanの名前と、値によって構成されたBeanDefinitionsを持つマップがあります。次に、これらのBeanを作成する必要があります。これはBeanFactoryが行うことです。ただし、ここでは、FactoryBeanを作成して登録することにより、カスタマイズを追加できます。 FactoryBeanは次の形式のインターフェースです。

public interface FactoryBean {
    T getObject() throws Exception;
    Class<?> getObjectType();
    boolean isSingleton();
}

したがって、BeanFactoryは、Beanクラスに対応するFactoryBeanがない場合にBean自体を作成するか、このBeanの作成をFactoryBeanに要求します。微妙なニュアンスがあります。Beanのスコープがシングルトーンの場合、この段階でBeanが作成され、プロトタイプの場合、このBeanが必要になるたびに、BeanFactoryから要求されます。

その結果、再びマップを取得しますが、すでに少し異なります。キーには、Beanの名前とBean自体の値が含まれています。しかし、これはシングルトーンにのみ当てはまります。

Beanの構成


今最も興味深いステージがやって来ます。作成されたBeanを含むマップがありますが、これらのBeanはまだ構成されていません。つまり、Beanの状態を設定するアノテーションAutowired、Valueは処理しませんでした。また、Beanの動作を変更するアノテーション（Transactional、Async）も処理しませんでした。対応するインターフェースの実装であるBeanPostProcessorsにより、この問題を解決できます。

public interface BeanPostProcessor {
    Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException;
    Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException;
}

恐ろしいが包括的な名前を持つ2つのメソッドが表示されます。どちらのメソッドもBeanを入力として受け取り、そこからBeanがどのクラスかを確認し、Reflection APIを使用して注釈を処理します。 Beanメソッドを返します。プロキシに置き換えられる可能性があります。

各Beanをコンテキストに配置する前に、すべてのBeanPostProcessorsに対してpostProcessBeforeInitializationメソッドがトリガーされ、次にinitメソッドがトリガーされ、さらにすべてのBeanPostProcessorsに対してpostProcessAfterInitializationメソッドがトリガーされます。

これらの2つのメソッドのセマンティクスは異なります。postProcessBeforeInitializationは状態注釈を処理し、postProcessAfterInitializationは動作を処理します。これは、プロキシを使用して動作を処理するため、注釈が失われる可能性があるためです。それが最後の場所で行動が変わる理由です。

カスタマイズはどこにありますか？アノテーションであるBeanPostProcessorを記述すれば、Springがそれを処理します。ただし、BeanPostProcessorが機能するには、Beanとしても登録されている必要があります。

たとえば、フィールドに乱数を埋め込むには、InjectRandomIntアノテーション（フィールドにぶら下がっている）を作成し、作成したアノテーションを処理する最初のメソッドでInjectRandomIntBeanPostProcessorを作成して登録します。2番目のメソッドでは、着信Beanを返します。

Beanのプロファイルを作成するには、メソッドにブロードキャストするProfileアノテーションを作成し、ProfileBeanPostProcessorを作成して登録します。最初のメソッドでは受信Beanを返し、2番目のメソッドでは、時間と実行ログを使用して元のメソッドへの呼び出しをラップするプロキシを返します。
