軍拡競争



5月、Google I / O 2019は、Jetpack Composeと呼ばれる宣言型Android UIを開発するための新しいフレームワークを発表しました。1か月後のWWDC 2019で、SwiftUIと呼ばれるiOS用の宣言型UIフレームワークが発表されました。これらの会議の後、モバイル開発が何を目指しているのかが明らかになり、現在これらのフレームワークで何ができるのか、それらの違いは何かを知りたいと思いました。

こんにちは世界


SwiftUIを使い始めるのは非常に簡単です。「SwiftUIを使用」チェックボックスを選択しながら、macOSをベータ版にアップグレードし、Xcode 11ベータ版をインストールして、新しいプロジェクトを作成するだけです。



Jetpack Composeの場合、すべてがより複雑になります。Jetpackから最新のビルドをダウンロードする必要があります。そして、ui-demosモジュールに基づいて、入力することにより、フレームワークの操作方法と必要な依存関係を把握します。SwiftUIと比較したComposeの非常に悲しい第一印象。



作成例

    @Composable
    fun HelloWorld() {
        Text(text = "Hello world")
    }

SwiftUIの例

struct ContentView: View {
    var body: some View {
        Text("Hello SwiftUI!")
    }
}

UIのレイアウトは非常によく似ています。これは、宣言的な外観になる傾向があるため、論理的です。

ビジュアルエディター


SwiftUiでは、ビジュアルエディタ（キャンバス）付属し
ている：

-コードを変更した後すぐに画面を再描画。



-一度に複数のデバイスを表示できます。



-エディターから要素の属性UIを変更できるようにします。


-そして最も面白くてクール-エディタからuiコードを実行して、画面がどのように機能するかを確認できます。


Jetpack Composeの場合、ビジュアルエディターはありません。

リストを操作する


SwiftUIで作業するには、文字列のビューを作成するだけです

struct ListRow: View {
    let number:Int
    
    var body: some View {
        Text.init(verbatim: "Text \(number)")
    }
}

リストに貼り付けます。

  List(Array(0...44)) { number in
                ListRow.init(number: number)
            }

リストの準備ができました。


Composeにはまだリストウィジェットがありません。一見すると、列を使用できます。

階層を表示


SwiftUIでは、UIViewControllerを置き換えるルート要素が、新しいライフサイクルを持つ新しいViewコンポーネントです。これは、UIViewController実装の長所と短所を分析したAppleが、すべてを新しい方法で実行し、最高の機能を実装しようとしたことを示唆しています。

struct ContentView : View {
   var body: some View {
        Text("ContentView2  ")
            .onDisappear(){
                print("onDisappear")
            }
            .onAppear{
                print("onAppear")
            }
    }
}

Composeの場合、ルート要素は、通常のXMLロードではなく、古き良きアクティビティまたはフラグメントのままです。

class MainActivity : Activity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
    }
}

Composeからウィジェットをダウンロードする。

class ComposeActivity : Activity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            CraneWrapper {
                MaterialTheme {
                    Text(text = "Hello world")
                }
            }
        }
    }
}

これは、Androidがレガシールートビューを削除しないという事実につながります。古いUIコンポーネントだけがXMLと通信しなくなり、Composeウィジェットと通信します。

ナビゲーション


SwiftUI内を移動するには、ボタンをクリックして、どのビューを開くかを指定する必要があります。

NavigationButton(destination: LandmarkDetail()) {
                   LandmarkRow(landmark: landmark)
               }

Compose内にはナビゲーションの実装はありません。ただし、ルート要素はActivityまたはFragmentであり、ナビゲーションが長期間実装されているため、おそらくそうではありません。

対応バージョン


SwiftUIは9月にリリースされるiOS 13で動作しますが、iPhone 5S、iPhone6、iPhone 6 Plusではサポートされなくなります。開発者は、これらのモデルを放棄する準備がどのようになるかについて考える必要があります。

Androidのすべてのバージョンでサポートされるように計画を作成します。

カンファレンスの後、UIの2つの新しい強力なフレームワークが登場すると思いました。結局のところ、Appleはクールな決定を下したことが判明しました。その決定の多くは「高揚状態」を持っているということです。そしてグーグルは、コンセプトレベルでさえいくつのポイントが機能するかはっきりしないほど粗雑なフレームワークを発表しました。
