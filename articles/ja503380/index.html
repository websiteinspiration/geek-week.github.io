Celeryを使用したDjangoの非同期ジョブ

この記事の翻訳は、Python Web Developerコースの開始前に準備されました。



アプリケーションに長いプロセスがある場合は、標準のリクエスト/レスポンスストリームではなく、バックグラウンドで処理できます。

たとえば、アプリケーションでは、ユーザーはサムネイル画像（ほとんどの場合、編集する必要があります）を送信し、電子メールアドレスを確認する必要があります。アプリケーションが画像を処理し、リクエストハンドラーで確認のために電子メールを送信する場合、エンドユーザーは何らかの理由で両方のタスクを完了してからページを再読み込みまたは閉じる必要があります。代わりに、これらの操作をタスクキューに転送し、別のプロセスに任せて、ユーザーに応答をすぐに送信することができます。この場合、エンドユーザーはバックグラウンドで処理しながら、クライアント側で他のことを実行できます。この場合、アプリケーションは他のユーザーやクライアントからのリクエストにも自由に応答できます。

今日は、このような問題を解決するために、Djangoアプリケーションで長時間実行されるプロセスを処理するように、CeleryとRedisをセットアップおよび構成するプロセスについて説明します。また、DockerとDocker Composeを使用してすべての要素をリンクし、単体テストと統合テストでCeleryジョブをテストする方法を確認します。

このガイドの終わりまでに、次のことを学びます。

  • セロリをDjangoに統合して、バックグラウンドジョブを作成します。
  • Django、Celery、RedisをDockerにパックします。
  • 別のワークフローを使用してバックグラウンドでプロセスを実行します。
  • Celeryログをファイルに保存します。
  • Celeryの仕事と労働者を監視および管理するためにFlowerを設定します。
  • 単体テストと統合テストでCeleryジョブをテストします。

バックグラウンドタスク


ユーザーエクスペリエンスを向上させるには、通常のHTTP要求/応答ストリームの外で、バックグラウンドで長いプロセスを実行する必要があります。

例えば：

  • 確認のための手紙を送る;
  • Web scapingとクロール。
  • データ分析;
  • 画像処理;
  • レポートの生成。

アプリケーションを作成するときは、リクエスト/レスポンスの有効期間中に実行する必要があるタスク（CRUD操作など）を、バックグラウンドで実行する必要があるタスクから分離するようにしてください。

作業プロセス


私たちの目標は、Celeryを使用して、リクエスト/レスポンスサイクル外の長いプロセスを処理するDjangoアプリケーションを開発することです。

  1. エンドユーザーは、POST要求をサーバーに送信して新しいジョブを生成します。
  2. このビューでは、ジョブがキューに追加され、ジョブIDがクライアントに送り返されます。
  3. AJAXを使用すると、ジョブ自体がバックグラウンドで実行されている間、クライアントは引き続きサーバーにクエリを送信してジョブのステータスを確認します。

次に、ファイルsample_tasks.pyを作成しますproject/tasks。 ここでは、shared_taskデコレーターを使用して、という新しいCeleryタスク関数を定義しました。 タスク自体はDjangoプロセスから実行されるのではなく、Celeryワーカーによって実行されることに注意してください。 次にファイルを追加します：

import time

from celery import shared_task

@shared_task
def create_task(task_type):
time.sleep(int(task_type) * 10)
return True

celery.py"project/core"

何が起きてる？

  1. まず、環境のデフォルト値を設定して、DJANGO_SETTINGS_MODULECeleryがDjangoプロジェクトを見つける方法を認識できるようにする必要があります。
  2. 次に、名前を付けてCeleryのインスタンスを作成し、それをcore変数に配置しましたapp。
  3. 次に、からの設定オブジェクトからCelery構成値をロードしましたdjango.conf。他のDjango設定との競合を防ぐために、namespace = "CELERY"を使用しました。したがって、Celeryのすべての構成設定は、プレフィックスで始まる必要がありますCELERY_。
  4. 最後に、でapp.autodiscover_tasks()定義されたアプリケーションからジョブを探すようにCeleryに指示しますsettings.INSTALLED_APPS。

AJAXリクエストから応答が返されると、getStatus()毎秒ジョブIDから送信されます。

function getStatus(taskID) {
  $.ajax({
    url: `/tasks/${taskID}/`,
    method: 'GET'
  })
  .done((res) => {
    const html = `
      <tr>
        <td>${res.task_id}</td>
        <td>${res.task_status}</td>
        <td>${res.task_result}</td>
      </tr>`
    $('#tasks').prepend(html);

    const taskStatus = res.task_status;

    if (taskStatus === 'SUCCESS' || taskStatus === 'FAILURE') return false;
    setTimeout(function() {
      getStatus(res.task_id);
    }, 1000);
  })
  .fail((err) => {
    console.log(err)
  });
}

答えが「はい」の場合、新しい行がDOMテーブルに追加されます。ビューget_statusを更新してステータスを返します。

@csrf_exempt
def get_status(request, task_id):
    task_result = AsyncResult(task_id)
    result = {
        "task_id": task_id,
        "task_status": task_result.status,
        "task_result": task_result.result
    }
    return JsonResponse(result, status=200)

完全な統合テストはどうですか？

def test_task_status(client):
    response = client.post(reverse("run_task"), {"type": 0})
    content = json.loads(response.content)
    task_id = content["task_id"]
    assert response.status_code == 202
    assert task_id

    response = client.get(reverse("get_status", args=[task_id]))
    content = json.loads(response.content)
    assert content == {"task_id": task_id, "task_status": "PENDING", "task_result": None}
    assert response.status_code == 200

    while content["task_status"] == "PENDING":
        response = client.get(reverse("get_status", args=[task_id]))
        content = json.loads(response.content)
    assert content == {"task_id": task_id, "task_status": "SUCCESS", "task_result": True}

結論


今日は、Django上のアプリケーションで長期的なタスクを実行するためのCeleryの基本構成を紹介しました。ユーザー側のコードを遅くする可能性のあるプロセスを処理キューに送信する必要があります。
