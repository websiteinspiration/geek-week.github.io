Kapelle



Chapel verwendet die forall- Schleife, um Threads zu parallelisieren. Es werden auch C-Zeiger auf jedes Element verwendet, und es wird kein Standardverweis auf Arrays verwendet, und es wird eine geführte Iteration durch Indizes angewendet : 



proc calculateKernelMatrix(K, data: [?D] ?T)
{
  var n = D.dim(0).last;
  var p = D.dim(1).last;
  var E: domain(2) = {D.dim(0), D.dim(0)};
  var mat: [E] T;
  var rowPointers: [1..n] c_ptr(T) =
    forall i in 1..n do c_ptrTo(data[i, 1]);

  forall j in guided(1..n by -1) {
    for i in j..n {
      mat[i, j] = K.kernel(rowPointers[i], rowPointers[j], p);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


Kapellencode war für die Leistung am schwierigsten zu optimieren und erforderte die meisten Codeänderungen.



D.



Um den D-Code zu parallelisieren, werden taskPool- Streams aus dem Paket std.parallel verwendet . D-Code wurde am wenigsten geändert, um die Leistung zu optimieren. Es ist von großem Vorteil, einen bestimmten Compiler und ausgewählte Kompilierungsschlüssel zu verwenden (wird später erläutert). Mit meiner Matrix-Implementierung kann ich Spalten mithilfe von refColumnSelect als Referenz auswählen . 



auto calculateKernelMatrix(alias K, T)(K!(T) kernel, Matrix!(T) data)
{
  long n = data.ncol;
  auto mat = Matrix!(T)(n, n);

  foreach(j; taskPool.parallel(iota(n)))
  {
    auto arrj = data.refColumnSelect(j).array;
    foreach(long i; j..n)
    {
      mat[i, j] = kernel(data.refColumnSelect(i).array, arrj);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


Julia



Julia-Code verwendet ein Makro Fäden zum Parallelisieren von Code und Makros Ansichtenfür Verweise auf Arrays. Das einzige, was mit Arrays in Julia verwirrt, ist ihr Referenzstatus. Manchmal, wie in diesem Fall, verhalten sich Arrays wie Wertobjekte und müssen mithilfe eines Makros referenziert werdenAnsichtenAndernfalls werden Kopien erstellt. In anderen Fällen verhalten sie sich beispielsweise wie Referenzobjekte, wenn sie an eine Funktion übergeben werden. Es kann etwas schwierig sein, damit umzugehen, da Sie nicht immer wissen, welche Vorgänge die Kopie generiert, sondern wo dies geschiehtAnsichtenbietet eine gute Lösung.



Der symmetrische Typ spart ein wenig zusätzliche Arbeit, die zum Reflektieren in der Matrix erforderlich ist. 



function calculateKernelMatrix(Kernel::K, data::Array{T}) where {K <: AbstractKernel,T <: AbstractFloat}
  n = size(data)[2]
  mat = zeros(T, n, n)
  @threads for j in 1:n
      @views for i in j:n
          mat[i,j] = kernel(Kernel, data[:, i], data[:, j])
      end
  end
  return Symmetric(mat, :L)
end


Die Makros @ bounds und @ simd in den Hauptfunktionen wurden verwendet, um die Grenzüberprüfung zu deaktivieren und die SIMD-Optimierung auf Berechnungen anzuwenden: 



struct DotProduct <: AbstractKernel end
@inline function kernel(K::DotProduct, x::AbstractArray{T, N}, y::AbstractArray{T, N}) where {T,N}
  ret = zero(T)
  m = length(x)
  @inbounds @simd for k in 1:m
      ret += x[k] * y[k]
  end
  return ret
end


Diese Optimierungen nehmen deutlich zu, sind aber sehr einfach zu bedienen.

Leistungsoptimierung



Der Prozess der Leistungsoptimierung in allen drei Sprachen war sehr unterschiedlich, und alle drei Communities waren in diesem Prozess sehr nützlich. Aber es gab allgemeine Punkte.



  • Statische Planung von Kernelfunktionen anstelle von Polymorphismus. Dies bedeutet, dass bei der Übertragung der Kernelfunktion der parametrische (Kompilierungszeit-) Polymorphismus und nicht der dynamische (Laufzeit-) Polymorphismus verwendet wird, bei dem die Planung mit virtuellen Funktionen Overhead erfordert.
  • Die Verwendung von Ansichten / Links anstelle des Kopierens von Daten im Multithread-Modus ist von großer Bedeutung.
  • Die Parallelisierung von Berechnungen ist von großer Bedeutung.
  • Zu wissen, dass ein Array für eine Zeile / Spalte grundlegend ist, und dies in Berechnungen zu verwenden, ist von größter Bedeutung.
  • Grenzprüfungen und Compileroptimierungen machen einen großen Unterschied, insbesondere in Chapel und D.
  • Die Aufnahme von SIMD in D und Julia hat zur Leistung beigetragen. In D wurde dies mit dem nativen Flag -mcpu = durchgeführt, und in Julia wurde dies mit dem Makro @ simd durchgeführt .


Aus sprachlicher Sicht war der Übergang zu einem produktiven Code in Chapel am schwierigsten, und der Code in Chapel änderte sich am meisten: von einfach zu lesenden Operationen mit Arrays bis zur Verwendung von Zeigern und kontrollierten Iterationen. Auf der Compilerseite war es jedoch relativ einfach, schnell hinzuzufügen und einen großen Leistungsschub zu erzielen.



Der D-Code hat sich kaum geändert, und der größte Teil der Leistung wurde durch Auswahl des Compilers und seiner Optimierungsflags erzielt. Der LDC-Compiler ist reich an Funktionen zur Leistungsoptimierung. Es hat 8-O-Optimierungsstufen, aber einige davon werden wiederholt. Zum Beispiel sind -O, -O3 und -O5 identisch, und es gibt unzählige andere Flags, die die Leistung beeinflussen. In diesem Fall wurden die Flags -O5 --boundscheck = off -ffast-math verwendet, bei denen es sich um aggressive Compileroptimierungen, Bounds Checking und LLVMs Fast-Math handelt, und -mcpu = native, um Vektorisierungsanweisungen zu ermöglichen.



In Julia verbesserten Makros in den zuvor überprüften Änderungen die Leistung deutlich, waren jedoch nicht zu verwirrend. Ich habe versucht, die Optimierungsstufe -O zu ändern, aber das hat die Leistung nicht verbessert.

"Arrays" beziehen sich hier nicht auf Arrays im Stil von C und C ++, die in D verfügbar sind, sondern auf mathematische Arrays. D hat jedoch eine lineare Lübeck-Algebra-Bibliothek, das beeindruckende Leistungsmerkmale und Schnittstellen zu allen gängigen BLAS-Implementierungen bietet. Arrays Julia natürlich die einfachste und bekannteste. Zungenkraft



Da Julia eine dynamische Programmiersprache ist, mögen einige sagen: "Nun, Julia ist eine dynamische Sprache, die viel freizügiger ist als statische Programmiersprachen, daher ist die Debatte vorbei", aber es ist viel komplizierter. Statische Systeme haben ihre Kraft. Julia hat ein Typsystem, das dem Typsystem aus statischen Sprachen ähnelt. Sie können also Code schreiben, als ob Sie eine statische Sprache verwenden würden, aber Sie können Dinge tun, die nur für dynamische Sprachen reserviert sind. Es verfügt über eine hochentwickelte Syntax für allgemeine und Metaprogrammierung sowie leistungsstarke Makros. Es hat auch ein sehr flexibles Objektsystem und Mehrfachversand. Diese Kombination von Funktionen macht Julia zur mächtigsten Sprache der drei.



D wurde als Ersatz für C ++ konzipiert und hat viel von C ++ übernommen (und auch von Java entlehnt), macht aber die Vorlagenprogrammierung und Berechnungen zur Kompilierungszeit (CTFE) viel benutzerfreundlicher als in C ++. Dies ist eine Sprache mit Einzelversand (obwohl es ein Paket mit mehreren Methoden gibt). Anstelle von Makros verfügt D über ein „Mixin“ für Zeichenfolgen und Muster, die einem ähnlichen Zweck dienen.



Chapel bietet generische Unterstützung und Unterstützung für OOP mit Einzelversand, es gibt keine Makrounterstützung und in diesen Angelegenheiten ist es nicht so ausgereift wie D oder Julia.



Wettbewerbsfähigkeit und gleichzeitige Programmierung



Gegenwärtig stehen neue Sprachen zunächst in Konkurrenz zu ihrer beliebten Teilmenge der Parallelität, die Details variieren jedoch je nach Sprache erheblich. Parallelität ist in diesem Beispiel relevanter und wird von allen drei Sprachen bereitgestellt. Das Schreiben paralleler Schleifen ist in allen drei Sprachen einfach.



Im Chapel-Parallelitätsmodell wird der Datenparallelität viel mehr Aufmerksamkeit geschenkt, es gibt jedoch Tools für die Parallelität von Aufgaben, und es wird eine clusterbasierte Parallelitätsunterstützung bereitgestellt.



Julia hat eine gute Unterstützung für Parallelität und Parallelität.



D hat eine industriell starke Unterstützung für Parallelität und Parallelität, obwohl seine Unterstützung für Multithreading durch Beispiele viel weniger gut dokumentiert ist.

Chapel hat einen Paketmanager, aber kein hoch entwickeltes Ökosystem.



C-Integration



Die C-Interoperabilität ist in allen drei Sprachen einfach zu verwenden. Kapelle hat gute Dokumentation, aber nicht so beliebt wie andere. Die D-Dokumentation ist besser und die Julia-Dokumentation ist die umfassendste. Seltsamerweise verfügt jedoch keine einzige Sprachdokumentation über die erforderlichen Befehle, um Ihren eigenen C-Code zu kompilieren und in die Sprache zu integrieren. Dies ist ein Versehen, insbesondere für Anfänger. In D und Julia ist es jedoch einfach, Beispiele für den Kompilierungsprozess zu suchen und zu finden.



Gemeinschaft



Für alle drei Sprachen gibt es praktische Orte, an denen Benutzer Fragen stellen können. Für Chapel ist Gitter der einfachste Ort, für Julia Discourse (obwohl es auch Julia Gitter gibt) und für D ist dies die offizielle Forumsseite. Die Julia-Community ist am aktivsten, dann D, dann Chapel. Ich habe festgestellt, dass Sie von allen drei Communities gute Antworten erhalten, aber wahrscheinlich erhalten Sie schnellere Antworten für D und Julia.

Zusammenfassung



Wenn Sie ein Anfängerprogrammierer sind, der numerische Algorithmen schreibt und wissenschaftliche Berechnungen durchführt, und eine schnelle und einfach zu verwendende Sprache wünschen, ist Julia die beste Wahl. Wenn Sie ein erfahrener Programmierer sind, der auf demselben Gebiet arbeitet, ist Julia immer noch eine großartige Option. Wenn Sie eine traditionellere, „industriellere“, statisch kompilierte Hochleistungssprache mit allen „Whistleblowern“ möchten, aber etwas Produktiveres, Sichereres und Schmerzfreieres als C ++ möchten, ist D die beste Option. Sie können „alles“ in D schreiben und dank der Compiler eine hervorragende Leistung erzielen. Wenn Sie Arrays für Cluster berechnen müssen, ist Chapel wahrscheinlich die bequemste Lösung.



In Bezug auf die Bruttoleistung bei dieser Aufgabe war D der Gewinner und zeigte deutlich die beste Leistung in 5 von 9 Referenzaufgaben. Die Studie hat gezeigt, dass das Label Julia als Hochleistungssprache mehr als nur ein Hype ist - es hat seine eigenen Vorteile im Vergleich zu wettbewerbsintensiven Sprachen. Es war schwieriger als erwartet, eine wettbewerbsfähige Leistung von Chapel zu erzielen - das Chapel-Team benötigte viel Forschung, um die aktuelle Lösung zu finden. Mit zunehmender Reife der Kapellensprache werden wir jedoch weitere Verbesserungen feststellen können.
