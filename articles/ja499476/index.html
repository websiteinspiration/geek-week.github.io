登録およびSMSなしのターンキーWebアプリケーションの自動化テスト

多くの場合、Webアプリケーションは、さまざまなテキストとコントロールを持つ動的に再構築する多数のフォームで構成されています。このようなアプリケーションのテストは悪夢に変わります。

100500ページをクリックしてすべての機能を確認する必要があります...次のリリースの前に、同じことをもう一度確認してください...そしてまた...明日また。ある時点で、検証は新しい機能の開発よりも長くかかり始めます。 「e2eテストはどうですか？」 - あなたが尋ねる。しかし、最初に、それらはまだ書かれる必要があります。そして次に、それらを書き始める前に、テストケースを書く必要があります。多くのテストケース。

これらの線を読んでいる間に額が汗で覆われていても心配しないでください。この記事では、単一のテストケースとe2eテストを作成せずに、TinkoffがWebアプリケーションの1つのテストをどのように自動化したかについて説明します。




テストケースの自動作成


たまたま、私たちのWebアプリケーションのテストは主にインターフェイスのチェックに関連していることが起こりました。ボタンが画面に表示されていること、目的のタイトルとテキストが表示されていることを確認する必要があります。inputに無効な値を入力すると、エラーメッセージが表示されます。

したがって、テストケースを作成するときは、すべてのアクションを記録する必要があります。

  • 「ボタンを押した」
  • 「XXXの値を入力しました」
  • 「ドロップダウンリストで値YYYを選択しました」

そしてチェック：

  • 「テキストが表示されました：XXX」
  • 「エラーメッセージが表示されました：YYY」
  • 「見出しが表示されました：ZZZ」

Webアプリケーションのすべての機能を分析した後、約30の固有のアクションとチェックを特定しました。このプロセスを自動化できることが明らかになりました。これを行うには、ページ上のテスターのすべてのアクションと、これらのアクションに対するサイトの反応を追跡する必要があります。

イベントをインターセプトすることから始めましょう。ボタン、ラジオボタン、チェックボックスなどのコントロールの操作を追跡するには、クリックイベントをサブスクライブする必要があります。各フレームワークには、このための独自のメソッドがあります。たとえば、AngularではfromEvent、JavaScriptおよびReactではdocument.addEventListenerです。カレンダーや入力などの入力コントロールの場合、サブスクライブするイベントのタイプのみが変更されます。クリックではなく、フォーカスアウトが変更されます。

fromEvent(this.elementRef.nativeElement, 'click')
.subscribe(tagName => {
 	if (tagName === 'BUTTON') {
   		this.testCaseService.addAction(`   "${action.name}"`);
 	} else if (tagName === 'INPUT-CALENDAR') {
   		this.testCaseService
     			.addAction(`  "${action.name}" "${action.value}"`);
 	}
});

そして最後に、最も重要なことはチェックです。テスターのアクションに応じてサイトが動作する方法。

テスターは通常何をチェックしますか？たとえば、彼はinputに無効な値を入力し、サイトはエラーメッセージで応答しました。または、ボタンをクリックして新しい画面が開き、タイトルが変更され、新しいテキストが表示され、コントロールが再構築されるとします。これらの変更はすべて、DOMツリーの変更に関連しています。それらを追跡するための多くのオプションがあります。たとえば、ReactとJavaScriptでMutationObserverを使用したり、AngularでngAfterViewInitを使用したりできます（サイトで対象となるフォーム要素にディレクティブを配置します）。

ngAfterViewInit() {
    const tagName = this.nativeElement.nodeName;
	const text = this.nativeElement.textContent;
    if (['SPAN', 'P'].includes(tagName)) {
 	 	 this.testCaseService.addContent(`** ** "${text}"\n`);
     } else if (tagName === 'H1') {
 	  	this.testCaseService.addContent(`** ** "${text}"\n`);
     } …	
}

ボタンはボタンタグとして表されないという事実にもかかわらず、マークアップを見ると、「input-button」CSSクラスを使用してページ上のすべてのボタンを選択でき、ネストされた「テキスト」CSSクラスからボタン名を取得できます。

仕事の半分は完了しました。追跡したすべてをテストケースに書き留めるだけです。

特定のキーの組み合わせについて、サイト上のすべてのアクションのインターセプトが含まれ、テスト回路のみが含まれます。特定のキーの組み合わせにより、サイト上のすべてのイベントの傍受を停止します。これにより、テストケースの自動記録をどこからでも開始および停止できます。

e2eテストの自動作成


自動生成されたテストケースを見ると、これらは基本的に同じ形式に縮小されたユーザースクリプトです。したがって、それらはe2eテストに変換できます。テストケースをバイパスして、すべてのアクションとチェックをインターセプトした後、すぐにe2eテストを作成することもできます。

現在、Specflow、xBehave.net。、Cucumber.js、CodeceptJSなどの動作スクリプトに基づいたガーキン表記のさまざまなフレームワークが多数あります。

テストケースから機能を取得するには、アクションの前と前にWhen キーワードを追加する必要がありますその後、すべてのチェック。

自動生成されたe2eテストを取得します。

Feature:   2-
  Background:
 	When  "" ""

  Scenario:
 	When    " - "
 	Then    "  "
 	And   " - "
 	When    "  " ""
 	When    "" "  "
 	When    ""

テスト実行を実行するには、生成される機能はほとんどありません。すべてのアクションとチェックのハンドラーを作成する必要があります。

朗報です。各機能のハンドラーを作成する必要はありません。すでに述べたように、サイトにはさまざまなフォームが多数ありますが、固有のアクションとチェックは30しかありません。つまり、すべてのe2eテストの汎用プロセッサには、まったく同じ数のメソッドが含まれます。コードは少し異なります-ガーキン表記とサイト上のレイアウトで選択したフレームワークによって異なります。ただし、ハンドラー自体の作成にはそれほど時間はかかりません。

When('   {string}', async function (button: string) {
	const xpath = "//button";
	const btn = await getItemByText(xpath, button);
	await waitAndClick(btn);
});
When('  {string} {string}', async function (label: string, text: string) {
	const xpath = "//*[contains(text(),'" + label + "')]/ancestor::outline";
	await inputSendKeys(currentBrowser().element(by.xpath(xpath)), text);
});

これで、次のタスクをチェックすると、テスター用のテストケースが自動的に作成され、自動生成されたe2eテストが実行されます。

つまり、次のことを行う必要があります。

  1. コントロールとの相互作用のイベントと、これらのアクションに対するサイトの反応（DOMツリーの再構築の追跡による）をサブスクライブします。
  2. パラグラフ1からe2eテストにデータを変換します。
  3. e2eテストを実行するための一般的なハンドラーを記述します。

このアプローチは、ルーチンから離れるのに役立ちます。インターフェイスに関連する簡単なチェックのために、テストケースとe2eテストの書き込みを自動化できます。さらに進んで、データベースのレコードも自動的にチェックし、サードパーティのサービスに送信しました。

これについて、バージョン管理、技術スタック、実装の最初の段階での問題、そしてモスクワでのHeisenbug-2019カンファレンスでの解決策について話しました。

この記事では、詳細には触れずに主要なアイデアを伝えようとしました。

結論


テストケースとe2eテストを作成するのに平均2分かかります。テストケースとe2eテストを手動で作成したい場合、これは最初の計算より60倍高速です。

チームのプロセスは変更しませんでした。テストケースを作成するためにテストキャパシティを割り当てて自動化チームに引き渡す必要はなくなりました。

私たちは回帰の概念から完全に姿を消しています。以前は、2週間のスプリントで、回帰に3日以上かかりましたが、今ではすべてのe2eテストを実行するために回帰に時間がかかり、これはわずか2時間です。

e2eテストを手動で作成する場合、テストと並行して行うのは非常に困難です。現在、e2eテストはタスクのテスト中に自動的に作成され、テスターは同じ機能を2回チェックする必要がありません。

その結果、チームは構成を変更することなく、はるかに効率的に作業を開始しました。
