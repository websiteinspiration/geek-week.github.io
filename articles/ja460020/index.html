Denoとは何ですか？このプロジェクトはNode.jsとどう違うのですか？

Node.jsの作成者であるRyan Dahlは、昨年1年半をかけてDenoプロジェクトに取り組んできました。これは、Node.jsに固有の問題を修正する新しいJavaScriptランタイムです。

私を誤解しないでください。 Node.jsプラットフォームは、JavaScriptを実行するための優れたサーバー側環境です。その人気は主に巨大なエコシステム、そして実際にはJavaScriptのサポートによるものです。ただし、Ryan Dahl氏は、Node.jsについて何か彼の注意を払う価値があると認めています。これは特に、セキュリティ、モジュール、依存関係の管理についてです。


彼の弁護では、かなり短い期間でNode.jsプラットフォームがどれほど人気になるかを知ることができなかったと言えるでしょう。さらに、2009年には、JavaScriptはまだ限定的で奇妙な言語のように見えました。また、当時は一般的なJavaScript機能の多くが存在しなかったことにも注意してください。

Denoとは何ですか？このプラットフォームの主な機能は何ですか？


Denoは、Googleが開発したV8 JSエンジンの上に構築されたTypeScriptセキュアランタイムです。Denoプラットフォームは、次のツールを使用して作成されました。

  • Rust（DenoコアはRustで記述され、NodeコアはC ++で記述されています）。
  • Tokio（Rustで記述されたイベントループ）。
  • TypeScript（Denoは追加設定なしで、JavaScriptとTypeScriptの両方をサポートします）。
  • V8（Chromeブラウザ、Node.js、その他のプロジェクトで使用されるGoogleのJSエンジン）

Denoプラットフォームが提供する機会について話しましょう。

セキュリティ（権限）


特に注意が必要なデノの最も重要な機能の中で、安全性に注意することができます。

Nodeとは異なり、Denoはデフォルトで、サンドボックスでコードを実行します。つまり、ランタイムは次のエンティティと機能にアクセスできません。

  • ファイルシステム。
  • 通信網。
  • 他のスクリプトの実行。
  • 環境変数。

デノの許可システムのしくみを見てみましょう。次のスクリプトについて考えてみます。

(async () => {
 const encoder = new TextEncoder();
 const data = encoder.encode('Hello world\n');

 await Deno.writeFile('hello.txt', data);
 await Deno.writeFile('hello2.txt', data);
})();

このスクリプトは、テキストファイルのカップルを作成- hello.txtとhello2.txt。テキストはこれらのファイルに配置されますHello world。コードはサンドボックスで実行されます。したがって、ファイルシステムにアクセスできません。

また、ここではNodeのDenoようにモジュールのようなものを参照するのではなく、名前空間を使用していることに注意してくださいfs。名前空間Denoは、開発者に多くの基本的なヘルパー関数を提供します。しかし、ネームスペースを使用すると、ブラウザとのコード互換性が失われます。これについては以下で説明します。

このスクリプトは、次のコマンドで実行できます。

deno run write-hello.ts

その後、次の内容の通知が表示されます。

Deno requests write access to "/Users/user/folder/hello.txt". Grant? [a/y/n/d (a = allow always, y = allow once, n = deny once, d = deny always)]

実際、各呼び出しの間にこれを2回見る可能性があります。もちろん、システムの質問にオプションを選択して回答した場合allow always、2回目はこの通知は届きません。

オプションの1つを選択denyすると、エラーが生成されPermissionDeniedます。エラーを処理するためのコードがないため、スクリプトプロセスが完了します。

スクリプトは次のように実行できます。

deno run --allow-write write-hello.ts

通知は表示されず、両方のファイルが作成されます。ファイルシステムの処理に影響を与える

フラグに加えて、--allow-writeスクリプトの実行時に他のフラグを使用できます。これは--allow-net、--allow-envと--allow-run。それらはそれぞれ、ネットワーク、環境、およびサブプロセスの起動へのスクリプトアクセスを開きます。

モジュール


Denoは、ブラウザーと同様に、モジュールをURLでロードします。最初に、多くの人々は、URLを含むサーバーコードのインポートコマンドに表示される内容に戸惑います。しかし、これは実際には理にかなっています。少し待ってください-あなたはそれを自分で理解するでしょう。

import { assertEquals } from "https://deno.land/std/testing/asserts.ts";

ここで、URLによるパッケージのインポートの特別な点について質問があるかもしれません。この質問への答えは簡単です。URLを使用することにより、nnoのような中央レジストリを使用せずにDenoパッケージを配布できます。 Npmは最近多くの問題を抱えています。

URLを介してコードインポートを整理することで、パッケージ作成者は適切と思われる場所であればどこでもコードをホストできます。これは、栄光のすべてにおける分権化です。もうpackage.jsonとはnode_modules。

アプリケーションを起動すると、Denoはインポートされたすべてのモジュールをロードしてキャッシュします。いったんキャッシュされると、フラグを使用して再ロードするよう明示的に要求しない限り、Denoは再ロードしません--reload。

このモジュール操作システムに関して、いくつかの重要な質問をすることができます。

モジュールコードが配置されているリソースにアクセスできない場合はどうなりますか？


モジュールコードは、一元化されたレジストリに保存されません。このコードをホストする一部のWebリソースは、多くの理由で使用できない場合があります。開発プロセス中、またはさらに悪いことに、プロジェクトを本番環境に導入しているときに、モジュールのホスティングが常に利用可能であることを期待するのは危険です。

すでに述べたように、Denoはロードされたモジュールをキャッシュします。キャッシュはローカルディスクに保存されるため、Denoの作成者は、バージョン管理システム（gitなど）を使用してキャッシュを処理し、プロジェクトリポジトリに含めることをお勧めします。この方法では、コードが格納されているリソースが利用できない場合でも、すべてのプロジェクト開発者がダウンロードしたモジュールのバージョンにアクセスできます。

Denoは、環境変数で指定されたフォルダにキャッシュを保存します$DENO_DIR。この変数を設定しない場合、Denoはキャッシュをキャッシュの標準システムディレクトリに格納します。この変数$DENO_DIRは、ローカルリポジトリ内のフォルダを指すように設定できます。このフォルダは、バージョン管理システムを使用して処理できます。

constantlyモジュールを常にURLでインポートする必要がありますか？


長いURLを定期的に入力すると、すぐに退屈することがあります。幸いなことに、Denoはこの作業を簡単にする2つの方法を提供しています。

最初の方法は、ローカルファイルからインポートされたモジュールを再エクスポートする機能を使用することです。たとえば、次のようになります。

export { test, assertEquals } from "https://deno.land/std/testing/mod.ts";

上記のコマンドが置かれているファイルが呼び出されたとしlocal-test-utils.tsます。これで、関数が再び必要になったtest場合assertEqual、または次のようにインポートできます。

import { test, assertEquals } from './local-test-utils.ts';

その結果、モジュールがURLによって読み込まれたかどうかは問題ではないことがわかりました。

2番目のオプションは、JSONファイルの形式でインポートマップを作成することです。

{
   "imports": {
      "http/": "https://deno.land/std/http/"
   }
}

同様のファイルを使用する場合、インポートコマンドは次のようになります。

import { serve } from "http/server.ts";

このスキームが機能するためには、スクリプトを実行するときにフラグを使用して、プロジェクトでのインポートマップの使用についてシステムに通知する必要があります--importmap。

deno run --importmap=import_map.json hello_server.ts

moduleモジュールのバージョン管理はどのように管理されますか？


パッケージのバージョン管理は彼らの責任です。顧客の選択の観点から見ると、パッケージの目的のバージョンは、URLへの追加に似ています
https://unpkg.com/liltest@0.0.5/dist/liltest.js。

ブラウザの互換性


Denoプラットフォームは、そのコードとブラウザとの互換性を追求しています。技術的な観点から見ると、ESモジュールを使用する場合、ブラウザーでアプリケーションを実行する機能を提供するwebpackなどのアセンブリツールを使用する必要はありません。

ただし、Babelのようなツールは、最新のJSコードをES5標準に準拠するコードに変換します。その結果、このコードは、最新のJavaScript機能をサポートしていない新しいブラウザーでも実行できます。しかし、このアプローチにはマイナスもあります。これは、Webプロジェクトのバンドルが多くの補助コードを取得するという事実により、Webプロジェクトのバンドルが成長するという事実にあります。

実際、私たちはプロジェクトの目標について決定を下します。適切なツールを選択します。

TypeScriptサポート


DenoはTypeScriptの使用を簡素化し、開発者がTSコードを実行するために何かを構成する必要をなくします。しかし、Denoプログラムは問題なくJavaScriptで作成することもできます。

概要


TypeScriptとJavaScriptの新しいランタイム環境であるDenoは、しばらくの間持続可能性を実証している興味深いプロジェクトです。しかし、それが生産の準備ができていると見なすことができる前に、彼にはまだ長い道のりがあります。

Denoで実装された、モジュールを操作するための分散型アプローチは、JavaScriptエコシステムを、現在はnpmであるパッケージの中央リポジトリから解放することを目的としています。

ライアン・ダール氏は、夏の終わりにDeno 1.0をリリースする予定であると語っています。このプロジェクトの将来に興味がある場合は、そのリポジトリに注意してください。
