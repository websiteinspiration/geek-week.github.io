QuakeをiPod Classicに移植する


iPod Classic（ビデオ）でQuakeを起動します。

TL; DR：MP3プレーヤーでQuakeを実行できました。この記事では、これがどのように発生したかについて説明します。

私は去年の夏の一部をお気に入りの2つに費やしました：RockboxとゲームQuake id Softwareです。 Quake を Rockboxに移植することで、これら2つの趣味を組み合わせる機会さえありました！これ以上は望めませんでした！

この投稿は、それがどのように機能したかを物語っています。これは、ほぼ2年間にわたる長い話です。加えて、これは、私の人生の中であまりにも多く書いてしまった完成した技術文書とは対照的に、詳細で無垢でない開発プロセスを文書化しようとする私の最初の試みです。記事には技術的な詳細も含まれますが、最初に、コードの作成につながった思考プロセスについて説明します。

悲しいかな、少なくとも短期的には、RockboxとQuakeに別れを告げる時が来ました。数か月間、私にとって空き時間は非常に少ないリソースになるので、急いで仕事に行く前に、自分の考えを急いで伝えます。

ロックボックス


Rockboxは好奇心の強いオープンソースプロジェクトであり、ハッキングに多くの時間を費やしました。それについての最もよい事はウェブページに書かれています：「Rockboxはデジタル音楽プレーヤーのための無料のファームウェアです。」そのとおりです。SandiskSansaプレーヤー、Apple iPod、およびその他の多くのサポートされているデバイスに付属する工場出荷時のソフトウェアの完全な代替品を作成しました。

私たちはオリジナルのファームウェアの機能を再現するよう努力しているだけでなく、プラグインと呼ばれるダウンロード可能な拡張機能のサポートも実装しています-MP3プレーヤーで実行される小さなプログラム。Rockboxにはすでに多くの素晴らしいゲームとデモがあり、その中で最も印象的なのは、おそらく一人称シューティングゲームのDoomとDuke Nukem 3Dでしょう1。しかし、彼には何かが欠けているように感じました。

地震がステージに登場


地震は完全に三次元の一人称シューティングゲームです。それが何を意味するか見てみましょう。ここでのキーワードは「完全立体」。DoomおよびDuke Nukem 3D（一般に2.5Dと呼ばれます（オプションの高さコンポーネントを含む2Dマップを想像してください））とは異なり、Quakeは完全な3Dで実装されます。各頂点とポリゴンは3D空間に存在します。つまり、古い疑似3Dトリックは機能しなくなり、すべてが完全な3Dで行われます。しかし、気が散っていました。つまり、Quakeは強力なものです。

そしてQuakeは冗談を許しません。私たちの調査によると、Quakeは、約100 MHzの周波数のx86プロセッサとFPU、および約32 MBのRAMを「必要」としています。笑い始める前に、Rockboxのターゲットプラットフォームは、ゲームを作成するときにJohn Carmackが焦点を当てたものと比較できないことを覚えておいてください。上、このようなデバイス）。これを念頭に置いて、徐々に減少しているデジタルオーディオプレーヤーのコレクションを見て、最も強力な生き残りを選びました：216 MHzのARMv5Eプロセッサーと64 MBのRAMを備えたApple iPod Classic / 6G（インデックスEARM DSP拡張の存在を示します-後でこれは私たちにとって重要になります）。深刻な仕様ですが、Quakeを実行するのにかろうじて十分です。

「まあ、それをロードしてください！」と思った。

そして起動しました！美しいQuakeコンソールの背景とメニューに迎えられました。すべて完璧に。しかし、時間をかけてください！私がゲームを始めたとき、何かがおかしい。 「はじめに」レベルは正常に読み込まれるように見えましたが、プレイヤーのスポーン位置は完全にマップの外にありました。奇妙なことだと思いました。私はさまざまなトリックを試し、デバッグを開始しましたsplashfが、それはすべて無駄でした-バグは私には複雑すぎる、またはそのように思えました。

そしてこの状況は数年間続きました。おそらくタイミングについて少し話をする価値があります。 Quakeを起動する最初の試みは2017年9月に行われた後、私はあきらめ、QuakeとRockboxの私のフランケンシュタインは棚に横たわり、2019年7月までほこりを集めていました。退屈とモチベーションの完璧な組み合わせを見つけたので、私は始めたものの完了に進むことにしました。

デバッグを始めました。私の流れの状態は、私が何をしていたかについての詳細をほとんど覚えていないようなものでしたが、作業の流れを再現するように努めます。

Quakeの構造は、Cのエンジンコードと、バイトコードでコンパイルされた言語であるQuakeCのゲームの高レベルロジックの2つの部分に分かれていることがわかりました。他の人のコードをデバッグするのは不合理な恐れがあるので、私は常にQuakeC VMに近づかないようにしました。しかし今、私はそれに突入することを余儀なくされました。バグの原因を探していたときの異常なストリーミングセッションを漠然と思い出します。多くの後grep、私は犯人を見つけました：pr_cmds.c:PF_setorigin。この関数は、何らかの理由で常に同じであった、マップのロード時にプレーヤーの新しい座標を設定する3次元ベクトルを受け取りました(0, 0, 0)。うーん...

私はデータストリームをバックトラックし、それがどこから来たかを見つけました：呼び出しからQ_atof()-文字列から浮動小数点への古典的な変換関数。そして、洞察が私に浮かびました：私はQ_atof()Quakeコードを再定義する一連のラッパー関数を書きました、そして私の実装atof()はおそらく間違っていました。修正はとても簡単でした。私は誤ったものを正しい-Quakeコードの関数に置き換えましたatof。そして出来上がり！「E1M1：The Slipgate Complex」と同様に、3つの廊下が問題なくロードされた有名なエントリーレベル。オーディオ出力は壊れた芝刈り機のように聞こえますが、MP3プレーヤーでQuakeを実行しました！

不思議な方向へ転がる


このプロジェクトは、ついに私が延期していたこと、つまりARM 2アセンブリ言語の学習の言い訳になりました。

問題は、速度に敏感な音の混合サイクルsnd_mix.cでした（芝刈り機の音を覚えていますか？）。

この関数SND_PaintChannelFrom8は、8ビットのモノラルオーディオサンプルの配列を受け取り、それらを16ビットのステレオストリームに混合します。その左右のチャネルは、2つの整数パラメーターに基づいて別々にスケーリングされます。 GCCは飽和演算を最適化するお粗末な仕事をしていたので、私はそれを自分で行うことにしました。結果は私を完全に満足させました。

ここに私が得たもののアセンブラバージョンがあります（Cバージョンは以下に示されています）：

SND_PaintChannelFrom8:
        ;; r0: int true_lvol
        ;; r1: int true_rvol
        ;; r2: char *sfx
        ;; r3: int count

        stmfd sp!, {r4, r5, r6, r7, r8, sl}

        ldr ip, =paintbuffer
        ldr ip, [ip]

        mov r0, r0, asl #16                 ; prescale by 2^16
        mov r1, r1, asl #16

        sub r3, r3, #1                      ; count backwards

        ldrh sl, =0xffff                    ; halfword mask

1:
        ldrsb r4, [r2, r3]                  ; load input sample
        ldr r8, [ip, r3, lsl #2]                ; load output sample pair from paintbuffer
                                ; (left:right in memory -> right:left in register)
        ;; right channel (high half)
        mul r5, r4, r1                      ; scaledright = sfx[i] * (true_rvol << 16) -- bottom half is zero
        qadd r7, r5, r8                     ; right = scaledright + right (in high half of word)
        bic r7, r7, sl                      ; zero bottom half of r7

        ;; left channel (low half)
        mul r5, r4, r0                      ; scaledleft = sfx[i] * (true_rvol << 16)
        mov r8, r8, lsl #16                 ; extract original left channel from paintbuffer
        qadd r8, r5, r8                     ; left = scaledleft + left

        orr r7, r7, r8, lsr #16                 ; combine right:left in r7
        str r7, [ip, r3, lsl #2]                ; write right:left to output buffer
        subs r3, r3, #1                         ; decrement and loop

        bgt 1b                          ; must use bgt instead of bne in case count=1

        ldmfd sp!, {r4, r5, r6, r7, r8, sl}

        bx lr

ここには説明する価値のあるトリッキーなハックがあります。qaddARMプロセッサのDSP命令を使用して低コストの彩度の加算を実装していqaddますが、32ビットのワードでしか機能せず、ゲームは16ビットのサウンドサンプルを使用します。ハックは、最初にサンプルを16ビット左にシフトすることです。サンプルをと組み合わせていqaddます。そして、逆シフトを行います。したがって、1つの命令で、GCCが7つとったことを行います。 （はい、すべて私がMMXのようなパックされた飽和演算を持っているARMv6の、で働いていた場合にハックなしで行うことが可能であろうqadd16が、残念ながら、人生はそれほど単純ではありません。また、ハッキングがクールであることが判明！）

ということにも注意一度に2つのステレオサンプルを読み取ります（単語ldrとstr）さらに2、3サイクル節約します。

以下は参照用のCバージョンです。

void SND_PaintChannelFrom8 (int true_lvol, int true_rvol, signed char *sfx, int count)
{
        int     data;
        int             i;

        // we have 8-bit sound in sfx[], which we want to scale to
        // 16bit and take the volume into account
        for (i=0 ; i<count ; i++)
        {
            // We could use the QADD16 instruction on ARMv6+
            // or just 32-bit QADD with pre-shifted arguments
            data = sfx[i];
            paintbuffer[2*i+0] = CLAMPADD(paintbuffer[2*i+0], data * true_lvol); // need saturation
            paintbuffer[2*i+1] = CLAMPADD(paintbuffer[2*i+1], data * true_rvol);
        }
}

最適化されたCバージョンと比較して、サンプルあたりの命令数が60％減少したと計算しました。ループのほとんどは、qadd算術の飽和とパッキングメモリ操作を使用して保存されました。

「素数」の陰謀


ここで私がプロセスで発見した別の興味深いバグがあります。アセンブリコードリストでは、命令の隣にbgt（分岐「以上bne」の場合）サンプル数が1でプログラムの速度が低下する境界線の場合（分岐「等しい場合」）を使用できないというコメントがあります。これにより、循環転送が発生します。整数がオン0xFFFFFFFFで、非常に長い遅延（最終的には終了します）。

この境界線のケースは、7325サンプル3の長さの特定のサウンドによってトリガーされます。7325の何がそんなに特別なのですか？2のべき乗で除算の残りを見つけてみましょう。

何か気づきましたか？つまり、偶然にも、7325は2の累乗で割ると "素数"になります。これはどういうわけか（私にはわかりません）、1つのサンプルの配列がサウンドミキシングコードに転送され、境界線のケースがトリガーされてハングするという事実につながります。

このバグの原因を特定するために、少なくとも1日は費やしましたが、その原因はすべて1つの誤った指示にあることがわかりました。人生で時々起こりますよね？
