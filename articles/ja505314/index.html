Linuxの時刻同期：NTP、Chrony、およびsystemd-timesyncd


ほとんどの人は時間を追跡します。時間通りに起きて、朝の儀式を行い、仕事に行き、昼食のために休憩し、プロジェクトの期限を守り、誕生日や休日を祝い、飛行機に乗り込みます。

さらに、私たちの何人かは時間に夢中です。私の時計は太陽エネルギーを利用しており、コロラド州フォートコリンズにある国立標準技術研究所（NIST）から、WWVBの長波長無線を介して正確な時刻を受け取ります。時間信号は、同じくフォートコリンズにある原子時計と同期しています。私のFitbitは私の携帯電話と同期します。これはNTPサーバーと同期し、最終的には原子時計と同期します。

デバイスも時間を追跡します


私たちのデバイスとコンピューターが正確な時間を必要とする理由はたくさんあります。たとえば、銀行、株式市場、およびその他の金融企業では、トランザクションを適切な順序で実行する必要があり、これには正確な時系列が重要です。

私たちの電話、タブレット、車、GPSシステム、コンピューターは正確な時刻と日付の設定を必要とします。コンピュータのデスクトップの時計に正しい時刻を表示したい。ローカルカレンダーに適切なタイミングでリマインダーを設定します。正しい時刻は、cronジョブとsystemdジョブが適切な時刻に開始されることも保証します。

日付と時刻もロギングにとって重要であるため、日付と時刻に基づいて特定のログを見つける方が少し簡単です。たとえば、私はかつてDevOpsで働いていて（当時はそう呼ばれていませんでした）、ノースカロライナにメールシステムをセットアップしました。以前は1日に2000万通以上のメールを処理していました。一連のサーバーを介して電子メールを追跡するか、地理的に分散したホスト上のログファイルを使用してイベントの正確なシーケンスを決定することは、コンピューターが時間的に同期している場合、はるかに簡単です。

一度-多くの時間


Linuxホストは、システム時刻とRTC時刻があることを認識する必要があります。 RTC（リアルタイムクロック）は少し奇妙で、ハードウェアクロックの名前としてはあまり正確ではありません。

ハードウェアクロックは、システムのマザーボードのバッテリーを使用してコンピューターの電源がオフになっている場合でも継続的に動作します。 RTCの主な機能は、タイムサーバーへの接続が利用できないときの時間を保存することです。インターネットを介してタイムサーバーに接続することが不可能だった当時、各コンピューターには正確な内部時計が必要でした。オペレーティングシステムは起動時にRTCにアクセスする必要があり、ユーザーはハードウェアBIOS構成インターフェイスを使用してシステム時刻を手動で設定し、それが正しいことを確認する必要がありました。

ハードウェアウォッチはタイムゾーンの概念を理解していません。RTCは時間のみを格納し、タイムゾーンやUTC（協定世界時、GMTまたはグリニッジ標準時とも呼ばれます）からのオフセットは格納しません。このツールを使用してRTCをインストールできます。このツールについては、この記事の後半で説明します。

システム時間は、OSがデスクトップのGUIクロックのdateコマンドの出力で、ログのタイムスタンプで表示する時間です。これは、ファイルが作成、変更、および開かれた時間にも適用されます。rtcのマニュアル

ページには、RTCとシステムクロックの詳細な説明があります。

NTPには何がありますか？


世界中のコンピューターは、NTP（Network Time Protocol）を使用して、NTPサーバーの階層を使用し、インターネットを介して標準の基準クロックと時刻を同期させます。メインのタイムサーバーはレベル1にあり、衛星、ラジオ、または電話回線を介してモデムを介して、レベル0のさまざまな国内タイムサービスに直接接続されています。レベル0のタイムサービスは、原子時計、原子時計が送信する信号に同調するラジオ受信機、またはGPS衛星が送信する高精度の時計信号を使用するGPS受信機です。

参照サーバーの大部分で、数千のパブリックNTPストラタム2サーバーが開いており、誰でも利用できます。NTPサーバーを必要とする多数のホストを持つ多くの組織とユーザー（私を含む）は、独自のタイムサーバーをインストールすることを好むため、1つのローカルホストだけがストラタム2または3を呼び出します。次に、ローカルタイムサーバーを使用するようにネットワーク上の残りのノードを構成します。 。私のホームネットワークの場合、これはレベル3のサーバーです。

さまざまなNTP実装


NTPの初期実装はntpdです。次に、chronydとsystemd-timesyncdの2つの新しいバージョンが彼女に加わりました。 3つすべてがローカルホスト時間をNTPタイムサーバーと同期します。 systemd-timesyncdサービスはchronydほど信頼性が高くありませんが、ほとんどの目的にはこれで十分です。 RTCが同期されていない場合は、ローカルシステムの時刻が少しずれたときに、システム時刻を徐々に調整してNTPサーバーと同期させることができます。 systemd-timesyncサービスはタイムサーバーとして使用できません。

Chronyは、2つのプログラム（chronydデーモンとchronycと呼ばれるコマンドラインインターフェイス）を含むNTP実装です。 Chronyには、多くの場合、単にかけがえのないいくつかの機能があります。

ここでも、NTPは、Chronyまたはsystemd-timesyncdを使用してLinuxホストに実装できるプロトコルです。

RPMパッケージのNTP、Chrony、およびsystemd-timesyncdは、標準のFedoraリポジトリーで入手できます。RPM systemd-udevは、Fedoraにデフォルトでインストールされるカーネルイベントマネージャーですが、必須ではありません。

3つすべてをインストールしてそれらを切り替えることができますが、これにより余分な頭痛が生じます。ですから、しない方がいいです。Fedora、CentOS、およびRHELの現在のリリースは、標準実装としてChronyに移行し、systemd-timesyncdも備えています。私は、Chronyがうまく機能し、NTPサービスよりも優れたインターフェースを提供し、はるかに多くの情報を提供し、制御を向上させると確信しています。

NTPサービスの無効化


ホストですでにNTPサービスが実行されている可能性があります。その場合は、他のものに切り替える前に無効にする必要があります。私はchronydを実行していたので、次のコマンドを使用して停止し、無効にしました。ホストで使用するすべてのNTPデーモンに適切なコマンドを実行します。

[root@testvm1 ~]# systemctl disable chronyd ; systemctl stop chronyd
Removed /etc/systemd/system/multi-user.target.wants/chronyd.service.
[root@testvm1 ~]#


サービスが停止して無効になっていることを確認します。

[root@testvm1 ~]# systemctl status chronyd
● chronyd.service - NTP client/server
     Loaded: loaded (/usr/lib/systemd/system/chronyd.service; disabled; vendor preset: enabled)
     Active: inactive (dead)
       Docs: man:chronyd(8)
             man:chrony.conf(5)
[root@testvm1 ~]#

起動前にステータスを確認する


システムクロックの同期ステータスにより、NTPサービスが実行されているかどうかを確認できます。まだNTPを開始していないため、timesync-statusコマンドは次のことを示唆します。

[root@testvm1 ~]# timedatectl timesync-status
Failed to query server: Could not activate remote peer.

直接ステータス要求は重要な情報を提供します。たとえば、引数やパラメータを指定しないでtimedatectlコマンドを実行すると、デフォルトのステータスサブコマンドが実行されます。

[root@testvm1 ~]# timedatectl status
           Local time: Fri 2020-05-15 08:43:10 EDT  
           Universal time: Fri 2020-05-15 12:43:10 UTC  
                 RTC time: Fri 2020-05-15 08:43:08      
                Time zone: America/New_York (EDT, -0400)
System clock synchronized: no                          
              NTP service: inactive                    
          RTC in local TZ: yes                    

Warning: The system is configured to read the RTC time in the local time zone.
         This mode cannot be fully supported. It will create various problems
         with time zone changes and daylight saving time adjustments. The RTC
         time is never updated, it relies on external facilities to maintain it.
         If at all possible, use RTC in UTC by calling
         'timedatectl set-local-rtc 0'.
[root@testvm1 ~]#

したがって、ホストの現地時間、UTC時間、RTC時間を取得できます。この場合、システム時刻はアメリカ/ニューヨーク（TZ）タイムゾーンに設定され、RTCはローカルタイムゾーンの時刻に設定され、NTPサービスはアクティブではありません。 RTC時刻がシステム時刻から少しずれ始めました。これは、クロックが同期されていないシステムでは正常です。ホスト上のオフセットの量は、最後のシステム同期からの経過時間によって異なります。

また、RTCに現地時間を使用することに関する警告を受け取りました-これは、タイムゾーンと夏時間の設定の変更に適用されます。変更が必要なときにコンピューターの電源がオフになっている場合、RTC時間は変更されません。しかし、24時間稼働するサーバーや他のホストの場合、これはまったく問題になりません。また、NTP時刻同期を提供するサービスは、起動の初期段階でホスト時刻を調整するため、起動が完了すると時刻が再び正しくなります。

タイムゾーン設定


通常、インストール時にタイムゾーンを指定しますが、将来変更する必要はありません。ただし、タイムゾーンを変更する必要がある場合があります。役立ついくつかのツールがあります。 Linuxは、タイムゾーンファイルを使用して、ホストのローカルタイムゾーンを決定します。これらのファイルは、/ usr / share / zoneinfoディレクトリにあります。デフォルトでは、私のタイムゾーンの場合、システムは次のように書き込みます。/ etc / localtime-> ../usr/share/zoneinfo/America/New_York。ただし、タイムゾーンを変更するために、そのような機微を知る必要はありません。

主なことは、あなたの場所と対応するチームのタイムゾーンの正式名称を知ることです。タイムゾーンをロサンゼルスに変更するとします。


[root@testvm2 ~]# timedatectl list-timezones | column
<SNIP>
America/La_Paz                  Europe/Budapest
America/Lima                    Europe/Chisinau
America/Los_Angeles             Europe/Copenhagen
America/Maceio                  Europe/Dublin
America/Managua                 Europe/Gibraltar
America/Manaus                  Europe/Helsinki
<SNIP>

これでタイムゾーンを設定できます。dateコマンドを使用して変更を確認しましたが、timedatectlを使用することもできます。

[root@testvm2 ~]# date
Tue 19 May 2020 04:47:49 PM EDT
[root@testvm2 ~]# timedatectl set-timezone America/Los_Angeles
[root@testvm2 ~]# date
Tue 19 May 2020 01:48:23 PM PDT
[root@testvm2 ~]#

これで、ホストのタイムゾーンを現地時間に再び変更できます。

systemd-timesyncd


systemd timesyncデーモンは、systemdのコンテキストで管理しやすいNTP実装を提供します。FedoraとUbuntuにデフォルトでインストールされます。ただし、デフォルトではUbuntuでのみ起動します。他のディストリビューションについてはわかりません。自分で確認できます。

[root@testvm1 ~]# systemctl status systemd-timesyncd

systemd-timesyncdの設定


systemd-timesyncdの設定ファイルは/etc/systemd/timesyncd.confです。これは、古いNTPおよびchronydサービスよりもオプションが少ない単純なファイルです。これが、Fedora仮想マシン上のこのファイルの内容です（これ以上の変更はありません）。

#  This file is part of systemd.
#
#  systemd is free software; you can redistribute it and/or modify it
#  under the terms of the GNU Lesser General Public License as published by
#  the Free Software Foundation; either version 2.1 of the License, or
#  (at your option) any later version.
#
# Entries in this file show the compile time defaults.
# You can change settings by editing this file.
# Defaults can be restored by simply deleting this file.
#
# See timesyncd.conf(5) for details.

[Time]
#NTP=
#FallbackNTP=0.fedora.pool.ntp.org 1.fedora.pool.ntp.org 2.fedora.pool.ntp.org 3.fedora.pool.ntp.org
#RootDistanceMaxSec=5
#PollIntervalMinSec=32
#PollIntervalMaxSec=2048

コメントとは別に、含まれている唯一のセクションは[Time]です。他のすべての行はコメント化されています。これらはデフォルト値であり、変更する必要はありません（理由がなければ）。NTP =行でNTPタイムサーバーを定義していない場合、FedoraはデフォルトでFedoraバックアップタイムサーバーを使用します。通常、タイムサーバーを追加します。

NTP=myntpserver

timesyncを起動する


次のようにsystemd-timesyncdを起動してアクティブにすることができます。

[root@testvm2 ~]# systemctl enable systemd-timesyncd.service
Created symlink /etc/systemd/system/dbus-org.freedesktop.timesync1.service → /usr/lib/systemd/system/systemd-timesyncd.service.
Created symlink /etc/systemd/system/sysinit.target.wants/systemd-timesyncd.service → /usr/lib/systemd/system/systemd-timesyncd.service.
[root@testvm2 ~]# systemctl start systemd-timesyncd.service
[root@testvm2 ~]#

システム時刻が設定またはリセットされていないことを示すログメッセージに注意してください。Timesyncサービスは、タイムスタンプに基づいてシステム時刻を設定します。タイムスタンプは、timesyncデーモンによって維持され、同期が成功するたびに作成されます。

timedatectlコマンドは、システムクロックからハードウェアクロックの値を取得できません。コマンドラインで入力した値からのみ、日付と時刻を設定できます。hwclockコマンドを使用して、RTCをシステム時刻と同じ値に設定できます。

[root@testvm2 ~]# /sbin/hwclock --systohc --localtime
[root@testvm2 ~]# timedatectl
               Local time: Mon 2020-05-18 13:56:46 EDT  
           Universal time: Mon 2020-05-18 17:56:46 UTC  
                 RTC time: Mon 2020-05-18 13:56:46      
                Time zone: America/New_York (EDT, -0400)
System clock synchronized: yes                          
              NTP service: active                      
          RTC in local TZ: yes

--localtimeオプションは、ハードウェアクロックがUTCではなく現地時間を表示することを示します。

なぜRTCが必要なのですか？


NTPを実装すると、起動時にシステムクロックが設定されます。そしてなぜRTCなのか？これは完全に正しいわけではありません。これは、タイムサーバーへのネットワーク接続がある場合にのみ発生します。ただし、多くのシステムはネットワーク接続に常時アクセスできないため、Linuxがそれらに基づいてシステム時刻を設定するには、ハードウェアクロックが役立ちます。これは、リアルタイムからずれている場合でも、手動で時刻を設定するよりも優れています。

結論


この記事では、日付、時刻、およびタイムゾーンを管理するためのいくつかのツールについて説明します。 systemd-timesyncdツールは、ローカルホストの時刻をNTPサーバーと同期できるNTPクライアントを提供します。ただし、systemd-timesyncdはサーバーサービスを提供しないため、ネットワークにNTPサーバーが必要な場合は、Chronyなど、サーバーとして機能するものを使用する必要があります。

私はネットワーク上のすべてのサービスに対して単一の実装を好むので、Chronyを使用しています。ローカルのNTPサーバーが必要ない場合、またはChronyをサーバーとして使用し、systemd-timesyncdをSNTPクライアントとして使用することに問題がない場合。結局のところ、systemd-timesyncdの機能に満足していれば、クライアントとしてChronyの追加機能を使用する必要はありません。

もう1つ注意：NTPを実装するためにsystemdツールを使用する必要はありません。古いバージョンのntpd、Chrony、または別のNTP実装を使用できます。結局のところ、systemdは多数のサービスで構成されています。それらの多くはオプションであるため、それらをオフにして、代わりに別のものを使用できます。これは巨大な一体型モンスターではありません。systemdやその一部は好きではないかもしれませんが、十分な情報に基づいて決定する必要があります。

systemdでのNTP実装は好きですが、自分のニーズにより適しているため、Chronyを好みます。これはLinuxです。
