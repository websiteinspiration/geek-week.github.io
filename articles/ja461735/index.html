FFmpeg DXVA2ハードウェアデコードプラクティス

こんにちは！この記事は、私のVisual Studioを使い始めるFFmpegの記事の続きです。ここでは、フルHD RTSPストリームのハードウェアデコードについて説明します。Intel ATOM Z8350でもこのタスクに簡単に対応できることを前に述べておきます。

タスク： RTSP h.264 IPカメラからの後続の並列処理（4つのプロセッサコア）のために、ハードウェアでデコードして最大4フレームをRAMに記録します。WinAPI関数を使用して、処理されたフレームを表示します。その結果、RTSPストリームを並列モードでコンピューター処理するための高速システムが得られます。次に、リアルタイムフレームを処理するためのコンピュータビジョンアルゴリズムを接続できます。


前書き


なぜハードウェアデコードが必要なのですか？弱くて安価なプロセッサを使用してリアルタイムビデオをデコードしたい場合、またはプロセッサをできるだけアンロードしたい場合は、ハードウェアデコードに慣れる必要があります。

DirectXビデオアクセラレーション（DXVA）は、ハードウェアアクセラレーションを使用してGPUによるビデオ処理を高速化するためのAPIです。 DXVA 2.0では、ビデオキャプチャやビデオ処理操作など、より多くの操作をGPUにリダイレクトできます。

以前の記事を書いた後、私はかなりの数の質問を受けました：「なぜそれはFFmpegを使用するのですか？」問題から始めましょう。ハードウェアデコードの主な困難は、デコードされたフレームをRAMに書き込むことです。フルHDの場合、これは1920 x 1080 x 3 = 6,220,800バイトです。フレームがNV12フォーマットで保存されているという事実を考慮に入れても、これは1920 x 1080 x 1.5 = 3 110 400バイトの多くです。1秒あたり75 MBを上書きすることは、どのプロセッサにとっても重要なタスクです。この問題を解決するために、インテルはSSE 4コマンドを追加しました。これにより、プロセッサーなしでデータを書き換えることができます。残念ながら、すべてのライブラリにこれが実装されているわけではありません。次のライブラリをテストしました。

  1. Ffmpeg
  2. VLC
  3. Opencv

VLC-ハードウェアデコード（プロセッサ負荷が非常に低い）を介してIPカメラで動作します。プリミティブRTSPストリームプレーヤーはわずか10行のコードで構築できますが、デコードされたフレームをRAMで受信すると、プロセッサ時間がかかりすぎます。

OpenCV -RTSPはFFmpegを使用してストリームを操作するため、仲介なしで機能することが決定されました。FFmpegライブラリを使用します。さらに、デフォルトでインストールされるFFmpegは、ハードウェアデコードなしでOpenCVに組み込まれています。

FFmpeg-私の意見では、結果は良好で、安定して動作します。唯一の欠点は、WindowsのバージョンX86（X64で作業できるように見える）のWEBカメラで動作しないように実装されていることです。

ハードウェアビデオのデコードは簡単


実際、FFmpegライブラリを使用したハードウェアデコードは、ソフトウェアほど複雑ではありません。プロジェクト設定はソフトウェア実装の場合と同じで、ブロック図は変更されていません。

FFmpegでサポートされているハードウェアデコード方法のリストを表示できます。

fprintf(stderr, " %s", av_hwdevice_get_type_name(type));

最初に行う必要があるのは、ビデオをデコードするハードウェアデコーダーをFFmpegに指示することです。私の場合、Windows10 + Intel Atom Z8350はDXVA2のみを残します。

type = av_hwdevice_find_type_by_name("dxva2");

ハードウェアデコーダーとして、CUDA、D3D11VA、QSV、またはVAAPI（Linuxのみ）を選択できます。したがって、このハードウェアソリューションを用意し、FFmpegをサポートしてビルドする必要があります。

ビデオストリームを開きます。

avformat_open_input(&input_ctx, filename, NULL, NULL;

ビデオストリームに関する情報を取得します。

av_find_best_stream(input_ctx, AVMEDIA_TYPE_VIDEO, -1, -1, &decoder, 0);

メモリを割り当てる：


frame = av_frame_alloc(); //      
sw_frame = av_frame_alloc(); //

この関数は、RAM内のデコードされたファイルを上書きします。

av_hwframe_transfer_data(sw_frame, frame, 0);

NV12フォーマットについて少し


それで、sw_frame構造体にフレームを取得しました。受信したフレームはNV12形式で保存されます。このフォーマットは、Microsoftによって発明されました。ピクセル情報を12ビットで保存できます。ここで、8ビットは強度であり、4ビットは色を表します（または、4つの隣接する2x2ピクセルの色がすぐに記述されます）。さらに、sw_frame.data [0]-強度が格納され、sw_frame.data [1]-色が格納されます。NV-12からRGBに変換するには、次の関数を使用できます。

NV12を使用すると、ぼかし、Retinex、グレースケールでの画像の取得（色を破棄するだけ）などの手順の実装を高速化できます。私のタスクでは、NV12形式をRGBに変換しません。これには時間がかかります。

そして、ハードウェアでビデオファイルをデコードしてウィンドウに表示する方法を学びました。私たちはNV12形式で会い、それを使い慣れたRGBに変換する方法を説明しました。

DLLハードウェアデコード


FFmpegは、40ミリ秒後にフレームを発行します（25フレーム/秒）。原則として、フルHDフレームの処理にはかなり時間がかかります。これには、4つのプロセッサコアすべての負荷を最大化するマルチスレッドが必要です。実際には、6つのスレッドを1度だけ開始し、それ以上は削除しません。これにより、作業が大幅に簡略化され、プログラムの信頼性が向上します。操作方法を図に示します。 1


図1 FFmpeg

を使用してマルチスレッドプログラムを構築するスキームデコーダーを* .dll（FFmpegD.DLL）の形式で作成して、プロジェクトに含めました。これにより、プロジェクトのコードを減らすことができます。これにより、コードの理解が深まり、それをアセンブラー（検証済み）までの任意のプログラミング言語に含めることができます。それを使用して、IPカメラからRTSPストリームプレーヤーを書き込みます。

DLLの操作を開始するには、int [13]配列へのポインター、新しいフレーム到着イベントのハンドル、カメラからの新しいデータパケットの処理を開始するハンドル、および文字カメラアドレスの配列を渡す必要があります。

配列構造を表1に示します。



呼び出す前に、フレーム番号1〜4をリセットする必要があります。

DLLはFFmpegを初期化するために必要なすべての手順を実行し、ポインターとフレーム番号を記録します。イベントを設定した後「新着フレーム」。着信フレームを処理し、フレーム番号の代わりに0を書き込むだけで十分です（これは、フレームが処理されて使用されなくなったことを意味します）。
