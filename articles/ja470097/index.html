パーティーでのGraphQLの紹介

GraphQLとRESTは、WebアプリケーションのAPIを作成するために使用される2つのパラダイムです。

RESTは、アプリケーションがデータの要求と送信に使用する一意の識別子（URL）のセットです。

GraphQLは、クライアントアプリケーションが単一のエンドポイントから必要なデータを正確に指定できるようにするクエリ言語です。

これらは関連するテクノロジーであり、主に同じものに使用されます（またはそれらは共存できます）が、非常に異なります。

少し新鮮ですね。それらの違いをより興味深い方法で説明しましょう。これは、GraphQLをよりよく理解するのに役立ち、たぶんそれを楽しませるだけです。

パーティーのメタファーで提示された概念をさらに開示するために、RESTを使用するときにしばしば発生する2つの制限に特に注目します。

1.関連リソースを取得する際のいくつか

の問題モバイルおよびWebアプリケーションのデータ管理では、多くの場合、関連リソースとデータセットが必要です。したがって、REST APIを使用してデータを取得すると、複数のエンドポイントに対する多数のリクエストが発生する可能性があります。たとえば、Postリクエストとそれに関連付けられたAuthorは、異なるエンドポイントへの2つのリクエストで作成できます。

someServer.com/authors/:id
someServer.com/posts/:id

APIを繰り返し呼び出すと、アプリケーションのパフォーマンスに影響します。これは、低帯域幅のデバイス（たとえば、スマートウォッチ、IoT、古いモバイルデバイスなど）にとってさらに重要です。

2.オーバーフェッチとアンダーフェッチ

RESTful APIを使用する場合、オーバーフェッチ/アンダーフェッチは避けられません。上記の例を使用すると、domainName.com / posts /：idエンドポイントは特定のレコードのデータを取得します。各エントリは、id、body、title、publishingDate、authorIdなどの属性で構成されています。 RESTでは、応答が事前定義されており、常に同じデータオブジェクトが返されます。

記録のタイトルとテキストのみが必要な状況では、実際に使用されるよりも多くのデータがネットワーク経由で送信されるため、オーバーフェッチが発生します。

出版物全体が作成者に関する関連データとともに必要な場合、実際に使用されるよりも少ないデータがネットワーク経由で送信されるため、アンダーフェッチが不十分です。データのアンダーフェッチが不十分であると、APIリクエストの数が増加し、その結果、インターネットリソースが過剰に使用されます。

GraphQLを使用したクライアントクエリ


GraphQLは、クライアントアプリケーションに多大な柔軟性を与える独自のアプローチを提供します。GraphQLを使用すると、リクエストがAPIに送信され、1回のリクエストで正確に必要なものが返されます。クエリ結果はクエリ自体と同じ形式で返されるため、応答構造が常に予測可能になります。これらの要因により、アプリケーションはサーバーではなく受信データを制御するため、アプリケーションをより高速に実行し、より安定させることができます。

「結果はクエリと同じ形式で返されます」


/* Query */
{
  myFriends(first: 2) {
    items {
      name
      age
    }
  }
}

/* Response */
{
  "data": {
    "items": [
      { "name": "Steve", "age": 27 },
      { "name": "Kelly", "age": 31 }
    ]
  }
}
