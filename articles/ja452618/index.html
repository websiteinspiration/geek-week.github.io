Badoo Developers on Google I / O 2019

0日目。まえがき


カンファレンスに参加するには、Google I / Oウェブサイトで2月に始まる抽選に当たる必要があります（通常、これはニュースから知られるようになります）。しかし、勝利はチケットを提供しませんが、それを1,150ドルで引き換えることを可能にするだけです。Code Jamなど、他のプログラムを使用すると、大幅な割引または無料でチケットを入手できます。学生と大学の従業員は、もっと安いチケットを375ドルで購入できます。

カンファレンスの前は、IT企業が参加者のためのパーティーを開いていました。私はそれらについて、150人以上のロシア語を話すユーザーを集めたTelegramでのチャットから学びました。通常、これらのチャットには、Telegramの専門のAndroidコミュニティからの招待によってアクセスできます。そのようなパーティーは、非公式の場で他の会議参加者と会う良い機会です。たとえば、主催者のメビウスと、飛行機で旅行する人のためのアプリを作成する開発者のチームに会いました。

会議は「駐車禁止」をスローガンに開催されました。グーグルは、この地域で最も人気のあるホテルへの無料送迎バスを組織し、Lyftタクシーサービス（Uberの米国の競合他社）にプロモーションコードを割り当てました。

Badooの開発者は5人いました。チケットを買う機会を勝ち取ったみんなに行きましょう。レポートは6〜10ストリームで配信され、多くの場合、より興味深いトピックをカバーするために分割されました。

Google折りたたみのヒント


最近、怠惰な人だけが折りたたみ装置について話していません。それらは現在わずかな市場シェアを占めていますが、それらのアプリケーションは何とか機能するはずです。

Googleの担当者は、たとえば、ベストプラクティスに従って画面の反転を正しく処理すれば、すべてがそのまま動作することを保証しました。折りたたみデバイスをサポートするために、タブレットとChrome OSのマルチウィンドウと同じメカニズムが使用されます。既存のandroid：maxAspectRatioに加えて、android：minAspectRatioが表示され、アプリケーションでサポートされるパーティーのアスペクト比に制限が追加されます。 Googleは、Android Q以降のAndroidデバイスの最小画面幅は2インチ（5.08 cm）になると主張しています。

デバイスを曲げたり展開したりするときに、アプリケーションで折りたたみデバイスのサポートを実装しているかどうかを確認するいくつかのこと：

  • アプリケーションは同じ状態を復元する必要があります。
  • スクロール位置を維持する必要があります。
  • キーボードのフォーカスは同じままにする必要があります。

アクティビティのサイズを変更したくない場合は、android：resizeableActivity = falseフラグが常に役立つとは限りません。これは、システムがアクティビティのサイズを変更したり、互換モードにしたりできるためです。



互換モード

折りたたみデバイスを使用しているユーザーのようですごくわずかですが、そのようなガジェットのコストは非常に高く、アプリケーションがどのように動作するかを確認する必要があります。

マルチモジュール性の長所と短所について


会議では、モジュール性に多くの注意が払われました。マルチモジュール性の主な利点：

  • テストは、現在のブランチの変更によって何らかの影響を受けるモジュールに対してのみ実行できます。
  • アプリケーションのさまざまな機能のテストを分離できます。たとえば、BadooにはすべてのUI要素を収集するギャラリーアプリケーションがあります。それらを開発する場合、依存関係の数が限られているため、このアプリケーションを十分に迅速に構築できます（同僚がMBLT DEVに関するレポートでこれについて話しました）。
  • 動的機能を追加する機能：スピーカーによると、ユーザーの80％がアプリケーションの機能の20％を使用するため、ほとんどの機能を動的モジュールに移動して後でダウンロードできます。たとえば、エキスパートユーザー向けの機能、お支払いが必要な機能、[アプリについて]画面などが適しています。同時に、オンボーディングはアプリケーションのすべてのユーザーに表示されるため、動的機能として実行しないでください。

また、マルチモジュール性は、多数の開発者に適しています。これは、大規模で急成長しているチームにとって大きな利点です。
マルチモジュール性にも問題があります。たとえば、データベースの作成方法は明確ではありません。3つのアプローチがあります。

  • アプリケーション全体に対して1つのデータベースを作成します。
  • 各モジュールのデータベースを作成します。
  • 論理的に接続できる複数のモジュールに対して1つのデータベースを作成します。

すべてのアプローチには欠点がありますが、GoogleのスタッフはRoomで明るい未来を約束します。Roomは、モジュールの分離を維持し、モジュール間のクエリをサポートし、動的な機能を持つモジュールをサポートしたいと考えています。残念ながら、これはまだ利用できません。

Badooアプリケーションには170以上のモジュールがあり、動的機能はまだ使用していませんが、マルチモジュールの他の利点と欠点があります。

新しい宣言型UI


Androidはすでに10年前のものであり、現在のUIは道徳的に時代遅れです。古いUIは維持が困難です。たとえば、Viewクラスにはコメントを含む29,188行のコードがあり、AppCompatバージョンはAndroidのさまざまなバージョンの多くのハックで大きくなりすぎています。この写真を見て、Googleの開発者たちは、アプリケーションに付属し、Androidから完全に解放されるUIフレームワークを作成することを決定しました。フレームワークの実際の名前はJetpack Composeです。

Flutter、React、Litho、Vue.jsは開発者のインスピレーションとして機能したため、新しいフレームワークは多くの人に馴染みがあるようです。主なアイデアは、UIにロジックがない場合でも、UIがモデルの変更に確実に応答するようにすることです。

ビュー階層は、@ Composableアノテーションが付けられた関数として表されます。フレームワークは、コンパイラプラグインを使用して、構成可能な関数への呼び出しをインターセプトします。
Googleチームは、（@ GenerateViewアノテーションを使用して）古いビュー階層で新しいフレームワークをサポートし、Android Studioで直接プレビューしてアニメーションをサポートすることを約束します。
Jetpack Composeはまだ非常に未加工であり、実際のアプリケーションで使用する準備ができていませんが、Androidの開発がどこに向かっているのかを理解するために、その作業の原理を研究する価値があります。

アニメーションの原則


彼のレポートで、Nik Butcherは、ユーザーエクスペリエンスを向上させるために反応性の時代にアニメーションを実装する方法について話しました。問題は、リアクティブアプリケーションではViewオブジェクトに状態がなく、逆にアニメーションに状態があることです。

優れたアニメーションは、次の3つの基準を満たしている必要があります。

  • 再起動可能（再入可能）：アニメーションをキャンセルして再開できます。
  • 連続性：アニメーションはある状態から別の状態にジャンプするべきではありません。
  • 滑らかさ：アニメーションは、速度/動きの方向を徐々に変化させます。

これを達成する方法：
  • アニメーションを開始するときは、最終値（終了する場所）のみを設定します。指定された初期値で、アニメーションはある状態から別の状態にジャンプする可能性があります。これは、ボタンを押して開始し、ユーザーが数回クリックすると発生する可能性があります。
  • 新しいアニメーションを開始する前に古いアニメーションをキャンセルします（Android SDK内にすでに実装されている場合もあります。たとえば、View＃animateから取得したViewPropertyAnimatorは、アニメーションプロパティの以前のアニメーションをキャンセルします）。
  • Spring Animationを使用します。このようなアニメーションは物理法則を考慮して移動します。つまり、滑らかさと連続性がより簡単に達成されます。つまり、オブジェクトがポイントAからポイントBに移動し、コマンドが到着してポイントCに移動すると、スプリングアニメーションを使用する場合、オブジェクトはスムーズに移動方向を変更します。
  • <animated-selector>を使用して、Drawableにアニメーションを追加します。考えられるすべての状態間の遷移を回避するために、中間状態（たとえば、ブート状態）に入り、それを通過することができます。

ただし、 100回聞くよりも1回読む方が良いので、レポート付きの動画をご覧ください。

性能試験


アプリケーションのパフォーマンスを測定するためのライブラリは、Jetpackの一部としてアルファ状態になりました。コードパフォーマンスの測定を行うことができ、多くの測定エラーを回避できます。AndroidStudioとの統合もあります。Jetpackベンチマークライブラリを使用

してパフォーマンステストを作成および実行する際の注意事項：

  • エミュレータで測定値を収集することは信頼できません。起動時の警告で親切に通知されます。
  • パフォーマンスを正しく測定するには、ProGuard / R8をオンにする必要があります。
  • 低レベルの充電が測定結果に影響を与えないように、デバイスには十分なレベルのバッテリー充電が必要です。
  • パフォーマンステストが記述されているモジュールには、パラメーター「debuggable = false」が必要です。
