C ++での強い型付けの利点：実地経験

私たちのプログラムはネットワークパケット、特にTCP / IP /などのヘッダーを処理します。それらでは、数値-オフセット、カウンター、アドレス-がネットワークバイト順（ビッグエンディアン）で表示されます。x86（リトルエンディアン）に取り組んでいます。ヘッダーを記述する標準的な構造では、これらのフィールドは単純な整数型（uint32_t、uint16_t）で表されます。バイトオーダーを変換するのを忘れたためにいくつかのバグが発生した後、フィールドタイプを暗黙的な変換と非定型の操作を禁止するクラスに置き換えることにしました。カットの下には、実用的なコードと、厳密なタイピングによって明らかになったエラーの具体例があります。

バイトオーダー


バイトオーダー（エンディアン、バイトオーダー）を知らない人のためのLikbez。詳細はすでに「Habré」にありました。

通常の数の表記では、左側の最も古い（左）から最も若い（右）へと進みます：432 10 = 4×10 2 + 3×10 1 + 2×10 0。整数データ型は、たとえば16ビット（0〜65535の数値）などの固定サイズです。それらは、2バイトとして、例えば432 10 = 01b0 16、つまりバイト01とb0 としてメモリーに保管されます。

この数値のバイトを出力します。

#include <cstdio>   // printf()
#include <cstdint>  // uint8_t, uint16_t

int main() {
    uint16_t value = 0x01b0;
    printf("%04x\n", value);

    const auto bytes = reinterpret_cast<const uint8_t*>(&value);
    for (auto i = 0; i < sizeof(value); i++) {
        printf("%02x ", bytes[i]);
    }
}

従来のIntelまたはAMD（x86）プロセッサーでは、次のようになります。

01b0
b0 01

メモリ内のバイトは、数字を書き込むときとは異なり、最も新しいものから最も古いものへと配置されます。この順序はリトルエンディアン（LE）と呼ばれます。 4バイトの数値についても同様です。バイト順は、プロセッサアーキテクチャによって決まります。プロセッサの「ネイティブ」の順序は、CPUまたはホストの順序（CPU /ホストのバイト順）とも呼ばれます。この場合、ホストのバイト順はリトルエンディアンです。

ただし、インターネットはx86で生まれたわけではなく、バイトの順序は最も古いものから最も新しいもの（ビッグエンディアン、BE）までさまざまでした。彼らはネットワークプロトコル（IP、TCP、UDP）のヘッダーでそれを使用し始めたので、ビッグエンディアンはネットワークバイトオーダーとも呼ばれます。

例：ポート443（1bb 16）、HTTPSが通過するTCPヘッダーにはバイトbb 01が書き込まれ、読み取られるとbb01 16 = 47873になります。

数値のバイトオーダーを変換できます。例えば、のためのuint16_t標準的な機能があるhtons()（H OST に Nためetwork S及びその逆-ホスト順からネットワーク短い整数ためにホルト整数）ntohs()。uint32_tそこと同様にhtonl()、そしてntohl()（long-long integer）。

// :  BE    BE .
if (tcp->dst_port == htons(443)) { ... }

//   BE     LE,   1,
//   LE    BE.
tcp->sent_seq = htonl(ntohl(tcp->sent_seq) + 1);

残念ながら、コンパイラーは型変数の特定の値がどこから来たかを認識してuint32_tおらず、異なるバイト順序で値を混合して誤った結果を得ても警告しません。

ネットワークで作業する場合、バイトオーダーは無視できないので、コードを書くときに無視できないようにしたいと思います。さらに、BEには番号だけではなく、ポート番号、IPアドレス、TCPシーケンス番号、チェックサムです。ビット数が一致していても、一方を他方に割り当てることはできません。

解決策は既知です-厳密な型指定、つまり、ポート、アドレス、番号の個別の型。さらに、これらのタイプはBE / LE変換をサポートする必要があります。プロジェクトにBoostがないため、Boost.Endianは適していません。

C ++ 17では、プロジェクトのサイズは約4万行です。安全なラッパータイプを作成し、それらのヘッダー構造を書き直すと、BEで機能するすべての場所でコンパイルが自動的に停止します。これらすべてを一度実行する必要がありますが、新しいコードは安全です。

ビッグエンディアンのクラス番号
#include <cstdint>
#include <iosfwd>

#define PACKED __attribute__((packed))

constexpr auto bswap(uint16_t value) noexcept {
    return __builtin_bswap16(value);
}

constexpr auto bswap(uint32_t value) noexcept {
    return __builtin_bswap32(value);
}

template<typename T>
struct Raw {
    T value;
};

template<typename T>
Raw(T) -> Raw<T>;

template<typename T>
struct BigEndian {
    using Underlying = T;
    using Native = T;

    constexpr BigEndian() noexcept = default;
    constexpr explicit BigEndian(Native value) noexcept : _value{bswap(value)} {}
    constexpr BigEndian(Raw<Underlying> raw) noexcept : _value{raw.value} {}

    constexpr Underlying raw() const { return _value; }
    constexpr Native native() const { return bswap(_value); }

    explicit operator bool() const {
        return static_cast<bool>(_value);
    }

    bool operator==(const BigEndian& other) const {
        return raw() == other.raw();
    }

    bool operator!=(const BigEndian& other) const {
        return raw() != other.raw();
    }

    friend std::ostream&
    operator<<(std::ostream& out, const BigEndian& value) {
        return out << value.native();
    }

private:
    Underlying _value{};
} PACKED;


  • このタイプのヘッダーファイルはどこにでも含まれるため、重いファイルの代わりに<iostream>軽量ファイルが使用され<iosfwd>ます。
  • 代わりにhtons()、など-高速コンパイラ組み込み関数。特に、一定の伝播がそれらに作用するため、設計者constexpr。
  • BEにはuint16_t/の値がすでに存在する場合がありuint32_tます。Raw<T>控除ガイド付きの構造により、そこから簡単に作成できますBigEndian<T>。

ここで論争の的になっている点はPACKED、パッケージ化された構造はあまり最適化できないと考えられていることです。唯一の答えは測定することです。コードのベンチマークでは、速度の低下は明らかになりませんでした。さらに、ネットワークパケットの場合、ヘッダー内のフィールドの位置は固定されたままです。

ほとんどの場合、BEは比較以外の操作を必要としません。シーケンス番号はLEで正しく折りたたむ必要があります。

using BE16 = BigEndian<uint16_t>;
using BE32 = BigEndian<uint32_t>;

struct Seqnum : BE32 {
    using BE32::BE32;

    template<typename Integral>
    Seqnum operator+(Integral increment) const {
        static_assert(std::is_integral_v<Integral>);
        return Seqnum{static_cast<uint32_t>(native() + increment)};
    }
} PACKED;

struct IP : BE32 {
    using BE32::BE32;
} PACKED;

struct L4Port : BE16 {
    using BE16::BE16;
} PACKED;

セキュアTCPヘッダー構造
enum TCPFlag : uint8_t {
    TH_FIN = 0x01,
    TH_SYN = 0x02,
    TH_RST = 0x04,
    TH_PUSH = 0x08,
    TH_ACK = 0x10,
    TH_URG = 0x20,
    TH_ECE = 0x40,
    TH_CWR = 0x80,
};

using TCPFlags = std::underlying_type_t<TCPFlag>;

struct TCPHeader {
    L4Port   th_sport;
    L4Port   th_dport;
    Seqnum   th_seq;
    Seqnum   th_ack;
    uint32_t th_flags2 : 4;
    uint32_t th_off    : 4;
    TCPFlags th_flags;
    BE16     th_win;
    uint16_t th_sum;
    BE16     th_urp;

    uint16_t header_length() const {
        return th_off << 2;
    }

    void set_header_length(uint16_t len) {
        th_off = len >> 2;
    }

    uint8_t* payload() {
        return reinterpret_cast<uint8_t*>(this) + header_length();
    }

    const uint8_t* payload() const {
        return reinterpret_cast<const uint8_t*>(this) + header_length();
    }
};

static_assert(sizeof(TCPHeader) == 20);

  • TCPFlag可能ですenum classが、実際にはフラグに対して2つの操作のみが実行されます。エントリのチェック（&）またはフラグの組み合わせ（|）への置き換え-混乱はありません。
  • ビットフィールドはプリミティブのままですが、安全なアクセスメソッドが作成されます。
  • フィールド名はクラシックのままです。
