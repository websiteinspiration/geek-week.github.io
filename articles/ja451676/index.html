最近のパフォーマンスチューニングとデータベースのトラブルシューティング

残念ながら、現在、パフォーマンスチューニングとデータベースのトラブルシューティングにおけるエキスパートの役割は、最後までしか削減されていません。トラブルシューティング：ほとんどの場合、問題がクリティカルポイントに達し、「昨日」に解決する必要がある場合にのみ、スペシャリストに依頼します。そしてそれでも、彼らが回って、詳細なパフォーマンス監査とストレステストなしでさらに高価で強力なハードウェアを購入することによって問題を遅らせないのは良いことです。実際、欲求不満で十分なことがよくあります。彼らは2〜5倍の高価な機器を購入し、パフォーマンスは30〜40％しか獲得しませんでした。全体の増加は、ユーザー数の増加またはデータの急激な増加と、ロジックの複雑化の両方によって数か月で消費されます。

そして今、アーキテクト、テスター、DevOpsエンジニアの数が急速に増加し、Java Core開発者が文字列の処理さえも最適化している今、データベースオプティマイザーにとってはゆっくりとではありますが確かな時が来ています。各リリースのDBMSは非常にスマートで複雑になるため、文書化された文書化されていない文書化されたニュアンスと最適化の両方を調査するには、膨大な時間が必要です。膨大な数の記事が毎月発行され、Oracleを専門とする主要な会議が開催されます。平凡なアナロジーで申し訳ありませんが、この状況で、データベース管理者が無数のトグルスイッチ、ボタン、ライト、画面を備えた飛行機のパイロットに似たときに、微妙なパフォーマンスの最適化でそれらをロードするのは不適切です。

もちろん、DBAはパイロットと同様に、さまざまな「トップ」（トップイベント、トップSQL、トップセグメントなど）で簡単に診断または認識できる場合、ほとんどの場合、明白で単純な問題を簡単に解決できます。また、ソリューションを知らなくても、MOSやGoogleで簡単に見つけることができます。症状でさえシステムの複雑さの背後に隠れており、Oracle DBMS自体によって収集された膨大な量の診断情報の中から取り除く必要がある場合は、さらに複雑になります。

最も単純で最も印象的な例の1つは、フィルターとアクセス予測の分析です。大規模で負荷の高いシステムでは、そのような問題が見落とされがちです。負荷はさまざまなリクエストにかなり均等に分散され（さまざまなテーブルの結合、条件のわずかな違いなど）、「これらのテーブルからのデータが最も頻繁に必要であり、それ以上のものがある」と言って、上位のセグメントには特別なものは何も表示されません。 。そのような場合、SYS.COL_USAGE $からの統計を使用して分析を開始できます：col_usage.sql

col owner format a30
col oname format a30 heading "Object name"
col cname format a30 heading "Column name"
accept owner_mask prompt "Enter owner mask: ";
accept tab_name prompt "Enter tab_name mask: ";
accept col_name prompt "Enter col_name mask: ";

SELECT a.username              as owner
      ,o.name                  as oname
      ,c.name                  as cname
      ,u.equality_preds        as equality_preds
      ,u.equijoin_preds        as equijoin_preds
      ,u.nonequijoin_preds     as nonequijoin_preds
      ,u.range_preds           as range_preds
      ,u.like_preds            as like_preds
      ,u.null_preds            as null_preds
      ,to_char(u.timestamp, 'yyyy-mm-dd hh24:mi:ss') when
FROM   
       sys.col_usage$ u
     , sys.obj$       o
     , sys.col$       c
     , all_users      a
WHERE  a.user_id = o.owner#
AND    u.obj#    = o.obj#
AND    u.obj#    = c.obj#
AND    u.intcol# = c.col#
AND    a.username like upper('&owner_mask')
AND    o.name     like upper('&tab_name')
AND    c.name     like upper('&col_name')
ORDER  BY a.username, o.name, c.name
;
col owner clear;
col oname clear;
col cname clear;
undef tab_name col_name owner_mask;

ただし、この情報を完全に分析するには十分ではありません。述語の組み合わせは表示されません。この場合、v $ active_session_historyとv $ sql_planの分析が役立ちます。

with 
 ash as (
   select 
      sql_id
     ,plan_hash_value
     ,table_name
     ,alias
     ,ACCESS_PREDICATES
     ,FILTER_PREDICATES
     ,count(*) cnt
   from (
      select 
         h.sql_id
        ,h.SQL_PLAN_HASH_VALUE plan_hash_value
        ,decode(p.OPERATION
                 ,'TABLE ACCESS',p.OBJECT_OWNER||'.'||p.OBJECT_NAME
                 ,(select i.TABLE_OWNER||'.'||i.TABLE_NAME from dba_indexes i where i.OWNER=p.OBJECT_OWNER and i.index_name=p.OBJECT_NAME)
               ) table_name
        ,OBJECT_ALIAS ALIAS
        ,p.ACCESS_PREDICATES
        ,p.FILTER_PREDICATES
      -- ,         :
      --  ,h.sql_plan_operation
      --  ,h.sql_plan_options
      --  ,decode(h.session_state,'ON CPU','ON CPU',h.event) event
      --  ,h.current_obj#
      from v$active_session_history h
          ,v$sql_plan p
      where h.sql_opname='SELECT'
        and h.IN_SQL_EXECUTION='Y'
        and h.sql_plan_operation in ('INDEX','TABLE ACCESS')
        and p.SQL_ID = h.sql_id
        and p.CHILD_NUMBER = h.SQL_CHILD_NUMBER
        and p.ID = h.SQL_PLAN_LINE_ID
        --     3 :
        -- and h.sample_time >= systimestamp - interval '3' hour
   )
   --       :
   -- where table_name='&OWNER.&TABNAME'
   group by 
      sql_id
     ,plan_hash_value
     ,table_name
     ,alias
     ,ACCESS_PREDICATES
     ,FILTER_PREDICATES
)
,agg_by_alias as (
   select
      table_name
     ,regexp_substr(ALIAS,'^[^@]+') ALIAS
     ,listagg(ACCESS_PREDICATES,' ') within group(order by ACCESS_PREDICATES) ACCESS_PREDICATES
     ,listagg(FILTER_PREDICATES,' ') within group(order by FILTER_PREDICATES) FILTER_PREDICATES
     ,sum(cnt) cnt
   from ash
   group by 
      sql_id
     ,plan_hash_value
     ,table_name
     ,alias
)
,agg as (
   select 
       table_name
      ,'ALIAS' alias
      ,replace(access_predicates,'"'||alias||'".','"ALIAS".') access_predicates
      ,replace(filter_predicates,'"'||alias||'".','"ALIAS".') filter_predicates
      ,sum(cnt) cnt
   from agg_by_alias 
   group by 
       table_name
      ,replace(access_predicates,'"'||alias||'".','"ALIAS".') 
      ,replace(filter_predicates,'"'||alias||'".','"ALIAS".') 
)
,cols as (
   select 
       table_name
      ,cols
      ,access_predicates
      ,filter_predicates
      ,sum(cnt)over(partition by table_name,cols) total_by_cols
      ,cnt
   from agg
       ,xmltable(
          'string-join(for $c in /ROWSET/ROW/COL order by $c return $c,",")'
          passing 
             xmltype(
                cursor(
                   (select distinct
                       nvl(
                       regexp_substr(
                          access_predicates||' '||filter_predicates
                         ,'("'||alias||'"\.|[^.]|^)"([A-Z0-9#_$]+)"([^.]|$)'
                         ,1
                         ,level
                         ,'i',2
                       ),' ')
                       col
                    from dual
                    connect by 
                       level<=regexp_count(
                                 access_predicates||' '||filter_predicates
                                ,'("'||alias||'"\.|[^.]|^)"([A-Z0-9#_$]+)"([^.]|$)'
                              )
                   )
               ))
          columns cols varchar2(400) path '.'
       )(+)
   order by total_by_cols desc, table_name, cnt desc
)
select 
   table_name
  ,cols
  ,sum(cnt)over(partition by table_name,cols) total_by_cols
  ,access_predicates
  ,filter_predicates
  ,cnt
from cols
where rownum<=50
order by total_by_cols desc, table_name, cnt desc;

クエリ自体からわかるように、過去3時間にASHがヒットした回数によって上位50の検索列と述語自体が表示されます。ASHは毎秒スナップショットのみを保存するという事実にもかかわらず、読み込まれたベースでのサンプリングは非常に代表的です。それはいくつかのポイントを明確にすることができます：

  • colsフィールド-検索列自体を表示し、total_by_cols-これらの列のコンテキスト内のエントリの合計。
  • この情報自体は問題の十分なマーカーではないことは明らかです。たとえば、いくつかのワンタイムフルスキャンは簡単に統計を台無しにする可能性があるため、クエリ自体とその頻度を考慮する必要があります（v $ sqlstats、dba_hist_sqlstat）
  • OBJECT_ALIAS SQL_ID,plan_hash_value , .. , :


必要に応じて、このスクリプトを簡単に変更して、パーティション分割や期待値などを考慮して、他のセクションの追加情報を収集できます。また、この情報を分析し、テーブル統計とそのインデックス、一般的なデータスキーム、ビジネスロジックの分析と組み合わせて、非正規化オプションやパーティションスキームやインデックスの変更などのソリューションを選択するための推奨事項を開発者またはアーキテクトに渡すことができます。

SQL *ネットトラフィックを分析することを忘れることもよくあります。たとえば、フェッチサイズ、SQLNET.COMPRESSION、ラウンドトリップの数を減らすことができる拡張データ型など、多くの微妙な点もありますが、これは既に別の記事のトピックです。
