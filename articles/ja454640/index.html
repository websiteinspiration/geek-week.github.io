VPNでのSSHによる4ターンのリモートマイクロサービスデバッグ

リモートマシンの複雑なケースを修正する必要がある状況がありますが、サーバーへのアクセスは、開いているポートに制限があるVPNの下でのみです。記事では、VPNでのSSHを介したリモートデバッグのトピックについて、セットアップにかかる貴重な時間を節約するために、同僚と少し話し合いたいと思います。この人が誰かに役立つならば、私は任務が達成されたと思います。

ネタバレ
Ubuntu, Ubuntu. — Mac, SSH- IDE xdebug, .

はじめに


したがって、保護された回線の背後にリモートサーバーがあります。Dockerのサーバーで、xdebugがオンになっているマイクロサービス。SSHとVPNを介してのみ外部からアクセスできます。

目的


目的：xdebugを介してローカルでマイクロサービスのリモートデバッグを開始します。

行く…

1.リモートサーバーでsshdを構成する


最初に注意する必要があるのは、サーバーでsshdを構成して、127.0.0.1だけでなく、任意のIPからの接続を受け入れることができるようにする必要があることです。デフォルトでは、このオプションは無効になっています。

ここでrootが必要です。私たちはあなたがそれを持っていると仮定します:-)

sudo echo "GatewayPorts yes" >> /etc/ssh/sshd_config
sudo service ssh restart

2. Dockerネットワークでホストマシンのアドレスを確認します


以来xdebugのドッキングウィンドウで実行され、ローカルマシンに自分自身を接続する（これはVPN接続であるため、そして私の場合には、このIPは解決しません）、ドッキングウィンドウネットワーク上のホストマシンのIPを見つけるために便利です。これは次のコマンドを使用して実行できます（サーバーで実行）。

ip -4 addr show docker0 | grep -Po 'inet \K[\d.]+'

チームが「172.17.0.1」を発行したとします

3.コンテナのxdebug設定にリモートホストのIPを登録します


sedによる置換の例ですが、エディターでペンを使用することもできます。

sed -i 's/xdebug.remote_host=.*/xdebug.remote_host=172.17.0.1/' /usr/local/etc/php/conf.d/xdebug.ini

レジから離れることなく、デバッグ用の「正しい」ポートを規定します。私の場合、マイクロサービスはnginxとphp-fpmの束で発生し、通常、ポート9000はphp-fpmの下でビジーであるため、デバッグにはポート9001を使用します。

sed -i 's/xdebug.remote_port=.*/xdebug.remote_port=9001/' /usr/local/etc/php/conf.d/xdebug.ini

リモートデバッグが原則的に有効になっていることをここで確認することは価値があります：「xdebug.remote_enable = 1」

grep xdebug.remote_enable /usr/local/etc/php/conf.d/xdebug.ini

通常、これらの設定はDockerfileに配線されるか、ボリュームを介してマウントされます。

サーバー上ですべての準備が整います。これで、トンネルを構築するためにローカルマシンに移動しました。

4. SSHトンネルをリモートサーバーに転送する


トンネルは次のコマンド（ローカルマシンで実行）によって発生します。

SSH -R 9001:0.0.0.0:9001 user@remote_server

この場所で魔法の音楽が鳴ります。基本的なセットアップが行われ、次にデバッグ環境のみが行われます。

IDEセットアップ


通常はvscodeを使用するため、ポートリスニングを介したデバッガーは問題なく起動します。vscodeの設定の例を示します（launch.jsonにノードを追加するだけです）：

        {
            "name": "Listen for XDebug",
            "type": "php",
            "request": "launch",
            "port": 9001,
            "cwd": "${workspaceFolder}",
            "pathMappings": {
                "/repo": "${workspaceRoot}"
            }
        },

「PathMappings」は、ローカルマシンとリモートマシンのディレクトリをマッピングするためのルールです。「/ repo」は、Dockerにデバッグされたコードがあるディレクトリです。デバッガーは、ファイルとブレークポイントをナビゲートできる必要があります。

PSさて、テストしてみましょうか？


ローカルマシン上でnetcatを実行し、待機します。

nc -l 9001

dockerで単一行のスクリプトを実行して印刷します。

php -r 'print("Hi!" . PHP_EOL);'

ローカルマシンでは、コールサインxdebugが表示されます。

<?xml version="1.0" encoding="iso-8859-1"?>
<init xmlns="urn:debugger_protocol_v1" xmlns:xdebug="https://xdebug.org/dbgp/xdebug"...

やったー！すべて準備ができています。

これで、IDEのリモートマシンでお気に入りのコードをデバッグできます。説明されている方法には明らかな欠点があります。サーバーでのsshdの再構成です。おそらく、もっと「静かな」方法があります。いずれにせよ、私はあなたのコメントとアドバイスを喜んでいたします。
