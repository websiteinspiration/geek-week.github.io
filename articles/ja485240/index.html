VPSでのgo goプロジェクトのレイアウト

こんにちは、ハブル！私の名前はArtyom Zheltakです。私はチームリーダーであり、OTUSの「Golang Developer」コースの教師でもあります。コースの新しいストリームの開始を見越して、私の記事をあなたと共有したいと思います。

Golangは素晴らしいと思いますが、VPS、VDSに取り組んでいるphpや他のプロジェクトは世界中にまだたくさんあります。あなたはそこにドッカーを置くことができますが、これは（作者によると）タスクの再複雑化です。ファイルをコンパイルしてFTP経由でアップロードすることができます-これは安全ではなく風水ではありません。SFTPの方が安全ですが、風水を再度使用しないでください。次に、CircleCIを通じてこのプロセスを自動化しましょう。CIの構成ファイルを1つずつ記述し、最後に結果を収集してデプロイを実行します。




実装要件


  1. 最小限のサーバー革新
  2. 展開を自動化する必要がある
  3. デプロイのエントリポイントは、開発アセンブリのタグ付けと、製品の追加の手動確認です
  4. アセンブリは自動テストに合格する必要があります
  5. 手動ロールバックメカニズムバージョン

CircleCIを選ぶ理由


当初から、プロジェクトはプライベートなbitbucketリポジトリを使用していました。（現在、プライベートリポジトリはすでにgithubにあります。）エコシステムを離れることなく、AtlasianはCircleCI（以下、CI）を採用することを決定しました。私はそれが好きだった：

  • 最小設定
  • sshデバッグ機能
  • 無料版ですが制限があります
    • 毎週2500クレジット（約250分の完了）#goは迅速に収集および展開されます。
    • シングルスレッド実行＃多くのペットプロジェクトがない
    • LinuxとWindowsのみ＃Linuxが必要

パート1、ワークフロー


.circleフォルダーを作成し、そこにconfig.ymlファイルを作成して、そこに期待されるワークフロー（タスクの実行順序）を記述します。

    workflows:
      version: 2
      tagged-build:
        jobs:
          - test
          - dev_deploy:
              requires:
                - test
          - approve_master_deploy:
              type: approval
              requires:
                - test
                - dev_deploy
          - prod_deploy:
              requires:
                - dev_deploy
                - approve_master_deploy

各コミットが最初にテストによってチェックされ、次にdevにロールアウトされ、手動で確認されてバトルサーバーに送信されるパターンを説明しました。光沢をガイドするには、タスクがタグによってのみ機能するようにフィルターを追加します。

- dev_deploy:
    requires:
      - test
    filters:
      branches:
        ignore: /.*/
      tags:
        only: /.*/

2番目のステップは、最も簡単です


テストを実行することから始めましょう。最小限のコードがあります。

jobs:
  test:
    docker:
      - image: circleci/golang:1.12
    working_directory: ~/go-example/
    steps:
      - checkout
      #   linter'  
      - run: go test -cover -v ./...

コードをテストしてコードスタイルチェックに合格したら、devにデプロイできます。監督者（執筆時点ではバージョン3.1.4）を使用してgoサービスを開始することをお勧めします。これらのログを収集します。

supervisor_ph.confファイルを.circleciフォルダーに追加します。CIPH_NAME内で、プロジェクト名に変更されます。そして、同じファイルにログの出力を書き込みます。

[program:PH_NAME]
stopasgroup=true
user=deploy-user
autostart=true
autorestart=true
stdout_logfile=/var/log/supervisor/PH_NAME.log
stderr_logfile=/var/log/supervisor/PH_NAME.log
redirect_stderr=true

展開時間


devおよびprodの場合、サーバーのみが変更され、アプリケーション名にサフィックスが追加されます。設定は環境変数に保存されます。（12要素アプリケーション）この部分を環境に出力し、残りを複製します。

prod_deploy:
  environment:
    TARGET_IP: 0.0.0.0
    TARGET_DIR: /var/www/deploy-user/go-example
    REMOTE_USER: deploy-user
    SERVICE_NAME: go_example_prod
  docker:
    - image: circleci/golang:1.12
  working_directory: ~/go-example/
  steps:
    - checkout
    - add_ssh_keys #   ci ,   
    - run: go build -ldflags "-X main.version=$CIRCLE_TAG" -o ./main ./src/main
    - run: ssh -o "StrictHostKeyChecking=no" $REMOTE_USER@$TARGET_IP "mkdir $TARGET_DIR/v$CIRCLE_TAG" #    ,       
    - run: scp main $REMOTE_USER@$TARGET_IP:$TARGET_DIR/v$CIRCLE_TAG/ #      
    - run: sed "s/PH_NAME/$SERVICE_NAME/g" .circleci/supervisor_ph.conf > .circleci/$SERVICE_NAME.conf
    - run: echo command=$TARGET_DIR/v$CIRCLE_TAG/main >> .circleci/$SERVICE_NAME.conf
    - run: scp .circleci/$SERVICE_NAME.conf $REMOTE_USER@$TARGET_IP:$TARGET_DIR/v$CIRCLE_TAG/
    - run: ssh $REMOTE_USER@$TARGET_IP "ln -sf $TARGET_DIR/v$CIRCLE_TAG/$SERVICE_NAME.conf /etc/supervisord.d"
    - run: ssh $REMOTE_USER@$TARGET_IP "supervisorctl  -c /etc/supervisord.conf reread && supervisorctl -c /etc/supervisord.conf update"
    - run: curl "$TELEGRAM_SERVICE?msg=$SERVICE_NAME%20v$CIRCLE_TAG%20deployed&channel=go_deploy"

通知には、curlを通じて呼び出される独自のボットを使用します。`when：on_fail`コマンドは、問題が発生した場合に機能し、変更をロールバックするためにも使用できます。私たちは電報ボットを持っていますが、一般的にはそれなしで行うことができ、標準の通知を使用できます：Slack、IRC。さらに、エラー通知はメールで送信されます。

`$ TELEGRAM_SERVICE`変数は、セクションBUILD SETTINGS→Environment Variablesを通じて追加されます。

- run:
    command: curl "$TELEGRAM_SERVICE?msg=$SERVICE_NAME%20v$CIRCLE_TAG%20failed&channel=go_deploy"
    when: on_fail
