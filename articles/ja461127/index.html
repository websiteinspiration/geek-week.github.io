VMware vSphereでのVMパフォーマンス分析。パート3：ストレージ



パート1. CPUについて
パート2.メモリについて

今日は、vSphereのディスクサブシステムメトリックを分析します。ストレージの問題は、仮想マシンの速度が低下する最も一般的な理由です。CPUとRAMの場合、トラブルシューティングがハイパーバイザーレベルで終了する場合、ディスクに問題がある場合は、データネットワークとストレージシステムに対処する必要があります。

ファイルアクセスの場合のカウンターはほぼ同じですが、ストレージへのブロックアクセスの例を使用してトピックを分析します。

理論のビット


仮想マシンのディスクサブシステムのパフォーマンスについて話すときは、通常、3つの関連パラメーターに注意してください。

  • 入力/出力操作の数（1秒あたりの入力/出力操作、IOPS）;
  • 帯域幅（スループット）;
  • 入出力操作の待ち時間（待ち時間）。

IOPSの数は通常、ランダムなワークロードにとって重要です。異なる場所にあるディスク上のブロックへのアクセスです。データベース、ビジネスアプリケーション（ERP、CRM）などは、このような負荷の例として役立ちます。

スループットは、シーケンシャルロードで重要です。次々に配置されたブロックへのアクセスです。たとえば、ファイルサーバー（常にではありません）およびビデオ監視システムは、このような負荷を生成する可能性があります。

スループットは、次のようにI / O操作の数に関連しています。

スループット= IOPS *ブロックサイズ。ここで、ブロックサイズはブロックのサイズです。

ブロックサイズはかなり重要な機能です。最新のESXiバージョンでは、最大32,767 KBのブロックを使用できます。ブロックがさらに大きい場合は、いくつかに分割されます。すべてのストレージシステムがこのような大きなブロックで効果的に機能するわけではないため、詳細設定ESXiにはDiskMaxIOSizeパラメータがあります。これを使用すると、ハイパーバイザーによってスキップされるブロックの最大サイズを減らすことができます（詳細はこちら）。このパラメーターを変更する前にストレージシステムの製造元に相談するか、少なくとも実験室で変更をテストすることをお勧めします。 

大きなブロックサイズは、ストレージのパフォーマンスに悪影響を与える可能性があります。 IOPSの数とスループットが比較的小さい場合でも、ブロックサイズが大きいと待ち時間が長くなる可能性があります。したがって、このパラメーターに注意してください。

レイテンシは最も興味深いパフォーマンスパラメータです。仮想マシンのI / Oレイテンシは、次の合計です。

  • ハイパーバイザー内の遅延（KAVG、平均カーネルMilliSec /読み取り）;
  • データネットワークとストレージ（DAVG、Average Driver MilliSec / Command）によって与えられる遅延。

ゲストOS（GAVG、平均ゲストMilliSec /コマンド）に表示される遅延の合計は、KAVGとDAVGの合計です。

GAVGとDAVGが測定され、KAVGが計算されます：GAVG – DAVG。


ソース KAVG

について詳しく見てみましょう。通常の操作では、KAVGはゼロになる傾向があるか、少なくともDAVGよりもはるかに小さい必要があります。 KAVGが高くなると予測される唯一のケースは、VMディスクのIOPS制限です。この場合、制限を超えようとすると、KAVGが増加します。

KAVGの最も重要なコンポーネントはQAVGです。これは、ハイパーバイザー内で処理するためのキュー内の時間です。 KAVGの残りのコンポーネントはごくわずかです。

ディスクアダプタードライバーのキューとムーンキューのサイズは固定されています。負荷の高い環境では、このサイズを増やすと便利です。これは、（同時に、衛星へのキューが増加します）アダプタドライバでキューを増やす方法について説明します。この設定は、月で動作するVMが1つだけの場合に機能します。これはまれです。月に複数のVMがある場合は、Disk.SchedNumReqOutstandingパラメータも増やす必要があります（手順は  こちら）。キューを増やすと、QAVGとKAVGがそれぞれ減少します。

ただし、繰り返しになりますが、最初にHBAベンダーのドキュメントを読み、実験室のベンチで変更をテストします。

月へのキューのサイズは、SIOC（ストレージI / Oコントロール）メカニズムを含めることによって影響を受ける可能性があります。サーバー上のキューを月に動的に変更することにより、すべてのクラスターサーバーから月への均一なアクセスを提供します。つまり、VMが一部のホストで実行されており、不均衡なパフォーマンスが必要な場合（騒々しいネイバーVM）、SIOCはキューの長さをこのホストの月まで減らします（DQLEN）。詳細はこちら。

KAVGが整理しました。DAVGについて少し説明します。ここではすべてが単純です。DAVGは、外部環境（データネットワークとストレージ）によって導入される遅延です。最近のストレージシステムには、独自のパフォーマンスカウンターがあります。 DAVGの問題を分析するには、それらを調べることは理にかなっています。 ESXiとストレージ側で問題がなければ、データネットワークを確認してください。

パフォーマンスの問題を回避するには、ストレージに適切なパス選択ポリシー（PSP）を選択します。最近のほとんどすべてのストレージシステムは、ラウンドロビンPSPをサポートしています（ALUAの有無にかかわらず、非対称論理ユニットアクセス）。このポリシーでは、ストレージへの使用可能なすべてのパスを使用できます。 ALUAの場合、月を所有するコントローラーへのパスのみが使用されます。 ESXiのすべてのストレージシステムに、ラウンドロビンポリシーを確立するデフォルトのルールがあるわけではありません。ストレージシステムにルールがない場合は、クラスター内のすべてのホストに対応するルールを作成するストレージメーカーのプラグインを使用するか、自分でルールを作成します。詳細はこちら。 

また、一部のストレージベンダーは、パスあたりのIOPSの数を標準値の1000から1に変更することを推奨しています。私たちの実践では、これにより、ストレージからより多くのパフォーマンスを「絞り込み」、コントローラーの障害または更新が発生した場合にフェイルオーバーにかかる時間を大幅に削減できました。ベンダーの推奨事項を確認し、禁忌がない場合は、このパラメーターを変更してみてください。詳細はこちら。

仮想マシンのディスクサブシステムの基本的なパフォーマンスカウンター


vCenterのディスクサブシステムパフォーマンスカウンターは、データストア、ディスク、仮想ディスクのセクションに収集されます



。データストアセクションには、VMディスクが配置されているvSphereディスクストレージ（データストア）のメトリックが含まれます。ここに標準のカウンターがあります：

  • IOPS（1秒あたりの平均読み取り/書き込み要求数）、 
  • 帯域幅（読み取り/書き込み速度）、 
  • 遅延（読み取り/書き込み/最大の遅延）。

カウンターの名前から、原則として、すべてが明らかです。ここでも、統計は特定のVM（またはVMディスク）ではなく、データストア全体に関するものであるという事実に注目します。私の意見では、少なくとも最小測定期間が2秒であることを基準にして、ESXTOPでこれらの統計を確認する方が便利です。Disk

セクションには、VMによって使用されるブロックデバイスのメトリックが含まれます。合計タイプのIOPSカウンター（測定期間あたりのI / O操作の数）と、ブロックアクセスに関連するいくつかのカウンター（コマンドの中止、バスのリセット）があります。私の意見では、この情報はESXTOPで見る方が便利です。 セクション仮想ディスク

-VMのディスクサブシステムのパフォーマンス問題のトラブルシューティングの観点から最も有用です。ここでは、各仮想ディスクのパフォーマンスを確認できます。この情報は、特定の仮想マシンに問題があるかどうかを理解するために必要です。このセクションには、入力/出力操作の数、読み取り/書き込みボリューム、および遅延の標準カウンターに加えて、ブロックサイズを示す便利なカウンターが含まれています。読み取り/書き込みリクエストサイズ。

次の図は、VMディスクのパフォーマンスのグラフです。IOPSの数、レイテンシ、ブロックサイズを確認できます。 



SIOCが有効になっている場合は、パフォーマンスメトリックもデータストア全体で表示できます。平均レイテンシとIOPSに関する基本的な情報をいくつか示します。デフォルトでは、この情報はリアルタイムでのみ表示できます。



ESXTOP


ESXTOPには、ホストディスクサブシステム全体、個々の仮想マシンおよびそれらのディスクに関する情報を表示するいくつかの画面があります。

仮想マシンに関する情報から始めましょう。 「ディスクVM」画面は「v」キーで呼び出されます



。NVDISKはVMディスクの数です。各ディスクの情報を表示するには、「e」を押して、関心のあるVMのGIDを入力します。

この画面の残りのパラメーターの意味は、その名前から明らかです。

トラブルシューティングに役立つもう1つの画面はディスクアダプタです。これは「d」キーによって呼び出されます（フィールドA、B、C、D、E、Gは下の図で選択されています）：



NPTH-このアダプターから見える月へのパスの数。アダプターの各パスに関する情報を取得するには、「e」を押して、アダプターの名前を入力します：



AQLEN-アダプタの最大キューサイズ。

この画面には、前述した遅延カウンターもあります。KAVG/ cmd、GAVG / cmd、DAVG / cmd、QAVG / cmdです。

「u」キーで呼び出されるディスクデバイス画面には、個々のブロックデバイスに関する情報が表示されます-月（下の図では、フィールドA、B、F、G、Iが選択されています）。ここでは、月へのキューのステータスを確認できます。



DQLEN-ブロックデバイスのキューサイズ。
ACTVは、ESXiコアのI / Oコマンドの数です。

米ドルが高い場合は、ラインナップの拡大を検討する必要があります。キューのチームが多いほど、QAVGが高くなり、それに応じてKAVGも高くなります。

ディスクデバイスの画面で、VAAI（vStorage API for Array Integration）がストレージで実行されているかどうかも確認できます。これを行うには、フィールドAおよびOを選択します

。VAAIメカニズムを使用すると、作業の一部をハイパーバイザーからストレージシステムに直接転送できます（たとえば、ゼロ化、ブロックのコピー、ブロックなど）。

ESXiディスク最適化のヒント


  • ブロックのサイズに注意してください。
  • HBAに最適なキューサイズを設定します。
  • データストアでSIOCを必ず有効にしてください。
  • 製造元の推奨事項に従ってPSPを選択してください。
  • VAAIが機能していることを確認します。
