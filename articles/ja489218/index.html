初心者です。スーパー：シンプルなゲームのコードとアーキテクチャ

私たちは複雑な世界に住んでおり、単純なことを忘れ始めているようです。たとえば、Occamのかみそりについては、その原則は次のとおりです。「小さい数に基づいて実行できることは、大きい数に基づいて実行されるべきではありません。」この記事では、単純なゲームの開発に使用できる最も信頼できるソリューションではなく、単純なソリューションについて説明します。



秋のモスクワのDotNextまでに、ゲームを開発することにしました。それは人気のある錬金術のITバリエーションでした。プレーヤーは、IT、ピザ、ドードーに関連する128のコンセプトを、利用可能な4つの要素から収集する必要がありました。そして、これをアイデアから実用的なゲームまで1か月弱で実現する必要がありました。

前回の記事で、私はこの作品のデザインコンポーネントである計画、フェイカピー、そして感情について書きました。そして、この記事は技術的な部分についてです。コードもあります！

免責事項：以下で説明するアプローチ、コード、およびアーキテクチャは、複雑でも、独創的でも、信頼できるものでもありません。それどころか、それらは非常にシンプルで、時にはナイーブであり、重い負荷向けに設計されていません。ただし、サーバーでロジックを使用するゲームやアプリケーションを作成したことがない場合は、この記事が最初のきっかけになります。

クライアントコード


プロジェクトのアーキテクチャについて簡単に説明すると、UnityにはAndroidとiOSのモバイルクライアントがあり、ストレージとしてCosmosDBを備えたASP.NETにはバックエンドサーバーがありました。

Unityのクライアントは、UIとの対話のみを表します。プレーヤーが要素をクリックすると、固定された方法で画面内を移動します。新しい要素が作成されると、ウィンドウにその説明が表示されます。


ゲーム

プロセスこのプロセスは、かなり単純なステートマシンで説明できます。このステートマシンの主なものは、次のステートへの遷移のアニメーションを待って、プレーヤーのUIをブロックすることです。



非常に優れたUnitRxライブラリを使用して、完全に非同期のスタイルでUnityコードを記述しました。最初はネイティブタスクを使おうとしましたが、iOSのビルドでは不安定になりました。しかし、UniRx.Asyncは時計のように機能しました。

アニメーションを必要とするアクションはすべて、AnimationRunnerクラスを通じて呼び出されます。

public static class AnimationRunner
   {
       private const int MinimumIntervalMs = 20;
     public static async UniTask Run(Action<float> action, float durationInSeconds)
       {
           float t = 0;
           var delta = MinimumIntervalMs / durationInSeconds / 1000;
           while (t <= 1)
           {
               action(t);
               t += delta;
               await UniTask.Delay(TimeSpan.FromMilliseconds(MinimumIntervalMs));
           }
       }
   }

これは実際には、古典的なコルーチンをUnitTaskに置き換えています。さらに、UIをブロックする必要のある呼び出しは、HandleUiOperationグローバルクラスメソッドを介して呼び出されますGameManager。

public async UniTask HandleUiOperation(UniTask uiOperation)
       {
           _inputLocked = true;
           await uiOperation;
           _inputLocked = false;
       }

したがって、すべてのコントロールで、InputLockedの値が最初にチェックされ、それがfalseの場合のみ、コントロールが反応します。

これにより、ロシアの人形のようにネストされた呼び出しでasync / awaitアプローチを使用して、ネットワーク呼び出しとI / Oを含む上記のステートマシンを実装するのが簡単になりました。

クライアントの2番目の重要な機能は、要素に関するデータを受信するすべてのプロバイダーがインターフェイスの形式で作成されたことです。会議後、バックエンドをオフにすると、ゲームが完全にオフラインになるように、文字どおり一晩でクライアントコードを書き直すことができました。このバージョンはGoogle Playからダウンロードできます。

クライアントとのやり取り


次に、クライアント/サーバーアーキテクチャの開発中に行った決定について説明します。



画像はクライアントに保存され、サーバーがすべてのロジックを担当しました。起動時に、サーバーはidとすべての要素の説明を含むcsvファイルを読み取り、メモリに保存しました。その後、彼は行く準備ができていました。

APIメソッドは必要最小限でした-5つだけです。彼らはゲームのすべてのロジックを実装しました。すべては非常に単純ですが、いくつか興味深い点について説明します。

認証およびスターター要素


複雑な認証システムや一般的にはパスワードはすべて破棄しました。プレーヤーがスタート画面で名前を入力して[スタート]ボタンを押すと、クライアントに一意のランダムトークン（ID）が作成されます。ただし、デバイスには接続されません。プレーヤーの名前とトークンがサーバーに送信されます。クライアントからバックへの他のすべての要求には、このトークンが含まれています。



このソリューションの明らかな欠点は次のとおりです。

  1. ユーザーがアプリケーションを解体して再インストールした場合、ユーザーは新しいプレーヤーと見なされ、進行状況はすべて失われます。
  2. 別のデバイスでゲームを続行することはできません。

これらは意図的な仮定でした。なぜなら、人々はたった1つのデバイスからのみ会議に参加することになるからです。また、別のデバイスに切り替える時間もありません。

したがって、計画されたシナリオでは、クライアントはサーバーメソッドをAddNewUser1回だけ呼び出しました。

ゲーム画面を読み込むときにGetBaseElements、このメソッドも1回呼び出され、id、スプライトの名前、4つの基本要素の説明が返されました。クライアントはリソースで必要なスプライトを見つけ、要素のオブジェクトを作成し、それらをローカルに書き込み、画面に描画しました。

繰り返し起動すると、クライアントはサーバーに登録されなくなり、開始要素を要求せず、ローカルストレージから取得しました。その結果、すぐにゲーム画面が開きました。

要素をマージ


プレーヤーが2つの要素を接続しようとするとMergeElements、新しい要素に関する情報を返すか、これらの2つの要素が収集されていないことを報告するメソッドが呼び出されます。プレーヤーが新しい要素を収集した場合、これに関する情報がデータベースに記録されます。



サーバーへの負荷を軽減するために、プレーヤーが2つの要素を追加しようとした後、結果がクライアント（メモリとcsv）にキャッシュされます。プレーヤーがアイテムを再スタックしようとすると、最初にキャッシュがチェックされます。そして、結果がそこにない場合のみ、リクエストはサーバーに送信されます。

したがって、各プレーヤーは背中との限られた数の相互作用を行うことができ、データベース内のエントリの数は利用可能な要素の数（うち128でした）を超えないことがわかります。他の多くの会議アプリケーションの問題を回避することができました。参加者の大規模な同時流入の後、しばしばバックアップされました。

参加者は私たちの「錬金術」をそのようにではなく、賞品のために演奏しました。したがって、スタンドの画面とゲームの別のウィンドウに表示するレコードのテーブルが必要でした。



レコードのテーブルを形成するために、最後の二つの方法が使用されているGetCurrentLadderとGetUser.好奇心旺盛なニュアンスもあります。プレーヤーが上位20の結果に含まれている場合、その名前がテーブルで強調表示されます。問題は、これら2つの方法の情報が直接関連していないことでした。

このメソッドGetCurrentLadderはコレクションStatsにアクセスし、20個の結果を取得してすばやく実行します。メソッドGetUserはコレクションにアクセスします。UsersUserIdによるもので、速すぎます。結果のマージはすでにクライアント側で行われています。これは、結果でUserIdを使用したくなかっただけなので、そこにはありません。比較は、プレーヤーの名前と得点の数に従って行われました。何千人ものプレイヤーの場合、衝突は必然的に起こります。しかし、すべてのプレイヤーの中で同じ名前とポイントを持つプレイヤーがいる可能性は低いという事実を頼りにしました。私たちの場合、このアプローチは完全に正当化されています。
