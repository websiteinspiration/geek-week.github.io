DVCセットアップ


データバージョン管理は、MLプロジェクトのモデルとデータバージョンを管理するために設計されたツールです。実験段階とモデルを運用に展開する際の両方に役立ちます。



DVCを使用すると、DSプロジェクトのモデル、データ、パイプラインにバージョンを付けることができます。
ソースはこちらです。

虹彩の色分類問題の例を使用して、DVC操作を見てみましょう。これを行うには、よく知られているデータセットIris Data Setを使用します。DVC を使用する他の例は、Jupyter Notebookに示されています。

何をすべきか：

  • リポジトリのクローンを作成します。
  • 仮想環境を作成します。
  • 必要なpythonパッケージをインストールします。
  • DVCを初期化します。

したがって、リポジトリを複製し、仮想環境を作成して、必要なパッケージをインストールします。インストールと起動の手順は、READMEリポジトリにあります。
1.このリポジトリのクローンを作成します

git clone https://gitlab.com/7labs.ru/tutorials-dvc/dvc-1-get-started.git
cd dvc-1-get-started

2.仮想環境を作成してアクティブ化する

pip install virtualenv
virtualenv venv
source venv/bin/activate

3. Pythonライブラリ（dvcを含む）をインストールします。

pip install -r requirements.txt

DVCをインストールするには、コマンドを使用しpip install dvcます。インストール後、プロジェクトフォルダーdvc initでDVCを初期化する必要があります。これにより、DVCをさらに操作するための一連のフォルダーが生成されます。
4.デモリポジトリの新しいブランチをチェックアウトします（マスターブランチのコンテンツをワイプしないため）

git checkout -b dvc-tutorial

5. DVCを初期化する

dvc init
commit dvc init
git commit -m "Initialize DVC"

DVCはGit上で実行され、そのインフラストラクチャを使用し、同様の構文を持っています。
このプロセスでは、DVCがメタファイルを作成して、プロジェクト履歴をGitに保存するために必要なパイプラインとバージョン付きファイルを記述します。そのため、実行後dvc initはgit commitすべての設定を確定するために実行する必要があります。

フォルダは、リポジトリに表示されます.dvcに基づいて行われます、cacheとconfig。

コンテンツ.dvcは次のようになります。

./      
../     
.gitignore  
cache/  
config

ConfigはDVC構成であり、キャッシュは、DVCがバージョン管理するすべてのデータとモデルを格納するシステムフォルダーです。

DVCは.gitignore、リポジトリにコミットする必要のないファイルとフォルダを書き込むファイルも作成します。Gitでバージョン管理するためにファイルをDVCに転送すると、バージョンとメタデータが保存され、ファイルはキャッシュに保存されます。

次に、すべての依存関係をインストールし、それcheckoutを新しいブランチdvc-tutorialにして、作業を行います。そして、Irisデータセットをダウンロードします。
データを取得する 

wget -P data/ https://raw.githubusercontent.com/uiuc-cse/data-fa14/gh-pages/data/iris.csv

DVCの機能


バージョン管理モデルとデータ

ソースはこちら。

DVCの制御下で一部のデータを転送すると、すべての変更の追跡が開始されることを思い出してください。そして、Gitと同じ方法でこのデータを操作できます。バージョンを保存し、それをリモートリポジトリに送信し、正しいバージョンのデータを取得し、バージョンを変更して切り替えます。 DVCでのインターフェースは非常にシンプルです。

コマンドを入力し、dvc addバージョン管理する必要があるファイルへのパスを指定します。 DVCは、拡張子が.dvcのiris.csvメタファイルを作成し、それに関する情報をキャッシュフォルダーに書き込みます。これらの変更をコミットして、バージョン管理の開始に関する情報がGit履歴に表示されるようにします。

dvc add data/iris.csv

生成されたdvcファイル内に、標準パラメーターを含むそのハッシュが格納されます。

Output-DVCの制御下で追加したdvcフォルダー内のファイルへのパス。システムはデータを取得してキャッシュに入れ、作業ディレクトリにキャッシュへのリンクを作成します。このファイルをGit履歴に追加して、バージョンを付けることができます。 DVCはデータ自体の管理を引き継ぎます。ハッシュの最初の2文字はキャッシュ内のフォルダーとして使用され、残りの文字は作成されたファイルの名前として使用されます。



MLパイプラインの自動化

データのバージョン管理に加えて、依存関係が定義される計算のチェーンであるパイプライン（パイプライン）を作成できます。トレーニングと分類子の評価の標準的なパイプラインは次のとおりです



。入力には、前処理してトレーニングとテストに分割し、特徴を計算してモデルをトレーニングして評価する必要のあるデータがあります。このパイプラインは、個別の部分に分割できます。たとえば、データの読み込みと前処理、データの分割、評価などの段階を区別し、これらのチェーンを接続します。

これを行うために、DVCにはdvc run特定のパラメーターを渡し、実行する必要があるPythonモジュールを指定する素晴らしいコマンドがあります。

今-たとえば、標識の計算の段階の開始。まず、featureization.pyモジュールの内容を見てみましょう：

import pandas as pd

def get_features(dataset):

   features = dataset.copy()

   # uncomment for step 5.2  Add features
   # features['sepal_length_to_sepal_width'] = features['sepal_length'] / features['sepal_width']
   # features['petal_length_to_petal_width'] = features['petal_length'] / features['petal_width']

   return features


if __name__ == '__main__':

   dataset = pd.read_csv('data/iris.csv')

   features  = get_features(dataset)
   features.to_csv('data/iris_featurized.csv', index=False)

このコードはデータセットを取得し、特性を計算して、iris_featurized.csvに保存します。追加の兆候の計算は次の段階に委ねました。

パイプラインを作成するには、計算の各段階でコマンドを実行する必要があります
dvc run。



まず、dvc runコマンドでstage_feature_extraction.dvcメタファイルの名前を指定します。DVCは、そこに計算ステージに関する必要なメタデータを記録します。引数を通じて、-d必要な依存関係（featureization.pyモジュールとiris.csvデータファイル）を示します。また、標識が保存されるiris_featurized.csvファイル、およびpython src / featurization.py起動コマンド自体も指定します。

dvc run -f stage_feature_extraction.dvc \
    -d src/featurization.py \
    -d data/iris.csv \
    -o data/iris_featurized.csv \
    python src/featurization.py

DVCはメタファイルを作成し、Pythonモジュールとiris.csvファイルの変更を追跡します。
それらに変更が発生した場合、DVCはパイプラインでこの計算ステップを再開します。

結果のstage_feature_extraction.dvcファイルには、そのハッシュ、開始コマンド、依存関係、および出力が含まれます（メタデータで見つけることができる追加のパラメーターがあります）。

次に、このファイルをGitコミットの履歴に保存する必要があります。したがって、新しいブランチを作成し、それをGitリポジトリーにプッシュできます。 Gitストーリーにコミットするには、各ステージを個別に作成するか、すべてのステージを一度に作成します。

実験全体に対してこのようなチェーンを構築すると、DVCは計算グラフ（DAG）を構築します。これにより、パイプライン全体または一部の再計算を開始できます。あるステージからの出力のハッシュは、別のステージの入力に送られます。彼らによれば、DVCは依存関係を追跡し、計算のグラフを作成します。 split_dataset.pyのどこかでコードを変更した場合、DVCはデータをロードせず、おそらく兆候を再カウントしますが、この段階とその後のトレーニングおよび評価段階を再開します。



メトリックの追跡

コマンドdvc metrics showを使用すると、現在の起動、つまり現在のブランチのメトリックを表示できます。パラメータを渡す-aと、DVCはGit履歴にあるすべてのメトリックを表示します。 DVCがメトリックの追跡を開始するために、評価ステップの作成時に、-mdata / eval.txtを介してパラメーターを渡します。 evaluate.pyモジュールは、この場合には、このファイルにメトリックを書き込み、f1およびconfusion metrics。この段階のDVCファイルに出力フォルダで、cacheとされてmetricstrueに設定します。つまり、dvc metrics showコマンドは、eval.txtファイルの内容をコンソールに出力します。また、このコマンドの引数を使用すると、のみf1_scoreまたはのみを表示できますconfusion_matrix。

パイプラインの再現性

このデータセットで作業した人は、このデータセットで適切なモデルを構築することが非常に難しいことを知っています。

これで、DVCを使用して作成されたパイプラインがあります。システムはデータとモデルの履歴を追跡し、全体または一部を再起動して、メトリックを表示できます。必要な自動化をすべて完了しました。

f1 = 0.78のモデルがありました。いくつかのパラメーターを変更して改善したいと考えています。これを行うには、理想的には1つのコマンドだけでパイプライン全体を再起動します。さらに、チームで作業している場合は、モデルとコードを同僚に渡し、同僚が引き続き作業できるようにすることができます。

このコマンドをdvc repro使用すると、パイプラインまたは個々のステージを再起動できます（この場合、コマンドの後に再現されたステージを指定する必要があります）。

コマンドを実行するdvc repro stage_evaluateと、ステージはパイプライン全体を再起動しようとします。しかし、現在の状態でこれを行うと、DVCは変更を認識せず、再起動しません。そして、何かを変更すると、彼はその変更を見つけて、この時点からパイプラインを再起動します。

$ dvc repro stage_evaluate.dvc

Stage 'data/iris.csv.dvc' didn't change.
Stage 'stage_feature_extraction.dvc' didn't change.
Stage 'stage_split_dataset.dvc' didn't change.
Stage 'stage_train.dvc' didn't change.
Stage 'stage_evaluate.dvc' didn't change.
Pipeline is up to date. Nothing to reproduce.

この場合、DVCはstage_evaluateステージの依存関係の変更を認識せず、再起動を拒否しました。パラメータを指定すると、-fすべての準備手順が再開され、追跡していたデータの以前のバージョンが削除されるという警告が表示されます。 DVCはステージを再起動するたびに、以前のキャッシュを削除し、実際にはデータを重複させないように上書きします。 DVCファイルが起動されると、そのハッシュがチェックされ、変更されている場合、パイプラインは再起動して、このパイプラインが持つすべての出力を上書きします。これを回避したい場合は、最初に特定のバージョンのデータをリモートリポジトリで実行する必要があります。

パイプラインを再起動して各ステージの依存関係を追跡する機能により、モデルをより速く実験できます。

たとえば、特性を変更できます（で特性を計算するための行のコメントを解除しますfeaturization.py）。 DVCはこれらの変更を確認し、パイプライン全体を再起動します。

リモート


DVC リポジトリへのデータの保存は、ローカルバージョンのストレージだけでなく機能します。コマンドを実行するdvc pushと、DVCはモデルとデータの現在のバージョンを事前構成されたリモートリポジトリストレージに送信します。その後、同僚がgit cloneリポジトリを作成して実行するdvc pullと、このブランチ用のデータとモデルのバージョンが受信されます。主なことは、誰もがこのリポジトリにアクセスできることです。



この場合、temp / dvcフォルダーの「リモート」ストレージをシミュレートします。ほぼ同じ方法で、リモートストレージがクラウドに作成されます。この変更をコミットして、Gitストーリーに残るようにします。これでdvc push、このストレージにデータを送信するための実行が可能になり、同僚はデータを受信するために実行するだけdvc pullです。

そこで、DVCと基本機能が役立つ3つの状況を検討しました。

  • データとモデルのバージョン管理。パイプやリモートリポジトリが必要ない場合は、ローカルマシンで作業している特定のプロジェクトのデータをバージョン管理できます。DVCを使用すると、数十ギガバイトのデータをすばやく処理できます。
  • チーム間のデータとモデルの交換。クラウドソリューションを使用してデータを保存できます。これは、チームが分散している場合、またはメールで送信されるファイルのサイズに制限がある場合に便利なオプションです。また、この手法は、ノートブックを相互に送信する場合に使用できますが、開始されません。
  • 大規模サーバー内でのチーム作業の編成。チームは、ビッグデータのローカルバージョン（たとえば、数十ギガバイトまたは数百ギガバイト）を操作できるため、データを相互にコピーするのではなく、重要なバージョンのモデルまたはデータのみを送信および保存する1つのリモートストレージを使用します。

プロジェクトにDVCを実装する方法は？

プロジェクトの再現性を確保するには、特定の要件を遵守する必要があります。
主なものは次のとおりです。

  • すべてのパイプラインは自動化されています。
  • 計算の各段階の起動パラメーターの制御。
  • コード、データ、モデルのバージョン管理。
  • 環境管理;
  • ドキュメンテーション。

これがすべて完了すると、プロジェクトは再現可能になる可能性が高くなります。DVCでは、このリストの3つの最初の要件を満たすことができます。

会社にDVCを実装しようとすると、「なぜこれが必要なのですか。Jupyter Notebookがあります。」おそらく、同僚の一部はJupyter Notebookのみを使用しており、そのようなパイプラインやコードをIDEで作成することははるかに困難です。この場合、段階的な実装を行うことができます。

  1. 開始する最も簡単な方法は、コードとモデルをバージョン管理することです。
    次に、パイプラインの自動化に進みます。
  2. 最初に、頻繁に再起動および変更されるステップを自動化し、
    次にパイプライン全体を自動化します。

チームに新しいプロジェクトと2、3人の愛好家がいる場合は、すぐにDVCを使用することをお勧めします。したがって、たとえば、それは私たちのチームで判明しました！新しいプロジェクトを始めるとき、私の同僚が私をサポートしてくれて、私たちは自分でDVCを使い始めました。その後、彼らは他の同僚やチームと共有し始めました。誰かが私たちの仕事を手に入れました。今日、DVCはまだ銀行で一般的に受け入れられているツールではありませんが、いくつかのプロジェクトで使用されています。
