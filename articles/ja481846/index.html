ElixirプロジェクトでのGitHub CIの使用

10月に、彼はこのコードが格納されているキャッシュデスクを離れることなくCIを実行できるアクションをGithub開始しました。とても便利です。誰かがプルリクエストを送信したり、サーバーに新しい変更をアップロードしたり、何か特別なもの（アクションを固定できるイベントのリストは公式のドキュメントに記載されています）になり次第、アセンブリが開始されます。スケジュールされた繰り返しタスク（cronベース）もサポートされています。


ワークフローと呼ばれるアクションパイプラインを作成できます。そして、これはすべて美しく、ドキュメントを除いて明るい未来のように見えます。


アプリケーションをテストするためのサードパーティのサービスを含むコンテナを作成する方法を理解するのに1時間以上かかりました。これが私がなんとか見つけたものです。公式ドキュメントは率直に扱いにくく、不完全であり、多くの場合単に誤りがあることに注意してください。


標準のCI（アクション）アクションは、CircleCIで使用されるものとよく似た構文の構成ファイルを使用します。これは、古き良きYAMLであり、ターゲットOS、環境、実行するコマンドなどを構成できます。アクション自体に一意の名前が付けられるため、他のアクションを参照してそれらに依存できます。


さらに、設定ではサービスを指定できます。サービスはクラウドのどこかで実行されている必要があり、GHは構成に従って、これらのサービスが公開するポートにコンテナーポートをマップします。この部分は公式ドキュメントでは不十分にカバーされており、説明されているものでさえエラーが含まれています。


以下は、RabbitMQとRedisサービスをテストする必要があるElixirプロジェクトの実際の構成例です。


name: Tests for My Project

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest

    container:
      image: elixir:1.9.1-slim

    services:
      rabbitmq:
        image: rabbitmq
        ports:
        - 5672:5672
        env:
          RABBITMQ_USER: guest
          RABBITMQ_PASSWORD: guest
          RABBITMQ_VHOST: "/"
      redis:
        image: redis
        ports:
        - 6379:6379

    steps:
    - uses: actions/checkout@v1
    - name: Install Dependencies
      run: |
        MIX_ENV=ci mix local.rebar --force
        MIX_ENV=ci mix local.hex --force
        MIX_ENV=ci mix deps.get
    - name: Run All Tests
      run: |
        MIX_ENV=ci mix test
      env:
        RABBITMQ_HOST: rabbitmq
        RABBITMQ_PORT: ${{ job.services.rabbitmq.ports[5672] }}
        REDIS_HOST: redis
        REDIS_PORT: ${{ job.services.redis.ports[6379] }}
