前書き


党と政府の政策に関連して、 BIMテクノロジー-ビルディングインフォメーションモデリングを導入するために、法律の積極的な変更があります。パーティーの続きとして、オープンプレゼンテーション形式のBIM-IFC（Industry Foundation Classes）を検討します。


IFCの歴史は、1995年（実際には1993年の夏[1]）に始まります。このとき、Autodesk Corporationと「仲間」のグループは、建築設計のためのさまざまなCADシステムの交換フォーマットを開発するためのカルテル契約を組織しました。 1年後、1996年に国際相互運用性アライアンスが登場したため、同志はオープンなメンバーシップを持つ組織によってこのフォーマットがオープンで開発されるべきであるという理解に達しました。その後、2008年に、組織はより魅力的になるbuildingSMARTに名前が変更されました。

IFC開発者は、豊かな想像力を持っていなかったし、それを適用する機会もありませんでした-彼らは非常に控えめな期限に設定され、タスクは非常にグローバルに見えました。したがって、彼らは基本として、STEP（製品モデルデータの交換のための標準）形式、またはアプリケーションプロトコル225：構築要素を採用しました。 STEPの周りには、ISO規格のステータスで、一連の仕様という形で豊富なインフラストラクチャが作成されていると言わざるを得ません。このインフラストラクチャは、EXPRESSデータモデリング言語とそのグラフィカルインカネーションであるEXPRESS-Gに基づいています。この言語は、さまざまなプログラミング言語でコードを自動生成するために開発されました。

IFCの開発は1995年9月に始まり、IFC 1.0は1996年6月に公開され、最終バージョンは1997年1月に発表されました。実際、IFCの最初のバージョンの目的は、目標を実現する可能性を示すことであり、さまざまな企業がこの形式で輸出/輸入のデモを行いました。

次のバージョンは1997年11月にリリースされた-1.5ですが、それを実装しようとすると、バージョン2.0の開発と並行して導入された修正バージョン1.5.1の開発を必要とする多くのエラーがすぐに明らかになりました-バージョン2.0の開発-1999年3月に導入されました。

これらのバージョンはすべて非推奨になりました。

バージョン2.1は2000年11月にリリースされました。これは、ドキュメントが利用可能な最も古いバージョンです。その後、ISO / PAS 16739：2005として発行されました。

現在、最も一般的なバージョン（ほとんどのプログラムが理解しています）はIFC 2.3です。

IFCのルート要素はIfcProjectです。ここでは、エンティティの作成に必要な属性のリストがどのように形成されるかを通知する必要があります。最初に、エンティティは独自の属性を持つことができます。次に、エンティティを祖先から継承できます。属性の順序は、祖先から子孫に指定されます。 IfcProjectの場合、継承チェーンは次のようになります：IfcRoot => IfcObjectDefinition => IfcObject => IfcProject。

IFCPROJECT(<GlobalId>,<OwnerHistory>,<>,<>,<ObjectType>, <LongName>,<Phase>, (<RepresentationContexts>),<UnitsInContext>);

ここで、IfcProjectを作成するには、すべての属性の値を設定する必要があります。 IfcRoot-GlobalIdから継承された最初の属性の値はIfcGloballyUniqueIdです。これは単純なタイプです。長さが22文字の文字列で、一意のGUIDまたはUUIDを書き込む必要があるため、128ビットの数値が22文字に入力されます。buildingSMARTWebサイトで公開されている特別なアルゴリズムがあります[3]。次のOwnerHistory属性はIfcOwnerHistoryです。この要素は、IFCのほぼすべてのオブジェクトについて、このIFC要素を誰が、どのように、いつ作成したかという質問に答えます。この要素を通じて、作成者を示すことができます。この属性を埋めるには、「所定の場所」に要素を作成できますが、別の場所で作成することをお勧めします。代わりに、＃<entity index>の形式のレコードでそれを参照するだけです。また、$記号はnull値を意味し、*記号は子孫が値を祖先の属性に割り当てる場合に使用されます。 enumタイプの値は、2つのポイント-.ELEMENTの間に書き込まれます。

結論


今、私の愛する読者、あなたはあなたの夢の家を書くことができます。残念ながら、オブジェクトの表示スタイルで拒否されたIfcMaterialDefinitionRepresentationは考慮しませんでした。IfcTopologyRepresentationは考慮していません。それが何のためにあるのか本当に理解できず、視覚化する方法もわかりません。 IFCオプションと追加のプロパティセットは考慮しませんでした。しかし、そうでなければ、これは簡単な紹介ではありません。

IFC形式には膨大な数のオブジェクトが含まれており、バージョンによってバージョンが増えるだけです。仕様のテキストには、EXPRESSスキームには反映されていないが、同時にファイルの処理に強く影響する注記があります。したがって、すべてのドキュメントを注意深く読まなければ、この形式を実装することは困難ですが、これは1人ではほとんど不可能であるため、完全に正しく読み取るプログラムはなく、それぞれに独自の特性があります。また、オープンソースプログラムの場合、検出されたエラーを修正する機会が常にある場合、プロプライエタリプログラムでは、これによりIFC形式を完全に使用することができなくなります。

IFC形式は、一般的な計画セクションの情報を格納するのにはまったく適していませんが、現在このセクションで活発に作業が行われています。この作業は、2020年4月末までに完了し、IFC5の一部になる予定です。現在、IFC Road、IFC Airport、IFC4precast（プレキャスト）にも取り組んでいます。 IFCブリッジはIFC4x2で登場し、特別な幾何学的オブジェクト-IfcSectionedSolidHorizo​​ntalが登場しました。

IFCの最近の変更により、IfcCoordinateReferenceSystem（測地座標系の説明）が登場したとしても、GMLに非常に近づいています。同時に、IFCはオブジェクトの内部構造の記述に焦点を合わせ、GMLはその外部表現を記述します。ただし、IFCの主な違いは、異なる場所にある同じオブジェクトを参照できることです。壁と窓のジオメトリの説明では、同じ点を使用できます。ただし、GMLでは、各幾何オブジェクトは完全に独立しています。
