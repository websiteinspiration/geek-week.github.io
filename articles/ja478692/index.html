AndroidでのJava 8のサポート方法

こんにちは、ハブル！Android Java 8のサポートについて、悪名高いJake Wortonによる一連の記事からのすばらしい記事の翻訳に注目します。元の記事はここにありますか？





私は数年間自宅で仕事をしていましたが、AndroidがさまざまなバージョンのJavaをサポートしているという同僚の不満をよく耳にしました。

これはかなり複雑なトピックです。最初に、「AndroidでのJavaサポート」の意味を決定する必要があります。言語の1つのバージョンでは、機能（ラムダなど）、バイトコード、ツール、API、JVMなど、さまざまなことが存在する可能性があるためです。

彼らがAndroidでのJava 8サポートについて話すとき、彼らは通常言語機能のサポートを意味します。それでは、それらから始めましょう。

ラムダス


Java 8の主な革新の1つはラムダでした。
コードはより簡潔でシンプルになり、ラムダにより、内部に単一のメソッドを持つインターフェースを使用してかさばる匿名クラスを作成する必要がなくなりました。

class Java8 {
  interface Logger {
    void log(String s);
  }

  public static void main(String... args) {
    sayHi(s -> System.out.println(s));
  }

  private static void sayHi(Logger logger) {
    logger.log("Hello!");
  }
}

javacとlegacyを使用してこれをコンパイルするとdx tool、次のエラーが発生します。

$ javac *.java
$ ls
Java8.java  Java8.class  Java8$Logger.class
$ $ANDROID_HOME/build-tools/28.0.2/dx --dex --output . *.class
Uncaught translation error: com.android.dx.cf.code.SimException:
  ERROR in Java8.main:([Ljava/lang/String;)V:
    invalid opcode ba - invokedynamic requires --min-sdk-version >= 26
    (currently 13)
1 error; aborting

このエラーは、ラムダinvokedynamicがJava 7で追加されたバイトコードの新しい命令を使用するために発生します。エラーテキストから、Androidが26 API（Android 8）以降でのみサポートしていることがわかります。

26 minApiでアプリケーションをリリースする人はほとんどいないので、あまり良くないように思えます。これを回避するために、いわゆるdesugaringプロセスが使用されます。これにより、APIのすべてのバージョンでラムダサポートが可能になります。

脱糖化の歴史


彼女はAndroidの世界ではかなりカラフルです。脱糖化の目的は常に同じです。すべてのデバイスで新しい言語機能を使用できるようにするためです。

たとえば、当初、Androidでラムダをサポートするために、開発者はRetrolambdaプラグインを接続しました。彼は、JVMと同じ組み込みメカニズムを使用してラムダをクラスに変換しましたが、コンパイル時ではなく実行時にそれを行いました。生成されたクラスは、メソッドの数の点で非常に高価でしたが、時間の経過とともに、改良と改善の後、このインジケーターは多かれ少なかれ妥当なものに落ちました。

その後、Androidチームは新しいコンパイラを発表しました、すべてのJava 8機能をサポートし、生​​産性が向上しました。 Eclipse Javaコンパイラーの上に構築されましたが、Javaバイトコードを生成する代わりに、Dalvikバイトコードを生成しました。しかし、その性能はまだ望まれるべき多くを残しました。

新しいコンパイラは（幸い）放棄された場合には、ジャグリングをしたJavaバイトコードのJavaバイトコードの変圧器は、アンドロイドのGradleプラグインに統合されたからBazel、Googleのビルドシステム。そして、そのパフォーマンスはまだ低かったので、並行して、より良いソリューションの検索が続けられました。

そして、私たち dexer- 取り替えられるはずだったD8dx tool。コンパイルされたJARファイルから.dex（デクシング）。D8はに比べてパフォーマンスがはるかに優れてdxおり、Android Gradle Plugin 3.1以降、デフォルトのdexerになっています。

Defaultおよびstaticインターフェースのメソッド


Java 8にもたらされたもう1つの重要かつ重要な変更は、宣言defaultとstaticメソッドのインターフェースでの機能です。

interface Logger {
  void log(String s);

  default void log(String tag, String s) {
    log(tag + ": " + s);
  }

  static Logger systemOut() {
    return System.out::println;
  }
}

これはすべてD8でもサポートされています。以前と同じツールを使用して、Logger'a defaultとstaticメソッドでzadeshugarennuyuバージョンを簡単に確認できます。ラムダとの違いの1つは、method referencesデフォルトメソッドと静的メソッドがAndroid VMに実装されており、24 API以降、D8がそれらを分離しないことです。

Desugaring API


これまでずっと、Java 8の機能について話しましたが、新しいAPIについては何も話しませんでした-ストリーム、CompletableFuture日付/時刻など。

ロガーの例に戻り、新しい日付/時刻APIを使用して、メッセージがいつ送信されたかを確認できます。

import java.time.*;

class Java8 {
  interface Logger {
    void log(LocalDateTime time, String s);
  }

  public static void main(String... args) {
    sayHi((time, s) -> System.out.println(time + " " + s));
  }

  private static void sayHi(Logger logger) {
    logger.log(LocalDateTime.now(), "Hello!");
  }
}
