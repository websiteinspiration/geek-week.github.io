ステップ1.ツールの準備


最初に行う必要があるのは、Visual Studio 2017をバージョン15.3以降にアップグレードすることです。そして、最新バージョンの.NET Core SDKをインストールします。

移行を開始する前に、.NET .Net Portability Analyzerの移植性分析ツールを使用することをお勧めします。これは、あるプラットフォームから別のプラットフォームへの移行がどれほど困難になるかを理解するための良い出発点です。しかし、もちろん、このツールはすべての問題を解決するわけではなく、その過程で多くの落とし穴があります。次に、通過する必要がある主なステージについて説明し、プロジェクトで使用するソリューションを示します。

最初に必要なことは、.NET標準をサポートするプロジェクトで使用されるライブラリへのリンクを更新することです。

ステップ6. HTTPハンドラーとHTTPモジュールをミドルウェアに転送する


HTTPハンドラーとHTTPモジュールは、基本的にASP.NET Coreのミドルウェアの概念に非常に似ていますが、モジュールとは異なり、ミドルウェアの順序は、要求パイプラインに挿入される順序に基づいています。モジュールの順序は、ほとんどの場合、アプリケーションのライフサイクルイベントに基づいています。回答のミドルウェアの順序は、要求の順序とは逆であり、要求と回答のモジュールの順序は同じです。これに基づいて、アップグレードを続行できます。

だから、更新されていないもの：

  • ミドルウェアのモジュールの移行（AuthenticationMiddleware、CultureMiddlewareなど）
  • 中間ソフトウェア・ハンドラ（ハンドラミドルウェアへ）
  • 新しいミドルウェアの使用

このプロジェクトの認証では、組み込みの資格情報システムを使用しません。これらの目的のために、新しいASP.NET Core構造に従って開発されたミドルウェアAuthenticationMiddlewareが使用されます。

public class AuthenticationMiddleware
{
   private readonly RequestDelegate _next;
   public AuthenticationMiddleware(IAuthenticationSchemeProvider schemes, RequestDelegate next)
   {
       Schemes = schemes ?? throw new ArgumentNullException(nameof(schemes));
       _next = next ?? throw new ArgumentNullException(nameof(next));
   }

   public IAuthenticationSchemeProvider Schemes { get; set; }
   
   public async Task Invoke(HttpContext context)
   {
       context.Features.Set<IAuthenticationFeature>(new AuthenticationFeature
       {
           OriginalPath = context.Request.Path,
           OriginalPathBase = context.Request.PathBase
       });
      
       var handlers = context.RequestServices.GetRequiredService<IAuthenticationHandlerProvider>();
       foreach (var scheme in await Schemes.GetRequestHandlerSchemesAsync())
       {
           try
           {
               if (await handlers.GetHandlerAsync(context, scheme.Name) is IAuthenticationRequestHandler handler && await handler.HandleRequestAsync())
                   return;
           }
           catch
           {
               // ignored
           }
       }

       var defaultAuthenticate = await Schemes.GetDefaultAuthenticateSchemeAsync();
       if (defaultAuthenticate != null)
       {
           var result = await context.AuthenticateAsync(defaultAuthenticate.Name);
           if (result?.Principal != null)
           {
               context.User = result.Principal;
           }
       }
       await _next(context);
   }
}

ASP.NETには、アプリケーションで使用できる多くの組み込みミドルウェアが用意されていますが、開発者は独自のミドルウェアを作成してHTTPリクエストパイプラインに追加できることに注意してください。このメカニズムを簡略化するために、特別なインターフェイスを追加しました。これで、それを実装するクラスを作成するだけで十分です。

public interface INopStartup
{
    /// <summary>
    /// Add and configure any of the middleware
    /// </summary>
    /// <param name="services">Collection of service descriptors</param>
    /// <param name="configuration">Configuration of the application</param>
    void ConfigureServices(IServiceCollection services, IConfiguration configuration);

    /// <summary>
    /// Configure the using of added middleware
    /// </summary>
    /// <param name="application">Builder for configuring an application's request pipeline</param>
    void Configure(IApplicationBuilder application);

    /// <summary>
    /// Gets order of this startup configuration implementation
    /// </summary>
    int Order { get; }
}

ここで、ミドルウェアを追加および構成できます。

/// <summary>
/// Represents object for the configuring authentication middleware on application startup
/// </summary>
public class AuthenticationStartup : INopStartup
{
    /// <summary>
    /// Add and configure any of the middleware
    /// </summary>
    /// <param name="services">Collection of service descriptors</param>
    /// <param name="configuration">Configuration of the application</param>
    public void ConfigureServices(IServiceCollection services, IConfiguration configuration)
    {
        //add data protection
        services.AddNopDataProtection();

        //add authentication
        services.AddNopAuthentication();
    }

    /// <summary>
    /// Configure the using of added middleware
    /// </summary>
    /// <param name="application">Builder for configuring an application's request pipeline</param>
    public void Configure(IApplicationBuilder application)
    {
        //configure authentication
        application.UseNopAuthentication();
    }

    /// <summary>
    /// Gets order of this startup configuration implementation
    /// </summary>
    public int Order => 500; //authentication should be loaded before MVC
}

/// <summary>
/// SQL Server specific extension method for Microsoft.EntityFrameworkCore.DbContextOptionsBuilder
/// </summary>
/// <param name="optionsBuilder">Database context options builder</param>
/// <param name="services">Collection of service descriptors</param>
public static void UseSqlServerWithLazyLoading(this DbContextOptionsBuilder optionsBuilder, IServiceCollection services)
{
    var nopConfig = services.BuildServiceProvider().GetRequiredService<NopConfig>();

    var dataSettings = DataSettingsManager.LoadSettings();
    if (!dataSettings?.IsValid ?? true)
        return;

    var dbContextOptionsBuilder = optionsBuilder.UseLazyLoadingProxies();

    if (nopConfig.UseRowNumberForPaging)
        dbContextOptionsBuilder.UseSqlServer(dataSettings.DataConnectionString, option => option.UseRowNumberForPaging());
    else
        dbContextOptionsBuilder.UseSqlServer(dataSettings.DataConnectionString);
}

ステップ15.ビューの移行


子アクションは使用されなくなりました。代わりに、ASP.NET Coreは、非同期で呼び出される新しい強力なツールであるViewComponentsを提供します。

ViewComponentから文字列を取得する方法：

/// <summary>
/// Render component to string
/// </summary>
/// <param name="componentName">Component name</param>
/// <param name="arguments">Arguments</param>
/// <returns>Result</returns>
protected virtual string RenderViewComponentToString(string componentName, object arguments = null)
{   
    if (string.IsNullOrEmpty(componentName))
        throw new ArgumentNullException(nameof(componentName));

    var actionContextAccessor = HttpContext.RequestServices.GetService(typeof(IActionContextAccessor)) as IActionContextAccessor;
    if (actionContextAccessor == null)
        throw new Exception("IActionContextAccessor cannot be resolved");

    var context = actionContextAccessor.ActionContext;

    var viewComponentResult = ViewComponent(componentName, arguments);

    var viewData = ViewData;
    if (viewData == null)
    {
        throw new NotImplementedException();       
    }

    var tempData = TempData;
    if (tempData == null)
    {
        throw new NotImplementedException();       
    }

    using (var writer = new StringWriter())
    {
        var viewContext = new ViewContext(
            context,
            NullView.Instance,
            viewData,
            tempData,
            writer,
            new HtmlHelperOptions());

        // IViewComponentHelper is stateful, we want to make sure to retrieve it every time we need it.
        var viewComponentHelper = context.HttpContext.RequestServices.GetRequiredService<IViewComponentHelper>();
        (viewComponentHelper as IViewContextAware)?.Contextualize(viewContext);

        var result = viewComponentResult.ViewComponentType == null ? 
            viewComponentHelper.InvokeAsync(viewComponentResult.ViewComponentName, viewComponentResult.Arguments):
            viewComponentHelper.InvokeAsync(viewComponentResult.ViewComponentType, viewComponentResult.Arguments);

        result.Result.WriteTo(writer, HtmlEncoder.Default);
        return writer.ToString();
    }
}

HtmlHelperを使用する必要はもうありません。ASP.NETCoreには、多数のヘルパータグ関数（タグヘルパー）が組み込まれています。アプリケーションが実行されると、サーバー側のRazorエンジンによって処理され、最終的に標準のhtml要素に変換されます。これにより、アプリケーション開発が大幅に簡素化されます。そしてもちろん、独自のタグヘルパーを実装することもできます。

を使用して設定とサービスを許可する代わりに、ビューで依存性注入を使用し始めましたEngineContext。

したがって、ビューの移行に関する主なポイントは次のとおりです。

  • 変換Views/web.config Views/_ViewImports.cshtml-名前空間のインポートと依存関係の挿入に使用されます（injection）。このファイルはRazor、関数やセクションの定義など、他の関数をサポートしていません。
  • 変換namespaces.addへ@using
  • メインアプリケーション構成への設定の転送
  • Scripts.RenderそしてStyles.Render存在しません。出力へのリンクlibmanまたはBundlerMinifier

結論として


私たちの経験から、大規模なWebアプリケーションの移行プロセスは非常に時間がかかる作業であり、落とし穴なくして実行することは困難です。最初の安定バージョンがリリースされたらすぐに新しいフレームワークに切り替えることを計画しましたが、すぐに完了することができませんでした。その時点ではまだ.NET Coreに転送されていない、特にEntityFrameworkに関連する重要な機能がいくつかありました。したがって、次のリリースは、.NET Frameworkに依存する.NET Coreアーキテクチャである混合アプローチを使用して最初にリリースする必要がありました。

.NET Core 2.1のリリース後、プロジェクトを完全に適応させることができました。そのときまでに、新しいアーキテクチャに取り組んでいる安定したソリューションがありました。残っているのは、一部のパッケージを置き換えてEF Coreで作業を書き直すことだけでした。したがって、新しいフレームワークへの完全な移行には数か月の作業が必要でした。GitHubのリポジトリ
