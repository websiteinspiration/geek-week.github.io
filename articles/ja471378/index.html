プログラミングはどこから始まりますか？



むかしむかし、lsFusionビジネスアプリケーションを開発するための高レベルのプラットフォームを開発しました。残された唯一のものはそれを書いてくれる人を見つけることでした。求人サイトに求人情報を掲載することにしました。しかし、疑問が生じました：ポジションまたは職業として何を示すか。

複雑さの点では、プラットフォーム言語は、もともとプログラマー向けではなくビジネスユーザー向けに作成されたSQLとそれほど変わりませんでした。そのため、「ビジネスアナリスト」も兼職することとしました。

簡単な例を少し改良する必要がある比較的単純なテストタスクを解決することを申請者に提案しました。一部の人々はそれをしました、しかし怒っている手紙は多くの人からのような憤慨を持って来ました：「あなたはプログラマーまたはビジネスアナリストを探していますか？」

その結果、必要な人を見つけましたが、この経験を分析して、私は思わず疑問に思いました。ビジネス分析はどこで終わり、プログラミングはどこで始まるのでしょうか。

大学を卒業したばかりの頃、すでに大規模なアウトソーシング会社でプログラマーとして働いていた友達がいました。それは大人の方法ですべてだった-プログラマーに加えて、ビジネスアナリストとテスターがいました。それから私は彼らがそこにすべてを構築する方法、特にビジネスアナリストが何をするかに非常に興味を持っていました。彼が彼らとどのようにやり取りしたかという質問に対して、次の回答が得られました。とにかく読んでいないので、すぐに削除します。」彼が少し不誠実で、何かを見ていたことは明らかです。

さて、さまざまな技術タスクと仕様を数多く読んで、私はそのような「ビジネス分析」の主な問題を理解しています。これが、分析とプログラミングの主な違いです。紙に完全に矛盾する条件を記述し、「良いものにしたい」などの絶対的に抽象的な要件を示し、それらを美しくし、作業が完了したと想定できます。プログラミングでは、そのようなトリックは機能しません-すべてが明確でなければなりません。このため、開発者は多くの場合、顧客からの機能の実装を回避し、単に矛盾を指摘し、要件の明確化を要求することができます。そして、実際にプログラマーの仕事をしている場合にのみ、顧客はそれを行うことができます。

プログラマーはプログラムを書かなければならないようです。プログラムと見なすことができるものは何ですか？理論で使用される形式の1つは、チューリングマシンです。これは、アルゴリズムを記述できる最も単純なコンピューターです。これは、プログラムの特性の1つが必須であることを示しています。プログラムには、実行の状態とプロセスがあります。 SQLの主な機能の1つは、宣言的な性質でした。 「プログラム」を記述する必要はなく、SQLで取得したいものを記述する必要があるため、一般の人々がそれを使用すると想定されていました。

子供の頃、チェスをする機会がありました。今では、ポジショナルプレーの基本原則を完全に理解していなかったことを理解しています。しかし、私は多くの前進のためのポジションをうまく計算することができました。私は常にゲームを「オープン」にしようとしましたが、さらにさまざまな組み合わせが原因で勝ちました。実際、私はデバッガを頭の中で動かしていて、ボード上の現在の位置を動かして時計で表すことができました。実際には、単純なコンピューターのようにプレイしました。それにもかかわらず、クラスの最初の年に、私はなんとか最初の成人ランクを完了し、私の年齢でベラルーシ選手権のトップ3に入ることができました。面白いですが、チェスはプログラミングと同じようなスキルを使用します。位置ゲームは建築のデザインです。戦術ゲーム-プログラムのデバッグ。

lsFusionでプログラミングする人々に教えると、命令論理は一般の人には宣言よりも知覚が難しいという結論に達しました。多くの先に進む状況を計算することは、日常生活で人が使用する基本的なスキルではありません。彼は状態を覚えていて、あらゆる段階でそれを変えることができなければなりません。同時に、一人一人が定期的に自分が欲しいものを定式化する必要があります。これは一種の「宣言型プログラミング」です。

宣言は他のツールに存在します。たとえば、Excelでは、数式を使用して特定のロジックを構築できます。Excelの数式の形で書かれたほぼすべての情報システムを目にする機会がありましたが、命令的なVisual Basicコードで少し風味が付けられていました。これらの人々はプログラマーですか、それとも高度なビジネスインテリジェンスですか？

現在開発で使用されている言語とテクノロジーのスタックを検討する場合、（ネットワークプロトコルスタックのように）いくつかのレベルに分けることができます。それぞれに新しいものが追加されました：

  1. アセンブラ。機械指向言語。
  2. C.手続き言語。
  3. C ++。オブジェクト指向プログラミング ..
  4. Java / C＃/ Pythonおよびその他。仮想マシンと管理メモリ。
  5. 1C /アクセス/ SAP NetWeaverなど ここでは、開発者の面倒から、メモリ、ディスク、ネットワークおよびその他の詳細の管理を削除しました。ビジネスロジックのみについて考える必要があります。lsFusionがプラットフォームとして配置されているのは、このレベルのどこかにあります。

次の各テクノロジーは、下位​​レベルのテクノロジーに基づいて記述されています。理論的には、低いレベルで書く人は高いレベルで書くことができるということになります。もちろん、これは彼らがそれを可能な限り効率的に行うという意味ではありません。車を作るエンジニアは、世界で最高のレーサーではありませんが、どういうわけかそれらに乗ることができなければなりません。反対方向では、これは機能しません。

主にこの理由により、原則として、開発者は常に低レベルのテクノロジーを使用する人を尊重し、高レベルのテクノロジーに書き込む人を軽視します。私は、アセンブラー開発者がオープナーを嫌っていた時代を捕らえるほど年をとっていません。しかし、C ++は邪悪なものであると信じていたシシュニコフも覚えています。 C ++開発者の中には、Java開発者をプログラマー不足と見なしている人もいます。なぜなら、彼らは実際にはメモリを管理できず、そのためJavaプログラムには多くの追加メモリが必要だからです。さて、最初の4つのレベルのすべての開発者は確かに1Cプログラマを軽んじています。これはひげを生やした冗談に最もよく反映されています。



ただし、これは、最初の4つのテクノロジーやlsFusionとは異なり、5番目のレベルのテクノロジーが閉鎖されて支払われているためと考えられます。彼らはそれが物事の順序である、流血の企業で使用されています。したがって、エコシステム全体は好きではありませんが、同時にエコシステムの開発者も好きです。

SQLやHTML + CSSなどの高レベルで宣言型の言語があるのは5番目のレベルであることは注目に値します。ちなみに、SQL開発者はSQL開発者に対してより忠実です。おそらく、データベース、バックエンド、およびフロントエンドを同時に処理する同じ1Cプログラマーとは対照的に、それらが一緒に動作し、別々のタスクブロックを解決するという事実が原因です。

公平に言えば、プログラミングだけでなく同様の状況が存在します。たとえば、オートマチックトランスミッションに関して非常に否定的に、機械式ギアボックスのみを使用して運転する人がたくさんいます。彼らは、機械が効率的にギアをシフトしないと信じており、手動でより良くすることができます（これは一般的に当てはまります）。もう1つは、ほとんどの人が、ありふれた利便性のために、効率、信頼性、および燃料消費の増加を犠牲にしても構わないということです。プログラミングと同じように、ギアをすばやく切り替えて必要なペダルを押すための十分な調整機能がないため、メカニックをまったく運転できない人がいます。

オートマチックトランスミッションを搭載したドライバーは、手動モードに切り替えることができることに注意してください。同様に、アセンブリコードをCに埋め込むか、lsFusionのレベルを下げてJavaコードを記述できます。

テクノロジーのレベルが高いほど、宣言性が高く、命令性が低くなります。これらのレベルはスケールとして考えることができ、「マシン」が一番下、「ユーザー」または「ビジネス」が一番上になります。下から上に移動するにつれて、プログラミングはますます少なくなっていると思われます。そしておそらくこの行のどこかに、プログラミングがビジネス分析になる時期があるでしょう。

すでに開催されているプログラマーを探しなかったのはなぜですか？これには少なくともいくつかの理由があります。

第一に、彼らはすでに何らかのテクノロジースタックを使用しており、それらを使って作業しています。比較的革新的な業界にもかかわらず、ほとんどのプログラマーは保守的です（確かにすべてではありません）。彼らは（一時的にであっても）需要のない誇大宣伝なしには未知の技術を学ばないでしょう。誰かが彼が働いている範囲を変えるとき、彼は自動的に「価値を失います」。したがって、あなたが知っていることをあきらめて、何か新しいことを学び始めるのは心理的に難しいです。

第2に、lsFusionプラットフォームでの開発と1Cでの開発は、Java、.Net、Pythonなどの低レベルのプラットフォームでの開発よりも、ビジネス分析により近いものです。他のタスク、問題、アプローチがあります。

しかし、主な理由の1つは経済的です。ベラルーシでは、人々は役人や治安当局者になることを求めていません。ここでは、牧師として働いたり、個人の飛行機で犬を展覧会に連れて行ったり、反対方向に酔ったりすることはできません。窓のドレッシングは歓迎されず、多くの役人が時々投獄され、その後集団農場を率いるために解放されます。また、天然資源で暮らす企業もありませんので、石油会社になるという夢も叶うものではありません。ほとんどすべての市場は十分に小さく、すでに原則としてすでに分割されています。したがって、新しいビジネスマンのための余地もほとんどありません。

このような背景に対して、IT領域は際立っています。これは次の図で最もよくわかります。



パイロットの非常に小さな領域と一部の金融コンサルタントを除外すると、プログラマーは平均して、他の高額な領域の従業員よりも3.5倍多く稼いでいることがわかります。国の従業員と比較すると、その差は7倍になります。

ベラルーシのプログラマーの仕事のもう一つのプラスは安定性です。ローカル市場は非常に小さいため、ほとんどの企業は西部市場で事業を行っています。したがって、原則として、プログラマーのすべての給与は外貨建てです。それが6月に今年の前半に比べて下落がある理由です-ルーブルは高く評価されました。現地通貨が常に世界通貨に対して切り下げられている国では、ドルとユーロの給与は安定性と静けさの鍵と考えられています。国の人々は多くの人から「人生の達人」であると考えられています。彼らはほとんどの市場で高価なセグメントの主な消費者です。多くの親が子供たちをプログラミングサークルに送り、関連部門にとって最高の競争をしています。

「Enter IT」のトレンドがベラルーシで繁栄していることは当然のことです。私たちはこれを利用して、lsFusionでの開発方法を教えることにより、他の職業の人々を雇うことを決心しました。ちなみに、今ではlsFusionの開発者が、元セールスマネージャー、システム管理者、エコノミスト、ERPシステムのコンサルタントなどを担当しています。

ベラルーシ最大の情報源であるメインページに有料広告を掲載しました。これは、現場からのエラーを考慮に入れ、ビジネスインテリジェンスではなく、実務経験のない開発者を探していることを示しています。最高の動機を与えるために、彼らはIT業界に対応する潜在的な給与を明示的に示しました。

必要な人数は非常に少なく、多くの人が必要なので、候補者を何らかの方法で排除する必要があることを理解しました。最も単純なスキームを選択しました。順位の例を取り、それを調整して、試合のスコアが手動ではなく、ユーザーが入力する必要のあるスコアに基づいて決定されるようにする必要があります。各候補者を個別に悩ませたり、時間を無駄にしたりしないようにするために、この問題の解決策が必要であることを発表で直接示しました。

この問題を解決するには、プラットフォームをダウンロードしてインストールし（インストーラーはサイトにありました）、そこにある例のコードを接続し、IDEで修正する必要がありました。私たちのlsFusion開発者の場合、約15分かかります。ソリューションはプラットフォームの追加知識を必要としないため、典型的なプログラマーが例を理解し、ダウンロード、実行、解決するのに最大3時間かかります。すべては既存のロジックとの類推によってそこで行われました。

統計によると、発表は数万人に開かれ、読まれました。テストタスク自体（別のリンクにありました）は数千人に読まれました。プラットフォームは約400人をダウンロードしました（IDEA、Java、PostgreSQLを含む1つの1.5 GBファイルでした）。 40人が何かを送った。これらのうち、約10人が適切な解決策を持っています。これらの人々の中で、私たちは私たちが雇う人を選びました。

この物語で私が理解できなかったのは、人々が欠けていたもの、つまり動機や能力でした。結局、誰かが彼に喜びをもたらさない低給の仕事に取り組んでいるなら、何が彼が問題を理解してそれを解決するために数時間も費やすのを妨げているのでしょう。私たちは最初から高い給与をすでに提供していますが、それは他の言語ですべてのジュニア開発者に支払われることにはほど遠いものです。たぶん、多くの人が「プログラマーのところに行って教えてあげる」という原則に基づいて行動するでしょう。実際、私たちの国では、プログラミングにおいて自己教育がいかに重要であるかを理解せずに、多くの人が受講するあらゆる種類の有料コースをすでに持っています。

しかし、おそらく問題は能力にあります。プログラミングを開始するには、分析的思考という1つの基本的なスキルが必要です。それは私が私将と一般（帰納）を区別する能力を意味し、逆もまた同様です（推論）。この能力は比較的少数の人々で開発されていますが、それがなければ、残念ながら、優れたプログラマーではありません。さらに、何らかの方法で分析的思考を測定し、このスキルを持つ人々の数への依存のグラフを作成すると、誇張表現が得られます。つまり、テクノロジーへの参入のしきい値が低いほど、それをはるかに多くの人々が克服できます。

分析的思考の問題は、それを学ぶことができないことです。それは幼い頃にどこかに形成され、大人はもはや何かを変えることができません。技術科学の試用期間候補を通過しました。しかし、彼は単純なドメインロジックを実装できませんでした。その男は試し、夕方自宅で働きましたが、たとえ話をするだけでよい仕事を与えられるたびに、それに対処することができませんでした。彼が候補者を同時に守ることができた方法は明確ではありません。ただし、これは分析的思考が日常生活で必須ではないことを示している可能性が高くなります。そして、このスキルが特に必要とされない職業はたくさんあります。

彼らがプログラミングへの道を始めるためのテクノロジーを尋ねたとき、私は答えます：SQLから始めます。いずれにしても、彼の知識は害を及ぼすことはなく、基本的な機能（SELECT / JOIN / UNION）を学ぶのに1日もかかりません。次に、インターネットから問題を解決する必要があります。それはうまくいきません-したがって、プログラミングから始めるべきではありません。何かが解決できる場合は、少なくとも1CまたはlsFusionがプログラマーになることができます。そして、それはどうなるのか。

結論


もちろん、記事のタイトルからの質問は修辞的です。プログラミングの概念は主観的であり、特定の人の見方に依存します。ある程度、すべての人がプログラミングに従事しています。たとえば、床暖房用のサーモスタットの設定を単純に行う場合や、Googleドキュメントで数式を使用してテーブルを作成する場合などです。ただし、テクノロジーのレベルが異なれば、参入基準も異なります。テクノロジーのレベルが高いほど、プログラミングできる人が増えます。このプロセスでの真のプログラミングは本当に小さくなっています。私は、これが他の開発者にとって侮辱の対象にならないようにしたかっただけです。
