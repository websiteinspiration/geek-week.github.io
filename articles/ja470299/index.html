効率的なSignalR接続管理

こんにちは、Habrahabr。私は現在、SignalRライブラリに基づくチャットエンジンに取り組んでいます。リアルタイムアプリケーションの世界での魅力的なイマージョンプロセスに加えて、多くの技術的な課題にも直面しました。そのうちの1つについて、この記事でお話ししたいと思います。

前書き


SignalRとは何か-それはオーバーファサードのようなものだWebSocketを、ロングポーリング、サーバー送信イベントの技術。このファサードのおかげで、これらのテクノロジーのいずれかと均一に作業でき、詳細について心配する必要がありません。さらに、ロングポーリングテクノロジーのおかげで、IE-8など、何らかの理由でWebソケットで作業できないクライアントをサポートできます。ファサードは、高レベルのRPCベースのAPIで表されます。さらに、SignalRは、API用語でグループと呼ばれる「パブリッシャー-サブスクライバー」の原則に従って通信を構築することを提案しています。これについては後で説明します。

挑戦


おそらく、プログラミングで最も興味深いことは、非標準の問題を解決する能力です。そして今日、これらのタスクの1つを指定し、その解決策を検討します。

スケーリングのアイデアが開発された時代、そして最初に水平的な時代の主な課題は、複数のサーバーを用意する必要があることです。また、指定されたライブラリの開発者はこの呼び出しにすでに対応しています。解決策の説明はMSDNにあります。つまり、パブリッシャー/サブスクライバーの原理を使用して、サーバー間の呼び出しを同期することが提案されています。各サーバーは共有バスにサブスクライブし、このサーバーから送信されるすべてのコマンドは最初にバスに送信されます。さらに、このコマンドはすべてのサーバーに適用され、その後クライアントにのみ適用されます。サーバーに



接続されている各クライアントには独自の一意の接続識別子-ConnectionIdがあることに注意することが重要です-そして、すべてのメッセージは最終的にこの識別子を使用して対処されます。したがって、各サーバーはこれらの接続を保存します。

ただし、不明な理由により、SignalRライブラリAPIはこのデータへのアクセスを提供していません。そしてここでは、これらのつながりへのアクセスという非常に深刻な問題に直面しています。これが私たちの挑戦です。

なぜ接続する必要があるのか


前述のように、SignalRはパブリッシャーサブスクライバーモデルを提供します。ここでは、メッセージルーティングの単位は、ConnectionIdではなくグループです。グループは接続の集まりです。メッセージをグループに送信することにより、このグループにあるすべてのConnectionIdにメッセージを送信します。グループを作成すると便利です。クライアントをサーバーに接続するときは、AddToGroupAsync APIメソッドを呼び出すだけです。

public override async Task OnConnectedAsync()
        {
            foreach (var chat in _options.Chats)
                await Groups.AddToGroupAsync(ConnectionId, chat);

            await Groups.AddToGroupAsync(ConnectionId, Client);
        }

そして、どのようにグループを去るのですか？開発者はAPIメソッドRemoveFromGroupAsyncを提供します：

public override async Task OnDisconnectedAsync(Exception exception)
        {
            foreach (var chat in _options.Chats)
                await Groups.RemoveFromGroupAsync(ConnectionId, chat);
            
            await Groups.RemoveFromGroupAsync(ConnectionId, Client);
        }

データ単位はConnectionIdであることに注意してください。ただし、ドメインモデルの観点からは、ConnectionIdは存在しませんが、クライアントは存在します。この点で、クライアントをConnectionId配列に、またはその逆にマッピングする編成は、指定されたライブラリーのユーザーに割り当てられます。

グループを離れるときに必要なすべてのConnectionIdクライアントの配列です。ただし、そのような配列は存在しません。自分で整理する必要があります。水平方向にスケーリングされたシステムの場合、タスクははるかに興味深いものになります。この場合、一部の接続を1つのサーバーに配置し、残りを他のサーバーに配置できます。

クライアントを接続にマップする方法


このトピックには、MSDNの完全なセクションがあります。以下の方法が検討のために提案されています。

  • インメモリストレージ
  • "ユーザー・グループ"
  • 永続的な外部ストレージ

接続を追跡するには？
OnConnectedAsyncおよびOnDisconnectedAsyncハブメソッドを使用して接続を追跡できます。

すぐに、スケーリングをサポートしないオプションは考慮されないことに注意します。これには、接続をサーバーのメモリに保存するオプションが含まれます。存在する場合、他のサーバー上のクライアント接続へのアクセスはありません。外部永続ストレージに保存するオプションは、非アクティブな接続のクリーニングの問題を含む、その欠点に関連しています。このような接続は、サーバーのハードリブート時に発生します。これらの接続の検出とクリーニングは簡単な作業ではありません。

上記のオプションの中で、「ユーザーグループ」オプションは興味深いものです。シンプルさは確かにその利点に適用されます-ライブラリ、リポジトリは必要ありません。同様に重要なのは、この方法の単純さの結果である信頼性です。

しかし、Redisはどうでしょうか？
ちなみに、Redisを使用して接続を保存することも悪い選択肢です。これは、メモリ内のデータを整理するという深刻な問題です。一方では、キーはクライアントであり、もう一方はグループです。

"ユーザー・グループ"


「ユーザーグループ」とは何ですか？これは、SignalR用語のグループであり、1つのクライアントのみがクライアントになることができます-彼自身。これにより、2つのことが保証されます。

  1. メッセージは1人だけに配信されます
  2. メッセージはすべての人間のデバイスに配信されます

これはどのように役立ちますか？私たちの課題は、クライアントをグループから離れることの問題を解決することであることを思い出させてください。必要なのは、グループを1つのデバイスから残して、残りのデバイスもサブスクライブを解除することですが、出口を開始したクライアントを除いて、このクライアントの接続リストはありませんでした。

「ユーザーグループ」は、この問題を解決するための最初のステップです。2番目のステップは、クライアント上に「ミラー」を構築することです。はい、はい、ミラー。

"鏡"


クライアントからサーバーに送信されるコマンドのソースはユーザーアクションです。メッセージの投稿-サーバーにコマンドを送信します。

this.state.hubConnection
      .invoke('post', {message, group, nick})
      .catch(err => console.error(err));

また、新しい投稿についてグループのすべてのクライアントに通知します。

public async Task PostMessage(PostMessage message)
        {
            await Clients.Group(message.Group).SendAsync("message", new
            {
                Message = message.Message,
                Group = message.Group,
                Nick = ClientNick
            });
        }

ただし、多くのコマンドは、すべてのデバイスで同期的に実行する必要があります。これを達成する方法は？接続の配列を用意し、特定のクライアントで接続ごとにコマンドを実行するか、以下で説明する方法を使用します。チャットを終了して、この方法を検討してください。

クライアントから到着したチームはまず特別な方法で「ユーザーグループ」に移動し、サーバーにリダイレクトします。「鏡」したがって、サーバーがデバイスの登録を解除するのではなく、デバイス自体が登録解除を要求します。

サーバーチャットの登録解除コマンドの例を次に示します。

public async Task LeaveChat(LeaveChatMessage message)
        {
            await Clients.OthersInGroup(message.Group).SendAsync("lost", new ClientCommand
            {
                Group = message.Group, Nick = Client
            });
            await Clients.Group(Client).SendAsync("mirror", new MirrorChatCommand
            {
                Method = "unsubscribe",
                Payload = new UnsubscribeChatMessage
                {
                    Group = message.Group
                }
            });
        }

public async Task Unsubscribe(UnsubscribeChatMessage message)
        {
            await Groups.RemoveFromGroupAsync(ConnectionId, message.Group);
        }

そしてここにクライアントコードがあります：

connection.on('mirror', (message) => {
          connection
            .invoke(message.method, message.payload)
            .catch(err => console.error(err));
        });

ここで何が起こっているのかを詳しく見てみましょう：

  1. クライアントは登録解除を開始します-"leave"コマンドをサーバーに送信します
  2. サーバーは、「ミラー」の「ユーザーグループ」に「登録解除」コマンドを送信します
  3. メッセージはすべてのクライアントデバイスに配信されます。
  4. サーバー上の指定された方法を使用して、クライアント上のメッセージがサーバーに送り返されます。
  5. 各サーバーで、クライアントはグループから登録解除されます

その結果、すべてのデバイス自体が、それらが接続されているサーバーから登録解除されます。それぞれが自分の購読を解除し、何も保存する必要はありません。サーバーのハードリブートが発生しても問題は発生しません。

では、なぜ接続する必要があるのでしょうか。


クライアント上に「ユーザーグループ」と「ミラー」があると、接続を操作する必要がなくなります。読者の皆さん、これについてどう思いますか？コメントであなたの意見を共有してください。
