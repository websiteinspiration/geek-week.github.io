Azure DevOpsで自動テストパイプラインを構築する

最近、私はDevOpsの世界であまり人気のない獣、Azure DevOpsパイプラインに出くわしました。このトピックに関する明確な指示や記事がないことをすぐに感じましたが、それが何に関連しているかはわかりませんが、Microsoftはツールの普及という点で明らかに取り組むべきことを持っています。今日は、Azureクラウド内で自動テスト用のパイプラインを構築します。

したがって、タスク：
WIX上のプロジェクトからアセンブルされた同じAzure DevOpsを使用して構築するソフトウェアがあります。興味があればこのツールについて書いていきます。実際、これは標準のInstallShieldに代わるWindowsインストーラーを構築するためのより自動化された方法です。したがって、私たちのソフトウェアは、特定のsetup.exeというアーティファクトを正常にビルドして生成し、それがアプリケーションをWindowsシステムに配置します。マージの前にブランチを良いか悪いかを検討するために、このアプリケーションを本番と同様の仮想マシンに入れ、そこでテストチームが準備した自動テストをコピーして実行し、結果を収集する必要があります。すべてGitLabで、...を介してのみ。

テストを実行する仮想化環境として、AzureサブスクリプションのエンティティであるAzure DevTest Labsを明らかに使用しています。

1.クラウド側の統合


まず、DevTest LabsとAzure DevOpsを統合する必要があります。そのためには、いくつかのサービスプリンシパルが必要です。実際には、クラウド内のパイプラインに移動して、自分でリソースを作成または削除できるサービスアカウントが必要です。

サブスクリプションに入り、Azure Active Directoryサービス



を見つけます。アプリの登録を見つけて[新規登録]をクリックすると、サービスプリンシパルが作成されます。作成時に選択する設定については説明しません。これは、サブスクリプションによって異なる場合があります。



次に、サービスディレクターに権限を与える必要があります。これを行うには、サブスクリプション、キー付きのアイコンに移動します。サブスクリプションを選択してください。



次に、[アクセス制御]で[役割の割り当て]をクリックし、作成した名前で検索してこのアカウントを探します。私たちは貢献者の役割を与えます、これで十分です。



次に、Azure ADのサービスプリンシパルに戻り、そのプロパティを開きます。後で、そこにあるすべてのIDが必要になるので、それらを保存します。

ここで、ポータル設定が終了し、Azure DevOpsに進みます。

2. Azure DevOps側の統合


まず、プロジェクト設定に移動して、サービス接続を選択します。Azure Resource Managerタイプの新しい要素を作成します。



ここで、記録したすべてのIDが必要です。フルバージョンのサービス接続ダイアログを使用するをクリックします。サービスプリンシパルから受け取ったすべてのデータを入力します。[確認]をクリックし、問題がなければ接続を維持します。これで、パイプラインはそれを使用してクラウドに接続できます。

5.パイプライン構成


最も重要で興味深い仮想マシンの作成から始めましょう。このために、これらすべての統合とテンプレートを作成しました。 Azure RMサブスクリプションで、段落2で構成したサービス接続を選択します。次に、使用可能なラボ環境がポップアップします。次に、生成したjsonを選択し、いくつかの必須変数を定義します。車のユーザー名とパスワードは直接または変数で設定できますが、そこに何を書いても、それが機能するかどうかはまったくわかりません。これらのクレジットでは車に乗れませんでした。主なことは、常に可能なようにマシンの名前を設定することですユニーク。これには、ビルド環境変数を使用します。



次に、別の重要なポイントを設定します。車が離陸した後も、何らかの形でそのパラメーターを知る必要があります。それは、私たちではなく、ペイラインよりも優れています。これを行うには、GetLabVMParams.ps1などのスクリプトを作成して、プロジェクトに配置します。 MicrosoftのWebサイトでスクリプトのテキストを取得しましたが、私の環境に合わせて少し修正しました。彼はPublicIPとFQDNマシンを取った。どちらも持っていませんが、入手が難しいPrivateIPがあるので追加しました。

Param( [string] $labVmId)

$labVmComputeId = (Get-AzureRmResource -Id $labVmId).Properties.ComputeId

# Get lab VM resource group name
$labVmRgName = (Get-AzureRmResource -Id $labVmComputeId).ResourceGroupName

# Get the lab VM Name
$labVmName = (Get-AzureRmResource -Id $labVmId).Name

# Get lab VM public IP address
# $labVMIpAddress = (Get-AzureRmPublicIpAddress -ResourceGroupName $labVmRgName -Name $labVmName).IpAddress

# Get lab VM FQDN
# $labVMFqdn = (Get-AzureRmPublicIpAddress -ResourceGroupName $labVmRgName -Name $labVmName).DnsSettings.Fqdn

# Get lab VM private IP address
$VmNetworkdetails= (((Get-AzureRmVM -ResourceGroupName $labVmRgName -Name $labVmName).NetworkProfile).NetworkInterfaces).Id
$nicname = $VmNetworkdetails.substring($VmNetworkdetails.LastIndexOf("/")+1)
$labVMnetwork = (Get-AzureRmNetworkInterface -Name $nicname -ResourceGroupName $labVmRgName)|Select-Object -ExpandProperty IPConfigurations 
$labVMIpAddress = $labVMnetwork.PrivateIpAddress

# Set a variable labVmRgName to store the lab VM resource group name
Write-Host "##vso[task.setvariable variable=labVmRgName;]$labVmRgName"

# Set a variable labVMIpAddress to store the lab VM Ip address
Write-Host "##vso[task.setvariable variable=labVMIpAddress;]$labVMIpAddress"

# Set a variable labVMFqdn to store the lab VM FQDN name
Write-Host "##vso[task.setvariable variable=labVMFqdn;]$labVMFqdn"

Write-Output $labVMIpAddress

Set-Item wsman:\localhost\client\trustedhosts * -Force
