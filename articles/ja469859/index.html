メソッド受信者の定義におけるポインターと値のセマンティクス

新しいデータ型を作成することは、すべてのプログラマーの作業の重要な部分です。ほとんどの言語では、型定義はそのフィールドとメソッドの説明で構成されています。 Golangでは、これに加えて、新しいタイプのメソッドで使用する受信者のセマンティクスを決定する必要があります。値（値）またはポインター（ポインター）。ほとんどの場合、プログラムは受信者の任意のセマンティクスで機能するため、一見すると、この決定は二次的なように思えるかもしれません。したがって、多くの人はこのポイントをスキップしてコードを記述し、メソッドの受信者のセマンティクスの影響を受ける最後まで理解していません。そしてそれを理解するには、Golangの動作を少し深く理解する必要があります。

小さな例を考えてみましょう。1つのNameフィールドとsayHello（人の文字列） methodで猫の構造を定義します。以下では、メソッドによって、特定の型に関連付けられた関数、オブジェクトをメソッドを持つ変数、メソッドの受信者をメソッドの説明でfuncという語の後の括弧内に示されている変数と呼びます。

type cat struct {
	Name string
}

func (c *cat) sayHello(person string) {
	fmt.Println(fmt.Sprintf("Meow, meow, %s!", person)
}

cat へのポインターを定義し、そこからNameフィールドを要求すると、フィールドがnilから呼び出されるため、明らかにエラーが発生します。


var c *cat // c=nil
fmt.Println(c.Name)
//panic: runtime error: invalid memory address or nil pointer dereference

ただし、sayHello（）メソッドが同じ変数で呼び出されても、エラーは発生しません。


var c *cat // c=nil
c.sayHello(“Human”)
//Meow, meow, Human!

https://play.golang.org/p/EMoFgKL1HEi

なぜこの例ではnilがメソッド呼び出しを持つことができ、言語自体のアーキテクチャーの観点からこれをどのように説明するのですか？これは、Goのメソッドがシンタックスシュガー、つまり受信者の引数の1つを持つ関数のラッパーであるために可能になります。c.sayHello（「Human」）メソッドが呼び出されると、（* cat）.sayHello（c、s）構成（https://play.golang.org/p/X9leJeIvxcA）が実際に呼び出されます。上記の例からnilメソッドを呼び出すことにより、実際には引数にnilを指定して関数を呼び出しますが、これはすでに非常に正常な状況です。したがって、Go nilでは、メソッドの正しい受信者です。

メソッドレシーバーは実際には引数であるため、メソッドレシーバーのセマンティクス「値」または「ポインター」を使用するための推奨事項は、関数引数の推奨事項に似ています。次に、それらはGoの基本ルールから推論されます。引数は常に値によって関数に渡されます。これは、関数への引数の転送は、そのコピーを通じて行われることを意味します。関数が構造を入力として受け入れる場合、この構造の完全なコピーがその内部に入ります。オブジェクトへのポインタを取る場合、新しい変数には同じオブジェクトへのポインタが付属します。これは、（関数内で引数のアドレスを関数に渡す前に、変数のアドレスを比較することによって理解することができhttps://play.golang.org/p/oc2ssC_Irs8、https://play.golang.org/p/FeQa2HUdX0a）。

リンク受け渡しを使用する場合：

  • 大きな構造物。ポインターは1マシンワード（32、64ビットはシステムによって異なります）を占めます。したがって、レシーバーでポインターを使用してメソッドを呼び出す場合、値で渡す場合のように、ポインターをコピーする方がオブジェクト全体をコピーするよりも安価です。
  • 呼び出されたメソッドがオブジェクト自体のデータを変更する場合。受信者が参照渡しされると、メソッドは間接的に変更を加えることにより、呼び出し元オブジェクトの状態に影響を与える可能性があります。値渡しでは不可能です。

値の転送を使用する場合：

  • 数値、文字列、ブールなどの単純な組み込み型の場合。ポインタを使用する場合、このタイプのオブジェクトとほぼ同じ量のメモリが使用され、以下で説明するように、ガベージコレクタによるメンテナンスのコストが増加します。
  • スライス、およびその他の参照タイプ：マップとチャネル-ポインタをとっても意味がありません。それら自体はすでにポインタです。
  • マルチスレッドでは、参照渡しとは異なり、値渡しは安全です。
  • 小さな構造物用。このような場合、値による送信の方が効率的です。これは、メソッドの内部データがスタックの別のフレームに配置されるためです。関数を終了すると、そのフレームはクリアされます。ポインターに沿って何かをかき回すと、このデータをスタックからヒープに転送します。そこから、このデータを他の関数で使用できる場合があります。ヒープを増やすと、ガベージコレクタに追加の負荷がかかります。ガベージコレクタの操作により、プログラムの速度が平均25％低下します。値単位の転送を使用する場合、データはスタックに残り、ガベージコレクターの追加の作業は必要ありません。
