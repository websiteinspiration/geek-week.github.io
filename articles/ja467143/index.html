Flutterアプリケーションでの不純物の使用



私はゲーム開発会社で働いていますが、最近は趣味でモバイルアプリケーションの開発に興味を持っています。したがって、Flutterフレームワークを使用したモバイルアプリケーションの開発に特化した会議に友人が招待してくれたとき、私は喜びに同意しました。そこでFlutterを実際に試してみて、このテクノロジーを確実に研究することにしました。開発に必要なダートは私には不慣れだったので、言語学習も必須プログラムに含まれていました。コード例に少し触れたところ、Dartは理解しやすく簡潔な言語で、本当に気に入っていました。私が気に入ったダートの特徴の一つは不純物です。

不純物とは何ですか？


最初の知人のために、ウィキペディアからの抜粋をあげます。
Admixture （英語ミックスイン）は、明確に定義された動作を実装するプログラミング言語（通常はクラスまたはモジュール）の要素です。他のクラスの動作を明確にするために使用され、独立して使用されるオブジェクトを生成することを意図していません。
Dartでは、そのような構成は名前の前の単語mixinによって定義されます。

上記の定義は、他のクラスに追加できる論理的に分離された動作の機能を取得することを意味します。

多重継承の可能性を思い起こさせますか？はい、しかし私には不純なアプローチの方が良いようです。そして、なぜ、例を見てみましょう。

抽象クラスAnimalがあるとします。

abstract class Animal {
    void voice();
}

また、Animalクラスを実装するCatおよびDogクラス。

class Cat extends Animal {
    void voice() {
        print(“Meow”);
    }
}

class Dog extends Animal {
    void voice() {
        print(“Woof”);
    }
}

そして突然必要になりました...


はい、はい、私は個人的に開発中にそのようなことはありません。

多重継承の場合も同様です。

class CatDog extends Cat, Dog {
}

しかし、音声コマンドをペットに向けて発声するとすぐに、非常に不愉快な状況が発生します。音声メソッドは両方のクラスで実装されているため、彼が正確に何を答えればよいか明確ではありません。この状況は広く知られており、「ダイヤモンドの問題」または「死のダイヤモンド」と呼ばれています。


不純物による販売の場合は遭遇しません。

lass Animal {
   void voice() {
      print(“Hakuna Matata!”);
   }
}

mixin Cat {
   void voice() {
      print(“Meow”);
   }
}

mixin Dog {
   void voice() {
      print(“Woof”);
   }
}

class CatDog extends Animal with Cat, Dog {
}

では、音声コマンドを使用するとどうなるでしょうか。この場合-ウーフ、そしてあなたがすでに理解したように、それは不純物の追加の順序に依存します。これは、それらを追加するのが並列ではなく順次であるために起こります。

特にDart 2.1で導入された機能をマークするために、Animalクラスを実装しました。それ以前は、Objectから継承するクラスにのみ不純物を追加できました。バージョン2.1以降、相続人へのクラスの追加が実装されています。

このメカニズムにより、関数の共通部分を作成して使用することが非常に便利になり、コードの重複の問題が解決されます。例を考えてみましょう。

abstract class Sportsman {
    void readySteadyGo();
}

mixin SkiRunner {
    void run() {
        print(“Ski, ski, ski”);
    }
}

mixin RifleShooter {
    void shot() {
        print(“Pew, pew, pew”);
    }
}

class Shooter() extends Sportsman with RifleShooter {
    void readySteadyGo() {
        shot();
    }
}

class Skier() extends Sportsman with SkiRunner {
    void readySteadyGo() {
        run();
    }
}

class Biathlete() extends Sportsman with SkiRunner, RifleShooter {
    void readySteadyGo() {
        run();
        shot();
    }
}

ご覧のとおり、重複したすべてのコードを不純に分散し、各実装で必要なコードのみを使用しました。

開発中に、すべてのクラスによる組み込みのために、いずれかの不純物の機能を公開してはならない状況が発生する可能性があります。また、これらの制限を課すことを可能にするメカニズムも利用できます。クラス名とともに広告の不純物に関するこのキーワード。したがって、不純物の使用は、指定されたクラスまたはそれから継承されたクラスを実装するクラスのみに制限します。

例えば：

class A {
}

abstract class B {
}

mixin M1 on A {
}

mixin M2 on B {
}

次に、同様のクラスを宣言できます。

class C extends A with M1 {
}
class D implements B with M2 {
}

しかし、次のようなものを宣言しようとするとエラーが発生します。
class E with M1, M2 {
}

Flutterアプリケーションでの使用


上で述べたように、不純物を使用すると、コードの重複を取り除き、再利用できる個別の論理パーツを作成できます。しかし、これはFlutterに一般的にどのように適用されますか？すべてがアトミックであり、特定の機能を担当するウィジェットに分割されていますか？例として、プロジェクトが多くのウィジェットを使用し、特定の内部状態に応じて表示が変化する状況をすぐに想像しました。この例をBLoCアーキテクチャで検討し、rxDartライブラリのエンティティを使用します。

フローコントローラーを閉じるためのインターフェースが必要です。

/// Interface for disposable objects
abstract class Disposable {
    void dispose();
}

状態サポートを実装するための混合。

/// Mixin for object which support state
mixin StateProvider implements Disposable {
    static const NONE_STATE = "None";

    final _stateController = BehaviorSubject<String>(seedValue: NONE_STATE);
    Observable<String> get stateOut => _stateController.stream;

    String get currentState => _stateController.value;

     void setState(String state) {
        _stateController.sink.add(state);
    }

    @override
    void dispose() {
        _stateController.close();
    }
}

ウィジェットの状態を制御する論理部分。メソッドを呼び出して状態を設定し、3秒後に別の状態に変更します。

/// Example BLoC
class ExampleBloc implements Disposable with StateProvider {
    static const EXAMPLE_STATE_1 = "EX1";
    static const EXAMPLE_STATE_2 = "EX2";

    Timer _timer;

    void init() {
        setState(EXAMPLE_STATE_1);

        _timer = new Timer(const Duration(seconds: 3), () {
            _timer = null;
            setState(EXAMPLE_STATE_2);
        });
    }

    @override
    void dispose() {
        if (_timer != null) {
            _timer.cancel();
            _timer = null;
        }
    }
}

そして、状態変化に応答するウィジェット自体。Dependency Injectionを使用して目的の論理コンポーネントを取得することを想像してください。

class ExampleWidget extends StatelessWidget {
    final bloc = di<HomePageBloc>();

    @override
    Widget build(BuildContext context) {
        return StreamBuilder(
            stream: bloc.stateOut,
            builder: (BuildContext context, AsyncSnapshot<String> snapshot) {
                // build widget by state
            },
        );
    }
}

必要に応じて、記述内容を組み合わせて追加することもでき、インターフェースを使用したビルダーメソッドの実装のみが必要ですが、これはすでに不要であると思われます。これは、例にすぎないため、プロジェクトにそのような単純なウィジェットが多数あるとは考えられないためです。それにもかかわらず、状態サポート機能の論理部分は、この混合を使用して任意のBLoCに簡単に追加されます。

結論


不純物を使用するメカニズムは、かなり興味深く、柔軟な開発ツールのように思えました。これにより、シンプルで理解しやすく、便利なアーキテクチャを構築できます。個人的には、このキットに含まれるこのツールは明らかに不必要ではないと判断しました。あなたにも役立つことを願っています。
