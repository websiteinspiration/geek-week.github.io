Injeção de dependência - mais fácil do que parece?

Olá Habr!



Estamos nos preparando para lançar a segunda edição do lendário livro de Mark Siman, “







Injeção de dependência no .NET”. Portanto, hoje decidimos atualizar brevemente o tópico injeção de dependência para especialistas em .NET e C #, e oferecemos uma tradução do artigo de Graham Downs, onde esse paradigma é considerado no contexto de inversão de controle (IoC) e uso de contêineres.



A maioria dos programadores provavelmente conhece o fenômeno da Injeção de Dependência, mas nem todo mundo entende o significado disso. Você provavelmente teve que lidar com interfaces e contêineres e, às vezes, trabalhar com eles o levava a um beco sem saída. Por outro lado, você pode ter acabado de ouvir algo sobre injeção de dependência e aberto este artigo porque deseja entender melhor a essência deles. Neste artigo, mostrarei como é simples o conceito de injeção de dependência e quão poderoso ele pode realmente ser.



A injeção de dependência é uma abordagem independente que pode ser usada por si só. Por outro lado, essa abordagem pode ser aplicada com interfaces e com contêineres para injeção de dependência / inversão de gerenciamento (DI / IoC). Ao aplicar a injeção de dependência nesse contexto, você pode encontrar alguma confusão, que eu experimentei inicialmente.



Ao longo de minha carreira (me especializei em desenvolvimento em Net / C #), usei a injeção de dependência em sua forma mais pura. Ao mesmo tempo, implementei o DI sem recorrer a contêineres ou inversão de controle. Tudo mudou recentemente quando recebi uma tarefa em que o uso de contêineres era indispensável. Então duvidei fortemente de tudo o que sabia antes.



Tendo trabalhado nesse estilo por várias semanas, percebi que contêineres e interfaces não complicam a implementação de dependências, mas, pelo contrário, expandem as capacidades desse paradigma.


(É importante observar aqui: interfaces e contêineres são usados ​​apenas no contexto de injeção de dependência. A injeção de dependência pode ser implementada sem interfaces / contêineres, mas, em essência, o único objetivo de interfaces ou contêineres é facilitar a implementação de dependências).



Este artigo mostrará como a injeção de dependência é feita, com e sem interfaces e contêineres. Espero que, após a leitura, você entenda claramente o princípio da injeção de dependência e seja capaz de tomar decisões informadas sobre quando e onde recorrer ao uso de interfaces e contêineres ao implementar dependências.



Treinamento



Para entender melhor a injeção de dependência em sua forma mais pura, vejamos um aplicativo de exemplo escrito em C #.



Primeiro, observe que esse aplicativo pode ser gravado sem nenhuma injeção de dependência. Em seguida, introduziremos a injeção de dependência, adicionando um recurso simples de registro.



À medida que você progride, verá como os requisitos para o registro estão se tornando gradualmente mais complicados e os satisfazemos usando injeção de dependência; enquanto a área de responsabilidade da classe Calculator



é minimizada. A injeção de dependência também nos impede de modificar a classe Calculator



sempre que quisermos alterar o dispositivo de registro.



inscrição



Considere o seguinte código. Ele foi escrito para um aplicativo de calculadora simples que pega dois números, um operador e exibe o resultado. (Esse é um aplicativo de trabalho simples para a linha de comando, portanto, é fácil reproduzi-lo como um Aplicativo de Console C # no Visual Studio e colar o código lá se você quiser seguir o exemplo. Tudo deve funcionar sem problemas.)



Temos uma classe Calculator



e uma classe principal Program



usando isso.



Program.cs: 



using System;
using System.Linq;

namespace OfferZenDiTutorial
{
    class Program
    {
        static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            var calc = new Calculator();
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }

        private static float GetNumber(string message)
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write(message);
                var input = Console.ReadLine();
                isValid = float.TryParse(input, out var number);
                if (isValid)
                    return number;

                Console.WriteLine("Please enter a valid number. Press ^C to quit.");
            }

            return -1;
        }

        private static char GetOperator()
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write("Please type the operator (/*+-) > ");
                var input = Console.ReadKey();
                Console.WriteLine();
                var operation = input.KeyChar;
                if ("/*+-".Contains(operation))
                {
                    isValid = true;
                    return operation;
                }

                Console.WriteLine("Please enter a valid operator (/, *, +, or -). " +
                                  "Press ^C to quit.");
            }

            return ' ';
        }

        private static float GetResult(Calculator calc, float number1, float number2, 
            char operation)
        {
            switch (operation)
            {
                case '/': return calc.Divide(number1, number2);
                case '*': return calc.Multiply(number1, number2);
                case '+': return calc.Add(number1, number2);
                case '-': return calc.Subtract(number1, number2);
                default:
                    //    ,        
                    throw new InvalidOperationException("Invalid operation passed: " + 
                                                        operation);
            }
        }
    }
}


O programa principal inicia, solicita ao usuário dois números e um operador e, em seguida, chama a classe Calculator



para executar uma operação aritmética simples nesses números. Em seguida, exibe o resultado da operação. Aqui está a turma Calculator



.



Calculator.cs:

Exploração madeireira



O aplicativo funciona bem, mas imagine: seu chefe pensou que agora todas as operações devem estar registradas em um arquivo no disco para que você possa ver o que as pessoas estão fazendo.



Parece que isso não é tão difícil, certo? Siga e adicione instruções, segundo as quais todas as operações executadas Calculator



devem ser inseridas em um arquivo de texto. Aqui está como é o seu agora Calculator



:

Isso funciona muito bem. Sempre que Calculator



algo acontece, ele é gravado em um arquivo Calculator.log



localizado no mesmo diretório de onde é executado.



Mas, a pergunta é possível: é realmente apropriado que a classe Calculator seja responsável por gravar em um arquivo de texto?



Classe filelogger



Não. Obviamente, essa não é sua área de responsabilidade. Portanto, para que o princípio de responsabilidade exclusiva não seja violado , tudo relacionado ao log de informações deve ocorrer no arquivo de log e, para isso, é necessário escrever uma classe auto-suficiente separada. Vamos fazer isso.



Primeiro de tudo, criamos uma classe completamente nova, vamos chamá-lo FileLogger



. É assim que ficará.



FileLogger.csh: 



using System;
using System.IO;

namespace OfferZenDiTutorial
{
    public class FileLogger
    {
        private const string FileName = "Calculator.log";
        private readonly string _newLine = Environment.NewLine;

        public void WriteLine(string message)
        {
            File.AppendAllText(FileName, $"{message}{_newLine}");
        }
    }
}


Agora, tudo relacionado à criação de um arquivo de log e à gravação de informações nele é processado nesta classe. Além disso, temos uma ninharia agradável: não importa o que essa classe consome, não é necessário colocar linhas em branco entre registros separados. Os registros devem apenas chamar nosso método WriteLine



, e nós cuidamos do resto. Não é legal?

Para usar uma classe, precisamos de um objeto que a instancia. Vamos resolver esse problema dentro da classe Calculator



. Substitua o conteúdo da classe pelo Calculator.cs



seguinte:



Calculator.cs: 



namespace OfferZenDiTutorial
{
    public class Calculator
    {
        private readonly FileLogger _logger;

        public Calculator()
        {
            _logger = new FileLogger();
        }

        public float Divide(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} / {number2}");
            return number1 / number2;
        }

        public float Multiply(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} * {number2}");
            return number1 * number2;
        }

        public float Add(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} + {number2}");
            return number1 + number2;
        }

        public float Subtract(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} - {number2}");
            return number1 - number2;
        }
    }
}


Portanto, agora nossa calculadora não se importa com a precisão com que o novo criador de logs grava informações no arquivo, ou onde ele está localizado, e se o arquivo foi ou não gravado. No entanto, um problema ainda existe: podemos esperar que a classe Calculator



saiba como o criador de logs é criado?



Injeção de dependência



Obviamente, a resposta para a última pergunta é negativa!



É aqui, caro leitor, que o vício entra em jogo. Vamos mudar o construtor da nossa classe Calculator



:



Calculator.cs: 



        public Calculator(FileLogger logger)
        {
            _logger = logger;
        }


Isso é tudo. Nada mais muda na classe.



A injeção de dependência é um elemento de um tópico maior chamado Inversão de gerenciamento, mas uma discussão detalhada está além do escopo deste artigo.



Nesse caso, você só precisa saber que estamos invertendo o controle da classe logger ou, falando metaforicamente, delegamos o problema de criar um arquivo para alguém FileLogger



incorporando uma instância FileLogger



em nossa calculadora e não esperando que a classe Calculator



saiba como criá-lo.



Então, de quem é essa responsabilidade?



Apenas quem instancia a turma Calculator



. No nosso caso, este é o programa principal.



Para demonstrar isso, modifique o método Main em nossa classe Program.cs da seguinte maneira:



Program.cs 



  static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            //    
            var logger = new FileLogger();
            var calc = new Calculator(logger);
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }


Assim, apenas duas linhas precisam ser alteradas. Não esperamos que a classe Calculator



instancie FileLogger



, ela fará isso por ela Main



e depois passará o resultado para ela.



Em essência, isso é injeção de dependência. Nem interfaces, nem contêineres para inversão de controle, nem nada disso são necessários. Basicamente, se você tivesse que fazer algo assim, estava lidando com injeção de dependência. Legal certo?



Ampliação de possibilidades: faremos outro logger



Apesar do exposto, as interfaces têm seu próprio lugar e, de fato, são reveladas precisamente em conjunto com a injeção de dependência.



Suponha que você tenha um cliente, do ponto de vista do qual registrar cada chamada Calculator



é um desperdício de tempo e espaço em disco, e é melhor não registrar nada.



Você acha que será necessário fazer alterações internas por causa disso Calculator



, o que potencialmente exigirá recompilar e redistribuir o assembly em que está localizado?



É aqui que as interfaces são úteis.



Vamos escrever uma interface. Vamos chamá-lo ILogger



, porque nossa classe estará envolvida em sua implementação FileLogger



.



ILogger.cs 



namespace OfferZenDiTutorial
{
    public interface ILogger
    {
        void WriteLine(string message);
    }
}


Como você pode ver, ele define um único método: WriteLine



implementado FileLogger



. Vamos dar outro passo e formalizar esse relacionamento, fazendo com que essa classe implemente oficialmente nossa nova interface:



FileLogger.cs 



public class FileLogger : ILogger


Essa é a única alteração que faremos neste arquivo. Tudo o resto será como antes.

Então, determinamos a atitude - o que devemos fazer agora?



Primeiro, modificaremos a classe Calculator para que ela use uma interface ILogger



, não uma implementação específica FileLogger



:



Calculator.cs 



private readonly ILogger _logger;

        public Calculator(ILogger logger)
        {
            _logger = logger;
        }


Neste ponto, o código ainda é compilado e executado sem problemas. Passamos a ele a FileLogger



partir do método principal do programa, aquele que implementa ILogger



. A única diferença é que Calculator



não é necessário apenas saber como criar FileLogger



, mas mesmo que tipo de criador de logs é emitido para ele.



Como tudo que você recebe implementa uma interface ILogger



(e, portanto, possui um método WriteLine



), não há problemas com o uso prático.



Agora vamos adicionar outra implementação da interface ILogger



. Esta será uma classe que não faz nada quando o método é chamado WriteLine



. Vamos ligar para ele NullLogger



, e aqui está como ele é:



NullLogger.cs 



namespace OfferZenDiTutorial
{
    public class NullLogger : ILogger
    {
        public void WriteLine(string message)
        {
            //      
        }
    }
}


Desta vez, não precisamos alterar nada da classe Calculator



se quisermos usar o novo NullLogger



, pois ele já aceita qualquer coisa que implemente a interface ILogger



.



Somente precisamos alterar o método Main em nosso arquivo Program.cs para passar outra implementação para ele. Vamos fazer isso e obter o método Main



parecido com este:



Program.cs

        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            var logger = new NullLogger(); //    
            var calc = new Calculator(logger);

Novamente, você só precisa alterar a linha que comentei. Se quisermos usar um mecanismo de log diferente (por exemplo, um no qual as informações sejam registradas no log de eventos do Windows ou usando notificações por SMS ou notificações por email), basta transferir uma implementação diferente da

interface ILogger



.



Pequeno aviso sobre interfaces



Como você pode ver, as interfaces são uma ferramenta muito poderosa, mas trazem um nível adicional de abstração ao aplicativo e, portanto, uma complexidade extra. Os contêineres podem ajudar na luta contra a complexidade, mas, como será mostrado na próxima seção, você deve registrar todas as suas interfaces e, ao trabalhar com tipos específicos, isso não é necessário.



Existe a chamada "ofuscação com a ajuda da abstração", que, em essência, se resume a uma re-complicação do projeto para realizar todos esses níveis diferentes. Se desejar, você pode ficar sem todas essas interfaces, a menos que haja razões específicas pelas quais você precisa delas. Em geral, você não deve criar uma interface que tenha apenas uma implementação.



Recipientes de injeção de dependência



O exemplo elaborado neste artigo é bastante simples: estamos lidando com uma única classe que possui apenas uma dependência. Agora, suponha que tenhamos muitas dependências, e cada uma delas esteja associada a outras dependências. Mesmo ao trabalhar em projetos moderadamente complexos, é provável que você tenha que lidar com essas situações, e não basta ter em mente que você precisa criar todas essas classes, mas também lembrar qual delas depende de qual delas - especialmente se você decidir use interfaces.



Familiarize-se com o contêiner de injeção de dependência. Isso simplifica sua vida, mas o princípio de operação de um recipiente assim pode parecer muito confuso, especialmente quando você está começando a dominá-lo. À primeira vista, essa oportunidade pode dar um pouco de mágica.

Neste exemplo, usaremos o contêiner do Unity, mas há muitos outros por onde escolher, vou citar apenas os mais populares: Castle Windsor , Ninject . Do ponto de vista funcional, esses contêineres praticamente não são diferentes. A diferença pode ser notada no nível de sintaxe e estilo, mas, no final, tudo se resume às suas preferências pessoais e experiência de desenvolvimento (bem como ao que é prescrito na sua empresa!).



Vamos dar uma olhada em um exemplo usando o Unity: tentarei explicar o que está acontecendo aqui.



A primeira coisa que você precisa fazer é adicionar um link ao Unity. Felizmente, há um pacote Nuget para isso, então clique com o botão direito do mouse no seu projeto no Visual Studio e selecione Gerenciar pacotes Nuget:







Encontre e instale o pacote Unity, concentre-se no projeto Unity Container:







Então, estamos prontos. Altere o método do Main



arquivo Program.cs



assim:



Program.cs 



 static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            //     
            var container = new UnityContainer();
            container.RegisterType<ILogger, NullLogger>();
            var calc = container.Resolve<Calculator>();
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }


Novamente, apenas as linhas marcadas precisam ser alteradas. Ao mesmo tempo, não precisamos alterar nada na classe Calculator, em nenhum dos registradores ou em sua interface: agora eles são implementados em tempo de execução; portanto, se precisarmos de um registrador diferente, precisaremos alterar apenas a classe específica, registrado para ILogger



.

Agora que este projeto funciona, o que exatamente está fazendo? Quando você escreve o código dessa maneira, fica tentado a assumir que algum tipo de mágica está envolvida. Agora vou lhe contar o que realmente vemos aqui e, espero, será uma sessão mágica com exposição, para que você não se preocupe em trabalhar com contêineres mais tarde.



Tudo começa com uma linha var calc = container.Resolve<Calculator>();



, portanto é daqui que vou explicar o significado desse código na forma de um "diálogo do contêiner consigo mesmo": o que ele "pensa" quando vê esta instrução.



  1. “Me pediram para permitir algo chamado Calculator



    . Eu sei o que é isso? "
  2. “, Calculator



    . , , . . ?”
  3. “, , - ILogger



    . , ?”
  4. “, . , ?”
  5. “, ! , , , ILogger



    , NullLogger



    .”
  6. “, NullLogger



    . . .”
  7. “ Calculator



    , var calc.”


Observe: se houvesse NullLogger



um construtor que solicitasse tipos adicionais, o construtor simplesmente repetiria todas as etapas para ele, começando com 3. Em princípio, ele examina todos os tipos e tenta resolver automaticamente todos os tipos encontrados para instâncias específicas.



Observe também que no parágrafo 2, os desenvolvedores não precisam registrar explicitamente tipos específicos. Se desejado, isso pode ser feito, por exemplo, para alterar o ciclo de vida ou passar nas classes derivadas, etc. Mas esse é outro motivo para evitar a criação de interfaces até que elas tenham uma necessidade óbvia.



Isso é tudo. Nada misterioso e especialmente místico.



Outras características



Vale a pena notar que os contêineres permitem que você faça mais algumas coisas que seriam bastante difíceis (mas não impossíveis) de implementar por conta própria. Entre essas coisas estão o gerenciamento do ciclo de vida e a implementação de métodos e propriedades. Uma discussão sobre esses tópicos está além do escopo deste artigo, pois é improvável que iniciantes precisem dessas técnicas. Mas eu recomendo que você, depois de dominar o tópico, leia atentamente a documentação para entender quais outras possibilidades existem.
