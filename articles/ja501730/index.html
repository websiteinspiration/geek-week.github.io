ProxySQL-接続を逆多重化するためのツール

こんにちは、私の名前はアレクサンダーヤコブレフです。私はシティモービルで働いて運営しています。今日は、非常に興味深いProxySQL製品について説明します。これは、多くのことを実行できる高性能のMySQLプロキシです。要求をマスクでキャッチして強制終了します。これにより、SQLインジェクションを検索したり、負荷を複製したりできます。彼との経験についてお話します。
遅かれ早かれ、開発が1組のサーバーで開始された大規模なITプロジェクトは、以下の状況に遭遇します。プロジェクトに最初は1つのデータベース（マスターサーバー）しかなかったとします。徐々に、奴隷の束がそれに追加されました。次に、シャーディングを導入しました。

そしてある晴れた日に、負荷は突然10倍に増加します。たとえば、主な競合他社が落ちて、顧客が急いでいるからです。この時点で、Webサーバーを追加するだけで負荷を単純に拡大できるように思えます。しかし、これを実行すると、不愉快な状況が発生します。

1つのウィザードの例を考えてみます。 50のWebサーバーがあり、200のphp-fpmプロセスごとにあるとします。次に、50 * 200の接続がマスターに到着し、その50 * 200 /スレーブの数が各スレーブに送信されます（もちろん、ラウンドロビンがhaproxyで構成されている場合）-下の図を参照してください。もちろん、マスターへの1万接続はたくさんありますが、それでも耐えられます。200のWebサイトがある場合、接続数はさらに多くなり、1つの接続= 1つのスレッドになります。
私たちが遭遇したのはこのボトルネックでした。



それから私たちは議論を始めました：ウィザードへの接続は常にコードにインストールされますが、すべてのfpmプロセスに必要ですか？おそらく違います。マスターへの永続的な接続の多くがスリップでハングしていることに気づきました。そして、逆多重化が必要であると彼らは決定しました。



これを行うために、ProxySQLという製品に注目しました。通常のリバースプロキシのように機能します。接続が確立され、構成で指定された特定のルールに従ってトラフィックが再配布されます。

ProxySQLをすべてのWebサーバーにインストールし、アプリケーション構成で、マスターデータベースへの呼び出しがアドレス127.0.0.1で実行されるように指定しました。以前は、各Webサーバーの200個のFPMワーカーがこのマシンからマスターベースへの200個の接続を意味していた場合、状況は変わりました。これらの200の接続はProxySQLで行われ、50〜70の接続が異なる時間に行われます。つまり、ProxySQLは既存の接続を再利用できます。

2番目のプロセスはまだ完了していませんが、ほぼ完了に近づいています。 ProxySQLを使用して、テスト環境で実際の負荷を複製することを計画しています。これは、戦闘トラフィックで新機能をチェックするために必要です。

これはどのように実装されますか？アプリケーションはProxySQLに移動し、2つのルートに沿ってトラフィックを送信します。アプリケーションが機能するための戦闘データベースと、負荷がかかっている状態で新機能をチェックするためのテスト環境です。

ProxySQLの機能は、設定があることです。この場合、puppetを介してロールアウトされます（puppetにはProxySQLモジュールがあります）が、変更を加えるためにランタイムゾーンの概念もあります（サーバーの追加、ユーザーの追加、サーバーとユーザーの削除）。 ）、通常の再起動/再読み込みは必要ありません。すべては、たとえば次のようにProxySQLコンソールを介して行われます。

mysql -ulogin -ppassword -h 127.0.0.1 -P6032 -e "INSERT INTO mysql_users(username,password,default_hostgroup) VALUES ('sm_username','pass',1);;LOAD MYSQL USERS TO RUNTIME;SAVE MYSQL USERS TO DISK;"
