Kubernetesの3レベルの自動スケーリング：それらを効果的に使用する方法


Kubernetesを完全にマスターするには、クラスターリソースをスケーリングするさまざまな方法を知る必要があります。システムの開発者によると、これはKubernetesの主要なタスクの1つです。水平および垂直の自動スケーリングとクラスターのサイズ変更のメカニズムの概要と、それらを効果的に使用する方法に関する推奨事項を用意しました。

スケーリングについて考えることがなぜ重要なのか


Kubernetesは、リソース管理およびオーケストレーションツールです。もちろん、クールなデプロイ、モニタリング、ポッド管理機能をいじくり回すと便利です（ポッドモジュールは、リクエストに応じて起動されるコンテナーのグループです）。

ただし、そのような問題について考える必要があります。

  1. モジュールとアプリケーションのスケーリング方法は？
  2. コンテナを運用可能かつ効率的に保つ方法は？
  3. ユーザーからのコードとワークロードの絶え間ない変化にどのように対応しますか？

リソースとパフォーマンスのバランスをとるようにKubernetesクラスタを構成するのは困難な場合があります。これには、Kubernetes内部の専門知識が必要です。アプリケーションまたはサービスのワークロードは1日または1時間で変動する可能性があるため、バランシングは継続的なプロセスとして最もよく表されます。

Kubernetesの自動スケールレベル


効果的な自動スケーリングには、2つのレベル間の調整が必要です。

  1. 水平（水平ポッドオートスケーラー、HPA）および垂直オートスケール（垂直ポッドオートスケーラー、VPA）を含むポッドレベル。これは、コンテナで使用可能なリソースをスケーリングしています。
  2. クラスターレベルは、クラスターオートスケーラーCAによって制御され、クラスター内のノードの数を増減します。

水平自動スケールモジュール（HPA）


名前が示すように、HPAはポッドレプリカの数をスケーリングします。レプリカの数を変更するためのトリガーとして、ほとんどの開発者はCPUとメモリの負荷を使用します。ただし、カスタム指標、それらの組み合わせ、または外部指標に基づいてシステムをスケーリングできます。

高レベルのHPAワークフロー：

  1. HPAは、インストール中に指定されたメトリック値をデフォルトの間隔である30秒で継続的にチェックします。
  2. HPAは、指定されたしきい値に達した場合にモジュールの数を増やすことを試みます。
  3. HPAは、デプロイメント/複製コントローラー内のレプリカの数を更新します。
  4. 次に、展開/レプリケーションコントローラは、必要なすべてのアドオンモジュールを展開します。

垂直自動スケール（VPA）は、既存のポッドに割り当てるプロセッサ時間またはメモリ時間を増やしたり減らしたりします。ステートフルまたはステートレスのポッドに適していますが、主にステートフルサービスを対象としています。ただし、最初に割り当てられたリソースの量を自動的に調整する必要がある場合は、ステートレスモジュールにVPAを適用できます。

VPAはOOMイベント（メモリ不足、メモリ不足）にも応答します。プロセッサ時間とメモリサイズを変更するには、ポッドの再起動が必要です。再起動すると、VPAはポッド配布バジェット（PDB）を尊重して、モジュールの最小数を保証します。

各モジュールのリソースの最小量と最大量を設定できます。したがって、割り当てられたメモリの最大量を8 GBの制限に制限できます。これは、現在のノードがコンテナーごとに8 GBを超えるメモリを正確に割り当てることができない場合に役立ちます。詳細な仕様と操作メカニズムは、公式VPA wikiに記載されています。

さらに、VPAには興味深いレコメンデーション機能（VPA Recommender）があります。すべてのモジュールのリソース使用率とOOMイベントを追跡し、履歴メトリックを考慮したインテリジェントなアルゴリズムに基づいて、メモリとプロセッサ時間の新しい値を提供します。ポッド記述子を受け取り、提案されたリソース値を返すAPIもあります。

VPA Recommenderがリソースの「制限」を監視しないことは注目に値します。これにより、モジュールがノード内のリソースを独占する可能性があります。メモリまたはプロセッサ時間の大きな浪費を避けるために、名前空間のレベルで制限値を設定することをお勧めします。

VPAの高レベルのスキーム：

  1. VPAは、インストール中に指定されたメトリック値をデフォルトの間隔である10秒で継続的にチェックします。
  2. 指定されたしきい値に達すると、VPAは割り当てられたリソースの量を変更しようとします。
  3. VPAは、デプロイメント/レプリケーションコントローラ内のリソースの量を更新します。
  4. モジュールを再起動すると、作成されたインスタンスにすべての新しいリソースが適用されます。

クラスターオートスケーラー（CA）は、待機中のポッドの数に基づいてノードの数を変更します。システムは定期的に保留中のモジュールをチェックし、さらにリソースが必要で、クラスターが設定された制限を超えない場合は、クラスターサイズを増やします。CAはクラウドサービスプロバイダーと対話し、クラウドサービスプロバイダーに追加のノードを要求するか、アイドル状態のノードを解放します。CAの最初の公開バージョンはKubernetes 1.8で導入されました。

高レベルの操作スキームCA：

  1. CAは、デフォルトの間隔が10秒のスタンバイ状態のモジュールをチェックします。
  2. クラスターで使用できるリソースが不足しているために1つまたは複数のモジュールがスタンバイ状態にある場合、モジュールは1つまたは複数の追加ノードを準備しようとします。
  3. クラウドサービスプロバイダーが必要なノードを割り当てると、クラスターに参加し、ポッドモジュールを提供する準備が整います。
  4. Kubernetes Schedulerは保留中のモジュールを新しいホストに配布します。この後、一部のモジュールがまだスタンバイ状態のままである場合、プロセスが繰り返され、新しいノードがクラスターに追加されます。


クラウド内のクラスターノードの自動割り当て

CAを使用する場合は、次の点を考慮してください。

  • CAは、プロセッサーの負荷に関係なく、クラスター内のすべてのモジュールを実行する場所を確保します。さらに、彼はクラスター内に不要なノードがないことを確認しようとします。
  • CAは、約30秒後にスケーリングの必要性を登録します。
  • ノードが不要になった後、CAはデフォルトで10分間待機してからシステムをスケーリングします。
  • オートスケールシステムには、エキスパンダーの概念があります。これらは、新しいノードが追加されるノードのグループを選択するためのさまざまな戦略です。
  • 責任を持ってオプションcluster-autoscaler.kubernetes.io/safe-to-evict（true）を使用してください。多くのポッドをインストールする場合、またはそれらの多くがすべてのノードに分散している場合、クラスターをダウンサイズする機能が大幅に失われます。
  • PodDisruptionBudgetsを使用して、ポッドが削除されないようにします。これにより、アプリケーションの一部が完全にクラッシュする可能性があります。

自動スケーリングを適用しようとするときに開発者が遭遇するいくつかの典型的な問題があります。

HPAとVPAは、メトリックといくつかの履歴データに依存しています。十分なリソースが割り当てられていない場合、モジュールは折りたたまれ、メトリックを生成できなくなります。この場合、自動スケーリングは行われません。

スケーリング操作自体は時間に敏感です。ユーザーが問題や障害に気付く前に、モジュールとクラスターをすばやくスケーリングしたいと考えています。したがって、ポッドとクラスターの平均スケーリング時間を考慮する必要があります。

理想的なシナリオ-4分：

  1. 30秒。ターゲット指標の更新：30〜60秒。
  2. 30秒。HPAはメトリック値をチェックします：30秒。
  3. 2秒未満。ポッドモジュールが作成され、スタンバイ状態になります：1秒。
  4. 2秒未満。CAは保留中のモジュールを確認し、ノードを準備するための呼び出しを送信します：1秒。
  5. 3分。クラウドプロバイダーはノードを割り当てます。K8は準備ができるまで待機します。最大10分です（いくつかの要因によって異なります）。

最悪の（より現実的な）シナリオ-12分：

  1. 30秒。ターゲット指標の更新。
  2. 30秒。HPAはメトリック値を検証します。
  3. 2秒未満。ポッドモジュールが作成され、スタンバイ状態になります。
  4. 2秒未満。CAは保留中のモジュールを確認し、ノードを準備するための呼び出しを送信します。
  5. 10分。クラウドプロバイダーはノードを割り当てます。K8は準備ができるまで待機します。待ち時間は、サプライヤーの遅延、OSの遅延、補助ツールの作業など、いくつかの要因に依存します。

クラウドプロバイダーのスケーリングメカニズムとCAを混同しないでください。後者はKubernetesクラスター内で機能し、クラウドプロバイダーメカニズムはノード割り当てに基づいて機能します。彼はあなたのポッドやアプリケーションで何が起こっているのか知りません。これらのシステムは並行して動作します。

Kubernetesでスケーリングを制御する方法


  1. Kubernetesは、リソース管理およびオーケストレーションツールです。クラスタポッドとリソース管理オペレーションは、Kubernetesの開発における重要なマイルストーンです。
  2. HPAおよびVPAのポッドスケーラビリティロジックについて学習します。
  3. CAは、ポッドとコンテナのニーズをよく理解している場合にのみ使用してください。
  4. クラスターを最適に構成するには、さまざまなスケーリングシステムがどのように連携するかを理解する必要があります。
  5. スケーリング時間を評価するときは、最悪のシナリオと最良のシナリオに留意してください。
