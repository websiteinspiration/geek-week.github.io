Webページで使用されるすべてのメモリを監視する：performance.measureMemory（）

本日の翻訳は、この記事の著者が、Webページに割り当てられたメモリを監視する方法について語っています。本番環境で動作しているページのメモリに細心の注意を払うと、Webプロジェクトの生産性を高いレベルで維持するのに役立ちます。



ブラウザは、Webページに割り当てられたメモリを自動的に制御します。ページがオブジェクトを作成すると、ブラウザはその内部メカニズムを使用して、このオブジェクトを格納するためのメモリを割り当てます。メモリは無限のリソースではないため、ブラウザは定期的にガベージコレクションを実行し、その間に不要なオブジェクトが検出されて、それらが占有しているメモリがクリアされます。ただし、そのようなオブジェクトを検出するプロセスは理想的ではありません。それは証明されましたそのようなオブジェクトの絶対的に正確で完全な識別は不溶性のタスクです。その結果、ブラウザーは「不要なオブジェクト」を見つけるという考えを「到達できないオブジェクト」を見つけるという考えに置き換えています。 Webページがその変数およびアクセス可能な他のオブジェクトのフィールドを介してオブジェクトにアクセスできない場合、これはブラウザがそのようなオブジェクトによって占有されているメモリを安全にクリアできることを意味します。次の例に示すように、「不要」と「到達不能」の違いにより、メモリリークが発生します。

const object = { a: new Array(1000), b: new Array(2000) };
setInterval(() => console.log(object.a), 1000);

ここには不要な大きな配列がありますbが、ブラウザはobject.b、コールバックのオブジェクトプロパティを通じてアクセスできるため、ブラウザが占有しているメモリを解放しません。その結果、このアレイが占有しているメモリがリークしています。

メモリリークは、Web開発でよく発生します。たとえば、開発者が誤って要素内のオブジェクトをキャプチャしたiframeとき、ワーカーを閉じ忘れたとき、配列内のオブジェクトを収集したときに、それらはプログラムに非常に簡単に表示されます。 Webページにメモリリークがあると、時間の経過とともにページのメモリ消費量が増加します。そのようなページは、ユーザーにとってどんどん遅くなります。

この問題を解決する最初のステップは、測定を行うことです。新しいperformance.measureMemory（） APIを使用すると、開発者は本番環境のWebページによるメモリ使用量を測定し、その結果、ローカルテストをすり抜けるメモリリークを検出できます。

新しいperformance.measureMemory（）APIは、以前のperformance.memoryとどう違うのですか？


既存の非標準API performance.memoryに精通している場合は、新しいAPIが古いAPIとどのように異なるかという質問に興味があるかもしれません。主な違いは、古いAPIはJavaScriptヒープのサイズを返し、新しいAPIはWebページ全体のメモリ使用量を評価することです。この区別は、Chromeが複数のWebページ（または同じページの複数のインスタンス）間のヒープ共有を編成する場合に重要です。このような場合、古いAPIによって返された結果が歪む可能性があります。古いAPIは、ヒープなどの実装に固有の用語で定義されているため、標準化は絶望的なビジネスです。

もう1つの違いは、Chromeでは、新しいAPIがガベージを収集するときにメモリ測定を行うことです。これにより、測定結果の「ノイズ」が減少しますが、結果が出るまでに時間がかかる場合があります。他のブラウザの作成者は、ガベージコレクションにバインドせずに新しいAPIを実装する場合があることに注意してください。

APIAPI使用の確認


SecurityErrorがスローされてperformance.measureMemory()、 関数呼び出しが失敗する場合があります。これは、環境が情報漏えいのセキュリティ要件を満たしていない場合に発生する可能性があります。ChromeでのOrigin Trialテスト中、このAPIにはサイト分離を含める必要があります。APIが通常の使用の準備ができると、それはcrossOriginIsolatedプロパティに依存します。ヘッダーCOOPおよびCOEPを設定することにより、Webページをこのモードで操作できます。 これがサンプルコードです：



if (performance.measureMemory) {
  let result;
  try {
    result = await performance.measureMemory();
  } catch (error) {
    if (error instanceof DOMException &&
        error.name === "SecurityError") {
      console.log("The context is not secure.");
    } else {
      throw error;
    }
  }
  console.log(result);
}

▍現地試験


Chromeは、ガベージコレクション時にメモリを測定します。つまり、APIにアクセスしても、Promiseがすぐに解決されるわけではありません。結果を取得するには、次のガベージコレクションセッションを待つ必要があります。APIは、現在20秒に設定されている特定のタイムアウト後にガベージコレクションを強制的に開始します。コマンドラインフラグを指定してChromeを実行する--enable-blink-features='ForceEagerMeasureMemory'と、タイムアウトがゼロになり、ローカルデバッグやローカルテストに役立ちます。

例


ページ全体のメモリ使用量のレベルを測定し、結果をサーバーに送信してそれらを集計および分析できるグローバルメモリモニターを定義して、新しいAPIを使用することをお勧めします。このAPIを使用する最も簡単な方法は、定期的な測定を行うことです。たとえば、毎M分実行できます。ただし、メモリ使用量のピークが測定間で発生する可能性があるため、データに歪みが生じます。次の例は、ポアソンプロセスを使用して、系統的誤差のない測定を行う方法を示しています。このアプローチにより、測定セッションが同じ確率でいつでも発生することが保証されます（ここでは、このアプローチのデモを示します。ここにソースコードがあります）。

最初に、セッションの次の開始を計画する関数を宣言setTimeout()して、ランダムに設定された間隔で関数を使用して消費されたメモリの量を測定します。この関数は、ブラウザウィンドウにページを読み込んだ後に呼び出す必要があります。

function scheduleMeasurement() {
  if (!performance.measureMemory) {
    console.log("performance.measureMemory() is not available.");
    return;
  }
  const interval = measurementInterval();
  console.log("Scheduling memory measurement in " +
      Math.round(interval / 1000) + " seconds.");
  setTimeout(performMeasurement, interval);
}

//       .
window.onload = function () {
  scheduleMeasurement();
}

この関数measurementInterval()は、1回の測定が約5分ごとに実行されるように設定された、ミリ秒で表されるランダムな間隔を見つけます。この関数の基礎となる数学的概念に興味がある場合は、指数分布についてお読みください。

function measurementInterval() {
  const MEAN_INTERVAL_IN_MS = 5 * 60 * 1000;
  return -Math.log(Math.random()) * MEAN_INTERVAL_IN_MS;
}

その結果、非同期関数performMeasurement()はAPIを呼び出し、結果を記録し、次の測定を計画します。

async function performMeasurement() {
  // 1.  performance.measureMemory().
  let result;
  try {
    result = await performance.measureMemory();
  } catch (error) {
    if (error instanceof DOMException &&
        error.name === "SecurityError") {
      console.log("The context is not secure.");
      return;
    }
    //    .
    throw error;
  }
  // 2.  .
  console.log("Memory usage:", result);
  // 3.   .
  scheduleMeasurement();
}

// ,    :
{

メモリ使用の全体的なレベルの推定値がフィールドに表示されますbytes。この推定値を導出するとき、数字の桁の区切り文字が使用されます。これらの値は、実装に大きく依存します。異なるブラウザーで受信された場合、それらを比較することはできません。それらを取得する方法は、同じブラウザーの異なるバージョンでも異なる場合があります。 Origin Trialプログラムが継続している間、戻り値には、メインウィンドウによるJavaScriptのメモリ使用量のインジケーターiframe、同じサイトの要素のメモリ使用量のインジケーター、および関連するウィンドウのインジケーターが含まれます。 APIの準備ができると、この値は、JavaScript、DOM、iframeウィンドウとWebワーカーに関連付けられているすべての要素によって消費されるメモリに関する情報になります。

リストbreakdown使用メモリに関する詳細情報を提供します。各エントリはメモリの一部を表し、このフラグメントをiframe、URLで識別される一連のウィンドウ、要素、またはワーカーに関連付けます。フィールドにuserAgentSpecificTypesは、実装機能によって決定されるメモリのタイプがリストされます。

これらのリストを一般的な方法で検討することが重要です。特定のブラウザーの機能に依存して、それらに基づいてすべてを分析しようとするのではありません。たとえば、一部のブラウザは空のリストbreakdownまたは空のフィールドを返す場合がありますattribution。他のブラウザは、要素内でattribution複数のURL を返す場合があります。これは、メモリがこれらのURLのどれに属しているかを特定できないことを示しています。
