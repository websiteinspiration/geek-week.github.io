実行時ではなく、設計時


むかしむかし、賢い老いた開発者が私にアドバイスをくれましたが、最近まで私は本当に感謝しませんでした。

コードのレビュー中に、A〜Zの文字のリスト（たとえば、特定の文字で始まる名前にジャンプできるボタンのセットを含む連絡先のリスト）を表示する必要がある関数を調べました。

それで、若く成功したプログラマーが現れました。 （まあ、それは私でした。）すべての文字の配列のハードコードの代わりに、65から90で実行されるforループを記述して、受け取った文字コードで文字を生成する方が簡単だと判断しました。

JavaScriptバリアントでは、次のようになります。

for (let i = 65; i <= 90; i++) {
 letters.push(String.fromCharCode(i))
}

賢明な古い開発者が私を見て、なぜこの配列をハードコーディングしないのかと尋ねました。アルファベットがセッションごとに異なるようには見えません。なぜ毎回それを計算するのですか？

この記事は、Vivaldiドキュメントリポジトリ診断システムを開発し、スタートアップに投資している EDISON Softwareのサポートにより翻訳されました。

怖かった。「ある種の子供のように、どのようにしてすべての文字をタイプすることを期待できますか。私はプロのソフトウェア開発者です！「天国のために、アルゴリズムとデータ構造、そして数学コプロセッサーを持っています！」

「いい」と彼は言った。「この機能を設計時に使用して配列を生成し、それをコピーして作業コードに貼り付けるだけです。」

そして彼はこれを言った：
「開発時に実行できることを実行時に避けてください。」
正直に言いましょう。私の小さなforループがアプリケーションの速度を低下させることはありませんでした。そして、最近のマシンはこのコードをすぐに理解できるので、誰も気付かないでしょう。しかし、原則として、これは賢明なアドバイスです。

多くの場合、リクエストごとにめったに変更されないデータをある形式から別の形式に変換するコードを記述します。年に1〜2回変更される可能性のあるデータベースからコンテンツを取得し、フォーマットしてブラウザーに送信するすべての状況を考慮してください。アプリケーションを遅くする必要はありません。これは、コンテンツ管理システムに関連付けられているサイトに特に当てはまります。

そのため、今後数年間で、Wordpress、Drupalなどの有名なプレーヤーは、次のような静的サイトジェネレーターから深刻な問題に直面すると思いますGatsby、Hugo、またはJekyllと、スムーズなビルドプロセス、ヘッドレスCMS、安価なCDN、高速で継続的な統合プロセスを組み合わせたもの。

このパターンはJAMstackと呼ばれ、「JavaScript、API、およびマークアップスタック」を意味します。そして、結果は非常に印象的です。

賢明な古い開発者からのアドバイスは、耳に聞こえます。「プログラムの実行中は、開発中にできることを避けてください。」そして時が経つにつれ、このアドバイスには広範囲にわたる影響があることに気づきました。ソフトウェア開発のためだけでなく、生活のためにも。

最近、レイダリオの素晴らしい本「原則：仕事と生活」を読みました。。この本の中心的なテーマは、実際の問題よりも問題の種類がはるかに少ないということです。したがって、事前にこれに取り組み、遭遇する可能性のある特定のタイプの問題にどのように取り組むかを見つければ、それが来ると、それに対処する準備がはるかによくなります。

実際、さまざまな種類の問題へのアプローチをソートすることで、より現実的な問題に直面してパニックに陥る「実行」ではなく、人生について静かに考えている「開発」中に、アプローチをソートすることで、より良い決定をより速く行うことができます。

ダリオは、彼のアプローチを一連の原則としてカタログ化することにより、この手法を実装しました。彼はさらに遠くまで行って、彼の意思決定プロセスを一連のコンピューターアルゴリズムに体系化し、膨大な量の履歴データでテストできるようにしました。

彼は数十億の億万長者であり、非常に成功した投資会社を経営しているとすれば、それはうまくいったと言えるでしょう。実際、ウォール街は株式トレーダーよりも多くのプログラマーを雇い始めています。したがって、間違った職業を選択したことに疑いがある場合、ソフトウェアが世界を食べているという多くの証拠があります。
