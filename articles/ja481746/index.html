WSL 2ファイルシステムとパフォーマンス


WSLバージョン2内のファイルはVHDX仮想ディスクに保存され、ext4がファイルシステムとして使用されます。次の形式のパスでrootfs WSLにアクセスできます。

\\wsl$\{DistroName}\

または、「explorer.exe」と入力できます。 CLIで、現在のディレクトリでWindowsブラウザーを開きます。

WSLバージョン1はVHDXを使用せず、Linuxファイルが置かれているディレクトリに簡単にアクセスできました。Microsoft は、WindowsからLinuxファイルを変更しないことを強く推奨しました。 WSLの新しいバージョンでは、仮想ディスク上のFSへのアクセスは、Plan 9 Filesystem Protocolファイルサーバーを使用して提供されます。



以前のバージョンのWSLでは、システムコールがWindows APIを介してエミュレートされていたため、ファイルシステムのパフォーマンスに問題があり、ファイルアクセスが遅く、不安定でした。 2019年末のWSL 2では、アーキテクチャが変更され、ネイティブLinuxカーネルが使用されました。Linuxアーキテクチャ用の新しいWindowsサブシステム（詳細）のYouTubeプレゼンテーションのスライドから判断すると、ディスク操作のパフォーマンスは2〜5倍に向上しました。 最大ディスク容量は256GBに制限されています。このボリュームを超えると、サイズを変更する必要があります。手順はドキュメントに記載されています。 WSLは当初、RAMの使用後にリソースを解放する際に問題がありました。19013のビルドでは、この問題は

逆に、WindowsアプリケーションからLinuxソケットへの接続が必要な場合は、WSLのIPアドレスにアクセスする必要があります。注意点があります-Linuxでは、サービスをlocalhost（127.0.0.1）ではなく、アドレス0.0.0.0で実行する必要があります。たとえば、SOCKS5プロキシをVPSにすばやく上げるには、動的ポート転送パラメーターを使用してSSHを実行する必要があります。

Linuxは毎回再起動後にIPアドレスを変更するため、ポート転送を実行して自動的に実行する必要があるシナリオでは、回避策を探す必要があります。この問題を解決するには多くの方法があり、松葉杖の程度はさまざまですが、github でこの問題の詳細を読むことができます。私はWindowsサービスを実装するgo-wsl2-hostユーティリティを使用しました。これにより、WSL IPアドレスがWindows ホストファイルに自動的に追加されるため、ubuntu.wslのようなホスト名をホストシステムに登録し、それを使用してLinuxにアクセスできます。ただし、これらの問題はすべて解決できず、あまりうまく機能しません。Microsoftがこれらの問題を修正した場合、それは予期されたままです。

Windows上のOpenSSHおよび自動開始サービス


Windows Server 2019と同様に、Windows 10にはOpenSSHフォークが付属しています。これには、使い慣れたssh-keygen、ssh-add、scp、およびssh-agentやsshdサーバー自体を含むその他のユーティリティがすべて含まれています。クライアントとサーバーの両方は、[アプリ]> [アプリと機能]> [オプション機能の管理]からインストールできますが、sshクライアントのバージョンは最新ではありません。ProxyJumpオプションを使用してジャンプホスト経由でホストに接続できないバグが発生し、この問題は修正されたことが判明しましたが、クライアントのSSHを手動で更新する必要がありました。現在のバージョンのWin32 OpenSSHをインストールするには、githubのリリースセクションからzipをダウンロードして、たとえばC：\ Program Files \ OpenSSHに解凍します。。サードパーティソフトウェアssh.exe（たとえば、VSCodeでリモート開発モードを使用している場合）が$ PATH（％SYSTEMROOT％\ System32 \ OpenSSH \）から呼び出された場合は、環境変数を変更する必要があります。環境変数は、[スタート]> [システム環境変数の編集]（[スタート]> [システム環境変数の変更]）のGUIで変更します。古いバージョンの上に新しい方法を追加する必要があります。 

SystemdはWSLでは機能しないため、システムの起動に伴うサービスの起動に問題があります。 WSLでsshサーバーの自動起動を設定するにはいくつかの方法があります。最も簡単な方法は、サーバースタートコマンドを指定できるタスクスケジューラでタスクを作成することです。インターネットでは、vbsスクリプトを使用してさまざまな起動手順を見つけることができます、PS1またはバット。ほとんどすべてのメソッドの問題は、トリガーがメインのWindows OSを起動することです。つまり、WSLが失敗してシステムを再起動する必要がある場合（wsl -t）、Linuxはサービスを実行せずに起動します。 Windowsが起動すると、WSL配布は最初にアクセスされたときにのみ開始されます。

マシン間をリモートで移動できるように、WSL内のラップトップでSSHサーバーを使用しています。また、sshポート転送技術と十分に検討されたSSHクライアントの集中構成を使用しているため、すべてのサーバーに透過的にアクセスでき、アドレスの代わりにホスト名を入力できます。つまり、ラップトップの1つをモバイルネットワークに接続した場合でも、autosshデーモンはジャンプホストに接続し、私はまだコンピューターにアクセスできます。NATが邪魔になることはありません。したがって、sshdが常に稼働していることが重要です。



SSHをWSLに取得する唯一の有効な方法は、SSHポートを転送することです。これは、RemoteForwardを使用してWSL自体から実行できます、またはホームネットワーク上の別のサーバーから。それを必要とする人はほとんどおらず、すでに高度なレベルになっているので、実際にコマンドを

❯❯ ssh -R '*:2363:*:22' -N -f mt.example.com

実行します。これで、アドレスmt.example.com:2263に接続すると、直接WSLにアクセスできます。

WSLでSSHサーバーを上げる予定の場合は、/ etc / ssh / sshd_configで必要なサーバー起動パラメーターを設定することを忘れないでください。ポート22でのサービスバインドとの競合を回避するために、WindowsのOpenSSHサーバーがインストールされている場合は無効にするか完全に削除する必要があります（アプリ>アプリと機能>オプション機能の管理）。 

X転送


Windows 10でstdoutを直接Windowsクリップボードにリダイレクトできるclip.exeユーティリティがあるのは楽しい瞬間でした。これはtmuxなどのプログラムで使用すると便利です。Xサーバーの転送により、テキストをリモートホストからコピーすることもできます。すべてを機能させるには、常にWindowsでXサーバーを実行し、$ DISPLAY変数を正しく設定する必要があります。 

少しつまらない理論。 Xを実行する* nixシステムには、さまざまなタイプのクリップボード（プライマリ、セカンダリ、クリップボード）があります。）、この記事の文脈では、これはそれほど重要ではありませんが、作業のメカニズムを一般的に理解するには重要です。 Linuxでクリップボードを操作するための2つのユーティリティ（xclipとxsel）があります。どちらのユーティリティも同様の機能を備えていますが、xselでは少し大きくなっていますが、バッファの内容を転送するために必要な基本機能は同じです。 Xアプリケーションでは、選択したテキストが1次選択に分類され、マウスの中央ボタンで挿入されます。xclipおよびxselでは、1次選択がデフォルトで使用されます。

たとえば、変数の内容をデフォルトバッファーにコピーするには、追加パラメーターなしでstdoutをxclipユーティリティのstdinに渡す必要があります。

❯❯ echo -n $DISPLAY | xclip

デフォルトバッファーの内容を表示するには、-oスイッチを指定してxclipを実行します。

❯❯ xclip -o
172.20.160.1:0

クリップボードがXサーバーを介してリダイレクトされ、グラフィックアプリケーションがローカルXサーバーで実行されるようにするには、WSLのデフォルトゲートウェイであるIPアドレスを$ DISPLAY変数に設定する必要があります。これまでのところ、（Windowsによって自動的に生成される）resolv.confから取得することよりも優れた方法は発明されていません。したがって、最も簡単な方法は、$ DISPLAY変数のエクスポートをシェルプロファイルに登録することです（たとえば、zshの場合は〜/ .zshrc）。

❯❯ echo "export DISPLAY=$(cat /etc/resolv.conf | grep nameserver | awk '{print $2}'):0" >> ~/.zshrc

chmod 700 /mnt/c/Users/${WIN_USER}/.ssh

Windowsターミナルをインストールして構成する


Windowsターミナルは、Windowsストアから、またはプロジェクトのgithubのリリースセクションからバイナリをダウンロードしてインストールできます。関連情報、手順、FAQもすべてあります。端末には、少なくともWindows 1903のバージョンとビルド18362が必要です。この場合、ストアから直接更新する方が簡単なので、Windowsストアからインストールすることをお勧めします。更新は定期的にリリースされ、開発計画がgithub（ロードマップ）端末の最初のバージョン。現時点では、バージョン1のすべての機能がすでに実装されており（計画によれば、2019年末までにすべての改善を実装する予定）、バグの修正作業から数か月後、2020年4月にターミナルv1.0の公式リリースが計画されています。 MSがgithub上にあるのは素晴らしいことです。彼らのソフトウェアはログ、わかりやすいエラーを表示することを学び、問題があれば簡単にグーグル検索できます。



端末にはまだ多くの設定はありませんが、快適な作業には十分です。製品は活発に開発されており、ユーザーは機能のリクエストやバグレポートを作成できるgithubにあります。開発者はユーザーとの問題の議論に参加し、バグが発見されたときに回避策を提供することがよくあります。



設定はjson形式で保存され、保存するとすぐに適用されます。これは、作業環境の構成を管理するための適切な方法を適用できる場合にのみ便利です-私はすべてのLinux構成をgitリポジトリに保存し、Windowsでは作業ツールからVSCodeのみを使用します。 。そのため、ターミナルはVSCodeと同じホットキーと設定形式を使用して、同じ方向に移動します。ちなみに、特にMSの優れたエディタを既に使用している場合は特に、構成の編集はVSCodeを使用すると便利です。端末構成ファイルにはすでに多くのデフォルト設定が含まれており、正しいエディターを使用すると、スキーマのキーと値のオプションの説明とともにすべてのオプションを表示できます（これは、プロジェクトに完全なドキュメントがまだない場合に特に便利です）。さらに、すべてのIDEチップは、オートコンプリート、インテリセンス、構文チェック、フォーマットなどの形で利用できます。
