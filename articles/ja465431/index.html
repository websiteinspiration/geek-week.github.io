PVS-Studio静的アナライザーを使用したApache DubboフレームワークのRPCソースコードの分析


Apache Dubboは、GitHubで最も人気のあるJavaプロジェクトの1つです。そして、これは驚くべきことではありません。8年前に作成され、高性能RPC環境として広く使用されています。もちろん、彼のコードのほとんどのエラーは長い間修正されており、コードの品質は高いレベルで維持されています。ただし、PVS-Studio静的コードアナライザーを使用して、このような興味深いプロジェクトのチェックを拒否する理由はありません。何とかして見つけたものを見てみましょう。

PVS-Studioについて


PVS-Studio 静的コードアナライザーは、IT市場に10年以上存在し、多機能で簡単に実装できるソフトウェアソリューションです。現在、アナライザーはC、C ++、C＃、Java言語をサポートしており、Windows、Linux、およびmacOSプラットフォームで動作します。

PVS-Studioは有料のB2Bソリューションであり、さまざまな企業の多数のチームで使用されています。アナライザーの機能を評価する場合は、配布キットをダウンロードして、こちらから試用キーをリクエストしてください。オープンソースプロジェクトで、または学生の場合は、PVS-Studioを無料で使用するオプション

もあります。

Apache Dubbo：それは何で何を食べますか？


現在、ほとんどすべての大規模なソフトウェアシステムが配布されています。分散システムで、応答時間が短く、送信データが比較的少ないリモートコンポーネント間にインタラクティブ接続がある場合、これはRPC（リモートプロシージャコール）環境を使用する良い理由です。

Apache Dubboは高性能RPC環境です（オープンプロシージャJavaベースのリモートプロシージャコール）。多くのRPCシステムと同様に、dubboは、パラメーターと戻りデータ型を使用してリモートで呼び出すことができるメソッドを定義するためのサービスを作成するという考えに基づいています。サーバー側では、インターフェイスが実装され、クライアント呼び出しを処理するためにダボサーバーが起動されます。サーバーと同じメソッドを提供するクライアント側のスタブがあります。Dubboは、フロントエンドのリモート呼び出し、フォールトトレランス、負荷分散、サービスの自動登録と検出など、3つの主要な機能を提供します。

V6007式「endKey [i] <0xff」は常に真です。OptionUtil.java（32）

public static final ByteSequence prefixEndOf(ByteSequence prefix) {
  byte[] endKey = prefix.getBytes().clone();
  for (int i = endKey.length - 1; i >= 0; i--) {
    if (endKey[i] < 0xff) {                                           // <=
      endKey[i] = (byte) (endKey[i] + 1);
      return ByteSequence.from(Arrays.copyOf(endKey, i + 1));
    }
  }
  return ByteSequence.from(NO_PREFIX_END);
}

バイト タイプの値（-128〜127の値）は、0xff（255）の値と比較されます。この条件では、Java のバイト型が重要であることは考慮されないため、条件は常に満たされるため、意味がありません。

V6009 'substring'関数は '-1'値を受け取る可能性がありますが、負でない値が予期されています。引数の検査：2. AbstractEtcdClient.java（169）

protected void createParentIfAbsent(String fixedPath) {
  int i = fixedPath.lastIndexOf('/');
  if (i > 0) {
    String parentPath = fixedPath.substring(0, i);
    if (categories.stream().anyMatch(c -> fixedPath.endsWith(c))) {
      if (!checkExists(parentPath)) {
          this.doCreatePersistent(parentPath);
      }
      } else if (categories.stream().anyMatch(c -> parentPath.endsWith(c))) {
        String grandfather = parentPath
        .substring(0, parentPath.lastIndexOf('/'));            // <=

      }
  }

lastIndexOf 関数の結果は2番目のパラメーターによってsubstring関数に渡されます。その2番目のパラメーターは負の数であってはなりませんが、lastIndexOfは、ストリング内に値が見つからない場合は-1を返すことがあります。サブストリングメソッドの2番目のパラメーターが最初のパラメーターより小さい（-1 <0）場合、StringIndexOutOfBoundsExceptionがスローされます。エラーを修正するには、lastIndexOf関数によって返された結果を確認し、それが正しい場合（少なくとも負ではない場合）、それを部分文字列関数に渡す必要があります。

結論


JavaでRPC を使用することに関心のある人は、おそらくApache Dubboについて聞いたことがあるでしょう。これは、長い歴史と多くの開発者によって書かれたコードを持つ人気のあるオープンソースプロジェクトです。このプロジェクトのコードは優れた品質ですが、PVS-Studioスタティックアナライザーはいくつかのエラーを見つけることができました。このことから、中規模から大規模のプロジェクトを開発する場合、コードがどんなに完璧であっても、静的分析は非常に重要であると結論付けることができます。

注意。このような1回限りのチェックは、静的コードアナライザーの機能を示していますが、それを使用する完全に間違った方法です。このアイデアについては、こことここで詳しく説明します。分析を定期的に使用してください！

このような素晴らしいツールを提供してくれたApache Dubbo開発者に感謝します。この記事がコードの改善に役立つことを願っています。この記事では、疑わしいコードのセクションすべてについて説明しているわけではないため、開発者が自分でプロジェクトをチェックして結果を評価する方がよいでしょう。
