CRDを使用してコード形式のポリシーのスタイルでコンテナーのセキュリティを自動化する方法



Kubernetes CRDを使用してセキュリティを自動化し、アプリケーションを保護する方法を紹介します。

雑誌Tomorrow Cloudy Mail.ru Cloud Solutionsのチームからの翻訳。出典：Niteen Kole CRDを使用してコンテナーのセキュリティを自動化し、セキュリティポリシーをコード付きのコードとして追加する方法。

CRDが必要な理由


セキュリティは、DevOpsチームにとって長い間頭痛の種でした。既存のソフトウェアツールは、アセンブリの自動化とアプリケーションのロールアウトの問題を正常に解決します。コンテナに基づくアプリケーションの自動ロールアウトが標準になりました。同時に、セキュリティ設定の自動化ははるかに遅れています。

自動化された脆弱性スキャンを実装できますが、アプリケーションセキュリティのポリシーを手動で構成する必要が頭痛の種になっています。Kubernetesカスタムリソース定義（CRD）

の導入は、アプリケーションのアセンブリおよびロールアウトの初期段階でセキュリティポリシーをコードとして定義する際の問題を解決し、アプリケーションを本番環境にロールアウトする際の使用を自動化するのに役立ちます。

CRDを使用すると、アプリケーションの動作を定義し、使用するいくつかのKubernetesクラスターのセキュリティを構成するグローバルセキュリティポリシーを実装できます。

CRDを使用して自動化し、セキュリティ設定をコードとして一元的に定義することで、セキュリティ設定を同時に強化し、アプリケーションを簡略化できます。これにより、最終的には、アプリケーションがより効率的になり、エラーが少なくなり、最も重要なことには、より安全になります。

CRDはどのように機能しますか？


セキュリティポリシーを定義するために、NeuVectorコンテナプラットフォーム内でNeuVector CRDを使用します。コンテナセキュリティのためのNeuVectorの代替：AquaSec、StackRox、Sysdig Secure、Twistlock。

NeuVector CRDには、最初に通常のアプリケーション動作の完全なプロファイルを収集するポリシーが含まれています。

収集された動作は、すべてのネットワークルール、プロセス、プロトコル、ファイル操作を含むホワイトリストに追加されます。これらすべてが、一連の標準アプリケーション操作を構成します。次に、アプリケーションを構成するコンテナ内で確認されたネットワーク接続のみを許可するセキュリティ設定が適用されます。これらの接続は、OSIモデルレイヤー7（アプリケーションプロトコルレイヤー）の検査中に識別されます。したがって、不正な外部接続の試みは防止されます。

セキュリティポリシーにより、攻撃者がコンテナの外部または内部の通信を使用してアプリケーションを自分の目的で使用することを防止します。

CRDでは、グローバルルールと各サービスのルールの両方を個別に定義できます。CRDはKubernetes RBACとも互換性があるため、サービスアカウントとKubernetesの役割を使用してセキュリティポリシーを適用できます。

さらに、アプリケーションのバージョンごとに独自のポリシーを作成するためのバージョニングを利用できます。OpenPolicy Agentなどのセキュリティポリシー管理ユーティリティとの統合がサポートされています。

準備完了-PrometheusとGrafanaに基づく特別に調整されたモニタリングシステムを備えたKubernetesクラスター、およびアクセス権の管理と分散チームでの開発の標準化のためのTLSとRBACは、Mail.ru Cloud Solutionsクラウドで無料でテストできます。

NeuVector CRDの作成


NeuVector CRDでは、ネイティブYAML Kubernetesファイルを使用してセキュリティルールを作成できます。

NeuVector CRDの説明を含むnvsecurityrule.yamlファイルを作成します。このファイルはNvSecurityRule、性質namespacedにNvClusterSecurityRule関連し、クラスターに関連します。

apiVersion: apiextensions.k8s.io/v1beta1
kind: CustomResourceDefinition
metadata:
  name: nvsecurityrules.neuvector.com
spec:
  group: neuvector.com
  names:
    kind: NvSecurityRule
    listKind: NvSecurityRuleList
    plural: nvsecurityrules
    singular: nvsecurityrule
  scope: Namespaced
  version: v1
  versions:
  — name: v1
    served: true
    storage: true

---

apiVersion: apiextensions.k8s.io/v1beta1
kind: CustomResourceDefinition
metadata:
  name: nvclustersecurityrules.neuvector.com
spec:
  group: neuvector.com
  names:
    kind: NvClusterSecurityRule
    listKind: NvClusterSecurityRuleList
    plural: nvclustersecurityrules
    singular: nvclustersecurityrule
  scope: Cluster
  version: v1
  versions:
  — name: v1
    served: true
    storage: true

CRDを作成するには、次のコマンドを実行します。

$ kubectl create -f nvsecurityrule.yaml

NeuVector CRDが作成されるとすぐに、後でパラメーターkind：NvSecurityRuleを使用して作成されたすべてのリソースがこのCRDによって処理されます。したがって、接続されたセキュリティポリシーを使用して独自のリソースを作成できます。何かを行う前に、NeuVectorのドキュメントを調べて、必要なclusterrolesand を追加することをお勧めしclusterrolebindingsます。

また、このCRDを使用してKubernetesクラスターにNeuVectorセキュリティポリシーを適用するには、適切な権限設定（RBAC）が必要です。任意のネームスペースに対してCRDによって定義されたセキュリティポリシーは、指定されたネームスペースに展開する権限を持つユーザーのみが適用できます。クラスタにセキュリティポリシーを適用するには、クラスタ管理者権限が必要です。

以下は、demo-security-v1.yamlのテストコードの一部です。これnginx-podは、名前空間内のコンテナーを制限しdemo、同じ名前空間の他のコンテナーへのアクセスをHTTP経由でのみ提供します。

apiVersion: v1
items:
- apiVersion: neuvector.com/v1
  kind: NvSecurityRule
  metadata:
    name: nv.nginx-pod.demo
  spec:
    egress:
    — Selector:
        criteria:
        — key: service
          op: =
          value: node-pod.demo
        — key: domain
          op: =
          value: demo
        name: nv.node-pod.demo
      action: allow
      applications:
      - HTTP
      name: nv.node-pod.demo-egress-0
      ports: any
    — Selector:
        criteria:
        — key: service
          op: =

スニペットの下には、名前空間内のコンテナに許可されているすべてのネットワーク接続（demoサーバーへの接続など）と、redis各コンテナに許可されているプロセスとディスクアクティビティの説明があります。最初にNeuVectorセキュリティポリシーをデプロイしてから、アプリケーションをデプロイすることを忘れないでください。したがって、セキュリティポリシーは、アプリケーションが起動した瞬間から有効になります。

セキュリティポリシーを適用するには：

$ kubectl create -f demo-security-v1.yaml

NeuVectorは、作成されたリソースのセキュリティポリシーを読み取り、REST APIを使用してNeuVectorコントローラーに接続します。NeuVectorコントローラーは、転送されたセキュリティポリシーに従ってルールと構成を作成します。

グローバルセキュリティポリシー


NeuVector CRDを使用すると、クラスター内の特定のアプリケーションまたはアプリケーションのグループに関連付けられていないグローバルセキュリティポリシーを定義できます。たとえば、セキュリティまたは実装チームは、グローバルネットワークルールを定義して、すべてのコンテナー内の接続をブロックするか、クラスター内のすべてのプロセスへの監視アクセスを構成します。



一般的なセキュリティポリシーとアプリケーションセキュリティポリシーを組み合わせて使用​​すると、組織に必要な複雑で正確なセキュリティ設定を構築できます。

以下は、コンテナーから外部へのssh接続を禁止する例です。

- apiVersion: neuvector.com/v1
  kind: NvClusterSecurityRule
  metadata:
    name: containers
    namespace: default
  spec:
    egress: []
    file: []
    ingress:
    — Selector:
        criteria: []
        name: external
      action: deny
      applications:
      - SSH
      name: containers-ingress-0
      ports: tcp/22
    process:
    — action: deny
      name: ssh
      path: /bin/ssh
    target:    
      Selector:
        criteria:
        — key: container
          op: =
          value: '*'
        name: containers
      policymode: null
    version: v1

テストから本番環境へのセキュリティポリシーの移行


NeuVector CRDを使用すると、テストポリシーから本番環境へのセキュリティポリシーの自動移行（すべてまたは必要なもののみ）を制御できます。NeuVectorコンソールでは、検出、監視、または保護のための新しいサービスのモードを構成できます。

監視モードまたは保護モードを選択することは、展開またはサービスの更新に、定義したセキュリティポリシーの設定が必ず含まれることを意味します。したがって、サービスは、セキュリティポリシーを適用した後でのみアクティブな状態になります。

開発者とDevOpsは、Kubernetes CRDの機能とセキュリティポリシーをコードとして使用して、アプリケーションのセキュリティポリシーの自動化を適用し、開発の初期段階から本番環境での作業まで、すべての段階でアプリケーションの保護を強化できます。
