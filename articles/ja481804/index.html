CompletableFutureのallOfおよびanyOfの改善



また会いましたね。コースの開始を見越して、「Java Developer」はあなたのために役立つ資料の翻訳を準備しました。

そこCompletableFutureデザインは私を驚かせる二つの方法があります：

  • CompletableFuture＃allOf
  • CompletableFuture＃anyOf


この記事では、それらの何が問題で、どのようにそれらをより便利にすることができるかを説明します。

CompletableFuture＃allOf


メソッドのシグネチャを見てみましょう：

public static CompletableFuture<Void> allOf(CompletableFuture<?>... cfs) {
    // ...
}


ここには少なくとも2つの論争の的な問題があります。

  1. このメソッドはCompletableFuture、異なるタイプのオブジェクトを返すいくつかのオブジェクトを受け入れます。
  2. メソッドが返すCompletableFuturereturnVoid


また、パラメーターの数が可変ではない場合があるので、この部分を見てみましょう。

CompletableFuture<Void>オペレーションの完了のシグナルとしてよく使用されますが、APIに小さな変更を加えることにより、このメソッドは、シグナリングデバイスとしても、完了したすべてのオペレーションの結果のキャリアとしても使用できます。やってみましょう。

非同期CompletableFuture＃allOf


最初に、正しい署名を考えましょう。

ほとんどの場合、同種のリストを処理して結果のリストをCompletableFuture返す必要があると想定するのは公平ですCompletableFuture。

public static <T> CompletableFuture<List<T>> allOf(
  Collection<CompletableFuture<T>> futures) {
    // ...
}


元のメソッドの内部は、おそらくあなたが期待するよりも複雑です：

static CompletableFuture<Void> andTree(
  CompletableFuture<?>[] cfs, int lo, int hi) {
    CompletableFuture<Void> d = new CompletableFuture<Void>();
    if (lo > hi) // empty
        d.result = NIL;
    else {
        CompletableFuture<?> a, b;
        int mid = (lo + hi) >>> 1;
        if ((a = (lo == mid ? cfs[lo] :
                  andTree(cfs, lo, mid))) == null ||
            (b = (lo == hi ? a : (hi == mid+1) ? cfs[hi] :
                  andTree(cfs, mid+1, hi)))  == null)
            throw new NullPointerException();
        if (!d.biRelay(a, b)) {
            BiRelay<?,?> c = new BiRelay<>(d, a, b);
            a.bipush(b, c);
            c.tryFire(SYNC);
        }
    }
    return d;
}


したがって、最初から作成するのではなく、完了シグナルデバイスとして使用するように意図されているかのように、元のメソッドにすでにあるものを再利用しようとします...次に、void結果を将来のリストに変更します。

CompletableFuture<List<CompletableFuture<T>>> i = futures.stream()
    .collect(collectingAndThen(
      toList(), 
      l -> CompletableFuture.allOf(l.toArray(new CompletableFuture[0]))
        .thenApply(__ -> l)));


ここまでは順調ですね。
CompletableFuture<List<CompletableFuture<T>>>代わりにCompletableFuture<Void>、なんとか手に入れました。しかし、結果のある未来のリストは必要ありません。結果のリストが必要です。

これで、リストを処理して、不要なフューチャーをリストから削除できます。CompletableFutureのjoinメソッドを呼び出すことは完全に正常です。なぜなら、これらのメソッドがブロックされることは決してないことがわかっているためです（この時点で、将来はすべて完了しています）。

CompletableFuture<List<T>> result = intermediate
    .thenApply(list -> list.stream()
        .map(CompletableFuture::join)
        .collect(toList()));


これをすべて組み合わせて最終的なソリューションにします。

public static <T> CompletableFuture<List<T>> allOf(
  Collection<CompletableFuture<T>> futures) {
    return futures.stream()
        .collect(collectingAndThen(
          toList(), 
          l -> CompletableFuture.allOf(l.toArray(new CompletableFuture[0]))
        .thenApply(__ -> l.stream()
           .map(CompletableFuture::join)
           .collect(Collectors.toList()))));
}


非同期で立ち下がりCompletableFuture＃allOf


例外がある場合、元のCompletableFuture＃allOfは残りのすべての操作が完了するまで待機します。

また、例外が発生したときに操作の完了を報告する場合は、実装を変更する必要があります。

これを行うには、新しいインスタンスCompletableFutureを作成し、いずれかの操作で例外が発生した後に手動で終了します。

CompletableFuture<List<T>> result = new CompletableFuture<>();
futures.forEach(f -> f
  .handle((__, ex) -> ex == null || result.completeExceptionally(ex)));


...しかし、すべての未来が正常に完了するときのシナリオに対処する必要があります。これは、改良されたallOf（）メソッドを使用して簡単に実行できます。その後、futureを手動で終了するだけです。

allOf(futures).thenAccept(result::complete);


これで、すべてを組み合わせて最終的なソリューションを形成できます。

public static <T> CompletableFuture<List<T>> 
  allOfShortcircuiting(Collection<CompletableFuture<T>> futures) {
    CompletableFuture<List<T>> result = new CompletableFuture<>();

    for (CompletableFuture<?> f : futures) {
        f.handle((__, ex) -> ex == null || result.completeExceptionally(ex));
    }

    allOf(futures).thenAccept(result::complete);

    return result;
}


CompletableFuture＃anyOf


メソッドシグネチャから始めましょう。

public static CompletableFuture<Object> anyOf(CompletableFuture<?>... cfs) {
    // ...
}


上記の方法と同じ問題をすぐに検出できます。

  1. このメソッドはCompletableFuture、異なるタイプのオブジェクトを含むいくつかのオブジェクトを受け入れます。
  2. このメソッドはCompletableFuture、型オブジェクトを含んで戻りますObject。


私の理解では、この方法CompletableFuture#allOfは信号装置として使用するように設計されています。しかしCompletableFuture#anyOf、この哲学と一致せず、CompletableFuture<Object>さらに混乱を招きます。

さまざまなタイプのデータを含むCompletableFutureを処理しようとしている次の例を見てください。

CompletableFuture<Integer> f1 = CompletableFuture.completedFuture(1);
CompletableFuture<String> f2 = CompletableFuture.completedFuture("2");

Integer result = CompletableFuture.anyOf(f1, f2)
  .thenApply(r -> {
      if (r instanceof Integer) {
          return (Integer) r;
      } else if (r instanceof String) {
          return Integer.valueOf((String) r);
      }
      throw new IllegalStateException("unexpected object type!");
  }).join();


かなり不快ですね。

幸いなことに、これは、シグネチャを変更して直接型キャストを導入することで、より一般的なシナリオ（同じ型の値を含む多くの将来の1つを待つ）に適応するのが非常に簡単です。

したがって、改善により、既存のメソッドを再利用して安全に結果をもたらすことができます。

public static <T> CompletableFuture<T> anyOf(List<CompletableFuture<T>> cfs) {
    return CompletableFuture.anyOf(cfs.toArray(new CompletableFuture[0]))
      .thenApply(o -> (T) o);
}
 
public static <T> CompletableFuture<T> anyOf(CompletableFuture<T>... cfs) {
    return CompletableFuture.anyOf(cfs).thenApply(o -> (T) o);
}
