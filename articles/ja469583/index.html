OpenVPNおよびActive Directory（ユーザー証明書なしのKerberos）

OpenVPNについては、Active Directoryによる認証を含め、多くのガイドが書かれています。それらのほとんどはLDAPを使用するように煮詰められます;記事全体に書かれているKerberosを使用するアプローチは見つかりませんでした。ただし、ここでは新しいものは発明されません。すでに実行されていることをコンパイルしただけで、PAMはKerberosで正常に動作し、OpenVPNにはネイティブのPAMプラグインがあります。私はまた、ユーザー証明書と、すべてのユーザーが1つの証明書を発行することを勧めるヒントを放棄することにしました。それらは私には適さなかったため、すべてのクライアントで機能する別の解決策が見つかりました。



NSSについて少し


私の場合、Centos 7をsssdと組み合わせて使用​​しました。ただし、winbindの使用を妨げるものはありません。主なことは、ドメイン内に車とKerberosチケットを持つことです。これがどのように行われるかについては書きません。私は何も新しいことは言いませんが、ネットワークにはたくさんの良いマニュアルがあります。

これまでに行ったこと/etc/nsswitch.confがない場合は、設定を変更する必要があります。使用したモジュールに応じて、module_nameの代わりに、それぞれsssまたはwinbind を入力する必要があります。パラメータの説明はで読むことができますman nsswitch.conf。

次の設定で十分です。

passwd:           files module_name
shadow:           files module_name
group:            files module_name

sssdの場合、とりわけ、/etc/sssd/sssd.confnssが[sssd]ブロックで記述されることを忘れないでください。

services = nss, pam

winbindについては、私は何かを言うのは難しいので、使用しませんでした。

最終的な目標は、getentから正しい結果を取得することです。マシンがドメインユーザーとグループに関する情報を受信できることを確認します。

> getent passwd kanlas
kanlas:*:14123583:1257570:Kanlas Kanlasovich:/home/kanlas@example.com:/bin/bash
> getent group VPN
vpn:*:13821391:kanlas,igor,marina

注意
sssd (.. kanlas@example.com), sssd.conf use_fully_qualified_names = False

PAMを構成する


次に、独自のPAMモジュールを作成する必要があります。システムのaccess.confを使用しないために、アクセスグループが読み取られる場所から別のファイルを作成します。私はPAMモジュールがあまり得意ではないので、以下の例ではすべての行が必要かどうかはわかりません。一般的に、関心があるのは文字列だけ account required pam_listfile.so onerr=fail item=group sense=allow file=/etc/openvpn/auth/access-groupsです。

最も簡単なオプションは、たとえば、ログインモジュールを取得して、そこに目的の行を追加することです。それが行われました。

我々が得る：

	
#%PAM-1.0
auth [user_unknown=ignore success=ok ignore=ignore default=bad] pam_securetty.so
auth       substack     system-auth
auth       include      postlogin
account    required     pam_listfile.so onerr=fail item=group sense=allow file=/etc/openvpn/auth/access-groups
account    include      system-auth
password   include      system-auth
session    required     pam_selinux.so close
session    required     pam_loginuid.so
session    optional     pam_console.so
session    required     pam_selinux.so open
session    required     pam_namespace.so
session    optional     pam_keyinit.so force revoke
session    include      system-auth
session    include      postlogin
-session   optional     pam_ck_connector.so

それをに入れて、/etc/pam.d私たちの裁量で呼び出します。

では/etc/openvpn/auth/access-groups、モジュールに示されているように、アクセスグループが記録されます。必要に応じてパスを指定できます。ドメインのグループの名前（共通名）を指定する必要があります。私の場合、VPN @ example.comをそこに書いた。

PAMを使用する必要があることをOpenVPNに説明する必要があります。サーバー構成に次の行を追加します。

plugin /usr/lib64/openvpn/plugins/openvpn-plugin-auth-pam.so __pam
username-as-common-name

ディストリビューションによってパスが異なる場合があります。パスでファイルを確認するか、findを使用します。

username-as-common-nameパラメータを使用するのは、それ以外の場合は、クライアントがauth-user-passを取得した後、証明書で指定された名前がサーバーに送信されるためです。

クライアント構成で、行を追加します

auth-user-pass

このセットアップが完了したら、テストできます。

ユーザー証明書を削除します


これで、ドメインを介した承認が得られ、ユーザー証明書は役に立たなくなり、論理的な決定のように見捨てられます。しかし、モバイルクライアントはこれに問題があり、単にそのような構成を受け入れないのに対し、デスクトップの構成は機能しますが、怒っています。最初に、証明書が不要になるようにサーバーを再構成します。

verify-client-cert none

次に、クライアント構成に追加します

setenv CLIENT_CERT 0

それだけです、今では顧客は証明書を誓うことはありません。このオプションがiOSの質問と回答のセクションに隠されているのはおかしいです。私はiOSでテストしていませんが、Android、およびMacOS（ベータクライアント）とWindowsはこれらの設定を一気に受け入れます。AndroidのサードパーティOpenVPNクライアントだけがまだこれを行うことができません（この記事の執筆時点では、すでに修正できていました）。
