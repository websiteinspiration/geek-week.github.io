HAProxyでリクエストの頻度を制限する方法：ステップバイステップの説明


著者は、特定のIPアドレスでHAProxy クエリ速度制限（レート制限）を実装する方法を説明します。Mail.ruクラウドソリューションチームが彼の記事を翻訳しました。これにより、費やす時間と労力を費やす必要がなくなることを願っています。

実際、これはサーバーをDoS攻撃から保護する最も一般的な方法の1つですが、インターネット上でサーバーを具体的に構成する方法を明確に説明するのは困難です。試行錯誤により、作者はHAProxyに、リアルタイムで更新されるIPアドレスのリストに対する要求の頻度を制限するように強制しました。

HAProxyを設定するために事前の知識は必要ありません。必要な手順はすべて以下に概説されています。

オープンソースで無料のHAProxyは、非常に手頃な価格のロードバランサーとプロキシサーバーです。近年、最小限のリソースで高いパフォーマンスを提供するため、非常に人気があります。代替プログラムとは異なり、HAProxy Community Editionの非営利バージョンは、信頼できるロードバランシングに十分な機能を提供します。

このプログラムは、最初は理解するのがかなり難しいです。ただし、彼女は非常に綿密で詳細な技術文書を持っています。これは彼がこれまでに使用したすべてのオープンソースプログラムの中で最も詳細なドキュメントであると著者は言います。
だから、ここにステップバイステップガイドがあります。

制限の設定


すべての設定は構成ファイルに登録されますhaproxy-ratelimiter.cfg。注意深く調べましょう。

defaults
mode http
timeout connect 5000ms
timeout client 50000ms
timeout server 50000ms

frontend proxy
bind *:80

# ACL function declarations
acl is_abuse src_http_req_rate(Abuse) ge 10
acl inc_abuse_cnt src_inc_gpc0(Abuse) gt 0
acl abuse_cnt src_get_gpc0(Abuse) gt 0

# Rules
tcp-request connection track-sc0 src table Abuse
tcp-request connection reject if abuse_cnt
http-request deny if abuse_cnt
http-request deny if is_abuse inc_abuse_cnt

use_backend api
backend api
balance roundrobin

server api01 api01:80
server api02 api02:80

backend Abuse
stick-table type ip size 100K expire 30m store gpc0,http_req_rate(10s)

HAProxyは一連の低レベルのプリミティブを提供し、柔軟性を高め、さまざまな使用例に適しています。そのカウンタは、CPUの累積レジスタ（加算器）を思い出させてくれます。中間結果を保存し、さまざまなセマンティクスを入力として受け取りますが、結局のところ、それらは単なる数値です。すべてをよく理解するには、設定ファイルの最後から始めるのが理にかなっています。

テーブル Abuse


backend Abuse
stick-table type ip size 100K expire 30m store gpc0,http_req_rate(10s)


ここでは、Abuse（ "abuse"）というダミーのバックエンドをセットアップします。架空のものです。これは、残りの構成が名前で参照できるスティックテーブルにのみ使用されるためAbuseです。スティックテーブルは、プロセスメモリに保存されたテーブルであり、各レコードについてライフタイムを決定できます。

私たちのテーブルには次の特徴があります。

  • type ip：リクエストはIPアドレスをキーとしてテーブルに保存されます。したがって、同じIPアドレスからの要求は同じレコードを参照します。基本的に、これはIPアドレスと関連データを追跡していることを意味します。
  • size 100K：テーブルには最大10万件のレコードが含まれます。
  • expire 30m：レコードの保持期間は30分の非アクティブです。
  • store gpc0,http_req_rate(10s)：gpc0最後の10秒間のカウンターとIPアドレス要求の数がエントリと共に保存されます。助けを借りて、gpc0虐待でIPアドレスが認識された回数を追跡します。実際、正のカウンタ値は、IPアドレスがすでに疑わしいとマークされていることを意味します。このカウンターを呼びましょうabuse indicator。

一般に、この表でAbuseは、IPアドレスが疑わしいものとしてマークされているかどうか、およびこのアドレスからの要求の現在の頻度を追跡できます。したがって、私たちは記録の履歴とリアルタイムの情報を持っています。

アクセス制御リスト（ACL）は、ルールが設定されている場合にのみ呼び出される関数宣言です。

3つすべてのACLエントリを詳しく見てみましょう。これらはすべて、AbuseIPアドレスをキーとして使用するテーブルを明示的に参照しているため、各関数は要求IPアドレスに適用されることに注意してください。

  • acl is_abuse src_http_req_rate(Abuse) ge 10：現在の要求頻度が10以上の場合、関数is_abuseは戻りますTrue。
  • acl inc_abuse_cnt src_inc_gpc0(Abuse) gt 0：関数inc_abuse_cntはTrue、増分値gpc0がゼロより大きい場合に戻ります。初期値gpc0はゼロなので、この関数は常にを返しますTrue。つまり、値が増加しabuse indicator、基本的にこのIPアドレスからの不正行為を通知します。
  • acl abuse_cnt src_get_gpc0(Abuse) gt 0：関数abuse_cntはTrue、値がgpc0ゼロより大きい場合に戻ります。言い換えれば、このIPアドレスがすでに悪用されているかどうかを確認します。

前述のように、ACLは単なる宣言、つまり関数宣言です。いくつかのルールがトリガーされるまで、それらは着信要求には適用されません。

同じセクションで定義されているルールを確認することは理にかなっていますfrontend。ルールは各着信要求に1つずつ適用され、定義したACLから関数を実行します。

各ルールの機能を見てみましょう：

  • tcp-request connection track-sc0 src table Abuse：テーブルにクエリを追加しますAbuse。キーはテーブル内のIPアドレスであるため、このルールはIPアドレスのリストに追加されます。
  • tcp-request connection reject if abuse_cnt: TCP-, IP- , abuse. , TCP- IP-.
  • http-request deny if abuse_cnt: , IP- . IP-, abuse.
  • http-request deny if is_abuse inc_abuse_cnt: , is_abuse inc_abuse_cnt True. , , IP- , IP- .

本質的には、リアルタイムとクエリ履歴のブラックリストの2種類のチェックを導入します。2番目のルールは、IPアドレスが悪用されている場合、すべての新しいTCP接続を拒否します。3番目のルールは、このアドレスからの要求の現在の頻度に関係なく、ブラックリストからのIPアドレスに対するHTTP要求のサービスを禁止します。4番目のルールは、要求頻度のしきい値を超えるとすぐに、IPアドレスからのHTTP要求が拒否されることを保証します。したがって、2番目のルールは主に新しいTCP接続で機能し、3番目と4番目のルールは既に確立された接続で機能します。1番目は履歴チェック、2番目はリアルタイムチェックです。

ご質問


10秒あたり10リクエストに制限されているのはなぜですか？


テーブルにはAbuse判断しhttp_req_rate(10s)、要求の頻度は10秒のウィンドウで測定されています、。is_abuseACLの関数はTrue、指定された間隔内で≥10の要求レートで戻ります。したがって、乱用は、10秒間に10回以上のリクエストが発生する頻度と見なされます。

たとえば、この記事では、リミッターの動作を確認しやすいように下限を設定することにしました。

http-requestとtcp-requestの接続ルールの違いは何ですか？


ドキュメント から：

http-request：http-requestステートメントは、ネットワーク層7で適用される一連のルールを定義します[OSIモデル]

ドキュメント から：
tcp-request connection：ネットワーク層4の状態に応じて、着信接続でアクションを実行します

HTTP-, TCP-?


サーバーへのHTTPリクエストが同じIPアドレスから複数のTCP接続を送信するとします。HTTP要求の頻度はすぐにしきい値を超えます。次に、4番目のルールが有効になり、リクエストが破棄され、IPアドレスがブラックリストに登録されます。

これで、同じIPアドレスからのHTTP接続が開いたままになり（永続的なHTTP接続を参照）、HTTP要求の頻度がしきい値を下回った可能性があります。3番目のルールはabuse indicator、このIPでトリガーされるため、HTTPリクエストの継続的なブロックを保証します。

ここで、数分後に同じIPがTCP接続の確立を試行するとします。2番目のルールが適用されるため、これらはすぐにドロップされます。ラベル付けされたIPアドレスを確認して、すぐに接続をドロップします。
