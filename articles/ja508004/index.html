チャペル



チャペルはスレッドを並列化するためにforallループを使用します。各要素へのCポインターも使用され、配列への標準参照ではなく、インデックスによるガイド付き反復が適用されます。 



proc calculateKernelMatrix(K, data: [?D] ?T)
{
  var n = D.dim(0).last;
  var p = D.dim(1).last;
  var E: domain(2) = {D.dim(0), D.dim(0)};
  var mat: [E] T;
  var rowPointers: [1..n] c_ptr(T) =
    forall i in 1..n do c_ptrTo(data[i, 1]);

  forall j in guided(1..n by -1) {
    for i in j..n {
      mat[i, j] = K.kernel(rowPointers[i], rowPointers[j], p);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


チャペルコードは、パフォーマンスを最適化するのが最も困難で、コードの変更もほとんど必要でした。



D



Dコードを並列化するには、std.parallelパッケージのtaskPoolストリームを使用します。Dコードは、パフォーマンスを最適化するために最小限の変更しか加えられていません。特定のコンパイラーと選択されたコンパイルキー（後で説明）を使用することは大きな利点です。私のMatrix実装では、refColumnSelectを使用して参照によって列を選択できます。 



auto calculateKernelMatrix(alias K, T)(K!(T) kernel, Matrix!(T) data)
{
  long n = data.ncol;
  auto mat = Matrix!(T)(n, n);

  foreach(j; taskPool.parallel(iota(n)))
  {
    auto arrj = data.refColumnSelect(j).array;
    foreach(long i; j..n)
    {
      mat[i, j] = kernel(data.refColumnSelect(i).array, arrj);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


ジュリア



ジュリアコードはマクロを使用しています スレッド コードとマクロを並列化するため 景色配列への参照用。Juliaで配列と混同されているのは、参照状態だけです。この場合のように、配列は値オブジェクトとして動作し、マクロを使用して参照する必要がある場合があります景色それ以外の場合は、コピーが生成されます。他の場合では、たとえば、それらを関数に渡すときに、参照オブジェクトのように動作します。コピーが生成する操作のセットが常にわからないが、どこで発生するかを理解するのは、少し扱いに​​くいかもしれません。景色良い解決策を提供します。



Symmetricタイプは、マトリックスに反映するために必要な少し余分な作業を節約します。 



function calculateKernelMatrix(Kernel::K, data::Array{T}) where {K <: AbstractKernel,T <: AbstractFloat}
  n = size(data)[2]
  mat = zeros(T, n, n)
  @threads for j in 1:n
      @views for i in j:n
          mat[i,j] = kernel(Kernel, data[:, i], data[:, j])
      end
  end
  return Symmetric(mat, :L)
end


メイン関数 の@境界と@ simdマクロを使用して、境界チェックを無効にし、計算にSIMD最適化を適用しました。 



struct DotProduct <: AbstractKernel end
@inline function kernel(K::DotProduct, x::AbstractArray{T, N}, y::AbstractArray{T, N}) where {T,N}
  ret = zero(T)
  m = length(x)
  @inbounds @simd for k in 1:m
      ret += x[k] * y[k]
  end
  return ret
end


これらの最適化はかなり顕著な増加をもたらしますが、非常に使いやすいです。

パフォーマンスの最適化



3つの言語すべてでパフォーマンスを最適化するプロセスは非常に異なり、3つのコミュニティすべてがこのプロセスで非常に役に立ちました。しかし、一般的なポイントがありました。



  • ポリモーフィズムを使用する代わりに、カーネル関数の静的スケジューリング。これは、カーネル関数を転送するときに、パラメトリック（コンパイル時間）ポリモーフィズムが使用され、動的（ランタイム）ポリモーフィズムが使用されないことを意味します。仮想ファンクションを使用したスケジューリングではオーバーヘッドが発生します。
  • マルチスレッドモードでデータをコピーする代わりにビュー/リンクを使用することは非常に重要です。
  • 計算の並列化は非常に重要です。
  • 配列は行/列の基本であることを理解し、これを計算に使用することが最も重要です。
  • 境界チェックとコンパイラの最適化は、特にチャペルとDで大きな違いをもたらします。
  • DとJuliaにSIMDを含めることで、パフォーマンスが向上しました。Dでは、これは-mcpu = nativeフラグを使用して行われ、Juliaでは、@ simdマクロを使用して行われました。

ここでの「配列」は、Dで使用可能なCおよびC ++のスタイルの配列ではなく、数学的配列を指します。 JuliaとChapelには配列サポートが付いていますが、Dにはありませんが、多次元配列（ndslice）を含むMirライブラリがあります。マトリックスコアの計算の実装では、マトリックスオブジェクトをDで記述しました。これは、原理を理解していれば簡単ですが、これはユーザーが望んでいることではありません。ただし、Dには線形リューベック代数ライブラリがあります。、優れたパフォーマンス機能を備え、すべての一般的なBLAS実装とのインターフェースを備えています。もちろん、ジュリアの配列は、最も単純で最もよく知られています。チャペルアレイは、ジュリアアレイよりもエントリーレベルの方が複雑ですが、シングルコア、マルチコアシステム、および単一または非常に類似したコードを使用するコンピュータークラスターで実行できるように設計されています。

ジュリアは動的プログラミング言語であるため、「ジュリアは静的プログラミング言語よりもはるかに寛容な動的言語なので、議論は終わった」と言う人もいるかもしれませんが、それははるかに複雑です。静的型システムにはその力があります。 Juliaには、静的言語の型システムと性質が似ている型システムがあるため、静的言語を使用しているかのようにコードを記述できますが、動的言語に対してのみ予約された処理を実行できます。これには、一般的なメタプログラミング用の高度に開発された構文と、強力なマクロがあります。また、非常に柔軟なオブジェクトシステムと複数のディスパッチがあります。この機能の組み合わせにより、Juliaは3つの言語の中で最も強力な言語になります。



DはC ++の代替として考えられ、C ++から多くを取り入れました（Javaからも借りました）が、テンプレートプログラミングとコンパイル時計算（CTFE）はC ++よりはるかにユーザーフレンドリーになります。これは単一のディスパッチを持つ言語です（ただし、マルチメソッドを持つパッケージはあります）。Dには、マクロの代わりに、同様の目的を果たす文字列とパターンの「ミックスイン」があります。



チャペルは、単一のディスパッチによるOOPの一般的なサポートと初期のサポートを備えています。マクロのサポートはありません。これらの問題では、Dやジュリアほど成熟していません。



競争力と並行プログラミング



現在、新しい言語は当初、人気のある並列処理のサブセットとの競争を維持していますが、詳細は言語によって大きく異なります。この例では並行性がより重要であり、3つの言語すべてがそれを提供します。並列ループの作成は、3つの言語すべてで簡単です。



チャペル並行性モデルでは、データ並行性により多くの注意が払われますが、タスク並行性のためのツールがあり、クラスターベースの並行性サポートが提供されます。



Juliaは、並行性と並行性の両方を十分にサポートしています。



Dは、同時実行性と同時実行性を産業的に強力にサポートしていますが、マルチスレッドのサポートについては、例が十分に文書化されていません。



標準ライブラリ



3つの言語すべての標準ライブラリは一般的にどの程度優れていますか？ユーザーが簡単に実行できるタスクは何ですか？これは、ライブラリの品質とドキュメント化の要因を考慮に入れるため、難しい質問です。3つの言語すべてに非常に優れた標準ライブラリがあります。Dには最も完全な標準ライブラリがありますが、ジュリアは2番目に優れており、次にチャペルですが、すべてがそれほど単純ではありません。たとえば、バイナリI / Oを記述したいユーザーは、ジュリアが初心者にとって最も簡単だと感じるかもしれません。IDXファイルを読み取るためのプログラムの実装ではI / O Dが最速でしたが、Juliaコードは以下で利用できない場合のために簡単に記述できました。他の2つの言語。



マネージャーとパッケージのエコシステム



ドキュメント、使用法、機能の点で、D-Dubパッケージマネージャーが最も包括的です。DもダブのWebサイトにパッケージの豊富なエコシステムを持っていますが、ジュリアのパッケージマネージャーはGitHubと緊密に統合されており、優れたドキュメントを備えた優れたパッケージシステムです。チャペルにはパッケージマネージャーがありますが、高度に開発されたエコシステムはありません。



C統合



Cの相互運用性は、3つの言語すべてで簡単に使用できます。チャペルには優れたドキュメントがありますが、他のものほど人気が​​ありません。Dのドキュメントの方が優れており、Juliaのドキュメントが最も包括的です。ただし、奇妙なことに、独自のCコードをコンパイルして言語に統合するために必要なコマンドが1つの言語のドキュメントに含まれていないため、特に初心者にとっては見落としです。ただし、DおよびJuliaでは、コンパイルプロセスの例を簡単に検索して見つけることができます。

概要



数値アルゴリズムを作成して科学的計算を行う初心者プログラマーで、高速で使いやすい言語が必要な場合は、ジュリアが最適です。あなたが同じ分野で働いている経験豊富なプログラマーなら、ジュリアはまだ素晴らしい選択肢です。すべての「ホイッスルブロワー」を備えた、より伝統的な「工業用」の静的にコンパイルされた高性能言語が必要だが、C ++より生産性が高く、安全で痛みの少ないものが必要な場合は、Dが最適なオプションです。Dで「何でも」書くことができ、そのコンパイラーのおかげで優れたパフォーマンスを得ることができます。クラスターで配列を計算する必要がある場合、Chapelがおそらく最も便利なソリューションです。



このタスクの総パフォーマンスの点では、Dが勝者であり、9つの参照タスクのうち5つで最高のパフォーマンスを明確に示しています。この研究は、高性能言語としてのジュリアのラベルは単なる誇大宣伝ではなく、競争の激しい言語と比較して独自の利点があることを示しています。チャペルから競争力のあるパフォーマンスを得るのは予想以上に困難でした-チャペルチームは現在のソリューションを思いつくために多くの研究を必要としました。しかし、チャペルの言語が成熟するにつれて、さらなる改善が見られます。
