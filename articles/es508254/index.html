Si usas moki, ¿estás probando al menos algo?







¿Sintió que por el simple hecho de probar está haciendo que el código sea más difícil de leer? Digamos que tiene un código que aún no se ha probado. Tiene una serie de efectos secundarios, y se le pide que ejecute las pruebas primero. Empiezas a seguir consejos como pasar variables globales como parámetros o extraer efectos secundarios problemáticos para crear stubs en las pruebas .



Pero cuanto más avanzas, más te sientes a gusto. Tiene un montón de comprobantes acumulados para estados y datos que le permiten escribir pruebas unitarias rápidas y confiables . Como resultado, tiene la sensación de haber creado una clase completamente diferente que no reproduce con precisión el código real.



Te detienes y piensas: “¿Está permitido cambiar las firmas de código por el simple hecho de probar? ¿Estoy probando código real o una clase completamente diferente en la que lo que se necesita no sucede? Puedes tener un dilema. ¿Está seguro de que debe seguir adhiriéndose a este enfoque? ¿O es una pérdida de tiempo?



Pregunta número un millón: ¿necesita escribir pruebas unitarias o pruebas de integración para código obsoleto?

Una mirada práctica a las pruebas



Me sumergí en este problema por un tiempo e intenté diferentes enfoques. La decisión se produjo después de una discusión memorable con JB Rainsberger .



Me di cuenta de que no hay paradoja. Hay diferentes definiciones de un "módulo".



Me parece que la palabra "módulo" es confusa. Las personas tienen una comprensión diferente de lo que es. Por lo general, los recién llegados a las pruebas consideran una función o método como un módulo. Entonces entienden que puede ser una clase o módulo completo. Sin embargo, la verdad, como mucho más en la vida, depende de la situación.









Pruebas aisladas



Creo que la división en pruebas "modulares" y de "integración" no es lo suficientemente clara. Dicha categorización genera problemas y disputas, aunque tenemos un objetivo: facilitar el cambio de software con la ayuda de comentarios rápidos y precisos cuando se rompe algo.



Por lo tanto, prefiero hablar sobre pruebas "aisladas".



No estoy seguro si este término es mejor. Pero me gusta por ahora, porque me hace hacer una pregunta extremadamente importante: ¿ aislado de qué?



Mi respuesta: aislado de lo que es difícil de probar.



¿Qué es difícil de probar?



Accidente, tiempo, red, sistema de archivos, bases de datos, etc.



Llamo a toda esta "infraestructura", y el resto se llama un "dominio". La combinación de dominio e infraestructura parece una mirada muy útil y práctica al diseño de software. Y no solo para mí. Esta separación está en el corazón de muchas arquitecturas mantenibles, como la Arquitectura Hexagonal (conocida como Puertos y Adaptadores), Arquitectura de Cebolla, Arquitectura Limpia y Núcleo Funcional / caparazón imperativo, etc.



Todos estos son los matices que dicen una cosa: es mejor separar el dominio y la infraestructura. La programación funcional también promueve esta idea: efectos secundarios aislados en el perímetro de su sistema. Los efectos secundarios son infraestructura. Y es difícil de probar.



Sí, pero ¿cómo ayuda?



En resumen, esto le dice exactamente qué quitar y reemplazar con tapones. No transfiera la lógica de negocios a la infraestructura, de lo contrario, será atormentado con las pruebas.



Un dominio aislado es fácil de probar. No necesita hacer apéndices para otros objetos de dominio. Puede usar el mismo código que se usa en el producto. Solo necesita deshacerse de la interacción con la infraestructura.



“¡ Pero puede haber errores en el código de infraestructura! »



Si lo mantiene lo suficientemente compacto, los riesgos se reducen. Pero básicamente, reduce el tamaño del código que necesita probar cuando se integra con una base de datos real.



La conclusión es esta: debe poder probar todo tipo de comportamiento de su aplicación sin guardar datos en PostgreSQL. Necesita una base de datos en memoria. Todavía necesita verificar la integración con PostgreSQL, pero solo unas pocas pruebas son suficientes , ¡y esta es una diferencia notable!



Todo era una teoría, volvamos a la realidad: tenemos un código que aún no se ha probado.



Volviendo al código obsoleto



Creo que esta visión de las pruebas te ayudará. No es necesario dejarse llevar por los talones, pero cuando trabaje con código obsoleto tendrá que dejarse llevar. Temporalmente. Porque este código es una vinagreta por el hecho de que es difícil de probar (infraestructura) y lógica de negocios (dominio).



Debe esforzarse por extraer el código de infraestructura del dominio.



Luego haga que la infraestructura dependa del dominio, y no al revés . Debe haber la menor infraestructura posible en la lógica empresarial. Obtendrá pruebas rápidas, confiables y aisladas que cubren la mayor parte de la lógica de la aplicación, y varias pruebas de integración para probar el funcionamiento del código de infraestructura con una base de datos real, API externa, reloj del sistema, etc.



¿Todavía tienes trozos? Si y no.



Tendrá implementaciones de infraestructura alternativas, por lo que aún puede ejecutar la lógica de producción sin usar una base de datos real.



¿Cómo descubrir que la implementación de la base de datos funciona correctamente? Escribe algunas pruebas de integración. Solo lo suficiente para verificar la corrección del comportamiento. JB Rainsberger lo llama "Pruebas de contrato". Creo que es mejor explicarlo con este ejemplo .
