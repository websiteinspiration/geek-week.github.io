React周辺機器の新しいフロントエンドライブラリ

こんにちは、ハブル！

今日、私たちは、フロントエンドが私たちの時代、または著者によると、Reactの後でどのように発展しているかについての興味深いテキストを提供したいと思いました。おそらく、投稿はもっとボリュームがあるかもしれません。そのため、作者が続編を書く場合は、見逃さないように翻訳します。

猫へようこそ！

かろうじて登場した後、Reactはすぐにゲームのルールを変更しました。彼はクライアントサイド開発の新時代を先導しました。その単方向のデータストリーム、JSX、および表現を定義する宣言的な方法は、ただクレイジーでした。 Reactは急速に絶大な人気を獲得し、6年経った現在でも、最も人気のあるフロントエンドライブラリの1つとして残っています。

ただし、フロントエンドのエコシステムは急速に発展しており、この分野では多くの興味深いプロジェクトが開発されています。私は常に、ゲームのルールを再び変える次のテクノロジーを探しています。昨日の部外者がエースに出て、6年前のReactのように、私たちの通常のアプローチが完全に損なわれてしまうのが好きです。

これまで以上に生産性を高めるのに役立つ新しいテクノロジーを見つける喜びは貴重です。自分がReactのスペシャリストだと思っていても、React以外に何が存在するかを調べることは非常に重要です。優れた開発者は、問題を解決するために必要なツールを選択します。彼は多くのパラダイム、言語、ツールを知っている必要があります。

最近、私はいくつかの調査を行い、実験を行ったところ、非常に興味深いライブラリーと、私の意見では潜在的な可能性がある新しいWeb標準が見つかりました。この投稿でお話しする内容はすべて、本番環境で使用できます。これらの技術を「最前線」と呼ぶことはできませんが、これまでのところ、広く配布されていません。

スヴェルト


すでに存在するものを変えるのではなく、新しい考え方を教え込む完全に新しいフレームワークから始めましょう。これはスヴェルトです。これは、クライアントエコシステムの新しいパラダイムです。



では、なぜSvelteはユニークなのでしょうか？主に2つの理由があります。まず、ランタイム環境がパッケージに含まれていないため、非常に小さなパッケージが作成されます。スヴェルト支持者はこれを「ゼロ実行環境」と呼んでいます。 Reactランタイムの重量は約100 kbであり、この量は各Reactアプリケーションのパッケージを超えて追加されます。

Svelteの2番目の重要な長所は、その構文です。構文がまったくないかのように、それは非常に最小限です。これに戻ります。

SvelteはReactとVueから多くを借りています。Reactの経験があれば、すでに多くを知っているので、Svelteを習得するのにほとんど時間はかかりません。

Svelteの印象を与えるいくつかのコードを見てみましょう。

新しいコンポーネントを定義するために、新しいファイルが作成され、このファイルの名前がコンポーネントの名前になります。このファイルには、オプションのスクリプトセクション、オプションのスタイルセクション、そして重要なことに、HTMLテンプレート自体があります。SvelteはJSXを使用しませんが、独自のテンプレート言語を使用します。

<script>
    export let counter; //     
</script>

<style>
    p {
        color: purple;
    }
</style>

<p>The counter is {counter}</p>

Vueに少し似ていますが、より単純です。

構文...そして特定の構文はありません、それは単なるJavaScriptです。ステータスを設定したいですか？変数を増やすだけですcounter。

<script>
    let count = 0;

    function handleClick() {
        count += 1;
    }
</script>

<button on:click={handleClick}>
    Clicked {count} {count === 1 ? 'time' : 'times'}
</button>

いいえthis.setState。フックなしuseState。ただのJavaScript。このような小さな例では、すべての最小限の構文を評価することは困難です。私はこれを数時間プログラムしましたが、問題を解決するためにはるかに少ないコードを記述する必要があったため、非常に生産的になりました。走っているように見え、誰かが後ろを押してくれた。浮気しているように。このような最小限の構文は、Svelteの作成者が非常に意識的に行った決定です。彼ら自身が投稿で説明します。この記事からの抜粋は次のとおりです。

...コードベースの増加に伴い、プロジェクトの開発時間とバグの数は、線形ではなく2次的に増加します。-より少ないコードを書く

Webフレームワークの最も重要なコンポーネントは、そのドキュメントです。 Reactはその素晴らしいドキュメントで有名であり、多くの人がドックでのみそれを学びます。 Svelteのドキュメントも、私が見る限り、非常に優れています。 Svelteの初心者として、私はこのフレームワークのドキュメントの例の中で最も気に入っています。



多くの場合、必要なものはだいたいわかっています。たとえば、コンポーネントの状態を変更し、イベントハンドラーを追加し、グローバルステータスを取得します（Reduxなど）。次に、これがどのように行われるかをすばやく確認する必要があります。例のセクションは、この種の例です。ブログの記事をGoogleで検索したり、Githubでリストを見たりする必要はありません。必要なセクションを参照してください。美しさ！

GatsbyやNext.jsに似たものもあります。呼ばれるものSapper、あなたはそれを非常に素早くコーディングし始めます。静的に生成されたHTMLページを使用して単一ページのアプリケーションを作成できます。これは、Gatsby / Next.jsに似ています。

Svelte Reduxには、stores（stores）と呼ばれる代替手段もあります。彼らは信じられないほどミニマルです。ストレージは次のように作成されます：

import { writable } from 'svelte/store';
export const count = writable(0);

その後、このようにリポジトリを更新できます。

count.set(0);

典型的なコードはまったくなく、この機能はすべてSvelteに完全に組み込まれており、追加の依存関係はありません。ここで、ReactプロジェクトにReduxを追加したときの問題を思い出してください。

ReactチームはSvelteとどのように関係していますか？


Reactチームは、私ほどスヴェルトに熱心ではありません。彼らは、ランタイム環境の破棄に関連するパフォーマンスの向上は無視できると信じています。非常に特殊な場合にのみ、たとえば組み込みウィジェットを使用する場合など。そのようなゲームはキャンドルの価値があります。さらに、ギャツビーとネクストがあるので、Reactコミュニティによると、速度が少し上がると、さらに遅くなります（申し訳ありませんが、どこで読んだか覚えていません。見つけたら、Twitterで知らせてください、@ karljakoblind）この場合、速度の改善が重要であり、「これはウィジェットでのみ役立つ」というのは悪い議論であるという理由だけで、各パッケージで100 kbの節約を一掃することが重要だと私は主張します。Webを優れた手頃な価格にするためには、可能な限りあらゆる種類のパフォーマンス最適化を行う必要があります。アプリケーションのパフォーマンスを改善するための微調整に携わったことがある場合は、わずかな改善でもどれだけの意味があるかがわかります。これは、魔法のように発生する超高速ダウンロードにつながる小さな改善すべての合計です。

だから、私はスベルトを試すべきですか？


次回は、Reactでうまく機能するプロジェクトを開始するときに、代わりにSvelteを使用するかどうかを真剣に考えます。ドキュメントを読んでいない場合でも、スヴェルトを研究していて、このフレームワークの生産性に驚いた。私がやったことはすべて、例を使って作業し、Sapperで少し練習することだけでした。それでも、この作業はReactよりも生産的であるように思えました。このような高い生産性は、どういうわけかミニマリストの構文に関連していると思います。追加の作業なしで、そのようなコンパクトなパッケージを手に入れることも非常に価値があります。すばやく読み込まれるアプリケーションは、ユーザーとGoogleの両方に非常に人気があります。

Svelteに欠陥はありますか？正直なところ、頭に浮かぶのは1つだけです。実際、Svelteはまだ新しく、あまり広く使用されていません。したがって、彼のコミュニティは限られており、おそらくさらに重要なことに、スヴェルトのスペシャリストを必要とする企業は（まだ？）あまりありません。あなたがジュニアデベロッパーで仕事を探しているなら、最初にReactを学ぶ方が賢明だと思います。ただし、上級開発者（または後輩で、時間が長い場合）は、Svelteを検討することを強くお勧めします。

Webコンポーネント


たぶん、Webコンポーネントは次の優れたテクノロジーでしょうか？彼女は確かに可能性を秘めていると思います。では、Webコンポーネントとは何でしょうか？

Webコンポーネントは、新しいカスタムHTMLタグを作成するためのw3c標準のセットです。これらの標準を使用すると、ウィジェットから大規模なアプリケーションまで何でも作成できます。



そして、ほとんどの大きなブラウザーでは、この新しい標準がサポート



されています。InternetExplorerでは、このような機能はポリフィルを使用してサポートされています。

Webコンポーネントを作成すると、新しいネイティブDOM要素が作成されます。 「仮想DOM要素」が作成され、その後DOMに実装されるReactの場合とは異なります。

<my-component hello="world"></my-component>

Svelteの最も優れた点は、ランタイム環境をクリーンアップすることであり、Webコンポーネントについても同様です。ネイティブDOM要素を作成する場合、ランタイム環境は必要ありません。アプリケーションを起動するには、通常のブラウザで十分です。
これらは単なるDOM要素なので、どのWebページでも使用できます。ウィジェットまたはアプリケーション全体を作成して、既存のアプリケーションに埋め込むことができます。プログラミングの対象は関係ありません-Angular、React、Vue、またはまったく異なるもの。これは、<div>タグなどの最も一般的なDOM要素についてです。

<body>
<my-app name="Hello world"></my-app>
<script src="/my-app.js" type="module"></script>
</body>

これにより、完全に新しい機会と、クライアント側の大規模オファーのアーキテクチャへの新しいアプローチが開かれます。この一例がマイクロフロントです。

Webコンポーネントを直接作成できますが、開発の観点からは、これは良い方法ではありません。 Webコンポーネントのレベルにコンパイルするフレームワークを使用する方がよい。この種の人気のある2つのフレームワークは、Stencil.jsとPolymer（Google製）です。

Webコンポーネントを使用する場合、ランタイム環境がないため、パッケージは非常にコンパクトです。次のベンチマークを見てください。



このリストで言及されているSvelteに気づきましたか？Svelteはコンポーネントレベルにコンパイルでき、生成するパッケージは非常に小さいです。
ただし、Webコンポーネントフレームワークのすべてのパッケージが、「クラシック」フレームワーク（ReactやAngularなど）のパッケージと比較してどれほど小さいかを見てみると、さらに興味深いです。

ReactのWebコンポーネントはどうですか？


ReactはWebコンポーネントにコンパイルすることもできますが、custom-elements-everywhere.comによると、それらはサポートされていません。



ただし、完全なサポートが続く場合があります。おそらく、Reactチームはこのステップを実行しようとしており、それを起動する前に適切に検討しただけです。この回避の理由の1つは、Webコンポーネントが、Reactのように宣言的なスタイルではなく、命令的なスタイルを使用することです。ReactではいつかWebコンポーネントの完全サポートが表示されるか、表示されないかもしれません。それまでの間、Webコンポーネントを作成する場合は、Reactの代わりに別のライブラリを使用してください。

Webコンポーネントを使用する必要がありますか？


Webコンポーネントは、microfrontアーキテクチャで使用するときに最も有用であるように思えます。異なるクライアントライブラリを使用して、多くのチームが同時に操作する大規模な組織で作業する場合、異なるチームのコードで一緒に作業することは簡単な作業ではありません。この場合、動作するアプローチはReactとは異なります。 Reactは、ウィジェットを作成するのではなく、アプリケーション全体のオーケストレーションを専門としています。ご覧のとおり、ReactコンポーネントとWebコンポーネントは互いに平和に共存できます。

Webコンポーネントには、Svelteとほぼ同じ欠点があります。ジョブを取得しようとしているときに履歴書を植字する場合、Webコンポーネントを優先度と呼ぶことはできません。少なくともReactでは優先されていません。私の経験では、企業はそれほど要求されていません（まだですか）。

最終的な考え


フロントエンドの開発では、クールで興味深いすべてのことが起こっています。Reactはフレームワークの1つにすぎません。その外で何が起こっているのかをナビゲートする必要があるだけです。
あなたが上級開発者であり、フロントエンドを専門とする場合は、もちろん、仕事に最適なツールを選択します。これは多くの場合Reactですが、おそらく次のプロジェクトにはWebコンポーネントの方が適しています。
