機能スイッチの概要


それは何ですか


簡単に言えば、スイッチの意味は次のとおりです。請負業者は、サブジェクト領域のロジックを表すコードを進め、条件付き演算子に遭遇します。このステートメントでは、請負業者は、関心のある機能がいつどのような状況で動作するかを知っているレジストリから、条件付きコードを実行する必要があるかどうかを尋ねます。


その結果、エグゼキュータは1つのブランチまたは別のブランチに沿って進み、縮退した場合、他のブランチは単に空になります。最も単純なケースでは、スイッチは、レジスターを管理する責任者の制御下で2つの値（「オン」と「オフ」）のいずれかを取ります。また、特定のユーザーのみ、特定の国のユーザー、特定の時間などの機能を含めるなど、さらに複雑なものを考え出すこともできます。

機能レジストリは、さまざまな形式をとることがあります。重要なのは、アプリケーションを停止したり再デプロイしたりせずにこのレジストリを管理できる場合です。たとえば、スイッチはファイルやデータベースに保存したり、スイッチを使用して個別のネットワークサービスを提供したりできます。以下では、スイッチの操作について詳しく説明します。

可能なスイッチシステム要件


ご覧のとおり、スイッチシステムは非常に複雑なものになる可能性があります。このようなシステムに課される可能性のある基本的な要件を示します。

  1. さまざまなパラメーターのオンとオフの機能の依存関係。パラメータの例：ユーザー（役割、識別子、地理的な場所）、時間（昼光または暗闇、就業日または平日、特定の期間）、環境（開発、テスト、産業用）。
  2. ドメインプロセスを分析する際の使用統計の収集：どのユーザーがどの条件下で機能を確認したか、または確認しなかったか、新しい機能を使用した回数。
  3. スイッチ自体で何が起こっているかの監査：誰がスイッチを押すか、いつ、何が変わるか。
  4. ローカルレジストリ機能とリモートレジストリ機能の間の柔軟な選択、またはリモートレジストリが利用できない場合の戦略の提供。
  5. 顧客と請負業者の両方のさまざまなカテゴリーの従業員（技術専門家、対象分野の専門家、およびアプリケーションユーザー自身）のスイッチシステムへのアクセスを構成します。

もちろん、これらすべての要件を満たすために努力する必要はありません。多くの場合、逆に、一部の要件は望ましくない場合があります。順守する必要のある要件が多いほど、独自のスイッチシステムの開発が明らかに負担になります。これは、スイッチの導入後のある時点で、独自のシステムのサポートが採算が取れなくなり、既製のソリューションを採用する時が来たことを意味します。

スイッチの使用に関しては、次の一般的な推奨事項を行うことができます。

切り替えポイントとロジックを一緒にする必要はありません。最も単純な例では、特定のスイッチの状態を問い合わせる条件ステートメントの後に、コードがすぐに含まれる機能を実装します。これは、ロジックに冗長な依存関係を追加するため、完全に便利ではありません。スイッチのインフラストラクチャに関する知識とレジストリからの特定の機能の名前です。実際には、これにより、スイッチのテストと削除のプロセスが複雑になります。

切り替えポイントを高く設定。前段落の作成。スイッチインフラストラクチャにアクセスするポイントは、サブジェクトエリアのロジックから可能な限り最後の機会まで除外する必要があります。つまり、リクエストが受信された場所の近くに「プッシュ」する必要があります。したがって、個々のモジュールのスイッチへの依存を減らし、サポートとテストのプロセスを簡素化します。

条件文の代わりに戦略を使用してください。スイッチが長期間存続する場合は、条件演算子を戦略に改善し、切り替え可能なロジックを個別に追跡およびテストするものに明示的に分離できます。

スイッチをグループ化しないでください。スイッチ間の依存関係を形成し、スイッチをグループ化し、階層にまとめる必要はありません。これにより、新しい機能とスイッチ自体のライフサイクルを備えたコードの保守が容易になります。

スイッチポイントを1つの機能に統合します。一度に複数のプログラムブロックの動作が同じスイッチに依存することが判明する場合があります。開発の調整が不十分な場合、切り替えポイントが別の場所に複製される可能性があり、これによりテストとサポートのコストが高くなります。スイッチングポイントをアプリケーションの入り口に「プッシュ」するように訓練されている場合、この推奨事項-システム全体にポイントを分散しないでください-は通常、自動的に実行されます。

スイッチの主なカテゴリ


この記事に記載 されているスイッチの分類を検討してください。これは、スイッチが「存続する」期間と状態が変化する頻度に基づいています。特定のクラスへのスイッチの割り当ては、コードでスイッチを使用する方法とスイッチの状態を格納する方法を決定するのに役立ちます。

リリーススイッチ

これは、スイッチのメインビューです。それらにより、開発を1つの主要なブランチに集中させることができ、さらに、定期的に商業運用に展開されます。別のブランチで開発し、それをメインブランチに挿入して目的のバージョンをリリースする代わりに、コードに切り替えポイントを導入し、ユーザーからまだ準備されていない機能を隠します。準備ができると、スイッチは新しい機能を起動します。

このようなスイッチは数日から数週間存続しますが、新しい機能の開発と実装は進行中です。機能がテストされ、適切であると見なされたら、コード内のスイッチと切り替えポイントを削除できます。通常、スイッチの状態は、新しいバージョンがリリースされたとき、またはアプリケーション構成が変更されたときに変更されます。このようなスイッチを構成ファイルに格納することは許可されています。新しい機能の切り替え点が唯一のものになる可能性が高いです。掘り下げて従来の条件付きステートメントの形式で配置しないことは理にかなっています。

実験用のスイッチ

新しい機能を実行するために、数日から数か月間存続するスイッチが使用されます。実験中、ユーザーが変更をどのように認識し、どのように行動が変化するかを監視できます。スイッチの状態は、要求ごとに非常に動的に変化できることが望ましい。データベースやネットワークサービスなどの一部の集中型ストレージは、ここで適している可能性が高くなります。実験が予測可能な数の問題に当てはまる場合、切り替えポイントは条件付きステートメントの形式で発行することもできます。

テクニカルスイッチ

テクニカルスイッチは、アプリケーション全体のパフォーマンスに影響を与えるアプリケーションのインフラストラクチャ部分の管理に役立ちます。パフォーマンスへの影響を評価するのが難しいアップデートをロールアウトするときに便利です。この場合、その使用が悲惨な結果をもたらすことが判明した場合、新しい機能を即座に無効にする「スイッチ」があればよいでしょう。

ほとんどの場合、このようなスイッチは数週間またはそれ以上存続し、アプリケーション構成が変更されるか、より頻繁に状態が変化します。これらのスイッチが実装する機能の重要性は、ネットワークサービスまたは少なくともデータベースを使用する必要があることを示唆しています。コード内で、同じ機能の切り替えポイントが別の場所に散在していないことを確認する必要があります。

機能スイッチにより、ユーザーをグループに分けて管理することが簡単になります。グループごとに、アプリケーションの次のリリースを待たずに、新しい機能を有効または無効にすることができます。これにより、前提条件を検証し、A / Bテストサイクル全体を完了するのにかかる時間が短縮されます。多くの企業は、週にいくつかの仮定の検証を確立しようとしています。

カナリアと青緑色の問題

カナリアはそれと何をしているのですか？鉱山業界ではカナリアが使用されていました。鉱山労働者は、高濃度の爆発性ガスが疑われる鉱山に降下したときに、カナリアをケージに入れて連れていきました。カナリアはとてもきれいな空気が好きで、人々がその中に有害で危険な不純物を感じるよりもずっと早い時期です。カナリアの歌声が止まったり、電源が切れたり、死亡したりした場合は、爆発するまで緊急に避難する必要があります。ここでこれについての詳細を読むことができます（英語）。

ここからは、私が理解しているように、「カナリアの証」という言葉が出てきました。抑圧的な国内政策を追求している州（つまり、いずれか）に住んでいる場合、ある日、ユーザーの個人データを承認された部門に転送するだけでなく、誰にも知らせないように指示されていることがあります。それらは引き渡されたのであって、あなたがそのような命令を受けたというわけではありません。もちろん、あなたは外に出ることはできませんが、場合によっては、この種のメッセージを定期的にユーザーに送信できます：「先月、私たちはあなたの個人データを開示するための命令を受けませんでした。」それはあなたがカナリアのように鳴く方法です、そしてあなたが個人データを提供するための注文を受けると、あなたは黙ります。したがって、当局の要求を満たし、ユーザーに警戒する理由を与えます。

そしてカナリア問題があります。ここでのメタファーは次のように機能します。ユーザーの一部が病気になったとき、私たちは緊急に「退避」します。ユーザーを病気にした新しい機能をオフにし、すべてのユーザーに届かないようにします（「ジャーク」しないでください）。これを行うには、新しい機能をさまざまなグループに拡張できる必要があります。ここでは、スイッチなしで行うことができます。たとえば、バランサーと、リクエストをリダイレクトする2つのノードがあります。 1つのノードには安定したバージョンがデプロイされ、2番目のノードには新しい機能を含む試験的なバージョンがあります。デフォルトでは、すべてのリクエストは安定バージョンのノードに送信されます。バランサの助けを借りて、実験バージョンのノードにいくつかのリクエストの送信を開始します。

ただし、このスイッチを使用すると、新しい機能をユーザーに拡張する際の柔軟性を高めることができます。スイッチでは、受信したリクエストのさまざまなパラメーターを組み合わせて、カナリア問題を管理するための便利なグラフィカルインターフェイスを責任者に提供できます。一般に、プロセスは同じです。たとえば、1％程度の小さなユーザーグループ向けの新機能を含めます。その後、アプリケーションの状態、新機能を使用するユーザーの動作を監視し、それをより多くのユーザーに拡張した場合の結果を予測します。新しい機能を使用するユーザーが徐々に増えていく中で、仮説をテストして修正することができます。マイナスの傾向に気づいたらその後、新しい機能を簡単に無効にできます。

青緑のリリースでは、一時的に青と緑と呼ばれる2つのサーバーが使用されます。リリースの前に、バランサーはすべてのリクエストをグリーンサーバーに送信すると想定します。リリース中、アプリケーションの新しいバージョンが青いサーバーにデプロイされ、バランサーもリクエストの送信を開始します。アプリケーションの新しいバージョンにエラーが含まれていることが判明した場合は、緑色のサーバーに切り替えます。

青緑色のリリースの基本バージョンは、ユーザーが新しいバージョンに含まれるすべての新機能を操作するか、または何も操作しないことを前提としています。また、スイッチを使用すると、リリース中に、すべての新機能のスイッチがオフになっている新バージョンのブルーサーバーにリクエストを送信することが可能になります。その後、新しい機能を徐々にパーツに組み込むことができます。

機能スイッチを使用すると、カナリアリリースと青緑リリースの利点を組み合わせることができます。通常、青緑色のリリースでは、アプリケーションの古いバージョンと新しいバージョンの2つの明確に分離されたノードがあり、バランサーの助けを借りて、すべての要求を一度に一方または他方のノードにリダイレクトします。スイッチを使用すると、新しいバージョンのノードにリクエストをリダイレクトできますが、カナリアリリース時と同様に、新しい機能をオフにしてから、徐々にオンにします。

カナリア問題の詳細については  、FowlerのWebサイトまたは「Habré」などの対応するメモを参照してください。青緑のリリースについて-  レビュー記事。また、青緑色のリリースでのファンクションスイッチの使用については、こちらをご覧ください。

データの保存方法を変更する

抽象化やインターフェイスからの開発などのプログラミング手法を研究しているときに、私はしばしば次の推奨事項に遭遇しました：プログラムを特定のデータストレージの方法（特定のデータベースなど）にバインドしないようにしてください。保存方法の変更が必要になる場合があります。私はそれを口ひげに巻いて、注文通りにやった。しかし、それ以来、ドメインロジックとストレージメソッドを共有する理由が変わり、逆に、商用製品のデータベースを変更する可能性について懐疑的になりました。

そしてこの資料を準備している間に、私は保証する人について学びました多くの企業が、工業的に運用されている製品のデータベースを変更することを実践しています。さらに、機能スイッチを使用すれば、おそらくデータベースの変更は完全に簡単です。言及された記事が興味のある人々によって書かれたことは明らかです（このサイトはスイッチを管理するための商用ツールの製造元によってサポートされています）。

次のように戦略を簡単に説明します。プロセスは、アプリケーションが1つの（古い）データベースで動作する瞬間から始まります。機能スイッチを使用して、アプリケーションで最初に新しいデータベースにデータを書き込み、次に新しいデータベースからデータを読み取るように強制します。同時に、古いデータベースとの対話が維持されます。両方のデータベースに対して書き込みと読み取りを行います。

記録が多かれ少なかれ明確である場合は、読みを明確にする必要があります。アプリケーションでデータが必要な場合は、両方のデータベースからデータを読み取ります。これらの2つの読み取りポイントは常に隣り合っているため、読み取り後、受信したデータを比較して整合性を確認できます。アプリケーションの安定化後（両方のデータベースが安定して同じデータを返し始めるとき）、アプリケーションは古いデータベースから切断され、スイッチが削除されます。

異なるプラットフォームでの機能の同時インクルード

既に言及されているレポート「機能の切り替え、またはリリースなしで機能をロールアウトする方法」では、サイトとモバイルアプリケーションの両方で同時に新しい機能を有効にする要件について説明しました（分/分！）。そして、同様にそれをオフにする必要があるかもしれません。

異なるプラットフォームに新しいバージョンを提供することは、同期が非常に難しいため、このような要件を満たすことは容易ではありません。サイトを備えたサーバーがあなたの管理下にある場合でも、何かを推測することができ（そして今回は配信が時計のように機能することを願っています）、モバイルアプリストアは更新を更新するためのポリシーを自由に変更できます。いずれにしても、アプリケーションの更新は長いプロセスであり、ストア自体による公開アプリケーションのチェックが含まれます。さらに、モバイルアプリケーション自体が定期的にそのAPIから更新をリクエストし、それ自体を「内部」にインストールすることを望んではなりません。私が理解しているように、このショップもプラットフォーム所有者の注意深い監督下にあります-任意のコードの実行には確かに問題があります。

ただし、スイッチは、アプリケーションの配信を完全に制御できない環境で機能を同時に含めるという要件を満たすのに役立ちます。当然のことながら、予想される包含まで、配信は確実に発生するはずです。この方法は、ブラックフライデーなどの一時的なプロモーション期間に役立つでしょう。

1つのシステムの複数のモジュールでも同様の状況が存在します。それらが個別に公開され、同時にサブジェクト領域の1つのプロセスのサポートに参加している場合、スイッチを使用すると、さまざまなモジュールの側でこのプロセスに調整された変更を加えることができます。

.NETのツールの切り替え


スイッチを操作するためのツールは、3つのグループに分類できます。まず、これらはユニバーサルヘビー製品（複合）であり、通常、プロバイダーと連携するネットワークサービスと、さまざまなプログラミング言語でこのサービスと通信できるようにする一連のライブラリが含まれます。ほとんどの場合、そのような製品を使用するには料金がかかります。第2に、これらは、コンピューターで実行する必要があるネットワークサービスであり、クライアントがREST APIを介してアクセスできるプロジェクトです。このグループのプロジェクト（簡潔にするために、サーバーと呼びます）のうち、十分に文書化されたAPIまたは.NETの公式クライアントを持つプロジェクトのみがレビューに含まれていました。第3に、これらは.NET用の（前の2つのグループに比べて）単純なライブラリです。機能レジストリを構成ファイルに保存するか、ネットワーク経由でこれらのプロジェクトの一部ではないリモートレジストリにアクセスすることをお勧めします。

注意深い読者は、レビューに含まれなかったソフトウェアツールの別のグループがあることに気づくでしょう。これらは、分散システムの構成リポジトリーとして使用できる製品です。そのような製品のよく知られた代表には、Apache ZooKeeper、Consul、  etcdが含まれ、この記事のコメントでは、.NETと簡単に友達にすることができるSpring Cloud Config Serverについても言及しています。。実際、スイッチは基本的な機能において構成リポジトリと非常に似ているため、これらのツールは独自のスイッチインフラストラクチャを作成するための開始点として使用できます。それにもかかわらず、機能スイッチの目的には特定の特性があるという事実のために、プロジェクト内のスイッチの文化の発展に伴い、ユニバーサル構成リポジトリーの欠点が感じられ始めます。このため、同様の製品についてはこれ以上説明しません。
