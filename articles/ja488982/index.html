S905Xプロセッサ（セキュアブート）ソフトウェアのセキュリティの概要

この記事では、S905Xプロセッサのソフトウェア保護について説明します。最終的な目標は、不正なソフトウェアを起動することです。

S905X


S905Xプロセッサは、最大1.5 GHzのクロック周波数を備えたARM Cortex-A53であり、H.265 4K、VP9、4KUHDをサポートするなど、ビデオおよびオーディオストリーム用のあらゆる種類のデコーダーを搭載しています。一般的に、悪い選択ではありません。その前身とは異なり、AMLogicはいわゆる「高度なTrustZoneセキュリティシステム」をこのプロセッサに統合しました。これは、ROMメモリの保護領域へのアクセス、署名のチェック、ソフトウェアの解読など、すべての重要なシステム操作を制御します。このトピックに関する詳細なドキュメントは、製造元の Webサイトにあります。 SecureOSはATOS-V1.5-g3e467d9として機能します（私はアサートしません、チェックしませんでした）。

トレーニング


プレフィックスを分解したので、UARTのRxD、TxD、およびGNDピンを簡単に見つけました。裏側には、リセットボタン用のコネクタ（確認されていません）とコネクタ用の場所がありました。位置から判断すると、コネクタは代替のeMMSカードのボード間であるように見えました（ピン配置は当然ありません）。



たとえば、起動シーケンスなどの重要なトピックに関するS905の無料アクセスドキュメントは、ほとんど影響を受けておらず、REの観点からは実質的に役に立たなかった。ドキュメントを別にして、私はUARTを接続しました...プロセッサの読み込み速度に驚きました。しかし、おそらくそれだけです。U-Bootはキーボードにまったく反応せず、いくつかのハードウェア設定の後、Androidカーネルをスマートにロードして起動しました。

U-Boot（bl33）自体がすでに「通常の世界」環境で実行されていたことに注目する価値があります（BL3-1は、制御をそれに移す直前にモードを切り替えました）。それら。どういうわけかU-Bootを制御しても、システムへのアクセスは制限されます。原則として、これはそれほど重要ではありません。結局のところ、ROMダンプを削除し、AESキーを抜き、RSAをヒューズから開いたとしても、これには大きな意味がありません。 RSA秘密鍵はまだありません。

U-Boot自体はキーを読み取りませんでした。彼はカーネルをメモリにロードし、セキュリティモニターにコードの復号化と検証を要求しました。 Security Monitorは必要なすべてのことを行い、U-Bootにカーネルをロードするかどうかを指示しました。一般に、独自のTrustedOSとBIOSなどのある種の機能を使用したソフトウェア保護への真剣なアプローチ。

私のさらなる計画は、eMMCメモリにアクセスすることでした。

eMMC


eMMCに接続するには、少なくともDAT_0、CLK、CMD、GNDピンを見つける必要がありました。また、eMMCコントローラーが機能した電圧（1V8または3V3）も確認します。これを視覚的に行うことは不可能で、トレーストラックの場合は自分でeMMCを引き出しました。チップを手にしたので、トレースした後、アダプターに接続してフルダンプをダンプしました。残念ながら、私はプロの機材を持っていなかったので、即席の素材を使用しました。結果はそのような「死んだバグ」です。


まあ、実際にはコネクタのピン配列：



ピン配置が完了した後、eMMCなしでプロセッサがどのように動作するかを調べました。ログから判断すると、eMMCからの起動に失敗したROMは、mSDカードからコードをロードしようとしました。これは心強いことでした。すぐにダンプをmSDカードにアップロードして、コンソールをオンにしました。ここでちょっとした驚きが待っていました。 ROMがカードからBL2コードをダウンロードし、それに制御を移しました（つまり、eMMCの「ブート」セクションは関与せず、ルートアクセスの必要がありませんでした）。さて、ターンがU-Bootになったとき、彼はeMMC（dev 1）の欠如について何度か不平を言いました。

U-Bootは抵抗せず、喜んでeMMCに切り替えました。もちろん、私はbootdelayパラメータの内容を調べました。そして、5つではなく0が表示されました。ある時点でU-Bootが0を書き留めたようです。誰かが突然何かを変更した場合に備えて。しかし、それはもう私には興味がなく、eMMCに関するトピックは締め切られました。

注：コネクタのピン配列があれば、チップをはんだ付けせずにeMMCに接続できます。これを行うには、電圧コンバータを備えた特別なアダプタが必要です。eMMCコントローラーは1V8で動作し、通常のSDアダプターを3V3に接続すると、1V8バス上にあるコンポーネントが焼損する危険があります。

また、この抵抗を削除する必要があります。そうしないと、プロセッサがアダプタに干渉します。

脱獄


U-Bootはenvパラメーター（「bootdelay」だけでなく）の変更に対して免疫がとられているため、envを介してブートプロセスに影響を与えることは不可能であり、そのソフトウェアをロードするために、プレフィックスをコンピューターに接続する必要がありました。このオプションは私にはまったく合いませんでした。ロードチェーン全体を分析したので、このコマンドは私の注意を引きました：

init_display=osd open;osd clear;imgread pic logo bootup $loadaddr;bmp display $bootup_offset;bmp scale

特に：

imgread pic logo bootup $loadaddr

ここで、U-Bootはアドレスloadaddr = 0x1080000で特定の「ブートアップ」リソースをロードしていました。リソース自体は、このような構造で記述されていました：

struct resource_header{
	unsigned int 	magic;		/* Image Header Magic Number	*/
	unsigned int 	hcrc;		/* Image Header CRC Checksum	*/
	unsigned int	size;		/* Image Data Size		*/
	unsigned int	start;		/* item data offset in the image*/
	unsigned int	end;		/* Entry Point Address		*/
	unsigned int	next;		/* Next item head offset in the image*/
	unsigned int	dcrc;		/* Image Data CRC Checksum	*/
	unsigned char	index;		/* Operating System		*/
	unsigned char	nums;		/* CPU architecture		*/
	unsigned char   type;		/* Image Type			*/
	unsigned char 	comp;		/* Compression Type		*/
	char 	name[32];		/* Image Name			*/
}

もちろん、「サイズ」と「開始」パラメータは興味深いものでした。それらと「loadaddr」パラメーターから進んで、U-BootはeMMCから直接読み取るための3つの新しいパラメーターを解決しました。理想的には、もちろん、彼は0x000B00C0の「ロゴ」セクションから0x1130000のメモリに画像をダウンロードしました...私の考えの道筋は今やはっきりしていると思います。新しいパラメーター「サイズ」と「開始」を解決し、eMMCに保存されたコードを記述しました。そのような操作の後、U-Bootはもはや画像を読み取ることはなく、エクスプロイトがあり、すぐに制御を獲得しました。エクスプロイトは、アセンブラのいくつかの行で構成されていました。

結論


接頭辞が棚でアイドル状態にならないようにするために、LibreElecを付けることにしました。新しく組み立てられたイメージは、ほとんどすぐに問題なく起動されました。カーネルは小さなファイルでファイナライズされ、WiFiとリモートコントロール用のドライバーがインストールされ、電源オフ機能がやり直されました。そして、すべてが正常に機能しましたが、コンソールはBluetoothもサポートしていることを思い出しましたが、これは当然LibreElecでは機能しませんでした。私はそれを必要としませんでしたが、私はそれをさらに補足するためにそれを設定することに決めました。
搭載されていたのはWiFiとBluetoothの両方を担当するAmpakのAP6255チップでした。 UART BluetoothがttyS1に接続されましたが、チップはHCIコマンドに応答せず、死んだふりをしました。さらに、ネイティブコアでは、予想通りチップがチームに応答しました。
このチップのドキュメントは非常に貧弱であり、そこから抽出できるのはピン配列のみです。まず、チップの電源とBT_WAKEピンを確認しました。すべてが正常でした。ドライバー自体とttyS1ポートを試したところ、結果は得られませんでした。チップはパルチザンとして静かで、オシロスコープを接続しました。その上で、すべての電子UART信号（RTS、TXD、RXD、CTS）をチェックして比較しました。すべてが正常でした。次に、rfkillを介して制御されるリセットシーケンスをチェックしました（これらはBT_WAKEおよびBT_REGONピンでした）。しかし、ここではすべてが大丈夫でした。次に、LPOピンを確認しました。ドキュメントによると、周波数が32.768KHzの信号が入るはずですが、ここでもオシロスコープに信号が表示されています。一般的に、すべてが順調でしたが、チップは私のコアで動作しませんでした。元のUARTドライバーを分解するのに数日を費やした後、LPO信号に戻ってその周波数を測定しました。結果は26KHzを示しました。ああ、どうだ。
wifi_dtドライバーがこの信号の原因でしたが、私のコアでは周波数を正しく生成していませんでした。私は元のドライバーを分解し、そこからpwm信号の初期化を引き継ぎ、そして...チップが生き返った（後に、正しい信号を持つこのドライバーのバージョンを見つけた）。これで、チップはさらに調整する準備ができましたが、どうすればアラートに設定できるかわかりませんでした。ドライバ/ファームウェア/ドキュメントが見つかりませんでした。元のAndroidでは、Bluetoothで作業しているとき、rfkillを除いてカーネルログには何もありませんでした。これは理解できるので、カーネルドライバーはなく、起動はユーザーレベルで行われました。どういうわけかAndroid自体のログにアクセスする必要があり、そのためには、デバイスへのrootアクセスが必要でした。これを行うために、私の初心者として、必要な設定をシステムセクションに追加し、adbをオンにしました。しかし、このシナリオでは、Androidは何かが間違っていることに気づき、そこでいくつかの操作を参照して、開始を拒否しました...まあ、いや、いや。そこにshを含めるだけで、システムパーティションを復元してrootfsをやり直しました。その後の起動後、ターミナルでrootアクセスができました。
これは、チップが初期化された方法です。
