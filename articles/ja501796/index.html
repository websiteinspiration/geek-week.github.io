C＃/ .NETで簡単にできる7つの危険な間違い

この記事の翻訳は、コース「C＃ASP.NET Core Developer」の開始前に準備されました。



C＃は素晴らしい言語であり、.NET Frameworkも非常に優れています。C＃での強い型付けは、他の言語と比較して、誘発できるエラーの数を減らすのに役立ちます。さらに、JavaScriptのようなもの（trueがfalseの場合）と比較すると、その一般的な直感的なデザインも非常に役立ちます。ただし、各言語には独自のレーキがあり、その上を簡単に進むことができます。また、言語およびインフラストラクチャの予想される動作に関する誤った考えも含まれます。これらのエラーのいくつかを詳しく説明しようと思います。

1.遅延（遅延）実行を理解していない


経験豊富な開発者はこの.NETメカニズムを知っていると思いますが、知識の少ない同僚を驚かせるかもしれません。簡単に言えば、各結果を返すIEnumerable<T>ために使用yieldするメソッド/演算子は、実際にそれらを呼び出すコード行では実行されません-結果のコレクションが何らかの方法でアクセスされたときに実行されます*。ほとんどのLINQ式は最終的に結果をyieldで返すことに注意してください。

例として、以下の悪質な単体テストを考えてみましょう。

[TestMethod]
[ExpectedException(typeof(ArgumentNullException))]
public void Ensure_Null_Exception_Is_Thrown()
{
   var result = RepeatString5Times(null);
}
[TestMethod]
[ExpectedException(typeof(InvalidOperationException))]
public void Ensure_Invalid_Operation_Exception_Is_Thrown()
{
   var result = RepeatString5Times("test");
   var firstItem = result.First();
}
private IEnumerable<string> RepeatString5Times(string toRepeat)
{
   if (toRepeat == null)
       throw new ArgumentNullException(nameof(toRepeat));
   for (int i = 0; i < 5; i++)
   {   
       if (i == 3)
            throw new InvalidOperationException("3 is a horrible number");
       yield return $"{toRepeat} - {i}";
   }
}

これらのテストは両方とも失敗します。結果はどこにも使用されないため、最初のテストは失敗し、メソッドの本体は実行されません。 2番目のテストは、もう少し重要な別の理由で失敗します。次に、メソッドを呼び出した最初の結果を取得して、メソッドが実際に実行されることを確認します。ただし、遅延実行メカニズムは、できるだけ早くメソッドを終了します。この場合、最初の要素のみを使用したため、最初の反復を渡すとすぐに、メソッドは実行を停止します（そのため、i == 3は真にはなりません）。

遅延実行は実際には興味深いメカニズムです。特に、LINQクエリを連鎖させ、クエリの準備ができたときにのみデータを取得できるためです。

3.流れの安全性を考慮しない


マルチスレッドは優れています。正しく実装されていれば、アプリケーションのパフォーマンスを大幅に向上させることができます。ただし、マルチスレッドに入るとすぐに、変更するオブジェクトには十分注意する必要があります。十分に注意しないと、ランダムに見えるエラーが発生し始める可能性があります。

簡単に言うと、.NETライブラリの多くの基本クラスはスレッドセーフではありません。-これは、Microsoftが複数のスレッドを使用してこのクラスを並行して使用できることを保証しないことを意味します。これに関連する問題をすぐに見つけることができれば、これは大きな問題ではありませんが、マルチスレッド化の性質は、発生する問題が非常に不安定で予測不可能であることを意味します。

たとえば、単純であるがスレッドセーフではないを使用する次のコードブロックを考えてみList<T>ます。

var items = new List<int>();
var tasks = new List<Task>();
for (int i = 0; i < 5; i++)
{
   tasks.Add(Task.Run(() => {
       for (int k = 0; k < 10000; k++)
       {
           items.Add(i);
       }
   }));
}
Task.WaitAll(tasks.ToArray());
Console.WriteLine(items.Count);

したがって、リストに0から4までの数値をそれぞれ10,000回追加します。これは、リストには最終的に50,000要素が含まれることを意味します。したほうがいい？まあ、最終的には可能性が少しありますが、以下は私の5つの異なるスタートの結果です。

実際、これはAddメソッドがアトミックではないためです。つまり、スレッドがメソッドに割り込むことができ、別のスレッドが追加処理を行っている間に配列を最終的にサイズ変更したり、同じインデックスの要素を追加したりできます。他のスレッドとして。 IndexOutOfRange例外が何度か発生しました。おそらく、配列の追加中に配列のサイズが変更されたためです。では、ここで何をしますか？ lockキーワードを使用して、（Add）アイテムを一度に1つのスレッドのみがリストに追加できるようにすることができますが、これはパフォーマンスに大きな影響を与える可能性があります。マイクロソフトは、優れた人材であり、優れたコレクションをいくつか提供しています。これらはスレッドセーフであり、パフォーマンスの点で高度に最適化されています。それらの使用方法を説明した記事はすでに公開しています。

4. LINQでのレイジー（遅延）ロードの悪用


遅延読み込みは、LINQ to SQLとLINQ to Entities（Entity Framework）の両方の優れた機能であり、必要に応じて関連するテーブル行を読み込むことができます。他のプロジェクトの1つに、「モジュール」テーブルと「結果」テーブルがあり、1対多の関係があります（モジュールには多くの結果が含まれる場合があります）。



特定のモジュールを取得したい場合、Entity FrameworkがModulesテーブルのすべての結果を返さないようにしたいのですが！したがって、彼は私が必要なときにのみ結果を取得するクエリを実行するのに十分賢いです。したがって、以下のコードは2つのクエリを実行します-1つはモジュールを取得するため、もう1つは結果を取得するため（モジュールごとに）、

using (var db = new DBEntities())
{
   var modules = db.Modules;
   foreach (var module in modules)
   {
       var moduleType = module.Results;
      //   
   }
}

しかし、何百ものモジュールがある場合はどうなりますか？これは、結果レコードを受信するための個別のSQLクエリがモジュールごとに実行されることを意味します！明らかに、これはサーバーに負担をかけ、アプリケーションの速度を大幅に低下させます。Entity Frameworkでは、答えは非常に簡単です。特定の結果セットをクエリに含めるように指定できます。以下の変更されたコードを参照してください。SQLクエリが1つだけ実行され、このモジュールの各モジュールと各結果が含まれます（1つのクエリに結合され、Entity Frameworkがモデルにインテリジェントに表示します）。

using (var db = new DBEntities())
{
   var modules = db.Modules.Include(b => b.Results);
   foreach (var module in modules)
   {
       var moduleType = module.Results;
      //   
   }
}

5. LINQ to SQL / Entity Frameworksがクエリを変換する方法がわからない


LINQトピックに触れたので、コードがLINQクエリ内にある場合、コードがどのように実行されるかについて言及する価値があると思います。高レベルで説明すると、LINQクエリ内のすべてのコードは式を使用してSQLに変換されます。これは明らかなようですが、現在のコンテキストを忘れて、最終的にコードベースに問題を引き起こすことは非常に簡単です。以下に、遭遇する可能性のある典型的な障害を説明するリストをまとめました。

ほとんどのメソッド呼び出しは機能しません。

したがって、以下のクエリですべてのモジュールの名前をコロンで区切り、2番目の部分を取得するとします。

var modules = from m in db.Modules
              select m.Name.Split(':')[1];

ほとんどのLINQプロバイダーでは例外が発生します。SplitメソッドにはSQL変換がありません。たとえば、日付に日数を追加するなど、一部のメソッドはサポートされている場合がありますが、すべてプロバイダーに依存します。

機能するものは予期しない結果をもたらす可能性があります...

以下のLINQ式を使用してください（これを実際に行う理由はわかりませんが、これが妥当な要求であることを想像してください）。

int modules = db.Modules.Sum(a => a.ID);

モジュールテーブルに行がある場合は、識別子の合計が表示されます。いいね！しかし、代わりにLINQ to Objectsを使用して実行するとどうなるでしょうか。これを行うには、Sumメソッドを実行する前に、モジュールのコレクションをリストに変換します。

int modules = db.Modules.ToList().Sum(a => a.ID);

ショック、ホラー-それはまったく同じです！ただし、モジュールテーブルに行がない場合はどうなりますか？ LINQ to Objectsは0を返し、Entity Framework / LINQ to SQLバージョンはInvalidOperationExceptionをスローします。これは、「int？」を変換できないことを示します「int」...で。これは、空のセットに対してSQLでSUMを実行すると、0ではなくNULLが返されるため、代わりにnull許容のintを返そうとするためです。このような問題が発生した場合にこれを修正するためのヒントをいくつか紹介します。

古き良きSQLをいつ使用する必要があるかを理解してください。

非常に複雑なリクエストを実行している場合、翻訳されたリクエストが吐き出され、何度も何度も食べ尽くされたように見える可能性があります。残念ながら、実例を示す例はありませんが、一般的な意見から判断すると、ネストされたビューを使用するのが本当に好きなので、コードのメンテナンスは悪夢になります。

さらに、パフォーマンスのボトルネックが発生した場合、生成されたSQLを直接制御できないため、それらを修正することは困難です。SQLでそれを行うか、データベース管理者に委任してください（あなたまたはあなたの会社にある場合）。

6.間違った丸め


さて、前の段落よりも少し単純なことについてですが、私はいつもそれを忘れてしまい、不快な間違いに終わりました（そして、それが財政に関連している場合は、怒っているひれ/遺伝子ディレクター）。

.NET FrameworkのRoundと呼ばれるMathクラスには、数値を受け取り、指定された小数点以下の桁数に丸める優れた静的メソッドが含まれています。ほとんどの場合完璧に機能しますが、2.25を小数第1位に四捨五入しようとするとどうなりますか？私はあなたがおそらく2.3に丸めることを期待していると思います-それは私たちがすべて慣れていることですよね？まあ、実際には、.NETは銀行丸めを使用していることがわかります与えられた例を2.2に丸めます！これは、数値が「中間点」にある場合、銀行員は最も近い偶数に丸められるためです。さいわい、これはMath.Roundメソッドで簡単にオーバーライドできます。

Math.Round(2.25,1, MidpointRounding.AwayFromZero)

7.恐ろしいクラス「DBNull」


これは一部の人にとって不愉快な思い出を引き起こす可能性があります-ORMはこの汚物を私たちから隠しますが、裸のADO.NET（SqlDataReaderなど）の世界を深く探ると、DBNull.Valueに出会います。

私は確かに理由の100％ではないよ、なぜデータベースからのNULL値は次のように処理されます（ご存知の場合はコメントしてください！）が、Microsoftは特殊なタイプのDBNull（静的フィールドValueを使用）でそれらを提示することにしました。これの利点の1つをあげることができます。NULLであるデータベースフィールドにアクセスするときに不快なNullReferenceExceptionが発生することはありません。ただし、NULL値をチェックする2番目の方法（忘れがちであり、重大なエラーが発生する可能性があります）をサポートするだけでなく、nullの操作に役立つC＃の優れた機能をすべて失う必要があります。簡単なことは何ですか

reader.GetString(0) ?? "NULL";

最終的には...

reader.GetString(0) != DBNull.Value ? reader.GetString(0) : "NULL";
