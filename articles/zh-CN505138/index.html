vMix呼叫和缩放之间的重要区别在于，它可以准确监视与扬声器连接的所有参数，并在必要时进行干预。例如，如果Internet连接的速度过低，您可以手动降低通过推回传输到扬声器的图像的质量。如果传出视频的质量下降，则传入视频（扬声器的网络摄像头）将更好地传输。

幻灯片：缩放


许多会议和聚会使用Zoom捕获幻灯片。不幸的是，这种幻灯片的显示质量直接取决于连接的质量。如果这是来自世界另一部分的居民，通过慢速ADSL线路连接，则质量可能会很差。

因此，在我们的会议上，幻灯片不会通过Zoom广播。扬声器会事先发送它们，并将它们上传到硬件室中的服务器。在报告期间，它们直接从此服务器广播。 “服务器”是指一台笔记本电脑，上面装有用于幻灯片放映的所有必要软件。这是一台装有PowerPoint或MacBook并带有Apple Keynote的Windows笔记本电脑。

通过在远程控制模式下通过“缩放”连接到此服务器，扬声器可以独立滚动幻灯片。要按下不必要的“前进”和“后退”按钮，通信质量会特别好。

这是一个不错的故障转移计划。例如，如果带有幻灯片的服务器坏了，那么我们总是有几台备用笔记本电脑将立即替换它，广播参与者甚至都不会注意到这种覆盖。如果“缩放”不起作用，则始终可以将其替换为Chrome远程桌面，VNC或任何其他远程控制工具。您甚至可以购买自己喜欢的TeamViewer，但它具有非常昂贵的商业许可证。

想象最坏的情况-圣彼得堡所有地区都关闭了电源。一枚核导弹飞进了我们在圣彼得堡的办公室。远程桌面由于某些根本原因已停止工作。但是在这里我们也有一个后备-我们可以继续通过位于莫斯科的机房进行广播。我们致电Zoom，三个来源同时参加：扬声器以及莫斯科和圣彼得堡的硬件。演示者从他的计算机播放幻灯片。如果其中一个城市关闭，则至少可以再切换一个。

如果演讲者想要在屏幕上进行某种演示或为几个人组织一份联合报告，则可以使用类似的技术。

当然，与此同时，幻灯片也变得非常清晰-普通，略带肥皂性，每个人都习惯了。但是您必须承认，这比根本没有报告要好得多。

如果总部与演讲者之间的网络连接丢失，并且演讲者位于世界的另一端，在美国的某个地方，我们将通过电话通过语音与他联系。我们将通过他的直接命令切换幻灯片。**应该始终有一个备份计划**。

演讲者和他自己的工作室


第四痛，最大程度


如果前面的示例中有一个在远程站点上讲话的人就像是暴风雨中的一艘船，那么具有自己工作室的讲话者就是团队叛逆的船。
它是什么样子的？

想象一下，演讲者拥有自己的笔记本电脑，并带有经过调节的Open Broadcaster程序-最受欢迎的免费流媒体解决方案。他可以在这台计算机上播放屏幕，进行实时编码，显示任何演示视频，自己更改计划等等。

然后，他通过RTMP将他的OBS连接到我们的服务器，并转发所有捕获的视频。该视频是简化的安装，可直接发送到Amazon的流媒体。

这里不受信任的环境不仅是Internet，而且演讲者一方的整个软件和硬件也很复杂。

对于演示者，能够独立控制情况听起来不错。但是在这种情况下，我们帮助发言人解决问题的能力非常有限。如果OBS开始下降怎么办？如果演讲者的计算机在尝试显示视频时突然开始变慢怎么办？各种各样的问题可能有数十个，数百个。

在某些情况下，说话者一侧的广播方案与我们狡猾的编辑系统根本不兼容。这发生在塞巴斯蒂安·达什纳（Sebastian Dashner）的身上，他不仅启动了一台计算机，而且还启动了两台计算机，从而“使”我们了。他参与其中的一项工作，第二项用作OBS的编辑站。我不得不通过Zoom进行流式处理，并绕过使用RTMP网关的漂亮方案。

应该始终有一个备份计划。在这种情况下，如果由于某种原因停止了他的OBS广播，您应该事先与发言人达成协议。最合乎逻辑的方法是切换到“常规”广播方式：通过Zoom连接并像往常一样继续报告。为此，演示者必须准备好使用这种“正常模式”-也就是说，他必须具有某种幻灯片，并且他必须想象要说的内容而不是显示屏幕。

亚马逊流媒体


“ CDN”代表“内容交付网络”。这是一个地理上分散的网络基础结构，使您可以优化向Internet用户的内容传递和分发。您可以直接在Habré上阅读有关CDN的更多信息。

换句话说，在您附近的某个地方，总是有一个服务器可以保存并快速提供您在浏览器中看到的所有图片和视频。如果每次您从位于主要亚马逊服务器所在的弗吉尼亚州的一台服务器下载所有图片，那么一半的站点将加载两倍的时间。否则根本就不会加载-因为网络服务器从负载中掉下来了。

在线会议上的视频需要通过CDN分发。但是，如果您能用自己的双手建造工作室或硬件室，甚至花费数百万卢布，那么您将无法以任何方式在世界范围内建造自己的CDN。

即使我们只想在莫斯科或圣彼得堡建立一台分发视频的服务器，也将是非常不愉快的。这种系统的特点是在负载下具有最大的可用性和容错能力。负载越大，要求资格较高的工程问题就越复杂。如果我们的服务器与哈巴罗夫斯克某人之间的网络连接消失，将会发生什么？如果由于不清楚的原因，带有视频服务器的容器每秒开始掉落五次，该怎么办？向已经知道如何做好的人付款比较容易。

我们已经研究了几种不同的视频流解决方案，目前专注于Amazon提供的产品。

流程如下：

  • 硬件的结果是视频流和预录制的视频文件。
  • 我们通过RTMP将视频流发送到Amazon MediaLive服务器 ;
  • 文件被发送到Amazon MediaConvert服务器；
  • 视频已上传到存储-Amazon S3；
  • 最后，数据从S3中提取出来并散布在CDN服务器-Amazon CloudFront中。

显然，所有服务都值得一些钱。确切的成本取决于区域和数量。刚开始时，这些金额并不令人印象深刻，但是当它们加起来时，已经相当不错了。例如，会议次数越多，需要存储的文件越多。

不幸的是，您不能立即将文件上传到CloudFront，必须先准备它们。具有不同Internet连接质量和不同屏幕尺寸的参与者需要不同质量的视频。

AWS Elemental MediaConvert是一种视频转码服务，具有成人广播公司级别的功能。视频文件是通过Web界面或API下载的，然后将其自身转换为任何所需的格式。如果您已将文件上传到MediaConvert，则提供编码的责任不再在您身上，而是在亚马逊上。费用在官方网站上。



AWS Elemental MediaLive大致相同，但用于视频流。这个想法是，您以最高的质量流式传输到MediaLive-并且它会从中生成适合不同设备和Internet速度的多个选项。



这些服务的功能看起来很酷，但并不便宜。例如，在购买代码转换服务时，可以理解它们可能会中断。如果代码转换器崩溃并在云端重新启动，则会浪费时间，并且会议参与者此时看不到任何内容，或者播放器中的广播中断。为避免发生这种恐怖情况，您需要一次为同一视频流购买多个转码器。

亚马逊简单存储服务（Amazon S3）是一种数据存储服务。您可以在那里无限量地填充数据。通常，S3可以用作任何事物的通用存储：您可以分发文件，可以分析大数据，可以同步IoT设备等等。在我们的案例中，需要使用S3作为MediaLive和MediaConvert呈现给我们的所有内容的中间存储。为了能够将视频传输到CDN，需要在设置中在S3中指定适当的存储段。



至于可靠性，S3提供了99.999999999％的可靠性（此处为11个9）。这样的可靠性甚至适用于银行，甚至更适合我们的视频。我们可以说这是整个方案中最稳定的部分-这种稳定性取决于我们为每兆字节向Amazon支付的钱。

Amazon CloudFront是通用CDN，它也可以交付视频。视频可以在俄罗斯以较低的延迟和较高的速度运行。CloudFront与AWS集成在一起，包括S3和Live服务。看起来像魔术：您在管理面板中配置了几个参数，突然，您的视频开始以高品质分发到世界各地。

使用CloudFront时需要付费- 仅当会议参与者真正观看视频时，金钱才会开始减少。非常重要的是，我们无需为S3和CloudFront之间的流量付费-亚马逊内部服务之间的流量是完全免费的。但是，如果我们尝试使用其他存储而不是S3，则将花费一分钱。

不幸的是，没有现成的平台可以从CloudFront接收视频并将其转变为真正的在线会议。更准确地说，有Amazon Media Package，但仍然可以修改文件。因此，我们必须基于HLS.js编写我们的网站，个人帐户，播放器，并为他们考虑特殊的“会议”功能。但这是另一篇文章的主题。
