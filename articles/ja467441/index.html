Dockerはcnab-to-ociをCNABプロジェクトに渡します...そして一般的にCNABは何ですか？

注意 ペレフ。：この記事は、コンテナーの世界からの最近の発表の翻訳です。先月、Dockerは次の開発をより広範なオープンソースコミュニティに移転することを発表しました。これは、CNABパッケージのメタデータをOCI標準（Open Container Initiative）のフォーマットに変換するためのツールに関するものでした-そのようなパッケージのコンテンツをコンテナレジストリ（Dockerレジストリなど）に配布する便利な可能性のために。しかし、これらすべてをよりよく理解するために、別のノート（ジャックウォーレンが新しいスタックのために書いたもの）を翻訳することから始めます-CNABとは何か。



CNABとは何ですか、なぜクラウドネイティブエコシステムにとって重要なのですか？


Cloud Native Application Bundle（CNAB）は、パッケージ化、インストール、コンテナ化されたアプリケーション管理を簡素化することを目的としたオープンな仕様です。ユーザーはこれらのパッケージを使用して、Docker、Azure、Kubernetes、Helm、自動化サービス（GitOpsで使用されるサービスなど）などのさまざまなランタイム環境にデプロイされるリソースを定義できます。

CNAB仕様は、Microsoft、HashiCorp、Bitnami、Intel、Pivo​​tal、DataDogなどの多くの企業間のコラボレーションの結果です。開発に関与している企業の多くは独自のクラウドサービスを提供していますが、CNABは特定のクラウド実装（クラウドに依存しない）に依存していません。言い換えれば、これはあらゆるインフラストラクチャおよびソフトウェアで機能し、特定のサービスプロバイダーに関連付けられていません。

しばらく前に、CNAB 1.0がリリースされました。つまり、プロジェクトは本番環境で使用できるようになりました。クラウドサービス/コンテナーに関連するITプロフェッショナルは、CNABで何が起こっているかを把握する必要があります。

追加の詳細


DockerのコラボレーションマーケティングディレクターであるRobert Duffnerは、次のように 述べています。 K8は非常に混乱し、非常に複雑になる可能性があるため、Kubernetesを使用したことがある人なら誰でもこの問題に精通しています。ダフナー氏は続けます。「現代のアプリケーションは、多くの多様なコンポーネントとサービスで構成されています。これらには、さまざまなクラウドリソース、マネージドサービス、SaaS製品、コンテナー、構成形式（ヘルムチャート、Kubernetes YAMLファイル、Docker Composeファイル）、機能などが含まれます。

しかし、CNABはそれとどのような関係があるのでしょうか。 「CNABはこれらの多様なコンポーネントを統合して、マルチサービスアプリケーションに単一のパッケージ形式を提供します。次に、これらのパッケージを操作し、（Docker Hubのようなレジストリを介して）特定の環境やクラウドの実装に依存しない複合で不変のオブジェクトであるかのように、それらを管理および共有できます。

この点を明確にするために、より具体的なトピックであるKubernetesとDockerに切り替えます（これらのテクノロジーはコンテナーデプロイメントの分野で最も重要な場所を占めています）。 DuffnerがKubernetesとDockerに対するCNABの利点を説明する方法は次のとおりです。「CNABは、ダイアログをアプリケーションレベルにプッシュすることで、コンテナーの構築、共有、および起動に関する考え方を変えるのに役立ちます。」

例の位置の例は、Dockerアプリです。：これは、まさにCNAB仕様の実装であるDockerアプリで起こったことです。これは、Dockerイメージに関連するシンプルさを、マルチサービスアプリケーションを構築するプロセス、その配布、およびさまざまな構成フォーマットを使用した起動に転送するためだけに開発されました。」
CNABで与えられている標準化への最大の注意は、コンテナーとサービスを単一の分割できない全体に組み立てることを可能にします。仕様は次の部分で構成されています。

  • CNAB -CNAB 1.0 Core Essentialsが含まれています
  • CNABレジストリ -OCIレジストリ内にCNABパッケージを格納する方法について説明します（このセクションはまだ完成していません）。
  • CNABセキュリティ -CNABパッケージの署名、検証、確認のメカニズムについて説明します。
  • CNABクレームは、CNABインストールレコードをストレージ用にフォーマットする方法を示す自己タイトルシステムに関するものです。
  • CNAB依存関係 -他のパッケージを依存関係としてパッケージに含める方法について説明します。

バンドルとは何ですか？


現在、コンテナ化されたアプリケーションをデプロイする方法はたくさんあります。Docker、Docker Compose、Kubernetes、およびその他のツールを使用できます。各ツールには、構成ファイルの形式に関する独自の要件があります。CNABは、単一のファイルに必要なすべてのメタデータを収集- bundle.jsonを。このファイルは、次のコンポーネントに分かれています。

  • パッケージスキーマのバージョン。
  • 高レベルのパッケージ情報。
  • 発信者情報（呼び出し）画像。
  • パッケージに含まれる画像のマップ。
  • 再定義されたパラメーターの仕様（および検証スキームへのリンク）。
  • アプリケーションが必要とする資格情報のリスト。
  • ユーザーアクションのオプションの説明。
  • アプリケーションによって表示される結果のリスト。
  • ユーザー入力の検証に使用される一連のスキーマ定義。

パッケージには2つのタイプがあります。

  • シン（シン）パッケージには、オブジェクトが1つだけ含まれています-記述子パケットのJSON形式（正規のJSONである必要があります）。
  • シック（シック）パッケージには、いくつかのオブジェクトパケット記述子、1つ以上の補助（誘導）イメージ、0以上のイメージが含まれています。シックパッケージにはbundle.json、圧縮アーカイブのルートにあるファイルが含まれています。

bundle.jsonは、イメージをデプロイするための包括的な構成を記述するという意味でdocker -compose.ymlに似ています。 2つの違いは、CNABがこのファイルの内容と形式、およびすべての関連ファイルの場所を明確に定義していることです。

パッケージのトピックに関して、Duffnerは次のように述べています。複雑なコンテナをデプロイしようとした人なら誰でもこれを知っています。CNABはこの複雑さをどのように処理しますか？「Dockerアプリ（CNAB実装）の利点の1つは、Docker Composeを使用してアプリケーションを簡素化し、それらを直接クラウドにデプロイできることです。そのため、明確に説明された、不変（不変）のアプローチと、クラウドネイティブアプリを展開する標準的な方法の可用性は、非常に重要です。」

誰が一番勝つ


当然のことながら、CNABは主に企業アプリケーションに焦点を当てています。確かに、コンテナーの単純な展開は、Fortune 500企業が使用するサービスの多面的な複雑さと比較できません。Duffnerは次のように述べています。「大企業はCNABの主な消費者です。それらの多くは、数百または数千もの分散アプリケーションを使用しており、それらのアセンブリを簡素化し、それらを管理し、ソフトウェアサプライチェーン全体のセキュリティを強化することを夢見ています。」

ただし、大企業だけがCNABの恩恵を受けるわけではありません。ダフナー氏は、最新のアプリケーションの展開の複雑さに現在取り組んでいるDevOps開発者とチームは、CNAB仕様の実装から大きな恩恵を受けると信じています。その支援により、彼らは「相互作用を改善し、ソフトウェアを市場に出すまでの時間を短縮する」ことができるようになります。

Dockerはcnab-to-ociライブラリをcnab.ioに渡します


2月12日、会社のDockerは、ライブラリのcnab-to-ociプロジェクトCNABの無償譲渡を発表しました。このプロジェクトは昨年、MicrosoftとDockerがCNAB仕様をLinux Foundationの一部であるJoint Development Foundationに移行した後のことです。次に、CNAB仕様リポジトリがGitHub のdeislabリポジトリから新しいcnabioリポジトリに移動しました。CNAB-Goのライブラリー、移動中に書かれており、仕様の実装を含む、およびダッフル（リファレンスCLIの実装では）また、新しいリポジトリに移動しました。

cnab-to-ociライブラリの目的は何ですか？


Dockerチームは、CNAB仕様とその元の実装の開発を支援し、cnab-to-ociライブラリも主導しました。後者の目的は、既存のコンテナーレジストリを介してCNABパッケージ（バンドル）を配布することです。現在、3つのCNABツールがこのライブラリを使用しています：Docker App、Porter and duffle、およびDocker Hub。レジストリの助けを借りて、レジストリはCNABパケットをプッシュおよびプルしたり、それらを共有したりできます。このライブラリの作成中に行われた開発は、CNABレジストリの将来の仕様の基礎として使用されます。

リポジトリの転送自体はすでに発生しているため、現時点から、Go-importsで次を参照する必要があります。github.com/cnabio/cnab-to-oci。

cnab-to-ociはCNABパケットをレジストリにどのように格納しますか？


おそらくご存知のとおり、OCIイメージ仕様にはOCIマニフェストとOCIイメージインデックスの 2つの主要なオブジェクトがあります。1つ目はよく知られており、古典的なDockerの外観を表しています。2番目は、最初はマルチアーキテクチャイメージを格納するために使用されました（例については、nginxを参照してください）。

ただし、仕様でOCIインデックスの使用がマルチアーキテクチャイメージのみに制限されていないことに気付かない場合があります。彼らの助けを借りて、あなたはあなたが欲しいものを保存することができます。主なことは、コンテンツが仕様を満たしていることです（ちなみに、これはかなりオープンです）。

cnab-to-ociはこのオープンファイルをプッシュファイルに利用しますbundle.json、呼び出しイメージ、コンポーネントイメージ（またはDockerアプリのサービスイメージ）。その結果、すべてが同じリポジトリに保存されます。したがって、誰かがパッケージをプルしたときに、すべてのコンポーネントを一緒に取得することもできます。
