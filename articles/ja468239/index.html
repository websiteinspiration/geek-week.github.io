Swiftで適切なデータ構造を選択する

また会いましたね。週末に出発する前に、基本コース「iOS Developer」のために特別に準備された資料の翻訳を皆さんと共有したいと思います。



特定の値のセットを表すために使用するデータ構造を決定することは、見かけよりもはるかに難しいことがよくあります。各タイプのデータ構造は特定の数のユースケースに合わせて最適化されているため、各データセットに適切なものを選択すると、コードのパフォーマンスに大きな影響を与えることがよくあります。

-スウィフト標準ライブラリは、3つの主要なデータ構造が付属しArray、DictionaryそしてSet最適化、プラスとマイナスの独自のセットをそれぞれ有します。それらの特性のいくつかと、目的に合った適切なデータ構造を見つけるために標準ライブラリを超える必要がある場合を見てみましょう。

アレイの直線性


Array（配列）はおそらくSwiftで最も一般的に使用されるデータ構造の1つであり、これには十分な理由があります。それはその要素を順番に格納し、予測可能な方法で簡単にソートされ、構造体からクラスのインスタンスや他のコレクションまで、あらゆる値を格納できます。

たとえば、ここでは配列を使用しCanvasて、描画アプリケーションに配置された図形のコレクションを格納しています。次に、キャンバスを画像にレンダリングする必要がある場合DrawingContext、次のように-を使用して配列を調べ、各要素を描画します。

struct Canvas {
    var shapes: [Shape]

    func render() -> Image {
        let context = DrawingContext()
        shapes.forEach(context.draw)
        return context.makeImage()
    }
}

上記で行ったように、すべての形状の順次レンダリングに関しては、配列は完全に適合します。配列は、非常に効率的な方法で要素を格納するだけでなく、並べ替え順序が保証されているため、追加の作業を行わなくても、予測可能なレンダリング順序を提供できます。

ただし、他のすべてのデータ構造と同様に、配列には欠点があります。私たちの場合、キャンバスから形状を削除するときに、欠点の1つに遭遇します。配列の要素はインデックスによって格納されるため、図を削除する前に、常に図が関連付けられているインデックスを常に見つける必要があります。

extension Canvas {
    mutating func remove(_ shape: Shape) {
        guard let index = shapes.firstIndex(of: shape) else {
            return
        }

        shapes.remove(at: index)
    }
}

最初は、上記のコードはそれほど問題に見えないかもしれませんがfirstIndex、時間の複雑さの点で線形（O（N））であるため、多数の形状を含むキャンバスのパフォーマンスのボトルネックになる可能性があります。

私たちのタイプのCanvasを使用すると、この制限を回避できます。たとえば、常に値やIDではなくインデックスで図を参照します。これにより、コードがより複雑で壊れやすくなります。これは、操作しているキャンバスのときにインデックスが期限切れにならないようにする必要があるためです。変更されます。

スピードセット


代わりに、Canvas基礎となるデータ構造を変更することで自分自身を最適化できるかどうかを見てみましょう。上記の問題を考慮すると、最初のアイデアの1つは、Set代わりに（セット）を使用することArrayです。「Swiftのセットの能力」ですでに説明したように、配列と比較した場合のセットの大きな利点の1つは、挿入と削除の両方を常に一定の（O（1））時間で実行できることです。アイテムはインデックスではなくハッシュ値で保存されるためです。セットがその中で使用されるように

更新Canvasすることにより、次のようになります。

struct Canvas {
    var shapes: Set<Shape>

    func render() -> Image {
        let context = DrawingContext()
        shapes.forEach(context.draw)
        return context.makeImage()
    }

    mutating func remove(_ shape: Shape) {
        shapes.remove(shape)
    }
}

繰り返しになりますが、上記のコードは正しく見え、問題なくコンパイルできます。ただし、削除の問題は解決しましたが、安定したレンダリング順序も失われました。これは、配列とは異なり、セットは保証された並べ替え順序を提供しないためです。これは、この状況では障害になるため、無作為に。

インデックス作成


実験を続けましょう。次に、（辞書）をCanvas入力して最適化できるかどうかを確認Dictionaryします。これにより、IDに基づいて任意の形状のインデックスを検索できます。私たちは、アレイのアクセスレベルを変更することで開始しますshapesするためprivateの新しい方法を使用して、要素の挿入を制御することができるようにしますadd。また、新しい図が追加されるたびに、そのインデックスも辞書に追加します。

struct Canvas {
    private var shapes = [Shape]()
    private var indexes = [Shape.ID : Int]()

    func render() -> Image {
        let context = DrawingContext()
        shapes.forEach(context.draw)
        return context.makeImage()
    }

    mutating func add(_ shape: Shape) {
        let index = shapes.count
        indexes[shape.id] = index
        shapes.append(shape)
    }
}

これで、与えられた図が格納されているインデックスが常にわかるので、セットを使用する場合と同様に、一定の時間ですばやく削除を実行できます。

extension Canvas {
    mutating func remove(_ shape: Shape) {
        guard let index = indexes[shape.id] else {
            return
        }

        shapes.remove(at: index)
        indexes[shape.id] = nil
    }
}

ただし、新しい実装Canvasにはかなり深刻なバグがあります。図形を削除するたびに、削除したインデックスよりも高いすべてのインデックスを実際に無効にします。これらの各要素は、配列の先頭に1ステップ移動するためです。削除するたびにこれらのインデックスを調整することでこの問題を解決できましたが、これにより再びO（N）テリトリーに戻ることになり、最初から回避しようとしました。

最新の実装にはメリットがあります。一般に、このような状況では2つのデータ構造を組み合わせて使用​​することをお勧めします。1つのデータ構造の長所を使用して他のデータ構造の短所を補ったり、その逆の場合が多いためです。

では、別の組み合わせでもう一度試してみましょう。ただし、今度は実際の要件を見てみましょう。

  • 一定の時間の複雑さを持たせるには挿入と削除の両方が必要であり、ベースインデックスを知らなくても図を削除できるはずです。
  • 安定したレンダリング順序を維持するには、ブルートフォースの保証された順序が必要です。

上記の要件を見ると、安定した並べ替え順序が必要ですが、実際にはインデックスは必要ないことがわかります。これにより、リンクリストが私たちのユースケースに最適になります。

リンクリストはノードで構成されており、各ノードにはリスト内の次のノードへのリンク（またはリンク）が含まれています。つまり、予測可能な方法でソートできます。項目が削除されたときにインデックスを更新する必要はありません。ただし、Swift標準ライブラリ（現時点では）にはリンクリストタイプが含まれていないため、それを使用する場合は、最初に作成する必要があります。

リンクリストを作成する


Listリストの最初と最後のノードを追跡 する構造（リスト）を宣言することから始めましょう。データの整合性を確保するために、これらのプロパティの両方をタイプの外で読み取り専用にします。

struct List<Value> {
    private(set) var firstNode: Node?
    private(set) var lastNode: Node?
}

次に、Nodeクラスを作成するタイプ（ノード）を作成しましょう。値ではなく参照でノードを参照できるようにするためです。リストは二重に接続されます。つまり、各ノードには次の隣接ノードと前のノードの両方へのリンクが含まれます。各ノードには、次のように値も格納されます。

extension List {
    class Node {
        var value: Value
        fileprivate(set) weak var previous: Node?
        fileprivate(set) var next: Node?

        init(value: Value) {
            self.value = value
        }
    }
}

以前のプロパティを弱くする理由は、両方向に強いリンクを保持した場合に表示される保持ループを回避するためです。保持サイクルの回避の詳細については、「メモリ管理」の記事を参照してください。
これが実際に必要なすべてのコードであり、リンクされたリストに値を格納できます。しかし、これはパズルの最初の部分にすぎません。他のコレクションと同様に、パズルを反復してその内容を変更できるようにしたいと考えています。反復から始めましょう。プロトコル指向のSwift設計のおかげで、プロトコルへの準拠を確実にしSequence、メソッドを実装することで簡単に実装できますmakeIterator。

extension List: Sequence {
    func makeIterator() -> AnyIterator<Value> {
        var node = firstNode

        return AnyIterator {
             //   ,    ,    :
            let value = node?.value
            node = node?.next
            return value
        }
    }
}

上記の反復は非常に単純であるため、AnyIterator標準のライブラリを使用して、カスタムの反復子タイプを実装する必要をなくしています。より複雑なシナリオでは、に一致を追加することで実装できますIteratorProtocol。
次に、リンクリストを変更するためのAPIを追加して、挿入から始めましょう。挿入された値の新しいノードを追加し、このノードを返すListメソッドappendを使用して、次のように展開します。

extension List {
    @discardableResult
    mutating func append(_ value: Value) -> Node {
        let node = Node(value: value)
        node.previous = lastNode

        lastNode?.next = node
        lastNode = node

        if firstNode == nil {
            firstNode = node
        }

        return node
    }
}

上記では、@discardableResult作成されたノードに常に関心があるわけではないため、メソッドの呼び出しの結果が使用されなかった場合に警告を生成しないようコンパイラーに指示する属性を使用しています。
リンクされたリストはインデックスに基づくのではなく、リンクを介した値のチェーンの維持に基づいているため、削除の実装は、リモートノードの次の隣接ノードと前の隣接ノードを更新して、それらが代わりに互いを指すようにするだけです。

extension List {
    mutating func remove(_ node: Node) {
        node.previous?.next = node.next
        node.next?.previous = node.previous

        //  « »,        ,    :
        if firstNode === node {
            firstNode = node.next
        }

        if lastNode === node {
            lastNode = node.previous
        }
    }
}

上記に基づいて、リストの初期バージョンが完成し、実際にチェックする準備が整いました。Canvasを更新して、新しいリストと、指定されたシェイプIDに一致するノードをすばやく見つけることができる辞書を使用します。

struct Canvas {
    private var shapes = List<Shape>()
    private var nodes = [Shape.ID : List<Shape>.Node]()

    func render() -> Image {
        let context = DrawingContext()
        shapes.forEach(context.draw)
        return context.makeImage()
    }

    mutating func add(_ shape: Shape) {
        nodes[shape.id] = shapes.append(shape)
    }

    mutating func remove(_ shape: Shape) {
        guard let node = nodes.removeValue(forKey: shape.id) else {
            return
        }

        shapes.remove(node)
    }
}

これで、呼び出しプロセスをさらに複雑にする必要なしに、挿入と削除の両方が簡単になり、並べ替え順序が予測可能になりました。これは非常に便利です。また、新しいListは完全にユニバーサルなタイプであるため、インデックスを使用せずに線形に値を格納する必要がある場合はいつでも使用できます。
