実際のプロジェクトでのアプリケーションに関するReact NativeとFlutterの比較

React NativeはFlutterとどのように異なりますか？ただし、異なるテクノロジーに基づいた異なるフレームワークについては例外です。クロスプラットフォームアプリケーションを開発するためのこれらのツールに精通していないが、次のプロジェクトのためにそれらの1つを選択したい人は、何をガイドする必要がありますか？



翻訳を公開している記事の著者は、最近React NativeとFlutterの多くの比較に遭遇したと述べています。これらすべての比較では、実際にはそれほど重要ではないことについて話していました。たとえば、プロジェクトの開発に使用される言語やコマンドラインツールについてです。ここでは、React NativeとFlutterを、実際のプロジェクトの重要性に関して、実際の状況について比較します。この比較は、必要なものを正確に選択するのに役立つ情報を全員に提供することを目的としています。

序論


ここでは、React NativeやFlutterを批判するつもりはありません。いわば、それらを荒らすつもりはありません。あなたが学んだことをどのように正確に処分するかは、あなた次第です-あなたがすでに知っていること、そしてどのように知っているか、プロジェクトのニーズに依存します。

3年前にReact Native（RN）を使い始めました。それ以前は、React.jsとNode.jsを使用してプロジェクトを開発していました。結果として、私がRNを選んだのは当然のことです。しかし、私はFlutterも本当に好きです。私はこのフレームワークの成長と発展を見てきましたが、彼には明るい未来が待っていると確信しています。 RNとFlutterはどちらも優れたツールであり、次のアプリケーションのベースを探している人が注目する価値があります。

私は毎日RNと仕事をしていますが、Flutterについて同じことを言うことはできません。したがって、私の計算を批判的に扱います。そして、私が何かを間違えているとわかったら、それを知らせてください。

言語


たとえば、DartおよびJavaScript言語、ネイティブコンポーネント、およびコードが自動的に生成されるコンポーネントに特化した多くの比較があります。ここではそのような比較は行いません。これについてはすでに多くのことが書かれています。必要な場合は、適切な資料を自分で見つけることができます。

Flutterアプリケーションの開発に使用されているDart言語は静的型付けを使用していると言ってよいでしょう。 JavaScriptの代替としてGoogleによって作成されました（ただし、Googleは他のブラウザメーカーにプロジェクトにDart仮想マシンのサポートを含めるように説得することはできませんでした。最終的にはChromeのみをサポートします）。 React Nativeプロジェクトにコードを書き込むJavaScriptは動的型付けを使用しますが、静的型付けに関心のある人は誰でもTypeScriptを使用してJSアプリケーションを作成できます。

ここでは、IDE、およびコードを自動生成するためのコマンドラインツールなどについて説明します。

しかし、実際には、これらすべてはそれほど重要ではありません。開発者は、特定のフレームワークが彼らの目標の達成において彼らに与える自由と行動の自由について最も懸念しており、一見、興味深いと思われるものではありません。何にでも慣れることができます。

したがって、私は今、特定のフレームワークを支持して表明された一般的な議論のいくつかを議論することを提案します。彼らは文字通りどこにでも見つけることができます。私はそれらについて読み、同僚から聞いた。それらは見た目ほど明確ではありません。

正直なところ、この領域ではFlutterとReact Nativeの間に大きなギャップがあります。Flutterチームは、開発ツールをVS CodeおよびAndroid Studioと統合するために一生懸命取り組んできました。プロジェクトをデバッグおよび分析するための多くのツールが作成されています。

一方、react-devtoolsパッケージといくつかの小さなプラグインを無視した場合、RNアプリケーションのデバッグは、主にそれぞれのプラットフォームの組み込み機能に依存します。変化をもたらすために、Facebookは最近モバイルアプリケーション用のデバッガーFlipperをリリースしました。しかし、これはまだパイロットプロジェクトです。

両方のフレームワークの資産には強い議論があることは容易に理解できます。次に、FlutterとReact Nativeの主な違いを見てみましょう。

主な違い


▍編集と「空気による更新」


JavaScriptコードのコンパイルは、事前に（AOTコンパイル）ではなく、実行する必要があるときに（JITコンパイル）実行されるため、React Nativeを使用すると、いわば「空気でアプリケーションを更新」できます。実際、実装は非常に簡単です。たとえば、Microsoft CodePushまたはExpo.ioを使用します。

これが私がFlutterに段階的に移行しない主な理由です（Reactパラダイムが好きであるという事実に加えて）。これは、特にいくつかの短期的なイベントに関連する特定のアプリケーションに関しては、力のバランスを根本的に変える要因です。このような状況では、アプリケーションの変更は非常に高速に行う必要があります。これらのアプリケーションは短期間で多くのユーザーを獲得しています。

開発環境がなく、コンパイルについて考えることなく、React Nativeアプリケーションを作成できます。たとえば、snack.expo.ioプロジェクトの使用。これは、特定の概念の操作性の証明を示す単純なプロジェクトや、エラーを報告してReact Native開発チームに示される予定のプロジェクトを作成するのに非常に便利です。

DartはJITコンパイルもサポートしますが、CodePushなどを使用して「無線で」Dartアプリケーションに更新を送信することはできなくなりました。もちろん、インストールインテントへの呼び出しを開始することもできますが、これはRNを使用して実現できることとはかけ離れています。

おそらく将来的には、フラッター開発チームが「無線で」アプリケーションの更新に関する質問に適切な回答を提供するでしょうが、これを実装する公式にサポートされているライブラリのようなものは今のところありません。

▍開発ツール


Flutter開発者は、優れたデバッグツールを使用できます。これはFlutterの開発の容易さを支持する強力な議論です。さらに、これはアプリケーションの最適化を非常に容易にします。

userユーザーインターフェースの説明


React NativeはJSXを使用してインターフェースを記述するため、作業がはるかに簡単になります。そしてFlutterはウィジェットベースのインターフェースを使用しています。このアプローチは詳細が多すぎて、開発者にとって特にフレンドリーではないようです。

Flutterインターフェイスの開発は、大量のボイラープレートコードの存在を意味します。 IDEを使用しない場合、インターフェースの説明は、ナビゲートが難しい入れ子構造の本当の地獄に変わります。これにより、プルリクエストのレビューが複雑になる可能性があります。

さらに、ここでは、私が知る限り、ウィジェット自体からクリーンなデザインを分離する通常の手段はありません。これは、RNのスタイル付きコンポーネントライブラリのようなもの、またはStyleSheet.createプロパティを通じて発生したことのコンポーネントへの転送を伴う呼び出しのようなものstyleです。

JSXを使用すると、シンプルで読みやすいインターフェース記述を作成できます。この状況は、styled-componentsなどの専用ライブラリによってさらに強化されます。これらすべてにより、RNはFlutterよりも優れた利点があります。以前にWebプロジェクトに関与し、React Nativeに切り替えた開発者は、非常に迅速に対応できます。

どちらのフレームワークも、マテリアルを「ホット」リロードする機能をサポートしています。これらの機能を使用すると便利です。これはiOSとAndroidの両方に適用されます。

コミュニティと図書館


JavaScriptエコシステムは巨大で、非常に多様なツールとライブラリがあります。しかし、それはJSコミュニティよりもはるかに若くて小さいですが、Dart開発者のコ​​ミュニティは良さそうです。

Dartアプリケーションを開発する場合、多くのパッケージを使用できます。ただし、ほとんどのITスタートアップは、現在、DartベースのSDKを、それらを使用したい人のために提供していません。しかし、Node.jsの開発ツールは、この環境で非常に人気があります。

Dartは、これがGoogleプロジェクトであり、この言語用の多くの有用なライブラリを作成することを会社が担当する可能性が高いという事実を支持しています。

厳密に言えば、コミュニティーのサポートとライブラリーの可用性の観点から、React NativeはFlutterをバイパスしています。
