Kubernetesの仕組み


私たちの話では、アリスとボブはしばらくKubernetesで作業した後、GitOpsを利用しました。実際、GitOpsはKubernetesと密接に関連しており、Kubernetesに基づくインフラストラクチャとアプリケーションの運用モデルです。

Kubernetesはユーザーに何を提供しますか？


主な機能は次のとおりです。

  1. Kubernetesモデルでは、すべてを宣言形式で記述できます。
  2. Kubernetes APIサーバーは、そのような宣言を入力として受け入れ、その後、クラスターを宣言に記述されている状態に常に切り替えようとします。
  3. 宣言は、さまざまなワークロード（「アプリケーション」）を記述および管理するのに十分です。
  4. その結果、アプリケーションとクラスターに対する変更は次の原因によるものです。
    • コンテナイメージの変更。
    • 宣言仕様の変更。
    • 環境のエラー-たとえば、コンテナのクラッシュ。

発散とは何ですか？


もう一度繰り返します。クラスターのすべての必要なプロパティは、クラスター自体で監視できる必要があります。

分岐のいくつかの例：

  • Gitのブランチのマージによる構成ファイルの変更。
  • GUIクライアントによるGitでのコミットによる構成ファイルの変更。
  • GitでのPRによる望ましい状態の複数の変更、それに続くコンテナーイメージのアセンブリと構成の変更。
  • エラー、「不適切な動作」につながるリソースの競合、または元の状態からの偶発的な逸脱によって、クラスターの状態が変化します。

収束メカニズムとは何ですか？


いくつかの例：

  • コンテナーとクラスターの場合、収束エンジンはKubernetesを提供します。
  • 同じメカニズムを使用して、Kubernetesベースのアプリケーションとデザイン（IstioやKubeflowなど）を管理できます。
  • Kubernetes、イメージリポジトリ、Gitの間の相互作用を管理するメカニズムは、Weave Cloudの一部であるWeave Flux GitOpsオペレーターによって提供されます。
  • 基本マシンの場合、収束メカニズムは宣言的で自律的でなければなりません。私たち自身の経験から、Terraformはこの定義に最も近いと言えますが、それでも人間による制御が必要です。この意味で、GitOpsはインフラストラクチャの伝統をコードとして拡張します。

GitOpsは、GitとKubernetesの優れた収束エンジンを組み合わせて、運用モデルを提供します。

GitOpsを使用すると、記述および監視できるシステムのみを自動化および制御できることを宣言できます。

GitOpsはクラウドネイティブスタック全体（Terraformなど）用です


GitOpsはKubernetesだけではありません。システム全体を宣言的に管理し、収束を使用する必要があります。システム全体とは、Kubernetesで動作する一連の環境を意味します。たとえば、「dev cluster 1」、「production」などです。各環境には、マシン、クラスター、アプリケーション、およびデータ、監視、など

テラフォームは、ブートストラップのために、この場合にいかに重要であるかに注意してください。 Kubernetesはどこかにデプロイする必要があります。Terraformを使用すると、同じGitOpsワークフローを使用して、Kubernetesおよびアプリケーションの基礎となるコントロールレイヤーを作成できます。これはベストプラクティスです。

Kubernetes上のレイヤーにGitOpsの概念を適用することに多くの注意が払われています。現在、Istio、Helm、Ksonnet、OpenFaaS、およびKubeflow用のGitOpsタイプのソリューションがあり、たとえば、Plumiもあり、クラウドネイティブのアプリケーションを開発するためのレイヤーを作成します。

Kubernetes CI / CD：GitOpsと他のアプローチの比較


述べたように、GitOpsは次の2つです。

  1. 上記のKubernetesおよびクラウドネイティブの運用モデル。
  2. 開発者中心のアプリケーション管理環境を編成する方法。

多くの場合、GitOpsは主にGitプッシュベースのワークフローです。私たちも彼が好きです。しかし、それだけではありません。CI/ CDパイプラインを見てみましょう。

GitOpsはKubernetesに継続的デプロイメント（CD）を提供します


GitOpsは、個別の「展開管理システム」の必要性を排除する継続的な展開メカニズムを提供します。Kubernetesがすべての作業を行います。

  • アプリケーションを更新するには、Gitで更新する必要があります。これは、望ましい状態へのトランザクションアップグレードです。その後、更新された説明に基づいて、Kubernetes自体によって「デプロイ」がクラスター内で実行されます。
  • Kubernetesの詳細により、これらの更新は収束的です。これは、すべての更新がアトミックである継続的な展開のメカニズムを提供します。
  • 注：Weave Cloudは、GitとKubernetesを統合するGitOpsオペレーターを提供し、クラスターの望ましい状態と現在の状態を一致させることによってCDを実行できるようにします。

CI CD


GitOpsは既存のCI / CDモデルを改善します。

最新のCIサーバーは、オーケストレーションのための手段です。特に、CIパイプラインを編成するための手段です。ビルド、テスト、トランクへのマージなどが含まれます。CIサーバーは、複雑なマルチステップパイプラインの管理を自動化します。一般的な誘惑は、Kubernetes更新セットのスクリプトを作成し、それを変更をクラスターにプッシュするためのパイプライン要素として実行することです。実際、これは多くの専門家が行うことです。ただし、これは最適ではありません。その理由は次のとおりです。

CDを「内部」で管理するには、トランクを更新するためにCIを使用する必要があり、Kubernetesクラスターはこれらの更新に基づいて自身を変更する必要があります。CDのプルモデルと呼んでいます、CIプッシュモデルとは異なります。CDは、ランタイムオーケストレーションの一部です。

CIサーバーがKubernetesの直接更新を通じてCDを作成してはならない理由


CIサーバーを使用して、Kubernetesでの直接更新を一連のCIタスクとして調整しないでください。これは、既にブログで説明したアンチパターンです。

アリスとボブに戻りましょう。

彼らはどのような問題に遭遇しましたか？BobのCIサーバーは変更をクラスターに適用しますが、プロセスでクラッシュした場合、Bobはクラスターの状態（またはその状態）とその修正方法を認識しません。成功した場合も同様です。

Bobのチームが新しいイメージをまとめてから、それらのデプロイメントにパッチを適用してイメージをデプロイするとします（すべてCIパイプラインから）。

イメージは正常にビルドされたが、パイプラインが落ちた場合、チームは以下を見つける必要があります。

  • アップデートは公開されましたか？
  • 新しいビルドを開始していますか？これは不必要な副作用につながりますか？同じ不変のイメージの2つのアセンブリを取得する機能がありますか？
  • アセンブリを開始する前に、次の更新を待つ必要がありますか？
  • 何が間違っていたのですか？どのステップを繰り返す必要がありますか（どのステップを繰り返しても安全ですか？）

Gitベースのワークフローを整理しても、Bobのチームがこれらの問題に遭遇しないとは限りません。それでも、タグやその他のパラメーターを使用して、コミットのプッシュプッシュと間違われる可能性があります。ただし、このアプローチは、明示的なオールオアナッシングに非常に近いものです。

要約すると、これがCIサーバーがCDを処理してはならない理由です。

  • 更新スクリプトは常に確定的ではありません。間違いを犯しやすいのです。
  • CIサーバーは宣言型クラスターモデルに変換されません。
