ベンチマークバッチコンベヤ


バッチパイプラインは、保存されているデータの最終的な量を処理します。処理結果のストリームはありません。集計関数の出力をデータセット全体に適用する必要があります。これによりパフォーマンス要件が変わります。データをリアルタイムで処理するため、ストリーミング処理の重要な要素であるレイテンシはここにはありません。唯一の重要なメトリックは、コンベヤの合計実行時間です。

したがって、Parallelを選択しました。テストの最初の段階で、1つのノードで作業しているとき、このコレクターは実際に最高のスループットを示しました（ただし、調整後のみ）。ただし、これは長い休止を犠牲にして取得されました。クラスターノードの1つがガベージコレクションで停止すると、パイプライン全体が停止します。また、ノードは異なるタイミングでガベージを収集するため、クラスターに各ノードを追加すると、総ビルド時間は増加します。1つのノードと3つのノードのクラスターのテスト結果を比較することにより、この影響を分析しました。

また、この段階では、実験的な低レイテンシコレクタについては考慮していません。それらの非常に短い休止はテスト結果に影響を与えません、さらに、これは帯域幅のために達成されます。

単一ノードでのテスト：コンベヤー


単一ノードのバッチベンチマークでは、単純なパイプラインを使用しました。完全なコードはこちら。

p.readFrom(longSource)
 .rebalance() // Introduced in Jet 4.2
 .groupingKey(n -> n % NUM_KEYS)
 .aggregate(summingLong(n -> n))
 .filter(e -> (e.getKey() & 0xFF_FFFFL) == 0)
 .writeTo(Sinks.logger())

ソースは、タイプ番号のシーケンスを生成するスタブですlong。キー関数はgroupingKey、キースペースを循環するように定義されています：0、1、2、... NUM_KEYS、0、1、2、...これは、最初のサイクル中にパイプラインがすべてのキーを参照し、結果を保存するための固定データ構造を作成することを意味します集約。そして、次のサイクルでは、システムは利用可能なデータのみを更新します。これは、異なる世代を考慮したガベージコレクションの仮説と完全に一致します。オブジェクトはすべての計算を通過するか、オブジェクトが長く存在せず、作成後すぐにガベージになります。

ソースは4億個の要素を生成し、1億個の個別のキーを作成しました。つまり、4回すべてのキーを処理しました。

オペレーター.rebalance()Jetのデフォルトの2ステージ集約を1 ステージ集約に置き換えます。これにより、ベンチマークでのエンジンの動作がより予測可能になりました。

また、集計操作でタイプとしてオブジェクトのインスタンスを状態として使用しLong、現在のアカウントが更新されるたびにガベージを生成するオプションもテストしました。この場合、多くのオブジェクトは古い世代でまともな時間の後に死にます。この状況では、1億ではコレクターの負荷が高すぎるため、キーの数を7000万に減らす必要がありました。

バッチパイプラインの場合には提供するものがないため、低レイテンシコレクターには焦点を当てませんでした。JDK 14はJDK 11とほぼ同じように機能することをすでに確認しているため、これを確認するために1つのテストを実行しました。次に、JDK 8とJDK 11に注目しました。また、JDK 8で使用されているデフォルトのParallelコレクターをG1と比較しました。

単一ノードでのテスト：結果


16 GBのRAMと6コアのIntel Core i7を搭載したラップトップでベンチマークを実行しました。ヒープサイズは10 GBでした。

最初は、Parallelが原因でパフォーマンスが非常に低く、ガベージコレクションを調整する必要がありました。したがって、VisualVMとVisual GCプラグインを使用することを強くお勧めします。最大フレームレート（10 fps）を設定すると、アプリケーションメモリの割り当てとガベージコレクターの作業との関係を非常に詳細に視覚化できます。アニメーションを見て、主な問題は、新しい世代に割り当てられたメモリのサイズが大きすぎることに気付きました。デフォルトでは、古い世代と新しい世代の比率は2：1のみで、実行中は動的に変化しません。そのため、設定を適用することにしました-XX:NewRatio=8、それが全体像を変えました。現在、Parallelが最も効果的でした。また-XX:MaxTenuringThreshold=2、一時オブジェクトはパイプラインですぐに停止するため、サバイバースペースを適用してスペース間のデータコピーを削減しました。

次に結果について。バッチパイプラインに適した唯一の指標は、タスクの期間です。結果を視覚化するために、それらの逆の値を取ったため、グラフはスループットを1秒あたりの要素数で示しています。 1つのノードの場合：



ガベージコレクションなしの集計は、より大きなキーのセットにもかかわらず、約30〜35％速く動作します。 G1とJDK 8のバンドルは最悪の状態で動作し、ParallelとJDK 11のバンドルは最適に調整されました。G1とJDK 11のバンドルはそれほど損失しませんでした。G1の構成には触れなかったことに注意してください。これは重要な点です。ガベージコレクションの構成は状況に大きく依存します。データ量の増加などにより、結果は劇的に変化する可能性があります。また、特定のタイプの負荷に対して、クラスター全体を構成する必要があります。

3つのノードでのテスト：コンベヤー


クラスタを正しくテストするには、より複雑なパイプラインを使用する必要がありました。

p.readFrom(longSource())
 .rebalance()
 .flatMap(n -> {
     Long[] items = new Long[SOURCE_STEP];
     Arrays.setAll(items, i -> n + i);
     return traverseArray(items);
 })
 .rebalance()
 .groupingKey(n -> n % NUM_KEYS)
 .aggregate(AggregateOperations.summingLong(n -> n))
 .filter(e -> e.getKey() % 1_000_000 == 0)
 .writeTo(Sinks.logger())
;

ソースは並行して動作しないため、システムのボトルネックにならないようにいくつかの最適化を行いました。ソースは数値0、10、20、...を生成しflatMap、不足している数値が補間されるステージを並列化しました。また、ソースとflatMap使用したrebalance()データの間で、クラスター全体にデータを分散しました。そして、メインステージ（キーアグリゲーション）の開始前に、再びバランスを調整しました。集計後、最初に100万個おきのキーと値のペアのみを残してから、ロガーに渡します。10億のデータ要素と5億のキーのセットが使用されました。

ノードが1つの場合と同様に、コンベヤをガベージなしのアグリゲーションとそれを使用してテストしました。

3つのノードでのテスト：結果


このベンチマークは、3つのc5d.4xlargeインスタンスからのAWSクラスターで実行されました。 16の仮想化されたプロセッサコアと32 GBのメモリを備えていました。チャネル帯域幅10 Gb /秒。結果：



ちなみに、単一のノードと比較してスループットが全体で約3倍に増加していることに注意してください。これは、分散処理の結果です。アセンブラに関しては、G1とJDK 11の組み合わせが両方のテストの勝者となりました。もう1つの顕著な結果は、G1とJDK 8のほぼ操作不能な組み合わせですが、これには他の次元に影響を与えるより深い理由があります。たとえば、JDK 8およびJDK 11でのParallelの明らかな利点は、冒頭で述べた効果によるものです。ガベージコレクションのためにノードが一時停止するとすぐに、クラスター全体の処理が停止します。そして、JDK 8のG1は、1分以上の非常に長い一時停止で上昇します。これは、クラスター内の障害検出器が機能し、ノードが停止したと判断するのに十分です。タスクは失敗し、クラスターはそれ自体を再編成し、タスクは2つのノードで再び開始されます。これにより、各ノードがより多くのデータを処理するようになるため、新しい障害がさらに速く発生します。同時に、排出されたノードが再び参加し、タスクは2つのノードで再開されますが、他のノードでも再開されます。タスクを再起動する無限ループがあります。

並列休止はクラスターを壊すほど長くはありませんが、同じノードでのテストでははるかに悪くなります。 3つのノードで、彼はG1とJDK 11の束で30％を失いました。そして、より大きなクラスターでは、状況はさらに悪化します。

すべてのテストを見ると、ParallelがJDK 11よりもJDK 8で高速に実行されているのは驚くべきことです。しかし、これは非常に良い一致によるものです。これらのテスト実行では、すべてのノードで完全な一時停止が同期的に開始され、コレクターの作業が並列化されました。明らかに、この効果に依存することはできません。

Parallelを使用したテスト構成では、クラスターの安定性に対する長い休止の壊滅的な影響は観察されなかったという事実にもかかわらず、これは非常にありそうなシナリオです。他のテストでは、ヒープとデータの量を増やすか、同じヒープのままにして、操作マージンを減らしたとき、Parallelは破壊的に動作しました。しかし、それが失敗につながらなかった場合でも、グラフ上では、同じノードで作業しているとその利点が失われました。クラスターに新しいノードを追加すると状況が悪化することが予想されます。

一方、G1とJDK 11の束はかなり短い休止時間で動作したため、コンベヤーは停止しませんでした。短い停止時間を緩和するメカニズムがあり、一時停止が150ミリ秒を超えない間、ガベージコレクションはローカルにのみ影響を与えます。
