ReduxはSwiftUIの状態コンテナーのようなものです。基礎



今週は、Reduxで使用されるのと同様の状態コンテナーの作成について説明します。開発中のアプリケーションの唯一の価値の源です。アプリケーション全体の単一の状態により、デバッグと検証が容易になります。真理値の単一のソースにより、アプリケーションで複数の状態を作成するときに発生する数千のエラーが排除されます。

真理値の単一のソース


主なアイデアは、単一の構造または構造の構成を通じてアプリケーション全体の状態を記述することです。Githubリポジトリ検索アプリケーションの作成に取り組んでいるとしましょう。ここで、stateは、Github APIを使用して特定のリクエストに従って選択するリポジトリの配列です。

struct AppState {
    var searchResult: [Repo] = []
}

次のステップは、アプリケーション内の各ビューに状態（読み取り専用）を渡すことです。これを達成する最良の方法は、SwiftUIの環境を使用することです。アプリケーション全体の状態を含むオブジェクトをベースビューの環境に渡すこともできます。ベースビューは、すべての子ビューと環境を共有します。 SwiftUIの環境の詳細については、SwiftUIパブリケーションの環境の力をご覧ください。

final class Store: ObservableObject {
    @Published private(set) var state: AppState
}

上記の例では、アプリケーションの状態を保存し、読み取り専用アクセスを提供するストアオブジェクトを作成しました。Stateプロパティは、@ Publishedプロパティのラッパーを使用し、SwiftUIに変更を通知します。これにより、アプリケーション全体を絶えず更新し、真の値の単一のソースから派生させることができます。以前、以前の記事でストレージオブジェクトについて説明しましたが、これについて詳しくは、記事「SwiftUIでのストアオブジェクトを使用したアプリケーション状態のモデリング」をお読みください。

レデューサーとアクション


状態の変化につながるユーザーアクション（アクション）について話す時が来ました。アクションは、状態の変化を表す単純な列挙または列挙のコレクションです。たとえば、データサンプリング中に負荷値を設定したり、結果のリポジトリを状態プロパティに割り当てたりします。次に、Actionの列挙のサンプルコードを考えます。

enum AppAction {
    case search(query: String)
    case setSearchResult(repos: [Repo])
}

レデューサーは、現在の状態を取り、その状態にアクションを適用し、新しい状態を生成する関数です。通常、減速または減速の組成物は、状態が変化するアプリケーションで唯一の場所です。単一の関数でアプリケーションの状態全体を変更できるという事実により、コードは非常にシンプルになり、テストが容易になり、デバッグが容易になります。以下は、reduce関数の例です。

struct Reducer<State, Action> {
    let reduce: (inout State, Action) -> Void
}

let appReducer: Reducer<AppState, AppAction> = Reducer { state, action in
    switch action {
    case let .setSearchResults(repos):
        state.searchResult = repos
    }
}

一方向の流れ


次に、データフローについて説明します。各ビューには、ストアオブジェクトを介した状態への読み取り専用アクセスがあります。ビューは、リポジトリオブジェクトにアクションを送信できます。レデューサーが状態を変更すると、SwiftUIがすべてのビューに状態の変更を通知します。SwiftUIには非常に効率的な比較アルゴリズムがあるため、アプリケーション全体の状態の表示と変更されたビューの更新は非常に高速です。

状態->表示->アクション->状態->表示

このアーキテクチャは、単方向のデータストリームでのみ機能します。これは、アプリケーション内のすべてのデータが同じパターンに従うことを意味します。これにより、作成されたアプリケーションのロジックが予測可能になり、理解しやすくなります。アクションの送信をサポートするようにストアオブジェクトを変更してみましょう。

final class Store<State, Action>: ObservableObject {
    @Published private(set) var state: State

    private let appReducer: Reducer<State, Action>

    init(initialState: State, appReducer: @escaping Reducer<State, Action>) {
        self.state = initialState
        self.appReducer = appReducer
    }

    func send(_ action: Action) {
        appReducer.reduce(&state, action)
    }
}

副作用


ユーザーアクションを受け入れて状態を変更 する単方向ストリームを既に実装していますが、通常は副作用と呼ばれる非同期アクションはどうですか。使用するストレージタイプに非同期タスクサポートを追加するにはどうすればよいですか？非同期タスクの処理に理想的なCombine Frameworkの使用を紹介する時がきたと思います。

import Foundation
import Combine

protocol Effect {
    associatedtype Action
    func mapToAction() -> AnyPublisher<Action, Never>
}

enum SideEffect: Effect {
    case search(query: String)

    func mapToAction() -> AnyPublisher<Action, Never> {
        switch self {
        case let .search(query):
            return dependencies.githubService
                .searchPublisher(matching: query)
                .replaceError(with: [])
                .map { AppAction.setSearchResults(repos: $0) }
                .eraseToAnyPublisher()
        }
    }
}

Effectプロトコルを導入することにより 、非同期タスクのサポートを追加しました。エフェクトは、Combineフレームワークのパブリッシャータイプを使用してパブリッシュできるアクションシーケンスです。これにより、Combineを使用して非同期ジョブを処理し、reducerがアクションを現在の状態に適用するために使用するアクションを公開できます。

final class Store<State, Action>: ObservableObject {
    @Published private(set) var state: State

    private let appReducer: Reducer<State, Action>
    private var cancellables: Set<AnyCancellable> = []

    init(initialState: State, appReducer: Reducer<State, Action>) {
        self.state = initialState
        self.appReducer = appReducer
    }

    func send(_ action: Action) {
        appReducer.reduce(&state, action)
    }

    func send<E: Effect>(_ effect: E) where E.Action == Action {
        effect
            .mapToAction()
            .receive(on: DispatchQueue.main)
            .sink(receiveValue: send)
            .store(in: &cancellables)
    }
}

実例


最後に、Github APIを非同期で呼び出し、クエリに一致するリポジトリを選択するリポジトリ検索アプリケーションを完成させることができます。アプリケーションの完全なソースコードはGithubで入手できます。

struct SearchContainerView: View {
    @EnvironmentObject var store: Store<AppState, AppAction>
    @State private var query: String = "Swift"

    var body: some View {
        SearchView(
            query: $query,
            repos: store.state.searchResult,
            onCommit: fetch
        ).onAppear(perform: fetch)
    }

    private func fetch() {
        store.send(SideEffect.search(query: query))
    }
}

struct SearchView : View {
    @Binding var query: String
    let repos: [Repo]
    let onCommit: () -> Void

    var body: some View {
        NavigationView {
            List {
                TextField("Type something", text: $query, onCommit: onCommit)

                if repos.isEmpty {
                    Text("Loading...")
                } else {
                    ForEach(repos) { repo in
                        RepoRow(repo: repo)
                    }
                }
            }.navigationBarTitle(Text("Search"))
        }
    }
}

画面をコンテナビューとレンダリングビューの 2つのビューに分割します。コンテナビューはアクションを管理し、グローバルステートから必要なパーツを選択します。レンダリングビューはデータを受信して​​表示します。以前の記事ですでにコンテナービューについて説明しましたが、詳細については、リンク「SwiftUIでのコンテナービューの紹介」に従ってください。
