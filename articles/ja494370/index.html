Python開発者の目を通したクリーンなアーキテクチャ

こんにちは！私はEugeneと申します。Python開発者です。過去1年半にわたって、私たちのチームは従来のMVCモデルから離れて、クリーンアーキテクチャの原則を積極的に適用し始めました。そして今日、私はこれにどのように到達したか、それによって何が得られるか、および他のPLからのアプローチの直接転送が常に良い解決策ではない理由について話します。



Pythonは、私の主な開発ツールとして7年以上使用されています。彼らが私が彼について私が最も好きなものを尋ねたとき、私はこれが彼の優れた読みやすさであると返信します。最初の知り合いは、「集団的心のプログラミング」という本を読むことから始まりました。そこに記載されているアルゴリズムに興味がありましたが、すべての例は当時はまだ馴染みのない言語でした。これは通常ではありませんでした（Pythonはまだ機械学習の主流ではありませんでした）、リストはしばしば疑似コードまたは図を使用して書かれていました。しかし、言語の簡単な紹介の後、私はその簡潔さを高く評価しました：すべてが簡単で明確であり、不必要で気を散らすものは何もありませんでした。説明したプロセスの本質のみ。これの主なメリットは、言語の驚くべきデザイン、非常に直感的な構文糖質です。この表現は常にコミュニティで高く評価されています。「これをインポートする」とは何か。これは必ずテキストの最初のページに表示されます。これは目に見えない監督のように見え、常にあなたの行動を評価します。フォーラムでは、リストの変数名に何らかの方法でCamelCaseを使用することは初心者の価値があったため、すぐにPEP8を参照して提案されたコードのイディオムに向かって議論の角度が変わりました。
エレガンスの追求に加えて言語の強力なダイナミズムにより、本当に楽しいAPIを備えた多くのライブラリが作成されました。 

それにもかかわらず、Pythonは強力ですが、表現力豊かな自己文書化コードを記述できるツールにすぎませんが、これは保証されておらず、PEP8への準拠も保証されていません。Djangoにある一見シンプルなオンラインストアが収益を上げ始め、その結果、機能が向上するようになると、ある時点でそれがそれほど単純ではなく、基本的な変更を行う場合でもますます努力が必要になることに気づきます。そして最も重要なのは、この傾向が高まっていることです。何が起こり、すべてがうまくいかなかったのですか？

悪いコード


不良コードは、PEP8に準拠していないコードでも、循環的複雑度の要件を満たしていないコードでもありません。悪いコードはまず第一に、制御されていない依存関係であり、プログラムのある場所での変更が他の部分での予測できない変更につながるという事実につながります。コードを制御できなくなり、機能を拡張するにはプロジェクトの詳細な調査が必要です。そのようなコードはその柔軟性を失い、変更されたことには抵抗しますが、プログラム自体は「脆弱」になります。 

クリーンなアーキテクチャ


アプリケーションのアーキテクチャを選択することでこの問題を回避できます。これに遭遇したのは私たちが最初ではありません。Javaコミュニティでは、最適なアプリケーション設計の作成について長い間議論されてきました。

2000年に戻ると、ロバートマーティン（ボックルアンクルとも呼ばれます）は、彼の記事「Principles of Design and Design」で、記憶に残る頭字語SOLIDでOOPアプリケーションを設計するための5つの原則をまとめました。。これらの原則はコミュニティーから高く評価されており、Javaエコシステムをはるかに超えています。それにもかかわらず、それらは本質的に非常に抽象的なものです。その後、SOLIDの原則に基づいて一般的なアプリケーション設計を開発する試みがいくつかありました。これらには、「六角形のアーキテクチャ」、「ポートとアダプタ」、「大胆なアーキテクチャ」が含まれ、実装の詳細は異なりますが、すべてに共通点があります。そして2012年、同じロバートマーティンが記事を発表し、「クリーンアーキテクチャ」と呼ばれる彼自身のバージョンを提案しました。



ボブおじさんによると、アーキテクチャは主に「境界と障壁」であり、アプリケーションの制御を失わないように、ニーズを明確に理解し、ソフトウェアインターフェイスを制限する必要があります。これを行うには、プログラムをレイヤーに分割します。ある層から別の層に移ると、データのみを転送できます（単純な構造とDTOオブジェクトはデータとして機能できます）。これが境界のルールです。「アプリケーションは絶叫する必要がある」という最も頻繁に引用されているもう1つのフレーズは、アプリケーションの主なものは、使用されているフレームワークやデータストレージテクノロジーではなく、このアプリケーションが実際に実行すること、アプリケーションが実行する機能- アプリケーションのビジネスロジックであることを意味します。したがって、レイヤーは線形構造ではなく、階層を持っています。したがって、さらに2つのルールがあります。

  • 内層の優先ルール -外層と相互作用するためのインターフェースを決定するのは内層です。
  • 依存関係ルール -依存関係は、内側の層から外側の層に向ける必要があります。

最後のルールは、Pythonの世界では非常に異例です。複雑なビジネスロジックシナリオを適用するには、常に外部サービス（データベースなど）にアクセスする必要がありますが、この依存関係を回避するには、ビジネスロジックレイヤー自体が外部との対話に使用するインターフェイスを宣言する必要があります。この手法は「依存関係の逆転」（SOLIDの文字D）と呼ばれ、静的型付けの言語で広く使用されています。Robert Martinによれば、これはOOPに付属する主な利点です。

これらの3つのルールは、クリーンアーキテクチャの本質です。

  • 国境通過ルール。
  • 依存関係ルール。
  • 内層の優先ルール。

このアプローチの利点は次のとおりです。

  • テストのしやすさ -レイヤーはそれぞれ分離されており、サルをパッチすることなくテストできます。重要度に応じて、さまざまなレイヤーのコーティングを細かく設定できます。
  • すべてのビジネスルールが1か所に集められているため、ビジネスルールの変更が容易で、プロジェクト全体に広がることはなく、低レベルのコードと混在することもありません。
  • 外部エージェントからの独立性：特定の場合にビジネスロジックと外部の世界の間に抽象化が存在するため、内部レイヤーに影響を与えることなく外部ソースを変更できます。ビジネスロジックを外部エージェントの特定の機能（データベーストランザクションなど）に関連付けていない場合に機能します。

不幸なことに、ボブおじさんのすべての規範に従って細心の注意を払って説明したにもかかわらず、この例はかなり怖いです。 

プロジェクトAPIは1つのメソッドで構成されています-使用可能なフィルターを使用してリストを取得します。初心者の開発者であっても、このようなプロジェクトのコードは15行以内で済むと思います。しかし、この場合、彼は6つのパケットを取った。完全に成功しているわけではないレイアウトを参照することもできますが、これは事実ですが、いずれの場合も、このプロジェクトを参照してこのアプローチの有効性を誰かが説明することは困難です。

もっと深刻な問題があります。記事を読まずにすぐにプロジェクトの研究を始めると、理解が非常に難しくなります。ビジネスロジックの実装を検討します。

from rentomatic.response_objects import response_objects as res

class RoomListUseCase(object):
   def __init__(self, repo):
       self.repo = repo
   def execute(self, request_object):
       if not request_object:
           return res.ResponseFailure.build_from_invalid_request_object(
               request_object)
       try:
           rooms = self.repo.list(filters=request_object.filters)
           return res.ResponseSuccess(rooms)
       except Exception as exc:
           return res.ResponseFailure.build_system_error(
               "{}: {}".format(exc.__class__.__name__, "{}".format(exc)))

プロジェクトのビジネスロジックを実装するRoomListUseCaseクラス（ビジネスロジックとあまり似ていませんよね？）プロジェクトのrepoオブジェクトによって初期化されます。しかし、レポとは何ですか？もちろん、コンテキストから、repoがデータにアクセスするためのリポジトリテンプレートを実装していることを理解できます。RoomListUseCaseの本体を見ると、1つのリストメソッドが必要であることがわかります。その入力はフィルターのリストであり、出力では明確ではありません。調べる必要があります。 ResponseSuccess。また、シナリオがより複雑で、データソースへの複数のアクセスがある場合はどうでしょうか。リポジトリが何であるかを理解していることがわかりました、あなたは実装を参照することができるだけです。しかし、彼女はどこにいますか？これは、RoomListUseCaseに関連付けられていない別のモジュールにあります。したがって、何が起こっているのかを理解するには、上位レベル（フレームワークのレベル）に上がり、オブジェクトの作成時にクラスの入力に何が供給されるかを確認する必要があります。

私は動的型付けの欠点を挙げていると思うかもしれませんが、これは完全に真実ではありません。表現力豊かでコンパクトなコードを記述できるのは動的型付けです。マイクロサービスとの類似性が思い浮かびます。モノリスをマイクロサービスにカットすると、マイクロサービス（PL、フレームワーク、アーキテクチャ）内で何でも実行できるため、設計はさらに厳格になりますが、宣言されたインターフェイスに準拠する必要があります。ここで：プロジェクトをレイヤーに分割すると、レイヤー間の関係はコントラクトと一致している必要がありますが、レイヤー内ではコントラクトはオプションです。そうでなければ、あなたはあなたの頭の中でかなり大きなコンテキストを保つ必要があります。覚えておきますが、悪いコードの問題は依存関係であるため、明示的なインターフェイスがなければ、明示的な原因と結果の関係がないため、やり直したい場所に戻ります。

一般に、そのとき私は新しいプロジェクトでClean Architectureを放棄し、再びクラシックMVCを適用しました。しかし、コーンの次のバッチを満たした後、彼は1年後にこの考えに戻り、ついに、Python 3.5以降でサービスを開始しました。ご存知のように、彼は型注釈とデータクラスをもたらしました：2つの強力なインターフェース記述ツール。それらに基づいて、サービスのプロトタイプをスケッチしましたが、結果はすでにはるかに良好でした。特にフレームワークと統合する場合には、まだ多くのコードがあったにもかかわらず、レイヤーの分散が停止しました。しかし、小規模プロジェクトでこのアプローチを適用し始めるにはそれで十分でした。徐々に、型注釈の最大の使用に焦点を当てたフレームワークが登場し始めました：apistar（現在のstarlette）、moltenframework。 pydantic / FastAPIバンドルが一般的になり、そのようなフレームワークとの統合がはるかに簡単になりました。これは、上記のrestomatic / services.pyの例のようになります。

from typing import Optional, List
from pydantic import BaseModel

class Room(BaseModel):
   code: str
   size: int
   price: int
   latitude: float
   longitude: float

class RoomFilter(BaseModel):
   code: Optional[str] = None
   price_min: Optional[int] = None
   price_max: Optional[int] = None

class RoomStorage:
   def get_rooms(self, filters: RoomFilter) -> List[Room]: ...

class RoomListUseCase:
   def __init__(self, repo: RoomStorage):
       self.repo = repo
   def show_rooms(self, filters: RoomFilter) -> List[Room]:
       rooms = self.repo.get_rooms(filters=filters)
       return rooms

RoomListUseCase-プロジェクトのビジネスロジックを実装するクラス。 show_roomsメソッドが実行するのはRoomStorageの呼び出しだけであるという事実に注意を払うべきではありません（この例は思いつきませんでした）。実際には、割引計算、広告に基づくリストのランキングなども行われます。ただし、モジュールは自給自足です。このシナリオを別のプロジェクトで使用する場合は、RoomStorageを実装する必要があります。そして、これに必要なものは、モジュールからはっきりと見えます。前の例とは異なり、このようなレイヤーは自己完結型であり、変更するときに、コンテキスト全体を覚えておく必要はありません。非体系的な依存関係からは、ただpydanticなのですが、それはフレームワークのプラグインで明らかになります。依存関係なし、追加のコンテキストではなくコードの可読性を向上させるもう1つの方法です。初心者の開発者でも、このモジュールの機能を理解できます。

ビジネスロジックシナリオはクラスである必要はありません。以下は、関数形式の同様のシナリオの例です。

def rool_list_use_case(filters: RoomFilter, repo: RoomStorage) -> List[Room]:
   rooms = repo.get_rooms(filters=filters)
   return rooms


そしてここにフレームワークへの接続があります：

from typing import List
from fastapi import FastAPI, Depends
from rentomatic import services, adapters
app = FastAPI()

def get_use_case() -> services.RoomListUseCase:
   return services.RoomListUseCase(adapters.MemoryStorage())

@app.post("/rooms", response_model=List[services.Room])
def rooms(filters: services.RoomFilter, use_case=Depends(get_use_case)):
   return use_case.show_rooms(filters)

FastAPIは、get_use_case関数を使用して、依存性注入パターンを実装します。データのシリアル化について心配する必要はありません。すべての作業はpydanticと組み合わせてFastAPIによって行われます。残念ながら、データは常にレストランでのライブブロードキャストに適したビジネスロジックフォーマットではありません<逆に、ビジネスロジックは、データがどこから来たのかを知りません-スラッシュ、リクエストボディ、Cookieなど。この場合、room関数の本体には入力データと出力データの特定の変換がありますが、ほとんどの場合、APIを使用する場合、このような簡単なプロキシ関数で十分です。

正規のClean Architectureモデルが示唆するように、私は意図的にビジネスロジックのレイヤーを分離しませんでした。Roomクラスは、ドメインリージョンを表すEntityドメインリージョンレイヤーにあると想定されていましたが、この例ではこれは必要ありません。EntityレイヤーとUseCaseレイヤーを組み合わせると、プロジェクトはクリーンアーキテクチャの実装でなくなることはありません。ロバートマーティン自身も、レイヤーの数は上下に変動する可能性があると繰り返し述べています。同時に、プロジェクトはクリーンアーキテクチャの主な基準を満たしています。

  • 国境通過ルール：本質的にDTOであるpydanticモデルは国境を越えます。
  • 依存関係ルール：ビジネスロジックレイヤーは他のレイヤーから独立しています。
  • 内部層の優先ルール：ビジネスロジックが外部の世界とやり取りするためのインターフェース（RoomStorage）を定義するのは、ビジネスロジック層です。

今日、私たちのチームのいくつかのプロジェクトは、説明されたアプローチを使用して実装され、製品に取り組んでいます。最小のサービスでもこのように整理するようにしています。それはよく訓練されます-私が来る前に私が考えていなかった質問。たとえば、ここでのビジネスロジックとは何ですか？たとえば、ある種のプロキシを作成している場合、これは常に明白とはほど遠いです。別の重要なポイントは、異なる考え方を学ぶことです。。タスクを取得するとき、私たちは通常、フレームワーク、使用するサービスについて、このデータを保存する方がよいキューが必要かどうかについて考え始めます。クリーンアーキテクチャを指示するアプローチでは、最初にビジネスロジックを実装し、次にインフラストラクチャとの相互作用の実装に進む必要があります。ロバートマーティンによると、アーキテクチャの主なタスクは、インフラストラクチャ層は、アプリケーションの不可欠な部分になります。

一般に、PythonでClean Architectureを使用するのに好ましい見通しがあります。ただし、その形式は、おそらく他のPLで受け入れられる方法とは大きく異なります。過去数年間、私はコミュニティの建築のトピックへの関心が大幅に高まってきました。だから、最後のPyConでDDDの使用に関するいくつかの報告があった、とからみんなドライラボが個別に注意すべきです。私たちの会社では、多くのチームがすでにある程度説明されているアプローチを実装しています。私たちは皆同じことをしています。成長し、プロジェクトも成長しました。Pythonコミュニティはこれに取り組み、たとえば、かつてすべてのDjangoに共通したスタイルと言語を定義しなければなりません。
