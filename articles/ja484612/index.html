プロジェクトのビルド時間を最適化する

私が作業している場所（Spot.IMスタートアップでは、サイズは中小規模のどこかです）で、Webpackを使用してさまざまなプロジェクトを構築しています。メイン製品の4年間の作業の後、数え切れないほど多くの人々がコードに貢献したとき、その最初の組み立ての時間は法外な90秒に達し、再組み立て時間

は14になりました。クリックするたびに待つのに約14秒かかります「保存」ボタンに。

誰でもプロジェクトに適用できるなど、いくつかの単純な最適化に頼った結果、上記の数値を、アセンブリの場合は20秒、プロジェクトの再構築の場合は1秒に減らすことができました。



この記事では、いくつかの簡単な変更についてお話ししたいと思います。これにより、プロジェクトによってアセンブリ時間が大幅に改善されます。CreateReactApp（または他のファッショナブルなアプリケーションジェネレーター）を使用している場合、この記事は特に役立つとは限りません。ただし、そのようなものを使用しない場合は、ここで説明していることが非常に役立ちます。

プロジェクトの構築にかかる時間の測定


何らかの最適化を行う前に、作業の結果を判断できる指標の測定を設定しましょう。これを行うには、speed-measure-webpack-plugin（SMP）パッケージを使用します。

const webpackConfig = require('./webpack.config')
const SpeedMeasurePlugin = require('speed-measure-webpack-plugin')

const smp = new SpeedMeasurePlugin({
  disable: !process.env.MEASURE,
})

module.exports = smp.wrap(webpackConfig)

Webpack構成ファイルをSMPラッパーに配置し（環境変数を使用してパフォーマンス測定を行うメカニズムを開始）、その後Webpack構成オブジェクトを転送します。その後、ビルド中に各ブートローダーが完了するまでにかかる時間を知ることができる快適なレポートを自由に使用できます。SMPを使用すると、2つのメリットがあります。まず、いくつかの改善を行ったので、それがプロジェクトのビルド時間にどのように影響したかがすぐにわかります。第二に、各ブートローダー（または、より正確には、「ブートローダーチェーン」）にかかる時間に関する完全なレポートをすぐに自由に利用できます。

プロジェクトの初期ビルド時間の改善


SMPの使用を開始した後、アセンブリプロセスを改善すると、プロジェクトのアセンブリ時間がどのように変化するかについての情報が得られました。最初に最適化を始めたのは、初期ビルド時間（つまり、Webpackが初期化されてからパッケージをビルドするのにかかった時間）です。最初のビルドプロセスを高速化するために、ブートローダーを使用することにしましたcache-loader。

キャッシュローダーは、ローダー（ローダー）の作業結果をキャッシュしてディスク（またはデータベース）に保存するローダーです。これは、次回Webpackを起動したときに、ビルド速度が大幅に改善されていること、または少なくとも期待されていることを意味します。

次に例を示します。

{
  rules: [
    {
      test: /\.jsx?$/,
      use: [
        'cache-loader',
        'babel-loader',
      ],
    },
    {
      test: /\.scss$/,
      use: [
        'style-loader',
        'cache-loader',
        'css-loader',
        'postcss-loader',
        'sass-loader',
      ],
    },
  ]
}

cache-loaderbefore css-loader（CSSの場合）およびbefore babel-loader（JSの場合）を 追加すると、プロジェクトの初期アセンブリに費やされた時間から約75秒を削除できました。

それでは、再組み立て時間について見ていきましょう。プロパティを変更することで、この時間の改善に努めますdevtool。

Webpackコードカード


Webpack設定では、devtoolプロパティを見つけることができます。これは、ドキュメントによれば、「デバッグ機能の向上に使用されるコードマップの作成スタイルを選択できます。セットポイントは、組み立てと再組み立ての速度に大きく影響する可能性があります。」

つまり、プロパティdevtoolを変更すると、プロジェクトのビルド後に開発者が使用できるコードカードに影響します。そして、これは、そのようなコードカードの作成にかかる時間に依存します。

このプロパティの実験は、プログラマーの人生を永久に変える可能性がある分野からのものです。これは、プロジェクトのビルド速度に大きな影響を与えます。つまり、値devtoolをsource-map（おそらくこれが最も遅いモードです）からeval-source-map また、プロジェクトの再組み立て時間を14秒から3.5秒に短縮できました。

{
  devtool: process.env.NODE_ENV === 'development'
    ? 'eval-source-map'
    : 'source-map'
}

プロパティはdevtool、12の値オプションを受け入れることができます。たとえば、CreateReactAppはcheap-module-source-mapを使用します。したがって、このプロパティを構成する場合は、実験して最適な値を見つけてください。

コードカードを作成するための高速な方法を使用すると、結果のカードの品質が低下することに注意してください。これらの劣化は、デバッグを開始することで感じることができます。幸い、最新のブラウザーはTC39に対応しています。その結果（少なくとも開発中）、大量のコードのトランスパイレーションを実際に行う必要はありません。このツールがJavaScriptを最新のブラウザーバージョンが理解できるレベルにトランスポートするようにBabelを構成する場合（CRAで行われたように））、コードのデバッグを行うと、コードカードがコード自体とあまり変わらないため、すべて正常に動作するはずです。コードをさまざまなブラウザの最新バージョンに明確な状態に変換することが決定された場合

、ファイルは次のようになりbabel.config.jsます。

module.exports = {
  presets: [
    [
      '@babel/preset-env',
      {
        targets: [
          'last 1 chrome version',
          'last 1 safari version',
          'last 1 firefox version',
        ].join(', '),
      },
    ],
  ],
  // ...
}

それで全部です。簡単な3つのステップで、プロジェクトのビルド時間が大幅に短縮されました。

同様の問題を解決し始めた人は、最初にWebpackのドキュメントを見て、正しく読みたいと思うかもしれません。ただし、これが唯一の知識源ではありません。

プロジェクトの構築に関する貴重な情報を見つける別の方法を見つけました。このアプローチは実際に実証されています。webpack.config既存のオープンソースプロジェクトのファイルを分析することで構成されます。特に、CreateReactAppプロジェクトファイル。このファイルを徹底的に読むと、多くの役立つことがわかります。

概要


注意深い読者は、最初は再組み立て時間を1秒に短縮することが問題であることに気づくでしょう。そして、テキストで言及されているこの指標の最良の値は3.5秒でした。明らかに、組み立てプロセスの最適化の進捗状況を説明するとき、何かが省略されました。それがそうです。特定のプロジェクト機能、つまりSpot.IMに大きく依存する微妙な最適化により、さらに2.5秒を獲得することで、プロジェクトの再組み立て時間を1秒に改善することができました。したがって、これらの改善点についてはここでは説明しません。
