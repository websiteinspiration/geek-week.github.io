STM32MP1-カーネル+ Linux =完璧なマイクロコントローラー

私の活動の性質上、測定、制御、制御など、さまざまなデバイスの開発に従事しています。デバイスの大部分は2つの部分に分かれています。

  1. 高リソースのマイクロコントローラーであり、ハードリアルタイムグラフィカルユーザーインターフェイス（GUI）を必要としません。
  2. 少数のリソースを消費し、デバイスのハードリアルタイムハードウェアで動作します。

そして原則として、新しいシリーズのマイクロコントローラーへの移行は、GUIの改善によって決定されました。色の増加、解像度の向上、設計要素の複雑化です。かつて、そのアイデアは、ある種のRaspberryボードをグラフィカルインターフェイスとして使用することでした。

しかし、STMのSTM32MP1マイクロコントローラーの新製品の登場により、私の苦痛は終わり、それが今までのところ起こっていました。

完璧なコントローラー


私にとって理想的なコントローラーは、次の特性を備えている必要があります。

  • さまざまなグラフィック形式の単純なサポート、ベクターフォントのサポートなど、適切なグラフィック機能を備えている
  • グラフィック用の大容量メモリ
  • USBキーボードとマウスのサポート
  • イーサネットのサポート
  • 多数のハードウェアタイマー
  • PWM生成
  • 多数のGPIO
  • 16ビットADC

まともなグラフィカルユーザーインターフェイスでさまざまなデバイスを構築するために必要なのは、これらの要件です。

そして、STMの新しいシリーズSTM32MP1を見つけてとても嬉しかったです。

最初の会議


新しいチップに慣れるために、STM32MP157C-DK2ボードを購入しました。これは1つです（インターネットからの写真）。



デモボードはSTM32MP157CACチップ上に構築されています。

  • 2コアA7、各650 MHz
  • 1コアM4、208 MHz

記憶


目を引く最初の違いは、チップにフラッシュメモリがないことです。本質的にマイクロプロセッサーであるA7コアの場合、これは通常の状況であり、コードは常にDRAMから実行されます。M4ベースのマイクロコントローラには、通常、実行可能コードが格納されているフラッシュメモリが含まれています。

この水晶では、M4カーネルにはRAMのみがあり、コードはそこから（データシートから）実行されます。

708
K バイトの内部SRAM：256 K バイトのAXI SYSRAM + 384 KバイトのAHB SRAM +
64 K バイトのAHB SRAM（バックアップドメイン）
および4 K バイトのSRAM（バックアップドメイン）

A7コアにはDDRコントローラーが組み込まれています（データシートから）。
STM32MP157C / Fデバイスは、
以下のデバイスをサポートする外部SDRAMのコントローラーを組み込んでいます：

  • LPDDR2またはLPDDR3、16ビットまたは32ビットのデータ、最大1 Gバイト、最大533 MHzクロック。
  • DDR3またはDDR3L、16ビットまたは32ビットのデータ、最大1 Gバイト、最大533 MHzクロック。

M4カーネルのメモリは、フラッシュメモリのバージョンのメモリよりも少ないようですが、RAMメモリはコア周波数で動作します。フラッシュメモリはこれを行うことができず、そこで分周器を使用する必要があります。

職場組織


Linuxはボードで動作するために必要なので、追加のハードドライブをコンピューターにインストールし、Ubuntu 16.04を展開する必要がありました。このOSはSTMによって推奨されています。

エンジニアリングと生産モード


ボードは2つのモードのいずれかで起動できます（ディップスイッチによって決定されます）。エンジニアリングモードでは、M4コアは個別に動作します。本番モードでは、M4コアはA7コアの制御下で実行されます。私はエンジニアリングモードを使用せず、プロダクションモードでのみ作業しました。このモードでは、コアクロックの調整について心配する必要はありません。タイミングは、起動プロセス中に最大値で構成されます。デバッガーでいくつかのレジスターをチェックし、MCOの周波数の一部を推定して、オシロスコープを見ました。A7は650 MHzの周波数で動作し、M4は208 MHzの周波数で動作します。

カーネルM4。プログラムをダウンロードして実行する


M4コアはA7を実行しているため、OpenSTLinuxを実行することを意味します。Linux remoteprocフレームワーク（RPROC）は、M4コアの制御に使用されます。このフレームワークでは、カーネルの有効化と無効化、ソフトウェア（ELFファイル）のダウンロードとデバッグを行うことができます。

プログラムをM4カーネルにロードするには、次のコマンドを使用します

。echo -n <firmware_name.elf >> / sys / class / remoteproc / remoteproc0 / firmware

プログラムを開始するには：

echo start> / sys / class / remoteproc / remoteproc0 / state

停止するには：

echo停止> / sys /クラス/ remoteproc / remoteprocX /状態

カーネルM4。SW4STM32


上記のコマンドを試してみましたが、すべて正常に機能します。STMでは、AC6 SW4STM32ソフトウェアを使用することをお勧めします。同時に、OpenSTM32 Webサイトには非常にすばらしいタイトルがあります。
LinuxのSystem Workbenchを使用すると、STのMPUのSTM32MP1ファミリーのEmbedded Linuxは、Linuxの世界の初心者であっても、構築と保守がそれほど簡単ではありませんでした。
また、System Workbench for LinuxをSystem Workbench for STM32にインストールすると、部分的にLinuxで、部分的にCortex-M4で実行される非対称アプリケーションをシームレスに開発およびデバッグできます。
つまり、m4コアとA7の両方のコードを記述できます。しかし、これは有料版です。また、無料でダウンロードできるM1コアのコードのみを記述してデバッグできます。

IDE SW4STM32がダウンロードされ、インストールされ、試されました。すべてが機能し、書き込みとコンパイルのプロセスはWindowsのバージョンと同じです。
EthenetとSSH接続がソフトウェアの記述に使用されるため、唯一の違いはコードのロードとデバッグです。ポイント単位でのデバッグや停止には、内蔵のST LINKボードを使用しています。

コア間データ交換


ほとんどの時間は、コア間のデータ交換のメカニズムの理解に費やされました。STMが提供する例では、そのような交換を実装するためのいくつかのオプションがあります。OpenSTLinuxのドライバーを直接操作するか、仮想UARTメカニズムを使用できます。仮想ポートを介した共有をより簡単にすることにしました。

すべては非常に単純ですが、微妙なニュアンスがあります。ttyRPMSG0デバイスを表示するには、この形式でA7コアにアクセスするためにM4コアが必要です。

これらはM4コアコードの行です。
