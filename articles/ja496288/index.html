ターゲット言語の1つとしてKotlinを選択した理由。パート1：回顧

DomKlik社は2015年から存在しており、この間に、主要なサービスを開発するためのターゲット言語をJavaからKotlinに変更することができました。

私の名前はAlexander Kotsyurubaです。DomKlikで内部サービスの開発を率いています。Kotlinに切り替えた理由の質問に答えて、2つの資料を公開します。

  1. 最初の部分は、言語の出現の回顧に捧げられます（実際、あなたはこれを読みます）
  2. 2番目の部分では、実用的な例に焦点を当てます（これは後でリリースされます）。

それでは始めましょう。



2010年：


  • 8月には惑星のパレード
  • 12月初旬、チューリッヒは2018 FIFAワールドカップの開催国としてロシアの発表を主催しました
  • JetBrainsは12月末にIntellij IDEA 10.0をリリースします

JetBrainsにとって、IDEが主な製品であり、IT企業として、製品の迅速な開発と品質に関心を持っています。 Intellij IDEAはJVM 6プラットフォームで開発されていますが、一方で仮想マシンは負荷に完全に対応し、カラシニコフ突撃銃のように信頼性が高く、印象的なコミュニティに代表されています。一方、Javaの速度は低下し始めています。これは、とりわけ、Sun Microsystemsの財務上の問題が原因です。Javaのリリース頻度から見ると、Sun Microsystemsは言語にあまり注意を払わないようになっています。

JavaおよびIntellij IDEAリリースのタイムライン：


ご覧のとおり、JetBrainsはIntellij IDEAの6つのメジャーバージョンをリリースすることに成功しました。 Javaが進歩したのは1つだけです。さらに、Java 5のように、Java 6は大きな進歩ではありませんでした。

2010年の時点では、Javaはすでに成熟しており（〜20歳、1990年に開発が開始されました）、50,000クラスを超える「コケに覆われた」状態になっています。アンドレイ・ブレスラフによればアブレスラフJavaのラムダ開発に外部の専門家として参加し、JVMを中断することなくそのような機能を追加することは非常に困難でした（Java 8での機能のリリースを延期する理由の1つ）。

これにより、JBは他の言語で製品を開発するようになりました。同社は当時人気のあったGroovyとScala言語の開発能力を持っていました。Groovyのそれがあるため、動的型付け言語の欠点を放棄しなければなりませんでした。



  1. 実行時の言語パフォーマンスが遅い
  2. リファクタリングの高コスト
  3. 本格的なIDEの欠如

Scalaは JetBrainsの内部開発にも適していませんでした。

  1. 完全な言語IDEの欠如
  2. 入場閾値が高い
  3. 開発者の高コスト
  4. コンパイラのパフォーマンス

Scalaが1つの言語で異なる概念を組み合わせるという点で成功したプロジェクトだったことは注目に値します。

同時に、JetBrainsは、特定の開発環境を開発するときに、対応する言語のフロントエンドコンパイラを作成します。これには、セマンティックツリーの解析、強調表示などが含まれます。つまり、コンパイラの作業の半分を説明します。その時点でIDEが7つあるため、JetBrainsは3.5コンパイラを開発したと言えます。



明らかに、簡潔なコードはビジネスロジックに集中するのに役立ちます。これにより、開発速度が向上し、エラーのリスクが軽減されます。 Java 6はこれに貢献せず、言語とプラットフォーム自体の寿命は理解できませんでした（SUNおよびOracle Corpでの噂の噂を参照）。これにより、同社は社内のニーズに合わせた言語を開発するようになりました。

主な言語要件は次のとおりです。

  1. Javaとの互換性（会社はすでにJavaで多くのコードを持っているため）
  2. 簡潔
  3. 速い
  4. タイプセーフ

したがって、Kotlinというプロジェクトが誕生しました。

2010〜2016：




2012：


このときまでに、MVP Kotlinが現れます。

JBはKotlinをオープンソースの状態に置き、IDEの内部開発に積極的に使用し始めます。内部での使用経験の結果に基づいて、製品の最初のリリースをリリースするかどうかを決定する予定です。

2016：


Kotlin 1.0がリリースされ、JVM 6とAndroidがサポートされました。現時点では、DomKlikはKotlinも試験運用し、それに書かれた安全な決済サービスをリリースします。これは、買い手と売り手の間で現金以外の方法で物件を決済する方法です。開発者は、言語の簡潔さ、機能、およびアイデアに満足し、「許可された開発言語」のリストにKotlinを含めることを決定しました。

2017-私たちの日：




2017年3月：


Kotlin 1.1がリリースされました。重要な機能の中で、注目に値します：

  1. JVM 8のサポート
  2. コルーチン（実験版）
  3. JavaScriptコンパイラ

JSでのコンパイルについて少し説明します。フロントエンドコンパイラの開発経験と開発ニーズを考慮して、Kotlinはほとんど最初からマルチプラットフォームをサポートするタスクを設定しました（Kotlin MPP、記事の残りの部分で詳しく説明します）。

2017年11月：


Kotlin 1.2がリリースされました。リリースに含まれるもの：

  1. マルチプラットフォームのサポート
  2. ネイティブコードへのコンパイラ（試験版）

マルチプラットフォームのサポートとは、特定のプラットフォームに固有の、またはプラットフォームに共通のプロジェクトにモジュールを接続する機能を意味します。



同時に、GoogleはAndroid Studio 3.0にデフォルトのKotlin言語サポートを含めています。私の意見では、これはKotlinコミュニティにとって重要なイベントです。実際、この言語は、巨大なGoogleに直面してAndroidコミュニティによって認識されています。

2018：


10月に、安定した下位互換性のあるコルーチンとネイティブコードのコンパイラを含むリリース1.3がリリースされます。

同じ年、DomKlikの建築委員会で、対象の開発言語をJavaからKotlinに変更することが決定されました。

2019：


2019年には、Kotlinコミュニティにとって重要なイベントが開催されます。すべて同じように、会議でのGoogleは、KotlinをAndroidアプリケーション開発の標準として設定しています。


2019年の春に、JavaをKotlinに変更する転換点に到達します。私たちのグラフは、KotlinがJava機能よりも優れていることを示しています。つまり、JavaよりもKotlinの方が書き始め、事実上、コアコンピタンスがKotlinに変わりました。

私たちの日々：


今日、データサイエンスのためのKotlinの存在に注目する価値があります。このトピックを個別に強調し、次の記事でより深く掘り下げて利点を示したいと思います。
私たちの会社では、サーバー側では、KotlinがJavaと比較して3倍以上も優位を占めています。

一般に、当社にはさまざまなプログラミング言語があります。


Kotlinコンパイラ：




Kotlin MPPを使用すると、会社の開発におけるデフォルト言語としてKotlinを使用できます。DomKlikの私たちは、Kotlin / JVM（サーバー側+ Android）+ Kotlin / Native（iOS）プラットフォーム用のKotlin MPPで住宅ローン計算機を開発するパイロットを成功させました。今後の計画は、パイロットをKotlin / JSに拡大することです。
