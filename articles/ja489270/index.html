WebAuthnとは何か、なぜそれが必要なのか


現代の世界では、ほとんどのインターネットサービスがユーザー認証にパスワードを使用しています。パスワードの利点は、その単純さです。必要なリソースにアクセスするには、パスワードを知って、正しく入力するだけです。

ただし、パスワードの短所は、長所を上回ることがよくあります。

  • パスワードが単純な場合-パスワードを取得できます。
  • 同じパスワードが複数のサービスで使用されている場合、そのうちの1つを破ると、攻撃者は他のすべてのサービスにアクセスできます。
  • パスワードはフィッシング攻撃に対して脆弱です。
  • 強力で一意のパスワードは覚えるのが難しいため、ユーザーはステッカーにパスワードを書き留め、それをモニターに貼り付けなければなりません。そこから、簡単にスパイ、盗難、または紛失する可能性があります。

パスワードの欠点を取り除き、認証プロセスをより安全にするために、パスワードを置き換えるさまざまな方法が表示されます。これは、SMSまたはPUSH通知を介してユーザーに送信されるワンタイムコードの使用、またはユーザーがログインできる特別なTOTPコードジェネレーターアプリケーションの使用です。パスワードを入力せずにアカウント。Microsoft、Yahoo、Amazon

などの企業は、パスワードなしの認証方法を使用することを考えており、サービスでのパスワードの使用を完全に中止しています。 Mail.ruメールも例外ではありません、1回限りのコードを使用したログインを長い間サポートしてきました。これにより、ユーザーは複雑で強力なパスワードを覚えずに、自分のアカウントにすばやくアクセスできます。

パスワードのもう1つの代替手段は、電子キー（セキュリティキー、別名-電子認証システム）を使用した認証です。その動作の原理は、非対称暗号の使用に基づいており、FIDO（Fast IDentity Online）アライアンスによって開発されたFIDO2プロトコルファミリで説明されています。

2019年3月、W3C は標準の最初のバージョンをリリースしました。は、電子キーを操作できるブラウザベースのJS APIについて説明しています。標準は、推奨ステータスと名前「Web認証：公開鍵を使用して資格情報にアクセスするためのAPI：レベル1（Web認証：公開鍵資格情報レベル1にアクセスするためのAPI）-WebAuthnと略称」を受け取りました。

WebAuthnの仕組み


WebAuthnを使用した認証プロセスで以下の主要なアクターが識別されます。

  • クライアント（WebAuthnクライアント）-WebAuthn APIをサポートするブラウザ。
  • Webアプリケーション-WebAuthn APIを使用して資格情報と対話するクライアント上で実行されるアプリケーション。
  • 資格情報（公開キー資格情報）-ユーザーアカウントに関連付けられた公開および秘密暗号キーのペア。
  • オーセンティケーター（デバイスまたはプログラム）は、ユーザー資格情報を作成し、これらの資格情報を使用して証明書利用者からの要求に署名します（別の名前は電子キーです）。
  • 証明書利用者（WebAuthn証明書利用者）-Webサーバー-ユーザーアカウントに関連付けられた公開鍵を保存し、認証システムに保存された秘密鍵を使用して要求の署名の有効性を検証します。

WebAuthn APIでは、2つの操作のみを実行できます。これにより、新しい資格情報を作成し、既に作成された資格情報を使用してサーバーからの要求に署名できます。

したがって、WebAuthnの作業は電子キーの使用に基づいています。それは何ですか？

電子キー


電子キー（「WebAuthnキー」、セキュリティセキュリティキーとも呼ばれます）は、FIDO2対話プロトコルを実装するデバイスまたはプログラムです。

WebAuthn仕様および世帯レベルでは、すべての電子キーは次の2つのカテゴリのいずれかに分類されます。

  • プラットフォームに依存しないキー（ローミング認証システム）;
  • プラットフォーム認証

プラットフォームに依存しないキーは、YubicoのYubikeyやGoogleのTitan Security Keyなどの外部の物理デバイスです。通常、このような認証システムは、USB、NFC、またはBLEを介してユーザーのコンピューターまたはスマートフォンに接続されます。このようなデバイスとの通信は、特別なCTAPプロトコル（Client to Authenticator Protocol）を使用して行われます。

物理的な電子キーの使用を開始するには、ユーザーは自分のアカウントに一度バインドする必要があります。その後、ユーザーはこの電子キーを使用して、他のデバイスやブラウザーに入力する機会を得ます。theverge.com


から取得した、プラットフォームに依存しないさまざまなキーの例。

最初のタイプのキーの操作メカニズムが多かれ少なかれ明確である場合は、2番目のカテゴリについて詳しく説明します。

2番目のケースでは、公開および秘密暗号化キーが生成され、外部の物理デバイス内ではなく、コンピューターまたはスマートフォン内のいくつかのソフトウェアモジュールで保存されます。このソフトウェアモジュールは、特定のアプリケーションレベルまたはオペレーティングシステムレベルで実装できます。たとえば、コンピューター内の安全なチップでは、OSは、ユーザーがログインしていて、本人であることを証明した場合にのみアクセスを許可します。

さまざまなブラウザーでの最新の実装では、OSはユーザーに指紋で、またはユーザーアカウントのパスワードを入力して自分自身を確認することを要求します。このようなキーを使用するには、ユーザーが指紋スキャナーに指を置く必要がありますが、指紋自体はどこにも保存されず、どこにも送信されないことを明確にすることが重要です。これは、オペレーティングシステムまたはブラウザがユーザーを検証する方法にすぎません。


プラットフォーム固有の認証システムの使用。

エンコードされた公開鍵情報または署名されたランダムトークンのみがWebAuthn APIからWebサービスに返され、サーバー側で保存およびチェックされます。

したがって、組み込みの認証システムの場合は、アカウントに関連付けられているブラウザーとデバイスでのみ使用できます。つまり、プラットフォームに依存するキーは、アカウントにログインする予定のデバイス/ブラウザごとに個別に登録する必要があります。

将来的には、たとえばフェイススキャンを使用したり、デバイスからPINコードを入力したりするなど、ユーザーがプラットフォーム固有の認証システムの使用をアクティブ化できるようにする方法がさらに期待されています。

資格情報を作成するとき、または署名を計算するときに、2つの値を取る特別なパラメーターで認証タイプの優先タイプを指定できます- cross-platformとplatform。この場合、ユーザーはいずれかのタイプの電子キーのみを使用するように求められます。

const credential = await navigator.credentials.create({
    publicKey: {
        authenticatorSelection: {
            authenticatorAttachment: 'cross-platform',
        },
        ...,
    },
});

WebAuthnサポート


ドングル認証がデバイスで機能するかどうかは、いくつかの異なる要因に依存します。ブラウザーがWebAuthn APIをサポートするかどうかから始まり、コンピューターにあるコネクターと、オーセンティケーターがサポートする通信方式で終わります。 WebAuthnのパフォーマンスは、使用するデバイスとOSにも大きく依存します。

執筆時点では、caniuse.comによると、WebAuthn APIは80.5％のユーザーによってサポートされています。 Mail.ruのユーザーに関する統計によると、この数字は同じ順序-79.8％です。ただし、これらすべてのブラウザーでWebAuthnを使用するには、外部の電子キーが必要です。

WebAuthn APIをサポートするすべてのブラウザーがプラットフォーム依存のキーで動作できるわけではありません（便宜上、これらのキーを後で「フィンガープリント」と呼びます）。さらに、そのようなキーを操作するには、それらのキーを操作できるブラウザをインストールするだけでは不十分です。デバイスとOSに適切なモジュール/センサーがあり、それとやり取りできることも必要です。すべてのMail.ruユーザーのうち、9.0％のみがプラットフォーム固有のキーを追加できます。

マックOS


Microsoft Edge、Google Chrome、Opera、およびMozilla Firefoxは、macOSでの外部キーの操作をサポートしています。Google Chromeはフィンガープリントもサポートしているため、Touch IDでWebAuthnを使用できます。


Edge、Opera、ChromeでWebAuthnと対話するためのインターフェースが同じである場合、Firefoxはここで際立っています。 Firefoxのかわいらしいポップアップの代わりに、WebAuthnを使用すると、画面の隅に小さな通知が表示されます。誤ってページをクリックした場合、ページは折りたたまれ、ユーザーは現在何が起こっているのか途方に暮れます。



ただし、SafariはまだWebAuthnをサポートしていません。 WebAuthnのサポートはmacOS 10.15 Catalinaで利用可能になるSafari 13で発表されました。ただし、執筆時点で、私のチェックでは、SafariのWebAuthn APIは存在しますが、非常に不安定で、すべての認証システムで機能するわけではないことを示しました。モバイル版と同様に、Safariは組み込みの電子キーでは機能しません。また、外部の電子キーはまだサポートされていません。

そのため、サポートの問題に加えて、WebAuthnのUIにはさらに大きな違いがあることがわかりました。これらの違いは、電子キーを介して入口を使用するためにユーザーに何が必要かをユーザーにさらに詳しく説明する必要があるという事実につながります。さらに、新しいブラウザがリリースされるたびに、これらのポップアップが変更される可能性があり、今日のWebAuthnを使用するプロセスは、昨日とは大きく異なる場合があります。

これがなぜ起こっているのかは明らかです。WebAuthnテクノロジーはまだかなり若いので、ブラウザー開発者はまださまざまなタイプの実装を実験しています。ブラウザーでのWebAuthnのサポートが時間の経過とともに安定し、制限なくそれを使用できることを期待することができます。

オーセンティケーターの登録


アカウントにさまざまな電子キーを登録する機会をユーザーに提供する場合、ユーザーは自分のリストを表示し、古いまたは不要なものをリストから削除する機会を持つ必要があります。これは、たとえば、キーの1つが侵害された場合に役立ちます。

WebAuthn APIは、資格情報を作成および使用するときに、使用される認証子のタイプ、タイプ、および名前に関する情報をクライアントが使用できないように設計されています。したがって、1つのキーを別のキーと区別できるデータはありません。リスト内のすべてのキーは、機能を区別せずに等しく表示されます。質問：何をすべきか？


資格情報を作成するときに、いわゆる資格情報を要求すると、使用されるオーセンティケーターに関するいくつかの情報を取得できます。認証。認証は、オーセンティケーターがサーバーに対する信頼性を証明するための方法です。場合によっては、キーの製造元に関する情報を認定から取得できます。しかし、一般的なケースでは、使用されるデータは、アカウントに関連付けられた1つの電子キーを別の電子キーと明確に区​​別するにはまだ十分ではありません。

const credential = await navigator.credentials.create({
    publicKey: {
        attestation: 'direct',

したがって、Mailのアカウントにキーを接続した後、新しく作成したレコードに名前を割り当てる機会をユーザーに提供します。さらに、ユーザーはこの名前によって、あるキーを別のキーと区別できます。

クライアントとサーバーが利用できる情報に認証システムのタイプに関するデータが含まれていないという事実は、WebAuthnの別の不愉快な機能につながります。

スマートフォンなどで、アカウントに指紋を1つだけ追加したユーザーを想像してみてください。 WebAuthn APIを使用してログインする場合は、関数呼び出しを渡しますnavigator.credentials.getアカウントに関連付けられているすべてのキーのリスト。ただし、ブラウザは、このリストから、デバイスに存在する認証子と存在しない認証子を判別できません。したがって、彼は常にユーザーにWebAuthnの使用を提供することを余儀なくされています。

したがって、指紋による認証をサポートしていないコンピューターのアカウントにログインしようとした場合でも、ユーザーはWebAuthnを使用するように求められます。

このような状況で正しい動作を実装するには、WebAuthn標準自体を改良する必要があります。たとえば、クロスプラットフォームキーまたはフィンガープリントが作成に使用されたかどうかに関する情報をエンコードし、使用できないことが事前にわかっている場合にWebAuthnユーザーに提供しない。

状況によっては、この動作を回避する方法があります。たとえば、ユーザーが指紋と物理キーを個別にのみ登録できるようにします。また、キーを使用する場合は、明らかにそれらをサポートしていないデバイスで指紋をフィルタリングします。しかし、この方法では問題が完全に解決されるわけではありません。そして、この問題を解決する信頼できる方法はありません。

ユーザーはこの動作についてまだ苦情を受けていないため、現在この機能を調査しており、今後何をするかを決定しています。

WebAuthnはさまざまなサブドメインで機能します


前に述べたように、WebAuthnはフィッシングに対する箱からの保護を提供します。電子キーを登録すると、originこのキーが登録された情報が保存されます。 WebAuthnでは、この電子キーを別ので使用することはできませんorigin。

Mail.ruなどの大規模なWebサービスでは、作業に複数の異なるドメインを使用することがよくあります。たとえばe.mail.ru、メールcloud.mail.ru用とクラウド用のドメインがあります。そして、それらのそれぞれに、私たちは共通の認可形式を持っています。この場合、標準のWebAuthn設定では不十分です。origin一方originで他方に登録されているキーを使用できるようにするには、これらの2つのドメインに共通のサフィックス（最初のレベル以上の共通のサブドメイン）が必要です。

たとえばドメインでe.mail.ruとcloud.mail.ru一般的なサフィックスはmail.ruです。

この場合、登録時と電子キーの使用時に、リクエストオプションでrpId同等のパラメーターを指定できます。mail.ru次に、https://mail.ruサブドメイン自体とそのキーのいずれにも同じキーを使用できます。

const credential = await navigator.credentials.create({
    publicKey: {
        rp: {
            name: 'Mail.ru Group',
            id: 'mail.ru',
        },
        ...,
    },
});

WebAuthnはiframeで動作します


セキュリティ上の理由から、クロスオリジンのiframe内からのWebAuthnメソッドの呼び出しは禁止されています。私たちのプロジェクトは単一の承認フォームを使用します。これはhttps://account.mail.ru/loginにあり、メールやクラウドなどの他のプロジェクトで承認フォームを表示したい場合、iframeが実際にページに追加されます内部でこのURLが開きます。このソリューションにより、フォームを使用するすべてのプロジェクトでフォーム自体を同時に更新できるようになり、統計の収集が簡素化され、ユーザーのUXも改善され、ユーザーは元と同じサービスを継続できるようになります。



今回のケースでは、iframe内からのWebAuthnメソッドへの呼び出しに対する制限により、認証フォームを表示する任意の場所にWebAuthnを介してログインする機会を与えたかったため、iframeを回避する方法を模索しました。

何をしましたか。すべてのサービスで承認フォームを開くには、小さなライブラリを使用します。これは基本的に、正しいURLでページにiframeを作成するだけで、コンテンツをロードした後、ページにiframeを表示します。このライブラリは、iframe postMessageとの通信をサポートし、それを使用して、たとえば、コンテンツのサイズを変更するときにiframeのサイズを変更します。

私たちは以下のメカニズムを考え出し、実装しました：

  1. WebAuthnを使用する際の承認アプリケーションでは、iframe内で開いているかどうかを判断します。
  2. iframe内で開いている場合は、ブラウザーAPI WebAuthnを呼び出す代わりに、呼び出しパラメーターをシリアル化して、postMessage親ウィンドウから送信します。
  3. 親ウィンドウで、これらのパラメーターを逆シリアル化し、WebAuthnメソッドを呼び出します。
  4. 回答を受け取ったら、同じ方法でシリアル化し、postMessageiframe の内部を介して送信します。そこで、この回答を受け入れて、さらに処理を実行します。

したがって、この禁止を回避し、会社のすべてのサービスで認証に同じキーを使用する可能性を認識しました。

WebAuthnテストの自動化


私たちのチームでは、すべての新しいプロジェクトと機能について、セレンのようなソリューションとWebDriverプロトコルを使用して、統合UI自動テストを常に作成しています。したがって、WebAuthnの開発中に、自動テストUIをどのように記述するかという問題が生じました。

このような自動テストを作成する際の難しさは、WebDriverプロトコルにWebAuthnとの対話を自動化するメソッドがまだないことです。また、標準自体では、まだ自動化WebAuthn APIのテストはサポートされていません（ただし、このトピックには問題があります）。この自動化の編成方法に関する最初のアイデアは、Web認証のドラフトに記載されています。公開鍵資格レベル2にアクセスするためのAPIであり、他の場所でサポートされていることは言うまでもありません。したがって、私は代替ソリューションを考え出す必要がありました。

なぜなら 自動テストでWebAuthn関数のネイティブ実装を操作することはできません（ブラウザーを制御するためのメソッドはありません）。その場合、以下を実行する必要がありました。

自動テストのどこかで、WebAuthnを使用する前に、ブラウザーのグローバルホストオブジェクトにパッチを適用し、ネイティブ関数を実装に置き換えます。ここでは、ネイティブ関数が呼び出された引数を変数に保存し、promiseを返します。

//    WebAuthn   
navigator.credentials.create = (options) => {
    window.credentialsCreateArgs = options;

    return new Promise((resolve, reject) => {
        window.credentialsCreateSuccess = resolve;
        window.credentialsCreateFail = reject;
    });
};

次に、関数の結果を何らかの形で取得して、テストでWebAuthnの作業をエミュレートする必要があります。常に一定のハードコードされた回答を返すことができます。

// -    
const credentialsCreateResponse = { /* constant object */ };

window.credentialsCreateSuccess(credentialsCreateResponse);

この場合、サーバーにこの回答を受け入れて検証しないように教える必要がありますが、自動的に正しいと見なす必要があります。

そのようなテストの欠点は何ですか？この場合、次のことはできません。

  • クライアントがパラメーターを正しく形成してWebAuthn関数にロールするかどうかを確認します。
  • バックエンドリリースをローリングするときに、バックエンドの変更が正しいことを確認します。

したがって、このようなテストは十分に信頼できず、これは私たちには適していません。

私たちはより困難な方法を採用することを決定し、その結果、Node.jsにFIDO2プロトコルとアルゴリズムの独自の実装を記述しました。その助けを借りて、これらの関数をネイティブ実装にできるだけ近づけてロックしました。

つまり、リクエストパラメータに基づいて、ロックされたWebAuthn関数の応答を計算し、サーバーがそれを完全に正しいと見なす関数を記述しました。

// -    
const credentialsCreateResponse =
    calcCredentialsCreateResponse(window.credentialsCreateArgs);
 
window.credentialsCreateSuccess(credentialsCreateResponse);

その結果、自動テストの操作スキームは次のようになります。

  1. WebAuthnメソッドが呼び出された引数を取得します。
  2. 実際の認証システムの内部と同じアルゴリズムを実装する関数を介してこれらの引数を実行します。
  3. 結果を取得し、置き換えられた関数から返します。
  4. 私たちのサービスの残りのコードはすべて、通常モードでこれらの応答を処理します。その結果、サービスのすべての動作は、実際のユーザーの動作と変わりません。

そのような自動テストのソースと最も隠された認証システムの実装はgithubリポジトリにあります。後で作業を開始して調査できます。オーセンティケーターを作成するときは、w3.orgの仕様とNode.jsのドキュメントのみがガイドでした。

電子キーを紛失した場合はどうなりますか？


電子キーを使用する場合は、アパートや車のキーと同じように扱う必要があります。アパートの鍵を紛失した場合、通常は鍵を交換して新しい鍵を取得します。同じことが電子キーにも当てはまります。紛失した場合は、リンクされているすべてのアカウントからすぐに削除し、すべてのアカウントに新しい認証システムをアタッチする必要があります。

メインの電子キーを紛失した場合、いくつかの追加の認証方法が必要です。たとえば、コードを生成するアプリケーションを使用したり、特別に保護された場所（金庫や銀行のセルなど）に保存できるスペアキーを使用したりします。

他の誰かが私の電子キーにアクセスした場合はどうすればよいですか？


2要素認証がアカウントに含まれていて、電子キーが要素の1つにすぎない場合、この場合、アカウントはハッキングから保護されます。攻撃者が2番目の要素にアクセスできない限り。

電子キーがアカウントを入力するのに十分な唯一の要素である場合、この場合でも、攻撃者はこの電子キーが登録されているアカウントの名前を知っている必要があります。この情報は電子キーの内部には保存されないため、偶然に紛失した可能性の高い電子キーは、それを発見した通行人にとって完全に役に立たなくなります。

ただし、パスワードだけでなく、ログインの代わりに電子キーを使用できる方式もあります。このシナリオでは、サーバーは、資格情報を要求するときにのみ資格情報を提供します。origin、および資格情報自体は完全にオーセンティケータに格納されます。この場合、紛失した電子キーはすでにアカウントに入力するために簡単に使用できるため、認証システムについてはより注意する必要があります。そのようなシナリオでの保護のために、PINコードを入力する必要があるアクティベーション用の追加のセキュリティ手段（たとえば、キー）を備えた電子キーを使用できます。

いずれの場合でも、キーの紛失に気づいたらすぐに、キーを使用して入力できるすべてのサービスのすべてのアカウントからキーを解放する必要があります。そのため、すべてのサービスは、バインドされた認証システムのリストを管理する機能を提供する必要があります。損失に早く気づくほど、攻撃者がデータにアクセスするまでの時間が短くなります。

これで私の生体認証データはMail.ru/Google/Microsoftサーバーに保存されますか？


WebAuthnが組み込みの認証システムで動作する場合（Mac OSのTouch IDを使用するなど）、対応するセンサーとオペレーティングシステムのみが生体認証データにアクセスできます。Webサービスは生体認証情報を受信または処理せず、公開暗号鍵と秘密鍵で署名されたデータでのみ機能します。

サーバー自体には、公開鍵に関する情報のみが保存されます。したがって、WebAuthnは、生体認証データを使用して組み込みの認証システムを操作することはありません。
