マイクロサービスアーキテクチャのオンライン分析：Postgres FDWのヘルプと提案̶ポストポストPostп̶р̶о̶с̶т̶и̶тиь̶

マイクロサービスアーキテクチャは、この世界のすべてのものと同様に、長所と短所があります。一部のプロセスはより簡単になり、他のプロセスはより複雑になります。そして、変化のスピードとスケーラビリティの向上のために、犠牲を払わなければなりません。それらの1つは、分析の複雑さです。モノリスですべての運用分析を分析レプリカのSQLクエリに減らすことができる場合、マルチサービスアーキテクチャでは各サービスに独自のベースがあり、1つのクエリを省略できない（または省略できるか）ようです。当社の運用分析の問題をどのように解決したか、およびこのソリューションをどのように活用できるかを知りたい方は、ようこそ。


私の名前はPavel Sivashです。DomKlikでは、分析データウェアハウスの保守を担当するチームで働いています。従来、私たちの活動はエンジニアリングの日付に起因することがありましたが、実際には、タスクの範囲ははるかに広いです。データ分析および独自のツールの開発のためのツールの日付エンジニアリング、サポート、および適応のためのETL / ELT標準があります。特に、運用レポートの場合、私たちはモノリスを持っているように "偽装"し、アナリストに必要なすべてのデータを提供する1つの基盤を与えることにしました。

一般的に、さまざまなオプションを検討しました。本格的なリポジトリを構築することは可能でした-私たちは試してみましたが、正直に言って、リポジトリを構築して変更を加えるというかなり遅いプロセスで、ロジックを頻繁に変更することができませんでした（誰かが成功した場合は、コメントにコメントを書き込んでください）。 「ガイ、パイソンを学び、分析の手掛かりに行く」とアナリストに言うことができますが、これはスタッフ募集の追加要件であり、可能であればこれを回避する必要があるように思われました。 FDW（Foreign Data Wrapper）テクノロジーを使用することにしました。実際、これはSQL標準にある標準のdblinkですが、はるかに便利なインターフェースを備えています。これに基づいて、私たちは決定を下しましたが、最終的には落ち着きました。その詳細は別の記事の主題であるか、おそらく1つではありません。データベーススキームの同期から、アクセス制御と個人データの匿名化まで、多くのことを話したいので。また、このソリューションは実際の分析データベースやリポジトリの代用ではなく、特定の問題のみを解決することを予約する必要があります。

トップレベルは次のようになります。


ユーザーが作業データを保存できるPostgreSQLデータベースがあり、最も重要なのは、すべてのサービスの分析レプリカがFDWを介してこのデータベースに接続されていることです。これにより、PostgreSQL、MySQL、MongoDB、またはその他（データベース、API、突然適切なラッパーがなくなった場合は独自に作成できます）に関係なく、いくつかのデータベースにクエリを書き込むことができます。さて、すべてがスーパーのようです！私たちは同意しませんか？

すべてが非常に迅速かつ簡単に終わったとしたら、おそらく記事はなかっただろう。

postgresがリモートサーバーへのリクエストを処理する方法を明確に理解することが重要です。これは理にかなっているように見えますが、注意を払っていないことがよくあります。postgresはリクエストをリモートサーバーで個別に実行される部分に分割し、このデータを収集し、最終的な計算はそれ自体で実行されるため、リクエストの速度はリクエストの記述方法に大きく依存します。また、注意が必要です。データがリモートサーバーからのものである場合、それらにはもはやインデックスがありません。スケジューラーを助けることができるものは何もないため、私たちだけがそれを助けて提案できます。それについてもっとお話ししたいと思います。

これは、クエリを記述するときに注意を払う必要がある瞬間です。フィルターはリモートサーバーに転送されませんでした。つまり、それを実行するために、postgresは600万行すべてを拡張し、その後ローカルでフィルター（フィルター行）して集計を実行します。成功の鍵は、フィルターがリモートマシンに転送されるようにリクエストを記述し、必要な行のみを受信して​​集計することです。

フレディ対 Jsonb


一般に、ブール型のフィールドと日付は既にクエリを高速化していますが、もう1つのデータ型がありました。率直に言って、フィルタリングでの戦いはまだ終わっていませんが、成功はあります。したがって、jsonbフィールドによるフィルターをリモートサーバーに渡す方法を以下に示します。

explain analyze verbose
SELECT count(1)
FROM fdw_schema.table 
WHERE is_active is True
AND created_dt BETWEEN CURRENT_DATE - INTERVAL '7 month' 
AND CURRENT_DATE - INTERVAL '6 month'
AND meta @> '{"source":"test"}'::jsonb;

Aggregate  (cost=245463.60..245463.61 rows=1 width=8) (actual time=6727.589..6727.590 rows=1 loops=1)
  Output: count(1)
  ->  Foreign Scan on fdw_schema."table"  (cost=1100.00..245459.90 rows=1478 width=0) (actual time=16.213..6634.794 rows=1360025 loops=1)
        Output: "table".id, "table".is_active, "table".meta, "table".created_dt
        Filter: (("table".is_active IS TRUE) AND ("table".created_dt >= (('now'::cstring)::date - '7 mons'::interval)) AND ("table".created_dt <= ((('now'::cstring)::date)::timestamp with time zone - '6 mons'::interval)))
        Rows Removed by Filter: 619961
        Remote SQL: SELECT created_dt, is_active FROM fdw_schema.table WHERE ((meta @> '{"source": "test"}'::jsonb))
Planning time: 0.747 ms
Execution time: 6727.815 ms

フィルタリング演算子の代わりに、あるjsonbを別のjsonbに含める演算子を使用する必要があります。最初の29ではなく7秒。これまでのところ、これはjsonb経由でリモートサーバーにフィルターを転送する唯一の成功したオプションですが、1つの制限を考慮することが重要です。データベースバージョン9.6を使用しますが、最新のテストを完了し、4月末までにバージョン12に移行する予定です。更新の際には、json_path、新しいCTEの動作、プッシュダウン（バージョン10から）など、多くの期待が寄せられている多くの変更があるため、それがどのように影響したかを説明します。すぐ試してみたいです。

彼を終える


それぞれの変更がリクエストの速度にどのように影響するかを個別に確認しました。次に、3つのフィルターがすべて正しく記述されているとどうなるかを見てみましょう。

explain analyze verbose
SELECT count(1)
FROM fdw_schema.table 
WHERE is_active = True
AND created_dt >= (SELECT CURRENT_DATE::timestamptz - INTERVAL '7 month') 
AND created_dt <(SELECT CURRENT_DATE::timestamptz - INTERVAL '6 month')
AND meta @> '{"source":"test"}'::jsonb;

Aggregate  (cost=322041.51..322041.52 rows=1 width=8) (actual time=2278.867..2278.867 rows=1 loops=1)
  Output: count(1)
  InitPlan 1 (returns $0)
    ->  Result  (cost=0.00..0.02 rows=1 width=8) (actual time=0.010..0.010 rows=1 loops=1)
          Output: ((('now'::cstring)::date)::timestamp with time zone - '7 mons'::interval)
  InitPlan 2 (returns $1)
    ->  Result  (cost=0.00..0.02 rows=1 width=8) (actual time=0.003..0.003 rows=1 loops=1)

  ->  Foreign Scan on fdw_schema."table"  (cost=100.02..322041.41 rows=25 width=0) (actual time=8.597..2153.809 rows=1360025 loops=1)

Planning time: 0.820 ms

はい、リクエストはより複雑に見えます。これは強制ボードですが、実行速度は2秒で、10倍以上高速です！そして、比較的小さなデータセットに対する単純なクエリについて話している。実際のリクエストでは、数百倍にも及ぶ成長を遂げました。

要約すると、FDWでPostgreSQLを使用する場合は、常にすべてのフィルターがリモートサーバーに送信されていることを確認してください。少なくとも、異なるサーバーのテーブル間の結合に到達するまでは満足しています。しかし、これは別の記事の話です。
