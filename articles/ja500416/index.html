なぜゲッターは必要ないのですか？

エンティティを操作する方法としてのセッター/ゲッター（PHPのSymfonyを例として使用）に関する以前の記事は、白熱した議論を受けました。この記事では、ゲッターについての私の考えを個別に表現するようにします。なぜ、いつ、いつ何かを入手するか、どのような責任を決定するか、いつ使用するのが適切か、適切でない場合です。私は一箇所に考えをまとめ、あなたと一緒に形式化しようとしました。

getHumanFactory（）getHuman（）。getDoneWork（）。getWorkTime（）


現在のオブジェクトの状態を取得するには、ゲッターが必要です。OOP言語では、これは通常はプライベートであるクラス変数の値です。

class Article {
     private String name

     public function getName(): String {
          return this.name
     }
}

クラス変数自体の値は、必要に応じて取得できます。オブジェクトにそれが何であるかを尋ねるだけです。メソッドの名前そのものが「与える」と言いますが、「実行」、「送信」、「作成」、「カウント」は行わないでください。与えると言う-何を与えるか。これは、あらゆる種類のデータオブジェクトで非常に正常に機能します。その責任は、情報を提供/伝達することです。

単語のセマンティクスは単純であるため、ゲッターにはロジックがありません。 「ママは、私にパイを与える含まれていません」「ママ、購入小麦粉、火災パイを、最終的には私のパイを与えます。」 「パイを提供してください」というフレーズでこれらのアクションをすべてカプセル化できるかもしれませんが、「与える」ことは間違いありません。

あなたは驚かれることでしょうが、私はすべてのメソッドに複数回名前を付けるこの方法に出会いました。ゲッターチェーンでは、アプリケーションレイヤーで何が起こっているのか、どのような接続が関係しているのかが常に明確であるとは限らないため、コードの複雑さはかなり大きくなります。一般に、この種類の設計と信頼性がすべての順序であるわけではなく、エラーの原因です。

getValue（）スローWhyDoYouNeedIt？{}


データオブジェクトの目的は理解できます。これは、退却の必要性についてのメッセージを伝えるためにどこかに逃げている1917年の映画の伍長です。

しかし、ビジネスオブジェクトをどうするか。エンティティ「ドキュメント」がフィールドのリストを誰かに与えるのはなぜですか？

これがビジネスオブジェクトの場合、ドキュメントを投稿、拒否、検証（これらのフィールドを含む）、入力、または確認できます。

フィールドを「与える」必要がある場合、このドキュメントはビジネスプロセスの一部ではありません。誰かにフィールドを与える必要があるのはなぜですか？多分出版？または抽出、アーカイブ、またはコピーを郵送またはレポートで送信しますか？理由と対象者が完全に明確ではありません。古き良きデータオブジェクトのように「与える」という別個の責任があり、ドキュメント自体を理解する上で基本的ではない別のビジネスプロセスのコンテキストで読むためのソースの形式での使用を明確に見ることができます。

doEverythingEverywhere（world.getGod（））


事業体に戻ります。ゲッターをエンティティーに残しておくと、そのゲッターをあらゆる場所で好きなように使用するという誘惑が大きくなります。特定のオブジェクトの状態に関連付けられており、キャンペーンのロジックを完全に逆転させます。カプセル化を壊していないように見えるかもしれません。しかし、どうですか？ある場所での状態、別の場所での動作-古典的なカプセル化違反。

たとえば、いくつかのエンティティがあります：

class Order
{
     private Status status
     private Boolean closed

     public function getStatus(): Status {
          return this.status
     }

     public function setClosed(Boolean closed): void {
           this.closed = closed
     }
}

確かに、そのようなアーキテクチャでは、ダース/ 100か所のステータスを要求します。あらゆる種類のサービス、コントローラー、その他のモジュールを使用できます。コードではなく、開発者向けの一連のルールによってこのコードの配布に人為的に制限が作成されたのは一度だけでした...カプセル化は、コード設計ではなくコーディング標準によって提供されました:)。

このようなことをどこかで行う必要がある場合：

if(order.getStatus() == input.getStatus()) {
      order.setClosed(true)
}

ほとんどの場合、エンティティには状態マシンが含まれていません。これは、内部からのオブジェクトの不変条件がまったく制御されないことを意味します。内部からの各操作中にデータの検証は行われません。その結果、高接続性、複雑な機能テスト、エンティティの状態を変更するコードのユニットロジックをチェックするだけでは不十分であるため、コード外部のエンティティの状態が有効であることを確認する必要があります。その結果、複雑さが増し、バグの可能性が高くなり、コードが増え、テストが複雑になります。

結論：私とあなたの同僚が、結果を返す任意の処理を行うメソッドのようにゲッターをあまり使わないことを願っています。

そして、より多くの開発者がCQRSの概念に注意を払うことを望みます。CQRSの概念では、読書とビジネス操作の責任が分割されます。
