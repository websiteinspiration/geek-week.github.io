サーバーのどのような負荷がネットワークメカニズムを作成しますか？

サーバーのネットワークサブシステムの動作を分析するときは、通常、待ち時間、システムのスループット、1秒あたりに処理できるパケット数（PPS、1秒あたりのパケット数）などの指標に注意が払われます。これらのインジケータは、調査中のコンピュータがどの最大負荷で動作できるかを理解するために使用されます。また、これらのメトリックは重要であり、システムについて多くのことを語ることができますが、ネットワークパケットの処理がサーバーで実行されているプログラムに与える影響に関する情報は提供されません。 この資料は、サーバーのネットワークメカニズムによって生じる負荷を調査することを目的としています。特に、ネットワーク問題の解決策がLinuxシステムで実行されているさまざまなプロセスから「盗む」ことができるプロセッサー時間について説明します。





Linuxでのネットワークパケット処理


Linuxは、対応するIRQの処理時にプロセッサーによって実行されるプロセスのコンテキストで、かなりの数のパケットを処理します。システムアカウンティングエンジンは、このために使用されるプロセッササイクルを、現在実行中のプロセスに割り当てます。これは、このプロセスがネットワークパケット処理と関係がない場合でも行われます。たとえば、チームtopはプロセスがプロセッサのリソースの99％以上を使用しているように見えるが、実際にはプロセッサ時間の60％がパケットの処理に費やされることを示す場合があります。これは、プロセス自体が問題を解決し、CPUリソースの40％しか使用しないことを意味します。

受信ハンドラーnet_rx_action通常、非常に高速に実行されます。たとえば、25μs未満です。 （このデータはeBPFを使用した測定から取得されました。詳細に興味がある場合は、こちらをご覧ください net_rx_action。）プロセッサは、タスクを別のSoftIRQサイクルに延期する前に、NAPIインスタンス（NICまたはRPS）ごとに最大64パケットを処理できます。休憩なしで次々と最大10のSoftIRQサイクルが続く可能性があり、約2ミリ秒かかります（これについての詳細は、について読むことで確認できます__do_softirq）。最大サイクル数が経過した後、または時間が経過した後もSoftIRQベクトルにまだ未解決の問題がある場合、これらの問題の解決はスレッドでの実行のために遅延されますksoftirqd特定のCPU。これが発生すると、システムは、ネットワーク操作によって作成されたプロセッサの負荷に関する情報を取得するという意味で、もう少し透過的であることがわかります（このような分析は、調査対象がSoftIRQであり、パケット処理に関連しており、他のものには関連していないという前提で実行されます）。 。

上記のインジケーターを取得する1つの方法は、以下を使用することperfです。

sudo perf record -a \
        -e irq:irq_handler_entry,irq:irq_handler_exit
        -e irq:softirq_entry --filter="vec == 3" \
        -e irq:softirq_exit --filter="vec == 3"  \
        -e napi:napi_poll \
        -- sleep 1

sudo perf script

この場合、プロセッサはアイドル状態であり（そのswapperため、プロセスのエントリが表示されます）、CPU 5のRxキューに対してIRQが呼び出され、SoftIRQ処理が2回呼び出され、64パケットが最初に処理され、次に27が処理されます。次のIRQは229μs後に呼び出され、サイクルを再び開始します。

このデータは、アイドル状態のシステムで取得されました。しかし、プロセッサ上では、任意のタスクを実行できます。この場合、上記の一連のイベントが発生し、このタスクが中断され、IRQ / SoftIRQタスクが実行されます。同時に、システムアカウンティングは、中断されたプロセスにプロセッサによって作成された負荷を割り当てます。その結果、ネットワークパケット処理タスクは通常、従来のプロセッサ負荷監視ツールから隠されます。それらは、ランダムに選択されたプロセスのコンテキストで、「犠牲者プロセス」のコンテキストで実行されます。これはいくつかの質問につながります。パケットの処理のためにプロセスが中断される時間を推定する方法は？ 2つの異なるネットワークソリューションを比較して、コンピューターで解決されるさまざまなタスクに与える影響が少ないものを理解するにはどうすればよいですか？

RSS、RPS、RFSメカニズムを使用する場合、パケット処理は通常、プロセッサコア間で分散されます。したがって、上記のパケット処理シーケンスは特定の各CPUに関連しています。パケットの到着率が上がると（毎秒100,000パケット以上の速度について話すことができると思います）、各CPUは毎秒数千または数万のパケットを処理する必要があります。非常に多くのパケットを処理すると、サーバーで実行される他のタスクに必ず影響が出ます。

この効果を評価する1つの方法を検討してください。

分散パケット処理の無効化


まず、RPSを無効にし、特定のMACアドレスに関連するすべてのパケットの処理を私たちが知っている唯一のCPUで整理することを目的としたフロー制御ルールを設定して、パケットの分散処理を停止しましょう。私のシステムには、802.3ad構成で2つのNICが集約されています。ネットワークタスクは、コンピューターで実行されている単一の仮想マシンに割り当てられます。

ネットワークアダプターのRPSは、次のように無効になっています。

for d in eth0 eth1; do
    find /sys/class/net/${d}/queues -name rps_cpus |
    while read f; do
            echo 0 | sudo tee ${f}
    done
done

次に、フロー制御ルールを設定して、パケットが単一のCPUを使用してテスト仮想マシンに入るようにします。

DMAC=12:34:de:ad:ca:fe
sudo ethtool -N eth0 flow-type ether dst ${DMAC} action 2
sudo ethtool -N eth1 flow-type ether dst ${DMAC} action 2

RPSを無効にし、フロー制御ルールを使用すると、仮想マシン宛てのすべてのパケットが同じCPUで処理されるようにすることができます。パケットを送信先のキューに確実に送信するには、ethqのようなコマンドを使用できます。次に、このキューがどのCPUに属しているかを調べます/proc/interrupts。私の場合、ターン2はCPU 5によって処理されます。

openssl speedコマンド


ユーティリティを使用しperfたり、着信トラフィックの処理を担当するSoftIRQランタイムを分析したりできますbpfが、このアプローチは非常に複雑です。さらに、観察プロセス自体が結果に確実に影響します。より簡単で理解しやすい解決策は、システム上の既知の負荷を作成するタスクを使用して、システム上のネットワーク操作によって作成された負荷を特定することです。たとえば、これはopenssl speedOpenSSLのパフォーマンスをテストするために使用されるコマンドです。これにより、プログラムが実際に取得するプロセッサリソースの量を確認し、プログラムが受け取るはずのリソースの量と比較することができます（これは、ネットワークタスクに費やされるリソースの量を見つけるのに役立ちます）。

チームはopenssl speedほぼ100％ユーザー空間チームです。特定のCPUに関連付けると、テストの実行中に、使用可能なすべてのリソースが使用されます。チームは、タイマーを指定された間隔（ここでは、たとえば、計算を簡単にするために10秒かかります）に設定し、テストを実行しtimes()てから、タイマーがトリガーされたら、それを使用して、プログラムが実際に取得したプロセッサー時間を調べます。見たsyscallところ、次のようになります。

alarm(10)                               = 0
times({tms_utime=0, tms_stime=0, tms_cutime=0, tms_cstime=0}) = 1726601344
--- SIGALRM {si_signo=SIGALRM, si_code=SI_KERNEL} ---
rt_sigaction(SIGALRM, ...) = 0
rt_sigreturn({mask=[]}) = 2782545353
times({tms_utime=1000, tms_stime=0, tms_cutime=0, tms_cstime=0}) = 1726602344

つまり、呼び出しalarm()と結果の検証の間に、システム呼び出しがほとんど行われていないことがわかります。プログラムが中断されなかった場合、またはごくまれに中断さtms_utimeれた場合、時間はテスト時間（この場合は10秒）と一致します。

これはユーザー空間でのみ実行されるテストであるため、表示さtimes()れるシステム時間は、システムに追加の負荷がかかることを意味します。これはopensslCPUで実行されるプロセスですが、CPU自体が他の処理でビジー状態になっている可能性があることがわかりました。たとえば、ネットワークパケットの処理：

alarm(10)                               = 0
times({tms_utime=0, tms_stime=0, tms_cutime=0, tms_cstime=0}) = 1726617896
--- SIGALRM {si_signo=SIGALRM, si_code=SI_KERNEL} ---
rt_sigaction(SIGALRM, ...) = 0
rt_sigreturn({mask=[]}) = 4079301579
times({tms_utime=178, tms_stime=571, tms_cutime=0, tms_cstime=0}) = 1726618896

ここではopenssl、7.49秒間プロセッサで作業することができたことがわかります（0.01秒に対応する測定単位で178 + 571）。しかし同時に5.71秒。この間隔はシステム時間で表されます。彼はopensslカーネル空間での仕事で忙しくないので、これは5.71秒ということです。-これは、システムに追加の負荷がかかった結果です。つまり、システムのニーズを満たすためにプロセスが「盗まれた」時期です。
