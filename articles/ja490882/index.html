Dockerでの問題を修正します。どうやら、GITはそれと何をしているのでしょうか？



Windows上のDockerは継続的な冒険です。次に、OSを更新する必要があります。そうしないと、最新バージョンがインストールされず、ネットワークへの接続方法を忘れてしまいます。一般的に、毎日彼からのニュース。 「設定して忘れる」はDocker Desktop for Windowsに関するものではありません。特に、開発者が推奨するとおりに使用されていない場合。また、何らかの理由で、外部ウィンドウをローカルとしてネットワークドライブに接続することを承認していません。また、ホストマシンにもあるネットワークフォルダーへのアクセスは承認しません。彼らはこれがセキュリティの観点からホラーホラーであると書いて おり、mountコマンドがコンテナで機能するためなど、あらゆる種類のキーが必要です。

cap_add:
- SYS_ADMIN
- DAC_READ_SEARCH



一般的に、コンテナーが顧客のサーバーにアップロードされた後、サービスがネットワークドライブを認識しなくなったとき、私は特に驚くことはありませんでした。これはすでに発生しており、Dockerのネットワーク設定が壊れたときにどのように、何を再起動するかについて、サポートグループ向けの段階的な説明さえ書かれています。

だから私は私の指示を開いて行動します。コンテナを再起動しても効果はありません。インフラストラクチャの再作成でdocker-composeを介して再起動します-助けにはなりません。 Dockerの設定を工場出荷時の設定にリセットし、仮想マシンの設定を復元し、イメージをリロードし、docker-composeを実行します。ここでも、すべてが以前と同じです-ネットワークが表示されません。より正確には、コンテナからSMBサーバーへのpingは正常ですが、ネットワークボールには接続しません。最後のポイントは、サーバーを再起動してDockerを再インストールすることですが、実際にはサーバーを再起動したくないので、スキップします。この命令で終了しました。

では、自宅のマシンに移動します。ここにはWindows用のDockerもありますが、少し新しいバージョンです。確認します。同じ卵：



うん。まあ、本当に、Dockerは何らかのセキュリティを備えた更新プログラムをロールアップしたと思いますが、これが原因でスクリプトが開始されないのですか？最後のチェックは、マシンからDockerを完全に削除し、再インストールすることです。これは、出荷時の設定にリセットするよりも涼しいはずです。前のステップのリスト全体を実行していますが、これに加えて、完全に皮肉なように車を再起動しています。私はDockerを最初から置き、画像をアップロードし、docker-composeを起動します-最初に！すべてのサービスはどちらもネットワークボール

を認識せず、読み込み時に「mount error（22）：Invalid argument」を書き込み続けます。コマンドラインから1行ずつスクリプトを実行しようとします。コンテナに接続し、コマンドを順番に実行して、すべてが接続され、次のように動作することを確認しますする必要があります：

mkdir -p $SMB_MOUNT_POINT
mount -t cifs $SMB_SHARE $SMB_MOUNT_POINT -o user=$SMB_USER,password=$SMB_PASSWORD,vers=2.0
ls $SMB_MOUNT_POINT

つまり、これは何ですか、コンテナの起動時にスクリプトにパラメータを渡すなんらかのがらくたですか。

より多くのアイデアを探しています。 Dockerの再起動に関するすべてのオプションは浅く、親イメージで変更の可能性があるオプションがあります。私のイメージはopenjdk：8-jdk-alpineに基づいて構築されており、特定のバージョンが指定されていないため、セキュリティの改善によりスクリプトが壊れる可能性があります。多分彼らはOpenJDKまたはアルパインの子会社で何かを変更しましたか？

プロジェクトログを確認し、古いopenjdk：8-jdk-alpine-3.8、openjdk：8-jdk-alpine-3.7などを選択してみます。 -コンテナを再構築するたびに、確認してください-すべてが以前と同じです。

翌朝、アイデアが浮かびます。今こそ、それを見つけ出す時期であり、スクリプトを実行したときに正確に何が気に入らないのでしょうか。

「mount error（22）：Invalid argument」というメッセージは、それほど有益ではありません。スクリプトを実行するときにデバッグ情報を表示するbashのマジックキーがあることがわかりました。

sh内でデバッグを開始します。

/ # sh -x /entry-point.sh
' echo 'Mount //<private_ip>/Exchange/Pipeline to /pipeline
Mount //<private_ip>/Exchange/Pipeline to /pipeline
' mkdir -p '/pipeline

mount error(22): Invalid argument
Refer to the mount.cifs(8) manual page (e.g. man mount.cifs)

' ls '/pipeline
+ exec

そして、ここでいくつかの奇妙な瞬間が表示されます-行は引用符で始まり、最後に引用符があります...引用符はどこからですか？

アイデア-実際のbashを使用して起動すると、より情報が得られますか？

BASHコンテナーにインストールします。

apk add bash

デバッグを開始します。

/ # bash -x /entry-point.sh
' echo 'Mount //<private_ip>/Exchange/Pipeline to /pipeline
Mount //<private_ip>/Exchange/Pipeline to /pipeline
+ mkdir -p $'/pipeline\r'
+ mount -t cifs //<private_ip>/Exchange/Pipeline /pipeline -o $'user=<private_user>,password=<private_password>,vers=2.0\r'
mount error(22): Invalid argument
Refer to the mount.cifs(8) manual page (e.g. man mount.cifs)
mount: mounting //<private_ip>/Exchange/Pipeline on /pipeline failed: Invalid argument
+ ls $'/pipeline\r'
+ exec

くそー、ラインがプラスで始まっているようです-良いですが、\ rと$ '...'のオプションが表示

されました。MidnightCommanderを設定して、アメニティを試しapk add mc、スクリプトを開いて編集します



。^各行末のM。まあ、まあ、ローカルプロジェクトを見てください-行末はどうですか???? CRLF。ただし、Windowsで動作します。

この特定のCRLFファイルをLF（長いライブのメモ帳++！）に変更し、プロジェクトを収集します-ビンゴ！正常に動作します。

なぜ以前は大丈夫だったのに、今はすべてが飛んでしまったのですか私はコミットを調べます-変更はありませんでした。GITはその場で改行を編集できることを覚えていますテキストファイル。そして先日、私は新しいリポジトリを接続し、CRLFに変換してそこからすべてのファイルをアップロードした可能性があります。

その結果、プロジェクトに.gitattributesファイルを追加しました。これは、UNIXの場合と同様に、別のファイルで行末文字を保存する必要があることを示しています。

# Set the default behavior, in case people don't have core.autocrlf set.
* text=auto

# Declare files that will always have LF line endings on checkout.
*.sh text eol=lf

Docker Desktop for Windows 2.2.0.3を最新バージョンに更新した後、すべてが動作しなくなりました。今回、私はすぐにコンテナーの内部に入り、デバッグを開始しました。ホストマシンにアクセスするための10.0.75.1 IPアドレスが機能しなくなり、DockerNatがシステムから削除され、DockerDesktopVM仮想マシンには外部ネットワークアダプターさえないことがわかりました。それとの通信は標準的なネットワークを経由しないが、どういうわけか異なる。また、ホストにアクセスするには、host.docker.internalを使用する必要があります。仲間の開発者とそれを書いた
ホストからコンテナーへの通信にIPアドレスを使用することはサポートされている機能ではないため、DockerNATはDocker Desktop 2.2.0.0から削除されました。コンテナーからホストに通信するには、特別なDNS名host.docker.internalを使用する必要があります。

さて、私はテスト環境の構成を修正しました、データベースがピックアップされ、コンテナーからhost.docker.internalパスにpingしましたが、ネットワークドライブは接続されていません。シェルから手動でマウントを実行しようとすると、よくあるエラー「マウントエラー（22）：無効な引数」が表示されます。

引数を順番に削除します- 「mount -t cifs //host.docker.internal/playground / pipeline」を実行するだけです-機能しているようですが、rootユーザーからノックされています。

ユーザーを追加します。mount - t cifs //host.docker.internal/playground / pipeline -o user = smbuser-パスワードを要求して接続します！

フルバージョンも機能します：

mount -t cifs //host.docker.internal/playground /pipeline -o user=smbuser,password=smbpassword

しかし、「mount -t cifs //host.docker.internal/playground / pipeline -o user = smbuser、password = smbpassword、vers = 2.0」は耕起しません。

最後のパラメーターをvers = 2.1に変更します-乾杯、うまくいきます！

最新バージョンのDockerは、ブラックジャックを備えたSMBサーバーの独自の実装を行ったようですが、2.0のサポートはありません。もちろん、他のニュースと比較すると、ナンセンスです。
