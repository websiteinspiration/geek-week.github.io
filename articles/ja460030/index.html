ネイティブ開発、React NativeおよびFlutter：選択基準

モバイルアプリケーションの設計段階では、特定のプロジェクトでどの言語を使用する方が収益性が高いかを理解することが重要です。ネイティブ開発（たとえば、iOSの場合-SwiftまたはObjective-C、Androidの場合-JavaまたはKotlin）とともに、React NativeやFlutterなどのクロスプラットフォームフレームワークが使用されます。SimbirSoftは、ビジネス開発者とモバイル開発者の両方を選択するのに役立ついくつかの基準を提供しています。



選択の問題


さまざまな見積もりによると、世界にはすでに約50億のスマートフォンがあり、そのうち80％までがAndroidオペレーティングシステムを使用しており、20％未満がiOSを使用しています。それでも、国ごとに独自の特徴があります。たとえば、米国では65％以上のスマートフォンがiOSで動作します。モバイルアプリケーションを作成する場合、iOSとAndroidの両方でバージョンをリリースする必要があります。これを行うには、ネイティブまたはクロスプラットフォーム（「ハイブリッド」）開発に切り替えます。

ネイティブ開発は、さまざまな言語を使用し、各プラットフォームの機能を考慮して、各プラットフォームのアプリケーションを個別に作成する必要がある古典的なソリューションです。プロジェクトの複数のバージョンを作成する場合、複数のチームが同時に作業します。

クロスプラットフォームのフレームワークのおかげで、一度に「1石で2羽の鳥を殺す」ことが可能になり、1つのツールを使用してiOSとAndroidのバージョンを準備することができました。フレームワークの中には、特に広く普及しているものがあります。

ネイティブ開発


ネイティブ開発とクロスプラットフォーム開発の両方に独自の特性があります。モバイル部門mobile.SimbirSoftの実践では、両方のアプローチを使用しました。ネイティブ開発の利点には、次のものがあります。

  • アプリケーションの安定した高速操作。
  • アプリケーションの最大寿命。
  • 「ネイティブ」ツールの使用により、アプリケーションはより柔軟でスケーラブルになります。
  • アーキテクチャと機能に対する制限が少ない。
  • インターフェイスはプラットフォームと完全に一致します。

ハイブリッド開発


クロスプラットフォームフレームワークは、複数のオペレーティングシステム用にアプリケーションを「カスタマイズ」するため、プラットフォームごとに固有の要素を作成する必要はありません。結果として：

  • 必要な専門家は少なくなります。
  • 時間とリソースが少なくて済みます。
  • 開発速度が向上しています。

アプリケーションロジックがすべてのプラットフォームで同じであり、インターフェースがシンプルである場合、ハイブリッド開発は製品の市場投入を加速します。以下の条件下では、React NativeやFlutterなどのフレームワークをお勧めします。

  • 短時間で複数のプラットフォーム用のプロトタイプアプリケーションを作成する必要がある場合。
  • アプリケーションがサーバーからすべての情報を取得する場合、ロジックはサーバーに実装され、アニメーションとインターフェースは重要ではありません。
  • プッシュ通知、アプリ内購入、地理位置情報など、ネイティブ機能の最小数が使用されている場合。
  • 適度なパフォーマンスが許容され、コンテンツがインターネットからダウンロードされる場合。

1.可用性


一部の企業は、モバイルアプリケーションの開発を依頼されたときに、コストを最も重要な基準の1つとして指摘しています。同時に、スペシャリストのコストとニーズは、開発方法によって異なります。

ネイティブ

原則として、モバイルスタジオには複雑なネイティブアプリケーションの開発に十分なリソースがあります。

React NativeとFlutter

これらの分野の専門家の数は少なくなっています。大規模なアプリケーションの開発は困難な場合があります。

アプリケーションが長期間の運用（つまり、「寿命」）を考慮して設計されている場合は、ネイティブテクノロジーをビジネスに推奨します。高性能、複雑なインターフェースとアニメーション、低消費電力、サードパーティのリソース（APIなど）との統合が必要な場合にも役立ちます。ネイティブアプリケーションは、テクニカルサポートコストが低いため、長期的には収益性が高くなります。

たとえば、特定のイベントのために開発された、仮説のテスト、プロトタイプの開発、および動作寿命が明確に制限されたアプリケーションの開発には、クロスプラットフォームテクノロジーをお勧めします。

市場投入までの時間（市場投入までの時間）は、主に利用可能なチームの規模とモバイルアプリケーションの機能によって決まります。

カメラへのアクセス、ファイルシステムとフィンガープリントの操作、実行時の許可など、プラットフォーム固有の詳細情報がなく、同じUIでアプリケーションがシンプルな場合は、クロスプラットフォームフレームワークを使用する方が収益性と速度が向上します。ここで、ハイブリッド開発は、2つのプラットフォームのほとんどすべてのコードとUIを再利用することで時間を節約します。ただし、複雑なカスタムビューを作成する必要がある場合、クロスプラットフォーム開発は遅くなります。

3.セキュリティと展望


言語を選択するとき、ビジネスの信頼性とセキュリティを、技術的な観点から、また今後の開発の見通しと陳腐化のリスクから、今日評価することが重要です。

信頼性の面では、ネイティブ開発は競合他社よりも優れています。メインライブラリの開発は数年前から行われており、ほとんどのバグはすでに修正されており、Java、Objective-C、Swift、Kotlinなどのネイティブ言語は常に進化しています。モバイルデベロッパーの間では、2020年代にネイティブAndroid開発がFlutterに部分的に置き換えられるという意見を聞くことができますが、これはこれまでのところ単なる仮定です。

React Nativeは、Skype、Instagram、Facebook、その他の有名な製品など、安全なモバイルアプリケーションを作成するためのすべてのツールを提供します。セキュリティの問題は、開発中にサードパーティのモジュールを使用する場合にのみ可能です。同時に、JavaScriptは活発に開発されており、新機能がリリースされています。近い将来、陳腐化のリスクは最小限に抑えられます。

Flutterでの開発の場合、リスクは高くなります。フレームワークは若いため、リリースは2018年12月にのみリリースされました。これまでのところ、たとえば、ライブラリアセンブリは1つのプラットフォームでしか使用できない、Android Studioでクラッシュが発生する、一部のプラグインとライブラリにバグがあるなどの問題があります。一方、これはすべて確定され、修正されています。GoogleがFlutterのサポートを停止するリスクを排除することはできません。ただし、Fuchsia OSはFlutterで作成されており、一部の開発者はAndroidの代替品を目にしています。

開発者向け


上記では、ビジネスと請負業者の両方が考慮に入れる要素について説明しました。また、主にプロジェクトマネージャーが管理する技術基準もあります。例：

-母国語とチームの好みに関する知識のレベル

各モバイルスタジオには、テクノロジーの選択に独自の好みがあります。ネイティブ開発には、それぞれの言語の最も完全な知識が必要です。ただし、システムのネイティブツールを使用するため、プラットフォーム固有のツールをカスタマイズまたはアクセスする際の制限や問題は少なくなります（React NativeやFlutterとは異なります）。 JavaScriptの経験がある場合、モバイル開発者はReact Native（Flutterの場合のようにDart言語を追加で学ぶ必要はありません）またはDart（TypeScriptの知識は大きなプラスになるでしょう）に非常に簡単に切り替えることができます。

React Nativeは内部でネイティブモジュールを使用します。その結果、カスタマイズが必要な場合（そして、これはそのままではサポートされていません）、ネイティブモジュールを使用する必要があります。たとえば、私たちの実践では、Yandex.Martライブラリをカスタマイズして、カスタムビジュアルコンポーネントをマップに表示する必要がある場合がありました。

Flutterは、React Nativeとは異なり、独自のグラフィックエンジンで際立っています。一方で、これにより、単純なアプリケーションを開発するときにネイティブにまったく触れないようにすることができます。一方、ネイティブにアクセスする必要がある場合、これは追加の問題（たとえば、基本データ型とJSONを使用したメッセージの交換）とネイティブのグラフィカルコンポーネントを使用できないことを意味します。

-エントリーしきい値

開発者が新しい言語を学ぶことを決定した場合、コミュニティの可用性、および背景情報とドキュメントが重要な問題になります。

FlutterとReact Nativeは常に進化しており、アクティブなプロフェッショナルコミュニティと優れたドキュメントがあります。同時に、コミュニティの拡大と、複雑なコンポーネントの開発プロセスを説明するトレーニング資料やフォーラムのおかげで、ネイティブ開発はフレームワークよりも進んでいます。

推奨事項


フレームワークを優先して選択する場合は、作業の次の側面に注意を払うことをお勧めします。

評価関連

するすべてのプラットフォーム（iOS、Android）をテストする必要があります。すべてのプロジェクト参加者の知識と経験のレベルを客観的に評価して、数時間での評価が過小評価されないようにすることが重要です。開発中のReact NativeおよびFlutterフレームワーク自体のバグのリスクを考慮してください。

デザイン

一部の要素は、FlutterやReact Nativeでレンダリングすることが難しい（または不可能でさえある）場合があります。このため、デザインは開発者と合意する必要があります-顧客が美しく描かれた絵に恋する前に。

CI / CD

React Nativeでは、自動アセンブリに関する特定の問題は除外されていません（たとえば、異なるプラットフォームにライブラリがインストールされているため）。より多くのリスク予備を置く必要があります。

スプラッシュ画面

Flutterでのスプラッシュ画面の実装は、この要素をネイティブでのみ描画できるため、バグが発生する可能性が高いReact Nativeよりも高速です。すべてのレンダリングとバグ修正を含むスプラッシュスクリーンでReact Nativeを使用する場合は、さらに時間をかけることをお勧めします。

レイアウト

React Nativeを使用する場合、いずれかのシステムにレイアウトを適合させる際の問題を回避するために、iOSとAndroidでのレイアウトを同時に行う必要があります。

ウェブとモバイルの並行開発

アプリケーションのWebバージョンがReactで記述されている場合、コンポーネントのロジックが同じであるため、React Nativeでのモバイルアプリケーションの開発に費やす時間が短縮されます。

デバッグ

アプリケーションが大きい場合、React Nativeを使用すると、テストと単体テストが簡単になります。Flutterでは、ログが情報を提供しないため、バグ修正にもっと時間をかける必要があります。

システムファイルを使用したアプリケーションの動作

すべてのファイルで名前とパスを取得することはできませんが、SDカードへのアクセス許可を要求する必要があります。ファイルを送信するには、ContentResolverを使用する必要があります。リスクを最小限に抑えるために、ファイルシステムに関連するすべての操作に時間をかけます。

クライアントへのアセンブリの配信

ネイティブ開発と大きな違いはありません。Crashlytics、TestFairy、TestFlightなどの便利なサービスを選択できます。

React Native vs Flutter


SimbirSoftでは、アプリケーションの性質に応じて、React NativeとFlutterの両方を使用しています。特定のフレームワークでの作業の機能を提供するのに役立つ、私たちの実践からのいくつかの観察を共有します。



要約する
