Go：構造のコピーの代わりにポインタを使用する必要がありますか？


ルネフレンチが作成したオリジナルのgopherからA Journey With Go用に作成されたイラスト。

パフォーマンスに関しては、多くのGo開発者にとって、構造自体をコピーして構造を共有するのではなく、ポインターを体系的に使用するのが最良のオプションのようです。構造のコピーの代わりにポインタを使用する効果を理解するために、2つの使用例を検討します。

集中的なデータ配信


構造体を共有してその値にアクセスする場合の簡単な例を見てみましょう。

type S struct {
  a, b, c int64
  d, e, f string
  g, h, i float64
}

以下は、コピーまたはポインタで共有できる基本構造です。

func byCopy() S {
  return S{
     a: 1, b: 1, c: 1,
     e: "foo", f: "foo",
     g: 1.0, h: 1.0, i: 1.0,
  }
}

func byPointer() *S {
  return &S{
     a: 1, b: 1, c: 1,
     e: "foo", f: "foo",
     g: 1.0, h: 1.0, i: 1.0,
  }
}

これら2つの方法に基づいて、2つのベンチマークを作成できます。1つ目は、構造がコピーと共に渡される場所です。

func BenchmarkMemoryStack(b *testing.B) {
  var s S

  f, err := os.Create("stack.out")
  if err != nil {
     panic(err)
  }
  defer f.Close()

  err = trace.Start(f)
  if err != nil {
     panic(err)
  }

  for i := 0; i < b.N; i++ {
     s = byCopy()
  }

  trace.Stop()

  b.StopTimer()

  _ = fmt.Sprintf("%v", s.a)
}

2番目-最初と非常に似ています-構造体はポインターによって渡されます：

func BenchmarkMemoryHeap(b *testing.B) {
  var s *S

  f, err := os.Create("heap.out")
  if err != nil {
     panic(err)
  }
  defer f.Close()

  err = trace.Start(f)
  if err != nil {
     panic(err)
  }

  for i := 0; i < b.N; i++ {
     s = byPointer()
  }

  trace.Stop()

  b.StopTimer()

  _ = fmt.Sprintf("%v", s.a)
}

ベンチマークを実行してみましょう：

go test ./... -bench=BenchmarkMemoryHeap -benchmem -run=^$ -count=10 > head.txt && benchstat head.txt
go test ./... -bench=BenchmarkMemoryStack -benchmem -run=^$ -count=10 > stack.txt && benchstat stack.txt

構造体のコピーを使用すると、ポインタを使用するよりも8倍速くなりました。

理由を理解するために、トレースによって生成されたグラフを見てみましょう。


コピーによって渡された構造の


グラフコピーによってポインターによって渡された構造の

グラフ最初のグラフはかなり単純です。ヒープを使用しないため、ガベージコレクターや余分なゴルチンがありません。

2番目のケースでは、ポインターを使用すると、Goコンパイラーが変数をヒープに移動し、ガベージコレクターとして機能します。グラフのスケールを大きくすると、ガベージコレクターがプロセスの重要な部分を占めることがわかります。



このグラフは、ガベージコレクターが4ミリ秒ごとに起動することを示しています。

もう一度ズームインすると、何が起こっているのかについての詳細情報を取得できます。



青、ピンク、赤のストライプはガベージコレクターのフェーズで、茶色のストライプはヒープ内の割り当てに関連付けられています（グラフで「runtime.bgsweep」とマークされています）。

スイープは、使用済みとしてマークされていないメモリのデータ関連セクションのヒープからの解放です。このアクションは、ゴルーチンがヒープメモリ内の新しい値を分離しようとしたときに発生します。スイープ遅延は、ヒープメモリで割り当てを実行するコストに追加され、ガベージコレクションに関連する遅延には適用されません。

www.ardanlabs.com/blog/2018/12/garbage-collection-in-go-part1-semantics.html

この例が少し極端な場合でも、変数をスタックではなくヒープに割り当てると、コストが高くなる可能性があることがわかります。この例では、構造はスタックに割り当てられ、ヒープに作成されるよりもはるかに速くコピーされ、そのアドレスが共有されます。

スタック/ヒープに精通しておらず、内部の詳細についてもっと知りたい場合は、インターネットで多くの情報を見つけることができます。たとえば、Paul Gribble による記事です。

GOMAXPROCS = 1を使用してプロセッサを1に制限すると、事態はさらに悪化する可能性があります。

集中的な関数呼び出し


2つの空のメソッドを構造に追加し、ベンチマークを少し調整します。

func (s S) stack(s1 S) {}

func (s *S) heap(s1 *S) {}

スタックに配置されたベンチマークは構造を作成し、それにコピーを渡します：

func BenchmarkMemoryStack(b *testing.B) {
  var s S
  var s1 S

  s = byCopy()
  s1 = byCopy()
  for i := 0; i < b.N; i++ {
     for i := 0; i < 1000000; i++  {
        s.stack(s1)
     }
  }
}

そして、ヒープのベンチマークはポインタによって構造を渡します：

func BenchmarkMemoryHeap(b *testing.B) {
  var s *S
  var s1 *S

  s = byPointer()
  s1 = byPointer()
  for i := 0; i < b.N; i++ {
     for i := 0; i < 1000000; i++ {
        s.heap(s1)
     }
  }
}

結論


goで構造体のコピーの代わりにポインターを使用することは、常に良いとは限りません。データに適切なセマンティクスを選択するには、Bill Kennedyによって書かれた値/ポインターセマンティクスに関する投稿を読むことを強くお勧めします。これにより、構造と組み込み型で使用できる戦略のより良いアイデアが得られます。さらに、メモリ使用量のプロファイリングは、割り当てとヒープで何が起こっているかを理解するのに役立ちます。
