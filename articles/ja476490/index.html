最新のアプリケーション保護システム（WAF）の機能は、OWASPトップ10の脆弱性のリストよりもはるかに広いはずです。

回顧


アプリケーションのサイバー脅威の規模、構成、および構造は急速に進化しています。長年にわたり、ユーザーは一般的なWebブラウザーを使用してインターネット経由でWebアプリケーションにアクセスしてきました。各時点で、2〜5個のWebブラウザーをサポートする必要があり、Webアプリケーションの開発とテストのための一連の標準は十分に制限されていました。たとえば、ほとんどすべてのデータベースはSQLを使用して構築されています。残念ながら、しばらくすると、ハッカーはWebアプリケーションを使用してデータを盗んだり、削除したり、変更したりすることを学びました。彼らは違法アクセスを取得し、アプリケーションのユーザーをだまして、コードを実装してリモートで実行するなど、さまざまなトリックを使用してアプリケーションの機能を悪用しました。まもなく、Webアプリケーションファイアウォール（WAF）と呼ばれる商用Webアプリケーションセキュリティツールが市場に登場しました。安全なアプリケーションを開発するための標準と方法論を定義および維持するために、社会はそれに対応してオープンソースのWebアプリケーションセキュリティプロジェクトであるOpen Web Application Security Project（OWASP）を作成しました。

基本的なアプリケーション保護


OWASPトップ10リストは、アプリケーションを保護するための開始点であり、アプリケーションの脆弱性につながる可能性のある最も危険な脅威と誤った設定のリスト、および攻撃を検出して撃退するための戦術が含まれています。OWASPトップ10は、世界中のサイバーセキュリティアプリケーション業界で認められたベンチマークであり、Webアプリケーションセキュリティシステム（WAF）が持つべき機能の基本的なリストを定義しています。

さらに、WAF機能では、クロスサイトリクエストフォージェリ（CSRF）、クリックジャッキング、Webスクレイピング、ファイルインクルード（RFI / LFI）など、Webアプリケーションに対するその他の一般的な攻撃を考慮する必要があります。

最新のアプリケーションの脅威とセキュリティの課題


これまでのところ、すべてのアプリケーションがネットワークバージョンで実行されるわけではありません。クラウドアプリケーション、モバイルアプリケーション、APIがあり、最新のアーキテクチャでは、個々のソフトウェア機能さえあります。これらのタイプのアプリケーションはすべて、データを作成、変更、および処理するときに、同期および制御する必要があります。新しいテクノロジーとパラダイムの出現により、アプリケーションのライフサイクルのすべての段階で新しい課題と課題が発生しています。これには、開発と運用（DevOps）、コンテナー、モノのインターネット（IoT）、オープンソースツール、APIなどの統合が含まれます。

分散アプリケーションの展開とさまざまなテクノロジにより、情報セキュリティの専門家だけでなく、統一されたアプローチに依存できなくなったセキュリティソリューションメーカーにとっても、複雑で複雑なタスクが発生します。アプリケーションのセキュリティ機能では、ビジネスの詳細を考慮して、誤操作を防止し、ユーザーのサービス品質を低下させる必要があります。

ハッカーの最終的な目標は通常、データを盗むか、サービスの可用性を混乱させることです。攻撃者は技術の進化も利用します。まず、新しいテクノロジーの開発により、潜在的なギャップや脆弱性が生まれます。第二に、彼らの武器には、伝統的な救済策を回避するためのより多くのツールと知識があります。これにより、いわゆる「攻撃面」と組織が新しいリスクにさらされる可能性が大幅に増加します。セキュリティポリシーは、テクノロジーとアプリケーションの変化に伴って常に変化している必要があります。

したがって、アプリケーションは増え続けるさまざまな攻撃方法や攻撃源から保護されるべきであり、自動攻撃は情報に基づいた決定に基づいてリアルタイムで反映されるべきです。その結果、セキュリティポジションが弱体化する中、運用コストと手作業が増加します。

タスク番号1：ボット管理


インターネットトラフィックの60％以上がボットによって生成され、ボットの半分は「不良」トラフィックと呼ばれます（ラドウェアのセキュリティレポートによる）。組織はネットワーク帯域幅の拡大に投資しており、基本的にダミーの負荷に対応しています。実際のユーザートラフィックとボットトラフィック、および「良い」ボット（検索ロボットや価格比較サービスなど）と「悪い」ボットを正確に区別すると、大幅なコスト削減とユーザーのサービス品質の向上につながります。

ボットはこのタスクを促進するつもりはなく、実際のユーザーの行動を模倣し、CAPTCHAやその他の障害を回避できます。さらに、動的IPアドレスを使用した攻撃の場合、IPアドレスによるフィルタリングに基づく保護は無効になります。多くの場合、クライアント側のJavaScriptを実行できるオープンソース開発ツール（Phantom JSなど）を使用して、ブルートフォース攻撃、資格情報の詰め込み攻撃、DDoS攻撃、および自動化されたボット攻撃を仕掛けます。

効果的なボットトラフィック管理には、そのソースの一意の識別情報（指紋など）が必要です。ボット攻撃の場合には多くのエントリが発生するため、そのフィンガープリントを使用すると、疑わしいアクティビティを識別し、ポイントを割り当てることができます。これに基づいて、アプリケーション保護システムが情報に基づいた決定を行います-ブロック/許可-最小の誤検知率。



タスク＃2：API保護


多くのアプリケーションは、APIを通じてやり取りするサービスから情報とデータを収集します。APIを介して機密データを転送する場合、50％以上の組織がサイバー攻撃を検出するためにAPIをチェックまたは保護していません。

APIの使用例：

  • IoT統合
  • 機械の相互作用
  • サーバーレス環境
  • モバイルアプリケーション
  • イベント駆動型アプリケーション

APIの脆弱性はアプリケーションの脆弱性に似ており、インジェクション、プロトコル攻撃、パラメーター操作、リダイレクト、ボット攻撃が含まれます。専用APIゲートウェイは、APIを介して通信するアプリケーションサービスの互換性を保証します。ただし、WAFはHTTPヘッダーの解析、レイヤー7アクセスコントロールリスト（ACL）、JSON / XMLペイロードの解析とチェック、およびすべての脆弱性からの保護などの必要なセキュリティツールで使用できるため、アプリケーションにエンドツーエンドのセキュリティを提供しません。 OWASPトップ10リスト。これは、ポジティブモデルとネガティブモデルを使用して主要なAPI値を検査することで達成されます。

タスク3：サービス拒否


古いサービス拒否（DoS）攻撃ベクトルは、アプリケーションへの攻撃におけるその効果を証明し続けています。攻撃者は、HTTPまたはHTTPSフラッド、低速攻撃（SlowLoris、LOIC、Torshammerなど）、動的IPアドレスを使用した攻撃、バッファオーバーフロー、ブルートフォースなど、アプリケーションサービスを妨害する多くの成功したテクニックを持っています-攻撃、および他の多く。モノのインターネットの開発とそれに続くIoTボットネットの出現により、アプリケーションへの攻撃がDDoS攻撃の主な焦点になっています。ほとんどの接続追跡WAFは、限られた量の負荷にしか耐えることができません。ただし、HTTP / Sトラフィックフローを検査して、攻撃トラフィックと悪意のある接続を削除できます。攻撃が検出されると、このトラフィックを再度通過させることは意味がありません。攻撃を撃退するWAFの帯域幅は限られているため、次の「不良」パケットを自動的にブロックするには、ネットワークの境界で追加のソリューションが必要です。このセキュリティシナリオでは、両方のソリューションが相互に対話して、攻撃に関する情報を交換できる必要があります。
