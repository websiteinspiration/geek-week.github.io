コンテナー内でBuildahを実行するためのベストプラクティス

コンテナーのランタイムを個別のインストゥルメンタルコンポーネントに分割することの利点は何ですか？特に、これらのツールを組み合わせて互いに保護し始めることができるという事実。



多くは、Kubernetesまたは同様のシステム内にコンテナーOCIを構築するというアイデアに惹かれています。常にイメージを収集するCI / CDがあるとすると、Red Hat OpenShift / Kubernetesのようなものがアセンブリ中の負荷分散の点で非常に役立ちます。最近まで、ほとんどの人はコンテナーにドッカーソケットへのアクセス権を与えるだけで、ドッカービルドコマンドの実行を許可されていました。これは非常に危険であることを数年前に示しました。実際、パスワードなしのrootまたはsudoを与えるよりもさらに悪いです。

したがって、人々は常にコンテナでBuildahを実行しようとしています。要するに、私たちは、私たちの意見では、コンテナ内でBuildahを実行し、適切なイメージをquay.io/buildahに配置するのが最善の方法の例を作成しました。始めましょう ...

カスタマイズ


これらのイメージは、buildahimageフォルダーのBuildahリポジトリにあるDockerfilesからコンパイルされます。
ここでは、Dockerfileの安定版を見てみましょう。

# stable/Dockerfile
#
# Build a Buildah container image from the latest
# stable version of Buildah on the Fedoras Updates System.
# https://bodhi.fedoraproject.org/updates/?search=buildah
# This image can be used to create a secured container
# that runs safely with privileges within the container.
#
FROM fedora:latest

# Don't include container-selinux and remove
# directories used by dnf that are just taking
# up space.
RUN yum -y install buildah fuse-overlayfs --exclude container-selinux; rm -rf /var/cache /var/log/dnf* /var/log/yum.*

# Adjust storage.conf to enable Fuse storage.
RUN sed -i -e 's|^#mount_program|mount_program|g' -e '/additionalimage.*/a "/var/lib/shared",' /etc/containers/storage.conf

ホストのLinuxカーネルレベルで実装されたOverlayFSの代わりに、コンテナー内でfuse-overlayプログラムを使用します。現時点では、OverlayFSはLinux機能によってSYS_ADMIN特権が付与されている場合にのみマウントできるためです。そして、ルート権限なしでBuildahコンテナを実行したいと思います。ヒューズオーバーレイは、VFSストレージドライバーよりもかなり高速でパフォーマンスが優れています。 Fuseを使用してBuildahコンテナーを起動する場合は、デバイス/ dev / fuseを指定する必要があることに注意してください。

podman run --device /dev/fuse quay.io/buildahctr ...
RUN mkdir -p /var/lib/shared/overlay-images /var/lib/shared/overlay-layers; touch /var/lib/shared/overlay-images/images.lock; touch /var/lib/shared/overlay-layers/layers.lock

次に、追加のストレージ用のディレクトリを作成します。コンテナー/ストレージは、追加の読み取り専用イメージストレージを接続する概念をサポートします。たとえば、あるマシンでオーバーレイストレージエリアを構成し、NFSを使用してこのストレージを別のマシンにマウントし、プル経由でダウンロードせずにそこからイメージを使用できます。ホストからのある種のイメージストレージをボリュームとして接続し、それをコンテナー内で使用できるようにするために、このストレージが必要です。

# Set up environment variables to note that this is
# not starting with user namespace and default to
# isolate the filesystem with chroot.
ENV _BUILDAH_STARTED_IN_USERNS="" BUILDAH_ISOLATION=chroot

最後に、BUILDAH_ISOLATION環境変数を使用して、デフォルトで、Buildahコンテナーはchroot分離で開始する必要があると言います。ここではすでにコンテナ内で作業しているため、ここでは追加の分離は必要ありません。Buildahが名前空間を分離して独自のコンテナーを作成するには、SYS_ADMIN特権が必要です。これを行うには、コンテナーのSELinuxおよびSECCOMPルールを弱める必要があります。これは、セキュアなコンテナーからビルドするためのインストールと矛盾します。

コンテナー内でBuildahを実行する


上記のBuildahコンテナーイメージスキーマを使用すると、このようなコンテナーの実行方法を柔軟に変更できます。

スピードとセキュリティ


コンピュータのセキュリティは常に、プロセスの速度とそのプロセスにどの程度の保護が適用されるかとの間の妥協です。この説明は、コンテナを組み立てる場合にも当てはまるので、以下では、そのような妥協策のオプションを検討します。

上記のコンテナーイメージは、/ var / lib / containersにストレージを保持します。したがって、コンテンツをこのフォルダーにマウントする必要があります。これを行う方法は、コンテナーイメージのアセンブリ速度に大きく影響します。

3つのオプションを考えてみましょう。

オプション1.最大のセキュリティが必要な場合、各コンテナーに対してコンテナー/イメージ用の独自のフォルダーを作成し、ボリュームマウントを介してコンテナーに接続できます。さらに、contextディレクトリをコンテナー自体の/ビルドフォルダーに配置します。

# mkdir /var/lib/containers1
# podman run -v ./build:/build:z -v /var/lib/containers1:/var/lib/containers:Z quay.io/buildah/stable\
buildah  -t image1 bud /build
# podman run -v /var/lib/containers1:/var/lib/containers:Z quay.io/buildah/stable buildah  push \ image1 registry.company.com/myuser
# rm -rf /var/lib/containers1

安全性。このようなコンテナで実行されるBuildahは、最大のセキュリティを備えています。ケイパビリティツールではroot権限が付与されず、SECOMPおよびSELinuxのすべての制限が適用されます。このようなコンテナは、-uidmap 0：100000：10000のようなオプションを追加して、ユーザー名前空間分離で実行することもできます。 。

パフォーマンス。ただし、コンテナレジストリからのイメージは毎回ホストにコピーされ、キャッシングは「ノーウェイ」という言葉からは機能しないため、ここでのパフォーマンスは最小限です。作業が完了すると、Buildahコンテナはイメージをレジストリに送信し、ホスト上のコンテンツを破棄する必要があります。コンテナイメージが次回収集されるときに、ホストに何も残っていないため、レジストリから再度ダウンロードする必要があります。

オプション2 Dockerレベルのパフォーマンスが必要な場合は、ホストのコンテナー/ストレージをコンテナーに直接マウントできます。

# podman run -v ./build:/build:z -v /var/lib/containers:/var/lib/containers --security-opt label:disabled quay.io/buildah/stable buildah  -t image2 bud /build
# podman run -v /var/lib/containers:/var/lib/containers --security-opt label:disabled \ quay.io/buildah/stable buildah push image2 registry.company.com/myuser

安全性。これは、コンテナーを構築するための最も安全でない方法です。これは、コンテナーがホスト上のストレージを変更することが許可されており、PodmanまたはCRI-Oに悪意のあるイメージが侵入する可能性があるためです。さらに、Buildahコンテナー内のプロセスがホスト上のストレージと対話できるように、SELinux分離を無効にする必要があります。このオプションはDockerソケットよりも優れていることに注意してください。これは、コンテナーが残りのセキュリティ機能によってブロックされ、ホスト上でコンテナーを取得して実行できないためです。

パフォーマンス。キャッシングは完全に関係しているため、ここでは最大です。 PodmanまたはCRI-Oがすでに目的のイメージをホストにダウンロードできている場合は、コンテナー内のBuildahプロセスがイメージを再度ダウンロードする必要はなく、このイメージに基づく後続のアセンブリも必要なものをキャッシュから取得できます。

オプション3.この方法の本質は、1つのプロジェクトで複数の画像をコンテナ画像の共有フォルダと結合することです。

# mkdir /var/lib/project3
# podman run --security-opt label:level=s0:C100, C200 -v ./build:/build:z \
-v /var/lib/project3:/var/lib/containers:Z quay.io/buildah/stable buildah  -t image3 bud /build
# podman run --security-opt label:level=s0:C100, C200 \
-v /var/lib/project3:/var/lib/containers quay.io/buildah/stable buildah push image3 \ registry.company.com/myuser

この例では、開始の間にプロジェクトフォルダー（/ var / lib / project3）を削除しないため、プロジェクト内の後続のすべてのビルドでキャッシュが利用されます。

安全性。オプション1と2の間の何か。一方で、コンテナーはホスト上のコンテンツにアクセスできないため、Podman / CRI-Oイメージストレージに何か悪いものを入れることはできません。一方、プロジェクトの一環として、コンテナは他のコンテナの組み立てを妨害する可能性があります。

パフォーマンス。ここでは、Podman / CRI-Oを使用して以前にダウンロードしたイメージを使用できないため、ホストレベルで共有キャッシュを使用する場合よりも悪いです。ただし、Buildahがイメージをダウンロードした後、このイメージはプロジェクト内の後続のビルドで使用できます。

追加ストレージ


容器/貯蔵で起動し、容器アセンブリコンテナエンジンは、読み取り専用モードオーバーレイ外部記憶画像を使用することができることにより、追加の記憶装置（追加記憶）としてクラス片があります。実際、1つ以上の読み取り専用ストレージをstorage.confファイルに追加して、コンテナーの起動時にコンテナーエンジンが目的のイメージを検索できるようにすることができます。さらに、彼はこれらのリポジトリのいずれにも見つからない場合にのみ、レジストリからイメージをダウンロードします。コンテナーエンジンは書き込み

安全なストレージにのみ書き込むことができます。上にスクロールして、イメージquay.io/buildah/stableのビルドに使用するDockerfileを表示すると、次のような行があります。

# Adjust storage.conf to enable Fuse storage.
RUN sed -i -e 's|^#mount_program|mount_program|g' -e '/additionalimage.*/a "/var/lib/shared",' /etc/containers/storage.conf
RUN mkdir -p /var/lib/shared/overlay-images /var/lib/shared/overlay-layers; touch /var/lib/shared/overlay-images/images.lock; touch /var/lib/shared/overlay-layers/layers.lock

最初の行では、コンテナーイメージ内の/etc/containers/storage.confを変更して、ストレージドライバーに/ var / lib /共有フォルダーの「additionalimagestores」を使用するように指示しています。そして、次の行で、共有フォルダーを作成し、いくつかのロックファイルを追加して、コンテナー/ストレージからの誤用がないようにします。基本的に、空のコンテナーイメージストレージを作成するだけです。

このフォルダーの上にコンテナー/ストレージをマウントすると、Buildahはイメージを使用できるようになります。

ここで上記のオプション2に戻ります。Buildahコンテナは、コンテナへの読み書きとホストへの保存が可能であり、したがって、Podman / CRI-Oレベルでのイメージキャッシングにより最大のパフォーマンスを発揮しますが、直接書き込むことができるため、セキュリティは最小限になります。保管中。そして、ここで追加のストレージを固定して、2つの世界のベストを利用します。

# mkdir /var/lib/containers4
# podman run -v ./build:/build:z -v /var/lib/containers/storage:/var/lib/shared:ro -v \ /var/lib/containers4:/var/lib/containers:Z  quay.io/buildah/stable \
 buildah  -t image4 bud /build
# podman run -v /var/lib/containers/storage:/var/lib/shared:ro  \
-v >/var/lib/containers4:/var/lib/containers:Z quay.io/buildah/stable buildah push image4 \ registry.company.com/myuser
# rm -rf /var/lib/continers4

/ var / lib / containers /ストレージホストは、読み取り専用モードでコンテナ内で共有される/ var / lib /にマウントされていることに注意してください。したがって、Buildahはコンテナーで作業するときに、以前にPodman / CRI-O（hi、speed）を使用してダウンロードしたイメージを使用できますが、独自のリポジトリー（hi、security）にのみ書き込むことができます。これは、コンテナーのSELinux分離を無効にすることなく行われることにも注意してください。

重要なニュアンス


基盤となるストレージから画像を削除してはなりません。そうしないと、Buildahコンテナが飛び出す可能性があります。

そして、これはすべての利点ではありません。


追加ストレージの機能は、上記のシナリオに限定されません。たとえば、すべてのコンテナイメージを共有ネットワークストレージに配置して、すべてのBuildahコンテナへのアクセスを許可できます。コンテナイメージを構築するためにCI / CDシステムが定期的に使用する何百ものイメージがあるとします。これらすべてのイメージを1つのストレージホストに集中させ、優先ネットワークストレージツール（NFS、Gluster、Ceph、ISCSI、S3 ...）を使用して、このストレージへの共有アクセスをすべてのBuildahノードまたはKubernetesノードに開きます。

これで、このネットワークストレージを/ var / lib / sharedのBuildahコンテナーにマウントするだけで十分です。Buildahコンテナーはプル経由でイメージをダウンロードする必要がなくなりました。したがって、事前入力フェーズを破棄し、すぐにコンテナを展開する準備が整います。

そしてもちろん、これを既存のKubernetesシステムまたはコンテナーインフラストラクチャ内で使用して、プル経由でイメージをダウンロードすることなく、どこでもコンテナーを起動および実行できます。さらに、コンテナーレジストリは、更新されたイメージをロードするためのプッシュリクエストを受信すると、このイメージを共有ネットワークストレージに自動的に送信し、すべてのノードですぐに使用できるようになります。

コンテナイメージのサイズは、数ギガバイトに達することがあります。追加ストレージの機能により、そのようなイメージをノードごとに複製することなく実行でき、コンテナーの起動をほぼ瞬時に行うことができます。

さらに、現在、コンテナーアセンブリをさらに高速化する新しいオーバーレイボリュームマウント機能に取り組んでいます。

結論


Kubernetes / CRI-O、Podman、またはDockerのコンテナー内でBuildahを実行することは現実的であり、docker.socketを使用するよりも簡単ではるかに安全です。画像の操作の柔軟性が大幅に向上し、さまざまな方法で画像を起動して、セキュリティとパフォーマンスの最適なバランスをとることができます。

追加ストレージの機能により、ノードへのイメージのダウンロードを高速化したり、完全になくしたりすることができます。
