チャネル上のすべてのIPv6ホストにpingを実行します


最初の最も簡単なヒントは、チャネル上のすべてのIPv6ホストにpingを送信することです。

IPv6は、すべてのタイプの1対多通信にマルチキャストアドレスを使用します。ブロードキャスト（またはブロードキャスト）IPv6アドレスはありません。これは、IPv6をIPv4と区別します。IPv4にはいくつかのタイプのブロードキャストアドレスがあります。たとえば、「制限されたブロードキャスト」アドレス255.255.255.255 [RFC1122]です。

ただし、「全ノードマルチキャスト」のIPv6アドレスがあるため、チャネル内のすべてのIPv6ノードにpingするために使用します。 （「ブロードキャスト」アドレスは実際には、特別な名前のマ​​ルチキャストアドレスであり、すべてのノードを含むマルチキャストグループです。たとえば、「グループ」またはマルチキャストアドレスビットは、リンクレベルでイーサネットブロードキャストアドレスに含まれていることに注意してください）。

チャネルの全ノードマルチキャストIPv6アドレス：ff02::1. ffマルチキャストIPv6アドレスを示します。次の0は、ビットが設定されていないフラグの一部です。マルチキャストグループの領域を

さらに2定義します。 IPv4マルチキャストアドレスとは異なり、IPv6マルチキャストアドレスにはスコープがあります。スコープ値は、マルチキャストパケットが許可されるネットワークの部分を示します。パケットが指定されたスコープの境界に到達すると、そのホップカウントフィールドがゼロ以外であるかどうかに関係なく、パケットは破棄されます。もちろん、マルチキャストグループの指定された境界に到達する前に変換カウンターがゼロに達した場合も、すぐにリセットされます。以下は、マルチキャストスコープIPv6の完全なリストです。

最後に::1、すべてのノードのマルチキャストグループを示します。

住所についてff02::1はあいまいですのでご注意ください。ルーターやマルチホームホストなど、複数のインターフェイスを持つIPv6ホストでは、ff02::1ICMPv6 pingを送信するインターフェイスを指定したり、ICMPv6 pingが到着したときに待機するように指定できるアドレスはありません。ff02::1有効であり、マルチインターフェースノードに接続されているすべてのインターフェースとチャネルで使用できます。

したがって、チャネル上のすべてのIPv6ノードにpingを実行するときは、IPv6のユーティリティpingに使用するインターフェイスを何らかの方法で通知する必要もあります。

インターフェース定義-ゾーンID


使用するインターフェイスを提供する別の方法は、IPv6アドレスパラメータの一部です。

ping出力でこの例を確認できます。応答するIPv6ノードのアドレスにもサフィックスが付いています%enp3s2。次に例を示します。

64 bytes from fe80::1d36:1fff:fefd:82be%enp3s2: icmp_seq=1 ttl=64 time=0.438 ms


このインターフェイスの定義方法は、[RFC4007]の「IPv6アドレスを指定したアーキテクチャ」で正式に説明されています。一般にオペレーティングシステムインターフェイスと呼ばれますが、実際には「ゾーン」または「スコープ」というより一般的なものを定義します。

[RFC4007]で説明されているように、IPv6ホストは複数の異なるIPv6インターフェイスを同じチャネルに接続できるため、より一般的なゾーンまたはスコープゾーンが存在する理由は次のとおりです。これらのインターフェースは同じゾーンのメンバーです。

オペレーティングシステムのゾーン内のいくつかのインターフェイスをグループ化できるはずです。現在、これがLinuxで可能かどうか、またその方法はわかりません。

リンクローカルアドレス応答


この全ノードマルチキャストpingから、合計6つの固有の応答を受け取りました。

これらの応答は、ユニキャストリンクローカルIPv6ホストアドレスから送信されました。たとえば、最初の答えは次のとおりです。

64 bytes from fe80::2392:6213:a15b:66ff%enp3s2: icmp_seq=1 ttl=64 time=0.106 ms


すべてのIPv6対応インターフェース[RFC4291]で必要なユニキャストリンクローカルIPv6アドレス、IPバージョン6アドレス指定アーキテクチャ。この理由は、IPv6ホストは常に自動的にユニキャストIPv6アドレスを持っているため、少なくとも直接接続されたチャネルを介して他のホストと通信するために使用できます。これには、リンクローカルホストアドレスを介した他のホストのアプリケーションとの通信が含まれます。

これにより、IPv6近隣探索やOSPFv3などのプロトコルの開発と実装が簡素化されます。また、ホスト上のエンドユーザーアプリケーションが、チャネル上でサポートされている他のIPv6インフラストラクチャを必要とせずに、チャネルを介して通信できるようにします。接続されているIPv6ホスト間の直接通信には、IPv6ルーターやDHCPv6サーバーを接続する必要はありません。

リンクローカルアドレスは、10ビットのプレフィックスfe80で始まり、その後に54個のゼロビットが続き、その後に64ビットのインターフェイス識別子（IID）が続きます。上記の最初の回答では2392:6213:a15b:66ff、これは64ビットのIIDです。

ループマルチキャスト


デフォルトでは、マルチキャストパケットは、それらを送信するノードに内部的に返されます。これは、IPv6アドレスとIPv4アドレスの両方で発生します。

このデフォルトの動作の理由は、マルチキャストパケットを送信するときに、ネットワーク上のどこかだけでなく、送信側のホスト自体でもリスニングローカルマルチキャストアプリケーションが実行されている可能性があるためです。このローカルアプリケーションは、マルチキャストパケットも受信する必要があります。

このマルチキャストローカルループは、ping出力で確認できます。

[mark@opy ~]$ ping -w 1 ff02::1%enp3s2

64 bytes from fe80::1d36:1fff:fefd:82be%enp3s2: icmp_seq=1 ttl=64 time=0.453 ms (DUP!)
...


最初の最も速い応答（0.453ミリ秒と比較して0.106ミリ秒）は、インターフェース自体に構成されたリンクローカルアドレスからのものenp3s2です。

[mark@opy ~]$ ip addr show dev enp3s2 | grep fe80
    inet6 fe80::2392:6213:a15b:66ff/64 scope link noprefixroute
