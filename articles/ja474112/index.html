サーバーのラック分散を最適化

チャットの1つで、彼らは私に質問をしました：

-そして、何かを読む必要があります、ラックにサーバーを正しくパックする方法は？

そんな文章を知らないことに気づき、自分で書きました。

まず、このテキストは、物理データセンター（DC）内の物理サーバーに関するものです。第2に、サーバーの数は多く、数百または数千と考えられます。数が少ない場合、このテキストは意味がありません。 3番目に、3つのリミッターがあると考えています。ラック内の物理的な場所、ラックへの電力供給、およびラックを列に並べることで、1つのToRスイッチを使用して隣接するラック内のサーバーを接続できます。

質問に対する答えは、最適化するパラメーターと、最良の結果を達成するために何を変えることができるかによって大きく異なります。たとえば、さらなる成長のためにより多くのスペースを残すために、最小限のスペースを取るだけで済みます。または、ラックの高さ、ラックあたりの電力、PDUのソケット、スイッチのグループ内のラックの数（1、2、または3ラックあたり1つのスイッチ）、ワイヤーの長さ、および引き作業（これは列の端で重要です）を自由に選択できます。列に10ラック、スイッチに3ラックの場合、別の列のワイヤーを引っ張るか、スイッチのポートを十分に活用する必要があります）など。個別のストーリー：サーバーの選択とDCの選択。これらは選択されていると想定しています。

微妙な違いや詳細、特に平均/最大サーバー消費量、および電力の供給方法を理解しておくとよいでしょう。したがって、ロシアの電源装置が230Vで、ラックごとに1つのフェーズがある場合、32Aマシンは約7kWを保持できます。公称で、ラックあたり6kWを支払うと仮定します。プロバイダーが各ラックではなく10ラックの数のみについて消費量を測定し、マシンが7 kWの従来のカットオフにある場合、技術的には別のラックで6.9 kWを別の5.1 kWで取得でき、すべてが大丈夫です-罰せられません。

通常、私たちの主な目標はコストを最小限に抑えることです。最良の測定基準は、TCO（総所有コスト）の削減です。次の要素で構成されています。

  • CAPEX：DCインフラストラクチャ、サーバー、ネットワークハードウェア、およびケーブルの調達
  • OPEX：DCレンタル、消費電力、メンテナンス。OPEXは耐用年数に依存します。これは3年に等しいと想定するのが妥当です。



個々のピースがパイ全体でどれだけ大きいかに応じて、最も高価なものを最適化し、残りの残りのリソースをできるだけ効率的に使用する必要があります。

既存のDCがあり、ラックの高さがHユニット（たとえば、H = 47）、ラックへの電力がP ラック（P ラック = 6 kW）であり、h = 2Uの2ユニットサーバーを使用するとします。2..4ユニットをラックからスイッチ、パッチパネル、オーガナイザーまで取り外します。それら。物理的には、S h =丸め（（H-2..4）/ h）サーバーがラックにあります（つまり、S h =丸め（（47-4）/ 2）= 21サーバー/ラック）。これがS hであることを思い出してください。

単純なケースでは、ラック内のすべてのサーバーは同じです。合計、ラックをサーバーでハンマーで打った場合、各サーバーで平均して消費できる電力はP serv = P ラック / S h（P serv = 6000 W / 21 = 287 W）です。簡単にするため、ここではスイッチの消費を無視します。

横に一歩進んで、サーバーの最大消費量がP maxであるかどうかを判断します。非常にシンプルで非効率的で完全に安全な場合は、サーバーの電源に書かれていることを読みます。それだけです。

より複雑で効率的な場合は、すべてのコンポーネントのTDP（熱設計パッケージ）を使用して要約します（これはあまり当てはまりませんが、そうなる可能性があります）。

通常、TDPコンポーネント（CPUを除いて）がわからないので、最も正確で最も難しいアプローチ（実験室が必要）も採用します。たとえば、Linpack（CPUとメモリ）やfio（ディスク）を使用して、必要な構成の実験サーバーを取得し、ロードします。消費を測定します。真剣に受け止める場合は、テスト中に冷たい廊下で最も暖かい環境を作成する必要があります。これは、ファンの消費とCPUの消費の両方に影響するためです。この特定の負荷の下で、これらの特定の条件における特定の構成の特定のサーバーの最大消費量を取得します。システムの新しいファームウェア、ソフトウェアの別のバージョン、その他の条件が結果に影響を与える可能性があることを意味します。

全体として、P servに戻り、それをP maxとどのように比較するか。これは、サービスがどのように機能し、技術者に対する神経がどれほど強いかを理解するための問題です。

まったくリスクを負わない場合は、すべてのサーバーがすぐに最大値を消費し始めると考えられます。同時に、DCへの1つの入力を形成できます。これらの条件でのインフラはそのためP、サービスを提供する必要がありますSERV ≡P 最大。これは、信頼性が極めて重要なアプローチです。

techdirが完全な安全性だけでなく、会社のお金についても考え、非常に勇気がある場合は、

  • ベンダーの管理を開始します。特に、1つの入力の低下を最小限に抑えるために、計画されたピーク負荷時にスケジュールされたメンテナンスを禁止します。
  • および/または私たちのアーキテクチャでは、ラック/列/ DCを失うことができ、サービスは引き続き機能します。
  • および/または、負荷をラック全体に水平に分散するため、サービスが1つのラックの最大消費量にすべて同時にジャンプすることはありません。

これは、推測するだけでなく、消費量を監視し、サーバーが通常およびピーク状態で実際に電力を消費する方法を知るのに非常に役立ちます。したがって、いくつかの分析の後、techdirはそれが持つすべてを圧縮し、次のように述べています。「ラックあたりの最大サーバー消費量の最大達成可能平均は、最大消費量よりも**非常に低い**と条件付きで決定する」P serv = 0.8 * P max。

そして、6kWラックでは、P max = 375Wの16サーバーではなく、P servの 20サーバー= 375W \ * 0.8 = 300W。それら。サーバー数が25％増加。これは非常に大きな節約です-結局のところ、必要なラックはすぐに25％少なくなります（また、PDU、スイッチ、ケーブルも節約されます）。そのような決定の深刻なマイナス-私たちの仮定が依然として真であることを常に監視する必要があります。新しいバージョンのファームウェアがファンの動作と消費を大幅に変更しないこと、新しいリリースの開発が突然サーバーの使用を大幅に効率的に開始しなかったこと（読み取り、サーバーの負荷と消費量が増えた）。結局のところ、最初の仮定と結論の両方がすぐに正しくなくなります。これは、責任を持って（または回避して、明らかに過負荷状態のラックに対して支払いを行う）必要があるリスクです。

重要な注意-可能であれば、ラック上のさまざまなサービスからサーバーを水平に分散するようにしてください。これは、1つのサービスのサーバーの1つのバッチが到着したときにストーリーが発生しないようにするために必要です。ラックは垂直に詰まっているため、「密度」が高くなります（簡単なため）。実際には、1つのラックには1つのサービスの同じ低負荷サーバーが詰め込まれ、他のラックも同様に高負荷であることがわかります。秒の落下の確率ははるかに高いです、負荷プロファイルは同じであり、このラック内のすべてのサーバーは、負荷の増加の結果として同じ量を消費し始めます。

ラック内のサーバーの分散に戻ります。ラックスペースの物理的な制限と電力の制限を調べ、次にネットワークを調べます。 24/32/48ポートNのスイッチを使用できます（たとえば、48ポートのToRスイッチがあります）。幸い、ブレイクアウトケーブルを考慮しない場合、多くのオプションはありません。ラックごとに1つのスイッチがあり、R ネットグループの2つまたは3つのラックに1つのスイッチがある場合のシナリオを検討します。グループで3つ以上のラックがすでに多すぎるように思えます。ラック間のケーブル接続の問題ははるかに大きくなります。

したがって、各ネットワークシナリオ（グループ内の1、2、または3ラック）では、サーバーをラックに分散します

。S ラック =最小（S h、切り捨て（P ラック / Pserv）、rounddown（N / R net））

したがって、グループに2つのラックがあるオプションの場合：

S ラック2 =最小（21、切り捨て（6000/300）、切り下げ（48/2））=最小（21、20 、24）=ラックあたり20サーバー。

同様に、残りのオプションを検討します

。S ラック1 = 20
S ラック3 = 16

そして、ほぼ完了です。すべてのSサーバーの分散用にラックの数を数えます（1000とします）：

R =切り上げ（S /（S ラック * R ネット））* R ネット

R 1 =切り上げ（1000 /（20 * 1））* 1 = 50 * 1 = 50

R 2ラック=切り上げ（1000 /（20 * 2））* 2 = 25 * 2 = 50ラック

R 3 =切り上げ（1000 /（16 * 3））* 3 = 21 * 3 = 63ラック

次に、以下に基づいて各オプションのTCOを検討しますラックの数、必要なスイッチの数、ケーブルなどTCOが少ないオプションを選択します。利益！

オプション1と2に必要なラック数は同じですが、価格が異なることに注意してください。 2番目のオプションのスイッチの数は半分で、必要なケーブルの長さは長くなります。
