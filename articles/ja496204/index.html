準備：リポジトリの変換


誰かが私たちの前で、私たちの代わりにこの問題を明確に解決したようです。既製の実用的なソリューションを見つける必要があるだけです。高速エクスポートはそれほど高速ではないことが判明しました。さらに、それは機能しませんでした。残念ながら、ログやスクリーンショットは残っていません。事実は彼ができなかったということです。Bitbucketは独自のコンバータを提供しません（可能です）。いくつかのグーグルメソッド、およびまた。私は自分でスクリプトを書くことにしました。とにかく、これが唯一のMercurialリポジトリではなく、将来的に役立つでしょう。初期の開発はここで紹介されます（それらはまだ同じ形式のままであるため）。作業のロジックは次のようになります。

  1. mercurial hggit拡張機能をベースにしています。 
  2. Mercurialリポジトリのすべてのブランチのリストを取得します。 
  3. ブランチ名を変換します（ブランチ名にスペースを入れてくれたmercurialと開発者に感謝します。ウムラウトなど、人生に喜びを与えるキャラクターに感謝します）。
  4. ブックマーク（hgブックマーク）を作成し、中間リポジトリにプッシュします。何のために？Bitbucketのため、およびブランチの名前と同じ名前（ステージングなど）でブックマークを作成できないためです。 
  5. 新しい（すでにgit）リポジトリで、ブランチ名からpostfixを削除し、hgignoreをgitignoreに移行します。 
  6. メインリポジトリにプッシュします。

ポイントに従ってコマンドをリストする：

  1. $ cd /path/to/hg/repo
$ cat << EOF >> ./.hg/hgrc
[extensions]
hggit=
EOF

  2. $ hg branches > ../branches

  3. #!/usr/bin/env bash
hgBranchList="./branches"
sed -i 's/:.*//g' ${hgBranchList}
symbolsToDelete=$(awk '{print $NF}' FS=" " ${hgBranchList} > sym_to_del.tmp)
 
i=0
while read hgBranch; do
  hgBranches[$i]=${hgBranch}
  i=$((i+1))
done <${hgBranchList}
 
i=0
while read str; do
  strToDel[$i]=${str}
  i=$((i+1))
done < ./sym_to_del.tmp
 
for i in ${!strToDel[@]}
do
  echo ${hgBranches[$i]} | sed "s/${strToDel[$i]}//" >> result.tmp
done
 
sed -i 's/[ \t]*$//' result.tmp
sed 's/^/"/g' result.tmp > branches_hg
sed -i 's/$/"/g' branches_hg
sed 's/ /-/g' result.tmp > branches_git
sed -i 's/-\/-/\//g' branches_git
sed -i 's/-\//\//g' branches_git
sed -i 's/\/-/\//g' branches_git
sed -i 's/---/-/g' branches_git
sed -i 's/--/-/g' branches_git
rm sym_to_del.tmp
rm result.tmp

  4. #!/usr/bin/env bash
gitBranchList="./branches_git"
hgBranchList="./branches_hg"
hgRepo="/repos/reponame"
 
i=0
while read hgBranch; do
  hgBranches[$i]=${hgBranch}
  i=$((i+1))
done <${hgBranchList}
 
i=0
while read gitBranch; do
  gitBranches[$i]=${gitBranch}
  i=$((i+1))
done <${gitBranchList}
 
cd ${hgRepo}
for i in ${!gitBranches[@]}
do
  hg bookmark -r "${hgBranches[$i]}" "${gitBranches[$i]}-cnv"
done
 
hg push git+ssh://git@bitbucket.org:username/reponame-temp.git
echo "Done."

  5. #!/bin/bash
# clone repo, run git branch -a, delete remotes/origin words to leave only branch names, delete -cnv postfix, delete default branch string because we can't delete it
repo="/repos/repo"
gitBranchList="./branches_git"
defaultBranch="default-cnv"
while read gitBranch; do   gitBranches[$i]=${gitBranch};   i=$((i+1)); done < $gitBranchList
cd $repo
for i in ${!gitBranches[@]}; do git checkout ${gitBranches[$i]}-cnv; done
git checkout $defaultBranch
for i in ${!gitBranches[@]}; do
    git branch -m ${gitBranches[$i]}-cnv ${gitBranches[$i]}
    git push origin :${gitBranches[$i]}-cnv ${gitBranches[$i]}
    git push origin -u ${gitBranches[$i]}
done

  6. #!/bin/bash
# clone repo, run git branch -a, delete remotes/origin words to leave only branch names, delete -cnv postfix, delete default branch string because we can't delete it
repo="/repos/repo"
gitBranchList="./branches_git"
defaultBranch="default"
while read gitBranch; do   gitBranches[$i]=${gitBranch};   i=$((i+1)); done < $gitBranchList
cd $repo
for i in ${!gitBranches[@]}; do
    git checkout ${gitBranches[$i]}
    sed -i '1d' .hgignore
    mv .hgignore .gitignore
    git add .
    git commit -m "Migrated ignore file"
done

いくつかのアクションの意味、つまり中間リポジトリの使用について説明しようと思います。最初、変換後のリポジトリでは、ブランチ名に「-cnv」という接尾辞が含まれています。これは、hgブックマーク機能によるものです。このポストフィックスを削除し、hgignoreの代わりにgitignoreファイルを作成する必要があります。これらすべてが履歴を追加するため、リポジトリのサイズが大きくなります（そして不当に）。別の例として、以下を引用できます。リポジトリを作成して、最初のコミットをコードを含めて300Mにプッシュします。次に、それをgitignoreに追加し、それなしでプッシュします。彼は歴史に残るでしょう。次に、履歴から削除してみます（git filter-branch）。特定の数のコミットを使用すると、結果のリポジトリサイズは減少せず、増加します。これは最適化によって解決されますが、bitbucketでは開始できません。リポジトリのインポート時のみ実行されます。したがって、すべての大まかな操作は中間で実行され、新しい操作へのインポートが行われます。ファイナルの中間リポジトリのサイズは1.15Gで、結果の350Mのサイズです。 

結論


移行プロセス全体は、次のようないくつかの段階に分かれています。

  • 準備（1つのビルドの例を使用した顧客へのデモ、ソフトウェアのインストール、リポジトリの変換、既存のccnet構成の更新）;
  • 開発環境とそのテストのためのCI / CD構成（古いシステムは並行して動作し続けます）。
  • 残りの環境のCI / CD設定（既存の「良好」と並行）およびテスト。
  • 移行開発、ステージング、テスト。
  • 本番環境を移行し、ドメインを古いIISインスタンスから新しいインスタンスに移行します。

正常に完了した準備フェーズの説明は現在完了中です。既存のシステムが壊れず、gitのmercurialリポジトリが移行されたことを除いて、ここでは大きな成功は得られませんでした。ただし、これらのプロセスの説明がなければ、より技術的な部分のコンテキストはありません。次のパートでは、チームシティに基づいてCIシステムをセットアップするプロセスについて説明します。
