Apache Bigtopと今日のHadoopディストリビューションの選択



昨年がApache Hadoopにとって大きな変化の年であったことはおそらく秘密ではありません。昨年、ClouderaとHortonworksは合併し（本質的には2番目の買収）、Maprは深刻な財政問題のためHewlett Packardに売却されました。そして、数年前であれば、オンプレミスのインストールの場合、ClouderaとHortonworksの間で選択を行わなければならないことが多かったのですが、残念ながら、今日、この選択はありませんでした。もう1つの驚きは、Clouderaが今年の2月以降、そのディストリビューションのバイナリアセンブリのパブリックリポジトリへのリリースの終了を発表し、現在は有料サブスクリプションでのみ利用できるという事実でした。もちろん、2019年末までにリリースされたCDHおよびHDPの最新バージョンをダウンロードする機能はまだあり、それらのサポートは1〜2年間期待されます。しかし、次に何をすべきか？それらのための、以前にサブスクリプションの支払いをした人は、何も変わっていません。また、有料版の配布キットに切り替えたくないが、クラスターコンポーネントの最新バージョン、パッチ、その他のアップデートを入手できるようにしたい方のために、この記事を用意しました。その中で、私たちはこの状況から可能な方法を検討します。

Arenadata Hadoop




これは完全に新しく、今のところあまり知られていない国内開発の分布です。残念ながら、現時点ではHabréについてはこの記事しかありません。

詳細については、プロジェクトの公式ウェブサイトをご覧ください。最新のディストリビューションは、バージョン3のHadoop 3.1.2、およびバージョン2の2.8.5に基づいています。

ロードマップの情報はここにあります。


Arenadata Cluster ManagerインターフェイスArenadataの

主要製品はArenadata Cluster Manager（ADCM）です。、会社のさまざまなソフトウェアソリューションをインストール、構成、監視するために使用されます。 ADCMは無料で、その機能は、それにansible-playbookのセットであるバンドルを追加することによって拡張されます。バンドルは、エンタープライズとコミュニティの2つのタイプに分けられます。後者はArenadataから無料でダウンロードできます。独自のバンドルを開発してADCMに接続することもできます。

Hadoop 3の導入と管理には、ADCMと組み合わせたコミュニティバージョンのバンドルが提供され、hadoop 2にはApache Ambariのみが存在します代替として。パッケージを含むリポジトリに関しては、それらはパブリックアクセス用に開かれており、すべてのクラスターコンポーネントに対して通常の方法でダウンロードおよびインストールできます。一般的に、このディストリビューションは非常に興味深いものです。Cloudera ManagerやAmbariなどのソリューションに慣れていて、ADCM自体が好きな人もいると思います。一部の人にとって、配布キットがインポート置換ソフトウェアレジストリに含まれているという事実は、大きなプラスになるでしょう。

短所について話すと、他のすべてのHadoopディストリビューションと同じです。つまり：

  • いわゆる「ベンダーロックイン」。ClouderaとHortonworksの例を使用して、会社のポリシーを変更するリスクが常にあることにすでに気づきました。
  • Apacheアップストリームに大幅な遅れ。

バニラハドゥープ




ご存知のように、Hadoopはモノリシック製品ではありませんが、実際には、分散HDFSファイルシステムに関連するサービスの全銀河です。 1つのファイルクラスタを必要とする人はほとんどいません。 1つはHiveを必要とし、もう1つはPrestoを必要とします。HBaseとPhoenixがあり、Sparkがますます使用されています。 Oozie、Sqoop、およびFlumeは、データを編成してダウンロードすることがわかっています。また、セキュリティの問題が発生した場合は、レンジャーと連携したKerberosがすぐに記憶されます。

Hadoopコンポーネントのバイナリバージョンは、各エコシステムプロジェクトのWebサイトでtarballの形式で入手できます。それらはダウンロードしてインストールを開始できますが、1つの条件があります。実行する可能性が最も高い「未加工」バイナリからのパッケージの自己アセンブリに加えて、ダウンロードしたコンポーネントのバージョン同士の互換性に自信がありません。推奨されるオプションは、Apache Bigtopを使用してビルドすることです。Bigtopを使用すると、ApacheのMavenリポジトリからビルドし、テストを実行して、パッケージをビルドできます。しかし、これは私たちにとって非常に重要です。Bigtopは、相互に互換性のあるコンポーネントのバージョンを収集します。以下で詳しく説明します。

Apache bigtop




Apache Bigtopは
、HadoopやGreenplumなどの多くのオープンソースプロジェクトを構築、パッケージ化、テストするためのツールです。 Bigtopには多くの
リリースがあります。執筆時点では、最新の安定版リリースはバージョン1.4で
、マスター版は1.5でした。異なるバージョンのリリースは、異なるバージョンの
コンポーネントを使用します。たとえば、1.4の場合、Hadoopコアコンポーネントはバージョン2.8.5で、マスターは
2.10.0です。サポートされるコンポーネントの構成も変更されています。古くて
再生不可能なものが残されており、その代わりに新しいもの、より需要の
あるもの、そして必ずしもApache自体のファミリーからのものではありません。

Bigtopには多くのフォークもあります。

Bigtopに慣れ始めたとき、他のApacheプロジェクト、普及率、名声、および非常に小さなコミュニティと比較して、その素朴さに主に驚きました。したがって、最小限の製品情報があり、フォーラムやニュースレターを通じて発生した問題の解決策を検索しても、何も生成されない可能性があります。最初は、ツール自体の機能により、配布キットの組み立てを完了するのは難しい作業でしたが、これについては後で説明します。

ティーザーとして、GentooやLFSなどのLinuxユニバースプロジェクトに一度アクセスしたことがある人にとっては、このことで作業し、私たち自身がebuildを検索（または作成）したときの「昔」の時間を思い出すのは懐かしいかもしれません。そして定期的に新しいパッチmozillaで再構築されました。

Bigtopの大きなプラスは、Bigtopのベースとなっているツールのオープン性と多様性と考えることができます。その基盤はGradleとApache Mavenです。Gradleは、GoogleがAndroidを収集するためのツールとしてかなり有名です。それは柔軟性があり、彼らが言うように、「戦闘でテストされました」。MavenはApache自体でプロジェクトを構築するためのフルタイムのツールであり、その製品のほとんどはMavenを通じてリリースされているため、それなしでは実現できません。POM（プロジェクトオブジェクトモデル）に注意を払う価値があります。これは、Mavenがプロジェクトで作業するために必要なすべてのものが記述された「基本的な」xmlファイルであり、その周りにすべての作業が構築されます。それはである
通常、彼らはBIGTOPを取る初めて遭遇したいくつかの障害が生じるとMavenの一部。

環境の準備


そして、ここで小さな余談が必要です。多かれ少なかれ複雑な製品をビルドするには、特定の環境が必要です-私たちの場合、それはJDK、同じ共有ライブラリ、ヘッダーファイルなど、ツール、たとえばant、ivy2などです。Bigtopに必要な環境を取得するためのオプションの1つは、必要なコンポーネントをアセンブリホストにインストールすることです。私は年代順に誤解しているかもしれませんが、バージョン1.0から、事前設定されたアクセス可能なDockerイメージにビルドオプションもあったようですが、ここで見つけることができます。

環境の準備に関しては、このためのアシスタントがあります-パペット。

次のコマンドを使用できます。起動は
ツールのルートディレクトリから行われます。./bigtop:

./gradlew toolchain
./gradlew toolchain-devtools
./gradlew toolchain-puppetmodules

または直接人形を介して：

puppet apply --modulepath=<path_to_bigtop> -e "include bigtop_toolchain::installer"
puppet apply --modulepath=<path_to_bigtop> -e "include bigtop_toolchain::deployment-tools"
puppet apply --modulepath=<path_to_bigtop> -e "include bigtop_toolchain::development-tools"

残念ながら、この段階ではすでに問題が発生する可能性があります。ここでの一般的なアドバイスは、ビルドホストで最新のサポートされているディストリビューションを使用するか、Dockerでパスを試すことです。

アセンブリ


何を集めようとしますか？この質問に対する答えは、コマンドの出力を提供します

./gradlew tasks

パッケージタスクセクションには、Bigtopの最終成果物である製品がいくつかあります。
これらは、接尾辞-rpmまたは-pkg-ind（dockerでのアセンブリの場合）で識別できます
。私たちの場合、最も興味深いのはHadoopです。

ビルドサーバーの環境でビルドしてみましょう。

./gradlew hadoop-rpm

Bigtop自体が、特定のコンポーネントに必要なソースをダウンロードし、ビルドを開始します。したがって、このツールはMavenリポジトリやその他のソースに関連付けられています。つまり、このツールにはインターネットアクセスが必要です。

このプロセスでは、標準出力が形成されます。時々あなたはそれとエラーメッセージから何がうまくいかなかったかを理解することができます。そして、時にはあなたはより多くの情報を必要とします。この場合、引数--info orを追加する価値--debugがあり、便利な場合もあります–stacktrace。後でメーリングリストを参照するためのデータセットを生成する便利な方法があります --scan。

これにより、bigtopはすべての情報を収集してグラドルに配置し、その後リンクを提供
します。その後、有能な人物がアセンブリが失敗した理由を理解できます。
このオプションを使用すると、ユーザー名、ノード、環境変数などの情報が一般に公開されなくなる可能性があるので注意してください。

多くの場合、エラーは、アセンブリに必要なコンポーネントを取得できないことが原因です。原則として、ソースの何かを修正するパッチを作成することで問題を修正できます。たとえば、ソースのルートディレクトリにあるpom.xmlのアドレスです。これは、適切な ./bigtop/bigtop-packages/src/common/oozie/パッチディレクトリ（たとえば、patch2-fix.diffの形式）に作成して配置することによって行われます。

--- a/pom.xml
+++ b/pom.xml
@@ -136,7 +136,7 @@
<repositories>
<repository>
<id>central</id>
- <url>http://repo1.maven.org/maven2</url>
+ <url>https://repo1.maven.org/maven2</url>
<snapshots>
<enabled>false</enabled>
</snapshots>

ほとんどの場合、この記事を読んだ時点で、上記の修正を自分で行う必要はありません。

ビルドはCentOSで行われましたが、Ubuntuで行うこともできます。

./gradlew -POS=ubuntu-16.04 -Pprefix=1.2.1 hadoop-pkg-ind

さまざまなLinuxディストリビューション用のパッケージを組み立てるだけでなく、ツールは組み立てられたパッケージを含むリポジトリを作成できます。たとえば、

./gradlew yum

Dockerでの煙のテストと展開も思い出してください。

3つのノードのクラスターを作成します。

./gradlew -Pnum_instances=3 docker-provisioner

3つのノードのクラスターで煙のテストを実行します。

./gradlew -Pnum_instances=3 -Prun_smoke_tests docker-provisioner

クラスターを削除：

./gradlew docker-provisioner-destroy

Dockerコンテナー内に接続するためのコマンドを取得します。

./gradlew docker-provisioner-ssh

ステータスを表示：

./gradlew docker-provisioner-status

ドキュメントで展開タスクの詳細を確認できます。

テストについて話すと、それらの多くは、主にスモークと統合テストです。彼らの分析はこの記事の範囲を超えています。一見すると、配布キットの作成はそれほど難しくないと言えます。製品で使用するすべてのコンポーネントを収集して渡し、それらをテストして管理することができました。また、テスト環境でのそれらの展開と基本的な操作に問題はありませんでした。

Bigtopの既存のコンポーネントに加えて、独自のソフトウェア開発であっても、何かを追加することができます。これらはすべて完全に自動化されており、CI / CDのコンセプトに適合しています。

結論


もちろん、この方法で構築されたディストリビューションをすぐに本番環境に送信するべきではありません。ディストリビューションを構築して維持することが本当に必要な場合は、資金と時間を投資する必要があることを理解する必要があります。

それにもかかわらず、適切なアプローチと専門家チームとの組み合わせで、商用ソリューションなしで実行することはかなり可能です。

Bigtopプロジェクト自体を開発する必要があり、今日では活発な開発は行われていないようです。また、Hadoop 3の外観の見通しは明確ではありません。ちなみに、Hadoop 3を構築する必要がある場合は、標準 コンポーネントに加えていくつかの追加コンポーネント（Ranger、Knox、NiFi）があるArenadata のフォークを見ることができます
。

Rostelecomに関しては、Bigtopは今日検討されているオプションの1つです。私たちがそれを止めるかどうかに関係なく、時間がわかります。

付録


アセンブリに新しいコンポーネントを含めるには、bigtop.bomおよび./bigtop-packagesにその説明を追加する必要があります。既存のコンポーネントとの類推によってこれを試すことができます。それを理解してみてください。一見思えるほど難しくありません。
