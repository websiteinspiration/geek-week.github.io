tecnología aprox .: QATOK # 2



El 5 de marzo, en la oficina de Odnoklassniki en San Petersburgo, se llevará a cabo aproximadamente Tech: QATOK # 2 , una reunión dedicada a las pruebas y todo lo relacionado con el aseguramiento de la calidad en proyectos de TI. Estamos a la espera de 3 informes técnicos de representantes de OK, Yandex y VK, los chicos hablarán sobre los problemas de las autoevaluaciones Espresso-Android en el mundo real, compartirán su experiencia usando Openapi-generator para escribir autoevaluaciones en REST-API y mostrarán cómo evaluar la cobertura de la prueba API sin eliminar el Código overage.

Regístrese en ok.tech: QATOK # 2

Resolviendo los problemas de las pruebas automáticas Espresso-Android en el mundo real -
Alexey Tyurin, VKontakte



A menudo, los proyectos que ya han ganado impulso y tienen una gran base de código, llegan a la conclusión de que necesita escribir pruebas automáticas. Incluyendo UI-autotests. En esta situación, no tiene la oportunidad de rehacer significativamente la arquitectura del proyecto para las pruebas. Como resultado, las pruebas creadas se vuelven inestables y distraen al equipo de desarrollo de productos más de lo que lo ayudan.
El propósito del informe: mostrar a las personas cómo se resuelven problemas específicos en la automatización de las pruebas de IU de una aplicación de Android (caso por caso). Además, estas soluciones no requieren reelaborar la arquitectura de la aplicación ni ninguna revisión significativa para las pruebas automáticas.

Evaluar la cobertura de la prueba API sin eliminar la cobertura del código -
Sergey Alekseev, Classmates



API es uno de los servicios más importantes en Odnoklassniki, se requiere para el trabajo de plataformas de usuario y una gran cantidad de socios externos, por lo tanto, probar API es una tarea primordial que requiere atención especial y muchas soluciones no estándar.
En el informe, hablaré sobre cómo se organizan las pruebas de API en Odnoklassniki, compartiré mi experiencia en el desarrollo de una herramienta interna que nos ayuda a encontrar puntos descubiertos / débiles en las pruebas automáticas de API, luego mostraré cómo puede armar un analizador para cubrir los métodos de API con pruebas automáticas para su infraestructura para requisitos personalizados, y al final prestaré especial atención a los enfoques para la recopilación y el análisis de la cobertura de prueba.

Cómo usar el generador OpenAPI para probar la API REST -
Viktor Orlovsky, Yandex.



En mi informe, le diré cómo utilizamos el generador de Openapi en Yandex. Las verticales para escribir pruebas automáticas en la REST-API, compartir mi experiencia de cómo podemos cubrir rápida y fácilmente una gran cantidad de API de servicios nuevos con pruebas automáticas, y también prestar atención a qué problemas encontramos en el camino y cómo resolverlos. 
Pasaremos de las pruebas heredadas en el cliente http a la API a la generación de pruebas reales basadas en la especificación.
