新しいプロジェクトでRobot Frameworkを採用した理由

最近、プロジェクトを変更しました-新しい開発に行きました。以前は、手動でも自動でもテストはありませんでした。お客様はツールキットに条件を課さなかった（それがPythonであることを除いて）ので、私は自分で選択しました。この記事では、このような状況でRobot Frameworkを選択した理由を説明します。そして最後に、私たちが話していることを説明する記事のために特別に書かれたいくつかの例があります。



私は10年以上にわたってテストの自動化を行っており、そのうちの約3つはRobot Frameworkと対話しました。

上記で述べたように、少し前に新しいプロジェクトに着手し、自動化を最初からテストし始めました。プロジェクトについてお話しすることはできません-NDA。これはクールな自動化ツールであり、将来的には多くの人的資源を節約するはずです。マイクロサービスから構築されています。これまでのところ、私の仕事はそのうちの4つに関係していますが、将来的には自分の活動を他の人にも広げます-それはすべて、私が1日8営業時間しかないという事実に帰着します。すべてに時間がかかります。

このプロジェクトの場合のテストの欠如は、推奨ツールの欠如と同じでした。Pythonスタックの一部として、私には選択の自由がありました。そして私はそれを利用しました。

ロボットフレームワークを選ぶ理由


私はRobot Frameworkのファンに帰することができると思います。ほとんどすべてのことがそれでできるように私には思えます。

他のプロジェクトでは、同社はpytestを使用しています。しかし、私の意見では、Pythonの機能によって制限されるため、そのような機能はありません。あなたはPythonコードを書くだけで、そのような変数に特定の値が期待されることに注意してください。私の好みでは、シンプルですが抽象的すぎます。多くのものが開発者に引き渡され、開発者はすべての便利な機能を手動で追加する必要があります。Robot Frameworkはそれを独自に行います。ここでは、プロジェクトに取り組む4つのポイントを示します。

柔軟性


Robot Framework自体はPythonで記述されており（つまり、Pythonで可能なすべてのことを実行できます）、そのためにコミュニティによって作成された多くのライブラリがあり、解決すべき問題の範囲が大幅に拡大されています。

Robot Frameworkの下では、ライブラリを簡単に作成し、ほとんどすべてのものと統合できます。すべてがすぐに機能します。松葉杖や自転車は必要ありません。
さらに、Robot Frameworkを使用すると、パラメトリックテスト（テンプレート）を記述できます。これにより、自動化ツールの作業が大幅に高速化されます。

テストにおけるロシア語


基本的なポイントの1つは、キーワード（Robot Frameworkの従来のメソッドのアナログであり、歴史的に何らかの理由で呼び出されます）はロシア語で記述できることです。結果として、テストテキストはもちろん、文学的言語からはかけ離れていますが、一見するとそこで何が起こっているのかが明らかになります。これはまず第一に、開発者を終わらせるためではなく、たとえば利害関係者にとって必要です。彼らは独立してテストを開き、そこで何が起こるかを正確に見ることができます：「からランダム要素を選択してください...」

タグ付け


Robot Frameworkはタグとうまく連動します。特定のタグを使用してテストを実行できるだけでなく、その助けを借りてレポートを分析することもできます。ログファイルには、タグの統計が含まれており、タグの配置を事前に検討した場合、それに基づいてアクションを実行できます。

便利なことに、各テストですべてのタグを長い布に書く必要はありませんが、テストをツリー構造に埋め込むことができます。私の場合、このように見えます。最初のレベルはマイクロサービス、2番目はテストのタイプ、3番目はテスト自体です（内部に埋め込まれているもののタグを含むinitファイルを置くと便利です）。

各マイクロサービス、各テストアプローチ（煙のテスト、CRUDテスト、統合など）のタグを作成しました。特定のタイプのテストまたは特定のサービスのみのテストを実行できます。関数のタイプにもタグを付けます-リストまたは詳細。また、リスト機能の製品が製品で「機能しない」場合、このタグが付いているすべてのテストは、それらがどこにあるか、何に関連しているかに関係なく、赤に変わります。

タグにより、自動テストをJiraにバインドします。トラッカーからのバグが閉じて、赤からのテストが緑に変わっても、正確に何が変わったのかについてのストーリーがあります。数か月後、テストが再び赤くなれば、前回問題を修正するためにどのような手順がとられたかを確認でき、それがバグの再現につながったと想定することさえできます。

重要でないバグのために追加した別の特別なタグ。GitLabでは、少なくとも1つのテストがクラッシュした場合、アセンブリで何かを行うことはできません。これは論理的です。すべてのバグが修正されるまで、製品をリリースすることも、週ごとのスプリントをリリースすることもできません。ただし、優先度は低く、バグはわずかです。具体的には、これらのテスト（このタグを使用したテスト）に合格しなかった場合にRobot Frameworkがアセンブリ全体をドロップしないようにするタグを選択しました。

素晴らしいログ


ログ分析はテストの不可欠な部分です。テストの実行時に何が起こっても、Robot Frameworkは完全にすべてを書き込みます。データベースに接続するためにログからログインとパスワードを隠す特別なラッパーを書かなければならないところまで。

このような詳細は、テストがクラッシュする理由をより早く理解するのに役立ちます。テスト中のシステムが正しく機能していないのか、それともテストで考慮されていないので修正する必要があるのか​​。この質問に対する答えは必ずしも明白ではありません。開発者とテスターのエラーは50/50に分散されます。
他のツール-同じpytest-で、同じ詳細ログを生成できます。しかし、それを生成するタスクは、テストを作成する開発者にあります。彼は、どのような種類のログエントリが本当に必要かについて考える必要があります。

現在プロジェクトにあるもの


Robot Frameworkを使い始めてから数か月が経ちました。現時点では、Robot Frameworkは200以上のテストを実装しており、前述のように、GitLabとの統合があり、開発された製品に加えられた変更を検証するのに役立ちます（自動テストをそれらにバインドできる識別子を持つテストケースはテストレールに保存されます）。

カバレッジを計算するために、SwaggerからバックエンドAPIのリストを取得して、テストされたものと比較するユーティリティを作成しました。したがって、現在のカバレッジを明確に理解しています。たとえば、今日、このマイクロサービスは完全にカバーされており、もう1つは98％カバーされています。また、テストにまだ反映されていない新しい機能を追加すると、カバレッジが低下します。したがって、私が実行する次のスプリントを計画することができます。

当然、プロジェクトはさらに発展しています。たとえば、今はモックサーバーをデプロイしています（それが何で、なぜ必要なのかについては、同僚がHabréで既に書いています）。

今練習する


上記の例は、上記のアイデアを説明するために記事のために特別に設計されています。これらの例を自分で試してみたい場合は、すべてリポジトリに投稿されています。

最も単純なテストの例


Robot Frameworkの最も単純なテストから始めましょう。

以下の例では、最初に特定の明らかにアクセス可能なリソース（この場合はen.wikipedia.org/wiki）に対してセッションが作成されます。Get root（/）を呼び出すことで、ステータスコード200を確認します。

*** Settings ***
Documentation        smoke-.
Library             RequestsLibrary

*** Variables ***
${base_url}         https://en.wikipedia.org/wiki
${url}              /

*** Test Cases ***
  Wiki
                    Create session     conn     ${base_url}    disable_warnings=1
    ${response}     Get request        conn     ${url}
                    Delete all sessions
                    Should be equal    ${response.status_code}    ${200}

ログのタグ付けと読み取り


最も単純なテストを改善し続けます。

テンプレートを単一のエントリポイントとして使用すると、科学者の誕生年の検証を簡単に追加できます。このようにして、読み込まれたページに正しいデータが表示されていることを確認します。

さらに、以下の例では、既存のキーワードをロシア語の名前で囲みました-私の好みのため、テストはより有機的に読みます。テスターとして、私は常に「まるで英語」と呼ばれる方法に悩まされていましたが、完全に読み書きができませんでした。いつもあなたが知っている言語で書く方が良いです。

...                  .

Test Setup           
Test Teardown         
Test Template       Smoke-

*** Variables ***

*** Test Cases ***

                    Should contain    ${text}    ${expected_word}    msg=    ${expected_word}!

に注意してください[Tags]。ここにタグを追加できます。これは、上で書いたように、レポートレベルで問題を評価するのに役立ちます。同様に、Force Tags __ init __。Robotファイル（リポジトリの例を参照）では、ネストされたものを含め、ディレクトリ内のすべてのテストにタグを設定できます。タグ付けが正しく実行され、テスト自体の名前を読み取らず、ロジックにクロールしない場合、テストプロジェクトでは機能しないとかなり正確に推測できます。

これらのテストの開始に関するレポートを見てください。明確にするために、エラーを見つけるテストを追加しました。

レポート統計はその最も重要な部分です。



この例では、タグ付きのテストnumbers（1に1つありますが、実際には、たとえば20のうち17になります）。このページに問題があると考えられます。

タグは、テストを選択的に実行するのに役立ちます。起動行に特定のタグを使用してすべてのテストを実行するには、以下を指定する必要があります。

--include <tag>

タグを使用した論理演算もサポートされています。 

-- include <tag>AND<tag>

たとえば、numbersタグを使用したテストでSmokeテストのみを実行する場合は、次を追加する必要があります。 

--include smokeANDnumbers

重要でないテスト


作業を大幅に簡略化するトリックに移りましょう。

テストにタグを付けたら、1つ以上のタグを「非クリティカル」として定義できます。そのようなタグを使用したテストでは、エラーがあれば表示されますが、最終的にはこれらのエラーは「無効」とは解釈されません。このオプションは、いくつかのマイナーなバグが考慮され、バグトラッカーに入力されたが、まだ修正されていない場合に使用します。これがないと、CIに含まれている自動テストでは、そのような既知の問題が検出されると、プロジェクトをアセンブルできなくなり、必ずしも便利ではありません。

新しいテストを追加します。

    (   )
    [Tags]              Letters    Known
    /abcdefghi          1799

起動時に、追加されたタグは次のキーを使用して「非クリティカル」として定義されます。 

--noncritical Known

Pythonのキーワード


次の例は、ライブラリを追加する方法を示しています。

「数値の配列を生成する」という新しいキーワードを作成します。その目的は明白です（このため、私はロシア語の名前が大好きです）。

from random import randint
from typing import List

from robot.api.deco import keyword

class ArrayGeneratorLibrary:
    ROBOT_LIBRARY_SCOPE = 'GLOBAL'

    @keyword("  ")
    def generate_array(self, length: int, minimal: int, maximal: int) -> List[int]:
        result = []
        for i in range(int(length)):
            result.append(randint(int(minimal), int(maximal)))
        return result

テストでライブラリを接続します。

Library             libraries.ArrayGeneratorLibrary

そしてそれを使う：

  ,   python.
    ${array}              ${5}    ${2}    ${8}
    Log to console    ${array}

ネストされた構造がある場合は、例のようにドットで区切る必要があることを忘れないでください。
別のトリック：渡された数値は${}文字列としてではなく、intとして扱われます！

インライン引数


もう1つの優れた点は、組み込みの引数です。これは、通常のように、呼び出しの最後ではなく、本体で直接渡されます。

説明のために、上記で作成した配列ジェネレーターのラッパーを作成します。これにより、組み込み引数を使用できます。

 ${n} ,  ${from}  ${to}
    ${result}             ${n}    ${from}    ${to}
    [Return]        ${result}

これで次のように書くことができます：

      .
    ${array}         5 ,  2  8
    Log to console    ${array}

メソッド名の一部の置換、Pythonの挿入とループ


Robot Frameworkについて私が本当に気に入っている次のトリックは、名前の一部を置き換えることです。2つの方法があるとします。1つは偶数を選択し、もう1つは奇数を選択します。

    
    [Arguments]     ${list}
    ${evens}        Evaluate    [i for i in $list if i % 2 == 0]
    [Return]        ${evens}

    
    [Arguments]     ${list}
    ${odds}         Evaluate    [i for i in $list if i % 2 != 0]
    [Return]        ${odds}

上記でEvaluate使用したキーワードを使用すると、Pythonコードの行を「ここに」実行できます。文字列の一部を変数のコンテンツで置き換えたくない場合、つまりリンクを渡したくない場合は、$中括弧なしで記号の直後に変数名を指定する必要があります。

したがって、両方のメソッドを呼び出して、名前の異なる部分を置き換えることができます。

    ,   . .
    ${types}        Create list        
    ${array}         5 ,  12  28
    FOR   ${type}   IN    @{types}
        ${numbers}      Run keyword     ${type}       ${array}
        log to console     ${numbers}
    END

メソッドデコレータ


はい、Robot Frameworkを使用すると、他のキーワードのデコレータを作成できます。

この可能性を説明するために、リストを返すメソッドの応答で負の数を選択するデコレーターを作成します。

    , 
    [Arguments]     ${keyword}    @{args}   &{kwargs}
    ${list}         Run keyword    ${keyword}    @{args}    &{kwargs}
    ${negs}         Evaluate    [i for i in $list if i < 0]
    [Return]        ${negs}

注意：
@{args}-これらはすべて名前のない引数です。
&{kwargs}すべて名前付き引数です。
この束があれば、リダイレクトして実際にデコレータを作成できます。
デコレーターの操作は次のようになります。

  
    ${negs}             ,       10  -5  5
    log to console  ${negs}
