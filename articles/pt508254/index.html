Uma visão prática dos testes



Mergulhei nesse problema por um tempo e tentei abordagens diferentes. A decisão veio após uma discussão memorável com JB Rainsberger .



Eu percebi que não há paradoxo. Existem diferentes definições de um "módulo".



Parece-me que a palavra "módulo" é confusa. As pessoas têm uma compreensão diferente do que é. Normalmente, os novatos nos testes consideram uma função ou método como um módulo. Então eles entendem que pode ser uma classe ou módulo inteiro. No entanto, a verdade, como muito mais na vida, depende da situação.









Testes isolados



Eu acredito que a divisão em testes "modulares" e "integração" não é clara o suficiente. Essa categorização leva a problemas e disputas, embora tenhamos um objetivo: facilitar a alteração do software com a ajuda de feedback rápido e preciso quando você quebra algo.



Portanto, prefiro falar sobre testes "isolados".



Não tenho certeza se este termo é melhor. Mas, por enquanto, gosto disso, porque me faz uma pergunta extremamente importante: isolada do que?



Minha resposta: isolado do que é difícil de testar.



O que é difícil de testar



Acidente, hora, rede, sistema de arquivos, bancos de dados, etc.



Eu chamo toda essa "infraestrutura", e o resto é chamado de "domínio". A combinação de domínio e infraestrutura parece muito útil e prática no design de software. E não só para mim. Essa separação está no centro de muitas arquiteturas de manutenção, como Arquitetura Hexagonal (conhecida como Portas e Adaptadores), Arquitetura Onion, Arquitetura Limpa e Núcleo Funcional / casca imperativa, etc.



Todas essas são as nuances que dizem uma coisa: é melhor separar o domínio e a infraestrutura. A programação funcional também promove essa idéia: efeitos colaterais isolados no perímetro do seu sistema. Efeitos colaterais são infraestrutura. E é difícil de testar.



Sim, mas como isso ajuda?



Em resumo, isso diz exatamente o que remover e substituir por plugues. Não transfira a lógica de negócios para a infraestrutura, caso contrário você será atormentado com o teste.



Um domínio isolado é fácil de testar. Você não precisa fazer stubs para outros objetos de domínio. Você pode usar o mesmo código usado no produto. Você só precisa se livrar da interação com a infraestrutura.



“ Mas pode haver erros no código de infraestrutura! »



Se você o mantiver compacto o suficiente, os riscos serão reduzidos. Mas, basicamente, você reduz o tamanho do código que precisa testar ao integrar com um banco de dados real.



A conclusão é a seguinte: você deve poder testar todos os tipos de comportamento de sua aplicação sem salvar dados no PostgreSQL. Precisa de um banco de dados na memória. Você ainda precisa verificar a integração com o PostgreSQL, mas apenas alguns testes são suficientes , e essa é uma diferença notável!



Era tudo uma teoria, voltemos à realidade: temos um código que ainda não foi testado.



Voltando ao código desatualizado



Eu acho que essa visão de teste irá ajudá-lo. Não há necessidade de se deixar levar pelos stubs, mas ao trabalhar com códigos desatualizados, você terá que se deixar levar. Temporariamente. Como esse código é um vinagrete, é difícil testar (infraestrutura) e lógica de negócios (domínio).



Você precisa se esforçar para extrair o código de infraestrutura do domínio.



Em seguida, torne a infraestrutura dependente do domínio, e não vice-versa . Deve haver o mínimo de infraestrutura possível na lógica de negócios. Você obterá testes rápidos, confiáveis ​​e isolados, cobrindo a maior parte da lógica do aplicativo e vários testes de integração para testar a operação do código de infraestrutura com um banco de dados real, API externa, relógio do sistema etc.



Você ainda tem stubs? Sim e não.



Você terá implementações de infraestrutura alternativas, para poder executar a lógica de produção sem usar um banco de dados real.



Como descobrir que a implementação do banco de dados está funcionando corretamente? Escreva alguns testes de integração. Apenas o suficiente para verificar a correção do comportamento. JB Rainsberger chama isso de "Testes de contrato". Eu acho que é melhor explicar com este exemplo .
