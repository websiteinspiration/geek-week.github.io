Kotlinの拡張機能。危険なアタビズムまたは便利なツール？



コトリンはまだ若い言語ですが、すでに私たちの生活に飛び込んできました。このため、これまたはその機能を正しく実装する方法、および適用するベストプラクティスが常に明確であるとは限りません。

Javaにはない言語の機能の場合は特に困難です。これらの障害の1つは拡張です。

これはコードを読みやすくする便利なツールであり、見返りにほとんど何も必要としません。しかし、同時に、少なくとも1人の人が、拡張が悪だと考えていなければ、間違いなく彼らに懐疑的であると知っています。以下では、論争や誤解を引き起こす可能性のあるこのメカニズムの機能について説明します。

DTOの拡張-データ転送オブジェクトテンプレートの違反


たとえば、クラスUserがあります。

class User(val name: String, val age: Int, val sex: String)

自分用のかなりのDTO！さらに、いくつかの場所のコードでは、ユーザーが大人であるかどうかを確認するためのチェックが必要です。最も簡単なオプションは、すべての場所で条件を付けることです

if (user.age >= 18) { ... }

ただし、そのような場所はいくつでも存在する可能性があるため、このチェックをメソッドに組み込むことは理にかなっています。
ここには3つのオプションがあります。

  1. 関数fun isAdult（ユーザー：ユーザー）-通常、ユーティリティクラスはそのような関数で構成されます。
  2. isAdult関数をUserクラス内に配置します。

    class User(val name: String, val age: Int, val sex: String) {
    fun isAdult() = age >= 18
}

  3. 同様の関数を含むUserのラッパーを記述します。

3つのオプションはすべて、技術的には生命の権利があります。しかし、最初のものは、すべてのユーティリティ関数を知る必要があるという形で不便さを追加しますが、これはもちろん、大きな問題ではありません。
クラスはゲッターとセッターだけではないため、2番目のオプションはデータ転送オブジェクトパターンに違反しているようです。しかし、パターンを壊すことは悪いことです。

3番目のオプションはOOPやテンプレートの原則に違反していませんが、同様の機能を使用したい場合は毎回ラッパーを作成する必要があります。このオプションもあまり好きではありません。結局、あなたはまだ犠牲を払わなければならないことがわかりました。

私の意見では、DTOテンプレートを犠牲にする方が簡単です。まず、DTOで関数（ゲッターとセッターを除く）を作成することが不可能である理由を1つも説明できませんでした。次に、意味の面から見ると、このようなコードは、操作対象のデータの隣に配置すると便利です。

ただし、開発者が作業するクラスを編集することが常に可能であるとは限らないため、DTOシェック内にそのようなコードを配置できるとは限りません。たとえば、これらはxsdから生成されたクラスである場合があります。さらに、誰かにとって、Dataクラスでそのようなコードを書くのは珍しく不快な場合があります。Kotlinは、関数と拡張フィールドの形式でそのような状況の解決策を提供します。

fun User.isAdult() = age >= 18

このコードは、Userクラス内で宣言されているかのように使用できます。

if(user.isAdult()) {...}

その結果、最小限の妥協で私たちのニーズを満たすかなりきちんとしたソリューションです。DTOテンプレートに違反しているという事実について話す場合、Javaでは通常の静的メソッドであることに注意してください。

public static final boolean isAdult(@NotNull User receiver)

ご覧のとおり、正式にはテンプレートにも違反していません。この関数の使用は、Userで宣言されているかのように見え、Ideaはオートコンプリートで提供します。とても快適です。

拡張機能は固有です。あなたはそれらの存在について知ることができず、拡張機能を持つエンティティのメソッドとフィールドを混乱させます


アイデアは、開発者がプロ​​ジェクトに来て、拡張機能で実装されたコードの周りで、どのメソッドがオリジナルでどれが拡張メソッドであるかは明確ではないということです。

これは問題ではありません。アイデアは開発者を支援し、そのような機能を強調するからです。公平を期して言えば、違いはDarculaテーマでより顕著であると言わなければなりません。ライトに変更すると、すべてがわかりにくくなり、拡張子はイタリック体のみが異なります。

以下に2つのメソッドを呼び出す例を示します。isAdultは拡張メソッド、isMaleはUserクラス内の通常のメソッドです。左側のスクリーンショットはDarculaテーマ、右側は通常のLightテーマです。



やや悪いのはフィールドです。たとえば、isAdultを拡張フィールドとして実装する場合、フォントタイプによってそれを通常のフィールドとのみ区別できます。この例では、名前は通常のフィールドです。拡張フィールドはイタリックフォントのみを生成します。



アイデア開発環境では、オートコンプリートの際に、どのメソッドが拡張機能であり、どのメソッドがオリジナルであるかを判断するのに役立ちます。快適です。



状況はフィールドでも同様です。



「<root>のユーザーの場合」は、拡張機能であることを意味します。

さらに、拡張メソッドとフィールドがオートコンプリート用に提案されているので、アイデアが拡張可能なエンティティに拡張を「バインド」するという事実は、開発に大いに役立ちます。

拡張機能はプロジェクト全体に散らばっており、ゴミ箱を形成しています


独自のデバイスに拡張機能をスローしたり、パブリック拡張機能を含むコードを別のファイルやパッケージに転送したりしないため、プロジェクトでこのような問題は発生しません。

たとえば、上記の例のisAdult関数は、拡張パッケージのUserファイルにあります。パッケージが十分ではなく、クラスの場所と関数ファイルの場所を正確に混同しないようにしたい場合は、_User.ktなどの名前を付けることができます。コレクションのJetBrainsの開発者もそうでした。または、良心がアンダースコアでのファイルの開始を禁止している場合は、user.ktを呼び出すことができます。実際、どちらの方法を使用しても違いはありません。主なことは、チーム全体が一貫している均一性があることです。

言語の作成者は、コレクションの拡張メソッドを開発するときに、それらを_Collections.ktファイルに配置しました。

これは一般的にコードの整理の問題であり、拡張機能の問題ではありません。Javaの静的関数は、静的関数だけでなく、拡張機能よりもランダムに分散させることができます。

単体テスト中に拡張機能を見落とさないでください


私の意見では、静的メソッドを濡らす必要がないのと同じように、拡張機能を濡らす必要はありません。拡張機能では、既存のデータを操作するロジックを配置する必要があります。たとえば、UserクラスのisAdult関数の場合、必要なものはすべてisAdultにあります。濡れる必要はありません。

もう少し複雑な例を考えてみましょう。外部システムからユーザーを取得するために役立つ特定のコンポーネント-UserComponentがあります。ユーザーを取得するためのメソッドは、getUsersと呼ばれます。すべてのアクティブユーザーを取得する必要があり、フィルタリングロジックを関数（拡張機能）の形式で追加することにしたと仮定します。その結果、次の関数が得られました。

fun UserComponent.getActiveUsers(): List<Users> = this.getUsers().filter{it.status == “Active”}

ここにあるように見えるかもしれません-拡張のためにモックが必要な状況。しかし、getActiveUsersが単なる静的メソッドであることを覚えている場合は、モックは必要ないことがわかります。Dipは、拡張機能で呼び出されるメソッドと関数でなければなりません。

拡張機能が拡張クラス内にある同名の機能と重複している可能性があります


最初の段落の例を使用して、このケースを検討します。ユーザーが大人かどうかをチェックする関数拡張isAdultがあるとします。

fun User.isAdult() = age >= 18

その後、User内に同じ名前の関数を実装します。

class User(val name: String, val age: Int, val sex: String){
    fun isAdult() = age >= 21
}

user.isAdult（）が呼び出されると、同じ名前の拡張と適切な関数があるにもかかわらず、クラスの関数が呼び出されます。クラス内で宣言された関数を認識していないユーザーは、拡張関数が完了するまで待機するため、このような場合は混乱する可能性があります。これは非常に深刻な結果をもたらす可能性のある不愉快な状況です。この場合、レビューまたはテンプレート違反の不便の可能性についてではなく、コードの誤った動作の可能性について説明しています。

上記の状況は、拡張機能を使用するときに、実際の問題が発生する可能性があることを示しています。

それらを避けるために、拡張機能を単体テストでできるだけカバーすることを忘れないでください。最悪の場合、テストが失敗すると、同じように機能する2つの関数が存在します。1つは拡張機能で、もう1つはクラス自体にあります。テストが失敗した場合、ある関数が別の関数と重複しているという事実に注意を向けます。

拡張機能はオブジェクトではなくクラスに関連付けられているため、混乱が生じる可能性があります


たとえば、最初の段落のUserクラスを考えてみます。それを開いて、後継のStudentを作成しましょう。

class Student(name: String, age: Int, sex: String): User(name, age, sex)

Studentの拡張関数を定義します。これは、学生が成人かどうかも決定します。学生のみが条件を変更します：

fun Student.isAdult() = this.age >= 16

そして今、私たちは次のコードを書きます：

val user: User = Student("", 17, "M")

user.isAdult（））は何を返しますか？
タイプStudentのオブジェクトと関数はtrueを返すように見えます。しかし、それはそれほど単純ではありません。拡張機能はオブジェクトではなくクラスにアタッチされ、結果はfalseになります。

拡張機能が静的メソッドであり、拡張可能なエンティティがこのメソッドの最初のパラメーターであることを思い出せば、これには奇妙なことは何もありません。これは、このメカニズムを使用するときに留意すべきもう1つのポイントです。そうしないと、不愉快で予期せぬ結果が生じる可能性があります。
