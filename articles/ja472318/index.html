Istioでの分散トレース

ある時点で、マイクロサービスアーキテクチャに基づいた本番システムを開発するときに、サービスの個々の要素を監視するだけでは深刻な問題に対処できないという結論に達しました。時間の経過とともに、リクエストトポロジ、ネットワーク遅延、個々のコマンドの継続時間に関する詳細情報とともに、アプリケーション全体のコールスタック全体の全体像を同時に把握する必要があります。この問題を解決するために、エンジニアは通常、分散トレースを使用します。

この投稿では、分散トレースの概念をマイクロサービスアーキテクチャのプリズムを通して検証します。Istioによってすべてが統合および自動化され、その後、プロセス全体がバックヤードを通じて簡略化および処理される方法 -Istioのサービス製品。

前書き


以前に、Istioをインストールして構成するだけでどれほど複雑になるかについて説明しました。そのため、この状況を簡略化するために、Banzai Cloud Istioのオープンソース開発を選択しました。 Istioに加えて、Helmで多くの便利なツール（Prometheus、Grfana、Jaeger、Kali）を使用できます。いずれの場合も、サポート/統合リクエストのほとんどはそれらからのものです。誰もが自分の好みに合わせてツールを選択できると信じているため、リストされているツールをIstioに厳密に実装するのではなく、これらのコンポーネントを簡単に統合する機能を提供しただけです。

さらに、クラスター内のリストされたコンポーネントを管理するための別の強力なツール（つまり、Istio以外のもの）をユーザーに提供したかったので、Backyardsも作成しました。これを使用すると、Prometheus、Grafana、Jaegerを簡単にインストールできます。また、Kialiとは異なり、Backyardsは単なるWebベースの監視インターフェイスではなく、サービスネットワークを管理するための本格的な多機能ツールです。一度に1つまたは複数のクラスターと互換性があり、強力なCLIおよびGraphQL APIが含まれています。

この投稿では、分散トレースとJaegerに焦点を当てます。

分散トレースの概要


マイクロサービスのアーキテクチャでは、複数のサービスが同時に相互に呼び出すと、問題を見つけてデバッグすることが困難になることがよくあります。実際、このステートメントには、サービスに対するリクエストの効率が低い主な理由があります。どのサービスがボトルネックになっているか、リクエスト間のネットワーク遅延の大きさはどれですか。

分散トレースのおかげで、完全な呼び出しツリーを視覚化し、どのサービスがどのサービスにアクセスしたか、各呼び出しにかかる時間、応答の遅延を確認できます。障害が発生した場所と、応答時間制限を超えたサービスを判別するのに役立つのは、分散トレースです。
分散トレースは、システム内のコールスタック全体にわたって個々の要求を追跡するプロセスです。


分散トレースを使用しないと、マイクロサービス間で交換される多くの呼び出しで迷うのは確かに簡単です。また、ログのみに依存する必要があるため、それなしでは問題をデバッグすることも困難です。しかし、分散トレースのおかげで、問題の診断に必要なすべての情報を含むコールスタック全体が利用可能になります。

分散トレースの内部


スパンは、分散トレースの最小単位です。彼女には「開始」と「期間」というラベルがあります。スパンは相互に親子関係を持ち、一緒にTraceを形成します。

トレースを正しく収集するための3つのタスクがあります

1. Incoming request spans
2. Outgoing request spans
3. Context propagation

受信リクエストスパン：
リクエストがサービスに到着したら、トレースヘッダーをチェックする必要があります。ヘッダーがない場合は、トレースを開始するためにルート（親）スパンを作成する必要があります。必要な場合は、子スパンを作成してトレースを続行します。

送信リクエストの範囲：
要求が1つのサービスから別のサービスに送信されると、スパンが最初に作成されるため、要求を受信する側は前述のように追跡を続けます。

コンテキストの伝播：
サービスは通常、一度に複数の要求を送受信します。変更を加えないと、着信要求と発信要求の間の接続を追跡することはできません。コンテキストを伝えるのに最適な時間です。 HTTPの場合、これは、着信呼び出しのトレースヘッダーをアプリケーションから発信呼び出しに渡すことで実行できます。これにより、完全なトレースを構築できます。

これらのタスクに加えて、トレースを収集、グループ化し、完全に視覚化する必要があります。

これらすべては、システム内のすべてのサービスに実装するには十分に複雑に見えるかもしれませんが、自動モードで説明されているほとんどすべてのタスクを解決できるツールがあります。その結果、開発者は実際に箱から出して分散トレースを取得できます。

Istio


Istioでは、Envoyプロキシが分散トレースの汚い作業を行います。 Istioにはサイドカーインジェクションがあります。これにより、サービスとの間のすべての着信および発信リクエストは、最初にEnvoyプロキシサーバーを通過します。この段階で、分散トレースを作成するために必要な3つの条件のうち最初の2つである親と娘のトレースが生成されます。

3番目の条件（コンテキストの転送）は、アプリケーションロジックの変更を通じて実行する必要があります。IstioのBookInfo、フルトレースは、インストール直後に得られるので、これは既に、各microserviceのレベルで実装されています。独自のサービスで分散トレースを使用する場合は、コンテキスト伝播を実装する必要があります自分で。

いくつかの一般的な誤解に反して、コンテキスト伝播による分散トレースの完全サポートは、通常のサービスグリッドのレベルでも、アプリケーションのロジックを変更することによってのみ作成できることに注意する価値があります。良い例が警告1に示されています。

Istio Webサイトには、使用するヘッダー（ヘッダー）に関するドキュメントが含まれています。すべてがZipkinヘッダー形式と互換性があります。

Istioでの仕組み


Jaegerを使用して分散トレースを行う方法を要約しましょう（後でこれを裏庭での同じプロセスと比較します）。

  1. Istioとともに、必要なすべてのサービスとデプロイメントを備えたJaegerを入手できます。
  2. トレースサービスのアドレスは、Envoyプロキシの構成に従ってデフォルトで設定され、スパンレポートとサイドカーサービスが機能します。
  3. Bookinfoは、すべてのサービスがヘッドヘッダーをトレースに配布する前にインストールできます。
  4. 十分な情報が収集されると、トレースが使用可能になります。
  5. Jaegerコントロールパネルをリダイレクトして、ブラウザーからユーザーパーツを開くことができます。
