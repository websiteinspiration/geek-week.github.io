This code launches a browser window and opens a page in it at the specified url. As a result, we get html code already with the desired data. But there is one snag. The result is only 100 entries, and the total number of races is almost 2000. How so? The fact is that initially only the first 100 entries are displayed in the browser, and only if you scroll to the very bottom of the page, the next 100 are loaded, and so on. Therefore, it is necessary to implement scrolling programmatically. To do this, use the command:

driver.execute_script("window.scrollTo(0, document.body.scrollHeight);")

And with each scrolling, we will check whether the code of the loaded page has changed or not. If it hasn‚Äôt changed, we‚Äôll check several times for reliability, for example 10, then the whole page is loaded and you can stop. Between the scrolls we set the timeout to one second so that the page has time to load. (Even if she doesn‚Äôt have time, we have a reserve - another nine seconds).

And the full code will look like this:

import pickle as pkl

def worker(args):
    names = args[0]
    ars=args[1]
    num=args[2]
    ars = ars.sort_values(by='name')
    ars = ars.reset_index(drop=True)  

    for n in names:   
        ind = ars[ars['name'] == n].index 
        yos = ars.loc[ind, 'year'].min()
        ars.loc[ind, 'exp'] = ars.loc[ind, 'year'] - yos + 1                      

    with open(r'D:\tri\par\prog' + str(num) + '.pkl', 'wb') as f:
        pkl.dump(ars,f)

The procedure is transferred to a portion of the names of names , part datafreyma ar only with these names, and the serial number of parallel tasks - num . Computations are written to the data frame and, at the end, the data frame is written to the file. In the laptop that calls this worker , we prepare the arguments accordingly:

However, I could not calm down and even after the publication I was constantly thinking about how to do the calculation using vectorization, that is, operations on whole columns of the pandas Series data frame. Such calculations are an order of magnitude faster than any parallelized cycles, even on a supercluster. And came up with. It turns out that for each name to find the year of the beginning of a career, it is necessary on the contrary - for each year to find the participants who started in it. To do this, you must first determine all the names for the first year from our sample, this is 2010. Accordingly, we process all records with these names using this year. Next, we take the next year - 2011.

Again we find all the names with entries this year, but we take from them only unprocessed ones, that is, those that were not met in 2010 and are processed using them in 2011. And so on for the rest of the year. The same cycle, but not two hundred thousand iterations, but nine in total.

for y in range(ars['year'].min(),ars['year'].max()):

Conclusion


Well, finally, a lot of work has been completed. For me, this was, in fact, the first project of its kind. When I took it, the main goal was to practice using python and its libraries. This task is more than completed. And the results themselves were quite presentable. What conclusions did I draw for myself upon completion?

First: Data is imperfect. This is probably true for almost any analysis task. Even if they are completely structured, and it often happens differently, you need to be prepared to tinker with them before you start calculating the characteristics and searching for trends - find errors, outliers, deviations from standards, etc.

Second:Any task has a solution. This is more like a slogan, but often it is. It‚Äôs just that this solution may not be so obvious and does not lie in the data itself, but outside of the box, so to speak. As an example - the processing of the names of participants described above, or website scraping.

Third: Domain knowledge is crucial. This will make it possible to better prepare the data, removing obviously invalid or non-standard data, avoid interpretation errors, use information that is not in the data, for example, distances in this project, present the results in the form accepted in the community, while avoiding stupid, incorrect conclusions.

Fourth: To work in pythonThere is a rich set of tools. Sometimes it seems that it is worth thinking about something, you begin to search - it already exists. This is just great! Many thanks to the creators for this contribution, especially for the tools that came in handy here: selenium for scraping, pycountry for determining the country code according to the ISO standard, country codes (datahub) for Olympic codes, geopy for determining the coordinates at the address, folium - for visualization of geodata, gender-guesser - for the analysis of names, multiprocessing - for parallel computing, matplotlib , numpy , and of course pandas - without it there‚Äôs nowhere to go.

Fifth: Vectorization is our everything. It is extremely important to be able to use the built-in pandas tools , it is very effective. I suppose, in most cases, when the number of records is measured starting from tens of thousands, this skill becomes simply necessary.

Sixth:Handling data is a bad idea. It is necessary to try to minimize any manual intervention - firstly, it does not scale, that is, when the amount of data increases several times, the time for manual processing will increase to unacceptable values, and secondly, there will be poor repeatability - you will forget something, you will make a mistake somewhere . Everything is only programmatic, if something falls out of the general standard for a software solution, well, it's okay, you can sacrifice some part of the data, there will still be more pluses.

Seventh:The code must be kept in working order. It would seem that it could be more obvious! In fact, when it comes to code for your own use, the purpose of which is to publish the results of this code, everything is not so strict here. I worked in Jupiter Notebooks, and this environment, in my opinion, just does not have to create integral software products. It is configured for line-by-line, piecewise launch, this has its advantages - it is fast: development, debugging and execution at the same time. But often the temptation is to just edit some line and quickly get a new result, instead of duplicating or wrapping in def. Of course, such a temptation should be avoided. One should strive for a good code, even ‚Äúfor oneself‚Äù, at least because even for one analysis work the launch is done many times, and investing time in the beginning will surely pay off in the future. And you can add tests, even on laptops, in the form of checks of critical parameters and throwing exceptions - it is very useful.

Eighth: Save more often. At each step, I saved a new version of the file. In total, they turned out to be about 10. This is convenient, since when an error is detected it helps to quickly determine at what stage it occurred. Plus, I saved the source data in the columns marked raw - this allows you to quickly check the result and see the discrepancy.

Ninth:It is necessary to measure the investment of time and the result. In some places, I took a very long time to restore data, which form a fraction of a percent of the total. In fact, this made no sense, you just had to throw them away, and that‚Äôs all. And I would do so if it were a commercial project, not an auto-training. This would allow you to get the result much faster. The Pareto principle works here - 80% of the result is achieved in 20% of the time.
