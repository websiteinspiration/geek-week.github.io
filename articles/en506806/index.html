MVP for Android - the benefits of using Moxy as a helper library

This article describes the benefits of using Moxy as a helper library when using MVP for an Android application.


Important: MVP is not compared with other architectural approaches like MVVM, MVI, etc. The article describes why, if MVP is chosen as the architecture of the UI part of the application, then it is better not to use the native implementation of MVP, but to use Moxy.


The Moxy library allows you to avoid boilerplate code for handling lifecycle fragments and activities, and work with View as if it is always active.
Further, View is understood as the implementation of View as a fragment or activity.
Interactor is understood as the essence of business logic, i.e. a class that is at a lower level of abstraction than Presenter.


Common Benefits of Moxy


  • Active support and development of the library.
  • Kotlin feature support like val presenter by moxyPresenter {component.myPresenter} and presenterScope for coroutine.
  • Automatically restore View state.
  • Automatic linking with the life cycle (and hence the absence of leaks of Activiti and other similar charms).
  • Views are accessed through a non-nullable viewState. There is no risk that any View command is lost.
