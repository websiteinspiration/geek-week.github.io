JVM内でのポリモーフィズムの実装方法

この記事の翻訳は、特にJava開発者コースの受講生向けに用意されています。




以前の記事「メソッドのオーバーロードとメソッドのオーバーライドについてのすべて」では、メソッドのオーバーロードとオーバーライドのルールと違いについて説明しました。この記事では、JVM内でメソッドのオーバーロードとオーバーライドがどのように処理されるかを説明します。

たとえば、前の記事のクラスである、親Mammal（哺乳類）と娘Human（人間）を取り上げます。

public class OverridingInternalExample {

    private static class Mammal {
        public void speak() { System.out.println("ohlllalalalalalaoaoaoa"); }
    }

    private static class Human extends Mammal {
        @Override
        public void speak() { System.out.println("Hello"); }

        //   speak()
        public void speak(String language) {
            if (language.equals("Hindi")) System.out.println("Namaste");
            else System.out.println("Hello");
        }

        @Override
        public String toString() { return "Human Class"; }
    }

    //           
    public static void main(String[] args) {
        Mammal anyMammal = new Mammal();
        anyMammal.speak();  // Output - ohlllalalalalalaoaoaoa
        // 10: invokevirtual #4 // Method org/programming/mitra/exercises/OverridingInternalExample$Mammal.speak:()V

        Mammal humanMammal = new Human();
        humanMammal.speak(); // Output - Hello
        // 23: invokevirtual #4 // Method org/programming/mitra/exercises/OverridingInternalExample$Mammal.speak:()V

        Human human = new Human();
        human.speak(); // Output - Hello
        // 36: invokevirtual #7 // Method org/programming/mitra/exercises/OverridingInternalExample$Human.speak:()V

        human.speak("Hindi"); // Output - Namaste
        // 42: invokevirtual #9 // Method org/programming/mitra/exercises/OverridingInternalExample$Human.speak:(Ljava/lang/String;)V
    }
}

多態性の問題は、「論理的」と「物理的」の2つの側面から見ることができます。まず、問題の論理的な側面を見てみましょう。

論理的な視点


論理的な観点から、コンパイル段階では、呼び出されたメソッドは参照のタイプに属していると見なされます。しかし、実行時に、参照されるオブジェクトのメソッドが呼び出されます。

たとえば、行では、humanMammal.speak();コンパイラーはとして宣言されMammal.speak()ているため、呼び出されるとhumanMammal見なしますMammal。しかし、実行時に、JVMはhumanMammalオブジェクトの内容を認識し、Human実際にメソッドを呼び出しますHuman.speak()。

概念的なレベルにある限り、それはすべて非常に単純です。しかし、JVMはこれをすべて内部的にどのように処理するのでしょうか。 JVMはどのメソッドを呼び出す必要があるかをどのように計算しますか？

オーバーロードされたメソッドはポリモーフィックと呼ばれず、コンパイル時に解決されることもわかっています。メソッドのオーバーロードが呼び出されることもありますがコンパイル時のポリモーフィズムまたは早期/静的バインディング。

オーバーライドされたメソッド（オーバーライド）は実行時に解決されます。これは、リンクに割り当てられているオブジェクトにオーバーライドされたメソッドがあるかどうかコンパイラーが認識しないためです。

物理的な視点


このセクションでは、上記のステートメントすべての「物理的な」証拠を見つけようとします。これを行うには、を実行して取得できるバイトコードを確認しますjavap -verbose OverridingInternalExample。パラメータ-verboseにより、Javaプログラムに対応するより直感的なバイトコードを取得できます。

上記のコマンドは、バイトコードの2つのセクションを表示します。

1.定数のプール。プログラムの実行に必要なほとんどすべてのものが含まれています。たとえば、メソッド（#Methodref）、クラス（#Class）、文字列リテラル（#String）への参照。



2.プログラムのバイトコード。実行可能なバイトコード命令。



メソッドのオーバーロードが静的バインディングと呼ばれる理由


上記の例では、コンパイラーはメソッドhumanMammal.speak()がクラスから呼び出されると考えていますMammalが、実行時には参照先のオブジェクトから呼び出されますhumanMammal-クラスのオブジェクトになりHumanます。

我々のコードと結果を見るとjavap、私たちは、メソッドを呼び出すためにそれを参照してくださいhumanMammal.speak()、human.speak()そしてhuman.speak("Hindi")異なるバイトコードを使用するため、コンパイラは、クラスを参照することによって、それらを区別することができます。

したがって、メソッドのオーバーロードが発生した場合、コンパイラーはコンパイル時にバイトコード命令とメソッドアドレスを識別できます。これが、静的リンケージまたはコンパイル時のポリモーフィズムと呼ばれる理由です。

メソッドのオーバーライドが動的バインディングと呼ばれる理由


メソッドを呼び出すためanyMammal.speak()、humanMammal.speak()コンパイラの観点から、両方の方法は、クラスのために呼ばれているので、バイトコードは、同じですMammal。

invokevirtual #4 // Method org/programming/mitra/exercises/OverridingInternalExample$Mammal.speak:()V

したがって、問題は、両方の呼び出しに同じバイトコードがある場合、JVMはどのメソッドを呼び出すかをどのようにして知るのでしょうか。

答えはバイトコード自体と指示に隠されていますinvokevirtual。JVM仕様によると（翻訳者注：JVM仕様2.11.8を参照）：
invokevirtual命令は、オブジェクトの（仮想）タイプでのディスパッチを通じてインスタンスメソッドを呼び出します。これは、Javaプログラミング言語でのメソッドの通常のディスパッチです。
JVMは、命令invokevirtualを使用して、C ++仮想メソッドに相当するJavaのメソッドを呼び出します。C ++では、別のクラスのメソッドをオーバーライドするには、メソッドを仮想として宣言する必要があります。ただし、Javaでは、デフォルトで、すべてのメソッドが仮想（finalメソッドとstaticメソッドを除く）であるため、子クラスで任意のメソッドをオーバーライドできます。

この命令invokevirtualは、呼び出されるメソッドへのポインタを取ります（＃4は定数プールのインデックスです）。

invokevirtual #4   // Method org/programming/mitra/exercises/OverridingInternalExample$Mammal.speak:()V

しかし、参照番号4はさらに別のメソッドとクラスを参照しています。

#4 = Methodref   #2.#27   // org/programming/mitra/exercises/OverridingInternalExample$Mammal.speak:()V
#2 = Class   #25   // org/programming/mitra/exercises/OverridingInternalExample$Mammal
#25 = Utf8   org/programming/mitra/exercises/OverridingInternalExample$Mammal
#27 = NameAndType   #35:#17   // speak:()V
#35 = Utf8   speak
#17 = Utf8  ()V

これらすべてのリンクを一緒に使用して、目的のメソッドが配置されているメソッドとクラスへの参照を取得します。これは、JVM仕様でも言及されています（翻訳者注：JVM仕様2.7への参照）。
Java仮想マシンは、オブジェクトの特定の内部構造を必要としません。
一部のOracle Virtual Machine Java実装では、クラスインスタンスへの参照はハンドラーへの参照であり、ハンドラ自体へのリンクで構成されています。1つはオブジェクトメソッドのテーブルを指し、もう1つはオブジェクトのタイプを表すClassオブジェクトへのポインタ、もう1つは領域へのポインタです。オブジェクトデータを含むヒープ上のデータ。

つまり、各参照変数には2つの非表示ポインターが含まれています。

  1. Classたとえば、オブジェクトのメソッドを含むテーブルへのポインタとオブジェクトへのポインタ[speak(), speak(String) Class object]
  2. オブジェクトフィールド値などのオブジェクトデータに割り当てられたヒープ上のメモリへのポインター。

しかし、再び疑問が生じます。それはどのように機能しinvokevirtualますか？残念ながら、この問題はすべてJVMの実装に依存し、JVMによって異なるため、誰もこの質問に答えることはできません。

上記の理由から、オブジェクトへの参照には、このオブジェクトのメソッドへのすべての参照を含むテーブルへのリンク/ポインターが間接的に含まれていると結論付けることができます。 JavaはC ++からこの概念を借用しました。このテーブルは、仮想メソッドテーブル（VMT）、仮想関数テーブル（vftable）、仮想テーブル（vtable）、ディスパッチテーブルなど、さまざまな名前で知られています。

特定のJVMに依存しているため、Javaでvtableがどのように実装されているかはわかりません。しかし、戦略はC ++とほぼ同じになると予想できます。vtableは、メソッド名とその参照を含む配列のような構造です。 JVMが仮想メソッドを実行しようとするたびに、JVMはvtableでそのアドレスを要求します。

各クラスには、vtableが1つしかありません。つまり、テーブルはクラスオブジェクトと同様に、クラスのすべてのオブジェクトに対して一意で同じです。クラスオブジェクトについては、「なぜ外部Javaクラスを静的にすることができないのか、なぜJavaが純粋にオブジェクト指向言語なのか、なぜそうでないのか」の記事で詳しく説明しています。

したがって、クラスのvtableは1つだけですObjectには、11のすべてのメソッド（考慮されていない場合registerNatives）と、それらの実装に対応するリンクが含まれています。



JVMがMammalクラスをメモリにロードすると、JVMはそのオブジェクトをClass作成Objectし、同じ参照を持つvtableクラスのすべてのメソッドを含むvtableを作成し（からMammalのメソッドをオーバーライドしないObjectため）、メソッドの新しいエントリを追加しますspeak()。



次に、クラスが入りHuman、JVMはすべてのエントリをvtableクラスMammalからvtableクラスにコピーHumanし、オーバーロードされたバージョンの新しいエントリを追加しますspeak(String)。

JVMは、クラスがいることを知っているHuman：2つのメソッドをオーバーライドするtoString()からObjectとspeak()からMammal。これらのメソッドでは、リンクが更新された新しいレコードを作成する代わりに、JVMは既存のメソッドへのリンクを、それらが以前存在していたのと同じインデックスに変更し、同じメソッド名を保持します。



この命令invokevirtualは、JVMにメソッド＃4への参照の値をアドレスとしてではなく、現在のオブジェクトのvtableで検索されるメソッドの名前として処理するように強制します。
JVMが定数プールと仮想メソッドテーブルを使用して、呼び出すメソッドを決定する方法がより明確になったと思います。
