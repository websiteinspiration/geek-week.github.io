これまでに遭遇したことのない、驚くほど狡猾なDDoS攻撃について説明します。油断できないのは、被害者のサーバー自体に対して攻撃が行われないことです。代わりに、攻撃者はサードパーティの攻撃検出システムの動作を引き起こし、サーバーに対して完全な実際の苦情（一般的な人々の「乱用」）を生成させます。

ホスティング事業者の側では、悪意のある活動に従事しているようですが、実際にはそうではありません。多くの大規模なホスティングプロバイダーは、問題の原因を深く理解する準備ができておらず、ルールを破ることを単に禁止することを好みます。

この記事では、実際のケースでこのタイプの攻撃について詳しく説明します。

使い方


着信RSTパケットは、プライベートポートへのTCP接続を確立しようとする試みに対する応答です。同時に、私のサーバーからRSTを送信するサーバーへの送信トラフィックはありません。これは、攻撃者が発信アドレスを自分のものに置き換え、DDoS攻撃と同様のトラフィックを生成することを意味します。しかし、攻撃者ができることは発信パケットを送信することだけなので、サーバーからのすべての応答は私のサーバーに届きます。


偽のリクエストに対する回答のみが被害者のサーバーに到達する
通常、このような攻撃はDNSの増幅に使用されます。攻撃者が被害者に代わって小さなリクエストを送信し、被害者がリクエストしていない大きな応答を受信した場合です。これは、チャネル枯渇攻撃の古典的なメカニズムです。
私の場合、攻撃者は被害者のサーバーのチャネルを使い果たしようとはしませんでした。その活動は、チャネルの消費量グラフでは完全に見えませんでした。その主な目的は、私のプロバイダーのwhoisサブネットで指定された乱用アドレスについて不平を言う手紙を送るネットワーク攻撃の自動通知のシステムを引き起こすことでした。SnortやSuricataなどの検出と侵入防止（侵入の防止/システムの検出）を実行できます。

その結果、ホスティング業者は合法的な会社から完全に本物の手紙を受け取ります。この手紙には、私の悪意のある活動に関する苦情と、その中の実際のログが含まれています。同時に、攻撃者はIDS / IPSシステムがインストールされているサーバーのアドレスと自動苦情が機能するために必要な最小パケット数を事前に知っているため、大きなチャネルは必要ありません。

攻撃者にとっての唯一の困難は、送信IPアドレスをパケットで置換できるサーバーを見つけることです。通常のホスティング業者はすべて、このようなパケットをブロックします。ホスティングには2つのタイプしかないため、クライアントは発信IPを置き換えることができます。非常に読み書きのできない構成、またはサイバー犯罪用に特別に作成されたもののいずれかです。

IPアドレスのなりすましの可能性の確認


発信IPの置換の可能性についてホストを確認することをお勧めします。これには、トラフィックの受信用と送信用の2つのサーバーが必要です。

受信サーバー側では、着信トラフィックのロギングを開始します。通常はトラフィックが発生しないはずのフィルターとして、まれなポートを指定します。

tcpdump -i any port 9912

テストするサーバー上で、送信IPアドレスを1.1.1.1に置き換えて、ポート9912に送信するパッケージを生成します。これには、nmap開発者からのクールなユーティリティnpingを使用します。L2およびL3レベルで非標準パッケージを生成できます。

nping --tcp -p 9912 -S 1.1.1.1 receiver-server.com

愚かな技術サポート



苦情の理由を理解したので、私はホスティング業者にすべてを詳しく説明しました：
Hello,
I finally understand what happened.

They spoof my IP address and DDoS random hosts using my address as source address. So victims generate automatic abuse reports to my hosting providers.

You can see on abuse log that connections are only in SYN_RECV state (no full TCP-connection established) because they can send only one packet using spoofed IP and can't finish TCP-handshake.
I can prove this. There are many TCP RST incoming packets coming right now to my server from hosts whom I never sent any packets.

You can check it right now by running:

tcpdump -i eth0 «tcp[tcpflags] & (tcp-rst) != 0»

You will see that many RST packets came from hosts to which I've never sent any data.
This proves that attacker is spoofing my IP-address to compromise me and generate abuses.

次に、資格のあるテクニカルサポートエンジニアが状況を理解し、質問は終了するように思えました。しかし、代わりに、ウイルス対策ソフトウェアでサーバーをチェックするか、OSを完全に再インストールするように要求しました。私たちが技術サポートに対応している間、虐待は続いて来ました、そして、一日で私は禁止されました。

私が実際に額装されたのは、非常に侮辱的でした。この状況は、理解するのではなく、スクリプトを軽率にフォローしている脆弱な人々がいかに存在するかを示しています。それ以来、私はこのケースを、重要なプロジェクトのホスティングを選択することについての議論が起こったときの例としてしばしば引用します。残念ながら、多くのホスティング会社は、小規模なクライアントの非標準的なケースに多くの時間を費やす余裕がなく、問題を整理するよりも禁止する方が簡単です。
