Yandex.MapsにWebAssemblyを実装する方法と、JavaScriptを残す理由

私はヴァレリーシャベルと申します。Yandex.Mapsベクトルエンジンの開発チームの出身です。最近、エンジンにWebAssemblyテクノロジーを導入しました。以下に、なぜそれを選択したのか、どのような結果が得られ、このテクノロジーをプロジェクトでどのように使用できるのかを説明します。



Yandex.Mapsでは、ベクトルマップはタイルと呼ばれる部分で構成されています。実際、タイルはマップのインデックス付き領域です。マップはベクトルであるため、各タイルには多くの幾何プリミティブが含まれています。タイルはサーバーからクライアントにエンコードされ、すべてのプリミティブは表示する前に処理する必要があります。かなりの時間がかかる場合があります。タイルには、2,000を超えるポリラインとポリゴンを含めることができます。



プリミティブを処理する場合、パフォーマンスが最も重要です。タイルが十分に速く準備されていない場合、ユーザーはそれを見るのが遅すぎて、次のタイルがキューで遅れます。処理を高速化するために、比較的新しいWebAssembly（Wasm）テクノロジーを試すことにしました。

マップでのWebAssemblyの使用


現在、プリミティブの処理のほとんどはバックグラウンドスレッド（Webワーカー）で行われ、別の人生を送っています。これは、メインスレッドをできるだけアンロードするために行われます。したがって、カードを表示するためのコードがサービスページに埋め込まれている場合、それ自体が大きな負荷になる可能性があり、ブレーキが少なくなります。欠点は、メインスレッドとWebワーカー間のメッセージングを適切に構成する必要があることです。

バックグラウンドスレッドで行われる処理の部分は、基本的に次の2つのステップで構成されます。

  1. サーバーからのprotobuf形式がデコードされます。
  2. ジオメトリが生成され、バッファに書き込まれます。

2番目のステップでは、WebGLの頂点バッファーとインデックスバッファーが形成されます。これらのバッファは、次のようにレンダリングするときに使用されます。頂点バッファーには、各頂点のパラメーターが含まれています。これらのパラメーターは、特定の瞬間の画面上の位置を決定するために必要です。インデックスバッファは、3つのインデックスで構成されます。各トリプルは、指定されたインデックスにある頂点バッファーからの頂点を持つ三角形が画面に表示されることを意味します。したがって、プリミティブを三角形に分割する必要があります。これも時間のかかる作業です。



明らかに、2番目のステップでは、プリミティブを正しくレンダリングするために、プリミティブの各頂点に関する多くの情報が必要になるため、多くのメモリ操作と数学的計算が行われます。



JavaScriptコードのパフォーマンスに満足できませんでした。この頃、誰もがWebAssemblyについて書き始めました。テクノロジーは常に進化し、改善されていました。調査を読んだ後、私たちはWasmが私たちの業務をスピードアップできることを提案しました。私たちはこれを完全に確信していませんでした：そのような大規模なプロジェクトでのWasmの使用に関するデータを見つけるのは難しいことがわかりました。

WasmはTypeScriptよりもやや悪い：

  1. 必要な機能を備えた特別なモジュールを初期化する必要があります。これにより、この機能が動作する前に遅延が発生する可能性があります。
  2. Wasmのコンパイル時のソースコードのサイズは、TSの場合よりもはるかに大きくなります。
  3. 開発者は、コード実行の代替バージョンをサポートする必要があります。これも、フロントエンドでは一般的でない言語で記述されています。

しかし、これらすべてにもかかわらず、Wasmを使用してコードの一部を書き換えるリスクがありました。

WebAssemblyの一般情報




Wasmはバイナリ形式です。さまざまな言語をコンパイルして、ブラウザーでコードを実行できます。多くの場合、このような事前コンパイルされたコードは、従来のJavaScriptよりも高速です。WebAssembly形式のコードは、ページのDOM要素にアクセスできず、原則として、クライアントで時間のかかる計算タスクを実行するために使用されます。

非常に便利で高速なC ++をコンパイル言語として選択しました。

WebAssemblyでC ++をコンパイルするために、emscriptenを使用しました。インストールしてC ++プロジェクトに追加した後、モジュールを取得するには、メインプロジェクトファイルを特定の方法で書き込む必要があります。たとえば、次のようになります。

#include <emscripten/bind.h>
#include <emscripten.h>
#include <math.h>

struct Point {
    double x;
    double y;
};

double sqr(double x) {
    return x * x;
}

EMSCRIPTEN_BINDINGS(my_value_example) {
    emscripten::value_object<Point>("Point")
        .field("x", &Point::x)
        .field("y", &Point::y)
    ;

    emscripten::register_vector<Point>("vector<Point>");

    emscripten::function("distance", emscripten::optional_override(
        [](Point point1, Point point2) {
            return sqrt(sqr(point1.x - point2.x) + sqr(point1.y - point2.y)) ;
    }));
}

次に、TypeScriptプロジェクトでこのコードを使用する方法について説明します。

コードでは、Point構造を定義し、それをTypeScriptのPointインターフェイスにマップします。xとyの2つのフィールドがあり、これらは構造のフィールドに対応しています。

さらに、標準のベクターコンテナーをC ++からTypeScriptに返す場合は、Point型に登録する必要があります。次に、TypeScriptでは、必要な関数を備えたインターフェイスがそれに対応します。

そして最後に、コードは、対応する名前でTypeScriptから呼び出すために関数を登録する方法を示しています。

emscriptenを使用してファイルをコンパイルし、結果のモジュールをTypeScriptプロジェクトに追加します。これで、任意のemscriptenモジュールに対して、便利な関数と型が事前定義されている共通のd.tsファイルを作成できます。

declare module "emscripten_module" {
    interface EmscriptenModule {
        readonly wasmMemory: WebAssembly.Memory;
        readonly HEAPU8: Uint8Array;
        readonly HEAPF64: Float64Array;

        locateFile: (path: string) => string;
        onRuntimeInitialized: () => void;
        _malloc: (size: size_t) => uintptr_t;
        _free: (addr: size_t) => uintptr_t;
    }

    export default EmscriptenModule;
    export type uintptr_t = number;
    export type size_t = number;
}

そして、モジュールのd.tsファイルを書くことができます：

declare module "emscripten_point" {
    import EmscriptenModule, {uintptr_t, size_t} from 'emscripten_module';

    interface NativeObject {
        delete: () => void;
    }

    interface Vector<T> extends NativeObject {
        get(index: number): T;
        size(): number;
    }

    interface Point {
        readonly x: number;
        readonly y: number;
    }

    interface PointModule extends EmscriptenModule {
        distance: (point1: Point, point2: Point) => number;
    }

    type PointModuleUninitialized = Partial<PointModule>;

    export default function createModuleApi(Module: Partial<PointModule>): PointModule;
}

これで、モジュールの初期化のためのPromiseを作成する関数を記述して使用できます。

import EmscriptenModule from 'emscripten_module';
import createPointModuleApi, {PointModule} from 'emscripten_point';
import * as pointModule from 'emscripten_point.wasm';

/**
 * Promisifies initialization of emscripten module.
 *
 * @param moduleUrl URL to wasm file, it could be encoded data URL.
 * @param moduleInitializer Escripten module factory,
 *        see https://emscripten.org/docs/compiling/WebAssembly.html#compiler-output.
 */
export default function initEmscriptenModule<ModuleT extends EmscriptenModule>(
    moduleUrl: string,
    moduleInitializer: (module: Partial<EmscriptenModule>) => ModuleT
): Promise<ModuleT> {
    return new Promise((resolve) => {
        const module = moduleInitializer({
            locateFile: () => moduleUrl,
            onRuntimeInitialized: function (): void {
                // module itself is thenable, to prevent infinite promise resolution
                delete (<any>module).then;

                resolve(module);
            }
        });
    });
}

const initialization = initEmscriptenModule(
    'data:application/wasm;base64,' + pointModule,
    createPointModuleApi
);

次に、このPromiseで距離関数と一緒にモジュールを取得します。

残念ながら、ブラウザでWasmコードを1行ずつデバッグすることはできません。したがって、通常のC ++のようにテストを記述してコードを実行する必要があります。そうすれば、便利なデバッグの機会が得られます。それにもかかわらず、ブラウザーでも、すべてをブラウザーコンソールに出力する標準のcoutストリームにアクセスできます。記事のプロジェクト例は、

結果


そこで、コードの一部を書き直し、ポリゴンとポリゴンの解析を検討する実験を開始しました。この図は、WasmとJavaScriptの1つのタイルの結果の中央値を示しています。



その結果、各メトリックに対してこのような相対係数が得



られました。純粋なプリミティブの解析時間とタイルのデコード時間からわかるように、Wasmは4倍以上高速です。総解析時間を見ると、その差も大きくなりますが、それでもまだ少し少ないです。これは、Wasmにデータを転送して結果を収集するコストが原因です。最初のタイルでは全体的なゲインが非常に高い（最初の10で-5倍以上）ことにも注意してください。ただし、相対係数は約3に減少します。

その結果、これらすべてを合わせて、バックグラウンドスレッドの1つのタイルの処理時間を20〜25％削減することができました。もちろん、この違いは以前の違いほど大きくはありませんが、ポリゴンとポリゴンの解析がすべてタイル処理ではないことを理解する必要があります。

そのため、モジュールを初期化する必要性について話すと、最初のタイルを解析する前に、約半分のユーザーに遅延がありました。遅延の中央値は188ミリ秒です。遅延は最初のタイルの前でのみ発生し、解析の成功率は一定であるため、最初に少し間を置いても我慢でき、深刻な問題とは見なされません。

もう1つのマイナス面は、ソースコードファイルのサイズです。 Wasmなしのベクトルマップエンジン全体のGzip圧縮された縮小コードは85 KBで、Wasm-191 KBです。同時に、破線および四角形の解析のみがWasmに実装されており、タイル内にあるすべてのプリミティブではありません。さらに、protobufをデコードするには、純粋なCでのライブラリ実装を選択する必要があり、C ++実装ではサイズがさらに大きくなりました。この違いは、C ++のコンパイル時に-O3の代わりに-Ozコンパイラフラグを使用するとわずかに減少しますが、それでも重要です。また、このような交換を行うと、パフォーマンスが低下します。

それにもかかわらず、ソースのサイズはカードの初期化の速度に大きな影響を与えませんでした。 Wasmは遅いデバイスでのみ悪化し、その差は2％未満です。しかし、Wasmを使用した実装で最初に表示されたベクタータイルのセットは、JS実装を使用した場合よりも少し速くユーザーに表示されました。これは、JSがまだ最適化されていない一方で、最初に処理されたタイルのゲインが大きいためです。

したがって、JavaScriptコードのパフォーマンスに慣れていない場合、Wasmは価値のあるオプションです。同時に、私たちよりも少ないパフォーマンスの向上を得ることができるか、まったくそれを得ることができません。これは、JavaScript自体が非常に高速に動作することがあり、Wasmではデータを転送して結果を収集する必要があるためです。

マップは通常のJavaScriptを実行します。これは、一般的な背景に対して構文解析のゲインがそれほど大きくないという事実と、Wasmで実装されているプリミティブのタイプがいくつかしかないという事実によるものです。これが変わったら、おそらくWasmを使い始めるでしょう。これに対するもう1つの強力な議論は、アセンブリとデバッグの複雑さです。2つの言語でプロジェクトをサポートすることは、パフォーマンスの向上に価値がある場合にのみ意味があります。
