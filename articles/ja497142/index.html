Harmony OSサポートをinDriver Androidアプリケーションに追加する方法

それはすべて、米国商務省がアメリカ企業とのビジネスを行うことが禁止されている企業のリストにHuaweiを含めたという事実から始まりました。Huaweiの対応は、スマートフォンのHarmony OSオペレーティングシステムであり、その中でGoogleサービスを使用することを拒否しました。また、市場の一部を失うことのないように、そのサポートをinDriverに統合しました。「OS統合」は大声で聞こえますが、私たちのアプリケーションは、Android用に作成された他のアプリケーションと同様に、Harmonyで実行されますが、完全に機能させるには、Googleサービスを類似のHuaweiサービスで置き換える必要があります。



inDriverでは、Googleのサービスを3つの目的で使用します。

  • プッシュ通知の送信。
  • 場所の定義;
  • グーグルマップ。

プッシュキット-プッシュ通知を送信する


まず第一に、Huaweiにはかなり優れたドキュメントがあります。ただし、クイックスタートには、ステップバイステップチュートリアルを使用してください。

ここには何も書かれていないので、ここではコメントが必要なので、いくつかの点についてのみ説明します。

サービスの有効化

データの保存場所を設定するよう求められます。注意あなたが変更することはできません、それを後で。場所は、プッシュの直接操作には影響しませんが、プッシュに関するデータを保存および処理するための場所にすぎません。

HMS SDKの統合

統合を完了すると、おそらくプロジェクトのビルドに問題が発生します。いずれにせよ、これは私たちのUbuntu 16.04.5 LTSビルドサーバー（GNU / Linux 4.4.0-174-generic x86_64）の場合でしたが、ローカルで動作しているMac mini（2018年後半、Mojave）ではすべて正常に動作しました。 追加するとこれを修正できます

メッセージの送信：HUAWEIプッシュキットコンソール。

プッシュの動作確認ができます。興味深い事実：コンソールType: Notification Messageで選択すると、ハンドラーをバイパスしてonMessageReceived(RemoteMessage message)すぐに電話に表示されます。

プッシュを設定するとき、コンソールを使用しなければならないことがよくありました。彼女とサーバーから送信されたプッシュの両方に最大30分の遅延がありました。これらが、開発者がすでに対処した一時的な問題であることを願っています。

これで電話に2つのプッシュサービスがありますが、どのようにして共同作業を整理できますか？まず、電話で利用可能なサービスを確認する必要があります。私たちの場所では、Googleのサービスを優先することにしました。 GoogleとHuaweiの両方のサービスが電話で利用できる場合、Googleと連携するとします。

サービスのステータスフラグを可能な値とともにサーバーに送信します。

- UNKNOWN
- GOOGLE
- HUAWEI

fun checkServices(context: Context): PlayServicesState {
   val googleAPI = GoogleApiAvailability.getInstance()
   when (googleAPI.isGooglePlayServicesAvailable(context)) {
       ConnectionResult.SUCCESS,
       ConnectionResult.SERVICE_MISSING,
       ConnectionResult.SERVICE_VERSION_UPDATE_REQUIRED,
       ConnectionResult.SERVICE_DISABLED -> return PlayServicesState.GOOGLE
   }
   val huaweiAPI = HuaweiApiAvailability.getInstance()
   when (huaweiAPI.isHuaweiMobileNoticeAvailable(context)) {
       com.huawei.hms.api.ConnectionResult.SUCCESS,
       com.huawei.hms.api.ConnectionResult.SERVICE_MISSING,
       com.huawei.hms.api.ConnectionResult.SERVICE_VERSION_UPDATE_REQUIRED,
       com.huawei.hms.api.ConnectionResult.SERVICE_DISABLED -> return PlayServicesState.HUAWEI
   }
   return PlayServicesState.UNKNOWN
}

受信した値に応じて、サーバーは対応するサービスにプッシュ通知を送信します。

彼らの仕事を整理することは技術の問題です。私はそれをやった。

FCMのイベント処理：

class AppFcmListenerService : FirebaseMessagingService() {
   private val cloudMessageHandler: CloudMessageHandler = GoogleMessageHandler(this)
   override fun onMessageReceived(message: RemoteMessage) {
       cloudMessageHandler.onMessageReceived(message)
   }
}

およびHCM：

class AppHmsListenerService : HmsMessageService() {
   private val cloudMessageHandler: CloudMessageHandler = HuaweiMessageHandler(this)
   override fun onMessageReceived(message: RemoteMessage) {
       cloudMessageHandler.onMessageReceived(message)
   }
}

抽象クラスに委託：

abstract class CloudMessageHandler { 
fun onMessageReceived(message: Any) {
   val data = getData(message)
   onMessageReceived(data)
}
private fun onMessageReceived(data: Map<String, String>) {
   //   
}
abstract fun getData(message: Any) :  Map<String, String> {
}



FCMの場合、次の 2つの実装があります。

class FirebaseMessageHandler(override val context: Context) : CloudMessageHandler(context) {
   override fun getData(message: Any): Map<String, String> {
       return (message as RemoteMessage).data
   }
}

HCMの場合：

class HuaweiMessageHandler(override val context: Context) : CloudMessageHandler(context) {
   override fun getData(message: Any): Map<String, String> {
       return (message as RemoteMessage).dataOfMap
   }
}

Huaweiクラスは、Googleサービスとほぼ同様のインターフェースを備えています。これは、「ほとんど」私は個人的に事実で明らかにしているRemoteMessageアウトcom.google.firebase.messagingのためにMap<String, String>あなたが呼び出したいRemoteMessage.getData()、とのためにRemoteMessage出てcom.huawei.hms.push- getDataOfMap()。

ロケーションキット-ロケーション決定


ロケーションはLOCATION_SERVICE、GoogleとHuaweiのサービスなしで利用可能なに対応できます。しかし、旅客輸送サービスにとって、正確な位置決定は重要なパラメーターです。幸いにも、ロケーションキットクラスのインターフェースは、Googleサービスの同じ名前のインターフェースと変わらなかったため、コード内のアプリケーションに違いはありませんでした。ただし、Huaweiのサービスの運用はそれほど正確ではありません。これが一時的な問題であることを願っています。
また、ジオロケーション要求がHMSコアを介して送信されるという事実にも遭遇しました。これは、正しく動作させるために、ジオロケーションの権限を「常に許可」に設定する必要があることを意味します。
