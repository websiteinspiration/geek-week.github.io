Windows Server 2019と 静止を伴うVMwareスナップショット：問題に対するエレガントなソリューション



こんにちは、ハブル！

VMwareでEFIエミュレーションを使用するWindows Server 2019仮想マシンには、アプリケーション対応スナップショットに問題があります。スナップショットが作成され、100％に達し、約5分間ハングした後、エラー「仮想マシンの静止に失敗しました」に陥ります。
 


調査の結果、このエラーの原因は、Windows Server 2019 VSSサービスと、アプリケーションの休止を担当するVMwareスナップショットプロバイダーの間の競合であることが判明しました。これにより、スナップショット用の仮想マシンが準備されます。これにより、アプリケーションと書き込み操作が停止し、スナップショットからの回復後、すべてのデータが整合します。

Microsoftは、Application-Aware SnapshotsはOSによって完全にサポートされていると主張し、主張して​​います。実際、このような問題はHyper-Vでは発生しません。VMware は、休止をオフにするようユーザーに控えめに招待しました（そのため、そのスナップショットはアプリケーションレベルで一貫性がなくなります）。または、GPTマーキングとEFIエミュレーション（2020年！）でシステムディスクを拒否します。 

今日は、この問題を美しく回避し、Windows Server 2019仮想マシンの一貫したアプリケーション対応のスナップショットを取得する方法を紹介すると同時に、仮想マシンにWindows ServerをインストールするためのMicrosoftがサポートする方法についてもお知らせします。

誰が有罪ですか？


最初からやり直してください。VMオプションでVMware ESXiに仮想マシンを作成する場合、デフォルトのエミュレーションはBIOSエミュレーションです。また、セキュリティやシステムディスクのサイズに特別な要件はありませんが、このようなシナリオは非常に適切です。ただし、セキュアブートを構成する必要があるか、vCPUの数が128を超えるとすぐに、UEFIエミュレーションを備えたVMを作成する必要があります。あるだけの場合：

  • セキュアブート-無関係なメディアからのブートに対する保護。
  • 暗号化サポート（TPMサポートが必要ですが、ESXi 6.7でも同じ問題が発生します）。
  • 2TBを超えるドライブのサポート（疑わしいですが、起こります）。
  • 多数のプロセッサーのサポート。

以前と同じようにWindowsをVMware VMにインストールする場合（VMの作成、ISOのパームオフ、ハンドルを使用した起動、次、次...）、システムディスクをGPTパーティション化すると、[リカバリ]セクションがEFIパーティションとシステムのパーティションの横に表示されます。

私たちは物資を思い出します：

リカバリパーティション -問題が発生した場合にシステムの工場出荷時の設定を復元するために使用されるシステムハードドライブ上の保護されたパーティション。 GPTディスクにWindows Serverをインストールすると、RecoveryはWindows Recovery Environment（WinRE）を書き込みます。コンピュータメーカーは、通常、特定のハードウェアに合わせて調整されたカスタム設定とドライバーでこのセクションを補足します。

やめる！ 

ただし、ハードウェアではなく、仮想マシンで作業しています。工場設定へのロールバックは必要ありません。最後に、VMを元の状態に戻したい場合は、単に再デプロイします。したがって、VMのリカバリパーティションは意味がなく、特に必要ありません。

リカバリセクションは、変更や誤って削除されないように保護されています。「フリーズしません」。その結果、VSS VMwareスナップショットプロバイダーとMicrosoft Windows VSSの間に競合が発生します。VMwareプロバイダーがディスクのすべてのパーティションのスナップショットを作成しようとしているため、Windows Serverはこれを許可していません。この良いニュースはログによって私たちに報告されています：



何をすべきか？


単純に扱われます。Convert Windows Imageコマンドレットを使用して、VMにWindowsをインストールできます。実際、このようなインストールスクリプトだけが推奨され、Microsoft自体によってサポートされています（このスクリプトは、Nano ServerイメージがあるディレクトリのWindows Server 2016 RTMイメージにありますが、これもまた興味深い話です）。

スクリプトは、更新されたsysprepされたWindows Server、必要なドライバー、さらには役割を含むテンプレートを自動的に作成します。スクリプトがunattend.xmlをサポートしているため、OSのインストールと構成を自動化できるため、このアプローチは興味深いものです。さらに、このオプションはエンジニアリングの自由を提供します。Hyper-VでのVMの事前起動から、Invoke-Commandを介したインストール、レジストリの編集を介したカスタムサービスの作成まで。 

出力では、厚いVHDディスクを取得します。これをVMwareに入れ、VMを起動し、VMwareツールを配置して、テンプレートに変換できます。その結果、VMには内部にGPTマークアップがあり、EFI VMでの実行に適しており、設計上、リカバリセクションはありません。おまけとして：私の例では、パフォーマンスを提供する準仮想アダプターをすぐに作成します。そして最も重要なこととして、このような仮想マシンでは、アプリケーションを意識したスナップショットを問題なく実行できます。

しかし、すでに「欠陥のある」テンプレートがある場合はどうでしょうか。


diskpart.exeを使用してリカバリパーティションを削除し、EFIおよびシステムパーティションを移動してマークアップを美しくするだけで十分です（ただし、これはできません）。サポートされていますか？仮想マシン上-間違いなく！

  1. cmd.exeを起動します。
  2. diskpartを起動します。
  3. ドライブのリストを取得します。
    list disk
  4. ディスクを選択：
    select disk n

    ここで、nはディスク番号です。
  5. セクションをリストします。
    list partition
  6. リカバリ（n）パーティション番号を確認して選択します。
    select partition n
  7. セクションから保護を削除します。
    gpt attributes=0x8000000000000000
  8. 削除：
    delete partition
