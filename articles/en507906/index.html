We use DI in Angular to the maximum - the concept of private providers

Angular has a very powerful Dependency Injection mechanism. It allows you to transfer any data through your application, convert and redefine them in the right parts.



Therefore, we can make the application architecture simpler and more flexible: an understandable data flow, minimal code coherence, ease of testing or replacing dependencies.



Nevertheless, DI in applications is used quite modestly. As a rule, this is the introduction of services or the transfer of some global data from top to bottom through the dependency injection tree.



In this article, I would like to show an alternative way of working with data obtained from DI. Purpose: To simplify the components, directives and services that use this data.







How DI is commonly used in Angular



I conduct daily reviews of Angular code at work and in open source. Typically, in most DI applications, it comes down to the following functionality:



  1. Get Angular entities from the dependency tree: ChangeDetectorRef, ElementRef and so on.
  2. Get the service to use it in the component.
  3. Get some global config for the token, which is declared somewhere above. For example, set the API_URL token in the root module and get it from the DI anywhere in the application, if necessary.


Less common are cases when developers go further and transform an existing global token into a more convenient form. A good example of such a conversion is the WINDOW token from the @ ng-web-apis / common package .



Angular provides the DOCUMENT token so that you can get the page object from anywhere in the application: your components are independent of global objects, easy to test, nothing will break with SSR.



If you regularly need access to the WINDOW object, you can write this token: 



import {DOCUMENT} from '@angular/common';
import {inject, InjectionToken} from '@angular/core';

export const WINDOW = new InjectionToken<Window>(
    'An abstraction over global window object',
    {
        factory: () => {
            const {defaultView} = inject(DOCUMENT);

            if (!defaultView) {
                throw new Error('Window is not available');
            }

            return defaultView;
        },
    },
);


When someone requests the WINDOW token for the first time from the DI tree, the token factory will be executed - he will receive a DOCUMENT object from Angular and get a link to the window object from it.



Further, I propose to consider a different approach to such transformations - when they are performed directly in the providers of the component or directive, which injects the result.



Private providers



In our team, we actively use DI when working with Angular and began to notice that often we need to make some transformations of the data obtained from the DI before using them. In fact, our component needs some data, and we embed others in it and execute the transformation logic inside it.

We have:



  • The component that is responsible for displaying information on a certain entity - the organization.
  • Query-parameter of the route, which indicates the id of the organization with which we are currently working.
  • A service that by id returns an Observable with information about the organization.


What do we want to do?



Take the organization id from the query parameters, pass it to the service method, and get a stream with information about the organization in response. Display this information in the component.



Consider three ways to achieve the desired and analyze them.



How to do it is not necessary



Sometimes I come across this style of working with data in components. Please do not do this: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent implements OnInit {
   organization: Organization;

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}

   ngOnInit() {
       this.activatedRoute.params
           .pipe(
               switchMap(params => {
                   const id = params.get('orgId');

                   return this.organizationService.getOrganizationById$(id);
               }),
           )
           .subscribe(organization => {
               this.organization = organization;
           });
   }
}

Let's do well.



Erin's report from the video that I applied above is done well. With its variant it turns out something like this: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent {
   readonly organization$: Observable<Organization> = this.activatedRoute.params.pipe(
       switchMap(params => {
           const id = params.get('orgId');
           return this.organizationService.getOrganizationById$(id);
       }),
   );

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}
}


To use the received data in the template: 

<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


This code works fine and is free from the drawbacks of the previous approach: it looks neat enough, it does not have extra fields. If we want to expand the component with a similar stream, we just add one more - to make it easier, because to add a new stream we do not need to touch the code of the previous one.



In addition, the data stream becomes more transparent: we only have the stream that is created when the component class is created. When it issues data, the information in our template will be displayed.



Let's make it even cooler: private providers



Let's take a closer look at the previous solution.



In fact, the component is independent of the router and even of the OrganizationService. It depends on organization $. But there is no such entity in our dependency injection tree, so we are forced to perform transformations in the component.



But what if you perform this conversion before the data gets into the component? Let's write Provider specifically for the component in which the necessary transformations will occur.



For convenience, we place the providers in a separate file next to the component, obtaining the following file structure:







Provider will be located in the organization.providers.ts file for data conversion and a token for receiving them in the component: 



export const ORGANIZATION_INFO = new InjectionToken<Observable<Organization>>(
   'A stream with current organization information',
);
         :

export const ORGANIZATION_PROVIDERS: Provider[] = [
   {
       provide: ORGANIZATION_INFO,
       deps: [ActivatedRoute, OrganizationService],
       useFactory: organizationFactory,
   },
];

export function organizationFactory(
   {params}: ActivatedRoute,
   organizationService: OrganizationService,
): Observable<Organization> {
   return params.pipe(
       switchMap(params => {
           const id = params.get('orgId');

           return organizationService.getOrganizationById$(id);
       }),
   );
}


Define an array of providers for the component. The value for the ORGANIZATION_INFO token will be obtained from the factory in which we will make the necessary data conversion.



A note on the operation of DI - deps allows you to take the necessary entities from the DI tree and pass them as arguments to the token value factory. In them, you can get any entity from DI, including using DI decorators, for example: 



{
       provide: ACTIVE_TAB,
       deps: [
           [new Optional(), new Self(), RouterLinkActive],
       ],
       useFactory: activeTabFactory,
}


Declare providers in the component: 



@Component({
   ..
   providers: [ORGANIZATION_PROVIDERS],
})


And we are ready to use the data in the component: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
   providers: [ORGANIZATION_PROVIDERS],
})
export class OrganizationComponent {
   constructor(
       @Inject(ORGANIZATION_INFO) readonly organization$: Observable<Organization>,
   ) {}
}


The component class is reduced to one line with the receipt of data.



The template remains the same: 



<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


What does this approach give us?



  1. Pure dependencies: the component does not implement and does not store unnecessary entities. It works only with the data that it needs, while it itself remains clean and contains only logic for displaying data.

After implementing this approach, our components and directives began to look much cleaner and simpler. Separation of the logic of data transformation and their display accelerated the process of their refinement and expansion. The time to catch bugs was also reduced due to the fact that you can immediately cut off the problem area: either the problem is in the data conversion, or in how it is displayed to the user.



Conclusion



The described approach will not save you from all the design problems. Adding providers to any little thing is also not worth it: sometimes the code is clearer if you use the conversion in the method or use Pipe.



Nevertheless, I hope that private providers will help you simplify components with a large number of dependencies or provide an alternative for the gradual refactoring of large pieces of logic.
