実行中のサーバーをRAIDに移行する

むかしむかし、Ubuntu 12.04には単一のドライブで動作するLAMPサーバーがありました。したがって、サーバーをよりフォールトトレラントな構成であるRAID1に確実に転送するという問題が発生しました。このサーバーを担当する仲間は2番目のディスクをインストールし、残りは私に割り当てられました。サーバーへのアクセスはssh経由でしか行われず、タスクが複雑になりました。

検索後、ガイドが見つかり、仮想マシンでトレーニングが始まりました。期待通りの結果が得られて興奮しましたが、それでも前述のサーバーをRAID1に正常に移行し、必要なアクションの大まかな概要を作成しました。移管プロセスをより詳しく説明した記事を「Habr」に掲載することにしました。

ディスクの内訳は次のとおりです。

/dev/sda1            2048     1574911      786432  swap
/dev/sda2   *     1574912    16254975     7340032  / 
/dev/sda3        16254976    31457279     7601152  /home

2番目のディスクは新しいので、前のパーティションを削除するための操作は必要ありませんでした。

された最初のもの に変更となった/etc/default/grub：
GRUB_TIMEOUT = 5（私たちは、タイムアウトを短くします）。
GRUB_RECORDFAIL_TIMEOUT = 10（エラーが発生した場合にシステムが対話式の介入を必要とせずに起動できるように追加します;試してみるにはタイムアウトします）;
GRUB_CMDLINE_LINUX_DEFAULT =“ bootdegraded”（システムが下位アレイから起動できるように、必ず“ bootdegraded”を追加してください）;
＃GRUB_HIDDEN_TIMEOUT_QUIET = true（GRUBメニューが常に表示されるようにコメント化する必要があります）。

一時的にスワップを無効にし、fdiskセットタイプfdで/ dev / sda1を一覧表示します。 （Linux raid autodetect）、2番目のディスクのパーティションを最初のディスクと同じにします（すべての操作はスーパーユーザー権限で実行されます）。

sfdisk –d /dev/sda | sfdisk –f /dev/sdb

次に、段階的にRAIDの構築を開始します。

mdadm --create --verbose /dev/md0 --raid-devices=2 --level=1 --metadata=1.2 /dev/sda1 /dev/sdb1

の下にセクションを作成しますswap。
mkswap /dev/md0

/etc/fstab最初のディスクのパーティションをRAIDパーティションに置き換え、6列目の数値（fsckを確認する必要がある）をゼロに変更し て修正します。

/dev/sda1 → /dev/md0 # SWAP
/dev/sda2 → /dev/md1 # /
/dev/sda3 → /dev/md2 # /home

アクティベートswap：
swapon -a

サーバーは現在最初のディスクで実行されているため、これまでのところ2番目のディスクのみを使用してRAIDを作成しています。

mdadm --create --verbose /dev/md1 --raid-devices=2 --level=1 --metadata=1.2 missing /dev/sdb2 
mkfs.ext4 /dev/md1
mdadm --create --verbose /dev/md2 --raid-devices=2 --level=1 --metadata=1.2 missing /dev/sdb3 
mkfs.ext4 /dev/md2

構成の更新mdadm.conf：

mdadm --examine --scan >> /etc/mdadm/mdadm.conf

私たちは、initramfsそれが私たちのRAIDについての情報を保存するように更新します。

update-initramfs -u

次に、最も長いプロセスが始まります-データの同期。サーバーがいくつかのタスクを実行するため、同期が完了した後、アレイのセクションの情報の一部が、作業ディスクの対応するセクションのデータと異なる場合があります。いくつかのオプションがあります。最小の負荷の時間を選択したり、同期中にサービスの一部を停止したり、差異を無視したりできます。通常、マウントして同期します。

mount /dev/md1 /mnt/ && [tmux|screen] rsync -axu / /mnt
mount /dev/md2 /mnt/home && [tmux|screen] rsync -axu /home/ /mnt/home

ssh接続の予期しない中断による同期を防ぐために、ターミナルマルチプレクサーの使用が妨げられることはありません。

同期の終了を待った後、システムディレクトリを新しいルートにマウントします。

mount --bind /proc /mnt/proc
mount --bind /dev /mnt/dev
mount --bind /var /mnt/var
mount --bind /run /mnt/run
mount --bind /sys /mnt/sys

新しいシステムに移行しています 
chroot /mnt
ブートローダーを両方のディスクにインストールします 

grub-install -–recheck /dev/sda
grub-install --recheck /dev/sdb

ブート構成を更新して、RAID（mdraid1x）に必要なモジュールをロードします。

update-grub

元のシステムに戻り、必要に応じてセクションを再同期します。

[tmux|screen] rsync -axu / /mnt
[tmux|screen] rsync -axu /home/ /mnt/home

コンソールまたはサーバーのいずれにも物理的にアクセスできず、RAIDが起動するドライブを選択できないため、このトリックを使用します。準備したブートローダー構成を、システムが起動するドライブとRAIDにないドライブに転送します。 RAIDについて誰が知っているか。 RAIDにあるディスクから現在の起動ディスクにgrub.cfgをコピーします。これにより、システムは/ dev / sdaで開始できますが、アレイをマウントし、すでにRAIDにあるパーティションから起動し続けます。最初に、RAIDから起動できない場合にシステムを復元するときに必要になる可能性がある古いファイルを保存してから、「戦闘」構成ファイルをコピーします。

cp -p /boot/grub/grub.cfg /boot/grub/grub.old 
cp -p /mnt/boot/grub/grub.cfg /boot/grub/grub.cfg

さらに、これらのファイルを比較して、新しいブートローダー構成ファイルでルートパーティションがRAID上にあるものとしてリストされていることを確認できます。

次に、作業の最も重要な段階に進み、再起動します。pingサーバーへのコンソールでそれを実行して、サーバーが再び使用可能になる時期が明確になるようにすることができます。ログインして、期待通りの結果が得られたことを確認します。lsblkディレクトリ/と/homeRAIDパーティション上にあることを示します。

前菜には簡単で楽しい仕事が残っています。最初のディスクの2つのパーティションをfdfdiskタイプを使用して設定した後、アレイに追加します。

mdadm /dev/md1 --add /dev/sda2
mdadm /dev/md2 --add /dev/sda3

その後、定期的にミラー同期がどのように行われているかを確認します。 

watch -n 5 cat /proc/mdstat

ここで終わりましょう。ご清聴ありがとうございました！
