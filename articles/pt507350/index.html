Analisador PVS-Studio: identificando possíveis problemas de compatibilidade da API Java SE











2019 foi um ano muito movimentado em termos de conferências. Nossa equipe pode fazer viagens de negócios por semanas. E como você sabe, a conferência é a hora de compartilhar conhecimento. Além de termos feito apresentações e contado muitas coisas interessantes em nosso estande, também aprendemos muito com a comunicação com os participantes da conferência e com os palestrantes. Portanto, na conferência de outono Joker 2019, um relatório de Dalia Abo Sheasha “Migrando além do Java 8” nos inspirou a implementar uma nova regra de diagnóstico que permite identificar incompatibilidades na API do Java SE entre diferentes versões do Java. Isso será discutido.



Relevância dos problemas de compatibilidade do Java SE



Já foi lançado o Java SE 14. Apesar disso, muitas empresas continuam a usar versões anteriores do Java (Java SE 6, 7, 8, ...). Como o tempo está em execução e o Java está sendo atualizado o tempo todo, o problema de compatibilidade entre diferentes versões da API do Java SE está se tornando cada vez mais relevante a cada ano.



Quando novas versões do Java SE são lançadas, elas geralmente são compatíveis com versões anteriores, ou seja, por exemplo, um aplicativo desenvolvido com base no Java SE 8 deve ser executado em 11 versões do Java sem problemas. No entanto, na prática, algumas incompatibilidades podem surgir em várias classes e métodos. Essa incompatibilidade está no fato de algumas APIs sofrerem alterações: elas são excluídas, alteradas no comportamento, marcadas como obsoletas e muito mais.



Esse problema só piorará quando você começar a pensar em migrar seu projeto para o Java SE mais recente. Ou quando o suporte técnico do seu aplicativo receber mais e mais cartas de que o aplicativo se comporta incorretamente ou não pode ser iniciado.



Eu acho que isso é suficiente para chamar a atenção!



Kit de ferramentas existente



Não existe uma solução única para alternar de uma versão do Java para outra. E se o seu aplicativo é o resultado de muitos anos de desenvolvimento contínuo usando uma variedade de soluções não triviais, quando você fica intrigado com a transição para a versão mais recente do Java, é provável que encontre um processo bastante trabalhoso.



Esse processo consiste em identificar uma API problemática ou potencialmente problemática que precisa ser revisada e substituída por uma solução alternativa. Isso pode afetar seriamente a lógica de negócios, que pode levar muito tempo para corrigir e testar. Se você precisar não apenas mudar para a nova versão do Java SE, mas também garantir a compatibilidade com várias versões, essa tarefa se tornará muito mais complicada.



Tendo vasculhado na Internet o tópico de identificação de incompatibilidades de API entre diferentes Java SEs, encontrei apenas ferramentas que acompanham o JDK: javac, jdeps, jdeprscan.



Não havia nenhuma ferramenta de terceiros nesse negócio, exceto a que tive a honra de conhecer no relatório Joker 2019 - Migration Toolkit for Binários de aplicativos .



javac



Ao desenvolver um aplicativo, não se esqueça dos avisos do compilador. Todo mundo tem uma atitude diferente em relação aos avisos:



  • Alguém os observa estritamente e um pouco de algo imediatamente soa o alarme. Talvez um pouco paranóico, mas pode muito bem reduzir os riscos técnicos no futuro;
  • Alguém os coleta em um arquivo separado com o pensamento de que ele necessariamente os revisará, como ele será libertado;
  • Alguém geralmente ignora avisos.


A primeira opção é boa porque parte dos problemas pode ser resolvida sem demora, por exemplo, apenas não usando um método ou classe que esteja marcado como obsoleto. O uso de uma API desse tipo não é um problema de bloqueio, mas você deve prestar atenção, pois já é provável que, ao usar a versão recém-lançada do Java, seu aplicativo se comporte de maneira diferente ou até mesmo travar.



jdeps



Se você já está intrigado com a questão de migrar seu aplicativo para uma versão mais recente do Java SE, o jdeps está com pressa para ajudá-lo.



O jdeps é uma ferramenta de linha de comando que executa análise estática das dependências de seu aplicativo e bibliotecas, aceitando arquivos * .class ou * .jar como entrada . A partir do Java SE 8, ele vem com o JDK.



E aqui estamos interessados ​​no fato de que, se você executar esta ferramenta com a opção --jdk-internals , ela informará em qual API JDK interna sua classe depende. Este é um ponto muito importante devido ao fato de a API interna não garantir que não será alterada nas versões futuras do Java.



Vejamos um exemplo. Suponha que você esteja desenvolvendo seu aplicativo Java 8 há muito tempo e nunca tenha se confundido com a compatibilidade da API do Java SE usada com versões mais recentes. Houve uma pergunta sobre a migração de seu aplicativo para, por exemplo, o Java 11. Nesse caso, você pode seguir o caminho de resistência mínima e iniciar imediatamente o aplicativo usando o Java 11. Mas algo deu errado, e suponha que o aplicativo tenha começado a falhar. Nesse caso, você pode executar o jdeps a partir do Java 11, configurando os arquivos * .jar do seu aplicativo.



O resultado será aproximadamente o seguinte:













A saída mostra que a dependência sun.misc.BASE64Encoder no Java 11 será removida. E a falha do seu aplicativo quando ele foi iniciado pela primeira vez no Java 11 é provavelmente devido ao erro java.lang.NoClassDefFoundError . Além de informar que isso não pode agradar, o jdeps pode oferecer uma dependência alternativa que pode ser usada em vez da atual. Nesse caso, ele sugeriu a substituição da dependência remota pelo java.util.Base64 .



O segundo aviso da ferramenta indica que ainda estamos usando outra dependência interna no código, mas até agora está correto. Haverá alterações nesta API em versões futuras do Java, mas isso deve ser levado em consideração.



Obviamente, o jdeps pode fazer mais do que apenas testar o uso de componentes internos do JDK. Isso não está incluído no escopo deste artigo, mas você pode se familiarizar com os recursos na página oficial.



jdeprscan



Os objetivos do jdeprscan são exatamente os mesmos do jdeps, a saber, ajudar a encontrar uma API indesejada e problemática.



O jdeprescan é uma ferramenta de análise estática que verifica um arquivo * .jar (ou alguma outra coleção de arquivos * .class ) para usar elementos desatualizados da API. O uso de uma API desatualizada não é um problema de bloqueio, mas você deve prestar atenção. A partir do Java SE 9, ele vem com o JDK.



Suponha também que haja uma questão de migrar o aplicativo para Java 11. Nesse caso, executando o comando 



jdeprscan --release 8 app.jar


você receberá uma lista de APIs que foram descontinuadas para Java 8, ou seja, a API que pode ser removida em versões futuras do Java. Depois de corrigir todos os avisos, você pode executar 

jdeprscan --release 11 app.jar


que produzirá uma lista de APIs que já foram descontinuadas para Java 11. Dessa maneira, você poderá encontrar e corrigir (se necessário) todas as APIs não recomendadas.



Kit de Ferramentas de Migração para Binários de Aplicativos



Essa ferramenta foi projetada para ajudá-lo a avaliar rapidamente seu aplicativo de usuário em busca de possíveis problemas antes de implantar em vários servidores (JBOSS, WebShere, Tomcat, WebLogic, ...). Além de toda a funcionalidade, a ferramenta também permite identificar diferenças na API do Java SE de diferentes versões.



Vamos dar uma rápida olhada no que é essa ferramenta.



Um rápido lançamento da ferramenta se parece com isso: 



java -jar binaryAppScanner.jar yourApp.jar --analyzeJavaSE 
--sourceJava=oracle8 --targetJava=java11 ....


A opção analyJavaSE implica o uso de vários parâmetros, sobre os quais você pode aprender chamando ajuda.

Implementação no PVS-Studio



Após estudar as ferramentas consideradas, chegamos à conclusão de que a busca por possíveis problemas de compatibilidade de diferentes versões da API do Java SE é uma tarefa valiosa para a análise estática.



As ferramentas discutidas realmente facilitarão a tarefa de migrar um aplicativo ou procurar uma API que possa prejudicar o desempenho do seu aplicativo nas versões mais recentes do Java SE (se você não quiser migrar o aplicativo). Mas, pensando que essas ferramentas o tempo todo você precisa executar na linha de comando separadamente do processo de desenvolvimento para identificar problemas, chegamos à decisão de que isso não é muito conveniente. E, partindo do fato de que a análise estática é necessária para detectar código problemático ou potencialmente problemático nos estágios iniciais do desenvolvimento, implementamos a regra de diagnóstico V6078, que sinalizará você sobre a API "problem".



A regra V6078 avisa com antecedência que seu código depende de certas funções e classes da API do Java SE, o que pode causar dificuldades em versões futuras do Java. E você, no início da implementação de um recurso específico, não estará vinculado a essa API, reduzindo assim os riscos técnicos no futuro.



A regra de diagnóstico emite avisos nos seguintes casos:



  • Se o método / classe / pacote for removido na versão de destino do Java;
  • Se o método / classe / pacote estiver marcado como obsoleto na versão de destino do Java;
  • Se o método mudou a assinatura.


A regra atualmente permite analisar a compatibilidade do Oracle Java SE das versões 8 a 14. Para uma regra se tornar ativa, ela deve ser configurada.



IntelliJ IDEA



No plug-in IntelliJ IDEA, é necessário ativar a regra e especificar os parâmetros na guia Configurações> PVS-Studio> Detecção de problemas de compatibilidade da API, a saber:



  • Source Java SE - a versão do Java na qual seu aplicativo é desenvolvido;
  • Alvo Java SE - a versão do Java com a qual você deseja verificar a compatibilidade da API usada no seu aplicativo (Java de origem SE);
  • Excluir pacotes - pacotes que você deseja excluir da análise de compatibilidade (os pacotes são listados com vírgula).












Plugin for gradle



Usando o plug-in gradle, você precisa definir as configurações do analisador em build.gardle: 



apply plugin: com.pvsstudio.PvsStudioGradlePlugin
pvsstudio {
    ....
    compatibility = true
    sourceJava = /*version*/  
    targetJava = /*version*/
    excludePackages = [/*pack1, pack2, ...*/]
}


Plugin para Maven



Usando o plug-in maven, é necessário definir as configurações do analisador em pom.xml: 



<build>
  <plugins>
    <plugin>
      <groupId>com.pvsstudio</groupId>
      <artifactId>pvsstudio-maven-plugin</artifactId>
      ....
      <configuration>
        <analyzer>
          ....
          <compatibility>true</compatibility>
          <sourceJava>/*version*/</sourceJava>
          <targetJava>/*version*/</targetJava>
          <excludePackages>/*pack1, pack2, ...*/</excludePackages>        
        </analyzer>
      </configuration>
    </plugin>
  </plugins>
</build>


Usando o kernel diretamente



Se você usar o analisador diretamente através da linha de comando, para ativar a análise de compatibilidade das APIs Java SE selecionadas, você deverá usar os seguintes parâmetros: 



java -jar pvs-studio.jar /*other options*/ --compatibility 
--source-java /*version*/ --target-java /*version*/ 
--exclude-packages /*pack1 pack2 ... */


Operação do analisador



Vamos supor que estamos desenvolvendo um aplicativo baseado no Java SE 8 e que temos uma classe com o seguinte conteúdo: 



/* imports */
import java.util.jar.Pack200;
public class SomeClass
{
  /* code */
  public static void someFunction(Pack200.Packer packer, ...)
  {
    /* code */
    packer.addPropertyChangeListener(evt -> {/* code */});
    /* code */
  }
}


Executando uma análise estática com várias configurações para a regra de diagnóstico, observaremos a seguinte figura:



  • Origem Java SE - 8, Destino Java SE - 9
    • O método 'addPropertyChangeListener' será removido.


  • Origem Java SE - 8, Destino Java SE - 11
    • O método 'addPropertyChangeListener' será removido.
    • A classe 'Pack200' será marcada como obsoleta.


  • Origem Java SE - 8, Destino Java SE - 14
    • A classe 'Pack200' será removida.




Primeiro, no Java SE 9, o método 'addPropertyChangeListener' na classe 'Pack200.Packer' foi removido. Na versão 11, o fato de a classe 'Pack200' ter sido marcada como obsoleta foi adicionada a isso. E na versão 14, essa classe foi completamente excluída.



Portanto, ao iniciar o aplicativo no Java 11, você obterá 'java.lang.NoSuchMethodError' e, se você executar no Java 14 - 'java.lang.NoClassDefFoundError'.



Conhecendo essas informações, ao desenvolver seu aplicativo, você considerará soluções alternativas para a tarefa.



Ideias para um maior desenvolvimento



Durante a implementação da regra de diagnóstico, surgiram idéias sobre a expansão:



  • Como o jdeprscan e o jdeps não podem alertar sobre o uso da reflexão para acessar a API encapsulada, faz sentido modificar a regra para tentar descobrir que tipo de API eles estão tentando usar. O resultado pode não ser perfeito, mas por que não ?!
  • Existe uma grande variedade de implementações JDK (da Oracle, IBM, Red Hat, ...). Como regra, eles são compatíveis entre si. Mas quanto a API JDK interna difere? Afinal, os desenvolvedores podem estar ligados a ele, o que pode levar a possíveis problemas ao migrar um aplicativo de um JDK para outro.


Todas essas são questões de pesquisa e, no final, o tempo dirá =) Se você conhece cenários interessantes para esse diagnóstico e gostaria de vê-los no PVS-Studio, escreva-nos .



Conclusão



A busca por possíveis erros de incompatibilidade é totalmente consistente com a nossa ideologia - usando o PVS-Studio para encontrar e corrigir erros nos estágios iniciais da escrita do código. Como um erro de digitação, uma chamada para uma função específica pode ser inserida no código a qualquer momento, portanto, executar regularmente o PVS-Studio em um projeto agora é duas vezes mais útil. Os



diagnósticos do V6078 estão disponíveis no analisador a partir da versão 7.08 . Você pode baixar e experimentar o analisador em seu projeto na página de download .
