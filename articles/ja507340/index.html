ロシア語を含む多くの記事がサーバーラグの補償について書かれています。このテクノロジーは、90年代後半からマルチユーザーFPSを作成するために積極的に使用されてきたため、当然のことです。たとえば、QuakeWorld modを思い出すことができます。

この記事では、私の目標は、何千回も書かれたことを繰り返すことではなく、技術スタックとコアゲームプレイの機能を考慮して、ゲームにラグ補正を実装する方法を説明することです。

ラグ補正について初めて聞いた人のために、ここに問題への短い遠足があります。



マルチプレイヤーFPSゲームでは、通常、マッチはリモートサーバーでシミュレートされます。プレイヤーはサーバーに入力（送信されたキーに関する情報）を送信し、それに応答してサーバーは受信したデータを考慮して、更新されたゲームの状態を送信します。このインタラクション方式では、「進む」キーを押してからプレーヤーのキャラクターが画面上を移動する瞬間までの遅延は、常により多くのpingになります。



ローカルネットワーク上でこの遅延（一般に入力遅延と呼ばれる）が見えない場合、インターネットを介して再生すると、キャラクターを操作するときに「氷の上を滑る」感覚が生じます。この問題は、モバイルネットワークに2重に関連しています。プレーヤーのpingが200ミリ秒の場合は、優れた接続と見なされます。 pingは通常350、500、および1000ミリ秒です。そうすれば、高速シューティングゲームで入力ラグを再生することはすでにほとんど不可能です。



この問題の解決策は、クライアント側のシミュレーション予測です。ここでは、クライアント自体がサーバーからの応答を待たずに、入力をプレイヤーのキャラクターに適用します。そして、回答を受け取ると、結果を比較して対戦相手の位置を更新するだけです。この場合、キーを押してから結果を画面に表示するまでの遅延は最小限です。



ここでは、ニュアンスを理解することが重要です。クライアントは常に、前回の入力と敵に従って自分自身を描画します-以前のように、サーバーからのデータからネットワーク遅延が発生します。つまり、対戦相手に発砲するとき、プレイヤーは自分を基準にして過去に彼を見ます。クライアントの予測については、以前に詳しく説明しました。



したがって、クライアントの予測は1つの問題を解決しますが、別の問題を引き起こします。プレイヤーが敵が過去にいた場所で発砲した場合、サーバーが同じ場所にいない可能性があります。サーバーの遅延補償はこの問題を解決しようとしています。武器を発砲すると、サーバーは、ショット時にローカルでプレイヤーが見たゲームの状態を復元し、本当に敵を攻撃できるかどうかを確認します。答えが「はい」の場合、敵がサーバーのこの時点でなくなっていても、ヒットはカウントされます。



この知識を武器に、Dino Squadにサーバーベースのラグ補償の実装を開始しました。まず、クライアントに表示されたものをサーバーに復元する方法を理解する必要がありましたか？そして、正確に何を復元する必要がありますか？私たちのゲームでは、武器と能力のヒットは、レーキキャストとオーバーラップを通じて、つまり、敵の物理的なコライダーとの相互作用を通じて計算されます。したがって、プレーヤーがローカルで「見た」これらのコライダーの位置は、サーバーで再生する必要がありました。当時、私たちはUnityバージョン2018.xを使用していました。物理APIはそこで静的であり、物理世界は単一のコピーに存在します。彼を後で箱から戻すことができるように彼の状態を保存する機会はありません。じゃあ何をすればいいの？

これらすべての要素を組み合わせて、物理的な世界の状態を適切な瞬間にロールバックできる「タイムマシン」を手に入れました。コードは簡単です： 



public class TimeMachine : ITimeMachine
{
     //  
     private readonly IGameStateHistory _history;

     //    
     private readonly ExecutableSystem[] _systems;

     // ,      
     //    
     private readonly GameState _presentState;

     public TimeMachine(IGameStateHistory history, GameState presentState, ExecutableSystem[] timeInitSystems)
     {
         _history = history; 
         _presentState = presentState;
         _systems = timeInitSystems;  
     }

     public GameState TravelToTime(int tick)
     {
         var pastState = tick == _presentState.Time ? _presentState : _history.Get(tick);
         foreach (var system in _systems)
         {
             system.Execute(pastState);
         }
         return pastState;
     }
}

public sealed class LagCompensationSystemGroup : ExecutableSystem
{
     // 
     private readonly ITimeMachine _timeMachine;

     //  
     private readonly LagCompensationSystem[] _systems;
     
     //  
     private readonly TimeTravelMap _travelMap = new TimeTravelMap();

    public LagCompensationSystemGroup(ITimeMachine timeMachine, 
        LagCompensationSystem[] lagCompensationSystems)
     {
         _timeMachine = timeMachine;
         _systems = lagCompensationSystems;
     }

     public override void Execute(GameState gs)
     {
         //      ,
         //     «».     ,
         //          .
         var buckets = _travelMap.RefillBuckets(gs);

         for (int bucketIndex = 0; bucketIndex < buckets.Count; bucketIndex++)
         {
             ProcessBucket(gs, buckets[bucketIndex]);
         }

         //       
         //  
         _timeMachine.TravelToTime(gs.Time);
     }

     private void ProcessBucket(GameState presentState, TimeTravelMap.Bucket bucket)
     {
         //      
         var pastState = _timeMachine.TravelToTime(bucket.Time);

         foreach (var system in _systems)
         {
               system.PastState = pastState;
               system.PresentState = presentState;

               foreach (var entity in bucket)
               {
                   system.Execute(entity);
               }
          }
     }
}


詳細を設定するだけでした：



1.時間内の最大移動範囲を制限する量を理解します。



モバイルネットワークの状態が悪いときにゲームにできるだけアクセスできるようにすることが重要だったため、ストーリーを30ティックのマージン（ティックレートは20 Hz）に制限しました。これにより、プレイヤーは非常に高いpingでも対戦相手にヒットすることができます。



2.時間内に移動できるオブジェクトと移動できないオブジェクトを決定します。



明らかに、私たちは敵を動かしています。しかし、インストールされているエネルギーシールド、たとえば、ありません。ここでは、ネットワークシューターでよく行われるように、防御能力を優先する方が良いと判断しました。プレイヤーが現在シールドをすでに設置している場合、過去の遅延補正された弾丸はそれを通り抜けるべきではありません。



3.恐竜の能力（バイト、テールパンチなど）を補う必要があるかどうかを判断します。必要なものを決定し、弾丸と同じルールに従ってそれらを処理しました。



4.遅延補正を実行するプレーヤーのコライダーをどうするかを決定します。良い意味で、彼らの立場は過去にシフトされるべきではありません：プレイヤーは彼が今サーバーにいるのと同じ時間に自分自身を見なければなりません。それでも、射撃プレイヤーのコライダーもロールバックしますが、これにはいくつかの理由があります。



まず、クラスタリングが向上します。pingが近いすべてのプレーヤーに同じ物理状態を使用できます。



第2に、すべてのレーキキャストとオーバーラップで、アビリティまたは発射体を所有するプレーヤーのコライダーを常に除外します。 Dino Squadでは、プレイヤーはシューティングゲームの基準により、かなり珍しいジオメトリを持つ恐竜を操作します。プレーヤーが異常な角度で発射し、弾丸の軌跡がプレーヤーの恐竜コライダーを通過しても、弾丸はそれを無視します。



3番目に、遅延補償の開始前であっても、ECSからのデータに基づいて、恐竜の武器の位置または能力の適用ポイントを計算します。



その結果、ラグ補正されたプレーヤーのコライダーの実際の位置は重要ではないため、生産性を向上させると同時に、より簡単な方法を採用しました。



ネットワーク遅延は単に取り除くことはできず、マスクすることしかできません。他の偽装方法と同様に、サーバーの遅延補償には独自のトレードオフがあります。撃たれているプレイヤーを犠牲にして、シューティングプレイヤーのゲーム体験を向上させます。しかし、Dino Squadにとって、ここでの選択は明白でした。



もちろん、プログラマーとゲームデザイナの両方にとって、サーバーコード全体の複雑さが増すにつれ、これらすべての費用を支払う必要がありました。以前のシミュレーションがシステムの単純な順次呼び出しであった場合、ラグ補償を使用すると、ネストされたループと分岐がその中に現れました。作業をしやすくするために、多くの労力を費やしました。



2019バージョン（またはおそらく少し前のバージョン）では、Unityは独立した物理シーンを完全または完全にサポートしています。すべての部屋に共通する物理的な世界をすばやく取り除きたいので、更新のほぼ直後にサーバーに実装しました。



各ゲームルームに独自の物理シーンを与えたので、シミュレーションを計算する前に、隣の部屋のデータからシーンを「クリア」する必要がなくなりました。まず、生産性が大幅に向上しました。次に、プログラマーが新しいゲーム要素を追加するときにシーンをクリアするためのコードを間違えた場合に発生するバグのクラス全体を取り除くことができました。このようなエラーはデバッグが難しく、ある部屋のシーンの物理オブジェクトの状態が別の部屋に「流れ込んだ」という事実につながることがよくありました。



さらに、物理的なシーンを使用して物理的な世界の履歴を保存できるかどうかについても少し調査しました。つまり、条件付きで、各部屋に1つのシーンではなく30のシーンを割り当て、それらから履歴が格納される循環バッファを作成することです。一般的に、オプションは機能することが判明しましたが、導入しませんでした。生産性が大幅に向上することはありませんでしたが、かなり危険な変更が必要でした。非常に多くのシーンでの長時間の作業中にサーバーがどのように動作するかを予測することは困難でした。したがって、「壊れていない場合は修正しない」というルールに従いました。
