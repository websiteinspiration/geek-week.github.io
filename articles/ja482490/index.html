おそらく、マイクロコントローラ用のプログラムの開発者は、コードのセキュリティと移植性を向上させるために設計された特別なコーディング標準について、少なくとも一度は聞いたことがあるでしょう。そのような標準の1つがMISRAです。この記事では、この標準とは何か、その標準とは何か、プロジェクトでどのように使用するかについて詳しく説明します。

読者の多くは、PVS-StudioがMISRA規格に従ってアラートの分類をサポートしていると聞いています。現在、PVS-Studio はMISRA C：2012およびMISRA C ++：2008の100以上のルールをカバーしています。

MISRAストーリー


MISRAの歴史はずっと前に始まりました。次に、90年代初頭、「安全IT」という表示名のもとに英国政府のプログラムが、電子システムのセキュリティに関連するさまざまなプロジェクトに資金を割り当てました。 MISRA（モーター産業ソフトウェア信頼性協会）プロジェクト自体は、陸上車両-自動車全般のマイクロコントローラー用ソフトウェアを開発するためのガイドを作成するために設立されました。

州からの資金提供を受けて、MISRAチームは作業を開始し、1994年11月までに最初のガイドを発表しました：「車両ベースのソフトウェアの開発ガイドライン。」このガイドでは、まだ特定の言語に縛られていないが、私は認めなければならない：。仕事が印象的だったが、それはおそらく組み込みソフトウェアを開発する考えられる全ての側面を懸念ところで、このガイドの開発者は、最近の25周年祝った彼らのために、このような重要な日付を。

とき州からの資金提供が終わり、MISRAメンバーは非公式なベースで一緒に作業を続けることを決定しました-これは今日まで続きます。

哲学とルール例


MISRAの最も特徴的な機能は、細部への信じられないほどの注意と安全を確保する際の細心の注意です。著者たちは、頭に浮かんだすべての「穴」CおよびC ++を1か所に集めただけでなく（たとえば、CERTの著者として）、これらの言語の国際標準を慎重に検討し、考えられるすべての考えられない方法で間違いを犯した方法を書き留めました。次に、上からコードの読みやすさに関するルールを適用して追加しました。これにより、既にクリーンなコードに新しいエラーを導入することがより困難になります。

深刻さの程度を理解するには、標準から取られたいくつかのルールを検討してください。

一方では、プロジェクトの目的に関係なく、常に従うべき適切で適切なルールが数多くあります。ほとんどの場合、これらは未定義/未指定/実装依存の動作を排除するように設計されています。例えば：

  • 初期化されていない変数の値を使用しないでください
  • ストリームを閉じた後はFILEポインターを使用しないでください
  • すべての非void関数は値を返す必要があります。
  • ループカウンターは浮動小数点型であってはなりません
  • …など

一方、ルールはありますが、その利点は表面にありますが、（通常のプロジェクトの観点から）どれを破ることはそれほど罪深いものではありません。

  • gotoとlongjmpは使用しないでください
  • 各スイッチはデフォルトで終了する必要があります
  • 到達できないコードを書かない
  • 変数関数を使用しない
  • アドレス演算を使用しない（[]および++を除く）
  • ...

このようなルールも悪くはありません。以前のルールと組み合わせると、すでにセキュリティが目に見えて向上していますが、責任の高い組み込みシステムにはこれで十分ですか？自動車産業だけでなく、航空機製造、航空宇宙、軍事、医療にも使用されています。

ソフトウェアエラーのために、一部のX線装置が患者に20,000 radの線量を照射することを望まないため、通常の「日常的な」ルールではもはや十分ではありません。人命と莫大な金が危機に瀕しており、細心の注意を払わなければならない。ここに残りのMISRAルールが登場します：
  • リテラルの接尾辞「L」は常に大文字にする必要があります（小さな「l」は単位と混同される可能性があります）
  • カンマ演算子は使用しないでください（間違いをする可能性が高くなります）
  • ( )
  • if, else, for, while, do, switch ( )
  • ( , )
  • … , .

多くの場合、MISRAに遭遇した人々は、規格の哲学は「これを禁止し、これを禁止すること」であるという意見を持っています。実際、これはそうですが、一部だけです。

はい、標準には多くの同様のルールがありますが、その目的は可能な限りすべてを禁止することではなく、コードのセキュリティに何らかの形で違反するすべての方法をリストすることです。ほとんどのルールでは、ルールに従うかどうかを選択します。これについて詳しく説明します。

MISRA Cでは、ルールは3つの主要なカテゴリに分類されます：必須、必須、および勧告。必須-これらはいかなる口実にも違反してはならない規則です。たとえば、このセクションには、「初期化されていない変数の値を使用しない」というルールが含まれています。必要なルールはそれほど厳格ではありません。拒否の可能性を許可しますが、これらの逸脱が注意深く文書化され、書面で正当化される場合に限ります。残りのルールはアドバイザリカテゴリに含まれます-これらは従う必要のないルールです。

MISRA C ++は少し異なります。必須のカテゴリがありません。ほとんどのルールは必須のカテゴリに属しています。したがって、実際には、あなたはあらゆる規則に違反する権利を持っています-逸脱を文書化することを忘れないでください。「ドキュメント」というカテゴリもあります。これらは必須のルールであり（逸脱は許可されません）、「アセンブラの使用ごとに文書化する必要がある」または「プラグインライブラリはMISRA C ++に準拠する必要がある」などの一般的な慣行に関連付けられています。

他に何がありますか


実際、MISRAは一連のルールだけで構成されているわけではありません。実際、これはマイクロコントローラー向けの安全なコードを書くためのマニュアルであり、あらゆる種類の有用性に満ちています。詳しく見てみましょう。

まず、標準には背景のかなり完全な説明が含まれています。標準が作成された理由、CまたはC ++が選択された理由、これらの言語の利点と欠点。

私たちはこれらの言語の長所をよく知っています。はい、そして一般的には欠点についても:)非常に複雑で、標準と構文の仕様が不完全で、間違いを犯しやすく、長時間エラーを探します。たとえば、誤って次のように書く可能性があります。

for (int i = 0; i < n; ++i);
{
  do_something();
}

それでも、余分なセミコロンに気付かない可能性がありますよね？次のように書くこともできます：
void SpendTime(bool doWantToKillPeople)
{
  if (doWantToKillPeople = true)
  {
    StartNuclearWar();
  }
  else
  {
    PlayComputerGames();
  }
}

1番目と2番目の 両方のケースがMISRAルールで簡単にキャッチされるのは良いことです（1番目はMISRA C：13.4 / MISRA C ++：6.2.1、2番目はMISRA C：13.4 / MISRA C ++：6.2.1）。

問題を説明することに加えて、この標準には、開始する前に知っておくべきこと、MISRA開発プロセスの設定方法、スタティックアナライザーを使用したコードのコンプライアンスチェックの方法、保持する必要のあるドキュメント、および方法に関する多くのヒントが含まれています埋めるなど。

また、最後には、アプリケーションのリストが含まれています。ルールの短いリストと要約表、C / C ++の脆弱性の小さなリスト、ルールからの逸脱に関するドキュメントの例、およびこの官僚機構全体で迷子にならないように設計されたいくつかのチェックリスト。

ご覧のように、MISRAは単なるルールのセットではなく、組み込みシステム用の安全なコードを作成するためのほぼ全体のインフラストラクチャです。

プロジェクトで使用する


ある状況を想像してみてください。非常に必要で責任のある組み込みシステム用のプログラムを作成します。または、すでにプログラムを持っていますが、それをMISRAに「転送」する必要があります。コードが標準に準拠しているかどうかを確認するにはどうすればよいですか？本当に手動で行う必要がありますか？

手動でのコード検証は、簡単で潜在的に不可能な作業ではありません。各レビュアーは、すべてのコード行を注意深く調べる必要があるだけでなく、標準についても詳細に知る必要があります。ホラー！

したがって、MISRA開発者自身は、静的分析を使用してコードをテストすることをお勧めします。実際、本質的に、静的分析は自動化されたコードレビュープロセスです。プログラムでアナライザーを実行するだけで、数分で標準違反の可能性に関するレポートを受け取ります。必要なものですよね？ログを表示して、応答を修正するだけです。

次の質問は、MISRAを使い始めるのはいつですか。答えは簡単です。早いほど良いです。理想的には、コードを書き始める前に、MISRAはコードの存続期間を通じて標準に従うことを想定しているためです。

もちろん、最初からMISRAを作成できるとは限りません。たとえば、プロジェクトがすでに部分的または完全に実装されていることがよくありますが、顧客はプロジェクトが標準に準拠することを望んでいます。この場合、既存のコードを完全にリファクタリングする必要があります。

ここで落とし穴が現れます。水中の岩がポップアップすることもあるでしょう。静的アナライザーを使用して、「通常の」プロジェクトでMISRA規格に準拠しているかどうかを確認するとどうなりますか？スポイラー：あなたは怖くなることができます。



もちろん、写真の例は誇張されています。ここでは、十分に大きなプロジェクトをチェックした結果を確認できます。実際には、マイクロコントローラーで作業することは考えられていません。それにもかかわらず、既存のコードをチェックすると、1、2、5、1万のアナライザー操作が表示される場合があります。そして、このすべてのヒープで、作成または変更されたばかりのコードに対して発行された新しい操作は失われます。

それについて何をしますか？すべてのものを脇に置き、すべての古いポジティブを修正するために座る必要がありますか？

私たちは、プロジェクトが初めてチェックされたとき、多くのポジティブがかなり頻繁に現れることを知っており、作業を停止することなく、アナライザーからすぐにメリットを得るのに役立つソリューションを開発しました。このソリューションは「抑制ベース」と呼ばれます。

Suppress-baseは、アナライザーメッセージの大量抑制を可能にするPVS-Studioメカニズムです。初めてプロジェクトをチェックしてそこに数千のポジティブを発見した場合は、それらを抑制ベースに追加するだけで、次にアナライザーを実行したときに警告が表示されなくなります。

したがって、通常の方法でコードの記述と変更を続けることができ、同時に、プロジェクトに導入されたばかりのエラーについてのみ警告が表示されます。同時に、古いエラーをかき集めることによって気を散らされることなく、今すぐアナライザーから最大のメリットを得ることができます。数回クリックするだけで、アナライザーが実装されます。これに関する詳細な説明は、こちらで読むことができます。

あなたはおそらく尋ねるかもしれません：「待って、隠された応答をどうするか？」答えは非常に簡単です。忘れないで、ゆっくり修正してください。たとえば、バージョン管理システムに抑制ベースを配置し、操作の数を増加させないコミットのみを許可することができます。したがって、徐々に「水中の岩」は遅かれ早かれ排水され、その痕跡はなくなります。

さて、アナライザーが実装され、続行する準備が整いました。次はどうする？明確なビジネス-コードを操作します。しかし、標準への準拠を宣言するには何が必要ですか？プロジェクトがMISRAに準拠していることをどのように証明しますか？

実際、コードがMISRAに準拠しているという特別な「証明書」はありません。規格自体で規定されているように、コンプライアンスコードの追跡は、ソフトウェアの顧客とソフトウェアの提供者の2者で行う必要があります。サプライヤーは、標準に準拠し、必要なドキュメントに記入するソフトウェアを開発します。顧客は、自分の側で、これらのドキュメントのデータが正しいことを確認する必要があります。

あなた自身があなた自身のソフトウェアの顧客であり開発者でもある場合、標準への準拠の責任はあなたの肩にのみあります:)

一般に、プロジェクトの適合性を証明するには、証拠書類が必要です。プロジェクト開発者が準備する必要があるドキュメントのリストは異なる場合がありますが、MISRAは参考としていくつか提供しています。このセットをさらに詳しく見てみましょう。

標準への準拠を主張するには、いくつかのものが必要です。

  • 実際には、コードが必須および必須のルールに準拠しているプロジェクト
  • ガイド実施計画
  • すべてのコンパイラと静的アナライザの警告のドキュメント
  • 必要なルールからのすべての逸脱の文書化
  • ガイドラインコンプライアンスの概要

1つはコンプライアンス計画です。これはあなたの最も重要な表であり、審査官を他のすべての文書に送るのはそれです。最初の列にはMISRAルールのリストがあり、残りにはこれらのルールからの逸脱が特定されているかどうかが示されています。このテーブルは次のようになります。


この規格では、複数のコンパイラーを使用してプロジェクトをビルドすること、および2つ以上の静的アナライザーを使用してコードのコンプライアンスをチェックすることを推奨しています。コンパイラーまたはアナライザーがルールに関連付けられたある種の警告を生成する場合は、表と文書でこれに注意する必要があります。操作を削除できない理由、それがfalseかどうかなどです。

静的アナライザーでチェックできないルールがある場合は、コードレビュー手順を実行する必要があります。この手順も文書化する必要があります。その後、この文書へのリンクをコンプライアンス計画に追加する必要があります。

コンパイラーまたは静的アナライザーが正しいことが判明した場合、またはコードレビュープロセスで実際のコード違反が見つかった場合は、それらを修正するか文書化する必要があります。ここでも、表のドキュメントへのリンクを添付します。

したがって、コンプライアンス計画は、コードで識別された逸脱のドキュメントを見つけることができるドキュメントです。

次に、ルールからの逸脱の文書化について少し説明します。すでに述べたように、そのようなドキュメントは必須ルールにのみ必要です。必須レベルのルールに違反することはできず、アドバイザリルールはドキュメントなしでは従うことができないためです。

ルールから逸脱する場合は、ドキュメントに以下を含める必要があります。

  • 壊れたルール番号
  • 正確な偏差位置
  • 偏差の妥当性
  • 逸脱してもセキュリティが損なわれないことの証拠
  • 潜在的なユーザーへの影響

ご覧のように、このドキュメントへのアプローチは、違反の価値があるかどうかを真剣に考えさせます。これは、特にRequired-rulesに違反するために行われたものではありませんでした:)

中央の列はコードの操作を開始する前に入力され、右端の列はプロジェクトの準備ができた後です。

質問するのはかなり理にかなっています。ルール自体がすでに標準自体で指定されているのに、なぜルールのカテゴリを指定する必要があるのですか？実際、この規格では、より厳密なカテゴリでのルールの「増加」が許可されています。たとえば、顧客がアドバイザリルールをカテゴリに移動するように依頼する場合があります。このような「増加」は、コードで作業する前に行う必要があり、ルールの遵守の概要により、これを明確に示すことができます。

最後の列では、すべてが単純です。ルールが使用されているかどうかを確認するだけで十分です。使用されている場合は、ルールからの逸脱があります。

このテーブル全体が必要なのは、ルールの優先順位と、コードがそれらに一致するかどうかをすばやく確認できるためです。拒否の正確な理由を知りたくなった場合は、いつでもコンプライアンス計画を参照して、必要なドキュメントを見つけることができます。

標準の未来


最後に、規格の将来について少しお話ししたいと思います。

MISRAは現在、発展途上にあります。たとえば、2019年の初めに、「MISRA C：2012第3版（最初の改訂）」が発表されました。更新され、2012年の新しい規則によって補足されています。同時に、彼らはMISRA C：2012 Amendment 2-C11 Coreの次期リリースを発表しました。これにはルールが追加され、2011年と2018年のC言語の最初のバージョンをカバーします。

静止せず、MISRA C ++。ご存知のように、最新のMISRA C ++標準は2008年にさかのぼるため、対象となる言語の最も古いバージョンはC ++ 03です。このため、MISRAに似た別の規格が登場し、AUTOSAR C ++と呼ばれています。もともとはMISRA C ++の続きとして考えられ、それ以降のバージョンの言語をカバーすることを目的としていました。首謀者とは異なり、AUTOSAR C ++は年に2回更新され、現在C ++ 14をサポートしています。 C ++ 17、次にC ++ 20へとさらにアップグレードする予定です。

他の標準について何を始めましたか？事実は、1年弱前に、両方の組織が標準を1つに統合することを発表したことです。これでMISRA C ++とAUTOSAR C ++は単一の標準になり、一緒に開発されるようになります。これは、C ++マイクロコントローラーで作成している開発者にとっては素晴らしいニュースであり、静的アナライザーの開発者にとっても同じくらい素晴らしいニュースだと思います。まだお気に入りの作品がたくさんあります！:)
