いくつかのアナリストに手を貸します：典型的な銀行業務の自動化のためのAPI Livy

こんにちは、ハブル！

銀行が顧客のソルベンシーを評価するためにさまざまなソース（信用調査会社、携帯電話会社など）からのデータを使用することは秘密ではありません。外部パートナーの数は数十に達する可能性があり、私たちのチームのアナリストは数人だけを募集します。小さなチームの作業を最適化し、日常的なタスクをコンピューティングシステムに転送するという問題が発生します。

このデータが銀行にどのように送られ、アナリストのチームがこのプロセスを監視する方法を分析します。



順番に始めましょう。

Hadoopに基づく分散システムとそれに関連するすべてのプロセスを、SmartDataと簡単に呼びます。SmartDataは、外部エージェントからAPIデータを受信します。（さらに、その代理人は銀行の外部パートナーと内部システムの両方です）。もちろん、クライアントごとに「関連プロファイル」を収集しておくと便利です。ソースからの更新されたデータはOperprofilに分類されます。OperprofileはCustomer 360のアイデアを実装し、Hbaseテーブルとして保存されます。クライアントとのさらなる作業に便利です。

顧客360
Customer 360-組織のすべてのプロセスで使用されるクライアントデータのすべての種類の属性を備えた運用ストレージを実装するためのアプローチ。

エージェントとの連携は継続中であり、監視する必要があります。インタラクションの品質とヒット率、およびこの情報を他のチームに転送する容易さをすばやく確認するために、たとえばTableauのレポートなどの視覚化を使用します。

ソースデータはKafkaに送信され、前処理されて、HDFSに基づいて構築されたDataLakeに配置されます。 HDFSからのログファイルの解析、その処理、および分析と視覚化システムへの毎日のアップロードを整理する方法を考え出す必要がありました。また、これをPythonラップトップのアナリストの愛情と組み合わせます。

内部キッチンで終了し、練習に移ります。

私たちのソリューションは、Livy APIを使用することでした。Livyでは、Jupyterから直接クラスターにコードを送信できます。Python（またはScala）で記述されたコードとメタデータを含むHTTPリクエストがLivyに送信されます。Livyは、Yarnリソースマネージャーによって管理されるクラスターでSparkセッションの起動を開始します。要求モジュールは、HTTP要求の送信に適しています。サイトを解析したい人はおそらく彼にすでに慣れているでしょう（そうでない場合は、ここで彼について少し学ぶ機会があります）。

必要なモジュールをインポートしてセッションを作成します。（セッションのアドレスもすぐにわかります。将来的には便利になります）。ユーザー認証のデータと、クラスターが実行するスクリプト言語の名前をパラメーターに転送します。

import json, requests, schedule, time

deleteメソッドはセッションを削除します。

requests.delete(session_url, headers=headers)

毎日のアップロードには、いくつかのオプションを使用できます。それらはハブのcronについてすでに書いていますが、ユーザーフレンドリーなスケジュールモジュールについては-いいえ。コードに追加するだけで、説明は必要ありません。また、便宜上、すべての計算を1か所にまとめます。

import json, requests, schedule, time
 
schedule.every().day.at("16:05").do(job, 300)
while True:
    schedule.run_pending()
 
def job(wait_time):
    host = 'http://***:8998'
    data = {'kind': 'spark', 'proxyUser': 'user'}
    headers = {'Content-Type': 'application/json'}
    r = requests.post(host + '/sessions', data=json.dumps(data), headers=headers)
 
    session_id = r.json().get('id')
    print("session_id: " + str(session_id))
    session_url = host + r.headers['location']
    r = requests.get(session_url, headers=headers)
 
    timeout = time.time() + wait_time
    sess_state = ['starting', 'success', 'idle']
    while(True):
        time.sleep(7)
        req_st = requests.get(session_url, headers=headers).json().get('state')
        if req_st != 'idle' and time.time() > timeout:
            requests.delete(session_url, headers=headers)
            break
        if req_st == 'idle':
            break
        if req_st not in sess_state:

def send_message(subject, text):
    import smtplib
    from email.mime.multipart import MIMEMultipart
    from email.mime.text import MIMEText
    me = "my_email_adress"
    you = "email_adress"
    msg = MIMEMultipart('alternative')
    msg['Subject'] = subject
    msg['From'] = me
    msg['To'] = you
    text = text
    part1 = MIMEText(text, 'plain')
    msg.attach(part1)
    s = smtplib.SMTP('domain.org')
    s.ehlo()
    s.starttls()
 
    s.login("user", "password")
    s.sendmail(me, you, msg.as_string())
    s.quit()


結論：


おそらく、このソリューションは最善であるとは言えませんが、アナリストのチームには透過的です。私がそれに見ている長所：

  • 自動化に使い慣れたJupyterを使用する機能
  • 視覚的相互作用
  • チームメンバーは、ファイルの操作方法（spark-zoo）を選択する権利を持っているため、既存のスクリプトを書き換える必要はありません。

もちろん、多数のタスクを開始するときは、解放されたリソースを監視し、アンロード間の通信を構成する必要があります。これらの問題は個別に解決され、同僚と合意しています。

少なくとも1つのチームがこの決定を下すことは素晴らしいことです。
