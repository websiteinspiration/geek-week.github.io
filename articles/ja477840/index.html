ゼロデイ脅威


ゼロデイ脅威とは、開発者によって許可されたがまだ発見されていないギャップや脆弱性を識別する用語です。脆弱性が修正されるまで、ネットワークへのアクセス、コンピュータのリモート制御、データ操作などに使用できます。この用語の名前は、まだ誰も知らないため、開発者が欠陥を修正する日がないために確立されました。かつては、Adobe、Windows、Torブラウザーなどの大企業やソフトウェアなど、多くの脆弱性がありました。

一部の組織は幸運でした、それらの脆弱性はそれを使用するつもりのない人々によって発見され、彼らは単に問題に関する情報を提供しました。たとえば、これはMacOSの。または、新機能に加えて、誤ってゼロデイ脅威を修正するアップデートがありました。

しかし、他の状況がありました。たとえば、Google Chromeは、攻撃者が被害者のデバイスで任意のコードをリモートで実行できるようにする脆弱性を一度に緊急に修正する必要がありました。

この脅威の問題は、それを100％防御することが不可能であることです。これは、まだ知られていないものに対して防御することが難しいためです。ただし、プロジェクトでこのような脅威の可能性を低減する方法はあります。後で説明しますが、少し理論から始めます。

CVE CWE


Common Vulnerabilities and Exposures（CVE）は、サイバー犯罪者が使用できるソフトウェアエラーのデータベースです。 CVEは、既知のソフトウェアの欠陥を合理化するために作成されました。ほとんどの情報セキュリティツールは独自のデータベースと名前を使用しており、この混乱を解消し、さまざまなツールとの互換性を追加するために、1999年にMITERがCVEを作成しました。ただし、CVEではコードのセキュリティを評価するには不十分でした。これには、問題の詳細な説明と、彼女よりも粗雑でない、より正確なものが必要です。したがって、これらの要件を満たすCommon Weakness Enumeration（CWE）ベースが作成されました。エラーがCWEリストにある場合、攻撃者が悪用してCVEリストに入れる可能性のある脆弱性につながる可能性があります。明確にするために、以下のオイラー図を見ることができます。


一部の静的アナライザーは、たとえば、プロジェクトが脆弱性が見つかったライブラリーを使用していることを開発者に通知できます。これにより、脆弱性が既に修正されているライブラリの新しいバージョンを選択し、他の誰かのコードから生じる脅威の問題の可能性を減らすことができます。

CVEおよびCWEリストの開発の出現により、静的アナライザーを含む多くの情報セキュリティツールがそれらのサポートを処理してきました。このようなアナライザーは、SASTソリューションと見なすことができます。 SAST（静的アプリケーションセキュリティテスト）を使用すると、開発者は、ソフトウェア開発ライフサイクルの初期段階にあるアプリケーションソースコードの脆弱性を見つけることができます。

開発でSASTを使用することは、ゼロデイ脅威の可能性を最小限に抑える別のオプションです。 CWEに従ってエラーを分類するアナライザーは、潜在的な脆弱性がどこに隠れているかを知ることができます。そして、これらのエラーを修正することにより、開発者はアプリケーションの信頼性を高め、ゼロデイ脅威の可能性を減らします。

静的セキュリティテストにはさまざまなツールがあります。脆弱性に対処する機能を示すために、PVS-Studioツールについて詳しく見てみましょう。このアナライザーからの警告は、CWEとして分類できます。いくつかの例を見てみましょう。

警告PVS-Studio：CWE-561：Dead Code（V3021）。

public string EncodeImage(....)
{
  if (string.IsNullOrWhiteSpace(inputPath))
  {
    throw new ArgumentNullException("inputPath");
  }
  if (string.IsNullOrWhiteSpace(inputPath))
  {
    throw new ArgumentNullException("outputPath");
  }
  ....
}

このコードは誤ってタイプミスをしました。 2つのif条件では、同じ変数がチェックされます。生成された例外から判断すると、2番目の条件では、変数outputPathをチェックする必要があります。その結果、コードの一部に到達できません。

このようなエラーは一見無害に見えます。ただし、この印象は非常に誤解を招く可能性があります。goto演算子の複製に関連する非常に単純で、一見すると無害なエラーについて考えてみます。

かつて、このエラーはiOSオペレーティングシステムに脆弱性を引き起こしました。

脆弱性の説明CVE-2014-1266：Apple iOS 6.xより前の6.1.6および7.xより前の7.x、Apple TV 6.xより前の6.0.2のデータセキュリティコンポーネントのセキュアトランスポート機能のlibsecurity_ssl / lib / sslKeyExchange.cにあるSSLVerifySignedServerKeyExchange関数、および10.9.2より前のApple OS X 10.9.xはTLS Server Key Exchangeメッセージの署名をチェックしないため、中間者攻撃者が署名ステップに任意の秘密鍵を使用してSSLサーバーを偽装したり、署名ステップを省略します。

二重gotoが原因で、コードに到達できない状況も発生します。条件に関係なく、2番目のgoto ステートメントはif ステートメントで実行されます。これにより、署名の検証が行われなくなります。この関数は0を返します。つまり、署名に問題がない場合でも、プログラムはサーバーからキーを受け取ります。このキーは、送信中にデータを暗号化するために必要です。

そのような単純な間違いの結果は非常に深刻でした。したがって、これまたはそのエラーがどれほど危険であるかをCWEとして分類しても意味がありません。修正するだけで、コードの安全性が高まります。

ちなみに、上記のエラーはPVS-Studioアナライザーで簡単に検出できます。彼はここですぐに2つのCWE警告を出します。

  • CWE-561 (V779): Dead Code
  • CWE-483 (V640): Incorrect Block Delimitation

別の例を見てみましょう。 2012年に、攻撃者がMySQLデータベースに侵入する可能性があるMySQLのセキュリティ問題について知られました。この理由となったコードのスニペットを提供します。CVE-2012-2122の

説明：sql / password.c MySQL 5.1.x以前の5/1 / 63、5.5.x以前の5.5.24、および5.6.x以前の5.6.6、および5.1.x MariaDB以前の05/01/62、 5.2.xより前の5.2.x、5.3.xより前の5.3.6、および5.5.xより前の5.5.23は、memcmp関数の特定の実装がある特定の環境で実行されている場合、リモートの攻撃者が同じ不正なパスワード。不適切にチェックされた戻り値が原因で、最終的にトークンの比較が成功します。

typedef char my_bool;
my_bool

  return memcmp(hash_stage2, hash_stage2_reassured, SHA1_HASH_SIZE);

memcmp 関数の戻り値の型はintで、check_scramble関数の戻り値の型はmy_bool、実際にはcharです。その結果、intがcharにキャストされ、最上位ビットが破棄されます。これは、256のうち約1ケースで、ユーザー名を知っていれば、任意のパスワードで接続できるという事実につながりました。

繰り返しますが、このCWEエラーは無効化され、コードの作成段階でもCVEに変換されないようにすることができます。たとえば、PVS-Studioスタティックアナライザーは次の警告を生成します：CWE-197（V642）：数値切り捨てエラー。

結論


0日間の脆弱性-保護が保証されていないもの。しかし、それらが発生する可能性は大幅に低減できます。これには、PVS-Studioなどの特殊なSASTソリューションを使用できます。プロジェクトがCWEとして分類できるエラーを検出した場合は、それらに注意を払い、修正する必要があります。CWEエラーを排除することでCVEリストを埋めるCWEはごくわずかですが、多くの潜在的な脅威からアプリケーションを保護します。
