開発者が議論しない解決策を見せてください、そして私はあなたにビールを置きます



2つの調整できないキャンプがあります。一部の人は言う：あなたはあなたのコミットを正確かつ有能に説明しなければなりません。各コミットは、完全で意味のある作業です。コミットを明確かつ簡単に説明できない場合は、コミットが間違っています。

他の人は、あなたが好きなようにコミットすることを考えています、これはあなたの個人的なワークフローの一部です。しかし、要求のプールは詳細に説明されています。何が行われ、どのように行われ、なぜ行われるのですか。テストしたように、どの問題が解決するか、何に注意を払うべきか。

私は2番目のアプローチの頑固な支持者です-私の仕事をマイクロピースに打ち負かすのは私にとって不便です。私は小さなタスクを取り、ランダムにコードベースを駆け巡り、実際に行われた順序で実験して変更を加えます。ボタンをクリックして、私の仕事が適切なコミットに再構築されるとしたら、それをクリックします。しかし、ボタンはありませんが、自分を壊したくありません。同時に、私は要求のプールを説明する特定のスキルに達しました。たまたまマイクロソフトのコードを見つけました（アウトスタッフを通じて、それはカウントされません）。そこで、リクエストのプールを処理するための最高の標準を手に入れました。長年にわたって、私はこの実践を発展させてきました。通常、私は何とかそのようなアプローチを使用するようにチームを説得することができました。

しかし、最後の仕事で、詳細なコミットの強力なサポーターであるdevlidを取得しました。ああ、長い間議論した。私の部下の役割が役割を果たしました、主なものに同意するソフコフスキーの癖は黙るのは簡単ではありません。私はいつものようにはっきりとしたものではなく、両方の刃に横たえられました。積み重ねられていますが、確信はありません。


突然、開発者が単一の政府に団結し、ルールを確立し、特定のアプローチが非合法であると宣言し始めると、内戦がその翌日に始まります。開発者の世界では、目的と測定可能性への信念があるからです。人間の知覚から独立した真実では。そして、これらすべてが存在する場合、それは紛争当事者の1人が明らかに客観的に間違っていることを意味します。

私の最初の深刻な労働争議はかなり早い時期に起こりました。私はまだ緑の6月だった。私のチームの実際の開発者の1人がPRを投げました。チームメンバー全員がレビューコードを作成するという習慣がありました。コードを開いたところ、すぐに問題が発生しました。ある数値をとる関数のテストを男が書いた。したがって、彼は彼女に0、1000、およびランダム（0、1000）を与えました。

その時、他のチームメイトが私をバカな新人だと思っていることを痛感しました。そして彼は彼のビジョンで彼らを汚すために少し待った。私はラッキーだった-テストでランダム！

私はユニットテストの理論を実際には理解していませんでしたが、数冊の本を読んでしっかりと覚えました。同じコードベースで同じユニットテストを実行しても、同じ結果が得られるはずです。有毒に見えないようにコメント全体を1時間ほど考えましたが、昨日数えるように教えられたサルだけがそのような決定を下すことができることが明らかになりました。もちろん、結局のところ、彼は昨日インターンが自分たちを絞り込んでいる開発者であると想像していたすべてのもののように、非常に愚かに見えました。

翌日、私は地獄へのポータルを開いたことに気づきました。私のコメントの下にはさらに100がありました。残りの日は、6人の開発者のチームではなく、6匹のサルのチームでした。全員がロシア人でしたが、PR自体では英語で議論しました。言葉の在庫が十分ではなかったとき、彼らは呼びかけました。彼らは、リンク、本からの引用、研究で互いに向き合って、個人に向けました。同僚は私の英語を嘲笑するために英語の辞書からのスクリーンショットで私のコメントの1つに答えました。しかし、妥協はしていません。

質問は複雑でした。一方では、このパラメーターが何の原因であるかを誰もまったく知りませんでした。私たちの関数は、ドックのない内部ネットワークから他のパラメーターにそれを渡しました。したがって、理論的には、関数は任意の値に該当する可能性があります。つまり、テストがこれをキャッチし、製品のエラーを予測できます。その一方で、ビルドをランダムに処理するという問題がすでにありました。このため、プロジェクトがローカルでビルドされてクラッシュした場合、私たちは愚かに新しいビルドを注文し、ログを確認しませんでした。結局、ビルドには4時間かかり、CIを通過するまでプルリクエストを見ることはできませんでした。

さらに、車はしばしば忙しかった-潜在的な「問題」を探すよりも、まだ注文されている間にビルドを注文する方が実際的でした（ほとんどの場合、統合テストでフライングタイムアウトになることが判明しました）。これは、ランダムテストが失敗しても、注意を払わないことを意味します。

すべては次の取引で解決されました。私たちはチーム全体に電話をかけ、30分の間、ロシア語で主張し続けました。私たちはアメリカ人の開発マネージャーがどのようにして電話に参加したか気づかず、高揚した口論でお互いをぶつけ続けました。彼らが「やあみんな、それは絶対に重要ではない」と聞くまで。そのままマージしてください。」彼が以前どのくらい私たちの言うことを聞いたのかはわかりません。彼が私たちの話していることを理解したので、私たちは突然論争をやめました。彼らは凍ってそれを忘れていました。平和に別れたが、確信はしなかった。


それ以来、私は理解するために経験を積んでいます-たわごと、ランダム、ハンディキャップ、または境界ケースをまったく与えません。このいまいましい単体テストは干渉します-書き直します。一日中議論するよりもすべてが良い。しかし、私が理想的な世界にいると想像すると、正しい判断だけを常に行う必要があるため、どうすればよいかわかりません。私たちはランダム性のあるテストを凍結しましたが、信頼性を優先する選択肢としてこれを理解しましたが、開発の利便性を損なうことになります。

先日も同様のジレンマに直面しました。新しいタイムコードで遊んでこんな感じに使えるものを書いてみました

// LessThan<T>  MoreThan<T> -    ,   . 
// ,        

//      , 
//   ,   100
const consumeNumberLessThan100 = (v: LessThan<100>) => 
    doWork(v);

//     ,   100
const consumeNumberMoreThan100 = (v: MoreThan<100>) => doWork(v);

const sample = (x: number) => {

    //     check  , 
    //   x  100
    //  "<"  ">"  , 
    //     -   
    const lessThan100 = check(x,'<', 100);

    if (lessThan100) {
        //     -  
        //       
        //    ,  lessThan100  - LessThan<100>
        //    
        consumeNumberLessThan100(lessThan100); 

        //   -  ,   , 
        //  ,   > 100.
        consumeNumberMoreThan100(lessThan100);
    }

    const moreThan100 = check(x, '>', 100);

    if (moreThan100) {
        consumeNumberMoreThan100(moreThan100); // 
        consumeNumberLessThan100(moreThan100); // 
    }

    consumeNumberLessThan100(x); // 
    consumeNumberMoreThan100(x); // 
}

くそー、これは非常にクールだと思ったので、早い段階で多数のエラーをキャッチできます。私の価値制限ツールのデザインはひどいですが、今のところそれは単なる概念です。将来的には、簡単に拡張して強力なDSLを構築し、コンパイル段階で保証されるパラメーターマッチングのための非常に複雑な条件を作成できます。私はそれがどのようにしてコードベースの信頼性を向上させることができるかを考え出し、思い切って、さまざまな使い慣れた開発者にスニペットを送りました。

意見は再び分かれ、私の方向ではありませんでした。再複雑化、過剰設計はサポートされておらず、誰もがeniへのキャストの助けを借りてガードを妨害します。読めない。現在のプロジェクトでは実験が悪いので良い。使用例は指で描いています。フィル、ビジネスに取り掛かります。

このアプローチの支持者は、そう、信頼できると述べた。間違いを見つければ早くするほど良いでしょう。さらに、限られた数で機能する関数を作成する場合でも、チェックを作成する必要がありますが、実行時にのみ機能し、関数の本体のコード量が増加します。

今、私は以前より少し賢くなり、議論を聞くことを学びました。自分のタイプで保護された関数を実際のプロジェクトで書くことを想像しました。それを使う誰もがこれが一体何であるかを私に尋ねているように。どのように、チップをいじった後、彼らはコードベースをカスタムの腐った見た目のDSLでコーティングし始めます。 300回チェックするので、実際には許容値を超えることは決してありません。アプローチは本当にひどいです、いくつかの問題は解決または平滑化できますが、たとえば、そのような場合

consumeNumberLessThan100(90);

決して滑らかにしないでください。90が100未満であることをコンパイラーに証明する必要があります。アクティビティーを提供すると、それが判明します。

consumeNumberLessThan100(assert(90, '<', 100));

とてもかっこよく見えません。反対意見はすべて手元にありますが、反対意見はありません。開発の容易さ、または信頼性というジレンマが判明します。ここで私たちは罠に陥り、そこにはどんな種類の利便性があり、どんな種類の信頼性があるのか​​を計算する必要があると考え始めます。しかし、開発における利便性と信頼性は非常に複雑なものです。それらは何千ものパラメータで構成されています。

たとえば、IDEが入力された1組の文字でコードをコンパイルする場合、コードが読みやすい場合、ドキュメントを確認せずにメソッドの機能を変更できる場合などに便利です。コンパイラーが静的分析で読み込まれていない場合、ビルドは高速で、テキストエディターは即座に文字をレンダリングします。しかし、コンパイラーがエラーを検出して強調表示すると、これも便利です。そしてまたそれは信頼性です。これは、膨大な数の完全に異なるものから組み立てられます。

座って、プロジェクトの継続時間、プロジェクトの進行方向、人類の歴史の中で誰かがコードベースのメソッドを呼び出す回数を計算する必要があります。開発者はここで作業します。これは、正解を計算することが不可能である良い半分の質問の無限のリストです。ただの推測。


かつて、ある友人から、Andrei Breslavへのインタビューのための技術的な質問を準備するように依頼されました。 Kotlinのドックに行き、問題のあるデザインポイントを見つけました。彼らは、他の場所と同じように、そこに闇があります。しかし、私が最も興味を持ったのは、例外を処理するためのアプローチでした。 Kotlinの例外ハンドラーは式であり、完全に機能的で信頼性の高いアプローチです。エラーを処理するためだけに必要なものではありません。すべての信頼性がゼロになります。そして、これは興味深いことです。ドックでは、開発者は自分の選択を説明するのに怠惰ではありませんでした。

クレイジーです。コードが正しく機能しない場合の対処法がわからない場合、どうすればコードを記述できますか？例外の処理方法がわからない場合は、処理しないでください。問題の解決策ではなく、保留します。ある時点で、コードは脱落し、常に存在する問題が予期されていた可能性のある損傷を引き起こします。

しかし、反対の論理的な議論は必要ありません。統計だけを行うことができます。 Kotlinの開発者にとって、研究は哲学を持っているため、論理を壊します。彼らの哲学は実用主義です。このプログラミング言語のすべての機能に一貫して組み込まれている、鉄、破壊できない実用主義。 Haskels / Idrisを見た理想主義者と、golangを書いたgovnokodersはまったく同じです。合理的な妥協の理念は、F＃コードベースで支配されます。そして、それらのうちの1つが正しいと感じることはなく、残りはばかです。

これらの人々はすべて、私のような人々よりもはるかに賢く、経験豊富であり、彼らはあなたがあなた自身のために哲学を開発していることを長い間理解しており、その後あなたは単にそれに従います。哲学のキラー機能は、ジレンマを解決することです。

そして哲学はITのすべてに現れました、そして哲学はあなた自身のために主観的な真実を選ぶことを示唆しているので、哲学は単一の真の客観性の考えの正反対です。

私たち一人ひとりが独自の哲学を持っています-それは一言で説明されていない、それは意思決定のための複雑なパターンのセットです。そして、私たちはしばしばそれらを変更または拡張します。実際には、独自の哲学を持つプロジェクトがあり、それは独自の哲学を持つプログラミング言語で記述され、それぞれ独自の哲学を持つフレームワークとツールを使用していることがわかります。そして、開発者はこのプロジェクトを書き、それぞれが独自の哲学を持っています。そして、ある種の決定を下す必要があるときはいつでも、状況の組み合わせだけがどちらに決定するかに影響します。複雑さの管理、経験、アプローチ、知識ではなく、単なる事故です。

そして、これらすべてのプロジェクトが機能します。開発者が解決するすべてのタスクで最も重要なのは、他の開発者のエラーの影響を排除することですが、プロジェクトは機能し、人々の問題を解決します。賢い人々は実践と建築パターンを思いつきます。強力な型制御とコントラクトを備えたプログラミング言語-開発者の間違いが少ないとすれば、すべて。毎日、職場でEdge Boardを開いていますが、タスクよりもバグの方が多いようです。すべてのバグはどこにありますか？

開発の哲学は、ファイナルを手に入れる方法の選択に過ぎません。しかし、哲学がなく、純粋な客観的な論理だけがある場合、あらゆる議論は全能の創造者と耐え難い石の問題に帰着します。


私は長い間開発に携わっており、根本的に同意しない場合でも、彼らが言うことを実行する方法を学び、非常に詳細なコミットを書き始めました。私のdevlidは本物の吸血鬼です。それは単に説明を必要とするのではなく、彼は私に各ファイルが何をするのか、なぜそれをするのか、どのようにそれをするのかを書いてほしいと思っています。コミットはどのような問題をコミットしますか？

私は8つの追加の詳細なコミットを追加しましたが、リクエストプールの詳細な説明と同じです。それ以来、私たちはこの特定のプロジェクトにほとんど取り組みませんでしたが、時々私はこれらのコミットを賞賛するためにそこに行きます。本当に、本当に、本当にクール。そして、私の個人的なプロジェクトを除いて、他のすべてのプロジェクトでは、このアプローチを適用しています。
