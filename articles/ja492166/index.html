Android上のウィジェット。理解する珍しい機能

こんにちは、ハブル！私の名前はアレクサンダーカーキモフです。FINCHのAndroid開発者です。

あなたのデザインがiOS用であり、それをAndroidに適合させなければならないということは起こりましたか？もしそうなら、あなたのデザイナーはウィジェットをよく使用しますか？残念ながら、ウィジェットはほとんど誰も使用しないため、多くの開発者にとってまれなケースです。

この記事では、ウィジェットの作成方法について詳しく説明します。

ウィジェット作成


知っておく必要があるウィジェットを作成するには：

  1. ウィジェットコンポーネントの機能。
  2. 画面グリッドにウィジェットを表示する機能。
  3. 機能のウィジェットの更新。

各項目を個別に分析します。

ウィジェットコンポーネントの機能


少なくとも一度はRemoteViewsを使用したことがある開発者なら誰でも、この項目に精通しています。あなたがこれらの一人なら、次のポイントに進んでください。

RemoteViewsは、別のアプリケーションのプロセスに属するビュー階層を記述および管理するように設計されています。階層管理を使用すると、別のアプリケーションの一部であるビューに属するプロパティを変更したり、メソッドを呼び出すことができます。RemoteViewsには、標準のandroid.widgetコンポーネントライブラリの限定されたコンポーネントセットが含まれています。

ウィジェット内の表示は別のプロセス（通常、これはホーム画面です）で機能するため、ウィジェットのUIを変更するには、アプリケーションで機能するBroadcastReceiver拡張機能-AppWidgetProviderを使用します。

画面の「グリッド」にウィジェットを表示する機能


実際、公式ガイドラインを見ると、この点はそれほど複雑ではありません。
各ウィジェットはminWidthおよびminHeightを定義する必要があります。これは、デフォルトで消費する必要があるスペースの最小量を示します。ユーザーがウィジェットをホーム画面に追加すると、通常、指定した最小の幅と高さよりも多くのウィジェットが占有されます。Androidホーム画面は、ウィジェットとアイコンを配置できる利用可能なスペースのグリッドをユーザーに提供します。このグリッドはデバイスによって異なります。たとえば、多くの携帯電話は4x4グリッドを提供し、タブレットはより大きな8x7グリッドを提供できます。

ロシア語への翻訳：各ウィジェットは、最小の幅と高さを設定して、デフォルトで占める最小のスペースを示す必要があります。


Android Studioで作成するときのウィジェット設定の例

ホーム画面に追加されたウィジェットは、通常、設定した画面の最小の幅と高さよりも多くのスペースを占有します。 Androidホーム画面は、ウィジェットとアイコンを配置できる利用可能なスペースのグリッドをユーザーに提供します。このグリッドはデバイスによって異なる場合があります。たとえば、多くの携帯電話は4x4グリッドを提供し、タブレットは大規模な8x4グリッドを提供できます。

このことから、デバイスのグリッドは何でも可能であり、セルのサイズはグリッドのサイズによって異なる場合があることが明らかになります。したがって、ウィジェットのコンテンツは、これらの機能を考慮して設計する必要があります。

特定の数の列と行に対するウィジェットの最小の幅と高さは、式

minSideSizeDp = 70×n-30 を使用して計算できます

。ここで、nは行または列の数です。現在、設定できる最大の最小グリッドは4x4です。これにより、ウィジェットがすべてのデバイスに表示されます。

ウィジェットの更新機能


AppWidgetProviderは本質的にBroadcastReceiverの拡張であるため、通常のBroadcastReceiverの場合と同じことができます。 AppWidgetProviderは、onReceiveで受け取ったIntentの対応するフィールドを解析し、受け取ったエクストラを使用してインターセプトメソッドを呼び出します。

コンテンツの更新頻度に問題が発生しました-重要なのは、iOSとAndroidのウィジェットの内部操作の違いです。実際、iOSウィジェットのデータは、ウィジェットがユーザーに表示されるようになると更新されます。 Androidでは、そのようなイベントは存在しません。ユーザーがウィジェットをいつ表示するかはわかりません。

Android上のウィジェットの場合、推奨される更新方法はタイマー更新です。タイマー設定は、ウィジェットパラメーターupdatePeriodMillisによって設定されます。残念ながら、この設定では、ウィジェットを30分ごとに複数回更新することはできません。以下で、これについて詳しく説明します。

さらに、州の宝くじに参加するためのStolotoアプリケーションと一緒に、FINCHで大規模な宝くじアプリケーションで発生したケースについて説明します。

私たちの場合、ウィジェットの設計には2つの状態が含まれています。

これは実際にはアダプタの薄いラッパーです。彼のおかげで、データをリモートコレクションビューに関連付けることができます。RemoteViewsFactoryは、データセット内の各アイテムのRemoteViewを生成するためのメソッドを提供します。コンストラクターには要件はありません。私が行うのは、その中にコンテキストを渡すことだけです。

次に、主なメソッドについて少し説明します。

  1. onCreate-アダプターの作成。
  2. getLoadingView-このメソッドは、ビューが返されることを提案します。ビューが作成されると、リストアイテムの代わりに表示されます。ここで何も作成しない場合、システムはデフォルトのビューを使用します。
  3. getViewAt-このメソッドは、リストアイテムの作成を提案します。RemoteViewsの標準的な使用法を紹介します。
  4. onDataSetChangedは、リスト内のデータを更新する要求を受け取ったときに呼び出されます。それら。このメソッドでは、リストのデータを準備します。このメソッドは、長くて重いコードを実行することでよりシャープになります。
  5. onDestroyは、アダプターが使用した最後のリストが削除されたときに呼び出されます（1つのアダプターを複数のリストで使用できます）。
  6. RemoteViewsFactoryは、リストのすべてのインスタンスが生きている間は生きているため、現在のアイテムのリストなど、現在のデータをリストに格納できます。

ウィジェットの手動更新


ウィジェットの更新時間は30分に設定されています。たとえば、WorkManagerでダンスすることで、より頻繁に更新できますが、なぜネットワークとバッテリーをロードするのですか？開発の初期段階でのそのような行動は適切であるように見えました。

ユーザーがウィジェットを見ると、無関係なデータがウィジェットに表示されていることに「ビジネス」が気づいたとき、すべてが変化しました。

状況はごく一般的です。iOSはすべてのウィジェット表示用に新しいカードを生成します。これは、そのために特別な画面があり、Androidには原則としてウィジェット用のそのようなイベントがないためです。一部の宝くじは15分ごとに開催されるため、ウィジェットは最新の情報を提供する必要があることを考慮に入れなければなりませんでした。ある種の抽選に参加したいのですが、すでに合格しています。

この不愉快な状況から抜け出し、データの更新に関する問題を何らかの形で解決するために、私は時間をかけてテストされたソリューションである「更新」ボタンを提案して実装しました。

最初の開発は悲しい絵を示しました：「更新」ボタンを押すことから実際の更新まで、数秒が経過する可能性があります。ウィジェットはアプリケーションによって生成されますが、実際にはシステムの制御下にあり、ブロードキャストを介してアプリケーションと通信します。

それら。ウィジェットの「更新」ボタンをクリックすると、チェーンが開始します。

  1. onReceiveプロバイダーの「アクション」でインテントを取得します。
  2. AppWidgetManager.ACTION_APPWIDGET_UPDATE。
  3. intent-eで指定されたすべてのwidgetsIdsに対してonUpdateを呼び出します。
  4. 新しいデータをオンラインで入手してください。
  5. ローカルデータを更新し、新しいリストカードを表示します。

その結果、ボタンをクリックすることで同じウィジェットを数秒間見たため、ウィジェットの更新はあまり見栄えがよくありませんでした。データが更新されたかどうかは不明でした。視覚応答の問題を解決する方法は？

最初に、インタラクターを介したグローバルアクセスでisWidgetLoadingフラグを追加しました。このパラメーターの役割は非常に単純です。ウィジェットデータのロード中に更新ボタンを表示しないでください。

次に、工場でのデータ読み込みプロセスを3つの段階に分けました。

enum class LoadingStep {
   START,
   MIDDLE,
   END
}

START-ダウンロードの開始。この段階で、すべてのアダプタービューの状態とグローバルダウンロードフラグが「読み込み中」に変わります。

MIDDLE-メインデータの読み込みの段階。ダウンロード後、グローバルダウンロードフラグが「ロード済み」状態に設定され、ダウンロードされたデータがアダプターに表示されます。

END-ダウンロードの終了。アダプターは、このステップでアダプターデータを変更する必要はありません。このステップは、WidgetProviderのビューの更新ステージを正しく処理するために必要です。

次に、アダプターで何が起こるかを見てみましょう。

private fun updateDataSync() {
   when (loadingStep) {
       START -> {
           widgetItems.forEach { it.isLoading = true }
           widgetLoadingStateInteractor.isWidgetLoading = true
           loadingStep = MIDDLE
           widgetManager.updateWidgets()
       }
       MIDDLE -> {
           widgetItems.clear()
           updateProfileSync()
           updateGamesSync()
           widgetLoadingStateInteractor.isWidgetLoading = false
           loadingStep = END
           widgetManager.updateWidgets()
       }
       END -> {
           loadingStep = START
       }
   }
}

  1. STARTおよびMIDDLEステップの最後に、updateWidgetsメソッドを呼び出して、プロバイダーが管理するビューステートを更新します。
  2. START «» , MIDDLE.
  3. MIDDLE, «».
  4. MIDDLE, END.
  5. , END, «». , END loadingStep START.


このような実装の助けを借りて、ウィジェット上の実際のデータを表示するための「ビジネス」の要件と、更新を頻繁に「プル」する必要性の間の妥協点に到達しました。
