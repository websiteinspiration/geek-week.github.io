機能的なDBMS

データベースの世界は、SQL言語を使用するリレーショナルDBMSによって長い間捕獲されてきました。そのため、新興種はNoSQLと呼ばれています。彼らはこの市場で特定の場所を取り戻すことに成功しましたが、リレーショナルDBMSは死ぬことはなく、その目的のために引き続き積極的に使用されています。

この記事では、機能的なデータベースの概念について説明します。よりよく理解するために、私は古典的な関係モデルと比較することによってこれを行います。例として、インターネットで見つかったさまざまなSQLテストのタスクを使用します。

前書き


リレーショナルデータベースは、テーブルとフィールドで動作します。機能データベースでは、クラスと関数が代わりに使用されます。N個のキーを持つテーブルのフィールドは、N個のパラメーターの関数として表示されます。テーブル間の関係の代わりに、リンクされているクラスのオブジェクトを返す関数が使用されます。JOIN関数の代わりに構成が使用されます。

タスクに直接進む前に、ドメインロジックのタスクについて説明します。DDLには、PostgreSQL構文を使用します。関数型の場合は、独自の構文。

テーブルとフィールド


フィールド名と価格を持つ単純なSkuオブジェクト：

Relational

CREATE TABLE Sku
(
    id bigint NOT NULL,
    name character varying(100),
    price numeric(10,5),
    CONSTRAINT id_pkey PRIMARY KEY (id)
)

機能的
CLASS Sku;
name = DATA STRING[100] (Sku);
price = DATA NUMERIC[10,5] (Sku);

1つのSkuパラメーターを入力として取り、プリミティブ型を返す 2つの関数を宣言します。

機能しているDBMSでは、各オブジェクトには、自動的に生成され、必要に応じてアクセスできる特定の内部コードがあると想定されています。

商品/店舗/サプライヤーの価格を設定します。時間の経過とともに変化する可能性があるため、時間フィールドをテーブルに追加します。私はコードを短くするために、リレーショナル・データベース内のディレクトリのためのテーブルの宣言をスキップします：

リレーショナル

CREATE TABLE prices
(
    skuId bigint NOT NULL,
    storeId bigint NOT NULL,
    supplierId bigint NOT NULL,
    dateTime timestamp without time zone,
    price numeric(10,5),
    CONSTRAINT prices_pkey PRIMARY KEY (skuId, storeId, supplierId)
)

機能的
CLASS Sku;
CLASS Store;
CLASS Supplier;
dateTime = DATA DATETIME (Sku, Store, Supplier);
price = DATA NUMERIC[10,5] (Sku, Store, Supplier);

タスク


Habréの 対応する記事から取った比較的単純なタスクから始めましょう。

まず、ドメインロジックを宣言します（リレーショナルデータベースの場合、これは上記の記事で直接行われます）。

CLASS Department;
name = DATA STRING[100] (Department);

CLASS Employee;
department = DATA Department (Employee);
chief = DATA Employee (Employee);
name = DATA STRING[100] (Employee);
salary = DATA NUMERIC[14,2] (Employee);

タスク1.1


直属の上司よりも高い給与を受け取っている従業員のリストを表示します。

関連した

select a.*
from   employee a, employee b
where  b.id = a.chief_id
and    a.salary > b.salary

機能的
SELECT name(Employee a) WHERE salary(a) > salary(chief(a));

タスク1.2


自分の部署で最大の給与を受けている従業員のリストを表示します。

リレーショナル

select a.*
from   employee a
where  a.salary = ( select max(salary) from employee b
                    where  b.department_id = a.department_id )

機能的
maxSalary ' ' (Department s) = 
    GROUP MAX salary(Employee e) IF department(e) = s;
SELECT name(Employee a) WHERE salary(a) = maxSalary(department(a));

//   ""
SELECT name(Employee a) WHERE 
    salary(a) = maxSalary(GROUP MAX salary(Employee e) IF department(e) = department(a));

どちらの実装も同等です。最初のケースでは、リレーショナルデータベースでCREATE VIEWを使用できます。これは、同じ方法で、最初にその特定の部門の最大給与を計算します。将来は、明確にするために、最初のケースを使用します。これは、ソリューションをよりよく反映しているためです。

タスク1.3


部門IDのリスト、つまり3人を超えない従業員の数を表示します。

関連した

select department_id
from   employee
group  by department_id
having count(*) <= 3

機能的
countEmployees ' ' (Department d) = 
    GROUP SUM 1 IF department(Employee e) = d;
SELECT Department d WHERE countEmployees(d) <= 3;

タスク1.4


同じ部門で働く任命されたマネージャーがいない従業員のリストを表示します。

関連した

select a.*
from   employee a
left   join employee b on (b.id = a.chief_id and b.department_id = a.department_id)
where  b.id is null

機能的
SELECT name(Employee a) WHERE NOT (department(chief(a)) = department(a));

タスク1.5


従業員の給与の合計が最大の部門IDのリストを見つけます。

関連した

with sum_salary as
  ( select department_id, sum(salary) salary
    from   employee
    group  by department_id )
select department_id
from   sum_salary a       
where  a.salary = ( select max(salary) from sum_salary )

機能的
salarySum ' ' (Department d) = 
    GROUP SUM salary(Employee e) IF department(e) = d;
maxSalarySum '  ' () = 
    GROUP MAX salarySum(Department d);
SELECT Department d WHERE salarySum(d) = maxSalarySum();

タスク2.1


1997年にNo. 1の商品30点以上を販売した販売者はどれですか

ドメインロジック（以前と同様、RDBMSでのアナウンスはスキップします）：
CLASS Employee '';
lastName '' = DATA STRING[100] (Employee);

CLASS Product '';
id = DATA INTEGER (Product);
name = DATA STRING[100] (Product);

CLASS Order '';
date = DATA DATE (Order);
employee = DATA Employee (Order);

CLASS Detail ' ';

order = DATA Order (Detail);
product = DATA Product (Detail);
quantity = DATA NUMERIC[10,5] (Detail);

関連した

select LastName
from Employees as e
where (
  select sum(od.Quantity)
  from [Order Details] as od
  where od.ProductID = 1 and od.OrderID in (
    select o.OrderID
    from Orders as o
    where year(o.OrderDate) = 1997 and e.EmployeeID = o.EmployeeID)
) > 30

機能的
sold (Employee e, INTEGER productId, INTEGER year) = 
    GROUP SUM quantity(OrderDetail d) IF 
        employee(order(d)) = e AND 
        id(product(d)) = productId AND 
        extractYear(date(order(d))) = year;
SELECT lastName(Employee e) WHERE sold(e, 1, 1997) > 30;

タスク2.2


各購入者（名前、姓）について、購入者が1997年に最も多くのお金を費やした2つの製品（名前）を見つけます。

前の例のドメインロジックを拡張します。
CLASS Customer '';
contactName '' = DATA STRING[100] (Customer);

customer = DATA Customer (Order);

unitPrice = DATA NUMERIC[14,2] (Detail);
discount = DATA NUMERIC[6,2] (Detail);

関連した

SELECT ContactName, ProductName FROM (
SELECT c.ContactName, p.ProductName
, ROW_NUMBER() OVER (
    PARTITION BY c.ContactName
    ORDER BY SUM(od.Quantity * od.UnitPrice * (1 - od.Discount)) DESC
) AS RatingByAmt
FROM Customers c
JOIN Orders o ON o.CustomerID = c.CustomerID
JOIN [Order Details] od ON od.OrderID = o.OrderID
JOIN Products p ON p.ProductID = od.ProductID
WHERE YEAR(o.OrderDate) = 1997
GROUP BY c.ContactName, p.ProductName
) t
WHERE RatingByAmt < 3

機能的
sum (Detail d) = quantity(d) * unitPrice(d) * (1 - discount(d));
bought '' (Customer c, Product p, INTEGER y) = 
    GROUP SUM sum(Detail d) IF 
        customer(order(d)) = c AND 
        product(d) = p AND 
        extractYear(date(order(d))) = y;
rating '' (Customer c, Product p, INTEGER y) = 
    PARTITION SUM 1 ORDER DESC bought(c, p, y), p BY c, y;
SELECT contactName(Customer c), name(Product p) WHERE rating(c, p, 1997) < 3;

PARTITION演算子は、次の原則に従って機能します。指定されたグループ（ここではCustomerとYearですが、任意の式を使用できます）内でSUM（ここでは1）の後に指定された式を合計し、ORDERで指定された式（ここで購入します。製品の内部コードと同じです）。

タスク2.3


現在の注文を満たすためにサプライヤに注文する必要がある商品の数。

ドメインロジックの拡張：
CLASS Supplier '';
companyName = DATA STRING[100] (Supplier);

supplier = DATA Supplier (Product);

unitsInStock '  ' = DATA NUMERIC[10,3] (Product);
reorderLevel ' ' = DATA NUMERIC[10,3] (Product);

関連した
select s.CompanyName, p.ProductName, sum(od.Quantity) + p.ReorderLevel — p.UnitsInStock as ToOrder
from Orders o
join [Order Details] od on o.OrderID = od.OrderID
join Products p on od.ProductID = p.ProductID
join Suppliers s on p.SupplierID = s.SupplierID
where o.ShippedDate is null
group by s.CompanyName, p.ProductName, p.UnitsInStock, p.ReorderLevel
having p.UnitsInStock < sum(od.Quantity) + p.ReorderLevel

機能的
orderedNotShipped ',   ' (Product p) = 
    GROUP SUM quantity(OrderDetail d) IF product(d) = p;
toOrder ' ' (Product p) = orderedNotShipped(p) + reorderLevel(p) - unitsInStock(p);
SELECT companyName(supplier(Product p)), name(p), toOrder(p) WHERE toOrder(p) > 0;

アスタリスク付きのタスク


そして最後の例は個人的なものです。ソーシャルネットワークの論理があります。人々はお互いに友達になり、お互いを好きになることができます。機能データベースの観点からは、次のようになります。
CLASS Person;
likes = DATA BOOLEAN (Person, Person);
friends = DATA BOOLEAN (Person, Person);

友情の候補者を見つける必要がある。より形式化すると、AはBと友達で、BはCと友達で、AはCが好きですが、AはCと友達ではないように、すべての人A、B、Cを見つける必要があります。
機能データベースの観点から、クエリは次のようになります。
SELECT Person a, Person b, Person c WHERE 
    likes(a, c) AND NOT friends(a, c) AND 
    friends(a, b) AND friends(b, c);

読者は、SQLでこの問題を個別に解決するよう招待されています。友人は好きな人よりもはるかに少ないと想定されています。したがって、それらは別々のテーブルにあります。ソリューションが成功した場合、2つのアスタリスクに関する問題もあります。その中で、友情は対称的ではありません。機能するデータベースでは、次のようになります。
SELECT Person a, Person b, Person c WHERE 
    likes(a, c) AND NOT friends(a, c) AND 
    (friends(a, b) OR friends(b, a)) AND 
    (friends(b, c) OR friends(c, b));

結論


言語の指定された構文は、上記の概念を実装するためのオプションの1つにすぎないことに注意してください。基礎として採用されたのはSQLであり、それを可能な限りそれに似せることが目標でした。もちろん、誰かがキーワードの名前や単語帳などを好まないかもしれません。ここでの主なものはコンセプトそのものです。必要に応じて、C ++とPythonの両方を同様の構文にすることができます。

これは単なる概念であるという事実にもかかわらず、すべての機能ロジックをリレーショナルロジックに変換するJava実装がすでにいくつかあります。さらに、プレゼンテーションロジックや他の多くの要素が見事にボルトで固定されているため、プラットフォーム全体が得られます。実際、「仮想マシン」としてRDBMS（これまでのところPostgreSQLのみ）を使用しています。RDBMSのクエリオプティマイザーはFSBDが知っている特定の統計を知らないため、このような変換は問題を引き起こすことがあります。理論的には、機能論理に特化した構造をストレージとして使用するデータベース管理システムを実装することが可能です。
