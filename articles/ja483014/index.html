PgQを使用したPostgreSQLメッセージキュー



メッセージキューは、保留中の操作、サービス間の相互作用、「バッチ処理」などを実行するために使用されます。RabbitMQ、ActiveMQ、ZeroMQなど、そのようなキューを整理するための特別なソリューションがありますが、それらはそれほど必要ではなく、それらのインストールとサポートは、メリットよりも多くの苦痛を引き起こします。登録時にユーザーに確認の電子メールが送信されるサービスがあるとします。Postgresを使用している場合は、運が良ければ、Postgresでは、箱から出してすぐに、すべての汚い作業を行うPgQ拡張があります。

この記事では、PgQ拡張機能を使用したPostgreSQLでのメッセージキューイング（タスク）について説明します。この記事は、PgQを使用したことがない場合、またはPostgresで自己記述型のキューを使用している場合に役立ちます。

タブレットを作成してそこでタスクを書くことができるのに、なぜPgQが必要なのですか？それは可能であるように見えますが、タスクへの並列アクセス、起こり得るエラー（タスクを処理するプロセスが落ちるとどうなりますか？）、およびパフォーマンス（PgQは非常に高速であり、特にトランザクションが実行されている場合は自己記述型のソリューションは通常ありません）を考慮する必要がありますタスク全体の間にデータベースが閉じない）が、私の意見ではPgQを使用する必要がある最も重要な理由は、PgQがすでに作成されて機能しており、自己作成ソリューションを作成する必要があることです（UPD：自己作成キューを使用する価値がない理由について） 、たとえば、こちらをご覧ください）。
（UPD：PgQはPostgresの上で実行されるため、トランザクションのすべての楽しみをその中で使用することもできます）

しかし、PgQには1つの大きなマイナスがあります。ドキュメントがないため、この記事でこの欠点を補おうとしています。

端末


PgQはパーツ（少なくとも2つ）で構成されています。1-postgresのpgq拡張機能、2-pgqdデーモン（少し後でインストールする場合）。

キューとのすべての対話は、Postgres内の関数を使用して実行されます。

たとえば、キュ​​ーを作成するには、次を実行する必要があります

select * from pgq.create_queue({ } text);

キューが作成されたら、メッセージをキューに追加できます。

select * from pgq.insert_event({ } text, { } text, {  } text);

次に、録音されたメッセージを受信する方法を学ぶ必要があります。このため、「コンシューマー」（私はコンシューマーを作成します）などのエンティティがあり、メッセージ（イベント）ではなく、「バッチ」（バッチ）を受信します。バッハは連続したメッセージのグループであり、バッハはpgqdを使用して作成されます。定期的に（構成ファイルの「ticker_period」パラメーター）pgqdは、蓄積されたすべてのメッセージを取得し、新しいバッチに書き込みます。pgqdが機能しない場合、新しいbachが作成されないことが重要です。つまり、コンシューマーは何も読み取る必要がありません。また、pgqdが長時間機能せず、オンにされた場合、この間に蓄積されたメッセージから1つの大きなbachが作成されるため、pgqdはオフにしてください。

（消費者を登録する重要！消費者にのみ記録されたイベントを受け取ることになります後 その登録なので、最初にコンシューマーを作成し、次にイベントを書き込む必要があります）：

select * from pgq.register_consumer({ } text, { } text);

（pgq.unregister_consumerと同様）
各コンシューマーは、作成後に発生したすべてのイベントを受信します（別のコンシューマーによって処理された場合でも）。つまり、1ターンに必要なコンシューマーは1つだけです。次に、複数のサーバーで負荷を分割する方法を説明します。

バッハを取得するには、まずそのIDを調べる必要があります。

select * from pgq.next_batch({ } text, { } text);

コンシューマがすべてのバッチを処理した場合、関数はNULLを返すことができます。この場合、pgqdが新しいbachを作成するまで待つ必要があります。

この場合、バッハは処理されませんが、この関数は同じ値を返します。
以下を使用して、バッチ内のすべてのイベントを取得できます。

select * from pgq.get_batch_events({id } bigint);

（バッハは空の場合があります。）

それらの1つを処理中にエラーが発生した場合は、後でこのイベントを処理することができます。

select * from pgq.event_retry({id } bigint, {id } bigint, {   } integer);

バッハの終了を通知し、新しいものを開始する機会を得るために、それが使用されます 

select * from pgq.finish_batch({id } bigint);

もちろん、これらは拡張機能のすべての機能ではありません。pgq.github.io / extension / pgq / files / external-sql.htmlとgithub.com/pgq/pgq/tree/master/functionsを読むことをお勧めします（各ファイルには定義と説明が含まれています）対応する機能）。

負荷分散


複数のハンドラーが同時にイベントを処理するために、pgq_coop拡張機能があります。これはpgqのように機能し、「サブコンシューマー」と呼ばれる新しいエンティティを追加します。これにより、もちろん、すでに処理されたものを除いて、親コンシューマーの登録の瞬間からすべてのイベントが受信されます。

select * from pgq_coop.register_subconsumer({ } text, { } text, { } text);

select * from pgq_coop.next_batch({ } text, { } text, { } text);

select * from pgq_coop.next_batch({ } text, { } text, { } text, {        ,      } interval);

select * from pgq_coop.finish_batch({id } bigint);

ここで すべての機能についてお読みください。

取り付け


pgq拡張機能とpgqdデーモンはPGDGリポジトリに含まれており、ほとんどのディストリビューションに非常に簡単にインストールされます。たとえば、Debianの場合

sudo apt install postgresql-XX-pgq3 pgqd（XXはバージョン番号）。

pgqdは、での使用について学ぶことができる小さなプログラムです。autorun pgqd --helpに追加することを忘れないでください（sudo systemctl enable pgqd.serviceデフォルトの構成はです/etc/pgqd.ini）。

PgQの使用を開始するには、データベースで拡張機能を接続するだけです。

create extension if not exists pgq;


pgq_coopを使用すると、すべてが少し複雑になり、リポジトリにはありませんが、ソースからコンパイルすることは難しくありません（Debianの例）：

sudo apt install postgresql-server-dev-XX
git clone https://github.com/pgq/pgq-coop.git
cd pgq-coop
sudo make install

を使用して拡張機能を接続します 

create extension if not exists pgq_coop;
