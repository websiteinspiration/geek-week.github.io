werfとGitLab CIを使用した同じタイプのマイクロサービスの組み立てとデプロイ



2年前に「GitLab CIを使用したプロジェクトの構築：数百のアプリケーションに1つの.gitlab-ci.ymlを使用する」という記事を公開しました。次に、今日は同様の問題の解決について説明します。新素材は、あなたと同様の多数のアプリケーションのためのCI / CDプロセスを構築する方法についてですDAPPを交換するWERFの到来includeに.gitlab-ci.ymlとWERFで。

はじめに


記事に記載されている詳細な説明では、次の状況が考慮されます。

  • 多くのリポジトリに分割されている大きなクライアントアプリケーションがあります。
  • 各リポジトリは、Kubernetesクラスターで実行する必要のある個別のアプリケーションです。
  • CIシステムとして、GitLab CIを使用しています。
  • デプロイメント（コードがデプロイされるインフラストラクチャ）は、Helmチャートによって記述されます。
  • イメージをビルドし、werfを使用してKubernetesにデプロイします。

シンプルさと利便性のために（そしてファッションへのオマージュとして）、これらのアプリケーションを引き続きマイクロサービスと呼びます。これらすべてのマイクロサービスは、同じ方法でアセンブル、デプロイ、起動され、特定の設定は環境変数を使用して構成されます。

それは、コピー.gitlab-ci.yml、werf.yamlおよび.helm多くの問題をもたらすことが理解されています。結局のところ、CIでの編集、アセンブリプロセス、またはHelmチャートの説明は、他のリポジトリに追加する必要があります...

.gitlab-ci.ymlでテンプレートを接続する


GitLab CE include:file（バージョン11.7以降）でのディレクティブの出現により、共通のCIを作成することが可能になりました。それ自体includeは少し前（11.4）に表示されましたが、テンプレートを接続できるのはパブリック URL からのみであり、機能が多少制限されていました。 GitLabのドキュメントには、すべての機能と使用例が完全に記載されています。

したがって、.gitlab-ci.ymlリポジトリ間でのコピーを拒否し、その関連性をサポートすることができました。以下に例.gitlab-ci.ymlを示しincludeます。

include:
 - project: 'infra/gitlab-ci'
   ref: 1.0.0
   file: base-gitlab-ci.yaml
 - project: 'infra/gitlab-ci'
   ref: 1.0.0
   file: cleanup.yaml

ブランチ名は注意 して使用することを強くお勧めしますref。Include'yはパイプラインの作成時に計算されるため、CIの変更は、最も都合の悪いときに自動的に本番環境パイプラインに反映されます。ただし、タグを使用refすると、CI / CDプロセスの説明のバージョン管理が簡単になります。アップグレード時には、すべてが可能な限り透過的に見え、タグにセマンティックバージョニングを使用すると、パイプラインバージョンの履歴を簡単に追跡できます。

別のリポジトリから.helmを接続する


これらのマイクロサービスはデプロイされ、同じように実行されるため、同じ一連のHelmテンプレートが必要です。.helmリポジトリ間でのディレクトリのコピーを回避するために、Helmテンプレートが格納されているリポジトリのクローンを作成し、目的のタグでチェックしていました。それはこのようなものでした：

   - git clone https://gitlab-ci-token:${CI_JOB_TOKEN}@gitlab.example.com/infra/helm.git .helm
   - cd .helm && git checkout tags/1.0.0
   - type multiwerf && source <(multiwerf use 1.0 beta)
   - type werf && source <(werf ci-env gitlab --tagging-strategy tag-or-branch --verbose)
   - werf deploy --stages-storage :local

gitサブモジュールを使用したバリエーションもありましたが、すべて回避策のように見えました...

そして、最近のwerfリリースでは、外部リポジトリーからのチャートを接続する機能を備えています。次に、パッケージマネージャーの機能を完全にサポートすることで、アプリケーションの展開の依存関係を透過的に説明できるようになりました。

シーケンス


マイクロサービスでの問題の解決に戻りましょう。Helmチャート（ChartMuseumなど）を格納するためのリポジトリを作成しましょう。Kubernetesクラスターに簡単にデプロイできます。

helm repo add stable https://kubernetes-charts.storage.googleapis.com
helm install stable/chartmuseum --name flant-chartmuseum

イングレスを追加：

apiVersion: extensions/v1beta1
kind: Ingress
metadata:
  annotations:
    kubernetes.io/ingress.class: nginx
    nginx.ingress.kubernetes.io/force-ssl-redirect: "false"
    nginx.ingress.kubernetes.io/proxy-body-size: 10m
    nginx.ingress.kubernetes.io/ssl-redirect: "false"
  name: chart-museum
spec:
  rules:
  - host: flant-chartmuseum.example.net
    http:
      paths:
      - backend:
          serviceName: flant-chartmuseum
          servicePort: 8080
        path: /
status:
  loadBalancer: {}

デプロイメントでflant-chartmuseumは、環境変数DISABLE_APIを値に変更する必要がありますfalse。それ以外の場合（デフォルト）、ChartMuseum API要求は機能せず、新しいグラフを作成できません。

次に、共有Helmチャートが保存されるリポジトリについて説明します。そのディレクトリの構造は次のとおりです。

.
├── charts
│   └── yii2-microservice
│       ├── Chart.yaml
│       └── templates
│           ├── app.yaml
└── README.md

Chart.yaml 次のようになります。

name: yii2-microservice
version: 1.0.4

ディレクトリにtemplatesは、アプリケーションをクラスターにデプロイするために必要なすべての必要なKubernetesプリミティブが含まれている必要があります。ご想像のとおり、この場合のマイクロサービスはyii2フレームワークに基づくPHPアプリケーションです。werfを使用して構築された2つのnginxおよびphp-fpmコンテナーを使用した最小のデプロイメントについて説明します。

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Values.global.werf.name }}
spec:
  replicas: 1
  revisionHistoryLimit: 3
  template:
    metadata:
      labels:
        service: {{ .Values.global.werf.name }}
    spec:
      imagePullSecrets:
      - name: registrysecret
      containers:
      - name: backend
{{ tuple "backend" . | include "werf_container_image" | indent 8 }}
        command: [ '/usr/sbin/php-fpm7', "-F" ]
        ports:
        - containerPort: 9000
          protocol: TCP
          name: http
        env:
{{ tuple "backend" . | include "werf_container_env" | indent 8 }}
      - name: frontend
        command: ['/usr/sbin/nginx']
{{ tuple "frontend" . | include "werf_container_image" | indent 8 }}
        ports:
        - containerPort: 80
          name: http
        lifecycle:
          preStop:
            exec:
              command: ["/usr/sbin/nginx", "-s", "quit"]
        env:
{{ tuple "frontend" . | include "werf_container_env" | indent 8 }}
---
apiVersion: v1
kind: Service
metadata:
  name: {{ .Values.global.werf.name }}
spec:
  selector:
    service: {{ .Values.global.werf.name }}
  ports:
  - name: http
    port: 80
    protocol: TCP

変数に.Values.global.werf.nameは、ファイルからプロジェクトの名前が含まれますwerf.yaml。これにより、サービスとデプロイメントの必要な名前を取得できます。

masterブランチにコミットするときに、チャートのChartMuseumをプッシュする最も簡単な自動化を実行してみましょう。これを行うには、次のように説明し.gitlab-ci.ymlます。

Build and push to chartmuseum:
 script:
 - for i in $(ls charts); do helm package "charts/$i"; done;
 - for i in $(find . -type f -name "*.tgz" -printf "%f\n"); do curl --data-binary "@$i" http://flant-chartmuseum.example.net/api/charts; done;
 stage: build
 environment:
   name: infra
 only:
   - master
 tags:
   - my-shell-runner-tag

に変更versionしてチャートをバージョン管理しChart.yamlます。新しいグラフはすべて自動的にChartMuseumに追加されます。

フィニッシュラインへ！プロジェクトリポジトリで.helm/requirements.yaml、グラフの依存関係を記述します。

dependencies:
- name: yii2-microservice
 version: "1.0.4"
 repository: "@flant"

...そしてリポジトリのあるディレクトリで実行します：

werf helm repo init
werf helm repo add flant http://flant-chartmuseum.example.net
werf helm dependency update

入り.helm/requirements.lockます。これで、アプリケーションをクラスターにデプロイするには、werf helm dependency build開始する前にコマンドを実行するだけで十分werf deployです。

アプリケーションのデプロイの説明を更新するには、マイクロサービスを使用してリポジトリを調べ、とのハッシュとタグに変更を加えた小さなパッチを適用する必要がrequirements.yamlありrequirements.lockます。必要に応じて、CIを介してこの操作を自動化することもできます。これを行う方法については、前述の記事ですでに説明しています。

結論


同様のアプリケーションにサービスを提供するために説明した一連のアクションが、同様の問題に遭遇したエンジニアに役立つことを願っています。また、werfを使用するためのその他の実用的なレシピを共有させていただきます。したがって、実装が困難または単に理解できないと思われる困難がある場合は、遠慮なくテレグラムに連絡するか、ここのコメントに将来の資料のリクエストを残してください。
