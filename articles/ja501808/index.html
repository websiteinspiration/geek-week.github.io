ポインターの所有権/借用システム（OBとも呼ばれる）は、逆参照されたポインターが有効なメモリオブジェクトを指すようにします。

プロトタイプOBシステムのスコープ


これは、Dに適合したOBシステムのプロトタイプです。最初は、ポインターのみを対象としており、動的配列、クラス参照、参照、または集合体のポインターフィールドを対象としていません。この種のサポートを追加すると、実装は複雑になりますが、その本質は変わりません。したがって、後で延期されます。RAIIオブジェクトは自身のメモリを安全に管理できるため、OBの対象にはなりません。メソッドに関係なく、ポインターがGCまたは他のアロケーターを使用してメモリを割り当てるかどうかに関係なく、これはOBにとって重要ではありません。それらは同じであり、同じように処理されます。

OBシステムは、@ live属性で注釈が付けられた関数でのみアクティブです。これは、意味処理の後に、有機物質の規則違反のチェックとしてのみ適用されます。新しい構文は追加されません。生成されたコードは変更されません。場合は、ライブ@関数以外の関数呼び出しのライブ@を、これらと呼ばれる機能があることが期待されるの生@それがテストされていませんが、互換インターフェース。非場合は@ライブ機能が呼び出すライブ@関数、渡された引数が続くことが予想されているライブ@規則。

OBシステムがエラーを検出します：

  • 無効な状態のポインターの逆参照
  • 変化したメモリオブジェクトへの複数のアクティブなポインタ。

ヌルポインターまたは場合によってはヌルポインターを逆参照する試みは検出されません。これは機能しません。現在、型にnull不可のポインタとして注釈を付ける方法がないためです。

OBの基本原則


OBの設計は次の原則に従います。

各メモリオブジェクトに対して、そのオブジェクトへの1つだけの変化するポインタ、またはいくつかの変化しない（読み取り専用）ポインタが存在する可能性があります。

設計


単一の変化するポインタは、メモリオブジェクトの「所有者」と呼ばれます。彼は推移的にメモリオブジェクトと、メモリオブジェクトからアクセス可能なメモリ内のすべてのオブジェクト（つまり、オブジェクトのグラフ）を所有しています。これはこのメモリオブジェクトへの唯一のポインタであるため、メモリ上に存在する可能性のある他のポインタ（変異するかどうかに関係なく）の足元の土を破壊することなく、メモリを安全に管理（形状の変更、割り当て、解放、サイズ変更）できます（グラフメモリ）を示します。

読み取り専用のメモリオブジェクトのグラフへのポインタがいくつかある場合、メモリ内のオブジェクトのグラフの突然の変化を心配することなく、それらを安全に読み取ることができます。

テクノロジーの残りの詳細は、ポインターが読み取り専用になり無効になる方法と、OBのコア原則が常に維持される方法に関連しています。

ポインタは述べて
各ポインタは、次のいずれかの状態にある：

不定
ポインターは無効な状態です。このようなポインターの逆参照は誤りです。
所有
者所有者は、メモリ内のオブジェクトのグラフへの唯一のポインタです。通常、ポインター所有者にはスコープ属性はありません。scope属性を持つポインターが、追跡されたポインターから派生していない式で初期化された場合、そのポインターは所有者になります。
オーナーポインターが別のポインター-オーナーに割り当てられている場合、最初のポインターは未定義状態になります。
借りた
借用ポインタは、一時的にメモリオブジェクトのグラフへの唯一のポインタになるポインタです。彼は、ポインター-所有者からの割り当てを通じてこの状態に移行しますが、所有者は、借りたポインターの使用が完了するまで、借り手の状態に移行します。
借用ポインタは、スコープ属性を持ち、変更可能なオブジェクトへのポインタである必要があります。
読み取り専用読み取り
専用インデックスは所有者によって割り当てられます。読み取り専用ポインタが有効である限り、この所有者からは読み取り専用ポインタのみを割り当てることができます。読み取り専用ポインタには、スコープ属性が必要です。また、可変オブジェクトへのポインタであってはなりません。

それらは、GCによって制御されることが想定されています。

所有

者以外からの借用および読み取り所有者は、リークがないか監視されますが、他のインジケーターは監視されません。借り手は、ポインターで初期化されていない場合、所有者と見なされます。

scope などのポインタをマークしても意味がないように思われますが、これをエラーとして扱うことで修正できる可能性があります。

例外

分析は、例外がスローされないと想定しています。

@live void leaky()
{
    auto p = malloc();
    pitcher();  // throws exception, p leaks
    free(p);
}

1つの解決策は、スコープ（exit）を使用することです。

@live void waterTight()
{
    auto p = malloc();
    scope(exit) free(p);
    pitcher();
}

RAIIオブジェクトを使用するか、nothrow関数のみを呼び出します。

異なるメモリグループの混在異なるメモリ

プールの組み合わせ：

void* xmalloc(size_t);
void xfree(void*);

void* ymalloc(size_t);

auto p = xmalloc(20);
yfree(p);  // should call xfree() instead

見つかりませんでした。

これは、特定のタイプに固有のプールを使用することで回避できます。

U* umalloc();
void ufree(U*);

V* vmalloc();
void vfree(V*);

auto p = umalloc();
vfree(p);  // type mismatch

また、@ live関数でvoid *への暗黙的な変換を許可しないこともあります。可変個引数関数の 引数（たとえば、printf）は消費されたと見なされます。これは安全ですが、あまり実用的ではないようで、確認が必要になる可能性があります。
