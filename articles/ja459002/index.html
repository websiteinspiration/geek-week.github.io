Mozillaが開発したSSL構成ツールについて話しています。

カットの下-サイトを設定するためのその機能とその他のユーティリティについて。

なぜ発電機が必要なのですか


ツールの機能について説明する前に、その目的について説明します。 HTTPSを使用する場合、暗号化は次の4つの場合に使用されます。キー交換時、SSL証明書時、メッセージの転送時、およびハッシュ合計（ダイジェスト）のコンパイル時。

それらのそれぞれは、クライアントとサーバーが同意する異なるアルゴリズムのセットを使用します。 「ハンドシェイク」には非対称暗号、メッセージのエンコードには対称暗号、ダイジェストにはハッシュアルゴリズムを選択します。

たとえば、ECDHE-ECDSA-CHACHA20-POLY1305暗号スイートは、楕円曲線のDiffie-Hellmanプロトコル（ECDHE）に従って鍵交換が行われることを意味します。これは一時的なキーを使用します（使い捨て）1つの接続のみを確立します。証明機関は楕円曲線デジタル署名アルゴリズム（ECDSA）を使用して証明書に署名し、インラインChaCha20アルゴリズムはメッセージの暗号化に使用されます。16バイトのオーセンティケーターを計算する POLY1305は、その整合性を担います。

アルゴリズムの利用可能なすべての組み合わせの完全なリストは、Mozilla wikiページにあります。

サーバーが使用する暗号化方式を構成するための特別なツールがネットワーク上にあります。この機能には、Mozillaで開発されたSSL Configuration Generatorがあります。

たとえば、最初のケースでは、ジェネレーターはAES128 / 256暗号化アルゴリズム、SHA256 / 384ハッシュアルゴリズム、および対称ブロック暗号化モードGCMを使用します。次に、暗号スイートの例を示します。TLS_AES_256_GCM_SHA384。

2番目のケースでは、セキュリティを強化するためにTLS 1.3から除外されている暗号の数がはるかに多い。さらに、TLS 1.3暗号スイートは、証明書のタイプと鍵交換メカニズムを記述していません。したがって、中間構成には、一時鍵とRSAを使用するDiffie-Hellmanプロトコルがあります。

これらの要件に基づいて、SSL構成ジェネレーターは構成ファイル（OpenSSL）を作成します。ビルド時に、必要なサーバーソフトウェア（Apache、HAProxy、MySQL、nginx、PostgreSQL、その他5つ）を選択できます。 Apacheの最新の構成の例を次に示します。

# generated 2019-07-04, https://ssl-config.mozilla.org/#server=apache&server-version=2.4.39&config=modern
# requires mod_ssl, mod_socache_shmcb, mod_rewrite, and mod_headers
<VirtualHost *:80>
    RewriteEngine On
    RewriteRule ^(.*)$ https://%{HTTP_HOST}$1 [R=301,L]
</VirtualHost>

<VirtualHost *:443>
    SSLEngine on
    SSLCertificateFile      /path/to/signed_cert_and_intermediate_certs
    SSLCertificateKeyFile   /path/to/private_key

    # enable HTTP/2, if available
    Protocols h2 http/1.1

    # HTTP Strict Transport Security (mod_headers is required) (63072000 seconds)
    Header always set Strict-Transport-Security "max-age=63072000"
</VirtualHost>

# modern configuration, tweak to your needs
SSLProtocol             all -SSLv3 -TLSv1 -TLSv1.1 -TLSv1.2
SSLHonorCipherOrder     off
SSLSessionTickets       off

SSLUseStapling On
SSLStaplingCache "shmcb:logs/ssl_stapling(32768)"

生成された構成はプロジェクトで使用でき、証明書と秘密鍵のパスを編集して設定をロードするだけです。ただし、ハッカーニュースの居住者の1人が言うように、正しい結果を得るにはサーバーのバージョンに注意を払うことが重要です。特に、nginx 1.0とnginx 1.4の出力は大きく異なります。また、下位互換性を維持し、クロールサイトのベンチマークで高い評価を得るために、生成された暗号スイートの一部を手動で修正する必要がある場合があるという意見もあります。

他にどのようなユーティリティがサイト保護に役立ちますか？


Mozillaポートフォリオには、SSLの構成後にリソースの信頼性を確認するのに役立つユーティリティがいくつかあります。

最初はMozilla Observatoryです。同社は当初、自社ドメインのセキュリティをチェックするツールを開発しました。今では、ソースコードとともに誰でも利用できます。 Observatoryは、サイトの中で最も人気のある脆弱性をスキャンします。潜在的に危険なCookie、XSSの脆弱性、リダイレクトです。システムをスキャンした後、インターネットリソースのセキュリティを向上させるための推奨事項のセットを提供します。
