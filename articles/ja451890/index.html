BINDの24未満の逆サブネットゾーンの委任。使い方

クライアントの1つに、割り当てられたサブネットのPTRレコードを編集する権利を与えるという課題に直面したら/ 28。外部からBIND設定を編集するための自動化がありません。したがって、私は別の方法をとることに決めました-PTR / 24サブネットゾーンの一部をクライアントに委任することです。

それは思われるでしょう-何が簡単になるでしょうか？サブネットを正しい方法で規定し、サブドメインで行われるように、それを目的のNSに送信するだけです。しかし、違います。それはそれほど単純ではありません（実際には一般的には原始的ですが、直感は役に立ちません）、それが私がこの記事を書いている理由です。

自分でそれを理解したい人はRFCを読むことができます。
既製のソリューションが必要な人は、猫へようこそ。

コピー・ペースト法が好きな人を遅らせないために、最初に実用的な部分を投稿し、その後に理論的な部分を投稿します。

1.練習。ゾーンを委任します/ 28

サブネット7.8.9.0/24があるとします。サブネット7.8.9.240/28をDNSクライアント7.8.7.8（ns1.client.domain）に委任する必要があります。

プロバイダーDNSで、このサブネットの逆ゾーンを記述するファイルを見つける必要があります。それはしてみましょう9.8.7.in-addr.arpa。
240から255までの投稿にはコメントがあります。そして、ファイルの最後に次のように書きます：

255-240  IN  NS      7.8.7.8
$GENERATE 240-255 $ CNAME $.255-240

シリアルゾーンを増やすことを忘れないでください 

rndc reload

これで、プロバイダー部分は終わりました。クライアントのDNSに渡します。

まず、次の内容で/etc/bind/master/255-240.9.8.7.in-addr.arpaファイルを作成します。

$ORIGIN 255-240.9.8.7.in-addr.arpa.
$TTL 1W
@                       1D IN SOA       ns1.client.domain. root.client.domain. (
                        2008152607      ; serial
                        3H              ; refresh
                        15M             ; retry
                        1W              ; expiry
                        1D )            ; minimum
@                       IN NS        ns1.client.domain.
@                       IN NS        ns2.client.domain.
241                     IN PTR          test.client.domain.
242                     IN PTR          test2.client.domain.
245                     IN PTR          test5.client.domain.

そして、中にnamed.confの私たちは、私たちの新しいファイルの記述を追加します。

zone "255-240.9.8.7.in-addr.arpa." IN {
        type master;
        file "master/255-240.9.8.7.in-addr.arpa";
};

Bバインドプロセスを再開します。

/etc/init.d/named restart

すべて。これで確認できます。

#>  host 7.8.9.245 
245.9.8.7.in-addr.arpa is an alias for 245.255-240.9.8.7.in-addr.arpa.
245.255-240.9.8.7.in-addr.arpa domain name pointer test5.client.domain.

PTRレコードだけでなくCNAMEも提供されることに注意してください。そうなるはずです。理由がわからない場合は、次の章へようこそ。

2.理論。使い方。

ブラックボックスの構成とデバッグは困難です。内部で何が起こっているかを理解すると、はるかに簡単になります。

ドメインdomainにサブドメインを委任するときは、次のように記述します。

client.domain.	NS	ns1.client.domain.
ns1.client.domain.	A	7.8.7.8

私たちはこのサイトに対して私たちが責任を負わないものを尋ねているすべての人に話し、誰が責任があるかを言います。そして、client.domainへのすべてのリクエストは 7.8.7.8にリダイレクトされます。確認すると、次の画像が表示されます（クライアントが持っているものは省略します。問題ではありません）。

# host test.client.domain
test.client.domain has address 7.8.9.241

それら。そのようなAレコードとそのIP 7.8.9.241があることが通知されました。不要な情報はありません。

しかし、サブネットで同じことをどのように行うことができますか？

なぜなら DNSサーバーはRIPEに登録されているため、ネットワークからPTR IPアドレスを要求すると、最初の要求は引き続き送信されます。ロジックはドメインの場合と同じです。しかし、ゾーンファイルにサブネットを入力する方法は？

次のように入力しようとします。

255-240  IN  NS      7.8.7.8

そして…奇跡は起こらなかった。リクエストのリダイレクトは受けません。問題は、逆ゾーンファイル内のこれらのエントリがIPアドレスであることをbindがまったく認識していないため、さらに範囲エントリを理解していないということです。彼にとって、これは単なるシンボリックサブドメインです。それら。 bindの場合、「255-240」と「oursuperclient」の間に違いはありません。そして、リクエストが本来あるべき場所に行くためのリクエストでは、リクエスト内のアドレスは241.255-240.9.8.7.in-addr.arpaのようになります。または、文字サブドメイン241.oursuperclient.9.8.7.in-addr.arpaを使用する場合、次のようになります。これは通常とは異なります：241.9.8.7.in-addr.arpa。

このようなリクエストを処理することは問題になります。そして、それがうまくいったとしても、それを実際の生活にどのように適用するかはまだはっきりしていません。結局のところ、7.8.9.241のリクエストでは、クライアントではなくプロバイダーDNSが応答します。

そして、ここでCNAMEが登場します。

プロバイダー側​​では、要求をクライアントDNSに転送する形式で、サブネットのすべてのIPアドレスのエイリアスを作成する必要があります。

255-240  IN  NS      ns1.client.domain.
241     IN  CNAME   241.255-240
242     IN  CNAME   242.255-240
 ..

これは勤勉です=）。

そして怠惰な人にとっては、以下のデザインがより適切です：

255-240  IN  NS      ns1.client.domain.
$GENERATE 240-255 $ CNAME $.255-240

今では情報の要求7.8.9.241から241.9.8.7.in-addr.arpaプロバイダのDNSサーバーにはに変換されます241.255-240.9.8.7.in-addr.arpaとDNSクライアントに移動します。

クライアントはそのようなリクエストを処理する必要があります。したがって、255-240.9.8.7.in-addr.arpaのゾーンを作成します。その中で、基本的には/ 24サブネット全体の任意のIPのリバースレコードをポストできますが、プロバイダーがリダイレクトするレコードについてのみ尋ねられるので、=）で遊ぶことはできません。
説明のために、クライアント側の逆ゾーンファイルの内容の例を示します。

$ORIGIN 255-240.9.8.7.in-addr.arpa.
$TTL 1W
@                       1D IN SOA       ns1.client.domain. root.client.domain. (
                        2008152607      ; serial
                        3H              ; refresh
                        15M             ; retry
                        1W              ; expiry
                        1D )            ; minimum
@                       IN NS        ns1.client.domain.
@                       IN NS        ns2.client.domain.
241                     IN PTR          test.client.domain.
242                     IN PTR          test2.client.domain.
245                     IN PTR          test5.client.domain.

ここでは、プロバイダー側​​でCNAMEを使用しているため、ip-addressによるデータの要求に対する応答として、1つではなく2つのレコードを受け取ります。

#>  host 7.8.9.245 
245.9.8.7.in-addr.arpa is an alias for 245.255-240.9.8.7.in-addr.arpa.
245.255-240.9.8.7.in-addr.arpa domain name pointer test5.client.domain.

また、ACLを正しく構成することを忘れないでください。自分でPTRゾーンを取得し、外部からだれにも応答しないことは意味がないためです=）。
