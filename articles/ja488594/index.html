作業を説明するために、リクエストを作成し、テスラ車の総数とタクシーでの作業の割合を決定する単純な関数を記述します。フラグメント読み取りのトリックがないと、このようなリクエストは最初にすべてのメモリを使い果たし、その後長時間かかり、最後にランプが落ちます。

フラグメントを読み取ると、関数は次のようになります。

def pandas_chunky_query():
    print('reading csv file with pandas in chunks')
    filtered_chunk_list=[]
    for chunk in pd.read_csv('C:\Open_data\RDW_full.CSV', chunksize=1E+6):
        filtered_chunk=chunk[chunk['Merk'].isin(['TESLA MOTORS','TESLA'])]
        filtered_chunk_list.append(filtered_chunk)
    model_df = pd.concat(filtered_chunk_list)
    print(model_df['Taxi indicator'].value_counts())

完全に妥当な数百万行を指定することで、1：46でクエリを実行し、ピーク時に1965 Mのメモリを使用できます。古代の8コア、約8 GBのメモリを搭載し、7番目のWindowsの下にある、おかしなデスクトップのすべての数値。

Daskで作業するときに最初に喜ばれることは、厚いデータを持っているからといって、余分なものを書く必要がないということです。ランプではなく、daskがインポートされるという事実に注意を払わない場合、すべてがランプに100行のファイルを処理するのと同じように見えます（さらに、プロファイリング用の装飾的なホイッスルがいくつかあります）。

def dask_query():
    print('reading CSV file with dask')
    with ProgressBar(), ResourceProfiler(dt=0.25) as rprof:

最後に、遅延コンピューティングについて。 daskの機能の1つは、遅延計算を使用することです。つまり、計算はコードで検出されたときにすぐには実行されませんが、本当に必要な場合、またはcomputeメソッドを使用して明示的に要求した場合です。たとえば、この関数では、ファイルの読み取りを指示したときにdaskがすべてのデータをメモリに読み取るわけではありません。彼は後でそれらを読み、リクエストに関連する列のみを読みます。

これは、次の例で簡単に確認できます。最初の64から12列しか残っていないフィルター済みファイルを使用します。圧縮された寄木細工は203 MBを使用します。通常のリクエストを実行すると、8.8秒で実行され、ピークメモリ使用量は約300 MBになります。これは、単純なcsvでそれを追い越した場合の圧縮ファイルの10分の1に相当します。
