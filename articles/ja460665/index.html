ドラフトFAQ：C ++標準が3年ごとに出てくるのはなぜですか？


WG21には、3年ごとの標準リリースの厳しいスケジュール（P1000を参照）があります。そして遅延なし。

各サイクルの間、特にその歴史と現状の理由に精通していない委員会の新メンバーから、「なぜそれほど厳格なのか」という質問を定期的に受け取ります。そして、ケルン政権との予備的な電話会議中に、何人かの人々が、なぜ私たちがこれを行ったのか、そしてこのスケジュールを採用する決定がどのように行われたのかを説明することを推奨しました。

P1000の次のドラフトの質問と回答の形でこれらすべてを描き、ケルンに行く途中で委員会メンバーにコピーを送りました。この資料は、P1000の次の公開バージョンで公開されます。現時点から数週間で送信されます。

しかし、ドラフトFAQは一般に興味があるかもしれないので、そのコピーを提供します。ほとんどの場合、それがあなたにとって有用であり、いくつかの点で啓発され、そしておそらく少しでも楽しまれることを願っています。


標準にはバグがありますが、C ++ 20を延期する必要がありますか？


もちろんそうです。

選択した速度で特定の方向に進んでいます：昨年はバグ修正が計画されているため、C ++“ 19”（Kona）の初期のスケジュールでは、C ++“ 20”の機能の追加（機能のフリーズ）を停止する期限を設定しています。この夏、さまざまな国からのコメントを処理するなど、バグを修正するための1年がありました。2020年（ケルン、ベルファスト、プラハでの会議）が始まる前に、フィードバックを提供し、問題に対するその他の解決策とバグ修正を適用する必要があります。

1〜2回の会議を開催する場合は、ほぼ準備が整った<機能名>を追加できます。C++ 20を延期する必要がありますか？


もちろんそうです。

さらに2度の会議が行われるのを待って（プラハの後）、C ++ 23が営業します。まず、C ++ 23の作業草案に<機能名>を追加することを投票します。これは、私たちがコンセプトで行ったことです。TSからC ++ 17に直接移行する準備ができていませんでした。したがって、最初のC ++ 20会議（トロントで）で、彼らは概念の基本機能をC ++ 20ドラフトに転送することを投票しました。これにより、導入されたTS（非「テンプレート」構文）の残りの部分の改善と改良に多くの時間が与えられました。来年（サンディエゴ）。これですべての機能が準備できました。

これは厳しすぎるようです。なぜISリリースを一定の間隔（3年）でリリースするのですか？


C ++ ISリリースの場合、これはプロジェクト管理の2つの主要なオプションの1つであり、経験から、このオプションは2番目よりも優れていることがわかっています。

私は説明します：

（1）「何を」：機能を選択して出荷するため、リリース時期を選択する必要はありません。ドラフト規格の機能を完成させるのにさらに時間が必要であることが判明した場合、全世界があなたを待たなければなりません。数年の開発を必要とする大規模な機能に取り組み、その後、リリースを安定させる一方で、新機能の開発をすべて中止しようとします。

したがって、C ++ 98（1994年頃に予定されていたため、リリースが間に合わなかった場合は失敗となる）とC ++ 11（2007年までにxが期待されていたため、0xと呼ばれていました） ）これは、無期限に「患者を保護しないままにする」アプローチであり、統合テストとリリースの遅延につながりました。そして、これは、次の標準のタイミング、そしてそれがまったくリリースされるかどうかについて大きな市場の不確実性につながりました（そうです、開発参加者だけでなく、1996年と2009年に深刻に疑われた一部の委員会メンバーでさえ、現れるでしょう）関連するリリースがあります）。数年の間、委員会が新しいリリースで展開する互換性のない変更の数を誰も知らなかったため、ほとんどのコンパイラは標準を満たしませんでした。またはいつ彼を待つか？これにより、コミュニティで利用可能なコンパイラでのC ++サポートの多様性と断片化がもたらされました。

なぜ私たちはこれをしたのですか、私たちは馬鹿ですか？実際にはそうではなく、彼らは単に経験の浅い人でした、そして、「楽観的」だとしましょう。善意で舗装された道路でした。 1994年から1996年と2007年から2009年にかけて、さらに1、2、3回の会議に移行し、すべてを行い、そのたびに最大4年間延期されると本当に信じていました。そして今、彼らは彼ら自身の経験から、1年か2年は転勤ができないと確信しています。

幸い、オプション（2）のおかげですべてが変更されました。

（2）「いつ」：リリース日を選択して、準備ができている機能を出荷します。一連の機能を選択する必要はありません。。ドラフト標準から機能を完成させるのにさらに時間が必要であることが判明した場合は、それを破棄し、準備が整ったものを出荷します。いくつかのリリースの作成には時間がかかりますが、サードパーティの「ブランチ」でそれを実行し、準備ができたらすぐにISマスターブランチに追加する、大きな機能で作業を続けることができます。また、機能の開発は現在のリリースから完全に分離されているため、常に機能に取り組んでいます（大きな接続ポイントはありません）。

私たちは2012年からこのアプローチに固執しており、それを放棄したくありません。これは「定期的に患者にパッチを当てる」アプローチです。これは、通常の機能統合内で、一定レベルの安定性に達するまでISドラフトへの作業の追加を拒否することによる、定期的な統合による高品質の期待につながります。また、市場が信頼できる予測可能なリリースサイクルも作成されます。長年にわたり、コンパイラーの作成者は次のリリース以降、以前にはなかった標準に準拠した製品のバージョンをリリースすることから始めました。そして2020年には、これまでになかった標準のリリースとともに、完全に準拠した実装が1年以内にリリースされることを期待しています。これは、開発者、ユーザー、教師-市場全体の利益のためだけです。

また、このアプローチを採用し始めてから、より大きな（大、中、小の機能で測定した場合）より高い品質（バグレポートの数を厳密に削減して測定した場合、および各規格の草案）。準備ができたものは出荷します（管理しなかった場合は出荷しません）。

アプローチ（2）について、どの程度深刻ですか。委員会の権威あるメンバーによると、いくつかの大きな機能が「ほとんど準備ができている」場合、少し待ってみたいと思うでしょう。


非常に深刻です。

統計があります。2016年にジャクソンビルでC ++ 17の機能を最終的に決定したとき、BjörnStraustrupが本会議でC ++ 17に概念を含めるという提案をしました。合意に達しなかったとき、ストラストルプは、コンセプトを含めるためにC ++ 17のリリースを1年間延期したいかどうかを直接尋ねられました。ビョルンはためらうことなく回避し、「いいえ」と答え、概念を持たないC ++ 17は、概念を含むC ++ 18またはC ++ 19よりも重要であると付け加えました。選択は次のとおりです：（2）コンセプトなしでC ++ 17をリリースし、次にコンセプト付きのC ++ 20（これを行いました）、または（1）C ++ 17をC ++ 20に名前変更します（これは同型です）（2） C ++ 17をスキップし、C ++ 17の準備ができているものをリリースすることを拒否することを除いて。

（1）と（2）の間のトレードオフはどうですか？たとえば、通常は（2）を順守しますが、機能を調整する必要がある場合は、「少し」の柔軟性を得ることで、「少し」の余分な時間を確保できますか？


いいえ、判明しました（1）。The Mythical Man-Monthの

Fred Brooksは、「神話的な小さな転移」についてよく説明し、「小さな転移は許さない」と結論付けました。

C ++ 20を移植したとしましょう。 （2）から（1）に戻る必要があります。これをどの程度難しくしようとも、何のメリットも得られません。それを磨くためにC ++ 20を延期することを決定した場合、標準を少なくとも2年間遅らせます。 1つか3つの会議の転送などの概念はありません。この間、他の人は（公正に）続けて次のように言うためです：「まあ、私の機能はもう1つの会議しか必要ないので、スケジュールを変更しました。別の会議を転送しましょう。」そして、少なくとも2年転送すると、C ++ 20はC ++ 22になり、おそらくC ++ 23 ...になりますが、すでにC ++ 23を出荷する予定です！ -つまり、どのような場合には、我々はC ++ 23を出荷する、との唯一の違いは、私たちがいることですしないでください大量の作業が完了し、リリースの準備が整ったC ++ 20を転送します。全世界をさらに3年間待たせることはありません。遅延はこれらの機能、それらのほとんどまたはすべてを一緒にした場合にはメリットがありません。

したがって、この文は「C ++ 20をC ++ 22またはC ++ 23に変えましょう」と簡単に答え、「はい、C ++ 23がありますが、C ++ 20に加えて、彼の代わりに。」C ++ 20の遅延とは、良好で安定した最終製品をリリースする代わりにC ++ 20をスキップすることを意味します。これによるメリットはありません。

しかし、機能Xは壊れています/ C ++ 20のバグを修正するために残した時間よりも多くの時間が必要です！


問題ない！切るだけ。

この場合、誰かがEWGまたはLEWGに（状況に応じて）状況の説明を含む手紙を書き、ワーキングドラフトISから機能を削除することを提案する必要があります。これらのグループはアピールを検討し、機能が壊れていると判断した場合（およびプレナムが同意した場合）、機能は次のC ++リリースまで延期されます。これはすでにC ++ 0xの概念で行っています。

ただし、（1）の場合、この機能だけでなく、C ++ 20からC ++ 23への機能セット全体を転送します。それは…バストでしょう。

アプローチ（2）はメジャーリリースまたはマイナーリリースを意味しますか？


番号。（2）は、「メイン/マイナー」リリースの観点からも、機能のセットを選択する必要がないことを意味するだけであることを認識するまで、最初にこれを述べました。

アプローチ（2）は、「準備が整ったものを出荷する」ことだけを意味します。リリースが取得されます：

  • 機能の同じサイズ（つまり、通常は平均）は「小さい」。これは、開発に費やされる時間が少ない（たとえば、それぞれ3年未満）ためであり、一般に、リリースで完了した機能の数は同じです。
  • そして、より多くの時間がかかる機能（たとえば、それぞれ3年以上）には可変サイズ（必要ではない場合があります）があり、各ISリリースには、リリースのために完了するまで管理できる数のこれらの機能が含まれています。したがって、一部のリリースではそれ以上のものがあり、他のリリースではそうではありません。

C ++ 14とC ++ 17は比較的小規模でした。これは、TSの実装の提案（たとえば、コントラクト）と「機能ブランチ」（概念など）で説明されている長期的な機能に多くの標準化の労力が費やされたためです。

C ++ 20は素晴らしいリリースです...


はい。C ++ 20には多くの主要な機能があります。最大の3つは「ko」（概念、契約、コルーチン）で始まるため、co_cpp20と呼ぶことができます。またはco_dependent。

...そして、C ++ 20の3年サイクルであまり多くは行われていませんか？


いいえ、上記の「一度は必要ないとき」を参照してください。

C ++ 20が大きいのは、3年間でより多くのことをしたからではなく、長い開発がたくさんあるからです（PセンテンスとTSブランチの形で、現在の形で2012年以来取り組んでいる少なくとも2つを含む）。 ）準備段階になり、同じリリースのISドラフトに含めることにしました。

ほとんどの場合、主な機能は長年にわたって開発されています。 C ++ 98およびC ++ 11のアプローチ（1）とアプローチ（2）の主な違いは、C ++ 98およびC ++ 11では、これらのすべての機能の準備ができるまでリリースが遅れ、現在は大規模に出荷されていることです。準備が整い次第、リリースします。

C ++ 20は、C ++ 14およびC ++ 17と同じ3年間のサイクルを経ました。過去3年間は、前の2サイクルよりも多くのことを行っておらず、主な機能を追加しただけです。それらのいずれかが準備ができていなかった場合は、それを捨てて、C ++ 23ですでに完成させていただろう。これが発生した場合は、実装提案で報告し、理由を説明します。

C ++ 14 + 17 + 20は、C ++ 98（1989-1998）およびC ++ 11（2002-2011）に続く3番目の9年サイクル（2011-2020）を構成しています。ただし、アプローチ（2）を順守したため、3年サイクルと6年サイクルの終わりに備えた開発もリリースしました。

製品がリリースされた後ではなく、開発中のときにバグをキャッチする方が良いのではないですか？


もちろん良いです。

  • 標準がリリースされる前は、機能が公開されておらず、使用されていませんでした（多くの場合、既に生産の経験があります）。
  • また、標準がリリースされるまで（許可されない）、すべての機能を一緒に使用できます。

私が説明します：

  1. C ++ 20の主な機能のほとんどは、少なくとも1つのコンパイラの標準の現在のドラフトに反映されている形で実装されており、ほとんどの場合、すでに量産コードで使用されています（つまり、非常に満足しているユーザーはすでに利用できます）。 。たとえば、コルーチン（この記事のわずか5か月前に導入されたもの）は、MSVCで2年間、Clangで1年間使用され、大規模な顧客（AzureやFacebookなど）に非常に満足していました。

最後に、インタラクション機能の問題を忘れないでください。準備ができたときにリリースするだけでなく、機能間の相互作用の問題を見つけてそのような相互作用のサポートを追加するための時間も必要です。これは、新しい機能が広く使用されるようになる前に見つけることができません。そして、標準のリリースをどれだけ遅らせるかは問題ではありません。私たちが探ることのできる相互作用は常にずっと後になります。柔軟な設計を利用してこのリスクを管理し、機能の互換性を確保し、すべてのリスクを取り除くのを待つ必要はありません。

品質がアプローチ（1）を使用するよりも優れていることを確信していますか？


はい。

客観的な指標によると、さまざまな国からのコメントの量とバグレポート、C ++ 14およびC ++ 17が最も安定したリリースであり、これらの指標によると、C ++ 98およびC ++ 11の3〜4倍でした。そして、その理由は、リリースの規則性、TSブランチでの最初の大きな機能の配置（メイン標準との統合の完全な説明を含む）、および準備が整っていると確信した場合のその後のインフュージョンです。
