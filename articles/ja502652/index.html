公共契約、それらの一貫性を確保する方法

  • システムは相互接続された多くのサービスで構成されていますか？
  • パブリックAPIを変更するときにサービスコードを手動で更新しますか？
  • サービスの変更は他の人の仕事を損なうことがよくあり、他の開発者はこれを嫌いますか？

少なくとも1回は「はい」と答えた場合は、歓迎します。

条項


パブリックコントラクト、仕様 -サービスとやり取りできるパブリックインターフェイス。本文では同じ意味です。

記事の内容はなんですか


契約の統合記述と自動コード生成のためのツールを使用して、Webサービスの開発に費やす時間を削減する方法を学びます。

以下で説明する手法とツールを適切に使用することで、新しい機能をすばやく展開し、古い機能を壊さないようにすることができます。

問題はどのように見えますか


複数のサービスで構成されるシステムがあります。これらのサービスは異なるチームに割り当てられています。



コンシューマーサービスは、サービスプロバイダーによって異なります。
システムが進化し、ある日、サービスプロバイダーが公共契約を変更します。



コンシューマーサービスが変更の準備ができていない場合、システムは完全に機能しなくなります。



この問題を解決する方法


サプライヤーサービスチームがすべてを修正します


これは、プロバイダーチームが他のサービスのサブジェクトエリアを所有し、そのgitリポジトリにアクセスできる場合に実行できます。これは、依存するサービスがほとんどない小規模なプロジェクトでのみ機能します。これは最も安いオプションです。可能であれば、それを使用する必要があります。

サービスのコードをコンシューマーチームに更新する


他の人が壊れているのになぜ修理しているのですか？

しかし、主な質問はサービスを修正する方法です。契約はどのように見えますか？新しいプロバイダーサービスコードを学ぶか、彼らのチームに連絡する必要があります。私たちはコードの研究と他のチームとのやり取りに時間を費やしています。

問題が発生しないようにするために何をすべきかを考える


長期的に見て最も妥当なオプションです。次のセクションで検討してください。

問題の顕在化を防ぐ方法


ソフトウェア開発のライフサイクルは、設計、実装、テストの3つの段階で表すことができます。

各ステップは、次のように拡張する必要があります。

  1. 設計段階では、契約を宣言的に定義します。
  2. 実装時には、契約に基づいてサーバーとクライアントのコードを生成します。
  3. テスト時には、契約を確認し、顧客のニーズ（CDC）を考慮に入れます。

問題の例として、各ステップについてさらに説明します。

問題がどのように見えるか




これは私たちのエコシステムがどのように見えるかです。
円はサービスであり、矢印はそれらの間の通信チャネルです。

フロントエンドは、Webベースのクライアントアプリケーションです。

ほとんどの矢印はストレージサービスにつながります。ドキュメントを保存します。これは最も重要なサービスです。結局のところ、私たちの製品は電子文書管理システムです。

このサービスが契約を変更すると、システムはすぐに動作を停止します。



私たちのシステムのソースは主にc＃で書かれていますが、GoとPythonにもサービスがあります。このコンテキストでは、図の他のサービスが何をするかは問題ではありません。



各サービスには、ストレージサービスを操作するための独自のクライアント実装があります。契約を変更するときは、各プロジェクトのコードを手動で更新する必要があります。

手動更新から自動更新に移行したいと思います。これにより、クライアントコードが変更される頻度を増やし、エラーを減らすことができます。エラーは、URLのタイプミス、不注意によるエラーなどです。

ただし、この方法ではクライアントのビジネスロジックのエラーは修正されません。手動でのみ調整できます。

問題からタスクへ


この場合、クライアントコードの自動生成を実装する必要があります。
その際、次の点を考慮してください。

  • サーバー側-コントローラーは既に作成されています。
  • ブラウザはサービスのクライアントです。
  • サービスはHTTPを介して通信します。
  • 世代を調整する必要があります。たとえば、JWTをサポートします。

スワッシュバックルの結論は適切ではありません。仕様のみを取得できます。クライアントコードを生成するには、追加のソリューションを使用する必要があります。

OpenApiToolsは一連の設定を持つ興味深いツールですが、最初にコードをサポートしていません。その利点は、多くの言語でサーバーコードを生成できることです。

NSwagは、Nugetパッケージであるため便利です。プロジェクトを構築するときに簡単に接続できます。私たちが必要とするすべてをサポートします：コードファーストアプローチとc＃でのクライアントコードの生成。それを選択します。

契約をどこで手配するか。契約のサービスにアクセスする方法


契約保管を整理するためのソリューションを次に示します。ソリューションは、複雑さが増す順にリストされています。

  • プロバイダーサービスプロジェクトフォルダーが最も簡単なオプションです。アプローチで実行する必要がある場合は、それを選択します。
  • 目的のプロジェクトが同じリポジトリにある場合、共有フォルダは有効なオプションです。長期的には、フォルダー内のコントラクトの整合性を維持することは困難です。これには、契約の異なるバージョンなどを説明する追加のツールが必要になる場合があります。
  • 仕様用の別のリポジトリ-プロジェクトが異なるリポジトリにある場合、契約は公共の場所に移動する必要があります。欠点は、共有フォルダーと同じです。
  • サービスAPI（swagger.ui、swaggerhub）を介して-仕様管理を扱う別のサービス。

サービスプロバイダーのプロジェクトフォルダーに契約を保存するために、最も簡単なオプションを使用することにしました。この段階ではこれで十分なので、なぜもっと支払うのですか？

どの時点で生成しますか


次に、コード生成を実行するポイントを決定する必要があります。
契約が共有されている場合、消費者サービスは契約を受け取り、必要に応じてコードを生成できます。

サービスプロバイダープロジェクトのあるフォルダーに契約を配置することにしました。つまり、サプライヤサービスプロジェクト自体のアセンブリ後に生成を行うことができます。

クライアントコードを配置する場所


クライアントコードは契約により生成されます。それを配置する場所を見つけることは残っています。
クライアントコードを別のStorageServiceClientProxyプロジェクトに配置することをお勧めします。各プロジェクトは、このアセンブリを接続できます。

このソリューションの利点：

  • クライアントコードはサービスに近く、常に最新です。
  • コンシューマは、1つのリポジトリ内のプロジェクトへのリンクを使用できます。

短所：

  • 別のリポジトリなど、システムの別の部分でクライアントを生成する必要がある場合は機能しません。少なくとも契約用の共有フォルダを使用して解決されます。
  • 消費者は同じ言語で書かれている必要があります。別の言語のクライアントが必要な場合は、OpenApiToolsを使用する必要があります。

[Microsoft.AspNetCore.Mvc.HttpPost("{pluginName}/{binaryDataId}/{fileName}")]
[OurNamespace.FileUploadOperation] //  
public async Task SaveFile() { // ... }

NSwagでは、ソフトウェアAPIを使用して仕様/コード生成を強制的にカスタマイズすることもできます。

JWTのサポートを追加する方法


不正なリクエストからサービスを保護する必要があります。これには、JWTトークンを使用します。それらは、各HTTP要求のヘッダーで送信する必要があります。これにより、サービスプロバイダーは、それらをチェックして、要求を満たすかどうかを決定できます。

JWTの詳細については、jwt.ioを参照してください。

タスクは結局のところ、発信HTTPリクエストのヘッダーを変更する必要があるということです。
これを行うために、NSwagコードジェネレーターは拡張ポイント（CreateHttpRequestMessageAsyncメソッド）を生成できます。このメソッド内では、送信前にHTTPリクエストにアクセスできます。

コード例 
protected Task<HttpRequestMessage> CreateHttpRequestMessageAsync(CancellationToken cancellationToken) {
      var message = new HttpRequestMessage();

      if (!string.IsNullOrWhiteSpace(this.AuthorizationToken)) {
        message.Headers.Authorization =
          new System.Net.Http.Headers.AuthenticationHeaderValue(BearerScheme, this.AuthorizationToken);
      }

      return Task.FromResult(message);
    }

公共契約のバージョン管理


なぜ公共契約をバージョン管理するのですか？


サービスプロバイダーの変更後、システム全体を一貫した運用状態に保つ必要があります。

クライアントを壊さないように、パブリックAPIの重大な変更は避けなければなりません。

ソリューションオプション


公共契約のバージョン管理なし


サービスプロバイダーチーム自身がコンシューマーサービスを修正します。



このアプローチは、サービスプロバイダーチームがコンシューマーサービスリポジトリにアクセスできないか、コンピテンシーがない場合は機能しません。そのような問題がない場合は、バージョン管理なしで実行できます。



公共契約のバージョン管理を使用する


サービスプロバイダーチームは、以前のバージョンの契約を残します。



このアプローチには、前のアプローチの欠点はありませんが、他の困難が追加されます。
以下について決定する必要があります。

  • 使用するツール。
  • 新しいバージョンをいつ導入するか;
  • 古いバージョンを維持する期間。

新しいバージョンを導入する時期


コントラクトへの変更で下位互換性が維持されない場合は、新しいバージョンを導入する必要があります。

変更が契約の新旧バージョン間の互換性に違反していることを確認するにはどうすればよいですか？

  • OpenAPIには、2つの仕様github.com/Azure/openapi-diff間の互換性をチェックするためのAzure opeanapi-diffツールがあります。
  • gRPCの自動ツールは見つかりませんでした。バージョン管理ポリシーのみがあります。
  • 一般的なgithub.com/grpc/grpc/blob/master/doc/versioning.md;
  • .Net docs.microsoft.com/en-us/aspnet/core/grpc/versioning?view=aspnetcore-3.1にあります。

バージョンを維持する期間


この質問に対する正しい答えはありません。

古いバージョンのサポートを削除するには、誰がサービスを使用しているかを知る必要があります。

バージョンが削除され、他の誰かがそれを使用する場合、それは悪いことです。あなたがあなたの顧客をコントロールしないならば、それは特に難しいです。

では、この状況で何ができるでしょうか？

  • 古いバージョンはサポートされなくなることをお客様に通知します。この場合、お客様の収入を失う可能性があります。
  • バージョンのセット全体をサポートします。ソフトウェアサポートのコストは増加しています。
  • この質問に答えるには、ビジネスに質問する必要があります。古い顧客からの収入は、古いバージョンのソフトウェアをサポートするコストを超えていますか？顧客にアップグレードを依頼する方が収益性が高いでしょうか？

この状況での唯一のアドバイスは、変更の頻度を減らすために公共契約にもっと注意を払うことです。

クローズドシステムで公共契約が使用されている場合は、CDCアプローチを使用できます。したがって、お客様が古いバージョンのソフトウェアの使用をいつ停止したかを知ることができます。その後、古いバージョンのサポートを削除できます。

結論


バージョン管理は、それなしでは実行できない場合にのみ使用してください。バージョン管理を使用する場合は、コントラクトを設計するときにバージョンの互換性を考慮してください。古いバージョンをサポートするコストとそれが提供する利点の間でバランスをとる必要があります。また、古いバージョンのサポートをいつ停止できるかを決定する価値もあります。

契約とCDCのテスト


このセクションは、次のように表面的に照らされています このアプローチの実装には、重大な前提条件はありません。

消費者主導の契約（CDC）


CDCは、サプライヤとコンシューマが同じ契約を使用することを確実にする方法の質問に対する答えです。これらは、契約の確認を目的としたある種の統合テストです。
アイデアは次のとおりです。

  1. 消費者は契約を説明します。
  2. サプライヤーはこの契約を自宅で実施します。
  3. この契約は、消費者とサプライヤーのCIプロセスで使用されます。プロセスに違反した場合、誰かが契約の遵守をやめました。

協定


PACTは、このアイデアを実装するツールです。

  1. 消費者は、PACTライブラリを使用してテストを作成します。
  2. これらのテストは、成果物-協定ファイルに変換されます。契約に関する情報が含まれています。
  3. プロバイダーとコンシューマーは、pactファイルを使用してテストを実行します。

クライアントテスト中にはプロバイダースタブが作成され、サプライヤーテスト中にはクライアントスタブが作成されます。これらのスタブはどちらもPactファイルを使用します。

同様のスタブ作成動作は、Swagger Mock Validator bitbucket.org/atlassian/swagger-mock-validator/src/masterを使用して実現できます。

結論


契約はコンプライアンスを確保するために必要です。契約の変更が消費者サービスの期待に違反する場合に表示されます。

このツールは、サプライヤが顧客（顧客）に適応する場合に適しています。これは、分離されたシステム内でのみ可能です。

外の世界にサービスを提供していて、顧客が誰なのかわからない場合、Pactはあなたには適していません。
