火星気候オービター

着陸時に墜落し、火星の大気圏に落下したのは、ロッキードが開発したソフトウェアコンポーネントがポンド力秒で測定されたインパルス値を提供したため、NASAが開発した他のコンポーネントがインパルス値を取ったためです。ニュートン秒

NASAによって開発されたコンポーネントは、おおよそ次の形式で想像できます。

//    ,  >= 2 N s
void trajectory_correction(double momentum)
{
    if (momentum < 2 /* N s */)
    {
        disintegrate();
    }
    /* ... */
}

Lockheedコンポーネントが上記のコードを次のように呼び出したと想像することもできます。

void main()
{
    trajectory_correction(1.5 /* lbf s */);
}

ポンド力秒（lbfs）は、約4.448222ニュートン/秒（Ns）です。したがって、ロッキードの観点からは、1.5 lbfsの送信trajectory_correctionは完全に正常であるはずです。1.5lbfsは約6.672333 Nsであり、2 Nsのしきい値をはるかに上回ります。

問題はデータの解釈です。その結果、NASAコンポーネントは変換せずにlbfsをNsと比較し、lbfsへの入力をNsへの入力として誤って解釈します。 1.5は2未満なので、オービターは崩壊しました。これは、「プリミティブオブセッション」と呼ばれる有名なアンチパターンです。

プリミティブへの執着

プリミティブの固定は、プリミティブデータ型を使用して問題ドメインの値を表し、上記のような状況を可能にするときに現れます。郵便番号を数字で表し、電話番号を文字列で表し、Nsとlbfsを倍精度数で表すと、これがまさに起こります。

単純な型を定義する方がはるかに安全Nsです。

struct Ns
{
    double value;
};

bool operator<(const Ns& a, const Ns& b)
{
    return a.value < b.value;
}

同様に、単純なタイプを定義できますlbfs。

struct lbfs
{
    double value;
};

bool operator<(const lbfs& a, const lbfs& b)
{
    return a.value < b.value;
}

これで、タイプセーフオプションを実装できますtrajectory_correction。

//  ,   >= 2 N s
void trajectory_correction(Ns momentum)
{
    if (momentum < Ns{ 2 })
    {
        disintegrate();
    }
    /* ... */
}

lbfs上記の例のように、 これをで呼び出すと、型の非互換性のためにコードがコンパイルされません。

void main()
{
    trajectory_correction(lbfs{ 1.5 });
}

通常はコメントで示される値の型に関する情報（2 /*Ns */, /* lbfs */）がどのように型システムに取り込まれ、コードで表現されるかに注意してください：（Ns{ 2 }, lbfs{ 1.5 }）。

もちろん、明示的な演算子の形式でキャストlbfsを提供できますNs：

struct lbfs
{
    double value;

    explicit operator Ns()
    {
        return value * 4.448222;
    }
};

この手法で武装するとtrajectory_correction、静的キャストを使用して呼び出すことができます。

void main()
{
    trajectory_correction(static_cast<Ns>(lbfs{ 1.5 }));
}

ここで、コードの正確さは、係数を掛けることによって達成されます。キャストは暗黙的に（キーワードを使用してimplicit）実行することもできます。その場合、キャストは自動的に適用されます。経験則として、Pythonコアの1つをここで使用できます。
明示的は暗黙的より良い
この話の教訓は、今日の非常にスマートな型チェックメカニズムはありますが、このタイプのエラーを検出するために十分な情報を提供する必要があるということです。サブジェクトエリアの詳細を考慮して型を宣言すると、この情報がプログラムに取り込まれます。

状態空間

Adversityは、プログラムが不良状態で終了したときに発生します。タイプを使用すると、フィールドをその出現範囲に絞り込むことができます。タイプを可能な値のセットとして解釈してみましょう。たとえば、boolは、{true, false}このタイプの変数がこれら2つの値のいずれかを取ることができるセットです。同様に、uint32_tこのセット{0 ...4294967295}。この方法で型を考えると、プログラムの状態空間を、ある時点でのすべての生存変数の型の積として定義できます。

型変数boolと型変数uint32_tがある場合、状態空間は等しくなり{true, false} X {0 ...4294967295}ます。これは、両方の変数が可能なすべての状態になる可能性があることを意味します。2つの変数があるため、プログラムはこれら2つの型の任意の組み合わせ状態になる可能性があります。

値を初期化する関数を考えると、すべてがはるかに興味深いものになります。

bool get_momentum(Ns& momentum)
{
    if (!some_condition()) return false;

    momentum = Ns{ 3 };

    return true;
}

上記の例では、Nを参照で取得し、何らかの条件が満たされた場合に初期化します。関数はtrue、値が正しく初期化された場合に戻ります。何らかの理由で関数が値を設定できない場合は、を返しますfalse。

状態空間の観点からこの状況を考えると、状態空間は製品であると言えますbool X Ns。関数がtrueを返す場合、それはインパルスが設定されたことを意味し、可能な値の1つですNs。問題はこれです：関数が戻る場合false、これはインパルスが設定されていないことを意味します。何らかの方法で、インパルスはNsの可能な値のセットに属していますが、有効な値ではありません。多くの場合、次の無効な状態が誤って拡大し始めるバグがあります。

void example()
{
    Ns momenum;

    get_momentum(momentum);

    trajectory_correction(momentum);
}

代わりに、これを行う必要があります。

void example()
{
    Ns momentum;

    if (get_momentum(momentum))
    {
        trajectory_correction(momentum);
    }
}

ただし、これを強制的に実行できるより良い方法があります。 

std::optional<Ns> get_momentum()
{
    if (!some_condition()) return std::nullopt;

    return std::make_optional(Ns{ 3 });
}

使用した場合optional、この関数の状態空間は大幅に減少しbool X NsますNs + 1。代わりにを取得します。この関数は、有効な値を返すか、値がNsないnulloptことを示します。今ではNs、システムに広まり始めてしまう、容認できないを手に入れることはできません。また、optionalは暗黙的に変換できないため、戻り値を確認することを忘れないようになりましNsた。特別にアンパックする必要があります。

void example()
{
    auto maybeMomentum = get_momentum();

    if (maybeMomentum)
    {
        trajectory_correction(*maybeMomentum);
    }
}

基本的に、関数が結果とエラーではなく結果またはエラーを返すように努めています。したがって、エラーのある条件を除外し、許容できない結果から安全であるため、さらに計算に漏れる可能性があります。

この観点から見ると、例外のスローは上記の原則に対応しているため正常です。関数は結果を返すか、例外をスローします。

RAII

RAIIは、リソース獲得が初期化であることを意味しますが、この原則は、リソースの解放に大きく関係しています。この名前はC ++で最初に登場しましたが、このパターンは任意の言語で実装できます（たとえば、IDisposable.NETから参照）。 RAIIは、リソースの自動クリーニングを提供します。

リソースとは何ですか？次に、動的メモリ、データベース接続、OS記述子などの例を示します。原則として、リソースは外部の世界から取得されたものであり、不要になったときに返却されることがあります。適切な操作を使用してリソースを返します。解放、削除、閉じるなどです。

これらのリソースは外部のものであるため、タイプシステムでは明示的に表現されていません。たとえば、動的メモリのフラグメントを選択すると、次のように呼び出すポインターが取得されますdelete。

struct Foo {};

void example()
{
    Foo* foo = new Foo();

    /*  foo */

    delete foo;
}

しかし、これを忘れた場合、または何かが呼び出しを停止した場合はdeleteどうなりますか？

void example()
{
    Foo* foo = new Foo();

    throw std::exception();

    delete foo;
}

この場合、呼び出しdeleteてリソースリークが発生することはありません。原則として、そのような手作業によるリソースのクリーニングは望ましくありません。動的メモリの場合、それをunique_ptr管理するのに役立つメモリがあります。

void example()
{
    auto foo = std::make_unique<Foo>();

    throw std::exception();
}

私たちのものunique_ptrはスタックオブジェクトであるため、スコープを離れると（関数が例外をスローしたとき、または例外がスローされたときにスタックが展開されたとき）、そのデストラクタが呼び出されます。呼び出しを実装するのはこのデストラクタですdelete。したがって、リソースメモリを管理する必要がなくなりました。この作業は、それを所有し、リリースを担当するラッパーに転送します。

このようなラッパーは、他のリソース用に存在します（または作成できます）（たとえば、WindowsのOS HANDLEを型にラップできますCloseHandle。その場合、デストラクタがを呼び出します）。

この場合の主な結論は、手動でリソースをクリーニングすることではありません。既存のラッパーを使用するか、特定のシナリオに適したラッパーがない場合は、自分で実装します。
