オープンソース-インターネットの基盤


よるに Linux Foundationの、フォーチュン2000企業の72％は、彼らの問題を解決するためのオープンソースツールを使用します。同時に、55％が商用製品でオープンソースを使用しています。オープンソースソフトウェアはデータセンターで配布されています。たとえば、Facebook、Rackspace、NASA、AT＆Tはそれと連携しています。多くのクラウドプロバイダーやIT企業がOpen Compute Projectを設立しました。彼女は、オープン標準のサーバーラックアーキテクチャ（Open Rack）と、クラウドデータセンター（OpenCloudサーバー）のモジュラーサーバー要件を開発しています。

人気のあるオープンソース製品の重要な部分は、Kubernetes、TensorFlow、Ansibleなどの大規模プロジェクトです。それらは、大規模なIT企業によって開発および資金提供されています。しかし、愛好家がサポートする小さな製品（cURLなど）もあります。多くの場合、彼らは自発的かつ自由な時間にこれを行います。そしてここに落とし穴があります。

このモデルが批判されるのはなぜですか


オープンソースの概念は、誰もがソースを変更し、それらのエラーを修正できることを意味します。共同作業により、コードベースの品質が向上し、バグの数が減少します。しかし、残念ながら、このモデルは常に機能するとは限りません。

オープンソースプロジェクトへの変更の重要な部分は、小規模なチームまたは1人のメンテナーによって行われます。たとえば、cURLリポジトリでの 25,000のコミットのうち、14,000は作者であるDaniel Stenbergに属しています。長い間、OpenSSLライブラリの開発者数は4人を超えませんでした。ほとんどのコミットは、そのうちの1人-Steve Henson によって行われました。したがって、そのような状況では、バグを見落として「スキップ」するのは簡単です。

したがって、5年前にOpenSSLでソフトウェアの最大の脆弱性の1つであるHeartbleedが発見されました。サーバーまたはクライアント上のメモリを不正に読み取ることができます。次に、脆弱なWebサイトの数は50万と推定されました。パッチを一度にリリースしたが、でも、2017年にされて働いていた 20万を。サイトがちハートブリード。

「開発者は、趣味として自由な時間にオープンソースプロジェクトに従事することがあります。したがって、一部のアプリケーションが放棄されていることは当然のことです。誰もプロジェクトを浮かせたくない場合、その周りに形成されたコミュニティは崩壊します。
最悪の場合、システムのユーザーがハッカーの攻撃の標的になる可能性があります。その例は、npmイベントストリームモジュールに対する昨年の攻撃です。」

プロジェクトの作者であるドミニクターは、他のタスクに切り替え、彼の頭脳を無人のままにしました。一部のユーザーは、モジュールのサポートを引き継ぐことを提案しました。

Tarrは同意し、GitHubとnpmのリポジトリへのアクセスを許可しました。時間の経過とともに、新しいメンテナはユーティリティにスクリプトを導入し、ビットコインのウォレットデータを盗んでサーバーにアップロードしました。イベントストリームの週のダウンロード数が190万件であることを考えると、この脆弱性は多数のユーザーに影響を及ぼしました。

状況を修正する方法


米国国家経済研究局に よると、オープンソース開発の主な動機付けの要因は経済的利益です。したがって、オープンソースの開発者はそれを収益化する方法を探しています。たとえば、モジュールの一部を制限付きまたは商用ライセンスに転送します。 MongoDB、Redis、その他の企業がこの道を歩んでいます。

状況についてはすでに詳しく説明しました。開発者は、コードの部分的な商用化でも追加の収入源を開き、プロジェクトに新しい人々を引き付けると信じています。しかし、そのようなモデルは、ITコミュニティにしばしば敵対的です。

このアプローチは、オープンソースソフトウェアの概念と矛盾すると考えられています。ただし、すべての人に適しているわけではありません。 2017年、Caddy は HTTP / 2 Webサーバーを発表しました商用ライセンス。しかし、何らかの理由で、1か月前に、プロジェクトは再びオープンソースに戻りました。
