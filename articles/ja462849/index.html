iノードについて

時々、中枢神経系に移行するために、サンクトペテルブルクとモスクワを中心に、DevOpsのポジションについてさまざまな大企業にインタビューします。多くの企業（Yandexなどの多くの優れた企業）では、同様の質問が2つあることに気付きました。

  • iノードとは
  • どのような理由で、ディスクへの書き込みエラーが発生する可能性があります（または、たとえば、ディスク領域が不足する理由、1つの本質）。

よくあることですが、私はこのトピックをよく知っていると確信していましたが、説明を始めるとすぐに、知識に著しいギャップが生じました。私の知識を体系化し、ギャップを埋め、不快感をなくすために、この記事を書いていますが、それでも重宝するかもしれません。

「下から」、つまり ハードドライブから（フラッシュドライブ、SSDなどの最新のものは破棄します。たとえば、ブロックサイズが512バイトであるため、20または80ギガバイトの古いドライブはすべて破棄します）。

ハードドライブは、そのスペースをバイトでアドレス指定する方法を認識していません。条件付きでブロックに分割されています。ブロック番号は0から始まります（これはLBAと呼ばれます。詳細はこちら：en.wikipedia.org/wiki/LBA）



図からわかるように、LBAブロックをHDDレベルとして指定しました。ちなみに、ディスクのブロックサイズを確認できます。

root@ubuntu:/home/serp# blockdev --getpbsz /dev/sdb
512

より高いレベルのパーティションがマークアップされ、1つはディスク全体に対応します（これも簡単にするためです）。ほとんどの場合、msdosとgptの2種類のパーティションマークアップが使用されます。したがって、msdosは2Tbまでのディスクをサポートする古いフォーマットであり、gptは512バイトブロックの1ゼタバイトまでアドレスできる新しいフォーマットです。この例では、図からわかるように、msdosタイプのセクションがあります。この場合のセクションはブロックNo. 1から始まり、MBRにはゼロセクションが使用されています。

最初のセクションでは、ext2ファイルシステムを作成しました。デフォルトでは、ブロックサイズは4096バイトで、これも図に示されています。次のようなファイルシステムのブロックサイズを確認できます。

ここで最も興味深いのは、ファイル/ home / serp / testfileを読み取る方法ですか？ファイルは、データが格納されているファイルシステムの1つ以上のブロックで構成されます。ファイル名を知って、どのように見つけるのですか？読むべきブロックは？

ここでiノードが役に立ちます。 ext2fsファイルシステムには、すべてのiノードに関する情報を含む「テーブル」があります。 ext2fsの場合のiノード数は、ファイルシステムの作成時に設定されます。 tune2fs出力の「inode count」パラメーターで必要な数を調べます。 65536個あります。 iノードには、必要な情報が含まれています。つまり、探しているファイルのファイルシステムブロックのリストです。指定されたファイルのiノード番号を見つける方法は？

名前とiノード番号の対応はディレクトリに含まれており、ext2fsのディレクトリは特殊なタイプのファイルです。また、独自のiノード番号があります。この悪循環を断ち切るために、ルートディレクトリに「固定」iノード番号「2」が割り当てられました。iノード番号2の内容を確認します。

ご覧のとおり、必要なディレクトリは番号579のブロックに含まれています。その中にホームフォルダーのノード番号が見つかり、serpディレクトリ内の要求されたファイルのノード番号が表示されるまで、チェーンに沿って次々と検索されます。突然誰かが番号が正しいかどうかを確認し、そこに適切な情報がある場合、それは難しくありません。私たちはします：

root@ubuntu:/# dd if=/dev/sdb1 of=/home/serp/dd_image bs=4096 count=1 skip=579
1+0 records in
1+0 records out
4096 bytes (4,1 kB) copied, 0,000184088 s, 22,3 MB/s
root@ubuntu:/# hexdump -c /home/serp/dd_image

出力では、ディレクトリ内のファイル名を読み取ることができます。

それで、私は主な質問に行きました：「どのような理由で記録エラーが起こり得るか」？

当然、これはファイルシステムに空きブロックがない場合に発生します。この場合、何ができますか？明らかな「不要なものを削除する」以外に、ext2、3、および4のファイルシステムには「予約済みブロック数」などの要素があることを覚えておいてください。上記のリストを見ると、「13094」というブロックがあります。これらのブロックはrootのみが書き込み可能です。しかし、問題を迅速に解決する必要がある場合、一時的な解決策をすべての人が利用できるようにして、空きスペースを少なくするにはどうすればよいですか。

root@ubuntu:/mnt# tune2fs -m 0 /dev/sdb1
tune2fs 1.42.9 (4-Feb-2014)
Setting reserved blocks percentage to 0% (0 blocks)

それら。デフォルトでは、書き込みに使用できるディスク領域の5％がなく、最新のディスクのボリュームを考えると、数百ギガバイトになる可能性があります。

他に何ができますか？空きブロックはあるが、ノードが終わっている場合、別の状況が考えられます。これは通常、ファイルシステムのブロックサイズよりも小さいファイルシステムに多数のファイルがある場合に発生します。 1つのiノードが1つのファイルまたはディレクトリに費やされており、合計で（このファイルシステムの場合）65536あることを考えると、状況は実際よりも多くなっています。これは、dfコマンドの出力から明確に確認できます。

私がiノードを使い果たした場合、私はあなたに呪文を教えません、なぜなら それらは正しくありません（正しくない場合は、お知らせください）。したがって、小さなファイルが重複するセクションでは、ファイルシステムを正しく選択する必要があります。したがって、たとえばbtrfsでは、iノードを終了できません。必要に応じて新しいものを動的に作成します。
