OOMkiller en Docker es más difícil de lo que piensas

Hola de nuevo. En previsión del inicio del curso, "Java Developer" preparó una traducción de otro pequeño material.



Recientemente, uno de los usuarios de Plumbr APM tuvo un extraño problema con una parada de emergencia del contenedor acoplable con código 137. La configuración era simple con varios contenedores anidados y máquinas virtuales, similar a una muñeca anidada:

  • propio servidor de hierro con Ubuntu;
  • muchos contenedores acoplables con Ubuntu adentro;
  • Máquina virtual Java dentro de contenedores acoplables.

Durante la investigación del problema, encontramos documentación de Docker sobre este tema. Quedó claro que la razón era una parada manual del contenedor, o una falta de memoria y la posterior intervención de oomkiller (asesino sin memoria).

Observamos syslog y vemos que, de hecho, se llamó oomkiller :

[138805.608851] java invoked oom-killer: gfp_mask=0xd0, order=0, oom_score_adj=0
[138805.608887] [<ffffffff8116d20e>] oom_kill_process+0x24e/0x3b0
[138805.608916] Task in /docker/264b771811d88f4dbd3249a54261f224a69ebffed6c7f76c7aa3bc83b3aaaf71 killed as a result of limit of /docker/264b771811d88f4dbd3249a54261f224a69ebffed6c7f76c7aa3bc83b3aaaf71
[138805.608902] [<ffffffff8116da84>] pagefault_out_of_memory+0x14/0x90
[138805.608918] memory: usage 3140120kB, limit 3145728kB, failcnt 616038
[138805.608940] memory+swap: usage 6291456kB, limit 6291456kB, failcnt 2837
[138805.609043] Memory cgroup out of memory: Kill process 20611 (java) score 1068 or sacrifice child

Como puede ver, el proceso de Java alcanzó un límite de 3145728 KB (aproximadamente 3 GB), lo que provocó que el contenedor se detuviera. Esto es bastante extraño, ya que Docker se lanzó con un límite de 4 GB (en el archivo docker-compose).

Probablemente sepa que la JVM también impone sus limitaciones en el uso de la memoria. Aunque el docker se estableció en un límite de 4 GB, JVM se lanzó con la opción Xmx=3GB. Esto puede ser aún más confuso, pero tenga en cuenta que la JVM puede usar más memoria de lo que -Xmx indicó (consulte el artículo sobre el análisis del uso de memoria en la JVM ).

Hasta que entendamos lo que está sucediendo. Docker debería permitir el uso de 4 GB. Entonces, ¿por qué OOMkiller trabajó en 3 GB? Una búsqueda adicional de información nos llevó al hecho de que hay otra limitación de memoria en el sistema operativo, que se implementó en el hardware.

Decir gracias a cgroups (grupos de control). cgroups es el motor del kernel de Linux para restringir, controlar y contabilizar el uso de recursos por parte de los grupos de procesos. En comparación con otras soluciones (equipo niceo /etc/security/limits.conf), cgroups ofrece más flexibilidad ya que pueden trabajar con (debajo) conjuntos de procesos.

En nuestra situación, cgroups limitó el uso de memoria a 3 GB (vía memory.limit_in_bytes). ¡Tenemos algunos progresos!

Un estudio de la memoria y los eventos de GC con Plumbr mostró que la mayoría de las veces la JVM usaba alrededor de 700 MB. La excepción fue solo inmediatamente antes de la parada, cuando hubo un aumento en la asignación de memoria. Fue seguido por una larga pausa de GC. Entonces parece que sucedió lo siguiente:

  • El código Java que se ejecuta dentro de la JVM está tratando de obtener mucha memoria.
  • La JVM, después de verificar que el límite Xmx de 3 GB aún está lejos, solicita asignar memoria para el sistema operativo.
  • Docker también comprueba y ve que tampoco se ha alcanzado su límite de 4 GB.
  • El núcleo del sistema operativo comprueba el límite de cgroup de 3 GB y elimina el contenedor.
  • La JVM se detiene con el contenedor antes de que pueda procesar el suyo OutOfMemoryError.

Entendiendo esto, hemos configurado todas las limitaciones de 2.5 GB para docker y 1.5 para java. Después de eso, la JVM podría manejar OutOfMemoryErrory lanzar una excepción OutOfMemoryError. Esto permitió a Plumbr hacer su magia: obtener una instantánea de la memoria con los volcados correspondientes de la pila y mostrar que había una consulta en la base de datos, que en cierta situación intentó cargar casi toda la base de datos.

recomendaciones


Incluso en una situación tan simple, había tres limitaciones de memoria:

  • JVM mediante parámetro -Xmx
  • Docker a través de las opciones en el archivo docker-compose
  • Sistema operativo a través de parámetros memory.limit_in_bytes cgroups

Por lo tanto, cuando encuentre un asesino OOM, debe prestar atención a todas las limitaciones de memoria involucradas.

Otra conclusión para los desarrolladores de docker. Parece que no tiene sentido permitir que tales "muñecas anidadas" se ejecuten en las cuales el límite de memoria del contenedor incluido es más alto que el límite de cgroups. Una simple comprobación de esto cuando se inicia el contenedor con la advertencia adecuada puede ahorrar cientos de horas de depuración para sus usuarios.
