HTTP / 3とHTTP / 2のパフォーマンス比較



Cloudflareは、9月の誕生日を祝っている昨年の9月にHTTP / 3サポートを発表しました。私たちの目標は、常にインターネットを改善することでした。標準の分野でのコラボレーションはプロセスの重要な部分であり、HTTP / 3の開発に携わることができて幸運です。

HTTP / 3はまだドラフト段階ですが、ユーザーからの新しいプロトコルへの大きな関心に気づきました（Cloudflareインフラストラクチャは、インターネットサイトの10％以上にサービスを提供しています-約）。現在までに、113,000を超えるゾーンでHTTP / 3サポートが有効になっています。実験的なブラウザを使用している場合は、新しいプロトコルを使用してこれらのゾーンにアクセスできます。非常に多くの人々がそれを組み込んでいるのは素晴らしいことです。多数の実際のWebサイトのHTTP / 3で作業することは、ブラウザからより多くのさまざまなプロパティをテストできることを意味します。

最初にGoogleと提携してHTTP / 3をリリースしました。これには、Chrome Canaryの試験的なサポートも含まれていました。それ以来、実験的なサポートを追加するブラウザーが増えています。FirefoxNightlyビルド、OperaやEdgeなどのさまざまなChromiumベースのブラウザー（Chromiumベースエンジン経由）、および予備の Safari ビルドで表示されます。私たちはこれらの進展を注意深く監視し、可能な限り支援します。HTTP / 3を有効にしたサイトの大規模なネットワークがあることで、ブラウザ開発者はコードをテストするための優れたテストの場を得ることができます。

それでは、現在の状況はどうですか？


IETF標準化プロセスには、RFC標準としてラベル付けできる「最終ドラフト」を準備する一連のドラフトが含まれます。 QUICチームメンバーは、分析、実装、相互運用性について協力して、潜在的な問題を特定します。第23ドラフトバージョンの段階でプロトコルサポートを開始し、すべての変更を追跡しました。執筆時点（2020年4月14日）で、最後は第27バージョンでした。。各ドラフトでは、QUIC定義の品質が向上し、プロトコルの動作に関する「大まかなコンセンサス」に近づきます。ドラフトの新しいバージョンごとに、仕様に変更を加えるためのバーが増えるので、無限にドピリバニヤの設定で理想を永続的に分析しないようにします。これは、バージョンごとに変更が少なくなることを意味し、最終的なRFCは、本番環境ですでに開始したプロトコルに厳密に対応します。

利点


HTTP / 3の主な利点の1つは、特に複数のオブジェクトを同時にリクエストする場合のパフォーマンスの向上です。 HTTP / 2では、TCP接続での干渉（パケット損失）により、すべてのフローが一度にブロックされます（行の先頭がブロックされます）。 HTTP / 3はUDPに基づいているため、パケットが失われたときに中断されるストリームは1つだけですが、すべてではありません。

さらに、HTTP / 3は0-RTT（ゼロの送受信時間）をサポートしています。つまり、後続の接続は最初の接続よりもはるかに速く開始できるため、接続を確立するときにサーバーからTLSを確認する必要がありません。これは、クライアントが完全なTLSネゴシエーションよりもはるかに早くデータを要求し始めることを意味します。つまり、Webサイトがブラウザーでより早くロードを開始します。

以下のアニメーションは、パケット損失の影響を示しています。最初の例では、2つのリソースを受信するために、HTTP / 2を介してクライアントからサーバーに要求が受信されます。リクエストとそれに関連するレスポンスは緑と黄色で表示されます。応答はいくつかのパケットに分割されます。悲しいかな、1つのパケットが失われる-その結果、両方のリソースの転送が遅れます。



2番目の例では、2つのリソースのリクエストがHTTP / 3経由で受信されます。 1つのパケットが失われ、黄色のリソースの転送がブロックされますが、緑色のリソースには影響しません。



セッションのネゴシエーション手順の改善は、サーバーとの「接続」がはるかに速く確立されることを意味します。つまり、データはブラウザーにより速く到着します。これが実際のトラフィックにどの程度反映されるか知りたくて、いくつかのテストを実施しました。 0-RTTの寄与を評価するために、最初のバイトまでの時間（最初のバイトまでの時間、TTFB）を測定するいくつかの制御テストを開始しました。平均して、HTTP / 3では、176ミリ秒後に最初のバイトが表示されます。 HTTP / 2の場合、201ミリ秒になります。つまり、ここでHTTP / 3を使用すると、遅延が12.4％減少します。



興味深いことに、ドラフトとRFCはプロトコルのすべての側面を規制しているわけではありません。たとえば、パケット転送方法はパフォーマンスに影響します。そして交通渋滞制御アルゴリズム。輻輳制御は、クライアントとサーバー側の輻輳したネットワークに適応する方法です。パケットが失われると、通信の品質が低下します。 QUICは新しいプロトコルであるため、過負荷管理システムを適切に設計および実装するには、実験と調整が必要です。

シンプルで安全な出発点として、損失検出と輻輳制御の仕様ではRenoアルゴリズムを推奨していますが、その他のアルゴリズムも使用できます。New Renoアルゴリズムから始めましたが、経験に基づいて、それが最適ではないと想定しました。最近CUBICに切り替えました -また、送信サイズが大きくパケット損失が大きいネットワークでは、CUBICはNew Renoよりもパフォーマンスが優れていました。すぐにこれについてもっと話します。ブログの更新をお楽しみに。

現在のHTTP / 2スタックには、BBR v1（TCP）があります。つまり、これらの輻輳制御アルゴリズムはギアサイズによって動作が異なるため、テストではリンゴとリンゴの正確な比較は行われません。ただし、HTTP / 2からHTTP / 3に切り替えると、小規模なサイトで改善が見られます。大規模な領域では、微調整および最適化されたHTTP / 2スタックによって素晴らしいパフォーマンスが示されます。

小さな15Kテストページは、HTTP / 2の458ミリ秒と比較して、平均443ミリ秒でHTTP / 3を介してロードされます。ただし、ページサイズを1 MBに増やすと、この利点はなくなります。具体的には、ネットワークでは、HTTP / 3プロトコルは、HTTP / 2-2.33秒よりも2.30秒よりも少し遅く動作します。







合成ベンチマークは興味深いですが、HTTP / 3が実際にどのように表示されるかは興味深いものです。

比較のために、Webブラウザをシミュレートして、ネットワークからサイトをダウンロードできるサードパーティサービスを引き付けたかったのです。 WebPageTestは、ページの読み込み時間を測定するための一般的なフレームワークであり、優れたカスケード図（ウォーターフォール）を備えています。バックエンドを分析するために、独自のBrowser Insightsシステムを使用して、ネットワークのエッジでタイミングを修正しました。次に、少しの自動化によって両方の部分を接続しました。

パフォーマンスを測定するためのテストケースとして、独自のブログを使用しました。 HTTP / 2とHTTP / 3の両方で、世界中の場所からサイトをロードするようにWebPageTestインスタンスを設定しました。含まれているHTTP / 3とBrowser Insights。したがって、HTTP / 3をサポートするブラウザーがページをロードするたびに、テストスクリプトはブラウザーから収集された分析データを要求しました。同じ手順をHTTP / 2でも繰り返して、比較できるようにしました。

次のグラフは、実際のblog.cloudflare.comページの読み込み時間と、HTTP / 3およびHTTP / 2メトリックの比較を示しています。地理的なポイントごとにこのような指標があります。



ご覧のとおり、HTTP / 3のパフォーマンスは、まだHTTP / 2のパフォーマンスよりも劣っています。北米では平均で約1〜4％の差があります。ヨーロッパ、アジア、南アメリカで同様の結果。これは、輻輳制御アルゴリズムの違いが原因であると考えられます。BBRv1はHTTP / 2で動作し、CUBICはHTTP / 3で動作します。将来的には、リンゴとリンゴをより正確に比較するために、両方のプロトコルに同じオーバーロード制御アルゴリズムを実装することを試みます。

結論


全体として、私たちは新しい基準の前進を支援できることを非常に嬉しく思います。私たちの実装はうまく機能し、状況によってはより良いパフォーマンスを示し、最悪の場合-同様のHTTP / 2です。標準が完成するにつれ、ブラウザーがメジャーバージョンでHTTP / 3サポートを追加することを楽しみにしています。私たちに関しては、最新のドラフトをサポートし、輻輳制御、優先順位付け、またはシステム構成（CPUおよびチャネル帯域幅）の設定に関係なく、HTTP / 3を最適化してパフォーマンスをさらに改善する新しい方法を探します。

それまでの間、実際にHTTP / 3を試してみたい場合は、ダッシュボードでアクティブにして、メインブラウザーのテストアセンブリ（Nightly、Canary）をダウンロードしてください。HTTP / 3を有効にする手順については、開発者向けドキュメントを参照してください。
