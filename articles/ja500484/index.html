Reactの代わりにWebコンポーネント-別の試み

こんにちは、ハブル！

私は最近、完全にバニラのWebコンポーネントで記述されたクライアント側のWebアプリケーションがフレームワークを使用せずにどのように見えるかを理解することを決めました。最終的に私はプロトタイプで使用するモバイルPWAテンプレートをスケッチしました。次の前提から進む：

  • DOMは状態です。私たちはフレームワークを持っていないので、すぐに機能主義を忘れて、命令型OOPに戻ります。Webコンポーネントは、状態をカプセル化し、パブリックAPIを持つ、存続期間の長いDOMノードです。それらは再作成されませんが、変更されます。これは、DOMを表現としてだけでなく、ビジネスオブジェクトのリポジトリとしても考慮する必要があることを意味します。したがって、対話の利便性を考慮してコンポーネントの階層を構築する必要があります。
  • コンポーネントの相互作用。コンポーネントは、直接呼び出し、コールバック交換、またはユーザーのDOMイベントのアップ/ダウンを介して対話できます。後者の方法は、相互のかみ合い（結合）を減らし、結合のグラフを順序付けするため、最も好ましい方法です（下の例を参照）。
    DOMイベントは階層内でのみ機能します-祖先のチェーンの下から上にポップアップしたり、すべての子孫にブロードキャストしたりできます。他の場合では、標準のブラウザAPIを使用してコンポーネントを処理します：document.querySelector（ 'page-home'）。一部のコンポーネントは、ウィンドウに登録してグローバルに使用できます：APP.route（ 'page-login'）。
  • インナースタイル。Shadow DOMは使用されないため、コンポーネントはグローバルスタイルを継承しますが、独自のスタイルを持つこともできます。<style scoped>は近い将来実装される可能性が低いため、コンポーネント名の接頭辞を使用して内部スタイルを宣言する必要がありますが、これは簡潔であり、正常に機能します（以下の例を参照）。
  • HTML / DOMとの相互作用。 DOMは状態なので、データソースはHTML要素自体の値です（value、checked、innerHTML for contenteditable = "true"など）。追加のJS変数は必要ありません。フォームの値にアクセスする便宜のために、ゲッター/セッターを作成し、それらを祖先オブジェクト（小さなライブラリが必要）に追加するだけです。フォーム値のアドレス指定は、クラス変数のアドレス指定と同じになりました。たとえば、this.passは、現在のコンポーネントの<input>子に入力されたパスワード値です。したがって、仮想DOMも双方向バインディングも必要ありません。フォームを再度開いたときにフォームを再描画する必要もありません。フォームに入力したデータは、特にパージしない限り、ナビゲーション中に保存されます。
  • ナビゲーション。ページコンポーネントは<main>コンテナ内にあり、一度作成されると削除されず、単に非表示になります。これにより、location.hashを使用してナビゲーションを実装できるようになり、標準のブラウザボタンが前後に正しく動作します。既存のコンポーネントに移動すると、onRoute（）メソッドが呼び出され、データを更新できます。

アプリケーションの構造


私たちのアプリケーションは以下で構成されています：

  • ルートコンポーネント<app-app>、window.APPからアクセス可能、ページルーターとグローバル機能を含みます。
  • コンテキストボタン付きのパネル（個別のコンポーネントに配置しませんでしたが、イベント処理を簡素化するために、それらを<app-app>レイアウトの一部にしました）;
  • ドロップダウンメニュー（個別のコンポーネント）;
  • ページコンポーネントが追加される<main>コンテナ：<page-home>、<page-login>、<page-work>。

ページは前後のナビゲーションで積み重ねられます。さらに、ボトムアップとトップダウンのデータフローを示します。

  • 承認ステータスと現在のユーザー名は<app-app>コンポーネントに保存されますが、ポップアップイベントを通じて<page-login>コンポーネントから取得されます。
  • <app-app>コンポーネントのタイマーティックは、<page-work>の子孫でのみキャッチされるブロードキャストイベントを通じて現在の値を送信します。

ポップアップイベントを通じて、アプリケーションパネルに状況依存ボタンのインストールも実装されます。ボタン自体がハンドラーと一緒にページコンポーネントで作成され、上部に送信されて、アプリレベルでインターセプトされ、パネルに挿入されます。

実装


コンポーネントの内部DOMを操作し、ダウンストリームイベントを送信するには、小さなWcMixin.jsライブラリを使用します-200行未満のコードで、その半分（ユーザー入力イベントの統合）もスローできます。それ以外はすべて純粋なバニラです。典型的なコンポーネント（承認ページ）は次のようになります。

import wcmixin from './WcMixin.js'

const me = 'page-login'
customElements.define(me, class extends HTMLElement {
   _but = null

   connectedCallback() {
      this.innerHTML = `
         <style scoped>
            ${me} {
               height: 90%; width: 100%;
               display: flex; flex-direction: column;
               justify-content: center; align-items: center;
            }
            ${me} input { width: 60%; }
         </style>
         <input w-id='userInp/user' placeholder='user'/> 
         <input w-id='passInp/pass' type='password' placeholder='password'/>
      `
      wcmixin(this)

      this.userInp.oninput = (ev) => {
         this.bubbleEvent('login-change', {logged: false, user: this.user})
      }

      this.passInp.onkeypress = (ev) => {
         if (ev.key === 'Enter') this.login()
      }
   }

   onRoute() {
      this.userInp.focus()
      this._but = document.createElement('button')
      this._but.innerHTML = 'Log in<br>⇒'
      this._but.onclick = () => this.login()
      this.bubbleEvent('set-buts', { custom: [this._but] })
   }

   async login() {
      APP.msg = 'Authorization...'
      this._but.disabled = true
      setTimeout(() => {
         this._but.disabled = false
         if (this.user) {
            this.bubbleEvent('login-change', {logged: true, user: this.user})
            APP.route('page-work')
         } else {
            APP.msg = 'Empty user !'
            this.userInp.focus()
         }
      }, 1500)
   }
})

まず、コンポーネントのローカルスタイルを確認します。次に、唯一の非標準属性w-id = `` userInp / user ''がHTMLマークアップに追加されました。 wcmixin（）関数は、この属性でマークされたすべての要素を処理し、現在のコンポーネントに変数を追加します。this.userInpは<input>要素自体（ハンドラーをハングさせることができます）を参照し、this.userは要素（username）の値です。要素へのアクセスが必要ない場合は、w-id = `` / user ''を指定できます。値のみが作成されます。

ユーザー名を入力すると、ポップアップイベントを通じて現在の値が送信され、onRoute（）メソッドで状況依存ボタンが作成されて送信されます。

承認コンポーネントがアプリケーション/パネルの上位コンポーネントについて何も知らない、つまりフックされていないことが重要です。彼は単にイベントを2階に送信し、それらをインターセプトする人は開発者次第です。<page-work>コンポーネントの<app-app>アプリケーションからのイベントの受信は、同じ方法で実装されます。

import wcmixin from './WcMixin.js'

const me = 'page-work'
customElements.define(me, class extends HTMLElement {

   connectedCallback() {
      this.innerHTML = `
         <p w-id='/msg'>Enter text:</p>
         <p w-id='textDiv/text' contenteditable='true'>1)<br>2)<br>3)</p>
      `
      wcmixin(this)

      this.addEventListener('notify-timer', (ev) => {
         this.msg = `Enter text (elapsed ${ev.val}s):`
      })
   }

   async onRoute() {
      this.textDiv.focus()
      document.execCommand('selectAll',false,null)
      const but = document.createElement('button')
      but.innerHTML = 'Done<br>⇒'
      but.onclick = () => alert(this.text)
      this.bubbleEvent('set-buts', { custom: [but] })
   }
})

そして、<app-app>コンポーネントで次のように記述します。

setInterval(() => {
   this._elapsed += 1
   this.drownEvent('notify-timer', this._elapsed)
}, 1000)

また、<app-app>コンポーネントは、そのカウンターを使用するページコンポーネントについて何も認識していません。つまり、その子孫にフックされていません。開発者がイベントの署名に同意するだけで十分です。DOMイベントは軽量で、降順のイベントはWebコンポーネントのみに送信され（単純な要素は送信されません）、昇順のイベントは、祖先のチェーン全体を標準として通過します。

実際、私が言いたかったのはそれだけです。

反対意見と提案


このようなアプローチでは、ビジネスロジックと表示が1つの「厚い」コンポーネントに混在し、一般に受け入れられているパターンに違反することがよくあります。ただし、ここではユーザー入力の表示と検証のロジックについてのみ説明します。残りのビジネスロジックは、独自の階層と対話メソッドを使用して、個別のJSクラスまたはサービスに簡単に移動できます。

なぜこれが必要なのですか。依然としてレンダリングパフォーマンスの問題があり（ガベージコレクションは無料ではありません）、ネイティブツールを使用した命令型アプローチは、JSライブラリとVDOMを使用した宣言型/関数型よりも常に高速でリソース消費が少なくなります。必要に応じて、ベンチマーク機能を引き受ければ、合意されたTORで任意のフレームワークの代表者と競争する準備ができています（これは不十分です）。
