アーキテクトの方法：認定と製品の没入

ほとんどすべての開発者は、自分のスキルをどのように伸ばすか、どの成長方向を選択するかを尋ねます。垂直-つまり、マネージャーになるか、水平-フルスタックです。神話に反して、1つの製品で何年にもわたる作業は制限ではなく、有益な機会です。この記事では、バックエンドデベロッパーであるAlexeiの経験を紹介します。Alexeiは、6年間の認定に専念し、この期間中に建築家への道を歩みました。



建築家は誰ですか？


ITアーキテクト（テクニカルリーダーシップ）は、ITプロジェクトのグローバルな問題に対処する高レベルの開発者です。彼は顧客のビジネスプロセスに没頭し、テクノロジの助けを借りて自分のタスクを解決するのを助け、これまたはその情報システムをどのように配置するかを決定します。

そのような専門家は、個々の主題領域を理解するだけでなく、プロセス全体を確認する必要もあります。

  • ビジネスタスクの設定。
  • データのプログラミング、準備、保存、処理を含む開発。
  • インフラストラクチャの展開とサポート。
  • テスト。
  • デプロイ。
  • 分析とメンテナンス。

これは、開発ライフサイクルの専門家やチームの代わりに自分を置き、システムの現在の状態を内部から理解し、誤りを特定し、目標を策定する能力を意味します。時々-個人的に外科的介入を行うため。

開発者から建築家までの専門能力開発の道のりには長い時間がかかります-通常数年です。このため、開発者は実践的なスキルと理論の知識の両方を必要とします。これらは国際的な認定で確認できます。

1つのプロジェクトで5年以上-ルーチンまたは成長の機会？


数年前、私たちは外国の顧客向けの大規模な医療ITシステムの開発に着手しました。この大規模プロジェクトには特定の問題がありました：

  • アクセス制限;
  • 不安定な製品。
  • 信じられないほど伸びたスプリントと長い調整。

「今こそスキルをアップグレードする時です」と、主要な開発者の1人であるアレクセイは、これらの困難を克服してシステムをよりよく理解するためにこの決断を下しました。

アレクセイは、トレーニングを開始する方法、取得することが重要な証明書、それを行う方法と理由に関する彼の経験を共有しました。

Lifehack：

基本的な英語トレーニングプログラムを避けてください。スキルをターゲットにする必要があります。ビジネスコミュニケーションに英語が必要な場合は、英語を学ぶ必要があります。極端に行かずにCAE（上級英語の証明書）を取らないでください。その特徴は洗練された言葉、国際的なコミュニケーションではほとんど使われない特定の表現です。



ステップ2：テクノロジースタック全体の認定


当初、このプロジェクトはORMオブジェクトリレーショナルマッピングテクノロジーに基づいていました。顧客の開発チームは彼らの発案を誇りに思っていました。すべてが高度な概念を使用して行われたため、それは難しくてクールです。

それにもかかわらず、製品、特に常に凍結するSQLサーバーの問題は珍しくありませんでした。問題の典型的な解決策はサービスを再起動することでした。顧客はチームリーダーに電話をかけ、再起動する時がきたと言いました。最後に、これを終了することにしました。

お客様はシステムのパフォーマンスを確認したいと考えていました。そのためには、プロファイリングを導入し、定期的に最適化を実行する必要がありました。当時、およそ2015年に、Ants Profilerがプロファイリングツールとして選択されましたが、それ自体は不十分でした。詳細がほとんどないため、コードの重要なブロックに関する情報を取得することは困難でした。 Ants Profilerは、システムのパフォーマンスが危険にさらされるようにコードを変更し始めました-プロファイリングが構成されている場合、すべてが単純に低下しました。そのため、アプローチを変更しました。

統計の分析から始めました

販売の統計を分析したところ、サーバー上の作業の95％が基本的な4行のビジネスロジックを持っていることが明らかになりました。彼らにとっては、単一のSQLクエリで十分であり、ORMを使用してビジネスロジックブロックによって生成されたクエリの完全なセットではありませんでした。

Alexは、ORMなしでジョブを移動するためのストアドプロシージャを提案および実装しました。このアイデアはプロジェクトの通常のパラダイムと矛盾し、チームリーダーは慎重に彼女に会いましたが、顧客はすべてを受け入れ、実装を要求しました。新しい方法では、製品での処理の遅延を4時間から数分（平均98倍）に減らすことができたため、これは当然のことでした。

それでもなお、私たちは疑問を抱きました。これは正しい決定なのか、それとも個人的な好みの問題なのでしょうか。全能C＃とORMへの信念は偶然に揺さぶられ、シンプルなソリューションの全力を示しました。
ケース2

：チームは、ORMパラダイムのフレームワーク内でデータを操作するためのリクエストを作成し、すべてのルールに従ってコンパイルされ、エラーは発生しませんでした。その処理には2〜3分かかり、これらのパラメーターは許容範囲内にありました。ただし、最も単純な選択と表示を使用した別の実装では、結果がより速く（2秒で）提供されました。

すべてのニュアンスを理解し、最良の方法を選択するために、プロジェクトスタック全体で認定されるスペシャリストを選択するときがきたことが明らかになりました。このタスクはAlexeyによって行われました。

最初の証明書

本質を理解するために、アレックスはプロジェクトの技術スタック全体をカバーするいくつかのマイクロソフト認定を通過しました：

  • TS：Microsoft .NET Framework 4を使用したWindowsアプリケーション開発
  • TS：CでのMicrosoft .NET Framework 4プログラミングによるデータへのアクセス＃
  • TS：Microsoft .NET Framework 3.5 Windowsフォームアプリケーション開発
  • PRO：Microsoft .NET Framework 3.5を使用したWindowsアプリケーションの設計と開発
  • PRO：Microsoft .NET Frameworkを使用したWindowsベースのアプリケーションの設計と開発
  • TS：Microsoft .NET Framework 2.0-Windowsベースのクライアント開発

新しいプロジェクトの作業を最適化しようとすると、チームは次の結論に達しました。

  • システムが機能するためには、コードを書くための規則に従う必要があります。インデントやコメントではなく、技術仕様-データベースへの呼び出しの数、サーバーの負荷などです。
  • 矛盾する概念の使用は問題を伴います。データベースの概念は集合論であり、ORMは操作の概念です。
  • 通常の順序に違反するアイデアは、チーム内の抵抗を満たすことができます。開発はまた、関係であり、自分の視点を主張する能力でもあります。
  • 認定により、心が広がり、使用できるものと使用できないものを理解できます。



ステップ3：コードだけではなく、もっと学ぶ


大規模なITソリューションに取り組む場合、多くの要因が重要です。たとえば、すべての開発者がネットワークパラメータに注意を払っているわけではありませんが、その帯域幅でさえビジネス問題の解決に影響を与える可能性があります。

これについての理解は、98シリーズの認定によって得られます。

  • 98-361：ソフトウェア開発の基礎
  • 98-364：データベースの基礎

それらは、物事をより広く見て、「コードのみ」という限られた概念から抜け出すことを可能にします。これらは基礎的なものですが、すべてをより深いレベルで理解するために重要です。

認定98シリーズは小規模なテストです-30問45分間。

ステップ4：プロセス管理


クリニックでの作業は、たとえばモバイルゲームの作成よりも責任のあるイベントです。ここでは、機能を追加して製品に展開することはできません。人々の健康と生活が危機に瀕しているため、調整のプロセスに従い、顧客から多くの変更を加えることが重要です。

典型的なアジャイルはこのプロジェクトで望ましい結果を生み出さず、各スプリントは十分に長く続きました。展開間は6か月から1年で行われました。

さらに、サービスを提供する10の診療所のプロセスを共通の基準にすることは技術的に不可能でした。

これらの条件下で結果をより迅速に取得するために、開発者は個人の責任とプロセスの大規模なビジョンを必要としました。これは、一定の注意の集中と高い資格を意味します。

スペシャリストがプロセスに没頭すると、彼は結果、原因、影響、全体像を明確に見ることができます。これは同時に、モチベーションと意識を高める要因でもあり、問題を解決する能力を向上させます。

十分に確立されたインフラストラクチャ、十分に構築されたアーキテクチャ、最適なコードにより、1人で多くのプロセスを実行できます。しかし、これはプロジェクトを単独で率いることができる普遍的な兵士を育てる必要があることを意味しません。コミュニケーションとチームワークが重要です。

チームでは、各開発者は、同僚が自分の行動に依存していることを理解しています。開発段階で5分の節約は、おそらくさらに5時間のテストです。これを理解するには、コミュニケーションを確立することが重要です。

彼のレベルを上げると、開発者は原則として次の結論に達します。

  • 多くはありませんが、効率的に作業できます。
  • 外部を追求して生活を複雑にしないでください。高度なテクノロジーが必ずしも問題をよりよく解決するとは限りません。
  • サイクルのすべての段階で専門家と友達になり、彼らの痛みを見つけてください。アーキテクトは、問題の特定、問題の記述、ネットワークトポロジの設計、開発、テスト、サポート、運用のプロセスを所有している必要があります。
  • からとまでのすべての機能を確認してください。
  • ITプロセスがビジネスプロセスに対応していないことがあり、これは戦わなければなりません。



ステップ5：ビッグデータのプリズムを通してアーキテクチャを理解する


このプロジェクトでは、かなり大きなデータベースを扱っていました。少なくとも、それはある点に思えた。AlexがedXでビッグデータの研究を始めたとき、プロジェクトの1.5 Tbは小さなデータベースであることがわかりました。深刻なスケール-10 Tbから、その他の方法が必要です。

認定に向けた次のステップは、ビッグデータのコースでした。彼はデータフローの構成を理解し、製品の運用を加速するのに役立ちました。また、小さなツールにも注意してください。たとえば、Excelを使用して個々のマイクロタスクを解決します。

証明書：
マイクロソフトプロフェッショナルプログラム：ビッグデータ証明書



ステップ6：開発者から設計者へ


これらすべての証明書を受け取った後、アレクセイはまだ開発者でしたが、受け取った情報には高度な抽象化があり、これは決して悪いことではないことに気付き始めました。

プロセスの大規模なビジョンは、最高レベルの認定の1つであるアーキテクトのレベルにつながります。

アーキテクト認定を求めて、アレックスはSundblad＆Sundbladによる認定ソフトウェアアーキテクト -Microsoft Platformに行きました。これはマイクロソフトによって承認されたプログラムであり、その開発は14年前に同社の本社とスウェーデンのオフィスの協力を得て始まりました。 .NET Framework、要件の収集、情報フロー管理、およびその他の多くの高レベルの問題をカバーし、アーキテクトのスキルを強く確認するものと見なされています。

プログラムには、学習コースが含まれていました。認定は体系化された知識であり、開発者から建築家に至るまで、開発の新しいラウンドに入ることができました。



要約する


Alexeyが指摘しているように、大規模なITシステムを使用するプロセスでは、プログラミングは高価な娯楽ではなく、ビジネス上の問題を解決するためのツールであることを覚えておくことが重要です。特定の課題に直面した場合、プロジェクトが停止しないように、ビジネス価値を明確に規定する必要があります。

アーキテクトは、プログラミングとその基本的なコンポーネントを特別に見ています。

  • データフローの作成および/またはサポート
  • データストリームに基づく情報ストリームの取得
  • 情報のストリームに基づくバリューストリームの取得
  • バリューストリームの収益化

アーキテクトの目を通してプロジェクトを見る場合、最後から開始する必要があります。値を定式化してから、データストリームを介してそこに移動します。

アーキテクトは、プロジェクトのグローバルなビジョンを掲げてデザインルールに従います。練習や自分の過ちを通してそれを達成することはほとんど不可能です-より正確には、それは可能ですが、非常に長い間です。認定資格を取得すると、視野が広がり、各質問の完全なコンテキストを確認し、何千人もの専門家の経験に慣れ、問題を効果的に解決するスキルを身に付けることができます。
