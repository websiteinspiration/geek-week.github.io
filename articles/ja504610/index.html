ECIES-X25519-AEAD-Ratchet-新しい高速I2Pエンドツーエンドネットワーク暗号化プロトコル

現在のElGamal / AESプロトコルにはいくつかの重大な欠点があり、I2Pネットワークの動作が比較的遅い理由の1つです。この記事で検討されているプロトコルは、ネットワークの速度と信頼性を高めるように設計されており、ストリーミングオーディオやビデオなどの新しい可能性を切り開きます。ノイズプロトコルとシグナルメッセンジャーアルゴリズムに基づいています。詳細な説明はこちら。特に、新しいプロトコルは、ElGamal / AESと共に既存のI2Pアドレスで使用できます。この記事はi2pdでの実装に特化しています



新しい暗号とエリゲーター


ほとんどの場合、NTCP2と同じ暗号化が使用されます。ElGamalがx25519に、AESがAEAD / Chacha20 / Poly1305に置き換わります。さらに、HKDF-SHA256が追加されました。これは、32または64バイトの長さのキーを作成し、OpenSSLバージョン1.1.1で使用できます。以前のバージョンでは、HMAC-SHA256による独自の実装を使用します。

ランダムな256バイトのElGamal一時公開鍵とは異なり、x25519公開鍵は楕円曲線上の点であり、攻撃者は32バイトがx25519鍵であるかどうかを判断して特定の結論を導き出すことができます。これを防ぐために、公開鍵はElligatorと呼ばれる変換を受けます。これは、それ自体が複雑であり、別の記事に値するものです。それがどのようにそしてなぜ働くかの詳細はここに記述されています。
ただし、OpenSSLに基づく実際の実装はそれほど複雑ではありません。

  1. キーの半分だけが変換に適しています。ルジャンドル記号を計算することで適合性が決まります
  2. 適切な鍵を254ビットのランダムなシーケンスに変換し、逆変換によってそこから復元できます。
  3. 任意の254ビットを任意のキーx25519に変換して戻すことができます
  4. I2Pでは、上位2ビットは最大32バイトまでのランダムな値で埋められます

同じ問題がNTCP2にも存在します。公開鍵x25519は、ルーターのアドレスをキーとしてAESによって暗号化されますが、攻撃者がI2Pの内部にいて、キーを隠すこの方法が適切でないため、別の脅威モデルがあります。

新しいプロトコルは、アドレス暗号化キー間のセッションの概念を導入します。暗号化キーはアドレスを一意に識別しますが、たとえばマルチホーミング時や再起動時など、1つのアドレスに複数のキーを含めることができます。キーのペアの間には、セッションは1つしか存在できず、セッションは双方向です。パケットを受信すると、セッションは送信者を決定します。最初のパケットを除くすべてのパケットは、パケットをタグセットとセッションに関連付ける8バイトのタグで始まります。タグは送信されませんが、セッションの両側で同時に計算されます。

パケットはAEAD / Chacha20 / Poly1305で暗号化され、暗号化キーはパケットごとに一意であり、HKDFを使用して前のパケットに基づいて両側で計算されます。

i2p::crypto::HKDF (m_CurrentSymmKeyCK, nullptr, 0, "SymmetricRatchet", m_CurrentSymmKeyCK);

接続を確立する


接続が確立されると、2つのメッセージが交換され、4つのx25519キーペアが含まれます。アドレス暗号化キーの2つのペアと一時キーの2つのペアです。サーバー（Bob）の公開暗号化キーはLeaseSetアドレスから取得されます。公開一時キーはElligatorを使用して非表示のメッセージと共に送信され、クライアント（Alice）の公開暗号化キーは別の暗号化ブロックで送信されます。

NewSession（NS）-----------> Bob
Alice <-NewSessionReply（NSR）

接続中に、計算された共有キーx25519（共有シークレット）に基づいて、ノイズプロトコルを使用してck（チェーンキー）キーが計算されます。 MixKey操作はHKDFを使用します

i2p::crypto::HKDF (m_CK, sharedSecret, 32, "", m_CK);

調整が正常に完了すると、最初のタグセットがckをキーとして作成されます

最初のタグセット 

uint8_t keydata[64];
i2p::crypto::HKDF (m_CK, nullptr, 0, "", keydata); // keydata = HKDF(chainKey, ZEROLEN, "", 64)
// k_ab = keydata[0:31], k_ba = keydata[32:63]
auto receiveTagset = std::make_shared<RatchetTagSet>(shared_from_this ());
receiveTagset->DHInitialize (m_CK, keydata); // tagset_ab = DH_INITIALIZE(chainKey, k_ab)
receiveTagset->NextSessionTagRatchet ();
m_SendTagset = std::make_shared<RatchetTagSet>(shared_from_this ());
m_SendTagset->DHInitialize (m_CK, keydata + 32); // tagset_ba = DH_INITIALIZE(chainKey, k_ba)
m_SendTagset->NextSessionTagRatchet ();


アリスはボブからNSRを受信するまでNSメッセージを送信します。最初に受信したNSRが現在のセッションを決定し、セッションがすでに存在する場合は、パケットのコンテンツのみが処理されます。各NSは、NSRの受信と復号化のみを目的とした個別の一時タグセットを作成します。

ボブは最初のNSを受信すると新しいセッションを作成し、確立されたセッションの最初のメッセージを受信するまで、キーとタグを付けてNSRを送信し続けます。

ニンニクのメッセージ構造


新しいプロトコルでのGarlicの暗号化されていないメッセージは、ElGamal / AESとは異なる構造を持っています。I2NPメッセージを配信するための指示を本質的に表す「ガベージメン」の代わりに、新しいプロトコルはNTCP2と同様の異なるタイプのブロックを使用します。各ブロックは、タイプ、長さ、内容を含むヘッダーで構成されています。現在、次のブロックタイプが使用されています。

  • ニンニクのクローブ-配信の手順が記載されたI2NPメッセージが含まれています。通常、データまたはLeaseSet。i2pdへの配信手順は現在使用されておらず、受信時に無視されますが、Javaルーターとの互換性のために送信時に入力されます。データ転送を減らすためにI2NPヘッダーが変更されました
  • 次のキー-新しいタグセットのキーをネゴシエートするために使用されます
  • ACKリクエスト-メッセージの受信確認のリクエスト。これは通常、新しいLeaseSetの送信とともに要求されます。
  • ACK-ACK要求への応答
  • パディング-0〜16バイトのランダムな長さのブロック。常に続く

タグセットと新しいキーの作成


前述のタグセットは、新しいパッケージごとのトリプルジェネレーター（数値、タグ、キー）です。番号は、0から始まる順に配置されます。番号は2バイトの番号として送信されるため、1つのタグセットで65535を超えるタグを生成することはできません。その後、新しいタグセットが必要になります。実際には、はるかに小さい値が使用されます。特に、i2pdでは、4Kパケットの送信後に新しいタグセットが作成されます。

タグセットは、共有キーと前のタグセットのDH_INITIALIZE結果に基づくDH_INITIALIZE操作で始まります。

DH_INITIALIZE 

// DH_INITIALIZE(rootKey, k)
uint8_t keydata[64];
i2p::crypto::HKDF (rootKey, k, 32, "KDFDHRatchetStep", keydata); // keydata = HKDF(rootKey, k, "KDFDHRatchetStep", 64)
memcpy (m_NextRootKey, keydata, 32); // nextRootKey = keydata[0:31]
i2p::crypto::HKDF (keydata + 32, nullptr, 0, "TagAndKeyGenKeys", m_KeyData.buf);
// [sessTag_ck, symmKey_ck] = HKDF(keydata[32:63], ZEROLEN, "TagAndKeyGenKeys", 64)
memcpy (m_SymmKeyCK, m_KeyData.buf + 32, 32);


次に、SESSTAG_CONSTANTが計算され、タグの計算に使用されます。

memcpy (m_SessTagConstant, m_KeyData.GetSessTagConstant (), 32);

新しいタグセットを作成する必要がある場合、送信者は新しいx25519キーを作成し、公開キーを含むNext Keyブロックを次のパッケージに追加します。受信者も新しいキーを作成し、応答としてNext Keyを送信します。ネゴシエーションが成功すると、当事者は新しいタグセットの使用を開始します。それ以外の場合は、以前のタグセットを使用して次のキーを送信し続けます。
