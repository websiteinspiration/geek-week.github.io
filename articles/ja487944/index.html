データサイエンスチームは、さまざまな機械学習モデルを使用して実験を行い、分類と回帰の問題を解決し、b2b領域でのビジネスプロセスを最適化しました。しかし、時系列を予測するタスクが登場したとき、ネットワーク上のこのトピックに関する利用可能な資料では、迅速なソリューションを開発するには不十分であることがわかりました。


このタスクの本質は、最大の精度（パーセントでの平均絶対誤差、またはMAPE <3％）で、1つの大規模な発電会社のニーズについて、今後3日間（72時間）の都市全体の1時間あたりの消費量を予測することでした。

3％の最大許容誤差は、予測の精度を示す非常に高い指標です。機械学習を使用する前は、エラーは3〜23％の範囲でした。これは、電気エネルギーの生成が不十分なため、卸売電力市場（自家発電よりはるかに高価）で追加の容量を購入し、過剰生産で市場で販売する必要があったためです。彼らが街に売るよりも安い。また、計画負荷と実際の負荷との偏差による悪影響は、ネットワーク内の交流電流の周波数の変化です。

多くの要因が予測の高精度の達成を妨げています。たとえば、気温の急激な上昇により、人々はすぐにエアコンからコンソールに手を伸ばし、落ちると中二階からヒーターを手に入れます。休暇中や大きなサッカーの試合にはテレビなどが含まれますいくつかのイベントは周期的であり、潜在的に予測可能ですが、他のイベントは完全に欠落しています。突然、事故のために製鉄所が機能しなくなり、すべての予測がカボチャに変わったとします（対象の都市には大きな生産企業はありませんでした）。発電量の変化の急上昇の1つが湖に座礁した船によって引き起こされ、船長が水力発電所の所有者と排水量を減らすことで合意した場合、エネルギー部門で予測を行う際の問題のかなり興味深い例がありました。当然、機械学習モデルは、履歴データに基づいてこのイベントを予測できませんでした。

サリマアックス


この問題を解決する2番目の試みは、SARIMAXを使用することでした。この方法は係数の選択に非常に時間がかかりますが、予測エラーをProphetと比較して6〜11％に減らすことができました。 

残念ながら、毎週の予測の結果のみが残りましたが、ブースティングモデルの追加機能として使用したのはこれらの予測データでした。

予測には、SARIMAパラメータ（p、d、q）（P、D、Q、s）を選択する必要があります。

  • pは自己回帰順序（AR）であり、時系列の過去の値を追加できます（「明日、過去数日間に雪が降っていた場合、おそらく雪が降る」と表現できます）。
  • dは、ソースデータの統合の順序です（現在の値から差し引く前の時点の数を決定します）。
  • qは移動平均（MA）の次数です。これにより、以前に観測されたエラー値の線形結合の形でモデルのエラーを設定できます。
  • P-p、ただし季節性。
  • D-d、ただし季節的。
  • Q-q、ただし季節的。
  • sは季節性の次元（月、四半期など）です。

毎週の履歴の季節ごとの展開で、トレンドと季節性を確認できます。これにより、seasonal_decomposeを表示できます。

ブースト


この時点で、追加の外部要因を使用しないと、必要な精度を達成できないことが明らかになりました。まず第一に、これらは気象要素です：温度、圧力、湿度、風の強さと方向、降水量。 

最初の試みは、ヤンデックスの天気のアーカイブを購入する試みでした。スタッフは優れたAPIと技術サポートを持っていますが、特に私たちの街では、データにかなりのギャップがありました。その結果、openweathermap.orgサービスで10ドルでアーカイブを購入しました。天気が非常に有用な要素であるという事実にもかかわらず、予測誤差は明らかに最終的なMAPEモデルを台無しにすることに注意することが重要です。この問題の正しい解決策は、実際の気象の過去の値ではなく、3日間の天気予報を使用するトレーニングであるという情報を知りましたが、残念ながら、そのような機会はありませんでした。 

また、時刻、曜日、休日、年の通し番号、過去の期間の多数のタイムラグと平均値のサインも追加しました。

スケーリング後のすべてのデータ（MinMaxScale）と1つのホットエンコーディング（各カテゴリ項目の値は0と1の別々の列になるため、3つの値を持つ項目は3つの異なる列になり、単位はそれらの1つだけになります） 3つの人気のあるブーストモデルXGBoost、LightGBM、CatBoostの競争。 

XGBoost


XGBoostについては多くの良いことが書かれています。2016年のSIGKDD会議で発表された彼は、注目を浴び、現在も主導的な地位を占めています。それについて聞いたことがない人のための興味深い資料。

モデルのデータを準備しましょう： 

from sklearn.preprocessing import MinMaxScaler, StandardScaler
from sklearn.model_selection import train_test_split


def gen_data(
    stop_day = '2018-06-01 23:00', 
    drop_columns = [],
    test_size=0.15,
    is_dummies=True,
    is_target_scale=False):

    df = pd.read_pickle('full.pkl')
    df = df.loc[:stop_day]
    scaler = MinMaxScaler()
    target_scaler = StandardScaler()
    y = df.hour_total
    
    if is_target_scale:        
        y = target_scaler.fit_transform(y.values.reshape(-1, 1)).reshape(1,-1)[0]
        
    X = df.drop(['hour_value', *drop_columns], axis=1)
    num_columns = X.select_dtypes(include=['float64']).columns
    X[num_columns] = scaler.fit_transform(X[num_columns])
    
    if is_dummies:
        X = pd.get_dummies(X)

    train_count_hours = len(X) - 72
    valid_count_hours = len(X) - int(len(X) * 0.2)
    X_test  = X[train_count_hours:]
    y_test  = y[train_count_hours:]
    X = X[:train_count_hours]
    y = y[:train_count_hours]

    #           ,       
    X_train, X_val, y_train, y_val = train_test_split(X, y, test_size=test_size, random_state=42)
    train_dates = X_train.index
    val_dates = X_test.index
    return X, y, X_train, X_val, y_train, y_val, X_test, y_test, target_scaler
