Bandingkan operator c # ?: Vs jika-selain vs.

Hari ini, pabrik hewan lain terbang ke tinjauan:

public static class AnimalsFactory
{
    public static Animal CreateAnimalByTernaryOperator(bool isCat)
    {
        return isCat ? (Animal)new Cat() : new Dog();
    }
}

Sekali lagi saya kesal karena C # memaksa Anda untuk melemparkan objek Cat ke Animal. Tetapi biarkan kasta menjadi lebih baik, karena melalui pernyataan if-else kodenya bahkan lebih lama:
public static class AnimalsFactory
{
    public static Animal CreateAnimalByIfElseOperator(bool isCat)
    {
        if (isCat)
            return new Cat();

        return new Dog();
    }
}

Mari kita menyimpang sebentar dari ulasan dan mencoba mencari tahu:

  • Akankah kode IL berbeda dalam contoh-contoh ini?
  • akankah salah satu contoh bermanfaat dalam kinerja?


Jawaban untuk pertanyaan pertama adalah ya, kode-IL berbeda, saya akan tunjukkan di bawah ini apa.

Mari kita beralih ke masalah kinerja. Unduh paket nuget untuk benchmark BenchmarkDotNet dan tulis tes:

public class AnimalFactoryPerformanceTests
{
    [ParamsAllValues]
    public bool IsCat { get; set; }

    [Benchmark]
    public void CreateAnimalByTernaryOperator() =>
        AnimalsFactory.CreateAnimalByTernaryOperator(IsCat);

    [Benchmark]
    public void CreateAnimalByIfElseOperator() =>
        AnimalsFactory.CreateAnimalByIfElseOperator(IsCat);
}

|  CreateAnimalByIfElseOperator | False | 1.636 ns |
|  CreateAnimalByIfElseOperator |  True | 1.360 ns |

Sungguh mengejutkan bahwa "untuk anjing" ternary bekerja lebih cepat, dan "untuk kucing" - pernyataan if-else.
Kami melihat kode-IL metode dengan operator ternary:

{

Saat membuat objek Dog , perintah IL_0000 - IL_0008 dijalankan secara berurutan , sementara saat membuat objek Cat , lompatan bersyarat terjadi ( IL_0001: brtrue.s IL_0009 ).

Seperti yang Anda tebak, untuk pernyataan if-else, kode IL dihasilkan yang tidak memerlukan lompatan kondisional untuk membuat objek Cat . Sementara objek Dog dibuat melalui cabang kondisional:

    IL_0008: ret
    IL_0009: newobj       instance void AnimalPerformance.Dog::.ctor()
    IL_000e: ret

Tambahkan pembuatan Parrot dan metode baru dengan pernyataan beralih ke pabrik :

public static class AnimalFactory
{
    public static Animal CreateAnimalByTernaryOperator(AnimalType animalType)
    {
        return animalType == AnimalType.Cat
            ? new Cat()
            : animalType == AnimalType.Dog
                ? (Animal)new Dog()
                : new Parrot();
    }

    public static Animal CreateAnimalByIfElseOperator(AnimalType animalType)
    {
        if (animalType == AnimalType.Cat)
            return new Cat();

        if (animalType == AnimalType.Dog)
            return new Dog();

        return new Parrot();
    }

    public static Animal CreateAnimalBySwitchOperator(AnimalType animalType)
    {
        switch (animalType)
        {
            case AnimalType.Cat:
                return new Cat();
            case AnimalType.Dog:
                return new Dog();
            case AnimalType.Parrot:
                return new Parrot();
            default:
                throw new InvalidOperationException();
        }
    }
}

Kesimpulan 1: Untuk pernyataan terner dan if-else, runtime secara langsung tergantung pada jumlah lompatan bersyarat yang telah berlalu di utas eksekusi.

Kesimpulan 2: Pernyataan sakelar C # dikonversi dalam kode IL menjadi pernyataan sakelar dan, secara rata-rata, bekerja sedikit lebih lama dari pernyataan percabangan biasa.

Kesimpulan 3: Segala sesuatu di atas hanya relevan untuk .NET Framework v.4.8. Setelah menjalankan tes yang sama di .NetCore, kami mendapatkan hasil yang sama sekali berbeda yang belum ditafsirkan.
