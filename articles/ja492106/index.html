前書き


JSで2つの数値を加算する関数を書くのは難しいですか？素朴な実装をする

function sum(a, b) {
	return a + b;
}

`sum（2、2）=== 4`の実装を確認してみましょう。すべてが機能しているように見えますか？実際には、関数を説明するときは、すべての種類の入力値と、関数が返すことができるものについて考える必要があります

1.1 + 2.7   // 3.8000000000000003
NaN + 2     // NaN
99999999999999992 + 99999999999999992 // 200000000000000000
2n + 2      // Uncaught TypeError: 
            // Cannot mix BigInt and other types, use explicit conversions.
{} + true   // 1
2 + '2'     // '22'

健全性は、プログラムの実行中にエラーがないことを証明するアナライザーの機能です。プログラムがアナライザーによって受け入れられた場合、そのプログラムは安全であることが保証されています。

安全なプログラムは、エラーなしで永久に機能するプログラムです。それら。プログラムはクラッシュしたりエラーをスローしたりしません。

正しいプログラム-本来あるべきことを実行し、本来すべきでないことを実行しないプログラム。正確さはビジネスロジックの実行に依存します。

タイプは、プログラム全体が安全であることを証明でき、プログラムがテストデータ内でのみ安全で正しいことをテストします（カバレッジ100％、ストライカーからの「ミュータント」の欠如、プロパティベースのテストのパスなどは何も証明できません。また、ライセンスリスクを軽減します）。定理の証明者がプログラムの正確さを証明できるという伝説があります。

TSの哲学を理解し、ツールが何を解決しようとしているか、何が重要で何を解決しようとしていないかを理解することが重要です。

構造型と名義型


構造型と名義型のタイプ1.問題


構造型付け -タイプを比較する場合、タイプの名前や宣言場所は考慮されず、タイプは「構造」に従って比較されます。

`sendEmail`を正しいアドレス` ValidatedEmail`に送信したいのですが、正しいアドレス `ValidatedEmail`を返すアドレス` validateEmail`をチェックする関数があります。残念ながら、TSを使用すると、任意の文字列を `sendEmail`に送信できます。TSの `ValidatedEmail`は` string`と同じです

type ValidatedEmail = string;
declare function validateEmail(email: string): ValidatedEmail;

declare function sendEmail(mail: ValidatedEmail): void;
sendEmail(validateEmail("asdf@gmail.com"));

// Should be error!
sendEmail("asdf@gmail.com");

構造タイピングと名義タイピング1.オファー


github.com/microsoft/TypeScript/issues/202
タイプが名目上チェックされるようにキーワード「nominal」を入力します。これで、 `ValidatedEmail`が期待される場所に` string`だけを渡すことを禁止できます

nominal type ValidatedEmail = string;
declare function validateEmail(email: string): ValidatedEmail;

declare function sendEmail(mail: ValidatedEmail): void;
sendEmail(validateEmail('asdf@gmail.com'));

// Error!
sendEmail('asdf@gmail.com');

構造型と名義型の入力1.ヒント


`Opaque`タイプを作成できます。これは、いくつかの` T`を受け取り、渡された `K`から作成されたタイプと組み合わせることで一意性を与えます。`K`は一意の記号（` unique symbol`）または文字列（これらの文字列が一意であることを確認する必要がある）のいずれかです。

type Opaque<K extends symbol | string, T> 
	= T & { [X in K]: never };

declare const validatedEmailK: unique symbol;
type ValidatedEmail = Opaque<typeof validatedEmailK, string>;
// type ValidatedEmail = Opaque<'ValidatedEmail', string>;

declare function validateEmail(email: string): ValidatedEmail;

declare function sendEmail(mail: ValidatedEmail): void;
sendEmail(validateEmail('asdf@gmail.com'));

// Argument of type '"asdf@gmail.com"' is not assignable
//  to parameter of type 'Opaque<unique symbol, string>'.
sendEmail('asdf@gmail.com');

構造型と名義型のタイプ2.問題


ドルとユーロのクラスがあり、各クラスにはドルをドルに、ユーロをユーロに追加するためのaddメソッドがあります。TSの場合、これらのクラスは構造的に同等であり、ユーロにドルを追加できます。

export class Dollar {
  value: number;

  constructor(value: number) {
    this.value = value;
  }

  add(dollar: Dollar): Dollar {
    return new Dollar(dollar.value + this.value);
  }
}

class Euro {
  value: number;

  constructor(value: number) {
    this.value = value;
  }

  add(euro: Euro): Euro {
    return new Euro(euro.value + this.value);
  }
}

const dollars100 = new Dollar(100);
const euro100 = new Euro(100);

// Correct
dollars100.add(dollars100);
euro100.add(euro100);

// Should be error!
dollars100.add(euro100);

構造型と名義型のタイプ2.オファー


github.com/microsoft/TypeScript/issues/202
文はすべて同じで、 `nominal`が付いていますが、クラスは魔法のように名義型になる可能性があるため（後で詳しく説明します）、そのような変換をより明示的な方法で行う可能性が考慮されます。

構造型と名義型の入力1.ヒント


クラスにプライベートフィールド（ `＃`またはTS c `private`のネイティブ）がある場合、クラスは魔法のように名目上になり、名前と値は何でもかまいません。`！`（明確な代入アサーション）は、TSが初期化されていないフィールドを使用しないようにするために使用されます（strictNullChecks、strictPropertyInitializationフラグが有効になっています）。

  }

class Euro {
  // #desc!: never;
  private desc!: never;
  value: number;

  constructor(value: number) {
    this.value = value;
  }

  add(euro: Euro) {
    return new Euro(euro.value + this.value);
  }
}

const dollars100 = new Dollar(100);
const euro100 = new Euro(100);

// Correct
dollars100.add(dollars100);

import { JokeError } from '../helpers';

function getJoke(isFunny: boolean): string | throws JokeError {
  /*...*/}

function getJokeSafe(isFunny: boolean): string {
  try {
    return getJoke(isFunny);
  } catch (error) {
    if (error instanceof JokeError) {
      return "";
    } else {
      // Should infer error correctly, should cast to never
      return error as never;
    }
  }
}

固定サイズのタプルがある場合、TSは要求されたインデックスに何かがあることを保証できます。これはアレイでは機能せず、TSは私たちを信頼します

安全でない操作。文


github.com/microsoft/TypeScript/issues/13778
配列へのインデックスアクセスのために、戻り値の型Tにundefinedを追加することが提案されています。ただし、この場合、インデックスでアクセスするときは、 `？`を使用するか、明示的なチェックを行う必要があります。

// interface Array<T> {
//   [n: number]: T | undefined;
// }

const dynamicNumbers: number[] = [1, 2, 3];
// Error: Object is possibly 'undefined'.
console.log(dynamicNumbers[100].toFixed());

// Optional chaining `?`
console.log(dynamicNumbers[100]?.toFixed());

// type refinement
if (typeof dynamicNumbers[100] === 'number') {
  console.log(dynamicNumbers[100].toFixed());
}

安全でない操作。ヒント


エンティティが必要以上に生成されないようにするために、以前に知られているコンテナ `Either`を使用して、インデックスを操作するための安全な関数を記述します。これにより、` Either <null、T> `が返されます。

import { Either } from "../helpers";

function safeIndex<T>(
  array: T[],
  index: number,
): Either<null, T> {
  if (index in array) {
    return Either.right(array[index]);
  }
  return Either.left(null);
}

const dynamicNumbers: number[] = [1, 2, 3];

safeIndex(dynamicNumbers, 100)
  .mapLeft(() => console.log("Nothing"))
  .mapRight(el => el + 2)
  .mapRight(el => console.log(el.toFixed()));

ボーナス タイプガード


TSは、「isSuperUser」がだれが「SuperUser」であるかを正しく決定するプログラマを信頼し、「Vasya」が追加された場合、プロンプトは表示されません。

PSは、組合の段階ですでにタイプをどのように区別するかについて考える価値があります-タグ付き組合

type SimpleUser = { name: string };
type SuperUser = { 
  name: string; 
  isAdmin: true; 
  permissions: string[] 
};
type Vasya = { name: string; isAdmin: true; isGod: true };
type User = SimpleUser | SuperUser | Vasya;

function isSuperUser(user: User): user is SuperUser {
  return "isAdmin" in user && user.isAdmin;
}

function doSomethings(user: User) {
  // Error: Cannot read property 'join' of undefined
  if (isSuperUser(user)) {
    console.log(user.permissions.join(","));
  }
}
