Linux：ロックプール/ dev /ランダムを削除する

ご存知のように、暗号的に強力な疑似乱数ジェネレータ（CSPRNG）である/ dev / randomには、ブロッキングという不快な問題が1つあります。この記事では、それを解決する方法について説明します。

過去数か月間、カーネルで乱数を生成する方法は少し改良されましたが、このサブシステムの問題はより広い時間枠で解決されました。最も最近の変更は、システム起動中に長い時間のためにブロックされるからgetrandom（）システムコールを防ぐために作られたが、この背後にある理由は、ブロッキングランダムプールの行動でした。最近のパッチでこのプールが削除され、メインコアに移動することが予想されていました。

Andy Lutomirskiは12月末にパッチの3番目のバージョンをリリースしました。それは作る「ランダムのLinuxのAPIという2つの主要なセマンティックの変更を。」パッチは新しいGRND_INSECUREフラグをgetrandom（）システムコールに追加します（Lutomirskyはそれをgetentropy（）と呼んでいますが、これはglibcでgetrandom（）を使用して固定フラグを実装しています）。このフラグは、呼び出しが常に要求されたデータ量を返すことを強制しますが、データがランダムであることを保証しません。カーネルは、最善を尽くして、ある時点で持っている最良のランダムデータを提供します。「おそらくあなたができる最善のことは、それを「安全でない」（安全でない）と呼び、セキュリティを必要とするものに使用されないようにすることです。」

パッチはブロッキングプールも削除します。現在、カーネルは2つのランダムデータプールをサポートしています。1つは/ dev / randomに対応し、もう1つは/ dev / urandomに対応しています。この2015年の記事で説明しています。ブロッキングプールは、/ dev /ランダムのプールです。このデバイスの読み取りは、要求を満たすためにシステムから「十分な」エントロピーが収集されるまでブロックされます（その名前を意味します）。プールに十分なエントロピーがない場合、このファイルからの以降の読み取りもブロックされます。

ロックプールを削除すると、/ dev / randomからの読み取りは、フラグ値がゼロに設定されたgetrandom（）のように動作します（そしてGRND_RANDOMフラグがnoopになります）。暗号化乱数ジェネレーター（CRNG）を初期化した後、/ dev / randomから読み取り、getrandom（...、0）を呼び出してもブロックされず、要求された量のランダムデータが返されます。

Lutomirsky氏は次のように述べています。「Linuxのブロッキングプールは時代遅れになっていると思います。 CRNG Linuxは、キーの生成にも使用できる十分な出力を生成します。ブロッキングプールは、物質的な意味で強力ではなく、それを維持するために、疑わしい値の多くのインフラストラクチャが必要です。」

この変更は、既存のプログラムが実際に影響を受けないようにすることを目的として行われました。実際には、GnuPGキーの生成などの長い待機の問題は小さくなるでしょう。

「これらのシリーズは既存のプログラムに違反するべきではありません。 / dev / urandomは変更されません。 / dev / randomはロード後すぐにブロックしますが、以前よりはブロックしません。既存のフラグを使用したgetentropy（）は、以前と同じように目的に対して実用的な結果を返します。

Lutomirskyは、コアがいわゆる「真の乱数」を提供する必要があるかどうかという問題は未解決のままであると指摘しました。これはある程度、ブロッキングコアによって行われるべきでした。彼は、これに対する唯一の理由を見ています：「州の基準への準拠」。 Lutomirskyは、カーネルがこれを提供する必要がある場合、これは完全に異なるインターフェースを介して実行するか、ユーザースペースに転送して、そのようなロックプールの作成に使用できる未処理のイベントパターンを取得できるようにすることを提案しました。

ステファンミュラーは彼のパッチのセットがLinux乱数ジェネレーター（LRNG）（バージョン26が現在リリースされています）は、それを必要とするアプリケーションに真の乱数を提供する方法の1つです。 LRNGは「ランダムビットの生成に使用されるエントロピーソースに関する推奨事項」SP800-90Bの要件に完全に準拠しているため、州の標準の問題を解決できます。
マシューギャレットは、「真のランダムデータ」という用語に異議を唱え、原則として、選択可能なデバイスは、予測可能なほど正確にモデル化できることを指摘しました。

ミュラー氏は、この用語は「基本的なノイズ源がエントロピーを生成するのと同じ速度で」結果を生成するだけの乱数ジェネレーターを説明するためにドイツの標準AIS 31に由来すると答えた。

用語の異なる解釈に加えて、LRNGパッチによって示唆されているように、ロックプールの存在は、少なくとも特権なしで利用できる場合、さまざまな問題を引き起こすだけです。
