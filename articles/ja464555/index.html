Bug Bountyでお金を稼ぐ簡単な方法


確かに「バグハンティング」という表現は2回以上聞いたことがあるはずです。他の誰かのプログラムに潜在的な脆弱性を見つけた場合、数百ドル（場合によっては数千ドル）を稼ぐことを拒否しないと私は確信しています。この記事では、このような脆弱性のオープンソースプロジェクトを研究するのに役立つトリックについて説明します。

フリーおよびオープンソースソフトウェアのバグ報奨金-それは何ですか？


Bug Bountyは、Webサイトやソフトウェアの開発者がバグや脆弱性を発見するための報酬を提供するさまざまなプログラムの一般的な名前です。AppleやMicrosoftなどの大企業の有名なバグバウンティプログラムに加えて、オープンソースプロジェクトの脆弱性を見つけるためのプログラムもあります。

それらの多くはHackerOneにありますが、おそらく最大のものはFOSSA-Free and Open Source Software Auditです。これは、EUが後援するさまざまなオープンソースプロジェクトの脆弱性を検索するプログラムです。賞金総額は、850,000ユーロにもなります。

参加するには？


まず、HackerOneに登録する必要があります。オープンソースのプロジェクトが必要です。 HackerOneにはリスト全体があります。

欧州連合からのバグバウンティに参加したい場合は、このプログラムに参加しているプロジェクトのリストがここにあります。ほとんどのプロジェクトでは、HackerOneに登録するだけで十分ですが、リストにリストされているプログラムの多くはintigriti.comにあります。

参加するには、自分に合ったプロジェクトを選び、参加条件をよく読んでください。彼らがあなたを満足させるなら、それは練習する時です。

脆弱性を見つけてお金を稼ぐには、プロジェクトをダウンロード（またはGitHubから複製）し、コードの各行を注意深く分析し、各式で潜在的なエラーを調べます。プログラムのセキュリティに影響を与える可能性のある何かを見つけた場合は、レポートに書き込んで開発者に送信してください。彼らがあなたの発見を価値があると評価した場合、あなたのお金はあなたのポケットにあります:)。

待って、軽さはどこですか？


そして簡単なのは、コードを手作業で排他的に分析する必要がないことです。自動モードでコードのエラーを検索するツールがあります。たとえば、静的コードアナライザー。私は開発-PVS-Studioを使用することを好みます。 PVS-Studioアナライザーは、C ++、C＃、Javaで記述されたコードのエラーを検出でき、便利なインターフェイスも備えています。さらに、無料で使用するためのオプションがいくつかあります。他にも多くのコードアナライザがあります。

もちろん、静的アナライザーはすべてのエラーから遠く離れた場所を検出できます。はい、神は彼らを祝福します！結局のところ、私たちの目標は、間違いをすばやく簡単に見つけることであり、すべてを見つけることではありません。

プロジェクトをダウンロードしてアセンブルしたら、数回クリックするだけで分析を開始できます。その結果は、アナライザーによって生成された警告（通常はかなりの量）を伴うレポートになります。 PVS-Studioでは、これらは3つの信頼レベルに分類されます。オレンジと黄色のレベルを分析結果からフィルタリングできるように、最初のレベルの警告から開始する必要があります。



分析結果のフィルタリングの例。

したがって、残りの警告を確認し、最大の危険を構成する可能性のある場所をそれらから選択するだけです。プログラムの運用中に直接再現できるか確認する必要があります。これをうまくやれば、開発者がレポートを受け入れる可能性が高まるだけでなく、支払い金額も確実に増えるでしょう。この点で、可視性はあなたの親友です。

見つかったエラーがプログラムのセキュリティに影響を与えるかどうかも検討する価値があります。確かに、この場合、あなたに支払われる金額は数倍大きくなります:)

スクリーンショットは、Visual Studioインターフェイスを示しています。ただし、これにだまされないでください。アナライザーは、Visual Studioのプラグインとしてだけでなく、LinuxやmacOSを含む独立して使用することもできます。

このアプローチの長所


まず、静的アナライザーを使用することは、エラーを見つける最も簡単な方法の1つです。コードアナライザーを使用するには、特別な知識は必要ありません。テストするコードが記述されている言語を理解するだけで済みます。

第二に、アナライザーは慎重です。人とは異なり、疲れたり警戒心を失うことはありません。したがって、それらを使用すると、ほとんど最小限のコストで任意の大きなコードベースを分析できます。

第3に、分析者は多くの場合、人よりも多くの知識を持っています。どういう意味ですか？ Androidカーネルのコードの例に関する私の考えを説明しましょう。

static void FwdLockGlue_InitializeRoundKeys() {
  unsigned char keyEncryptionKey[KEY_SIZE];
  ....
  memset(keyEncryptionKey, 0, KEY_SIZE); // Zero out key data.
}

どこに間違いがあるのでしょうか？

コンパイラーは、keyEncryptionKey配列が他で使用されていないことを確認して、コードを最適化し、そこからmemset関数呼び出しを削除できることがわかりました。そして、彼はリリース構成でのアセンブリ中にのみこれを行います。すべては問題ありませんが、しばらくの間は暗号化キーのみがRAMにロック解除されたままになるため、攻撃者が取得することができます。本当のセキュリティホール！

そして結局のところ、このエラーを自分で見つけることはほとんど不可能です。デバッグモードでは、memset呼び出しは正常に機能します。そして、このためのテストを作成することはありません...この機能について知って、自分で覚えるだけです。

しかし、プロジェクトの開発者がこの機能について知らない場合はどうなりますか？バグを検索するときにこの機能について知らない場合はどうなりますか？これはアナライザーにとって重要ではありません。これはV597診断機能を備えているため、レポートを表示すると、それについて確実に理解できます。

最後に、第四に。バグバウンティを追跡する際に静的分析を使用する最も有用な利点の1つは速度です。はい、これで、2つ、3つ、4つのプロジェクトを夕方に確認できますが、それだけではありません。

最も重要なことは、あなたが一番になることです。任意のプロジェクトでバグを発見したことに対して報酬が提供されますが、プロジェクトは引き続き確定および開発されます。開発者は新しいリリースと新機能を展開し、それに伴って新しいコードとエラーのための新しいオープンスペースが生まれます。私が説明したアプローチを使用すると、公開の初日に新しいエラーと潜在的な脆弱性を正確に調べることができます。



潜在的な脆弱性


注意深い読者は困惑するかもしれません

。一方では、プログラムのエラーをコードで検索しますが、他方では、潜在的な脆弱性について言及します。さらに、バグバウンティの観点から見ると、脆弱性はより興味深いものです。どうか明らかにしてください！

実際、エラーと潜在的な脆弱性は本質的に同じものです。もちろん、今後の調​​査でほんの少しのエラー/潜在的な脆弱性だけが、実際の脆弱性であることが判明する可能性があります。ただし、無害なブローパーと深刻な脆弱性は、コードではまったく同じに見える可能性があります。 「PVS-Studioはどのようにして脆弱性の発見に役立つのですか？」という記事は、現在では脆弱性として知られているいくつかのそのような（一見通常の）エラーを提示しています。

ちなみに、米国連邦情報・技術局（NIST）のレポートによると、アプリケーションの脆弱性の約64％はソフトウェアのエラーに関連しており、セキュリティの欠陥（セキュリティ機能の不足ではない）に関連しています。

自信を持ってPVS-Studioを入手し、バグとセキュリティの欠陥を探し始めてください！ちなみに、CWEに従って警告を分類すると、この問題の解決に役立ちます。

結論


私は読者に彼に少しの認識と金銭的報酬をもたらすであろうまさにそのバグを見つけるのを助けたと思います。静解析がこれを助けると確信しています！原則として、開発者には検出されたエラーを詳細に分析する時間がないため、検出結果が実際にプログラムに影響を与える可能性があることを証明する必要があります。最善の方法は、それを視覚化することです。そして覚えておいてください：コードのバグがセキュリティに違反する可能性が高いほど、彼らはそれに対してより多く支払うでしょう。

おそらくそれだけです。あなたの報酬を見つけて頑張ってください！
