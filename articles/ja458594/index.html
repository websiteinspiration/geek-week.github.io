L7バランシングで繰り返し要求を使用してクライアントへの応答の可能性を高めることを忘れないでください

nginxを使用してL7レベルでHTTPトラフィックのバランスをとると、ターゲットが肯定応答を返さない場合、クライアント要求を次のアプリケーションサーバーに送信できます。アプリケーションサーバーの状態をパッシブに検証するメカニズムのテストでは、ドキュメントのあいまいさと、サーバーを運用サーバーのプールから除外するためのアルゴリズムの特異性が示されました。

HTTPトラフィックの分散の概要


HTTPトラフィックのバランスをとるにはさまざまな方法があります。 OSIモデルのレベルによって、ネットワーク、トランスポート、アプリケーションの各レベルでテクノロジーのバランスをとることができます。アプリケーションのサイズに応じて、組み合わせを使用できます。

トラフィックバランシングのテクノロジーは、アプリケーションとそのメンテナンスにプラスの効果をもたらします。それらのいくつかを次に示します。負荷が複数のノードに分散されるアプリケーションの水平スケーリング。アプリケーションサーバーからのクライアント要求のフローを削除することによる、アプリケーションサーバーの使用停止の計画。アプリケーションの変更された機能に対するA / Bテスト戦略の実装。正常に機能しているアプリケーションサーバーにリクエストを送信して、アプリケーションの復元力を向上させる。

最後の関数は2つのモードで実装されます。パッシブモードでは、クライアントトラフィックのバランサーがターゲットアプリケーションサーバーの応答を評価し、特定の条件下でそれを運用サーバーのプールから除外します。アクティブモードでは、バランサーは特定のURIでアプリケーションサーバーに定期的に独立して要求を送信し、応答の特定の兆候がある場合、それを実動サーバーのプールから除外することを決定します。その後、バランサーは、特定の条件下で、アプリケーションサーバーを運用サーバーのプールに戻します。

アプリケーションサーバーのパッシブ検証と、本番サーバーのプールからの除外


nginx / 1.17.0のフリーウェア版のパッシブアプリケーションサーバーチェックを詳しく見てみましょう。アプリケーションサーバーはラウンドロビンアルゴリズムによって順番に選択され、それらの重みは同じです。

3ステップの図は、アプリケーションサーバー2にクライアント要求を送信することから始まる時間セクションを示しています。明るいインジケーターは、クライアントとバランサー間の要求/応答を特徴付けます。暗いインジケータ-nginxとアプリケーションサーバー間のリクエスト/レスポンス。



3番目のステップは、ターゲットサーバーがエラー応答を返した場合、またはまったく応答しなかった場合に、バランサーがクライアントのリクエストを次のアプリケーションサーバーにリダイレクトする方法を示しています。

サーバーが次のサーバーを使用するHTTPおよびTCPエラーのリストは、proxy_next_upstreamディレクティブで指定されます。

デフォルトでは、nginxはべき等HTTPメソッドのリクエストのみを次のアプリケーションサーバーにリダイレクトします。

クライアントは何を取得しますか？一方では、要求を次のアプリケーションサーバーにリダイレクトする機能により、ターゲットサーバーで障害が発生したときに、クライアントに満足のいく応答を提供できる可能性が高くなります。一方、最初にターゲットサーバーを呼び出し、次に次の呼び出しを行うと、クライアントへの合計応答時間が長くなることは明らかです。

最後に、アプリケーションサーバーの応答が クライアントに返され、そこでproxy_next_upstream_tries 許容試行カウンターが終了します。

次の動作中のサーバーへのリダイレクト機能を使用する場合、バランサーサーバーとアプリケーションサーバーのタイムアウトをさらに調和させる必要があります。アプリケーションサーバーとバランサーの間でリクエストを「移動」する時間の上限は、クライアントのタイムアウト、またはビジネスで指定された待機時間です。タイムアウトを計算するときは、ネットワークイベントのマージン（パケット配信中の遅延/損失）も考慮する必要があります。クライアントが毎回タイムアウトによってセッションを終了する一方で、バランサーが保証された回答を取得する場合、アプリケーションを信頼できるものにするという善意は無駄になります。



アプリケーションサーバーのパッシブヘルスチェックは、ディレクティブによって制御されます。たとえば、値に次のオプションがあります。

upstream backend {
    server app01:80 weight=1 max_fails=5 fail_timeout=100s;
    server app02:80 weight=1 max_fails=5 fail_timeout=100s;
}
server {
    location / {
        proxy_pass			http://backend;
        proxy_next_upstream		timeout http_500;
        proxy_next_upstream_tries 	1;
        ...
    }
    ...
}

パラメーターは、サーバーが使用できないと見なされるために、サーバーでの作業の指定された数の失敗した試行が発生しなければならない時間を設定します。サーバーが使用不可と見なされる時間。 上記の例では、構成ファイルの一部として、100秒以内に5つの失敗した呼び出しをキャッチするようにバランサーが構成されています。



アプリケーションサーバーを本番サーバープールに戻す


ドキュメントからわかるように、fail_timeoutの期限が切れた後のバランサーは、サーバーが作動不能であると見なすことができません。ただし、残念ながら、ドキュメントではサーバーのパフォーマンスの評価方法を明示的に規定していません。

実験がなければ、状態をチェックするメカニズムは前述のものと同様であるとしか想定できません。

何が欠けている？


  1. nginx / 1.17.0に実装されたアルゴリズムは、サーバーを稼働サーバープールに戻す前にサーバーのパフォーマンスを確認する最も公平な方法ではない場合があります。現在のドキュメントによると、少なくとも1つの要求は予期されていませんが、max_failsで指定された量です。
