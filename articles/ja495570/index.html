PVS-Studioでのウサギの穴の深さまたはC ++インタビュー


著者：Andrey Karpov、 khandeliantsPhilip Handelyants。

私がインタビューで使用した質問が著者の意図したものよりも複雑であることが判明したときの興味深い状況を共有したいと思います。C ++言語とコンパイラでは、常に目を光らせておく必要があります。退屈しないでください。

他のプログラミング会社と同様に、C ++、C＃、およびJavaの開発者の求人についてインタビューするための一連の質問があります。二重底または三重底に関する多くの質問があります。 C＃とJavaに関する質問については、他の著者がいるため、はっきりとは言えません。しかし、C ++でのインタビューのためにAndrei Karpovが編集した質問の多くは、言語機能の知識の深さを調査するためにすぐに考案されました。

これらの質問には、単純な正解を与えることができます。あなたはより深く行くことができます。これに応じて、インタビュー中に、言語のニュアンスにどれだけ精通しているかを判断します。コードアナライザーを開発しており、言語の微妙な点や「ジョーク」を非常によく理解しているため、これは私たちにとって重要です。

今日は、インタビューで最初に尋ねられた質問の1つが私たちの計画よりもさらに深かったことがどのように判明したかについての短い話をします。したがって、次のコードを示します。

void F1()
{
  int i = 1;
  printf("%d, %d\n", i++, i++);
}

そして、「何が印刷されますか？」

良い質問。すぐに知識について多くを言うことができます。人がまったく答えられない場合は考慮されません。これらは、HeadHunter Webサイト（hh.ru）での予備テストによって排除されます。いいえ、彼らはでたらめです。私たちの記憶には、精神で何かに答えたいくつかのユニークな個性がありました：

このコードは、最初にパーセンテージ、次にd、次に別のパーセンテージ、d、ワンド、n、そして2つのユニットを出力します。

明らかに、そのような場合、インタビューはすぐに終了します。

それで、今度は通常のインタビューに戻ります:)。多くの場合、彼らは次のように答えます：

1と2は印刷されます

これはインターンレベルの答えです。もちろん、そのような値を出力することはできますが、おおよそ次の答えを待っています。

これは、このコードが正確に何を出力するかということではありません。これは不特定（または未定義）の動作です。引数が計算される順序は定義されていません。呼び出された関数の本体が実行される前にすべての引数を評価する必要がありますが、これが行われる順序はコンパイラーの裁量に任されています。したがって、コードは「1、2」とその逆の「2、1」の両方を非常に適切に印刷できます。一般に、そのようなコードを書くことは非常に望ましくありません。少なくとも2つのコンパイラーによってビルドされる場合、「足で撃つ」ことが可能です。そして、多くのコンパイラはここで警告を出します。

実際、Clangを使用すると、「1、2」を取得できます。

また、GCCを使用すると、「2、1」を取得できます。

むかしむかし、MSVCコンパイラを試してみると、「2、1」も生成されました。トラブルの兆候はありません。

最近、一般的なサードパーティの目的で、最新のVisual C ++を使用してこのコードをコンパイルし、実行する必要がありました。/ O2最適化を有効にしたリリース構成でアセンブルされます。そして、彼らが言うように、彼らは自分の頭に冒険を見出しました:)。何が起こったと思いますか？ハ！ 「1、1」。

だからあなたが欲しいものを考えてください。質問はさらに深く、より混乱していることがわかります。私たち自身はこれが起こることを期待していませんでした。

C ++標準は、引数の計算をいかなる方法でも規制しないため、コンパイラーはこのタイプの指定されていない動作を非常に独特な方法で解釈します。MSVC 19.25コンパイラーによって生成されたアセンブラーコード（Microsoft Visual Studio Community 2019、バージョン16.5.1）を見てみましょう。言語標準のフラグバージョンは '/ std：c ++ 14'です。


正式には、オプティマイザは上記のコードを次のように変更しました。

void F1()
{
  int i = 1;
  int tmp = i;
  i += 2;
  printf("%d, %d\n", tmp, tmp);
}

コンパイラーの観点からは、そのような最適化はプログラムの観察された動作を変更しません。これを見ると、スマートポインタに加えて、C ++ 11標準が「マジック」関数make_shared（およびC ++ 14もmake_uniqueを追加）を追加したことを理解し始めます。このような無害な例であり、「薪を割る」こともできます。

void foo(std::unique_ptr<int>, std::unique_ptr<double>);

int main()
{
  foo(std::unique_ptr<int> { new int { 0 } },
      std::unique_ptr<double> { new double { 0.0 } });
}

狡猾なコンパイラは、これを次の計算順序に変えることができます（たとえば、同じMSVC）。

new int { .... };
new double { .... };
std::unique_ptr<int>::unique_ptr
std::unique_ptr<double>::unique_ptr

new 演算子の2回目の呼び出しで例外がスローされると、メモリリークが発生します。

しかし、元のトピックに戻ります。コンパイラーの観点からすべてが問題なかったという事実にもかかわらず、出力「1、1」は、開発者が期待する動作を考慮するために正しくないことが確実でした。次に、MSVCコンパイラで標準バージョンフラグ '/ std：c ++ 17'を使用してソースコードをコンパイルしようとしました。そして、すべてが期待どおりに動作し始め、「2、1」が印刷されます。アセンブラコードを見てください：


すべてが公正で、コンパイラは値として2と1を引数として渡しましたが、なぜすべてが劇的に変化したのですか？次の点がC ++ 17標準に追加されたことが

わかります。postfix-expressionは、expression-list内の各式とデフォルト引数の前にシーケンスされます。関連するすべての値の計算と副作用を含むパラメーターの初期化は、他のパラメーターの初期化に対して不確定に順序付けられます。

コンパイラーは依然として引数を任意の順序で計算する権利を持っていますが、現在はC ++ 17標準から始めて、次の引数とその副作用の計算を開始する権利を持ち、前の引数のすべての計算と副作用が完了した瞬間からのみです。

ちなみに、 '/ std：c ++ 17'フラグを使用してスマートポインターを使用して同じ例をコンパイルすると、すべてが正常になります。std:: make_uniqueの使用はオプションになりました。

これが判明した問題の深さの別の測定値です。理論はありますが、特定のコンパイラまたは標準の異なる解釈の形式での実践があります:)。 C ++の世界は、見た目よりも常に複雑で予想外です。

誰かが起こっていることをより正確に説明できる場合は、コメントでお知らせください。少なくとも面接で答えを知るためには、ようやく質問を理解する必要があります。 :)
