RustとLLVMでnumpy、scikit、pandasを100倍高速化：開発者Weldへのインタビュー

こんにちは、ハブル！「Weldの主な貢献者へのインタビュー：RustとLLVMでnumpy、scikit、pandasを100倍も高速化する」という記事の翻訳を紹介します。

PythonとRのデータサイエンスツールで数週間働いた後、異なる言語で使用できるCUDAのような中間表現（IR）があるかどうか疑問に思い始めました。すべての言語で同じメソッドを再実装および最適化するよりも優れたものが必要です。これに加えて、各関数を個別に最適化するのではなく、プログラム全体を最適化するために共通のランタイムがあると便利です。

さまざまなプロジェクトの調査とテストを数日間行った後、私は溶接を見つけました（学術記事を読むことができます）。

Weldは何のために設計されましたか、それはどのような問題を解決しますか？


Weldの目標は、NumPyやPandasなどの高レベルAPIを使用するアプリケーションの生産性を向上させることです。彼が解決する主な問題は、他のライブラリでは現在提供されていない、機能横断的でライブラリ間の最適化です。特に、広く使用されている多くのライブラリには、個々の関数のアルゴリズムの最新の実装（たとえば、CによるPandasで実装された高速結合アルゴリズム、またはNumPyでの高速行列乗算）がありますが、これらの関数の組み合わせを最適化する可能性はありません。たとえば、行列の乗算とそれに続く集計を実行するときに、不要なメモリスキャンを防止します。

Weldは、ライブラリが計算を共通の中間表現（IR）で表現できるようにする共通のランタイム環境を提供します。次に、このIRをコンパイラオプティマイザを使用して最適化し、ループマージ、ベクトル化などの最適化を使用して並列マシンコードにオンザフライ（JIT）でコンパイルできます。WeldのIRは最初は並列であるため、その中で表現されるプログラムは常に簡単に並列化できます。

また、Splitアノテーションと呼ばれる新しいプロジェクトもあります。これは、Weldと統合され、既存のライブラリにそのような最適化を含めるための障壁を減らすように設計されます。

numpy、pandas、scikitを最適化する方が簡単ではないでしょうか？それはどれくらい速いですか？


Weldはこれらのライブラリの関数の組み合わせの最適化を提供しますが、ライブラリの最適化は個々の関数のみの呼び出しを高速化できます。実際、これらのライブラリの多くはすでに個々の機能に対して非常に最適化されていますが、並列処理を使用しないか、メモリ階層を効率的に使用しないため、最新の機器の限界を下回るパフォーマンスを提供します。

たとえば、多次元配列（ndarray）の多くのNumPy関数は低レベルC言語で実装されていますが、各関数の呼び出しには入力データのフルスキャンが必要です。これらの配列がCPUキャッシュに収まらない場合、実行時間のほとんどは、計算を実行するのではなく、メインメモリからのデータのロードに費やすことができます。 Weldは、個々の関数呼び出しを表示し、キャッシュやCPUレジスタにデータを格納するループを組み合わせるなどの最適化を実行できます。このような最適化により、スケーラビリティが向上するため、マルチコアシステムではパフォーマンスが1桁以上向上します。



WeldプロトタイプをSpark（左上）、NumPy（右上）、TensorFlow（左下）と統合すると、ユーザーアプリケーションコードを変更することなく、独自のインフラストラクチャの実装に比べて最大30倍の改善が見られます。PandasとNumPy（右下）のライブラリ間の最適化により、パフォーマンスが2桁向上します。

Balooとは何ですか？


Balooは、Weldを使用してPandas APIのサブセットを実装するライブラリです。CWI（Centrum Wiskunde＆Informatica、Amsterdam）のマスターであるRadu Jicaによって開発されました。Balooの目標は、上記の最適化タイプをPandasに適用して、シングルスレッドのパフォーマンスを向上させ、メモリ使用量を削減し、同時実行性を確保することです。

Weld / Balooは、メモリに収まらないデータを処理するための外部操作（Daskなど）をサポートしていますか？


WeldとBalooは現在、外部操作（コア外、外部メモリ）をサポートしていませんが、この方向でオープンソース開発を喜んで受けられます！

Weldの実装にRustとLLVMを選択したのはなぜですか？すぐにルストに来ましたか？


Rustを選択した理由：

  • ランタイムは最小限であり（実際、配列の境界をチェックするだけです）、JavaやPythonなどの他の言語に簡単に組み込むことができます
  • これには、パターンマッチングなどの関数型プログラミングパラダイムが含まれています。これにより、たとえば、パターンマッチングコンパイラを最適化するためのコードの記述が容易になります。
  • ( Rust crates), .

広く使用されサポートされているオープンソースのコンパイルフレームワークであるため、LLVMを選択しました。 C / C ++の代わりにLLVMを直接生成するため、Cコンパイラは不要であり、C / C ++コードを解析しないため、コンパイル時間も短縮されます。

Rustは、Weldが最初に実装された言語ではありませんでした。最初の実装はScalaでしたが、これは代数的データ型とパターンマッチングなどの強力な機能の存在により選択されました。これにより、コンパイラーの主要部分であるオプティマイザーの作成が簡素化されました。オリジナルのオプティマイザーは、Spark SQLの拡張可能なオプティマイザーであるCatalystのように作成されました。

他のランタイムや言語にJVMベースの言語を埋め込むのは難しすぎるため、Scalaから離れました。

Weld CPU GPU, RAPIDS, data science Python GPU?


WeldとRAPIDSなどのシステムの主な違いは、さまざまなカーネル（CUDA用語では関数）を含むアプリケーションをオンザフライでコンパイルして最適化することであり、個々の関数の実装を最適化することではありません。たとえば、Weld GPUバックエンドは、個別のカーネルを呼び出すのではなく、最終的なアプリケーション用に最適化された単一のCUDAカーネルをコンパイルします。

さらに、Weld IRはハードウェアに依存しないため、GPUとCPUの両方、およびベクトルプロセッサなどの非標準の機器に使用できます。
もちろん、Weldは本質的にRAPIDSを含む同じ分野の他のプロジェクトと交差し、それらの影響下で作成されます。

Bohrium（NumPyでレイジーコンピューティングを実装）やNumba（Pythonライブラリ、JITコードコンパイラ）などのランタイム環境は、Weldの高レベルの目標を共有しています。また、Spark SQLなどのオプティマイザシステムは、溶接設計に直接影響します。

Weldには、データサイエンスライブラリの最適化以外の用途がありますか？


Weld IRの最もエキサイティングな側面の1つは、データの並行性をネイティブでサポートすることです。つまり、Weld IRのループ並列化は常に安全です。これにより、Weld IRは新しいタイプの機器にとって魅力的です。

たとえば、NECの従業員はWeldを使用して、カスタム高帯域幅ベクトルプロセッサでPythonワークロードを実行し、新しいバックエンドを既存のWeld IRに追加するだけでした。

IRは、データベースに物理操作のレイヤーを実装するためにも使用できます。また、PythonのサブセットをWeldコードにコンパイルできる機能も追加する予定です。

ライブラリは実際のプロジェクトで使用する準備ができていますか？そうでない場合は、いつ最終的な結果を期待できますか？


これらのライブラリをテストしたサンプルとベンチマークの多くは、実際のワークロードから取得されています。したがって、私たちはユーザーにアプリケーションで現在のバージョンを試してもらい、フィードバックを残してほしいと思っています。そして何よりも、彼らは独自のパッチを提案しました。

一般に、現時点では、実際のアプリケーションではすべてがそのまま機能するとは言えません。

今後数か月にわたる次のリリースでは、Pythonライブラリの使いやすさと信頼性にのみ焦点を当てます。私たちの目標は、ライブラリを実際のプロジェクトに含めるのに十分なものにすることです。また、サポートがまだ追加されていない場所で非溶接ライブラリバージョンを使用する機能。

最初の質問で述べたように、Splitアノテーションプロジェクト（ソースコードと学術記事）はこの移行を単純化するはずです。

注釈の分割は、注釈を既存のコードに追加して、それを分割、変換、および並列化する方法を決定できるシステムです。これは、Weldで最も効果的であると考える最適化を提供します（データセット全体をスキャンするのではなく、関数呼び出し間でCPUキャッシュにデータチャンクを格納します）。ただし、Splitアノテーションは、IRコンパイラに依存せずに既存のライブラリコードを使用するため、Weldよりも統合がはるかに簡単です。また、メンテナンスとデバッグが容易になり、信頼性が向上します。

まだ完全なWeldサポートがないライブラリは、Splitアノテーションを使用できます。これにより、ユーザーのフィードバックに基づいて溶接サポートを徐々に追加しながら、新しい最適化を組み込むことができます。
