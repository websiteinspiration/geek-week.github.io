ドライ


自分を繰り返さないでください。繰り返さないでください。これはソフトウェア開発の原則であり、システムの重複情報の量を減らすことを主な目的としています。目標は、システムの複雑さを減らし、その管理性を高めることです。

オリジナル（The Pragmatic Programmer、Andrew Hunt、David Thomas）では、この原則は次のように定式化されています。「各知識は、システム内で単一の一貫した信頼できる表現を持つ必要があります。」この場合、知識は、対象領域またはアルゴリズムの任意の部分（コード、データベーススキーマ、特定の対話プロトコルなど）を意味すると理解されています。したがって、システムに1つの変更を加えるには、1つの場所で1つの「知識」だけを更新する必要があります。

ばかげた例：クライアントとサーバーは互いに構造化データを送信します。なぜならこれらは異なるマシン上で実行される異なるアプリケーションであり、それらは両方ともこれらの構造の独自の実装を持つ必要があります。何かが変わった場合、2つの場所で変更を行う必要があります。この繰り返しを避けるための明白なステップは、共通のコードを別のライブラリに割り当てることです。次のステップは、構造体のフィールドにアクセスするために同じタイプのコードを記述しないように、構造体（Googleプロトコルバッファーなど）の説明に従って生成することです。

コードの重複は、DRY違反の特別なケースにすぎません。そして、それは必ずしもそうではありません。 2つのコードが同じに見えても、それぞれが独自のビジネスロジックを実装している場合、DRYは機能します。

他のDRY原則と同様-教義ではなくツール。システムが大きいほど、この原則を破るのは簡単です。まず、理想は達成不可能です。そして第二に、盲目的にDRYをフォローするとコードがより複雑になり、理解が難しくなる場合は、それを放棄することをお勧めします。

アプリケーションバイナリインターフェイス。バイナリアプリケーションインターフェイス。これは、バイナリモジュール（実行可能ファイル、ライブラリ、OS）の相互作用を定義する一連の規則です。 1つのABIに準拠して2つのモジュールを作成する必要があります。これは、バイナリ互換性のための必須条件です。この場合、問題なく対話できます（たとえば、実行可能ファイルがライブラリにリンクされ、オペレーティングシステムによって実行されます）。ABIの

例は、LinuxのELF実行可能ファイル形式とWindowsのPEです。各OSは、必要なデータ（リソース、エントリポイントなど）が、対応する形式に従ってバイナリファイルにあることを想定しています。 LinuxプログラムはWindows上で直接実行されず、その逆も同様であるため、ELFとPEは明らかに異なります。

ライブラリと実行可能ファイルのレベルで、ABIはクラス内のフィールドの配置、子孫内の基本クラス、仮想関数を実装するメカニズム、呼び出しスタックフレームの形式、呼び出された関数に引数を渡すためのルールなどを決定

できます。C++には単一のアーキテクチャとOSに依存するため、驚くべきことではない標準のABI。たとえば、x86_64上の多くのUnixライクなオペレーティングシステム（Linux、FreeBSD、MacOS）用のC ++コンパイラは、ARM- ARM C ++ ABI上のSystem V AMD64 ABIに準拠しています。 Visual C ++ ABIは公式には公開されていませんが、少なくとも部分的に再設計されています。これはSystem V ABIとは非常に異なり、名前を非表示にする（マングリング）ルールと呼び出された関数に引数を渡す（Linuxは6つのレジスターを使用し、Windowsは4つのレジスターを使用する）、およびその他の多くの違い。

APIとABIが同じままで、実装の詳細のみが変更された場合でも、バイナリ互換性が失われる可能性があります。たとえば、C ++ 11では、文字列を（ベクトルのように）文字を順次格納する必要がありました。このため、GCC 5は文字列の実装を変更する必要があり（以前はCOWが使用されていました）、バイナリの非互換性が発生しました。

牛


書き込み時にコピー。録音時にコピーします。これは、暗黙的な共有と遅延コピーとも呼ばれるリソース管理メカニズムです。コピーが必要な場合、リソースは実際にはコピーされず、リソースへのリンクが作成されるという考え方です。そして、オリジナルまたは「コピー」の変更要求がある場合にのみ、完全なコピーの作成が行われます。COW

の利点は明白です。オブジェクトのコピーは即座に行われます。オブジェクトが頻繁にコピーされてもほとんど変更されない場合は、パフォーマンスが大幅に向上する可能性があります。COW

の使用例：

  • Linuxの仮想プロセスメモリ管理。fork()ページが呼び出されると、プロセスメモリはコピーされず、共有としてのみマークされます。
  • 一部のファイルシステム（Btrfs、ZFS）およびデータベース（MS SQL Server）のスナップショット。
  • C ++ 11以前は、一部の実装でstd::stringはCOWを使用していました。C ++ 11では、要件がstd::string変更されました（ABIを参照）。
  • Qtの多くのタイプはCOWを使用しています。

FBC、FBCP


壊れやすい基本クラス（問題）。壊れやすい基本クラスの問題。これは基本的なOOP問題であり、その本質は、基本クラスを正しく変更すると、相続人の1人にエラーが発生する可能性があることです。

たとえば、無限再帰へ
struct Base
{
    virtual void method1()
    {
        // ...
    }

    virtual void method2()
    {
        // ...
        method1();  // <--     
    }
};

struct Derived : Base
{
    void method1() override
    {
        method2();
    }
};

たとえば、構成を優先して継承を放棄するか、Java用語のインターフェースを拡張することによってのみFBC問題を解決できます（C ++では、これは状態とメソッドの実装のない抽象基本クラスにのみ継承されます）。他の場合では、以下のヒントを使用して、FBCPの可能性を最小限に抑えることのみを試みることができます。

  • 必要のない場合は継承または再定義を禁止します（finalC ++およびJavaのキーワード）。
  • 相続人は基本クラスの内部にアクセスできません。通信はパブリックインターフェイスを介してのみ行われます。
  • 継承メソッドは、基本クラスの再定義されたメソッドで呼び出される仮想メソッドと再定義されたメソッド自体のみを呼び出すことができます。

LRU


最近使用されていない。押し出しは長く未使用です。これはキャッシングアルゴリズムの1つです（これらはプリエンプティブポリシーでもあります）。一般的なケースでは、キャッシュはキーと値のペアの高速ストレージと見なすことができます。その主な特徴の1つはヒット率です。このレベルが高いほど、高速キャッシュに必要な値が含まれる頻度が高くなり、低速ストレージで必要な値を見つける頻度が低くなります。しかし、メモリは決して弾力性がないため、キャッシュサイズを制限する必要があります。キャッシュアルゴリズムのタスクは、ヒットのレベルを最大化するために、必要に応じて、満たされたキャッシュからスローする要素を決定することです。

LRU誰も最も長くアクセスしていないキャッシュ要素を置き換えます。これはおそらく最も有名なキャッシングアルゴリズムです。おそらく効率とシンプルさの組み合わせによるものです。LRUのメモリ消費量はO（n）、値への平均アクセス時間はO（1）、要素を追加する平均時間もO（1）です。実装には、通常、ハッシュテーブルと二重にリンクされたリストが使用されます。

たとえば、
template <class K, class V>
class LRU
{
private:
    using Queue = std::list<std::pair<K, V>>;
    using Iterator = typename Queue::iterator;
    using Hash = std::unordered_map<K, Iterator>;

    Queue queue_;
    Hash hash_;
    const size_t limit_;

public:
    LRU(size_t limit)
        : limit_(limit)
    {
    }

    std::optional<V> get(const K& key)
    {
        const auto it = hash_.find(key);
        if (it == hash_.end())
        {
            return {};
        }
        it->second = reorder(it->second);
        return { it->second->second };
    }

    void add(K&& key, V&& value)
    {
        if (hash_.size() >= limit_)
        {
            pop();
        }
        queue_.emplace_front(std::move(key), std::move(value));
        const auto it = queue_.begin();
        hash_[it->first] = it;
    }

private:
    Iterator reorder(Iterator it)
    {
        queue_.emplace_front(std::move(it->first), std::move(it->second));
        queue_.erase(it);
        return queue_.begin();
    }

    void pop()
    {
        hash_.erase(queue_.back().first);
        queue_.pop_back();
    }
};

LRU の明らかな欠点は、それぞれn要素の2つの構造を使用するため、メモリ消費量が多いことです。LRUに加えて、 MRU（最も最近使用された）、LFU（最も頻繁に使用されなかった）、セグメント化されたLRU、2Qなど、さまざまなケースに対応する他の多くのキャッシュアルゴリズムがあります
