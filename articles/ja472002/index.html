Gradleでのパッケージの再梱包

私の記事では、Gradleを使用して非常に簡単に実装できる別のトリック、つまりライブラリパッケージの再パッケージについてお話したいと思います。このビルドシステムで少しでも作業したことがある人なら誰でも、異なるバージョンのライブラリの競合を解決する方法を自動的に知っていることを知っています。必要に応じて、たとえば、特定のバージョンのライブラリを上書きするなど、これに影響を与えることができます。


configurations.all {
    resolutionStrategy {
        force "org.ow2.asm:asm:7.2"
    }
}

残念ながら、これは常にバージョンの競合の問題を解決するのに役立ちません。たとえば、ファームウェア内の一部のhtcデバイスにすでにgsonライブラリがあるという既知の問題があり、gsonのバージョンが組み込みのものと異なる場合、ClassLoaderがメモリに1つのクラスのみをロードし、この場合はシステムクラスになるため、問題が発生する可能性があります。


この問題は、ライブラリの開発時にも発生する可能性があります。異なるバージョンの同じサードパーティライブラリ（1と2など）を使用する2つのライブラリをプロジェクトに接続すると、Gradleは最新バージョンの2番目を解決して取得します。ただし、このサードパーティライブラリに下位互換性がなく、2番目のバージョンを最初のライブラリの代わりに使用できない場合、トレースで追跡することは非常に困難な問題が発生します。最初のバージョンを待っているライブラリは2番目のクラスを受け取り、クラッシュします。


gradプラグインを作成するときにバージョンの競合が発生しました。競合するasmライブラリを使用しています。プラグインを作成した後、私はテストプロジェクトでそのパフォーマンスをチェックしました。すべてが正常で、ペットプロジェクトがすべてチェック済みですが、サードパーティの依存関係がたくさんある実際の作業プロジェクトに接続すると、問題が発生しました。
