多くの場合、あなたは、例えば、書き込むことによって、ドッカーコンテナの基本的なイメージを持つプロジェクトを開始することができますDockerfileしFROM node、「デフォルト」。ただし、ノードイメージを指定する場合は、完全にインストールされたDebian Stretchディストリビューションが、そのビルドに使用されるベースイメージであることを覚えておいてください。プロジェクトに共通のシステムライブラリやユーティリティが必要ない場合は、完全に機能するオペレーティングシステム（OS）をベースイメージとして使用しないことをお勧めします。

セキュリティレポートからわかるように、Docker Hubでテストした最も人気のある10個のDockerイメージのそれぞれに既知の脆弱性が含まれていました。プロジェクトの実行に必要なシステムツールとライブラリのみを組み合わせた最小限のイメージを優先することで、攻撃者が攻撃するスペースを最小限に抑え、安全なOSを確実に提供できます。

2.最小特権ユーザー


Dockerfile指定しない 場合USER、デフォルトでは、コンテナーはrootユーザーを使用して実行されます。実際には、コンテナにroot権限が必要な理由はほとんどありません。 Dockerはデフォルトでrootユーザーを使用してコンテナーを起動します。次に、この名前空間が実行中のコンテナーでrootユーザーにマップされると、コンテナーはDockerホストでrootアクセス権を持つ可能性があります。 rootユーザーを含むコンテナでアプリケーションを実行すると、攻撃領域がさらに拡大し、アプリケーション自体がエクスプロイトに対して脆弱である場合に、特権を昇格する簡単な方法が提供されます。

露出を最小限に抑えるには、アプリケーションのDockerイメージに専用のユーザーとグループを作成できるようにします。でディレクティブUSERを使用するDockerfileコンテナが最低限の特権のアクセスでアプリケーションを起動するようにします。

専用ユーザーが画像に存在しない場合があります。の手順を使用して、このユーザーを作成しますDockerfile。

以下は、ユニバーサルUbuntuイメージに対してこれを行う方法の完全な例です。

FROM ubuntu
RUN mkdir /app
RUN groupadd -r lirantal && useradd -r -s /bin/false -g lirantal lirantal
WORKDIR /app
COPY . /app
RUN chown -R lirantal:lirantal /app
USER lirantal
CMD node index.js

上記の例：

  • パスワードなし、ホームディレクトリのインストールなし、シェルなしのシステムユーザー（-r）を作成します
  • 作成したユーザーを、事前に作成した既存のグループに追加します（groupaddを使用）。
  • 最後の引数を、作成したいユーザーの名前に、作成したグループと組み合わせて追加します

Node.jsと高山の画像には、と呼ばれるユニバーサルユーザーがすでに含まれていますnode。以下は、一般的なユーザーノードを使用したNode.jsの例です。

FROM node:10-alpine 
RUN mkdir /app
COPY . /app
RUN chown -R node:node /app
USER node
CMD [“node”, “index.js”]

Node.jsアプリケーションを開発している場合は、公式のDockerおよびNode.jsのベストプラクティスを参照できます。

3. MITM攻撃を回避するために画像に署名して検証する


Dockerイメージの信頼性が問題です。これらの画像は、本番環境でコードを実行するコンテナとして文字通り使用するため、信頼されています。したがって、使用する画像が出版社が提供する画像とまったく同じであり、どちらの側も変更していないことを確認することが重要です。偽造は、Dockerクライアントとレジストリ間の有線接続を介して、または所有者のアカウントのレジストリをハッキングしてイメージを置き換えることによって発生する可能性があります。

Dockerイメージの検証


デフォルトのDocker設定では、信頼性を検証せずにDockerイメージを取得できます。これにより、出所と作成者が検証されていないDockerイメージが使用される可能性があります。

ポリシーに関係なく、使用する前に常にイメージを検証することをお勧めします。検証を試すには、次のコマンドで一時的にDocker Content Trustを有効にします。

export DOCKER_CONTENT_TRUST=1

ここで、署名されていないことがわかっている画像をプルアップしてみてください。リクエストは拒否され、画像は受信されません。

シグネチャーDockerイメージ


優先ドッカー認定代わりにその起源と信憑あなたが確認できない画像で、ドッカーハブによって検証し、監督されている信頼できるパートナーから得られた画像を。

Dockerを使用すると、イメージに署名できるため、別のレベルの保護が提供されます。画像に署名するには、Docker Notaryを使用します。公証人は画像の署名を確認し、画像の署名が無効な場合、画像の起動をブロックします。

上記で示したように、Docker Content Trustが有効になっていると、Dockerイメージのアセンブリがイメージに署名します。イメージが最初にサインインすると、Dockerは~/docker/trustユーザーの秘密鍵を作成して保存します。次に、この秘密鍵を使用して、追加の画像が作成されたときにそれらに署名します。

署名付きイメージの設定の詳細な手順については、公式のDockerドキュメントを参照してください。

4.オープンソースコンポーネントの脆弱性を検出、修正、追跡する


Dockerコンテナのベースイメージを選択すると、ベースイメージに関連付けられているすべてのセキュリティ問題のリスクを間接的に負います。これらは、オペレーティングシステムのセキュリティに寄与しない不適切に構成されたデフォルト設定、および選択したベースイメージに関連付けられているシステムライブラリである可能性があります。

最初のステップとして、最小限の基本イメージを使用して、問題なくアプリケーションを実行することをお勧めします。これは、脆弱性を制限することで攻撃空間を減らすのに役立ちます。一方、独自のチェックは行われず、使用されているベースイメージのバージョンで特定される可能性のある将来の脆弱性からユーザーを保護しません。

したがって、オープンソースソフトウェアの脆弱性から保護する1つの方法は、Snykなどのツールを使用して、使用するDockerイメージのすべてのレイヤーに存在する可能性のある脆弱性の継続的なスキャンと追跡を追加することです。



Dockerイメージは、次のコマンドを使用して既知の脆弱性をスキャンされます。

# fetch the image to be tested so it exists locally
$ docker pull node:10
# scan the image with snyk
$ snyk test --docker node:10 --file=path/to/Dockerfile

Dockerイメージは既知の脆弱性が監視されているため、Snykイメージの新しい脆弱性を発見した後、次のように通知して修正の推奨を提供できます。

$ snyk monitor --docker node:10
