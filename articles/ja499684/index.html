製品カードを自動的に作成し始めた方法



前回の記事では、製品名で製品を自動的に照合する方法、つまり、たとえば
ヘッドセットA4Tech Bloody G501黒
そして
A4 G501、黒（赤）{マイク付きヘッドフォン、2.2m}

- 同じです。これにより、価格と在庫状況に関連するすべてを自動化することが可能になりました。この記事では、商品の特性とイメージを使用して作業を自動化し、作業を自動化した方法について説明します。

これは、名前による商品の自動マッチングの信頼できる操作を前提とした、製品開発の論理的な継続です。私たちは自動化プロジェクトを別の会社に分離しました。私は、クライアントの1つを例として使用してプロジェクトについて話し、クライアントのデータを許可を得て共有しています。次は簡単な紹介です。次に、使用したアルゴリズムについて説明します。

どこから始めましたか


1年前の初期状態は、12万点の商品カタログ、そのうち7万点が在庫あり。商品の自動比較と作成が開始された後（最後の記事についてのように）、カタログはすぐに100万個になり、そのうちのいくつかは60から65万個の在庫でした。しかし、残りが特性を持っていなかったため、小売購入に利用できるのは35万のポジションのみでした。カテゴリに応じて、製品が販売されるためには、一定の割合の特性が満たされている必要があります。一部のカテゴリでは、写真が必要です。 35万点の商品のうち、12万点には画像がありませんでした。在庫はあるが小売りで販売されていない商品は卸売りで販売できることを説明します。通常、名前と価格が記載されたExcelファイルを送信します。しかし、カードを備えたb2bポータルは間違いなくプラスです。また、アグリゲーターに配置することもできます。アグリゲーターには、製品に関するより詳細な情報があります。

画像と仕様は半自動モードで入力されました。今年は他の顧客も現れましたが、今ではこのプロセスが12つか2つの比較的大きな店舗でどのように構築されたかを知っています。これは多かれ少なかれ標準的なものです。時々サイトを管理するプログラマーは、ストアのテーマでサイトパーサーを作成します。さらに、管理パネルのコンテンツ部門は、カタログの蒸し物と製品、ソースのカテゴリとカタログのカテゴリを比較します。次に、各カテゴリで、プロパティごとに、対応、置換、単位変換を構成します。一部の商品のカードは完全に手動で入力されます。商品のごく一部が外部の請負業者に渡されます。だから、それは私たちのクライアントと一緒でした。

カタログの約6％が1か月に使用されなくなります。つまり、サイトで50万個の製品を販売するには、数量を同じレベルに維持するためだけに、月に3万枚のカードを作成する必要があります。

私たちはできるだけ体系的に問題に取り組み、製品カードの作成プロセスをできるだけ自動化することを決定しました。

データはどこで入手しますか


発生する最初の質問は、ソースデータをどこで取得するか、およびどれだけ取得できるかです。主なソースは2つあります。

  • ベンダーファイルとAPI
  • インターネットで見つけたデータ

2番目のポイントには、サイトの解析が含まれます。パーサーを作成するためのインフラストラクチャを作成しました。これにより、サイトの解析を約30分で構成できます。次に、このインフラストラクチャで約100のパーサーが作成されました。現時点では、1600万個の製品に関する情報を収集し、3500万個の画像とドキュメント（手順、データシート）をダウンロードしてCDNにアップロードしています。CDNとして、Bunny CDNを使用します。画像を保存するときは、知覚的ハッシュを考慮して、後で重複をすばやく見つけることができるようにします。これについては、以下で詳しく説明します。ヒントと設定のないパーサー自体が任意のサイトの製品に関する情報を収集できるのであれば理想的ですが、これらは遠い計画です。

製品カードに画像をどのように記入するか


これは最も単純で最も明白な部分です。製品を名前で照合するメカニズムはすでにあり、迅速かつ正確に機能します。この場合、約0.1％の誤検知に関心があります。偽陰性は実際には結果に影響を与えません。大きな違いはないため、最終的には10または11のソースで作業します。比較は毎秒5〜6千製品の速度で動作し、1500万製品すべてが1時間未満でどのカタログにも適合します。比較すると商品のすべての画像がわかるため、これで終了することは可能です。ただし、実際には、それらは異なるサイトで異なる組み合わせで複製されます。類似した画像のセットから最高の品質を選択し、製品カードを後で手動で編集する場合のために残りをオプションとして残しておくのが正しいでしょう。

類似画像の識別は2つの段階で行われます。最初の段階では、ファイルをダウンロードするときに、画像ごとに知覚ハッシュを考慮します。最初に、無地の背景を削除し、フィールドをトリミングします。次に、結果の画像にDCTハッシュを適用します。Habréに関する記事があったときにこのアルゴリズムについて。

各製品のマッチングが完了すると、ハッシュが既に計算された一連の画像が得られます。これをクラスターに分割し、各クラスターから最高品質の画像を選択する必要があります。このようなセットのサイズは、数ユニットから数百ユニットまでさまざまです。一般的な値は10〜40です。このサイズでは、2次複雑度アルゴリズムを適用できます。主な演算（ハミング距離の計算）は非常に高速です。

次のように画像をクラスターに分割します。

  1. セットの最初の画像は最初のクラスターを形成します
  2. 次の各画像は、各クラスターの最初の画像と比較されます
  3. 最も高い類似度係数がしきい値より大きい場合、既存のクラスターに画像を追加します
  4. それ以外の場合は、1つのイメージから新しいクラスターを作成します

その後、各クラスターから最高品質の画像を取得します。最初の近似では、品質は画像のサイズと解像度によって決まります。サイズと解像度は大きいほど良いですが、サイズと解像度の比率は妥当な範囲内である必要があります。このような分析は実際には十分です。

さらにもう1つのニュアンスがあります。商品の写真は自動的に見つかりましたが、それでもなお、人の方が重要です。また、画像のシーケンスを変更することを決定した場合、そのような機会があるはずです。しかし同時に、手動で編集した後にカードを「フリーズ」するという考えは好きではありません。アルゴリズムを改善し、新しい画像を見つけて、将来的にカードを自動的に改善したいと考えています。

これを念頭に置いて、次のアプローチが有効なオプションであることが判明しました：手動で変更されたすべての画像（およびここでの変更は、可能な限り幅広いアクションセット（確認、除外、順序の変更）を意味します）は常にそのまま残ります。自動モードで将来変更できる他のすべての画像。

製品カードにプロパティをどのように記入するか


プロパティを使用すると、写真よりも少し複雑になります。異なるソースでは、同じプロパティが異なる方法で呼び出されたり、異なる単位で表現されたり、明示的な対応がない場合があります（たとえば、HDMI：はい、インターフェース：HDMI、USB）。

プロパティの定義から始め、次のタイプを考え出しました。

  • うーん、ダメ
  • 数+単位
  • 単一の選択
  • 複数の選択肢
  • ライン

さらに、「寸法」と「測定単位」という用語を使用しますが、これらの意味は異なります。

次に、測定の単位とそれらを変換する方法について説明しました。現在、メートルからデシベルまで、すでに100以上あります。それらはすべて、次元に応じてグループに分けられ、グループ内で相互に関連付けることができます。たとえば、馬力はワットに、ルーメンはニュートンメートルに変換できます。これは不可能です。

次に、プロパティを独立したステップに分割するプロセスを分割して、各ステップを他のステップとは別に改善およびテストできるようにしました。カタログの各製品について、次の操作を実行します。

  1. 商品をソースからの商品と名前で比較します（複雑なプロセスがあり、前の記事で詳しく説明しています）
  2. 入力するプロパティ定義のリストを形成する
  3. すべての関連製品のプロパティのリストを作成します
  4. 2項の各定義に3項の値のリストを関連付けます
  5. リストを単一の値に変換する

プロジェクトをテストする方法


このプロジェクトには注目すべき機能があります。コアは、最適化タスクのみで構成されるテクノロジーです。さらに、これらのタスクのほとんどには正確な解決策がなく、速度と精度を考慮した独自の基準が考案されました。これは、テストが本当に必要な場所です。

プロジェクトのアーキテクチャは、複雑なアルゴリズムが可能な限り単純なステップに分割されるように設計されています。各ステップには、最適化基準と一連のテストがあります。正確さだけでなくスピードも重視。少なくとも、私たちはあらゆる変化の世話をしていました。このような各ステップはスポーツプログラミングのタスクに似ており、変更のたびに結果を比較する機会があります。
