自動構成


Spring AMQPを使用してRabbitMQとの対話（SSLなどを使用）を実装するとします。私たちは、次のようないくつかの豆を、作成する必要がありConnectionFactory、RabbitAdmin、RabbitTemplateとAbstractRabbitListenerContainerFactory。

しかし、RabbitMQにいくつかの仮想ホストがあると想像してください。デフォルトでは、Spring AMQPでは、との手動構成を使用して、仮想ホストごとにそのようなエコシステムを作成する必要がRabbitTemplateありRabbitListenerます。



ご覧のとおり、この構成はすべて時間がかかり、頭痛の種になる可能性があります。

この問題を認識して、提案されたライブラリーはこれらすべてのBeanを自動的に構成します。あなたがする必要があるのは、で設定を決定することだけですapplication.properties、そして魔法が起こります！
うわー、しかし設定はどのように見えますapplication.propertiesか？

すべてが非常に簡単です。まず、プロパティのプレフィックスがあります-this spring.rabbitmq.custom。

その後、イベントの名前を指定する必要があります。ライブラリが行うすべての設定はこの名前に基づいているため、これは非常に重要な部分です。

イベント名の後に、プロパティと値を指定できます。すべてのプロパティはドキュメントで説明されています。

したがって、テンプレートは次のとおりです。 以下は、2つの異なる接続の構成例です。

しかし、この新しい戦略を説明する前に、RabbitMQとSpring AMQPのデフォルトの動作を見てみましょう。

デフォルトでは、RabbitMQはメッセージのライフサイクル全体を制御できる再試行処理を提供していません。

たとえば、RabbitMQ 3.8より前のバージョンでは、メッセージヘッダーには、再試行の回数を制御するプロパティがありませんでした。

RabbitMQのデフォルトの動作：

  • 存続時間（TTL、存続期間）を定義していない場合、RabbitMQは常にメッセージをキューに入れようとします。
  • TTLを定義したがdlxを定義しなかった場合、TTLの後、メッセージはキューから削除され、失われます。
  • TTLとdlxを定義した場合、TTLの後、メッセージはdlxで定義された交換器に送信されます。

Spring RabbitMQのチューニング


このライブラリは、再試行用に別のキューを使用する別のアプローチを採用しています。このように、expirationメッセージパラメータとx-deathパラメータを使用してTTLを制御し、再試行回数を制御できます。
しかし、どうやって？

再試行キューでdlxの概念を使用して、メッセージをメインキューに再送信します。したがって、x-deathパラメーターにアクセスでき、メッセージの存続期間をプログラムで判別できます。

注意。このライブラリはSpring AMQPの拡張であるため、デフォルトの再試行戦略を使用でき、このライブラリはBeanの自動構成にのみ使用できます。

出版社


ライブラリにはクラスがあり、RabbitTemplateHandlerとても使いやすいです。getRabbitTemplateBeanを取得するには、メソッドを呼び出し、仮想ホストをパラメータとして渡す必要がありますRabbitTemplate。その後、convertAndSendメソッドを呼び出し、それを交換およびルーティングキーパラメータに渡すことができます。

注：交換およびルーティングキーは、アノテーションを使用して取得できます値。

例：

@Value("${spring.rabbitmq.custom.some-event.exchange}")
private String exchangeSomeEvent;

@Value("${spring.rabbitmq.custom.some-event.queueRoutingKey}")
private String routingKeySomeEvent;

@Autowired
private final RabbitTemplateHandler rabbitTemplateHandler;

public void sendMessage(final String message) {
   rabbitTemplateHandler.getRabbitTemplate("some-event").convertAndSend(exchangeSomeEvent, routingKeySomeEvent, message);
}

加入者


加入者も非常に簡単です。必要なのは、メソッドにRabbitListenerアノテーション（デフォルトではSpring AMQPアノテーション）を追加してnameを渡すことだけですcontainerFactory。
仮想ホストの正しいcontainerFactory名を見つける方法は？

あなたはそれを知る必要はありません、ただイベントの名前を渡すだけで、ライブラリはあなたのためにすべての魔法を行います！このライブラリには、推奨される再試行とdlqを含める機能もあります。

それらをオンにするのは非常に簡単です。メソッドにアノテーションを追加しEnableRabbitRetryAndDlq、プロパティの名前を引数として渡す必要があります。

注：再試行およびdlqで処理する例外を指定することもできます。デフォルトException.classで処理されます。つまり、すべての例外を処理します。

例：

@RabbitListener(containerFactory = "some-event", queues = "${spring.rabbitmq.custom.some-event.queue}")
@EnableRabbitRetryAndDlq(event = "some-event", exceptions = { IllegalArgumentException.class, RuntimeException.class })
public void onMessage(Message message) {
   ...
}
