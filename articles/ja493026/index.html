製品のテスト：カナリアデプロイメント

カナリアは絶えず歌う小鳥です。これらの鳥は、メタンと一酸化炭素に敏感です。空気中の過剰なガスの濃度が低くても、意識を失ったり、死んだりします。金鉱山労働者と鉱山労働者は鳥を獲物として捕獲しました。カナリア諸島が歌っている間、あなたは仕事をすることができます。黙っていれば、鉱山にはガスがあり、出発する時間です。鉱山労働者は地雷から生き残るために小鳥を犠牲にした。



同様の慣行がITにも見られます。たとえば、サービスまたはアプリケーションの新しいバージョンを本番環境にデプロイし、その前にテストを行うという標準的なタスクで。テスト環境は高すぎる可能性があり、自動化されたテストは私たちが望むすべてをカバーしていないため、テストして品質を犠牲にすることは危険です。このような場合でも、カナリアデプロイメントアプローチは、実際の本番トラフィックが少し新しいバージョンで起動されるときに役立ちます。このアプローチは、本番用に新しいバージョンを安全にテストするのに役立ち、大きな目的のために小さなものを犠牲にします。より詳細には、アプローチがどのように機能するか、何が有用であり、どのようにそれを実装するかが、Andrey Markelov（Andrey_V_Markelov）、Infobipでの実装例を使用します。

Infobipの主要なソフトウェアエンジニアであるAndrey Markelovは、11年間、金融および電気通信分野でJavaアプリケーションを開発してきました。彼はオープンソース製品を開発し、アトラシアンコミュニティに積極的に参加し、アトラシアン製品のプラグインを作成しています。伝道者のプロメテウス、Docker、Redis。


Infobipについて


これは、銀行、小売業者、オンラインストア、運送会社がSMS、プッシュ、レター、ボイスメッセージを使用して顧客にメッセージを送信できるようにするグローバルな通信プラットフォームです。このようなビジネスでは、顧客が時間通りにメッセージを受信できるように、安定性と信頼性が重要です。

Infobip ITインフラストラクチャの数：

  • 世界中に15のデータセンター。
  • 稼働中の500の固有のサービス。
  • チームの数をはるかに超える2500インスタンスのサービス。
  • 毎月4.5 TBのトラフィック。
  • 45億の電話番号

ビジネスは成長しており、それに伴いリリースの数も増えています。お客様はより多くの機能を求めているため、1日60回のリリースを行っています。しかし、これは困難です。多くのサービスがありますが、チームはほとんどありません。エラーなしで本番環境で機能するコードをすばやく作成する必要があります。

いずれの場合でも、バージョンをテストしても、ほとんどの場合、展開後に問題が発生します。あなたはそれをあなたの手でテストすることができます、それは自動化することができます、あなたはそれをテストすることができません-どんな場合でも問題が発生します。それらを解決する最も簡単で最も正しい方法は、作業バージョンにロールバックすることです。そうしてはじめて、被害に対処し、原因と修正をすることができます。

それで、私たちは何が欲しいですか？

問題は必要ありません。お客様が私たちより速くそれらを見つけた場合、それは私たちの評判を打つでしょう。したがって、お客様よりも早く問題を見つける必要があります。積極的に対処することで、被害を最小限に抑えます。

同時に、導入を加速したいこれにより、迅速かつ簡単に、それ自体で、チームからのストレスなしに行われます。エンジニア、DevOpsエンジニア、プログラマーを保護する必要があります。新しいバージョンのリリースにはストレスが伴います。チームは消耗品ではなく、人材の合理的な活用に努めています。

展開の問題


クライアントのトラフィックは予測不可能です。クライアントのトラフィックが最小になる時期を予測することは不可能です。クライアントがキャンペーンをどこでいつ開始するかはわかりません。おそらく今夜はインドで、明日は香港で。大きな時差があることを考えると、午前2時の配置であっても、顧客が苦しむことがないという保証はありません。

プロバイダーの問題。メッセンジャーとプロバイダーは私たちのパートナーです。新しいバージョンのデプロイ中にエラーが発生するクラッシュが発生する場合があります。

分散チーム。クライアント側とバックエンドを開発するチームは、異なるタイムゾーンにいます。このため、彼らはしばしば彼らの間で合意することができません。

データセンターをステージ上で繰り返すことはできません。1つのデータセンターには200のラックがあります。これをサンドボックスで繰り返すと、ほぼうまくいきません。

ダウンタイムは許されません！たとえば、99.99％の時間で作業する場合、許容範囲のアクセシビリティ（エラーバジェット）があり、残りの割合は「ミスをする権利」です。100％の信頼性を達成することは不可能ですが、ダウンタイムとダウンタイムを常に監視することが重要です。

煙のテスト


展開の問題を解決する別の方法。前の例で、チームAが新しいバージョンをデプロイしたい場合のスモークテストの動作を見てみましょう。

まず、チームは1つのインスタンスを本番環境にデプロイします。モックからインスタンスへのメッセージは実際のトラフィックをシミュレートするため、通常の毎日のトラフィックと一致します。問題がなければ、チームは新しいバージョンをユーザートラフィックに切り替えます。



2番目のオプションは、追加の鉄で展開することです。チームは本番環境でテストしてから切り替え、すべてが機能します。



煙のテストの欠点：

  • テストは信頼できません。プロダクションと同じトラフィックをどこで取得できますか？昨日または1週間前に使用できますが、必ずしも現在のものとは一致しません。
  • 維持するのは難しいです。アクティブなレコードがリポジトリに送信される場合、テストアカウントをサポートし、展開する前に常にそれらをリセットする必要があります。これは、サンドボックスでテストを作成するよりも困難です。

ここでの唯一のボーナスは、パフォーマンスを確認できることです。

カナリアリリース


煙テストの欠陥のため、カナリアリリースを使い始めました。

鉱山労働者がカナリアを使用してガスレベルを示す方法と同様の方法で、ITで自分自身を見つけました。サービスレベルアグリーメント（SLA）を満たそうとしている間に、新しいバージョンに実際の本番トラフィックを少し投入します。SLAは「ミスをする権利」であり、年に1回（または他の期間）使用できます。すべてがうまくいけば、さらにトラフィックを追加します。そうでない場合は、以前のバージョンを返します。



実装とニュアンス


カナリアリリースをどのように実装しましたか？たとえば、顧客のグループがサービスを介してメッセージを送信します。



デプロイは次のようになります。バランサーの下から1つのノードを削除し（1）、バージョンを変更し（2）、トラフィックを個別に開始します（3）。



一般に、1人のユーザーが不満であっても、グループの全員が幸せになります。すべてが順調なら-すべてのバージョンを変更します。



ほとんどの場合にマイクロサービスがどのように見えるかを概略的に示します。

Service Discoveryと2つのサービス（S1N1とS2）があります。最初のサービス（S1N1）は開始時にService Discoveryに通知し、Service Discoveryはそれを記憶します。 2つのノード（S2N1およびS2N2）を持つ2番目のサービスも、起動時にService Discoveryに通知します。



最初の2番目のサービスはサーバーとして機能します。最初のサービスは、サーバーに関する情報をService Discoveryに要求し、それを受信すると、サーバーを検索してチェックします（「ヘルスチェック」）。チェックすると、メッセージを送信します。

誰かが2番目のサービスの新しいバージョンをデプロイしたい場合、彼は2番目のノードがカナリアノードになることをサービスディスカバリに伝えます。カナリアノードをバランサーの下から削除し、最初のサービスはそこにトラフィックを送信しません。



バージョンを変更すると、Service Discoveryは2番目のノードがカナリアになったことを認識します。負荷を軽減できます（5％）。問題がなければ、バージョンを変更し、ロードを返して作業します。

最も単純なオプションは、1つのノードが常にカナリアである場合です。このノードは常にトラフィックが少なく、そこから展開を開始します。問題が発生した場合は、彼女の作業を配備と比較します。たとえば、エラーが2倍ある場合、ダメージは2倍に増加しています。

バージョン分析


いくつかのバージョン管理戦略があります。

カナリアノードのみのメトリックを表示します。最も単純なオプションの1つは、新しいバージョンをデプロイして、作業のみを調査することです。しかし、この時点でエンジニアがログの調査を開始し、神経質にページをリロードしている場合、このソリューションは他のソリューションと変わりません。

カナリアノードは他のノードと比較されます。これは、フルトラフィックで実行される他のインスタンスとの比較です。たとえば、トラフィックが少ない場合、状況が実際のインスタンスよりも悪い、または良くない場合、何かが間違っています。

カナリアノードは過去にそれ自体と比較されます。カナリアに割り当てられたノードは、履歴データと比較できます。たとえば、1週間前にすべてが問題なかった場合、現在の状況を理解するためにこのデータに焦点を合わせることができます。

オートメーション


エンジニアを手動の比較から解放したいので、自動化を実装することが重要です。デプロイメントパイプラインプロセスは通常、次のようになります。

これはGroovyへのパイプラインです。

while (System.currentTimeMillis() < endCanaryTs) {

    if (isOk) {

    }   else {

このサイクルでは、新しいノードを1時間比較するように設定しました。カナリアプロセスがまだプロセスを終了していない場合は、関数を呼び出します。彼女はすべてが順調かどうかを報告しますdef isOk = compare(srv, canary, time, base, offset, metrics)。

すべてが順調である場合- sleep DEFAULT SLEEPたとえば、少しの間、続行します。そうでない場合は、終了します-デプロイメントは失敗しました。

メトリックの説明。関数がcompareDSLの例でどのように見えるかを見てみましょう。

metric(
    'errorCounts',
    'rate(errorCounts{node=~"$canaryInst"}[5m] offset $offset)',
    {   baseValue, canaryValue ->
        if (canaryValue > baseValue * 1.3) return false 
        return true
    }
)

エラー数を比較し、過去5分間の1秒あたりのエラー数を知りたいとします。

ベースノードとカナリアノードの2つの値があります。カナリアノードの値は現在のものです。基本- baseValueその他の非カナリアノードの値です。経験と観察に基づいて設定した式に従って、値を互いに比較します。値がcanaryValue悪い場合、展開は失敗し、ロールバックします。

なぜこれがすべて必要なのですか？

人間は数百、数千の測定基準をチェックできない特にそれを迅速に行うために。自動比較により、すべてのメトリックをチェックし、問題をすばやく警告できます。通知時間は重要です。最後の2秒間に何かが発生した場合、被害は15分前に発生した場合ほど大きくありません。誰かが問題に気づくまで、サポートを書きます。サポートをロールバックするために顧客を失う可能性があります。

プロセスがうまくいき、すべてが問題なければ、他のすべてのノードを自動的にデプロイします。現在、エンジニアは何もしていません。彼らはカナリアを実行するときにのみ、どの指標を使用するか、どのくらいの期間比較を行うか、どの戦略を使用するかを決定します。



問題がある場合は、カナリアノードを自動的にロールバックし、以前のバージョンで作業し、見つかったエラーを修正します。メトリックにより、新しいバージョンからの損傷を簡単に見つけて確認できます。
