Node.js開発者ツール ジョブキュー

Webアプリケーションやモバイルアプリケーションのバックエンドを実装する場合、最も単純なものでも、データベース、メール（smtp）サーバー、redisサーバーなどのツールを使用することが慣例になっています。使用されるツールのセットは常に拡大しています。たとえば、amqplibパッケージのインストール数（週あたり65万インストール）で判断されるメッセージキューは、リレーショナルデータベース（mysqlパッケージは週あたり46万インストール、週あたり80万インストール）とともに使用されます。

今日、ほとんどすべての実際のプロジェクトで、必要性は生じますが、これまでのところ1桁少ないジョブキューについてお話ししたいと思います。

したがって、ジョブキューを使用すると、一部のタスクを非同期で実行でき、実際には、指定された入力パラメーターを使用して、設定された時間に関数を実行できます。

パラメータに応じて、タスクを実行できます。

  • ジョブのキューに追加した直後。
  • 設定された時間に一度;
  • 予定通り何度も。

ジョブキューを使用すると、実行中のジョブにパラメータを転送し、失敗したジョブを追跡して再実行し、同時に実行するジョブの数に制限を設定できます。

Node.js上のアプリケーションの大部分は、Webおよびモバイルアプリケーション用のREST-APIの開発に関連しています。 REST-APIの実行時間を短縮することは、ユーザーがアプリケーションを快適に使用するために重要です。同時に、REST-APIの呼び出しにより、時間のかかる操作やリソースを大量に消費する操作が開始される可能性があります。たとえば、購入後、モバイルアプリケーションにプッシュメッセージをユーザーに送信するか、CRM REST-APIで購入を行うためのリクエストを送信する必要があります。これらのクエリは非同期で実行できます。ジョブキューを操作するためのツールがない場合、それを正しく行うにはどうすればよいですか？たとえば、メッセージキューにメッセージを送信し、これらのメッセージを読み取り、これらのメッセージに基づいて必要な作業を実行するワーカーを開始できます。

実際、これがジョブキューの役割です。ただし、よく見ると、ジョブキューにはメッセージキューといくつかの根本的な違いがあります。最初に、メッセージ（静的）はメッセージキューに入れられ、ジョブキューには何らかの機能（関数の呼び出し）が含まれます。次に、ジョブキューは、特定の作業を実行するある種のプロセッサ（ワーカー）の存在を意味します。同時に、追加の機能が必要です。負荷が増加した場合に備えて、プロセッサプロセッサの数を透過的にスケーリングする必要があります。一方、ピーク負荷を平準化し、サービス拒否を防止するために、1つのプロセッサワーカーで同時に実行するタスクの数を制限する必要があります。これは、非同期ジョブを実行できるツールが必要であることを示しています。さまざまなパラメータの設定は、REST-APIを使用してリクエストを行うのと同じくらい簡単です（さらに簡単であれば、より良い）。

メッセージキューを使用すると、ジョブがキューに入れられた直後に実行されるジョブキューを実装することは比較的簡単です。しかし、多くの場合、設定された時間に1回、またはスケジュールに従ってタスクを完了する必要があります。これらのタスクでは、Linuxでcronロジックを実装する多くのパッケージが広く使用されています。根拠がないようにするために、node-cronパッケージには週に48万のインストール、node-scheduleには週に17万のインストールがあると言います。

今日は、私が一緒に作業するブルパッケージの使用を検討します。なぜこの特定のパッケージを選択したのですか（自分の選択を他のパッケージに課すことはありませんが）。その瞬間、メッセージキューの便利な実装を探し始めたとき、bee-queueプロジェクトはすでに停止していました。 bee-queueリポジトリで指定されたベンチマークによると、kueの実装は他の実装よりもはるかに遅れており、さらに定期的に実行されるタスクを実行する手段が含まれていませんでした。アジェンダプロジェクトは、mongodbデータベースに格納するキューを実装します。これは、キューにタスクを配置するときに超信頼性が必要な場合に、大きな利点になります。ただし、これが決定的な要素であるだけではありません。当然、私はすべての持久力ライブラリオプションをテストし、キューに多数のタスクを生成しましたが、アジェンダから中断されない作業を取得することはできませんでした。特定の数のタスクを超えると、アジェンダが停止し、タスクを機能させることができなくなりました。

そのため、bullパッケージはredisサーバーをバックエンドとして使用するため、十分な速度とスケーラビリティーを備えた便利なAPIを実装するbullに落ち着きました。特に、redisサーバーのクラスターを使用できます。

キューを作成するときは、ジョブキューに最適なパラメータを選択することが非常に重要です。多くのパラメータがあり、それらのいくつかの値はすぐに私に届きませんでした。数多くの実験を経て、私は次のパラメータに落ち着きました。

const Bull = require('bull');

const redis = {
  host: 'localhost',
  port: 6379,
  maxRetriesPerRequest: null,
  connectTimeout: 180000
  };

const defaultJobOptions = {
  removeOnComplete: true,
  removeOnFail: false,
};

const limiter = {
  max: 10000,
  duration: 1000,
  bounceBack: false,
};

const settings = {
  lockDuration: 600000, // Key expiration time for job locks.
  stalledInterval: 5000, // How often check for stalled jobs (use 0 for never checking).
  maxStalledCount: 2, // Max amount of times a stalled job will be re-processed.
  guardInterval: 5000, // Poll interval for delayed jobs and added jobs.
  retryProcessDelay: 30000, // delay before processing next job in case of internal error.
  drainDelay: 5, // A timeout for when the queue is in drained state (empty waiting for jobs).
};

const bull = new Bull('my_queue', { redis, defaultJobOptions, settings, limiter });

module.exports = { bull };

私は強気の「そのまま」に行く機会を使用します-いくつかのコアでプロセッサープロセッサーを並列化するために。これを行うには、2番目のパラメーターで、プロセッサーワーカーを起動するコアの数を設定し、3番目のパラメーターで、ジョブ処理関数の定義を含むファイル名を設定します。このような機能が必要ない場合は、2番目のパラメーターとしてコールバック関数を渡すだけです。

add（）メソッドを呼び出すことによってタスクがキューに入れられ、キュー名とオブジェクトがパラメーターで渡されます。これらは、将来タスクハンドラーに渡されます。たとえば、ORMフックで、新しいニュースのエントリを作成した後、すべてのクライアントに非同期にプッシュメッセージを送信できます。

  afterCreate(instance) {
      bull.add('push:news', _.pick(instance, 'id', 'title', 'message'), options);
  }

パラメータのイベントハンドラは、add（）メソッドとdone（）関数に渡されたパラメータを持つタスクオブジェクトを受け入れます。これらの関数は、タスクが完了したことを確認したり、タスクがエラーで終了したことを通知したりするために呼び出す必要があります。

const { firebase: { admin } } = require('../firebase');
const { makePayload } = require('./makePayload');

module.exports = (job, done) => {
  const { id, title, message } = job.data;
  const data = {
    id: String(id),
    type: 'news',
  };
  const payloadRu = makePayload(title.ru, message.ru, data);
  const payloadEn = makePayload(title.en, message.en, data);
  return Promise.all([
    admin.messaging().send({ ...payloadRu, condition: "'news' in topics && 'ru' in topics" }),
    admin.messaging().send({ ...payloadEn, condition: "'news' in topics && 'en' in topics" }),
  ])
    .then(response => done(null, response))
    .catch(done);
};

そして最後に、ちょっとしたライフハック。私が言ったように、ブルはバックエンドとしてredisサーバーを使用します。Redisサーバーの再起動時にタスクが非常に少なくなる可能性。しかし、システム管理者は単に「大根のキャッシュをクリア」することができるという事実を知っていて、特にすべてのタスクを削除している間、私は主に定期的に実行されるタスクについて心配しました。この点で、私はそのような定期的なタスクを再開する方法の機会を見つけました：

const cron = '*/10 * * * * *';

const { bull } = require('./app/services/bull');

bull.getRepeatableJobs()
  .then(jobs => Promise.all(_.map(jobs, (job) => {
    const [name, cron] = job.key.split(/:{2,}/);
    return bull.removeRepeatable(name, { cron });
  })))
  .then(() => bull.add('check:status', {}, { priority: 1, repeat: { cron } }));

setInterval(() => bull.add('check:status', {}, { priority: 1, repeat: { cron } }), 60000);

つまり、タスクは最初にキューから除外され、次に再度設定されます。また、すべて（alas）はsetInterval（）によって設定されます。実際、そのようなライフハックがなければ、私はおそらく雄牛に定期的なタスクを使用することを決定しなかったでしょう。
