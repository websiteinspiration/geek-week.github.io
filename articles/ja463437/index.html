サーバーコードを記述せずにモバイルデバイスをデータベースに接続する

こんにちは、ハブル！Foresightモバイルプラットフォームを使用した経験を共有したいと思います。

独自のサーバーを作成せずにリモートでデータを保存するタスクがある場合、最初に頭に浮かぶのはFirebase Realtime Databaseツールです。ほとんどの読者は何が問題になっているのかを知っていますが、残りについては説明します。Firebase Databaseを使用すると、NoSql形式でデータをリモートで保存できます。


Firebaseからの画像


使用の利点から：

  • グーグルによってサポートされています
  • ベース側でコードを記述する必要はありません
  • クイックワーク
  • 5GBまで無料
  • Firebase認証サービスとの統合とかなりの数の承認プロバイダー
  • Android、iO、Unityなどのサポート
  • リアルタイムで同期：データベースのデータを更新すると、クライアントで更新されます

その後、モバイルプラットフォームのForesightサーバー（以下、FMFと呼びます）に切り替えます。プラットフォームのデプロイ方法と必要なものは、ドキュメントに記載されています。 FMFは、データベースサーバーへのコネクタを持つサーバーです。また、必要なPostgreSQLに接続することもできます。このためのコードを記述する必要はありません。私が使用しなければならないのは、ブラウザーを介した管理パネルです。サーバー、データベースの必要なアドレスを入力します。管理パネルには、設定と環境ツリー



があります。設定について詳しくは、ドキュメントをご覧ください。

残念ながら、FMFには試用版や無料期間はありません。また、FMFは安全な顧客インフラストラクチャに展開されます。そのため、サーバーへのリンクを提供することはできません。ただし、作業の本質を理解するには、次の情報で十分です。

環境ツリーに新しい環境を作成します。たとえば、Leonid_environmentと呼びます。環境内で、Leonid_projectなどのプロジェクトを作成します。私はプロジェクトに入り、そこでデータソースを作成します。ここではPostgreSQLです。テーブルが保存されているサーバーとデータベースのパラメーターを入力します。



データソースの中に行き、「インポート」をクリックします。ソースの名前に、PostgreSQLで導入したプロシージャの名前を入力します。この場合は、fruits_get_by_colorです。モバイルクライアントの場合、必ずしもそうとは限りませんが、別の名前を付けることができます。Fruits_get_by_color_for_androidという名前を付けます。

Fmp + Android


新しいプロジェクトを作成しています。プラットフォームとともに、それを操作するためのフレームワークが配布されます。フレームワークをプロジェクトにプルしますファイル→インポートモジュール→* .aar



プロジェクトでは、サーバーアドレスを貼り付ける一連の定数と、環境の名前、プロジェクトの名前、ログイン、パスワード、リソース名を発明したFMPから宣言します。

    private static final String MY_URL = "http://mobilefmp.dev.fs.world";
    private static final VersionAPI MY_VERSION_API = VersionAPI.V_1;
    private static final String MY_ENVIRONMENT = "Leonid_environment";
    private static final String MY_PROJECT = "Leonid_project";
    private static final String MY_VERSION = "v1";
    private static final String MY_LOGIN = "Leonid";
    private static final String MY_PASSWORD = "123123";
    private static final String MY_RESOURCE = "fruits_get_by_color_for_android";

ここでは、直接PostgreSQLを指すのではなく、FMPを指すことに注意してください。次に、HyperHiveオブジェクトを作成し、宣言した変数を設定します。

hyperHive = new HyperHiveState(getApplicationContext())
                .setHostWithSchema(MY_URL)
                .setApiVersion(MY_VERSION_API)
                .setEnvironmentSlug(MY_ENVIRONMENT)
                .setProjectSlug(MY_PROJECT)
                .setVersionProject(MY_VERSION)
                .buildHyperHive();

次に、認証を実行する必要があります。これを行うには、ユーザー名とパスワードをauth（）メソッドに渡します。

boolean status = hyperHive.authAPI.auth(MY_LOGIN, MY_PASSWORD, true).execute().isOk();

trueの場合、テーブルの内容をクエリできます。 tableStreamCallParamsオブジェクトで、jsonの形式でfruit_colorパラメータを渡し、黄色の値を入力します。フルーツテーブルを作成し、カラーパラメータを入力として受け取るプロシージャをPostreSQLで作成したことを思い出してください。これは、デバイスでこのパラメーターを入力するために必要でした。

        TableStreamCallParams tableStreamCallParams = new TableStreamCallParams();
        String data = "{\"fruit_color\": \"yellow\"}";
        tableStreamCallParams.setData(data);
        String status = hyperHive.requestAPI.tableStream(MY_RESOURCE, tableStreamCallParams).execute();

要求の後、FMPサーバーに移ります。FMPサーバーはPostgreSQLサーバーにアクセスします。その結果、黄色い果物のリストが表示されるはずです



。SQLiteデータベースのLinesストリームイン。この場合、黄色はバナナとレモンだけでした。FMF側のキャッシュデータのダウンロード速度は、通常のインターネット速度で1秒あたり約10,000行です。

私が説明したすべての手順は、ドキュメントで読むことができます。また、iOおよびその他のオペレーティングシステムへの接続に関する情報もあります。完全なアクティベーションコード

を提供します。

メインスレッドからリクエストを行い、コード量を減らすためにエラーを処理しませんでした。

機能的なFMP


FMF機能はこれに限定されません。データベースのサーバー側で、受信だけでなく、渡された値に応じて新しい行を作成するためのプロシージャを作成できます。値の配列を渡すことができます。制限はデータベース自体の内部にのみあります。

FMPの基本機能の大まかな手順を説明します。

  • プラットフォーム側でデータをキャッシュし、デルタを受け取る（テーブル全体ではなく、変更された行のみ）
  • ユーザーアクションのロギング
  • ユーザー、ユーザーグループ、デバイスIDによるアクセス制御
  • LDAPによる認証
  • SMTP、Citrix XenMobile、Sentryとの統合
  • CryptoPro証明機関でのデータ署名
  • ファイルストレージはFMFの側面ではありません
  • FMP側のローカルデータベース（サードパーティのデータベースには接続できませんが、Firebaseのようにローカルでデータベースを作成できます）
  • Firebase Cloud Messaging、Appleプッシュ通知、Windowsプッシュ通知用のコネクタ。メーリングリストを作成し、通知テンプレートをプッシュする

次のフレームワークがあります。

  • アンドロイド
  • iOS
  • UWP
  • WinCE
  • セイルフィッシュ、オーロラ（ロシアのセイルフィッシュ）

次のコネクタがあります。

  • 先見分析プラットフォーム

調査結果


FMFプラットフォームとFirebase Realtime Databaseの比較は、少なくとも、おなじみのAndroidおよびiO開発者と何かを比較したかった。実際、PMFの機能と目的は少し異なります。競合他社の中には、SAPモバイルプラットフォーム、IBM初のモバイルプラットフォーム、Optimum CDCがあります。

結論として、以下の場合はFMFに注意を払う価値があります。

  • システムの代金を喜んで支払うかなり大規模な顧客
  • モバイルデバイスとデータベースサーバーの間にレイヤーを書きたくない人
  • 商品の倉庫など、大量のデータがある
  • ロシアのソフトウェアの使用が基本である
  • データ保護が重要な役割を果たす
