Princípios do SOLID nas imagens



Se você conhece a programação orientada a objetos , provavelmente já ouviu falar dos princípios do SOLID . Essas cinco regras para o desenvolvimento de software definem o caminho a seguir quando você escreve programas para torná-los mais fáceis de escalar e manter. Eles ganharam fama graças ao programador Robert Martin.

Existem muitos artigos excelentes na Web que falam sobre os princípios do SOLID, mas eu dificilmente me deparei com ilustrações entre eles. Por causa disso, é difícil para as pessoas com tendência à percepção visual de informações - como eu - apreender a essência e não se distrair.

O objetivo principal deste artigo é entender melhor os princípios do SOLID por meio de ilustrações, além de determinar a finalidade de cada princípio. O fato é que alguns dos princípios parecem semelhantes, mas as funções desempenham diferentes. Pode acontecer que você siga um princípio e viole o outro, embora na aparência não exista muita diferença entre eles.

Para facilitar a leitura, mencionei apenas as classes aqui, no entanto, tudo o que foi dito no artigo também se aplica a funções, métodos e módulos, portanto, lembre-se disso.

Bem, vamos começar.

Princípios do SOLID



S - Responsabilidade Única


Cada classe deve ser responsável por apenas uma operação.



Se uma classe é responsável por várias operações ao mesmo tempo, a probabilidade de erros aumenta - ao fazer alterações em uma das operações, você mesmo, sem suspeitar, pode afetar outras.

Objetivo

O princípio serve para separar os tipos de comportamento, devido aos quais os erros causados ​​por modificações em um comportamento não se estendem a outros tipos que não estão associados a ele.

O - Aberto Fechado (Aberto Fechado)


As classes devem estar abertas para extensão, mas fechadas para modificação.



Quando você altera o comportamento atual de uma classe, essas alterações afetam todos os sistemas que trabalham com essa classe. Se você deseja que a classe realize mais operações, a opção ideal não é substituir as antigas por novas, mas adicionar novas às existentes.

Objetivo

O princípio serve para tornar o comportamento da classe mais diversificado sem interferir nas operações atuais que ela executa. Graças a isso, você evita erros nos fragmentos de código em que esta classe está envolvida.

L - Substituição Liskov (Princípio da Substituição Barbara Liskov)


Se P é um subtipo de T, qualquer objeto do tipo T presente no programa pode ser substituído por objetos do tipo P sem consequências negativas para a funcionalidade do programa.


Nos casos em que a classe filho não é capaz de executar as mesmas ações que a classe pai, existe o risco de erros.

Se você tem uma classe e cria outra em sua base, a classe original se torna a mãe e a nova se torna sua descendente. A classe filho deve executar as mesmas operações que a classe pai. Isso se chama hereditariedade.

É necessário que a classe descendente possa processar as mesmas solicitações que o pai e produzir o mesmo resultado. Ou o resultado pode ser diferente, mas ao mesmo tempo ser do mesmo tipo. Isso é mostrado na figura a seguir: a classe dos pais serve café (de qualquer forma), o que significa que é aceitável que a classe descendente sirva cappuccino (um tipo de café), mas é inaceitável servir água.

Se a classe descendente não atender a esses requisitos, significa que ela é muito diferente da dos pais e viola o princípio.

Objetivo

O princípio serve para garantir a constância: a classe pai e a classe descendente podem ser usadas da mesma maneira sem interromper o programa.

I - Segregação de Interface


O cliente não deve depender de métodos que ele não usa.



Quando uma classe precisa executar ações que não têm nenhum benefício real, isso se traduz em um desperdício de recursos e, se a classe não puder executar essas ações, isso causará bugs.

Uma classe deve realizar apenas as operações necessárias para a implementação de suas funções. Todas as outras ações devem ser excluídas completamente ou movidas se houver uma chance de serem necessárias por outra classe no futuro.

Objetivo

O princípio serve para dividir um único conjunto de ações em vários conjuntos menores - assim, cada classe faz o que é realmente necessário e nada mais.

D - Inversão de Dependência (Princípio de Inversão de Dependência)


. , . . .


Para começar, explicarei os termos usados ​​aqui, em palavras simples.

Módulos (ou classes) do nível superior = classes que executam a operação com a ferramenta
Módulos (ou classes) do nível inferior = ferramentas necessárias para executar operações
Abstrações - representam a interface que conecta as duas classes
Detalhes = características específicas da ferramenta

De acordo com este princípio, a classe não deve estar conectado a uma ferramenta usada para executar uma operação. Em vez disso, ele deve estar conectado a uma interface que ajude a estabelecer uma conexão entre a ferramenta e a classe.

Além disso, o princípio afirma que nem a interface nem a classe são obrigadas a se aprofundar nas especificidades da ferramenta. Pelo contrário, esta ferramenta deve atender aos requisitos da interface.

Objetivo

Este princípio serve para eliminar a dependência de classes de nível superior em relação às classes de nível inferior, introduzindo interfaces.

Resumindo o que foi dito


Examinamos todos os cinco princípios e formulamos uma tarefa para cada um. Tudo isso foi projetado para ajudá-lo a escrever código que pode ser modificado, expandido e testado com um mínimo de problemas.
