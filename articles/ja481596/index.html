Mikrotikログ分析のためのELK 7.5設定の解析

ELKとログと統計の即興ソースで何ができるかを確認することは長い間考えられてきました。 Habrのページでは、ホームミニサーバーを使用して、たとえばELKスタックに基づくログ分析システムでハニーポットを作成する方法の実用的な例を示す予定です。この記事では、ELKスタックを使用してファイアウォールログを分析する最も簡単な例について説明します。将来的には、ZeekによるNetflowトラフィックとpcapダンプを分析するための環境設定について説明したいと思います。



パブリックIPアドレスがあり、ゲートウェイ/ファイアウォールとして多かれ少なかれスマートデバイスがある場合、「おいしい」TCPおよびUDPポートへの着信要求のログを構成することにより、パッシブハニーポットを編成できます。猫の下にMikrotikルーターを構成する例がありますが、別のベンダールーター（または他のセキュリティシステム）を手元に持っている場合は、データ形式とベンダー固有の設定を少し把握するだけで、同じ結果が得られます。

Mikrotikでは、CLIを介してリモートサーバーへのロギングを構成するには、いくつかのコマンドを入力します。

Elasticsearchのセットアップ


Elasticsearchに戻ります。まず、サーバーが稼働中であることを確認する必要があります。Elasticは、CLIのRest APIを通じて最も効率的に操作されます。curlを使用すると、ノードの状態を確認できます（localhostをホストDocker IPアドレスに置き換えます）。

❯❯ curl localhost:9200

次に、Kibanaを開いてみてください。 localhost：5601。 Kibana Webインターフェースで何も構成する必要はありません（テーマをダークに変更しない限り）。インデックスが作成されたかどうかを確認します。これを行うには 、[管理]セクションを開き、 左上の[ Elasticsearchインデックス管理]を選択します 。ここでは、インデックスに登録されているドキュメントの数、ディスクスペースをどれだけ使用しているかを確認でき、インデックスマッピングに関する情報を確認できます。



次に、正しいマッピングテンプレートを登録する必要があります。この情報は、Elasticがどのタイプのデータがどのフィールドに属するかを理解するために必要です。たとえば、IPアドレスに基づいて特別な選択を行うには、 src_ipフィールドに ipデータ型を明示的に指定する必要があります、地理的な場所を特定するには、geoip.locationフィールドを特定の形式で定義し、タイプgeo_pointを登録する必要があります。新しいフィールドの場合、データ型は動的パターン（数値の場合は長い、文字列のキーワード）に基づいて自動的に決定されるため、すべての可能なフィールドを説明する必要はありません。

新しいテンプレートを作成するには、curlを使用するか、Kibanaコンソール（開発ツールセクション）から直接作成します。

❯❯ curl -X POST -H "Content-Type: application/json" -d @elasticsearch/logstash_mikrot-template.json http://192.168.88.130:9200/_template/logstash-mikrot

マッピングを変更したら、インデックスを削除する必要があります。

❯❯ curl -X DELETE http://192.168.88.130:9200/logstash-mikrot-2019.12.16

少なくとも1つのメッセージがインデックスに到着したら、マッピングを確認します。

❯❯ curl http://192.168.88.130:9200/logstash-mikrot-2019.12.16/_mapping

Kibanaでデータをさらに使用するには、[管理]> [Kibanaインデックスパターン]でパターンを作成する必要があり  ます   。 すべてのインデックスが一致するように 、記号*（logstash-mikrot *）を使用してインデックス名を入力 し、日付と時刻のフィールドとしてタイムスタンプフィールドを選択します  。[ カスタムインデックスパターンID]フィールドに   パターンID（たとえば、logstash-mikrot）を入力  できます。将来的には、これによりオブジェクトへのアクセスが簡略化される可能性があります。

インデックスパターン を作成したら  、最も興味深い部分であるデータ分析と視覚化に進むことができます。Kibanaには多くの機能とセクションがありますが、今のところ、私たちにとって重要なのは2つだけです。

発見する


ここでは、ドキュメントをインデックスで表示し、受信した情報をフィルタリング、検索、および表示できます。検索条件に時間枠を設定するタイムラインを忘れないようにすることが重要です。



視覚化


このセクションでは、収集したデータに基づいて視覚化を構築できます。最も簡単なのは、スキャンボットネットのソースを点線またはヒートマップの形式で地理マップに表示することです。グラフを作成したり、選択を行ったりする方法もたくさんあります。



将来的には、データ処理について、おそらく視覚化について、さらに他の興味深いものについて詳しく説明する予定です。勉強の過程で、チュートリアルを補足しようと思います。

トラブルシューティング


インデックスがElasticsearchに表示されない場合は、まずLogstashログを確認する必要があります。

❯❯ docker logs logstash --tail 100 -f

Elasticsearchとの接続がない場合、またはパイプライン構成のエラーが主な理由である場合、Logstashは機能しません。これは、デフォルトでjson dockerに書き込まれるログを注意深く調査すると明らかになります。

ログにエラーがない場合は、Logstashが構成済みのソケットでメッセージをキャッチすることを確認する必要があります。デバッグの目的で、出力  としてstdoutを  使用でき  ます。

stdout { codec => rubydebug }

その後、メッセージがログに直接受信されると、Logstashはデバッグ情報を書き込みます。

Elasticsearchのチェックは非常に簡単です。サーバーのIPアドレスとポート、または特定のエンドポイントAPIでcurlを使用してGETリクエストを作成するだけです。たとえば、人間が読める表でインデックスのステータスを確認します。

❯❯ curl -s 'http://192.168.88.130:9200/_cat/indices?v'



Elasticsearchへの接続がない場合も、Kibanaは起動しません。ログでこれを確認するのは簡単です。

Webインターフェースが開かない場合は、Linuxでファイアウォールが正しく構成または無効化されていることを確認する必要があります（ Centosではiptables  とdockerに問題があり  、トピックのアドバイスに従って解決されました  ）。あまり生産的でない機器では、すべてのコンポーネントが数分間ロードされる可能性があることも考慮する価値があります。メモリが不足すると、サービスがまったくロードされない場合があります。コンテナリソースの使用状況を表示します。

❯❯ docker stats

突然誰かがdocker-compose.ymlファイル内のコンテナーの構成を正しく変更してコンテナー を再起動する方法を知らない場合  、これはdocker-compose.yml  を編集し、同じパラメーターを使用して同じコマンドを再起動することで行われ  ます。

❯❯ docker-compose --compatibility up -d

同時に、変更されたセクションでは、古いオブジェクト（コンテナー、ネットワーク、ボリューム）が消去され、新しいオブジェクトが構成に従って再作成されます。名前付きボリュームが使用され、コンテナーで削除されず、ホストシステムから構成がマウントされるため、サービスのデータは同時に失われません  。Logstashは、構成ファイルを監視し、ファイルが変更されたときにパイプライン構成を再開することもできます。docker restartコマンドを使用

して、サービスを個別に再起動できます   （docker-compose.ymlがあるディレクトリにある必要はありません  ）。

❯❯ docker restart logstash

docker inspectコマンドでdocker オブジェクトの設定を確認できます  。jqで  使用するとより便利です。 



結論


このプロジェクトのセキュリティはテスト（開発）環境であり、ルーターの外部にリリースする予定がないため、報告されていません。より深刻な用途で展開する場合は、ベストプラクティスに従い、HTTPS用の証明書をインストールし、バックアップを作成し、通常の監視（メインシステムの横で開始されない）を行う必要があります。ちなみに、Traefikは私のサーバーの私のdockerで実行されます。これは一部のサービスのリバースプロキシであり、それ自体でTLSを終了し、認証も行います。つまり、構成されたDNSとリバースプロキシのおかげで、未構成のHTTPSとパスワードを使用してインターネットからKibana Webインターフェイスにアクセスできるようになります（私が理解しているように、Kibanaのコミュニティバージョンでは、Webインターフェイスのパスワード保護をサポートしていません）。私は、Dockerを使用してホームネットワークで使用するためにTraefikをセットアップした経験についてさらに説明する予定です。
