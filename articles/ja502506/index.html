GOスケジューラー：今は協調的ではない？

バージョンGO 1.14のリリースノートを読むと、言語のランタイムにかなり興味深い変更があったことに気付くでしょう。「Goroutineは非同期でプリエンプティブルになりました」ので、私は非常に興味を持っていました。GOスケジューラー（スケジューラー）は現在、協調的でないことがわかりましたか？さて、斜めに対応案を読んだところ、好奇心は満たされました。

しかし、しばらくして、私はイノベーションをより詳細に研究することを決定しました。これらの研究の結果を共有したいと思います。



システム要求


以下で説明するものは、GO言語の知識に加えて、追加の知識、つまり次のことを読者に要求します。

  • スケジューラーの原理を理解していること（私は以下で「指先で」説明しようとしますが）
  • ガベージコレクターのしくみを理解する
  • GOアセンブラとは何かを理解する

最後に、これらのトピックをうまくカバーするリンクをいくつか残しておきます。

プランナーについて簡単に


最初に、協調的および非協調的マルチタスクとは何かを思い出させてください。

非協調的（クラウドアウト）なマルチタスキングでは、例としてOSスケジューラーに慣れています。このスケジューラーは、バックグラウンドで動作し、さまざまなヒューリスティックに基づいてスレッドをアンロードします。CPU時間をアンロードする代わりに、他のスレッドが受信を開始します。

協力プランナーは異なる行動を特徴とします-彼はゴルーチンの1つが明確に覚醒して別の場所に自分の場所を与える準備のヒントで目覚めるまで眠ります。次に、計画担当者は、現在のゴルーチンをコンテキストから削除する必要があるかどうかを判断し、必要な場合は誰をその場所に置くかを決定します。これがGOスケジューラの機能です。

さらに、スケジューラーが動作するための基礎を考慮します。

  • P-論理プロセッサー（runtime.GOMAXPROCS関数でその数を変更できます）。各論理プロセッサーで、一度に1つのgoroutineを独立して実行できます。
  • M-OSスレッド。各PはMからのスレッドで実行されます。Pが常にMに等しいとは限らないことに注意してください。たとえば、あるスレッドがsyscallによってブロックされ、別のスレッドがそのPに割り当てられます。また、CGOやその他のニュアンスもあります。
  • G-ゴルチン。さてここでは明らかですが、Gは各Pで実行する必要があり、スケジューラはこれを監視します。

そして、あなたが知る必要がある最後のこと、そしてスケジューラーが実際にいつゴルーチンを呼び出すのですか？それは単純です。通常、呼び出しの命令は、コンパイラーによって関数の本体（プロローグ）の先頭に挿入されます（これについては、後で詳しく説明します）。

そして、実際の問題は何ですか？




記事の冒頭から、GOでスケジューラの作業の原則が変更されたことをすでに理解しているので、これらの変更が行われた理由を考えてみましょう。コードを見てください：

スポイラーの下 
func main() {
	runtime.GOMAXPROCS(1)
	go func() {
		var u int
		for {
			u -= 2
			if u == 1 {
				break
			}
		}
	}()
	<-time.After(time.Millisecond * 5) //    main   ,         

	fmt.Println("go 1.13 has never been here")
}


バージョンGO <1.14でコンパイルすると、「go 1.13は今までここになかった」という行が画面に表示されません。これは、スケジューラが無限ループのあるゴルーチンにプロセッサ時間を与えるとすぐに、Pを完全にキャプチャし、このゴルーチン内で関数呼び出しが発生しないため、スケジューラをウェイクアップしないためです。そして、runtime.Gosched（）への明示的な呼び出しのみがプログラムを終了させます。

これは、goroutineがPをキャプチャし、長い間、他のgoroutineがこのPで実行されないようにする1つの例にすぎません。この動作が問題を引き起こす場合のオプションは、提案を読んで見つけることができます。

解析提案


この問題の解決策は非常に簡単です。 OSスケジューラーと同じようにしましょう！ GOにPからgoroutineを実行させ、そこに別のルーチンを配置するだけです。これには、OSツールを使用します。

これを実装する方法は？ランタイムがgoroutineが機能するフローにシグナルを送信できるようにします。 Mからの各ストリームにこの信号のプロセッサを登録します。プロセッサのタスクは、現在のゴルーチンを置き換えることができるかどうかを判断することです。もしそうなら、その現在の状態（レジスタとスタックの状態）を保存し、別のリソースにリソースを与えます。そうでなければ、現在のゴルーチンを実行し続けます。シグナルを使用したコンセプトはUNIXベースのシステムのソリューションであることは注目に値しますが、たとえば、Windowsの実装は少し異なります。ちなみに、SIGURGは送信する信号として選択されました。

この実装の最も難しい部分は、goroutineを強制的に実行できるかどうかを判断することです。実際のところ、ガベージコレクターの観点から見ると、コードのいくつかの場所はアトミックである必要があります。このような場所を危険な場所と呼びます。コードの実行時にunsafe-pointからgoroutineをスクイーズしてGCが起動すると、unsafe-point'eに取り込まれたgoroutineの状態が置き換えられ、処理を実行できます。安全/安全でないコンセプトを詳しく見てみましょう。

あなたはそこに行きました、GC？




1.12より前のバージョンでは、ランタイムGoschedは、GCのコードのアトミックセクションになることを心配せずにスケジューラを確実に呼び出すことができる場所にセーフポイントを使用していました。すでに述べたように、セーフポイントデータは関数のプロローグにあります（ただし、すべての関数のプロローグではありません）。go-shnアセンブラを解析した場合は、異議を唱えることができます-そこには明らかなスケジューラ呼び出しはありません。はい、ありますが、runtime.morestack呼び出し命令がそこにあります。この関数の内部を見ると、スケジューラー呼び出しが見つかります。スポイラーの下で、GOソースからコメントを非表示にします。または、自分でmorestackのアセンブラーを見つけることができます。

ソースで見つかりました
Synchronous safe-points are implemented by overloading the stack bound check in function prologues. To preempt a goroutine at the next synchronous safe-point, the runtime poisons the goroutine's stack bound to a value that will cause the next stack bound check to fail and enter the stack growth implementation, which will detect that it was actually a preemption and redirect to preemption handling.

明らかに、群集アウトコンセプトに切り替えると、群集アウトシグナルはどこでも私たちのgorutinをキャッチできます。しかし、GOの作成者はセーフポイントを残さないことを決定しましたが、どこにでもセーフポイントを宣言しました！もちろん、実際にはほぼどこにでも問題があります。上記のように、私たちが誰も強制的に出さないいくつかの危険なポイントがあります。簡単な危険なポイントを書いてみましょう。


j := &someStruct{}
p := unsafe.Pointer(j)
// unsafe-point start
u := uintptr(p)
//do some stuff here
p = unsafe.Pointer(u)
// unsafe-point end

問題が何であるかを理解するために、ガベージコレクターの肌を試してみましょう。作業に行くたびに、マーキングを開始するルートノード（スタックおよびレジスター内のポインター）を見つける必要があります。ランタイムでは、メモリ内の64バイトがポインターであるか単なる数値であるかを言うことは不可能であるため、GOコンパイラーから提供されたスタックとレジスターカード（メタ情報を持つキャッシュ）を登録します。これらのマップの情報により、ポインターを見つけることができます。したがって、GOが行番号4を実行したときに、私たちは起こされて職場に送られました。その場所に到着してカードを見ると、空であることがわかりました（GCの観点からのuintptrは数値であり、ポインターではないため、これは事実です）。さて、昨日、jのメモリの割り当てについて聞いたので、今、このメモリに到達できません。クリーンアップする必要があり、メモリを削除した後、スリープ状態になります。次は何ですか？ええと、当局は目覚めました、夜、叫んで、あなたはあなた自身が理解しました。

理論については以上です。実際には、これらすべての信号、危険なポイント、レジスターカードとスタックがどのように機能するかを検討することを提案します。

練習に移りましょう


実行中のシステムコールを確認して比較するために、perfプロファイラーによる記事の最初からダブルラン（go 1.14とgo 1.13を実行）しました。14番目のバージョンで必要なsyscallはすぐに見つかりました。

15.652 ( 0.003 ms): main/29614 tgkill(tgid: 29613 (main), pid: 29613 (main), sig: URG                ) = 0

まあ、明らかにランタイムはgoroutineが回転しているスレッドにSIGURGを送信しました。この知識を出発点として、私はGOのコミットを調べて、このシグナルがどこにどのような理由で送信されているか、およびシグナルハンドラーがインストールされている場所を見つけました。送信から始めましょう。信号の送信機能はランタイム/ os_linux.goにあります。


func signalM(mp *m, sig int) {
	tgkill(getpid(), int(mp.procid), sig)
}

これで、ランタイムコード内の場所を見つけ、そこから信号を送信します。

  1. goroutineが実行状態にある場合は、サスペンド時。サスペンド要求は、ガベージコレクターから送信されます。ここでは、おそらくコードを追加しませんが、runtime / preempt.go（suspendG）ファイルで見つけることができます
  2. goroutineが長すぎるとスケジューラが判断した場合は、runtime / proc.go（retake）
    
if pd.schedwhen+forcePreemptNS <= now {
	signalM(_p_)
}

    forcePreemptNS-10ミリ秒に等しい定数、pd.schedwhen-pdストリームのスケジューラが最後に呼び出された時刻
  3. すべてのストリームと同様に、この信号はパニック、StopTheWorld（GC）、およびいくつかのケース（記事のサイズがすでに境界を超えているため、バイパスする必要があります）中に送信されます

ランタイムがMに信号を送信する方法とタイミングを理解しました。次に、このシグナルのハンドラーを見つけて、ストリームが受信されたときに何が行われるかを確認します。


func doSigPreempt(gp *g, ctxt *sigctxt) {
	if wantAsyncPreempt(gp) && isAsyncSafePoint(gp, ctxt.sigpc(), ctxt.sigsp(), ctxt.siglr()) {
		// Inject a call to asyncPreempt.
		ctxt.pushCall(funcPC(asyncPreempt))
	}
}

この関数から、「ロックイン」するには2つのチェックを行う必要があることは明らかです。

  1. wantAsyncPreempt-Gが強制的に排除されるかどうかを確認します。ここでは、たとえば、現在のgoroutineステータスの有効性が確認されます。
  2. isAsyncSafePoint-現在混み合っているかどうかを確認します。ここでのチェックで最も興味深いのは、Gが安全な点か安全でない点かです。さらに、Gが実行されているスレッドがGをプリエンプトする準備ができていることを確認する必要があります。

両方のチェックに合格すると、状態Gを保存してスケジューラーを呼び出す実行可能コードから命令が呼び出されます。

そして、安全ではない


私は新しい例を分析することを提案します、それは危険な点を持つ別のケースを示します：

別の無限のプログラム 

//go:nosplit
func infiniteLoop() {
	var u int
	for {
		u -= 2
		if u == 1 {
			break
		}
	}
}

func main() {
	runtime.GOMAXPROCS(1)
	go infiniteLoop()
	<-time.After(time.Millisecond * 5)

	fmt.Println("go 1.13 and 1.14 has never been here")
}


ご想像のとおり、GO 1.14には「go 1.13と1.14はここにいなかった」という表記はありません。これは、infiniteLoop（go：nosplit）関数の中断を明示的に禁止しているためです。このような禁止は、機能全体であるunsafe-pointを使用するだけで実装されます。コンパイラーがinfiniteLoop関数用に生成したものを見てみましょう。
