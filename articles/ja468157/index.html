Ocmの一般理論に関するEcmaScriptの視点

こんにちは、ハブル！

今日まで、私は英語を話す作家による興味深い記事の翻訳にのみ携わってきました。そして今、自分で何かを書く時が来ました。最初の記事では、「中間」に成長したいジュニア開発者に役立つと思うトピックを選びました。JavaScriptと従来のプログラミング言語（C ++、C＃、Java）の類似性/違いをOOPの観点から分析します。それでは始めましょう！

一般的なパラダイム規定


ウィキペディア でJavaScriptの定義を見ると、次の概念がわかります。
JavaScript（/ ˈdʒɑːvɑːˌskrɪpt /; abbr。JS /ˈdʒeɪ.ɛs./）は、マルチパラダイムプログラミング言語です。オブジェクト指向、命令的、機能的なスタイルをサポートします。ECMAScript言語（ECMA-262標準）の実装です。

この定義からわかるように、JavaScriptはそれ自体では存在しませんが、一部のEcmaScript仕様の実装です。それに加えて、他の言語がこの仕様を実装しています。

EcmaScript（以下ES）には、次のパラダイムがあります。

  • 構造的
  • OOP
  • 機能的
  • 命令的
  • アスペクト指向（まれなケース）

ESのOOPは、プロトタイプ組織で実装されます。 「JSのOOPは従来の言語のOOPとどう違うのですか？」原則として、「クラスは古典的な言語であり、プロトタイプはJSである」と非常にあいまいになります。

実際には、状況はもう少し複雑です。動作の点では、動的クラス組織とプロトタイプ組織の違いはわずかです（確かに存在しますが、それほどグローバルではありません）。

PythonまたはRubyを見てください。これらの言語では、OOPは動的なクラス編成に基づいています。これらの言語の両方で、プログラムの進行に応じてオブジェクトのクラスを動的に変更できます。クラス内の変更も、生成するエンティティに動的に影響します。JSと同じですが、JSでは、OOPはプロトタイプに基づいています。

静的クラス組織とプロトタイプ組織の言語間の大きな違い。それ自体の違いは、「クラスがあります。ここでプロトタイプはそれほど重要ではありません。

静的クラス編成は何に基づいていますか？


このタイプのOOPの基礎は、「クラス」と「エッセンス」の概念です。クラスは、生成できるエンティティの特定の形式化された一般化された特性のセットです。それら。これは、それによって生成されるすべてのオブジェクトの特定の一般的な計画です。

特性には2つのタイプがあります。プロパティ（エンティティの説明）およびメソッド（エンティティのアクティビティ、それらの動作）。クラスによって生成された

エンティティは、このクラスのコピーですが、初期化されたプロパティがあります。ご覧のように、クラスはエンティティの説明（厳密に定義されたプロパティのセットを提供）とその動作（厳密に定義されたメソッドのリストを提供）を厳密に規制します。

JAVAの小さな例を次に示します。

class Person{
     
    String name;        // 
    int age;            // 
    void displayInfo(){
        System.out.printf("Name: %s \tAge: %d\n", name, age);
    }
}

次に、クラスのインスタンスを作成します。 

public class Program{
      
    public static void main(String[] args) {
         
        Person tom;
    }
}

tom エンティティにはPersonクラスのすべての特性があり、そのクラスのすべてのメソッドもあります。

OOPパラダイムは、コードを再利用するための非常に幅広い機能パレットを提供します。これらの機能の1つは継承です。

あるクラスは別のクラスを拡張して、汎化と特殊化の関係を作成できます。この場合、一般クラス（スーパークラス）のプロパティは、作成時に子孫クラスのエッセンスにコピーされ、メソッドは参照によって（継承の階層チェーンによって）利用できます。静的クラス型付けの場合、このチェーンは静的です、動的の場合は、プログラムの実行中に変更される可能性があります。これが最も重要な違いです。この瞬間を覚えておくことをお勧めします。さらに、プロトタイプ編成にたどり着くと、「クラスはある、プロトタイプはある」という答えの本質が明らかになります。

このアプローチの欠点は何ですか？

次のことは明らかだと思います。

  • 本質的に、決して役に立たない特性があるかもしれません。
  • クラスは、生成されたエンティティに提供するプロパティとメソッドを動的に変更、追加、削除することはできません。彼の署名を変更することはできません。
  • 本質的に、親のクラス（または親の階層チェーン）に存在しないプロパティまたはメソッドは存在できません
  • メモリ使用量は、継承階層内のリンクの数に比例します（プロパティをコピーするため）。

プロトタイプ組織は何に基づいていますか？


プロトタイプ編成の重要な概念は、動的な可変オブジェクト（dmo）です。 DMOにはクラスは必要ありません。彼自身はすべてのプロパティとメソッドを保存できます。

プロパティのDMOを設定するとき、それはその中にこのプロパティの存在をチェックします。プロパティがある場合は単に割り当てられ、ない場合はプロパティが追加され、渡された値で初期化されます。 DMOは、プログラム中に何度でも署名を変更できます。

次に例を示します。

//       
const Person = {
  name: null,
  age: null,
  sayHi() {
    return `Hi! My name is ${this.name}. I'm ${this.age} years old.`
 }
}

const Tom = {
  //-      
}

Tom.__proto__ = Person;

主題の誰もがクラス構文がES6で登場したことを知っていると思いますが、これは構文糖にすぎません。ボンネットの下のプロトタイプ。上記のコードは、優れたコーディング手法と見なすべきではありません。これは単なる実例であり、読者を混乱させたり、理論的概念の違いを強調したりしないように、この形式で表示されます（現在、すべての一般の人々がES6クラスを使用しています）。

Tomオブジェクトをコンソールに出力すると、オブジェクト自体には_proto_リンクのみがあり、デフォルトでは常にそこに存在していることがわかります。参照は、TomオブジェクトのプロトタイプであるPersonオブジェクトを指します。

プロトタイプは、他のオブジェクトのプロトタイプとして機能するオブジェクト、または必要に応じて別のオブジェクトがプロパティとメソッドを描画できるオブジェクトです。

オブジェクトのプロトタイプは任意のオブジェクトにすることができ、さらに、オブジェクトはプログラム中にプロトタイプを再割り当てすることができます。

トムに戻りましょう。

Tom.name = 'Tom'; //   
Tom.surname = 'Williams'; //   
Tom.age = 28;//   

Tom.sayHi();//  sayHi,      ,    ,      

const tomSon = {
  name: 'John',
  age: 5,
  sayHi() {
    return this.__proto__.sayHi.call(this) + `My father is ${this.__proto__.name} ${this.surname}`;
  }
}
//,    
tomSon.__proto__ = Tom;
tomSon.sayHi();//  "Hi! My name is John. I'm 5 years old.My father is Tom Williams"

name、ageプロパティ、sayHiメソッドはtomSonオブジェクトのプロパティであることに注意してください。同時に、TomオブジェクトにあるかのようにtomSon sayHiでsayHiプロトタイプメソッドを明示的に呼び出しますが、実際には存在せず、Personプロトタイプから暗黙的に返されます。また、プロトタイププロパティ名を明示的に操作し、暗黙的に取得します。 tomSonオブジェクトの独自のプロパティとして呼び出す姓プロパティですが、実際にはありません。姓のプロパティは、プロトタイプから__proto__リンクを介して暗黙的に取得されます。

プログラム中に、既に作成されたオブジェクトのプロトタイプを変更したことに注意してください。これは、プロトタイプ組織と動的クラス組織の類似点です。それが「古典的な言語とJavaScriptの違いは何ですか？」という質問に対する「クラスはあり、プロトタイプはある」という答えです。完全に正しくなく、OOPの理論とクラスやプロトタイプへの実装に関する誤解を示しています。

プロトタイプ編成では、静的クラス1とは異なり、このプロトタイプからプロパティを継承するエンティティを作成した後にプロトタイプを変更する機会があり、これらの変更は既に作成されているエンティティに影響します。

Ben.hobbies = ['chess', 'badminton'];
// tomSon   ,             ,     
tomSon.sayAboutFathersHobies = function () {
  const reducer = (accumulator, current) => {`${accumulator} and ${current}`}
  return `My Father play ${this.hobbies.reduce(reducer)}`
}

tomSon.sayAboutFathersHobies(); //  'My Father play chess and badminton'

これは、プロトタイプ組織委任モデルまたはプロトタイプ継承と呼ばれます。

エンティティが特定の動作を実装する能力はどのように決定されますか？


静的クラス編成では、この操作には、必要な動作を実装する特定のクラスのメンバーシップについてエンティティーをチェックすることが含まれます。プロトタイプ編成には、アヒルのタイピングという概念があります。ダックタイピングの場合、特定の動作を実装する機能についてエンティティをチェックすることは、特定の時点でこの動作を実装する機能についてエンティティを直接テストすることを意味します。プログラムの異なる部分では、チェックの結果が正反対になる場合があります。
