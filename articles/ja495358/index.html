前書き


Kubernetesは非常に動的なシステムです。 K8sクラスターのインフラストラクチャを使用して、ポッドが常に存在すると想定します-はい、ポッド：ノード全体があります。 -削除できます。これを検証するために、主にランダムなKubernetesノードを強制終了してクライアントアプリケーションがポッドを再作成する準備ができているかどうかを確認することにより、さまざまなカオスエンジニアリングアプローチを使用します。

このような動的なKubernetesの条件下でアプリケーションが正しく機能するためには、PodDisruptionBudgetsの作成、複数のアプリケーションレプリカの同時デプロイ、podAffinity、nodeAffinityの正しい構成など、最低限のルールに従う必要があります。

しかし、そのような明白なポリシーにもかかわらず、私たちは常に彼らのアプリケーションに影響を与えることができません。現実には、クライアントが何らかの理由で単一のレプリカでアプリケーションを実行するときに問題に遭遇する必要がありました（一度ではありません！）。これは数か月間このように機能し、ほとんどデプロイされません。そのまれなロールアウトは、私たちの参加なしで、そしてもちろん、werf なしで実行されます（このユーティリティはそれ自体ですべてを実行できます）。同時に、レジストリはすべての古い画像を自動的に削除するように構成されており、一度何らかの理由でポッドを再作成する必要がある場合、修復不可能なことが起こります。

最近、K8sノードの通常の並べ替え操作によって1時間のダウンタイムが発生した後、アプリケーションを構築できる人物を探していたため、k8s-image-availability-exporterを作成することにしました。ユーティリティのアイデアは自動化にあり、組織のポリシーやその他の「ランダムな」要素の遵守に関係なく、そのような状況の結果を防ぐのに役立ちます。

実装


一般的な実装アルゴリズムは次のとおりです。

  • 5つの情報提供者を起動します。したがって、次のすべてのクラスターオブジェクトの状態のコピーがメモリに保持されます：Deployment、StatefulSet、DaemonSet、CronJobs、およびSecrets。
  • すべての画像をPodTemplatesにグループ化し、優先キューに配置します。
  • 15秒ごとに（オプションで設定--check-period）、最も検証されていない画像の一部をキューから長時間抽出し、レジストリ内の存在を確認します。イメージの数は動的に変化するため、10分間でクラスターのすべての可能なイメージをチェックできます。

    • PodSpec存在する場合imagePullSecretsは、必要な秘密を取得します。
    • シークレットのimagePullSecrets有無にかかわらず（ある場合）、コンテナーレジストリに移動し、イメージが存在するかどうかを確認します。

この作業の結果としてのメトリックの次のタイプは、（下にエクスポートされる暗黙deployment、statefulset、daemonsetおよびcronjob）：

  • k8s_image_availability_exporter__bad_image_format-ゼロ以外の値は、無効なフィールド形式を示しますimage。

  • k8s_image_availability_exporter__authentication_failure — ( imagePullSecrets);

コードはGoで記述され、無料のApache 2.0ライセンスで配布されており、GitHubで入手できます。それを改善するための提案やユーティリティの使用に関するコメントがあれば、私たちは幸せになります！

このユーティリティはすでに数十のKubernetesクラスタに実装されており、その使用結果に非常に満足しています。ただし、正式には現在の開発状況はアルファ版ですので、自己責任でご利用ください。

使い始めるには？


3つのステップで十分です。

  1. git clone https://github.com/flant/k8s-image-availability-exporter.git
  2. kubectl apply -f deploy/
  3. Prometheusとの統合をセットアップします。スクレイピングルールとアラートルール。

詳細（Prometheusの構成済み構成の例を含む）については、READMEを参照してください。

概要


これで、k8s-image-availability-exporterにより、実行中のKubernetesコントローラーのいずれかがレジストリー内のコンテナーのイメージを失ったというアラートが表示されます。これにより、問題が顕在化する前に問題を解決できます。

別の解決策は、レジストリをまったくクリーンアップしないか、この問題がすでに体系的に解決されているすべての場所でユーティリティを使用することです（werfで行われるように）。
