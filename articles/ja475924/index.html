Github Security Labに関するお知らせ：すべてのコードを一緒に保護する



私たちは皆、オープンソースソフトウェアのセキュリティを確保するための集団的責任を負っています-一人でそれを行うことはできません。本日Github Universeで、Github Security Labを発表しました。業界全体のセキュリティ研究者、エスコート、企業が集まり、オープンソースのセキュリティは誰にとっても重要であるという私たちの信念を共有する場所です。

この目標に貢献する最初のパートナーがいることを嬉しく思います。同時に、オープンソースエコシステムの保護に役立つツール、リソース、受賞歴、数千時間のセキュリティ研究を提供しています。

本日の発表の一部として、GitHub Security LabがCodeQLを作成しましたオープンソースの脆弱性を発見する可能性がある人なら誰でも自由にアクセスできます。 CodeQLは、世界中の多くの研究グループがセマンティックコード分析を行うために使用するツールであり、私たちは自分たち自身を使用して、人気のあるオープンソースプロジェクトで100を超える登録済みCVE（Common Vulnerabilities and Exposures）を見つけました。

また、GitHub上で作成された、パブリックにアクセス可能なレコメンデーションデータベースであるGitHub Advisory Databaseに加えて、GitHub依存関係グラフによって追跡されるパッケージにマップされた追加のデータもリリースします。

セキュリティに対するGitHubのアプローチは、オープンソースプロジェクトのセキュリティライフサイクル全体に及びます。GitHub Security Labはオープンソースプロジェクトの脆弱性を特定して報告するのに役立ちます。一方、メンテナーと開発者はGitHubを使用してパッチを作成し、依存するプロジェクトの開示と更新を脆弱性の解決に調整します。



Githubセキュリティラボ


GitHubセキュリティラボの使命は、セキュリティ研究者のグローバルコミュニティを刺激し、世界中のコードを保護できるようにすることです。私たちのチームは、重要なオープンソースプロジェクトの脆弱性を見つけて報告するために継続的なリソースを投入する例を設定します。 Komadnaはすでに、脆弱性検出用の100を超えるCVEをリリースしています。

世界のオープンソースプロジェクトを保護することは簡単なことではありません。まず、スケール：JavaScriptの1つのエコシステムには、100万を超えるオープンソースパッケージが含まれています。さらに、セキュリティの専門家が不足しており、1人の専門家に対して約500人の開発者がいます。最後に、調整があります。世界中のセキュリティ専門家が何千もの企業で働いています。 GitHub Security labとCodeQLがこれを支援します。

この作業では、オープンソースプロジェクトの脆弱性を見つけて報告するために時間と経験を寄付する企業が参加します。それぞれが彼自身の方法で貢献することを誓いました、そして、私たちは他の人が将来私たちに加わることを望みます。

機能を拡張するために、オープンソースプロジェクトで使用できる協調的なCodeQLコード分析エンジンも無料にしています。 CodeQLを使用すると、データのようにコードをクエリできます。脆弱性の原因となったコーディングエラーを知っている場合は、このコードのすべてのバリアントを見つけるクエリを記述して、脆弱性のクラス全体を永久に破壊できます。CodeQLの使用を開始する方法をご覧ください。

あなたがセキュリティ研究者であるか、セキュリティチームで働いている場合、私たちはあなたの助けを必要としています。世界のオープンソースプロジェクトを保護するには、コミュニティ全体の取り組みが必要です。 GitHub Security Labはイベントを主催し、ベストプラクティスを共有して全員が参加できるようにします。GHSecurityLabアカウントを追跡する 詳しくはTwitterをご覧ください。

オープンソースでのセキュリティワークフローの改善


セキュリティの世界の研究者がますます脆弱性を発見するにつれて、付随するエンドユーザーはそれらを修正するためのより洗練されたツールを必要としています。

今日、新しい脆弱性を修正するプロセスはしばしば一時的なものです。オープンソースプロジェクトの新しい脆弱性の40％は、宣言されたときにCVEに識別子がありません。つまり、それらはどのパブリックデータベースにも含まれていません。重大な脆弱性の70％は、開発者に通知されてから30日間は未解決のままです。

修正中です。メンテナーと開発者はGitHubで直接連携して、メンテナーの準備が整ったときにのみ新しい脆弱性が公開されるようになり、開発者は迅速かつ簡単に修正バージョンにアップグレードできます。

Githubセキュリティ勧告


セキュリティのヒントのおかげで、メンテナーはプライベートスペースの修正に関してセキュリティリサーチャーと協力し、GitHubから直接CVEを申請し、構造化された脆弱性情報を提供できます。次に、セキュリティに関する推奨事項を公開する準備ができると、GitHubは影響を受けるプロジェクトに関するアラートを送信します。



自動セキュリティ更新


脆弱な依存関係の通知を受け取ることは便利ですが、修正を伴うプルプルリクエストを取得することはさらに優れています。開発者が新しい脆弱性にすばやく対応できるように、GitHubは自動化されたセキュリティ更新プログラム-プルリクエストを作成して、脆弱な依存関係をパッチ適用済みバージョンに更新します。



システムの自動セキュリティアップデートはGitHub Satellite 2019のベータ版でリリースされ、現在、セキュリティアラートが有効になっているアクティブなリポジトリごとにほとんど利用可能であり、デプロイされています。

トークンスキャン


最も一般的な間違いの1つは、プロジェクト内のトークンまたは資格情報のハードコードです。GitHubにコミットを送信した後、またはプロジェクトをパブリックに切り替えてから数秒以内に、20の異なるクラウドプロバイダーからのフォーマットをスキャンします。一致するものが見つかると、プロバイダーに通知し、プロバイダーは通常、トークンをキャンセルして影響を受けるユーザーに通知することでアクションを実行します。そして今日、私たちは4つの新しいパートナーを発表しました：GoCardless、HashiCorp、Postman、Tencent。

Githubアドバイザリーデータベース


メンテナーがGitHubのセキュリティヒントで作成するすべての変更と追加のデータを作成し、GitHub依存関係グラフによって監視されるパッケージにマッピングしました。これらは無料で利用できます。ブラウザで新しいGitHubアドバイザリデータベースを探索し、コメントにCVE識別子を含む投稿への直接リンクを作成するか、セキュリティアドバイザリAPIエンドポイントを使用してプログラムでデータにアクセスします。
