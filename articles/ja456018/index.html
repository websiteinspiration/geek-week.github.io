Webhookを使用してGitHubから本番サーバーへのデプロイを実装する方法



私は長い間、GitHubにリポジトリーを作成する習慣がありました。これは、すべてをGoogleドライブに保存するよりもはるかに効果的です。しかし、ここで問題がすぐに発生します。稼働中のサーバーにデプロイする方法は？

ほとんどの検索で、Jenkinsやその他の継続的デプロイメントツールが使用されました。しかし、私は別の解決策を見つけたかったのです。そこで私は無料のWebhookサービスに行きました。

Skillboxの推奨事項：実践コース「モバイルデベロッパーPRO」。

私たちはあなたを思い出させる： 「Habr」のすべての読者のために-万ルーブルの割引を「Habr」プロモーションコードを使用して、もちろんすべてのSkillbox登録時。

Webhookの技術的な基礎は、Ubuntu 16.04を本番サーバーとして使用した新しいDigital Oceanドロップレットです。計画の実装に必要な手順の数を減らすために、すべてのアクションはrootユーザーによって実行されます。

githubから始めましょう


リポジトリがあり、それを使用したい場合は、このステップをスキップできます。SSHURIを使用するだけです。そうでない場合は、作成してSSH URIも使用します。



GoとWebhook（デジタルオーシャンドロップレット）をインストールします。

始める前に、すぐにアップデートして、新しいUbuntu 16.04インストーラーでアップグレードする価値があります。

sudo apt update -y && sudo apt upgrade -y

WebHookをインストールするには、Goプログラミング言語をインストールする必要があります。この記事の執筆時点では、バージョン1.11.4が関連していたため、別のバージョンを使用している場合は変更する必要があります。

wget https://dl.google.com/go/go1.11.4.linux-amd64.tar.gz
sudo tar -C /usr/local -xzf go1.11.4.linux-amd64.tar.gz
export PATH=$PATH:/usr/local/go/bin

Webhookの最新バージョンをダウンロードします。

go get github.com/adnanh/webhook

ダウンロード中はプログレスバーは表示されないので、待つ必要があります。プロセスが完了したら、〜/ go / bin / webhookを実行します。

Webhookはインストールされていますが、すべてが正常に機能するように、ディレクトリとファイル構造を作成する必要があります。

mkdir ~/webhooks
mkdir ~/webhooks/deployment-tutorial
touch ~/webhooks/hooks.json
touch ~/webhooks/deployment-tutorial/deploy.sh
chmod +x ~/webhooks/deployment-tutorial/deploy.sh

hooks.jsonファイルは構成とルーティングを担当し、deploy.shはGitHubからのアップグレードに必要なコマンドを実行するためのツールとして機能します。

まず、hooks.jsonをテキストエディターで開いて構成します。このファイルには、Webhookの起動後に作成されるエンドポイントの構成が含まれており、それぞれが一意のエンドポイントであるオブジェクトの配列です。

[{
    "id": "deployment-tutorial",
    "execute-command": "/root/webhooks/deployment-tutorial/deploy.sh",
    "command-working-directory": "/root/deployed-site/",
    "response-message": "Executing deploy script...",
    "trigger-rule": {
        "match": {
            "type": "payload-hash-sha1",
            "secret": "The Returners",
            "parameter": {
                "source": "header",
                "name": "X-Hub-Signature"
            }
        }
    }
}]

Idは、エンドポイントURLに使用される一意の名前です。
execute- command-エンドポイントがアクティブ化されたときに実行されるスクリプト。
command-working-directory -execute-commandの実行中に使用されるディレクトリ。
trigger-rule-情報セキュリティの目的で使用されるオプション。これはエンドポイントの秘密のフレーズです。

すべてのアクションはルートから実行するため、開始アドレスは/ルートです。通常のユーザーとしてログインする場合、/ home / usernameを登録する必要があります。usernameは論理的なもので、このユーザーの名前です。 〜/を使用する必要はありません。パスは絶対パスでなければなりません。そうでない場合、エラーが発生します。

まだ存在しない作業フォルダを設定したことに気づいたかもしれません。作成する前に、deploy.shで終了する必要があります。

スクリプトは常に「shebang」で始まる必要があります。ファイルを開く前に、どのbashを実行する必要があります。それでは、次のコマンドをスクリプトで実行します。

#!/bin/bash
 
git fetch --all
git checkout --force "origin/master"

GoとWebhookの両方がインストールされたので、hooks.jsonで構成を構成し、デプロイメントを担当するスクリプトを作成する必要があります。彼は、GitHubリポジトリのメインブランチと連携して、宛先ディレクトリを変更します。

最後に、Webhookをアクティブモードにして、000,000,000,000をDopletの有効なIPアドレスに置き換えます。

/root/go/bin/webhook -hooks /root/webhooks/hooks.json -ip "000.000.000.000" -verbose

実行時に、最後に{id}が付いたURLの出力に気付く場合があります。これは、hooks.json：deployment-tutorialファイルにすでに作成されているオブジェクトのIDになります。

Git（Digital Ocean Droplet）のセットアップ


サーバーのセットアップはまだ完了していません。それを完了するには、最初のウィンドウがWebhookを実行している間に、新しいターミナルウィンドウを開いてサーバーで再度認証する必要があります。最初に、リポジトリURIを設定しましたが、これを使用する必要があります。

これを行うには、hooks.jsonで指定されている作業ディレクトリに移動し、次のように記述します。

git init
git remote add origin git@github.com:jhsu98/deployment-tutorial.git

この場合、URIを、私が示したものではなく、独自のものに置き換える必要があります。

最後のステップは、GitHubに接続するためにSSHキーを生成することです。ターミナルウィンドウでssh-keygenと入力し、キーが生成されるまでEnterキーを押して確定します。次に、cat〜/ .ssh / id_rsa.pubと入力して、公開鍵を表示する必要があります。あなたはこのようなものを手に入れます：

展開キーとWebhookを構成する（GitHub）


ブラウザで、リポジトリの設定に移動します。最初に、展開キーを追加する必要があります。



キーを追加するときは、その名前を指定して、上記のキー出力に挿入する必要があります。GitHubでは公開鍵のみを操作できるため、必要に応じて、いくつかの鍵を指定する必要があります。すべてが完了したら、Webhookセクションが番です。



このセクションの3つのフィールドには特別な注意が必要です。これらは、ペイロードURL、コンテンツタイプ、およびシークレットです。ペイロードURLはWebhookがリッスンするエンドポイント、コンテンツタイプはデータ形式、シークレットはhooks.jsonファイルのカスタム行です。私たちの場合、それは「The Returners」です。



Webhookを作成すると、設定に対応するマークが表示され、サーバーが使用可能であることを示します。動作しているWebhookでターミナルを確認すると、リポジトリがすでに動作していることがわかります。

ワークフローを確認するには、リポジトリをローカルマシンにコピーし（まだ行っていない場合）、メインブランチに変更を加えます。その後、サーバーに表示されます。できました。

これらすべてをすでに数回行っている開発者は、設定に約10分を費やしています。しかし、これは何時間もの試行錯誤が費やされた後です。次の4つの重要なカスタマイズ機能を覚えておいてください。

  • hooks.jsonには絶対パスを使用してください。
  • deploy.shを実行可能にしないでください。
  • GitHubと本番サーバーの最初の「連絡」中にSSH接続を確認します。
