プライベートSNAFUはDBAに行きます


知らない人にとって、SNAFUは戦争中にアメリカ人によって作成された軍事愛国的な漫画のキャラクターです。このガウジングは、自然な愚かさのために、常に破滅的な状況に陥り、原則として、シリーズの最後に死にます。確かに、次のシリーズで彼は再び生きています-この意味で、彼はサウスパークからケニーの遠い先祖と見なすことができます。

SQLサーバー開発者のポジションに人を募集するとき、私はよく彼らが質問にどのように答えるかに応じました。私は、同僚が提案した小さな1行のタスクで救われていなかった場合は、YESと伝える準備ができていました。このタスクが1つのSQL文字列でどれだけ生成できるかは驚くべきことです。そして今、候補者はすでに熱狂的に熊手を歩いています。あなたが見るように、熊手はたくさんあります。もちろん、可能なレーキをすべて収集した人は一人もいませんでした。しかし、それらすべてを表示するには、SNAFUが必要でした。

したがって、タスク：

SELECT * FROM Events where DT='2020/02/03'

  • コードの作成者は何をしたいですか？
  • なぜこれが期待どおりに機能しないのですか？
  • それを修正するには？

SNAFUとの会話はさらに進みます。

SNAFU：もちろん、クエリは1日分のデータを選択します。
私：それで何が悪いのですか？
SNAFU：ええと、アスタリスクがあります。フィールドを示す方が良いでしょう...
私：そして、テーブルに何が保存されていると思いますか？そして、DTはどのタイプのフィールドですか？
SNAFU：イベント...イベントの時間...つまり、datetime
Me：ええと？
SNAFU：（審議後）ああ、時間があれば、ノートは真夜中
に読まれるI：すばらしい

最初の2つのポイントを克服し、3つ目のポイントに移動しました。

私：それを修正するには？
SNAFU： where convert（varchar、DT、102）= '2020/02/03'（Rake＃1 works）
I：しかし、DTインデックスが存在する場合にSQLがDTインデックスを使用できなくなる可能性があることは気になりませんか？
SNAFU： '2020/02/03'と '2020/02/04'の間のDT（Rake＃2は機能しました）
私：次の番号（真夜中）のデータもここに含まれますか？
SNAFU： '2020/02/03'から '2020/02/03 23:59:59'までのDT（Rake＃3は機能しました）
I：お金があるときに秒がある銀行を扱いますか、あなたが貢献したものは失われますか？
SNAFU： '2020/02/03'と '2020/02/03 23：59：59.999'の間のDT（レーキ＃4は機能しました）
I： 23：59：59.999が真夜中まで再び丸められることを知っていますか？
SNAFU：正確には、粒度は13〜16ミリ秒です。次に： '2020/02/03'から '2020/02/03 23：59：59.986'までのDT（Rake＃5は機能しました）
私：つまり、DTがsmalldatetimeまたはdatetime2になると、すべてが機能しなくなりますか？



以下の瞬間への移行がなぜそれほど難しいのか、私にはわかりません。実際には、候補者の半分以上が愚かに絶望的です。時々プロンプトを表示する必要があります。しかし、私たちのSNAFUは解決策を拷問しました：

SNAFU：ここでDT> = '2020/02/03'およびDT <'2020/02/04'
私：最後に。

SNAFUは額から汗を拭き取ります。コントロールショットの時間です。
私：そして、他に何が本当にここで悪いのですか？
SNAFUが揺れています。彼の手が震えています。視線がさまよっている。
私： 2月3日ですか、3月2日ですか？

そのようなタスクは、ばかげた「テストタスク」よりもはるかに速く理解を与えるように思えます。この場合、SNAFUは良い人かもしれませんが、彼は時系列データをほとんど処理しなかったので、彼を金融セクターに入れないようにした方がよいでしょう。
