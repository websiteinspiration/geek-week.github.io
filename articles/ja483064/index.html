すべてのカノンで最小の別名小さなブログのVue



みなさん、こんにちは！この記事では、Vuexとルーターを含むVueのすべての魅力を使用して、シンプルなVueブログのフロントの開発を検討します。また、アプリケーションの構造について話し、コンテナーとルーターを操作します。

最初に、アプリケーションのフロント（この場合はブログ）を作成する段階を定義します。

  1. 企画
  2. アプリケーションスケルトン
  3. モデルの作成
  4. ビジネスロジックの実装
  5. ページとルートを追加する
  6. コンポーネントの追加
  7. 編集

1.計画


まず、SPAの内容を書き留めます。これはユーザーが直接操作するものであるため、ページから直接開始する必要があります（TDDテストのように切り上げるのが非常に大雑把な場合は、最初にアプリケーションが何を実行する必要があるか、ユーザーがどのように操作するかを説明し、その後、実装に取り​​組んでいます）。

だからどのページになるでしょう：

  • ホーム-人気のカテゴリ、最近のコメント、記事が掲載されます
  • カテゴリの参照-特定のカテゴリの記事のリスト
  • ニュースを表示-ニュースのコンテンツとコメントのリスト、およびカテゴリの他の記事を直接

ページのプロトタイプを配置すると、次のようになります。

完全な美しさ


明確なものはありませんが、それはプロトタイプです:-)プロトタイピング用のソフトウェアを使用して描かれなかった理由をすぐに説明します。紙の上でははるかに高速で簡単であり、時には手を動かす必要があるためです。とても悲しい場所に署名する必要があります。

取得したページに基づいて、コンポーネントをリストします。

  • 記事一覧
  • カテゴリー一覧
  • コメント一覧
  • コメントフォーム

コンポーネント自体の実装時にすでにコンポーネントの最適化を扱いますが、この段階ではこれは必要ありません。

最後に、すべてのユーザーインタラクションポイントを説明したので、アプリケーションの本質を説明します。

  • 記事（タイトル、内容、コメントのリスト）
  • カテゴリ（見出し、ニュースリスト）
  • コメント（内容）

ビジネスロジック（BL）の本質が記述されており、ベーステーブルではないことに注意してください。パーツの前面と背面の大部分（データレイヤーのみを除く）を開発および計画する場合、「テーブル」ではなくBLエンティティを操作する必要があります。さらに、エンティティはアプリケーション自体で使用されているもののみを持つ必要があります。未来に触れることは良いことですが、この未来が来ることはめったになく、使用された機能のみが配置され、アプリケーション構造が拡張可能な場合、将来の機能の追加に問題はなく、現時点で余分なものはありません。

3.モデルの作成


アプリケーションには複雑なロジックはありませんが、それでも、いずれにしてもモデルを作成する必要があります。
これにはいくつかの理由があります。

  • オブジェクトの指定 -任意のオブジェクト。プロパティとメソッドを含めることができます。クラスを使用すると、特定のオブジェクトがどのプロパティとメソッドを持っているかがわかります
  • コンポーネントパラメータの型指定 -前の手順の続き：Vueコンポーネントは入力プロパティの型を制御します
  • 利便性 -クラスの観点から、サブジェクトエリアのエンティティを操作します。これにより、コードがより明確になります。クラスは、get / set / staticなどの追加機能も提供します

すべてのモデルは適切なディレクトリに配置されます。記事のクラスは次のようになります。

src /モデル/ Article.js
export default class Article
{
  constructor(id, title, content) {
    this.id = id;
    this.title = title;
    this.content = content;
    this.comments = [];
  }

  addComment(item) {
    this.comments.push(item);
  }

  static createFrom(data) {
    const {id, title, content} = data;
    return new this(id, title, content);
  }
}

export default {
  getArticles() {
    const comments = this.getComments();
    const items = [

      },
      {
        id: 2, title: ' 2', content: '  2',
      },
      {
        id: 3, title: ' 3', content: '  3',
      },
      {
        id: 4, title: ' 4', content: '  4',
      },
      {
        id: 5, title: ' 5', content: '  5',
      },
      {
        id: 6, title: ' 6', content: '  6',
      },
    ];
    return items.map((item) => {
      const article = Article.createFrom(item);
      article.comments = comments.filter((comment) => comment.article_id == article.id);

      return article;

  getComments() {

        id: 1, article_id: 1, content: '   1',

      },
      {
        id: 3, title: '', articles: [],
      },
    ];
    return items.map((item) => Category.createFrom(item))
  },
  addComment(comment) {
    if (comment) {
      //    
    }
  },
};


vuexの使用方法：

  • 純粋さ -コンポーネントは、「知りすぎ」ている神のオブジェクトにはならない
  • 一貫性 -同じデータを使用する複数のコンポーネントがある場合、データを変更するときにvuexコンテナーを使用すると、更新を開始したコンポーネントだけでなく、アプリケーション全体でそれら（データ）が更新されます。
  • 利便性 -各コンポーネントでパラメーターの型やAPI呼び出しを記述するよりも、ストアにアクセスする方がはるかに簡単で便利です。

  name: 'ArticleItems',

    ArticleItem,

コメントを追加するためのフォームにも注意を払う価値があります。リクエストはAPIに直接行われませんが、Vuexアクションが実行され、APIへのリクエストはすでに「内部」にあり、必要なArticleモデルが更新され、コメントのリストが更新されます。コンポーネント自体のコードは次のようになります。

src /コンポーネント/ CommentForm.vue
<template>
  <form @submit.prevent="onSubmit">
    <textarea class='form-control' v-model="content"></textarea>
    <br>
    <button type="submit" class="btn btn-primary"></button>
  </form>
</template>

<script>
export default {
  name: 'CommentForm',
  props: {
    articleId: Number,
  },
  data() {
    return {
      content: '',
    };
  },
  methods: {
    onSubmit() {
      if (this.content) {
        this.$store.dispatch('addComment', {
          content: this.content,
          article_id: this.articleId,
        });
        this.content = '';
      }
    },
  },
}
</script>


すべてのコンポーネントとページを実装し、スタブを実装してテストデータを追加したら、この特定のケースでは、ブログの前半部分が完成します。しかし、通常は実際に発生するので、作業の完了後に変更が始まるため、これはプロジェクトの終わりにはほど遠いです:-)

  • コンポーネントはビジュアルのみを扱い、データはリポジトリからモデルの形で取得され、直接のAPIリクエストは行われません。
  • モデルには、必要なビジネスロジックとエンティティ間のすべての関係が含まれています。また、APIにはアクセスしないでください。
  • ストレージ（vuex）はすべてのコンポーネントの接続リンクです。データはAPIからリクエストされ、モデルに変換され、コンポーネントによってアクセスされます。
