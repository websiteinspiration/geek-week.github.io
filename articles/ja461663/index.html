LinuxがWindowsを導入した経緯

マイクロソフトで働いている間は常に、Linux開発者向けのツールを作成してきました。バージニア大学でコンピュータサイエンスと管理を学び、2016年8月に仕事を始めました。勉強中は主にC ++でプログラミングしました。私の主なオペレーティングシステムはLinuxでした。 私の経験はMicrosoftが必要とするものと完全には一致していないように思えるかもしれませんが、当時、同社はテクノロジーと文化の両面で大きな変化を遂げていました。同社は、Linuxを含むすべてのオペレーティングシステムが重要な新しい状態に移行していました。





Microsoftでの私の仕事はLinuxチームで始まりました。そこでSQL Server製品を使用していました。私はこのチームに招待され、Linuxでの経験を活かしてもらいたいと思っていました。省略したばかりですが、自分の経験からチームにとって価値のあるものになり得るという事実に非常に感銘を受けました。

数年前、Linux用のSQL Serverのバリアントを開発するという考えはエイプリルフールのジョークにすぎませんでしたが、2016年にはそれは完全に現実のものでした。彼らが製品の最初のバージョンをリリースした直後に私はチームに加わりました。特に管理者向けに、SQL Serverツールキットを改善し始めました。このツールキットは、Linuxサーバーとアプリケーションの管理を目的としています。 LinuxでSQL Serverを使用するには、コマンドラインツールをLinuxユーザーが慣れている外観に合わせる必要がありました。

さらに、Linux用のSQL Server GUIの最初のバージョンを設計する機会がありました。私はVisual Studio Codeのフォークから始めました。これは現在、Azure Data Studioと呼ばれています。これは、オペレーティングシステムに関係なく、あらゆる種類のSQL Serverで動作するElectronベースのアプリケーションです。

Microsoftでの最初の1年間、Linux用のSQL Serverから多くのことを学びました。この知識には、テクノロジーとビジネス思考の組み合わせに基づいてプロジェクトの開発と保守を管理するアプローチの開発を含めることができます。


Microsoft Build 2018のWSLチーム、Chocolatey、およびBoxstarter

2017年8月、私はプロジェクトマネージャーとしてWSLチーム（LinuxのWindowsサブシステム、LinuxのWindowsサブシステム）に参加しました。 WSLについて初めて聞いたのは、Microsoft Build 2016での発表のときでした。チャンネル9の関連動画「 WindowsでUbuntuでBashを実行する！"、リリース直後にバイラルとなった。それは会議で行われた他の多くの発表よりも明らかに聴衆に興味を持っていました。 WSLテクノロジーは、会議の主要な論文を発表する枠組みの中で、文字通り数分以内に簡潔に議論されました。しかし、これからの聴衆は、ジャーナリストは言うまでもなく、直接狂っていました。 Channel9サポートチームが、このビデオクリップに対するユーザーの関心の高さは、実際にはDDoS攻撃によるものだと恐れたことがあります。 Microsoftの広報担当者がWindows内で実行されているUbuntuでBashを起動...このアクションはすぐにヒットしました。

WSLのユーザーニーズを発見した最初のチームは、Windowsコンソールで作業したチームです。開発中に、人々はLinuxのBashのようなものを求めていると繰り返し耳にしました。その結果、チームは次のように考えました。「WindowsでBashシェルを正しく実行できるのに、なぜBashに似ているのですか？」

簡単だったと言っているのではありません。 WSLの作成には、システムの中核となる開発チームが所有するWindowsの詳細な知識と、Microsoft Researchで開発されたpicoprocessと呼ばれるテクノロジの組み合わせが必要でした。さらに興味深いことに、picoprocessesは、SQL ServerをLinuxで動作させるまさにそのテクノロジーです。

picoprocessは、ユーザーモードで未変更のLinux実装を提供することになっています。 Windowsカーネルチームは、LinuxシステムコールをWindowsに接続するように設計されたシムを開発しています。つまり、WSLのおかげで、Linux用にコンパイルされたコードをWindows NTカーネルで実行できるようになりました。コードを再コンパイルしたり、仮想マシンを使用したりする必要はありませんでした。

独自のLinuxディストリビューションのみを作成したわけではありません。実際には、そのような分布がたくさんあったのです。 WSLで利用可能な最初のLinuxバージョンはUbuntuでした。 Canonicalの専門家に連絡して、WSLでの作業を支援してほしいかどうかを確認しました。彼らは私たちのアイデアに熱心でした。これにより、UbuntuがMicrosoft Storeに登場した。ちなみに、前の文自体は、かなり変わったように聞こえます。現在、Microsoft Storeでは、6つの配布をカウントできます。特定のオペレーティングシステムで利用可能な他のアプリケーションストアには他のオペレーティングシステムがあるのでしょうか。


現在、Microsoft Storeには、WSLで使用できる6つのLinuxディストリビューションがあります。

私たちが作成したコードは、LinuxプロセスとWindowsカーネル間のインターフェイスとして機能するLinux互換のカーネルシステムコールでした。 Linuxには約340のシステムコールがあります。問題は、最初に実装するシステムコールを決定することでした。他のオペレーティングシステムと同様に、Linuxでは新しいバージョンのOSがリリースされると新しいシステムコールが追加されますが、古いコールは削除されないため、下位互換性が保証されます。最初は、特定のシステムコールのセットの処理が実装され、その他はすべて「まだ実装されていません」などのイベントにラップされていました。これにより、WSLチームはLinuxユーザーが必要とするシステムコールを正確に分析し始めることができました。

どのシステムコールを最初に実装する必要があるかという質問に対する答えは、WSLを使用する人々とのコミュニケーションを確立する必要性を意味しました。 Buildカンファレンスでのこのテクノロジーに関するメッセージは、人々にWSLの使用を開始してフィードバックを提供することを目的としています。 WSLを取得するには、Windows Insider Programのメンバーである必要があります。誰でもこのプログラムに接続できます。プログラムの参加者はWindowsに興味がある人だけだったと思うかもしれませんが、1,000万人を超えるサブスクライバーの中には、さまざまな興味を持つ人々がいました。彼らはWindowsだけでなく、ゲーム、Bluetoothの新機能、WSLなどにも興味を持っていました。

BashシェルをWindowsで実行することに関心があるユーザーグループの1つは、Linuxサーバーで実行するWebアプリケーションの構築に携わるWeb開発者でした。それらのアプリケーションを組み立てるプロセス全体は、多くの場合、一連のBashコマンドでした。さらに、誰かがWebアプリケーションの開発、たとえばStack Overflowの開発に協力を求めた場合、彼が見つけることができるほとんどのコードサンプルはLinuxで実行するように設計されます。これは、Web開発用のコンピューターとしてWindowsを使用するユーザーには特に適していません。多くの場合、そのような開発者がMacプラットフォームに切り替えるのが最も簡単です。 macOSでは、同様のコード例が問題なく実行されます。

WindowsでのWSLの存在の最初の数週間で、1人のエンタープライズユーザーがXEyesを実行します。このプログラムは、WSLで実行されているX11ウィンドウシステムで実行されました。XEyesはシンプルなプログラムです。彼女はマウスカーソルに続く漫画の目を表示します。しかし、このデモはソーシャルネットワークを爆破しました。


XEyesはWSLを介してWindowsで実行されます

WSLに関するユーザーレビューを収集する方法について、多くのことを議論しました。フィードバックを収集するために使用された従来のツールは、UserVoiceでした。 WSL用に設計されたUserVoiceサイトがあり、さまざまな問題に対する数百のアイデアと数千の投票が集められました。 GitHubを使用するべきかどうかということに関して、真の疑問が浮上しました。 Web開発者はWSLに最初に興味を持ったユーザーグループの1つだったので、GitHubの質問は非常に重要でした。しかし、WSLはオープンソースプロジェクトではありませんでした。そのようなプロジェクトをGitHubに配置するのは奇妙に見えます。開発者の要件を満たすことを決定し、GitHubに問題の報告、フィードバック、ディスカッションのためのページを作成しました。それ以来、WindowsでのLinuxの使用に関連する多くの問題に関する数千のメッセージを受け取りました。

何千人もの人がWSL GitHubリポジトリにエラーメッセージを残しています。このようなメッセージはそれぞれ、WSLチームによってレビューおよび議論され、それぞれにコメントが付けられました。その後、さらなる行動が決定されました。特定の機会を実現するため、または何らかのエラーを修正するために、新しいコードを記述する必要があった場合、このコードが作成されてWSLプロジェクトに追加され、その後、Windows Insiderプログラムの下で配布されるすべてのWindowsアセンブリに分類されます。エラーメッセージが表示されてから修正されるまでの全体のサイクルに、数週間はかかりませんでした。

その結果、ユーザーの要求に対するWSLチームの運用応答のおかげで、コミュニティはWSLの作成プロセスに関与していたと言えます。人々はUserVoiceまたはGitHubを通じて問題や希望を報告し、チームはこれらすべてをレビューし、プロジェクトに変更を加え、それがWindows Insiderプロジェクトのビルドに表示されました。

プロジェクトマネージャーとしてWSLチームに参加したとき、私はWSLをベータ版から解放することに集中しました。ユーザーの不満は主に互換性とパフォーマンスに関連しています。しかし、私の意見では、ユーザーがそのようなことを心配する場合、それは彼らが私たちの開発を真剣に使用していることを意味します。パフォーマンスは、特定のソフトウェアシステムの助けを借りていくつかの大きな問題を解決する人によってのみ処理されます。その結果、やらなければならないことがまだたくさんありましたが、WSLを使用してより多くの問題を解決できるように、そして問題をより早く解決できるようにするために、私たちは理由を持ってそれを行いました。

WSLの機能が拡大し始めたので、マイクロソフトは従来からMicrosoftエコシステムを使用して作業しているユーザーだけでなく、開発者にWSLを近づけるように努力しました。 PyConやOSCONなどのイベントに参加することは非常に興味深いものでした。そこにいた開発者たちは、マイクロソフトの代表もこれらのイベントに参加したことに驚いた。開発者は、Windows環境でLinuxを実行するという考えに不信感を抱いていました。これらのイベントでは、SQL Server、WSL、およびVisual Studio Codeを紹介しました。


さまざまなイベントでのWSLのデモンストレーション、

私は彼らの懐疑的なコメントに、私が示したものを試すよう提案することで答えました。疑い深い人が独自のコマンド、小さなスクリプト、スニペットを実行し始めたとき、私は常に起こっていることに激しい反応を示しました。「ちょっと待って、これは本当にLinuxです。どうやったの？なぜこれを知らなかったのですか？」多くの場合、彼らは私たちが彼らのニーズを満たすもの、非常に興味深いものを作成したという結論に達しました。

WSLの互換性とパフォーマンスに関するユーザーの不満を考慮して、新しいシステムアーキテクチャ -WSL 2をリリースしました。 LinuxカーネルをWindowsに含めることで完全な互換性を提供し、速度が20倍向上します。 WSL 2の基盤を作成し、2019年5月に開催されたビルド会議でこのテクノロジーの発表を見て興味深い経験をしました。今日、WSLプロジェクトはすでにベータ版を超えており、バージョン2になっています。

さらに、マイクロソフトで他のチームと協力して、WSLが彼らの製品で確実に使用できるように努めました。 WSLのこの使用の重要な例の1つは、Visual Studio Codeです。これは、JavaScriptおよびNode.jsプロジェクトの開発に使用される最も一般的なコード環境です。このエディターを使用する開発者がWSLから大きなメリットを得ることができることに気付いたとき、私はVisual Studio Codeに興味を持ちました。最初は、大量のコードを記述する必要はありませんでした。主な目的は、WSL環境で実行されているNode.jsプログラムのデバッグを簡素化することでした。これにより、開発者はWSLを使用して、Windowsコンピューター上でLinuxバージョンのNode.js用に設計されたプログラムを開発する機会が得られます。このようなプログラムのデバッグは、Linuxでのデバッグとまったく同じです。


Visual Studio Code WSL Node.js

これがJavaScriptとNode.jsで可能であることが判明した後、他の言語（C ++やPythonなど）でも同様のことを行うよう求める多くのリクエストを受け取り始めました。私はWSLとVSコードの統合のこの例に魅了されました。私はすべてこれに非常に興味を持っていました。これにより、Linux開発者向けのツールを構築する上での新しい役割に私を導いた。今、私はVS CodeのC ++開発者向けのツールに焦点を当てています。もちろん、この仕事ではLinuxに焦点を当てています。対応するWSL拡張機能がリリースされた今年のPyConイベントで、Visual Studioコードのリモート開発のデモを見ることができて良かったです。同時に、私のチームが開発していたC ++の拡張機能が導入されました。

マイクロソフトにあまり時間をかけなかったにもかかわらず、Linux開発者向けの多くのツールの作成に参加できたことを嬉しく思います。このデータベースとLinux on Windowsのサポート、およびコードの作成とデバッグのためのツール。今後もLinuxの開発に取り組み、世界中の開発者が楽しめるツールを作成していきます。
