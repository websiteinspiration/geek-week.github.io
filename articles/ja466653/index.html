NESのゲームのデバッグ：今日の状況


プログラミングに関わったことがあれば、バグの概念に精通しています。彼らが私たちを気にしないのであれば、開発プロセスははるかに速く、より楽しくなります。しかし、これらのバグは、コード、作業スケジュール、創造的な流れを台無しにする瞬間を待っています。幸いにも、レトロなゲームプログラマーのコードでさえ、バグを根絶するための多くのツールと戦略があります。

デバッグツール


コードをデバッグする最良の方法の1つは、デバッガーを使用することです。FCEUXおよびMesenエミュレータの一部のバージョンには、いつでもプログラムを中断してコードの状態を確認できるデバッガが組み込まれています。


FCEUXエミュレーターデバッガー

この方法は、アセンブリ言語を使用する上級プログラマーに適しています。しかし、私たちは新しいので、C（cc65）で記述します。もちろん、コンパイラーは独自のルールで動作するため、Cコードからコンパイルされたマシンコードを処理することは困難です。

Hex Editor FCEUX

ある種の変数または配列を観察する必要があるとしましょう。リンカオプション（ld65）に次の行を追加-Ln labels.txt

しますlabels.txt。プロジェクトをコンパイルすると、そのフォルダにファイルが表示されます。テキストを表示するために任意のプログラムでそれを開き、監視する変数の名前を探します。

（注：静的変数を宣言した場合、このリストには含まれません。したがって、代わりにstatic unsigned char playerX使用してくださいunsigned char playerX）

PPUビューアー、名前テーブルビューアーなど、他の便利なFCEUXエミュレーターのデバッグメニューツールを探索します。

デバッグプロセスを簡略化する


また、変数を監視するために毎回デバッガを実行したくない場合はどうでしょうか。より高度な方法は、画面に値を表示するプロシージャを記述することです。インターフェースのスコアを使用して、Y軸上のプレーヤーの位置を表示してみましょう。


それは完璧に動作します！

レトロコーダーとnesdougブログのオーナーである Doug Frakerは、デバッグ目的で画面上の視覚化を使用するための同様の方法を作成しました。以下の手順では、CPU負荷の程度を明確に示す灰色の線が画面に表示されます。

// void gray_line(void);
// For debugging. Insert at the end of the game loop, to see how much frame is left.
// Will print a gray line on the screen. Distance to the bottom = how much is left.
// No line, possibly means that you are in v-blank.
 
_gray_line:

   lda <PPU_MASK_VAR
   and #$1f ;no color emphasis bits
   ora #1 ;yes gray bit
   sta PPU_MASK
 
ldx #20 ;wait
 
@loop2:
 
   dex
   bne @loop2
   lda <PPU_MASK_VAR ;normal
   sta PPU_MASK
   rts

このプロシージャをコードにコピーするか、プロジェクトにnesdoug.hライブラリを含めるだけです。プロシージャは、ゲームサイクルの完了後に呼び出す必要があります。そうすると、灰色のバーが画面に表示されます。


うまくいきましたが、もう1つバグがあるようです！私たちは後で彼を取り除くでしょう。その間、先に進みましょう。

マクロの力


マクロは、便利なデバッグツールにもなります。バグの原因となったコード内の場所を見つけることができます。

サウンドを再生したり、必要な値のゼロパレットを選択したりするなど、適切なタイミングで信号を提供する何らかのマクロを作成してみましょう。ゼロパレットを赤、青、ランダムカラーに変更し、サウンドを再生するマクロがいくつかあります。


使い方？プロジェクトが正常にコンパイルされ、ゲームでエミュレーターを起動し、[スタート]ボタンをクリックして、...


白い画面しか無いようです。さらに、一部のエミュレータはステータスバーに「CPU jam！」と報告する場合があります。次はどうする？

まず、エラーが発生するコードをローカライズする必要があります。そして、ここで私のマクロの音が出てきます。

メインメニューが機能していることは確かです。その後に何が起こるか見てみましょう：

playMainMenu();

player.lives = 9;
points = 0;
gameFlags = 0;

while(current_level<7 && player.lives>0)

{

       set_world(current_world);

      debugSound;

      playCurrentLevel();

}

手順中にゲームがクラッシュする疑いがありますset_world。この直感を見てみましょう。手順を確認した後、次の行にマクロ名を入力するだけです。

私たちはプロジェクトを開始し、...音が聞こえます！つまり、この手順は成功し、次のことを確認する必要がありますplayCurrentLevel。以下のデバッグマクロを移動してみましょう。

while(current_level<7 && player.lives>0)

{

       set_world();

       playCurrentLevel():

      debugSound;

}

プロジェクトを再開しましたが、音が聞こえません。これは、手順が完了しておらず、内部で障害が発生していることを意味します。

このような場合は、手順コードを開いて、バグの可能性のある場所の検索を絞り込むことができるまでこの手法を適用し続ける必要があります。

パレット変更マクロは、状態をチェックするのにも役立ちます。たとえば、コードはいくつかの条件の複雑なテストを実行します。

原子力オプション


私は最近、ゲームの幽霊の1つが何らかの疑わしい動作をしていることを発見しました。時々、彼はプレーヤーを攻撃することを拒否しました。

バグに見舞われたこのゴーストを見てください。キャラクターが画面の中央に近いときにのみ攻撃します。


この手順のコードをどれほど熱心に研究しても、バグがどこに隠れているのかわからなかったので、極端な対策を講じて、このコードの動作を最新の開発環境でテストすることにしました。

スクリーンマップ、メタファイル属性を含む配列、プロシージャコードを必要なものすべて取り、Visual Studio 2017に単純に挿入しました。


PCでは、コードはまったく同じように機能しました。結局のところ、バグはプレイヤーと敵の間の障害物を見つけるためにキャッシュを埋める手順に隠れていました。配列が正しく入力されていません。私は0x80の代わりに0があるはずだと確信しています。


そのため、コードを段階的にデバッグして、これがなぜ起こっているのかを調べます。


おかしいですが、間違った順序でアクションを実行していたようです。修正して、もう一度アレイをチェックしてみましょう！


現在、配列は正しく読み込まれているようです。つまり、cc65コードを修正して、NESプロジェクトを再度コンパイルするだけです。


したがって、最新の開発ツールは、アルゴリズムのデバッグとバグの修正に役立ちます。
