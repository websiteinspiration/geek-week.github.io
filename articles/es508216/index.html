Qué es CI (Integración continua)

CI (integración continua) : traducido literalmente como "integración continua". Esto se refiere a la integración de piezas individuales de código de aplicación entre ellas. Cuanto más a menudo recopilamos el código juntos y verificamos:



  • ¿Él va a?
  • ¿Pasan las pruebas automáticas?


¡Todo lo mejor! CI le permite hacer tales comprobaciones automáticamente. Se utiliza en equipos de desarrollo avanzado que escriben no solo código, sino también pruebas automáticas. Se le pregunta en las entrevistas, al menos una comprensión de lo que es. Sí, incluso con probadores.



Por lo tanto, contaré en un artículo sobre qué es. Cómo funciona CI y cómo será útil para su proyecto.

¿Qué es el CI?



CI es el ensamblaje, despliegue y prueba de una aplicación sin intervención humana. Ahora lo explicaré con un ejemplo.



Digamos que tenemos dos desarrolladores: Masha y Vanya. Y el probador Katya.



Masha escribe el código. Lo agrega al sistema de control de versiones (del Sistema de control de versiones en inglés , VCS ). Esto es algo así como un buzón para el código: una ubicación de almacenamiento donde se guardan todos los cambios y en cualquier momento puede ver quién cambió qué y cuándo.



Entonces Vanya termina su pieza de funcionalidad. Y también guarda el código en VCS.







Pero este es solo el código fuente: un conjunto de archivos con la extensión .java o cualquier otro. Para que Katya pueda probar los cambios, necesita:



  1. Construir una compilación a partir del código fuente
  2. Ejecútalo en una máquina de prueba


Construir una compilación es cuando creamos un archivo ejecutable a partir de un conjunto de archivos de código fuente:







puede compilar una compilación manualmente, pero esto es una hemorroide innecesaria: recuerde que en qué orden ejecutar, qué archivos dependen el uno del otro, no se equivoquen en el comando ... Por lo general, usan un comando especial el programa. Para java, esto es Ant, Maven o Gradle. Con la ayuda del recopilador, configura el proceso de compilación una vez y luego lo ejecuta con un comando. Ejemplo de lanzamiento para Maven: 



mvn clean install


Esto es semiautomatización: aún necesita una persona que ingrese a un equipo y ensamble la construcción con “plumas”. Digamos que los desarrolladores hacen esto. Cuando Katya le pide a la compilación que pruebe, Vanya actualiza la versión del repositorio y recopila la compilación.







Pero construya una compilación ≠ obtenga una aplicación para probar. ¡Todavía necesita ser lanzado! Esto lo hace el servidor de aplicaciones. Los servidores son diferentes: wildfly, apache, jetty ...



Si esto es wildfly, entonces necesitas:



  1. Adjuntar compilación al directorio independiente / implementaciones
  2. Inicie el servidor (después de configurar el servicio una vez)


Y esto es nuevamente semiautomatización. Porque el desarrollador necesita copiar el archivo resultante después del ensamblaje en el banco de pruebas y habilitar el servicio. Sí, esto lo hacen un par de equipos, pero sigue siendo una persona.







Pero si eliminamos a una persona de este esquema, ¡obtenemos CI!







CI es una aplicación que le permite automatizar todo el proceso. Extrae cambios del repositorio con código. ¡Por sí mismo! Hay dos opciones de configuración:



  • CI sondea el repositorio "Hola cuco, ¿tienes algún cambio?" cada N horas / minutos, cómo configurarlo.



  • El repositorio saluda a CI cuando se compromete: “¡Hola, hola! ¡Y tengo una actualización aquí! (¿Es un gancho git o equivalente en su VCS)







Cuando el CI recibe los cambios, comienza a construir la compilación y las pruebas automáticas.



Si el ensamblaje falla (las pruebas caen o el proyecto no se pudo ensamblar), el sistema escribe un correo electrónico a todas las partes interesadas:



  • Project Manager (para saber lo que se está haciendo!)
  • El desarrollador que hizo los cambios.
  • Cualquier otra persona: a medida que lo configure, será así.








Si la compilación es exitosa, CI implementa la aplicación en la máquina de prueba. ¡Y al final Katka puede probar la nueva asamblea!







Sí, por supuesto, una vez que tenga que configurarlo todo: dígale al servidor de CI de dónde obtener los cambios, qué pruebas automáticas ejecutará, cómo compilar el proyecto, dónde construirlo más tarde ... Pero luego lo configuré una vez, ¡y lo hace!



Las pruebas automáticas también deberán ser escritas por nosotros mismos, pero bien hechas =)







Si está en sus dedos, entonces el sistema CI ( integración continua ) es un tipo de programa que monitorea su control de origen, y cuando aparecen cambios allí, los reúne automáticamente, los construye, ejecuta pruebas automáticas (por supuesto, si están escritos).



En caso de falla, se lo hace saber a todas las partes interesadas, en primer lugar, hasta el último responsable. (c) habr.com/en/post/352282






Programas de CI



Los más populares son Jenkins y TeamCity.



Pero hay muchas otras variaciones: CruiseControl, CruiseControl.Net, Atlassian Bamboo, Hudson, Microsoft Team Foundation Server.





Cómo se ve



Veamos cómo se ve desde el punto de vista del usuario. Lo mostraré con un ejemplo del sistema TeamCity.



Cuando inicio sesión, veo todas las tareas. Las tareas son diferentes:



  • Construir construir
  • Conducir pruebas automáticas
  • Implemente la aplicación en el banco de pruebas.
  • Ejecute pruebas de GUI (o pruebas de cartero) en este soporte
  • Notifique a todos los interesados ​​por correo electrónico sobre los resultados del montaje y las pruebas.


Las tareas se pueden agrupar. Aquí, digamos, tenemos un proyecto CDI. Al entrar, veo las tareas para este proyecto en particular:







  • CDI Archetype y CDI Core son compilaciones. Verifican que la aplicación esté generalmente en construcción. Haga ejercicio en un par de minutos y corra para cada cambio de código.
  • CDI Core con pruebas : ensamblaje del proyecto con todas las pruebas automáticas, que, como se puede ver en la pantalla, más de 4000 piezas. Las pruebas toman media hora, pero también se ejecutan para cada confirmación.

Cómo funciona CI



¿Cómo y dónde recopila CI las compilaciones y ejecuta las pruebas automáticas? Usaré TeamCity como ejemplo, pero otros sistemas funcionan de manera similar.



TeamCity en sí no recoge nada. El montaje y la ejecución de las pruebas automáticas tienen lugar en otras máquinas llamadas "agentes":







"Agente" es una computadora simple. Una pieza de hierro o una máquina virtual, no el punto. Pero, ¿cómo entiende esta computadora lo que necesita hacer?



TeamCity tiene un servidor y un cliente. El servidor es la aplicación en la que luego presionarás los botones y verás una hermosa imagen "cómo todo salió bien". Se instala en una máquina.







Y la aplicación "cliente" se instala en las máquinas "agente". Y cuando hacemos clic en el botón "Ejecutar" en el servidor:







El servidor selecciona un cliente gratuito y le da todas las instrucciones: qué debe hacerse exactamente. El cliente recopila la compilación, realiza pruebas automáticas, recopila el resultado y regresa al servidor: "Encendido, retener, dibujar".







El servidor muestra el resultado al usuario y envía un correo electrónico a todas las partes interesadas.

También podemos configurar para cada conjunto una lista de agentes en los que se puede ejecutar. ¿Para qué? Por ejemplo, tenemos la mitad de los agentes de Linux y la mitad de Windows. E hicimos el montaje solo para un sistema. O en Windows, algún tipo de error flotante se arrastra, pero solucionarlo es largo y costoso, y todo el cliente en Linux, bueno, ¿por qué entonces?



Y también sucede que los agentes se dividen entre proyectos para que no haya lucha: este proyecto usa Burana y Apollo, y eso Chip y Dale. La cuestión es que solo se puede realizar una tarea en un agente. Por lo tanto, no tiene sentido comprar un auto genial para un agente; de ​​todos modos, de inmediato no puede conducir un montón de pruebas allí.



Al final, cómo funciona: primero, el administrador compra computadoras bajo los "agentes" e instala la aplicación cliente de TeamCity en ellas. No deberían ser demasiado geniales, porque no harán muchas tareas a la vez.







En este caso, TeamCity paga por la cantidad de licencias para los agentes de compilación. Entonces, cuantos menos sean, mejor.



En una máquina separada, el administrador instala el servidor TeamCity. Y lo configura: configura ensamblajes, indica qué ensamblajes se pueden manejar en qué máquinas, etc. El servidor necesita un lugar para almacenar artefactos: los resultados del ensamblaje.



Tenemos dos proyectos: Cliente único y Factor, que interactúan entre sí. El Probador de cliente único no puede recopilar Factor localmente. ¡Lanza la compilación en TeamCity y descarga la compilación terminada de los artefactos!


Además, el desarrollador elige qué ensamblaje desea ejecutar y hace clic en "Ejecutar". Qué sucede en este momento:



1. El servidor de TeamCity verifica la lista en qué agentes se puede ejecutar este ensamblado. Luego comprueba cuál de estos agentes está actualmente libre: ¿







Encontrado libre? Le di el rompecabezas!







Si todos los agentes están ocupados, la tarea se pone en cola. La cola funciona según el principio FIFO: primero en entrar, primero en salir. Quién se levantó primero: eso y zapatillas.







La cola se puede ajustar manualmente. Entonces, si veo que la cola está llena de ensamblajes lanzados por el sistema de control de versiones, subiré la mía a la cima. Si veo que las personas comenzaron asambleas, significa que también son importantes, tienes que esperar.







Esta es una práctica normal si las capacidades del agente no son suficientes y se crea una cola. Mira quién lo lanzó:



  • ¿Robot? Entonces, esto es solo una verificación de rutina de que nada más se ha roto. Esto puede esperar 5-10-30 minutos, está bien
  • ¿Colega? Esta asamblea es importante para él, ya que no esperó el lanzamiento previsto. Hacemos cola, no debes avanzar.


A veces, incluso puede cancelar la compilación desde el sistema de control de versiones, si hace mucho calor, y todos los agentes están ocupados con las pruebas por hora. En este caso, puedes:



  • eleva tu turno a la cima para que comience con el primer agente libre
  • inicie sesión en el agente, cancele la compilación actual
  • reiniciarla! Aunque caerá al final de la línea, es feo cancelar el ensamblaje


2. El agente realiza la tarea y devuelve el resultado al servidor







3. El servidor dibuja el resultado en la interfaz gráfica y guarda los artefactos. Entonces puedo ingresar a TeamCity y ver en los artefactos los registros completos de las pruebas automáticas anteriores, o descargar el ensamblaje del proyecto para implementarlo localmente.



Se recomienda encarecidamente que preconfigure la cantidad de ensamblajes que almacenará CI. Porque si los artefactos contienen compilaciones de más de 200 mb y habrá muchos de ellos, el próximo lanzamiento del ensamblado fallará con el error "sin espacio en disco":







4. El servidor envía el correo electrónico, así es como lo configura. Él puede hacer un boletín positivo "la asamblea fue exitosa", y puede enviar correos solo en caso de falla: "¡Oh, ella, algo salió mal!".





Integración con VCS



Hablé sobre diferentes opciones para configurar la integración de CI-VCS:



  • CI sondea el repositorio "Hola cuco, ¿tienes algún cambio?" cada N horas / minutos, cómo configurarlo.



  • El repositorio saluda a CI cuando se compromete: “¡Hola, hola! ¡Y tengo una actualización aquí! (¿Es un gancho git o equivalente en su VCS)







¿Pero cuándo se usa?



Lo mejor de todo, por supuesto, es que el sistema de control de versiones notifique al servidor de CI. Y ejecute todo el ciclo para cada cambio: ensamblar, probar, implementar. Luego, cualquier cambio en el código irá inmediatamente al entorno de prueba, que será lo más relevante posible.







Además, cada cambio ejecuta pruebas automáticas. Y si las pruebas fallan, queda claro de inmediato quién los rompió. Después de todo, solía funcionar y después de las ediciones de Vasya, de repente se rompió, lo que significa que su compromiso condujo a una caída. Con raras excepciones, cuando la caída está flotando.







Pero en la vida real, este esquema rara vez es aplicable. Solo piense: puede tener muchos proyectos, muchos desarrolladores. Cada uno comete algo al menos una vez cada media hora. ¡Y si ejecuta 10 ensamblajes en media hora por cada confirmación, las colas en TeamCity nunca se clasificarán!



Uno de nuestros productos tiene un módulo central y hay más de 15 clientes. Cada uno tiene sus propias pruebas automáticas. El ensamblaje del cliente es el núcleo + características del cliente. Es decir, un cambio en el proyecto raíz puede afectar a 15 ensamblajes diferentes. Por lo tanto, todos deben ejecutarse cuando se comprometen en el núcleo.



Cuando tuvimos 4 agentes de compilación, el martes por la noche se lanzaron compilaciones y pruebas generales para estos clientes. Y a las 10 a.m., todavía había un par de horas en TeamCity.


Otra opción es comprar muchos agentes. Pero este es el precio del auto en sí + de la licencia en TeamCity, que ya es mucho más caro, e incluso paga todos los meses.



Por lo tanto, por lo general hacen lo siguiente:



1. Se pueden dejar ensamblajes muy rápidos e importantes para cualquier compromiso: si demora de 1 a 2 minutos, déjelos perseguir.







2. El resto de las compilaciones comprueban si ha habido cambios en VCS, por ejemplo, una vez cada 15 minutos. Si hubo, entonces lanzamos.







3. Pruebas largas (por ejemplo, pruebas de rendimiento): una vez cada pocos días por la noche.











CI en pruebas



Si hablamos de desarrollar nuestra aplicación, las pruebas se incluyen en el ciclo estándar. Usted o sus desarrolladores escriben pruebas automáticas, que luego son controladas por CI. Pueden ser pruebas de unidad, api, gui o estrés.







Pero, ¿qué pasa si estás probando una caja negra? Hay una aplicación, no hay código fuente. Estas son las duras realidades de los probadores de integración: el proveedor le ofrece una nueva versión de la aplicación, que debe verificar antes de ponerla en producción.



Aquí, supongamos que tiene pruebas de API en Postman. O pruebas de GUI en selenio. ¿Puedo personalizar el bucle CI para ellos?







¡Por supuesto que puede!



CI no establece un marco estricto como "Solo trabajo en proyectos con pruebas automáticas" o "Solo trabajo cuando tengo acceso al código fuente". Puede buscar en el sistema de control de versiones, o puede que no lo haga. Esta es una condición opcional!



Escribió autotests? Dígale al servidor de CI cómo ejecutarlos con frecuencia, y disfrute el resultado =)









Total



CI - integración continua. Aquí es cuando su aplicación se prueba constantemente: ¿está todo bien? ¿Pasan las pruebas? ¿Se va la asamblea? Además, todos los controles se llevan a cabo automáticamente, sin intervención humana.



Es especialmente importante para los equipos donde varios desarrolladores trabajan en el código de una aplicación. ¿Como sucedió esto? Por separado, partes del programa funcionan, pero juntas ya no. CI le permite detectar rápidamente tales problemas. Y cuanto más rápido lo encuentre, más barato será arreglarlo.







De ahí el nombre: una verificación constante de la integración de piezas de código entre sí.

Tareas típicas de CI:



  • Verifique si hubo una actualización en el código
  • Construir construir
  • Conducir pruebas automáticas
  • Implemente la aplicación en el banco de pruebas.
  • Ejecute pruebas de GUI (o pruebas de cartero) en este soporte
  • Notifique a todos los interesados ​​por correo electrónico sobre los resultados del montaje y las pruebas.








Y todo esto, automáticamente, ¡sin intervención humana! Es decir, lo configuraron una vez, y luego lo hizo.



Si CI está configurado en el proyecto, tendrá bancos de pruebas constantemente actualizados. Y si algo rompe el código, lo sabrá de inmediato, el servidor de CI enviará una carta. ¿Y puede acceder a la interfaz gráfica y ver si todas las compilaciones son exitosas y las pruebas son verdes? Califica la imagen del proyecto en un minuto.
