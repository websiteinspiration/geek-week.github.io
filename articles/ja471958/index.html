開始


それはすべて、私たちの会社のサービスがkubernetesに移行することが決定されたという事実から始まりました。それは70％のウィンドウと30％のLinuxです。このため、AWS EKSクラウドサービスは、可能なオプションの1つと見なされていました。 2019年10月8日まで、AWS EKS Windowsはパブリックプレビューでしたが、私はそれを使い始めました。kubernetesのバージョンは古い1.11で使用されていましたが、とにかく確認して、このクラウドサービスがどの段階で機能しているかを確認し、まったく機能していないことがわかりました。ハースの削除が追加されたバグ。古いものはWindowsワーカーノードと同じサブネットから内部IP経由で応答しなくなりました。

したがって、同じEC2上のkubernetes上の独自のクラスターを支持してAWS EKSの使用を放棄することが決定され、すべてのバランシングとHAのみがCloudFormationを通じて記述される必要があります。

EKSクラスターを現在のVPCとサブネットに統合する


すべてのソースで、発表の上のリンクとドキュメントでも、Amazonのパブリックサブネットのみを使用して、独自のユーティリティeksctlまたはCloudFormation + kubectlを介してクラスターをデプロイし、新しいクラスターに個別のVPCを作成することが提案されました。

このオプションは多くのユーザーには適していません。まず、個別のVPCは、そのコスト+現在のVPCへのピアリングトラフィックの追加コストです。複数のAWSアカウント、VPC、サブネット、ルートテーブル、トランジットゲートウェイなどを備えたAWSに既製のインフラストラクチャをすでに持っている人はどうしたらいいですか？もちろん、分解したり、すべてやり直したりしたくないので、既存のVPCを使用して新しいEKSクラスターを現在のネットワークインフラストラクチャに統合し、最大数を分割して、クラスターの新しいサブネットを作成する必要があります。

私の場合、このパスが選択されました。既存のVPCを使用し、2つのパブリックサブネットと2つのプライベートサブネットのみを新しいクラスターに追加しました。もちろん、Amazon EKSクラスターVPCの作成のドキュメントに従って、すべてのルールが考慮されました。

EIPを使用するパブリックサブネットにワーカーノードがないという条件も1つありました。

eksctlとCloudFormation


私はすぐに両方のクラスター展開方法を試したことを予約します。どちらの場合も画像は同じでした。

ここではコードが短いため、eksctlを使用した場合のみの例を示します。eksctlを使用して、クラスターは3つのステップでデプロイされます

。1.システムコンテナーと不幸なvpc-controllerが後で配置されるクラスター自体+ Linuxワーカーノードを作成します。

eksctl create cluster \
--name yyy \
--region www \
--version 1.14 \
--vpc-private-subnets=subnet-xxxxx,subnet-xxxxx \
--vpc-public-subnets=subnet-xxxxx,subnet-xxxxx \
--asg-access \
--nodegroup-name linux-workers \
--node-type t3.small \
--node-volume-size 20 \
--ssh-public-key wwwwwwww \
--nodes 1 \
--nodes-min 1 \
--nodes-max 2 \
--node-ami auto \
--node-private-networking

既存のVPCにデプロイするには、サブネットのIDを指定するだけで、eksctlがVPC自体を決定します。

ワーカーノードをプライベートサブネットにのみデプロイするには、ノードグループに--node-private-networkingを指定する必要があります。

2.クラスターにvpc-controllerをインストールします。これにより、空きIPアドレスの数とインスタンスのENIの数をカウントし、それらを追加および削除してワーカーノードを処理します。

eksctl utils install-vpc-controllers --name yyy --approve

3. vpc-controllerを含むLinuxワーカーノードでシステムコンテナーが正常に起動した後、Windowsワーカーで別のノードグループを作成するだけです。

eksctl create nodegroup \
--region www \
--cluster yyy \
--version 1.14 \
--name windows-workers \
--node-type t3.small \
--ssh-public-key wwwwwwwwww \
--nodes 1 \
--nodes-min 1 \
--nodes-max 2 \
--node-ami-family WindowsServer2019CoreContainer \
--node-ami ami-0573336fc96252d05 \
--node-private-networking

ノードがクラスターに正常にフックし、すべてが正常であるように見えた後、ノードは準備完了ステータスになっていますが、そうではありません。

問題：
アプリケーションをKubernetesにデプロイし、デプロイメント、レプリカ> 1を配置して、次の図を確認します。新しいポッドは正常に起動して動作しますが、古いポッドはネットワークインターフェースを失います。はい、はい。古いポッドは完全にネットワークがありませんが、実行状態でハングし続けます。レプリカを減らしたり増やしたりし、ポッドを削除して、最後のポッドが実行状態になったポッドだけが常に機能するわけではないようにします。残りはすべて機能しません。いずれにせよ、ポッドまたは異なるポッドは同じノードで起動します。

決定：
はい、問題はVPCのカスタム構成にあることが判明しました。つまり、ドメイン名フィールドのカスタム値を示すDHCPオプションセットを使用した場合、または完全に空の場合（私の場合、ドメイン名サーバーのみを変更したため、残りは必要ありませんでした）起動後にポッド内のネットワークインターフェイスが消えるという、理解できない問題が発生します。

DHCPオプションセットに登録する必要があります。

domain-name = <aws-region-name>.compute.internal;
