プログラミング値の簡単な歴史


資料が頭と心にしっかりと収まるように、過去50年間に人々がプログラミングで何をしたかったのか、そして彼らが何をしたのかを簡単に思い出してください。20年の開発経験に裏付けられた、個人的な主観的意見とホリバーだけの違反はありません。

低水準言語：C、C ++


あなたがプログラムをマシンコード上の数字の形ですぐに書くことができることは明らかです、そして多くはZX Spectrum、BK0010-01とPCでそれを書きました-コードは非常に高速であることが判明しますアセンブラの発明でさえあまり役に立たなかった-このような低レベルのコードが書かれることは非常にまれであり、非常に正確であり、おそらくドライバ、マイクロコントローラ、または狡猾な組み込みシステムを開発していない場合、これは実際には役に立たないでしょう。


70年代前半に、Bell LabsはC言語を思い付きました。C言語は簡潔な構文と非常に「安価な」抽象化のおかげで定着し、ほぼ「ポータブルアセンブラ」になりました。緊張したり、Cナイトで10年間書いたり、肉を食べたり、祈ったりせず、ソーシャルネットワークや公正なセックスに気を取られない場合、GNUが優れた生産性の高いゲームであることを雄弁に証明する非常に便利で高速なプログラムを作成できることは明らかです。愛されていますが、Windowsの品質は変更できません。さらに多くの例をあげることができます。
しかし、コインの裏側は常に感じられています-C言語自体の概念の穴によって引き起こされるセキュリティの穴を定期的に開く（「ソフトウェアの穴」の業界全体が作成されました）-コンパイラは、信じられないほどの力、激しいオルガスム、短いメモリを備えた無責任な牛のようなものです。過失-プログラムをドロップするだけではなく（nullポインターを参照し、ポインターを2度解放し、配列から出る）、データを破壊し、クライアントが呼び出しを開始するまでの間、そしてそれが遅すぎる場合（「未定義の動作」、異なるコンパイラからコンパイラへ）。

BjörnStraustrupは、80年代初頭に状況をさらに混乱させ、CにOOP機能を追加しました。C++は非常に人気がありますが、一般的に、一連のプログラミング実験として見なされ、さまざまな成功結果が含まれています。致死。時々、C ++には当初から意味がなかったように見えることもあれば、徐々に失われ、客観的に複雑で矛盾する概念のヒープを生み出し、新しい標準ごとにますます増えている。「ゼロコストの抽象化」という優れた目標にもかかわらず、Cのように高速なコードを取得し、信頼性の高いソリューションを作成できるようにするには、次の条件が必要です。
  • 「非常に」という言葉から経験豊富なチーム（長年の実践、プログラミングの「僧侶」）
  • 優れた静的コードアナライザー
  • テスト部門（無責任なコンパイラによって残されたコードに、長期間感じることができる穴がある可能性があります）
  • 注意深く監視されているすべてのチームメンバーが合意した要件（生のポインター、厳密なカプセル化、変数の厳密な名前付け、オブジェクトの初期化などは使用されません）


これらの要件を遵守することは、特に「突然のサプライズなしで機能するコード」に対するビジネスニーズが高まっている状況では、非常にコストがかかることは明らかです。そのようなプロジェクトのコードは長い間書かれているため、長期間慎重にテストする必要がありますが、C / C ++がないと、Rustが発明される前に、それを行うのが本当に困難でした。

もう一度、C / C ++の要約-「現在の抽象化」を備えた強力であるが「無責任」なコンパイラーがあり、開発者をほとんど助けません。その結果、すべての問題はプログラマーの肩に委ねられます。チーム内の少なくとも1人のプログラマーが経験がなく、注意深くなく、コンパイラーのすべての機微を知らない場合（実際、誰もすべての機微を知っておらず、ユーザーは後で見つけます）、問題が発生するまで待ちます。しかし、プログラムはすぐに機能し、おそらく正しく動作します:-)もちろん、これは「クランチ」の市場全体を生み出しました。静的アナライザーは、結局のところ、顧客が支払う必要があります。問題が発生します-開発者を助け、驚きや低レベルのセキュリティホールなしでプログラムを生み出す、より厳密で安全なコンパイラを作成することはできませんか？

C＃、Scala、Kotlin


「そして、それはいつラストのことになるのですか？」 -待って、この甘い柿を食べるにはもう少し準備が必要です。他のテクノロジーの機能について話さなければ、なぜRustが登場したのか、なぜRustが非常に人気があり需要があったのか理解できません。

関数型プログラミング、Lisp、Haskell、F＃


開発者のキャリア全体として、多くの人がここに来ることはなく、無駄になっています。FP（関数型プログラミング）が登場した理由と、一部の地域でFPが非常に人気がある理由を理解することは非常に役立ちます。

簡単に説明します。数学には「停止問題」などの解決できない問題があります。非常に単純な言葉で非常に厳密ではないがそれが明確である場合、プログラムがバグなしで機能することを証明するアルゴリズムを思い付くことができません。どうして？当初、人々は積極的かつ命令的にプログラミングを開始しました：
  • 可変変数
  • 条件付きサイクル

Haskellは率直に言って、積極的な「見せかけ」のために離陸しませんでした。彼の専門用語をよく理解するには、現代数学の主要な理論の1つであるカテゴリ理論に特化した科学の候補者である必要があります。そして、Haskellでは、実行グラフがオーバーフローするか、実行が開始され、メモリが後で終了するなど、「怠惰」が明らかに賢明でした。これは、戦闘での使用に大きな影響を与えます。そのため、敵の領土での戦闘に参加している兵士がハスケルを投げ、カラシニコフ突撃銃を持ちます。

どのようにしてガベージコレクタを拒否しましたか？


しかし、このように！コンパイラのメモリは、リソースとして提示されるようになり、その「所有権」は1つの変数のみにすることができます。変数が「ala of scope」（より正確には、「ライフタイム」、Rustブックの詳細ですが、すべてが直感的にシンプルです）を超えた場合、コンパイラーはデストラクターへの呼び出しを挿入します（デストラクターがない場合、何も呼び出されません）。
変数が関数/メソッドに渡されると、所有権がそこで転送され、関数の最後にデストラクタがコンパイラによって呼び出されます。変数が関数/メソッドから返される場合、所有権もより高いレベルに戻り、デストラクタはより高いレベルで呼び出されます。
このような単純な方法で、コピーコンストラクターと移動セマンティクスに関連するC ++のひどい濁りを取り除きました。説明が複雑に思われる場合は、錆びたデストラクタを使用してコードを記述してみてください。すべてが非常に厳密で論理的であることがわかります。

どのようにしてゼロコストの抽象化を達成しましたか？


言語にはヒープOOPと継承はありませんが、強力で柔軟な特性とカプセル化ツールがあります。もちろん、この言語ではGolangに似ています。すべてはそれらの構造とメソッドを中心に展開しますが、代数的データタイプとパターンマッチングにより、コードは正確で厳密です（残念ながら、コードはGolangに追加するのを忘れていました）。構造/データへの読み取りリンクと書き込みリンクを作成できます。ただし、同時に作成できるタイプは1つだけです。1つの書き込みリンクまたは多くの読み取りリンクです。言語用語では、これは借用と呼ばれます。（「借用」）。これにより、オブジェクトをコピーする必要性が減り、データの競合が回避されます（借用によって「読み取り/書き込みロック」が実装されるため）。このすべては、コンパイラーによって厳密にチェックされ、コードがコンパイルされない場合は、コードを修正して、エラーがないことを悲観的なコンパイラーに証明する必要があります。
文字列やその他の操作を処理するために、低レベルのスライスタイプが使用されます。しかし、問題があります-行は「少し複雑」に配置されており、コンパイラーの変更の結果として、それら自体が開発者の意外な言語で表示されているようです。しかし、それらは機能し、迅速かつ予測可能に機能します。それが今のところ、これで終わりです。

マルチスレッドプログラミング


言語に固有でコンパイラに組み込まれている所有機能と借用機能により、ロックの設定と削除の手順に問題はなく、マルチスレッドプログラミングはなんらかの理由で簡単で、驚くことはありません。何かが間違っている場合、コードは単にコンパイルされません。 Sync / Sendトレイトの詳細については説明しません。ライブで確認する必要がありますが、複雑で神秘的なことは何も起こりません。ロジックを実装し、コンパイラーが誓ったら、修正してください。それだけです。

ヒープとスタック


デフォルトでは、構造体とオブジェクトのメモリはスタックに割り当てられますが、割り当てるのは非常に簡単で、ヒープ内で自動的に解放されます。すでに上で説明したように、その所有者（変数を参照）が「alaスコープ」（より正確には「寿命」）を離れると、メモリは自動的に解放されます。また、本当に必要な場合は、さまざまなタイプのスマートポインターも用意されています（hello Swift）。上記が複雑に思える場合は、そうではありません。メモリをヒープに割り当てるコードを記述し、デストラクタでメモリを解放することに関するメッセージを出力すると、すべてが明確に理解できるようになります。トリックはありません-メモリは常に解放され、それをチェックするのはプログラマではありませんが、コンパイラが保証します。

プロジェクトの詳細を少し明らかにします。 Amazon SQSは、毎秒数百のデータパケットをスピルします。キューが読み込まれ、ワーカーによってローカルで解析されます。各メッセージはブローカーによって処理され、別の外部サーバーにリダイレクトされます。複数の外部サーバーがあります。当初、ソリューションはスクリプトによって実装されました。スクリプト、オペレーティングシステムでのプロセスの実行、メッセージの読み取り、ネットワーク経由での処理と送信です。いくつかの強力な鉄のサーバー（8コア、16 GBのRAM）で、何百ものスクリプトが（それぞれに！）起動され、SQSからの読み取り、データの処理と送信が同時に行われました。シンプルで信頼できるが、鉄の摂取量が気になり始めた。鉄のコストは常に増加していました。

一般的な印象は、Rustでの開発は、システムプログラミングではなく、「厳密なスクリプト」に似ており、リソースとテストの両方の点で、スクリプトまたはWeb開発よりも長くはないということです。この場合、厳密な静的コンパイル、ガベージコレクターがないこと、および代数的にデータ型。

全体的な感じはとてもポジティブです。それでも、いくつかの鉄のサーバー（8コア、16 GBのRAM）の代わりに、1つのプロセス（数十のスレッド）で5 GBを超えるRAMを消費せず、コアにあまり目立たない負荷が発生し、トラフィックが0.5〜1ギガビットの領域で問題を解決するようになりました。 。
