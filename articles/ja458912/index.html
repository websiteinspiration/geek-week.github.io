imapsyncを使用してZimbraに移行する

階層型アドレス帳やUbuntu 18.10 LTSの公式サポートなどの待望の機能に加えて、Zimbra 8.8.15のリリースにより、他のメールシステムからZimbraコラボレーションスイートへの移行に必要な多数のユーティリティのサポートが削除されるというニュースが伝えられました。特に、DominoのZCS移行ウィザードのサポートは完全に廃止され、ExchangeのZCS移行ウィザードはOutlookからのPSTファイルのインポートのみをサポートするようになりました。本格的な移行を実装するために、Zimbra Collaboration Suiteの開発者は有料のオンラインサービスを利用することを提案していますが、すべての企業がセキュリティ上の理由からサードパーティにメールボックスデータを提供する準備ができているわけではありません。特にそのようなユーザーのために、メールシステムからZimbraへの信頼できる増分移行を提供できるimapsyncプログラムがあります。



Imapsyncは、公式にサポートされているZimbra Linuxディストリビューションのリポジトリで利用できる無料の無料ユーティリティです。このプログラムの目的は、異なるメールサーバー間で電子メールを転送することであり、完全に対応します。設定の柔軟性のおかげで、メールは段階的かつ完全自動モードで転送できます。imapsyncを使用してLotus DominoおよびMS ExchangeからZimbra Collaboration Suiteに移行する方法を見てみましょう。

MS ExchangeからZimbraへの移行


メールの移行を成功させるには、一連の準備を完了する必要があります。まず、Zimbraサーバー上に、MS Exchangeサーバー上と同じユーザーを同じパスワードで作成する必要があります。このタスクを実行する際、ADを使用したZimbraユーザーの自動チューニングは非常に役立ちます。そのような自動チューニングのプロセスは、以前の記事の1つで説明されています。

ADまたはExchangeのすべてのアカウントが移行されたら、Zimbraを使用してサーバーにimapsyncパッケージをインストールする必要があります。 ZimbraのOSとしてUbuntuを使用している場合は、コマンドsudo apt-get install imapsync -yを使用してプログラムをインストールできます。CentOSおよびRHELでは、インストールは次のコマンドを使用して実行されます。sudo yum install imapsync -y。

パッケージとすべての依存関係のインストールが完了したら、Exchangeサーバーの構成に進みましょう。その上で、IMAP4サービスを有効にする必要があります。これを行うには、services.mscファイルを実行します。その後、[ツール]> [サービス]-> [IMAP4 Microsoft Exchange]-> [開始]に移動する必要があります。その後、移行中にユーザーパスワードを使用しないようにするために、ユーザーメールボックスへの管理者アクセス権を付与する必要があります。また、速度制限ポリシーを無効にするか構成して、移行の障害にならないようにします。これで、サーバーからサーバーへのメールの転送を開始できます。

私たちのインフラストラクチャでは、MS Exchangeはexchange.company.ruでホストされます、およびZimbraはzimbra.company.ruによってホストされます。コマンドラインを使用してユーザーが1人ずつ転送されます。移行に必要ないくつかのパラメーターを指定してimapsyncユーティリティを実行します。まず、MS Exchangeと同様に、Zimbraでは、管理者アカウントがすべてのユーザーメールボックスの内容にアクセスできることに注意してください。つまり、システム管理者は、新しいメールサーバーに移行するために、通常のユーザーアカウントのパスワードを知っている必要はありません。そのため、MS ExchangeからZimbraへの各ユーザーの移行は、次の形式のコマンドを使用して実行されます。

imapsync --host1 exchange.company.ru --user1 "xxxxx" --authuser1 "yyyyy" --password1 "zzzzz" --nosslcheck --host2 zimbra.company.ru --user2 "XXXXX"  --authuser1 "YYYYY" --password2 "ZZZZZ"

このコマンドでは、imapsyncがアカウントデータを取得するソースホスト（--host1）と、このデータが書き込まれるホスト（--host2）を指定します。また、ユーザ名（指定--user1を最初にホストとそれに対応するユーザ名（上）--user2第二のホスト上）。そのデータにアクセスするために、最初のホストの管理者アカウントの名前（--authuser1）と管理者パスワード（--password1）を指定し、送信されたデータを記録するために、管理者アカウントの名前（--authuser2）とパスワードを指定します2番目のホストの管理者（--password2）。

理想的な世界では、このようなコマンドでMS ExchangeとZimbraを備えたサーバー間でユーザーデータを転送するのに十分ですが、よくあることですが、ソリューション間の違いにより独自の修正が行われるため、MS Exchangeからの単純なデータ転送はZimbraメールボックスでひどい混乱を招きます。この場合の障害は、メールボックス内のフォルダーの異なる名前です。データ転送中の混乱を避けるために、私たちのチームをおおよそ次の形式にする必要があります。

imapsync --subscribe_all --maxmessagespersecond 4 --addheader --f1f2 "Inbox=INBOX" \
--f1f2 "Sent Items=Sent" --f1f2 "Sent items=Sent" --exclude "^Journal|^Tasks|^Trash|Contacts|Calendar|RSS Feeds" \
--useheader "Message-ID" --buffersize 8192000 --nosyncacls  --allowsizemismatch --usecache --syncinternaldates --skipsize --nofoldersizes \
--tls1 --port1 143 --user1 "xxxxxxxxxx" --authuser1 "besadmin" --password1 "XXXXXXX" \
--host2 'xxxxxxx' --user2 "xxxxxx" --authuser2 "XXXXXX" --password2 "XXXX" --tls2 --port2 7143 \
--regextrans2 "s/Inbox1/INBOX/i" --regextrans2 "s/Sent Items/Sent/i" --regextrans2 "s/sent-mail/Sent/i" --regextrans2 "s/Deleted Items/Trash/i" --regextrans2 "s/Deleted Messages/Trash/" \
--delete2duplicates --timeout 600 --pidfile "impasync.local.pid" --logfile "imapsync.log" --sep1 "/" --errorsmax "100"

これは、どのインフラストラクチャでも問題なく機能するユニバーサルスクリプトではないことに注意してください。その中の多くの事柄は、インフラストラクチャのさまざまな機能に依存する可能性があります。そのため、本格的な移行を開始する前に、重要でないビジネスアカウントのシステム間でのデータ転送を必ずテストしてください。

imapsyncの起動に使用されるパラメーターが豊富であっても、怖がらないと仮定します。メールボックスからフォルダーを転送するときに混乱がなく、MS Exchangeのすべての要素がZimbraに転送されるように、それらのほとんどが正確に必要です。たとえば、-subscribe_all オプションを使用すると、imapsyncはアカウント内のすべてのフォルダーを取得できるため、後でそれらを操作できます。オプション--f1f2同期中にフォルダの名前を変更できます。つまり、たとえば同期する場合です。送信済みアイテムフォルダーの名前が[送信済み]に変更され、その内容をZimbra Collaboration Suiteのフォルダー構造に完全に合わせることができます。--regextrans2パラメータを使用すると、フォルダ名を変更することもできます。これは、Perlプログラミング言語の正規表現メカニズムに基づいているため、この言語に精通している人はこのパラメータを簡単に学習できます。--addheaderおよび--useheaderオプション

の機能を説明するため、imapsyncで文字の同期がどのように正確に行われるかを詳しく調べる必要があります。実際、imapsyncを同期するには、両方のサーバーでレターを識別して、同期するかどうかを決定する必要があります。それらを検証するために、このユーティリティは、下書きなどの電子メールにはない「Message-Id：」や「Received：」（識別番号と受信時刻）などのパラメーターを使用します。それらを正常に同期するには、-addheaderコマンドが必要です。同期中にこのような文字を識別して比較するために、ID番号を追加することができます。 --useheaderパラメーターを使用すると、同期中に2つのパラメーターのいずれか1つのみを使用して文字を比較できます。

MS ExchangeからZimbraへのアカウントの完全な転送を可能にする完璧なスクリプトを記述した後、サーバー間でユーザーを一度に1つずつ転送する必要があるという事実に関連する重大な問題が残ります。これは、特に従業員が多く、メールボックスの数が数千単位で測定される企業では、あまり最適ではありません。ただし、アカウントのリストを作成し、アカウント名に基づいてメールを自動的に転送する小さなスクリプトを作成することで、このプロセスを大幅に最適化できます。これを行うには、imap_usersファイルを作成し、次の形式で必要なデータをすべて追加します。

domains=(
domain1\
domain2)

users=(
user1@domain1\
user2@domain1\
user3@domain1\
user1@domain2\
user2@domain2\
user3@domain2)

その後、スクリプト自体を記述します。私たちはこのようなものを得ました：

. imap_users
src_srv = exchange.company.ru
dest_srv = zimbra.company.ru
for ((i = 0; i < ${#users[@]}; i++ ))
do
/usr/bin/imapsync --subscribe_all --maxmessagespersecond 4 --addheader --f1f2 "Inbox=INBOX" \
--f1f2 "Sent Items=Sent" --f1f2 "Sent items=Sent" --exclude "^Journal|^Tasks|^Trash|Contacts|Calendar|RSS Feeds" \
--useheader "Message-ID" --buffersize 8192000 --nosyncacls  --allowsizemismatch --usecache --syncinternaldates --skipsize --nofoldersizes \
--tls1 --port1 143 --host1 $src_srv --user1 ${users[$i]} --authuser1 "besadmin" --password1 "XXXXXXX" --nosslcheck \
--host2 $dest_srv --user2 ${users[$i]} --authuser2 "XXXXXX" --password2 "XXXX" --tls2 --port2 7143 \
--regextrans2 "s/Inbox1/INBOX/i" --regextrans2 "s/Sent Items/Sent/i" --regextrans2 "s/sent-mail/Sent/i" --regextrans2 "s/Deleted Items/Trash/i" --regextrans2 "s/Deleted Messages/Trash/" \ 
--delete2duplicates --timeout 600 --pidfile "impasync.local.pid" --logfile "imapsync.log" --sep1 "/" --errorsmax "100"
done

データ転送はインクリメンタルなので、最初にサーバー間でデータの大部分を転送し、次に短い技術的な中断内で、最後の同期以降に蓄積された少量のデータを転送し、すべての受信メールをZimbraに転送して、メインのメールシステムにすることができます。新しいメールシステムに移行する前に、今後の移行について従業員に通知し、トレーニングを実施するとともに、Zimbraの通常の機能がどこにあるかについてのメモを準備する必要があることを思い出してください。これにより、移行がよりスムーズに行われ、ビジネスに問題が生じることはありません。

Lotus / DominoからZimbraへの移行


多くの点で、imapsyncを使用したIBMからのメールシステムからの移行プロセスは、MS Exchangeおよびその他のメールシステムからの移行プロセスを繰り返します。ただし、直接移行のプロセスでは、多くの管理者がIMAPを介した切断とそれに伴う同期の終了のエラーを受け取ります。同時に、ログにエラーメッセージは表示されません。Lotus / Dominoからの移行中にこの問題が発生した場合、次のアクションがこの問題の解決に役立ちます。

  • 管理者権限でLotus Notesを起動し、アドレス帳に移動します。
  • アドレス帳の設定で、デフォルトのメッセージ形式を「優先MIME」に変更します
  • 管理者権限で実行されている同じLotus Notesで、ユーザーデータベースの形式をメール（R6）に変更します。これは、メールクライアントからではなく、サーバーから取得されます。
  • Lotus Domino Router Domino «tell router quit»
  • Domino «load compact userdb.nsf»
  • , Domino ( ) «fixup userdb.nsf»
  • Domino «convert -e userdb.nsf», imap
  • Domino «convert -h userdb.nsf» imap- .

これらすべての手順を完了すると、imapsyncを使用してLotus / DominoからZimbraにメールを転送しても問題なく動作するはずです。

移行用のユーティリティに加えて、デスクトップコンピューターおよびモバイルデバイス用のZimbraクライアントはサポートされずに残されました。また、HTML Webクライアントはサポートから削除されました。これは、古いブラウザーや低電力デバイスとの互換性を確保するために使用されていました。 Zimbraはほとんどすべてのオープンな標準とプロトコルをサポートしているため、サードパーティのお客様はそれらすべてを簡単に置き換えることができます。
