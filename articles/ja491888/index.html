いいね？しかし、よく見ると何かが目に留まります…なぜ生産を変えただけvariables.CLUSTERでなく、2度目の処方も変えたのvariables.MY_VAR=10ですか？この変数を取得する必要がありbase_deployますか？ YAMLが機能するため、アンカーから受け取ったものを再定義し、一致するフィールドの内容を拡張せずに置き換えます。したがって、一致する段落ですでにわかっている変数をリストする必要があります。

はい、「エキスパンド」は正しい言葉です。これがまさに問題の機能が呼び出されるものです。Extendsアンカーで発生するようにフィールドを書き換えるだけでなく、スマートマージを実行できます。

.base_deploy: 
  stage: deploy
  script:
    - my_deploy_command.sh
  variables:
    CLUSTER: "default-cluster"
    MY_VAR: "10"

Deploy Production:
  extends: .base_deploy
  environment:
    url: production.example.com
    name: production
  variables:
    CLUSTER: "prod-cluster"

ここで、最後のジョブDeploy ProductionにはMY_VAR、デフォルト値を持つ変数とオーバーライドされた変数の両方がありますCLUSTER。

これは簡単なことのようですが、想像してみてください。1つとbase_deploy20の回路が同じように配置されています。特定の変数のセットまたは他の一致するフィールドを保持しながらcluster、それらを他の人に渡す必要がありenvironment.nameます...この小さな快適さにより、多くの開発回路の展開の説明を2〜3倍に減らすことができました。

番号4：参照のみ/参照以外


  • 変数やファイルの変更を含む包括的な条件。
  • これは関数のファミリ全体であるため、一部の部分はGitLab 10.0に登場し、他の部分（たとえばchanges）は11.4 に登場し始めました。

時々、これは私たちを聞くパイプラインではないようですが、私たち彼です。優れた管理ツールがonly/ except統合されました。これは何を意味するのでしょうか？

最も単純な（そしておそらく最も楽しい）ケースでは、ステージをスキップします。

Tests:
  only:
    - master
  except:
    refs:
    - schedules
    - triggers
    variables:
    - $CI_COMMIT_MESSAGE =~ /skip tests/

サンプルジョブでは、マスターブランチでのみ実行されますが、スケジュールまたはトリガーによってトリガーすることはできません（GitLabはAPI呼び出しとトリガーを共有しますが、これは基本的に同じAPIです）。コミットメッセージにスキップテストのパスフレーズがある場合、ジョブは実行されません。たとえば、README.mdプロジェクトまたはドキュメントのタイプミスが修正されました-なぜテスト結果を待つのですか？

「ねえ、2020は外にある！」ドキュメントを変更するときにテストを実行する必要がないことを毎回アイアンボックスに説明する必要があるのはなぜですか？」そして本当に：only:changesそれはあなたが特定のディレクトリでのみファイルを変更するときにテストを実行することを可能にします。例えば：

  only:
    refs:
      - master
      - merge_requests
    changes:
      - "front/**/*"
      - "jest.config.js"
      - "package.json"

そして逆のアクションのために-つまり 実行しない-はいexcept:changes。

＃6：環境：auto_stop_in


  • 活動のない環境を停止します。
  • GitLab 12.8で導入されました。
  • docs.gitlab.com/ce/ci/yaml/#environmentauto_stop_in

記事の公開直前にこの機会を知り、実際にテストする時間はありませんでしたが、これはいくつかのプロジェクトで期待されていた「同じこと」です。

GitLab環境でパラメーターを指定できますon_stop。たとえば、各ブランチに対して動的に環境を作成および削除する場合に非常に便利です。 k on_stopでマークされたジョブは、たとえば、MRのマージがマスターブランチにあるとき、またはMRが閉じられたとき（またはボタンをクリックするだけで）に実行され、不要な環境が自動的に削除されます。

すべてが便利で、論理的で、うまくいきます...人的要因がなければ。多くの開発者は、GitLabのボタンをクリックするのではなく、ローカルでMRをマージしますgit merge。あなたはそれらを理解することができます：それは便利です！しかし、この場合、ロジックon_stop機能しません。忘れられた環境が蓄積されています...待望の便利な場所auto_stop_inです。

ボーナス：十分な機会がない場合の一時的な小屋


これらすべて（および他の多く）のGitLabの新しい、要求された機能にもかかわらず、残念ながら、ジョブを実行するための条件を、現在利用可能な機能のフレームワーク内で説明することは単に不可能です。

GitLabは完璧ではありませんが、夢のパイプラインを構築するための基本的なツールを提供します。ささやかなDSLを超えて、スクリプトの世界に飛び込む準備ができている場合。以下に、私たちの経験からのいくつかのソリューションを示します。これらは、イデオロギー的に正しいまたは推奨されるふりをするものではありませんが、組み込みAPI機能の欠如によるさまざまな可能性を示すために、より多く提示されています。

回避策2：イメージ内のrubocopのMR rb-filesで転送が変更されました


Rubocop:
  stage: test
  allow_failure: false
  script:
    ...
    - export VARFILE=$(mktemp)
    - export MASTERCOMMIT=$(git merge-base origin/master HEAD)
    - echo -ne 'CHANGED_FILES=' > ${VARFILE}
    - if [ $(git --no-pager diff --name-only ${MASTERCOMMIT} | grep '.rb$' | wc -w |awk '{print $1}') -gt 0 ]; then
        git --no-pager diff --name-only ${MASTERCOMMIT} | grep '.rb$' |tr '\n' ' ' >> ${VARFILE} ;
      fi
    - if [ $(wc -w ${VARFILE} | awk '{print $1}') -gt 1 ]; then
        werf --stages-storage :local run rails-dev --docker-options="--rm --user app --env-file=${VARFILE}" -- bash -c /scripts/rubocop.sh ;
      fi
    - rm ${VARFILE}

内部.gitには画像がないため、変更されたファイルのみをチェックするために外に出なければなりませんでした。

注：これは非常に標準的な状況ではなく、問題の多くの条件に準拠するための必死の試みであり、その説明はこの記事の範囲には含まれていません。

回避策＃3：ロールアウト時にトリガーして他のリポジトリからジョブを開始する


  before_script:
    - |
      echo '### Trigger review: infra'
      curl -s -X POST \
        -F "token=$REVIEW_TOKEN_INFRA" \
        -F "ref=master" \
        -F "variables[REVIEW_NS]=$CI_ENVIRONMENT_SLUG" \

このような単純で必要な（マイクロサービスの世界では）事柄が、依存関係として別のマイクロサービスを新しく作成された回路に展開しているように見えます。ただし、これはAPI呼び出しではなく、既におなじみの（上記の）機能が必要です。

  only:
    refs:
    - triggers
    variables:
    - $ACTION == "auto_review_start"
