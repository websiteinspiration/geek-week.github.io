Webコンポーネント角度コンポーネントライブラリを構築する

Angular Elementsに関する多くの記事が書かれており、定期的にレポートを読んでいます。同様に、いくつかの本格的なアングルを配置する必要がなくなりました-Webコンポーネントを収集してページで使用するだけです。

ただし、原則として、これらの資料はかなりユートピア的な状況を考慮したものに限定されています。個別のプロジェクトを作成し、角度コンポーネントを作成し、プロジェクトを構成して要素をアセンブルし、最後に複数のJSファイルをコンパイルして、それらを通常のページに接続すると、望ましい結果が得られます。コンポーネントが機能します。



実際には、できれば現在の開発と使用に影響を与えないように、既製の作業用角度プロジェクトからいくつかのコンポーネントを引き出す必要が生じます。この記事は、これらの状況の1つが原因で判明しました。プロジェクトの個々の要素を組み立てるだけでなく、AngularのUIライブラリ全体をネイティブWebコンポーネントを含む一連のファイルにコンパイルするプロセスを作りたかったのです。

モジュールの準備


まず、Angular Elementsをコンパイルするためのモジュールがどのようになるかを思い出してみましょう。

@NgModule({
   imports: [BrowserModule],
   entryComponents: [SomeComponent],
})
export class AppModule {
   constructor(readonly injector: Injector) {
       const ngElement = createCustomElement(SomeComponent, {
           injector,
       });

       customElements.define('some-component', ngElement);
   }

   ngDoBootstrap() {}
}

するべき：

  1. 角度要素を作成する予定のentryComponentsコンポーネントに追加し、コンポーネントに必要なモジュールをインポートします。
  2. createCustomElementとインジェクターを使用して角度要素を作成します。
  3. ブラウザーのcustomElementsでWebコンポーネントを宣言します。
  4. ngDoBootstrapメソッドをオーバーライドして空にします。

最初の項目はコンポーネント自体とその依存関係の指定であり、他の3つはWebコンポーネントがブラウザーに表示されるために必要なプロセスです。この分離により、要素を作成するロジックを抽象スーパークラスに個別に配置できます。

export abstract class MyElementModule {
   constructor(injector: Injector, component: InstanceType<any>, name: string) {
       const ngElement = createCustomElement(component, {
           injector,
       });

       customElements.define(`${MY_PREFIX}-${name}`, ngElement);
   }

   ngDoBootstrap() {}
}

スーパークラスは、インジェクター、コンポーネント、名前から本格的なネイティブコンポーネントを組み立てて登録できます。特定の要素を作成するためのモジュールは、次のようになります。

@NgModule({
   imports: [BrowserModule, MyButtonModule],
   entryComponents: [MyButtonComponent],
})
export class ButtonModule extends MyElementModule {
   constructor(injector: Injector) {
       super(injector, MyButtonComponent, 'button');
   }
}

この例では、ボタンのモジュールをNgModuleメタデータに収集し、entryComponentsでこのモジュールからコンポーネントを宣言し、Angular依存関係注入メカニズムからインジェクターを取得します。

モジュールは組み立ての準備ができており、別個のWebコンポーネントに折りたたむことができる一連のJSファイルを提供します。この方法で、いくつかのモジュールを作成し、それらから順番にWebコンポーネントを組み立てることができます。

いくつかのコンポーネントをまとめる


次に、結果のモジュールのブートストラッププロセスを設定する必要があります。何よりも、このロジックを個別の実行可能ファイルに入れて、特定のモジュールをコンパイルするという考えが気に入っています。

要素の構造は次のように



なります。最も単純なバージョンの個別のコンパイルファイルは次のようになります。

enableProdMode();

platformBrowserDynamic()
   .bootstrapModule(ButtonModule)
   .catch(err => console.error(err));

このアプローチは、準備されたモジュールを簡単にバイパスし、要素を明確かつ単純にしてプロジェクト構造を維持するのに役立ちます。

angular.jsonビルド設定で、収集されたファイルのdist内の一時フォルダーへのパスを指定します。

"outputPath": "projects/elements/dist/tmp"

モジュールのアセンブリ後、一連の出力ファイルが作成されます。

アセンブリ自体については、angular-cliで通常のビルドコマンドを使用します。

ng run elements:build:production --main='projects/elements/src/${project}/${component}/compile.ts'

別の要素が最終製品になるので、Ahead-of-Time-compilationで製品フラグをオンにし、その後、プロジェクトとコンポーネントの名前で構成される実行可能ファイルへのパスを置き換えます。

次に、結果を個別のファイルに収集します。これは、個別のWebコンポーネントの最終的なバンドルになります。これを行うには、通常の猫を使用します。

cat dist/tmp/runtime.js dist/tmp/main.js > dist/tmp/my-${component}.js

今後、同じページで複数のコンポーネントを使用すると重複が発生するため、各コンポーネントのバンドルにpolyfills.jsファイルを配置しないことに注意してください。もちろん、angular.jsonのoutputHashingオプションを無効にする必要があります。

作成されたバンドルを一時フォルダーからコンポーネントを格納するフォルダーに転送します。たとえば、次のようになります。

cp dist/tmp/my-${component}.js dist/components/

すべてをまとめるだけで残り、コンパイルスクリプトの準備が整います。

// compileComponents.js

projects.forEach(project => {
   const components = fs.readdirSync(`src/${project}`);

   components.forEach(component => compileComponent(project, component));
});

function compileComponent(project, component) {
   const buildJsFiles = `ng run elements:build:production --aot --main='projects/elements/src/${project}/${component}/compile.ts'`;
   const bundleIntoSingleFile = `cat dist/tmp/runtime.js dist/tmp/main.js > dist/tmp/my-${component}.js`;
   const copyBundledComponent = `cp dist/tmp/my-${component}.js dist/components/`;

   execSync(`${buildJsFiles} && ${bundleIntoSingleFile} && ${copyBundledComponent}`);
}

これで、一連のWebコンポーネントを持つきちんとしたお父さんができました。



コンポーネントを通常のページに接続する


組み立てられたWebコンポーネントはページに個別に挿入でき、必要に応じてJSバンドルを接続します。

<my-elements-input id="input"> </<my-elements-input>

<script src="my-input.js"></script>

zone.js全体を各コンポーネントと一緒にドラッグしないようにするために、ドキュメントの最初に一度接続します。

<script src="zone.min.js"></script>

コンポーネントがページに表示され、すべて正常です。

そして、別のボタンを追加しましょう：

<my-elements-button size="l" onclick="onClick()"></my-elements-button>

<script src="my-button.js"></script>

私たちはページを開始します...



ああ、すべてが壊れました！

バンドルを調べると、そのような目立たない行があることがわかります。

window.webpackJsonp=window.webpackJsonp||[]



同じモジュールのロードを重複しないように、Webpackパッチウィンドウ。ページに追加された最初のコンポーネントだけが、それ自体をcustomElementsに追加できることがわかりました。

この問題を解決するには、カスタムWebパックを使用する必要があります。

  1. 要素を含むカスタムWebパックをプロジェクトに追加します。

    ng add @angular-builders/custom-webpack --project=elements

  2. angular.jsonの構成：

    "builder": "@angular-builders/custom-webpack:browser",
    "options": {
       "customWebpackConfig": {
           "path": "./projects/elements/elements-webpack.config.js"
       },
...

  3. カスタムWebpack構成ファイルを作成します。

    module.exports = {
   output: {
       jsonpFunction: 'myElements-' + uuidv1(),
       library: 'elements',
   },
};

    その中で、便利な方法で各アセンブリに一意のIDを生成する必要があります。私はuuidを利用しました。

ビルドスクリプトを再度実行できます。新しいコンポーネントは同じページでうまく動作します。

美しさをもたらす


コンポーネントは、コンポーネントのカラーテーマとサイズを指定するグローバルCSS変数を使用します。

ライブラリを使用する角度のあるアプリケーションでは、それらはプロジェクトのルートコンポーネントに配置されます。独立したWebコンポーネントではこれは不可能なので、スタイルをコンパイルして、Webコンポーネントが使用されているページに接続するだけです。

// compileHelpers.js
compileMainTheme();

function compileMainTheme() {
   const pathFrom = `../../main-project/styles/themes`;
   const pathTo = `dist/helpers`;

   execSync(
       `lessc ${pathFrom}/theme-default-vars.less ${pathTo}/main-theme.css`,;
   );
}

lessを使用するので、lessc変数をコンパイルし、結果のファイルをヘルパーフォルダーに配置します。

このアプローチにより、ページのすべてのWebコンポーネントのスタイルを制御することができ、それらを再コンパイルする必要はありません。

最終スクリプト


実際、上記の要素を組み立てるプロセス全体を、一連のアクションに減らすことができます。

#!/bin/sh

rm -r -f dist/ &&
mkdir -p dist/components &&
node compileElements.js &&
node compileHelpers.js &&
rm -r -f dist/tmp

メインのpackage.jsonからこのスクリプトを呼び出して、実際の角度コンポーネントをコンパイルするプロセス全体を1つのコマンドの起動に減らすだけです。

上記のすべてのスクリプト、および角度コンポーネントとネイティブコンポーネントを使用するためのデモは、githubにあります。

合計


新しいWebコンポーネントを追加するための主要な角度のあるプロジェクトの構造を維持しながら、新しいWebコンポーネントを追加するのに数分かかるプロセスを編成しました。

すべての開発者は、一連の要素にコンポーネントを追加し、それらをAngular Elementsでの作業の詳細に踏み込むことなく、Webコンポーネントの別個のJSバンドルのセットにアセンブルできます。
