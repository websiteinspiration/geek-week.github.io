GameDev TDDまたはRabbit Hell

TDDがゲーム開発で使用されることはほとんどありません。通常、テストを作成する開発者を確保するよりも、テスターを雇う方が簡単です。これにより、リソースと時間の両方が節約されます。したがって、TDDの使用が成功するたびに、より興味深いものになります。カットの下では、この開発技法がゲームElemenTerraでキャラクターの動きを作成するために使用された素材の翻訳。



テスト駆動開発またはTDD（テストによる開発）は、プロセス全体が多数の小さなサイクルに分割されるソフトウェア開発手法です。ユニットテストが記述され、次にこれらのテストに合格するコードが記述され、リファクタリングが行われます。そしてアルゴリズムが繰り返されます。

TDDの基本


2つの数値を加算する関数を作成するとします。通常のワークフローでは、それを書くだけです。ただし、TDDを使用するには、まずプレースホルダー関数と単体テストを作成する必要があります。

// Placeholder-,    :
int add(int a, int b){
 return -1;
}
// Unit-,   ,  add    :
void runTests(){
 if (add(1, 1) is not equal to 2)
   throw error;
 if (add(2, 2) is not equal to 4)
   throw error;
}

最初は、プレースホルダー関数が入力ごとに-1を返すため、単体テストは機能しません。これで、addを正しく実行してa + bを返すことができます。テストに合格します。これは回避策のように思えるかもしれませんが、いくつかの利点があり

ます。誤ってaddをa-bとして追加すると、テストが失敗し、関数の修正方法がすぐにわかります。テストがないと、このエラーをキャッチできず、デバッグに時間がかかる非標準の反応を確認できません。
コードを書いている間、いつでもテストを継続して実行できます。これは、別のプログラマが誤ってaddを変更した場合、すぐにエラーを認識し、テストが再び失敗することを意味します。

ゲーム開発におけるTDD


ゲーム開発者のTDDには2つの問題があります。まず、多くのゲーム機能には、測定できない主観的な目標があります。第二に、複雑な相互作用オブジェクトで満ちている世界の空間のすべての可能性をカバーするテストを書くことは困難です。キャラクターの動きを「見栄え」にしたり、物理シミュレーションを「ぎくしゃくさせない」ようにしたい開発者は、これらの測定基準を決定論的な「合格/不合格」条件として表現するのが難しいことに気づくでしょう。

ただし、TDD手法は、キャラクターの動きなど、複雑で主観的な機能に適用できます。そして、ゲームElemenTerraでそれを行いました。

デバッグレベルに対する単体テスト


練習を始める前に、自動ユニットテストと従来の「デバッグレベル」を区別したいと思います。人工的な条件で隠れた場所を作成することは、gamedevでは一般的なことです。これにより、プログラマとQAは個々のイベントを監視できます。


Legend of Zelda：The Wind Waker

ElemenTerraのシークレットデバッグレベルには、そのようなレベルがたくさんあります。プレイヤーのキャラクターの問題のあるジオメトリでいっぱいのレベル、特定のゲームステートなどをトリガーする特別なユーザーインターフェイスを持つレベルです。

単体テストと同様に、これらのデバッグレベルはエラーの再現と診断に使用できます。ただし、いくつかの点で異なります。

ユニットテストは、システムをパーツに分割し、それぞれを個別に評価する一方で、デバッグレベルは、より包括的な方法でテストを実行します。デバッグレベルでエラーを見つけた後も、開発者は手動でエラーポイントを見つける必要がある場合があります。
ユニットテストは自動化されており、常に確定的な結果を提供する必要がありますが、多くのデバッグレベルはプレーヤーによって「制御」されます。これにより、セッションに違いが生じます。

ただし、これは単体テストがデバッグレベルよりも優れていることを意味するものではありません。後者の方が実用的です。ただし、単体テストは、従来は存在していなかったシステムでも使用できます。

ElemenTerraでは、プレイヤーは自然の神秘的な力を利用して、宇宙嵐の影響を受けた生き物を救います。そのような力の1つは、生き物を食べ物や避難所に導く道を開く能力です。これらのパスはプレーヤーによって作成された動的なグリッドであるため、クリーチャーの動きは、異常な幾何学的なケースと任意に複雑な地形を処理する必要があります。

キャラクターの動きは、「すべてが他のすべてに影響を与える」複雑なシステムの1つです。これを行ったことがある場合は、新しいコードを作成するときに、既存の機能を破壊するのは非常に簡単です。小さな突起物を登るにはウサギが必要ですか？さて、しかし今、彼らは斜面を登って、けいれんしています。トカゲの道が交差しないようにしますか？それは機能しましたが、今、彼らの典型的な振る舞いは台無しにされています。

AIとほとんどのゲームプレイコードの責任者として、私は突然のエラーのための時間がないことを知っていました。私はすぐにリグレッションに気づきたかったので、TDDを使用した開発は私にとって良いオプションのように思えました。

私の実装と「クリーン」TDDの類似点と相違点について話しましょう。

私のシステムはこの点でTDDに似ていました：

  • テストを作成することで関数の作業を開始し、それらを実行するために必要なコードを記述しました。
  • 私は古いテストを続け、新しい機能を追加しました。
  • 各テストでシステムの一部を正確に測定したため、問題をすばやく見つけることができました。
  • テストは自動化されており、プレーヤーの入力は必要ありませんでした。

そして、これは異なっていました：

  • テストを評価するとき、主観の要素がありました。移動の実際の間違い（キャラクターがAからBに移行しなかった）は、プログラムで検出できます。つまり、たとえば、スキューの位置、アニメーションの同期の問題、およびけいれんの動きには、人間による評価が必要でした。
  • テストは完全に確定的ではありませんでした。フレームレートの変動などのランダムな要因により、わずかなずれが発生しました。しかし、一般的に、クリーチャーは通常、同じパスをたどり、セッション間で同じ成功/失敗をします。

TDD — ?


開発者は、プレーヤーが理解できないほどのユニットテストレベルに多くの労力を費やす場合があります。私はそれを否定しません、私自身は「ウサギの地獄」を作成することから多くの喜びを受け取りました。そのような内部機能は時間がかかり、より重要なマイルストーンを危険にさらす可能性があります。これが起こらないようにするには、ユニットテストを使用する場所とタイミングを慎重に検討してください。以下では、エレメンテラクリーチャーの移動についてTDDを正当化するいくつかの基準を強調しました。

1.テストタスクを手動で完了するのに長い時間がかかりますか？

自動テストに時間をかける前に、従来のゲームコントロールを使用して機能を評価できるかどうかを確認する必要があります。キーでドアのロックを解除したい場合は、キーを生成してドアを開けてください。この関数の単体テストを作成するのは時間の無駄です-手動テストは数秒しかかかりません。

2.テストケースを手動で作成することは困難ですか？

自動化された単体テストは、既知で再現が難しいケースがある場合に正当化されます。 「ウサギの地獄」のテストNo. 7は、レッジに沿ってどのように歩くかをチェックします。これは、AIが通常​​避けようとすることです。このような状況は、ゲームコントロールを使用して再現することが困難または不可能な場合があり、テストは簡単です。

3.望ましい結果が変わらないことを知っていますか？

ゲームのデザインは完全に反復に基づいているため、ゲームがリメイクされると機能の目標が変わる可能性があります。小さな変更でも、機能を評価するためのメトリック、つまりユニットテストを無効にする可能性があります。食事中、睡眠中、プレイヤーとのインタラクション中の生き物の行動が数回変化した場合、ポイントAからポイントBへの遷移は変化しませんでした。したがって、ムーブメントコードとその単体テストは、開発全体を通じて適切なままでした。

4.リグレッションは気付かれない可能性がありますか？

ゲームを送信する前に最後のタスクの1つを完了し、突然ルールに違反するエラーを見つけたという状況はありましたか？そして、あなたが何年も前に終えた機能で。ゲームは相互接続された巨大なシステムであるため、新しい関数Bを追加すると古い関数Aが失敗する可能性があるのは当然です。

壊れた関数がどこでも使用されている場合（ジャンプなど）はそれほど悪くありません。すぐにメカニズムの故障に気付くはずです。後の開発で発見されたエラーはスケジュールを混乱させる可能性があり、起動後にゲームプレイに害を及ぼす可能性があります。

5.テストを使用した場合と使用しない場合に発生する可能性のある最悪の事態？

テストの作成は、リスク管理の1つの形式です。自動車保険を購入するかどうかを決定するとします。次の3つの質問に答える必要があります。

  • 月々の保険料はいくらですか？
  • 車が損傷する可能性はどのくらいありますか？
  • 保険がかけられていない場合、最悪のケースのシナリオはどのくらい高価になりますか？

TDDの場合、ユニットテストのサービスを提供するための生産コスト、バグが発生する確率としての車の損傷の確率、回帰エラーの最悪のシナリオとしての完全な車の交換という形で、毎月の貢献を想像できます。

機能テストの作成に時間がかかる場合、それは単純で変更される可能性が低い（または、後の開発で中断した場合に対処できる）場合、単体テストは問題よりも多くの問題を引き起こす可能性があります。テストが簡単で、機能が不安定で相互接続されている場合（または、エラーに時間がかかる場合）、テストが役立ちます。

自動化の制限


ユニットテストは、エラーを見つけて排除するための優れた追加機能ですが、大規模なゲームでの専門的な品質管理の必要性を置き換えるものではありません。QAは、創造性、主観的な判断、優れたテクニカルコミュニケーションを必要とする芸術です。
