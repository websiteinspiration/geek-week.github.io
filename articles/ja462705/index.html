ReactJSアプリケーションの上位5つのバグ

4年以上前にReactJSに夢中になり、それ以来、この素晴らしいツールを使用してすべてのフロントエンドアプリケーションを開発しています。この間、私と一緒に作業できて幸運だったチームはたくさんのミスを​​犯しましたが、その多くはうまく修正されました。多くの最適なソリューションが困難で費用のかかる実験で見つかりました。

今日、私は他の人より頻繁に行われる最も重大で痛みを伴う間違いを共有したいと思います。この記事を書く前に、もちろん、同様の記事を探すためにインターネットを研究していましたが、それらのほとんどが時代遅れであり、2019年にはあまり関係のないことについて話していることに驚きました。そこで、現時点で最も差し迫った問題のリストをまとめようとしました。

2.匿名関数を小道具として使用する


最初のエラーが依然としてファッションへのオマージュとして認識できる場合、2番目のエラーの知識は、眠れない夜や頭痛からあなたを救います。結局のところ、アプリケーションを不十分に機能させるのは彼女であり、そのユーザーはReactJSに永遠に失望する可能性があります。しかし、ユーザーと同じように、ユーザーにも彼を愛してほしいのです。このエラーを回避するには、非常に単純なルールを使用できます。絶対に、匿名関数をプロパティとしてコンポーネントに渡さないでください。

export default function MyOtherComponent() {
  return (
    <MyComponent getValue={i => i.value} /> {/*     */}
  );
}

このエラーのより高度なバージョンは次のようになります（Reduxに慣れていない場合は読んでください）。

import { connect } from 'react-redux';
import { createStructuredSelector } from 'reselect';
import MyComponent from './MyComponent';
import { fetchProjectData, projectId } from "./store/projects"

const mapStateToProps = createStructuredSelector({
  projectId,
});

const mapDispatchToProps = {
  fetchProjectData,
};

const mergeProps = (
  { projectId, ...restState }: any,
  { fetchProjectData, ...restDispatch }: any,
  { memberId, ...restOwn }: any
) => ({
  ...restState,
  ...restDispatch,
  ...restOwn,
  fetchProjectData: () => fetchProjectData(projectId),
});

export default connect(
  mapStateToProps,
  mapDispatchToProps,
  mergeProps
)(MyComponent);

どちらの場合も、最終的には、無名関数がpropsコンポーネントに入ります。親要素をレンダリングするたびに、この関数はメモリ内の新しいオブジェクトを参照するため、これは悪いことです。つまり、前のオブジェクトと同じにならず、コンポーネントが不必要に正常にレンダリングされます。アプリケーションのパフォーマンスを大幅に低下させる可能性があるため、Reactで自分自身が唾を吐き、失望するようになりますが、全体の要点は小道具の匿名関数にあります。決してそれをしないでください-そして幸せになります。

問題は、多くの場合、このようなエラーは何も問題を起こさないことです。コードはそれ自体で機能します-それだけです。そして、目立って悪いことは何も起こりません。もう一度匿名の呼び出しをプッシュしてサーバーからデータを受信する瞬間まで（2番目の例）、問題の深刻さを理解できます。結果として、このような匿名の小道具が蓄積すると、アプリケーションを遅くして、1995年に、ページをダウンロードするために電話線を解放するように近所の人に依頼しなければならなかった経験のレベルに達します。

ほんの数語、正しい書き方。方法は次のとおりです。

const getValue = i => i.value;

return default function MyOtherComponent() {
  return (
    <MyComponent getValue={getValue} />
  );
}


import { connect } from 'react-redux';
import { createStructuredSelector } from 'reselect';
import MyComponent from './MyComponent';
import { fetchProjectData, projectId } from "./store/projects"

const mapStateToProps = createStructuredSelector({
  projectId,
});

const mapDispatchToProps = {
  fetchProjectData,
};

export default connect(
  mapStateToProps,
  mapDispatchToProps
)(MyComponent);

    fetchProjectData(projectId);
  }, [fetchProjectData, projectId]);

3.アプリケーション内の複数のReactインスタンス


このエラーは、特にReactJSだけでなく、アプリケーション全体のアーキテクチャーに関連している可能性があります。しかし、この実用上の問題は、多くの場合、百あると、開発者とあまりにも多くの記事のでに関する ITそれら眠れない夜。

1つのページにReactアプリケーションの複数のインスタンスを詰め込もうとしないでください。実際、Reactのドキュメントではこのアプローチは禁止されていません。一部の記事ではそれを行うための推奨事項にさえ会いました（もちろん、私は自分のアプリケーションでこれを行いました）、このアプローチを最適化し、アプリケーションのすべての部分を調和させます（この場合）全労働時間の半分以上を占め始めます。これは簡単に回避できます。たとえば、新しいReactのレガシーコードでいくつかのイベントに応答する必要がある場合-appendix-イベントモデルを使用できます。たとえば、次のようになります。

import React, { useCallback, useEffect } from 'react';
export default function MyComponent() {
  const reactHandlerOfLegacyEvent = useCallback((event) => {/* event handler */}, []);
  
  useEffect(() => {
    document.addEventListener("myLegacyEvent", reactHandlerOfLegacyEvent);
    return () => {
      document.removeEventListener("myLegacyEvent", reactHandlerOfLegacyEvent);
    };
  }, [reactHandlerOfLegacyEvent]);

  return ({/*  -  */});
}

4.既存のオープンソースの代わりに、独自のライブラリを作成する


この問題はReactJSに関するものではなく、開発全体に関するものです。もちろん、単に学習している間は、独自のライブラリを多数作成することで、より速く成長し、大きくなることができますが、プログラミングの最前線になりたい場合は、問題を解決するオープンソースライブラリをそれぞれ試す必要があります。あなたの問題を解決するライブラリがあるかどうか、検索ロボットに尋ねてください-彼らはそのような質問に完璧に答えることができます。

私が自分で使用するライブラリの例は示しません。私はそれらの半分が数ヶ月で時代遅れになり、他の人が彼らの代わりに来ることを知っています。そして、これは元のステートメントと矛盾しているように思われます。そして、本当に、数か月で廃止されるライブラリを使用するのはなぜですか？答えは非常に簡単です。まだ発生していない問題の解決策を見ることができます。既存の開発を改善したり、既存のライブラリの問題の例を使用して問題をより適切に解決する方法を理解したりできます。これは、オープンソースライブラリを使用して開発の世界と対話することによってのみ実行できます。

5.他人のコードを使用することへの恐怖


前のエラーと同様に、これはReactJSアプリケーションに固有のものではありませんが、それらのアプリケーションではかなり一般的です。素晴らしい6月が勇敢に戦いに突入し、前の4つのエラーの1つについて読んだだけで問題なく機能するコードの一部を書き換える方法をどのくらいの頻度で確認しますか。私自身もそうでした。しかし、不正行為をするために何があるのか​​、私は今、しばしば時間を浪費しています。

しかし、私は他の開発者、彼らが遭遇した彼らの考えや問題を理解することも学びました。問題の解決に費やされた（無駄ではない）時間と労力の量を確認します。5つのうち3つのケースで、他の誰かのコードを「改善」しようとすると、私とほぼ同じ結果が得られます。タスクの開始時に、通常、将来のすべての問題が表示されるわけではありません。したがって、私は、誰かにとって奇妙で「時代遅れ」なコードに思えても、他の誰かのコードを尊重します。私はあなたに助言します。
