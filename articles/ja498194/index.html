Service Locator-神話を払拭する


あるプラットフォームの優れたパフォーマンスと使いやすさを示す実践が、別のプラットフォームの支持者の陣営でどのように示されるかは驚くべきことです。この運命は、.Netで非常に人気があり、iOSでの評判が悪いService Locatorのパターンによって完全に感じられます。

一部の記事では、ServiceLocatorが無視され、「自転車」と呼ばれています。他の人はそれがアンチパターンであると主張します。ロケーターのプラス面とマイナス面の両方を説明して中立性を維持しようとする人がいます。大部分は、ハブラハブ自身がこれらの迫害に関与しており、相互参照を持ついくつかの同様の記事が含まれています。Latitudeの実装に遭遇する新しい開発者は、ほぼ即座に放置されます。まあ、多くの開発者は客観的な理由なしに同じことを批判することはできません。

まず、サービスロケータとは何か、それを何に使用できるかを思い出してください。

サービスは、それ自体の中にビジネスロジックをカプセル化し、他のオブジェクトへのリンクを持つことができる自律オブジェクトです。実際、どのクラスのインスタンスもサービスとして機能できます。多くの場合、サービスの概念の代わりに、「マネージャー」の概念が使用されます。ただし、一般的な状況は、静的クラスがリポジトリデータを操作するマネージャとして機能する場合です。サービスの本質は、それがクラスのインスタンスであり、その後のすべての結果を伴うことです。これは、それが真空状態で存在することができないことを意味します-それは、アプリケーションの存続期間中、それに取り付けられるキャリアを持っている必要があります。そのようなキャリアはサービスロケータです。

ユーザー（アプリケーションまたは開発者）は、指定されたタイプのサービスをサービスロケーターに要求し、使用可能なインスタンスを受け取ります。抽象ファクトリーとよく似ていますよね？違いは、サービスロケータから指定されたタイプのインスタンスをリクエストするたびに、同じインスタンスが何度も受信され、すでに使用されているデータが保存されていることです。サービスは通常のシングルトンのように動作するように見えますが、これはそうではありません。サービスのインスタンスはいくつでも作成でき、自由に独立して使用できます。同時に、それらのそれぞれは、あなたが一生を通じてそこに投稿するデータをカプセル化します。


なぜこれが必要なのでしょうか？最も明白で愛される例は、ユーザープロファイルです。 UserSettingsまたはCoreDataの形式でストレージを使用しない場合、アプリケーションの存続期間中、アプリケーションのさまざまな画面でそれを使用するには、UserProfileクラスのインスタンスへのリンクをどこかに保持する必要があります。さらに、そのようなインスタンスがシングルトンでない場合は、あるフォームから別のフォームに転送する必要があります。アプリケーションの開発サイクルで一時ユーザーまたは独立した他のユーザーを作成しなければならない場合、必然的に問題が発生します。シントンはすぐにボトルネックになります。そして、独立したインスタンスはアプリケーションロジックに過負荷をかけ始め、インスタンスに関心のあるコントローラーが追加されるにつれて、その複雑さは指数関数的に増加します。


サービスロケーターはこの問題をエレガントに解決します。抽象的なUserProfileクラスがあり、特定のクラスDefaultUserPrifileおよびTemporatyUserProfileが継承されている場合（完全に空の実装、つまり実質的に同一）、サービスロケーターにアクセスすると、2つの同一の独立したオブジェクトが返されます。

ロケーターのもう1つのアプリケーションは、コントローラーのチェーンを介したデータインスタンスの転送です。最初のコントローラーでは、オブジェクトを作成（受信）して変更し、最後のコントローラーでは、最初のオブジェクトに入力したデータを使用します。コントローラーの数が非常に多く、それらがスタック上にある場合、これらの目的でデリゲートを使用することは非常に困難です。同様に、スタックを最小化した直後に変更された情報をスタックのルートに表示する必要があることがよくあります（スタックはコピーで作成されたすべてのインスタンスを削除することを好むことを覚えています）。ただし、スタックの最上部でサービスを取得して変更し、その後スタックの折りたたみを開始した場合、ユーザーがルートコントローラーを使用できるようになると、変更されたデータが保存され、表示できるようになります。

一般的に、ほとんどのチュートリアルで説明されている「コーディネーター」パターンを使用する場合（たとえば、ここ、ここ、ここ、ここ、ここ、またはここ）、クラスのインスタンスをAppDelegateに配置するか、コーディネーターへのリンクをすべてのViewControllerに渡す必要があります。それを使用します。必要？そして、なぜ、実際に？


個人的に、私はAppDelegateがきれいに輝くことを好みます。そして、ICoordinatableから継承してコーディネーターフィールドを設定するには、時間を費やすだけでなく（これは、ご存じのように、お金に相当します）、ストーリーボードを通じて人間による宣言型プログラミングの可能性を奪います。いいえ、これは私たちの方法ではありません。

コーディネーターをサービスとして適切に作成すると、利点の欠点が生じます。

  • コーディネーターの整合性を維持することを気にする必要はありません。
  • コーディネーターは、ICoordinatableから継承されていないコントローラーでも、アプリケーション全体で使用可能になります。
  • 必要なときにだけコーディネーターを開始します。
  • コーディネーターは、ストーリーボードと一緒に任意の順序で使用できます（便利です）。
  • ストーリーボードでコーディネーターを使用すると、自明ではないが効果的なナビゲーションメカニズムを作成できます。

ただし、コーディネーターは「ナビゲーター」パターン（ルーターを使用して計算されたパスを介してアプリケーション内を移動するメカニズム）の半分です。その実装により、複雑さは桁違いに増加します。ナビゲーターとサービスロケーターの機能-これは別の幅広いトピックです。ロケーターに戻りましょう。

サービスロケーターが不良であるという理由につながる客観的な理由には、特定のサービスの保守の複雑さと、ロケーターのメモリの状態を制御できないことが含まれます。

彼はひどいですか？最初にサービスを登録し、それを使用するために抽出する必要があります。最初のケースでは、要求されたときに存在することが保証されています。しかし、2番目は必要になる前にサービスを作成しません。多くの場所でサービスを利用する必要がある場合は、選択肢を増やすことで夢中になります。

しかし、これらすべてをそのようなコードに変換するのは簡単です。

ProfileService.service.update(name: "MyName")

ここでは、サービスインスタンスが存在することが保証されています。存在しない場合は、サービス自体によって作成されるためです。余分なものはありません。

ロケーターに対する2番目の主張は、C＃でパターントレースを作成する開発者がガベージコレクターの作業を忘れ、不必要なインスタンスからロケーターをクリーンアップする機能を提供しないことに起因するようですが、これはまったく難しいことではありません。

ProfileService.service.remove()

これを行わない場合、ProfileService.service（）に切り替えると、以前にアプリケーションの任意の場所ですでに処理したサービスのインスタンスが取得されます。ただし、（）を削除すると、サービスにアクセスしたときにクリーンなコピーが取得されます。場合によっては、削除（）の代わりに、データの事前定義された部分をクリアして同じインスタンスで作業を続けることにより、クリア（）を行うことができます。

ビデオは、フィールドに入力された値がアプリケーションの後続の各画面に送信されることを示しています。そして最後の画面では、最初の画面からアプリケーションを起動するためにコーディネーターが呼び出されます。ここでは、従来のナビゲーションスタックの折りたたみは発生しません。メモリから完全に押し出され、代わりに新しいスタックが開始されます。プロファイルサービスを削除する行をコメントアウトすると、ナビゲーションスタックを最小化した場合と同様に、ユーザー名が最初の画面に転送されます。

ロケータ：

import Foundation

protocol IService {
    static var service: Self {get}
    
    func clear()
    func remove()
}
protocol IServiceLocator {
    func service<T>() -> T?
}

final class ServiceLocator: IServiceLocator {
    
    private static let instance = ServiceLocator()
    private lazy var services: [String: Any] = [:]
    
    // MARK: - Public methods
    class func service<T>() -> T? {
        return instance.service()
    }
    
    class func addService<T>(_ service: T) {
        return instance.addService(service)
    }
    
    class func clear() {
        instance.services.removeAll()
    }
    
    class func removeService<T>(_ service: T) {
        instance.removeService(service)
    }
    
    func service<T>() -> T? {
        let key = typeName(T.self)
        return services[key] as? T
    }
    
    // MARK: - Private methods
    private fun caddService<T>(_ service: T) {
        let key = typeName(T.self)
        services[key] = service
    }
    
    private func removeService<T>(_ service: T) {
        let key = typeName(T.self)
        services.removeValue(forKey: key)
    }
    
    private func typeName(_ some: Any) -> String {
        return (some isAny.Type) ? "\(some)" : "\(type(of: some))"
    }
}

サービスプロファイルはそれほど複雑ではありません。

import UIKit

final class ProfileService: IService {
    
    private (set) var info = ProfileInfo()
    
    class var service: ProfileService {
        if let service: ProfileService = ServiceLocator.service() {
            return service
        }
        
        let service = ProfileService()
        ServiceLocator.addService(service)
        return service
    }

    func clear() {
        self.info = ProfileInfo()
    }

    func remove() {
        ServiceLocator.removeService(self)
    }
    
    func update(name: String) {
        self.info.name = name
    }
}

struct ProfileInfo {
    varname = ""
}

サービス自体の内部でデータ領域を移動することにより、さらに簡素化できます。しかし、この形式では、データモデルとサービスの責任の領域が明確になります。

final class CoordinatorService: IService {
    
    private (set)var coordinator: MainCoordinator!
    
    var navController: UINavigationController {
        return self.coordinator.navigationController
    }
    
    class var service: CoordinatorService {

            return service
        }
        
        let service = CoordinatorService()
        service.load()
        ServiceLocator.addService(service)
        return service
    }

    func clear() {
    }

    func remove() {
        ServiceLocator.removeService(self)
    }
    
    // MARK - Private
    private func load() {
        let nc                    = UINavigationController()
