BpfTrace-最後に、Linux上のDtraceの完全な代替品

システムにバグがあり、速度が低下し、故障することがあります。システムが大きくなるほど、原因の特定が難しくなります。何かが期待どおりに機能しない理由を突き止め、将来の問題を修正または防止するには、内部を調べる必要があります。そのためには、システムは可観測性の特性を備えている必要があります。これは、広義のインストルメンテーションによって実現されます。


でHighLoad ++、ピーター・ザイツェフ（Percona）は、Linuxのトレースに利用できるインフラを見直し、bpfTrace、（名前が示すように）申し出多くの利点について話しました。レポートのテキスト版を作成したので、詳細を確認するのに便利で、追加の資料が常に手元にありました。

計装は2つの大きなブロックに分割できます。

  • 静的、情報のコレクションがコードに関連付けられている場合：記録ログ、カウンター、時間など。
  • 動的、コード自体がインストルメント化されていないが、必要なときにそれを実行できる場合。

別の分類オプションは、データを記録する方法に基づいています。

  • トレース -特定のコードが機能した場合にイベントが生成されます。
  • サンプリング -システムのステータスがチェックされます。たとえば、毎秒100回、システムで何が起こっているかを決定します。

静的計装は何年も前から存在しており、ほとんどすべてに含まれています。Linuxでは、Vmstatやtopなどの多くの標準ツールがそれを使用します。procfsからデータを読み取ります。大まかに言えば、カーネルコードからさまざまなタイマーとカウンターが書き込まれます。

しかし、これらのカウンターをあまり多く挿入することはできません。これらのカウンターで世界のすべてをカバーすることはできません。したがって、動的なインストルメンテーションが役立ち、必要なものを正確に監視できます。たとえば、TCP / IPスタックに問題がある場合は、非常に深く掘り下げて特定の詳細を指示できます。



Dtrace


DTraceは、Sun Microsystemsによって作成された最初の既知の動的トレースフレームワークの1つです。 2001年に作成が開始され、2005年に初めてSolaris 10でリリースされました。このアプローチは非常に人気があり、後に他の多くのディストリビューションにも採用されました。

興味深いことに、DTraceではカーネル空間とユーザー空間の両方を計測できます。関数呼び出しにトレースを配置し、プログラムに具体的に指示することができます。特別なDTraceトレースポイントを導入します。これにより、ユーザーは関数名よりも理解しやすくなります。

これは、オープンオペレーティングシステムではないため、Solarisにとって特に重要でした。新しいオープンソースLinuxソフトウェアで実行できるようになったため、コードを調べてトレースポイントをそのような関数に配置する必要があることを理解することはできませんでした。

特に当時のDTraceのユニークな機能の1つは、トレースが有効になっていなくても費用がかからないことです。これは、CPU命令の一部を、戻り時にこれらの命令を実行するDTrace呼び出しに置き換えるだけで動作します。

DTraceでは、インストルメンテーションはCやAwkと同様の特別なD言語で記述されています。

その後、DTraceはLinuxを除くほぼすべての場所に登場しました。2007年にはMacOS、2008年にはFreeBSD、2010年にはNetBSDが登場しました。2011年のOracleには、Oracle Unbreakable LinuxにDTraceが含まれていました。しかし、Oracle Linuxを使用する人はほとんどいないため、DTraceがメインLinuxに入ったことはありません。

興味深いことに、2017年、オラクルはついにGPLv2の下でDTraceのライセンスを取得しました。これにより、ライセンス供与の問題なしにLinuxをメインラインのLinuxに含めることが可能になりましたが、すでに遅すぎました。当時、Linuxには優れたBPFがあり、主に標準化に使用されていました。

DTraceはWindowsにも含まれる予定ですが、一部のテストバージョンで利用できるようになりました。

eBPF — Linux


これらのすべてのフレームワークにより、eBPFは近年Linuxの標準になりました。これは、ほとんどすべてを可能にする、より高度で柔軟性が高く効果的なツールです。

eBPFとは何で、どこから来たのですか？実際、eBPFは拡張バークレーパケットフィルターであり、BPFは1992年にファイアウォールによる効率的なパケットフィルタリングのための仮想マシンとして開発されました。当初、彼は監視、観察可能性、または追跡とは関係がありませんでした。

最新のバージョンでは、イベントを処理するための一般的なフレームワークとして、eBPFが拡張されています（そのため、拡張された単語です）。現在のバージョンは、JITコンパイラーと統合されており、効率が向上しています。

従来のBPFとのeBPFの違い：

  • レジスターが追加されました。
  • スタックが現れました。
  • 追加のデータ構造（マップ）があります。



今では、ほとんどの場合、古いBPFがあったことを忘れ、eBPFは単にBPFと呼ばれます。最近のほとんどの式では、eBPFとBPFは同じものです。したがって、ツールはeBpfTraceではなくbpfTraceと呼ばれます。

eBPFは、2014年以降、メインラインLinuxに含まれており、Perf、SystemTap、SysDigなどの多くのLinuxツールに徐々に含まれています。標準化があります。

EBPFプログラム


では、eBPFとは何で、なぜそれが興味深いのでしょうか。

eBPFは特別なバイトコードのプログラムであり、カーネルに直接含まれており、トレースイベントの処理を実行します。さらに、特別なバイトコードで作成されているため、カーネルはコードが非常に安全であることを確認できます。たとえば、カーネルのクリティカルセクションのループがシステム全体をハングさせる可能性があるため、ループを使用していないことを確認してください。

しかし、これは完全に安全であることを可能にしません。たとえば、非常に複雑なeBPFプログラムを作成する場合、毎秒1000万回発生するカーネルのイベントに挿入すると、すべてが非常に遅くなる可能性があります。しかし、同時に、eBPFは、一部のカーネルモジュールがinsmodを介して挿入されただけで、これらのモジュールに何でも含まれる可能性がある場合、以前のアプローチよりもはるかに安全です。誰かがミスをした場合、または単にバイナリの非互換性のために、コア全体が落ちる可能性があります。

eBPFコードはLLVM Clangでコンパイルできます。つまり、概して、Cのサブセットを使用してeBPFプログラムを作成しますが、これはもちろん非常に複雑です。また、コンパイルはカーネルに依存することが重要です。ヘッダーは、どの構造が使用され、何に使用されているかなどを理解するために使用されます。これは、特定のコアに関連するいくつかのモジュールが常に提供されるか、または再コンパイルする必要があるという意味で、あまり便利ではありません。

EBPFパフォーマンス


パフォーマンスの点では、eBPFは非常に効果的です。オーバーヘッドの程度とオーバーヘッドがあるかどうかを理解するには、プローブを追加します。プローブは1秒あたり数回切り替わり、実行にかかる時間を確認できます。



Cloudflareのメンバーがベンチマークを作成しました。単純なeBPFプローブでは約100ナノ秒かかりましたが、より複雑なプローブでは300ナノ秒かかりました。これは、複雑なプローブであっても、1秒あたり約300万回、単一のコアで呼び出すことができることを意味します。プローブがマルチコアプロセッサで毎秒10万回または100万回ジャーク動作する場合、これはパフォーマンスにあまり影響しません。

DTraceとbpfTrace


一般的に、DTraceとbpfTraceは同じものに使用されます。


http://www.brendangregg.com/blog/2018-10-08/dtrace-for-linux-2018.html

違いは、複雑なツールに使用できるBPFエコシステムにもBCCがあることです。DTraceには同等のBCCがないため、複雑なツールキットを作成するには、通常、Shell + DTraceバンドルを使用します。

bpfTraceを作成するときに、DTraceを完全にエミュレートするタスクはありませんでした。つまり、DTraceスクリプトを取得してbpfTraceで実行することはできません。しかし、これはあまり意味がありません。下位レベルのツールのロジックは非常に単純だからです。 Linux、Solaris、FreeBSDでは、接続する必要のあるトレースポイント、システムコールの名前、およびそれらが低レベルで直接何を行うかを理解することが通常より重要です。それが違いが生じるところです。

この場合、bpfTraceはDTraceから15年後に作成されます。 DTraceにはないいくつかの追加機能があります。たとえば、スタックトレースを実行できます。

しかし、もちろん、多くはDTraceから継承されます。たとえば、関数名と構文は似ていますが、完全に同等ではありません。


DTraceスクリプトとbpfTraceスクリプトは、コードサイズが近く、複雑さと言語機能が似ています。



bpfTrace

すべての機能を使用するには、少なくとも4.9のバージョンが必要です。BpfTraceを使用すると、ユーザーアプリケーションやカーネルプローブなどで関数呼び出しを計測するためのアップローブから始めて、さまざまなプローブを作成できます。


興味深いことに、カスタムのアップローブ関数に相当するuretprobeがあります。カーネルの場合、同じことがkprobeとkretprobeです。つまり、実際には、トレースフレームワークでは、関数が呼び出されたとき、およびこの関数の完了時にイベントを生成できます。これは、タイミングによく使用されます。または、関数が返した値を分析し、関数の呼び出しに使用されたパラメーターに従ってそれらをグループ化することもできます。関数呼び出しをキャッチしてそこから戻ると、多くの素晴らしいことができます。

bpfTrace内では、次のように機能します。解析され、Cに変換され、Clangで処理されるbpfプログラムを記述します。これにより、bpfバイトコードが生成され、その後プログラムが読み込まれます。


プロセスは非常に難しいため、制限があります。強力なサーバーでは、bpfTraceが適切に機能します。しかし、Clangを小さな組み込みデバイスにドラッグして、何が起こっているのかを把握することはお勧めできません。プライはこれに適しています。もちろん、bpfTraceのすべての機能を備えているわけではありませんが、直接バイトコードを生成します。

Linuxサポート


bpfTraceの安定バージョンは約1年前にリリースされたため、古いLinuxディストリビューションでは使用できません。パッケージを取得するか、IO Visorが配布する最新バージョンをコンパイルすることをお勧めします。

興味深いことに、最新のUbuntu LTS 18.04にはbpfTraceがありませんが、snapパッケージを使用して配信できます。これは便利ですが、スナップパッケージの作成方法と分離方法により、すべての機能が動作するわけではありません。カーネルトレースの場合、snapを含むパッケージは適切に機能しますが、ユーザートレースの場合、正しく機能しない可能性があります。



プロセストレースの例


IOリクエストの統計を取得できる最も単純な例を考えてみます。

bpftrace -e 'kprobe:vfs_read { @start[tid] = nsecs; }
kretprobe:vfs_read /@start[tid]/ { @ns[comm] = hist(nsecs - @start[tid]); delete(@start[tid]); }'

ここではvfs_read、kretprobeとkprobeの両方の関数に接続します。さらに、各スレッドID（tid）について、つまり各要求について、その実行の開始と終了を追跡します。データは、システム全体の全体だけでなく、さまざまなプロセスによってもグループ化できます。以下は、MySQLのIO出力です。



古典的なバイモーダルI / O分布が表示されます。高速リクエストの多くは、キャッシュから読み取られるデータです。2番目のピークは、レイテンシがはるかに高いディスクからデータを読み取ることです。

これをスクリプトとして保存し（通常、bt拡張子が使用されます）、コメントを記述し、フォーマットして、さらに使用し#bpftrace read.btます。

// read.bt file
tracepoint:syscalls:sys_enter_read
{
  @start[tid] = nsecs;
}
tracepoint:syscalls:sys_exit_read / @start[tid]/ 
{
  @times = hist(nsecs - @start[tid]);
  delete(@start[tid]);
}

言語の一般的な概念は非常に単純です。

  • 構文：接続するプローブを選択しますprobe[,probe,...] /filter/ { action }。
  • フィルター：フィルターを指定します。たとえば、特定のPidの特定のプロセスに関するデータのみを指定します。
  • アクション： bpfプログラムに直接変換し、bpfTraceが呼び出されたときに実行されるミニプログラム。

MySQLトレースの例


MySQLの例とその仕組みを紹介します。MySQLには、dispatch_commandすべてのMySQLクエリ実行が発生する機能があります。

bpftrace -e 'uprobe:/usr/sbin/mysqld:dispatch_command { printf("%s\n", str(arg2)); }'
failed to stat uprobe target file /usr/sbin/mysqld: No such file or directory

私は、MySQLに送信されるクエリのテキストを印刷するためにアップローブを接続したかっただけです-基本的なタスクです。問題が発生しました-そのようなファイルがないと言います。ここではないときのように：

root@mysql1:/# ls -la /usr/sbin/mysqld
-rwxr-xr-x 1 root root 60718384 Oct 25 09:19 /usr/sbin/mysqld

これらは、スナップによる単なる驚きです。スナップを介して設定した場合、アプリケーションレベルで問題が発生する可能性があります。

その後、新しいUbuntuであるaptバージョンを使用してインストールを再開しました。

root@mysql1:~# bpftrace -e 'uprobe:/usr/sbin/mysqld:dispatch_command { printf("%s\n", str(arg2)); }'
Attaching 1 probe...
Could not resolve symbol: /usr/sbin/mysqld:dispatch_command

「そのようなシンボルはありません」-いかがですか？私nmはそのようなシンボルがあるかどうかを調べます：

root@mysql1:~# nm -D /usr/sbin/mysqld | grep dispatch_command
00000000005af770 T 
_Z16dispatch_command19enum_server_commandP3THDPcjbb

root@localhost:~# bpftrace -e 'uprobe:/usr/sbin/mysqld:_Z16dispatch_command19enum_server_commandP3THDPcjbb { printf("%s\n", str(arg2)); }'
Attaching 1 probe...
select @@version_comment limit 1
select 1

そのようなシンボルがありますが、MySQLはC ++からコンパイルされるため、マングリングが使用されます。実際、このコマンドで使用される関数の現在の名前は次のとおりです_Z16dispatch_command19enum_server_commandP3THDPcjbb。関数で使用すると、接続して結果を取得できます。perfエコシステムでは、多くのツールが自動的にアンマングルを解除しますが、bpfTraceはまだできません。

のフラグ-Dにも注意してくださいnm。MySQLと他の多くのパッケージには動的シンボル（デバッグシンボル）がないため、これは重要です。これらは他のパッケージで提供されます。これらの文字を使用する場合はフラグが必要-Dです。それ以外の場合、nmはそれらを表示しません。
