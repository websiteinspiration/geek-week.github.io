元の形式を見てください。

template<typename T>
class MyClass
{
    public:
        void MyClass(T const& x){}
        template<typename T_ = T>
        void f(T&& x,
            typename std::enable_if<!std::is_reference<T_>::value,
            std::nullptr_t>::type = nullptr){}
};


そして、それを次のより表現力豊かな形式と比較してください。

template<typename T>
using IsNotReference = std::enable_if_t<!std::is_reference_v<T>>;
template<typename T>
class MyClass
{
public:
    void f(T const& x){}
    template<typename T_ = T, typename = IsNotReference <T_>>
    void f(T&& x){}
};

リラックスして本番環境で使用を開始することはすでに可能であると合理的に信じています。可能ですが、ほとんどの場合は機能しますが、インターフェイスについては、コードは安全で信頼できるものでなければなりません。そうですか？ハックしよう！

欠陥＃1：SFINAEがバイパスされる可能性がある


通常、SFINAEは条件に応じてコードの一部を無効にするために使用されます。これは、たとえば、何らかの理由でユーザー定義のabs関数（ユーザー定義の算術クラス、特定の機器の最適化、トレーニング目的など）を実装する必要がある場合に非常に役立ちます。

template< typename T >
T myAbs( T val ) {
    return( ( val <= -1 ) ? -val : val );
}
 
int main()
{
    int a{ std::numeric_limits< int >::max() };
    std::cout << "a: " << a << " myAbs( a ): " << myAbs( a ) << std::endl;
}

このプログラムは次のように表示しますが、かなり正常に見えます。

a: 2147483647 myAbs( a ): 2147483647

しかしabs、符号なしの引数Tで関数を呼び出すことができ、その結果は壊滅的なものになります。

nt main()
{
    unsigned int a{ std::numeric_limits< unsigned int >::max() };
    std::cout << "a: " << a << " myAbs( a ): " << myAbs( a ) << std::endl;
}

実際、プログラムは次

a: 4294967295 myAbs( a ): 1

のように表示します。関数は符号なし引数で機能するように設計されていないため、TSFINAEを使用して可能なセットを制限する必要があります。

template< typename T >
using IsSigned = std::enable_if_t< std::is_signed_v< T > >;
 
template< typename T, typename = IsSigned< T > >
T myAbs( T val ) {
    return( ( val <= -1 ) ? -val : val );
}

コードは期待どおりに機能します。符号なしの型でmyAbsを呼び出すと、コンパイル時エラーが発生します。

candidate template ignored: requirement 'std::is_signed_v<unsigned int>' was not satisfied [with T = unsigned int]

SFINAE状態のハッキング


次に、この関数の何が問題になっていますか？この質問に答えるには、myAbsがSFINAEを実装する方法を確認する必要があります。

template< typename T, typename = IsSigned<T> >
T myAbs( T val );

myAbs

int a{ myAbs( -5 ) };
int b{ myAbs< int >( -5 ) };

最初と2番目の呼び出しは単純ですが、3番目の呼び出しは興味深いものです。このテンプレートの引数はvoid何ですか？

2番目のテンプレートパラメーターは匿名であり、既定の型がありますが、それでもテンプレートパラメーターであるため、明示的に指定できます。これは問題ですか？この場合、これは本当に大きな問題です。3番目のフォームを使用して、SFINAEチェックを回避できます。

unsigned int d{ myAbs< unsigned int, void >( 5u ) };
unsigned int e{ myAbs< unsigned int, void >( std::numeric_limits< unsigned int >::max() ) };

欠点＃2：特定の実装ができない


SFINAEのもう1つの一般的な用途は、特定のコンパイル時の条件に特定の実装を提供することです。myAbs符号なしの値での呼び出しを完全に禁止し、これらのケースに簡単な実装を提供したくない場合はどうなりますか？if if constexpr in C ++ 17（これについては後で説明します）、または次のことができます。

 template< typename T >
using IsSigned = std::enable_if_t< std::is_signed_v< T > >;
 
template< typename T >
using IsUnsigned = std::enable_if_t< std::is_unsigned_v< T > >;
 
template< typename T, typename = IsSigned< T > >
T myAbs( T val ) {
    return( ( val <= -1 ) ? -val : val );
}
 
template< typename T, typename = IsUnsigned< T > >
T myAbs( T val ) {
    return val;
}

しかし、それは何ですか？

error: template parameter redefines default argument
template< typename T, typename = IsUnsigned< T > >
note: previous default template argument defined here
template< typename T, typename = IsSigned< T > >

ああ、C ++標準（C ++ 17;§17.1.16）は次のように述べています：

「デフォルトの引数は、同じスコープ内の2つの異なる宣言によってテンプレートパラメータに提供されるべきではありません。」

おっと、これはまさに私たちがやったことです...

なぜレギュラーを使わないのですか？


代わりに、実行時に次のように使用できます。

template< typename T >
T myAbs( T val ) {
    if( std::is_signed_v< T > ) {
        return ( ( val <= -1 ) ? -val : val );
    } else {
        return val;
    }
}

それがあるため、コンパイラは、条件を最適化するだろうif (std::is_signed_v <T>)なっif (true)たりif (false)、テンプレートを作成した後。はい、現在の実装ではmyAbsこれで動作します。しかし、一般的に、これは巨大な制限課し：文をifし、else皆のために有効である必要がありますT。実装を少し変更するとどうなるでしょうか。

template< typename T >
T myAbs( T val ) {
    if( std::is_signed_v< T > ) {
     	return std::abs( val );
    } else {
     	return val;
    }
}
 
int main() {
    unsigned int a{ myAbs( 5u ) };
}

コードはすぐにクラッシュします。

error: call of overloaded ‘abs(unsigned int&)’ is ambiguous

この制限により、SFINAEは排除されます。Tのサブセットに対してのみ有効なコードを記述できます（myAbsでは、符号なしの型に対してのみ有効、または符号付きの型に対してのみ有効です）。

解決策：SFINAEの別のフォーム


これらの欠点を克服するために何ができるでしょうか？最初の問題では、ユーザーが関数を呼び出す方法に関係なく、SFINAEチェックを強制する必要があります。現在、コンパイラーが2番目のテンプレートパラメーターのデフォルトタイプを必要としない場合、テストを回避できます。

SFINAEコードを使用して、デフォルトのタイプを提供する代わりにテンプレートパラメータタイプを宣言するとどうなりますか？やってみよう：

template< typename T >
using IsSigned = std::enable_if_t< std::is_signed_v< T >, bool >;
 
template< typename T, IsSigned< T > = true >
T myAbs( T val ) {
    return( ( val <= -1 ) ? -val : val );
}
 
int main() {
    //int a{ myAbs( 5u ) };
    int b{ myAbs< int >( 5u ) };
    //int c{ myAbs< unsigned int, true >( 5u ) };
}

この型のデフォルト値を提供するため、有効な場合はIsSignedをvoid以外の型にする必要があります。void型には値がないため、bool、int、enum、nullptr_tなど、他のものを使用する必要があります。通常はboolを使用します。この場合、式は意味がありそうです：

template< typename T, IsSigned< T > = true >

できます！myAbs (5u)以前のように、コンパイラはエラーを出します：

candidate template ignored: requirement 'std::is_signed_v<unsigned int>' was not satisfied [with T = unsigned int

2番目の呼び出しmyAbs <int> (5u)は引き続き有効です。コンパイラに型をT明示的に通知するため、に変換さ5uれintます。

最後に、myAbs指の周りをトレースできなくなりmyAbs <unsigned int, true> (5u)ます。エラーが発生します。コンパイラが匿名のテンプレート値の引数型を必要とするため、呼び出しでデフォルト値を提供するかどうかは関係ありません。いずれにしても、SFINAE式の一部が評価されます。

次の問題に進むことができます-少し待ってください！同じテンプレートパラメータのデフォルト引数をオーバーライドする必要はなくなったと思います。元の状況はどうでしたか？

template< typename T, typename = IsUnsigned< T > >
T myAbs( T val );
 
template< typename T, typename = IsSigned< T > >
T myAbs( T val );

しかし、現在のコードでは：

template< typename T, IsUnsigned< T > = true >
T myAbs( T val );
 
template< typename T, IsSigned< T > = true >
T myAbs( T val );

前のコードと非常によく似ているので、これも機能しないと思うかもしれませんが、実際にはこのコードには同じ問題はありません。なにIsUnsigned <T>？ブールまたはルックアップの失敗。なにIsSigned <T>？同じことですが、一方がBoolの場合、もう一方はルックアップの失敗です。

つまり、テンプレート引数boolを持つ関数は1つしかなく、もう1つは失敗した置換であるため、デフォルト引数はオーバーライドされず、存在しません。

古いバージョンのC ++


上記のすべてがC ++ 11で機能します。唯一の違いは、標準バージョン間の制限の定義の冗長性です。

//C++11
template< typename T >
using IsSigned = typename std::enable_if< std::is_signed< T >::value, bool >::type;
 
//C++14 - std::enable_if_t
template< typename T >
using IsSigned = std::enable_if_t< std::is_signed< T >::value, bool >;
 
//C++17 - std::is_signed_v
template< typename T >
using IsSigned = std::enable_if_t< std::is_signed_v< T >, bool >;

ただし、テンプレートは同じです。

template< typename T, IsSigned< T > = true >

古き良きC ++ 98では、テンプレートエイリアスはありません。さらに、関数テンプレートは型またはデフォルト値を持つことができません。SFINAEコードを結果タイプに挿入することも、関数パラメーターのリストにのみ挿入することもできます。コンストラクターには結果タイプがないため、2番目のオプションをお勧めします。私たちにできる最善のことは次のようなものです：

template< typename T >
T myAbs( T val, typename my_enable_if< my_is_signed< T >::value, bool >::type = true ) {
    return( ( val <= -1 ) ? -val : val );
}

比較のためだけに-C ++の最新バージョン：

template< typename T, IsSigned< T > = true >
T myAbs( T val ) {
    return( ( val <= -1 ) ? -val : val );
}

C ++ 98バージョンは醜く、意味のないパラメーターが導入されていますが、機能します-どうしても必要な場合は使用できます。そしてはい：my_enable_ifそれらmy_is_signedは実装されるべきです（std :: enable_if std :: is_signedC ++ 11の新機能）。

現在の状態


C ++ 17が導入されましたif constexpr-コンパイル時の条件に基づいてコードを破棄する方法。 ifステートメントとelseステートメントの両方が構文的に正しい必要がありますが、条件はコンパイル時に評価されます。

template< typename T >
T myAbs( T val ) {
    if constexpr( std::is_signed_v< T > ) {
        return( ( val <= -1 ) ? -val : val );
    } else {
        if constexpr( std::is_unsigned_v< T > ) {
            return val;
        } /*else {
            static_assert( false, "T must be signed or unsigned arithmetic type." );
        }*/
    }
}

ご覧のように、abs関数はよりコンパクトで読みやすくなっています。ただし、不適切なタイプの処理は簡単ではありません。コメント付きの無条件の場合static_assert、このステートメントの調整が不十分になります。これは、破棄されるかどうかに関係なく、標準で禁止されています。

幸い、抜け穴があります。テンプレートオブジェクトでは、条件が値に依存しない場合、破棄された演算子は作成されません。いいね！

したがって、コードの唯一の問題は、テンプレートの定義中にクラッシュすることです。static_assertテンプレートが作成されるまで評価を延期できれば、問題は解決されます。つまり、すべての条件がfalseの場合にのみ作成されます。しかし、どうすれば延期できますかstatic_assertテンプレートを作成する前に？タイプに応じて状態を変えてください！

template< typename >
inline constexpr bool dependent_false_v{ false };
 
template< typename T >
T myAbs( T val ) {
    if constexpr( std::is_signed_v< T > ) {
        return( ( val <= -1 ) ? -val : val );
    } else {
        if constexpr( std::is_unsigned_v< T > ) {
            return val;
        } else {
            static_assert( dependent_false_v< T >, "Unsupported type" );
        }
    }
}

今後について


私たちはすでに非常に近いですが、C ++ 20が最終的な解決策をもたらすまで、少し待つ必要があります：概念！これにより、テンプレート（およびSFINAE）の使用方法が完全に変わります。

簡単に言えば、概念を使用して、テンプレートパラメータとして受け入れられる引数のセットを制限できます。abs関数には、次の概念を使用できます。

template< typename T >
concept bool Arithmetic() {
    return std::is_arithmetic_v< T >;
}

そして、どうすればコンセプトを使用できますか？3つの方法があります。

//  
template< typename T >
requires Arithmetic< T >()
T myAbs( T val );
 
 
//  
template< Arithmetic T >
T myAbs( T val );
 
// 
Arithmetic myAbs( Arithmetic val );

3番目のフォームはテンプレート関数を宣言していることに注意してください！C ++ 20でのmyAbsの完全な実装は次のとおりです。

template< typename T >
concept bool Arithmetic() {
    return std::is_arithmetic_v< T >;
}
 
Arithmetic myAbs( Arithmetic val ) {
    if constexpr( std::is_signed_v< decltype( val ) > ) {
        return( ( val <= -1 ) ? -val : val );
    } else {
        return val;
    }
}
 
int main()
{
    unsigned int a{ myAbs( 5u ) };
    int b{ myAbs< int >( 5u ) };
    //std::string c{ myAbs( "d" ) };
}

コメントアウトされた呼び出しは次のエラーを出します：

error: cannot call function 'auto myAbs(auto:1) [with auto:1 = const char*]'
constraints not satisfied
within 'template<class T> concept bool Arithmetic() [with T = const char*]'
concept bool Arithmetic(){
^~~~~~~~~~
'std::is_arithmetic_v' evaluated to false

これらの方法を量産コードで大胆に使用することを全員に強く要請します;コンパイル時間はランタイムよりも安いです。SFINAEingをお楽しみください。
