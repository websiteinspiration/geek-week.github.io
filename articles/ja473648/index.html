馬は死んだ-叫び：tslintからeslintへの移行

最近まで、フロントのすべてのプロジェクトで、Dodo Pizza Engineeringの開発者はtslintを使用していました-tslint は、コードをめちゃくちゃにして不正確にしたときに、1つのスタイルでコードを維持し、多くのコメントを修正するのに役立つ便利なツールです。しかし、その後tslintは奪取して死亡しました。カットの下で、なぜそれが起こったのか、故人に涙を注ぐのをやめてエスリントツールに切り替える方法、そして非常に親密なものを示す方法についてお話します。



実際、それはずっと前に始まりました：tslintカーネルの最後のリリースはすでに2016年にありました。そして、そのリリースが本当に最後だったので、誰かがまだ「最後」と言っているなら、これが「最後」と言い始める時です。2019年2月19日、tslintの開発を停止するための公式投稿がリリースされました。その中で、開発会社（ちなみに、Microsoftでさえありません）は、このリンターでのTypeScriptサポートの改善を目指しているため、eslintに切り替えることを強くお勧めします。

1つの言語、1つのスタック、1つのコミュニティ


Microsoftは、TypeScriptをJava / ECMAスクリプトに取って代わる主要なWeb開発言語と見なしています。明らかに、このような野心的な目標は、フロントエンド開発全体のための単一のスタックのツールを意味します。これにより、大規模なJSコミュニティからTypeScriptへの移行が大幅に簡略化されます。 Microsoftからの信頼の保証に加えて、eslintはtslintよりも優れたアーキテクチャを備えています。たとえば、パーサーを接続でき、接続されたルールの選択肢が増えます。

マイクロソフトが欲しかっただけの場合、それ自体はマイクロソフトではありません。彼らのソフトウェアの品質について私たちが何を言っても、彼らは素晴らしい開発ツール（そして、ところで、入力デバイス）を実行します。だから今回は手ぶらで来たのではなく、移行計画を書いた。この計画に従って、tslintルールの開発は2019年8月1日にすでに中止されており、tslint自体の開発は2019年11月1日に終了します。正直なところ、開発はずっと前に中止されています（最新のリリースについては上記を参照）。

ここで、読者に、eslintに切り替える時が来たことを明らかにします。他に選択肢はありません。錠剤を甘くするために、それは言う価値があります：

  • tslintはTypeScriptに重点を置いており、型の正しい使用と構文チェックに重点を置いていますが、eslintは、Reactコンポーネントの構文を含む、前にあるすべてのものをカバーしています。
  • eslintには、さらに多くの定義済みルールがあります。
  • ブロックレベルでコードをチェックするルール（およびプラグイン）があります（コードの重複、知覚される複雑さなど）。
  • コードをまったくチェックしないプラグインがありますが、たとえば、正規表現です。

一般的に、主流の製品である新しいリンターへの移行は、私たちにとってこれまで見られなかった機会の全世界を開くように見えます。じゃあ、やってみよう！

eslintをプロジェクトに追加する


ルールの移行については以下で説明します。それまでの間、eslintと連携するプロジェクトをセットアップします。
tslintを含むプロジェクトが既にある場合は、まずtslintに関連するすべてのパッケージを削除します。tslint自体、tslint-react、tslint-config-prettierなどです。

次に、eslintパッケージをプロジェクトに追加します（すべてをdevDependenciesとして設定します）。

  • エスリント自体;
  • @ typescript-eslint / parser-TypeScriptを解析するためのエンジン。
  • @ typescript-eslint / eslint-plugin-TypeScriptのルールセット

Eslint最小セットアップ


.eslintrc.json構成ファイルを作成します。 Eslintはその構成で多くのファイル形式をサポートしていますが、JSONが最も便利なようです。最小限の作業オプションは次のようになります。
{
    //  
    "env": {
		//   
		"browser": true,
		//   ES6
		"es6": true,
		//   ES2017
		"es2017": true
    },
    //  
    "extends": [
		//    eslint
		"eslint:recommended",
		//     
		"plugin:@typescript-eslint/eslint-recommended",
		//    TypeScript
		"plugin:@typescript-eslint/recommended",
		 //  TS,    
		"plugin:@typescript-eslint/recommended-requiring-type-checking"
	],
	//  
	"parser": "@typescript-eslint/parser",
	"parserOptions": {
		//    TS    
		"project": "tsconfig.json",
		"tsconfigRootDir": ".",
	},
	//      TypeScript
	"plugins": ["@typescript-eslint"],
	"rules": {}
}

このセクションでenvは、プロジェクトのオプションについてeslintに伝えます。私の例では、これはブラウザー用のプロジェクトです（つまり、コードはブラウザーで機能します）。 Node.JSの書き込み-ノードを設定：true。次の2つのオプションは、テストされるJSの方言を示します。一般的に、TypeScriptのコードをチェックしますが、プロジェクトにJSのコードもある場合は、それらを強化することを忘れないでください。自分自身では、これらのパラメーターをtsconfig.jsonのtargetと同じ値に設定することにしました。

式の最後に必要なセミコロンやスペース/タブなど、標準のeslintルールセットには、議論の余地はありません。すべてのルールはユニークに役立ちます。ここに含まれているルールとレベルを確認できます。無効にする

必要がある次の行ルールの半分。 TypeScriptでは機能せず、通常の操作の代わりに一連のエラーをスローするため、これは必要です。

次に、TypeScriptの推奨ルールを別のバッグに接続する必要があります。ここでは、一般的な構文規則（varの禁止など）がそのように機能することを覚えておく必要があります。

ただし、TSタイプを使用するルール（たとえば、@ typescript-eslint / no-unnecessary-type-assertion）の場合、TypeScriptエンジンが必要です。また、エンジンにはtsconfig.jsonファイルが必要です。このファイルへのパスを指定する必要があります。

tsconfig.jsonで、Dodo Pizza Engineeringは通常、プロジェクトでビルドされないように、excludeおよびthrow outテストを指定します。ただし、eslintを機能させるには、指定して含める必要があります。つまり、リントする必要のあるすべてのファイルをプロジェクトに明示的に含める必要があります。これがないと、eslintは見つけたすべてのファイルを誓います。「ファイルはプロジェクトにありません。私は何もしません。たくさんのエラーを投げます。」プロジェクトファイルを明示的に指定しないオプションがあります-パラメータを設定しますcreateDefaultProgram: true。これは、本質的には、「Parsiだけが見つかる」という意味です。ただし、パフォーマンスが大幅に低下するため、開発者はそうしないことを強くお勧めします。

ForkTsCheckerWebpackPluginを使用してTypeScriptファイルを処理する場合は、そのパラメーター（webpack.config.ts内）tslint: trueをに置き換えますeslint: true。

コマンドラインからリンターの起動を設定することも価値があります。その前に、この値をのスクリプトセクションに追加しますpackage.json。

"eslint": "eslint --cache --ext .js,.jsx,.ts,.tsx src",
"eslint:dump": "eslint --print-config ./.eslintrc.json",

最初の行は、プロジェクトをビルドせずにeslint検証を開始するだけです。2番目には現在のeslint設定が表示され、ルールパラメータの設定を確認できます。

このバージョンでは、eslintはすでにプロジェクトで機能しており、グローバル、未使用の変数などの再定義を行っています。

Visual Studio Codeのセットアップ


この方法をすべて完了したら、コマンドラインからリンターを開始できます。また、プロジェクトのビルド時に暗黙的に起動されます。ただし、Visual Studio Codeでは、リンターからのコメントは表示されません。どうして ？！

スタジオ用のeslintプラグイン（dbaeumer.vscode-eslint）があり、インストールする必要があります。その後は、何も機能せず、強調も修正もされません。どうして？プラグインには設定があるので、JavaScriptファイルでのみ作業する必要があることを示しています。

この卑劣な設定はUIで行われないため、スタジオ設定ファイルに移動して、必要な言語をeslint.validateパラメータに手動で追加する必要があります。言語の完全なリストは、スタジオのドキュメントの腸にあります。この設定は、次のようになります。

"eslint.validate": [
        "javascript",
        "javascriptreact",
        "typescriptreact",
        "typescript",
    ],

その後、スタジオを再起動すると、すべてが最終的に動作し始めます。

ルールを構成するために残ります


プロジェクトが設定されました。上記の例ではルールのリストが空だったので、ルールについて説明します。

tslintが、正式に正しいコードにめちゃくちゃになるのを止めなかったと言わざるを得ません。たとえば、待つのを忘れます。Eslintは、そのようなセマンティックエラーを見つけてそれを誓う方法を知っています。戻り値がPromiseであることを報告するために、何らかの理由で、awaitが書き込まれていません。これには、中程度の複雑さの文体上の問題も含まれます。ラムダや関数を使用するなど、Prettierはこれを実行できません。

単純なルールについて：括弧の位置、タブと スペースなどは、それらはPrettierまたは同様のパッケージに与えられるべきであると考えられています。しかし、リンターではとにかく残しておく必要があります。これが最後のフロンティアであり、倒れたプロジェクトアセンブリの怠慢な開発者を止めることができます。さらに、この行は自動化できます。たとえば、huskyを使用すると、コミットごとに自動的にリンターを開始できます。

私たちは、持っているtslintルールセットを移行しないことにしました。そして、独自のセットを最初から作成します。

eslintには事前定義されたルールセットがあります。

  • ESLint Recommendedは、ホリバーを生成しないという考えで作られた中立的なルールのセットです。明らかに必要なチェックだけが含まれています：未使用の変数など 後続のすべてのセットはこれを拡張します。
  • グーグル-既にホリバーの理由があります：インデントには厳密にスペースがあり、セミコロンが必要です。
  • AirBnB-必須のセミコロンを含む、厳密なスタイルのルールもあります。
  • 標準-セミコロンはここでは禁止されていますが、末尾のコンマも禁止されています。

既製のパッケージは好きではありませんでした。これは奇妙に聞こえるかもしれませんが、スタイルの戦争を回避して、新しいリンターに切り替えることが重要です。どこでもこのように書く場合（タブ、セミコロンなし、末尾のコンマは必須）、そのままにしておきます。主なことは、すべてのプロジェクトで同じであることです。

約束のセックス：独自のルールセット


正直なところ、eslintルールセットを表示することは、おっぱいを見せている女の子のようなものです。これ以上の秘密はありません。私はこれをするかどうか長い間考えました。でも、仲間の女の子と相談した結果、何に価値があるか決めました。

使用するプラグインから始めます。

  • react-反応コンポーネントコードをチェックします。基本的なルールと私たちのルール。重要なことから：私たちは純粋な機能コンポーネントに溺れています。
  • react-hooks-フックの使用に関する反応開発者からのルール。
  • promise-Promiseの使用時に一般的なエラーをチェックします。TypeScriptコードでは少し奇妙に動作します。重要なことから：コードを読みやすくするために、どこでもPromiseを使用し、コールバックを使用せずに/キャッチしようとします。
  • optimize-regexは、正規表現を改善するためのヒントを提供する楽しいプラグインです。正規表現は少しあるので、あまり役に立ちません。しかし、正規表現の魔法を持っているわけではありません。だからそれは便利ですが、尋ねない人がたくさんいます。
  • sonarjsは、コードの複雑さと一般的なリファクタリングエラーをチェックするFireプラグインです。1つ目は面白いことです。プラグインは、コードの複雑さを評価し、コードを単純化する価値があるときにアドバイスを提供します。多くの場合、リファクタリングエラーを検索すると、コードを簡略化したり、少なくとも読みやすさを向上させることができます。
  • @ typescript-eslint-TypeScriptコードをチェックするためのeslintルール。また、TSと互換性のない基本ルールを無効にするためのセット。
