JSONスキーマ。存在するかしないか？

建築：不必要に必要とする技術。

フレデリックキースラー

XML形式のSOAPプロトコルを使用するWebサービスの場合、信頼できる実績のあるソリューションは、XMLメッセージのデータ型と構造を記述するXMLスキーマ（xsd-scheme）の予備的な開発であることは誰にとっても秘密です。このアプローチにより、開発者には明確な利点があります。スキームで設定されるメッセージの構造に対して厳密な標準化されたルールがあり、ルールの数は有限であり、XML形式での新しいメッセージの検証を自動化することができます。

しかし、XML言語は、その重さ（XML heaviness）と、分散システム向けのREST（REpresentational State Transfer）ソフトウェア開発のアーキテクチャスタイルの普及により、JSON（JavaScript Object Notation）マークアップ言語に取って代わられたことも知られています。 RESTスタイル自体はJSONを使用する必要はありませんが（何も必要とは言えないが、「推奨」）、実際には、REST APIの開発では、JSONを使用してメッセージ本文を記述することがよくあります。

したがって、JSONメッセージを使用してREST APIを開発する慣行は、ロシア（そして私たちだけでなく）のITの生活にしっかりと浸透しましたが、XMLスキーマの形式でメッセージ構造を説明した経験により、一度にWebサービス開発者の生活が大幅に簡素化されました。 JSONメッセージの場合、永続的に無視されます。しかし、喜ばざるを得ない人すべてではありません。

XMLスキーマに精通している開発者が、ドキュメントの構文解析と検証ロジックを再発明するたびに車輪を再発明する必要に直面したとき、同様のJSONスキーマドラフトが形成されました。json-schema.orgから入手できます。、および変更の履歴と使用例に関する多数のドキュメント。また、ドラフトステータスで公開されているという事実にもかかわらず、さまざまな言語のすべての一般的な開発プラットフォームとライブラリで長い間サポートされてきました。JSONスキーマ

自体は、XMLスキーマよりも少ないメッセージ構造化機能を提供します。 XMLスキーマで簡単に説明できるものは、可能な場合でも、JSONスキーマを使用して繰り返すことは必ずしも簡単な作業ではありません。しかし、ここで私はこの事実を利点として考えます。どうして？ システムのアルゴリズムが単純で線形性が高いほど、信頼性が高くなり、ドキュメントの構造が単純になります。認識が容易になることなどが知られています。 引用せずにはいられない：



「独創的なものはすべてシンプルで、単純なものはすべて独創的です。」そして、スキームを使用してドキュメントの複雑な構造と受け入れ可能なオプションのセットを説明することができない場合、おそらくこのドキュメントの形成の構造とロジックを単純化する方向を検討する価値がありますか？

序文


それで、この記事は何ですか？

JSONスキーマによって送信されたJSONメッセージを記述することの利点にもっと注目したいと思います。「入り口」であるにもかかわらず、JSONスキームを使用せずにREST APIを開発することは常にシンプルで高速ですが、システムの成長に伴い、何らかの方法で欠落すると、システムの保守とサポートのコストが上昇します。また、メッセージ構造の予備調査は、データ交換とその処理に関する一般的なルールの不必要な重複なしに、メッセージ交換のより良い編成に貢献します。

また、JSONスキーマの可能性とそれを使用するためのルールについてロシア語圏のコミュニティーに情報を広めるために、この記事のフレームワークの具体的な例を使用して、私の経験を共有します。

JavaScriptオブジェクト表記（JSON）


JSON（English JavaScript Object Notation）は、JavaScriptに基づくテキストベースのデータ交換フォーマットです。

理論


JSONマークアップ言語は、限られた一連のデータ型を定義します。「キー」のペア「{キー "：値}」には常に文字列タイプを使用し、「値」には次のタイプを適用できます。文字列、数値、オブジェクト（JSONタイプ）、配列、ブール（trueまたはfalse）、null。

結論の代わりに


入荷時期です。では、JSONスキーマは最終的に何を提供できるのでしょうか？

  1. これにより、開発者の作業が楽になり、JSONメッセージを検証するためのコードが改善されます。
    つまり、ソフトウェアのサポートと統合が簡素化されます。
  2. サービスを開発し、システムの将来の開発のための「予備」を使用してデータのフォーマットと構成を検討することができます。
  3. ドキュメント指向、オブジェクト指向データベースのドキュメントの検証を適用します。
  4. JSON-Schemaは、APIのテストと文書化の時間を節約するのに役立ちます。
  5. 下位互換性APIサポートを簡素化します。
  6. データストリームを管理できます。
  7. プログラムの実行段階で取得された「列挙型」の値を使用して実行時にJSONスキーマを生成するときの柔軟な検証
    「列挙型」のステータスマシンまたはワークフローに使用できます（アプリケーション例Tsdk そして VolCh）
  8. JSONスキーマの実装に適用することができるDTO
    の（例えば、使用からアマルケビッチ）
