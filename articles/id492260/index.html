Konfigurasi jaringan


Jika layanan net.lo tidak ditambahkan ke runlevel boot, maka harus ditambahkan di sana:

rc-config add net.lo boot

Dengan cara yang sama, mereka dibuat dan dikonfigurasikan untuk berjalan untuk tingkat standar layanan bersih. br0 (jembatan untuk mesin virtual dan smartphone), bersih. enp0s25 (antarmuka kabel) dan bersih. wlp3s0 (antarmuka nirkabel), misalnya:

ln -s net.lo /etc/init.d/net.br0
rc-config add net.br0 default

Pengaturan proxy


Karena port normal (buram) dari server proxy akan diperbaiki ke alamat jembatan ( 192.168.120.1 ), tambahkan baris berikut ke /etc/rc.conf:

rc_squid_need="net.br0"

Pengaturan proxy berikut tidak akan dipertimbangkan dalam artikel ini:

  • Verifikasi identitas pengguna pada port reguler (tidak transparan);

Pengaturan firewall


Siapkan layanan nftables dan ulogd pada level run "default" (yang terakhir diperlukan untuk pemberitahuan paket):

rc-config add nftables default
rc-config add ulogd default

Karena akan ada jembatan di pengaturan firewall, tambahkan baris berikut ke /etc/rc.conf:

rc_nftables_need="net.br0"

Agar pemberitahuan berfungsi di /etc/sysctl.conf, Anda harus menambahkan baris berikut:

net.netfilter.nf_log.2 = nfnetlink_log

Agar file konfigurasi ada di folder / etc / nftables, kami membuat perubahan berikut di /etc/conf.d/nftables:

NFTABLES_SAVE="/etc/nftables/rules-save"

Untuk mencegah perubahan sementara agar tidak disimpan di file konfigurasi, buat perubahan lain ke /etc/conf.d/nftables:

SAVE_ON_STOP="no"

Saya akan menjelaskan tugas yang diberikan ke firewall:

  1. Aturan default di semua rantai tabel filter ip harus menghapus paket (bahkan di rantai output!). Demikian pula untuk tabel filter netdev;
  2. Pertukaran dalam bentuk apa pun, kecuali untuk IPv4 dan pertukaran layanan terkait (misalnya, ARP), ditekan sesegera mungkin (baik IPv4 maupun IPSec tidak digunakan);
  3. Paket multicast IPv4 juga dihapus sedini mungkin;
  4. Untuk melindungi ponsel cerdas Android (saat terhubung ke jembatan menggunakan aplikasi hostapd), pertukaran siaran yang dilakukan oleh mesin virtual yang berjalan di bawah sistem operasi Microsoft Windows tidak boleh ditransmisikan ke antarmuka nirkabel.
  5. Anda hanya dapat menggunakan perangkat Wi-Fi kami dengan alamat MAC yang dikenal.

Kustomisasi konsumen


Di PC itu sendiri


Untuk tes yang lebih menyeluruh dari server proxy transparan, saya memutuskan untuk menggunakannya untuk semua aplikasi (dengan beberapa pengecualian, lebih lanjut tentang itu nanti) pada PC itu sendiri.

Tambahkan sertifikat server proxy ke toko bersama:

mkdir -p /usr/local/share/ca-certificates
cp /etc/squid/squid.pem /usr/local/share/ca-certificates/squid.crt

Perbarui repositori:

update-ca-certificates

Aplikasi - Firefox dan browser web Chromium tidak menggunakan toko sertifikat umum, jadi itu harus ditambahkan ke toko yang sesuai dari aplikasi ini.

Buat file konfigurasi /etc/env.d/38proxy dengan baris berikut:

http_proxy=http://192.168.120.1:3128
https_proxy=http://192.168.120.1:3128
ftp_proxy=http://192.168.120.1:3128

Perbarui variabel lingkungan:

env-update

Setelah memasukkan kembali pengaturan akan diterapkan ke semua aplikasi.

Tentang pengecualian:

  1. Untuk berjaga-jaga, mari kita tinggalkan browser web Chromium, itu akan selalu terhubung ke jaringan secara langsung, tanpa server proxy, untuk ini kita akan mengubah file konfigurasi / etc / chromium / default dengan cara ini: 

    CHROMIUM_FLAGS="--enable-seccomp-sandbox ‚Äîno-proxy-server"
  2. Di hadapan variabel lingkungan https_proxy, aplikasi xfreerdp mencoba untuk terhubung melalui server proxy, jadi tambahkan baris berikut ke file konfigurasi shell: 

    alias xfreerdp='https_proxy= xfreerdp'
  3. Aplikasi youtube-dl tidak dapat memverifikasi sertifikat, mungkin karena sifat toko sertifikat Python. Solusi permanen belum ditemukan, jadi tambahkan baris berikut ke file konfigurasi shell:

    alias youtube-dl='youtube-dl --no-check-certificate'

Di Red Hat Enterprise Linux OS 6 dan 7 mesin virtual


Di mesin virtual, aktifkan sertifikat PEM:

update-ca-trust force-enable

Kami menulis ulang sertifikat server proxy dari PC ke mesin virtual:

scp squid.crt root@192.168.120.66:/etc/pki/ca-trust/source/anchors/

Di mesin virtual kami memperbarui sertifikat:

update-ca-trust extract

Pada mesin virtual dengan sistem operasi Microsoft Windows edisi yang berbeda


Sertifikat diunggah ke toko sertifikat umum mesin (bukan pengguna!) Di bagian "Root Tepercaya ...". Peramban web Firefox tidak menggunakan toko sertifikat umum, jadi sertifikat tersebut harus ditambahkan ke toko aplikasi yang sesuai.

Pada PC Raspberry Pi dengan Raspbian OS


Pembaruan aplikasi berhasil tanpa menginstal sertifikat.

Di ponsel cerdas dengan OS Android


Sertifikat dimuat sebelumnya ke perangkat, lalu dipasang menggunakan aplikasi manajemen sertifikat. Setelah itu, pengguna akan diminta untuk meningkatkan keamanan memasuki perangkat (jika belum dilakukan) menggunakan kunci grafis atau kata sandi.
