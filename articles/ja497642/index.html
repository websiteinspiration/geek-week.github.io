PVS-Studioを使用したGCC 10コンパイラのテスト


GCCコンパイラーは、マクロを多用するように作成されています。PVS-Studioを使用してGCCコードをもう一度チェックすると、マクロが悪いというチームの意見が再度確認されます。このようなコードは、静的アナライザーだけでなく、プログラマーにとっても理解するのが困難です。もちろん、GCCの開発者は既にプロジェクトに慣れており、プロジェクトに精通しています。しかし、側面から見ると、何かを理解することは非常に困難です。実際には、マクロのために、コード検証を完全に実行することはできませんでした。それにもかかわらず、PVS-Studioアナライザーは、いつものように、コンパイラーでもエラーを検出できることを示しました。

GCCコンパイラコードを再確認する時間


前回、 4年前にGCCコンパイラを確認しました。時間は速く、いつの間にか飛んでいき、どういうわけか私はすべてこのプロジェクトに戻ってそれを再確認するのを忘れていました。出版物「GCC 10の静的分析」はこの考えを押し戻しました。

実際、コンパイラに独自の静的コードアナライザが組み込まれていて、それらも開発中であることは秘密ではありません。したがって、PVS-Studioスタティックアナライザーがコンパイラー内でもエラーを検出でき、パンを食べているわけではないという記事をときどき書きます。

実際、従来の静的アナライザーとコンパイラーを比較することはできません。静的アナライザーは、コードのエラーを検索するだけでなく、開発されたインフラストラクチャーでもあります。たとえば、これはSonarQube、PlatformIO、Azure DevOps、Travis CI、CircleCI、GitLab CI / CD、Jenkins、Visual Studioなどのシステムとの統合です。これらは、大量の警告を抑制するために開発されたメカニズムであり、大規模な古いプロジェクトでもPVS-Studioの使用をすばやく開始できます。これは通知メーリングリストです。等々。ただし、とにかく最初に尋ねられる質問は、「PVS-Studioはコンパイラーが見つけられないものを見つけることができるのか？」です。したがって、これらのコンパイラー自体をチェックすることについて、何度も記事を書きます。

GCCプロジェクト検証の話題に戻りましょう。このプロジェクトについて詳しく説明する必要はありません。このプロジェクトの内容について話し合いましょう。

内部には、検証を妨害する多数のマクロがあります。まず、PVS-Studioアナライザーは多数の誤検知を生成します。それは何の問題もありませんが、彼が発行したレポートを取得して調査することはそれほど簡単ではありません。良い方法では、マクロの誤った警告を抑制するための作業を行う必要があります。それ以外の場合は、有用な警告がノイズの流れに溺れてしまいます。この設定は、この記事の執筆タスクの範囲を超えています。実際、私は完全に正直になります- 複雑なことは何もありませんが、私はこれを行うのが面倒です。ノイズが原因で、レポートの表示は非常に表面的なものでした。

第二に、プロジェクトに慣れていない人にとって、コードを理解することは非常に困難です。マクロ、マクロ...アナライザーが警告を生成する理由を理解するには、それらがデプロイされているものを確認する必要があります。とても厳しい。マクロは好きではありません。誰かがCのマクロなしではできないと言うかもしれません。しかし、GCCは長い間Cで書かれていませんでしたはい、歴史的な理由でファイルには拡張子が付いています。

これは明らかにCではなく、C ++です。

一般に、マクロとコーディングスタイルは、アナライザーレポートの調査を非常に困難にします。だから今回は様々なエラーの長いリストを喜ばせません。数杯のコーヒーを使うのが困難で、10個の興味深い断片を書きました。

PVS-Studio警告：V560条件式の一部は常にfalseです：

  • GET_MODE（xop1）== xmode1

static bool

PVS-Studioの警告：V501「&&」演算子の左側と右側に、同じサブ式「！Strcmp（a-> v.val_vms_delta.lbl1、b-> v.val_vms_delta.lbl1）」があります。dwarf2out.c 1481の

二つのstrcmp関数は、同じポインタを比較します。つまり、明らかに冗長なチェックが実行されます。以前の記事で、私はそれがタイプミスであり、実際に書く必要があることを示唆しました：

return (   !strcmp (a->v.val_vms_delta.lbl1, b->v.val_vms_delta.lbl1)
        && !strcmp (a->v.val_vms_delta.lbl2, b->v.val_vms_delta.lbl2));

ただし、4年間このコードは修正されていません。同時に、記事で説明したコードの疑わしいセクションについて著者に通知しました。今、私はこれが間違いであることを確信していません。おそらくこれは単なる冗長なコードです。この場合、式は簡略化できます。

return (!strcmp (a->v.val_vms_delta.lbl1, b->v.val_vms_delta.lbl1));
