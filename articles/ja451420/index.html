XSS


さて、最初のタイプの攻撃はXSSです。はい、誰もが聞いている古き良きXSS。XSS（クロスサイトスクリプティング）は、サイト訪問者を標的とする攻撃の一種です。これがどのように行われるか：攻撃者は入力フィールドを介して、データベースに侵入してその仕事をする悪意のあるコードを記述します。通常、このようにして、ユーザーからCookieが盗まれ、パスワードやログインなしでアカウントにログインできます。

何が起こったのですか？

JohnはJavaScriptコードを使用してコメントを追加しました。データをページに出力すると、テキストコメントがHTMLコードに変換されます。 HTMLタグは、scriptタグの使用を確認し、マークアップに追加しました。インタープリターはすでにJavaScriptコードを実行しています。つまり、ジョンは単にjsコードを既存のサイトコードに追加しただけです。

どのように修正しますか？

SQLインジェクション


もう忘れられ始めている、最も一般的なタイプの攻撃のもう1つ。準備されたリクエストとフレームワークがあるので、彼らはそれを忘れます。

準備されたリクエストについて話します。

攻撃の本質は何ですか。攻撃者はSQLクエリの一部を入力フィールドに入力し、フォームを送信します。クエリの実行中に、受信したデータがデータベースに追加されます。しかし、テキストにはコードがあるため、エントリを追加すると、スクリプトのロジックが変更されます。

さて、状況をシミュレートします。開発者はフォームをXSS攻撃から保護しました。そしてジョンは彼の知識を使い続け、不幸な開発者の欠陥を指摘しました。

つまり、コメントを追加するために同じフォームを引き続き使用します。
いくつかの変更を加えましょう：

1）コメントの事前モデレート。

つまり、モデレーターによって承認されたコメントのみがページに表示されます。記事の主要部分に気を取られないように、事前モデレーションは最も単純な形式で実装します。

アイデアを実装するには、コメント付きのテーブルにフィールド「is_moderate」を追加します。これには、「1」（コメントを表示）または「0」（表示しない）の2つの値を取ります。もちろん、デフォルトでは「0」です。

2）リクエストを変更します。

これは明確にするためです。コメントを追加するリクエストを次のようにします。

"INSERT INTO `comments` SET `username`='$username', `message`='$comment'"

フォーム作業コードは次のようになります。

Index.phpファイル
<?php
    $opt = [
            PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
            PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC
            ];

    $pdo->exec("SET CHARSET utf8");
        
    $query = $pdo->prepare("SELECT * FROM `comments` WHERE `is_moderate`='1'");
    $query->execute();
    $comments = $query->fetchAll();    

    if ($_POST) {
        $username = htmlspecialchars(trim($_POST['name']));
        $comment = htmlspecialchars(trim($_POST['comment']));

        $query = $pdo->prepare("INSERT INTO `comments` SET `username`='$username', `message`='$comment'");
        $query->execute();

        if ($query) {
            echo ' !';
        } else {
            echo ' !';
        }
    }
?>


さて、ジョンはサイトにログオンし、コメントが管理され始めたのを見て、開発者をあざけることにしました。さらに、XSS攻撃の形は今ではうまく反映されており、Johnはすでに楽しむ機会を奪われています。彼はこのタイプのコメント「LOL '、is_moderate =' 1」を残し、モデレートをバイパスします。

どうして？

クエリでジョンのコメントを置き換えると、引用符が壊れます。つまり、前述のように、Johnは任意のSQLコードを実行することができました。

Johnのコメントを含むリクエストを実行すると、リクエストは次のようになります。

"INSERT INTO `comments` SET `username`='John', `message`='LOL', `is_moderate`='1'"

さらに、SQLインジェクションは、フォームを送信するときだけでなく実装することもできます。これは、識別子でレコードを受け取ったり、検索フォームを処理したりする場合など、それほど明白ではない状況でも発生します。

直し方？

問題を解決する方法は、長い間知られており、準備されたクエリです。準備されたリクエストは、実行される前に特別な処理を行うリクエストです。処理は、追加の引用符をエスケープすることで構成されます。あなたはそのような特徴を聞いたに違いありません。 PHPでは、「\ '」のように実装されています。

最も一般的なソリューションはPDOです。 PDOはデータベースを操作するためのインターフェースです。かなり便利なインターフェースとは何ですか。あなただけがそれを賢く使う必要があります。

PDOは、マスクとプレースホルダーを使用して準備済みクエリを実装する機能を提供します。

次に、マスクを使用するときのリクエストは次のようになります。

Index.phpファイル
<?php
    $query = $pdo->prepare("INSERT INTO `comments` SET `username`=:username, `message`=:comment");

    $params = ['username' => $username,'comment' => $comment];
    $query->execute($params);


そして、このようなプレースホルダーを使用する場合：

Index.phpファイル
<?php 
    $query = $pdo->prepare("INSERT INTO `comments` SET `username`=?, `message`=?");

    $params = [$username,$comment];
    $query->execute($params);


現在、ジョンの攻撃は関係ありません。少なくともこのフォームでは。

ちなみに、私たちが実装したモデレーションは、この形式でも、別の種類の攻撃（スパム）からすでに保護しています。私たちは皆彼のことを聞いた。SPAMは、ソーシャルエンジニアリングの知識を利用して攻撃者が攻撃を実行するメッセージの送信です。攻撃されるのはモデレーターだけです。そして、もし彼がそれほど愚かでなければ、そのゴミをデータベースから削除するか、もし怠け者であれば、彼はその出版物を拒否し、それだけです。

CSRF攻撃


CSRF-クロスサイトリクエストフォージェリ。知っている人が少ないので危険です。これを行うことは非常に簡単ですが。

発生方法：別のサイトの攻撃者がフォームを偽造し、被害者にこのフォームに従うよう強制します。つまり、POST要求が送信されます。したがって、HTTPリクエストが偽装され、被害者のサイトで悪意のあるアクションが実行されます。

たとえば、攻撃者があなたに代わって友人にVKで手紙を送る可能性がありますが、あなたはそれを知らないでしょう。

少し紛らわしく聞こえます。実際に検討することを提案します。

どのように脆弱性を修正しますか？

ある時点で、開発者はcsrfが何であるかをグーグルし、保護ロジックは次のようになります。保護のために、csrfトークン（文字と数字のセット）を作成し、フォームに掛ける必要があります。同じトークンをユーザーに（たとえば、セッションを通じて）修正することも必要です。そして、フォームを処理するときに、これらのトークンを比較します。一致する場合、コメントを追加できます。

これを実装します。

開発者のサイトのindex.phpファイル
<?php
    session_start();

    $token = '';
    if (function_exists('mcrypt_create_iv')) {
        $token = bin2hex(mcrypt_create_iv(32, MCRYPT_DEV_URANDOM));
    } else {
        $token = bin2hex(openssl_random_pseudo_bytes(32));
    }

    $_SESSION['token'] = $token;
?>
...
<form action="action.php" method="POST">
    <input type="text" name="username">
    <textarea name="message"></textarea>
    <input type="hidden" name="csrf_token" value="<?php echo $token;?>">
    <input type="submit" value=" ">
</form>


Action.phpファイル
<?php
    session_start();

    if ($_POST) {
        if ($_SESSION['token'] == $_POST['csrf_token']) {
            echo ' !';
        } else {
            echo '!';
        }
    }


ブルートフォースとPublickのパスワード


おそらく最も有名なタイプの攻撃です。彼はハッカーなどに関するほとんどすべての最初の映画を聞いた。

本質は何ですか：サイト管理パネルに承認用のフォームがあります。Johnが知らないユーザー名とパスワードが必要です。しかし、彼は人気のあるユーザー名とパスワードを含むファイルを持っています。そして彼は私たちのサイトでそれらを試すために喜んで走ります。

開発者は何に反対することができますか？たとえば、特定の期間の認証試行回数の制限。

承認フォームの初期バージョンを次のようにします。

        $query->execute(['username' => $username,'password' => $password]);
        $find_user = $query->fetchAll();

        if ($find_user) {
            echo ' !';
        } else {
            echo '  !';
        }
    }
?>
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>Brute Force  Public Passwords</title>
</head>
<body>
    <form method="POST">

        <input type="password" name="password" placeholder="Password">

修正方法：既に述べたように、最も簡単なオプションは、一定期間あたりの認証試行回数を制限することです。

これを行うには、ログインしようとしたときに、cookieに現在の時間の値をユーザーに追加します。そして今、ログインしようとすると、ユーザーが5秒間に1回しかログインできないことがわかります。また、パスワードやログインが正しく入力されていない場合、次回の試行までのタイムアウトを5秒増やします。

Index.phpファイル
<?php 
    $count_next_minit = $_COOKIE['count_try'] ? $_COOKIE['count_try'] : 1;
    $seconds_to_new_try = 5;
    
    $opt = [PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
            PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC];
    $pdo = new PDO("mysql:host=localhost;dbname=".$db,$user,$pass,$opt);
    $pdo->exec("SET CHARSET utf8");

    if (isset($_POST['autorize'])) {
        if ($_COOKIE['last_try']) {
            if ($_COOKIE['last_try'] < time() - $seconds_to_new_try * $count_next_minit) {
                $username = htmlspecialchars(trim($_POST['login']));
                $password = htmlspecialchars(trim($_POST['password']));
                $query = $pdo->prepare("SELECT * FROM `users` WHERE `username`=:username AND `password`=:password");
                $query->execute(['username'=>$username,'password'=>$password]);
                $find_user = $query->fetchAll();

                setcookie('last_try', time(), time() + 3600);
                if ($_COOKIE['count_try']) {
                    $old_value = (int)$_COOKIE['count_try'];
                    setcookie('count_try', $old_value + 1, time() + 3600);
                } else {
                    setcookie('count_try', 1, time() + 3600);
                }

                if ($find_user) {
                    var_dump(' !');
                } else {
                    var_dump('  !');
                }
            }else{
                var_dump('   !    ' . $seconds_to_new_try * $count_next_minit . ' ');
            }
        }else{
            setcookie('last_try', time(), time() + 3600);
        }
    }
?>


バックトレース


バックトレースは、システムエラーメッセージを介して攻撃する方法です。これはMySQLとPHPの両方です。

たとえば、Johnは無効なURLを入力し、そのようなIDを持つデータベースにレコードがないというエラーを受け取りました（レコードがアドレスバーからIDを介して受信された場合-site.ru/article?id=12）。入力時にユーザーがエラーを表示する、いわゆる「ドック」-特定のパターンのウェブサイトアドレスもあります。そして、これにより、ジョンがボットを使用してこのアドレスのリストを調べ、サイトでこの脆弱性を見つけようとする可能性が開かれます。

直し方？この例では、エラー出力を閉じるだけで問題が解決されるため、例はありません。通常、これはホスティングによって実装され、管理パネルでログを別のファイルに書き込むことができますが、エラーの出力を自分で制限することは不必要ではありません。

これは、error_reporting（）関数を使用して実行できます。

引数には、E_ERROR、E_WARNING、E_PARSE、E_NOTICE、E_ALLがあります。名前はその名のとおりです。

たとえば、error_reporting（E_NOTICE）を使用すると、通知タイプのエラー（たとえば、$ _POST配列にデータがないことなどの警告）を除いて、すべてのエラーが非表示になります。
すべてのエラー（実際に必要）の出力を無効にするには、この関数を次のように使用する必要があります：error_reporting（0）

論理エラー


論理エラーは最悪のものの一部です。これらは不注意によるエラーだからです。それらは予期せずポップアップし、問題の根本がどこにあるのかさえわからないこともあります。これらはサイトのロジックのエラーです。

たとえば、管理パネルのいずれかのページで、セッション内の承認データとCookieの存在を確認するのを忘れている場合があります。そのため、彼らはすべてのユーザーがこのページにアクセスできるようにしました。

この場合、節約できるのは1つだけです。プログラムコードを作成するときは、ハッキングの方法を考えてください。

DDOS


DOSは、技術、特にコンピュータに対する攻撃の一種です。攻撃は、過負荷のためにマシンを無効にすることを目的としています。 DDOSは、より多くのコンピュータが攻撃に関与しているという点でのみ異なります。

つまり、ジョンは彼の友人に電話をかけ、彼らは一緒にサイトにリクエストを送信し始めます。ボットネットはそれと何をするのですか？ボットネットは感染したコンピューターの多くであり、攻撃者が自分の仕事（プロセスの開始など）をある程度制御できるという特徴があります。それらはサーバーが負荷に耐えられないほど多くの要求を送信し、せいぜい、非常にゆっくりと動作し始めるか、不特定の時間の間拒否します。

このような攻撃からの保護は、ホスト自体、またはCloudflareなどの特別なサービスによって提供されます。

保護の仕組み：Cloudflareサービスは、トラフィックが通過する独自のDNSサーバーを提供します。そこでフィルタリングされ、サービスの所有者と開発者だけが知っているアルゴリズムを通過します。その後、ユーザーはあなたのサイトにアクセスします。はい、もちろん、サーバー操作、ページ生成、その他すべてがありますが、今はそれについて話していません。
さらに、DDOSといえば、Cloudflareが提供するIPアドレスブロッキングについても言及する必要があります。

はい、これはすべて保護を100％保証するものではありませんが、攻撃時にサイトがフロート状態に留まる可能性が高くなる場合があります。

MITM


Man In The Middleは、攻撃者がパケットを傍受してスプーフィングするタイプの攻撃です。データはネットワークを介してパケットで送信されると聞いています。したがって、httpプロトコルを使用する場合、データは暗号化されていない通常の形式で送信されます。

たとえば、あなたは友人に「こんにちは」と書いて、彼は「私にお金を送ってください、これが財布です」を受け取ります。

この問題を解決するために、httpsプロトコルが作成されました。これを使用すると、データは暗号化され、ジョンは受信したトラフィックに対して何もできなくなります。
また、サイトのhttpsプロトコルを取得するには、SSL証明書を取得する必要があります。さて、またはTLS。一般に、TLSはSSL 3.0に基づいているため、本質的にSSLレシーバーです。彼らの仕事に大きな違いはありません。

SSL証明書は同じCloudflareを無料で提供します。

バックドア


もう少し理論。このタイプの攻撃では、さまざまな方法で実装でき、本質を理解するだけで済みます。バックドアは、スクリプト自体がバックグラウンドで何かを実行する一種の秘密攻撃です。ほとんどの場合、これらは急流からダウンロードされたプラグインまたはWordPressテーマです。プラグイン/テーマ自体は非常に適切に機能しますが、プラグイン/テーマコードに追加されたスクリプトの特定の部分は秘密裏に何かを行います。たとえば、同じスパム。これが、急流からファイルをダウンロードすることの望ましくないことに関するすべての警告の理由です。
