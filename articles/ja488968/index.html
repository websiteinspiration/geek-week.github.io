PostgreSQLのパフォーマンス問題の簡単な検出

時々パフォーマンスの問題に悩まされない非常に大規模で大規模なデータベースはありますか？そんなに多くはないでしょう。したがって、PostgreSQLを担当するすべてのDBA（データベース管理者）は、実際に何が起こっているかを見つけるために、潜在的なパフォーマンスの問題を追跡する方法を知る必要があります。

オプション設定後のPostgreSQLパフォーマンスの向上


多くの人々は、postgresql.confのパラメーターを変更することが成功への真の道だと考えています。ただし、これは常に当てはまるわけではありません。もちろん、ほとんどの場合、優れたデータベース構成オプションは非常に役立ちます。ただし、多くの場合、一部のアプリケーションロジックの奥に隠された奇妙な要求が原因で、実際の問題が発生します。実際の問題を引き起こしているクエリが、あなたが気づいたものではない可能性すらあります。自然な疑問が生じます：これらのリクエストをどのように追跡し、実際に何が起こっているのかを知るにはどうすればよいですか？このための私のお気に入りのツールはpg_stat_statementsで、PostgreSQL 9.2以降を使用している場合は常に有効になっているはずです（古いバージョンでは使用しないでください）。

pg_stat_statementsを有効にする


サーバーでpg_stat_statementsを有効にするには、postgresql.confの次の行を変更し、PostgreSQLを再起動します。

shared_preload_libraries = ‘pg_stat_statements’

このモジュールをサーバーにダウンロードした後、PostgreSQLは自動的に情報の収集を開始します。良い点は、モジュールのオーバーヘッドが非常に低いことです（オーバーヘッドは基本的に単なる「ノイズ」です）。

次に、次のコマンドを実行して、データへのアクセスに必要なビューを作成します。

CREATE EXTENSION pg_stat_statements;

拡張機能はpg_stat_statementsと呼ばれるビューを作成し、データに簡単にアクセスできるようにします。

PostgreSQLの遅いクエリの検出


最も興味深いクエリを見つける最も簡単な方法は、pg_stat_statementsの出力をtotal_timeでソートすることです。

	
SELECT * FROM pg_stat_statements ORDER BY total_time DESC;

ここでの美しさは、最も時間がかかるリクエストのタイプがリストの一番上に表示されることです。最善の方法は、最初のリクエストから10番目のリクエストに行き、そこで何が起こるかを確認することです。

私の意見では、データベースサーバーで最も時間のかかるクエリを表示せずにシステムを構成することは不可能です。

PostgreSQLの詳細なパフォーマンス分析

stddev_time列も確認すると非常に便利です。標準偏差が大きい場合、これらのクエリの一部は高速で一部は低速であると予想され、ユーザーエクスペリエンスの低下につながる可能性があります。

「行」列も非常に参考になります。 1,000回の呼び出しが1,000,000,000行を返したとします。実際、これは、各呼び出しが平均100万行を返したことを意味します。常に大量のデータを返すことはあまり良いことではないことを理解するのは簡単です。

特定のタイプのリクエストが不十分なキャッシュパフォーマンスを示すかどうかを確認する場合は、shared_ *が役立ちます。つまり、pg_stat_statementsが有効になっている場合、PostgreSQLは個々のクエリタイプごとにキャッシュヒット率を通知できます。

temp_blks_ *フィールドを確認することも意味があります。PostgreSQLがソートまたは実体化のためにディスクにアクセスする必要があるたびに、一時的なブロックが必要です。

最後に、blk_read_timeとblk_write_timeがあります。通常、track_io_timingが有効になっていない場合、これらのフィールドは空です。ここでの考え方は、特定のタイプのリクエストがI / Oに費やす時間を測定できるようにすることです。これにより、システムがI / OまたはCPUに関連付けられているかどうかの質問に答えることができます。ほとんどの場合、I / Oタイミングを有効にすることをお勧めします。これにより、重要な情報が得られます。

JavaとHibernateを使用する


pg_stat_statementsは適切な情報を提供します。ただし、場合によっては、構成変数が原因で要求が中止されることがあります。

test=# SHOW track_activity_query_size;
track_activity_query_size
---------------------------
1024
(1 row)

ほとんどのアプリケーションでは、1024バイトで十分です。ただし、HibernateまたはJavaを使用している場合、これは通常当てはまりません。Hibernateは非常に長いクエリをデータベースに送信する傾向があるため、関連する部分（FROM句など）が実行される前にSQLコードをカットできます。したがって、track_activity_query_sizeをより高い値（おそらく32.786）に増やすことは理にかなっています。

PostgreSQLのボトルネックを特定するための便利なクエリ
