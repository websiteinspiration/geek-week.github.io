ReactアプリケーションでのCSS-in-JSライブラリの隠れた価格

最近のフロントエンドアプリケーションでは、CSS-in-JSテクノロジーが非常に人気があります。問題は、通常のCSSよりも便利なスタイルを操作するためのメカニズムを開発者に提供することです。私を誤解しないでください。CSSは本当に好きですが、優れたCSSアーキテクチャを作成するのは簡単なことではありません。CSS-in-JSテクノロジーには、従来のCSSスタイルに比べていくつかの大きな利点があります。ただし、残念ながら、特定のアプリケーションでは、CSS-in-JSを使用するとパフォーマンスの問題が発生する可能性があります。この記事では、最も一般的なCSS-in-JSライブラリの高レベルの機能を分析し、それらを使用するときに発生する可能性がある問題のいくつかについて説明し、これらの問題を軽減する方法を提案します。



状況の概要


私の会社では、UIライブラリを作成することにしました。これは私たちにかなりの利益をもたらし、さまざまなプロジェクトでインターフェースの標準的なフラグメントを再利用することを可能にします。私はその仕事を引き受けたボランティアの一人でした。最も人気のあるCSS-in-JSライブラリのスタイリングAPIにはすでに慣れているので、CSS-in-JSテクノロジーを使用することにしました。仕事の過程で、私は合理的に行動するよう努めました。再利用可能なロジックを設計し、コンポーネントが共有するプロパティを適用しました。そのため、コンポーネントの構成を取り上げました。たとえば、コンポーネントはコンポーネントを<IconButton />拡張し、コンポーネント<BaseButton />は単純なエンティティの実装でしたstyled.button。残念ながら、それが判明したIconButton独自のスタイルが必要なので、このコンポーネントを様式化されたコンポーネントに変換しました。

const IconButton = styled(BaseButton)`
  border-radius: 3px;
`;

ライブラリに表示されるコンポーネントが増えるにつれて、使用する合成操作も増えていきました。これは不自然に見えませんでした。結局のところ、実際には、構成はReactの基礎です。コンポーネントを作成するまで、すべてが順調でしたTable。このコンポーネントのレンダリングが遅いと感じ始めました。特に、テーブルの行数が50を超える状況では、これは正しくありませんでした。したがって、私は開発者ツールに頼ることによって問題を理解し始めました。

ところで、なぜ開発者ツールインスペクタを使用してCSSルールを編集できないのか疑問に思った場合は、CSSStyleSheet.insertRule（）を使用しているためです。。これは、スタイルシートを変更する非常に迅速な方法です。しかし、その欠点の1つは、対応するスタイルシートをインスペクターが編集できなくなることです。

言うまでもなく、Reactによって生成されたツリーは本当に巨大でした。コンポーネントの数Context.Consumerが多すぎて、睡眠が奪われる可能性があります。実際のところ、通常のReactコンポーネントの作成に加えて、単一のスタイル付きコンポーネントがstyled-componentsまたはemotionを使用してレンダリングされるたびに、追加のコンポーネントも作成されますContext.Consumer。これは、対応するスクリプト（ほとんどのCSS-in-JSライブラリは、ページの実行中に実行されるスクリプトに依存する）が生成されたスタイルルールを正しく処理できるようにするために必要です。通常、これによって特別な問題が発生することはありませんが、コンポーネントがトピックにアクセスできる必要があることを忘れないでください。これContext.Consumerは、様式化された要素ごとに追加でレンダリングする必要性につながり、コンポーネントからテーマを「読み取る」ことができますThemeProvider。その結果、テーマのあるアプリケーションで様式化されたコンポーネントを作成すると、3つのコンポーネントが作成されます。これは1つのコンポーネントStyledXXXと2つ以上のコンポーネントContext.Consumerです。

確かに、Reactはすぐに仕事をするので、特にひどいことはありません。つまり、ほとんどの場合、心配する必要はありません。しかし、より複雑なコンポーネントを作成するために、いくつかの様式化されたコンポーネントが組み立てられるとどうなるでしょうか。この複雑なコンポーネントが長いリストの一部である場合、またはこれらのコンポーネントの少なくとも100がレンダリングされる大きなテーブルの場合はどうなりますか？ここでそのような状況では、私たちは問題に直面しています...

実行中のアプリケーションの分析


CSS-in-JSアプリケーションの速度が低下する問題に対する明確な答えは次のとおりです。「100個のCSS-in-JSライブラリがContext.Consumerコンポーネントごとに2つレンダリングすると言われています。」しかし、これらすべてについて注意深く考えると、Context.ConsumerこれはJS変数にアクセスするためのメカニズムにすぎません。もちろん、Reactは対応する値を読み取る場所を見つけるためにいくつかの作業を行う必要がありますが、これだけでは上記の測定結果を説明できません。この質問への本当の答えは、使用の理由を分析することによって見つけることができますContext.Consumer。実際のところ、ほとんどのCSS-in-JSライブラリは、ブラウザでのページ出力中に実行されるスクリプトに依存しており、ライブラリがコンポーネントスタイルを動的に更新するのに役立ちます。これらのライブラリは、ページのアセンブリ中にCSSクラスを作成しません。代わり<style>に、ドキュメント内のタグを動的に生成および更新します。これは、コンポーネントがマウントされたとき、またはそのプロパティが変更されたときに行われます。これらのタグには通常、ハッシュ化された名前が単一のReactコンポーネントにマッピングされる単一のCSSクラスが含まれます。このコンポーネントのプロパティが変更されると、対応するタグも変更される必要があります<style>。このプロセス中に行われていることを説明する方法は次のとおりです。 

  • タグに必要なCSSルールが再生成され<style>ます。
  • 前述のCSSルールを格納するために使用される新しいハッシュクラス名が作成されます。 
  • classname対応するReactコンポーネントのプロパティが、作成したクラスを指す新しいプロパティに更新されます。

たとえば、ライブラリを考えてみましょうstyled-components。コンポーネントを作成するとき、styled.divライブラリはこのコンポーネントに内部識別子（ID）を割り当て、HTML <head>タグに新しいタグを追加します<style>。このタグには、対応するスタイルが属するReactコンポーネントの内部識別子を参照する単一のコメントが含まれています。

<style data-styled-components>  
  /* sc-component-id: sc-bdVaJa */
</style>

そして、対応するReactコンポーネントを表示するときにスタイル付きコンポーネントライブラリが実行するアクションを次に示します。

  1. テンプレート文字列styled-componentsからCSSルールを解析します。
  2. 新しいCSSクラス名を生成します（または以前の名前を保持するかどうかを調べます）。
  3. スタイリスを使用してスタイルの前処理を実行します。
  4. 前処理の結果得られた CSSを対応する<style>HTMLタグに埋め込みます<head>。

このプロセスのステップ1のトピックを使用できるようにするには、Context.Consumerが必要です。このコンポーネントのおかげで、値はテンプレート文字列のトピックから読み取られます。このコンポーネントに関連付けられたタグをコンポーネントから変更できるようにするには、<style>もう1つ必要Context.Consumerです。これにより、スタイルシートのインスタンスにアクセスできます。これが、ほとんどのCSS-in-JSライブラリで2つのインスタンスに遭遇する理由Context.Consumerです。

これに加えて、これらの計算はすべてユーザーインターフェイスに影響を与えるため、コンポーネントのレンダリング段階で実行する必要があることに注意してください。これらは、Reactコンポーネントのライフサイクルのイベントハンドラーのコードでは実行できません（このようにすると、実行が遅延する可能性があり、ユーザーにとってページの形成が遅いように見えます）。これが、styled.divアプリケーションのレンダリングが通常のアプリケーションのレンダリングよりも遅い理由です。

これはすべて、styled-components開発者によって注目されています。コンポーネントを再レンダリングする時間を短縮するために、ライブラリを最適化しました。特に、ライブラリは、様式化されたコンポーネントが「静的」であるかどうかを調べます。つまり、コンポーネントのスタイルがテーマまたはコンポーネントに渡されるプロパティに依存するかどうか。たとえば、静的コンポーネントを以下に示します。

概要


CSS-in-JSテクノロジーは、かつてはスタイリングの分野における革命のように見えました。これにより、多くの開発者の作業が楽になり、追加の作業を行わなくても、名前の衝突やブラウザの製造元の接頭辞の使用など、多くの問題を解決できるようになりました。この記事は、人気のあるCSS-in-JSライブラリ（ページの実行中にスタイルを制御するライブラリ）がWebプロジェクトのパフォーマンスにどのように影響するかという疑問を明らかにするために書かれています。これらのライブラリがパフォーマンスに与える影響が、必ずしも顕著な問題の発生につながるとは限らないという事実に特に注意を払いたいと思います。実際、ほとんどのアプリケーションでは、この影響は完全に見えません。数百の複雑なコンポーネントを含むページを持つアプリケーションで問題が発生する可能性があります。

CSS-in-JSの利点は、通常、このテクノロジーを使用することの潜在的な悪影響を上回ります。ただし、CSS-in-JSの短所は、アプリケーションが大量のデータをレンダリングする開発者、プロジェクトに絶えず変化する多くのインターフェース要素が含まれる開発者が心に留めておく必要があります。アプリケーションがCSS-in-JSの悪影響を受けていると思われる場合は、リファクタリングを行う前に、すべてを適切に評価および測定することをお勧めします。
