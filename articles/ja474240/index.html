Orleans 3.0がリリースされました

これは、Orleansチームからのゲスト投稿です。Orleansは、.NETを使用して分散アプリケーションを作成するためのクロスプラットフォームフレームワークです。詳細については、https：//github.com/dotnet/orleansを参照して  ください。

Orleans 3.0のリリースをお知らせします。Orleans 2.0と比較すると、多くの改善と修正、およびいくつかの新機能があります。これらの変更は、幅広いシナリオと環境で本番環境でOrleansベースのアプリケーションを使用する多くの人々の経験と、フレームワークがより良く、より速く、より柔軟になるのを助けるグローバルOrleansコミュニティの熱意によって決まります。このリリースにさまざまな方法で貢献してくれたすべての人に感謝します。



Orleans 2.0と比較した主な変更点


Orleans 2.0は18か月以上前にリリースされ、それ以来フレームワークは大きな進歩を遂げています。2.0と比較したいくつかの変更：

  • 分散ACIDトランザクション
  • 場合によっては生産性が30％以上向上した新しいスケジューラー
  • 新しいRoslynベースのコードジェネレーター
  • クラスターメンバーシップを書き換えて、回復速度を向上
  • 共同ホスティングのサポート

その他の多くの改善と修正。

ネットワーク層をASP.NET Bedrockで置き換える


TLSを 使用して安全な通信を維持する   ことは、しばらくの間主要な関心事でした。さらに、コミュニティと内部パートナーの両方から  。バージョン3.0では、Microsoft.Orleans.Connections.Securityパッケージを介して利用可能なTLSサポートを追加します。詳細については、TransportLayerSecurityの例を参照してください  。

TLSサポートの実装は、以前のバージョンのOrleansでネットワークレイヤーがどのように実装されたかという主な懸念事項でした。TLSを実装SslStreamする最も一般的な方法であるTLSのサポートに簡単に適合させることができませんでした。 TLSを原動力として、Orleansネットワーク層を書き直しました。

Orleans 3.0では、ネットワークレイヤー全体が、ASP.NETグループの機能であるProject Bedrockに基づいて作成されたレイヤーに置き換えられて  います。 Bedrockの目標は、開発者が高速で信頼性の高いネットワーククライアントとサーバーを作成できるようにすることです。

ASP.NETチームとOrleansチームは協力して、ネットワーククライアントとサーバーの両方をサポートし、不可知論的であり、ミドルウェアを使用して構成できる抽象化を作成しました。これらの抽象化により、Orleans固有の内部ネットワークコードを変更せずに、構成を通じてネットワークの「トランスポート」を変更できます。 Orleans TLSサポートはBedrockミドルウェアとして実装されており、.NETエコシステム内の他のユーザーと共有できるように、それをユニバーサルにする予定です。

この革新の動機はTLSサポートの追加でしたが、夜間の負荷テストでは、スループットは平均で約30％増加します。

ネットワークレイヤーの書き換えには、カスタムバッファープーリングを使用法MemoryPool<byte> に置き換えることも含まれています  Span<T>。この変更により、シリアライゼーションの利点が増えました  。以前は呼び出しを行う専用スレッドによるロックに依存していた一部のコードパスが   、非同期メッセージパッシングにBlockingCollection<T>使用されるようになり  Channel<T>ました。これにより、専用スレッドが少なくなり、代わりに.NETスレッドプールに作業が移動します。

ジェネリックホストによる共同ホスティング


.NET Generic Hostの おかげで、同じプロセスでASP.NET Coreなどの他のプラットフォームとOrleansを共同ホスティングすることが以前より簡単になりました  。

以下は、ASP.NET CoreとともにOrleansをホストに追加する例です  UseOrleans。
var host = new HostBuilder()
  .ConfigureWebHostDefaults(webBuilder =>
  {
    //  ASP.NET Core
    webBuilder.UseStartup<Startup>();
  })
  .UseOrleans(siloBuilder =>
  {
    //  Orleans
    siloBuilder.UseLocalHostClustering();
  })
  .ConfigureLogging(logging =>
  {
    /*   ,    */
  })
  .ConfigureServices(services =>
  {
    /*    */
  })
  .UseConsoleLifetime()
  .Build();

//      .
await host.RunAsync();

この機能を使用して、展開トポロジを簡素化したり、既存のアプリケーションに機能を追加したりできます。一部のチームは 、ASP.NET Core Health Checksを使用してOrleansプロジェクトにKubernetesテストを追加するために内部で共有ホスティング  を使用してい  ます。

信頼性の向上


高度なうわさのおかげで、クラスターの障害からの回復が速くなりました。さらに、メッセージングエラーはより一貫して処理されるようになり、その結果、エラーが呼び出し元に返されます。これにより、開発者はバグをより早く見つけることができます。たとえば、メッセージを完全にシリアル化または逆シリアル化できない場合、詳細な例外が元の呼び出し元に渡されます。

拡張された拡張性


ストリームは、独自のデータアダプターを持つことができるようになりました。これにより、任意の形式でデータを受信できます。これにより、開発者はリポジトリでフローアイテムをどのように表現するかをより詳細に制御できます。また、ストリームプロバイダーはデータの書き込み方法を制御できるため、ストリームを既存のシステムやOrleans以外のサービスと統合できます。

トランザクションのカスタム状態は、トランザクションで果たすことができる役割を宣言できるようになりました。たとえば、トランザクションライフサイクルイベントをサービスバスキューに書き込むトランザクション状態の実装は、書き込み専用であるため、トランザクションマネージャの役割を果たせません。
