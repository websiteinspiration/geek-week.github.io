ドワーフ要塞タルンアダムスがゲーム開発について語る


長い間、エンターテインメントに強力なプロセッサを使用する最良の方法の1つは、ドワーフフォートレスでした。これは、全世界がASCII文字で構成されており、ギガバイトのメモリとプロセッサ時間の大部分を喜んで消費するゲームです。

しかし、他のいくつかのゲームとは異なり、DFの場合、プレイヤーは自分が必要とするすべてのものを本当に必要としていると感じます。彼女の詳細な計算は、建物、都市、商人、川、火山、モンスター、そしてもちろんノームで全世界を作り出します。一人がこれをすべて作成した場合、それは素晴らしい成果になります。ドワーフフォートレスは、これらすべてのオブジェクトを独自に作成するプログラムです。

ゲームの作者であるTarn Adamsは、彼の作成についての私たちの質問に答えることに同意しました。これは、多くの模造品が存在するにもかかわらず、依然として完全にユニークなゲームのままです。

Gamasutra：Dwarf Fortressが Steamとitch.ioでシンプルなインターフェースを備えた有料版で登場することを聞きました。これはあなたの次の治療費が原因であると彼らは言います。インディーズの開発者が浮かぶのは難しいですか？

Adams：はい、あなたはビジネスを離れないように多くの努力をしなければなりません、そしてほとんどすべての開発者は賃貸住宅や他の主要なニーズに支払うことができるように創造的な方法を発明することを余儀なくされています。また、処理できない問題が発生することもあります。私たちは今のところ幸運でしたが、時間をかけて変更する必要がありました。終了して新しい仕事を取得し、Patreonに切り替えて、今はSteamでかゆみを伴うか、鉛筆画を描きます。これは、ビデオゲームの作成とはかけ離れています。

ドワーフフォートレスの毎日のコードビルドはどのように機能しますか？どの言語を使用していますか？どんな図書館？それらを使用する場合、どのIDEですか？どのコンピューターでゲームを開発していますか？

私はWindows 10を搭載した通常の目立たないToshibaラップトップで作業しています。MSVCコミュニティのひどいC / C ++ミックスでコードを書いています。レガシーバージョンではOpenGLを使用し、メインバージョンではSDLを使用し、サウンドにはFModを使用します。私は何十年も使用しているMSVC（およびその前）からのヘッダーの一部を除いて、Windowsでは何も使用していません。 Linux / Mac版を定期的に開発していないので、Linux / Mac版で何が起こっているのか完全にはわかりません。

私は厳密な命名システムを使用しており、変数や関数の長い名前を保存しないので、数年経ってもすべてが読みやすくなっています。一般的に、私は未来を自分で管理しようとします。コード内のすべての私のコメントはこれを目的としています。 「ファイルを探す」機能を積極的に利用しています。しかし、たとえば、古いシステムを拡張したり、バグを修正したりするときに、何が起こるのかを改めて考えなければならない状況があります。この場合、調査には1時間以上かかることがあります。これにより、最初は考えていなかった有用なコメントを追加することができます。

Threetoe（ゲーム開発におけるTarnのパートナー）がストーリーを書いていて、それを実現できるゲームエンジンを作成しようとしていることを覚えています。これは今でも刺激的な仕事のやり方だと私には思えます。拒否したストーリーは複雑すぎて実装できませんでしたか？それらの1つのプロットがゲームで完全に繰り返されたことが起こったことがありますか？

ハ、ある意味で、それらはすべて複雑すぎると思います。キャラクターの動機、目標の設定などそれらがストーリーでどのように発生するかについて遅れを取り続けるそれでも、履歴を生成するための要素は常に軽いため、これは依然として有用なプロセスです。さらに、たとえ到達しなくても、キャラクターの基本的な仕組みにアプローチできます。

ウィキペディアによると、ゲームのバージョン番号（現在は.44）は、完了からどれだけ離れているか（つまり、44％）を示しています。ドワーフフォートレスには、将来何が待ち受けていますか？何が起こるかについての予見はありますか？実装するために残された深刻な側面は何ですか？

私は悪党とのリリースを終えています、それは次の数ヶ月でリリースされます。彼はかなり好奇心が強いはずです。次に、グラフィックを実装し、Steam / itchのバージョンのゲームの使いやすさを向上させます。次に、包囲プロセスを改善し、さらにいくつかの作業を行ってから、ビッグウェイトに進みます。これはDFの歴史の中で最大の再編と拡大です。作成に関する神話を生成し、完全に魔法の手続き型システムを作成し、世界のさまざまな部分を表示するためにいくつかのウィンドウを開くことができます。これはすばらしい追加になります。次に、プロパティ/法律/税関を含むリリースがあります。その後、注文はまだ決定されていませんが、経済、船舶、およびその他のまだ利用できない重要なコンポーネントに取り組みます。まだやることがたくさんあります！まだバージョン1.0に移行していません。しかし、バージョン1.0では、ゲームの開発は実際には終了しません...おそらく、そのリリースの時点までに、残り時間はほとんどありません。

プログラムは世界を構築するためにどのようなステップを踏んでいますか？

彼女はカードにメモリを割り当てます。次に、彼女は自分が持つ極（たとえば、北、南）を選択します（またはプレーヤーから渡されたパラメーターを考慮に入れます）。乱数ジェネレーターのシードは、さまざまなパラメーター（海、島のサイズ、その他の種類）を考慮して可変サイズのグリッドのマップフィールド（高度、降水量、温度、排水、火山活動、野生生物）の基本値を設定し、プログラムがそれらをフラクタルに塗りつぶします。温度の変化は極に依存し、プログラムは最も高いピークのポイントを選択します。ここで彼女は最初のパスを作成してプロセスがどのように実行されるかを確認し、マップが目的のパラメーターに合うように高さを変更しようとします。この段階で、世界が修正できない場合、それは破棄され、すべてが新たに始まります。

次に、高さ、降水量、温度などに応じて、一次導関数フィールドが設定されます-植生。プログラムは、バイオームがパラメーターで指定された間隔に対応しているかどうかをチェックします。この段階で、中間レベルの高さはより平坦な領域を作成するために平滑化され、火山は火山活動のフィールドに従って配置されます。

その後、浸食と河川の段階に進みます。小さな海は排水され、プログラムは山の斜面の端を見つけ、そこからテスト川を走らせることができます。さらに、プレーヤーがプロセスを追跡できるように、彼女はそれらの1つにカメラを備えています。多くの偽の川が海への道を見つけることができない場合、これらのポイントから流れ落ち、水路を壊します。高さが高すぎると、マップ全体が峡谷にならないように平坦化されることがあります。理想的には、鉱物系の物質を使用する必要がありますが、今のところ使用していません。湖は川のいくつかの地点で育ちます。


高さは山から海に向かって再び滑らかになり、山と火山のために局所的な調整が行われます。高さの作成が完了すると、プログラムは、雨の影と山岳地域の降水量に基づいて降水量を調整します。高さと降水量、および森林の抑制効果に基づいて、温度が再設定され、その後プログラムは新しい値を使用して最終的に植生レベルを設定します。塩分値は、海とその隣接タイルに設定されています。

これをすべて定義したら、バイオームの最後の領域の境界を見つけて、名前と個性を与えることができます。ここでは、地層と地下層も追加していますが、前述のように、地質情報は以前に追加されているはずです。次に、パラメータへの準拠の最終検証プロセスを実行して、プレイヤーが望んでいるものから大きく逸脱していないことを確認します。これが終了すると、プログラムは各地域の動物の世界の初期個体数を生成し、気象変数を設定します。

物語自体はこの瞬間から始まります。次に、文明と洞窟があります。次に何が起こるかを説明するのはかなり難しいですが、主な考え方は、大規模な戦略的ゲームは、プレーヤーのゼロとかなり恣意的な動きのルールと貧弱なAI（ただし数千のエージェント）でシミュレートされ、どの履歴が書かれているかに基づいています。シミュレーションログを使用したスト​​ーリーの手続き型生成は完全に機能するアプローチですが、欠点もあります。これは大変な作業であり、強調したい良い点を見つけるために後処理と調査を行う必要があり、ダイナミクスとメカニズムが十分でない場合、結果は退屈になる可能性があります。
