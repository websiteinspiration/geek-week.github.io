ジンブラとメール爆弾防御

電子メール爆撃は、最も古いタイプのサイバー攻撃の1つです。基本的には通常のDoS攻撃に似ていますが、さまざまなIPアドレスからのリクエストの波の代わりに、電子メールのシャフトがサーバーに送信され、大量のメールがいずれかのメールアドレスに届くため、負荷が大幅に増加します。このような攻撃は、メールボックスを使用できなくなる可能性があり、場合によってはサーバー全体の障害につながる可能性さえあります。このタイプのサイバー攻撃の長い歴史は、システム管理者に多くのプラスとマイナスの結果をもたらしました。肯定的な要素には、電子メール爆撃に関する十分な知識と、そのような攻撃を防御する簡単な方法の利用可能性が含まれます。マイナスの要因には、これらのタイプの攻撃を実行するための公に利用可能な多数のソフトウェアソリューション、および攻撃者が自分自身を検出から確実に保護する能力が含まれます。



このサイバー攻撃の重要な特徴は、利益のために使用することがほとんど不可能であることです。まあ、攻撃者はメールボックスの1つにメールのシャフトを送信しました。まあ、メールの通常の使用を許可していません。攻撃者は誰かの企業メールをハッキングし、GAL全体で何千ものメールを大量に送信し始めたため、サーバーがクラッシュしたか、速度が低下し始めました。使えなくなったので、次に何を？このようなサイバー犯罪をリアルマネーに変換することはほとんど不可能です。そのため、電子メール爆撃は現在非常にまれであり、インフラストラクチャを設計するときのシステム管理者は、そのようなサイバー攻撃から保護する必要性を覚えていないかもしれません。

それにもかかわらず、電子メール爆撃自体はかなり無意味な商業活動であるという事実にもかかわらず、それはしばしば他のより複雑で多段階のサイバー攻撃の不可欠な部分です。たとえば、メールをハッキングしてそれを使用して一部の公共サービスのアカウントを盗む場合、攻撃者はしばしば無意味な文字で被害者のメールボックスを「爆撃」し、確認メッセージがストリームで失われて気付かれないようにします。電子メール爆撃は、企業への経済的圧力の手段としても使用できます。したがって、顧客からのアプリケーションを受け取る企業のパブリックメールボックスへの積極的な攻撃は、顧客との作業を非常に複雑にし、その結果、機器のダウンタイム、未処理の注文、および評判の喪失と利益の損失につながる可能性があります。

そのため、システム管理者は、電子メールの爆撃を行う可能性を忘れず、常にこの脅威から保護するために必要な対策を講じる必要があります。これは、メールインフラストラクチャの構築段階でも実行できること、およびシステム管理者に時間と労力がほとんどかからないこと、インフラストラクチャに電子メールの爆撃からの保護を提供しないための客観的な理由が残るだけではない。このサイバー攻撃に対する保護がZimbra Collaboration Suite Open-Source Editionにどのように実装されているかを見てみましょう。

Zimbraの中心にはPostfixがあります。これは、現時点で最も信頼性が高く機能的なオープンソースメール転送エージェントの1つです。また、そのオープン性の主な利点の1つは、機能を拡張するためのさまざまなサードパーティソリューションをサポートしていることです。特に、Postfixはメールサーバー用の高度なサイバーセキュリティユーティリティであるcbpolicydを完全にサポートしています。スパムから保護し、ホワイト、ブラック、およびグレーリストを作成することに加えて、cbpolicydを使用すると、Zimbra管理者はSPF署名の検証を構成し、電子メールまたはデータの送受信の制限を設定できます。スパムやフィッシングメールに対する信頼性の高い保護を提供するだけでなく、メールの爆弾からサーバーを保護することもできます。

システム管理者が最初に必要とするのは、インフラストラクチャMTAサーバー上のZimbra Collaboration Suite OSEにプリインストールされているcbpolicydモジュールのアクティブ化です。これは、コマンドzmprov ms `zmhostname` + zimbraServiceEnabled cbpolicydを使用して行われます。その後、cbpolicydを快適に管理できるように、Webインターフェイスをアクティブにする必要があります。これを行うには、Webポート番号7780で接続を有効にし、コマンドln -s / opt / zimbra / common / share / webui / opt / zimbra / data / httpd / htdocs / webuiを使用してシンボリックリンクを作成してから、設定ファイルを編集しますnano /opt/zimbra/data/httpd/htdocs/webui/includes/config.phpコマンドを使用します。次の行を記述する必要があります。

$ DB_DSN = "sqlite：/opt/zimbra/data/cbpolicyd/db/cbpolicyd.sqlitedb";
$ DB_USER = "root";
$ DB_TABLE_PREFIX = "";

その後は、zmcontrol restartおよびzmapachectl restartコマンドを使用してZimbraおよびZimbra Apacheサービスを再起動するだけです。その後、example.com：7780 / webui / index.phpのWebインターフェースにアクセスできます。主なニュアンスは、このWebインターフェースへの入り口はいかなる方法でも保護されていないことです。許可されていない人がそこに入るのを防ぐために、Webインターフェースへの入り口ごとにポート7780の接続を閉じることができます。

電子メールを送信するためのクォータは、cbpolicydを使用して設定できるため、内部ネットワークから送信される手紙の流れから身を守ることができます。このようなクォータにより、1つのメールボックスから1つの時間単位に送信できる文字の最大数に制限を設定できます。たとえば、会社のマネージャーが1時間あたり平均60〜80通の電子メールを送信する場合、マージンが少ない場合、1時間あたり100通の電子メールの割り当てを設定できます。このような割り当てを使い果たすには、マネージャーは36秒ごとに1通の手紙を送る必要があります。一方では、これで十分に機能し、他方では、そのようなクォータがあれば、上司の1人のメールにアクセスする攻撃者は、企業への電子メール爆撃や大量のスパム攻撃を仕掛けることはありません。

このようなクォータを設定するには、Webインターフェイスでレターを送信するための新しい制限ポリシーを作成し、ドメイン内で送信されたレターと外部アドレスで機能するレターの両方に機能するように指定する必要があります。これは次のように行われます。



その後、レターの送信に関連する制限をより詳細に指定できます。特に、制限が更新されるまでの時間間隔と、制限を超えたユーザーが受信するメッセージを設定できます。その後、手紙の送信制限を設定できます。送信文字数と送信情報のバイト数の両方で設定できます。この場合、指定された制限を超えて送信されるレターでは、別の方法で行います。たとえば、メッセージをすぐに削除したり、メッセージの送信制限を更新した直後に保存するように保存したりできます。 2番目のオプションは、従業員が電子メールを送信するための最適な制限を識別するときに使用できます。

レターの送信に関する制限に加えて、cbpolicydではレターの受信制限を設定できます。このような制限は、一見すると、電子メール爆撃に対する優れた解決策ですが、実際には、このような制限を設定すると、たとえ大きな条件であっても、特定の条件下では重要な手紙が届かない場合があります。そのため、受信メールの制限を含めることは強くお勧めできません。ただし、まだチャンスがあると判断した場合、着信メッセージの制限の設定に近づくには、特別な注意を払う必要があります。たとえば、信頼できる取引先からの受信メールの数を制限して、メールサーバーが侵害された場合に会社にスパム攻撃を行わないようにすることができます。

電子メールの爆撃中に受信メッセージのシャフトから身を守るために、システム管理者は単に受信メールを制限するよりも賢いことをする必要があります。そのような解決策は、グレイリストの使用かもしれません。彼らの行動の原則は、信頼できない送信者からメッセージを配信しようとする最初の試みであり、サーバーへの接続が突然中断されるため、メッセージの配信が失敗する理由です。ただし、特定の期間中に信頼できないサーバーが同じメッセージを再度送信しようとした場合、サーバーは切断されず、配信は成功します。

これらすべてのアクションの意味は、電子メールの自動大量メール送信プログラムは通常、送信されたメッセージの配信の成功を確認せず、2回目の送信を試みないということです。一方、ユーザーは手紙がアドレスに送信されたかどうかを確実に確認します

cbpolicyd Webインターフェイスでグレーリストを有効にすることもできます。すべてが機能するためには、サーバー上のユーザー宛てのすべての受信メッセージを含むポリシーを作成し、このポリシーに基づいてグレイリストルールを作成する必要があります。ここで、cbpolicydが見知らぬ人からの繰り返しの応答を待つ間隔を設定できます。送信者。通常は4〜5分です。同時に、グレーリストを構成して、さまざまな送信者からの手紙の配信の成功および失敗の試みをすべて考慮し、それらの番号に基づいて、送信者をホワイトリストまたはブラックリストに自動的に追加するかどうかを決定できます。

グレイリストの使用には最大限の責任をもって取り組むべきであるという事実に注意を向けます。このテクノロジーの使用は、企業にとって本当に重要な文字を失う可能性を排除するために、ホワイトリストとブラックリストを常に維持することと密接に関連していることが最善です。

さらに、SPF、DMARC、およびDKIMのチェックを追加すると、メール爆弾からの保護に役立ちます。多くの場合、メール爆弾処理中に受け取った手紙はそのようなチェックに合格しません。これを行う方法は、以前の記事の1つで説明されています。

したがって、メール爆弾などの脅威から身を守ることは非常に簡単であり、企業のZimbraインフラストラクチャを構築する段階でもこれを行うことができます。ただし、そのような保護を使用することによるリスクが、得られるメリットを超えないようにすることが常に重要です。
