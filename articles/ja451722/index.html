Qt-非同期非同期ウィジェットライブラリ

全てにおいて良い日。私のqt-asyncプロジェクトについて少しお話ししたいと思います。多分それは誰かにとって興味深く、あるいは役に立つように思えるかもしれません。

非同期性とマルチスレッド化は、開発者の日常生活に長い間真剣に組み込まれてきました。多くの最新の言語とライブラリは、非同期での使用を考慮して設計されています。 C ++言語もこの方向にゆっくりと動いています。std:: thread、std :: promise / futureが登場し、コルーチンとネットワークを導入しようとしています。 Qtライブラリーも遅れることはなく、類似のQThread、QRunnable、QThreadPool、QFutureなどを提供します。同時に、Qtで非同期アクションを表示するためのウィジェットが見つかりませんでした（おそらく見栄えが悪かったので、間違っている場合は修正してください）。

したがって、私は欠点を補い、そのようなウィジェットを自分で実装しようとすることにしました。マルチスレッド開発は複雑ですが、興味深いビジネスです。

ウィジェットの実装に進む前に、ウィジェットの形でユーザーに提示するモデルをウィンドウの形式で説明する必要があります。最も一般的な形式では、ウィジェットの操作は次のように見えます。ある時点で、ユーザーまたはシステムが非同期操作を開始します。この時点で、ウィジェットは操作の進行状況または単に操作の表示を示します。オプションで、ユーザーは操作をキャンセルできます。次に、非同期操作は2つの方法で完了します。エラーが発生してウィジェットにエラーが表示されるか、ウィジェットに正常な操作の結果が表示されます。

したがって、モデルは次の3つの状態のいずれかになります。

  1. 進行中-非同期操作が進行中です
  2. エラー-非同期操作が失敗しました
  3. 値-非同期操作が正常に完了しました

各状態で、モデルは対応するデータを格納する必要があるため、AsyncValueモデルを呼び出しました。非同期操作自体はモデルの一部ではなく、状態を切り替えるだけであることに注意してください。AsyncValueは任意の非同期ライブラリで使用できることがわかり、簡単な使用パターンが確認できます。

  1. 非同期操作の開始時に、AsuncValueをProgressに設定します
  2. 最後に-操作の成功に応じて、エラーまたは値のいずれかで
  3. 必要に応じて、操作中に、ユーザーが操作を停止する機会がある場合は、進行状況データを更新し、停止フラグをリッスンできます。

QRunnableを使用した例を以下に示します。

class MyRunnable : public QRunnable
{
public:
    MyRunnable(AsyncValue& value)
     : m_value(value)
    {}

    void run() final
    {
        m_value.setProgress(...);

        // do calculation

        if (success)
            m_value.setValue(...);
        else
            m_value.setError(...);
    }

private:
    AsyncValue& m_value;
}

std :: threadを操作するための同じスキーム：

AsyncValue value;
std::thread thread([&value] () {
    value.setProgress(...);

    // do calculation

    if (success)
        value.setValue(...);
    else
        value.setError(...);
});

したがって、クラスの最初のバージョンは次のようになります。

template <typename ValueType_t, typename ErrorType_t, typename ProgressType_t>
class AsyncValue
{
public:
    using ValueType = ValueType_t;
    using ErrorType = ErrorType_t;
    using ProgressType = ProgressType_t;

    // public API

private:
    QReadWriteLock m_lock;
    std::variant<ValueType, ErrorType, ProgressType> m_value;
};

マルチスレッドをサポートするクラスに出くわす人なら誰でも、そのようなクラスのインターフェースがシングルスレッドの類似物とは異なることを知っています。たとえば、サイズ（）関数は役に立たず、マルチスレッドベクトルでは危険です。ベクトルは現時点で別のスレッドで変更できるため、その結果はすぐに無効になる可能性があります。

AsyncValueクラスのユーザーは、クラスデータにアクセスできる必要があります。データのコピーを提供すると負荷がかかる可能性があり、ValueType / ErrorType / ProgressTypeのタイプはどれも重い可能性があります。内部データへのリンクの発行は危険です-いつでも無効になる可能性があります。以下のソリューションが提案され

ています。1.対応するデータを受け取るラムダが渡される関数accessValue / accessError / accessProgressを介してデータへのアクセスを提供します。例えば：

template <typename Pred>
bool accessValue(Pred valuePred)
{
    QReadLocker locker(&m_lock);

    if (m_value.index() != 0)
        return false;

    valuePred(std::get<0>(m_value));
    return true;
}

したがって、内部値へのアクセスは参照によって実行され、読み取りロックの下にあります。つまり、アクセス時のリンクが無効になることはありません。

2. accessValue関数のAsyncValueユーザーは、stateChanged信号をサブスクライブし、信号の処理後にこのリンクを使用する必要がない場合、内部データへのリンクを記憶できます。彼女は無効になります。

このような状況では、AsyncValueコンシューマーは有効で便利なデータアクセスを常に保証されています。このソリューションには、AsyncValueクラスの実装に影響を与えるいくつかの影響があります。

まず、クラスは状態が変化したときにシグナルを送信する必要がありますが、同時にテンプレートです。基本的なQtクラスを追加する必要があります。ここで、ウィジェットがコンテンツを更新する信号を決定でき、関係者全員が内部データへのリンクを更新します。

lass AsyncValueBase : public QObject
{
    Q_OBJECT
    Q_DISABLE_COPY(AsyncValueBase)

signals:
    void stateChanged();
};

第2に、信号を送信する瞬間は読み取り用にブロックする必要があり（誰もが信号を処理するまでAsyncValueを変更できないようにする必要があります）、最も重要なのは、その瞬間に新しいデータと古いデータへの有効なリンクが存在することです。信号を送信する過程で、一部のAsyncValueコンシューマーは引き続き古いリンクを使用し、信号を処理したコンシューマーは新しいリンクを使用します。

std ::バリアントは適しておらず、データを動的メモリに格納して、新しいデータと古いデータのアドレスが変更されないことがわかります。

少し余談です。

動的割り当てを必要としないAsyncValueクラスの他の実装を検討できます。

  1. AsyncValue内部データのコピーのみをユーザーに提供します。以前に書いたように、データが大きい場合、そのようなソリューションは次善の策になる可能性があります。
  2. 1つではなく2つの信号を定義します：stateWillChange / stateDidChange。最初の信号で古いリンクを削除し、2番目の信号で新しいリンクを受信するようにコンシューマーに義務付ける。このスキームは、AsyncValueコンシューマを不必要に複雑にします。AsyncValueへのアクセスが拒否される時間間隔があります。

ご覧のとおり、書き込み用にm_lockロックを増やし、読み取り用に戻す必要があります。残念ながら、QReadWriteLockクラスではそのようなサポートはありません。QMutex / QReadWriteLockのペアで目的の機能を実現できます。以下は、AsyncValueクラスの実装で、実際に近いものです。

//   AsyncValue
enum class ASYNC_VALUE_STATE
{
    VALUE,
    ERROR,
    PROGRESS
};
Q_DECLARE_METATYPE(ASYNC_VALUE_STATE);

//       
class AsyncValueBase : public QObject
{
    Q_OBJECT
    Q_DISABLE_COPY(AsyncValueBase)

signals:
    void stateChanged(ASYNC_VALUE_STATE state);

protected:
    explicit AsyncValueBase(ASYNC_VALUE_STATE state, QObject* parent = nullptr);

    //     PromoteToWriteLock/DemoteToReadLock
    QMutex m_writeLock;
    QReadWriteLock m_contentLock;
    //  
    ASYNC_VALUE_STATE m_state;
};

template <typename ValueType_t, typename ErrorType_t, typename ProgressType_t>
class AsyncValueTemplate : public AsyncValueBase
{
    // 
    struct Content
    {
        std::unique_ptr<ValueType_t> value;
        std::unique_ptr<ErrorType_t> error;
        std::unique_ptr<ProgressType+t> progress;
    };
    Content m_content;

public:
    using ValueType = ValueType_t;
    using ErrorType = ErrorType_t;
    using ProgressType = ProgressType_t;

    //   
    template <typename... Args>
    void emplaceValue(Args&& ...arguments)
    {
        moveValue(std::make_unique<ValueType>(std::forward<Args>(arguments)...));
    }

    //   
    void moveValue(std::unique_ptr<ValueType> value)
    {
        //      
        Content oldContent;

        //   emplaceXXX/moveXXX   
        QMutexLocker writeLocker(&m_writeLock);
        {
            //      
            QWriteLocker locker(&m_contentLock);

            //     
            oldContent = std::move(m_content);
            //   
            m_content.value = std::move(value);
            //   
            m_state = ASYNC_VALUE_STATE::VALUE;

            //    
        }

        //  
        emitStateChanged();

        //    emplaceXXX/moveXXX 
        //   
    }

    //   value
    void emplaceError(Args&& ...arguments);
    void moveError(std::unique_ptr<ErrorType> error);
    void emplaceProgress(Args&& ...arguments);
    void moveProgress(std::unique_ptr<ProgressType> progress);

    template <typename Pred>
    bool accessValue(Pred valuePred)
    {
        //    
        QReadLocker locker(&m_contentLock);

        //   
        if (m_state != ASYNC_VALUE_STATE::VALUE)
            return false;

        //     
        valuePred(*m_content.value);

        //    
        return true;
     }

    //  accessValue
    bool accessError(Pred errorPred)
    bool accessProgress(Pred progressPred)
};

ご覧のとおり、AsyncValueが対応する状態になるまで待機せず、単にfalseを返すaccessXXX関数があります。AsyncValueに値またはエラーが表示されるまで同期的に待機すると便利な場合があります。実際、std :: future :: getの類似物が必要です。関数のシグネチャは次のとおりです。

template <typename ValuePred, typename ErrorPred>
void wait(ValuePred valuePred, ErrorPred errorPred);

この関数が機能するためには、条件変数-あるスレッドで予期され、別のスレッドでアクティブ化される同期オブジェクトが必要です。待機関数では待機する必要があり、AsyncValueの状態をProgressからValueまたはErrorに変更する場合は、ウェイターに通知する必要があります。

AsyncValueクラスに別のフィールドを追加すると、待機関数が使用されるときにまれに必要になるため、このフィールドをオプションにすることはできますか？答えは明らかです。もちろん、std :: unique_ptrを格納し、必要に応じて作成すれば可能です。 2番目の質問が発生しました-このフィールドをオプションにして、動的割り当てを行わないようにすることは可能ですか？気になっている人は、次のコードを見てください。主な考え方は次のとおりです。最初の待機呼び出しはスタックにQWaitCondition構造を作成し、そのポインターをAsyncValueに書き込みます。その後の待機呼び出しは、ポインターが空でないかどうかを確認し、このポインターによって構造を使用します。ポインターが空の場合は、最初の待機呼び出しについて上記を参照してください。 。

class AsyncValueBase : public QObject
{
    ...
    struct Waiter
    {
        //     
        QWaitCondition waitValue;
        //   wait
        quint16 subWaiters = 0;
        //  wait    
        QWaitCondition waitSubWaiters;
    };
    //   
    Waiter* m_waiter = nullptr;
};

    template <typename ValuePred, typename ErrorPred>
    void wait(ValuePred valuePred, ErrorPred errorPred)
    {
        //   -     
        if (access(valuePred, errorPred))
            return;

        //  AsyncValue  
        QMutexLocker writeLocker(&m_writeLock);
        //    
        if (access(valuePred, errorPred))
            return;

        //    wait 
        if (!m_waiter)
        {
            //  Waiter  
            Waiter theWaiter;

            //       if
            SCOPE_EXIT {
                //     wait,
                //    theWaiter 
                if (m_waiter->subWaiters > 0)
                {
                    //    subWaiters  
                    do
                    {
                         m_waiter->waitSubWaiters.wait(&m_writeLock);
                    } while (m_waiter->subWaiters != 0);
                }

                //   wait  ,
                //       Waiter
                m_waiter = nullptr;
            };

            //    Waiter  AsyncValue
            //    wait  
            m_waiter = &theWaiter;

            //   AsyncValue     Value  Error
            //   
            do
            {
                m_waiter->waitValue.wait(&m_writeLock);
            } while (!access(valuePred, errorPred));
        }
        //   wait  
        else
        {
            //       else
            SCOPE_EXIT {
                //     
                m_waiter->subWaiters -= 1;
                //     ->   wait
                if (m_waiter->subWaiters == 0)
                    m_waiter->waitSubWaiters.wakeAll();
            };

            //     
            m_waiter->subWaiters += 1;

            //   AsyncValue     Value  Error
            //   
            do
            {
                m_waiter->waitValue.wait(&m_writeLock);
            } while (!access(valuePred, errorPred));
        }
    }

すでに述べたように、AsyncValueには、特定のライブラリにバインドされないように非同期コンピューティングを行うメソッドがありません。代わりに、何らかの方法で非同期を実装する無料の関数が使用されます。以下は、スレッドプールでAsyncValueを計算する例です。

template <typename AsyncValueType, typename Func, typename... ProgressArgs>
bool asyncValueRunThreadPool(QThreadPool *pool, AsyncValueType& value, Func&& func, ProgressArgs&& ...progressArgs)
{
    //   
    auto progress = std::make_unique<typename AsyncValueType::ProgressType>(std::forward<ProgressArgs>(progressArgs)...);
    //    
    auto progressPtr = progress.get();

    //    AsyncValue
    if (!value.startProgress(std::move(progress)))
        return false;

    QtConcurrent::run(pool, [&value, progressPtr, func = std::forward<Func>(func)](){
        SCOPE_EXIT {
            //     AsyncValue,   
            value.completeProgress(progressPtr);
        };

        //  AsyncValue
        func(*progressPtr, value);
    });

    return true;
}

ライブラリは、ネットワーク要求を処理するためのasyncValueRunNetworkと、新しく作成されたスレッドで操作を実行するasyncValueRunThreadの2つの類似した関数を実装しています。ライブラリのユーザーは簡単に独自の関数を作成し、そこで使用して、他の場所で使用する非同期ツールを使用できます。
