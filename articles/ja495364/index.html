APIスタイルガイド、またはユーザーに考えさせない



こんにちは！ Lesha Rutskoiと申します。Wrikeのプロダクトマネージャーです。それ以前は、AdformとPandaDocで働いていました。過去5年間、私は統合とAPIに関連するすべてを行ってきました。

WrikeはSaaSコラボレーションおよびプロジェクト管理製品です。開発者にはWrikeに基づいたソリューションを構築してもらいたいので、そのためにはAPIが便利である必要があります。同時に、世界中に9つのオフィスがあり、そのうち3つは開発オフィスです。異なる言語を話す分散チームを使用して一貫したAPIを作成することは非常に困難です。彼らの決定が互いに矛盾し始める可能性が高まっています。この場合、万人のための統一されたルールのセットなしでは済まされません。

分散して作業して独自のAPIを作成する場合は、スタイルガイドAPIが役立ちます。それが解決する一般的な問題と、それが開発者にとってどのように生活を楽にするかについてお話したいと思います。また、社内で独自のAPIスタイルガイドを作成および実装した経験も共有します。

なぜ便利なAPIが必要なのですか？


多くの読者はおそらくそれほど便利ではないAPIを使用して問題を解決しなければなりませんでした。はい、これはそれほど単純ではなく、時間がかかりますが、通常は問題を解決できます。

では、なぜAPIを改善するために時間とチームの努力を無駄にするのでしょうか。答えは非常に簡単です。製品の使用に関する決定は、特にAPIと統合に関しては、開発者によって行われることがよくあります。したがって、ユーザーエクスペリエンス（UX）だけでなく、DX（開発者エクスペリエンス）についても考えることが重要です。

例を挙げましょう。私はかつて、すべての製品統合を担当するチームで働いていました。多くの場合、私たちの計画がすでに概説されていた四半​​期の真ん中に、マーケティング担当者が来て言った。間もなく、彼らは私たちの統合と成功について話すことができるクールな公開イベントを開催します！今すぐにこの統合をすぐに書けばすばらしいでしょう。」この場合、私たちは何をしましたか？チームの経験豊富な開発者が3日間で最低限のプロトタイプを作成できれば、ほとんどの場合2〜3週間でタスクを完璧な状態にすることができました。数日足りなかった場合、APIとドキュメントはまあまあでした。私たちが計画を再作成して新しい統合を行うことは無意味でした-それでも、期日までに価値のあるものは何もなかったでしょう。製品がこのような状況にある可能性があることを忘れないでください。パートナーは、APIの使用を迅速かつ便利に開始できるかどうかに応じて、対処するかどうかを決定します。

しかし、悪いAPIを設計したいと思う人はほとんどいません。なぜ皆が違うのですか？通常、APIはこのように開発されます。スタートアップが成長し始めて最初のクライアントが来ると、チームの1つがエンドポイントを作成し、バージョンをURLに転送することを決定します。

開発者として、私がそのようなAPIを使用することは不便でしょう。そして、その理由は、バージョン管理に1つのアプローチを他のすべての人よりも好むということではありません。同時にいくつかの方法を使用するだけでは不便です。 APIで動作するコードを3回記述するか、3つの異なるライブラリを使用する必要があります。これは、エラーが3倍になることを意味します。

そして、日付と時刻があります。標準的なアプローチを使用していても、データはさまざまな方法で送信できます。サーバーのタイムゾーン、GMT、クライアントの時刻、Unix時間で送信されます。クライアントはすべてのデータを1つの形式にする必要があります。

私の意見では、優れたAPIで最も重要なことは一貫性です。選択するバージョン管理の方法は関係ありません。主なことは、API全体で同じであることです。さらに成功する解決策は、他の企業がすでに採用している一般的なアプローチを取ることです。ほとんどの場合、クライアントはすでに問題に直面しており、既成のコードまたはライブラリーを持っています。もう一度考える必要はありません。既成のツールを使用できます。

APIを作成する単一のアプローチは、分散したチームを持つ大企業にとって特に重要です。部屋にいる10人の人がスタートアップを作成する場合、各開発者は隣人を肩にぶつけ、コードレビュー中に主な問題に同意または確認できます。大企業では、さらに難しくなります。

スタイルガイドAPIは、このような問題の解決に役立ちます。これは、APIの設計原則を説明するドキュメントです。全員が同じアプローチを使用している場合、APIは1人で開発されたものであり、多くの異なるチームではないようです。このアプローチは、コードスタイルガイドによく似ており、チームがコードの記述方法について合意します。あるチームの人は、別のチームのコードベースを見て、そこで何が起こっているのかをすばやく理解できます。

独自のAPIスタイルガイドを作成する方法


Arnauld Lauret（API Handymanとして知られています）は、クールなサイト-API Stylebookを作成しました。彼は一般にアクセス可能なスタイルガイドをまとめ、さまざまな企業（Cisco、Google、Red Hat、政府機関、その他多くの企業）がパブリックドメインに配置しました。

ガイドラインは、サイト上のトピックに分かれています。たとえば、バージョン管理に関心がある場合は、[バージョン管理]タブで、さまざまな企業がこの問題をどのように解決するかを確認できます。

他の人の経験から学びましょう！ただし、他人のスタイルガイドを完全にコピーすることはお勧めできません。それでも、彼はあなたではなく、他の会社の問題を解決します。

スタイルガイドに、会社に近いトピックを追加して、問題を解決する必要があります。多分あなたは多くのモバイルアプリケーションを持っていて、答えのサイズはあなたにとって重要です。次に、この側面に関連する原則を説明します。または、メッセージブローカーを介して相互に通信するマイクロサービスへのモノリスを確認するときがきたという内部チームに同意しました。この場合、Kafkaまたはその他のツールを使用する際の原則をAPIスタイルガイドに含める必要があります。

API Style Guide?


APIスタイルガイドを作成しましたが、実装は残ります。これも簡単ではありません。私が働いていた場所の1つにAPIスタイルガイドを実装した経験を共有します。同社は複雑な構造を持っていました。何百人ものエンジニア、4つの開発オフィスに分散した数十のチーム、同僚が異なる言語を話しました。

まず、利害関係者を説得する必要があります。ここでは、さまざまな分野の技術リーダーとビジネス担当者の両方についてよく話し合っています。誰もが私たちが解決したい問題とスタイルガイドがどのように役立つかを理解する必要があります。私の例では、すべての関係者が、優れたAPIが重要な利点であることを理解しました。多くのクライアントがそのようなリクエストを持ち、競合他社は居眠りせず、APIを通じて利用できる機会の範囲を積極的に拡大しました。

次のステップは、これを行うチームを見つけることです。私の場合、会社には外部APIをサポートする別のチームがすでにありました。彼女は常に自分の仕事の困難に直面していたので、変化の必要性を最もよく理解しました。このチームのメンバーは、プロジェクトに参加できてうれしいです。

チームはどのようにスタイルガイドAPIを実装しましたか？

  • 彼女は社内会議やハッカソンでイニシアチブについて話し、社内の誰もが議論されていることを理解できるようにしました。これは、次の段階で非常に役立つ志を同じくする人々を引き付けるのに役立ちました。
  • ドラフトスタイルガイドを作成し、それはその後APIチャンピオンに同意しました（それらについてもう少し）。
  • プロジェクトをチームに提示しました。担当者は各開発オフィスに行き、ほとんどすべてのチームと話し、プロジェクトの意味を説明し、トレーニングを実施しました。
  • APIの設計を支援しました。多くのチームがガイドのすべてを初めて行うのは困難でした。ここでは、特に異常な状況で、ヘルプと追加の説明が役立ちます。
  • レビューを実施しました。すべてのチームがスタイルガイドに従って行動することを確認することが重要でした。

大企業の1つのチームがこのような大きな変更を実装することは非常に困難です。サポートが必要です。ここでは、APIチャンピオンがつながっていました。各方向のアーキテクトとテクライドは、チームや部門にアイデアを導入するのに役立ちました。

重要な変更が行われると、不満がしばしば現れます。この場合、これらの人々に変更の必要性を説得できる権限が必要です。Tehlidsはこれに最も適した人々です。

別のSlackチャネルのAPIチャンピオンは、新たな問題と提案について議論しました。小さな変更はすぐに導入されましたが、大きな変更は詳細な議論を必要としました。この場合、イニシエーターは公式の提案を書きました：解決する必要がある問題、解決策の方法はすでに存在していますか、彼らが持つ利点と欠点です。その後、1〜2日間、オフィス外のオフサイトに集まり、アイデアについて話し合い、解決策を選択して修正しました。次に、私たちが下した決定についてチームに伝えました。なぜそれを行ったのか、次に何をする必要があるのか​​です。

また、APIチャンピオンは彼らの分野でレビューをするのを助けました。彼らのチームのメンバーは最初に彼らに目を向け、次に最終的なレビューはスタイルガイドを作成していたチームによって行われました。

しばらくして、APIコミュニティが社内で形成され始めました。これには、複数のレビューサイクルを経て、優れた正しいデザインが何であるか、なぜそれが必要なのか、どのように作成するのかを理解した人々が含まれていました。私たちは誰もが質問できる大規模なチャットを作成し、APIコミュニティのメンバーがそれを解決するのに役立ちました。

エンジニアリングレベルでのみアプローチを実装することはできません。セールス、サポート、マーケティングの各チームは、APIの対象となるユーザーシナリオ、APIの価値、会社の戦略との関連性、APIに関する情報を顧客に適切に伝える方法についても理解する必要があります。

これらすべてを、エンジニアリング以外のチームを含む社内のほとんどのチームのオンボーディングに導入しようとしました。これは、誰がいつ、どのような場合に誰に助けを求めることができるかを理解するためにも必要です。

オートメーション


最後の重要なトピックは、ルーチンプロセスの自動化です。レビューは第一の例として役立ちます。チーム数が多い大企業では、多くの場合、多くのレビューを行う必要があります。エラーの説明がない、IDの形式が正しくないなど、基本的な問題に時間を費やしたくありません。

いくつかのツールがこれに役立ちます：

  • ZalandoのZallyは、OpenAPI仕様がAPIスタイルガイドと一致するかどうかを確認します。Zallyの基本構成は、Zalandoスタイルガイドに準拠しているかどうかをチェックしますが、独自のルールを再定義して追加できます。
  • ApiaryのDreddは、最終的なREST API実装が設計、つまりOpenAPI、ブループリント、またはRAML仕様と一致しているかどうかを確認します。
