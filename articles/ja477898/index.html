Kotlinコンテキスト指向プログラミングの概要

これは記事の翻訳です。Kotlinでのコンテキスト指向プログラミングの概要

この記事では、Kotlin言語の急速な発展の副産物として生じた新しい現象について説明します。これは、アプリケーションとライブラリのアーキテクチャを設計するための新しいアプローチであり、コンテキスト指向プログラミングと呼びます。

機能の許可について一言


よく知られているように、3つの主要なプログラミングパラダイムがあります（追記：他のパラダイムがあります）。

  • 手続き型プログラミング
  • オブジェクト指向プログラミング
  • 関数型プログラミング

これらのアプローチはすべて、何らかの方法で関数を処理します。関数の解決またはそれらの呼び出しのスケジューリング（この場所で使用する関数の選択を意味する）の観点からこれを見てみましょう。手続き型プログラミングの特徴は、グローバル関数の使用と、関数名と引数の型に基づく静的解決です。もちろん、型は静的型付け言語でのみ使用できます。たとえば、Pythonでは関数が名前で呼び出され、引数が正しくない場合、プログラムの実行中にランタイムで例外がスローされます。手続き型アプローチの言語での関数の解決は、手続き/関数の名前とそのパラメーターのみに基づいており、ほとんどの場合、静的に行われます。

オブジェクト指向プログラミングスタイルは、関数のスコープを制限します。関数はグローバルではなく、クラスの一部であり、対応するクラスのインスタンスでのみ呼び出すことができます（Pedantの注：一部の古典的な手続き型言語にはモジュラーシステムがあるため、スコープ;手続き型言語！= C）。

もちろん、クラスのメンバー関数を、呼び出されたオブジェクトのタイプの追加引数を持つグローバル関数でいつでも置き換えることができますが、構文の観点から見ると、その違いは非常に重要です。たとえば、この場合、メソッドは参照するクラスにグループ化されているため、このタイプのオブジェクトが提供する動作の種類がより明確になります。

もちろん、カプセル化はここで最も重要です。これは、クラスまたはその動作の一部のフィールドがプライベートであり、このクラスのメンバーだけがアクセスできるためです（純粋に手続き型のアプローチではこれを提供できません）。これにより、実際に使用されるメソッドは名前だけでなく、メソッドだけでなく、それが呼び出されるオブジェクトのタイプにも基づいています。オブジェクト指向アプローチでのメソッド呼び出しのディスパッチは、ランタイムで定義されたオブジェクトのタイプ、メソッドの名前、およびコンパイル段階での引数のタイプによって異なります。

機能的アプローチは、機能解決の点で根本的に新しいものをもたらすものではありません。機能指向言語には、通常、可視性のある領域を区別するためのより良いルールがあります（注意：ここでも、Cはすべて手続き型言語ではありません。スコープが明確に区切られている言語もあります）。これにより、モジュールのシステムに基づいて関数の可視性をより詳細に制御できますが、さらに、解決は引数のタイプに基づいてコンパイル時に実行されます。

これは何ですか？


オブジェクトアプローチの場合、オブジェクトのメソッドを呼び出すときにその引数がありますが、さらに、呼び出されたクラスのインスタンスを表す明示的（Pythonの場合）または暗黙的パラメーターがあります（以下、すべての例はKotlinで記述されます）。

class A{
    fun doSomething(){
        println("    $this")
    }
}

ネストされたクラスとクロージャーは少し物事を複雑にします：

interface B{
    fun doBSomething()
}
 
class A{
    fun doASomething(){
        val b = object: B{
            override fun doBSomething(){
                println("    $this  ${this@A}")
            }
        }
        b.doBSomething()
    }
}

この場合、2つの暗黙的なこの関数doBSomethingがあります-クラスBの 1つのインスタンスに対応しますが、回路Aの別のインスタンスがあります。ラムダクロージャのより一般的なケースでも同じことが起こります。この場合、これは暗黙的なパラメーターとしてだけでなく、レキシカルスコープで呼び出されるすべての関数とオブジェクトのスコープまたはコンテキストとしても機能することに注意することが重要です。したがって、doBSomethingメソッドは、実際にはクラスAのメンバー（パブリックまたはプライベート）だけでなく、B自体のメンバーにもアクセスできます。

そして、これはコトリンです


Kotlinは、まったく新しい「おもちゃ」、つまり拡張機能を提供します。（Pedantによる注記：実際、それほど新しいものではなく、C＃にも存在します）。A内だけでなく、プログラム内の任意の場所でA.doASomething（）のような関数を定義できます。この関数内には、メソッドが呼び出されるインスタンスAを指す、レシーバーと呼ばれる暗黙のthisパラメーターがあります。

class A
 
fun A.doASomthing(){
    println(" -   $this")
}
 
fun main(){
    val a = A()
    a.doASomthing()
}

拡張機能は受信者のプライベートメンバーにアクセスできないため、カプセル化に違反しません。

Kotlinで次に重要なことは、レシーバー付きのコードブロックです。何かを受信者として使用して、コードの任意のブロックを実行できます。

class A{
    fun doInternalSomething(){}
}
 
fun A.doASomthing(){}
 
fun main(){
    val a = A()
    with(a){
        doInternalSomething()
        doASomthing()
    }
}

この例では、with関数は後続のブロックのすべてのコードをaのコンテキスト内に配置するため、最初に両方の関数を「a。」なしで呼び出すことができます。つまり、このブロックのすべての関数は、（明示的に渡された）オブジェクトaで呼び出されたかのように呼び出されます。

この時点でのコンテキスト指向プログラミングへの最後のステップは、クラスのメンバーとして拡張機能を宣言する機能です。この場合、拡張関数は次のように別のクラス内で定義されます。

class B
 
class A{
    fun B.doBSomething(){}
}
 
fun main(){
    val a = A()
    val b = B()
    with(a){
        b.doBSomething() //  
    }
    b.doBSomething() //  
}

ここでBがいくつかの新しい動作を取得することが重要ですが、それが特定の字句コンテキストにある場合のみです。関数拡張は、クラスAの通常のメンバーです。つまり、関数の解決は、関数が呼び出されるコンテキストに基づいて静的に行われますが、実際の実装は、コンテキストとして渡されるAのインスタンスによって決定されます。関数は、オブジェクトaの状態と相互作用することもあります。

コンテキスト指向のディスパッチ


記事の冒頭で、関数呼び出しをディスパッチするためのさまざまなアプローチについて説明しましたが、これには理由があります。実際、Kotlinの拡張関数を使用すると、ディスパッチを新しい方法で処理できます。現在、特定の関数を使用するかどうかの決定は、そのパラメーターのタイプだけでなく、その呼び出しの字句コンテキストにも基づいています。つまり、異なるコンテキストでの同じ表現は、異なる意味を持つ可能性があります。もちろん、実装の観点からは何も変化せず、クラス自体のボディに記述されたメソッドと拡張（メンバー拡張）のディスパッチを定義する明示的なレシーバーオブジェクトはまだありますが、構文の観点からは、これは別のアプローチです。

例として、Javaの数値に対する算術演算の古典的な問題を使用して、コンテキスト指向のアプローチが従来のオブジェクト指向のアプローチとどのように異なるかを見てみましょう。JavaとKotlin のNumberクラスはすべての数値の親ですが、Doubleなどの特殊な数値とは異なり、数学的な演算を定義しません。たとえば、次のように書くことはできません。


val n: Number = 1.0
 
n + 1.0 //  `plus`     `Number`

ここでの理由は、すべての数値型に対して算術演算を一貫して定義することができないためです。たとえば、整数除算は浮動小数点除算とは異なります。一部の特殊なケースでは、ユーザーは必要な操作の種類を知っていますが、通常、そのようなことをグローバルに定義しても意味がありません。オブジェクト指向の（そして実際には機能的な）ソリューションは、Numberクラスの新しい継承タイプを定義し、それに必要な操作を定義して、必要な場所で使用することです（Kotlin 1.3ではインラインクラスを使用できます）。代わりに、これらの操作でコンテキストを定義してローカルに適用しましょう：


interface NumberOperations{
    operator fun Number.plus(other: Number) : Number
    operator fun Number.minus(other: Number) : Number
    operator fun Number.times(other: Number) : Number
    operator fun Number.div(other: Number) : Number
}
 
object DoubleOperations: NumberOperations{
    override fun Number.plus(other: Number) = this.toDouble() + other.toDouble()
    override fun Number.minus(other: Number) = this.toDouble() - other.toDouble()
    override fun Number.times(other: Number) = this.toDouble() * other.toDouble()
    override fun Number.div(other: Number) = this.toDouble() / other.toDouble()
}
 
fun main(){
    val n1: Number = 1.0
    val n2: Number = 2
    val res = with(DoubleOperations){
        (n1 + n2)/2
    }
     
    println(res)
}

この例では、resの計算は、追加の操作を定義するコンテキスト内で行われます。コンテキストはローカルで定義する必要はありませんが、関数のレシーバーとして暗黙的に渡すことができます。たとえば、これを行うことができます：


fun NumberOperations.calculate(n1: Number, n2: Number) = (n1 + n2)/2
 
val res = DoubleOperations.calculate(n1, n2)

つまり、コンテキスト内の操作のロジックは、このコンテキストの実装から完全に分離されており、プログラムの別の部分または別のモジュールに書き込むこともできます。この単純な例では、コンテキストはステートレスシングルトンですが、ステートコンテキストも使用できます。

コンテキストはネストできることも覚えておくとよいでしょう。

with(a){
    with(b){
        doSomething()
    }
}

これにより、両方のクラスの動作を組み合わせる効果が得られますが、複数の受信者による拡張機能がないため、この機能を制御することは今日では困難です（KT-10468）。

明示的なコルーチンの力


コンテキスト指向アプローチの最良の例の1つは、Kotlinx-coroutinesライブラリで使用されています。アイデアの説明は、Roman Elizarovの記事にあります。ここで、CoroutineScopeはステートフルなコンテキストを備えたコンテキスト指向の設計のケースであることを強調したいだけです。CoroutineScopeは2つの役割を果たします。

  • これには、コルーチンを実行するために必要なCoroutineContextが含まれ、新しいコルーチンが起動されると継承されます。
  • これには、親コルーチンの状態が含まれています。これにより、生成されたコルーチンがエラーをスローした場合、それをキャンセルできます。

また、構造化された並行性は、コンテキスト指向アーキテクチャーの優れた例を提供します。

suspend fun CoroutineScope.doSomeWork(){}
 
GlobalScope.launch{
    launch{
        delay(100)
        doSomeWork()
    }
}

ここで、doSomeWorkはコンテキスト関数ですが、そのコンテキストの外部で定義されています。起動メソッドは、対応する関数の字句領域に相当する2つのネストされたコンテキストを作成します（この場合、両方のコンテキストは同じタイプなので、内部コンテキストが外部コンテキストをシェーディングします）。Kotlinコルーチンを学習するための良い出発点は、公式ガイドです。

DSL


Kotlinには幅広いクラスのタスクがあり、これらは通常DSL（ドメイン固有言語）を構築するタスクと呼ばれます。この場合、DSLは、ある種の複雑な構造のユーザーフレンドリーなビルダーを提供するいくつかのコードとして理解されています。実際、DSLという用語の使用は、ここでは完全に正しいわけではありません。このような場合、基本的なKotlin構文は特別なトリックなしで単純に使用されますが、この一般的な用語を使用してみましょう。

ほとんどの場合、DSLビルダーはコンテキスト指向です。たとえば、HTML要素を作成する場合は、まずこの特定の要素をこの場所に追加できるかどうかを確認する必要があります。Kotlinx.htmlライブラリ特定のタグを表すコンテキストベースのクラス拡張を提供することにより、これを実現します。実際、ライブラリ全体は、既存のDOM要素のコンテキスト拡張で構成されています。

もう1つの例は、TornadoFX GUIビルダーです。シーングラフのビルダー全体は、ネストされたコンテキストビルダーのシーケンスとして配置されます。内部ブロックは、外部ブロックの子の構築または親のパラメーターの調整を担当します。以下は公式ドキュメントの例です：


override val root = gridPane{
    tabpane {
        gridpaneConstraints {
            vhGrow = Priority.ALWAYS
        }
        tab("Report", HBox()) {
            label("Report goes here")
        }
        tab("Data", GridPane()) {
            tableview<Person> {
                items = persons
                column("ID", Person::idProperty)
                column("Name", Person::nameProperty)
                column("Birthday", Person::birthdayProperty)
                column("Age", Person::ageProperty).cellFormat {
                    if (it < 18) {
                        style = "-fx-background-color:#8b0000; -fx-text-fill:white"
                        text = it.toString()
                    } else {
                        text = it.toString()
                    }
                }
            }
        }
    }
}

この例では、字句ドメインはコンテキスト（GUIセクションとその内部構造を表すため論理的）を定義し、親コンテキストへのアクセス権を持っています。

次のステップ：複数の受信者


コンテキスト指向プログラミングは、Kotlin開発者に多くのツールを提供し、アプリケーションアーキテクチャを設計する新しい方法を開きます。他に何か必要ですか？多分そう。

現在、コンテキストアプローチでの開発は、ある種のコンテキスト限定のクラス動作を実現するために拡張機能を定義する必要があるという事実によって制限されています。これはカスタムクラスに関しては問題ありませんが、ライブラリのクラスに対して同じことをしたい場合はどうでしょうか。または、既に制限されている動作の拡張を作成する場合（たとえば、CoroutineScope内にある種の拡張を追加するなど）？ Kotlinは現在、拡張機能が複数の受信者を持つことを許可していません。ただし、下位互換性を損なうことなく、複数の受信者を言語に追加できます。複数の受信者を使用する可能性は現在議論されており（KT-10468）、KEEPリクエストとして発行されます（UPD：すでに発行されています））ネストされたコンテキストの問題（またはおそらくチップ）は、提案された機能のもう1つの非常に望ましい型クラス（type-classes）を使用するためのすべてではないにしても、ほとんどのオプションをカバーできることです。これらの機能の両方が同時に言語で実装される可能性はかなり低いです。
