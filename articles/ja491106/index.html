トランザクションはどのようにコミットされますか


このトピックはレプリケーションに直接関係しないので、別の記事を書くことができます。ただし、トランザクションのコミットメカニズムをこれ以上理解しないと、それ以上の読み取りは役に立たないため、最も基本的なことを思い出させてください。トランザクションは3つの段階でコミットされます。

  1. トランザクションをデータベースログに書き込みます。
  2. データベースエンジンでのトランザクションの適用。
  3. トランザクションの適用の成功についてクライアントに確認を返します。

このアルゴリズムでは、さまざまなデータベースでニュアンスが発生する可能性があります。たとえば、MySQLデータベースのInnoDBエンジンには、2つのログがあります。1つはレプリケーション（バイナリログ）を維持し、もう1つはACID（元に戻す/やり直しログ）を維持します。一方、PostgreSQLでは1つのログを実行します。両方の機能（ログ先行書き込み= WAL）。しかし、上記はまさにそのようなニュアンスを無視できるようにする一般的な概念です。

同期（同期）レプリケーション


受信した変更を複製するためのロジックをトランザクションコミットアルゴリズムに追加しましょう。

  1. トランザクションをデータベースログに書き込みます。
  2. データベースエンジンでのトランザクションの適用。
  3. すべてのレプリカにデータを送信しています。
  4. それらのトランザクションに関するすべてのレプリカから確認を受け取ります。
  5. トランザクションの適用の成功についてクライアントに確認を返します。

このアプローチには、いくつかの欠点があります。

  • クライアントは、すべてのレプリカに変更が適用されるのを待っています。
  • クラスター内のノード数が増えると、書き込み操作が成功する可能性が低くなります。

1番目の段落ですべてがある程度明確である場合は、2番目の段落の理由を明確にする必要があります。同期レプリケーション中に少なくとも1つのノードからの応答がない場合は、トランザクションをロールバックします。したがって、クラスター内のノード数を増やすと、書き込み操作が失敗する可能性が高くなります。

51％（クォーラム）など、ノードの特定の部分からのみ確認を待機できますか？はい、できますが、クラシックバージョンでは、すべてのノードからの確認が必要です。これは、このタイプのレプリケーションの疑いのない利点である、クラスター内のデータの完全な一貫性を保証できる方法だからです。

非同期複製


以前のアルゴリズムを変更してみましょう。「いつか後で」レプリカにデータを送信し、「いつか後で」変更をレプリカに適用します。

  1. トランザクションをデータベースログに書き込みます。
  2. データベースエンジンでのトランザクションの適用。
  3. トランザクションの適用の成功についてクライアントに確認を返します。
  4. レプリカにデータを送信し、それらに変更を適用します。

このアプローチでは、データがレプリカに到達し、通信されるまでクライアントを待機させないため、クラスターが高速であるという事実につながります。

しかし、「しばらくして」レプリカにデータをドロップする状態は、トランザクションの損失につながる可能性があり、トランザクションの損失がユーザーに確認される可能性があります。これは、データを複製する時間がなかった場合、操作の成功に関する確認がクライアントに送信され、変更が到着したノードが飛んだためです。 HDD、私たちはトランザクションを失っています。それは非常に不愉快な結果につながる可能性があります。

セミシンクレプリケーション


最後に、準同期レプリケーションに移行しました。このタイプのレプリケーションはあまり知られておらず、あまり一般的ではありませんが、同期レプリケーションと非同期レプリケーションの両方の利点を組み合わせることができるため、非常に興味深いものです。

以前の2つのアプローチを組み合わせてみましょう。クライアントを長期間保持することはしませんが、データを複製する必要があります。

  1. トランザクションをデータベースログに書き込みます。
  2. データベースエンジンでのトランザクションの適用。
  3. レプリカへのデータの送信。
  4. 変更の受信についてレプリカから確認を受信します（「後で」適用されます）。
  5. トランザクションの適用の成功についてクライアントに確認を返します。

このアルゴリズムでは、変更を受け入れるノードとレプリカノードが落ちた場合にのみトランザクションの損失が発生することに注意してください。そのような誤動作の可能性は小さいと見なされ、これらのリスクは受け入れられます。

しかし、このアプローチでは、ファントム読み取りのリスクが発生する可能性があります。次のシナリオを想像してみてください。ステップ4で、どのレプリカからも確認を受け取りませんでした。このトランザクションをロールバックする必要があり、確認をクライアントに返さないでください。データはステップ2で適用されたため、ステップ2の終了とトランザクションのロールバックの間に時間差があり、その間に並列トランザクションはデータベースにあるべきでない変更を確認できます。

ロスレスセミシンクレプリケーション


少し考えてみれば、このシナリオで幻像の読み取りの問題を修正するためにアルゴリズムのステップを変更することしかできません。

  1. トランザクションをデータベースログに書き込みます。
  2. レプリカデータを送信しています。
  3. 変更の受信についてレプリカから確認を受信します（「後で」適用されます）。
  4. データベースエンジンでのトランザクションの適用。
  5. トランザクションの適用の成功についてクライアントに確認を返します。

これで、変更が複製された場合にのみ変更をコミットします。

結論


いつものように、完璧な解決策はありません。一連の解決策があり、それぞれに独自の利点と欠点があり、さまざまな種類の問題を解決するのに適しています。これは、複製されたデータベースのデータを同期するメカニズムを選択する場合にも当てはまります。準同期レプリケーションがもたらす一連の利点は、その普及率が低いにもかかわらず、注目に値すると見なすのに十分であり、興味深いものです。
