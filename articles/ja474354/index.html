コンテナ内のスマートホーム（ioBroker + Docker内のZigbee）

前書き


しばらくの間、Google Home Miniを介して照明を制御する標準のSonoffリレーをいくつか使用しました。でも結局もっと欲しかった。標準的な機能では足りないので、少しずつシステムを柔軟にしていくことにしました。私はioBrokerを選びました。

最初はいつものように、機能の一部を見て、選択し、チェックしました。基本的な必需品が別々に働いていたとき、私はそれらを一緒にまとめ始めました。そしてもちろん、私は問題に遭遇しました。

主な困難：

  • 正確には何を選択しますか？結局のところ、私たちの計画を実行するには多くの方法があります。そして、選ばれたソリューションでも多くのオプションがあります...
  • 私が選択したソリューションのセットを正確に組み立てる方法、および構成に正確に組み込む方法に関する既成の指示はありません。

私が選んだもの、なぜ、どのような困難に遭遇したか、そしてそれらをどのように解決するか、そしてそれは議論されます。

今後は、ioBrokerが古いラップトップのdockerで起動し、Zigbeeをそれに投入して、ゲートウェイなしでXiaomiセンサーと直接やり取りする方法について説明します。標準的な指示はしません。「バンプ」のみを指示します。

まだ生産的ではないため、何かを変更することはかなり可能です。だから私はアドバイスと修正に感謝します:)

ウィッシュリストと私の考えのトレイン


Sonoffリール（eWeLinkで巻かれた）、ある種の延長コード（Tuya SmartLife）、複数のセンサーを備えたXiaomiゲートウェイ（Mi Home）、Google Home Miniカラムがありました。

eWeLinkとSmartLifeは通常、Google Homeで取り上げられ、「テーブルの上の照明をつけて」などの音声コマンドを聞いていました。 Mi Homeが起動しませんでした（Google HomeでサポートされているXiaomiデバイスはほとんどありません）。

私はioBrokerを置き、Xiaomi Gatewayをピックアップしました。原則として、すべてが機能しました。すべてが美しいので、スクリプトを確認しました。スクリプトは記述されており（Node-Redを選択しました）、これについてすべてを行うことにしました。

ただし、私はプロの管理者ではありません。準備する方法はわかりますが、ライブラリやコンポーネントなどの微妙な要素がどのように組み込まれるのかわかりません（深く掘り下げたくありません）。そのため、ioBrokerがnodejsを必要としていることを少し気になりました。バージョンに問題があります（リポジトリの標準のyumが古すぎるnodejsを置くなど）。

ええと私はすべてを始めましたが、それはうまくいきましたが、方法が理解できないという恐怖が私の心に残りました。また、たとえば、アップグレード中に何かが壊れた場合、その修正方法がわかりません。しかし、ioBrokerに加えて、ラップトップに他のシステムを積み重ねたいとも思っていました。

たとえば、動作しているように見えるものに更新を加えます。そしてしばらくすると、うまく機能しないことがわかりました。 1か月前にバックアップをロールバックする必要があります。さらに、バックアップはこのシステムだけでなく、他のすべてのものでもあります。実行可能ファイルがどこにあり、構成ファイルがどこにあり、データ自体がどこにあるのかが最後までわからない

ので、非常に煩わしく、Dockerを使用することにしました。コンテナ内のコード。データは個別に、ディレクトリはホストにマウントされます。バックアップが簡単。

新しいバージョンはありますか？はい、別の仮想マシン上でも、このディレクトリのデータを含むコンテナの新しいバージョンがどのように機能するかを簡単に確認できます。異なるシステムが互いに干渉することはありません。一部のシステムのバイナリを簡単にロールバックできます。繰り返しになりますが、他のものに転送するのは簡単です（コンテナー内のioBrokerはSynologyとシングルボードの両方で機能します）。美しさ！

（インターネット経由で利用可能にするが、セキュリティのために公開されないようにするための）要件もあり、構成を選択するときにさらに影響します。

取り付け


ホストOSは重要ではありません。CentOSをダウンロードしました（古いメモリから、すべてのネットワークタスクで安定してバグなく動作したことを思い出しました）。実際のバージョンはCent OS 8

でした。ホスト名、fail2banなどの基本的な設定を行いました（ホストはLANにしかありませんが、単なる習慣です）。Dockerを置く。これについては詳しく説明しません。例命令。

ioBrokerを起動します。しかし、どのネットワークを選択するのでしょうか？ホストかMacvlan？

まず、Macvlanは各コンテナーがルーターからIPアドレスを取得することを望んでいました。しかし、それから彼はこの冒険を断念することを決めました：

  • もちろん、ホストでは、転送するポートを明示的に指定する必要があります。他のコンテナと重複しないようにしてください。だが
  • iptablesは各コンテナー内で構成する必要があります。他のパラメータを使用して再起動するたびに、アップグレード/交換を含めます。ホストモードでは、これはセキュリティ管理の単一ポイントです。
  • 自宅のWiFiだけでなく、今後も利用できるようにする予定です。この場合、複数のホストを構成するよりも、外部から1つのホストにアクセスできるように（そしてその上にポートマッピングを行う）方が便利です。

現在、ZeroTier Oneをチェックしています。ホストにのみインストールしました。ZeroTierクライアントが完全に実行されているときに、GPRSを介して携帯電話からこのホストのIPアドレス（ローカルではなく、ZeroTierによって発行されます）およびポート8082にアクセスすると、visインターフェースが開きます。

だから標準

docker run -d --name ioBroker -p 8081:8081 -p 8082:8082 -v /opt/iobroker/:/opt/iobroker/ --device=/dev/ttyACM0 --env-file /opt/ioBroker_env.list --restart=always buanet/iobroker:latest

おっとっと問題が発生しました。

docker logs ioBroker最後のステップでは、外部リソースへの接続がないことを示しています。名前がずれるわけにはいきません。

docker exec -it ioBroker bashは、IPへのpingがIPから正常に通過するが、名前では通過しないことを示しています。

グーグル、私はドッカーがどのようにDNSサーバーを誤って代用するかについてのたくさんのリンクを見つけました、私は/etc/docker/daemon.jsonを支配します、私はdnsmasqを理解します-何も助けにはなりません。

思考が忍び寄り、突然何かがネットワークレベルでブロックされます。コンテナにtelnetやカールはありません。確認できません。インストールも簡単ではありません-yum installは機能しません。もちろん、必要なホストを/ etc / hostsに手動で指定することもできますが、これは時間がかかりすぎるので、他のバージョンを確認することをお勧めします。

たとえば、すべてが開かれることを期待して、ホスト上のファイアウォールを愚かに停止します。しかし、違います。

ポートの可用性は引き続きwgetで確認できることを思い出します。そして彼はコンテナにいます！IPでもダウンロードできません。ホームルーターのWebインターフェースでも接続できません。まあ、それは問題が間違いなくDNSではなく、iptablesにあることを意味します。

その結果、Dockerインターフェイスを信頼済みゾーンに追加した後、すべてが機能しました。

sudo firewall-cmd --permanent --zone=trusted --change-interface=docker0
sudo firewall-cmd --reload

それはさらに興味深いことですが、説明のどこかで見落としていたのでしょうか？

または、CentOS 8を使用しませんが、他の問題はありません（他のOSではデフォルトでファイアウォールを使用しません）？

それとも、誰もが指示を書いていないのは明らかなので、私は長い間一人でばかでした。

ジグビー


したがって、私のioBrokerはコンテナー内にあり、公開されているポートは数個しかありません。現在は管理者8081とvis 8082です。mqtt1883が追加され、おそらくTuyaをサポートするための何かが追加されます（そのようなドライバーは見ましたが、まだわかりません）。

残念ながら、ゲートウェイを介してXiaomiデバイスと対話するには、マルチキャストが必要であり、これにより、この複雑な構成で必要になります。そこで、USBスティックのコンテナを投入することにしました。また、通常の操作。

コマンドラインでは、これについてはすでに見てき--device=/dev/ttyACM0ました。コンテナー内のデバイスが表示されました。 ioBrokerで、標準ドライバー「Xiaomiおよびその他のデバイス用のZigbee」をアクティブ化しましたが、機能しませんでした。

シリアルポートにアクセスするには、ユーザーをダイヤルアウトグループに追加する必要があることをお勧めします。私はコンテナに入り、このグループにiobrokerを追加します-それは役に立ちません。

npmを介してシリアルポートパッケージをインストールする必要があるというヒントが表示されます。

できません、権利はありません。さらにGoogle。

Bluefoxだけで、これは/opt/iobroker/node_modules/iobroker.javascript/ディレクトリから行う必要があると誰かに伝えます-私にはそれがなく、まだインストール権限がありません（つまり、インストールが開始されてから、クラッシュ）。

最後に、コマンドラインでどのディレクトリに配置するかを明示的に指定する必要があることがわかってきました。

npm install -g serialport  --production --save --prefix "/opt/iobroker"

インストールされていますが、役に立ちません。

私はあなたがまだアクセス権を扱う必要があると疑い始めています。チェック（もちろんコンテナー内から）：

test -w /dev/ttyACM0 && echo success || echo failure

成功 それら。それでも、Dockerはデバイスを正しく投げました。

sudo -H -u  iobroker test -w /dev/ttyACM0 && echo success || echo failure

グーフィー！

コンテナー内の障害bashはルートの下で実行されますが、iobrokerユーザーからはポートにアクセスできません。ダイヤルアウトグループに事前に追加したにもかかわらず。

ls -l /dev/ttyACM0与える

crw-rw----. 1 root 18 166, 0 Nov  3 18:14 /dev/ttyACM0

ハ！グループの名前の代わりに18は何ですか？ メインホストで

はすべてが正しい：crw-rw----. 1 root dialout 166, 0 Nov 3 15:15 /dev/ttyACM0

メインホストの/ etc / group dialout:x:18とコンテナー内にあることdialout:x:20

がわかりました。同じ名前のグループにユーザーを追加しましたが、意味がありません。番号は同じではありません。そのため、識別子18で別のグループを作成し、ユーザーを既に追加しています。

groupadd -g 18 serial
usermod -a -G serial iobroker

確実性を高めるため、すべてを再起動しました。そして、これは私の対決の終わりでした:)

ioBrokerに関連付けられたXiaomiゲートウェイからすべてのセンサーを穏やかに解きました。

私はそれらをioBroker自体のオブジェクトとして表示します。Visの読み取り値が



読み取られます。



漏れセンサーの接点が閉じられると、データが受信されます。そして状況が変わります



。Node-Redでは信号が送られてきます。したがって、電子メールなどが送信されていても、音声またはMP3ファイルがGH Mini列に送信されます。



ちなみに、オブジェクトを表示すると、驚きが待っていました



。XiaomiCubeを別の側に向けました。最新の変更は緑色で表示されます。
Flip90が変更されました-これは理解できます。この信号は制御のためにキャッチされます。しかし、結局のところ、flip90_fromとflip90_toがまだあります。

理論的には、キューブからさらに多くの制御信号を取得できることがわかります。たとえば、エッジに矢印を描く場合（まるで円のように）、「90度回転」するだけでなく、どの方向（自分から、または自分へ、左または右）でも追跡できます。

flip180の場合も機能します。また、他のジェスチャーについても同様の追加情報があります（フリップの上面、スライドの上面、タップの上面）

。しかし、標準のMi Homeでは、顔に関する情報はありませんでした。Xiaomi Gateway経由の以前の接続では、私もそれを見ていないようで、各顔に番号があることを知りませんでした。以前、私は落下（自由落下）の追加のアクションについてしか知りませんでしたが、それらはMi Homeから追放されました（どうやら、彼らはそれを頻繁に落としました）。
