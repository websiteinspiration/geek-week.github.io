ブート時のLUKSコンテナーのデコード

こんにちは、夜！この投稿は、LUKSデータ暗号化を使用していて、ルートパーティションを復号化する段階で、Linux（Debian、Ubuntu）でディスクを復号化したい場合に役立ちます。そして私はインターネットでそのような情報を見つけることができませんでした。 最近では、シェルフ内のディスク数の増加に伴い、/ etc / crypttabを介して、よく知られた方法を使用してディスクを復号化するという問題に遭遇しました。個人的には、この方法を使用していくつかの問題を強調します。つまり、ファイルがルートパーティションがマウント（マウント）された後にのみ読み取られるため、特に* _cryptデバイスまたはmdadm raidsのパーティションから収集された場合、ZFSのインポートに悪影響を及ぼします。セクションからも収集されます。

LUKSコンテナーでpartedを使用できることは誰もが知っていますか？また、アレイにない他のサービスの早期開始の問題は、使用する必要があります（私はiSCSI上のクラスター化されたProxmox VE 5.xおよびZFSで作業しています）。

ほとんどの場合、インターネットでは、キーファイル（これの前にスロットに追加されたcryptsetup luksAddKeyコマンド）、またはまれな例外（ロシア語のインターネットは非常に少ない）を使用します-/ lib / cryptsetup / script / （もちろん、まだ方法はありますが、私はこの2つを正確に使用して、記事の基礎を形成しました）。また、再起動後、コンソールにコマンドを追加することなく、完全に自律的に組み込まれるようにして、すべてが一度に「離陸」するようにしました。では、なぜ待つのですか？ -

始めましょう！

暗号化パーティションsda3_cryptにインストールされたシステム（Debianなど）と、暗号化して好きなものを作成する準備ができている1ダースのディスクを想定しています。sda3_cryptのロックを解除するためのパスフレーズがあり、このセクションから、実行中の（復号化された）システムのパスワードから「ハッシュ」を削除し、それを他のドライブに追加します。すべてが初歩的で、コンソールで実行します。

/lib/cryptsetup/scripts/decrypt_derived sda3_crypt | cryptsetup luksFormat /dev/sdX

ここで、Xはディスク、パーティションなどです。

パスフレーズの「ハッシュ」を使用してドライブを暗号化した後、誰に、何に慣れているかに応じて、UUIDまたはIDを見つける必要があります。/ dev / disk / by-uuidとby-idからそれぞれデータを取得します。

次の段階は、必要な機能のためのファイルとミニスクリプトの準備です。次に進みます。

cp -p /usr/share/initramfs-tools/hooks/cryptroot /etc/initramfs-tools/hooks/
cp -p /usr/share/initramfs-tools/scripts/local-top/cryptroot /etc/initramfs-tools/scripts/local-top/

さらに

touch /etc/initramfs-tools/hooks/decrypt && chmod +x /etc/initramfs-tools/hooks/decrypt

コンテンツ../decrypt
#!/bin/sh

cp -p /lib/cryptsetup/scripts/decrypt_derived "$DESTDIR/bin/decrypt_derived"


さらに

touch /etc/initramfs-tools/hooks/partcopy && chmod +x /etc/initramfs-tools/hooks/partcopy

コンテンツ../partcopy
#!/bin/sh

cp -p /sbin/partprobe "$DESTDIR/bin/partprobe"
cp -p /lib/x86_64-linux-gnu/libparted.so.2 "$DESTDIR/lib/x86_64-linux-gnu/libparted.so.2"
cp -p /lib/x86_64-linux-gnu/libreadline.so.7 "$DESTDIR/lib/x86_64-linux-gnu/libreadline.so.7"


もうちょっと

touch /etc/initramfs-tools/scripts/local-bottom/partprobe && chmod +x /etc/initramfs-tools/scripts/local-bottom/partprobe

コンテンツ../partprobe
#!/bin/sh

$DESTDIR/bin/partprobe


そして最後の、update-initramfsの前に、ファイル/ etc / initramfs-tools / scripts / local-top / cryptrootを編集する必要があります。

元の

                # decrease $count by 1, apparently last try was successful.
                count=$(( $count - 1 ))
                
                message "cryptsetup ($crypttarget): set up successfully"
                break


この形にする

編集済み

                # decrease $count by 1, apparently last try was successful.
                count=$(( $count - 1 ))
                

                /bin/decrypt_derived $crypttarget | cryptsetup luksOpen /dev/disk/by-uuid/ *CRYPT_MAP*
                /bin/decrypt_derived $crypttarget | cryptsetup luksOpen /dev/disk/by-id/ *CRYPT_MAP*

                message "cryptsetup ($crypttarget): set up successfully"
                break


ここではUUIDまたはIDのいずれかを使用できることに注意してください。主なことは、HDD / SSDデバイスに必要なドライバーが/ etc / initramfs-tools / modulesに追加されることです。udevadm info -a -n / dev / sdXを使用して、使用するドライバーを見つけることができます。egrep 'looking | DRIVER'。

これですべてのファイルが揃ったので、update-initramfs -u -k all -vを実行します。ロギングはありません。スクリプトの実行エラー。ディスクの数に応じて、再起動してパスフレーズを入力し、少し待ちます。次に、システムが起動し、起動の最終段階で、つまりルートパーティションを「マウント」した後、partprobeコマンドが実行されます。LUFSデバイス上に作成されたすべてのパーティションが検出されてピックアップされ、ZFSとmdadmのいずれのアレイも問題なくアセンブルされます。そして、これらすべては、これらのディスク/アレイを必要とする基本的なサービスとサービスをロードする前に行います。
