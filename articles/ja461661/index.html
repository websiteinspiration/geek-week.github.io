コンポーネントベースの開発ガイド

モジュール化は、1960年代以降、ソフトウェア開発の主要な原則の1つです。この原則を適用すると、プログラミングに多くの有用性がもたらされます。モジュール性は、責任の分離の原則の効果的な使用に貢献し、コードの作成、再利用、ビルドの能力の向上につながります。

今日、ソフトウェア設計におけるモジュール性の原則の適用は、コンポーネントに組み込まれた新しい形をとっています。これはコンポーネント駆動開発（CDD）です。React、Vue、Angularなどのユーザーインターフェースや、BitなどのCDD指向のツールを開発するための最新のライブラリとフレームワーク、モジュール式コンポーネントに基づいてアプリケーションを作成できます。プログラマーは、コンポーネントを分離して開発し、コンポーネント構成を構築するために必要なパターンとツールを自由に使用できます。


コンポーネントは、アプリケーションインターフェイスの明確に定義された独立したフラグメントです。コンポーネントの例として、ボタン、スライダー、チャットメッセージを表示するためのウィンドウなどのエンティティを引用できます。CDDの機能を理解し、このアプローチを開発に適用する方法を理解すると、アプリケーションの基礎としてコンポーネントを使用できます。これは、ソフトウェアプロジェクトを作成するときに、有用なすべてのものを提供します。つまり、モジュール式プログラミングの原則を適用します。Webコンポーネント

の分野で現在起こっていることをよく見ると、CDDがフロントエンド開発への標準化されたアプローチになっていることがわかります。 今日公開している資料は、コンポーネント開発ガイドです。

ビットでの作業：コンポーネントの作成、分離、再利用、およびコラボレーション

簡単に言うと、コンポーネントベースの開発は、疎結合の独立したコードブロックを作成してアプリケーションを設計することです。それぞれに、システムの他の部分との相互作用を整理するように設計されたインターフェースがあります。構成を通じて互いに結合された多くのコンポーネントは、モジュール式アプリケーションを形成します。

たとえば、Reactアプリケーションの作成にCDDを使用するということは、最初にアプリケーションの基礎を形成するコンポーネントを作成し、次に、ページ全体や機能の大きなブロックなど、アプリケーションのより大きな部分をそれらからアセンブルすることを意味します。

CDDはアトミックデザインに関連しています（ここでは、このトピックに関する有用な資料です）、および「マイクロフロントエンド」として知られる、Webプロジェクトのクライアント側部分を開発するためのアプローチ。

CDDは、大きなプロジェクトの開発プロセスを個々のコンポーネントの小さな開発プロセスに分解するのに役立ちます。各コンポーネントは、アプリケーションの他の部分とは独立して設計されており、システムの他の部分との相互作用の可能性を考慮して作成されます。各コンポーネントを独立したエンティティとして設計すると、開発者に多くの便利な機能が提供されます。

CDD


開発がコンポーネントに基づいている場合、これはプログラマーが特殊なツールを必要としていることを意味します。このようなツールは、コンポーネントの作成、テスト、コンポーネントへの一般的なアクセスの整理、およびコンポーネントでのコラボレーションのサポートを目的としています。

特に、コンポーネントを分離して設計およびテストすることが重要です。これにより、アプリケーションで使用できる独立したユニットの形で作業できます。さらに、コンポーネントの再利用のサポートとそれらを共有する機能も重要です。これにより、主要なプロジェクトでチームの一部として働いている人が、コンポーネントの形で、チームメンバーの1人がすでに発明したホイールを再発明しないことができます。

ここでは、CDDスタイルのプロジェクトでの作業を整理するのに役立ついくつかの便利なツールを紹介します。以下のセクションの1つで、CDD原則の実際の実装に推奨されるプロジェクトアーキテクチャについて説明します。

▍Bit：コンポーネントの個別およびチーム開発


ビットは、本質的にCDD方法論の実用的なアプリケーションをサポートするために設計されたオープンソースツールです。これはビットについてのビデオです。このツールは、コンポーネントの開発、コンポーネントのコラボレーション、およびそれらを使用したWebプロジェクトの作成に役立ちます。

問題は、ビットを使用すると、プロジェクトの一部として開発されたコンポーネントを分離できるということです。言う-アプリケーションまたはライブラリ内。ビットはコマンドラインツールをサポートし、コンポーネントのカプセル化をすべてのファイルと依存関係とともに整理するのに役立ちます。ビットを使用すると、カプセル化されたコンポーネントの仮想表現を分離して開発およびテストできます。

これは、アプリケーション内で作成されたコンポーネントにすべての依存関係を備え、カプセル化して、アプリケーションの外部で使用およびテストできるエンティティに変換できることを意味します。

さらに、ビットを使用すると、スタンドアロンのカプセル化されたコンポーネントをパックし、クラウドツールを使用してそれらの共有を整理できます。これにより、プロジェクトに取り組んでいるチームは、共有されているすべてのコンポーネントを使用できます。ビットコミュニティには約50,000人の開発者がいます。彼らの努力により、誰もが利用できる何千ものオープンソースコンポーネントが作成されました。


カプセル化されたコンポーネントを使用したプロジェクトの開発

クラウドプラットフォームの機能により、開発チームはアプリケーションにオープンソースコンポーネントをインストールできます。チームメンバーは、コンポーネントの作成者に、自分の作業環境から直接コンポーネントを改善するためのアイデアを提供することもできます。 BitはGitの機能を拡張して、プロジェクト全体でコンポーネントのソースコードの変更を追跡および同期します。これにより、開発者はコンポーネントの変更と更新を管理できます。

ビットは、コンポーネントの依存関係の完全なツリーを格納します。これにより、開発者は、コンポーネントの更新が依存コンポーネントにどのように影響するか、コンポーネントに変更を加えるときにプロジェクトで「壊れる」可能性があることについて学ぶ機会が得られます。これは、ビットのおかげで、プログラマーはコンポーネントに基づいて開発を整理するための本格的な環境を自由に利用できることを意味します。


CDDのクラウド機能

インターフェース設計者、プロジェクト顧客の代表、およびプログラマーは、プロジェクトでの共同作業の過程で、一般的なビジュアルツールを使用できます。これは、デザイナーとプログラマーの間のギャップを埋めます。さらに、この状況では、設計者やプログラマーだけでなく、異種性やエラーが少ないアプリケーションのエンドユーザーにもメリットがあることに注意してください。ここ、ここと以下は、ビットの使用に関するさまざまな側面に関する資料です。

UIUIコンポーネントの開発と研究：StoryBookとStyleguidist


StoryBookとStyleguidistは、Reactを使用してユーザーインターフェイス要素をすばやく開発するための環境です。これらのプロジェクトはどちらも、コンポーネント開発を加速する優れたツールです。

ストーリーブック


StoryBookは、UIコンポーネントを迅速に開発するための環境です。コンポーネントのライブラリを操作したり、コンポーネントのさまざまな状態を表示したり、コンポーネントのインタラクティブな開発やテストを行ったりすることができます。


StoryBookの操作

StoryBookは、アプリケーションから分離してコンポーネントを設計するのに役立ちます。これにより、コンポーネントの再利用性が向上し、コンポーネントのテスト性が向上します。

StoryBookを使用すると、ライブラリに保存されているコンポーネントを表示し、オンラインでそれらのプロパティを試すことができます。ページをリロードせずに、コンポーネントにすぐに加えられた変更が視覚化されます。ここでは絵本で作成されたコンポーネントのいくつかの例があります。

StoryBookを使用してコンポーネントの開発をスピードアップできるさまざまなプラグインがあります。これにより、コンポーネントコードを変更してから視覚的に表現するまでの時間を短縮できます。Storybookは、Reactに加えて、より多くのReact NativeとVue.jsをサポートしていることに注意してください。

React styleguidist


React Styleguidistはコンポーネント開発環境です。この環境には、ホットブートをサポートする開発サーバーが含まれています。また、propTypesコンポーネントを表示し、.mdファイルに基づくコンポーネントの編集可能な例を開発者に提供するインタラクティブなスタイル管理システムも備えています。


React Styleguidist

このプラットフォームは、JavaScript ES6、Flow、およびTypeScriptをサポートしています。彼女は、追加の設定なしで、Create Reactアプリを操作できます。Styleguidistには、コンポーネントのドキュメントを自動的に作成する機能があります。これにより、このシステムは、一部のチームが取り組んでいるコンポーネントのビジュアルドキュメントポータルの役割を果たすことができます。

StoryBookやStyleguidistなどのプロジェクトに興味がある場合は、Formidableによって開発されている React Liveプロジェクトをご覧ください。

StoryBookとStyleguidistの違い


StoryBookを使用する場合、プログラマはJavaScriptファイルに「ストーリー」を書き込みます。Stuleguidistで作業するとき、彼はMarkdownファイルに「例」を書きます。StoryBookは一度にコンポーネントの1つのバリエーションのみを表示しますが、Styleguidistはさまざまなコンポーネントのいくつかのバリエーションを表示できます。StoryBookは、コンポーネントのさまざまな状態を分析するのに最適です。Styleguidistは、コンポーネントのドキュメントを生成し、コンポーネントの機能を示すのに最適です。

CDD方法論を使用して使用されるアーキテクチャ上の決定


CDDのスタイルで作業するということは、アプリケーションを開発するときに、主にコンポーネントを作成するということです。さらに、そのようなコンポーネントは、可能な限り互いに独立している必要があります。これは、開発者が「コンポーネントのセット」を作成するだけではないことを意味します。また、ユーザーインターフェイスコンポーネントのいわゆる設計システムも実装しています。

コンポーネントは、アプリケーション自体のフレームワーク内（つまり、同じプロジェクト、リポジトリ内）でも、コンポーネントのライブラリとして別のプロジェクト（リポジトリ）の形式でも作成できます。

ビットなどのツールを使用すると、コンポーネントを分離してカプセル化できます。これにより、開発者はコンポーネントを作成、テスト、再利用する十分な機会が得られ、コンポーネントがどこで作成されたかに関係なく、どこででも完成させることができます。

CDDを使用すると、プロジェクトで作業する最初のステップの形でコンポーネントを開発できると言うと、コンポーネントは繰り返し使用するのに適している傾向があることも意味します。これにより、さまざまなアプリケーションを作成するときにそれらを適用できます。したがって、開発者はそのようなコンポーネントを作成するために自分がどのように行う必要があるかを正確に見つける必要があります。ライブラリを作成するのに6か月を費やすことほど悪いことはありません結局、誰も使用しません。残念ながら、これは多くのチームで起こります。

コンポーネントライブラリを作成する理由


正直なところ、Gitリポジトリは、さまざまなプロジェクトで共有される予定のアトミックコンポーネントコードを格納する可能性を考慮して考えられていません。この問題を解決するには、パッケージマネージャーは適していません。どちらもコードリポジトリをサポートするために作成されたものであり、理解可能です。コンポーネントはリポジトリではありません。


さまざまなプロジェクトでのコンポーネントの共有

そのため、あるアプリケーションからコンポーネントを取得して別のアプリケーションで使用する必要がある場合は、新しいリポジトリを作成する必要があります。そのようなコンポーネントごとに個別のリポジトリを作成するという不必要な作業から身を守るために、ほとんどのチームは、20〜30個のコンポーネントを格納するように設計された1つの共有リポジトリを作成します。

Bitなどのツールを使用する場合、コンポーネントライブラリ自体は必要ありません。このようなツールを使用すると、アプリケーションからクラウドにコンポーネントをダウンロードし、このコンポーネントを他のプロジェクトに実装できます。この作業スキームを従来のリポジトリと比較すると、これは音楽CDとSpotifyを比較するようなものだと言えます。確かに、コンポーネントライブラリの開発と使用のアイデアに近い場合は、BitやStoryBookなどのプラットフォームの機能を使用してライブラリを操作できます。

ライブラリを設計するときは、いくつかの重要な決定を行う必要があります。以下では、これを支援するいくつかの重要な原則を検討します。これらの原則の要点は、独立したコンポーネントを作成することです。プロジェクトの残りの作業段階は、レゴコンストラクターのアセンブリに似ています。独立したコンポーネントを開発するという原則が尊重されていない場合、いつか問題が発生する可能性があります。ライブラリにあるものとは異なる何かが必要になったときに、問題が発生します。実際、これが発生すると、チームはコンポーネントライブラリの使用を停止します。

CDDチームの利点



CDDの原則を使用するチームは、開発の加速、コードの再利用性の向上、TDDの改善、一貫した設計システムインターフェイスの恩恵を受けます。

プログラマは、すでに記述されているコンポーネントにアクセスできるコードを記述し、連携してコンポーネントに変更を加えることができます。新しい機能やアプリケーションの開発は、基本的なコンポーネントのカスタマイズと拡張に帰着します。さらに、これは本番環境でのみ発生するエラーを防ぐのに役立ちます。

コードの共有は、とりわけ、サポートする必要のあるコードの量を減らすことを意味します。これにより、プログラマーは何か新しいものを作成することに集中できます。コンポーネントに基づいてアプリケーションが開発される場合、誰もがアプリケーションのビルディングブロックの単一のベースを使用するという事実により、これは作業を標準化します。コンポーネントアプローチも柔軟性に貢献します。

一部のチームは、Reactコンポーネントのセットとして実装された設計システムに基づくモジュール式コンポーネントを使用することで、ワークフローが最大60％速くなると報告しています。一部の組織は、CDDの導入により、コードベースから約30％のコードを削除できることを発見しました。
