マイクロコントローラーのプログラムでprintfを使用する場合の最も一般的なレーキ

私のプロジェクトでは時々、シリアルポート（UARTまたはUSB経由の抽象化、シリアルポートをシミュレート）と組み合わせてprintfを使用する必要があります。そして、いつものように、そのアプリケーションの間には多くの時間が経過し、大規模なプロジェクトで正常に機能するように考慮に入れる必要があるすべてのニュアンスを完全に忘れてしまいます。

この記事では、マイクロコントローラー用のプログラムでprintfを使用するときに発生する独自のトップニュアンスを、最も明白なものから完全に明らかでないものまでの証拠で並べ替えて収集しました。

簡単な紹介


実際、マイクロコントローラーのプログラムでprintfを使用するには、次のように十分です。
  • プロジェクトコードにヘッダーファイルを含めます。
  • シリアルポートに出力するように_writeシステム関数を再定義します。
  • リンカが必要とするシステムコールのスタブを記述します（_fork、_waitなど）。
  • プロジェクトでprintf呼び出しを使用します。

実際、すべてがそれほど単純なわけではありません。

使用されているものだけでなく、すべてのスタブについて説明してください。


最初にプロジェクトを構築するときにあいまいなリンクの束が存在することは驚くべきことですが、少し読んだ後、何が、なぜかが明らかになります。私のすべてのプロジェクトでは、このサブモジュールを接続しています。したがって、メインプロジェクトでは、必要なメソッドのみ（この場合は_writeのみ）を再定義し、残りは変更しません。

すべてのスタブはC関数でなければならないことに注意することが重要です。C ++ではありません（またはextern“ C”でラップされています）。そうしないと、レイアウトが失敗します（G ++でビルドするときに名前を変更することを忘れないでください）。

_writeには1文字が入ります


_writeメソッドのプロトタイプには、表示されたメッセージの長さを渡す引数がありますが、値は1です（実際、私たちは常にそれを1にしますが、後で詳しく説明します）。
int _write (int file, char *data, int len) {
   ...
}

このような実装には、次の欠点があります。
  • 低い生産性;
  • ストリーミングの不安;
  • 他の目的でシリアルポートを使用できないこと。


低性能


パフォーマンスが遅いのは、プロセッサリソースを使用してバイトを送信するためです。同じDMAを使用する代わりに、ステータスレジスタを監視する必要があります。この問題を解決するには、事前に送信用のバッファーを準備し、行末の文字を受け取ったときに（またはバッファーを埋めるように）送信します。この方法ではバッファメモリが必要ですが、頻繁に送信する場合はパフォーマンスが大幅に向上します。
バッファを使用した_writeの実装例
#include "uart.h"

#include  <errno.h>
#include  <sys/unistd.h>

extern mc::uart uart_1;

extern "C" {

//      uart.
static const uint32_t buf_size = 254;
static uint8_t tx_buf[buf_size] = {0};
static uint32_t buf_p = 0;

static inline int _add_char (char data) {
    tx_buf[buf_p++] = data;

    if (buf_p >= buf_size) {
        if (uart_1.tx(tx_buf, buf_p, 100) != mc_interfaces::res::ok) {
            errno = EIO;
            return -1;
        }
        buf_p = 0;
    }

    return 0;
}

// Putty  \r\n   
//    .
static inline int _add_endl () {
    if (_add_char('\r') != 0) {
        return -1;
    }

    if (_add_char('\n') != 0) {
        return -1;
    }

    uint32_t len = buf_p;
    buf_p = 0;
    if (uart_1.tx(tx_buf, len, 100) != mc_interfaces::res::ok) {
        errno = EIO;
        return -1;
    }

    return 0;
}

int _write (int file, char *data, int len) {
    len = len;

    //   .
    if ((file != STDOUT_FILENO) && (file != STDERR_FILENO)) {
        errno = EBADF;
        return -1;
    }

    //    
    //   \n.
    if (*data != '\n') {
        if (_add_char(*data) != 0) {
            return -1;
        }
    } else {
        if (_add_endl() != 0) {
            return -1;
        }
    }

    return 1;
}

}

ここでは、dmaを使用して直接送信する場合、uartクラスのオブジェクトが責任を負います-uart_1。オブジェクトはFreeRTOSメソッドを使用して、バッファーからデータを送信するときに（mutexを取得して返す）、サードパーティによるオブジェクトへのアクセスをブロックします。したがって、別のスレッドから送信している間は、誰もuartオブジェクトを使用できません。

この実装も保護されないままです。これは、隣接するFreeRTOSストリームで、printfに別の行の送信を開始して、現在送信されているバッファーをグラインドする必要がないためです（uart内のmutexは、オブジェクトが異なるストリームで使用されるのを防ぎますが、データは送信されません） ）別のprintfがprintfによって呼び出されるリスクがある場合は、printfへのアクセスを完全にブロックするレイヤーオブジェクトを実装する必要があります。私の特定のケースでは、printfと対話するスレッドは1つだけであるため、追加の複雑化によってパフォーマンスが低下するだけです（レイヤー内のmutexの継続的な取得と解放）。

他の目的でシリアルポートを使用できない


文字列全体が受信された（またはバッファーがいっぱいになった）後にのみ送信するので、uartオブジェクトではなく、コンバーターメソッドを呼び出して、後続のパケット転送（たとえば、パケットと同様の伝送プロトコルによる保証付きの配信）を行うために、トップレベルのインターフェイスを呼び出すことができます。トランザクションmodbus）。これにより、1つのuartをデバッグ情報の出力と、たとえば、管理コンソールとのユーザー操作（デバイスで利用可能な場合）の両方に使用できます。受信側で解凍プログラムを書くだけで十分です。

デフォルトでは、フロート出力は機能しません


newlib-nanoを使用する場合、デフォルトでは、printf（およびsprintf / snprintfなどのすべての派生物）は、float値の出力をサポートしません。これは、次のリンカーフラグをプロジェクトに追加することで簡単に解決できます。
SET(LD_FLAGS
        -Wl,-u,vfprintf;
        -Wl,-u,_printf_float;
        -Wl,-u,_scanf_float;
        "_")

フラグの完全なリストはこちらをご覧ください。

printfはマイクロコントローラーを強制的にハードフォールトにします


少なくとも2つの理由が考えられます。
  • スタックの問題。
  • _sbrkに関する問題;

スタックの問題


この問題は、FreeRTOSまたはその他のOSを使用しているときに実際に発生します。問題はバッファの使用です。最初の段落では、_writeはそれぞれ1バイトになると述べました。これを実現するには、printfを初めて使用する前に、コードでのバッファリングの使用を禁止する必要があります。
setvbuf(stdin, NULL, _IONBF, 0);
setvbuf(stdout, NULL, _IONBF, 0);
setvbuf(stderr, NULL, _IONBF, 0);

関数の説明から、次のいずれかの値を同じ方法で設定できることがわかります。
#define	_IOFBF	0		/* setvbuf should set fully buffered */
#define	_IOLBF	1		/* setvbuf should set line buffered */
#define	_IONBF	2		/* setvbuf should set unbuffered */

ただし、これによりタスクスタックがオーバーフローする可能性があります（または、割り込みからprintfを呼び出す非常に悪い人の場合は、割り込み）。

純粋に技術的には、各ストリームに対して非常に注意深くスタックを配置することが可能ですが、このアプローチは慎重な計画を必要とし、それが運ぶエラーをキャッチすることは困難です。より簡単な解決策は、それぞれ1バイトを受け取り、それを独自のバッファーに格納してから、以前に解析された必要な形式で出力することです。

_sbrkの問題


この問題は私にとって個人的に最も暗黙のうちにありました。では、_sbrkについて何を知っているのでしょうか。
  • 標準ライブラリのかなりの部分をサポートするために実装する必要がある別のスタブ。
  • ヒープにメモリを割り当てるために必要です。
  • malloc、freeなど、あらゆる種類のライブラリメソッドで使用されます。

個人的に、私のプロジェクトの95％のケースでは、FreeRTOSの束を使用する再定義されたメソッドnew / delete / mallocでFreeRTOSを使用しています。したがって、メモリを割り当てるとき、割り当てはFreeRTOSヒープ上にあると確信しています。これは、bss領域で所定の量のメモリを消費します。ここでレイヤーを見ることができます。したがって、純粋に技術的には問題はないはずです。関数は単に呼び出されるべきではありません。しかし、彼女が電話をかけた場合、彼女はどこで自分の記憶を得ようとするでしょうか？

問題3の解決策：
  • bssとスタックの間のインデント。

標準の_malloc_r（libg_nano.a（lib_a-nano-mallocr.o）内にあります）を置き換えることができなかったため、最初のオプションで解決しました（メソッドは__attribute__（（weak））として宣言されていませんが、単一の関数のみをバイライブラリから除外しますリンクできませんでした）。1回の呼び出しでsbrkを書き直したくなかったのです。

最終的な解決策は、初期スタックと_sbrkに個別のパーティションをRAMに割り当てることでした。これにより、セットアップフェーズ中にセクションが互いの上にスタックされなくなります。sbrkの内部には、セクションから出るためのチェックもあります。海外での移行を検出したときにフローがwhileループでハングするように、少し修正を加える必要がありました（sbrkの使用は初期化の初期段階でのみ発生し、デバイスのデバッグ段階で処理する必要があるため）。
