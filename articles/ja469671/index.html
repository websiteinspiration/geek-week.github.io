Ada言語を使用してシステムをプログラムする理由



Adaプログラミング言語は、米国国防総省と英国国防総省が軍事プロジェクトでますます使用される組み込みコンピューティングシステム用の何百もの特殊なプログラミング言語を置き換えることを決定した1970年代半ばに誕生しました。Ada言語は、これらすべての組み込みシステムで機能する唯一の言語であると同時に、信頼性とパフォーマンスを特殊な言語よりも劣らないレベルで提供できるように設計されています。

1995年からの更新後、言語は汎用システムに適合し、オブジェクト指向プログラミングを追加しましたが、信頼性、サポートの容易さ、効率性などの主要な値を見失うことはありませんでした。今日、エイダで書かれたソフトウェアは、軍事機器だけでなく、航空電子工学および航空交通管制システムの分野における商業プロジェクトの基礎を形成しています。Adaコードは、Arian 4や5 などのミサイル、多数の衛星、および小さな誤動作が深刻な結果をもたらす可能性のある他の無数のシステムを制御します。

組み込みのデフォルト保護


地獄の型システムは厳密ではありません-暗黙的な型変換を許可しないため、超厳密と呼ばれることもあります。たとえば、次のCコードスニペットを見てください。

typedef uint32_t myInt;
myInt foo = 42;
uint32_t bar = foo;

これは有効なコードです。それはコンパイルされ、起動し、生命、宇宙などの主要な質問への答えを示す明白な結果を生成します。地獄では、これは機能しません：

type MyInt is Integer;
foo: MyInt;
bar: Integer;
foo := 42;
bar := foo;

IntegerとMyIntは同じではないため、コンパイラーはエラーをスローします。このアプローチの主な利点は、プログラマーが型定義を変更した場合、コードベース全体で何千もの暗黙的な型変換によってプログラムが破壊されないことです。代わりに、型を明示的にキャストする必要があります。これにより、適切なコードが促進され、「ほぼ同じ」である型の混同が防止されます。

タイプC、Linux、Win32の標準的な定義が混在する沼地で立ち往生しているプログラマーは、無数のドキュメントページや不適切にフォーマットされたコードを調べて、どのtypedefまたはマクロだけが実際の定義のみを含むかを理解する必要がないことを理解できます。デバッグ中にコンパイルを妨げたり、クロールアウトしたもの。



Adaは、コンパイルおよび起動段階でのチェックに追加の保護レイヤーを追加します。 Adaでは、プログラマーは、ブロックの終了ステートメントと変数の値が適合する境界を明示的に指定する必要があります。 Adaはintやfloatのような標準型を定義していませんが、プログラマーは最初から特定の範囲の型を作成する必要があります。これは文字列にも当てはまります。無制限の文字列を除いて、すべての文字列は固定長です。

作業段階では、メモリへの不正アクセス、バッファオーバーフロー、設定制限の超過、エラー±1、アレイへのアクセスなどのエラーをチェックできます。その後、アプリケーション全体をドロップする代わりに、それらを安全に処理できます。

Adaは、低レベルポインターの代わりに参照型のモデルを実装します。各参照型は、デフォルトで、または必要に応じて特定のプログラマーによって、よりエキゾチックなNUMAメモリ実装で動作するように、メモリプールによって処理されます。プログラマはメモリに直接アクセスする必要はなく、メモリプールハンドラを使用する必要があります。

最後に、実行時にコンパイラまたはプログラムが、関数との間でデータをやり取りする方法を決定します。また、各パラメーターの送信方向（「in」、「out」、または「in out」）を示す必要がありますが、データがレジスター、ヒープ、または参照によって転送されるかどうかの最終決定は、コンパイラーまたはプログラムによって実行時に行われますが、プログラマーではありません。これにより、スタックオーバーフローの問題が回避されます。

レイヴンズカーのプロフィールと方言SPARKはAdaのサブセットであり、後者は契約に重点を置いています。時間の経過とともに、これらのサブセットの機能はメイン言語の仕様に移行されました。

今日のエイダプログラミング


ANSIは1983年にAda 83仕様を確立しました。その後、Intel 80286がリリースされたばかりで、Motorola 68000はわずか4歳でした。マイクロコントローラの人気が高まり始めたのは、1970年代から80年代への不器用な移行と同様に、家庭用コンピュータの黎明期でした。Intel 8051マイクロコントローラーとその驚くべき4kb EPROMと128BのRAMを想像してみてください。



今日人気のあるマイクロコントローラーは、1983年のマイクロコントローラーよりも何倍も強力です。ARM、AVR、RISC-Vなどを使用できます。 （またはレゴマインドストームNXTキット）、同じCベースのツールを使用して開発を開始します。人気のあるGNAT Adaコンパイラが GCCに基づいていることは驚くに値しません。DragonEggプロジェクトの一環として開発中LLVMに基づくツールです。

GCCベースのAdaツールには2つのバージョンがあります。 AdaCoreオプションは商用サポートされていますが、独自の特性があります。 Free Software Foundationのオプションは自然に無料で機能的にはAdaCoreに匹敵します。

簡単に開始するには、AdaCoreにバンドルされているGNATプログラミングスタジオ IDE（GPS）（Githubのコピー）を使用するか、テキストエディターでコードを記述して手動でコンパイルするか、Makefileを使用します。ここでのツールキットはCまたはC ++よりも少し複雑ですが、開発はすべてのツールを含み、GCCとほぼ同様に機能するgnatmakeユーティリティによって促進されます。小さな例



、しかし、控えめな僕がコマンドライン引数パーサーとして書いたAdaの重要なプロジェクト。Makefileはada / projectフォルダーにあります。フォルダーは、パッケージ仕様ファイル（.ads）とパッケージ自体（.adb）を見つけることができる場所として定義されています。

これらのファイルは、CとC ++のヘッダーとコードを含むファイルにほぼ対応していますが、重要な違いもあります。Cとは異なり、Adaにはプリプロセッサがなく、コードとヘッダーを組み合わせてコンパイル済みファイルを作成しません。代わりに、仕様で指定されているパッケージ名へのリンクがあります。.adsファイルの名前も、パッケージの名前と一致する必要はありません。これにより、柔軟性が向上し、循環依存や、ヘッダーを特定の順序でリンクする必要があるという一般的なCの問題が回避されます。
