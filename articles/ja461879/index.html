抜粋「10.1」OpenVPNトンネルの作成»


この本では、暗号化についてすでに多くのことを話しました。 SSHとSCPはリモート接続を介して送信されるデータを保護でき（第3章）、ファイル暗号化によりサーバーに保存されているデータを保護でき（第8章）、TLS / SSL証明書はサイトとクライアントブラウザー間の送信中のデータを保護できます（第9章）。ただし、データをより広範囲の接続で保護する必要がある場合があります。たとえば、チームの一部のメンバーが公衆アクセスポイントを介してWi-Fi経由でネットワークに接続し、外出先で作業しなければならない可能性があります。そのようなアクセスポイントがすべて安全であるとは絶対に想定すべきではありませんが、従業員は会社のリソースに接続する方法を本当に必要としています。その場合、VPNが役立ちます。

適切に設計されたVPNトンネルは、データが安全でないネットワークを介して送信されるときにデータを隠すような方法で、リモートクライアントとサーバー間の直接接続を提供します。だから何？暗号化を使用してこれを行うことができる多くのツールをすでに見てきました。 VPNの真の価値は、トンネルを開くことにより、リモートネットワークをすべてローカルに接続しているかのように接続できることです。ある意味で、バイパスを使用しています。

このような拡張ネットワークを使用すると、管理者はどこからでもサーバーで作業を行うことができます。しかし、さらに重要なのは、リソースが複数のブランチに分散している企業は、どこにいても、それらを必要とするすべてのグループがそれらをすべて表示してアクセスできるようにすることです（図10.1）。

トンネル自体はセキュリティを保証しません。ただし、暗号化標準の1つをネットワーク構造に含めることができるため、セキュリティレベルが大幅に向上します。 OpenVPNオープンソースパッケージを使用して作成されたトンネルは、すでに読んだものと同じTLS / SSL暗号化を使用します。 OpenVPNはトンネリングに利用できる唯一のオプションではありませんが、最も有名なものの1つです。これは、IPsec暗号化を使用する代替レイヤー2トンネリングプロトコルよりもわずかに高速で安全であると考えられています。

出張中や別の建物で作業しているときに、チームの全員が安全に互いに通信できるようにしますか？これを行うには、OpenVPNサーバーを作成して、アプリケーションの共有とサーバーのローカルネットワーク環境へのアクセスを許可する必要があります。これを機能させるには、2つの仮想マシンまたは2つのコンテナーを起動するだけで十分です。1つはサーバー/ホストとして機能するため、もう1つはクライアントのためです。VPNの作成は簡単なプロセスではないため、全体像を把握するには数分を費やす価値があるでしょう。

OpenVPN


サーバーにOpenVPNをインストールするには、openvpnとeasy-rsa（暗号化キーの生成プロセスを制御するため）の2つのパッケージが必要です。必要に応じて、CentOSユーザーは、第2章で行ったように、最初にepel-releaseリポジトリをインストールする必要があります。サーバーアプリケーションへのアクセスを確認できるように、Apache Webサーバー（UbuntuのApache2およびCentOSのhttpd）をインストールすることもできます。

サーバーをセットアップしている間、22（SSH）と1194（デフォルトのOpenVPNポート）を除くすべてのポートをブロックするファイアウォールをアクティブにすることをお勧めします。この例はufwがUbuntuでどのように機能するかを示していますが、9章のファイアウォールCentOSプログラムを覚えていると思います。

# ufw enable
# ufw allow 22
# ufw allow 1194

サーバー上のネットワークインターフェース間の内部ルーティングを許可するには、/ etc / sysctl.confファイルの1行（net.ipv4.ip_forward = 1）のコメントを外す必要があります。これにより、接続後にリモートクライアントを必要に応じてリダイレクトできます。新しいパラメーターを機能させるには、sysctl -pを実行します。

# nano /etc/sysctl.conf
# sysctl -p

これでサーバー環境は完全に構成されましたが、準備が整う前に他に行うことは次のとおりです。次の手順を完了する必要があります（以下で詳しく説明します）。

  1. easy-rsaパッケージに付属のスクリプトを使用して、公開鍵基盤（PKI）を暗号化するための鍵セットをサーバーに作成します。本質的に、OpenVPNサーバーは、独自の証明機関（CA）としても機能します。
  2. クライアントに適切なキーを準備する
  3. サーバーのserver.confファイルを構成する
  4. OpenVPNクライアントを構成する
  5. VPNを確認する

暗号化キーの生成


生活を複雑にしないために、OpenVPNサーバーが実行されているのと同じマシン上でキーインフラストラクチャを構成できます。ただし、セキュリティに関する推奨事項では、通常、運用環境での展開には別のCAサーバーを使用することを推奨しています。OpenVPNで使用するための暗号化キーリソースを生成して割り当てるプロセスを図1に示します。10.2


OpenVPNをインストールすると、/ etc / openvpn /ディレクトリが自動的に作成されましたが、まだそこには何もありません。openvpnおよびeasy-rsaパッケージには、構成の基礎として使用できるサンプルテンプレートファイルが付属しています。認証プロセスを開始するには、easy-rsaテンプレートディレクトリを/ usr / share /から/ etc / openvpnにコピーし、easy-rsa /ディレクトリに移動します。

# cp -r /usr/share/easy-rsa/ /etc/openvpn
$ cd /etc/openvpn/easy-rsa

easy-rsaディレクトリには、かなりの数のスクリプトが含まれます。テーブルの中。10.1に、キーの作成に使用するツールを示します。


これらの操作にはroot権限が必要なので、sudo suを使用してrootになる必要があります。

使用する最初のファイルはvarsと呼ばれ、easy-rsaがキーを生成するときに使用する環境変数が含まれています。既存のデフォルト値ではなく、独自の値を使用するようにファイルを編集する必要があります。これが私のファイルの外観です（リスト10.1）。

リスト10.1 ファイルのメインフラグメント/ etc / openvpn / easy-rsa / vars

export KEY_COUNTRY="CA"
export KEY_PROVINCE="ON"
export KEY_CITY="Toronto"
export KEY_ORG="Bootstrap IT"
export KEY_EMAIL="info@bootstrap-it.com"
export KEY_OU="IT"

varsファイルを実行すると、その値がシェル環境に転送され、そこから新しいキーのコンテンツに含まれるようになります。 sudoコマンドだけでは機能しないのはなぜですか？最初の段階では、varsという名前のスクリプトを編集して適用します。アプリケーションとは、varsファイルがその値をシェル環境に転送することを意味し、そこから新しいキーのコンテンツに含まれます。

終了していないプロセスを完了するには、必ず新しいシェルを使用してファイルを再実行してください。これが完了すると、スクリプトは別のスクリプト（clean-all）を実行して、/ etc / openvpn / easy-rsa / keys /ディレクトリ内のコンテンツを削除するように要求します。


当然、次の手順は、clean-allスクリプトを実行し、次にbuild-caを実行して、pkitoolスクリプトを使用してルート証明書を作成します。varsによって提供される認証設定を確認するように求められます。

# ./clean-all
# ./build-ca
Generating a 2048 bit RSA private key

次はbuild-key-serverスクリプトです。新しいルート証明書とともに同じpkitoolスクリプトを使用しているため、キーペアの作成を確認するために同じ質問が表示されます。キーには、渡した引数に基づいて名前が付けられます。このコンピューターで複数のVPNを開始しない限り、通常は例のようにサーバーになります。

# ./build-key-server server
[...]

1 out of 1 certificate requests certified, commit? [y/n]y
Write out database with 1 new entries

OpenVPNは、Diffie-Hellmanアルゴリズム（build-dhを使用）によって生成されたパラメーターを使用して、新しい接続の認証をネゴシエートします。ここで作成されるファイルは秘密にすべきではありませんが、現在アクティブなRSA鍵のbuild-dhスクリプトを使用して生成する必要があります。将来、新しいRSA鍵を作成する場合は、Diffie-Hellmanアルゴリズムに基づいてファイルを更新する必要もあります。

# ./build-dh

サーバー側のキーは/ etc / openvpn / easy-rsa / keys /ディレクトリにありますが、OpenVPNはこれを認識しません。デフォルトでは、OpenVPNは/ etc / openvpn /でキーを探すので、それらをコピーします。

# cp /etc/openvpn/easy-rsa/keys/server* /etc/openvpn
# cp /etc/openvpn/easy-rsa/keys/dh2048.pem /etc/openvpn
# cp /etc/openvpn/easy-rsa/keys/ca.crt /etc/openvpn

クライアント暗号化キーの準備


すでに見たように、TLS暗号化は対応する鍵のペアを使用します。1つはサーバーにインストールされ、もう1つはリモートクライアントにインストールされます。つまり、クライアントキーが必要になります。私たちの古くからの友人pkitoolは、まさにこれに必要なものです。この例では、プログラムを/ etc / openvpn / easy-rsa /ディレクトリで実行し、それにclient引数を渡してclient.crtおよびclient.keyというファイルを生成します。

# ./pkitool client

これで、2つのクライアントファイルと、キー/ディレクトリにある元のca.crtファイルが安全にクライアントに転送されます。彼らの所属とアクセス権のため、これはそれほど簡単ではないかもしれません。最も簡単な方法は、PCのデスクトップで実行されているターミナルにソースファイルのコンテンツ（およびこのコンテンツのみ）を手動でコピーすることです（テキストを選択して右クリックし、メニューから[コピー]を選択します）。次に、クライアントに接続されている2番目のターミナルで作成したのと同じ名前の新しいファイルにこれを貼り付けます。

しかし、誰でもカットアンドペーストできます。代わりに、カットアンドペースト操作が可能なGUIに常にアクセスできるとは限らないため、管理者として考えてください。リモートのscp操作でファイルにアクセスできるように、ファイルをユーザーのホームディレクトリにコピーしてから、chownを使用してファイルの所有者をrootから通常の非rootユーザーに変更し、リモートscpアクションを実行できるようにします。すべてのファイルが現在インストールされ、アクセス可能であることを確認してください。少し後でクライアントに移動します。

# cp /etc/openvpn/easy-rsa/keys/client.key /home/ubuntu/
# cp /etc/openvpn/easy-rsa/keys/ca.crt /home/ubuntu/
# cp /etc/openvpn/easy-rsa/keys/client.crt /home/ubuntu/
# chown ubuntu:ubuntu /home/ubuntu/client.key
# chown ubuntu:ubuntu /home/ubuntu/client.crt
# chown ubuntu:ubuntu /home/ubuntu/ca.crt

一連の暗号化キーの準備が整ったら、VPNの作成方法をサーバーに通知する必要があります。これは、server.confファイルを使用して行われます。

キー入力の数を減らし

すぎて入力しすぎていませんか？ブラケット付きの拡張機能は、これらの6つのコマンドを2つに減らすのに役立ちます。これらの2つの例を調べて、何が起こっているのかを理解できると思います。さらに重要なのは、これらの原則を数十または数百の要素を含む操作に適用する方法を理解できることです。

# cp /etc/openvpn/easy-rsa/keys/{ca.crt,client.{key,crt}} /home/ubuntu/
# chown ubuntu:ubuntu /home/ubuntu/{ca.crt,client.{key,crt}}


server.confファイルの設定


server.confファイルがどのように見えるかをどのようにして知ることができますか？/ usr / share /からコピーしたeasy-rsaディレクトリテンプレートを覚えていますか？OpenVPNをインストールすると、圧縮された構成テンプレートファイルが残り、それを/ etc / openvpn /にコピーできます。テンプレートがアーカイブされているという事実に基づいて、便利なツールであるzcatを紹介します。

catコマンドを使用してファイルのテキストコンテンツを表示することはすでに知っていますが、gzipを使用してファイルを圧縮するとどうなりますか？ファイルはいつでも解凍でき、catはそれを喜んで表示しますが、これは必要以上に1または2ステップです。代わりに、おそらくご想像のとおり、zcatコマンドを入力して、解凍したテキストを1ステップでメモリにロードできます。次の例では、画面にテキストを印刷する代わりに、server.confという新しいファイルにリダイレクトします。

# zcat \
  /usr/share/doc/openvpn/examples/sample-config-files/server.conf.gz \
  > /etc/openvpn/server.conf
$ cd /etc/openvpn

10.1.2。OpenVPNクライアントの構成


伝統的に、トンネルは少なくとも2つの出口で構築されます（そうでなければ、洞窟と呼ばれます）。サーバー上で適切に構成されたOpenVPNは、一方でトンネルとの間でトラフィックを転送します。ただし、なんらかのクライアント側ソフトウェア、つまりトンネルの反対側にも必要です。

このセクションでは、OpenVPNクライアントとして機能するように手動でLinuxコンピューターを構成することに焦点を当てます。しかし、これはそのような機会を利用できる唯一の方法ではありません。 OpenVPNは、WindowsまたはmacOSを搭載したデスクトップコンピューターやラップトップ、およびAndroidやiOSベースのスマートフォンやタブレットにインストールして使用できるクライアントアプリケーションをサポートしています。詳細については、openvpn.netを参照してください。

OpenVPNパッケージはサーバーにインストールされたので、クライアントコンピューターにインストールする必要があります。ただし、easy-rsaは必要ありません。使用するキーはすでに存在しているためです。作成した/ etc / openvpn /ディレクトリにclient.confテンプレートファイルをコピーする必要があります。今回はファイルはアーカイブされないので、通常のcpコマンドはこのタスクを完全に実行します。

# apt install openvpn
# cp /usr/share/doc/openvpn/examples/sample-config-files/client.conf \
  /etc/openvpn/

client.confファイルの設定のほとんどは非常に簡単です。これらはサーバー上の値と一致する必要があります。次のファイルの例からわかるように、一意のパラメーターはリモート192.168.1.23 1194で、クライアントにサーバーのIPアドレスを通知します。繰り返しますが、これがサーバーのアドレスであることを確認してください。また、クライアントコンピュータがサーバー証明書を認証するように強制して、中間者攻撃の可能性を防ぐ必要があります。これを行う1つの方法は、remote-cert-tls serverという行を追加することです（リスト10.3）。


これで、/ etc / openvpn /ディレクトリに移動して、サーバーから認証キーを抽出できます。例のサーバーIPアドレスまたはドメイン名を実際の値に置き換えます。


クライアントでOpenVPNを実行するまで、エキサイティングなことは起こりません。いくつかの引数を渡す必要があるため、コマンドラインから実行します。--tls-client引数は、クライアントとして機能し、TLS暗号化を使用して接続することをOpenVPNに通知し、-configが構成ファイルをポイントします。

# openvpn --tls-client --config /etc/openvpn/client.conf

コマンドの出力を注意深く読んで、正しく接続されていることを確認してください。初めて問題が発生する場合は、サーバーとクライアントの構成ファイル間の設定の不一致、またはネットワーク接続/ファイアウォールの問題が原因である可能性があります。トラブルシューティングのヒントをいくつか紹介します。
