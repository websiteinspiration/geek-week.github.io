在2020年重新设计的React Native架构

React Native（RN）框架于2015年出现。它旨在使用React库开发跨平台的移动应用程序。这些应用程序支持为其创建平台的本机功能。原始的React Native架构具有某些缺陷。但是，尽管如此，RN得到了良好的社区支持，尤其是由于React的高知名度，其受欢迎程度正在逐步提高。



在2018年讨论了React Native架构项目。这项工作由Facebook团队完成。重新设计的目的是使框架更加稳定，并解决多年来在RN中积累的最常见问题。该材料（我们今天将要发布的翻译版）专门考虑RN体系结构的处理如何提高应用程序性能和程序员的速度。

旧建筑


React Native是可以称为独立于平台的解决方案的一个示例。结果，事实证明，该框架的主要目标是允许开发人员使用React机制编写JavaScript代码。React Native的目标是将React组件树转变为可以在各种移动平台上运行的组件。这意味着：

  • 正确的接口形成。
  • 提供对各种平台的本机功能的访问。

为了组织JS与本机线程之间的交互，使用了Bridge模块。该模块是用C ++编写的，并且基于异步队列。当网桥从各方之一接收数据时，它将对该数据进行序列化，将其转换为字符串形式，然后将其传递通过队列。当数据到达其目的地时，将对其进行反序列化。

这意味着所有线程都依赖跨网桥发送的异步JSON消息。各方发送这些消息，期望（但不能保证）将来的某个时候，这些消息将得到答复。使用这种数据交换方案，存在通信信道过载的风险。

这是通常给出的示例，以说明这种数据交换方案如何引起应用程序性能问题。假设应用程序用户滚动了一个巨大的列表。在本机环境中发生onScroll事件时，信息将异步传递到JavaScript环境。但是，本机机制不会等到应用程序的JavaScript部分完成其工作并向其报告为止。因此，在显示其内容之前，会有一个延迟，以列表中的外观表示。

同样，在重新计算布局的结果到达屏幕之前，系统必须执行几次数据交换会话。因此，在布局数据进入本机环境之前，需要在Yoga中对其进行处理。当然，这意味着还必须通过网桥传输此类数据。

如您所见，使用异步机制发送JSON数据以及对该数据进行序列化和反序列化会导致性能问题。这种机制是不完善的。但是，本机机制和JavaScript之间还可以如何进行交互？您可以使用JSI技术回答此问题。

新架构


在重新设计React Native架构的过程中，桥函数被逐步淘汰，取而代之的是称为JavaScript接口（JSI）的新机制。

使用JSI带来了一些重大改进。

第一个此类改进是JS捆绑包不再依赖JSC。换句话说，JSC引擎现在可以很容易地用其他东西替换，很有可能具有更高的性能。例如-V8引擎。

第二个改进是新的React Native架构的基础。事实是，由于使用了JSI，因此可以在JavaScript中存储指向C ++的链接-对象（主机对象）和这些对象的调用方法。结果，事实证明，应用程序的JavaScript部分和本机机制将比以前更加了解彼此。

换句话说，JSI允许所有线程之间的完全交互。通过使用共享所有权概念，JavaScript代码可以直接从JS流中运行本机方法。无需使用JSON来传输数据。这样，您就可以摆脱与网桥的使用有关的特定问题，该问题与可能的队列溢出和异步数据传输相关。


新的React Native架构（源）

除了上述方案显着改善了不同线程之间的交互这一事实之外，新架构还允许直接控制本地模块。这意味着然后可以使用本机模块，然后需要它们，而不必在应用程序启动时将其全部初始化。这导致启动应用程序的速度大大增加。

这种新的机制可以为开发新项目开辟道路，而这些新项目具有旧RN应用程序所不具备的能力。事实是，现在可供我们使用的是C ++的功能。这意味着基于React Native，现在可以创建比以前更多种类的应用程序。

摘要


经过多年的发展，React Native基础设施中积累了很多东西，如今这些东西已经被认为是过时的和未使用的。 RN开发人员渴望从中删除不必要的内容并提高支持它的能力，从而导致了其某些功能的删除。例如，我们正在谈论一个事实，即它的内核模块（例如Webview和AsyncStorage）正逐渐从框架中删除，并被作为React Native社区开发的存储库。

得益于新的紧凑型内核以及JavaScript代码与本机代码之间令人印象深刻的交互机制，React Native架构的重新设计应导致RN应用程序变得更加高效，并且它们将更快，更方便地开发。

React Native重组预计将于2020年第四季度完成。我对重新设计RN体系结构所带来的应用程序性能和可用性方面的观点感到满意。
