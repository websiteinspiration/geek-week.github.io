C ++でのシリアル化

この記事では、C ++でのシリアル化プロセスの自動化に焦点を当てます。最初に、入力/出力ストリームへのデータの読み取り/書き込みを簡素化するための基本的なメカニズムについて説明します。その後、基本的なlibclangベースのコード生成システムについて説明します。ライブラリのデモ版を含むリポジトリへのリンクは、記事の最後にあります。

ruSOでは、C ++でのデータのシリアル化に関して時々質問があります。これらの質問は、本質的に一般的なものであり、TCが基本的にどこから始めればよいのかわからない場合、特定の問題を説明する質問である場合があります。この記事の目的は、C ++でシリアル化を実装するための可能な方法の1つを要約することです。これにより、システムを実際に使用できる場合に、システムを構築するステップを最初のステップから論理的な結論までたどることができます。

1.初期情報


この記事では、バイナリデータ形式を使用します。その構造は、シリアル化可能なオブジェクトの種類に基づいて決定されます。このアプローチにより、サードパーティのライブラリを使用する必要がなくなり、標準のC ++ライブラリによって提供されるツールのみに制限されます。

シリアル化プロセスは、オブジェクトの状態をバイトのストリームに変換することから成り、書き込み操作を伴う必要があるため、低レベルの詳細を説明するときに「シリアル化」という用語の代わりに後者を使用します。同様に、読み取り/デシリアライズします。

記事の量を減らすために、オブジェクトのシリアル化の例のみを示します（逆シリアル化に言及する価値のある詳細が含まれている場合を除く）。完全なコードは上記のリポジトリにあります。

2.サポートされているタイプ


まず、サポートする予定の型を決定する価値があります。これは、ライブラリの実装方法に直接依存します。

たとえば、選択がC ++の基本的な型に限定されている場合、関数テンプレート（整数型の値を操作するための関数のファミリーです）とその明示的な特殊化で十分です。プライマリテンプレート（タイプstd :: int32_t、std :: uint16_tなどに使用）：

template<typename T>
auto write(std::ostream& os, T value) -> std::size_t
{
    const auto pos = os.tellp();
    os.write(reinterpret_cast<const char*>(&value), sizeof(value));
    return static_cast<std::size_t>(os.tellp() - pos);
}

注：シリアル化中に取得したデータがバイトオーダーが異なるマシン間で転送される予定の場合、たとえば、値をローカルバイトオーダーからネットワークバイトに変換してから、リモートマシンで逆の操作を実行する必要があるため、書き込み機能と同様に変更する必要があります。出力ストリームへのデータ、および入力ストリームからの読み取り機能用。

boolの専門分野：

constexpr auto t_value = static_cast<std::uint8_t>('T');
constexpr auto f_value = static_cast<std::uint8_t>('F');

template<>
auto write(std::ostream& os, bool value) -> std::size_t
{
    const auto pos = os.tellp();
    const auto tmp = (value) ? t_value : f_value;
    os.write(reinterpret_cast<const char*>(&tmp), sizeof(tmp));
    return static_cast<std::size_t>(os.tellp() - pos);
}

このアプローチは次のルールを定義します。タイプTの値が長さsizeof（T）のバイトのシーケンスとして表すことができる場合、プライマリテンプレートの定義を使用できます。それ以外の場合は、特殊化を決定する必要があります。この要件は、メモリ内のT型のオブジェクトの表現の機能によって指定できます。

コンテナーstd ::文字列を考えてみます。指定したタイプのオブジェクトのアドレスを取得してcharへのポインターにキャストし、出力ストリームに書き込むことができないことは明らかです。つまり、特殊化が必要です。

template<>
auto write(std::ostream& os, const std::string& value) -> std::size_t
{
    const auto pos = os.tellp();
    const auto len = static_cast<std::uint32_t>(value.size());
    os.write(reinterpret_cast<const char*>(&len), sizeof(len));
    if (len > 0)
        os.write(value.data(), len);
    return static_cast<std::size_t>(os.tellp() - pos);
}

ここで重要な点が2つあります。 

  1. 文字列の内容だけでなく、そのサイズも出力ストリームに書き込まれます。 
  2. std :: string :: size_typeをstd :: uint32_t型にキャストします。この場合、ターゲットタイプのサイズではなく、固定長であることに注意する必要があります。このような削減により、たとえば、マシンワードサイズが異なるマシン間でネットワークを介してデータが送信される場合の問題を回避できます。

したがって、基本的な型の値（およびstd :: string型のオブジェクトでさえ）は、関数writeテンプレートを使用して出力ストリームに書き込むことができることがわかりました。次に、サポートされているタイプのリストにコンテナーを追加する場合に必要な変更を分析します。オーバーロードのオプションは1つだけです-コンテナー要素のタイプとしてTパラメーターを使用します。std ::ベクトルの場合、これはうまくいきます：

template<typename T>
auto write(std::ostream& os, const std::vector<T>& value) -> std::size_t
{
    const auto pos = os.tellp();
    const auto len = static_cast<std::uint16_t>(value.size());
    os.write(reinterpret_cast<const char*>(&len), sizeof(len));
    auto size = static_cast<std::size_t>(os.tellp() - pos);
    if (len > 0)
    {
        std::for_each(value.cbegin(), value.cend(), [&](const auto& e)
            { size += ::write(os, e); });
    }
    return size;
}

、次にstd：map-いいえ。std:: mapテンプレートには少なくとも2つのパラメータ（キータイプと値タイプ）が必要なため。したがって、この段階では、関数テンプレートを使用できなくなり、より普遍的なソリューションが必要になります。コンテナーサポートを追加する方法を理解する前に、カスタムクラスがまだあることを思い出してください。明らかに、現在のソリューションを使用しても、シリアル化が必要なクラスごとに書き込み関数をオーバーロードするのは賢明ではありません。せいぜい、1つの書き込みパターンの特殊化が必要です。カスタムデータ型で機能します。ただし、このためには、クラスがそれぞれシリアル化を個別に制御できる必要があります。これらのクラスには、ユーザーがこのクラスのオブジェクトをシリアル化および逆シリアル化できるインターフェイスが必要です。少し後でわかるように、このクラスは、カスタムクラスを操作するときに、書き込みテンプレートの「共通の特徴」として機能します。それを定義しましょう。

class ISerializable
{
protected:
    ~ISerializable() = default;

public:
    virtual auto serialize(std::ostream& os) const -> std::size_t = 0;
    virtual auto deserialize(std::istream& is) -> std::size_t = 0;
    virtual auto serialized_size() const noexcept -> std::size_t = 0;
};

ISerializable から継承するクラスはすべて、以下に同意します。

  1. シリアル化のオーバーライド-状態（データメンバー）を出力ストリームに書き込みます。
  2. オーバーライドdeserialize —入力ストリームから状態（データメンバーの初期化）を読み取ります。
  3. オーバーライドされたserialized_size-オブジェクトの現在の状態のシリアル化されたデータのサイズを計算します。

 したがって、書き込み関数テンプレートに戻ります。一般に、ISerializableクラスの特殊化を実装できますが、それを使用することはできません。

template<>
auto write(std::ostream& os, const ISerializable& value) -> std::size_t
{
    return value.serialize(os);
}

毎回、この特殊化を利用するには、相続人タイプをISerializableにキャストする必要があります。最初に、シリアライゼーションに関連するコードの記述を単純化することを目標に設定しました。したがって、ライブラリでサポートされている型が基本型に限定されていない場合は、別の解決策を探す必要があります。

3. stream_writer


関数テンプレートを使用して、ストリームにデータを書き込むためのユニバーサルインターフェイスを実装することは、あまり適切なソリューションではないことがわかりました。チェックする必要がある次のオプションは、クラステンプレートです。関数テンプレートで使用されるのと同じ方法に従います-デフォルトでプライマリテンプレートが使用され、必要なタイプをサポートするために明示的な特殊化が追加されます。

さらに、ISerializableに関する上記のすべてを考慮する必要があります。明らかに、type_traitsに頼らずに多くの後続クラスの問題を解決することはできません。C++ 11以降では、標準ライブラリにstd :: enable_ifテンプレートが表示され、テンプレートクラスを無視できます。コンパイル中の特定の条件-これがまさに私たちが利用しようとしているものです。

Stream_writer クラステンプレート：

template<typename T, typename U = void>
class stream_writer
{
public:
    static auto write(std::ostream& os, const T& value) -> std::size_t;
};

書き込み メソッドの定義：

template<typename T, typename U>
auto stream_writer<T, U>::write(std::ostream& os, const T& value) -> std::size_t
{
    const auto pos = os.tellp();
    os.write(reinterpret_cast<const char*>(&value), sizeof(value));
    return static_cast<std::size_t>(os.tellp() - pos);
}

ISerializable の専門分野は次のとおりです。

template<typename T>
class stream_writer<T, only_if_serializable<T>>
    : public stream_io<T>
{
public:
    static auto write(std::ostream& os, const T& value) -> std::size_t;
};

ここで、only_if_serializableはヘルパータイプです。

template<typename T>
using only_if_serializable =
    std::enable_if_t<std::is_base_of_v<ISerializable, T>>;

したがって、タイプTがISerializableから派生したクラスである場合、この特殊化はそれぞれインスタンス化の候補と見なされます。タイプTがISerializableと同じクラス階層にない場合、候補は候補から除外されます。

ここで次の質問をするのは公平でしょう。結局のところ、プライマリテンプレートは、その特殊化と同じ一般的なパラメータの値を持ちます-<T、void>。スペシャライゼーションが優先されるのはなぜですか？回答：そのような動作は標準（ソース）で規定されているため、次のようになります。

std ::文字列の特殊化は次のようになります。

4.コンセプト


最初に、使用される概念について簡単に説明し、その後に更新された例を示します。

template<typename T>
concept String = std::is_same_v<T, std::string>;

正直なところ、この概念は詐欺のために定義されたものであり、次の行で説明します。

template<typename T>
concept Container = !String<T> && requires (T a)
{
    typename T::value_type;
    typename T::reference;
    typename T::const_reference;
    typename T::iterator;
    typename T::const_iterator;
    typename T::size_type;
    { a.begin() } -> typename T::iterator;
    { a.end() } -> typename T::iterator;
    { a.cbegin() } -> typename T::const_iterator;
    { a.cend() } -> typename T::const_iterator;
    { a.clear() } -> void;
};

コンテナには、タイプに対して「作成」する要件が含まれ、それがコンテナタイプの1つであることを実際に確認します。これは、stream_writerを実装するときに必要となる要件のセットです。もちろん、標準にはさらに多くの要件があります。

template<typename T>
concept SequenceContainer = Container<T> &&
    requires (T a, typename T::size_type count)
{
    { a.resize(count) } -> void;
};

順次コンテナーの概念：std ::ベクトル、std ::リストなど

template<typename T>
concept AssociativeContainer = Container<T> && requires (T a)
{
    typename T::key_type;
};

連想コンテナの概念：std ::マップ、std ::セット、std :: unordered_mapなど

ここで、連続するコンテナの特殊化を判断するために、あとはT型に制限を課すだけです。

template<typename T> requires SequenceContainer<T>
class stream_writer<T, void>
{
public:
    static auto write(std::ostream& os, const T& value) -> std::size_t;
};

template<typename T> requires SequenceContainer<T>
auto stream_writer<T, void>::write(std::ostream& os, const T& value) -> std::size_t
{
    const auto pos = os.tellp();
    // to support std::forward_list we have to use std::distance()
    const auto len = static_cast<std::uint16_t>(
std::distance(value.cbegin(), value.cend()));
    os.write(reinterpret_cast<const char*>(&len), sizeof(len));
    auto size = static_cast<std::size_t>(os.tellp() - pos);
    if (len > 0)
    {
        using value_t = typename stream_writer::value_type;
        std::for_each(value.cbegin(), value.cend(), [&](const auto& item)
            { size += stream_writer<value_t>::write(os, item); });
    }
    return size;
}

  • std ::ベクトル
  • std :: deque
  • std ::リスト
  • std :: forward_list

mapの場合、小さなニュアンスがあり、stream_readerの実装に関係しています。std :: map <K、T>のvalue_typeはそれぞれstd ::ペア<const K、T>です。入力ストリームから読み取るときにconst Kへのポインターをcharへのポインターにキャストしようとすると、コンパイルエラーが発生します。この問題は次のようにして解決できます。連想コンテナの場合、value_typeは単一のタイプKまたはstd ::ペア<const K、V>のいずれかであり、value_typeと内部でパラメーター化される小さなテンプレートヘルパークラスを記述できます。必要なタイプを決定します。

std :: setの場合、すべてが変更されません。

template<typename U, typename V = void>
struct converter
{
    using type = U;
};

std :: mapの場合-constを削除します。

template<typename U>
struct converter<U, only_if_pair<U>>
{
    using type = std::pair<std::remove_const_t<typename U::first_type>, typename U::second_type>;
};

連想コンテナ の読み取りの定義：

template<typename T> requires AssociativeContainer<T>
auto stream_reader<T, void>::read(std::istream& is, T& value) -> std::size_t
{
    const auto pos = is.tellg();
    typename stream_reader::size_type len = 0;
    is.read(reinterpret_cast<char*>(&len), sizeof(len));
    auto size = static_cast<std::size_t>(is.tellg() - pos);
    if (len > 0)
    {
        for (auto i = 0U; i < len; ++i)
        {
            using value_t = typename converter<typename stream_reader::value_type>::type;
            value_t v {};
            size += stream_reader<value_t>::read(is, v);
            value.insert(std::move(v));
        }
    }
    return size;
}


5.補助機能


例を考えてみましょう：

class User
    : public ISerializable
{
public:
    User(std::string_view username, std::string_view password)
        : m_username(username)
        , m_password(password)
    {}

    SERIALIZABLE_INTERFACE

protected:
    std::string m_username {};
    std::string m_password {};
};

このクラスのシリアル化メソッド（std :: ostream＆）の定義は次のようになります。

auto User::serialize(std::ostream& os) const -> std::size_t
{
	auto size = 0U;
	size += stream_writer<std::string>::write(os, m_username);
	size += stream_writer<std::string>::write(os, m_password);
	return size;
}

ただし、出力ストリームに書き込まれるオブジェクトのタイプを毎回示すのは不便であることを認めなければなりません。タイプTを自動的に推定する補助関数を作成します。

template<typename T>
auto write(std::ostream& os, const T& value) -> std::size_t
{
    return stream_writer<T>::write(os, value);
}

現在の定義は次のとおりです。

auto User::serialize(std::ostream& os) const -> std::size_t
{
	auto size = 0U;
	size += ::write(os, m_username);
	size += ::write(os, m_password);
	return size;
}

最後の章では、さらにいくつかのヘルパー関数が必要になります。

template<typename T>
auto write_recursive(std::ostream& os, const T& value) -> std::size_t
{
    return ::write(os, value);
}

template<typename T, typename... Ts>
auto write_recursive(std::ostream& os, const T& value, const Ts&... values)
{
    auto size = write_recursive(os, value);
    return size + write_recursive(os, values...);
}

template<typename... Ts>
auto write_all(std::ostream& os, const Ts&... values) -> std::size_t
{
    return write_recursive(os, values...);
}

write_all 関数を使用すると、シリアル化するすべてのオブジェクトを一度に一覧表示できます。一方、write_recursiveは、出力ストリームへの書き込みの正しい順序を保証します。計算の順序が折り返し式に対して定義されている場合（2項+演算子を使用する場合）、それらを使用できます。特に、関数size_of_all（前述していません。シリアル化されたデータのサイズの計算に使用されます）では、入出力操作がないために使用される折り返し式です。

6.コード生成


libclang-clangのC APIを使用してコードを生成します。高レベルのこのタスクは次のように説明できます。ソースコードを含むディレクトリを再帰的に巡回し、特別な属性でマークされたクラスのすべてのヘッダーファイルを確認し、ヘッダーファイルがある場合は、同じ属性のデータメンバーを確認し、データメンバー名の文字列をコンパイルする必要があります。コンマでリストされます。あとは、ISerializableクラスの関数の定義テンプレートを記述するだけです（必要なデータメンバーの列挙を配置するだけです）。

コードが生成されるクラスの例：

processTranslationUnit 関数の定義：

auto processTranslationUnit(const fs::path& path, const fs::path& targetDir) -> void
{
    const auto pathname = path.string();

    arg::Context context { false, false };
    auto translationUnit = arg::TranslationUnit::parse(context, pathname.c_str(), CXTranslationUnit_None);

    arg::ClassExtractor extractor;
    extractor.extract(translationUnit.cursor());

    const auto& classes = extractor.classes();

    for (const auto& [name, c] : classes)
    {
        SerializableDefGenerator::processClass(c, path, targetDir.string());
    }
}

			/* -    - */ 
                        /* -     */
            }
            return CXChildVisit_Continue;
        }
        , this);
}

ここでは、すでにclangのC API関数が直接表示されています。 libclangの使用方法を理解するために必要なコードのみを意図的に残しました。 「舞台裏」に残っているすべてのものは重要な情報を含んでいません-それは単にクラス名、データメンバーなどの登録です。より詳細なコードはリポジトリにあります。

そして最後に、関数processClassは、見つかった各クラスのシリアル化属性の存在をチェックし、存在する場合は、必要な関数の定義を含むファイルが生成されます。リポジトリは特定の例を提供します：名前空間の名前/ sを取得する場所（この情報はClassクラスに直接保存されます）とヘッダーファイルへのパス。

上記のタスクにはArgentumライブラリが使用されていますが、残念ながら使用はお勧めしません。他の目的で開発を開始しましたが、このタスクにはそこで実装されている機能が必要だっただけで、面倒でした。私はコードを書き直さなかったが、それをBintrayに投稿し、Conanパッケージマネージャーを介してCMakeファイルに接続した。このライブラリが提供するのは、クラスとデータメンバーのためのclang C APIに対する単純なラッパーです。
