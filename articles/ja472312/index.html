Mavenプロジェクトをマルチリリースjarに転送した経験：すでに可能ですが、それでも難しい

Java 8 Stream APIの機能を拡張する小さなStreamExライブラリがあります。私は伝統的にライブラリをMavenでコンパイルしましたが、ほとんどの場合、すべてが私に適しています。しかし、私は実験したかった。


ライブラリの一部の機能は、Javaのバージョンによって動作が異なります。最も印象的な例は、takeWhileJava 9でのみ登場した新しいStream APIメソッドです。私のライブラリは、Java 8でもこれらのメソッドの実装を提供しますが、Stream APIを自分で拡張すると、ここでは触れないいくつかの制限が適用されます。 Java 9以上のユーザーが標準実装にアクセスできることを願っています。


プロジェクトがJava 8を使用してコンパイルを続けるためには、通常、これはリフレクションツールを使用して行われます。標準ライブラリに対応するメソッドがあるかどうかを調べ、対応するメソッドがある場合はそれを呼び出します。ない場合は、実装を使用します。ただし、呼び出しのオーバーヘッドが少ないと宣言されているため、MethodHandle APIを使用することにしました。事前に取得MethodHandleして静的フィールドに保存できます。


MethodHandles.Lookup lookup = MethodHandles.publicLookup();
MethodType type = MethodType.methodType(Stream.class, Predicate.class);
MethodHandle method = null;
try {
  method = lookup.findVirtual(Stream.class, "takeWhile", type);
} catch (NoSuchMethodException | IllegalAccessException e) {
  // ignore
}

そしてそれを使います：


if (method != null) {
  return (Stream<T>)method.invokeExact(stream, predicate);
} else {
  // Java 8 polyfill
}
