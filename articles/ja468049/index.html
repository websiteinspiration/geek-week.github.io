werfを使用して複雑なHelmチャートを展開する



この記事は、チャートリポジトリの既製のソリューションを使用したKubernetesのHelmチャートの開発に特化しています。このアプローチでは、ユーザーはコミュニティレシピまたは彼自身のレシピを適用し、すべてのプロジェクトの標準コンポーネントをタイムリーに更新し、ソリューション全般を維持するための利便性を確保します。

この便利な機能がGitOps-utility werfに組み込まれました。これにより、Kubernetesアプリケーションでの収集と展開のためのインフラストラクチャを運用するプロセス全体が簡略化されます。

内蔵ヘルム


おそらく、Helmの主な成功-「Kubernetesのパッケージマネージャー」-は、公式のチャートリポジトリや一般的なリポジトリのサポートほど、その機能に関連しているわけではありません。レシピとチャートの設定には巨大なコミュニティが伴います。スペシャリストは、エンドユーザーが必要とする現在のソリューションをサポートします。公式リポジトリの各チャートは標準化され、十分に文書化されています。

些細なケースでは、ユーザーはアプリケーションを展開するためのチャートを作成する必要さえありません。既製のソリューションを見つけ、ドキュメントを読み、設定を準備し、公式のチャートを使用するだけです。

長い間、werf内でHelmを積極的に使用してアプリケーションインフラストラクチャを展開してきましたが、今ではさらに進んでいます。バージョンv1.0.4-alpha.10以降、werf は、Helmクライアントを完全に破棄するために、依存関係とチャートリポジトリを操作するコマンドを追加しました。

このための主なコマンド：

  • werf helm repo

    add         Add a chart repository
fetch       Download a chart from a repository and (optionally) unpack it in local directory
init        Init default chart repositories configuration
list        List chart repositories
remove      Remove a chart repository
search      Search for a keyword in charts
update      Update information of available charts locally from chart repositories
  • werf helm dependency

    build       Rebuild the charts/ directory based on the requirements.lock file
list        List the dependencies
update      Update charts/ based on the contents of requirements.yaml

さらに、チームwerf helm deploy-chartはチャート参照のサポートを追加しました。

そして、これがこの魔法のすべての動作の様子です-より正確には、ここでは、werf（左）とHelm（右）での同じチャートのロールアウトの比較を示します。



もっと練習


-全体のアプリケーション展開のための既製のソリューションを使用しての利便性に戻ると平凡な例ワードプレスを。werfを使用して彼のHelmチャートをKubernetesクラスターにロールアウトすると、次のようになります。

$ cat << EOF > values.yaml
wordpressBlogName: flant
wordpressUsername: admin       
wordpressPassword: password
mariadb:                                                          
  mariadbRootPassword: password
EOF
$ werf helm deploy-chart --values values.yaml stable/wordpress my-web-app

→ アシシネマでのデモンストレーション。

リポジトリには多くのソリューションが含まれているため、それらから独自のアプリケーションをキューブとして構築できます。キューブは、システム全体が依存するさまざまなHelmチャートになります。ほとんどの場合、必要に応じてコンポーネントを適切に構成するだけで済みます。

たとえば、Webアプリケーションには、キュー、キャッシュ、およびデータストレージが必要です。これらのコンポーネントをwerfを介してどのように展開しますか？

はじめに-CLIを使用して、必要なものすべての検索を高速化しましょう。

$ werf helm repo search queue 
NAME                  CHART VERSION    APP VERSION    DESCRIPTION                                                 
stable/rabbitmq       6.4.1            3.7.17         Open source message broker software that implements the A...
stable/rabbitmq-ha    1.31.0           3.7.15         Highly available RabbitMQ cluster, the open source messag...

類推すると、登録されているすべてのチャートリポジトリの残りのコンポーネントが見つかります（使用されているリポジトリのリストは、コマンドを実行して確認できますwerf helm repo list）。

すべてが見つかったら、それをチャートに追加するだけです。これは、いくつかの方法で実行できます。コンポーネントチャートをディレクトリに追加するかcharts、ファイルrequirements.yamlを作成して依存関係を記述し、特別なコマンドを使用してそれらと対話しますwerf helm dependency build|list|update。

2番目のパスの図：

$ cat << EOF > .helm/requirements.yaml
dependencies:
- name: mysql
  version: 0.3.4
  repository: "@stable"
- name: redis
  version: 1.1.11
  repository: "@stable"
- name: rabbitmq
  version: 0.6.16
  repository: "@stable"
EOF
$ werf helm dependency build
$ werf deploy --env test

→ アシシネマでのデモンストレーション。

ロールアウトプロセスでは、werf がすべての依存関係を作成し、残りのリソースとともにそれらを追跡します。追加のアクションは必要ありません。

その結果、経験豊富なコミュニティの専門知識を考慮に入れると、チャートの開発と保守にかかる時間を大幅に節約できます。ただし、公開チャートの使用に制限される人はいません。結局のところ、チャートを展開して同等の成功を収めることができます。その構成は、必要な詳細を考慮して準備されています。ここでは、werfのリポジトリおよび依存関係コマンドに関する

ドキュメントを示します。またについての詳細は、ドキュメントを読んで興味がある可能性がありWERFに展開し、Helmとの主な違い。

結論


既存のオープンソースソリューションで重要な機能がないため、対応する機能、バインディング、統合を作成する必要があります。werfプロジェクトは、まず第一に、DevOpsエンジニアの日常的なタスクを解決するために作成およびサポートされました。

例として、私たちは長い間（2年目！）作業してきました。Helmでリソースを追跡するという提案 が、プロジェクトのメインコードベース（上流）の代替モードとして受け入れられるようにするためです。コミュニティはアイデアを受け入れ、サポートしました。多くの人がこの機会を必要としています。しかし、この方向への進歩はHelm 3で始まり、今だけ...

これまで、Helmでのこのアイデアの実装は、開発者によって実質的にブロックされていました。ただし、この期間中に、適切なリソース追跡機能を別のオープンソースライブラリ（kubedog）で開発し、werfですでに使用しています。そして、GitHubでの活動から判断すると、ライブラリは他のKubernetes / Helmユーザーの間で興味を示しました。これは非常に喜ばしいことです。

GitHubのkubedogプロジェクトにスターを付けたり、Helmの元の問題を高く評価したりすることで、私たちのアイデア（および実装）をサポートできます。感謝！
