お金を地中に埋めないように製品を設計する方法


システムのアーキテクチャ設計を接続するための製品またはシステムの作成のどの段階で、後で費やしたお金のために耐え難いほど苦痛にならないようにしますか？CQRSとマイクロサービスを組み合わせるかどうかを決定する方法。


この記事は、ITソリューションの開発を要求するビジネス担当者を対象としています。製品の発売方法と、アーキテクチャに関連する不当なコストを回避する方法を紹介します。また、CQRSの使用がさまざまなアプリケーションクライアントでの機能の実装にどのように役立つか、およびマイクロサービスがまさに万能薬であるかどうかも確認してください。


CQRSについて簡単に


CQRS（Command-Query Responsibility Segregation） -システムの開発で使用されるテンプレートで、システムの任意のメソッドが要求（システムの状態を変更しない）またはコマンド（システムの状態を変更する）のいずれかであると述べています。実践が示すように、これはソフトウェア開発で最も一般的に使用されるパターンの1つです。さまざまなレベルで、さまざまな状況で使用できます。たとえば、データがOLTPシステムに頻繁に書き込まれ、OLAPシステムから読み取られる場合、従来のシステムのOLTP / OLAPへの分割は、データベースアーキテクチャでのCQRSテンプレートの使用にすぎません。

/// <summary>
///     Ninject,     
/// </summary>
public class NinjectFactory : IHandlersFactory
    {
        private readonly IResolutionRoot _resolutionRoot;

        public NinjectFactory(IResolutionRoot resolutionRoot)
        {
            _resolutionRoot = resolutionRoot;
        }

        public IAsyncCommandHandler<TCommand> CreateAsyncCommandHandler<TCommand>()
        {
            return _resolutionRoot.Get<IAsyncCommandHandler<TCommand>>();
        }

        public IAsyncQueryHandler<TQuery, TResult> CreateAsyncQueryHandler<TQuery, TResult>()
        {
            return _resolutionRoot.Get<IAsyncQueryHandler<TQuery, TResult>>();
        }

        public ICommandHandler<TCommand> CreateCommandHandler<TCommand>()
        {
            return _resolutionRoot.Get<ICommandHandler<TCommand>>();
        }

        public IQueryHandler<TQuery, TResult> CreateQueryHandler<TQuery, TResult>()
        {
            return _resolutionRoot.Get<IQueryHandler<TQuery, TResult>>();
        }
    }
