落とし穴のあるシングルトン

前書き


プログラミングには膨大な数のパターンとアンチパターンがあります。多くの場合、パターンを使用すると、自分自身の経験と実際の知識が決まります。この記事では、シングルトンパターンの適用、つまりUnityに適用されるネットでの実装について説明します。


シングルトン


私はチームでコードを書いているので、チームをアンロードし、Unityで特定のパターンを実装する際のいくつかの困難を考える必要をなくすために、コード内でできるだけ多くの作業を行います。


この質問2に関連して、ネット1の文献を研究し、いくつかのプロジェクトでの作業の結果、次のクラスが生まれました。


using UnityEngine;

/// <summary>
///    .
/// </summary>
/// <typeparam name="T">,    </typeparam>
/// <remarks>
///        OnDestroy  OnApplicationQuit
///      IsAlive.     
/// ,        .
/// 
/// 
///      Awake, OnDestroy, 
/// OnApplicationQuit    
/// base.Awake()  .
/// 
///    Initialization -      
///  .
/// 
///     unity,    
///   .    ,    
///       .
/// 
///  :
///     -  "CLR via C#"
///     - Chris Dickinson "Unity 2017 Game optimization"
///</remarks>

public class Singleton<T> : MonoBehaviour where T : Singleton<T>
{

    private static T instance = null;

    private bool alive = true;

    public static T Instance
    {
        get
        {
            if (instance != null)
            {
                return instance;
            }
            else
            {
                //Find T
                T[] managers = GameObject.FindObjectsOfType<T>();
                if (managers != null)
                {
                    if (managers.Length == 1)
                    {
                        instance = managers[0];
                        DontDestroyOnLoad(instance);
                        return instance;
                    }
                    else
                    {
                        if (managers.Length > 1)
                        {
                            Debug.LogError($"Have more that one {typeof(T).Name} in scene. " +
                                            "But this is Singleton! Check project.");
                            for (int i = 0; i < managers.Length; ++i)
                            {
                                T manager = managers[i];
                                Destroy(manager.gameObject);
                            }
                        }
                    }
                }
                //create 
                GameObject go = new GameObject(typeof(T).Name, typeof(T));
                instance = go.GetComponent<T>();
        instance.Initialization();
                DontDestroyOnLoad(instance.gameObject);
                return instance;
            }
        }

        //Can be initialized externally
        set
        {
            instance = value as T;
        }
    }

    /// <summary>
    /// Check flag if need work from OnDestroy or OnApplicationExit
    /// </summary>
    public static bool IsAlive
    {
        get
        {
            if (instance == null)
                return false;
            return instance.alive;
        }
    }

    protected void Awake()
    {
        if (instance == null)
        {
            DontDestroyOnLoad(gameObject);
            instance = this as T;
            Initialization();
        }
        else
        {
            Debug.LogError($"Have more that one {typeof(T).Name} in scene. " +
                            "But this is Singleton! Check project.");
            DestroyImmediate(this);
        }
    }

    protected void OnDestroy() { alive = false; }

    protected void OnApplicationQuit() { alive = false; }

    protected virtual void Initialization() { }
}

オブジェクトの作成


プロジェクト（特にチームワーク> 3人）を拡張する場合、オブジェクトを作成する順序が不明確になると、状況が発生することがよくあります。厳密に言えば3、Awake（）の呼び出しのシーケンスはランダムです（もちろん、これは完全に真実ではなく、プロセスに影響を与える可能性がありますが、ドキュメントは神聖です）。そのため、インスタンス{get;}プロパティを実装することにより、この厄介な欠点を排除する必要があります。その結果、他のクラスのAwake（）からシングルトンに完全にアクセスできます。
