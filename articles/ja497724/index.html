PythonでWebアプリを公開するためのサーバーの準備

こんにちは、ハブロフスク！

自由な時間があり、小さなウェブアプリケーションを作りたいという欲望がありました。centosに無料のサーバーもあるという考え（気象センサーからデータを受け取り、データベースに保存してから何か面白いことをする）があります。セットアップのチュートリアルも同じようです...しかし、執筆の時点では、完全に機能するものは1つも見つかりませんでした。python3を使用してCentOS 7.4サーバーにもアプリケーションをデプロイする場合*バンドル、uwsgi、nginx、ようこそcat。

それで、この記事ですでにあるべきものとカバーされないものは：

  1. CentOS 7.4を実行する物理サーバーまたは仮想サーバー（他のオペレーティングシステムおよびバージョンのパフォーマンスは保証されません）。
  2. スーパーユーザー特権でのサーバーへのアクセス*サーバーが仮想の場合、SSHを介してサーバーに接続する機能があると想定されます。

rootユーザーとして接続した後の最初のステップは、管理者権限を持つ新しいユーザーを作成することです（これは必須ではありませんが、少なくとも適切な口調です）。

adduser developer
passwd developer

ここで、ユーザーパスワードを入力して確認する必要があります

usermod -aG wheel developer

ローカル管理者グループにユーザーを追加する

su - developer

新しく作成したユーザーに切り替えます。

次に環境を準備します。

sudo yum install -y epel-release
sudo yum install -y python3 python3-pip python3-devel nginx gcc
sudo yum update -y python36 python2
sudo yum groupinstall -y "Development Tools" "Development Libraries"
sudo yum install -y python3-devel python36-virtualenv

最後から2番目の行に注目したいと思います。仮想環境を正しく起動するには、「開発ツール」と「開発ライブラリ」が必要です。フラグを使用して、確認を求めずにチームが実行できるようにします。

これで、作業ディレクトリの作成を開始できます。

sudo mkdir /opt/myproject && cd /opt/myproject
sudo chown -R developer /opt/
python3 -m venv myprojectenv
source myprojectenv/bin/activate

pip3 install --upgrade pip
pip3 install uwsgi flask

次に、Webアプリケーションのエントリポイントを準備します

vi /opt/myproject/wsgi.py（以下、viエディターを使用しますが、これは好みの問題です）。

def application(environ, start_response):
    start_response('200 OK', [('Content-Type', 'text/html')])
    return ["<h1 style='color:blue'>Testing Success!</h1>"]

起動後のこのコードにより、前に行ったすべての手順が成功したことを確認できます。テストするには、サービスを実行します。

uwsgi --socket 0.0.0.0:8080 --protocol=http -w wsgi

物理的なljcnegtまたはrdp接続では、結果はブラウザで0.0.0.0:8080で確認できます。SSH接続で、新しいセッションを開き、curlでテストします。

curl -v 0.0.0.0:8080

-vフラグを使用すると、完全な出力を表示できます。

テストが完了したら、CTRL + Cを押してサービスを終了する必要があります（それ以外の場合は、プロセスを探してkillで強制終了する必要があります。これも終了ですが、明らかに不便です）。

これで、メインのWebアプリケーションを作成できます。

vi /opt/myproject/myproject.py

from flask import Flask
application = Flask(__name__)

@application.route("/")
def hello():
   return "<h1 style='color:blue'>Hello There!</h1>"

if __name__ == "__main__":
   application.run(host='0.0.0.0')

テストプロセスは前のプロセスと非常に似ていますpython3 /opt/myproject/myproject.py。コマンドとチェックを使用してアプリケーションを起動します。結果は同じになるはずです。

エントリポイントを編集して、呼び出されたときにアプリケーションが起動するようにします。

vi /opt/myproject/wsgi.py

from myproject import application

if __name__ == "__main__":
   application.run()

今最も困難で興味深い。このアプリケーションから、バックグラウンドで自動的に機能するサービスを作成するときがきました。

まず、uwsgiのiniファイルを準備します。

vi /opt/myproject/myproject.ini

[uwsgi]
wsgi-file = myproject/wsgi.py

master = true
processes = 2

uid = developer
socket = /tmp/uwsgi/myproject.sock
chown-socket = nginx:nginx
chmod-socket = 666
vacuum = true

die-on-term = true

ここで、次の行に注意を払います。

  1. socket = /tmp/uwsgi/myproject.sock-サービスを高速化するには、unix-socketを使用する必要があります。nginxがそれに接続するには、一時フォルダー/ tmp / uwsgiに作成します。
  2. chown-socket = nginx：nginxはソケットの使用権をnginxユーザーに転送します
  3. chmod-socket = 666-前のものと同じ目的を果たします。異なるマニュアルやヒントには、異なるパラメーター値664、665、777がありますが、666は最低限の動作をすることが実験的に確立されています。

これで、サービスの作成に直接進むことができます。

sudo vi /etc/systemd/system/myproject.service

[Unit]
Description=uWSGI instance to serve myproject app
After=network.target

[Service]
ExecStartPre=-/usr/bin/bash -c 'mkdir -p /tmp/uwsgi chown nginx:nginx /tmp/uwsgi'
ExecStart=/usr/bin/bash -c 'cd /opt/myproject; source myprojectenv/bin/activate; uwsgi --ini myproject.ini'
ExecStartPost=/usr/bin/bash -c 'setenforce 0'
PrivateTmp=false
[Install]
WantedBy=multi-user.target

興味深い行はすべてServiceブロックにあります。

  1. ExecStartPreは、サービスを開始する前にシステムマネージャが実行するコマンドです。

    -/ usr / bin / bash -cは、コマンドの次のシーケンスの開始を通知し、「mkdir -p / tmp / uwsgi chown nginx：nginx / tmp / uwsgi」は、ソケット用の一時フォルダを作成し、権限をnginxユーザーに転送します。
  2. ExecStartはサービスを直接起動するコマンドです。このコマンドでは、作業ディレクトリに移動し、仮想環境をアクティブにして、iniファイルからuwsgi-serverを起動します。
  3. ExecStartPost-サービスの開始後に実行されるコマンド。私たちのケースでは、nginxサーバーからuwsgiサーバーへのリクエストを正しく転送するために必要です。
  4. PrivateTmp = falseは、作成された一時フォルダーを他のプロセスから見えるようにするパラメーターです。

ここでサービスを実行します。

systemctl start myproject
systemctl status myproject
systemctl enable myproject

最後のコマンドは、サーバーの再起動後に自動起動します。

そして最後のジャーク-nginxサーバーを構成し、外部ネットワークでWebアプリケーションを使用できるようにします。彼の作業のログは、コマンドを使用していつでも表示できます

journalctl -u  myproject

sudo vi /usr/lib/systemd/system/nginx.service

[Service]ブロックを見つけ、最後にPrivateTmp = falseを追加します。

その後、コマンドを使用してデーモンを再起動systemctl daemon-reload

し、サーバー構成に直接進みます。

vi /etc/nginx/nginx.conf

http {
    . . .

    include /etc/nginx/conf.d/*.conf;

    server {
        listen 80 default_server;

        . . .

httpブロックを見つけて、新しいサーバーを追加します。

server {
    listen 80;
    server_name server_domain_or_IP;

    location / {
        include uwsgi_params;
        uwsgi_pass unix:/tmp/uwsgi/myproject.sock;
    }
}

サーバーの電源を入れ、ファイアウォールのルールを編集する必要があります。

systemctl start nginx
systemctl enable nginx

firewall-cmd --zone=public --permanent --add-service=http
firewall-cmd --zone=public --permanent --add-service=https

firewall-cmd --reload

これで、ドメイン名またはIPアドレスでサーバーにアクセスすると、Flaskアプリケーションから応答が返されます。

この資料がお役に立てば幸いです。
