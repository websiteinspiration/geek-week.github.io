アジャイルが必要ですか？テストする5つのモデル

アジャイル宣言で署名された年に生まれた子供たちは、今年の成人を祝います。そして、大人はアジャイルがどこに適用できるかについて議論し続けます。通常、地域によって影響を受けます。ITの外部でアジャイルを使用することは可能ですか？時々ドラマが追加されます：アジャイル原子力発電所を建設してみましたか？芸術的効果の場合、これは確かに優れています。ただし、スピーカーの競争に勝たずに製品を作りたい場合は、特定の状況に関連して検討することをお勧めします。

この記事では、いくつかのアジャイル適用性評価モデルについて説明し、そのうちの1つであるPMIおよびアジャイルアライアンスのアジャイルプラクティスガイドで提示されているアジャイル適合性モデルについて詳しく説明します。

DSDM適合性フィルター


1994年、DSDMコンソーシアムはプロジェクト適合性フィルター（PSF）および組織適合性フィルター（OSF）アンケートを開発しました。これらのアンケートは、プロジェクトと組織の柔軟性を評価し、潜在的な問題の原因を特定するために設計されました。

アジャイルビジネスコンソーシアムは現在、DSDM プロジェクトアプローチアンケート（PAQ）をホストしています。これは、DSDMを使用するリスクを特定する17のステートメントのリストです。ステートメントは、DSDMの哲学と実践の理解、チームでの役割の遵守に関係しています。すでにアプローチを理解している場合は、慣れることは理にかなっています。



Alistair Cockburnの重要性とチームサイズの要因


Alistair Cockburnは、「システムの重要度」と「チームのサイズ」の指標を使用して、要因の組み合わせに応じて、Crystalファミリのメソッドのどれを適用する必要があるかを示しました。重要度とは、問題が発生した場合に実行できる被害のレベルを指します。Alistarは、TEDx でのスピーチでモデルの意味を説明しています。



ベームとターナー-レーダーチャート


このモデルでは、評価に5つの基準が使用され、結果が花びらチャートに適用されます。

著者は以下を評価することを提案します：

  1. スタッフ研修レベル
  2. 毎月変更される要件の割合
  3. 文化
  4. チームサイズ
  5. 重要度（前のモデルと同様）

モデルはオンラインで構築できます。スライダーを動かして図を表示します。



ステイシー複雑度モデル


モデルは、不確実性に関してプロジェクトを評価するために使用されます。

縦軸は、やりたいことが明確かどうかを示しています。横軸は、これを達成する方法を理解しているかどうかを示します。

軸は、プロジェクトが分類できる4つのドメイン（単純、複雑、複雑、カオス）を形成します。ドメインを定義した後、アプローチを選択できます。



他のモデルについては、たとえば、DSDMの開発者の1人であるMike GriffithsとAgile Practice Guide から読むことができます。次に、上記のモデルを統合したアジャイル適合性モデルに進みます。

アジャイル適合性モデル


使い方


理想的にはスポンサー、チーム、顧客の代表を含む数人が3つのドメインにグループ化された質問に答えます。

  • 文化-環境はアプローチにどのように貢献していますか？
  • チーム-チーム自体がアプローチを利用できますか？
  • プロジェクト-プロジェクト自体はどうですか？柔軟になるためにどのくらい必要であり、可能ですか？

グループは問題を議論し、一般的な評価に来て、それを花びら（レーダー）チャートに配置する必要があります。（ダウンロードできます）。質問に行きましょう。

ドメイン「カルチャー」


サポート：プロジェクトスポンサー（キュレーター）は、このプロジェクトでの柔軟なメソッドの使用をサポートしていますか？



注意：スポンサーが「やりたいことをしてください」と言った場合、これはまだサポートされていません。

信頼：チームと協力しているプロジェクトスポンサーと顧客代表を検討してください。これらの利害関係者は、チームが彼らのビジョンと製品またはサービスのニーズを、彼らのサポートと絶え間ないフィードバックで変えることができると思いますか？



入り口であなたが戦争と平和のように厚いTKで迎えられ、出口で悲観的な顔とレポート保護を備えた委員会があなたを待っている場合-これは非常に悪い兆候です。もう一回考えて。

決定：プロジェクトで作業を実行するためのローカルな決定を行う際に、チームには自律性がありますか？



私の知人の一人が誰かに「アジャイルを実装する」ようにアドバイスしました。会社の詳細を知っていたので、私はそれがすでに壁にボードを掛けていたかどうか尋ねました。ショックを受けた少女は、ボードを壁に掛けるべきではないと言った。

ボードを壁に掛けることができるかどうか自分で判断できない場合は、自律性を期待しないでください。この言葉はあなたについてではありません。

ドメイン「チーム」


チームサイズ：メインプロジェクトチームのサイズを次のスケールで見積もります
。1-9= 1; 10-20 = 2; 21-30 = 3; 31-45 = 4; 46-60 = 5; 61-80 = 6; 81-110 = 7; 111-150 = 8; 151-200 = 9; 201+ = 10.



おそらく100+チームでは、自己組織化では少し難しくなります;）

経験：チームの主要な役割の経験とスキルを検討します。チームには、役割ごとに1人の経験豊富なチームメンバーがいますか？



あなたがすべての穴を閉じる訓練生は考慮されません。

アクセス：チームは、フィードバックを受け取って質問に答えるために、少なくとも1人の顧客/ビジネス担当者に毎日アクセスできますか？



空想しないでください。前回の様子を思い出してください。

ドメイン「プロジェクト」


変更：毎月何パーセントの要件が変更される可能性がありますか？



Stacey Complexity Modelを思い出してください。何も変わらないとしたら、なぜ柔軟性が必要なのでしょうか。

重要度：欠陥が原因で発生する可能性のある損失を検討し、障害がどのように終了するかを判断します。



バグをピックアップして修正することは常に可能とは限りません。物理オブジェクトは特に悪いです。

納品：製品を開発して部品で評価できますか？顧客/ビジネス担当者は、増分についてタイムリーなフィードバックを提供できますか？



増分は、デモで表示するだけでなく、使用できるようにする必要があります。

結果


最後に、ポイントは値に接続されます。



焦点を絞った結果は、俊敏性への取り組みを示しています。ハイブリッドゾーンの結果は、柔軟で予測的なアプローチを組み合わせる可能性を示しています。結果が中心から離れるほど、最適なアプローチが予測的である確率が高くなります。グラフの両方向が極端になる可能性があります。

主なことは、結果を医学的診断として認識しないことです。これは、現在の状況を分析し、必要に応じて変更にさらに取り組むための開始点です。
