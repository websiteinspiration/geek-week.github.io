少し前まで、Okkoオンラインシネマ-Rekkoチャレンジ2019の推奨システムのコンテストがBoostersプラットフォームで開催されました。私にとっては、リーダーボードを使用したコンテストに参加した最初の経験でした（以前はハッカソンでのみ力を試しました）。タスクは面白く、練習でもよく知っています。賞金プールがあり、参加することが理にかなっています。その結果、主催者が記念のTシャツを出した14位になりました。いいね。感謝。 この記事では、簡単にタスクに没頭し、私が提案した仮説について説明します。また、推奨システムで競争を引きずり、スタッキングの経験なしにトップ15に入る方法について説明します。これは、コンテストに参加しようとしている人に特に役立ちます。



推奨システム


レコメンダーシステムの主な目的は、ユーザーが購入したいものをユーザーに提供することです（残念ながら、このような肥大化したビューは、商用アプリケーションによって課されます）。
タスクにはさまざまなステートメント（ランキング、類似するものの検索、特定の要素の予測）があり、したがって、それらを解決する方法があります。さて、私たちは皆、選択の変動性を気に入っています。これは、各問題に対するいくつかの潜在的な解決策のセットによって提供されます。さまざまなアプローチについては、Anatomy of Recommender Systemsの記事で詳しく説明されています。もちろん、誰もNFLの定理をキャンセルしませんでした。つまり、競争問題では、さまざまなアルゴリズムを試すことができます。

1.単純なモデル

それほど単純ではないことに気づき、私は何であるかを謙虚に扱い始めました。まず、シンプルなものから始めたかったのです。業界でよく使用されるいくつかのソリューションを試してみました。つまり、協調フィルタリング（この場合はアイテムベース）と行列の因数分解です。

コミュニティは、これらのタスクに適したいくつかのpythonライブラリ（暗黙的およびLightFM）を広範囲に使用します。1つ目は、ALSに基づく因数分解と、項目間マトリックスを前処理するためのいくつかのオプションを備えた最近傍協調フィルタリングです。2番目には、2つの特徴的な要素があります。

  • 因数分解はSGDに基づいています。これにより、WARPを含むサンプリングベースの損失関数を使用できるようになります。
  • ハイブリッドアプローチを使用して、ユーザーの潜在的なベクトルがその属性の潜在的なベクトルの合計になるように、モデルのユーザー属性とアイテムに関する情報を組み合わせます。同様にアイテムについても同様です。このアプローチは、ユーザー/アイテムにコールドスタートの問題がある場合に非常に便利です。

ユーザーの属性（映画の操作に基づいて表示する機能を除く）がなかったため、アイテムの属性のみを使用しました。

        interactions, lack_of_interactions = self.make_two_tables(interaction_matrix)
        if interaction_matrix_2 is not None:
            interactions_2, lack_of_interactions_2 = self.make_two_tables(interaction_matrix_2)
        else:
            interactions_2, lack_of_interactions_2 = interactions, lack_of_interactions

        self.num_items = interaction_matrix.shape[1]
        self.llr_matrix = np.zeros((self.num_items, self.num_items))

        # k11 - item-item co-occurrence
        self.k_11 = np.dot(interactions, interactions_2.T)
        # k12 - how many times row elements was bought without column elements
        self.k_12 = np.dot(interactions, lack_of_interactions_2.T)
        # k21 - how many times column elements was bought without row elements
        self.k_21 = np.dot(lack_of_interactions, interactions_2.T)
        # k22 - how many times elements was not bought together
        self.k_22 = np.dot(lack_of_interactions, lack_of_interactions_2.T)

    def make_two_tables(self, interaction_matrix):
        interactions = interaction_matrix
        if type(interactions) == csr_matrix:
            interactions = interactions.todense()
        interactions = np.array(interactions.astype(bool).T)
        lack_of_interactions = ~interactions
        interactions = np.array(interactions, dtype=np.float32)
        lack_of_interactions = np.array(lack_of_interactions, dtype=np.float32)
        return interactions, lack_of_interactions

    def entropy(self, k):
        N = np.sum(k)
        return np.nansum((k / N + (k == 0)) * np.log(k / N))

    def get_LLR(self, item_1, item_2):
        k = np.array([[self.k_11[item_1, item_2], self.k_12[item_1, item_2]],
                      [self.k_21[item_1, item_2], self.k_22[item_1, item_2]]])

        LLR = 2 * np.sum(k) * (self.entropy(k) - self.entropy(np.sum(k, axis=0)) - self.entropy(np.sum(k, axis=1)))
        return LLR

    def compute_llr_matrix(self):
        for item_1 in range(self.num_items):
            for item_2 in range(item_1, self.num_items):

                self.llr_matrix[item_1, item_2] = self.get_LLR(item_1, item_2)

                if item_1 != item_2:
                    self.llr_matrix[item_2, item_1] = self.llr_matrix[item_1, item_2]


    def get_top_n(self, n=100, mask=False):
        filtered_matrix = self.llr_matrix.copy()
        for i in tqdm(range(filtered_matrix.shape[0])):
            ind = np.argpartition(filtered_matrix[i], -n)[-n:]
            filtered_matrix[i][[x for x in range(filtered_matrix.shape[0]) if x not in ind]] = 0

        if mask:
            return filtered_matrix != 0
        else:
            return filtered_matrix


その結果、結果の行列をマスクとして使用して、最も重要な相互作用のみを残すことができます。ここでは、しきい値を使用するか、最高の値を持つtop-k要素を残すという2つのオプションがあります。同様に、1つの速度での購入に対するいくつかの影響の組み合わせを使用して、アイテムをランク付けします。つまり、テストは、たとえば、購入への変換の可能性に関するお気に入りに追加することがいかに重要かを示します。有望に見えますが、LLRスコアを使用したフィルタリングは非常に小さな増加をもたらし、いくつかの速度を組み合わせても結果が悪化するだけであることを使用が示しました。どうやら、メソッドはこのデータ用ではありません。プラスのうち、私はこのアイデアを実装する方法を理解している間、私は内部で暗黙のうちに掘り下げなければならなかったことにのみ注意することができます。

単純な協調フィルタリングの予測を改善できる別のアイデアが思い付きました。業界では、古い見積もりにある種の減衰関数を使用することがよくありますが、それほど単純なケースはありません。おそらく、次の2つのケースを考慮する必要があります。

  1. サービスを見ているユーザーはほとんどが新しい
  2. 「検査官。」つまり、比較的最近来て以前人気のあった映画を見ることができる人たちです。

したがって、コラボレーティブコンポーネントを2つの異なるグループに自動的に分割することが可能です。これを行うために、ユーザーと映画の交点で「1」または「0」に設定された暗黙の値の代わりに、この映画がユーザーの視聴履歴でどれほど重要であるかを示す関数を発明しました。

confidence(movie)=α∗StartTime+β∗ΔWatchTime


ここで、StartTimeは映画のリリース日であり、ΔWatchTimeはリリース日とユーザーがそれを視聴した日付の差であり、αとβはハイパーパラメーターです。

ここで、最初の用語は最近リリースされた映画のスピードを上げることに責任があり、2番目の用語は古い映画に対するユーザーの依存を考慮することです。映画がずっと前にリリースされ、ユーザーがすぐにそれを購入した場合、今日、この事実はあまり考慮されるべきではありません。これが目新しさである場合は、新しいアイテムを見る多くのユーザーに推奨する必要があります。映画がかなり古く、ユーザーがそれを今だけ見た場合、これは彼のような人にとっても重要になる可能性があります。

係数αとβを整理するのは簡単です。夜のために出発し、仕事は完了です。範囲についての仮定はまったくなかったため、当初は範囲が広く、以下は局所最適検索の結果です。
