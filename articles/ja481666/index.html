カスタムSwiftLintルール

こんにちはハブル！私の名前はアレックスです。FINCHのiOSデベロッパーです。間もなく新年は別の生活を始める時であり、SwiftLintのようなクールなものがこれに役立ちます。記事では、レガシープロジェクトやペットプロジェクトを含むすべてのプロジェクトに実装する必要がある理由を説明し、通常のラインを使用してこのツールを最大限に活用する方法も示します。

SwiftLintの概要とインストール方法については説明しません。ツールに慣れていない場合は、公式ドキュメントを読むことをお勧めします。

大規模なプロジェクトで作業するときに発生する一般的な問題-ホットフィックスまたはその他のものを装ってスタイルガイドに準拠していない-に直行することをお勧めします。しかし、実際には、スタイルガイドを尊重し、それを酔っ払ったとしても、論理的な内訳は含まれていませんが、審美的な喜びに明らかに影響を与えるような、ありふれたタイプミスが発生しないことを誰も保証できません。

したがって、覚えておいてください

。1. SwiftLintを使用すると、次のことができます。

  • スタイルガイド付きの1つのスタイル

実際にはすべて。

記事はここで終わりますが、もしそうなら、この記事を書き始めませんでした。興味深いことに、SwiftLint ではこれを行うことができません - ホットフィックスコードを記述してください。

2. SwiftLintを使用すると、次のことを防止できます。

  • 強制的に展開
  • 強い代表
  • サイクロマティック
  • 他に…

そのような間違いから安全であることは良いことだと思いますよね？これは、初心者の開発者にとって特に役立ちます。なぜなら、彼は学習するだけで、時にはそのようなエラーを疑わないからです。

3. Swiftlintは、独自のルールに従って拡張できます。

最初に、他の親にはならないクラスのfinalを指定します。 finalのおかげで、プロジェクトの組み立て時間を節約できます。 Appleのドキュメントは最終クラスについて私たちに言っているものです：
内部アクセスを持つ宣言（何も宣言されていない場合のデフォルト）は、宣言されているモジュール内でのみ表示されます。通常、Swiftはモジュールを構成するファイルを個別にコンパイルするため、コンパイラーは内部宣言が別のファイルでオーバーライドされているかどうかを確認できません。ただし、モジュール全体の最適化が有効になっている場合は、すべてのモジュールが同時にコンパイルされます。これにより、コンパイラーはモジュール全体について一緒に推論し、目に見えるオーバーライドがない場合は内部での宣言の最終を推論できます。

そんな不注意を簡単なレギュラーシーズンで解決していきます。次に、コードをプロジェクトに直接埋め込めるように、すぐにrubyに書き込みます。

final_class:
    included: ".*.swift"
    name: "Final class requrement"
    regex: '^class'
    message: "All classes must be final or nonfinal"
    saverity: error

小さくてかなり単純な例。ルールごとに同じようなものを書くことはしませんが、ソースコードは記事の最後にあります。

class SomeClass { } // 
internal class SomeClass { } // 
/* @non-final */ class SomeClass { } //

次のポイントはinitが必要です。会社ではストーリーボードを使用しないため、各UIViewクラスで致命的な初期化子を指定することは完全に正常なわけではありません。この場合は、独自のNLView（NL-NibLess）-必要なinitが1回だけ実装されるクラスがあります。プロジェクトの新しい開発者はこれを知らないかもしれませんが、SwiftLintの利点は責任あるリードではなく常に叱責されます。またはそれと一緒に。

required_init:
    regex: 'required init\?\(coder: NSCoder\)'
    message: "Use NL class instead"

ストーリーボードを引き続き使用している場合は、次のルールを使用して、すべてのストーリーボードがプライベートであることを確認できます。

open_iboutlets:
    included: ".*.swift"
    name: "IBOutlet opening"
    regex: '@IBOutlet ?(weak){0,1} var'
    message: "IBOutlet should be private"
    severity: error


open_ibaction:
    included: ".*.swift"
    name: "IBAction opening"
    regex: '@IBAction func'
    message: "IBAction should be private"
    severity: error

Foundationが完全に不要な場所で使用されることがよくあります。したがって、忘れないように毎回強調表示することをお勧めします。

 foundation_using:
    included: ".*.swift"
    regex: 'import Foundation'
    message: "Do you really need for Foundation ???"

印刷はかなり難しい操作であり、アプリケーションのパフォーマンス（特にループ）に大きな悪影響を与える可能性があることを皆さんに知ってもらいたいです。唯一の評決-プリントはまったくすべきではありません。

print_using:
    regex: 'print'
    message: "Print decrease performance of the app"
    severity: error

また、クラスのみのプロトコルは作成しないでください。そのような構文はすぐに廃止される可能性があり、これはSwift開発者には推奨されていません。

class_protocol:
    regex: ': class'
    message: "Use Anyobject instead"

以下は、R.swiftを使用するプロジェクトのルールです。 

image_name_initialization:
    included: ".*.swift"
    name: "Image initialization without R.swift"
    regex: 'UIImage\(named:[^)]+\)'
    message: "Use R.image.name() or typealias of this instead"
    severity: error

これは私が思いつくことのほんの一部にすぎませんが、インターネット上での例はさらに少なくなっています。githubで私の「コレクション」全体を見ることができます。
