SQL Server 2019のメモリ許可フィードバック

こんにちは！「MS SQL Server Developer」コースの開始を期待して、興味深い翻訳をもう1つ用意しました。





オプティマイザがクエリを実行するために必要なメモリ量を正しく計算しない場合、別のプロセスで使用される可能性のあるメモリの浪費であるか、ディスクへのデータの流出（ディスクの流出）があります。この問題を解決するために、マイクロソフトはMemory Grantフィードバックを追加しました。この記事では、Greg Larsenがこの仕組みについて説明します。


以前のバージョンのSQL Server（SQL Server 2019または15.xより前）のメモリ許可フィードバックは、バッチモードで実行されるクエリに対してのみ実装されていました。バッチモードのクエリは、一度に1行のみが処理される行モードのクエリとは対照的に、一度に最大900行をスキャンして計算します。バージョン15.xでは、メモリ許可フィードバックが拡張され、文字列モードでのクエリをサポートします。



メモリ許可のフィードバックとは何ですか？これは、以前の実行中に使用されたメモリ量を考慮して、クエリに必要なメモリの計算を調整するプロセスです。つまり、キャッシュされたクエリが最後の実行で使用したメモリが多すぎる場合、SQL Serverは次回の実行時のメモリ割り当てを減らします。または、SQL Serverが最後にメモリ不足を割り当てられたためにディスクを使用するクエリを検出した場合、クエリのメモリを増やします。メモリ許可に関するフィードバックの目標は、クエリが処理される行数に対応するメモリ量を使用するまで、クエリが実行されるたびにメモリ要件を調整することです。



メモリ許可フィードバック 機能（バッチモードと文字列モードの両方）は、インテリジェントクエリ処理機能ファミリ（IQP、インテリジェントクエリ処理）の一部です。図1は、Azure SQLデータベースとSQL Server 2019に存在するすべてのIQP機能と、以前のバージョンのAzure SQLデータベースとSQL Server 2017に含まれていたアダプティブクエリ処理の一部であった機能を示す図を示しています。





図1.インテリジェントなクエリ処理



メモリ許可フィードバックがどのように機能するかをさらに理解するために、クエリの初回および以降の実行時におけるクエリのメモリ割り当てを見てみましょう。



メモリの付与が時間とともにどのように変化するかを確認します。



時間の経過に伴うメモリ割り当ての変化を確認するために、同じクエリを数回実行します。起動のたびに、割り当てられたメモリの量と使用された量、およびデータベースエンジンがメモリ要件を調整するのに役立つ追加のフィードバック情報が表示されます。テストには、ここからダウンロードできるWideWorldImportersデータベースを使用します。リスト1に、文字列モードでメモリ許可に関するフィードバックを確認するためのテストリクエストを示します。

必要なメモリの見積もりが誤っている可能性がある理由



リクエストに必要なメモリの量に影響を与える多くの要因があります。必要なメモリの合計を計算するとき、クエリオプティマイザは、実行に関して各操作のメモリ要件を考慮する必要があります。クエリを実行するために必要なメモリを計算するときにオプティマイザが考慮に入れるいくつかのパラメータを次に示します。



  • クエリに含まれるテーブルのカーディナリティ
  • 選択可能なスピーカー
  • 推定行サイズ
  • データをソートおよび/または結合する必要がありますか
  • クエリは並列実行されていますか


必要なメモリを正しく計算するには、SQL Serverは、クエリの実行時にメモリが必要になる可能性のある多くの側面を考慮する必要があります。割り当てられているメモリが多すぎると、SQL Serverは他のプロセスで使用できるときにメモリを浪費します。または、十分なメモリが割り当てられていない場合、リクエストはデータをディスクにマージしますが、これは負荷の高い操作です。



SQL Serverは、初回のメモリ計算に必ずしも優れているとは限りません。 SQL Serverが誤ったメモリ量を計算する最も一般的な理由は、古い統計や不正確な統計です。ただし、統計が正しい場合でも、SQL Serverは他の何らかの理由で必要なメモリ量を過大または過小に見積もることができます。これらの理由の1つは、SQL Serverが返された列の推定行サイズを正しく評価しないことです。次の例では、行のサイズを大きくすると、常にNULLである3つのNVARCHAR（MAX）列が原因で、クエリが過度の許可警告を受け取る方法がわかります。



しかし今、SQL Serverが最初にクエリを実行したときに必要なメモリを誤って計算した場合、その後の実行では、メモリ許可フィードバックが使用されます。クエリが実行されるたびに、このクエリの前回の実行に割り当てられたメモリに関する情報が分析されます。前回の開始時にメモリ割り当てが正しくなかった場合、データベースエンジンはメモリ割り当てを修正します。

メモリ許可フィードバックの無効化



何らかの理由で行モードでメモリ許可フィードバックを無効にする場合（メモリ許可フィードバック-行モード）、これを行う方法はいくつかあります。1つ目は、データベースの互換性レベルを150未満に設定することです。この方法には1つの問題しかありません。バージョン15.xで登場した他のすべての機能も無効になります。または、オプションでデータベースレベルのフィードバックを無効にすることもできますROW_MODE_MEMORY_GRANT_FEEDBACK
