大規模なReactアプリケーションを開発するための実用的なガイドライン。計画、アクション、データソース、およびAPI

今日は、大規模なReactアプリケーションの開発に特化した資料の翻訳の最初の部分に注目します。 Reactを使用して1ページのアプリケーションを作成する場合、コードベースを混乱させることは非常に簡単です。これはアプリケーションのデバッグを複雑にし、プロジェクトコードの更新または拡張を困難にします。



Reactエコシステムには、アプリケーションの特定の側面を管理するために使用できる多くの優れたライブラリーがあります。それらのいくつかについて詳しく説明します。さらに、いくつかの実際的な推奨事項をここに示します。プロジェクトが適切にスケーリングする必要がある場合は、プロジェクトの作業の最初からこれらの推奨事項に従うと便利です。翻訳のこの部分では、計画、アクション、データソース、APIについて説明します。これから説明する大規模なReactアプリケーションを開発する最初のステップは計画です。

企画


ほとんどの場合、開発者はアプリケーションのこの作業段階をスキップします。これは、計画プロセス中にコードの記述が行われないためです。しかし、このステップの重要性を過小評価することはできません。これがなぜそうなのかはすぐにわかります。

applicationsアプリケーション開発時の計画はなぜですか？


ソフトウェア開発には、多くのプロセスの調整が必要です。同時に、すべてが非常に簡単に制御不能になる可能性があります。開発プロセスで遭遇する障害や不確実性は、プロジェクトのタイミングを脅かす可能性があります。 

締め切りに間に合うように支援することは、プロジェクト計画フェーズがあなたを助けることができるところです。この段階で、アプリケーションに必要なすべての機能を「棚の上に置きます」。プロジェクト全体の開発のタイミングを見積もるよりも、プログラマーにリストされている小さな個々のモジュールの作成にかかる時間を予測する方がはるかに簡単です。

複数のプログラマーが大規模なプロジェクトに参加する場合（これは通常発生します）、事前に開発された計画、特定のドキュメントの存在により、相互のやり取りが大幅に促進されます。実際、このドキュメントで作成されたさまざまなタスクは、個々の開発者に割り当てることができます。その存在は、チームメンバーが同僚が何をしているかを認識するのに役立ちます。

そして最後に、このドキュメントのおかげで、プロジェクトの作業がどのように進んでいるかを非常に明確に見ることができます。多くの場合、プログラマーはアプリケーションのある部分から別の部分に作業を移し、以前よりもはるかに遅れて以前の作業に戻ります。
アプリケーションの計画プロセスを検討してください。

▍ステップ3：データとモデル


アプリケーションの各コンポーネントに関連するいくつかのデータがあります。同じデータが複数のアプリケーションコンポーネントで使用されている場合、それらは集中状態ツリーの一部になります。状態ツリーはReduxを使用して管理されます。

このデータは多くのコンポーネントで使用されます。その結果、1つのコンポーネントによってデータが変更されると、そのデータは他のコンポーネントに反映されます。

アプリケーションの類似データのリストを作成します。モデル図になります。このリストに基づいて、レデューサーを作成することができます。

products: {
  productId: {productId, productName, category, image, price},
  productId: {productId, productName, category, image, price},
  productId: {productId, productName, category, image, price},
}

オンラインストアを使用して上記の例に戻りましょう。推奨製品と新製品のセクションでは、個々の製品を表すために使用されるのと同じデータ型を使用しています（などproduct）。このタイプは、アプリケーションレデューサーの1つを作成するための基礎として機能します。

アクションプランを文書化した後、データの操作を担当するアプリケーションレイヤーを設定するために必要な詳細のいくつかを検討します。

アクション、データソース、およびAPI


アプリケーションが大きくなると、Reduxリポジトリに関連付けられているメソッドの数が過剰になることがよくあります。アプリケーションの実際のニーズから離れて、ディレクトリ構造が悪化することがあります。これはすべて維持するのが難しくなり、アプリケーションに新しい機能を追加することはより複雑になります。

Reduxリポジトリのコードを長期的にクリーンに保つために、いくつかのことを調整する方法について話しましょう。最初からモジュールを再利用に適したものにすると、多くの問題を回避できます。最初は過剰に思えるかもしれませんが、プロジェクトを不必要に複雑にしているだけで、それを行うだけの価値があります。

▍API設計とクライアントアプリケーション


リポジトリの初期設定中に、APIから取得したデータの形式は、リポジトリの構造に大きく影響します。多くの場合、データはレデューサーに転送する前に変換する必要があります。

最近、APIを設計するときに何が必要で何をする必要がないかについて多くのことが言われています。バックエンドフレームワークやアプリケーションサイズなどの要素は、APIの設計方法に追加の影響を与えます。

サーバーアプリケーションの開発と同様に、補助機能を別のフォルダーに保存することをお勧めします。これらは、たとえば、データをフォーマットおよびマッピングするための関数です。これらの関数に副作用がないことを確認してください（純粋関数に関するこの記事を参照）。

export function formatTweet (tweet, author, authedUser, parentTweet) {
  const { id, likes, replies, text, timestamp } = tweet
  const { name, avatarURL } = author

  return {
    name,
    id,
    timestamp,
    text,
    avatar: avatarURL,
    likes: likes.length,
    replies: replies.length,
    hasLiked: likes.includes(authedUser),
    parent: !parentTweet ? null : {
      author: parentTweet.author,
      id: parentTweet.id,
    
  
}

このコード例では、関数formatTweetは新しいキー（parent）をフロントエンドアプリケーションのツイートオブジェクトに追加します。この関数は、渡されたパラメーターに基づいて、外部のデータに影響を与えることなくデータを返します。

ここでは、フロントエンドアプリケーションのニーズに対応する構造を持つ前述のオブジェクトにデータをマッピングすることで、さらに先に進むことができます。ただし、一部のキーを検証できます。
