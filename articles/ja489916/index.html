プロトブフまたはJSON構造のフロントエンド通信プロトコル？



私たちのチームの新しいプロジェクトでは、新製品用のフロントエンドフレームワークVUEを選択しました。バックエンドはPHPで記述されており、17年間正常に動作しています。

コードが大きくなり始めたとき、私はサーバーとのデータ交換を単純化することを考えなければなりませんでした。

バックエンドについて


プロジェクトは十分に大きく、機能は非常に混乱しています。そのため、DDDで記述されたコードには特定のデータ構造があり、プロジェクト全体としての汎用性のために複雑で膨大でした。

フロントエンドについて


4ヶ月 フロントの開発では、JSONをサーバーからの応答として使用し、State Vuexで便利な形式でマッピングしました。しかし、サーバーに戻るために、サーバーがDTOオブジェクトを読み取ってマップできるように、逆方向に変換する必要がありました（奇妙に見えるかもしれませんが、そうでなければなりません:)）

問題


それは何もないようです、彼らは何で働いたか、状態は大きなオブジェクトに成長しました。彼らはさらに小さなモジュールに分割し始め、それぞれに独自の状態や変更などがありました... APIはマネージャーからの新しいタスクの後に変化し始め、これをすべて管理することがますます困難になり、それが間違ってマッピングされ、フィールドが変更されました...

そしてここに解析やマッピングなどのエラーを排除するために、サーバーとフロントのユニバーサルデータ構造について考え始めました。

いくつか検索した後、2つのオプションが見つかりました。

  1. プロトコルバッファ
  2. フロント用のサーバー側JS DTO自動生成。これらのDTOでさらにJSON処理が行われます。

ペンを試した後、GoogleのProtobufを使用するのが慣習でした。

そしてそれが理由です：

  1. PHPやJSを含む、多くのプラットフォーム用に記述された構造をコンパイルするファンクショナルがすでにあります。
  2. 作成された構造.protoのドキュメントジェネレータがあります。
  3. 構造のバージョン管理を簡単にねじ込むことができます。
  4. PHPとJSの両方でリファクタリングすると、オブジェクトの検索が容易になります。
  5. そして、必要に応じて、gRPCなどの他のチップ。

話をやめて、すべてがどのように見えるか見てみましょう


PHP側での外観については説明しません。すべてがほぼ同じで、オブジェクトは同じです。

Node.js上のシンプルなクライアントJSとミニサーバーの例を紹介します。

まず、必要なデータ構造について説明します。ドカ。

product.proto

syntax = "proto3";

package api;

import "price.proto";

message Product {
    message Id {
        uint32 value = 1;
    }
    Id id = 1;
    string name = 2;
    string text = 3;
    string url = 4;
    Price price = 5;
}

price.proto

syntax = "proto3";
package api;

message Price {
    float value = 1;
    uint32 tax = 2;
}

service.proto

syntax = "proto3";

package api;

import "product.proto";

service ApiService {
    rpc getById (Product.Id) returns (Product);
}

このサービスについて少し説明します。使用しなくても、なぜ必要なのかを説明します。このサービスは、私たちの場合の文書化のためだけに説明され、何を受け入れ、何を与えるかを示しているので、必要なオブジェクトを置き換えることができます。これはgRPCにのみ必要です。

サーバーで表現

const express = require('express');
const cors = require('cors');

const app = express();
app.use(cors());

//           .
const Product = require('./src/proto/product_pb').Product;
const Price = require('./src/proto/price_pb').Price;

//    , .     .
app.use (function(req, res, next) {
  let data = [];
  req.on('data', function(chunk) {
    data.push(chunk);
  });
  req.on('end', function() {
    if (data.length <= 0 ) return next();
    data = Buffer.concat(data);
    console.log('Received buffer', data);
    req.raw = data;
    next();
  })
});

app.post('/api/getById', function (req, res) {
  //     Product.Id,   
  const prId = Product.Id.deserializeBinary(req.raw);
  const id = prId.toObject().value;

  //   " "      
  const product = new Product();
  product.setId(new Product.Id().setValue(id));
  product.setName('Sony PSP');
  product.setUrl('http://mysite.ru/product/psp/');

  const price = new Price();
  price.setValue(35500.00);
  price.setTax(20);

  product.setPrice(price);

  //       Product
  res.send(Buffer.from(product.serializeBinary()));
});

app.listen(3008, function () {
  console.log('Example app listening on port 3008!');
});

合計は何ですか


  1. 多くのプラットフォームで一度説明された構造に基づいて生成されたオブジェクトの形での単一の真のポイント。
  2. 混乱はなく、自動生成されたHTMLの形式と.protoファイルの表示の両方の明確なドキュメントがあります。
  3. どこでも、変更などなしで、特定のエンティティで作業が行われています（そして、フロントエンドはギャグを愛していることを私たちは皆知っています:)）
  4. このプロトコルの非常に便利な操作は、Webソケットを介して交換されます。

もちろん小さなマイナスがあります。これはシリアライズとデシリアライズの速度です。ここに例があります。
