ストレージフェイルオーバーの提供



みなさん、こんにちは！最近、オープンウェビナー「フォールトトレラントストレージの提供」が開催されました。アーキテクチャの設計で発生する問題、サーバー障害がサーバークラッシュの言い訳にならない理由、およびダウンタイムを最小限に抑える方法を検討しました。このウェビナーは、シティモービルのサーバー開発責任者であり、コース「高負荷アーキテクト」の教師であるIvan Remenによって実施されました。


なぜストレージの復元力が気になるのですか？


スケーラブルなストレージのフォールトトレランスについて考え、基本的なキャッシュの問題を理解することは、起動段階である必要があります。スタートアップを書くとき、最初に製品の最小バージョンを作成することは明らかです。しかし、成長すればするほど、生産性が向上し、ビジネスが完全に停止する可能性があります。そして、もしあなたが投資家からお金を得たら、もちろん、彼らはまた、一定の成長と新しいビジネス機能を必要とするでしょう。適切なバランスを見つけるには、速度と品質のどちらかを選択する必要があります。同時に、どちらか一方を犠牲にすることはできません。犠牲にする場合は、意識的に、一定の制限内で行います。ただし、普遍的なレシピや理想的なソリューションはありません。

読書のベースに寄り添う


これが最初のシナリオです。プロセッサまたはハードドライブの負荷が99％のサーバーが1台あるとします。ここで：

  • リクエストの90％が読み取られます。
  • リクエストの10％はレコードです。

この状況での最善の解決策は、レプリカについて考えることです。どうして？これが最も安価で簡単なソリューションです。



複製は次のように分類されます

。1.同期によって：

  • 同期;
  • 非同期;
  • 半同期。

2.ポータブルデータによると：

  • 論理（行ベース、ステートメントベース、混合）;
  • 物理的。

3.レコードあたりのノード数で：

  • マスター/スレーブ;
  • マスター/マスター。

4.イニシエーター別：

  • 引く
  • 押す

そして今、仕事は水の入ったバケツについてです。MySQLと非同期マスター/スレーブレプリケーションがあるとします。DCでクリーニングが進行中です。その結果、クリーナーがつまずき、マスターベースのサーバーにバケツの水を注ぎます。自動化により、最新のスレーブの1つがマスターモードに正常に切り替えられました。そして、すべてが機能し続けます。キャッチはどこですか？

答えは簡単です。複製できなかったトランザクションは失われます。その結果、ACIDのプロパティDに違反します。

次に、非同期レプリケーション（MySQL）の仕組みについて説明します。

  1. トランザクションをストレージエンジン（InnoDB）に記録する。
  2. トランザクションをバイナリログに記録する。
  3. ストレージエンジンでのトランザクションの完了。
  4. 確認を顧客に返す。
  5. ログの一部をレプリカに転送します。
  6. レプリカでのトランザクションの実行（p。1-3）。

そして今、問題は、レプリケーションで終わらないように、上記の段落で何を変更する必要があるかということです。

また、交換する必要があるのは、4番目と5番目の2つのポイントのみです（「ログの一部をレプリカに転送する」と「クライアントに確認を返す」）。したがって、マスターノードが停止した場合、トランザクションログは常にどこかにあります（アイテム2）。また、トランザクションがバイナリログに記録されている場合、トランザクションもいつか発生します。

その結果、次のように機能する準同期レプリケーション（MySQL）が得られます。

  1. トランザクションをストレージエンジン（InnoDB）に記録する。
  2. トランザクションをバイナリログに記録する。
  3. ストレージエンジンでのトランザクションの完了。
  4. ログの一部をレプリカに転送します。
  5. 確認を顧客に返す。
  6. レプリカでのトランザクションの実行（p。1-3）。

レコードベースに寄り添う


私たちがいるとき、正反対の問題について話しましょう：

  • リクエストの90％-記録。
  • 要求の10％が読み取られます。
  • 1サーバー;
  • 負荷-99％（プロセッサまたはハードディスク）。

ここでは、よく知られたシャーディングが役に立ちます。しかし、今度は別のことについて話しましょう：



そのような場合に非常に頻繁に、彼らはマスターマスターを使い始めます。ただし、この状況では役に立ちません。どうして？すべてがシンプルです。サーバー上にエントリは少なくありません。結局のところ、レプリケーションはすべてのノードにデータがあることを意味します。ステートメントベースのレプリケーションでは、SQLは実質的にすべてのノードで実行されます。C行ベースは少し簡単ですが、それでも高価です。また、マスターマスターには競合に関する問題があります。

実際、以下の状況でmaster-masterを使用することには意味があります。

  • ライトスルーフォールトトレランス（つまり、常に1つのマスターにのみ書き込むという考え方です）。仮想IPアドレスを使用して実装できます。
  • 地理分散システム。

ただし、マスターとマスターのレプリケーションは常に難しいことに注意してください。そして多くの場合、マスターマスターはそれが解決する以上の問題をもたらします。

シャーディング


シャーディングについてはすでに説明しました。つまり、シャーディングはレコードをスケーリングする確実な方法です。アイデアは、独立した（常にではない）サーバーにデータを分散することです。各シャードは独立して複製できます。

シャーディングの最初のルールは、一緒に使用されるデータは同じシャードにある必要があるということです。ここで数式が機能しsharding_key -> shard_idます。したがって、sharding_key一緒に使用されるデータは一致する必要があります。最初の難点は、間違って拾った場合sharding_key、すべてを再定義することが非常に困難になることです。次に、がある場合sharding_key、一部のクエリは実行が非常に困難になります。たとえば、平均値を見つけることができません。

これを実証するために、それぞれに3つの値を持つ（2; 2; 3）（0; 0; 500）の2つのシャードがあるとします。平均値は、（1 + 2 + 3 + 500）/ 6 = 84.33333になります。

次に、2つの独立したサーバーがあるとします。そして、各シャードの個別に平均値を再計算します。それらのうちの最初のものでは、2番目に2を取得します-166.66667。そして、これらの値を平均しても、正しい値とは異なる数値が得られます：（2 + 166.66667）/ 2 = 86.33334。

つまり、平均の平均はすべての平均と等しくありません。

avg(a, b, c, d) != avg(avg(a, b) + (avg(c, d))

単純な数学ですが、覚えておくことが重要です。

ソーシャルネットワークに対話システムがあるとします。対話には2人しか参加できません。すべてのメッセージは1つのテーブルにあります。

  • crc32（1 // 2）！= crc32（2 // 1）;

キャッシュ


そして、キャッシュについて一言。キャッシュはアンチパターンであると言えますが、このステートメントについては議論の余地があります（多くの人はキャッシュの使用を好みます）。しかし、概して、キャッシュは応答率を上げるためにのみ必要です。また、負荷を保持するように設定することはできません。

結論は簡単です-キャッシュなしで静かに生きるべきです。それらが必要となる可能性がある唯一の理由は、プロセッサでそれらが必要とされるのとまったく同じ理由、つまり応答速度を上げるためです。キャッシュが消えた結果としてデータベースが負荷に耐えられない場合、これは悪いことです。これは非常に失敗したアーキテクチャパターンであるため、そうするべきではありません。そして、どんなリソースがあっても、いつの日かキャッシュは確実に落ちます。

キャッシュの問題は論文です：

  • コールドキャッシュから開始します。
  • キャッシュ無効化の問題。
  • キャッシュの一貫性。

それでもキャッシュを使用する場合は、一貫したハッシュが役立ちます。これは分散ハッシュテーブルを作成する方法であり、1つ以上のストレージサーバーで障害が発生しても、格納されているすべてのキーと値を完全に再配置する必要はありません。ただし、これについて詳しくは、こちらをご覧ください。
