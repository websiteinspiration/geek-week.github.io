Qu'est-ce que le CI (intégration continue)

CI (intégration continue) - traduit littéralement par «intégration continue». Cela fait référence à l'intégration de morceaux individuels de code d'application entre eux. Le plus souvent nous collectons le code ensemble et vérifions:



  • Va-t-il?
  • Les tests automatiques réussissent-ils?


Tout le meilleur! CI vous permet d'effectuer ces vérifications automatiquement. Il est utilisé dans les équipes de développement avancées qui écrivent non seulement du code, mais aussi des autotests. On lui demande lors des entretiens - au moins une compréhension de ce que c'est. Oui, même avec des testeurs.



Par conséquent, je vais dire dans un article ce que c'est. Comment CI fonctionne et comment il sera utile pour votre projet.

Qu'est-ce que CI?



CI est l'assemblage, le déploiement et le test d'une application sans intervention humaine. Je vais maintenant expliquer avec un exemple.



Disons que nous avons deux développeurs - Masha et Vanya. Et le testeur Katya.



Masha écrit le code. L'ajoute au système de contrôle de version (du système de contrôle de version anglais , VCS ). Cela ressemble à une boîte de dépôt pour le code - un emplacement de stockage où toutes les modifications sont enregistrées et à tout moment, vous pouvez voir qui a changé quoi, quand.



Puis Vanya termine son morceau de fonctionnalité. Et enregistre également le code dans VCS.







Mais ce n'est que le code source - un ensemble de fichiers avec l'extension .java, ou tout autre. Pour que Katya puisse tester les changements, il vous faut:



  1. Construisez une build à partir du code source
  2. Exécutez-le sur une machine de test


Construire une build, c'est quand nous créons un fichier exécutable à partir d'un ensemble de fichiers de code source: vous







pouvez construire une build manuellement, mais c'est une hémorroïde inutile: rappelez-vous que dans quel ordre exécuter, quels fichiers dépendent les uns des autres, ne faites pas d'erreur dans la commande ... Habituellement, ils utilisent un spécial le programme. Pour java, il s'agit de Ant, Maven ou Gradle. Avec l'aide du collecteur, vous configurez le processus de génération une fois, puis vous l'exécutez avec une seule commande. Exemple de lancement pour Maven: 



mvn clean install


Il s'agit de semi-automatisation - vous avez toujours besoin d'une personne qui entrera dans une équipe et assemblera la construction avec des «stylos». Disons que les développeurs font cela. Lorsque Katya demande à la build de tester, Vanya met à jour la version à partir du référentiel et collecte la build.







Mais construisez un build, obtenez une application pour les tests. Elle doit encore être lancée! Cette opération est effectuée par le serveur d'applications. Les serveurs sont différents: wildfly, apache, jetty ...



S'il s'agit de wildfly, il vous faut:



  1. Attacher la génération au répertoire autonome / déploiements
  2. Démarrez le serveur (après avoir configuré le service une fois)


Et c'est encore une semi-automatisation. Parce que le développeur doit copier l'archive résultante après assemblage sur le banc de test et activer le service. Oui, cela se fait par quelques équipes, mais toujours par une personne.







Mais si nous supprimons une personne de ce schéma, nous obtenons CI!







CI est une application qui vous permet d'automatiser l'ensemble du processus. Il extrait les modifications du référentiel avec du code. Par lui-même! Il existe deux options de configuration:



  • CI interroge le référentiel "Hey coucou, avez-vous des changements ??" toutes les N heures / minutes, comment configurer.



  • Le référentiel fait signe à CI de s'engager: «Hé bonjour! Et j'ai une mise à jour ici! " (est-ce un crochet git ou équivalent dans votre VCS)







Lorsque le CI reçoit les modifications, il commence à construire la génération et les autotests.



Si l'assemblage a échoué (les tests ont échoué ou le projet n'a pas pu être assemblé), le système écrit un e-mail à toutes les parties intéressées:



  • Chef de projet (pour savoir ce qui se fait!)
  • Le développeur qui a apporté les modifications
  • N'importe qui d'autre - tel que vous l'installez, il en sera ainsi.








Si la génération réussit, CI déploie l'application sur la machine de test. Et à la fin, Katka peut tester le nouveau montage!







Oui, bien sûr, une fois que vous devez tout configurer - dites au serveur CI d'où obtenir les modifications, quels autotests exécuter, comment construire le projet, où le construire plus tard ... Mais ensuite je l'ai configuré une fois - et puis il le fait!



Les autotests devront également être écrits par nous-mêmes, mais bien fait =)







Si c'est sur vos doigts, alors le système CI ( intégration continue ) est une sorte de programme qui surveille votre contrôle de source, et lorsque des modifications y apparaissent, il les rassemble automatiquement, les construit, exécute des tests automatiques (bien sûr, s'ils sont écrits).



En cas d'échec, il le signale à toutes les parties intéressées, en premier lieu au dernier committer. (c) habr.com/en/post/352282






Programmes CI



Les plus populaires sont Jenkins et TeamCity.



Mais il existe un tas d'autres variantes - CruiseControl, CruiseControl.Net, Atlassian Bamboo, Hudson, Microsoft Team Foundation Server.





À quoi cela ressemble-t-il



Voyons à quoi cela ressemble du point de vue de l'utilisateur. Je vais montrer avec un exemple du système TeamCity.



Lorsque je me connecte, je vois toutes les tâches. Les tâches sont différentes:



  • Build build
  • Tests automatiques de conduite
  • Déployer l'application sur le banc de test
  • Exécutez des tests GUI (ou des tests Postman) sur ce support
  • Informer toutes les personnes intéressées par e-mail des résultats de l'assemblage et des tests


Les tâches peuvent être regroupées. Ici, disons, nous avons un projet CDI. En entrant, je vois les tâches de ce projet particulier:







  • CDI Archetype et CDI Core sont des versions. Ils vérifient que l'application est généralement en construction. Entraînez-vous en quelques minutes et exécutez pour chaque changement de code.
  • CDI Core avec tests - montage du projet avec tous les autotests, qui, comme on peut le voir à l'écran, plus de 4000 pièces. Les tests durent une demi-heure, mais sont également exécutés pour chaque validation.

Comment fonctionne CI



Comment et où CI collecte-t-il les builds et exécute-t-il les tests automatiques? J'utiliserai TeamCity comme exemple, mais d'autres systèmes fonctionnent de la même manière.



TeamCity lui-même ne collecte rien. L'assemblage et l'exécution des autotests ont lieu sur d'autres machines appelées «agents»:







«Agent» est un simple ordinateur. Un morceau de fer ou une machine virtuelle, pas le but. Mais comment cet ordinateur comprend-il ce qu'il doit faire?



TeamCity a un serveur et un client. Le serveur est l'application même dans laquelle vous piquez ensuite des boutons et regardez une belle image "comment tout s'est bien passé". Il est installé sur une seule machine.







Et l'application «client» est installée sur les machines «agent». Et lorsque nous cliquons sur le bouton "Exécuter" sur le serveur:







Le serveur sélectionne un client gratuit et lui donne toutes les instructions: ce qui doit être fait exactement. Le client collecte la génération, effectue des autotests, recueille le résultat et retourne au serveur: «On, hold, draw».







Le serveur affiche le résultat à l'utilisateur et envoie un e-mail à toutes les parties intéressées.

Nous pouvons également configurer pour chaque assembly une liste d'agents sur lesquels il peut s'exécuter. Pourquoi? Par exemple, nous avons la moitié des agents Linux et la moitié de Windows. Et nous avons fait l'assemblage pour un seul système. Ou une sorte de bogue flottant rampe sous Windows, mais le réparer est long et coûteux, et tout le client sous Linux - eh bien, pourquoi alors?



Et il arrive aussi que les agents soient répartis entre les projets pour qu'il n'y ait pas de combat - ce projet utilise Burana et Apollo, et que Chip and Dale. Le fait est qu'une seule tâche peut être effectuée sur un seul agent. Par conséquent, cela n'a aucun sens d'acheter une voiture cool pour un agent, immédiatement un tas de tests ne peut toujours pas être chassé.



En fin de compte, comment cela fonctionne: d'abord, l'administrateur achète des ordinateurs sous les «agents» et installe l'application client TeamCity sur eux. Ils ne devraient pas être trop cool, car ils ne feront pas beaucoup de tâches à la fois.







Dans ce cas, TeamCity vous payez le nombre de licences pour les agents de build. Donc, moins ils sont, mieux c'est.



Sur une machine distincte, l'administrateur installe le serveur TeamCity. Et le configure - configure les assemblages, indique quels assemblages peuvent être entraînés sur quelles machines, etc. Le serveur a besoin d'un emplacement pour stocker les artefacts - les résultats de l'assemblage.



Nous avons deux projets - Single Client et Factor, qui interagissent les uns avec les autres. Le testeur de client unique ne peut pas collecter le facteur localement. Il lance la construction dans TeamCity et télécharge la construction terminée à partir d'artefacts!


De plus, le développeur choisit l'assembly qu'il souhaite exécuter et clique sur «Exécuter». Que se passe-t-il en ce moment:



1. Le serveur TeamCity vérifie la liste des agents sur lesquels cet assembly peut être exécuté. Il vérifie ensuite lequel de ces agents est actuellement libre:







Trouvé gratuitement? Je lui ai donné le puzzle!







Si tous les agents sont occupés, la tâche est mise en file d'attente. La file d'attente fonctionne selon le principe FIFO - premier entré, premier sorti. Qui s'est levé le premier - ça et les pantoufles.







La file d'attente peut être ajustée manuellement. Donc, si je vois que la file d'attente est pleine d'assemblages lancés par le système de contrôle de version, je vais élever le mien au sommet. Si je vois que les gens ont commencé les assemblées, cela signifie qu'ils sont également importants, vous devez attendre.







C'est une pratique normale si les capacités des agents ne sont pas suffisantes tout au long et qu'une file d'attente est créée. Vous regardez qui l'a lancé:



  • Robot? Donc, c'est juste une vérification de routine que rien de plus n'a été cassé. Cela peut attendre 5-10-30 minutes, ça va
  • Collègue? Cette assemblée est importante pour lui, puisqu'il n'a pas attendu le lancement prévu. Nous faisons la queue, vous ne devez pas avancer.


Parfois, vous pouvez même annuler la génération à partir du système de contrôle de version, si elle est vraiment chaude, et tous les agents sont occupés avec des tests toutes les heures. Dans ce cas, vous pouvez:



  • élevez votre tour vers le haut pour qu'il commence au premier agent libre
  • connectez-vous à l'agent, annulez la version actuelle
  • redémarrez-la! Bien que cela tombe tout en bas de la ligne, il est juste moche d'annuler l'assemblage


2. L'agent exécute la tâche et renvoie le résultat au serveur







3. Le serveur dessine le résultat dans l'interface graphique et enregistre les artefacts. Je peux donc aller dans TeamCity et voir dans les artefacts les journaux complets des autotests passés, ou télécharger l'assembly du projet pour le déployer localement.



Il est fortement recommandé de préconfigurer le nombre d'assemblys que CI stockera. Parce que si les artefacts contiennent des builds de plus de 200 Mo et qu'il y en aura beaucoup, alors le prochain lancement de l'assembly échouera avec l'erreur «manque d'espace disque»:







4. Le serveur envoie l'e-mail - c'est comme ça que vous le configurez. Il peut faire une newsletter positive «l'assemblée a réussi», et il ne peut envoyer de mail qu'en cas d'échec «Oh, elle, quelque chose s'est mal passé!».





Intégration avec VCS



J'ai parlé de différentes options pour configurer l'intégration CI-VCS:



  • CI interroge le référentiel "Hey coucou, avez-vous des changements ??" toutes les N heures / minutes, comment configurer.



  • Le référentiel fait signe à CI de s'engager: «Hé bonjour! Et j'ai une mise à jour ici! " (est-ce un crochet git ou équivalent dans votre VCS)







Mais quand est-ce qui est utilisé?



Le meilleur de tous, bien sûr, est que le système de contrôle de version informe le serveur CI. Et exécutez le cycle complet pour chaque changement: assembler, tester, déployer. Ensuite, toute modification du code ira immédiatement à l'environnement de test, qui sera aussi pertinent que possible.







De plus, chaque modification exécute des autotests. Et si les tests échouent, il est immédiatement clair qui a commis la rupture. Après tout, cela fonctionnait et après les modifications de Vasya, il a soudainement cassé - ce qui signifie que son engagement a entraîné une chute. À de rares exceptions près, lorsque l'automne flotte.







Mais dans la vraie vie, un tel schéma est rarement applicable. Pensez-y, vous pouvez avoir de nombreux projets, de nombreux développeurs. Chacun commet quelque chose au moins une fois toutes les demi-heures. Et si vous exécutez 10 assemblys en une demi-heure pour chaque commit, les files d'attente dans TeamCity ne seront jamais ratissées!



L'un de nos produits a un module principal et il y a plus de 15 clients. Chacun a ses propres autotests. L'assemblage client est le noyau + les fonctionnalités client. En d'autres termes, une modification du projet racine peut affecter 15 assemblys différents. Donc, ils doivent tous être exécutés lors de la validation dans le noyau.



Lorsque nous avions 4 agents de build, des builds et des tests tout-en-un pour ces clients ont été lancés mardi soir. Et à 10 heures, il restait encore quelques heures à TeamCity.


Une autre option consiste à acheter de nombreux agents. Mais c'est le prix de la voiture elle-même + pour la licence dans TeamCity, qui est déjà beaucoup plus cher, et même payé tous les mois.



Par conséquent, ils font généralement ce



qui suit : 1. Les assemblages très rapides et importants peuvent être laissés à n'importe quel commit - si cela prend 1-2 minutes, laissez-les poursuivre.







2. Le reste des versions vérifie s'il y a eu des changements dans VCS - par exemple, toutes les 15 minutes. S'il y en avait, alors nous lançons.







3. Tests longs (par exemple, tests de performance) - une fois tous les quelques jours la nuit.











CI dans les tests



Si nous parlons de développer notre application, les tests sont inclus dans le cycle standard. Vous ou vos développeurs écrivez des autotests, qui sont ensuite pilotés par CI. Il peut s'agir de tests unitaires, api, gui ou de stress.







Mais que faire si vous testez une boîte noire? Il y a une application, pas de code source. Ce sont les dures réalités des testeurs d'intégration - le fournisseur vous donne une nouvelle version de l'application, que vous devez vérifier avant de la mettre en production.



Ici, disons que vous avez des tests d'API dans Postman. Ou des tests GUI dans Selenium. Puis-je personnaliser la boucle CI pour eux?







Bien sûr vous pouvez!



CI ne définit pas un cadre strict tel que «Je ne travaille que dans des projets avec des tests automatiques» ou «Je ne travaille que lorsque j'ai accès au code source». Il peut regarder dans le système de contrôle de version, ou il peut ne pas regarder. Ceci est une condition facultative!



A écrit des autotests? Dites au serveur CI comment les exécuter souvent - et profitez du résultat =)









Total



CI - intégration continue. C'est alors que votre application est constamment testée: tout est-il bon avec elle? Les tests réussissent-ils? L'assemblée va-t-elle? De plus, tous les contrôles sont effectués automatiquement, sans intervention humaine.



C'est particulièrement important pour les équipes où plusieurs développeurs travaillent sur le code d'une application. Comment cela peut-il arriver? Séparément, certaines parties du programme fonctionnent, mais ensemble plus. CI vous permet de détecter très rapidement de tels problèmes. Et plus vous le trouvez rapidement, moins il est coûteux de le réparer.







D'où le nom - une vérification constante de l'intégration de morceaux de code entre eux.

Tâches CI typiques:



  • Vérifiez s'il y a eu une mise à jour dans le code
  • Build build
  • Tests automatiques de conduite
  • Déployer l'application sur le banc de test
  • Exécutez des tests GUI (ou des tests Postman) sur ce support
  • Informer toutes les personnes intéressées par e-mail des résultats de l'assemblage et des tests








Et tout cela - automatiquement, sans intervention humaine! Autrement dit, ils l'ont mis en place une fois, puis il lui-même.



Si CI est configuré dans le projet, vous disposerez de bancs d'essai constamment à jour. Et si quelque chose casse dans le code, vous le saurez immédiatement, le serveur CI enverra une lettre. Et vous pouvez entrer dans l'interface graphique et voir si toutes les builds sont réussies, et les tests sont verts? Notez l'image du projet en une minute.
