自己文書化コードの10の原則

こんにちは！今日は、ピーター・グッドリフの著書「プログラマーズ・クラフト//良いコードを書くための実践」から引用した、完璧で理解可能なコードを書くためのヒントを共有したいと思います。

もちろん、コードを書く人なら誰でもこの面白い本を読むのは良いことですが、特に怠惰で、苦しみを減らし、同僚を誤解させたい（良心を持っている）人々のために、猫の下でコードを自己文書化する10の原則を示します。

1.適切なフォーマットで簡単なコードを書く


プレゼンテーション形式は、コードの理解しやすさに大きな影響を与えます。合理的な表現は、コードの構造を伝えます。関数、ループ、および条件ステートメントがより明確になります。

int fibonacci(int position)
{
     if (position < 2)
     {
         return 1;
     }
     int previousButOne = 1;
     int previous = 1;
     int answer = 2;

     for (int n = 2; n < position; ++n)
     {
         previousButOne = previous;
         previous = answer;
         answer = previous + previousButOne;
     }
     return answer;
}

すべての変数、型、ファイル、関数の名前は、わかりやすく、誤解を招くものではないはずです。名前はそれが何であるかを正しく説明する必要があります。意味のある名前が見つからない場合は、コードの操作を理解しているかどうかに疑問があります。

命名システムは一貫しており、不快な驚きを引き起こさないようにする必要があります。変数は、その名前が示す目的にのみ常に使用されるようにしてください。

名前の適切な選択は、おそらく不要なコメントを避けるための最良の方法です。名前は、コードを自然言語の表現力に近づけるための最良の方法です。

4.意味のあるタイプ名を選択する


可能な範囲で、使用可能な言語機能を使用して制限または動作を説明してください。例えば：

  • 変更されない値を決定するときは、定数タイプを割り当てます（Cではconstを使用）。
  • 変数が負の値をとってはならない場合は、符号なし型を使用します（言語に存在する場合）。
  • 列挙を使用して、関連するデータセットを記述します。
  • 変数のタイプを正しく選択してください。C / C ++では、size_t型の変数にサイズを書き込み、ptrdiff_t型の変数へのポインターを使用した算術演算の結果を書き込みます。

5.名前付き定数を使用する


if（counter == 76）が困惑している ようなコード。76の魔法の意味は何ですか？このチェックの意味は何ですか？マジックナンバーの実践は悪質です。それらはコードの意味を不明瞭にします。次のように書く方がはるかに良いです：

const size_t bananas_per_cake = 76;
...
if (count == bananas_per_cake)
{
    //   
}

定数76のコードがコードで頻繁に見つかる場合（すみません、bananas_per_cake）、追加の利点が得られます：パイのバナナの内容を変更する必要がある場合、エラーが多い76のグローバル検索/変更を実行するのではなく、コードを1か所で変更するだけで十分です。

これは、数値だけでなく、定数文字列にも当てはまります。見てみましょう任意の彼らが繰り返し起こる場合は特に、あなたのコード内のリテラルを。代わりに名前付き定数を使用した方がよいでしょうか？

8.ファイルにラベルを付ける


ファイルの内容と関連するプロジェクトの説明を含むコメントブロックをファイルの先頭に配置します。これは多くの作業を必要としませんが、大きな利点です。このファイルに同行しなければならない人は誰でも、それが何を扱っているかをよく理解できます。このタイトルには特別な意味がある場合があります。ほとんどのソフトウェア会社は、法的な理由により、各ソースファイルに著作権表示が必要です。通常、ファイルヘッダーは次のようになります。

/*********************************************************
* File: Foo.java
* Purpose: Foo class implementation
* Notice: (c) 1066 Foo industries. All rights reserved.
********************************************************/

9.エラーを正しく処理する


すべてのエラーの処理を最も適切なコンテキストに置きます。ディスクの読み書きに問題がある場合は、ディスクへのアクセスを処理するコードで処理する必要があります。このエラーを処理するには、別のエラー（「ファイルを読み込めません」などの例外）を生成して、より高いレベルに渡す必要がある場合があります。つまり、プログラムの各レベルでのエラーは、そのコンテキストでの問題の正確な説明でなければなりません。ユーザーインターフェイスコードでディスク障害に関連するエラーを処理しても意味がありません。

自己文書化コードは、エラーが発生した場所、それが何を意味し、現時点でのプログラムへの影響が何かを読者が理解するのに役立ちます。
