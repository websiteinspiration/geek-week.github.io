C ++ 20の新しい宇宙船オペレーター

C ++ 20は、「宇宙船」と呼ばれる新しい演算子を追加します  <=>。つい最近まで、Simon Brandは、この演算子が何であり、どのような目的で使用されているかについての詳細な概念情報を含む投稿を 公開しました。この投稿の主な目的は、「奇妙な」新しい演算子とその類似体の特定のアプリケーションを調査し、  operator==日常のコーディングで使用するための推奨事項を策定することです。


比較


次のようなコードが表示されることは珍しいことではありません。

struct IntWrapper {
  int value;
  constexpr IntWrapper(int value): value{value} { }
  bool operator==(const IntWrapper& rhs) const { return value == rhs.value; }
  bool operator!=(const IntWrapper& rhs) const { return !(*this == rhs);    }
  bool operator<(const IntWrapper& rhs)  const { return value < rhs.value;  }
  bool operator<=(const IntWrapper& rhs) const { return !(rhs < *this);     }
  bool operator>(const IntWrapper& rhs)  const { return rhs < *this;        }
  bool operator>=(const IntWrapper& rhs) const { return !(*this < rhs);     }
};

注：注意深い読者は、これがC ++ 20より前のコードよりも実際には冗長ではないことに気付くでしょう。これについては後で詳しく説明します。

私たちの型が同じ型の何かと比較できることを確認するには、たくさんの標準コードを書く必要があります。まあ、しばらくはこれに対処します。次に、このように書く人が来る：

constexpr bool is_lt(const IntWrapper& a, const IntWrapper& b) {
  return a < b;
}
int main() {
  static_assert(is_lt(0, 1));
}

最初に気づくのは、プログラムがコンパイルされないことです。

error C3615: constexpr function 'is_lt' cannot result in a constant expression

問題は、それがconstexpr 比較機能で忘れられていたことです。次にconstexpr 、すべての比較演算子に追加されます。数日後、誰かがアシスタントを追加  is_gtしますが、すべての比較演算子に例外指定がないためnoexcept 、5つのオーバーロードのそれぞれに追加するのと同じ面倒なプロセスを実行する必要があります  。

これは、新しいC ++ 20宇宙船オペレーターが私たちを助けるために来るところです。IntWrapper C ++ 20の世界でソースを作成する方法を見てみましょう。

#include <compare>
struct IntWrapper {
  int value;
  constexpr IntWrapper(int value): value{value} { }
  auto operator<=>(const IntWrapper&) const = default;
};

あなたが気づくかもしれない最初の違いは、新しい包含<compare>です。ヘッダー<compare> は、宇宙船オペレーターに必要なすべてのタイプの比較カテゴリをコンパイラーに入力する役割を果たし、デフォルトの関数に適したタイプを返します。上記のスニペットでは、戻り値の型  auto はになります  std::strong_ordering。

余分な5行を削除しただけでなく、何も決定する必要がなく、コンパイラーがそれを実行します。デフォルトでは明示的に指定していませんがis_lt、変更されずにそのまま機能し、同時に機能しconstexprますoperator<=>。これは良いことですがis_lt、たとえ宇宙船のオペレーターをまったく使用していなくても、なぜコンパイルが許可されているのか疑問に思う人もいます。この質問に対する答えを見つけましょう。

式の書き換え


C ++ 20では、「書き換えられた」式に関連する新しい概念にコンパイラーが導入されています。宇宙船のオペレーターは、とともに、operator==書き換え可能な最初の2つの候補の1つです。式の書き換えのより具体的な例については、に示す例を見てみましょうis_lt。

オーバーロードを解決する間、コンパイラーは最適な候補のセットから選択します。候補はそれぞれ、必要な演算子に対応しています。比較操作と等価操作の選択プロセスは、コンパイラーが特別な転記および合成された候補も収集する必要がある場合（[over.match.oper] /3.4）にわずかに変更されます。

私たちの表現では、a < b標準はタイプを検索できると述べていますaこのタイプを受け入れるfor operator<=>関数またはfor 関数operator<=>。これはコンパイラが行うことであり、型に実際にa含まれているものを発見しますIntWrapper::operator<=>。その後、コンパイラはこの演算子を使用して式a < bをに書き換えることができます(a <=> b) < 0。この書き換えられた式は、通常のオーバーロード解決の候補として使用されます。

この書き直された式が正しい理由を尋ねるかもしれません。式の正確さは、実際には宇宙船のオペレーターが提供するセマンティクスからきています。<=>-これは3者間比較です。これは、バイナリの結果だけでなく、（ほとんどの場合）順序も取得することを意味します。注文がある場合は、比較操作の観点からこの注文を表現できます。簡単な例として、C ++ 20の式4 <=> 5は結果を返しますstd::strong_ordering::less。結果std::strong_ordering::lessは、この値4と異なるだけで5なく、厳密にこの値よりも小さいことを意味します。これにより、操作の適用が(4 <=> 5) < 0正確で正確になり、結果を説明できます。

上記の情報を使用して、コンパイラは（つまり、任意の一般比較演算子を取ることができ<、>,,など）と宇宙船のオペレータの観点から、それを書き換えます。標準では、書き直された式は、多くの場合(a <=> b) @ 0、@ 比較演算を表します。

式の合成


読者は、上記の「合成された」式への微妙な言及に気付いたかもしれませんが、このステートメントを書き換えるプロセスでも役割を果たしています。次の関数について考えてみます。

constexpr bool is_gt_42(const IntWrapper& a) {
  return 42 < a;
}

の元の定義を使用するとIntWrapper、このコードはコンパイルされません。

error C2677: binary '<': no global operator found which takes type 'const IntWrapper' (or there is no acceptable conversion)

これはC ++ 20より前のバージョンでは理にかなっており、この問題を解決する方法は、の左側の機能friend にいくつかの追加機能を追加するIntWrapper ことですint。コンパイラとIntWrapper C ++ 20 定義を使用してこの例をビルドしようとすると、再び機能することに気付くでしょう。上記のコードがまだC ++ 20でコンパイルされている理由を見てみましょう。

オーバーロードを解決する間、コンパイラーは、標準が「合成された」候補と呼ぶもの、またはパラメーターの逆の順序で書き換えられた式も収集します。上記の例では、コンパイラは書き換えられた式を使用しようとします(42 <=> a) < 0、ただし、からIntWrapper への変換が行われていないintため、書き直された式は破棄されます。また、コンパイラーは「合成」式0 < (a <=> 42)を呼び出し、変換コンストラクターからint への変換が行われることを検出するIntWrapper ため、この候補が使用されます。

合成式の目的はfriend 、オブジェクトを他の型から変換できるギャップを埋めるために関数テンプレートを作成する必要があるかどうかの混乱を避けることです。合成された式はに一般化され0 @ (b <=> a)ます。

より複雑なタイプ


コンパイラーによって生成された宇宙船演算子は、クラスの個々のメンバーで停止するのではなく、タイプ内のすべてのサブオブジェクトに対して正しい比較のセットを生成します。

struct Basics {
  int i;
  char c;
  float f;
  double d;
  auto operator<=>(const Basics&) const = default;
};
 
struct Arrays {
  int ai[1];
  char ac[2];
  float af[3];
  double ad[2][2];
  auto operator<=>(const Arrays&) const = default;
};
 
struct Bases : Basics, Arrays {
  auto operator<=>(const Bases&) const = default;
};
 
int main() {
  constexpr Bases a = { { 0, 'c', 1.f, 1. },
                        { { 1 }, { 'a', 'b' }, { 1.f, 2.f, 3.f }, { { 1., 2. }, { 3., 4. } } } };
  constexpr Bases b = { { 0, 'c', 1.f, 1. },
                        { { 1 }, { 'a', 'b' }, { 1.f, 2.f, 3.f }, { { 1., 2. }, { 3., 4. } } } };
  static_assert(a == b);
  static_assert(!(a != b));
  static_assert(!(a < b));
  static_assert(a <= b);
  static_assert(!(a > b));
  static_assert(a >= b);
}

コンパイラは、配列であるクラスメンバーをサブオブジェクトのリストに展開し、それらを再帰的に比較する方法を知っています。もちろん、これらの関数の本体を自分で書きたい場合でも、コンパイラーによって式を書き換えることでメリットが得られます。

アヒルのように見え、アヒルのように泳ぎ、クワクワのように operator==


標準化委員会の非常に賢い人の中には、宇宙船の運営者が何であれ、常に要素の辞書式比較を行うことに気づいています。辞書式比較を無条件に実行すると、特に等価演算子を使用したコードが非効率になる可能性があります。

2つの行を比較する標準的な例。文字列が"foobar"あり"foo"、==を使用して文字列と比較する場合、この操作はほぼ一定であると期待できます。効果的な文字列比較アルゴリズムは次のとおりです。

  • まず、2つの線のサイズを比較します。サイズが異なる場合は、false
  • それ以外の場合は、2行の各要素を1つずつ確認し、違いがあるかすべての要素が終了するまで比較します。結果を返します。

宇宙船オペレーターの規則に従い、異なる要素が見つかるまで各要素を比較することから始める必要があります。この例では、"foobar"と"foo"する場合にのみ、比較、'b'そして'\0'、あなたは最終的に戻りますfalse。

これに対抗するため に、宇宙船のオペレーターとは無関係にコンパイラーがどのように書き換えて生成するかを詳しく説明する記事P1185R2がありました  operator==。私たちのもの  IntWrapper は次のように書くことができます：

#include <compare>
struct IntWrapper {
  int value;
  constexpr IntWrapper(int value): value{value} { }
  auto operator<=>(const IntWrapper&) const = default;
  bool operator==(const IntWrapper&) const = default;
};

もう1つステップ...しかし、朗報があります。上記のコードを書く必要はまったくありません。それを書くauto operator<=>(const IntWrapper&) const = defaultだけで、コンパイラが暗黙的に別のより効率的なコードを生成するのに十分operator==です！

コンパイラーは、==と!=に固有の、わずかに変更された「書き換え」ルールを適用します。これらのステートメントではoperator==、ではなくで書き換えられoperator<=>ます。これは、!=最適化からも恩恵を受けることを意味します。

古いコードは壊れません


この時点で、あなたは考えるかもしれません：まあ、コンパイラがこの演算子の書き換え操作を実行することが許可されている場合、コンパイラを裏返そうとするとどうなるでしょう：

struct IntWrapper {
  int value;
  constexpr IntWrapper(int value): value{value} { }
  auto operator<=>(const IntWrapper&) const = default;
  bool operator<(const IntWrapper& rhs) const { return value < rhs.value; }
};
constexpr bool is_lt(const IntWrapper& a, const IntWrapper& b) {
  return a < b;
}

答えは大したことではありません。C ++の過負荷解決モデルは、すべての候補者が直面する分野です。この特定の戦いでは、次の3つがあります。

  • IntWrapper::operator<(const IntWrapper& a, const IntWrapper& b)
  • IntWrapper::operator<=>(const IntWrapper& a, const IntWrapper& b)

（書き換え）

  • IntWrapper::operator<=>(const IntWrapper& b, const IntWrapper& a)

（合成）

C ++ 17でオーバーロード解決ルールを採用した場合、この呼び出しの結果は混合されますが、コンパイラーがこの状況を最も論理的なオーバーロードに解決できるようにC ++ 20オーバーロード解決ルールが変更されました。

コンパイラーが一連の追加パスを完了する必要がある場合、過負荷解決フェーズがあります。新しいメカニズムがC ++ 20で登場しました。その中で、書き換えまたは合成されないオーバーロードが優先されるため、オーバーロードがIntWrapper::operator<最良の候補になり、あいまいさを解決します。同じメカニズムにより、通常の書き直された式の代わりに、合成された候補の使用が防止されます。
