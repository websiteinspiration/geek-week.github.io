IoTアーキテクチャ

ほぼ1年前、私はIoTソリューションのアーキテクチャーに関する一連の記事を発表し始めました。（最初の記事habr.com/en/post/420173へのリンク）。そして最後に、シリーズの2番目の記事があなたの法廷に行きます。


IoTプロジェクトは似ていますが、共通のコンポーネントがあるのは事実です。しかし同時に、IoTプロジェクトには、市場の要求と解決すべきタスクのクラスから始まる根本的な違いがあります。

明らかに、各ソリューションを「ゼロから」開発することは非常に非効率的です。多くのコンポーネントは、プログラミングパターンの原則に従って再利用できます。

この記事では、IoTソリューションのさまざまなクラスとそれぞれの実装を分析します。その結果、IoTアーキテクチャーの実装手順について推奨事項が提示され、IoTソリューションの一般的な傾向を強調するように努めます。

また、特定のIoTソリューションのアーキテクチャの構築を開始する価値があることと、アーキテクチャの考えられる問題領域を検討することから、主要な問題領域の定式化も試みます。

IoTアプリケーション


今日、IoTソリューションのアプリケーションの3つの主要な地理的領域に分割されています。

ヨーロッパ


ヨーロッパのIoT市場では、天然資源の保護に関連するプロジェクトが検討されています。典型的な例は、昼/夜の照明、暖房の自動化、水管理、火災警報器です。

極東地域


この市場では、IoTソリューションの最優先事項はセキュリティです。典型的なアプリケーションは、道路、交通機関、および家庭での異常な状況を検出するカメラです。2番目の主要な領域は、地震活動、ハリケーン、台風、津波などのセキュリティおよび防災ソリューションです。

北米市場


輸送の最適化、商品の効率的な配送、スマートホームや都市の要素など、ビジネスプロセスの最適化を目的としたIoTプロジェクトは、主にこの市場で代表されています。

もちろん、IoTソリューションは1つの市場から別の市場に浸透し、排他的な特権ではありません。各地域では、IoTソリューションのすべてのクラスを見つけることができます。今日のグローバル市場に存在するこれらのクラスのIoT製品を分類することが重要です。以下は、そのようなクラスのIoTソリューションのリストです。

次の図は、IoTソリューションの階層化アーキテクチャを示しています。 IoTトポロジは、OSIなどの従来のレイヤーモデルとは異なります。これは線形または複雑なフローグラフではありません。一部のコンポーネントはオプションであり、特定のクラスのソリューションでは使用できない場合があります。 M2M（車から車へ）とM2P（車から人へ）の2種類のロジック、およびC2C（車から車へ、通常は1つのLTEモバイル通信セル内）などの特殊なケースが存在する可能性があります。

IoTソリューションには2つの物理的な場所があります。1つ目はエンド（周辺機器）デバイスで、2つ目はサーバーまたはクラウドのバックエンドデータセンターです。同時に、後で説明するように、これは古典的なクライアント/サーバーアプリケーションアーキテクチャではありません。



以下では、各レベルを個別に分析し、その機能を実際のクラスのIoTソリューションと比較します。

物理層-物理層


このレベルは、情報の収集（センサー）と機械的作業の実行（実行メカニズム）の2つのタイプの操作を表します。

センサーは、次のカテゴリに分類できます。

  • センサー：

    • 光：フォトダイオード/トランジスタ/抵抗器、PIR検出器
    • サウンド：マイク、超音波センサー
    • 機械的な動きの極限点を記録するスイッチ、特にリミットスイッチ。回転角度または回転速度の測定。
    • 電気容量、インダクタンス、抵抗などの物理的特性の変化を測定する電磁センサー。
  • 複雑または複合センサー。これらには、ガス、スペクトルなどの特殊なセンサーや、アプリケーションが増加している別のタイプの情報収集デバイスであるビデオカメラが含まれます。

IoTソリューションでは、物理要素には特定の共通要件があります。

  • IoTソリューションの量が多いため、可能な限り低価格。
  • 電池式で、低消費電力が必要です。今日の市場の需要は、1〜10年のメンテナンスなしの周辺機器の操作です。
  • 多くの場合、設置やメンテナンスのコストを最小限に抑えながら、遠隔地にアクセスできない場所にあります。
  • ビデオカメラを使用する場合、人工知能に基づく意思決定を伴う一次画像処理

IoTソリューションの実行メカニズムは、玄関ドアのロック、エンジンの作動、シンクロ、ライトのオン/オフ、暖房、水道、ガスなどを開きます。実行メカニズムの実装には大きな変更はありません。したがって、IoTソリューションのこの部分については、この記事では取り上げません。

エッジレイヤー-ペリフェラルコンピューティングレベル


このレベルは通常、単一のセンサーまたはアクチュエーターに接続されます。アナログ情報をデジタルに、またはその逆に変換するための最小限の機能を提供します。センサーを接続するには、価格と消費電力について同じ要件があります。これらのタイプのデバイスを製造する多くの製造元には、データモデル、構成、および操作に関する単一の標準がないため、個々の統合の問題が生じます。

消費電力を削減するために、周辺機器には通常4つの動作モードがあります。

  • 睡眠モード
  • センサーからの情報の測定と収集
  • 情報の通信、送信、受信のモード
  • インストールと接続モード

周辺デバイスは通常、物理、周辺コンピューティング、通信の3つのレベルを組み合わせます。周辺機器のコンピューティングレベルの主な機能は、ローカルETL（抽出、変換、ロード）です-センサーからの情報を受信、変換、保存します。このレベルは、センサーから情報を収集するだけでなく、それを標準形式にして、干渉をフィルタリングし、予備的な分析とローカルストレージにも責任があります。

ローカルネットワーク層-周辺通信層


データ転送は、周辺機器の中で最もエネルギーを消費する部分です。ほとんどの周辺機器は、主電源や有線通信に接続されていません。さらに、周辺機器はゲートウェイからかなり離れた場所（数km以内）に配置できます。一方、送信される情報量は通常非常に少量です。周辺通信レベルでは、次のプロトコルが使用されます。

  • ZigBee / Zwave
  • ブレ
  • ロラ
  • 独自のローバンド

アドホックとメッシュは、距離と信頼性を高めるために、このレベルで今日広く使用されています。

設定の目的で、NFCプロトコルも使用できます。最初のインストールまたはメンテナンス、あるいはその両方の際に、モバイルアプリケーションのサービスエンジニアは、周辺通信層を介して周辺デバイスに接続できます。周辺機器に印刷されたQコードが認証に使用されることもあります。

ゲートウェイレイヤー-ゲートウェイレベル


IoTソリューションでゲートウェイレイヤーを使用する理由はいくつかあります。

  • バックエンドが生の情報を受け取ると、その能力が高まり、コストが非常に高くなります。
  • バックエンドは、多数の周辺機器のリアルタイム応答を保証できません。
  • セキュリティ上の制限により、一部の情報はバックエンドに送信できず、人間が継続的に監視できません。そのような情報には、街路監視カメラからのデータ、医療情報などが含まれます。

ゲートウェイは、次の基本機能を提供する必要があります。

  • 周辺機器から第2レベルのETLを実装します。
  • バックエンドとの通信がなくても、重大な状況を修正してローカルの反応を示します。これは、脳の関与なしの人の心拍や肺呼吸の信号と比較できます。
  • Backndと通信します。処理された情報を周辺機器からサーバーに送信し、周辺機器の構成データを受信します。
  • 周辺機器の状態に関する情報、およびそれらによって収集されたデータを保存します。

場合によっては、AI / ML（人工知能/機械学習）機能がゲートウェイレベルに存在する必要があります。ゲートウェイデバイスは主に電源から給電されるか、大きな内蔵バッテリーを備えていますが、一部のソリューションでは低消費電力も必要です。この状況では、追加の問題が発生します-周辺機器との通信のための同期プロトコル。それらの1つ（ゲートウェイまたは周辺デバイス）は、他のデバイスが通信する準備ができているよりも頻繁に「通信準備完了」メッセージを送信する必要があります。どちらを選択するかは、各デバイスの総消費電力と、メンテナンスなしの所要時間によって異なります。

今日、私たちはビデオカメラの形で情報源を持つアプリケーションの数を増やしています。これらの特定のソリューションでは、ゲートウェイとエッジを一緒に統合できます。このようなアプリケーションのAI / ML機能は非常に人気が高まっています。組み込みシステム向けの新しいマシンインテリジェンスアクセラレータにより、このソリューションは現実のものとなりました。
それとは別に、スマートホームソリューションのゲートウェイについても言及する必要があります。このクラスのソリューションのゲートウェイは、多くの場合、STBデバイス（テレビアダプターまたはホームセキュリティコントロールユニット）と組み合わされます。最初の統合には、オープンなRDK-Vプラットフォームがすでに存在します。近い将来、3つのコンポーネントすべて（ゲートウェイ+ STB +セキュリティ）の1つのデバイスへのさらなる統合が期待されます。また、NAS（ローカルファイルストレージ）およびマシンビデオ/オーディオ認識用のAI / MLサービスも特徴とするでしょう。Alexaなどの音声認識デバイスはクラウドインフラストラクチャに基づいていますが、主な認識は周辺レベルにまで適用される可能性があります。

Wide Network Layer —


このレイヤーは、ソリューション全体の周辺部分とバックエンド部分を分離します。ゲートウェイは主に4G / 5Gなどのモバイルワイヤレスを使用してバックエンドに接続されますが、有線インターネットアクセスが使用されることもあります。外部通信の論理層には、LvM2Mと呼ばれるIoTソリューション用の標準プロトコルがあります。 LvM2Mプロトコルは各周辺機器にアクセスするように設計されていますが、多くの周辺機器ベンダーはLvM2Mインターフェイスをサポートしていないため、ゲートウェイデバイスはこの問題を解決し、周辺機器と通信するためのラッパーを作成できます。

外部通信層には、通信サービスとISOモデルも含まれています。これには、DNSベースのバランシングおよびロケーションサービス、COAPトランスポートプロトコル、DTLS暗号化、およびこの記事の範囲を超える他の多くのコンポーネントが含まれます。

ここで重要なコメントが1つあります。LvM2MプロトコルはDTLS暗号化プロトコルを使用します。DTLSプロトコルは、セキュリティキーとハンドシェイクセッションを備えたプロトコルです。ポイントツーポイントベースで機能します。DTLSパケットを復号化するには、接続セッション中に使用したのと同じバックエンドインスタンスを使用する必要があります。これは、回路のセキュリティレベルの一部であるロードバランサーに問題を引き起こします。次に、ロードバランサは、システム負荷が高い場合の自動スケーリングに必要です。この制限を回避するには、DNSをロードバランサーとして使用します。すべてのN DNSクエリは、セキュリティレベルインスタンスの新しいIPアドレスを受け取ります。

Middleware Layer —


このレイヤーは、負荷分散、メッセージキューイング、およびストリーミングのための内部クラウド機能を提供します。このレイヤーのコンポーネントは複製され、自動的にスケーリングされます。レベルは、主にクラウドプロバイダーからのマイクロサービスまたはPaaSに基づいて実装されます。この要件は、データ量のジャンプとディップのパラダイムから生じています。自動スケーリングにより、バックエンド実装のコストが削減されます。サービスの実際の実装は異なる場合がありますが、一般的な原則は1つのままです。つまり、非同期メッセージ転送にバッファリングとロードバランシングを提供します。このようにして、さまざまなバックエンドコンポーネントが独立してジョブを実行し、負荷に応じて水平方向にスケーリングできます。

この図は、サーバー内通信パターンの概略ブロック図を示しています。ロードバランサは、異なるサービス間で負荷を分散するように設計されています。キュー-キューは、順次サービスの非同期操作を実装するための中間バッファリングを提供します。サブスクライバー-受信者は、前のメッセージを処理した後にメッセージを順次受信するために、そのロジックに対応するキューにサブスクライブします。

ETLレイヤー-データの収集、処理、保存のレベル


内部ETLレベル（抽出、変換、およびロード）は、3番目のETL操作です。1つは周辺機器で、2つ目はゲートウェイです。バックエンドETLは、すべての周辺機器とゲートウェイからデータを収集し、次の操作を担当します。

  • 情報収集
  • 標準ビューに情報をもたらす

このレイヤーの一般的な実装スキームを図に示します。データ収集（抽出）の操作には、関連するキューからの情報の読み取りが含まれます。変換操作は、Lambdaなどの特殊なクラウドサービスによって、またはコンテナー内のコンピューティング手段と仮想マシンのみによって実行できます。上記の各方法には、それぞれ正と負の特性があります。たとえば、Lambdaサービスはほぼ完全な自動化には便利ですが、作成にかなりの時間がかかるため、表示されたイベントに迅速に対応する必要がある場合は適用できません。ラムダは使用時間に対して課金されるため、連続処理にもあまり適していません。最も一般的に使用されるサービスは、コンテナ化されたコンピューティングです。これらは便利にスケーラブルで、さまざまなバックエンドに簡単に移植できます。この操作の主な目的は、データを格納、並べ替え、検索に便利な形式にすることです。これを行うために、さまざまなメッセージやキューからのデータが結合されることがよくあります。

ストレージ操作（ロード）は、保存、ソート、およびその後の情報検索を目的としています。情報の種類とその使用オプションに応じて、さまざまなツールが使用されます。データに厳密なスキーム（テーブル列）がない場合は、NoSQLデータベースに格納されます。ただし、データが固定スキーマによって体系化できる場合は、SQLデータベースタイプが使用されます。後者には、OLTP（オンライントランザクション処理）とOLAP（オンライン分析処理）という2つのタイプがあります。名前が示すように、最初のタイプはETLプロセス自体により適しています-新しい値をデータベースに書き込むのに対し、2番目のタイプはデータの検索と分析により便利です。したがって、OLTPデータベースにダウンロードした後、バックグラウンドでデータがOLAPにコピーされることがよくあります。データがデータベースに保存するのに不便または不可能な場合があります。たとえば、レコードとして、このデータはバケットに記録され、レコードのメタデータはデータベースに保存されます。ストレージコストを削減するために、古いデータはアーカイブまたは削除されます。そして、このレベルの最後のコンポーネントは、顧客への提示および分析サービスのための新しい保存データの可用性の内部通知です。

どこでIoTアーキテクチャソリューションの構築を開始しますか？この質問に対する答えへの単一のアプローチはありません。そしてここで私は個人的な意見を述べます：

  1. ゲートウェイから取得できるデータモデルを定義します。バックエンドに転送されました。
  2. データを収集できる周辺機器と、ゲートウェイから送信されたモデルに取り込むためにデータをどのように処理する必要があるかを確認します。
  3. 周辺機器の要件（距離、情報量、消費電力など）を確認します。
  4. 適切な周辺コンピューティングデバイス、センサーとの関連でのその場所、それらの作業のプロトコルを選択します。
  5. 以下を含むクラウドパーツのアーキテクチャを解決します。

    • 安全性
    • 負荷分散
    • クラウド非同期データ転送
    • 要素の保存、フォーム、およびデータのライフサイクル
    • システム情報転送のグラフを作成する
    • 分析モデル、AI / MLコンポーネントを構築する
    • 通知の種類と内容を開発する
    • サービスの冗長性と自動スケーラビリティを設定する
    • コストを見積もり、最適化する
  6. モバイルクライアントのUI / UXデザイン
  7. 周辺機器のデータ転送フィードバックを構築する
