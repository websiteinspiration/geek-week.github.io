REXX言語、40周年

REXXは、当時としてはかなり古く、高度な言語です。ウィキペディアによれば、1979年に彼は現れた、つまり彼は最近40歳になった。もちろん、昔ながらの言語ではありませんが、年齢はまともです。

Rexxは略語であり、Restructured Extended Executorを意味します。これはおそらく祖先のEXEC / EXEC2を指します。私はVM / SP 3オペレーティングシステムで彼に会いました。そこで彼はこれらのスクリプト言語を正確に置き換えるようになりました。その後、それはOS / 2で人気があり、他の多くのOSで使用され、多くの実装と派生方言が作成されました。


彼はハブレにあまり注意を払わなかったので、この記事のギャップを埋めようとします。

祖先、起源、作者


元の言語の実装には、1人の著者、IBM Mike Cowlishawの従業員がいました。この実装は、S / 360アセンブラーで作成されました。

ウィキペディアは、REXXがPL / 1、EXECおよびEXEC2、およびAlgolの影響を受けたと考えています。一般に、はい、この言語は間違いなくその構文にあります-Algol（または、Pascalの場合）の相続人。つまり、ループ、ブロックなどです。構文は中括弧ではなく、キーワードで囲まれています。たとえば、do / endです。EXECとEXEC2に関しては、REXXはそれらの代わりとして作成されましたが、おそらくそれらとは関係がありません。しかし、PL / 1と比較すると、REXXははるかに単純であり、言語の目的は完全に異なります。

REXXコードは次のようになります。

/*    1  10 */
sum = 0
do count = 1 to 10
    say count
    sum = sum + count
end
say "    " sum"."

特徴


私の意見では、この言語で最も興味深いまたは珍しいものは何ですか？

1.言語の構文はかなり自由で、もちろんFortranではありません。 AlgolとPascalが最もよく見えます。基本的にも一般的です。

構造プログラミングに典型的な演算子があります。

機能と手順があります。ここで、プロシージャがデフォルトですべての変数を呼び出しコードから非表示にすることは少し珍しいですが、それらの一部は、公開<変数リスト>キー構成を使用して利用可能にすることができます。

タイピングは動的です。変数は、実行のさまざまな時点で文字列または数値として解釈され、定義されない場合があります（曲がっていても確認できます）。実際、言語には1つのプリミティブデータ型があります。文字列が正しい数であれば、算術演算を実行できる文字列です。

Wikipediaの小さな例を次に示します。

say hello /* => HELLO */
hello = 25
say hello /* => 25 */
hello = "say 5 + 3"
say hello /* => say 5 + 3 */
interpret hello /* => 8 */
drop hello
say hello /* => HELLO */

これは、最初の（および最後の）行の未定義の変数がその名前で大文字で計算されていることを示しています。これは不確実性をチェックする方法です。

解釈ステートメントは、今日、通常evalと呼ばれるものに類似しています。

エラー処理はおそらくBasicに最も似ています。

signal on halt
do forever
... /*   */
end
halt: /* .    halt,    */
exit

プログラム終了処理（端末からのボタンによる）です。停止に加えて、たとえば構文（解釈されたコードの構文エラー）などの他の「シグナル」があります。実際に適用したことは覚えていませんが、たとえば、解釈入力に何が入力されているかなど、構文の正しさをチェックすることは非常に論理的です。

2. Perlで言うように、正規表現はありませんでした（ただし、Perlの前にほぼ10年が残されました）。正規表現の概念そのものは以前に登場しましたが、それらは確かにすでに、たとえば、grepユーティリティがすでに存在していたものです。代わりに、パース演算子がありました。これは、グループを変数に置き換えた、正規化のいくつかの単純化された類似物です。

parse [origin] [template]

それら。ソースの内容（関数、変数、式などの引数である可能性があります）を分析して、テンプレートに準拠しているかどうかを確認します。テンプレートには、値を区切るリテラルと、これらの区切り記号の間のソースから値を受け取った変数を含めることができます。ある意味では、正規表現ではグループのように見えます。

fio = ', '
parse var fio firstName ',' LastName
say firstName
say LastName

もちろん、これは正規表現よりもはるかに単純です。それは、\ dのようなメタ文字がなく、*または+の形式の繰り返しがないため、代替手段がないためです。

3. Rexxには、OS（CMS）だけでなく、このAPIをサポートする他のプログラムのスクリプト言語としても使用できるAPIがありました。それら。実際、それはDSLライティング言語でした。おそらく最初の一つでしょう。同時に、ランタイムは、結果としてプログラムで使用できる追加の機能またはコマンドをインタープリターに実装して提供できます。

XEDITテキストエディタを起動すると、アクセス可能なディスクのどこかで、REXXスクリプトを含むPROFILE XEDITというファイルを探しました。このスクリプトは、ファンクションキーを割り当て、XEDITから使用できる追加のコマンドを定義できます。つまり、今日では、たとえば.bashrcに最も近くなっています。またはそのvim用のアナログ。

したがって、PROFILE XEDITで定義されたREXXプロシージャおよび関数は、エディタコマンドを実行し、ファイル自体の内容を含む、開いているファイルに関する情報にアクセスできます。

4.コマンドをホスト環境（OS、またはテキストエディタなど）に送信するための特別な構文は必要ありませんでした。代わりに、Rexxが認識しなかったすべての式が実行に渡されました。したがって、スクリプト内にOSコマンドを書き込むには、それを引用符で囲むだけで十分です。次に例を示します。

'list * * * (stack'

コマンドは、文字列定数と変数を連結することによって作成できます。連結演算のシンボルは単にスペースでした。

コマンドを別のアプリケーションに送信するために、アドレスステートメントがありました。たとえば、環境でXEDITテキストエディターを使用している場合、XEDITではなくOS（CMS）のコマンドを実行するには、次のようにする必要がありました。

address cms
'list * * * (stack'

5. CMSは、Rexxがデータの交換にも使用したOSコマンドスタックにアクセスするためのAPIを実装しました。上記のコマンドは、受信したファイルのリストをスタックに返し、そこからスクリプトがファイルを取得し、1行を標準入力として読み取りました。いくつかの点で、これはUnixパイプに似ていますが、それとは少し異なります。パイプは依然としてバイトのストリームであり、CMSのスタックは行のコレクションです。一般に、これは、UNIXのイデオロギーとIBMのOSの典型的な違いです。ファイルの大部分は、固定または可変の文字列（同時に既知の長さ）に向けられていました。

6. APIの一部として、アプリケーションはRexxスクリプト変数にアクセスでき、それらを読み取って変更できます。これは、データを交換するもう1つの方法でした。

7.任意精度のかなり効果的な演算が実装されました。任意-これは、たとえば100文字以上を意味します-十分なメモリ容量。

8.メモリ管理-自動。たとえば、変数（連想配列またはその要素）の値を削除するためのドロップ演算子がありましたが、いずれの場合も、ポインターとブロックサイズを扱う必要がありませんでした。

9.唯一の、しかし同時に非常に普遍的なデータ構造には、たとえばJavaScriptによく似た連想配列がありました。つまり、a.1を書き込んだ場合、これはキー1の配列aの要素です。abを書き込んだ場合、これはキーbの要素です。

辞書に単語を追加する手順を考えてみましょう：

add_word: procedure expose dictionary.
   parse arg w
   n = dictionary.0 + 1
   dictionary.n = w
   dictionary.0 = n
return

このフラグメントの辞書変数は、連想配列です。プロシージャはこの配列を返しませんが、外部からアクセスできるようにします（exposeを使用）。dictionary.0は、ディクショナリ内の要素の数を格納します（これは単なる合意であり、ご覧のとおり、私たち自身で実行します）、dictionary.nはディクショナリのn番目の単語です。

10.言語は、メモリまたはOC APIへの低レベルのアクセス権を持っていませんでしたが、原則として、同様の何かを書くことはかなり可能でした。言語拡張機能の例をいくつか挙げます。1つはSQL / DSデータベース（DB2）で動作するように設計され、もう1つはVM / SP内の仮想マシン間でメッセージを交換するためのものです。つまり、原則として、DSLとしてアレンジできるすべてのものを実装できます。

アプリケーションでのREXXサポート


CMS自体に加えて、REXXはメインのXEDITテキストエディターをすぐにサポートしました。これにより、REXXでマクロを記述したり、ファンクションキーでハングアップしたり、コマンドラインから、またはいわゆるコマンドを呼び出したりすることができます。テキストの行の横に入力され、操作できるようにする「接頭辞」コマンド。たとえば、dコマンドを使用すると、入力した次の行を削除でき、copyコマンドやmoveコマンドなどがありました。 REXXのマクロは、例えば、ストリングを大文字または小文字に変換できます。

私の記憶にある言語をサポートする2番目のアプリケーションはDMS（CMS（DMS / CMS）用のディスプレイ管理システム）と呼ばれ、パネルで構成される対話型プログラムを開発するためのツールでした。 REXXから、事前に描画されたフォームの1つを選択し、そのフィールドに値を入力して、画面に表示することができました。ユーザーがファンクションキーを押した後、プログラムは制御を取り戻し、REXX変数には変更されたフィールドの値が含まれていました。私がHTMLとそのフォームと呼ぶDMSの最も近い機能的な類似物。そして、開発の複雑さ（単純さ）の点では、HTMLは確かに表現力の点で勝っていますが（たとえば、テキスト端末で理解できるDMSには画像がまったくありませんでした）、これはおそらく少し簡単だったでしょう。

言及する価値のある別のアプリケーションは、ISPFと呼ばれていました。しかし、それはすでにDMSよりはるかに広く、今のようにテキスト端末用に調整された統合開発環境に似ていました。そして、はい、テキストエディターもREXXでプログラムされました。

VM / SP 6以降、CMSは、端末画面に可変サイズの仮想ウィンドウを作成し、それらにテキストを表示するためのツールを導入しました。対応するAPIがREXX用に登場しました（より正確には、主にREXXの拡張でした）。

REXXのもう1つの非常に興味深いアプリケーションは、いわゆる自動オペレーターでした。これは、OSオペレーターの仮想マシンで実行されるアプリケーションであり、通常はオペレーターの端末に届くすべてのメッセージを受信し、それらをソフトウェアで処理できるようにしました。コマンドハンドラ-REXXスクリプト。したがって、たとえば、一部のオペレーター機能を多くの人々に提供することが可能であり、これらの人々は単にオペレーターの仮想マシンにメッセージを送信し、ハンドラー（特権ユーザーに代わってオペレーター）がいくつかのコマンドを実行しました。たとえば、システムにディスクまたはテープボリュームをマウントしました。

しかし、可能性はそのような単純なものに限定されませんでした。たとえば、システムにない場合、ユーザーにメッセージを送信できません。それを自動オペレーターに送信し、キューに保存してから、（ユーザーがログインしているというメッセージを受け取った後）必要な人に送信します。実際、これは小さなREXXスクリプトを使用した単純な電子メール実装です。

開発への影響


VM / SP 3でのREXXの登場により、以前はPL / 1などで記述されていたものの重要な部分を記述し始めたと思います。これは非常に強力であり、同時に比較的大きなアプリケーションを適切に構造化できる単純な言語です。まず第一に、おそらくそれがDSLとして使用されたときの統合機能によるものです。

データベースクエリ（SQL / DS）の実行、画面フォームの表示、ファイルの読み取りと書き込みの機能は、ビジネスアプリケーション開発のニーズの重要な部分をカバーしています。

明らかに、多くの人々が言語を好んだ。彼はOS / 2、網上、Windows、その他多くの場所にいました。その後、オブジェクト拡張機能付きのオブジェクトREXXと、JVM用のNetREXXが登場しました。主なことは、この言語がまだ生きているということです。そして、一般的に、今日スクリプトを書くものを選択する必要がある場合は、たとえばRexx、bash、完全に鳥の言語であるcmd.exeを検討します-私は間違いなく最初のものを選択します。しかし、Perl、Python、その他の多くの新しい言語と比較すると、すべてがそれほど明白ではありません。たとえば、コマンドと変数をランタイムに転送するというRexxのアイデアは良かったですが、オブジェクト指向のCOMなどのアイデアはさらに機能的です。

過去40年間にもかかわらず、IBM z OSで使用されており、Wikipediaによると、ごく最近、2012年頃に言語の本がリリースされました。さまざまなOSのライブ実装をダウンロードして、自分で試すことはかなり可能です。
