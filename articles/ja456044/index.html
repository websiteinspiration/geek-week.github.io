「開発者のためのKubernetes：DevOpsには何人の開発者がいますか？」


Grigory Nikonov、Wunderman Thompson、マネージングディレクター

500ノードのクラスターはありません。厳しいDevOpsはありません。専任の製品チームはありません。しかし、私たちは多くの興味深いプロジェクトを抱えており、これらのプロジェクトの開発とサポート中に見つけた質問への回答があります。まず第一に、私たちは開発者であり、後で使用するツールを自分で作成することに慣れています。多分それらはあなたの仕事であなたを助けるでしょう。

エージェンシーワンダーマントンプソンは、ロシアでのインターネットソリューション開発のパイオニアの1人であり、現在、単純なランディングページと複雑な分散システムの両方を開発しています。 Kubernetesは、開発へのアプローチを最適化し、代理店のお客様向けに、作成されたソリューションのホスティングと運用を支援します。

多数の統合と内部コンポーネントを備えた分散システムでは、マイクロサービスアーキテクチャはソリューションの更新と保守の要件への自然な対応ですが、このようなアーキテクチャへの移行は、バージョン管理と公開に関連する一連の問題を引き起こします。私たちは代理店であり、専任の製品チームではなく、開発者は特定のソリューションの詳細なコンテキストを常にマシンに維持しているわけではなく、開発環境の再現性、複数のチームに同時に変更を加え、しばらくしてからプロジェクトに戻る能力に要件を課します。 。これらの課題に対する答えは、私たちが開発し、開発者とDevOpsがソリューションの開発と保守を容易にするプロセスとツールです。

DevOpsがOpsよりもDevである理由と、開発/サポートの時間とコストを削減するのにどれほど時間がかかるのかがわかります。

  • Kubernetesがプロジェクト開発へのアプローチをどのように変更したか。
  • コードのライフサイクルはどのように見えるか。
  • マイクロサービスの管理された公開に使用するツール。
  • 古いアーティファクトを組み立てる問題をどのように解決するか;
  • 喜んでクラスターにデプロイする方法。

「Kubernetes Cluster Autoscalerを使用したアプリケーションのスケーリング：オートスケーラーのニュアンスとMail.ruクラウドソリューションの実装」


Alexander Chadin、Mail.ruクラウドソリューション、PaaSサービス開発者

今日の世界では、ユーザーはアプリケーションが常にオンラインであり、いつでも利用できることを期待しています。つまり、どんなに大きなトラフィックストリームにも耐えることができます。 Kubernetesは、負荷に応じて自分自身をスケーリングできる、かなり洗練されたソリューションを提供します-Kubernetes Cluster Autoscaler。

一般に、Kubernetesには、スケーラブルなものという点で2種類のスケーリングがあります。アプリケーションのより多くのコピーまたはより多くのリソースです。既存のノード内のアプリケーションレプリカの数を増やす場合の垂直スケーリング。そして、より複雑な水平スケーリング-ノード自体の数を増やします。

2番目のケースでは、アプリケーションのコピーをさらに増やすことができます。これにより、アプリケーションの高可用性が保証されます。Cluster Autoscalerを使用した水平スケーリングについて説明します。負荷に応じて、ノードの数を増やすだけでなく減らすこともできます。たとえば、負荷のピークが過ぎると、オートスケーラー自体がノードの数を必要な数に減らし、プロバイダーのリソースの料金を減らします。

会議では、Kubernetesクラスターオートスケーラーのニュアンス、およびクラスターオートスケーラーの実装をMail.ru Cloud Containersサービスの一部として起動するときに発生した問題について詳しく説明します。あなたは学びます：

  • Kubernetesにはどのスケーラーがあり、それらの使用の特殊性は何ですか。
  • スケーラーを使用するときに注意する必要があるもの;
  • Node Groups;
  • Kubernetes Cluster Autoscaler MCS.

«R&D : K8S OpenStack»


Maxim Kletskin、Gazprombank、製品マネージャー

すべてのサービスとしてトレンドが設定されている世界では、市場投入期間は何よりも重要です。仮説をテストし、最初の形成時に新しい市場を見つけるためには、アプリケーションを迅速に開発する必要があります。銀行にとってスピードは特に重要であり、新しいテクノロジー、特にコンテナー化テクノロジーとKubernetesがここで役立ちます。

Maxim KletskinはGazprombankの製品マネージャーであり、パイロット製品をリリースするためのサンドボックスを開発しています。 GazprombankのR＆Dは、OpenStackであるクラウドでさまざまな研究を行っています。 Kubernetesは2つの方法で使用されます。1）OpenStackクラウドの管理レイヤーとしてのベアメタル上のKubernetes、および2）開発用のOpenShiftディストリビューションとしてのK8S。

レポートでは、最初のケースについて説明し、GazprombankがKubernetesを使用してOpenStackを管理する方法について説明します。OpenStackアーキテクチャーを見ると、それがかなりアトミックであることがわかります。そのため、OpenStackコントロールレイヤーとしてKubernetesを使用することは、非常に興味深く、論理的であるように見えます。さらに、これによりOpenStackクラスターへのノードの追加が容易になり、コントロールプレーンの信頼性が向上します。そして、ケーキのチェリーのように、クラスターからのテレメトリーの収集を簡素化します。

あなたは学びます：

  • なぜ銀行への研究開発：テストと実験;
  • OpenStackのコンテナ化方法。
  • OpenStackをK8Sにデプロイする方法と理由。
