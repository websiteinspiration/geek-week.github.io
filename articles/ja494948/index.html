Zimbra Collaboration Open-Source EditionでVIPメールボックスを作成する

企業では、いわゆるVIPメールボックスの作成が必要な場合に状況がしばしば発生し、特定の人々の輪だけがそれに手紙を送ることができます。このようなメールボックスは、個人秘書を持つトップマネージャには一般的です。トップマネージャーの代わりに、すべての電子メールが秘書によって受信され、秘書は一般的なストリームから本当に重要で必要なメッセージを選択して、マネージャーのメールボックスにリダイレクトします。企業の通常の従業員は、VIPメールボックスにメッセージを送信しようとすると、メッセージが配信されなかったという通知を受け取る必要があります。このようなVIPメールボックスは、Amavisおよびcbpolicydと呼ばれる組み込みのメールフィルタリングツールを使用して、Zimbra Open-Source Editionに簡単に実装できます。


Amavisは、Postfixメール転送エージェント、Spam Assassinスパムフィルター、ClamAVアンチウイルスなどのアクションを調整します。これらのユーティリティはすべて、スパムやウイルス、および企業の情報セキュリティに深刻な脅威をもたらすフィッシングメールに対する信頼性の高い保護を提供します。デフォルトでは、Amavisは例外なくすべてのメールをフィルタリングしますが、組み込みのポリシー制限機能を使用すると、メールボックスに対してこのフィルタリングを無効にするだけでなく、メールボックスにメールを送信できるユーザーのリストを原則的に制限できます。

たとえば、直接メッセージを送信できる人の輪を制限する必要があるCEO ceo@example.ruのメールボックスを取り上げます。リストは、CTO、CIO、CFO、および個人秘書の形で、会社のトップマネジメントに限定されます。このため、アマビスのポリシー制限を利用します。コマンドzmprov ma ceo@example.ru + amavisWhitelistSender secretary@example.ru cto@example.ru cio@example.ru cfo@example.ruを使用して、CEOに直接メールを書き込むことができるアドレスをホワイトリストに追加します。コマンドzmprov ma ceo@example.ru amavisBlacklistSender example.ruを使用して、残りをブラックリストに追加します。

すべてが正しく行われていることを確認するために、コマンドを実行できますzmprov -l ga ceo@example.ru | grep amavis | grep送信者。チームの出力には、CEOへのメッセージの送信が許可されている人と、彼のメールアドレスへのメッセージの送信が禁止されている人の両方が表示されます。ただし、企業のドメインのユーザーに加えて、CEOのメールボックスへの手紙は、外部インターネット上のドメインからも送信されます。したがって、このメールボックスの外部インターネットからの電子メールの受信を無条件に禁止することは、合理的なステップです。

これらの制限は、Zimbra Collaboration Suite Open-Source Editionに組み込まれているポリシーサービスであるcbpolicydを使用して導入できます。これを使用するには、まずZimbra OSEで対応するモジュールをアクティブ化する必要があります。これを行うには、次のコマンドを入力しますzmprov ms mail.example.ru + zimbraServiceEnabled cbpolicyd。 Zimbra OSEがマルチサーバーインフラストラクチャで実行されている場合は、MTAホストでこのコマンドを実行します。また、Webインターフェースからcbpolicydへのアクセスを開く必要があります。これを行うには、コマンドln -s / opt / zimbra / data / cbpolicyd / share / webui / opt / zimbra / httpd / htdocs /を入力します。これにより、Apacheサーバーにシンボリックリンクが作成されます。さらに、接続するデータベースに関する情報をファイル/opt/zimbra/cbpolicyd/share/webui/includes/config.phpに追加する必要があります。これを行うには、任意のテキストエディターで、行$ DB_DSN = "sqlite：/opt/zimbra/data/cbpolicyd/db/cbpolicyd.sqlitedb"をこのファイルに追加します。;。その後は、zmcontrol restartコマンドとzmapachectl restartコマンドを使用して、Zimbra OSEとApache Webサーバーを再起動するだけです。 現在、cbpolicydを管理するためのWebインターフェースはmail.example.ru：7780 / webui / index.phpで利用できますが、ファイアウォールの設定により利用できない場合があることに注意してください。 cbpolicyd Webインターフェースにアクセスするように構成します。 ：あなたはcbpolicyd Webインターフェイスを入力した後、次の2つのグループを作成する必要がusers_local_onlyのみローカルドメインのユーザーからのメールを受信できるユーザーのためだけでなく、LOCAL_DOMAIN

ローカルドメインの場合、[無効]をオフにして有効にします。その後、参加者をグループに追加する必要があります。私たちのケースでは、我々はにceo@example.ruが追加されますusers_local_onlyグループにしてexample.ruドメインLOCAL_DOMAINのグループ。





次に、ポリシー設定について説明します。ここでは「Receiving Local Only」と呼び、配布先のユーザーグループと、ローカルと見なされるドメインを示します。user_local_onlyおよびlocal_domainグループは、それらとして指定する必要があります、およびポリシーの適用の優先順位も示します。これは、複数のポリシーが同じメールボックスに同時に適用される場合に必要です。優先順位に基づいて、cbpolicydは適用されるポリシーと無視されるポリシーを決定します。この場合、優先度を100に設定できます。将来、重複するポリシーが表示された場合は、優先度を調整できます。



もう1つの重要なポイントは、ポリシーがトリガーされたときのアクションの構成です。つまり、CEOのメールボックスが外部ドメインからの手紙を受け取ることを禁じている場合、彼のメールボックスに送信され続ける手紙に何が起こるかを決定する必要があります。REJECTオプションを使用して、このようなメールを単に拒否することをお勧めします。変更後は、ポリシーを適用し、コマンドzmprov ms mail.example.ru zimbraCBPolicydAccessControlEnabled TRUEおよびzmcbpolicydctl restartを使用してcbpolicydを再起動するだけです。

したがって、Zimbra Collaboration Suite Open-Source Editionの組み込みツールを使用して、VIPメールボックスを作成しました。このテンプレートを使用すると、VIPメールボックスの数や、メールを送信できるアカウントの数を増やすことができます。
