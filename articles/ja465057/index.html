LinuxおよびWindowsではWindowsコンテナを実行できません*


コンテナー技術は、再構成されたアプリケーションイメージにより、環境の取り扱いを容易にします。これはApple AppstoreやGoogle Playに似ていますが、エンジニアと開発者専用です。モバイルアプリケーションのストアと同様に、iOSのGoogle Playからアプリケーションを配置することはできません。したがって、Linuxオペレーティングシステムを使用するDockerホストでは、Windowsオペレーティングシステムを使用してコンテナを起動することはできません。 Windowsを備えたDockerホストがLinuxにコンテナーを実行するための環境を提供できるため、その逆も当てはまります。

また、オペレーティングシステムのバージョンに互換性があることを確認しないと、Windows環境でWindowsコンテナを起動できません。 Microsoftのコンテナを使用する場合は、Windowsコンテナのバージョンの互換性を振り返る必要がありますこのドキュメントを定期的に開きます。

バージョニングと言えば-コンテナの出現により、MicrosoftはWindowsの半年ごとの新しい半年ごとのバージョンをリリースすることを決定しました。これらは、windows server 1703、1709、1803、1809、1903などのバージョンです。数字はリリースの年と月を示し、18か月間サポートされます。最初の2つはすでに平和で休んでいて、サービスの終わりにあります。さらに、Windows Server 2016やWindows Server 2019などのLTSバージョンもあります。バージョンリスト。

したがって、Windows 1803のバージョンを備えたホストでコンテナをアセンブルした場合、このコンテナはWindows 1803を備えたホストでのみ実行できます。したがって、毎回コンテナ自体を再構築しないようにするには、最新のテクノロジー開発速度を持つLTSバージョンのWindowsを使用する必要があります。常に正当化されるわけではありません。または、それでも、バージョン管理について考え、半年ごとの段階的なプログラムに従って、常にコンテナーを常に再構築します。

Windowsコンテナー用Dokerfileの最新のタグは常に存在するわけではなく、一般的に非推奨です。良い方法では、Windowsのバージョンを常に把握し、Dockerfileに適切な変更を加える必要があります。

上記のすべては、プロセス分離モードで実行されているコンテナーに当てはまります。 Hyper-V分離モードでは、下位互換性が有効です。現在のバージョンと以前のバージョンで構築されたすべてのコンテナーを実行できます。一般に、Hyper-V分離を使用すると、WindowsホストでLinuxコンテナーを実行することもできます。ただし、これまでのところ、このモードでサポートできる機能は少なく、Kubernetesが不足しているだけです。

プロセス分離とHyper-V分離の違いは、別の記事のトピックです。今のところ、Hyper-V分離のシナリオは私には完全には明らかではなく、デフォルトでWindowsはプロセス分離を使用していると言います。

もう1つの問題は、Docker Hubで正しいバージョンのイメージを見つけることです。一部の画像は、Windowsでは一般的に欠落しています。たとえば、Nginx、MySQL、Nodejs、その他数百のWindowsアプリケーションの公式ビルドは見つからないため、コンテナを自分で収集するか、コミュニティメンバーが収集して提供したコンテナを使用する必要があります。

Windowsコンテナは軽量ではありません。

コンテナは軽量であることは一般的に認められていますが、Linuxの真実はWindowsの真実であるとは限りません。 Windowsコンテナーの大部分は、一見すると非常に重くなっています。そして一見すると印象は変わりません。たとえば、基本的なaspnet：4.8イメージの重量は約7.5 GBです。

管理するには、CMDとPowershellをよく理解している必要があります。

ほとんどの場合、DockerホストでWindows Serverのコアバージョンを使用する必要があります。 Windows Serverには、豊富なリモート管理機能があります。一般的なアプローチは、グラフィカルインターフェイスを備えたWindowsサーバーを使用して、すべてのグラフィックスナップインを任意のコアサーバーに接続できることです。

このアプローチはコンテナーのシナリオでは機能しませんが、コンテナーにはWindows Serverのフルバージョンが含まれています。 Windowsコンテナ内では、理論的にはWMIを介して接続することが可能ですが、ホストOSがこのトラフィックを傍受するだけであるため、これはそれほど単純ではありません。ホストには数十から数百のコンテナが存在する可能性があり、この場合、トラフィックを目的のコンテナに転送するだけで十分です。

CMDとPowershellは、コンテナーの管理と、Dockerがインストールされているホストの両方に必要です。また、すべてのRUN命令は前述のシェルで実行されるため、Dockerfileを作成するにはこれらのシェルの知識が必要です。

長いPowershellコマンドレットをすべて覚えるのはかなり難しいです。これらは、簡潔なbashコマンドではありません。現在、ほとんどのコマンドレットには、どのLinuxエンジニアにとっても明確なエイリアスがあります。 powershellでは、以下を使用できます。

紹介記事の結論として、オーケストレーションとクラスター管理の問題には意図的には触れなかったと言いたいです。 Windows上のDockerは追いついており、Windows上のSwarmやKubernetesなどのオーケストレーションアプリケーションはそれらの機能を完全には実現していません。

また、現時点では、Dockerクラスターを上げたい場合は、おそらくマルチプラットフォームです。つまり、クラスター内に1つ以上のLinuxホストが必要です。たとえば、Kubernetesの場合、マスターノードはLinux上にある必要があります。また、Swarmでは、たとえばNginxにバランサーを実装したり、Linuxでのみ利用可能な他の一般的なクラスターアプリケーションを実行したりするために、Linuxコンテナーが必要です。
