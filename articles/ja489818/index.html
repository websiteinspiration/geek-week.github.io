（Service Workerではなく）Web Workerで無限同期ループを使用する


Webワーカーは基本的にWebブラウザーで実行されるスレッドであるため、メインスレッドをブロックするリスクなしに無限ループを実行できます。これにより、ミリ秒未満の期間で作業できます。この精度は、労働者が時間に大きく依存する決定を行う必要がある場合に特に適しています。これらの決定の採用は（非常に高いレベルの精度で）メインストリームに報告できます。たとえば、特定のイベントから経過したマイクロ秒数が素数で表される場合に何かを導出できます。時間の経過とともにマイクロ秒の精度で作業するために、performance.nowメソッドを使用できます。

WebアニメーションAPIの使用


WebアニメーションAPIでは、JavaScriptコードを使用してDOM要素をアニメーション化できます。

興味深いのは、マウントされていない要素をアニメーション化できることです！これにより、純粋なJavaScript（およびWeb API）で使用可能な時間メカニズムにアクセスできます。

ここに代替の実装がありますsetTimeout：

function ownSetTimeout(callback, duration) {
  const div = document.createElement('div');

  const keyframes = new KeyframeEffect(div, [], { duration, iterations: 1 });

  const animation = new Animation(
    keyframes,
    document.timeline
  );

  animation.play();

  animation.addEventListener('finish', () => {
    callback();
  });
}

きちんとね？

▍利点


  1. DOMとの対話を必要としない自己完結型のソリューション。
  2. このアプローチに不慣れなプログラマは、対応するコードの意味を簡単に理解できます。
  3. 「タイマー」は非アクティブなタブで一時停止します。

▍短所


  1. WebアニメーションAPI-このテクノロジーはまだ実験段階です。本番環境では使用しないでください。
  2. 非常に貧弱なブラウザのサポート。この方法は、Chromiumでのみ機能する可能性があります。
  3. このソリューションの利点はすべてありますが、それでも直感的とは言えない人に見えるかもしれません。
  4. 「タイマー」が非アクティブなタブで一時停止されているという事実は、代替手段として使用される場合、このソリューションのマイナスになる可能性がありますsetTimeout。
  5. この方法は、間隔のカウントには使用できません。プログラマが利用できるのはイベントのみonfinishです。
  6. 精度が低い。私の実験によれば、誤差は簡単に±5ミリ秒になる可能性があります。
