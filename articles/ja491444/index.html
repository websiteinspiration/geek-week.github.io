Azure DevOpsパイプラインでの変数の使用

私たちは、Azure DevOpsだけでなく、Windows向けの開発のための素晴らしいツールを引き続き検討しています。今回は、環境変数に悩まされ、すべての経験を1つの記事にまとめることにしました。

ランタイム環境ごとに構文が異なるという事実から始まり、パイプラインのあるステージから別のステージに変数を転送する標準的な機能の欠如で終わります。

YAMLはまだリリースパイプラインに達しておらず、多くのステージと多くのアーティファクトの機能が必要なので、主な例はリリースパイプラインにあることを予約します。これは、通常のパイプラインで使用できるようになり、機能的に実質的に平準化されました。パイプラインでは、YAMLがテキストビューに追加され、設定可能なパラメーターを備えた小さなグラフィックツールチップが追加されました。非常に便利で、各モジュールのドキュメントにアクセスする必要はありません。しかし、これについては次の記事で説明しますが、ここでは、イノベーション自体の写真を示します。



保管と使用


まず、システムにデフォルトの変数があります。それらは、起源に応じて、リリース、システムなどの単語で始まります。完全なリスト（結局のところ、ありません）は、ドキュメントにあります。構文のある統合失調症はすべて、以下のドキュメントの例で示されています。同じ変数には、呼び出し先に応じて3つの表現があります。

steps:
 - bash: echo This script could use $SYSTEM_ACCESSTOKEN
    env:
      SYSTEM_ACCESSTOKEN: $(System.AccessToken)
  - powershell: Write-Host "This is a script that could use $env:SYSTEM_ACCESSTOKEN"
    env:
      SYSTEM_ACCESSTOKEN: $(System.AccessToken)

タスクを実行しているエージェントに変数を設定した場合、これは$（System.AccessToken）です。同じエージェントのPowerShellスクリプト内で使用する場合は、すでに$ env：SYSTEM_ACCESSTOKENになります。禁じられていることですが、ターゲットマシンのPowerShellタスクを使用してリモートホストでこの変数を使用する場合は、paramを使用して、引数を介してスクリプトに渡す必要があります。単純なbashでは、引数と構文$ SYSTEM_ACCESSTOKENを使用して、それを投入することができます。

同じ法律はあなた自身の変数には適用されません、ここであなたはすでに構文に対して責任があります。各タスクでローカルに変数を設定できます。



または、変数のリポジトリでグローバルに、リポジトリからそれらをリンクします。とても快適。



これのボーナスとして、変数が非常に秘密である場合、それらはAzure Vaultと呼ばれるストレージのAzureクラウドに格納でき、Vaultはライブラリのプロジェクトにリンクできます。



一般的に、変数はすべて明確です。パイプラインでは、起動ごとに手動で変数を設定できますが、リリースではそのような機能はありません。パイプラインに何を転送するかはエージェント初期化ログで確認できますが、それらはすでに変換された形式になっていることに注意してください。



動的変数


最も興味深い部分は、ある段階で何らかの価値を得て次の段階に渡したいときに始まります。



このような機能は提供されていません。しかし、私たちの手は退屈ではなく、Googleの助けを借りて解決策が見つかりました。感謝します。AzureDevOpsには、インターフェイスで描いた以上のことを実行できるAPIがあります。

したがって、グローバル変数を更新するための呼び出しが必要です。これは、パイプラインの内部から直接実行します。アドレスは、環境変数から取得されます。前述のように、ドキュメントには単語がありません。あなたは彼らにあなた自身に尋ねるか、彼らが店を閉めるかどうか、そこにあるものをハードコードすることができます。

$releaseurl = ('{0}{1}/_apis/release/releases/{2}?api-version=5.0' -f $($env:SYSTEM_TEAMFOUNDATIONSERVERURI), $($env:SYSTEM_TEAMPROJECTID), $($env:RELEASE_RELEASEID)  )

渡す変数の空の値を設定し、Scope-Releaseを設定します。



たとえば、値のランダムジェネレーターを作成します。このステージ内の変数宣言の構文に注意してください。そのような機能が導入されました。



次のステップでは、変数をスクリプトに渡します。はい、はい、直接は不可能です。引数を通じて必要です。



スポイラーの下のスクリプト

Powerhell
#Script requires stageVar variable in release variables set to Release scope

param ( [string] $expVar )
#region variables
$ReleaseVariableName = 'StageVar'
$releaseurl = ('{0}{1}/_apis/release/releases/{2}?api-version=5.0' -f $($env:SYSTEM_TEAMFOUNDATIONSERVERURI), $($env:SYSTEM_TEAMPROJECTID), $($env:RELEASE_RELEASEID)  )
#endregion


#region Get Release Definition
Write-Host "URL: $releaseurl"
$Release = Invoke-RestMethod -Uri $releaseurl -Headers @{
    Authorization = "Bearer $env:SYSTEM_ACCESSTOKEN"
}
#endregion

#region Output current Release Pipeline
Write-Output ('Release Pipeline variables output: {0}' -f $($Release.variables | ConvertTo-Json -Depth 10))
#endregion


#region Update StageVar with new value
$release.variables.($ReleaseVariableName).value = "$expVar"
#endregion

#region update release pipeline
Write-Output ('Updating Release Definition')
$json = @($release) | ConvertTo-Json -Depth 99
Invoke-RestMethod -Uri $releaseurl -Method Put -Body $json -ContentType "application/json" -Headers @{Authorization = "Bearer $env:SYSTEM_ACCESSTOKEN" }
#endregion

#region Get updated Release Definition
Write-Output ('Get updated Release Definition')
Write-Host "URL: $releaseurl"
$Release = Invoke-RestMethod -Uri $releaseurl -Headers @{
    Authorization = "Bearer $env:SYSTEM_ACCESSTOKEN"
}
#endregion

#region Output Updated Release Pipeline
Write-Output ('Updated Release Pipeline variables output: {0}' -f $($Release.variables | ConvertTo-Json -Depth 10))
#endregion
