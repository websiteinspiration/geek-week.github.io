VPNキラー。バトルサーバーへの適切なリモートアクセス

この記事で述べられている意見は著者の個人的な意見です。彼はそれが彼の雇用主、上司、セキュリティ部門の意見と一致しないかもしれないことを強調します。

インフラストラクチャの点で当社の最も優れた点の1つは、リモートアクセスの実装方法です。これはまさに超巨大な魔法です。私は仲間の多くの警備員や監査人と話をしました、そして私達は偶然に商用ソリューションと私たちのソフトウェアを混合することによって全く新しい話を発明したようです。そのため、業界が古いリモートアクセスソリューションでどのように機能し、どのように実装するかについての技術的な詳細に深く掘り下げるのは興味深いことだと思いました。

良い人のVPNはアドバイスしません


まず、強力な声明から始めます。すべてのVPNはゴミです。

VPNは、他のデジタルテクノロジーと同様に、ハッキングされても世界が終わらないように構成できます。しかし、誰もこれをしません...しかし理論的にはできます！

99.95％の場合、VPNは次のように構成されます。

  1. デバイス（ラップトップまたは他のサーバー）間のネットワークブリッジの作成
  2. ...およびサーバーのより大きなネットワーク（たとえば、クラウド内またはオンプレミス用）
  3. ...および追加の暗号化レイヤーで保護されたインターネット



これは良い考えではありません。ウイルスがラップトップに定着し、VPNを介して戦闘サーバーのネットワークに接続している場合を想像してみてください。タダム！これで、ウイルスはプロダクションにLANアクセスできます。排気は何ですか？悲しみ。悲しみがいっぱい。

さて、ウイルスのケースははるかにフェッチされているとしましょう。しかし、ハッカーが、たとえばVPNが機能するデバイスまたはソフトウェア内の脆弱性を使用して、VPN自体に侵入できるという事実についてはどうでしょうか。それでは、ターゲットネットワークに直接進んだのですか？これは深刻な問題であり、理論化にはほど遠いものです。Heartbleedが攻撃方法を使用してVPNをキャプチャするためにどのように使用されたかについては、ブログで直接警告しました。

つい最近まで、VPNの脆弱性の波が世界中に蔓延し、標的のネットワークへのアクセスを得るためにサイバー犯罪者によってすぐに使用されました。しかし、これは驚くべきことではありませんよね？これらのシステムは、ネットワークの前に他の保護メカニズムがなくても、ネットワークを直接「調べます」。通常、修正とパッチは自動的には適用されず、独自仕様のOSで実行されている独自仕様のソフトウェアによって制御される独自仕様のメカニズムが含まれます。さて、幸運があります。

これらのVPNデバイスを見つけるのは難しいですか？この記事を書く前に、私は試したことがないので、確信が持てませんでした。Shodan.ioで約30分してください-出力からのいくつかの主要な結果：

  • Thomson Reutersは41億ドル規模の企業で、従業員数は26,000人で、その半分は金融サービスです。
  • SAP Concurは旅行管理サービスであり、それを壊すと、あらゆる種類の個人データと支払いデータが大量に提供されます。
  • Progressive Insurance-個人データと個人医療情報、およびいくつかの支払いデータ。
  • シェブロンフィリップスケミカル -その名前はそれ自体が物語っています。

まあ、これはおそらくあまり良くありません。これらのものを見つけるのが非常に簡単である場合、それらをオンラインにするのに最適なソリューションとは思えません。他に選択肢はありますか？

ゼロトラスト


ゼロトラスト[文字通り：「ゼロトラスト」]-本質的には、各接続を承認し、それらがすでにネットワーク内にあるために、それらの一部が信頼できると想定しないことを意味します。この用語をよりよく理解し、ビジネスへのアプローチを再考したい場合は、ネットワークワールドに関するこの記事を読んでください（別の恥知らずな自己宣伝をお詫びします）。戦闘サーバーにゼロトラスト認証を導入するために、Okta Advanced Server Access（OASA）ソリューション

を購入しました。OASAは 3つの理由でクールです。


1.これは、ステロイド上のOpenSSHの構成のラッパーにすぎません


内部的には、OASAプラットフォームはOpenSSH（コンソールの同じsshコマンド）の適切に管理されたデプロイメントです。また、OpenSSHは非常に十分にテストされた安全なリモート管理ソリューションです。ちなみに、2003年以降、OpenSSHには、標準構成での不正なリモートアクセスにつながる可能性がある単一の脆弱性がありません。
このスキームのエントリポイントは通常のAmazon Linux 2 EC2インスタンスであるため、攻撃範囲は非常に小さくなります。注意：VPNの使用に関する主な問題の1つは、自動パッチを妨げる独自のソフトウェア/ OS構成です。ネットワークエントリポイントを他のインフラストラクチャと一緒に更新できることは大きなメリットです。

2.ネットワークブリッジはありません


上で述べたように、ほとんどのVPNは、ラップトップなどのネットワークデバイスとインターネット上のサーバーのネットワークとの間にネットワークブリッジを作成するように構成されています。 VPNで私の最も苦痛な点の1つは、VPNがすべてのネットワークトラフィックを傍受することです。もちろん、別の動作に再構成することもできますが、クライアントとNIST 800-53 SC-7（7）セキュリティプロトコルでは通常、それだけで十分です。

これは、いくつかのセキュリティ対策が業界の発展をはるかに下回っている良い例です。昔ながらの世界では、VPNはトラフィックを暗号化するための唯一のレイヤーでした。監査人は、VPN保護がなければ、暗号化されていないチャネルを介してデータを送信すると信じていることがあります。

幸い、もっと良い方法があります。 OASAモデルでは、ユーザーとサーバーの間で接続が個別に確立されます。たとえば、「EC2インスタンスi-028d62efa6f0b36b5に移動したい」という要求に応じて、システムはネットワークの入力ポイントに接続してから、ターゲットサーバーに接続します。 OASAはまた、SSOによるデータの最初の検証後に10分のクライアント証明書を発行し、目的の（そして確認された）信頼できるデバイス上にいることを検証することによって、これらの接続を保護します。

ネットを歩き回る自由はほとんどありません。管理者はネットワークエントリポイントを入力してポートを別の宛先に転送できますが、この機能はデフォルトで無効になっており、接続を確立するときに明示的に要求する必要があります。そして、最良の部分は、「VPN」と呼ばれていないため、Virtual Private Cloudを介してすべてのトラフィックを伝送する必要がないということです。

3.専用ネットワークアクセスとランダムIP


これらのネットワークエントリポイントは、仮想プライベートクラウドごとに個別にデプロイされます（たとえば、1つは製品用、1つはステージング用、1つは開発用など）。さらに、すべてのアクティビティをログに記録してトラフィックをフィルタリングするセキュリティソリューションによって、それぞれが厳密に監視されます。クラッカーがエントリポイントの1つにある場合、何かもうまくいきません。どのような場合でも、すでにVirtual Private Cloud内にいるというだけの理由で、保護されたリソースへのアクセスは許可されません。

私のお気に入りの防御機構の1つが偶然現れました。最初にエントリポイントの構成を開始したとき、それぞれにAWSからの静的IPアドレスが構成されていました。すぐに、これらのIPアドレスが適切なタイミングでEC2インスタンスに関連付けられないことがあり、OASAの誤った自己設定につながる可能性があることがわかりました。本番環境でさまざまな修正を行って喜んだので、我慢できず、静的IPに関する話をすべて削除しただけで、うまくいきました。



OASAはインターネットに直接接続するIPを必要とするだけであり、事前に知る必要はありません。クライアントが接続を確立する準備ができると、一意のGUIDが要求され、その助けを借りてIPが決定されます。

  • ユーザー：ログインしてvpc-99f2acffに接続したい
  • OASAクライアントアプリケーション：vpc-99f2acffをGUID 25af5d4f-e657-4583-b0bd-beb1ca4f0c1fの既知のサーバーとして定義しました
  • OASAサーバー：K 25af5d4f-e657-4583-b0bd-beb1ca4f0c1fには3.22.198.24でアクセスできます。ここに証明書があります
  • OASAクライアントアプリケーション：証明書がインストールされ、SSH経由で3.22.198.24に接続しています...

これは、エントリポイントインフラストラクチャの各インスタンス（注意については別の投稿）に、まったく新しいIPアドレスのセットが付属していることを意味します。したがって、ランダムなハッカーは何千万もの中から1つのIPを探す必要があり、その数は毎日増加しています。残念なことに、この検索は...のおかげで完全に無駄になっています。

エンタープライズポートノッキング


ポートノッキング [文字通り：ポートをノックする]は、原則として現実の世界では誰も使用しないものですが、設定するのはとても楽しいものです。つまり、ポートノッキングはさまざまな閉じたネットワークポートへの「バンプ」のシーケンスであり、シーケンスが正しく繰り返されると、「実際の」ポートがIPに開きます。エレガントですが、実際の大規模なプロジェクトでは実用的ではありません。
私はアイデアに触発され、アイデアを改善する方法を考えました。私がエンタープライズポートノッキングと呼ぶソリューションが生まれました。

誰かがインターネットにアクセスできるようになるまで、インターネットへのエントリポイントの近さを保証するメカニズムを作成したかったのです。このメカニズムは、使いやすく、信頼性があり、既存の方法で認証されるはずでした。
私はメカニズムのアーキテクチャをスケッチし、彼女と一緒に私たちの信じられないほど優秀なエンジニアのチームに行きました。数週間後、私たちは始めました。

このサービスは非常にシンプルで、AWS Lambda関数としてデプロイされ、AWS API Gatewayを介してアクセスできます（サーバーレスアーキテクチャの喜びです）。簡単で信頼性の高い使用のために。このメカニズムは次のように機能します。

  1. ユーザーはSSOを介して正常に認証されます。
  2. アプリケーションは、特別にマークされたセキュリティグループ（ファイアウォールルールのAWSコンセプト）を検索して、構成されたAWSアカウントをバイパスします
  3. アプリケーションはセキュリティグループを更新し、要求者のIPアドレスへのアクセスを許可します。セキュリティグループのルールは、追加のタイムスタンプも保存します。
  4. cronタスクは、指定された期間ごとに許可されたIPのリストをクリアします。

このような状況のおかげで、インターネットに完全に閉じており、ファイアウォールでポートを開く前にユーザーデータベースによる2要素認証を必要とするリモートアクセスソリューションを自慢しています。

そして、それは簡単です！


これらのメカニズムの動作がいかに単純であるかについては、まだ触れていません。多くのコンポーネントがあることは知っていますが、一緒に使用すると非常に単純な承認が作成されます。

  1. SSOを介してログインします。
  2. SSOポータルでEnterprise Port Knockingの起動をクリックします。
  3. ターミナルで、SSHコマンドを使用して、ターゲットEC2インスタンスのIDとして宛先を指定します。OASAシステムは、どのエントリーポイントを使用するかを決定するのに十分なほどスマートで、他のすべては自動的に行われます！



このシステム全体は、インフラストラクチャ、すべての安全協定の遵守、およびお客様の安全にとって大きな勝利です。ユーザーは、追加の認証を行ったり、使用するVPNを覚えたりする必要がないため、サーバーへの接続がいかに簡単かを本当に気に入っています。そして、私は私の穏やかな睡眠が本当に好きです。皆新モデルで勝ちました！

もちろん、ハッカーを除いて。
