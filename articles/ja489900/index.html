PostgreSQLアンチパターン：トリガーをバイパスしてデータを変更する

遅かれ早かれ、多くの人がテーブルエントリの何かを大規模に修正する必要に直面しています。私はすでにそれをより良くする方法、そしてどのように-それをしない方が良いのかをすでに述べました。今日は、大量更新の2番目の側面であるトリガーのトリガーについて説明します。

たとえば、何かを修正する必要があるテーブルで、悪意のあるトリガーがハングしON UPDATE、すべての変更が一部の集計に転送されます。また、これらのユニットが影響を受けないように、すべてを更新する（新しいフィールドを初期化するなど）必要があります。

トリガーをオフにしましょう！


BEGIN;
  ALTER TABLE ... DISABLE TRIGGER ...;
  UPDATE ...; --  -
  ALTER TABLE ... ENABLE TRIGGER ...;
COMMIT;

実際には、これですべてです。すべてがすでに停止しています。

これALTER TABLEはAccessExclusiveロックを課すため、その下では、並列実行している誰も、単純なものSELECTでも、テーブルから何も読み取ることができません。つまり、このトランザクションが完了するまで、「ただ読む」ことを望むすべての人が待機します。そして、UPDATE私たちは私たちがo-o-olgi ...

すぐにオフにして、すぐにオンにしましょう！


BEGIN;
  ALTER TABLE ... DISABLE TRIGGER ...;
COMMIT;

UPDATE ...;

BEGIN;
  ALTER TABLE ... ENABLE TRIGGER ...;
COMMIT;

ここでは状況が良くなり、待機時間がはるかに短くなります。しかし、2つの問題だけがすべての美しさを台無しにします。

  • トリガーがオフになっている間は、テーブルの変更はによって飛んでいきます。まあ、それはユニットには入りませんが、入ります。トラブル！

セッション変数管理


したがって、以前のバージョンでは、重要なポイントに遭遇しました。テーブルの「私たちの」変更と「私たちの」ではない変更を区別するためのトリガーを何らかの方法で教える必要があります。「私たちのもの」をそのままスキップしますが、「私たちのものではありません」-トリガー。これにはセッション変数を使用できます。

session_replication_role


マニュアル を読む：
トリガー変数は、session_replication_role構成変数の影響も受けます。追加の指示なしで有効になっているトリガー（デフォルト）は、レプリケーションの役割が「元の」（デフォルト）または「ローカル」のときに起動します。インジケーションによって有効にされたトリガーENABLE REPLICAは、現在のセッションモードが「レプリカ」の場合にのみ起動し、インジケーションによって有効にされたトリガーENABLE ALWAYSは、現在のレプリケーションモードに関係なく起動します。
この設定は、のように一度にすべてに適用されるのではなくALTER TABLE、個別の特別な接続にのみ適用されることを強調します。アプリケーションが起動しないように合計：

SET session_replication_role = replica; --  
UPDATE ...;
SET session_replication_role = DEFAULT; --
