どこから始めれば？


さまざまな言語とフレームワークの既成の例がいっぱいです。これは、プロジェクトリポジトリにあるYAML構成ファイルを使用して構成されます。



Androidの最小限の例：

name: Android CI

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v1
      - name: set up JDK 1.8
        uses: actions/setup-java@v1
        with:
          java-version: 1.8
      - name: Build with Gradle
        run: ./gradlew assembleDebug

説明：プッシュするたびに、ubuntu OSを搭載したgithub仮想マシンの任意のブランチでタスクが起動されます。タスクの手順：コードをチェックアウトし、jdkを構成し、アセンブリのGradleタスクを実行します。

3. UIテスト

UIテストの作成は想像力の具現化です。私のアプローチは、アプリケーションでのユーザーの標準アクションを模倣する1つの「煙」テストです-ログイン、製品の選択、注文、注文の追跡。 UIAutomator、Espresso、Kaspressoを使用できます。

-ここに起動するための2つのオプションもありますgithubの仮想マシンまたはなどのクラウドサービス上のエミュレータ。Firebaseテストラボは
：githubの内部でエミュレータを使用するには、既製の実装がある1と2は。

Firebase Test Labの場合、gcloud CLIを介してGoogle Cloud Platformを操作する必要があります

- name: prepare gcloud
  uses: GoogleCloudPlatform/github-actions/setup-gcloud@master
  with:
    version: latest
    service_account_email:  ${{ secrets.SA_EMAIL }}
    service_account_key: ${{ secrets.GOOGLE_APPLICATION_CREDENTIALS }}
- name: gcloud Set up project
  run: |
    gcloud config set project ${{ secrets.PROJECT_ID }}
- name: Assemble apks for smoke test
  run: ./gradlew Smoke
- name: Run tests in test lab
  run: |
     gcloud firebase test android run \
       --app app/build/outputs/apk/production/debug/app.apk \
       --test app/build/outputs/apk/androidTest/production/debug/appTest.apk \
       --device model=Nexus6P,version=25,orientation=portrait,locale=en_US \
       --device model=athene,version=23,orientation=portrait,locale=en_US \
       --device model=sailfish,version=26,orientation=portrait,locale=en_US

これを機能させるには、Firebaseでプロジェクトを作成し、Google Cloud Consoleで管理者権限を持つサービスアカウントを作成し、gcloudでの承認のために受け取ったjsonキーをgithubシークレットの base64にアップロードする必要があります。

私の場合の一般的な設定は次のようになりました。タスクはマスターのPRイベントによってトリガーされます

name: Android CI

on:
  pull_request:
    branches:
      - 'master'

jobs:
  build:

    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v1
      - name: set up JDK 1.8
        uses: actions/setup-java@v1
        with:
          java-version: 1.8
      - name: Run static checks
        run: ./gradlew checkAll
      - name: Run some unit tests
        run: ./gradlew testStageDebugUnitTest

      - name: prepare gcloud
        uses: GoogleCloudPlatform/github-actions/setup-gcloud@master
        with:
          version: latest
          service_account_email:  ${{ secrets.SA_EMAIL }}
          service_account_key: ${{ secrets.GOOGLE_APPLICATION_CREDENTIALS }}
      - name: gcloud Set up project
        run: |
          gcloud config set project ${{ secrets.PROJECT_ID }}
      - name: Assemble apks for smoke test
        run: ./gradlew Smoke
      - name: Run tests in test lab
        run: |
          gcloud firebase test android run \
            --app app/build/outputs/apk/production/debug/app.apk \
            --test app/build/outputs/apk/androidTest/production/debug/appTest.apk \
            --device model=Nexus6P,version=25,orientation=portrait,locale=en_US \
            --device model=athene,version=23,orientation=portrait,locale=en_US \
            --device model=sailfish,version=26,orientation=portrait,locale=en_US

簡単そうです。効率は、書かれたテストと選択されたコード分析ルールに依存します。複数の独立したタスク（ジョブ）を記述して、それらを並行して実行できます。私の場合、すべてが順番に行われます。検証プロセスは、私たちのプロジェクト（仮想マシンgithub 2コアCPU、7 GB RAM、14 GB SSD）で約15分かかりますが、目で「コーディング」する限り、これらのテストの結果も届きますが、それほど重要ではありません。

UIテストは最も役立ちます-通過中に、ライブラリを更新した後に分析がクラッシュし、ライブラリを更新してはいけないことを理解しています。
