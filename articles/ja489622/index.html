スマート環境ライブラリ


独自の発想を書く前に、インターネットをサーフィンして既製のソリューションを探すことは有用です。もちろん、必要なものが見つからないというリスクはありますが、むしろ「保険事件」です。原則として、このアプローチは機能し、多くの時間と労力を節約します。検索結果に

よると、次のことが判明しました。

  • os.environへの呼び出しを実際にラップするパッケージがありますが、同時に、一連の注意散漫なアクション（クラスのインスタンスの作成、呼び出しでの特別なパラメーターなど）が必要です。
  • ただし、特定のエコシステム（主にDjangoのようなWebフレームワーク）に密接に結び付いている優れたパッケージがあるため、ファイルがなければ普遍的ではありません。
  • 何か新しいことをしようとするまれな試みがあります。たとえば、入力を追加し、次の形式のメソッドを呼び出して変数値を明示的に解析します

    get_<typename>(var_name)

    または、ここで別の解決策を示しますが、今は不名誉なPython 2をサポートしていません（公式のRIPにもかかわらず、記述されたコードやエコシステム全体が山のように残っています）。
  • 学生向けの工芸品があります。なぜそれらが上流のPyPIになってしまい、新しいパッケージの命名で問題が発生するのかはまったく明らかではありません（特に、「smart-env」という名前は必要な手段です）。

そしてリストはどんどん続きます。しかし、上記の点は、何かを便利で普遍的なものにするという考えを捉えるのに十分でした。

smart-envの要件：

  • 最もシンプルな使用方法
  • 簡単に構成可能なデータ型サポート
  • Python 2.7互換
  • 良好なテストカバレッジ

結局、これはすべて実現されました。次に使用例を示します。

from smart_env import ENV

print(ENV.HOME)  # Equals print(os.environ['HOME'])

# assuming you set env variable MYVAR to "True"

ENV.enable_automatic_type_cast()

my_var = ENV.MY_VAR  # Equals boolean True

ENV.NEW_VAR = 100  # Sets a new environment variable

例からわかるように、新しいクラスを操作するには、それをインポートするだけで十分です（インスタンスを作成する必要はありません-追加のアクションは必要ありません）。任意の環境変数へのアクセスは、それをENVクラス変数として参照することによって実現されます。これにより、実際には、このクラスはネイティブシステム環境の直感的なラッパーになり、同時に、ほとんどすべてのシステムの構成オブジェクトの可能なバリアントに変わります（たとえば、Djangoでも同様のアプローチが実現されます） 、そこにのみ構成オブジェクトは直接モジュール/設定パッケージです）。

自動タイピングサポートモードの有効化/無効化は、enable_automatic_type_cast（）およびdisable_automatic_type_cast（）の2つのメソッドを使用して実現されます。これは、環境変数にシリアル化されたJSONのようなオブジェクトまたはブール定数のみが含まれている場合に便利です（環境変数を「有効な」文字列と比較して、Djangoで明示的にDEBUG変数を設定するのは、最も一般的なケースの1つです）。しかし、今は明示的に行を変換する必要はありません-必要なアクションのほとんどはすでにライブラリの腸に埋め込まれており、アクションのシグナルを待っているだけです。 :)一般に、タイピングは透過的に機能し、ほとんどすべての利用可能な組み込みデータ型をサポートします（フローズンセット、複素数、バイトはテストされていません）。

特に、ユビキタス6（メタクラスの使用に関する問題を解決するため）のおかげで、Python 2のサポート要件はほとんど犠牲を払わずに実装されました（タイピングの拒否と最新バージョンのPython 3からの砂糖菓子）。

ただし、いくつかの制限があります。

  • Python 3のサポートはバージョン3.5以降を意味します（プロジェクト内に存在するのは、怠惰または改善の必要性の欠如の結果です。3.4のままでいるという客観的な理由を考え出すことが難しいためです）。
  • Python 2.7では、ライブラリはセットリテラルの逆シリアル化をサポートしていません。説明はこちら。しかし、誰かが実装したい場合-ようこそ:);

ライブラリは、解析エラーの場合の例外メカニズムも提供します。使用可能なアナライザーのいずれかで文字列を認識できなかった場合、値は文字列のままです（便宜上、およびBashの変数の通常のロジックとの下位互換性のため）。

Pythonシェルライブラリ


今、私は2番目のライブラリについて説明します（私は可能な類似体の欠点の説明は省略します- 。それはスマート-ENV類似体のために説明したものと同様である- こことここ）。

一般に、実装の概念とその要件は、例からわかるように、smart-envについて説明したものと同様です。

from python_shell import Shell

Shell.ls('-l', '$HOME')  # Equals "ls -l $HOME"

command = Shell.whoami()  # Equals "whoami"
print(command.output)  # prints your current user name

print(command.command)  # prints "whoami"
print(command.return_code)  # prints "0"
print(command.arguments)  # prints ""

Shell.mkdir('-p', '/tmp/new_folder')  # makes a new folder

アイデアはこれです：

  1. Pythonの世界でBashを表す単一のクラス。
  2. 各Bashコマンドは、Shellクラスの関数として呼び出されます。
  3. 各関数の呼び出しパラメーターは、対応するBashコマンドの呼び出しに転送されます。
  4. 各コマンドは、その呼び出し時に「ここと今」実行されます。同期アプローチで機能します。
  5. stdoutの排気コマンドとその戻りコードにアクセスできます。
  6. コマンドがシステムにない場合、例外がスローされます。

smart-envと同様に、Python 2のサポートが提供されます（ただし、犠牲の血液がもう少しかかります）。Python3.0-3.4のサポートはありません。
