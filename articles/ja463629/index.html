Dockerを使用して同じサーバーでNextCloud + ONLYOFFICEを構成する

こんにちは、ハブル！「Dockerを使用する単一サーバーでのNextCloudとONLYOFFICEのセットアップ」という記事の翻訳を紹介します。

技術志向の人々の生活の中で、Googleドキュメントやクラウドストレージなどのオンラインオフィススイートパッケージの利点を過小評価することは困難です。テクノロジが非常に普及し、Officeアプリケーション市場を長く独占してきたMicrosoftでさえ、最近Office 365 Webアプリケーションの開発と、ユーザーが独自のサービスを使用するサブスクリプションモデルに切り替えるよう説得することに焦点を合わせています。自分のストレージをインストールして構成するプロセスに興味がある人は、猫に招待されます。

少し前に、私たちは零細企業で使用するために簡単に展開できるクラウドストレージソリューションとWebアプリケーションのオープンソースオフィススイートを検討しました。すべてのドキュメンテーションをオンラインで保存する主な動機は、紙のワークフローの量を最小限に抑え、操作の量が少ないにもかかわらず、積極的なビジネス慣行を実装することです。裏返して言うと、サーバーまたはトラフィックへの物理的なアクセスを監査する手段がないため、このサービスを提供するためにクラウドサーバーをレンタルすることは、施設に直接保管するよりも安全性が低いということです。したがって、エンドツーエンドの暗号化とオープンソースソフトウェアも必要です。

オープンソースソリューションに関するすべての利用可能な情報を考慮して、クラウドストレージ用に設計された2つの既存プロジェクト（過去12か月間にgitリポジトリでコミットされたもの）が見つかりました：NextCloudとOwnCloud、そして唯一のアクティブなオフィスアプリケーションパッケージONLYOFFICE。クラウドストレージの両方のツールはほぼ同じ機能を備えており、NextCloudを選択するかどうかの決定は、ソフトウェアとの快適なユーザー操作のためにONLYOFFICEと統合できるという証拠の存在に基づいていました。ただし、サービスの展開を開始したときに、上記のサービスの統合に関する情報の不足が明らかになりました。統合方法に関する3つのチュートリアルビデオが見つかりました。

  • www.youtube.com/watch?v=GAd-x_sP6ng
  • www.youtube.com/watch?v=ceOld6P16wI
  • www.youtube.com/watch?v=DvTXyl-OOZA

3つのビデオのいずれも、共有nginxを使用してNextCloudと同じ物理サーバーにONLYOFFICEドキュメントサービスをインストールするという質問に答えていませんでした。代わりに、APIドキュメントサービスに個別のポートを使用するなどの分離テクノロジーを使用していました。別の提案されたオプションは、ドキュメントサービスに組み込まれたnginxインスタンスの手動構成を使用してドキュメントサービス用に別のサーバーをデプロイし、アクセスキー（データクラウドへのアクセスを確認する既知のアクセスキー）とTLS証明書をインストールすることでした。上記のアプローチは安全ではなく、効果も十分ではないと考えられたため、Docker-composeを使用して、NextCloud、ONLYOFFICE、およびドメイン名でリクエストを共有する一般的なnginxを統合しました。これを行う方法について、順を追って説明します。

ステップ1：nginxコンテナー


これは非常に単純なインストールですが、この手順では、リバースプロキシサーバーを構成するために最も多くの作業が必要です。最初に、nginx：stableイメージのdocker-compose構成を作成しました。

version: '2'
services:
  nginx:
    image : nginx:stable
    restart: always
    volumes:
      - ./nginx/nginx-vhost.conf:/etc/nginx/conf.d/default.conf:ro
      - ./nginx/certificates:/mycerts 
    ports:
      - 443:443
      - 80:80

これにより、ポート80および443がパブリックアクセス用に開かれたコンテナーが作成され、構成がnginx / nginx-vhost.confにマップされ、自己署名証明書として、または/ nginx /証明書からLet's Go Encrypt証明書を使用して生成された証明書のストアが定義されます。この場所には、office.yourdomain.comとcloud.yourdomain.comのフォルダーが含まれ、それぞれ証明書チェーンとサーバーの秘密鍵にfullchain1.pemとprivkey1.pemのファイルが含まれている必要があります。自己署名証明書の生成方法の詳細については、www.akadia.com / services / ssh_test_certificate.htmlを参照してください（.keyおよび.crtの名前を.pemに変更すると、nginxのファイル構造を変換せずに機能します）。

その後、vhostファイルを定義しました。まず、ポート80の動作をhttpsへの単純なリダイレクトとして定義します。HTTPトラフィックを許可したくない

server {
    listen 80;
    location / {
        return 301
            https://$host$request_uri;
    }
}

次に、サービス用にポート443に2つの仮想サーバーを作成しました。 

server {
    listen 443 ssl;
    server_name cloud.yourdomain.com ;
    root /var/www/html;

    ssl_certificate     /mycerts/cloud.yourdomain.com/fullchain1.pem;
    ssl_certificate_key /mycerts/cloud.yourdomain.com/privkey1.pem;
    ssl_protocols       TLSv1 TLSv1.1 TLSv1.2;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    location / {
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Host $http_host;
        proxy_redirect off;
        proxy_pass http://app:80;
    }
}
server {
    listen 443 ssl;
    server_name office.yourdomain.com;
    root /var/www/html;

    ssl_certificate     /mycerts/office.yourdomain.com/fullchain1.pem;
    ssl_certificate_key /mycerts/office.yourdomain.com/privkey1.pem;
    ssl_protocols       TLSv1 TLSv1.1 TLSv1.2;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    location / {
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Host $http_host;
        proxy_redirect off;
        proxy_pass http://onlyoffice:80;
    }
}

ステップ2：ドキュメントサービス


次に、docker-compose.ymlにドキュメントサービスコンテナーを追加する必要があります。ここで構成する特別なものはありません。

services:
...
  onlyoffice:
    image: onlyoffice/documentserver
    restart: always

ただし、nginxコンテナをドキュメントサービスに関連付けることを忘れないでください。

services:
...
  nginx:
    ...
    depends_on:
      - onlyoffice

ステップ3：NextCloud


まず、新しいサービスを追加します。

services:
...
  db:
    image: mariadb
    command: --transaction-isolation=READ-COMMITTED --binlog-format=ROW
    restart: always
    volumes:
      - /data/nextcloud_db:/var/lib/mysql
    environment:
      - MYSQL_ROOT_PASSWORD=#put some password here
      - MYSQL_PASSWORD=#put some other password here
      - MYSQL_DATABASE=nextcloud
      - MYSQL_USER=nextcloud
  app:
    image: nextcloud
    depends_on:
      - db
      - onlyoffice
    restart: always

nginxへのリンクを追加します。

services:
...
  nginx:
    ...
    depends_on:
      - app

さあ、コンテナをロードします。 

docker-compose up -d

しばらくすると、nginxはNextCloudフロントエンド（デフォルトの構成ページ）へのリダイレクトを開始します。最初の管理ユーザーのユーザー名とパスワード、およびdocker-compose.ymlで指定したデータベース認証情報を入力する必要があります。構成が完了したら、ログインできます。私たちの場合、待機には1分近くかかり、クラウドサービスに入る前にログインページをさらに更新する必要がありました。

ステップ4：NextCloudとONLYOFFICEを接続する


このステップでは、ONLYOFFICE機能に接続するNextCloudアプリケーションをインストールする必要があります。メニューの右上隅にあるアプリケーションコントロールパネルから始めましょう。ONLYOFFICEアプリケーション（「Office＆text」セクションまたは検索を使用）を見つけ、インストールしてアクティブ化します。

その後、右上隅のメニューから[設定]に移動すると、左側のメニューでONLYOFFICE要素を見つける必要があります。それに入る。以下のようにアドレスを登録する必要があります。

統合アプリケーションの設定


最初のアドレスは、ブラウザーで実行されているアプリケーションから直接jsおよびcssファイルにリンクするために使用されます（これが、nginxを介してONLYOFFICEサービスへのアクセスを開く必要がある理由です）。永続的な認証キーよりもDocker分離層を信頼するため、秘密キーは使用されません。 3番目のアドレスは、ONLYOFFICE APIに直接接続するためにNextCloudコンテナーによって使用され、Dockerからのデフォルトの内部ホスト名を使用します。さて、最後のフィールドは、Dockerネットワークを使用する場合にONLYOFFICEが外部IPアドレスまたは内部Dockerアドレスを使用してNextCloud APIにリクエストを返すことができるように使用されますが、この場合は使用されません。ファイアウォールの設定でこれらのタイプの相互作用が許可されていることを確認してください。

保存後、NextCloudは接続をテストし、すべてが正しければ、統合に関連する設定を表示します。たとえば、この統合で編集できるファイルのタイプは何ですか。必要に応じてカスタマイズしてください。

最後のステップ：エディターを探す場所


クラウドストレージのフォルダーに戻り、「+」をクリックして新しいファイルを作成すると、ドキュメント、スプレッドシート、またはプレゼンテーションを作成するための新しいオプションが利用可能になります。彼らの助けを借りて、あなたはONLYOFFICEを使用してこれらのファイルタイプを作成し、すぐに編集できるようになります。
