Raspberry Pi上のEmbox RTOS

こんにちは！EmboxがRaspberry Piをサポートしている

かどうかをよく尋ねられます。はいあります。ここで今すぐ実行する方法の説明。この記事では、これについてもう少しお話ししたいと思います。

私たちは長い間Raspberry Pi Model B rev 2.0ボード（これは最初のRpi 1です）を購入し、移植の最初のステップを踏みました：UART、割り込みコントローラー、タイマー、さらにはフレームバッファーさえも何らかの形で実装されました。しかし、それを開始する方法に関するデータが失われたので、私はそれを再び覚える/理解する必要がありました。

まず、QEMUエミュレータで起動しました。通常のQEMUはraspi2マシンをサポートしていますが、raspiはサポートしていません。しかし、移植プロセスがあったときは、raspiサポートだけが追加されました。リポジトリにはまだバージョンがありますが、最新の環境にコンパイルするためにドラフトコミットを作成する必要がありましたが、最終的にはrpi1をサポートするQEMUエミュレーターのバージョン、「-M raspi」マシンを取得しました（wikiで実行する方法を確認できます））その結果、Emboxがそこでローンチされ、ビデオメモリに勾配が描画されました。

鉄から始めることについての質問がありました。最初は起動方法が明確ではなかったため、標準のRaspbianを試すことにしました。2020-02-13-raspbian-buster-lite.imgが公式サイトからダウンロードされます。すべてのダウンロードはmicroSDカードから行われるため、準備します-結果の画像をコピーします
dd bs=4M if=2020-02-13-raspbian-buster-lite.img of=/dev/sdb conv=fsync
ここで、「/ dev / sdb」はSDカードです。

ここでlsblkを見ると、次のようなものがあります。


sdb      8:16   1  14,6G  0 disk 
├─sdb1   8:17   1   256M  0 part 
└─sdb2   8:18   1  14,3G  0 part

sdb1をマウントして、そこに何があるか見てみましょう。


$ sudo mount /dev/sdb1 /mnt
$ ls /mnt/
bcm2708-rpi-b.dtb       bcm2710-rpi-3-b.dtb       COPYING.linux  fixup_db.dat      start_db.elf
bcm2708-rpi-b-plus.dtb  bcm2710-rpi-3-b-plus.dtb  fixup4cd.dat   fixup_x.dat       start.elf
bcm2708-rpi-cm.dtb      bcm2710-rpi-cm3.dtb       fixup4.dat     issue.txt         start_x.elf
bcm2708-rpi-zero.dtb    bcm2711-rpi-4-b.dtb       fixup4db.dat   kernel.img
bcm2708-rpi-zero-w.dtb  bootcode.bin              fixup4x.dat    LICENCE.broadcom
bcm2709-rpi-2-b.dtb     cmdline.txt               fixup_cd.dat   overlays
bcm2710-rpi-2-b.dtb     config.txt                fixup.dat      start_cd.elf

ご覧のように、さまざまなバージョンのRaspberry向けに、すべての状況に対応する* .dtbのセット全体があります。ブートローダー-bootcode.binも表示され、kernel.imgはLinuxです。

モニターをHDMI経由でボードに接続し、起動し、Raspbianを見ました。すべて問題ありません。次に、ブートローダーがそれをロードできるように、Emboxを含むバイナリをkernel.imgにコピーする必要があります。
Emboxをビルド：


make confload-arm/rpi1-model-b
make

結果のバイナリをコピーします。


cp build/base/bin/embox.bin /mnt/kernel.img

SDカードをRpiに戻し、電源を入れます。モニターでは、Emboxによって描画されるはずだった四角が表示されませんでした。さて、シリアルポート経由で接続してバイナリをデバッグしてみましょう。これを行うには、RDC1-USB-UARTアダプターを使用します。次のようにボードに接続します。


RDC1      Rapi

GND <---> GND
5V  <---> 5V
RX  <---> TXD0/GPIO14
TX  <---> RXD0/GPIO15

ここでは、写真にピン配置が表示されているため、許されています。これで、USBアダプターへの電源はPCのUSBハブから供給され、Rpiへの電源はUSBアダプターから供給されることがわかりました。



オンにする。 minicom経由で接続：


sudo minicom -d /dev/ttyUSB0

結論は見えません。シリアルポートドライバになんらかの問題があるはずです。レジスタの初期化を無効にし、ブートローダーによって構成されたものを使用して、SDカードに新しいイメージを入力します-出力が表示されます。すべてが明確です。つまり、正しく接続されていましたが、シリアルポートドライバーは正しくプログラムできません。 PL011シリアルポートドライバーをどのように修復したかについては触れませんが、このプラットフォーム用に開発する際に役立つ可能性があるいくつかの点に注意します。まず、Rpiのベースアドレスはなんとなく混乱して計算されます。それらはドキュメントではすぐに見つかりません。そのため、レジスタのベースアドレスを確認する良い方法は、Linuxでそれらを調べることです。


pi@raspberrypi:~$ ls /sys/bus/amba/devices/20201000.serial

アドレスが0x20201000であることがわかります。

-第二のポイントは、Linuxが組み立てとのkernel.imgにコピーすることができるRPI、ために修飾が存在することであるwww.raspberrypi.org/documentation/linux/kernel/building.md

、それは実際の入力周波数を理解する助けたシリアルポートの場合ボーレートのプログラミングに必要なUARTCLK-ドライバー/ tty /シリアル/ amba-pl011.cの適切な場所にprintk（）を挿入するだけです。

その結果、大きな変更なしに、QEMU on ironと同じイメージを起動することができました。

ドライバーのソースコードを逆アセンブルすることは意味がないので、おそらくこれですべてです。一般的に、Raspberryでは、他の非Linux OSの移植はほとんどないようです。たとえば、FreeRTOSは見つかりませんでした（ある種のリポジトリはGitHubにありますが、公式リリースにはありませんでした）、NuttX、またはそのようなものは見つかりませんでした。はい、フォーラムでの議論は、RTOSがおそらくが必要とされていないことを示唆しているとのArduinoをプレイするために、より良いそれを取る- www.raspberrypi.org/forums/viewtopic.php?t=201540 Osdevの

例はなって誰かがsomething-を望んでいるそうだとすれば、動作不能であることが判明それからベアメタルでそれを行い、そして私たちからのより良い例を見てください:)

新しいRpi 2/3/4の完全なサポートはまだ計画中です。ただし、一部のサポートはGSoC 2020の一部として提供される可能性が高く、これは学生の間で最も人気のあるアイデアの1つです。実際、GSoCについては、EmboxでRpi1サポートを復元しました。誰かが、彼らはRPI RTOS、コメント欄に書き込みを必要とする理由についての考えを持っている場合、私たちは幸せになります:)
