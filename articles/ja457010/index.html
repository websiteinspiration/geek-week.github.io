Webコンポーネントを使用しない理由

私はこれを主に将来のために自分で書くので、なぜWebコンポーネントに懐疑的で、SvelteがデフォルトでWebコンポーネントにコンパイルされないのかと誰かが尋ねられたときに参照する場所があります。（ただし、カスタム要素のあらゆる場所での優れた評価によって証明されているように、Webコンポーネントにコンパイルすることも、それらを統合することもできます）。


以下のいずれも、Webコンポーネントで行われたハードワークに対する批判として解釈されるべきではありません。おそらく、私はこの出版物でも間違いを犯したかもしれません。その場合、私はそれを訂正させていただきます。また、Webコンポーネントを使用してはならないことも宣言しません。彼らには独自のスコープがあります。それらが私に適さない理由を説明します。

class Adder extends HTMLElement {
  constructor() {
    super();

    this.attachShadow({ mode: 'open' });

    this.shadowRoot.innerHTML = `
      <input type="number">
      <input type="number">
      <p></p>
    `;

    this.inputs = this.shadowRoot.querySelectorAll('input');
    this.p = this.shadowRoot.querySelector('p');

    this.update();

    this.inputs[0].addEventListener('input', e => {
      this.a = +e.target.value;
    });

    this.inputs[1].addEventListener('input', e => {
      this.b = +e.target.value;
    });
  }

  static get observedAttributes() {
    return ['a', 'b'];
  }

  get a() {
    return +this.getAttribute('a');
  }

  set a(value) {
    this.setAttribute('a', value);
  }

  get b() {
    return +this.getAttribute('b');
  }

  set b(value) {
    this.setAttribute('b', value);
  }

  attributeChangedCallback() {
    this.update();
  }

  update() {
    this.inputs[0].value = this.a;
    this.inputs[1].value = this.b;

    this.p.textContent = `${this.a} + ${this.b} = ${this.a + this.b}`;
  }
}

customElements.define('my-adder', Adder);
