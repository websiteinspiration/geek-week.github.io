使用Dex在Kubernetes中进行身份验证：固定LDAP

今天，我将详细介绍结合使用Dex和LDAP在Kubernetes中配置身份验证的详细信息，并展示如何将静态用户添加到Dex。 

在本文中，我不会详细介绍Dex的基本原理，但马上我将继续安装和配置LDAP。您可以在本文中熟悉Dex的原理。

我们做什么：

  1. 安装OpenLDAP并在其上配置STARTTLS支持。 
  2. 我们描述了我们组织的LDAP目录的结构。
  3. 在kube-api服务器上启用OIDC（OpenID Connect）支持。
  4. 获取Dex将使用的域的SAN证书。
  5. 安装Dex和Dex-auth，我们在其中描述LDAP目录和静态用户。 
  6. 我们将生成用户的kubeconfig以使用集群。
  7. 为群集中的组和用户设置RBAC身份验证。

所以走吧



我将以一个带有Helm版本3和Ingress以及三个域名的现成Kubernetes集群的示例为您展示。

我们连接对OpenID Connect的支持


让我们继续进行Kubernetes集群设置。 

我们使用域dex.ash.dtln.cloud访问Dex API服务器，并使用login.ash.dtln.cloud访问集群。

我们部署的集群没有kubeadm或kops安装程序，因此OIDC配置可以立即添加到systemd单元中。在其他情况下，最好使用这些实用程序来完成配置。

  1. 我们编辑单元/etc/systemd/system/kube-apiserver.service并将启动参数添加到ExecStart部分：

    --oidc-client-id=dex-k8s-authenticator \
--oidc-groups-claim=groups \
--oidc-username-claim=email \
--oidc-issuer-url=https://dex.ash.dtln.cloud/ \
  2. 我们重新启动我们的API，检查它们是否已增加：

    sudo systemctl daemon-reload
sudo systemctl restart kube-apiserver
sudo systemctl status kube-apiserver

创建一个多域证书


现在转到Kubernetes集群。 

  1. 我们使用Helm安装cert-manager： 

    helm repo add jetstack https://charts.jetstack.io
helm repo update
helm install cert-manager --namespace kube-system jetstack/cert-manager --version v0.13.0

  2. 我们在cert.yml中描述了对我们的域的SAN证书的请求：

    ---
apiVersion: cert-manager.io/v1alpha2
kind: ClusterIssuer
metadata:
  name: letsencrypt-dex
spec:
  acme:
    email: kubernetes@dataline.ru
    server: https://acme-v02.api.letsencrypt.org/directory
    privateKeySecretRef:
      name: letsencrypt-key-dex
    solvers:
    - http01:
        ingress:
          class: nginx
---
apiVersion: cert-manager.io/v1alpha2
kind: Certificate
metadata:
  name: auth-dex
  namespace: kube-system
spec:
  secretName: cert-auth-dex
  issuerRef:
    kind: ClusterIssuer
    name: letsencrypt-dex
  commonName: dex.ash.dtln.cloud
  dnsNames:
  - dex.ash.dtln.cloud
  - login.ash.dtln.cloud

  3. 我们执行命令：

    kubectl apply -f cert.yaml

  4. 现在，使用以下命令查看证书请求的状态： 

    kubectl get certificates --all-namespaces
kubectl get challenges --all-namespaces

我们配置RBAC授权


  1. 例如，我们将静态用户分配给cluster-admin系统角色，将开发人员组的用户分配给视图角色，该角色仅允许查看资源。然后，crb.yml文件的内容应如下所示：

    ---
apiVersion: rbac.authorization.k8s.io/v1beta1
kind: ClusterRoleBinding
metadata:
  name: dex-admin
  namespace: kube-system
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cluster-admin
subjects:
- kind: User
  name: "admin@dtln.cloud"

---
apiVersion: rbac.authorization.k8s.io/v1beta1
kind: ClusterRoleBinding
metadata:
  name: dex-developers
  namespace: kube-system
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: view
subjects:
- kind: Group
  name: "developers"

  2. 切换到主上下文，并将创建的yaml文件应用于集群：

    kubectl config set-context default
kubectl apply -f crb.yml

  3. 我们查看可用的上下文并切换回我们的用户：

    kubectl config get-contexts
kubectl config set-context johndoe-ash.dtln.cloud

这样就完成了与OpenLDAP结合的Dex的设置。
