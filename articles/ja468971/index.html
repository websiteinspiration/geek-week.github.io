Pstros nds


秘蔵アーカイブをサイトからダウンロードした後、調べ始めました。Readmeを読んだ後、Pstros NDSはニンテンドーDSのKVMポート（Sun Microsystemsによって開発されたJava仮想マシン）であることがわかりました。インターネット上の記事（記事1、記事2）からわかるように、このJavaマシンは限られた量のRAMを備えた小さなデバイス用に設計されており、JavaプラットフォームのサブセットであるJ2MEに属しています。

J2MEアーキテクチャ


J2MEには、構成とプロファイルがコア要素であるモジュール式アーキテクチャがあります。

構成とは、モバイルデバイスの特定のファミリ（Java言語の機能、仮想マシンの機能と機能、サポートされる最低限のライブラリ）の開発に使用できるツールを説明する仕様です。

このプロファイルは、特定のモバイルデバイスの構成機能を拡張および補完します。グラフィックディスプレイ、通信機能、特定の周辺機器（この場合はタッチスクリーン）を操作できます。

エミュレータ


ニンテンドーDSをコンピューターに直接接続することは、私の知る限り、不可能です。microSDを常にコンピューターからデバイスに貼り付けて、結果を確認するには長すぎる。したがって、アプリケーションのテスト起動には、人気のあるDeSmuMEエミュレーターを使用します。

DeSmuMEの自作アプリケーションの実行にはいくつかの機能があります。私が質問を理解している限り、これはフラッシュカートリッジ上のファイルシステムでの作業の問題に起因します。

自作アプリケーションとファイルシステムとの相互作用のために、libfatライブラリが作成されました。ただし、Nintendo DSには非常に多くのフラッシュカートリッジがあり、それらはすべて特定のファイル読み取り/書き込みコマンドを使用します。 libfatライブラリはこれらのコマンドを認識できず、エラーメッセージを自作アプリケーションに送信します。これを回避するために、DLDI（Dynamically-Linked Device Interface）と呼ばれる技術が発明されました。これは、homebrewアプリケーションとフラッシュカートリッジの間のレイヤーとして機能し、特定の読み取り/書き込みコマンドをlibfatライブラリが理解できるコマンドに変換します。

開発環境のセットアップ


jvmでアーカイブを調べたところ、_rebuild.batファイルに気づきました。その内容を以下に示します。

コンテンツ_rebuild.bat
SET WTKDIR=c:/wtk22
SET CP=%WTKDIR%/lib/cldcapi10.jar;%WTKDIR%/lib/midpapi20.jar;./classes.zip 
del .\output\*.* /Q
"c:/program files/java/jdk1.5.0_06/bin/javac.exe" -source 1.3 -target 1.1 -bootclasspath %CP% ./src/*.java
%WTKDIR%/bin/preverify -classpath %CP% ./src
del .\src\*.class /Q
copy output\*.* .\


これから、JVM用のアプリケーションを構築する方法と、必要な環境の種類が明らかになります。オラクルのWebサイトでは、アーカイブには必要なバージョンのwtkおよびjdkが含まれています。batファイルで指定されたパスに従ってインストールする（またはbatファイルのパスを編集する）と、開発を開始できます。

アプリケーション開発


私の仕事の性質上、Swing GUIを作成するためにライブラリーとたくさん協力しました。私はこのライブラリーが好きなので、例として、独自の実装を作成することにしました。私は「ブラックボックス」の原則に基づいて開発することを決めました。それが外にどのように見えるべきか知っているので、それがどのように内部に配置されているかを見ないでください。第一に、私が計画したライブラリは機能性が大幅に低下するため（エンターテインメントのために開発しているため）、おそらく、Swingで使用されるソリューションの多くは単に私にとって必要ではないでしょう。第二に、独自の実装を考え出すことははるかに興味深いです。

ビルドスクリプトからわかるように、Java 1.1で開発する必要があります。

要件に関して、ライブラリユーザーに最初に求めたのは、ダイアログフォームを呼び出すための一般的なSwingコードをサポートすることでした。

MyDialogForm myDialogForm = new MyDialogForm(this, true); 

myDialogForm.setVisible(true); 

if (myDialogForm.getAnsewer()) { 
    // TODO - do something here 
}

それら。この場合、ダイアログフォームが表示され、ダイアログフォームが閉じられるまで、実行はsetVisible（true）を超えません。同時に、呼び出されたフォームにある他のコンポーネントについては、それらのユーザーアクションの処理を続行する必要があります。

ライブラリ操作スキームの最初のバージョンを次の図に示します。



スキームからわかるように、各ユーザーアクションは個別のスレッドで実行されます。この実装により、ダイアログフォームを操作するための要件は満たされますが、ハンドラーのユーザーが2つのスレッドから同じデータを同時に操作できる（たとえば、2つのボタンをすばやく押す）ことができるという問題があります。これにより、ユーザーがそのようなデータへのアクセスを保護する必要性に関連する追加の不便が生じます。

すべてのフォームがインタラクティブであることを前提としています。1つのフォームを表示する場合、他のフォームを操作することはできません。この場合、ニンテンドーDSの画面は大きくなく、一度に複数のフォームを表示するのは不便なので、これは適切な前提です。

この場合、ライブラリーの回路は次のようになります。



すなわち 各フォームは、アプリケーションのユーザーアクションを処理する独自のフローを作成します。アクションの処理の結果として新しいフォームが呼び出されると、それを呼び出したスレッドはフォームが閉じるのを待機し、この時点でユーザーアクションを処理するための新しいストリームが作成されます。
