3番目のSteam Windowsクライアントの脆弱性（0dayではない）

前のシリーズ


少し前に、CVE-2019-14743とCVE-2019-15316という Steamの2つの脆弱性について話しました。私がそれらを登録しようとした方法についての全体的な話がありました、私は成功しませんでした、彼らは私を禁止しました、そして私が結果を達成するためにどうにかして公の開示とコミュニティの助けの後にのみ。Valveは、HackerOneで私を謝罪して禁止を解除するふりをしたので、このサービスを通じて次の脆弱性を転送することにしました。3回目（Half-Life 3について明らかな冗談があるはずです）、すべてが多かれ少なかれ成功しました。



脆弱性の説明


この脆弱性により、部分的に制御されたコンテンツを持つファイルの作成（または既存のファイルへの部分的に制御されたコンテンツの追加）が可能になります。脆弱なサービスのバージョンは5.31.28.21（SteamService.exeファイルのデータ）です。最初に、この脆弱性を悪用する方法について説明し、次に考えられる結果について説明します。

ステップ1.環境


Steamアプリケーションを完了し、Steamクライアントサービスが実行されている場合は停止する必要があります。ほとんどの場合、管理者権限のないユーザーはサービスを開始および停止できません。ただし、特にこのサービスでは、Valveはすべてのユーザーがサービスを停止および開始できるように権限を設定しました。

ユーザーがアクセスできる任意の場所にフォルダーを作成します（たとえば、「C：\ test」）。このフォルダーでは、SteamソースフォルダーからSteam.exeファイルとsteamclient.dllファイルをコピーする必要があります（デフォルトでは、これは「C：\ Program Files（x86）\ Steam」です）。空のログサブフォルダーを作成します（ "C：\ test \ logs"）。

次に、レジストリを修正します。「HKLM \ Software \ wow6432node \ valve \ steam」ブランチで、「InstallPath」パラメータの値を「C：\ test \ 1 \ ..」に変更します。通常、非管理者ユーザーの場合、HKLM内のレジストリブランチは書き込み可能ではありませんが、この場合は書き込み可能ではありません。Valveをインストールするときに、HKLM内のブランチにそのような権限を設定し、その中ですべてのアクションをすべてのユーザーが使用できるようにします（「ユーザー」グループの「フルコントロール」）。

ステップ2.少しテストしてみましょう


Steamクライアントサービスを起動します。停止後（数秒後に自動的に発生します）、「C：\ test \ logs」フォルダーの内容を確認します。「service_log.txt」ファイルがそこにあります。ログの内容は次のようになります。

08/27/19 13:45:01 : ERROR: SteamService: Invalid file signature C:\test\1\..\bin\SteamService.dll

パス "C：\ test \ 1 \ .."はパス "C：\ test"と同等であるため、Windowsは2番目のパスを使用して機能し、最初のパスはメッセージに入りました。「service_log.txt」ファイルを削除して続行します。

ステップ3.さらにテキストを追加します。


興味深い事実：Windowsが「\ ..」を含むパスを処理する場合、そのようなパスは自動的に単純化されます。ステージングフォルダーのチェックを行いません。

たとえば、パス名 "C：\ 1 \ <test> \ .."は、フォルダー名に山かっこを使用できないにもかかわらず、 "C：\ 1"に変換されます。

最初のステップで、レジストリにパスを登録しました。次に、改行を追加します。これは簡単なコードを書くことで実行できますが、regeditインターフェースからも実行できます。レジストリブランチ「HKLM \ Software \ wow6432node \ valve \ steam」を開き、「InstallPath」パラメーターのコンテキストメニューで「Modify binary data ..」を選択するだけで十分です。 16進エディタのようなものが表示され、必要な編集を行うことができます。



サービスの別のテスト起動を実行し、アクションの結果を確認します。



テスト後、ファイル「service_log.txt」を再度削除する必要があります。

ステップ4.作成したファイルをリダイレクトする


管理者権限のないユーザーは、あるファイルから別のファイルへのシンボリックリンクを作成できません。しかし、ここにはトリックがあります。ファイルからファイルへのシンボリックリンクに近い効果を得るために、管理者権限なしでユーザーが利用できる他のタイプのリンクを組み合わせることができます。

最初に、フォルダ「C：\ test \ logs」から「\ RPC Control \」にNTFS再解析ポイント（NTFSマウントポイントの別名）を作成します。 「\ RPC Control \」は、通常の意味での通常のフォルダではなく、エクスプローラなどでは表示できません。これはシステムオブジェクトディレクトリであり、その中には、名前付きミューテックス、イベント、およびその他の同様のオブジェクトが含まれています。 NTFS再解析ポイントによるリダイレクトが機能する理由は明確ではありませんが、おそらくファイルシステムとオブジェクトディレクトリのフォルダーに同じ抽象化を使用することがポイントです。オブジェクトディレクトリから、管理者権限なしでファイルへのシンボリックリンクを作成できます。 "\ RPC Control \ service_log.txt" <-> "C：\ Path \ to \ file"という形式のシンボリックリンクを作成します。

その結果、「C：\ test \ logs \ service_log.txt」へのすべての呼び出しは、ファイル「C：\ Path \ to \ file」にリダイレクトされます。このようなリダイレクトを作成するには、2つの主要な要件があります。NTFS再解析ポイントの作成元のフォルダーは空である必要があり、ユーザーが書き込み可能である必要もあります。最初の条件を満たすために、各テストの後にファイル「service_logs.txt」を削除しました。2番目の条件は、ユーザーが制御する場所にソースフォルダーを作成したという事実によって確認されています。

このようなシンボリックリンクのペアを作成する特別なユーティリティ-CreateSymlinkがあり、GitHubからダウンロードできます。ユーティリティの使用：

CreateSymlink.exe <> <>

私たちの場合、それは：

CreateSymlink.exe "C:\test\logs\service_log.txt" "C:\\\"

まとめると、Steamクライアントサービスが開始すると、シンボリックリンクの作成時に指定されたパスに沿ってファイルが作成され、このファイルには制御可能なコンテンツが含まれます（最初と最後の行を除いて）。既存のファイルへのパスを指定すると、コンテンツはファイルの最後に追加されます。これはすべて、NT AUTHORITY \ SYSTEM権限を持つSteamクライアントサービスに代わって行われます。

調査結果


これが私がSteamリサーチの投稿を終えるところです-かなり表面的な分析で発見された3つの脆弱性、これは十分ではありません。より深く行くには、より多くの時間と欲求が必要です。残念ながら、Valveの態度とHackerOne従業員の無能さは、非常に大きな障壁です。

Steamをより安全にするのを助けてくれたすべての読者に改めて感謝したい。脆弱性を修正し、陰謀論に反論してくれたValveに感謝します。基本的に脆弱性をValveに伝えることができなかったにもかかわらず、プラットフォームを提供してくれたHackerOneに感謝します。
