最初の部分を読む

オブジェクトの説明


オブジェクトの説明では、インスタンスを作成する機能を持つクラスと間違えられないシングルトンのみを許可します。したがって、静的クラスの変数として、またはアプリケーションクラスにシングレットを格納する必要はありません。

たとえば、スレッドに関連付けられた静的メソッドを持つユーティリティクラスがあり、アプリケーション全体にアクセスしたい場合は、次のようにします。

package com.myapps.example.util
import android.os.Handler
import android.os.Looper
// ,   ,   
object ThreadUtil {
    fun onMainThread(runnable: Runnable) {
        val mainHandler = Handler(Looper.getMainLooper())
        mainHandler.post(runnable)
    }
}

ThreadUtilは、静的クラスメソッドを呼び出すときと同じ方法で後で呼び出すことができます。

ThreadUtil.onMainThread(runnable)

つまり、プライベートコンストラクターを使用して静的クラスの動作を模倣する必要がなくなり、インスタンスが格納されている場所を把握する必要がなくなります。実際、オブジェクトは言語の元の要素です。同じ原理で、内部クラスの代わりにオブジェクトを作成します。

iewPager.addOnPageChangeListener(object : ViewPager.OnPageChangeListener {
    override fun onPageScrollStateChanged(state: Int) {}
    override fun onPageScrolled(position: Int, positionOffset: Float, positionOffsetPixels: Int) {}
    override fun onPageSelected(position: Int) {
        bindUser(position)
    }
});

どちらも基本的に同じことを行います-宣言されたオブジェクトとしてクラスの1つのインスタンスを作成します。

サポートするオブジェクト


一見すると、Kotlinには静的変数とメソッドがありません。この言語にはそのような概念はありませんが、補助オブジェクトの概念があります。これらは、静的な方法でアクセスできるメソッドと変数を含むクラス内のシングルトンオブジェクトです。コンパニオンオブジェクトでは、Javaの静的クラスと同様に、特定の定数とメソッドを使用できます。これにより、newInstanceスニペットパターンに従うことができます。

最も単純な形式のコンパニオンオブジェクトを見てみましょう。

class User {
    companion object {
        const val DEFAULT_USER_AGE = 30
    }
}
//      :
user.age = User.DEFAULT_USER_AGE


Androidでは、通常、静的メソッドと変数を使用してフラグメントの静的ファクトリーを作成します。例えば：

class ViewUserActivity : AppCompatActivity() {
    companion object {
const val KEY_USER = "user"
        fun intent(context: Context, user: User): Intent {
            val intent = Intent(context, ViewUserActivity::class.java)
            intent.putExtra(KEY_USER, user)
            return intent
        }
    }    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_cooking)
        val user = intent.getParcelableExtra<User>(KEY_USER)
        //...
    }
}

インテントの 作成は、Javaでの同様のアクションに似ています。

val intent = ViewUserActivity.intent(context, user)
startActivity(intent)

このパターンは、ユーザー定義またはその他のコンテンツを表示するために必要なデータがインテントまたはフラグメントに含まれない可能性が低くなるため、優れています。コンパニオンオブジェクトは、Kotlinで静的アクセスフォームを保持する方法であり、クラスの不足を補うために使用する必要があります。

グローバル定数


Kotlinでは、アプリケーションの1つの場所に表示される定数を定義できます（該当する場合）。ただし、定数の範囲はできる限り最小限に抑える必要があります。また、リージョンをグローバルにしたい場合、Kotlinには定数クラスを使用せずにこれを行う優れた方法があります。何かのようなもの：

package com.myapps.example
import android.support.annotation.StringDef
//   ,  !
const val PRESENTATION_MODE_PRESENTING = "presenting"
const val PRESENTATION_MODE_EDITING = "editing"

次に、プロジェクト内の任意の場所で定数として使用できます。

import com.savvyapps.example.PRESENTATION_MODE_EDITING
val currentPresentationMode = PRESENTATION_MODE_EDITING

プロジェクトの定数は、その複雑さを軽減するためにできるだけ小さくする必要があります。カスタムクラスにのみ適用される値がある場合は、ヘルパーオブジェクトに配置することをお勧めします。

拡張


拡張機能は、クラス機能を継承せずに追加できるので便利です。たとえば、アクティビティにメソッドを追加する方法はhideKeyboard()？拡張機能を使用すると、これを簡単に行うことができます。

fun Activity.hideKeyboard(): Boolean {
    val view = currentFocus
    view?.let {
        val inputMethodManager = getSystemService(Context.INPUT_METHOD_SERVICE) as InputMethodManager
        return inputMethodManager.hideSoftInputFromWindow(view.windowToken,
                InputMethodManager.HIDE_NOT_ALWAYS)
    }
    return false
}

拡張機能は次の点で便利です。

  • コードを読みやすくする
  • ユーティリティクラスとメソッドを作成する必要がなくなります。

さらに進んで、コードのアーキテクチャーを改善できます。APIによってソースから抽出されたデータのクラスと見なされるArticleなどの基本モデルがあるとします。

class Article(val title: String, val numberOfViews: Int, val topic: String)

何らかの式に基づいて、ユーザーの記事の関連性を決定する必要があります。Articleクラスに直接配置する必要がありますか？また、モデルにAPIからのデータのみを含める必要がある場合は、拡張機能を再度使用できます。

fun Article.isArticleRelevant(user: User): Boolean {
    return user.favoriteTopics.contains(topic)
}

これは現在、ユーザーのお気に入りのトピックのリストでArticleトピックの存在を確認する簡単な方法です。

このロジックは、ユーザーの動作のこれらの属性やその他の属性をテストする場所によって異なります。このロジックはArticleモデルに関係なくある程度サポートされているため、目的、方法、変更する機能に応じて変更できます。
