DBA：役に立たないインデックスを見つける

私は定期的に、PostgreSQLのインデックスがクエリのパフォーマンスの問題に広く役立つスイスのナイフであると多くの開発者が心から信じている状況に遭遇します。テーブルに新しいインデックスを追加するか、既存のインデックスのどこかにフィールドを含めるだけで十分です。その後、（magic-magic！）すべてのクエリがこのインデックスを効果的に使用します。

最初に、もちろん、そうではないか、効率的でないか、またはすべてではありません。第二に、余分なインデックスは、書き込み時にパフォーマンスの問題を追加するだけです。

ほとんどの場合、このような状況は「長期的な」開発中に発生します。カスタム製品が「一度だけ書いた、与えられた、忘れられた」モデルに従って作成されていないが、私たちの場合のように作成されている場合ライフサイクルの長いサービス。

改善は、多くの分散チームの力によって繰り返し発生します。これらのチームは、空間だけでなく時間内にも分散されます。そして、プロジェクトの開発の全履歴や、データベース内のデータの適用された分布の特徴を知らなくても、インデックスを簡単に「混乱」させることができます。しかし、考慮事項とカットの下でのテストリクエストにより、問題の一部を事前に予測および検出できます。

  • 未使用のインデックス
  • 接頭辞「クローン」
  • 「真ん中」のタイムスタンプ
  • インデックス可能なブール
  • インデックス内の配列
  • ヌルゴミ

最も簡単なことは、パスがまったくないインデックスを見つけることです。統計情報（pg_stat_reset()）のリセットがかなり前に行われたことを確認するだけでよく、使用された「まれに、しかし適切に」使用されたものを削除したくないです。システムビューを使用しますpg_stat_user_indexes。

SELECT * FROM pg_stat_user_indexes WHERE idx_scan = 0;

ただし、インデックスが使用されていて、この選択に当てはまらなかった場合でも、クエリに適しているとは限りません。

どのインデックスが[適切ではありません]


一部のクエリが「インデックスに悪影響を与える」理由を理解するために、通常最も頻繁に使用されるインスタンスである通常のbtreeインデックスの構造について考えます。通常、単一のフィールドのインデックスは問題を引き起こしません。そのため、フィールドのペアの複合で発生する問題を考慮します。

想像できるように、非常に簡略化された方法は「レイヤードケーキ」で、各レイヤーには、対応するフィールドの値に従って順番にツリーが並べられています。



これで、フィールドAがグローバルに順序付けられ、Bが特定の値A内のみであることは明らかです。実際のクエリで発生する条件の例と、それらがインデックス内をどのように「ウォーク」するかを見てみましょう。

良い：前置条件


インデックスbtree(A, B)には「サブインデックス」が含まれることに注意してくださいbtree(A)。これは、以下に説明するすべてのルールが任意のプレフィックスインデックスに対して機能することを意味します。

つまり、この例のタイプよりも複雑なインデックスを作成btree(A, B, C)すると、データベースが自動的に「出現」すると見なすことができます。

  • btree(A, B, C)
  • btree(A, B)
  • btree(A)

これは、データベースにプレフィックスインデックスが「物理的に」存在することは、ほとんどの場合冗長であることを意味します。すべての後に、さらに悪いことがある-複数のインデックステーブルには、書き込みに持っている -それは書き込み増幅を呼び出しているので、PostgreSQLのユーバーは、この不満（とここであなたができる彼らの主張の分析を見つけます）。

そして何かが基地がうまく生きることを妨げるなら、それを見つけて取り除く価値があります。例を見てみましょう：

CREATE TABLE tbl(A integer, B integer, val integer);
CREATE INDEX ON tbl(A, B)
  WHERE val IS NULL;
CREATE INDEX ON tbl(A) --  #1
  WHERE val IS NULL;
CREATE INDEX ON tbl(A, B, val);
CREATE INDEX ON tbl(A); --  #2

プレフィックスインデックス検索クエリ
WITH sch AS (
  SELECT
    'public'::text sch -- schema
)
, def AS (
  SELECT
    clr.relname nmt
  , cli.relname nmi
  , pg_get_indexdef(cli.oid) def
  , cli.oid clioid
  , clr
  , cli
  , idx
, (
    SELECT
      array_agg(T::text ORDER BY f.i)
    FROM
      (
        SELECT
          clr.oid rel
        , i
        , idx.indkey[i] ik
        FROM
          generate_subscripts(idx.indkey, 1) i
      ) f
    JOIN
      pg_attribute T
        ON (T.attrelid, T.attnum) = (f.rel, f.ik)
  ) fld$
  FROM
    pg_class clr
  JOIN
    pg_index idx
      ON idx.indrelid = clr.oid AND
      idx.indexprs IS NULL
  JOIN
    pg_class cli
      ON cli.oid = idx.indexrelid
  JOIN
    pg_namespace nsp
      ON nsp.oid = cli.relnamespace AND
      nsp.nspname = (TABLE sch)
  WHERE
    NOT idx.indisunique AND
    idx.indisready AND
    idx.indisvalid
  ORDER BY
    clr.relname, cli.relname
)
, fld AS (
  SELECT
    *
  , ARRAY(
      SELECT
        (att::pg_attribute).attname
      FROM
        unnest(fld$) att
    ) nmf$
  , ARRAY(
      SELECT
        (
          SELECT
            typname
          FROM
            pg_type
          WHERE
            oid = (att::pg_attribute).atttypid
        )
      FROM
        unnest(fld$) att
    ) tpf$
  , CASE
      WHEN def ~ ' WHERE ' THEN regexp_replace(def, E'.* WHERE ', '')
    END wh
  FROM
    def
)
, pre AS (
  SELECT
    nmt
  , wh
  , nmf$
  , tpf$
  , nmi
  , def
  FROM
    fld
  ORDER BY
    1, 2, 3
)
SELECT DISTINCT
  Y.*
FROM
  pre X
JOIN
  pre Y
    ON Y.nmi <> X.nmi AND
    (Y.nmt, Y.wh) IS NOT DISTINCT FROM (X.nmt, X.wh) AND
    (
      Y.nmf$[1:array_length(X.nmf$, 1)] = X.nmf$ OR
      X.nmf$[1:array_length(Y.nmf$, 1)] = Y.nmf$
    )
ORDER BY
  1, 2, 3;

理想的には、空の選択を取得する必要がありますが、見てください-疑わしいインデックスグループです。

nmt | wh            | nmf$      | tpf$             | nmi             | def
---------------------------------------------------------------------------------------
tbl | (val IS NULL) | {a}       | {int4}           | tbl_a_idx       | CREATE INDEX ...
tbl | (val IS NULL) | {a,b}     | {int4,int4}      | tbl_a_b_idx     | CREATE INDEX ...
tbl |               | {a}       | {int4}           | tbl_a_idx1      | CREATE INDEX ...
tbl |               | {a,b,val} | {int4,int4,int4} | tbl_a_b_val_idx | CREATE INDEX ...

次に、各グループについて、短いインデックスを削除する価値があるのか​​、それとも長いインデックスを削除する必要がないのかを自分で決定します。

良い：最後のフィールドを除くすべての定数


最後を除くインデックスのすべてのフィールド の値が定数によって設定されている場合（この例では、これはフィールドAです）、インデックスは通常どおり使用できます。この場合、最後のフィールドの値は任意に設定できます：定数、不等式、間隔、ダイヤル、IN (...)または= ANY(...)。また、並べ替えることもできます。



  • WHERE A = constA AND B [op] constB / = ANY(...) / IN (...)
    op : { =, >, >=, <, <= }
  • WHERE A = constA AND B BETWEEN constB1 AND constB2
  • WHERE A = constA ORDER BY B

上記のプレフィックスインデックスに基づいて、これはうまく機能します。

  • WHERE A [op] const / = ANY(...) / IN (...)
    op : { =, >, >=, <, <= }
  • WHERE A BETWEEN const1 AND const2
  • ORDER BY A
  • WHERE (A, B) [op] (constA, constB) / = ANY(...) / IN (...)
    op : { =, >, >=, <, <= }
  • ORDER BY A, B

悪い例：「レイヤー」の完全な列挙


クエリの一部では、インデックス内の動きの唯一の列挙は、「レイヤー」の1つにあるすべての値の完全な列挙になります。そのような値が1であり、何千もある場合は幸運です。

通常、このような問題は、クエリで不等式が使用されている場合に発生します。条件は、インデックスの順序で前のフィールドを決定しないか、並べ替え中にこの順序に違反しています。

  • WHERE A <> const
  • WHERE B [op] const / = ANY(...) / IN (...)
  • ORDER BY B
  • ORDER BY B, A

悪い：間隔またはセットが最後のフィールドにありません


前のものの結果として、いくつかの値またはそれらの範囲を中間の「レイヤー」で見つけて、インデックスの「より深い」にあるフィールドでフィルターまたはソートする必要がある場合、インデックスの「中央」にある一意の値の数がすごい。

  • WHERE A BETWEEN constA1 AND constA2 AND B BETWEEN constB1 AND constB2
  • WHERE A = ANY(...) AND B = const
  • WHERE A = ANY(...) ORDER BY B
  • WHERE A = ANY(...) AND B = ANY(...)

悪い例：フィールドではなく式


開発者が無意識のうちにクエリの列を別の何か-インデックスのない式に変えることがあります。これは、目的の式からインデックスを作成するか、逆変換を実行することで修正できます。

  • WHERE A - const1 [op] const2
    修正： WHERE A [op] const1 + const2
  • WHERE A::typeOfConst = const
    修正： WHERE A = const::typeOfA

フィールドのカーディナリティを考慮します


あなたは、インデックスを必要とし(A, B)、あなたがしたいだけ平等で選びます：(A, B) = (constA, constB)。ハッシュインデックスの使用は理想的ですが、...バージョン10までのそのようなインデックスの非ジャーナリング（Walロギング）に加えて、それらはいくつかのフィールドに存在することもできません。

CREATE INDEX ON tbl USING hash(A, B);
-- ERROR:  access method "hash" does not support multicolumn indexes

通常、btreeを選択しました。では、列を配置する最良の方法は何ですか- (A, B)または(B, A)？この質問に答えるには、対応する列のデータのカーディナリティ、つまり、それに含まれる一意の値の数などのパラメーターを考慮する必要があります。

それを想像してA = {1,2}, B = {1,2,3,4}、両方のオプションのインデックスツリーのアウトラインを描画します。



実際、描画するツリー内の各ノードはインデックス内のページです。また、その数が多いほど、インデックスが占有するディスク領域が多くなるため、インデックスからの読み取りに時間がかかります。

この例では、オプションに(A, B)は10個のノードと(B, A)-12があります。つまり、「フィールド」を可能な限り少数の一意の値で「最初」に置くと、より収益性が高くなります。

悪い例：たくさんの場違い（タイムスタンプ「途中」）

CREATE INDEX ON tbl(B, A); -- -

btreeの配列


別のポイントは、btreeインデックスを使用して「配列にインデックスを付ける」試みです。対応する演算子が適用されるため、これは完全に可能です。
(<, >, = . .) , B-, , . ( ). , , .
しかし問題は、包含と交差の演算子に彼が望むものを使用することです<@, @>, &&。もちろん、他のタイプのインデックスが必要なため、これは機能しません。特定の要素にアクセスする機能に対して、そのようなbtreeがどのように機能しないかarr[i]。

私たちはそのようなものを見つけることを学びます：

CREATE TABLE tbl(
  id
    serial
      PRIMARY KEY
, pid
    integer
, list
    integer[]
);
CREATE INDEX ON tbl(pid);
CREATE INDEX ON tbl(list); -- -

  , cli.relname nmi
  , pg_get_indexdef(cli.oid) def

NULLインデックスエントリ


最後の非常に一般的な問題は、完全にNULLのエントリでインデックスを「散らかす」ことです。つまり、各列のインデックス付き式がNULLであるレコード。このようなレコードには実用的なメリットはありませんが、挿入するたびに害が生じます。

通常、これらは、テーブルでオプションのパディングを使用してFKフィールドまたは値の関係を作成するときに表示されます。次に、インデックスを転がして、FKがすばやく機能するようにします。接続が満たされる頻度が少ないほど、インデックスに含まれる「ガベージ」が多くなります。シミュレートします：

CREATE TABLE tbl(
  id
    serial
      PRIMARY KEY
, fk
    integer
);
CREATE INDEX ON tbl(fk);

INSERT INTO tbl(fk)
SELECT
  CASE WHEN i % 10 = 0 THEN i END
FROM
  generate_series(1, 1000000) i;

ほとんどの場合、このようなインデックスは条件付きインデックスに変換できます。

CREATE INDEX ON tbl(fk) WHERE (fk) IS NOT NULL;

そのようなインデックスを見つけるには、データの実際の分布を知る必要があります。つまり、結局のところ、テーブルのすべての内容を読み取り、発生のWHERE条件に従って重ね合わせます（これはdblinkを使用して行います）。これには非常に長い時間がかかる場合があります。

インデックス内のNULLエントリの検索クエリ
WITH sch AS (
  SELECT
    'public'::text sch -- schema
)
, def AS (
  SELECT
    clr.relname nmt
  , cli.relname nmi
  , pg_get_indexdef(cli.oid) def
  , cli.oid clioid
  , clr
  , cli
  FROM
    pg_class clr
  JOIN
    pg_index idx
      ON idx.indrelid = clr.oid
  JOIN
    pg_class cli
      ON cli.oid = idx.indexrelid
  JOIN
    pg_namespace nsp
      ON nsp.oid = cli.relnamespace AND
      nsp.nspname = (TABLE sch)
  WHERE
    NOT idx.indisprimary AND
    idx.indisready AND
    idx.indisvalid AND
    NOT EXISTS(
      SELECT
        NULL
      FROM
        pg_constraint
      WHERE
        conindid = cli.oid
      LIMIT 1
    ) AND
    pg_relation_size(cli.oid) > 1 << 20 --  1MB   
  ORDER BY
    clr.relname, cli.relname
)
, fld AS (
  SELECT
    *
  , regexp_replace(
      CASE
        WHEN def ~ ' USING btree ' THEN
          regexp_replace(def, E'.* USING btree (.*?)($| WHERE .*)', E'\\1')
      END
    , E' ([a-z]*_pattern_ops|(ASC|DESC)|NULLS\\s?(?:FIRST|LAST))'
    , ''
    , 'ig'
    ) fld
  , CASE
      WHEN def ~ ' WHERE ' THEN regexp_replace(def, E'.* WHERE ', '')
    END wh
  FROM
    def
)
, q AS (
  SELECT
    nmt
  , $q$-- $q$ || quote_ident(nmt) || $q$
      SET search_path = $q$ || quote_ident((TABLE sch)) || $q$, public;
      SELECT
        ARRAY[
          count(*)
        $q$ || string_agg(
          ', coalesce(sum((' || coalesce(wh, 'TRUE') || ')::integer), 0)' || E'\n' ||
          ', coalesce(sum(((' || coalesce(wh, 'TRUE') || ') AND (' || fld || ' IS NULL))::integer), 0)' || E'\n'
        , '' ORDER BY nmi) || $q$
        ]
      FROM
        $q$ || quote_ident((TABLE sch)) || $q$.$q$ || quote_ident(nmt) || $q$
    $q$ q
  , array_agg(clioid ORDER BY nmi) oid$
  , array_agg(nmi ORDER BY nmi) idx$
  , array_agg(fld ORDER BY nmi) fld$
  , array_agg(wh ORDER BY nmi) wh$
  FROM
    fld
  WHERE
    fld IS NOT NULL
  GROUP BY
    1
  ORDER BY
    1
)
, res AS (
  SELECT
    *
  , (
      SELECT
        qty
      FROM
        dblink(
          'dbname=' || current_database() || ' port=' || current_setting('port')
        , q
        ) T(qty bigint[])
    ) qty
  FROM
    q
)
, iter AS (
  SELECT
    *
  , generate_subscripts(idx$, 1) i
  FROM
    res
)
, stat AS (
  SELECT
    nmt table_name
  , idx$[i] index_name
  , pg_relation_size(oid$[i]) index_size
  , pg_size_pretty(pg_relation_size(oid$[i])) index_size_humanize
  , regexp_replace(fld$[i], E'^\\((.*)\\)$', E'\\1') index_fields
  , regexp_replace(wh$[i], E'^\\((.*)\\)$', E'\\1') index_cond
  , qty[1] table_rec_count
  , qty[i * 2] index_rec_count
  , qty[i * 2 + 1] index_rec_count_null
  FROM
    iter
)
SELECT
  *
, CASE
    WHEN table_rec_count > 0
      THEN index_rec_count::double precision / table_rec_count::double precision * 100
    ELSE 0
  END::numeric(32,2) index_cover_prc
, CASE
    WHEN index_rec_count > 0
      THEN index_rec_count_null::double precision / index_rec_count::double precision * 100
    ELSE 0
  END::numeric(32,2) index_null_prc
FROM
  stat
WHERE
  index_rec_count_null * 4 > index_rec_count --   NULL-
ORDER BY
  1, 2;
