分散型ホイールセットレジストリ：Hyperledger Fabricの使用経験

こんにちは、私はRRP KP（ホイールセットのライフサイクルを監視するための分散データレジスタ）のプロジェクトチームで働いています。ここでは、テクノロジーによって課せられた条件下でこのプロジェクトの企業ブロックチェーンを開発したときのチームの経験を共有したいと思います。ほとんどの部分でHyperledger Fabricについて説明しますが、ここで説明するアプローチは、許可されたブロックチェーンに当てはめることができます。私たちの研究の最終的な目標は、最終製品が使いやすく、メンテナンスが難しくないように、企業のブロックチェーンソリューションを準備することです。

発見や予期しない解決策はなく、ユニークな開発についてもここでは取り上げません（私にはそれらがないためです）。私の控えめな経験を共有し、「可能だった」ことを示し、おそらく、良い意思決定とあまり良くない意思決定を行う他の人の経験についてコメントで読みたいと思います。

問題：ブロックチェーンはまだスケーリングされていません


今日、多くの開発者の努力は、ブロックチェーンを本当に便利なテクノロジーにすることを目的としており、美しいラッパーの時限爆弾ではありません。州のチャネル、楽観的なロールアップ、プラズマ、およびシャーディングが毎日行われる場合があります。いつか。または、TONが打ち上げを6か月間遅らせ、次のプラズマグループが存在しなくなる可能性があります。私たちは別のロードマップを信じて、その夜の素晴らしいホワイトペーパーを読むことができますが、ここで私たちは自分たちが持っているもので何かをする必要があります。たわごとを成し遂げる。

現在のプロジェクトで私たちのチームに設定されたタスクは、一般的に次のようになります。信頼関係を構築したくない多くのエンティティが数千に達しています。特別なパフォーマンス要件なしで通常のPCで機能し、集中会計システムと同等のユーザーエクスペリエンスを提供するDLTでこのようなソリューションを構築する必要があります。ソリューションの基礎となるテクノロジーは、データの悪意のある操作の可能性を最小限に抑える必要があります。これがブロックチェーンがここにある理由です。

ホワイトペーパーとメディアのスローガンは、次の開発により、毎秒数百万のトランザクションが可能になると約束しています。本当に何ですか？

Mainnet Ethereumは、現在30 tpsで実行されています。すでにこのため、それを企業のニーズに適したブロックチェーンとして認識することは困難です。許可されたソリューションの中には、2000 tps（Quorum）または3000 tps（Hyperledger Fabric、出版物は少し小さいですが、ベンチマークが古いコンセンサスエンジンで実行されたことを考慮する必要があります）を示すベンチマークが知られています。最悪の結果である20,000 tpsをもたらさない、Fabricを根本的に改訂する試みがありましたが、これは今のところ、安定した実装を待っている学術研究にすぎません。ブロックチェーン開発者の部門を維持する余裕のある企業がそのような指標に耐える可能性は低いです。しかし、問題はスループットだけではなく、まだ待ち時間があります。

待ち時間


トランザクションが開始されてからシステムによる最終承認までの遅延は、検証と順序付けのすべての段階を通過するメッセージの速度だけでなく、ブロックの形成のパラメーターにも依存します。ブロックチェーンで1,000,000 tpsの速度でコミットできても、488 MBのブロックを形成するのに10分かかる場合でも、それは私たちにとってより簡単になりますか？

Hyperledger Fabricのトランザクションライフサイクルを詳しく見て、時間が費やされている時間と、ブロック形成のパラメーターにどのように関連しているかを理解しましょう。


ここから取得：hyperledger-fabric.readthedocs.io/en/release-1.4/arch-deep-dive.html#swimlane

（1）クライアントはトランザクションを生成し、それを推奨ピアに送信します。後者はトランザクションをシミュレートし（チェーンコードによって行われた変更を現在の状態に適用しますが、元帳にはコミットしません）、RWSetを取得します-キー名、バージョン、およびCouchDBのコレクションから取得した値（ 2）承認者が署名されたRWSetをクライアントに送り返す、（3）クライアントは必要なすべてのピア（承認者）の署名を確認してからトランザクションを注文サービスに送信するか、検証せずに送信する（検証は後で行われる）、注文サービスはブロックを形成し、（ 4）承認者だけでなく、すべてのピアに返信します。ピアは、読み取りセットの鍵バージョンがデータベースのバージョン、すべての承認者の署名と一致することを確認し、最後にブロックをコミットします。

しかし、それだけではありません。 「注文がブロックを形成する」という言葉は、トランザクションの順序だけでなく、リーダーからフォロワーへの、およびその逆の3つの連続したネットワーク要求も隠します。リーダーはログにメッセージを追加し、フォロワーに送信し、後者はログに追加し、レプリケーションの確認をリーダーに送信し、リーダーはメッセージをコミットします、コミット確認をフォロワーに送信し、フォロワーはコミットします。ブロックを形成するサイズと時間が短いほど、注文サービスがコンセンサスを確立する必要がより頻繁になります。Hyperledger Fabricには、ブロック形成の2つのパラメーターがあります。BatchTimeout-ブロック形成の時間とBatchSize-ブロックサイズ（トランザクション数とブロック自体のサイズ（バイト単位））。パラメータの1つが制限に達するとすぐに、新しいブロックが解放されます。保証ノードが多いほど、時間がかかります。したがって、BatchTimeoutとBatchSizeを増やす必要があります。ただし、RWSetsはバージョン管理されているため、ブロックを作成するほど、MVCCが競合する可能性が高くなります。さらに、BatchTimeoutの増加に伴い、UXは壊滅的に低下しています。これらの問題を解決するための次のスキームは合理的かつ明白です。

ブロックの確定を待つのを避け、トランザクションのステータスを追跡する機能を失わない


形成時間とブロックサイズが長いほど、ブロックチェーンのスループットは高くなります。もう1つは直接フォローしませんが、RAFTでの合意形成には、リーダーからフォロワーへ、およびその逆への3つのネットワーク要求が必要であることを忘れないでください。ノードの数が多いほど、時間がかかります。ブロックのサイズと形成時間が短いほど、そのような相互作用は多くなります。エンドユーザーのシステム応答の待機時間を増やすことなく、形成時間とブロックサイズを増やす方法は？

まず、同じバージョンの異なるRWSetが含まれている可能性がある大きなブロックサイズによって引き起こされるMVCCの競合を何らかの方法で解決する必要があります。明らかに、クライアント側では（ブロックチェーンネットワークに関連して、これはバックエンドである可能性があり、私はそれを意味します）、MVCC競合ハンドラーが必要です。これは、個別のサービスか、再試行ロジックを使用したトランザクショントリガー呼び出しに対する通常のデコレータのいずれかです。

再試行は指数関数的に実装できますが、レイテンシは指数関数的に低下します。したがって、特定の小さな制限内のランダム化された再試行、または永続的な再試行のいずれかを使用する必要があります。第1の実施形態で起こり得る競合に注意して。

次のステップは、システムとのクライアントの対話を非同期にして、15、30、または10,000,000秒待機しないようにすることです。これをBatchTimeoutとして設定します。ただし、同時に、トランザクションによって開始された変更がブロックチェーンに書き込まれる/書き込まれないことを確認する機会を保存する必要があります。
データベースを使用して、トランザクションのステータスを保存できます。使いやすさの点で最も簡単なオプションはCouchDBです。データベースにはすぐに使用できるUI、REST APIがあり、レプリケーションとシャーディングを簡単に構成できます。 Fabricが世界の状態を保存するために使用する同じCouchDBインスタンスに、別個のコレクションを作成するだけで済みます。この種のドキュメントを保存する必要があります。

{
 Status string //  : "pending", "done", "failed"
 TxID: string // ID 
 Error: string // optional,   
}


このドキュメントは、トランザクションがピアに転送される前にデータベースに書き込まれます。これが何かを作成する操作である場合、エンティティIDがユーザーに返され（同じIDがキーとして使用されます）、ピアからの関連情報を受信すると、ステータス、TxID、およびエラーフィールドが更新されます。



このスキームでは、ユーザーはブロックが最終的に形成されるのを待たずに、画面上の回転ホイールを10秒間見ます。システムから瞬時に応答を受け取り、作業を続けます。

トランザクションステータスの保存にはBoltDBを選択しました。これは、メモリを節約する必要があり、スタンドアロンデータベースサーバーとのネットワークインタラクションに時間を費やしたくないためです（特に、プレーンテキストプロトコルを使用してこのインタラクションが行われる場合）。ちなみに、CouchDBを使用して上記のスキームを実装するか、単に世界の状態を保存します。いずれにしても、CouchDBへのデータの保存方法を最適化することは理にかなっています。デフォルトでは、CouchDBでは、bツリーノードのサイズは1279バイトです。これは、ディスクのセクターサイズよりもはるかに小さいため、ツリーの読み取りと再調整の両方で、より多くの物理ディスクアクセスが必要になります。最適なサイズはAdvanced Format標準に準拠しており、4キロバイトです。最適化のために、btree_chunk_sizeパラメータを4096に設定する必要がありますCouchDB構成ファイル内。BoltDBの場合、そのような手動による介入は必要ありません。

2つの新しいアクションがスキームに追加されました：（1）APIのリクエストを受信した後、トランザクションを呼び出すために必要なパラメーターを含むメッセージがキューに入れられ、クライアントはトランザクションがシステムによって受け入れられたというメッセージを受信します、（2）バックエンドは設定で指定された速度でデータを読み取りますキューから; トランザクションを開始し、ステータスストアのデータを更新します。
これで、形成時間とブロック容量を好きなだけ増やして、ユーザーからの遅延を隠すことができます。

結論
