偉大なエジプトのファイアウォール

世界的にパンデミックが蔓延しており、人々はトイレットペーパーやソバを産業規模で購入しており、ほとんどのIT企業は従業員を遠隔地に移動させています。私の雇用主もそうでした-ドイツの準統計局。

基本的には問題ありませんでしたが、1か月前の従業員の1人で、すべてがまだ怖くないようでしたが、休暇でエジプトの親戚に行き、国境閉鎖のため無事に立ち往生しました。ええと、彼女は健康で、ラップトップを使って作業しているラップトップです-彼女は隔離され、VPNを介して作業しています。 1週間で2回... 3週目でVPNの接続が停止しました。最初の行のサポートは、再起動のように一般的なものをチェックしました-それは助けにはなりませんでした。 2行目が診断を開始しました。接続はTLSハンドシェイク段階で永続的なタイムアウトになります。ローカルファイアウォールを無効にした-それは助けにはならなかった。私たちは別の車を試しました-それは機能しません。別のプロバイダーが機能しない。この時点で、サポートチームはあきらめ、「ネットワークマネージャーは責任がある」という古き良き原則に従って、問題を喜んで私に押し込みました。

サーバーログを調べます。最初のパケットに応答した後、サーバーログに到達しようとする試みは見られません。おかしいとかなりおなじみ。私は従業員に電話しましたが、彼らが一般的な人権、特にインターネットの自由にどのように対処しているかに興味がありました。物事は悪い、インターネットはそれらをブロックしているのでラクダがしゃっくり、Roskomnadzorは緊張して傍観者を吸い込んでいると彼は言います。うん...すぐにグーグルで見ると、2017年以降、エジプトで同様のVPNの問題がたくさん寄せられています。写真を完成させるために、私は従業員が近年2週間以上故国に滞在しているかどうか尋ねます-いいえ、彼女はそうではありませんでした。パズルが形になり始めます。

無料のホワイトIPで企業VPNサーバーのコピーを作成します-接続はありません。期待された。
ポートを変更します-接続がありません。これは悲しいです。

プロトコルを変更します-接続はありません。パズルが発達しました-私たちの目の前には、素晴らしい中国のファイアウォールのようなDPIがあります。

従業員は悲しい、上司は露骨に「あなたはロシアのハッカーです、何かをしてください」と尋ねます。さて... Darknetの重砲を発見し、obfsproxyを回転させます。

サーバー（CentOS 7）の場合、次のようになります。

~ sudo pip install virtualenv
~ cd /etc/openvpn && virtualenv venv && source venv/bin/activate
~ sudo pip install obfsproxy
~ sudo -u openvpn /etc/openvpn/venv/bin/python /etc/openvpn/venv/bin/obfsproxy obfs3 --dest=127.0.0.1:1194 server 1.2.3.4:49416

次のようなクライアント（MacOS）の場合：

~ brew install pip
~ pip install pyopenssl obfsproxy
~ obfsproxy obfs3 socks 127.0.0.1:8443

クライアントのOpenVPN構成で、以下を追加します。

socks-proxy-retry
socks-proxy 127.0.0.1 8443

利益 obfsproxyラッパーのOpenVPNがローカルシグネチャ検出アルゴリズムによって検出されず、セッションが開始され、pingが送信され、トラフィックが実行され、従業員は満足しています。この「明白な」方法でオートロードするobfsproxyクライアントパーツを追加するだけです（ポピーは嫌いです）。私は囚人に安心して別れを告げ、「この問題はこのように解決されますが、安定性を保証することはできません。この回避策は他に方法がない場合にのみ使用できます」という精神でサポートに手紙を書きます。

どうやら、エジプトには特に狡猾なDPIが必要な場所があり、最初は新しい加入者への通信や新しいホストへのトラフィックをブロックせず、「観光客」の条件付きカテゴリを参照しています。そして、一定のタイムアウトが経過すると、ユーザーは「自分のもの」として分類され、喜んでトラフィックをカットして地元の王様を喜ばせます。
