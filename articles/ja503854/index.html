ページ構成のリダイレクト


このオプションを使用すると、Jiraのページへのリダイレクトをインテリジェントに管理できます。ベンダーからの情報はこちらから入手できます。実を言うと、このオプションが必要な理由とその使用方法を理解するのに何時間もかかったので、このオプションについて例を挙げて説明します。
ここに例があります。

Jiraで、毎月、すべての従業員がINVプロジェクトでチケットの形でお金を受け取るための口座に入るとします。



ご覧のとおり、Alexey Matveevは3月と4月に2つのアカウントを開設しました。

しかし、人事部門はJiraではなく一部の外部システムで機能しており、この外部システムには各従業員用のページがあり、Jiraにリンクがあり、最後のアカウントを持つ従業員のチケットにつながります。

このリンクはどのように見えますか？

もちろん、このようにリンクできますhttp：// localhost：2990 / jira / browse / INV-2。しかし、INV-2は4月の最後のスコアになります。 5月には別のアカウントのチケットがあります。人事部門の社員のページで毎月リンクを変更することもできますが、複雑すぎます。もっと簡単なものが必要です。

そして、リダイレクトページ構成だけが私たちの助けになります。

設定しましょう。

人事部門のシステムでは、リンクは次のようになります。

http://localhost:2990/jira/plugins/servlet/kredi?userName=Alexey+Matveev

ここではチケットではなく、必要なチケットをリダイレクトするplugins / servlet / kredi servletを参照しています。完全なユーザー名をパラメーターとして渡し、このパラメーターに基づいて、サーブレットが目的のチケットにリダイレクトします。
リンクを作成しました。ここでロジックを作成する必要があります。そして、Redirect Page Configurationメニューでロジックを実行するだけです。メニューを開き、次のコードを入力します。

string userName = argv["userName"];

if (isNull(userName)) {
    return "/plugins/servlet/kredierror?customErrorTitle=User Not Provided&customErrorMessage=Provide user with userName parameter";
}
if (isNull(getUserByFullName(userName))) {
    return "/plugins/servlet/kredierror?customErrorTitle=User Not Found&customErrorMessage=User " + userName + " not found";
}
string [] k = selectIssues("project = INV and reporter = " + getUserByFullName(userName).username + " order by \"Invoice Date\" desc");
if (size(k) == 0) {
   return "/plugins/servlet/kredierror?customErrorTitle=Invoice Not Found&customErrorMessage=No invoices for "+ userName; 
}
return "/browse/" + k[0];

コードは次のことを行います。

  • ユーザーのフルネームを含むuserNameパラメータの値を取得します。
  • パラメータが渡されていることを確認し、渡されていない場合は、エラーページにリダイレクトします。
  • そのフルネームのユーザーがJiraにログインしていることを確認します。そうでない場合は、エラーページにリダイレクトします。
  • このユーザーのアカウントを持つ最後のチケットを選択します。チケットが見つからない場合は、エラーページにリダイレクトします。
  • 最後のカウントでチケットにリダイレクトします。

Sil構成


このメニューでは、SILを構成できます。ベンダーからの情報はここにあります。



SILホームディレクトリでは、すべてのスクリプトを保存するディレクトリを指定できます。デフォルトはJIRA_HOME / silprogramsです。

Charsetを使用すると、スクリプトを格納するエンコーディングを決定できます。エンコードはUTF-8のままにしておくことをお勧めします。なぜこの設定が必要になるのかよくわかりません。

S ILキャッシュサイズを使用すると、コードがJavaバイトコードに変換され、SILキャッシュに格納されているスクリプトの数を指定できます。

なぜSILキャッシュが必要なのですか？

以下はサンプルのSILコードです。

string message = "My message";
runnerLog(message);

このコードを実行するには、Javaバイトコードに変換する必要があります。SILには、構造、関数、ループ、条件、代入など、多数の構成要素が含まれているため、これらすべてをJavaバイトコードに変換するには、時間が必要であり、SILキャッシュを使用するだけでこの時間を節約できます。コードは一度Javaバイトコードに変換され、キャッシュに保存されます。このコードを再度実行する必要がある場合、既製のJavaバイトコードがSILキャッシュから取得されます。

したがって、何千ものSILスクリプトがある場合、キャッシュのサイズを増やすことは理にかなっています。

Asynchronous Runner


ベンダーからの情報はここで読むことができます：

スレッドはSILのスレッド数を設定します。デフォルトでは、スレッドの数は10です。つまり、一度に実行されるスクリプトは10のみです。残りは並んでいて、スレッドの1つが解放されるまで待機します。多数のスクリプトを同時に実行している場合は、このパラメータを増やします。存続可能

時間（TTL）は、スレッドの存続時間を設定します。デフォルト値は1時間です。つまり、スクリプトが1時間以上実行された場合、そのスクリプトは強制終了され、スクリプトの結果は、この時間内に管理できたものだけになります。実行時間の長いスクリプトがある場合は、このパラメータを増やします。

チェックポイント間隔TTL時間パラメーターよりも長く実行されるスクリプトがあるかどうかをSILがチェックする頻度を決定します。そして、そのようなスクリプトが見つかった場合、それらは殺されます。

また、この画面では、現在実行中のすべてのスクリプトを確認できます。

カスタムフィールドのマッピング


このメニュー項目では、カスタムフィールドタイプのSIL値タイプへのマッピングを指定できます。ベンダーからの情報はこちらで読むことができます。



たとえば、カスタムチェックボックスフィールドを見てください。このフィールドの値はタイプ文字列[]にマップされます。つまり、チェックボックスタイプのフィールドから値を読み取りたい場合は、次のコードを記述する必要があります。

string[] value =  #{My Checkbox Field};

値を割り当てるには、次のようなコードを使用します。

string[] value =  {"value1", "value2"};
#{My Checkbox Field} =  value;

このフィールドは標準フィールドであるため、マッピングを変更できません。Cprimeはすでに正しいマッピングを示しています。

次に、カスタムフィールドグリッドのタイプを見てみましょう。このタイプは、Table Grid Next Generationプラグインから取得しました。Cprimeはそのままではこのフィールドのマッピングを提供しないため、デフォルトのマッピングが割り当てられています。デフォルトのマッピングは文字列です。ただし、デフォルトのマッピングを別のマッピングに変更することができます。



場合によっては、組み込みのマッピングでは不十分であり、この場合、マッピングを使用してSILの拡張機能を作成できます。これを行う方法の詳細については、こちらをご覧ください。
