ASP.NET MVC-Entity Framework、MySQL、およびDependency Resolverを使用してリポジトリを選択

レガシーテクノロジー
: ASP.NET MVC . ASP.NET Core. , .

ASP.NET MVCとMySQLに関する以前の記事 を少し拡張することにしました。これは、ほぼ標準的なORMエンティティフレームワーク（EF）ではなく、ADO.NETを介してDBMSに直接アクセスすることで、ASP.NET MVCでMySQLを操作することに関するものでした。そして、このアクセス方法の実装が与えられました。また、この方法は古く、使用が推奨されていませんが、たとえば、高負荷のアプリケーションや、ORMが正しく機能するSQLクエリを生成できない状況に直面している場合など、便利な場合があります。また、ORMとADO.NETの両方を使用して、アプリケーションで両方の方法を組み合わせることができる場合もあります。その結果、私はアプリケーションを完成させることを考え、決定しました。EntityFrameworkのリポジトリ実装をアプリケーションに追加し、Dependency Resolverを使用して、アプリケーションのパラメーターに依存するように選択します。

すべてのコードはこのアドレスで取得できます。このコードの下には、前のプロジェクトに関連する小さなリンクと説明が部分的に表示されます。そして、あなたはアプリケーションを見ることができます。

プロジェクトを変更


1. Entity FrameworkをMySQLで使用するには、MySQL.Data.EntityFrameworkライブラリをインストールする必要があります（もちろん、別のライブラリを用意することもできます。これは、OracleのMySQLの所有者です）。MySQL.DataとEntityFramework自体


をプルします。以下の変更が行われましたweb.configファイル：

<entityFramework>
  <providers>
      <provider invariantName="MySql.Data.MySqlClient" type="MySql.Data.MySqlClient.MySqlProviderServices, MySql.Data.EntityFramework, Version=8.0.19.0, Culture=neutral, PublicKeyToken=c5687fc88969c44d" />
  </providers>
</entityFramework>

興味深いの競合が発生した とMySQL.Data以来- MySQL.Data.EntityFrameworkはのバージョンに必要なのMySQL.Data 8.0.19少なくとも、それが更新されましたが...そしてプロジェクトが動作を停止しました。エラーが表示され始めました：



«Ubiety.Dns.Core» . . , , . ( HRESULT: 0x80131045)

どうやら、署名されていないアセンブリUbiety.Dns.CoreがMySQL.Data 8.0.19のバージョンに追加されました。また、プロジェクトにNugetを通じてこのコンポーネントを含める必要がありました。エラーはなくなりました。

2.さらに、依存関係の実装を実装するために、依存関係（DI）を実装するためのコンテナーであるプロジェクトNinjectを追加します。

3.プロジェクトの構造を少し変更します。リポジトリファイルを別のリポジトリディレクトリに配置し、その中にADO.NETサブディレクトリを作成します（既存のLanguagesRepository.csファイルとUsersRepository.csファイルをそこに移動します）とEF（ここにEntity Frameworkのリポジトリファイルがあります）。

4.ファイルweb.configに加えて、セクションappConfigオプションがアプリケーションに追加されました<add key="ConnectionMethod" value="ADO.NET" />。アプリケーションは、「Entity Framework」または「ADO.NET」の2つの値を取ります。このパラメーターへのリンクをBase.csファイルに追加しました：

public static string ConnectionMethod
{
    get
    {
        return System.Configuration.ConfigurationManager.AppSettings["ConnectionMethod"];
    }
}

Entity FrameworkとMySQL-リポジトリ


クラス を含むDbContext.csファイルをRepository \ EF ディレクトリに追加しますEFDbContext。

public class EFDbContext : DbContext {
    public EFDbContext() : base(Base.ConnectionString)
    { }
    public DbSet<UserClass> Users { get; set; }
    public DbSet<LanguageClass> Languages { get; set; }
}

その中で、我々は、使用するDBMS接続文字列とデータセットを決定UsersしてLanguages。

次のクラスを使用して、LanguagesRepository.csファイルを追加しますLanguagesRepositoryEF。

public class LanguagesRepositoryEF : ILanguagesRepository
{
    private EFDbContext context = new EFDbContext();
 
    public IList<LanguageClass> List()
    {
        return context.Languages.OrderBy(x => x.LanguageName).ToList();
    }
}

そして、次のクラスを持つUsersRepository.csファイルUsersRepositoryEF：

public class UsersRepositoryEF : IUsersRepository
{
    private EFDbContext context = new EFDbContext();
 
    public IList<UserClass> List()
    { 
        return context.Users.ToList();
    }
 
    public IList<UserClass> List(string sortName, SortDirection sortDir, int page, int pageSize, out int count)
    {
        count = context.Users.Count();
        if (sortName != null) return context.Users.OrderByDynamic(sortName, sortDir).Skip((page - 1) * pageSize).Take(pageSize).ToList();
        else return context.Users.OrderBy(o => o.UserID).Skip((page - 1) * pageSize).Take(pageSize).ToList();
    }
 
    public bool AddUser(UserClass user)
    {
        user.Language = context.Languages.Find(user.Language.LanguageID);
        if (user.Language != null && context.Users.Add(user) != null)
        {
            try { context.SaveChanges(); }
            catch (System.Exception ex) {}
        }
        return user.UserID > 0;
    }
 
    public UserClass FetchByID(int userID)
    {
        UserClass user = null;
        try { user = context.Users.Find(userID); }
        catch (System.Exception ex) { }
        return user;
    }
 
    public bool ChangeUser(UserClass user)
    {
        bool result = false;
        user.Language = context.Languages.Find(user.Language.LanguageID);
        if (user.Language != null)
        {
            UserClass olduser = context.Users.Find(user.UserID);
            if (olduser != null)
            {
                olduser.Email = user.Email;
                olduser.Loginname = user.Loginname;
                olduser.Language = user.Language;
                olduser.SupporterTier = user.SupporterTier;
                try { result = context.SaveChanges() > 0; }
                catch (System.Exception ex) { }
            }
        }
        return result;
    }
 
    public bool RemoveUser(UserClass user)
    {
        bool result = false;
        UserClass olduser = context.Users.Find(user.UserID);
        if (olduser != null) context.Users.Remove(olduser);
        try { result = context.SaveChanges() > 0; }
        catch (System.Exception ex) { }
        return result;
    }
}

ファイルサイズがADO.NETのファイルサイズよりも明らかに短いことがわかります— ORMは私たちのために汚い仕事をします—それはそれ自体でSQLクエリを作成します。

しかし、ADO.NETの実装に組み込まれたいくつかの点に出くわしましたが、EFでは機能しませんでした。

私はに変更しなければならなかった最初の事UserClass.csのファイル（内ドメインディレクトリを）：テーブルリンクの正常な動作のために別のフィールドを追加するUsersとLanguages：

[HiddenInput(DisplayValue = false)]
public int? LanguageID { get; set; }

そして2番目-MySQLタイプのフィールドはEnumEFで機能しないことがわかりました。最も可能性が高いのは、コード内の列挙が整数値であるためですが、データベースからEFまでの値はテキストとして読み取られます（MySQLからのクエリでenum型のフィールドの値を読み取る場合、MySQLはこの列挙のテキスト値のみを返します）。そして、ADO.NETのバージョンでデザインを使用してこれを回避できるCAST(u.SupporterTier AS UNSIGNED) as SupporterTier場合、EF を使用すると、そのような変態は私にとって乗り越えられないことが判明しました。さて、テクノロジーコードファーストタイプのフィールドEnumはINT タイプのフィールドを生成するため、データベース内のフィールドタイプを変更する必要がありましたSupporterTier。

CHANGE COLUMN `SupporterTier` `SupporterTier` INT(4) UNSIGNED NOT NULL DEFAULT '1' ;

アプリケーションパラメータを使用してリポジトリを選択


教科書に書かれているように、コンストラクタを介して実装を使用します。最初に、共有リポジトリのインターフェイスを作成する必要があります。次の内容で、LanguagesRepository.csファイルをリポジトリディレクトリに作成します。

public interface ILanguagesRepository
{
    IList<LanguageClass> List();
}

そして、内容を含むUsersRepository.csファイル：

public interface IUsersRepository
{
    IList<UserClass> List();
 
    IList<UserClass> List(string sortName, SortDirection sortDir, int page, int pageSize, out int count);
 
    bool AddUser(UserClass user);
 
    UserClass FetchByID(int userID);
 
    bool ChangeUser(UserClass user);
 
    bool RemoveUser(UserClass user);
}

さて、これらのインターフェースから対応するクラスを継承します：

public class LanguagesRepositoryADO : ILanguagesRepository
public class UsersRepositoryADO : IUsersRepository
public class LanguagesRepositoryEF : ILanguagesRepository
public class UsersRepositoryEF : IUsersRepository

まあ、UsersControllerコントローラーで、これらのインターフェイスで動作するように追加します。

private ILanguagesRepository repLanguages;
private IUsersRepository repUsers;
 
public UsersController(ILanguagesRepository langsParam, IUsersRepository usersParam) 
{ 
    repLanguages = langsParam;
    repUsers = usersParam;
}

そしてコントローラーで、これらのクラスのオブジェクトにアクセスする場所をそれぞれオブジェクトrepLanguagesとrepUsersに変更します。ただし、当然ながら不便なコントローラコンストラクタを介してリポジトリクラスのインスタンスを渡す必要があります。これを回避するには、Dependency Resolver（DR）のような強力な魔術が必要です。このために、Ninjectを使用

します。メソッドのGlobal.asax.csファイルにDRを登録しますApplication_Start。

DependencyResolver.SetResolver(new NinjectDependencyResolver());

NinjectDependencyResolver.cs ファイルを次のクラスでインフラストラクチャディレクトリに作成しますNinjectDependencyResolver（インターフェイスから継承IDependencyResolver）。

public class NinjectDependencyResolver : IDependencyResolver 
    { 
        private IKernel kernel; 
        public NinjectDependencyResolver() 
        { 
            kernel = new StandardKernel(); 
            AddBindings(); 
        } 
        public object GetService(Type serviceType) 
        { 
            return kernel.TryGet(serviceType); 
        } 
        public IEnumerable<object> GetServices(Type serviceType) 
        { 
            return kernel.GetAll(serviceType); 
        } 
        private void AddBindings() 
        { 
            if (Domain.Base.ConnectionMethod == "Entity Framework")
            {
                kernel.Bind<ILanguagesRepository>().To<LanguagesRepositoryEF>();
                kernel.Bind<IUsersRepository>().To<UsersRepositoryEF>();
            }
            else
            {
                kernel.Bind<ILanguagesRepository>().To<LanguagesRepositoryADO>();
                kernel.Bind<IUsersRepository>().To<UsersRepositoryADO>();
            }
        }
    }

また、DBMSのどのメソッドを使用するか（直接、ADO.NETまたはEntity Frameworkを介して）が決定される唯一の場所AddBindingsは、クラスのメソッドであることがわかりNinjectDependencyResolverます。それがどのように機能するかわからない場合の本当の魔法。

この方法ではAddBindings、「ConnectionMethod」アプリケーションパラメータの値に応じて、インターフェイスの束が起こるILanguagesRepositoryとIUsersRepository、特定のクラスとの具体的なインタフェースメソッドを実装します。アプリケーションを開始したとき、DRをクラスのオブジェクトとして登録NinjectDependencyResolverし、クラスでリポジトリインターフェイスのバインディングを特定のクラスに指定したので、MVCフレームワークにコントローラーオブジェクトの作成を要求するとUsersController、Ninjectはクラスを分析するときに、インターフェイスの実装ILanguagesRepositoryとIUsersRepository 特定のクラスをインスタンス化し、それらをコントローラーコンストラクターに（DRおよびMVCフレームワーク経由で）渡します。

合計


アプリケーションは、ORMエンティティフレームワークを介してDBMSにアクセスする方法もサポートするようになりました。同時に、ADO.NETを介したアクセスメソッドは廃止されておらず、アプリケーションがパラメーターによって起動されるときに選択されます。このため、Ninjectライブラリを使用して、コントローラーのコンストラクターを通じて依存関係注入メソッドを使用しました。
