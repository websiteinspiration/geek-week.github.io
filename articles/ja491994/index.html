CocoapodsからSwift Package Managerへの移行



Cocoapodsは、iOSで最も人気のある依存関係マネージャーと見なされています。近年、AppleはネイティブのSwift Package Manager（SPM）依存関係マネージャーの開発に取り組んでいます。

当初、その使用はサーバー側のSwiftまたはターミナルアプリケーションでのみ可能でした。 SPMはこのようなアプリケーションで実行および変更され、コミュニティはその作業に精通し、Appleチームはベータテスターを受け取りました。

Xcode 11のリリースに伴い、SPMはiOSの開発の世界に入ってきました。現在、それはすでに使用可能な本格的なツールですが、これまでのところ制限があります。

現在のバージョンでは、SPMはリソースをサポートしていません（SE-0271を待っています）。私たちの国では、各モジュールは、プロジェクトに接続できるアトミックな自給自足依存関係であるため、リソース（ローカリゼーション、アセット）が必要です。

それまでの間、移行の準備をすることができます。それがどれほど難しいか、自動化できるかどうか、どのような問題が発生するかを理解することです。

CocoapodsからSPMに移行する理由


  • ネイティブ性とエコシステムへの統合。XcodeはすでにSPMパッケージの作成または追加を提案しています。
  • Cocoapodsとは異なり、依存関係プロジェクトで作業するためにワークスペースを用意する必要がなくなりました。
  • 新しい依存関係を追加した後、ポッドインストールを実行してプロジェクト全体を再ビルドする必要はありません。
  • ローカルの依存関係を使用すると便利です。フォルダー構造は変更されず、より速く再構築されます。

プロジェクトに多くの依存関係がある場合、移行プロセスは1日で完了することができません。Podfile内の依存関係の数は、依存関係の総数と同じではありません。1つの依存関係にはさらに10個の依存関係が存在する可能性があるためです。依存関係はそれほど多くないが、依存関係を開発している場合でも、プライベートリポジトリであっても、迅速に移行しない可能性があります。幸い、SPMはCocoapodsとうまく調和しており、移行プロセスを繰り返すことができます。主なことは、ニュアンスを考慮することです：

  • SPMとCocoapodsの依存関係の間に交差点があってはなりません。
  • 移行はコンポーネント開発プロセスに統合する必要があります。コンポーネントが開発中の場合は、CocoapodsとSPMの両方に対してバージョンアップグレードを実行する必要があります。
  • チームに問題はないはずです。つまり、すべての依存関係のインストールを単一のcliコマンドで実装する必要があります。
  • 移行はCI CDに組み込む必要があります。プロジェクトは、すべての構成で、ローカルとCIエージェントの両方でビルドする必要があります。

どこから始めますか？


まず、依存関係の完全なリストを収集します-それはPodfile.lockに表示されます。SPMReadyなどのユーティリティを使用できます。



次に、それらを関係によってグループ化します。たとえば、ほとんどすべてのコンポーネントはSMENetworkに依存しています。つまり、SMENetwork自体が移行するまで、コンポーネントをSPMに移行できません。

次に、ほとんどのモジュールが依存する1つのモジュール、SMENetworkを選択します。
最後に、このモジュールのリポジトリに移動し、フォルダにチェックインして、このフォルダのルートにモジュールを作成します。
