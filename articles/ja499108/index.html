序文


jvm言語では、リンクの操作を非表示にするのが通例です。つまり、ほとんどの場合、参照データ型を操作するため、アンパサンド（＆）を非表示にすることにしました。

ラスタには明示的なリンクがあり、たとえば整数-`＆i32`へのリンクがあります。リンクは` * `を介して逆参照できます。リンクへのリンクもあり、2回逆参照する必要があります**。

イテレータ


コードを記述する場合、多くの場合、コレクションを条件（述語）でフィルタリングする必要があります。岩の中では、要素を取ることは次のようになります：

    val vec = Vector(1,2,3,4)
    val result = vec.filter(e => e % 2 == 0)

種類を見てみましょう：

  private[scala] def filterImpl(p: A => Boolean, isFlipped: Boolean): Repr = {
    val b = newBuilder
    for (x <- this)
      if (p(x) != isFlipped) b += x

    b.result
  }

`newBuilder`の詳細には触れずに、新しいコレクションが作成されていることは明らかです。古いコレクションを反復処理し、述語がtrueを返す場合は要素を追加します。コレクションが新しいという事実にもかかわらず、その要素は実際には最初のコレクションの要素へのリンクであり、突然これらの要素が変更可能になった場合、それらの変更は両方のコレクションに共通になります。

さあ、同じことを頑張ってやってみましょう。すぐに実際の例を示し、その違いを検討します。

    let v: Vec<i32> = vec![1, 2, 3, 4];
    let result: Vec<&i32> = v.iter().filter(|e| **e % 2 == 0).collect();

わあ、すごい？二重ポインター逆参照？ベクトルをフィルタリングするだけですか？ハード:(しかし、これには理由があります。

このコードが岩とどのように異なるかを選びましょう：

  1. ベクトルのイテレータを明示的に取得します（ `iter（）`）
  2. 述語関数では、何らかの理由で、ポインタを2回逆参照します
  3. `collect（）`を呼び出す
  4. また、通常の整数ではなく、参照型Vec <＆i32>のベクトルも生成されました

イテレータに戻ります。それらの間の「キャプチャ」の概念はIntoIterです。彼はコレクションを「飲み込み」、その要素を所持しています。コードでは、このアイデアは次のように反映されます。

coll_1で `into_iter（）`を呼び出すことにより、それをイテレータに「変換」し、前の例のように、すべての要素を吸収しました。その後、coll_1への呼び出しは、コンパイル中に借用チェッカーによって罰せられます。次に、 `collect`関数を使用してこれらの要素を収集し、新しいベクターを作成しました。イテレータからコレクションを収集するには、 `collect`関数が必要です。これを行うには、収集する対象のタイプを明示的に指定する必要があります。したがって、coll_2はタイプを明確に示します。

変更可能なリンクの数に制限を設定することにより！

ある時点での変更可能なリンクは、1つだけです（不変ではありません）。つまり、1つまたは複数の不変、または1つの可変です。コードは次のようになります。


// 
struct X {
    x: i32,
} 

fn test_borrow_checker() -> X {
    let mut first = X { x: 20 }; //  
    let second: &mut X = &mut first; //   
    let third: &mut X = &mut first; //    .        `second`        - .
//    second.x = 33;  //    ,             ,    
    third.x = 33;

    return first;
}

前の相対的な例の変更を見てみましょう。まず、構造に1つのフィールドを追加しました。変更が必要なため、変更が必要なためです。次に、 `mut`が変数の宣言に現れました` let mut first = ... `、これは、` val`や `var`のように、可変性に関するコンパイラーのマーカーです。 3番目に、すべてのリンクのタイプが「＆X」から「＆mut X」に変更されました（もちろん、見た目は巨大です。これには寿命がありません...）。リンクによって格納されている値を変更できます。

この場合、1つだけを使用するため、2つの可変リンクの存在が同時に許可されます。つまり、2番目のリンクは破棄され、何も変更されません。また、 `second`は、「3番目」を作成すると、すべてが正常になります。しかし、 `second.x = 33;`のコメントを外すと、2つの可変リンクが同時に存在し、ここから抜け出すことができないことが判明します-コンパイル時エラー。

イテレータ


したがって、3種類の送信があります。

  1. 吸収、借用、引っ越し
  2. リンク
  3. 可変リンク

各タイプには独自のイテレータが必要です。

  1. IntoIterは元のコレクションからオブジェクトを吸収します
  2. Iterはオブジェクトリンクで実行されます
  3. IterMutは可変オブジェクト参照で実行されます

問題が発生します-いつ使用するか。特効薬はありません。他人のコードや記事を読んで練習する必要があります。このアイデアを実証する例を示します。

学校があり、その中にクラスがあり、クラスの生徒がいるとします。


#[derive(PartialEq, Eq)]
enum Sex {
    Male,
    Female
}

struct Scholar {
    name: String,
    age: i32,
    sex: Sex
}

let scholars: Vec<Scholar> = ...;

たとえば、データベースにクエリを実行することで、小学生のベクトルを取得しました。次に、クラスの女の子の数を数える必要がありました。`into_iter（）`を介してベクターを「飲み込む」場合、カウント後、このコレクションを使用して男の子をカウントすることはできなくなります。


fn bad_idea() {
    let scholars: Vec<Scholar> = Vec::new();
    let girls_c = scholars
        .into_iter()
        .filter(|s| (*s).sex == Sex::Female)
        .count();

    let boys_c = scholars 
        .into_iter()
        .filter(|s| (*s).sex == Sex::Male)
        .count();
}

男の子をカウントするための行に「移動後にここで使用される値」というエラーが発生します。また、可変イテレータが役に立たないことも明らかです。それが、 `iter（）`であり、二重リンクで動作している理由です。


fn good_idea() {
    let scholars: Vec<Scholar> = Vec::new();
    let girls_c = scholars.iter().filter(|s| (**s).sex == Sex::Female).count();
    let boys_c = scholars.iter().filter(|s| (**s).sex == Sex::Male).count();
}

ここでは、国の潜在的な新入社員の数を増やすために、変更可能な反復子がすでに必要です。


fn very_good_idea() {
    let mut scholars: Vec<Scholar> = Vec::new();
    scholars.iter_mut().for_each(|s| (*s).sex = Sex::Male);
}

最後の質問は残っています-`filter`のリンクの二重逆参照はどこから来たのですか？実際のところ、述語は引数への参照を受け取る（それをキャプチャしないようにする）関数です。


    fn filter<P>(self, predicate: P) -> Filter<Self, P> where
        Self: Sized, P: FnMut(&Self::Item) -> bool,

述語はFnMut（大まかに言えば関数）であり、その（自己）アイテムへの参照を取得してboolを返します。すでにイテレータ `.iter（）`からのリンクがあったので、2番目はフィルタに現れました。イテレーター（ `into_iter`）に吸収されると、リンクの二重逆参照が通常の参照に変わります。
