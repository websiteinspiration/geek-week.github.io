すべてのOSトラフィックをTorでラップします



記事で説明されているものはすべて、GitHubで入手可能なToroxyツールとして実装されています。
最近、オンラインの匿名性は激しい議論の対象となっています。ローカルデバイスからのインターネットリソースへのアクセスに関するデータをさまざまなレベルで収集して、後でユーザーに対して使用できる（または可能性がある）ユーザーの「モデル」を構築できることは明らかです。したがって、ますます多くのアクティブなインターネットユーザーがプロキシと匿名化のメカニズムの必要性を確信していることは当然のことです。この点に関して、新しいVPNクライアントはますます増えていますが、実際にすべてが信頼できるわけではありません。すべてがそのまま機能するわけではなく、HTTPトラフィックのみが匿名化され、実装の品質が悪い、開発者はユーザーに関するデータをマージすることで罪を犯します。

この記事では、いくつかのソフトウェアコンポーネントから独自のUIツールを組み立てようとします。これにより、ローカルシステムのトラフィックを完全に匿名化し、作業のどの段階でも「リスニング」チャネルを介したリークを防ぐことができます。

私たちの主な目標は、既製のツールから信頼できるユーティリティを「構築」することです。明らかに、高品質のツールをゼロから作成するという考えには誤りが多いため、既製のコンポーネントを選択して正しく接続する方がより速く信頼性が高くなります！

ツールでできることは何ですか？

  1. ソースを確実にマスクするために、ターゲットシステムのすべてのトラフィックを中間ノード（できればいくつか）にリダイレクトします。
  2. 匿名性違反の可能性を追跡して修正し、UI通知を使用して報告する

ツールを作成するために選択したコンポーネント：

  • tor
  • iptables
  • python3
  • systemd

「Linux」と呼ばれるシェルにすべてのコンポーネントを混在させることで、最終的な目標の達成に役立つ価値のあるものを確実に得ることができます。

コンポーネント＃1：Tor


ツールインフラストラクチャの残りの部分が構築されるのは、このコンポーネントの周りです。 Torは、VPNクライアントの一部であるメカニズムを提供します—外部オブザーバーに対して匿名の中間ノードを介してトラフィックをラップするメカニズム（そのようなノードの標準構成では3）。

デフォルトでは、インストール後の標準のバッチリポジトリからのTorクライアントは、ソックス可能なすべてのクライアントを受け入れるポート9050のリスニングを開始します。問題は、システム内のsocks-trafficに加えて、このプロトコルで動作しないアプリケーションからの他のトラフィックが大量に存在する可能性があることです。これに関しては、まず、ローカルシステム内で、新しいネットワーク接続のためにウィンドウをTorネットワークに割り込む必要があります。これは、透過プロキシを上げるだけで簡単に行えますtorrc構成で：

/etc/tor/torrc
...
TransPort 9040
#           python
ControlPort 9051
...

UDPトラフィックには特別な注意が必要です。実際のところ、オニオンルーティングの原則は「ストリーム」の概念に基づいており、ご存じのように、TCPにのみ存在します。Torを介してUDPパケットを送信すると、応答パケットが戻りパスを見つけられないため、ターゲットシステムは応答を受信できなくなります。しかし、この特殊性にもかかわらず、DNSクエリを匿名化する機会がまだあります。これは、ご存知のように、UDPで実行され、同時に.onion解決を含みます。

/etc/tor/torrc
...
AutomapHostsOnResolve 1
DNSPort 53
...

これに関して、Torへのアクセスはループバック内で開かれています。

コンポーネント＃2：Iptables


私たちのタスクはシステム全体の外部オブザーバーからトラフィックの真のソースを隠すことであり、Torのウィンドウはすでに開いているため、このウィンドウですべてのトラフィックをラップするだけです。Linuxカーネルにバンドルされているシステムファイアウォールは、これに役立ちます。

#  tcp
iptables -t nat -A OUTPUT -p tcp --syn -j REDIRECT --to-ports $TRANS_PORT
#   udp (dns only)
iptables -t nat -A OUTPUT -p udp --dport 53 -j REDIRECT --to-ports 53
iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
#      
iptables -A OUTPUT -j REJECT
ip6tables -A OUTPUT -j REJECT

この段階で、すべての発信トラフィックを確実に偽装する実用的なユーティリティが得られますが、これは作業の半分にすぎません。

コンポーネント＃3：python +デスクトップ環境UI


毎回、コンソールから手動で構成する（bashスクリプトを実行する場合でも）のは非常に退屈な作業になるので、次の点で特に役立つ小さなユーティリティの作成を開始します。

  1. 自動構成
  2. Tor内でいつでも自分のアイデンティティを変更する
  3. iptablesルールの整合性を監視し、違反した場合は書き換えます
  4. 現在のID（IP）の追跡
  5. グラフィカルな通知による前の2つの段落の通知

ユーティリティは最初の起動時に必要なすべてのコンポーネントを独自にダウンロードし、その後の起動時に上記のようにTorをiptablesと組み合わせて構成します。

外部IPアドレスを変更する場合は、IPの変更を自動化するために最初に開いていたTor-9051サービスポートを操作します。

with Controller.from_port(port = 9051) as controller:
    controller.authenticate()
    controller.signal(Signal.NEWNYM)

整合性の追跡は、iptablesルール構造を定期的に読み取り、それらのSHA256合計をチェックすることにより、かなりひどく実装することができます（ひざ上で行います）。

def rulesOk(self):
    RULES_CHECKSUM_CMD = "{ iptables-save && ip6tables-save; } | sed s/\-\-uid\-owner\\\\s[0-9]\\\\+\\\\s//g | grep -viE '^#' | grep -viE '^\:' | sort | uniq | sha256sum | cut -d' ' -f 1"
    checkSum = getoutput(RULES_CHECKSUM_CMD).strip()
    alright = checkSum == Strings.RULES_CHECKSUM_CORRECT_HASH

    if not alright:
        rules = getoutput('iptables-save && ip6tables-save')
        self.lastSnapshotFileName = "/tmp/broken-rules-%s.log" % time.strftime("%d-%m-%Y_%I-%M-%S")
        open(self.lastSnapshotFileName, "w").write(rules)
        return False
     else:
         return True

また、予想されるチェックサムに不整合がある場合は、iptablesルールダンプを保存して/tmp/broken-rules-%d-%m-%Y_%I-%M-%S.log、さらに処理を進めることができます。それが判明した場合
rulesOk() == False
これにより、iptablesルールテーブルの書き換えが開始されます。

現在のIPを監視するには、IPクライアントを提供する外部リソースへの一定の参照を使用します（例：）ident.me。

最後に、DE UIを使用して、ルールの問題やIPの変更に関する問題を報告します。特にデーモンプロセスからUIを使用する場合、各グラフィカル環境はいくつかの点で一意ですが、ほとんどのLinuxシステムでは、Pythonから呼び出されるそのようなbashコードは通知を正常に表示します。

# root UI
eval "export $(egrep -z DBUS_SESSION_BUS_ADDRESS /proc/$(pgrep -u $LOGNAME gnome-session)/environ)"

export DISPLAY=:0
for USR in `ls /home && echo root`
do
    # ubuntu gnome + root UI
    export XAUTHORITY=/home/$USR/.Xauthority
    notify-send -u {0} '{1}' '{2}'

    # ubuntu parallels
    for UID in `ls /run/user/`
    do
        su $USR -c "DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/$UID/bus notify-send -u {0} '{1}' '{2}'"
    done
done

コンポーネント＃4：systemd


そして最後に、私たちは間違いなく1回限りのセットアップを行い、将来の安全性については考えたくないので、自動実行とサービスが役に立ちます。Linuxには、systemd、sysV、initなど、いくつかの標準デーモン管理サブシステムがあります。私たちのケースでは、設定の柔軟性のため、systemdを選びました。

前のステップで記述されたpythonスクリプトが「toroxy」と呼ば/usr/bin/れ、にあるとすると、デーモンを制御するための一定の柔軟性を備えた自動実行とその後の監視は次のようになります。

[Unit]
Description=Toroxy
After=network.target
StartLimitIntervalSec=0

[Service]
Type=simple
Restart=always
RestartSec=1
User=root
# service toroxy start
ExecStart=/usr/bin/toroxy service
# service toroxy stop
ExecStop=/usr/bin/toroxy stop
# service toroxy reload
ExecReload=/usr/bin/toroxy switch

[Install]
#   init 3,       UI     Tor
WantedBy=multi-user.target

ほとんどすべてが「産業」運用の準備ができています。信頼性を高めるためにツールに追加したい最後のタッチは、iptables-persistentを使用したシステム起動時のiptablesルールの自動初期化です（ご存知のように、再起動するとiptablesルールがリセットされます）。

iptables-save > /etc/iptables/rules.v4
ip6tables-save > /etc/iptables/rules.v6
netfilter-persistent start && netfilter-persistent save

結論


そこで、さまざまなコンポーネントの組み合わせから独自のツールを組み立てました。これにより、かなり高い信頼性で、ネットワーク上のLinuxユーザーの継続的な匿名性を提供できます。結論として、記事で説明されているものはすべて、GitHubで入手可能なToroxyツールとして実装されていると言うべきです。
