microconfig.ioでマイクロサービス構成を簡単に管理

マイクロサービスの開発とその後の操作における主な問題の1つは、インスタンスの適切で正確なチューニングです。私の意見では、新しいmicroconfig.ioフレームワークが役立ちます。これにより、アプリケーションをセットアップするための日常的なタスクのいくつかを非常にエレガントに解決できます。

多数のマイクロサービスがあり、それぞれに独自のファイル/設定ファイルが付属している場合、適切な巧妙さやロギングシステムがないと、それらの1つでエラーが発生する可能性が高く、ログシステムを見つけるのは非常に困難です。フレームワーク自体が設定する主なタスクは、重複するインスタンス設定を最小限に抑え、それによってエラーを追加する可能性を減らすことです。

例を考えてみましょう。yaml構成ファイルを使用した単純なアプリケーションがあるとします。どの言語のどのマイクロサービスでもかまいません。フレームワークをこのサービスに適用する方法を見てみましょう。

ただし、最初に、利便性を高めるために、最初にmicroconfig.ioプラグインをインストールして、Idea IDEに空のプロジェクトを作成します。



プラグインの起動構成を構成します。上のスクリーンショットのように、デフォルトの構成を使用できます。

私たちのサービスはorderと呼ばれ、次に新しいプロジェクトで同様の構造を作成



します：サービス名のフォルダーに設定ファイル-application.yamlを置きます。すべてのマイクロサービスは何らかの環境で起動されるため、サービス自体の構成を作成することに加えて、環境自体を記述する必要があります。これを行うには、envsフォルダーを作成し、作業環境の名前を含むファイルをそれに追加します。したがって、このパラメーターはプラグインの設定で設定されるため、フレームワークはdev環境でサービスの構成ファイルを作成します。dev.yaml

ファイルの構造は非常にシンプルです。

mainorder:
    components:
         - order

フレームワークは、グループ化された構成で機能します。私たちのサービスでは、メインオーダーグループの名前を選択します。フレームワークは、環境ファイルでそのような各アプリケーショングループを見つけ、対応するフォルダで見つけたすべての構成を作成します。注文

サービス設定ファイル自体では、これまで1つのパラメーターのみを示します。

spring.application.name: order

今すぐプラグインを実行し、それがプロパティで指定されたパスに従って、私たちのために私たちのサービスの必要な設定を生成します：



あなたがすることができません単にフレームワークの配布キットをダウンロードして、コマンドラインから起動してプラグインをインストールせずに。
このソリューションは、ビルドサーバーでの使用に適しています。

フレームワークがプロパティ構文、つまりyaml構成で一緒に使用できる通常のプロパティファイルを完全に理解していることは注目に値します。

別の支払いサービスを追加し、同時に既存の支払いサービスを複雑にします。
では順：

eureka:
 instance.preferIpAddress: true
 client:
   serviceUrl:
     defaultZone: http://192.89.89.111:6782/eureka/
server.port: 9999
spring.application.name: order
db.url: 192.168.0.100

お支払い：

eureka:
 instance.preferIpAddress: true
 client:
   serviceUrl:
     defaultZone: http://192.89.89.111:6782/eureka/
server.port: 9998
spring.application.name: payments
db.url: 192.168.0.100

これらの構成の主な問題は、サービス設定に大量のコピーペーストが存在することです。フレームワークがそれを取り除くのにどのように役立つか見てみましょう。最も明白なものから始めましょう- 各マイクロサービスの説明におけるユーレカ構成の存在。設定ファイルを使用して新しいディレクトリを作成し、そこに新しい構成を追加します。



各プロジェクトに#include eurekaという行を追加します。

フレームワークは自動的にeureka構成を見つけてサービス構成ファイルにコピーしますが、dev.yaml環境ファイルで指定しないため、別個のeureka構成は作成されません。サービスオーダー：

#include eureka
server.port: 9999
spring.application.name: order
db.url: 192.168.0.100

また、インポート行を#include eureka、oracleに変更することで、別の構成でデータベース設定を行うこともできます。

構成ファイルの再生成中の各変更はフレームワークによって監視され、メイン構成ファイルの横の特別なファイルに配置されることに注意してください。彼のログのエントリは次のようになります：「保存された1プロパティがorder / diff-application.yamlに変更されました」。これにより、大きな構成ファイルの変更をすばやく検出できます。

構成の共通部分を削除すると、多くの不要なコピーと貼り付けを取り除くことができますが、さまざまな環境の構成を柔軟に作成することはできません。サービスのエンドポイントは一意であり、ハードコード化されています。これは悪いことです。削除してみましょう。

良い解決策は、他の人が参照できる1つの構成にすべてのエンドポイントを保持することです。これを行うために、プレースホルダーのサポートがフレームワークに導入されました。eureka構成ファイルの変更方法は次のとおりです。

 client:
   serviceUrl:
     defaultZone: http://${endpoints@eurekaip}:6782/eureka/

次に、このプレースホルダーがどのように機能するかを見てみましょう。システムはエンドポイントと呼ばれるコンポーネントを見つけ、その中でeurekaipを探し、それを構成で置き換えます。しかし、異なる環境はどうですか？これを行うには、次のタイプapplication.dev.yamlのエンドポイントに設定ファイルを作成します。フレームワークは独立して、ファイル拡張子に従って、この構成が属する環境を決定し、それをロードし



ます。devファイルの内容：

eurekaip: 192.89.89.111
dbip: 192.168.0.100

サービスのポートに同じ構成を作成できます。

server.port: ${ports@order}.

すべての重要な設定が1か所にあるため、構成ファイルに散在するパラメーターが原因でエラーが発生する可能性が低くなります。

フレームワークは、多くの既製のプレースホルダーを提供します。例えば、構成ファイルが配置されているディレクトリーの名前を取得して割り当てることができます。

#include eureka, oracle
server.port: ${ports@order}
spring.application.name: ${this@name}

このため、構成でアプリケーションの名前を追加で指定する必要はありません。また、同じモジュールなどの共通モジュールに移動することもできます。

client:
   serviceUrl:
     defaultZone: http://${endpoints@eurekaip}:6782/eureka/
 spring.application.name: ${this@name}

注文 構成ファイルは1行に削減されます。

#include eureka, oracle
server.port: ${ports@order}

親構成からの設定が不要な場合は、構成で指定でき、生成時に使用されます。つまり、何らかの理由で注文サービスに一意の名前が必要な場合は、spring.application.nameパラメータをそのままにします。

サービスにカスタムロギング設定を追加する必要があるとしましょう。これらの設定は、logback.xmlなどの別のファイルに格納されています。個別の設定グループを作成してみましょう。



基本構成では、@ ConfigDir プレースホルダーを使用して、必要なログ設定ファイルを配置するフレームワークを指定します。

microconfig.template.logback.fromFile: ${logback@configDir}/logback.xml

logback.xml ファイルでは、標準のアペンダーを構成します。これには、プレースホルダーを含めることもできます。プレースホルダーは、構成の生成中にフレームワークによって変更されます。次に例を示します。

<file>logs/${this@name}.log</file>

サービス構成にlogback importを追加すると、サービスごとに構成されたログが自動的に取得されます。

#include eureka, oracle, logback
server.port: ${ports@order}

利用可能なすべてのフレームワークプレースホルダーをより詳しく理解する時が来ました：

$ {this @ env} -現在の環境の名前を返します。
$ {... @ name} -コンポーネントの名前を返します。
$ {... @ configDir} -コンポーネントの構成ディレクトリへの完全パスを返します。
$ {... @ resultDir} -コンポーネントの宛先ディレクトリへのフルパスを返します（受信したファイルはこのディレクトリに配置されます）。
$ {this @ configRoot} -構成ストアのルートディレクトリへのフルパスを返します。

システムは、javaへのパスなどの環境変数を取得することもできます。
$ {env @ JAVA_HOME}
または、フレームワークはJAVAで記述されているため、次のようなシステム変数を取得できます。System ::次の種類の構成を使用するgetProperty：
${system@os.name} Spring EL
拡張言語のサポートについて言及する価値があります。構成では、同様の式が適用されます。

connection.timeoutInMs: #{5 * 60 * 1000}
datasource.maximum-pool-size: #{${this@datasource.minimum-pool-size} + 10}

#var 式を使用して、構成ファイルでローカル変数を使用できます。

#var feedRoot: ${system@user.home}/feed
folder:
 root: ${this@feedRoot}
 success: ${this@feedRoot}/archive
 error: ${this@feedRoot}/error

したがって、フレームワークは、マイクロサービスを細かく柔軟に構成するためのかなり強力なツールです。フレームワークは完全にその主要なタスクを実行します-設定のコピーと貼り付けをなくし、設定を統合し、結果として起こり得るエラーを最小限に抑えながら、構成を組み合わせて異なる環境に簡単に変更できます。

このフレームワークに興味がある場合は、その公式ページにアクセスして完全なドキュメントを読むか、ここのソースを詳しく調べることをお勧めします。
