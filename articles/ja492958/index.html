自動化された再帰コンピューティング

1.はじめに


サブルーチン（英語のサブルーチン）が誇張なしのプログラミングに与える影響は計り知れません。プログラミングの黎明期に導入され、彼らは今日に関連性を失うことはありません。それらがなければ、実用的なプログラミングは想像することは不可能です。正式な観点からは、それほど必要ではありませんが、純粋な理論は、アルゴリズムのサイズよりもアルゴリズムの特性に関心があります。

オートマトンの理論では、オートマトンサブプログラム（AMP）の実践の基礎となるネストされたオートマトンの概念については、ほとんど議論されていません。オートマトンの同様の（ネストされた）階層構造は、考えれば非常に表面的なものです。この関係の理由の1つは、ハードウェアレベルでネストされた階層を実装する複雑さである可能性があります[1、2]。

プログラミングは柔軟性が高く、デジタル回路を設計する場合よりも多くの機能を提供します。ネストされたオートマトンのソフトウェア実装と、オートマトン再帰アルゴリズムのコンセプトをさらに考慮して、これを確認する必要があります。

ネストされたオートマトンモデルを形成する際のすべての特定の問題について、その正式な定義は問題を引き起こしません。ただし、その一方で、モデル階層を構築するという選択は、ソフトウェアの実装に大きな影響を与えます。

2.自動化された再帰アルゴリズム


前回の記事[3]では、自動プログラム（AP）の制御モデルの正式な定義のみが示され、その実装や具体的な使用例は考慮されていませんでした。再帰的アルゴリズムを実装する可能性についても言及されました。次に、階乗計算の例を使用して、最初に、オブジェクトオートマトンC ++パラダイムのフレームワーク内で自動ルーチンを作成するためのメカニズムのソフトウェア実装を検討し、次に、再帰的アルゴリズムを検討するので、そのような実装の一般原則を本質的に定義します自動プログラミングのフレームワークにおけるアルゴリズム。

APIで再帰関数を使用するのは非常に簡単です。これは、オブジェクトマシンプログラムのリスト1のコードを示しています。ここで、FSimpleFactorial automatonクラスはプロセスではなく、オートマトンルーチンです。最終状態「00」が含まれます（サブプログラムの詳細については、[3]を参照してください）。動作のレベルでは、作成されたオートマトンオブジェクトは、初期状態「f0」から最終状態「00」への1つの無条件遷移を備えたオートマトンに対応し、通常の階乗階乗計算関数因子（...）のアクションy1のフレームワーク内でこの遷移を呼び出します。

#include "lfsaappl.h"

class FSimpleFactorial : public LFsaAppl
{
public:
    double dF{1};		//	 
    FSimpleFactorial(int n);
    virtual ~FSimpleFactorial() {};
private:
    int nN{0};
    void y1();
    double factorial(int n);
};

#include "stdafx.h"
#include "FSimpleFactorial.h"

LArc TT_SimpleFactorial[] = {
    LArc("f0", "00", "--",	"y1"),		//
    LArc()
};
//////////////////////////////////////////////////////////////////////
// Construction/Destruction
//////////////////////////////////////////////////////////////////////
FSimpleFactorial::FSimpleFactorial(int n)
    :LFsaAppl(TT_SimpleFactorial, "FSimpleFactorial")
{
    nN = n;
}
//
void FSimpleFactorial::y1() { dF = factorial(nN); }

double FSimpleFactorial::factorial(int n)
{
    if (n == 0) return 1;
    else return n*factorial(n-1);
}

    pVarStart->SetDataSrc(this, 0.0);

    // creating and calling a subroutine
    pFCall->FCall(this);
}


初期状態 "st"から状態 "t1"への遷移におけるこのプロセスは、ネストされたオートマトン（通常の意味では、サブルーチンの呼び出し）を作成するための基礎となるFSimpleFactorialタイプのオブジェクトを作成します。さらに、ブール型bStartのローカル変数の現在の状態に基づいて（述語コードx1も参照）、状態「t1」から「t2」に遷移します。この遷移では、アクションy1が最初にこのローカル変数の値をリセットし（誤った再起動を防止するため）、次にFCallプロシージャを呼び出して、ネストされたオートマトンを作成します。

VKPa環境の自動マシンのインタープリターは、ネストされたオートマトンの初期状態「f0」への遷移を編成し、後者の状態がプロセスの現在の状態になります。ネストされたオートマトンの最終状態「00」への遷移は、計算を前のネストレベルに戻します。この例では、最上位のオートマトンの状態t2への遷移が完了します。次に、プロセスは、状態「t2」から状態「t1」への無条件遷移で、アクションd2のローカル変数の階乗の計算結果を設定するアクションy2を実行します。

ネストされたオートマトンの結果は、その作業が完了し、トップレベルのオートマトンの次の状態に移行した後でのみ利用可能になることが非常に重要です。ネストされたオートマトンのオブジェクトへのポインターを介してそれ（結果）を取得することが可能になります。

そして、それが再帰的な関数のランタイムのためでなければ、すべてがうまくいくでしょう。通常のサブルーチンと比較すると、ネストのため、原則としてはるかに大きくなるため、自動プロセスのアクションの条件付き原子性に違反する可能性があり、その合計実行時間は、離散クロックサイクルの値に収まるはずです。

私たちの場合、階乗の計算は非常に小さく、n = 170のdoubleの結果の最大値までは1マイクロ秒に適合します。大きな値を計算するには、長い演算に切り替える必要があります（たとえば、[4、5]を参照）。同時に、階乗計算時間はさらに増加し​​、ディスクリートクロックサイクルを超えることがほぼ保証され、非プリエンプティブマルチタスクモードで動作している残りのネットワークマシンの速度と、アプリケーション全体の応答の反応性に影響を与え、アプリケーションの「ブレーキ」として現れます。

計算を段階的に拡張することにより、離散クロックサイクル内に保ち、「ブレーキ」の欠陥を取り除くことができます。これらの目的のために、通常の階乗アルゴリズムを自動形式に変換すると便利です。確かに、オートマトンの解釈モードにより、計算時間は増加しますが、アプリケーションの一時的なプロパティは通常に戻ります。

リスト3は、通常の形式とその後のオートマトン形式への変換のために準備された形式での階乗の計算を示しています。後者は、プログラムのソースコードに隠されているアトミックステップを明示的に表します。ここでは、y = f（...）という形式の演算子について説明します。fは通常のプログラム関数または再帰的なプログラム関数です。このようなレコードは、関数の動作時間をマスクし、変数yの値の「瞬間的な」割り当ての誤った印象を作成します。実際にはどちらもありません。

リスト3.階乗を計算するためのソフトウェア関数
int factorial(int n)
{
    if (n == 0) return 1;
    else return n*factorial(n-1);
}

double Factorial(int n)
{
    double dF = 1;
    if (n == 0)
        return dF;
    else {
        double dPrev = Factorial(n-1);
        double dF = n*dPrev;
        return dF;
    }
}

3.結論


解決すべきタスクの詳細により、トレーニング目的とネストの実装のためのカーネルのテストを除いて、実際のプロジェクトで実際に再帰アルゴリズムを使用および開発する必要はなかったと認めます。自動化ルーチン-ほとんど常に、大量に。その中でも、慣性と呼ばれるサブルーチンが特に興味深い（詳細については、[3]を参照）。しかし、彼らは彼らの考慮に別の記事を捧げることになっています。

再帰的アルゴリズムはプログラム理論の重要な部分です[6]。ルーチンと再帰を実装しないと、すべての観点から効率的で便利な計算モデルを想像することは困難です。また、オブジェクトプログラミングの可能性を考慮して（記事の例を参照）、再帰を実装するには、サブプログラムのモデルを作成した後、それほど難しくありません。もちろん、再帰を通常の形式に変換する特定の方法を使用して、同じサイクルを使用して再帰をバイパスすることができます。しかし、愚かにそれを取り除くのではなく、その実装のための直接的なメカニズムを持つことが望ましいです。もちろん、それは遅くなりますが、リソース（同じスタックメモリの費用）の面でコストは高くなりますが、既存のハードウェア機能ではそれほど重要ではありません。

しかし、再帰アルゴリズムは、アルゴリズムモデルの機能、普遍性、および有効性を評価するためのテストとしても興味深いものです。この意味で、オートマトンモデルは再帰を非常に効果的に実装するだけでなく、ボーナスとして「オートマトンチップ」を追加します-再帰アルゴリズムの内部状態を制御する機能、パフォーマンスを評価するステップをカウントする機能、アルゴリズムの実行時間を制御する機能など。等

再帰との戦いは十分に無意味です。使用できる必要があります。 「私は一見、再帰が複雑に見えるかもしれません。しかし、場合によっては、すべてが正しく行われていれば、再帰的な方法は非常に効果的です。ただし、ループを使用する方が良い場合があります。両方の方法とそれらを効果的に使用する能力を理解することは、あなたの仕事であなたを助け、インタビューでの利点となります” [7]。再帰アルゴリズムの自動モデルは、計算モデルの「自動プロパティ」を使用して、再帰を理解し、そのようなアルゴリズムをはるかに速くデバッグおよび改良することを可能にすることを自分で追加するだけです。
