30分でWebサービスを監視するボット：電報+ bash + cron



新しいサービスをすばやく監視する必要がある場合もありますが、手元に既成のインフラストラクチャ/専門知識がない場合があります。このガイドでは、30分で、組み込みのubuntuツール（bash、cron、curl）のみを使用してWebサービスを監視するためのツールを実装します。電報を使用してアラートを配信します。

「ケーキの上のチェリー」は、ユーザーの感情的な関与になります。それは人々にチェックされます-それは動作します。

Doctor Near遠隔医療サービスでチャットボットを作成してユーザーストレスのレベルを特定する際、監視が必要でした。数時間で、ミニプロジェクトが作成されました。これは、うまく機能するだけでなく、肯定的なフィードバックも追加します。

はじめに、スクリプトを含むリポジトリを取得します。

git clone https://github.com/rshekhovtsov/msms.git

msmsフォルダーに移動し、その中で作業します。

テレグラムがブロックされている場合は、プロキシを使用してください。最も簡単で信頼性の高いオプションは、トルソックです。

sudo apt install tor
sudo apt install torsocks

ステップ2.モニタリングを設定する


サービスの説明は、servicesフォルダーにiniファイルを作成することによって行われます。5つのパラメータを設定する必要があります。

  1. MSMS_SERVICE_NAME：サービス名-警告と監視ログで使用されます。
  2. MSMS_SERVICE_ENDPOINT：curlにアクセスするサービスのエンドポイント。
  3. MSMS_CURL_PARAMS：追加のカールオプション。以下の例を参照してください。
  4. MSMS_EXPECTED：予期されるサービス応答。答えが短い場合に使用されます。
  5. MSMS_EXPECTED_FILE：予期されるサービス応答を含むファイル名。指定した場合、MSMS_EXPECTEDを上書きします。
  6. MSMS_RECIPIENTS：通知受信者のリストを含むファイル。

google.comでのリクエストは、リダイレクト付きの固定htmlを返します。これを、予期されるサーバー応答として使用します。

curl google.com > services/google-response.html

services / google.iniファイルを作成します。

MSMS_SERVICE_NAME='google front page'

# service endpoint
MSMS_SERVICE_ENDPOINT='google.com'

# curl parameters
MSMS_CURL_PARAMS='-s --connect-timeout 3 -m 7'

# expected service response
MSMS_EXPECTED_FILE='google-response.html'

# recipients list file
MSMS_RECIPIENTS='google-recipients.txt'

MSMS_CURL_PARAMScurlが実行できるすべての ことを指定できます。

  1. コンソールとログが詰まらないように、curlメッセージを無効にします。 -s
  2. チェックするサービスとの接続のタイムアウトを設定します（秒単位）： --connect-timeout 3
  3. 応答を受信するためのタイムアウトを設定します。 -m 7
  4. SSLの証明書検証を無効にします（たとえば、自己署名証明書が使用されている場合）。 --insecure
  5. httpリクエストのタイプを指定します。 -X POST
  6. ヘッダーを指定します。 -H "Content-Type: application/json"
  7. リクエストの本文を文字列またはファイルとして指定します。ファイルの例：-d @request.json

通知をオフにし、タイムアウトを3秒に設定しました。接続時と7秒。サービスからの応答を受信します。

注意：例のように、パラメーター値を一重引用符で指定します。残念ながら、この意味でのbashはかなり脆弱であり、誤って間違った引用に飛んでいる蝶は、エラーの診断が困難な宇宙の死につながる可能性があります。

監視を設定します。すべてが正常であることを確認します。

sudo chmod +x ./monitoring.sh
torsocks ./monitoring.sh

スクリプトは次の形式のメッセージを表示する必要があります。

2020-01-10 12:14:31
health-check "google front page": OK

ステップ3.スケジュールを調整する


cronで監視スケジュールを設定します。

sudo crontab -e

毎分google.comをチェックする行を追加します。

*/1 * * * * torsocks <   >/monitoring.sh >> <   >/monitoring.log 2>&1

監視自体のパフォーマンスを確認する通知を毎日11.00に追加します。これを行うには、DAILYパラメータをスクリプトに渡します。

0 11 * * * torsocks <   >/monitoring.sh DAILY >> <   >/monitoring.log 2>&1

2>&1-エラーをメイン出力ストリームにリダイレクトする標準的な手法。その結果、それらは監視ログにも含まれます。

変更を保存し、次のコマンドでそれらをキャッチします。

 sudo service cron reload

cronの設定の詳細については、たとえばこちらをご覧ください。

したがって、監視スクリプトが毎分起動され、google.comのgolからアクセスされます。受信した応答が予期された応答と異なる場合、スクリプトは電信で通知を受信者のリストに送信します。監査ログは、monitoring.logファイルに保持されます。

別のサービスを追加する必要がある場合は、servicesフォルダーにそのサービスの新しいiniファイルを作成し、必要に応じて、別の受信者リストを作成します。他はすべて自動的に動作します。

チェックされたサービスが利用できなくなった場合、アラートが毎分送信されます。サービスをすぐに復元できない場合は、電報のボットプロパティで通知を一時的に無効にできます。

それでは、スクリプトの追加機能と実装について詳しく見ていきましょう。

監視スクリプト


monitoring.shは単純な自動検出を実行します-servicesフォルダーからすべてのiniファイルを取得し、それぞれがアラートをチェックして送信するためのロジックを含むメインスクリプトを実行します。

#!/bin/bash
cd $(dirname "$0")/services

for service_ini  in $(ls *.ini); do
    bash ../msms.sh "$1" "$service_ini"
done

監視ステータスに関する毎日のメッセージを生成するには、スクリプトにDAILYパラメータを渡すことができます。

スクリプトが開始すると、現在のフォルダーがサービスに変更されることに注意してください。これにより、iniファイルはサービスに関連するファイルパスを指定できます。

アラートを確認および送信するためのスクリプト


msms.shには、サービスをチェックしてアラートを送信する基本的なロジックが含まれています。

電報を扱う：

# telegram endpoint
TG_API_URL="https://api.telegram.org/bot$(cat ../telegram-api-key.txt)/sendMessage"

#################################################################
# send message to telegram
# parameter: message text
#################################################################
function send_message {
    for chat_id  in $(cat ../$MSMS_RECIPIENTS); do
	curl -s -X POST --connect-timeout 10 $TG_API_URL -d chat_id=$chat_id -d parse_mode="Markdown" -d text="$1"
	echo
    done
}

ファイルに格納されている秘密鍵を使用して、電報REST APIにアクセスするためのURLを作成します。

send_message関数は、curlを使用してこのREST APIにメッセージを送信し、iniで指定したファイルから受信者IDを取得します。データを送信するときに、マークアップ投稿を使用するように指定しますparse_mode="Markdown"。

現在の日時を出力し、iniファイルをロードします。

echo $(date '+%Y-%m-%d %H:%M:%S')

# load variables from .ini file:
. $2

マジックライン. $2は、入力に渡されたiniファイルを2番目のパラメーターとして通常のスクリプトとして実行し、その中に指定された値を環境変数に入れます。

パラメータが指定されている場合、ファイルから予期される応答をダウンロードしますMSMS_EXPECTED_FILE。

if [ -n "$MSMS_EXPECTED_FILE" ]; then
 MSMS_EXPECTED="$(cat "$MSMS_EXPECTED_FILE")"
fi

必要に応じて、アラートを送信してサービスチェックを実行します。

RESPONSE="$(eval curl $MSMS_CURL_PARAMS \"$MSMS_SERVICE_ENDPOINT\")"
EXIT_CODE=$?
if [[ $EXIT_CODE != 0 ]]; then
    echo health-check \"$MSMS_SERVICE_NAME\" FAILED: CURL EXIT WITH $EXIT_CODE
    MESSAGE="$(cat ../templates/curl-fail.txt)"
    MESSAGE=$(eval echo $MESSAGE)
    send_message "$MESSAGE"
elif [[ "$RESPONSE" != "$MSMS_EXPECTED" ]]; then
    echo health-check \"$MSMS_SERVICE_NAME\" FAILED: "$RESPONSE"
    MESSAGE="$(cat ../templates/service-fail.txt)"
    MESSAGE=$(eval echo $MESSAGE)
    send_message "$MESSAGE"
else
    echo health-check \"$MSMS_SERVICE_NAME\": OK
fi

まず、RESPONSEこのサービスのcurlコマンドの結果に変数を割り当てます。

式EXIT_CODE=$?は最後のコマンドの結果を変数に入れます。カール。アラートを送信する必要がある場合は、対応するファイルからテンプレートが読み込まれ、受信者へのメール送信がを使用して実行されsend_messageます。

最後のブロックはDAILYパラメータを処理します。

if test "$1" = "DAILY"; then
    echo health-check \"$MSMS_SERVICE_NAME\" DAILY
    MESSAGE="$(cat ../templates/daily.txt)"
    MESSAGE=$(eval echo $MESSAGE)
    send_message "$MESSAGE"
fi

監視自体の状態を確認するメッセージを送信します。
