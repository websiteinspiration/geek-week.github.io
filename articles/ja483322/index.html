コードファーストアプローチとは何ですか？


REST APIやSOAP APIなどのサービスを開発するときはいつでも、次の2つの方法のいずれかを選択できます。

  • 最初にコードし、コードから契約を生成する
  • 契約優先および契約ベースのコード開発

まず、Code Firstの簡単な例から始めましょう。

Spring Boot Code First REST APIの例


Spring Boot Frameworkを使用してAPIを生成するRESTful Webサービスを開発しています。たとえば、retrieveAllUsers（）APIでは、「/ users」URIを開き 、サービスメソッドを呼び出して
すべてのユーザー（/ usersリソース）を返します

このURLにアクセスすると、すべてのユーザーが返されます。


他のサービスメソッドも同じ方法で定義され、それぞれに独自のURIがあります。この例では、コードを取得し、そこからドキュメントを生成します。このドキュメントでは、ユーザーがサービスを使用する方法について説明します。これを行うには、Swaggerドキュメント形式を使用します

受け取った応答メッセージのタイプとそれに付随する応答ステータスが表示されます。Swaggerからこのサービスを呼び出して回答を取得する

コードファーストのメリット


このアプローチの主な利点：

  • 最小限の労力での契約：契約の生成に追加の労力は必要ありません。コードから自動的に生成できるため、これはサービス開発の副産物にすぎません。
  • コードとコントラクトの同期：コントラクトはコードから生成されるため、常に互いに同期されます

コードファーストの短所


このアプローチの欠点は次のとおりです。

並行開発なし


サービスプロデューサーとサービスコンシューマーを並行して開発することはできません。最初にサービスを開発し、次に契約を生成する必要があります。その後、契約に準拠するコンシューマコードを記述できます。契約を理解しなければ、消費者を開発することはできません。

チームの目標なし


サービスが開発される前に契約を知ることができないため、開発におけるさまざまな利害関係者の目的はありません。したがって、方向がずれたり、不必要な変更が加えられたりして、無駄な労力が発生する可能性があります。

クロスプラットフォーム互換性なし


一部の古いプラットフォームでは、コードからコントラクトを生成するのはそれほど簡単ではありません。この結果、生成されたコントラクトでは、プラットフォーム間の非互換性が非常に頻繁に発生します。

この主題に関する著者のビデオがあります。
