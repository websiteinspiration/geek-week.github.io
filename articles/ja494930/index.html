Unity3D：自動スクリプトマネージャーアグリゲーター

前書き


この記事では、スクリプトマネージャー間の相互作用の1つのタイプの編成（シングルトーンと呼ばれる）に焦点を当て、具体的には、すべてのインスタンスマネージャーへのリンクを含む個別のアグリゲータークラスの使用に焦点を当てます。この記事を読んだ後、アグリゲータークラスを作成するというアイデアが思い浮かびました。

タスク


私は、毎回自分のペンでマネージャーをアグリゲータークラスにハンマーで打つことは非常に不便であるという結論に達しました。新しいマネージャーを作成するときは、アグリゲータークラスを開いて変更する必要があります。十分な経験があれば、このような面倒なプロセスを自動化できます。したがって、タスクが設定されました：シングルトーンの自動作成とその自動コレクター。

クラスシングルトン


using UnityEngine;
public class Singleton<T>: MonoBehaviour where T: MonoBehaviour
{
	public static T instance { get; private set; }

	public void Awake()
	{
		if (instance == null)
		{
			instance = GetComponent<T>();
			ManagersAregator.addManager(instance);
		}
		else
		{
			Debug.Log($"<color=red>   {typeof(T).ToString()}</color>");
			Destroy(gameObject);
		}
	}

	public void OnDestroy()
	{
		ManagersAregator.removeManager<T>();
	}
}

Unityの古典的なシングルトン実装は、インスタンスの静的変数を使用します。Awake（）メソッドは、インスタンスが定義されているかどうかを確認します。そうでない場合は、thisキーワードを使用してクラスインスタンスへの参照を取得します。しかしそれ以来特定の実装では、クラスTの「テンプレート」変数が使用されますが、これは使用できませんでした。しかし、スクリプトがハングしているオブジェクトからTコンポーネントを簡単に取得できます。場合は、インスタンスがされて定義され、それが破壊されなければなりません。したがって、オブジェクトは常に1つのコピーでステージ上にあります。Awake（）

メソッドで新しい要素がaggregatorクラスに追加され、OnDestroy（）メソッドで、要素がaggregatorクラスから削除されます。MyClass

クラスアグリゲーター


using System.Collections.Generic;
using UnityEngine;

public static class ManagersAregator
{
	static Dictionary<string, MonoBehaviour> Managers = new Dictionary<string, MonoBehaviour>();

	public static void addManager<T>(T newManager)
	{
		string keyWord = typeof(T).ToString();

		if(Managers.ContainsKey(keyWord))
		{
			Debug.Log($"[ManagersAregator]  -{newManager}-   -{keyWord}-  ");
		}
		else
		{
			Managers.Add(keyWord, newManager as MonoBehaviour);
			Debug.Log($"<color=green>[ManagersAregator]    -{newManager}-   -{keyWord}-</color>");
		}
	}

	public static T getManager<T>(string callback) where T: Singleton<T>
	{
		string keyWord = typeof(T).ToString();

		if(Managers.ContainsKey(keyWord))
		{
			Debug.Log($"<color=yellow>[{callback}]   -{keyWord}-</color>");

			MonoBehaviour mbTemp = null;
			T manager = null;

			if(Managers.TryGetValue(keyWord, out mbTemp))
			{
				manager = (T)mbTemp;
				Debug.Log($"<color=green>[{callback}]  -{manager}- </color>");
			}
			else
			{
				Debug.Log($"<color=red>[{callback}]    -{keyWord}-</color>");
			}

			return manager;
		}

		Debug.Log($"<color=red>[ManagersAregator]    -{keyWord}-   .</color>");
		return null;
	}

	public static void removeManager<T>()
	{
		string keyWord = typeof(T).ToString();

		if(Managers.ContainsKey(keyWord))
		{
			Managers.Remove(keyWord);
			Debug.Log($"[ManagersAregator]    -{keyWord}-   ");
		}
		else
		{
			Debug.Log($"[ManagersAregator]    -{keyWord}-   .");
		}
	}
}

クラスは静的です。彼はステージ上にインスタンスを持つことができません。したがって、いつでもどのシーンからでも利用できます。

すべてのスクリプトマネージャー（singletones）は、マネージャーディクショナリに格納されます。各マネージャーのキーは、そのマネージャーのクラス名です。おそらく、プログラミングの初心者は「Bah、しかしこのディクショナリはMonoBehaviourを何に格納し、すべてのクラスはSingletonから継承するのか」という質問をするでしょう。これは良い質問です。その答えは、あらゆるクラスのマネージャーの自動アグリゲーターを実装するための鍵です。

プログラミングには、アップキャスト、つまり基本クラスへの型変換の概念があります。 UnityのすべてのクラスはMonoBehaviourを継承しているため、MonoBehabiourに適用できます。したがって、マネージャディクショナリには、MonoBehaviourクラスのオブジェクトのみが含まれます。

アグリゲータークラスのメソッドを考えてみましょう：

void addManager<T>(T newManager)

このメソッドは、SingletonクラスのAwake（）メソッドで呼び出されます。引数はインスタンスクラスの静的変数です。次に、インスタンスが属するクラスの名前でキーが作成され、マネージャーが辞書に追加されます。

T getManager<T>(string callback) where T: Singleton<T>

関数は、メソッドの呼び出し元のクラスの名前を文字列として引数に取ります。これは、便利なデバッグのためだけに行われます（コンソールには、メソッドが呼び出された場所からクラスが表示されます）。AnotherMyClassクラスでこのメソッドを使用する例：

public class AnotherMyClass: MonoBehaviour
	void Start()
	{
		string cb = GetType().ToString(); //     
		MyClass MC = ManagersAregator.getManager<MyClass >(cb);
	}

「Hu ** I、やり直し [AnotherMyClass] Manager -MyClass-を受信しました。」 というメッセージがコンソールに表示されます。

void removeManager<T>()

タイプTのマネージャがディクショナリに含まれている場合、ディクショナリから削除します。

概要


実際、アグリゲータクラスの3つの関数のうち、開発者はgetManagerメソッドを使用するだけで済みます。特別なプラスは、あらゆる状況でのデバッグメッセージの優れた可視性です。もちろん、オプションでそれらをオフにすることもできます。いつ、どのクラスが何かを取得しようとしているか、何を取得しようとしているのかを確認することは非常に便利だと思います。
