CO₂Xiaomi ClearGrass Air Detectorセンサーを中国のサーバーなしでローカルに使用します



長い間、私はホームオートメーションに二酸化炭素センサーCO automationを導入することを計画していました。価格/品質/機能/外観の点で、Xiaomi ClearGrass Air Detectorは私にとって最高であることが判明しました。空気質分析装置にはセンサーが含まれています：

  1. CO₂
  2. tVOC（揮発性有機化合物）
  3. PM2.5
  4. 温度
  5. 湿度

ClearGrassは、視野角が大きく、バッテリーが6時間持続する高品質の画面を備えています。そのようなデバイスの130ドルの地域の価格は、それを必須のセグメントに変換します！素晴らしいレビューはmysku.ruで読むことができます。

アナライザーは、ネイティブのqingping +またはMiHomeアプリケーションに追加できます。どちらの場合も、データは中国のサーバーを経由します。サードパーティのリモートサーバーを使用せずにセンサーからローカルにデータを取得する方法を見つけることにしました。

1.トラフィックを調査する


最初のステップは、ClearGrassがデータをqingping +アプリケーションに転送する方法を確認することでした。ClearGrassはWi-Fi経由でインターネットに接続します。トラフィックを聞くために、Raspberry Pi Wi-Fiでアクセスポイントを上げ、tcpdumpを開始して情報を収集しました。

sudo tcpdump -i wlan0 -vv -s0 -X -n port 1883 -s 65535 -w cleargrass.pcap

トラフィックを分析すると、ClearGrassは約5つの異なるIPアドレスにアクセスし、154.8.191.174ではMQTTプロトコルを使用して暗号化されていない電波品質データを送信することがわかりました。



2. ClearGrassからのトラフィックをRaspberry Piでラップします


iptablesで 少し実験して、私はこのルールに行きました：

sudo iptables -i wlan0 -t nat -A PREROUTING -s 192.168.115.19 -j REDIRECT

「192.168.115.19（IP ClearGrass）からのwlan0インターフェース上のすべての新しいトラフィックは、ローカルにリダイレクトする必要があります。」私はiptablesの専門家ではないので、提案や改善をうれしく思います。このルールにはマイナスがあります。アナライザーが既にRaspberry Piに接続されている場合、トラフィックはリダイレクトされません。最初にルールを実行してから、Wi-Fiを介してClearGrassをRaspberry Piに接続する必要があります。

その結果、Raspberry PiでMQTT モスキートブローカーを受け取ったときに、アナライザーが1分に1回、空気品質データを送信しているのがわかりました。

3.ホームオートメーションZウェイ用のJS上のMQTTナノブローカー


私はホームオートメーションサーバーとして、多くのZ-WaveデバイスとJSでスクリプトを作成する機能をサポートするZ-Wayを使用しています。



残念ながら、Z-Wayには（node.jsに基づくシステムとは異なり）JS用のMQTTブローカーがないため、このアナライザーからのデータのみを受け入れ、他には何もできない最小限のブローカーを作成することにしました。特にドキュメントを読まずに、アナライザーと蚊の間の通信を調べ、次のシーケンスをまとめました。

MQTT PROTOCOL

Connect Command (sensor -> broker)
	0x10 - Connect Command

Connect Ack (broker -> sensor)
	0x20 - Connect Ack
	0x02 - Len 2
	0x00
	0x00 - Connection Accepted

Subscribe Request (sensor -> broker)
	0x82 - 0b1000 0010; 0b1000 - Subscribe Request

Subscribe Ack (broker -> sensor)
	0x90 - 0b1001 0000; 0b1001 - Subscribe Ack
	0x03 - Len 3
	0x00
	0x08 - Message identifier 8
	0x00 - Fire and Forget

Ping Request (sensor -> broker)
	0xC0 - Ping Request
	0x00 - Len 0

Ping Response (broker -> sensor)
	0xD0 - Ping Response
	0x00 - Len 0

Publish Message (sensor -> broker)
	0x30 - Publish Message
	0x96
	0x04 - Len 534

その結果、簡単なJSスクリプトが生まれました。

mqttSocket.reusable();
mqttSocket.bind(1883);
mqttSocket.onrecv = function(data, host, port) {
	var arr = new Uint8Array(data);

	switch(arr[0]) {
		// PING
		case 0xC0:
			console.log("---------- MQTT PING RESPONSE");
			this.send([0xD0, 0x00]);
			break;
		// CONNECT
		case 0x10:
			console.log("---------- MQTT CONNECT ACK");
			this.send([0x20, 0x02, 0x00, 0x00]);
			break;
		// SUBSCRIBE
		case 0x82:
			console.log("---------- MQTT SUBSCRIBE ACK");
			this.send([0x90, 0x03, arr[2], arr[3], 0x00]);
			break;
		// PUBLISH
		case 0x30:
			var sensorPayload = self.getPayload(arr);
			var sensorMessage = sensorPayload.substr(sensorPayload.indexOf('{'), sensorPayload.lastIndexOf('}'));
			var sensorObj = JSON.parse(sensorMessage);
			console.logJS("---------- MQTT MESSAGE:", sensorObj);
			console.logJS("---------- CO2: ", sensorObj.data.co2);
			self.vDevCO2.set("metrics:level", sensorObj.data.co2);
			break;
	}
};
mqttSocket.listen();

もちろん、多くのことは考慮されていませんが、たとえば、ある前提では、PINGとMESSAGEの両方を使用することができますが、私はこれの一部を見逃します。おそらく将来的には、Aedesコードベースを使用して、Z-Way用のMQTTブローカーを作成します。そして現時点では、この目標はXiaomi ClearGrass空気検知器分析装置から地域の大気質データを取得するための基本的な機会であり、この目標は達成されました。

将来的には、Z-WaveバージョンのTION S3ブリーザーをインストールし、ClearGrassのデータに基づいて制御したいと考えています。
