ビルダー向けのB2Bサービスの例でのデータベースクエリの最適化

より効率的なサーバーに移動せずにデータベースクエリの数で10倍に成長させ、システムを稼働させ続けるにはどうすればよいですか？データベースのパフォーマンス低下にどのように取り組んだか、SQLクエリをどのように最適化して、できるだけ多くのユーザーにサービスを提供し、コンピューティングリソースのコストを増加させないかについて説明します。

建設会社の業務管理サービスを行っています。約3000社が私たちと協力しています。1万人以上が毎日4〜10時間、私たちのシステムで働いています。計画、アラート、警告、検証などのさまざまなタスクを解決します。PostgreSQL9.6を使用しています。データベースには約300のテーブルがあり、毎日最大2億のリクエスト（1万の異なる）がデータベースに送信されます。平均して、1秒あたり3〜4,000のリクエストがあり、最もアクティブな瞬間には1秒あたり1万を超えるリクエストがあります。ほとんどのリクエストはOLAPです。追加、変更、削除がはるかに少ないため、OLTPの負荷は比較的小さいです。プロジェクトの範囲を評価し、私たちの経験がどのように役立つかを理解できるように、これらすべての数値を示しました。

最初の写真。リリカル


開発を開始したとき、データベースにどのような負荷がかかるのか、サーバーがプルを停止した場合にどうするのかについてはあまり考えていませんでした。データベースを設計するときは、一般的な推奨事項に従い、足を踏み入れないようにしましたが、「エンティティ属性値パターンを使用しない」などの一般的なヒントを超えました私たちは入りませんでした。データの冗長性を回避する正規化の原則に基づいて設計されており、特定のクエリの高速化を気にしませんでした。最初のユーザーが到着するとすぐに、パフォーマンスの問題が発生しました。いつものように、私たちはこれに対して完全に準備ができていませんでした。最初の問題は単純でした。原則として、すべてが新しいインデックスを追加することによって決定されました。しかし、単純なパッチが機能しなくなったときがきました。十分な経験がなく、問題の原因を理解することがますます困難になっていることに気づき、サーバーを正しくセットアップし、監視を接続し、統計を取得するためにどこを探すべきかを示すのに役立つ専門家を雇いました。

2枚目の写真。統計的


したがって、1日に約1万のクエリがデータベースで実行されます。これらの1万のうち、平均実行時間が0.1〜0.3ミリ秒で200〜300万回実行されるモンスターがあり、1日あたり100回呼び出される平均実行時間が30秒のクエリがあります。

1万件のクエリすべてを最適化することは不可能だったため、データベースのパフォーマンスを正しく向上させるために、どこに努力を向ければよいかを判断することにしました。数回繰り返した後、リクエストをタイプに分割し始めました。

TOPクエリ


これらは、最も時間がかかる（合計時間）最も難しいクエリです。これらは、非常に頻繁に呼び出されるクエリ、または非常に長い時間がかかるクエリです（速度を上げるための闘争の最初の反復でも、長くて頻繁なクエリは最適化されています）。結果として、サーバーは合計で実行に最も多くの時間を費やしています。さらに、上位のリクエストを合計実行時間で分離し、IO時間で分離することが重要です。このようなクエリを最適化する方法は少し異なります。

すべての企業の通常の慣行は、TOPリクエストを処理することです。それらの数は少なく、1つの要求を最適化してもリソースの5-10％が解放されます。ただし、プロジェクトが古くなるにつれて、TOPクエリの最適化はますます重要なタスクになります。すべての簡単な方法はすでに解決されており、最も「難しい」リクエストではリソースの「わずか5〜5％」しか取り除かれません。 TOPクエリの合計時間が30〜40％未満の場合は、すばやく動作するように既に努力しているので、次のグループからのクエリの最適化に移りましょう。
このグループに含める上位のクエリの数に関する質問に答える必要があります。私は通常10以上20以下を取ります。TOPグループの最初と最後の時間の違いが10倍以下になるようにしています。つまり、クエリの実行時間が1か所から10か所に急激に減少した場合は、TOP-10を使用し、ドロップがより滑らかな場合は、グループサイズを15または20に増やします。


ミドル農民（ミディアム）


これらは、最後の5〜10％を除いて、TOPの直後に続くすべての要求です。通常、これらの特定の要求を最適化するには、サーバーのパフォーマンスを大幅に向上させる機能があります。これらのクエリは、最大80％の「重み」があります。しかし、彼らのシェアが50％を超えたとしても、彼らをもっと詳しく見る時が来ました。

尾


言われたように、これらの要求は最後に行き、時間の5-10％かかります。自動クエリ分析ツールを使用しない場合にのみ、それらを忘れることができます。その場合、最適化も安価になります。

各グループを評価するには？

私はPostgreSQLのそのような評価を行うのに役立つSQLクエリを使用します（他の多くのDBMSでも同様のクエリを記述できると確信しています）

TOP-MEDIUM-TAILグループのサイズを評価するSQLクエリ
SELECT sum(time_top) AS sum_top, sum(time_medium) AS sum_medium, sum(time_tail) AS sum_tail
FROM
(
  SELECT CASE WHEN rn <= 20              THEN tt_percent ELSE 0 END AS time_top,
         CASE WHEN rn > 20 AND rn <= 800 THEN tt_percent ELSE 0 END AS time_medium,
         CASE WHEN rn > 800              THEN tt_percent ELSE 0 END AS time_tail
  FROM (
    SELECT total_time / (SELECT sum(total_time) FROM pg_stat_statements) * 100 AS tt_percent, query,
    ROW_NUMBER () OVER (ORDER BY total_time DESC) AS rn
    FROM pg_stat_statements
    ORDER BY total_time DESC
  ) AS t
)
AS ts


クエリ結果は3つの列で、各列には、このグループからの要求の処理に費やされた時間の割合が含まれています。クエリ内には、1つのグループからの要求を別のグループから分離する2つの番号（私の場合は20と800）があります。

これが、最適化作業の開始時のリク​​エストの比率がおおよそ相関する方法です。

最初、TOPの失敗はTOPクエリをヒットしました。時間が経つにつれ、小児疾患がなくなり、TOPリクエストの割合が減少し、難しいリクエストを加速するためにより多くの努力をしなければなりませんでした。

リクエストのテキストを取得するには、そのようなリクエストを使用します
SELECT * FROM (
  SELECT ROW_NUMBER () OVER (ORDER BY total_time DESC) AS rn, total_time / (SELECT sum(total_time) FROM pg_stat_statements) * 100 AS tt_percent, query
  FROM pg_stat_statements
  ORDER BY total_time DESC
) AS T
WHERE
rn <= 20 -- TOP
-- rn > 20 AND rn <= 800 -- MEDIUM
-- rn > 800  -- TAIL

特定の要求がそれぞれ3〜1000倍高速化される場合があります。印象的なパフォーマンスにもかかわらず、最初は10ミリ秒実行されたクエリを最適化する意味がなく、最も重いクエリの300分の1に含まれ、データベースの総ロード時間には100分の1パーセントかかりました。しかし、同じレシピを同様のリクエストのグループに適用すると、数パーセントを取り戻しました。何百ものクエリをすべて手動で表示する時間を無駄にしないために、正規表現を使用して類似のクエリを見つける簡単なスクリプトをいくつか作成しました。その結果、クエリグループの自動検索により、適度な労力を費やすことでパフォーマンスをさらに向上させることができました。

その結果、同じハードウェアに3年間取り組んできました。 1日の平均負荷は約30％で、ピーク時には70％に達します。リクエスト数とユーザー数は約10倍に増加しています。そして、これらすべては、TOP-MEDIUMクエリのこれらの非常にグループの絶え間ない監視のおかげです。 TOPグループに新しいリクエストが表示されたらすぐに、それを分析してスピードアップを試みます。クエリ分析スクリプトを使用して、MEDIUMグループを週に1回確認します。すでに最適化の方法がわかっている新しいリクエストに遭遇した場合は、すぐに変更します。時には、一度に複数のクエリに適用できる新しい最適化方法を見つけることがあります。

予測によると、現在のサーバーはユーザー数の増加にさらに3〜5倍耐えます。確かに、スリーブにはもう1つの切り札があり、SELECTクエリはミラーにまだ翻訳されていません。しかし、「重砲」をオンにする前に、まず「スマート」最適化の可能性を完全に使い果たしたいので、私たちはこれを意識的に行いません。
行われた作業を批判的に見ると、垂直スケーリングの使用が提案される場合があります。専門家の時間を無駄にする代わりに、より強力なサーバーを購入します。特に垂直スケーリングの限界をまだ使い果たしていないため、サーバーのコストはそれほど高くない場合があります。ただし、要求の数のみが10倍に増加しました。数年前から、システムの機能性は向上しており、今ではより多くの種類のクエリがあります。キャッシュが原因であった機能は、より少ないリクエスト、さらにはより効率的なリクエストによって実行されます。したがって、別の5を安全に乗算して、実際の加速係数を取得できます。したがって、最も保守的な見積もりによると、加速度は50倍以上だったと言えます。サーバーを垂直に50回振ると、コストが高くなります。特に考慮最適化が完了するといつでも機能し、レンタルサーバーの料金は毎月発生します。
