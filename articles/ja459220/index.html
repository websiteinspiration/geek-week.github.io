メモリーリークをチェックするための統合テスト

iOS用のSoundCloudアプリケーションの開発中に、多くの単体テストを作成します。単体テストは素晴らしいですね。それらは短く、（うまくいけば）読みやすく、私たちが書いたコードが期待通りに機能するという確信を与えてくれます。しかし、ユニットテストは、その名前が示すように、コードの1ブロックのみをカバーします。ほとんどの場合、関数またはクラスです。では、クラス間の相互作用に存在するエラー（メモリリークなどのエラー）をどのようにキャッチするのでしょうか。


メモリーリーク


場合によっては、メモリリークエラーの検出が非常に困難です。デリゲートへの強い参照の可能性がありますが、検出がはるかに難しいエラーもあります。たとえば、次のコードにメモリリークが含まれている可能性があることは明らかですか。

final class UseCase {
    weak var delegate: UseCaseDelegate?
    private let service: Service

    init(service: Service) {
        self.service = service
    }

    func run() {
        service.makeRequest(handleResponse)
    }

    private func handleResponse(response: ServiceResponse) {
        // some business logic and then...
        delegate.operationDidComplete()
    }
}


以来サービスをされて、すでに実施され、その動作に関する保証はありません。自分をキャプチャするプライベート関数にhandleResponseを渡すことにより、サービスにUseCaseへの強い参照を提供します。場合はサービスがこのリンクを維持することを決定-私たちはこれが起こらないという保証がない-そして、メモリリークが発生します。しかし、コードをざっと見てみると、これが実際に起こり得ることは明らかではありません。John Sundellによる 素晴らしい投稿もある

ユニットテストを使用してクラスのメモリリークを検出することについて。しかし、メモリリークをスキップするのが非常に簡単な上記の例では、そのような単体テストの記述方法が必ずしも明確ではありません。 （もちろん、ここでは経験の観点から話をしていません。）

Guillermeが最近の投稿で書いたように、iOS用のSoundCloudアプリケーションの新機能は「クリーンなアーキテクチャパターン」に従って書かれています。これはほとんどの場合VIPERの一種です。これらのVIPERモジュールのほとんどは、ModuleFactoryと呼ばれるものを使用して構築されています。このようなModuleFactoryは、いくつかの入力、依存関係、および構成を受け取り、UIViewControllerを作成しますこれはすでにモジュールの残りの部分に接続されており、ナビゲーションスタックにプッシュできます。

このVIPERモジュールには、いくつかのデリゲート、オブザーバー、および暴走フォールトが存在する可能性があり、それぞれがコントローラーをナビゲーションスタックから削除した後、コントローラーがメモリに残る可能性があります。これが発生すると、メモリの量が増加し、オペレーティングシステムがアプリケーションを停止することを決定する場合があります。

では、可能な限り少数の単体テストを記述することで、非常に多くの潜在的なリークをカバーすることは可能ですか？そうでない場合、これはすべて時間の浪費でした。

統合テスト


この投稿のタイトルから推測したように、答えはイエスです。そして、これは統合テストを通じて行います。統合テストの目的は、オブジェクトが互いにどのように相互作用するかをテストすることです。もちろん、VIPERモジュールはオブジェクトのグループです。メモリリークは、絶対に回避したい相互作用の1つの形式です。

私たちの計画は単純です：ModuleFactoryを使用して、VIPERモジュールをインスタンス化します。次に、UIViewControllerへのリンクを削除し、モジュールのすべての重要な部分が一緒に破棄されることを確認します。

私たちが直面している最初の問題は、本質的にはVIPERのどの部分にも簡単にアクセスできないことですUIViewControllerを除くモジュール。ModuleFactoryの唯一のパブリック関数はfunc make（）-> UIViewControllerです。しかし、テスト専用の別のエントリポイントを追加するとどうなるでしょうか。この新しいメソッドは内部で宣言されるため、@ testableインポート（ModuleFactoryフレームワーク）を介してのみアクセスできます。モジュールのすべての最も重要な部分へのリンクを返します。テストに弱いリンクが入るのを待つことができます。最終的には次のようになります。

public final class ModuleFactory {
    //     ,   ...
    public func make() -> UIViewController {
        makeAndExpose().view
    }

    typealias ModuleComponents = (
        view: UIViewController,
        presenter: Presenter,
        Interactor: Interactor
    )

    func makeAndExpose() -> ModuleComponents {
      // Set up code, and then...
      return (
          view: viewController,
          presenter: presenter,
          interactor: interactor
      )
    }
}


これにより、オブジェクトデータに直接アクセスできないという問題が解決されます。明らかに、これは理想的ではありませんが、私たちのニーズを満たしているので、テストの作成に移りましょう。次のようになります。

final class ModuleMemoryLeakTests: XCTestCase {
    //      .    
    //    .
    private var view: UIViewController?
    //       
    //   ,   
    // UIKit,   UIViewController  .
    private weak var presenter: Presenter?
    private weak var interactor: Interactor?

    //   setUp    ModuleFactory 
    //   makeAndExpose.     ,  
    //     ModuleComponents
    // ,          .
    //     .
    func setUp() {
        super.setUp()
        let moduleFactory = ModuleFactory(/* mocked dependencies & config */)
        let components = moduleFactory.makeAndExpose()
        view = components.view
        presenter = components.presenter
        interactor = components.interactor
    }

    //   ,   tearDown   ,
    //        ,     ,  
    //     .
    func tearDown() {
        view = nil
        presenter = nil
        interactor = nil
        super.tearDown()
    }

    func test_module_doesNotLeakMemory() {
        //   ,      .
        //      , 
        //          setUp.
        XCTAssertNotNil(presenter)
        XCTAssertNotNil(interactor)

        //        .
        //    ,  
        //     ,   
        //      .
        view = nil

        // ,  ,   
        //  Presenter  Interactor   .
        //  ,      
        //  ,    .
        XCTAssertNil(presenter)
        XCTAssertNil(interactor)
    }
}


したがって、VIPERモジュールでメモリリークを検出する簡単な方法があります。これは決して理想的ではなく、テストする新しいモジュールごとにある程度のユーザー作業が必要ですが、これは、考えられるすべてのメモリリークに対して個別の単体テストを作成するよりもはるかに少ない作業です。また、疑わしいメモリリークの特定にも役立ちます。実際、これらのテストのいくつかを書き込んだ後、テストに合格しないことが判明し、調査の結果、モジュールにメモリリークが見つかりました。修正後、テストを繰り返す必要があります。

また、モジュールの統合テストのより一般的なセットを記述するための出発点にもなります。最後に、Presenterとの強いリンクを維持するだけでUIViewControllerをmockに置き換えます。ユーザー入力を偽装してから、プレゼンターのメソッドを呼び出し、Viewでのデータのダミー表示を確認します。
