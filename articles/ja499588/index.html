JWTを使用した.NET Core gRpcでの認証

この記事では、JWTを使用したgRpcサービスでのAPI認証の機能について説明します。.NET Core WebAPIでJWTおよびHTTPヘッダーを使用することに慣れていると思いますので、これらの詳細については説明しません。gRpcで認証を実装しようとしたときに、ほとんどの例がコンソールアプリケーションを使用して記述されているという事実に遭遇しました。これは、私の意見では開発者が住んでいる現実とはかけ離れています。たとえば、サービスメソッドを呼び出すたびにチャネルを作成したくありません。また、各リクエストでトークンとユーザー情報を送信することについて心配したくありません。代わりに、これらすべてを自分で処理するインフラストラクチャレベルが欲しいです。このトピックがあなたにとって興味深いものである場合は、カットの下にもっとあります。この記事のすべての例は、.NET Core 3.1に有効です。

使用例


トピックを掘り下げる前に、記事で使用されている例を説明する価値があります。ソリューション全体は、WebサイトとgRpcサービス（以下、API）の2つのアプリケーションで構成されます。どちらも.NET Core 3.1で記述されています。これが許可されていれば、ユーザーはログインして一部のデータを表示できます。 Webサイトはユーザーデータを保存せず、認証プロセスでAPIに依存します。 gRpcサービスと通信するには、ウェブサイトに有効なJWTトークンが必要ですが、このトークンはアプリケーションでのユーザー認証とは関係ありません。 WebアプリケーションはサイドでCookieを使用します。 APIがサービスへのリクエストを行うユーザーを認識するために、これに関する情報がJWTトークンとともに送信されますが、トークン自体ではなく、追加のHTTPヘッダーとともに送信されます。下の図は、先ほど説明したシステムのスキーマの例を示しています。


ここで、この例を実行したとき、APIに最も適切な認証方法を実装するという目標がなかったことに注意してください。いくつかのベストプラクティスを見たい場合は、OpenID Connect仕様をご覧ください。しかし、問題を解決して時間と費用を節約できるものと比較して、最も正しい解決策は冗長である場合があるように思えます。

gRpcサービスでJWT認証を有効にする


gRpcサービスの構成は、.NET Core APIが必要とする通常の構成と同じです。追加のプラスは、HTTPプロトコルとHTTPSプロトコルで違いがないことです。簡単に言うと、Startup.csファイルにミドルウェアだけでなく、標準の認証サービスと承認サービスを追加する必要があります。ミドルウェアを追加する場所は重要です。ルーティングとポイントの間に正確に追加する必要があります（一部のコードが欠落しています）。

public void Configure(...) {
    app.UseRouting();
    
    app.UseAuthentication();
    app.UseAuthorization();
    
    app.UseEndpoints(...
}

ただし、サービスが登録される場所はそれほど重要ではありません。メソッドにConfigureServices（）メソッドを追加するだけです。ただし、ここではJWTトークンチェックを設定する必要があります。ここで定義できますが、別のクラスにプルすることをお勧めします。したがって、コードは次のようになります。

public void ConfigureServices(...) {
    services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
        .AddJwtBearer(o => {
            var validator = new JwtTokenValidator(...);
            o.SecurityTokenValidators.Add(validator);
        });
    services.AddAuthorization();
}

JwtTokenValidator クラスは、検証ロジックを定義するクラスです。正しい設定でTokenValidationParametersクラスを作成する必要があります。これにより、残りのJWT検証作業が実行されます。おまけとして、ここにセキュリティの層を追加できます。JWTはよく知られている形式であるため、必要になる場合があります。JWTを使用している場合は、jwt.ioにアクセスして情報を確認できます。私はJWTに追加の暗号化を追加することを好みます。これにより、復号化がより困難になります。バリデーターは次のようになります。

public class JwtTokenValidator : ISecurityTokenValidator
{
    public bool CanReadToken(string securityToken) => true;
    
    public ClaimsPrincipal ValidateToken(string securityToken, TokenValidationParameters validationParameters, out SecurityToken validatedToken)
    {
        var handler = new JwtSecurityTokenHandler();
        var tokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer = "your string",
            ValidAudience = "your string",
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("your secrete code"))
        };
        
        var claimsPrincipal = handler.ValidateToken(token, tokenValidationParameters, out validatedToken);
        return claimsPrincipal;
    }
    
    public bool CanValidateToken { get; } = true;
    public int MaximumTokenSizeInBytes { get; set; } = int.MaxValue;
}

そして、それがすべてのAPI側のニーズです。クライアントのセットアップ履歴は、選択したHTTPまたはHTTPSプロトコルに応じて少し長く、少し異なります。

すべてのリクエストでHTTPヘッダーをgRpcサービスに送信する


これは公式ドキュメントからわかるかもしれませんが、実際には、ダムコンソールプログラム以外では使用できません。たとえば、こちらで確認できます。

var channel = GrpcChannel.ForAddress("https://localhost:5001");
var client = new Greeter.GreeterClient(channel);
var response = await client.SayHelloAsync(
    new HelloRequest { Name = "World" });
Console.WriteLine(response.Message);

これを実際のプロジェクトで使用するには、集中構成とDIが必要ですが、ほとんど考慮されていません。これがあなたがしなければならないことです。まず、必要なNuGetパッケージをプロジェクトに追加する必要があります。Grpc.Tools パッケージはプロジェクトのビルド時にプロトタイプを作成するのに役立ち、Grpc.Net.ClientFactoryはDIをセットアップするのに役立ちます。 gRpcを使用する場合、要求/応答チェーンの途中のどこかに処理を実装する必要がある場合は、gRpc.Coreの一部であるInterceptorから継承したクラスを使用する必要があります。サービス内でHttpContext.User.Identityにアクセスする必要がある場合は、IHttpContextAccessorインターフェイスを追加できます

dotnet add package Grpc.Net.Client
dotnet add package Google.Protobuf
dotnet add package Grpc.Tools
dotnet add package Grpc.Net.ClientFactory



サービスに追加します（これにはサービスへの追加登録が必要です）。以下をStartup.csファイルに追加する必要があります。

services.AddTransient<AuthHeadersInterceptor>();
services.AddHttpContextAccessor();

var httpClientBuilder = services.AddGrpcClient<MygRpcService.MygRpcServiceClient>(o => { o.Address = new Uri("grpc-endpoint-url"); });
httpClientBuilder.AddInterceptor<AuthHeadersInterceptor>();              
httpClientBuilder.ConfigureChannel(o => o.Credentials = ChannelCredentials.Insecure);

AuthHeadersInterceptor クラスは、Interceptorクラスから派生した独自のクラスです。これはIHttpContextAccessorを使用し、.AddHttpContextAccessor（）を登録するとこれを行うことができます。

httpClientBuilder.ConfigureChannel(o => o.Credentials = ChannelCredentials.Insecure);

HTTPでの作業には必要ですが、これだけでは不十分です。また、この行をConfigure（）メソッドから除外する必要があります。

app.UseHttpsRedirection();

さらに、gRpcチャネルを作成する前に、特別な設定を確立するために踊る必要があります。これは、アプリケーションの起動時に1回だけ実行できます。そのため、上記の削除した行とほぼ同じ位置に追加しました。これはHTTPの場合にのみ呼び出す必要があります。

AppContext.SetSwitch("System.Net.Http.SocketsHttpHandler.Http2UnencryptedSupport", true);

HTTPSの構成機能


WindowsおよびLinuxでSSLを使用する場合、いくつかの問題があります。Windowsコンピューターで開発し、Linuxベースのイメージを使用してDocker / Kubernetesにデプロイする場合があります。この場合、構成は多くの投稿で説明されているほど単純ではありません。この構成については別の記事で説明します。ここでは、コードのみに触れます。

SSL資格情報を使用するには、gRpcチャネルを再構成する必要があります。DockerにデプロイしてLinuxベースのイメージを作成する場合、無効な証明書を許可するようにHttpClientを構成する必要がある場合もあります。HttpClientはチャネルごとに作成されます。

httpClientBuilder.ConfigureChannel(o =>
{
    // add SSL credentials
    o.Credentials = new SslCredentials();
    // allow invalid/untrusted certificates
    var httpClientHandler = new HttpClientHandler
    {
        ServerCertificateCustomValidationCallback = HttpClientHandler.DangerousAcceptAnyServerCertificateValidator
    };
    var httpClient = new HttpClient(httpClientHandler);
    o.HttpClient = httpClient;
});

HTTPヘッダーの追加


ヘッダーは、インターセプタークラス（Interceptorの後続）に追加されます。gRpcはメタデータの概念を使用します。メタデータの概念は、リクエストとしてヘッダーとして送信されます。インターセプタークラスは、呼び出しコンテキストのメタデータを追加する必要があります。

public class AuthHeadersInterceptor : Interceptor
{
    public AuthHeadersInterceptor(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }
    
    public override AsyncUnaryCall<TResponse> AsyncUnaryCall<TRequest, TResponse>(TRequest request, ClientInterceptorContext<TRequest, TResponse> context, AsyncUnaryCallContinuation<TRequest, TResponse> continuation)
    {
        var metadata = new Metadata
        {
            {HttpHeaderNames.Authorization, $"Bearer <JWT_TOKEN>"}
        };
        var userIdentity = _httpContextAccessor.HttpContext.User.Identity;
        if (userIdentity.IsAuthenticated)
        {
            metadata.Add(HttpHeaderNames.User, userIdentity.Name);
        }
        var callOption = context.Options.WithHeaders(metadata);
        context = new ClientInterceptorContext<TRequest, TResponse>(context.Method, context.Host, callOption);
        
        return base.AsyncUnaryCall(request, context, continuation);
    }
}

シナリオでは、gRpcサービスを呼び出すだけの場合、AsyncUnaryCallメソッドをオーバーライドするだけで済みます。もちろん、JWTトークンは構成ファイルに保存できます。
