Patroniで信頼性の高いPostgreSQLクラスターを構築した方法



今日、サービスの高可用性は、大規模で高価なプロジェクトだけでなく、いつでもどこでも必要です。「申し訳ありませんが、メンテナンスを行っています」というメッセージが表示された一時的に利用できないサイトも引き続き発生しますが、通常は見事に笑顔になります。これにクラウドでの生活を追加してください。追加のサーバーを起動するときは、APIを1回呼び出すだけでよく、「鉄」の操作について考える必要はありません。また、重要なシステムがクラスタテクノロジーと冗長性を使用して信頼性の高い方法で作成されなかった理由はもはやありません。

私たちは、サービス内のデータベースの信頼性を確保するために検討したソリューションと、その結果をお知らせします。さらに、広範囲にわたる結論のあるデモ。

高可用性アーキテクチャのレガシー


これは、さまざまなオープンソースシステムの開発のコンテキストでさらによく見られます。古いソリューションでは、需要の増加に応じて高可用性テクノロジーを追加する必要がありました。そしてそれらの品質は異なっていました。次世代ソリューションは、アーキテクチャのコアに高可用性をもたらします。たとえば、MongoDBはクラスターを主な使用例として位置付けています。クラスタは水平方向にスケーリングします。これは、このDBMSの強力な競争力です。

PostgreSQLに戻ります。これは最も人気のあるオープンソースプロジェクトの1つで、最初のリリースは前世紀の95年目に行われました。長い間、プロジェクトチームは、高可用性をシステムが対処する必要のあるタスクとは考えていませんでした。したがって、データコピーを作成するためのレプリケーションテクノロジは、2006年にバージョン8.2でのみ統合されましたが、ファイルされました（ログ配布）。 2010年に、バージョン9.0はストリーミングレプリケーションを導入し、さまざまなクラスターを作成するための基礎となっています。実際、これは、Enterprise SQLまたは最新のNoSQLの後にPostgreSQLを知る人にとって非常に驚くべきことです。コミュニティの標準的なソリューションは、同期または非同期のレプリケーションを備えたマスターレプリカのカップルです。同時に、ウィザードはドレインで手動で切り替えられ、クライアントの切り替えの問題も個別に解決することが提案されています。

信頼性の高いPostgreSQLを作成することを決めた方法とこれのために選択したもの


ただし、常に注意を払う必要のないフォールトトレラントソリューションの構築に役立つプロジェクトやツールが大量になければ、PostgreSQLはそれほど普及していなかっただろう。クラウドではMail.ruクラウドソリューションの開始DBaaSから（MCS）単一のPostgreSQLサーバと非同期レプリケーションで、マスターレプリカのペア使用可能でした。

当然、私たちは全員の生活を簡素化し、PostgreSQLのインストールを利用できるようにしたいと考えました。PostgreSQLのインストールは、非常にアクセスしやすいサービスの基盤として機能し、常に監視して夜中に目を覚まして切り替える必要はありませんでした。このセグメントには、実績のある古いソリューションと、最新の開発を使用する新世代のユーティリティの両方があります。

今日、高可用性の問題は冗長性（言うまでもなく）に依存していませんが、コンセンサス-リーダーを選択するアルゴリズム（リーダー選挙）に依存しています。ほとんどの場合、重大な事故はサーバーの不足が原因ではなく、コンセンサスの問題が原因で発生します。新しいリーダーが出なかった、2人のリーダーが異なるデータセンターに現れた、など。例として、Github MySQLクラスターでのクラッシュがあります。詳細な事後分析が書き込まれています。

この問題の数学的根拠は非常に深刻です。一方では、HAソリューションを構築する可能性に理論的な制限を課すCAP定理があり、他方では、PaxosやRaftなどの数学的に証明されたコンセンサス決定アルゴリズムがあります。。これに基づいて、Zookeeperなどの非常に人気のあるDCS（分散型コンセンサスシステム）があります。したがって、意思決定システムが独自に記述された独自のアルゴリズムの一部で動作する場合は、非常に注意する必要があります。膨大な数のシステムを分析した後、主にZalandoによって開発されたオープンソースシステムであるPatroniに落ち着きました。

叙情的な余談として、マルチマスターソリューション、つまり録音に合わせて水平方向にスケーリングできるクラスターも検討したと言います。ただし、2つの主な理由により、このようなクラスターを作成しないことにしました。第一に、そのようなソリューションは非常に複雑であり、したがってより多くの脆弱性があります。すべてのケースで安定した決定を下すのは難しいでしょう。次に、この場合、PostgreSQLは純粋（ネイティブ）でなくなり、一部の関数は使用できなくなります。一部のアプリケーションでは、作業中にバグが隠れている可能性があります。

パトロニ


では、Patroniはどのように機能するのでしょうか。開発者はホイールを再発明せず、実証済みのDCSソリューションの1つをベースとして使用することを提案しました。構成の同期に関するすべての問題、リーダーの選択、およびクォーラムが彼に与えられます。これにはetcdを選択しました。

さらに、Patroniは、PostgreSQLのすべての設定とレプリケーション設定の適切な適用、およびスイッチオーバーとフェイルオーバー（つまり、通常の非標準の切り替えウィザード）でのコマンドの実行に従事しています。具体的には、MCSクラウドでは、ウィザード、同期レプリカ、および1つ以上の非同期レプリカからクラスターを作成できます。同期レプリカの存在により、少なくとも2台のサーバー上のデータの安全性が確保され、このレプリカが主な「マスターの候補」になります。

etcdは同じサーバーにデプロイされるため、最適なクォーラム値のために推奨されるサーバーの数は3または5です。そのようなクラスターは、読み取り用に水平方向にスケーリングされます（上記の書き込み用のスケーリングについて書きました）。それにもかかわらず、特に高負荷では、非同期レプリカが遅れていることに注意してください。

このような読み取りスタンバイレプリカの使用は、レポートまたは分析タスクに正当化され、マスターサーバーの負荷を軽減します。

このようなクラスターを自分で作成する場合は、次のものが必要です。

  • 3台以上のサーバーを準備し、それらの間のIPアドレスとファイアウォールルールを構成します。
  • サービスetcd、Patroni、PostgreSQLのパッケージをインストールします。
  • etcdクラスターを構成します。
  • PostgreSQLで動作するようにpatroniサービスを構成します。

つまり、全体で、ダースの構成ファイルを正しく構成し、どこでもミスをしないようにする必要があります。これを行うには、たとえば、Ansibleなどの構成管理ツールを使用する価値があります。ただし、高可用性のTCPバランサーはまだありません。それをすることは別の仕事です。

既製のクラスターを必要とするが、これをすべて試したくない人のために、私たちは私たちの生活を単純化しようとし、クラウド内のPatroniで既製のクラスターを作成しました。無料でテストできます。クラスタ自体に加えて、次のことを行いました。

  • TCPバランサー 異なるポートでは、常に現在のマスター、同期または非同期のレプリカをそれぞれポイントします。
  • アクティブなPatroniウィザードを切り替えるためのAPI。

これらは、MCSクラウドAPIとWebコンソールの両方を介して接続できます。

デモ


MCSクラウドでPostgreSQLクラスターの機能をテストするために、DBMSに問題が発生した場合のライブアプリケーションの動作を見てみましょう。

以下は、人工的なイベントをログに記録し、これを画面に報告するアプリケーションのコードです。エラーが発生した場合は、これを報告し、Ctrl + Cの組み合わせで停止するまで、サイクルで作業を続行します。

from __future__ import print_function

from datetime import datetime
from random import randint
from time import sleep
import psycopg2


def main():
    try:
        connection = psycopg2.connect(user = "admin",
                                      password = "P@ssw0rd",
                                      host = "89.208.87.38",
                                      port = "5432",
                                      database = "myproddb")

        cursor = connection.cursor()
        cursor.execute("SELECT version();")
        record = cursor.fetchone()
        print("Connection opened to", record[0])

        cursor.execute(
            "INSERT INTO log VALUES ({});".format(randint(1, 10000)))
        connection.commit()
        cursor.execute("SELECT COUNT(event_id) from log;")
        record = cursor.fetchone()
        print("Logged a value, overall count: {}".format(record[0]))
    except Exception as error:
        print ("Error while connecting to PostgreSQL", error)
    finally:
        if connection:
            cursor.close()
            connection.close()
            print("Connection closed")


if __name__ == '__main__':
    try:
        while True:
            try:
                print(datetime.now())
                main()
                sleep(3)
            except Exception as e:
                print("Caught error:\n", e)
                sleep(1)
    except KeyboardInterrupt:
        print("exit")

結論


そのようなクラスターは、管理者に多大な利点を提供するように思えます。実際、データベースサーバーの深刻な故障や誤動作は、アプリケーションにとって、したがってユーザーにとっては目立ちません。急いで何かを修理したり、一時的な構成やサーバーなどに切り替える必要はありません。また、このソリューションを既製のサービスとしてクラウドで使用する場合、準備に時間を費やす必要はありません。もっと面白いことをすることができるでしょう。
