Goプログラムの実行時に変数のサイズを決定する方法

要約：このノートでは、Goアプリケーションのコンポーネントによるメモリ消費を分析する方法の1つについて説明します。



多くの場合、データ構造はプログラムのメモリに格納され、プログラムの実行中に動的にサイズが変化します。このような構造の例としては、データキャッシュやプログラム操作ログ、外部システムから受信したデータなどがあります。この場合、メモリの消費量が増加し、機器の機能が不十分で、特定のリークメカニズムが明確でない状況が発生する可能性があります。



Goアプリケーションのプロファイルを作成する主な方法は、「net / http / pprof」パッケージからpprofツールを接続することです。その結果、実行中のプログラムでメモリ割り当てを含むテーブルまたはグラフを取得できます。ただし、このツールを使用するには非常に大きなオーバーヘッドが必要であり、特に実際のデータでプログラムの複数のインスタンスを実行できない場合は、適用できない場合があります。



この場合、たとえば、システム統計を表示したり、測定値を監視システムに転送したりするために、プログラムオブジェクトによるメモリ消費をオンデマンドで測定したいという要望があります。ただし、言語では、これは一般的に不可能です。 Goには、プログラムの実行中に変数のサイズを変更するツールはありません。



したがって、私はそのような機会を提供する小さなパッケージを書くことにしました。主なツールはリフレクションです（パッケージ"reflection"）このようなプロファイリングアプリケーションの問題に関心のあるすべての人に、さらに読んでもらいます。







まず、組み込み関数について簡単に説明します。 



unsafe.Sizeof(value)


そして 



reflect.TypeOf(value).Size()


これらの関数は同等であり、変数のサイズを決定するためにインターネットで推奨されることがよくあります。しかし、これらの関数は実際の変数のサイズを返しませんが、変数コンテナーのサイズ（バイト単位）を返します（大体-ポインターのサイズ）。たとえば、int64型の変数の場合、これらの関数は正しい結果を返します。これは、この型の変数には実際の値が含まれ、それへの参照は含まれないためです。ただし、スライスや文字列など、実際の値へのポインタを含むデータ型の場合、これらの関数は、この型のすべての変数に対して同じ値を返します。この値は、変数データへのリンクを含むコンテナのサイズに対応しています。例を挙げて説明します。 



func main() {
	s1 := "ABC"
	s2 := "ABCDEF"
	arr1 := []int{1, 2}
	arr2 := []int{1, 2, 3, 4, 5, 6}
	fmt.Printf("Var: %s, Size: %v\n", s1, unsafe.Sizeof(s1))
	fmt.Printf("Var: %s, Size: %v\n", s2, unsafe.Sizeof(s2))
	fmt.Printf("Var: %v, Size: %v\n", arr1, reflect.TypeOf(arr1).Size())
	fmt.Printf("Var: %v, Size: %v\n", arr2, reflect.TypeOf(arr2).Size())
}


その結果、以下が得られます。 



Var: ABC, Size: 16
Var: ABCDEF, Size: 16
Var: [1 2], Size: 24
Var: [1 2 3 4 5 6], Size: 24


ご覧のとおり、変数の実際のサイズは計算されていません。



標準ライブラリには、変数のサイズをバイト単位で返すbinary.Size（）関数がありますが、固定サイズのタイプの場合のみです。つまり、構造体のフィールドで文字列、スライス、連想配列、または単にintが検出された場合、関数は適用されません。ただし、この機能をサイズパッケージの基礎として使用し、上記のメカニズムの機能を固定サイズのないデータ型に拡張しようとしました。



プログラムの操作中にオブジェクトのサイズを決定するには、そのタイプと、それが構造体である場合はすべてのネストされたオブジェクトのタイプを理解する必要があります。分析が必要な最終的な構造は、通常、ツリーの形で表示されます。したがって、複雑なデータ型のサイズを決定するには、再帰を使用する必要があります。



したがって、任意のオブジェクトで消費されるメモリ量の計算は次のように表されます。



  • 単純な（非複合）型の変数のサイズを決定するアルゴリズム。
  • 配列の要素、構造体フィールド、キー、および連想配列の値のアルゴリズムの再帰呼び出し;
  • 無限ループの定義;


（配列や構造体ではなく）単純型変数の実際のサイズを決定するには、上記のリフレクションパッケージのSize（）関数を使用できます。この関数は、実際の値を含む変数に対して正しく機能します。配列、文字列である変数の場合。要素またはフィールドを通過して各要素の値を計算するために必要な値へのリンクが含まれています。 変数のタイプと値を分析するために、リフレクションパッケージは変数を空のインターフェイス（インターフェイス{}）にパックします。 Goでは、空のインターフェースに任意のオブジェクトを含めることができます。さらに、Goのインターフェースは、実際の値のタイプと実際の値へのリンクの2つのフィールドを含むコンテナーで表されます。







-それは、メソッド自体の名前のための基礎として役立っているその逆空インターフェースと副への分析値のマッピングた反射。



Goでリフレクションがどのように機能するかをよりよく理解するには、Go の公式ブログにあるRob Pikeの記事をお勧めします。この記事の翻訳はハブレにありました。



最後に、サイズパッケージが開発されました。これは、次のようにプログラムで使用できます。 



package main

import (
	"fmt"

	"github.com/DmitriyVTitov/size"
)

func main() {
	a := struct {
		a int
		b string
		c bool
		d int32
		e []byte
		f [3]int64
	}{
		a: 10,                    // 8 bytes
		b: "Text",                // 4 bytes
		c: true,                  // 1 byte
		d: 25,                    // 4 bytes
		e: []byte{'c', 'd', 'e'}, // 3 bytes
		f: [3]int64{1, 2, 3},     // 24 bytes
	}

	fmt.Println(size.Of(a))
}

// Output: 44


備考：



  • 実際には、大きな入れ子で約10 GBのボリューム構造のサイズを計算するには、10〜20分かかります。これは、リフレクションが非常に負荷の高い操作であり、各変数を空のインターフェースにパックし、その後分析する必要があるという事実の結果です（上のリンクの記事を参照）。
  • 大規模な構造の分析中の実際のシステムでは実際のデータが変化する可能性が高いため、比較的低速であるため、変数のサイズを概算するにはパッケージを使用する必要があります。または、必要に応じて、ミューテックスを使用して計算時にデータへの排他的アクセスを提供します。
  • プログラムは、配列、インターフェース、および連想配列の「コンテナー」のサイズを考慮しません（これらは、配列とスライスの場合は24バイト、マップとインターフェースの場合は8バイトです）。したがって、小さいサイズの要素が多数ある場合、損失は大きくなります。
