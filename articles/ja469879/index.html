ワンクリックでダブルVPN。入口と出口のIPアドレスを簡単に分割する方法



TL; DR この記事では、VPNクライアントを接続するためのIPアドレスが、クライアントがインターネットにアクセスするIPアドレスとは異なるVPNサーバーを構成する最も簡単な方法について説明します。

VPNを使用してインターネット上のプライバシーを保護し、独自のサーバーをレンタルしていますか？同時に、あなたは世界中でこのサーバーに接続する唯一のクライアントですか？実際のIPアドレスを見つけるのは難しいですか？ Springパッケージの発効により、それははるかに簡単になります。

ダブルVPN-憶測が多い人気トピック。多くの場合、この用語は完全に異なるテクノロジーと呼ばれますが、ほとんどの場合、IPアドレスのレベルでインターネット接続とアクセスポイントが離れていることを意味します。このモードでVPNサーバーを構成する最も簡単な方法を検討します。サーバー側で追加の構成を必要とせず、最大の速度と最小の遅延を得ることができます。

脅威モデル


何かから身を守るためには、脅威モデルを明確に理解する必要があります。プロバイダーがすべてのクライアントトラフィックを保存することを要求する新しい法律については触れませんが、接続データ、いわゆるNetflow、ストレージは非常にシンプルで、長い間成功しています。つまり、条件付きIPアドレス1.1.1.1を2.2.2.21日の特定の時刻のアドレスに接続するという事実が記録されます。

プロバイダー、都市、または国の規模でこのような情報にアクセスできるので、VPNの背後に誰が隠れているかを確認するだけで十分です。

VPNの使用時にプライバシーのレベルを上げるには、IPレベルで接続ポイントとインターネットアクセスポイントを分離する必要があります。上の画像では、ユーザーはIrina Yarovoyの監視下のフェンスの後ろにいます。フェンスを通過するすべての接続は、イリーナは厳密に覚えています。まともな市民としてのユーザーはアドレスgood.citizen.vpnに接続し、アドレスから戻りますsuper.cool.guy.vpn。その結果、イリーナの場合、これら2つの接続は接続されていないように見えます。

デュアルVPNとは何ですか？


「デュアル」VPNの名の下で、さまざまなことがよく理解されていますが、ほとんどの場合、地理的に離れた、またはネットワークレベルでインターネットに接続およびアクセスするノードを意味します。VPNプロバイダーのマーケティングトリックにすぎない場合もあります。つまり、まったく何も意味しません。このようなサービスは、「トリプル」および「クワッド」VPNと呼ばれることがあります。

実際に使用される最も一般的なスキームを分析します。

サーバー間のVPN


最も一般的な方法。このモードでは、クライアントは最初のサーバーのみへのVPN接続を確立します。 2番目のサーバーへのトンネルは最初のサーバーで構成され、クライアントからのすべてのトラフィックは2番目のサーバーに送られます。複数の中間サーバーが存在する場合があります。同時に、サーバー間のトンネルは、クライアントが接続されているプロトコル以外のプロトコル（IPsecなど）を使用して、またはGREやIPIPなどの暗号化なしでインストールできます。このモードでは、すべての中間サーバーがルートトレースに表示されます。クライアント側の中間サーバーがどのように接続されているかを確認する方法はないため、信頼できるのはプロバイダーのみです。

トラフィックルート全体で、最小MTU（最大伝送ユニット）は最初のトンネルの値と同じままであり、各中間サーバーは復号化されたクライアントトラフィックにアクセスできます。



プロキシ経由のVPN


また、かなり一般的な方法です。中国など、別のプロトコルでVPNトラフィックをマスクするためによく使用されます。VPNを使用すると、すべてのシステムトラフィックをトンネルに簡単にルーティングできるため、この方法は一連のプロキシよりも便利です。また、プログラムのシステムコールをインターセプトしてプロキシにリダイレクトするためのツールもあります。ProxyCap、Proxifierですが、リクエストを逃したり、プロキシをバイパスしたり、一部のプログラムで正しく機能しないため、安定性が低下します。

このモードでは、プロキシサーバーはルートトレースに表示されません。



VPN内のVPN


最も偏執的で最も遅い方法：すべてのトンネルはクライアント側で立ち上がり、それぞれが他のトンネルの内側にあります。この方法では、クライアント側でのルートのトリッキーな設定とすべてのVPNクライアントの正しい順序での起動が必要です。これはレイテンシとパフォーマンスに悪影響を及ぼしますが、中間サーバーはクライアントのオープントラフィックにアクセスできません。すべてのカプセル化オーバーヘッドが要約され、クライアントが最終的に利用できる最大パケットサイズ（MTU）は、トンネルの数に応じて減少します。中間サーバーはルートトレースに表示されません。



VDSを構成する


入口と出口を分割してVPNを設定する最も簡単な方法は、複数のIPアドレスを単一の仮想サーバーに接続することです。この方法では、実際にはトラフィックが1つのサーバーで終了するため、最大速度と最小遅延を得ることができます。ここVdsina.ruでは、コントロールパネルから自分でこれを行うことができます。IPv4はどこでも終了しますが、60ルーブルのサーバーでも追加のIPアドレスを発行します。

追加のIPアドレスを接続する


追加のIPアドレスを購入したら、指示に従って設定する必要があります。



明確にするために、PTRレコードをIPに割り当てます。これは、IPアドレスが変換されてドメインに戻されるときに表示されるドメイン名です。これは、存在しないドメインを含む任意の値にすることができます。

例として、次の値を使用します。

xxx.xxx.38.220 — super.cool.guy.vpn #   ( )
xxx.xxx.39.154 — good.citizen.vpn #   ( )



2つのIP検証


最初にサーバーに設定されたIPアドレスが出口点になるため、新しいアドレスが入口点になることを覚えておくことが重要です。SSH経由でサーバーに接続し、どのアドレスが外部アドレスとして使用されているかを確認します。

これを行うには、コンソールからifconfig.coサービスを使用するのが最も簡単です。curlを介してリクエストする場合、リクエスト元のIPアドレスを返します。

$ curl ifconfig.co
xxx.xxx.38.220

最後の数字は、外部アドレスが本当に出口点と一致していることを示しています。エントリポイントとして2番目のIPの正しい動作を確認してみましょう。これを行うには、SSHに組み込まれているSOCKSプロキシ機能を使用するだけです。

コマンドはクライアントで実行されます。

ssh -D 9999 root@good.citizen.vpn

#   
curl -x socks5h://127.0.0.1:9999 ifconfig.co
super.cool.guy.vpn

最初のコマンドは、good.citizen.vpnアドレスでSSHセッションを確立し、同時にこのセッション内でローカルポートで利用可能なSOCKSプロキシをアクティブにします。2番目は、このプロキシを介して通常のHTTPリクエストを行います。
例では、クエリに架空のドメイン名を使用していることを覚えておくことが重要です。それらはPTR解像度でのみ表示され、それらへの完全な要求を行うことはできません。したがって、この段階では、IPアドレスを使用してサーバーにアクセスする必要があります。

IKEv2サーバーの構成



IPsec IKEv2は、ほとんどすべてのオペレーティングシステムでそのまま使用できる最新のVPNプロトコルです。Windows、macOS、iOSのデフォルトプロトコルとして使用されます。同時に、サードパーティのソフトウェアをインストールする必要がなく、ほとんどの場合、OpenVPNよりも高速に動作します。ハブでのIKEv2サーバーの設定に関する記事はすでにありましたが、それらはすべて自己署名証明書の使用について説明しており、VPNクライアントの側にルート証明書をインストールする必要があるという点で不便です。

Let's Encryptからの信頼できる証明書を使用したサーバー構成の例を分析します。これにより、クライアントに無関係なルート証明書をインストールせずに、ユーザー名とパスワードのみを発行できます。

証明書の場所は次のとおりです。

cp /etc/letsencrypt/live/good.citizen.vpn/privkey.pem /etc/ipsec.d/private/
cp /etc/letsencrypt/live/good.citizen.vpn/chain.pem /etc/ipsec.d/cacerts/

letsencrypt証明書は頻繁に再発行されるため、手動で行うのは不便です。したがって、certbotのフックを使用してこのプロセスを自動化します。

スクリプトのタスクは、証明書が更新されるたびに3つのファイルを目的のフォルダーにコピーし、strongswanコマンドを送信して証明書を再度読​​み取ることです。/etc/letsencrypt/renewal-hooks/deploy/renew-copy.sh

ファイルを作成し、実行可能にします。

#!/bin/sh

set -e

for domain in $RENEWED_DOMAINS; do
        case $domain in
        good.citizen.vpn)
                daemon_cert_root=/etc/ipsec.d/

                # Make sure the certificate and private key files are
                # never world readable, even just for an instant while
                # we're copying them into daemon_cert_root.
                umask 077

                cp "$RENEWED_LINEAGE/cert.pem" "$daemon_cert_root/certs/"
                cp "$RENEWED_LINEAGE/chain.pem" "$daemon_cert_root/cacerts/"
                cp "$RENEWED_LINEAGE/privkey.pem" "$daemon_cert_root/private/"

                 # Reread certificates
                /usr/sbin/ipsec reload
                /usr/sbin/ipsec purgecerts
                /usr/sbin/ipsec rereadall
                ;;
        esac
done

これで、証明書が再発行されるたびに、スクリプトは新しいファイルをstrongswanフォルダーにコピーし、デーモンにコマンドを送信して証明書を再度読​​み取ります。

構成が正常にアクティブ化され、証明書が接続されていることがわかります。この段階では、すでにVPNサーバーに接続できますが、インターネットにアクセスできなくなります。インターネット上のクライアントを解放するには、転送を有効にし、NATを構成する必要があります。

NATセットアップ


転送パッケージをアクティブ化します。

echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf
sysctl -p

NATをオンにします。これは、最小限のiptables構成の単なる例であることを覚えておくことが重要です。必要に応じて、残りのルールをカスタマイズします。

ethName0-インターフェース名で置き換えます
10.0.1.0/24-VPNクライアントに発行されるIPアドレスの範囲。私たちは彼を尋ねた/etc/ipsec.conf
111.111.111.111 -出口ポイントのIPアドレスを、この例では、アドレスsuper.cool.guy.vpnを

iptables -t nat -A POSTROUTING -s 10.0.1.0/24 -o ethName0 -j SNAT --to-source 111.111.111.111

デバッグ中


構成のこの段階で、クライアントが既に接続できる完全に機能するサーバーを取得する必要があります。続行する前に、接続を確認してこれを確認することをお勧めします。

接続に問題がある場合は、ログをリアルタイムで監視できます。

journalctl -f -u strongswan

スタートアップ時のスタートアップ


すべてが成功した場合は、起動時にstrongswanを起動に追加できます。

systemctl enable strongswan

iptablesルールの保存


再起動後にiptablesルールを保存するために、特別なiptables-persistentパッケージがあります。現在のルールセットが保存され、スタートアップに追加されることを覚えておくことが重要です。

apt install iptables-persistent

合計


エントリーポイントとエグジットポイントを間隔を空けて配置した最も単純なサーバーセットアップオプションを示しました。この構成では、入口と出口のIPアドレスが異なるサブネット上にあるにもかかわらず、サーバー間で追加のトンネルを使用しないため、最大VPN速度を得ることができます。このアプローチでは、3つ以上のIPアドレスをサーバーに接続して、さらに実験を行うことができます。

IKEv2プロトコルは、システムに最大限ネイティブに統合されており、他のすべての条件が同じであれば、サードパーティのVPNプログラムを使用する場合よりも速度を上げることができるため、日常の作業でデスクトップOSで使用するのに最適です。
