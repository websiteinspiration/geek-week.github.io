クロスプラットフォームナビゲーション：NavControllerがIonic 4でAngular / Routerを破った理由



みなさん、こんにちは！私の名前はニキータZhigamovsky、のプログラマーでKitApp、と私はイオン4でナビゲーションを構築する上で、私の経験について話をしたい：私は、発生した問題とその解決策。

2018年から、モバイルアプリケーション向けのクロスプラットフォームソリューションを開発しています。以前はIonic 3rdバージョンで作業していましたが、時間の経過とともに機能が開発され、新しいバージョンに切り替えることにしました。Ionic4の以前のモデルの迷惑な瞬間やバグはすでに修正されているようです。

何がうまくいかないように思われるでしょう。最後に、機能には通常のAngularルーティングがあり、すべての欠点を持つ古いNavControllerはありません。ルーティングガイドの公式のIonic Webサイトでも、プログラムを使用してページをナビゲートすることは、メソッドを使用する価値があることが示されています角度のある/ルーター。しかし、私を古いNavControllerに戻す原因がありました。

問題の本質


興味深いバグに気づきました。サイドメニューがあるとします。イオンスプリットペインを使用して作成しました。また、メニューとは別のページがあり、メニューから他のページに移動したいとします。Router.navigateByUrl（ '/ menu / ...'）を使用して移動します。次に、メニューページAを呼び出し、メニューとは別のページ-Bを呼び出します。ただし、1つありますが、

ページAで、ngOnInitイベントで特定のロジックがトリガーされたとします。ルーターを使用してページBに移動し、メニューページがまだアクティブであることを確認します-削除されていません。したがって、ページAに戻ると、このページのngOnDestroyイベントが機能しなかったため、ngOnInitイベントは機能しません。すべてが論理的であるように思われるでしょう。このような場合、通常は角度ではなく、ライフサイクルメソッドの1つ、つまりイオン-ionViewWillEnterを使用します。それがアクティブになるとすぐにページに移動すると発生します。すべてが問題ないようで、完璧に適合しますが、いくつかの規則があります。

この遷移がメニュー内のページからのものでない場合、ページAで適切なアクションを実行するためのオプションはどれも機能しません。このページへの移行を追跡することはできません。繰り返しますが、ページBの下など、他のページの下では開いたまま静かに機能します。

いくつかの明確な例：

このページ構造がある場合、ionViewWillEnterは機能します：

1）個別のページ

-ページ1
-ページ2
- PAGE3

各ページに行くときに、この例では、ionViewWillEnterは完璧に動作します。 （page1 => page2、page2 => page3など）

2）メニュー/タブ

-メニュー
-menuPage1
-menuPage2
-menuPage3
この例では、すべてがうまくいきます。ionViewWillEnterメソッドは、ページ（menuPage1 => menuPage2、menuPage1 => menuPage3など）に移動するたびに起動します。

しかし、以下の例では、すべてがより複雑である：

-メニュー
- menuPage1
- menuPage2
- menuPage3
- loginPage
- signupPage

標準のAngularルーティングの問題はここから始まります。メニューページ内を移動する場合（menuPage1 => menuPage2 => menuPage3）-個々のページ間を移動する場合（loginPage => signupPage）と同じように、ionViewWillEnterメソッドは通常どおり機能します。しかし、別のページとメニューページ（loginPage => menu / menuPage1またはmenu / menuPage3 => signupPage）の間で移動を開始するとすぐに、ngOnInitメソッドもionViewWillEnterも機能しません。 ngOnInitはページが破損していないため機能しません。これは論理的です。しかし、ionViewWillEnterが機能しなかったのはなぜですか？

ドキュメントに基づいて、ionViewWillEnterは、個別のルーティングスタック（キーワード「個別」）内、または個別のページ間、またはメニュー/タブ内で機能します。しかし、個々のページとメニュー/タブの混合構造ではありません。奇妙ですが、これは正常な動作と見なされます。同時に、これは、特にライフサイクルフックの名前を考えると、ユーザーが期待する動作とは異なります:)。

この問題を解決するにはどうすればよいですか？


多くのフォーラムを訪問したが、通常の解決策を見ていなかったし、常に機能するとは限らない疑わしいライフハックを見たので、他に何かが必要であることが明らかになった。任意のタイプのページ間の遷移機能を変更するもの。

この場合はどうしますか？もちろん、Routerを地獄に投げて忘れてしまいます。以前は嫌われていたので、今では非常に優れたNavControllerがあるからです。

NavController.navigateRoot（）メソッドの主な違いは、別のページに切り替えた後、前のページが自動的に破棄されることです。そして再びそれに切り替えると、ngOnInitメソッドとionViewWillEnterの両方が機能します！実際、これは完璧なソリューションです-松葉杖や怪しげな自己記述関数なしで。

最もクールなのは、前の例のように、通常のページ間でも、メニュー内でも、混合タイプでも、どのページ構造でも機能することです。

良い面を要約します。

  1. NavControllerは、前のページに戻るときに、前のページをそれぞれ削除します。ページは更新され、ionViewWillEnterメソッドとngOnInitメソッドが機能します。たとえば、それらのロジックを再度呼び出して、ページの情報を更新できます。
  2. 古いpush（）、setRoot（）、pop（）メソッドを忘れて、クラス要素をナビゲートします。結局のところ、これが多くの問題を引き起こしたのです。navCtrlのメソッドが更新され、ルーターのメソッドと同じパスが渡されます。

「BUT」がない

場合の注意点が1つあります。Androidのハードウェアの「戻る」ボタンにイベントハンドラーを追加し、このハンドラーでルーターまたはnavControllerを使用してどこかに移動しようとすると、コンソールで次のように選択されます。エラー：「ナビゲーションがAngularゾーンの外でトリガーされました」。

はい、ナビゲーションは機能し、ページは開きますが、何も機能しません-プロパティの初期化もライフサイクルメソッドも。残念ながら、戻るボタンを押すことによるナビゲーションは、Angularゾーンの外側でトリガーされ、実際には、テンプレートを開くだけです。

解決策は、実際には非常に簡単です。 Angularゾーン内でナビゲーションを明示的に強制するだけです。

例：

import { Component, NgZone } from '@angular/core';
import { NavController } from '@ionic/angular';
@Component({
  selector: 'app-root',
  templateUrl: 'app.component.html'
})
export class AppComponent {
  constructor(private navCtrl: NavController, private ngZone: NgZone){}
  this.ngZone.run(() => this.navCtrl.navigateForward('menu')).then();
  
  this.ngZone.run(() => this.router.navigateByUrl('/menu/my-orders')).then();
}

そして今、すべてがうまくいきます！

ngZoneについては興味深い記事がたくさんあります。幸運を！

navControllerのメソッドについて少し：



  • this.navCtrl.setDirection（ 'root'）-スタックにルートページを設定し、以前のページをすべて削除します。
  • this.navCtrl.navigateRoot（ 'homePage'）-navCtrl.setDirection（ 'root'）+ router.navigateByUrl（ 'homePage'）に似ていますが、スタックの前のページを削除する必要があります（これが必要です）。
  • this.navCtrl.navigateForward（ 'examplePage'）-router.navigateByUrl（ '/ examplePage）と似ていますが、どこに移動するかを明示的に示すことで、スタックの前のページを削除できます。
  • this.navCtrl.back（）-location.back（）に似ていますが、アニメーションがあります。
  • this.navCtrl.navigateBack('backPage') — navCtrl.setDirection('back') + router.navigateByUrl('backPage').

ここで、menu / page1にいて、



別のメニュースタックがあり、menu / page1からログインページに移動した後で、menu / page1ページを削除する必要があるとします。これにより、再び切り替えた後、ngOnInitまたはionViewWillEnter。遷移にrouter.navigateByUrl（ 'login）を使用する場合、その後はログインページが表示されますが、



ログインからmenu / page1に切り替えた後は、それぞれngOnInitもionViewWillEnterも機能しません。

navCtrl.navigateRoot（ 'login'）を使用して移動する場合、ログインページを開いた後、前のページは削除されます。そして、ngOnInitメソッドとionViewWillEnterメソッドが機能します。



これがnavControllerを使用する利点です。予想される動作は、現在の動作と完全に一致しています。
