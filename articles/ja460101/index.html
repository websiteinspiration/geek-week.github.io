かなり長い間、私はHackerOneプラットフォームの脆弱性を探しており、メインの作業以外に一定の時間を割り当てて、お気に入りのプログラムや新しいプログラムをチェックしています。この記事の主人公となるCookieベースのXSSの脆弱性に何度も遭遇しました。このタイプの脆弱性は、Cookieパラメータの値がページに反映されるときに発生します。デフォルトでは、私たちが順番に危険を証明しない限り、それらは自己XSSと見なされます。実際、今日はCookieベースのXSS脆弱性を悪用する方法を説明します。また、調査のために合計7300ドルを受け取った1つの会社をテストした例も示します。

ユーザーサイドでJavaScriptを実行するには、Cookieを設定する方法を見つけ、必要に応じて、犠牲者をCookieが埋め込まれているページに誘導する必要があります。このバグを悪用する可能な方法：

⠀1 . CRLFインジェクション。この脆弱性は、適切な検証および改行文字のブロックがない場合に発生します。 Set-cookieヘッダーを実装して、目的の名前とCookieの値を応答し、ブラウザに設定できます。実際の例：リダイレクトでのtwitter.comへの滑りやすいCRLFインジェクション-https://twitter.com/login?redirect_after_login=/jjjkkk嘊嘍Set-Cookie：jjjjj = a; domain = twitter.com



このタイプに関するレポート脆弱性はHackerOneで読むことができますhackerone.com/hacktivity?order_direction=DESC&order_field=popular&filter=type%3Apublic&querystring=crlf%20injection

2. XSSサブドメインの脆弱性。 XSSはパブリックにアクセス可能であり、ワイルドカード* .vulnerabledomain.comに配置されている必要があります。多くのバグ報奨金プログラムでは、サブドメインは範囲外です。つまり、ほとんどの場合、バグはまったく受け入れられないか、「報奨金の対象外」のマークが付けられて受信されます。そのような場合、後退するべきではありませんが、CookieベースのXSSと接続するために、XSS検索に時間を費やして報酬を得ることができます。 XSSが検出された場合、document.cookie関数を使用してCookieを設定または削除できます。

影響の強化：多くの場合、被害者は、jira.vulnerabledomain.comなどよりもさらに、/ plugins / servlet / oauth / users / icon-uri？consumerUri = https：//maliciousdomain.comなどのURLを使用して、vulnerabledomain.comのメインドメインを信頼します。このサブドメインが個人アカウントまたは認証に関連付けられていない場合、サブドメインよりもメインドメインに切り替わる可能性が高くなります。上記に基づいて、我々はにリダイレクトするには、サイトのリダイレクト機能を使用することができますvulnerabledomain.com/login?redirectUrl=https：//jira.vulnerabledomain.com/pathサブドメインを、より良い効果のために。

被害者がアクティブなセッションを持っている場合、リダイレクトは自動的に行われ、そうでない場合は、承認が必要です。ユーザーがこのようなリンクをクリックすると、リフレクトされたXSSが存在するサブドメインからCookieもインストールされ、さらに上流に送信されます。CookieベースのXSSページにエクスプロイトが機能し、トークンのCSRF値を取得します。メールアドレスの変更リクエストを完了します。したがって、2つのXSS脆弱性の組み合わせは、古いメールボックスからの電子メールパスワードまたはコードの変更の追加確認などの関連する問題がない場合、アカウントの乗っ取りにつながる可能性があります。

3。Cookieの設定を許可するテストファイルの検出。コンテンツ検出ツール（dirb、dirserachなど）を見つけて掘り始めるだけで十分です。開発者がクリーニングを実行するのを忘れた場合は、そのようなファイルに遭遇する可能性があります。

最近、任意の名前と値のCookieをインストールできるテストサーブレットHTMLページを見つけました。もちろん、POSTリクエストの保護は存在しなかったため、被害者がCSRFエクスプロイトにアクセスした場合（またはPOSTをGETに変更できる場合）は、ブラウザにCookieをインストールすることが可能です。



このバグはCRLFインジェクションの代替案として認定され、/ examples /を削除することで修正され、重大度の低いバグの場合は150ドルが支払われました。 h1トリアージャーは中程度の配信を行いましたが、開発者はそれが深刻度が低いと信じる傾向がありました。



4。中間者攻撃（MITM）。このメソッドは、Cookieに安全なフラグがない場合にのみ適用できます。フラグの種類がわからない場合や、記憶を更新したい場合は、OWASP London 2017 www.owasp.org/images/a/a0/OWASPLondon20171130_Cookie_Security_Myths_Misconceptions_David_Johansson.pdfによる Cookie Securityのプレゼンテーションをご覧になることをお勧めします。

攻撃を成功させるには、被害者が攻撃者のネットワーク上にあるか、DNS解決が影響を受ける可能性があることが必要です。脆弱性をチェックするには、次のことが必要です

。1）次の内容のindex.phpファイルをホストする：

<?php if ($_SERVER['HTTP_HOST'] == 'non-existed-subdomain.vulnerabledomain.com') { setrawcookie("VID",'\'+alert(123123123)+\'', time()+36000, "/", ".vulnerabledomain.com",0,1); } ?>

2） / etc / hosts /ファイルに次の行を追加します：127.0.0.1 non-existed-subdomain.vulnerabledomain.com

3） non-existed-subdomain.vulnerabledomain.comにアクセスし、その後、Cookieが反映されているページを開きます。
