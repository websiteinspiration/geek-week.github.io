サーバー要求偽造、ブラインドSSRF操作

SSRFと呼ばれるものがあります。彼女についてはたくさん書かれていますが、それでも、簡単に説明します。

サイトにアクセスして、プロフィールに記入し、「アバターをアップロード」アイテムにアクセスするとします。そして、あなたには選択肢があります-ファイルをアップロードするか、リンクを指定します。

この場合、2番目の点に関心があります。画像を含むリソースへのリンクを提供すると、Webアプリケーションは次のようになります。

  1. ダウンロードする
  2. 画像が画像であることを確認してください
  3. サイズが合わない可能性があるため、サイズを確認してください
  4. ユーザーに表示（切断用）

だからここに。サイトがイメージのダウンロード元を確認しない場合、これは（原則として）脆弱性です。さらに、画像のダウンロードなどの小さな機能に対する攻撃のベクトルは非常に広範囲であり、すべてのオプションを実行するのに十分なほどバーに集まっているわけではありません。



一度尋ねられました-「http（s）リンクを1つしか配置できない場合はどうすればよいですか？それは何も与えません！」

私はあなたに言います。

最も簡単なオプションは、内部のサービスを特定することです。画像について話している場合は、favicon.ico、ロゴ、アイコンディレクトリなどの標準パスを確認して、Apacheの使用を提案できます。リクエストを送信することで、人気のローカルアドレスだけでなく、内部インフラストラクチャで機能するサイトのサブドメインを反復処理できます。これらはすべての種類の竹、ジラ、Gitlabなど、あらゆる企業に愛されています。

それには何が必要ですか？しかし、知識は力です。結局のところ、盲目的であっても、さまざまな脆弱性やエクスプロイトを使用できます。使用するWebサーバーまたはサービスのベンダーまたはバージョンを知っているので、適用できる攻撃の範囲を絞り込みます。現在ではないかもしれませんが、将来的には、内部インフラストラクチャに関する技術情報を知ることが、他の脆弱性の悪用に役立つでしょう。

さて、私たちはIPアドレスを入力することを禁止されていました。インフラストラクチャ内に重要なリソースがあり、そのIPアドレスが192.168.1.1であると仮定します。

最初に、このIPを割り当てるドメインを精神的に作成します。my-test-site.comとします。実際には、必要なIPにアドレスが向けられるサブドメインを作成する必要がありますが、後で詳しく説明します。

パスワード総当たり


内部にルーターがあることを夢見てみましょう。/ admin /ディレクトリはBasic authの下にあります。リンクを変更することで、ルーターのログインとパスワードを選択できます。しかし、ここでは非常に単純です。フォームのリンクを作成するだけです

http://login:password@my-test-site.com/admin/

したがって、コロンの前の値はログイン、パスワードの後のパスワードになります。そして@記号を介して、このデータが送信されるドメインがあります。フォームに入力する必要があるすべての種類の連絡先担当者には対応していません。したがって、基本認証、つまりサーバーからの401応答を含むポップアップウィンドウが必要です。

運が良ければ、サイトが少なくとも応答コード（200、401、503）を返すと、はるかに簡単になります。次に、プロセスを明確に観察し、勝利を確認します。

http://admin:password@my-test-site.com/admin/ - 401
http://admin:admin@my-test-site.com/admin/ - 401
http://admin:123456@my-test-site.com/admin/ - 200

このような要求を12個または2個送信すると、このルーターのパスワードを見つけることができます。次に、スクリプトを使用して、独自のDNSまたは/reboot.cgiを保存します。

答えがなく、常に同じである場合はどうでしょうか。

ここでタイミングが役立ちます。

タイミング


すべてに時間がかかります。私はあなたの記事を読むのにあなたの時間を費やしているので、サービスはそれに応答するのにあなたの時間を費やしています。

特徴は、内部リソースにアクセスしてタイミングを測定し、質問に答えることができることです-サービスはありますか？

多くのリクエストを送信することで、内部サービス、ポート、さらにはディレクトリやファイルまで盲目的に並べ替えることができ、回答の異常性に依存します。

1302 ms - http://test.company.com
1307 ms - http://db.company.com
5483 ms - http://jira.company.com
1410 ms - http://docs.company.com
1366 ms - http://kafka.company.com

jiraサブドメインが最も長く応答しました。最も可能性が高いのは内部であり、Webサーバーがページを読み込もうとしたところ、画像ではないことに気づいたため、違いが顕著になっています。さらに、「誰が一番長く答えたのか」という事実ではなく、「誰もが誰のように答えなかったのか」という事実ではありません。タイミングが遅延のようになる可能性があるため-たとえば、実行に長い時間がかかる大きなファイルまたはスクリプトを見つけた場合。またはその逆、簡単な答え。

1302 ms - http://test.company.com
1307 ms - http://db.company.com
423 ms - http://jira.company.com
1410 ms - http://docs.company.com
1366 ms - http://kafka.company.com

この場合、答えは、画像パーサーを処理しないのは401またはリダイレクトのいずれかであると述べています。または、サイトはアクセス可能ですが、最初のバイトまたはコンテンツタイプを確認した後、ページを完全にダウンロードする前にWebアプリケーションが拒否しました。この例の他のサイトは接続を待機していませんでした（またはホスト名が落ち着きませんでした）。

IP検証メカニズム


多くのサイトが、IPアドレスが内部ではないことの確認を入力しています。ただし、ロジックが間違っている可能性があり、アプリケーションを内部IPアドレスに接続できる場合もあります。たとえば、サイトは2つの論理ブロックをチェックインします。まず、外部IPをポイントするホストが正確に指定されているかどうかを確認します。最初のチェックに合格した後、サイトが接続を確立し、前の段落のIPアドレスをキャッシュしていない場合は、おかしなトリックを試すことができます。

ドメインmy-test-site.comの最初のリクエストで、外部IPを与えます（例：123.123.123.123）。
検証に合格するとすぐに、同じドメインに内部IPを与え始めます。

この場合、Emil Lernerは1u.msというすばらしいサービスを提供しました。

1u.msドメインは、指定したIPアドレスで応答します。

ドメインの形式は次のとおりです。

make-%IP%-rebind-%IP-rr.1u.ms

例えば

make-123.123.123.123-rebind-127.0.0.1-rr.1u.ms

最初の要求はアドレス123.123.123.123で応答され、2番目の要求はすでに127.0.0.1で応答されます（次から次の要求の場合、5秒以内）。

ちなみに、サブドメインが必要な場合は、IPアドレスをドットなしで記述できます。

make-123-123-123-123-rebind-127-0-0-1-rr.1u.ms

ちなみに、makeの前とrrの後で、キャッシュされたデータが使用されないようにするために、任意の単語を書き込むことができます。

bo0om-make-123-123-123-123-rebind-127-0-0.1-rr-test.1u.ms

そして、ログを表示するために-tail -fのアナログ：

curl http://1u.ms:8080/log

（またはブラウザの同じリンク）

DNSを使用したポートスキャン


実際、DNSレコードの管理では、ポートをスキャンしてみることができます。そして、ちょっとしたトリックがこれを助けてくれます。

ドメインmy-test-site.comがあるとします。

通常、リソースを開くためのAレコードが少なくとも1つ含まれています。
私たちのサイトのIPが172.217.20.46（google.comから取得）であるとしましょう。しかし、そのようなエントリをいくつか指定することができます！それらを作成し、DNSレコードが次のようになっているとします。

my-test-site.com 172.217.20.46
my-test-site.com 192.168.1.1

私たちのリソースは開かれますか？はい、そうです。そして、最初のレコードが最初になるからです。

次のようにDNSレコードを変更します。

my-test-site.com 192.168.1.1
my-test-site.com 172.217.20.46

この機能を使用すると、開いているポートと開いていないポートを確認できます。実際、多くの（すべてではない）ライブラリは、最初にDNSレコードにリストされている最初のリソースにアクセスし、次に2番目のリソースにアクセスしようとします。

ここでは、アバターをロードする際のリソースへのリンクを示しました。

ドメインを指定し、ポートを変更することにより、例外メソッドを使用して、使用可能なポートを判別しようとします。 最初のレコードとして192.168.1.1を指定したので、たとえこのアドレスが正しくなくても、192.168.1.1（ポート8080および3306）でライブラリに応答したと結論付けます。したがって、これらのポートは開いており、サービスが提供されています。この場合、2番目のアドレスに切り替わりません。 ここの1u.msサービスも役立ちます。この場合、次の構成になります。

make-%IP%-and-%IP%rr.1u.ms

タイプの

make-192.168.1.1-and-172.217.20.46rr.1u.ms

前の段落のように、他のすべての機能である2つのエントリを返します。

ちなみに、このアプローチは、DNSの再バインドを高速化するために使用でき、「ハングした」サーバーから機能しているサーバーにブラウザーを強制的に切り替えます。DNSキャッシュを更新するのに1分待つよりも速いと思います。ただし、たとえばChromeでは、ランダムなIPが必要になるため、このトリックは機能しません。

もう1つの問題は、Webアプリケーションがドメインアドレスを決定するDNSサーバーに組み込みのラウンドロビンを使用できることです。つまり、レコードの順序を変更して、すべてのサーバーに負荷を均等に分散します。たとえば、8.8.8.8はこの機能を拒否しましたが、1.1.1.1には存在しています。

リダイレクト


リダイレクトしてみてください！まず、リダイレクトを使用すると、たとえばDNSを使用してポートをスキャンするときに、Webアプリケーションに対する要求の数を減らすことができます。答えが届いた場合は、別のポートまたはドメインにリダイレクトします。そうでない場合、多分彼は開いているポートでつまずいた。

ただし、最善の方法は、リダイレクトを使用してプロトコルを変更することです。私の実践では、ファイルへのリダイレクト/// etc / passwdが機能し、ファイルの内容を表示する場合がありました。ブラインドssrfを備えたバージョンでは、プロトコルをgopher（まだ存在している間）に変更してみることができます。また、コマンドを使用してSMTPやその他の魔法に手紙を送ることができます。
