Kotlin MultiplatformのMVIアーキテクチャテンプレート、パート1



約1年前、私は新しいKotlin Multiplatformテクノロジーに興味を持ちました。 APIにアクセスしながら、一般的なコードを記述してさまざまなプラットフォーム用にコンパイルできます。それ以来、私はこの分野で積極的に実験し、このツールを当社で宣伝してきました。たとえば、1つの結果は、Reaktiveライブラリ-Kotlin Multiplatform用のReactive Extensionsです。

Android開発用のBadooおよびBumbleアプリケーションでは、MVIアーキテクチャテンプレートを使用します（アーキテクチャの詳細については、Zsolt Kocsiの記事をご覧ください：“ Kotlinに基づく最新のMVIアーキテクチャ"）。私はさまざまなプロジェクトに取り組んでおり、このアプローチの大ファンになりました。もちろん、Kotlin MultiplatformでMVIを試す機会を逃すことはできませんでした。さらに、ケースは適切でした。Reaktiveライブラリの例を記述する必要がありました。これらの私の実験の後、私はMVIにさらにインスピレーションを受けました。

私は常に、開発者がKotlin Multiplatformをどのように使用するか、そしてそのようなプロジェクトのアーキテクチャをどのように構築するかに注意を払っています。私の観察によれば、平均的なKotlin Multiplatform開発者は、実際にはMVVMテンプレートに慣れているという理由だけで彼の仕事でMVVMテンプレートを使用するAndroid開発者です。さらに、「クリーンアーキテクチャ」を適用するものもあります。ただし、私の意見では、MVIはKotlin Multiplatformに最適であり、「クリーンアーキテクチャ」は不要な複雑化です。

したがって、私は次のトピックに関する3つの記事からなるこのシリーズを書くことにしました。

  1. MVIテンプレートの簡単な説明、問題の説明、およびKotlin Multiplatformを使用した共通モジュールの作成。
  2. iOSおよびAndroidアプリケーションでの共通モジュールの統合。
  3. 単体テストと統合テスト。

以下はシリーズの最初の記事です。これは、Kotlin Multiplatformをすでに使用している、または使用を計画しているすべてのユーザーにとって興味深いものです。

この記事の目的は、Kotlin Multiplatform自体の操作方法を教えることではありません。この領域について十分な知識がないと思われる場合は、最初に概要とドキュメント（特に、「同時実行性」と「不変性」セクションで Kotlin /ネイティブメモリモデルの機能を理解すること）に慣れることをお勧めします。この記事では、プロジェクトの構成、モジュール、およびトピックに関連しないその他のものについては説明しません。

MVI


まず、MVIとは何かを思い出してみましょう。省略形はModel-View-Intentを表します。システムには2つの主要コンポーネントしかありません。

  • モデル-ロジックとデータのレイヤー（モデルはシステムの現在の状態も格納します）;
  • (View) — UI-, (states) (intents).

次の図は、おそらくすでに多くの人に馴染みのあるものです。



つまり、これらの非常に基本的なコンポーネントであるモデルとプレゼンテーションが表示されます。他のすべてはそれらの間で循環するデータです。

データが一方向にのみ移動することが簡単にわかります。状態はモデルから取得され、表示のためにビューに分類されます。意図はビューから取得され、処理のためにモデルに挿入されます。この循環は単方向データフローと呼ばれます。

実際には、モデルはしばしばStoreと呼ばれるエンティティによって表されます（これはReduxから借用されます）。ただし、これは常に発生するわけではありません。たとえば、MVICoreライブラリでは、モデルはFeatureと呼ばれます。

また、MVIは反応性と密接に関連していることにも注意してください。データストリームとその変換のプレゼンテーション、およびサブスクリプションのライフサイクル管理は、リアクティブプログラミングライブラリを使用して実装するのに非常に便利です。現在、それらのかなりの数が利用可能ですが、Kotlin Multiplatformで一般的なコードを記述する場合、マルチプラットフォームライブラリしか使用できません。データストリームの抽象化が必要です。また、入力と出力を接続および切断したり、変換を実行したりする機能が必要です。現在、私はそのような2つのライブラリーを知っています。

  • 私たちのReaktiveライブラリ-KotlinマルチプラットフォームでのReactive Extensionsの実装;
  • コルーチンとフロー -Kotlinコルーチンによるコールドストリームの実装。

問題の定式化


この記事の目的は、Kotlin MultiplatformでMVIテンプレートを使用する方法と、このアプローチの利点と欠点を示すことです。したがって、私は特定のMVI実装に執着することはありません。ただし、データストリームがまだ必要なので、Reaktiveを使用します。必要に応じて、アイデアを理解した後、Reaktiveをコルーチンとフローに置き換えることができます。一般的に、MVIは不必要な複雑化をせずにできるだけ単純にするようにします。

MVIを説明するために、以下の要件を満たす最も単純なプロジェクトを実装するようにします。

  • AndroidおよびiOSのサポート。
  • 非同期操作のデモンストレーション（入出力、データ処理など）。
  • できるだけ多くの一般的なコード。
  • 各プラットフォームのネイティブツールによるUI実装。
  • プラットフォーム側にRxがない（Rxへの依存関係を「api」として指定する必要がないようにするため）

例として、非常に単純なアプリケーションを選択しました。ボタンのある1つの画面をクリックして、猫の任意の画像を含むリストをダウンロードして表示します。画像をアップロードするには、オープンAPI：https : //thecatapi.comを使用します。これにより、WebからリストをダウンロードしてJSONファイルを解析する必要があるため、非同期操作の要件を満たすことができます。GitHubで

したがって、ストアには次のプロパティがあります。

  • 2つの汎用パラメーターがあります。入力インテントと出力状態です。
  • 意図の消費者である（Consumer <Intent>）;
  • 状態ストリームです（Observable <State>）;
  • 破壊可能です（使い捨て）。

そのようなインターフェースを毎回実装することはあまり便利ではないので、特定のアシスタントが必要になります。

class StoreHelper < in Intent ： Any、out State ： Any、in Effect ： Any >（
initialState ： State、
private val actor ： Actor < State、Intent、Effect >、
private val reducer: Reducer<State, Effect>
) : Observable<State>, DisposableScope by DisposableScope() {
init {
ensureNeverFrozen()
}
private val subject = BehaviorSubject(initialState)
fun onIntent(intent: Intent) {
actor(subject.value, intent).subscribeScoped(isThreadLocal = true, onNext = ::onEffect)
}
fun onEffect（effect ： Effect）{
subject.onNext（reducer（subject.value、effect））
}
fun subscribeをオーバーライドします（observer ： ObservableObserver < State >）{
subject.subscribe（オブザーバー）
}
}

StoreHelperは、私たちがストアを簡単に作成できるようにする小さなクラスです。次のプロパティがあります。

  • 3つの一般的なパラメーターがあります。入力インテントとエフェクトおよび出力状態。
  • コンストラクター、アクター、ギアボックスを介して初期状態を受け入れます。
  • 状態のストリームです。
  • 破壊可能（使い捨て）;
  • 非凍結（サブスクライバーも凍結されないようにするため）;
  • DisposableScope（サブスクリプションを管理するためのReaktiveからのインターフェース）を実装します。
  • 意図と効果を受け入れて処理します。

onIntentメソッドをさらに詳しく考えてみましょう：

  • 意図を引数として受け入れます。
  • アクターを呼び出し、意図と現在の状態をそれに渡します。
  • アクターから返されたエフェクトストリームをサブスクライブします。
  • すべての効果をonEffectメソッドに送信します。
  • エフェクトのサブスクライブは、isThreadLocalフラグを使用して実行されます（これにより、Kotlin / Nativeでのフリーズが回避されます）。

次に、onEffectメソッドを詳しく見てみましょう。

  • 引数として有効になります。
  • ギアボックスを呼び出し、エフェクトと現在の状態をギアボックスに転送します。
  • 新しい状態をBehaviorSubjectに渡します。これにより、すべてのサブスクライバーが新しい状態を受け取ります。

ビューには次のプロパティがあります。

  • 2つの一般的なパラメーターがあります。入力モデルと出力イベントです。

Androidビューとの混同を避けるために、MviViewの名前にMviプレフィックスを追加しました。また、ConsumerインターフェースとObservableインターフェースは拡張せず、単にプロパティとメソッドを使用しました。これは、Rxを「api」依存関係としてエクスポートせずに、プレゼンテーションインターフェイスを実装用のプラットフォーム（AndroidまたはiOS）に設定できるようにするためです。トリックは、クライアントが「イベント」プロパティと直接対話するのではなく、MviViewインターフェースを実装して、抽象クラスを拡張することです。

この抽象クラスをすぐに追加して、以下を表します。

abstract class AbstractMviView<in Model : Any, Event : Any> : MviView<Model, Event> {
private val subject = PublishSubject<Event>()
override val events: Observable<Event> = subject
protected fun dispatch(event: Event) {
subject.onNext(event)
}
}

多くのAndroid開発者のライフサイクルに精通しています。iOSやJavaScriptにも最適です。ファサードのライフサイクルの状態間の遷移の図は次のようになり


ます。ここで何が起こっているのかを簡単に説明します。

  • 最初にonCreateメソッドが呼び出され、その後-onViewCreatedが呼び出され、次に-onStart：これによりファサードが作動状態になります（開始）。
  • この後のある時点で、onStopメソッドが呼び出されます。これにより、ファサードが停止状態（停止）になります。
  • 停止状態では、2つのメソッドのいずれかを呼び出すことができます。
  • ビューが破棄されると、ビューを再度作成する（onViewCreated）か、ファサード全体を破棄する（onDestroy）ことができます。

使い方：

  • まず、KittenStoreのインスタンスを作成します。
  • onViewCreatedメソッドでは、KittenViewへのリンクを覚えています。
  • onStartでは、相互にKittenStoreとKittenViewに署名します。

結論


これは、Kotlin Multiplatformでの私のMVIシリーズの最初の記事でした。その中で：

  • MVIとは何か、それがどのように機能するかを思い出した。
  • Reaktiveライブラリを使用して、Kotlin Multiplatformで最も単純なMVI実装を作成しました。
  • MVIを使用して画像のリストをロードするための共通モジュールを作成しました。

共通モジュールの最も重要なプロパティに注意してください：

  • UIコードを除くすべてのコードをマルチプラットフォームモジュールに配置できました。すべてのロジックに加えて、ロジックとUI間の接続と変換が一般的です。
  • ロジックとUIは完全に無関係です。
  • UIの実装は非常に簡単です。必要なのは、着信ビューモデルを表示してイベントをスローすることだけです。
  • モジュールの統合も簡単です。必要なものは次のとおりです。
    • KittenViewインターフェイス（プロトコル）を実装します。
    • KittenComponentのインスタンスを作成します。
    • 適切なタイミングで彼のライフサイクルメソッドを呼び出す。
  • このアプローチにより、プラットフォームへのRx（またはコルーチン）の「フロー」が回避されます。つまり、アプリケーションレベルでサブスクリプションを管理する必要がありません。
  • すべての重要なクラスはインターフェースによって抽象化され、テストされます。
