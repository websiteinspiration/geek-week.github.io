ReactiveX Redux

Reduxを使用するすべての人は、遅かれ早かれ、非同期操作の問題に直面します。しかし、最新のアプリケーションはそれらなしでは開発できません。これらは、バックエンドへのhttpリクエストと、あらゆる種類のタイマー/遅延です。 Reduxの作成者は明確に話します-デフォルトでは、同期データフローのみがサポートされ、すべての非同期アクションはミドルウェアに配置する必要があります。

もちろん、これは冗長すぎて不便なので、「ネイティブ」ミドルウェアのみを使用する開発者を見つけるのは困難です。 Thunk、Sagaなどのライブラリとフレームワークが常に助けになります。

ほとんどのタスクでは、それらで十分です。しかし、1つの要求を送信したり1つのタイマーを作成したりするよりも少し複雑なロジックが必要な場合はどうでしょうか。ここに小さな例があります：

async dispatch => {
   setTimeout(() => {
      try {
         await Promise
            .all([fetchOne, fetchTwo])
            .then(([respOne, respTwo]) => {
                dispatch({ type: 'SUCCESS', respOne, respTwo });
             });
      } catch (error) {
          dispatch({ type: 'FAILED', error });
      }   
   }, 2000);
}

そのようなコードを見るのも苦痛ですが、維持して拡張することは単に不可能です。より複雑なエラー処理が必要な場合はどうすればよいですか？再リクエストが必要な場合はどうなりますか？この機能を再利用したい場合はどうすればよいですか？

私の名前はドミトリーサモクヴァロフです。この投稿では、Observableの概念と、それをReduxと組み合わせてどのように実践するかを説明し、すべてをRedux-Sagaの機能と比較します。

原則として、そのような場合は、redux-sagaを使用してください。OK、私たちはサガを書き直します：

try {
    yield call(delay, 2000);
    const [respOne, respTwo] = yield [
       call(fetchOne),
       call(fetchTwo)
    ];
    yield put({ type: 'SUCCESS', respOne, respTwo });
} catch (error) {
    yield put({ type: 'FAILED', error });    
}

それは著しく良くなりました-コードはほぼ線形で、見た目も読みやすくなります。しかし、サガはサンクと同じくらい不可欠なので、拡張と再利用は依然として困難です。

別のアプローチがあります。これはまさにアプローチであり、非同期コードを記述するための別のライブラリではありません。これはRxと呼ばれます（これらはObservable、Reactive Streamsなどです）。これを使用して、Observableの例を書き直します。

action$
  .delay(2000)
  .switchMap(() => 
     Observable.merge(fetchOne, fetchTwo)
       .map(([respOne, respTwo]) => ({ type: 'SUCCESS', respOne, respTwo }))
       .catch(error => ({ type: 'FAILED', error }))

コードはフラットになり、ボリュームが減少しただけでなく、非同期アクションを記述するという原則自体が変更されました。これで、クエリを直接操作するのではなく、Observableと呼ばれる特別なオブジェクトに対して操作を実行します。

Observableを値のストリーム（シーケンス）を提供する関数として表すと便利です。 Observableには3つの主要な状態があります-次（「次の値を与える」）、エラー（「エラーが発生しました」）、完了（「値が終わった、これ以上与えることはありません」）。この点で、これはPromiseに少し似ていますが、これらの値に対して反復できるという点が異なります（これは観測可能な超大国の1つです）。 Observableで何でもラップできます-タイムアウト、httpリクエスト、DOMイベント、単なるjsオブジェクト。



2番目の超大国Observableはオペレーターです。演算子は、Observableを受け入れて返す関数ですが、値のストリームに対して何らかのアクションを実行します。最も近い類推は、JavaScriptからのマップとフィルターです（ちなみに、そのような演算子はRxにあります）。



私にとって個人的に最も有用なのは、zip、forkJoin、およびflatMapオペレーターでした。これらの例を使用すると、オペレーターの作業を説明するのが最も簡単です。

zip演算子は非常に単純に機能します-いくつかのObservable（9以下）を取り、それらが出力する値を配列として返します。

const first = fromEvent("mousedown");
const second = fromEvent("mouseup");


zip(first, second)
    .subscribe(e => 
       console.log(`${e[0].x} ${e[1].x}`));



//output
[119,120]
[120,233]
…

一般に、zipの操作はスキームで表すことができます



。Zipは、複数のObservableがあり、それらから一貫して値を受け取る必要がある場合に使用されます（同期しているかどうかに関係なく、異なる間隔で出力できるという事実にもかかわらず）。 DOMイベントを操作するときに非常に役立ちます。

forkJoin演算子は、1つの例外を除いてzipに似ています-各Observableから最新の値のみを返します。



したがって、ストリームからの有限値のみが必要な場合に使用するのが妥当です。
もう少し複雑なのは、flatMapオペレーターです。 Observableを入力として受け取り、新しいObservableを返し、セレクター関数または別のObservableを使用して、値から新しいObservableにマップします。混乱するように聞こえますが、図ではすべてが非常に単純



です。コードではさらに明確になっています。

const observable = of("Hello");

const promise = value => 
new Promise(resolve => resolve(`${value} World`);

observable
  .flatMap(value => promise(value))
  .subscribe(result => console.log(result));

//output
"Hello World"

ほとんどの場合、flatMapは、switchMapおよびconcatMapとともに、バックエンドリクエストで使用されます。
ReduxでRxを使用するにはどうすればよいですか？これには、reduxで観測できるすばらしいライブラリがあります。そのアーキテクチャは次のようになります



。Observable、オペレーター、およびそれらに対するアクションはすべて、epicと呼ばれる特別なミドルウェアの形式で作成されます。各エピックはアクション入力を受け取り、それをObservableにラップし、これもObservableの形式でアクションを返す必要があります。通常のアクションを返すことはできません。これは無限ループを作成します。 APIリクエストを行う小さなエピックを書きましょう。

const fetchEpic = action$ => 
    action$
      .ofType('FETCH_INFO')
      .map(() => ({ type: 'FETCH_START' }))
      .flatMap(() => 
        Observable
          .from(apiRequest)
          .map(data => ({ type: 'FETCH_SUCCESS', data }))
          .catch(error => ({ type: 'FETCH_ERROR', error }))
      )

redux-observableとredux-sagaを比較せずに行うことは不可能です。多くの人には機能と機能が近いようですが、これはまったく当てはまりません。Sagasは完全に必須のツールであり、本質的には副作用を処理するための一連のメソッドです。Observableは、基本的に異なるスタイルの非同期コードであり、必要に応じて、異なる哲学を使用します。

問題解決の可能性とアプローチを説明するために、いくつかの例を書きました。

アクションによって停止するタイマーを実装する必要があるとします。これがサガでの様子です：

while(true) {
   const timer = yield race({
     stopped: take('STOP'),
     tick: call(wait, 1000)
   })

   if (!timer.stopped) {
      yield put(actions.tick())
   } else {
      break
   }
}

Rxを使用します。

interval(1000)
   .takeUntil(action$.ofType('STOP'))


sagasにキャンセル付きのリクエストを実装するタスクがあるとします。

function* fetchSaga() {
  yield call(fetchUser);
}

while (yield take('FETCH')) {
  const fetchSaga = yield fork(fetchSaga);
  yield take('FETCH_CANCEL');        
  yield cancel(fetchSaga);
}

Rxではすべてが簡単です。

switchMap(() => fetchUser())
  .takeUntil(action$.ofType('FETCH_CANCEL'))

Rxは完全に異なる状況にあります



。Rxは魔法のハンマーであり、銀の弾丸であるように見えるかもしれません。残念ながら、そうではありません。 Rxに入るしきい値ははるかに高いため、Rxを積極的に使用するプロジェクトに新しい人を紹介することは困難です。

さらに、Observableを使用する場合は、何が起こっているのかを注意深く常に理解することが特に重要です。そうしないと、明らかでないエラーや未定義の動作に遭遇する可能性があります。

action$
  .ofType('DELETE')
  .switchMap(() => 
     Observable
       .fromPromise(deleteRequest)
       .map(() => ({ type: 'DELETE_SUCCESS'})))

かなり単純な仕事をするエピックを書いたら、「DELETE」タイプの各アクションで、要素を削除するAPIメソッドが呼び出されました。ただし、テスト中に問題が発生しました。テスターは奇妙な振る舞いについて不満を述べました-削除ボタンをクリックしても何も起こらなかった。switchMapオペレーターは、一度に1つのObservableのみの実行をサポートすることがわかりました。これは、競合状態に対する一種の保護です。
