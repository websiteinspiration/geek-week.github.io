（ほとんど）役に立たないJavaScriptウェブカメラストリーミングを作成した方法

この記事では、Adobe Flash Playerのようなサードパーティのブラウザプラグインを使用せずに、WebSocketを介してビデオをストリーミングする試みを共有したいと思います。これから何を読んだか。

Adobe Flash-以前のMacromedia Flashは、Webブラウザーで実行するアプリケーションを作成するためのプラットフォームです。 Media Stream APIの実装以前は、Webカメラからビデオや音声をストリーミングしたり、ブラウザーでさまざまな会議やチャットを作成したりするための実質的に唯一のプラットフォームでした。メディア情報を送信するためのプロトコルRTMP（Real Time Messaging Protocol）は、実際には長い間閉鎖されていました。つまり、ストリーミングサービスを向上させたい場合は、Adobe自体のソフトウェアであるAdobe Media Server（AMS）を使用してください。

2012年にしばらくして、アドビは仕様を「放棄し、公開裁判所に提出」しました。エラーを含み、実際には完全ではなかったRTMPプロトコル。その時までに、開発者はこのプロトコルの実装を開始し、Wowzaサーバーが登場しました。 2011年、アドビはRTMPに関連する特許の違法使用についてWowzaに対して訴訟を起こしました。4年後、紛争は世界中で解決されました。

Adobe Flashプラットフォームは20年以上前から存在しており、この間に多くの重大な脆弱性が発見され、2020年までにサポートを停止することが約束されたため、ストリーミングサービスに代わるものはそれほど多くありません。

私のプロジェクトでは、ブラウザーでのFlashの使用を完全に断念することにしました。上記で示した主な理由、またモバイルプラットフォームではFlashがまったくサポートされていないため、Windows（ワインエミュレーター）での開発用にAdobe Flashを展開したくありませんでした。そこで私はJavaScriptでクライアントを書き始めました。これは単なるプロトタイプになります。ストリーミングはp2pに基づいてはるかに効率的に実行できることを後で学びました。ピア-サーバー-ピアのみが存在しますが、まだ準備が整っていないため、さらに別の機会に使用します。

開始するには、websocketsサーバー自体が必要です。私はメロディ囲碁パッケージに基づいて最も簡単なものを作りました：

サーバーコード
package main

import (
	"errors"
	"github.com/go-chi/chi"
	"gopkg.in/olahol/melody.v1"
	"log"
	"net/http"
	"time"
)

func main() {
	r := chi.NewRouter()
	m := melody.New()

	m.Config.MaxMessageSize = 204800

	r.Get("/", func(w http.ResponseWriter, r *http.Request) {
		http.ServeFile(w, r, "public/index.html")
	})
	r.Get("/ws", func(w http.ResponseWriter, r *http.Request) {
		m.HandleRequest(w, r)
	})

         //    
	m.HandleMessageBinary(func(s *melody.Session, msg []byte) {
		m.BroadcastBinary(msg)
	})

	log.Println("Starting server...")

	http.ListenAndServe(":3000", r)
}


クライアント（放送側）では、まずカメラにアクセスする必要があります。これはMediaStream APIを介して行われます。Media Devices API

を通じてカメラ/マイクへのアクセス（解像度）を取得します。このAPIは、ポップアップを表示するMediaDevices.getUserMedia（）メソッドを提供します。カメラやマイクへのアクセス許可をユーザーに求めるウィンドウ。私はGoogle Chromeですべての実験を行いましたが、Firefoxではすべてがほぼ同じように機能すると思います。

次に、getUserMedia（）はPromiseを返します。Promiseには、MediaStreamオブジェクトが渡されます-ビデオとオーディオデータのストリームです。このオブジェクトをsrcのビデオ要素プロパティに割り当てます。コード：

放送側
<style>

</style>
</head>

<script type="application/javascript">
  var
        video = document.getElementById('videoObjectHtml5ApiServer');

//   MediaDevices API,      (    )
// getUserMedia  ,           video   

if (navigator.mediaDevices.getUserMedia) {
        navigator.mediaDevices.getUserMedia({video: true}).then(function (stream) {
          //     video ,        
          video.srcObject = stream;
        });
}
</script>

放送側の準備ができました！次に、ビデオストリームを取得してクライアントに表示してみましょう。これには何が必要ですか？まず第一に、もちろん、ソケット接続。WebSocketオブジェクトに「リスナー」をぶら下げ、「message」イベントをサブスクライブします。バイナリデータを受信すると、サーバーはそれをサブスクライバー、つまりクライアントにブロードキャストします。同時に、「メッセージ」イベントの「リスナー」に接続されたコールバック関数がクライアントでトリガーされ、オブジェクト自体が関数引数（vp8でエンコードされたビデオストリームの一部）に渡されます。

長い間、受信したピースをすぐにビデオ要素に送信して再生することが不可能である理由を理解しようとしましたが、それは不可能であることがわかりました。もちろん、最初にその要素をビデオ要素に接続された特別なバッファーに入れる必要があります。そうしないと、ビデオストリームの再生が開始されません。これを行うには、MediaSource APIとFileReader APIが必要です。

MediaSourceは、メディア再生オブジェクトとこのメディアストリームのソースとの間の一種の仲介役として機能します。MediaSourceオブジェクトには、ビデオ/オーディオストリームソース用のプラグ可能なバッファーが含まれています。1つの機能は、バッファーにはUint8データのみを含めることができるため、そのようなバッファーを作成するにはFileReaderが必要です。コードを見てみましょう。コードはより明確になります。

ビデオストリームを再生する
<style>
  #videoObjectHtml5ApiServer { width: 320px; height: 240px; background: #666; }
</style>
</head>
<body>
<!--    ""     -->
<video autoplay id="videoObjectHtml5ApiServer"></video>

<script type="application/javascript">
  var
        video = document.getElementById('videoObjectHtml5ApiServer'),
         socket = new WebSocket('ws://127.0.0.1:3000/ws'),
        mediaSource = new MediaSource(), //  MediaSource
        vid2url = URL.createObjectURL(mediaSource), //   URL     
        arrayOfBlobs = [],
        sourceBuffer = null; // ,  -

         socket.addEventListener('message', function (event) {
                // ""     
                arrayOfBlobs.push(event.data);
                //    
                readChunk();
            });

         //   MediaSource   ,      
        // / 
        //   ,   ,        
        //     ,      
         mediaSource.addEventListener('sourceopen', function() {
            var mediaSource = this;
            sourceBuffer = mediaSource.addSourceBuffer("video/webm; codecs=\"vp8\"");
        });

      function readChunk() {
        var reader = new FileReader();
        reader.onload = function(e) { 
          //   FileReader  ,     
          //  ""   Uint8Array ( Blob)   , 
          //  ,       /
          sourceBuffer.appendBuffer(new Uint8Array(e.target.result));

          reader.onload = null;
        }
        reader.readAsArrayBuffer(arrayOfBlobs.shift());
      }
</script>


ストリーミングサービスのプロトタイプが完成しました。主な欠点は、ビデオの再生が送信側から100ミリ秒遅れることです。サーバーに送信する前にビデオストリームを分割するときに、これを自分で設定します。また、ノートパソコンで確認したところ、送信側と受信側の間に次第にずれが生じてきましたが、はっきりと見えました。この欠点を克服する方法を探し始めたところ、RTCPeerConnection APIに出会いました。これにより、ストリームを断片に分割するようなトリックなしでビデオストリームを転送できます。蓄積された遅延は、転送前のブラウザでは、各部分がwebm形式にトランスコードされているためだと思います。私はこれ以上掘り下げませんでしたが、WebRTCの研究を開始しました。自分の研究の結果について考え、このコミュニティに興味があれば別の記事を書きます。
