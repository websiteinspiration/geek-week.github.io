gRPCとprotobuf用のもう1つのQtラッパー



少し前まで、Qtに完全に準拠しており、protobufとgRPCの便利でシンプルなラッパーとジェネレーターが十分にないことに戸惑いました。私は記事を見つけました ここではラッパーについて説明しますが、既存のC ++ APIに比べても、その使用方法は実用的ではないように思えました。

gRPCとprotobufについて少し


状況をシミュレーションしてみましょう。マルチプラットフォームプロジェクトを作成し、サービスと通信するためのRPCフレームワークを選択する必要があります。いつでも胸の底をくぐり抜け、「私は自分のフレームワークです」と言うことができますが、私たちは既製のソリューションの時代に生きているように思えます。そのような決定の1つは、有名な会社から長い間提示されました。RPCフレームワークを比較することは想定していません。これはこの記事の目的ではありません。ここでは、gRPCの好きなところをリストします。

  • IDLを簡潔かつ明確にする
  • さまざまなプラットフォーム用の多数のジェネレータの存在
  • プロトタイピングとテストアプリケーションの迅速かつ簡単な作成のためのクライアント/サーバーコードの生成

ポイントへ


Qtが型リフレクションで非常にうまく機能しており、メタ情報の量が一般的に最高レベルであるため、サードパーティのライブラリを散在させることなく、「純粋な」Qtコードを生成する独自のジェネレータが必要であることがわかりました。そこでqtprotobufgenが誕生しました。

qtprotobufgen


qtprotobufgenは、libprotocによって提供されるAPIに基づく、最も単純な本質的にジェネレーターです。あなたがあなたのニーズのためにそのようなものを作りたいのであれば、私は小さなチートを残しておきます。

  • プラグインクラス:: google :: protobuf ::コンパイラ:: CodeGeneratorへの単一のエントリポイントがあり、そこから継承する必要があります
  • 仮想メソッドを生成すると、別の.protoファイルで作業するときに生成が決定されます
  • 仮想GenerateAllメソッドは、生成用に提供された、または依存関係である.protoファイルの完全な配列を操作するときに生成を決定します
  • 仮想のHasGenerateAllメソッドは、本質的には以前のバージョンから残っている遺物です。真を返す

protobuf開発者による既製のソリューションがあるので、自分のパーサー/ジェネレーターをゼロから作成する意欲はなかったと、すぐに言っておく必要があります。ただし、必要に応じて、protocが発行するバイナリストリームを読み取るか、独自のプロトファイルパーサーを作成できます。

開発中に、コンパイルされた言語で記述されたジェネレーターの1つの重大な欠点が明らかになりました。生成とコンパイルを1つのCMakeスタックに入れるのが困難でした。 Qtがメタオブジェクト情報の生成を行うのは、ヘッダーファイルで宣言されたクラスの本体にQ_OBJECTマクロが含まれるヘッダーファイルに基づいているため、mocがさらにコードを生成するために提供するファイルについて、構成段階（cmakeの読み取り）で把握する必要があるためです。解決策として、追加の依存関係を作成せず、その仕事を完全に行いましたが、十分なテストに合格しなかったインタプリタ言語Go（Lang）に頼らなければなりませんでした。

ジェネレーターは既存のプロトコルルールの対象であり、執筆時点では、追加の生成オプションは導入されていません。

protoc --plugin=protoc-gen-qtprotobuf=<path/to/bin>/qtprotobufgen --qtprotobuf_out=<output_dir> <protofile>.proto [--qtprotobuf_opt=out=<output_dir>]

シンプルで使いやすいように、特別に準備されたcmakeルーチンを使用してコードを生成し、それらをcmakeプロジェクトに埋め込むことができます。もっと詳しく

ライブラリについて


APIを詳細に説明してもあまり意味がありません。ドキュメントを作成し、現在利用可能なAPIについてもう少し読むことができます。

プロジェクトは2つの論理部分qtprotobufとqtgrpcに分かれています。名前から、各コンポーネントの目的は明らかだと思います。システムに事前にアセンブルおよびインストールされたライブラリと、サブプロジェクトのcmakeプロジェクトへの統合の両方の統合オプションがあるため、できるだけ便利に使用できるようにしました。

生成されたコードは完全にQMLに*エクスポートされるため、gRPC APIの操作がはるかに簡単になります。

使用する


プロジェクトと統合して生成を実行すると、一連のソースファイルを受け取ります。これらのファイルは、後でスタティックライブラリに収集され、バイナリファイルにリンクされます。最近の変更により、生成されたプロトタイプの静的な登録の可能性が排除されました。したがって、プロジェクトへの登録を管理する必要があります。

...
#include <QtProtobufTypes>
...
int main(int argc, char *argv[])
{
    QtProtobuf::registerProtoTypes();
    ... //   Qt 
}

執筆の時点では、プロトパッケージ用に生成されたすべてのタイプを登録する単一のメソッドはないため、アプリケーションで使用されるすべてのタイプに対してqRegisterProtobufTypeメソッドを呼び出す必要があります。

...
qRegisterProtobufType<MyProtoType>();
...

ライブラリとジェネレータの使用法はREADMEで説明されており、プロジェクトにはいくつかの例が付属しています。gRPC / protobufをまったく知らない人のために、公式ドキュメントを読むことをお勧めします

開発者向け


開発中はTDDに準拠しようとしましたが、TDDから逸脱したくありません。私たちの経験が示しているように、TDDはAPIをリファクタリングまたは更新するときにあなたを救い、隠れた問題を検出するのに役立ちます。したがって、貢献したい場合は、ユニット、ユニット、および機能テストを作成する準備をしてください。

*既知の問題


現在、Qt関連の問題がいくつかあります。それらのいくつかは、私たちの参加の有無にかかわらず解決されましたが、すべてが現在のQtリリースに含まれていませんでした。主なものは、qmlコードからいくつかの基本的なprotobufタイプにアクセスできないことです。JSエンジンとしてV8を使用しているために、QMLで使用できるタイプのセットが非常に制限されていることは誰にも秘密ではないと思います。QMLをカスタムタイプ（fixed32、sint32など）に少し親しみやすくする試みは失敗しましたが、問題の原因を修正することが判明しました。QtNetworkの現在の実装にもいくつかの問題がありますが、Qtチームはすぐに修正します。

予定

  6. CIの実装
