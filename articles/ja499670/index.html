私はjavascriptに切り替えています

Goで5年間執筆した後、次の段階に進むことにしました。私によく仕えて行きます。それはおそらく私が長い間使用できる最高の言語でしたが、今はGoを離れるときです。

Goの制限と問題を十分に見たとは言えません。何年にもわたって、明日の要件をよりよく満たす言語や、より成功したコミュニティが発展してきた言語に切り替えるように促されたものは何もありませんでした。 GoからJavaScriptに切り替えた理由について、Goの欠点を挙げて資料を書きたくありません。私はそのような資料は人生から離婚され、読者にほとんど利益をもたらさないと信じています。それで、JavaScriptの好きなところ、移行のきっかけとなったものについての資料を書きました。





システムを入力して数字を操作する


JavaScriptには、長い間整数データ型がありませんでした。この言語には奇妙な倍精度の数しかありませんでしたが、整数の役割を許容できる場合がありました。今ではすべてがそうではありません。JavaScriptに整数と整数の配列が追加されました。この種類の異なる値を相互に変換できます。これらすべてにより、JSは最高の数値型言語になります。

var sb = new ArrayBuffer(4)
var intArr = new Int32Array(sb)
intArr[0] = 1 << 16
// 65536
intArr[0]
// 65536

これはまさに私が待ち望んでいたことです。正確な計算です。選択した言語がそのような機能をサポートしていることを知っておくのはとても良いことです。

ただし、上記の機能は他の多くの言語で非常に一般的であるため、JavaScriptはこれだけではありません。Int32ArrayどのようInt16Arrayに使用でき、例外をスローせずにすべてが期待どおりに機能します。

var sb = new ArrayBuffer(4)
var intArr = new Int32Array(sb)
intArr[0] = 1 << 16
// 65536
intArr[0]
// 65536
//      
var shortArray = new Int16Array(sb)
shortArray[0]
// 0
shortArray[1]
// 1

ここで質問できます。「結果は、コードが実行されるシステムのバイトオーダーに依存しますか？もしそうなら-なぜ私はアーキテクチャを使用するのですか？上位バイトが最初にどこにあるのですか？」そして私はこれに答えます：「尋ねてくれてありがとう。良い質問"。

次に、これらの整数のもう1つの優れた機能について説明します。つまり、他の言語とは異なり、オーバーフローは期待どおりに処理されます。

var shortArr = new Int16Array(1)
var c = shortArr[0] = 1 << 15 // :   
c == shortArr[0]
// false
shortArr[0]
// -32768
c
// 32768

この時点で、JavaScriptで数値の力をすでに実証していると思いますが、私のポイントを完全に明確にするために、さらに例を挙げます。浮動小数点数を処理するときに

、除算の余り（%）を取得するための演算子を使用できます。そして、この演算子はそれが機能するはずの方法で正確に振る舞います：

3.14 % 5
// 3.14
13.14 % 5
// 3.1400000000000006

さらに、整数の配列は、「定位置」にあるように簡単にソートできます。この場合、プログラマは並べ替えを実行する標準コードを確認する必要さえありません。これにより、JavaScriptは強力な静的型付けにより多くの言語と区別されます。

[-2, -7, 0.0000001, 0.0000000006, 6, 10].sort()
// [-2, -7, 10, 1e-7, 6, 6e-10]
       :
(1,2,3,4,5,6) === (2,4,6)
true

ここで、ユーザーが入力した数値を表す文字列があるとします。この数は1ずつ増やす必要があります。多くの言語では、これを行うために、型変換を実行する大量の退屈なコードを書くのに時間を費やす必要があります。JavaScriptでは、これはコードがエレガントで読みやすいように行われます。

var a = "41"
a += 1
// 411, , , .
var b = "41"
b -=- 1
// 42,

それから、頭の中で覚えるのが難しいものを1つだけ見つけました（しかし、それに慣れる必要があります）。実際、日付を使用する場合、月の番号は0から始まり、それ以外のすべての番号は1から始まるという事実を考慮する必要があります。覚えておく価値のあるものは他にありませんでした。

最後のわら：競争


これにたどり着いたとき、私は一般に、JavaScriptに切り替える必要があることをすでに確信していました。

私がチェックしたかった最後のことは、私が最も好きだったものでした：競争と並行性。

JavaScriptに標準ライブラリがまったくないこと、予測できない型システム、演算子が奇妙に振る舞い、エンティティを比較するときにエンティティの状態を変更し、コンパイルエラー時にシステムをサイレントに保つことは確固たる基盤であり、私がプロジェクトを作成するときに私が通常目指しているのはまさにそれです。しかし、最終的な決定を下す前に、JavaScriptの競争力と並行性が同様に機能することを確認する必要がありました。

GoからJavaScriptに移動するので、このコードが3を3回出力する理由を理解するのは非常に簡単でした。

for (i = 1; i <= 2; ++i) {
  setTimeout(function(){
    console.log(i);
  }, 0);
}

ただし、次のコードフラグメントの結果は、一見すると、同じレベルの直感性で違いがないことを認めなければなりません。

console.log(0)
setTimeout(_ => console.log(1), 0)
requestAnimationFrame(_ => console.log(2))
Promise.resolve().then(_ => console.log(3))
console.log(4)
// 0
// 4
// 3
// 2
// 1

ここで2つの同期関数を呼び出し、「マクロタスク」を計画し、「マイクロタスク」を起動し、アニメーションを作成することをブラウザーに通知すると考えると、 すべてがより明確になります。

これを知っていれば、JavaScriptでの競合するコード実行のメカニズムを理解するのに費用はかかりません。並行性はどうですか？

ここに私がここで使用するコードがあります。彼にcodediodeioをありがとう。

//         :
const start = Date.now();
function log(v){console.log(`${v} \n Elapsed: ${Date.now() - start}ms`);}
//    ,   
function doWork(){
  for(let i = 0; i < 1000000000; i++); //      
  return 'work done';
}
log('before work');
doWork();
log('after work');
// before work
//  Elapsed: 0ms
// after work
//  Elapsed: 624ms

このコードを使用すると、同期モードでタスクを完了するのにかかる時間を調べることができます。

それではプロミスを使ってみましょう：

function doWork(){
  return new Promise((resolve, reject) => {
    for(let i = 0; i < 1000000000; i++);
    resolve('work done');
  })
}
log('before work');
doWork().then(log);
log('after work');
// before work
//  Elapsed: 0ms
// after work
//  Elapsed: 637ms
// work done
//  Elapsed: 637ms

ここでも同じ問題に直面しています。問題はコードがまだ「マクロタスク」として実行されていること、またはこのタスクが同じスレッドをブロックしていることであると誰かが反対するかもしれません。したがって、重いサイクル全体が「マイクロタスク」として実行されるようにコードを再設計しましょう。

function doWork(){
  //  `resolve`        
  return Promise.resolve().then(v =>  {
    for(let i = 0; i < 1000000000; i++);
    return 'work done';
  })
}
log('before work');
doWork().then(log);
requestAnimationFrame(()=>{log('time to next frame')});
log('after work');
// before work
//  Elapsed: 0ms
// after work
//  Elapsed: 1ms
// work done
//  Elapsed: 631ms
// time to next frame
//  Elapsed: 630ms

出力の最初の行を見ると、問題が解決されたと感じるでしょう。しかし、これはそうではありません。メインスレッドで次に使用可能な実行スロットにジョブを移動しただけです。ある程度の同期コードをより高速に実行できたかもしれませんが、非同期コードは依然としてフレームの処理に必要な時間に影響を与えるため、アプリケーションは約0.5秒間ユーザーのアクションに応答しません。

これは本当に私を襲ったものです：JavaScriptで実行する必要がある重い計算がある場合、それらはメインスレッドをブロックし、アプリケーションインターフェイスの速度を低下させます。 async / awaitコンストラクトの使用方法に関するこのビデオやチュートリアルや、この問題を解決するための約束を見つけることができますが、すべてトピックから外れています。これらのプリミティブにより、コードの並列実行ではなく競争力のある実行が可能になります。実際の並列モードで実行できた唯一の種類のプロミスは、組み込みのブラウザプロミスでしたfetch。そして、コードで記述されているものは決して並列に実行されません。

コードを並列モードで実行する方法は？Webワーカーを作成する必要がある-他の言語からのストリームのようなもの。これは、この言語で非常に必要なレースの状態を、並列レースコード実行に導入する優れた新機能です。

ウェブワーカーの使用は非常に簡単です。並行して実行する予定のコードを別のファイルに配置し、を使用してデータを交換できる新しいワーカーのインスタンスを作成する必要がありますpostMessage。これにより、真の並行性が得られます。さらに、すべてがMDNで十分に文書化されています。これらすべてが少し面倒に見えますか？はい、そうですが、プログラマーが自由に使える機会です。さらに、このプリミティブの使用を容易にするライブラリーが作成されます。結局のところ、別の中毒の何が問題になっていますか？

JavaScriptにはこの素晴らしい機能があるので、これは言語が非同期タスクを調整または同期するためのプリミティブによって提供されるメカニズムを持つ必要があることを意味します。正しい？正しい。言語は私たちにpostMessageとMessageChannel（これはのようですchan）を与えてくれます。これはとても良いことです。これらのメカニズムのみを使用する場合、コード内に「スレッドの競合」が存在することはなく、このコード内を移動するのは非常に簡単であり、それについて話すのも簡単です。素晴らしい。

ただし、より生産的なものが必要な場合、イベントをトリガーしてコールバックの計画を待つ必要がないもの、非常に高速に動作するものが必要な場合は、SharedArrayBufferが役立ちます。。ここでは、異なるスレッドで共有できるメモリのフラグメントについて話し、これらのフラグメントからのデータに対してアトミック操作を実行します。そこもされませんMutex、でもselect、でもWaitGroup。これらのプリミティブが必要な場合は、自分で（私のように）記述する必要があります。いかがですか？このAPIは、正しく使用できない値も返します！

はい、結局のところ、誰が簡単な作業を好きになるでしょうか？（これは、JavaScriptオペレーターにはまだ含まれていない、鈍い印です）。

概要


JavaScriptは、競争力、並行性、強く型付けされた変数など、必要なすべてを提供する成熟した言語です。さらに、JSはプログラミングを活性化し、作業をより楽しくして、追加のライブラリーなしで、あらゆる種類の予測不能で神秘的なものを箱から出します。その結果、JavaScriptで書く人は誰でも、デバッグに飽きることはありません。ここで嫌いなことは何ですか？
