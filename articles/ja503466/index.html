SSHエージェントについて

前書き


SSHエージェントはOpenSSHの一部です。この投稿では、エージェントとは何か、それを使用する方法、およびキーを安全に保つためにどのように機能するかについて説明します。エージェント転送とその仕組みについても説明します。エージェント転送を使用するリスクを軽減し、要塞を介して内部ホストにアクセスするときに使用できるエージェント転送の代替方法を紹介します。

SSHエージェントとは


ssh-agentSSHのキーマネージャです。鍵と証明書をメモリに保存し、暗号化せずにすぐに使用できますssh。これにより、サーバーに接続するたびにパスワードを入力する必要がなくなります。システムのバックグラウンドで実行されssh、とは別に、通常は最初の起動時に開始されsshます。

SSHエージェントは以下を行わないため、秘密鍵を安全に保ちます。

  • 重要な情報はディスクに書き込まれません。
  • 秘密鍵をエクスポートすることはできません。

エージェントに保存されている秘密鍵は、メッセージへの署名という1つの目的にのみ使用できます。

しかし、エージェントがメッセージにしか署名できない場合、SSHはトラフィックをどのように暗号化および復号化しますか？

SSH公開鍵と秘密鍵の最初の調査では、SSHがこれらの鍵ペアを使用してトラフィックを暗号化および復号化すると想定するのは当然です。それはまさに私が考えたものです。しかし、そうではありません。SSHキーペアは、初期接続時の認証にのみ使用されます。

たとえば、サーバーの観点から、SSH接続中にユーザーキーを確認する方法は次のとおりです。

  • クライアントはサーバーに公開鍵を提供します。
  • サーバーは、クライアントに秘密鍵を使用して署名するように求める短いランダムメッセージを生成して送信します。
  • クライアントはSSHエージェントにメッセージの署名を要求し、結果をサーバーに送り返します。
  • サーバーは、クライアントの公開鍵を使用して署名を検証します。
  • これでサーバーは、クライアントが秘密鍵を所有していることを証明しました。

その後、接続プロセス中に、SSHセッショントラフィックを暗号化するために使用される、新しい一時的な対称キーのセットが生成されます。これらのキーは、セッション全体を持続させることさえできません。キーの再生成イベントは定期的に発生します。

エージェントプロトコル


SSHは、Unixドメインソケットを使用して、SSHエージェントプロトコルを介してエージェントと通信します。ほとんどの人はssh-agentOpenSSHに付属しているものを使用しますが、多くのオープンソースの選択肢があります。

エージェントプロトコルは非常にシンプルなので、基本的なSSHエージェントを1日か2日で作成できます。基本的な操作はわずかです。

  • 通常の鍵ペア（公開鍵と復号化された秘密鍵）を追加する
  • 限定された鍵ペア（公開鍵と復号化された秘密鍵）を追加する
  • スマートカードからキー（通常または制限付き）を追加（公開キーのみ）
  • キーを削除
  • エージェントに保存されているキーの一覧表示
  • エージェントに保存されているキーでメッセージに署名する
  • パスワードでエージェント全体をロックまたはロック解除する

制限付きキーとは何ですか？これは通常、寿命が限られているか、使用時にユーザーによる明示的な確認が必要なキーです。

コマンドssh-addは、SSHエージェントへのゲートウェイです。彼は、署名以外のすべての操作を実行します。パラメータなしでssh-addを実行すると、ホームディレクトリをスキャンしていくつかの標準キーを探し、それらをエージェントに追加します。デフォルトでは、以下を検索します。

  • ~/.ssh/id_rsa
  • ~/.ssh/id_ed25519
  • ~/.ssh/id_dsa
  • ~/.ssh/id_ecdsa

キーチェーンにキーを追加するとすぐに、キーが自動的に使用されsshます。

ssh-また、macOS
ssh-agentに同梱されているmacOS Keychainは、macOS Keychainにキーのパスフレーズを保存できるため、再起動後にエージェントにキーを簡単に再追加できます。キーチェーンの設定によっては、再起動後にキーロックを解除する必要がある場合があります。キーパスフレーズをキーチェーンに保存するには、コマンドを実行しssh-add -K [ ]ます。パスフレーズは通常、「ローカルアイテム」に保存されます。ssh-agentこれらの保存されたパスフレーズは、必要に応じて自動的に使用されます。

エージェント転送とは


エージェント転送機能により、ローカルSSHエージェントは既存のSSH接続を介して通信し、よりリモートのサーバーに対して透過的に認証できます。たとえば、SSH経由でEC2にログインし、そこからプライベートGitHubリポジトリのクローンを作成するとします。エージェント転送を行わない場合は、GitHub秘密鍵のコピーをEC2ホストに保存する必要があります。エージェントがリダイレクトされると、EC2のSSHクライアントはGitHubでの認証にローカルコンピューターのキーを使用できます。

エージェント転送の仕組み


まず、少し背景。 SSH接続は複数のチャネルを持つことができます。次に、一般的な例を示します。bastion-host（ジャンプボックス）へのインタラクティブな接続は、1つのチャネルで実行されます。接続でエージェント転送が有効になっている場合（通常はを使用ssh -A）、2番目のチャネルがバックグラウンドで開き、エージェント要求をローカルコンピュータに転送します。

に関してはssh、リモートとローカルの間に違いはありませんssh-agent。 SSHは常に環境変数$SSH_AUTH_SOCKを調べて、エージェントのUnixドメインソケットを見つけます。 SSHDは、エージェント転送が有効になっているリモートホストに接続すると、エージェント転送チャネルに関連付けられたリモートUnixドメインソケットを作成し、$SSH_AUTH_SOCKそれを指すようにエクスポートします。

使用 ProxyJump：より安全な代替手段


要塞ホスト（ジャンプボックス）を通過する場合、エージェントの転送は実際には必要ありません。最良のアプローチは、ディレクティブを使用すること ProxyJumpです。

別のチャネルを介してエージェントをリダイレクトする代わりに、ProxyJump要塞を介してローカルSSHクライアントの標準入出力をリダイレクトし、さらにリモートホストにリダイレクトします。仕組みは次のとおりです。

  1. 実行ssh -J bastion.example.com cloud.computer.internalしcloud.computer.internalて、要塞ホストを介してに接続しますbastion.example.com。cloud.computer.internalでDNSを検索して見つけることができるホスト名ですbastion.example.com。
  2. SSHクライアントは、エージェントからのキーを使用してに接続しbastion.example.comます。
  3. SSHDを要塞に接続した後、cloud.computer.internalこの接続をローカルのSSHクライアントに接続して転送します。
  4. ローカルSSHクライアントは、接続を再び通過しますが、今回は cloud.computer.internal

SSHセッションでは、SSHと考えることができます。sshが要塞で実行されることはありません。代わりに、それはsshdに接続cloud.computer.internalした後、第2の接続を行うローカルのSSHに戻って、この接続（標準入力および出力）の制御を提供します。

ProxyJumpの設定

bastion-host thisとしましょうbastion.example.com。私は~/.ssh/configこのように設定できます：

Host bastion.example.com
	User carl

Host *.computer.internal
	ProxyJump bastion.example.com
	User carl

次にssh cloud.computer.internal、要塞を介して内部宛先に接続するために実行します-エージェント転送なし。

ProxyJumpが機能しない場合...

古いバージョンのSSHおよびSSHD（2016年にリリースされたバージョン7.2より前）は、をサポートしていませんProxyJump。ただしProxyCommand、netcatを使用して同等の操作を実行できます。次に例を示します。

ssh -o ProxyCommand="ssh bastion.example.com nc %h %p" cloud.computer.internal

ここでの魔法は、SSH自体がSSHに使用するプロキシサーバーであることです。パーツnc %h %pは、cloud.computer.internalポート22 へのrawソケット接続を開くだけです。親sshコマンドの標準入力/出力は直接に渡されるProxyCommandため、親sshはプロキシ接続を介して内部ホストに認証できます。
