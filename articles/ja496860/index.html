最初のDI：Typescriptアプリケーションのインターフェイスに関する最初のDI

私のライブラリの1つであるFirst DIを共有しています。React、Preact、Mithrilなどのライブラリーのブラウザー・アプリケーションでの依存性注入の問題を解決するために長年私を助けてきました。First DIを作成するとき、C＃のDIライブラリと、autofac、java spring autowired、ninjectなどのJava言語のイデオロギーが基礎として採用されました。これらの言語のライブラリと同様に、First DIはリフレクションとTypescriptインターフェイスに基づいて動作します。

DIとは何ですか？


つまり、Dependecy Injection（以下DI）は、クリーンアーキテクチャを構築するための主要なパーツの1つです。これにより、プログラムの任意の部分を、プログラムのこの部分の別の実装に置き換えることができます。また、DIはプログラムの一部を実装および置換するためのツールです。

特に、次のことが可能になります。

  • 各プラットフォームのロジック実装を個別に置き換えます。たとえば、ブラウザでNotifications APIを介して通知を表示し、モバイルアプリケーションをアセンブルするときに、モバイルプラットフォームのAPIを介して通知に置き換えます。
  • さまざまな環境でロジックの実装を置き換えます。たとえば、開発者のローカルマシンからアクセスできない支払いゲートウェイまたは通知配信システムを製品回路で使用できます。したがって、開発段階で実際の支払いや郵送を行う代わりに、ロジックをスタブに置き換えることができます。
  • データソースを置き換えます。たとえば、サーバーへの実際のリクエストの代わりに、ビジネスロジックやレイアウトをテストしてレイアウトに準拠しているかどうかをテストするために、事前に準備されたデータを利用します。
  • さらに多くの便利な用途がありますが、今はそうではありません...

まず、作業用のツールを準備します。

使用準備


使用するには、3つの簡単なことを行うだけです。

  1. Reflect -metadata polyfillを接続します。
  2. tsconfigファイルで、emitDecoratorMetadataオプションとexperimentalDecoratorsオプションを有効にします。1つ目はリフレクションの生成を可能にし、2つ目はデコレーターのサポートを含みます
  3. 空のデコレータを作成します。たとえば、const Reflection =（..._ params：Object []）：void => {}、またはライブラリから既製のものを使用します。

次に、その最も単純な形式での使用を示します。

シンプルな使い方


たとえば、純粋なアーキテクチャを使用して記述されたプログラムを見てみましょう。

import { autowired, override, reflection } from "first-di";

@reflection // typescript  
class ProdRepository { //   

    public async getData(): Promise<string> {
        return await Promise.resolve("production");
    }

}

@reflection
class MockRepository { //       

    public async getData(): Promise<string> {
        return await Promise.resolve("mock");
    }

}

@reflection
class ProdService {

    constructor(private readonly prodRepository: ProdRepository) { }

    public async getData(): Promise<string> {
        return await this.prodRepository.getData();
    }

}

class ProdController { //   React, Preact, Mithril  .

    @autowired() //  
    private readonly prodService!: ProdService;

    // constructor  ,   React, Preact, Mithril
    //      

    public async getData(): Promise<string> {
        return await this.prodService.getData();
    }

}

if (process.env.NODE_ENV === "test") { //    
    override(ProdRepository, MockRepository);
}

const controllerInstance = new ProdController(); //   
const data = await controllerInstance.getData();

if (process.env.NODE_ENV === "test") {
    assert.strictEqual(data, "mock"); //  
} else {
    assert.strictEqual(data, "production"); //  
}

純粋なアーキテクチャでは、DIが機能を開始するために、自動配線とオーバーライドの2行のコードのみを追加する必要があります。そして今、このプログラムにはすでに実稼働用の実装とテスト用の実装があります。

First DIは主にクライアントアプリケーション用に記述されているため、デフォルトでは依存関係はシングルトンとして実装されます。この動作をグローバルに変更するために、デフォルトのオプションのほか、自動配線されたデコレーターとオーバーライド（）メソッドのパラメーターがあります。

次のように変更できます。

//  1 
import { DI, AutowiredLifetimes } from "first-di";
DI.defaultOptions.lifeTime = AutowiredLifetimes.PER_INSTANCE;

//  2  
import { autowired, AutowiredLifetimes } from "first-di";
@autowired({lifeTime: AutowiredLifetimes.PER_INSTANCE})
private readonly prodService!: ProdService;

//  3  
import { override, AutowiredLifetimes } from "first-di";
override(ProdRepository, MockRepository, {lifeTime: AutowiredLifetimes.PER_INSTANCE});

これは、DIを使用する最も簡単な方法です。しかし、企業では、特定の実装の代わりに、実装される実装の契約であるインターフェースが使用されます。これには2つ目の操作モードがあります。

専門的な使用


業務用として、特定の実装をインターフェースに置き換えます。しかし、typescriptは実行時にインターフェイスを実行するためのコードを生成しません。幸いなことに、簡単な解決策があります。理解するには、理論を思い出す必要があります...インターフェイスとは何ですか？これは完全に抽象クラスです！

さいわい、typescriptは完全に抽象クラスをサポートし、ランタイムコードを生成し、抽象クラスを使用して型を記述することができます。

この情報を使用して、エンタープライズプログラムを作成します。

import { autowired, override, reflection } from "first-di";

abstract class AbstractRepository { //    

    abstract getData(): Promise<string>;

}

@reflection
class ProdRepository implements AbstractRepository {

    public async getData(): Promise<string> {
        return await Promise.resolve("production");
    }

}

@reflection
class MockRepository implements AbstractRepository {

    public async getData(): Promise<string> {
        return await Promise.resolve("mock");
    }

}

abstract class AbstractService { //    

    abstract getData(): Promise<string>;

}

@reflection
class ProdService implements AbstractService {

    private readonly prodRepository: AbstractRepository;

    constructor(prodRepository: AbstractRepository) {
        this.prodRepository = prodRepository;
    }

    public async getData(): Promise<string> {
        return await this.prodRepository.getData();
    }

}

class ProdController { //   React, Preact, Mithril  .

    @autowired()
    private readonly prodService!: AbstractService;

    // constructor  ,   React, Preact, Mithril
    //      

    public async getData(): Promise<string> {
        return await this.prodService.getData();
    }

}

override(AbstractService, ProdService);

if (process.env.NODE_ENV === "test") {
    override(AbstractRepository, MockRepository);
} else {
    override(AbstractRepository, ProdRepository);
}

const controllerInstance = new ProdController();
const data = await controllerInstance.getData();

if (process.env.NODE_ENV === "test") {
    assert.strictEqual(data, "mock");
} else {
    assert.strictEqual(data, "production");
}

これで、デフォルトの実装がない既製のProfessionalプログラムができました。すべてのコントラクトは、アプリケーションのアセンブリ段階ですでに実装されています。

アプリケーションにDIを組み込むのはとても簡単です。また、C＃とJavaで記述した人は、Web開発で既存のエクスペリエンスを使用できます。

その他の特徴


DIの複数のコピーを使用する：

import { DI } from "first-di";
import { ProductionService } from "../services/ProductionService";

const scopeA = new DI();
const scopeB = new DI();

export class Controller {

    @scopeA.autowired()
    private readonly serviceScopeA!: ProductionService;

    @scopeB.autowired()
    private readonly serviceScopeB!: ProductionService;

    // constructor  ,   React, Preact, Mithril
    //      

    public async getDataScopeA(): Promise<string> {
        return await this.serviceScopeA.getData();
    }

    public async getDataScopeB(): Promise<string> {
        return await this.serviceScopeB.getData();
    }

}

拡張性、あなた自身のDIを書くことができます：

import { DI } from "first-di";

class MyDI extends DI {
    // extended method
    public getAllSingletons(): IterableIterator<object> {
        return this.singletonsList.values();
    }
}

なぜあなた自身が書かれていて使われていないのですか？


その理由は簡単です。このDIが作成されたとき、代替手段はありませんでした。これは、angular 1がもはや関連せず、angular 2が終了しない時期でした。クラスはJavascriptに現れ、typescriptに反映されました。ちなみに、反射の出現は行動の主な原動力となっています。

InversifyJS-その現在の形でさえ私には合いません。定型文が多すぎます。また、行または文字で登録すると、依存関係をリファクタリングする可能性がなくなります。
