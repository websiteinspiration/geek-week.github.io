キャップはまた、上記の方法の両方が、ユーザーが情報の取得に関与しているという事実に大きな欠点があることを示唆しています。
誰もがWindowsで自分のマシンの名前をどこで見るべきかを知っているとき、私はコンピュータリテラシーを高めるオプションを考慮しません。ビジネスは高貴ですが、非常に困難です。そして、会社に従業員の離職があった場合、それは完全に悲惨です。ほとんどの場合、彼らはログインを覚えていません。

彼は自分の魂を注ぎ、今では要点を述べた。
ハブロフスク市民のアイデアが基礎として取られましたミッテルこの記事から。
ユーザーがWindowsにログオンすると、ログオンスクリプトは必要な情報（時間とマシン名）をユーザーアカウントの特定の属性に入力するという考え方です。システムを終了すると、同様のログオフスクリプトが実行されます。

アイデア自体は気に入りましたが、実装に問題がありました。
  1. ユーザー用のログオンスクリプトとログオフスクリプトが記述されているグループポリシーはドメイン全体に適用されるため、ユーザーがログインするすべてのマシンでスクリプトが機能します。ワークステーション（Microsoft RDSやCitrix製品など）と共にターミナルソリューションを使用する場合、このアプローチは不便です。
  2. データは、平均的なユーザーが読み取り専用アクセス権を持つユーザーアカウントのDepartment属性に入力されます。スクリプトは、ユーザーアカウント属性に加えて、コンピューターアカウントの部門属性も変更します。ユーザーは、デフォルトでは変更できません。したがって、ソリューションを機能させるために、作成者はADオブジェクトのデフォルトのセキュリティ設定を変更することを推奨します。
  3. 日付形式は宛先マシンのローカリゼーション設定に依存するため、1つのマシンから2018年11月10日14:53を取得し、他の11/10/18 2:53 pmを取得できます。


これらの欠点に対処するために、次のことが行われました。
  1. GPOはドメインにリンクしていませんが、マシンのOUにリンクしています（ユーザーとマシンを異なるOUで分離し、他のユーザーにアドバイスしています）。同時に、ループバックポリシー処理モードがマージモードに設定されます。
  2. スクリプトは、Info属性のユーザーアカウントにのみデータを入力します。この属性は、ユーザーが自分のアカウントに対して個別に変更できます。
  3. 属性値を生成するコードを変更しました


スクリプトは次のようになります。
SaveLogonInfoToAdUserAttrib.vbs
On Error Resume Next
Set wshShell = CreateObject("WScript.Shell")
strComputerName = wshShell.ExpandEnvironmentStrings("%COMPUTERNAME%")
Set adsinfo = CreateObject("ADSystemInfo")
Set oUser = GetObject("LDAP://" & adsinfo.UserName)
strMonth = Month(Now())
If Len(strMonth) < 2 then
  strMonth = "0" & strMonth
End If
strDay = Day(Now())
If Len(strDay) < 2 then
  strDay = "0" & strDay
End If
strTime = FormatDateTime(Now(),vbLongTime)
If Len(strTime) < 8 then
  strTime = "0" & strTime
End If
strTimeStamp = Year(Now()) & "/" & strMonth & "/" & strDay & " " & strTime
oUser.put "info", strTimeStamp & " <logon>" & " @ " & strComputerName
oUser.Setinfo
