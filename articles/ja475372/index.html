TLS 1.3に基づくフロントエンドドメイン

前書き



Cisco、BlueCoat、FireEyeなどの著名なメーカーが提供する最新の企業コンテンツフィルタリングシステムは、より強力な対応製品であるDPIシステムと非常に多くの共通点があります。DPIシステムは、国レベルで集中的に実装されています。両方の作業の本質は、着信および発信のインターネットトラフィックを検索し、ブラック/ホワイトリストに基づいて、インターネット接続を禁止することを決定することです。また、どちらも作業の基本において同様の原則に依存しているため、それらを回避する方法にも多くの共通点があります。

DPIと企業システムの両方を非常に効率的にバイパスできるテクノロジの1つは、ドメイン前面のテクノロジです。その本質は、ブロックされたリソースに移動し、評判の良い別のパブリックドメインの後ろに隠れているという事実にあります。これは、明らかにgoogle.comなどのシステムによってブロックされません。

このテクノロジーについてはすでに多くの記事が書かれており、多くの例が示されています。ただし、DNS-over-HTTPSと暗号化されたSNIテクノロジ、および最近議論されたテクノロジ、およびTLS 1.3プロトコルの新しいバージョンは、ドメイン前面に向けた別のオプションを検討する機会を提供します。

テクノロジーを扱う


最初に、基本概念について少し決めて、誰が誰で、なぜこれがすべて必要なのかを誰もが理解できるようにします。 eSNIメカニズムについても触れましたが、その仕組みについては後で説明します。 eSNI（暗号化サーバー名表示）メカニズムは、TLS 1.3プロトコルでのみ使用可能なSNIの安全なバージョンです。主なポイントは、要求が送信されるドメインに関する情報を含めて暗号化することです。

次に、eSNIエンジンが実際にどのように機能するかを見てみましょう。

最新のDPIソリューションによってブロックされているインターネットリソースがあるとします（たとえば、有名な急流トラッカー-rutracker.nlを取り上げます）。トレントトラッカーのWebサイトにアクセスしようとすると、リソースがブロックされている標準のプロバイダースタブ



が表示されます。ILVWebサイトでは、このドメインは実際にはストップリストにリストされています。



whoisをリクエストすると、ドメイン自体がクラウドプロバイダーCloudflareの背後に「隠れている」ことがわかります。



しかし、ILVの「スペシャリスト」とは異なり、より技術的に精通している（または有名な規制機関の苦い経験によって教えられた）従業員は、IPアドレスでサイトを愚かに禁止せず、ストップリストにドメイン名を入力しました。これは、同じIPアドレスの背後に隠れている他のドメインを確認し、それらの1つにアクセスして、アクセスがブロックされていないことを確認することで簡単に確認できます



。すべての通信はhttpsプロトコルを介して行われており、ビーラインからのhttps証明書の置換にまだ気付いていないため、プロバイダーのDPIはブラウザーがどのドメインに移動するかをどのようにして見つけますか？彼は千里眼ですか、それとも私がついているのですか？

Wireshark経由のトラフィックを見て、この質問に答えてみましょう。



スクリーンショットは、最初にブラウザがDNS経由でサーバーのIPアドレスを受信し、次に宛先サーバーとの標準TCPハンドシェイクが発生し、次にブラウザーがサーバーへのssl接続の確立を試行することを示しています。これを行うには、SSLクライアントのHelloパケットを送信します。このパケットには、ソースドメイン名がクリアテキストで含まれています。このフィールドは、接続を適切にルーティングするために、cloudflareフロントエンドサーバーで必要です。これは、プロバイダーDPIが私たちをキャッチし、接続を切断する場所です。同時に、プロバイダーからスタブを受け取りません。サイトが無効になっているか、単に機能していないかのように、標準のブラウザーエラーが表示されます。



次に、Firefoxの手順に記載されているように、ブラウザーでeSNIメカニズムをオンにします。
これを行うには、Firefoxの設定ページを開きます：configと次の設定を有効にします：

network.trr.mode = 2;
network.trr.uri = https://mozilla.cloudflare-dns.com/dns-query
network.security.esni.enabled = true

その後、リンクを使用してcloudflare Webサイトの設定が正しいことを確認し、トレントトラッカーでフォーカスを再試行します。



出来上がり。私たちのお気に入りのトラッカーは、VPNやプロキシなしで開きました。次に、Wiresharkのトラフィックダンプを見てみましょう。



今回は、ssl client helloパッケージには宛先ドメインが明示的に含まれていませんが、代わりにパッケージに新しいフィールドが表示されました-encrypted_server_name-これはrutracker.nlの値が含まれ、フロントエンドのcloudflareサーバーのみがこのフィールドを復号化できます。その場合、プロバイダーのDPIは手を洗って、そのようなトラフィックを許可するしかありません。ただし、暗号化に関する他のオプションはありません。

それで、テクノロジーがブラウザでどのように機能するか-私たちは見ました。それでは、より具体的で興味深いものに適用してみましょう。また、最初に、同じカールでeSNIを使用してTLS 1.3を操作する方法を説明します。同時に、eSNIベースのフロントエンドドメイン自体がどのように機能するかを確認します。

eSNIを使用したドメインフロントエンド


curlは標準のopensslライブラリを使用してhttps経由で接続するため、最初に、そこでeSNIサポートを提供する必要があります。opensslマスターブランチではまだeSNIがサポートされていないため、特別なopensslブランチをダウンロードし、コンパイルしてインストールする必要があります。

githubからリポジトリを複製し、通常どおりコンパイルします。

$ git clone https://github.com/sftcd/openssl
$ cd openssl
$ ./config

$ make
$ cd esnistuff
$ make

次に、curlでリポジトリのクローンを作成し、アセンブルされたopensslライブラリを使用してそのコンパイルを構成します。

$ cd $HOME/code
$ git clone https://github.com/niallor/curl.git curl-esni
$ cd curl-esni

$ export LD_LIBRARY_PATH=/opt/openssl
$ ./buildconf
$ LDFLAGS="-L/opt/openssl" ./configure --with-ssl=/opt/openssl --enable-esni --enable-debug

ここでは、opensslが配置されているすべてのディレクトリ（この場合、/ opt / openssl /）を正しく示し、構成プロセスがエラーなしで成功することを確認することが重要です。

構成が成功すると、次の行

が表示されます。警告：esni ESNIが有効になっていますが、実験的とマークされています。注意して使用してください！

$ make

パッケージを正常にビルドした後、特別なopenssl bashファイルを使用してcurlを構成および実行します。便宜上、curlを含むディレクトリにコピーします。

cp /opt/openssl/esnistuff/curl-esni

DNSとTLSパケットをWiresharkに同時に書き込みながら、cloudflareサーバーへのテストhttpsリクエストを実行します。

$ ESNI_COVER="www.hello-rkn.ru" ./curl-esni https://cloudflare.com/

サーバーの応答では、opensslとcurlからの多くのデバッグ情報に加えて、cloudflareからコード301のHTTP応答が返されます。

HTTP/1.1 301 Moved Permanently
< Date: Sun, 03 Nov 2019 13:12:55 GMT
< Transfer-Encoding: chunked
< Connection: keep-alive
< Cache-Control: max-age=3600
< Expires: Sun, 03 Nov 2019 14:12:55 GMT
< Location: https://www.cloudflare.com/

これは、リクエストが宛先サーバーに正常に配信され、聞いて処理されたことを示しています。

次に、wiresharkのトラフィックダンプを見てみましょう。この場合、プロバイダーDPIが見たもの。



最初のcurlが、cloudflareサーバーの公開eSNIキーをDNSサーバーに向けたことがわかります-_esni.cloudflare.comへのTXT DNSリクエスト（パッケージ番号13）。次に、opensslライブラリを使用して、curlはTLS 1.3リクエストをcloudflareサーバーに送信しました。このサーバーでは、SNIフィールドが前の手順で取得した公開鍵で暗号化されています（パッケージ番号22）。ただし、eSNIフィールドに加えて、SSL-helloパッケージの一部として、フィールドが通常の-オープンSNIで挿入されました。このSNIは任意の順序で指定できます（この場合はwww.hello-rkn.ru）。

このオープンSNIのフィールドは、cloudflareサーバーによる処理時には考慮されず、プロバイダーのDPIを偽装しただけでした。 cloudflareサーバーはssl-helloパッケージを受信し、eSNIを復号化し、そこから元のSNIを抽出し、何も起こらなかったかのように処理しました（eSNIの開発中に計画どおりにすべてを行いました）。

この場合の唯一のことは、DPI（_esni.cloudflare.comでのプライマリDNSクエリ）に頼ることができます。ただし、DNSクエリは、このメカニズムが内部からどのように機能するかを示すためにのみ開いています。

最後にDPIから土を取り除くために、すでに述べたDNS-over-HTTPSメカニズムを使用します。簡単な説明は、DOHです。これは、HTTPS経由でDNSクエリを送信することにより、中間者攻撃から身を守ることができるプロトコルです。

リクエストを再度実行しますが、今回はDNSではなくhttpsプロトコルを使用して公開eSNIキーを取得します。

ESNI_COVER="www.hello-rkn.ru" DOH_URL=https://mozilla.cloudflare-dns.com/dns-query ./curl-esni https://cloudflare.com/

リクエストトラフィックダンプは下のスクリーンショットに示されています。



最初にcurlがDoHプロトコルを介してmozilla.cloudflare-dns.comサーバーにアクセスし（サーバー104.16.249.249へのhttps接続）、SNI暗号化のための公開鍵値を取得し、次にドメインwww.hello-rkn.ruの背後に隠れて、宛先サーバーに。

上記で指定されたmozilla.cloudflare-dns.comリゾルバーDoHに加えて、有名な悪の企業などの他の人気のあるDoHサービスを使用できます。
次のリクエストを実行します。

ESNI_COVER="www.kremlin.ru" DOH_URL=https://dns.google/dns-query ./curl-esni https://rutracker.nl/

そして私たちは答えを得ます：

< HTTP/1.1 301 Moved Permanently
< Date: Sun, 03 Nov 2019 14:10:22 GMT
< Content-Type: text/html
< Transfer-Encoding: chunked
< Connection: keep-alive
< Set-Cookie: __cfduid=da0144d982437e77b0b37af7d00438b1a1572790222; expires=Mon, 02-Nov-20 14:10:22 GMT; path=/; domain=.rutracker.nl; HttpOnly; Secure
< Location: https://rutracker.nl/forum/index.php
< CF-Cache-Status: DYNAMIC
< Expect-CT: max-age=604800, report-uri="https://report-uri.cloudflare.com/cdn-cgi/beacon/expect-ct"
< Server: cloudflare
< CF-RAY: 52feee696f42d891-CPH



この場合、dns.google DoHリゾルバーを使用してブロックされたサーバーrutracker.nlに切り替え（タイプミスはなく、今では有名な企業には独自のファーストレベルドメインがあります）、すべてのDPIによってブロックすることは厳しく禁止されている別のドメインで覆われています死の痛みの下で。回答によれば、リクエストが正常に処理されたことを理解できます。

プロバイダーDPIがカバーとして渡すオープンSNIに応答することの追加チェックとして、他の禁止されたリソース、たとえば別の「良い」トレントトラッカーの背後にあるrutracker.nlへのリクエストを実行できます。

$ ESNI_COVER="rutor.info" DOH_URL=https://dns.google/dns-query ./curl-esni https://rutracker.nl/

サーバーからの応答はありません。リクエストはDPIシステムによってブロックされます。
