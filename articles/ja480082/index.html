多数の監視オブジェクトを含むZabbixのMySQLでパーティショニングを使用する

サーバーとサービスを監視するために、NagiosとMuninに基づく複合ソリューションを長い間使用してきました。ただし、この束には多くの欠点があるため、多くの人と同様に、Zabbixを積極的に利用しています。この記事では、削除するメトリックの数を増やし、MySQLデータベースのボリュームを増やすときに、最小限の労力でパフォーマンスの問題を解決する方法について説明します

ZabbixでMySQLデータベースを使用する際の問題


データベースは小さく、データベースに格納されているメトリックの数は少なかったものの、すべてが素晴らしいものでした。 Zabbixサーバー自体を起動する通常のハウスキーパープロセスは、古くなったレコードをデータベースから正常に削除し、成長を防ぎました。ただし、キャプチャされたメトリックの数が増加し、データベースのサイズが特定のサイズに達するとすぐに、すべてが悪化しました。ハウスキーパーは、割り当てられた時間間隔でデータを削除するための管理を停止し、古いデータがデータベースに残り始めました。ハウスキーパーの運用中に、Zabbixサーバーの負荷が増大し、長時間続く可能性がありました。現状をどうにかして解決する必要があることが明らかになった。

これは既知の問題であり、Zabbixで大量の監視を行うほとんどすべての人が同じ問題に直面していました。MySQLをPostgreSQLやElasticsearchに置き換えるなど、いくつかのソリューションもありましたが、最も簡単で最も実績のあるソリューションは、MySQLデータベースにメトリックデータを格納するパーティションテーブルに切り替えることでした。私たちはこの方法で行くことにしました。

通常のMySQLテーブルからパーティションテーブルへの移行


Zabbixは十分に文書化されており、メトリックを格納するテーブルは既知です。これらはテーブルです：historyフロート値が格納されているhistory_str場所、短い文字列値history_textが格納されている場所、長いテキスト値history_uintが格納されている場所、整数値が格納されている場所trends変更のダイナミクスを格納するテーブルもありますが、サイズが小さく、後で戻ってくるので、変更しないことにしました。

一般に、処理する必要のあるテーブルは明確でした。私たちは、月の番号に基づいて、最後の週を除いて、毎週のパーティションを作成することにしました。1か月あたり4つのパーティション：1日から7日、8日から14日、15日から21日、22日から1日（翌月）。問題は、Zabbixサーバーの作業を中断したり、メトリックを収集したりせずに、必要なテーブルを「オンザフライ」でパーティション化する必要があったことです。

奇妙なことに、これらのテーブルの構造そのものが、これを支援してくれました。たとえば、テーブルのhistory 構造は次のとおりです。

`itemid` bigint(20) unsigned NOT NULL,
`clock` int(11) NOT NULL DEFAULT '0',
`value` double(16,4) NOT NULL DEFAULT '0.0000',
`ns` int(11) NOT NULL DEFAULT '0',

ここで

KEY `history_1` (`itemid`,`clock`)

ご覧のとおり、各メトリックは最終的にテーブルに入力され、2つの非常に重要で便利なitemidフィールドとclockフィールドが含まれています。したがって、我々はよく名前で、例えば、一時テーブルを作成することができhistory_tmp、それを分割設定し、その後、テーブルからすべてのデータをそこに転送しhistory、その後、テーブル名前を変更historyするhistory_oldと、テーブルhistory_tmpにはhistory、私たちから十分ではない持っていること、およびそのデータを追加history_oldしますhistory そして削除しhistory_oldます。これは完全に安全に実行でき、何も失うことはありません。上記のitemidフィールドとclockフィールドは、序数ではなく特定の時間へのリンクメトリックを提供するためです。

したがって、何もオフにしたり、停止したりしないでください。主なことは、MySQLサーバー自体に十分な空きディスク領域があることです。上記の表のそれぞれにhistory、history_text、history_str、history_uint接尾辞«_tmp»、それが元のテーブルと同じボリュームになることを考えるとテーブルを作成するには、少なくとも十分なスペース。

上記の各テーブルについてすべてを数回説明することはせず、そのうちの1つだけのテーブルの例ですべてを検討しますhistory。

そのため、テーブルhistory_tmp の構造に基づいて空のテーブルを作成しますhistory。

CREATE TABLE `history_tmp` LIKE `history`;

必要なパーティションを作成します。たとえば、1か月間やってみましょう。各パーティションは、クロックフィールドの値に基づくパーティションルールに基づいて作成され、タイムスタンプと比較します。

ALTER TABLE `history_tmp` PARTITION BY RANGE( clock ) (
PARTITION p20190201 VALUES LESS THAN (UNIX_TIMESTAMP("2019-02-01 00:00:00")),
PARTITION p20190207 VALUES LESS THAN (UNIX_TIMESTAMP("2019-02-07 00:00:00")),
PARTITION p20190214 VALUES LESS THAN (UNIX_TIMESTAMP("2019-02-14 00:00:00")),
PARTITION p20190221 VALUES LESS THAN (UNIX_TIMESTAMP("2019-02-21 00:00:00")),
PARTITION p20190301 VALUES LESS THAN (UNIX_TIMESTAMP("2019-03-01 00:00:00"))
);

この演算子は、作成したテーブルにパーティションを追加しますhistory_tmp。クロックフィールドの値が「2019-02-01 00:00:00」より小さいデータがパーティションp20190201に分類されることを明確にして、次にクロックフィールドの値が「2019-02-01 00:00:00」より大きいが小さいデータ「2019-02-07 00:00:00」はパーティーp20190207に該当します。
重要なメモ：クロックフィールドの値が「2019-03-01 00:00:00」以上のパーティション分割テーブルにデータがある場合はどうなりますか？このデータには適切なパーティションがないため、それらはテーブルに分類されず、失われます。したがって、そのようなデータの損失を回避するために、追加のパーティションをタイムリーに作成することを忘れないでください（以下について）。
これで、一時テーブルが準備されました。データを入力します。プロセスには非常に長い時間がかかる可能性がありますが、幸い、他の要求はブロックされないため、辛抱する必要があります。

INSERT IGNORE INTO `history_tmp` SELECT * FROM history;

IGNOREキーワードは、テーブルにまだデータがないため、最初の入力時には必要ありませんが、データを追加するときに必要になります。さらに、このプロセスを中断して、データを入力するときに最初からやり直す必要がある場合にも役立ちます。

そのため、しばらくして（おそらく数時間）、最初のデータのアップロードが完了しました。ご存知のように、テーブルにhistory_tmp はテーブルからのすべてのデータが含まれるのではなくhistory、クエリの開始時にテーブルに含まれていたデータのみが含まれます。ここでは、実際には、選択肢があります。もう1つのパスを作成するか（充填プロセスが長時間続いた場合）、すぐに上記のテーブルの名前を変更します。まず、2パス目を見てみましょう。まず、最後に挿入されたレコードの時刻を理解する必要がありますhistory_tmp。

SELECT max(clock) FROM history_tmp;

受け取ったとします：1551045645。次に、取得した値をデータ入力の2番目のパスで使用します。

INSERT IGNORE INTO `history_tmp` SELECT * FROM history WHERE clock>=1551045645;

この一節はもっと速く終わるはずです。ただし、最初のパスが数時間実行され、2番目のパスも長時間実行された場合、3番目のパスを作成することは正しいかもしれません。

最後にhistory_tmp、次のようにして、レコードの最後の挿入時刻を取得する操作を再度実行します。

SELECT max(clock) FROM history_tmp;

1551085645 を取得したとしましょう。この値を保存します-再入力するために必要になります。

そして今、実際には、一次データの入力がhistory_tmp 終わったら、テーブルの名前を変更します：

BEGIN;
RENAME TABLE history TO history_old;
RENAME TABLE history_tmp TO history;
COMMIT;

最初のRENAMEの後、2番目のRENAMEが実行されるまでテーブルが存在しないため、存在しないテーブルにデータを挿入する瞬間を回避するために、このブロックを1つのトランザクションとして設計history しました。ただし、RENAME操作の間にテーブルにhistory データが入り、テーブル自体がまだ存在していない場合でも（名前の変更により）、無視できる少数の挿入エラーが発生します（バンクではなく監視しています）。

これでhistory、パーティション化された新しいテーブルが作成されましたが、テーブルにデータを挿入する最後のパスで受信した十分なデータがありませんhistory_tmp。しかし、このデータは表にありますhistory_old そこから共有します。これを行うには、以前に保存した値1551085645が必要になります。なぜこの値を保存し、現在のテーブルの最大充填時間を使用しなかったのhistoryですか？新しいデータがすでに入って来ており、間違った時間を取得するからです。したがって、データを測定します。

INSERT IGNORE INTO `history` SELECT * FROM history_old WHERE clock>=1551045645;

この操作の終了後、新しいパーティションテーブルhistory には、古いテーブルにあったすべてのデータと、テーブルの名前が変更された後のデータが含まれています。テーブルhistory_old はもう必要ありません。すぐに削除することも、削除する前にバックアップ（パラノイアがある場合）を作成することもできます。

全体のプロセスは、テーブルに対して繰り返す必要があり、上述のhistory_str、history_text とhistory_uint。

Zabbixサーバー設定で修正する必要があるもの


現在、データの履歴に関するデータベースの保守は私たちの肩にかかっています。これは、Zabbixが古いデータを削除しないようにする必要があることを意味します。 Zabbixサーバーがデータ自体をクリーンアップしようとしないようにするには、Zabbix Webインターフェイスに移動し、メニューで「管理」を選択し、次に「一般」サブメニューを選択して、右側のドロップダウンリストで「履歴をクリア」を選択する必要があります。表示されるページで、「履歴」グループのチェックマークをすべて外し、「更新」ボタンをクリックします。これhistory*により、ハウスキーパーによる不要なテーブルクリーニングが防止されます。

同じページのグループ「変更のダイナミックス」に注意してください。これは単なるテーブルです。trendsこれに戻ると約束しました。また、サイズが大きくなりすぎてパーティション化する必要がある場合は、このグループもオフにして、このテーブルをテーブルの場合とまったく同じように処理しますhistory*。

さらなるデータベースのメンテナンス


以前に作成されたように、パーティションテーブルでの通常の操作では、時間内にパーティションを作成する必要があります。これは次のようにして行うことができます：

ALTER TABLE `history` ADD PARTITION (PARTITION p20190307 VALUES LESS THAN (UNIX_TIMESTAMP("2019-03-07 00:00:00")));

さらに、パーティションテーブルを作成し、Zabbixサーバーでそれらをクリーンアップすることを禁止したため、古いデータを削除することが今や懸念事項となっています。幸い、何の問題もありません。これは、データが不要になったパーティションを削除するだけです。

例えば：

ALTER TABLE history DROP PARTITION p20190201;

日付範囲を指定したDELETE FROMステートメントとは異なり、DROP PARTITIONは数秒で完了し、サーバーをまったくロードせず、MySQLでレプリケーションを使用する場合と同じようにスムーズに機能します。
