GRASPテンプレート：情報エキスパート

こんにちは、ハブロフスク。ウラジスラフ・ロダンと接触。現在、OTUSポータルでソフトウェアアーキテクチャと高負荷ソフトウェアアーキテクチャのコースを教えています。今回は、新しいコース「建築とデザインパターン」の開始を見越して、少し著作権資料を書くことにしました。読書を楽しむ。




前書き


クレイグ・ラーマンの著書 『UMLとパターンの適用、第3版』に記載されているGRASPパターンは、GoFパターンの一般化であり、OOP原則の直接の帰結です。これらは、論理ラダーで欠落しているステップを補完します。これにより、OOPの原則からGoFパターンを取得できます。GRASPテンプレートは、（GoFのような）デザインパターンではなく、クラス間で責任を分散するための基本原則です。実際にはそれほど人気が​​ないことが示されていますが、GRASPパターンのフルセットを使用して設計されたクラスを分析することは、優れたコードを記述するための前提条件です。

GRASPテンプレートの完全なリストは、9つの要素で構成されています。

  • 情報専門家
  • クリエーター
  • コントローラ
  • 低カップリング
  • 高い凝集力
  • ポリモーフィズム
  • 純粋な製造
  • 間接
  • 保護されたバリエーション

リストから最も明白で最も重要なパターンであるInformation Expertを検討することをお勧めします。

情報専門家


言い回し


科学的定式化を避け、このパターンの本質は次のように表現できます。情報が含まれている場所で情報を処理する必要があります。

違反の例


明らかな単純さと自明性にもかかわらず、どのプロジェクトのコードでも、この原則に対する多くの違反を見つけることができると私は確信しています。

最も単純なクラスシステムを考えてみましょう：OrderItem'ov（注文明細）のリストを含むOrder（注文）、その要素には次にGood（製品）とその数量が含まれ、製品には価格、名前などが含まれる場合があります。

@Getter
@AllArgsConstructor
public class Order {
    private List<OrderItem> orderItems;
    private String destinationAddress;
}

@Getter
@AllArgsConstructor
public class OrderItem {
    private Good good;
    private int amount;
}

@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}


注文の量を計算するという単純なタスクがあります。この問題の解決策にあまり注意を払っていない場合は、Orderクラスのオブジェクトを処理するクライアントコードで次のようなものをすぐに作成できます。

public class Client {
    public void doSmth() {
        
    }
    
    private int getOrderPrice(Order order) {
        List<OrderItem> orderItems = order.getOrderItems();
        
        int result = 0;
        
        for (OrderItem orderItem : orderItems) {
            int amount = orderItem.getAmount();
            
            Good good = orderItem.getGood();
            int price = good.getPrice();
            
            result += price * amount;
        }
        
        return result;
    }
}


このソリューションを分析してみましょう。

まず、価格設定に関連するビジネスロジックを追加し始めると、Client :: getOrderPriceメソッドコードは必然的に大きくなるだけでなく、あらゆる種類のif-s（年金受給者向け割引、休日割引、卸売購入）、最終的には、このコードを読み取ることが不可能になるという事実につながり、変更ははるかに少なくなります。

次に、UMLダイアグラムを作成すると、Clientクラスの依存関係がOrder、OrderItem、Goodの3つのクラスにあることがわかります。これらのクラスを操作するためのすべてのビジネスロジックを描画します。つまり、OrderItemまたはGoodをOrderとは別に再利用する場合（たとえば、倉庫に残っている商品の価格を計算する場合）は、ビジネスロジックがクライアントコード内にあるため、これを実行できず、コードの重複が避けられなくなります。

この例では、get'ovのチェーンがあるほとんどすべての場所と同様に、クライアントコードが情報を処理し、そのOrderを含んでいるため、Information Expertの原則に違反しています。

応用例


原則に従って責任を再配分してみましょう：

@Getter
@AllArgsConstructor
public class Order {
    private List<OrderItem> orderItems;
    private String destinationAddress;
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }
}

@Getter
@AllArgsConstructor
public class OrderItem {
    private Good good;
    private int amount;

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

public class Client {
    public void doSmth() {
        Order order = new Order(new ArrayList<>(), "");
        order.getPrice();
    }
}


これで、情報はそれを含むクラスで処理され、クライアントコードはOrderにのみ依存し、その内部構造については何も疑いません。Order、OrderItemとGood、またはOrderItemとGoodのクラスは、プロジェクトのさまざまな部分で使用できる個別のライブラリにアセンブルできます。

結論


カプセル化から生まれた情報エキスパートは、GRASPの責任分担の最も基本的な原則の1つです。その違反は、コードの知覚を単純化し（驚きを最小限にするという原則）、再利用の可能性を追加し、クラス間の接続数を減らすことで、簡単に判別および除去できます。モノリシックアプリケーション、マルチレベル、サーバーレスアーキテクチャの機能を学習できるフレームワーク内の無料のウェビナーにご参加
