新しいシリアル化がJavaで表示される可能性があります

OpenJDKのWebサイトに、新しい改良されたシリアル化を言語に導入して古いものを置き換えるというアイデアを説明する新しい研究ドキュメントが登場しました。

Javaでのシリアライゼーションはバージョン1.1以降、つまりほぼ誕生してから存在しています。一方、シリアライゼーションは、java.io.Serializableインターフェースからこのクラスを継承することにより、クラスをシリアライズ可能にする非常に便利なメカニズムです。おそらく、この単純さでさえ、Javaがネットワークアプリケーションを迅速かつ効率的に作成できるようになったため、Javaが世界中で非常に人気を得た主な理由の1つになったのでしょう。

一方、Javaでのシリアライゼーションの実装方法には、アプリケーションのサポートのコストを増加させ、セキュリティを低下させ、プラットフォームの進化を遅らせる問題が多数含まれています。

Javaでのシリアル化の何が問題になっていますか？最も深刻な問題をリストします：

  • シリアライゼーション（およびデシリアライゼーション）は、言語メカニズムをバイパスします。フィールドアクセス修飾子（プライベート、保護）を無視し、コンストラクターを使用せずにオブジェクトを作成します。つまり、これらのコンストラクターに存在する可能性のある不変条件を無視します。攻撃者は、無効なデータでデータを置き換えることにより、このような脆弱性を悪用する可能性があり、逆シリアル化中にそれらが正常に飲み込まれる可能性があります。
  • シリアライズ可能なクラスを書くとき、コンパイラーは何の助けにもならず、エラーを検出しません。たとえば、直列化可能クラスのすべてのフィールドがそれ自体が直列化可能であることを静的に保証することはできません。または、readObject、writeObject、readResolveなどのメソッドの名前を間違えると、シリアル化中にこれらのメソッドが使用されなくなります。
  • シリアライゼーションは通常のバージョン管理メカニズムをサポートしていないため、シリアライズ可能なクラスを変更して古いバージョンとの互換性を維持することは非常に困難です。
  • シリアライゼーションはストリーミングエンコーディング/デコーディングに強く関連しています。つまり、エンコーディング形式を標準のものとは異なるものに変更することは非常に困難です。さらに、標準形式はコンパクトでも、効率的でも、人が読める形式でもありません。

Javaでの既存のシリアライゼーションの根本的な間違いは、それがプログラマーには「見えない」ようにしようとすることです。それは単にjava.io.Serializableを継承し、仮想マシンによって実行されるある種の暗黙の魔法を受け取ります。
それどころか、プログラマーはオブジェクトの構築と分解を担当する構造を明示的に記述する必要があります。これらの構成要素は言語レベルである必要があり、リフレクションではなくフィールドへの静的アクセスを介して記述する必要があります。

別のシリアライゼーションエラーは、あまりにも多くのことを実行しようとしていることです。彼女は、オブジェクト（ループを含む可能性がある）の任意のグラフをシリアル化し、その状態を壊すことなく逆シリアル化できるようにするタスクを自分自身に設定します。

このエラーは、タスクを簡略化し、オブジェクトのグラフではなく、アイデンティティの概念がないデータツリー（JSONなど）をシリアル化する場合に修正できます。

オブジェクトモデルに自然に適合し、逆シリアル化にコンストラクターを使用し、エンコード形式から分離され、バージョン管理をサポートするシリアル化を行う方法は？この目的のために、アノテーションが救いとなり、Javaにまだ含まれていない言語、つまりパターンマッチングの可能性があります。例えば：

public class Range {
    int lo;
    int hi;
  
    private Range(int lo, int hi) {
        if (lo > hi)
            throw new IllegalArgumentException(String.format("(%d,%d)",
                                                             lo, hi));
        this.lo = lo;
        this.hi = hi;
    }
 
    @Serializer
    public pattern Range(int lo, int hi) { 
        lo = this.lo;
        hi = this.hi;
    }
    
    @Deserializer
    public static Range make(int lo, int hi) {
        return new Range(lo, hi);
    }
}

この例では、Rangeクラスが宣言されています。これは、クラスの2つの特別なメンバー、シリアライザと@Serializerおよび@Deserializerアノテーションが付けられたデシリアライザを使用してシリアル化の準備ができています。シリアライザはパターンのデコンストラクタを通じて実装され、デシリアライザはコンストラクタが呼び出される静的メソッドを通じて実装されます。したがって、逆シリアル化中に、コンストラクターで指定された不変のhi> = loが必ずチェックされます。
このアプローチには魔法はなく、通常の注釈が使用されるため、Javaプラットフォーム自体だけでなく、どのフレームワークでもシリアライゼーションを実行できます。つまり、エンコード形式は絶対に何でもかまいません（バイナリ、XML、JSON、YAMLなど）。

シリアライザとデシリアライザは一般的な方法であるため、プログラマはその実装に大きな自由度を持っています。たとえば、オブジェクトのメモリ内での表現方法とは異なるオブジェクトの表現を選択する場合があります。たとえば、LinkedListは一連のリンクではなく、1つの連続した配列にシリアル化できます。これにより、プレゼンテーションがよりシンプルで効率的でコンパクトになります。

このアプローチのバージョン管理は、@ Serializerおよび@Deserializerアノテーションの特別なバージョンフィールドを使用して実装されます。

class C {
    int a;
    int b;
    int c;
   
    @Deserializer(version = 3)
    public C(int a, int b, int c) { 
        this a = a;
        this.b = b;
        this.c = c;
    }
    
    @Deserializer(version = 2)
    public C(int a, int b) { 
        this(a, b, 0);
    }

    @Deserializer(version = 1)
    public C(int a) { 
        this(a, 0, 0);
    }
    
    @Serializer(version = 3)
    public pattern C(int a, int b, int c) {
        a = this.a;
        b = this.b;
        c = this.c;
    }
}

この例では、バージョンに応じて、3つのデシリアライザーの1つが呼び出されます。
シリアライザとデシリアライザをシリアライゼーション以外の目的で使用したくない場合はどうなりますか？これを行うには、それらをプライベートにできます。ただし、この場合、特定のシリアル化フレームワークは、そのようなコードがパッケージが深いリフレクトアクセス用に開かれていないモジュール内にある場合、リフレクションを介してそれらにアクセスできません。そのような場合のために、言語に別の新しい構成を導入することを提案します：オープンクラスメンバー。例えば：

class Foo {
    private final InternalState is;

    public Foo(ExternalState es) {
        this(new InternalState(es));
    }
    
    @Deserializer
    private open Foo(InternalState is) { 
        this.is = is;
    }
    
    @Serializer
    private open pattern serialize(InternalState is) { 
        is = this.is;
    }
}

ここでは、シリアライザーとデシリアライザーはopenキーワードでマークされており、setAccessibleに対してオープンになります。

したがって、新しいアプローチは古いアプローチとは根本的に異なります。そのアプローチでは、クラスはシリアライズ可能として設計されており、そのままプラットフォームに提供されるわけではありません。これには追加の労力が必要ですが、シリアル化がより予測可能で安全であり、エンコード形式とシリアル化フレームワークから独立しています。
