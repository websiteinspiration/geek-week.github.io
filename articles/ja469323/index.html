symfonyプロジェクトのゲッター/セッターとカプセル化の問題

最近では、Symfonyを使用しており（1年弱）、私が取り組む機会があったすべてのプロジェクトでは、エンティティは常に、プライベートフィールドとそれらのベアセッター/ゲッターのみを含むように作成されました。

この記事では、このアプローチが危険である理由と例について説明します。つまり、古き良きカプセル化に違反し、バグを含むコードの記述を引き起こし、システムの複雑さを増します。
この記事では、さまざまなビルダーでのセッターのトピックと、セッターによる依存関係注入のトピックは省略します（私たちは承認しないとだけ言います）。 DDD、Rich Modelのような複雑なトピック、カップリング/凝集、その他のスマートワードについては何もありません。カプセル化についてお話しください。猫へようこそ。

コードに移動します。タイプヒントとタイプリターンの表示を少し省略して、オブジェクトを操作するときに、次のコードが実用的な観点とどのように異なるかを考えてみましょう。

$userName = $user->name;
$user->name = $newUserName;
コードから：
$userName = $user->getName();
$user->setName($newUserName);

繰り返しますが、タイプを無視すると、何も起こりません。

この場合のオブジェクト自体は、通常の構造とほとんど変わりません。つまり、本質的に、そのようなコードは、手続き型コード（データを処理して変更する手続き）を書くことを私たちに促します。

カプセル化


カプセル化とは何ですか？カプセル化は通常、データ、動作、詳細、および条件を隠すこととして理解されています。詳細を明らかにすると、私たちの頭脳と貧弱な育成はすぐに、ここで利用可能なすべてのものを使用するように私たちを促し、今は最小の抵抗の道に沿ってタスクを達成します。

単純なオブジェクトを「破壊」しようとします。つまり、オブジェクトを無効な状態にします。

class Order
{
     private const STATUS_OPEN = 'open';
     private const STATUS_DELIVERED = 'delivered';

     private ProductCollection $products;
     private string $deliveryStatus;
     private ?DateTime $deliveryDate = null;

     public function deliver()
     {
          if ($this->isDelivered()) {
               throw new OrderDomainException('Order already delivered.');
          }

          $this->deliveryDate = new DateTime();
          $this->deliveryStatus = self::STATUS_DELIVERED;
     }

     private function isDelivered(): bool
     {
         return $this->deliveryStatus === self::STATUS_DELIVERED;
     }

     //  
}

注文を施設の外のどこかに「間違って」配達することは可能ですか？たとえば、ステータスを示したり、現在の日付を記したりしないでください。条件でロジックのフレーミングを開始することは可能ですか？注文は送信されましたか？この形式では、そのような可能性はありません。これはカプセル化の例です-動作とデータは非表示になっています。使用できるのは1つの方法だけです。さらに、このコードには優れた内部凝集力があります。

次に、この例を見てください。

class Order
{
    private const STATUS_OPEN = 'open';
    private const STATUS_DELIVERED = 'delivered';

    private ProductCollection $products;
    private string $deliveryStatus;
    private ?DateTime $deliveryDate = null;

    public function getDeliveryStatus(): string
    {
        return $this->deliveryStatus;
    }

    public function setDeliveryStatus(string $deliveryStatus): void
    {
        $this->deliveryStatus = $deliveryStatus;
    }

    public function getDeliveryDate(): ?DateTime
    {
        return $this->deliveryDate;
    }

    public function setDeliveryDate(?DateTime $deliveryDate): void
    {
        $this->deliveryDate = $deliveryDate;
    }

    public function getProducts(): ProductCollection
    {
        return $this->products;
    }

    public function setProducts(ProductCollection $products): void
    {
        $this->products = $products;
    }
}

このクラスのオブジェクトで何ができるでしょうか？（質問は修辞的です）。条件：注文は配送されましたか-どこに配置されますか、それはまったく同じですか、それとも同じである必要がある場所とまったく同じですか？または、それは塗りつけられて複製され、さらには異なりますか？誰もそれがどうなるかを教えてくれません。コードは私たちにはっきりと伝えます-あなたが望むように、それがどうなるか-私たちは見ていきます。カプセル化と呼ばれるOOP原則は崩壊しました。その結果、このオブジェクトは数十の手順で処理されます。そして今、再び高い質問です-このオブジェクトは、それを操作する上で、パブリッククラスフィールドを持つオブジェクトとどのように違うのですか？

セッター


カプセル化 するメソッドであるセッターは、オブジェクトフィールドをパブリックにします。実際、それらはそのような手続き型アプローチの主な問題です-外部のすべてのロジック、データとロジックの整合性に対する制御の欠如。

これは、非常に複雑で紛らわしいコード-デバッグが難しく、ロジックを理解するのが難しいコードに満ちています。オブジェクトパラメータの設定が予期しない場所、レイヤー、および野生の条件で発生し、他のデータから分離されているためです。

まあ、そのようなオブジェクトは簡単に無効になります。教義の本質の例：

ゲッター


ゲッターはセッターに勝るものはありません。また、カプセル化を解除するのにも役立ちます。
もちろん、私はデータ転送に必要なメソッド（DTOの読み取りと作成、シリアル化など）について話しているのではありません。

以下は、フィールドを「読み取る」ことができるために発生した問題にすでに驚いているコードです。コードは私の頭からのものですが、それは私の目が見たものと私の手を刻印したものに近いものです。

$order = $this->orderRepository->find($orderId);

if ($order->getStatus() === Order::STATUS_ACTIVE

このコードでは、すべての条件、ビジネス上の意思決定を行うための論理式は外にあります。

これらすべてから、ビジネスロジックの詳細は無関係なコード行で調査する必要があることがわかります。本質的に、これは非常に複雑なコードです。何をするのか理解するのが難しくなるからです。あなたは心に留めておくか、同僚に心に留めておく必要があります。常に読んでよく考え、常にエラーやバグ、そして多くのコピーと貼り付けを行う必要があります。コードは、手続き型の「サービス」と「マネージャー」によって膨張します。

合計


セッター/ゲッターを含むコードはコードを著しく複雑にし、複雑なドメインではそれほど単純ではないため、ロジックは理由の単位に適合しませんが、「実際に」いくつかの制御構造またはデータパーティクルを具体化するコードからのWebが邪魔になると、プログラマーの神経系のために、クリーンでテスト可能な穏やかなコードを書くのはかなり難しいです。

残念なことに、多くのSymfonyプログラマーは貧血モデルの問題を完全に認識しておらず、そのようなオブジェクトを単なるデータウェアハウス（DAO）と見なし、サービス/マネージャーが生成します。上記の引数が、OOPラッパーの手続き型コードの問題について考えるように促してくれることを願っています（このトピックでは、手続き型プログラミングに関する不満はありません）。

概念的に間違いや誤りをすばやく修正する準備ができています-譲歩はありません:)
