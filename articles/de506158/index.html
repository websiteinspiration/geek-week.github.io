useSWR ist meine neue Lieblingsbibliothek von React

Vor Beginn des Kurses "React.js Entwickler" wurde eine Übersetzung des Artikels erstellt .


In den letzten Monaten habe ich an einer Anwendung für NextJS gearbeitet. Jede Woche wird es mehr und mehr. Die Anwendung verwendet Axios für API-Aufrufe und Unstated-Next für die Statusverwaltung . Es gibt viele API-Aufrufe, aber wir möchten nicht, dass Benutzer eine Reihe von Ladebildschirmen sehen. Daher lagern wir die Ergebnisse von axios Anrufe in unstated Repositories.

Wir sind jedoch auf ein Problem gestoßen. Gewölbe selbst werden immer komplexer. Seiten erfordern manchmal mehrere API-Aufrufe und stützen sich wiederum auf die Ergebnisse anderer API-Aufrufe. Aus Tagen wurden Wochen und aus Wochen Monate, während unsere nicht genannten Gewölbe immer umständlicher wurden. Wir sind auf seltsame Fehler gestoßen, als unsere hausgemachte Caching-Logik Schwierigkeiten hatte, mit unerwarteten Extremfällen umzugehen.

Dann dachten wir, dass der Weg einfacher sein sollte.
Und hier ist er. Und es wurde sogar von Vercel , dem Erfinder von NextJS, entwickelt .

SWR kennenlernen


Der Name SWR stammt von Stale-While-Revalidate , einer Caching-Methode, die in der Frontend-Entwicklung immer beliebter wird. Sie können zwischengespeicherte Inhalte sofort herunterladen und sofort aktualisieren, um in Zukunft neue Inhalte zu verarbeiten. In unserem Fall haben wir den perfekten Kompromiss zwischen Leistung und Benutzererfahrung erzielt.

Verwendung von useSWR


useSWR ist eine von Vercel entwickelte React-Hook-Bibliothek . Sie können Daten von einer API oder einer anderen externen Quelle abrufen, im Cache speichern und dann rendern.

Schauen wir uns zunächst ein Beispiel für eine React-Komponente an, die eine TODO-Liste von einem JSON-Server empfängt und rendert.

import React from "react";
import "./App.css";
const todosEndpoint = "http://localhost:3001/todos";
const TodoApp = () => {
  const [todos, setTodos] = React.useState([]);
  React.useEffect(() => {
    const getData = async () => {
      const response = await fetch(todosEndpoint);
      const data = await response.json();
      setTodos(data);
    };
     getData();
    }, []);
  return (
    <div className="App">
      {todos.map((todo) => (
        <div key={todo.id}>{todo.title}</div>
      ))}
    </div>
  );
};
export default TodoApp;

Schauen wir uns nun dieselbe Komponente an, die jedoch mit useSWR neu geschrieben wurde .

import React from "react";
import useSWR from "swr";
const todosEndpoint = "http://localhost:3001/todos";
const getData = async () => {
  const response = await fetch(todosEndpoint);
  return await response.json();
};
const TodoApp = () => {
  const { data: todos } = useSWR(todosEndpoint, getData);
  
return (
    <div>
      {todos && todos.map(todo => (
        <div key={todo.id}>{todo.title}</div>
      )}
    </div>
  );
};
export default TodoApp;

Wie Sie sehen können, ist es der vorherigen Implementierung sehr ähnlich.

In diesem Beispiel verwenden wir useSWR (Schlüssel, Abruf, Optionen) , um unsere TODO-Liste abzurufen . Schlüssel in useSWR wird für das Caching benötigt. In unserem Fall haben wir todosEndpoint verwendet . Als Abrufprogramm haben wir eine asynchrone Funktion übergeben, die eine TODO-Liste abruft.

Es ist zu beachten, dass useSWR keine Ahnung hat, wie Sie Daten abrufen. Sie können jede beliebige asynchrone Data Mining-Strategie verwenden. Sie können Fetch , Axios oder sogar GraphQL verwenden . Während Ihre Funktion Daten asynchron zurückgibt,useSWR wird sich freuen.

Der Aufruf useSWR gibt die folgenden Parameter zurück:

  • Daten : Schlüsseldaten, die mit dem Abruf abgerufen wurden (nicht definiert, falls nicht bereits geladen);
  • Fehler : Fehler durch Abrufer verursacht ;
  • isValidating : Ein Boolescher Wert, der angibt , ob eine Anforderung oder eine erneute Validierung ausgeführt wird.
  • mutate (data ?, shouldRevalidate) : Funktion zum Ändern zwischengespeicherter Daten.

Komplexeres Beispiel


Schauen wir uns ein komplexeres Beispiel an, das mehr von useSWR zurückgegebene Parameter verwendet . In diesem Beispiel rufen wir ein Element der TODO-Liste ab und rendern das Kontrollkästchen für den vollständigen Status unserer Aufgabe aus der Liste. Wenn das Kontrollkästchen angeklickt wird, senden wir eine PUT - Anfrage zu aktualisieren TODO, und dann rufen wir mutieren die aktualisieren useSWR Cache .

import React from "react";
import useSWR from "swr";
import "./App.css";
const todosEndpoint = "http://localhost:3001/todos";
const getTodo = async (id) => {
  const response = await fetch(`${todosEndpoint}/${id}`);
  return await response.json();
};
const updateTodo = async (id, todo) => {
  const response = await fetch(`${todosEndpoint}/${id}`, {
    method: "PUT",
    headers: {
      "Content-type": "application/json; charset=UTF-8",
    },
    body: JSON.stringify(todo),
  });
  return await response.json();
};
const TodoApp = () => {
  const todoId = 1;
  const key = `${todosEndpoint}/${todoId}`;
  const { data: todo, mutate } = useSWR(key, () =>
    getTodo(todoId)
  );
  const toggleCompleted = async () => {
    const newTodo = {
      ...todo,
      completed: !todo.completed,
    };
    await updateTodo(todoId, newTodo);
    mutate(newTodo);
  };
  if (!todo) {
    return <div>Loading...</div>;
  }
  return (
    <div>
      <p>{todo.title}</p>
      <div>
        <input
          type="checkbox"
          id="completed"
          name="completed"
          value="completed"
          checked={todo.completed}
          onChange={toggleCompleted}
        />
        <label htmlFor="completed">Completed</label>
      </div>
    </div>
  );
};

Die Verwendung von mutate ist eine hervorragende Möglichkeit, die sichtbare Leistung Ihrer Webanwendung zu steigern. Wir können Daten lokal ändern und die Ansicht ändern, ohne warten zu müssen, bis die entfernte Datenquelle aktualisiert ist. useSWR validiert sie sogar erneut und ersetzt sie durch die neuesten Daten im Hintergrund.

Warum brauche ich dafür eine Bibliothek?


Sie können diese Frage stellen. Möglicherweise verfügen Sie bereits über einen Statusverwaltungsmechanismus in Ihrer Anwendung, und Sie sehen keinen Sinn darin, eine Bibliothek eines Drittanbieters für etwas so Einfaches wie das Extrahieren und Zwischenspeichern von Daten zu verwenden.

Und ich werde Ihnen antworten, dass das Zwischenspeichern von Daten nicht einfach ist. Und wenn Ihre Anwendung wächst, wird es noch schwieriger. Wenn Sie auf extreme Fälle stoßen und diese verarbeiten, erhalten Sie komplexe Repositorys und Anbieter, die schwer zu verstehen und schwer zu warten sind. Bugs werden aus dem Nichts verschüttet.
"In der Informatik gibt es nur zwei komplexe Probleme: die Ungültigmachung des Caches und das Erfinden von Namen" - Phil Carleton

Anstatt Ihre eigene Data Mining-Lösung zu implementieren, sollten Sie sich auf die bewährte Lösung verlassen, die von einem der angesehensten Unternehmen im React-Ökosystem entwickelt wurde.

Wie uns useSWR geholfen hat


Die Portierung des Problems der Datenextraktion in unserer Anwendung auf die Verwendung von SWR hat viele Vorteile gebracht.

1. Er erlaubte uns, den Code zu entfernen

Pull-Anfragen, die unnötigen Code löschen, aber keinen neuen hinzufügen - meine Favoriten. Je weniger Code in der Anwendung enthalten ist, desto unwahrscheinlicher ist es, dass Fehler auftreten. Und das ist gut. Mit zunehmendem Alter schätze ich die Einfachheit immer mehr. Mit useSWR konnten wir nicht angegebenen Speicher vollständig entfernen, was unsere Anwendung vereinfachte und das Verständnis erleichterte.

2. Er vereinfachte die Verbindung neuer Entwickler mit dem useSWR- Projekt

Wie die meisten Open Source-Projekte verfügt es über eine hervorragende Dokumentation. Die Implementierung unserer eigenen Lösung würde bedeuten, dass wir unsere eigene Dokumentation schreiben und neuen Entwicklern im Projekt den Umgang mit der Datenextraktion beibringen müssen. Jetzt, da wir useSWR verwenden , müssen wir dies nicht mehr tun.

3. Er hat schwierige Dinge vereinfacht.

Unsere Anwendung enthält wie die meisten anderen Anwendungen viele API-Aufrufe. Einige dieser Aufrufe hängen von anderen API-Aufrufen ab. Mit useSWR einfach zu schreibende Hooks für abhängige Abfragen.

4. Die sichtbare Leistung wurde verbessert. Die

Anwendung sieht schneller aus. Benutzer bewerteten es und gaben gutes Feedback.

5. Es aktualisiert veraltete Daten

useSWR aktualisiert veraltete Daten im Fokus. Dies bedeutet, dass Benutzer immer die aktuellste Version der Daten haben, ausgenommen die Downloadzeit.

Fazit


useSWR hatte einen großen Einfluss auf unsere Anwendung. Diese Bibliothek hat unseren Code vereinfacht und die Benutzererfahrung verbessert. Ich kann mir nicht einmal eine andere Bibliothek vorstellen, die so einfach zu implementieren ist und so viele Vorteile bietet wie useSWR . Um mehr über useSWR zu erfahren , können Sie die Site oder das Repository auf GitHub besuchen .

Alternative Bibliotheken


useSWR ist nicht die einzige gute Bibliothek zum Abrufen von Daten. Während es in unserem Fall gut funktioniert hat, ist vielleicht eine andere Bibliothek für Sie besser für die Datenextraktion geeignet.

Reaktionsabfrage


React -Query ist eine Bibliothek von Hooks zum Abrufen, Zwischenspeichern und Aktualisieren von asynchronen Daten in React. Diese beliebte Lösung, die von Tanner Linsi geschrieben wurde , ist im Umfang der Verwendung von SWR sehr ähnlich .

Apollo


Der Apollo- Client ist ein GraphQL- Client , der sich um das Abrufen und Zwischenspeichern Ihrer Daten sowie das Aktualisieren der Benutzeroberfläche kümmert. Wenn Sie GraphQL verwenden , ist Apollo eine großartige Lösung.
