在不增加Laravel内存消耗的情况下解决N + 1查询问题

开发人员使用ORM创建应用程序时的主要问题之一是其应用程序中的N + 1请求。当应用程序生成对对象的每次调用的请求时，N + 1查询问题不是访问数据库的有效方法。当我们从数据库中获取数据列表而不使用延迟加载或急切加载时，通常会发生此问题。幸运的是，Laravel及其ORM Eloquent提供了方便操作的工具，但它们也有一些缺点。

在本文中，我们考虑了N + 1问题，解决该问题的方法以及优化内存消耗的方法。

Class User extends Authenticatable
{
    public function articles()
    {
        return $this->hasMany(Aricle::class, 'writter_id');

    public function first_article()
    {
        return $this->hasOne(Aricle::class, 'writter_id')->orderBy('created_at', 'asc');
    }

    public function scopeWithFirstArticle($query)
    {
        $query->addSelect(['first_article_id' => Article::select('id')
                ->whereColumn('writter_id', 'users.id')

实际上，我们没有为动态通信实现延迟加载。我们只需分配查询结果即可接收用户的第一篇文章。在为贪婪加载和延迟加载访问first_article属性时，这应该同样有效。







Laravel 5.X中的动态连接



不幸的是，我们的解决方案仅适用于Laravel 6及更高版本。Laravel 6和以前的版本使用addSelect的不同实现。为了在框架的旧版本中使用，我们需要更改代码。我们必须使用selectSub来执行子查询：







public function scopeWithFirstArticle($query)
{
    if(is_null($query->toBase()->columns)) {
        $query->select([$query->toBase()->from . '.*']);
    }

    $query->selectSub(
                Article::select('id')
                ->whereColumn('writter_id', 'users.id')
                ->orderBy('created_at', 'asc')
                ->take(1)
                ->toSql(), 
            'first_article_id'
            )->with('first_article');
}
