中国のUSB IRトランシーバープロトコルのリバースエンジニアリング


中国のMicroUSB IRトランシーバーに出会い、Windowsを搭載したコンピューターに接続したいという要望がありました。トランシーバは、Micro USBコネクタを備えた非常にコンパクトなデバイスです。これを使用する唯一の「公式」オプションは、ZaZaRemoteと呼ばれるAndroidアプリケーションを使用することです。

アダプターを介してコンピューターに接続すると、デバイスはHID互換のUSBデバイス\ VID_10C4＆PID_8468として定義されました。このIDによるグーグルでは結果が得られなかったため、プロトコルを逆にする必要がありました。

無効なHID


デバイスクラスは、USB \ Class_03＆SubClass_FF＆Prot_FFとして定義されました。Class_03-HIDデバイス、SubClass_FF-ベンダー固有。hidusb.sysドライバーは、システムによって自動的にインストールされました。HID APIを介してこのドライバーを操作できます。

簡単な小さなプログラムをスケッチしたので、デバイスに関するさまざまな情報を得ることができました。

この機能はもう少し複雑です。送信されるメッセージの最大長が1024バイトに制限されていることがわかります。メッセージはフラグメントに分割されます。フラグメントは、5バイトのヘッダーと最大56バイトのデータ（合計61バイト）で構成されます。ヘッダー構造：

  • buf [0] = 2は定数です。ReportID 2を覚えていますか？これだそうです。また、ReportCount 60-残りのデータのサイズ-も同じです。
  • buf [1] = fragmSize + 3-フラグメントデータサイズ+ 3、つまり サイズは、この変数に続くバイトから計算されます。
  • buf [2] = UsbPackCounterは単なるカウンタ、1..15です。
  • 派手な式によって計算されたbuf [3]の値は、フラグメントの数にすぎません。次のように書き換えることができます。

    buf[3] = tsize / 56;
if (tsize % 56) buf[3]++;
  • buf [4] = fragmCnt-フラグメント番号、1..buf [3]。

形成されたフラグメントは、CallIntMethodへの呼び出しを通じて送信されます。次のタイプがあります。

jint (JNICALL *CallIntMethod)(JNIEnv *env, jobject obj, jmethodID methodID, ...);

HexRaysが今回は管理しなかったことがわかります-引数ではv6 = JNIEnv * envのみです。ただし、アセンブラコードではすべてが整っています。

LDR     R2, [R2,#(dword_B2A4 - 0xB284)] ; jmethodID methodID

ZaZaRemoteとの交換のダンプ


以前の成功にもかかわらず、これまでのところ私は主要なことを達成できていません-デバイスにIRコマンドを送信させることです。アプリケーションが大きすぎて混乱し、USBトラフィックをダンプしたかっただけです。しかし、Androidアプリケーションの場合にこれを行う方法は？エントリはVirtualBox-eでAndroid-x86として見つかりました。x86は決してARMではないという事実にもかかわらず、それでもNativeBridgeメカニズムを通じてネイティブARMバイナリを実行できます。必要なソフトウェアをインストールして構成することで、このアプリケーションをVirtualBoxで動作させることができました。

要求された許可は間違いなくこのソフトウェアへの信頼を刺激します。


アプリケーションを起動してUSB転送を設定することで、USBトラフィックを盗聴することができました。そのため、IRコマンドの送受信を初期化する一連のコマンドと、IRパケットのサンプルデータを取得しました。

このデバイスはIRコマンドを送信するだけでなく受信することもできることがわかりましたが、受信は10 cmの距離から、その後は1回おきに、かなりまあまあのように機能しました。

概要


調査の過程で、Tiqiaa Tview IRトランシーバーのUSBプロトコルが完全に復元され、ドライバーのinfファイルとそれを操作するためのソフトウェアが書き込まれました。
