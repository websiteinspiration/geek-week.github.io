DockerのOOMkillerは思ったより難しい

また会いましたね。コースの開始を見越して、「Java Developer」は別の小さな資料の翻訳を準備しました。



最近、Plumbr APMユーザーの1人が、コード137でdocker-containerを緊急停止するという奇妙な問題を抱えていました。ネストされた人形に似た、いくつかのネストされたコンテナーと仮想マシンの構成はシンプルです。

  • Ubuntuを備えた独自のIronサーバー。
  • 内部にUbuntuを備えた多くのDockerコンテナー。
  • Dockerコンテナー内のJava仮想マシン。

問題の調査中に、このトピックに関するDocker ドキュメントが見つかりました。理由は、コンテナーの手動停止、またはメモリ不足とそれに続くoomkiller（メモリ不足キラー）の介入のいずれかであることが明らかになりました。

syslogを見て、実際にoomkillerが呼び出されたことを確認します。

[138805.608851] java invoked oom-killer: gfp_mask=0xd0, order=0, oom_score_adj=0
[138805.608887] [<ffffffff8116d20e>] oom_kill_process+0x24e/0x3b0
[138805.608916] Task in /docker/264b771811d88f4dbd3249a54261f224a69ebffed6c7f76c7aa3bc83b3aaaf71 killed as a result of limit of /docker/264b771811d88f4dbd3249a54261f224a69ebffed6c7f76c7aa3bc83b3aaaf71
[138805.608902] [<ffffffff8116da84>] pagefault_out_of_memory+0x14/0x90
[138805.608918] memory: usage 3140120kB, limit 3145728kB, failcnt 616038
[138805.608940] memory+swap: usage 6291456kB, limit 6291456kB, failcnt 2837
[138805.609043] Memory cgroup out of memory: Kill process 20611 (java) score 1068 or sacrifice child

ご覧のとおり、Javaプロセスが3145728 KB（これは約3 GB）の制限に達したため、コンテナーが停止しました。 Docker自体が4 GBの制限で起動されたため（ファイル内docker-compose）、これはかなり奇妙です。

JVMもメモリ使用量に制限を課していることをご存じでしょう。 docker自体は4 GBの制限に設定されていましたが、JVMはオプションで起動されましたXmx=3GB。これはさらに混乱する可能性がありますが、JVMは-Xmxが示したものよりも多くのメモリを使用できることに注意してください（JVMでのメモリ使用量の分析に関する記事を参照）。

何が起こっているのか理解するまで。 Dockerは4 GBの使用を許可する必要があります。では、なぜOOMkillerは3 GBで動作したのでしょうか。さらに情報を検索した結果、ハードウェアに展開されたOSのメモリには別の制限があるという事実がわかりました。

cgroups（コントロールグループ）に感謝します。 cgroupsは、プロセスグループによるリソース使用を制限、制御、およびアカウンティングするためのLinuxカーネルエンジンです。他のソリューション（チームniceまたは/etc/security/limits.conf）と比較すると、cgroupはプロセスのセット（の下）で作業できるため、より高い柔軟性を提供します。

私たちの状況では、cgroupは（を介してmemory.limit_in_bytes）メモリ使用量を3 GBに制限しました。進捗があります！

Plumbrを使用したGCメモリとイベントの調査では、ほとんどの場合、JVMが約700 MBを使用していることがわかりました。例外は、メモリ割り当てが急増した停止の直前のみでした。その後、GCの長い休止が続きました。したがって、次のことが起こったようです：

  • JVM内で実行されているJavaコードが大量のメモリを取得しようとしています。
  • 3 GBのXmx制限がまだ遠いことを確認したJVMは、オペレーティングシステムにメモリを割り当てるように要求します。
  • Dockerもチェックして、4 GBの制限に達していないことを確認します。
  • OSカーネルはcgroupの制限である3 GBをチェックし、コンテナーを強制終了します。
  • JVMはコンテナを処理する前に停止しOutOfMemoryErrorます。

これを理解して、Dockerの2.5 GBとJavaの1.5のすべての制限を構成しました。その後、JVMはOutOfMemoryErrorOutOfMemoryError例外を処理してスローできます。これにより、Plumbrは魔法を使うことができました-対応するスタックダンプでメモリスナップショットを取得し、特定の状況でデータベースのほぼ全体をロードしようとしたデータベースへのクエリが1つあったことを示しました。

調査結果


このような単純な状況でも、3つのメモリ制限がありました。

  • パラメータによるJVM -Xmx
  • ファイルのオプションによるDocker docker-compose
  • パラメータによるオペレーティングシステム memory.limit_in_bytes cgroups

したがって、OOMキラーが発生した場合は、関連するすべてのメモリ制限に注意を払う必要があります。

Docker開発者にとってのもう1つの結論。囲まれたコンテナのメモリ制限がcgroupsの制限よりも高いような「ネストされた人形」を実行することを許可しても意味がないようです。適切な警告でコンテナーを起動するときにこれを簡単にチェックすると、ユーザーのデバッグに数百時間も節約できます。
