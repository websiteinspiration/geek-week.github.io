アプリ内アップデート：Androidアプリケーションアップデートの高速化



Android Dev Summitで 発表されたさまざまなツールの中で、アプリ内アップデート（IAU）アプリケーションのアップデートメカニズムに特に注意を払います。これにより、開発者は新機能の追加、バグ修正、パフォーマンスの向上を迅速化できます。この機能はGoogle I / O 2019の後に公開されたため、この記事ではIAUについて詳しく説明し、推奨される実装スキームについて説明し、コード例をいくつか示します。 IAUをPandaoに統合した経験についても説明します。これは、中国からの商品の注文アプリケーションです。

新しいAPIにより、開発者はGoogle Playで入手可能な最新バージョンへのアプリケーションの更新を開始できます。このように、IAUは既存のGoogle Play自動更新メカニズムを補完します。IAUには、ユーザー操作の点で根本的に異なるいくつかの実装スキームが含まれています。

  1. Flexible Flowを使用すると、ユーザーはバックグラウンドで更新をダウンロードし、ユーザーにとって都合のよいときにインストールできます。これは、ユーザーが古いバージョンを引き続き使用できるが、新しいバージョンがすでに使用可能な場合を対象としています。


  2. 即時フローでは、アプリケーションを使用し続ける前に、ユーザーは更新をダウンロードしてインストールする必要があります。これは、開発者がアプリケーションを更新することが重要である場合を対象としています。



2番目のオプションはそれほど重要ではなく、Pandaoアプリケーションにはあまり適していないため、Flexible Flowシナリオをより詳細に分析します。

ユースケース


IAUを使用したアップグレードプロセスは、いくつかのステップで構成されています。

  1. Play Coreライブラリを使用するアプリケーション。利用可能なアップデートがあるかどうかをGoogle Playでチェックします。
  2. そうである場合、アプリケーションはGoogle PlayにIAUsダイアログを表示するように要求します。Google Playは、更新を提案するダイアログをユーザーに表示します。
  3. ユーザーが同意すると、バックグラウンドでGoogle Playが更新をダウンロードし、ステータスバーにダウンロードの進行状況を表示します。

    2. アプリケーションは、バックグラウンドになるまで待機してから、更新を完了します。一方では、ユーザーとアプリケーションとの対話が中断されないため、これはUXの観点から見ると、それほど煩わしくない動作です。しかしその一方で、アプリケーションがバックグラウンドにあるかどうかを判断するロジックを実装する必要があります。

ダウンロードしたアップデートのインストールが完了していない場合、Google Playはバックグラウンドでインストールを完了する場合があります。このオプションは、更新のインストールを保証しないため、明示的に使用しない方が適切です。

テストデバイスで更新プロセス全体を手動で完了するには、ソースとターゲットのアセンブリ番号が異なるアプリケーションの少なくとも2つのバージョンが必要です。

ここでは、IAUs Flexible Flowを使用するためのサンプルコードを確認します。これは、公式ドキュメントにも記載されています。最初に、モジュールレベルでbuild.gradleファイルにPlay Coreライブラリを追加する必要があります。

dependencies {
   ...
   implementation "com.google.android.play:core:1.5.0"
}

次に、インスタンスAppUpdateManagerを作成し、にコールバック関数を追加します。AppUpdateInfoこれにより、更新の可用性、更新を開始するオブジェクト（利用可能な場合）、現在のダウンロードの進行状況（既に開始されている場合）に関する情報が返されます。

// Create instance of the IAUs manager.
val appUpdateManager = AppUpdateManagerFactory.create(context)

// Add state listener to app update info task.
appUpdateManager.appUpdateInfo.addOnSuccessListener { appUpdateInfo ->
    // If there is an update available, prepare to promote it.
    if (appUpdateInfo.updateAvailability() == UpdateAvailability.UPDATE_AVAILABLE) {
        // ...
    }

    // If the process of downloading is finished, start the completion flow.
    if (appUpdateInfo.installStatus() == InstallStatus.DOWNLOADED) {
        // ...
    }
}
    .addOnFailureListener { e ->
        // Handle the error.
    }

Google Playからの更新をリクエストするダイアログを表示するには、受信したオブジェクトAppUpdateInfoをメソッドに渡す必要がありますstartIntentSenderForResult。

   appUpdateManager.startUpdateFlowForResult(
          // Pass the intent that is returned by 'getAppUpdateInfo()'.
           appUpdateInfo,
           // Or 'AppUpdateType.IMMEDIATE for immediate updates.
           AppUpdateType.FLEXIBLE,
          // The current activity.
           activity,
           REQUEST_CODE
   )

イベントリスナーをIAUsマネージャーに追加して、更新ステータスを追跡できますInstallStateUpdatedListener。

// Create a listener to track downloading state updates.
val listener = InstallStateUpdatedListener { state -> 
    // Update progress indicator, request user to approve app reload, etc.
}

// At some point before starting an update, register a listener for updates.
appUpdateManager.registerListener(listener)

// ...

// At some point when status updates are no longer needed, unregister the listener.
appUpdateManager.unregisterListener(listener)

更新がダウンロードされたら（status DOWNLOADED）、アプリケーションを再起動して更新を完了する必要があります。呼び出しappUpdateManager.completeUpdate()で再起動を開始できますが、その前にダイアログボックスを表示して、ユーザーがアプリケーションを再起動する準備ができていることを明確に確認することをお勧めします。

Snackbar.make(
    rootView,
    "An update has just been downloaded from Google Play",
    Snackbar.LENGTH_INDEFINITE
).apply {
    setAction("RELOAD") { appUpdateManager.completeUpdate() }
    show()
}

エラー「アップデートは利用できません」


最初に、「基本的な実装要件」セクションにリストされている要件への準拠を再確認します。すべてを実行してonSuccessも、呼び出しに従って更新できない場合は、キャッシングに問題がある可能性があります。内部キャッシュメカニズムが原因で、Google Playアプリケーションが利用可能な更新を認識していない可能性があります。手動テスト中にこれを回避するには、Google Playの[マイアプリとゲーム]ページに移動して、強制的にキャッシュをリセットします。または、Google Playアプリケーションの設定でキャッシュをクリアすることもできます。この問題はテスト中にのみ発生することに注意してください。キャッシュは毎日更新されるため、エンドユーザーには影響しません。

PandaoアプリでのIAUの柔軟なフロー


早期アクセスプログラムに参加し、IAUのフレキシブルフロー（推奨実装）をPandaoアプリケーションに統合しました。これは、製造業者とベンダーが中国の商品を取引できるプラットフォームです。 IAUsダイアログは、最大数のユーザーが操作できるようにメイン画面に表示されました。当初、対話を1日1回だけ表示して、人々がアプリケーションと対話するのを邪魔しないようにしたかったのです。

A / Bテストは新しい機能のライフサイクルで重要な役割を果たすため、アプリケーションでIAUの効果を評価することにしました。ユーザーをランダムに2つのグループに分けました。最初のグループはIAUを使用しないコントロールのグループで、2番目のグループはテストのグループで、これらのユーザーにIAUの対話を示しました。


PandaoアプリのIAUフレキシブルフローA / Bテスト。

過去数回のリリースで、アプリケーションの各バージョンのアクティブユーザーの割合を測定しました。現時点で最新バージョンを使用しているアクティブユーザーのうち、主要な部分はグループBの参加者、つまりIAU機能で構成されていることがわかりました。グラフの紫色の線は、バージョン1.29.1の公開後の最初の数日間で、IAUを持つアクティブユーザーの数が、この機能を持たないユーザーの数を超えたことを示しています。したがって、IAUを使用するユーザーはアプリケーションをより速く更新すると主張できます。


PandaoアプリのIAUs Flexible Flowダイアログ。

私たちのデータ（上記のグラフを参照）によると、ユーザーは、リリース後の最初の数日間、IAUsダイアログの確認ボタンをクリックすることが最も多く、その後、アプリケーションの次のバージョンが公開されるまで、変換は常に減少します。ダイアログボックスのインストールボタンでも同じことが見られ、ダウンロードしたアップデートのインストールを開始します。したがって、両方の場合の平均コンバージョン値は、リリースの頻度に正比例すると言えます。 Pandaoでは、1か月以内の平均コンバージョンは、確認ボタンをクリックすると35％、インストールボタンをクリックすると7％に達します。

新しいバージョンに関心のある人はすぐに更新され、更新に関心のない人は興味がないので、時間の経過に伴う確認の割合の減少はユーザーエクスペリエンスの問題にすぎないと想定しています。この仮定に基づいて、更新に興味のない人の邪魔をせず、毎日尋ねることはしないことにしました。「陳腐化」に基づく別のクエリロジックを使用することをお勧めします。つまり、ユーザーの邪魔にならないようにするために、ユーザーが古いバージョンをどれだけ使用しているか、および更新をすでに提案している頻度を評価しています。

一般に、IAUはA / Bテスト中に良好な結果を示したため、すべてのユーザーにIAUを展開しました。
