Angular 8バージョンの新機能

したがって、angular 8がリリースされました。これには、Ivyプレビュー、サービスワーカーのサポート、差分読み込み、およびその他のいくつかの仕上げが含まれています。Manfred Steyerは、最新リリースでの最も重要な変更について説明しています。

計画通り、驚きはありませんでした。フレームワークとCLIの更新はng updateを使用して行うことができ、新機能は「革命ではなく進化」というモットーに従って素晴らしい追加です。

この記事では、著者がAngular 8とAngular CLI 8の最も重要な新機能について説明します。この記事で使用されている例は、GitHubにあります。

アイビーを最初に見て


Angularコミュニティが待ち望んでいる次の大きなニュースは、新しいコンパイラであり、新しいレンダリングエンジンでもあるIvyです。 Ivyは非常に小さなバンドル、インクリメンタルコンパイルを生成でき、Angularの将来の革新の基礎にもなります。

Angularの基本部分の多くが変更されたため、Angularチームは以前のバージョンとの互換性に特に注意を払いました。Ivyにアップグレードした後、既存のアプリケーションは以前と同じように動作するはずです。せいぜい、はるかに小さなバンドルになります。グーグル上の600以上のアプリケーションが公式にAngularに基づいているので、これは無関心ではありません-実際の数ははるかに高いと噂されています。

Angular 8では、Ivyのプレビューがテストに利用できます。このバージョンの目的は、迅速なフィードバックを得ることです。したがって、角度のあるチームは、現時点では製品にIvyを使用しないことをお勧めしますが、クラシックビューエンジン（図1）を引き続き使用することをお勧めします


。

GoogleのAngularチームのCTOであるBrad Greenによると、ngconf 2019で、Ivyは、差分ロードと組み合わせた互換モードでパッケージサイズを大幅に改善する予定です。したがって、命知らずはすでに将来のIvy APIをテストできます。特に、このモードには最適化の大きな可能性があります。APIはまだ非公開としてマークされています。そのクラスと関数を見ると、あなたは言うことができます：それらは特殊文字withで始まります。

Ivyをすでに試したい場合は、enable-ivyスイッチを使用して新しいプロジェクトを作成できます。
ng new ivy-project --enable-ivy

このキーは、tsconfig.app.json構成に次のエントリを保存するようにCLIに指示します。
"angularCompilerOptions": { 
        "enableIvy": true
}

このエントリは、バージョン8にアップグレードした後に手動で追加して、Ivyを使用して既存のアプリケーションをテストすることもできます。
アプリケーションをデバッグモードで実行するには、AOTを使用することをお勧めします。
ng serve --aot

また、ng buildを使用して作成したアプリケーションのサイズにも注意する必要があります。角度が9の場合、Ivyはデフォルトでアクティブになります。それまで、Angularチームは、古いバージョンとの互換性を確保するための作業を継続する予定です。

Webワーカー


JavaScriptは、定義によりシングルスレッドです。このため、データのクエリなどの時間のかかるタスクは、通常、非同期で実行されます。言うまでもなく、これは複雑な計算には役立ちません。特に広範なJavaScriptソリューションで一般的になってきているため、ほとんどすべてのWebブラウザーでWebワーカーをサポートしています。これらは、ブラウザによって別のスレッドで起動されるスクリプトです。ブラウザタブのストリームとの通信は、メッセージを介して行われます。
WebワーカーはAngular自体には関係ありませんが、構築する際には考慮に入れる必要があります。目標は、Webワーカーごとに1つのパッケージを提供することです。このタスクは、新しいCLIによって完了しました。
新しい機能を示すために、いわゆるN クイーン問題の JavaScript実装を示します） "。アイデアは、お互いを脅かすことなく、チェス盤に1つのクイーンを並べることです。これは、同じ行、列、または対角線に別のクイーンが存在してはならないことを意味します。

可能なすべての解を計算するアルゴリズムチェス盤は計算が複雑であると考えられていますが、通常の8行8列のチェス盤の計算はかなり高速ですが、従来のコンピューターは12 x 12ボードで限界に達しています。27x 27ボードの解決策は現在の記録です。 。
バックグラウンドでこのような計算を変換するには、まずCLIを使用してWeb worker'aを作成する必要があります。
ng generate worker n-queens

この命令は、従業員だけでなく、ビルドプロセスに必要な構成ファイルと既存のファイルのエントリにもファイルを作成します。同じフォルダーに共通の.component.tsファイル拡張子を持つ同じ名前のコンポーネントが含まれている場合、CLIはWebワーカーと対話するためのコードも追加します。
ワーカー自体は、イベントのリスナーで構成されています。
import nQueens from './n-queens';
 
addEventListener('message', ({ data }) => {
  const result = nQueens(data.count);
  postMessage(result, undefined);
});

このイベントは、メインスレッドがワーカーにメッセージを送信したときに実行されます。パラメータには、メインストリームから送信された情報が含まれます。この場合、チェス盤のサイズを設定するcountプロパティによって制限されます。ここでは省略されているnQueens関数を評価した後、eventListenerはpostMessageを通じて結果をメインスレッドに送り返します。したがって、ブラウザはメッセージイベントを発生させます。
Workerクラスは、Workingスクリプトと対話するためにusingコンポーネントで使用されます。
const count = parseInt(this.count, 10);
 
const worker = new Worker('../logic/n-queens.worker', {
    type: 'module' // Worker uses EcmaScript modules
});
 
worker.postMessage({count});
 
worker.addEventListener('message', (event) => {
  // tslint:disable-next-line: no-console
  console.debug('worker result', event.data);
 
  // Update chessboard
  this.processResult(event.data);
});

コンポーネントは、postMessageを介して、チェス盤の希望するサイズのメッセージをワーカーに送信し、そこで計算を開始します。メッセージイベントを通じて結果を受け取ります。
将来的には、CLIがワーカースクリプトの正しいアセンブリを処理します。 TypeScriptコンパイラは、ng generate workerコマンドによって作成されたtsconfig.worker.jsonに登録されている.worker.tsの末尾でそれらを認識します。メインアプリケーションのビルド時にCLIがこれらのファイルに再び影響しないようにするために、ng generate workerは同じファイルテンプレートをtsconfig.app.jsonのexcludeセクションに配置します。
完全な実装は、作者の例とともにプロジェクトに含まれています。比較のために、N queensタスクの例は、メインスレッドとWebワーカーの両方で解決できます。たとえば、12 x 12のチェスボードの問題を解決しようとすると、最初のケースではUIがハングしますが、Webワーカーでのバックグラウンド計算ではパフォーマンスが低下しません。

差圧


これまで、このバージョンの「父親のJavaScript」はほとんどどこでも機能するため、アプリケーションを古き良きES 5にコンパイルするのが通例でした。つまり、IE11とGoogle検索エンジンのWebクローラーの両方がこのコードを実行できます。
ただし、新しいES 2015以降のバージョンはより効率的です。これらを使用すると、よりコンパクトなパッケージを作成でき、ブラウザーでパッケージをより効率的に解釈することもできます。以前は、ES 5に最小公分母としてロールバックすることが慣習であったため、残念ながら、最新のブラウザーは新しいバージョンの言語を利用できませんでした。
これで終わりです。バージョン8以降、CLIには差分ロードと呼ばれる機能があります。アイデアは、2つのパッケージグループを提供することです。1つはECMAScript 5に基づいており、古いブラウザー用に設計され、もう1つはECMAScript 2015などの新しいバージョンのECMAScriptに基づいており、前述の利点を最新のブラウザーに提供します。
差分読み込みを有効にするために多くの作業を行う必要はありません。ECMAScriptのサポートされているバージョンの上限と下限を設定するだけです。上限は、tsconfig.jsonで次のように示されます。
"target": "es2015"

下限はbrowserslistファイルで定義されています。このファイルには、たとえば市場シェアなどの特定の基準に従ってサポートされるブラウザーが含まれています。たとえば、新しいプロジェクトを作成するときにCLIがプロジェクトのルートに作成するbrowserslistファイルに保存できます 。この場合、ブラウザーのリストには、IE 9-11エントリを持つES 5ブラウザーが含まれます。したがって、CLIは下限しきい値をこのバージョンとして定義します。 CLIがng buildコマンドを受信すると、両方のバージョンでビルドプロセスが実行されます。 このプロセスの欠点は、ビルドに必要な時間が2倍になることです。

ブラウザは、ダウンロードするパッケージのバージョンを決定できるようになりました。これを行うには、index.htmlアドオンのスクリプトへのリンクを取得します。ECMAScript5パッケージを指すユーザーには、nomoduleが追加されます。したがって、ECMAScriptをサポートしているブラウザー、したがってECMAScript 2015+をサポートしているブラウザーは、このスクリプトを無視しません。一方、ECMAScript 2015+パッケージは、type =” module”のCLIによって実装されます。したがって、古いブラウザはこれらのスクリプトを無視します：
<script src="main-es2015.js" type="module"></script>

<script src="main-es5.js" nomodule></script>

ng buildとは異なり、他のCLIコマンドは（！）ESサポートの上限のみを使用します。私たちの場合、これはECMAScript 2015です。これは、特に効率上の理由で発生します。通常、デバッグとテスト中、開発者は2番目のアセンブリを待たずに、できるだけ早く結果を確認したいと考えています。

遅延読み込みモジュール


Angularルーターは最初の日から遅延読み込みをサポートしています。これまで、これはロード可能なモジュールの魔法の定義で達成されていました：
{
    path: 'lazy',
    loadChildren: () => './lazy/lazy.module#LayzModule'
}

＃の前の値は、モジュール実装ファイルへのパスを示しています。後の値は、それに含まれるクラスを意味します。この記述スタイルはAngular 8で機能しますが、ECMAScript動的インポートに関連して廃止されました：
{
    path: 'lazy',
    loadChildren: () => import('./lazy/lazy.module').then(m => m.LazyModule)
}

新しいレコードオプションには、ファイル名がマジック値として含まれています。ただし、インポートは多くのIDEでサポートされているため、無効な値はすぐにエラーを返します。

ViewChildおよびContentChildの重要な変更


ViewChildとContentChildの使用には重大な変更がありますが、残念ながら、以前は常に予測どおりに機能するとは限りませんでした。以前のバージョンでは、ngIfやngForなどの構造体ディレクティブ内にない要素をクエリするためにコンポーネントによって使用された場合、クエリ結果はngOnInitですでに利用可能でした。それ以外の場合は、ngAfterViewInit（またはContentChildの場合はngAfterContentInit）よりも早くアクセスできません。データバインディングのために後でDOMにロードされた要素の場合、プログラムコードにはngAfterViewChecked、またはngAfterContentCheckedが必要です。
この動作は紛らわしいため、コンポーネントは解決が発生するタイミングを示す必要があります。
@ViewChild('info', { static: false })
  paragraph: ElementRef;

staticフラグがtrueの場合、Angularはコンポーネントが初期化されるときに要素を見つけようとします。これが機能するのは、構造指令に含まれていない場合のみです。static：falseを使用すると、ビューの初期化または更新後に解決が実行されます。
ng updateは自動的に正しい値を入力しようとします。これが不可能な場合は、TODOでコメントを追加します。
この変更は、ViewChildrenおよびContentChildrenデコレータを使用したクエリには影響しません。彼らは常に静的な意味での新しい意味での動的な振る舞いを持っていました：false。

新機能ngUpgrade


これまでのところ、AngularJS 1.XとAngularがngUpgradeと混同していた問題の1つは、URLが原因で両方のフレームワークのルーターが競合していたことです。これにより、副作用の説明が困難になりました。これを回避するために、両方のバージョンで単一のURLロケーションサービスを使用する機能が追加されました。
これを行うために、AngularチームはAngular位置情報サービスの機能を拡張し、AngularJSの$位置の代わりを提供しました。
このため、ロケーションサービスに新しいonUrlChangeメソッドが追加され、URLの変更を追跡します。
export class AppComponent {
  constructor(loc: Location, pLoc: PlatformLocation) {
    loc.onUrlChange((url) => console.debug('url change', url));
    console.debug('hostname: ', pLoc.hostname);
  }
}

PlatformLocationサービスは、URLの特定の部分への追加アクセスを提供します。フレームワークをより適切に統合するために、それに基づく$ロケーションの置き換えがどのように使用されるかの詳細な説明は、こちらにあります。さらに、前述の動的ECMAScriptインポートに基づくAngularJS遅延読み込みソリューションを見つけることができます。

結論


再び、Angularチームは彼らの言葉を守りました：Angularの新しいバージョンへの移行は簡単で、大きな変更は含まれていません。それどころか、いくつかのコーナーは滑らかになり、GoogleのSPAフレームワークでの作業がさらに快適になりました。古いブラウザがサポートされていないか、別のパッケージでサポートされている場合、差分読み込みはパッケージサイズをさらに最適化する機会を提供します。Webワーカーのサポートは、計算集中型のタスクがブラウザーで処理する方法を見つけることを示しています。愛好家は、Ivyで最初のステップを実行できるようになりました。
