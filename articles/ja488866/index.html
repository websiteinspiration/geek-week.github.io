Gitlabのセキュリティチェック要件


次の手順を完了するには、次のものが必要です。

  • Gitlabアカウント。
  • プラットフォームSysdig the Secure DevOpsのアカウント。

Sysdig Secureを使用した画像検査のためのGitlab CI / CDパイプラインの構成


これら2つのプラットフォームを統合する方法を実際に見ていきます。このガイドでは、Sysdigセキュアイメージスキャンの操作方法を数分で学習します。

アクセス資格情報を構成する


Gitlab CI / CDパイプラインは、Sysdig Secureバックエンドと通信するためにアクセス資格情報を必要とします。すべてを正しく行うために、Sysdig UIの[設定]> [ユーザープロファイル]からアクセストークンをコピーします。

次に、Gitlabプロジェクトの新しいグローバル変数を設定します。

  1. プロジェクトで、[設定]> [CI / CD]に移動し、[ 変数]を選択します。
  2. 新しい変数SYSDIG_SECURE_TOKENを作成し、値フィールドにSysdig Secure APIキーを追加します。
  3. マスクされたボタンをクリックして、APIトークンがパイプラインログに出力されないようにします。

Gitlabパイプラインの定義


まず、収集するイメージを定義するDockerファイルが必要です。プロジェクトに任意のDockerファイルをアップロードするか、次の例を使用できます。

FROM debian:stretch
RUN apt update && apt install python-pip python-numpy openssh-server -y && rm -rf /var/lib/apt
RUN pip install flask
COPY app.py /app.py
EXPOSE 5000 22
ENTRYPOINT ["python", "./app.py"]

次に.gitlab-ci.yml、プロジェクトのルートに新しいファイルを作成する必要があります。このファイルには、環境の構築に必要なすべてのステップ（ステージ）が記述されています。彼はすべての必要な段階を決定します。

パイプラインには3つの段階があります。

  • コンテナイメージのアセンブリ。
  • 脆弱性またはポリシー違反がないか画像をスキャンします。
  • イメージを宛先リポジトリに移動します。

variables:
  CI_REGISTRY_IMAGE: "sysdiglabs/dummy-vuln-app"
  SCAN_IMAGE_NAME: "sysdiglabs/secure_inline_scan:latest"
  SYSDIG_SECURE_ENDPOINT: "https://secure.sysdig.com"

docker-build-master:
  image: docker:latest
  stage: build
  services:
    - docker:dind
  before_script:
    - docker login -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD" "$CI_REGISTRY"
  script:
    - docker build --pull -t "$CI_REGISTRY_IMAGE" .
    - docker run --rm -v /var/run/docker.sock:/var/run/docker.sock "$SCAN_IMAGE_NAME" /bin/inline_scan analyze -s "$SYSDIG_SECURE_ENDPOINT" -k "$SYSDIG_SECURE_TOKEN" "$CI_REGISTRY_IMAGE"
    - docker push "$CI_REGISTRY_IMAGE"
  only:
    - master

コンテナイメージを作成する


画像の組み立てはで行われdocker build --pull -t "$CI_REGISTRY_NAME"ます。Dockerファイルから指示を受け取り、次のステップでチェックされる新しいローカルイメージを生成します。

脆弱性のスキャン


次の段階はコンテナのチェックです。ここでは、イメージの脆弱性をチェックし、構成をチェックして、結果をSysdigバックエンドに保存します。

Sysdigローカルスキャンの利点の1つは、スキャンを実行するためにイメージを別のレジストリに移動したり、外部から公開したりする必要がないため、イメージを制御できなくなることです。これはランナー内で行われ、結果のみがSysdig Secureに送信されます。

docker run --rm -v /var/run/docker.sock:/var/run/docker.sock "$SCAN_IMAGE_NAME" /bin/inline_scan analyze -s "$SYSDIG_SECURE_ENDPOINT" -k "$SYSDIG_SECURE_TOKEN" "$CI_REGISTRY_IMAGE"

この時点で、Sysdig Secureは、イメージがポリシーで指定された停止条件（重大な脆弱性など）のいずれかを満たしている場合、エラーコードを返します。パイプラインを停止すると、脆弱な画像がコンテナレジストリに転送されなくなります。





Sysdig Secureでは、追加情報を見て、セキュリティチェックが完了しなかった理由を理解



できます。スクリーンショットでわかるように、コンテナーはポート22を開いたままにします。これは禁止ポートのリストにあり、Pythonライブラリにいくつかの重大な脆弱性も含まれています。

イメージを宛先リポジトリに移動します


脆弱性のチェックに成功した後、パイプラインがドッキングプッシュされ、画像がコンテナレジストリに公開されます。資格情報またはリモートストレージを指定しない場合、Gitlabはデフォルトのを使用します。

結論


Sysdig Secureのイメージスキャン を使用すると、Gitlab CI / CDパイプラインでイメージをインフラストラクチャからパブリックまたは中間レジストリに送信せずにチェックし、構成をチェックして、本番環境への脆弱性の漏洩を防ぐことができます。

既知の脆弱性を見つけて、Dockerfileの手順、ベースイメージにインストールされているパッケージまたはサードパーティライブラリのホワイトリストとブラックリスト（JavaのJAR / WARファイル、Javascriptのnpmなどの言語パッケージマネージャーなど）を含む、セキュリティの推奨事項に準拠しているかどうか構成を確認します。 Pythonではpip、Rubyではgem。

障害が発生した場合、これをコンテナの作成者にすばやく報告して、問題をすばやく解決し、デフォルトで保護されたコンテナのセキュリティポリシーを作成できます。 Sysdig Secureを試すには、今日デモ！
