データ移行システムの比較と選択




開発プロセスのデータモデルには変更するプロパティがあり、ある時点でデータベースに対応しなくなります。もちろん、データベースを削除すると、ORMはモデルに対応する新しいバージョンを作成しますが、そのような手順を実行すると、既存のデータが失われます。したがって、移行システムの機能は、スキームを変更した結果、既存のデータを失うことなく、それをアプリケーションのデータモデルと確実に同期させることです。

この記事では、データベースの移行を管理するためのさまざまなツールについて検討します。このレビューがこの選択に直面している開発者に役立つことを願っています。

仕事


当社は現在、次世代製品であるDocs Security Suite（DSS）の開発を積極的に行っています。サーバー部分は.Net Coreで記述されており、Entity Framework Coreはそれに応じてDBMSとして使用されます。アプリケーションを設計するときは、Code Firstアプローチを使用します。

アプリケーションのドメインモデルは、同時に複数の開発者によって作成されます-それぞれがシステムの独自の論理部分を担当します。

前世代のDSSでは、従来のEntity Framework Migrations（EF 6）が移行管理システムとして使用されていました。しかし、いくつかの主張はそれに対して蓄積されています。その主なものは、EFではバージョンの競合を解決するための健全なアプローチがないというものでした。この事実は、サポートのフレームワーク内でバグを修正するときに私たちをまだ動揺させるので、代替オプションを検討することが決定されました。

議論の結果、移行管理システムに関する次の要件が形成されました。

  1. さまざまなDBMSのサポート。必須のMS SQL Server、PostgreSQL、Oracle。ただし、他の
  2. ORMを使用します。当初、EF Coreの使用が想定されていましたが、設計段階では、他のORMを検討する準備ができていました
  3. マイグレーションの自動生成。コードファーストの開発を考えると、「ペンでペイントする」移行の必要性を避けたい
  4. バージョンの競合。マージを伴う分散開発環境では、EF Coreが衝突してクラッシュする可能性があります。アプリケーションの異なる部分は異なる開発者によって作成されるため、これは重大な問題となり、それぞれに多くの時間を費やす必要があります
  5. 高度なドキュメントとサポート。ここでは、私たちには、説明は必要ありません
  6. 無料です。条件付きの基準。非常に高価なシステムや高価ではありませんが、便宜上理想的であるため、検討する準備も整いました。

小規模な調査の結果、次のオプションが見つかり、検討に望ましいと考えられました。

  1. EFコアの移行
  2. Dbup
  3. RoundhousE
  4. ThinkingHome.Migrator
  5. 流暢な移行者

そしてもう少し



EntityFramework Core Migrations

当然、これは選択のための最初の主要なオプションでした。タンバリンで踊ることなく箱から出して動作するネイティブツール。大量のドキュメント、公式であり、それほど単純ではないなど。ただし、クラシックEFに提示された主張はEFコアに非常に関連しています。

したがって、EF Coreの利点が強調されています。

  • マイクロソフトのサポート、ロシア語を含むドキュメント、巨大なコミュニティ
  • CodeFirstベースの自動移行
  • EF 6と比較すると、データベーススナップショットはEF Coreに保存されなくなりました。コードファーストでEF Coreを使用する場合、データベースを展開する必要はありません
  • 私たちはCode Firstから踊っているので、必要なすべてのデータアクセスプロバイダーに1回の移行を実行することが可能です
  • プロバイダーについて-PostgreSQL、Oracleなど、さらに-MS SQL Serverもサポートされています

同様に短所：

  • 紛争解決は同じレベルにとどまった。一連の移行を構築し、データベースイメージを更新する必要があります。
  • 生成されるマイグレーションに基づくモデルへの依存

DbUp


dbup.github.io

DbUpはNuGetによってインストールされる.NETライブラリであり、SQL Serverへの変更のロールを支援します。どの変更スクリプトがすでに実行されたかを追跡し、データベースの更新に必要なスクリプトを起動します。ライブラリはASP.NETオープンソースブログエンジンプロジェクトから生まれ、MITライセンスで存在し、コードはGitHubにあります。移行はT-SQLを使用して説明されています。

利点は何ですか：

  • 多数のDBMSのサポート（MS SQL Server、PstgreSQL、MySQL）
  • スクリプトはT-SQLで記述されているため、非常にシンプルに見えます
  • 競合もSQLを使用して解決されます

短所：

  • サポートされているさまざまなDBMSのすべてに、Oracleは含まれていません。
  • ORMと相互作用しない
  • ペンでT-SQLスクリプトを書くことは、私たちが目指していたものではありません
  • ドキュメンテーションとコミュニティはまあまあですが、SQLスクリプトを書くという状況では必要ないかもしれません。

RoundhousE


github.com/chucknorris/roundhouse

この移行管理ツールは、前のライセンスと同様に、Apache 2.0ライセンスで配布され、T-SQL移行エンジンで実行されます。どうやら、開発者は快適な開発プロセスを作成するのではなく、DBMSサポートに関する技術的な問題を解決することに集中しました。

長所：

  • 必要なDBMSをサポート（Oracleを含む）

マイナス：

  • Oracle（およびAccessに関係のないAccess）は.NET Coreではサポートされず、.NET Full Frameworkでのみサポートされます
  • ORMでは機能しません
  • 以前のツールよりもドキュメントが少ない
  • 繰り返しますが、移行はスクリプトで書かれています





データベーススキーマを.NET Coreプラットフォームにバージョン付きで移行するためのThinkingHome.Migratorツール。MITライセンスで配布されています。開発者自身がほぼ1年前に最新バージョンについて書きました。

長所：

  • .NET Coreでシャープ
  • 移行の分岐シーケンスを実装
  • 移行ログを実装

マイナス：

  • 最終更新-1年前。どうやら、プロジェクトはサポートされていません。
  • Oracleでサポートされていません（この記事は、.NET Coreの安定した実装がないためであると述べていますが、これは1年前です）
  • マイグレーションの自動生成がありません

一般的に、このプロジェクトは、特に発展する可能性がある場合には有望であるように見えますが、ここで決定を下す必要がありました。

Fluent Migrator


github.com/fluentmigrator/fluentmigrator

多数のファンを持つ最も人気のある移行ツール。Apache 2.0ライセンスで配布されています。説明で述べたように、これは.NETの移行プラットフォームであり、Ruby on Railsの移行に似ています。データベーススキーマへの変更は、C＃のクラスで説明されています。

プラスがあります：

  • 必要なDBMSのサポート
  • .NET Coreサポート
  • 大規模な開発コミュニティ
  • 移行の競合は順次解決されます-移行の実行順序が示されます。さらに、1つのエンティティの周りに競合がある場合、コードをマージすると、そのソリューションは他のコードと同じ方法で実行されます
  • 移行が成功した後に実行されるプロファイルがあります。最後の更新は1か月前でした。つまり、プロジェクトは

短所については、こちら：

  • マイグレーションの自動生成がありません
  • EFモデルとの接続なし
  • データベーススナップショットなし

私たちの選択は何でしたか？




最も激しく議論されたのは、移行の自動生成と正気な競合解決の2つのパラメーターでした。他の要因ははるかに少ない怖がっています。その結果、ディスカッションの結果によると、チームはFluent Migratorを新しいプロジェクトで使用することを決定しました。将来の紛争の解決のためにはるかに多くの利点をもたらすでしょう。

調査結果


もちろん、完璧なツールはありません。したがって、選択肢として「ウィッシュリスト」に優先順位を付ける必要がありました。ただし、他のチームや他のタスクでは、他の要因が決定的な要素となる場合があります。この記事があなたの選択に役立つことを願っています。
