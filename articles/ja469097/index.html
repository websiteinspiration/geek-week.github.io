php7、node.js、redisを備えたCentOS 8 Webサーバー

序文


CentOSオペレーティングシステムの新しいバージョン、つまりCentOS 8がリリースされてから2日が経過しました。これまでのところ、インターネットで何かが行われる方法に関する記事はほとんどないため、このギャップを埋めることにしました。さらに、このプログラムのペアをインストールする方法だけでなく、ディスクのパーティション分割などの一般的なタスクのために、現代の仮想環境にLinuxをインストールする方法についても説明します。

ただし、最初に、以前のバージョンからこのバージョンに切り替える必要がある理由について簡単に説明します。これには2つの理由があります。

  1. php7！CentOSの以前のバージョンでは、「Orthodox」php5.4がインストールされていました...

    さて、もう少し深刻な場合、多くのパッケージがいくつかのバージョンを一括して飛び越えました。私たち（redhatのようなOSのファン）がついに参入しました。未来ではないにしても、少なくとも現在はそうです。そして、Ubuntuのサポーターはもはや私たちを笑わせたり、私たちに指を突いたりしなくなります...少なくともしばらくの間;）。
  2. yumからdnfへの移行。主な違いは、いくつかのバージョンのパッケージでの動作が正式にサポートされていることです。ここの図8では、これはまだ私にはあまり役に立ちませんが、有望に聞こえます。

仮想マシンを作成する


ハイパーバイザーは異なり、特定の読者のために読者を研ぐ目的はありません。一般的な原則についてお話しします。

記憶


最初に...確かに7から始まるCentOSシステムをインストールしますが、私の意見では6でもそうです（「しかし、正確ではありません」）、少なくとも 2 GBのRAMが必要です。したがって、私はあなたに多くを始め、与えることをお勧めします。

ただし、その場合、インストール後にメモリ容量を削減できます。1 GBで、ベアシステムは非常にうまく機能します。

ディスク


通常のインストールでは、20〜30 GBの仮想ディスクを作成します。これはシステムにとって十分です。そして、データ用の2番目のディスク。仮想マシンの作成段階でも追加後でも追加できます。通常は後で追加します。

CPU


1つのコアでは、ベアシステムはスローダウンしません。また、リソースは自由にスケーラブルであるため、インストール段階でリソースを増やしても意味がありません（要件を十分に理解していて、コンフィギュレータに再び入るのが面倒でない限り）

。残りは通常、デフォルトのままにしておくことができます。

インストール自体


ですから...インストーラーを起動します...個人的に、私はこれらのサービスを仮想マシンの形でのみ長い間置いてきたので、USBフラッシュドライブ上のすべての種類の配布レコードについては説明しません。好きなハイパーバイザーにISOをCDとしてマウントし、ダウンロードして実行しました。

基本的なインストールは非常に典型的であり、いくつかの点のみに焦点を当てます。

ソースの選択


8番目のバージョンのリリース以降、Yandexのミラーは1日中横になっています。まあ、つまり、それは定期的に上昇し、その後再びエラーを示し始めます。問題はサービスへの過度の負荷であると確信しています。したがって、ソースを示すために、私は通常のアドレスを入力する代わりに、ここに行き、好きなミラーを選択し、インストーラーウィンドウでアドレスを手動で入力する必要がありました。repodataディレクトリがあるフォルダへのパスを指定する必要があることを覚えておくことが重要です。たとえば、mirror.corbina.net / pub / Linux / centos / 8 / BaseOS / x86_64 / osです。

ディスクの故障


この質問は私の考えではかなり宗教的です。各管理者は、このテーマに関して独自の立場を持っています。しかし、私はまだ質問に対する私の見解を共有します。

はい、原則として、場所全体をルートに割り当てて作業できます。ほとんどの場合、それは非常に優れています。では、なぜ異なるセクションのある庭にフェンスを張るのでしょうか。 -私の意見では、これの主な理由：割り当てと移植性。

たとえば、何か問題が発生し、メインデータセクションでエラーが発生した場合は、とにかくシステムをロードして、蘇生対策を実行できるようにしたいと考えています。したがって、/ bootの下に個別のパーティションを割り当てます。カーネルとブートローダーがあります。通常は500メガバイトで十分ですが、まれにそれ以上かかることもあります。テラバイトでスペースを測定することに慣れているため、このセクションには2GBを割り当てます。そして、ここでは、lvmを実行できないことが重要です。

次は、システムのルートです。通常のインストールでは、システムごとに4 GBを超える必要はありませんでしたが、スケジュールされたイベント中に/ tmpディレクトリを使用してディストリビューションを解凍し、それを別のセクションとして強調表示するポイントを確認しません-最近のシステムでは自動的にクリーンアップされるため、いっぱいになりません。ルートの下に8GBを割り当てます。

スワップ...概して、それによる実用的なメリットはほとんどありません。サーバーでスワップの使用を開始した場合、今日の現実の世界では、サーバーがRAMを追加する必要があるだけです。それ以外の場合は、パフォーマンスの問題が保証されます（または一部のプログラムでメモリリークが発生します）。したがって、このセクションは診断のみを目的としています。したがって、2 GBは大きな数値です。はい、サーバーのメモリ量に関係なく。はい、私はそれがスワップボリュームに対するメモリの比率について書かれているすべての記事を読みました...私見、それらは時代遅れです。 10年間の実践において、これは私にとって決して有用ではありませんでした。 15年前に使用しました。

誰もが別のセクションの私見で/ホームを割り当てるかどうかを決めることができます。誰かがサーバー上のこのディレクトリを積極的に使用している場合は、強調することをお勧めします。誰もいない場合-理由はありません。

次に、/ var。私の意見では、それを選び出す必要があります。手始めに、あなたは自分を4GBの数字に制限し、それがどうなるかを決めることができます。「はい」とは、つまり

  1. まず、いつでも/ varサブディレクトリに別のディスクをマウントできます（これについては後で例を示します）。
  2. 次に、lvmがあります。いつでも追加できます。そして、通常は、そこにあまりにも多くのログが流れ始めたときに追加する必要があります。しかし、この数値を事前に予測することはできなかったので、2 GBから始めて、次に調べます。

未割り当て領域はボリュームグループ内で空きのままになり、いつでも使用できます。

LVM


LVMで/ブート以外のすべてのパーティションを作成することは理にかなっています。はい、スワップを含みます。はい、すべてのヒントの交換はディスクの最初に行う必要があります。LVMの場合、その場所は原則として特定できません。しかし、私が上で書いたように、あなたのシステムはまったくスワップを使用すべきではありません。したがって、彼がどこにいるかは関係ありません。まあ、95年ではなく、私たちは正直に生きています！

さらに、LVMには、一緒に生活できる必要のあるいくつかの基本的なエンティティがあります。

  • 物理ボリューム
  • ボリュームグループ
  • 論理ボリューム

物理ボリュームはグループ化され、各物理ボリュームは1つのグループにのみ含まれ、グループは一度に複数の物理ボリュームに存在できます。
また、論理ボリュームはそれぞれ同じグループにあります。

しかし...私たちは、いまいましいところ、再び21世紀の庭にいます。そして、サーバーは仮想です。物理的なメカニズムに適用されたのと同じメカニズムをそれらに適用しても意味がありません。そして仮想の場合、データをシステムから分離することが重要です！これは特に、データを別の仮想マシンにすばやく切り替える機能（たとえば、新しいOSに切り替える場合）および一般的にあらゆる種類の便利な機能（たとえば、ハイパーバイザーを使用するセクションごとの個別のバックアップ）にとって非常に重要です。したがって、1つのボリュームグループがシステムに使用され、必ず別のボリュームグループがデータに使用されます。この論理的な分離は、人生の多くを助けます！

仮想マシンの作成時に仮想ハードディスクを1つだけ作成した場合、これで構成は終了です。そして、もし2つの場合は、2番目をまだマークしないでください。

sudo -i

dnf -y update

残りのディスクを構成します


システムにxvdaパーティションがあるとすると、データディスクはxvdbになります。 OK。

ほとんどのヒントは、「fdiskを実行してパーティションを作成...」という言葉から始まります。

そのため、これは正しくありません。

パンケーキをもう一度繰り返すつもりです。この場合、仮想ディスクであっても全体を占めるLVMで作業するには、その上にパーティションを作成するのは危険です。このフレーズでは、すべての単語が重要です。 LVMなしで作業する場合-必要です。ディスクシステムとデータについて言えば-それは必要です。何らかの理由でディスクの半分を空のままにする必要がある場合-必要です。しかし、通常これらすべての仮定は純粋に理論的なものです。既存のセクションに場所を追加する場合は、この構成でこれを行うのが最も簡単です。そして、管理のしやすさは、この構成に意図的に移行するほどに勝っています。

そして、便利なのは、データセクションを拡張する場合は、仮想セクションにスペースを追加し、vgextendを使用してグループを拡張するだけです。まれに、別のものが必要になることもありますが、少なくとも最初から論理的なものを拡張する必要はありません。次に、このボリューム自体を拡張するには、最初に既存のボリュームを削除してから、新しいボリュームを上に作成することをお勧めします。見栄えがよくなく、ライブで実行することもできません。指定したシナリオに従って拡張を実行すると、パーティションをアンマウントすることさえありません。

全体として、物理ボリュームを作成し、次にそれを含むボリュームのグループを作成してから、サーバーのセクションを作成します。

pvcreate /dev/xvdb
vgcreate data /dev/xvdb
lvcreate -n www -L40G data
mke2fs -t ext4 /dev/mapper/data-www

ここでは、大文字の「L」（およびGBでのサイズ）の代わりに、小さいサイズを指定して、絶対サイズではなく相対サイズを指定できます。たとえば、ボリュームグループで現在使用可能なスペースの半分を使用するには、「-l + 50％FREE」を指定する必要があります。

最後このコマンドは、パーティションをext4ファイルシステムでフォーマットします（これまでのところ、私の慣例では、すべてが壊れた場合の最大の安定性を示しているので、私はそれを好みます）。

次に、パーティションを正しい場所にマウントします。これを行うには、/ etc / fstabに正しい行を追加します。

/dev/mapper/data-www    /var/www                ext4    defaults        1 2

そして、 

mount /var/www

エラーがポップアップした場合-アラームを鳴らします！ / etc / fstabにエラーがあることを意味します。そして、次の再起動時に非常に大きな問題が発生します。システムがまったく起動しない場合があります。これは、多くの場合、クラウドサービスにとって非常に悲しいことです。したがって、最後に追加された行を緊急に修正するか、完全に削除する必要があります。そのため、手動でマウントコマンドを記述していませんでした。ダイレクトサーバーの構成を確認する絶好の機会はありませんでした。

今、私たちは実際に私たちが望むすべてを置き、ウェブの下のポートを開きます：

dnf groupinstall "Development Tools"
dnf -y install httpd @nodejs @redis php
firewall-cmd --add-service http --permanent
firewall-cmd --add-service https --permanent

必要に応じて、ここにデータベースを配置することもできますが、個人的には、Webサーバーから分離しておくようにしています。彼女を近くに保つことはより速いですが、はい。仮想ネットワークアダプターの速度は通常ギガビットリージョンにあり、同じマシンで作業している場合、アクセスはほぼ瞬時に行われます。しかし、安全性は低くなります。ここでは誰にとってより重要です。

次に、構成ファイルにパラメーターを追加します（このような新しい最新のCentOSイデオロギーを作成します）

echo "vm.overcommit_memory = 1"> /etc/sysctl.d/98-sysctl.conf

サーバーを再起動します。
コメントの中で、私はSeLinuxをオフにするように助言するように叱られたので、それを修正し、その後、SeLinuxの設定を忘れないようにする必要があるという事実について書きます。
実際には、利益！:)
