彼は何ができますか？


本質的に、Badoo Jira PHP Clientは、Jira API応答用の既製のラッパークラスのセットであり、そのほとんどはActiveRecordのように動作できます：彼らは自分自身に関するデータを取得する方法とサーバーでそれらを更新する方法を知っており、レベルでの遅延初期化とデータキャッシュをサポートしていますコード。常に処理する必要のあるすべてのJiraエンティティがラップされています：問題、ステータス、優先度、変更ログ、ユーザー、バージョン、コンポーネントなど（ほとんどすべてがインターフェースに表示されます）。

さらに、Badoo Jira PHPクライアントは、すべてのJiraカスタムフィールドに単一のクラス階層を提供し、必要な各カスタムフィールドのクラス定義を生成できます。

$Issue = new \Badoo\Jira\Issue('SMPL-1');

$Issue->addComment('Sample comment text');
$Issue->attachFile('kitten.jpeg', 'pretty!', 'image/jpeg');

if ($Issue->getStatus()->getName() === 'Open') {
   $Issue->step('In Progress');
}


$DeveloperField = new \Example\CustomFields\Developer($Issue);
$DeveloperField->setValue('username')->save();


$User = \Badoo\Jira\User::get('username');
$User->watchIssue('SMPL-1');
$User->assign('SMPL-2');

これにより、PHPからのAPIとの対話がよりシンプルで便利になり、Jiraのドキュメントがコードに直接移動するため、IDEでオートコンプリートを使用して多くの標準アクションを実行できます。

私たちが遭遇したAPI機能


Jira APIを積極的に使用し始めたとき、それはSOAPプロトコルを通じてのみ利用可能でした。そのRESTバージョンは後で登場し、私たちはその最初のユーザーの1人でした。当時、PHPで作成された公開されているRESTクライアントはほとんどありませんでした。 SOAPベースからRESTに徐々に移行して、コードベースに簡単に統合できるものを見つけるのはさらに困難でした。したがって、私たちは選択の余地がありませんでした。私たちは独自のクライアントを開発し続けることを決定しました。

そして、私たちは生きていて、あらゆる種類のハックと松葉杖をSOAPクライアントからドラッグし、RESTの特殊性のために新しいものを取得しました。その結果、重複したコードがたくさんある非常に大胆なクラスがいくつか増えましたが、この混乱を解消する必要があります。

カスタムフィールドは常に私たちにとって最も苦痛な場所でした。300を超えるフィールドがあり（この記事の執筆時点では338）、この数は徐々に増加しています。

クライアントは何で構成されていますか？


カスタムフィールドを操作するためのクラス


まず最初に、APIとやり取りするためのデータ構造の記憶を取り除き、エラーが発生した場合に読み取り可能なフィールド名を取得する必要がありました。

その結果、すべてのカスタムフィールドに対して単一のクラス階層を作成しました。それは3つの層になった：

  1. すべての人に共通の抽象的な親：\ Badoo \ Jira \ CustomFields \ CustomField。
  2. フィールドのタイプごとに抽象クラスによる：SelectField、UserField、TextFieldなど
  3. 特定のフィールドごとのクラス別（例：DeveloperまたはReviewer）。

これらのクラスは個別に作成することも、特別なスクリプトジェネレーターを使用して自動的に作成することもできます（後で説明します）。


この構造のおかげで、Select Listタイプ（複数選択）のカスタムフィールドの値を更新するようにコードに教えるには、SelectFieldから継承したPHPクラスを作成するだけで十分です。実際、すべてのカスタムJiraフィールドは、PHPコードで通常のActiveRecordに変わります。

namespace \Example\CustomFields;

class Developer extends \Badoo\Jira\CustomFields\SingleUserField
{
	const ID = 'customfield_10200';
	const NAME = 'Developer';
}

// ,  ,  !


同じクラスに、フィールドに関する情報を格納します。デフォルトでは、これはID、フィールドの名前、および制限されている場合（たとえば、チェックボックスと選択）の使用可能な値のリストです。

作成リクエストを発行する


Jiraでのタスクの作成は、かなり複雑な手順です。これをWebインターフェースから実行すると、一連のフィールドを備えた特別な画面（「画面の作成」）が表示されます。必要に応じて、一部を入力できます。一部は必須としてマークされています。同時に、Create Screenはプロジェクトごとに、また1つのプロジェクトのさまざまなタイプのタスクに対しても一意にすることができます。したがって、フィールドの値、およびタスクの作成中にフィールドの値を設定する機能には、あらゆる種類の制限があります。

この状況で開発者にとって最も不愉快なことは、これらの制限がAPIに適用されることです。後者には特別なリクエストがあります（create-metaバージョン5.0以降のREST APIで使用できます）。これにより、タスクの作成時に使用できるフィールド設定のリストを取得できます。ただし、「今すぐ簡単なことをする」必要がある開発者は、これを気にする必要はほとんどありません。

その結果、次のようになりました。タスクを作成するリクエストは非常に大きくなる可能性があるため、データを徐々に追加することが多く、すべてをJiraに送信しようとするとエラーが発生しました。その後、リクエストで何かが変更されたすべての場所のコードを調べ、正確に何がうまくいかなかったかを理解するための長くて退屈な試みをしなければなりませんでした。

そのため、\ Badoo \ Jira \ Issue \ CreateRequestを実行しました。これにより、何か間違ったことをしようとしている場所でエラーを早期に確認できます。フィールドに何らかの曲線の値を指定するか、使用できないフィールドを変更します。たとえば、プロジェクトに存在しないコンポーネントを指定しようとすると、最終的にAPIにリクエストを送信した場所ではなく、実行した場所で例外がクラッシュします。

CreateRequestでの作業の流れは次のようになります
$Request = new \Badoo\Jira\Issue\CreateRequest('DD', 'Task', $Client);
$Request
   ->setSummary('summary')
   ->setDescription('description')
   ->setFieldValue('For QA', 'custom field with some comments for QA who will check the issue');

$Request->send();


APIを直接操作する


上記のクラスのセットはほとんどのニーズをカバーします。ただし、大多数がすべてから離れていることは十分に承知しています。したがって、APIを直接操作するための小さなクライアント- \ Badoo \ Jira \ REST \ Clientもあります。

お客様の使用例
$Jira = \Badoo\Jira\REST\Client::instance();
$Jira->setJiraUrl('https://jira.example.com/');
$Jira->setAuth('user', 'password')

$IssueInfo = $Jira->issue()->get('SMPL-1');


カスタムフィールドのクラスジェネレーター


カスタムフィールドを使いやすくするには、各フィールドのコードに独自のクラスが必要です。私たちの場所では、必要に応じて手動で作成しましたが、クライアントを公開する前に、このアプローチは新規ユーザーにとってはあまり便利ではないと判断しました。したがって、Jira APIにアクセスしてカスタムフィールドのリストを取得し、既知のフィールドタイプのテンプレートクラスを作成できる特別なジェネレーターを作成しました。

ほとんどのタスクでは、リポジトリからbin / generate CLIスクリプトを使用するだけで十分だと考えています。--help / -hオプションを使用して、彼に自分自身について話すように依頼できます。

./bin/generate --help

最も単純なケースでは、生成には、JiraのURL、ユーザー、パスワード、クラスの名前空間、およびコードを配置するディレクトリを指定するだけで十分です。

./bin/generate -u user -p password --jira-url https://jira.mlan --target-dir custom-fields --namespace CustomFields

また、独自のテンプレートを追加し、個々のフィールドのクラスを生成する機能も実装しました。これはドキュメントにあります。
