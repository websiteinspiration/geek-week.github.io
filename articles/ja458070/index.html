私たちの記事の多くは何かに向けられていますが、PVS-Studioツール自体には向けられていません。しかし、開発者が私たちのツールを使いやすくするために多くのことをします。しかし、これはまさに裏でしばしば行われていることです。私はこの状況を修正し、Visual StudioのPVS-Studioプラグインについて話すことにしました。Visual Studioを使用している場合は、この記事が役立ちます。

静的コード分析とは何か、なぜそれが必要なのか


静的コード分析は、プログラムのソースコードのエラーや欠点を特定するプロセスです。静的分析は、自動化されたコードレビュープロセスと見なすことができます。協調的なコードレビューは素晴らしい方法論です。しかし、それには大きな欠点があります-高コストです。新しいコードを確認するために定期的に数人のプログラマーを集めるか、推奨を行った後にコードを再確認する必要があります。

一方では、コードを定期的に確認したいと思います。一方、それは高すぎます。トレードオフは静的コード分析ツールです。彼らは精力的にプログラムのソースコードを処理し、コードの特定のセクションにさらに注意を払うようプログラマに推奨します。もちろん、プログラムは、プログラマーのチームによって実行されたコードの完全なレビューを置き換えるものではありません。ただし、利点/価格比により、静的分析を使用することは多くの企業にとって非常に有用な方法になります。読者が特定の数値に興味がある場合は、記事「PVS-Studio ROI」を理解することをお勧めします。

商用および無料の静的コードアナライザーは多数あります。静的アナライザーの大規模なリストは、WikipediaのWebサイトで入手できます。静的コード分析のためのツールのリスト。静的コードアナライザーが存在する言語のリストも非常に大きい（C、C ++、C＃、Java、Ada、Fortran、Perl、Rubyなど）。当然、PVS-Studioアナライザーについて説明します。

静的解析の主な利点は、プログラムの欠陥を排除するコストを大幅に削減できる可能性があることです。エラーが早期に特定されるほど、エラーを修正するコストは低くなります。したがって、McConnellの著書「Perfect Code」で提供されているデータによると、テスト段階で誤りを修正することは、設計（書き込み）段階の10倍の費用がかかります。


図1.導入と検出の時間に応じた欠陥修正の平均コスト（表のデータは、S。McConnell著の本「Perfect Code」から引用）

静的分析ツールを使用すると、設計段階で多数のエラーを特定できるため、プロジェクト全体の開発コストを大幅に削減できます。たとえば、PVS-Studio静的コードアナライザーはコンパイル直後にバックグラウンドで起動でき、潜在的なエラーが見つかった場合はプログラマーに通知します。このモードの詳細については、以下で説明します。

PVS-Studio Static Code Analyzer


PVS-Studioは、C、C ++（拡張機能もサポートされているC ++ / CLIおよびC ++ / CX）、Windows、LinuxおよびmacOSプラットフォーム上のC＃およびJava のアプリケーションのソースコードのエラーと潜在的な脆弱性を検出する静的アナライザーです。アナライザーは、Visual Studio 2010-2019およびIntelliJ IDEAと完全に統合されます。この記事では、C、C ++、C＃のコードをチェックするためのPVS-Studioの作業を詳しく見ていきます。 PVS-Studioを使用してIntelliJ IDEA で Javaコードをテストする方法については、こちらをご覧ください。

PVS-StudioをインストールしてVisual Studioに統合すると、ユーザーはメインメニューと診断メッセージを操作するためのウィンドウに追加の「PVS-Studio」アイテムを受け取ります（図2を参照）。

初期設定


分析装置は、設置後すぐに使用できます。ほとんどの場合、最初の実行を完了するために何も構成する必要はありません。最初に必要となる唯一の設定は、サードパーティライブラリの除外です。ソースファイル（jpegライブラリなど）を編集することはできないため、チェックする必要はありません。さらに、不要なフォルダーを除外すると、プロジェクトの分析時間が短縮されます。分析から除外されるディレクトリは、PVS-Studio>オプション...>ファイルをチェックしない>パスマスク（図3を参照）で設定されます。


図3.アナライザーがチェックしないディレクトリーのリストの編集。

示された名前のいずれかがファイルの絶対パスにある場合、分析は実行されません。デフォルトでは、一部のディレクトリの名前はすでにリストに含まれています。ただし、プロジェクトでは、ZLibライブラリを含むフォルダーは「zlib」ではなく、「zip_lib」などと呼ばれる場合があります。したがって、このリストを編集する必要があります。編集を開始するには、3つのドットが付いたボタンをクリックします。

PathMasksリストの有効なマスクの例：

  • c：\ Libs \-このフォルダーとそのサブフォルダーにあるすべてのプロジェクトファイルは除外されます。
  • \ Libs \または* \ Libs \ *-Libsサブフォルダーを含むパスへのパスが除外されるディレクトリにあるすべてのファイル。「*」文字が指定されていない場合でも、自動的に追加されるため、両方の録音オプションは同じです。
  • Libsまたは* Libs *-すべてのファイルが除外されます。パスには、「Libs」が名前または名前のフラグメントとして含まれるサブフォルダーが含まれています。また、この場合、名前にLibsが含まれているファイル（c：\ project \ mylibs.cppなど）は除外されます。混乱を避けるため、常にスラッシュを使用することをお勧めします。

フォルダ全体を除外することに加えて、個々のファイルを除外するためにマスクを指定できます。これには、FileNameMasks設定があります。除外リストの操作方法の詳細については、ドキュメント：設定：ファイルをチェックしないをご覧ください。

図4. PVS-Studioアナライザーを使用したソリューションの検証。

コンテキストメニュー


したがって、メッセージをダブルクリックすると、目的のコードフラグメントに移動します。マウスの右ボタンを押すと、コンテキストメニューが開きます。

メニューは非常にシンプルであり、各項目の説明で記事をオーバーロードしないでください。不明な点がある場合は、ドキュメントをご覧ください。

ただし、非常に便利な機能について詳しく説明します。設定では、除外するフォルダー/ファイルを追加できます（図2を参照）。そのため、何かを追加するのは思ったよりもずっと簡単です！

「ファイルをチェックせず、すべてのメッセージを非表示にする...」という項目に注意してください。クリックすると、例外に追加できるパスのリストが表示されます（図6を参照）。


図6.ファイルをスキャンから除外する。

単一のファイルまたはディレクトリの1つを選択できます。この図は、「SDL2-2.0.9 \ src \ haptic \ windows」というフォルダーが選択されていることを示しています。つまり、このフォルダー内のすべてのファイルとすべてのサブフォルダーが分析から除外されます。さらに、これらのファイルに関連するすべてのメッセージは、リストからすぐに消えます。とても快適。テスト関連のメッセージをすべて削除するために分析を再開する必要はありません。

インクリメンタル分析モード


最も重要な機能の1つであるインクリメンタルコード分析について話さないと、PVS-Studioの紹介は不完全になります。

エラーが早期に発見されるほど、それを修正する方が安上がりです。プログラムの編集されたテキストのエラーをすぐに強調するのが理想的です。ただし、これは技術的に難しく、リソースを大量に消費します。したがって、PVS-Studioは、修正されたコードが正常にコンパイルされた瞬間にバックグラウンドで起動します。したがって、変更されたばかりのコードでエラーが検索されます。分析が行われていることは、システム通知領域のアイコンで判断できます。

エラーが検出されると、ポップアップウィンドウが表示され、危険を警告します（図7を参照）。

PVS-Studioの機能


簡単に説明しましょう。PVS-Studioで利用できるすべての診断を簡潔に説明することは不可能です。診断とその詳細な説明の完全なリストは、ドキュメント：診断されたエラーの説明にあります。診断はタイプ別にグループ化されたテーブルに限定されます。一部の診断は複数のグループに含まれています。実際には、分割は非常に恣意的です。たとえば、タイプミスにより、初期化されていないメモリが使用される可能性があります。それどころか、いくつかのエラーはテーブル内の場所を見つけることができませんでした。それにもかかわらず、表は全体として、静的コードアナライザーの機能を示しています（図8を参照）。


図8. PVS-Studioの機能。

ご覧のとおり、アナライザーは、タイプミス、コピー、貼り付けが原因で発生したエラーの検索などの領域でできるだけ多く現れます。コードのセキュリティに関連する問題を診断します。

これが実際にどのように機能するかは、エラーデータベースを調べることでわかります。このデータベースには、さまざまなオープンソースプロジェクトをチェックして見つけたすべてのエラーが収集されています。

コマンドラインからプロジェクトを確認する


PVS-Studio_Cmd.exeは、コマンドラインからC ++ / C＃Visual Studioプロジェクト（.vcxproj / .csproj）および.slnソリューションをチェックするためのユーティリティです。分析の自動化に役立ちます。プログラムは、インストールが行われたディレクトリにあります。デフォルトでは、「C：\ Program Files（x86）\ PVS-Studio」です。

プログラムには多くのパラメーターがありますが、最初に必要なのはそのうちの3つだけです。

  • --target：検証するプロジェクトまたはソリューションファイル。
  • --output：レポートを書き込むplogファイル。
  • --progress：検証の進行状況を表示します。

ローンチは次のようになります。


図12. PVS-Studio_Cmd.exeプログラムの出力

実行後、レポートを含むplogファイル（起動オプションで指定したパス）を取得します。このレポートは、PlogConverter.exeユーティリティを使用して他の形式に変換できます。IDEでレポートを表示するには、エクスプローラーでplogファイルをダブルクリックします。

また、Extensions> PVS-Studio> Open / Save> Open Analysis Report ...の拡張メニューでレポートファイルを開くこともできます。

誤った警告の抑制


アナライザーによって生成される一部のメッセージは、必然的にfalseになります。それについては何もできません。静的アナライザーは、人工知能を持たないプログラムであり、実際のエラーを検出したかどうかを正確に判別できません。

誤検知に対抗するために、アナライザーには一連の異なるメカニズムが用意されています。これらについては、ドキュメントの次のセクションで詳しく説明しています。
