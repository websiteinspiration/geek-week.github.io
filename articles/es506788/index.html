Migraciones de bases de datos con Flyway

Se preparó una traducción del artículo antes del inicio del curso de Desarrollador Java .


1. Introducción


Este artículo describe los conceptos clave de Flyway y un ejemplo del uso de este marco para cambiar continuamente el esquema de la base de datos usando la base de datos H2 en memoria como ejemplo usando el complemento flyway maven.

Flyway actualiza las versiones de la base de datos mediante migraciones. Las migraciones pueden escribirse en SQL (con una sintaxis específica para un DBMS específico) o en Java.

Las migraciones pueden ser versionadas o recurrentes. Los primeros tienen una versión única y se aplican exactamente una vez. Los segundos no tienen un número de versión y se aplican cuando cambia su suma de verificación.

Las migraciones repetidas dentro de una sola ejecución siempre se aplican después de completar las migraciones versionadas. Las migraciones repetidas se aplican en el orden en que se describen. En una migración, todas las operaciones se realizan dentro de una sola transacción de base de datos.

En este artículo, nos centraremos en usar el complemento Maven para migraciones de bases de datos.

2. Complemento Flyway Maven


Agregue el complemento flyway maven a pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
</plugin>

La versión actual del complemento se puede ver en Maven Central .
Puede encontrar una lista de parámetros de complementos en la documentación . Los parámetros del complemento se pueden configurar de cuatro maneras diferentes.

2.1. <configuration>Los
parámetros de la sección de complementos se pueden especificar directamente en la etiqueta en la sección de complementos en pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
    <configuration>
        <user>databaseUser</user>
        <password>databasePassword</password>
        <schemas>
            <schema>schemaName</schema>
        </schemas>
        ...
    </configuration>
</plugin>

2.2. Maven propiedades


También puede configurar el complemento especificando parámetros en <properties>:

<project>
    ...
    <properties>
        <flyway.user>databaseUser</flyway.user>
        <flyway.password>databasePassword</flyway.password>
        <flyway.schemas>schemaName</flyway.schemas>
        ...
    </properties>
    ...
</project>

2.3. Archivo de configuración externo


O describa la configuración en un .propertiesarchivo separado :

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=schemaName
...

El nombre predeterminado es el archivo de configuración flyway.properties. Este archivo debe estar en el mismo directorio que el archivo pom.xml. La codificación se establece en el parámetro flyway.encoding(el valor predeterminado es UTF-8).

Si usa un nombre diferente para el archivo (por ejemplo, customConfig.properties), debe especificarlo explícitamente al llamar a maven:

$ mvn -Dflyway.configFile=customConfig.properties

2.4. Propiedades del sistema


Finalmente, los parámetros se pueden especificar como propiedades del sistema cuando se llama a maven desde la línea de comandos:

$ mvn -Dflyway.user=databaseUser -Dflyway.password=databasePassword
  -Dflyway.schemas=schemaName

Si la configuración se especifica de varias maneras, la prioridad será la siguiente:

  1. Propiedades del sistema
  2. Archivo de configuración externo
  3. <propiedades de la sección>
  4. Sección de <configuración del >complemento

3. Ejemplo de migración


En esta sección, consideraremos los pasos necesarios para migrar un esquema de base de datos usando la base de datos H2 en memoria como ejemplo usando el complemento maven. Para configurar Flyway, utilizaremos un archivo externo.

3.1. POM cambios


Para comenzar, agregue una dependencia en H2:

<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>1.4.196</version>
</dependency>

Aquí también podemos consultar la última versión de controlador disponible en Maven Central . Agregue un complemento para Flyway, como se describió anteriormente.

3.2. Configurar Flyway en un archivo externo


Cree un $PROJECT_ROOTarchivo myFlywayConfig.propertiescon los siguientes contenidos:

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=app-db
flyway.url=jdbc:h2:mem:DATABASE
flyway.locations=filesystem:db/migration

La configuración anterior indica que los scripts de migración están en el directorio db/migrationy para conectarse a la base de datos utilizando H2 databaseUsery databasePassword.

Esquema de base de datos para la aplicación app-db.

Por supuesto, en los parámetros flyway.user, flyway.passwordy flyway.urldebe especificar el nombre, contraseña y URL de su base de datos.

3.3. Primera migración


Según las convenciones de Flyway, los nombres de los scripts de migración deben estar en el siguiente formato:

<Prefix<code>><</code>Version>__<code><</code>Description>.sql

Dónde:

  • <Prefijo> - prefijo. Para las migraciones versionadas, el valor predeterminado es "V". El prefijo se configura a través de la propiedad flyway.sqlMigrationPrefix.
  • <Versión>: número de versión de migración. Las versiones mayor y menor se pueden separar subrayando. La versión siempre debe comenzar con 1.
  • <Descripción>: una descripción textual de la migración. La descripción debe estar separada del número de versión por dos guiones bajos.

Ejemplo: V1_1_0__my_first_migration.sql

Entonces, creemos un directorio db/migrationen $PROJECT_ROOTla migración del script V1_0__create_employee_schema.sqly SQL para crear la tabla employee:

CREATE TABLE IF NOT EXISTS `employee` (
 
    `id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name` varchar(20),
    `email` varchar(50),
    `date_of_birth` timestamp
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;

3.4. Realizamos migraciones


A continuación, $PROJECT_ROOTejecute el siguiente comando maven para aplicar las migraciones de la base de datos:

$ mvn clean flyway:migrate -Dflyway.configFile=myFlywayConfig.properties

Nuestra primera migración debe completarse.

Ahora el esquema de la base de datos se ve así:
empleado:

+----+------+-------+---------------+
| id | name | email | date_of_birth |
+----+------+-------+---------------+

Podemos repetir los pasos anteriores para realizar otras migraciones.

3.5. Segunda migración


Para la segunda migración, cree un archivo con el nombre que V2_0_create_department_schema.sqlcontenga las siguientes dos solicitudes:

CREATE TABLE IF NOT EXISTS `department` (
 
`id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
`name` varchar(20)
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;
 
ALTER TABLE `employee` ADD `dept_id` int AFTER `email`;

Realicemos la migración, tal como lo hicimos para la primera migración.

Ahora el esquema de nuestra base de datos ha cambiado: employeese ha agregado una nueva columna y se ha creado una nueva tabla department:

employee:
+----+------+-------+---------+---------------+
| id | name | email | dept_id | date_of_birth |
+----+------+-------+---------+---------------+

department:

+----+------+
| id | name |
+----+------+

Para verificar que ambas migraciones fueron exitosas, ejecute el siguiente comando maven:

$ mvn flyway:info -Dflyway.configFile=myFlywayConfig.properties

4. Desactivar la ruta de vuelo en Spring Boot


A veces puede que necesite deshabilitar las migraciones de Flyway.

Esto puede ser necesario durante las pruebas cuando se genera un esquema de base de datos basado en entidades. En este caso, puede deshabilitar Flyway para el perfil de prueba.

En Spring Boot, esto es muy simple.

4.1. Spring boot 1.x


Todo lo que necesitamos hacer es establecer la propiedad flyway.enableden el archivo application-test.properties:

flyway.enabled=false


4.2. Spring boot 2.x


En versiones posteriores de Spring Boot, esta propiedad se ha cambiado a 

spring.flyway.enabled:
spring.flyway.enabled=false

4.3 Vía aérea vacía Estrategia de migración


Si queremos deshabilitar solo la migración automática de Flyway al inicio, pero queremos comenzar las migraciones manualmente, entonces usar las propiedades anteriores no funcionará para nosotros.

Esto se debe al hecho de que Spring Boot no configurará automáticamente los contenedores Flyway y, por lo tanto, tendremos que configurarlos nosotros mismos, lo que no es muy conveniente.

En este caso, podemos dejar Flyway habilitado e implementar una FlywayMigrationStrategy vacía :

@Configuration
public class EmptyMigrationStrategyConfig {
 
    @Bean
    public FlywayMigrationStrategy flywayMigrationStrategy() {
        return flyway -> {
            // do nothing 
        };
    }
}

De hecho, esto deshabilitará las migraciones de Flyway cuando se inicie la aplicación.

Pero aún podemos comenzar la migración manualmente:

@RunWith(SpringRunner.class)
@SpringBootTest
public class ManualFlywayMigrationIntegrationTest {
 
    @Autowired
    private Flyway flyway;
 
    @Test
    public void skipAutomaticAndTriggerManualFlywayMigration() {
        flyway.migrate();
    }
}

5. Cómo funciona Flyway


Para rastrear cuándo, por quién y qué migraciones se aplicaron, se agrega una tabla especial con metadatos al esquema de la base de datos. Esta tabla también almacena las sumas de verificación de las migraciones e información sobre si la migración fue exitosa o no.

El marco funciona de la siguiente manera:

  1. Comprueba el esquema de la base de datos para una tabla de metadatos (predeterminado SCHEMA_VERSION). Si la tabla de metadatos no existe, la crea.
  2. Escanea el classpath para las migraciones disponibles.
  3. Compara las migraciones con una tabla de metadatos. Si el número de versión es menor o igual que la versión marcada como actual, entonces se ignora.
  4. Marca todas las migraciones restantes como pendientes. Luego los ordena en números de versión ascendentes y los ejecuta en el orden especificado.
  5. A medida que se aplican las migraciones, actualiza la tabla de metadatos.

6. equipos


Flyway tiene los siguientes equipos principales de gestión de la migración:

  • Info . Muestra el estado / versión actual del esquema de la base de datos. Información sobre qué migraciones se esperan, cuáles se aplicaron, el estado de las migraciones completadas y la fecha de su finalización.
  • Migrar . Actualización del esquema de la base de datos a la versión actual. Escanee un classpath para encontrar las migraciones disponibles y aplicar las migraciones pendientes.
  • La línea de base . Instalar la versión del esquema de la base de datos, ignorando las migraciones hasta e incluyendo baselineVersion. La línea de base lo ayuda a usar Flyway en una base de datos existente. Las nuevas migraciones se aplican como de costumbre.
  • El validar . Comprobación del esquema de base de datos actual contra las migraciones disponibles.
  • La reparación . Restaurar tabla de metadatos.
