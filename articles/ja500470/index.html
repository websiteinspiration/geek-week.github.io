今日は、アーキテクチャーの機能、および既存のビジネスロジックをSwiftUIアプリケーションに転送して埋め込む方法について説明します。

SwiftUIの標準データストリームは、Viewと、プロパティと状態変数を含む特定のモデル、またはそのような状態変数の相互作用に基づいています。したがって、MVVMがSwiftUIアプリケーションの推奨アーキテクチャパートナーであることは当然です。AppleはそれをCombineフレームワークと組み合わせて使用​​することを推奨しています。Combineフレームワークは、時間の経過とともに値を処理するための宣言型Api SwiftUIを導入します。ViewModelはObservableObjectプロトコルを実装し、ObservedObjectとして特定のビューに接続します。



変更可能なモデルプロパティは@Publishedとして宣言されます。 

class NewsItemModel: ObservableObject {
    @Published var title: String = ""
    @Published var description: String = ""
    @Published var image: String = ""
    @Published var dateFormatted: String = ""
}

従来のMVVMと同様に、ViewModelはデータモデル（ビジネスロジックなど）と通信し、データを1つの形式または別のビューで転送します。

struct NewsItemContentView: View {
    @ObservedObject var moder: NewsItemModel
    
    init(model: NewsItemModel) {
        self.model = model 
    }
    //... - 
}

MVVMは、他のほとんどのパターンと同様に、輻輳と
冗長性の傾向があります。オーバーロードされたViewModelは常に、ビジネスロジックがどれだけ適切に強調表示および抽象化されるかに依存します。ビューの負荷は、要素の状態変数への依存の複雑さと他のビューへの遷移によって決まります。

SwiftUIで追加されたのは、Viewがクラスではなく構造であるため、継承をサポートしておらず、コードの重複を強制することです。
小規模なアプリケーションではこれが重要ではない場合、機能の成長とロジックの複雑さにより、過負荷が重要になり、大量のコピーと貼り付けが阻害されます。

この場合、クリーンなコードとクリーンなアーキテクチャのアプローチを使用してみましょう。 MVVMを完全に放棄することはできません。結局のところ、DataFlow SwiftUIはMVVMに基づいて構築されていますが、再構築するにはかなり時間がかかります。

コードの純粋性を保証するには、少なくともコードを機能層とモジュールに分割し、問題の一般的な解決策を使用し、コンポーネント間の相互作用の抽象化を実装する必要があります。そして少なくとも、UIコードをいわゆるビジネスロジックから分離する必要があります。

選択したアーキテクチャパターンに関係なく、データベースとネットワークの操作、データの処理と保存のロジックは、UIとアプリケーション自体のモジュールから分離されています。同時に、モジュールはサービスまたはストレージの実装で機能し、サービスまたはストレージは、ネットワーク要求の一般的なサービスまたは一般的なデータストレージにアクセスします。 1つまたは別のサービスにアクセスできる変数の初期化は、アプリケーションモジュール（モジュールビジネスロジック）が最終的にアクセスする特定の一般的なコンテナーで実行されます。



ビジネスロジックを選択して抽象化した場合は、モジュールのコンポーネント間の相互作用を好きなように調整できます。

原則として、iOSアプリケーションの既存のパターンはすべて同じ原理で動作します。



常にビジネスロジックがあり、データがあります。出力用のデータの表示と変換、および変換されたデータの出力先を担当するコールマネージャーもあります。唯一の違いは、コンポーネント間での役割の分散方法です。


なぜなら私たちはアプリケーションを読みやすくするために努力しており、現在および将来の変更を単純化するために、これらすべての役割を分離することが論理的です。私たちのビジネスロジックはすでに強調されており、データは常に分離されています。ディスパッチャー、プレゼンター、ビューを維持します。その結果、View-Interactor-Presenterで構成されるアーキテクチャが得られます。インタラクターはビジネスロジックサービスと対話し、プレゼンターはデータを変換し、一種のViewModelとしてビューに渡します。良い方法では、ナビゲーションと構成もViewから個別のコンポーネントに取り込まれます。



論争の的となる役割をさまざまなコンポーネントに分割したVIP + Rアーキテクチャを採用しています。

例を見てみましょう。
SwiftUIとMVVMで書かれた小さなニュースアグリゲーターアプリケーションがあります。



アプリケーションには、独自のロジックを持つ3つの個別の画面、つまり3つのモジュールがあります。 

  • ニュースリストモジュール。
  • ニュース画面モジュール。
  • ニュース検索モジュール。

各モジュールは、選択したビジネスロジックとやり取りするViewModelと、ViewModelがブロードキャストするものを表示するViewで構成されています。



ViewModelは、表示の準備ができているデータの格納のみに関係するように努めています。現在、彼はサービスへのアクセスと結果の処理の両方に従事しています。

これらの役割をプレゼンターとインタラクターに転送し
ます。プレゼンターとインタラクターは、モジュールごとに設定します。



インタラクターは、サービスから受け取ったデータをプレゼンターに渡します。プレゼンターは、ビューにバインドされた既存のViewModelに準備済みデータを入力します。原則として、モジュールのビジネスロジックの分離に関しては、すべてが単純です。  

今ビューに行きます。強制的なコードの重複に対処してみましょう。ある種のコントロールを扱っているのであれば、それは彼のスタイルや設定かもしれません。画面上のビューの場合は、次のようになります。

  • 画面スタイル
  • 共通のUI要素（LoadingView）;
  • 情報アラート;
  • いくつかの一般的な方法。

継承は使用できませんが、合成はかなり使用できます。この原則により、SwiftUIのすべてのカスタムビューが作成されます。

そこで、Viewコンテナを作成し、そこにすべて同じロジックを転送して、画面上のViewをコンテナ初期化子に渡し、それを本体内のコンテンツViewとして使用します。

struct ContainerView<Content>: IContainer, View where Content: View {
    @ObservedObject var containerModel = ContainerModel()
    private var content: Content

    public init(content: Content) {
        self.content = content
    }

    var body : some View {
        ZStack {
            content
            if (self.containerModel.isLoading) {
                LoaderView()
            }
        }.alert(isPresented: $containerModel.hasError){
            Alert(title: Text(""), message: Text(containerModel.errorText),
                 dismissButton: .default(Text("OK")){
                self.containerModel.errorShown()
                })
        }
    }

画面上のビューは、本文ContainerView内のZStackに埋め込まれています。これには、LoadingViewを表示するためのコードと情報アラートを表示するためのコードも含まれています。

また、内部ビューのViewModelから信号を受信して​​その状態を更新するためにContainerViewが必要です。@Observedを介し
て内部ビューと同じモデルにサブスクライブすることはできません。これは、その信号をドラッグするためです。



したがって、デリゲートパターンを介してそれとの通信を確立し、コンテナーの現在の状態には独自のContainerModelを使用します。

ContainerViewはIContainerプロトコルを実装し、インスタンス参照は埋め込みビューモデルに割り当てられます。 

protocol  IContainer {
    func showError(error: String)
    
    func showLoading()
    
    func hideLoading()
}

struct ContainerView<Content>: IContainer, View where Content: View&IModelView {
    @ObservedObject var containerModel = ContainerModel()
    private var content: Content

    public init(content: Content) {
        self.content = content
        self.content.viewModel?.listener = self
    }
    //- 
}

Viewは、IModelViewプロトコルを実装して、モデルアクセスをカプセル化し、いくつかのロジックを統合します。同じ目的のモデルは、IModelプロトコルを実装します。

protocol IModelView {
    var viewModel: IModel? {get}
}

protocol  IModel:class {
   //....
    var listener:IContainer? {get set}
}

次に、このモデルではすでに、必要に応じてデリゲートメソッドが呼び出され、たとえば、コンテナーモデルの状態変数が変化するエラーを含むアラートが表示されます。

struct ContainerView<Content>: IContainer, View where Content: View&IModelView {
    @ObservedObject var containerModel = ContainerModel()
    private var content: Content

    //- 
    func showError(error: String) {
        self.containerModel.setupError(error: error)
    }
    
    func showLoading() {
        self.containerModel.showLoading()
    }
    
    func hideLoading() {
        self.containerModel.hideLoading()
    }
}

これで、ContainerViewを介して動作するように切り替えて、ビューの動作を統合できます。
