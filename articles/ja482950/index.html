Integer.valueOf（15）== Integer.valueOf（15）; Integer.valueOf（128）！= Integer.valueOf（128）


免責事項：これはjlsの一部です。

2つのラッパーを==（！=）演算子で比較すると、オートボクシングが発生しないことは明らかです。一般的に言って、混乱するのは最初の平等です。実際のところ、整数値の場合、i：-129 <i <128整数ラッパーオブジェクトがキャッシュされます。したがって、この範囲のiの場合、Integer.valueOf（i）は毎回新しいオブジェクトを作成せず、既に作成されているオブジェクトを返します。この範囲に入らないiの場合、Integer.valueOf（i）は常に新しいオブジェクトを作成します。したがって、正確に比較する方法と厳密に比較する方法を厳密に監視しない場合、機能しているように見え、テストでさえカバーされているが、同時にそのような「レーキ」を含むコードを書くことができます。

Oracle（HotSpot）のjvmでは、キャッシュの上限はプロパティ「java.lang.Integer.IntegerCache.high」を使用して変更できます。。

場合によっては、別のクラスのプリミティブまたは文字列final staticフィールドの値がコンパイル時に解決されます


混乱するように聞こえ、ステートメントは少し長くなります。意味はこれです。プリミティブ型または文字列の定数を最終的な静的フィールドとして定義し、すぐに初期化するクラスがある場合、
class AnotherClass {
    public final static String CASE_1 = "case_1";
    public final static String CASE_2 = "case_2";
}

他のクラスで使用される場合、
class TheClass {
    // ...
    public static int getCaseNumber(String caseName) {
        switch (caseName) {
            case AnotherClass.CASE_1:
                return 1;
            case AnotherClass.CASE_2:
                return 2;
            default:
                throw new IllegalArgumentException("value of the argument caseName does not belong to the allowed value set");
        }
    }
}

これらの定数の値（ "case_1"、 "case_2"）はコンパイル時に解決されます。そして、それらはリンクとしてではなく、値としてコードに挿入されます。つまり、ライブラリからそのような定数を使用し、定数の値が変更された新しいバージョンのライブラリを取得した場合、プロジェクトを再コンパイルする必要があります。そうしないと、古い定数値がコードで引き続き使用される可能性があります。

この動作は、定数式（たとえば、スイッチ/ケース）を使用する必要があるすべての場所で観察されます。または、コンパイラーが式を定数に変換することを許可され、それを実行できます。

これらのフィールドは、初期化を静的ブロックに移動して即時初期化を削除するとすぐに、定数式で使用できなくなります。

シングルトンの古典的な「遅延」初期化：二重ロックが必要


このトピックには、非常に論理的に見える次のアプローチ（ダブルチェックイディオム）が常に機能するという誤解があります。
public class UnsafeDCLFactory {
    private Singleton instance;

    public Singleton get() {
        if (instance == null) {  // read 1, check 1
            synchronized (this) {
                if (instance == null) { // read 2, check 2
                    instance = new Singleton();
                }
            }
        }
        return instance; // read 3
    }
}

オブジェクトが作成されたかどうかを確認します（読み取り1、チェック1）。もしそうなら、それを返します。そうでない場合は、ロックを設定し、オブジェクトが作成されていないことを確認し、オブジェクトを作成して（ロックが解除されて）、オブジェクトを返します。

このアプローチは、次の理由で機能しません。（読み取り1、チェック1）と（読み取り3）が同期していません。Javaメモリモデルの概念によれば、別のスレッドで行われた変更は、同期するまでスレッドに表示されない場合があります。感謝mk2 コメントについては、ここに問題の正しい説明があります：
はい、read1とread3は同期していませんが、問題は別のスレッドにはありません。そして、非同期の読み取り値は並べ替えることができます。read1！= nullですが、read3 == nullです。同時に、「インスタンス=新しいシングルトン（）;」による オブジェクトが完全に構築される前にオブジェクトへの参照を取得できます。これは実際には別のスレッドとの同期の問題ですが、read1とread3ではなく、read3とインスタンスメンバーへのアクセスです。
これは、読み取り中に同期を追加するか、シングルトンへのリンクが存続する変数を揮発性としてマークすることによって処理されます。（volatileを使用したソリューションは、Java 5+でのみ機能します。それ以前は、この状況ではjavaに不確実性のあるメモリモデルがありました。）これが作業オプションです（追加の最適化-ローカル変数 `res`が追加され、volatileフィールドからの読み取り数が削減されました）。
public class SafeLocalDCLFactory {
    private volatile Singleton instance;

    public Singleton getInstance() {
        Singleton res = instance;     // read 1
        if (res == null) {            // check 1
            synchronized (this) {
                res = instance;       // read 2
                if (res == null) {    // check 2
                    res = new Singleton();
                    instance = res;
                }
            }
        }
        return res;
    }
}

「初期化オンデマンドホルダーイディオム」-シングルトンの非常に美しい「遅延」初期化


javaは、クラス（Classオブジェクト）を必要なときにだけ初期化します。そして、あなたはこれを利用することができます！初期化オンデマンドホルダーイディオムメカニズムはまさにそれを行います。（コードはここからです。）
public class Something {
    private Something() {}

    private static class LazyHolder {
        static final Something INSTANCE = new Something();
    }

    public static Something getInstance() {
        return LazyHolder.INSTANCE;
    }
}

LazyHolderクラスは、Something.getInstance（）が初めて呼び出されたときにのみ初期化されます。JVMはこれが1回だけ行われ、しかも非常に効率的に行われるようにします。クラスがすでに初期化されている場合、オーバーヘッドはありません。したがって、LazyHolder.INSTANCEも「lazy」でスレッドセーフに1回初期化されます。
オーバーヘッドに関する仕様の一部
If this initialization procedure completes normally and the Class object is fully initialized and ready for use, then the invocation of the initialization procedure is no longer necessary and it may be eliminated from the code — for example, by patching it out or otherwise regenerating the code.
