「ゼロ」地獄とそれから抜け出す方法

Null値は、軽率に使用すると、あなたの人生を耐え難くする可能性があり、そのような痛みの原因を正確に理解していない場合もあります。説明させてください。


デフォルト値


私たちは皆、多くの引数を取るメソッドを見ましたが、それらの半分以上はオプションです。結果は次のようになります。

public function insertDiscount(
  string $name,
  int $amountInCents,
  bool $isActive = true,
  string $description = '',
  int $productIdConstraint = null,
  DateTimeImmutable $startDateConstraint = null,
  DateTimeImmutable $endDateConstraint = null,
  int $paymentMethodConstraint = null
): int

上記の例では、デフォルトですべての場所に適用される割引を作成しますが、作成時に非アクティブにしたり、特定の製品にのみ適用したり、特定の時間にのみ機能したり、ユーザーが特定の支払い方法を選択したときに適用したりできます。

特定の支払い方法の割引を作成する場合は、次のようにメソッドを呼び出す必要があります。

insertDiscount('Discount name', 100, true, '', null, null, null, 5);

このコードは機能しますが、それを読む人には完全に理解できません。分析が非常に難しくなるため、アプリケーションのサポートは容易ではありません。

この例を引数ごとに見てみましょう。

有効な割引とは何ですか？


無制限の割引がどこにでも適用されることはすでにわかっています。したがって、有効な割引には、後で追加できる制限を除くすべてが含まれます。引数isActiveのデフォルト値はtrueです。したがって、このメソッドは次のように呼び出すことができます。

insertDiscount('Discount name', 100);

コードを読んだだけでは、割引がすぐに有効になることはわかりません。調べるには、メソッドシグネチャにデフォルト値があるかどうかを確認する必要があります。

ここで、200行のコードを読み取る必要があると想像してください。呼び出されたメソッドのすべてのシグネチャで非表示の情報を確認してもよろしいですか？何も探すことなくコードを読むだけです。

説明を担当する議論についても同様です。デフォルトでは、空の文字列です。これにより、説明が表示されると予想される場所で多くの問題が発生する可能性があります。たとえば、小切手に印刷することはできますが、それが空であるため、金額の行の横に空の行が表示されます。システムは、これが発生することを許可してはなりません。

このメソッドを次のように書き直します。

public function insertDiscount(
  string $name,
  string $description,
  int $amountInCents,
  bool $isActive
): int

後で別の方法を使用して追加することにしたため、制限を完全に削除しました。すべてのパラメーターが必要になったため、任意の順序で配置できます。コードが近くにある方が読みやすいので、名前の直後に説明を配置しました。

insertDiscount(
  'Discount name',
  'Discount description',
  100,
  Discount::STATUS_ACTIVE
);

また、割引活動ステータスには定数を使用しました。これで、この引数の真の意味を見つけるためにメソッドのシグネチャを調べる必要はありません。アクティブな割引を作成していることが明らかになります。将来的には、この方法をさらに改善することができます（ネタバレ：値オブジェクトの使用）。

制約を追加する


これで、さまざまな制限を追加できます。ゼロ、ゼロ、ゼロの地獄を回避するために、個別のメソッドを作成します。

public function addProductConstraint(
  Discount $discount,
  int $productId
): Discount;

public function addDateConstraint(
  Discount $discount,
  DateTimeImmutable $startDate,
  DateTimeImmutable $endDate
): Discount;

public function addPaymentMethodConstraint(
  Discount $discount,
  int $paymentMethod
): Discount;

したがって、特定の制限付きの新しい割引を作成する場合は、次のようにします。

$discountId = insertDiscount(
  'Discount name',
  'Discount description',
  100,
  Discount::STATUS_ACTIVE
);

addPaymentMethodConstraint(
  $discountId,
  PaymentMethod::CREDIT_CARD
);

これを元の呼び出しと比較します。読むのがとても便利になったことがわかります。

オブジェクトプロパティがnull


オブジェクトプロパティのゼロを解決すると、問題が発生します。私はこのようなものをどれくらいの頻度で見るかを伝えることができません：

$currencyCode = strtolower(
  $record->currencyCode
);

ブーム！ 「nullをstrtolowerに渡すことはできません。」これは、開発者がcurrencyCodeがnullの可能性があることを忘れたために発生しました。多くの開発者はまだIDEを使用していないか、または警告を抑制していないため、これは何年も気付かれない可能性があります。エラーは未読のログに引き続き記録され、クライアントは明らかにこれに関連しないと思われる定期的な問題を報告するため、誰もこのコード行を見る気になりません。

もちろん、currencyCodeにアクセスできるところならどこでもnullチェックを追加できます。しかし、それから私たちは別の種類の地獄に終わるでしょう：

if ($record->currencyCode === null) {
  throw new \RuntimeException('Currency code cannot be null');
}

if ($record->amount === null) {
  throw new \RuntimeException('Amount cannot be null');
}

if ($record->amount > 0) {
  throw new \RuntimeException('Amount must be a positive value');
}

しかし、すでに理解したように、これは最良の解決策ではありません。メソッドが煩雑になるだけでなく、このテストをどこでも繰り返す必要があります。そして、別のnullプロパティを追加するたびに、そのような別のチェックを行うことを忘れないでください！幸い、値オブジェクトという単純な解決策があります。

値オブジェクト


値オブジェクトは強力ですが単純なものです。私たちが解決しようとしていた問題は、すべてのプロパティを常に検証する必要があるということでした。しかし、オブジェクトのプロパティを信頼できるかどうか、それらが有効かどうかわからないため、これを行います。もし私たちができるとしたら？

値を信頼するには、2つの属性が必要です。それらは検証される必要があり、検証後に変更されてはなりません。このクラスを見てください：

final class Amount
{
  private $amountInCents;
  private $currencyCode;

  public function __construct(int $amountInCents, string $currencyCode): self
  {
    Assert::that($amountInCents)->greaterThan(0);

    $this->amountInCents = $amountInCents;
    $this->currencyCode = $currencyCode;
  }

  public function getAmountInCents(): int
  {
    return $this->amountInCents;
  }

  public function getCurrencyCode(): string
  {
    return $this->currencyCode;
  }
}

私はbeberlei / assertパッケージを使用しています。チェックが失敗すると、例外がスローされます。これは、検証をこのコンストラクターに移動していない限り、ソースコードのnullの例外と同じです。

型宣言を使用するため、型も正しいことが保証されます。したがって、intをstrtolowerに渡すことはできません。型宣言をサポートしていない古いバージョンのPHPを使用している場合は、このパッケージを使用して->整数（）および->文字列（）で型をチェックできます。

オブジェクトを作成した後は、getterのみがあり、setterがないため、値を変更することはできません。これは免疫と呼ばれます。finalを追加しても、このクラスを拡張してセッターやマジックメソッドを追加することはできません。メソッドのパラメーターにAmount $が表示されている場合、そのすべてのプロパティが検証され、オブジェクトが安全に使用できることを100％確認できます。値が有効でない場合、オブジェクトを作成できません。

ここで、値オブジェクトを使用して、例をさらに改善できます。

$discount = new Discount(
  'Discount name',
  'Discount description',
  new Amount(100, 'CAD'),
  Discount::STATUS_ACTIVE
)

insertDiscount($discount);

最初にディスカウントを作成し、内部で量を引数として使用することに注意してください。これにより、insertDiscountメソッドが有効なディスカウントオブジェクトを受け取ることに加えて、このコードブロック全体がはるかに理解しやすくなります。

ゼロホラーストーリー


アプリケーションでnullが有害になる可能性のある興味深いケースを見てみましょう。アイデアは、データベースからコレクションを抽出し、それをフィルタリングすることです。

$collection = $this->findBy(['key' => 'value']);
$result = $this->filter($collection, $someFilterMethod);

if ($result === null) {
   $result = $collection;
}

結果がnullの場合、元のコレクションを結果として使用しますか？適切な値が見つからなかった場合、フィルタリングメソッドはnullを返すため、これには問題があります。したがって、すべてがフィルターされている場合、フィルターを無視してすべての値を返します。これは完全に論理を壊します。

結果として元のコレクションが使用されるのはなぜですか？私たちは決して知りません。開発者はこのコンテキストでnullが何を意味するかについて一定の仮定を持っていたと思いますが、それは間違っていることが判明しました。

これはnull値の問題です。ほとんどの場合、それらが何を意味するのかは明確ではないため、それらにどのように反応するかを推測することしかできません。間違いを犯すのはとても簡単です。一方、例外は非常に明確です。

try {
  $result = $this->filter($collection, $someFilterMethod);
} catch (CollectionCannotBeEmpty $e) {
  // ...
}

このコードは一意です。開発者がそれを誤って解釈することはほとんどありません。

努力する価値はありますか？


同じことをするコードを書くための余分な努力のように見えます。それがそうです。しかし同時に、コードの読み取りと理解に費やす時間が大幅に短縮されるため、その努力は報われるでしょう。コードの記述に費やす時間が1時間増えるごとに、コードを変更したり新しい機能を追加したりする必要がない日を節約できます。保証された高利回りの投資と考えてください。
