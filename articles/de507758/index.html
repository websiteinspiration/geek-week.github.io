JavaScript-Speicher - keine Lecks









Wie Sie Ihre Daten erstellen und darauf zugreifen, kann sich auf die Leistung Ihrer Anwendung auswirken. Mal sehen wie.



Einführung



JavaScript ist eine sehr hohe Sprache. Daher denken die meisten Entwickler nicht darüber nach, wie die Daten im Speicher gespeichert werden. In diesem Artikel wird untersucht, wie Daten im Speicher gespeichert werden, wie sie sich auf den Prozessor und den Speicher auswirken und wie sich die Art und Weise, wie Sie Daten an JS verteilen und darauf zugreifen, auf die Leistung auswirkt.



Dreiecksbeziehung



Wenn ein Computer einige Berechnungen durchführen muss, benötigt ein Prozessor (CPU) Daten zur Verarbeitung. Entsprechend der Aufgabe sendet er somit eine Anforderung zur Datenabtastung über den Bus an den Speicher.



Es sieht so aus:







Das ist also unser romantisches Dreieck - Prozessor -> Bus -> Speicher



Es ist schwierig, eine gesunde Beziehung im Trio aufrechtzuerhalten



Der Prozessor ist viel schneller als der Speicher. Somit verarbeitet dieser Prozessor -> Bus -> Speicher -> Bus -> Der Prozessor Zeit für Berechnungen. Während der Speicher gescannt wird, befindet sich der Prozessor im Leerlauf.



Um Ausfallzeiten zu vermeiden, wurde dem System ein Cache hinzugefügt. Wir werden nicht auf Details darüber eingehen, was ein Cache ist oder welche Arten von Cache es gibt, aber es genügt zu sagen, dass der Cache der interne Speicher des Prozessors ist.



Wenn die CPU einen Befehl zum Ausführen empfängt, sucht sie zuerst nach Daten im Cache, und wenn dort keine Daten vorhanden sind, sendet sie eine Anforderung über den Bus.



Der Bus wiederum überträgt die angeforderten Daten sowie den bedingten Teil des Speichers und speichert sie zur schnellen Referenz im Cache.



Somit ist die CPU weniger unzufrieden damit, wie langsam der Speicher ist, und daher hat die CPU weniger Ausfallzeiten.



Streitigkeiten sind Teil jeder Beziehung.



Das Problem, das auftreten kann - insbesondere bei der Verarbeitung großer Datenmengen -, ist ein Phänomen, das als Cache-Miss bezeichnet wird.



Ein Cache-Fehler bedeutet, dass die CPU während der Berechnungen feststellt, dass sie nicht über die erforderlichen Daten im Cache verfügt, und diese Daten über reguläre Kanäle anfordern muss. Sie erinnern sich, dass viel langsamer Speicher vorhanden ist.







Geldscheinillustration. Daten aus dem Array werden verarbeitet, aber Daten außerhalb des Cache-Limits werden ebenfalls zur Berechnung angefordert - und dies führt zu einem „Cache-Miss“.



Ok, aber ich bin ein JavaScript-Entwickler. Warum sollte es mich interessieren?



Gute Frage. In den meisten Fällen denken Sie nicht darüber nach. Wenn heute jedoch immer mehr Daten über in Node.js geschriebene Server übertragen werden, tritt beim Sortieren großer Datenmengen mit größerer Wahrscheinlichkeit ein Cache-Miss-Problem auf.



Ich werde es glauben, wenn ich es sehe !!!



Messe. Schauen wir uns ein Beispiel an.



Hier ist eine Klasse namens Boom. 



class Boom {
  constructor(id) {
    this.id = id;
  }
  setPosition(x, y) {
    this.x = x;
    this.y = y;
  }
}


Diese Klasse (Boom) hat nur 3 Eigenschaften - id, x und y.



Erstellen wir nun eine Methode, die x und y auffüllt.



Lassen Sie uns die Daten einstellen: 



const ROWS = 1000;
const COLS = 1000;
const repeats = 100;
const arr = new Array(ROWS * COLS).fill(0).map((a, i) => new Boom(i));


Jetzt werden wir diese Daten in der Methode verwenden: 



function localAccess() {
  for (let i = 0; i < ROWS; i++) {
    for (let j = 0; j < COLS; j++) {
      arr[i * ROWS + j].x = 0;
    }
  }
}


LocalAccess durchläuft das Array linear und setzt x auf 0.



Wenn wir diese Funktion 100 Mal wiederholen (siehe Wiederholungskonstante), können wir messen, wie lange die Ausführung dauert: 



function repeat(cb, type) {
  console.log(`%c Started data ${type}`, 'color: red');
  const start = performance.now();
  for (let i = 0; i < repeats; i++) {
    cb();
  }
  const end = performance.now();
  console.log('Finished data locality test run in ', ((end - start) / 1000).toFixed(4), ' seconds');
  return end - start;
}
repeat(localAccess, 'Local');

Preis für Cache Miss



Nach dem, was wir oben gelernt haben, können wir nun das Cache-Miss-Problem berücksichtigen, wenn wir Daten verarbeiten, die während der Iteration weit entfernt sind. Gelöschte Daten sind Daten, die sich nicht im benachbarten Index befinden. Hier ist die Funktion dafür: 



function farAccess() {
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      arr[j * ROWS + i].x = 0;
    }
  }
}


Was hier passiert, ist, dass wir uns bei jeder Iteration dem Index zuwenden, der sich in einem ROWS-Abstand von der letzten Iteration befindet. Wenn die ROWS 1000 ist (wie in unserem Fall), erhalten wir daher die folgende Iteration: [0, 1000, 2000, ..., 1, 1001, 2001, ...].



Fügen wir dies unserem Geschwindigkeitstest hinzu: 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local');
}, 2000);


Und hier ist das Endergebnis: Die







nichtlokale Iteration war fast viermal langsamer. Dieser Unterschied nimmt mit zunehmenden Daten zu. Dies liegt daran, dass der Prozessor aufgrund eines Cache-Fehlers im Leerlauf ist.



Welchen Preis zahlen Sie? Es hängt alles von der Größe Ihrer Daten ab.



Okay, ich schwöre, das werde ich nie tun!



Vielleicht denken Sie nicht darüber nach, aber ... es gibt Fälle, in denen Sie auf ein Array mit einer Logik zugreifen möchten, die nicht linear (z. B. 1,2,3,4,5) oder nicht bedingt (z. B. für) ist. sei i = 0; i <n; i + = 1000)).



Beispielsweise erhalten Sie Daten von einem Dienst oder einer Datenbank und müssen diese mithilfe komplexer Logik verarbeiten, sortieren oder filtern. Dies kann zu einer ähnlichen Situation führen wie in der farAccess-Funktion.



Hier ist eine Illustration, die (hoffentlich) die Situation verdeutlicht:







Die Originaldaten im Speicher gegenüber den sortierten Daten im Speicher. Zahlen geben die Indizes der Objekte im Quellarray an.



Wenn wir uns das Bild oben ansehen, sehen wir die Daten in der Form, in der sie gespeichert sind (oberer grauer Balken). Unten sehen wir das Array, das erstellt wurde, als die Daten vom Server kamen. Schließlich sehen wir ein sortiertes Array, das Verweise auf Objekte enthält, die an verschiedenen Positionen im Speicher gespeichert sind.



Das Iterieren über ein sortiertes Array kann daher zu mehreren Fehlern im Cache führen, wie im obigen Beispiel gezeigt.

Beachten Sie, dass dieses Beispiel für ein kleines Array gilt. Cache-Fehler sind für viel größere Daten relevant.
In einer Welt, in der Sie reibungslose Animationen im Frontend benötigen und jede Millisekunde Prozessorzeit im Backend laden können, kann dies kritisch werden.



Oh nein! Alles ist verloren !!!



Nein überhaupt nicht.



Es gibt verschiedene Lösungen für dieses Problem, aber jetzt, da Sie den Grund für diesen Leistungsabfall kennen, können Sie wahrscheinlich selbst eine Lösung finden. Sie müssen nur Daten, die so nah wie möglich verarbeitet werden, im Speicher speichern.

Diese Technik wird als Datenlokalitäts-Entwurfsmuster bezeichnet .
Lassen Sie uns unser Beispiel fortsetzen. Angenommen, in unserer Anwendung ist der häufigste Prozess die Datenverarbeitung unter Verwendung der in der Funktion farAccess gezeigten Logik. Wir möchten die Daten so optimieren, dass sie in der gängigsten for-Schleife schnell ausgeführt werden.



Wir organisieren folgende Daten: 



const diffArr = new Array(ROWS * COLS).fill(0);
for (let col = 0; col < COLS; col++) {
  for (let row = 0; row < ROWS; row++) {
    diffArr[row * ROWS + col] = arr[col * COLS + row];
  }
}


In diffArr werden nun die Objekte, die sich in den Indizes [0,1,2, ...] des ursprünglichen Arrays befinden, wie folgt festgelegt: [0,1000,2000, ..., 1, 1001, 2001, ..., 2, 1002 , 2002, ...]. Die Zahlen geben den Index des Objekts an. Dies simuliert die Sortierung eines Arrays. Dies ist eine Möglichkeit, das Entwurfsmuster für die Datenlokalität zu implementieren.



Um dies zu überprüfen, werden wir unsere farAccess-Funktion geringfügig ändern, um ein benutzerdefiniertes Array zu erhalten: 



function farAccess(array) {
  let data = arr;
  if (array) {
    data = array;
  }
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      data[j * ROWS + i].x = 0;
    }
  }
}


Fügen Sie nun das Skript zu unserem Test hinzu: 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local')
  setTimeout(() => {
    repeat(() => farAccess(diffArr), 'Non Local Sorted')
  }, 2000);
}, 2000);


Wir führen es und wir bekommen:







Und voila! „Wir haben unsere Daten für einen allgemeineren Ansatz optimiert.“

Ausgabe



In diesem Artikel haben wir das Entwurfsmuster für die Datenlokalität demonstriert. Tatsächlich haben wir gezeigt, dass der Zugriff auf eine Datenstruktur auf eine Weise, für die sie nicht optimiert ist, die Leistung verringern kann. Dann haben wir die Daten entsprechend unserer Art der Verarbeitung optimiert - und gesehen, wie sie die Leistung verbessern.



Das Entwurfsmuster für die Datenlokalität ist im Bereich der Spieleentwicklung üblich, wo Sie häufig mit vielen Entitäten umgehen müssen, die viele Male wiederholt werden. In den obigen Ergebnissen sehen wir, dass die Datenlokalität auch in Hochsprachen wie JavaScript und nicht nur in Spielen immer noch wichtig ist.



Angesichts der Datenmenge, die zwischen Servern übertragen oder in Browser heruntergeladen wird, müssen Anwendungsentwickler heute Entwurfsmuster berücksichtigen, die für Spieleentwickler früher alltäglich waren - sowohl auf der Serverseite als auch auf der Clientseite.
