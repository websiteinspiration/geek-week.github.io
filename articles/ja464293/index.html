React Reduxでフックは置き換えられますか？

フックはReactに登場して以来、Reduxを置き換えることができるかどうかについて多くの質問がありました。

フックとReduxにはほとんど共通点がないと思います。フックは私たちに運命を扱うためのいくつかの新しい驚くべき機会を与えません。代わりに、APIを拡張して、Reactですでに可能だったことができるようにします。ただし、フックAPIにより、Reactの状態管理機能をより便利に使用できるようになりました。クラスに基づくコンポーネントで利用可能だった古いものよりも、状態を操作するための新しい機会を使用する方が簡単であることがわかりました。今では、コンポーネントの状態を操作するためのツールを以前よりもずっと頻繁に使用しています。当然、これは適切な場合にのみ行います。



ReactフックとReduxに対する私の態度を説明するために、最初にReduxが通常使用される状況についてお話します。

Reduxとは何ですか？


Reduxは、アプリケーションの状態の予測可能なストレージを実装するライブラリです。また、Reactとシームレスに統合するアーキテクチャでもあります。

Reduxの主な長所は次のとおりです。

  • 状態の確定的な表現（純粋なコンポーネントと組み合わせることで、確定的な視覚要素の形成が可能になります）。
  • トランザクションの状態変更のサポート。
  • I / Oメカニズムと副作用からの状態管理の分離。
  • 州の信頼できるデータの単一のソースの存在。
  • さまざまなコンポーネントの状態とのコラボレーションの簡単な編成。
  • トランザクション分析ツール（アクションオブジェクトの自動ロギング）。
  • プログラム実行プロセスを記録および再生する機能を備えたデバッグ（タイムトラベルデバッグ、TTD）。

つまり、Reduxを使用すると、コードを適切に整理し、便利にデバッグすることができます。Reduxは、保守が容易なアプリケーションの開発に役立ちます。このライブラリを使用すると、プログラムで発生する問題の原因を見つけやすくなります。

Reactフックとは何ですか？


Reactフックを使用すると、機能コンポーネントを操作するときに、クラスに基づいたコンポーネントの状態の類似物と、そのライフサイクルメソッドの類似物を使用できます。フックはReact 16.8で登場しました。

フックの主な長所は次のとおりです。

  • .
  • , ( render prop).

これらの優れた機能は、実際にはReduxを圧倒するものではないことに注意してください。Reactフックは確定的状態の更新を実行するために使用できますが、これを使用する必要がありますが、これは常にReact機能の1つであり、Redux確定的状態モデルはこの機能とうまく組み合わされています。これが、視覚要素の出力においてReactが決定論を達成する方法であり、これは、誇張なしに、Reactを作成するための原動力の1つです。フック付きのreact-redux APIやReact useReducerフックの

ようなツールを使用する場合、フックまたはReduxのどちらを選択するかを尋ねる理由がないことがわかります。これらのテクノロジーを組み合わせたり組み合わせたりして、両方を使用できます。

フックAPIの出現後、私は次のテクノロジーの使用をやめました。

  • Render-propパターン。

フックを置き換えないものは何ですか？


私は今でも次のテクノロジーをよく使用しています。

  • Redux-上記の理由のすべてのため。
  • 高次コンポーネント-アプリケーションのビジュアルコンポーネントのすべてまたは一部で共有されるエンドツーエンドの機能を実装する必要がある場合にコンポーネント構成を実行するため。このような機能には、Reduxプロバイダー、ページレイアウトシステム、アプリケーション設定サポートシステム、認証および承認ツール、アプリケーション国際化ツールなどが含まれます。
  • コンテナーコンポーネントと視覚的表現を持つコンポーネント間の分離。これにより、アプリケーションのモジュール性とテスト容易性を向上させることができます。エフェクトと純粋なロジックを分離することをお勧めします。

いつフックを使用するのですか？


すべてのアプリケーションとすべてのコンポーネントでReduxを使用するように努力する必要はありません。プロジェクトが1つのビジュアルコンポーネントで構成されており、データを保存せず、そこからデータをロードしない場合、非同期I / O操作が実行されない場合、Reduxを使用してこのプロジェクトを複雑にするに値する理由は考えられません。

同じことが、次の機能を持つコンポーネントについても言えます。

  • ネットワークリソースは使用しません。
  • それらはデータを状態に保存せず、そこからロードしません。
  • 子孫ではない他のコンポーネントと状態を共有しません。
  • 短期的なデータストレージに使用される特定の状態はありません。

以前は、コンポーネントの状態の操作には、クラスベースのコンポーネントの使用、クラスプロパティを宣言するメカニズム（またはクラスコンストラクター内）を使用した状態への初期データの書き込みなどが含まれていました。その結果、Reduxを使用しないようにするためには、コンポーネントが複雑すぎる必要があることがわかりました。 Reduxは、Reduxを使用してフォームの状態を管理するための便利なツールの存在にも賛成しました。その結果、以前は、フォームの一時的な状態が、より長い有効期間を持つデータと同じ場所に格納されることを心配していませんでした。

私は多かれ少なかれ複雑なすべてのアプリケーションでReduxをすでに使用しているので、コンポーネントの状態を保存するためのテクノロジーの選択はあまり考えられませんでした。私はほとんどすべての場合にReduxを使用しました。

現代の状況では、選択を行うことも難しくありません。コンポーネントの状態の操作は、標準のReactメカニズムを使用して編成され、Reduxを使用してアプリケーションの状態を管理します。

いつReduxを使用するのですか？


状態管理に関するもう1つの一般的な質問は次のとおりです。「すべてをReduxリポジトリに置く必要がありますか？これを行わない場合、TTDメカニズムを使用してアプリケーションをデバッグする機能に違反しますか？」

Reduxリポジトリのすべてを完全にホストする必要はありません。実際のところ、アプリケーションでは、ログに記録されたり、デバッグ中に使用されたりして、開発者が問題を発見する際に目立った助けとなる情報を提供するために、散在しすぎている多くの一時データを使用します。おそらく、リアルタイムエディタアプリケーションを作成しているのでない限り、マウスの動きやキーストロークをすべて状態に書き込む必要はありません。何かをRedux状態にすると、アプリケーションに抽象化のレベルが追加され、それに伴う複雑さのレベルも追加されます。

つまり、Reduxを安全に使用できますが、これには何らかの理由があるはずです。コンポーネントで次の機能が異なる場合、コンポーネントでのRedux機能の使用は正当化できます。

  • I / Oを使用します。たとえば、ネットワークや特定のデバイスで動作します。
  • データを保存するか、データをロードします。
  • 子孫ではないコンポーネントと組み合わせて状態を操作します。
  • また、アプリケーションの他の部分が扱うビジネスロジックも処理します。アプリケーションの他の部分で使用されるデータを処理します。

このコードが動作する状態フラグメントは、さまざまなコンポーネントによって使用され、1つのコンポーネントだけでは処理されません。これはデータではなく、短期間しか存在しません。このデータは永続的と見なすことができ、アプリケーションのさまざまな画面や複数のセッションで使用できます。これらはすべて、データを保存するためのコンポーネントの状態を適用できないシナリオです。ただし、これは引き続き可能ですが、アプリケーションの作成者が、React APIに基づいて、状態を管理するための独自のライブラリを作成する場合に限ります。これは、単にReduxを使用するよりもはるかに困難です。

React Suspense APIは、将来、ある状態でデータストレージを実行し、そこからデータをロードするときに便利になるかもしれません。そのリリースを待ち、Reduxデータを保存およびロードするためのテンプレートを置き換えることができるかどうかを確認する必要があります。Reduxを使用すると、副作用を残りのコンポーネントロジックから明確に分離できますが、特別な方法でI / Oサービスを操作する必要はありません。（私がライブラリredux-sagaミドルウェアredux-a thunkを好む理由は、効果を分離するためです）。このシナリオでReduxと競合するには、React APIが効果の分離を提供する必要があります。

Reduxはアーキテクチャです


Reduxは、状態管理ライブラリよりもはるかに多く（多くの場合はるかに少ない）です。これはFluxアーキテクチャのサブセットでもあり、状態変更の実装方法をより厳密に定義しています。 Reduxアーキテクチャの詳細については、こちらをご覧ください。

コンポーネントの複雑な状態を維持する必要があるが、Reduxライブラリを使用する必要がない場合は、Reduxスタイルのレデューサーをよく使用します。さらに、Reduxの精神で作成されたアクション（およびAutoduxやredux-sagaなどのReduxツールでさえ）を使用して、Node.jsアプリケーションでアクションを送信します。ただし、このようなアプリケーションにReduxをインポートすることすらしていません。

Reduxプロジェクトは、ライブラリというよりも、常にアーキテクチャであり、一連の自発的な合意でした。実際、Reduxの基本的な実装は、文字通り数十行のコードでレイアウトできます。

これは、コンポーネントのローカル状態をフックでより頻繁に使用し、すべてをReduxにバインドしたくない人にとっては朗報です。

ReactはフックをサポートしますuseReducerReduxスタイルで記述されたレデューサーと連携できます。これは、状態を操作する重要なロジックの実装、依存する状態のフラグメントの操作などに適しています。個々のコンポーネントの一時的な状態が適切なタスクに直面している場合は、Reduxアーキテクチャを使用してこの状態を処理できますが、Reduxライブラリの代わりに、フックを使用して状態を管理できますuseReducer。

以前は一時的に保存していたデータの永続的な保存を後で確立する必要がある場合は、そのような変更に90％の準備ができています。コンポーネントをReduxリポジトリに接続し、対応するレデューサーをそこに追加するだけです。

質問と答え


RedReduxがすべてのアプリケーションデータを管理しないと、決定論は壊れますか？


いいえ、壊れていません。実際、Reduxを使用してもプロジェクトは確定的ではありません。しかし、合意はそうです。Reduxの状態を確定的にする場合は、純粋な関数を使用します。ローカルコンポーネントの一時的な状態を確定する必要がある場合も同様です。

Red Reduxライブラリは、信頼できるデータの単一のソースの役割を果たす必要がありますか？


信頼できるデータの単一のソースの原則は、アプリケーションの状態に含まれるすべてのデータが1つの場所に格納される必要があることを示していません。この原則の意味は、状態の各フラグメントが信頼できるデータのソースを1つだけ持つべきであることです。その結果、多くの状態フラグメントが存在する可能性があり、それぞれに信頼できるデータの独自のソースがあります。

これは、プログラマがReduxに何を転送し、コンポーネントの状態に何を転送するかを決定できることを意味します。状態決定データは、他のソースから取得することもできます。たとえば、表示しているページのアドレスに関する情報を操作できるブラウザAPIから。

Reduxは、アプリケーションの状態に関する信頼できるデータの単一のソースをサポートする優れたツールです。ただし、コンポーネントの状態がこのコンポーネント内でのみ特定され、使用される場合、定義により、この状態には信頼性の高いデータの単一のソース（Reactコンポーネントの状態）がすでにあります。

一部のデータをRedux状態にする場合は、常にこのデータをRedux状態から読み取る必要があります。 Reduxリポジトリにあるものすべてについて、このリポジトリは信頼できるデータの唯一のソースでなければなりません。

必要に応じて、すべてをRedux状態にすることは完全に正常です。頻繁に更新する必要がある状態フラグメントを使用する場合、または依存状態フラグメントが頻繁に使用されるコンポーネントの状態を保存することについて話している場合、これはおそらくパフォーマンスに影響します。パフォーマンスに問題があるまで、パフォーマンスについて心配する必要はありません。ただし、パフォーマンスの問題が心配な場合は、両方の方法で状態を操作し、パフォーマンスへの影響を評価してください。プロジェクトのプロファイリングを行い、RAILパフォーマンスモデルを覚えておきます。

react react-reduxから接続機能を使用する必要がありますか、それともフックを使用する方が良いですか？


それは多くに依存します。この関数connectは繰り返し使用するのに適した高次コンポーネントを作成し、フックは単一コンポーネントとの統合用に最適化されています。

同じプロパティを別のコンポーネントに接続する必要がありますか？もしそうなら、それを使用してくださいconnect。そうでなければ、私はフックを選ぶことを好みます。たとえば、ユーザーアクションの権限の承認を担当するコンポーネントがあるとします。

import { connect } from 'react-redux';
import RequiresPermission from './requires-permission-component';
import { userHasPermission } from '../../features/user-profile/user-profile-reducer';
import curry from 'lodash/fp/curry';

const requiresPermission = curry(
  (NotPermittedComponent, { permission }, PermittedComponent) => {
    const mapStateToProps = state => ({
      NotPermittedComponent,
      PermittedComponent,
      isPermitted: userHasPermission(state, permission),
    });

    return connect(mapStateToProps)(RequiresPermission);
  },
);

export default requiresPermission;

これで、管理者がアプリケーションで集中的に作業し、そのすべてのアクションに特別な権限が必要な場合、これらのすべての権限と必要なすべてのエンドツーエンド機能を組み合わせた高次コンポーネントを作成できます。

import NextError from 'next/error';
import compose from 'lodash/fp/compose';
import React from 'react';
import requiresPermission from '../requires-permission';
import withFeatures from '../with-features';
import withAuth from '../with-auth';
import withEnv from '../with-env';
import withLoader from '../with-loader';
import withLayout from '../with-layout';

export default compose(
  withEnv,
  withAuth,
  withLoader,
  withLayout(),
  withFeatures,
  requiresPermission(() => <NextError statusCode={404} />, {
    permission: 'admin',
  }),
);

このタスクには、高次コンポーネントAPIが便利です。フックを使用するよりも少ないコードを使用して、より簡潔に解決できます。ただし、この関数を使用するにはconnect、最初の引数としてそれを受け入れmapStateToProps、2番目の引数として- を覚えておく必要がありますmapDispatchToProps。この関数は関数またはオブジェクトリテラルを取ることができることを忘れないでください。さまざまなユースケースがどのように異なるかconnect、そしてこれはカリー化された関数ですが、そのカリー化は自動的には実行されないことを知る必要があります。

言い換えれば、私は開発の際にconnectコードを簡潔にするために多くの作業が行われましたが、結果として得られたコードは、特に読みやすく、特に便利でもありません。複数のコンポーネントを処理する必要がない場合はconnect、コード量の増加につながるという事実を考慮に入れても、はるかに便利なフックAPIよりも不便なAPIを優先したいと思います。

aシングルトンがアンチパターンと見なされ、Reduxがシングルトンである場合、これはReduxがアンチパターンであることを意味しますか？


いいえ、ありません。コードでのシングルトンパターンの使用は、このコードの疑わしい品質を示唆し、コード内に共有された可変状態の存在を示します。これは本当のアンチパターンです。一方、Reduxは、カプセル化（レデューサーの外でアプリケーションの状態を直接変更しないでください。Reduxが状態変更の問題を解決します）とメッセージの送信（送信されたイベントオブジェクトのみが状態変更を引き起こす可能性があります）によって、共有状態の変更を防ぎます。
