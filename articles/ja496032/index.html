3ノードのZooKeeperクラスター構成とApache Kafkaブローカー

良い一日！

この記事では、3つのZooKeeperノード（分散システム調整サービス）のクラスターをセットアップすることを検討します。そのうち2つはKafkaメッセージブローカーで、3つ目はマネージャーです。

その結果、次のコンポーネントスキームが実装されます。



1台のマシン上のすべてのコンポーネントでは、簡単にするために、組み込みのコンソールパブリッシャー（プロデューサー）メッセージとサブスクライバー（コンシューマー）メッセージを使用します。

ZooKeeperモジュールはKafkaパッケージに組み込まれているので、それを使用します。

インストールとセットアップ


Kafkaパッケージのインストール

この場合、オペレーティングシステムはUbuntu 16.04 LTSです。

現在のバージョンと詳細な手順は公式ウェブサイトにあります。

wget http://mirror.linux-ia64.org/apache/kafka/2.4.1/kafka_2.12-2.4.1.tgz
tar -xvzf kafka_2.12-2.4.1.tgz

server_1、server_2、server_3の3つのノードの作業ディレクトリを作成します。

mv kafka_2.12-2.4.1 kafka_server_1
cp -r kafka_server_1/ kafka_server_2/
cp -r kafka_server_1/ kafka_server_3/

ノードの

構成Kafkaブローカーの構成
（それぞれディレクトリーkafka_server_1、kafka_server_2、およびkafka_server_3から）

vim config/server.properties

  • ブローカーID（それぞれ0、1、2）
  • クライアントポート（9092、9093、9094）
  • ZooKeeperポート（2181、2182、2183）
  • ログディレクトリ（/ tmp / kafka-logs-1、/ tmp / kafka-logs-2、/ tmp / kafka-logs-3）

broker.id=0
listeners=PLAINTEXT://:9092
zookeeper.connect=localhost:2181   
log.dirs=/tmp/kafka-logs-1

Znode構成：

vim config/zookeeper.properties

  • データのディレクトリ（/ tmp / zookeeper_1、/ tmp / zookeeper_2、/ tmp / zookeeper_3）
  • クライアントポート（2181、2182、2183）
  • クライアント接続の最大数と接続制限
  • ノード間のデータ交換用ポート（つまり、各ノードに他のノードの存在を通知します）

dataDir=/tmp/zookeeper_1
clientPort=2181
maxClientCnxns=60
initLimit=10
syncLimit=5
tickTime=2000
server.1=localhost:2888:3888
server.2=localhost:2889:3889
server.3=localhost:2890:3890

ZooKeeperノードのディレクトリを作成し、サービスファイルにノードIDを書き込みます。

mkdir -p /tmp/zookeeper_[1..3]
echo "1" >> /tmp/zookeeper_1/myid
echo "2" >> /tmp/zookeeper_2/myid
echo "3" >> /tmp/zookeeper_3/myid

アクセス権に問題がある場合は、ディレクトリとファイルで変更します。

sudo chmod 777 /tmp/zookeeper_1

Zノードとブローカーの実行


最初の2つのノード（kafka_server_1 /、kafka_server_2 /）では、ZooKeeperおよびKafka-serversのスクリプトを実行します。引数は対応する構成ファイルです。

sudo bin/zookeeper-server-start.sh config/zookeeper.properties
sudo bin/kafka-server-start.sh config/server.properties

3番目のノード（kafka_server_3 /）-ZooKeeperのみ。

サーバーを停止するためのスクリプト：

sudo bin/kafka-server-stop.sh
sudo bin/zookeeper-server-stop.sh

テーマ、コンソールプロデューサー、コンシューマーの作成


トピック（トピック）-特定のタイプのメッセージのストリーム。パーティションに分割されます（数量は--partitionキーで指定されます）。各パーティションは2つのサーバーで複製されます（--replication-factor）。--bootstrap-serverキーの後に、Kafkaブローカーのポートをカンマで区切って指定します。--topicスイッチは、トピックの名前を指定します。

sudo bin/kafka-topics.sh --create --bootstrap-server localhost:9092,localhost:9093 --replication-factor 2 --partitions 2 --topic TestTopic

ポートのトピックのリストを見つけるには、各トピックに関する情報を次のコマンドで実行できます。

sudo bin/kafka-topics.sh --list --zookeeper localhost:2181           
sudo bin/kafka-topics.sh --zookeeper localhost:2181 --describe --topic TestTopic

リーダー-パーティションのメインインスタンスを持つサーバー、レプリカ-情報が複製されるサーバー、ISR-リーダーに障害が発生した場合にリーダーの役割を果たすサーバー。

適切なスクリプトを使用して、コンソールプロデューサー、コンソールコンシューマーを作成します。

sudo bin/kafka-console-producer.sh --broker-list localhost:9092,localhost:9093 --topic TestTopic
sudo bin/kafka-console-consumer.sh --bootstrap-server localhost:9092,localhost:9093 --from-beginning --topic TestTopic

これにより、1つのメッセージブローカーで2番目のプロセスが失敗した場合に、受信者にメッセージを継続的に送信できます。

systemctlのzookeeperとkafkaのサービス


クラスターを簡単に開始できるように、systemctlでサービスを作成できます：zookeeper_1.service、zookeeper_2.service、zookeeper_3.service、kafka_1.service、kafka_2.service。

ファイル/etc/systemd/system/zookeeper_1.serviceを編集します（ディレクトリ/ home / userおよびuser userを必要なものに変更します）。

[Unit]
Requires=network.target remote-fs.target
After=network.target remote-fs.target

[Service]
Type=simple
User=user
ExecStart=home/user/kafka_server_1/bin/zookeeper-server-start.sh /home/user/kafka_server_1/config/zookeeper.properties
ExecStop=/home/user/kafka_server_1/bin/zookeeper-server-stop.sh
Restart=on-abnormal

[Install]
WantedBy=multi-user.target

サービスzookeeper_2.serviceとzookeeper_3.serviceの場合も同様です。

ファイル/etc/systemd/system/kafka_1.service：
[Unit]
Requires=zookeeper.service
After=zookeeper.service

[Service]
Type=simple
User=user
ExecStart=/bin/sh -c '/home/user/kafka_server_1/bin/kafka-server-start.sh /home/user/kafka_server_1/config/server.properties > /tmp/kafka-logs-1 2>&1'
ExecStop=/home/user/kafka_server_1/bin/kafka-server-stop.sh
Restart=on-abnormal

[Install]
WantedBy=multi-user.target

kafka_2.serviceも同様です。

サービスのアクティブ化と検証

systemctl daemon-reload
systemctl enable zookeeper_1.service #  
systemctl start zookeeper_1.service
sudo journalctl -u zookeeper_1.service #   
systemctl stop zookeeper_1.service

同じことがzookeeper_2.service、zookeeper_3.service、kafka_1.service、kafka_1.serviceにも当てはまります。
