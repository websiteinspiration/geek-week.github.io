There are few differences, but the debate around them does not subside due to those unprepared to abandon the usual style of developers. Therefore, we need a compromise. Here it is proposed to discuss it further.



A compromise solution might look like this:







What actually happened here? Just a little magic or trick. Because the real braces are still here. They were replaced by the editor with upper and lower underscores. In the source file, all the brackets are in place, but when reading the source file, your favorite IDE replaces the brackets with underscores according to the proposed algorithm.



As a result, we satisfy both conflicting camps. After all, what do the brackets supporters want to the right of the previous expression? They want to save an empty line, which would be occupied only by a curly bracket, which would show obvious wastefulness in relation to the final size of the screen.



And supporters of brackets at one level want to see the same boundaries of a block of program text. The symmetric arrangement of the brackets makes the code slimmer, highlights the block more explicitly, makes it easier to highlight with its eyes its beginning and end. True, at the same time the same extra line appears that I would like to remove, but beauty requires sacrifice!



Now let's think about how these wishes get along in the proposed approach. Lines are now saved not only at the top of the code block, but also at the bottom! Both camps should appreciate such a huge plus.



And on the other hand - beauty remains untouched! Visually, the code block is highlighted by the most symmetric of all the symmetric constructs. Eyes will just as easily find a beginning and an end. Well, there will simply be no one to object to this style, because the opponents will surely all move to the camp of the advocates of the approach.



Thus, we can only consider the algorithm for replacing curly braces with underscores.



Bracket Replacement Algorithm



Any decent IDE parses the source code of the program and selects pieces of text in it that correspond to the target language model. In particular, the text between curly braces usually corresponds to a model element called a ‚Äúcode block‚Äù. Therefore, most IDEs have already prepared everything for us - they have allocated blocks and are ready to give us information about them, if we are not too lazy to write the corresponding plug-in.



After the plugin receives the block, it skips all spaces, tabs, and line breaks above and below the block until it finds a bracket. Then he writes somewhere inside himself two meanings - the position of the opening and closing brackets. After that, the brackets can be cut and pasted underscores. And of course, if the brackets are not found, then nothing needs to be edited.



To insert an underline, the algorithm must detect a text shift after the start of the block. If there is no shift, then the author of the code did not intend to use any of the considered styles, and therefore, we will not insert any underscores to him (and we will not remove the brackets either).



But if there is a shift, then in the first line of the code block at the position before the shift (that is, with a shift equal to the shift of the text above) we insert the upper underline. We do the same with the last line of the block - put the underscore in front of it. That's the whole replacement.



But there remains little complexity in the form of a seamless combination of the editing process with the display of replacements in the form of underlines.



Major changes first. If the block is deleted together with both brackets, then nothing needs to be done, the editor will display everything correctly. If a block is deleted without brackets, but in its entirety, or if a part of a block is deleted with one of the brackets, then we restore the undeleted bracket (brackets) and give further creativity to the author of the code.



Now the changes are smaller. If something is corrected inside the block, then again nothing needs to be done. But if a new line is added to the block, it would be most logical to expect the developer to position the cursor at the end of the last line and press enter. After that, we wait for the first character to be entered and remove the underscore from the beginning of the last line and insert it at the beginning of the next (still empty). Then the developer writes the text after underlining and a certain set of spaces (or tabs), indicating the indentation of the block.



If the developer wants to finish the block, then after inserting a new line, he presses the backspace button , which gives us a signal to restore the underline in the line above and delete it in the current line. Thus, you can shift any number of indents, passing into an increasingly external program block. In this case, the underscores will be offset, corresponding to the block in which the last line is located.



We derive general rules, the observance of which will make it quite simple to make replacements.



While we are in a block of text, including the extreme lines, we do not need to do anything. When we press enter anywhere on the last line except the end, we need to transfer the underline. If enter is pressed at the end of the line, we are waiting for the character to be entered. When we press backspace in the last line at the underline position - we need to move the current underline one line higher. And of course, with all operations, you must remember to insert and remove the brackets in the text that will be saved to disk.



That's the whole algorithm. It remains to wait for the plugin experts to give us all such a nice gift. Well, and after them the language specification will also catch up, because how can it not correspond to what is convenient, and even used by so many?



And finally, about the overscore symbol , which I conditionally called "top underscore" here. Unfortunately, it is not on the standard keyboard, and therefore there is no way to enter it directly without the help of an editor. That is why the help of the IDE is important. Although it is theoretically possible to enter a Unicode sequence (\ u00af = ¬Ø), which some editors automatically convert to the overscore characterbut still doing it every time we need a bracket would be a mockery of the developers. Therefore, plugins are needed, well, changes in the specifications of languages.



That's it, we are waiting for urgent updates of language specifications and a lot of convenient plug-ins for all possible IDEs.
