茂みの周り


ドキュメントでの作業に関して開発者が利用できるものを見てみましょう。

Pythonは、モジュール、クラス、関数のドキュメント行をサポートしています。それは良いです。コメントは、ドキュメントに使用されることもあります。doxygenやsphinx

などのドキュメントジェネレータがあります。それも良いです。 ただし、もう少し必要な場合もあります。関数またはクラスの記述が大きくなりすぎ、ドキュメントの行（またはコメント）が不必要に「膨らむ」場合があります。非実行可能行の数は、実行可能行の数を抑制し始めます。これにより、コードがぼやけて読み取りが困難になる可能性があります。別の問題は、さまざまなグラフです。完成した画像をテキストに挿入することはできません。挿入できるのはチャートの説明のみです（例：Plant UML）

ダイアグラムですが、この場合でも、レンダリングを提供する必要があります。これは、プログラムテキストで直接行うことは困難です。

この問題の解決策は簡単です。長い説明を別のファイルに保存し、ドキュメントにドキュメントがあることをコードに残します。 1つのファイルを適切なタイミングで適切にレンダリングできます。また、コード内のマークはインタラクティブであることが望ましいです。つまり、マウスを1回クリックするだけでドキュメントに移行できます。原則として、プログラムのテキスト内のマーキングはより広く考えることができます。URLのほか、ソーステキストと画像の別のファイル内のどこか、およびドキュメント付きの外部ファイルにすることができます。

バックリンクを想像するのは簡単です。ドキュメントがプログラムのソースコードからのクラス、または特定のコードブロックに言及しているとしましょう。この場合、目的のコードにすばやく移行できると便利です。

考慮すべきもう1つの重要なシナリオ。開発中に外部ドキュメントファイルを作成したいとします。ドキュメントを作成する正当な衝動を抑えることができる不要な障壁がないように、組織の質問-どこに保存するか、何を呼び出すか、ドキュメントへのリンクを配置する方法-をできるだけ自動的に解決することが非常に望ましいです。

個別の議論は、ドキュメントファイルの形式の問題に値します。最近、マークダウンが批判されているにもかかわらず、人気が高まっています（たとえば、こちら）。プロジェクトページに入ると、githubが自動的にマークダウンドキュメントを表示するとします。これは非常に便利です。その際、ファイルの名前と場所について、非常にシンプルで直感的な合意に従う必要があります。マークダウンを操作するためのツールも豊富であるため、この形式が適しています。

Markdown


Codimension IDEはqutepartをテキストエディターのコンポーネントとして使用し、qutepartはそのままでマークダウンをサポートします。構文の強調表示は既に行われています。自動レンダリングでは、Pythonファイルと同じ方法を使用できます。メインフィールドは2つの部分に分かれています。左側はマークダウンテキスト、右側はレンダリングです。


自動レンダリングによるマークダウンの編集

もちろんレンダリングは自動的に行われます。 IDEはテキスト編集の一時停止を定義し、必要に応じてレンダリングを更新します。マークダウンビューでは、テキストエディターの左側が抑制されます。レンダリングにmistune

ライブラリを使用すると便利でした、テキストをHTMLにすばやく変換できます。準備が整ったHTMLがQ​​Tコンポーネントに送信されて表示されます。ミスチューンライブラリは、plantUMLダイアグラムの説明に認識を追加するのに十分な柔軟性も備えています。チャートは、適切な言語のコードブロックとして追加されます。次に例を示します。


チャート認識plantUML

plantUMLは、さまざまなタイプのチャートをサポートしています。以下のために余次元、図の説明であること、透明であり、それが分析されていないが、plantUMLがそのまま送信されます。したがって、サポートされているすべてのタイプがビューポートに表示されます。執筆時点では、plantUMLは、さまざまなタイプの図の次の開始/終了タグをサポートしています。

  • @startuml / @enduml
  • @startgantt / @endgantt
  • @startsalt / @endsalt
  • @startmindmap / @endmindmap
  • @startwbs / @endwbs
  • @startditaa / @endditaa
  • @startjcckit / @endjcckit

編集可能なマークダウンファイルには、ネットワークからダウンロードした画像だけでなく、いくつかの図が含まれている場合があります。頻繁な状況は、テキストの変更がほとんどなく、グラフィックリソースが変更されていない場合です。毎回同じものを汲み出してダイアグラムを再生成しないようにするために、同様のリソースのキャッシュを実装する必要がありました。

QTライブラリから、HTMLをサポートするQTextBrowserコンポーネントが使用されます。残念ながら、サポートされているHTMLの方言は制限されているため、最終的な結果を完全にすることはほとんど不可能です。おそらく、この欠陥は将来修正される可能性があります。

リンク上のマウスクリックの処理には、介入も必要でした。次の一部がリンクに表示される場合があります。

  • 外部リソース（http：//またはhttps：//で始まる）
  • プロジェクトのルートまたは現在のファイルからの相対パスを持つファイル（プロジェクトディレクトリがファイルシステム内を移動するときに便利です）
  • ファイルcの絶対パス

また、それがファイルの場合は、このファイル内のアンカーのIDまたは行番号のいずれか（オプションの要素）を指定できます。どちらの場合も、ファイルを目的の場所にスクロールするために追加情報が使用されます。

グラフィック要素


たとえばグループの場合のように、新しいCMLコメントを使用してグラフィック要素を入力することは理にかなっています。CMLは、新しいグラフィック要素にも必要な属性をサポートしています。

# cml 1 doc ....

ドキュメンテーションへのリンクは、解説に似ています。プログラムの実行可能な行ではありませんが、いくつかのコンテキストに関する追加情報を提供します。Codimension IDEいくつかのタイプのコメントを認識します：独立、リーディング、ラテラル。ドキュメンテーションについては、コメントの場合と同じ方法で独立した主要なグラフィック要素を維持するのが賢明なようです。ドキュメントのside要素と混同されています。ポイントは、複数行のコードブロックに対してサイドコメントがどのようにレンダリングされるかです。ブロックの右側に長方形が描画され、サイドコメントのすべての行をカバーし、一致する行番号をサポートします。また、CMLドキュメントがサイドコメントの途中で会った場合の対処方法は完全には明確ではありません。一方、リンクは依然として他の場所につながるため、サイドCMLドキュメントのサポートは冗長に見えます。コードブロックの特定の行のコンテキストは非常に狭いです。関数とクラスのサイドCMLドキュメントのサポートも冗長に思われます。したがって、この段階では、独立した主要なCMLドキュメントのみが実装されます。将来的にラテラルCMLドキュメントをサポートする合理的な必要性があり、それらを表示する方法が適切な場合は、この機能を追加できることに注意してください。

グラフィック要素に何を表示する必要があるかを説明しましょう。明らかに、要素のテキストが必要です。テキストは開発者によって設定され、リンクが何を指しているかを示します。ワンクリックでドキュメントへの移行を提供することが望ましいと前述しました。他の要素の動作間の連続性を尊重する必要があるため、グラフィック要素のテキストは適切ではありません。マウスをクリックすると、グラフィック要素が選択されます。ただし、問題の解決は簡単です。たとえば、テキストの左側にアイコンを追加できます。アイコンをクリックすると、遷移が発生します。

これで、CML docコメントの属性のリストが明確になりました。

  • link-別の場所へのリンク
  • anchor-他の場所からのリンクのアンカー識別子
  • タイトル-グラフィック要素に表示するテキスト
  • bg、fg、border-特別な方法で強調表示する必要がある場合は、要素の個々の色

2つの属性linkおよびanchorの少なくとも1つが存在している必要があります。それらがなければ、そのようなCML docコメントはあまり意味がありません。その他の属性はオプションです。テキストはまったくない場合があり、色は標準である場合があります。

独立および先行のCMLドキュメントコメント

先頭と独立の要素の違いは、コネクタが特定のブロックにつながるか、ブロック間のコネクタにつながるかだけです。

マウスカーソルがアイコン上にあり、要素にリンク属性がある場合、カーソルの形状が変化し、クリックが進むことを示します。

グラフィック要素は、コンテキストメニューもサポートしています。色を含む要素の属性を表示または編集するためのオプションと、要素を削除するためのオプションを使用できます。

その他のグラフィック要素


コードのグラフィカル表現の他のほとんどすべての要素には、ドキュメントがある場合があります（例外は、コメントとドキュメント文字列など）。したがって、コンテキストメニューからドキュメントアイテムを作成する方法があります。


ドキュメントを操作するためのコンテキストメニュー

オプションは2つだけです。最初の「ドキュメントリンクの追加/編集/アンカー...」では、リンク、アンカー、タイトルの属性を手動で入力するためのモーダルダイアログが表示されます。ここで、開発者はリンクをどこに送信するか、どこにファイルを配置するかなど、完全に自由です。

2番目のオプションはより興味深いものです。実行されたアクション「ドキュメントファイルを作成し、リンクを追加して編集用に開く」のため、アイテムの名前は長くなります。すべてのアクションは入力なしで自動的に実行されるため、組織の問題をすばやく解決できます。

  • , ( )
  • # cml 1 doc ..., . doc< >. : See documentation
  • ,
  • , , , , # cml 1 doc ... .

ファイルに関する決定は、プロジェクトがロードされているかどうかによって異なります。ロードされると、プロジェクトのルートにdocサブディレクトリが作成され、次にプロジェクトのルートからソースファイルへの相対パスが作成されます。ファイル名は、拡張子を.mdに置き換えて形成されます。たとえば、プロジェクトが

/home/me/myProject

プロジェクトファイルのドキュメントが追加されます

/home/me/myProject/src/utils.py

ドキュメントファイルが作成されます

/home/me/myProject/doc/src/utils.md

したがって、ソースコードのドキュメントの構造はソースコードの構造を繰り返し、プロジェクトのdocサブディレクトリをざっと見ても直感的なナビゲーションには十分です。このスキームは、デフォルトの動作には妥当であるように思われ、ドキュメントでの作業を高速化します。

プロジェクトが読み込まれていない場合は、ファイルの横にdocサブディレクトリが作成され、その中に拡張子が.mdに置き換えられたファイルが作成されます。ただし、そのようなシナリオはありそうにありません。ドキュメンテーションについて話している場合、それはほぼ確実にプロジェクトでの作業であり、個別のファイルではありません。

もちろん、# cml 1 doc ...テキストエディターまたはグラフィック表現のダイアログで編集することにより、いつでも自動生成された要素を変更できます。

プロジェクトドキュメントの開始点


なぜドキュメントの出発点が必要なのですか？私の意見では、最初にプロジェクトを開いた人がプロジェクトに参加するプロセスを容易にすることができます。たとえば、開発チームの構成を拡張する場合や、プロジェクトのユーザー向けです。プロジェクトにドキュメントがあり、それが多数のファイルで構成されていると想像してください。どこから始めますか？文書はどのように構成されていますか？それらを推測してからチェックする代わりに、推奨されるエントリポイントを明示的に示す方が便利です。

ここでは、githubアプローチを少し拡張して、デフォルトと特定のドキュメント編成オプションの両方が機能するようにすることができます。何も言われていない場合は、プロジェクトのルートでREADME.mdファイルを探します。また、プロジェクトのプロパティは、ドキュメントの開始点の特定のファイルを示す別のフィールドで展開できます。

プロジェクトドキュメントを開くには、2つのオプションがあります。IDEのメインウィンドウのツールバーに、マークダウンアイコンの付いたボタンが追加されました。クリックすると、ドキュメントファイルが表示モードで開かれます（利用可能な場合）。ファイルがない場合は、ドキュメントファイルの場所が提供され、新しいタブがテキスト付きの編集モードで開きます-スタブ。2番目のオプションは、開いているタブが他にない場合に表示されるCodimensionウェルカムページへのリンクです。付随するテキストには、プロジェクトドキュメントに関する単語と、ドキュメントが見つかった場合のリンクが含まれます。初めてプロジェクトを開いたユーザーに表示されるのは、このウェルカムページです。
