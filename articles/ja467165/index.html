Kotlin / Nativeには、iOSとAndroid間のコード共有という特定のアプリケーションがあります。これだけである種の興味が生まれます。 Scala Nativeにはこれがなく、しがみつくことは何もありません。 ScalaコードとTensorflowのような高速計算を組み合わせる必要がある場合、Scala Nativeを計算に使用できます。ただし、Scala Nativeは、ScalaとNativeの間で互いに安全に呼び出したり、共通のコードをいじったりできるという意味ではありません。

Kotlinでは、同じコードを作成してコンパイルすることが急増しています。これは、POSIXを使用していないという設計の意味での作業の地獄ですが、JVMでなくてもJVMで同じように機能するある種の中間層ライブラリです。そのようなライブラリーがない場合は、それで終わりです。この場所ではコードが異なります。コードが異なる場合、どのように保守するのですか？さまざまなエンドポイントをコンパイル、デプロイする必要があるライブラリをデプロイする方法、ビルドシステム、依存関係もありますか？あなたが掘るなら、質問-奈落の底。 Scala Nativeを開発する1人ですべてに答えることはできません。アプリケーションは別の話であり、これについては私のレポートを見ることができます。

ある程度、準備ができていました。最初に、私がチームを辞めると、コリャトロピンが私の代わりになることに気づきました。彼は現在チームを率いています。私は準備をしていました、そして私はそれを独特の方法で準備しました、私は彼が私の場所に立つことができることを理解しました。 2番目-すべてのサブシステムをゆっくりと分散し、それらを拒否しました。プログラムする必要はもうありませんでした。

Kotlin、Android、そして起こったことすべて、そして今やマルチプラットフォームで、それがすべてです。 Kotlinは1つのAndroid、1つのニッチに固執せず、さらに進んだ。

Java 8以降、これらの人々の流れは鈍化しました。彼らはScalaやKotlinには行かず、ラムダを得ました。これで開始するには十分であり、最高のものはもはや興味がありません。腰を下ろして考えてみると、規模が違うことに気づきました。

私の頭の中では、Scalaプラグインほど大きくない規模のチームに切り替えました。当時、ユーザーは10万人、Kotlinは4万人でした。完全に異なるスケール。面白くて、事前に教えてくれなかったことがわかりました。試してみるのが面白くて、野心的で野心的な目標ではなかったので、コトリンに行くことにしました。

一般に、Scalaはプログラミングに適しています。 Scalaでの9年間のプログラミングの後にKotlinを書き始めると、手がなくなります。これにはいくつかの悲しみがあります。

私がCTOだったらどうしますか？ Kotlinでの簡単なタスクの場合、チームをまとめる方が簡単でしょう。それはより安くまたはより効率的でしょう。私はScalaプログラマーをより安く採用することができたかもしれませんが、彼らがこのScalaでひどいことをするので、私はより悪い結果を得たでしょう。労働市場でそれが難しい場合でも、Javaプログラマーの採用を開始します。Kotlinの方がはるかに簡単にトレーニングできます。これは、真剣な技術的な事柄がない限り、新しいプロジェクトを開始するという観点からです。

たとえば銀行業界では、Scalaを使用する理由があります。 Kotlinはそれらを解決できません。 Scalaでの同じメタプログラミングははるかに便利です。

このトピックについてよく考えると、Scalaで優れたインフラストラクチャーを作成できますが、Kotlinでは不可能です。同じ注釈プロセッサがあります。いつかそれは可能になるでしょうが、数年の観点からのみです。

ジェイソンと私は適応に多くの時間を費やしました。ブートストラッププロジェクトを実行できるように、コンパイラでプログラミングして、少なくともこれらの複雑なファイルにコードを入力できるようにします。最初はすべて赤でしたが、私はすべてを緑に変えるのに多くの時間を費やしました。コンパイラのフロントエンドが緑色であるか、エラーがほとんどないことを確認するテストも行いました。そして、それに応じて、コンパイラチームは完全にIDEAに切り替えました。

アレクサンダー：流出にはいくつかの理由があります。最初に、IDEAはコミュニティバージョンをリリースしました。つまり、無料バージョンのユーザーにScalaプラグインをインストールできます。私たちは当初からオープンソースでしたが、私たちが始めたときはオープンソースで独占的でした。しかし、それは有料IDEにインストールされました。このため、無料で使用できる標準ツールが必要だったため、Eclipseサポートを作成しました。したがって、私たちも自由になった。

2番目-プラットフォームとしてのEclipseは、14年に衰退し始めました。彼は死ななかった、と言うのはばかげていますが、資金調達は突然破綻しました。 Eclipse Foundationはより少ない資金を集め始め、そのような状況でプラットフォームを開発する方法は明確ではありません。

3番目の理由は、高速IDEを作成できなかったことです。プレゼンテーションコンパイラそれはゆっくりと機能しました。ある種の機能について話した場合、それは一般的にはトリネットであり、リファクタリングは非常に困難です。コンパイラーでは、最適化の観点から、ツリーは一方向に構築され、親を取ることはできません。親ノードを見つけるには、上からファイルを取得し、それらに移動して調べる必要があります-ひどい大騒ぎ。

4番目の理由は、コンパイラーの表示で機能することです。これは分離後に得られる木です。エディターは前文で記述されており、プレゼンテーションコンパイラーでは、通常、どのようなコードかは明確ではありません。これを使用してリファクタリングを整理する方法は不明です。この前向きな発言をどのように抽出するのですか？したがって、機能がゆっくり登場しました。

同時に、フロントエンドは悲惨で、うまく機能しませんでした。しかし、彼が少なくともそのようになるためには、彼はひどいことをしなければなりませんでした。 forメソッド{i <- 0 to 10} println(i)は、言語の80％をサポートする必要がありました。これは分離されており、の暗黙的な変換Range - 1 to 10のパスに沿ってmap、暗黙的なパラメーターであるの暗黙的な変換がありCanBuildFromます。さらに、コレクションとラムダのクレイジーな階層がまだあります。パラメータが不明な他の匿名のラムダ-完全な缶。この例が機能するようになったら、フロントエンドの80％で機能することを理解する必要があります。したがって、貧弱なフロントエンドであっても、信じられないほどの量のScala機能をサポートしていました。

現在IDEAは多くの言語コンバーターを作成しています。それらは奇妙ですが異なります、そしてそれがすべて始まったとき誰も覚えていません。 2009年に、私は会議に行って、そこでショックを受けました。JavaからScalaに変換してみませんか？セミコロンに飽きてタイプを削除したり展開したりすることについて話し合いました-どういうわけか移動する必要があります。私は座って、夕方に千行を書きました-ある言語から別の言語への簡単な変換器。

しかし、私はさらに進んで、この機能を見つけやすくする方法を見つけました。Javaコードにコピーするときは、セミコロンを挿入したり削除したりする予定です。これは、発見に最適な機能を作成する場合の私のお気に入りのケースです。あなたは彼女を必要としませんが、あなたは彼女について知りません。必要なときにすぐに入手できます。これはAppleスタイルのようなもので、シンプルで簡単です。したがって、それ以来、多くのコンバーターがJetBrainsによって作成されており、広告機能はそれらが挿入物で提供されることです。しかし、はい、私はそれを思いつきました、これは私の誇りです。
