Spring BootでのMQ JMSアプリケーション開発



Java開発者は、JMSインターフェースを使用する場合、多くの場合、Spring Frameworkでの作業を好みます。 Springは、一般的なパターンのテンプレートを提供することにより、新しいアプリケーションのコーディングを簡略化でき、MQ JMSクラスで長年にわたって使用されてきました。 Spring JmsTemplateクラスは重要なインターフェースですが、それでもコードの構成と依存関係に依存しています。

Spring Frameworkには、さまざまなニーズに対応するさまざまなモジュールが含まれています。そのようなコンポーネントの1つがSpring Bootです。 Spring Bootスターターは、特定のテクノロジーを使用するために必要なすべての依存関係と自動構成ライブラリーを便利に引き出します。これにより、JmsTemplateなどのクラスを直接操作するよりも速く、新しいアプリケーションとテクノロジーを簡単に使い始めることができます。では、MQアプリケーションがこのように簡単にアクセスできるようにするにはどうすればよいでしょうか。@middle_java

年12月6日更新11コメントこの記事では、Maven Central Repository Java からMQクラスを直接ダウンロードする方法について説明しました。そして、これを使用してMQ用のSpring Boot Starterを作成します。GitHubからモジュールの完全なソースコードをダウンロードできます。

MQ Spring Bootの開始


まず、実行中のキューマネージャが必要です。次に、このキューマネージャにアクセスするアプリケーションを作成します。

コンテナーでのキュー・マネージャーの開始


すぐに開始するには、サーバープロセスを実行する開発者向けのIBM MQコンテナーを使用できます。このコンテナを起動すると、デフォルトでいくつかのオブジェクトが作成され、それらの定義は自動構成モジュールに認識されます。デフォルト値は、コンテナーキューマネージャーが必要とする値に対応しています。

つまり、Docker環境を使用してキューマネージャーを起動し、追加の設定なしでそれに接続できます。次のコマンドを使用して、Linuxでコンテナを起動できます。

  docker run --env LICENSE=accept --env MQ_QMGR_NAME=QM1 \
             --publish 1414:1414 \
             --publish 9443:9443 \
             --detach \
             ibmcom/mq

応用例


これは、「Spring JMS Getting Started Guide」のJMSアプリケーションモデルと一致するアプローチを使用します。このプロジェクトをEclipseワークスペースにインポートする方法に従い、gradleビルドプロセスを使用しましたが、mavenベースのプロセスも機能します。演習として、Eclipse IDEではなくコマンドラインからMaven設定を使用しました。
例の同じコードは、最小限の変更でMQで動作します。

  • IBM MQパッケージを指すように、依存関係リストのソースメッセージングプロバイダーを変更します。
  • Application.javaのキュー名（例では「メールボックス」を使用）を「DEV.QUEUE.1」に変更します。これは、DockerコンテナのDeveloper構成によって事前定義されたキューです。デフォルトのDeveloper構成パラメーターを使用しない場合は、キュー名を既存のキューと同じに設定するか、対応する新しいキューを定義する必要があります。

コンパイルを制御する 変更されたbuild.gradleファイルには、次のものが含まれています。

  dependencies {
    compile("com.ibm.mq:mq-jms-spring-boot-starter:+")
    compile("com.fasterxml.jackson.core:jackson-databind")
  }

Mavenを使用する場合、pom.xmlの対応するセクションは次のとおりです。

<dependencies>
  <dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
  </dependency>
  <dependency>
    <groupId>com.ibm.mq</groupId>
    <artifactId>mq-jms-spring-boot-starter</artifactId>
    <version>0.0.3</version>
  </dependency>
</dependencies>

実際の唯一の違いは、MQ Spring Bootモジュールの特定のバージョンがMaven構成で指定されていることです。

デフォルトの構成


MQブートスターター構成属性名は、ibm.mqプレフィックスで始まります。デフォルトの属性

  ibm.mq.queueManager=QM1
  ibm.mq.channel=DEV.ADMIN.SVRCONN
  ibm.mq.connName=localhost(1414)
  ibm.mq.user=admin
  ibm.mq.password=passw0rd

これらは、Dockerコンテナーのデフォルト値に対応しています。

高度な構成オプション


使用したい実行中のMQキューマネージャがすでにある場合は、オーバーライドされた値を指定することで、デフォルトの設定を簡単に変更できます。キュー・マネージャーは、ローカル・サービスの場合もあれば、IBM Cloudでホストされるサービスの場合もあります。Javaプロジェクトのapplication.propertiesファイルは、これらの属性をプロジェクトから除外する1つの方法です。設定は、コードを変更せずに有効になります。

キューマネージャ属性の名前を指定します。

ibm.mq.queueManager=QM1

キュー・マネージャーへのクライアント接続の場合は、以下も指定する必要があります

ibm.mq.channel 
ibm.mq.connName

channelもconnNameも指定されていない場合は、ローカルキューマネージャが使用されていると見なされます。構成コンポーネントは、一部のTLS関連の属性もサポートしています。これらはREADMEファイルで詳細に説明されていますが、デフォルト値はnullです。インストールする必要があるかもしれません

ibm.mq.user 
ibm.mq.password=passw0rd

デフォルト値を上書きします。これらの属性を空白のままにして、キューマネージャーで許可されている場合は、認証なしでオペレーティングシステムのローカルユーザーIDを使用できます。application.properties

ファイルの内容の例：

  ibm.mq.queueManager=QM1
  ibm.mq.channel=SYSTEM.DEF.SVRCONN
  ibm.mq.connName=server.example.com(1414)
  ibm.mq.user=user1
  ibm.mq.password=passw0rd

その後、Spring Bootはキューマネージャーとの対話に使用できるConnectionFactoryを作成します。

調査結果


MQでSpring Boot Starterを使用すると、非常に迅速に開始できます。次に、最初のアプリケーションを起動した後、それを拡張して他のMQ機能を使用できます。試した後、ここまたはGitHubの問題のページでフィードバックを送信できます。
