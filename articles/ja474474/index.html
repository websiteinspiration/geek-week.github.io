サーバーおよびWebアプリケーションの負荷テストのオープンベンチマーク

これは、ハッカーニュースとGitHubの居住者の推奨に基づくユーティリティのコレクションです。リストには、Locust、Vegeta、Slow_cooker、k6、Siegeが含まれます。これらは、DICE、EA、およびブイアントのエンジニア、ならびにKubernetesおよびLoad Impactの開発者によって使用されます。これらのツールについて話しましょう。

Locust.io


サイトのストレステスト用ツール。すべてのスクリプトはPythonで記述されています。Flask上に構築された特別なWebインターフェースにより、リアルタイムで結果を監視できます。 Locustの作者はスイスのエンジニアで、その中にはコンピューターゲームの開発と出版に携わるDICEおよびEA企業の従業員がいます。

このツールは興味深い概念に基づいています。イナゴ（ "イナゴ"）は、昆虫（仮想ユーザー）の群れの動作をエミュレートし、テスト中にサイトを "攻撃"します。リクエストは、並列コンピューティングを編成するためのネットワークライブラリ-geventを使用して形成されます。以下は、プロジェクトの公式Webサイトで提供されている簡単なテストの例です。

from locust import HttpLocust, TaskSet, task

class WebsiteTasks(TaskSet):
    def on_start(self):
        self.client.post("/login", {
            "username": "test_user",
            "password": ""
        })

    @task
    def index(self):
        self.client.get("/")

    @task
    def about(self):
        self.client.get("/about/")

class WebsiteUser(HttpLocust):
    task_set = WebsiteTasks
    min_wait = 5000
    max_wait = 15000

Locustはリクエストライブラリを使用します。標準のPythonツール用のこのアドオンは、HTTPおよびSSLの操作を簡素化し、コードをより視覚的にします。ちなみに、リクエストドキュメントはLocustのテストをデバッグするためのチートシートとして使用できます。

このストレステストツールは、7年以上前から存在しています。この間、大規模なコミュニティが形成され、GitHub には1万を超えるスターがいます。 Locustは、BattlefieldシリーズのゲームのBattlelogネットワークのパフォーマンスを評価するために使用されました。Flaskフレームワークの作者であるArmin Ronacher は、このツールについて積極的に語りました。

イナゴの欠点の中で区別しますサイトの応答時間を推定する際のパフォーマンスの低下と定期的なエラー。このツールではグラフの作成方法もわかりませんが、CSVファイルの形式で結果をアップロードし、テーブルエディターでグラフをレンダリングすることで問題が解決します。

Locustを詳しく調べたい場合は、ツールのドキュメントに注意を払う必要があります。また、Python MeetupでWargamingのAlexei Romanov によるパフォーマンスを推奨することもできます。ユーザーの行動をエミュレートするスクリプトの作成方法について語っています。

野菜


Goで記述されたHTTPサービスをテストするためのGoコマンドラインユーティリティ。ライブラリとして接続して、独自の負荷テストツールを作成できます。 Vegetaは、Lyft、Uber、およびYelpが使用するソースコードを確認およびナビゲートするためのエンジンである、オープンソースのSourcegraphプラットフォームの作成者の1人によって開発されました。

ベジータはネットワークリソースの機能を評価し、設定された頻度でリクエストを「衝撃」します。たとえば、localhostを確認するには、次のコマンドを入力します。

echo "GET http://localhost/" | vegeta attack -duration=5s | tee results.bin | vegeta report

デフォルトでは、Vegetaは標準コマンド読み取りストリーム（stdin）で動作するため、テスト用のリソースはエコーを介して送信されます。期間パラメータは、テストの期間を示します。レポートはresults.binファイルに生成されます。ベジータはテキスト形式でレポートを生成しますが、グラフを描くことができます。次のコマンドで生成できます。

vegeta plot -title=Results results.bin > results-plot.html

大規模なコミュニティがVegetaを中心に形成され、GitHubには1万2,000の星があります。このツールは、Kubernetes開発者がプラットフォームのパフォーマンスを評価するためにも使用されました。Vegetaは、数千のノードのクラスターに対して毎秒約1000万のリクエストを生成しました。

Vegetaテストのすべての関数とフラグを説明するドキュメントは、GitHubのリポジトリにあります。あなたはまた、見つけることができますが、実行可能ファイルをプリコンパイル。

通常、負荷テストユーティリティは、できるだけ多くのリクエストを短時間で送信して、サーバーの機能をチェックします。 slow_cookerの作成者は、彼らのツールを使用すると、予測可能な負荷の下で長時間鉄の仕事を評価できると述べています。

浮力のある専門家は、その開発を使用して、Linkerdおよびnginxなどの他のサービスをテストします。楽器はかなり若いです-彼は約3歳です-それで、彼が大きなコミュニティを得るまで。しかし、状況は将来変化する可能性があります。たとえば、そのリポジトリは、航空券を検索する国際サービスであるSkysannerですでにフォークされています。

k6


クラウドでホストされているマイクロサービス、コンテナー、サイトの負荷および回帰テスト用のツール。 Goで書かれており、Load ImpactのJavaScript開発者によって書かれています-これは、サイトの「持続性」をテストするためのアプリケーションです。

テストロジックとすべての設定がJavaScriptで記述されている場合、k6の操作はコードモデルとしてのすべてに基づいています。スクリプトでは、個々のステップをグループにまとめることができます。これは、BDDの原則に従うことに慣れているユーザーにとって便利な場合があります。そのようなグループの例を次に示します。

import { group } from "k6";

export default function() {
  group("user flow: returning user", function() {
    group("visit homepage", function() {
      // load homepage resources
    });
    group("login", function() {
      // perform login
    });
  });
};

このツールでは、スクリプトを記録してグラフを作成することもできます。最後の関数はInfluxDBとGrafanaに実装されています。また、Jenkins、Circle CI、Team City、GitLabなどのCIシステムと統合されています。

ユーザーは、k6はリソース集中型ではなく、便利なAPIを持っていると言っています。ただし、いくつかの欠点があります。特に、k6 は WebSocketをサポートしていないため、分散システムでテストを実行できません。ただし、ハッカーニュースのテーマ別スレッドのk6開発者は、これらの機能は将来登場する予定であると述べています。

自分でk6の可能性について知りたい場合は、HNの住民は技術文書から始めることをお勧めします-それは詳細で例があります。ご不明な点がございましたら、公式フォーラムにご連絡ください。

包囲


Siegeでは、テストWebサーバーをロードできます。このユーティリティは、開発者が戦闘に近い状況でのコードのリソース消費を確認できるように、エンジニアのJeff Fulmerによって作成されました。 Siegeは、「siegeの下で」サーバーを保持しているかのように、つまりツールの名前であるかのように、多くのユーザーからのサイトへの継続的なアクセスストリームをエミュレートします。

テスト後、ユーティリティには、スキャン時間、1秒あたりのトランザクション数、スループット、成功したリクエストと失敗したリクエストの数、および応答コード200のリクエストの数が表示されます。Siegeによって生成されたレポートの例を次に示します。

SiegeはITコミュニティで非常に広まっています。たとえば、「NGINX High Performance」という本のセクション全体が、その助けを借りて負荷テストに専念しています。彼も一部のクラウドプロバイダーを使用します。

Siegeの欠点の1つは、テストパラメータを計算するための非標準の構文と自明ではない方法を区別できることです。たとえば、リダイレクトは成功したトランザクションと見なされるため、その数はリクエストの総数を超える可能性があります。Siegeを実際に試してみたい場合は、オンラインマニュアルをご覧ください。システムにはいくつかの「奇妙な点」があります。
