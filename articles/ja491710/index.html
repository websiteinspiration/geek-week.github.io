React、Express、MongoDBに基づいて構築されたアプリケーションのドッキング

この記事の著者は、その翻訳を本日公開していますが、React、Express、MongoDBベースのWebアプリケーションをDockerコンテナーにパックする方法について話したいと思います。ここでは、そのようなプロジェクトのファイルおよびフォルダー構造の形成、ファイルの作成、DockerfileおよびDocker Composeテクノロジーの使用の機能について検討します。



仕事の始まり


簡単にするために、クライアントとサーバーのパーツによって提示され、データベースに接続された動作中のアプリケーションがすでにあると仮定します。

クライアントとサーバーのコードが同じフォルダーにある場合が最適です。コードは1つのリポジトリに配置できますが、別のリポジトリに格納することもできます。この場合、コマンドを使用して、プロジェクトを1つのフォルダーに結合する必要がありますgit submodule。私はそれをやった。

React Application


ここでは、Create Reactアプリを使用して作成され、TypeScriptをサポートするように構成されたプロジェクトを使用しました。これは、いくつかの視覚要素を含むシンプルなブログです。

まず、Dockerfileルートディレクトリにファイルを作成しますclient。これを行うには、次のコマンドを実行します。

$ touch Dockerfile

ファイルを開き、以下のコマンドを入力します。すでに述べたように、私はTypeScriptアプリケーションで使用するので、最初にそれをビルドする必要があります。次に、何が起こったのかを調べて、すべて静的リソースの形式で展開する必要があります。これを実現するために、私はDockerイメージを構築する2段階のプロセスを使用します。

最初のステップは、Node.jsを使用してアプリケーションを構築することです。ベース画像としてアルパイン画像を使用しています。これは非常にコンパクトなイメージであり、コンテナーのサイズに有利に影響します。

FROM node:12-alpine as builder
WORKDIR /app
COPY package.json /app/package.json
RUN npm install --only=prod
COPY . /app
RUN npm run build

だから私たちのものを始めますDockerfile。最初にチームが来るnode:12-alpine as builder。次に、作業ディレクトリを設定します/app。この場合、これです。これにより、コンテナ内に新しいフォルダが作成されます。このコンテナフォルダにpackage.json、依存関係をコピーしてインストールします。次に/app、フォルダからすべてをコピーします/services/client。作業はプロジェクトの組み立てによって完了します。

次に、新しく作成したアセンブリのホスティングを整理する必要があります。これを行うには、NGINXを使用します。そして、再び、これはシステムのアルパインバージョンになります。これは、以前と同様に、スペースを節約するためです。

FROM nginx:1.16.0-alpine
COPY --from=builder /app/build /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

ここではnginx、前の手順で取得したプロジェクトアセンブリの結果がフォルダーにコピーされます。次に、ポートを開きます80。コンテナーが接続を待機するのは、このポート上です。ファイルの最後の行は、NGINXを起動するために使用されます。

アプリケーションのクライアント部分をドッキングするために必要なのはこれだけです。結果Dockerfileは次のようになります。

FROM node:12-alpine as build
WORKDIR /app
COPY package.json /app/package.json
RUN npm install --only=prod
COPY . /app
RUN npm run build
FROM nginx:1.16.0-alpine
COPY --from=build /app/build /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Express API


Express APIも非常にシンプルです。ここでは、エンドポイントを整理するために、RESTfulテクノロジーが使用されています。エンドポイントは、パブリケーションの作成、承認のサポート、およびその他の問題の解決に使用されます。まずDockerfile、ルートディレクトリに作成しapiます。以前と同じように行動します。

アプリケーションのサーバー側の開発中に、ES6の機能を使用しました。したがって、コードを実行するには、コードをコンパイルする必要があります。私はBabelを使用してコードを処理することにしました。ご想像のとおり、ここでも多段階の組み立てプロセスが使用されます。

FROM node:12-alpine as builder
WORKDIR /app
COPY package.json /app/package.json
RUN apk --no-cache add --virtual builds-deps build-base python
RUN npm install
COPY . /app
RUN npm run build

ここでのすべてDockerfileは、プロジェクトのクライアント部分で使用したものと非常に似ているため、詳細には触れません。ただし、1つの機能があります。

RUN apk --no-cache add --virtual builds-deps build-base python

パスワードをデータベースに保存する前に、bcryptを使用してパスワードをハッシュします。これは非常に人気のあるパッケージですが、アルパインベースのイメージでの使用にはいくつかの問題があります。ここで、次のようなエラーメッセージが表示される場合があります。

node-pre-gyp WARN Pre-built binaries not found for bcrypt@3.0.8 and node@12.16.1 (node-v72 ABI, musl) (falling back to source compile with node-gyp)
npm ERR! Failed at the bcrypt@3.0.8 install script.

これは広く知られている問題です。彼女の解決策は、npmパッケージをインストールする前に追加のパッケージとPythonをインストールすることです。

クライアントの場合のように、イメージを構築する次のステップは、前のステップで形成されたものを取り、Node.jsを使用して実行することです。

FROM node:12-alpine
WORKDIR /app
COPY --from=builder /app/dist /app
COPY package.json /app/package.json
RUN apk --no-cache add --virtual builds-deps build-base python
RUN npm install --only=prod
EXPOSE 8080 
USER node
CMD ["node", "index.js"]

ここには、プロジェクトが本番環境で機能するように設計されたパッケージのみをインストールするという機能があります。バベルはもう必要ありません-結局のところ、すべてがアセンブリの最初のステップでコンパイルされています。次に、8080アプリケーションのサーバー側がリクエストの到着を待機するポートを開き、Node.jsを実行します。

ここに要約がありDockerfileます：

FROM node:12-alpine as builder
WORKDIR /app
COPY package.json /app/package.json
RUN apk --no-cache add --virtual builds-deps build-base python
RUN npm install
COPY . /app
RUN npm run build
FROM node:12-alpine
WORKDIR /app
COPY --from=builder /app/dist /app
COPY package.json /app/package.json
RUN apk --no-cache add --virtual builds-deps build-base python
RUN npm install --only=prod
EXPOSE 8080 
USER node
CMD ["node", "index.js"]

Docker作成


作業の最後のフェーズは、コンテナーapiとclientMongoDBを含むコンテナーをもたらすことです。これを行うにdocker-compose.ymlは、親リポジトリのルートディレクトリにあるファイルを使用します。これは、この場所からDockerfileプロジェクトのクライアント部分とサーバー部分のファイルにアクセスできるためです。

ファイルを作成しますdocker-compose.yml。

$ touch docker-compose.yml

プロジェクトファイルの構造は次のようになります。


プロジェクトファイルの最終的な構造次のコマンドに

追加しdocker-compose.ymlます。

version: "3"
services:
  api:
    build: ./services/api
    ports:
      - "8080:8080"
    depends_on:
      - db
    container_name: blog-api
  client:
    build: ./services/client
    ports:
      - "80:80"
    container_name: blog-client
  db:
    image: mongo
    ports:
      - "27017:27017"
    container_name: blog-db

すべてが非常にシンプルに配置されています。 3つのサービスclientがapiありdbます。 MongoDBの選択肢はありませんDockerfile-Dockerはハブから適切なイメージをダウンロードし、そこからコンテナーを作成します。これは、データベースが空になることを意味しますが、初心者にはこれが適しています。

セクションapiとclientキーにbuildは、Dockerfile対応するサービスのファイルへのパス（ルートディレクトリapiとへのパス）が含まれるキーがありますclient。ファイルで割り当てられたコンテナーポートDockerfileは、Docker Composeがホストするネットワーク上で開かれます。これにより、アプリケーションが対話できるようになります。サービスを設定するときapiに、さらにキーが使用されますdepends_on。彼はDockerに、このサービスを開始する前に、コンテナーが完全に起動するまで待つ必要があると伝えていますdb。これにより、コンテナのエラーを防ぐことができapiます。

そして-MongoDBに関連するもう1つの小さなことを次に示します。バックエンドコードベースでは、データベース接続文字列を更新する必要があります。通常それは示しますlocalhost：

mongodb://localhost:27017/blog

ただし、Docker Composeテクノロジーを使用して、コンテナーの名前を指すようにする必要があります。

mongodb://blog-db:27017/blog

作業の最後のステップは、プロジェクトのルートフォルダー（ファイルがある場所docker-compose.yml）で次のコマンドを実行して、これらすべてを開始することです。

$ docker-compose up
