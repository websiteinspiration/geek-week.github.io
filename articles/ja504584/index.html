JS開発者がノードの代わりにDenoを使用する5つの理由

Node.jsの作成者Ryan Dahlは、Node.js。の問題の多くを解決するように設計された新しいプラットフォームをリリースしました。おそらく、このニュースに対するあなたの最初の反応は次のようになります。私はただ彼がいなかった...」私はこのニュースにまったく同じように反応しました。しかし、特にDenoについて少し理解した後、2020年にサーバー側JavaScript開発者が必要とするものをDenoと正確に呼ぶことができる理由を理解しました。 DenoとNode.jsを比較し、このプラットフォームが開発者に良い結果をもたらすことができる5つの主な理由を検討してください。特に、Denoのおかげで、以前にNode.jsを使用していた人たちが、より便利で最新のツールを自由に使えるようになるという事実について話しています。





1.最新のJavaScript機能：ESモジュール


ノード：

const moment = require("moment")

デノ：

import { moment } from "https://deno.land/x/moment/moment.ts"

私と同じように、React開発者であれば、Node.jsにパッケージをインポートするための構文が慣れているものとは異なることに長い間気づいていました。実際のところ、Node.jsプラットフォームはずっと前の2009年に作成されました。それ以来、JavaScriptには多くの新しいものが登場しています。言語の多くが改善されました。

React（およびDeno）は、最新の構文を使用してビューの依存関係をインポートしますimport package from 'package'。Node.jsはデザインを使用しますconst package = require(«package»)。

ESモジュールの構文を使用すると、開発者に2つの重大な利点がもたらされます。

  1. キーワードを使用すると、import必要なものだけをパッケージから選択してダウンロードできます。これはメモリの節約に役立ちます。
  2. キーワードを使用するとrequire、モジュールは同期して読み込まれます。また、メカニズムimportはモジュールを非同期モードでロードし、パフォーマンスを向上させます。

上記のコードスニペットを注意深く読むと、Denoの例では、パッケージがmomentURLからインポートされていることがわかります。そしてそれが、ノードに対するデノの次の利点へと私たちを導きます。

2.分散型パッケージストレージ


Denoを使用する場合、開発者はNPMに拘束されなくなりました。あなたは私を正しく理解しましたpackage.json。各パッケージはURLを使用してダウンロードされます。

Node.jsでは、パッケージを使用するために、まずNPMからインストールする必要があります。

npm i moment

次に、インストールが完了するのを待つ必要があります。その後、アプリケーションのパッケージを接続します。

const moment = require("moment")

さらに、誰かがNode.jsプロジェクトをローカルで実行する必要があるたびに、このプロジェクトのすべての依存関係をNPMからインストールする必要があります。

Deno環境では、パッケージはURLを使用してインポートされます。その結果、プロジェクトにライブラリが必要な場合momentは、URL https://deno.land/x/moment/moment.tsを使用してライブラリをインポートする必要があります。これは、見てきたように、次のようになります。

import { moment } from "https://deno.land/x/moment/moment.ts"

Denoには、Node.jsよりもパッケージの作業に大きな利点があります。これは、インストール後、各パッケージがハードドライブにキャッシュされるという事実にあります。つまり、パッケージは一度だけインストールされます。既に別のプロジェクトにインポートされているプロジェクトに依存関係をインポートする必要がある場合、対応するコードを再ロードする必要はありません。

3. DenoにはTypeScriptが組み込まれています


Node.jsでTypeScriptに書き込むには、作業環境をセットアップするかなり長いプロセスを実行する必要があります。TypeScriptのインストール、アップグレードpackage.json、モジュールでの型サポートの提供が必要です。

Denoでは、TypeScriptコードの操作性を確保するために、.ts代わりに拡張子を使用して、そのようなコードを含むファイルを保存するだけで十分.jsです。TypeScriptコンパイラはDenoに組み込まれています。

4.非同期関数以外のサポートを待つ


ノード：

const fakeData = async () => {
    const data = await fetch("https://jsonplaceholder.typicode.com/todos/1");
    const result = await data.json();
}

デノ：

const data = await fetch("https://jsonplaceholder.typicode.com/todos/1");
const result = await data.json();

Node.jsでは、キーワードawaitは、キーワードを使用して宣言された関数でのみ使用できasyncます。

Denoでは、awaitどこでも使用できるので、期待する結果を整理できます。await構文を非同期関数に入れることはオプションです。

ほとんどすべてのJavaScriptアプリケーションには、多くの非同期関数が含まれています。結果として、この改善によりコードがよりクリーンでシンプルになったと言えます。

5.ブラウザAPI（ウィンドウ、フェッチ）へのアクセス


JavaScriptを使用してHTTPリクエストを実行するには、Fetch APIを使用できます。Node.jsはブラウザAPIにアクセスできません。この例では、標準のプラットフォーム機能のみを使用してFetchにアクセスすることはできません。まず、適切なパッケージをインストールする必要があります。

npm i node-fetch

次に、それをコードにインポートする必要があります。

const fetch = require("node-fetch")

その後、対応する呼び出しを行うことができます。

そして、Denoはオブジェクトにアクセスできますwindow。つまり、同じFetch APIを使用するには、次のようにするだけです。

fetch("https://something.com")

これは、他のブラウザAPIの使用にも適用されます。それらを使用するために、追加のライブラリをインストールする必要はありません。

これをawait非同期関数の外部で使用する可能性と組み合わせると、DenoコードがNode.jsコードよりも簡単である理由と方法が明らかになります。

ノード：

const fetch = require("node-fetch")

const fakeData = async () => {
    const data = await fetch("https://jsonplaceholder.typicode.com/todos/1");
    const result = await data.json();
}

デノ：

const data = await fetch("https://jsonplaceholder.typicode.com/todos/1");
const result = await data.json();

結論：DenoにはNodeよりも他の利点があります


Denoには、Node.jsよりも多くの利点があります。たとえば、標準的な形式のDenoの方が安全です。 Deno環境では、Wasmコードを実行できます。多くの組み込みライブラリがあります。

この記事で説明するDenoの長所はすべて相互に関連しています。これらが一緒になって、Node.jsよりもモダンと呼ぶことができる環境、つまりバックエンドプロジェクトを開発するための環境を形成します。私はReact開発者です。Denoの機能は私のニーズと密接に関係しています。これで、サーバーコードを作成し、キーワードを使用できます。importどこにでも配置できますawait。特に何も設定しなくてもTypeScriptで記述でき、追加のパッケージをインストールせずにFetchを使用できます。私の仕事は簡単になっています。

Denoは最終的にNodeの代わりになるのでしょうか？多分。これにはおそらく数年かかる可能性があります。Node.jsのエコシステムは巨大であり、Denoがそのような規模に成長するには時間が必要です。

JavaScript開発者は、新しいプロジェクトですでにDenoを使用しています。したがって、今何か新しいことに取り掛かろうとしている場合は、間違いなくDenoを確認する必要があります。
