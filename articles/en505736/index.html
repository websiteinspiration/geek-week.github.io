Run, PHPUnit, run: how I optimized test execution time

From the very beginning of work on one of the web-projects, we strove for a high level of code coverage with tests, and at the initial stage of development I did not think about optimizing the speed of their execution. As a result, with the growth of the project, the ever greater coverage of its tests and the growth of the team, the test execution time increased from a few seconds to tens of minutes. And having quick tests can be just as important as the performance of the entire application.


How did I fight this and what happened in the end?


What came before the optimizations


The application is being developed in PHP 7.2 and Symfony 4.2. To write tests we use PHPUnit. Most of the tests are integration tests for the REST API.


Run the tests:


> bin/phpunit
...
Time: 20.17 minutes, Memory: 400.25 MB
OK (1494 tests, 5536 assertions)

All tests took about 20 minutes. This led to the following problems:


  • the automatic CI process began to be blocked for the duration of the tests, leading to the exhaustion of free CI runners,
  • developers were forced to line up in a virtual queue, waiting for the completion of CI pipelines,
  • for the duration of all tests, you could go locally to make a cup of coffee and leaf through the news, which only fueled the desire to simply ignore their launch.

#!/bin/sh

php_no_xdebug() {
  temporaryPath="$(mktemp -t php.XXXXXX).ini"

  # Using awk to ensure that files ending without newlines do not lead to configuration error
  php -i | grep "\.ini" | grep -o -e '\(/[a-z0-9._-]\+\)\+\.ini' | grep -v xdebug | xargs awk 'FNR==1{print ""}1' | grep -v xdebug >"$temporaryPath"

  php -n -c "$temporaryPath" "$@"
  rm -f "$temporaryPath"
}
