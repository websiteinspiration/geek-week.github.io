Ngrok v2プロトコルのリバースエンジニアリング

ngrokは、ユーザーのローカルコンピューターへのトンネルを作成できるサービスです。つまり、パブリックアドレスは予約されており、すべての呼び出しはローカルポートに転送されます。

残念ながら、2016年のクライアントのオープンソースバージョン（ngrok v1）のサポートは廃止されており、このサービスを使用するには、クローズドバージョン（ngrok v2）を実行する必要があります。この記事では、公式クライアントが使用するプロトコルを学習し、代替のオープンクライアントを作成するプロセスについて説明します。


しかし、それは必要ですか？Ngrokの代替

奇妙なことに、このサービスには代替手段がほとんどありません。具体的には、次の3つです。


  • serveo.net。同様の機能を提供しますが、カスタムクライアントではなく、SSHリバースポート転送を使用します。残念ながら、プロジェクトは現在閉鎖されています。

    フィッシングのため、Serveoは一時的に無効になっています。

    Serveoは、悪用をやめるのに役立ついくつかの新しい制限付きで、数日で戻ってきます。お待ち頂きまして、ありがとうございます！

  • localtunnel.me。これは、Hostヘッダーに基づいて配布されるHTTPトンネルのみを提供し、HTTPSの場合、データはサーバー上で復号化され、クリアテキストでクライアントアプリケーションに送信されます。プロジェクトサイトは現在利用できません。
  • pagekite.net。HTTPおよびTLSトンネルを提供します。30日間の試用期間後は、それ以上の使用には料金を支払う必要があります。

（PSコメントから、serveo.netと同様に、SSHポート転送を介してHTTPトンネルを提供するlocalhost.runがあることが示唆されました。ただし、サービスはngrokとは異なり、HTTPトンネルのみを提供します）

import sys

stream = sys.stdin.buffer

while True:
    chunk = stream.read(44)
    if not chunk: break
    assert len(chunk) == 44
    kind = chunk[:4].decode('ascii')
    assert kind in ('read', 'writ')
    str_id = hex(int.from_bytes(chunk[4:12], 'little'))
    l = int.from_bytes(chunk[20:28], 'little')
    n = int.from_bytes(chunk[36:], 'little') if kind == 'read' else l
    buf = stream.read(l)
    assert len(buf) == l
    if '--full-data' not in sys.argv: buf = buf[:n]
    print('((%r, %s, %d), (%r, %d))'%(kind, str_id, l, buf, n))
