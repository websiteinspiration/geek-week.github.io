Active Restoreのサービス、またはInnopolisの1つの産業プロジェクトの履歴

こんにちは、ハブル！私の名前はローマンです。今日は、イノポリス大学で、Acronis Active Restoreシステム用のテストベンチとサービスを開発した経緯についてお話ししたいと思います。これは、まもなく会社の製品ラインの一部になる予定です。大学が産業界のパートナーとどのように関係を築いているかに興味があるすべての人に、猫の下でフォローアップすることを勧めます。



Active Restoreの開発はアクロニスの内部で始まりましたが、私たちはイノポリス大学の学生として、教育産業プロジェクトの一環としてこのプロセスに参加しました。私の学芸員（そして現在は同僚）のDaulet Tumbaevは、そのアイデアとアーキテクチャについて、彼の投稿で既に書いています。今日は、イノポリスからのサービスの準備方法についてお話します。

それはすべて、夏の初めに始まりました。最初の学期に、IT企業が私たちのところにやって来て、実際の仕事のために彼らのアイデアを提供すると知らされていました。そして、2018年12月に15の異なるプロジェクトが提示され、月末に優先順位を設定して、誰が一番好きかを判断しました。

すべての学部生が、私たちが参加したい4つのプロジェクトを選択する必要があるフォームに記入しました。なぜ私が、なぜ正確にこれらのプロジェクトをやる気にさせることが必要でした。たとえば、私はすでにC / C ++でのシステムプログラミングと開発の経験があることを指摘しました。しかし、最も重要なことは、このプロジェクトによってスキルが向上し、成長し続けることができました。

2週間後、配属され、2学期の初めからプロジェクトへの取り組みが始まりました。チームが結成され、最初の会議でお互いの長所と短所を評価し、役割を割り当てました。

  • Roman RybkinはPython / C ++開発者です。
  • Eugene Ishutin-会社とのやり取りを担当するPython / C ++開発者。
  • Anastasia Rodionovaは、ドキュメントの作成を担当するPython / C ++開発者です。
  • Brandon Acosta-環境のセットアップ、実験とテストのためのスタンドの準備。

プロセスを開始するために必要な最初の2週間。お客様との連絡を確立し、プロジェクトの要件を正式化し、反復プロセスを開始し、作業環境をセットアップしました。

ところで、私たちが選択科目を始めたとき、顧客との私たちの仕事は本当に沸騰し始めました。事実は、アクロニスが選択した科目（だけでなく）でイノポリス大学をリードしていることです。カーネルチームの主要な開発者であるAlexey Kostyushkoは、リバースエンジニアリングとWindowsカーネルアーキテクチャとドライバーの2つのコースを継続的に教えています。私の知る限り、システムプログラミングとマルチスレッドコンピューティングのコースも将来計画されています。しかし、主なことは、これらのコースはすべて、学生が産業プロジェクトに対処できるように設計されていることです。彼らは主題領域の理解に真剣に取り組んで、それによってプロジェクトの作業を簡素化します。

このため、他のチームよりも精力的にスタートし、Acronis自体とのやり取りが密になっています。Alexey Kostyushkoは、製品所有者の役割で私たちの代理を務めました。彼から、対象分野で必要な知識を得ました。彼の選択科目のおかげで、私たちのハードスキルとコンピテンシーは非常に強く刺激され、私たちが直面するタスクを実行する準備が本当にできました。

考えからプロジェクトへ


すべてのチームの最初の月は、可能な限り困難でした。誰もが道に迷い、どこから始めればいいのか分からなかった-おそらく文書を使ったり、逆にコードに飛び込んだりするかもしれません。当初、矛盾するコメントは大学の学芸員やメンターと企業の代表から寄せられました。

すべてが（少なくとも私の頭の中で）所定の位置に収まると、大学のメンターがチームの内部関係を構築し、ドキュメントを準備するのに役立つことが明らかになりました。しかし、本当の進歩は、3月にDauletが到着したことです。私たちはただ座って、週末を通してプロジェクトに取り組みました。次に、プロジェクトの本質を再考し、再起動して、タスクの優先順位を再配布し、すぐに前進しました。実験を開始して（少し後で）、サービスを開発するために何が必要かを理解しました。その瞬間から、一般的な考えは明確な計画に変わりました。コードの実際の開発が始まり、2週間で、仮想マシン、実験の自動化とデータ収集に必要なサービスとコードを含む、テストベンチの最初のバージョンを開発しました。

注目に値するのは、産業プロジェクトと並行して、プロジェクトの有能なアーキテクチャの構築と品質管理の組織化に役立つトレーニングコースがあったことです。最初は、これらのタスクには週あたり70〜90％の時間がかかりましたが、結局のところ、開発プロセス中の問題を回避するために時間が必要でした。大学の目標は、私たちが開発プロセスを適切に構築することを学び、企業は顧客として結果にもっと関心を持つことでした。もちろん、これは多くの混乱をもたらしましたが、理論的スキルと実践的スキルを組み合わせるのに役立ちました。十分な複雑さと負荷が動機づけの存在を保証し、プロジェクトを成功に導きました。

最初は、私たちのチームの2人が純粋な開発に携わっていました。1人はドキュメントを引き継ぎ、もう1人は環境の設定に飛び込みました。しかし、後にさらに3人の学士が加わり、私たちは1つのチームになりました。大学は、3年生向けのテスト産業プロジェクトを立ち上げることを決定しました。チームを4人から7人に拡大すると、学士が開発に関連するタスクを簡単に実行できるため、プロセスが大幅に加速しました。 Ekaterina Levchenkoは、テストベンチ用のpythonコードとバッチスクリプトの作成を支援しました。 Ansat AbirovとRuslan Kimは開発者として行動し、アルゴリズムの選択と最適化に従事しました。

実験が開始された5月末までこの形式で作業しました。この時点で、学士のための産業プロジェクトは終了しました。そのうちの2人は、アクロニスのインターンシップを開始し、私たちと一緒に働き続けました。そのため、5月以降は、すでに6人1チームとして活動していた。

私たちの前は3学期でした。イノポリスでは学業はありません。選択科目は2つしかなく、残りの時間は産業プロジェクトに費やされました。サービスに関する作業が集中的に行われたのは3学期でした。開発プロセスは完全にレールに落ち、デモとレポートは定期的になりました。この形式では、1.5か月間作業し、7月末には開発のほぼ完成に近づきました。

技術詳細


最初に、ファイルシステムミニフィルタードライバー（ここで読むことができる）と適切に相互作用するサービスの要件が策定され、そのアーキテクチャが検討されました。さらなるコードサポートのシンプルさを念頭に置いて、すぐにモジュール式のアプローチを提供しました。私たちのサービスには、いくつかのマネージャー、エージェント、ハンドラーが含まれており、コーディングを始める前から、並列モードで作業する機能が提供されていました。

ただし、アクロニスの担当者とのミーティングでアーキテクチャについて話し合った後、最初に実験を行い、次にサービス自体を採用することにしました。その結果、開発にかかった時間はわずか2.5か月でした。残りの時間は、Windowsを実行できる最小限のファイルリストを見つける実験を行いました。実際のシステムでは、このファイルのセットはドライバーを使用して生成されますが、ドライバーの動作を確認するために、このセットを半分割法を使用して発見的に見つけることにしました。


実験のスタンド。


これを行うには、2つの仮想マシンからPythonでスタンドを組み立てます。1つはLinuxで動作し、2つ目はWindowsをロードしました。それらに対して2つのディスクが構成されました：仮想HD1と仮想HD2。両方のドライブは、LinuxがインストールされているVM1に接続されていました。HD1上のこの仮想マシンに、HD2を「損傷」したKillerアプリケーションがインストールされました。損傷とは、ディスクから一部のファイルが削除されることを指します。HD2は、Windowsで動作するVM2の起動ディスクでした。ディスクへの「損傷」の後、VM2を起動しようとしました。これが可能である場合、ディスクから削除されたファイルは実行する必要がないと見なされました。

このプロセスを自動化するために、ランダムにではなく、事前に考え抜かれたアプローチの一部として、ファイルを削除しようとしました。アルゴリズムは3つのステップで構成されていました。

  1. ファイルリストを半分に分割します。
  2. 半分のファイルの1つを削除します。
  3. システムを起動してみてください。システムが起動した場合は、削除したファイルを不要なもののリストに追加します。それ以外の場合は、ステップ1に戻ります。

まず、アルゴリズムをシミュレートすることにしました。ファイルシステムに1,000,000個のファイルがあるとします。この場合、重要なファイルが全体の約15％である場合に、重要なファイルの最も効果的な検索が行われました。


ハーフ除算方式。

最初は、実験に多くの問題がありました。2〜3週間、テストスタンドの準備が整いました。そしてさらに1〜1.5か月、バグをキャッチし、コードを追加し、スタンドを機能させるためにさまざまなトリックを適用する必要がありました。

最も難しいことは、ディスク操作のキャッシュに関連するバグを見つけることでした。実験は2日間機能し、シミュレーションよりも数倍速い非常に楽観的な結果が得られました。ただし、重要なファイルのテストは失敗し、システムは起動しませんでした。仮想マシンの強制シャットダウン中に、ファイルシステムによってキャッシュされた削除操作が実行されなかったため、ディスクが完全に消去されていないことがわかりました。その結果、アルゴリズムは誤った結果を受け取り、数日間、すべての畳み込みを実行してすべてを把握しました。

ある時点で、連続操作中にアルゴリズムがファイルシステムのセグメントの1つに埋め込まれ、同じファイルを削除しようとした（新しい結果を期待して）ことに気づきました。これは、削除の間隔を誤って選択しているときに、アルゴリズムがほとんどが必要な領域にあるときに発生しました。この時点で、再シャッフルファイルリストを追加することにしました。つまり、数回の反復で、ファイルのリストがシャッフルされました。これは、そのような棒からアルゴリズムをノックアウトするのに役立ちました。

すべての準備が整ったら、これら2つのVMを3日間起動しました。合計で約600回の繰り返しが行われ、そのうち20回以上の打ち上げが成功しました。この実験は、Windowsの実行に最適なファイルサイズを見つけるために、より強力なマシンだけでなく、長期間実行できることが明らかになりました。また、このプロセスをさらに加速するために、アルゴリズムを複数のマシンに分散させることができます

私たちの場合、Windowsに加えて、ディスク上にはPythonとサービスのみがありました。3日間で、ファイル数を7万から5万に減らすことに成功しました。ファイルのリストは28％だけ減りましたが、このアプローチが機能していることが明らかになり、OSをロードするために必要なファイルの最小セットを決定できるようになりました。

サービス構造


サービスの構造について少し触れましょう。メインのサービスモジュールはキューマネージャです。ドライバーからファイルのリストを取得するため、このリストからファイルを復元する必要があります。これを行うために、優先度のあるターンを作成しました。



順番に復元されるファイルのリストがあります。また、新しいアクセス要求が発生した場合、緊急に必要なファイルが優先的に復元されます。このため、キューの最初に、ユーザーが現在本当に必要とするファイルがあり、行の最後には、将来必要になる可能性のあるファイルがあります。しかし、ユーザーのアクティブな作業により、「異常なオブジェクトのキュー」と、現在復元されているファイルのリストを形成できます。さらに、検索操作はこれらのキューすべてに一度に適用されているはずです。悲しいかな、いくつかのファイルの優先順位を設定できるキューのそのような実装は見つかりませんでしたが、検索をサポートし、その場で優先順位を変更しました。既存のデータ構造に適応したくなかったため、自分で書いて、それを使用する機能をセットアップする必要がありました。



私たちのサービスは、最初にDauletが作業したドライバーと通信し、その後、ファイルの復元を担当するコンポーネントと通信する必要があります...したがって、最初に、リカバリシステムの独自の小さなエミュレーターを作成することにしました。サービスを復元してテストします。

合計で、通常モードとリカバリモードの2つの動作モードが提供されました。通常モードでは、ドライバーはOSの起動時に影響を受けるファイルのリストを送信します。次に、システムの実行中に、ドライバーはファイルに関するすべての操作を監視し、サービスに通知を送信します。次に、ファイルのリストを変更します。回復モードでは、ドライバーはシステムの回復が必要であることをサービスに通知します。サービスはファイルをキューに入れ、バックアップからファイルを要求するソフトウェアエージェントを実行し、回復プロセスを開始します。

卒業証書、仕事への招待、新しいプロジェクト


サービスの準備ができてテストが完了すると、プロジェクトの最後のアクティビティが完了しました。蓄積したすべてのアーティファクトを更新して構造化し、結果を顧客と大学に提示する必要がありました。同社にとって、これは私たちの卒業論文を持つ大学にとって、プロジェクトの実施に向けた別のステップでした。

プレゼンテーションに続いて、学生に提案が行われました。そして数週間後、私はアクロニスに勤務します。プロジェクトの結果、開発者たちは、ネイティブWindowsアプリケーションのレベルまでサービスを下げることでサービスをより効率的にすることが可能であると考えました。しかしそれについては次の記事で詳しく説明します。
