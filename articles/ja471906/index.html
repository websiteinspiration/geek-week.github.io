不適切な最適化の危険性

最大のパフォーマンスを得るためにシステムを最適化することになると、他人の慣習を無謀に適用すると、間違いを犯しやすくなります。そのような方法の1つは、ファイルシステムをマウントするときにnobarrierを指定することです。

このメモが生まれた方法


私はMail.Ru Cloud Solutionsのエンジニアとして働いており、主にユーザーの仮想マシンが置かれているブロックストレージの「周り」のあらゆる種類の問題を扱っています。そのため、実行中の仮想マシンのパフォーマンスと安定性に関連する興味深いケースがよく発生しますアプリケーション-特にデータベース。

原則として、「報告」する場合のほぼ半分の場合、同じことがわかります-nobarrierオプションでマウントされたファイルシステム。そして、「なぜこのオプションを書いたのですか」と尋ねると、ほとんどの場合、「より速いと言われた/より速いと読んだ/そのように設定された」という答えのオプションの1つを取得します-その後、丁寧かつ慎重に説明しようとしますそうする必要はありません。どうして？これはデータ損失への道のりにおける最初の自信のあるステップだからです。

短い遠足


ファイルシステム-構造は非常に複雑で負荷が高い。キャッシング中に最大のパフォーマンスを確保するために、キャッシングと並列記録が積極的に使用されます。したがって、データの一部はキャッシュに移動し、可能/必要な場合、または「オンデマンド」の場合は常に破棄されます。バリアは、すべてのキャッシュを強制的にディスクにフラッシュする特別な操作です。

データベースに関しては、クライアント（クライアントアプリケーション）に対して確認されたトランザクションが永続的であり、消失しないことを確認する必要があります。一方で、DBMSは独自のキャッシュを積極的に使用して最大のパフォーマンスを達成し、一貫性を確保します。ジャーナリングが使用されます-変更がログに書き込まれ、ログが「同期」され、変更がデータに書き込まれます（書き込まれると、キャッシュに入れられます）。ログがいっぱいになると、キャッシュ内のすべてのデータに対して強制的な同期が行われ、ログが再び読み込まれ始めます。

同期操作


同期が実行されると、オペレーティングシステムはページキャッシュをフラッシュするだけでなく、デフォルトですべてのディスクキャッシュをフラッシュするコマンドを送信します（おそらくこれを繰り返します）。流す。バッファをフラッシュする操作は「高価」であり、かなりの時間がかかります。ただし、ファイルシステムでは書き込みの順序が重要であるため、必要です。これに違反すると、（たとえば）突然の再起動が発生すると、ファイルの代わりにデータではなくガベージが含まれます-デバイスが原因でレコードを再注文することにしました。そして、フラッシュがすべてのキャッシュを強制的にフラッシュするとき-これにより、フラッシュ前が最初に書き込まれ、その後がフラッシュされます-つまり、「バリア」が作成され、エントリが「バリアの前」と「バリアの後」に分割されます（ここからおよび「バリア書き込み」という名前）-これにより、バリア後のレコードがバリア前のレコードよりも早く適用されないことを保証できます。

バリア効果なし


nobarrierオプションは、ファイルシステムの実行中に強制フラッシュの送信を無効にします。これは、レコードを並べ替えることができるという事実につながります-障害が発生した場合、ファイルシステム（および一般的なケースでは一般にデータ）に一貫性がない可能性があります-前の段落で記録の順序について述べたことを思い出してください。

なぜこのオプションが含まれているのですか？低コストのSSDの場合、フラッシュ操作は非常にコストがかかります。たとえば、フラッシュのない低コストのSSD（および「サーバー」として配置されている多くの高価なSSD）は、毎秒10〜2万の書き込み操作を実行し、フラッシュを有効にすると、1〜2千に低下します。このような場合、nobarrierはパフォーマンスを大幅に向上させ、上記のデータ整合性に対するリスクを引き起こします。

仮想環境


仮想マシンの場合-たとえば、Linuxハイパーバイザー上の仮想マシンの従来の構成について話している場合、QEMUがあります。これは、ゲストオペレーティングシステムのI / Oのエミュレーションを実際に担当するプロセスです。そして最も重要なのは、仮想マシンでファイルバックアップされていないディスクを使用する場合、そのような仮想ディスク（突然！？）のキャッシュは、対応するQEMUプロセスのアドレス空間のユーザー空間にあります。そして、このプロセスがクラッシュした場合（たとえば、SEGFAULT / SIGSEGVによると）、そのすべてのキャッシュがそれに伴って停止します。このようなブロックデバイスドライバーの例は、RBD（Ceph）ドライバーです。

たとえば、Cephを使用せず、iSCSI / FCを使用する場合でも、障害レベルは消えません。QEMUからホストオペレーティングシステム（ハイパーバイザー）に移行するだけです。ハイパーバイザーが落ちました-ページキャッシュが停止しました（これは、io = 'threads'とcache = 'writeback'またはcache = 'unsafe'の組み合わせに当てはまります）。おっとっと
