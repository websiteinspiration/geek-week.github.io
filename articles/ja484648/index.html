ユーザーインターフェイスに動的リストを実装する機能


現代の各アプリケーションのインターフェースには、オブジェクトのリストが含まれています。それらを使用する場合、ユーザーはソート、フィルタリング、エクスポートなどの同じタイプのアクションを必要とすることがよくあります。これらの操作の実装は、リストが「動的」になる可能性があるという事実により、しばしば複雑になります。この場合、データはサーバーからクライアントへだけでなく、データベースサーバーからアプリケーションサーバーへも必要に応じて読み取られます。

オープンで無料のlsFusionプラットフォームでは、すべてのリストはデフォルトで動的であり、数行のコードで任意のフォームに追加されます。この記事では、実装の技術的な詳細と、フォームのリストを操作するときにユーザーに自動的に提供されるインターフェースの機能について説明します。

生き物


lsFusionのフォームのリストは、OBJECTSステートメントによって追加されます。
OBJECTS i = Item
テーブルがフォームに追加され、Itemクラスのすべてのオブジェクトが行に含まれます。1つのリストに複数のオブジェクトを追加できます。例えば、
OBJECTS (i = Item, s = Stock)
この場合、テーブルにはItemクラスとStockクラスのオブジェクトのすべての可能なペアが表示されます。

列は、PROPERTIESステートメントを使用してリストに追加されます。
PROPERTIES name(i), name(s), quantityOnHand(i, s)
単純なオブジェクトの詳細を、グループ化、パーティション、再帰などの式と同様にテーブルに追加できます。

デフォルトでは、リストにはデータベース内のすべてのオブジェクトが表示されます。それらを制限するには、FILTERSステートメントを使用できます。
FILTERS quantityOnHand(i, s) > 0
フィルター条件では、フォーム上のオブジェクトに依存する任意の式を使用できます。

ナビゲーション


ユーザーがフォームを開くと、プラットフォームはテーブルのサイズに応じて、表示されるレコードの数を自動的に決定します。表示を簡略化するために、このようなレコードは50あると想定し、プラットフォームは常にクライアントとサーバーに150のレコードを保存します。この場合、現在アクティブなオブジェクトは、この「ウィンドウ」の中央にある必要があります。50番目から99番目のレコードまでです。現在のオブジェクトがリストの最初または最後にある場合、エントリが少なくなる可能性があります。

フォームを開いたときに特定のレコードをアクティブにする必要がある場合、データベースサーバーに対して2つの要求が行われ、それぞれが目的のレコードの両側で75レコードを読み取ります。次に、それらの結果から一般的なリストを接着します。リストを最初または最後から初期化する必要がある場合は、100レコードに対して1つの要求が行われ、最初または最後に受信したレコードがアクティブに設定されます。ユーザーがリストのCTRL + HOMEまたはCTRL + ENDを押してリストの先頭または末尾に移動した場合も同じことが起こります。

ユーザーが現在のウィンドウの中央の外側（50番目の前または99番目の後ろ）でアクティブな録音を行うとすぐに、プラットフォームは追加のレコードを読み取り、現在のレコードが新しいウィンドウの「中央」に来るようにします。

このような動的リストの実装の特徴は、現在のオブジェクトがウィンドウの中央の外側にあることはできないということです。したがって、リストをスクロールすると、現在のオブジェクトは自動的に表示領域に移動します。


リスト内のデータの読み取りは、常に2つのクエリで発生します。最初のクエリは、一時テーブル内の必要なレコードのキーのみを読み取ります。2番目のクエリは、既に読み取られたキーのすべての列の値を読み取ります。これは、サブクエリやその他の複雑なSQL構造にコンパイルできる式を列に含めることができるためです。この場合、プラットフォーム自体がこれらのキーをサブクエリにプッシュするため、これらの列の値の計算はデータベース全体ではなく、必要なキーに従ってのみ行われます。1つのクエリではなく2つのクエリが作成されるため、これによりわずかなオーバーヘッドが発生しますが、非効率的なデータベースサーバープランへの偶発的な「取得」から保護されます。

仕分け


デフォルトでは、リストのエントリは内部オブジェクト識別子の昇順でソートされます。これらの識別子は、オブジェクトの作成時に昇順で自動的に生成され（同時に、すべてのクラス内でグローバルに一意です）、インデックスは常にそれらから構築されます。

リストの並べ替えは、次のように変更できます。

  • コードにソートを介して実行される列を示すことによって注文ステートメント：
    ORDER column1(o) DESC, column2(o)
  • 列見出しをマウスでダブルクリックする（CTRLを押したままにすると、「ネストされた」ソートが追加されます）。

現在のソートに応じて、レコードキーを読み取るときに、対応する列式がリクエストのORDER BYブロックに追加されます。同時に、オブジェクトの一意の識別子が常にソートに追加され、すべてのレコードのキーが一意になるようにします。

次の形式の式：column1> value1 OR（column1 = value1 AND column2> value2）OR（column1 = value1 AND column2 = value2 AND key> value）がリクエストのWHERE句に追加されます。また、キーを読み取るときに、必要な数の読み取りレコードを含むLIMIT命令がリクエストに追加されます。 「アップ」レコードを読み取る場合、ORDER BYの順序とWHEREの式は、逆方向にレコードを読み取るために「反転」されます。

適切なインデックスがある場合、これらのクエリを実行する複雑さは比較的小さいことに注意してください（インデックスは、現在のキーから指定された数のレコードだけ上または下に向かってマイレージになるため）。したがって、列column1、column2でソートするときに動的リストでの作業を高速化するには、次のインデックスを作成することをお勧めします。
INDEX column1(Object o), column2(o), o;
並べ替えが計算された列に基づいている場合は、この記事で説明されているように、列を永続的に保存してから、その列にインデックスを作成できます。

このような実装の機能の1つは、「正直な」スクロールバーがないことです。レコードを読み取る場合、必要な数のレコードのみが読み取られます。目的のフィルターを使用してCOUNT（*）を介してリスト内の行の総数を取得するリクエストを行うと、テーブルまたはインデックスが完全に実行され、パフォーマンスに悪影響を与える可能性があります。OFFSET構文を使用してレコードを読み取るときにも同じ問題が発生します。さらに、リストをナビゲートするとき、リストのエントリ数は、他のユーザーが新しい変更を行うことによって変更できることに注意してください。

編集


驚いたことに、一部の商用プラットフォームでも、動的リストで編集する機能を実現できませんでした。このメカニズムを実装する上での主な問題は、表示されているウィンドウのみがサーバーとクライアントに保存され、リスト全体で変更を加えることができることです。

lsFusionには、リスト編集を直接実装する特定のメカニズムはありません。この機能は、一般的なセッションメカニズムの一部として実装されます。

現在の変更セッションで行われたすべての変更は、一時テーブルに保存されます。ユーザーがフォームで何かを編集すると（いずれかのエントリの値を含む）、新しい値が適切なキーを使用して一時テーブルに書き込まれます。次に、2番目のクエリ（キーを受け取った後）が列の値を読み取ると、対応する一時テーブルに変更を加えたJOINがクエリに追加されます。

変更セッションを保存すると、クエリが実行され、1つのトランザクションで一時テーブルのすべての値がデータベースに書き込まれます。

グループ調整


多くの場合、ユーザーはリストで選択したすべてのオブジェクトの列の値を一度に変更する必要があります。このため、各リストのツールバーには特別なボタン（F12ホットキー付き）があります。これを押すと、通常のセル編集モードがアクティブになりますが、変更は現在のレコードではなく、選択したすべてのレコードに適用されます。

このメカニズムにより、指定した基準に従って多数のオブジェクトをすばやく編集できます。


通常の編集中と同様に、変更はすぐにデータベースに保存されるのではなく、一時テーブルに記録されます。次に、ユーザーは[ 保存 ]ボタンをクリックして、それらをデータベースに書き込む必要があります。このアプローチの欠点は、ユーザーが誤って余分なデータを変更することです。しかし、ここで彼らが言うように、原理は機能します。「大きな力には大きな責任が伴います」。

リストの要約


どのリストでも、ユーザーには、現在の選択の特定の列ごとにレコード数または金額を確認する機会が与えられます。これを行うには、ユーザーは特定のリストのツールバーの適切なボタンをクリックする必要があります。


このデータを取得するには、式COUNT（*）またはSUMを使用して要求が自動的に生成され、その中に現在の選択式が追加されます。この機能を使用すると、レポートを生成することなく、リストから合計をすばやく取得できます。

クライアントのデスクトップバージョンでは、選択したセルの合計をExcelと同様に計算することもできます。


コピーペースト


デスクトップバージョンでは、ユーザーは特定のセルをマークし、CTRL + Cを押して、それらの値をクリップボードに貼り付けることができます。


同様に、テーブルをクリップボードから任意のフォームの編集可能なリストに貼り付けることができます。

このような機会は、多くの場合、特殊な輸入品の開発に代わるものです。

テーブルセッティング


どのリストでも、パラメータの一部を変更できます。

列の構成、サイズ、見出し、マスクなどを変更できます。現在のユーザーとすべてのユーザー（現在のユーザーに必要な権限がある場合）の両方のテーブル設定を保存できます。

[ページサイズ]オプションに注意してください。これを使用して、記事の冒頭で説明した「ウィンドウ」のサイズを変更できます。たとえば、自動50エントリの代わりに、より大きな値を指定できます。その後、クライアントとサーバーに大量のデータがダウンロードされますが、リクエストの発生頻度は低くなります。このパラメーターの値を0に設定すると、動的リストから正常になります。つまり、リスト内のすべてのエントリーが常に読み取られます。ウィンドウサイズの値は、パラメーターpageSize命令DESIGNを使用してコードで指定することもできます。

ピボット


デフォルトでは、すべてのリストがテーブルとして表示されます。しかし、ユーザーがさまざまなレポートや図を作成できる特別な表示モードに切り替える可能性があります。


リスト内のエントリの数が少ない場合、データはクライアントで直接処理されます。データの量が特定のしきい値を超えるとすぐに、サーバー上で自動的に生成されたデータベースクエリがデータのグループ化に使用されます。

オルタナティブ


合理的な質問が発生します。この機能がサポートされていないプラットフォームで動的リストを編集する問題を解決する方法。ほとんどの場合、それらは単純なリストを作成し、ユーザーにいくつかの選択によってエントリの数を制限するように強制します（たとえば、カテゴリ、サプライヤー、または他の関連ディレクトリによるフィルターを指定します）。

ただし、このアプローチには1つの重大な問題があります。選択が確立されていても、かなりの数のレコードがサンプルに含まれないという保証はありません。多くの場合、そのような操作の複雑さはすべてのデータを取得する複雑さに直接匹敵するため、取得される値の数を事前に計算することは不可能です。したがって、場合によっては、多数のエントリがリストに追加され、サーバーまたはクライアントのリソースを大量に消費するだけでなく、パフォーマンスも大幅に低下します。

結論


現代の開発における動的リストの実装は、クライアントとサーバーの両方の部分が関与しているため、最も簡単な作業ではありません。この機能を迅速かつ便利に実装できるオープンライセンスのオープンソースライブラリは世界中にほとんどありません。

オープンで無料のlsFusionプラットフォームでは、動的リストが数行のコードで作成され、ユーザーにそれらを操作するための多数の機会を提供します。このようなリストを編集する機能と組み合わせて、このプラットフォームを使用すると、数十万行のドキュメントをすばやく操作したり、ドキュメント内のレコードの選択に関する便利な作業を整理したり、ディレクトリのグループ変更を実行したりできます。

動的リストを使用すると、限られたデータセットを処理することで、サーバーとクライアントのプロセッサ時間とメモリの消費を大幅に削減でき、サーバーとクライアント間の通信チャネルの負荷を軽減できます。そのような高い効率性により、8つの最大の小売チェーンのうち5つは、ベラルーシのlsFusionプラットフォームに基づくソリューションで主要な運用活動を実行します。
