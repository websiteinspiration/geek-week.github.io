Vkontakteのドキュメントを通じて、Messages APIの禁止を回避した方法

Habroコミュニティの皆さん、こんにちは。私にとって、この最初の記事は特定の陶酔の下で書かれているので、この記事を文学的部分に対してあまり厳密に判断しないでください。しかし、まあ、言葉は少なくてビジネスに取り掛かります。

すべての始まり


VCにはAPIがあることは誰もが知っています。ほとんどの人が自分の目的でそれを使用しようとしたと思います。個人的に、私はそれに関連する多くのプロジェクトを持っています：5つの強力なボットの断片、グループの投稿からの大規模なデータセットのコンパイルなど。そして、友人が私に数回、会話の添付ファイル、写真から曲をダウンロードするように頼んだり、別のファイルに誰かとの通信のテキストを保存したりするのは当然のことです。

しかし、「それ」が出てきて、その瞬間から、そのような小さなリクエストの実装は簡単なタスクではなくなりました。



そのため、数日前に、この問題を一度に取り除くために、ラッパーを通常のユーザーのふりをしてHTTPリクエストで書くことにしました。メッセージセクションの公式APIと同じ強力なツールがあります。

仕事に取り掛かろう


それで、私は認可から始めました。httpsスニファーとFirefoxを装備したことで、許可のすべての「ステップ」を実行し、最終的なCookieを取得することができました。これからは、クエリがどのように行われたかを理解することだけが残りました。ほとんどのデータはhttps://vk.com/wkview.phpからのPOSTリクエストによって受信され、さまざまな状況のパラメーターのみが毎回変化することがわかりました。あらゆる種類の投資を完全に排出するための関数をなんとか作成しましたが、一時的にすべてが劇的に変化したため、これについては詳しく説明しません。
承認Cookieを受信するためのファイルへのリンク（ほとんどの人がかかるため、2要素認証用にのみ作成しました）

予期しない発見


友人が近づいてきて、何をしているのかと聞いたとき、私はラップトップで作業していました。問題全体を指ですばやく説明することができなかったので、メッセージセクションの公式ドキュメントを開き、これらの「禁止された」メソッドの主な説明の下にあるものを見て驚いた：



いいえ、あなたは正しく理解しています。私は最初ではありません。この機会をご覧ください。他のメソッドで何度も使用しましたが、メッセージセクションのメソッドに関数「サンプルリクエスト」が残るとは思えませんでした。そして、トラフィックをスクランブルしたときの驚きはさらに大きかった。これらは通常のAPIリクエストであり、サイト上でのみ、Webフォームのパラメーター名がわずかに異なり、何らかのハッシュIDがありました。



数分して、ハッシュIDがボタンタグのデータハッシュ属性にある文字列にすぎないことに気付き、数分後、「テストリクエスト」のエミュレーションを実装するためにすでに懸命に努力しており、それが機能することを完全には信じていませんでした。結局のところ、これらのリクエストには確かに数などの制限があります。しかし、膝の上に書かれたこの30行のスクリプト（Cookieの受信を含まない）が4分間でダイアログの添付ファイルから5万枚の写真を送り出すことができたときの私の驚きは何でしたか。



使用したコードを適用します
import requests, pickle, re, json

with open('cookies_vk_auth.pickle', 'rb') as handle:
    cookies_final = pickle.load(handle)

session = requests.Session()
peer_id = int(input('  :  '))

response = session.get(f'https://vk.com/dev/messages.getHistoryAttachments', cookies=cookies_final)
hash_data =  re.findall(r'data-hash="(\S*)"', response.text)[0]

session = requests.Session()
response = session.post(f'https://vk.com/dev',
            data=f'act=a_run_method&al=1&hash={hash_data}&method=messages.getHistoryAttachments&param_count=20&param_max_forwards_level=45&param_media_type=photo&param_peer_id={peer_id}&param_photo_sizes=0&param_preserve_order=0&param_v=5.103', cookies=cookies_final)

count=20

for i in range(200):
    response_json = json.loads(json.loads(response.text[4:])['payload'][1][0])['response']['items']

    for photo in response_json:
        ph = photo['attachment']['photo']['sizes'][-1]['url']
        r = session.get(ph, timeout=10)
        
        if r.status_code == 200:
            with open(f'D://dev/'+str(ph.split('/')[-1]), 'wb') as f:
                f.write(r.content)

    m_id = photo['message_id']
    response = session.post(f'https://vk.com/dev',
            data=f'act=a_run_method&al=1&hash={hash_data}&method=messages.getHistoryAttachments&param_count=20&param_start_from={m_id}&param_max_forwards_level=45&param_media_type=photo&param_peer_id={peer_id}&param_photo_sizes=0&param_preserve_order=0&param_v=5.103', cookies=cookies_final)


私は非常に感銘を受けたので、この時点でクールダウンして他の方法を実装することにしました（突然、私はただ間違えられました）。Historyメソッドを取り上げたところ、結果は同様でした。0.1秒の遅延を設定する必要があったのは、サーバーが要求が多すぎるというエラーを出さないためです。（誰かが繰り返す場合は、メソッドを変更するときに、URLをドキュメントに変更する必要があることを覚えておいてください。ハッシュデータはここから取得されます）。つまり、この方法では、パスワードとユーザーログインのみを使用して、公式ドキュメントからメッセージセクションにアクセスできるようになりました。信頼性を高めるために、別のアカウントで同じ手順を実行しても同じ結果が得られました。

要約する


だから、私は、これが1年間ドキュメントに保管されてきた私たちの個人データの保護の侵害であり、何人がすでにそれを使用したかは分からないことをすでに理解していると思います。さらに、このギャップは非常に大きく、すぐに閉じる必要があります。これがこのように機能しないことをもう一度証明するために、VK開発者自身を引用します。
2019年2月15日以降にメッセンジャーの開発を開始する場合は、サポートでテストアクセスを取得する必要があります。これは、スタンドアロンアプリケーションの管理者のキーを使用したメッセージセクションのメソッドの作業を意味します。

つまり、ユーザーの通信にアクセスできる内部アプリケーションのトークンを取得する場合でも、通常のパスワードとログインでのアクセスはもちろん、VKからの個人的な許可が必要です。

私の個人的な意見


メッセージセクションの禁止は、ユーザーのセキュリティに根本的な変更をもたらしませんでした。彼は国境を指定し、彼らが何をしていたかさえ理解していなくても、データへの完全なアクセスを得ることができる「アンダーハッカー」のグループをカットオフしました。プログラミングの経験が豊富な他の人々にとって、通信へのアクセスは時間の問題です。そして、記事の最初の部分で、私は自分の例で、添付ファイルを汲み出すためのプログラムを作成したことで、ユーザーになりすますことができるライブラリーの出現が遠くないことを証明しました。多分私自身がそれを終わらせるでしょう、そしてVK開発者はこれに備え、私たちのデータのプライバシーが本当に重要であるなら、あまりにも疑わしいユーザー活動を認識する方法を考え出す必要があります。
