インターネットブロッキングのしくみ：実際の例を使用した最新の方法の概要



インドの科学者のグループは、自国の例を使用して、政府機関によって導入されたインターネットブロックの最新の方法のレビューを公開しました。彼らは、インターネットプロバイダーが禁止情報へのアクセスを制限するために使用するメカニズムを調査し、その正確性とそのようなブロックを回避する能力を評価しました。この作品のポイントをお伝えします。

入力


近年、さまざまな国の研究者が、「無料」ではないと見なされている国、たとえば中国やイランで使用されているブロッキング方法について多くの研究を行ってきました。しかし、インドのような民主主義国でさえ、近年インターネットを検閲するための大規模なインフラストラクチャを開発しました。

調査中、科学者は国内でブロックされている可能性のある1,200のサイトのリストをまとめました。データはCitizen LabやHerdictなどのオープンソースから収集されました。その後、インターネットアクセスは、最も人気のある9つのインターネットサービスプロバイダーを使用して編成されました。

検閲の事実を判断し、サイトをブロックするために、OONIツールが最初に使用されました。

ロックを検索するOONIと独自のスクリプト


研究者は当初、OONIと呼ばれる人気のある検閲検出ツールを使用することを意図していました。ただし、すでに実験中に、多くの誤検知が発生することが判明しました。結果を手動で確認すると、多くの不正確さが明らかになりました。

検閲の不十分な定義は、古いOONIメカニズムが原因である可能性があります。したがって、DNSフィルタリングが検出されると、ツールは、Google DNSによって返された特定のホストのIPアドレス（無修正と見なされます）を、インターネットプロバイダーによってサイトに割り当てられたIPアドレスと比較します。

アドレスが一致しない場合、OONIはロックの存在を通知します。ただし、現代のインターネットの現実では、異なるIPアドレスは何も言わず、たとえば、CDNネットワークの使用の証拠である可能性があります。

したがって、研究者はロックを検出するために独自のスクリプトを作成する必要がありました。以下は、インターネット上のコンテンツをブロックする一般的な方法の概要と、現代の状況におけるそれらの効果の分析です。

ロックの実行方法またはミドルボックスとは


分析の結果、さまざまなタイプのロックのすべてのケースで、それらは組み込みネットワーク要素を使用して実装されていることがわかりました。研究者はそれらをミドルボックスと呼びました-彼らはユーザーのトラフィックを傍受し、それを分析し、禁止されたサイトに接続する試みを見つけた場合、トラフィックに特別なパケットを埋め込みます。

ミドルボックスを検出するために、研究者はtracerouteユーティリティの原理を使用する独自の反復ネットワークトレース（INT）メソッドを開発しました。その本質は、IPヘッダーのTTL値の増加により、ブロックされたサイトにWebリクエストを送信することです。



ミドルボックス傍受メカニズム

DNSロック


DNS解決プロセスは、あらゆるWebサイトへのアクセスを得るための主要なステップです。ユーザーが入力したURLは、最初に関連付けられたIPアドレスに解決されます。 DNSブロッキングを使用する場合、検閲は常にこのステップで正確に介入します。解決されたリゾルバーはユーザーに無効なIPアドレスを返します。その結果、サイトは単に開かれません（DNSポイズニング）。

別のブロック方法は、DNSインジェクションの使用です。この場合、クライアントとリゾルバー間のミドルボックスがDNSクエリをインターセプトし、不正なIPアドレスを含む独自の応答を送信します。

インターネットプロバイダーによるDNSブロッキングを特定するために、調査者は検閲のない国の出力ノードでTORを使用しました-サイトを開いたがプロバイダーを介した単純な接続で-いいえ、ブロックの事実があります。

DNSによってブロックされたサイトを特定した後、研究者たちはブロックの方法を決定しました。



反復的なネットワークトレース方式：クライアントは、ブロックされたサイトと増え続けるTTLを含む特別なリクエスト（DNS / HTTP GET）を送信します

TCP / IPパケットフィルタリング


パケットヘッダーによるフィルタリングによるロックは、オンラインの検閲の一般的な方法と見なされています。インターネット上では、多くの調査結果を見つけることができます。それらの作成者は、サイトをブロックするこのような方法を特定しようとしています。

実際には、この方法は、従来のシステム障害と簡単に混同される可能性があり、ネットワークに問題を引き起こし、帯域幅を減少させるという問題があります。 HTTPブロッキングとは異なり、TCP / IPをフィルタリングする場合、ユーザーは必要なサイトがブロックされているという通知を受け取りません-単に開かないだけです。ブロッキングのケースを検証し、ネットワークの通常の障害やエラーから切り離すことは非常に困難です。

しかし、研究者たちはこれを試みました。このため、ハンドシェイク手順が使用されました。無検閲の国の出力ノードでTorをトンネルしたハンドシェイクパケット。Torを使用して接続を確立することができたサイトの場合、ハンドシェイク手順は、約2秒の遅延を伴って、さらに5回続けて実行されました。試行のそれぞれが失敗した場合、高い確率で、意図的なフィルタリングの問題でした。

その結果、このようなブロッキング方法は、テストされたすべてのインターネットプロバイダーで見つかりませんでした。

HTTPフィルタリング


ただし、9つのうち5つのプロバイダーの場合、HTTPフィルタリングが検出されました。この方法では、HTTPパケットの内容を分析します。これらの非常に中間的なネットワーク要素（ミドルボックス）を使用して実装できます。

HTTPフィルタリングを識別するために、研究者たちはインターネット検閲のない国で終わるTorループを作成しました。次に、国内でTorを使用して作成されたブロックされたサイトへのリクエストに応じて受け取ったコンテンツを比較しました。

最初のタスクの1つは、ロックが発生した瞬間を識別することでした。たとえば、一部のプロバイダーの場合、HTTP GETリクエストを送信した後の応答は、TCP FINビットが設定され、ブロッキング通知を含むHTTP 200 OK応答でした。クライアントのブラウザーをターゲットサイトから切断したのは、彼でした。しかしその後、サイトからのパッケージも届きました。そのような場合、何がブロッキングトリガーになったのか（クライアントの要求、またはサイトの応答）は明確ではありませんでした。

簡単な操作で確認できました。GETリクエストのHTTPパケットヘッダーで、HostフィールドがHOSTに置き換えられました。これは、ブロックされたサイトを開くのに十分であることがわかりました。これは、検閲がサーバーの応答ではなくクライアントの要求のみをチェックすることを証明しています。

結論：すべてのプロバイダーがブロックしていますか


多くの場合、特定のインターネットプロバイダーはサイト自体をブロックしませんが、この点では「隣接」ネットワークを管理するプロバイダーに依存しています。検討中の実験では、いくつかのインターネットサービスプロバイダーが独自のロックを使用することはありませんでしたが、同時に、国内でユーザーからブロックされたサイトを開くことができませんでした。
