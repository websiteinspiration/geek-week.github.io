ネーミング


コーディングスタイルの基本的な規則は命名です。名前をすぐに（広告を検索せずに）表示すると、タイプ、変数、関数、定数、マクロなどの名前がわかります。命名規則は任意にすることができますが、一貫性が重要であり、規則に従う必要があります。

一般的な命名原則


  • 他のチームのメンバーでも理解できる名前を使用してください。
  • 名前は、オブジェクトの目的または適用性を表す必要があります。
  • 名前の長さを節約しないでください。（初心者でも）長くてわかりやすい名前を使用することをお勧めします。
  • 特にプロジェクトの外でよく知らない場合は、略語が少なくなります。
  • 既知の略語のみを使用してください（ウィキペディアはそれらについて知っていますか？）。
  • 言葉を短くしないでください。

一般に、名前の長さはスコープのサイズと一致する必要があります。たとえば、nは5行の関数内の適切な名前ですが、クラスを記述する場合、これは少し短くなる可能性があります。

class MyClass {
 public:
  int CountFooErrors(const std::vector<Foo>& foos) {
    int n = 0;  //      
    for (const auto& foo : foos) {
      ...
      ++n;
    }
    return n;
  }
  void DoSomethingImportant() {
    std::string fqdn = ...;  //     
  }
 private:
  const int kMaxAllowedConnections = ...;  //    
};


class MyClass {
 public:
  int CountFooErrors(const std::vector<Foo>& foos) {
    int total_number_of_foo_errors = 0;  //      
    for (int foo_index = 0; foo_index < foos.size(); ++foo_index) {  //   `i`
      ...
      ++total_number_of_foo_errors;
    }
    return total_number_of_foo_errors;
  }
  void DoSomethingImportant() {
    int cstmr_id = ...;  //   ( )
  }
 private:
  const int kNum = ...;  //      
};

一般的な名前も有効であることに注意してください。iはイテレーターまたはカウンター、Tはテンプレートパラメーターです。

さらに、ルールを説明するとき、「単語」/「単語」は、省略形を含め、スペースなしで英語で書かれたすべてです。つまり、最初の文字は大文字にすることができます（スタイルに応じて： " キャメルケース "または "パスカルケース"）。残りの文字は小文字です。例えば、好ましくはStartRpc（）、好ましくはStartRPC（）。

テンプレートパラメータもカテゴリのルールに従います：タイプ名、変数名など...

ファイル名


ファイル名は小文字でのみ記述する必要があります。アンダースコア（_）またはハイフン（-）を使用してファイル名を区切ることができます。プロジェクトで使用されているセパレータを使用します。単一のアプローチがない場合は、「_」を使用します。

適切な名前の例：

  • my_useful_class.cc
  • my-useful-class.cc
  • myusefulclass.cc
  • myusefulclass_test.cc // _unittestと_regtestは非推奨です。

C ++ファイルは.ccで終わり、ヘッダーは
.hで終わる必要があります。テキストとして含まれるファイルは.incで終わる必要があります（セクション「独立したヘッダー」も参照してください）。/ usr / includeに

すでに存在する名前（db.hなど）を使用しないでください。 ファイルに特定の名前を付けてください。たとえば、http_server_logs.hは、logs.hよりも優れています。ファイルをペアで使用する場合は、同じ名前を付けることをお勧めします。たとえば、foo_bar.hおよびfoo_bar.cc（およびFooBarクラスが含まれます）。



タイプ名


タイプ名は大文字で始まります。新しい単語もそれぞれ大文字で始まります。アンダースコアは使用されません：MyExcitingClass、MyExcitingEnum。

すべてのタイプの名前（クラス、構造、エイリアス、列挙、テンプレートパラメータ）は、同じスタイルで名前が付けられます。タイプ名は大文字で始まります。新しい単語もそれぞれ大文字で始まります。アンダースコアは使用されません。例えば：

// classes and structs
class UrlTable { ...
class UrlTableTester { ...
struct UrlTableProperties { ...

// typedefs
typedef hash_map<UrlTableProperties *, std::string> PropertiesMap;

// using aliases
using PropertiesMap = hash_map<UrlTableProperties *, std::string>;

// enums
enum UrlTableErrors { ...

変数名


変数（関数パラメーターを含む）とデータメンバーの名前は、単語間にアンダースコアを付けた小文字で記述されます。これらのクラスのメンバー（構造ではない）は、名前の末尾にアンダースコアが付いています。たとえば、次のようにa_local_variable、a_struct_data_member、a_class_data_member_。

共通の変数名


例えば：

std::string table_name;  // OK -    


std::string tableName;   //  -

クラスデータメンバー


これらのクラスのメンバーである静的および非静的は、末尾にアンダースコアが追加された通常の変数と呼ばれます。

class TableInfo {
  ...
 private:
  std::string table_name_;  // OK -   
  static Pool<TableInfo>* pool_;  // OK.
};

構造体データメンバー


静的および非静的の構造データのメンバーは、通常の変数と呼ばれます。最後にアンダースコアは追加されません。

struct UrlTableProperties {
  std::string name;
  int num_entries;
  static Pool<UrlTableProperties>* pool;
};

いつ構造体を使用するか、いつクラスを使用するかについて説明している構造体とクラス も参照してください。

定数の名前


オブジェクトはconstexprまたはconstとして宣言されるため、実行中に値は変更されません。定数の名前は記号「k」で始まり、次に大文字と小文字が混在したスタイルの名前になります。アンダースコアは、大文字が分離に使用できない場合にまれに使用できます。例えば：

const int kDaysInAWeek = 7;
const int kAndroid8_0_0 = 24;  // Android 8.0.0

静的タイプのストレージ（つまり、静的またはグローバル、詳細はこちら：ストレージ期間）を持つ同様のすべての定数オブジェクトにも名前が付けられます。この規則は、他のタイプのストレージ（たとえば、自動定数オブジェクト）の変数ではオプションです。

関数名


一般的な関数は、大文字と小文字が混在したスタイルで名前が付けられます。変数アクセス関数（アクセサーとミューテーター）は、ターゲット変数と同様のスタイルを持つ必要があります。

通常、関数名は大文字で始まり、名前の各単語は大文字になります。

void AddTableEntry();
void DeleteUrl();
void OpenFileOrDie();

（APIの一部であり、関数のように見えるはずのクラスまたは名前空間領域の定数にも同様の規則が適用されます（そして関数ではないという事実は重要ではありません））

アクセサーとミューテーター（関数の取得と設定）対応する変数のように名前を付けることができます。それらはしばしば実際のメンバー変数に対応しますが、これは必須ではありません。たとえば、int count（）およびvoid set_count（int count）。

名前空間名前空間


名前空間は小文字と呼ばれます。トップレベルのネームスペースは、プロジェクトの名前に基づいています。名前と他のよく知られている名前空間の衝突を避けてください。

トップレベルの名前空間は通常、（コードを作成した）プロジェクトまたはチームの名前です。コードは、名前空間に対応する名前を持つディレクトリ（またはサブディレクトリ）に配置する必要があります。略語を使用し

ないというルールを忘れないでください-これは名前空間にも適用されます。内部のコードは名前空間参照を必要とする可能性が低いため、省略形は不必要です。 ネストされた名前空間に既知の名前を使用しないでください。名前間の衝突は、アセンブリーに驚きをもたらす可能性があります。特に、名前がネストされた名前空間を作成しないでください

std。一意のプロジェクト識別子（websearch :: index、websearch :: index_util）は、衝突に対して安全ではなく、websearch :: utilが推奨されます。

以下のために内部/内部別のコードが追加されたときに、名前空間、衝突が（内部ヘルパーは異なるチームに繰り返す傾向がある）が発生することがあります。この場合、ファイル名を使用して名前空間に名前を付けると、非常に役立ちます。（websearch ::インデックス:: frobber.hで使用するためのfrobber_internal）

列挙名


転送（スコープの制限あり（スコープあり）、なし（スコープなし）のいずれか）は、定数またはマクロと呼ばれます。つまり、kEnumNameまたはENUM_NAMEのいずれかです。

列挙子の個々の値に定数として名前を付けることをお勧めします。ただし、マクロと呼ぶことは許容されます。UrlTableErrors（およびAlternateUrlTableErrors）列挙自体の名前はタイプです。したがって、混合スタイルが使用されます。

enum UrlTableErrors {
  kOk = 0,
  kErrorOutOfMemory,
  kErrorMalformedInput,
};
enum AlternateUrlTableErrors {
  OK = 0,
  OUT_OF_MEMORY = 1,
  MALFORMED_INPUT = 2,
};

2009年1月までは、列挙値の命名スタイルはマクロのスタイルに似ていました。これにより、マクロ名と列挙値が重複するという問題が発生しました。定数スタイルを適用すると問題が解決するため、新しいコードでは定数スタイルを使用することをお勧めします。ただし、古いコードを書き換える必要はありません（複製の問題がない限り）。
