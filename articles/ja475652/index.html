Aurora OS（Sailfish Mobile OS RUS）でモバイルアプリケーションを開発した方法

私たちのチームは、大規模な顧客向けの支払いシステムのフレームワーク内で支払いを受け取るための企業モバイルアプリケーションを開発してサポートしています。現在、ロシアの37の地域でAuroraオペレーティングシステム（以前のSailfish Mobile OS RUS）を実行しているデバイスで顧客の従業員が使用しています。この投稿では、このプロジェクトについて簡単に説明し、OS自体について詳しく説明します。


バックグラウンド


当社のNORBITは、大口顧客向けの決済システムを開発しました。システムには、サーバー部分といくつかのクライアント（Androidオペレーティングシステム上のデスクトップ、ウェブ、モバイルアプリケーション）が含まれています。このシステムは当初、ロシア連邦のいくつかの地域で導入されました。 2016年、お客様は地域での存在感を拡大することを決定しました。現在、このシステムは37の地域で実装され、正常に動作しています。

2017年に、顧客はデバイスを国内のソフトウェアで使用することを決定し、Aurora OS（Sailfish Mobile OS RUS）を備えた数千のINOI R7デバイスを購入しました。私たちのチームは、別のモバイル決済システムクライアントの作成を任されました。 

開発に数ヶ月しかかかりませんでした。すべての作業をゼロから整理する必要があったため、これは興味深い課題でした。チームを編成し、新しいバージョンの開発、テスト、定期リリースのプロセスを構築する必要がありました。これらの段階について簡単に説明します。


プロジェクトはいかがでしたか


そもそも、これを実現する力を理解する必要がありました。チームが必要でしたが、Aurora OS市場には既製のスペシャリストがいませんでした。

ここでは、プラットフォーム自体に敬意を表する必要があります。Aurora OSでの開発はQtフレームワークを使用して行われます。単純なアプリケーションを作成する場合は、宣言型言語QMLを使用できます。これにより、ほとんどの場合、C ++での低レベルのプログラミングに頼ることができなくなります。結果として、Qtの賢明な開発者を探す作業に要約され、全体としては十分です。

さて、チームが結成されました。機能面で他の顧客に追いつく必要があります。締め切りは厳しく、既存の機能を更新して新しい機能を実装するのに数か月しかかかりませんでした。何よりも、ビジネスロジックとの戦いに加えて、プラットフォーム自体との戦いが待ち受けていました。多くの質問を解決する必要がありましたが、それらに対する既製の答えはありませんでした。stackoverflowを使用すると、方向が新しく、最初の1つであったため、ソリューションを迅速にコピーできません。それらのほんの一部です。

KKMによるBluetooth接続


このプロジェクトでは、ATOLおよびShtrikhのキャッシュレジスタ（KKM）のサポートが必要でした。どちらのKKMメーカーも、Bluetooth経由で接続できる製品を製造しています。Bluetoothの使用例は、Qt Webサイトにあります。

バーコードは、リンクからダウンロードできるソースコードの形式でKKMを操作するためのドライバーを提供します。ドライバを使用するプロジェクトの例もあります。

ATOL用のドライバーは、「業務用機器のドライバー」という検索キーワードでサイトからダウンロードできます。このプロジェクトでは、ドライバーの8番目のバージョンを使用しました。

プロジェクトの最初には、54-FZのサポートありとサポートなしの両方のKKMがありました。したがって、4種類のキャッシュレジスタ（ATOL /54-付きバーコード/54-なしのバーコード）のサポートが必要でした。

大規模なデータベース更新を処理する


顧客の従業員は毎朝、電話で相手の参照情報を更新します。非常に多くの請負業者がいるため（地域によっては数千）、ディレクトリの更新には最大1分かかりました。最初は、各カウンターパーティを挿入した後にコミットが行われました。すべてのヘルプ情報を挿入した後でコミットを開始した後、ディレクトリを更新する時間が半分になりました（1つの大きなコミット）。

アプリケーション構築の最適化


最初に、プロジェクトで「子供の過ち」が起こりました-ヘッダーファイルをヘッダーファイルに含めました。フォワード宣言への移行により、プロジェクトのビルド時間が大幅に短縮されました。また、プロジェクトをビルドしてインストールし、rpmパッケージを作成するには、かなりの時間がかかります。インストールの種類として「バイナリファイルのコピー」を選択すると、プロジェクトの再組み立て、インストール、デバッグの開始が数秒で完了します。また、エミュレータでのアセンブリを高速化するために、バーコードとキャッシュレジスタを操作するためのクラスのアセンブリを無効にすることができます。非rpmインストールオプションは、依存関係が引き上げられないため、潜在的に危険です。

自動ビルドアプリケーション


開発環境を開始しなくても、アプリケーションのアセンブリを整理できます。開発環境からビルドする場合は、アセンブリコンソールで実行中のコマンドを確認できます。同じコマンドをbashまたはバッチで実行できます。

しかし、すべての困難にもかかわらず、私たちはお客様が設定したタスクを完了しました。2017年12月31日、アプリケーションが正常に起動し、最初の支払いが承認されました。現時点では、アプリケーションはAurora OSを実行している数千のモバイルデバイスで実行されます。 

OSオーロラ 


次に、オペレーティングシステムとその機能について詳しく説明します。


Aurora OS（以前のSailfish Mobile OS RUS）は、モバイルデバイス（スマートフォンおよびタブレット）向けの信頼できるオペレーティングシステムです。これは、Open Mobile Platform社によって開発されています。このシステムはSailfish OSプラットフォームに基づいており、企業ユーザーおよび政府機関向けに設計されています。  

2018年、RostelecomはOpen Mobile Platformの会社の75％とフィンランドのプロジェクトの支配的株式を買収しました。同時に、Sailfish Mobile OS Rusという名前をAuroraオペレーティングシステムに置き換えることが決定されました。これは国内市場により適しており、著者が考案したように、製品の目標に対応しており、ロシアと海外の両方でポジティブな連想を引き起こしています。

「信頼できる」という言葉は、Aurora OS上のデバイスを従業員に提供する組織が、デバイス自体とデバイスが使用するデータの両方を完全に制御できることを意味します。データの保存と送信のセキュリティは、GOSTに準拠した暗号化アルゴリズムによって保証され、FSBおよびFSTECの証明書によって確認されます。

モバイルデバイスを管理するために、SF Cloud製品もOpen Mobile Platformによって開発されています。これはお客様の組織の機器に展開できるサーバーソリューションであり、従業員のモバイルデバイスのステータスをいつでも監視し、デバイスのアプリケーションをインストール、更新、削除し、OSの更新を管理し、デバイスへのアクセスをブロックし、データを安全に削除することができます（ワイプ） ）


Aurora OSの完全な開発サイクルがロシアで行われています。これにより、彼女はロシアのコンピュータープログラムとデータベースの統一登録簿に入ることができました。現時点では、これがこのステータスを持つ唯一のモバイルOSであるため、レジストリからのモバイルアプリケーションの互換性は必須です。 

Aurora OSの内部にはPOSIX互換の環境があります。つまり、モバイルデバイス向けの本格的なLinuxです。たとえば、Androidとは異なり、PCの「大規模」ディストリビューションに馴染みのあるsystemd、D-Bus、ssh、およびその他のサービスとユーティリティがあります。同時に、システムにはlibhybrisコンポーネントが含まれています。これは、POSIX互換システムでBionic環境のドライバーを使用するためのものです。これにより、元々Android用に設計されたデバイス上のAurora OS。これには、 Sony Xperia XおよびSony Xperia XA2の公式サポートが含まれます。

必要なAPIのほとんどすべてが、Qtフレームワークとそれに基づく追加のライブラリを使用して提供されます。これらは、システムアプリケーションとサービスの実装にも使用されます。Qtライセンスでは、オープンアプリケーションと独自仕様の両方を作成できます。 2番目のケースでは、基本的な要件はLGPLルールへの準拠ですが、Qtライブラリを商用ベースで使用する場合、この要件はオプションになります。

Sailfish SDK は公開されており、Qtでの作業にも重点を置いています。次のコンポーネントが含まれています。

Qtクリエーター-Qtプロジェクトによって開発され、Qtライブラリとの便利な作業のための高度な統合とツールを提供するIDE。 Sailfish Mobile OS RUSのサポートは、ARMおよびx86アーキテクチャ用の適切なビルドキットとビルドおよびエミュレーションコントロールを構成するプラグインを通じて実現されます。


ビルド環境は、VirtualBoxの仮想マシンにパッケージ化された一連のツールと目標を備えたLinux環境です。この形式の配布では、開発者のオペレーティングシステムに関係なく、同じビルドツールを使用できます。Linux、Windows、およびmacOSがサポートされていますが、いくつかの制限もあります。たとえば、Aurora OSのプロジェクトは、仮想マシンから見える特定のディレクトリ（ユーザーのホームディレクトリまたはSDKのインストール時に指定された代替ディレクトリ）に配置する必要があります。

エミュレーターは、x86用にコンパイルされたSailfish OSイメージを含むVirtualBoxの仮想マシンです。実際、これはオペレーティングシステムの完全なアセンブリであり、アプリケーションのさまざまな側面を確認できます。しかし、同時に、たとえばセンサーの使用を確認するために物理デバイスを使用すると便利です。

Qt QML Liveは、インストールパッケージを再ビルドせずに、プロジェクトのQMLファイルに変更を加えるときに、アプリケーションのグラフィカルインターフェイスをオンザフライで「変更」できるツールです。このアプローチにより、ユーザーインターフェイスの開発を大幅に加速できます。

ドキュメントは、Qtによって提供されるAPIの説明と、Aurora OSに固有のコンポーネント（Sailfish Silicaドキュメントなど）の両方をカバーする一連のリファレンス記事です。ドキュメントは個別に読むことができ、ソースコードエディタで選択した要素に対して呼び出されるコンテキストヘルプ形式で利用できます。

Aurora OS用のアプリケーションの開発を開始するには、ドキュメントを勉強するだけでなく、トレーニングコースを受講することもできます。Stepikプラットフォームには、開発の最初のステップと主な側面を説明する公開入門コースがあります。また、オープンモバイルプラットフォーム企業のパートナーの従業員は、リクエストに応じて、edu @ omprussia.ruにメールを送信できます。センサー、ナビゲーション、マルチメディアなど

の使用に関するレッスンを含む、より包括的なトレーニングコースへのアクセスを取得します。アプリケーションの開発結果は、インストール用のrpmパッケージです。彼は顧客企業のデバイスの従業員に入る前に、次の手順を実行する必要があります。

1.有効な開発者証明書で署名されている。実際、rpmファイルは署名されているだけでなく、その構成に含まれているファイルも含まれているため、この段階はインストールパッケージアセンブリの不可欠な部分です。可用性により、インストールパッケージの発行元とその構造の整合性を確認できます。署名を完了するには、鍵生成ツールと開発者証明書を取得する必要があります。これを行うには、Open Mobile Platform企業のパートナーが適切なリクエストをdev-support@omprussia.ruに送信できます。

2.インストールパッケージをSF Cloud管理者に転送します。ワークフローの構成に応じて、このステージはさまざまな方法で実装できます。その結果、パッケージをSF Cloudリポジトリにダウンロードし、コントロールパネルからデバイスに配布できるようにする必要があります。同時に、リポジトリにダウンロードするとき、パッケージの署名の検証だけでなく、rpmファイル構造の正確さもあり、これは多くの要件（アプリケーションコンポーネントの場所の標準化とセキュリティ要件の両方による）の影響を受けます。例えば：

  • パッケージのビルドに使用されるスペックファイルでは、セクション％pre、％post、％preun、％postun、％verifyscriptを使用しないでください。
  • アプリケーションパッケージのスペックファイル内のスクリプトは、既存のファイルを変更または削除しないでください。
  • , ;
  • /usr/bin/{ };
  • desktop- /usr/share/applications/{_}.desktop;
  • /usr/share/icons/hicolor/{}/apps/{ }.png;
  • , , /usr/share/{ }.


このような要件の検証を自動化するには、rpm-validatorスクリプトを使用します。同様のチェックがSDKで利用でき、ビルドエンジンコントロールセンターのIDEから呼び出すことができます。 rpm-validatorスクリプトの設定は、ターゲットプラットフォームとお客様の要件によって決まる可能性があることを明確にすることが重要です。dev-support@omprussia.ruでのリクエストに応じて、「Open Mobile Platform」でAurora OSの認定ビルド用のrpm-validatorを取得できます。

3.管理パネルSF Cloudを介した従業員のデバイスへの配布。この段階には、原則として、デバイスの小さなグループでの技術的および機能的なテストが含まれます。Aurora OSの認定ビルドを備えたデバイスにrpmファイルをインストールする場合、署名と構造もチェックされます。開発者は、すべての従業員をデバイスにダウンロードする前に発生するエラーの修正に関与できます。

Aurora OSは、FSBおよびFSTECの要件を満たす唯一のモバイルオペレーティングシステムです。 

Aurora OSでのモバイルアプリケーション開発の機能について説明するコメントが用意されています。
