Angular：統合テスト（浅いテスト）



アプリケーションが大きくなったとき、またはリファクタリングで正しく機能することが非常に重要なときは、単体テストまたはe2eテストについて考え始めます。

Angularでの作業の数年間-コーポレートセグメントのアプリケーションは、リファクタリングや新しい機能の作成時に多くの問題を抱えていたため、テストは時間の無駄ではなく、プログラマーの手に渡る安定性の鍵のように見えます。

次に、Angularで基本的なアプリケーションをテストする方法を理解し、少し理論に触れてみましょう。

アプリケーション開発者が実行できるテストの種類（この場合はフロントエンド）から始めましょう。



（Martin FowlerとGoogle Testing Blog）

Angularには、すぐに使えるそのようなメソッドがあります。

Angularアプリケーションの例を使用して、テストピラミッドを詳しく見てみましょう。

ユニットテスト -クラスメソッドの独立したテスト。メソッドの動作をチェックし、パラメーターのさまざまな組み合わせを入力に渡し、結果を期待されるものと比較します。

たとえば、16進文字列がrgbに変換されるコンポーネントがあります。この情報はユーザーに表示され、残りのサーバーにも送信されます。

単体テストに目を向けると、コンポーネントクラスのメソッドごとに多くのテストを記述できます。

メソッドrgbToHex（hex：string）=> stringがある場合、そのようなシグネチャを持つメソッドをテストするには、次のことを行う必要があります：expect（rgbToHex（ '777'））ToBe（ 'rgb（119、119、119）'） 。

すばらしいように見えますが、カバーする必要のある関数の数が多いため問題があり、テストを書くのが面倒です。また、コンポーネント内の各メソッドの単体テストを記述しても、それらの共同作業の正確性は保証されません。

たとえば、テーブル出力コンポーネントを作成してテストした後、それを「電卓」コンポーネントで呼び出すことができますが、誤ってバインディングを誤って指定してしまいます。その場合、データはテーブルに分類されず、完全に機能するコンポーネントは正しく機能しません。



統合テスト-いくつかのコンポーネントの束をテストします。この段階から、クラスメソッドだけでなく、HTMLへのバインディング（つまり、コンポーネント内の要素をクリックします。浅いテストは、本質的に統合テストであるAngular表記でよく見られます。

浅いテストでは、以下に詳しく説明します。

E2E（エンドツーエンド）テスト-単体テストの問題を解決するためにアプリケーションを完全にテストする方法。

このアプローチでは、完全にレンダリングされたアプリケーションのテストスクリプトを記述します。すべてのコンポーネントとサービスが一緒に組み立てられ、ユーザーの操作が再現されます。
これは非常に優れていますが、スタブ（通常はnode.jsの静的jsonサーバー）を動的に変更するという問題があり、エミュレートされたサーバーとは異なるデータが必要になる場合があります。

たとえば、ユーザーのリストがあるページがあります。このパネルのコンポーネントの1つはページネーションです。ユーザー数によって動作が異なります。ただし、エミュレートされたサーバー上のデータがjsonを介して設定されている場合、ユーザーの数は常に1になるため、異なるページ数ですべてのケースをチェックすることはできません。

その結果、メソッドではなくインターフェイスの論理ユニットによってスタブを柔軟に変更してテストする場合は、e2eとユニットテストの間に何かを使用する必要があると結論付けることができます。ここで、浅いテスト（統合テスト）が役に立ちます。

用語集：

  • モックは、さまざまなユースケースで応答をシミュレートするためのオブジェクトです。
  • スタブは、ロジックのない最も愚かなスタブです（Martinを読むことができます）。
  • KarmaはAngularに組み込まれたテストランナーです（多くの場合、代わりにjestを使用することをお勧めしますが、今日はそうではありません）。
  • Jasmineは、テストを仕様の形式で記述するためのフレームワークです（以下を参照）。
  • spec-テストファイル拡張子（BDDスタイルの仕様記述）。
  • これは、Jasmineでのテストのメソッドの名前です。
  • xitは、実行されないメソッドの名前です。
  • 適合-スペックに同じ名前のメソッドがある場合、それらのみが起動されます。

Angularや他のフレームワークの浅いテストは、十分なサイズのコンポーネントがレンダリングされ、独自の機能を備えたuiの個別のユニットとして存在できるようになる場合のユニットテストアプローチです。

たとえば、hex-> rgbに変換するためのコンポーネントがあります。このコンポーネントのみをレンダリングし、さまざまな状況のスタブを生成し、エンドユーザーの観点からこのコンポーネントの可能なユースケースを実行し、コンポーネントの動作を確認できます。

例を考えてみましょう（リポジトリ）。

PageObjectに従ってコンポーネントのコンポーネントにアクセスするためのクラスを準備し、プロジェクトのルートにヘルパーを追加します。

ヘルパー-レンダリング用に選択されたコンポーネント内の要素を検索するのに役立ちます。このように、Angular Materialを使用すると、作業が楽になります。selectタイプの要素は、別のブロックにオプション付きのリストを作成します。これらの要素を検索すると、ボイラープレートにつながり、ヘルパー形式のラッパーが役立ちます。

export class PageObjectBase {

  constructor(private root: HTMLDivElement) { }
  //    input  
  _inputValue(cssSelector: string, value: string) { 
    if (value) {
      this.root.querySelector<HTMLInputElement>(cssSelector).value = value;
      this.root.querySelector<HTMLInputElement>(cssSelector).dispatchEvent(new Event('input'));
    }
    else {
      return this.root.querySelector<HTMLInputElement>(cssSelector).value
    }
  }
  //          
  _buttonClick(cssSelector: string) {
    this.root.querySelector<HTMLButtonElement>(cssSelector).dispatchEvent(new Event('click'));
  }

}

PageObjectは人気のある自動テストテンプレートです。書かれたテストのサポートを簡素化するために使用されます。UIを変更した場合、テストを書き直す必要はありません。要素セレクターを変更するだけです。


export class ConverterFromHexPageObject extends PageObjectBase {

  constructor(root: HTMLDivElement) {
    super(root)
  }

  hex(text?: string) {
    return this._inputValue('.input-hex', text);
  }

  rgb(text?: string) {
    return this._inputValue('.input-rgb', text);
  }

  clear() {
    this._buttonClick('.btn-clear')
  }

  calc() {
    this._buttonClick('.btn-calc')
  }

}

const urlToSave = 'http://localhost:4200/save-hex';


//  -   
describe('ConverterFromHexComponent', () => {
  let component: ConverterFromHexComponent;
  let fixture: ComponentFixture<ConverterFromHexComponent>;
  let page: ConverterFromHexPageObject;
  let httpTestingController: HttpTestingController;


  //      ,   
  beforeEach(async(() => {
    TestBed.configureTestingModule({
      imports: [ConverterModule, HttpClientTestingModule]
    })
      .compileComponents();
    httpTestingController = TestBed.get(HttpTestingController);
  }));


  //  ,   
  beforeEach(() => {
    fixture = TestBed.createComponent(ConverterFromHexComponent);
    component = fixture.componentInstance;
    page = new ConverterFromHexPageObject(fixture.nativeElement);
    fixture.detectChanges();
  });


  //    ,     http 
  afterEach(() => {
    httpTestingController.verify();
  });

  it('should create', () => {
    expect(component).toBeTruthy();
  });

  it('should clear', async () => {
    page.hex('112233'); //  input  
    expect(page.hex()).toBe('112233'); // ,    
    await page.clear(); //      
    fixture.detectChanges(); //    
    expect(page.hex()).toBe(''); // ,   clear 
  });

  it('should convert', async () => {
    page.hex('123123');
    expect(page.rgb()).toBe('');
    page.calc();
    const req = httpTestingController.expectOne(urlToSave);
    expect(req.request.method).toEqual('POST');
    expect(req.request.body.hex).toEqual('123123');
    req.flush({});
    await fixture.detectChanges();
    expect(page.rgb()).toBe('rgb(18, 49, 35)');
  });

  it('should convert three-digit hex', async () => {
    page.hex('567');
    expect(page.rgb()).toBe('');
    page.calc();
    const req = httpTestingController.expectOne(urlToSave);
    expect(req.request.method).toEqual('POST');
    req.flush({});
    await fixture.detectChanges();
    expect(page.rgb()).toBe('rgb(85, 102, 119)');
  });

  it('rgb should be empty when entered incorrectly hex', async () => {
    page.hex('qw123we');
    page.calc();
    const req = httpTestingController.expectNone(urlToSave);
    await fixture.detectChanges();
    expect(page.rgb()).toBe('');
  });

});
