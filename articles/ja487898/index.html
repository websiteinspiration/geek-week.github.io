膝のメモリ帯域幅の測定

数週間前の夕食の会話の中で、ある同僚はある種の遅いプロセスについて不平を言っていました。彼は生成されたバイト数、処理サイクル数、そして最終的にはRAMの容量を計算しました。同僚は、メモリ帯域幅が500 GB /秒を超える最新のGPUは、そのタスクを使い果たして窒息することはないと述べています。

これは興味深いアプローチだと私には思えました。個人的に、私はこれまでこの観点からパフォーマンス目標を評価したことがありません。はい、プロセッサとメモリのパフォーマンスの違いについて知っています。



キャッシュを多用するコードの書き方を知っています。おおよその遅延値を知っています。しかし、これはメモリ帯域幅をすぐに評価するには不十分です。

これが思考実験です。10億の32ビット整数の連続した配列をメモリに想像してみてください。これは4ギガバイトです。この配列を反復処理して値を合計するにはどのくらい時間がかかりますか？CPUはRAMから1秒あたり何バイトを読み取ることができますか？継続的なデータ？ランダムアクセス？このプロセスはどれだけ並列化できますか？

これらは役に立たない質問だと言うでしょう。実際のプログラムは複雑すぎて、このような単純なランドマークを作成できません。そこには！本当の答えは「状況による」です。

ただし、この問題は調査する価値があると思います。私は答えを見つけようとはしていません。しかし、上限と下限、途中の興味深いポイントを定義し、その過程で何かを学ぶことができると思います。

2020年の遅延


そのリストは2012年に編集され、2020年のこの記事は時代が変わりました。StackOverflowを搭載したIntel i7の数値は次のとおりです。

L1キャッシュでヒット、約4サイクル（2.1-1.2 ns）
L2キャッシュでヒット、約10サイクル（5.3-3.0 ns）
L3キャッシュでヒット、シングルコアの場合〜40サイクル（21.4-12.0 ns）
L3キャッシュでヒットし、別のカーネルで約65サイクル（34.8-19.5 ns）
L3キャッシュをヒットし、別のカーネルを変更して〜75サイクル（40.2-22.5 ns）
ローカルRAM〜60 ns

面白い！何が変わったの？

  • L1が遅くなっています。 0,5 → 1,5
  • L2の高速化。 7 → 4,2
  • L1とL2の比率は大幅に減少します。2,5x 14（ワオ！）
  • L3キャッシュが標準になりました。 12 40
  • RAMが高速になりました。 100 → 60

広範囲にわたる結論を導き出すことはしません。元の数値がどのように計算されたかは不明です。リンゴとオレンジの比較はしません。

素朴なベンチマーク


いくつかのテストをしてみましょう。帯域幅を測定するために、単純なC ++プログラムを作成しました。おおよそ彼女はこのように見えます。

// Generate random elements
std::vector<int> nums;
for (size_t i = 0; i < 1024*1024*1024; ++i) // one billion ints
    nums.push_back(rng() % 1024); // small nums to prevent overflow

// Run test with 1 to 12 threads
for (int thread_count = 1; thread_count <= MAX_THREADS; ++thread_count) {
    auto slice_len = nums.size() / thread_count;
    
    // for-each thread
    for (size_t thread = 0; thread < thread_count; ++thread) {
        
        // partition data
        auto begin = nums.begin() + thread * slice_len;
        auto end = (thread == thread_count - 1)
            ? nums.end() : begin + slice_len;

        // spawn threads
        futures.push_back(std::async([begin, end] { 
            
            // sum ints sequentially
            int64_t sum = 0;
            for (auto ptr = begin; ptr < end; ++ptr)
                sum += *ptr;
            return sum;
        }));
    }

    // combine results
    int64_t sum = 0;
    for (auto& future : futures)
        sum += future.get();
}

いくつかの詳細は省略されています。しかし、あなたはその考えを理解しました。要素の大きな連続した配列を作成します。アレイを個別のフラグメントに分割します。各フラグメントを個別のスレッドで処理します。結果を蓄積します。

ランダムアクセスも測定する必要があります。それはとても難しいです。私はいくつかの方法を試しましたが、最終的には事前計算されたインデックスを混在させることにしました。各インデックスは1回だけ存在します。次に、内側のループがインデックスを反復処理して計算しsum += nums[index]ます。

std::vector<int> nums = /* ... */;
std::vector<uint32_t> indices = /* shuffled */;

// random access
int64_t sum = 0;
for (auto ptr = indices.begin(); ptr < indices.end(); ++ptr) {
    auto idx = *ptr;
    sum += nums[idx];
}
return sum;

スループットの計算では、インデックス配列のメモリは考慮していません。合計に寄与するバイトのみがカウントされsumます。私はハードウェアのベンチマークを行いませんが、さまざまなサイズのデータ​​セットとさまざまなアクセススキームで機能する能力を評価します。

大きなブロック


最初のテストは、大きなメモリブロックで動作します。 1 GBのNアイテムのブロックが強調表示され、小さなランダムな値で埋められます。単純なループは配列をN回反復するため、ボリュームN を使用してメモリにアクセスし、合計を計算しint64_tます。いくつかのスレッドが配列を分割し、それぞれが同じ数の要素にアクセスします。



TA-ダー！このグラフでは、合計演算の平均実行時間を取得して、からruntime_in_nanosecondsに変換していgigabytes_per_secondます。

かなり良い結果です。int321つのストリームで11 GB /秒を順次読み取ることができます。 38 GB /秒に達するまで線形にスケーリングします。テストmatrix4x4とmatrix4x4_simd高速ですが、同じ天井に寄りかかっています。

1秒あたりにRAMから読み取ることができるデータ量には、明確な上限があります。私のシステムでは、これは約40 GB /秒です。これは、上記の現在の仕様に準拠しています。

下の3つのグラフから判断すると、ランダムアクセスは低速です。とても、とても遅い。シングルスレッドのパフォーマンスint32は、わずか0.46 GB /秒です。これは、11.03 GB /秒の順次スタッキングより24倍遅いです！matrix4x4完全なキャッシュラインで実行されるため、テストは最良の結果を示します。ただし、シーケンシャルアクセスよりも4〜7倍遅く、ピークはわずか8 GB /秒です。

小ブロック：順次読み取り


私のシステムでは、各ストリームのL1 / L2 / L3キャッシュサイズは32 KB、256 KB、2 MBです。要素の32キロバイトのブロックを取り、それを125,000回繰り返した場合はどうなりますか？これは4 GBのメモリですが、常にキャッシュにアクセスします。



驚くばかり！シングルスレッドのパフォーマンスは、約12 GB /秒の大きなブロックを読み取るのと似ています。今回を除いて、マルチスレッドは40 GB /秒の上限を突破します。それは理にかなっている。データはキャッシュに残るため、RAMのボトルネックは発生しません。 L3キャッシュに収まらないデータについては、約38 GB /秒の同じ上限が適用されます。

テストmatrix4x4は回路と同様の結果を示していますが、さらに高速です。シングルスレッドモードでは31 GB /秒、マルチスレッドモードでは171 GB /秒。



では、を見てみましょうmatrix4x4_simd。 y軸に注意してください。



matrix4x4_simd非常に高速に実行されました。の10倍の速度ですint32。16 KBのブロックでは、1000 GB /秒を突破します！

明らかに、これは表面合成テストです。ほとんどのアプリケーションは、同じデータを使用して同じ操作を続けて100万回実行しません。テストでは、実際のパフォーマンスは示されていません。

しかし、教訓は明らかです。キャッシュ内では、データは迅速に処理されます。SIMD使用時の天井が非常に高い場合：シングルスレッドモードでは100 GB /秒を超え、マルチスレッドモードでは1000 GB /秒を超えます。キャッシュへのデータの書き込みは遅く、ハード制限は約40 GB /秒です。

小ブロック：ランダム読み取り


同じことをしましょう。ただし、ランダムアクセスを使用します。これは私の記事の私のお気に入りの部分です。



RAMからのランダムな値の読み取りは遅く、わずか0.46 GB /秒です。L1キャッシュからのランダムな値の読み取りは非常に高速です：13 GB /秒。これはint32、RAMからシリアルデータを読み取るよりも高速です（11 GB /秒）。



テストmatrix4x4は同じテンプレートで同様の結果を示していますが、の約2倍の速さintです。



ランダムアクセスはmatrix4x4_simdめちゃくちゃ速いです。

ランダムアクセスの結果


メモリからの無料読み取りが遅い。壊滅的に遅い。両方のテストケースで1 GB /秒未満int32。同時に、キャッシュからのランダム読み取りは驚くほど高速です。これは、RAM からの順次読み取りに相当します。



消化する必要があります。キャッシュへのランダムアクセスの速度は、RAMへのシーケンシャルアクセスと同等です。L1 16 KBからL2 256 KBへのドロップは半分以下です。

これには大きな影響があると思います。

フレームの予算見積もり


ゲーム開発者は、CPUとメモリの負荷に制限（予算）を設定するのが一般的です。しかし、帯域幅の予算を見たことがありません。

最近のゲームでは、FPSは成長を続けています。現在は60 FPSです。 VRは90 Hzの周波数で動作します。 144 Hzのゲームモニターを持っています。それは素晴らしいので、60 FPSはたわごとのようです。私は古いモニターに戻ることは決してありません。 eスポーツとストリーマーTwitchは240 Hzを監視します。今年、AsusはCESで360 Hzのモンスターを紹介しました。

私のプロセッサの上限は約40 GB /秒です。大きな数字のようです！ただし、240 Hzの周波数では、フレームあたり167 MBしか取得されません。現実的なアプリケーションでは、144 Hzで5 GB /秒のトラフィックを生成できます。これは、フレームあたりわずか69 MBです。

この観点から問題を評価することは有用であるように私には思えます。これにより、一部のアイデアが実行不可能であることは明らかです。240 Hzに到達するのは簡単ではありません。これだけでは起こりません。
