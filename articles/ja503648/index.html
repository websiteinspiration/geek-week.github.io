「ss -s」の出力はどういう意味ですか

netstatユーティリティはssユーティリティに置き換えられ、集約された（概要）情報「ss -s」（または「ss --summary」）の出力がニーズの監視に使用されることがよくあります。ただし、表示された各フィールドは何を意味しますか？

# ss -s
Total: 15046 (kernel 16739)
TCP:   39306 (estab 11458, closed 25092, orphaned 110, synrecv 0, timewait 24929/0), ports 0

Transport Total     IP        IPv6
*	  16739     -         -        
RAW	  0         0         0        
UDP	  15        5         10       
TCP	  14214     1214      13000    
INET	  14229     1219      13010    
FRAG	  0         0         0

結局のところ、微妙な点があります。

真実の源はソースコードです。iproute2 / misc / ss.cにあるユーティリティ«ss»のソースコード。「ss -s」の出力は、print_summary関数で行われます。

バージョン4.17.0より前では、「ss -s」の出力が上に表示されています。後90ee99dを犯し、それは次のようになり始めました。

# ss -s
Total: 15046
TCP:   39306 (estab 11458, closed 25092, orphaned 110, timewait 24929)

Transport Total     IP        IPv6
RAW	  0         0         0        
UDP	  15        5         10       
TCP	  14214     1214      13000    
INET	  14229     1219      13010    
FRAG	  0         0         0

ご覧のとおり、「合計」行から「カーネル」フィールドが削除されています。「TCP」の行で、フィールド「synrecv」が削除され、1つの（最初の）番号「timewait」が残り、フィールド「ports」が削除されました。テーブル "Transport"の行 "*"が削除されました。コミットの一般的な意味は次のように翻訳できます：「長い間壊れていたものを出力から削除しました。」

何が壊れましたか？


でprint_summaryバージョン4.16.0が使用される構造のあるslabstat呼び出すことによって満たされ、get_slabstatを。get_slabstat呼び出しをシェルコマンドに変換しようとすると、次のようになります。

# egrep '^(sock|tcp_bind_bucket|tcp_tw_bucket|tcp_open_request|skbuff_head_cache)' /proc/slabinfo | awk '{ print $1, $2; }'

「/ proc / slabinfo」のキーは、「slabstat」構造のフィールドに対応しています。

struct slabstat {
	int socks;       // sock, net/core/sock.c:sk_init,   2.6.12
	int tcp_ports;   // tcp_bind_bucket, net/ipv4/tcp.c:tcp_init
	int tcp_tws;     // tcp_tw_bucket, net/ipv4/tcp.c:tcp_init,   2.6.14
	int tcp_syns;    // tcp_open_request, net/ipv4/tcp.c:tcp_init,   2.6.13
	int skbs;        // skbuff_head_cache, net/core/skbuff:skb_init,  
};

つまり、今日「/ proc / slabinfo」から取得されるすべての値のうち、「ポート」フィールドに表示される値「tcp_bind_bucket」のみを取得できます。私は彼の本「TCP / IPアーキテクチャ、設計、およびLinuxでの実装」から説明します。S. SethとMA Venkatesulu 2008、IEEE Computer Society」：
This structure keeps information about the port number usage by sockets and the way the port number is being used. The information is useful enough to tell the new binding socket whether it can bind itself to a particular port number that is already in use. The data structure also keeps track of all the socket’s that are associated with this port number.
もう一度同じですが、写真の形をしています（私からのアーティストは



まあまあです）。スラブ「sock」（2.6.12で削除）にゼロ以外の値があります。説明は非常に単純です。文字列比較の特殊性のため、「sock」で始まるキーを持つ数値はすべてフィールド値に入ります。私の場合、これは「sockfs」に関連する「net / socket.c：init_inodecache」からの「sock_inode_cache」です-「socket struct」を含む「inodes」のリスト（たぶん、これはユーティリティの作成者が意図したものではないかもしれません）：

# egrep '^sock' /proc/slabinfo | awk '{ print $1, $2; }'
sock_inode_cache 16739

まあ、「tcp_bind_bucket」は、カーネルビルドオプションのために欠落しています（したがって、「ports」フィールドの値は常に「0」です）。
