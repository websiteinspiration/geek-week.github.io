作業はキャッシュから始まります


すでにクライアントアプリケーションにデータを送信しています。これは、ページの読み込み中にできる限り早く行います。つまり、データを配信する最も速い方法は、クライアントからの情報の要求またはサーバーのイニシアチブでのクライアントへの情報の送信に関連する手順を含まない方法のみです。これは、キャッシュが前面に来るページ形成へのこのアプローチを使用して行うことができます。ただし、これは、ごく短時間ではありますが、ユーザーに古い情報を表示する必要があることを意味します。このアプローチを使用すると、ページを読み込んだ後、すぐにフィードとストーリーのキャッシュされたコピーをユーザーに表示し、最新のデータが利用可能になった後、これをすべてそのようなデータに置き換えます。

Reduxを使用してinstagram.comのステータスを管理します。その結果、上記のスキームの全体的な実装計画は次のようになります。 ReduxリポジトリのサブセットをクライアントのindexedDBテーブルに格納し、ページが初めて読み込まれたときにこのリポジトリにデータを入力します。ただし、indexedDBの操作、サーバーからのデータのダウンロード、ユーザーによるページの操作は非同期プロセスです。その結果、問題が発生する可能性があります。それらは、ユーザーが古いキャッシュ状態で作業しているという事実にあり、サーバーから受信したときにユーザーのアクションを新しい状態に適用する必要があります。

たとえば、キャッシュの操作に標準のメカニズムを使用すると、次の問題が発生する可能性があります。キャッシュとネットワークからのデータの並列ロードを開始しています。キャッシュからのデータはネットワークデータよりも速く準備ができるので、ユーザーに表示します。次に、たとえばユーザーは投稿を気に入ったが、最新の情報を伝えるサーバーの応答がクライアントに届いた後、この情報はいいねされた投稿に関する情報を上書きする。この新しいデータには、ユーザーがキャッシュバージョンの投稿を設定したいいね！に関する情報はありません。これがどのように見えるかです。


ユーザーがキャッシュされたデータを操作するときに発生する競合状態（Reduxアクションは緑色で強調表示され、ステータスは灰色です）

この問題を解決するには、ユーザーのアクションに応じてキャッシュされた状態を変更し、これらのアクションに関する情報を保存する必要がありました。これにより、サーバーから新しい状態に適用されたときにそれらを再現できます。 Gitまたは別のバージョン管理システムを使用したことがある場合、このタスクはなじみのあるものに見えるかもしれません。テープのキャッシュ状態がローカルリポジトリブランチであり、最新のデータを含むサーバーの応答がマスターブランチであるとします。その場合、再配置操作を実行したい、つまり、あるブランチに記録された変更（いいね、コメントなど）を取得して別のブランチに適用したいと言えます。

中間状態を使用して競合状態によって引き起こされる問題を解決します（Reduxアクションは緑色で強調表示され、状態は灰色です）

中間状態のおかげで、既存のすべてのレデューサーを再利用できます。さらに、これにより、現在の状態とは別に中間状態（最新のデータを含む）を保存できます。また、中間状態での作業はReduxを使用して実装されているため、この状態を使用するアクションを送信するだけで十分です。

API


中間状態APIは、2つの主要な関数で構成されています。これはあるstagingActionとstagingCommit：

function stagingAction(
    key: string,
    promise: Promise<Action>,
): AsyncAction<State, Action>

function stagingCommit(key: string): AsyncAction<State, Action>

変更を元に戻したり、境界ケースを処理したりすることを目的とした関数は他にもいくつかありますが、ここではそれらを考慮しません。

この関数stagingActionは、中間状態に送信する必要があるイベントによって解決されたpromiseを受け入れます。この関数は、中間状態を初期化し、初期化以降に送信されたアクションを監視します。これをバージョン管理システムと比較すると、ローカルブランチの作成を扱っていることがわかります。進行中のアクションはキューに入れられ、新しいデータが到着した後の暫定状態に適用されます。

関数stagingCommit現在の状態を中間の状態に置き換えます。さらに、中間状態で実行される非同期操作が完了することが予想される場合、システムはこれらの操作が完了するのを待ってから置き換えます。これは、（キャッシュを格納するブランチからの）ローカル変更がマスターブランチ（サーバーから受信した新しいデータの上）に適用される場合の再配置操作に似ており、状態のローカルバージョンが関連しているという事実につながります。

中間状態の作業システムを有効にするために、ルートリデューサーをリデューサーのエクステンダー機能でラップしました。彼はアクションを処理しますstagingCommit以前に保存したアクションを新しい状態に適用します。これらすべてを利用するには、アクションを送信するだけでよく、それ以外はすべて自動的に実行されます。たとえば、新しいテープをロードしてそのデータを中間状態にしたい場合は、次のようにします。

function fetchAndStageFeed() {
    return stagingAction(
        'feed',
        (async () => {
            const {data} = await fetchFeedTimeline();
            return {
                type: FEED_LOADED,
                ...data,
            };
        })(),
    );
}

//         
store.dispatch(fetchAndStageFeed());

//   ,    stagingCommit,
//      'feed'

//     
store.dispatch(stagingCommit('feed'));
