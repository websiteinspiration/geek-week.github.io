LinuxでC ++プログラムを開発する際のPVS-Studio静的アナライザーの知識

PVS-Studioは、C、C ++、C＃、およびJavaでのプロジェクトの分析をサポートしています。アナライザーは、Windows、Linux、およびmacOSシステムで使用できます。この投稿では、Linux環境でのCおよびC ++で記述されたコードの分析に焦点を当てます。

取り付け


LinuxでPVS-Studioをインストールするには、ディストリビューションのタイプに応じて、いくつかの方法があります。最も便利で好ましい方法はリポジトリを使用することです。これにより、新しいバージョンがリリースされたときにアナライザーを自動的に更新できます。2番目のオプションは、ここからダウンロードできるインストールパッケージを使用することです。

インストールするために完了する必要があるコマンドは、使用しているLinuxディストリビューションによって異なります。たとえば、Debianベースのシステムの場合、リポジトリからのインストールは次のようになります。

wget -q -O - https://files.viva64.com/etc/pubkey.txt | sudo apt-key add -

sudo wget -O /etc/apt/sources.list.d/viva64.list \
https://files.viva64.com/etc/viva64.list

sudo apt update
sudo apt install pvs-studio

インストールパッケージからPVS-Studioをインストールするには、gdebiユーティリティを使用できます。

sudo gdebi pvs-studio-VERSION.deb

インストールプロセスの詳細については、ドキュメントの「LinuxでのPVS-Studioのインストールと更新」セクションをご覧ください。また、Debianベースではないシステムに関する情報もそこにあります。

インストール後、ライセンスデータを入力する必要があります。これを行うには、次のコマンドを使用します

pvs-studio-analyzer credentials NAME KEY [-o LIC-FILE]

どこNAMEとKEYは、ライセンスが登録されている名前とライセンスキーです。オプションの-oパラメーターを使用すると、ライセンスファイルが生成されるパスを指定できます。デフォルトでは、これは〜/ .config / PVS-Studio /ディレクトリに保存されます。

「ダウンロードしてPVS-Studioを試す」ページで試用キーを入手できます。

プロジェクトの検証


アナライザーをインストールした後、プロジェクトのチェックを開始できます。これを行うには、主に2つの方法があります。

  1. コンパイル監視;
  2. アセンブリシステムから直接、アセンブリプロセス中に開始します。

最初の方法を最初に検討しましょう。Linuxで監視を開始するには、straceユーティリティが必要です。アナライザーはこれを使用して、プロジェクトのアセンブリ中に起動されたプロセスに関する情報を収集します。

起動は次のコマンドで実行されます。

pvs-studio-analyzer trace -- make

この例ではmakeを使用してビルドを使用していますが、ビルドを開始するために使用する他のコマンドをmake呼び出しの代わりに使用できます。コマンドラインパラメータは通常の方法で渡すことができます。

アセンブリ後、straceは、アナライザーがソースコードの検証に使用するファイルを作成します。次のコマンドで分析を実行できます

pvs-studio-analyzer analyze -o /path/to/project.log

出力は結果がエンコードされたファイルになり、サポートされている形式のいずれかに変換できます。次のセクションでレポートの操作を検討します。

straceの使用に加えて、compile_commands.jsonファイル（JSONコンパイルデータベース）に基づいて分析を実行することもできます。多くのビルドシステムでは、コンパイラ呼び出しコマンドをエクスポートできます。または、このためにBEARユーティリティを使用できます。この場合、次のコマンドで分析を実行できます

pvs-studio-analyzer analyze –f /path/to/compile_commands.json

アナライザーは、実行ファイルの名前で使用されるコンパイラーを認識することに注意してください。チェック中に「コンパイルユニットが見つかりません」というエラーが表示される場合は、–compilerまたは–cパラメーターを使用してコンパイラーの名前を指定してみてください。
pvs-studio-analyzer analyze -c MyCompiler

これは、クロスコンパイルし、非標準の実行可能ファイル名でコンパイラを使用するときに必要になる場合があります。

監視モードでの起動に加えて、アナライザーをアセンブリシステムまたはIDEに統合できます。GitHubのPVS_Studioリポジトリの統合がすでに構成されているプロジェクトの例を見つけることができます。

  • pvs-studio-cmake-examples
  • pvs-studio-qmake-examples
  • pvs-studio-makefile-examples
  • pvs-studio-waf-examples
  • pvs-studio-eclipse-examples

Linuxで分析を実行する方法の詳細については、ドキュメントを参照してください。

レポートを操作する


プロジェクトをチェックした後、アナライザーはコード化されたレポートを作成します。サポートされている形式のいずれかに変換するには、PVS-Studioと共にインストールされるplog-converterユーティリティを使用します。

ユーティリティは次の形式をサポートしています。

  • xml-SonarQubeのプラグインによってサポートされる、分析結果の追加処理のための便利なフォーマット。
  • csvは、表形式のデータを表示するためのテキスト形式です。
  • errorfile-gccおよびclangの出力形式。
  • tasklist-QtCreatorで開くことができるエラー形式。
  • html-分析結果の簡単な説明を含むhtmlレポート。
  • fullhtml-分析結果をさまざまなパラメーターでソートし、ソースコードをナビゲートするHTMLレポート。

fullhtml形式は、警告が発生したソースコード行に移動できるため、レポートを表示するのに最も便利です。次のコマンドを使用すると、アナライザーレポートをこの形式に変換できます。

plog-converter -a GA:1,2 -t fullhtml /path/project.log -o /path/report_dir

コマンドが実行されると、レポートファイルが置かれるディレクトリ/パス/ report_dirが作成されます。-a

スイッチに注意してください。レポートに含める警告を指定できます。これは、アナライザの出力をフィルタリングする必要がある場合に便利です。上記のチームは、1番目と2番目の信頼レベル（高および中）の一般分析グループからの警告のみを含むレポートを作成します。 レポートの例：

アナライザの応答抑制


静的アナライザーでコードをチェックすると、誤検知または単に不要なメッセージ（ノイズ）が発生する可能性があります。 PVS-Studioは、このような警告を抑制するメカニズムを提供します。アラームを個別に抑制するには、ドキュメントのセクション「誤ったアラートの抑制」で説明されている方法のいずれかを使用できます。

また、古いコードを分析する場合、すべてのメッセージを大幅に抑制する必要がある場合があります。原則として、これは既存のコードベースに追加されたコードのみをチェックするために必要です。これには、pvs-studio-analyzerユーティリティの抑制パラメーターが使用されます。

次のコマンドを使用して、レポート内のメッセージをまとめて抑制することができます。

pvs-studio-analyzer suppress /path/to/report.log

抑制されたメッセージに関する情報は、プロジェクトの横にあるsuppress_base.jsonファイルに格納されます。このようなメッセージは、その後の検査でレポートから除外されます。

このメカニズムについては、こちらのドキュメントで詳しく説明しています。
