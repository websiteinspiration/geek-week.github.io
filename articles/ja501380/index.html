ボイラープレートコードがない場合のビジネス価値と分析に焦点を当てます。DWHをコードベースとして管理：バージョン管理、レビュー、自動テスト、CI。モジュール性、拡張性、オープンソース、コミュニティ。わかりやすいユーザードキュメントと依存関係の視覚化（データリネージ）。

短いレビュー


DBTフレームワークは、頭字語ELT（Extract-Transform-Load）のTに関するすべてです。

BigQuery、Redshift、Snowflakeなどの生産的でスケーラブルな分析データベースの登場により、データウェアハウスの外で変換を行う意味がなくなりました。 

DBTはソースからデータをアンロードしませんが、（内部または外部ストレージの）ストレージに既にロードされているデータを操作する大きな機会を提供します。


DBTの主な目的は、コードを取得してSQLでコンパイルし、リポジトリでコマンドを正しい順序で実行することです。

DBTプロジェクトの構造


プロジェクトは、次の2種類のディレクトリとファイルで構成されています。

  • モデル（.sql）-SELECTクエリによって表現される変換の単位
  • 設定ファイル（.yml）-パラメータ、設定、テスト、ドキュメント

基本的なレベルでは、作業は次のように構成されています。

  • ユーザーが任意の便利なIDEでモデルコードを準備する
  • CLIを使用してモデルを起動し、DBTがSQLでモデルコードをコンパイルします
  • コンパイルされたSQLコードは、指定された順序（グラフ）でウェアハウスで実行されます。

すべてがSELECTです


これは、Data Build Toolフレームワークのキラー機能です。つまり、DBTは、ウェアハウスでのクエリの具体化に関連するすべてのコード（CREATE、INSERT、UPDATE、DELETE、ALTER、...コマンドのバリエーション）を抽象化します。

どのモデルでも、結果のデータセットを定義する1つのSELECTクエリを作成する必要があります。

同時に、変換ロジックはマルチレベルにすることができ、他のいくつかのモデルからのデータを統合できます。注文のショーケース（f_orders）を作成するモデルの例：

ここでどんな興味深いことがわかりますか？

1つ目は、CTE（Common Table Expressions）を使用して、多くの変換とビジネスロジックを含むコードを整理して理解する

2つ目：モデルコードは、SQLとJinja言語（テンプレート言語）の混合です。

この例では、forループを使用して、セット式で指定された各支払い方法の金額を形成しています。ref関数も使用されます-コード内で他のモデルを参照する機能：

  • コンパイル時に、refはリポジトリ内のテーブルまたはビューへのターゲットポインターに変換されます
  • refを使用すると、モデルの依存関係のグラフを作成できます

DBTにほぼ無制限の可能性を追加するの はJinjaです。最も一般的に使用されるもの：

  • if / elseステートメント-分岐ステートメント
  • forループ-ループ
  • 変数-変数
  • マクロ-マクロを作成する

実体化：テーブル、ビュー、インクリメンタル


実体化戦略-結果として得られるモデルデータのセットがリポジトリに格納される方法。

基本的な検討事項は次のとおりです。

  • テーブル-ストレージ内の物理テーブル
  • 表示-表示、リポジトリ内の仮想テーブル

より複雑な具体化戦略があります。

  • 増分-増分ロード（大きなファクトテーブル）。新しい行が追加され、変更された行が更新され、削除された行がクリアされます 
  • エフェメラル-モデルは直接具体化せず、CTEとして他のモデルに参加します
  • 自分で追加できるその他の戦略

具体化戦略に加えて、特定の倉庫の最適化の機会が開かれます。次に例を示します。

  • スノーフレーク：一時テーブル、マージ動作、テーブルのクラスタリング、権限のコピー、ビューの保護
  • Redshift：Distkey、Sortkey（インターリーブ、複合）、遅延バインディングビュー
  • BigQuery：テーブルのパーティショニングとクラスタリング、マージ動作、KMS暗号化、ラベルとタグ
  • Spark：ファイル形式（parquet、csv、json、orc、delta）、partition_by、clustered_by、buckets、incremental_strategy

現在、次のリポジトリがサポートされています。

  • Postgres
  • レッドシフト
  • Bigquery
  • スノーフレーク
  • プレスト（一部）
  • スパーク（一部）
  • Microsoft SQL Server（コミュニティアダプタ）

モデル依存関係グラフ


彼は依存関係のツリーです。彼はDAG（Directed Acyclic Graph-Directional Acyclic Graph）です。

DBTは、すべてのプロジェクトモデルの構成に基づいてグラフを作成します。または、モデル内の他のモデルへのref（）リンクを作成します。グラフを作成すると、次のことが可能になります。

  • 正しい順序でモデルを実行する
  • ウィンドウドレッシングの並列化
  • 任意のサブグラフを実行する

データ品質とドキュメント


モデル自体の形成に加えて、DBTを使用すると、次のような結果のデータセットに関する多くのアサーションをテストできます。

  • nullではない
  • ユニークな
  • 参照整合性-参照整合性（たとえば、注文テーブルのcustomer_idは顧客テーブルのidに対応します）
  • 一致する有効なリスト

独自のテスト（カスタムデータテスト）を追加できます。たとえば、1日、1週間、1か月前のインジケーターを使用した収益の偏差率（％）などです。SQLクエリとして定式化された仮定はすべてテストになり得ます。

このようにして、データの不要な逸脱やエラーをストレージのストアフロントでキャッチできます。

ドキュメントに関しては、DBTは、メタデータとコメントをモデルレベルおよび属性でさえ追加、バージョン管理、および配布するためのメカニズムを提供します。 

構成ファイルレベルでのテストとドキュメントの追加は次のようになります。

マクロとモジュール


DBTの目的は、SQLスクリプトのセットになることではなく、独自の変換を構築してこれらのモジュールを配布するための強力で機能豊富なツールをユーザーに提供することです。

マクロは、モデル内の関数として呼び出すことができる構造と式のセットです。マクロを使用すると、DRY（Do n't Repeat Yourself）エンジニアリング原則に従って、モデルとプロジェクトの間でSQLを再利用できます。

マクロの例：

{% macro rename_category(column_name) %}
case
 when {{ column_name }} ilike  '%osx%' then 'osx'
 when {{ column_name }} ilike  '%android%' then 'android'
 when {{ column_name }} ilike  '%ios%' then 'ios'
 else 'other'
end as renamed_product
{% endmacro %}

そしてその使用：

{% set column_name = 'product' %}
select
 product,
 {{ rename_category(column_name) }} --  
from my_table

DBTには、ユーザーが個々のモジュールとマクロを公開して再利用できるようにするパッケージマネージャーが付属しています。

これは、次のようなライブラリをダウンロードして使用する機能を意味します。

  • dbt_utils：日付/時刻、サロゲートキー、スキーマテスト、ピボット/アンピボットなどの操作
  • SnowplowやStripeなどのサービス用の既製のショーケーステンプレート 
  • Redshiftなどの特定のデータウェアハウス用のライブラリ 
  • ロギング -DBT ロギングモジュール

パッケージの完全なリストはdbt hubで入手できます。

さらに多くの機能


ここでは、私とチームがWheelyでデータウェアハウスを構築するために使用するその他の興味深い機能と実装について説明します。

ランタイム環境の分離DEV-TEST-PROD


同じDWHクラスター内でも（異なるスキーム内で）。たとえば、次の式を使用します。

with source as (
 
   select * from {{ source('salesforce', 'users') }}
   where 1=1
   {%- if target.name in ['dev', 'test', 'ci'] -%}           
       where timestamp >= dateadd(day, -3, current_date)   
   {%- endif -%}
 
)

このコードは文字通り言う：開発、テスト、ci環境の場合、過去3日間のみデータを取得し、それ以上は取得しない。つまり、これらの環境での実行ははるかに高速になり、必要なリソースが少なくなります。立ち上げたときのprod環境、フィルタ条件が無視されます。

代替列エンコードの具体化


Redshiftは、個々の列ごとにデータ圧縮アルゴリズムを指定できる列DBMSです。最適なアルゴリズムを選択すると、占有ディスクスペースを20〜50％削減できます。

マクロredshift.compress_table推奨列が符号化アルゴリズムを使用して新しいテーブルを作成し、ANALYZE COMPRESSIONコマンドを実行しますが、それにデータを転送、セグメンテーションキー（dist_key）とソート（key_key）で示され、必要に応じて古いコピーを削除します。

マクロ署名：

{{ compress_table(schema, table,
                 drop_backup=False,
                 comprows=none|Integer,
                 sort_style=none|compound|interleaved,
                 sort_keys=none|List<String>,
                 dist_style=none|all|even,
                 dist_key=none|String) }}

ストレージの自動化


UDF（ユーザー定義関数）など、使用するストレージの機能の拡張を使用する場合、これらの関数のバージョン管理、アクセス制御、および新しいリリースの自動ロールアウトは、DBTでの実装に非常に便利です。

PythonではUDFを使用して、ハッシュ値、メールアドレスのドメイン、およびビットマスクのデコードを計算します。

任意のランタイム（dev、test、prod）でUDFを作成するマクロの例：

{% macro create_udf() -%}
 
 {% set sql %}
       CREATE OR REPLACE FUNCTION {{ target.schema }}.f_sha256(mes "varchar")
           RETURNS varchar
           LANGUAGE plpythonu
           STABLE
       AS $$  
           import hashlib
           return hashlib.sha256(mes).hexdigest()
       $$
       ;
 {% endset %}
  
 {% set table = run_query(sql) %}
 
{%- endmacro %}

Wheelyでは、PostgreSQLベースのAmazon Redshiftを使用しています。Redshiftの場合、テーブルの統計を定期的に収集し、ディスク領域を解放することが重要です。それぞれANALYZEコマンドとVACUUMコマンドです。

これを行うには、redshift_maintenanceマクロからのコマンドを毎晩実行します。

{% macro redshift_maintenance() %}
 
   {% set vacuumable_tables=run_query(vacuumable_tables_sql) %}
 
   {% for row in vacuumable_tables %}
       {% set message_prefix=loop.index ~ " of " ~ loop.length %}
 
       {%- set relation_to_vacuum = adapter.get_relation(
                                               database=row['table_database'],
                                               schema=row['table_schema'],
                                               identifier=row['table_name']
                                   ) -%}
       {% do run_query("commit") %}
 
       {% if relation_to_vacuum %}
           {% set start=modules.datetime.datetime.now() %}
           {{ dbt_utils.log_info(message_prefix ~ " Vacuuming " ~ relation_to_vacuum) }}
           {% do run_query("VACUUM " ~ relation_to_vacuum ~ " BOOST") %}
           {{ dbt_utils.log_info(message_prefix ~ " Analyzing " ~ relation_to_vacuum) }}
           {% do run_query("ANALYZE " ~ relation_to_vacuum) %}
           {% set end=modules.datetime.datetime.now() %}
           {% set total_seconds = (end - start).total_seconds() | round(2)  %}
           {{ dbt_utils.log_info(message_prefix ~ " Finished " ~ relation_to_vacuum ~ " in " ~ total_seconds ~ "s") }}
       {% else %}
           {{ dbt_utils.log_info(message_prefix ~ ' Skipping relation "' ~ row.values() | join ('"."') ~ '" as it does not exist') }}
       {% endif %}
 
   {% endfor %}
 
{% endmacro %}

結論


DWHを調理して消費することは、スムージーを飲むことと同じくらい楽しくて有益です。DBTは、Jinja、カスタム拡張（モジュール）、コンパイラ、エンジン（エグゼキュータ）、およびパッケージマネージャで構成されています。これらの要素を一緒に収集すると、データウェアハウスの本格的な作業環境が得られます。今日、DWH内で変換を管理するためのより良い方法はほとんどありません。



DBT開発者が従う信念は、次のように定式化されています。

  • 複雑な分析ロジックを表現するには、GUIではなくコードが最適です
  • データを操作することで、ソフトウェア開発のベストプラクティスを適応させる必要があります（ソフトウェアエンジニアリング）

  • 重要なデータインフラストラクチャは、オープンソースソフトウェアとしてユーザーコミュニティによって制御される必要があります。
  • 分析ツールだけでなく、コードがますますオープンソースコミュニティの一部になるでしょう。

これらの中核的な信念は、850社以上の企業で現在使用されている製品を生み出し、将来的に作成される多くの興味深い拡張機能の基礎を形成しています。
