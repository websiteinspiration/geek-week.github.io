マイクロサービスブログの作成-パート1「一般的な説明」

この記事で私は私たちのcを共有したいと思います セルゲイマスロフ 例として「ブログの作成」タスクを使用してマイクロサービスアーキテクチャを使用して典型的な問題を解決するためのベストプラクティス（読者がブログの構成を想像でき、これが機能に関する質問を引き起こさないことを期待して）：

したがって、ブログはgolangで記述された5つのマイクロサービスで構成されます。

  • ゲートウェイAPI（api-gw）-リクエストのルーティング、認証、ロギング、トレースを担当します
  • ユーザー（ユーザー）-ユーザーの登録/認証、要求のロギング、トレース
  • 記事（投稿）-記事の作成/読み取り/変更/削除（CRUD）、要求のロギング、トレース、および承認
  • コメント（コメント）-リクエストの作成/読み取り/変更/削除（CRUD）、ロギング、トレース、認証
  • カテゴリー（カテゴリー）-カテゴリーの作成/読み取り/変更/削除（CRUD）、要求のロギング、トレース、および許可

クライアントアプリケーション（ウェブ/フロントエンド）はvue.jsに実装され、REST APIを介してマイクロサービスとやり取りし、マイクロサービス自体はgRPCを介して互いにやり取りします。

ストレージとして、MongoDBを使用します。

最小限の労力で活発に開発中のプロジェクトでAPIドキュメント（swagger形式）を最新に保つ方法を別のチェリーで示します。

ブログコンポーネント図




各マイクロサービスは個別のDockerコンテナーに実装され、プロジェクトはdocker-composeを使用して起動されます。

この例ではすぐに予約します。開発プロセスを簡略化するために、本番環境では使用しない2つの仮定を使用します。

  • データベースは、Dockerコンテナーにデプロイされます。このアプローチは、ストレージの信頼性を低下させます（HighLoad 2018で説明されているスキームを除く）。
  • プロジェクト全体が1つのgitリポジトリでホストされています。このアプローチは、マイクロサービスアーキテクチャの基本原則の1つである分離と矛盾し、コンポーネント間の接続の可能性を高めます。

開発プロセスの構築方法


先に述べたように、マイクロサービス間の相互作用はgRPCに基づいて構築されます。一言で言えば、gRPCはリモートプロシージャ（RPC）を呼び出すためにGoogleが開発した高性能フレームワークです。HTTP/ 2の上で動作します。GRPCは、いわゆるprotofile（以下の例を参照）に基づいています。その主なタスクは、2つのことをコンパクトな形式で宣言することです。

  • サービスインターフェイスの完全なリスト（APIインターフェイスのアナログ）を提供します。
  • 各インターフェイスの入力に何が供給され、出力で何が得られるかを説明します。

以下に、例として、カテゴリサービスのプロトファイルを示します。

syntax = "proto3";
package protobuf; 
import "google/api/annotations.proto";

//   Category
service CategoryService {
 
  // 
  rpc Create (CreateCategoryRequest) returns (CreateCategoryResponse) {
    option (google.api.http) = {
      post: "/api/v1/category"
    };
  }              

  // 
  rpc Update (UpdateCategoryRequest) returns (UpdateCategoryResponse) {
    option (google.api.http) = {
      post: "/api/v1/category/{Slug}"
    };
  }     

  // 
  rpc Delete (DeleteCategoryRequest) returns (DeleteCategoryResponse) {
    option (google.api.http) = {
      delete: "/api/v1/category/{Slug}"
    };
  }     

  //   SLUG
  rpc Get (GetCategoryRequest) returns (GetCategoryResponse) {
    option (google.api.http) = {
      get: "/api/v1/category/{Slug}"
    };
  }     

  //
  rpc Find (FindCategoryRequest) returns (FindCategoryResponse) {
    option (google.api.http) = {
      get: "/api/v1/category"
    };
  }                    

}

//------------------------------------------
//  CREATE
//------------------------------------------
message CreateCategoryRequest {
  string ParentId = 1;
  string Name = 2;
  string Path = 3;
}
message CreateCategoryResponse {
  Category Category = 1;
}

//------------------------------------------
//  UPDATE
//------------------------------------------
message UpdateCategoryRequest {
  string Slug = 1;
  string ParentId = 2;
  string Name = 4;
  string Path = 5;
  int32 Status = 6;
}
message UpdateCategoryResponse {
  int32 Status =1;
}

//------------------------------------------
//  DELETE
//------------------------------------------
message DeleteCategoryRequest {
  string Slug = 1;
}
message DeleteCategoryResponse {
  int32 Status =1;
}

//------------------------------------------
//  GET
//------------------------------------------
message GetCategoryRequest {
  string Slug = 1;
}
message GetCategoryResponse {
  Category Category = 1;
}

//------------------------------------------
//  FIND
//------------------------------------------
message FindCategoryRequest {
  string Slug = 1;
}
message FindCategoryResponse {
  repeated Category Categories = 1;
}


//------------------------------------------
//  CATEGORY
//------------------------------------------
message Category {
  string Slug = 1;
  string ParentId = 2;
  string Path = 3;
  string Name = 4;
  int32 Status = 5;
}

プロトファイルが必要な理由を一般的に理解したところで、マイクロサービスの開発プロセスがどのようになるか見てみましょう。

  1. protofileでサービスの構造を記述します。
  2. コードジェネレーター（./bin/protogen.sh）を起動すると、サーバーコードの主要部分が生成され、クライアントコードが作成されます。たとえば、API Gatewayの場合、最新のドキュメントがswagger形式で作成されます。
  3. 自分の手で行う必要があるのは、特別なファイル/protobuf/functions.goにインターフェイスを実装するためのコードを記述することだけです。

さらに、マイクロサービスの1つに変更を加える場合は、上記のアルゴリズムに従って続行します。プロトファイルを編集し、protogenを実行し、functions.goの実装を編集すると、変更がドキュメントとクライアントに自動的に「残されます」。「ゲートウェイAPIのマイクロサービスパート2に関するブログの作成」
