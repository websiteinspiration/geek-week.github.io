しかし今日、私はここのキャンペーンで浮上した一つの「問題」について考えたいと思います。粒子または化学要素の無秩序な動きを想像してみてください-それらは条件付き有限体積で移動し、互いに衝突するだけ、または反発する-最も純粋な形のカオス。そのようなシステムが起動された場合、この一見役に立たない蟻塚でも、構造化された動きが時々観察されます。特定の瞬間にランダムな衝突が発生し、衝突の方向（衝撃）が多かれ少なかれ一方向に変化し、その後衝突します。粒子は一方向に動き、波を形成します。将来、外部衝突からのこの波は減衰し始め、再び無秩序な画像が表示されます。

それら。本質的にランダムに構造を形成しますが、構造はこのランダムの一時的な形式にすぎません。このように流れると、カオスはさまざまな形を取り、合理的なものとして認識されます。順序付けられました。概して、合理的とは、そのような繰り返しが人に気付かれるように繰り返されるものです。ただし、ここではこれ以上掘り下げることはしませんが、もう少し広範に考えます。

ほとんどの場合、いわゆる地球上の「生命」は、化学元素の無秩序な相互作用の結果として生まれました。ある分子が別の分子から電子を引き出すことができなかった場合、またはその逆の場合、その分子は別の分子と分け隔てられていなければ、このような人生は決して起こらなかっただろう。この機会自体それは、化合物が元素から、また化合物から最初のレプリカントと有機体から出現したという事実だけによるものでした。したがって、デジタルライフを作成したい場合は、相互作用の機会を与える必要があります。

私たちが宇宙の偉大な建築家であり、初等から始めてその法則をすべて正確に知っていれば、それらをデジタル化する問題は深刻な問題にはなりません。しかし理由は私たちは（残念ながら）そうではないので、生物学的生命の到来と同じ条件を推測して作成でき、それによってデジタル生命を可能にするという事実にのみ頼ることができます。

私がフリーコードの原則を説明しようとしているのはこのためです。可能な最大の無料コード。人生がどのように配置されるべきかについての私たち自身の理解から解放されます-そのような理解はおそらく誤りであるか、あまりに不完全であり、あまり意味がありません。私たちの小さな生き物は、身の回りのあらゆるものを研究し、説明するために、一生懸命に努力しています。しかし、誰も試みを禁じませんでした。

無秩序なコードを書くとき、必然的にフォークに出くわします。ランダム体験。それら。無秩序に移動するか、経験の形成を可能にします。それら。パーティクルの衝突だけでなく、相互作用も可能にします。私たちは、経験の伝達と再生なしには考えられない生物学的生命の類似体を探しています。 RNA / DNAは、伝わる記録された体験です。

一方、どのようにして発展したのか、私たちにはよくわからないので、それを決定することはできません。この誘惑が取り除かれなければ、間違いは避けられません。言い換えれば、私たちはプログラムに経験がどのように見えるべきか、どのように形成されるべきかをプログラムに伝えることができません。しかし、私たちは彼女に彼が原則として形成されることができる道具を与えることができます。

経験の蓄積の構造は非常に簡単に説明できるように思えたので、私は非常に長い間苦労しました。単純さについての私たち自身の考えに基づいて私たちには単純に思える単純さを考えることは、私たち自身を欺く確かな方法です。本当のシンプルさは本当に複雑だと思います。しかし、私はここで暴力団に入らないことを提案します。

ランダムな入力数値セットに応答して、実行可能なコード行をランダムに生成したとします。応答の生成または反応は経験であると想定できます。何かが同じ入力信号に対して同じように安定して反応する場合、それは経験を使用していると言えます。そのため、生成された文字列を覚えておきたいと思います。しかし、これは直接行うことはできません！この行を取得して、クラスインスタンスの配列変数のどこかに押し込み、IF [input.set in memory_ array] THEN [memory .-> reaction]のようなものを書き込んで、一種の反応ライブラリの作成を開始することはできません。それは無理だ！これが「生物学的記憶システム」が機能する方法であるという事実ではないからです。そして、それらの構造が元々私たちが現在目にするものと同じであったという事実ではありません。

同時に、少なくとも256バイトの文字列を取得してランダムに生成して実行することはできません。これは、最近のコンピューターでは非常に時間がかかり、私たちに適さないためです。そのような方法が理想的である可能性がありますが、悲しいことに、現在の未開発の技術でこれを検証することは不可能です。

加えて、私たちはまだプログラミング言語を持っていますし、誰が言っても、少なくとも何かについての私たち自身の考えがあります。何もないところから何かを構築することはできません。したがって、主観的な外観が必要です。たとえば、あるフォームが別のフォームにどのように移行するか、またその逆の様子、どのようにしてカオスがフォームを簡単に引き継ぎ、ボールが床から持ち上げられてリリースされたかがその床に落ちてマークを残すのと同じように手放す方法を見るしかありません。その結果、正確に何も決定したくありませんが、すべてをランダム化することもできません。

問題を理解するために、コードに2つの可能なアクションがあると想像してください。どちらを選択するかを直接指定することはできません。これは決定のためです。そして、すべてをランダム化する場合、確率を使用する必要があります。何だけ？ 50/50？ 30/70？ 80/20？他の条件が等しいとすると、50/50は確かに最も不明確な関係です。その後、コードは次のようになります。

if random.random() >= 0.5:
      1
else:
     2

このコードの何が問題になっていますか？そして、ループの各反復で、このコードは常に最初のアクションか2番目のアクションのいずれかをランダムに選択するという事実。言い換えれば、そのような理論的な可能性すら持っていないので、そのようなカオスは決して構造化されません。概して、ここでは同じ決定を扱っています。この状況から抜け出す方法があります。これは、action1とaction2自体が、適用される条件を変更する必要があるという事実にあります。選択肢はあると思いますが、以下を提供したいと思います。

ご存知のように、オペレーティングシステムは優先順位を使用して実行キュー内のスレッドのバランスをとります。実行時に実行中のスレッドは徐々に優先度を失いますが、キューでアイドル状態のスレッドはこの優先度を上げます。あるビートでは、後者の優先順位が最初のビートより高くなり、フローが入れ替わり、状況が繰り返されます。したがって、フローは次々とかなり速く交互に変化し、ほぼ同時に実行されているように見えます。

私たちの場合、優先順位の原則により、カーネルで実行するスレッドではなく、制御が移されるコードの部分を変更できます。そして、コードの1つの部分を「ランダム」に、2番目の部分を「経験済み」にすると、それらの代替の変更が、まさに私たちが探しているソリューションになります。

言い換えれば、経験を適用するコードは、次にアクションがランダムになり、ランダムに作用する可能性を毎回増加させ、サイクルの後続の反復で経験を適用する可能性を増加させます。したがって、私たちはカオスの望ましいフローを経験に、そしてその逆を得る-まさに私たちが必要とするものです。

この設計の信じられないほどの利点は、システムの骨化にまったく問題がないことです。これは、たとえば同じニューラルネットワークで知られています。一方、安定性は犠牲になりますが、ここでは必要ありません。

ここで特に強調したい通常の古典的なプログラマーは天国からのマナとしてコードの安定性を必要とすること：認識は安定して認識しなければならず、毎回1回ではなく、サーバーは対応するクライアント要求に対して所定の応答で時々応答する必要があります。 、次に別のものなど等通常のプログラミングでは、コードの不安定性はテストで見つかる明らかなバグです。私たちの場合、明確なコードタスクがないため、この安定性はまったく必要ありません。より正確には、それはまったくありません。しかし、別の言い方をすれば、コードの制御の移行における変更の安定性が必要であり、それを実現します。振り子は安定しているとも言えますが、あるサイクルからあるポイントから反対のポイントまで何とかして行きます。

根拠がないようにするために、上記のすべての考慮事項に基づいて、今日スケッチしたPythonの次の基本コードを確認することをお勧めします。 

import random

deter_numb = 69
p_deter = 0.01
p_random = 0.99
iteration = 1

while True:
        print('iter = ',iteration)
        print('p_rand =  ', p_random)         
        print('p_deter = ', p_deter)
        input()
        if p_random > random.random():
            p_random-=0.01                     #  
            p_deter+=0.01
            print(random.random())    #  
        if p_deter > random.random():       
            p_deter-=0.01                      #  
            p_random+=0.01
            print(deter_numb)            #  
        iteration+=1


このスクリプトを自分で実行してみて、動作を確認することをお勧めします。これは自己均衡コードです。アクション1が立っているように
print(random.random())
-つまり 単なる乱数出力とアクション2-
print(deter_numb)
-確定数の出力。コードでは、その場所にアクションがあります。ここでは、原理を示すためだけに印刷しました。

コントロールが最初の条件（コードのランダムな部分）に渡ると、コード自体が繰り返しの可能性を減らし（p_random- = 0.01-それ自体を減らします）、同時に、エクスペリエンスが適用されるコードの一部が実行される確率を増やします（p_deter + = 0.01-他の確率の増加）。制御を2番目の条件に移すときにも同じことが起こります。したがって、これらのコードの両方が実行されると、それら自体が「スローダウン」し、同時にコードの他の「競合する」部分を「加速」します。

制御がコードのランダムな部分に移る最初の確率は99％で、実験的な確率は1％になることに注意してください。これは、そのような根本的に異なる確率でもシステムがバランスしていることを示すためです。コードを実行すると、この比率が100回の反復で50/50になり、0.5の明示的なピークと約6％の動作近傍（次の3〜500回の反復）で正規分布を形成する方法がわかります。

コードを実行すると、確率が平衡状態になり、約0.5に落ち着き、このポイントを超えてバランスをとるように、その下部のポイントの近くで振り子がバランスをとるようになります。

もう1つの機能がここに表示されます：乱数または決定的なもののいずれかが表示されます（69）、またはそれらの両方が一度に表示されます。つまり、システムは経験を適用してからランダムに行動するか、またはその逆、または経験を適用するか、単に行動することができます。無作為に。つまり、コード自体の実行はランダムのままですが、システムで経験を適用できるようになりました。行動を決定するときに決意をなくしました。それはまさに私たちが必要としていたことです。
