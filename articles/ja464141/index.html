メタクラスとは何ですか？


では、メタクラスとは何でしょうか。これは、他のタイプを説明する特別なタイプです。C＃にはタイプタイプという非常によく似たものがあります。しかし、似ているだけです。Type型の値は任意の型を記述でき、メタクラスはメタクラスが宣言されたときに指定されたクラスの相続人のみを記述できます。

これを行うために、私たちの仮想バージョンのC＃は、Typeの後継であるType Type <T>型を取得します。ただし、タイプ<T>は、タイプTまたはその子孫の記述にのみ適しています。
例を挙げて説明します。

class A { }

class A2 : A { }

class B { }

static class Program
{
    static void Main()
    {
        Type<A> ta;
        ta = typeof(A);  //  
        ta = typeof(A2); //   
        ta = typeof(B);  //   – Type<B>   Type<A>
        ta = (Type<A>)typeof(B); //      -  

        Type tx = typeof(A);
        ta = tx; //   –    Type  Type<A>
        ta = (Type<A>)tx; //   
        Type<B> tb = (Type<B>)tx; // 
    }
}

上記の例は、メタクラスの出現への最初のステップです。タイプタイプ<T>を使用すると、対応する値で記述できるタイプを制限できます。この可能性はそれ自体で役立つ可能性がありますが、メタクラスの可能性はこれに限定されません。

メタクラスと静的クラスメンバー


クラスXに静的メンバーがある場合、メタクラスType <X>はそれに類似したメンバーを取得し、静的ではなくなります。これにより、Xの静的メンバーにアクセスできます。この紛らわしい句を例で説明しましょう。

class X
{
    public static void DoSomething() { }
}

static class Program
{
    static void Main()
    {
        Type<X> tx = typeof(X);
        tx.DoSomething(); //   ,     X.DoSomething();
    }
}

ここで、一般的に言えば、問題が発生します-クラスXで静的メソッドが宣言され、その名前とパラメータセットがTypeクラスのメソッドの1つの名前とパラメータセットと一致し、その継承者がType <X>である場合はどうなりますか？この問題を解決するためのかなり単純なオプションがいくつかありますが、ここでは詳しく説明しません。単純化のために、紛争のファンタジー言語には魔法の名前はないと考えています。

上記の通常の人のコードは戸惑うはずです。このメソッドを直接呼び出すことができるのに、なぜメソッドを呼び出すために変数が必要なのでしょうか。確かに、この形では、この機会は役に立たない。しかし、クラスメソッドを追加すると、メリットがあります。

クラスメソッド


クラスメソッドはDelphiのもう1つの構成要素ですが、C＃にはありません。宣言すると、これらのメソッドはクラスという単語でマークされ、静的メソッドとインスタンスメソッドのクロスになります。静的メソッドと同様に、それらは特定のインスタンスにバインドされておらず、インスタンスを作成せずにクラス名を通じて呼び出すことができます。ただし、静的メソッドとは異なり、これには暗黙のパラメーターがあります。この場合、これだけがクラスのインスタンスではなく、メタクラスです。クラスメソッドがクラスXで記述されている場合、そのthisパラメーターのタイプはType <X>になります。そして、あなたはこのようにそれを使うことができます：

class X
{
    public class void Report()
    {
        Console.WriteLine($”    {this.Name}”);
    }
}

class Y : X
{
}

static class Program
{
    static void Main()
    {
        X.Report() // : «    X»
        Y.Report() // : «    Y»
    }
}

これまでのところ、この機能はそれほど印象的ではありません。しかし、そのおかげで、静的メソッドとは異なり、クラスメソッドを仮想化することができます。より正確には、静的メソッドを仮想化することもできますが、この仮想性で次に何をするかは明確ではありません。しかし、クラスメソッドでは、このような問題は発生しません。例を挙げて考えてみましょう。

class X
{
    protected static virtual DoReport()
    {
        Console.WriteLine(“!”);
    }

    public static Report()
    {
        DoReport();
    }
}

class Y : X
{
    protected static override DoReport()
    {
        Consloe.WriteLine(“!”);
    }
}

static class Program
{
    static void Main()
    {
        X.Report() // : «!»
        Y.Report() // : ???
    }
}

論理的には、Y.Reportを呼び出すと「Bye！」が表示されます。ただし、X.Reportメソッドには、どのクラスから呼び出されたかについての情報がないため、X.DoReportまたはY.DoReportを動的に選択することはできません。その結果、ReportがYを介して呼び出された場合でも、X.Reportは常にX.DoReportを呼び出します。DoReportメソッドを仮想化しても意味がありません。したがって、C＃では静的メソッドを仮想化することはできません。それらを仮想化することは可能ですが、仮想化のメリットを享受することはできません。

もう1つはクラスメソッドです。前の例のReportが静的ではなくクラスの場合、Xを介して呼び出されたとき、およびYを介して呼び出されたときに「認識」します。したがって、コンパイラーは目的のDoReportを選択するコードを生成でき、Y.Reportの呼び出しは結論「バイ！」へ。

この機能自体は便利ですが、メタクラスを介してクラス変数を呼び出す機能を追加すると、さらに便利になります。そんな感じ：

class X
{
    public static virtual Report()
    {
        Console.WriteLine(“!”);
    }
}

class Y : X
{
    public static override Report()
    {
        Consloe.WriteLine(“!”);
    }
}

static class Program
{
    static void Main()
    {
        Type<X> tx = typeof(X);
        tx.Report() // : «!»
        tx = typeof(Y);
        tx.Report() // : «!»
    }
}

メタクラスと仮想クラスメソッドなしでこのようなポリモーフィズムを実現するには、クラスXとその子孫それぞれに対して、通常の仮想メソッドを使用して補助クラスを記述する必要があります。これにはかなりの労力が必要であり、コンパイラーによる制御はそれほど完全ではないため、どこかで間違いを犯す可能性が高くなります。一方、インスタンスレベルではなく型のレベルでポリモーフィズムが必要となる状況が定期的に発生します。言語がそのようなポリモーフィズムをサポートしている場合、これは非常に便利なプロパティです。

仮想コンストラクター


メタクラスが言語で表示された場合、仮想コンストラクターをメタクラスに追加する必要があります。仮想コンストラクタがクラスで宣言されている場合、そのすべての子孫はそれと重複する必要があります。同じパラメータセットを持つ独自のコンストラクタがあります。次に例を示します。

class A
{
    public virtual A(int x, int y)
    {
        ...
    }
}

class B : A
{
    public override B(int x, int y)
        : base(x, y)
    {
    }
}

class C : A
{
    public C(int z)
    {
        ...
    }
}

このコードでは、クラスCにはパラメーターint x、int yのコンストラクターがないため、クラスBはコンパイルされませんが、クラスBはエラーなしでコンパイルされます。

別のオプションも可能です。祖先の仮想コンストラクターが相続人で重複していない場合、コンパイラーは自動的に重複します。これは、デフォルトのコンストラクターを自動的に作成するようになっています。どちらのアプローチにも明らかな長所と短所がありますが、全体像としては重要ではありません。

仮想コンストラクタは、通常のコンストラクタを使用できる場所であればどこでも使用できます。さらに、クラスに仮想コンストラクターがある場合、そのメタクラスには、コンストラクターと同じパラメーターのセットを持つCreateInstanceメソッドがあり、このメソッドは、以下の例に示すように、クラスのインスタンスを作成します。

class A
{
    public virtual A(int x, int y)
    {
        ...
    }
}

class B : A
{
    public override B(int x, int y)
        : base(x, y)
    {
    }
}

static class Program
{
    static void Main()
    {
        Type<A> ta = typeof(A);
        A a1 = ta.CreateInstance(10, 12); //    A
        ta = typeof(B);
        A a2 = ta.CreateInstance(2, 7); //    B
    }
}

つまり、実行時に型が決定されるオブジェクトを作成する機会を得ます。これは、Activator.CreateInstanceを使用しても実行できます。しかし、この方法はリフレクションを介して機能するため、パラメーターのセットの正確性は実行段階でのみチェックされます。しかし、メタクラスがある場合、間違ったパラメーターを持つコードはコンパイルできません。さらに、リフレクションを使用する場合、作業の速度には多くの課題が残されており、メタクラスを使用するとコストを最小限に抑えることができます。
