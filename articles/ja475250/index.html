RedashがPythonコードのパフォーマンス低下を引き起こす問題に気づき、修正したように

最近、Redashは1つのタスク実行システムを別のシステムに置き換え始めました。つまり、CeleryからRQへの移行を開始しました。最初の段階では、直接要求を実行しないタスクのみが新しいプラットフォームに転送されました。これらのタスクの中には、電子メールの送信、更新する必要があるリクエストの把握、ユーザーイベントの記録、その他のサポートタスクがあります。 これらすべてを展開した後、RQワーカーは、Celeryが解決するために使用したのと同じ量のタスクを解決するために、はるかに多くのコンピューティングリソースを必要とすることがわかりました。 今日公開している翻訳は、Redashが問題の原因を突き止めて対処した経緯を物語っています。







セロリとRQの違いについて一言


セロリとRQにはプロセスワーカーの概念があります。あちこちで、並列タスクの編成のために、フォークの作成が使用されます。 Celeryワーカーが開始すると、いくつかのforkプロセスが作成され、それぞれが自律的にタスクを処理します。 RQの場合、ワーカーのインスタンスには1つのサブプロセス（「主力」と呼ばれる）のみが含まれ、1つのタスクを実行してから破棄されます。ワーカーがキューから次のタスクをダウンロードすると、新しい「働き者」が作成されます。

RQを使用する場合、より多くのワーカープロセスを実行するだけで、Celeryを使用する場合と同じレベルの並列処理を実現できます。ただし、CeleryとRQの間には微妙な違いが1つあります。Celeryでは、ワーカーは起動時にサブプロセスの多くのインスタンスを作成し、それらを繰り返し使用して多くのタスクを完了します。また、RQの場合、ジョブごとに新しいフォークを作成する必要があります。どちらのアプローチにも長所と短所がありますが、ここではこれについては触れません。

パフォーマンス測定


プロファイリングを始める前に、ワーカーコンテナーが1000個のジョブを処理するのに必要な時間を調べることで、システムのパフォーマンスを測定することにしました。record_eventこれは一般的な軽量操作であるため、タスクに集中することにしました。パフォーマンスを測定するために、コマンドを使用しましたtime。これには、プロジェクトコードにいくつかの変更が必要です。

  1. 1000個のタスクを実行するパフォーマンスを測定するために、RQバッチモードを使用することにしました。このモードでは、タスクを処理した後、プロセスが終了します。
  2. システムパフォーマンスの測定中にスケジュールされた可能性のある他のタスクが測定に影響を与えないようにしました。だから私は移動record_eventと呼ばれる別のキューにbenchmark置き換えること@job(‘default’)で@job(‘benchmark’)。これは、の発表の直前に行われましrecord_eventたtasks/general.py。

これで測定を開始することができました。まず、負荷のない状態でワーカーを開始および停止するのにかかる時間を知りたいと思いました。この時間は、後で得られる最終結果から差し引くことができます。

$ docker-compose exec worker bash -c "time ./manage.py rq workers 4 benchmark"

real	
0m14.728s
user	
0m6.810s
sys	
0m2.750s

コンピューターのワーカーを初期化するのに14.7秒かかりました。それを覚えてる。

次に、1000個のテストジョブrecord_eventをキューに入れましたbenchmark。

$ docker-compose run --rm server manage shell <<< "from redash.tasks.general import record_event; [record_event.delay({ 'action': 'create', 'timestamp': 0, 'org_id': 1, 'user_id': 1, 'object_id': 0, 'object_type': 'dummy' }) for i in range(1000)]"

その後、以前と同じようにシステムを起動し、1000個のジョブを処理するのにかかる時間を調べました。

$ docker-compose exec worker bash -c "time ./manage.py rq workers 4 benchmark"

real	
1m57.332s
user	
1m11.320s
sys	
0m27.540s

起こったことから14.7秒を差し引くと、4人のワーカーが102秒で1000個のタスクを処理することがわかりました。では、なぜそうなのかを調べてみましょう。これを行うために、私たちは、労働者が仕事で忙しい間、助けを借りて彼らを調べますpy-spy。

プロファイリング


さらに1,000個のタスクをキューに追加し（これは、前の測定中にすべてのタスクが処理されたために実行する必要があります）、ワーカーを実行してスパイします。

$ docker-compose run --rm server manage shell <<< "from redash.tasks.general import record_event; [record_event.delay({ 'action': 'create', 'timestamp': 0, 'org_id': 1, 'user_id': 1, 'object_id': 0, 'object_type': 'dummy' }) for i in range(1000)]"
$ docker-compose exec worker bash -c 'nohup ./manage.py rq workers 4 benchmark & sleep 15 && pip install py-spy && rq info -u "redis://redis:6379/0" | grep busy | awk "{print $3}" | grep -o -P "\s\d+" | head -n 1 | xargs py-spy record -d 10 --subprocesses -o profile.svg -p'
$ open -a "Google Chrome" profile.svg

前のチームが非常に長いことを知っています。理想的には、読みやすさを向上させるために、文字のシーケンスが見つかる場所で分割して、個別のフラグメントに分割することをお勧めします&&。ただし、コマンドは同じセッション内で順次実行する必要があるdocker-compose exec worker bashため、すべてが同じように見えます。このコマンドの機能の説明は次のとおりです。

  1. バックグラウンドで4つのバッチワーカーを起動します。
  2. 15秒待機します（ダウンロードを完了するのに必要な時間はおよそそれだけです）。
  3. をインストールしpy-spyます。
  4. 開始rq-infoし、ワーカーの1つのPIDを見つけます。
  5. 以前に受信したPIDを持つワーカーの作業に関する情報を10秒間記録し、データをファイルに保存します profile.svg

その結果、以下の「燃えるようなスケジュール」が得られました。


py-spyによって収集されたデータの可視化

このデータを分析したところ、タスクrecord_eventがでの実行に長時間を費やしていることに気付きましたsqlalchemy.orm.configure_mappers。これは、各タスク中に発生します。ドキュメントから、興味のあるときに、以前に作成されたすべてのマッパーの関係が初期化されることがわかりました。

このようなことは、すべてのフォークで発生する必要はありません。親ワーカーで一度関係を初期化して、「主力」でこのタスクを繰り返すのを避けることができます。

その結果、sqlalchemy.org.configure_mappers()「主力製品」を起動する前にコードへの呼び出しを追加し、再度測定を行いました。

$ docker-compose run --rm server manage shell <<< "from redash.tasks.general import record_event; [record_event.delay({ 'action': 'create', 'timestamp': 0, 'org_id': 1, 'user_id': 1, 'object_id': 0, 'object_type': 'dummy' }) for i in range(1000)]
$ docker-compose exec worker bash -c "time ./manage.py rq workers 4 benchmark"

real	
0m39.348s
user	
0m15.190s
sys	
0m10.330s

これらの結果から14.7秒を差し引くと、4人のワーカーが1000個のタスクを処理するために必要な時間が102秒から24.6秒に改善されたことがわかります。これは4倍のパフォーマンス向上です！この修正により、RQプロダクションリソースを4倍にして、同じシステムスループットを維持することができました。

概要


これらすべてから、私は次の結論を出しました。アプリケーションが唯一のプロセスである場合とフォークの場合は、アプリケーションの動作が異なることを覚えておく価値があります。各タスク中にいくつかの難しい公式タスクを解決する必要がある場合は、フォークが完了する前に一度実行して、事前に解決することをお勧めします。このようなことは、テストや開発中には検出されないため、プロジェクトに問題があると感じ、その速度を測定し、パフォーマンスの問題の原因を探すときに最後までたどり着きます。
