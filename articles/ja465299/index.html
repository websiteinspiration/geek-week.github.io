Apache NIFI-実際の機能の概要

前書き


たまたま、私の現在の職場では、このテクノロジーに精通する必要がありました。少し背景を説明します。次の会議で、私たちのチームは、有名なシステムとの統合を作成する必要があると言われました。統合により、この有名なシステムはHTTP経由で特定のエンドポイントにリクエストを送信し、奇妙なことに、SOAPメッセージの形式で応答を返信します。すべてがシンプルで些細なようです。それは必要なものに従います...


仕事


3つのサービスを作成します。 1つ目は、データベース更新サービスです。このサービスは、サードパーティシステムから新しいデータが到着すると、データベース内のデータを更新し、次のシステムに転送するためにCSV形式で特定のファイルを生成します。 2番目のサービスのエンドポイントが呼び出されます。転送されたファイルを受信して​​検証し、FTP経由でファイルストレージに配置するFTPトランスポートサービスです。 3番目のサービスである、コンシューマーへのデータ転送サービスは、最初の2つと非同期で動作します。サードパーティの外部システムからリクエストを受け取り、前述のファイルを受け取り、完成した応答ファイルを取得して変更し（id、description、linkToFileフィールドを更新）、SOAPメッセージの形式で応答を送信します。つまり、全体像は次のとおりです。最初の2つのサービスは、更新用のデータが到着したときにのみ作業を開始します。3番目のサービスは、1分間にデータを受信するための約1000リクエストの情報を消費するユーザーが多いため、常に機能します。サービスは常に利用可能であり、それらのインスタンスはテスト、デモ、プリプロド、プロドなどの異なる環境に配置されています。以下は、これらのサービスの作業の図です。不要な複雑さを避けるために一部の詳細が簡略化されていることをすぐに説明します。

def flowFile = session.get()
if (!flowFile) return
try {
    flowFile = session.write(flowFile, { inputStream, outputStream ->
        String result = IOUtils.toString(inputStream, "UTF-8");
        def recordIn = new XmlSlurper().parseText(result)
        def element = recordIn.depthFirst().find {
            it.name() == 'id'
        }

        def newId = Integer.parseInt(element.toString()) + 1
        def recordOut = new XmlSlurper().parseText(result)
        recordOut.Body.ClientMessage.RequestMessage.RequestContent.content.MessagePrimaryContent.ResponseBody.id = newId

        def res = new StreamingMarkupBuilder().bind { mkp.yield recordOut }.toString()
        outputStream.write(res.getBytes(StandardCharsets.UTF_8))
} as StreamCallback)
     session.transfer(flowFile, REL_SUCCESS)
}
