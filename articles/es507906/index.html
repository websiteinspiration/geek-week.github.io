Utilizamos DI en angular al máximo: el concepto de proveedores privados

Angular tiene un mecanismo de inyección de dependencia muy poderoso. Le permite transferir cualquier dato a través de su aplicación, convertirlo y redefinirlo en las partes correctas.



Por lo tanto, podemos hacer que la arquitectura de la aplicación sea más simple y flexible: un flujo de datos comprensible, una coherencia de código mínima, facilidad de prueba o reemplazo de dependencias.



Sin embargo, la DI en las aplicaciones se usa con bastante modestia. Como regla, esta es la introducción de servicios o la transferencia de algunos datos globales de arriba a abajo a través del árbol de inyección de dependencias.



En este artículo, me gustaría mostrar una forma alternativa de trabajar con datos obtenidos de DI. Objetivo: simplificar los componentes, directivas y servicios que utilizan estos datos.







Cómo DI se usa comúnmente en angular



Realizo revisiones diarias del código angular en el trabajo y en código abierto. Por lo general, en la mayoría de las aplicaciones DI, se reduce a la siguiente funcionalidad:



  1. Obtenga entidades angulares del árbol de dependencias: ChangeDetectorRef, ElementRef, etc.
  2. Obtenga el servicio para usarlo en el componente.
  3. Obtenga alguna configuración global para el token, que se declara en algún lugar arriba. Por ejemplo, establezca el token API_URL en el módulo raíz y obténgalo del DI en cualquier lugar de la aplicación, si es necesario.


Menos comunes son los casos en que los desarrolladores van más allá y transforman un token global existente en una forma más conveniente. Un buen ejemplo de tal conversión es el token WINDOW del paquete @ ng-web-apis / common .



Angular proporciona el token DOCUMENTO para que pueda obtener el objeto de página desde cualquier lugar de la aplicación: sus componentes son independientes de los objetos globales, fáciles de probar, nada se romperá con SSR.



Si regularmente necesita acceso al objeto WINDOW, puede escribir este token: 



import {DOCUMENT} from '@angular/common';
import {inject, InjectionToken} from '@angular/core';

export const WINDOW = new InjectionToken<Window>(
    'An abstraction over global window object',
    {
        factory: () => {
            const {defaultView} = inject(DOCUMENT);

            if (!defaultView) {
                throw new Error('Window is not available');
            }

            return defaultView;
        },
    },
);


Cuando alguien solicita el token WINDOW por primera vez desde el árbol DI, se ejecutará la fábrica de tokens: recibirá un objeto DOCUMENT de Angular y obtendrá un enlace al objeto de la ventana.



Además, propongo considerar un enfoque diferente para tales transformaciones, cuando se realizan directamente en los proveedores del componente o directiva, que inyecta el resultado.



Proveedores privados



En nuestro equipo, utilizamos activamente DI cuando trabajamos con Angular y comenzamos a notar que a menudo necesitamos hacer algunas transformaciones de los datos obtenidos de DI antes de usarlos. De hecho, nuestro componente necesita algunos datos, e incorporamos otros en él y ejecutamos la lógica de transformación dentro de él.







Veamos un ejemplo sólido de inmediato. Erin Coglar, en su informe The Architecture of Components en una gran conferencia internacional Angular Connect, mostró un ejemplo:





Si no es conveniente para usted abrir y ver el video, entonces firmemos el caso aquí.



Tenemos:



  • El componente responsable de mostrar información sobre una determinada entidad: la organización.
  • Parámetro de consulta de la ruta, que indica la identificación de la organización con la que estamos trabajando actualmente.
  • Un servicio que por id devuelve un Observable con información sobre la organización.


¿Qué queremos hacer?



Tomar la identificación de la organización de los parámetros de consulta, pasarla al método de servicio y obtener una secuencia con información sobre la organización en respuesta. Mostrar esta información en el componente.



Considere tres formas de lograr lo deseado y analícelas.



Cómo hacerlo no es necesario



A veces me encuentro con este estilo de trabajar con datos en componentes. Por favor no hagas esto: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent implements OnInit {
   organization: Organization;

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}

   ngOnInit() {
       this.activatedRoute.params
           .pipe(
               switchMap(params => {
                   const id = params.get('orgId');

                   return this.organizationService.getOrganizationById$(id);
               }),
           )
           .subscribe(organization => {
               this.organization = organization;
           });
   }
}


Para usar los datos recibidos en la plantilla: 



<p *ngIf="organization">
   {{organization.name}} from {{organization.city}}
</p>

Hagámoslo bien.



El informe de Erin del video que apliqué arriba está bien hecho. Con su variante resulta algo como esto: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent {
   readonly organization$: Observable<Organization> = this.activatedRoute.params.pipe(
       switchMap(params => {
           const id = params.get('orgId');
           return this.organizationService.getOrganizationById$(id);
       }),
   );

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}
}


Para usar los datos recibidos en la plantilla: 

<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


Este código funciona bien y está libre de los inconvenientes del enfoque anterior: se ve lo suficientemente ordenado, no tiene campos adicionales. Si queremos expandir el componente con una secuencia similar, simplemente agregamos una más, para que sea más fácil, porque para agregar una nueva secuencia no necesitamos tocar el código de la anterior.



Además, el flujo de datos se vuelve más transparente: solo tenemos el flujo que se crea cuando se crea la clase de componente. Cuando emite datos, se mostrará la información en nuestra plantilla.



Hagámoslo aún más genial: proveedores privados



Echemos un vistazo más de cerca a la solución anterior.



De hecho, el componente es independiente del enrutador e incluso del Servicio de organización. Depende de la organización $. Pero no existe tal entidad en nuestro árbol de inyección de dependencias, por lo que nos vemos obligados a realizar transformaciones en el componente.



Pero, ¿qué sucede si realiza esta conversión antes de que los datos ingresen al componente? Escribamos Proveedor específicamente para el componente en el que ocurrirán las transformaciones necesarias.



Por conveniencia, colocamos a los proveedores en un archivo separado al lado del componente, obteniendo la siguiente estructura de archivos: El







proveedor se ubicará en el archivo organization.providers.ts para la conversión de datos y un token para recibirlos en el componente: 



export const ORGANIZATION_INFO = new InjectionToken<Observable<Organization>>(
   'A stream with current organization information',
);
         :

export const ORGANIZATION_PROVIDERS: Provider[] = [
   {
       provide: ORGANIZATION_INFO,
       deps: [ActivatedRoute, OrganizationService],
       useFactory: organizationFactory,
   },
];

export function organizationFactory(
   {params}: ActivatedRoute,
   organizationService: OrganizationService,
): Observable<Organization> {
   return params.pipe(
       switchMap(params => {
           const id = params.get('orgId');

           return organizationService.getOrganizationById$(id);
       }),
   );
}


Defina una matriz de proveedores para el componente. El valor para el token ORGANIZATION_INFO se obtendrá de la fábrica en la que realizaremos la conversión de datos necesaria.



Una nota sobre el funcionamiento de DI - deps le permite tomar las entidades necesarias del árbol DI y pasarlas como argumentos a la fábrica de valores de tokens. En ellos, puede obtener cualquier entidad de DI, incluido el uso de decoradores DI, por ejemplo: 



{
       provide: ACTIVE_TAB,
       deps: [
           [new Optional(), new Self(), RouterLinkActive],
       ],
       useFactory: activeTabFactory,
}


Declarar proveedores en el componente: 



@Component({
   ..
   providers: [ORGANIZATION_PROVIDERS],
})


Y estamos listos para usar los datos en el componente: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
   providers: [ORGANIZATION_PROVIDERS],
})
export class OrganizationComponent {
   constructor(
       @Inject(ORGANIZATION_INFO) readonly organization$: Observable<Organization>,
   ) {}
}


La clase de componente se reduce a una línea con la recepción de datos.



La plantilla sigue siendo la misma: 



<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


¿Qué nos brinda este enfoque?



  1. Dependencias puras: el componente no se implementa y no almacena entidades innecesarias. Funciona solo con los datos que necesita, mientras que en sí mismo permanece limpio y contiene solo lógica para mostrar datos.

Después de implementar este enfoque, nuestros componentes y directivas comenzaron a verse mucho más limpios y simples. La separación de la lógica de la transformación de datos y su visualización aceleró el proceso de su refinamiento y expansión. El tiempo para detectar errores también se redujo debido al hecho de que puede cortar inmediatamente el área del problema: o bien el problema está en la conversión de datos o en la forma en que se muestra al usuario.



Conclusión



El enfoque descrito no lo salvará de todos los problemas de diseño. Agregar proveedores a cualquier pequeña cosa tampoco vale la pena: a veces el código es más claro si usa la conversión en el método o usa Pipe.



Sin embargo, espero que los proveedores privados lo ayuden a simplificar los componentes con una gran cantidad de dependencias o proporcionen una alternativa para la refactorización gradual de grandes piezas de lógica.
