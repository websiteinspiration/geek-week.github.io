Enumの値が不明です


簡単な例を見てみましょう：

type Status uint32

const (
	StatusOpen Status = iota
	StatusClosed
	StatusUnknown
)

ここでは、iotaを使用して列挙子を作成します。これにより、この状態になります。

StatusOpen = 0
StatusClosed = 1
StatusUnknown = 2

次に、このタイプのステータスが、パック/アンパックされるJSONリクエストの一部であると想像してみましょう。次の構造を設計できます。

type Request struct {
	ID        int    `json:"Id"`
	Timestamp int    `json:"Timestamp"`
	Status    Status `json:"Status"`
}

次に、次のクエリ結果を取得します。

{
  "Id": 1234,
  "Timestamp": 1563362390,
  "Status": 0
}

一般に、特別なことは何もありません-ステータスはStatusOpenにアンパックされます。
次に、ステータス値が設定されていない別の答えを取得しましょう。

{
  "Id": 1235,
  "Timestamp": 1563362390
}

この場合、Request構造体のStatusフィールドはゼロに初期化されます（uint32の場合は0です）。したがって、StatusUnknownではなくStatusOpenを再び取得します。

この場合、最初に列挙子の不明な値を設定することをお勧めします-つまり 0：

type Status uint32

const (
	StatusUnknown Status = iota
	StatusOpen
	StatusClosed
)

ステータスがJSONリクエストの一部でない場合は、予想どおり、StatusUnknownに初期化されます。

ベンチマーク


正しいベンチマークは非常に困難です。要素が多すぎると、結果に影響を与える可能性があります。

よくある間違いの1つに、コンパイラの最適化があります。teivah / bitvectorライブラリの具体的な例を見てみましょう。

func clear(n uint64, i, j uint8) uint64 {
	return (math.MaxUint64<<j | ((1 << i) - 1)) & n
}

この関数は、指定された範囲のビットをクリアします。この方法でパフォーマンスをテストできます。

func BenchmarkWrong(b *testing.B) {
	for i := 0; i < b.N; i++ {
		clear(1221892080809121, 10, 63)
	}
}

このテストでは、コンパイラーはclearが他の関数を呼び出さないことに気づくので、そのままそのまま埋め込みます。組み込み後、コンパイラは副作用が発生しないことを確認します。したがって、明確な呼び出しは単純に削除され、結果は不正確になります。

1つの解決策は、次のようなグローバル変数に結果を設定することです。

var result uint64

func BenchmarkCorrect(b *testing.B) {
	var r uint64
	for i := 0; i < b.N; i++ {
		r = clear(1221892080809121, 10, 63)
	}
	result = r
}

ここで、コンパイラーは、呼び出しが副作用を引き起こすかどうかを知りません。したがって、ベンチマークは正確になります。

ポインタ！ポインタはどこにでもあります！


変数を値で渡すと、この変数のコピーが作成されます。ポインタを渡している間、アドレスをメモリにコピーするだけです。

したがって、ポインタを渡す方が常に高速ですよね？

もしそうなら、この例を見てください。これは、最初にポインタで送受信し、次に値で送受信する0.3 KBのデータ構造のベンチマークです。 0.3 KBは少しです。私たちが毎日作業している通常のデータ構造については、それだけです。

これらのテストをローカル環境で実行すると、値ごとの転送が4倍以上速くなります。かなり思いがけないですね。

この結果の説明は、Goでメモリ管理がどのように行われるかを理解することに関連しています。私はそれを見事に説明することはできませんウィリアム・ケネディですが、要約してみましょう。

変数はヒープまたはスタックに配置できます。
  • スタックには、このプログラムの現在の変数が含まれています。関数が戻るとすぐに、変数がスタックからポップされます。
  • ヒープには、共通の変数（グローバル変数など）が含まれています。

値を返す簡単な例を見てみましょう：

func getFooValue() foo {
	var result foo
	// Do something
	return result
}

ここで、結果変数は現在のゴルーチンによって作成されます。この変数は現在のスタックにプッシュされます。関数が戻るとすぐに、クライアントはこの変数のコピーを受け取ります。変数自体がスタックからポップされます。別の変数が上書きされるまでメモリ内に存在しますが、アクセスできなくなります。
これで同じ例ですが、ポインタがあります：

func getFooPointer() *foo {
	var result foo
	// Do something
	return &result
}

結果変数は現在のgoroutineによって引き続き作成されますが、クライアントはポインター（変数のアドレスのコピー）を受け取ります。結果変数がスタックからプルされた場合、この関数のクライアントはそれにアクセスできなくなります。

このシナリオでは、Goコンパイラーは変数を共有できる場所に結果変数を出力します。束で。

ポインターを渡す別のシナリオ：

func main()  {
	p := &foo{}
	f(p)
}

同じプログラムでfを呼び出すので、変数pをヒープする必要はありません。それは単にスタックにプッシュされ、サブ関数がそれにアクセスできます。

たとえば、この方法では、io.ReaderのReadメソッドでスライスが取得されます。スライス（ポインター）を返すと、ヒープに入れられます。

なぜスタックはそんなに速いのですか？2つの理由があります。
  • . , (push) , (pop) , return. , . .
  • , , , .

結論として、関数を作成するときのデフォルトのアクションは、ポインターの代わりに値を使用することです。ポインターは、変数を共有する場合にのみ使用してください。

また、パフォーマンスの問題が発生している場合、可能な最適化の1つは、特定の状況でポインターが役立つかどうかを確認することです。コンパイラが変数をヒープに出力するかどうかは、次のコマンドで確認できます。
go build -gcflags "-m -m"
。
ただし、繰り返しになりますが、ほとんどの日常業務では、値を使用するのが最善です。

/スイッチまたは/ /選択の中止


fがtrueを返す場合、次の例ではどうなりますか？

for {
  switch f() {
  case true:
    break
  case false:
    // Do something
  }
}

休憩といいます。この中断のみがスイッチを切断し、forループは切断しません。

ここで同じ問題：

for {
  select {
  case <-ch:
  // Do something
  case <-ctx.Done():
    break
  }
}

Breakは、forループではなく、selectステートメントに関連付けられています。

/スイッチまたは/選択を中断する1つの可能な解決策は、ラベルを使用することです。

loop:
	for {
		select {
		case <-ch:
		// Do something
		case <-ctx.Done():
			break loop
		}
	}

エラー処理


特にエラー処理の分野では、囲碁はまだ若いです。この欠点を克服することは、Go 2で最も予想される革新の1つです。

現在の標準ライブラリ（Go 1.13より前）は、エラーを構築するための関数のみを提供しています。したがって、pkg / errorsパッケージを見ると興味深いでしょう。

このライブラリは、常に尊重されているわけではない規則に従うための良い方法です。
エラーは一度だけ処理する必要があります。エラーログはエラー処理です
。したがって、エラーはログに記録されるか、より高く投げられるべきです。

現在の標準ライブラリでは、エラーにコンテキストを追加して何らかの階層を作成する必要がある場合があるため、この原則を観察することは困難です。

データベースエラーの原因となるREST呼び出しの例を見てみましょう。

unable to server HTTP POST request for customer 1234
 |_ unable to insert customer contract abcd
     |_ unable to commit transaction

pkg / errorsを使用すると、次のことができます。

func postHandler(customer Customer) Status {
	err := insert(customer.Contract)
	if err != nil {
		log.WithError(err).Errorf("unable to server HTTP POST request for customer %s", customer.ID)
		return Status{ok: false}
	}
	return Status{ok: true}
}

func insert(contract Contract) error {
	err := dbQuery(contract)
	if err != nil {
		return errors.Wrapf(err, "unable to insert customer contract %s", contract.ID)
	}
	return nil
}

func dbQuery(contract Contract) error {
	// Do something then fail
	return errors.New("unable to commit transaction")
}

最初のエラー（外部ライブラリによって返されない場合）は、errors.Newを使用して作成できます。中間層である挿入は、このエラーをラップし、コンテキストを追加します。次に、親がそれを記録します。したがって、各レベルはエラーを返すか、処理します。

また、コールバックなど、エラーの原因を特定することもできます。データベースにアクセスできる外部ライブラリからのdbパッケージがあるとします。このライブラリは、db.DBErrorと呼ばれる一時的なエラーを返す場合があります。再試行する必要があるかどうかを判断するには、エラーの原因を特定する必要があります。

func postHandler(customer Customer) Status {
	err := insert(customer.Contract)
	if err != nil {
		switch errors.Cause(err).(type) {
		default:
			log.WithError(err).Errorf("unable to server HTTP POST request for customer %s", customer.ID)
			return Status{ok: false}
		case *db.DBError:
			return retry(customer)
		}

	}
	return Status{ok: true}
}

func insert(contract Contract) error {
	err := db.dbQuery(contract)
	if err != nil {
		return errors.Wrapf(err, "unable to insert customer contract %s", contract.ID)
	}
	return nil
}

また、これはに含まれているerrors.Cause、使用して行われPKG /エラー：

遭遇した私が使用したことを一般的なエラーの一つのpkg /エラー部分的にしかし。たとえば、エラーチェックは次のように実行されました。

switch err.(type) {
default:
  log.WithError(err).Errorf("unable to server HTTP POST request for customer %s", customer.ID)
  return Status{ok: false}
case *db.DBError:
  return retry(customer)
}

この例では、db.DBErrorがラップされている場合、2回目の呼び出しは行われません。

スライスの初期化


時には、スライスの最終的な長さを知っています。たとえば、FooスライスをBarスライスに変換するとします。つまり、これら2つのスライスは同じ長さになります。

私はしばしばこのように初期化されたスライスに出くわします：

var bars []Bar
bars := make([]Bar, 0)

スライスは魔法のような構造ではありません。内部的には、空き領域がなくなった場合にサイズを増やす戦略を実装します。この場合、新しいアレイが自動的に作成され（容量が大きくなります）、すべての要素がそのアレイにコピーされます。

ここで、[] Fooには何千もの要素が含まれているため、サイズを増やすこの操作を数回繰り返す必要があると想像してみましょう。挿入アルゴリズムの複雑さはO（1）のままですが、実際にはこれはパフォーマンスに影響します。

したがって、最終的な長さがわかっている場合は、次のいずれかを実行できます。

  • 事前定義された長さで初期化します。

func convert(foos []Foo) []Bar {
	bars := make([]Bar, len(foos))
	for i, foo := range foos {
		bars[i] = fooToBar(foo)
	}
	return bars
}

  • または、長さ0と所定の容量で初期化します。

func convert(foos []Foo) []Bar {
	bars := make([]Bar, 0, len(foos))
	for _, foo := range foos {
		bars = append(bars, fooToBar(foo))
	}
	return bars
}

最良の選択肢は何ですか？最初は少し速いです。ただし、後者の方が一貫しているため、後者をお勧めします。初期サイズがわかっているかどうかに関係なく、スライスの最後に要素を追加するには、appendを使用します。

context.Contextは、開発者によって誤解されることがよくあります。公式ドキュメントによると：
コンテキストは、APIの境界を越えて、期限、キャンセル信号、およびその他の値を伝達します。
この説明は非常に一般的であるため、プログラマーが正しく使用する方法を困惑させる可能性があります。

それを理解してみましょう。コンテキストは以下を運ぶことができます：
  • — (, 250 ), - (, 2019-01-08 01:00:00), , , ( -), . .).
  • ( <-chan struct {}). . , . , , . , ( , ). , , , .
  • / ( interface{}).

あと2点。まず、コンテキストは構成可能です。したがって、たとえば、期限とキー/値リストを含むコンテキストがある場合があります。さらに、複数のゴルーチンが同じコンテキストを共有できるため、キャンセル信号が複数のジョブを停止する可能性があります。

トピックに戻りますが、ここで私が遭遇した間違いがあります。

Goアプリはurfave / cliに基づいていました（知らない場合、これはGoコマンドラインアプリケーションを構築するための優れたライブラリです）。起動すると、開発者は一種のアプリケーションコンテキストを継承します。つまり、アプリケーションが停止すると、ライブラリはコンテキストを使用してキャンセル信号を送信します。

たとえば、gRPCエンドポイントが呼び出されたときに、このコンテキストが直接送信されることに気付きました。これは私たちが必要とするものではありません。

代わりに、gRPCライブラリに通知します。たとえば、アプリケーションが停止したとき、または100ミリ秒後にリクエストをキャンセルしてください。

これを達成するには、単純に複合コンテキストを作成します。親が（urfave / cliによって作成された）アプリケーションコンテキストの名前である場合、次のようにするだけです。

ctx, cancel := context.WithTimeout(parent, 100 * time.Millisecond)
response, err := grpcClient.Send(ctx, request)

コンテキストは理解するのがそれほど難しくなく、私の意見では、これは言語の最高の機能の1つです。

入力としてファイル名を使用する


もう1つのよくある間違いは、ファイル名を関数に渡すことです。

ファイル内の空の行数をカウントする関数を実装する必要があるとします。最も自然な実装は次のようになります。

func count(filename string) (int, error) {
	file, err := os.Open(filename)
	if err != nil {
		return 0, errors.Wrapf(err, "unable to open %s", filename)
	}
	defer file.Close()

	scanner := bufio.NewScanner(file)
	count := 0
	for scanner.Scan() {
		if scanner.Text() == "" {
			count++
		}
	}
	return count, nil
}

ファイル名は入力として設定されているので、それを開いてロジックを実装しますよね？

次に、この関数を単体テストでカバーしたいとします。通常のファイル、空のファイル、エンコーディングの種類が異なるファイルなどでテストします。管理が非常に困難な場合があります。

さらに、HTTPボディなどに同じロジックを実装する場合は、このための別の関数を作成する必要があります。

Goには、io.Readerとio.Writerの2つの優れた抽象概念が付属しています。ファイル名を渡す代わりに、データソースを抽象化するio.Readerを渡すだけです。
これはファイルですか？ HTTPボディ？バイトバッファ？同じReadメソッドを使用するため、問題ありません。

私たちの場合、入力をバッファリングして1行ずつ読み取ることもできます。これを行うには、bufio.ReaderとそのReadLineメソッドを使用できます。

func count(reader *bufio.Reader) (int, error) {
	count := 0
	for {
		line, _, err := reader.ReadLine()
		if err != nil {
			switch err {
			default:
				return 0, errors.Wrapf(err, "unable to read")
			case io.EOF:
				return count, nil
			}
		}
		if len(line) == 0 {
			count++
		}
	}
}

これで、ファイルを開く責任がカウントクライアントに委任されました。

file, err := os.Open(filename)
if err != nil {
  return errors.Wrapf(err, "unable to open %s", filename)
}
defer file.Close()
count, err := count(bufio.NewReader(file))

2番目の実装では、実際のデータソースに関係なく関数を呼び出すことができます。一方、次の行からbufio.Readerを作成するだけでよいので、これによりユニットテストが容易になります。

count, err := count(bufio.NewReader(strings.NewReader("input")))

ゴルーチンとサイクル変数


私が遭遇した最後の一般的なエラーは、ループ変数でゴルーチンを使用するときでした。

次の例の結論はどうなりますか？

ints := []int{1, 2, 3}
for _, i := range ints {
  go func() {
    fmt.Printf("%v\n", i)
  }()
}

1 2 3ランダムに？いいえ。

この例では、各ゴルーチンは変数の同じインスタンスを使用しているため、3 3 3（ほとんどの場合）を出力します。

この問題には2つの解決策があります。1つ目は、変数iの値をクロージャー（内部関数）に渡すことです。

ints := []int{1, 2, 3}
for _, i := range ints {
  go func(i int) {
    fmt.Printf("%v\n", i)
  }(i)
}

2つ目は、forループ内に別の変数を作成することです。

ints := []int{1, 2, 3}
for _, i := range ints {
  i := i
  go func() {
    fmt.Printf("%v\n", i)
  }()
}

i：= iの割り当ては少し奇妙に思えるかもしれませんが、この設計は完全に有効です。ループにいるということは、別のスコープにいるということです。したがって、i：= iは変数iの別のインスタンスを作成します。もちろん、読みやすくするために別の名前で呼び出すこともできます。

他のよくある間違いを知っている場合は、コメントに自由に書いてください。
