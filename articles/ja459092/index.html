X.Spectator-.NETでのステータス監視



今日、ほとんどの情報システムは、かなり複雑なアーキテクチャと多数の相互依存関係を持つ複雑なソリューションです。このようなシステムの動作中、ピーク負荷時に、一部のモジュールが失敗するか、正しく機能しない可能性があります。この場合、システムは安定しなくなり、すべての着信要求を正しく処理しなくなる可能性があります。システムの安定した運用を確保するために、さまざまな戦略を実装できます。

場合によっては（これが許容される場合）、システムのいわゆるへの変換。 "セーフモード"。このモードでは、システムはより多くの呼び出しを処理できますが、クエリ処理の結果の精度はわずかに低下します。他の場合では、コンピューティングリソースのスケーリングと、着信要求の処理を担当するシステムの個々のモジュールの数の増加が役立ちます。同時に、最も重要なタスクは、システムの状態を判別することです。これにより、1つまたは別のアクションを適用して、その作業を安定させる必要があります。

私のプロジェクトでは、かなり前からさまざまな方法で同様の問題を解決してきました。大学在学中から、.NET 4.0で記述された小さなプロジェクトがあり、実際のプロジェクト用に小さなコードを随時使用していました。私は長い間、このプロジェクトをリファクタリングし、クリーンアップして、システムの状態を監視する問題を美しく最小限に解決できる個別のミニフレームワークの形式で配置することを計画してきました。数晩と数日の休日を過ごした後、私はこのコードを片付けてGitHubに投稿しました。さらに、このプロジェクトで何がどのように実施されているかをより詳細に検討することを提案します。

システムの動作モードを変更する必要性を判断するために、仮想「センサー」と「オブザーバー」を使用するアプローチを使用することが提案されています。



▌エンティティと定義


使用する基本エンティティ：

  • センサー -システムのインジケーターの1つのステータスをチェックします。
  • オブザーバー -1つまたは複数のセンサーに問い合わせます。センサーの現在の読み取り値に応じてその状態を変更します。
  • 状態計算機 -メトリックログに基づいて現在の状態を計算します。
  • 状態ログ -ポーリング時間を示す各センサーのインジケーターのセット。

抽象化ごとに、基本的な実装と、単純で便利な拡張のためのメカニズムがあります。より詳細に検討してみましょう。

センサー


基本的なIProbeインターフェイス。IProbeを実装するクラスは、要求に応じて、監視するシステムパラメータの値、または特定のモジュール/サービスを提供します。

    public interface IProbe
    {
        string Name { get; }
        
        Task<ProbeResult> Check();
    }

Check メソッドの結果として、IProbeインスタンスはProbeResult構造を返します。

    public struct ProbeResult
    {
        public string ProbeName { get; set; }
        public DateTime Time { get; set; }
        public bool Success { get; set; }
        public string Data { get; set; }
        public Exception Exception { get; set; }

        public override string ToString() => $"{Time}: {Success}";
    }

ここで、Successフィールドは、センサーの観点からパラメーターが正常にテストされたかどうかを決定し、DataおよびExceptionフィールドは、デバッグまたはロギングに必要な場合に備えて、追加情報を格納できます。

観察者


ISpectator の基本的なインターフェース。システムを監視し、システムの状態変更時にイベントを生成して、これらのイベントにサブスクライブしているすべてのモジュールに通知します。IStateEvaluatorインターフェイスを実装するクラスのインスタンスを使用して、現在の状態を計算します。

    public interface ISpectator<TState> where TState : struct, IConvertible
    {
        event EventHandler<StateEventArgs<TState>> StateChanged;
        event EventHandler<HealthCheckEventArgs> HealthChecked;
        
        TState State { get; }
        
        TimeSpan Uptime { get; }
        
        string Name { get; }
        
        void AddProbe(IProbe probe);

        void CheckHealth();
    }

各センサーのポーリング中に、オブザーバーがStateChangedイベントを発生させ、システムの現在の状態に関する情報を含むStateEventArgsタイプのオブジェクトがこのイベントのサブスクライバーに渡されます。

状態計算機


基本IEvaluatorインターフェイス。センサーステータスログに基づいてシステムの現在の状態を計算します。

    public interface IStateEvaluator<TState>
    {
        TState Evaluate(
            TState currentState, 
            DateTime stateChangedLastTime, 
            IReadOnlyCollection<JournalRecord> journal);
    }

状態ログ


JournalRecord構造のインスタンスのコレクション。JournalRecordインスタンスは、オブザーバーによってポーリングが開始されたときにポーリングされたすべてのセンサーに関する情報を格納します。

    public struct JournalRecord
    {
        public JournalRecord(DateTime time, IEnumerable<ProbeResult> values)
        {
            Time = time;
            Values = values.ToImmutableList();
        }

        public DateTime Time { get; set; }
        
        public IReadOnlyCollection<ProbeResult> Values { get; set; }

        public override string ToString() => $"{Time}: [{string.Join(",", Values)}]";
    }

▌仕組み


システムの状態を計算するプロセスは、次のように説明できます。システムに埋め込まれた各センサーは、観測されたシステム/モジュール/サービスのパラメーターの1つを決定できます。たとえば、APIへの外部アクティブリクエストの数、使用されるRAMの量、キャッシュ内のエントリ数などを修正できます。

各センサーを1人または複数のオブザーバーに割り当てることができます。各オブザーバーは1つまたは複数のセンサーを使用できます。オブザーバーはISpectatorインターフェイスを実装し、状態の変化またはセンサーのポーリングのイベントでイベントを生成する必要があります。

システムの状態の次のチェック中に、オブザーバーはすべての「センサー」をポーリングし、状態ログに書き込むための配列を形成します。システムの状態が変化した場合、オブザーバーは適切なイベントを生成します。変更に関する情報を受信するイベントサブスクライバーは、システムの動作パラメーターを変更できます。さらに、システムの状態を判断するために、さまざまなタイプの「電卓」を使用できます。

同期および非同期の動作モード


オブザーバーが「センサー」の調査を開始する2つの主なシナリオを検討してください。

同期モード


センサーの調査に続いてシステムの状態の再計算が行われるのは、システムモジュールの1つがオブザーバーに直接アピールされるためです。
この場合、オブザーバーとセンサーは同じスレッドで動作します。システムの状態の誤った計算は、システム内の操作の一部として実行されます。

プロジェクトにはすでにそのようなオブザーバーのSpectatorBaseの基本的な実装があります。

コードを表示
    public class SpectatorBase<TState> : ISpectator<TState> 
        where TState : struct, IConvertible
    {
        private TState _state;

        private readonly IList<IProbe> _probes;
        private readonly IStateEvaluator<TState> _stateEvaluator;
        private readonly List<JournalRecord> _journal;
        private readonly ReaderWriterLockSlim _journalLock;
        private readonly ReaderWriterLockSlim _stateLock;
        private readonly Stopwatch _stopwatch;

        public event EventHandler<StateEventArgs<TState>> StateChanged;
        public event EventHandler<HealthCheckEventArgs> HealthChecked;

        public virtual TState State
        {
            get
            {
                _stateLock.EnterReadLock();
                
                try
                {
                    return _state;
                }
                finally
                {
                    _stateLock.ExitReadLock();
                }
            }
        }

        public TimeSpan Uptime => _stopwatch.Elapsed;

        public string Name { get; set; }

        public IReadOnlyCollection<JournalRecord> Journal
        {
            get
            {
                _journalLock.EnterReadLock();
                
                try
                {
                    return _journal;
                }
                finally
                {
                    _journalLock.ExitReadLock();
                }
            }
        }
        
        public DateTime StateChangedDate { get; private set; }

        public TimeSpan RetentionPeriod { get; private set; }

        public SpectatorBase(IStateEvaluator<TState> stateEvaluator, TimeSpan retentionPeriod, TState initialState)
        {
            RetentionPeriod = retentionPeriod;
            _state = initialState;
            StateChangedDate = DateTime.UtcNow;
            
            _stateEvaluator = stateEvaluator;
            _stopwatch = Stopwatch.StartNew();
            _probes = new List<IProbe>();
            _journal = new List<JournalRecord>();
            _journalLock = new ReaderWriterLockSlim();
            _stateLock = new ReaderWriterLockSlim();
        }

        public void AddProbe(IProbe probe) => _probes.Add(probe);

        protected virtual void ChangeState(TState state, IEnumerable<string> failedProbes)
        {
            _stateLock.EnterWriteLock();

            try
            {
                _state = state;
            }
            finally
            {
                _stateLock.ExitWriteLock();
            }

            StateChangedDate = DateTime.UtcNow;

            StateChanged?.Invoke(this, new StateEventArgs<TState>(state, StateChangedDate, failedProbes));
        }

        public virtual void CheckHealth()
        {
            var results = new Stack<ProbeResult>();

            var tasks = _probes
                .Select(async o => { results.Push(await o.Check().ConfigureAwait(false)); })
                .ToArray();

            Task.WaitAll(tasks);

            var now = DateTime.UtcNow;
            
            _journalLock.EnterWriteLock();
            
            try
            {
                //cleanup state records
                _journal.RemoveAll(o => o.Time < now.Subtract(RetentionPeriod));
                
                _journal.Add(new JournalRecord(now, results));
            }
            finally
            {
                _journalLock.ExitWriteLock();
            }

            //Recalculate state
            var state = _stateEvaluator.Evaluate(State, StateChangedDate, _journal);

            if (!EqualityComparer<TState>.Default.Equals(State, state))
            {
                ChangeState(state, results.Where(o => !o.Success).Select(o => o.ProbeName));
            }

            OnHealthChecked(now, results);
        }

        protected virtual void OnHealthChecked(DateTime now, IReadOnlyCollection<ProbeResult> results) =>
            HealthChecked?.Invoke(this, new HealthCheckEventArgs(now, results));
    }


非同期モード


この場合、センサーへの問い合わせはシステムのプロセスとは非同期で行われ、別のスレッドで実行できます。このようなオブザーバーの基本的な実装も、すでにプロジェクトAutomatedSpectatorです。

コードを表示
public class AutomatedSpectator<TState> : SpectatorBase<TState>, IAutomatedSpectator<TState> 
        where TState : struct, IConvertible
    {
        public TimeSpan CheckHealthPeriod { get; }

        private readonly System.Timers.Timer _timer;

        public AutomatedSpectator(
            TimeSpan checkHealthPeriod,
            TimeSpan retentionPeriod,
            IStateEvaluator<TState> stateEvaluator, TState initialState)
            : base(stateEvaluator, retentionPeriod, initialState)
        {
            CheckHealthPeriod = checkHealthPeriod;

            _timer = new System.Timers.Timer(CheckHealthPeriod.TotalMilliseconds);
            _timer.Elapsed += (sender, args) => CheckHealth();
            _timer.AutoReset = true;
        }

        public void Start() => _timer.Start();
    }


▌まとめ


X.Spectatorを使用することで、負荷の高いいくつかのプロジェクトで個人的に、いくつかのサービスの安定性を大幅に向上させることができました。提案されたフレームワークは、マイクロサービスアーキテクチャに基づいて構築された分散システムに実装されたときに最高であることが判明しました。最も最適な統合オプションは、センサーの実装プロセスがIoCコンテナーを使用して実装されており、オブザーバーがシングルトーンの形式で提示されている場合に、制御の反転の原理、つまり依存関係の実装を使用することです。オブザーバーの単一のインスタンスには、モジュールとサービスの異なるクラスからアクセスできます。
