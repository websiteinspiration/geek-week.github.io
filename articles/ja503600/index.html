STM32MP1：U-Boot、Buildroot、Arch LinuxおよびDebianの一部

こんにちは、ハブル！

しばらく前に、STMicroelectronicsは興味深いSTM32MP1シリーズプロセッサを発売しました。ついにこのプロセッサをベースにしたデバッグボードを手に入れたとき、人気のあるディストリビューション（Debian、Arch Linuxなど）をベースにしたビルドが不足していることに驚きました。残ったのは、このボード用の配布キットを自分で調整しようとすることだけでした。この結果を踏まえて、この記事が掲載されました。



特徴は何ですか？


この記事は、少なくともSTM32MP1シリーズプロセッサの特性の簡単な概要がなければ完了しません。STM32MP1シリーズには、STM32MP151、STM32MP153、STM32MP157の3つのプロセッサフ​​ァミリがあります。主な特徴を表に示します。



表からわかるように、ファミリー間の違いは、STM32MP151にはCortex-A7のコアが1つありますが、STM32MP153およびSTM32MP157にはそのようなコアが2つあり、STM32MP157も3D GPUをサポートしています。しかし、一般的に、2020年のこれらのプロセッサの特性は印象に残りません。かなり控えめです。なぜ私はまだそれらに注意を払ったのですか？

なぜSTM32MP1なのか？


実際、完全に論理的な問題が発生する可能性があります。ラズベリーPi、バナナPi、オレンジPiがあり、最後に-なぜ他のSTM32MP1が必要なのでしょうか。さらに、これらすべてのボードは通常、調査対象よりも大幅に高いパフォーマンスを発揮します。答えは簡単です-家庭用の工芸品をやっている間、それはラズベリーを取る必要があり、それは正しいでしょう。しかし、産業用アプリケーション向けの量産製品について話している場合、STM32MP1が優勝したおかげで、他のものが決定的な役割を果たすようになりました。

  • 使用温度範囲。STM32MP1の場合、マイナス40度から始まりますが、他のシングルボードコンピュータの多くのプロセッサでは、マイナス20度であるのが適切です。

  • . DigiKey Mouser STM32MP1, .

もちろん、ST32MP1は、産業用アプリケーションの市場で唯一のプロセッサではありません。NXPとTIの両方があります。TIについては、それに基づいたかなり複雑なモジュールのプロジェクトがあり、ドキュメントに記載されていないかなりの数のハードウェア機能からの沈殿物がありましたが、それを順守しないと、プロセッサーは完全に故障する可能性があり、すぐにではなく、時間とともに最も不都合な瞬間に。さらに、これはシングルコアプロセッサであり、割り当てられるタスクの数が増えるにつれて、パフォーマンスの問題がますます頻繁に発生しました。同時に、私はSTMicroelectronicsマイクロコントローラーを扱っていましたが、それらは非常に優れていることがわかったので、この新しい小さなコントローラーを選択することにしました。

デバッグボード


実験のために、STM32MP157A-DK1デバッグボードを購入しました。このボードは、機器の面ではかなり控えめです。STM32MP157C-DK2やSTM32MP157A-EV1などの豊富な周辺機器のようなLCDディスプレイはありません。ただし、microSDカードスロット、USB-UARTコンソール、いくつかのUSBポート、イーサネットがあります。最初のスタートとしては、十分すぎるほどです。そして、ドライストーリーを写真で薄めるために、このデバッグボードの写真を添付し​​ています。



既製のソフトウェアから何ですか？


STMicroelectronicsでは、ハードウェアに関してはすべてが通常非常に優れていますが、ソフトウェアに関してはひどいものです。これらのAtollic True Studio、CubeMX、CubeIDEのすべての変更は、新しいリリースごとにだんだんバグを含んでおり、いくらかの苦痛を呼び起こします。 STM32MP1をサポートすると、状況は少し良くなります。 STMicroelectronicsは、OpenSTLinuxの特定のアセンブリのみを提供しています。このアセンブリは、Yoctoプロジェクトを使用して構築されたディストリビューションです。もちろん、これはすべてこの形式で存在できますが、主な欠点は、有名なディストリビューションのリポジトリにアクセスできないことでした。これは、apt-get installなどのコマンドを実行するだけでは、人気のあるディストリビューションのリポジトリからユーティリティをボードに配置できないことを意味します。多くの場合、これは組み込みソリューションでは必要ありませんが、状況が発生する可能性があります。そのような機会は絶対に不要ではありません。

私たちは何をしますか？


したがって、タスクは明確です。デバッグボードで人気のあるディストリビューションを実行する必要があります。私の選択はArch Linuxにかかった。これは最も簡単なディストリビューションではありませんが、ARMデバイスによく適合しています。既製のアセンブリとこれに特化した公式サイトがあります。

最初にスナップで問題を解決しようとしました-armv7の下でアセンブルされたArch Linuxのディストリビューションからブートローダー対応コアを外しました。これは時々他のボードで動作しましたが、私を待っている大失敗がありました：カーネルが正しいアーキテクチャーのためにアセンブルされたという事実にもかかわらず、それは始まりませんでした。さて、あなたはあなたのカーネルと同時にあなたのローダーを組み立てる必要があります。私の行動計画はこれでした：

  1. U-Bootブートローダーをビルドします。
  2. Linuxカーネルをビルドします。
  3. microSDカードにマークを付けます。
  4. ブートローダー、カーネル、ルートファイルシステムをmicroSDカードに書き込みます。
  5. 利益

組立準備


この計画を実行するには、Linuxを搭載したコンピューターとmicroSDカードに記録するためのカードリーダーが必要です。私はDebian 10でラップトップを使用しましたが、一般的にこれは重要ではありません。ユーティリティの名前はわずかに異なるだけです。したがって、必要なユーティリティを配置します。私はすぐに、すべてのコマンドはrootとして、またはsudoを介して実行する必要があることに注意してください。

apt-get install git
apt-get install make
apt-get install gcc
apt-get install gcc-arm-linux-gnueabihf
apt-get install bison
apt-get install flex
apt-get install g++
apt-get install rsync
apt-get install libncurses-dev

アセンブリの準備として、作業ディレクトリに3つのディレクトリを作成します。u-boot（ブートローダー用）、buildroot（システムビルド用）、archlinux（ディストリビューション用）：

mkdir u-boot
mkdir buildroot
mkdir archlinux

これらのディレクトリはさらに必要になります。これらの名前については、記事の本文で後ほど参照します。

Uブーツアセンブリ


U-Bootについてはすでに多くの記事が書かれていますが、その一部として、それが何であるのか、それが何のためにあるのか、どのように機能するのかについては詳しく説明しません。これは、ARMデバイスでLinuxを起動するためのブートローダーであると言えます。 U-BootブートローダーのソースコードはGitHubで入手できます。

U-Bootを構築するために、最初に、U-Bootリポジトリを以前に作成したu-bootディレクトリに複製します。

git clone https://github.com/u-boot/u-boot

U-Bootを正常にビルドするには、デバイスツリーファイルとU-Boot構成ファイルが必要です。

デバイスツリーファイルは、デバイスに依存するファイルです。このファイルには、特定のボードのプロセッサの構成が記述されています。 ARMプロセッサをベースにハードウェアを作成し、その上でLinuxを実行する予定の場合は、そのためのデバイスツリーファイルを開発する必要があります（または既製のファイルを適応させる必要があります）。ただし、多くのデバッグボードにはすでに既製のファイルがあります。思いやりのあるU-Boot開発者は、それらをリポジトリに含めます。だから、u-boot / arch / arm / dtsディレクトリを見てください。stm32mp157a-dk1.dtbファイルが含まれている必要があります。これは、デバッグボードのデバイスツリーファイルです。

UPD：コメントに正しく記載されているように、stm32mp157a-dk1.dtbファイルは最初のU-Bootビルド後に表示されます。ビルドプロセスを開始する前は、u-boot / arch / arm / dtsディレクトリにソースファイルstm32mp157a-dk1.dtsが含まれており、

U-Boot構成ファイルには基本的なブートローダー設定が含まれています。

U-Bootを最初から構成することは、非常に多くの設定があるため、かなり長く骨の折れるプロセスです。これらの目的のために、コンソールとグラフィックの両方のコンフィギュレーターがあります。ただし、ここでは幸運でした。u-boot/ configsディレクトリにstm32mp15_basic_defconfigファイルがあります。これは、STM32MP15デバッグボード用のU-Boot基本構成ファイルです。このファイルを開いて、すばやく開始するには、1行だけ変更すれば十分であることがわかります。

CONFIG_DEFAULT_DEVICE_TREE=”stm32mp157c-ev1”

書く

CONFIG_DEFAULT_DEVICE_TREE=”stm32mp157a-dk1”

この行で、ボードにデバイスツリーファイルを使用する必要があることをブートローダーに通知します。

これでU-Bootをビルドする準備ができました。設定を使用します：

make CROSS_COMPILE=arm-linux-gnueabihf- stm32mp15_basic_defconfig

アセンブリを実行します。

make CROSS_COMPILE=arm-linux-gnueabihf-

すべてが順調に進んだら、u-bootディレクトリにたくさんのファイルができているはずです。これらのうち、2つはu-boot-spl.stm32とu- boot.imgです。

最初のファイルは、いわゆるFirst Stage Boot Loader（FSBL）です。U-Bootの前にあり、最初に起動し、U-Bootの起動に必要なDDR3メモリを初期化します。他のボードでは、FSBLはしばしば1つのイメージにU-Bootと組み合わされますが、ここでは各イメージを個別にUSBフラッシュドライブに書き込む必要があります。

これでU-Bootはこれで終了です。指定されたファイルを保存して、Linuxカーネルアセンブリに直接進んでください。

Linuxカーネルアセンブリ


Buildrootを使用してLinuxカーネルをビルドします。もちろん、これらの目的のために、同様に人気のあるYoctoを使用することも、kernel.orgのソースからカーネルを構築することもできます。しかし、私はBuildrootでの作業経験があったので、それを決めました。さらに、Buildrootはルートファイルシステム（rootfs）やU-Bootローダーも構築します。

次に、利用可能な手段で、公式ウェブサイトからBuildrootからアーカイブをダウンロードし、buildrootディレクトリに解凍してそこに移動します。

U-Bootの場合と同様に、最初に注意する必要があるのは、ハードウェアの構成ファイルです。

buildroot / configsディレクトリに移動し、開発者がすでにボードの構成ファイルを追加していることを確認します。ファイルがあります。stm32mp157a_dk1_defconfig（Buildroot-2020.05ビルドではtrue、このファイルの以前のバージョンではまだありませんでした）。

この構成ファイルを使用して5.4.26カーネルをビルドしようとしましたが、ボード上で正常に起動しました。ただし、何らかの理由で、このアセンブリのLinuxデバイスツリーファイルは切り捨てられることが判明しました。デフォルトでは、USBポートのサポートさえありませんでした。時間の経過とともにこのバグが修正されることを期待しましょう。

私はこの問題をググって行って、STMicroelectronicsリポジトリーに出くわしました。そこでは、製品のパッチが適用されたLinux 4.19ソースを見つけました。含めて、正しいDTBファイルもそこにありました。カーネルを構築するときにこのリポジトリを使用するようにBuildrootに指示するだけです。これを行うには、stm32mp157a_dk1_defconfigファイルをコピーして、名前をstm32mp157a_dk1_new_defconfigに変更します。それを開き、次のように変更します。

代わりに、

BR2_PACKAGE_HOST_LINUX_HEADERS_CUSTOM_5_4=y

私たちは書く 

BR2_PACKAGE_HOST_LINUX_HEADERS_CUSTOM_4_19=y

代わりに 

BR2_LINUX_KERNEL_CUSTOM_VERSION=y
BR2_LINUX_KERNEL_CUSTOM_VERSION_VALUE="5.4.26"

私たちは書く

BR2_LINUX_KERNEL_CUSTOM_TARBALL=y
BR2_LINUX_KERNEL_CUSTOM_TARBALL_LOCATION="$(call github,STMicroelectronics,linux,v4.19-stm32mp-r1.2)/linux-v4.19-stm32mp-r1.2.tar.gz"

ファイルを保存して閉じます。構成ファイルの準備ができました。適用しましょう（buildrootディレクトリから実行する必要があります）：

make CROSS_COMPILE=arm-linux-gnueabihf- stm32mp157a_dk1_new_defconfig

このコマンドは、構成ファイルstm32mp157a_dk1_defconfigから、buildrootディレクトリにある.configファイルに情報を転送します。将来的には、アセンブリは.configファイルに基づいて構築されます。

これで、ビルドプロセスを開始する準備がほぼ整いましたが、その前にカーネルを構成する必要があります。

microSDカードのパーティションとセクション


microSDカードにマークを付けてパーティションを作成することは、特定のハードウェアプラットフォームに密接に関連する非常に重要な段階です。残念ながら、特定のプロセッサでこの問題に関する情報を見つけるのは必ずしも簡単ではありません。完全に機能するU-BootとLinuxカーネルを収集しても、microSDカードのレイアウトでわずかなエラーが発生しても機能しません。

すぐに、STM32MP1でシステムが起動されるmicroSDカードにはGPTマークアップが必要であることに注意します。gdiskユーティリティはこれに役立ちますが、後で詳しく説明します。

microSDカードのセクションは次のようになります。



図からわかるように、カードにはfsbl1、fsbl2、ssbl、kernel、rootfsの少なくとも5つのパーティションが含まれている必要があります。さらに、1つ以上のデータセクションを作成して、それらに関する情報を格納することもできます。fsbl1とfsbl2のセクションでは、完全に同一であり、主なブートローダは（あなたは覚えているとして、これはそれらに書かれているU-ブートspl.stm32ファイル我々はU-ブート組立工程中に受信していること）。すべてが機能するはずであり、そのようなセクションが1つしかないにもかかわらず、STM2MP1のドキュメントでは、2つを実行することをお勧めしています。これらのセクションにはその他の要件が適用されます。



  • 各パーティションのサイズは256 KBである必要があります。
  • , fsbl (fsbl1 fsbl2). : , .

ssbl セクションは、U-Bootブートローダー（U-Bootのアセンブリプロセス中に受け取ったu-boot.imgファイル）を記述するためのものです。推奨されるssblパーティションサイズは2 MBです。カーネル
セクションは、Linuxカーネル（zImageファイル）、デバイスツリー（stm32mp157a-dk1.dtbファイル）、およびシステムを起動するU-Bootのスクリプトを書き込むためのものです。推奨されるカーネルパーティションサイズは64 MBです。rootfs セクションは、ルートファイルシステムを記述するためのものです。 BuildrootによってコンパイルされたルートファイルシステムとArch Linuxのルートファイルシステムに書き込みを試みます。推奨されるrootfsパーティションサイズは1 GB以上です。


データセクションは、ユーザーデータを格納するためのものです。このようなセクションを1つまたは複数作成できます。そして、あなたはそれなしで全く行うことができます。この記事では、このセクションは作成しません。

MicroSDカードの記録


したがって、現在の段階では、すべてがmicroSDカードに記録する準備ができています。これをLinuxコンピューターのカードリーダーに挿入し、mocroSDカードの最初と2番目のセクションにプライマリブートローダー（FSBL）を書き込みます。

dd if=u-boot/u-boot-spl.stm32 of=/dev/sdb1
dd if=u-boot/u-boot-spl.stm32 of=/dev/sdb2

次に、microSDカードの3番目のセクションにU-Bootを書き込みます。

dd if=u-boot/u-boot.img of=/dev/sdb3

次に、カーネル、デバイスツリーファイル、ブートスクリプトをmicroSDカードの4番目のセクションにコピーする必要があります。

ファイルのコピーを開始する前に、ダウンロードスクリプトについて少し説明する必要があります。実際、このスクリプトでは、U-Bootに関するさまざまな情報が示されています。この情報を使用して、システムを起動し、カーネルに制御を移すことができます。これらのスクリプトを記述する方法はいくつかありますが、STM32MP1のドキュメントには最も単純な（私の意見では）と説明されています。カーネルセクションのルートに/ extlinuxディレクトリを作成し、extlinux.confという名前で次の内容のテキストファイルを作成する必要があります。

LABEL stm32mp157a-dk1
KERNEL /zImage
FDT /stm32mp157a-dk1.dtb
APPEND root=/dev/mmcblk0p5 rootwait rw console=ttySTM0,115200

ここではすべてが非常に簡単です。ローダーにカーネル、デバイスツリー、ルートファイルシステムを取得する場所を指示し、作業コンソールとしてttySTM0ポートを使用するとします。

カーネルをコピーします。

cp -a buildroot/output/images/zImage /media/myuser/kernel/

注：/ media / myuser /ディレクトリで、カードリーダーに取り付けたときにmicroSDカードをマウントします。あなたの場合、それは別のディレクトリかもしれません。

デバイスツリーファイルをコピーします。

cp -a buildroot/output/images/stm32mp157a-dk1.dtb /media/myuser/kernel/

ディレクトリを作成します。

mkdir /media/myuser/kernel/extlinux

ファイルを作成します。

nano /media/myuser/kernel/extlinux/extlinux.conf

内容を記入してください：

LABEL stm32mp157a-dk1
KERNEL /zImage
FDT /stm32mp157a-dk1.dtb
APPEND root=/dev/mmcblk0p5 rootwait rw console=ttySTM0,115200

ファイルを保存してエディターを閉じます。

これで、microSDカードの4番目のセクションの準備が整いました。Linuxカーネルとそのすべての補助ファイルがすでに書き込まれています。すでにこの段階で、microSDカードをデバッグボードに挿入すると、Linuxカーネルが読み込まれますが、最終的にはルートファイルシステムをマウントできないため、カーネルパニックに陥ります。これまでに記録したので、これは当然のことです。

最終段階では、ルートファイルシステムをmicroSDカードに書き込みます。そしてここでは様々なオプションが可能です：

  1. Buildrootによって生成されたルートファイルシステムを書き込む
  2. Arch Linuxルートファイルシステムを書き換える

まず、Buildrootが生成したルートファイルシステムを書き留めて、それから始めます。これはこの記事の目的ではありませんでしたが、特にこのアクションにはあまり時間がかからないため、一般的にどのアプリケーションにも役立つ可能性があるように思えました。ルートファイルシステムは、1つのコマンドでmicroSDカードの5番目のセクションに書き込まれます。

dd if=buildroot/output/images/rootfs.ext4 of=/dev/sdb5

メモリカードをデバッグボードに挿入して、システムを起動します。 USB-UARTコンソールを介してデバッグ情報の出力を観察します。STM32MP157A-DK1ボードのmicroUSBポートを介してアクセスできます。表示された情報は、PuttyやMinicomなどの端末プログラムで表示できます。この記事では、Debianで別のターミナルウィンドウを開いて後者を使用しました。

自家製のルートファイルシステムが正常に起動することを確認したら、次はArch Linuxを起動します。もう一度、microSDカードをコンピューターのカードリーダーに挿入し、メモリカードの5番目のセクションを再度フォーマットします。

cp -a archlinux/* /media/myuser/rootfs

rm –rf /media/myuser/rootfs/boot/*

結論


この記事の中で、STM32MP157A-DK1デバッグボードを十分に試してみました。U-Boot、Linuxカーネル、独自のルートファイルシステムを構築し、Arch LinuxとDebianも起動しました。この資料が、STM32MP1ファミリのプロセッサを使用する場合、またはARM上の他のシングルボードを使用する場合の両方に役立つことを願っています。
