デザイナーなしの製品設計



KORUS Consulting CISに3年間勤務しており、この期間、19のB2Bサービスの設計に参加しました。インタラクションデザインは通常、Axure、InVision、Moqups、Framer（お気に入りのオプションを挿入）に関連付けられていますが、私のツールはHTML、SCSS、AngularJsです。 HTMLテンプレートを保存する通常の方法が本格的なレイアウトのアルマナックでどのように成長したか、およびアートディレクターが率いるタイプセッターのグループがすべてのKORUS Consulting CIS製品のインターフェイスとの相互作用を6年間どのように設計したかを説明します。

そして、それはなぜうまくいったのか。

「相互作用の設計者はどこから来たのか」の章の「患者の手にある精神病院」で、アランクーパーは、必要なスキルは開発のどの参加者でも簡単に見つけられると書いています。設計は、プロジェクトマネージャー、アナリスト、テクニカルライター、マーケティング担当者、開発者が実施できます。この機能が別の専門家、できれば設計者に引き継がれるのは良いことです。KORUSはこの役割をコーダーに与えました。しかし、すぐには。

拒否されたPhotoshop


2019年まで、KORUSにはデザイナーはいませんでした。そして2011年まで、ユーザーエクスペリエンスなどのことを聞く人はいませんでした。自尊心のあるソフトウェア開発者として、同社は製品を迅速に市場に投入することを目指し、アナリストと開発者はインターフェースの外観と動作に責任を負っていました。主に開発者。

したがって、2011年にデザイナーのスキルを持つコーダーが会社に登場したとき、開発の前には最終段階でのみ許可されていました。

KORUSスペシャリストの新しい（読み取り-異常）は、主な機能がすでに実装されているときに、ピクセルの移動とペイントを許可された典型的な開発スキームに遭遇しました。開発後のデザインのように聞こえます。そして、これは精神病院、クーパー氏に非常に詳細に説明されています。 KORUSは彼の物語の教科書のヒーローと見なすことができました。

Photoshopや他のグラフィックエディターは、この開発プロセスに苦労しました。特に要件が常に変化する状況では、編集が容易になるため、当初からレイアウトに依存していました。そして、あなたは開発者にレイアウトを与えることができます。美しさをデザインし、組版し、刺激を与えると同時に、はるかに効率的です。このアプローチでは、任意のエディターでの描画は不要な中間リンクであり、2011年にはおそらく多くの人がそれに同意するでしょう。

設計ツールの拒否は、定評のあるスキームの上に新しい道を築くのに役立ち、インターフェースの正しい設計への長い道のりを開始しました。開発時間が限られている場合、PSDレイアウトは単なる推奨になり、レイアウトは一連のルールのようになり、向きを変えて見えないケースを忘れるのが難しくなります。または、便利さについてのあなた自身の考えに基づいて、それを考慮してください。

レイアウトでは、次にコンポーネントに時間を浪費しないように個々のコンポーネントを保存することもできます。そのようなコンポーネントを含む最初のHTMLページを以下に示します。



これは、AngularJの将来のキットのリポジトリの最初のブリックになりました。当初は、電子文書管理サービス（以下、クーリエと呼ぶ）の印刷フォームと個別ページが追加されていました。

リポジトリが大きくなり、さらに多くのレイアウトがあったとき、それはUIKITと呼ばれ、ロゴには子猫忍者と呼ばれていました。



2015年、当社はSberbankとのやり取りに全力を注ぎ、サービスの再描画に関する疑問が生じました。KORUSは、コンポーネントとスタイルにすでに実績があるという事実を繰り返し使用しています。ビーライン、アルファバンクなどの企業スタイルに合わせて色を変更するだけで十分です。この経験のおかげで、クーリエはわずか1時間で再描画されました。



また、最初にデザイナーがグラフィックレイアウトを再描画し、レイアウトデザイナーがスタイルを変更したとしたらどうでしょう。2つのタスク-1つのソリューション：すでにUIKITでレイアウト。

彼らが「悪い」ことをしないようにするには？


驚いたことに、最初の4年間は、1人の専門家だけがすべての設計作業を担当していました。UIKITでの開発は保存されました。これにより、時間を大幅に節約できます。設計者は、開発にできるだけ早く接続し、ひどいインターフェイスソリューションを防ぐことができました。

製品ポートフォリオの拡大は当然、新しいレイアウトデザイナーの登場につながりました。これにより、クジラ忍者はより多くのリソースを割り当てることができ、レイアウトはよりインタラクティブになりました。HTMLリンクを相互にリンクするだけでは不十分です。JQueryとAngularJが使用されました。ライブプロトタイプは、主にアナリストによって評価されました。すべての州をクリックして、スクリーンショットを撮り、TKに貼り付けることができます。
「BEMではなくBEMと同様のことを行ったため、すべてのプロジェクトでレイアウトの統一に取り掛かりました」
しかし、レイアウトデザイナですべてが明らかな場合、レイアウトデザイナは、ガイドラインと、単純なものを設計するためにさまざまなレイアウトを組版する彼自身の豊富な経験にガイドされるだけでしょうか？レイアウトデザイナは、インターフェイスをデザイナとして組み立てることができ、誰かがボタンを配置する場所を教えてくれるのを待つ必要はありません。彼らはすでにツールバーにこのボタンを何度も見ています。デザイナーはこれをあいまいにしています。そして、KORUSは、「ナチュラルデザイン」と呼ばれる理解可能なライフケースを必要としていました。また、1つのエコシステムの製品は通常互いに類似しており、一意性がそれらに害を与えるだけであることを思い出した場合、私の答えは「はい」です。

UIKITの登場により、プロトタイピングプロセスが加速しました。クリック可能なレイアウトのデモンストレーションは、プロジェクトの開始前にプレセール、顧客との詳細の調整に使用されました。そして最後に、開発者はインターフェースの振る舞いに関する必要な入力を受け取り、コーディングプロセスでそれ自体が思い付かないようにしました。
「前線が彼らを楽にしてくれました。UIKITは部分的に私にそれを正しくさせました。」

AngularJsとの対話性


UIKITの登場後にKORUSにもたらされたのは、まさにそのような作業スキームでした。ニーズが発生し、技術要件で形式化され、レイアウトデザイナーはそれをWebプロトタイプにすぐに具体化しました。
インターネットの奥深くからいくつかのフレームワークを取得し、それをグリーンラベルの下に再描画し、製品に実装して、ボタンとヘッダーだけでなくすべてのサービスを統一する必要がありました。



フレームワークのコントロールに加えて、UIKITには一連のアイコン、ウィジェット、独自の開発およびスタイルが補充されました。

サービスに似たページレイアウトのタスクがあった場合、完成したものを取得し、コンテンツを要件に変更しましたが、ラッパーとロジックは同じままでした。ページ間の遷移を構成でき、すべてのボタンとフィールドはクリック可能です。検証、モーダルウィンドウ、ファイルアップロードの成功を表示できます。ロールモデル全体の表現で、ポジティブシナリオとネガティブシナリオでユーザーパス全体を最初から最後まで表示できます。

均一なインターフェース


インターフェイスの事前に設計された部分を再利用することは、2015年の新機能ではありません。 10年前でも、Webスタジオのレイアウトデザイナーは、同じ要素を毎回タイプセットし直さないようにプロセスを最適化していました。私たちは銀河を発見しませんでしたが、これは始まりに過ぎませんでした。

UIKITにレイアウトを追加すればするほど、既存のケースがカバーされるようになり、新しいプロジェクトにさらに適用できるようになることは明らかです。

新しいユーザー登録レイアウトが必要ですか？これはすでにあります。ビジネスのニーズに合わせて調整し、2、3時間以内で済むようになりました。これで、登録は開発者に転送できるようになりましたが、別のサービスで行われます。

会社データのテーブルが必要ですか？サービスのこの良さをまとめて：CRTL + C CTRL +V。

この中でレイアウトチームにとって最も魅力的なのは、統一されたスタイルです。ガイドラインは、インターフェースのアトミック要素がどのように見えるべきかを指示するものです。これらのコンポーネントのスタイルを修正し、それを最後の手段としてのみ変更することは理にかなっているようです。UIKITで紹介する新しい製品レイアウトがいくつあっても、コンポーネントのスタイルは1か所にあります。プロジェクト自体のニーズのために、何かをカスタマイズできる個別の製品CSSがあります。

ガイドライン、最適化、および私たち自身の経験の影響を受けて、私たちは常に手元にありました：

  • 入力、ボタン、ドロップダウン。
  • テーブル、両面、メニュー、ツールバー。
  • フォーム、タイムライン、モーダルウィンドウ。
  • ある種のローン申請の処理または支払文書の作成の全体的な流れ。

これらのことから、すでに本格的なインターフェースを組み立てることが可能であり、AngularJsのおかげで、レイアウト上で生産の双方向性が実現され、顧客に非常に人気がありました。

秘密裏に言えば、UIKITでお客様のために数回デモを行ったと言います。デモの直前にテストスタンドが落ちていたからです。

作業回路


時間の経過とともに、UIKITのイデオロギーのインスピレーターでありクリエーターがアートディレクターになり、プロジェクトはますます増えました。誰がこの素晴らしい仕事を続けるのか-すでに開発された機能を使用してHTMLでインターフェースを設計するのか？もちろんレイアウトデザイナー。

レイアウト設計者がアナリストと緊密に連携し、ユーザビリティを介して独自に考えるという見通しに非常に満足していると言うのは少しトリックです。それでも、誰もが（私は？）レイアウトデザイナーは、デザイナーからレイアウトを受け取り、それをカットしてWebページに変換するスペシャリストであることに慣れています。相互作用の設計により、機械的な作業に、まだ研究されていない異常な責任が追加されます。しかし、私たちは常に私たちの仕事を修正し、正しいやり方を提案する人の存在に助けられました。また、彼はただプロンプトを出すだけでなく、信頼できる情報源を参照したので、それらを読んで間違いを減らすことができました。アートディレクターは、フィッツのいくつかの法則、次にビジネスライティングのルール、次にテキストの書式設定を常にポイントし、要素を強調しました...抵抗しましたが、情報を吸収しました。

今日、2つの緑色のボタンを並べて配置することができました。明日は、私たちが知っていたので、それを実行しませんでした。ターゲットアクションを持つボタンは、ページごとに1つである必要があります。

今日、すべての入力が同じ幅で動かなくなっていました。明日は、コンテンツのTINのフィールドを正確に作成します。したがって、ユーザーは、データが予想される期間をよりよく読み取ることができます。

プロセスは順調に進んだ。レイアウトデザイナーは、アナリストからのビジネス要件に依存してインターフェイスを収集し、その決定をアートディレクターと調整し、開発に割り当てます。

私たちはデザイナーとして成長し、意思決定を学び、アートディレクターに深く没頭する必要のないタスクの範囲を拡大するための自立を育みました。

そのパイプラインのどこかに、レイアウトのレビューであるGitLabが現れ、jsを使用した視覚化が改善されました。

レイアウトデザイナが標準的でない困難な作業に直面した場合、アートディレクターがつながり、解決策を考え出すのに役立ちました。そのとき彼自身が新製品のコンセプトをデザインしました。
私たちは問題を理解するために2〜3つのオプションを提供し、彼は1つを選択し、それをどのように改善できるかを述べました。時々、彼は単に「大丈夫」と言っただけで、私たちは目標を達成したことを理解して喜んでいました。ただし、私たちのソリューションがユーザーにとって便利で理解しやすい方法を理解するための十分なフィードバックがなかったり、それがアートディレクターにとって明確であるかどうかさえわからなかったりすることがあります。疑いは私たちに何度も何度も何度も試して、いくつかの方法で要件を解釈させました。

私たちのリーダーはこれについて彼自身の意見を持っていました。このように考えさせられたと彼は言った。それぞれ独自の方法でプロセスを評価しました。私は自分と彼の時間を節約していると思いました。 ３回歩かないように３つのオプションを一度にご紹介しましょう。 A / B / Cのような大砲のテストです。

会社の認識

時間の経過とともに、新しいプラグインを接続し、UIKITのいくつかのスタンドを整理し（現在の開発では内部、固定バージョンでは外部、今後開発される）、スタイルのバージョン管理を追加し、スタイルパックをアーティファクトとネクサスにインポートし、npmパッケージを容易にする必要がありました。メディアファイルをCDNに転送しています。その後、SASSに切り替えました。一見普通のレイアウトのサイトコレクションは、その展開、チームからの更新、および将来の改善に関する活発な議論により、本格的なプロジェクトに変わりました。つまり、当初はラピッドプロトタイピングの支援に過ぎなかった私たちのプロジェクトは、固有のプロセスを備えた本格的な製品になりました。
