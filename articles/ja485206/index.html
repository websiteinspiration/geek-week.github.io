Typescriptはどのように私をがっかりさせましたか？



始める前に、私はTypeScriptのファンであることを述べておきます。これは、ReactのフロントエンドプロジェクトおよびNodeで行うすべてのバックエンド作業のための私の主要なプログラミング言語です。私は完全にTypescriptに興味がありますが、気になる瞬間や、この記事で伝えたい瞬間があります。

私は多くの異なる会社のために過去3年間にわたってTypeScriptについてのみ書いたので、私の意見では、TypeScriptは少なくとも何かを正しく行うか、特定のニーズを閉じます。

TypeScriptはその不完全性にもかかわらず、主流の開発フロントエンドに入り、HackerRank開発者スキルレポートによると、最も人気のあるプログラミング言語のリストで7位にランクされています。

大小に関わらず、開発チームはTypeScriptで書くかどうかに関わらず、セキュリティのために常にその価値があります。

  • 適切に作成された単体テストが、本番環境で可能な限り多くのコードをカバーするようにします
  • ペアプログラミングを使用する：追加の目を使用すると、構文エラーだけでなく、より深刻な問題を検出できます
  • コードレビュープロセスを定性的に構築し、マシンが検出できないエラーを特定する
  • リンターを使用-eslintなど

TypeScriptはこれに加えてセキュリティのレベルを追加しますが、私の意見では、この点で他の言語よりもはるかに遅れています。その理由を説明します。

TypeScriptは信頼できる型システムではありません


私は、これは活字体の主な問題かもしれないと思うが、最初に私が何を定義してみましょう、信頼性と信頼性の低いタイプのシステムです。

ロバストタイプシステム


信頼性の高い型システムは、プログラムが無効な状態にならないようにします。たとえば、式の静的な型がstringの場合、実行時に評価されると、必ずstringのみが取得されます。

信頼できる型システムでは、コンパイル時または実行時に、式が期待される型と一致しない状況になることは決してありません。

もちろん、信頼性にはさまざまな程度があり、信頼性の解釈もさまざまです。TypeScriptはある程度信頼でき、型エラーをキャッチします。

// Type 'string' is not assignable to type 'number'
const increment = (i: number): number => { return i + "1"; }

// Argument of type '"98765432"' is not assignable to parameter of type 'number'.
const countdown: number = increment("98765432");

安全でない型システム


Typescriptは、100％の信頼性がその目的ではないことを完全に公然と報告しています。TypeScriptの「非ターゲット」リストの「非ターゲット」番号3でさえ、明確に次のように述べています。
信頼性のある、または「おそらく正しい」タイプのシステムを持つことは、私たちの目標ではありません。代わりに、精度とパフォーマンスのバランスを取るよう努めています。

つまり、実行時に変数が特定の型であるという保証はありません。次のやや工夫された例でこれを説明できます。

interface A {
    x: number;
}

let a: A = {x: 3}
let b: {x: number | string} = a; 
b.x = "unsound";
let x: number = a.x; // unsound

a.x.toFixed(0); // WTF is it?

上記のコードは、ax-がインターフェースAの番号であることがわかっているため機能しません。残念ながら、再割り当てによるいくつかのフェイントの後、それは文字列に変わり、このコードはコンパイルされますが、実行時にエラーが発生します。

残念ながら、この式はエラーなしでコンパイルされます。

a.x.toFixed(0);

その信頼性が言語の目標ではないことは、おそらくTypeScriptの最大の問題の1つです。実行時にtscコンパイラーがキャッチしない多くの実行時エラーが発生し続けますが、TypeScriptに信頼できる型システムが存在する場合、コンパイラーはこれに気づくでしょう。 TypeScriptは、「信頼性のある」言語の陣営の1つになり、もう1つは「信頼性のない」言語の陣営になりました。このハーフメジャーアプローチは、後で説明するanyタイプに基づいています。

TypeScriptへの移行によって、作成するテストの数がまったく減らなかったので、私は不満を感じています。始めたばかりの頃、大量の単体テストを作成する煩わしいルーチンを減らすことができると誤って判断しました。

TypeScriptは、現在の順序に異議を唱え、型を使用するときの認識コストを下げることは、型の信頼性よりも重要であると主張します。

TypesScriptがこのような方法を選択した理由を理解しています。また、型システムの信頼性が100％保証されている場合、TypeScriptはそれほど人気が​​ないだろうという意見があります。この意見はテストに耐えられませんでした-Dutter言語は Flutterの広範な使用と共に急速に人気を博しています。そして、型の信頼性がダートの目標であると主張されています。

TypeScriptが強力な型付けから「緊急脱出」するための不安とさまざまな方法により、効率が低下し、残念ながら、「何もないよりはまし」になります。現在。TypeScriptの人気が高まるにつれて、より多くのコンパイラーオプションが利用可能になり、経験豊富なユーザーが100％の信頼性を追求できるようになればうれしいです。

TypeScriptは実行時の型チェックを保証しません


実行時の型チェックはTypeScriptの目的ではないので、私の願いはおそらく叶わないでしょう。ランタイム型チェックは、たとえば、API呼び出しから返されたJSONデータを操作するときに役立ちます。これらのプロセスを型システムのレベルで制御できれば、エラーのカテゴリ全体と多くの単体テストを取り除くことができます。

実行時には何も保証できないため、これは簡単に起こります。

const getFullName = async (): string => {
  const person: AxiosResponse = await api();
  
  //response.name.fullName may result in undefined at runtime
  return response.name.fullName
}

io-ts などのヘルパーライブラリはいくつかありますが、これはすばらしいことですが、モデルを複製する必要がある場合があります。

結論


私はTypeScriptのファンであり、日常の仕事でそれを使用することを繰り返す必要がありますが、それは不完全であり、周囲の誇大宣伝は完全に正当化されていないと感じています。Airbnb はTypeScriptがエラーの38％を防ぐのに役立つと主張しています。私はそのように正確に述べられているパーセンテージについて非常に懐疑的です。TypeScriptは、優れたコードのすべての既存のプラクティスを改善または組み合わせません。私はまだたくさんのテストを書かなければなりません。私はより多くのコードを書くと主張することができるので、私は非常に多くのテストを書かなければなりません。予期しないランタイムエラーが何度も発生します。

TypeScriptは基本的な型チェックのみを提供し、実行時の信頼性と型チェックは目標ではないという事実により、TypeScriptは、最高の世界と現在コーディングしている世界の中間の半分の測定の世界に残ります。

TypeScriptは、vscodeなど、印刷プロセス中に視覚的なフィードバックを得るIDEの優れたサポートのおかげで非常に優れています。

TypeScriptは優れた型チェックを提供し、型チェックなしや単純なeslintよりも間違いなく優れていますが、TypeScriptの方がはるかに優れており、必要なコンパイラーオプションによって、言語をもっと必要とする人を満足させることができます。
