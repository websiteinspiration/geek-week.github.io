Spring Boot + Apache Kafka and SSL in the Docker container

Hi friends. In this article, I will tell you how to build a solution to quickly start your development from standalone Java application, broker and Kookka Zookeeper, raised in separate Docker containers and configured with plaintext authorization for access from both the local machine and the external circuit.

The finished project on github can be picked up at the end of the article



Choosing a docker Apache Kafka + Zookeeper image


First you’ll have to decide on the Apache Kafka image vendor for our container. The requirements of each specific project vary in the necessary level of security and reliability of the solution, in some cases, of course, you will have to assemble your own image, but for most projects it will be wise to choose one available on the docker hub. For 2020, there are three main popular vendor images kafka + zookeper:

  • confluent images [subjectively, a big plus - they are supplied by the developers of Apache Kafka itself and are the most reliable in terms of information security]
  • images from bitnami [plus, in comparison with confluent, that the images need a minimal configuration and “start out of the box”]
  • wurstmeister [ , github hub.docker, ]

Docker-compose


To launch and configure images, docker-compose is used , which makes it easy to run multi-container applications. You can not do without it, because the minimal configuration of kafka provides for a bunch of zookeeper and at least one broker.

ompose is bundled with mac / windows - the Docker distribution, so installing the necessary environment comes down to downloading and installing the distribution from docs.docker.com/engine/install .

But linux users need to install it separately. For a docker + compose installation guide, it's best to consult the official docs.docker.com/compose/install documentation .

Writing a yaml file for compose


The general scheme of our configuration for all clients and containers will be as follows:



One broker with open ports 9093 will be connected to one coordinator - for localhost connections and 9092 for connections from other machines.

To access from external networks, enable authorization via login password without ssl encryption. To work through ssl, you need to issue your own keys and sign certificates, the whole process is well described in the official documentation .

As a result, the file will look like this (the address needs to be replaced with your public IP for access from other networks):

version : ' 2 '
services :
zookeeper :
image : wurstmeister / zookeeper
expose:
- "2181"
kafka:
image: wurstmeister/kafka
ports:
- "9092:9092"
expose:
- "9093"
environment:
KAFKA_AUTO_CREATE_TOPICS_ENABLE: 'true'
KAFKA_ADVERTISED_LISTENERS: INSIDE://kafka:9093,OUTSIDE://<your public IP here>:9092
KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: INSIDE:PLAINTEXT,OUTSIDE:SASL_PLAINTEXT
KAFKA_LISTENERS: INSIDE://0.0.0.0:9093,OUTSIDE://0.0.0.0:9092
KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
KAFKA_INTER_BROKER_LISTENER_NAME: INSIDE
KAFKA_OPTS: "-Djava.security.auth.login.config=/etc/kafka/kafka_server_jaas.conf"
KAFKA_SASL_ENABLED_MECHANISMS: PLAIN
KAFKA_SASL_MECHANISM_INTER_BROKER_PROTOCOL: PLAIN
depends_on:
- zookeeper
volumes:
- ./:/etc/kafka

In order to enable authorization, you must transfer the kafka_server_jaas.conf file to kafka. In the folder with the compose yaml file, create kafka_server_jaas.conf and add one admin user with the admin-secret password in it:

KafkaServer {
org.apache.kafka.common.security.plain.PlainLoginModule required
username = "admin"
password = "admin-secret"
user_admin = "admin-secret";
};
Client {};
view raw kafka_server_jaas.conf hosted with ❤ by GitHub

This file will be transferred to the image with kafka broker through:

  1. specifying the path to the file in the kafka config directly
    KAFKA_OPTS: "-Djava.security.auth.login.config=/etc/kafka/kafka_server_jaas.conf"
  2. mapping of the local directory with the file in / etc / kafka the image directory
    volumes:
    - ./:/etc/kafka

Container launch


As a result, we should get 2 files that are in the same directory:

  1. docker-compose.yml
  2. kafka_server_jaas.conf

In this directory you need to call the command:

$ docker-compose up -d

the -d flag allows you to start in Detached mode and close the console if necessary without turning off the images.

Java client on Spring Boot


We will write a minimalistic client with the ability to publish and read messages from kafka. To do this, create a new project through the Spring Initializr or any other tool familiar to you. It is enough to connect one dependency:



I prefer to configure Spring Boot applications whenever possible through configuration yaml files with minimal configuration in Java code. Due to the built-in structuring in the Yaml format, these files are much more convenient for human perception than .properties files and allow you to group settings by modules. The only drawback of the .yml configuration is that it is not supported in tests, but it is easily bypassed by the built-in boot tools.
I highly recommend trying to see how your perception of the old .properties file changes (especially for large projects) if you rewrite it in yaml. This can be done in a couple of clicks at www.toyaml.com/index.html .

In the project resources, create an application.yml file with the following contents:

example :
kafka :
consumer-enabled : $ {consumer-enabled: true}
spring :
kafka :

properties :
sasl :

config: org.apache.kafka.common.security.plain.PlainLoginModule required username=${kafka_username:'admin'} password=${kafka_password:'admin-secret'};
mechanism: PLAIN

client-id: example

This setup works for sequential proofreading on one message ( max-poll-records: 1 ) by one listener ( concurrency: 1 ) and requires manual confirmation of message processing in listener ( ack-mode: manual_immediate ).

@Service
@SuppressWarnings ( " SpringJavaInjectionPointsAutowiringInspection " )
public class Producer {
private final KafkaTemplate < String , String > kafkaTemplate;
public Producer(KafkaTemplate<String, String> kafkaTemplate) {
this.kafkaTemplate = kafkaTemplate;
}
public ListenableFuture<SendResult<String, String>> sendMessage(String topic, String key, String message) {
return this.kafkaTemplate.send(topic, key, message);
}
}

@Service

}
