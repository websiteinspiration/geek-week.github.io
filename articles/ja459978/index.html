Node.js開発者ツール。Webソケットでのリモートプロシージャコール

ホラーストーリーは、多くの場合、たとえばWebブラウザーでサポートされていない、またはプロバイダー/管理者がWebソケットトラフィックを抑制しているため、アプリケーションで使用できないことなど、Webソケットテクノロジーについて語っています。一方、開発者は、他のテクノロジーと同様に、websocketテクノロジーの落とし穴を常に予測しているわけではありません。申し立てられている制限については、今すぐWebブラウザーの96.8％がWebSocketテクノロジーをサポートしていると言います。船外に3.2％残っているのはたくさんあり、これらは何百万ものユーザーです。まったく同感です。すべてを比較しただけで知られています。誰もがAjaxで長年使用しているのと同じXmlHttpRequestは、97.17％のWebブラウザーでサポートされています（それ以上ではありませんか？）。そして、フェッチ-一般に、93.08％のWebブラウザー。 WebSocketとは異なり、そのようなパーセンテージ（以前はそれより少なかった）は、Ajaxテクノロジーの使用を長い間止めてきました。そのため、ロングポーリングでフォールバックを使用しても、現時点では意味がありません。 WebSocketをサポートしていないWebブラウザーがXmlHttpRequestをサポートしていないWebブラウザーと同じであるためであり、実際にはフォールバックは発生しません。

企業ネットワークのプロバイダーまたは管理者によるWebSocketの禁止に関する2番目の恐怖の話も不合理です。これは、誰もがhttpsプロトコルを使用しており、WebSocket接続が開いている（httpsを壊さずに）ことを理解することができないためです。

実際の制限とそれらを克服する方法については、この投稿で、アプリケーションのWeb管理領域を開発する例について説明します。

したがって、WebブラウザのWebSocketオブジェクトには、率直に言って、send（）とclose（）の非常に簡潔なメソッドのセットと、EventTargetオブジェクトから継承されたaddEventListener（）、removeEventListener（）、およびdispatchEvent（）メソッドがあります。したがって、開発者はいくつかの問題を解決するためにライブラリ（通常）または独立して（ほとんど不可能）を使用する必要があります。

最も理解しやすいタスクから始めましょう。サーバーへの接続は定期的に中断されます。再接続は簡単です。しかし、この時点でメッセージがクライアントとサーバーの両方から送信され続けることを覚えている場合は、すべてが即座に、しかもはるかに複雑になります。一般に、受信したメッセージの確認メカニズムが提供されていない場合、メッセージは失われるか、確認メカニズムが提供されている場合、メッセージは再配信されます（複数回でも）。

保証されたメッセージ配信やテイクなしのメッセージ配信が必要な場合は、これを実装するための特別なプロトコル、たとえば、WebSocketトランスポートで機能するAMQPやMQTTがあります。しかし、今日は考慮しません。

ほとんどのWebSocketライブラリは、サーバーへの透過的なプログラマの再接続をサポートしています。このようなライブラリを使用すると、独自の実装を開発するよりも常に信頼性が高くなります。

次に、非同期メッセージを送受信するためのインフラストラクチャを実装する必要があります。これを行うには、追加のバインディングなしの「裸の」onmessageイベントハンドラーを使用します。このようなインフラストラクチャは、たとえば、リモートプロシージャコール（RPC）です。id idはjson-rpc仕様に導入されました。特にwebsocketトランスポートを操作するために、クライアントによるリモートプロシージャコールをwebサーバーからの応答メッセージにマッピングできるようになりました。私は他のすべての可能性よりもこのプロトコルを好みますが、これまでのところ、node.jsのサーバー部分にこのプロトコルをうまく実装できていません。

そして最後に、スケーリングを実装する必要があります。クライアントとサーバー間の接続は定期的に発生することを思い出してください。 1台のサーバーの能力では不十分な場合は、さらに数台のサーバーを調達できます。この場合、接続が切断された後、同じサーバーへの接続は保証されません。通常、redisサーバーまたはredisサーバーのクラスターは、複数のWebSocketサーバーを調整するために使用されます。

そして、残念ながら、遅かれ早かれ、同時にシステムパフォーマンスが発生します。同時に開くwebsocket接続の数におけるnode.jsの機能（これを速度と混同しないでください）は、メッセージキューやブローカーなどの専用サーバーの機能よりも大幅に低いためです。また、重要なポイントが発生した後、redisサーバーのクラスターを介してwebsocketサーバーのすべてのインスタンス間で交換を行う必要が生じても、開いている接続の数は大幅に増加しません。この問題を解決する方法は、WebSocketトランスポートを含めて機能するAMQPやMQTTなどの専用サーバーを使用することです。しかし、今日は考慮しません。

リストされているタスクのリストからわかるように、WebSocketでの作業中のサイクリングは非常に時間がかかり、ソリューションを複数のWebSocketサーバーに拡張する必要がある場合は不可能です。

したがって、私はwebsocketでの作業を実装するいくつかの一般的なライブラリを検討することを提案します。

廃止されたトランスポートモードにのみフォールバックを実装するライブラリは考慮から除外します。今日、この機能は関係がなく、より広範な機能を実装するライブラリは、原則として、廃止されたトランスポートモードにフォールバックも実装しています。

まず、最も人気のあるライブラリであるsocket.ioから始めます。これで、このライブラリは遅く、リソースを大量に消費するという意見を聞くことができます。最も可能性が高く、ネイティブのWebSocketよりも動作が遅くなります。しかし、今日、それはその手段によって最も開発されたライブラリです。また、WebSocketを使用する場合の主な制限要因は速度ではなく、一意のクライアントとの同時オープン接続の数です。そして、この問題は、クライアントと特殊なサーバーに接続することですでに解決されています。

そのため、サーバーから切断し、サーバーまたはRedisサーバーのクラスターを使用してスケーリングする場合、soket.ioは信頼性の高い回復を実装します。実際、socket.ioは独自のメッセージングプロトコルを実装しています。これにより、特定のプログラミング言語に縛られることなく、クライアントとサーバー間のメッセージングを実装できます。

socket.ioの興味深い機能は、任意のオブジェクトをサーバーからクライアントに返すことができるイベント処理の確認です。これにより、リモートプロシージャコールが可能になります（ただし、json-rpc標準には準拠していません）。

import io from 'socket.io-client';

const socket = io({
  path: '/ws',
  transports: ['websocket']
});

export default (action, collection, payload = {}) =>
  new Promise((resolve, reject) => {
    socket.emit('remote-call', {action, collection, payload}, (response) => {
      if (response.error) {
        reject(response);
      } else {
        resolve(response);
      }
    });
});

残念ながら、サーバー側でもう少し作業を行う必要がありました。リモート呼び出しを処理する関数のマッピングを整理するために、express.jsルーターに似たルーターが開発されました。ミドルウェア（req、res、next）署名の代わりにのみ、実装は署名（ソケット、ペイロード、コールバック）に依存します。その結果、通常のコードを取得しました。


const Router = require('./router');

const router = Router();

router.use('GET_LIST',  (socket, payload, callback) => {
  const limit = Number(payload.pagination.perPage);
  const offset = (Number(payload.pagination.page) - 1) * limit
  return callback({data: users.slice(offset, offset + limit ), total: users.length});
});

router.use('GET_ONE',  (socket, payload, callback) => {
  return callback({ data: users[payload.id]});
});

router.use('UPDATE',  (socket, payload, callback) => {
  users[payload.id] = payload.data
  return callback({ data: users[payload.id] });
});
