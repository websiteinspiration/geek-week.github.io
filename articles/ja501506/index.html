GDIリークと運の重要性について


2019年5月に、潜在的に危険なChromeバグを確認するように依頼されました。最初、私は彼を重要ではないと診断し、このように2週間も無駄にしました。その後、調査に戻ったところ、Chromeベータチャネルでブラウザプロセスがクラッシュする最大の原因になりました。おっとっと

6月6日、出発のデータを解釈する際の間違いに気付いたその日、バグはReleaseBlock-Stableとマークされました。つまり、何が起こっているのかを理解するまで、ほとんどのユーザーに新しいバージョンのChromeをリリースすることはできません。

クラッシュは、GDIオブジェクト（グラフィックスデバイスインターフェイス）が不足しているために発生しましたが、それらがどのタイプのGDIオブジェクトであるかを知りませんでした。診断データは、問題がどこにあるのかについての手掛かりを与えず、再作成できませんでした。

私たちのチームの多くの人々が6月6〜7日にこのバグに懸命に取り組み、彼らの理論をテストしましたが、前進しませんでした。 6月8日にメールを確認することにしたところ、Chromeがすぐにクラッシュしました。それはだった同じ失敗。

なんと皮肉なことでしょう。変更を探してクラッシュレポートを調べ、ChromeブラウザープロセスがGDIオブジェクトをリークする原因を突き止めようとしていたところ、ブラウザー内のGDIオブジェクトの数は絶え間なく増加し、6月8日の朝までに10,000という魔法の数を超えました。この時点で、GDIオブジェクトにメモリを割り当てる操作の1つが失敗し、意図的にブラウザーをクラッシュさせました。それは信じられないほどの幸運でした。

バグを再現できれば、必然的に修正できます。私はこのバグをどのようにして引き起こしたのかを理解する必要があり、その後それを排除することができます。

手始めに、問題の短い歴史



Chromiumコードのほとんどの場所で、GDIオブジェクトにメモリを割り当てようとするとき、最初にこの割り当てが成功したかどうかを確認します。メモリを割り当てることができなかった場合は、このソースコードからわかるように、スタックに情報を書き込んで意図的にクラッシュを実行します。 GDIオブジェクトにメモリを割り当てることができない場合、画面にレンダリングできないため、障害は意図的に発生します。問題を報告し（クラッシュレポートが有効になっている場合）、空のUIを表示するよりもプロセスを再起動することをお勧めします。既定では、プロセスごとに最大10,000のGDIオブジェクトを作成でき、通常は数百のみが使用されます。したがって、この制限を超えた場合、何かが完全に失敗しました。

GDIオブジェクトのメモリ割り当てエラーに関するクラッシュレポートの1つを取得すると、コールスタックとその他のあらゆる種類の有用な情報が得られます。いいね！しかし問題は、そのようなクラッシュダンプが必ずしもバグに関連しているとは限らないことです。これは、GDIオブジェクトのリークを引き起こすコードと、エラーを報告するコードが同じコードではない可能性があるためです。

つまり、大まかに言えば、2種類のコードがあります。

void GoodCode（）{
   auto x = AllocateGDIObject（）;
   もし（！x）
     CollectGDIUsageAndDie（）;
   UseGDIObject（x）;
   FreeGDIObject（x）;
}

void BadCode（）{
   auto x = AllocateGDIObject（）;
   UseGDIObject（x）;
}

良いコードは、メモリの割り当てに失敗したことに気づき、これを報告します。悪いコードは、クラッシュを無視してオブジェクトをこぼし、責任があるように良いコードを「置き換え」ます。

Chromiumには数百万行のコードが含まれています。エラーが発生した関数や、リークしているGDIオブジェクトの種類すらわかりませんでした。同僚の 1人が、クラッシュ前にプロセス環境ブロックをバイパスして各タイプのGDIオブジェクトの数を取得するコードを追加しましたが、列挙されたすべてのタイプ（デバイスコンテキスト、エリア、ビットマップ、パレット、ブラシ、羽、不明）の数は100を超えませんでした。変わっている。

メモリーを直接割り当てるオブジェクトはこのテーブルにありますが、カーネルによって作成されたオブジェクトはなく、Windowsオブジェクトマネージャーのどこかに存在していることがわかりました。これは、GDIViewがこの問題に対して私たちと同じくらい盲目であることを意味しました（加えて、GDIViewはローカルで障害を再生する場合にのみ役立ちます）。リークしたカーソルがあり、カーソルはGDIオブジェクトがアタッチされたUSER32オブジェクトであるためです。これらのGDIオブジェクトのメモリはカーネルによって割り当てられ、何が起こっているのかを確認できませんでした。

誤解


私たちの関数CollectGDIUsageAndDieは非常に鮮明な名前を持っています、そして私はこれについて私に同意すると思います。とても表情豊か。

問題は、実行するアクションが多すぎることです。 CollectGDIUsageAndDieは、GDIオブジェクトの約12種類の異なるメモリ割り当てエラーをチェックし、コードの埋め込みにより、結果として同じエラーシグネチャを受け取りました。これらはすべてメイン関数にクラッシュし、マージされました。したがって、私の同僚の1人が賢明に変更を加え、異なるチェックを個別の（組み込みではない）関数に分割しました。これにより、一見したところ、どのチェックが失敗したかがわかりました。

悲しいかな、これにより、CrashIfExcessiveHandlesからクラッシュレポートを取得し始めたときに、、自信を持って言った：「これは失敗の原因ではなく、単に署名の変更が原因です。」

しかし、私は間違っていました。これが失敗の原因であり、署名が変更されました。おっとっと ぎこちない分析、ドーソン。あなたのためのクッキーはありません。

ストーリーに戻る


この時点で、6月7日に行った作業で1日あたり約10,000個のGDIオブジェクトが使用されていることをすでに知っていました。それが理解できたらなぞなぞを解きます。


Windowsタスクマネージャーには、リークを見つけるために使用できる追加のGDIオブジェクト列があります。 6月7日、自宅から作業機に接続して作業していましたが、テストを実行して障害シナリオを再現しようとしたため、この列は作業機でオンになりました。しかし、その間、私の自宅のマシンのブラウザにGDIオブジェクトのリークがありました。

自宅でブラウザを使用した主なタスクは、Chromeリモートデスクトップ（CRD）アプリケーションを使用して、稼働中のマシンに接続することです。そこで、ホームマシンのGDIオブジェクト列をオンにして、実験を開始しました。すぐに結果が出ました。

実際、このバグのタイムラインは、「私は失敗した」（14:00）から「なんらかの理由でCRDに接続されている」から、「カーソルのケース」まで35分しか経過していないことを示しています。バグをローカルで再生できると、バグの調査がどれほど簡単になるかはすでに述べました。

CRDアプリケーション（またはChromeアプリケーション？）がカーソルを変更するたびに、6つのGDIオブジェクトのリークが発生することがわかりました。 Chromeリモートデスクトップでの作業中に画面の目的の部分にマウスを移動すると、1分あたり数百、1時間あたり数千のGDIオブジェクトがリークする可能性があります。

この問題の解決に何の進展もない1か月後、それは突然、除去不可能な問題から単純な修正に変わりました。すぐに修正案を書き、それから同僚の1人（私はこのバグに取り組みませんでした）が実際の修正を作成しました。6月10日11:16にダウンロードされ、13：00にリリースされました。いくつかのマージの後、バグは消えました。

それで全部です？


私たちはバグを修正しましたが、それは素晴らしいことですが、そのようなバグが二度と起こらないことがはるかに重要です。明らかに、リソース管理にC ++（RAII）オブジェクトを使用することは適切ですが、この場合、バグはWebCursorクラスに含まれていました。

メモリリークに関しては、信頼できるシステムのセットがあります。Microsoftにはヒープスナップショットがあり、Chromiumにはユーザーバージョン用のヒーププロファイリングとリークエリミネーターがあります。テストマシンで。しかし、GDIオブジェクトのリークは注意を奪われているようです。プロセス情報ブロックには不完全な情報が含まれており、一部のGDIオブジェクトはカーネルモードでのみ表示でき、トレースを容易にするオブジェクトのメモリを割り当てたり解放したりするための単一のポイントはありません。これは、私が対処しなければならなかったGDIオブジェクトの最初のリークではありませんでした。追跡できる信頼できる方法がないため、最後のリークにはなりません。以下は、次のWindowsリリースに対する私の推奨事項です。

  • PEBをあいまいに読み取らずに（そしてカーソルを無視せずに）、すべてのタイプのGDIオブジェクトの数を取得するプロセスを簡単にする
  • 信頼できる追跡のためにGDIオブジェクトを作成および破棄するすべての操作を傍受および追跡するサポートされた方法を作成します。間接的に作成されたものを含む
  • これをすべてドキュメントに反映します

それで全部です。GDIオブジェクトは、メモリが制限されない方法で必然的に制限されるため、このような追跡の実装は難しくありません。これらの奇妙であるが避けられないGDIオブジェクトを使用する方が安全であるとしたらすばらしいでしょう。ああどうぞ。
