JavaでのGPUプログラミング

JavaからGPUにアクセスすると、驚異的なパワーが明らかになります。GPUの仕組みとJavaからのアクセス方法について説明します。

GPUプログラミングは、Javaプログラマーにとって非常に高い世界です。通常のJavaタスクはGPUに適していないため、これは理解できます。ただし、GPUにはテラフロップスのパフォーマンスがあるので、GPUの機能を調べてみましょう。
このトピックをわかりやすくするために、Ironプログラミングへの没入を促進する少しの歴史とともに、GPUのアーキテクチャーを説明するのに少し時間をかけます。

GPUとCPUコンピューティングの違いを示したら、Javaの世界でGPUを使用する方法を説明します。最後に、Javaコードを記述してGPUで実行するために使用できる主なフレームワークとライブラリについて説明し、いくつかのコード例を示します。

少し背景


GPUは、1999年にNVIDIAによって最初に普及しました。これは、グラフィックデータをディスプレイに転送する前に処理するように設計された特別なプロセッサです。多くの場合、これにより一部の計算でCPUをオフロードできるため、CPUリソースを解放して、これらのアンロードされた計算を高速化できます。その結果、大きな入力をより高い出力解像度で処理および表示できるため、視覚的な表示がより魅力的になり、フレームレートがよりスムーズになります。

2D / 3D処理の本質は主に行列の操作にあり、これは分散型アプローチを使用して制御できます。画像処理の効果的なアプローチは何ですか？これに答えるために、標準のCPUアーキテクチャ（図1に示す）とGPUを比較してみましょう。

GPUでプログラムを実行する


ゲームの初期のグラフィック効果のほとんどは、GPUで実行され、CPUからのデータストリームに適用される本当にハードコードされた小さなプログラムでした。

これは、ハードコードされたアルゴリズムが不十分な場合でも、特に視覚効果が主要な魔法の方向の1つであるゲームデザインでさえ、明白でした。それに応えて、大手の売り手がGPUへのアクセスを開始し、サードパーティの開発者がそれらをプログラミングできるようになりました。

典型的なアプローチは、シェーダーと呼ばれる小さなプログラムを特別な言語（通常はCの亜種）で記述し、目的のアーキテクチャー用の特別なコンパイラーを使用してコンパイルすることでした。シェーダーは光と影の効果を制御するためによく使用されるため、シェーダーという用語が選択されましたが、これは、他の特殊効果を制御できるという意味ではありません。

各GPUベンダーは、独自のプログラミング言語と、自社のアーキテクチャ用のシェーダーを作成するためのインフラストラクチャを備えていました。このアプローチでは、多くのプラットフォームが作成されています。

主なものは次のとおりです。

  • DirectCompute：DirectX 10以降、Direct3Dの一部であるMicrosoftのプライベートシェーダー言語/ API。
  • AMD FireStream：AMDが時代遅れのプライベートATI / Radeonテクノロジー。
  • OpenACC：マルチベンダーコンソーシアム、並列コンピューティングソリューション
  • ++ AMP: Microsoft C++
  • CUDA: Nvidia,
  • OpenL: , Apple, Khronos Group

ほとんどの場合、GPUの操作は低レベルのプログラミングです。これを開発者が理解しやすくするために、コーディングのために、いくつかの抽象化が提供されました。最も有名なのは、MicrosoftのDirectXと、Khronos GroupのOpenGLです。これらは高レベルのコードを記述するためのAPIであり、GPU向けに、より意味論的にはプログラマ向けに簡素化できます。

私の知る限り、DirectXにはJavaインフラストラクチャはありませんが、OpenGLには優れたソリューションがあります。JSR 231は2002年に開始され、GPUプログラマーを対象としていますが、2008年に廃止され、OpenGL 2.0のみをサポートしています。

OpenGLのサポートは、独立したJOCLプロジェクト（OpenCLもサポートしています）で継続されており、利用者が利用できます。したがって、有名なMinecraftゲームはJOCLを使用して作成されました。

GPGPUが登場


これまでのところ、JavaとGPUには共通点はありませんでしたが、共通しているはずです。Javaは、多くのコンピューティングがあり、多くのコンピューティングパワーが必要とされる企業、データサイエンス、および金融セクターでよく使用されます。これが、汎用GPU（GPGPU）の考え方です。この道に沿ってGPUを使用するという考えは、ビデオアダプターのメーカーがプログラムフレームバッファーへのアクセスを許可し始め、開発者がコンテンツを読み取れるようになったときに始まりました。一部のハッカーは、ユニバーサルコンピューティングにGPUの全機能を使用できると判断しています。
レシピはこんな感じでした：

  1. データをラスター配列としてエンコードします。
  2. それらを処理するシェーダーを作成します。
  3. 両方をグラフィックスカードに送信します。
  4. フレームバッファーから結果を取得する
  5. ラスター配列からデータをデコードします。

これは非常に簡単な説明です。これが本番環境で機能するかどうかはわかりませんが、本当に機能します。

その後、スタンフォードインスティテュートの数多くの研究により、GPUの使用が簡素化されました。 2005年に、彼らはプログラミング言語、コンパイラ、ランタイムを含む小さなエコシステムであるBrookGPUを作りました。

ANSI CバリアントであるBrookスレッドプログラミング言語で記述されたBrookGPUコンパイル済みプログラム。サーバーコンピューティングパーツのOpenGL v1.3 +、DirectX v9 +、またはAMD Close to Metalをターゲットにでき、Microsoft WindowsおよびLinuxで動作します。デバッグのために、BrookGPUはCPU上の仮想グラフィックスカードをシミュレートすることもできます。
しかし、当時の設備が整っていたため、離陸できませんでした。 GPGPUの世界では、データをデバイスにコピーする必要があります（この場合、デバイスはGPUとGPUが配置されているデバイスを指します）。GPUがデータを計算するのを待ってから、データを制御プログラムにコピーします。これにより、多くの遅延が生じます。また、2000年代半ばには、プロジェクトが活発に開発されていたため、これらの遅延により、基本的なコンピューティングでのGPUの集中的な使用も排除されました。

しかし、多くの企業がこのテクノロジーの将来を見てきました。ビデオアダプターのいくつかの開発者がGPGPUに独自のテクノロジーを提供し始め、他の形成された提携は、大量のハードウェアで機能するそれほど基本的でなく多目的なプログラミングモデルを提供しました。

これですべてを説明したので、最も成功している2つのGPUコンピューティングテクノロジー（OpenCLとCUDA）を確認してみましょう。Javaがどのように機能するかについてもご覧ください。

OpenCLとJava


他のインフラストラクチャパッケージと同様に、OpenCLはCで基本的な実装を提供します。これは、Java Native Interface（JNI）またはJava Native Access（JNA）を使用して技術的に利用可能ですが、このアプローチはほとんどの開発者にとって難しすぎます。

幸い、この作業はすでにJOCL、JogAmp、JavaCLなどのいくつかのライブラリによって行われています。残念ながら、JavaCLは死んだプロジェクトになっています。しかし、JOCLプロジェクトは生きており、非常に適応しています。次の例で使用します。

しかし、最初にOpenCLとは何かを説明する必要があります。 OpenCLは、GPUやCPUだけでなく、DSPプロセッサやFPGAでさえ、あらゆる種類のデバイスのプログラミングに適した非常に基本的なモデルを提供することを先に述べました。

最も単純な例を見てみましょう。折りたたみベクトルはおそらく最も明るく単純な例です。加算用と結果用の2つの数値配列があります。図1に示すように、最初の配列の要素と2番目の配列の要素を取得し、その合計を結果の配列に入れます。 5.


図5. 2つの配列の要素を追加し、結果の配列に合計を格納する

ご覧のとおり、操作は非常に一貫性があり、それにもかかわらず分散されています。各追加操作を異なるコアGPUにプッシュできます。つまり、Nvidia 1080のように2048コアがある場合、2048の追加操作を同時に実行できます。つまり、ここでは、コンピュータパワーの潜在的なテラフロップスがあなたを待っています。 1000万の配列のこのコードは、JOCL Webサイトから取得されます。

public class ArrayGPU {
    /**
     * The source code of the OpenCL program 
     */
    private static String programSource =
        "__kernel void "+
        "sampleKernel(__global const float *a,"+
        "             __global const float *b,"+
        "             __global float *c)"+
        "{"+
        "    int gid = get_global_id(0);"+
        "    c[gid] = a[gid] + b[gid];"+
        "}";
    
    public static void main(String args[])
    {
        int n = 10_000_000;
        float srcArrayA[] = new float[n];
        float srcArrayB[] = new float[n];
        float dstArray[] = new float[n];
        for (int i=0; i<n; i++)
        {
            srcArrayA[i] = i;
            srcArrayB[i] = i;
        }
        Pointer srcA = Pointer.to(srcArrayA);
        Pointer srcB = Pointer.to(srcArrayB);
        Pointer dst = Pointer.to(dstArray);


        // The platform, device type and device number
        // that will be used
        final int platformIndex = 0;
        final long deviceType = CL.CL_DEVICE_TYPE_ALL;
        final int deviceIndex = 0;

        // Enable exceptions and subsequently omit error checks in this sample
        CL.setExceptionsEnabled(true);

        // Obtain the number of platforms
        int numPlatformsArray[] = new int[1];
        CL.clGetPlatformIDs(0, null, numPlatformsArray);
        int numPlatforms = numPlatformsArray[0];

        // Obtain a platform ID
        cl_platform_id platforms[] = new cl_platform_id[numPlatforms];
        CL.clGetPlatformIDs(platforms.length, platforms, null);
        cl_platform_id platform = platforms[platformIndex];

        // Initialize the context properties
        cl_context_properties contextProperties = new cl_context_properties();
        contextProperties.addProperty(CL.CL_CONTEXT_PLATFORM, platform);
        
        // Obtain the number of devices for the platform
        int numDevicesArray[] = new int[1];
        CL.clGetDeviceIDs(platform, deviceType, 0, null, numDevicesArray);
        int numDevices = numDevicesArray[0];
        
        // Obtain a device ID 
        cl_device_id devices[] = new cl_device_id[numDevices];
        CL.clGetDeviceIDs(platform, deviceType, numDevices, devices, null);
        cl_device_id device = devices[deviceIndex];

        // Create a context for the selected device
        cl_context context = CL.clCreateContext(
            contextProperties, 1, new cl_device_id[]{device}, 
            null, null, null);
        
        // Create a command-queue for the selected device
        cl_command_queue commandQueue = 
            CL.clCreateCommandQueue(context, device, 0, null);

        // Allocate the memory objects for the input and output data
        cl_mem memObjects[] = new cl_mem[3];
        memObjects[0] = CL.clCreateBuffer(context,
            CL.CL_MEM_READ_ONLY | CL.CL_MEM_COPY_HOST_PTR,
            Sizeof.cl_float * n, srcA, null);
        memObjects[1] = CL.clCreateBuffer(context,
            CL.CL_MEM_READ_ONLY | CL.CL_MEM_COPY_HOST_PTR,
            Sizeof.cl_float * n, srcB, null);
        memObjects[2] = CL.clCreateBuffer(context,
            CL.CL_MEM_READ_WRITE,
            Sizeof.cl_float * n, null, null);
        
        // Create the program from the source code
        cl_program program = CL.clCreateProgramWithSource(context,
            1, new String[]{ programSource }, null, null);
        
        // Build the program
        CL.clBuildProgram(program, 0, null, null, null, null);
        
        // Create the kernel
        cl_kernel kernel = CL.clCreateKernel(program, "sampleKernel", null);
        
        // Set the arguments for the kernel
        CL.clSetKernelArg(kernel, 0,
            Sizeof.cl_mem, Pointer.to(memObjects[0]));
        CL.clSetKernelArg(kernel, 1,
            Sizeof.cl_mem, Pointer.to(memObjects[1]));
        CL.clSetKernelArg(kernel, 2,
            Sizeof.cl_mem, Pointer.to(memObjects[2]));
        
        // Set the work-item dimensions
        long global_work_size[] = new long[]{n};
        long local_work_size[] = new long[]{1};
        
        // Execute the kernel
        CL.clEnqueueNDRangeKernel(commandQueue, kernel, 1, null,
            global_work_size, local_work_size, 0, null, null);
        
        // Read the output data
        CL.clEnqueueReadBuffer(commandQueue, memObjects[2], CL.CL_TRUE, 0,
            n * Sizeof.cl_float, dst, 0, null, null);
        
        // Release kernel, program, and memory objects
        CL.clReleaseMemObject(memObjects[0]);
        CL.clReleaseMemObject(memObjects[1]);
        CL.clReleaseMemObject(memObjects[2]);
        CL.clReleaseKernel(kernel);
        CL.clReleaseProgram(program);
        CL.clReleaseCommandQueue(commandQueue);
        CL.clReleaseContext(context);

    }

    private static String getString(cl_device_id device, int paramName) {
        // Obtain the length of the string that will be queried
        long size[] = new long[1];
        CL.clGetDeviceInfo(device, paramName, 0, null, size);

        // Create a buffer of the appropriate size and fill it with the info
        byte buffer[] = new byte[(int)size[0]];
        CL.clGetDeviceInfo(device, paramName, buffer.length, Pointer.to(buffer), null);

        // Create a string from the buffer (excluding the trailing \0 byte)
        return new String(buffer, 0, buffer.length-1);
    }
}

コードはドキュメント化されますが、少しウォークスルーしてみましょう。ご覧のとおり、コードはCのコードと非常によく似ています。JOCLはOpenCLにすぎないため、これは正常です。最初に、ここにコードがあり、このコードは最も重要な部分です。コードはOpenCLを使用してコンパイルされ、ビデオカードに送信されて実行されます。このコードはカーネルと呼ばれます。この用語をOCカーネルと混同しないでください。これはデバイスコードです。このコードはCのサブセットで記述されてい

ます。カーネルがデバイスをインストールして構成するJavaコードを受け取った後、データを分割し、結果のデータに適切なメモリバッファーを作成します。

要約すると、ここに「ホストコード」があります。これは通常、言語バインディング（この場合はJava）であり、「デバイスコード」です。ホストがデバイスを制御するため、ホストで何が機能し、デバイスで何が機能するかを常に強調表示します。

上記のコードは、「Hello World！」に相当するGPUを示しているはずです。ご覧のとおり、そのほとんどは巨大です。

SIMD機能を忘れないでください。デバイスがSIMD拡張機能をサポートしている場合は、算術コードを高速化できます。例として、カーネル行列乗算コードを見てみましょう。このコードは、アプリケーションの単純なJava行にあります。

__kernel void MatrixMul_kernel_basic(int dim,
                  __global float *A,
                  __global float *B,
                  __global float *C){

    int iCol = get_global_id(0);
    int iRow = get_global_id(1);
    float result = 0.0;
    for(int i=0; i< dim; ++i)
    {
          result +=
          A[iRow*dim + i]*B[i*dim + iCol];
    }
    C[iRow*dim + iCol] = result;
}

技術的には、このコードは、OpenCLフレームワークによってインストールされたデータの一部を処理し、準備の部分で呼び出した命令を使用します。

ビデオカードがSIMD命令をサポートし、4つの浮動小数点数のベクトルを処理できる場合、小さな最適化で前のコードを次のように変えることができます。

#define VECTOR_SIZE 4    
__kernel void MatrixMul_kernel_basic_vector4(
    size_t dim, // dimension is in single floats
    const float4 *A,
    const float4 *B,
    float4 *C)
{
    size_t globalIdx = get_global_id(0);
    size_t globalIdy = get_global_id(1);
    float4 resultVec = (float4){ 0, 0, 0, 0 };
    size_t dimVec = dim / 4;
    for(size_t i = 0; i < dimVec; ++i) {
        float4 Avector = A[dimVec * globalIdy + i];
        float4 Bvector[4];
        Bvector[0] = B[dimVec * (i * 4 + 0) + globalIdx];
        Bvector[1] = B[dimVec * (i * 4 + 1) + globalIdx];
        Bvector[2] = B[dimVec * (i * 4 + 2) + globalIdx];
        Bvector[3] = B[dimVec * (i * 4 + 3) + globalIdx];
        resultVec += Avector[0] * Bvector[0];
        resultVec += Avector[1] * Bvector[1];
        resultVec += Avector[2] * Bvector[2];
        resultVec += Avector[3] * Bvector[3];
    }

    C[dimVec * globalIdy + globalIdx] = resultVec;
}

このコードを使用すると、パフォーマンスを2倍にできます。

CUDAとJava


CUDAは、このプログラミングの問題に対するNvidiaのソリューションです。CUDAは、行列、ヒストグラム、さらにはディープニューラルネットワークなど、標準のGPUオペレーション用にすぐに使用できる多くのライブラリを提供します。ライブラリのリストは、既製のソリューションの束とともにすでに現れています。これはすべてJCudaプロジェクトからのものです。

  • JCublas：マトリックスのすべて
  • JCufft：高速フーリエ変換
  • JCurand：乱数のすべて
  • JCusparse：まれな行列
  • JCusolver：数値の因数分解
  • JNvgraph：グラフのすべて
  • JCudpp：プリミティブパラレルデータといくつかのソートアルゴリズムのCUDAライブラリ
  • JNpp：GPU画像処理
  • JCudnn：ディープニューラルネットワークライブラリ

乱数を生成するJCurandの利用を検討しています。別の特別なカーネル言語がなくても、Javaコードからこれを使用できます。例えば：

GPUを使用して、非常に強力な数学に基づいて、非常に高品質の多数の乱数を作成します。

JCudaでは、汎用CUDAコードを記述して、クラスパスでJARファイルを呼び出すことにより、Javaからそれを呼び出すこともできます。優れた例については、JCudaのドキュメントを参照してください。

低レベルのコードを超えて


見た目はすべて素晴らしいですが、コードが多すぎ、インストールが多すぎ、さまざまな言語が多すぎてすべてを実行できません。 GPUを少なくとも部分的に使用する方法はありますか？

このすべてのOpenCL、CUDA、およびその他の不要なことについて考えたくない場合はどうしますか？ Javaでプログラミングするだけで、明らかでないすべてのことを考えたくない場合はどうでしょうか。 Aparapiプロジェクトが役立ちます。 Aparapiは「並列API」に基づいています。これは、OpenCLを内部で使用するHibernate for GPUプログラミングの一部と考えています。ベクトル加算の例を見てみましょう。

public static void main(String[] _args) {
    final int size = 512;
    final float[] a = new float[size];
    final float[] b = new float[size];

    /* fill the arrays with random values */
    for (int i = 0; i < size; i++){
        a[i] = (float) (Math.random() * 100);
        b[i] = (float) (Math.random() * 100);
    }
    final float[] sum = new float[size];

    Kernel kernel = new Kernel(){
        @Override public void run() {
I           int gid = getGlobalId();
            sum[gid] = a[gid] + b[gid];
        }
    };

    kernel.execute(Range.create(size));
    for(int i = 0; i < size; i++) {
        System.out.printf("%6.2f + %6.2f = %8.2f\n", a[i], b[i], sum[i])
    }
    kernel.dispose();
}

これは純粋なJavaコード（Aparapiのドキュメントから抜粋）です。また、あちこちで、特定の用語KernelおよびgetGlobalIdを確認できます。GPUのプログラミング方法を理解する必要がありますが、GPGPUアプローチをよりJavaに似た方法で使用できます。さらに、Aparapiは、OpenGLコンテキストをOpenCLレイヤーに使用する簡単な方法を提供します。これにより、データをグラフィックカードに完全に残すことができ、メモリの待ち時間の問題を回避できます。

多くの独立した計算を行う必要がある場合は、Aparapiをご覧ください。並列計算の使用例は数多くあります。

さらに、TornadoVMと呼ばれるプロジェクトがいくつかあります。これは、適切な計算をCPUからGPUに自動的に転送するので、箱から出してすぐに最適化を行うことができます。
