静かな知性。潜在的なWEB脆弱性を特定する方法

注意


このガイドは、情報セキュリティの最低限の知識をまだ持っている初心者を対象としています。時間が多く、何か生産的なことをしたい若者を励ますようにしています。

前書き


すでにご存じのとおり、Bug Bountyプログラムの効果を高めるために企業と連携する多くの仲介サイトがあります。

  • hackerone.com

使い方 ？


  • 脆弱性を探す
  • 脆弱性を悪用するために取った手順を書きます
  • この技術レポートを破棄します。サポート。
  • コインをもらいます。

これには何が必要ですか？


VPS、できればDebian（他のものも使用できます）+良好なインターネット接続（alexhost.comから取得、VPS 1.5 GB RAM、1コア、10 GB SSD、年間わずか11ユーロ。）

  • sublist3r
  • httprobe
  • ウェブスクリーンショット
  • ヘッダー応答
  • 相対URL抽出
  • jsfilesエクストラクター

プリファレンスの設定


ツールのインストールを容易にするために小さなスクリプトを書きました。

root@debian-s-1vcpu-2gb-ams3-01:~# git clone https://github.com/airwawekz/xrec.git
root@debian-s-1vcpu-2gb-ams3-01:~# cd xrec
root@debian-s-1vcpu-2gb-ams3-01:~# chmod +x xrec-install.sh
root@debian-s-1vcpu-2gb-ams3-01:~# ./xrec-install.sh

識別プロセス


参加したいプログラムが、メインドメインを含むすべてのサブドメインの脆弱性を検索することを目的としているとします。通常、プライマリドメインはそのサブドメインよりも安全です。つまり、サブドメインに重点を置く必要があります。

例：* .example.com

最初のステップ：

example.comに属するすべてのサブドメインを特定します。これにはSublist3rを使用します。 最良の結果を得るには、スクリプト構成にAPIを追加することをお勧めします。 （例：Virustotal） スキャンが完了するのを待っています。最後に、スキャンプロセスの結果が保存される「domains.txt」ファイルを作成します。2番目のステップ： どのドメインが有効かを確認します。このため、httprobeを使用します。



root@debian-s-1vcpu-2gb-ams3-01:~# cd /root/xrec/Sublist3r/
root@debian-s-1vcpu-2gb-ams3-01:~# python sublist3r.py -d example.com -o domains.txt











root@debian-s-1vcpu-2gb-ams3-01:~#   cat domains.txt | httprobe | tee -a alive.txt

このプロセスが完了したら、有効なサブドメインを含む「alive.txt」というファイルを作成します。



3番目のステップ：

アクティブなサイトごとにスクリーンショットを撮ります。有効なドメインのリストが多数ある場合、このステップにより、潜在的な脆弱なサブドメインをすばやく特定できます。

root@debian-s-1vcpu-2gb-ams3-01:~# python  /root/xrec/webscreenshot/webscreenshot.py -r chromium -i alive.txt -o /root/xrec/webscreenshot/screens/



結果は、フォルダー/ root / xrec / webscreenshot / screen /に* example.com.pngとして保存されます。

このプロセスが完了したら、各スクリーンショットを個別に確認することをお勧めします。私はしばしば、脆弱なcmsを持つ忘れられたサブドメインを見つけます。

4番目のステップ：

ヘッダーの応答を確認します。このステップは非常に重要です。ヘッダーを介して使用できる多くの脆弱性があります。

例：Yahooリモートコード実行

root@debian-s-1vcpu-2gb-ams3-01:~# cd /root/xrec/ | sudo chmod +x response.sh
root@debian-s-1vcpu-2gb-ams3-01:~/xrec# ./response.sh /root/xrec/Sublist3r/alive.txt



このプロセスの最後に、「headers」と「responsebody」という2つのフォルダーが作成され、検証の結果が保存されます。

5番目のステップ：

最後のステップは、JSファイルからエンドポイントを抽出することです。これを行うには、jsfiles抽出プログラムを使用します。

root@debian-s-1vcpu-2gb-ams3-01:~# cd /root/xrec/ | chmod +x jsfiles.sh
root@debian-s-1vcpu-2gb-ams3-01:~# ./jsfiles.sh

このプロセスが完了すると、すべてのファイルがwww.example.comのようになる「scripts」というフォルダーが作成されます。



現時点では、十分な情報を収集しており、テストする必要があります。

例：

終了リンクがある場合、「details.php？Id = 3」または「details / id / 3」の場合、最初のステップは、XSS / SQLiなどの脆弱性をテストすることです。

別の例、エンドリンクがある場合：= redirectUrl = / example / photoオープンリダイレクトを試すことができます。

/pingcheck.php?s=google.comという形式のエンドポイントがある場合、/ viewpage.php？s = google.comのようなものがあれば、RCEまたはSSRFのテストを実行できます。﻿﻿
