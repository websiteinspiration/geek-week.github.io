MacBook Pro 2018 T2をArchLinux（デュアルブート）と連携させる

新しいT2チップが原因で、タッチバー付きの新しい2018 MacBookにLinuxをインストールすることが不可能であるという事実について多くの誇大宣伝がありました。時が経ち、2019年末までに、サードパーティの開発者は、T2チップとやり取りするための多数のドライバーとカーネルパッチを実装しました。MacBookモデル2018以降のメインドライバーは、VHCIの動作（タッチ/キーボードなどの操作）とサウンドの機能を実装します。mbp2018-bridge-drv

プロジェクトは、3つの主要コンポーネントに分かれています。

  • BCE（バッファコピーエンジン）-T2との通信のメインチャネルを確立します。VHCIとオーディオには、このコンポーネントが必要です。
  • VHCIはUSB仮想ホストコントローラーです。キーボード、マウス、およびその他のシステムコンポーネントがこのコンポーネントによって提供されます（他のドライバーはこのホストコントローラーを使用してより多くの機能を提供します。
  • オーディオ-T2オーディオインターフェイス用のドライバー。現在、MacBookの内蔵スピーカーからのオーディオ出力のみがサポートされています。

2番目のプロジェクトはmacbook12-spi-driverと呼ばれ、キーボードの入力ドライバー、トラックパッドSPI、MacBook Proのタッチバー（2016年後半以降）を操作する機能を実装しています。バージョン5.3以降、キーボード/トラックパッドの一部のドライバーがカーネルに含まれるようになりました。

カーネルパッチを使用して、wi-fi、タッチパッドなどのデバイスのサポートも実装されました。現在、カーネルのバージョンは5.3.5-1です。

現時点で機能するもの


  1. NVMe
  2. キーボード
  3. USB-C（Thunderboltはテストされていません。モジュールが自動的にロードされると、システムがしっかりと中断されます）
  4. タッチバー（Fnキー、バックライト、ESCなどをオンにする機能付き）
  5. サウンド（内蔵スピーカーのみ）
  6. Wi-Fiモジュール（brcmfmac経由、iw経由のみ）
  7. USB-C経由のDisplayPort
  8. センサー
  9. 一時停止/再開（一部）
  10. など

このチュートリアルは、macbookpro15.1およびmacbookpro15.2に適用されます。記事はここから英語でgithubから取られました。この記事のすべてが機能したわけではないため、自分で解決策を探す必要がありました。

インストールする必要があるもの


  • USB-C USBドックアダプター（テザリングモードでマウス、キーボード、USBモデム、または電話を接続するための少なくとも3つのUSB入力）。これはインストールの初期段階でのみ必要です。
  • USBキーボード
  • USB / USB-Cフラッシュドライブ、少なくとも4GB

  2. 画像を作成し、自分はarchliveを通じて（あなたはarcha分布を持つシステムが必要です）

_pacman ()
{
    _msg_info "Installing packages to '${work_dir}/airootfs/'..."

4.最初の起動


挿入したフラッシュドライブとキーボードで再起動します。アップルが表示されたらオプションを押し、EFI BOOTを選択します。

次に、「e」キーを押して、コマンドラインの最後にmodule_blacklist = thunderboltと入力する必要があります。これを行わないと、システムが起動せず、Thunderbolt ICMエラーがクラッシュします。

fdisk / cfdiskを使用して、セクション（私はnvme0n1p4を持っています）を見つけ、archをフォーマットしてインストールします。公式の手順またはサードパーティを使用できます。

ブートパーティションは作成せず、/ dev / nvme0n1p1にブートローダーを書き込みます。/mnt
の環境が完全に形成された後、arch-chrootに移動する前に、次のように書き込みます。

mount /dev/nvme0n1p1 /mnt/boot
arch-chroot /mnt /bin/bash

/etc/pacman.confに追加：


[mbp]
Server = https://packages.aunali1.com/archlinux/$repo/$arch

カーネルをインストールします。


sudo pacman -S linux-mbp linux-mbp-headers
sudo mkinitcpio -p linux-mbp

/etc/modprobe.d/blacklist.confにthunderboltとapplesmcを配置します

blacklist thunderbolt
blacklist applesmc

キーボード、タッチバーなど


yayをインストールします。


sudo pacman -S git gcc make fakeroot binutils
git clone https://aur.archlinux.org/yay.git
cd yay
makepkg -si

タッチバー用のモジュールをインストールします。


git clone --branch mbp15 https://github.com/roadrunner2/macbook12-spi-driver.git
cd macbook12-spi-driver
make install

モジュールをオートロードに追加します：/etc/modules-load.d/apple.conf

industrialio_triggered_buffer
apple-ibridge
apple-ib-tb
apple-ib-als

キーボードのカーネルモジュールをインストールします。リポジトリanuali1には既製のパッケージがあり、apple-bce-dkms-gitと呼ばれています。それをインストールするには、コンソールに書きます：

pacman -S apple-bce-dkms-git

この場合、カーネルモジュールはapple-bceと呼ばれます。自己組織化の場合、それはbceと呼ばれます。したがって、mkinicpio.confファイルのMODULESセクションにモジュールを登録する場合は、インストールしたモジュールを忘れないでください。

手動組立：


git clone https://github.com/MCMrARM/mbp2018-bridge-drv.git
cd mbp2018-bridge-drv
make
cp bce.ko /usr/lib/modules/extramodules-mbp/bce.ko

bceまたはapple-bceモジュールをオートロードに追加します：/etc/modules-load.d/bce.conf

bce

デフォルトでFnボタンを使用する場合は、/ etc / modprobe.d / apple-tb.confファイルに次のように記述します。

options apple-ib-tb fnmode=2

カーネルとinitramfsを更新します。 


mkinitcpio -p linux-mbp

iwdをインストールします。

sudo pacman -S networkmanager iwd

5.ブートローダー


chroot内にすべてのメインパッケージがインストールされたら、ブートローダーのインストールに進むことができます。

私はどうにかグラブを働かせることができませんでした。外付けのUSBドライブから、GRUBは起動しますが、nvmeにそれを登録しようとすると
grub-install --target = x86_64-efi --efi-directory = / boot --bootloader-id = grub
システムがカーネルパニックにクラッシュし、再起動後にオプションを介して新しい項目が表示されませんでした。私はこの問題のわかりやすい解決策を見つけられなかったため、systemd-bootを使用してブートを実装することにしました。

  1. 発売 

    bootctl --path=/boot install

    カーネルパニックに飛びます。MacBookの電源を切り、再度電源を入れ、オプションをクリックします（キーボードでUSB-Cハブの電源を切らないでください）。
  2. 外部デバイスに加えて新しいEFI BOOTレコードが表示されていることを確認します
  3. 最初のインストール時と同様に、外部USBドライブからのダウンロードを選択します（module_blacklist = thunderboltと書くことを忘れないでください）
  4. ディスクをマウントし、arch-chrootを介して環境に入ります


mount /dev/nvme0n1p4 /mnt
mount /dev/nvme0n1p1 /mnt/boot
arch-chroot /mnt

システムが完全にロードされるまでキーボードが機能する必要がある場合（これは、luks / dm-crypt暗号化が使用されている場合に必要です）、MODULESセクションの/etc/mkinicpio.confファイルに書き込みます。

MODULES=(ext4 applespi intel_lpss_pci spi_pxa2xx_platform bce)

カーネルとinitramfsを更新します。 


mkinicpio -p linux-mbp

systemd-bootを構成する


/boot/loader/loader.confファイルを編集し、その中のすべてを削除して、以下を追加します。

default arch
timeout 5
editor 1

/ boot / loader / entriesフォルダーに移動し、arch.confファイルを作成して、次のように書き込みます。

title arch
linux /vmlinuz-linux-mbp
initrd /initramfs-linux-mbp.img
options root=/dev/<b>nvme0n1p4</b> rw pcie_ports=compat

luksとlvmを使用した場合、 

options cryptdevice=/dev/<b>nvme0n1p4</b>:luks root=/dev/mapper/vz0-root rw pcie_ports=compat

MacOSで再起動します。

6. Wi-fiセットアップ


結局のところ、MacOSはwi-fiアダプターのファームウェアファイルを/ usr / share / firmware / wifiフォルダーに保存しており、ブロブの形式でそこから取得して、brcmfmacカーネルモジュールにフィードできます。アダプターが使用するファイルを正確に見つけるには、MacOSでターミナルを開いて次のように記述します。

ioreg -l | grep C-4364

長いリストを取得します。RequestedFilesセクションのファイルのみが必要です。

"RequestedFiles" = ({"Firmware"="<b>C-4364__s-B2/maui.trx</b>","TxCap"="C-4364__s-B2/maui-X3.txcb","Regulatory"="C-4364__s-B2/<b>maui-X3.clmb</b>","NVRAM"="C-4364__s-B2/<b>P-maui-X3_M-HRPN_V-m__m-7.7.txt</b>"})

あなたの場合、ファイル名は異なるかもしれません。それらを/ usr / share / firmware / wifiフォルダーからUSBフラッシュドライブにコピーし、名前を次のビューに変更します。

    maui.trx -> brcmfmac4364-pcie.bin
    maui-X3.clmb -> brcmfmac4364-pcie.clm_blob
    P-maui-X3_M-HRPN_V-m__m-7.7.txt -> brcmfmac4364-pcie.Apple Inc.-<b>MacBookPro15,2.txt</b>

この場合、最後のテキストファイルにはモデル名が含まれています。モデルがmacbookpro15,2でない場合は、Macbookモデルに従ってこのファイルの名前を変更する必要があります。

Archで再起動します。

USBフラッシュドライブから/ lib / firmware / brcm /フォルダーにファイルをコピーします


sudo cp brcmfmac4364-pcie.bin /lib/firmware/brcm/
sudo cp brcmfmac4364-pcie.clm_blob /lib/firmware/brcm/
sudo cp 'brcmfmac4364-pcie.Apple Inc.-<b>MacBookPro15,2.txt' /lib/firmware/brcm/

モジュールの状態を確認します。


rmmod brcmfmac
modprobe brcmfmac

ネットワークインターフェースがifconfig / ipを介して表示されることを確認します。iwctl
を介してwifi を構成します

。netctl、nmcliなどを使用 インターフェースは機能せず、iwdを介してのみ機能します。

NetworkManagerにiwdを使用させる。これを行うには、ファイル/etc/NetworkManager/NetworkManager.confを作成して次のように記述します。

[device]
wifi.backend=iwd

NetworkManagerサービスの開始


sudo systemctl start NetworkManager.service
sudo systemctl enable NetworkManager.service

8.一時停止/再開


現時点では、2019年10月16日、サウンドまたは一時停止/再開のいずれかを選択する必要があります。bceモジュールの作成者が機能を完了するのを待っています。

susped / resumeをサポートするモジュールをビルドするには、次のことを行う必要があります。


git clone https://github.com/MCMrARM/mbp2018-bridge-drv.git
cd mbp2018-bridge-drv
git checkout suspend
make
cp bce.ko /usr/lib/modules/extramodules-mbp/bce.ko
modprobe bce

anuali1リポジトリから既製のapple-bceモジュールをインストールした場合は、まずそれを削除してから、サスペンドモードをサポートするbceモジュールを収集してインストールする必要があります。

また、applesmcモジュールをブラックリストに追加する必要があります（これをまだ行っていない場合）。pcie_ports= compatパラメーターが/boot/loader/entries/arch.confの最後のオプション行に追加されていることを確認してください。

現在、サスペンドモードに入るとタッチバードライバーがクラッシュし、thunderboltドライバーはシステムを30秒以上サスペンドし、再開すると数分間サスペンドすることがあります。これは問題のあるモジュールを自動的にダウンロードすることで修正できます。

スクリプト/lib/systemd/system-sleep/rmmod.shを作成します。

#!/bin/sh
if [ "${1}" == "pre" ]; then
        rmmod thunderbolt
        rmmod apple_ib_tb
elif [ "${1}" == "post" ]; then
        modprobe apple_ib_tb
        modprobe thunderbolt
fi

実行可能にします。

sudo chmod +x /lib/systemd/system-sleep/rmmod.sh

それは今のところすべてです。その結果、システムは完全に機能しますが、サスペンド/レジュームのニュアンスは一部異なります。クラッシュやカーネルパニックは数日の稼働時間の間観察されていません。近い将来、bceモジュールの作者がそれを完成させて、サスペンド/レジュームとサウンドを完全にサポートしてくれることを願っています。
