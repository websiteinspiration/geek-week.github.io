サーバーレステクノロジーに関する誤解


サーバーレスと非サーバーデータ処理（サービスとしての機能、FaaS）はほとんど同じものであると多くの人が信じています。そのため、違いはそれほど大きくなく、新製品を導入する価値があります。 AWS Lambdaは、サーバーレステクノロジーの全盛期のスターの1つであり、サーバーレスアーキテクチャの最も人気のある要素の1つでしたが、このアーキテクチャはFaaSよりも優れています。

サーバーレステクノロジーの基本原則は、インフラストラクチャの管理とスケーリングについて心配する必要はなく、使用した分だけを支払うことです。 AWS DynamoDB、S3、SNSまたはSQS、Graphcool、Auth0、Now、Netlify、Firebaseなど、多くのサービスがこれらの基準に適しています。一般に、サーバーレスとは​​、スケーリングのためにインフラストラクチャとその最適化を管理する必要なく、クラウドコンピューティングのすべての機能を使用することを意味します。また、インフラストラクチャレベルのセキュリティはもはや問題ではなく、セキュリティ標準への準拠の難しさと複雑さを考えると、大きな利点です。最後に、提供されているインフラストラクチャを購入して使用する必要はありません。

サーバーレス性は「心の状態」と考えることができます。これは、ソリューションの設計における特定の考え方です。インフラストラクチャのメンテナンスを必要とするアプローチは避けてください。サーバーレスアプローチでは、プロジェクトに直接影響を与え、ユーザーにメリットをもたらす問題の解決に時間を費やしています。持続可能なビジネスロジックを作成し、ユーザーインターフェイスを開発し、適応性と信頼性の高いAPIを開発します。

たとえば、無料のテキスト検索プラットフォームの管理と維持を回避できる場合、それが私たちの仕事です。アプリケーションを構築するこのアプローチは、複雑なインフラストラクチャの管理について考える必要がなくなるため、市場での製品の発売を大幅に加速できます。インフラストラクチャを管理する責任とコストを取り除き、顧客が必要とするアプリケーションとサービスの作成に集中します。パトリック・デボワはこのアプローチを「サービスフル」と呼んだ、この用語はサーバーレスコミュニティで採用されています。関数は、（ライブラリ全体またはWebアプリケーション全体をデプロイするのではなく）デプロイ可能なモジュールの形でサービスへの接続リンクと見なされます。これにより、アプリケーションの展開と変更の管理に驚異的な粒度が提供されます。この方法で関数をデプロイできない場合は、関数が実行するタスクが多すぎるため、リファクタリングする必要があることを示している可能性があります。

一部は、クラウドアプリケーションの開発におけるベンダー依存によって混乱しています。サーバーレステクノロジでも同じことが起こりますが、これは混乱の結果ではほとんどありません。私たちの経験では、AWS上でサーバーレスアプリケーションを作成し、他のAWSサービスを統合するAWS Lambdaの機能と組み合わせています。これらすべてが、サーバーレスアーキテクチャの利点の一部を形成しています。これは、結合の結果が単に項の合計以上である場合の相乗効果の良い例です。ベンダーへの依存を回避しようとすると、さらに大きな問題に遭遇する可能性があります。コンテナを操作する場合、クラウドプロバイダー間の独自の抽象化レベルを管理する方が簡単です。しかし、サーバーレスソリューションの場合、特に最初から経済効率を考慮に入れれば、努力は報われません。ベンダーがサービスを提供する方法を必ず確認してください。一部の特殊なサービスは、他のベンダーとの統合ポイントに依存しており、追加設定なしでプラグアンドプレイ接続を提供できます。リクエストをいくつかのコンテナーまたはEC2インスタンスにプロキシするよりも、ゲートウェイAPIエンドポイントからLambda呼び出しを提供する方が簡単です。 GraphcoolはAuth0を使用して簡単に構成できるため、サードパーティの認証を使用するよりも簡単です。

サーバーレスアプリケーションに適したベンダーを選択することは、アーキテクチャレベルのソリューションです。アプリケーションを作成すると、いつかサーバー管理に戻ることは期待できません。クラウドベンダーの選択は、コンテナーやデータベース、さらにはプログラミング言語の使用を選択することと同じです。

検討してください：

  • 必要なサービスとその理由。
  • クラウドプロバイダーが提供するサービスと、選択したFaaSソリューションを使用してそれらを組み合わせる方法。
  • サポートされているプログラミング言語（動的または静的型付け、コンパイルまたは解釈、ベンチマークの有無、コールドスタート時のパフォーマンス、オープンソースエコシステムとは何かなど）。
  • セキュリティ要件は何ですか（SLA、2FA、OAuth、HTTPS、SSLなど）。
  • CI / CDおよびソフトウェア開発サイクルの管理方法。
  • どのコードとしてのインフラストラクチャクラスソリューションを利用できますか？

既存のアプリケーションを拡張し、サーバーレス機能を段階的に追加すると、使用可能なオプションが多少制限される場合があります。ただし、ほとんどすべてのサーバーレステクノロジーは、（RESTまたはメッセージキューを介して）ある種のAPIを提供します。これにより、アプリケーションカーネルに関係なく、単純な統合で拡張機能を作成できます。明確なAPI、優れたドキュメント、強力なコミュニティを備えたサービスを探してください。間違いはありません。多くの場合、統合の容易さは重要な測定基準となる可能性があり、これがおそらく、2015年のLambdaのリリース以降のAWSの成功の主な理由の1つです。

サーバーレスが役立つ場合


サーバーレステクノロジは、ほぼすべての場所に適用できます。ただし、それらの利点は1つのアプリケーションだけに限定されません。現在、サーバーレステクノロジーのおかげで、クラウドエントリのしきい値は非常に低くなっています。開発者はアイデアを持っているが、クラウドインフラストラクチャを管理し、コストを最適化する方法がわからない場合は、これについて何らかのエンジニアを探す必要はありません。スタートアップがプラットフォームの作成を望んでいても、コストが手に負えなくなるのではないかと心配している場合、サーバーレスソリューションを簡単に利用できます。

コスト削減とスケーリングの容易さにより、サーバーレスソリューションは、内部システムと外部の両方に、数百万のオーディエンスを持つWebアプリケーションに等しく適用できます。口座はユーロではなくセントで測定されます。 AWS EC2の最も単純なインスタンス（t1.micro）を1か月レンタルすると、たとえそれを何もしなくても（誰がオフにするのを忘れたことがないのですか？）15ユーロかかります。比較すると、同じ期間にわたってこのようなレベルの支出を達成するには、512 MBのサイズを1秒間で約300万回実行する必要があります。また、この機能を使用しない場合は、何も支払いません。

サーバーレステクノロジーは主にイベントに依存しているため、古いシステムにサーバーレスインフラストラクチャを追加するのはかなり簡単です。たとえば、AWS S3、Lambda、Kinesisを使用して、API経由でデータを受信できる古い小売システム用の分析サービスを作成できます。

ほとんどのサーバーレスプラットフォームは、さまざまな言語をサポートしています。ほとんどの場合、これらはPython、JavaScript、C＃、Java、Goです。通常、すべての言語でライブラリの使用に制限はないため、お気に入りのオープンソースライブラリを使用できます。ただし、依存関係を悪用しないことをお勧めします。これにより、機能が最適に実行され、サーバーレスアプリケーションのスケーラビリティのメリットが失われないようになります。コンテナーにロードする必要のあるパッケージが多いほど、コールドスタートにかかる時間が長くなります。

コールドスタートとは、コンテナー、ランタイム、およびエラーハンドラーを使用する前にそれらを初期化する必要がある場合です。このため、関数の実行の遅延は3秒に達する可能性があり、これはせっかちなユーザーには最適なオプションではありません。ただし、コールドスタートは、数分のダウンタイム後の最初の使用時に発生します。そのため、多くの人々は、これをマイナーな不便さであると考えています。これは、関数を定期的にpingしてアイドル状態を維持することで回避できます。またはこの側面を無視します。

サーバーレステクノロジーが開発サイクルに与える影響


インフラストラクチャは単なる構成であるため、シェルスクリプトなどのスクリプトを使用してコードを定義およびデプロイできます。または、AWS CloudFormationのようなコードとしての構成クラスソリューションに頼ることもできます。このサービスはすべての領域の構成を提供するわけではありませんが、Lambda関数として使用する特定のリソースを定義できます。つまり、CloudFormationがあなたを失望させたところで、このギャップを埋める独自のリソース（Lambda関数）を書くことができます。このようにして、AWS環境の外部で依存関係を構成することもできます。

これらはすべて構成にすぎないため、特にCloudFormationのようなコードとしてのインフラストラクチャクラスソリューションを使用する場合は、特定の環境、リージョン、ユーザー向けにデプロイスクリプトをパラメータ化できます。たとえば、リポジトリ内の各ブランチのインフラストラクチャのコピーをデプロイして、開発中にそれらを完全にテストすることができます。これにより、開発者のコ​​ードがライブ環境で適切に機能するかどうかを理解したい場合、開発者はフィードバックを大幅に高速化できます。実際の使用料のみが支払われるため、管理者は多数の環境を展開するコストを心配する必要はありません。

DevOpsは、開発者が正しい構成を持っていることを確認するだけでよいので、心配が少なくなります。インスタンス、バランサー、セキュリティグループを管理する必要はなくなりました。したがって、特にIAM構成とクラウドリソースの最適化に関しては、インフラストラクチャを構成できることが重要であるにもかかわらず、NoOpsという用語がますます使用されています。

Epsagon、Thundra、Dashbird、IOPipeなどの非常に強力な監視および視覚ツールがあります。これにより、サーバーレスアプリケーションの現在の状態を監視し、ログとトレースを提供し、パフォーマンスメトリックとアーキテクチャのボトルネックを記録し、コスト分析と予測などを実行できます。それらは、DevOpsエンジニア、開発者、およびアーキテクトにアプリケーションの動作の包括的なアイデアを提供するだけでなく、マネージャーが1秒あたりのリソースコストとコスト予測で状況をリアルタイムで監視できるようにします。管理されたインフラストラクチャでこれを管理することは、はるかに困難です。

サーバーレスアプリケーションの設計は、ウェブサーバーのデプロイ、仮想マシンまたはコンテナの管理、パッチサーバー、オペレーティングシステム、インターネットゲートウェイなどを必要としないため、はるかに簡単です。これらすべての責任からの抽象化により、サーバーレスアーキテクチャは主なものであるソリューションに集中できます。ビジネスと顧客のニーズ。

ただし、ツールキットの方が優れている（毎日改善されている）場合もありますが、開発者はビジネスロジックの実装と、アーキテクチャ内のさまざまなサービス間でのアプリケーションの複雑さの最適な分散に集中できます。サーバーレスアプリケーションはイベントドリブンであり、クラウドプロバイダーによって抽象化されます（SQS、S3イベント、DynamoDBストリームなど）。したがって、開発者は特定のイベントに応答するためのビジネスロジックを規定するだけでよく、データベースとメッセージキューをより適切に実装する方法や、特定のハードウェアストレージのデータを使用して最適な作業を整理する方法について心配する必要はありません。

コードは、他の開発プロセスと同様に、ローカルで実行およびデバッグできます。単体テストは同じままです。アプリケーションインフラストラクチャ全体をカスタムスタック構成で展開できるため、開発者はテストのコストや高価な管理環境への影響を心配することなく、重要なフィードバックをすばやく得ることができます。

サーバーレスアプリケーションを構築するためのツールと手法


サーバーレスアプリケーションを構築する特定の方法はありません。このタスクの一連のサービスと同様に。 AWSは今日の強力なサーバーレスソリューションのリーダーですが、Google Cloud、Zeit、Firebaseに注意してください。 AWSを使用する場合、Visual Studioは優れたツールキットであるため、特にC＃を使用する場合、アプリケーションを収集するアプローチとしてサーバーレスアプリケーションモデル（SAM）を推奨できます。 SAM CLIはすべてVisual Studioと同じように実行できるため、別のIDEまたはテキストエディターに切り替えても何も失われません。もちろん、SAMは他の言語でも動作します。

他の言語で記述する場合、Serverless Frameworkは非常に強力な構成YAMLファイルを使用して何でも構成できる優れたオープンソースツールです。サーバーレスフレームワークはさまざまなクラウドサービスもサポートしているので、マルチクラウドソリューションを探している方にお勧めします。彼には、あらゆるニーズに対応するためのプラグインを多数作成した巨大なコミュニティがあります。

ローカルテストには、オープンソースツールのDocker-Lambda、Serverless Local、DynamoDB Local、LocalStackが適しています。サーバーレステクノロジーは、それらのツールと同様に、まだ開発の初期段階にあります。そのため、複雑なテストシナリオを設定するときは、汗をかく必要があります。ただし、環境内でスタックを拡張してテストするだけで、非常に安価になります。また、クラウド環境の正確なローカルコピーを作成する必要はありません。

AWS Lambdaレイヤーを使用して、デプロイされたパッケージのサイズを縮小し、読み込みを高速化します。

特定のタスクには正しいプログラミング言語を使用してください。さまざまな言語には、独自の長所と短所があります。多くのベンチマークがありますが、JavaScript、Python、C＃（.NET Core 2.1+）はAWS Lambdaのパフォーマンスの面でリーダーです。ランタイムAPIは最近AWS Lambdaに登場しました。これにより、必要な言語とランタイムを指定できるので、実験してみてください。

展開用にパッケージサイズを小さくしてください。サイズが小さいほど、読み込みが速くなります。特にライブラリの機能をいくつか使用する場合は、大きなライブラリを使用しないでください。JavaScriptでプログラミングしている場合は、Webpackなどのビルドツールを使用してビルドを最適化し、本当に必要なものだけを含めます。.NET Core 3.0には、パフォーマンスを向上させ、コールドスタートで多くのことを支援するQuickJitおよび階層化コンパイルがあります。

サーバーレス機能のイベントへの依存は、最初はビジネスロジックの調整を複雑にする可能性があります。この点で、メッセージキューとステートマシンは非常に便利です。ラムダ関数は相互に呼び出すことができますが、応答を受け取ることを期待しない場合（「ショットアンドフォーゴット」）にのみ実行します。別の関数の完了を待機するためのアカウントを取得したくない場合。メッセージキューは、ビジネスロジックの一部の分離、アプリケーションのボトルネックの管理、およびトランザクションの処理（FIFOキューを使用）に役立ちます。 AWS Lambda関数は、後で分析するために失敗したメッセージを追跡するハングメッセージキューとしてSQSキューに割り当てることができます。 AWSステップ関数（ステートマシン）は、関数チェーンの作成を必要とする複雑なプロセスを管理するのに非常に役立ちます。Lambda関数を別の関数を呼び出す代わりに、Step関数は状態遷移を調整し、関数間でデータを転送し、関数のグローバル状態を制御できます。これにより、再試行の条件、または特定のエラーが発生したときに実行する必要があることを特定できます。特定の条件下では、非常に強力なツールです。
