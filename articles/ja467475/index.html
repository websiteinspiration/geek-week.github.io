Thread.Abort（）を.NET Coreに返します。CoreCLRおよびCoreFXのバージョンをアプリケーションに提供する

.NET Frameworkから.NET Coreに移行するときに、不快な瞬間が発生する場合があります。たとえば、アプリケーションでドメインを使用している場合は、ロジックを書き換える必要があります。似たような状況Thread.Abort（） ：マイクロソフトがされるように嫌わ彼らが最初にこのメソッドが宣言されていることを、この練習（と当然のように）非推奨、その後、完全なフレームワークのそれを切り取って、今それが裏切り投げているPlatformNotSupportedExceptionを。

しかし、アプリケーションがThread.Abort（）を使用していて、本当にそれを.NET Coreに変換したい場合はどうでしょうか。何も書き直さずに？ええと、プラットフォームがこの機能をサポートしていることはよくわかっているので、喜ばしく思います。解決策はあります。CLRの独自のバージョンをアセンブルする必要があるだけです。

免責事項：これは、最小限の理論を備えた純粋に実用的な記事であり、開発者と.NET環境との間のやり取りの新しいオプションを示すことだけを目的としています。本番環境ではこれを行わないでください。しかし、本当にしたい場合...

CoreCLRを再び素晴らしいものにする


次に、Thread.Abort（）を返す2番目の部分に進みましょう。不愉快な驚きはここで私たちを待っています：スレッドの実装はCoreCLRにあります。これはCoreFXの一部ではなく、マシンに個別にプリインストールされています。まず、デモプロジェクトを作成します。

var runtimeInformation = RuntimeInformation.FrameworkDescription;
Console.WriteLine(runtimeInformation);
var thr = new Thread(() =>
{
	try
	{
		while (true)
		{
			Console.WriteLine(".");
			Thread.Sleep(500);
		}
	}
	catch (ThreadAbortException)
	{
		Console.WriteLine("Thread aborted!");
	}
});

foreach (var asm in AppDomain.CurrentDomain.GetAssemblies())
{
	Console.WriteLine(asm.Location);
}

thr.Start();
Thread.Sleep(2000);
thr.Abort();

coreclrを収縮します。ファイルを見つける

coreclr\src\System.Private.CoreLib\shared\System\Threading\Thread.cs

そしてAbort（）を

[SecuritySafeCritical]
[SecurityPermissionAttribute(SecurityAction.Demand, ControlThread = true)]
public void Abort()
{
   AbortInternal();
}

[System.Security.SecurityCritical]  // auto-generated
[ResourceExposure(ResourceScope.None)]
[MethodImplAttribute(MethodImplOptions.InternalCall)]
private extern void AbortInternal();

次に、属性とC ++実装を返す必要があります。.NET Frameworkのさまざまなオープンリポジトリから断片的に収集し、便宜上、すべての変更をプルリクエストの形式で設計しました。

注：ビルド時に、「新しい」属性のリソースに問題がありました
。これをスタブで「修正」しました。あなたは自宅の実験以外の場所にこのコードを使用したい場合はこの点に注意してください。

コードにこれらの変更を統合した後、実行がbuild.cmdからcoreclr。後の段階でのアセンブリはエラーで散らばり始めるかもしれませんが、それは恐ろしいことではありません、私たちにとっての主なことはCoreCLRがアセンブリできることです。それは以下にあります：

coreclr\bin\Product\Windows_NT.x64.Debug
