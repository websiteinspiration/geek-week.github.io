MonbMQパート3でのRabbitMQの使用

当社では、以前の記事で、私たちは、タスクキューを作成しました。メッセージ形式の1つのタスクが1人の受信者に配信されることを前提としています。この記事では別のことを行います。一度に1つのメッセージを複数の受信者に送信します。2つのプログラムで構成されるロギングシステムを作成します。1つ目はメッセージを送信し、2つ目はメッセージを受信して​​表示します。私たちのシステムでは、実行中のすべての受信者が送信者から送信されたメッセージを受信します。

取引所（交換機）


以前の記事では、キューとの間でメッセージを送受信しましたが、今度は完全なメッセージ転送モデルをRabbitMQに導入します。

以前の記事で取り上げた内容を簡単に見ていきましょう。

  • プロデューサー-メッセージを送信するアプリケーション
  • キュー-メッセージを格納するバッファ
  • コンシューマー-メッセージを受信して​​処理するアプリケーション

RabbitMQ（またはAMQP）の重要なアイデアは、送信者（プロデューサー）がメッセージをキューに直接送信しないことです。さらに、彼はそれが待ち行列にあるかどうかさえ知りません。代わりに、送信者は交換機（Exchange）にメッセージを送信します。交換器は非常に単純なもので、2つのことを行います。送信者からメッセージを受信し、キューにリダイレクトします。エクスチェンジャーにはさまざまなタイプがあります。特定のキューにメッセージを送信するもの（ダイレクトタイプ）、同じメッセージを複数のキューに一度に送信するもの（ファンアウトタイプ）、特定の指定されたリダイレクトルール（トピックタイプ）に基づいてメッセージをキューにリダイレクトするものがあります。


（RabbitMQ公式サイトからの画像）

ファンアウト型の交換器を見てみましょう。これを作成するには、次のコードを記述します。

$producer->newFanoutExchange('logs');

ファンアウトエクスチェンジャーは非常に単純なスキームに従って機能し、受信したメッセージをそれに接続されているすべてのキューに送信するだけです。

サーバーに存在するすべての交換器をリストするには、rabbitmqctlを呼び出します

sudo rabbitmqctl list_exchanges

このリストにはamqが含まれます*交換器と標準の名前のない（空のライン）交換器。まだ必要ないので注意しないでください。

前回の記事では、交換器について何も知りませんでしたが、それでもメッセージを送信することができました。 MonsterMQに明示的に3番目の引数として交換を指定しない場合はこれが起こったプロデューサー::公開（）メソッドを、ライブラリーは、名前の2番目の引数として渡されたキューにメッセージを送信RabbitMQの、交換標準の無名を使用するプロデューサー::公開（）メソッド。

これで、名前をProducer :: publish（）メソッドの3番目の引数として指定して、新しいエクスチェンジャーにメッセージを送信できます。

$producer->publish($message, '', 'logs');

次に、2つのキューを持つ2つのワーカーを作成します。これらは、それらを発表したワーカーがアクティブである限り存在します。これは次のようにして行うことができます。

//Worker 1
$producer->queue('queue-1')->setExclusive()->declare();


//Worker 2
$producer->queue('queue-2')->setExclusive()->declare();

キューをアナウンスしたワーカーがセッションを終了して切断すると、自動的に削除されます。両方のワーカーが作業を完了すると、両方のキューが削除されます。

キューを交換器に接続します


交換とキューはすでに作成されています。次に、受信したメッセージをキューに送信するように交換機に指示する必要があります。交換器とキューをバインドする必要があります。これを行うには、次のコードを記述します。

//Worker 1
$producer->queue('queue-1')->bind('logs');


//Worker 2
$producer->queue('queue-2')->bind('logs');

今後、ログエクスチェンジャーはキューに接続され、受信したメッセージをキューに転送します。

Linuxで次のコマンドを使用して、既存のリンクをすべて一覧表示できます

rabbitmqctl list_bindings

送信者スクリプトは、前のレッスンと比べてそれほど変わっていません。主な違いは、標準の交換機ではなく、以前にアナウンスされた交換機にメッセージを送信することです（ボックスからアクセス可能）。これがsend.phpコードです

try {
   $producer = \MonsterMQ\Client\Producer();

   $producer->connect('127.0.0.1', 5672);
   $producer->logIn('guest', 'guest');

   $producer->newFanoutExchange('logs');

   $message = implode(' ', array_slice($argv, 1));
   $message = empty($message) ? 'Hello world!' : $message;

   $producer->publish($message, '', 'logs');

   echo "\n Sent {$message} \n";
} catch(\Exception $e) {
   var_dump($e);
}

キューに関連付けられていない交換器にメッセージを送信すると、メッセージが失われることに言及する価値があります。しかし、受信者をまだローンチしていないため、今ではそれが適しています。

これが最初のworker-1.phpワーカーのコードです

try {
   $consumer = \MonsterMQ\Client\Consumer();

   $consumer->connect('127.0.0.1', 5672);
   $consumer->logIn('guest', 'guest');

   $producer->queue('queue-1')->setExclusive()->declare();
   $producer->queue('queue-1')->bind('logs');

   $consumer->consume('queue-1');

   $consumer->wait(function ($message, $channelNumber) use ($consumer){
      echo "\n $message \n";
   });
} catch(\Exception $e) {
   var_dump($e);
}

2番目のworker-2.phpワーカーのコードは次のとおりです

try {
   $consumer = \MonsterMQ\Client\Consumer();

   $consumer->connect('127.0.0.1', 5672);
   $consumer->logIn('guest', 'guest');

   $producer->queue('queue-2')->setExclusive()->declare();
   $producer->queue('queue-2')->bind('logs');

   $consumer->consume('queue-2');

   $consumer->wait(function ($message, $channelNumber) use ($consumer){
      echo "\n $message \n";
   });
} catch(\Exception $e) {
   var_dump($e);
}

最初のワーカーに送信されたメッセージをログファイルに保存するには、コンソールで次のコマンドを呼び出します。

php worker-1.php > logs_from_rabbit.log

両方のワーカーを開始し、ターミナルウィンドウにメッセージを表示するには、それぞれのウィンドウで次のコマンドを呼び出します。

php worker-1.php

php worker-2.php

メッセージを送信するには、次のコマンドを呼び出します。

php send.php
