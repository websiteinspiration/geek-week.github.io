Terraform、モノリポジトリ、およびコードとしてのコンプライアンス

みなさん、こんにちは。OTUSは、コース「Kubernetesに基づくインフラストラクチャプラットフォーム」で新しいグループの募集を開始しました。これに関連して、このトピックに関する興味深い資料の翻訳を準備しました。



あなたはインフラストラクチャをコードとしてterraformを使用している人の1人であり、より生産的かつ安全にそれを使用する方法を考えています。一般に、最近、これは多くを悩ませました。私たちは皆、さまざまなツール、言語を使用して構成とコードを作成し、かなりの時間を費やして読みやすく、拡張可能で、スケーラブルなものにします。


多分問題は私たち自身にありますか？

記述されたコードは、価値を生み出すか、問題を解決し、重複排除に再利用できる必要があります。通常、この種のディスカッションは「モジュールを使用しましょう」で終わります。私たちは皆テラフォームモジュールを使用していますよね？過剰なモジュール性が原因で問題のある多くの物語を書くことができましたが、これは完全に異なる物語であり、私はしません。


いいえ、私はしません。主張しないでください、いいえ...はい、多分後で。

モジュールを使用してルートモジュールをロックするときにコードにタグを付けて、モジュールコードを変更した場合でもその動作を保証することはよく知られています。このアプローチは、適切なモジュールにタグを付けて適切に使用するチームの原則になるはずです。

...しかし、依存関係はどうですか？120の異なるリポジトリに120のモジュールがあり、1つのモジュールを変更すると他の20のモジュールに影響する場合はどうでしょうか。これは、20 + 1のプルリクエストを実行する必要があることを意味しますか？レビューアの最小数が2の場合、21 x 2 = 44を意味しますレビュー。マジで！ 「1つのモジュールを変更する」ことでコマンドを麻痺させるだけで、全員がロードオブザリングのミームまたはGIFを送信し始め、残りの日は失われます。


全員に通知する1つのPR、全員を1つにまとめる1つのPR、束縛して暗闇に突入

する働く価値はありますか？レビューアの数を減らす必要がありますか？または、変更が大きな影響を与える場合は、モジュールの例外を作成し、PRは不要です。本当に？深く暗い森の中を盲目的に歩きたいですか？または、それらをすべてまとめて、退屈させて暗闇に突入しますか？

いいえ、レビューの順序を変更しないでください。 PRでの作業が正しいと思われる場合は、それに固執してください。スマートパイプラインがある場合、またはマスターにプッシュする場合は、このアプローチを使用してください。

この場合、問題は「作業方法」ではなく、「gitリポジトリの構造」です。



これは、以下の提案を最初に適用したときに感じたものと似ています。

基本に戻りましょう。Terraformモジュールを備えたリポジトリの一般的な要件は何ですか？

  1. 重大な変更がないようにタグを付ける必要があります。
  2. どのような変更でもテストできるはずです。
  3. 変更は相互レビューを経る必要があります。

それから私は以下を提案します- テラフォームモジュールにマイクロリポジトリを使用しないで ください。1つのモノリポジトリを使用します。

  • 変更/要件がある場合、リポジトリ全体にタグを付けることができます
  • 変更、PR、プッシュをテストできます。
  • 変更はレビューを通過できます。


力がある！

わかりましたが、このリポジトリの構造はどうなりますか？過去4年間、これに関連して多くの失敗があり、モジュール用の別のディレクトリが最善の解決策であるという結論に達しました。


モノリポジトリのディレクトリ構造の例。 tags_overrideの変更を確認しますか？

したがって、20の他のモジュールに影響を与えるモジュール変更は1 PRのみです！このPRに5人のレビュアーを追加した場合でも、マイクロリポジトリに比べてレビューは非常に高速になります。 Githubを使用している場合、これはさらに優れています！メンテナ/所有者がいるモジュールにはCODEOWNERSを使用できます。これらのモジュールに対する変更は、この所有者によって承認される必要があります。

すばらしいですが、mono-repositoryディレクトリにあるこのようなモジュールの使用方法は？

簡単：

module "from_mono_repo" {
 source = "git::ssh://.../<org>/<repo>.git//<my_module_dir>"
 ...
}
module "from_mono_repo_with_tags" {
  source = "git::ssh://..../<org>/<repo>.git//<mod_dir>?ref=1.2.4"
  ...
}
module "from_micro_repo" {
  source = "git::ssh://.../<org>/<mod_repo>.git"
  ...
}
module "from_micro_repo_with_tags" {
  source = "git::ssh://.../<org>/<mod_repo>.git?ref=1.2.4"
  ...
}

この種の構造の欠点は何ですか？まあ、PR /変更で「すべてのモジュール」をテストしようとすると、1.5時間のCIパイプラインを取得できます。PRで変更されたモジュールを見つける必要があります。私はそれをこのようにします：

changed_modules=$(git diff --name-only $(git rev-parse origin/master) HEAD | cut -d "/" -f1 | grep ^aws- | uniq)

別の欠点もあります。「terraform init」を実行すると、リポジトリ全体が.terraformディレクトリにロードされます。しかし、スケーラブルなAWS CodeBuildコンテナーでパイプを実行しているので、これで問題が発生したことはありません。 Jenkinsおよび永続的なJenkinsスレーブを使用する場合、この問題が発生する可能性があります。


泣かせないで。

単一リポジトリーでも、マイクロリポジトリーのすべての利点を利用できますが、おまけとして、モジュールの保守コストを削減できます。

正直なところ、このモードで作業した後、テラフォームモジュールにマイクロリポジトリを使用するという提案は犯罪と見なされます。

ユニットテストはどうですか？これは本当に必要ですか？ ...一般的に、ユニットテストとは正確にはどういう意味ですか。 AWSリソースが正しく作成されているかどうかを本当にチェックしますか？これは誰の責任ですか？terraformまたはリソースの作成を処理するAPI？おそらく、コードのネガティブテストとべき等性により重点を置く必要があります。

コードのべき等性をチェックするために、terraformはと呼ばれる優れたパラメーターを提供します-detailed-exitcode。ただ走れ：

> terraform plan -detailed-exitcode

その後、実行すればそれだけterraform applyです。少なくとも、コードがべき等であり、ランダムな文字列やその他の理由で新しいリソースを作成しないことが保証されます。

否定的なテストはどうですか？一般的に、ネガティブテストとは何ですか？実際、これは単体テストと大差ありませんが、否定的な状況に注意を払います。

たとえば、暗号化されていないパブリックS3バケットを作成することはできません。

したがって、実際には、S3バケットが実際に作成されているかどうかを確認する代わりに、一連のポリシーに基づいて、コードがリソースを作成しているかどうかを確認します。どうやってするの？ Terraform Enterpriseは、このための優れたツールであるSentinelを提供しています。

...しかし、オープンソースの選択肢もあります。現在、HCLコードの静的分析のための多くのツールがあります。これらのツールは、一般的なベストプラクティスに基づいており、不要なことを行うことはできませんが、必要なテストがない場合はどうなるか、さらに悪いことに、状況が少し異なる場合はどうでしょうか。たとえば、特定の条件に基づいて一部のS3バケットを公開できるようにしたい場合、実際にはこれらのツールのセキュリティバグになります。

これがterraform-complianceが表示される場所です。このツールを使用すると、独自のテストを記述して、企業のポリシーとして必要なものを決定できるだけでなく、セキュリティを左に移動して、セキュリティと開発を分離することもできます。かなり物議を醸すようですね。番号。ところが？


Terraform-complianceのロゴ

まず、terraform-complianceはBehavior Driven Development（BDD）を使用します。

Feature: Ensure that we have encryption everywhere.

    Scenario: Reject if an S3 bucket is not encrypted
        Given I have aws_s3_bucket defined
        Then it must contain server_side_encryption_configuration

暗号化が有効になっているかどうかを確認します。

これで十分ではない場合は、詳細を記述できます。

Feature: Ensure that we have encryption everywhere.

    Scenario: Reject if an S3 bucket is not encrypted with KMS
        Given I have aws_s3_bucket defined
        Then it must contain server_side_encryption_configuration
        And it must contain rule
        And it must contain apply_server_side_encryption_by_default
        And it must contain sse_algorithm
        And its value must match the "aws:kms" regex

さらに深く

調べて、暗号化にKMSが使用されていることを確認します。このテストのterraformコード：

resource "aws_kms_key" "mykey" {
 description             = "This key is used to encrypt bucket objects"
 deletion_window_in_days = 10
}

resource "aws_s3_bucket" "mybucket" {
 bucket = "mybucket"

 server_side_encryption_configuration {
   rule {
     apply_server_side_encryption_by_default {
       kms_master_key_id = "${aws_kms_key.mykey.arn}"
       sse_algorithm     = "aws:kms"
     }
   }
 }
}

したがって、テストは組織内の文字通りすべてによって理解されます。ここで、これらのテストの作成をセキュリティサービスまたはセキュリティについて十分な知識を持つ開発者に委任できます。このツールを使用すると、BDDファイルを別のリポジトリに保存することもできます。これは、コードの変更とコードに関連付けられたセキュリティポリシーの変更が2つの異なるエンティティである場合の責任を区別するのに役立ちます。これらは異なるライフサイクルを持つ異なるチームである場合があります。すごいでしょ？まあ、少なくとも私にとってはそうだった。
