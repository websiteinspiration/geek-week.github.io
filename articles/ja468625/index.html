サーバーレス：15％遅く、8倍高価

私は最近、CardGames.io WebサイトでAPI を試し、サーバーレスフレームワークを試すことにしました。過去数年間、それは技術の世界でホットな話題となっている、と私はprocrastinated私が今までに私の技術的なスキルを維持し、何か新しいことを試してみたかったです。したがって、私はサーバーレスを数時間研究して、そのようなAPI配置に意味があるかどうかを確認することにしました。

現在の構成


CardGames.ioはAWSで実行されます。すべてのHTMLページ、CSS、JavaScript、画像はS3に保存されます。 Elastic BeanstalkでホストされているC＃APIがあり、.NET CoreとDockerを実行しているLinuxサーバーで実行されます。最後に、S3とAPIの統計の前にCloudFront CDNを使用します。以下は、2019年8月のEC2スコアです。他にもいくつかインスタンスがありますが、APIは従来のロードバランシングでm1.small（はい、t2.smallはおそらくよりうまく機能します）で動作します。ハイライトされた部分を赤で要約すると、月額164.21ドルと悪くないことがわかります。 EC2にはいくつかのプロジェクトがあるので、これらのコストをどのように削減するのかわからないため、そこにEBSも含めました。


Elastic BeanstalkのAWSアカウントAWSで

.NET CoreをDockerにデプロイするのに数分かかるため、それぞれ1-3のインスタンスを持つ2つの環境があります。非アクティブな環境でそれを実行し、CNAMEレコードを最近配備されました。展開が遅いことが、新しいことを試してみたかった理由の1つでした。Beanstalkにnode.jsアプリケーションを備えたサーバーがいくつかあり、それらは数秒でデプロイされます。APIでも同じにしたかったのです。

サーバーレスへの移行


サーバーレスフレームワークを使用し て非常に優れた ASP.NET Web APIホスティングチュートリアルを調査したところ、既存のAPIプロジェクトに単純な構成ファイル、1つの依存関係、小さなスタートアップクラスを追加するだけでよいことがわかりました。デプロイには約20秒かかりました-Beanstalkよりもはるかに優れています。Lambda（ただし、2.2のみ）の.NET Coreの組み込みサポートのおかげで、Beanstalkでは独立制御でDockerを使用する場合にのみサポートされると思います。いずれにせよ、自動スケーリンググループや最大インスタンスなどについて考えずに満足しました。

性能試験


AWSのサーバーレスには、実際に機能をホストするLambdaと、速度制限、APIキーなどを追加できるフロントゲートウェイAPIがあります。私はBeanstalkサーバーがあるus-west-2リージョンでLambda機能をホストしました。次に、CloudFrontインスタンスを構成して、ゲームの1つから新しいサーバーレス構成にリクエストをルーティングし、もう1つを古いBeanstalk構成にルーティングします。次に、2つのURLで簡単なテストを開始しました。1つはサーバーレス、もう1つはBeanstalkです。両方のURLが同じAPIを呼び出し、データベースに同じイベントを保存しました。それぞれに対して100個のクエリを実行しました。結果は次のとおりです。


パフォーマンスの比較

複数の実行で、サーバーレス構成は 15％遅くなりました（どちらかと言えば、アイスランドから実行したため、大きなpingになります）。速度はがっかりしましたが、それでもかなり高いままでした。APIの前にあるAPI Gatewayのオーバーヘッドがいくらかあることに気づきました。たとえ使用しなくても、関数が多すぎます。悲しいことですが、重要ではありません！

価格


正直なところ、最初は価格については考えていませんでした。実際の使用料を支払うことは、インスタンスを24時間年中無休で支払うことよりもおそらく安くなると私はちょうど決めました。したがって、新しいサーバーレスインストールが数日間機能することを許可し、アカウントのチェックを開始しました。おっとっと！ Lambda + API Gatewayの請求額はすでに100ドルを超えています。最初はLambdaの設定をいじくり回して、節約するために割り当てるメモリを少なくしましたが、問題をよく見てみると、ゲートウェイに問題があることが明らかになりました。 APIゲートウェイのレートは次のとおりです。


ゲートウェイAPIの料金

当社のAPIは、1日あたり約1000万のリクエストを受け入れます。これは、毎日ゲートウェイあたり約35ドルです。さらに、Lambdaのコストは1日あたり約10ドルですが、割り当てるメモリを少なくすることで削減できます。合計すると、Elastic Beanstalkの月額$ 164に対して、1日あたり約$ 45、つまり月額$ 1350になります。では 8倍高価！私は新しいテクノロジーと迅速な導入が好きですが、月額1200ドルを追加で支払うつもりはありません。Beanstalkに戻る！

結論


おそらく、私は事前に価格に精通し、いくつかの数学的計算をしたはずです！しかしもちろん、私は実際の仕事をしなければならず、貴重な技術スキルを学ばないでしょう！状況によっては、Gateway APIとLambda APIがElastic Beanstalkよりも優れていると確信しています。それだけではありません。おそらく、APIキー、速度制限、およびその他のゲートウェイAPI関数を使用する場合、100万リクエストあたり$ 3.50を支払うことは理にかなっています。通常のロードバランサーをLambdaの前に置くだけの方が良いでしょう。私の知る限り、これは不可能です。LambdaへのhttpアクセスにはゲートウェイAPIが必要です。

しかし、Lambdaの料金を支払ったとしても、1日10ドルで164ドルではなく、1か月に300ドルになります。多くのクエリがありますが、各クエリはほとんど機能しません。基本的に、クエリごとに1つのDB呼び出しです。おそらく、より多くの計算時間を使用するより重いクエリは、100ミリ秒の計算時間を支払うLambdaに適しています。以下は、1つのリクエストのレポートです。ご覧のとおり、計算時間は3.50ミリ秒ですが、請求書は100ミリ秒に設定されています。これは弱い丸めではありません。


1つのリクエストに対するLambdaレポート

最後に、ゲートウェイ、Lambda、またはサーバーレスAPIについてはまったく批判しません。一部のワークロードについては、退屈な古いEC2やElastic Beanstalkよりもはるかに高価であることを示しています。私たちが残るものについて。また、システムを構成する方法としては、はるかに優れた、またはより効率的な方法がある可能性があります。私はAWSのエキスパートではないため、重大なエラーが発生した場合は、コメントに明記してください。
