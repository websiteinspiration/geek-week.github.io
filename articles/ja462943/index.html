4.虫のにおいがする


コンテンツ

Vampus Huntingは、1972年にGregory Yobによって発明された古典的なテキストゲームです。同年、彼はゲームの説明とソースコードを雑誌の記事で紹介されました。

ゲームの本質は、邪悪なヴァンパスの本拠地である十二面体の迷路をプレイヤーが調査することであり、ゲームログに表示されるインジケーターメッセージに基づいて、トップルームにある推測を試みます。 Vampus自体（不快なにおいがする）に加えて、ランダムな部屋にプレイヤーを運ぶコウモリ（ノイズが聞こえる）と、ゲームの終了につながるピット（貫通）があります。ゲームの目的は、プレイヤーが一度に1室から5室を飛ぶことができる5つの矢を持っているVampusを殺すことです（プレーヤー自身がショットを行うための「パワー」の種類を決定します）。したがって、プレーヤーには2つのアクションがあります。弓から撃つ、部屋に行く。結果がどうなるかは、運の分担と意識の度合いによって異なります。

一般的に、私はメカニズムが好きでした。それは単純ですが、同時にリスクの要素も伴います。Google PlayのVampusには興味深いゲームはありませんでした（当時、ラブクラフトの世界で新しいゲームがあったことを除いて、後でわかったように、同じVampusの仕組みに基づいていました。ただし、これはHabréでのゲームの作成に関する記事です）。 Vampusをベースとする決定。クラシックゲームを維持することを目標に設定しましたが、少し更新して新しい機能を追加しています。

6. Vampusの迷路の生成とそれを使った作業


内容

Vampus Labyrinthは12面体であり、20x20 の次元の行列Gとして表すことができます。頂点には0から19までの番号を付けます。行列要素が1の場合、頂点（部屋）の間に通路があり、それ以外の場合、通路はありません。

また、各部屋の隣人のインデックスを格納する20×3 行列Nを導入します。この行列はGでの作業を高速化します。


行列GとNはゲームコードに縫い込まれ、変更されません（もちろん、Nでしか作業できないため、Gを保存する必要はありませんが、ここではそのままにしておきます）。これらは、与えられた十二面体の「真の」頂点インデックスです。プレーヤーの場合、「ゲーム」インデックスは、「真の」インデックスの一種のマスクであり、次のように次元20のベクトルVに形成されます。

「true」と「ゲーム」のルームインデックス間の対応を確立するために、変換メソッドindByNmbが使用されます。

public byte indByNmb(int room) { 
    byte ind = -1;
    for (byte i = 0; i < V.length; i++) {
        if (V[i] == room) {
            ind = i;
            break;
        }
    }
    return ind;
}

public void printNearRooms(byte currentRoom) {
    byte ind = indByNmb(currentRoom);
    appendText(V[N[ind][0]], V[N[ind][1]], V[N[ind][2]]);

入力で、printNearRoomsメソッドはcurrentRoomルームの現在の「ゲーム」インデックスを取得します。

public boolean isBatsNear() {
    boolean answer = false;
    byte indP = indByNmb(P);
    for (int i = 0; i < 3; i++) {
        if ((V[N[indP][i]] == Bats[0]) || (V[N[indP][i]] == Bats[1])) {
            answer = true;
            break;
        }
    }
    return answer;
}


7.ゲームメッセージの保存とプレイヤーへの出力


コンテンツ

クラシックゲームは、プレーヤーがコンソールと対話するプロセスでした。ゲームは純粋にテキストでした。この機能を保存したかったので、次のようにすべてのゲームメッセージをブロックに分割しました。

  • 新しいゲームの最初のメッセージ。
  • プレイヤーが移動したときのメッセージ。
  • ピット付近のメッセージ。
  • Vampusを移動するときのメッセージ。
  • 穴に落ちた時のメッセージ。

テキストはXMLファイルに保存されます。各ブロックには、さまざまなゲームプレイのためにいくつかのメッセージオプションがあります。

隣接する部屋の1つに穴がある場合に表示されるメッセージブロックの例：

<string name="g_pitsNear_1">—  \n</string>
<string name="g_pitsNear_2">—    \n</string>
<string name="g_pitsNear_3">—     \n</string>
<string name="g_pitsNear_4">—  , \n</string>
<string name="g_pitsNear_5">—   \n</string>
<string-array name="g_pitsNear">
    <item>@string/g_pitsNear_1</item>
    <item>@string/g_pitsNear_2</item>
    <item>@string/g_pitsNear_3</item>
    <item>@string/g_pitsNear_4</item>
    <item>@string/g_pitsNear_5</item>
</string-array>

この構造により、Javaコードに影響を与えることなく、既存のメッセージを簡単に編集したり、新しいメッセージを追加したりできます。

たとえば、部屋をチェックするときに、前に示したisBatsNearメソッドがtrueを返した場合、目的のメッセージブロックをXMLから取得し、ランダムに1つをappendTextの引数として取得します。

if (isBatsNear()) {
    String[] g_batsNear = getResources().getStringArray(R.array.g_batsNear);
    appendText(g_batsNear[random.nextInt(g_batsNear.length)]);
}

ゲームメッセージは、TextViewオブジェクトであるコンソールに出力されます。appendTextメソッドを見てみましょう。

public void appendText(final String str) {
    msgBuffer.add(str);
    if (!isTimerGameMsgWork) {
        mTimerGameMsg.run();
        isTimerGameMsgWork = true;
    }
}

ゲームメッセージを表示する必要が生じると、appendTextメソッドが呼び出され、それが引数として使用されます。メソッド自体では、まずmsgBufferバッファーに行が追加されます。その後、ブール変数isTimerGameMsgWorkをチェックします。mTimerGameMsgタイマーが実行されている場合はtrueになります。このタイマーの作品は、その後からするとmsgBufferバッファ、 FIFO原理（先入れ先出し）によると、メッセージがされている受信間隔指定でmIntervalGameMsg -とゲームログに追加txtViewGameLog。 完全なメッセージ出力コード：



コード
ArrayList<String> msgBuffer = new ArrayList<>();
Handler mHandlerGameMsg;
private int mIntervalGameMsg = 1000;
boolean isTimerGameMsgWork = false;

public void appendText(final String str) {
    msgBuffer.add(str);
    if (!isTimerGameMsgWork) {
        mTimerGameMsg.run();
        isTimerGameMsgWork = true;
    }
}

final Runnable mTimerGameMsg = new Runnable() {
    @Override
    public void run() {
        if (msgBuffer.size() == 0) {
            mHandlerGameMsg.removeCallbacks(mTimerGameMsg);
            isTimerGameMsgWork = false;
        } else {
            txtViewGameLog.append(msgBuffer.get(0));
            msgBuffer.remove(0);
            mHandlerGameMsg.postDelayed(mTimerGameMsg, mIntervalGameMsg);
        }
    }
};

9.小さなゲームでも歴史に値する。


目次

ある人が魂と一緒に働きかけようとするとき、プロジェクトはこれからのみ利益を得ます。アーティストのアナスタシア・フロリコヴァがそのような人物であることが判明したのは幸運でした。それら。彼女は私が必要とするものを単に描くのではなく、ゲームの世界に興味を持ち、それがどのように機能するかを理解したかったのです。そして、突然、概して平安がないことが判明しました！この吸血鬼は誰ですか？そして、なぜプレーヤーは彼を殺すべきですか？ Vampusの部屋はどのように見えますか？などなど、私は考えていなかったし、プレイヤーに伝えるつもりもなかった。その結果、私たちは、そのような一見小さなゲームでも独自のストーリーを持つべきであることに同意しました。そして彼女は現れた。

私たちの伝説によれば、ヴァンプスは古代ですが、人々をからかうのが大好きな邪悪な神話上の生き物ではありません。はい、彼は迷宮に住んでいますが、この迷宮は古典的な暗いダンジョンの形ではなく、内容がヴァンプスの特徴である部屋の山からなる想像を絶する家の形をしています。たとえば、ある部屋には映画館があり、その壁にはお気に入りの映画のポスターがあり、もう1つの部屋にはクローゼットがあり、そこで彼は「いたずら」を準備しています。


名も無きハンターのプレイヤーは、Vampusの存在を証明したいクリプトゾロジーフォーラムの常連になりました。私たちは古典的な弓と矢をカメラとフィルムに置き換えました、そしてゲームの目標はヴァンパスを殺すことではなく、彼の写真を得ることでした。ちなみに、メインメニューはVampusについて議論するフォーラムのために再利用されており、ディスカッションからプレイヤーはそれについて学ぶことができます。

フォーラムの形式で作成されたメインメニューの最初のバージョン

他の側面については、それらはほとんど変わっていませんでした：マウスは同じように動作し、ピットに入るとクラッシュ、カメラの故障、およびゲームの終了（そしてプレーヤーの死ではない）につながり始めました。


カメラについての別の瞬間。クラシックゲームでは、プレイヤーは1〜5部屋の範囲で矢を放つことができ、論理的に見えました。私たちは弓の代わりにカメラを持っています（一度に1〜3部屋の写真を撮りますが、Vampusを攻撃する古典的な矢のように機能します）。そして、それは...奇妙に見えませんか？カメラの射程を1部屋に減らして次の部屋だけを撮影できるようにするというアイデアがありましたが、これは最初にゲームを複雑にし、次にゲームに勝てない状況が発生する可能性がありますが、これは誤りです。一般的に、これは個人的にまだ私を悩ませている瞬間ですが、私はまだ解決策を見つけていません。

ゲームのスタイルとムードに関して。ヴァンパスに関するほとんどすべてのゲームは退屈な灰色のトーンで作られ、アクションはダンジョンの暗い場所で行われます。したがって、私たちは私たちのゲームはこれとは異なり、ユーモアと明るい色で行われるべきだと決めました。




10. Vampus Transformationと最終結果


内容次

に、ゲームの作業がさらに2か月になるのを待っていました。 Vampusには20室の部屋があるため、それぞれに独自のインテリアがあります。また、アチーブメントのアイコン、ゲーム内のアイコンが描かれ、デザイン全般やUIの決定がなされました。ゲーム全体のテキストも追加され、コードが補足および最適化され、新しい機能が追加されました（たとえば、ゲーム中に情報を記録するためのメモ帳が表示されました）。一般的に、ヴァンパスは大きな変化を遂げました。
