CI（継続的インテグレーション）とは

CI（継続的統合） -文字通り「継続的統合」と解釈されます。これは、アプリケーションコードの個々の部分の統合を指します。より頻繁にコードを収集してチェックする：



  • 彼は行くのですか？
  • 自動テストに合格していますか？


よし！CIでは、このようなチェックを自動的に行うことができます。これは、コードだけでなく自動テストも作成する高度な開発チームで使用されます。彼はインタビューで尋ねられます-少なくともそれが何であるかについての理解。はい、テスターであっても。



したがって、それが何であるかについての記事でお知らせします。CIがどのように機能し、どのようにプロジェクトに役立つか。

CIとは？



CIは、人間の介入なしでのアプリケーションのアセンブリ、展開、およびテストです。例を挙げて説明します。



2人の開発者、MashaとVanyaがいるとします。そしてテスターカティア。



マーシャがコードを書きます。それをバージョン管理システムに追加します（英語のバージョン管理システム、VCSから）。これは、コードのドロップボックスのようなものです。すべての変更が保存され、いつ誰が何を変更したかをいつでも確認できる保管場所です。



その後、Vanyaは機能の一部を完成させます。また、コードをVCSに保存します。







ただし、これは単なるソースコードです。拡張子が.javaのファイルのセットなどです。したがって、Katyaは変更をテストできます。以下が必要です。



  1. ソースコードからビルドをビルドする
  2. テストマシンで実行する


ビルドのビルドとは、一連のソースコードファイルから1つの実行可能ファイルを作成する







ことです。ビルドは手動でビルドできますが、これは不必要な痔です。実行する順序、相互に依存するファイルは、コマンドを間違えないでください。通常は特別なコマンドを使用します。プログラム。 Javaの場合、これはAnt、Maven、またはGradleです。コレクターの助けを借りて、ビルドプロセスを一度設定してから、1つのコマンドで実行します。 Mavenの起動例： 



mvn clean install


これは半自動化です。チームに参加し、「ペン」でビルドを組み立てる人が必要です。開発者がこれを行うとしましょう。Katyaがビルドにテストを要求すると、Vanyaはリポジトリーからバージョンを更新し、ビルドを収集します。







ただし、ビルドをビルドするtestingテスト用のアプリを入手します。まだ起動する必要があります！これは、アプリケーションサーバーによって行われます。サーバーは異なります：wildfly、apache、jetty ...



これがwildflyの場合、次のものが必要です。



  1. ビルドをスタンドアロン/デプロイメントディレクトリにアタッチする
  2. サーバーを起動します（サービスを一度セットアップした後）。


そして、これもまた半自動化です。開発者は、アセンブリ後にテストベンチに結果のアーカイブをコピーし、サービスを有効にする必要があるためです。はい、これはいくつかのチームによって行われますが、それでも1人です。







しかし、このスキームから人を削除すると、CIが取得されます。







CIは、プロセス全体を自動化できるアプリケーションです。コードでリポジトリから変更をプルします。単独で！2つの構成オプションがあります。



  • CIはリポジトリをポーリングします「ねuckカッコウ、何か変更はありますか？」N時間/分ごと、構成方法。



  • リポジトリは、コミット時にCIに手を振ります。そして、私はここでアップデートを得ました！」（それはgitフックまたはVCSの同等物ですか）







CIが変更を受け取ると、ビルドと自動テストの構築を開始します。



アセンブリが失敗した場合（テストが失敗した場合、またはプロジェクトをアセンブルできなかった場合）、システムはすべての関係者に電子メールを書き込みます。



  • プロジェクトマネージャー（何が行われているかを知るために！）
  • 変更を加えた開発者
  • 他の誰でも-設定すると、そうなります。








ビルドが成功すると、CIはアプリケーションをテストマシンにデプロイします。そして最後に、カトカは新しいアセンブリをテストできます！







はい、もちろん、すべてを設定する必要がある場合-変更の取得元、実行する自動テスト、プロジェクトのビルド方法、後でビルドする場所をCIサーバーに指示します...しかし、私は一度セットアップしてから、実行します！



自動テストも自分で書く必要がありますが、よくできています=）







手元にある場合、CI（継続的インテグレーション）システムはソース管理を監視する一種のプログラムであり、変更がそこに表示されると、自動的にまとめられ、ビルドされ、自動テストが実行されます（もちろん、作成されている場合）。



失敗した場合は、最初から最後のコミッターまで、すべての関係者にこれを知らせます。（c）habr.com/en/post/352282

最も人気のあるのはJenkinsとTeamCityです。



しかし、他にも多くのバリエーションがあります-CruiseControl、CruiseControl.Net、Atlassian Bamboo、Hudson、Microsoft Team Foundation Server。

  • テスト付きのCDIコア -すべての自動テストを含むプロジェクトのアセンブリ。画面に表示されるように、4000以上のピースがあります。テストには30分かかりますが、コミットごとに実行されます。

CIの仕組み



CIがビルドを収集して自動テストを実行する方法と場所は？例としてTeamCityを使用しますが、他のシステムも同様に機能します。



TeamCity自体は何も収集しません。自動テストの組み立てと実行は、「エージェント」と呼ばれる他のマシンで行われ







ます。「エージェント」は単純なコンピューターです。重要ではなく、鉄片または仮想マシン。しかし、このコンピュータは、どうすればよいかをどのように理解するのでしょうか。



TeamCityにはサーバーとクライアントがあります。サーバーはまさにアプリケーションであり、そこでボタンを突き、「すべてがうまくいった」美しい写真を見るでしょう。 1台のマシンにインストールされます。







そして、「クライアント」アプリケーションが「エージェント」マシンにインストールされます。そして、サーバーの「実行」ボタンをクリックすると、







サーバーは無料のクライアントを選択し、それにすべての指示を与えます。クライアントはビルドを収集し、自動テストを実行し、結果を収集してサーバーに戻ります：「オン、ホールド、ドロー」。







サーバーは結果をユーザーに表示し、さらにすべての関係者に電子メールを送信します。



同時に、アセンブリが発生した特定のエージェントを常に確認し

VCSとの統合



CI-VCS統合を構成するためのさまざまなオプションについて説明しました。



  • CIはリポジトリをポーリングします「ねuckカッコウ、何か変更はありますか？」N時間/分ごと、構成方法。



  • リポジトリは、コミット時にCIに手を振ります。そして、私はここでアップデートを得ました！」（それはgitフックまたはVCSの同等物ですか）







しかし、いつ使用されるのですか？



もちろん、最も優れているのは、バージョン管理システムがCIサーバーに通知することです。そして、変更ごとにサイクル全体を実行します：組み立て、テスト、展開。その後、コードへの変更はすぐにテスト環境に送られ、可能な限り関連性が高くなります。







さらに、変更ごとに自動テストが実行されます。そして、テストが失敗した場合、誰のコミットがそれらを破ったかはすぐにわかります。結局のところ、それは以前は機能していましたが、Vasyaの編集後に突然壊れました-つまり、彼のコミットメントは失敗につながりました。まれな例外を除いて、秋が浮いているとき。







しかし、実際には、そのようなスキームが適用されることはめったにありません。ただ考えてください-あなたは多くのプロジェクト、多くの開発者を持つことができます。それぞれが少なくとも30分に1回何かをコミットします。また、コミットごとに30分で10個のアセンブリを実行する場合、TeamCityのキューはレイキングされません。



当社の製品の1つにコアモジュールがあり、15人以上の顧客がいます。それぞれに独自の自動テストがあります。カスタマーアセンブリは、コア+カスタマー機能です。つまり、ルートプロジェクトの変更は15の異なるアセンブリに影響を与える可能性があります。したがって、コアでコミットする場合は、すべて実行する必要があります。



ビルドエージェントが4つあった場合、火曜日の夜にこれらの顧客向けのすべてのビルドとテストが開始されました。そして午前10時までに、TeamCityにはまだ数時間ありました。


別のオプションは、多くのエージェントを購入することです。しかし、これは車自体の価格+ TeamCityのライセンスの価格です。TeamCityはすでにはるかに高価で、毎月支払うことさえあります。



したがって、通常は次のようになります



。1.非常に高速で重要なアセンブリは、すべてのコミットに任せることができます。1〜2分かかる場合は、追跡を許可します。







2.残りのビルドは、VCSに変更があったかどうかを確認します-たとえば、15分ごとに1回。あった場合は、起動します。







3.長いテスト（たとえば、パフォーマンステスト）-数日ごとに1回夜間。











テスト中のCI



アプリケーションの開発について話している場合、テストは標準サイクルに含まれます。あなたまたはあなたの開発者が自動テストを作成し、それがCIによって駆動されます。ユニット、API、GUI、ストレステストのいずれかになります。







しかし、ブラックボックスをテストしている場合はどうでしょうか。アプリケーションはありますが、ソースコードはありません。これらは統合テスターの厳しい現実です-サプライヤーはあなたに新しいアプリケーションリリースを提供します、それを本稼働する前にチェックする必要があります。



ここで、PostmanにAPIテストがあるとします。または、SeleniumでのGUIテスト。それらのCIループをカスタマイズできますか？







もちろんできます！



CIは、「自動テストを行うプロジェクトでのみ機能する」、「ソースコードにアクセスできる場合にのみ機能する」などの厳密なフレームワークを設定していません。バージョン管理システムを調べたり、調べなかったりする場合があります。これはオプションの条件です！



自動テストを書きましたか？それらを頻繁に実行する方法をCIサーバーに通知し、その結果を楽しんでください=）









合計



CI-継続的な統合。これは、アプリケーションが常にテストされているときです。すべてがうまくいきますか？テストは合格ですか？組み立ては進んでいますか？さらに、すべてのチェックは人間の介入なしに自動的に実行されます。



これは、複数の開発者が1つのアプリケーションのコードに取り組むチームにとって特に重要です。これはどのように起こりますか？別に、プログラムの一部は機能しますが、一緒に機能しなくなります。CIを使用すると、このような問題を非常に迅速に検出できます。そして、あなたがそれをより早く見つければ見つけるほど、それを修正するのはより安くなります。







したがって、名前-コード間の統合の継続的なチェック。

典型的なCIタスク：



  • コードに更新があったかどうかを確認する
  • ビルドビルド
  • 自動テストをドライブする
  • アプリケーションをテストベンチにデプロイします
  • このスタンドでGUIテスト（またはPostmanテスト）を実行する
  • 組み立てとテストの結果をメールで興味のある人全員に通知








そして、これらすべて-人間の介入なしに、自動的に！つまり、彼らは一度それを設定し、それからそれ自体を設定します。



CIがプロジェクトで構成されている場合は、常に最新のテストスタンドがあります。そして、コードで何かが壊れた場合、すぐにそれについて知ることができ、CIサーバーは手紙を送信します。そして、グラフィカルインターフェイスに移動して、すべてのビルドが成功し、テストが成功したかどうかを確認できます。プロジェクトの写真を1分で評価してください。
