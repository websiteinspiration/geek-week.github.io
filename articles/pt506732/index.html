Reutilização em toda a organização de componentes da interface do usuário

Embora o React e outras bibliotecas e estruturas de desenvolvimento da Web facilitem a reutilização de componentes, esse esquema não funciona por si só. Nesta área, ainda existem tarefas complexas que são especialmente características para esses casos quando se trata da escala de grandes organizações. A solução para esses problemas implica a necessidade de um desperdício de tempo e energia para coordenar as atividades das equipes de desenvolvimento envolvidas em vários projetos nas organizações. Na realidade, até encontrar o componente necessário ao programador desenvolvido por outra equipe já é muito difícil.



O autor do material, cuja tradução publicamos hoje, quer falar sobre como é organizada a reutilização de componentes no PayPal. Ele diz que, embora a empresa não esteja pronta para divulgar a base de código de sua solução, será um prazer compartilhar as idéias subjacentes ao sistema interno de reutilização de componentes. Campo para inserir uma quantia em dinheiro

Na realidade, ao desenvolver uma interface, nem toda equipe leva em consideração absolutamente todos os fatores que afetam o resultado final. Por exemplo, o desenvolvimento de um site acessível requer a verificação de váriosrequisitos. Estima-se que 20% da população sejam pessoas com deficiência. Isso significa que aproximadamente 60 milhões de nossos clientes podem sofrer inconvenientes ao trabalhar com nosso site, se não conseguirmos fazer com que o site cumpra totalmente os requisitos de acessibilidade de projetos da web. Da mesma forma, o desenvolvimento de uma interface localizada geralmente não é considerado uma tarefa de suma importância. Frequentemente, os programadores perdem de vista os problemas que podem ocorrer com os layouts de página quando textos localizados são exibidos neles, ou textos em idiomas com letras da direita para a esquerda. Idealmente, cada componente deve resolver esses problemas de forma independente.

Perguntas sobre os componentes

A organização de componentes juntos, por outro lado, é um grande fardo para as equipes envolvidas no compartilhamento de componentes. As equipes precisam fazer esforços consideráveis ​​para fazer isso, e esses esforços devem ser justificados. Por exemplo, as equipes estão sob pressão para liberar novas oportunidades para seu desenvolvimento. Se você comparar isso com o desenvolvimento de componentes destinados ao uso apenas no aplicativo em que uma determinada equipe está envolvida, a organização do uso conjunto de componentes requer mais esforços para projetá-los e criar as abstrações corretas. Mas e se os componentes projetados para uso repetido, ninguém, de fato, não usará? Projeto preliminar de componentes, o que parece ser necessário para os componentes,destinado a compartilhar, em princípio, priva os programadores do desejo de publicar esses componentes.

Nossa decisão



Visão geral da solução

Nós, no caminho de nossa própria solução para reutilizar componentes, tivemos que praticar constantemente em busca de um equilíbrio entre a viabilidade técnica de uma determinada idéia e a conveniência de uma solução. Estamos falando da relação entre a automação da solução de certos problemas e o trabalho manual. Decidimos usar um sistema baseado em um ambiente projetado para trabalhar com componentes. Com sua ajuda, íamos criar e visualizar componentes. Essa abordagem evita a complexidade da criação de componentes, enquanto ajuda os autores do componente a controlar seu desenvolvimento. Muitas ferramentas para organizar ambientes para trabalhar com componentes assumem a configuração do Webpack / Babel, que permite usá-los sem configurações adicionais, mas ao mesmo tempo possibilita, se necessário, alterar os parâmetros padrão.

Entre muitas dessas ferramentas, escolhemos o Storybook. A razão para isso foi o uso generalizado desse ambiente, tanto na comunidade de código aberto quanto no PayPal. Além disso, o Storybook suporta um poderoso sistema de plug-ins que poderíamos usar para organizar auditorias de qualidade de componentes e outras tarefas.

O Component Explorer é uma solução proprietária do PayPal usada pela empresa para compartilhar componentes. Esta solução, em um nível alto, permite encontrar componentes. É responsável por hospedar componentes e por sua saída usando uma interface uniforme. O Component Explorer usa um Storybook e uma ferramenta de linha de comando component-clipara publicar componentes. Um aplicativo GitHub é usado para automatizar o trabalho.component-bot. Em outras palavras, os componentes são criados e documentados usando o Storybook, após o qual são automaticamente compartilhados com component-clie component-bot. Além disso, as auditorias de qualidade dos componentes são realizadas automaticamente. Isso é feito usando o conjunto de complementos do Storybook incluído em nosso pré-configurado storybook-preset.

O Component Explorer permite compartilhar os componentes projetados como pacotes npm e os componentes que ainda não foram publicados no npm. Isso é muito importante, pois a maioria dos componentes criados pelas equipes de produto não são apresentados como pacotes npm. Esses componentes existem apenas como partes dos respectivos aplicativos. A capacidade de compartilhar componentes que não são pacotes npm permite publicar componentes para um público maior de desenvolvedores. Os comandos não têm uma necessidade urgente de extrair o componente em um repositório separado e publicá-lo no npm. O componente pode ser compartilhado publicamente a partir de seu próprio repositório.

Além disso, a organização do acesso geral a componentes "não empacotados" permite que você se livre do problema do design preliminar de componentes, sobre o qual já falamos. Isso possibilita um modelo especial de colaboração do desenvolvedor. O componente é apresentado em domínio público por padrão: a equipe só precisa fazer algum esforço se membros de outra equipe que decidem usar seu componente entrar em contato com ele. Ou, ainda mais interessante, graças à tecnologia de federação de módulos do Webpack 5, você pode compartilhar componentes diretamente do Cx sem publicá-los no npm.

E, finalmente, a organização do acesso geral a componentes "não empacotados" é aplicável a componentes que não estão planejados para serem usados ​​repetidamente. Tais componentes, por exemplo, páginas apresentadas na forma de componentes, são convenientemente usados ​​para organizar o acesso simples e rápido a alguns materiais.

Páginas apresentadas como componentes



O que é um componente?

Os componentes podem ser não apenas unidades atômicas da interface do usuário, mas também algo mais. Os componentes podem ser combinados, uma estrutura maior pode ser formada a partir deles. Tais estruturas, reunidas, podem formar uma página inteira. Com isso em mente, os programadores podem organizar componentes de várias escalas no Cx: de botões pequenos a widgets, como formulários para inserir dados de cartão de crédito e até páginas e seções de sites inteiras.

Se os programadores compartilham algo como páginas inteiras, isso ajuda as pessoas a começar a trabalhar de maneira mais aberta e eficiente.. Além disso, isso ajuda as equipes e o gerenciamento da empresa a navegar melhor no portfólio de produtos existente. Nossa visão para esse problema é que os funcionários da empresa tenham acesso aberto e gratuito a todas as páginas de produtos do PayPal no Cx.

Comunidade


Nosso sistema Component Explorer deve seu sucesso à comunidade de especialistas da empresa, que consiste em programadores, designers e gerentes de produto. Criar uma comunidade não é apenas o uso de certas ferramentas. É também a formação de uma cultura de organização e a criação da atitude correta entre as pessoas. O uso do Component Explorer é apenas um bom começo para mudar a maneira como as equipes projetam interfaces de usuário. É, por exemplo, que os componentes, por padrão, são dispostos em domínio público. Esses componentes abrem novas oportunidades de interação entre equipes. Vemos o valor do Component Explorer no seguinte:

  • Esse sistema permite que as equipes interajam de maneira mais aberta e eficiente, o que aumenta a velocidade do desenvolvimento e da inovação.
  • Isso enfraquece o impacto na organização de fenômenos como o isolamento das equipes, o desperdício de esforços de diferentes equipes na implementação da mesma funcionalidade, a inconsistência das equipes.
  • Contribui para criar valores que apóiam a comunidade e vão além do sucesso de equipes individuais.
  • Ajuda aqueles que criaram algo bem-sucedido a obter reconhecimento da comunidade.
