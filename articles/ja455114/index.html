CPythonでの整数型の実装

CPythonのメモリマネージャーであるPandasの実装の詳細については、Habréに関する記事が既にありました。辞書の実装に関する記事を書きました。

あなたは通常の整数型について書くことができるように思えますか？ただし、すべてがそれほど単純なわけではなく、整数型はそれほど明白ではありません。

なぜx * 2がx << 1よりも速いのか疑問に思っている場合。

そして、次のトリックを上げる方法：

>>> 42 == 4
True
>>> 42
4
>>> 1 + 41
4

次に、この記事を読む必要があります。

Pythonにはプリミティブ型はありません。すべての変数はオブジェクトであり、動的メモリに割り当てられます。同時に、整数は1つのタイプ（10進数は考慮しません）-PyLongObject のみで表されます。その実装と宣言は、ファイルlongobject.h、longintrepr.h、およびlongobject.cにあります。

struct _longobject {
    PyObject_VAR_HEAD   //       
    digit ob_digit[1];  //   
};

CPythonの任意のオブジェクトには2つのフィールドがあります：ob_refcnt-オブジェクトへのリンクのカウンターとob_type-オブジェクトのタイプへのポインター、長さを変更できるオブジェクトへのポインター、ob_sizeフィールドが追加されます-現在割り当てられているサイズ（使用されるサイズは小さい場合があります）。

したがって、整数型は個別の桁からの可変長の配列で表されるため、そのままのpythonは長い演算をサポートします。言語の2番目のバージョンでは、「通常の」整数の個別の型がありました。「長い」整数は、文字Lを使用して作成されたか、通常のものによってオーバーフローを引き起こしました; 3番目のバージョンではそれを拒否することが決定されました。

    # python2
    num1 = 42
    print num1, type(num1)  # 42 <type 'int'>
    num2 = 42L
    print num2, type(num2)  # 42 <type 'long'>
    num3 = 2 ** 100
    print num3, type(num3)  # 1267650600228229401496703205376 <type 'long'>

_longobject構造体によって格納される整数値は、次の値に等しくなります。

∑i=0abs(ob_size)−1ob_digiti∗2SHIFT∗i


ビットとしては、64ビットシステムでは32ビット符号なしタイプ（uint32_t）、32ビットシステムでは16ビット符号なしshortタイプが使用されます。

実装で使用されるアルゴリズムは、前の式からのSHIFTに厳格な制限を課します。特に、15で除算する必要があるため、cpythonは64ビットシステムと32ビットシステムでそれぞれ30と15の2つの値をサポートします。負の値の場合、ob_sizeは負の値を持ち、ob_size = 0のような数値によってゼロが与えられます。

算術演算を実行するとき、インタプリタはオペランドの現在の長さをチェックし、両方が1ビットで構成されている場合は、標準の演算が実行されます。

ただし、シフトコマンドでは、「短い」整数の場合のチェックは行われず、長い演算の場合は常に実行されます。したがって、2を掛けることはシフトより速いです。除算は右シフトよりも遅いことに注意してください。これは、1桁の整数の場合もチェックしません。しかし、除算はより複雑です。最初に商を計算し、次に剰余であるNULLがそれに渡されるメソッドが1つあります。1つのことを計算する必要がある場合、つまり、メソッドもこのケースをチェックする必要がある場合、すべてオーバーヘッドが増加します。

比較関数には、「短」整数の特別なケースもありません。

static int
long_compare(PyLongObject *a, PyLongObject *b)
{
    Py_ssize_t sign;

    // ,       
    //       
    if (Py_SIZE(a) != Py_SIZE(b)) {
        sign = Py_SIZE(a) - Py_SIZE(b);
    }
    else {
        Py_ssize_t i = Py_ABS(Py_SIZE(a));
        //  ""       
        while (--i >= 0 && a->ob_digit[i] == b->ob_digit[i])
            ;
        if (i < 0)
            sign = 0;
        else {
            sign = (sdigit)a->ob_digit[i] - (sdigit)b->ob_digit[i];
            if (Py_SIZE(a) < 0)
                sign = -sign;
        }
    }
    return sign < 0 ? -1 : sign > 0 ? 1 : 0;
}

数値の配列（リスト）


整数型の変数を作成する場合、インタープリターは動的メモリに十分なスペースを割り当て、参照カウンター（型ssize_t）、型PyLongObjectへのポインター、ビット配列の現在のサイズ（ssize_t）を設定し、配列自体を初期化する必要があります。 64ビットシステムの最小構造サイズは、2 * ssize_t +ポインタ+数字= 2 * 8 + 8 + 4 = 28バイトです。数値のリストを作成すると、追加の問題が発生します。数値はプリミティブ型ではなく、Pythonのリストはオブジェクトへの参照を格納するため、オブジェクトは動的メモリに順番に配置されません。

この配置により、アレイ全体の反復が遅くなります。実際、ランダムアクセスが実行され、遷移を予測することができなくなります。

動作時間を遅くし、メモリの断片化に寄与する動的メモリの過剰な割り当てを回避するために、cpythonに最適化が実装されています。起動フェーズでは、小さな整数の配列が-5〜256に事前に割り当てられています。

#ifndef NSMALLPOSINTS
#define NSMALLPOSINTS           257  //     python    .
#endif
#ifndef NSMALLNEGINTS
#define NSMALLNEGINTS           5
#endif

import ctypes

#  PyLongObject
class IntStruct(ctypes.Structure):
    #  
    _fields_ = [("ob_refcnt", ctypes.c_long),
                ("ob_type", ctypes.c_void_p),
                ("ob_size", ctypes.c_long),
                ("ob_digit", ctypes.c_int)]

    def __repr__(self):
        return ("IntStruct(ob_digit={self.ob_digit}, ob_size={self.ob_size}, "
                "refcount={self.ob_refcnt})").format(self=self)


if __name__ == '__main__':
    #    42
    int42 = IntStruct.from_address(id(42))
    print(int42)

    int_minus_2 = IntStruct.from_address(id(-2))
    print(int_minus_2)  # ob_digit=2, ob_size=-1

    #      
    int42.ob_digit = 4
    print(4 == 42)  # True
    print(1 + 41)   # 4

Honest PyLongObjectはこのリストに保存されます。つまり、それらには参照カウンターがあります。たとえば、スクリプトとインタープリターが使用するゼロの数を確認できます。

整数の内部実装から、cpythonでの算術演算は高速ではないことがわかります。他の言語は、配列を順次反復し、数値をレジスタに読み取り、いくつかのプロセッサ命令を直接呼び出します。cpythonはメモリ全体にタグ付けされ、かなり複雑なメソッドを実行します。 1ビットの整数の最も単純なケースでは、1つの命令を呼び出すだけで十分ですが、インタープリターはサイズを比較してから、動的メモリにオブジェクトを作成し、サービスフィールドに入力して、そのポインターを返す必要があります。さらに、これらのアクションにはGILロックが必要です。これで、numpyライブラリが生まれた理由とそれが非常に人気がある理由が理解できました。

cpythonの型全体に関する記事を、突然、ブール型に関する情報で仕上げたいと思います。

事実、Pythonでは長い間、ブール変数に個別の型はなく、すべての論理関数が0と1を返していましたが、新しい型を導入することにしました。同時に、全体として子型として実装されました。

さらに、ブール型の各値はシングルトンであり、ブール変数はTrueまたはFalseのインスタンスへのポインターです（Noneも同様に実装されています）。

struct _longobject _Py_FalseStruct = {
    PyVarObject_HEAD_INIT(&PyBool_Type, 0)
    { 0 }
};

struct _longobject _Py_TrueStruct = {
    PyVarObject_HEAD_INIT(&PyBool_Type, 1)
    { 1 }
};

static PyObject *
bool_new(PyTypeObject *type, PyObject *args, PyObject *kwds)
{
    PyObject *x = Py_False;
    long ok;

    if (!_PyArg_NoKeywords("bool", kwds))
        return NULL;
    if (!PyArg_UnpackTuple(args, "bool", 0, 1, &x))
        return NULL;
    ok = PyObject_IsTrue(x);
    if (ok < 0)
        return NULL;
    return PyBool_FromLong(ok);
}

PyObject *PyBool_FromLong(long ok)
{
    PyObject *result;

    if (ok)
        result = Py_True;
    else
        result = Py_False;
    Py_INCREF(result);
    return result;
}

PyObject_IsTrue（x）はブール値を計算するためのトリッキーなメカニズムであり、あなたはそれを見ることができ、ここでブール機能上またはセクションでのドキュメント。

このような遺産は、Trueと1の正確な等価、ディクショナリとセットのキーとしてTrueと1を持つことができない、ブール型の算術演算の許容性など、いくつかの面白い効果をもたらします。
