﻿複数文字定数を使用することの危険性


コード分​​析中、PVS-Studioはデータフローを分析し、変数の値を操作します。値は定数から取得されるか、条件式から推定されます。それらを仮想値と呼びます。最近、複数文字定数で機能するようにそれらを改善しました。これが、新しい診断ルールを作成する理由でした。

前書き


複数文字リテラルは実装で定義されるため、コンパイラーによって、これらのリテラルをさまざまな方法でエンコードできます。たとえば、GCCとClangはリテラル内の文字の順序に基づいて値を設定しますが、MSVCは文字のタイプ（通常またはエスケープ）に応じてそれらを移動します。

たとえば、リテラル 'T \ x65s \ x74'は、コンパイラーに応じてさまざまな方法でエンコードされます。同様のロジックをアナライザーに追加する必要がありました。その結果、新しい診断ルールV1039を作成して、コード内のそのようなリテラルを検出しました。このようなリテラルは、アセンブリに複数のコンパイラを使用するクロスプラットフォームプロジェクトでは危険です。

Diagnostics V1039


例を考えてみましょう。さまざまなコンパイラーによってコンパイルされた以下のコードは、異なる動作をします。

#include <stdio.h>

void foo(int c)
{
  if (c == 'T\x65s\x74')                       // <= V1039
  {
    printf("Compiled with GCC or Clang.\n");
  }
  else
  {
    printf("It's another compiler (for example, MSVC).\n");
  }
}

int main(int argc, char** argv)
{
  foo('Test');
  return 0;
}

異なるコンパイラでコンパイルされたプログラムは、画面に異なるメッセージを出力します。

特定のコンパイラを使用するプロジェクトの場合、これは顕著ではありませんが、移植によって問題が発生する可能性があるため、このようなリテラルを単純な数値定数に置き換える必要があります。たとえば、「Test」を0x54657374に変更します。

コンパイラーの違いを示すために、3文字と4文字のシーケンス（たとえば、「GHIJ」と「GHI」）が取られ、コンパイル後のメモリー内でのそれらの表現が表示される小さなユーティリティーを作成します。

ユーティリティコード：

#include <stdio.h>

typedef int char_t;

void PrintBytes(const char* format, char_t lit)
{
  printf("%20s : ", format);

  const unsigned char *ptr = (const unsigned char*)&lit;
  for (int i = sizeof(lit); i--;)
  {
    printf("%c", *ptr++);
  }
  putchar('\n');
}

int main(int argc, char** argv)
{
  printf("Hex codes are: G(%02X) H(%02X) I(%02X) J(%02X)\n",'G','H','I','J');
  PrintBytes("'GHIJ'", 'GHIJ');
  PrintBytes("'\\x47\\x48\\x49\\x4A'", '\x47\x48\x49\x4A');
  PrintBytes("'G\\x48\\x49\\x4A'", 'G\x48\x49\x4A');
  PrintBytes("'GH\\x49\\x4A'", 'GH\x49\x4A');
  PrintBytes("'G\\x48I\\x4A'", 'G\x48I\x4A');
  PrintBytes("'GHI\\x4A'", 'GHI\x4A');
  PrintBytes("'GHI'", 'GHI');
  PrintBytes("'\\x47\\x48\\x49'", '\x47\x48\x49');
  PrintBytes("'GH\\x49'", 'GH\x49');
  PrintBytes("'\\x47H\\x49'", '\x47H\x49');
  PrintBytes("'\\x47HI'", '\x47HI');
  return 0;
}
