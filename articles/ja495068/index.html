OpenGL RAIIスタイルのプリミティブ

こんにちは、Habraユーザー。私は長い間書いておらず、おそらく誰かが私からの記事を待っていたでしょう-もちろんそうではありません。空き時間が少し増えて、GitHubが完全に空になったので、クローンMein kampf Minecraftを作成することにしました。高い確率で、私はこれを文書化します-habr.com上の私の記事に従ってください。今日は、面白い場合は猫の下で、私がOpenGLプリミティブをRAIIスタイルでラップする方法を示します。

読みたいものをコメントに書き込んでください。私はMinecraftを一から書くべきですか？それでは始めましょう。例としてバッファオブジェクトを示します。最初に、メインの呼び出し（glGenBuffers、glDeleteBuffers）を別のクラスで行います。

このような場所：

class buffer_object
{
public:
    buffer_object() noexcept {
        glGenBuffers(1, m_object);
    }
    
    ~buffer_object() noexcept {
        glDeleteBuffers(1, m_object);
    }
    
    uint32_t get_object() const noexcept {
        return m_object;
    }
private:
    uint32_t m_object;
};

長所：

  • バッファの削除を監視する必要はありません
  • OpenGLヘッダーを有効にしても突き出ない（関数の定義を* .cppに転送する場合）

マイナス：

  • バインド時にバッファのタイプ（配列、要素...）を指定することはできません
  • glGenBuffersへの1回の呼び出しで複数のバッファを作成することはできません
  • バッファがコピーされた場合のリモートオブジェクトへのアクセスに関する問題

コピーの問題を解決するために、私たちはそれを単に禁止します。これを行うには、コピー不可能なクラスを作成して継承します。

コピーできないクラスの例：

struct noncopyable
{
    noncopyable() = default;
            
    noncopyable(noncopyable&&) = default;
    noncopyable& operator = (noncopyable&&) = default;
            
    noncopyable(const noncopyable&) = delete;
    noncopyable& operator = (const noncopyable&) = delete;
};

素晴らしい、マイナス1つの問題。さらに、バインディングの問題をどのように解決するかを考えます... bind / unbind関数を追加します。

void bind(buffer_type type) noexcept {
    glBindBuffer(static_cast<GLenum>(type), m_object);
}
    
void unbind() noexcept {
    glBindBuffer(static_cast<GLenum>(type), 0);
}

buffer_typeは列挙型クラスですが、extern const（in * .hpp）、および（* .cpp）で次のようなことを行うことができます。

const uint32_t array_buffer = GL_ARRAY_BUFFER;

繰り返しますが、それははみ出しません 

#include <GL/glew.h>

長所：

  • バッファの削除を監視する必要はありません
  • OpenGLヘッダーを有効にしても突き出ない（関数の定義を* .cppに転送する場合）
  • バインド時にバッファのタイプ（配列、要素...）を示すことができます
  • コピーなし-コピーの問題なし

マイナス：

  • glGenBuffersへの1回の呼び出しで複数のバッファを作成することはできません

だから、ほとんどすべてが素晴らしいですが、それでもいくつかのバッファを作成することはできません...修正しましょう。

glGenBuffersは、配列へのポインタと生成するバッファの数を取ることができます。
配列が必要です。std:: vectorを使用できますが、メモリを割り当てる必要があるのは1回だけです。ここでは、std ::配列を使用することをお勧めしますが、これにより、さらに1レベルの抽象化を行う必要があります。

クラスをstd ::配列に書き換えて、テンプレートを少し追加します。

constexpr uint8_t default_object_count = 1;
constexpr size_t  default_index        = 0;

template <type::buffer_type type, uint32_t count = default_object_count>
class buffer_object : noncopyable
{
public:
    buffer_object() noexcept {
        glGenBuffers(count, m_object.data());
    }
    
    ~buffer_object() noexcept {
        glDeleteBuffers(count, m_object.data());
    }
    
    template <size_t index = default_index>
    void bind() noexcept
    {
        static_assert(index < count, "index larger than array size");
        glBindBuffer(static_cast<GLenum>(type), m_object[index]);
    }
    
    void unbind() noexcept {
        glBindBuffer(static_cast<GLenum>(type), 0);
    }
    
    buffer_object(buffer_object&&) = default;
    buffer_object& operator = (buffer_object&&) = default;
    
private:
    std::array<uint32_t, count> m_object;
};

そして、ここに私たちが得たものがあります。

長所：

  • バッファの削除を監視する必要はありません
  • バインド時にバッファのタイプ（配列、要素...）を示すことができます
  • コピーなし-コピーの問題なし
  • glGenBuffersを1回呼び出すだけで複数のバッファを作成できます

マイナス：

  • 同じオブジェクトに異なるバッファをバインドすることはできません
  • OpenGLヘッダーを有効にする

まあ...短所が多すぎる、何ができるのでしょうか？

OpenGL関数が呼び出される抽象化の別のレベルを追加することで、GL / glew.hのインクルードを削除できます（OpenGL + DirectXのサポートが計画されている場合は、これを行う必要があります）。異なるバッファーとのバインドは少し複雑です。オプションとして、どのバッファーでどのインデックスが禁止されているかを忘れることができるため、別の配列を追加し、それにバッファータイプを書き込むことができます。私はまだこれを行っていませんが、開発のこの段階では、これで十分です。

より使いやすくするために、スコープ付きバインドクラスを作成しました。そこで彼は：

constexpr uint32_t default_bind_index = 0;
template <class T>
class binder : utils::noncopyable
{
public:
      template <uint32_t index = default_bind_index>
      binder(T& ref) : m_ref(ref) { m_ref.template bind<index>(); }
    ~binder()                            { m_ref.unbind();}
        
private:
    T& m_ref;
};

この行に戸惑う人もいます。 

m_ref.template bind<index>();

templateキーワードを指定しない場合、次のエラーが発生します。

バインダー.hpp：22：46：依存するテンプレート名 'bind'の前に 'template'キーワードがありません
これは、この場合、Tが依存する型であることを意味します。コンパイラはまだm_refのタイプを認識していません。バインディングはまだないので、コンパイラはそれを純粋に構文的に処理します。したがって、<はより小さい演算子として解釈されます。これが実際にはメンバー関数テンプレートの特殊化であることをコンパイラーに示すには、ドット演算子の直後にテンプレートキーワードを追加する必要があります。
