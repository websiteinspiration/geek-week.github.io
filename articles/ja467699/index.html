Visual Studio 2019 Communityで最速のstrlenを実行して欠陥を見つける方法

最近のプロセッサーでの「奇妙な」ポップカウント命令の使用に関する記事に私は促されました。これは、ユニットの数を数えることではなく、C行（nullで終了する行）の終わりのサインを検出することについてです。
nullで終了する文字列は、プログラミング言語で文字列を表現する方法であり、特別な文字列型を導入する代わりに、文字の配列が使用され、文字列の終わりは、最初に検出された特別なnull文字（値が0のASCIIコードからのNUL）です。

そのような用語の長さを決定するために、標準関数が使用されます。 

size_t __cdecl strlen(char const* str)

Cで記述できる操作アルゴリズムは次のとおりです。


size_t strlen_algo(const char* str)
{
	size_t length = 0;
	while (*str++)
		length++;
	return length;
}

MS Visual Studio 2019コミュニティコンパイラ（Release、x86）がそれをどのように変換するかを見てみましょう。

08811F7h:
mov         al,byte ptr [ecx]  
inc         ecx  
test        al,al  
jne         main+0D7h (08811F7h)

つまり、1バイトがメモリからロードされ、ゼロと比較されます。リリースでプロジェクトをビルドすると、strlen呼び出しの代わりに同じコードが使用されます。アルゴリズムは正しいですが、速度は十分ではないようです。Debugで標準strlenを呼び出してコードをコンパイルするとどうなりますか？-strlenライブラリ関数は期待どおりに呼び出されますが、アセンブラで手動で記述されます。

size_t strlen2(const char* str)
{
	register size_t len = 0;
	// align to 16 bytes
	while ((((intptr_t)str) & (sizeof(__m128i) - 1)) != 0)
	{
		if (*str++ == 0)
			return len;
		++len;
	}
	// search for 0
	__m128i xmm0 = _mm_setzero_si128();
	__m128i xmm1;
	int mask = 0;
	for (;;)
	{
		xmm1 = _mm_load_si128((__m128i*)str);
		xmm1 = _mm_cmpeq_epi8(xmm1, xmm0);
		if ((mask = _mm_movemask_epi8(xmm1)) != 0)
		{
			// got 0 somewhere within 16 bytes in xmm1, or within 16 bits in mask
			// find index of first set bit

#ifndef _DISABLE_ASM_BSF // define it to disable ASM
#if (_MSC_VER >= 1300)   // make sure <intrin.h> is included
			unsigned long pos;
			_BitScanForward(&pos, mask);
			len += (size_t)pos;
#elif defined(_MSC_VER)  // earlier MSVC's do not have _BitScanForward, use inline asm
			__asm bsf edx, mask; edx = bsf(mask)
			__asm add edx, len; edx += len
			__asm mov len, edx; len = edx
#elif ((__GNUC__ >= 4) || ((__GNUC__ == 3) && (__GNUC_MINOR__ >= 4))) // modern GCC has built-in __builtin_ctz
			len += __builtin_ctz(mask);
#elif defined(__GNUC__) // older GCC shall use inline asm
			unsigned int pos;
			asm("bsf %1, %0" : "=r" (pos) : "rm" (mask));
			len += (size_t)pos;
#else                    // none of choices exist, use local BSF implementation
			len += count_bits_to_0(mask);
#endif
#else
			len += count_bits_to_0(mask);
#endif

			break;
		}
		str += sizeof(__m128i);
		len += sizeof(__m128i);
	}
	return len;
}

最新のプロセッサのベクトル命令を使用してより高速に実行することは可能ですか？試してみよう。インテルイントリンシクスガイド

を使用して、文字列を操作するためだけに設計された組み込み_mm_cmpistri SSE4.2を見つけます。長さ128ビットの2つのベクトルと操作のマスクが入力に供給されます。マスクとして使用するのは、_SIDD_UBYTE_OPS = 0-データ型、_SIDD_CMP_EQUAL_EACH = 8-バイトアライメントの操作であり、ゼロベクトルと比較します。戻り値は、最初のペアごとに等しくない要素の数になります（つまり、要素が左から右に一致する場合、カウントは停止します。誰かが動作を確認したら喜んでいます）。

ロードされた行のアドレスを調整するために使用されます。アドレスは、ほとんどのSSE命令が機能するために16の倍数で必要です。私たちが使用するpcmpistri命令の場合、アラインメントは厳密には必要ありません。アクセス例外は発生しません。
本質的

__m128i data = _mm_load_si128((__m128i*)(str + length));
if ((res = _mm_cmpistri(z128, data, _SIDD_CMP_EQUAL_EACH)) != 16)

にコンパイル：
pcmpistri   xmm1,xmmword ptr [eax+edx],8

ただし、16に揃えることも私たちの場合に役立ちます。速度が少し向上するので、16バイトの読み取りサイクルが未割り当ての可能性のあるページ（4Kメモリページ）に移動しないことが確実です。

サイクル：

while (str[length])
    length++;

文字列の終わりが検出された場合、文字列のサイズを「取得」します（_mm_cmpistriの操作アルゴリズムが完全にわからないため）。
コメントの後に削除ピクル、小さいサイズのラインが増加しました。
