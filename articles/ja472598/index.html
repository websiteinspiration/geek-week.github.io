動的アスペクト指向

多くの有効なフィールド値が他の値に依存する特定のサブジェクトモデルに関するストーリー。

仕事


特定の例を使用すると分解が簡単です。多くのパラメーターでセンサーを構成する必要がありますが、パラメーターは相互に依存しています。たとえば、応答しきい値はセンサーのタイプ、モデルと感度に依存し、可能なモデルはセンサーのタイプなどに依存します。

この例では、センサーのタイプとその値（トリガーとなるしきい値）のみを取得します。

public class Sensor
{
    // Voltage, Temperature
    public SensorType Type { get; internal set; }
    //-400..400 for Voltage, 200..600 for Temperature
    public decimal Value { get; internal set; }
}

電圧センサーと温度センサーの場合、値がそれぞれ-400..400と200..600の範囲のみであることを確認してください。すべての変更を追跡してログに記録できます。

「シンプルなソリューション


データの一貫性を維持するための最も単純な実装は、手動で制限と依存関係をセッターとゲッターに設定することです。

public class Sensor
{
	private SensorType _type;
	private decimal _value;
	public SensorType Type
	{
	   get { return _type; }
	   set
	   {
		  _type = value;
		  if (value == SensorType.Temperature) Value = 273;
		  if (value == SensorType.Voltage) Value = 0;
	   }
	}
	public decimal Value
	{
	   get { return _value; }
	   set
	   {
		  if (Type == SensorType.Temperature && value >= 200 && value <= 600
			 || Type == SensorType.Voltage && value >= -400 && value <= 400)
			 _value = value;
	   }
	}
}

1つの依存関係により、読みにくいコードが大量に生成されます。条件を変更したり、新しい依存関係を追加したりすることは困難です。



実際のプロジェクトでは、そのようなオブジェクトには30を超える依存フィールドと、それぞれに200を超えるルールがありました。記述された解決策は機能していますが、そのようなシステムの開発とサポートに大きな頭痛の種をもたらします。

「理想的」だが非現実的


ルールは短い形式で簡単に記述でき、関連するフィールドの横に配置できます。完璧に：

public class Sensor
{
	public SensorType Type { get; set; }

	[Number(Type = SensorType.Temperature, Min = 200, Max = 600, Force = 273)]
	[Number(Type = SensorType.Voltage, Min = -400, Max = 400, Force = 0)]
	public decimal Value { get; set; }
}

Forceは、状態が変化した場合に設定する値です。

ターゲットプロパティが依存するフィールドのリストは事前定義されていないため、C＃構文のみで属性にこれを書き込むことはできません。

働き方


ルールは次のように記述します。

public class Sensor
{
    public SensorType Type { get; set; }

    [Number("Type=Temperature", "200..600", Force = "273")]
    [Number("Type=Voltage", "-400..400", Force = "0")]
    public decimal Value { get; set; }
}

それはそれを機能させるために残っています。そのようなクラスは単に役に立たない。

発車係


アイデアは単純です-セッターを閉じて、特定のディスパッチャーを通じてフィールド値を変更します。ディスパッチャーはすべてのルールを理解し、その実行を監視し、フィールドの変更について通知し、すべての変更をログに記録します。



オプションは機能していますが、コードはひどく見えます：

someDispatcher.Set(mySensor, "Type", SensorType.Voltage);

もちろん、ディスパッチャを依存関係のあるオブジェクトの不可欠な部分にすることができます。

mySensor.Set("Type", SensorType.Voltage)

しかし、私のオブジェクトは他の開発者によって使用され、時にはなぜそれがそれほど必要なのかが完全に明確にならない場合があります。結局のところ、簡単に書きたいのです。

mySensor.Type=SensorType.Voltage;

継承


具体的には、私たちのモデルでは、依存関係のあるオブジェクトのライフサイクルを私たち自身で管理しました。モデル自体でのみ作成し、編集のみを外部に提供しました。したがって、すべてのフィールドを仮想化し、モデルの外部インターフェースを変更せずに残しますが、チェックのロジックを実装する「ラッパー」のクラスで機能します。



これは外部ユーザーにとって理想的なオプションです。彼は通常の方法でそのようなオブジェクトを操作します

mySensor.Type=SensorType.Voltage

そのようなラッパーを作成する方法を学ぶことは残っています

クラス生成


実際に生成するには2つの方法があります。

  • 完全な属性ベースのコードを生成する
  • 属性チェックを呼び出す完全なコードを生成する

属性に基づいてコードを生成することは間違いなくクールで、すぐに機能します。しかし、どれだけの力が必要になるでしょう。そして、最も重要なのは、新しい制限/ルールを追加する必要がある場合、いくつの変更が必要になるか、そしてどの程度複雑か？

各セッターの標準コードを生成し、属性を分析してチェックを実行するメソッドを呼び出します。

ゲッターは変更しません。

MethodBuilder getPropMthdBldr = typeBuilder.DefineMethod("get_" + property.Name,
	  MethodAttributes.Public |
	  MethodAttributes.SpecialName |
	  MethodAttributes.HideBySig |
	  MethodAttributes.Virtual,
	  property.PropertyType, Type.EmptyTypes);
ILGenerator getIl = getPropMthdBldr.GetILGenerator();

getIl.Emit(OpCodes.Ldarg_0);
getIl.Emit(OpCodes.Call, property.GetMethod);
getIl.Emit(OpCodes.Ret);

ここでは、メソッドに渡された最初のパラメーターがスタックに置かれます。これはオブジェクト（this）への参照です。次に、基本クラスのゲッターが呼び出され、結果が返され、スタックの一番上に置かれます。それら。ゲッターは、呼び出しを基本クラスに転送するだけです。

セッターは少しトリッキーです。分析のために、静的メソッドを作成します。これは、ほぼ次の方法で分析を生成します。

if (StrongValidate(this, property, value))
{	
	value = SoftValidate(this, property, value);
	if (oldValue != value)
	{
		<    value>;
		ForceValidate(baseModel, property);
		Log(baseModel, property, value, oldValue);
	}
}

StrongValidate-ルールに適合する値に変換できない値を破棄します。たとえば、テキストボックスには「y」と「n」のみを入力できます。"u"を書こうとするときは、モデルが破壊されないように変更を拒否する必要があります。

    [String("", "y, n")]

SoftValidate-値を不適切から有効に変換します。たとえば、intフィールドは数値のみを受け入れることができます。111を書き込もうとすると、値を最も近い適切な値「9」に変換できます。

    [Number("", "0..9")]

<値を指定して基本セッターを呼び出す>-有効な値を取得したら、基本クラスのセッターを呼び出してフィールドの値を変更する必要があります。

ForceValidate-変更後、フィールドに依存するフィールドで無効なモデルを取得する可能性があります。たとえば、タイプを変更すると、値が変更されます。

ログは単なる通知とログです。

このようなメソッドを呼び出すには、オブジェクト自体、その新旧の値、および変化するフィールドが必要です。このようなセッターのコードは次のようになります。

MethodBuilder setPropMthdBldr = typeBuilder.DefineMethod("set_" + property.Name,
	  MethodAttributes.Public |
	  MethodAttributes.SpecialName |
	  MethodAttributes.HideBySig |
	  MethodAttributes.Virtual,
	  null, new[] { property.PropertyType });
//  ,   
var setter = typeof(DinamicWrapper).GetMethod("Setter", BindingFlags.Static | BindingFlags.Public);
 
ILGenerator setIl = setPropMthdBldr.GetILGenerator();
setIl.Emit(OpCodes.Ldarg_0);//     - this
setIl.Emit(OpCodes.Ldarg_1);//     -  setter    (value)
if (property.PropertyType.IsValueType) //  Boxing,    Value,         object
{
	setIl.Emit(OpCodes.Box, property.PropertyType);
}
setIl.Emit(OpCodes.Ldstr, property.Name); //    
setIl.Emit(OpCodes.Call, setter);

setIl.Emit(OpCodes.Ret);

基本クラスの値を直接変更する別のメソッドが必要です。コードは単純なゲッターに似ていますが、パラメーターは2つしかありません-これと値：

MethodBuilder setPureMthdBldr = typeBuilder.DefineMethod("set_Pure_" + property.Name,
	  MethodAttributes.Public, CallingConventions.Standard, null, new[] { property.PropertyType });
ILGenerator setPureIl = setPureMthdBldr.GetILGenerator();

setPureIl.Emit(OpCodes.Ldarg_0);
setPureIl.Emit(OpCodes.Ldarg_1);
setPureIl.Emit(OpCodes.Call, property.GetSetMethod());
setPureIl.Emit(OpCodes.Ret);

検証


検証自体のコードは単純です。最も長い条件の原則に従って現在のアクティブな属性を探し、新しい値が有効かどうかを尋ねます。ルールを選択するときは、次の2つのことを考慮する必要があります（ルールを解析して適切なルールを計算する）。

  • GetCustomAttributesの結果をキャッシュします。フィールドから属性を取得する関数は、毎回属性を作成するため、動作が遅くなります。結果をキャッシュします。基本クラスBaseModelで実装しました
  • 適切なルールを計算するときは、フィールドタイプを処理する必要があります。すべての値を文字列に減らして比較すると、動作が遅くなります。特に列挙型。DependencyAttribute基本属性クラスに実装されています
