小さなUnityをゼロから作成する方法



当社には独自のゲームエンジンがあり、開発したすべてのゲームで使用されています。重要な基本機能をすべて提供します。 

  • レンダリング
  • SDKを操作します。
  • オペレーティングシステムを操作します。
  • ネットワークとリソース。 

ただし、Unityがそれほど重要視されているもの、つまりシーンやゲームオブジェクトを整理するための便利なシステムや、それらのエディターが欠けていました。

ここでは、これらすべての設備をどのように実装したか、および何を実現したかを説明します。

今は


これで、すべての重要なサブシステムとエディターを備えたUnityのコンポーネントシステムに似たものがあります。ただし、特定のプロジェクトのニーズから始めたため、かなり大きな違いがあります。

シーンに格納されているビジュアルオブジェクトがあります。これらのオブジェクトは、階層に編成されたノードで構成され、各ノードには次のような多数のエンティティを含めることができます。

  • 変換-ノードの変換。
  • コンポーネント-レンダリングに関与しており、1つだけ存在することも、まったく存在しないこともあります。コンポーネントは、表示可能なスプライト、メッシュ、パーティクル、およびその他のエンティティです。Unityに最も近いものはレンダラーです。
  • 振る舞い-振る舞いに責任があり、いくつかあるかもしれません。これはUnityのMonoBehaviourの直接の類似物であり、ロジックはそれらに記述されています。
  • 並べ替えは、シーン内のノードが表示される順序を担当するエンティティです。私たちのシステムは、オブジェクトを表示するための既存の多様なロジックを使用して、すでに実行中のゲームに簡単に統合できたはずなので、新しいエンティティを古いものに統合できる必要がありました。したがって、並べ替えにより、表示順序の制御を外部コードに移すことができます。

Unityと同様に、プログラマーはコンポーネント、動作、または並べ替えを作成します。これを行うには、クラスを作成し、必要なイベント（Update、OnStartなど）を再定義し、特別な方法で必要なフィールドをマークします。UnrealEngineでは、これはマクロで行われ、コメントでタグを使用することにしました。

/// @category(VSO.Basic)
	class SpriteComponent : public MaterialComponent
	{
		VISUAL_CLASS(MaterialComponent)

	public:
		/// @getter
		const std::string& GetId() const;
		/// @setter
		void SetId(const std::string& id);

	protected:
		void OnInit() override;
		void Draw() override;

	protected:
		/// @property
		Color _color = Color::WHITE;
		/// @property
		Sprite _sprite;
	};

さらにクラスでは、タグを考慮に入れて、すべてのコードが生成されます。これは、データの保存と読み込み、エディターの動作、クローン作成やその他の小さな関数のサポートに必要です。

エディターの自動シリアル化と生成は、ビジュアルオブジェクトに格納されているエンティティだけでなく、任意のクラスでもサポートされています。これを行うには、特別なSerializableクラスからそれを継承し、タグで必要なプロパティをマークするだけで十分です。また、クラスのインスタンスを完全なアセット（UnityのScriptableObjectに類似）にする場合は、クラスをAssetクラスから継承する必要があります。

その結果、ライブラリは新しい機能をすばやく開発する機会を提供します。そして今では、エフェクトの作成、UIのレイアウト、ゲームシーンのデザインなど、ゲームの開発に関する作業の一部を、プログラマーよりも上手に対処できる専門家に転送できます。

コード生成


多くのシステムの操作では、C ++にリフレクション（リフレクション -プログラムコード内の型に関する情報にアクセスする機能）が欠如しているために必要な、かなり多くのルーチンコードを記述する必要があります。したがって、このテクニカルコードのほとんどを生成します。 

ジェネレーターは、ヘッダーファイルを解析し、それらに基づいて必要なコードを生成するPythonスクリプトのセットです。柔軟な生成設定のために、コメントで特別なタグが使用されます。

次のサブシステムのコードを生成できます。

  • シリアル化-ディスクからデータを保存/ロードするため、またはネットワーク経由で送信するときに使用されます。後で詳しく検討します。
  • リフレクションライブラリのバインディング-エディターをデータに自動的に表示するために使用されます。エディターの章で議論されます。
  • エンティティを複製するためのコード-エディターとゲームでエンティティを複製するために使用されます。
  • 軽量ランタイムリフレクションのコード。

C ++の解析


ヘッダーファイルを解析するためのほとんどすべてのソリューションは、clangを使用したコードの解析につながりました。しかし、実験の結果、そのような解決策の速度は私たちにはまったく合わないことが明らかになりました。さらに、clangが提供するパワーは、私たちには必要ありませんでした。

したがって、別の解決策が見つかりました：CppHeaderParser。これは、ヘッダーファイルを読み取ることができるpython単一ファイルライブラリです。これは非常に原始的で、＃includeに従わず、マクロをスキップし、文字を解析せず、非常に高速に動作します。 

今日まで使用していますが、バグを修正して機能を拡張するために、かなりの量の編集を行う必要がありました。特に、C ++ 17からのイノベーションのサポートが追加されました。

コード生成のステータスの不確実性に関する誤解を避けたかったのです。したがって、生成は完全に自動的に行われるべきであることが決定されました。CMakeを使用します。この場合、生成はコンパイルごとに開始されます（依存関係が変更された場合にのみ生成を開始するように構成することはできませんでした）。これに時間がかからず、煩わしくないように、すべてのファイルとディレクトリの内容を解析した結果をキャッシュに保存します。その結果、コード生成のアイドル開始には数秒しかかかりません。

コードジェネレーター


世代によって、すべてが簡単になります。テンプレートから何かを生成するためのライブラリはたくさんあります。Templite +を選択しました。非常に小さく、必要な機能があり、適切に動作します。

生成には2つのアプローチがありました。最初のバージョンには多くの条件、チェック、およびその他のコードが含まれていたため、テンプレート自体は最小限であり、生成されたロジックとテキストのほとんどはPythonコードでした。 Pythonコードではテンプレートよりも書く方が便利であり、任意のトリッキーなロジックを簡単に台無しにする可能性があるため、これは便利でした。しかし、これもひどいものでした。C++コードの膨大な数の行と混合されたpythonコードは、読み取りまたは書き込みに不便だったからです。使用されたpython-generatorsは状況を簡素化しましたが、全体として問題を排除しませんでした。

したがって、現在のバージョンの生成はテンプレートに基づいており、Pythonコードは必要なデータを準備するだけで、見栄えがよくなりました。

連載


シリアライゼーションでは、protobuf、FlexBuffers、cerealなどのさまざまなライブラリが検討されました。

手書きの構造があり、生成された構造をユーザーコードに統合する方法がないため、コード生成ライブラリ（Protobuf、FlatBuffersなど）は適合しませんでした。また、シリアル化のためだけにクラスの数を2倍にするのは、もったいないです。  シリアル

ライブラリが最適な候補のようです。構文が素晴らしく、実装が明確で、シリアル化コードを生成するのに便利です。ただし、他のほとんどのライブラリの形式と同様に、そのバイナリ形式は適していませんでした。重要なフォーマット要件はハードウェアからの独立性であり（データはバイト順やビット深度に関係なく読み取る必要があります）、バイナリフォーマットはPythonからの書き込みに便利です。

データをテキストビューからバイナリビューに変換するプラットフォームやプロジェクトに依存しないユニバーサルスクリプトが必要だったため、Pythonからバイナリファイルを書き込むことが重要でした。したがって、非常に便利なシリアル化ツールであることが判明したスクリプトを作成しました。

主なアイデアは穀物から得られたもので、データの読み取りと書き込みのための基本的なアーカイブに基づいています。それらから、レコードを異なる形式（xml、json、binary）で実装する異なる相続人が作成されます。また、シリアル化コードはクラスによって生成され、これらのアーカイブを使用してデータを書き込みます。



編集者


エディターにはImGuiライブラリーを使用します。その上で、すべてのメインエディターウィンドウ（シーンコンテンツ、ファイルおよびアセットビューアー、アセットインスペクター、アニメーションエディターなど）を作成しました。エディター

のメインコードは手動で記述されていますが、使用する特定のクラスのプロパティを表示および編集しますrttrライブラリ、そのために生成されたバインディング、およびrttrで機能する一般化されたインスペクターコード。

リフレクションライブラリ-rttr


C ++でリフレクションを整理するために、rttrライブラリが選択されました。クラス自体への介入を必要とせず、便利で理解可能なAPIを備え、ラッパーを登録する機能を備えたタイプ（スマートポインターなど）のコレクションとラッパーをサポートし、必要なことをすべて実行できるようにします（タイプの作成、クラスメンバーの反復、プロパティの変更、メソッドの呼び出しなど）。 

また、通常のフィールドと同様にポインターを操作でき、nullオブジェクトパターンを使用するため、操作が大幅に簡略化されます。

ライブラリのマイナス点は、かさばり、それほど高速ではないことです。そのため、エディターにのみ使用します。たとえばアニメーションシステムなど、オブジェクトのパラメーターを操作するためのゲームコードでは、独自のプロダクションの最も単純なリフレクションライブラリを使用します。

rttrライブラリでは、クラスのすべてのメソッドとプロパティの宣言を使用してバインディングを作成する必要があります。このバインディングは、編集サポートを必要とするすべてのクラスのPythonコードから生成されます。そして、rttrが任意のエンティティのメタデータを追加できるという事実により、コードジェネレーターは、クラスメンバーにさまざまな設定を設定できます：ツールチップ、数値フィールドの許容値境界のパラメーター、フィールドの特別なインスペクターなど。これらのメタデータは、編集インターフェイスを表示するためにインスペクターで使用されます。

→rttrでクラスを宣言するためのサンプルコードはここにあります

検査官


エディター自体のコードがrttrで直接機能することはほとんどありません。最も一般的に使用されるレイヤーは、オブジェクトがそのためにImGuiインスペクターを描画できることです。これは、rttrからのデータを処理し、ImGuiコントロールを描画する手書きのコードです。

データ編集インターフェースの表示をカスタマイズするには、rttrへの登録時に指定されたメタデータを使用します。私たちはすべてのプリミティブ型、コレクションをサポートしています。値とポインタによって格納されるオブジェクトを作成することが可能です。クラスメンバーが基本クラスへのポインターである場合、作成時に特定の子孫を選択できます。

インスペクターのコードは、操作のキャンセルのサポートも想定しています。値を変更すると、データを変更するコマンドが作成され、ロールバックできます。 

これまでのところ、アトミックな変更を表示および保存する機能を備えた決定を行うシステムはありません。つまり、オブジェクトの変更されたプロパティをシーンに保存し、プレハブを読み込んだ後にこれらの変更を適用することはサポートされていません。また、オブジェクトのプロパティを変更するときに、アニメーショントラックが自動的に作成されることはありません。

ウィンドウとエディター


現時点では、エディター、コード生成、アセット作成システムに基づいて、さまざまなサブシステムとエディターが作成されています。

  • ゲームインターフェイスのシステムは、柔軟で便利なレイアウトを提供し、必要なすべてのインターフェイス要素を含みます。ウィンドウの動作を視覚的にスクリプト化するシステムが彼女のために作られました。
  • アニメーションの状態を切り替えるためのシステムは、Unityのアニメーションの状態エディターに似ていますが、動作原理によって多少異なり、より幅広いアプリケーションがあります。 
  • クエストとイベントのデザイナーは、プログラマーの参加なしに、ゲームイベント、クエスト、チュートリアルを柔軟にカスタマイズできます。 

これらすべてのサブシステムとエディターを開発するとき、私たちはUnityとUnreal Engineを注意深く調べ、それらから最大限を引き出そうとしました。そして、これらのサブシステムのいくつかはゲームプロジェクトの側で作られています。

要約する


結論として、開発がどのように行われたかを説明したいと思います。最初の作業バージョンは、わずか2か月で数人が作成していくつかのゲームプロジェクトに統合しました。コード生成はまだ行われておらず、現在は豊富なエディターを備えています。同時に、それは前向きの動きが始まった作業バージョンでした。当時、これはエンジン開発の主要なベクトルに対応しているとは言えませんでした。すべてが数人の熱意と私たちの活動の必要性と正確さに対する明確な理解に基づいていました。

その後の開発はすべて、非常に積極的かつ進化的に段階的に行われましたが、常にゲームプロジェクトの利益を考慮に入れています。現在、10人以上の人が「私たちの小さなUnity」の開発に取り組んでおり、新しいバージョンの開発は、最初の頃ほど速くありません。

それにもかかわらず、私たちはほんの数年で素晴らしい結果を達成し、止まることはありません。自分にとって、そして会社全体にとって正しい、重要であるとあなたが考えていることに前進してほしいと思います。
