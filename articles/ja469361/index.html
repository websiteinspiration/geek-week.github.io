スヴェルト3の痛みと涙

序文の代わりに


この記事は、私たちがライブプロジェクトでSvelteを試すことにしたように、役に立ちます。私たちの小さな会社は、Mongodb Stitchのバックエンドを備えたサービスのWeb管理者を開発するための注文を受けました。ここ数年、フロントエンドは（プロジェクトのサイズとReactNativeが必要かどうかに応じて）ReactまたはVueで作成しますが、Svelteの美しさについて聞いた後、それが良いかどうかを自分で理解することを試みることにしました。VueやReactの代わりに使用する必要があるかもしれません。

スヴェルトとは誰ですか？




一言で言えば-これは新しいjsフレームワークです（しかし、彼は自分自身をそのように考えていません）。これは、ReactとVueのキラーであり、何とか何とか何とかしています...実際のプロジェクトでの使用の便利さのビュー。
私たちは車を製造せず、運転します。そして、私たちには悪の顧客がいます。

チュートリアル


新しい言語またはフレームワークについて最初に学習するのは、サイトのチュートリアルです。Svelteでは、これはSvelte.dev/Tutorialです。

残念ながら、一部のチュートリアルはプレビューのサイトでは機能しませんが、実際に機能するかどうかは完全には明らかではありません。チュートリアルが古くなっているか、SvelteのWebサイトで問題が発生したのかがはっきりしないため、時間をかけて自分で手動で確認する必要があります。本当に時間がかかります。

のみが見つかりましたが、現時点では非常に未加工に見えました。そこから 単純にインポートすると、アセンブリ中にエラーがパッケージから抜け、UIキットのエラーを解決する時間がありませんでした。 ドキュメントをすばやく確認したので、CDNからのバニラのインポートを支持してそれを放棄することにしました。 プロジェクトでMaterial UIを使用したかったのは、管理パネルがモバイルデバイスからも使用される可能性があり、Materialがこれに非常に適しているためです（私の意見では、デスクトップではさらに悪く見えます）。

様式


スタイルはすべて非常に明確であり、Vueのようにすべてのスタイルを詰め込みます。あなたはスタイルを書き、すべてがうまくいき、次に幅と高さなどの小道具パラメーターを受け取る必要のあるUIコンポーネント（UIKitがないため）を書き、論理的には次のようにします。

<script>
export let width = '20px';
export let height = '20px';
</script>
<style>
.loader {
	width: { width },
	height: { height },
}
</style>

そして...いいえ、このスタイルでは変数を挿入できません。「ReactWay」を使用してこの状況から抜け出し、「スクリプト」でスタイルを返す変数または関数として動的なスタイルを作成できます。

その結果、スクリプトのスタイル部分にスタイルの一部であるお粥が再びあります。純粋なコードは、スタイルにパラメーターがない場合、または動的パラメーターのみがあり、それらがスクリプト内にある場合にのみ取得されます。

ルーティングとルーター


通常のルーターがなければ、1ページのアプリを作成することもできません。

ルーターを自分で選択する必要があります。非常によく似たソリューション間でも選択できます。実際、これまでのところgithub.com/EmilTholin/svelte-routingとgithub.com/kazzkiq/sveroのルーターは 2つしかありません。

どちらも非常に似ていますが、最初の星を選びました。星の数で選びました（はい、私はひどい人です）。

間違い


単にバンドルを作成できず、エラーについて強く叫ぶReactとは異なり、Svelteはすべてのエラーを完全に収集します。同時に、何かを忘れた場合は、npmのインストールまたはエクスポートをインストールすると、すべてが正常になり、白い画面が表示されます（ちなみに、これはReactの最初のバージョンの場合です）。

ReactとVueでは、ビルドまたはlintの段階でこのようなエラーを確認していました。

その結果、2つの場所でエラーを検出する必要があります。公平に言えば、これはReactとVueにありますが、それほど頻繁ではなく、そのような明らかなエラーは糸くずの段階で捕捉されます。

Svelteは命名規則が必要です


この問題はかなり頻繁に発生します-命名規則がないためです。

たとえば、ルーターに戻ります。

そのようなインポートがあります：

    import { Router, link, Link ,Route } from "svelte-routing";

このリンクとリンクは何ですか？私のようなこのような小さなプロジェクトでも、Svelteで使用されているプロジェクトでの命名規則の欠如にはすでに混乱がありました。

もちろん、ReactManの目には次のようになっているはずです。

 
import { Router, useLink, LinkComponent ,RouteComponent } from "svelte-routing";

Svelteにはreactのようにrefの類似があります。これはdom要素またはコンポーネントへの参照です。
公式チュートリアルでは、次のようにするように指示されています。

<script>
let dialog;
</script>
<dialog bind:this={dialog} class="mdl-dialog» />

コンポーネントに15個の変数がある場合はどうでしょうか。

<script>
let loading;
let pins;
let dialog;
let stitch;
</script>
<dialog bind:this={dialog} class="mdl-dialog» />

これらのどれが実際に変数であり、どこにコンポーネントまたは要素へのリンクがあるか。
チュートリアルを変更して、初心者と変数の規則の観点から例をより正確にします（良い例については、Reactチュートリアルを参照してください）。あなたはビネグレット変数をやらせる場合ならば、それはいいだろうリンターやバンドラ、このようなAのビネグレットソースで誓ったが、例えばとして、これはgolangによって行われました。

<script>
let dialogElement;
</script>
<dialog bind:this={dialogElement} class="mdl-dialog» />

コンポーネント関数で一部のデータを受け取り、このデータをViewに送信する必要がある場合の「ライフ」のもう1つの例は、いくつかの変数がUI要素の一部にねじ込まれている場合です。そして、それは私たち（またはむしろ、私たちのプログラマー）と次のように見えます：

  
async function ReloadPins() {
        loading =  true;
        pins = await getAllPins();
        status = 0;
    }

私は、そのコードを開いたとき、それはすぐにここに変数であるように私には「明確になった」状態 1は、私たちのに取り付けられているUI。

もちろん冗談ですが、何もはっきりしていません。さらに、状態は明確でもなく、単なる補助変数でもあります。Reactでは、これは状態によって決定され、少なくとも何らかの形で明確になります。

どこで誰がそれをしたのか？


Svelteコンポーネント内の変数はグローバルであり、どこでも変化します（こんにちはAngular）。
しかし、良いニュースがあります。それらは1つのコンポーネント内で（Angularの間）グローバルであるため、このすべての変更は、1つのコンポーネント内ではそれほど怖いものではありません。

同時に、コードの「美しさ」を節約し、状態を変数にするために何が必要かについて考えるのではなく、関数の内部に何を残すべきかを考えます（こんにちはReact）-すべてとどこにでもあります。



その結果、コンポーネントをすばやく作成できますが、ジュニアがすでに「忘れた」ときに、この「サラダ」でジュニアが何を書いたかを理解するのは非常に難しい場合があります（時には名前も）。このため、プロジェクトまたはチーム内でコードを明確に編成しないと、内部が何であるかを理解することはできません。

<script src="https://s3.amazonaws.com/stitch-sdks/js/bundles/4.4.0/stitch.js"></script>

使用しているとき以来NPMと、通常のインポートパッケージを、バッファエラーの問題が開始されます。
