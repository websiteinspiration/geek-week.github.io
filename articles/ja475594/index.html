面白いJavaScript：ほぼ線形の方程式

素晴らしい数学（つまり、線形方程式）と同様に素晴らしいJavaScriptを受け取り、一方を他方に重ね合わせたらどうなるでしょうか。js-environmentの制限と詳細の条件では、単純な数学的問題が非常に好奇心が強く、js-stoneの落とし穴に満ちている可能性があります。モスクワで開催された前回のHolyJS 19カンファレンスでは、そのような線形方程式（SEMrushからの他のタスクの中でも特に）が激動しました。



そして、はい、これもまた、Entertainment JavaScriptの見出しです。私は、気になっているすべての人をカットするように求めます。

前のタスク




もちろん、以下に説明されているすべてのこと-これは楽しいことのために2つの素晴らしいことを共生させるための軽薄な試みです-真剣に受け取られるべきではありません。そして、この資料は、それが会議の参加者の熱心な関心のためでなければ、特別な感謝を込めて存在しなかったでしょう。

言い回し


1.方程式のすべての整数解を見つけます：

9 +~ x >> 6 / 3 = -x / 3

2.方程式のすべての整数解を見つけます：

9 +~ x >> 6 / 3 = -x / 3 | 0

2番目の方程式は、右側の追加の演算のみが最初の方程式と異なります。

数学的近似


最初の方程式に移ります。まず、表に従って、使用される操作の優先順位を理解します。

(9 +(~ x)) >> (6 / 3) = -x / 3

x からビットごとの否定を取り、それを9に加算します。加算の結果は、6を3で除算した結果に等しいビット数だけビットごとに右にシフトされます。

明らかに、問題は、目的のxでのビットごとの演算の使用にあります。しかし、さらに推論するための条件付きの根を見つけるためには、方程式を近似の数学的な類似体にすることを試みる価値があります。

ビット演算は、オペランドを符号付き32ビット整数として扱います。ビット単位のNOTは、インクリメントからの整数否定と置き換えることができます。

(9 -(x + 1)) >> (6 / 3) = -x / 3
(8 - x) >> 2 = -x / 3

右へのビット単位のシフト（符号を保持）は、右側のオペランドと等しい次数での2 による整数除算で置き換えることができます。

(8 - x) / 2^2 = -x / 3
(8 - x) / 4 = -x / 3

もちろん、これらの置き換えは非常に恣意的であり、これについては後で説明します。これで通常の線形方程式が得られました。その唯一の根は-24です。元の方程式の左辺と右辺の値を代入します。

9 +~ (-24) >> 6 / 3; // = 8
-(-24) / 3; // = 8

両方の部分は同じです。つまり、すべてがそれほど絶望的ではなく、-24が本当に解決策です。

怠惰を検索


関数f1（x）=（8 -x）/ 4およびf2（x）= -x / 3のグラフを描くと、もちろんx = -24で2本の線の唯一の交点がわかります。



ただし、左側の式でビットごとの演算を使用して、等しくない置換をいくつか行ったため、実際には関数f1のグラフは少し異なります。任意のxの場合、関数の値は、実線f1の値とは異なる整数で、-1から1の範囲でシフトする可能性があります。これは、ソリューション検索領域を-24の左と右に制限できることを意味します。ここで、関数f1とf2の値複数の違いが出始めます。

検索領域の境界を見つけるには、1）モジュールで不等式を解決するか、2）関数のグラフを詳しく調べます。xが間隔[-36、-12]を調べる価値があることがわかります。

| (8 - x) / 4 + x / 3 | <= 1



いくつかの閉じた範囲[beg、end]で解全体を反復処理するには、findx関数を記述します。

const findx = (f, beg, end) => 
    [...Array(end - beg + 1)].map((_, i) => i + beg).filter(f);

この関数は、渡された関数fの値がtrueに削減された整数の配列を返します。解を見つけるために、等式演算子を使用して方程式をjs関数として表します。

let eq1 = x => 9 +~ x >> 6 / 3 == -x / 3;
findx(eq1, -36, -12); // [-24, -21, -18, -15]

したがって、x = [-24、-21、-18、-15]は、最初の方程式の望ましい解です。

グラフィカルなソリューション


列挙はもちろん成功ですが、関数f1のグラフを最後まで理解して、方程式をグラフィカルに解いてみましょう。さらに、ソリューションはブラウザコンソールの必須の所有権を意味しません。

ビットごとのNOT演算子は小数部分を単に破棄するだけなので、結果-（x + 1）は切り捨てられます。ビットシフト演算子はもう少し複雑です。これを整数除算に置き換えましたが、被除数の符号に応じて、この演算は結果を切り捨てまたは切り上げます。

10 >> 2; // = 2
-10 >> 2; // = -3

ただし、目的のxが[-36、-12]の範囲にあることはわかっています。したがって、ビット単位のシフトの左オペランド（8 -x）は[20、44]の範囲にあります。つまり、常に正です。これは、切り捨てによる整数除算を意味します。

業務の性質を考え出した、私たちは関数の正しいグラフ描画することができF1を：



私たちは、同じ座標の関数の4つの交点を見つけるのx = [-24、-21、-18、-15]。

2番目の方程式


したがって、2番目の方程式に到達しました。

9 +~ x >> 6 / 3 = -x / 3 | 0

最初とはビットごとのORが追加されています。そのような演算の右のオペランドがゼロの場合、結果は小数部分が破棄された左のオペランドの値になります。

はじめに、同じ検索を行ってみましょう。検索領域を決定します。現在、関数f2はf1と同様の特性を持っているため、信頼性のために、可能なシフトを合計し、関数の絶対値が2単位以上異なる場合に検索を制限する必要があります。これは[-48、0]です。

let eq2 = x => 9 +~ x >> 6 / 3 == -x / 3 | 0;
findx(eq2, -48, 0); // [-24, -21, -18, -15]

そして、同じ答えを得ました。結局のところ、何かがおかしいのではないかという疑惑があります。しかし、実際には、元の方程式をそのようなjs関数として表していたため、等価演算子を介して2つの式（左と右）を1つに結合しました。また、等価演算子には、ビットごとのORの優先度よりも高い優先度があります。関数は次と同じです。

x => (9 +~ x >> 6 / 3 == -x / 3) | 0;

この場合、true |であるため、ビット単位のORは効果がありません。0 = 1。これを回避するには、2つの部分式の結果を比較していることを関数本体で明示的に示す必要があります。

let eq2 = x => (9 +~ x >> 6 / 3) == (-x / 3 | 0);
findx(eq2, -48, 0); // [-32, -29, -28, -26, -25, -24, -23, -22, -21, -19, -18, -15]

関数グラフのプロットが重なるが、我々は唯一の整数値を有する点に興味を持っているX座標[-32、-29、-28、-26、-25、-24 、-23、-22、-21、-19、-18、-15]、合計12のソリューション。必要に応じて、2つの「はしご」とステップ3および4の交点をアルゴリズムで見つけることができます。

追加の質問


会議で提案された問題では、式2の最小解を見つける必要があるという追加の質問がありました。これは必ずしも整数であるとは言えなかったため、x = -32 の答えは正しくないことがわかりました。

力ずくで解決策を見つけることはここでは機能しませんが、グラフィカルに解決することは非常に簡単です。これは、右側の-33に最も近いx値



です。x = -32。（9）のようです。しかし、これはまだ真実ではありません。私たちの環境はJavaScriptであるため、数値の表現では、使用されるデータ型によって制限されます。タイプ番号はfloat64、倍精度浮動小数点数（IEEE 754）です。これを覚えて、おおよその正確さを挙げれば、豪華なキツネを手に入れるのに十分でした！

ビット演算のダークサイド


上記のように、ビットごとの演算は、オペランドをシーケンス0および1で表される32ビットの数値に変換します。これは、範囲[-2147483648、2147483647]です。数値がそれに適合しない場合、最上位ビットは破棄されます。

最初の方程式では、右側にビット単位の演算がないため、これは何の役割も果たしません。しかし、2番目に、この数値の変換は興味深い効果をもたらします。

たとえば、数値-24は次のように表されます。

11111111111111111111111111101000

数値の負の値は、正の数値のレコードのビットを反転（ビットごとのNOT）して、1を加算したものです。

この32ビットシーケンスで終了する変換後の範囲外の数値は、バイナリ演算では数値-24と同じになります。たとえば、これらは数値です。

4294967272 | 0; // -24
8589934568 | 0; // -24, prepend '1'
12884901864 | 0; // -24, prepend '10'
17179869160 | 0; // -24, prepend '11'
21474836456 | 0; // -24, prepend '100'
// ...

方程式の右側で、ビットごとの演算の前に、xを3で除算します。「等価物」-24からxを見つけます。これは、完全に3で割り切れます。そのような最も近い数は12884901864です。方程式で代入してください。

9 +~ 12884901864 >> 6 / 3 = -12884901864 / 3 | 0
9 +~ 12884901864 >> 2 = -4294967288 | 0
9 + 23 >> 2 = 8
8 = 8

3で割った結果（-4294967288）は、割り当てられた32桁に収まりません。ビットを反転すると、符号は最終的に失われ、残りは8つだけになります。

00000000000000000000000000001000

さらに、eq2関数を呼び出して、結果が正しいかどうかを確認できます。

eq2(12884901864); // true

考えてみると、この根の隣に残りの11の整数解の射影があります。

したがって、膨大な数の新しいソリューションが表示され、最も近い正の-24のみが考慮されます。それにもかかわらず、これは主要なタスクほど面白くなく、参加者からの決定を分析するとき、そのような非常にまれな回答が個別に評価されました。混乱しないようにするために、必要な整数に対する制限を、符号付き32ビット整数として問題の状態に導入できます。

そして、あなたは作ることができません！次に、最小のルートを見つけるには、負の符号のあるNumber.MAX_SAFE_INTEGERの近傍に注意を払う必要があります。これは、この数値が整数で、極端な精度のfloat64であるためです。さて、それではご自分で。
