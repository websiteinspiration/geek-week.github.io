Xbox 360の保護とハッキング（パート1）

Xbox 360ゲームコンソールについて聞いたことがあると思いますが、それは「点滅」しているということです。ここで「ファームウェア」とは、ゲームのコピーと独自仕様のソフトウェアを起動するための組み込み保護メカニズムのバイパスを意味します。そしてここで質問が発生します！どのようなメカニズム、どうやって回避するのですか？開発者は何をしましたか、どのようにして回避しましたか？実際、このトピックは特にXbox 360にとって非常に広範囲で興味深いものです。ここでは、ソフトウェア、ハードウェアの欠陥、および絶対的な魔法の脆弱性を見つけることができます。面白い？お立ち寄りください！最初の部分では、ハイパーバイザー、ドライブ、およびファームウェアを紹介します...


主題に会う


Xbox 360ゲームコンソールは2005年にリリースされ、それ以来鉄の特性に変更はありません。それがリリースされた間ずっと、彼らは同じでした：

  • 3.2 GHz PowerPC CPU、/ 3コア
  • 500 MHz GPU
  • 512 MBのRAM
  • SATA DVD-ROM
  • SATA HDD（オプション）

はい、設計は時間とともに変化し、ナノメートルは減少しました：


それにもかかわらず、すべてのゲームはコンソールのすべての「改訂版」で同等にうまく機能しました。これは、2005年の機器で最新のゲームを実行できる場合にまさに当てはまります。

リリースの時点で、コンソールは可能な限り安全に開発されたという大声が出されました。ハードウェアレベルで保護されたカスタムチップであり、一般に、ハッカーはまだこれを見ていません。
このボックスには、ハッカーコミュニティがこれまでに見たことのないセキュリティレベルがあります。

開発者は何を考えましたか？

まず、システムのプログラムコードを取得できないようにすべてを行いました。プライマリローダー（1BL）を含む32 KB ROMと、それが実行される64 KB SRAMが中央プロセッサに組み込まれました。CPUチップからROMの内容を取得することは非常に困難です。


次に、特別なヒューズが同じプロセッサに組み込まれました-書き込み可能な（文字通り、高電圧）ジャンパ。含まれるヒューズ：

  • JTAGインターフェイスロックビット
  • プレフィックスの目的を決定するビット（retail / devkit / testkit）
  • 固有の128ビットプロセッサキー
  • ダウングレードを無効にするロックダウン値（LDV）カウンター


はい、融合の量は限られています。コンソールを80回続けて更新した場合、CFLDVカウンターは終了します...わかりません。これを試みたことはありません。おそらく、プレフィックスは更新されなくなります。

3番目に、開発者は信頼の連鎖を実装しました。ブートローダーの信頼性を検証するために、最新の（当時）SHA-1アルゴリズムとRSA-2048アルゴリズムの組み合わせが使用され、コンソールからすべてのキーを取得してシステムを再構築できたとしても、独自のコードを起動したり、ブートローダーを不正に変更したりする可能性を排除しました。


第4に、開発者は「誰も信用しない」という原則にさらに踏み込み、RAMを保護するための特別なハードウェアモジュールを同じ残念なCPUに組み込むことを決定しました。その助けにより、プログラムコードのあるすべての領域が暗号化され、最も重要な領域（ハイパーバイザー）の整合性監視がオンになりました！


これにより、開発者は、RAMにアクセスできる外部デバイスを介して、RAM内のシステムプログラムコードを変更するDMA攻撃から身を守りました。

最後に、ハイパーバイザーは権利のメモリ領域への区別を扱いました。もちろん、デジタル署名をチェックする前に、彼だけがページを実行可能にすることができました。そのため、署名されていないコードをダウンロードしたり、データ領域から何かを実行したりすることができませんでした。 。

その結果、Xbox 360オペレーティングシステムは十分に保護されたため、DVD-ROMが最初の攻撃手段として選択されました。

...バックアップを開始します！


Xbox 360では、ゲームのメインメディアとして2層DVDが選択されました。当然、防御メカニズムもここにありました：

  • DVD-ROMとのデータ交換は一意のキーで暗号化されました
  • ディスクの最初には、ライセンスを確認するための特別な「セキュリティセクター」がありました。
  • ディスク上の実行可能ファイルはデジタル署名されています

しかし、DVDドライブの保護には、メインシステムよりも注意が払われていませんでした。ゲームコンソールは、DVDドライブに過度の信頼を示しました。ディスクのライセンスを決定したのはDVD-ROMでした。さらに、ファームウェアは外部メモリに保存され、プログラマが読み取ることができました。


その結果、2006年5月14日に、commodore4eva（c4eva）は、TS-H943モデルを駆動するために最初に修正ファームウェアをリリースしました：

軸を壊す！


ストーリーの最初の部分から思い出されるように、Xbox 360システムには、すべてを制御するハイパーバイザーがありました。だからここに。システムの1つのバージョンに、脆弱性が突然現れました！研究者がハイパーバイザのコードサンプルをどのように正確に受け取ったかは、私にははっきりとはわかりません。しかし、実際は-2006年の終わりに、研究者はXbox 360で署名されていないコードを起動し、2007年の初めに開発者によって脆弱性が修正されました。
Timeline:
Oct 31, 2006 — release of 4532 kernel, which is the first version containing the bug
Nov 16, 2006 — proof of concept completed; unsigned code running in hypervisor context
Nov 30, 2006 — release of 4548 kernel, bug still not fixed
Dec 15, 2006 — first attempt to contact vendor to report bug
Dec 30, 2006 — public demonstration
Jan 03, 2007 — vendor contact established, full details disclosed
Jan 09, 2007 — vendor releases patch
Feb 28, 2007 — full public release

ハイパーバイザーには1つの機能がありました。残りのコードとは異なり、ハイパーバイザーは仮想アドレス空間ではなく、物理（リアルモード）で実行されました。ブロードキャストは使用されず、呼び出しは直接行われました（0x00000000'xxxxxx形式のアドレス）。これは、速度を上げるため、または簡単にするために行われました...そして、Xbox 360のアドレススペースの1つの機能があり

、メモリへのアクセスモードは、その物理アドレスによって決まります。つまり、アドレスの最上位ビットには公式の目的がありました。たとえば、アドレス0x00000 0 00'0000201Cはアドレス0x201Cへの直接アクセスを意味し、0x00000 1 00'0000201Cは同じ物理アドレス0x201Cを読み取るときにオンザフライで整合性を復号化および検証する必要があることを意味しました。


したがって、暗号化と保護を有効にして実行するには、0x00000 1 00'xxxxxxxxのようなアドレスを参照する必要があります。その後、ハードウェアモジュールに保護メカニズムが組み込まれました。したがって、ハードウェアレベルでは、目的のビットが自動的に追加されました（特別なHRMORレジスタがこれを担当しました-ハイパーバイザーリアルモードオフセットレジスタ）。

もう一度-ハイパーバイザーが0x00000 0 00'xxxxxxxxのようなアドレスにアクセスするとすぐに、MMUは自動的にこのアドレスを0x00000 1 00'xxxxxxxxに変更します（暗号化と保護を含む）。したがって、保護と暗号化なしで物理メモリから「直接」コードを実行しようとすると、失敗する運命にありますか？

ハイパーバイザーバージョン4532の脆弱なコードを見てみましょう。
// %r0 –
13D8: cmplwi %r0, 0x61 // ID
13DC: bge illegal_syscall // 0x61, ,
...
13F0: rldicr %r1, %r0, 2, 61 // 4
13F4: lwz %r4, syscall_table(%r1) //
13F8: mtlr %r4 // lr
...
1414: blrl //

ホリネズミを見た？しかし、彼はそうです！ Cmplwi命令は32ビット値で動作しますが、rldicr- 64 ビット値で動作します！つまり、システムコール番号として値0x20000000'0000002Aを渡すと、テストに合格します（32ビットの下位部分が0x61より小さいため）。その結果、アドレス0x10ECの代わりに、ハンドラーのアドレスが0x80000000'000010ECから取得されます。

そして、彼らが言うように、あなたの手を見てください。アドレスの最上位ビットはそれぞれゼロではなく、HRMORは追加されません！また、実際のアドレス空間は32ビットであるため、私たちが設定した63ビットは無視されます。間違ったシステムコール番号を送信するだけで、ハイパーバイザーを暗号化されていない安全でないメモリにリダイレクトしました。


しかし、待って、ジャンプする場所があるようにするには、物理​​メモリにデータを書き込むことができる必要があります。これを達成する方法は？

ここで、2番目の要素が出てきます。Xbox360のGPUは、非常にスマートでさえありました。彼は、ジオメトリデータを物理メモリにアップロードするための特別なシェーダー命令MemExportをサポートしました。つまり、シェーダーをコンパイルしてGPUで実行し、どこにでも何でも記録できます。そして最も重要なことに、シェーダーは署名されていません。ゲームディスクを何らかの方法で変更すると、シェーダーコードを簡単に置き換えることができます。


一つの質問が残った。ゲームディスクのシェーダーコードを置き換える方法は？ここで、コンソールのDVDドライブをハッキングすることは非常に役に立ちました。シェーダーを作成し、ゲームのイメージで置き換え、ディスクに記録してLinuxを起動しました。

はい、それを実行するには、毎回ゲームを開始し、エクスプロイトが機能するのを待ち、Linuxでブートディスクを置く必要がありましたが、それは少なくとも何かでした！

すでに述べたように、Microsoftは、ハイパーバイザーの脆弱性を修正したシステムアップデートをリリースしました。しかし、脆弱なカーネルを返すとどうなりますか？

ダウングレード！


一般に、Xbox 360のアーキテクチャには、ダウングレードからの保護メカニズムが定められていました。プロセッサのハードウェアヒューズでは、更新中にビットが書き込まれるたびに（ロックダウン値、LDVが増加）、このLDVがヒューズとシステムで一致しない場合、コンソールは単純に起動しませんでした。

Xbox 360ブートローダーの構造、つまり「ブートローダーセクション」について考えてみましょう。


イメージにはいくつかのブートローダーのセットがあり、それぞれがいくつかのLDVに対応していることがわかります。一例では、これらは1888を LDV 0、のために16767 LDV 3および16756 LDV 2

だから、それ自体が単にセクション6BL / 7BLに記録されたシステムのすべてのアップデートベースの「コア」5BLにパッチとして「適用」1888。しかし、適用するパッチのセットは、ヒューズとブートローダーヘッダーの規定されたLDVに従って選択されました！そして、ちょうど5BLで、1つの大きなBUTでヘッダーを変更できます-ヘッダーは、そこに記録されたHMAC-SHA-1ハッシュ合計に対してチェックされました。そして、それは普通のmemcmpによってチェックされました。

それでも状況がわからない場合は、ここでタイムアタック（タイミングアタック）を適用できました。標準のmemcmp関数は、最初の不一致の直後に比較を完了します。したがって、ハッシュの最初のバイトを変更してmemcmpの実行時間を検出することで、目的の値を選択できます（検証時間はそれに伴って増加します）。さらに続けて、ハッシュ合計のすべてのバイトを取得できます。

測定には、POST_OUTデバッグバスを使用しました。これはPCのように動作し、1バイトの値をロードするさまざまなタイミングで表示されます。これにより、プロセッサが現在実行されている場所と発生したエラーを判断できます。実際、これらはマザーボード上の8つのポイントであり、それぞれが特定の値のビットを担当します。


これらのポイントにはんだ付けしたら、各ダウンロードステージの実行時間を測定し、エラーが発生したかどうかを確認できます。

ハッシュ選択のプロセス全体には約1時間かかります。


その結果、パッチが適用されておらず、最も古いバージョンの1888システムが起動されているため、コアカーネルに現在のLDVがインストールされているイメージが取得されます。脆弱なバージョン4532にアップグレードすることがすでに可能な場所から：



もちろん、Microsoftは最初に更新されたブートローダー（2BL、「CB」）を更新し、CBLDVヒューズを焼くことによってこの脆弱性を修正し、ダウングレードを再び不可能にしました。現在はmemcmpの代わりに、入力値に関係なく同じランタイムを使用した安全なバージョンが使用されていました。

JTAGハック！


しかし、ここでは、研究者たちはあきらめず、抜け穴を発見しました。はい、開発者が想像もできなかったようなものです。

コンソールの通常の動作モードでは、すべてのブートローダーがチェーンで相互に接続されています。その結果、各ブートローダーの復号化は、前のブートローダー（ペアリングデータ）のヘッダーのデータ領域に依存します。さらに、コードの暗号化は一意のプロセッサキーに依存するため、CPU_Keyを知らなければ、作業イメージを取得してアセンブルすることはできません。それとも可能ですか？

コンソールの製造段階（プロセッサキーがまだヒューズに書き込まれていない場合）では、ペアリングデータがゼロ（ゼロペアリング）のときに特別なXbox 360起動モードが使用されます。そして、そのようなイメージは（脆弱なカーネルを使って！）プロセッサキーさえ知らなくても、どのコンソールでも実行できます！
残念ながら、本格的な打ち上げはありません。これがエラーです。


つまり、キングコングゲームを開始できず、シェーダーによるエクスプロイトをアクティブ化できません...しかし、脆弱なカーネルがすでに開始されています！RAMを記録する別の方法があるのでしょうか？あることがわかりました。

まず、コンソールのサウスコンソールの3つの空きGPIOをGPU JTAGピンにはんだ付けします。


次に、サウスブリッジのファームウェアを変更し（暗号化されていますが、署名はされていません）、脆弱なカーネルでイメージを収集します。その後、魔法が起こります：

  • 起動時に、JTAG GPUのサウスブリッジがPCI Expressに登り、NANDコントローラーを構成します
  • これで、読み取り時に、NANDコントローラーは必要なメモリ領域にデータを書き込みます
  • システムのコアが起動し、システムが起動したことをサウスブリッジに通知します
  • サウスブリッジはNANDコントローラーを引き出し、RAMを適切な場所に上書きし、ハイパーバイザーの脆弱性を利用します！
  • コントロールがコードに転送され、必要な処理を実行します

要するに、彼らはキングコングシェーダーエクスプロイトと同じようにすべてを行いましたが、クールです-ゲームを開始してディスクを交換する必要はありません。

JTAG Hackに基づいて、システムの修正バージョンが作成されました-XBReboot、Freeboot、署名の検証が無効になっており、海賊はすでに歩き回っていました。ゲームは、USBドライブやディスクからだけでなく、コンピューターから直接SMBプロトコルを介して起動することもできます。


重要なのは、本格的なシステムハックによって、DVDキーを紛失して再生できない人にチャンスが与えられたということです。プロセッサキーがあれば、DVDキーを取り出すのは難しくありませんでした。

もちろん、ここでマイクロソフトはすぐに脆弱性を閉じ、2BLを更新し、CBLDVの価値を高めました。これで、脆弱なハイパーバイザーの叙事詩は終わりました、人々は店で「JTAG互換」コンソールの残りを購入するために走りました-誰もが問題なくUSBフラッシュドライブで遊んでいたかったです。ハッキングに適したリリース日が含まれているバンドルについてフォーラムで議論が行われました
