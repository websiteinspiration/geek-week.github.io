How to use third-party assemblies in the plug-in

Problem



I, and I think many developers of MS Dynamic CRM (in my case this is 2016 on premise) ran into a problem, in order to use third-party assemblies, you need to combine them with the main assembly and then register it in the Plugin Registration Util.



As an example, many use ILMerge or ILRepack.



Search



I searched for solution options at various sites, but everywhere except using ILMerge or ILRepack, I did not find any other options. I would like to implement the task so that the assemblies could move together with the database to another server, and everything would work without problems.

using System;

using System.Data.SqlClient;

using System.Reflection;

using System.Web;



namespace CrmExternalAssemblyLoader

{

public class AssemblyLoaderModule: IHttpModule

{

private static bool IsLoaded;

private readonly object _lock = new object();

public void Dispose()

{

throw new NotImplementedException();

}



public void Init(HttpApplication context)

{

lock (_lock)

{

if (!IsLoaded)

{

AppDomain.CurrentDomain.AssemblyResolve += CurrentDomain_AssemblyResolve;

IsLoaded = true;

}

}

}



private Assembly CurrentDomain_AssemblyResolve(object sender, ResolveEventArgs args)

{

var assemblyName = new AssemblyName(args.Name);

var assembly = GetAssembly(assemblyName.Name + ".dll");

if (assembly != null)

{

return Assembly.Load(assembly);

}

return null;

}

Done.



When the plugin execution starts, CRM will try to find the assembly in memory, will not find it and climb into AppDomain.CurrentDomain.AssemblyResolve, where the assembly is subtracted from the database and loaded into the domain memory. The assembly will live until the site is reloaded.



Pros \ Cons



+ You do not need to merge assemblies into the plugin, thus the assembly size will not increase, there will not be much load on SQL when the plugin starts, especially if the plugin is registered to none.

+ The access speed to the assembly is higher since it is already loaded into the domain.

+ All third-party assemblies are in your database and migrate with the base.

+ Ability to store assemblies of different versions ( This moment is poorly tested! So there is still something to check)



- Perhaps when updating CRM, you will need to add the module line to web.config CRM again.
