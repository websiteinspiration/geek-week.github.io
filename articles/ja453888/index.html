PLCとセンサー


プログラマブルロジックコントローラー-サイクリックプログラムが組み込まれた小型コンピューター-プロセスの自動化に最もよく使用されます。実際には、PLCの助けを借りて、センサーから読み取り値を取得します。たとえば、加速度と速度、電圧レベル、軸に沿った振動、温度（この場合は17個のインジケーター）です。センサーはよく間違えられます。私たちのプロジェクトは8か月以上前のものですが、センサーを使って実験を行い、最も適切なモデルを選択する独自のラボがあります。現在、例えば、超音波センサーの使用を検討しています。

個人的に、私は最初にPLCを見ました。開発者としては、これまで一度も遭遇したことがなく、かなり不快でした。2相、3相、4相モーターよりも深く掘り下げて会話を始めると、すぐにスレッドを失い始めました。言葉の約80％はまだ理解可能でしたが、一般的な意味は頑固にずれてしまいました。一般的に、これは深刻な問題であり、そのルーツはPLCプログラミングを開始するためのかなり高いしきい値にあります-実際に何かを行うことができるマイクロコンピューターは、少なくとも200〜300ドルのコストがかかります。この場合、プログラミング自体は簡単であり、問​​題が発生するのは、センサーが実際のコンベヤーまたはモーターに接続されている場合のみです。



37-in-1標準センサーセット

ご存知のように、センサーは異なります。18ドルからのコストを見つけることができた最も単純なもの。主な特徴-「帯域幅と解像度」-センサーが1分間に送信するデータ量。私自身の経験から、メーカーが1分あたり30データポイントと主張する場合、実際にはその数は15を超えることはほとんどありません。また、これは深刻な問題を引き起こします。このトピックは流行であり、一部の企業はこの誇大宣伝で利益を上げようとしています。158ドル相当のセンサーをテストしました。その帯域幅により、理論的にはコードの一部を破棄することが可能になりました。しかし、実際には、これらは同じデバイスの絶対的な類似品であり、価格は18ドルでした。

最初の段階：センサーを取り付け、データを収集します


実際、プロジェクトの最初のフェーズはハードウェアのインストールでした。インストール自体は長くて退屈なプロセスです。これも全体の科学です。センサーが収集するデータは、センサーをモーターまたはボックスにどのように取り付けるかによって異なります。 2つの同じセンサーの1つがボックスの内側に、もう1つが外側に取り付けられている場合がありました。ロジックは、内部の温度を高くすることを推奨していますが、収集されたデータは別の方法で示しています。システムがクラッシュしたことが判明しましたが、開発者が工場に到着したとき、彼はセンサーが箱の中にあるだけでなく、そこにあるファンのすぐ上にあるのを見ました。



この図は、最初のデータがどのようにシステムに入力されたかを示しています。ゲートウェイがあり、PLCとセンサーが関連付けられています。さらに、もちろん、キャッシュ-機器は通常モバイルカードで実行され、すべてのデータはモバイルインターネット経由で送信されます。お客様の仕分けセンターの1つは、ハリケーンが頻繁に発生するエリアにあり、接続が切断される可能性があるため、復元されるまでゲートウェイにデータを蓄積します。

次に、データをクラウド（AWS）にフィードするAmazonのGreengrassサービスを使用します。

データがクラウド内に入るとすぐに、一連のイベントがトリガーされます。たとえば、ファイルシステムデータを保存する生データのイベントがあります。通常のシステムパフォーマンスを示す「ハートビート」があります。 UIでの表示に使用される「ダウンサンプリング」があり、処理（特定のインジケーターの1分あたりの平均値など）が行われます。つまり、生データに加えて、システムを監視するユーザーの画面に表示されるダウンサンプリングデータがあります。

生データは寄木細工の形式で保存されます。最初はJSONを選択し、次にCSVを試しましたが、最終的には分析チームと開発チームの両方が「寄せ集め」に満足しているという結論に達しました。

実際、システムの最初のバージョンはDynamoDBに基づいて構築されており、このデータベースについて悪いことは言いたくありません。アナリスト（受け取ったデータを操作する必要がある数学者）がいるとすぐに、DynamoDBのクエリ言語が複雑すぎることがわかりました。MLと分析のために特別にデータを準備する必要がありました。そのため、AWSのクエリエディターであるAthenaを採用しました。私たちにとっての利点は、Parquetデータの読み取り、SQLの書き込み、CSVファイルでの結果の収集が可能になることです。分析チームが必要とするものだけです。

6つのテスラ工場で約1万人が働いており、そのような規模の生産にはこれで十分ではありません。興味深いことに、メルセデスの工場はさらに自動化されています。関与するすべてのロボットが常に監視する必要があることは明らかで

SageMakerでは、約80のアルゴリズムをそのまま使用できます。これには、Caffe2、Mxnet、Gluon、TensorFlow、Pytorch、Microsoftコグニティブツールキットのフレームワークが含まれています。現時点ではKeras + TensorFlowを使用していますが、Microsoftコグニティブツールキットを除くすべての人がなんとかやってみました。このように広い範囲をカバーすることで、自分の分析チームを制限する必要がなくなります。



最初の3〜4か月間、人々は単純な数学の助けを借りてすべての作業を行いましたが、実際にはMLはありませんでした。システムの一部は純粋に数学的な法則に基づいており、統計データ用に設計されています。つまり、平均温度レベルを監視し、それがスケールから外れた場合は、アラートがトリガーされます。

次に、モデルのトレーニングが行われます。すべてが簡単でシンプルに見えますが、実装が始まる前はそうです。

ビルド、トレーニング、デプロイ...




どのようにしてこの状況を脱したのかを簡単に説明します。 2番目の列を見てください。データの収集、処理、クリーニング、S3バケットと接着剤を使用してイベントを開始し、「パーティション」を作成します。 AthenaはS3の上に構築されているため、すべてのデータがAthenaのパーティションに配置されていますが、これも重要なニュアンスです。アテナ自体はとても安いです。ただし、各リクエストは非常に高価になる可能性があるため、データの読み取りとS3からのデータの取得には料金がかかります。したがって、パーティションの大規模なシステムがあります。

ダウンタイマーがあります。また、Amazon EMRを使用すると、データをすばやく収集できます。実際、機能エンジニアリングのために、私たちのクラウドでは、各アナリストのために、Jupyter Notebookが提起されました-これは彼ら自身のインスタンスです。そして、クラウド内で直接すべてを分析します。

SageMakerのおかげで、主にトレーニングクラスターフェーズをスキップできました。このプラットフォームを使用しなかった場合、Amazonでクラスターを作成する必要があり、DevOpsエンジニアの1人がそれに従う必要があります。SageMakerでは、メソッドのパラメーター、Docker上のイメージを使用してクラスターを生成できます。これは、使用するパラメーターのインスタンス数を示すためだけに残ります。

さらに、スケーリングを処理する必要はありません。大きなアルゴリズムを処理したい場合、または何かを緊急に計算する必要がある場合は、自動スケーリングをオンにします（CPUまたはGPUのどちらを使用するかによって異なります）。

さらに、すべてのモデルは暗号化されています。これは、SageMakerの箱から出して、S3に含まれているバイナリです。

モデルの展開


環境に展開された最初のモデルに近づいています。実際、SageMakerではモデルのアーティファクトを保存できますが、SageMakerには独自のモデル形式があるため、この段階では多くの論争がありました。制限を取り除いて、それを避けたかったので、モデルはピクルス形式で保存され、必要に応じてKeras、TensorFlowなどを使用することもできます。 SageMakerの最初のモデルをそのままネイティブAPIで使用しましたが、

SageMakerはさらに3つのステップで作業を簡素化します。何かを予測しようとするたびに、特定のプロセスを開始して、データを与え、予測値を取得する必要があります。カスタムアルゴリズムが必要になるまで、すべてがうまくいきました。



アナリストは、CIとリポジトリがあることを知っています。 CIリポジトリには、3つのファイルを配置するフォルダがあります。 Serve.pyは、SageMakerがFlaskサービスを生成し、SageMaker自体と通信できるようにするファイルです。 Train.pyは、モデルに必要なすべてのものを配置する必要のあるtrainメソッドを持つクラスです。最後に、predict.py-その助けを借りて、このクラスを発生させます。その中にメソッドがあります。アクセス権があれば、S3からあらゆる種類のリソースをそこから調達できます。SageMaker内には、インターフェイスからプログラムで何でも実行できるイメージがあります（制限はありません）。

SageMakerから、predict.py-内部画像-にアクセスできます。これは、特定のパラメーターを使用して予測を呼び出したりトレーニングしたりできるFlaskアプリケーションです。これらはすべてS3に関連付けられており、さらに、Jupyter Notebookからモデルを保存する機能があります。つまり、Jupyter Notebookでは、アナリストはすべてのデータにアクセスでき、何らかの実験を行うことができます。

本番環境では、これらはすべて次のようになります。私たちにはユーザーがいて、予測値のエンドポイントがあります。データはS3にあり、Athenaに送信されます。 2時間ごとに、次の2時間の予測を計算するアルゴリズムが起動されます。このタイムステップは、私たちのケースでは、モーターに問題があると言うには約6時間の分析で十分であるという事実によるものです。スイッチを入れた瞬間でも、モーターは5〜10分で加熱され、急激なジャンプは発生しません。

重要なシステムでは、たとえば、エールフランスが航空機のタービンをチェックする場合、予測は10分の速度で行われます。この場合、精度は96.5％です。

何か問題が発生している場合は、通知システムがオンになります。次に、時計やその他のデバイスの多くのユーザーの1人が、特定のモーターが異常に動作しているという通知を受け取ります。彼は行って彼の状態をチェックします。

トレーニングジョブを作成する


SageMakerはトレーニングジョブを理解しています。その結果、API（S3に格納されているバイナリ）のみを使用する場合：指定したパラメーターから、モデルが取得されます。

sagemaker = boto3.client('sagemaker')

sagemaker.create_training_job(**create_training_params)

status = sagemaker.describe_training_job(TrainingJobName=job_name)['TrainingJobStatus']
print(status)

try:
    sagemaker.get_waiter('training_job_completed_or_stopped').wait(TrainingJobName=job_name)
finally:
    status = sagemaker.describe_training_job(TrainingJobName=job_name)['TrainingJobStatus']
    print("Training job ended with status: " + status)
    if status == 'Failed':
        message = sagemaker.describe_training_job(TrainingJobName=job_name)['FailureReason']
        print('Training failed with the following error: {}'.format(message))
        raise Exception('Training job failed')
