JavaScriptコードの信頼性を向上させるための7つの推奨事項

本日の翻訳を公開している記事の著者は、JavaScriptに関する7つの推奨事項を読者と共有することを決定しました。これらの推奨事項は、著者が望んでいるように、より信頼性の高いプログラムを作成するのに役立ちます。



1.ファクトリー機能を使用する


誰も知らない場合、ファクトリ関数はオブジェクトを返す通常の関数（クラスやコンストラクタではない）です。このシンプルなコンセプトにより、JavaScriptの強力な機能を利用して、強力で信頼性の高いアプリケーションを作成できます。

キーワードを使用せずにファクトリ関数が呼び出されることを知っておくことが重要newです。このキーワードで彼女に電話をかけると、彼女は本来あるべき行動を停止します。

factoryファクトリ関数を使用する理由


ファクトリ関数を使用すると、クラスやキーワードにバインドしなくても、オブジェクトのインスタンスを簡単に作成できますnew。

ファクトリ関数の本質は、それらが最も一般的な関数と見なされることです。つまり、オブジェクト、その他の関数、さらにはプロミスの構築に使用できます。つまり、そのような関数を組み合わせてより強力なファクトリ関数を作成し、さらに他の関数やオブジェクトと組み合わせてさらに高度なファクトリ関数を作成することができます。ファクトリー機能は、プログラマーに無限の可能性を開きます。

上記を考慮し、ファクトリ関数の使用とコードを記述するための推奨アプローチを組み合わせると、これらの関数は強力で便利なツールになります。

以下は、ファクトリー関数の簡単な例です。

function createFrog(name) {
  const children = []
  
  return {
    addChild(frog) {
      children.push(frog)
    },
  }
}

const mikeTheFrog = createFrog('mike')

ファクトリー関数をしばらく使用すると、コンストラクター関数の形での競合製品と比較して、ファクトリー関数がコードの再利用のレベルを向上させることが理解できるようになります。その結果、プログラマーはより少ないコードを書く必要があります。ファクトリ関数は、任意のオブジェクトを返すことができ、コードベースでの作業を簡素化できるため、リファクタリングを容易にします。

2.コンストラクター関数を作成するときに、プロトタイプにメソッドを追加します


JavaScriptの学習を始めたばかりの場合は、プロトタイプオブジェクトを操作することは、何か新しいことに思えるかもしれません。ですから、最初は私と一緒でした。

この手法はクラスには適していません。クラスは、プログラマーの側で追加の作業をせずに、メソッドをプロトタイプに個別に保存します。

以下は、コンストラクター関数の例です。

function Frog(name, gender) {
  this.name = name
  this.gender = gender
}

Frog.prototype.leap = function(feet) {
  console.log(`Leaping ${feet}ft into the air`)
}

コンストラクタによって作成されたオブジェクトにFrog.prototype.leapメソッドleapを書き込むだけでなく、なぜコンストラクトを使用するのですか？たとえば、次のようになります。

function Frog(name, gender) {
  this.name = name
  this.gender = gender
  
  this.leap = function(feet) {
    console.log(`Leaping ${feet}ft into the air`)
  }
}

実際、メソッドがコンストラクタープロパティに直接アタッチされている場合prototype、これは、このメソッドが、コンストラクターによって作成されたオブジェクトのすべてのインスタンスで共有されることを意味します。

つまり、前の例を使用this.leapすると、オブジェクトの複数のインスタンスを作成するときにFrog、それぞれに独自のメソッドがあることがわかりleapます。つまり、このメソッドのコピーがいくつか作成されます。この場合、これはシステムリソースの不合理な使用を示しています。これらすべてのオブジェクトには同じメソッドのコピーがあり、どこでも同じように動作するためです。オブジェクトの各インスタンスでそのようなメソッドのコピーを作成する必要はありません。

その結果、プログラムのパフォーマンスが低下します。しかし、これを回避することは難しくありません。プロパティことに留意すべきであるthis.nameとはthis.gender、彼らが特定のオブジェクトに属している必要がありますので、この形で宣言する必要があります。実際のカエルに類似点を描くと、その仮想表現はコンストラクターを使用して記述されFrogますが、カエルには独自の名前を付けることができ、カエルには異なる性別があることがわかります。その結果、各オブジェクトにカエルに関する一意の情報を格納するには、オブジェクトインスタンスのレベルで正確に使用されるように、オブジェクトのプロパティを宣言することが理にかなっています。

4. TypeScriptを使用する


TypeScriptは、この言語がプログラマーにタイプを安全に操作するための強力なツールを提供し、実行中のコードに表示される前でもエラーを検出できるため、JavaScriptコミュニティーで広く普及しています。

TypeScriptを使用すると、コードを起動する前であっても、コードのコンパイル段階で潜在的なエラーを見つけることができます。コードに問題がある場合、コンパイル時にコンパイル通知が表示されます。

しかし、タイプセーフとエラーの早期検出は、あらゆる状況でTypeScriptを使用する理由の完全なリストとはほど遠いものです。TypeScriptの優れた点の1つは、主要なブラウザーに表示される前に新しいJavaScript機能を利用できることです。TypeScriptコードは、最新のブラウザーではなく最新のブラウザーの両方でサポートされているJavaScriptコードにコンパイルされるのが事実です。

5.テストを書く


プロジェクトに取り組んでいて、この作業に真剣に取り組んでいる場合、実際にはテストを作成する必要があります。これにより、コードが予測可能になり、エラーが発生しにくくなります。これにより、コードに変更を加えるときに、コードの品質に高いレベルの信頼が与えられます。言い換えると、コードを時の試練に耐えさせたいのであれば、テストを書くことよりもこのテストのためにコードを準備するより良い方法はありません。プロジェクトで行うテストが多いほど、コードを本番環境にデプロイするときのコードの信頼度が高くなります。

テストの最も重要でポジティブな機能を1つだけ選択する必要があるとしたら、それは何でしょうか。これは、テストが実際のプログラムに入る前にエラーを見つけるのに役立つという事実だと思います。どのようなプログラマーがそのような機会を望まないのですか？私は確かにそのようなことを拒否しません。それが私が自分のプロジェクトの単体テストを書いている理由です。

プロジェクトのテストの作成を開始する場合は、コードテストを整理するためのツールとフレームワークが数多くあることに注意してください。ここにいくつかの良い点があります。

ご存じのとおり、問題のないJavaScriptを使用すると、一度に多くのタスクを解決する巨大な関数を作成できます。

プログラミングに慣れていない場合、これはポジティブに思えるかもしれません。自分のことを言いますが、昔は、自分が必要とすることを行う大量のコードを書いたとき、とても気分が良かったと思います。それは主に私にとって重要でした。私のコードが問題なく機能していることを確認したときは、自信が持てました。言うまでもなく、十分に機能するコードの巨大なブロックを書くことができたという事実によって私の自信が強化されました。そのとき私はなんて素朴だったのだろう。

保守が容易なコード、単純でエラーが発生しにくいコードを記述したい場合は、関数を可能な限り単純かつ小さくすることが最善の策です。関数が単純であるほど、システムの他の部分から分離してテストすることが容易になります。

これは、関数型プログラミングの原則を仕事で使用したい人にとって特に重要です。これに関して、関数のよく知られた要件の1つを思い出すことができます。関数は1つの問題のみを解決する必要があり、この問題を非常によく解決する必要があります。
