Dimensions- und Faktentabellen in DATA VAULT


Die am schwierigsten zu verstehenden Informationen:

  • Messtabellen basieren auf den Informationen von Hubs und ihren Satelliten.
  • Faktentabellen basieren auf den Informationen von Links und ihren Satelliten.

Dies ist offensichtlich, nachdem Sie den Artikel zu den Grundlagen von DATA VAULT gelesen haben . Hubs speichern eindeutige Schlüssel von Geschäftsobjekten, ihre Satelliten den Status von Attributen von Geschäftsobjekten, die an die Zeit gebunden sind, Satelliten, die an Links gebunden sind, die Transaktionen unterstützen, speichern die numerischen Merkmale dieser Transaktionen.

Auf dieser Theorie endet im Prinzip.

Meiner Meinung nach ist es jedoch notwendig, einige Konzepte zu beachten, die in Artikeln zur DATA VAULT-Methodik zu finden sind:

  • Raw Data Marts - Fenster mit Rohdaten;
  • Information Marts - Informationsspeicher.

Der Begriff „Raw Data Marts“ bezieht sich auf Storefronts, die auf DATA VAULT-Daten basieren, indem relativ einfache JOINs ausgeführt werden. Mit dem Ansatz „Raw Data Marts“ können Sie Ihr Lagerprojekt flexibel und schnell um Informationen erweitern, die für die Analyse geeignet sind. Ein solcher Ansatz impliziert nicht die Implementierung komplexer Datentransformationen und die Implementierung von Geschäftsregeln, bevor diese in eine Storefront gestellt werden. Die Daten von „Raw Data Marts“ sollten jedoch für den Geschäftsbenutzer verständlich sein und als Grundlage für die weitere Transformation dienen, beispielsweise mit BI-Tools.

Das Konzept der „Information Marts“ wurde in der Data Vault 2.0-Methodik eingeführt und ersetzte das alte Konzept der „Data Marts“. Diese Änderung ist darauf zurückzuführen, dass die Aufgabe der Implementierung eines Datenmodells zum Erstellen von Berichten als Konvertieren von Daten in Informationen bekannt ist. Das „Information Marts“ -Programm sollte dem Unternehmen in erster Linie Informationen liefern, die für die Entscheidungsfindung geeignet sind.

Ausreichend ausführliche Definitionen spiegeln zwei einfache Tatsachen wider:

  1. Die Vitrinen für „Raw Data Marts“ basieren auf Raw (RAW) DATA VAULT, einem Repository, das nur grundlegende Konzepte enthält: HUBS, LINKS, SATELLITES;
  2. Information Marts Storefronts werden mit BUSINESS VAULT-Elementen erstellt: PIT, BRIDGE.

Wenn wir uns Beispiele zum Speichern von Informationen über einen Mitarbeiter ansehen, können wir sagen, dass eine Storefront, in der die aktuelle (für heute aktuelle) Telefonnummer des Mitarbeiters angezeigt wird, eine Storefront vom Typ „Raw Data Marts“ ist. Um ein solches Schaufenster zu erstellen, werden der Geschäftsschlüssel des Mitarbeiters und die Funktion MAX () verwendet, die für das Attribut des Satelliten-Upload-Datums (MAX (SatLoadDate)) verwendet wird. Wenn der Verlauf der Attributänderung in der Storefront gespeichert werden soll - wenn er verwendet wird, müssen Sie wissen, zu welchem ​​Datum das Telefon auf dem neuesten Stand war. Die Kompilierung des Geschäftsschlüssels und das Datum des Ladens in den Satelliten werden dieser Tabelle als Primärschlüssel hinzugefügt. Das Datum des Ablaufdatumsfelds wird ebenfalls hinzugefügt.

Wenn Sie ein Schaufenster erstellen, in dem die relevanten Informationen zu jedem Attribut mehrerer im Hub enthaltener Satelliten gespeichert sind, z. B. Telefonnummer, Adresse, Name, müssen Sie die PIT-Tabelle verwenden, über die alle relevanten Daten leicht abgerufen werden können. Diese Art von Vitrine wird als "Information Marts" bezeichnet.

Beide Ansätze sind sowohl für Messungen als auch für Fakten relevant.

Zum Erstellen von Fensteranzeigen, in denen Informationen zu mehreren Links und Hubs gespeichert sind, kann der Zugriff auf BRIDGE-Tabellen verwendet werden.
