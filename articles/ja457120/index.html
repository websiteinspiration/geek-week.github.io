卵の皮とDevOpsの共通点は何ですか？

これは、hackernoon.comに投稿されたPatrick Lee Scottの記事の翻訳です。著者は、DevOpsを活用するのに役立ついくつかの重要な原則を理解することを提案しています。



数日前、私は愚かな方法で卵を剥がそうとしました、そして私のガールフレンドのアンジェリはなぜ私がこれをしているのかと尋ねました。

彼女は別の固ゆで卵をとって、まな板の上にぶつけ、それを押して、テーブルの上に転がしました。掃除には3秒しかかかりませんでした。そして、私は立って、殻を少しずつ外しました。

これで私が言いたいのは、私たちは努力の価値を誇張しているということです。

最良のソリューションはシンプルで効果的です。錆びたボルトの掃除は難しいですか？多分はい。しかし、これがコカコーラの助けを借りて行うことができることを知らない場合にのみ。まだ複雑ですか？番号。ボルトをグラスに入れて待つだけです。

簡単なテクニックを知らなければ、それらを適用することはできません。実装する代わりに実験を行い、複製する代わりに研究を行います。

プログラミングである種のアプローチを何度も使用する場合、解決される問題を単純化できるため、このアプローチはテンプレートまたはいわゆるベストプラクティスになります。

Command Query Responsibility Segregation（CQRS）やEvent Sourcing（ES）などの複雑で恐ろしい名前にもかかわらず、これらのプラクティスは問題の解決に役立ちます。特に、分散システムを構築するときに発生するもの。

開発全体を見ると、「Keep it Simple、Stupid」（KISS）や「Do n't Repeat Yourself」（DRY）など、より普遍的な原則があることがわかります。DevOpsに関連する同様のパターンと原則についてお話ししたいと思います。

DevOpsは、鳥が歌い、太陽が輝く約束の地として提示されることがよくあります。しかし、適切な方法を使用しないと、DevOpsは地獄に変わり、すべての指にシェル（私のような）を突き刺します。

DevOpsシステムを作成して、KISSやDRYのような普遍的な原則に加えて、自分用のいくつかのソリューションを見つけました。テンプレートとは言えませんが、卵をすばやくきれいにするのに役立ちます。これらのソリューション（順不同）：

  • 他人があなたの前にしたことをしないでください。
  • 開発者ができる限り生産的になるようにします。
  • 生産は神話です。

ホイールを再発明しないでください-購入してください。

Craigslistと同じメールサーバーを使用できることをご存知ですか。そしてそれは無料で何ですか？メールサーバーが必要です。新しいサーバーを作成せず、既存のサーバーで作業してください。

私は素晴らしい自己ホスト型リストでツールを探すか、Helm Hubを使用してこれを行います。

Helmはソフトウェアを見つけるためのかなり新しいツールであるという事実にもかかわらず、このサイトは既に作成されています。https：//v3.helm.sh/

Helmを使用すると、既成の自転車を簡単に検索できます。

プログラムを始めたばかりの頃にさかのぼってみましょう。 9年生で、私は最初の「本当の」言語-QBasicを学びました。それ以前は、HTMLとCSSでWebサイトを作成して数年になります。その後、インターネットは新しくなり、パッケージを作成する方法を知っている人もいましたが、現在のようにパッケージを共有することはありませんでした。通常、ストレージにはフロッピーディスクを使用しました。グレード11のJava Swingで書いたアルカノイドのようなゲームと一緒に見つけるのは素晴らしいこと

です。少なくとも13歳の時、私はそれらについてしか知りませんでした。一部のJavaプロ（こんにちは、VladおよびNick！）はすでにクールでした。 ;）

今、すべてが間違っています。今日では、UIライブラリ全体を見つけることができます。 1つのコマンドでエレガントかつ簡単に日付を処理できるライブラリを見つけることができます。

完全に機能するサブシステムを使用およびインストールするためのツールがあれば、すばらしいと思いませんか？データベースが必要ですか？データベースのインストール

朗報：そのようなツールが存在します！ Helmでは、パッケージ化されてすぐに使用できる、完全に機能するサブシステムをインストールすることもできます。

原理はNPMとGradleの場合と同じですが、Helmで管理するパッケージはチャートと呼ばれます。チャートはコンテナをマークアップし、さまざまな方法でKubernetesで実行できるようにします。

自転車を購入することで、既成の解決策がわかります。チャートはステアリングホイールであり、Kubernetes内で実行される説明付きのコンテナはホイールです。

グラフの優れている点は、サービスをパックして任意のKubernetesクラスターで実行し、必要に応じて他のユーザーと共有できることです。

つまり、コードを使用して環境全体を記述することができます。

- name: backup
  repository: http://jenkins-x-chartmuseum:8080
  version: 0.0.2
- name: monitor
  repository: http://jenkins-x-chartmuseum:8080
  version: 0.0.3
- name: marketing-site
  repository: http://jenkins-x-chartmuseum:8080
  version: 1.1.10
- name: denormalizer-service
  repository: http://jenkins-x-chartmuseum:8080
  version: 1.0.0
- name: mongo
  repository: https://kubernetes-charts.storage.googleapis.com/
  version: 1.0.0

マーケティングサイトを1.2.0にアップグレードする必要がありますか？変更してコミットするだけです。

私はかつてテーブルに座り、自分のコードを見て、バグを追跡しようとしました。ユーザーは数週間前から彼について不満を言っており、ついに私はぶらぶらする少しの自由時間を過ごしました。

多田さん！見つかりました！修繕！

私は日当たりの良い部屋の職場のパーティションの後ろに座って、他の人に「バグを修正しました！」と叫びました。

次の火曜日、ユーザーがリリースを見ると、私の努力に間違いなく感謝します！しかし、このために、我々は荷造りと場所からの解放を移動しようとする必要があります...多分それがうまくいく何も起こらない場合は...

まあが、大丈夫、場合リリースは来週火曜日起こるん、ユーザーは間違いなくそれを感謝します！

それが、私がジュニアプログラマーとして大学での最初の仕事をしたときの展開でした。

それ以来、多くの変化がありました。

現在、トランクベースの開発を使用し、モジュールを1日に何度も展開しています。プルリクエストを送信すると、テストに合格し、ビルドが収集された後、ボットは収集された環境のレビューコードとともに対応するコメントを投稿します。

今日、あなたはオフィスの仕切りを叫ぶ必要はありません。

プログラマーに与える自由、つまりインフラストラクチャの必要な部分を自由に制御できるほど、DevOpsエンジニアとしての作業が容易になります。

最初のレッスンで、あなたと私は本番環境での更新の場合、1桁を変更してコミットするだけで十分であることを確認しました。アプリケーションをグラフにまとめる理想的な世界では、チームの各プログラマーが、本番段階でのツールの動作に影響を与える機会があります。 Kubernetesはチャートをコンテナの説明として理解します。そのため、リソースの要件を説明します。

ロジックは次のようなものです。新しいサービスに必要なメモリ量やCPU設定を推測することはできません（私だけではないと思います）。したがって、私のチームと私はSlackでこれらの設定を調整する必要があることを通知されるルールに従って、監視とアラートを展開します。つまり、システムをデプロイするときに、必要な設定についてシステムから通知されます。以前はシェルを選んだように、何時間も座ってPrometheus経由でリクエストを送信し、設定を調整していました。そして今、私はすべてを賢くすることを学びました。

「複雑に聞こえますね！」番号。チャートを設定するだけです。

何かを自動化または簡略化できる場合-先に進んでください。たとえば、「expose：true」というラベルの付いたサービスをデプロイするだけでDNSパスを割り当てることができるとしたらどうでしょう。ここに演算子が表示されます。これはより高度なKubernetesツールであり、知っておく必要があります。ただし、詳細にはあまり深く入りません。

レッスン3.生産は神話


これは私にとって本当の啓示でした。物事を別の角度から見る必要があったので注意深く聞いてください。

10年以上の間、世界には数種類の環境しかないと思いました。最も単純なシナリオでは、環境のステージングと生成があります。最初にステージングに展開し、次にテストし、次のステージに移動し、展開し、テストしました。すべてがくっついたら-息子が統合されて-すぐに生産に入ることができます。

私はこのパターンを毎年続け、一度も疑ったことはありません。それは常にそうでした。

そして、このスキームは、シェルを取り除くための別の非生産的な方法です。

本質的に、チャートは依存チャートです。これを使用して環境を表すことができます。その後、本番環境でのデプロイメントプロセスは、単一のチャートをデプロイすることになります。

共通のコンテキストで接続された各チーム、プロジェクト、またはグループに独自のチャートがある場合、単一のトランザクションですべてのサービスをグループ化および更新できるいくつかの環境が表示されます。

プロダクションを大規模で包括的な環境として捉えないでください。実際、これは小規模なプロダクションです。

大きな変化はその規模で恐ろしいです。また、小規模な開発環境が複数ある場合は、変更を分離してシステムをより柔軟にすることができます。

さらに、すべてがチャートの形式で配置されている場合、変数は設定され、同じ方法で外部から変更できます。たとえば、それほど強力ではないkafka-clusterを本番環境に接続する必要がない場合は、構成を変更するだけです。

レッスン4.すべてをクラスターに転送し、全体をバックアップする


だから、私たちは生産を整理しました。データベースをどうするかカフカとは？セキュリティ上の問題がありますか？

注意深く読んだ場合は、覚えておいてください。データベースをチャートパッケージに含めることができることを書きました。

Kubernetesには、データベースやその他のステートフルアプリケーションを便利な形式で実行するための個別のAPI、StatefulSetsがあります。

Kubernetesの本質は、コンテナの起動の信頼性を向上させることです。これを使用して、Veleroツール（Helm Chart経由でインストール）を使用すると、Kubernetesクラスター全体のバックアップとそれに接続されているディスク（StatefulSetによって作成されたディスクなど）を作成し、1つのコマンドですべてを復元できます。さらに、スケジュールに基づいて自動バックアップを構成することも簡単です。

1つのチームでのバックアップ、リカバリ、およびKubernetesマネージャーは、完全に新しいクラスターをデプロイし、わずか2つのチームでバックアップを復元するのに役立ちます。新しいバックアップを最初に作成する場合は、最大3つ。

サーバーで考える代わりに、クラスター全体を操作できます。

プリプロダクションは迷惑ですか？DNSのバックアップ、リカバリ、変更の距離まで、視界から外してください。

レッスン5. VPNは複雑すぎる、より簡単な解決策がある


喜んでVPNを使用したことがありますか？

真剣ではありません。可能ですか？

Googleはそのような試みをしました。しかし、数年前、同社はVPNを使用しないことを発表しました。彼らもそれを楽しんでいないと思います。

Googleは信頼できないネットワークに切り替えました。どのネットワークにも適したマスターキーはありませんが、各サービスの入り口には、SSOログイン画面の形式のキーがあります。監視サービスにログインしてみませんか？会社のユーザー名とパスワードを使用してログインします。

わずかな変更のみが必要です。承認にVPNを使用する代わりに、プロキシを使用します。

VPNはプライベートネットワーク上のKubernetes管理リンクもホストしますが、SSOはホストしません。ただし、独自の認証メカニズムがあります。Google CloudとAWSの場合、これはIdentity and Access Management（IAM）であり、IPアドレスをホワイトリストに登録する機能です。

それほど損失のない、かさばらないアーキテクチャで作業できる場合は、どうしてでしょうか。Googleのように：VPNから信頼されていないシステムに移行します。

レッスン6.整理、自動化、および征服


ああ、体系化は長いことだと思いますか？ナンセンス！将来的には、これにより数時間、数日、さらには数週間も節約できます。仕事に！

体系化は、インフラストラクチャを複数回再作成するための唯一の現実的な方法です。

gitで何かを変更してコミットすることで各設定項目を改善できる場合、基本的に技術組織全体が宣言的です。 gitにアクセスできる開発者であれば、システムを稼働状態に簡単に維持できます。

これを行うには、いくつかの小さなリポジトリを使用します。 Monoreposは人々に手を抜いて、人工的に重要な構造に依存させます。代わりに、後でリンクできる小さなリポジトリを多数使用することをお勧めします。

友達のマットと私はメタというツールを作成していますこれは開発段階でこれを行うのに役立ちます。Helmはこれを生産段階で行います。彼にとって、すべてが依存チャートです！
