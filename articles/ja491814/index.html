コードとしてのネットワーク図

ここ数年で、私はドキュメンテーションをもっと扱うようになりました。これまたはそのシステムがどのように機能するかについての説明文を書きます-一般的に、それは非常に簡単です。すべての主要なオブジェクトが表示される図を描画します。これらのオブジェクト間の関係も非常に簡単です。

しかし、最も問題なのは、このドキュメントを最新の状態に保つことです。そして、わかりました、テキストですが、図...すべてのドキュメントはオンラインです。 html形式の場合、図が実際に描かれているテキストにgif / jpeg / png画像が添付されます。スキームは、Visioやdraw.ioのオンラインサービスなどのさまざまなプログラムで描画されます。次に、ダイアグラムをグラフィック形式にエクスポートして、htmlに添付します。すべてが簡単です。

何が問題ですか？

スキームは通常単純です。より正確には、それほど複雑ではありません。はい、オブジェクトの数は10または2で、接続の数はほぼ同じです。プラス署名、いくつかの指定。単純なスキームは言葉で説明できますが、複雑すぎるため、うーん...（s）「理解していません」多くのスキームがあり、それらの変更は定期的、定期的、つまり常に、なぜなら彼らは私たちの製品の開発に従います。

htmlサービスを埋め込むことができます。試しましたか？

はい、もちろん。たとえば、私はgliffy.comグラフィックスが好きです。ただし、変更については、サードパーティのサービスにアクセスして編集する必要があります。また、修正を同僚に委任することはより困難です。

何をすべきか？

最近、githubで、github.com / RaoulMeyer / diagram-as-codeリポジトリに出会いました。。コードとしてチャート。それら。必要な回路をjsで説明します。このjsは、他のドキュメントテキストと同じhtmlに直接記述します。

ちなみに、私はドキュメントをHTMLでまったく書いていません。通常、ドキュメントはマークダウンテキストを含むファイルのセットであり、その後、wintersmithなどのエンジンによって本格的なドキュメントサイトに変換されます。または、wikiシステム。

それは非常に便利であることがわかります。ここでテキストを記述してから、scriptタグを開き、回路のjsコードを記述します。

また何が悪いの？

私はこのリポジトリが好きでしたが、コードまたはテキストビューを使用して図を描画する場合の唯一の例ではありません。 （記事の最後に、コードとしてトピック図でグーグル化したプロジェクトや記事へのリンクがあります。）

また、ドキュメントを修正するのは私だけではありません。時には同僚も貢献します-単語を修正し、説明を変更し、新しい写真を挿入します。 

したがって、図を読みやすく理解しやすいテキスト形式で表示したいと考えています。これは、長期間調査する必要はありません。また、場所によっては、コピーと貼り付けを簡単にして、新しいスキームの追加を迅速化することもできます。 

そして、別の同僚は、コードはもちろん優れていると述べましたが、構造を使用すると、すべてが非常に厳密で表現力豊かになる可能性があります。

したがって、ノード、接続、ノードのグループ、およびノー​​ドの場所を記述するいくつかの小さな配列のセットとしてスキームを提示しようとしました。私の控えめな意見では、それは結局便利ですが、もちろん、味と色はわかりました ...

配列内のチャートはどうですか？

  • 各ノードは、ノードを一意に識別する識別子によって記述されます。
  • ノードにアイコンを追加したり、碑文を追加したりすることもできます。
  • 2つのノード間の関係を指定できます。
  • スキーム上のコミュニケーションのために、あなたは色、碑文を設定することができます。
  • 通信の方向は、ソースからターゲットへの方向として定義されます。そして、ソースとターゲットはノードの識別子によって示されます。
  • 1つ以上のノードをグループに追加できます。
  • グループからグループへのリンクも指定できます。

これらの単純なルールを使用して、このようなスキームを取得します。単に？かなり。



そして、それは次のjsコードで記述されています。ここでの主なものは、elementsオブジェクトです。ノード-ノード、エッジ-接続が示されます。
 
  const elements = {
    nodes: [       //  
      { id: 'client', type: 'smartphone', label: 'Mobile App'},
      { id: 'server', type: 'server', label: 'Main Server'},
      { id: 'db1', type: 'database', label: 'DB 1'},
      { id: 'db2', type: 'database', label: 'DB 2'},
    ],
    edges: [       //  
      { source: 'client', target: 'server', label: 'request' },
      { source: 'server', target: 'db1', label: 'request' },
      { source: 'server', target: 'db2', label: 'request' },
    ],
  };
  Diagram('scheme1', elements);

そして、なぜノードとは別にポジションが取り出されるのですか？

これはより快適です。最初にノードを指定します。次に、いくつかのグループを示し、それらをノードで示すことができます。次に、接続を示します。そして、主要なオブジェクトとそれらの間の接続が存在する場合、ダイアグラム上のこれらのオブジェクトの位置を取り上げます。またはその逆。

ポジションがなくても可能ですか？

ポジションがなくても可能です。しかし、それは少ししわくちゃになります。例では、このオプションを見ることができます。これはサイトスケープにはfcoseノードの位置を特定するアルゴリズムがあるためです。、グループの存在も考慮に入れます。位置を指定すると、スキームがより制御しやすくなりますが、スキームの最初のドラフトの段階では、位置がなくても可能です。

また、海軍戦闘のスタイルで位置を指定できます。それら。1つのノードはa1にあり、もう1つのノードはd5にあります。これは特に、サイトスケープがキャンバス上でオブジェクトを移動可能に、つまり それらを移動し、さまざまなレイアウトオプションを確認してから、コードで要素の好みの配置を修正できます。
