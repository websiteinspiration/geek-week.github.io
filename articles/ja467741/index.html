確定的アセンブリとは何ですか？


確定的アセンブリとは、同じ環境とアセンブリ命令で同じソースコードを構築するプロセスであり、異なるマシン、異なるディレクトリ、異なる名前で作成された場合でも、同じバイナリファイルが常に作成されます。 。このようなアセンブリは、異なるフォルダからコンパイルした場合でも同じバイナリが作成されることが保証されている場合、再生可能またはシールされたアセンブリと呼ばれることもあります。

確定的アセンブリは、それ自体で発生するものではありません。これらは通常のプロジェクトでは作成されません。これが発生しない理由は、オペレーティングシステムやコンパイラによって異なります。

確定的なアセンブリは、特定のアセンブリ環境で保証される必要があります。つまり、オペレーティングシステム、ビルドシステムのバージョン、ターゲットアーキテクチャなどの一部の変数は、おそらく異なるアセンブリでも同じままです。

近年では、などの各種団体、クロム、再現ビルド、またはYoctoは、決定論的なアセンブリを達成するために多大な努力を行っています。

確定的アセンブリの重要性


確定的アセンブリが非常に重要である主な理由は2つあります。

  • セキュリティ。ソースコードの代わりにバイナリを変更すると、変更が元の作成者から見えなくなります。これは、医学、航空、宇宙などの安全が重要な条件では致命的となる可能性があります。これらの資料の潜在的に同一の結果により、サードパーティは正しい結果について合意に達することができます。

オペレーティングシステムに応じて、C / C ++のビルドプロセス中に作成されるさまざまな種類のバイナリがあります。

Windowsマイクロソフト。最も重要なの拡張子を持つファイルです.obj、.lib、。dllと.exe。これらはすべて、ポータブル実行可能（PE）形式の仕様に準拠しています。これらのファイルは、dumpbinなどのツールで分析できます。
Linux。以下の拡張子を持つファイル.o、.a、.so（実行可能なバイナリ用）と拡張子なしの実行ファイルとリンク可能なファイル（実行可能ファイルとリンク可能フォーマット、ELF）のフォーマットに対応 。 ELFファイルの内容は、readelfを使用して分析できます。
マックOS。以下の拡張子を持つファイル.o、.a、.dylib（実行可能なバイナリの場合）と拡張せずフォーマット仕様のMach-Oに対応します。これらのファイルは、MacOSのXcodeツールキットの一部であるotoolアプリケーションを使用して確認できます。

バリエーションのソース


多くの異なる要因により、ビルドが非決定的になる可能性があります。要因は、オペレーティングシステムやコンパイラによって異なります。各コンパイラには、変動の原因を修正するための特定のパラメータがあります。現在まで、gccそしてclang-これらは修正のためのより多くのオプションを含むコンパイラです。以下のためにmsvcあなたが試すことができますが、最終的に、あなたはおそらく、確定アセンブリを取得するためにバイナリを修正する必要があり、いくつかの文書化されていないオプションがあります。

コンパイラー/リンカーによって追加されたタイムスタンプ


バイナリに再生できない時間情報が含まれている主な理由は2つあります。

  • マクロ__DATE__または__TIME__ソースコードでの使用。
  • ファイル形式により、時刻情報をオブジェクトファイルに格納する必要がある場合。これは、Windowsのポータブル実行可能形式とMacOSのMach-Oの場合です。Linuxでは、ELFファイルはタイムスタンプをエンコードしません。

CMakeを使用してプロジェクトをビルドします。

cmake_minimum_required(VERSION 3.0)

add_executable(helloA main.cpp)
add_executable(helloB main.cpp)
target_link_libraries(helloA HelloLibA)
target_link_libraries(helloB HelloLibB)

Microsoft Visual Studioの可能な解決策


Microsoft Visual Studioには、Microsoftによって文書化されていないリンカー/ Breproフラグがあります。次の図に示すように、このフラグは、ポータブル実行可能形式からのタイムスタンプを-1に設定します。



CMakeでこのフラグをアクティブにするには、作成時に次の行を追加する必要があります.exe。

add_link_options("/Brepro")

またはこれらの行 .lib

    TARGET
    PROPERTIES STATIC_LIBRARY_OPTIONS "/Brepro"
)

問題は、このフラグにより​​、最終的なバイナリ.exeファイルでバイナリが（ファイル形式のタイムスタンプに関連して）再生可能になるが、.libからすべてのタイムスタンプが削除されない（Mach-Oオブジェクトファイルと同じ問題）上記で説明しました）。TimeDateStampフィールドCOFFヘッダファイルのファイルについては.lib残ります。この情報をバイナリから削除する唯一の方法.libは.lib、TimeDateStampフィールドに対応するバイトを既知の値に置き換えることで修正することです。
