Webアプリケーションをオフラインで機能させる方法

JavaScriptとブラウザーAPIの力

世界はますます相互に接続されており、インターネットにアクセスできる人の数は45億人に増えています。


ただし、このデータは、インターネット接続が遅いか切断されている人の数を反映していません。米国でも、490万世帯が毎秒3メガビットを超える速度で有線インターネットアクセスにアクセスできません。

残りの世界—信頼できるインターネットアクセスのある人々—は、依然として接続を失う傾向があります。ネットワーク接続の品質に影響を与える可能性のあるいくつかの要因は次のとおりです。

  • プロバイダーからの悪いカバレッジ。
  • 極端な気象条件。
  • 停電。
  • ネットワーク接続をブロックする建物など、デッドゾーンに陥るユーザー。
  • 列車の旅とトンネルの旅。
  • サードパーティによって制御され、時間制限がある接続。
  • 特定の時間または日にインターネットへのアクセスを制限するか、まったく必要としない文化的慣習。

これを踏まえると、アプリケーションの開発および作成時に自律的な経験を考慮する必要があることは明らかです。


この記事は、華南からの優れた注文を処理し、Webアプリケーションとサイトを開発している企業であるEDISON Softwareのサポートにより翻訳されました。
最近、Service Worker、キャッシュストレージ、IndexedDBを使用して、既存のアプリケーションに自律性を追加する機会がありました。アプリケーションがオフラインで機能するために必要な技術的な作業は、この記事で説明する4つの個別のタスクに削減されました。

サービス労働者


オフラインで使用するために作成されたアプリケーションは、ネットワークに大きく依存するべきではありません。概念的には、これは、障害発生時に緊急オプションが存在する場合にのみ可能です。

Webアプリケーションの読み込みに失敗した場合は、ブラウザーのリソース（HTML / CSS / JavaScript）をどこかに取得する必要があります。これらのリソースは、ネットワークリクエストではない場合、どこから取得されますか？キャッシュについてはどうでしょう。ほとんどの人は、空白のページよりも古くなっている可能性のあるユーザーインターフェイスを提供するほうがよいことに同意します。

ブラウザーは常にデータを照会します。フォールバックとしてのデータキャッシングサービスでは、ブラウザー要求をインターセプトし、キャッシングルールを作成する必要があります。これがサービスワーカーの出番です。仲介者と考えてください。



Service Workerは単なるJavaScriptファイルであり、イベントをサブスクライブして、ネットワーク障害のキャッシュと処理に関する独自のルールを記述できます。

タスク1-静的リソースのキャッシュ


静的リソースは、頻繁に変更されないリソースです。HTML、CSS、JavaScript、および画像は、このカテゴリに分類される場合があります。ブラウザは、Service Workerによってインターセプトされる可能性のあるリクエストを使用して静的リソースをロードしようとします。

サービスワーカーの登録から始めましょう。

if ('serviceWorker' in navigator) {
  window.addEventListener('load', function() {
    navigator.serviceWorker.register('/sw.js');
  });
}

Service Workerは内部でWebワーカーであるため、別のJavaScriptファイルからインポートする必要があります。registerサイトを読み込んだ後、メソッドを使用して登録を行います。
Service Workerが読み込まれたので、静的リソースをキャッシュしましょう。

var CACHE_NAME = 'my-offline-cache';
var urlsToCache = [
  '/',
  '/static/css/main.c9699bb9.css',
  '/static/js/main.99348925.js'
];

self.addEventListener('install', function(event) {
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(function(cache) {
        return cache.addAll(urlsToCache);
      })
  );
});

静的リソースのURLを制御しているため、Service Workerの初期化の直後にそれらをキャッシュできますCache Storage。



キャッシュに最近リクエストされた静的リソースがいっぱいになったので、リクエストが失敗した場合にこれらのリソースをキャッシュからロードしましょう。

self.addEventListener('fetch', function(event) {
  event.respondWith(
    fetch(event.request).catch(function() {
      caches.match(event.request).then(function(response) {
        return response;
      }
    );
  );
});

このイベントはfetch、ブラウザーがリクエストを行うたびに発生します。新しいイベントハンドラーにfetch、ネットワークが停止した場合にキャッシュされた応答を返すためのロジックが追加されました。

タスク2-動的リソースのキャッシュ


シングルページアプリケーション（SPA）は通常、ページの最初の読み込み後に徐々にデータを要求します。デモアプリケーションも例外ではありません。書籍のリストはすぐには読み込まれません。このデータは通常、アプリケーションに新しい状態を提供するために頻繁に変化する応答を返すXHRリクエストから取得されるため、動的です。

動的リソースのキャッシュは、実際には静的リソースのキャッシュと非常によく似ています。主な違いは、キャッシュをより頻繁に更新する必要があることです。可能なすべての動的XHR要求の完全なリストを生成することも非常に難しいため、静的リソースの場合のように事前定義されたリストを用意するのではなく、到着時にそれらをキャッシュします。

ハンドラを見てみましょうfetch：

self.addEventListener('fetch', function(event) {
 event.respondWith(
   fetch(event.request).catch(function() {
     caches.match(event.request).then(function(response) {
       return response;
     }
   );
 );
});

成功した要求と応答をキャッシュするコードを追加することで、この実装をカスタマイズできます。これにより、常に新しいリクエストがキャッシュに追加され、キャッシュされたデータが常に更新されます。

self.addEventListener('fetch', function(event) {
 event.respondWith(
   fetch(event.request)
     .then(function(response) {
       caches.open(CACHE_NAME).then(function(cache) {
         cache.put(event.request, response);
       });
     })
     .catch(function() {
       caches.match(event.request).then(function(response) {
         return response;
       }
     );
 );
});

我々はCache Storage現在、いくつかのエントリがあります。

タスク3-楽観的なユーザーインターフェイスを作成する


現在のところ、このアプリケーションの問題は、データ収集ロジックが依然としてネットワーク応答に大きく依存していることです。チェックインまたはチェックアウトアクションは、サーバーに要求を送信し、正常な応答を期待します。これはデータの整合性には優れていますが、スタンドアロンエクスペリエンスには適していません。

これらのやり取りをオフラインで機能させるには、アプリケーションをより楽観的にする必要があります。楽観的相互作用はサーバーからの応答を必要とせず、データの更新されたビューを喜んで表示します。ほとんどのWebアプリケーションでの通常の楽観的な操作はdelete、必要なすべての情報がすでにある場合に、ユーザーに即時のフィードバックを提供しないのはなぜですか？

楽観的アプローチを使用してネットワークからアプリケーションを切断することは、実装が比較的簡単です。

case CHECK_OUT_SUCCESS:
case CHECK_OUT_FAILURE:
  list = [...state.list];
  list.push(action.payload);
  return {
    ...state,
    list,
  };
case CHECK_IN_SUCCESS:
case CHECK_IN_FAILURE;
  list = [...state.list];
  for (let i = 0; i < list.length; i++) {
    if (list[i].id === action.payload.id) {
      list.splice(i, 1, action.payload);
    }
  }
  return {
    ...state,
    list,
  };

重要なのは、ネットワーク要求が成功したかどうかに関係なく、同じ方法でユーザーアクションを処理することです。上記のコードは、私たちのアプリケーションの再来の減速から取られ、SUCCESSおよびFAILUREネットワークの可用性に応じて開始します。ネットワークリクエストがどのように完了するかに関係なく、書籍のリストを更新します。

タスク4-ユーザーアクションを同期のためにキューに入れる


ユーザーがネットワークに戻ったときにサーバーと同期できるように、ユーザーがオフラインのときにユーザーが実行したアクションを追跡する必要があります。ブラウザには、アクションのキューとして機能できるいくつかのストレージメカニズムがあり、ここではIndexedDBを使用します。IndexedDBには、LocalStorageにはないいくつかの機能があります。

  • 非同期の非ブロッキング操作
  • 大幅に高いストレージ制限
  • トランザクション管理

古いレデューサーコードを見てください。

case CHECK_OUT_FAILURE:
  list = [...state.list];
  list.push(action.payload);
  addToDB(action); // QUEUE IT UP
  return {
    ...state,

case CHECK_IN_FAILURE;

以下は、ヘルパーとともにIndexedDBを作成する実装ですaddToDB。

let db = indexedDB.open('actions', 1);
db.onupgradeneeded = function(event) {
  let db = event.target.result;
  db.createObjectStore('requests', { autoIncrement: true });
};

const addToDB = action => {
  var db = indexedDB.open('actions', 1);
  db.onsuccess = function(event) {
    var db = event.target.result;
    var objStore = db
      .transaction(['requests'], 'readwrite')
      .objectStore('requests');
    objStore.add(action);
  };
};

すべてのオフラインユーザーアクションがブラウザーのメモリに保存されたので、ブラウザーイベントリスナーonlineを使用して、接続が復元されたときにデータを同期できます。

window.addEventListener('online', () => {
  const db = indexedDB.open('actions', 1);
  db.onsuccess = function(event) {
    let db = event.target.result;
    let objStore = db
      .transaction(['requests'], 'readwrite')
      .objectStore('requests');
    objStore.getAll().onsuccess = function(event) {
      let requests = event.target.result;
      for (let request of requests) {
        send(request); // sync with the server
      }
    };
  };
});

この段階で、サーバーに正常に送信したすべてのリクエストのキューをクリアできます。
