calc.exeの小さなバグの修正

日曜日には、いつものようにラウンジでRedditを閲覧しました。子犬の楽しさとプログラマーの悪いユーモアをスクロールして、私の注意は特定の投稿に引き寄せられました。これはcalc.exeのバグでした。


Windows電卓での日付範囲の計算結果が正しくない

「まあ、これは奇妙なエラーのようです。何が原因なのでしょうか」と私は思いました。もちろん、週数によって、バグはある種のオーバーフローまたは範囲エラーのように見えますが、それは典型的な理由です。しかし、それは常に、友好的な宇宙の隣人からの高エネルギー光線によって反転されたビットになる可能性があります。

理由に興味を持って、私はそのような場合にあなたがしたことをしました：私は自分のマシンでそれを試し、「すべてがうまくいく」と投稿しました。そして、私のマシンで「7月31日〜12月31日」の投稿からの状況を繰り返すと、「5ヶ月」という正しい結果が得られました。しかし、少しテストしたところ、「7月31日〜12月30日」が実際にエラーを引き起こすことがわかりました。誤った値「5か月、613566756週間、3日」が表示されます。

プログラムのシェイクを終えていなかったので、次のことを思い出しました。そして本当に。この間違いはあまり複雑ではないので、見つけようと思った。ソースのダウンロードは非常に簡単で、必要なUWPワークロードをVisual Studioに追加することにも問題はありませんでした。

慣れていないコードベースをナビゲートすることは、時間に慣れるものです。特に、バグを見つけたオープンソースプロジェクトに貢献したい場合。ただし、XAMLやWinRTを知らないからといって、物事が簡単になるわけではありません。

ソリューションファイルを開いて、バグに関連するファイルを探すために「計算機」プロジェクトを調べました。見つかったDateCalculator.xamlのでDateDiff_FromDate to DateCalculatorViewModel.cpp、ついにタイトルにふさわしいようDateCalculator.cppです。

ブレークポイントを設定していくつかの変数を調べたところ、最終的な値がDateDifferenceすでに正しくないことがわかりました。つまり、それは単なる文字列への変換エラーではなく、実際の計算のエラーでした。単純化

された疑似コードでの実際の計算は次のようになります。

DateDifference calculate_difference(start_date, end_date) {
    uint[] diff_types = [year, month, week, day]

    date pivot_date = start_date
    uint days_diff = calculate_days_difference(start_date, end_date)

        uint current_guess = days_diff /typicalDaysInType[type]

        int diff_remaining
        bool best_guess_hit = false
        do{
            diff_remaining = calculate_days_difference(pivot_date, end_date)
            if(diff_remaining < 0) {
                // pivotDate has gone over the end date; start from the beginning of this unit
                current_guess = current_guess - 1
                pivot_date = temp_pivot_date
                pivot_date = advance_date_by(pivot_date, type, current_guess)
                best_guess_hit = true
            } else if(diff_remaining > 0) {
                // pivot_date is still below the end date
                if(best_guess_hit)
                    break;
                current_guess = current_guess + 1
                pivot_date = advance_date_by(pivot_date, type, 1)
            }
        } while(diff_remaining!=0)

        temp_pivot_date = advance_date_by(temp_pivot_date, type, current_guess)
        pivot_date = temp_pivot_date 
        calculated_difference[type] = current_guess
        days_diff = calculate_days_difference(pivot_date, end_date)
    }
    calculcated_difference[day] = days_diff
    return calculcated_difference
}

元気です。ロジックに問題はありません。基本的に、関数は次のことを行います。

  • 開始日から完全な年を数えます
  • 昨年の日付から月を数える
  • 先月の日付から週を数えます
  • 先週の日付から残りの日数を数えます

実際には、問題はシーケンシャルスタートが 

date = advance_date_by(date, month, somenumber)
date = advance_date_by(date, month, 1)

等しい

date = advance_date_by(date, month, somenumber + 1)

これは通常同じことです。しかし、疑問が生じます。「月の31日目、次の月は30日目にあたる場合、1か月追加してから、どこに行きますか？」

のためのように思えるWindows.Globalization.Calendar.AddMonths（Int32）を答えはなります「30日に。」

つまり、
「7月31日+ 4か月= 11月30日」
「11月30日+ 1か月= 12月30日」
「7月31日+ 5か月= 12月31日」

したがって、AddMonths演算は分散的ではありません（AddMonth-multiplication ）、可換でも連想でもありません。実際に「足し算」の操作とは何か。時間とカレンダーで作業するのは楽しいことではありませんか？

なぜこの場合、範囲を設定するエラーがそのような膨大な数の週につながるのですか？ご想像のとおり、これはdays_diff符号なしの型が原因です。これにより、-1日が膨大な量になり、それが次の週のサイクルの反復に渡されます。次にcurrent_guess、符号なし変数を減らすことではなく減らすことによって状況を修正しようとします。

まあ、それは日曜日を過ごす面白い方法でした。最小限の「修正」で、Githubにプルリクエストを作成しました。計算は次のようになるので、「訂正」を引用符で囲みます。



技術的には「7月31日+ 4か月= 11月30日」と仮定すると、これは正しい結果だと思います。このオプションは、日付の違いに関する人間の直感と完全に一致しているわけではありません。しかし、いずれにせよ、これは以前ほど間違っていません。
