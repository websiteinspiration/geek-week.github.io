Pascal（バックエンド）でのウェブサイト開発

この記事では、Pascalでサイトを作成し始めた理由、理由、および方法について説明します：Delphi / FPC。
おそらく「Pascalサイト」は次のようなものに関連付けられています。

writeln('Content-type: text/html');

しかし、いいえ、すべてがはるかに興味深いです！ただし、実際のサイトのソースコード（ほぼすべて）はGitHubで入手できます。

何のために？


一般的に、私はプロのウェブデベロッパーではありません。ゲームを作っています。ゲーム、特にオンラインのゲームには、ウェブサイトが必要です。そのため、たまたま私は自分のゲーム用のサイトを作り始めました。 PerlでのCGIの使用-2000年代初頭から中頃にかけて、人気が高まりました。問題が発生するまで、すべてが順調でした。

2013年、Spectromancerゲームのオンライントーナメントの開催を開始しました。これを行うために、ゲームのWebサイトにトーナメントページを作成し、誰とプレイするか、現在の結果などを表示しています。トーナメントの開始時に、プレーヤーのページが更新され、...が読み込まれませんでした。人々はF5を押し、問題をさらに悪化させました。 CGIスクリプトへの1秒あたり4〜5個のリクエストでも、個別のPerlプロセスとして実行され、サーバーの速度が大幅に低下し、1秒あたり10個以上のリクエストでは完全にアクセスできなくなることがわかりました。

このストレステストがリハーサルトーナメント中に行われたのは良いことです。後でトーナメントの更新された静的ページをすでに使用しました。

どうして？


したがって、新しいゲームのためにこのサイトを作成する必要が生じたとき、疑問が生じました-何についてですか？PerlのブレーキCGIはオプションではありません。PerlのFastCGI？Perlでマルチスレッドプログラムを作成してデバッグする方法を想像できません。通常のスクリプトで十分な問題がありました。Node.js？JSに対する敵意がなければ、おそらくそれが最良の選択でしょう。ゲーム自体とそのサーバーはPascal（実際にはDelphiですが、FPCも優れている）で書かれているため、アイデアが浮かびました。サイトを同じ言語で作成する必要がありますか？これにより、ゲームサーバーとの統合が簡単になります。「試すことは拷問ではありません！」と思って試してみることにしました。

どうやって？


インターフェイスとしてSimpleCGI（SCGI）を選択しました。FastCGIよりもいくらか単純で、FastCGIの利点は私には関係ありません。バックエンドを別のサーバーに配布する必要はなく、すべてが同じサーバーで実行されます。したがって、タスクはサーバーからのリクエストを処理し、いくつかの空白のテンプレートからの応答としてHTMLページを生成する特定のSCGIフレームワークの開発に要約されます。結果はそのようなモジュールフレームワークです。次の部分で構成されています。

  • メインループ：着信接続を受け入れ、要求を読み取り、それらを処理のためにキューに入れます。処理された要求に対する既製の応答を接続ソケットに書き込み、それらを閉じます。

Perlスクリプトの非常に便利な機能は、サイトに小さな変更を加えるのが非常に簡単なことです。スクリプトコードを編集するだけです。コンパイル、デプロイする必要はありません。もちろん、pascalはコンパイルされた言語であり、そのようには機能しませんが、プロセスを再起動せずに可能な限り変更を加えられるようにしたかったのです。そのため、私はテンプレートシステムを十分に柔軟にすることを試みました。

彼女はそのように働きます。テンプレートファイルは「templates」フォルダにあります。プロセスの開始時にロードされ、変更時に再ロードされます。これにより、プロセスを再起動せずに動的コンテンツを変更できます。各ファイルには、1つ以上のテンプレートを含めることができます。一緒に、それらはテンプレートの辞書（またはハッシュ）を形成します：{"name"-> "value"}。これはテンプレートの静的辞書です。すべてのリクエストに共通であり、その内容は変更されません（ファイルの内容が変更されるまで）。別の1つがあります-動的辞書です。これは、要求ごとに空で作成され、データベースなどから動的データハンドラーで満たされます。静的データと動的データを組み合わせて、最終結果が形成されます。

テンプレート宣言の例：

#NEWSFEED_ITEM:
<div class=NewsHeader>
 <a href='/$LANG_LC/forum/thread/$NEWS_ID'><IF_NEWS_PINNED>[TOP]  </IF_NEWS_PINNED>$NEWS_DATE   $NEWS_TITLE</a>
</div>
<div class=NewsText>$NEWS_TEXT
 <div align=right>
  <a href='/$LANG_LC/forum/thread/$NEWS_ID'>$COMMENTS</a>
 </div>
</div>

これは、NEWSFEED_ITEMという名前のニュースフィードの静的テンプレートで、中には他のいくつかのテンプレートが含まれています。たとえば、NEWS_TEXT-データベースからダウンロードされたニューステキストを含む動的テンプレートです。翻訳は、$ TEMPLATE_NAME形式のすべての部分文字列が、このテンプレートの値で再帰的に置き換えられることです。

ここでは、条件付き翻訳の疑似タグにも気づくでしょう：<IF_TEMPLATE_NAME>-翻訳中に、指定されたテンプレートの値に応じて、そのようなタグが削除され、その内容が残されるか削除されます。この条件の形式をHTMLタグの形式で具体的に選択しました。これにより、テキストエディターで編集するときに構文の強調表示が機能し、ペアのタグが見やすくなります。

このテンプレートを使用したフィード生成コードは次のようになります。


    result:='';
    //       NEWSFEED_ITEM      result
    for i:=0 to n-1 do begin
      id:=StrToIntDef(sa[i*c],0);
      title:=sa[i*c+1];
      cnt:=StrToIntDef(sa[i*c+2],1)-1;
      flags:=StrToIntDef(sa[i*c+3],0);
      //     
      db.Query('SELECT msg,created FROM messages WHERE topic=%d ORDER BY id LIMIT 1', 
        [id]);
      if db.lastErrorCode<>0 then continue;
      text:=db.Next;
      date:=db.NextDate;
      //    ( temp)
      temp.Put('NEWS_ID',id,true);
      temp.Put('NEWS_DATE',FormatDate(date,true),true);
      temp.Put('NEWS_TITLE',title,true);
      temp.Put('NEWS_PINNED',flags and 4>0,true);
      comLink:='$LNK_READ_MORE | ';
      if cnt>0 then comLink:=comLink+inttostr(cnt)+' $LNK_COMMENTS'
        else comLink:=comLink+'$LNK_LEAVE_COMMENT';
      temp.Put('NEWS_TEXT',text,true);
      temp.Put('COMMENTS',comLink,true);
      //   
      result:=result+BuildTemplate('#NEWSFEED_ITEM');
    end;

ローカリゼーション


テンプレートは、ローカライズにも便利です。これを行うには、グローバル（要求のコンテキスト内）clientLang変数を使用します。これは次のように機能します。リクエストハンドラは、クライアントがロシア語のページを必要とすることを発見すると、clientLangに値「RU」を書き込みます。その後、テンプレートトランスレータは、テキスト内で$ TEMPLATE_NAMEを見つけ、常に最初に$ TEMPLATE_NAME_RUを適用しようとします。したがって、ローカライズのために必要なのは、テキストを含む各テンプレートが別の言語用のバージョンを作成することだけです。

#TITLE_NEWS:News
#TITLE_NEWS_RU:
