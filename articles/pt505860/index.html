Spring Boot, Hibernate e Kotlin para iniciantes passo a passo


Olá pessoal, meu nome é Oleg, sou especialista técnico no DomKlik. Em nossa equipe, o núcleo da pilha é Kotlin e Spring Boot. Quero compartilhar com você minha experiência sobre a interação e os recursos do trabalho com o PostgreSQL e o Hibernate em conjunto com o Spring Boot e o Kotlin. Além disso, usando o exemplo de um microsserviço, mostrarei as vantagens do Kotlin e suas diferenças em relação a um aplicativo Java semelhante. Vou falar sobre as dificuldades não tão óbvias que os recém-chegados podem encontrar ao usar essa pilha com o Hibernate. Este artigo será útil para desenvolvedores que desejam mudar para o Kotlin e estão familiarizados com o Spring Boot, Hibernate Java.

Plugins


Para a aplicação no Kotlin, como colecionador de projetos, use o DSL Gradle Kotlin. A lista de plug-ins conectados será padrão para o Spring Boot e, para o Kotlin com o Hibernate, teremos vários novos:

plugins {
    id("org.springframework.boot") version "2.2.7.RELEASE"
    id("io.spring.dependency-management") version "1.0.9.RELEASE"
    kotlin("jvm") version "1.3.72"
    kotlin("plugin.spring") version "1.3.72"
    kotlin("plugin.jpa") version "1.3.72"
}

Considere os três últimos.

kotlin(«jvm»)- O plugin básico do Kotlin para trabalhar na JVM. Sem o qual nem um único aplicativo na pilha Java será iniciado.

kotlin(«plugin.spring»)- porque as classes padrão em Kotlin são finais, então este plugin irá automaticamente aulas marcadas com anotações @Component, @Async, @Transactional, @Cacheablee @SpringBootTestaberto a herança, e tópicos relacionados a este artigo, que irá permitir que as classes escritas em Kotlin ser proxy procuração CGLIB da Primavera.

É importante notar que as entidades que estão marcados anotações @Entity, @MappedSuperclasse @Embaddablenão se tornará opendepois de conectar o plugue. Além disso, get accessoreles também serão final, e depois vamos perder a oportunidade de trabalhar com entity reference. Para evitar isso e fazerEntitye seus campos open, adicione a build.gradle.kts:

allOpen {
   annotation("javax.persistence.Entity")
   annotation("javax.persistence.MappedSuperclass")
   annotation("javax.persistence.Embeddable")
}

kotlin(«plugin.jpa»)- Se os dois plugins anteriores forem aplicados a qualquer aplicativo no Kotlin + Spring Boot, o próximo já estará diretamente relacionado ao Hibernate. Entidades


Como em qualquer aplicativo com mais de uma entidade, criaremos um ancestral comum para todos entity.

BaseEntity:

@MappedSuperclass
abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

DepartmentEntity:

@Entity
@Table(name = "department")

       val name: String,

               mappedBy = "department",

               cascade = [CascadeType.ALL]
       )
       val employees: MutableList<EmployeeEntity> = mutableListOf()
) : BaseAuditEntity<Long>() {

   fun addEmployee(block: DepartmentEntity.() -> EmployeeEntity) {
       employees.add(block())
   }

   fun setEmployees(block: DepartmentEntity.() -> MutableSet<EmployeeEntity>) {
       employees.clear()

}

Nós não usamos classes de dados . Isso parece uma clara vantagem para o Kotlin sobre Java (anterior à versão 14), e há uma explicação para isso.

Por que não usar?


Classes de dados, além de ser final sobre a sua própria, têm certos campos para todos os campos equals, hashCodee toString. E isso é inaceitável em conjunto com o Hibernate.

Por quê? E também porque hashCodeé sempre igual a uma constante - a resposta está na documentação do próprio Hibernate . Especificamente, estamos interessados ​​nesta seção:

Embora o uso de um ID natural seja melhor para iguais e hashCode , às vezes você só possui o identificador de entidade que fornece uma restrição exclusiva.

É possível usar o identificador de entidade para verificação de igualdade, mas ele precisa de uma solução alternativa:

  • você precisa fornecer um valor constante para hashCode para que o valor do código de hash não seja alterado antes e depois da liberação da entidade.
  • você precisa comparar a igualdade do identificador de entidade apenas para entidades não transitórias.

Ou seja, você precisa comparar por natural id, ou, como no nosso exemplo, por primary key id. Isso evitará muitos problemas ao comparar a entidade e economizará sua perda ao usar a entidade como um elemento no Set.

A presença toString, definida por todos os campos, mata completamente toda a preguiça, por exemplo, ao registrar em diário uma entidade, uma vez que todos os campos serão inicializados para saída para uma sequência.

Dadas as características do Hibernate, essa funcionalidade do Kotlin não é adequada para nós.

Validade nula em campos de acordo com o DB apenas


Quanto à validade dos valores nullnos campos, nem tudo é tão simples. Anteriormente, mergulhamos um pouco nas profundezas do Hibernate: falando em plugin.jpa, mencionei o uso de um construtor sem argumentos ao inicializar uma entidade.

Ao inicializar campos, a reflexão também é usada. E se um valor foi armazenado no banco de dados na coluna correspondente null, a classe está calmamente inicializado com este campo com o valor null. Ao acessá-lo, corremos o risco de obter NPE, embora o campo esteja marcado como not nullable. Para evitar isso, é necessário monitorar o sincronismo da estrutura de tabelas e classes.

Se você observar o que é descrito nas duas últimas seções de maneira mais abrangente, essas regras se aplicarão não apenas às primitivas, mas também a várias entidades.

Repositórios


Ao usar o Kotlin, os repositórios prontos agora têm uma verificação de validação null. Portanto, se tivermos certeza de que, ao pesquisar departmentpor nome, o resultado será único e exclusivo, o tipo de retorno poderá ser especificado como non nullable:

interface DepartmentRepository : JpaRepository<DepartmentEntity, Long> {

   fun findOneByName(name: String) : DepartmentEntity
}

É DepartmentEntityindicado como único e não pode ser null. Se, por algum motivo, não encontrarmos o que procuramos department, não capturaremos o NPE, mas outra coisa:

org.springframework.dao.EmptyResultDataAccessException: Result must not be null!

Esse processamento é conseguido com a adição de suporte especializado da Kotlin aos MethodInvocationValidatore ReflectionUtilsnos dados comuns da primavera.

lateinit var


Outra característica do Kotlin que eu gostaria de considerar é lateinit var.

Adicionar uma nova classe ancestral: BaseAuditEntity.

@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class BaseAuditEntity<T> : BaseEntity<T>() {

   @CreatedDate
   @Column(updatable = false, nullable = false)
   lateinit var created: LocalDateTime

   @LastModifiedDate
   @Column(nullable = false)
   lateinit var modified: LocalDateTime
}

Considere o uso lateinit varde campos de auditoria ( created, modified) como um exemplo .

lateinit var- Este é um not nullcampo de inicialização atrasada. O acesso ao campo antes de sua inicialização gera um erro:

kotlin.UninitializedPropertyAccessException: lateinit property has not been initialized

Como regra, nos voltamos para os campos created, e modifiedmesmo depois, como uma entidade foi armazenada no banco de dados. Neste caso, os dados nestes polyahprostavlyayutsya a etapa de armazenamento e eles são not null, então lateinit varsomos mais do que adequados.
