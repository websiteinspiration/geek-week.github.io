Visual C ++でのUnicode文字のprintf関数形式指定子の悲しい運命

WindowsでのUnicodeサポートは、他のほとんどのオペレーティングシステムよりも早く登場しました。このため、Windowsでの文字の表現に関連する多くの問題は、他のシステムと同じ方法では解決されませんでした。その開発者は、新しい標準の導入をより良い時期まで延期しました[1]。最もわかりやすい例：Windowsでは、UCS-2エンコーディングがUnicode文字を表すために使用されます。バージョン1.0は65,536文字しかサポートしていないため、Unicodeコンソーシアムによって推奨されました[2]。5年後、コンソーシアムは考えを変えましたが、Win32、Windows NT 3.1、Windows NT 3.5、Windows NT 3.51、およびWindows 95はすでに市場にリリースされており、すべてがUCSエンコーディングを使用していたため、Windowsで何かを変更するには遅すぎました-2 [3]。

しかし、今日はprintf関数のフォーマット行について話します。

UnicodeはCよりも前にWindowsで採用されたため、Microsoft開発者はこの標準のサポートをCランタイムに実装する方法を理解する必要があり、その結果、wcscmp、wcschr、およびwprintfなどの機能が登場しました。printfでの文字列のフォーマットに関して、次の修飾子が導入されました。

  • ％sは、フォーマット文字列と同じ幅の文字列を表します。
  • ％Sは、フォーマット文字列の幅と逆の幅を持つ文字列を表します。
  • ％hsは、フォーマット文字列の幅に関係なく、通常の文字列を表します。
  • ％wsおよび％lsは、フォーマット文字列の幅に関係なく、ワイド文字列を表します。

アイデアは次のようなコードを書くことでした：

TCHAR buffer[256];
GetSomeString(buffer, 256);
_tprintf(TEXT("The string is %s.\n"), buffer);

そして、ANSIモードでコンパイルすると、次の結果が得られます。

char buffer[256];
GetSomeStringA(buffer, 256);
printf("The string is %s.\n", buffer);

そして、Unicodeモードでコンパイルするとき-このように[4]：

wchar_t buffer[256];
GetSomeStringW(buffer, 256);
wprintf(L"The string is %s.\n", buffer);

％s 指定子はフォーマット文字列と同じ幅の文字列を受け入れるため、このコードはANSI形式とUnicode形式の両方で正しく機能します。また、このソリューションでは、必要な幅の文字列が％s指定子の代わりに使用されるため、すでに記述されているコードをANSI形式からUnicode形式に簡単に変換できます。

Unicodeサポートが正式にC99に追加されたとき、C言語標準化委員会は、printf関数に別のフォーマット文字列モデルを採用しました。

  • ％sと％hsは通常の文字列を表します。
  • %ls .

それが問題の始まりです。その時までの過去6年間で、何十億行もの膨大な数のプログラムがWindows用に作成され、古い形式を使用していました。 Visual CおよびC ++コンパイラについてはどうですか？

世界中の既存のWindowsプログラムをすべて壊さないように、古い非標準モデルをそのまま使用することにしました。printf

の従来のルールとC標準のルールに従うランタイム環境の両方でコードを機能させる場合は、通常の文字列の％hs指定子とワイド文字列の％lsに制限する必要があります。この場合、フォーマット文字列がsprintf関数に渡されるかどうかに関係なく、結果の恒常性が保証されますまたはwsprintf。

#ifdef UNICODE
#define TSTRINGWIDTH TEXT("l")
#else
#define TSTRINGWIDTH TEXT("h")
#endif
TCHAR buffer[256];
GetSomeString(buffer, 256);
_tprintf(TEXT("The string is %") TSTRINGWIDTH TEXT("s\n"), buffer);
char buffer[256];
GetSomeStringA(buffer, 256);
printf("The string is %hs\n", buffer);
wchar_t buffer[256];
GetSomeStringW(buffer, 256);
wprintf("The string is %ls\n", buffer);

別のTSTRINGWIDTH定義を使用すると、たとえば次のコードを記述できます。

_tprintf(TEXT("The string is %10") TSTRINGWIDTH TEXT("s\n"), buffer);

人々は表形式の情報提示が好きなので、ここにあなたのための表があります。


Windowsで採用されている従来の形式と同じ方法でCで定義されている修飾子を使用して行を強調表示しました[5]。コードで両方の形式で同じ結果を生成する場合は、これらの修飾子を使用します。

注意

[1]他のシステムよりも前にWindowsにUnicodeを導入することで、Microsoftに最初の利点がもたらされたはずですが、少なくともUnicodeの場合、それは彼らにとって「先駆的な呪い」であることが判明しました。より有望な解決策（UTF-8エンコーディングなど）があり、その後システムにUnicodeが導入された場合。

[2]どうやら、彼らは65,536文字で十分だと信じていました。

[3]後でUTF-16に置き換えられました。さいわい、UTF-16は、両方のエンコーディングで表現できるコード文字について、UCS-2と下位互換性があります。

[4]正式には、Unicodeバージョンは次のようになります。

unsigned short buffer[256];
GetSomeStringW(buffer, 256);
wprintf(L"The string is %s.\n", buffer);

実際には、wchar_tはまだ独立した​​型ではなく、標準に追加されるまでは、unsigned shortの同義語でした。運命wchar_tのねじれと回転は、別の記事で見つけることができます。

[5] Windowsによって開発されたクラシックなフォーマットが最初に登場したため、C標準がそれに適応する必要があり、その逆ではない可能性が高かった。
