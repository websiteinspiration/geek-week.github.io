UI自動テスト用のセレン化されたシンプルで便利なテストフレームワークテンプレート

こんにちは。

この記事では、機能テストの自動化における私の経験を共有したいと思います。これは、便利で信頼性の高いテストフレームワークを作成することです。

使用するもの：Java、Selenide、Alure、TestNG、Maven。



前書き


GitHubプロジェクト-SelenideBoilerplate。

多くの場合、テスト自動化に関する記事では、例は現実とはかけ離れています。

driver.get (“URL”)
driver.find_element_by_id(“ID”).send_keys(“username”)
driver.find_element_by_id (“ID”).send_keys(“password”)
driver.find_element_by_id(“submit”).click()

期待、ページオブジェクトなどの多くの同様の例があります。その結果、初心者のテスターがすべてを正しく便利に整理するのは難しい場合があります。プロジェクトは松葉杖で覆われているため、新しいテストを作成して古いテストを維持することがますます難しくなっています。

私の意見では、冗長すぎて複雑すぎるツールもいくつかあります。

シンプルで便利で拡張が容易なテストフレームワークを紹介します。これは、通常のセレンよりもはるかに使いやすく、いくつかのプロジェクトでうまく使用できました。このプロジェクトが基本です。実際のプロジェクトでは、すべてが少し複雑です（並列化、remoteDriver、多くのテストなど）。

ツール


  • Selenideは、簡潔で安定したオープンソースUIテストを作成するためのライブラリです。Selenideは、タイムアウト、ロードに失敗したアイテムのクリックなどの問題のほとんどを解決します。StaleElementReferenceExceptionを忘れることもできます。非常に便利で簡単に習得できるツールで、セレンに戻りたくない作業に使用できます。
  • WebDriverManager - Selenideに含まれています。ブラウザーのドライバーのダウンロードとドライバーパスの設定のすべての作業を行うライブラリ-
     System.setProperty("webdriver.browser.driver", "/path_to_driver/driver");
  • レポートの魅力。
  • TestNGはテストフレームワークです。
  • Mavenはプロジェクトのアセンブリを自動化するためのツールです。

プロジェクトの構造




アプリとページの モジュールから始めましょう。

PageBuilderクラス


通常、例はすべてが1つのクラスに収まるかなり単純なページを提供します。しかし、実際のプロジェクトでは、非常に大きなページがあり、1つのクラスですべての機能を説明することは最良のアイデアではありません。たとえば、これはさまざまな支払いシステムのさまざまな形式の再充電ページである可能性があります。

この場合、ページをいくつかのクラスと要素（たとえば、バスケット内の製品）に分割し、すべてをページのメインクラスにまとめることをお勧めします。

したがって、すべてのページを1か所、つまりPageBuilderクラスで作成することをお勧めします。

package app;

import app.pages.LoginPage;

public class PageBuilder {

    public static LoginPage buildLoginPage() {
        return new LoginPage("/login");
    }

    public static BalancePage buildBalancePage() {
        DepositForm depositForm = new DepositForm();
        WithdrawalForm withdrawalForm = new WithdrawalForm();
        return new BalancePage("/balance", depositForm, withdrawalForm);
    }
}

クラスAppConfig


AppConfig クラスは、テスト中のWebアプリケーションの設定を格納します。たとえば、サイトのアドレス、テストユーザーなどです。このプロジェクトでは、これは単にサイトのアドレスです。

package app;

public class AppConfig {

    public static final String baseUrl = "https://google.com";

}

アプリクラス


これは、このモジュールのメインクラスです。Appクラスのコンストラクターでは、すべてのページが作成されます。

package app;

import app.pages.LoginPage;

public class App {

    public LoginPage loginPage;

    public App() {
        loginPage = PageBuilder.buildLoginPage();
    }
}

このアプローチのおかげで、テストで常にページオブジェクトを作成する必要はありません。必要なページが取得されるAppオブジェクトのみが作成されます。

また、Appクラスには、登録、登録、注文作成などのメソッドがある場合があります。

つまり、いくつかのページオブジェクトが含まれ、テストで必要になることが多い大規模な操作です。

ページオブジェクトに移りましょう


Selenideのおかげで、ページオブジェクトの操作は簡単です。すべてのページはBasePage基本クラスを継承します。相対ページURLは、ページオブジェクトコンストラクターに渡されます。

すべてのページ要素にはpublic access modifierがあるため、命令型と宣言型の両方のスタイルでテストを記述できます。また、要素から、テキストや属性などの必要なデータを取得できます。

ロケータは1か所にのみ保存されます。すべてのページロジックは、ページメソッドで記述する必要があります。

このアプローチでは、何かが壊れたり変更されたりした場合、ほとんどの場合、テストを書き直す必要はありません。メソッドが終了するか、ロケーターが実際のものに変更されます。


package app.pages;

import com.codeborne.selenide.SelenideElement;
import helpers.Driver;
import static com.codeborne.selenide.Selenide.*;

public class LoginPage extends BasePage {

    public SelenideElement loginField = $("#login__username");
    public SelenideElement passwordField = $("#login__password");
    public SelenideElement signInButton = $("#login_enter");
    public SelenideElement termsOfUseLabel = $("label[for=\"login_agree\"]");

    public LoginPage(String pageUrl) {
        super(pageUrl);
    }
    
    public void login(String email, String password) {
        loginField.setValue(email);
        passwordField.setValue(password);
        termsOfUseLabel.click();
        signInButton.click();
        Driver.waitForUrlContains("account/accounts");
    }
}

残り


ヘルパー モジュールには、2つの重要なクラスが含まれています

。TestConfig-このクラスから、テストを実行するための設定を取得できます。デフォルト設定もここに表示されます。

テストはコマンドによって実行され mvn test -Dbrowser=chrome -Dheadless=1

ます。変数の値はコマンドラインから取得され、TestConfigクラスのおかげで、テストとアプリケーションで使用できます。

たとえば、環境（dev、stage、production）に応じてアプリケーションのURLを変更できます。


package helpers;

public class TestConfig {

    public static String browser = "chrome";
    public static String headless = "1";

    public static void initConfig() {
        browser = System.getProperty("browser") == null ? "chrome" : System.getProperty("browser");
        headless = System.getProperty("headless") == null ? "1" : System.getProperty("headless");
    }

    public static boolean isHeadless() {
        return headless.contains("1");
    }
}

Driver クラスは、いくつかの便利なメソッドを備えたセレンおよびセレン化ドライバのラッパーです。

最も重要なメソッド：

Driver.initDriver（） -ここでドライバー/ブラウザーが初期化されます。

   public static void initDriver() {

        // Get settings from command line

        TestConfig.initConfig();

        // Set settings for selenide browser

        Configuration.pageLoadStrategy = "eager";
        Configuration.browserSize = "1920x1080";
        Configuration.holdBrowserOpen = false;
        Configuration.screenshots = false;

        if(TestConfig.isHeadless()) {
            Configuration.headless = true;
        } else {
            Configuration.headless = false;
        }

        switch (TestConfig.browser) {
            case "chrome":
                Configuration.browser = Browsers.CHROME;
                break;
            case "firefox":
                Configuration.browser = Browsers.FIREFOX;
                break;
            default:
                Configuration.browser = Browsers.CHROME;
                break;
        }
    }

テスト


すべてのテストクラスはA_BaseTestクラスから継承されます。このクラスでは、Appアプリケーションオブジェクトが作成され、ロガー、softAssert、ブラウザが開いたり閉じたりします。Cookieは各テストの後にクリアされます。エラーを記録できるA_BaseTestListener

もあります。 テストは次のようになります。読みやすく、メンテナンスも簡単。



import org.testng.annotations.Test;

public class ExampleTest extends A_BaseTest
{
    @Test
    public void loginViaEmail() {
        app.loginPage.open();
        app.loginPage.login("email@email.com", "passwords");
        
        logger.info("Sample info message");
               
        softAssert.assertEquals(2,2);
        softAssert.assertAll();
    }
}

テストクラスはtestng.xmlで指定されます。

test-outputフォルダーには、ログとスクリーンショット-Driver.takeScreenshot（）が含まれています。

レポートには、Allureが使用されます。テストが完了したら、コマンドallure serve target/allure-resultsを実行してレポートを表示できます。
