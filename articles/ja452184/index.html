アカウントベースのブロックチェーンの匿名性について

私たちは暗号通貨の匿名性のトピックに長い間関心を持っており、この分野の技術の発展を追跡しようとしています。私たちの記事では、Monero での機密取引の動作原理をすでに詳細に検討しており、この分野に存在するテクノロジーの比較レビューも実施しました。ただし、今日のすべての匿名暗号通貨は、Bitcoin-Unspent Transaction Output（以降、UTXO）によって提案されたデータモデルに基づいて構築されています。 Ethereumのようなアカウントベースのブロックチェーンの場合、匿名性とプライバシーの実装のための既存のソリューション（たとえば、MobiusやAztec）は、スマートコントラクトでUTXOモデルを繰り返すことを試みました。

2019年2月、スタンフォード大学とVisa Researchの研究者グループプレプリント「ゼザー：スマートコントラクトの世界におけるプライバシーに向けて」をリリース。著者らはまず、アカウントベースのブロックチェーンで匿名性を確保するアプローチを提案し、スマートコントラクトの2つのオプションを提示しました：機密（残高と送金金額を非表示）トランザクションと匿名（受信者と送信者を非表示）トランザクション。提案されたテクノロジーは興味深いものであり、そのデバイスを共有したいと思います。また、アカウントベースのブロックチェーンの匿名性の問題が非常に複雑であると考えられる理由と、著者がそれを完全に解決できたかどうかについて話します。



これらのデータモデルのデバイスについて


UTXOモデルでは、トランザクションは「入力」と「出力」で構成されます。 「出口」に直接類似するのは、ウォレットの紙幣です。各「出口」には特定の種類があります。誰かと（トランザクションを形成して）支払う場合、1つ以上の「出力」を使いますが、それらはトランザクションの「入力」になり、ブロックチェーンはそれらを使用済みとしてマークします。同時に、支払いの受取人（または、変更が必要な場合は自分）が、新しく生成された「出力」を受け取ります。概略的に、これは次のように表すことができます。



アカウントベースのブロックチェーンは、銀行口座のように構成されています。彼らはあなたの口座の金額と送金金額のみを操作します。アカウントから特定の金額を送金する場合、「出口」を燃やすことはありません。ネットワークは、どのコインが使用され、どのコインが使用されていないかを覚えておく必要はありません。最も単純なケースでは、トランザクションの検証は、送信者の署名と残高の金額をチェックすることに限定されます。



技術解析


次に、Zetherがトランザクションの量、受信者と送信者をどのように隠すかについて話します。彼の仕事の原則を説明する過程で、機密バージョンと匿名バージョンの違いに注意します。アカウントベースのブロックチェーンでプライバシーを確​​保する方がはるかに簡単なので、匿名化によって課せられる制限の一部は、テクノロジーの機密バージョンには関係しません。

ZetherはEl Gamal暗号化スキームを使用して、残高と送金金額を暗号化します。次のように動作します。アリスがボブbコインをアドレス（彼の公開鍵）Yに送信する場合、乱数rを選択し、金額を暗号化します。

ボブはこれらの値を受け取ると、同じ方法で暗号化して残高に追加するだけで、この方式に便利です。

同様に、アリスはバランスシートから同じ値を差し引き、公開鍵としてYのみを使用します。

宛先と送信者の隠蔽


UTXOでの「出口」の混在は、暗号通貨の黎明期に現れ、送信者を隠すのに役立ちます。これを行うために、送信者は、転送を行うときに、ブロックチェーンでランダムな「出口」を収集し、自分自身でこねます。次に、彼は「出口」にリング署名で署名します。これは、関与する「出口」の中に送信者のコインがあることを検証者に納得させる暗号化メカニズムです。もちろん、関係するコイン自体は無駄になりません。

ただし、受信者を隠すために、偽の「出力」を生成することはできません。したがって、UTXOでは、各「出口」には独自の一意のアドレスがあり、これらのコインの受取人のアドレスに暗号で関連付けられています。現時点では、一意の「出口」アドレスと受信者のアドレスの関係を、彼の秘密鍵を知らずに識別する方法はありません。

アカウントベースのモデルでは、1回限りのアドレスを使用できません（それ以外の場合は、「exit」のモデルになります）。したがって、受信者と送信者はブロックチェーン内の他のアカウントの間で練り合わせる必要があります。同時に、暗号化された0コインは、実際の残高を実際に変更することなく、こねられたアカウントから引き落とされます（または、受信者がこねる場合は0が追加されます）。

送信者と受信者の両方が常に恒久的なアドレスを持っているので、同じアドレスに混合するためにここで同じグループを使用することが必要になります。例を挙げて考えると簡単です。

アリスがボブの慈善活動に寄付することを決定したが、転送は外部の監視者に対して匿名のままであることを希望するとします。次に、送信者のフィールドで自分を偽装するために、アダムとアデルのアカウントも入力します。そして、ボブを隠すために-受信者フィールドでさらにベンとビルのアカウント。次の分割払いで、アリスは彼女の隣にアレックスとアマンダ、そしてボブの隣にブルースとベンゲンを入れることにしました。この場合、これらの2つのトランザクションでブロックチェーンを分析するとき、これらのトランザクションを匿名化する参加者の交差するペアが1つだけあります-アリスとボブ。



トランザクションレーシング


すでに述べたように、アカウントベースのシステムで残高を隠すために、ユーザーは残高と送金金額を暗号化します。同時に、アカウントの残高がマイナスにならないことを証明する必要があります。問題は、トランザクションを形成するときに、ユーザーが自分の現在のアカウントステータスに関する証拠を作成することです。しかし、ボブがアリスにトランザクションを送信し、それがアリスによって送信されるよりも早く受け入れられるとどうなりますか？その後、ボブのトランザクションの採用前にバランスの証明が作成されたため、アリスのトランザクションは無効と見なされます。



この状況で発生する最初の解決策は、トランザクションの前にアカウントを凍結することです。しかし、この方法は適切ではありません。分散システムでこのような問題を解決することの複雑さに加えて、匿名のスキームでは誰のアカウントをブロックするかが明確にならないためです。

この問題を解決するために、このテクノロジーは着信トランザクションと発信トランザクションを分離します。お金を使うとすぐにバランスシートに影響が出て、収入が延期されます。このために、「時代」の概念が導入されました-固定サイズのブロックのグループ。現在の「時代」は、ブロックの高さをグループのサイズで割ることによって決定されます。トランザクションを処理すると、ネットワークは送信者の残高を即座に更新し、受信者の資金をドライブに追加します。蓄積された資金は、新しい「時代」が始まるときにのみ、受取人が利用できます。

その結果、ユーザーは資金を受け取る頻度に関係なく（もちろん、残高が許す限り）トランザクションを送信できます。元号のサイズは、ブロックがネットワーク全体に広がる速度とトランザクションがブロックに落ちる速度に基づいて決定されます。

このソリューションは機密転送の場合にうまく機能しますが、匿名トランザクションの場合、後で説明するように、深刻な問題が発生します。

リプレイ攻撃保護


アカウントベースのブロックチェーンでは、各トランザクションは送信者の秘密鍵によって署名され、トランザクションが変更されておらず、このキーの所有者によって作成されたことを検証者に納得させます。しかし、伝送チャネルを聞いていた攻撃者がこのメッセージを傍受して、まったく同じ秒を送信した場合はどうなるでしょうか。検証者は、トランザクションの署名を検証し、その作成者を確信し、ネットワークは送信者の残高から同じ金額を引き落とします。

この攻撃は、リプレイ攻撃と呼ばれます。UTXOモデルでは、攻撃者は使用された出力を使用しようとしますが、それ自体は無効であり、ネットワークによって拒否されるため、このような攻撃は関係ありません。

これを防ぐために、トランザクションにランダムデータフィールドが挿入されます。これはノンスまたは単に「塩」と呼ばれます。ソルトを使用してトランザクションを再送信する場合、レビュー担当者はこのナンスが以前に使用されたかどうかを確認し、使用されていない場合は、このトランザクションが有効であると見なします。 nonceユーザーの履歴全体をブロックチェーンに保存しないようにするため、通常、最初のトランザクションではゼロであると見なされ、その後1ずつ増加します。ネットワークは、新しいトランザクションのnonceが過去と1つだけ異なることを確認できます。

匿名変換スキームでは、ナンストランザクションを検証する問題が発生します。 nonceを送信者のアドレスに明示的にバインドすることはできません。明らかに、これにより変換が匿名化されるためです。また、処理中の他の翻訳と競合する可能性があるため、すべての参加アカウントのnonceに1を追加することはできません。

ゼロ信頼の証拠


UTXOでは、送信者は彼が負の金額を費やしていないことをネットワークに証明する必要があります。そうしないと、空から新しいコインを生成することが可能になります（これが可能だった理由は、以前の記事の 1つで説明しました）。また、「入力」にリング署名を付けて、混練コインの中に彼に属する資金があることを証明します。

アカウントベースのブロックチェーンの匿名バージョンでは、証明のための式ははるかに複雑です。送信者は次のことを証明します。

  1. 送信された金額は正です。
  2. バランスは負ではありません。
  3. 送信者は、転送量（ゼロを含む）を正しく暗号化しました。
  4. 差引残高は送信者と受信者のみが変更します。
  5. 送信者は自分のアカウントから秘密鍵を所有しており、実際に（関係者の中で）送信者のリストに含まれています。
  6. トランザクションで使用されるナンスは正しく構成されています。

このような複雑な証明のために、著者はBulletproof（ちなみに、著者の1人がその作成に参加しました）とSigma-bulletと呼ばれるSigma プロトコルを組み合わせて使用​​しています。そのような声明の正式な証明はかなり難しい作業であり、テクノロジーの実装を取り上げたい人の数を大幅に制限します。

結果はどうですか？


私たちの意見では、アカウントベースのブロックチェーンにプライバシーをもたらすZetherの部分は、今では十分に使用されている可能性があります。しかし、現時点では、このテクノロジの匿名バージョンは、その使用と実装の複雑さに深刻な制限を課しています。しかし、著者がほんの数か月前にそれをリリースしたことを忘れないでください。多分誰かが今日の問題の解決策を見つけるでしょう。実際、これが科学のやり方です。
