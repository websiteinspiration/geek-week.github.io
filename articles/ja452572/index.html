MK-161のEFORTH：データ構造

この記事は、プログラマブル計算機に関するeForthの一連の記事の最後です。ここから始めます。

入力言語「Electronics MK-161」のコマンドは、ファイルeForth0.mklの半分しか占有しません。後半はテーブルで占められており、その開発はトランスレータのアルゴリズム部分を書くことと同じくらい困難ではありませんでした。これらのテーブルがどのように使用されるかを理解してみましょう。


ワース教授は、「小規模なプログラミング」は、アルゴリズムとデータ構造という2つの同等に重要なコンポーネントを開発することからなると教えています。

すでに1つのeForthデータ構造に遭遇しました。これは、バイトメモリにあるVCA（高レベルワード）の本体です。4つのハンドラーは、「自分の」VCAのパラメーターフィールドをさまざまな方法で解釈します。

.DB DOVAR ;     
.DB … ;     

.DB DOCON ;   
.DW _ ;  

.DB DOCONM ;    
.DW _ ;  

.DB DOLST ;    
.DW 1, 2,… EXITT ;

次の比較的単純なデータ構造は、TYPEの「標準メッセージ」に関連付けられています。すべてのeForthメッセージには番号が付けられ、安価なプログラムメモリに転送されます。単語TYPEが1文字を出力する場合、そのコードは、そのようなメッセージの番号（0〜7）になります。

;   TYPE
		.BASE
tblTYPE:	.DBB str7,str6, str5, str4, str3, str2, str1, str0

拡張MK言語では、.BASE疑似コマンドが.DBBコマンドの「ベース」を設定し、str7、str6などのラインオフセットを順番に配置します。基本ラベルtblTYPEを基準にしています。テーブルアドレスに0から7までの数を追加すると、このオフセットを読み取ることができます。見つかったオフセットをtblTYPEに追加すると、目的の行のアドレスが取得されます。

文字列の最初のバイトには、その長さが含まれています。 eForthは、このようなカウントラインを広範囲に使用します。

また、208個すべての埋め込みワードのコードアドレスをリストするtblTokensテーブルも見つけました。単語がプリミティブでない場合、テーブルには0が含まれます。アドレス0に移動すると、eForthがきしむように再起動します。

同じ208語の名前を参照するテーブルtblNamesについても説明しました。カウント行の形式のこれらの名前は、同じ「ゴム」プログラムメモリに保存されます。 tblNamesテーブル自体は、eForthの実行中は使用できませんが、それに含まれる情報は失われません。コンパイル時に、eForth.fは名前のアドレスを10進メモリーに格納されたより便利なデータ構造に転送します（2を参照）。

電卓の画面に文字を表示するときの制御コードの連想テーブルであるtblCHPUTについても話しました。 tblKeyNumからtblKeyRusFまでの別の7つのテーブルは、さまざまなキーボードモードで押されたボタンのコードを8ビットの文字コードに変換します。アクティブなキーボードモードを担当するルーチンのアドレスは、10進レジスターptrKbdIntにあります。

全体として、eForth0.mklファイルに1つのデータ構造のみがアセンブルされずに残っています。これらは名前認識テーブルです。メインコースの後にデザート（5を参照）に残しておきましょう。2つの見出しテーブルが10進メモリーに保存されています。最初に、これらの見出しを「詰め込む」ためのツールを用意します。

1.見出しを操作する：HEAD！とHEAD @


HEAD!      ( xt nfa r -- )              r,  xt  nfa.
HEAD@      ( r -- xt nfa lex )          r,  xt, nfa  .

1つの小数点レジスタMK-161は、小数点以下12桁を記憶できます。 eForthはこのレジスタを使用して、それぞれ0〜9999の3つの小さな数値を格納します。これらの数値を格納するための3つの「フィールド」は、A、B、Cと呼ばれます：AAAABBBBCCCC。小数点記号はフィールドAのみを参照します

。HEAD@プリミティブはレジスタ番号を取得し、そこからフィールドに分割し、HEAD！フィールドを長い数で収集し、結果の「モンスター」を指定されたレジスターに書き込みます。しかし、ニュアンスがあります。

単語の「10進見出し」のフィールドAには、彼の名前（nfa）のアドレスが含まれています。このアドレスが負の場合、名前はプログラムメモリに保存されます。フィールドBには、トークン（xt）という単語が含まれています。フィールドCはレキシコンと呼ばれます。これは、IMMEDIATEビットと、単語がコンパイルのみを目的としていることを示す記号を格納します。

HEAD @は、ヘッダーを複数の部分に分割します。スタックの一番上にはレキシコンフィールドCがあり、その下に名前フィールドAがあります。通常トークンを格納するフィールドBは一番下にあります。

頭！フィールドCをリセットします。

2.インライン見出し




208個の組み込みワード（0から207）のそれぞれのヘッダーは、R44から順に並んでいます。これらの単語の名前はプログラムメモリにハードコードされているため、フィールドAには常に負の数が含まれます。

フィールドBおよびCは編集可能です。したがって、ユーザーは組み込みの単語を再定義し、それらから必要なIMMEDIATEを作成できます（4を参照）。

3.ユーザーの見出し




定義済みの名前を208個だけ使用すると、バイトメモリが節約されますが、非常に退屈です。したがって、名前を選択する際のファンタジーが32文字のみに制限される別のデータ構造を開発しました。この構造は32のリストで構成されており、それぞれが特定の長さのユーザーの単語を処理します。これらの32個のリストにはそれぞれ個人的な見出しがあります。リスト自体は10進数のメモリを介して「ジャンプ」しますが、それらのヘッダーは常にR301 ... R332に格納されます。

名前で単語を並べ替えることは、161eForthの重要なハイライトです。ソートを行うと、名前で単語を検索するときの比較回数が大幅に減り、コンパイルが高速化されます。各名前の長さがわかっている場合、誰がハッシュ関数を必要としますか？

簡単にするために、リストの見出し単語のタイトルとしてフィールドA、B、Cを持つ同じ構造を持っています。これらのフィールドの目的は異なります。フィールドAには、リスト内の最初のレジスタの番号が含まれています。フィールドBには、リストに提供されるレジスターの数が含まれています。フィールドCには、見出しがすでにリストにある単語の数が格納されます。

作業の始めに、フィールドCはゼロに等しく、単語はすべてのリストにありません。フィールドBは2で、各リストにはスターター用のいくつかのレジスターが与えられます。フィールドAは、R333で始まる2つのレジスタのブロックを示します。

各リストには、単語の見出しが含まれています。それらはすでに分解されています（1を参照）。ここでは、おそらく、名前（nfa）のアドレスは正であり、従来はVCAの本体の前に格納されていたカウントラインを指します。また、フィールドBのトークンは、この名前の直後にバイナリメモリに入るコードフィールド（cfa）のアドレスです。 1つの例外は単語がすでに定義されている場合、フィールドAは古い名前を指します。なぜ文字列をもう一度保存するのですか？バイナリメモリは高価です。

リストのすべてのレジスタがいっぱいになると（B = C）、単語PUBLISHはさらに5つの空き場所を提供し、このデータ構造を適切な場所にプッシュし、リストヘッダーのリンク（A）を調整します。

4.新しい単語の公開：WORKおよびPUBLISH


LAST       ( -- a )                      .
WORK       ( -- a )                     .
PUBLISH    ( -- )                       .
$,n        ( nfa -- )                   ,    nfa.
?UNIQUE    ( a -- a )                ,    .

MK-161用に開発された単語のタイトルを格納するためのデータ構造は実用的であり、eForthに簡単に統合できました。 CREATE、CONSTANT、または：が新しい単語を作成するとき、システムワード$、nにアクセスして、指定された名前の単語のタイトルを作成します。 $、nは参照しますか？検証のための一意-新しい単語を作成するか、古い単語を再定義しますか？

同じ名前の単語が既に存在する場合、UNIQUEはこれについてユーザーに警告します。同時に、再定義されたヘッダーのアドレスがシステム変数LASTに入力されます。新しい単語の場合、LASTはリセットされます。

いずれの場合も、$、nはWORK変数に新しいヘッダーを作成します。これは、ヘッダーの12ビットを格納できる10進レジスターです。名前が見つからなかった場合、86eForthや他の多くの砦で行われているように、コードフィールドの前の辞書に含まれます。MK-161は、「通信フィールド」なしでなんとかできました。これにより、バイナリメモリも節約されます。

PUBLISHプリミティブは、単語の定義を完了します。コロンワードをコンパイルする場合、PUBLISHが呼び出されます;その結果、SMUDGEビットは必要ありませんでした。WORKのヘッダーがコピーされる場所は、LAST変数によって決定されます。LASTがゼロの場合、対応するリストに新しいヘッダーが作成されます（3を参照）。リストは完全ですか？次に、PUBLISHはさらに5つのレジスタを追加します。そのうち4つは将来のために追加されます。

PUBLISHの実行後、LAST変数は常に最後の単語のタイトルを指します。これは、IMMEDIATEが語彙フィールドを変更することによってその仕事をするのに役立ちます。

5.（FIND）および名前認識テーブル


(FIND)     ( a -- r T | a F )          r,        a.
FIND       ( a -- a F | xt 1 | xt -1)    .  1,  IMMEDIATE.

プリミティブ（FIND）は、名前による単語の検索を管理します。まず、以前に既知の名前を持つ組み込みの単語の中から名前を検索し、次に、目的の名前の長さを持つユーザーの単語のリストを確認します（3を参照）。名前認識テーブルは、この「最初の」作業を大幅に加速します。ここではそれらがどのように機能するかを示します。

最初（FIND）に、tblLen配列でメインの連想テーブルのアドレスを検索します。このテーブルには、必要な長さの既知の名前が「準備」されています。このテーブル（FIND）では、名前の最初の文字を検索します。ほとんどの場合、これにより、検索した単語のタイトルレジスタ番号を最初の文字と長さですぐに見つけることができます。

同じ長さのいくつかの単語の最初の文字が同じである場合があります。次に、レジスタ番号（FIND）の代わりに、次の連想テーブル（読み取り番号は300以上）のアドレスに遭遇し、2番目の文字で検索が続行されます。など、単語が見つかるか、そのような単語が存在しないことが確認されるまで。

もちろん、最初の文字が一致した後（FIND）、名前全体が検証されます。しかし、認識テーブルはeForthを高速にしました。この春、私はそれらに多くの時間を費やし、今では検索時間を節約しています。それらの「キー」はアルファベット順にソートされています。申し訳ありませんが、ファームウェアMK-161はそれを吐き出します。

互換性を保つため、Fort ANS [4]のFINDという単語を実装しました。これは、「ブラックワーク」プリミティブ（FIND）を信頼しています。単語はすでに考慮されていますか？UNIQUEは（FIND）を通じてその引数も検索します。
