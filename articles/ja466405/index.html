JoeがJimにメッセージを送信すると、フレームはスイッチAに送信されます。スイッチAはJimのMACアドレスを知らないため、Joeがメッセージを受信したポートを除くすべてのポートを介してブロードキャストフレームを送信します。ブロードキャストフレームがスイッチBのポートで受信されると、1つのインターフェースに到着したパケットはジムに送信され、2番目のインターフェースに到着したパケットは最初のポートに転送されてスイッチAに送信されます。



同時に、最初のインターフェースに到着した要求が転送されます。 2番目のポートもスイッチAを切り替えます。

スイッチAには、f0 / 1、f0 / 2およびf0 / 3の3つのインターフェイスがあります。ユーザーがAAA MACアドレスを持つコンピューターを持っていて、スイッチAにブロードキャストフレームを送信するとします。スイッチは、f0 / 1インターフェイスを介してこのフレームを受け入れます。コンピューターにBBB MACアドレスを持つ別のユーザーがネットワーク上にいます。したがって、ソースアドレスAAAと宛先アドレスBBBがあります。



スイッチAはBBBの宛先MACアドレスに到達する方法を認識していませんが、AAAの送信元MACアドレスがf0 / 1インターフェイスを介して到達できることを認識しており、それに関するレコードをMACアドレステーブルに書き込みます。次に、スイッチAは宛先アドレスの要求を他の2つのインターフェイス（f0 / 2およびf0 / 3）に送信します。

スイッチAは、AAA要求を受信すると、送信元f0 / 2から送信されたことを確認し、AAAデバイスがf0 / 2インターフェイスを介して到達可能であることをMACアドレステーブルに記録します。さらに、BB0宛先がf0 / 1インターフェースに対応しているという記録をすでに持っているので、彼はリクエストを宛先に送信します。



スイッチAはブロードキャストフレームを送信したため、f0 / 2インターフェイスを介してスイッチBに送信されるだけでなく、f0 / 3インターフェイスを介してスイッチCにも送信され、スイッチCがスイッチBのf0 / 3インターフェイスに転送しました。



フレームを受信すると、スイッチBはこのように考えています。「AAAソースが以前にf0 / 2インターフェイスに配置されていたことがわかっていますが、これからのフレームはf0 / 3インターフェイスを介して届いたため、MACアドレステーブルを更新してf0 / 2をf0 / 3 "。



次に、フレームはスイッチAに戻り、非常に「驚き」ます。以前は、スイッチAはAAA送信元がf0 / 1インターフェイスによって接続されていたと考えていましたが、今度はメッセージがf0 / 2インターフェイスから送信されたことがわかります。スイッチBとCを介したフレームの逆方向で、スイッチAは再び彼を混乱させるメッセージを受け取ります-今度は、AAAソースがf0 / 3インターフェイス上にあることがわかります。



したがって、このスイッチのMACアドレステーブルは、これら3つのインターフェイス間で常に更新されます。つまり、前述のMACアドレステーブルが不安定になるという問題が発生します。最初のケースと同様に、ここでフレームループが形成され、数秒ごとにテーブルが更新されます。

3番目のループ問題があります-フレームの複数のコピーです。ユーザーAAAはフレームをスイッチAに送信し、それをf0 / 2インターフェイスを介してスイッチBに送信します。スイッチBは、フレームをf0 / 1インターフェイスを介してBBB宛先に配信します。問題ありません。

ただし、同時に、スイッチAは2番目のインターフェイスf0 / 3を介してスイッチCに同じフレームを送信し、スイッチCはスイッチBに転送します。スイッチCからパケットまたはフレームを受信すると、スイッチBはそれがBBBにアドレス指定されていることを確認し、それを宛先に送信します。したがって、BBBユーザーは同じパケットを2回受信します。ここで問題が発生します-これがアプリケーションによって実行されたデータの分散である場合、同じフレームがユーザーに2回表示されることはありません。

これらは、フレームループが引き起こす可能性のある3つの問題です。これらすべては、STPプロトコルを使用して、前に説明した1つの方法で解決されます。前のビデオの1つでは、彼の番号を覚えていません。ポートモードについて説明したとき、このプロトコルはすでに言及されており、ネットワークでのトラフィックループを防止するのに役立ちます。

したがって、3つのデバイスが相互に接続されて閉じたネットワークループを形成し、同じブロードキャストドメインに属すると、ループが形成されます。この場合、STPプロトコルで使用されるアルゴリズムは、スイッチの1つをルートスイッチ（ルートブリッジ）として指定します。ルートスイッチとしてスイッチAを選択します。

ルートスイッチが他のスイッチと交換する情報は、BPDUと呼ばれます。デバイスは2秒ごとにBPDUメッセージを交換します。この時間は「ハロータイマー」と呼ばれます。 BPDUメッセージには、ルートスイッチのBIDとルートスイッチへのルートのコスト、またはルートパスコスト（これは実際にはルートスイッチまでの距離です）が含まれています。各ポートのパスのコストは、ルートスイッチへの最短パスを計算するのに役立ちますが、この概念については詳しく説明しません。

IEEE 802.1d標準であるSTPプロトコルの仕組みを次に示します。これは非常に古い標準であり、その欠点は、接続が切断されたときに情報を更新するための最大時間であり、50秒に相当します。ブロッキングポートブロッキングステートに加えて、さらに2つの中間ステート-リスニングとラーニングをサポートし、その後、転送転送ステートに切り替わります。

この状況を改善するために、別の標準が導入されました-IEEE 802.1w、またはRapid STP-RSTPと呼ばれる高速STPプロトコル。中間状態がなくなり、ブロッキング状態から転送状態に移行します。

シスコがRSTPを作成してSTPを改良したように、PVSTPの「高速化」バージョン-RPVSTPを開発しました。独自のISLプロトコルを使用してカプセル化されたこれらのプロトコルはどちらも、採用前に開発されたため、802.1q標準をサポートしていませんでした。相互運用性を向上させるために、シスコでは、802.1qサポートを追加してこれらのプロトコルを強化しています。 ISLと802.1qの両方をサポートする新しいプロトコルは、PVSTP +およびRPVSTP +と呼ばれます。現在、これらはシスコネットワークの業界標準です。
STPプロセスは、各ポートコストポートでのパスコストメトリックによって特徴付けられます。このインジケーターの基礎として、ポート速度の特性が使用されました-ポート速度（mb / s）。したがって、IEEE 1998標準によれば、10 Mb / sの速度はポート100のコストに対応し、100 Mb / sの速度-19、1 Gb / sのコスト-4および10 Gb / sのコスト-2のコスト。この標準は100 Gb / sの速度を考慮していません1 Tb / sなので、2004年に新しいIEEEが開発されました。ポートのコストの相対的な指標は200万から20です。

ビットが0001の順序で配置されている場合、これは数値4096が1の下にあることを意味します。つまり、優先度は4096になります。ビットの16の組み合わせに応じて、これらの数値の1つが優先度として使用されます。前のものより4096多い。

デフォルトでは、すべてのCiscoスイッチの優先順位は32768ですが、これらの番号のいずれかを優先順位として選択できます。拡張システムIDを使用する場合、VLAN番号がこの番号に追加されます。つまり、VLAN1を使用している場合、ブリッジIDの優先順位は32768 + 1 = 32769
