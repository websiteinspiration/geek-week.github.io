サイト統計と小さなリポジトリ

WebalizerユーティリティとGoogle Analyticsツールは、何年もの間、ウェブサイトで何が起こっているのかを理解するのに役立ちました。今では、それらが有用な情報をほとんど提供しないことを理解しています。 access.logファイルにアクセスできるので、統計を把握し、sqlite、html、sql、および任意のスクリプトプログラミング言語などの非常に基本的なツールを実装するのは非常に簡単です。

Webalizerのデータソースは、サーバーのaccess.logファイルです。これは、列と数値がどのように見えるかで、トラフィックの合計量のみが明確です。



Googleアナリティクスなどのツールは、読み込まれたページから独自にデータを収集します。彼らは私たちにいくつかの図と線を示していますが、それに基づいて正しい結論を出すのはしばしば困難です。おそらくもっと努力が必要でしたか？私は知らない。

では、サイト訪問の統計で何を見たいのでしょうか？

ユーザーとボットのトラフィック


多くの場合、サイトトラフィックには制限があり、使用されている有用なトラフィックの量を確認する必要があります。たとえば、次のようになります。



SQLレポート要求
SELECT
1 as 'StackedArea: Traffic generated by Users and Bots',
strftime('%d.%m', datetime(FCT.EVENT_DT, 'unixepoch')) AS 'Day',
SUM(CASE WHEN USG.AGENT_BOT!='n.a.' THEN FCT.BYTES ELSE 0 END)/1000 AS 'Bots, KB',
SUM(CASE WHEN USG.AGENT_BOT='n.a.' THEN FCT.BYTES ELSE 0 END)/1000 AS 'Users, KB'
FROM
  FCT_ACCESS_USER_AGENT_DD FCT,
  DIM_USER_AGENT USG
WHERE FCT.DIM_USER_AGENT_ID=USG.DIM_USER_AGENT_ID
  AND datetime(FCT.EVENT_DT, 'unixepoch') >= date('now', '-14 day')
GROUP BY strftime('%d.%m', datetime(FCT.EVENT_DT, 'unixepoch'))
ORDER BY FCT.EVENT_DT


グラフから、ボットの一定のアクティビティを確認できます。最も活発な代表者を詳細に研究することは興味深いでしょう。

迷惑なボット


ユーザーエージェント情報に基づいてボットを分類します。毎日のトラフィックに関する追加の統計、成功したリクエストと失敗したリクエストの数は、ボットの活動についての良い考えを与えます。



SQLレポート要求
SELECT 
1 AS 'Table: Annoying Bots',
MAX(USG.AGENT_BOT) AS 'Bot',
ROUND(SUM(FCT.BYTES)/1000 / 14.0, 1) AS 'KB per Day',
ROUND(SUM(FCT.IP_CNT) / 14.0, 1) AS 'IPs per Day',
ROUND(SUM(CASE WHEN STS.STATUS_GROUP IN ('Client Error', 'Server Error') THEN FCT.REQUEST_CNT / 14.0 ELSE 0 END), 1) AS 'Error Requests per Day',
ROUND(SUM(CASE WHEN STS.STATUS_GROUP IN ('Successful', 'Redirection') THEN FCT.REQUEST_CNT / 14.0 ELSE 0 END), 1) AS 'Success Requests per Day',
USG.USER_AGENT_NK AS 'Agent'
FROM FCT_ACCESS_USER_AGENT_DD FCT,
     DIM_USER_AGENT USG,
     DIM_HTTP_STATUS STS
WHERE FCT.DIM_USER_AGENT_ID = USG.DIM_USER_AGENT_ID
  AND FCT.DIM_HTTP_STATUS_ID = STS.DIM_HTTP_STATUS_ID
  AND USG.AGENT_BOT != 'n.a.'
  AND datetime(FCT.EVENT_DT, 'unixepoch') >= date('now', '-14 day')
GROUP BY USG.USER_AGENT_NK
ORDER BY 3 DESC
LIMIT 10


この場合、分析の結果、robots.txtをファイルに追加してサイトへのアクセスを制限する決定が下され、 最初の2つのボットがテーブルから消え、MSロボットが最初の行から下に移動しました。

最も活動の多い日時


トラフィックはアップを示しています。それらを詳細に検討するためには、それらの発生時間を割り当てる必要がありますが、時間測定のすべての時間と日を表示する必要はありません。したがって、詳細な分析が必要な場合は、ログファイルで個々のクエリを見つけやすくなります。



SQLレポート要求
SELECT
1 AS 'Line: Day and Hour of Hits from Users and Bots',
strftime('%d.%m-%H', datetime(EVENT_DT, 'unixepoch')) AS 'Date Time',
HIB AS 'Bots, Hits',
HIU AS 'Users, Hits'
FROM (
	SELECT
	EVENT_DT,
	SUM(CASE WHEN AGENT_BOT!='n.a.' THEN LINE_CNT ELSE 0 END) AS HIB,
	SUM(CASE WHEN AGENT_BOT='n.a.' THEN LINE_CNT ELSE 0 END) AS HIU
	FROM FCT_ACCESS_REQUEST_REF_HH
	WHERE datetime(EVENT_DT, 'unixepoch') >= date('now', '-14 day')
	GROUP BY EVENT_DT
	ORDER BY SUM(LINE_CNT) DESC
	LIMIT 10
) ORDER BY EVENT_DT


グラフの最初の日の最もアクティブな時間11、14、20を観察します。しかし、翌日の13時にボットはアクティブでした。

1週間あたりの1日の平均ユーザーアクティビティ


アクティビティとトラフィックが少しわかった。次の質問はユーザー自身の活動でした。このような統計では、たとえば1週間などの長い集計期間が望ましいです。



SQLレポート要求
SELECT
1 as 'Line: Average Daily User Activity by Week',
strftime('%W week', datetime(FCT.EVENT_DT, 'unixepoch')) AS 'Week',
ROUND(1.0*SUM(FCT.PAGE_CNT)/SUM(FCT.IP_CNT),1) AS 'Pages per IP per Day',
ROUND(1.0*SUM(FCT.FILE_CNT)/SUM(FCT.IP_CNT),1) AS 'Files per IP per Day'
FROM
  FCT_ACCESS_USER_AGENT_DD FCT,
  DIM_USER_AGENT USG,
  DIM_HTTP_STATUS HST
WHERE FCT.DIM_USER_AGENT_ID=USG.DIM_USER_AGENT_ID
  AND FCT.DIM_HTTP_STATUS_ID = HST.DIM_HTTP_STATUS_ID
  AND USG.AGENT_BOT='n.a.' /* users only */
  AND HST.STATUS_GROUP IN ('Successful') /* good pages */
  AND datetime(FCT.EVENT_DT, 'unixepoch') > date('now', '-3 month')
GROUP BY strftime('%W week', datetime(FCT.EVENT_DT, 'unixepoch'))
ORDER BY FCT.EVENT_DT


1週間の統計によると、平均して1人のユーザーが1日あたり1.6ページを開いています。この場合、ユーザーごとに要求されるファイルの数は、サイトに新しいファイルを追加することによって異なります。

すべてのリクエストとそのステータス


Webalizerは常に特定のページコードを表示し、常に成功したリクエストとエラーの数だけを見たいと思っていました。



SQLレポート要求
SELECT
1 as 'Line: All Requests by Status',
strftime('%d.%m', datetime(FCT.EVENT_DT, 'unixepoch')) AS 'Day',
SUM(CASE WHEN STS.STATUS_GROUP='Successful' THEN FCT.REQUEST_CNT ELSE 0 END) AS 'Success',
SUM(CASE WHEN STS.STATUS_GROUP='Redirection' THEN FCT.REQUEST_CNT ELSE 0 END) AS 'Redirect',
SUM(CASE WHEN STS.STATUS_GROUP='Client Error' THEN FCT.REQUEST_CNT ELSE 0 END) AS 'Customer Error',
SUM(CASE WHEN STS.STATUS_GROUP='Server Error' THEN FCT.REQUEST_CNT ELSE 0 END) AS 'Server Error'
FROM
  FCT_ACCESS_USER_AGENT_DD FCT,
  DIM_HTTP_STATUS STS
WHERE FCT.DIM_HTTP_STATUS_ID=STS.DIM_HTTP_STATUS_ID
  AND datetime(FCT.EVENT_DT, 'unixepoch') >= date('now', '-14 day')
GROUP BY strftime('%d.%m', datetime(FCT.EVENT_DT, 'unixepoch'))
ORDER BY FCT.EVENT_DT


レポートには、LINE_CNTとは異なり、クリック（ヒット）ではなくリクエストが表示されます。REQUEST_CNTメトリックはCOUNT（DISTINCT STG.REQUEST_NK）と見なされます。目標は、効果的なイベントを表示することです。たとえば、MSボットが1日数百回robots.txtファイルをポーリングし、この場合、そのようなポーリングは1回だけカウントされます。これにより、チャート上のジャンプをスムーズにすることができます。

グラフから、多くのエラーを確認できます-これらは存在しないページです。分析の結果、リモートページからのリダイレクトが追加されました。

誤ったリクエスト


リクエストの詳細なレビューのために、詳細な統計を表示できます。



SQLレポート要求
SELECT
  1 AS 'Table: Top Error Requests',
  REQ.REQUEST_NK AS 'Request',
  'Error' AS 'Request Status',
  ROUND(SUM(FCT.LINE_CNT) / 14.0, 1) AS 'Hits per Day',
  ROUND(SUM(FCT.IP_CNT) / 14.0, 1) AS 'IPs per Day',
  ROUND(SUM(FCT.BYTES)/1000 / 14.0, 1) AS 'KB per Day'
FROM
  FCT_ACCESS_REQUEST_REF_HH FCT,
  DIM_REQUEST_V_ACT REQ
WHERE FCT.DIM_REQUEST_ID = REQ.DIM_REQUEST_ID
  AND FCT.STATUS_GROUP IN ('Client Error', 'Server Error')
  AND datetime(FCT.EVENT_DT, 'unixepoch') >= date('now', '-14 day')
GROUP BY REQ.REQUEST_NK
ORDER BY 4 DESC
LIMIT 20


このリストには、たとえば/wp-login.phpへのリクエストなど、すべてのダイヤラが含まれます。サーバーによるリクエストの書き換えに関するルールを調整することにより、そのようなリクエストに対するサーバーの応答を調整して、スタートページに送信できます。

そのため、サーバーログファイルに基づくいくつかの簡単なレポートは、サイトで何が起こっているかをかなり完全に示しています。

キープロパガンダ


生データがデータベースにある場合、測定テーブルにはないキーを記録する必要があります。その後、測定値への参照を構築することが可能になります。たとえば、DIM_REFERRERテーブルでは、キーは3つのフィールドの組み合わせです。

SQLキー伝搬クエリ
/* Propagate the referrer from access log */
INSERT INTO DIM_REFERRER (HOST_NK, PATH_NK, QUERY_NK, UPDATE_DT)
SELECT
	CLS.HOST_NK,
	CLS.PATH_NK,
	CLS.QUERY_NK,
	STRFTIME('%s','now') AS UPDATE_DT
FROM (
	SELECT DISTINCT
	REFERRER_HOST AS HOST_NK,
	REFERRER_PATH AS PATH_NK,
	CASE WHEN INSTR(REFERRER_QUERY,'&sid')>0 THEN SUBSTR(REFERRER_QUERY, 1, INSTR(REFERRER_QUERY,'&sid')-1) /*  sid -   */
	ELSE REFERRER_QUERY END AS QUERY_NK
	FROM STG_ACCESS_LOG
) CLS
LEFT OUTER JOIN DIM_REFERRER TRG
ON (CLS.HOST_NK = TRG.HOST_NK AND CLS.PATH_NK = TRG.PATH_NK AND CLS.QUERY_NK = TRG.QUERY_NK)
WHERE TRG.DIM_REFERRER_ID IS NULL

ユニットテーブル


最後に、集計テーブルをロードします。たとえば、次のように日次テーブルをロードできます。

SQL集約ロード要求
/* Load fact from access log */
INSERT INTO FCT_ACCESS_USER_AGENT_DD (EVENT_DT, DIM_USER_AGENT_ID, DIM_HTTP_STATUS_ID, PAGE_CNT, FILE_CNT, REQUEST_CNT, LINE_CNT, IP_CNT, BYTES)
WITH STG AS (
SELECT
	STRFTIME( '%s', SUBSTR(TIME_NK,9,4) || '-' ||
	CASE SUBSTR(TIME_NK,5,3)
	WHEN 'Jan' THEN '01' WHEN 'Feb' THEN '02' WHEN 'Mar' THEN '03' WHEN 'Apr' THEN '04' WHEN 'May' THEN '05' WHEN 'Jun' THEN '06'
	WHEN 'Jul' THEN '07' WHEN 'Aug' THEN '08' WHEN 'Sep' THEN '09' WHEN 'Oct' THEN '10' WHEN 'Nov' THEN '11'
	ELSE '12' END || '-' || SUBSTR(TIME_NK,2,2) || ' 00:00:00' ) AS EVENT_DT,
	BROWSER AS USER_AGENT_NK,
	REQUEST_NK,
	IP_NR,
	STATUS,
	LINE_NK,
	BYTES
FROM STG_ACCESS_LOG
)
SELECT
	CAST(STG.EVENT_DT AS INTEGER) AS EVENT_DT,
	USG.DIM_USER_AGENT_ID,
	HST.DIM_HTTP_STATUS_ID,
	COUNT(DISTINCT (CASE WHEN INSTR(STG.REQUEST_NK,'.')=0 THEN STG.REQUEST_NK END) ) AS PAGE_CNT,
	COUNT(DISTINCT (CASE WHEN INSTR(STG.REQUEST_NK,'.')>0 THEN STG.REQUEST_NK END) ) AS FILE_CNT,
	COUNT(DISTINCT STG.REQUEST_NK) AS REQUEST_CNT,
	COUNT(DISTINCT STG.LINE_NK) AS LINE_CNT,
	COUNT(DISTINCT STG.IP_NR) AS IP_CNT,
	SUM(BYTES) AS BYTES
FROM STG,
	DIM_HTTP_STATUS HST,
	DIM_USER_AGENT USG
WHERE STG.STATUS = HST.STATUS_NK
  AND STG.USER_AGENT_NK = USG.USER_AGENT_NK
  AND CAST(STG.EVENT_DT AS INTEGER) > $param_epoch_from /* load epoch date */
  AND CAST(STG.EVENT_DT AS INTEGER) < strftime('%s', date('now', 'start of day'))
GROUP BY STG.EVENT_DT, HST.DIM_HTTP_STATUS_ID, USG.DIM_USER_AGENT_ID

sqliteデータベースを使用すると、複雑なクエリを作成できます。WITHには、データとキーの準備が含まれています。メインクエリは、ディメンションへのすべての参照を収集します。

この条件では、履歴を再度読み込むことができません：CAST（STG.EVENT_DT AS INTEGER）> $ param_epoch_from、ここで、パラメーターはクエリ
'SELECT COALESCE（MAX（EVENT_DT）、\' 3600 \ '）AS LAST_EVENT_EPOCH FROM FCT_ACCESS_USER_AGENT_DD'の結果

です：CAST（STG.EVENT_DT AS INTEGER）<strftime（ '％s'、date（ 'now'、 'start of day'））

ページまたはファイルのカウントは、ポイントを検索することにより、基本的な方法で行われます。
