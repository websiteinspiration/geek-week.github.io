Abhängigkeitsinjektion - einfacher als es klingt?

Hallo Habr!



Wir bereiten die Veröffentlichung der zweiten Ausgabe des legendären Buches von Mark Siman „Dependency Injection in .NET“ vor.







Daher haben wir uns heute entschlossen, das Thema der Abhängigkeitsinjektion für .NET- und C # -Spezialisten kurz zu aktualisieren und eine Übersetzung des Artikels von Graham Downs anzubieten, in dem dieses Paradigma im Kontext der Steuerungsinversion betrachtet wird (IoC) und Containernutzung.



Die meisten Programmierer kennen wahrscheinlich das Phänomen der Abhängigkeitsinjektion, aber nicht jeder versteht die Bedeutung davon. Sie mussten sich wahrscheinlich mit Schnittstellen und Containern befassen, und manchmal führte die Arbeit mit ihnen zu einer Sackgasse. Andererseits haben Sie vielleicht gerade etwas über Abhängigkeitsinjektion gehört und diesen Artikel geöffnet, weil Sie deren Wesen besser verstehen möchten. In diesem Artikel werde ich zeigen, wie einfach das Konzept der Abhängigkeitsinjektion ist und wie leistungsfähig es tatsächlich sein kann.



Die Abhängigkeitsinjektion ist ein in sich geschlossener Ansatz, der für sich allein verwendet werden kann. Andererseits kann dieser Ansatz sowohl mit Schnittstellen als auch mit Containern für die Abhängigkeitsinjektion / Managementinversion (DI / IoC) angewendet werden. Wenn Sie in diesem Zusammenhang die Abhängigkeitsinjektion anwenden, kann es zu Verwirrung kommen, die ich anfangs erlebt habe.



Während meiner Karriere (ich bin auf die Entwicklung von Net / C # spezialisiert) habe ich die Abhängigkeitsinjektion in ihrer reinsten Form verwendet. Gleichzeitig habe ich DI implementiert, ohne auf Container oder Umkehrung der Kontrolle zurückzugreifen. Alles hat sich kürzlich geändert, als ich eine Aufgabe erhielt, bei der der Einsatz von Containern unverzichtbar war. Dann zweifelte ich stark an allem, was ich vorher wusste.



Nachdem ich mehrere Wochen in diesem Stil gearbeitet hatte, stellte ich fest, dass Container und Schnittstellen die Implementierung von Abhängigkeiten nicht erschweren, sondern im Gegenteil die Fähigkeiten dieses Paradigmas erweitern.


(Hierbei ist zu beachten: Schnittstellen und Container werden nur im Zusammenhang mit der Abhängigkeitsinjektion verwendet. Die Abhängigkeitsinjektion kann ohne Schnittstellen / Container implementiert werden. Im Wesentlichen besteht der einzige Zweck von Schnittstellen oder Containern darin, die Implementierung von Abhängigkeiten zu erleichtern.)



Dieser Artikel zeigt, wie die Abhängigkeitsinjektion sowohl mit als auch ohne Schnittstellen und Container durchgeführt wird. Ich hoffe, dass Sie nach dem Lesen das Prinzip der Abhängigkeitsinjektion klar verstehen und fundierte Entscheidungen darüber treffen können, wann und wo Sie bei der Implementierung von Abhängigkeiten auf Schnittstellen und Container zurückgreifen müssen.



Ausbildung



Um die Abhängigkeitsinjektion in ihrer reinsten Form besser zu verstehen, schauen wir uns eine in C # geschriebene Beispielanwendung an.



Beachten Sie zunächst, dass eine solche Anwendung ohne Abhängigkeitsinjektion geschrieben werden kann. Anschließend werden wir die Abhängigkeitsinjektion einführen und eine einfache Protokollierungsfunktion hinzufügen.



Im weiteren Verlauf werden Sie feststellen, wie die Anforderungen für die Protokollierung allmählich komplizierter werden, und wir erfüllen diese Anforderungen mithilfe der Abhängigkeitsinjektion. während der Verantwortungsbereich der Klasse Calculator



minimiert wird. Durch die Abhängigkeitsinjektion müssen Calculator



wir die Klasse nicht ändern, wenn wir das Protokollierungsgerät ändern möchten.



Anwendung



Betrachten Sie den folgenden Code. Es wurde für eine einfache Taschenrechneranwendung geschrieben, die zwei Zahlen, einen Operator, verwendet und das Ergebnis anzeigt. (Dies ist eine einfache Arbeitsanwendung für die Befehlszeile. Sie können sie daher problemlos als C # -Konsolenanwendung in Visual Studio reproduzieren und den Code dort einfügen, wenn Sie dem Beispiel folgen möchten. Alles sollte problemlos funktionieren.)



Wir haben eine Klasse Calculator



und eine Hauptklasse Program



es benutzen.



Program.cs: 



using System;
using System.Linq;

namespace OfferZenDiTutorial
{
    class Program
    {
        static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            var calc = new Calculator();
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }

        private static float GetNumber(string message)
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write(message);
                var input = Console.ReadLine();
                isValid = float.TryParse(input, out var number);
                if (isValid)
                    return number;

                Console.WriteLine("Please enter a valid number. Press ^C to quit.");
            }

            return -1;
        }

        private static char GetOperator()
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write("Please type the operator (/*+-) > ");
                var input = Console.ReadKey();
                Console.WriteLine();
                var operation = input.KeyChar;
                if ("/*+-".Contains(operation))
                {
                    isValid = true;
                    return operation;
                }

                Console.WriteLine("Please enter a valid operator (/, *, +, or -). " +
                                  "Press ^C to quit.");
            }

            return ' ';
        }

        private static float GetResult(Calculator calc, float number1, float number2, 
            char operation)
        {
            switch (operation)
            {
                case '/': return calc.Divide(number1, number2);
                case '*': return calc.Multiply(number1, number2);
                case '+': return calc.Add(number1, number2);
                case '-': return calc.Subtract(number1, number2);
                default:
                    //    ,        
                    throw new InvalidOperationException("Invalid operation passed: " + 
                                                        operation);
            }
        }
    }
}


Das Hauptprogramm startet, fordert den Benutzer nach zwei Zahlen und einem Operator auf und ruft dann die Klasse Calculator



auf, um eine einfache arithmetische Operation für diese Zahlen durchzuführen. Zeigt dann das Ergebnis der Operation an. Hier ist die Klasse Calculator



.

Protokollierung



Die Anwendung funktioniert einwandfrei, aber stellen Sie sich vor: Ihr Chef war der Meinung, dass jetzt alle Vorgänge in einer Datei auf der Festplatte protokolliert werden sollten, damit Sie sehen können, was die Leute tun.



Es scheint, dass das nicht so schwierig ist, oder? Nehmen Sie Anweisungen an und fügen Sie sie hinzu, nach denen alle ausgeführten Vorgänge Calculator



in eine Textdatei eingegeben werden sollen. So sieht deine jetzt aus Calculator



:



Calculator.cs: 



using System.IO;

namespace OfferZenDiTutorial
{
    public class Calculator
    {
        private const string FileName = "Calculator.log";

        public float Divide(float number1, float number2)
        {
            File.WriteAllText(FileName, $"Running {number1} / {number2}");
            return number1 / number2;
        }

        public float Multiply(float number1, float number2)
        {
            File.WriteAllText(FileName, $"Running {number1} * {number2}");
            return number1 * number2;
        }

        public float Add(float number1, float number2)
        {
            File.WriteAllText(FileName, $"Running {number1} + {number2}");
            return number1 + number2;
        }

        public float Subtract(float number1, float number2)
        {
            File.WriteAllText(FileName, $"Running {number1} - {number2}");
            return number1 - number2;
        }
    }
}


Es funktioniert großartig. Wenn Calculator



etwas passiert, schreibt es es in eine Datei, die Calculator.log



sich im selben Verzeichnis befindet, in dem es ausgeführt wird.



Die Frage ist jedoch möglich: Ist es wirklich angemessen, dass die Calculator-Klasse für das Schreiben in eine Textdatei verantwortlich ist?



Klasse Filelogger



Nein. Dies ist natürlich nicht sein Verantwortungsbereich. Damit das Prinzip der alleinigen Verantwortung nicht verletzt wird , muss daher alles, was mit der Informationsprotokollierung zusammenhängt, in der Protokolldatei enthalten sein. Dazu muss eine separate autarke Klasse geschrieben werden. Machen wir das.



Zunächst erstellen wir eine völlig neue Klasse, nennen wir es FileLogger



. So wird es aussehen.



FileLogger.csh: 



using System;
using System.IO;

namespace OfferZenDiTutorial
{
    public class FileLogger
    {
        private const string FileName = "Calculator.log";
        private readonly string _newLine = Environment.NewLine;

        public void WriteLine(string message)
        {
            File.AppendAllText(FileName, $"{message}{_newLine}");
        }
    }
}


In dieser Klasse wird nun alles verarbeitet, was mit dem Erstellen einer Protokolldatei und dem Schreiben von Informationen zusammenhängt. Darüber hinaus erhalten wir eine angenehme Kleinigkeit: Unabhängig davon, was diese Klasse verbraucht, ist es nicht erforderlich, Leerzeilen zwischen separate Datensätze zu setzen. Aufzeichnungen sollten nur unsere Methode aufrufen WriteLine



, und wir kümmern uns um den Rest. Ist es nicht cool?

Um eine Klasse zu verwenden, benötigen wir ein Objekt, das sie instanziiert. Lassen Sie uns dieses Problem innerhalb der Klasse lösen Calculator



. Ersetzen Sie den Inhalt der Klasse durch Calculator.cs



Folgendes:



Calculator.cs: 



namespace OfferZenDiTutorial
{
    public class Calculator
    {
        private readonly FileLogger _logger;

        public Calculator()
        {
            _logger = new FileLogger();
        }

        public float Divide(float number1, float number2)

            _logger.WriteLine($"Running {number1} / {number2}");
            return number1 / number2;
        }

        public float Multiply(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} * {number2}");
            return number1 * number2;
        }

        public float Add(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} + {number2}");
            return number1 + number2;
        }

        public float Subtract(float number1, float number2)
        {
            _logger.WriteLine($"Running {number1} - {number2}");
            return number1 - number2;
        }
    }
}


Daher ist es unserem Taschenrechner jetzt egal, wie genau der neue Logger die Informationen in die Datei schreibt oder wo sich diese Datei befindet und ob die Datei überhaupt geschrieben wird. Es besteht jedoch immer noch ein Problem: Können wir überhaupt erwarten, dass die Klasse Calculator



weiß, wie der Logger erstellt wird?



Abhängigkeitsspritze



Offensichtlich ist die Antwort auf die letzte Frage negativ!



Hier, lieber Leser, kommt die Sucht ins Spiel. Lassen Sie uns den Konstruktor unserer Klasse ändern Calculator



:



Calculator.cs: 



        public Calculator(FileLogger logger)
        {
            _logger = logger;
        }


Das ist alles. An der Klasse ändert sich nichts anderes.



Die Abhängigkeitsinjektion ist ein Element eines größeren Themas namens Management Inversion, eine ausführliche Erörterung dieses Themas würde jedoch den Rahmen dieses Artikels sprengen.



In diesem Fall müssen Sie nur wissen, dass wir die Steuerung der Logger-Klasse invertieren , oder wir übertragen metaphorisch das Problem des Erstellens einer Datei an jemanden, indem wir FileLogger



eine Instanz FileLogger



in unseren Taschenrechner einbetten , anstatt zu erwarten, dass die Klasse Calculator



weiß, wie sie erstellt wird.



Wessen Verantwortung ist das?



Nur derjenige, der die Klasse instanziiert Calculator



. In unserem Fall ist dies das Hauptprogramm.



Um dies zu demonstrieren, ändern Sie die Main-Methode in unserer Program.cs-Klasse wie folgt:



Program.cs 



  static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            //    
            var logger = new FileLogger();
            var calc = new Calculator(logger);
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }


Somit müssen nur zwei Zeilen geändert werden. Wir erwarten nicht, dass die Klasse Calculator



instanziiert FileLogger



, sie wird es dafür tun Main



und dann das Ergebnis an sie weitergeben.



Im Wesentlichen handelt es sich hierbei um eine Abhängigkeitsinjektion. Weder Schnittstellen noch Container zur Umkehrung der Kontrolle oder ähnliches werden benötigt. Wenn Sie so etwas tun mussten, handelte es sich im Grunde genommen um eine Abhängigkeitsinjektion. Cool, oder?



Erweiterung der Möglichkeiten: Wir werden einen weiteren Logger herstellen



Trotz alledem haben Schnittstellen ihren eigenen Platz, und tatsächlich werden sie genau in Verbindung mit Dependency Injection enthüllt.



Angenommen, Sie haben einen Client, aus dessen Sicht das Protokollieren jedes Anrufs Calculator



eine Verschwendung von Zeit und Speicherplatz darstellt, und es ist besser, überhaupt nichts zu protokollieren.



Denken Sie, dass diesbezüglich Änderungen im Inneren Calculator



erforderlich sein werden, die möglicherweise ein erneutes Kompilieren und Verteilen der Baugruppe erfordern, in der sie sich befindet?



Hier bieten sich die Schnittstellen an.



Schreiben wir eine Schnittstelle. Wir werden es nennen ILogger



, weil unsere Klasse an der Umsetzung beteiligt sein wird FileLogger



.



ILogger.cs 



namespace OfferZenDiTutorial
{
    public interface ILogger
    {
        void WriteLine(string message);
    }
}


Wie Sie sehen können, wird eine einzige Methode definiert: WriteLine



implementiert FileLogger



. Machen wir einen weiteren Schritt und formalisieren diese Beziehung, sodass diese Klasse unsere neue Schnittstelle offiziell implementiert:



FileLogger.cs 



public class FileLogger : ILogger


Dies ist die einzige Änderung, die wir an dieser Datei vornehmen werden. Alles andere wird wie zuvor sein.

Also haben wir die Einstellung festgelegt - was sollen wir jetzt damit machen?



Zunächst ändern wir die Calculator-Klasse so, dass sie eine Schnittstelle ILogger



und keine bestimmte Implementierung verwendet FileLogger



:



Calculator.cs 



private readonly ILogger _logger;

        public Calculator(ILogger logger)
        {
            _logger = logger;
        }


Zu diesem Zeitpunkt wird der Code noch problemlos kompiliert und ausgeführt. Wir gehen FileLogger



von der Hauptmethode des Programms aus, die implementiert wird ILogger



. Der einzige Unterschied besteht darin, dass Calculator



nicht nur nicht wissen muss, wie erstellt wird FileLogger



, sondern auch, welche Art von Logger für ihn erstellt wird.



Da alles, was Sie erhalten, eine Schnittstelle implementiert ILogger



(und daher eine Methode hat WriteLine



), gibt es keine Probleme mit der praktischen Verwendung.



Fügen wir nun eine weitere Implementierung der Schnittstelle hinzu ILogger



. Dies ist eine Klasse, die beim Aufruf der Methode nichts unternimmt WriteLine



. Wir werden ihn anrufen NullLogger



und so sieht er aus:



NullLogger.cs 



namespace OfferZenDiTutorial
{
    public class NullLogger : ILogger
    {
        public void WriteLine(string message)
        {
            //      
        }
    }
}


Dieses Mal müssen wir in der Klasse überhaupt nichts ändern, Calculator



wenn wir die neue verwenden möchten NullLogger



, da sie bereits alles akzeptiert, was die Schnittstelle implementiert ILogger



.



Wir müssen nur die Main-Methode in unserer Program.cs-Datei ändern, um eine andere Implementierung an sie zu übergeben. Lassen Sie uns dies tun und die Methode so Main

Kleiner Haftungsausschluss zu Schnittstellen



Wie Sie sehen können, sind Schnittstellen ein sehr leistungsfähiges Werkzeug, das der Anwendung jedoch eine zusätzliche Abstraktionsebene und damit zusätzliche Komplexität verleiht. Container können im Kampf gegen die Komplexität hilfreich sein. Wie im nächsten Abschnitt gezeigt wird, müssen Sie jedoch alle Schnittstellen registrieren. Wenn Sie mit bestimmten Typen arbeiten, ist dies nicht erforderlich.



Es gibt die sogenannte „Verschleierung mit Hilfe der Abstraktion“, die im Wesentlichen auf eine erneute Komplikation des Projekts hinausläuft, um all diese verschiedenen Ebenen zu realisieren. Wenn Sie möchten, können Sie auf alle diese Schnittstellen verzichten, es sei denn, es gibt bestimmte Gründe, warum Sie sie benötigen. Im Allgemeinen sollten Sie keine solche Schnittstelle erstellen, die nur eine Implementierung hat.



Abhängigkeitsinjektionsbehälter



Das in diesem Artikel erarbeitete Beispiel ist recht einfach: Wir haben es mit einer einzelnen Klasse zu tun, die nur eine Abhängigkeit hat. Nehmen wir nun an, wir haben viele Abhängigkeiten, und jede davon ist anderen Abhängigkeiten zugeordnet. Selbst wenn Sie an mäßig komplexen Projekten arbeiten, müssen Sie sich wahrscheinlich mit solchen Situationen auseinandersetzen, und Sie müssen nicht nur alle diese Klassen erstellen, sondern sich auch daran erinnern, welche von welcher abhängt - insbesondere, wenn Sie sich entscheiden Verwenden Sie Schnittstellen.



Machen Sie sich mit dem Abhängigkeitsinjektionscontainer vertraut. Es vereinfacht Ihr Leben, aber das Funktionsprinzip eines solchen Containers kann sehr verwirrend erscheinen, insbesondere wenn Sie gerade erst anfangen, es zu beherrschen. Auf den ersten Blick kann diese Gelegenheit etwas Magie geben.

In diesem Beispiel verwenden wir den Container von Unity, aber es stehen viele andere zur Auswahl. Ich werde nur die beliebtesten nennen: Castle Windsor , Ninject . Aus funktionaler Sicht unterscheiden sich diese Behälter praktisch nicht. Der Unterschied mag auf der Ebene der Syntax und des Stils spürbar sein, aber letztendlich hängt alles von Ihren persönlichen Vorlieben und Ihrer Entwicklungserfahrung ab (sowie von den Vorgaben in Ihrem Unternehmen!).



Schauen wir uns ein Beispiel mit Unity genauer an: Ich werde versuchen zu erklären, was hier passiert.



Als erstes müssen Sie einen Link zu Unity hinzufügen. Glücklicherweise gibt es dafür ein Nuget-Paket. Klicken Sie also mit der rechten Maustaste auf Ihr Projekt in Visual Studio und wählen Sie Nuget-Pakete verwalten:







Suchen und installieren Sie das Unity-Paket und konzentrieren Sie sich auf das Unity Container-Projekt:







Wir sind also bereit. Ändern Sie die Main

Program.cs 



 static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            //     
            var container = new UnityContainer();
            container.RegisterType<ILogger, NullLogger>();

Bitte beachten Sie: Wenn es NullLogger



einen Konstruktor gibt, der zusätzliche Typen anfordert, wiederholt der Konstruktor einfach alle Schritte für sie, beginnend mit 3. Im Prinzip durchsucht er alle Typen und versucht, alle gefundenen Typen automatisch in bestimmte Instanzen aufzulösen.



Beachten Sie auch, dass Entwickler in Absatz 2 bestimmte Typen nicht explizit registrieren müssen. Falls gewünscht, kann dies beispielsweise durchgeführt werden, um den Lebenszyklus zu ändern oder abgeleitete Klassen usw. zu bestehen. Dies ist jedoch ein weiterer Grund, die Erstellung von Schnittstellen zu vermeiden, bis ein offensichtlicher Bedarf besteht.



Das ist alles. Nichts Geheimnisvolles und besonders Mystisches.



Andere Eigenschaften



Es ist erwähnenswert, dass Container es Ihnen ermöglichen, einige weitere Dinge zu tun, die sich nur schwer (aber nicht unmöglich) selbst implementieren lassen. Dazu gehören das Lebenszyklusmanagement und die Implementierung von Methoden und Eigenschaften. Eine Diskussion dieser Themen würde den Rahmen dieses Artikels sprengen, da es unwahrscheinlich ist, dass Anfänger solche Techniken benötigen. Ich empfehle Ihnen jedoch, die Dokumentation sorgfältig zu lesen, nachdem Sie das Thema beherrschen, um zu verstehen, welche anderen Möglichkeiten bestehen.
