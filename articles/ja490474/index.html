STM32パート1：基本

自分で完全に記述していないコードを信頼することはできません。-ケン・トンプソン
おそらく私のお気に入りの引用です。私がうさぎの穴の奥深くに飛び込むことにしたのは彼女だった。プログラミングの世界への旅を始めたのはごく最近で、たった1か月しか経っていないので、資料をまとめるための記事を書くことにしました。すべては、Arduinaを使用してフォトスタジオのランプを同期させるという簡単なタスクから始まりました。問題は解決しましたが、もう写真スタジオには行かず、時間がありません。その時から、マイコンのプログラミングに徹底的に取り組むことにしました。 Arduinは、その単純さには魅力がありますが、プラットフォームは好きではありませんでした。その選択はST社とその人気製品にかかった。その瞬間、まだ何が違うのかわかりませんでしたが、一般的な消費者として、「プロセッサ」の速度とメモリの量を比較して、STM32F746NGディスプレイを備えた印象的なボード-Discoveryを購入しました。私は絶望の瞬間を逃し、要点を正します。

プログラマーのイメージに没頭して、たくさん読んだり、勉強したり、実験したりしました。そして、すでに上で述べたように、私はよく勉強したかった、それだけです。そして、このために私は目標を設定しました。既製のソリューションではなく、私のものだけです。そして、すべてが私のためにうまくいったなら、あなたは成功するでしょう。

必要なものすべてのリスト：

  1. Ubuntu 16+ 仮想マシンなど
  2. アームコンパイラ-でダウンロードdeveloper.arm.com/tools-and-software/open-source-software/developer-tools/gnu-toolchain/gnu-rm/downloads
  3. openocdデバッガーおよびプログラマー-リンクからダウンロードすることはできません。ソースから収集され、説明が添付されています

    git clone https://git.code.sf.net/p/openocd/code openocd
  4. お好みに合わせてテキストエディター

すべてをインストールして組み立てたら、最初のプロジェクトに進むことができます！いいえ、点滅している電球でもありません。まず、マイクロウェル自体の初期化プロセスについて詳しく説明する必要があります。

私たちの必要なもの：

  1. Makefile
  2. Linker.ld
  3. Init.c

最後の段落Init.cから始めましょう。まず、mkはアドレスをロードする必要があります。「スタックポインタ」は、PUSH命令とPOP命令に使用されるメモリアドレスへのポインタです。すべての手順を詳しく説明するわけではないので、これら2つの手順を十分に検討することを強くお勧めします。これを実装する方法については、以下を参照してください。

extern void *_estack;
void *vectors[] __attribute__((section(".isr_vector"), used)) = {
    &_estack
}

次に、この例を見てみましょう。Externは、シンボルが外部であることを意味し、このシンボルをLinker.ldファイルで宣言したので、少し後で戻ってきます。

 __attribute__((section(".isr_vector"), used))

ここでは、isr_vectorセクションに配列を配置するようコンパイラーに指示する属性を使用し、コードでそれを使用しない場合でも、プログラムに含める必要があります。そして、その最初の要素はその同じポインターになります。

さて、この配列がなぜであるか、そしてそれが何と一緒に食べられるかを理解しましょう。従来のプロセッサーとは異なり、armアーキテクチャーのmicronは、ゼロアドレスからではなく、この配列のポインターが示すアドレスから実行を開始します。すべてが複雑です。また、この配列の最初のポインターは常にスタックの先頭を指していますが、2番目のポインターは既にコードの先頭を指しています。

例を挙げましょう。スタックが0x20010000で始まり、プログラムコードが0x0800008であると仮定しています。配列は次のように書くことができます

void *vectors[] __attribute__((section(".isr_vector"), used)) = {
    0x20010000,
    0x08000008
}

つまり、コントローラは最初にスタックを初期化し、次に最初の命令のアドレスを考慮してそれをプログラムカウンタレジスタにロードします。現在、最も重要なことは、モデルに応じて、これらの数値は異なる可能性がありますが、stm32f7の例では、この配列はアドレス0x08000000のメモリにあると自信を持って言えます。 mkがオンまたはリセット後に作業を開始するのは、このアドレスからです。

ここで、停止して、この配列を必要なセクションに配置する方法に注目します。これは「ld」またはリンカーによって行われます。このプログラムはプログラム全体を集め、Linker.ldスクリプトがこれに使用されます。例を挙げ、さらに分析します。

MEMORY{
	ROM_AXIM (rx) : ORIGIN = 0x08000000, LENGTH = 1M
	RAM_DTCM (rwx): ORIGIN = 0x20000000, LENGTH = 64K
}

_estack = LENGTH(RAM_DTCM) + ORIGIN(RAM_DTCM);

SECTIONS{
	.isr_vector : {
	KEEP(*(.isr_vector))
	} >ROM_AXIM</code>
}

ここで何が起こるか見てみましょう。 MEMORYはメモリのセクションを定義し、SECTIONSはセクションを定義します。ここでは、_estackと、メモリの最初とその長さの合計、つまりメモリの最後に等しいという事実がわかります。配列を配置する.isr_vectorセクションも定義されています。>ROM_AXIMセクションの終わりにあるということは、このセクションは、ミクロンで必要な0x08000000で始まるメモリセクションに配置する必要があることを意味します。

必要な場所に配列を配置し、ミクロンを機能させるための何らかの命令が必要になりました。拡張されたinit.cは次のとおりです。

extern void *_estack;

void Reset_Handler();

void *vectors[] __attribute__((section(".isr_vector"), used)) = {
    &_estack,
    &Reset_Handler
};

void __attribute__((naked, noreturn)) Reset_Handler()
{
    while(1);
}

前に述べたように、2番目のアドレスは最初の関数または命令へのポインターでなければなりません。繰り返しになりますが、属性はすべて単純ですが、関数は値を返さず、入り口でABIに従います。つまり、裸の「裸」です。そのような関数では、通常のアセンブラが使われます。

今こそコードをコンパイルし、内部の内容を確認する時です。ここでは、Makefileには触れません。

arm-none-eabi-gcc -c init.c -o init.o -mthumb

arm-none-eabi-gcc -TLinker.ld -o prog.elf init.o -Wl,--gc-sections -nostartfiles -nodefaultlibs -nostdlib

そして、ここには理解できないことがたくさんあります。順番に：

  1. -mthumbは、thumb命令のみを使用するarmv7用にコンパイルします。
  2. -TLinker.ldはリンカースクリプトを指定します。デフォルトでは、Linux環境で実行するためにコンパイルされます
  3. -Wl、-gc-sections -nostartfiles -nodefaultlibs -nostdlibは、すべての標準ライブラリ、環境C初期化ファイル、および数学などの他のすべての補助ライブラリを削除します。

もちろん、これをミクロン単位でロードしても意味がありません。しかし、バイナリを表示して研究することに意味があります。

objcopy -O ihex prog.elf prog.bin

hexdump prog.bin

そして、結論を見ていきます。「00000000：00 01 00 02 09 00 00 08」は私たちの成功を象徴しています。これは私の最初の記事であり、すべての資料と本質を完全に明らかにすることはできなかったので、次はメカニズムをより詳細に説明し、私たちの友人はライトを点滅させないプログラムに行くことができますが、プロセッサとバスのクロックを構成できます。
