JSON-RPC APIテストライブラリ

JSON-RPCプロトコルに従ってAPIが実装されたマイクロサービスのテストを最初に書いたときに、json要素の品質チェックを構築することは、以前考えていたよりもはるかに厳しいことに気付きました。

以下は簡単な例です。

サービスから応答を受け取りました：

{
  "result": 12,
  "id": 46929734,
  "jsonrpc": "2.0"
}

確認する必要があるもの：

  1. 応答にはjsonrpc要素が含まれ、その値は2.0です
  2. 応答には要素「id」が含まれており、その値はリクエストで渡された同様の要素と同じです。現時点では、これは46929734だとします。

回答には要素「result」が含まれており、その値は12です。


Java + TestNG + Gsonを使用してこれを確認する方法：

Assert.assertTrue(response.has("result"));
Assert.assertTrue(response.has("jsonrpc"));
Assert.assertTrue(response.has("id"));

Assert.assertTrue(response.get("result").isJsonPrimitive());
Assert.assertTrue(response.get("jsonrpc").isJsonPrimitive());
Assert.assertTrue(response.get("id").isJsonPrimitive());

Assert.assertEquals(response.get("result").getAsInt(), 12);
Assert.assertEquals(response.get("jsonrpc").getAsString(), "2.0");
Assert.assertEquals(response.get("id").getAsInt(), 46929734);

現時点で、「なぜこれほど多くの主張があるのですか？」という質問が頭に浮かぶかもしれません。はい、実際、テストする場合、最後の3つのアサーションのグループにのみ関心があり、それらは応答内の要素のjson値をチェックし、それらが合格した場合、回答は期待に応えます。しかし、テストが失敗した場合はどうなりますか？そのようなテストが失敗するいくつかの考えられる理由を検討してください。

  1. 回答-> 
    {
  "result": 12,
  "id": 46929734,
  "jsonrpc": "1.0"
}

    TestNG -> java.lang.AssertionError: expected [2.0] but found [1.0]
    «2.0», «1.0» — .
  2. -> 
    {
  "result": 12,
  "id": 46929734,
  "jsonrpc": {}
}

    TestNG -> java.lang.UnsupportedOperationException: JsonObject

    , .
  3. -> 

    {
  "result": 12,
  "id": 46929734
}

    TestNG -> java.lang.NullPointerException
    , .

最初の例では、すべてが正常です。エラーは理解でき、実際の値と期待される値はすぐに表示されます。しかし、2番目と3番目の例はこれを自慢することはできません。このような低下は、テスターが原因を分析するのに時間がかかります。このような表面的なエラーを引き落とすことは困難です。テストがどの要素を解析しようとしたのか、どのメソッドが解析しようとしたのか、そしてこの要素が実際にどのように答えを見つけたのかを理解するには時間がかかります。テストアセンブリのログから必要なすべての情報を収集するよりも速いため、ローカルでテストを再起動する必要がある場合もあります。

そのような数のチェック、それらを記述して最新の状態でサポートする必要性は私を非常に動揺させたので、私の意見では、json要素のチェックに必要なすべてを収集する別のライブラリーを作成しました。ライブラリを使用して上記のjsonを必要としたのと同じレベルのカバレッジと詳細は、9行ではなく3行で達成されます。

Gassert.verifyInteger(response, "result", 12);
Gassert.verifyString(response, "jsonrpc", "2.0");
Gassert.verifyInteger(response, "id", 46929734);

ライブラリはチェックするメソッドを実装します：

  • すべてのプリミティブ型の要素（ネストされているかどうかにかかわらず、値のチェックありとなし）
  • JsonObject、ネストされているかどうか、値のチェックありとなし
  • JsonArray、ネストされているかどうか、値のチェックありとなし
  • ネストされたJsonNull要素
  • 配列内の要素タイプ
  • 予期される要素配列の内容
  • 配列の次元
  • オブジェクトの寸法

検証のすべてのレベルで、TestNGがデフォルトで提供するエラーに比べて、より詳細なエラーが実装されます。

テストが落ちる理由として考えられるものをもう一度考えてみましょう。

  1. 回答-> 
    
{
  "result": 12,
  "id": 46929734,
  "jsonrpc": "1.0"
}

    Gassertからのエラー-> java.lang.AssertionError：要素[jsonrpc]の検証に失敗しました。[2.0]が必要ですが、[1.0]が見つかりました
  2. 回答-> 

    
{
  "result": 12,
  "id": 46929734,
  "jsonrpc": {}
}

    Gassertからのエラー-> java.lang.AssertionError：Element [jsonrpc] is a JsonPrimitive。[true]を見つけることを期待していなかったが、[false]を見つけた
  3. 回答-> 

    
{
  "result": 12,
  "id": 46929734
}

    Gassertからのエラー-> java.lang.AssertionError：Jsonに要素が含まれていません：[jsonrpc]。[true]を見つけることを期待していなかったが、[false]を見つけた

また、チェックのほとんどのメソッドは、追加の文字列パラメーターでオーバーロードされます。アサーションが低下した場合、表示されるエラーのテキストに追加されます。この機能を使用して、たとえば、応答の完全な本文または要求内のエラー+応答を表示できます。
