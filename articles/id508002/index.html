Perubahan yang lebih ambisius juga muncul terkait dengan kinerja, pendekatan arsitektur untuk membuat aplikasi dan konstruktor deklaratif:



1. Pekerjaan yang dioptimalkan dengan memori. Kinerja memori.



Ini adalah perubahan global untuk Swift 5.3 secara keseluruhan. Dengan masuk ke dalam struktur, Anda dapat menggunakan passing dengan nilai alih-alih tautan, sehingga mengurangi ukuran tumpukan (tumpukan), ukuran biner, dan waktu kompilasi.







Ngomong-ngomong, beberapa kontrol di SwiftUI sendiri mulai menggunakan pendekatan Malas. Misalnya, daftar yang sama dan LazyVGrid / LazyHGrid (analog dari UICollectionView). Secara teori, ini harus menghapus masalah inisialisasi tampilan segera. Namun, belum ada yang dikatakan tentang ... NavigationLink .



2. Menggunakan DSL di dalam blok-blok ViewBuilder.



Hore, sekarang kita bisa menambahkan if / else atau mengganti case dalam blok deklaratif untuk keperluan kita. Misalnya, buat pabrik Tampilan Anak di dalam Tampilan induk. Atau di dalam NavigationView.







Itu sangat keren.



Sekarang Anda juga dapat memeriksa kondisinya dengan @PropertyWrapper di dalam blok: 



@State private var isVisible = true
if isVisible == true {
    Text("Hello") // Only rendered when isVisible is true.
}


3. Sekarang Anda dapat membuat aplikasi 100% pada komponen SwiftUI.



Ya ya. Untuk melakukan ini, Apple membuat cara melakukannya tanpa AppDelegate, SceneDelegate, dan UIHostingViewController.



Menggunakan anotasiutama dan protokol Aplikasi, Adegan Anda dapat mencapai ini: 



@main
struct testmultiplatformApp: App {
  @SceneBuilder var body: some Scene {
        WindowGroup {
            MailViewer()
        }
        Settings {
            SettingsView()
        }
    }
}


utamamemberi sinyal bahwa ini adalah titik masuk baru aplikasi Anda. Diperlukan penerapan protokol Aplikasi. Perhatikan jenis properti tubuh dari struktur Aplikasi. Aplikasi dapat memiliki satu atau lebih yang disebut adegan, yang masing-masing mengimplementasikan protokol adegan. Setiap adegan memiliki View root dan siklus hidupnya sendiri. Jika Anda ingin menggunakan beberapa adegan (seperti, misalnya, dalam aplikasi multi-jendela), maka Anda harus mengatur atribut body SceneBuilder. Ini pada dasarnya adalah mekanisme enkapsulasi dari UISceneDelegate dan logikanya.



WindowGroup digunakan untuk membuat layar ukuran penuh tunggal, seperti UIWindow / UIWindowScene.



Namun, kami ingat bahwa SwiftUI adalah tambahan untuk UIKit. Dan UIKit tetap di dalam. Jika kita bahkan menjalankan aplikasi templat seperti itu, maka dalam hierarki Tampilan kita akan melihat:







Baik UIHostingViewController di tempat dan UIWindowScene. Tapi di dalam. Ya, untuk inisialisasi, ini sangat menyederhanakan pekerjaan. Tetapi apakah mereka memikirkan solusi sehingga mereka tidak perlu mengembalikan UISceneDelegate ke aplikasi dengan pekerjaan eksplisit dari seluruh struktur navigasi.



Melacak adegan siklus hidup dikurangi menjadi metode: 



public func onChange<V>(of value: V, perform action: @escaping (V) -> Void) -> some Scene where V : Equatable


Ini adalah cara mereka mengusulkan untuk menggunakan metode ini untuk melacak transisi aplikasi ke status latar belakang: 



  ///
    /// Use this modifier to trigger a side effect when a value changes, like
    /// the value associated with an ``SwiftUI/Environment`` key or a
    /// ``SwiftUI/Binding``. For example, you can clear a cache when you notice

    ///
    ///     struct MyScene: Scene {
