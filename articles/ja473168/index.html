ASP.NETおよびASP.NET Coreでの今後のSameSite Cookieの変更

SameSiteは、クロスサイトリクエストフォージェリ（CSRF）を防ぐために設計された  2016 HTTP Cookie拡張です。当初、その設計は、新しいSameSiteプロパティをCookieに追加することで使用できる追加機能でした。 LaxとStrictという2つの意味がありました。

Lax値を設定する場合、あるサイトをサーフィンするとき、またはGETサーフィンを介して他のサイトからあなたのサイトにCookieを送信する必要があると理解されています。 Strict値は、Cookieを1つのサイトからのみのリクエストに制限しました。プロパティを設定しなくても、リクエストでCookieが機能する方法に制限はありません。 OpenIdConnect認証操作（ログイン、ログアウトなど）および操作を要求するサイトに外部サイトからPOST要求を送信する他の機能は、Cookieを使用して、CSRFを関連付けたり保護したりできます。これらの操作は、プロパティをまったく設定せずにSameSiteを拒否して、特殊な要求フロー中にこれらのCookieが送信されるようにする必要があります。

今グーグル 標準  を更新し、 Chromeの将来のバージョンで提案された変更を導入します。この変更により、同じSameSiteの「なし」の値が追加され、デフォルトの動作が「緩い」に変更されます。これにより、OpenIdConnectログインが破壊され、ウェブサイトで構築できるその他の機能が破壊される可能性があります。これらの機能では、SameSiteプロパティがNoneに設定されているCookieを使用する必要があります。

ただし、元の標準に準拠し、新しい値を認識していないブラウザーは、新しい標準を使用するブラウザーとは異なる動作をします。つまり、.NET Webサイトでは、ユーザーエージェントスニッフィングを追加して、新しいNone値を送信するか、属性を送信しないかを決定する必要があります。



.NETは、.NET 4.7.2および.NET Core 2.1以降でSameSite属性の動作を変更するための更新を受け取り、Googleの新しい値の導入に適応します。 .NET Frameworkの更新は、更新の確認機能を使用する場合、11月19日にMicrosoft UpdateおよびWSUSを介した追加の更新として利用可能になります。 12月10日に更新プログラムが一般公開され、Microsoft Update Centerに表示されます。 .NET Coreの更新は、11月にプレビュー1以降の.NET Core 3.1のリリースで利用できるようになります。.NETCore

3.1には、SameSiteプロパティを設定しない、更新された列挙定義 SameSite.Unspecified が含まれ  ます。

Microsoft.Owin v4.1および.NET Core用のOpenIdConnectミドルウェアは、.NET Frameworkおよび.NETの更新と同時に更新されますが、ユーザーエージェントスニッフィングコードをフレームワークに追加することはできません。これは、サイトのコードに実装する必要があります。エージェントスニッフィングの実装は、使用しているASP.NETまたはASP.NET Coreのバージョンと、サポートするブラウザによって異なります。

ASP.NET 4.7.2とMicrosoft.Owin 4.1.0の場合、エージェントスニッフィングはICookieManagerを使用して実装できます  。

public class SameSiteCookieManager : ICookieManager
{
  private readonly ICookieManager _innerManager;

  public SameSiteCookieManager() : this(new CookieManager())
  {
  }

  public SameSiteCookieManager(ICookieManager innerManager)
  {
    _innerManager = innerManager;
  }

  public void AppendResponseCookie(IOwinContext context, string key, string value,
                                   CookieOptions options)
  {
    CheckSameSite(context, options);
    _innerManager.AppendResponseCookie(context, key, value, options);
  }

  public void DeleteCookie(IOwinContext context, string key, CookieOptions options)
  {
    CheckSameSite(context, options);
    _innerManager.DeleteCookie(context, key, options);
  }

  public string GetRequestCookie(IOwinContext context, string key)
  {
    return _innerManager.GetRequestCookie(context, key);
  }

  private void CheckSameSite(IOwinContext context, CookieOptions options)
  {
    if (DisallowsSameSiteNone(context) && options.SameSite == SameSiteMode.None)
    {
        options.SameSite = null;
    }
  }

  public static bool DisallowsSameSiteNone(IOwinContext context)
  {
    // TODO:    User Agent .
    var userAgent = context.Request.Headers["User-Agent"];
    return userAgent.Contains("BrokenUserAgent") ||
           userAgent.Contains("BrokenUserAgent2")
  }
}

そして、新しいCookieManagerを使用するようにOpenIdConnect設定を構成します。

app.UseOpenIdConnectAuthentication(
    new OpenIdConnectAuthenticationOptions
    {
    // …    … 
    CookieManager = new SameSiteCookieManager(new SystemWebCookieManager())
});

SystemWebCookieManagerが正しく機能するには、修正されたSameSite .NET 4.7.2以降をインストールする必要があります。

ASP.NET Coreの場合、パッチをインストールしてから、エージェントのスニッフィングコードをCookieポリシーに挿入する必要があります  。3.1より前のバージョンでは、SameSiteMode.Unspecifiedを（SameSiteMode）（-1）に置き換えます。

private void CheckSameSite(HttpContext httpContext, CookieOptions options)
{
    if (options.SameSite > SameSiteMode.Unspecified)
    {
        var userAgent = httpContext.Request.Headers["User-Agent"].ToString();
        // TODO:    User Agent .
        if (/* UserAgent doesn’t support new behavior */)
        {
               //  .NET Core < 3.1  SameSite = -1
               options.SameSite = SameSiteMode.Unspecified;
         }
    }
}

public void ConfigureServices(IServiceCollection services)
{
    services.Configure<CookiePolicyOptions>(options =>
    {
        options.MinimumSameSitePolicy = SameSiteMode.Unspecified;
        options.OnAppendCookie = cookieContext => 
            CheckSameSite(cookieContext.Context, cookieContext.CookieOptions);
        options.OnDeleteCookie = cookieContext => 
            CheckSameSite(cookieContext.Context, cookieContext.CookieOptions);
    });
}

public void Configure(IApplicationBuilder app)
{
    app.UseCookiePolicy(); // Before UseAuthentication or anything else that writes cookies.
    app.UseAuthentication();
    // …
}

Azure Active Directoryチームでテストしたところ、Azure Active Directoryによると、新しい値を理解していないすべての一般的なユーザーエージェントに対して次のチェックが機能することがわかりました。

public static bool DisallowsSameSiteNone(string userAgent)
{
    //      iOS.   :
    // - Safari  iOS 12  iPhone, iPod Touch, iPad
    // - WkWebview  iOS 12  iPhone, iPod Touch, iPad
    // - Chrome  iOS 12  iPhone, iPod Touch, iPad
    if (userAgent.Contains("CPU iPhone OS 12") || userAgent.Contains("iPad; CPU OS 12"))
    {
        return true;
    }

    //     Mac OS X.   :
    // - Safari  Mac OS X.
    //  :
    // - Chrome  Mac OS X
    //        Mac OS.
    if (userAgent.Contains("Macintosh; Intel Mac OS X 10_14") && 
        userAgent.Contains("Version/") && userAgent.Contains("Safari"))
    {
        return true;
    }

    //    Chrome 50-69,      SameSite=None.
    if (userAgent.Contains("Chrome/5") || userAgent.Contains("Chrome/6"))
    {
        return true;
    }

    return false;
}

このブラウザーのリストは、完全に信頼できるものではありません。システムがサポートする通常のブラウザーやその他のユーザーエージェントが、更新プログラムのインストール後に正しく動作することを確認する必要があります。

Chrome 80は   、2020年2月または3月に新しい動作に移行する予定です。新しい動作を確認したい場合は、古いバージョンのChromium  をダウンロードできます。
