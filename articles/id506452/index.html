Cara melindungi situs publik Anda dengan ESNI

Hai Habr, nama saya Ilya, saya bekerja di tim platform Exness. Kami mengembangkan dan mengimplementasikan komponen infrastruktur dasar yang menggunakan tim pengembangan produk kami.

Dalam artikel ini, saya ingin berbagi pengalaman menerapkan teknologi terenkripsi SNI (ESNI) dalam infrastruktur situs web publik.



Menggunakan teknologi ini akan meningkatkan tingkat keamanan saat bekerja dengan situs web publik dan mematuhi standar keamanan internal yang diadopsi oleh Perusahaan.

Pertama-tama, saya ingin memperhatikan bahwa teknologi ini tidak terstandarisasi dan masih dalam konsep, namun CloudFlare dan Mozilla sudah mendukungnya (dalam konsep01 ). Ini memotivasi kami untuk melakukan percobaan seperti itu.

Sedikit teori


ESNI adalah ekstensi protokol TLS 1.3 yang memungkinkan Anda untuk mengenkripsi SNI dalam pesan handshake TLS "Halo Klien". Inilah yang terlihat seperti Klien Hello dengan dukungan ESNI (alih-alih SNI yang biasa, kami melihat ESNI):



 Untuk menggunakan ESNI, Anda memerlukan tiga hal:

  • DNS 
  • Dukungan pelanggan
  • Dukungan sisi server.

DNS


Diperlukan untuk menambahkan dua catatan DNS - A , dan TXT (Catatan TXT berisi kunci publik yang dengannya klien dapat mengenkripsi SNI) - lihat di bawah ini. Selain itu, harus ada dukungan DoH (DNS over HTTPS), karena klien yang tersedia (lihat di bawah) tidak mengaktifkan dukungan ESNI tanpa DoH. Ini logis, karena ESNI menyiratkan enkripsi nama sumber daya yang kami akses, artinya, tidak masuk akal untuk mengakses DNS melalui UDP. Selain itu, penggunaan DNSSEC melindungi terhadap serangan keracunan cache dalam skenario ini.

Saat ini, beberapa penyedia DoH tersedia , di antaranya:

  • Cloudflare
  • Opendns
  • Google

CloudFlare menyatakan (Periksa Peramban Saya → Kode Terenkripsi → Pelajari Lebih Lanjut) bahwa server mereka sekarang mendukung ESNI, yaitu, untuk server CloudFlare di DNS, kami memiliki setidaknya dua catatan - A dan TXT. Pada contoh di bawah, kami meminta Google DNS (lebih HTTPS):

Dukungan pelanggan


Jika kita berbicara tentang browser, maka saat ini dukungan hanya diterapkan di FireFox . Inilah cara mengaktifkan dukungan ESNI dan DoH di FireFox. Setelah browser dikonfigurasi, kita akan melihat sesuatu seperti ini:



Tautan untuk memeriksa browser.

Tentu saja, TLS 1.3 harus digunakan untuk mendukung ESNI, karena ESNI adalah perpanjangan dari TLS 1.3.

Untuk menguji backend dengan dukungan ESNI, kami menerapkan klien saat bepergian , tetapi lebih lanjut tentang itu nanti.

Dukungan sisi server


ESNI saat ini tidak didukung oleh server web seperti nginx / apache, dll., Karena mereka bekerja dengan TLS melalui OpenSSL / BoringSSL, di mana ESNI tidak didukung secara resmi.

Oleh karena itu, kami memutuskan untuk membuat komponen front-end kami sendiri (ESNI reverse proxy), yang akan mendukung penghentian TLS 1.3 dengan ESNI dan proxy lalu lintas HTTP (S) ke hulu yang tidak mendukung ESNI. Ini memungkinkan Anda untuk menerapkan teknologi dalam infrastruktur yang ada, tanpa mengubah komponen utama - yaitu, menggunakan server web saat ini yang tidak mendukung ESNI. 

Untuk kejelasan, kami memberikan diagram:



Saya perhatikan bahwa proksi dikandung dengan kemampuan untuk mengakhiri koneksi TLS tanpa ESNI, untuk mendukung klien tanpa ESNI. Juga, protokol untuk berkomunikasi dengan upstream dapat berupa HTTP atau HTTPS dengan versi TLS lebih rendah dari 1,3 (jika upstream tidak mendukung 1,3). Desain ini memberikan fleksibilitas maksimum. Kami meminjam

implementasi dukungan ESNI saat bepergian dari CloudFlare . Segera, saya perhatikan bahwa implementasi itu sendiri cukup non-sepele, karena ini menyiratkan perubahan dalam pustaka crypto / tls standar dan karena itu memerlukan "patching" GOROOT sebelum perakitan.

Untuk menghasilkan kunci ESNI, kami menggunakan esnitool (juga gagasan CloudFlare). Kunci-kunci ini digunakan untuk mengenkripsi / mendekripsi SNI.
Kami menguji build menggunakan go 1.13 di Linux (Debian, Alpine) dan MacOS. 

Beberapa kata tentang fitur operasional


Proxy reverse ESNI memberikan metrik dalam format Prometheus, seperti rps, kode latensi & respons hulu, jabat tangan TLS yang gagal & durasi jabat tangan TLS. Pada pandangan pertama, ini tampaknya cukup untuk mengevaluasi bagaimana proxy menangani traffic.

Sebagai kesimpulan, saya perhatikan bahwa teknologi ESNI terlihat cukup menjanjikan. Ada banyak lagi pertanyaan terbuka, misalnya, pertanyaan tentang menyimpan kunci ESNI publik dalam DNS dan memutar kunci ESNI - masalah ini dibahas secara aktif, dan versi terbaru dari draft (pada saat penulisan) ESNI sudah 7 .
