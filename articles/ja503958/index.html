オープンAPIゲーム：Swagger Play


この記事では、実際の例とともに、SwaggerモジュールをPlay Frameworkに使用する方法を説明します。私が言います：

  1. 最新バージョンのSwagger-Play（swagger-apiアノテーションを使用してOpenAPI仕様に基づいてドキュメントを生成できるPlayモジュール）を固定する方法とswagger-ui（生成されたドキュメントを視覚化するために使用されるJavaScriptライブラリ）を構成する方法
  2. Swagger-Coreの主なアノテーションについて説明し、Scalaでのそれらの使用の機能について説明します
  3. データモデルクラスを正しく操作する方法を説明します
  4. Swaggerでジェネリック型の問題を回避する方法
  5. SwaggerにADT（代数データ型）を理解させる方法
  6. コレクションを説明する方法

この記事は、ScalaでPlay Frameworkを使用し、APIのドキュメントを自動化するすべての人にとって興味深いものになります。

依存関係を追加


インターネットで多くの情報源を調べてきたので、SwaggerとPlay Frameworkを友だちにするためには、Swagger Play2モジュールをインストールする必要があると思います。

github上のライブラリアドレス：

https : //github.com/swagger-api/swagger-play

依存関係を追加：

libraryDependencies ++= Seq(
  "io.swagger" %% "swagger-play2" % "2.0.1-SNAPSHOT"
)

そして、ここで問題が発生します

。この記事の執筆時点では、依存関係はMaven-centralまたはSonatypeリポジトリーから引き出されていませんでした。

Maven-centralで見つかったビルドはすべてScala 2.12で終了しました。一般に、Scala 2.13の単一のアセンブルバージョンはありませんでした。

今後登場することを期待しています。

Sonatype-releasesリポジトリを登っていくと、このライブラリの現在の分岐点を見つけました。githubのアドレス：

https : //github.com/iterable/swagger-play

したがって、依存関係を挿入します。

libraryDependencies ++= Seq(
  "com.iterable" %% "swagger-play" % "2.0.1"
)

Sonatypeリポジトリを追加します。

resolvers += Resolver.sonatypeRepo("releases")

（このアセンブリはMaven-centralであるため、必要ありません）

これで、application.conf構成ファイルでモジュールをアクティブ化できます。

play.modules.enabled += "play.modules.swagger.SwaggerModule"

また、ルートをルートに追加します。

GET     /swagger.json           controllers.ApiHelpController.getResources

そして、モジュールは準備が整いました。

Swagger Playモジュールは、ブラウザーで表示できるjsonファイルを生成します。

Swaggerの機能を完全に楽しむには、視覚化ライブラリーswagger-uiもダウンロードする必要があります。swagger.jsonファイルを読み取るための便利なグラフィカルインターフェイスと、残りのリクエストをサーバーに送信する機能を提供し、Postman、Rest-client、およびその他の同様のツールの優れた代替手段を提供します。

したがって、次のように追加します。

libraryDependencies += "org.webjars" % "swagger-ui" % "3.25.3"

コントローラーで、静的ライブラリのindex.htmlファイルへの呼び出しをリダイレクトするメソッドを作成します。

def redirectDocs: Action[AnyContent] = Action {
    Redirect(
       url = "/assets/lib/swagger-ui/index.html",
       queryStringParams = Map("url" -> Seq("/swagger.json")))
  }

まあ、私たちはルートファイルでルートを規定します：

GET   /docs                   controllers.HomeController.redirectDocs()

もちろん、webjars-playライブラリを接続する必要があります。依存して追加：

libraryDependencies +=  "org.webjars" %% "webjars-play" % "2.8.0"

ルートをルートファイルに追加します。

GET     /assets/*file               controllers.Assets.at(path="/public", file)

アプリケーションが実行されている場合、ブラウザに

http：// localhost：9000 / docs

と入力し、すべてが正しく行われれば、アプリケーションのswaggerページに移動します。



このページにはまだREST APIに関するデータが含まれていません。これを変更するには、Swagger-Playモジュールによってスキャンされる注釈を使用する必要があります。

def find(id: String): Action[AnyContent] = 
    safeAction(AllowRead(DrillObj)).async { implicit request =>
      drillsDao.findById(UUID.fromString(id))
        .map(x => x.fold(NotFound(s"Drill with id=$id not found"))(x =>

アノテーションを使用して、メソッドの説明、クエリ文字列から取得した入力パラメーター、サーバーからの応答を指定できます。成功した場合、メソッドはDrillクラスのインスタンスを返します。

 @ApiOperation(
    value = " ",
    response = classOf[Drill]
  )
  @ApiResponses(value = Array(
    new ApiResponse(code = 404, message = "Drill with id=$id not found")
  ))
  def find(@ApiParam(value = "String rep of UUID, id ") id: String)=
    safeAction(AllowRead(DrillObj)).async { implicit request =>
      drillsDao.findById(UUID.fromString(id))
        .map(x => x.fold(NotFound(s"Drill with id=$id not found"))(x =>
          Ok(write(x)))).recover(errorsPf)
    }



私たちは良い説明を得ました。Swaggerは、オブジェクトのシリアル化対象をほぼ推測しましたが、1つの例外があります。Drillクラスの開始フィールドと終了フィールドはInstantオブジェクトであり、Longでシリアル化されます。0をより適切な値に置き換えたい。これを行うには、@ ApiModel、@ ApiModelPropertyアノテーションをクラスに適用します。

@ApiModel
case class Drill(
                id: UUID,
                name: String,
                @ApiModelProperty(
                  dataType = "Long",
                  example = "1585818000000"
                )
                start: Instant,
                @ApiModelProperty(
                  dataType = "Long",
                  example = "1585904400000"
                )
                end: Option[Instant],
                isActive: Boolean
                )

これで、モデルの完全に正しい説明ができました。

例3


これまでのところ、すべてが簡単でした。次に、より複雑な例を示します。型パラメーターに応じて一般化されたクラスがあるとします。

case class SessionedResponse[T](
                            val ses: SessionData,
                            val payload: T
                          )

Swaggerは、少なくともジェネリックをまだ理解していません。注釈で示すことはできません：

@ApiOperation(
    value = " ",
    response = classOf[SessionedResponse[Drill]]
  )

この状況での唯一の方法は、必要な型ごとにジェネリック型をサブクラス化することです。たとえば、DrillSessionedResponseをサブクラス化できます。
唯一の問題は、caseクラスから継承できないことです。幸い、私のプロジェクトでは、ケースクラスをクラスに変更することを妨げるものは何もありません。次に：

class SessionedResponse[T](
                            val ses: SessionData,
                            val payload: T
                          )

object SessionedResponse {
  def apply[T](ses: SessionData, payload: T) = new SessionedResponse[T](ses, payload)
 
}

private[controllers] class DrillSessionedResponse(
          ses: SessionData,
          payload: List[Drill]
) extends SessionedResponse[List[Drill]](ses, payload)

これで、このクラスを注釈で指定できます。

@ApiOperation(
    value = " ",
    response = classOf[DrillSessionedResponse]
  )

実施例4


次に、ADTに関連するさらに複雑な例-代数的データ型。

SwaggerはADTを操作するためのメカニズムを提供します。

抽象@ApiModelには、この目的のために2つのオプションがあります

。1. subTypes-サブクラスの列挙

2. DISCRIMINATOR-サブクラスが互いに異なるフィールド。

私の場合、ケースクラスからJSONを生成するuPickleは、$タイプフィールド自体を追加し、ケース-オブジェクトを文字列にシリアル化します。したがって、弁別子フィールドを使用したアプローチは受け入れられませんでした。

別のアプローチを使用しました。あるとしましょう

sealed trait Permission

case class Write(obj: Obj) extends Permission
case class Read(obj: Obj) extends Permission

ここで、Objはケースオブジェクトで構成される別のADTです。

//  permission.drill
case object DrillObj extends Obj

// permission.team
case object TeamObj extends Obj

Swaggerがこのモデルを理解するには、実際のクラス（または特性）ではなく、この目的のために特別に作成されたクラスに必要なフィールドを提供する必要があります。

@ApiModel(value = "Permission")
case class FakePermission(
       @ApiModelProperty(
         name = "$type",
         allowableValues = "ru.myproject.shared.Read, ru.myproject.shared.Read"
       )
       t: String,
       @ApiModelProperty(allowableValues = "permission.drill, permission.team"
       obj: String
     )

次に、PermissionではなくFakePermissionをアノテーションに指定する必要があります。

@ApiImplicitParams(Array(
    new ApiImplicitParam(
      value = "",
      required = true,
      dataTypeClass = classOf[FakePermission],
      paramType = "body"
    )
  ))

コレクション


私が読者の注意を引き付けたかった最後のもの。私が言ったように、Swaggerはジェネリック型を理解しません。ただし、彼はコレクションの操作方法を知っています。

したがって、@ApiOperation アノテーションには、値「List」を渡すことができるresponseContainerパラメータがあります。

入力パラメータに関しては、表示

dataType = "List[ru.myproject.shared.roles.FakePermission]"

この属性をサポートするアノテーション内で、望ましい結果を生成します。ただし、scala.collection.Listを指定した場合は機能しません。

結論


私のプロジェクトでは、Swagger-Coreアノテーションを使用して、Rest-APIと、ジェネリック型や代数データ型を含むすべてのデータモデルを完全に説明することができました。私の意見では、Swagger-Playモジュールの使用は、API記述を自動的に生成するのに最適です。
