RubyRussia 2019：サーバーレスについてのニコライ・スベルチコフ

9月28日のRubyRussia カンファレンスで、Nikolai SverchkovがServerless is Ruby Rubyについて講演します。Ivan Solovyovはインタビューで、なぜこのトレンドが興味深いのか、そしてなぜルビストがそれに注目すべきなのかについて話しました。




Rubyに来た経緯を教えてください。

彼は大学でプログラミングを始めました。すべての実践はC ++で行われましたが、私はRubyで最終論文と論文を作成しました。 Rubyを選んだ理由は、はっきりとは言えませんが、おそらくその当時の言語は誇大宣伝の波に乗っていたためでしょう。 Ruby大学では誰も知りませんでした。でも卒業後、Javaで書こうと思った。それから、Javaは特別でクールな開発者、プログラマーの一種のより高いカーストであると私には思えました。私の最初の就職の面接はJavaジュニアでした、そして、私は首尾よく失敗しました。しかし、彼はRubyが必要な会社を得ることができました。多分それは最高です！ほんの数年がエリクサーに向かって移動しました。今私は邪悪な火星人でバックエンドをやっています。

サーバーレスに関するレポート。どのようにしてこの方向に取り組み始めましたか？技術への入り口は難しいですか？

彼は暇なときに勉強を始めた。私には本当の仕事がありましたが、主な仕事の仕事とは関係がありませんでした。その後、会社のプロジェクトでサーバーレスを使用し始めました。エントリーのしきい値はかなり低く、Herokuのしきい値より少し高いと思います。最初の「Hello、world！」を書いてくださいそれは非常に簡単です-Amazonからの記事、チュートリアル、非常に豊富なドキュメントがたくさんあります。そしてもちろん、それから困難があります。落とし穴があります、私はそれらのいくつかについてレポートで話します。

初心者はサーバーレスに飛び込み、本番環境で使用すべきですか？

私はイエスと思う！しかし、サーバーレスですべてを実装するために急ぐ必要はありません。最初に、アプリケーションを見て、その中に別のサービスに配置できる小さなビジネスロジックを見つけます。この新しいマイクロサービスにシンプルな通信インターフェイスがある場合、99％の確率で、同じAWS Lambdaを使用して実装できます。理想的には、それがビジネスロジックのステートレスな部分である場合、結果を保存する方法について考える必要はなく、ラムダ関数の実行のアーティファクトについて考える必要があります。例として、手紙を送ることは最初の良い仕事になりえます。私のレポートでは、サーバーレスコンピューティングモデルがどのタスクの範囲に最も適しているかという問題を個別に取り上げます。

主な推奨事項は、ビジネスロジックコードをラムダ自体から分離することです。これは、非常によく知られたルール「薄いコントローラー、厚いモデル」のバリエーションです。まず、この方法でテストする方が簡単です。次に、プロセスでサーバーレスが自分のタスクに適していないことを理解している場合は、実績のあるソリューションに簡単に移行できます。たとえば、サーバーレス入力レイヤーを通常のWebサーバーに置き換えることができます。この点で、非常に良いジェッツ。これはRubyサーバーレスフレームワークであり、すぐにAWS Lambdaと通常のAmazon EC2インスタンスの両方にデプロイできるアプリケーションを作成できます。

このRubyフレームワークについて詳しく教えてください。

ジェッツは一種です。特定の言語に合わせて調整された他のサーバーレスフレームワークがあるという事実にもかかわらず、Jetsが最も機能的です。現在、彼はマスターで2500以上のコミットをしています。このフレームワークは、Ruby開発者にとっておなじみの多くの規則を使用しています。これは、このような「サーバーレスのRuby on Rails」です。しかし、これには欠点があります。サーバーレスコンピューティングモデルでは、コードの実行時間に対して料金を支払うため、文字通り、大量の依存関係の初期化とロードにはコストがかかります。このトピックの開示についても、私の報告で時間が与えられます。

現在、サーバーレスで最も一般的なプラットフォームは何ですか？それらの違いは何ですか？どちらを選ぶのが良いですか？私が理解しているように、ほとんどの場合それについて話すので、あなたは今アマゾンウェブサービスに踏み込んでいます。

私が会話で「ラムダ」という単語をよく使用する理由を尋ねますか？物語はカーチャーやゼロックスのようなものだと思います。ニッチな製品と呼ぶのが通例のブランドもあります。 2014年、Amazonは最初にサーバーレスコンピューティングモデル-AWS Lambdaへのパブリックアクセスを提供しました。それ以外のすべてのユーザー：Azure Functionsを使用するMicrosoft、Cloud Functionを使用するGoogle、IBM Cloud Functionsを使用するIBMは、まだ精神的に追いついています。サーバーレス市場は活況を呈していますが、AWSのサービス価格は多くの場合、競合他社よりも高くなっています。したがって、Google Cloud Runなどの新しいリリースは、ゲームを一夜にして変えることができます。プラットフォームの比較についてより詳細な分析を行う場合は、DUMP 2019カンファレンスのDataArtのRuslan Serkinのビデオを視聴することをお勧めします。

, serverless?

ああ、これは私のレポートの最も興味深い部分です！私はネタバレなしでできる-聞いてください！代わりに、私は奉仕と生産について話すことができます。今年の4月に、同じJetsの作者が講演したミンスクでのRubyConfByカンファレンスに参加しました。アフターパーティでは、サーバーレス開発について説明し、主要な業界関係者からのサポートがある場合、ラムダ関数の大量配布が見られない理由について説明しました。モデルの利点は明白であり、エコシステムは透過的ですが、ITコミュニティの側からの広範な信頼はありません。その結果、今やサーバーレスは一種のシャドウプレーヤーであり、かつての状況はDockerでかつてあった状況を彷彿とさせるという結論に達しました。運用中のDockerは自殺であると長い間考えられてきました。コンテナー化がソフトウェア配布の主要なツールであることがわかります。私は将来、サーバーレスでも同じことを期待していると思います。より多くの人々が彼を信頼するでしょう。

サーバーレスはモノリスを殺しますか？

この質問は、「マイクロサービスアーキテクチャはモノリスを殺すか？」と言い換えることができます。サーバーレスは理想的なマイクロサーバーであるため、小型であり、ステートレスであり、外界と通信するための最小限のインターフェースを備えています。マイクロサービスがモノリスを殺さないように、ラムダもモノリスを殺しません。 3つのアーキテクチャにはすべて、それらに最適な特定の範囲のタスクがあります。逆もまた同様です。たとえば、サーバーレスを使用することが実用的でないタスクがあります。レポートで詳細を確認してください。

コードを使用するときに一般的なラムダをどうするか？たとえば、いくつかの一般的なコンポーネント、いくつかの実用的な機能などです。

この問題に関するベストプラクティスはまだありません。そして、質問は、「マイクロサービス間で共通のコードをいじくるにはどうすればいいのか？」にもう一度書き直すことができます。なぜなら、意味は同じだからです。再利用されたロジックを共有のどこかに保持し、関数自体でそれを接続するオプションがあります。このアプローチは、たとえば、出力で1つの実行可能ファイルを取得するGoでうまく機能します。別のオプションは、コンポーネントの接続を取り除き、複製を許可することです。おそらく試してみる価値があり、最も効果的なものを確認してください。私がアドバイスできる唯一のことは、普遍的なラムダ関数を作ろうとしないことです。ある時点で、1つのuber関数がコードの重複の問題に対する完全なソリューションであると思われるかもしれません。しかし、時間が経つにつれて、これはどこにも行かない道であることがわかります。 1番目と2番目のアーキテクチャのすべての問題を含む、特定の「サーバーレスのモノリシックバージョン」が得られます。
