Web应用程序：Micro Frontend Module Federation支持的框架

我想介绍一个支持Webpack Module Federation的Microfront框架。该框架允许您链接用任何库，香草JS甚至IFrame编写的应用程序（如果情况确实很糟）。


最近有一篇文章详细描述了市场上的东西。但是，大多数解决方案都强加了一些其他概念，无论是路由选择还是应用程序注册表，尽管实际上有时这些都不是必需的，但是仅需要一种简单的方法即可将一个应用程序实现为另一个应用程序。


但是，让我们退后一步，为什么我们需要一个微前端方法？在某些时候，任何大型应用程序本身都会增长，构建过程会减慢速度，每一个小的更改都会导致在监视模式下等待几分钟，代码过多，因此无法再有效地支持它。


如果团队希望切换到新的堆栈/框架，则问题会更加严重。我们试图从一个框架平稳地转换到另一个框架，用一个整体替换另一个。整个过渡时期必须同时支持两种产品，而当我们最终完成时，最后的整体绝对是巨大的。在某个时候，表达了将其分解为子应用程序的想法，但是每个人都是由不同的团队开发的，具有与主机应用程序进行通信的不同接口，并且某些子应用程序通常是IFrame，例如 需要最大程度的隔离。尽管有动物园的缺点，但这种方法仍然有效，而且效果很好。但是从长远来看，他没有工作。


如何将在一个框架上编写的应用程序包含在另一个框架上的另一个应用程序中？**需要一个抽象乐团！**

在直接进入框架之前，我们先概述一下基本术语：



  • 主机是包含较小应用程序的应用程序。
  • 子应用程序（或仅仅是应用程序）是包含在主机中的IFrame，全局应用程序或Web组件应用程序，它们又可以是更多嵌入式应用程序的主机。
  • Orchestrator：Web Apps Framework-一种代理，允许主机与应用程序通信，下载应用程序等。
  • 应用程序不直接相互通信，但是主机可以将它们连接

Web Apps框架是一种与框架无关的方法，可将任何应用程序实现到任何应用程序中，并具有通用的通信接口和其他优点。

import React from "react";
import {render, unmountComponentAtNode} from "react-dom";
import {App} from './app';

const template = document.createElement('template');

template.innerHTML = `
    <style>/* shadow CSS */</style>
    <div class="container"></div>
`;

customElements.define('web-app-%appId%', class extends HTMLElement {
    constructor() {
        super();
        this.attachShadow({mode: 'open'});
        this.shadowRoot.appendChild(
            document.importNode(template.content, true)
        );
        this.mount = this.shadowRoot.querySelector('.container');
    }

    connectedCallback() { render(
        <App 
            authtoken={this.getAttribute('authtoken')}
            node={this}
        />, 
        this.mount
    ) }

    disconnectedCallback() { unmountComponentAtNode(this) }    
});
