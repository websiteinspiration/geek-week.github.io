スレッドのトリミング：Puppet EnterpriseからAnsible Towerへの移行。パート1

国立環境衛星情報サービス（NESDIS）は、Puppet EnterpriseからAnsible Towerに移行することにより、Red Hat Enterprise Linux（RHEL）の構成管理コストを35％削減しました。「方法」カテゴリのこのビデオでは、システムエンジニアのMichael Rauがこの移行の実装を実証し、1つのSCMから別のSCMへの移行の結果として得られた有用なヒントと経験を共有しています。

このビデオから次のことを学びます。

  • Puppet EnterpriseからAnsible Towerへの移行の実現可能性を管理者に正当化する方法。
  • 最もスムーズな移行に使用する戦略;
  • Ansible PlaybookでPEマニフェストをトランスコードするためのヒント。
  • Ansible Towerをインストールするためのベストプラクティス。



みなさん、こんにちは。マイケル・ラウです。NESDISの全米海洋大気管理局（NOAA）で働いているActioNetの上級システムエンジニアです。今日は、ラインのトリミングについて説明します。私自身の経験は、Puppet EnterpriseからAnsible Towerへの移行です。このプレゼンテーションのテーマは、私が年初にこの移行を行った後に残った「私の傷跡を見る」ことです。この過程で学んだことを伝えたいと思います。これを行うと、私の経験を使用して、それほど難しいことなく移行を行うことができます。

Ansible Festでのすべてのプレゼンテーションの最初に、このようなスライドが表示されます。このスライドは、私の会社の自動化の歴史を示しています。 2007年からPuppet / Puppet Enterpriseを使用しているため、私はこのビジネスに慣れていません。私は2016年にAnsibleで作業を開始しました。この製品の他の多くのユーザーと同様に、コマンドラインと簡単なスクリプト（プレイブック）を使用した「トリック」の可能性に惹かれました。 2017年の終わりに、私はAnsible Towerに移行する正当な理由についてリーダーシップに向けました。すぐに、私がこのステップを踏むように促した理由について話します。経営陣の同意を得てから、計画が完了するまでにさらに数か月を要し、今年の1月から2月に移行しました。それで、Ansibleを支持して完全にPuppetを放棄しました。これは素晴らしいことです。



私をAnsibleに最も引き付けているのは、ロールとスクリプト（プレイブック）を作成して使用できることです。ロールは、異なるが相互に関連するタスク（タスク）を作成し、これらのタスクに関連するすべてのデータを1か所に配置するのに最適です。 Playbookは、1つ以上のホストのアクションを記述するスクリプトファイルであるYAML構文です。これらの機能についてユーザー、主にソフトウェア開発者に説明します。 Ansible Towerを使用すると、「いいえ、シェルアクセスにアクセスできませんが、すべてのTowerプロセスを開始し、必要なときにサービスを再起動する機会が与えられます。」私たちが使用する作業環境と設備についてお知らせします。



これは連邦政府のLAN、クラウドベースのMPLSを介して接続された7つの物理サイト、140のRHELサーバー、その99％は仮想（vSphere）、SuperMicroハードウェア、NexentaStore NAS、一連のCisco、Arista、Cumulusスイッチ、フォーティネットUTM統合脅威管理ツールです。 。

連邦ネットワークとは、立法行為によって提供されるあらゆる情報保護手段を使用しなければならないことを意味します。 Puppet Enterpriseは、使用しているほとんどの機器をサポートしていないことを覚えておいてください。政府機関がこの支出項目の資金調達に問題を抱えているため、予算ハードウェアの使用を余儀なくされています。そのため、SuperMicroクラスのハードウェアを購入し、政府の契約により保守が保証されている個々の部品から機器を組み立てます。私たちはLinuxを使用していますが、これがAnsibleに切り替える重要な理由の1つです。

パペットとの私たちの歴史はこれです。



2007年には、20〜25ノードの小規模なネットワークがあり、そこにPuppetを導入しました。基本的に、これらのノードはRedHatの単なる「ボックス」でした。 2010年に、45ノードのPuppet Dashboard Webインターフェースの使用を開始しました。ネットワークが拡大し続けたため、2014年にPE 3.3に切り替え、75ノードのマニフェストを書き換えて完全な移行を行いました。これは、Puppetがゲームのルールを変更することを好み、この場合は言語を完全に変更したためです。 1年後、Puppet Enterpriseのバージョン3のサポートが停止したため、PE 2015.2への移行を余儀なくされました。新しいサーバーのマニフェストを再度書き換えて、100ノードの予約でライセンスを購入する必要がありましたが、当時は85ノードしかありませんでした。

わずか2年が経過しましたが、PE 2016.4の新しいバージョンに切り替えるには、やはり多くの作業を行う必要がありました。私たちは300ノード、合計130ノードのライセンスを購入しました。新しいバージョンの言語の構文は2015バージョンの言語とは異なるため、マニフェストに大幅な変更を加える必要がありました。その結果、SCMはSVNバージョン管理システムからBitbucket（Git）に切り替わりました。それが人形との「関係」でした。

そのため、次の引数を使用して別のSCMに切り替える必要がある理由を経営者に説明する必要がありました。 1つ目は、サービスの高価格です。 RedHatの担当者と話しましたが、彼らは、Ansible Towerを使用して300ノードのネットワークを維持するコストは、Puppet Enterpriseのコストの半分であると述べました。別のAnsible Engineを購入した場合、コストはほぼ同じですが、PEよりもはるかに多くの機能を利用できます。私たちは連邦予算によって資金を供給されている国営企業であるため、これはかなり重要な議論です。



2番目の引数は汎用性です。 Puppetは、Puppetエージェントを備えたハードウェアのみをサポートします。つまり、すべてのスイッチにエージェントをインストールする必要があり、エージェントは最新バージョンである必要があります。また、スイッチの一部が1つのバージョンをサポートし、一部が別のバージョンをサポートしている場合、それらすべてに同じバージョンのPEエージェントをインストールして、すべて同じSCMシステムで動作できるようにする必要があります。

Ansible Towerシステムは、エージェントがないため、動作が異なりますが、Ciscoスイッチと他のすべてのスイッチをサポートするモジュールがあります。このSCMは、Qubes OS、Linux、および4.NET UTMをサポートしています。 Ansible Towerは、オープンソースのUnixベースのオペレーティングシステムであるIllumosカーネルに基づくNexentaStore NASもサポートしています。これはほとんどサポートされていませんが、とにかくAnsible Towerがサポートします。

3番目の議論は、私にとっても行政にとっても非常に重要であり、開発のしやすさです。私はPuppetマニフェストモジュールとコードを10年間習得しましたが、このSCMで作業する方がはるかに簡単なので、Ansibleを1週間勉強しました。もちろん、実行可能ファイルを実行する場合、不必要に実行しないと、合理的で応答性の高いハンドラーがそれらを使用して機能します。 YAMLベースのプレイブックスクリプトは、簡単に習得でき、すぐに使用できます。 YAMLについて聞いたことがない人は、スクリプトを読むだけで、YAMLの仕組みを簡単に理解できます。

正直なところ、PuppetはPuppet Masterの使用に基づいているため、開発者としての作業を複雑にします。これは、Puppetエージェントとの通信を許可された唯一のマシンです。マニフェストに変更を加えてコードをテストする場合は、Puppetマスターのコードを書き直す必要があります。つまり、すべてのクライアントを接続してPuppetサーバーサービスを開始するようにPuppetマスターファイル/ etc / hostsを構成する必要があります。そうして初めて、1台のホストでネットワーク機器の動作をテストできます。これはかなり苦しい手順です。
Ansibleでは、すべてがはるかに簡単です。必要なことは、SSHを介してテスト対象のホストと通信できるマシン用のコードを開発することだけです。これは作業がはるかに簡単です。

Ansible Towerの次の大きなプラスは、既存のサポートシステムを活用して、既存のハードウェア構成を保存できることです。追加のアクションなしのこのSCMは、インフラストラクチャと機器、仮想マシン、サーバーなどに関するすべての利用可能な情報を使用します。RH Satelliteサーバーがあればそれと通信することができ、Puppetを操作するときに決して得られない統合を提供します。

もう1つの重要なことは、詳細な制御です。Puppetはモジュラーシステムであり、クライアントサーバーアプリケーションであることを知っているので、1つの長いマニフェストですべてのマシンの操作の既存の側面を判断する必要があります。この場合、システムの各要素の状態を30分ごとにテストする必要があります。これはデフォルトの期間です。これがPuppetの仕組みです。

タワーはこれからあなたを救います。さまざまな機器でさまざまなプロセスを制限なしに実行できます。メインの作業を行ったり、他の重要なプロセスを開始したり、セキュリティシステムを構成したり、データベースを操作したりできます。 Puppet Enterpriseでは、特定の困難に伴うすべてのことを実行できます。そのため、1つのホストで構成した場合、他のホストで変更が有効になるまでに時間がかかります。 Ansibleでは、すべての変更が同時に有効になります。

最後に、セキュリティモジュールについて検討します。 Ansible Towerでは、非常に正確かつ完全に、驚くほど簡単に実装されています。ユーザーに特定のサービスまたは特定のホストへのアクセスを提供できます。私はこれをWindowsでの作業に慣れている従業員と行い、Linuxシェルへのアクセスを制限しています。私は彼らにタワーへのアクセスを提供して、彼らが仕事をし、彼らの能力の範囲内にあるそれらのサービスだけを実行できるようにします。



Ansible Towerへの移行を容易にするために事前に行う必要があることを見てみましょう。まず、機材を準備する必要があります。インフラストラクチャの要素がまだデータベースにない場合は、そこに追加する必要があります。システムの特性を変更しないためPuppetデータベースにはないシステムがありますが、Towerに移行する前にシステムを追加しないと、多くの利点が失われます。これは「ダーティ」な予備データベースの場合がありますが、所有しているすべての機器に関する情報が含まれている必要があります。したがって、データベースのすべてのインフラストラクチャを自動的に変更する動的なハードウェアスクリプトを作成する必要があります。そうすると、Ansibleは新しいシステムにあるホストを認識します。このSCMに通知する必要はありません。追加したホストと存在しなくなったホスト。これは、彼女がすべてを自動的に知っているためです。データベースに含まれるデータが多いほど、Ansibleはより便利で柔軟になります。データベースから機器のステータスのバーコードを読み取るかのように機能します。

Ansibleのコマンドラインを理解するために、少し時間をかけてください。いくつかの特別なコマンドを実行してハードウェアスクリプトをテストし、シンプルだが便利なプレイブックスクリプトを作成して実行し、必要に応じてJinja2テンプレートを使用します。頻繁に遭遇する標準的なハードウェア構成を使用して、複雑なマルチステージプロセスのロールとスクリプトを記述してみてください。これらのことを試して、どのように機能するかをテストしてください。このようにして、タワーで使用されるライブラリを作成するためのツールを使用する方法を学びます。移行の準備には約3か月かかったと既に述べました。私の経験に基づいて、あなたはそれをより速く行うことができると思います。この時間が失われたと考えないでください。後で作業の利点がすべて感じられるからです。

次に、Ansible Towerに何を期待するか、このシステムで何ができるかを決定する必要があります。



空の仮想マシン上の空のハードウェアにシステムを展開する必要がありますか？それとも、作業の元の状態と既存の機器の設定を保持しますか？これは国有企業の作業にとって非常に重要な側面であるため、Ansibleを既存の構成に移行してデプロイできることを確認する必要があります。自動化する日常的な管理プロセスを特定します。新しいシステムに特定のアプリケーションとサービスを展開する必要があるかどうかを確認します。やりたいことのリストを作成し、優先順位を付けます。

次に、計画したタスクを完了するのに役立つスクリプトとロールのコードの記述を開始します。それらを、関連するプレイブックスクリプトの論理的なコレクションであるプロジェクトで組み合わせます。各プロジェクトは、使用するコードマネージャーに応じて、個別のGitリポジトリまたは別のリポジトリに関連付けられます。 PlaybookスクリプトとPlaybookディレクトリを管理するには、タワーサーバーのProject Base Pathに手動で配置するか、Git、Subversion、Mercurial、Red Hat Insightsなどのタワーのソースコード管理（SCM）システムに配置します。 1つのプロジェクト内に、必要なだけスクリプトを配置できます。たとえば、1つの基本的なプロジェクトを作成し、RedHatの主要要素用のスクリプト、Linuxの基本用のスクリプトを配置しました。残りのベースラインのシナリオ。したがって、1つのプロジェクトには、1つのGitリポジトリから管理されるさまざまな役割とスクリプトがありました。

これらすべてをコマンドラインで実行します。これは、それらのパフォーマンスをテストするための良い方法です。このようにして、タワーを設置する準備が整います。

Puppetマニフェストのトランスコードについて少しお話ししましょう。実際に何をする必要があるのか​​を理解するまで、多くの時間を費やしたからです。



前に述べたように、Puppetはすべての設定と機器パラメーターを1つの長いマニフェストに格納し、このマニフェストにはこのSCMが実行する必要があるすべてのものが含まれています。移行中は、すべてのタスクを1つのリストにまとめる必要はありません。代わりに、新しいシステムの構造（役割、スクリプト、タグ、グループ、そしてそこに何を入れるべきか）について考えてください。一部のスタンドアロンネットワーク要素は、スクリプトを作成できるグループにグループ化する必要があります。自律クラスを含む、多数のリソースを含むより複雑なインフラストラクチャ要素は、役割で組み合わせることができます。移行する前に、これを決定する必要があります。 1つの画面に収まらない大量のロールまたはスクリプトを作成する場合は、インフラストラクチャの個々の部分をキャプチャできるようにタグを使用する必要があります。
