PVS-Studioを使用したTravis CI、Buddy、AppVeyorでのコミットとプルリクエストの分析


バージョン7.04以降、LinuxおよびmacOSのCおよびC ++言語用のPVS-Studioアナライザーには、指定されたファイルのリストをチェックするテスト機能があります。新しいモードを使用して、コミットとプル要求をチェックするようにアナライザーを構成できます。この記事では、Travis CI、Buddy、AppVeyorなどの一般的なCI（継続的インテグレーション）システムで変更されたGitHubプロジェクトファイルのリストをチェックするように構成する方法を示します。

ファイルリストチェックモード


PVS-Studioは、C、C ++、C＃、およびJavaで記述されたプログラムのソースコードのエラーおよび潜在的な脆弱性を検出するためのツールです。 Windows、Linux、macOSの64ビットシステムで動作します。

LinuxおよびmacOS用のPVS-Studioバージョン7.04では、ソースファイルのリストをチェックするモードが表示されました。これは、ビルドシステムでcompile_commands.jsonファイルの生成を許可するプロジェクトで機能します。これは、アナライザーが指定されたファイルのコンパイルに関する情報を抽出するために必要です。ビルドシステムがcompile_commands.jsonファイルの生成をサポートしていない場合は、Bearユーティリティを使用してそのようなファイルを生成することができます。

また、ファイルリストチェックモードは、コンパイラ起動時のstraceトレースログ（pvs-studio-analyzer trace）と併用できます。これを行うには、まずプロジェクトの完全なアセンブリを実行し、それを追跡して、アナライザーがすべてのテスト済みファイルのコンパイルパラメーターに関する完全な情報を収集するようにする必要があります。

ただし、このオプションには重大な欠点があります-開始時にプロジェクト全体のアセンブリの完全なトレースを作成する必要があるため、コミットを迅速にチェックするという考え方自体に矛盾します。または、トレース結果自体がキャッシュされている場合、トレース後にソースファイルの依存関係構造が変更されると（たとえば、新しい#includeがソースファイルの1つに追加されるなど）、後続のアナライザーの起動が完了しない場合があります。

したがって、トレースログでファイルリストチェックモードを使用してコミットまたはプルリクエストをチェックすることはお勧めしません。コミットのチェック時に増分アセンブリを実行できる場合は、増分分析モードの使用を検討してください。

分析用のソースファイルのリストはテキストファイルに保存され、-Sパラメーターを使用してアナライザーに転送されます。

pvs-studio-analyzer analyze ... -f build/compile_commands.json -S check-list.txt

このファイルでは、ファイルへの相対パスまたは絶対パスが示されています。新しいファイルはそれぞれ新しい行に配置する必要があります。分析用のファイル名だけでなく、さまざまなテキストを指定することもできます。アナライザは、これがファイルではないことを確認し、その行を無視します。これは、ファイルが手動で指定されている場合にコメントを付けるのに役立ちます。ただし、CIでの分析中にファイルのリストが生成されることがよくあります。たとえば、これらのファイルは、コミットまたはプルリクエストのファイルである場合があります。

このモードを使用すると、メインの開発ブランチに入る前に新しいコードをすばやく確認できます。検証システムがアナライザーの警告に応答するために、フラグ--indicate-warningsがplog-converter ユーティリティに追加されました。

plog-converter ... --indicate-warnings ... -o /path/to/report.tasks ...

このフラグを使用すると、アナライザーレポートに警告がある場合、コンバーターはゼロ以外のコードを返します。戻りコードを使用して、プリコミットフック、コミットまたはプルリクエストをブロックし、生成されたアナライザーレポートを画面に表示したり、メールで共有または送信したりできます。

注意。ファイルリスト分析を初めて実行すると、プロジェクト全体が分析されます。アナライザーは、ヘッダーファイルからプロジェクトソースファイルの依存関係のファイルを生成する必要があります。これは、CおよびC ++ファイルの分析の機能です。将来的には、依存関係ファイルをキャッシュすることができ、アナライザーによって自動的に更新されます。増分分析モードを使用するよりもファイルリスト検証モードを使用する場合にコミットをチェックする利点は、オブジェクトファイルではなく、このファイルのみをキャッシュする必要があることです。

プルリクエスト分析の一般原則


プロジェクト全体の分析には時間がかかるため、プロジェクトの特定の部分のみをチェックすることは理にかなっています。問題は、新しいファイルを残りのプロジェクトファイルから分離する必要があることです。

2つのブランチを持つコミットツリーの例を考えます。



コミットA1に、すでにテスト済みのかなり大量のコードが含まれているとしましょう。少し前に、コミットA1からブランチを作成し、いくつかのファイルを変更しました。

もちろん、A1の後はさらに2つのコミットがあることに気づきましたが、マスターではコミットしないため、これらは他のブランチのマージでもありました。そして今、修正プログラムの準備が整う時が来ました。したがって、B3とA3の合併のプルリクエストが表示されました。

もちろん、それらのマージの結果全体をチェックすることはできますが、変更されたファイルが少ないため、これは長すぎて正当化されません。したがって、変更されたものだけを分析する方が効率的です。

これを行うには、ブランチの違いを取得します。マスターにマージするブランチのHEADにあります。

git diff --name-only HEAD origin/$MERGE_BASE > .pvs-pr.list

$ MERGE_BASEは後で詳しく検討します。実際のところ、すべてのCIサービスがマージの基礎に関する必要な情報を提供するわけではないため、このデータを取得するための新しい方法を考え出す必要があります。これについては、以下で説明する各Webサービスで詳しく説明します。

したがって、ブランチ間の違い、つまり変更されたファイル名のリストが得られました。次に、.pvs-pr.listファイル（出力を上記にリダイレクトした）をアナライザーに渡す必要があります。

pvs-studio-analyzer analyze -j8 \
                            -o PVS-Studio.log \
                            -S .pvs-pr.list

分析後、ログファイル（PVS-Studio.log）を読みやすい形式に変換する必要があります。

plog-converter -t errorfile PVS-Studio.log --cerr -w

このコマンドは、stderr（標準エラーメッセージ出力ストリーム）にエラーをリストします。

エラーを表示するだけでなく、問題の組み立てとテストのためにサービスに通知する必要があります。これを行うために、-Wフラグ（--indicate-warnings）がコンバーターに追加されました。アナライザー警告が少なくとも1つある場合、plog-converterユーティリティーの戻りコードは2に変わり、プル要求ファイルに潜在的なエラーが存在することをCIサービスに通知します。

トラビスci


構成は.travis.ymlファイルの形式です。便宜上、すべてを.travis.ymlファイルから呼び出される関数（bash script_name.sh function_name）を含む個別のbashスクリプトに入れることをお勧めします。

必要なコードをbashスクリプトに追加して、より多くの機能を利用できるようにします。インストールセクションで、次のように記述します。

install:
  - bash .travis.sh travis_install

指示があった場合は、ハイフンを削除してスクリプトに転送できます。.travis.sh

ファイルを開き、アナライザーのインストールをtravis_install（）関数に追加します。

travis_install() {
  wget -q -O - https://files.viva64.com/etc/pubkey.txt \
    | sudo apt-key add -
  sudo wget -O /etc/apt/sources.list.d/viva64.list \
    https://files.viva64.com/etc/viva64.list
  
  sudo apt-get update -qq
  sudo apt-get install -qq pvs-studio 
}

スクリプト セクションに分析の実行を追加します。

script:
  - bash .travis.sh travis_script

そしてbashスクリプトで：

travis_script() {
  pvs-studio-analyzer credentials $PVS_USERNAME $PVS_KEY
  
  if [ "$TRAVIS_PULL_REQUEST" != "false" ]; then
    git diff --name-only origin/HEAD > .pvs-pr.list
    pvs-studio-analyzer analyze -j8 \
                                -o PVS-Studio.log \
                                -S .pvs-pr.list \
                                --disableLicenseExpirationCheck
  else
    pvs-studio-analyzer analyze -j8 \
                                -o PVS-Studio.log \
                                --disableLicenseExpirationCheck
  fi
  
  plog-converter -t errorfile PVS-Studio.log --cerr -w
}

このコードは、プロジェクトのビルド後に実行する必要があります。たとえば、CMakeでビルドした場合は、次のようになります。

travis_script() {
  CMAKE_ARGS="-DCMAKE_EXPORT_COMPILE_COMMANDS=On ${CMAKE_ARGS}"
  cmake $CMAKE_ARGS CMakeLists.txt
  make -j8
}

次のようになります。

travis_script() {
  CMAKE_ARGS="-DCMAKE_EXPORT_COMPILE_COMMANDS=On ${CMAKE_ARGS}"
  cmake $CMAKE_ARGS CMakeLists.txt
  make -j8
  
  pvs-studio-analyzer credentials $PVS_USERNAME $PVS_KEY
  
  if [ "$TRAVIS_PULL_REQUEST" != "false" ]; then

    pvs-studio-analyzer analyze -j8 \
                                -o PVS-Studio.log \
                                -S .pvs-pr.list \
                                --disableLicenseExpirationCheck
  else
    pvs-studio-analyzer analyze -j8 \
                                -o PVS-Studio.log \
                                --disableLicenseExpirationCheck
  fi
  
  plog-converter -t errorfile PVS-Studio.log --cerr -w
}

指定された環境変数$ TRAVIS_PULL_REQUESTと$ TRAVIS_BRANCHにすでに気付いているでしょう。Travis CIが独自に発表します。

  • $ TRAVIS_PULL_REQUESTはプルリクエスト番号を格納します。通常のブランチの場合はfalseを格納します。
  • $ TRAVIS_REPO_SLUGは、プロジェクトリポジトリの名前を格納します。

この関数のアルゴリズム：


Travis CIは戻りコードに応答するため、警告があると、サービスにコミットにエラーが含まれているとマークするよう指示されます。

それでは、次のコード行を詳しく見てみましょう。

git diff --name-only origin/HEAD > .pvs-pr.list

実際のところ、Travis CIはプルリクエスト分析中にブランチを自動的にマージします。


したがって、B3-> A3ではなくA4を分析します。この機能のため、originからのブランチの一番上にあるA3との差を計算する必要があります。 重要な詳細が1つ残っています。コンパイル済みの翻訳単位（* .c、* .cc、* .cppなど）のヘッダーファイルの依存関係をキャッシュします。アナライザーは、ファイルのリストをチェックするモードの最初の開始時にこれらの依存関係を計算し、次にそれらを.PVS-Studioディレクトリーに保存します。 Travis CIではフォルダをキャッシュできるため、.PVS-Studio /ディレクトリのデータを保存します。



cache:
  directories:
    - .PVS-Studio/

このコードは.travis.ymlファイルに追加する必要があります。このディレクトリには、分析後に収集されたさまざまなデータが格納されます。これにより、ファイルリスト分析または増分分析の以降の起動が大幅に高速化されます。これが行われない場合、アナライザーは実際に毎回すべてのファイルを実際に分析します。

バディ


Travis CIと同様に、BuddyはGitHubに保存されているプロジェクトを自動的にビルドしてテストする機能を提供します。Travis CIとは異なり、Webインターフェースで構成されているため（bashサポートが利用可能）、プロジェクトに構成ファイルを保存する必要はありません。

まず、アセンブリーラインに新しいアクションを追加する必要があります。


プロジェクトのビルドに使用されたコンパイラを示します。このアクションでインストールされるDockerコンテナーに注意してください。たとえば、GCCには特別なコンテナがあります。


次に、PVS-Studioと必要なユーティリティをインストールします。

  https://files.viva64.com/etc/viva64.list

apt-get update && apt-get -y install pvs-studio

    https://api.github.com/repos/${BUDDY_REPO_SLUG}/${PULL_REQUEST_ID} \

                              -o PVS-Studio.log \

Travs-CIのセクションを読んだ場合、このコードは既におなじみですが、新しいステージが表示されています。


事実は、今はマージの結果ではなく、プルリクエストが行われたブランチのHEADを分析しているということです。


したがって、私たちは条件付きコミットB3にあり、A3との違いを取得する必要があります。

PULL_REQUEST_ID="pulls/$BUDDY_EXECUTION_PULL_REQUEST_NO"
  MERGE_BASE=`wget -qO - \
    https://api.github.com/repos/${BUDDY_REPO_SLUG}/${PULL_REQUEST_ID} \
    | jq -r ".base.ref"`
git diff --name-only HEAD origin/$MERGE_BASE > .pvs-pr.list

A3 を決定するには、GitHub APIを使用します。

https://api.github.com/repos/${USERNAME}/${REPO}/pulls/${PULL_REQUEST_ID}

バディが提供する次の変数を使用しました。

  • $ BUDDY_EXECUTION_PULL_REQEUST_NO-プルリクエスト番号;
  • $ BUDDY_REPO_SLUG-ユーザー名とリポジトリの組み合わせ（たとえば、max / test）。

下のボタンを使用して変更を保存し、プルリクエスト分析を有効にします。


Travis CIとは異なり、Buddyは以降の起動のためにすべてのファイルを自動的にキャッシュするため、キャッシュに.pvs-studioを指定する必要はありません。したがって、最後に残っているのは、PVS-Studioのログインとパスワードをバディに保存することです。変更を保存したら、パイプラインに戻ります。変数の設定に進み、PVS-Studioのログインとキーを追加する必要があります。


その後、新しいプルリクエストまたはコミットの出現によりチェックがトリガーされます。コミットにエラーが含まれている場合、Buddyはプルリクエストページでこれを示します。

Appveyor


AppVeyorのセットアップはBuddyに似ています。すべてがWebインターフェイスで行われ、プロジェクトリポジトリに* .ymlファイルを追加する必要がないためです。

このページを下にスクロールし、キャッシュ保存をオンにしてプルリクエストを作成します。

結論


もちろん、継続的インテグレーションのすべての可能なサービスを検討したわけではありませんが、それらはすべて、作業の詳細に非常に類似しています。キャッシングを除いて、各サービスは独自の「バイク」を作成するため、すべてが常に異なります。

Travis-CIのようなどこかで、数行のコードとキャッシングが問題なく機能します。AppVeyorのように、設定でフォルダーを指定するだけです。しかし、どこかで一意のキーを作成し、システムに納得させて、キャッシュされたフラグメントを上書きする機会を与える必要があります。したがって、上で説明しなかった、継続的インテグレーションサービスでのプルリクエストの分析を設定する場合は、最初にキャッシングに問題がないことを確認してください。
