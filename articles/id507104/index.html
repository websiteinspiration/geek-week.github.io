Perpecahan besar dalam impor: mengklarifikasi ketidakpastian dengan impor dalam naskah

Terjemahan artikel disiapkan sebelum dimulainya kursus "Pengembang React.js"


Saya telah bekerja dengan naskah untuk beberapa waktu, dan saya punya banyak masalah dengan memilah modul dan pengaturan saran, dan harus saya katakan, ada banyak hal yang tidak dapat dipahami di sekitar mereka. Namespace import * as React from 'react', esModuleInteropdll. Karena itu, mari kita pahami mengapa semua hype muncul.

Saya tidak akan berbicara tentang ruang nama sebagai sistem modular dalam naskah, karena idenya bukan yang terbaik (terutama mengingat vektor pengembangan saat ini), dan tidak ada yang menggunakannya sekarang.

Jadi, bagaimana hal-hal sebelum mereka muncul esModuleInterop? Ada hampir semua modul yang sama yang dimiliki oleh babel atau browser, dan juga impor / ekspor. Namun, dalam hal ekspor dan impor secara default, naskah memiliki opsi sendiri: perlu untuk menulis import * as React from 'react'(sebagai gantinya import React from 'react'), dan, tentu saja, di sini kita berbicara tidak hanya tentang reaksi, tetapi tentang semua impor default dari commonjs. Bagaimana hal itu terjadi?

Untuk memahami ini, mari kita lihat bagaimana kompatibilitas antara beberapa pola commonjsdan modul bekerja es6. Misalnya, kami memiliki modul yang mengekspor foodan barsebagai kunci:

module.exports = { foo, bar }

Kita dapat mengimpor menggunakan persyaratan dan penghancuran:

const { foo, bar } = require('my-module')

Dan menerapkan prinsip yang sama menggunakan impor yang disebutkan (meskipun, jujur, ini bukan perusakan):

import { foo, bar } from 'my-module'

Namun, pola yang lebih umum di commonjs adalah const myModule = require('my-module')(karena belum ada perusakan), tetapi bagaimana Anda melakukannya es6?

Ketika mengembangkan spesifikasi untuk impor, es6salah satu aspek penting adalah kompatibilitas commonjs, karena commonjsbanyak kode sudah ditulis di atasnya. Ini adalah bagaimana impor dan ekspor default muncul. Ya, satu-satunya tujuan adalah memastikan kompatibilitas commonjs, sehingga kami dapat menulis import myModule from 'my-moduledan mendapatkan hasil yang persis sama. Namun, dari spesifikasi ini tidak jelas, dan selain itu, penerapan kompatibilitas adalah hak prerogatif dari pengembang transpiler. Dan di sini perpecahan besar baru saja terjadi: import React from 'react'atau import * as React from 'react'- itulah pertanyaannya.

Mengapa naskah memilih yang terakhir? Tempatkan diri Anda di tempat pengembang transpiler dan bertanya pada diri sendiri, apa cara termudah untuk impor transportasi dari es6ke commonjs? Katakanlah Anda memiliki set impor dan ekspor berikut:

export const foo = 1
export const bar = 2
export default () => {}
import { foo } from 'module'
import func from 'module'`

Jadi, kita akan menggunakan objek jsdengan kunci defaultuntuk ekspor secara default!

module.exports = {
  foo: 1,
  bar: 2,
  default: () => {}
}
const module = require('module')
const foo = module.foo
const func = module.default

Keren, tapi bagaimana dengan kompatibilitas? Jika impor secara default berarti kita mengambil bidang dengan nama default, maka ketika kita menulis import React from 'react'itu berarti const { default: React } = require('react'), tetapi itu tidak berhasil! Kemudian coba gunakan impor dengan tanda bintang. Sekarang pengguna harus menulis import * as React from 'react'untuk mendapatkan konten module.exports.

Namun, ada perbedaan semantik dari commonjs.
Commonjsseperti javascript biasa, tidak lebih. Hanya fungsi dan objek, tanpa memerlukan apa pun . Di sisi lain, dalam impor es6, requiresekarang bagian dari spesifikasi, oleh karena itumyModuledalam hal ini, ini bukan hanya objek javascript biasa, tetapi apa yang disebut namespace (jangan bingung dengan ruang nama dalam naskah), yang, karenanya, memiliki properti tertentu. Salah satunya adalah namespace tidak dapat dipanggil. Dan apa masalahnya di sini, Anda mungkin bertanya?

Mari kita coba pola lain commonjs, dengan satu fungsi sebagai ekspor:

module.exports = function() { // do something }

Kita dapat menggunakan requiredan menjalankannya:

const foo = require('my-module')
foo()

Meskipun jika Anda mencoba melakukan ini dalam lingkungan keluhan khusus dengan modul ES6, Anda akan mendapatkan kesalahan:

import * as foo from 'my-module'
foo() // Error

Itu karena namespace tidak sama dengan objek javascript, tetapi struktur terpisah yang menyimpan setiap es6 ekspor.

Tapi Babel sudah benar dan memberikan opsi kompatibilitas di mana kita bisa menulis import React from 'react'dan itu akan berhasil. Ketika melakukan transiling, ia menandai setiap modul ES6 dengan flag khusus module.exports, sehingga kami memahami bahwa jika flag itu benar, maka itu dikembalikan module.exports, dan jika itu salah (misalnya, jika itu adalah perpustakaan commonjsyang belum ditransformasikan), maka kita perlu membungkus ekspor saat ini { default: export }sehingga kita dapat digunakan setiap waktu default(lihat di sini ).

Mengetik naskah berhasil melalui impor dengan tanda bintang, tetapi akhirnya menyerah dan menambahkan opsiesModuleInteropke kompiler. Secara umum, opsi ini melakukan hal yang sama seperti babel, dan jika Anda mengaktifkannya, Anda dapat menulis impor reguler import React from 'react', dan naskah akan mengerti segalanya.

Masalahnya adalah bahwa meskipun diaktifkan secara default di proyek baru (saat runtime tsc --init), itu tidak akan berfungsi untuk proyek yang ada (bahkan jika Anda memutakhirkan ke TypeScript 3) karena tidak memiliki kompatibilitas ke belakang. Dengan demikian, Anda harus menulis ulang impor asterisk yang tidak perlu ke impor default. Bereaksi akan memperlakukan ini secara normal, karena masih merupakan kumpulan ekspor bernama, tetapi tidak misalnya dengan panggilan namespace. Tapi jangan takut jika semuanya sesuai dengan pengetikan ekspor (dan sebagian besar, semuanya sudah beres, karena banyak dari mereka secara otomatis diperbaiki), TypeScript 3 akan memungkinkan Anda untuk dengan cepat mengkonversi impor dengan tanda bintang ke standar.

Oleh karena itu, saya benar-benar menganjurkan penggunaan opsi esModuleInterop, jika hanya karena itu tidak hanya akan memungkinkan Anda untuk menulis lebih sedikit kode dan membuatnya lebih mudah dibaca (dan ini bukan hanya kata-kata, misalnya, rollup tidak akan memungkinkan Anda untuk menggunakan impor dengan tanda bintang seperti ini), tetapi juga mengurangi ketidaksetujuan antar komunitas naskah dan babel.

Perhatian: dulu ada opsi enableSyntheticDefaultImportsyang menutup kompiler ketika ia mencoba mengeluh tentang impor yang salah secara default, jadi kami membutuhkan cara kami sendiri untuk memastikan kompatibilitas dengan commonjs(misalnya,WebpackDefaultImportPlugin), tetapi itu bermasalah, karena, misalnya, jika Anda memiliki tes, maka Anda masih perlu memastikan kompatibilitas tersebut. Harap dicatat bahwa esModuleInteroptermasuk impor sintetis secara default hanya jika target Anda adalah <=ES5. Oleh karena itu, jika Anda mengaktifkan opsi ini, dan kompiler terus mengeluh tentang hal itu import React, maka pahami tujuan apa yang Anda kejar, dan mungkin penyertaan impor default akan menjadi pilihan Anda (atau restart vscode / webstorm, siapa tahu).

Saya harap penjelasan saya sedikit memperjelas situasinya, tetapi jika Anda masih memiliki pertanyaan, Anda dapat menanyakannya di twitter !
