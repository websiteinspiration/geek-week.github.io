リストで会った

  • public static final列挙で定義された値ごとに1つのフィールド
  • プライベート合成フィールド$VALUES、メソッドの実装の詳細values()
  • メソッドの実装values()とvalueOf()
  • プライベートコンストラクタ
  • 静的初期化のブロック。実際に最も興味深いことが起こります。より詳しく考えてみましょう。

Javaコードの形式では、後者は次のようになります。

    static  {
        Fizz = new FizzBuzz("Fizz", 0);
        Buzz = new FizzBuzz("Buzz", 1);
        FizzBuzz = new FizzBuzz("FizzBuzz", 2);

        $VALUES = new FizzBuzz[] {
            Fizz, 
            Buzz, 
            FizzBuzz
        };
    }

最初に、列挙要素のインスタンスが作成されます。作成されたインスタンスは、適切なpublic static finalフィールドにすぐに書き込まれます。

次に、配列が作成され、すべての列挙要素のインスタンスへのリンクで埋められます。リンクは、上の段落で初期化したクラスのフィールドから取得されます。入力された配列はprivate static finalフィールドに格納されます$VALUES。

その後、リストの準備は完了です。

JLS§8.9.3からすでに知っているように、同じ名前のフィールドが、結果のクラスの列挙要素ごとに作成されます。クラスのフィールド数は、16ビットの符号なしfields_countを定義します。これにより、1つのクラスファイルの65_535フィールドまたは65_534列挙要素に制限されます。1つのフィールドは$VALUES、メソッドが返すクローンの配列用に予約されていますvalues()。これは仕様で明示的に述べられていませんが、より洗練されたソリューションを考え出すことはほとんどありません。

フィールド、メソッド、クラス、定数値などの名前は、定数プールに格納されます。
定数プールの内部構造について何も知らない場合は、以下の古代の記事を読むことをお勧めします怠惰な. , , .
クラス定数のプールのサイズも65_535要素の数によって制限されます。正しく形成されたクラスの定数のプールが空になることはありません。少なくとも、このクラスの名前があります。

たとえば、デバッグ情報のないOpenJDK 14-ea + 29からjavacによってコンパイルされた空の列挙のクラスの定数のプールには、29のエントリが含まれています。

したがって、1つの列挙内の65_534要素の数も達成できません。最良のケースでは、65_505またはこれに近い数を数えることができます。

この長期にわたる概要の最後のコード：フィールドに

値を書き込むことstatic finalができるのは、静的初期化ブロック内だけであり、クラスファイルのレベルでは、名前の付いたメソッドによって表されます<clinit>。どのメソッドのバイトコードも、65_535バイトを超えることはできません。おなじみの数ですね。

静的フィールドに書き込むための1つの命令putstaticは3バイトを必要とします。これにより、の概算が得られます65_535 / 3 = 21_845。実際、この見積もりは誇張されています。命令は、スタックの最上部からフィールドに書き込む値を取得します。これは、前の命令の1つがそこに配置したものです。そして、この命令も貴重なバイトを消費します。ただし、これを考慮しなくても、結果の数値は65_505よりも大幅に少なくなります。

要約すれば：

  • クラスファイル形式は、列挙要素の最大数を約65_505に制限します
  • 静的な最終フィールド初期化メカニズムにより、さらに制限されます。理論的には-最大21_845要素、実際にはこの数はさらに少ない
