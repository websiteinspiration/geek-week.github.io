テストは初心者向けではありません

私はテストの大ファンです。私はブログとメーリングリストにそれについて書いて、自分の自由時間に他の開発者とそれについて話し合い、Goでのテストに関するトレーニングコースを作成することさえしました。

しかし、テストに対する私のすべての愛情にもかかわらず、私はそれを初心者にはお勧めしません。

ワイルドに聞こえますか？この記事では、私の見解をより詳細に説明しますが、全体の要点は、最終的には次の2つに分類されます。

  1. 初心者は、最も単純なテスト以外のものを書くのに十分な知識を持っていません。これは必然的に次のポイントにつながります...
  2. プログラミングを学びながら現実的なテストを書くために必要なスキルを訓練しようとすることは非常に困難です

これは原則として一つのポイントであることを理解しています。いずれにしても、わかりやすくするために2つに分けました。

多くの方が私に同意しないことは承知していますが、記事を読んでください。もし読んでも納得できない場合は、喜んで話し合います。最後に、私は勉強するためにここにいます

初心者は、最も簡単なテスト以外のものを書くための知識が不足しています


初心者がコードを書くときはいつでも、その主な目的は、タスクを分解したり、グローバル変数を避けたり、テストコードを書いたりすることではありません。正直なところ、ほとんどの初心者はおそらくそれが何であるかほとんどわかりません。実際、彼らの主な目標は単純です-このいまいましいことを機能させることです。すべて。

これを確認するのに労力が必要になることはほとんどありません。初心者が書いたコードを見てください。

Webアプリケーションに移動しますか？確かに、コードの至る所にSQLクエリが記述されており、データベースへの接続がグローバル変数に格納されることがほぼ保証されています。

Railsアプリケーション？ビューにはビジネスロジックが、コントローラーには大量のロジックジャムが表示されます。

PHP Webアプリケーション？すべてのロジックが1つのphpファイルに含まれていても驚くことはありません。

たとえば、フォームの解析、データベースとの相互作用などです。基本的な機能、たとえば機能が制限された計算機を使用しても、私が説明したような例に出くわすことになります。これは初心者が気にしないという事実によるものではなく、彼らはまだより良い方法を知らないだけです。

初心者は依存性注入が何であるかを知らず、グローバル変数がどのようにテストを複雑にするかを理解していません。ほとんどの場合、モックが何であるかを知らないので、コーディングが簡単なコードの設計方法を理解することを期待するのはかなり馬鹿げています。

結果として、初心者にとって、これらのような最も単純なテストだけが意味をなします：

func Add(a, b int) int {
  return a+b
}

// And a test...
func TestAdd(t *testing.T) {
  got := Add(2, 4)
  want := 6
  if got != want {
    t.Errorf("Add() = %d; want %d", got, want)
  }
}

これを初心者に見せて、テストとは何かについてのアイデアを与えることには何の問題もありませんが、このコードを見せて、実際のテストのようなものに見せかけるのは、とんでもないことだと思います。

結局、これは私たちが彼らにもっと教え始めるという事実につながります。依存関係の注入とは何か、なぜグローバル変数がtimeなどのテストを複雑にするのかを説明しようとしています。Now（）を使用すると境界のケースのチェックが困難になる可能性があります。そして、この場所で私は心配し始めます。なぜなら、私たちはもはや初心者にコードの書き方を教えないからです。このとき、コードの書き方とテスト方法を同時に教えます。そして、それが私を2番目のポイントに導きます...

プログラミングを学びながら現実的なテストを書くために必要なスキルを訓練しようとすることは非常に困難です


以前と同じように、初心者が作成したコードについて考えてもらいたいのですが、今回は、最初に作成したプログラムのいくつかを覚えておいてください。

あなたが私のようだったら、最初のWebアプリケーションは次のようになります。

<p>
  <?php
  // This may not work. I don't know PHP anymore.
  $name = $_GET['name'];
  echo "Hello, " . $name;
  ?>
</p>

芸術作品ですよね？

今、あなたがこのコードを初めて書いたばかりで、誰かがあなたがそれをテストすべきだと言ったと想像してください。そして、あなたはReactを使用する必要があります。そしてフレームワーク。ああ、それでデータベースを微調整したり、おそらくGraphQLを操作したりしても害はありません。

理由はわかりませんが、私たちの開発者は、長年の経験と実践で得たスキルを使用し、他の人、特に初心者に同じことを期待する習慣があります。これはとんでもないことです！これは、あなたがすでに三角法や代数などを学んでいて、特定の問題を解決するために数学を使用できるという理由だけで、彼が数学を理解することを誰かに期待することです。

何かがうまくいくとしても、初心者に向いているとは限りません。それらは、あなたが使用するものから利益を得るために必要な経験と実践がなければ、文脈から外れる可能性があります。あるいは、おそらく彼らが苦しんでいる問題は、実際にはこれらの複雑なソリューションすべてを使用するには単純すぎるものです。

HTTPリクエストがどのように機能するかを段階的に調査した方法を忘れたかのようです。ヘッダーの仕組み。クッキーフォーム。サーバーへのPOSTリクエストはどのように機能しますか、または一般的にEXTRA HTTPメソッドがあります。そして、これらすべてを古き良き試行錯誤の方法で学びました。

実際、テストのせいだとは思いません。本当の問題は、プログラミング、テスト、ウェブサイトの構築など、何百万ものことを同時に学ぶべきだと私たちは強く信じていることです。これがどのようにして起こったのか本当にわかりませんが、問題の一部は、これらすべてを研究するのに何がかかるのかを気にしないことです。初心者は「何を学ぶべきか」と尋ねます。そして、「テストを学び、反応し、graphqlを実行しますが、標準ライブラリのみを使用します...」ということを始めます。

いいえ、いいえ、もう一度いいえ。やめて



他の条件ではそのようなアプローチの不条理が明らかであるため、これはばかげています。サッカーの方法を誰かに教える場合は、荷造りやドリブルなどの基本から始めます。 「これがプロのやり方です」と言って始めるのではなく、ロナウドのビデオをオンにして、そのままにしてください。では、プログラミングの初心者に対して私たちは一体何をしているのでしょうか？

私たちは「まあ、もちろん、これらすべてを同時に学ぶことは不可能であることを彼らは理解しています」と言って自分自身を正当化しようとしますが、彼らは理解していません！初心者の開発者がこの罠に陥り、行き詰まっているためにがらくたのように感じるという事実によって、状況はさらに悪化します。彼らは開発者になるために必要なものを単に持っておらず、このレンガの壁にぶつからないとプログラミングを愛するので、あなたと私にとっては残念です。

そして、これは私が実際に言いたかったことに私を導きます-私たちの多くは、一度にいくつかのことを勉強することに焦点を合わせれば、よりよく勉強します。私たちは新しい挑戦をしたいのですが、新しいことに挑戦したいのですが、混乱して混乱に陥りたくありません。テストやその他のこと、たとえば、Webアプリケーションの作成方法、http、cookieなどの仕組みを学ぶことは、この罠に陥る簡単な方法です。そのため、通常は、残りを最初に調べ、最後にテストすることをお勧めします。あなたはいつでもあなたの古いプロジェクトに戻って、テストを勉強したとき、どのようにそれらを今再構築するかを見ることができますが、これはあなたが過負荷ではなく、動揺していない、そして結局このビジネスを放棄していない場合にのみ可能です。

しかし、テスト方法を学びたい場合はどうなりますか？（およびその他の100万人は「もしも」


クールフォワード！もちろん、最初にテストを学び、それから初めてWeb開発やその他のトピックを学ぶことができます。たぶんそれだけをした人もいるだろうし、あなたはこの方法をもっと好きかもしれない。テストは初心者向けではないと言うとき、これは勉強するのに悪いトピックだとは言いません。つまり、テストに加えて何かを学ぶことは間違いです。

ほとんどの人は最初にWebアプリケーションまたは何か視覚的なものを作成する方法を学びたいと思っていますが、これはあなたがこれから始める必要があることを意味するのではなく、テストから始めることができます。たぶん、理解するまで何も理解し始めないでしょうが、私があなたが何を求めているかを学ぶのをやめさせてはいけません。

ペアプログラミングなどで勉強できないわけでもありません。メンターを使用すると、同時に多くのことを学ぶことができ、この負荷にうまく耐えることができます。これは、これらすべてを案内する誰かがいるためです。この状況では、道に迷ったとしても行き止まりに陥ることはなく、失敗した気分にもなりません。「元気にやっているので、今のところ難しすぎます。次回は代わりにXとYを試してください！」簡単に言えば、このシナリオでは、混乱したり、失望したり、やめたりすることはありません。
