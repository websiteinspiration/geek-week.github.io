PostgreSQLアンチパターン：大量のJOINを含むヒット辞書

  • 珍しいレコードがJOINの真ん中に届く
  • シーシュポス結合配列
  • 有害なJOINおよびOR
  • CTE JOIN CTE

JOINがあまり好きでないとは思わないでください... :)

しかし、多くの場合、それがなければ、リクエストはそれを使用するよりもはるかに生産的になります。したがって、今日は、辞書を使用して、リソースを大量に消費するJOINを完全に取り除くようにします。



PostgreSQL 12以降、以下で説明する状況の一部は、デフォルトではCTEが非具体化されているため、少し異なる方法で再生できます。この動作は、キーを指定することで元に戻すことができますMATERIALIZED。

抽象の世界では、タスクの作成者は組織の全従業員に均等に分散する必要がありますが、実際には、タスクは、原則として、かなり限られた数の人から来ます。マーケティング ...）。

1000人の組織で、特定のアーティストごとにタスクを設定するのは20人（通常はさらに少ない）の著者だけであり、この主題の知識を使用して「伝統的な」要求を迅速化するとします。

hstore辞書


hstoreタイプ を使用して、キーと値の「辞書」を生成します。

CREATE EXTENSION hstore

著者のIDと名前を辞書に追加するだけで十分なので、後でこのキーを使用して抽出できます。

--   
WITH T AS (
  SELECT
    *
  FROM
    task
  WHERE
    owner_id = 777
  ORDER BY
    task_date DESC
  LIMIT 100
)
--     
, dict AS (
  SELECT
    hstore( -- hstore(keys::text[], values::text[])
      array_agg(id)::text[]
    , array_agg(name)::text[]
    )
  FROM
    person
  WHERE
    id = ANY(ARRAY(
      SELECT DISTINCT
        author_id
      FROM
        T
    ))
)
--    
SELECT
  *
, (TABLE dict) -> author_id::text -- hstore -> key
FROM
  T;


[explain.tensor.ruを見てください]

人に関する情報を取得するために費やされた時間は2分の1で、読み取られたデータは7分の1でした。「欺瞞」に加えて、これらの結果は、を使用して1回のパスでテーブルからレコードを大量に取得するのに役立ちました= ANY(ARRAY(...))。

テーブルエントリ：シリアル化と逆シリアル化


しかし、1つのテキストフィールドではなくレコード全体を辞書に保存する必要がある場合はどうでしょうか。この場合、PostgreSQL がテーブルを単一の値として書き込む機能を使用すると、次のようになります。

...
, dict AS (
  SELECT
    hstore(
      array_agg(id)::text[]
    , array_agg(p)::text[] --  #1
    )
  FROM
    person p
  WHERE
    ...
)
SELECT
  *
, (((TABLE dict) -> author_id::text)::person).* --  #2
FROM
  T;

ここで何が起こったか見てみましょう：

  1. personテーブルの完全なレコードのエイリアスとしてpを取り、それらから配列を組み立てました。
  2. (person[]::text[]), hstore- .
  3. .
  4. person ( ).
  5. «» (...).*.

json-


しかし、上記で適用したこのようなトリックは、「固定解除」を行うための対応するテーブルタイプがない場合は機能しません。まったく同じ状況が生じることになる、と直列化のためのデータソースとして、我々は、使用しようとCTEの行ではなく、「本当の」テーブルを。

この場合、jsonを操作するための関数が役立ちます。

...
, p AS ( --   CTE
  SELECT
    *
  FROM
    person
  WHERE
    ...
)
, dict AS (
  SELECT
    json_object( --    json
      array_agg(id)::text[]
    , array_agg(row_to_json(p))::text[] --   json   
    )
  FROM
    p
)
SELECT
  *
FROM
  T
, LATERAL(
    SELECT
      *
    FROM
      json_to_record(
        ((TABLE dict) ->> author_id::text)::json --     json
      ) AS j(name text, birth_date date) --    
  ) j;

ターゲット構造を説明するとき、ソース文字列のすべてのフィールドをリストすることはできませんが、本当に必要なフィールドのみをリストできることに注意してください。「ネイティブ」テーブルがある場合は、関数を使用することをお勧めしますjson_populate_record。

まだ辞書にアクセスできますが、json- [de]シリアライゼーションのコストは非常に高いため、この方法を使用するのは、「正直な」CTEスキャン自体が悪化した場合にのみ使用するのが妥当です。

テスト性能


したがって、データをディクショナリにシリアル化する方法は2つあります-hstore / json_objectです。さらに、キーと値の配列は、テキストへの内部または外部変換を使用して2つの方法で生成することもできます：array_agg（i :: text）/ array_agg（i）:: text []。

純粋に合成の例を使用して、さまざまなタイプのシリアル化の有効性を確認してみましょう。異なる数のキーをシリアル化します。

WITH dict AS (
  SELECT
    hstore(
      array_agg(i::text)
    , array_agg(i::text)
    )
  FROM
    generate_series(1, ...) i
)
TABLE dict;

評価スクリプト：シリアル化
WITH T AS (
  SELECT
    *
  , (
      SELECT
        regexp_replace(ea[array_length(ea, 1)], '^Execution Time: (\d+\.\d+) ms$', '\1')::real et
      FROM
        (
          SELECT
            array_agg(el) ea
          FROM
            dblink('port= ' || current_setting('port') || ' dbname=' || current_database(), $$
              explain analyze
              WITH dict AS (
                SELECT
                  hstore(
                    array_agg(i::text)
                  , array_agg(i::text)
                  )
                FROM
                  generate_series(1, $$ || (1 << v) || $$) i
              )
              TABLE dict
            $$) T(el text)
        ) T
    ) et
  FROM
    generate_series(0, 19) v
  ,   LATERAL generate_series(1, 7) i
  ORDER BY
    1, 2
)
SELECT
  v
, avg(et)::numeric(32,3)
FROM
  T
GROUP BY
  1
ORDER BY
  1;



PostgreSQL 11では、辞書サイズが最大で2 ^ 12キーであり、jsonでのシリアル化にかかる時間が短くなります。json_objectと「内部」型変換の組み合わせが最も効果的array_agg(i::text)です。

次に、各キーの値を8回読み取ってみましょう。辞書にアクセスしない場合、なぜ必要なのでしょうか。

  SELECT
    *

      SELECT

      FROM
        (

            array_agg(el) ea

              SELECT
                (TABLE dict) -> (i % ($$ || (1 << v) || $$) + 1)::text

SELECT
  v

  T
GROUP BY
  1
ORDER BY
  1;
