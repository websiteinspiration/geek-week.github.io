Unityの自動レイアウトとスタイル：UIの新しいパイプラインとツール



まず、便利なレイアウトツールを作成し、作業パイプラインを変更しました。順番に。

モバイルゲームには、HUDやメタ用の膨大な数の画面など、さまざまなインターフェイスがあります。UXデザイナーがそれらを設計し、UIデザイナーが描画します。これがすべてエンジンに表示されるようにするには、特別なトレーニングを受けた人々（テクニカルUIデザイナー）がいます。まあ、または単にレイアウトデザイナー。彼らの仕事の一部は、PSDレイアウトからプレハブにすべてを番号ごとに入念に転送することです。また、UIアニメーション、スプライトの入力、アダプティブレイアウトの作成、ローカリゼーションキーの配置なども行います。

そして、いくつかの目標を設定しました。

  • レイアウトデザイナーの作業を加速し、彼を日常から救う。
  • UI設計と開発の部門間の相互作用を最適化します。
  • UIを統一する：ガイドラインを作成し、それに基づいてUnityアセットを作成します。
  • UIの拡張性と多様性、アセットの再利用性、編集への抵抗、簡単な最適化を提供します。

これが起こった方法です。

UnityエンジンとAdobe Photoshopからある種のデザインシステムを作成しようとしました（ゲーム開発者を除いて、インターフェイスの作成にはほとんど使用されていないようです）。大まかに言えば、PhotoshopとUnityの友達を作るという課題に直面しました。



まず第一に、Unityで「ビジュアル言語」のセット全体を投影する必要がありました。これを行うために、私たちはスタイルを選択し、それらを私たち自身のニーズに合わせて調整しました。これは、企業ではかなり一般的な慣行です。したがって、フォントセットはフォントスタイルになり、レイアウトのセットはこれらのレイアウトのスプライトを持つスタイルになり、色のセットは色を持つスタイルになりました。これらすべてを一度作成して、ガイドラインを使用してpsd拡張と共に再利用および補足するだけで十分でした。したがって、UI設計者は、さまざまな画面で1001の基板を破棄し、いくつかのユニバーサルな基板を作成する必要があります。

現在のスタイルは次のとおりです。

  1. 色スタイル <名前、色>
  2. 文字スタイル <名前、フォント、フォントサイズ、フォント素材、行間>
  3. フォントスタイル <タイトル、フォント、フォント素材>
  4. ボタンのスタイル <名前、ボタン設定>
  5. レイアウトスタイル <名前、スプライト、インデント>
  6. スプライトスタイル <名前、スプライト>
  7. ローカリゼーションスタイル <名前、キー>
  8. アイコンのスタイル <名前、スプライト>

そして一般的には、ベースから継承して、スタイルを簡単かつ簡単に作成できます。例えば：

[CreateAssetMenu(menuName = "UIStyles/ColorStyle")]     
public class ColorStyle : BaseStyle<Color>    
{
}

唯一の制限：タイプはSerializableでなければなりません。タイプセッタでさえ、彼自身のタイプのスタイルの作成に対処します。



UIの状態を変更するには、いくつかのスタイルが必要です。たとえば、カラースタイル、テキストスタイル、スプライトスタイル、ボタンスタイル、ローカリゼーションスタイル。それらはすべて、さまざまなゲームオブジェクトに影響を与えます。写真は異なる色でペイントされ、異なるスプライトがそれらに置き換えられます。

これまでは、これをすべて使用する方法は明らかではありませんでした。簡単な例を示します。異なる色のオファーのボタンが1つあるとします。ボタンの色はオファー自体によって異なります。そして、それらは次のようになります。



次に、Imageを使用して1つのプレハブを作成し、その上にColorStyleComponentをぶら下げ、そこにColorStyleスタイルを挿入します。そこにボタンを再描画できるすべての色が格納され、コード内のオファーに応じてスタイルを適用します。



FontStyle、IconStyle、LayoutStyleの各スタイルは、自動ビルダーで使用されます。これは、プロジェクト内のすべてのフォント、アイコン、およびアンダーレイのライブラリです。

スタイルのあるシステムには、非常に多くの利点があります（プロジェクト内の一連のアセットのみが煩わしい場合があります）。

  • 再利用;
  • スタイルを変更してもプレハブは変更されませんが、何かを変更する必要がある場合、これはすべてのプレハブに影響します。
  • タイプセッターはスタイルを自分で作成し、変更して削除します。これにより、プログラマは花のあるモノベフのフィールドのヒープを忘れることができます。
  • エディターでは、スタイルを直接適用できます。エディターのBaseStyleComponent <T、TStyle>にはそのエディターが記述されているためです。

1つの大きなウィジェットの状態を変更するには、いくつかのスタイルを使用できます。さまざまなオブジェクトのオン/オフを切り替えることができます。これを行うために、プロジェクトでStateStyleComponentが作成され、オンとオフを切り替える必要のあるすべてのスタイルとオブジェクトが収集されます。次に、ウィジェット内で、可能なすべての状態が作成され、切り替えられます。これにより、色、フォント、テキスト、スプライトを切り替えるための同じコードの束からプログラマを節約できます。





次に、自動車メーカーを作成しました。これは、psdを貼り付けるツールであり、出力上に生のプレハブが作成されます。このスプライトは既に挿入されており、位置、名前、フォント、サイズ、エイリアスが配置されています。psdからのコピーはすべてレイアウトデザイナーによって慎重に処理されます。階層を大まかに固定して維持します。そして、それはほんの数秒でそれを行い、最終的に何時間もの骨の折れるコピーを節約します。



オートコンピュータウィンドウ自体は、設定が貧弱に見えます。使用中に、これらの設定が追加されます。



すべてのデータを取得するために、psdパーサーは、psd仕様に基づいてこれらのニーズのために特別に作成されました。これはかなり退屈なプロセスであり、既製のライブラリを使用できます。しかし、すべてを1つのボタンで実行したかったため、ゼロから作成する必要がありました。

自動車メーカーは、Photoshopで階層全体をコピーします。最初に、プレハブの名前でルートオブジェクトが作成され、画面の端にアンカーされます。その内部では、オブジェクトは上記の原則に従って作成されます。オブジェクトはルートオブジェクトで作成され、グローバル位置を維持しながら実際の親が割り当てられます。アンカーは、原則に従って配置されます-オブジェクトがどのエッジまたはセンターに最も近くに配置され、軸およびYに沿って個別に釘付けされます。すべての位置とサイズは整数になります。自動車メーカーは非表示のオブジェクトを無視します。

このツールの利点：高速レイアウト、階層の保存、必要なコンポーネントの追加、ゲームオブジェクトの名前の統一、名前によるスタイルの自動追加、不要な場所でのレーキキャストの無効化、おおよその貼り付け。

ただし、このアプローチには欠点があります。psdレイアウトが1つのボタンに対してのみ1000レイヤーで構成されている場合、それは実質的に役に立たないため、スマートオブジェクトにボタンを追加することをお勧めします。また、ツールにはUIデザイナーからの通常の正しい名前が必要です。レイヤーに「Rectangle_copy_21_1005」という名前を付けることはできなくなります。一方、これにより、他のUI設計者にとってレイアウトが読みやすくなります。

計画は、基板スプライトのエクスポートから節約するシステムを考え出すことです。

この記事のauto-makerポップアップを使用したレイアウトには、3分13秒かかりました（プロジェクトにスプライトをスローする場合を除く）。ペンは18分32秒かかります。同時に、ペンを使用したレイアウトにも問題があります。場所によっては、UIデザイナーの鋭い目が気づき、やり直さなければならない不正確な位置にあります。したがって、人的要因を排除するため、自動レイアウトも優れています。
