Quantas instruções do processador o compilador usa?

Há um mês, tentei contar quantas instruções diferentes são suportadas por processadores modernos e contei 945 em Ice Lake. Os comentaristas levantaram uma questão interessante : que parte de toda essa diversidade é realmente usada pelos compiladores? Por exemplo, um determinado Pepijn de Vos em 2016 calculou quantas instruções diferentes estão envolvidas nos binários em seu / usr / bin e contou 411 - ou seja, aproximadamente um terço de todas as instruções x86_64 existentes naquele momento não foram usadas em nenhum dos programas padrão em seu sistema operacional. Sua outra descoberta curiosa é que o código para x86_64 consiste em instruções para um terceiro mov. (Em geral, sabe-se que apenas as instruções são movsuficientes para escrever qualquer programa.)

Decidi desenvolver a pesquisa de Vos usando o compilador LLVM / Clang como o "código de referência". Possui várias vantagens sobre o conteúdo de / usr / bin de uma versão sem nome de um sistema operacional sem nome:

  1. É conveniente trabalhar com ele: este é um binário enorme, de tamanho comparável a todo o conteúdo do / usr / bin do Linux médio;
  2. Ele permite comparar diferentes ISAs: lançamentos oficiais para x86, ARM, SPARC, MIPS e PowerPC estão disponíveis em releases.llvm.org/download.html ;
  3. Permite rastrear tendências históricas: binários oficiais estão disponíveis para todos os lançamentos desde 2003;
  4. Finalmente, no estudo de compiladores, é lógico usar o compilador como um objeto experimental :-)

No último tópico, os comentaristas mencionaram que obtêm o código mais compacto para o SPARC. Aqui vemos que o binário do AArch64 é um terço menor em tamanho e no número total de instruções.

E aqui está a distribuição de acordo com o número de instruções:


Inesperadamente, o código de 11% para SPARC consiste em nops preenchendo slots de atraso de ramificação . Para o i386, entre as instruções mais frequentes, vemos int3preenchendo as lacunas entre as funções e nopusadas para alinhar loops às linhas de cache. A observação de De Vos de que o código é um terço mové confirmada nas duas variantes x86; mas mesmo nas arquiteturas de armazenamento de carga mov, se não for a instrução mais frequente, será a segunda após o carregamento.

E como o conjunto de instruções usadas mudou com o tempo?

O único ISA para o qual cada versão possui um binário oficial é o i386:


A linha cinza colocada no eixo direito é o número de instruções diferentes usadas no compilador. Como podemos ver, há algum tempo o compilador foi compilado muito mais diversificado. int3Foi usado para preencher as lacunas apenas em 2018; usado anteriormente o mesmo nopque para o alinhamento nas funções. Aqui você pode ver que o alinhamento dentro das funções é usado desde 2013; antes disso nophavia muito menos. Também é interessante que até 2016 eles movrepresentassem quase metade do compilador.

As primeiras versões do LLVM, antes do advento do clang, também foram lançadas com binários para o SPARC. Então o suporte ao SPARC perdeu sua relevância e, novamente, apareceu apenas após 14 anos - com uma ordem de magnitude aumentada no número de nops:


Historicamente, o próximo ISA para o qual os binários LLVM foram lançados foi o PowerPC: primeiro no Mac OS X e, depois de um hiato de dez anos, no RHEL. Ao contar as instruções do ARM, cortei os sufixos de condição - juntos com eles, foram obtidas cerca de mil instruções diferentes, mas mesmo sem elas, o ARM está muito à frente de outros ISAs na variedade de instruções geradas.
