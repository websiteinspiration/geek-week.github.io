Je vais interviewer des programmeurs Java. Un ensemble de questions pour tous les niveaux

Dans ma carrière, je devais être développeur et manager, et même une fois «CTO» dans une petite startup. Et, bien sûr, j'ai dû mener un grand nombre d'entretiens avec des cadidats. Par conséquent, pour moi, j'ai développé de telles règles pour la conduite des entretiens techniques:

  • poser à tous les candidats les mêmes questions
  • les questions doivent être telles que les juniors et les seniors y répondent, mais de différentes manières,
  • après trois questions, le niveau du candidat doit être compris.

Et donc j'ai eu ces trois questions pour les développeurs Java.

Parlez-nous de HashMap


Oui, oui, la question très standard avec laquelle 50% des entretiens commencent. Pourquoi je lui demande? Parce qu'on peut y répondre très, très différemment.

  • La réponse idéale commence par l'énoncé du problème. Pourquoi avons-nous besoin d'une telle structure? Quelle tâche spécifique résout-elle? À quoi la solution triviale à ce problème ressemble-t-elle «de front» et pourquoi est-elle mauvaise?
  • "Points bonus" pour une excursion historique, connaissance de l'auteur de la solution, au moins l'année approximative de son apparition (pas à Java, bien sûr)
  • La réponse d'au moins le junior doit inclure des façons d'utiliser la structure. Quelles méthodes soutient-elle?
  • Le développeur intermédiaire doit comprendre le périphérique interne, ainsi qu'une description des contrats des fonctions hashCode () et equals (), leurs exigences fonctionnelles
  • Points bonus pour décrire les exigences non fonctionnelles pour hashCode () et equals ()
  • Points bonus pour envisager d'utiliser HashMap dans un environnement multi-thread.
  • "Points bonus" pour considérer les alternatives HashMap - en les listant, en comparant la complexité de l'implémentation, l'utilisation, la vitesse en général, les cas idéaux et les pires.
  • "Points bonus" pour les alternatives HashMap sous forme de composants système séparés (par exemple, bases de données et caches "externes").
  • «Points bonus» pour décrire comment HashMap est utilisé dans les bases de données et pourquoi les index binaires basés sur des arbres sont généralement utilisés, plutôt que les index HASH.

Comme vous pouvez le voir, un très grand nombre "d'options bonus", dont vous pouvez parler avec senior'om. Et de combien de points pouvez-vous parler pendant au moins 10 minutes?

Si vous vous ennuyez, remplacez HashMap par TreeMap.

Ecrire SQL ...


Ouais, et encore la question "standard", qui était il y a 10 ans à chaque interview. Maintenant, c'est moins commun, apparemment en raison de la mode pour nosql. Mais la capacité de travailler avec des données est toujours nécessaire, et les programmeurs qui comprennent le produit cartésien sont toujours plus utiles (et plus chers) que ceux qui se limitent à la relecture générale du manuel.

Et la question est:
Soit deux tables dans la base de données: élèves et groupes. Chaque étudiant a un identifiant pour le groupe group_id auquel il appartient. Écrire une requête SQL sans utiliser d'imbriqué, répertoriant les groupes (ou le nombre de groupes) sans un seul étudiant

La bonne réponse contiendra, bien sûr, non seulement JOIN, mais LEFT JOIN en comparant la valeur à NULL. La chose la plus intéressante sera de discuter avec le candidat des erreurs qui peuvent être commises lors de la rédaction d'une demande, et pourquoi la demande ne fonctionnera pas si elle est faite. Pourquoi ne peut-on pas comparer avec NULL par égalité? Pourquoi un simple JOIN ne fonctionne-t-il pas?

Un bonus pour les index qui accélèreront le fonctionnement de cette requête et pour considérer les options SQL dans différents dialectes (différents SGBD).

Trouver des erreurs dans le code


Le candidat s'exprime comme suit:
Supposons que vous obteniez une demande d'extraction (demande de fusion) d'un autre développeur pour une révision de code. Cette requête a un grand nombre de fichiers différents, mais l'un d'eux ressemble à ceci (le code de classe entier est donné):

package ru.mycompany.utils;

/** Wrap around Lock to simplify interface */
class LockWrapper {
    private Lock lock;
    public wait() {
        this.lock.wait()
    }
}

Aurez-vous des commentaires sur ce code, et si oui, lesquels? Allez-vous ignorer la demande avec un tel fichier plus loin (à la branche principale), ou aurez-vous besoin d'un rétablissement, et pourquoi?

Bien sûr, il y a au moins trois niveaux ici.

  • Tout junior doit signaler les erreurs de syntaxe présentes dans ce code. Le développeur doit indiquer pourquoi le code ne se compile pas. Et le problème n'est pas seulement dans la syntaxe, bien sûr.
  • Le développeur doit savoir pourquoi même si le code compile, cela ne fonctionnera pas. Et comment corriger ce code pour qu'il fonctionne. Le développeur doit pouvoir utiliser à la fois la construction synchronize et les classes du package java.util.concurrent et savoir pourquoi une seconde n'est pas la troisième.
  • Un développeur idéal devrait commencer par demander pourquoi ce code est nécessaire, quel problème il résout et s'il peut le résoudre même s'il résout tous les problèmes existants dans ce code.

Dans le processus de réponse, avec le candidat, nous apportons le code à la forme «idéale». La forme finale dépend bien sûr des qualifications du candidat. Le meilleur de mes candidats a trouvé 13 erreurs ici. Mis à part «l'erreur la plus importante».

En prime, vous pouvez parler des moyens de contrôle automatique du code à utiliser pour trouver au moins une partie des erreurs présentes.

Au lieu d'une conclusion


Un conseil que je veux donner à tous ceux qui doivent interviewer de futurs collègues. N'essayez pas de répondre si une personne convient ou non. Formulez votre opinion soit sous la forme d'un grade (si votre entreprise l'a adopté), soit en montant que vous êtes prêt à payer au candidat pour travailler dans votre entreprise. Cela simplifie d'une part la décision pour vous, car elle ne sera pas en noir ou blanc, et d'autre part - à quelqu'un qui prendra la décision finale ou comparera les résultats de plusieurs entretiens.
