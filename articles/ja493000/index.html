今では、私たち自身の設計システムがどんどん増えているようです。これは素晴らしいことです。

設計システムには分離が含まれますが、GitLabはすでにBootstrapで作成されているため、設計システムの約50〜60％がBootstrap / Vueコンポーネントのラッパーであり、機能が低下しています。これは、設計システムでコンポーネントを誤って使用できないようにするために必要です。抽象ライブラリについて語る場合、たとえば、必要なボタンを作成する機能など、柔軟性が重要です。 GitLabのスピナーがデザイナーによって承認された4つのサイズである場合、物理的に他の人にそれをさせないようにする必要があります。

いつかは良いものが勝ち、私たちがもちろんIEとEdgeのサポートで得点した場合、フロントエンドプロジェクトを効果的にリファクタリングできる重要なツールを手に入れます-これはShadow DOMです。 Shadow DOMは、コンポーネントに流れ込むグローバルスタイルの問題を解決します。 Googleでさえすでに埋められているPolymerを服用しないでください。 lit-elementとlit-HTMLを使用すると、お気に入りのフレームワークを使用して分離されたスタイルを構築できます。

ReactにはCSSモジュールがあり、Vueには同じことを行うスコープ付きスタイルがあると言えます。 CSSモジュールはクラスでのみ機能するため、100％の分離は提供されません。また、Vueのスコープ付きスタイルを使用すると、トップコンポーネントのスタイルが親のルート要素に該当し、そこでデータ属性が使用されて速度が低下する場合、非常に興味深いシナリオが実現できます。

私はAngularを3年間叱ったにもかかわらず、今のところ、Angularが最適に実装されていることを認めざるを得ません。 Angularでは、スタイルの分離を確実に行うために、分離モードを切り替え、必要に応じてShadow DOMを使用するだけで十分です。それ以外の場合は通常のエミュレーションを使用します。

2.テスト


GitLabのテストはすべてのコードをカバーし、信頼性を提供します。したがって、パイプラインは98分で完了します。



GitLabはマージ要求ごとにパイプラインを収集するため、GitLab.comのパブリックランナーの時間の40％を収集します。

私は非常に刺激を受けました：私はついにすべてがテストでカバーされるプロジェクトに乗りました！バックエンドコードのカバレッジは100％に近く、到着時のフロントエンドコードは89.3％でカバーされていました。

残念ながら、次の理由により、この報道のほとんどがゴミであることが判明しました。

  • コンポーネントに関連しない変更が行われると故障します。
  • 変更しても壊れません。

例を挙げて説明します。Jestを採用したのは、特定の状況ではアサーションを記述せずにスナップショットを使用できるようになると考えていたためです。問題は、Jestを構成せずに正しいシリアライザーを追加しなかった場合、Vue Test Utilsは単にHTMLで小道具を出力することです。次に、たとえば、オブジェクトにオブジェクトが渡されたdataという名前のパラメーターにpropsがあったuserという名前のpropsがわかります。送信されたデータの形式の変更は、スナップショットの失敗につながりません。

Ruby開発者は、大まかに言って、すべてのメソッドをカバーするテストに慣れています。
VueまたはReactコンポーネントのテストを行う場合、パブリックAPIの動作をテストする必要があります。
したがって、一部のシナリオでは使用されなかった計算されたプロパティに対する巨大なテストがありましたが、他のシナリオでは、この計算が呼び出されたときに状態に到達することが物理的に不可能でした。テンプレートが文字列であるため、テンプレートのテストカバレッジを計算できないVueに特に感謝します。Vue 3では、ソースマップとそれを修正する機能が表示されますが、すぐには修正されません。

幸いにも、レガシーを効果的にリファクタリングできる簡単なスキルが1つあります。これは、大規模なテストの世界でピン留めテストと呼ばれるものを作成する機能です。

固定テスト


これは、リファクタリングしている動作をキャプチャしようとするテストです。固定テストがリポジトリにコミットされることはほとんどありません。つまり、ステージング環境を使用するなど、あらゆる種類の改良を通じて、コンポーネントのレンダリング方法を説明するテストを自分で作成します。リファクタリング後、ピン留めテストは同じHTMLを生成するはずであり、これはおそらく良い兆候であるか、発生した変更を理解する必要があります。

人生の例を挙げましょう。数か月前、私はドロップダウンリストをリファクタリングしてマージリクエストのレビューを行いました。従来のコンテキストはこれです。以前は、ドロップダウンリストのダッシュで友達の枝を互いに分離するために、テキスト文字列「divider」が単に渡されていました。したがって、ブランチがディバイダーと呼ばれていた場合、あなたは運が悪いです。リファクタリングの過程で、ある人がHTMLノード内の2つのクラスを入れ替え、これが本番環境に入り、それを台無しにしました。もちろん、公平に言えば、それほど生産的ではありませんが、ステージングではありますが、それでもなおです。

その結果、そのようなテストを書き始めたとき、クールなテストカバレッジインジケーターにもかかわらず、テストが誤って記述されていることがわかりました。なぜなら、最初に、カルマ、つまり古いもののテストがありました。次に、ほとんどすべてのテストで、コンポーネントの内部についての仮定が行われました。つまり、それらは単体テストのふりをして、本質的にエンドツーエンドのように機能し、特定のコンポーネントが特定の小道具でレンダリングされていることを確認する代わりに、特定のクラスの特定のタグがレンダリングされていることを確認します。違いを理解する：クラスはコンポーネントですか？

その結果、合計18〜9千行のリファクタリングテストを使用した18のマージリクエストでは、11千がカットされたため、変更ログの合計は約2万になります。



同時に、正式には、スピナークラスに関するアサーションを削除し、代わりに正しいプロップを持つスピナーがそこにレンダリングされていることを確認するために、これらすべてのテストを作り直しました。

一見、これはありがたい仕事です。しかし、適切なアーキテクチャのテストを書き直すことは、ビジネスに販売するのが非常に簡単でした。GitLabは商業的に有益な製品です。もちろん、20回のテストを書き換えるには3回の反復が必要であると製品マネージャーに伝えた場合、どこに送られるかを推測します。別のこと：「テストを書き直すには3回の反復が必要です。これにより、スピナーをより効率的に導入し、設計システムの新しい要素の将来の実装を加速することができます。」そしてここで私たちは重要になります。

3.抵抗


私のスピナーの多くがGitLab設計システムで待っている別の機能があります-これらは通常のSVGアイコンです。


メインプロジェクトで使用されているデザイナーによって描画されたアイコンがありますが、GitLabは子供時代が難しいため、デザインシステムにはありません。たとえば、2019では、CSSはWebpackではなくSprocketsと呼ばれる部分によって収集されます。これはパイプラインRubyです。これは、バックエンドとフロントエンドで同じCSSを再利用する必要があるためです。このため、アイコンはさまざまな方法でさまざまなプロジェクトに接続する必要があります。そのため、誰かがメインコードベースを3か月リファクタリングして、アイコンを設計システムから関連プロジェクトに接続できるようにしました。

ここで必然的に遭遇する重要なポイントがあります。リファクタリングは継続的な改善のプロセスです。しかし、遅かれ早かれ、あなたは止まらなければなりません。
リファクタリングを完了するのではなく、停止することは絶対に正常ですが、具体的な測定可能な改善を得ます。
しかし、レガシープロジェクトに取り組んでいる場合、必然的にこれを行う人々に出くわすことになります。

これは、慣れているため、古い方法で書くことを意味します。たとえば、バックベンダーは次のように言っています。 Karmaのテストを3年間書いてきたので、新しい機能を追加する必要があります。テストなしでは機能を利用できないため、ここにKarmaのテストがあります。」

あなたの仕事はこれにできるだけ抵抗することです。これは比較的簡単に戦うことができますが、それよりもさらに大きな罪があります。時々、リファクタリングの過程で問題に遭遇し、一般的に脇に置きたいという欲求があります。

つまり、特定の理由でリファクタリングを最後まで行うことができないという理由だけで新しい松葉杖を置き換えることです。たとえば、アイコンをメインコードベースに統合する際に問題が発生した場合は、グローバルアプリケーションCSSからプルされるユーティリティクラスを残すことができます。正式には、ビジネスタスクは解決されますが、実際には、Lernean hydraの歴史と同様に、8つのバグがあり、4つが修正され、13つが残っています。

家の修理のようなリファクタリング-終了することは不可能であり、停止することしかできません。
リファクタリングの最初の80％は20％の時間を要し、リファクタリングの残りの80％（同様）はさらに80％の時間を要します。
リファクタリングプロセス中に新しいハックを導入しないことが重要です。私を信じて、開発プロセス中に、それら自体が表示されます。

4.ツール


幸いにも、私が到着する前から、GitLabは、Prettier、Vue Test Utils、Jestなどの優れたツールを導入するための正しい道に乗り出しました。 Prettierは曲がって実装しましたが。

何が問題になっているのかを説明します。私は歴史的に何を、なぜそうしたのかを理解していましたが、私の検索の80％は37000行のかわいらしいコードのコミットに遭遇しました。履歴を使用することはほとんど不可能であり、変更の履歴を検索するときにこのコミットを除外するように、VS Codeのプラグインを構成する必要がありました。

もちろん、ツールは重要ですが、慎重に選択する必要があります。たとえば、Vueがあり、VueにはVue Test Utilsという優れたテストツールがあります。しかし、Vue 2が2〜3年前にリリースされた場合、Vue Test Utilsはまだベータ版から抜け出していません。さらに、インサイダー情報によると、現時点では、Vue Test Utilsの唯一の開発者がVueに書き込みを行っていません。

ツールを選択する過程で、あなたは運命でトスをプレイし、本当に勝とうとします。

GitLabは、CoffeeScriptで小児に怪我をしました。そのため、GitLabのTypeScriptで書くという理論的なアイデアすらプッシュすることは不可能です。すべてが1つの単純な引数に分解されます。JavaScriptにコンパイルされる言語が停止した場合、CoffeeScriptと同じではありませんか。
ツールを選択するときは、ツールを交換できるようにするか、極端な場合は独立して保守できるようにしてください。

5.抽象化


例から始めましょう。数か月前、私はニュースを見ました。「GitHubはjQueryを取り除きました。私たちは長い道のりを歩み、jQueryを使用しなくなりました。」当然、GitLabのjQueryも取り除く必要があると思いました。



クイック検索の結果、jQueryが300ファイルで使用されていることがわかりました。それは恐ろしいように見えますが、何もありません-目は恐れて、手はやっています。しかし、違います！Bootstrapがあるため、jQueryはGitLabコードベースに不可欠な接着剤です。

BootstrapはもともとjQueryで作成されました。これは、たとえば、Bootstrapでドロップダウン開始イベントをキャッチする必要がある場合、これはjQueryイベントであることを意味します。ネイティブで傍受することはできません。

これは、レガシーコードを操作するときに抽象化する必要がある最初のことです。捨てることができないjQueryがある場合は、jQueryイベントの内部作業を非表示にする独自のイベントエミッターを記述します。

明るい未来が来たら、jQueryを削除できますが、今のところ、govnokodに集中する必要があります。通常のレガシープロジェクトでは、コード全体に均等に分散されます。「化学防護服なしでは立ち入らない」というフラグの付いたボトルネックに収集します。

そのため、メトリックを持つことが非常に重要です。

ここで何が起こっているかをお話します。これはVueであり、その反応システムは値を追跡します。値が変更されると、この値に依存するすべての依存関係が呼び出されます。各行は、コメントを行に配置したり、コメントをサーバーから動的にロードしたりできるため、多くのもので構成されるVueコンポーネントです。これらはすべて、VueコンポーネントでもあるVueストアにサブスクライブされます。

マージ要求を閉じると、たとえばストアが更新されたときに、2万のストアサブスクリプションをすべて更新する必要があります。行が削除された場合は、依存関係から削除する必要があります。そして簡単な計算：削除する必要がある要素を見つけるには、2万要素の配列を調べる必要があります。そのような行が500あり、各行がいくつかのコンポーネントであるとしましょう。結果は数学演算O（N）、つまりO（20,000）* 500です。 JavaScriptはずっと実行されています。

依存関係の追加はアレイへのプッシュにすぎないため、展開は即座に行われます。数学演算O（1）。

コードの品質を改善すると、パフォーマンスやその他のメトリックスが低下する場合があります。それらを測定することは非常に重要です。
