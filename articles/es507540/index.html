Trabajar con eventos personalizados en htmlix.js

Este artículo cubrirá los principios básicos de trabajar con eventos de usuario en Htmlix y creará una pequeña aplicación con cuatro componentes: un formulario de entrada de nombre de usuario, un saludo, un botón de salida para una aplicación y una matriz con usuarios de la aplicación.



Puede hacer clic aquí para ver un ejemplo listo .



Puede leer la API htmlix aquí.



Después de ingresar el nombre de usuario en el formulario, la aplicación mostrará su nombre en el saludo, el botón de salida y creará un nuevo contenedor en la matriz con los usuarios, además de guardar los datos en localStorage. Después de volver a cargar la página, tomará el nombre de localStorage. Al hacer clic en el botón "salir del perfil" se eliminarán los datos de todos los componentes y la variable de localStorage.



Creemos cuatro componentes: el formulario de inicio de sesión - formulario, el saludo - saludo, el botón de cierre de sesión - cierre de sesión y la matriz con todos los visitantes - users_array, y veamos cómo crearíamos comunicación entre ellos si no utilizáramos eventos de usuario.

En el ejemplo anterior, registramos los datos de la propiedad "input" del componente de formulario en la variable "text". Ahora debemos mostrarlos en las propiedades de los componentes de saludo y cierre de sesión, y también crear un nuevo contenedor en la matriz users_array. Para comprender por qué se necesitan eventos personalizados, primero intentemos hacer esto de varias maneras sin usarlos.



1. forma: en el método de clic del formulario, vaya directamente a cada componente y establezca la propiedad de texto: 




        event.preventDefault();
        var text = this.parent.props.input.getProp();
                //  
                this.rootLink.state["greeting"].props.user_name.setProp(text);
                this.rootLink.state["logout"].props.user_name.setProp(text);
                  
                  ///  
                 this.rootLink.state["users_array"].add({user_name: text});


La desventaja de este enfoque es obvia, ¿qué pasa si obtenemos datos para esta propiedad en otro lugar? Por ejemplo, según el almacenamiento local, se cargarán automáticamente.



2. La forma es crear un método en el objeto stateMethods y llamarlo cuando los datos cambien: 




ststeMethods: {

         entryUser: function(text){

                this.state["greeting"].props.user_name.setProp(text);
                this.state["logout"].props.user_name.setProp(text);
                 this.state["users_array"].add({user_name: text});
       }
}

window.onload = function(){
             var name = window.localStorage.getItem('user_name');

             if(name)HM.stateMethods.entryUser.call.(HM, name); ///    = HM
}

// 
click: function(){
                  event.preventDefault();
		  var text = this.parent.props.input.getProp();
                  this.rootLink.stateMethods.entryUser.call.(this.rootLink, text); 
}


En este caso, logramos evitar la duplicación de código, pero el problema es que los métodos para trabajar con las propiedades del componente no están dentro de él, sino en funciones externas. ¿Qué pasa si antes de establecer una propiedad necesitamos formatearla de alguna manera, de manera diferente para cada componente, o si necesitamos establecer no una propiedad, sino varias, por lo tanto, es mejor cuando los componentes mismos trabajan con sus propiedades?



3. la forma es transferir toda la lógica de trabajar con propiedades en componentes, y crear métodos en ellos con los cuales invocar ciertas acciones. 



	
	greeting: {
		
		container: "greeting",

                //     
		props: [ "user_name", ["greet_user", "aux"] ],

                     }
	        }

	logout: {

               }

         entryUser: function(text){

                this.state["greeting"].methods.greet_user(text);
                this.state["logout"].methods.set_name(text);
                this.state["users_array"].methods.entry_user(text);
        }
   }
   //     entryUser


Ahora los componentes están encapsulados, sin embargo, accedemos al componente por su nombre. ¿Qué sucede si luego queremos cambiar el nombre del componente o método, o eliminar el componente por completo, o tendremos muchos de esos componentes que escuchan la misma propiedad? O tendremos muchas de esas propiedades, por lo que tendremos que crear muchos de estos métodos.



Para facilitar esta tarea, puede usar eventos personalizados. El componente se suscribe al evento y, cuando ocurre, hace algo con sus propiedades.



Editemos el código, ahora usando eventos personalizados: 



var StateMap = {

   eventEmiters: {	
             //   -  	
             ["emiter-entry-user"]: {prop: ""}, 		
   },
   form: {
	container: "form",
	props: ["input", "click"],
	methods: {
			
		click: function(){
		    event.preventDefault();
		    var text = this.parent.props.input.getProp();

		     //        
		    this.rootLink.eventProps["emiter-entry-user"].setEventProp(text); 	
		    window.localStorage.setItem('user_name', text);							
		}			
	},		
},	
greeting: {
		
	container: "greeting",
	props: [ "user_name", ['listen_entry_user', "emiter-entry-user", "" ] ], //   "emiter-entry-user"
	methods: {
			
		listen_entry_user: function(){
		   //      
		   this.parent.props.user_name.setProp( this.emiter.getEventProp() );
		},	
	},		
},
logout: { 
		
	container: "logout",
	props: [ "user_name", ["listen_entry_user", "emiter-entry-user", "" ]], //   "emiter-entry-user"
	methods: {
			
		listen_entry_user: function(){
		     //       
		     this.parent.props.user_name.setProp( this.emiter.getEventProp() );
		},			
	},	
},
    //      , ..            
users_array: { 
		
	arrayProps: [ ['listen_entry_user', "emiter-entry-user", ""] ], //   "emiter-entry-user"
	   arrayMethods: {
			
		listen_entry_user: function(){
			//       
			this.parent.add( {user_name: this.emiter.getEventProp()} );
		},			
	},		
	container: "user",
	props: [ "user_name",   ],
	methods: {						
		}
   },        	


}
window.onload = function(){
	
		
	var HM = new HTMLixState(StateMap);
	var name = window.localStorage.getItem('user_name');
	
	if(name != null)HM.eventProps["emiter-entry-user"].setEventProp(name); ///  "emiter-entry-user"        	
}


Por lo tanto, nos deshicimos del método intermedio "entryUser" y simplemente generamos el evento pasando nuevos datos en el formulario y después de cargar la página. Ahora, si necesitamos cambiar algo en el componente, o eliminarlo por completo, no necesitamos corregir el código en cada función, que puede ser cualquier número. Y para darse de baja de cualquier evento, simplemente puede eliminar el oyente en el componente, o deshabilitarlo temporalmente usando el método disableEvent () en la propiedad - suscriptor.
