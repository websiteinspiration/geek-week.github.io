スケジューラー


ここでの主な革新-それはプロファイル計画 （プロファイルのスケジュール）です。これは、クラスター内のワークロードがより異質になるほど、計画へのさまざまなアプローチの必要性がより早く発生するという事実と関連しています。

この問題を解決するために、著者は、スケジューラーがスケジューラーに割り当てられた異なる構成と呼び出されたプロファイルを使用することを提案しています。開始すると、kube-schedulerは使用可能なすべてのプロファイルをスキャンして、レジストリに保存します。構成にプロファイルがない場合は、デフォルトのオプション（default-scheduler）が選択されます。ポッドがキューに入れられた後、kube-schedulerは選択されたスケジューラーを考慮して計画を実行します。

述語に基づいサミ政策立案（PodFitsResources、PodMatchNodeSelector、PodToleratesNodeTaintsなど）と優先度（SelectorSpreadPriority、InterPodAffinityPriority、MostRequestedPriority、EvenPodsSpreadPriorityなど）。実装はすぐにプラグインのシステムを提供するため、すべてのプロファイルは特別なフレームワークを介して追加されます。

現在の構成構造（間もなく変更されます）：

type KubeSchedulerConfiguration struct {
   ...
   SchedulerName string
   AlgorithmSource SchedulerAlgorithmSource
   HardPodAffinitySymmetricWeight
   Plugins *Plugins
   PluginConfig []PluginConfig
   ...
}

...そして設定例：

profiles:
  - schedulerName: 'default-scheduler'
    pluginConfig:
      - name: 'InterPodAffinity'
      - args:
          hadPodAffinityWeight: <value>

HPAカスタムスケーリング速度


1年以上の間、HPAの構成可能なスケール速度と呼ばれる興味深い機能が、Kubernetesの機能強化炉で衰退してきました。カスタマイズ可能な水平ズーム速度。（ちなみに、私たちの同盟国が開発を開始しました。）新しいリリースでは、大量使用の最初の段階に移行しました-アルファ版で利用可能になりました。

ご存知のように、KubernetesのHorizo​​ntal Pod Autoscaler（HPA）は、サブリソースをサポートするリソースのポッド数をスケーリングしますscaleCPU消費またはその他のメトリックに基づきます。新しい機能を使用すると、このスケーリングが発生する速度を双方向に制御できます。これまでは、非常に限られた範囲で規制することが可能でした（たとえば、クラスターのパラメーターglobalを参照--horizontal-pod-autoscaler-downscale-stabilization-window）。

カスタムスケーリング速度を導入する主な動機は、ビジネスの重要性に応じてクラスターのワークロードを分離する機能であり、1つのアプリケーション（最も重要なトラフィックを処理する）のサイズを最大にして、ロールアップ速度を遅くすることができます（新しい負荷のバーストが発生する可能性があります）。他の人のために-他の基準に従ってスケーリングする（お金を節約するなど）。

提案されたソリューション-HPA構成のオブジェクトを追加behavior、両方向のスケーリングを制御するルールを定義できます（scaleUpおよびscaleDown）。たとえば、設定：

behavior:
  scaleUp:
    policies:
    - type: percent
      value: 900%

...現在実行中のポッドの数が900％増加するという事実につながります。つまり、アプリケーションが単一のポッドとして開始した場合、スケーリングが必要な場合は1→10→100→1000に成長し始めます。

他の例と実装の詳細については、KEPを参照してください。

貯蔵施設


アルファ版は、データが事前にロードされたボリュームを作成するための基盤であるGeneric Data Populators（KEP）を提供します。実装として、DataSource任意のタイプのオブジェクトをデータソースにできるように、フィールドの検証を弱めることが提案されています。

ボリュームをコンテナにバインドマウントする前に、そのすべてのファイルに対する権限が値に従って変更されますfsGroup。この操作は、一部のアプリケーション（たとえば、一般的なDBMS）の作業を中断させ、長時間かかる可能性があります（大容量の場合-1 TBを超える）。の新しいフィールドPermissionChangePolicyでは、ボリュームの内容の所有者を変更するかどうかPersistentVolumeClaimVolumeSource を決定できます。現在の実装はアルファ版であり、詳細はKEP。

オブジェクトのSecrets場合ConfigMaps 、 新しいフィールドがimmutable追加され、不変になりました。原則として、これらのオブジェクトはポッドでファイルとして使用されます。これらのファイルの変更は（約1分後に）すぐに、ファイルをマウントしたすべてのポッドに適用されます。したがって、シークレットまたはConfigMapの更新が失敗すると、アプリケーション全体が失敗する可能性があります。

この機能の作成者は、推奨される方法でアプリケーションを更新する場合でも-ローリングアップグレードを介して-既存のシークレット/ ConfigMapの更新の失敗が原因で障害が発生する可能性があると述べています。さらに、実行中のポッドでこれらのオブジェクトを更新するプロセスは、パフォーマンスとスケーラビリティの点で複雑です（各kubeletは、各固有のシークレット/ CMを常に監視する必要があります）。

現在の実装では、リソースが変更不可としてマークされた後、この変更をロールバックできないようになっています。オブジェクトを削除して再度作成するだけでなく、リモートシークレットを使用するポッドを再作成する必要があります。バージョン-アルファ、詳細-KEP。
