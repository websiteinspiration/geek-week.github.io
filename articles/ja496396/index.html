Alfa Digitalでのアクセシビリティテストの実施方法

デジタル製品は、美しく、便利で、収益性が高いだけでなく、障害を持つ人々にとっても利用しやすいものでなければなりません。これは、一見したところよりも重要です。時々、これをビジネス、プロダクトオーナー、および直接の同僚に伝える必要があります。しかし、それからあなたの製品を新しいレベルに引き上げることが判明しました。私たちはシニアiOS開発者と一緒にいますファミルコ私たちの経験について教えてください。

Alfa-Bankは、視覚障害を持つグループの金融商品の質を向上させることを目的として中央銀行が作成した特別なワーキンググループの一部です。作業と評価の特別な方法論はすでに開発されており、すべての銀行がこれを遵守しています。

そして、これが私たちがテストした方法です。



ユーザーとシナリオ


まず、（さまざまなグループの）視覚障害を持つ人々にテストを依頼しました。

視覚障害の最初のグループは、特別なデバイスとソフトウェア（スクリーンリーダー、VoiceOver）の使用を余儀なくされている完全に盲目の人々です。

2番目と3番目のグループは視覚障害があり、疾患に応じて、そのようなデバイスを使用するか、デバイスなしで正常に管理します。

はい。アプリケーション全体または個々の機能の完了または処理でさえあるテストを実施する場合、ビジネスにそのニーズを伝えることに支障はないことに注意することが重要です。コスト、時間、労力もかかるからです。ここでは、一方では視覚障害を持つ人々のためのそのような行動が会社の社会的責任であるというようなストーリーが判明し、他方では統計が示すように、国の人口の最大30％がさまざまな状況で一時的に視力の問題を経験する可能性があり、これはすでに印象的な姿。

したがって、Alpha Mobileを使用するための最も人気のある4つのシナリオを一緒にテストしました。どうぞ：

  • アプリケーションの承認
  • バランスチェック
  • アカウントを表示する（履歴、ステータス）
  • モバイルアカウントの再充電

もちろん、特定の人によってシナリオは異なります。QRコードを使用して住宅や共同サービスの料金を支払う人が多く、誰かが親戚に送金しますが、最も一般的なのはこれら4つです。

テスト方法とツール


特別なGOST R 52872-2012の「視覚障害者のためのアクセシビリティ要件」があり、すべての標準が十分に詳しく説明されています。これを使用して、見つかった各エラーに対応するラベルを割り当てます。合計で、見つかったすべての問題は3つのカテゴリに分類されました。

設計上の問題。たとえば、モバイルバンクの入り口では、これは全員がピンコードを入力するための通常のウィンドウであり、入力フィールドにフォーカスがなく、ピンを入力する必要がある試行回数を大声で言わない。


私たちもそうでした。それはまあまあの問題のように聞こえますが、これは重要なことです。ピンを正しく入力したかどうか、残りの試行回数を聞くことができない場合は、この試行回数を超える可能性があります。これは、アルファモバイルへの入り口が一時的にブロックされ、その結果生じるすべての不便さを意味します。

コード品質の問題。これは、正しく発音できない要素がある場合です。たとえば、一部の場所では、ナビゲーションの矢印を「テーブルの終わり」などのシステム要素として発声できます。


コントラストの問題。たとえば、ここでは、通常の視力でも、テキストを読むのは困難です。あなたはこれを取り除き、直ちにこれを考慮に入れなければなりません。

テストは、製品を作成するためのアップルのアプローチを本当に助けます。まず、すべてをデバイス上で直接テストすることは本当に便利です。クペルティニアンはすべてをクールに適応させました。

次に、アクセシビリティインスペクター付きのXcodeがあります。このユーティリティは、画面にカーソルを合わせるとボタンと要素の特定のビューを表示します。すべてをすばやく読み、正しく発声されるかどうかを理解できます。実際、私たちの場合、これが主な問題でした-VoiceOverの署名ボタン。


視覚障害のある人が使用するモバイルアプリケーションの機能と利便性を評価することにより、欠陥を発見しました。評価は、すべての基本的なユーザーシナリオの通過をテストすることによって実行されます。

クライアントが利用できる基本的なユーザーシナリオが多く、スクリプトの通過中に特定された障壁や困難が少ないほど、評価は高くなります。

スクリプトの可用性のレベルは、最も重大な問題によって決定されます。

  • criticalは、タスクをまったく完了できない重大な可用性の問題が検出されたシナリオを受け取ります。
  • 未成年者は、ユーザーがタスクの完了に大きな困難を経験したときに、中程度の重大度のアクセシビリティの問題が検出されるシナリオを受け取ります。
  • lowは、ユーザーがタスクの完了に何らかの困難を経験したときに、重要度の低い可用性の問題が検出されるシナリオを受け取ります。

アクセシビリティの欠陥を回避するには？


まず、UI Accessibility Elementプロトコルを使用します。

次に、Voice Over（視覚障害のあるユーザーが音声コマンドとキーボードを操作するのに役立つ特別なMakoshi機能）を改善する必要があります。

  • 署名ボタン。
  • 値を追加します。
  • ヒントを残してください。
  • グループコントロール。
  • 間違った碑文を修正してください。
  • コントロールのタイプを示します：ボタン、碑文、リンクなど。
  • ボタンまたはアイテムが狭すぎる場合は拡張する（最小44：44）

その他の推奨事項は次のとおり

です。1.ボタン-.accessibilityLabel

各ボタンには、短くて共鳴的な名前を付ける必要があります。VoiceOverはヘッジし、忘れた場合はボタンのテキストまたはアイコンの名前を読み込もうとします。

署名する必要があるもの：

  • アイコン付きでテキストなしのボタン。
  • 画像。可能であれば、写真に示されているものに署名することをお勧めします。
  • アセットのように、ラベルのないボタンと画像はアイコンの名前を示します



2.値--.accessibilityValue

名前に加えて、値を書き込むことができます。たとえば、textFieldで金額を入力するときは、アカウントの名前またはデジタルエンディングに署名し、ルーブルの数も示す必要があります。


3.ヒント-.accessibilityHint

アクションをさらに明確にしたい場合は、.accessibilityHintにヒントを記述できます。ただし、プロンプトに大きく依存するべきではありません。常に説明するのは面倒なので、一部のユーザーは電話の設定でオフにしています。

ボタンは「別の銀行へ」と発声されます。説明のために、ヒント、転送のタイプ、速度などを残すことができます。


4.グループコントロール-.accessibility

デフォルトでは、各要素は個別に読み上げられます。これは不便です。圧力ゾーンが減少し、気づかない場合があるため、一般化する必要があります。

これでセルにはいくつかのフィールドがあります：カード、お金、名前、セルごとに3つのコントロール。1つのセルと名前があったことを一般化する必要があるので、意味により近いことがわかります。


それを修正するには？

  1. セル全体をアクセス可能なコントロールにします。デフォルトでは、すべてのビューは他の要素のコンテナのみであり、VoiceOverはそれらを無視します。ビューを最終要素としてマークするには、セルisAccessibilityElement = trueを設定する必要があります。
  2. セルに名前を付けます。ラベルに焦点を合わせることができなくなったため、テキストを手動で指定する必要があります。accessibilityLabel = specialOffer.title

簡略化できます：

  1. セル全体をアクセス可能なコントロールにします。セルを設定しますisAccessibilityElement = true
  2. accessibilityLabelで、最も重要なことを書きます。カードとアカウントの名前です。カンマで区切られたVoiceOverは句読点を考慮します。
  3. accessibilityValueで追加情報を指定します。この場合、これはアカウント、金額です。
  4. セルを押すことができることを示します。これは基本的にボタンです。accessibilityTraits = .button
