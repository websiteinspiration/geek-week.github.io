インフラストラクチャをコードとしてPulumiでテストします。パート2

みなさん、こんにちは。今日は、記事「Plumiを使用したコードとしてのインフラストラクチャのテスト」の最後の部分を共有します。この翻訳は、コース「DevOpsの実践とツール」の学生のために特別に準備されたものです。



導入テスト


テスト済みのテストスタイルは強力なアプローチであり、インフラストラクチャコードの内部をチェックするためのホワイトボックスをテストできます。ただし、確認できる内容が多少制限されます。テストは、直接デプロイの前にPulumiによって作成されたインメモリデプロイメントプランに基づいて実行されるため、デプロイメント自体をテストすることはできません。このような場合のために、Pulumiには統合テストフレームワークがあります。そして、これらの2つのアプローチは一緒にうまく機能します！

Pulumi統合テストフレームワークはGoで記述されており、内部コードの大部分をテストするのに役立ちます。前述の単体テストのアプローチがホワイトボックステストに近い場合、統合テストはブラックボックスです。 （内部テストを徹底するためのオプションもあります。）このフレームワークは、完全なPulumiプログラムを取得し、新しいスタックを最初からデプロイする、バリエーションで更新する、削除するなど、さまざまなライフサイクル操作を実行するために作成されました。定期的に（たとえば、夜間）ストレステストとして実行します。

（私たちは取り組んでいますそのため、同様の統合テスト機能がネイティブ言語SDKにあります。Pulumiプログラムの記述言語に関係なく、Go統合テストフレームワークを使用できます。

このフレームワークを使用してプログラムを実行すると、次のことを確認できます。

  • プロジェクトコードは構文的に正しく、エラーなしで機能します。
  • スタックとシークレットの構成設定が機能し、正しく解釈されます。
  • 選択したクラウドプロバイダーにプロジェクトを正常にデプロイできます。
  • プロジェクトを初期状態から他のNつの状態に正常にアップグレードできます。
  • プロジェクトは正常に破棄され、クラウドプロバイダーから削除されます。

すぐにわかるように、このフレームワークはランタイム検証を実行するためにも使用できます。

簡単な統合テスト


この動作を確認するために、pulumi/examples私たちのチームとPulumiコミュニティがリポジトリを使用して、独自のリクエスト、コミット、およびナイトリービルドのプールをテストするときに、リポジトリを確認します。

以下は、S3バケットと他のいくつかのオブジェクトのプロビジョニングを行う、私たちの例の簡単なテストです。

example_test.go：


package test
 
import (
    "os"
    "path"
    "testing"
 
    "github.com/pulumi/pulumi/pkg/testing/integration"
)
 
func TestExamples(t *testing.T) {
    awsRegion := os.Getenv("AWS_REGION")
    if awsRegion == "" {
        awsRegion = "us-west-1"
    }
    cwd, _ := os.Getwd()
    integration.ProgramTest(t, &integration.ProgramTestOptions{
        Quick:       true,
        SkipRefresh: true,
        Dir:         path.Join(cwd, "..", "..", "aws-js-s3-folder"),
        Config: map[string]string{
            "aws:region": awsRegion,
        },
    })
}

このテストでは、フォルダーのスタックを作成、変更、破棄する基本的なライフサイクルを実行しますaws-js-s3-folder。合格したテストを報告するのに約1分かかります：

$ go test .
PASS
ok      ... 43.993s

これらのテストの動作をカスタマイズするための多くのオプションがあります。オプションの完全なリストについては、構造を 参照してくださいProgramTestOptions。たとえば、Jaegerエンドポイントをトレース用に構成（Tracing）し、ネガティブテスト中にテストが失敗すると予想することを示し（ExpectFailure）、一連の「編集」をプログラムに適用して連続的な状態遷移（EditDirs）などを行うことができます。それらを使用してアプリケーションのデプロイメントを検証する方法を見てみましょう。

リソースプロパティの確認


上記の統合により、プログラムが確実に「機能する」-クラッシュしない。しかし、結果のスタックのプロパティを確認したい場合はどうでしょうか？たとえば、特定のタイプのリソースが準備された（または準備されなかった）こと、およびそれらには特定の属性があること。

のパラメーターExtraRuntimeValidationをProgramTestOptions使用すると、デプロイ後にPulumiによって記録された状態（デプロイ後の状態）を確認できるため、追加のチェックを行うことができます。これには、構成、エクスポートされた出力値、すべてのリソースとそのプロパティ値、およびリソース間のすべての依存関係を含む、結果のスタックの状態の完全なスナップショットが含まれます。

この基本的な例を確認するために、プログラムが1つのS3バケットを作成することを確認してみましょう。

  integration.ProgramTest(t, &integration.ProgramTestOptions{
        // as before...
        ExtraRuntimeValidation: func(t *testing.T, stack integration.RuntimeValidationStackInfo) {
            var foundBuckets int
            for _, res := range stack.Deployment.Resources {
                if res.Type == "aws:s3/bucket:Bucket" {
                    foundBuckets++
                }
            }
            assert.Equal(t, 1, foundBuckets, "Expected to find a single AWS S3 Bucket")
        },
    })

これで、go testを実行すると、一連のライフサイクルテストが実行されるだけでなく、スタックが正常にデプロイされた後、結果の状態の追加チェックが実行されます。

ランタイムテスト


これまでのところ、すべてのテストは、デプロイメント動作とPulumiリソースモデルについてのみ行われてきました。準備したインフラストラクチャが実際に機能していることを確認したい場合はどうでしょうか？たとえば、仮想マシンが実行されている場合、S3バケットには予想されるものが含まれます。

あなたはすでにこれを行う方法を推測しているかもしれません：のオプションはこれのExtraRuntimeValidationためProgramTestOptionsの素晴らしい機会です。この時点で、プログラムのリソースの完全な状態にアクセスできる任意のGoテストを実行します。この状態には、仮想マシンのIPアドレス、URL、受信したクラウドアプリケーションやインフラストラクチャとの実際のやり取りに必要なすべての情報が含まれます。

たとえば、テストプログラムは次のwebEndpoint名前のバケットプロパティをエクスポートしますwebsiteUrlこれは、カスタマイズできる完全なURL index documentです。ステータスファイルを調べbucketてこのプロパティを直接検索して読み取ることもできますが、多くの場合、スタックは次のような便利なプロパティをエクスポートするので、確認に便利です。

integration.ProgramTest(t, &integration.ProgramTestOptions{
            // as before ...
        ExtraRuntimeValidation: func(t *testing.T, stack integration.RuntimeValidationStackInfo) {
            url := "http://" + stack.Outputs["websiteUrl"].(string)
            resp, err := http.Get(url)
            if !assert.NoError(t, err) {
                return
            }
            if !assert.Equal(t, 200, resp.StatusCode) {
                return
            }
            defer resp.Body.Close()
            body, err := ioutil.ReadAll(resp.Body)
            if !assert.NoError(t, err) {
                return
            }
            assert.Contains(t, string(body), "Hello, Pulumi!")
        },
    })

以前の実行時チェックと同様に、このチェックはスタックを上げた直後に実行され、これはすべて単純な呼び出しに応答して行われますgo test。そして、これは氷山の一角にすぎません。コードで記述できるすべてのGoテスト機能が利用できます。

継続的なインフラストラクチャ統合


コードレビューに送信する前にそれらをテストするためにインフラストラクチャに多くの変更が加えられた場合、ラップトップでテストを実行できることは良いことです。しかし、私たちと多くのクライアントは、開発ライフサイクルのさまざまな段階でインフラストラクチャをテストしています。

  • 各オープンプールで、合併前のテストのリクエスト。
  • コミットごとに、マージが正しく実行されたことを再確認します。
  • 定期的に、たとえば、追加のテストのために夜間または毎週。
  • パフォーマンステストまたはストレステストの一部として、通常は長期間にわたって実行され、テストを並行して実行するか、同じプログラムを数回展開します。

それぞれについて、Pulumiはお気に入りの継続的インテグレーションシステムとの統合をサポートします。継続的な統合により、アプリケーションソフトウェアの場合と同じようにインフラストラクチャのテストカバレッジを提供します。

Pulumiは一般的なCIシステムをサポートしています。それらの一部を次に示します。

  • AWSコードサービス
  • Azure DevOps
  • CircleCI
  • Githubアクション
  • Gitlab CI
  • Googleクラウドビルド
  • トラヴィス

詳細については、継続的デリバリーのドキュメントを参照してください。

一時的な環境


非常に強力な機能は、受け入れテストのみを目的として一時的な環境をデプロイできることです。Pulumiのプロジェクトとスタックの概念は簡単にすべてのいくつかの簡単なCLIコマンドまたはフレームワークをテストする統合により、完全に分離し、独立した環境を展開し、破壊するように設計されています。

GitHubを使用している場合、PulumiはGitHubアプリを提供します。これは、受け入れテストをCIパイプライン内のリクエストのプールに接続するのに役立ちます。アプリケーションをGitHubリポジトリにインストールするだけで、Pulumiがインフラストラクチャのプレビュー、更新、テスト結果に関する情報をCIとリクエストのプールに追加します。



基本的な受け入れテストにPulumiを使用すると、チームのパフォーマンスを向上させ、変更の品質に自信を与える新しい自動化機能が得られます。

合計


この記事では、汎用プログラミング言語を使用すると、アプリケーションの開発に役立つ多くのソフトウェア開発方法が利用できるようになることを確認しました。これには、単体テスト、統合テスト、および広範なランタイムテストを実施するための相互作用が含まれます。テストは、オンデマンドまたはCIシステムで簡単に実行できます。
