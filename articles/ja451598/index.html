ユニットフェアリーフェアリーフェアリー：DSL in C＃

実行中の単体テストを作成するときに、そのコードを確認する頻度はどれくらいですか。そして、あなたはそのように考えます：「これはテストです、私はそのようにそれを残します...」いいえ、％username％なので、そのままにしないでください。テストは、コードサポートを提供するシステムの重要な部分であり、この部分もサポートされることが非常に重要です。残念ながら、これを保証する方法は多くありません（テスト用のテストは作成しません）が、まだいくつかあります。


Dodo DevSchool開発者スクールでは、特に、優れたテストのために以下の基準を強調しています。

  • 再現性：同じコードと入力でテストを実行すると、常に同じ結果になります。
  • フォーカス：テストが落ちる理由は1つだけです。
  • 理解可能性：ええと、ここでは明らかです。:)

これらの基準に関して、このようなテストをどのように気に入っていますか？

[Fact]
public void AcceptOrder_Successful()
{
  var ingredient1 = new Ingredient("Ingredient1");
  var ingredient2 = new Ingredient("Ingredient2");
  var ingredient3 = new Ingredient("Ingredient3");
 
  var order = new Order(DateTime.Now);
 
  var product1 = new Product("Pizza1");
  product1.AddIngredient(ingredient1);
  product1.AddIngredient(ingredient2);
 
  var orderLine1 = new OrderLine(product1, 1, 500);
  order.AddLine(orderLine1);

  var product2 = new Product("Pizza2");
  product2.AddIngredient(ingredient1);
  product2.AddIngredient(ingredient3);
 
  var orderLine2 = new OrderLine(product2, 1, 650);
  order.AddLine(orderLine2);

  var orderRepositoryMock = new Mock<IOrderRepository>();
  var ingredientsRepositoryMock = new Mock<IIngredientRepository>();

  var service = new PizzeriaService(orderRepositoryMock.Object, ingredientsRepositoryMock.Object);
 
  service.AcceptOrder(order);
 
  orderRepositoryMock.Verify(r => r.Add(order), Times.Once);
  ingredientsRepositoryMock.Verify(r => r.ReserveIngredients(order), Times.Once);

私にとって-非常に悪い。

理解しにくいです。たとえば、Arrange、Act、Assertブロックを割り当てることさえできません。

再生不可：DateTime.Nowプロパティが使用されています。そして最後に、それは焦点が合っていないので、秋には2つの理由があります。2つのリポジトリのメソッドの呼び出しがチェックされます。

さらに、テストの名前はこの記事の範囲を超えていますが、私は名前に注意を払います。このような一連の否定的なプロパティでは、テストの名前を見ると、外部の人がこのテストがプロジェクトに一般的に含まれている理由をすぐに理解できるように公式化することは困難です。
テストに簡潔な名前を付けることができない場合は、テストに問題があります。
テストは理解できないので、テストで何が起こるかを説明しましょう。

  1. 材料が作成されます。
  2. 食材から製品（ピザ）を作ります。
  3. 製品から注文が作成されます。
  4. リポジトリが濡れているサービスが作成されます。
  5. 注文はサービスのAcceptOrderメソッドに渡されます。
  6. AddおよびReserveIngredientsメソッドがそれぞれのリポジトリで呼び出されていることを確認します。

それでは、このテストをどのように改善するのでしょうか。本当に重要なことだけをテストの本文に残しておく必要があります。そのために、Martin FowlerやRebecca Parsonsなどの賢い人々がDSL（ドメイン固有言語）を考案しました。ここでは、Dodoで使用するDSLパターンについて説明します。これは、ユニットテストが柔らかくて滑らかで、開発者が毎日自信を持っていることを確認するために使用します。

計画は次のとおりです。最初にこのテストをわかりやすくし、次に再現性に取り組み、最終的には焦点を絞ることにします。私たちは運転した ...

成分の廃棄（定義済みドメインオブジェクト）


注文作成ブロックから始めましょう。順序付けは、中心的なドメインエンティティの1つです。コードの書き方を知らなくても、ドメインロジックを理解している人でも、どのような順序で作成されているのかを理解できるような方法で順序を説明できればすばらしいと思います。これを行うには、まず、抽象的な「Ingredient1」と「Pizza1」を使用せずに、それらを実際の材料、ピザ、その他のドメインオブジェクトに置き換える必要があります。

最適化の最初の候補は成分です。それらはすべて簡単です。カスタマイズする必要はなく、コンストラクタを呼び出すだけです。それらを別のコンテナに入れて名前を付ければ、ドメインの専門家にわかりやすくなります。

public static class Ingredients
{
  public static readonly Ingredient Dough = new Ingredient("Dough");
  public static readonly Ingredient Pepperoni = new Ingredient("Pepperoni");
  public static readonly Ingredient Mozzarella = new Ingredient("Mozzarella");
}

完全に狂ったIngredient1、Ingredient2、Ingredient3の代わりに、Dough、Pepperoni、Mozzarellaを入手しました。
一般的に使用されるドメインエンティティには、定義済みのドメインオブジェクトを使用します。

製品のビルダー


次のドメインエンティティは製品です。それらはすべて少し複雑です。各製品はいくつかの成分で構成されており、使用する前にそれらを製品に追加する必要があります。

ここで、古き良きビルダーパターンが役に立ちます。これが製品のビルドバージョンです。

public class ProductBuilder
{
  private Product _product;

  public ProductBuilder(string name)
  {
     _product = new Product(name);
  }

  public ProductBuilder Containing(Ingredient ingredient)
  {
     _product.AddIngredient(ingredient);
     return this;
  }

  public Product Please()
  {
     return _product;
  }
}

これは、パラメーター化されたコンストラクター、カスタマイズメソッド、Containing およびターミナルメソッドで構成されますPlease。あなたがコードで親切であることを好きではない場合は、それを置き換えることができますPleaseしNow。ビルダーは、オブジェクトを構成する複雑なコンストラクターとメソッド呼び出しを非表示にします。コードがよりクリーンでわかりやすくなります。良い方法では、ビルダーはオブジェクトの作成を単純化して、ドメインの専門家がコードを明確に理解できるようにする必要があります。特に、作業を開始する前に構成が必要なオブジェクトには、ビルダーを使用する価値があります。

製品ビルダーを使用すると、次のようなデザインを作成できます。

var pepperoni = new ProductBuilder("Pepperoni")
  .Containing(Ingredients.Dough)
  .Containing(Ingredients.Pepperoni)
  .Please();

ビルドは、カスタマイズが必要なオブジェクトの作成に役立ちます。構成が1行で構成されている場合でも、ビルダーの作成を検討してください。

ObjectMother


製品の作成がはるかにまともになったという事実にもかかわらず、デザイナーはnew ProductBuilderまだかなり醜く見えます。ObjectMother（Father）パターンで修正します。

パターンは5 kopecksのように単純です。静的クラスを作成し、すべてのビルダーを収集します。

public static class Create
{
  public static ProductBuilder Product(string name) => new ProductBuilder(name);
}

これで次のように書くことができます：

var pepperoni = Create.Product("Pepperoni")
  .Containing(Ingredients.Dough)
  .Containing(Ingredients.Pepperoni)
  .Please();

ObjectMotherは、宣言的にオブジェクトを作成するように設計されています。さらに、ドメインに新しい開発者を紹介するのに役立ちます。CreateIDE自体を書くとき、このドメインで何を作成できるかがわかります。

このコードでは、ObjectMotherはnot Createではなくと呼ばれることがありGivenます。私は両方のオプションが好きです。他のアイデアがある場合-コメントで共有してください。
オブジェクトを宣言的に作成するには、ObjectMotherを使用します。コードはよりクリーンになり、新しい開発者がドメインを掘り下げることが容易になります。

注文のビルダー（後ろからの例）


次に、説明したパターンを適用してOrder Builderを作成しますが、ここではビルダーからではなく、受け取りたいものから移動しましょう。これは私が注文を作成したい方法です：

var order = Create.Order
  .Dated(DateTime.Now)
  .With(Pizza.Pepperoni.CountOf(1).For(500))
  .With(Pizza.Margarita.CountOf(1).For(650))
  .Please();

どうすればこれを達成できますか？注文と注文ラインにはビルダーが必要です。すべてを注文するビルダーを使用すると、すべてが明確になります。そこで彼は：

public class OrderBuilder
{
  private DateTime _date;
  private readonly List<OrderLine> _lines = new List<OrderLine>();
  
  public OrderBuilder Dated(DateTime date)
  {
    _date = date;
    return this;
  }

  public OrderBuilder With(OrderLine orderLine)
  {
    _lines.Add(orderLine);
    return this;
  }

  public Order Please()
  {
    var order = new Order(_date);
    foreach (var line in _lines)
    {
      order.AddLine(line);
    }
    
    return order;
  }
}

ただし、OrderLine状況はより興味深いものです。最初に、ターミナルのPleaseメソッドはここで呼び出されません。2番目に、ビルダーへのアクセスCreateは、ビルダー自体の静的またはコンストラクターによって提供されません。ヘルプで最初の問題を解決し、implicit operatorビルダーは次のようになります。

public class OrderLineBuilder
{
  private Product _product;
  private decimal _count;
  private decimal _price;
 
  public OrderLineBuilder Of(decimal count, Product product)
  {
     _product = product;
     _count = count;
     return this;
  }

  public OrderLineBuilder For(decimal price)
  {
     _price = price;
     return this;
  }

  public static implicit operator OrderLine(OrderLineBuilder b)
  {
     return new OrderLine(b._product, b._count, b._price);
  }
}

2番目の場合、クラスのExtensionメソッドはそれを理解するのに役立ちますProduct。

public static class ProductExtensions
{
  public static OrderLineBuilder CountOf(this Product product, decimal count)
  {
     return Create.OrderLine.Of(count, product)
  }
}

一般に、拡張メソッドはDSLの優れた友です。彼らは完全に地獄のような論理から宣言的で理解可能な説明をすることができます。
拡張メソッドを使用します。それらを使用してください。:)

サービスの作成（テスト可能なパターン）


今の注文で、すべてが多かれ少なかれ悪くはありません。リポジトリのmokaを処理する時が来ました。ここで私たちが検討しているテスト自体が動作のテストであることは、ここで言う価値があります。行動テストはメソッドの実装と強く関連しており、そのようなテストを記述できない場合は、記述しない方がよいでしょう。ただし、便利な場合もあれば、まったくなくてはならない場合もあります。次の手法は、動作のテストを正確に記述するのに役立ちます。それを使いたいと突然わかった場合は、まず、動作ではなく状態をチェックするようにテストを書き直すことができるかどうかを検討します。

だから、私は私のテストメソッドに単一のmokがないことを確認したいと思います。これを行うには、のラッパーを作成します。このラッパーではPizzeriaService、メソッド呼び出しをチェックするすべてのロジックをカプセル化します。

public class PizzeriaServiceTestable : PizzeriaService
{
  private readonly Mock<IOrderRepository> _orderRepositoryMock;
  private readonly Mock<IIngredientRepository> _ingredientRepositoryMock;

  public PizzeriaServiceTestable(Mock<IOrderRepository> orderRepositoryMock, Mock<IIngredientRepository> ingredientRepositoryMock)
     : base(orderRepositoryMock.Object, ingredientRepositoryMock.Object)
  {
     _orderRepositoryMock = orderRepositoryMock;
     _ingredientRepositoryMock = ingredientRepositoryMock;
  }

  public void VerifyAddWasCalledWith(Order order)
  {
     _orderRepositoryMock.Verify(r => r.Add(order), Times.Once);
  }

  public void VerifyReserveIngredientsWasCalledWith(Order order)
  {
     _ingredientRepositoryMock.Verify(r => r.ReserveIngredients(order), Times.Once);

  }
}

このクラスを使用すると、メソッド呼び出しを確認できますが、それでも何らかの方法で作成する必要があります。これを行うには、すでに知っているビルダーを使用します。

public class PizzeriaServiceBuilder
{
  public PizzeriaServiceTestable Please()
  {
     var orderRepositoryMock = new Mock<IOrderRepository>();
     var ingredientsRepositoryMock = new Mock<IIngredientRepository>();

     return new PizzeriaServiceTestable(orderRepositoryMock, ingredientsRepositoryMock);
  }
}

再現性（リテラル拡張）


リテラル拡張パターンは、再現性とは直接関係ありませんが、再現性に役立ちます。現時点での問題は、日付DateTime.Nowを注文日として使用することです。特定の日付から突然開始した場合、注文受付のロジックが変化します。ビジネスロジックでは、少なくともしばらくの間、注文受付の2つのロジックをサポートし、のようなチェックでそれらを分離する必要がありif (order.Date > edgeDate)ます。この場合、システム日付が境界を通過すると、テストが失敗する可能性があります。はい、これをすばやく修正し、2つのテストを行います。1つは境界日付の前にロジックをチェックし、もう1つは後でテストします。それでも、このような状況を回避し、すべての入力データをすぐに一定にすることをお勧めします。

「DSLはそれとどのような関係があるのですか？」- あなたが尋ねる。実際、たとえば、Extensionメソッドを使用してテストに日付を入力すると便利3.May(2019)です。この形式の録音は、開発者だけでなく、ビジネスにも理解できます。これを行うには、そのような静的クラスを作成するだけです

public static class DateConstructionExtensions
{
   public static DateTime May(this int day, int year) => new DateTime(year, 5, day);
}

当然、このパターンを使用するのは日付だけではありません。たとえば、製品の構成に成分の量を導入した場合、そのようなものを書くことができます42.Grams("flour")。
量的なオブジェクトと日付は、使い慣れた拡張メソッドを使用して簡単に作成できます。

フォーカス


テストに焦点を合わせ続けることが重要なのはなぜですか？焦点を絞ったテストは維持が容易ですが、サポートする必要があります。たとえば、コードを変更する場合は変更する必要があり、古い機能を削除する場合は削除する必要があります。テストに焦点が当てられていない場合、ロジックを変更するときに、大きなテストを理解し、テストされた機能の一部を切り取る必要があります。テストに焦点が当てられており、その名前が明確である場合は、廃止されたテストを削除して新しいテストを作成するだけです。テストに適切なDSLがある場合、これはまったく問題ありません。

したがって、DSLの作成を終えた後、このテストを2つのテストに分割することで、このテストに焦点を合わせる機会がありました。

[Fact]
public void WhenAcceptOrder_AddIsCalled()
{
  var order = Create.Order
     .Dated(3.May(2019))
     .With(Pizza.Pepperoni.CountOf(1).For(500))
     .With(Pizza.Margarita.CountOf(1).For(650))
     .Please();
 
  var service = Create.PizzeriaService.Please();

  service.AcceptOrder(order);
 
  service.VerifyAddWasCalledWith(order);
}

[Fact]
public void WhenAcceptOrder_ReserveIngredientsIsCalled()
{
  var order = Create.Order
     .Dated(3.May(2019))

     .With(Pizza.Margarita.CountOf(1).For(650))
     .Please();

テストの名前はそれらが書かれた目的を反映していることに注意してください。これで、私のプロジェクトに参加した開発者は、各テストが書かれた理由とこのテストで何が起こるかを理解できます。
テストの焦点はそれらをサポートします。優れたテストに焦点を当てる必要があります。
そして今、私があなたに叫んでいるのをすでに聞くことができます。いくつかのテストをきれいにするためだけに、100万のコードを書きました。」はい、正確に。テストは2、3しかありませんが、DSLに投資してこれらのテストをわかりやすくすることは理にかなっています。DSLを書くと、たくさんの特典があります。

  • 新しいテストを書くのが簡単になります。単体テストのために2時間セットアップする必要はありません。ただ受講して書いてください。
  • テストは理解しやすく、読みやすくなります。テストを見ている開発者は、テストが作成された理由とテストの内容を理解しています。
  • 新しい開発者の（またはドメイン内の）テストに参加するためのしきい値が削減されました。たとえば、ObjectMotherを使用すると、ドメイン内に作成できるオブジェクトを簡単に把握できます。
  • そして最後に、テストで作業するのは素晴らしいことです。その結果、コードがよりサポートされるようになります。
