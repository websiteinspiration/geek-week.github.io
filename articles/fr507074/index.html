Aide-mémoire SIMD, maintenant pour .NET Core

Ce n'est un secret pour personne que les intrinsicistes ont apporté. J'ai écrit à ce sujet avant leur apparition et après . De plus, poste toujours sur Habré, par exemple ceci . Et tout, semble-t-il, est merveilleux, si ce n’était pour un «mais»: ces intrinsèques sont appelées dans un style de sous-réseau, et pas du tout comme dans assembleur ou C ++.

Pour cette raison, il est difficile d'utiliser un code vectorisé prêt à l'emploi simplement en le portant des pros. Il sera également difficile avec l'adaptation des réponses et des exemples trouvés sur stackoverflow)) Après tout, pour chaque appel, vous devez lui trouver un nouveau nom. Et ce n'est pas si facile - si vous recherchez le nom d'un intrinsèque positif dans MSDN, il n'y a que des articles en plus, et peut-être quelque part à la fin ce dont nous avons besoin.

Quand j'ai dû écrire du code SIMD sur les pros, j'ai utilisé une très bonne ressource - officedaytime , où toutes les instructions de base pour la plate-forme x86 sont brièvement et clairement présentées. J'ai décidé que ce serait bien d'avoir une telle feuille de triche pour le net.



Donc, je ne verse pas beaucoup d'eau, voici cette page adaptée pour netcore .

Comment il a été généré


De toute évidence, je ne me suis pas assis et j'ai constamment dirigé cette table avec mes mains. Heureusement, pour de nombreuses méthodes (mais pas toutes) .NET Core, l'intrinsèque correspondant est écrit, comme on l'appelle chez les pros. Il est également pratique que tous ces conseils soient générés dans XML-Doc dans un fichier situé à côté de l'assembly: System.Runtime.Intrinsics.xml.

Pour travailler avec HTML, j'ai pris le bon vieux HtmlAgilityPack , qui non seulement fonctionne toujours bien, mais est également activement pris en charge.

La nature intrinsèque de la création de vecteurs, ainsi que la mise à zéro, ont apporté un petit inconvénient. Pour eux, nous avons dû ajouter des constantes distinctes dans le code du générateur.

Et aussi les instructions de décalage _mm_slli_si128 et _mm_srli_si128 pour une raison quelconque sont nommées différemment dans le sous-réseau, j'ai dû les corriger aussi.
