DRAMハードウェア暗号化は間近です。一般ユーザーをどのように脅迫していますか？

ディスク、パーティション、個々のドキュメントを暗号化するための多くのオプションがあります。 1台のデバイスが侵害された場合、連邦の鍵の配布さえあり、アクセスには複数の当事者の参加が必要です（Shamirの秘密共有スキームを参照）。ファイルシステム暗号化オプションは、静的データを保護するための標準的な方法と見なすことができる幅広いオプションを提供します。

もう1つは、RAM内のデータです。それらはオープン形式で保存され、メモリとCPUの間でオープンに転送されます。仮想化された環境で、攻撃者が隣接する仮想マシンからメモリを読み取る方法を見つけた場合、攻撃者はサーバー上の他のVMのデータにアクセスする可能性があります。物理的な攻撃は、メモリチップをコピーするか、バス上のデータを傍受することで可能です。停電後もデータが保存される読み取り専用メモリの場合、脅威はさらに深刻になります。

RAM暗号化テクノロジーは、これらの攻撃の一部に対処します。予想外の「副作用」がパーソナルコンピュータに現れるのではないかという恐れがありますが、たとえば、壊れないDRMなどです。

IntelとAMDは、ハードウェアインメモリ暗号化スキームを提供しています。

Intel暗号化


2017年末に、IntelはTotal Memory Encryption（TME）と呼ばれるx86命令セットの拡張を提案し、物理的なDRAMとNVRAMを単一の一時キーで完全に暗号化しました。近い将来、TMEにマルチキーサポートのあるマルチキートータルメモリ暗号化（MKTME）拡張機能を追加し、異なるキーと異なるアプリケーションでメモリのページ暗号化を行う可能性を計画しています。

現在のソフトウェアとの互換性を維持するために、プロセッサ内のデータは、CPUキャッシュ内だけでなく、開いたままになります。これらはメモリコントローラーレベルで暗号化され、DRAMで暗号化された形式で受信されます。



AES-XTS-128暗号化は、物理的にメモリバスの近くにあるAES-XTSエンジンによって実行されます。

TMEスキームは、プロセッサによるキー生成とメモリ全体の完全な暗号化を提供します。この暗号化はBIOSでオンとオフが切り替えられ、OSとアプリケーションに対して透過的です。

MKTMEは、メモリの異なるページを暗号化するための個別のキーを導入します。 keyID（物理アドレスの15ビット）を使用した特別なメモリアドレス指定スキームが開発されました。 MKTMEを機能させるには、BIOSでTMEをアクティブ化する必要がありますが、MKTMEを使用すると、TMEによって作成された個々のメモリ領域の暗号化を無効にできます。

個別のメモリ暗号化の主なタスクの1つは、データセンターでの仮想マシンのより信頼性の高い分離です。個別のキーでkeyIDを使用すると、実行時に各仮想マシンのメモリを暗号化できます。仮想マシンごとに、異なるkeyID。暗号化キーは、ハードウェアではなくソフトウェアレベルで生成できるようになりました。



LinuxカーネルでのMKTMEのサポート


インテルは将来のプロセッサーにMKTMEを実装する予定なので、2018年9月にLinuxカーネルレベルでMKTMEサポートを担当し、数か月後にMKTME対応APIの更新されたRFCを導入しました。両方のパッチの主な開発者の1人はベラルーシのハッカーであるKirill Shutemovです。Linuxカーネルのパッチの議論で彼のコメントを参照してください。

新しいソフトウェアインターフェイスは、暗号化とキーを構成するためのユーザーインターフェイスをサポートし、ホットスワップCPUの場合は、キーIDをメモリ領域とキーストレージに割り当てます。

Linuxカーネルのパッチには、新しいAPI関数の使用例が含まれています。

//    Add a 'user' type key::

char \*options_USER = "type=user
       algorithm=aes-xts-128
       key=12345678912345671234567891234567
       tweak=12345678912345671234567891234567";

key = add_key("mktme", "name", options_USER, strlen(options_USER),
       KEY_SPEC_THREAD_KEYRING);

//    Add a 'cpu' type key::

char \*options_USER = "type=cpu algorithm=aes-xts-128";

key = add_key("mktme", "name", options_CPU, strlen(options_CPU),
       KEY_SPEC_THREAD_KEYRING);

//   Update a key to 'Clear' type::

ret = keyctl(KEYCTL_UPDATE, key, "type=clear", strlen(options_CLEAR);

//   Add a "no-encrypt' type key::

key = add_key("mktme", "name", "no-encrypt", strlen(options_CPU),
       KEY_SPEC_THREAD_KEYRING);

いくつかのLinuxカーネル開発者は、APIへの提案された変更に反対し、キャッシュコヒーレンスの問題と、CPUが交換された場合に導入されるコールドストレージによるキーリークのリスクを指摘しています。メモリコントローラーは特定のメモリアクセス要求がどこから来たのかわからないので、

誰かがMKTMEが仮想マシンの分離に役立つのではないかと疑っていました。実際、ハイパーバイザー内で悪意のあるコードが実行された場合、MKTME は役立ちません。

開発者は、メインブランチにコミットする前にこのパッチに変更を加える可能性があります。

AMDの代替


AMDの同様のTMEメモリ暗号化テクノロジは、Secure Memory Encryption（SME）と呼ばれます。 TMEと同様に、BIOSに含める必要があります。その後、プロセッサが単一のキーを生成し、すべてのメモリをオペレーティングシステムとアプリケーションに対して透過的に暗号化します。

Secure Encrypted Virtualization（SEV）拡張機能は、仮想マシンごとに個別のキーを提供します。各VMは、暗号化するメモリのページを示します。鍵管理はAMD Secure Processorによって処理されます。

リストされているコンポーネントは、AMD EPYCサーバープロセッサに実装されています。説明から判断すると、SEVテクノロジーはMKTMEと非常によく似ています。同様に、ハイパーバイザーから悪意のあるコードが起動された場合、SEVテクノロジーは無力です（SEVered攻撃の説明を参照）。

2019年6月、SEVテクノロジーの完全サポートがSUSE Linux Enterprise 15 Service Pack 1に実装されました。
