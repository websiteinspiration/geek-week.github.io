クイックファイル検索

翻訳者から：1983年1月15日に発行された非常に古い記事の翻訳に注目します。そのような印象的な時代にもかかわらず、この記事は私には興味深く思われ、今日の誰かに役立つ可能性があります。ちなみに、この記事はopennet.ruのman Locate（1）ヘルプセクションで参照されています：https ://www.opennet.ru/man.shtml?topic=locate 。



概要


この記事では、UNIXでのクイックファイル検索メカニズムについて説明します。 2つのデータ圧縮方法と新しい行検索手法を組み合わせ、任意のファイルをすばやく検索するように設計されています。標準の検索ユーティリティに統合されたコードは、以前に作成され、毎日更新されるデータベースを検索します。これは、分散した（ディスク上の）ディレクトリ構造からオンザフライで生成される候補とのキー一致を検索する通常のメカニズムとは異なります。

ファイルパスデータベースは、インクリメンタルにエンコードされ、辞書式にソートされたリスト（「フロント圧縮」ファイルと呼ばれることもあります）であり、効果的な圧縮を得るために、通常のバイグラムエンコーディングも行われます。通常のASCII表現と比較して、圧縮率は5〜6です。アルゴリズムで費やされる通常の時間は通常の検索よりも40〜50％短くなりますが、リストは、線形増分検索を使用してスキャンされます。

前書き


コンピュータシステムまたはコンピュータネットワークでファイルを見つけることは複雑なプロセスです。 UNIXユーザーは、cd、ls、およびgrepコマンドの操作から、Berkeley whereisおよびfleeseで開発されたコマンドなどの特殊なコマンド、さらに一般的なfind Unixコマンドまで、さまざまな方法で実装できます。

残念ながら、フリース（Berkeley製）はホームディレクトリに限定されており、whereisは標準的な場所にあるシステムコードとドキュメントのみを検索できます。

探す：

find / -name "*<filename>*" -print

もちろん、任意のファイルを検索することはできますが、ファイルシステム全体で再帰的降下を使用し、ディスク全体に容赦なく分散しているため、非常に低速です。焦りがあったため、ファイルへのパスを検索する（「検索して見つける」方法に関連して）代替の検索を開発するように求められました。

予備計算


システム内のすべてのファイルの静的リストを作成し、その上でgrepを探してみませんか？/ usrを/ uに短縮しても、20,000ファイルを含む一般的なシステムには、1,000ブロックのファイル名が含まれます。アンロードされたPDP-11 / 70システムのGrepは1秒あたり30〜40ブロックを処理し、スキャンには30分かかります。これは、一般的に使用されるコマンドでは受け入れられません。

しかし、少し犠牲を払えば、1日以内にファイルを検索できない場合、そのようなファイルを作成した人がおそらく手の届く範囲にあるか、ファイルがまだ準備ができていないため、大きな問題は発生しません使用する。異なるファイル命名規則を持つ他のユーザーグループによって作成された古いファイルは、最も可能性の高い検索候補です。

圧縮


アプリケーションへのアクセスを高速化するには、バイナリ検索またはハッシュテーブルを使用することをお勧めしますが、名前の一部のみに関心がある場合、そのようなスキームは部分比較ではうまく機能しません。インクリメンタルコーディングとして知られている、順序付けられたデータを圧縮するための簡単に実装できる手法が、同様の辞書圧縮タスクに使用されました[Morris / Thompson、1974]。この場合、前の名前の最長プレフィックスが計算されます。例えば：

/usr/src
/usr/src/cmd/aardvark.c
/usr/src/cmd/armadillo.c
/usr/tmp/zoo


に変換 

0/usr/src
8/cmd/aardvark.c
14armadillo.c
5tmp/zoo

デコードは単純になります（変数の宣言は省略されます）

fp = fopen(COMPRESSED_FILELIST, "r");
while((count = (getc(fp) & 0177)) != EOF) {
    for(p = path + count; (*p++ = getc(fp)) < 0200; )
        ; /*overlay old path with new*/
    ungetc(*--p, fp);
    *p-- = NULL;
    if(math(path, name) == YES)
        puts(path);
}

ここで、mathは、名前の部分文字列がパス文字列に含まれているかどうかを判断する関数です。

実際、コード化されたリストはコード化されていないリストよりも約5倍短く、デコードは非常に単純なので、プログラムはテキストファイルでのgrepの 3〜4倍の速度で実行されます。

さらに速く


このアプローチはすでに有用ですが、さらに改善の余地があります。（注：このコードはfindユーティリティで使用されます。既存の同様のプログラムを改善できる場合、UNIXに別のコマンド[およびmanページ]を負担する必要はありません。幸い、2つの引数を使用したfind呼び出しはなく、装飾せずにvoidを埋めることができます。 ：

find name

上記のコードは、ディスクではなくメモリ内ではありますが、まだ解凍されたリストを検索することに注意してください。これは、文字列をサブストリングと反対方向に比較することで回避できます。 namendが文字列nameの最後の文字を指すようにし、mathを次のように置き換えます。

for(s = p, cutoff = path + count; s > cutoff; s--) {
    if(*s == namend) { /*quick first char check*/
        for(p = namend - 1, q = s - 1; *p != NULL; p--, q--)
            if(*q != *p)
                break;
        if(*p == NULL) {
            puts(path);
            break;
        }
    }
}

これは、3つのケースを見ると理解しやすくなります。サブストリングが完全にcutoffの右側にある場合、比較は正常に終了します。それらが重なる場合、カットオフは「ソフト」になり、比較が続行されます。サブストリングが完全にcutoffの左側にある場合、前の行との一致が明らかになるため、この検索を実行できません。技術的には、比較が行われた直後にカットオフをパスに移植できます。この条件は上記では省略されています。

2層機器


検索パターン 文字の処理による検索の速度低下を回避できます。その際、メタ文字を含まない名前の部分を処理し、find内で低速の再帰関数amathを使用します。つまり、

puts(path)

なる

if(globchars == NO | amatch(path, name))
    puts(path);

ここで、globcharsは、名前にメタ文字が含まれている場合に設定されます。単純なmanコマンドの検索パターンの使用例：

vtroff -man 'find '*man*'"$1"'.[1-9]

さらなる改善


findユーティリティの製品コードは、最も一般的な2文字の組み合わせを印刷不可能なASCIIコードに置き換えることにより、圧縮率をさらに20〜25％高めます。「.c」と「.P」は特に一般的です。

Huffmanアルゴリズム[Reghbati、1981]など、時間とデータサイズの間に他のトレードオフを実装する他のアルゴリズムは有望ではありません。I/ Oパフォーマンス制限をパフォーマンス制限に置き換えるだけです。

Boyer-Mooreサブリニア検索法[Boyer、1977]またはマクロモデル法[Storer / Szymanski、1982]を使用できます。

結論として、180ブロックと2ページに収まるCコードを使用して、19,000個のファイル名を数秒でスキャンしたことに注意してください。
