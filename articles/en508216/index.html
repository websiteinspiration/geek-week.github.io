What is CI (Continuous Integration)

CI (Continuous Integration) - literally translated "continuous integration". This refers to the integration of individual pieces of application code among themselves. The more often we collect the code together and check:



  • Is he going to?
  • Do auto tests pass?


All the better! CI allows you to do such checks automatically. It is used in advanced development teams that write not only code, but also autotests. He is asked at interviews - at least an understanding of what it is. Yes, even with testers.



Therefore, I will tell in an article about what it is. How CI works and how it will come in handy for your project.

What is CI?



CI is the assembly, deployment and testing of an application without human intervention. Now I will explain with an example.



Let's say that we have two developers - Masha and Vanya. And tester Katya.



Masha writes the code. Adds it to the version control system (from the English Version Control System, VCS ). This is something like a dropbox for code - a storage location where all changes are saved and at any time you can see who changed what, when.



Then Vanya finishes his piece of functionality. And also saves the code in VCS.







But this is just the source code - a set of files with the extension .java, or any other. So Katya can test the changes, you need:



  1. Build a build from source code
  2. Run it on a test machine


Building a build is when we create one executable file from a set of source code files: You







can build a build manually, but this is an unnecessary hemorrhoids: remember that in what order to run, which files depend on each other, do not make a mistake in the command ... Usually they use a special the program. For java, this is Ant, Maven, or Gradle. With the help of the collector, you set up the build process once, and then run it with one command. Launch example for Maven: 



mvn clean install


This is semi-automation - you still need a person who will enter a team and assemble the build with “pens”. Let's say developers do this. When Katya asks the build for testing, Vanya updates the version from the repository and collects the build.







But build a build ≠ get an app for testing. It still needs to be launched! This is done by the application server. Servers are different: wildfly, apache, jetty ...



If this is wildfly, then you need:



  1. Attach build to standalone / deployments directory
  2. Start the server (after setting up the service once)


And this is again semi-automation. Because the developer needs to copy the resulting archive after assembly to the test bench and enable the service. Yes, this is done by a couple of teams, but still a person.







But if we remove a person from this scheme, we get CI!







CI is an application that allows you to automate the entire process. It pulls changes from the repository with code. By itself! There are two configuration options:



  • CI polls the repository “Hey cuckoo, do you have any changes ??” every N hours / minutes, how to configure.



  • The repository waves its hand at CI when committing: “Hey hello! And I got an update here! ” (is it a git hook or equivalent in your VCS)







When the CI receives the changes, it starts building the build and autotests.



If the assembly failed (the tests fell, or the project could not be assembled), the system writes an e-mail to all interested parties:



  • Project Manager (to know what is being done!)
  • The developer who made the changes
  • Anyone else - as you set it up, it will be so.








If the build is successful, CI deploys the application on the test machine. And in the end Katka can test the new assembly!







Yes, of course, once you have to set it all up - tell the CI server where to get the changes from, which autotests to run, how to build the project, where to build it later ... But then I set it up once - and then it does!



Autotests will also have to be written by ourselves, but well done =)







If it’s on your fingers, then the CI ( Continuous Integration ) system is a kind of program that monitors your Source Control, and when changes appear there, it automatically pulls them together, builds them, runs auto tests (of course, if they are written).



In case of failure, it makes this known to all interested parties, in the first place to the last committer. (c) habr.com/en/post/352282

Let's see how it looks from the point of view of the user. I will show with an example of the TeamCity system.



When I log in, I see all the tasks. The tasks are different:



  • Build build
  • Drive Auto Tests
  • Deploy the application on the test bench
  • Run GUI tests (or Postman tests) on this stand

  • CDI Core with tests - assembly of the project with all autotests, which, as can be seen on the screen, 4000+ pieces. Tests take half an hour, but are also run for each commit.

How CI works



How and where does CI collect builds and run auto tests? I will use TeamCity as an example, but other systems work similarly.



TeamCity itself does not collect anything. The assembly and run of autotests takes place on other machines called “agents”:







“Agent” is a simple computer. A piece of iron or a virtual machine, not the point. But how does this computer understand what it needs to do?



TeamCity has a server and a client. The server is the very application in which you will then poke buttons and watch a beautiful picture "how everything went well". It is installed on one machine.







And the “client” application is installed on the “agent” machines. And when we click the “Run” button on the server:







The server selects a free client and gives it all the instructions: what exactly needs to be done. The client collects the build, performs autotests, collects the result and returns to the server: “On, hold, draw”.







The server displays the result to the user plus sends an email to all interested parties.



At the same time, we always see on which particular agent the assembly took place:







And you can choose where to run the auto tests yourself. Because it happens that autotests fall on only one build agent. This means that he has something wrong with the configuration.

We can also configure for each assembly a list of agents on which it can run. What for? For example, we have half of Linux agents, and half of Windows. And we did the assembly only for one system. Or on Windows some kind of floating bug crawls out, but fixing it is long and expensive, and all the client on Linux - well, why then?



And it also happens that agents are divided between projects so that there is no fight - this project uses Burana and Apollo, and that Chip and Dale. The thing is that only one task can be performed on one agent. Therefore, it makes no sense to buy a cool car for an agent; immediately you can’t drive a bunch of tests there anyway.



In the end, how it works: first, the admin purchases computers under the “agents” and installs the TeamCity client application on them. They should not be too cool, because they will not do many tasks at once.







In this case, TeamCity you pay for the number of licenses for build agents. So the fewer they are, the better.



On a separate machine, the admin installs the TeamCity server. And configures it - configures assemblies, indicates which assemblies can be driven on which machines, etc. The server needs a place to store artifacts - the results of the assembly.



We have two projects - Single Client and Factor, which interact with each other. The Single Client Tester may not collect Factor locally. He launches the build in TeamCity and downloads the finished build from artifacts!


Further, the developer chooses which assembly he wants to run and clicks “Run”. What happens at this moment:



1. The TeamCity server checks the list on which agents this assembly can be run. Then he checks which of these agents is currently free:







Found free? I gave him the puzzle!







If all agents are busy, the task is queued. The queue works according to the FIFO principle - first in, first out. Who first got up - that and slippers.







The queue can be adjusted manually. So, if I see that the queue is full of assemblies launched by the version control system, I will raise mine to the very top. If I see that people started assemblies, it means that they are also important, you have to wait.







This is normal practice if agent capacities are not enough throughout and a queue is created. You look who launched it:



  • Robot? So, this is just a routine check that nothing more has been broken. This can wait 5-10-30 minutes, that's okay
  • Colleague? This assembly is important to him, since he did not wait for the planned launch. We stand in line, you should not go forward.


Sometimes you can even cancel the build from the version control system, if it’s really hot, and all the agents are busy with hourly tests. In this case, you can:



  • raise your turn to the very top so that it starts on the first free agent
  • log in to the agent, cancel the current build
  • restart her! Although it will fall to the very bottom of the line, it’s just ugly to cancel the assembly


2. The agent performs the task and returns the result to the server







3. The server draws the result in the graphical interface and saves the artifacts. So I can go into TeamCity and see in the artifacts the full logs of the past autotests, or download the project assembly to deploy it locally.



It is highly recommended that you pre-configure the number of assemblies that CI will store. Because if the artifacts contain builds of 200+ mb and there will be a lot of them, then the next launch of the assembly will fail with the error “out of disk space”:







4. The server does the emailing - that's how you set it up. He can do a positive newsletter “the assembly was successful”, and he can send mail only in case of a failure “Oh, her, something went wrong!”.





Integration with VCS



I talked about different options for configuring CI-VCS integration:



  • CI polls the repository “Hey cuckoo, do you have any changes ??” every N hours / minutes, how to configure.



  • The repository waves its hand at CI when committing: “Hey hello! And I got an update here! ” (is it a git hook or equivalent in your VCS)







But when is which used?



Best of all, of course, is for the version control system to notify the CI server. And run the entire cycle for each change: assemble, test, deploy. Then any change to the code will immediately go to the test environment, which will be as relevant as possible.







Plus, each change runs autotests. And if the tests fail, it’s immediately clear whose commit broke them. After all, it used to work and after Vasya's edits it suddenly broke - which means that his commit led to a fall. With rare exceptions, when the fall is floating.







But in real life, such a scheme is rarely applicable. Just think - you can have many projects, many developers. Each commits something at least once every half an hour. And if you run 10 assemblies in half an hour for each commit, the queues in TeamCity will never be raked!



One of our products has a core module, and there are 15+ Customers. Each has its own autotests. Customer assembly is core + customer features. That is, a change in the root project can affect 15 different assemblies. So, they all need to be run when committing in core.



When we had 4 build agents, all-all-all builds and tests for these customers were launched on Tuesday night. And by 10 a.m. there was still a couple of hours in TeamCity.


Another option is to purchase many agents. But this is the price for the car itself + for the license in TeamCity, which is already much more expensive, and even pay every month.



Therefore, they usually do as:



1. Very fast and important assemblies can be left to any commit - if it takes 1-2 minutes, let them chase.







2. The rest of the builds check to see if there have been changes in VCS - for example, once every 15 minutes. If there were, then we launch.







3. Long tests (for example, performance tests) - once every few days at night.











CI in testing



If we are talking about developing our application, then testing is included in the standard cycle. You or your developers write autotests, which are then driven by CI. It can be unit, api, gui or stress tests.







But what if you are testing a black box? There is an application, no source code. These are the harsh realities of integration testers - the supplier gives you a new application release, which you need to check before putting it into production.



Here, let's say you have API tests in Postman. Or GUI tests in Selenium. Can I customize the CI loop for them?







Of course you can!



CI does not set a strict framework such as “I only work in projects with automatic tests” or “I only work when I have access to the source code.” It may look into the version control system, or it may not look. This is an optional condition!



Wrote autotests? Tell the CI server how to run them often - and enjoy the result =)









Total



CI - continuous integration. This is when your application is constantly being tested: is everything good with it? Do the tests pass? Is the assembly going? Moreover, all checks are carried out automatically, without human intervention.



It is especially important for teams where several developers work on the code of one application. How does this happen? Separately, parts of the program work, but together no longer. CI allows you to very quickly detect such problems. And the faster you find it, the cheaper it is to fix it.







Hence the name - a constant check of the integration of pieces of code among themselves.

Typical CI Tasks:



  • Check if there was an update in code
  • Build build
  • Drive Auto Tests
  • Deploy the application on the test bench
  • Run GUI tests (or Postman tests) on this stand
  • Notify everyone interested by email about the results of assembly and testing








And all this - automatically, without human intervention! That is, they set it up once, and then it itself.



If CI is configured in the project, you will have constantly up-to-date test stands. And if something breaks in the code, you will know about it right away, the CI server will send a letter. And you can go into the graphical interface and see if all the builds are successful, and the tests are green? Rate the picture of the project in a minute.
