Got Juno Meetup-9月12日、ミンスク

みなさん、こんにちは。私の名前はアントントゥピコフです。私はゲットのB2B開発のチームリーダーです。

Gettは単なるタクシーサービスではなく、ダイナミックな市場です。これは非常に負荷の高いシステムです。世界中の120の都市にある何万ものドライバーと何百万もの顧客が毎日Gettサービスを使用しています。

最初にB2Cアプリケーションとして開発されたGettは、過去6年間でいくつかの「新興企業のスタートアップ」を立ち上げました。これらは、配達、ミニバス、法人販売の実験でした。それらの多くは後に成功するビジネス分野に変わりました。たとえば、B2Bソリューションは、Google、Visa、Gucci、LVMH、Nestleなどのブランドを含む、15,000を超える大企業で正常に機能します。

ゲットが直面する次の大きな課題は、来年にIPOを取得することです。次世代の製品が私たちのIPOの成功に役立つことを願っています。そのため、最高のGettとJunoのエンジニアの努力を組み合わせてこの目標を達成することが決定されました。これは、最先端のテクノロジースタックを使用してゼロから革新的な製品ソリューションを開発することです。

9月12日にミンスクで開催される会議にGoWayコミュニティの支援を得て全員を招待したいと思います。そこでは、私の仲間の開発者が毎日直面する技術的な課題と、新しいソリューションに取り組むために今後直面する必要があることについて少し話し合います。

    In Gett much of the functionality is location based. During each order there is a need to retrieve configuration data based on GPS locations. Geo services are handling large number of requests and are on critical path of the business flow. It is required that the service will respond quickly and will be scalable.   • Ron Ludmer, Gett, Senior Backend Developer at Matching & Ride Exchange Team – Redis-based Semaphore ( c )

    As the popularity of microservices architecture style increases, the risks that rise from working with a distributed system are getting more focus. One common problem is having our shared resources being accessed from multiple instances simultaneously, causing data inconsistency. What is the best way to ensure correctness with minimal damage to efficiency? Arguably, the simplest way is using locking mechanism. Ron are going to talk about different approaches to distributed locking, and the ways they tackled this issue at Gett, leading up to our current solution: Redis-based distributed lock implemented in Golang.

そして、なぜ今、私たちはJunoと力を合わせることに決めました。

Gett B2Bセグメントが最初から運用上利益を上げていた（そして全体的な収益性のための戦略の対象であった）ことを考えると、これによりGettは企業顧客とそのニーズに関する価値提案を再構築するようになりました。Juno

を2017年4月に購入しました。、ニューヨークで事業を展開するスタートアップで、ミンスクに開発センターがあります。 Junoは2015年に設立された配車サービスです。当初から、ジュノは自分自身について大声で述べました。このサービスは2016年5月にニューヨークで開始され、すでに2016年9月に旅行の数は月50万を超えました。 2017年9月、成功した旅行の数は100万を超えました。今日、ジュノはニューヨークのオンラインタクシー輸送市場で主要な地位を占めています。

Gettは現在、B2Bソリューションの技術更新を優先しています。これらの目的のために、JunoのミンスクオフィスとモスクワおよびイスラエルのGett RnDチームの取り組みを組み合わせることが決定されました。

B2Bプラットフォームは、旅行に関連するすべての企業の問題を​​解決します。つまり、車の注文や監視から、財務部門による経費の管理やレポートの生成までです。

Junoチームは、Gettと一緒にサービスのいくつかの領域で作業します。

従業員の接続
多くの場合、企業では、従業員自身がタクシーや配達サービスを使用することはできません。別の従業員/サービスを通じて旅行のリクエストが行われる場合があります。各会社には独自の旅行ポリシーがあり、Gett B2Bのタスクは、各クライアントの詳細にできるだけ適応することです。個人アカウントをカスタマイズし、すべてのルールとポリシーを考慮してください。

原価管理
Gett B2Bが解決するクライアントにとって最も具体的な問題の1つは、企業の輸送コストの透明性です。透明性は、各旅行のアカウントの詳細だけでなく、企業が各都市または国で負担するすべての費用を1つの場所に集約する可能性にも関係します。将来的には、これらのレポートはVATの払い戻しなどに使用されます。

Gett B2Bは、管理者と金融サービスの手動作業を自動化することにより、これらの問題を解決します。小切手を集めたり、旅行の詳細を調べたり、異なる部門に費用を請求したり、各従業員の情報を手動で入力したり、何度も再確認したりする必要はありません。顧客はSFTPデータを同期したり、APIトリップを作成したりできます。

さらに、JunoとGettは、課金、コンシューマーサービス、ベンダーコネクションの分野で協力します。B2Bソリューションテクノロジー

の選択は、最新のデジタル製品が直面する問題を効率的に解決することを目的としています。

  • AWSとKubernetesはライドヘイリング市場に典型的な変動負荷に適応し、効果的にスケーリングするのに役立ちます
  • Jenkinsに関連付けられたプロセスを継続的に提供することで、新しい機能で顧客を毎日喜ばせることができる
  • GoとRedisを使用すると、システム負荷が最も高いポイントでリソースを効率的かつ柔軟に使用し、時代に遅れずについていくことが可能
  • React / Redux＆Rubyのおかげで、迅速に対応して製品の変更を実装し、優れたユーザーエクスペリエンスを提供する
  • CloudAMQPのおかげで、データの一貫性を維持しながらマイクロサービスアーキテクチャの一部になる
  • 製品のパフォーマンスを監視し、高品質であることを確認します。PagerDuty、Airbrake、New Relic、Datadog、および組み込みユニット、コンポーネント、Java / testNG / Seleniumの統合テストを含む継続的統合プロセスを含む異常を監視および検索するためのエコシステムの助けを借りて可能です
