FT8通信プロトコル-仕組み

こんにちは、ハブル。

おそらく、無線通信とアマチュア無線に少しでも興味を持っていたすべての人が、FT8デジタル通信プロトコルについて聞いたでしょう。このタイプの接続は2017年に登場し、それ以来その人気は高まっています。


出典：www.qsl.net/w1dyj/FT8%20for%20web.pdf

これがどのように機能するのか、なぜ必要なのかに興味がある人は、カットの下で続けてください。

物語


アマチュア無線通信から遠い人のために、簡単な歴史的背景。

自家製で最初に購入されたアマチュアラジオ局は、約100年前に登場しました。彼らが送受信できる唯一の信号は単純なトーンであり、送信にはモールス符号が使用されていました。もちろん、以前は耳でCWをとる方法を学ぶ必要がありましたが、このスキルはアマチュアだけでなくプロの間でも長い間求められていましたが、海軍のラジオ事業者の職業は60年代にのみ取り消されたようです。もちろん、エーテルのノイズに対する弱い信号の受信には一定のロマンスがありました。これは、コストロマ地域の単純な村のハムラジオ事業者が1928年に遭難した飛行船チームウンベルトノビレからSOS信号を受信する方法でした。

次のステップは音声通信でした。シングルバンド変調でマイクを使用して音声を送信すると、無線通信がより便利になります。この方法は引き続き使用されます。

最後に、90年代頃、デジタル通信はハムラジオ事業者の間で人気を集め始めました。この場合、信号（通常はRTTYまたはPSK）がコンピューターを形成し、オペレーターは一種のテキストチャットを使用して通信相手と通信します。



技術的に類似したプロトコルは非常に単純であり、ユーザーがキーボードから入力した文字はほとんど複雑な処理をせずにブロードキャストに送信されたと言う必要があります。そして不思議なことに、これには利点がありました。オペレーター自身が適切な周波数を見つけ、信号の品質を耳で評価し、デコードされた文字を画面で確認し、答えを入力しなければなりませんでした。

しかし、デジタル処理の可能性が高まり、ついにレビューの主人公、通信規格FT8が登場しました。FT8という名前は、FrankeおよびTaylor、8-FSK変調と解釈されます。スティーブン・フランケ教授と天体物理学者でノーベル賞を受賞したジョセフ・テイラーは、データ転送のための本当に興味深く効率的なプロトコルを作成しました。

FT8の機能：

  • メッセージの長さは15秒で、メッセージは固定タイムスロットで送信されるため、デコードが容易になります。
  • メッセージ長は77ビット+ 12ビットCRCです。
  • エラー訂正FEC LDPC（174.87）。
  • 周波数変調8-FSK、トーン間の距離6.25Hz。
  • 帯域幅50Hz。このような狭帯域では、複数のステーションが同時に動作してデコードできます。
  • デコードしきい値-20dB。
  • 自動的に機能する、応答を自動的に送信するなどのオプション機能。

一般的に、著者は、一方ではより効率的に情報を送信し、他方では芽の中の「アマチュア無線の精神」を殺すことができる、ほぼ完全に自動化された通信プロトコルを作成する上手な仕事をしました。ノイズのロマンスやエーテルのフェージングは​​なく、海の反対側から対応する弱い信号を受信します。PC画面のみがあり、残りはアルゴリズムと数学によって行われます。

個人的には、ほぼ完全に自動化された無線通信の楽しさはよくわかりませんが、FT8が人気があるので、これは誰かです必要。これがどのように機能するかをさらに詳しく見てみましょう。

ビットコーディング


実用的な例を考えてみましょう-「CQ RA1ABC KO50」というフレーズの送信。ここで、CQは一般的に受け入れられている呼び出しコード（詳細はこちら）、RA1ABCはアマチュアコールサイン、KO50はいわゆるグリッドロケーターで、アマチュアラジオが放送する地域を決定します。

最初のステップでは、一般的な略語は短いコードに置き換えられます。たとえば、CQは2hコードを受け取ります。アマチュア無線のコールサインは、文字と数字の形式が厳密に定義されているため、よりコンパクトな形式で録音することもできます。

const char A0[] = " 0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ+-./?";
const char A1[] = " 0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ";
const char A2[] = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ";
const char A3[] = "0123456789";

// Check for standard callsign
int i0, i1, i2, i3, i4, i5;
if ((i0 = char_index(A1, c6[0])) >= 0 && (i1 = char_index(A2, c6[1])) >= 0 &&
    (i2 = char_index(A3, c6[2])) >= 0 && (i3 = char_index(A4, c6[3])) >= 0 &&
    (i4 = char_index(A4, c6[4])) >= 0 && (i5 = char_index(A4, c6[5])) >= 0) {
        // This is a standard callsign
        int32_t n28 = i0;
        n28 = n28 * 36 + i1;
        n28 = n28 * 10 + i2;
        n28 = n28 * 27 + i3;
        n28 = n28 * 27 + i4;
        n28 = n28 * 27 + i5;
        printf("Pack28: n28=%d (%04xh)\n", n28, n28);
        return NTOKENS + MAX22 + n28;
}

文字をインデックスで置き換えるという考え方はここでは重要です。これは、辞書のサイズが小さい場合に効果的です。その結果、コールサイン「RA1ABC」は0BF1C2C1hとして書き込まれます-6ではなく4バイトを費やしました。

同様に、「KO50」グリッドロケーターは2バイトの4BFAh番号として書き込まれます。メッセージの最大長は77ビットであるため、これらすべてが必要です。もちろん、任意のテキストやテレメトリさえ送信できますが、これらの制限があります。 77ビットのメッセージ長は、送信時間（約13秒、それほど高速ではありません）と情報容量の間の妥協点として選択されました。メッセージを長くすることはできますが、送信時間が長くなります。

その結果、すべての「CQ RA1ABC KO50」メッセージはデータブロック00 00 00 26 28 9F D4 92 FE 88に変換されます。14ビットのCRCもそれに追加されます。

次のステップは、91ビットメッセージを長さ174ビットのいわゆるLDPC（低密度パリティチェック）コードに変換することです。これにより、受信中にエラーを訂正するために使用される冗長性が追加されます。このステップでは、メッセージは00 00 00 26 28 9F D4 92 FE 8A CA 0C F3 D1 34 33 88 D0 C2 9C 3D CCに変換されます。

周波数コーディング


ここにあるものはすべて非常に単純です。このステップでは、メッセージをさまざまな周波数の一連のトーンに変換します。ご存知のように、8つのトーン（変調タイプFSK-8）を使用できます。変換後、31406520 00000001 15353274 61112745 36563140 65201575 76054515 70523040 61407642 3140652という形式のシーケンスが得られます。各桁は0〜7のトーン番号です。

トーン間の周波数間隔は6.25Hzです。 1つのトーンの持続時間は0.16秒であり、上記のシーケンスがあるため、WAVファイルを簡単に生成したり、自律ビーコンの場合は周波数シンセサイザを直接制御したりできます。

結論


ご覧のとおり、数学的にFT8は非常に興味深いプロトコルです。WSJT-XソースコードのオリジナルはFortranで記述されていますが、それでもまだわかりません。 C ++のエンコーダーの別のバージョンはgithub.com/kgoba/ft8_libで表示できます。上記のコードスニペットはそこから取得されました。WSJT-Xバージョンをダウンロードすることもできます。無料です。送信機なしでサウンドカードだけをお試しいただけます。http://websdr.ewi.utwente.nl:8901を使用してFT8周波数にチューニングし、仮想オーディオケーブルを介してサウンドをリダイレクトすることもできるので、トランシーバーやr / lコールサインなしで動作中のプログラムを見ることができます。

アマチュアのラジオ界でのFT8の人気については...個人的には、これは私には完全に明確ではありません。私の意見では、そのようなタイプの通信にはすでにロマンスはなく、趣味や創造性よりも産業プロトコルに近いです。しかし、ほとんどは異なる意見を持っているようです。それは記事の最初のグラフに示されています。何百万ものハムラジオファンが間違っていることはありませんか？もちろん、FT8自体は技術的に興味深いものです。おそらく、プロトコルを研究するプロセスは、プロトコルの使用よりもはるかに興味深いものです。ただし、プログラムを少なくとも1回試すことも確かに興味深いです。
