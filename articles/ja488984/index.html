クロスブラウザーツイン


ユージーン：私は導入質問があります。Playwrightプロジェクトに出くわし、「クロスブラウザー」という言葉を目にしたとき、2つのトラックの間にひもを巻いたヴァンダムの広告を思い起こします。

さまざまな言語のさまざまなブラウザーで動作するように、そのようなツールを1つ実行するように感じています-そのような分割もあるはずです。それらがどこにあり、主要な技術的課題がどのように見えるかについて話すのは興味深いです。

Andrei：ええ、本当にたくさんの絡みがあります。私たちにとって、ユーザーとしては多くのことが理解可能で自然に見えますが、実際にはそうではありません。たとえば、リンクをクリックするなどの簡単なことです。

だから私はリンクをクリックしました-そしてリンクを取り、ページをスクロールして興味深い場所に移動します。 URLは変更されたようですが、サイトは同じです。ナビゲーションですか？個人の観点から言えば、それは結構ですが、ブラウザの観点からは、ナビゲーションが異なります。ドキュメントが再作成されない場合の同じページのナビゲーションと、ページがリロードされる場合の新しいドキュメントのナビゲーションです。同じページのナビゲーションは常にWebオートメーションの頭痛の種であり、SPA（シングルページアプリケーション）の時代とHistory APIの大量使用の時代では、生活は容易になりませんでした。

すべてを1つのAPIで制御できるようにすること、形式化を導入して人々がそれがどのように機能するかに驚かないようにすることは非常に困難です。これが最初の課題です。

ただし、APIを思い付くだけでは不十分です。Chromium、WebKit、FirefoxなどのすべてのブラウザーでこのAPIを何らかの形でサポートする必要があります。これを行うために、ブラウザエンジンの「計測」に従事し、ブラウザデバッグプロトコルを開発しています。この作業の結果は、ChromiumのChrome DevTools Protocol 、FirefoxのJuggler、WebKitのWebInspector Protocolです。デバッグプロトコルは、必要なAPIを実装できるように十分強力である必要があります。また、ブラウザ開発者が簡単にサポートできる必要があります。

たとえば、WebKitとChromium内の同じナビゲーションは、技術的に非常に異なる方法で実装されています。私たちはWebKitの担当者と多くのことを話しました。その結果、Chromeとは根本的に異なるインストルメンテーションに同意しましたが、それでもWebオートメーションのニーズは満たしています。したがって、APIと3つの異なるブラウザーをインスツルメントする機能との間のそのような技術的な妥協点を見つけることは本当に難しいですが、非常に興味深いものです。

技術的な課題としては、最も重要なのはクロスプラットフォームのWebKitです。実際には、ウェブキットは主にSafariの形式で、macOS / iOSにのみ存在します。しかし、すべての開発者がMacを持っているわけではないので、私たちのタスクはMac、Linux、WindowsでWebkitビルドをビルドすることでした。これにより、3つのブラウザーすべてで安価なLinuxベースのCIでWebテストを実行できるようになりました。

Vsevolod：WebKitは現在Appleまたは他の誰かをサポートしていますか？

Andrew：私の知る限り、WebKitは主にAppleとIgaliaによって開発されています。

WebKitには、プラットフォームごとにさまざまなポートがあります。 Igaliaは主に、WebKitGTKとWebKitWPEの Linuxポート、およびEpiphanyブラウザに関与しています。

Mac上のWebKitに関しては、これはもちろんAppleです。

Windowsには2つのポートがあります。1つはWebkit-on-Windowsで、これはAppleがWindows上のiTunesに対して行ったようです。このポートは特に開発されていません。2番目のポートはWebKit Cairoです。彼らは主にソニーに従事しています。

どうだった？どのプロトコルが使用されていますか？


Vsevolod：前の記事の読者の1人が、Mozillaには微妙な違いがあると述べています。そこにはDevToolsプロトコルはありませんが、自分のプロトコルですか？少なくとも1年前に、APIをPuppeteerと同じにしたいという連絡がありました。

Andrew：本当にニュアンスがあります:) Playwrightのニーズに合わせて、Jugglerを自分で開発しています。Firefox用のデバッグプロトコルであり、イデオロギー上はChrome DevTools Protocolに似ていますが、そうではありません。何故ですか？

実際、Chrome DevTools Protocolは非常に低レベルのプロトコルです。これにより、プロトコルクライアントはブラウザとのやり取りを最大限に活用できますが、ブラウザの開発と変更に伴い、プロトコルもそれに応じて変化します。 Chrome DevTools Protocolを効果的に使用するには、ブラウザエンジニアとしての知識が必要です。ソースをどこから読むかを理解し、その仕組みを理解する必要があります。残念ながら、Chromiumは大規模で複雑なプロジェクトであり、Chrome DevToolsプロトコルを正しく使用することは困難であり、プロトコルの変更に対応することは永遠の十字架です。 Chrome Devtools Protocolを使用するほとんどのプログラムは、きちんとサポートを受けず、作成後1年半から2年後に、新しいバージョンのChromiumでの作業を停止します。一部には、これによりChrome PuppeteerをChromium Automationの安定したAPIとして作成するように促されました。

Jugglerは、Firefoxの強力なデバッグプロトコルのビジョンであり、維持が簡単であり、Chromiumでのデバッグプロトコルに関するすべての経験を積んでいます。

JugglerをFirefoxチームに見せたとき、多くの熱意がありました。：実際には、Firefoxはすでにいくつかのプロトコルがあるということであるマリオネットセレン、とのニーズのために使用され、リモートデバッグプロトコル - Firefoxのデベロッパーツールの必要性のために使用されています。ただし、これらのプロトコルはさまざまな理由でWebオートメーションのニーズを満たしていません。

私たちはジャグラーの開発とサポートを続けています。

一方、多くの議論がありました。 -最後に、技術的な実現可能性についてのすべての私たちの疑問にもかかわらず、Firefoxチームは、新しいプロジェクトを開始することを決めたリモートプロトコル、その目標は直接FirefoxでChromeのデベロッパーツールプロトコル自体を実装することでしたが。彼らが成功した場合-まあ、それは素晴らしいでしょう！私たちは彼らの実装に移り、それを私たちのプレイライトで使用します。その間、私たちはジャグラーの開発と維持を続けています。

Vsevolod：私たちが理解しているように、あなたはジャグラーとパペティアの両方を行いました。理解したかった：Playwrightへの移行は、Puppeteer API全体がPlaywrightでサポートされることを意味しますか？

Andrei：簡単に言えば、それがすべてです。それ以上の場合、PlaywrightのAPIはPuppeteerのAPIと同じか、それ以上です:)

Puppeteerを始めたとき、私たちは本当に何をしているのか分かりませんでした。私たちはChromeデベロッパーツールを日常的に使用し、ブラウザーツールを使用し、Chrome Devtoolsプロトコルを開発しています。そして、突然そのようなトピック-Webオートメーション！そこで何が起こるかを確認することにしました。いくつかのPhantomJS、いくつかの理解できない世界があります。

Playwrightでは、人々がテストを書くことができるように、APIをシャープにしたいと考えています。テストを簡単で美しく、理解しやすいものにしています。

兄のイメージと肖像で、特に何も研ぎませんでした。彼らはそれを「汎用ブラウザ自動化ライブラリ」と呼び、人々があなたが何を望んでいるのかを理解してそれを使用するようにしました。多くの人がテストにそれを使用していることはすぐに明らかになり、テストではAPIはあまり人間工学的ではありませんでした。人々は多くの過ちを犯しました。

たとえば、ブラウザコンテキストは一般的に1年半後に理解しました（ブラウザコンテキストはPuppeteer v1.5.0に登場しました）。人々は多くのテストを行い、テスト内に別のページが作成され、最初のページは閉じる必要があり、2番目は残り、ページが蓄積し、メモリリークが発生し、Cookieがスタックします。一般的に、トラブル。

プレイライトでは、人形劇の重要な問題の多くを解決し、解決することに尽力しています。たとえば、Playwrightでは、コンテキストはすべての中心であり、コンテキストがなければ、ページを作成することさえできません。すべてのテストは、ブラウザコンテキストの作成で始まり、閉じることで終了します。別の例：Playwrightのコンテキストでは、ポップアップを含むすべてのページに継承される多くの設定を設定できます。 Puppeteerのフレームワークでは、この問題は私たちによって解決されませんでした。

Playwrightでは、人々がテストを書くことができるようにAPIをシャープにしたいと考えています。テストを簡単で美しく、理解しやすいものにしています。

Vsevolod：はい、Pascalを思い起こします=）そして、現在の6か月から1年間の技術チームの主なタスクは何ですか？多分いくつかの基本的な機能はまだいくつかの深刻な問題をカットまたは解決する必要がありますか？

私たちはブラウザ開発者なので、オペレーティングシステムは私たちにとって非常に重要であり、すべてのオペレーティングシステムでテストを実行しています。たとえば、各プラットフォームのWebKitは、独自のネットワークスタックの実装を使用します。ブラウザのヘッドレスモードも、プラットフォームによって実装方法が大きく異なります。これがすべて私たちの責任です。

Vsevolod：Playwrightはエミュレーションデバイスとどのように連携しますか？

Andrew：モバイルエミュレーションは非常にうまく機能します。 Chrome開発者ツールの時代、「デバイスモード」を使用していたときに、それをやり始めました。そこで私たちは多くのことを学び、この経験ですでにPlaywrightの他のブラウザーでエミュレートしました。

Playwrightは、メタビューポートタグなどのさまざまなWebプラットフォーム機能をエミュレートします、ビューポートサイズ、タッチイベント、useragent。エミュレーションをオンにして開いたサイト（iPhoneなど）は、実際のiPhoneと同じように見え、動作します。そして、突然これがどこか間違っている場合、これはバグです。ファイルしてください。修正します。

c継続的インテグレーションを実行する方法


Vsevolod：特定のニュアンスですべてを続けて実行できることはすでに示しました。セレノイドには、セレンには多くのツールがあることを知っています。 100500テストを並行して実行したい場合、プレイライトはどうでしょうか？

Andrei：何をどの程度正確に並列化したいかによります。たとえば、1000個のテストがあり、それらを1つのブラウザーでできるだけ早く実行したい場合、Playwrightにブラウザーコンテキストがあります。コンテキスト内の各テストを発見し、テスト後に、コンテキストを閉じてください。たとえば、jest-playwrightは機能します。

たとえば、同じOS上の2つのブラウザーでテストを並行して実行したい場合、これら2つのブラウザーを並行して起動してテストを実行することを妨げるものはありません。

しかし、たとえば、異なるOSで並行してテストを実行したい場合、異なるOSで同じタスクを実行することが非常に簡単な最新のCI / CDが役立ちます。もちろん、ここでは、Playwrightがこれらの最新のCI / CDで問題なく動作することが重要です。

セレノイドについて具体的に言えば、このシナリオは、テストが私と一緒にローカルで実行され、ブラウザがサービスの形で遠く離れた場所に住んでいる場合、Playwrightで技術的に可能です。ただし、ユーザーのケース自体は、私の意見では、主に過去のものです。

Vsevolod：dockerで起動するには、イメージDockerを作成する必要があることを正しく理解しています。そこからNode.JSとPlaywrightが起動され、すべて準備が整っていますか？

Andrei：はい、そうです。

ウラジミール：これらのブラウザーを毎回インストールすれば、それほど時間はかかりませんか？そして、インストールはどうですか？

Andrew：私たちは、ブラウザーが世界中で高速にダウンロードできるように努めています-これには優れたグローバルCDNがあります。私のオフィスのコンピューターでは、3つのブラウザーすべてが30秒でダウンロードされますが、CIでもすぐに判明し、node_modulesをキャッシュできる場所もたくさんあります。

インストールに関しては、システムにはありませんが、ローカルのもの、ブラウザ付きのバイナリだけが適切な場所に解凍されます。これには多くの時間がかからず、オペレーティングシステムは決して「汚染」しません。

テストでパッチを適用したブラウザに対するユーザーの信頼について


Vsevolod：人々は何がそこにあり、何が揺れるかについて不信を持っています。以前、私が覚えているように、Puppeteerにはブラウザーのバージョンを選択する機能がありました。そして、ここでは、私が理解しているように、さまざまなプラットフォーム（Chromium、WebKit、またはFirefox）のドライバーまたはブラウザーを自分で決定し、自分で置き、常に最新バージョンを使用するソリューションを提供していますか？

Andrew：あなたは環境変数PUPPETEER_CHROMIUM_REVISIONを意味していると思います。 Playwrightにはまだ追加していませんが、必要がないようです。

Playwrightの世界では、ユーザーはいつでも使用したいPlaywrightのバージョンを選択でき、適切なブラウザーが表示されます。ブラウザを更新したい場合は、ライブラリのバージョンを更新してください。このようなシステムにより、バグや不整合、およびユーザーのためのスペースを大幅に削減でき、システム構成のトラブルを減らすことができます。

可能な統合と単一のクロスブラウザプロトコルについて


Vsevolod：PlaywrightへのSeleniumテストの移植があることを確認する方法を考えましたか？

Andrew：Seleniumテストについて話している場合、どの言語を理解する必要があります。しかし、一般的にはそう考えました。 PuppeteerはすべてJavaScriptで作成されており、コミュニティは非常に迅速に登場し、PuppeteerをPython、C＃、Ruby、Rust、その他すべてに対応させました。

同様の状況がプレイライトでも得られます。たとえば、Puppeteer C＃の作者は現在、Playwright C＃の開発に携わっています。そのため、さまざまな言語をサポートするためにコミュニティに主に依存しています。

Vsevolod：他のツールと徐々に統合しますか？私はあなたのスラックチャンネルから素晴らしいツールQAWolfについて学びました個人的には使ったことも聞いたこともありません。このツールは私には興味深く思えました。私が理解しているように、あなたの中では誰かがそれを使って統合しようとしている。アクティブな統合に他のツールはありますか？

Andrew：QAWolfは良い例です。すでに劇作家を使用する大規模なプロジェクトのうち、活字体とVSCodeを呼び出すことができます。CodeceptJSテストフレームワークも最近、Playwrightをそのドライバーの1つとしてサポートしました。

通常、Playwrightに関するプロジェクトを掲載した別のページのショーケースがあります。 Playwrightを使用していて、これを世界と共有したい場合は、PRを送信してこのページに追加してください。

Vsevolod：SeleniumにIDEがあることは知っています。 IDEを使用している人がPlaywrightでテストを簡単に書き始めることができるように、VS Codeまたは別のIDEのプラグインを提出することを考えていますか？人々がより簡単にテスト領域に入ることができるようにするためのプラグインを使用したアプローチを考えていますか？

Andrei：Selenium IDEに関しては、このトピックに興味があり、この分野でいくつかのクールな実験を行っていますが、これまでのところ、土壌を調査しているだけです。さらに、今ではプレイライトに完全に集中しており、全力を注いでいます。

VSCodeのプラグインについては-いいえ、私たち自身はこれを行いませんが、コミュニティに依存しています。他のIDEについても同様です。
