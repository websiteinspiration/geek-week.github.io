La grande division de l'importation: clarifier l'incertitude avec l'importation dans Typescript

Une traduction de l'article a été préparée avant le début du cours "React.js Developer"


Je travaille avec tapuscrit depuis un certain temps, et j'ai eu beaucoup de problèmes à trier ses modules et à conseiller les paramètres, et je dois dire qu'il y a vraiment beaucoup d'incompréhensible autour d'eux. Espace de noms import * as React from 'react', esModuleInteropetc. Par conséquent, comprenons pourquoi tout le battage médiatique a surgi.

Je ne parlerai pas des espaces de noms comme d'un système modulaire en tapuscrit, car l'idée n'était pas la meilleure (surtout compte tenu du vecteur de développement actuel), et personne ne l'utilise actuellement.

Alors, comment étaient les choses avant leur apparition esModuleInterop? Il y avait presque tous les mêmes modules que Babel ou les navigateurs, ainsi que des importations / exportations nommées. Cependant, en matière d'exportations et d'importations par défaut, dactylographié avait sa propre option: il fallait écrire import * as React from 'react'(à la place import React from 'react'), et, bien sûr, ici nous ne parlons pas seulement de réagir, mais de toutes les importations par défaut de commonjs. Comment est-ce arrivé?

Pour comprendre cela, voyons comment fonctionne la compatibilité entre certains modèles commonjset modules es6. Par exemple, nous avons un module qui exporte fooet barcomme clés:

module.exports = { foo, bar }

Nous pouvons importer en utilisant require et destructuring:

const { foo, bar } = require('my-module')

Et appliquez le même principe en utilisant des importations nommées (bien que, pour être honnête, ce ne soit pas une déstructuration):

import { foo, bar } from 'my-module'

Cependant, le modèle le plus courant dans commonjs est const myModule = require('my-module')(car il n'y a pas encore eu de déstructuration), mais comment le faites-vous es6?

Lors du développement de la spécification pour l'importation, l' es6un des aspects importants était la compatibilité avec commonjs, car commonjsbeaucoup de code était déjà écrit dessus. C'est ainsi que l'importation et l'exportation par défaut sont apparues. Oui, le seul objectif était d'assurer la compatibilité avec commonjs, afin que nous puissions écrire import myModule from 'my-moduleet obtenir exactement le même résultat. Cependant, d'après les spécifications, cela n'était pas évident et, en outre, la mise en œuvre de la compatibilité était la prérogative des développeurs du transpilateur. Et ici, le grand schisme vient de se produire: import React from 'react'ou import * as React from 'react'- c'est la question.

Pourquoi le tapuscrit a-t-il choisi ce dernier? Mettez-vous à la place du développeur du transpilateur et demandez-vous quel est le moyen le plus simple de transporter les importations de es6vers commonjs? Supposons que vous ayez l'ensemble d'importations et d'exportations suivant:

export const foo = 1
export const bar = 2
export default () => {}
import { foo } from 'module'
import func from 'module'`

Ainsi, nous utiliserons l'objet jsavec la clé defaultpour l'export par défaut!

module.exports = {
  foo: 1,
  bar: 2,
  default: () => {}
}
const module = require('module')
const foo = module.foo
const func = module.default

Cool, mais qu'en est-il de la compatibilité? Si l'importation par défaut signifie que nous prenons un champ avec un nom default, alors quand nous l'écrirons, import React from 'react'cela signifiera const { default: React } = require('react'), mais cela ne fonctionne pas! Essayez ensuite d'utiliser l'importation avec un astérisque. Désormais, les utilisateurs devront écrire import * as React from 'react'pour accéder au contenu module.exports.

Cependant, il existe une différence sémantique par rapport à commonjs.
Commonjsétait comme javascript ordinaire, pas plus. Juste des fonctions et des objets, sans aucune exigence . En revanche, en import es6, il fait requiredésormais partie de la spécification, doncmyModuledans ce cas, ce n'est pas seulement un objet javascript normal, mais ce qu'on appelle un espace de noms (à ne pas confondre avec les espaces de noms en tapuscrit), qui, en conséquence, possède certaines propriétés. L'un d'eux est que l'espace de noms ne peut pas être appelé. Et quel est le problème ici, vous pouvez demander?

Essayons un autre modèle commonjs, avec une fonction en tant qu'export:

module.exports = function() { // do something }

Nous pouvons l'utiliser requireet l'exécuter:

const foo = require('my-module')
foo()

Bien que si vous essayez de le faire dans un environnement conforme aux spécifications avec les modules ES6, vous obtiendrez une erreur:

import * as foo from 'my-module'
foo() // Error

En effet, l'espace de noms n'est pas le même qu'un objet javascript, mais une structure distincte qui stocke chaque exportation es6.

Mais Babel a bien compris et a fourni une option de compatibilité où nous pouvons écrire import React from 'react»et cela fonctionnera. Au cours de la transpilation, il marque chaque module es6 avec un drapeau spécial module.exports, afin que nous comprenions que si le drapeau est vrai, il est renvoyé module.exportset s'il est faux (par exemple, s'il s'agit d'une bibliothèque commonjsqui n'a pas été transposée), alors nous devrons envelopper l'exportation actuelle { default: export }afin que nous pourrait utiliser à chaque fois default(jetez un oeil ici ).

Tapuscript a fait son chemin à travers les importations avec des astérisques, mais a finalement abandonné et ajouté une optionesModuleInteropau compilateur. En général, cette option fait la même chose que babel, et si vous l'activez, vous pouvez écrire l'importation régulière en tant que import React from 'react', et typescript comprendra tout.

Le problème est que bien qu'il soit activé par défaut dans les nouveaux projets (au moment de l'exécution tsc --init), il ne fonctionnera pas pour les projets existants (même si vous effectuez une mise à niveau vers TypeScript 3) car il n'a pas de compatibilité descendante. Ainsi, vous devrez réécrire les importations d'astérisques inutiles dans les importations par défaut. React traitera cela normalement, car il s'agit toujours d'une collection d'exportations nommées, mais pas par exemple avec un appel d'espace de noms. Mais n'ayez pas peur si tout est en ordre avec le typage d'exportation (et pour la plupart, tout est en ordre, car beaucoup d'entre eux sont automatiquement corrigés), TypeScript 3 vous permettra de convertir rapidement l'importation avec des astérisques en standard.

Par conséquent, je préconise vraiment l'utilisation de l'option esModuleInterop, ne serait-ce que parce qu'elle vous permettra non seulement d'écrire moins de code et de la rendre plus facile à lire (et ce ne sont pas seulement des mots, par exemple, le cumul ne vous permettra pas d'utiliser des importations avec des astérisques comme celui-ci), mais réduira également les désaccords entre les communautés dactylographié et babel.

Attention: il y avait une option enableSyntheticDefaultImportsqui fermait le compilateur lorsqu'il essayait de se plaindre d'une importation incorrecte par défaut, nous avions donc besoin de notre propre moyen pour assurer la compatibilité avec commonjs(par exemple,WebpackDefaultImportPlugin), mais c'était problématique, car, par exemple, si vous avez des tests, vous devez toujours garantir cette compatibilité. Veuillez noter que esModuleInteropcomprend l'importation synthétique par défaut uniquement si votre cible est <=ES5. Par conséquent, si vous activez cette option et que les compilateurs continuent de s'en plaindre import React, comprenez quel objectif vous poursuivez, et peut-être que l'inclusion d'importations par défaut sera votre option (ou redémarrez vscode / webstorm, qui sait).

J'espère que mon explication a un peu clarifié la situation, mais si vous avez encore des questions, vous pouvez me les poser sur twitter !
