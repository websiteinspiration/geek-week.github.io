AssemblyScriptの最初の紹介

WebAssembly（Wasm）テクノロジーのサポートは、比較的最近になってブラウザーに登場しました。しかし、このテクノロジはWebの機能を十分に拡張できるため、通常デスクトップとして認識されているアプリケーションをサポートできるプラットフォームになります。

WebAssemblyの習得は、Web開発者にとって困難な作業になる可能性があります。ただし、AssemblyScriptコンパイラは状況を改善できます。


本日の翻訳を公開しているこの記事の著者は、まずWebAssemblyが非常に有望なテクノロジーである理由について話し、次に、AssemblyScriptがWasmの可能性を引き出すのにどのように役立つかを見ていきます。

Webアセンブリ


WebAssemblyは、ブラウザーの低レベル言語と呼ぶことができます。開発者は、JavaScript以外のものにコンパイルするコードを作成することができます。これにより、Webページに含まれるプログラムは、さまざまなプラットフォームのネイティブアプリケーションとほぼ同じ速度で動作します。このようなプログラムは、制限された安全な環境で実行されます。

すべての主要なブラウザー（Chrome、Firefox、Safari、およびEdge）の開発を担当するチームの代表が、WebAssembly標準の作成に関与しました。彼らは、2017年の初めにシステムアーキテクチャについて合意しました。現在、上記のすべてのブラウザがWebAssemblyをサポートしています。実際、このテクノロジーは約87％のブラウザーで使用できます。

WebAssemblyコードはバイナリ形式で存在します。つまり、そのようなコードは、同様のJavaScriptコードよりも小さく、ロードも高速です。さらに、Wasmコードはテキスト形式で表示できるため、ユーザーはそれを読んで編集できます。

WebAssembly標準が最初に登場したとき、一部の開発者は、JavaScriptに代わってWebの主要言語になる可能性があると考えていました。しかし、WebAssemblyは、既存のWebプラットフォームとうまく統合する新しいツールとして最もよく見られています。これは彼の優先目標の 1つです。

この言語が長期間にわたって成功裏に使用されてきたJavaScriptを置き換える代わりに、WebAssemblyはWeb開発者に新しい興味深い機会を提供します。確かに、WasmコードはDOMに直接アクセスできないため、既存のほとんどのWebプロジェクトは引き続きJavaScriptを使用します。この言語は、長年にわたる開発と最適化のおかげで、すでに非常に高速です。また、WebAssemblyには独自のアプリケーションがあります。

  • ゲーム。
  • 科学計算、可視化、シミュレーション。
  • CADアプリケーション。
  • 画像や動画の編集。

これらのWasmの用途はすべて、それぞれのアプリケーションが通常デスクトップと見なされるものによって統一されています。しかし、WebAssemblyを使用すると、ネイティブに近いレベルのパフォーマンスに到達できるため、多くの同様のアプリケーションをWebプラットフォームを使用して実装できるようになりました。

WebAssemblyは、既存のWebプロジェクトを利用できます。その一例がFigmaプロジェクトです。 Wasmアプリケーションのおかげで、このプロジェクトのロード時間が大幅に改善されました。 Webサイトが重い計算を実行するコードを使用する場合、このWebサイトのパフォーマンスを向上させるために、そのようなコードのみをWebAssemblyアナログに置き換えることは理にかなっています。

独自のプロジェクトでWebAssemblyを使用してみてください。この言語は、すぐに学習して書くことができます。しかし、それにもかかわらず、WebAssemblyはもともと他の言語のコンパイルターゲットとして開発されました。それはして設計された CおよびC ++のために良いサポートを。Wasm 実験的サポートはGo 1.11で登場しました。Rustでの Wasmアプリケーションの作成にも多大な労力が費やされています。

ただし、Web開発者がWebAssemblyを使用するためだけにC、C ++、Go、またはRustを学習することを望まない可能性は十分にあります。彼らは何をしますか？この質問への答えは、AssemblyScriptを与えることができます。

AssemblyScript


AssemblyScriptは、TypeScriptコードをWebAssemblyコードに変換するコンパイラーです。 TypeScriptは、Microsoftによって開発された言語です。これはJavaScriptのスーパーセットであり、改良された型サポートとその他のいくつかの機能を備えています。 TypeScriptはかなり人気のある言語になっています。 AssemblyScriptは、限られた一連のTypeScript構成体のみをWasmに変換できることに注意してください。つまり、TypeScriptに慣れていない人でも、AssemblyScriptが理解できるコードを作成するのに十分なレベルで、この言語をすばやく習得できます。

同時に、TypeScriptがJavaScriptに非常に類似しているという事実を考えると、AssemblyScriptテクノロジーにより、Web開発者は完全に新しい言語を学ぶ必要なく、Wasmモジュールをプロジェクトに簡単に統合できます。

例


最初のAssemblyScriptモジュールを作成しましょう。これから説明するコードはすべてGitHubにあります。WebAssembly をサポートするには、少なくともNode.js 8 が必要です。

新しいディレクトリを作成し、npmプロジェクトを初期化して、AssemblyScriptをインストールします。

mkdir assemblyscript-demo
cd assemblyscript-demo
npm init
npm install --save-dev github:AssemblyScript/assemblyscript

AssemblyScriptは、プロジェクトのGitHubリポジトリから直接インストールする必要があることに注意してください。AssemblyScriptは、開発者がまだ広く使用できる状態にあるとはまだ考えていないため、npmではまだ公開されていません。

AssemblyScriptに含まれているコマンドを使用して補助ファイルを作成しますasinit。

npx asinit .

これでscripts、セクションpackage.jsonは次の形式になります。

{
  "scripts": {
    "asbuild:untouched": "asc assembly/index.ts -b build/untouched.wasm -t build/untouched.wat --sourceMap --validate --debug",
    "asbuild:optimized": "asc assembly/index.ts -b build/optimized.wasm -t build/optimized.wat --sourceMap --validate --optimize",
    "asbuild": "npm run asbuild:untouched && npm run asbuild:optimized"
  }
}

index.jsプロジェクトのルートフォルダーにある ファイルは次のようになります。

const fs = require("fs");
const compiled = new WebAssembly.Module(fs.readFileSync(__dirname + "/build/optimized.wasm"));
const imports = {
  env: {
    abort(_msg, _file, line, column) {
       console.error("abort called at index.ts:" + line + ":" + column);
    }
  }
};
Object.defineProperty(module, "exports", {
  get: () => new WebAssembly.Instance(compiled, imports).exports
});

これにより、requireコマンドを使用して、コードにWebAssemblyモジュールを含めることができます。つまり、通常のJavaScriptモジュールが接続されるのと同じ方法です。

フォルダassemblyにはファイルが含まれていますindex.ts。 AssemblyScriptルールに従って記述されたソースコードがあります。自動生成されるボイラープレートコードは、2つの数値を追加する単純な関数です。

export function add(a: i32, b: i32): i32 {
  return a + b;
}

おそらく、似たような関数のシグネチャがのようになることを期待していましたadd(a: number, b: number): number。したがって、通常のTypeScriptで記述されているかどうかを確認します。ただし、ここではtypeの代わりにtypeをnumber使用していi32ます。これは、AssemblyScriptコードがTypeScriptの一般的な数値型ではなく、整数と浮動小数点数に特定のWebAssembly型を使用するために発生します。 プロジェクトを組み立てましょう：



npm run asbuild

build以下のファイルがフォルダ内に表示される必要があります。

optimized.wasm
optimized.wasm.map
optimized.wat
untouched.wasm
untouched.wasm.map
untouched.wat

アセンブリには最適化されたバージョンと通常のバージョンがあります。アセンブリの各バージョンは、バイナリ.wasmファイル、マップマップ .wasm.map、および.watファイル内のバイナリコードのテキスト表現を提供します。Wasmコードのテストプレゼンテーションはプログラマーを対象としていますが、このファイルを見ることさえしません。実際、AssemblyScriptを使用する理由の1つは、Wasmコードを操作する必要がなくなることです。

次に、Node.jsをREPLモードで実行し、コンパイルされたWasmモジュールが通常のJSモジュールと同じように使用できることを確認します。

$ node
Welcome to Node.js v12.10.0.

Type ".help" for more information.

> const add = require('./index').add;
undefined
> add(3, 5)
8

一般に、Node.jsでWebAssemblyテクノロジーを使用するために必要なのはこれだけです。

プロジェクトにオブザーバースクリプトを装備する


開発中にモジュールを変更するときにモジュールを自動的に再構築するには、onchangeパッケージの使用をお勧めします。実際のところ、AssemblyScriptには、ファイルの変更を監視するための独自のシステムがまだありません。onchangeパッケージをインストールします。

npm install --save-dev onchange

package.jsonスクリプトに 追加しますasbuild:watch。このフラグは -iコマンドに含まれているため、イベントが発生する前に、スクリプトが呼び出されたときにビルドプロセスが1回開始されます。

{
  "scripts": {
    "asbuild:untouched": "asc assembly/index.ts -b build/untouched.wasm -t build/untouched.wat --sourceMap --validate --debug",
    "asbuild:optimized": "asc assembly/index.ts -b build/optimized.wasm -t build/optimized.wat --sourceMap --validate --optimize",
    "asbuild": "npm run asbuild:untouched && npm run asbuild:optimized",
    "asbuild:watch": "onchange -i 'assembly/**/*' -- npm run asbuild"
  }
}

これで、コマンドを常に実行するのではなく、asbuild一度だけ実行しasbuild:watchます。

パフォーマンス


Wasmコードのパフォーマンスレベルを評価する簡単なテストを作成します。WebAssemblyの主な範囲は、プロセッサを集中的に使用するタスクを解決することです。たとえば、これらはある種の「重い」計算です。特定の数が素数かどうかを調べる関数を作成してみましょう。

同様の関数の基本的なJS実装を以下に示します。非常に単純に配置されており、ブルートフォースで数をチェックしますが、大量の計算を実行するため、私たちの目的には適しています。

function isPrime(x) {
    if (x < 2) {
        return false;
    }

    for (let i = 2; i < x; i++) {
        if (x % i === 0) {
            return false;
        }
    }

    return true;
}

AssemblyScriptコンパイラー用に作成された同様の関数は、ほとんど同じに見えます。主な違いは、コード内の型注釈の存在です。

function isPrime(x: u32): bool {
    if (x < 2) {
        return false;
    }

    for (let i: u32 = 2; i < x; i++) {
        if (x % i === 0) {
            return false;
        }
    }

    return true;
}

コードのパフォーマンスを分析するために、Benchmark.jsパッケージを使用します。インストールしてください：

npm install --save-dev benchmark

ファイルを作成しますbenchmark.js。

const Benchmark = require('benchmark');

const assemblyScriptIsPrime = require('./index').isPrime;

function isPrime(x) {
    for (let i = 2; i < x; i++) {
        if (x % i === 0) {
            return false;
        }
    }

    return true;
}

const suite = new Benchmark.Suite;
const startNumber = 2;
const stopNumber = 10000;

suite.add('AssemblyScript isPrime', function () {
    for (let i = startNumber; i < stopNumber; i++) {
        assemblyScriptIsPrime(i);
    }
}).add('JavaScript isPrime', function () {
    for (let i = startNumber; i < stopNumber; i++) {
        isPrime(i);
    }
}).on('cycle', function (event) {
    console.log(String(event.target));
}).on('complete', function () {
    const fastest = this.filter('fastest');
    const slowest = this.filter('slowest');
    const difference = (fastest.map('hz') - slowest.map('hz')) / slowest.map('hz') * 100;
    console.log(`${fastest.map('name')} is ~${difference.toFixed(1)}% faster.`);
}).run();

これはnode benchmark、コンピューターでコマンドを実行した後に取得できたものです。

AssemblyScript isPrime x 74.00 ops/sec ±0.43% (76 runs sampled)
JavaScript isPrime x 61.56 ops/sec ±0.30% (64 runs sampled)
AssemblyScript isPrime is ~20.2% faster.

ご覧のとおり、アルゴリズムのAssemblyScript実装は、JS実装よりも20％高速でした。ただし、このテストはマイクロベンチマークであることに注意してください。その結果にあまり依存しないでください。

AssemblyScriptプロジェクトのパフォーマンス調査のより信頼できる結果を見つけるために、これとこのベンチマークを確認することをお勧めします。

WebページでのWasmモジュールの使用


WasmモジュールをWebページで使用してみましょう。まずindex.html、次の内容のファイルを作成します。

<!DOCTYPE html>
<html>
    <head>
        <meta charset="utf-8" />
        <title>AssemblyScript isPrime demo</title>
    </head>
    <body>
        <form id="prime-checker">
            <label for="number">Enter a number to check if it is prime:</label>
            <input name="number" type="number" />
            <button type="submit">Submit</button>
        </form>

        <p id="result"></p>

        <script src="demo.js"></script>
    </body>
</html>

demo.jsコードを以下に示す ファイルを作成します。WebAssemblyモジュールをロードするには多くの方法があります。最も効率的なのは、WebAssembly.instantiateStreaming関数を使用して、それらをストリーミングモードでコンパイルおよび初期化することです。ステートメントが実行されなかった場合に呼び出されるabort関数を再定義する必要があることに注意してください。

(async () => {
    const importObject = {
        env: {
            abort(_msg, _file, line, column) {
                console.error("abort called at index.ts:" + line + ":" + column);
            }
        }
    };
    const module = await WebAssembly.instantiateStreaming(
        fetch("build/optimized.wasm"),
        importObject
    );
    const isPrime = module.instance.exports.isPrime;

    const result = document.querySelector("#result");
    document.querySelector("#prime-checker").addEventListener("submit", event => {
        event.preventDefault();
        result.innerText = "";
        const number = event.target.elements.number.value;
        result.innerText = `${number} is ${isPrime(number) ? '' : 'not '}prime.`;
    });
})();

次に、静的サーバーパッケージをインストールします。この機能を使用するにはWebAssembly.instantiateStreaming、MIMEタイプを使用してモジュールにサービスを提供する必要があるため、サーバーが必要application/wasmです。

npm install --save-dev static-server

package.json適切なスクリプトに 追加します。

{
  "scripts": {
    "serve-demo": "static-server"
  }
}

次に、コマンドnpm run serve-demoを実行し、ローカルホストのURLをブラウザーで開きます。フォームに特定の数値を入力すると、それが単純かどうかを確認できます。現在、AssemblyScriptの開発では、コードの記述からNode.jsやWebページでのコードの使用まで、長い道のりを歩んできました。

概要


WebAssembly、つまりAssemblyScriptは、なんとかしてサイトを魔法のように高速化することができません。しかし、ワズムは絶対にすべてを加速することを任されたことはありません。このテクノロジーは、新しい種類のアプリケーションのWebへの道を開くという点で注目に値します。

AssemblyScriptについても同様のことが言えます。このテクノロジーにより、多数の開発者がWebAssemblyに簡単にアクセスできるようになります。JavaScriptに近い言語でコードを作成するときに、WebAssemblyの機能を使用して複雑な計算問題を解決できます。
