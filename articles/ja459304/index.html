ナンバー3。* ngForで生成されたリストを更新するときのコード実行（要素がDOMに表示された後）


スクロール可能なリストを整理するための興味深いカスタムディレクティブがあるとします。これを、Angularディレクティブを使用して作成されたリストに適用します*ngFor。

<div *ngFor="let item of itemsList; let i = index;"
     [customScroll]
     >
  <p *ngFor="let item of items" class="list-item">{{item}}</p>
    
</div>

通常、このような場合、リストを更新するときに、リストでscrollDirective.update発生した変更を考慮して、スクロール動作を構成するためにそのようなものを呼び出す必要があります。

これはフックを使用して行うことができるように見えるかもしれませんngOnChanges：

class SomeComponent implements OnChanges {
  @Input() itemsList = [];
  
  @ViewChild(CustomScrollDirective) scroll: CustomScrollDirective;

  ngOnChanges(changes) {
    if (changes.itemsList) {
      this.scroll.update();
    }
  }
...
}

確かに、ここで問題に直面しています。フックは、ブラウザーが更新されたリストを表示する前に呼び出されます。その結果、リストをスクロールするためのディレクティブのパラメーターの再計算が正しく実行されません。仕事が終わった

直後に電話をかけるには*ngFor？

これを行うには、次の3つの簡単な手順に従います。

▍ステップ番号1


*ngFor（#listItems） が適用される要素へのリンクを配置します。

<div [customScroll]>
    <p *ngFor="let item of items" #listItems>{{item}}</p>
</div>

▍ステップ番号2


Angularデコレーターを使用して、これらの要素のリストを取得しますViewChildren。タイプエンティティを返しますQueryList。

▍ステップ番号3


クラスにQueryListは、リストが変更されるたびにイベントをスローする読み取り専用の変更プロパティがあります。

class SomeComponent implements AfterViewInit {
  @Input() itemsList = [];
  
  @ViewChild(CustomScrollDirective) scroll: CustomScrollDirective;
  @ViewChildren('listItems') listItems: QueryList<any>;
  private sub: Subscription;
  
  ngAfterViewInit() {
    this.sub = this.listItems.changes.subscribe(() => this.scroll.update())
  }

}

ここでは、問題の本質について質問があるかもしれません。パラメータを取得しました。すべてが期待どおりに機能します。

上記のブラウザのスクリーンショットと、コンソールに表示される内容を詳しく見てください。ここでは、オブジェクトqueryParamsが2回発行されていることがわかります。最初のオブジェクトは空であり、Angularルーターの初期化プロセス中に発行されます。その後、クエリパラメータを含むオブジェクトを取得します（この場合は-{accessToken: ‘someTokenSequence’}）。

問題は、URLに要求パラメーターがない場合、ルーターが何も返さないことです。つまり、最初の空のオブジェクトを発行した後、パラメータがないことを示す可能性のある、2番目の空のオブジェクトも発行されません。


パラメータなしでリクエストを実行すると、2番目のオブジェクトは発行されません。

その結果、コードがリクエストデータを受信できる2番目のオブジェクトを予期している場合、URLにリクエストパラメータがなかった場合、そのオブジェクトは起動されません。

このコンポーネントは2つの問題を解決します。

  1. 要素の配列を表示します（この操作は1回実行されると想定されています）。また、メソッドを呼び出すことにより、画面に表示される内容をフォーマットしますformatItem。
  2. マウスの座標が表示されます（この値は明らかに非常に頻繁に更新されます）。

このコンポーネントにパフォーマンスの問題があるとは思われません。したがって、すべての手続きに準拠するためにのみパフォーマンステストを実行します。ただし、このテスト中にいくつかの奇妙なことが表示されます。


多くのformatItem呼び出しとかなり大きなプロセッサ負荷。

問題は何ですか？しかし、実際には、Angularがテンプレートを再描画すると、テンプレートからすべての関数（この場合は関数formatItem）が呼び出されます。その結果、テンプレート関数で重い計算が実行されると、プロセッサに負担がかかり、ユーザーが対応するページを認識する方法に影響します。

それを修正するには？formatItem事前に計算を行い、準備が整ったデータをページに表示するだけで十分です。

// home.component.html
<div class="wrapper" (mousemove)="mouseCoordinates = {x: $event.x, y: $event.y}">
  <div *ngFor="let item of displayedItems">
    <span>{{item}}</span>
  </div>
</div>

{{mouseCoordinates | json}}


// home.component.ts
@Component({
    selector: 'app-home',
    templateUrl: './home.component.html',
    styleUrls: ['./home.component.sass']
})
export class HomeComponent implements OnInit {

    items = [1, 2, 3, 4, 5, 6];
    displayedItems = [];

    mouseCoordinates = {};

    ngOnInit() {
        this.displayedItems = this.items.map((item) => this.formatItem(item));
    }

    formatItem(item) {
        console.log('formatItem');
        const t = Array.apply(null, Array(5)).map(() => 1);
        return item + '%';
    }
}

これで、パフォーマンステストははるかに適切に見えます。


formatItemの呼び出しが6回で、プロセッサの負荷が低いため
