前書き


最初の部分では、問題の説明とウィッシュリストがどのように変容したかについて書きました。結局、OpenVPNを使用することにしましたが、すべてをDockerコンテナーで実行することにしたため、それほど単純ではないことがわかりました。

後でもう一度すべてをやり直したため、外部VPSを拒否したことをすぐに言っておく必要があります。ただし、すべてがコンテナーに含まれているので、その過程で興味深い機能をいくつか見つけました。

取り付け


要点のみ説明します。

ioBroker


docker run -d --name iobhost  --net=host -v /opt/iobroker/:/opt/iobroker/ --device=/dev/ttyACM0 --env-file /opt/ioBroker_env.list --restart=always buanet/iobroker:latest

私はMiHomeゲートウェイを持っているので、センサーが接続されています。いくつかのシナリオがセットアップされていますが、まだ壊したくないので、ioBrokerをそれに接続しました。彼はセンサーを見て、Zigbeeスティックにそれらを再バインドする必要はありませんでした（ただし、いくつかあり、いくつかのボタンがそれに接続されています）。

ここでは、ioBrokerがMiHomeゲートウェイに接続し、-net = hostパラメーターで開始するために、それら。これはホストインターフェースを使用するため、コンテナーに転送するポートを指定する必要はありません。これがないと、マルチキャストで作業しているため、ゲートウェイを見ることができませんでした。

Zigbee USBスティックをコンテナーに転送するには、パラメーター--device = / dev / ttyACM0が必要です。また、/ opt / ioBroker_env.listにUSBDEVICES = "/ dev / ttyACM0"という行を追加する必要がありました。この行は、コンテナの最初の起動時に空のディレクトリを見つけて初期構成を開始するときに存在していることが重要です。

もちろん、後で構成することもできますが、追加の体の動きを行う必要があります。

MQTTサーバー


外部VPSでは、日食蚊を発射しました。まず、Let's暗号化証明書を発行して、TLSをセットアップしました。次に、クライアントは証明書を提示する必要があり、次に名前とパスワード（ブルートフォースからの保護）を提示する必要があると判断しました。そこで、クライアントが証明書を書き込めるように、自己署名にそれをやり直しました。

Openvpn


Docker Hubで人気のある画像を使用しました。サーバー（VPS）kylemanna / openvpnの場合。クライアント（ioBrokerがインストールされているホームサーバー）の場合-ekristen / openvpn-client。

VPSサーバー

例のアドレスは、ホームネットワークに192.168.6.0を使用し、ループバックに192.168.16.0を使用します。どこでも間違えないように頑張ります。

でopenvpn.conf追加

私は特別に192.168.16.0/25からループバックを作成し、192.168.16.128 / 25からクライアントにアドレスを提供します。これにより、iptablesの解決ルールを単一のグリッド192.168.16.0/24で構成できるようになります。

したがって、VPSには192.168.16.1のループバックがあり、mqttを持っています。

ホームサーバーには192.168.16.6があります。 iobroker、別名ホームクライアントのdnsがあり、ホームネットワークから、またはVPNを介して接続するためにいくつかのドメイン名を再定義します。
ネットワークから彼の「実際の」IPを登録するというアイデアが至る所にありました。 ccd / iobrokerと入力して、

しかし、これが自宅のラップトップのWiFiインターフェース上のネットワークからのアドレスであり、多くの場合それがデフォルトゲートウェイであるという事実から明らかに、グリッチがありました。スマートフォン付き。そして、私はホームネットワークがアクティブなクライアントの有無にかかわらず動作することを望んでいました。そして、私はクライアントを必死に切断する必要はなく、家に帰って他のリソースにアクセスできました。

したがって、VPNがアクティブかどうかに関係なく、常にループバックと対話するようにこのサーバーとそのコンテナーを構成しました。そして、同じアドレスが扱われていました。

したがって、ccd / iobroker

iroute 192.168.16.6 255.255.255.255

まあ、すべてのVPNクライアントは、VPNを通じて192.168.16.0/24ネットワークにアクセスできることを知っています。彼らが192.168.16.1（ループバックVPS）にパケットを送信する場合、パケットは暗号化され、openvpnコンテナーに入り、復号化され、デフォルトルートは172.17.0.1（デフォルトコンテナーのデフォルトゲートウェイ）に行き、ホストに到達し、すべてが正常です。

しかし、VPSホストからVPNクライアントに「ping」したり、アドレス192.168.16.6（OpenVPNコンテナー内のVPNトンネル上の一時IPではない）でホームサーバーにアクセスするにはどうすればよいでしょうか。

もちろん、メッシュ192.168.16.0はOpenVPNコンテナーにルーティングする必要があります。もちろん、172.17.0.3であることがわかります。しかし、いつかは変わるかもしれません。

OpenVPNがコンテナーではなくサーバーに直接デプロイされた場合、すべてが単独で機能します。そして、私はそれを狡猾に行わなければなりませんでした。私は、システムの最新版によって作成されたスクリプトを作成し、それに入れます。

ipaddr=$(docker inspect -f '{{.NetworkSettings.IPAddress}}' vpn-client)
route add -net 192.168.16.0/24 gw $ipaddr

それら。docker inspectを使用して、実行中のコンテナーのIPアドレスを確認し、通常の方法でグリッドにルーティングします。

通常のrc.localとは異なり、ググる必要がありましたが、実際には、最後に実行するスクリプトを作成する方法を説明しました。簡単な手順を

次に示します。/etc/systemd/system/custom.targetファイルを作成します。

[Unit] 
Description=My Custom Target
Requires=multi-user.target
After=multi-user.target
AllowIsolate=yes

ファイル/etc/systemd/system/last_command.serviceを作成します 

[Unit]
Description=My custom command
After=multi-user.target
[Service]
Type=simple
ExecStart=/usr/local/bin/my_last_command.sh
[Install]
WantedBy=custom.target

/etc/systemd/system/custom.target.wants ディレクトリを作成します

ln -s /etc/systemd/system/last_command.service \   /etc/systemd/system/custom.target.wants/last_command.service
systemctl daemon-reload
systemctl set-default custom.target

再起動を待たずにすぐに開始したい場合：systemctl isolate custom.target
再起動後、ExecStartで説明されている/usr/local/bin/my_last_command.shファイルが最後に起動されます。

イプテーブル


mqttサーバーでは、2つのポートが発生しています。TLSと認証を備えた8883で、インターネットからリモートセンサーにアクセスできます。はい、私自身もMQTT Explorerに接続して、何がどのように接続されているかを確認できます。

1883はすでにTLSなしで、ユーザー名とパスワードのみが必要です。 TLSが方法を知らない自宅のSonoff rfBridgeに必要です。

これは恐ろしいことではありません。家からのトラフィックは、192.168.16.0のデフォルトゲートウェイであるiobrokerを使用してサーバーに送信され、OpenVPNなどを使用してコンテナーにパケットを転送するためです。ただし、内部からのみポート1883へのアクセスを許可する必要があります。それら。 iptables。

標準的なアプローチは、どこからでもこのポートへのアクセスを拒否し、内部ネットワークからのアクセスを許可するルールに従うことです。

iptables -I INPUT -p tcp -m tcp --dport 1883 -j DROP
iptables -I INPUT -s 172.17.0.0/24 -p tcp -m tcp --dport 1883 -j ACCEPT

グリッドはここで172.17.0.0と示されています。これは、OpenVPNから「ネイバー」に移動するため、元のコンテナからではなく、非表示NAT（rfBridgeがWiFiネットワークからのものであるiobrokerコンテナー）として移動するためです。

そして、それはうまくいきます。しかし、ニュアンスがあります。私のポート1883はmqttコンテナーに転送されます。そして、結局のところ、iptablesは最初にDOCKER-USERチェーンを満たします。

つまり、このルールでは、ポート1883へのアクセスがブロッキングルールに許可されました。また、インターネットからも簡単に接続できます。

DOCKER-USERチェーンでブロッキングルールを作成する必要があります！

iptables -I DOCKR-USER -p tcp -m tcp --dport 1883 -j DROP
iptables -I INPUT -s 172.17.0.0/24 -p tcp -m tcp --dport 1883 -j ACCEPT

ただし、何らかの理由で内部ネットワークからのアクセスを許可する下の方には、INPUTが必要です。

ホームサーバー


本体は同じです。ただし、彼はクライアントでしたが、WiFiネットワーク（rfBridge）からVPS上のmqttにトラフィックをルーティングする必要がありました。それら。トラフィックの動き：

rfBridge（192.168.6.8）-> iobrokerホスト（192.168.6.6）-> vpn-clientコンテナー（172.17.0。？）-> VPS上のopevpnコンテナー->ループバックVPS（192.168.16.1）-> mqttコンテナー（ポート1883）

クライアントとループバックのあるグリッドに「転送」パケットを許可します。

iptables -A FORWARD -d 192.168.16.0/24 -j ACCEPT

コンテナの詳細に進みます。

タスク1


覚えているように、ループバックを介して構成したサブシステム間の相互作用。それら。vpn-clientからの192.168.16.6上のパケットは、VPNトンネルではなくホスト（172.17.0.1）に向かう必要があります。

実行中のコンテナでこのコマンドを実行すると、すべてが機能します。再起動後、これは忘れられますが、iobroker.ovpn構成ファイルで指定できます
route 192.168.16.6 255.255.255.255 172.17.0.1。

そして、openvpnは、コンテナーの起動時にこのルートをインストールします。これは標準的な方法で簡単に解決できます。

タスク2


ホームネットワーク192.168.6.0（rfBridgeなど）からのパケットはiobrokerホストに送信され、vpn-clientコンテナーに転送されます。

ただし、私は意図的に192.168.6.0ネットワークを暗号化ドメインに含めないでください。VPSのOpenVPNコンテナーは、このネットワークをどうするかを認識していません。明白な解決策は、VPS上のパケットがそのアドレスから来るように、vpn-client内にNATを作成することです。しかし、ニュアンスがあります。コンテナーを再起動した後、必要なiptablesコマンドを保存する方法は？ Iptables-persistentを配置するのは簡単ではありません。

もちろん、添加剤を入れた新しい容器を組み立てることもできます。しかし、アップグレードの手順がより複雑になるので、私はしたくありません。 「最新のものを殺して起動し、マウントされたフォルダーから設定をプルアップする」代わりに、アセンブリを開始する必要があります...それは私がコンテナと通信しているわけではありません。

そのため、「コンテナーの起動後、コンテナー内でiptablesにNATを実行するように指示するコマンドを強制的に実行する」ことを決定しました。私はこのためにチームを使いました
docker events --filter "container=vpn-client" --filter "event=start"。

彼女はハングし、フィルターで指定されたイベントを待ちます。私の場合、コンテナを起動します。その後、docker execを介して、ホストから必要なコマンドを実行します。

これを行うには、VPSと同様に、/ usr / local / bin / my_last_command.shを構成します

#!/bin/bash
cont="vpn-client"
ipaddr=$(docker inspect -f '{{.NetworkSettings.IPAddress}}' $cont)
route add -net 192.168.16.0/24 gw $ipaddr
for (( ; ; ))
do
  docker events --filter "container=$cont" --filter "event=start"
  docker exec -it $cont iptables -t nat -I POSTROUTING -s 192.168.16.0/255.255.255.0 -j MASQUERADE
  docker exec -it $cont iptables -t nat -I POSTROUTING -s 192.168.6.0/255.255.255.0 -j MASQUERADE
done

dockerイベントの出力が、コマンドを実行するawk入力に渡される例があります。しかし、イベントの前に「ハング」してコマンドを実行し、再びイベントを待つ方が簡単に思えました。
