「ジャッカル」：フロントエンドを絞る

こんにちは！私はTinkoff BusinessのプラットフォームチームのリーダーであるVanyaです。

私のお気に入りの娯楽は、[DevTools]タブを開いて、ウェブサイトのアーティファクトの重さを確認することです。この記事では、プラットフォームのフロントエンドチームの助けを借りて、サイトのコードを変更せずに1日でアプリケーションのウェイトを30％削減した方法について説明します。トリックや登録はありません-nginx、docker、node.js（オプション）のみです。




何のために


現在、フロントエンドアプリケーションは非常に重くなっています。収集されたアーティファクトは、2〜3 MB以上になる場合があります。ただし、圧縮アルゴリズムはユーザーの助けになります。

最近まで、1992年に世界に導入されたGzipのみを使用していました。これはおそらくWebで最も人気のある圧縮アルゴリズムであり、IE 6を超えるすべてのブラウザーでサポートされています

。Gzipの圧縮レベルは1から9（より効率的）まで変化し、オンザフライまたは静的に圧縮できることを思い出してください。

  • 「オンザフライ」（動的）-アーティファクトは組み立て後に受け取った形式で保存され、圧縮はクライアントへの配信中に行われます。私たちのケースでは、nginxレベルです。
  • 静的-アーティファクトはアセンブリ後に圧縮され、HTTPサーバーはそれらを「そのまま」クライアントに送信します。

明らかに、最初のオプションでは、各要求に対してより多くのサーバーリソースが必要です。2つ目は、アプリケーションの組み立てと準備の段階です。

一方、結果は明らかに印象的です。次に、圧縮をnginxに転送してみました- ドキュメントのgoogle。すべてが非常に単純であることがわかりました：

彼はdockerイメージを組み立て、コンテナーを起動し、ひどく驚いた：



ファイルのダウンロード時間が10倍になりました-100ミリ秒から5秒に！アプリケーションが使用できなくなりました。

ドキュメンテーションをより深く研究した結果、静的に配布できることがわかりました。以前に作成したスクリプトを使用して、同じアーティファクトを圧縮し、コンテナに入れて起動しました。時間をダウンロードして通常に戻します-勝利！ただし、このタイプの圧縮をサポートしているブラウザの割合は約80％であるため、喜ぶには早すぎます。

つまり、下位互換性を維持する必要があり、さらに最も効果的なレベルのGzipを使用する必要があります。そこで、後に「Jackal」という名前が付けられたファイル圧縮ユーティリティを作成するというアイデアが生まれました。



私たちは何が必要なのか？


Nginx、Docker、Node.js。ただし、必要に応じてbashを使用することもできます。
Nginxを使用すると、ほとんどすべてが明確になります。

brotli off;
brotli_static on;
gzip_static on;

しかし、まだDockerイメージを更新することができていないアプリケーションをどうするのでしょうか。そう、後方互換性を追加します：

gzip on;
gzip_level 4;
gzip_types text/plain text/css application/javascript;

動作原理を説明します。


各リクエストで、クライアントはAccept-Encodingヘッダーを送信します。このヘッダーには、サポートされている圧縮アルゴリズムがカンマで区切られてリストされています。通常は、deflate、gzip、brです。

クライアントがその行にbrを持っている場合、nginxは拡張子.brのファイルを探します。そのようなファイルがなく、クライアントがGzipをサポートしている場合は、.gzを探します。そのようなファイルがない場合、「オンザフライ」でシェイクし、第4レベルの圧縮でそれを戻します。

クライアントがどのタイプの圧縮もサポートしていない場合、サーバーはアーティファクトを元の形式で発行します。

ただし、問題が発生しました。私たちのnginx DockerイメージはBrotliモジュールをサポートしていません。基本として、既製のドッカー画像を使用しました。

プロジェクトにnginxを「パック」するためのDockerfile
FROM fholzer/nginx-brotli

#     
RUN rm -rf /usr/share/nginx/html/

#     
COPY app/nginx /etc/nginx/conf.d/

#     
COPY dist/ /usr/share/nginx/html/

# 
CMD nginx -c /etc/nginx/conf.d/nginx.conf


トラフィックのバランスを考え出しましたが、アーティファクトはどこから入手できますか？これがジャッカルが助けになるところです。

"ジャッカル"


これは、アプリケーションの静特性を圧縮するためのユーティリティです。

これで、node：alpineというdockerイメージにラップされた3つのnode.jsスクリプトができました。スクリプトについて見ていきましょう。

base-compressorは、基本的な圧縮ロジックを実装するスクリプトです。

入力引数：

  1. 圧縮関数-任意のJavaScript関数。独自の圧縮アルゴリズムを実装できます。
  2. 圧縮パラメータ-転送された関数に必要なパラメータを持つオブジェクト。
  3. 拡張-圧縮アーティファクトの拡張。ピリオド文字で始まる指定が必要です。

gzip.js - zlibパッケージから渡され、9番目のレベルの圧縮を示すGzip関数を含むbase-compressor呼び出しを含むファイル。

brotli.js- 同じnpmパッケージから渡されたBrotli関数を使用し、11番目の圧縮レベルを示すbase-compressor呼び出しを含むファイル。

Jackalイメージを作成するDockerfile
FROM node:8.12.0-alpine

#    
COPY scripts scripts

#  package.json  package-lock.json  
COPY package*.json scripts/

#     
WORKDIR scripts

#   
#    node_modules/  
#  ,    
RUN npm ci

#    
CMD node gzip.js | node brotli.js


私たちはそれがどのように機能するかを理解しました、これで安全に実行できます：

docker run \
   -v $(pwd)/dist:/scripts/dist \
   -e 'dirs=["dist/"]' \
   -i mngame/shakal

  • -v $（pwd）/ dist：/ scripts / dist-コンテナー内のディレクトリーと見なすローカル・ディレクトリーを指定します（マウントへのリンク）。スクリプトディレクトリはコンテナ内で機能するため、スクリプトディレクトリを指定する必要があります。
  • -e 'dirs = ["dist /"]'-環境パラメーターdirsを指定します-スクリプト内のディレクトリを記述する行の配列/圧縮されます。
  • -i mngame / shakal-docker.ioで画像を指定します。

指定されたディレクトリで、スクリプトは指定された拡張子.js、.json、.html、.cssを持つすべてのファイルを再帰的に圧縮し、その隣に拡張子.brおよび.gzを持つファイルを保存します。私たちのプロジェクトでは、このプロセスには約2分かかり、すべてのアーティファクトの重みは約6 MBです。

この時点で、そしておそらくもっと以前に、あなたは考えたことがあるかもしれません。どのノード？プロジェクトのpackage.jsonに2つのパッケージを追加して、ポストビルドを直接呼び出すだけではどうですか？」

個人的には、CIでリンターを実行するために、プロジェクトがそれ自体のために100以上のパッケージをインストールするときに、リンティング段階で最大10を必要とする時期を知るのは面倒です。これは、エージェントの時間、結局のところ、市場投入までの時間です。

Dockerの場合、圧縮に必要なすべてがインストールされた、事前に組み立てられたイメージを取得します。ここで何も圧縮する必要がない場合は、圧縮しないでください。lintが必要です-それだけを実行してください。テストが必要です-それらだけを実行してください。さらに、Jackalの適切なバージョンを取得します。各プロジェクトで依存関係を更新する必要はありません。新しいバージョンをリリースし、プロジェクトの最新のタグを使用するだけです。

結果：


  • アーティファクトのサイズが636 Kbから446 Kbに変更されました。
  • パーセンテージサイズが30％減少しました。
  • ダウンロード時間は10〜12％減少しました。
  • 記事に基づく減圧の時間は同じままでした。

合計


次のPRまでに、今すぐユーザーを支援できます。アセンブリの後に「ジャッカル」圧縮のステップを追加し、アーティファクトをコンテナに配信します。30分後、ユーザーは少し気分が良くなります。

フロントエンドの重量を30％削減できました-成功します！すべての簡単なサイト。
