PostgresqlのDIY生産カレンダー







こんにちは、私の名前はビクターで、Gems Developmentの開発者です。Postgresqlで本番カレンダーの作成と完了をどのように実装したかを教えてください。



ワークフローは、公式の国民の祝日と国際的な祝日によって異なります。企業アプリケーションでは、日付を計算するタスクで、営業日と週末が考慮されることがよくあります。たとえば、「建設許可証の発行」、「試運転許可証の発行」などのサービスは予定どおりに提供する必要があります。



タスクを分析した後、週末と休日をカレンダーに格納するだけで十分であるという結論に達しました。なぜなら、それらは計算が難しく、毎年の生産カレンダーに従って変更できるためです。



カレンダーを操作するためのルール：



  • 稼働日の定義は、日付が週末と休日のリストにあるかどうかによって異なります。
  • 転送に関連する稼働日が非稼働になると、それもテーブルに記録されます。
  • したがって、休日が営業日である場合、このテーブルのレコードは削除されます。


デモのために、個別のスキームを作成し、セッションで変数を設定して、リクエストでスキームを指定しないようにします。 



create schema calendar;
set search_path=calendar;


このような計算をサポートするために、データベースにテーブルが作成されました。 



CREATE TABLE d_holidays
(
  holiday_date date,
  comment character varying(1000),
  CONSTRAINT d_holidays_pk PRIMARY KEY (holiday_date)
);


また、テーブル内の目的の日付の検索を高速化できるインデックス： 

CREATE UNIQUE INDEX d_holidays_pk_idx
  ON d_holidays
  USING btree
  (holiday_date);


データを作成するには、セットを返すPostgresqlの標準関数を使用します。2017年1月1日から2025年12月31日までのすべての日である、生成されたデータの十分に大きなセットをテーブルに事前に入力して、インデックスの動作を確認してみましょう。 



insert into d_holidays(holiday_date)
select generate_series(to_date('01.01.2017','dd.mm.yyyy'),
            to_date('31.12.2025','dd.mm.yyyy'),
            '1 day');


指定した日付からの日数に対応する3287のレコードを取得しました。Postgresql構成でパラメーター「autovacuum = on」が有効になっていることを確認することが重要です。



クエリプランを見てみましょう。 



explain(analyze)
select comment from d_holidays where holiday_date='09.09.2020';


インデックスが使用されていることを確認しました：

  Index Cond: (holiday_date = '2020-09-09'::date)

祝日の計算は年内に行われます。指定年の1月1日から12月31日までのすべての日付を確認した後、休日を選択し、表に記入します。アルゴリズムを実装するには、手続き型言語plpgsqlと無名ブロックの構築を使用します。



1月1日と12月31日の日付になるデザインをご用意いたします。 



do
$$
declare
    calc_year integer:=2020; -- 
    begin_date date; --    
    end_date date; --    
    dow integer; --  . -1,...,-6,-0
begin
    --   
    if calc_year between 1988 and 2099 then
        begin_date :=to_date('01.01.'||calc_year, 'dd.mm.yyyy');
        end_date :=to_date('31.12.'||calc_year, 'dd.mm.yyyy');
    end if;
end$$;


曜日を計算するには、extract関数を使用します。これは、曜日の数を決定します（日曜日が0であることを考慮に入れて）。曜日の名前を決定するために、単純なケースコンストラクトを使用します。



次のスクリプトを取得します。 



do
$$
declare
    calc_year integer:=2020; -- 
    begin_date date; --    
    end_date date; --    
    dow_value integer; --  . -1,...,-6,-0
begin
    --   
    if calc_year between 1988 and 2099 then
        begin_date :=to_date('01.01.'||calc_year, 'dd.mm.yyyy');
        end_date :=to_date('31.12.'||calc_year, 'dd.mm.yyyy');
        
        while begin_date<=end_date
        loop
            raise info '%',to_char(begin_date,'dd.mm.yyyy');
            begin_date:=begin_date+interval '1 day';
            dow_value:=extract(dow from begin_date);
            if dow_value in(0,6) then
                insert into d_holidays(holiday_date,comment)
                        values(begin_date,
                            case when dow_value=6 then '' else '' end);
            end if;
        end loop;
    end if;
end$$;


休日を追加するためのアルゴリズムを分析しましょう： 



insert into d_holidays(holiday_date,comment) 
    values(to_date('01.01.2020','dd.mm.yyyy'),' ') 
        on conflict(holiday_date) do update set comment=EXCLUDED.comment;
…        
insert into d_holidays(holiday_date,comment) 
    values(to_date('04.05.2020','dd.mm.yyyy'),'  04.01.2020') 
        on conflict(holiday_date) do update set comment=EXCLUDED.comment;


休日が営業日となる場合がございます。この場合、アルゴリズムは次のとおりです（たとえば、2020年には休業日がないため、2018が選択されました）。 



delete from d_holidays where holiday_date=to_date('29.12.2018','dd.mm.yyyy'); 
delete from d_holidays where holiday_date=to_date('09.06.2018','dd.mm.yyyy');


設計に注意してください： 



on conflict(holiday_date) do update set comment=EXCLUDED.comment;


この革新はPostgresqlバージョン9.5で登場し、レコードを挿入するときの衝突を効果的に処理することができます。この例では、休日がすでに休日である場合、既存の日付でレコードを追加するときに主キーの競合を避けるために、重複を作成せずに日付の説明を更新します。この構文の詳細については、ドキュメントをご覧ください。
