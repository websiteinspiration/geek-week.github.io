PostgreSQLログのエラーとイベントの監視（grok_exporter）

こんにちは、同僚とhabretchiki！本日は、Prometheusとメトリックgrok_exporterのエクスポーターを使用して、PostgreSQLログに表示されるエラーとイベントの運用監視をどのように整理できるかについて、簡単に説明します。

これはもちろん、このエクスポーターを使用する特殊なケースであることをすぐに言っておく必要があります。それで、なぜそれが必要で、誰が興味を持つのでしょうか？

前書き


誰がこれを必要としますか？


Prometheusがすでにインフラストラクチャに存在し、ELKやGreylogなどのログを収集および分析するための別個のインフラストラクチャを作成するのは、費用がかかり、実用的ではありません（Prometheusがない場合、問題はなく、簡単にすばやくインストールできます）。

このツールは、DBAだけでなく、一般にアプリケーションログを操作するすべての人にとって、アプリケーション管理者にも役立ちます。これにより、フリーランスの行動に関する情報をすばやく取得し、状況をさらに分析するための特定の参照ポイントを得ることができます。

これは何のためですか？


監視ログを使用すると、承認の試み（失敗したものを含む）、さまざまなエラー、または特定のイベントなど、より完全な状況を把握し、新たな問題に迅速に対応できます。

ここには、イベントカテゴリ（ログイベントカテゴリのセクションを参照）で構成された適切なリストがあり、Prometheusでルールを作成するために使用できます。

grok_exporter


メトリックエクスポーターgrok_exporterを使用すると、ソースから非構造化データを読み取り、ルールに従って構造化データに変換し、Prometheusが理解できる形式でメトリックとして返すことができます。

エクスポーターは、ELK plugins-filters-grokのプラグインのイメージで作成されました。これにより、plugins-filters-grokテンプレートのセットをそのまま使用できます。

取り付け


インストールは簡単でシンプルです。これを行うには、リンクをクリックして、お好きなバージョンをダウンロードしてください（そうです、それらはすべてプレリリースであり、最新バージョンはまだリリース候補の段階にあります）。結果のアーカイブを解凍します。

練習


pg_grok_error_count


メトリックは、パターンに従って、エラー、警告、致命的、パニックのイベントの数をカウントします。緊急時の監視と警告に役立ちます。たとえば、次の場合にアラートを構成できます：失敗した認証の試行、単位時間あたりのエラー数のしきい値の超過、またはデータベースが障害後に回復状態にある場合など（ログイベントのカテゴリ）。

たとえば、承認の試みが失敗した場合のアラートを設定します。PostgreSQLログファイルでのログイン試行の失敗の例は、次のようになります。

2020-04-20 23:34:53 AEST datname:test,client:127.0.0.1,app:[unknown],usename:test_user,session:5e9da4fd.733 FATAL:  password authentication failed for user "test_user"
2020-04-20 23:34:53 AEST datname:test,client:127.0.0.1,app:[unknown],usename:test_user,session:5e9da4fd.733 DETAIL:  Password does not match for user "test_user".
        Connection matched pg_hba.conf line 86: "host    all             all             127.0.0.1/32            md5"

grok_exporterの出力では、失敗した認証試行は、サブストリングパスワード認証が失敗した...を含むメッセージラベルによって識別できます。

pg_grok_error_count{app_name="[unknown]",client_host="127.0.0.1",datname="postgres",log_level="FATAL",message="password authentication failed for user "postgres"", usename="postgres"} 15
pg_grok_error_count{app_name="[unknown]",client_host="127.0.0.1",datname="test",log_level="FATAL",message="password authentication failed for user \"test_user\"",usename="test_user"} 5

得られたデータは、プロメテウスのルールを策定するのに役立ちます。感度は自分の現実に基づいて調整できます。

groups:
- name: AuthAlert
  rules:
  - alert: AuthFail
    expr: sum(rate(pg_grok_error_count{message=~"password authentication failed for user.*"}[1m])) by (instance) > 0.1
    for: 30s
    labels:
      severity: warning
    annotations:
      summary: Too many fail authorization for {{ $labels.instance }}

pg_grok_reload_conf_count


同様に、pg_reload_confコマンドの実行を追跡できます。さらに、ラベルのリスト、または1つのセッションラベルのみが使用されているという事実に注意を払う価値があります。これは、イベントがユーザーセッションではなくサーバープロセスの一部として作成されるためです。

PostgreSQLログファイルでは、このイベントは次のようになります。

2020-04-21 01:20:26 AEST datname:,client:,app:,usename:,session:5e9d9371.564 LOG:  received SIGHUP, reloading configuration files

それら。前の例で使用したラベルが空であることがわかります。この場合、識別にセッション識別子を使用します。これは、PostgreSQLのインスタンスが停止または再起動されるまで変更されません。

同様の状況は、インスタンスを停止するなど、他の同様の場合にも当てはまります。

2020-04-21 01:32:52 AEST datname:,client:,app:,usename:,session:5e9d9371.564 LOG:  received fast shutdown request
2020-04-21 01:32:53 AEST datname:,client:,app:,usename:,session:5e9d9371.564 LOG:  database system is shut down

grok_exporterの出力では、次のようになります。

# HELP pg_grok_reload_conf_count Count of call pg_reload_conf
# TYPE pg_grok_reload_conf_count counter
pg_grok_reload_conf_count{session="5e9d9371.564"} 5

間違いなくgrok_exporterは、アプリケーションを監視するための適切なツールです。監視システムの「アクセスが難しい」を調査し、非常にシンプルで機能的で軽量なまま、追加の（制御）メトリックを使用して監視を強化できます。

同時に、監視システムを介してアプリケーションログに少なくとも間接的にアクセスできるため、開発者にも役立ちます。さまざまなソースからのメトリックを関連付けることができる場所。

もう1つの良い点は、プロジェクトが開発中であり、新しい機能を獲得していることです。
