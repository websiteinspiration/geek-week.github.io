自動化の日、または自動テストレイヤーの構築方法

チームで作業しているときに、自動化テストのタスクの一部がバックログに陥り、手動テストに永遠に留まる頻度はどれくらいありますか？私たちは常にこのような問題に直面しており、その理由は常に異なります。

  • 時間がない/十分な時間がない、そして忘れた;
  • 十分な経験がない;
  • 自動化タスクは非常に大きく、サードパーティの統合が必要であることがわかりました。

5月中旬にAvito の同僚のアイデアに触発され、ヤロムドル 最初の自動化デーを開催しました。このイベントの目的は、上記の理由により、テストを自動化していないテストを自動化することです。


その上で、関心のあるすべてのQAエンジニアを集め、事前に準備しました。

  • タスクのバックログ。
  • 行動規範;
  • 技術スタックについて合意した。
  • 準備されたツール;
  • 必要なアクセスを注文しました。

自動化のために、API統合テストレイヤーが選択されました。バックログの各タスクには以下が含まれています。

  • エンドポイントへのリンク、およびjson形式のRESTリクエストの入力および出力パラメーター。
  • テストの本質、それをチェックする方法と理由。

自動化のためのプロジェクトの開始時に、次のスタックを選択しました。その上で、自動テストで埋めるプロジェクトテンプレートを準備しました。

  • Java8
  • JUnit5
  • 春
  • ユニレスト

サンプルプロジェクトテンプレートはここにあります。

静的テストデータの統合自動テストでは既に悪いテストがあり、その下のレイヤーしかなかったため、以前はテストデータを自動生成するためのツールを以前から用意していたことに注意してください。これがないと、新しいフレームワークを作成したくありませんでした。この問題については、MAPの定性的なコミュニケーションで部分的に話しました。

ルール


自動化デーの参加者のためのルールは、可能な限りシンプルでした。開発を始める前に、事前に設計されたプロジェクト構造を示し、簡単な説明を行いました。開発時には、参加者は所定のプロジェクト構造に準拠する必要がありました。
10時から19時まで、昼食と夕食に気を取られることがあるので、参加者はJava + JUnit5でテストを作成します。 Javaでのテストのロジックの実装は、Unirestライブラリを使用したAPI呼び出しとして編成されました。

そして、私たちの居心地の良いゲームルームには、12人のQAエンジニアと2人のAutomation Dayイベントの主催者が集まりました。

QAエンジニアのレベルが異なっていた、誰かがスタック全体を知っていた、誰かがプロジェクトでSpringを使用したことがなかった、誰かがプロジェクトで安心しただけで、Unirestを見たことがない、一部はまだ自動化していないことは言うまでもありませんAPIをテストし、手でテストしました。

結果


8労働時間の集中開発で得られたもの：

  • + 12のプルリクエスト。
  • バックログから12個のクローズされたタスク。
  • + 14の統合自動テスト。
  • >起動ごとに自動テストに合格した99％以上。
  • プロジェクトに精通した12人のQAエンジニア。

一ヶ月後


約1か月後にテスト指標を取得すると、非常に興味深い指標が得られました。

  • + 30のプルリクエスト。
  • バックログから30のクローズされたタスク。
  • + 48の統合自動テスト。
  • UnirestをRetrofitに置き換えました。
  • >起動ごとに自動テストに97％以上合格。

特に、このようなイベントがQAチーム内のコミュニケーションに与えるプラスの影響に注意したいと思います。人々が実際の製品の課題に直面し、近くに助けて教えることができる人がいる場合、これは常にプラスの効果をもたらします。簡単な例、参加者の1人ilya_tadいくつかの理由から、Unirestの代わりにRetrofitを使用することを提案しました。直近では、同じ参加者が2日目の自動化デーを開催しました。その日は全員がテストも作成しましたが、すでにレトロフィットを使用しています。ちなみに、2日目の自動化の日には、Unirestを完全にRetrofitに置き換えました。改訂されたプロジェクトのおおよそのテンプレートはここにあります。

レビュー


Automation Dayの参加者全員に、印刷したてのスーパーヒーローTシャツが届きました。もちろん、最初の自動化デーの後に参加者からフィードバックを収集しました。「この自動化デーで最も価値があり、有用だったものは何ですか？」という質問への回答：

  • テストピラミッドから自動テストの別のレイヤーをカバーするためのすべての参加者の関与（自動テストの最上位レイヤーのエンド2エンドUIをアンロードする）;
  • 他のことに気を取られることなくタスクに集中する能力、およびメンタリングの貴重な経験。
  • 明確に指定された期間の特定のタスクに関する集合的な作業。特定の狭い範囲のタスクと問題の解決に焦点を当てます。
  • ローカルフレームワークの一般的な考えが現れました。

参加者からの要望も集めましたが、すべてを掲載するわけではありませんが、定期的に開催されるのがオートメーションデーでした。

結論


当然、定期的にそのようなイベントを実施するよう努めます。さまざまなトピックを実行し、新しいツールを試します。

Automation Dayを使用すると、実際の製品タスクについて人々を教育することができ、人々は経験を共有し始め、率先して取り組み、互いによりよくコミュニケーションをとることができます。このイベントでは、適切な準備と編成を行うことで、自動テストの層全体を比較的迅速に形成し、この層のさらなる開発に向けて人々を準備することができます。
