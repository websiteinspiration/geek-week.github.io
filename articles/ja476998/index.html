Kubernetes 1.17：主要なイノベーションの概要

昨日、12月9日、Kubernetes の次のリリース-1.17。私たちのブログの伝統に従って、私たちは新しいバージョンの最も重要な変更について話します。



この資料を準備するために使用される情報は、公式発表、Kubernetes拡張追跡テーブル、CHANGELOG-1.17および関連する問題、プルリクエスト、Kubernetes拡張提案（KEP）から取得されます。では、何が新しいのですか？

トポロジベースのルーティング


長い間、Kubernetesコミュニティはこの機能- トポロジ対応サービスルーティングを待っていました。KEPが 2018年10月にそれに基づいており、公式の拡張が2年前であれば、通常の問題（このような）はさらに数年前のものです...

一般的なアイデアは、「ローカル」を実装する機会を提供することですKubernetesにあるサービスのルーティング。この場合の「局所性」とは、「同じトポロジレベル」（トポロジレベル）を意味します。

  • サービスの同じノード、
  • 同じサーバーラック
  • 同じ地域
  • 同じクラウドプロバイダー
  • ...

このような機能の使用例：

  • (multi-AZ) — . , AZ AWS;
  • / ;
  • , ;
  • fluentd ( ) , ;
  • …

トポロジーについて「知っている」このルーティングは、ノードアフィニティ、ポッドアフィニティ/アンチアフィニティ、または最近 導入されたトポロジー対応ボリュームスケジューリング（およびボリュームプロビジョニング）と同様に、ネットワークアフィニティのアフィニティとも呼ばれます。ServiceTopologyKubernetes の現在の実装レベルはアルファ版です。

この機能がどのように配置され、すでにどのように使用できるかについて詳しくは、著者の1人がこの記事を読んでください。

  • kube-proxy (IPv4 IPv6);

CSI


K8s 1.12で最初に導入されたCSIベースのリポジトリ のトポロジサポートは、stableと宣言されています。CSIボリュームの移行プラグイン

・イニシアティブ- CSIの移行 -ベータ版に達しました。この機能は、既存のツリー内プラグインを、Kubernetesエンドユーザーが気付かない最新のインターフェース（CSI、ツリー外）に転送するために重要です。クラスタ管理者がCSI移行をアクティブ化するだけで十分です。その後、既存のステートフルリソースとワークロードは「そのまま機能」しますが、Kubernetesカーネルに含まれている古いドライバではなく、現在のCSIドライバを使用します。

AWS EBS（kubernetes.io/aws-ebs）およびGCE PD（kubernetes.io/gce-pd）ドライバーの移行は現在ベータ版のステータスです。他のリポジトリの予測は次のとおりです。この記事



では、K8の「従来の」ストレージサポートがCSIにどのようにもたらされるかについて話しました。CSIからベータステータスへの移行への移行は、プロジェクトのブログの別の出版物に捧げられています。 さらに、Kubersnetes 1.17リリースのベータ版のステータス（つまり、デフォルトでの組み込み）は、CSIのコンテキストにおける別の重要な機能によって実現され、K8s 1.12の（アルファ実装）に由来します- スナップショットの作成とそれらからの復元。ベータ版リリースの途中でKubernetesボリュームスナップショットに加えられた変更の中には、次のものがあります。



  • サイドカーCSI外部スナップショットを2つのコントローラーに分割し、
  • ボリュームのスナップショットの内容に注釈として削除シークレットを追加しました、
  • 残りのリンクがある場合にAPIオブジェクトスナップショットが削除されないようにする新しいファイナライザー（ファイナライザー）。

リリース1.17の時点では、この機能は3つのCSIドライバー（GCE永続ディスクCSIドライバー、Portworx CSIドライバー、NetApp Trident CSIドライバー）でサポートされています。このブログ投稿で、その実装と使用について詳しく読むことができます。

クラウドプロバイダーのラベル


使用するクラウドプロバイダーに応じて作成されたノードとボリュームに 自動的に割り当てられるラベルは、K8s 1.2 （2016年4月！）のリリース以降、非常に長い間ベータ版としてKubernetesで利用可能でした。長期間にわたって広く使用されていることを考えると、開発者たちは機能を安定版（GA）と宣言する時がきたと判断しました。

したがって、これらはすべてそれに応じて（トポロジーによって）名前が変更されました。

  • beta.kubernetes.io/instance-type → node.kubernetes.io/instance-type
  • failure-domain.beta.kubernetes.io/zone → topology.kubernetes.io/zone
  • failure-domain.beta.kubernetes.io/region → topology.kubernetes.io/region

...しかし、古い名前で引き続き使用できます（下位互換性のため）。ただし、すべての管理者は現在のラベルに切り替えることをお勧めします。関連する K8s ドキュメントが更新されました。

構造化出力kubeadm


アルファフォーマットでは、kubeadmユーティリティの構造化された出力が初めて表示されます。サポートされる形式：JSON、YAML、Goテンプレート。

この機能を実装する動機（KEPによる）は次のとおりです。

Kubernetesは手動でデプロイできますが、この操作の事実上の標準（適切でない場合）は、kubeadmを使用することです。Terraformのような一般的なシステム管理ツールは、Kubernetesのデプロイをkubeadmに依存しています。Cluster APIのスケジュールされた改善には、kubeadmおよびcloud-initを使用したKubernetesブートストラップ用の構成可能なパッケージが含まれます。

構造化された出力がないと、一見して最も無害な変更でさえ、Terraform、Cluster API、およびkubeadmの結果を使用するその他のソフトウェアを破壊する可能性があります。

近い将来、次のkubeadmコマンドが（構造化された出力の形式で）サポートされるようになります。

  • alpha certs
  • config images list
  • init
  • token create
  • token list
  • upgrade plan
  • version

他のイノベーションの安定化

  • カスタムリソースのデフォルト値;
  • pod' process namespaces;
  • ScheduleDaemonSetPods — pod' DaemonSet kube-scheduler ( DaemonSet);
  • ;
  • , subPath;
  • Kubelet heartbeats Lease API;
  • « » (Finalizer Protection) ( Service' LoadBalancer');
  • 同一のオブジェクトのセットを監視する複数のウォッチを操作するときのパフォーマンスにおけるkube-apiserverの最適化は、各ウォッチャーに対して同じオブジェクトを再シリアル化することを回避することによって実現されます。
