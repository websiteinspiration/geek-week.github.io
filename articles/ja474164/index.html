ジンブラは私たちのすべてです

この奇跡を鉄のマシンから仮想のマシンに移したのは、どういうわけかここで起こりました。状況は率直に憂慮すべきです-カーネル2バージョンとZimbra 6.0.4の古いDebianの再配置はCentOS 7とZImbra 8.8で計画されました。

このイベントを完了するには、BDSMを完全に愛している必要があることをすぐに言っておく必要があります。そうしないと、何も起こらず、すべてがパスの真ん中に立ってしまいます。

それでは、順番に見ていきましょう。

最初に、古いサーバーと同じバージョンのZimbraをデプロイする必要があります。私の場合、CentOS 5のZimbra 6.0.4でしたが、残念ながら、それは当時、このソフトウェアのOSの最も高度なバージョンでした。

ここからジンブラをダウンロードしてください。

wget https://files2.zimbra.com/downloads/6.0.4_GA/zcs-6.0.4_GA_2038.RHEL5_64.20091214192925.tgz

そして解凍します：

tar -xvf https://files2.zimbra.com/downloads/6.0.4_GA/zcs-6.0.4_GA_2038.RHEL5_64.20091214192925.tgz

インストールする前に、CentOSを構成してください。バージョン5では、Selinux、Postfix / Sendmail（いずれか）を無効にし、yumのカブを設定します。

vi /etc/sysconfig/selinux

そこで値を設定します：

SELINUX=disabled

インストールと更新の際にサービスが干渉しないように、サービスを停止してオフにします。

service iptables stop
chkconfig iptables off
service sendmail stop
chkconfig sendmail off
service postfix stop
chkconfig postfix off

yumのカブをセットアップしました。このファイルでは、mirrorで始まる行を＃コメントアウトする必要があります。

vi /etc/yum.repos.d/CentOS-Base.repo

そして編集 

'baseurl='

5.11の代わりに、バージョンを設定しました。

baseurl=http://vault.centos.org/5.11/os/$basearch/

次に、CentOSを更新します

yum update

OSを更新した後、1つの設定のみを行う必要があります。zimbraはDNSに非常に嫉妬しています。これは非常に重要な設定であり、これがないとインストールプロセスは機能しません。

vi /etc/hosts

その中に私たちはあなたのIPと完全なサーバー名を登録します-IP FQNS NS。

すべてがその順序であり、そうでなければ再び間違いです。私の場合、サーバーはsrvmail（または、そのmxレコード）と呼ばれていました。

192.168.0.2 srvmail.domail.local srvmail

次に、インストールを続行し、事前にzimbraを解凍したディレクトリに移動して、以下を実行できます。

./install.sh --platform-override

国旗 

"--platform-override"

どのような場合でも何をインストールするかをインストーラーに指示します（プラットフォームを強制的に書き換えます）。

別の有用なフラグは"-s"、実行可能パッケージのみをインストールし、構成しないことをインストーラーに通知するこのフラグです。

次に、インストーラーがインストールを開始し、すべてが正しく行われていれば、エラーは発生しません。インストールの最後に、zimbraサービスのステータスを確認します。

zmcontrol status

すべてのサブサービスがステータスになっているRunning場合は、問題ありません。

zimbruを停止します。

zmcontrol stop

zimbraをインストールしたら、データベースと設定ファイルを古いzimbraサーバーから新しいzimbraサーバーに転送する必要があります。

フォルダが必要です

db data index

にあります 

/opt/zimbra

同様にファイル 

/opt/zimbra/conf/localconfig.xml

新しいサーバー上のフォルダーデータは、削除またはバックアップできます。

転送中のフォルダーとファイルはホスト（権限）を変更する可能性があるため、このことを修正する必要があります。

chown -R zimbra.zimbra /opt/zimbra
/opt/zimbra/libexec/zmfixperms

また、テストを実行する前に、SQLデータベースでエラーを確認する必要があります。これは、標準ユーティリティを使用して行うことができます。 

mysqlcheck

または組み込みのZimbroユーティリティを使用する 

zmdbintegrityreport

mysqlを実行し、スクリプトを実行します。

/opt/zimbra/bin/mysql.server start
/opt/zimbra/libexec/zmdbintegrityreport
/opt/zimbra/libexec/zmdbintegrityreport -r

1行目はエラーをチェックし、2行目はチェックしてすぐに修復を試みます。

エラーが発生していない場合は、手をたたいて下を見てください。エラーの場合：

mysql.general_log
Error : You can't use locks with log tables.
mysql.slow_log
Error : You can't use locks with log tables.

次に、以下を実行する必要があります。

cd /opt/zimbra/db/data/mysql
ls -al *log.frm
mv *log.frm /tmp/
/opt/zimbra/libexec/zmdbintegrityreport

後者がエラーなしで完了したら、mysqlを停止して次に進みます。

/opt/zimbra/bin/mysql.server stop

次に、パスワードをldapベースに変更する必要があります。ldapを実行します。

ldap start

「r」フラグを使用する最初のバージョンでは、rootユーザーのパスワードを変更し、zimbraユーザーの2行目では、ldapを再度停止します。

zmldappasswd -r newrootpass
zmldappasswd newpass
ldap stop

あなたがWEBに侵入しようとすることができます、それがロードしないならば、絶望しないでくださいこれは正常です。

これらの問題は、同じバージョンへのアップグレードにより解消されます!!!
注意アップグレード時にはZimbraサービスを開始する必要があります。それ以外の場合は失敗します。


ダウンロードしたzimbraがあるフォルダーに移動して、次のように入力します。

./install.sh --platform-override

Zimbraは、更新が必要かどうかを数回尋ねます。「yes」と言って続行します。ちなみに、移動したlocalconfig.xmlからシステムの設定を取得します。これを小さなハックと考えてください:-)

インストール後、メーラーのステータスを確認してください：

zmcontrol status

すべて問題なければ、WEBにアクセスして、古いサーバーの任意のユーザーの下に行くことができます。パスワードは同じです。

さらに、親指と同じように、最新の7.x.x.xバージョンをダウンロードし、アンパックして、もう一度：

./install.sh --platform-override

この段階で、OSを次のステップにグレードアップする準備ができました。
