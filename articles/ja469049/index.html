Luaでビジネスロジックを作成する理由

こんにちは、ハブル。この投稿では、IPONWEBでLuaという美しい名前のプログラミング言語を使用する方法と理由について説明します。

Luaはスクリプトベースの組み込みプログラミング言語で、Cで無料のインタプリタとオープンソースを備えています。1993年にブラジルのリオデジャネイロカトリック大学のテクグラフ部門で開発され、その祖先はDEL（データ入力言語）とSOLでした。 （Simple Object Language）は以前に開発されました。祖先の1つであるSOL言語は、新生児の「バプテスマ」に間接的に参加しました。「Sol」はポルトガル語から「太陽」として翻訳され、新しい言語は「ルア」、「月」と名付けられました。

「システム」言語で記述されたエンジンにLuaを簡単に埋め込むことができるようになったため、Luaはビデオゲームで人気のスクリプト言語になりました。たとえば、スクリプトはGrim FandangoのBalaとBaldur's Gateで記述されています。World of Warcraftをプレイする人は、おそらくLuaについて1回または2回以上聞いたことがあるでしょう。ハードコアプレーヤー、カジュアルファン、ファンがゲームの世界のその他の人々の有効性を測定するのを楽にする、ゲームのアドオンが書かれているということです。Luaは、gamedevの外では、組み込みシステム（TV、プリンター、車のパネル）のスクリプト言語や、VLCメディアプレーヤーなどのアプリケーションとして使用されています。Luaは、Tarantool、Redis、OpenRestyなどのツールを埋め込み言語として使用します。Luaは、核燃料の熱機械的挙動をシミュレートするFortran計算コードの拡張言語としても使用されました。

なぜルア？


IPONWEBは、オンライン広告の分野で活動する企業（DSP、SSP、広告代理店、広告主）向けの高負荷プラットフォームの開発者です。この記事では、私たちの仕事について詳しく話しました。。当初、私たちはプラットフォームのビジネスロジックをC ++で開発しましたが、これが最良の選択ではないことにすぐに気付きました。コストを最小限に抑えるには、プラットフォームのパフォーマンスと開発速度が重要であり、C ++の開発は遅すぎることが判明し、機能の追加の複雑さも影響を受けました。ビジネスロジックの解釈を低レベルのサーバーコードから分離することに決め、これに適した言語を探し始め、Luaに落ち着きました。それは2008年のことでしたが、私たちに適したJavaScript実装はまだ存在していませんでした。Perl、Python、Rubyは遅すぎて統合が簡単ではありませんでした。また、あまり知られていないがゲーム開発者に人気のあるLua言語があり、私たちが望んでいたのはゲーム開発者のニーズと同様でした-低レベルの操作用の高速エンジンとビジネスロジック用のビルドが簡単な高速言語が必要でした。

Luaは非常に高速な言語です。トレースJITコンパイラーを含むLua 5.1のランタイム環境であるLuaJITを使用することで、速度をさらに上げることができます（私たちは、すでに書いた独自のフォークを使用しています）。RTBのビジネスロジックを記述するときシステム、速度は私たちにとって重要です。RTBでは、平均して、各着信要求を処理するために120ミリ秒があります。同時に、コードの実行には10〜15ミリ秒しか割り当てられず、残りの時間は他のサービスからの応答を待機しています。たとえば、ユーザーがネットワークにサイトをロードするときに0.5秒の遅延に気付かない場合、RTBの場合、これらの500ミリ秒は非常に長い時間です。 Lua言語はどれくらい速いのかという質問に対する答えは次のとおりです。それは、私たちが長年ビジネスロジックを記述して、RTBビジネスに留まることができるほど十分に高速です。私たちが選択した言語が十分に速くなかった場合、プラットフォームを作成する人がいなくなります。これは、RTBを他の言語で作成できないことを意味しますか？という意味ではない。しかし、私たちはLuaでRTBを作成し、私たちとクライアントのビジネスタスクにうまく対処しています。サーバー上のLuaの速度の良い例はこれですOpenResty ベンチマーク。

組み込み言語としてのLuaには多くの利点があります。それは非常に小さな標準ライブラリーを備えた、ミニマルでコンパクトです。その機能はCで完全に複製されており、LuaとCの間の簡単で「シームレスな」対話を提供します。Luaは、他の多くの言語と比較してログインしきい値がかなり低くなっています。完全に仕事に従事するために数日。

そして、これは単純なハンドラー（イベントハンドラー）のようになります。

local adm_cache = require 'modules.adm_cache' -- adm = "ad markup"
local config = require 'modules.config'
local util = require 'modules.util'
 
local AbstractHandler = require 'handlers.abstract'
local ImpRenderHandler = AbstractHandler:new({is_server_request = false})
local IMP_RENDER_TEMPLATE = config.get('imp_render_template')
 
local IMP_RENDER_BILLING = {name = 'free', type = 'in'}
 
--   ,  .
--   ( ,   ) .
--          .
function ImpRenderHandler:handle(params)
    local user_id = self.uuid
 
    local cache_id = get_param('id')
    if not cache_id or cache_id == '' then
        return self:process_bad_request({reason = '"id" parameter is expected', user_id = user_id})
    end
 
    local adm = adm_cache.get(cache_id)
    if not adm then
        return self:process_bad_request({reason = 'No adm in cache', user_id = user_id})
    end
 
    update_billing(IMP_RENDER_BILLING)
    self:log_request('imp_render', {adm = adm, user_id = user_id, cache_id = cache_id})
 
    local content = util.expand_macro(IMP_RENDER_TEMPLATE, {ADM = adm})
 
    return {200, content = content}
end
 
return ImpRenderHandler

Luaのシンプルさは、迅速な開発を可能にするだけでなく、少ない労力で多くの作業を行うことを可能にします。 IPONWEBプラットフォームは、特定のクライアントのニーズに合わせて調整された一般的なソリューションであり、1人の開発者と1人のマネージャーがプロジェクトを実行できます。 Luaのコードを読むことは、開発者自身だけでなく、プロジェクトマネージャー、さらには顧客にも可能です。一緒に、私たちは問題をすばやく発見し、その原因と解決策を見つけます。多くの場合、プロジェクトマネージャーは開発者に問題を通知し、すぐにそれを解決する方法を提案します。

同時に、Luaの単純さはだまされている可能性がありますが、ミニマリズムとコンパクトさには欠点があります。たとえば、PerlやPythonでコードが記述されている場合、開発者は既成のモジュールの巨大なリポジトリを自由に利用でき、RubyにはRubyGemsがあり、他の多くの言語には豊富なリポジトリがあります。そしてルアはLuaRocksとそこにある3000個のモジュール。さらに、LuaRocksに適切なモジュールがあったとしても、特定の会社でそれを使用するために努力する必要がある可能性があります。Luaは、コード（サンドボックス）を実行するための安全な環境を作成するための優れたツールを提供します。サンドボックスで作業する場合、一部の機能を無効にすることができます。つまり、LuaRocksモジュールは、会社の安全な環境によってブロックされている機能を使用していると機能しない可能性があります。これはコンパクトさと埋め込み可能性の代償ですが、それだけの価値はあります。たとえば、Pythonなどの電池を使用する言語は、Luaよりも複雑な方法で構築されていません。

使い方？


私たちのプラットフォームの基礎は、Lua開発者に一連の機能を提供し、広告市場のタスクに合わせて調整された、便利で拡張可能なAPIを備えたカスタマイズ可能なHTTPサーバーです。このサーバーは何億ものリクエストを処理し、1日あたりテラバイトのログを書き込みます。受信リクエストはシステムスレッド全体に均等に分散され、システムスレッドの内部はサンドボックスです。



リクエストがサーバーに到着すると、このリクエストが分類されたサンドボックス内にコルーチンが作成され、リクエストが処理されます。コルーチンは互いに独立して動作し、作成される各コルーチンは実行のためにキューに入れられます。各コルーチンの有効期間（関連するサービス（データベース、メトリック、バジェットサーバー）の予想される応答を考慮したリクエストの合計処理時間）は、120ミリ秒を超えてはなりません。



つまり、リクエスト処理プロセスは次のように説明できます。

  1. 受信した各要求は解析され、正当性の標準チェックに合格します。
  2. このリクエストの処理を担当するCorutinが起動されます。各サンドボックス内には、さまざまなステータスの多くのコルーチンがあります。
  3. リクエスト処理が開始され、2つの結果が得られます。
    • 処理が正常に完了しました。
    • Corutinは制御をサーバーに転送します。これは通常、コルーチンが他のサービスからの応答を待っているときに発生します。そのような場合、応答が到着するか、待機時間が経過するまで、コルチンの作業は中断されます。制御を転送すると、サーバーは次の要求の処理を開始します。これは、新しい要求か、関連するすべてのサービスから応答を受け取り、コードの実行を継続する準備ができている要求のいずれかです。リクエストの処理順序は、ビジネスロジックの要件によって決まります。


corutinの使用は、詳細な議論に値する別の興味深いトピックです。たとえば、この記事では、コルチンを使用してビデオゲームでカットシーンを作成する方法を詳しく説明しています。そして、アプリケーションサーバー上のコルーチンは別の記事を取り上げる必要があり、おそらく将来はそれを行う予定です。

次は何ですか？


そして、おそらく、IPONWEBでのLuaの使用が拡大されるでしょう。 Luaを私たちのビジネスでどのように使用できるかについてのアイデアがあり、これらのアイデアが実装されると、私たちは間違いなく新しい経験を共有します。組み込みのスクリプト言語としてのLuaの利便性と機能は、特にクライアントデータの処理を高速化するのに役立ちます。しかし、これはまだ計画と見通しの分野からのものです。

要約すると、11年前に作成されたビジネスロジックの言語の選択は、それ自体が正当化され続け、私たち自身のビジネス問題にうまく対処し、顧客の問題を解決するのに役立つと言えます。Luaは読みやすく、統合しやすく、速くて簡単に習得できます。Luaはこれまでも今も最高のスクリプト言語の1つであり、その範囲は決してゲーム開発に限定されません。その上に書かれたIPONWEBビジネスロジックは、この一例にすぎません。
