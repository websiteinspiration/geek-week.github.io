仕組み！重要


CSSでは、スタイルを優先するために特定性メカニズムを使用します。カスケードまたは複雑なセレクターを記述し、セレクターの重みを増やして、他のスタイルよりもいくつかのスタイルの優先順位をブラウザーに示します。

例えば：

.content .title {
    color: red;
}

.title {
    color: blue;
}

この場合、赤の後に見出しに青が適用されているにもかかわらず、結果として、赤の見出しが表示されます。最初のセレクターがより具体的であるため（突然これに慣れなくなった場合は、これ以上読み進めないでください。悪化するだけで、ここでより具体的になり、具体性を学びます）。

したがって、同じ例を使用しますが、重要です！重要なことに、彼はあなたの特異性を気にしていないことが突然わかりました。

.content .title {
    color: red;
}

.title {
    color: blue !important;
}

タイトルは青になります！このプロパティは、記述したルールの自然な流れに違反し、優先度の低いスタイルに最高の優先度を与えます。

さて、しかし、「唾を吐く」方法...実際には、そして唾を吐きます。別のインポーターを追加しましょう。

.content .title {
    color: red !important;
}

.title {
    color: blue !important;
}

そして、見出しは再び赤です。2つあるからです！重要な競合とこの競合は、セレクターの重みを比較することによって解決されます。

次の場合、インポーターは不良です。

.title {
    color: red;
}

「うーん、「赤」と書いたのに、見出しが赤くならないのはなぜですか？そしてそうならば？"

.title {
    color: red !important;
}

「ファイン！赤！" 本当に完璧ですか？

少なくともインポーターなしでスタイルが適用されなかった理由を理解せずにスタイルを再定義するため、このアプローチは少なくとも不適切です。おそらく、使用したセレクターの重みは単に不十分であることが判明し、少し正確で「重い」セレクターを作成すると、問題を解決し、スタイルを厳密に再定義することになります。このクラスが使用されましたが、使用されていない場合は、1つの場所で要素を従属させ、おそらく別の場所で壊れました。

修正プログラム用のツール


多くの場合、インポーターについて誰かに質問すると、「問題をすばやく修正する必要があるときに重要なものを使用し、自分で記述していないコードや非常に不十分に記述されているコードを理解する時間がありません。」

はい。ただし、常に「緊急」であるタスクのみです。そのようなタスクは、原則として、誰も終了しないため、そのままにしておきます。あなたは自分が「悪い人のコード」をさらに悪化させたことに気づいていません。

高速とはどういう意味ですか？再定義のためにセレクターに必要な重みを見つけることはそれほど難しく、長いですか？私たちは開発でDevToolsを使用することに非常に積極的であり、この問題を解決するための答えがあります。

変更したい要素を調べます。現在の色は黒です。この色が割り当てられているセレクターを見てみましょう。



同じセレクターを使用して、新しい色を割り当てたり、重みを増やしたりします。このセレクターがどんなに難しくても、ポイントは帽子にあります。



長いですか？これは執筆と時間的にほとんど差がないように思えます！重要ですが、はるかに正確で安全です。

サードパーティのソリューションと他の誰かのコード


インポーターを使用する必要性についての質問に対する2番目に人気のある回答：「私は使用します！サードパーティのライブラリとプラグインまたはCMS-okのスタイルを再定義するときに重要です。」

この答えは少し真実に近いですが、少し正確に定式化されている場合のみです。サードパーティのファイルのスタイルは自分のスタイルと同じ方法で上書きでき、他の人のスタイルを接続してセレクターの重みを増やした後に追加するだけなので、この形式では正しくありません。

ただし、場合によっては、この回答が質問に「入り込む」ことがあります。たとえば、スライドを切り替えたときにJavaScriptがCSSの一部のプロパティの値を上書きするように実装されているページにスライダーを接続した場合。



そして、何らかの理由で、プラグイン自体への変更を無視して、これらのプロパティをオーバーライドする必要があります。

JSはインラインスタイルを使用してのみプロパティ値を再割り当てできます。つまり、実際には、JavaScriptを使用して要素に対して変更されたプロパティは、単純にスタイル属性に書き込まれます。特異性メカニズムの観点からは、これらのスタイルの方が優先され、セレクターの重みをどのように増やしても、それらのスタイルを上書きすることはできません。

これは、私たちが重要なことなしに行うことができない最初のケースです。
インラインスタイルを再定義する必要がある場合は、重要なものだけが役立ちます。
インポーターを使用したスタイルの優先度は、インラインスタイルの優先度より高く、何らかの理由でインラインスタイルを書き換える必要がある場合は、使用するしかありません！重要。

善のために重要


さまざまなライブラリやプラグインのコードで使用されることがあるもう1つのケースを考えてみましょう。

BootstrapライブラリのCSSソースコードを開くと、ライブラリ開発者がインポーターを使用することがよくあることがわかります。おそらく、彼らはまた、すぐに修正する必要があるだけで、修正しました（皮肉）が、一般的には、これについてより適切な説明があります。この場合、重要なは「永久に」使用されます。このようにして、Bootstrap開発者はコードを「保護」します。

実際には、このライブラリをサイトに接続することにより、元のスタイルを再定義できます。したがって、開発者は安全であり、具体性がどのように機能するかを知らなければ、コードの動作を壊すことはより困難になります。しかし、CSSの特定性メカニズムがどのように機能するかを理解している場合は、インポーターを使用してスタイルを記述し、より「重い」セレクターを使用して、インポーターを再定義できます。

したがって、輸入者なしでは不可能である2番目のケースになります。
重要なもの（自分のものまたはサードパーティのライブラリからのもの）をオーバーライドする必要がある場合は、！重要なものが役立ちます。

そのため、「サードパーティのライブラリスタイルをオーバーライドする必要があるときにインポーターを使用する」と言うのは正しくありません。「インラインスタイルやその他をオーバーライドする必要があるときにインポーターを使用する！重要です」と言う方が正しいでしょう。そして、これらのスタイルがどこから来たのかは関係ありません：ファイルから、またはサードパーティから。

ヘルパー


実際、前の段落では、インポーターの使用が正常で正しい場合のすべてのケースについて説明しました。これらは、輸入業者の使用が避けられない場合です。しかし、おそらく完全に適切ではないかもしれませんが、輸入業者の使用が許容される場合はまだあります。

コードの再利用と簡略化を目的として、コードで使用する補助的なユニバーサルクラスが必要になる場合があります。
たとえば、ある種の.warningクラスを作成し、それをページのさまざまな要素に適用して、それらの重要性を示します。

.warning {
     color: red;
}

そして、このクラスを任意の要素に適用することで、この要素のテキストを確実に赤にします。しかし、要素はより正確なセレクターを使用して記録された優先度の高いスタイルを持つことができ、.warningクラスからの色は地面になります。この場合、このクラスで使用される色がより優先されるため、ヘルパークラスに重要を追加して、それをより普遍的にします。でも、いいとは言えません。結局のところ、これによりクラススタイルの優先度が高くなりますが、このヘルパーが別の重要なヘルパーに出くわす可能性があるため、100パーセントの作業が保証されるわけではなく、すべて同じことがうまくいきません。それでも、特定のケースで、このクラスはテキストを「それほど赤くしない」ようにする必要があることが判明した場合、インポーターを繰り返してすぐに新しいセレクターに「出産」します。
