2.エラスティックスタック：セキュリティログ分析。Logstash



前回の記事では、ELKが構成するソフトウェア製品のELKスタックについて説明しました。また、ELKスタックを操作するときにエンジニアが直面する最初のタスクは、ログを送信してElasticsearchに保存し、さらに分析することです。ただし、これは言葉で言えば、elasticsearchは特定のフィールドと値を含むドキュメントの形式でログを保存するため、エンジニアはさまざまなツールを使用してエンドシステムから送信されるメッセージを解析する必要があります。これはいくつかの方法で行うことができます-APIを使用してデータベースにドキュメントを追加するか、既製のソリューションを使用するプログラムを自分で作成することによって。このコースの一環として、Logstashソリューションについて説明します。ELKスタックの一部です。エンドシステムからLogstashにログを送信する方法を確認してから、Elasticsearchデータベースに解析およびリダイレクトするための構成ファイルを構成します。このため、Check Pointファイアウォールからのログを受信システムとして取得します。

コースの一部として、ELKスタックのインストールは考慮されていません。このトピックには多数の記事があるため、構成コンポーネントについて検討します。

Logstashを構成するためのアクションプランを作成しましょう。

  1. elasticsearchがログを受け入れることの検証（ポートの正常性と開放性の検証）。
  2. イベントをLogstashに送信し、メソッドを選択して実装する方法を検討します。
  3. Logstash構成ファイルで入力を構成します。
  4. ログメッセージがどのように見えるかを理解するために、デバッグモードでLogstash構成ファイルの出力を構成します。
  5. フィルターを構成します。
  6. ElasticSearchで正しい出力を構成します。
  7. Logstashが起動します。
  8. Kibanaのログを確認します。

各項目をさらに詳しく検討してみましょう。

チェックポイントファイアウォールからLogstashにログを送信する方法を検討する


Check Point管理サーバーを使用すると、log_exporterユーティリティを使用してsyslog経由でLogstashにログを送信できます。この記事で詳しく理解できます。ここでは、ストリームを作成するコマンドのみを残します

。cp_log_export add name check_point_syslog target-server << ip_address_logstash >> target-port 5555プロトコルtcp形式の一般的な読み取りモードの準統一

<< ip_address_logstash >>-Logstashが実行されているサーバーのアドレス、ターゲットポート5555-ログを送信するポート、tcp経由でログを送信するとサーバーをロードできるため、場合によってはより正確ですUDPを設定します。

Logstash構成ファイルでINPUTを構成します




デフォルトでは、構成ファイルは/etc/logstash/conf.d/ディレクトリーにあります。構成ファイルは、3つの意味のある部分で構成されています：INPUT、FILTER、OUTPUT。ではINPUT、我々はシステムがで、からログを取る場所を示す我々はFILTERログを解析する-私たちはフィールドと値にメッセージを分割する方法を設定、でOUTPUT我々は出力ストリーム設定-解析されたログが送信されます。

最初に、INPUTを構成し、ファイル、tcp、exeなどのタイプを検討します。

TCP：

input {
tcp {
    port => 5555
    host => “10.10.1.205”
    type => "checkpoint"
    mode => "server"
}
}

モード=>「サーバー」
Logstashが接続を受け入れると言います。

ポート=> 5555
ホスト=>「10.10.1.205」
IPアドレス10.10.1.205（Logstash）、ポート5555での接続を受け入れます—ポートはファイアウォールポリシーで許可されている必要があります。

タイプ=>「チェックポイント」
ドキュメントにマークを付けます。複数の着信接続がある場合に非常に便利です。以下では、接続ごとに、if論理構造を使用して独自のフィルターを作成できます。

ファイル：

イベントのタイプ。

ファイルを変更すると、ファイル全体が読み取られます。「end」を設定すると、システムはファイルの最後に新しいエントリが表示されるのを待ちます。

エグゼクティブ：

input {
  exec {
    command => "ls -alh"
    interval => 30
  }

この入力では、（のみ！）シェルコマンドが起動され、その出力がログメッセージにラップされます。

コマンドを呼び出す間隔（秒単位）。

フィルターをカスタマイズ


最後の段階で、彼らはkvを選択し、次にこのフィルターの構成を示します。

filter {
if [type] == "checkpoint"{
	kv {
		value_split => "="
		allow_duplicate_values => false
	}
}
}

フィールドと値を分割する記号「=」を選択します。ログに同じエントリがある場合、データベースにインスタンスを1つだけ保存します。それ以外の場合は、同じ値の配列を取得します。つまり、「foo = some foo = some」というメッセージが表示された場合は、foo = someのみを書き込みます。

ElasticSearchで正しい出力を構成する


フィルターを構成したら、ログをelasticsearchデータベースにアップロードできます。

output 
{
if [type] == "checkpoint"
{
 	elasticsearch 
        {
		hosts => ["10.10.1.200:9200"]
		index => "checkpoint-%{+YYYY.MM.dd}"
    		user => "tssolution"
    		password => "cool"
  	}
}
}

ドキュメントがチェックポイントタイプで署名されている場合は、イベントをelasticsearchデータベースに保存します。データベースは10.10.1.200でポート9200への接続をデフォルトで受け入れます。各ドキュメントは特定のインデックスに保存されます。この場合、「チェックポイント-」+現在の日付のインデックスに保存されます。各インデックスには特定のフィールドセットを含めることができます。または、メッセージに新しいフィールドが表示されたときに自動的に作成され、フィールド設定とそのタイプをマッピングで表示できます。

認証を構成した場合（後で検討します）、特定のインデックスに書き込むためのクレジットを指定する必要があります。この例では、パスワードは「cool」の「tssolution」です。特定のインデックスにのみログを書き込むユーザー権限を区別できます。
