新旧のCSS。ウェブデザインの歴史

90年代後半にウェブのデザインと開発に興味を持つようになりました。くそっ、どれくらい前か。そしてそれはどれほどひどかった。つまり、ウェブサイトの作成と公開はトリッキーなビジネスであり、顧客はすぐに数えることができました。

ほとんどの開発者は当時、または少なくとも次の10年を覚えているように思えますが、違います。最近、私はツイートに出会いました。作者は、角丸を表示する前に設定する手法に驚いていましたborder-radius（角丸ごとに個別の画像を作成し、正確に配置します）。それborder-radiusが標準になり、ブラウザの接頭辞が削除されたとき、私たちはどのように息を切らして待ったかをまだ覚えています。

一方、多くの人は昔、ウェブデザインを少しだけ試し、それ以来何も変わっていないと信じています。

この記事はすべての人を対象としています。CSSとWebデザインの歴史を覚えています。

（あなたが読むことができる。特に因果関係について、したがって、私はそのすべてが本当に正しく実行設定されている保証することはできません。メモリや研究が密接にこの記事に絡み合っていることを覚えておいてくださいW3Cからの正式なCSSの歴史を、はるかに短い、より良い試合現実と大幅に少ない呪いが含まれています）。

初日


最初はCSSがありませんでした。

とても悪かったです。

この時代の私のお気に入りのアーティファクトはO'Reillyの本です。私はこれをHTML：The Definitive Guideで勉強しました。このガイドのいくつかの版は、90年代半ばと後半に発行されました。この本では、CSSについてはまったく触れられておらず、HTMLについてのみ触れています。私はそれをもう持っておらず、インターネット上でスキャンを見つけることができませんが、この本の新しい版のページHTML＆XHTML：The Definitive Guide（XHTMLについては後で話します）のページがあります。 90年代の最先端のウェブデザインのヒントを次に示します。


「ヘッダーとフッターを水平線ではっきりと強調します。」

いいえ、それだけではありませんborder-top。あれ<hr>。ページタイトルはほぼ間違いなく中央に配置され<center>ます。

ページの背景、フォント、テキストの色は標準です。このガイドは最初に最初の概念を紹介するためです。それが白黒の本であるという理由と、それが現実を反映しているという理由で、ページ上の何かを着色することは巨大な痔でした。

すべてのヘッダーを赤にしたいとしましょう。これを行うには、次のタグを配置します。

<H1><FONT COLOR=red>...</FONT></H1>

... 各見出しのため。あなたが青に切り替えることを決して決心しないことを願っています！

ああ、HTMLタグは大文字でカスタム作成されました。なぜそれが良いアイデアだと私たち全員が決めたのか覚えていません。おそらくこれは、テキストエディタに構文の強調表示が表示される前でした（私は12歳で、メモ帳を使用していました）。見出しタグは、メインテキストと区別しやすくなっています。

したがって、サイトを維持することは本当に悪夢でした。多くの人は、デフォルトの設定だけで、デザインを使用しない方が簡単だと判断しました。ある意味で、それはすごく良かった。ブラウザではこれらのデフォルト値を変更できるので、インターネット上のテキストを好きな形で読むことができる。

多くのGeocitiesサイトで興味深い代替ソリューションがどのように登場したかを覚えています。すべての新しいWebページの外観がまったく新しいものです。くそ、それでしょ？すべての新しいページで必要なことを実行してください。

多分これはウェブデザインの頂点だった。

地獄、私はその日が恋しいです。完全にオープンなウェブ、Twitter、Facebookはありません。言いたいことがあるなら、あなた自身のウェブサイトを作るべきです。それはだった素晴らしいです。彼が何をしているのか誰も知らなかった。当時のウェブデザイナーの大多数は、他の無知なアマチュアティーンからページをコピーする無知なアマチュアティーン（私のような）だったに違いありません。インターネットの半分は、アニモルフについてのファンが作ったポータルであり、サイトが640x480の画面解像度で最適に機能するという不可解な警告画面があります（おそらく、画面解像度が不十分な12歳のユーザーは、新しいモニターを購入する必要があります）。すべてのクールな人がInternet Explorer 3を使用しています-Internet Explorer 3-最先端のブラウザーただし、一部の敗者はまだNetscape Navigatorに座っているため、メインページにアニメーションGIF「Best in IE」を配置する必要があります。

サムネイルグリッド


別のランダムなページを考えてみましょう。たとえば、「映画の写真」などです。（ちょっと待って、写真？当時、「フレーム」と「スクリーンショット」という言葉は知りませんでしたか？そうですね...）



別のフレームのセットですが、配置が異なります。

<body bgcolor="#7714bf" background="img/bg-jamcentral.gif" text="#ffffff" link="#edb2fc" vlink="#edb2fc" alink="#edb2fc">

ここで、開発者は重要なことを行いました。背景画像（不透明）だけでなく、背景色も示しました。そうしないと、背景画像が読み込まれなかった場合、白い背景に白いテキストが表示されます。

（現代の開発とのもう1つの興味深い違い。今日、多くの設計者はすべてのリソースがダウンロードされることを想定しており、回避する必要がある不便さとしてダウンロードを提示することがあります。しかし、メインから10メートルのサンフランシスコからの光ファイバー接続に座っているすべてとは限りません。チャネル）。

しかし、ページ自体に戻ります。サムネイルグリッドは、当時の古典的なWebデザインの問題でした。主な問題は、写真を並べて配置することです、そしてデフォルトではHTMLはそれらを1つの大きな列に積み上げます。当時、設計者が持っていたツールは1つだけでした。それはテーブルです。当サイトでは、次のように構成されています。

<table cellpadding=10>
<tr><td align=center><a href="..."><img src="..."></a></td>...</tr>
<tr>...</tr>
<tr>...</tr>
<table>

ブラウザーの裁量でフォーマットされた3×3のサムネイルグリッド（別の行の最後の画像は、実際には表の一部ではありません）列の数は画面の幅に依存しませんが、当時は誰もが小さな画面を持っていたため、これはデザイナーをあまり気にしませんでした。ここでは画像にキャプションはありませんが、それぞれの画像は個別のセルに配置されているため、簡単に追加できます。

これは、1996年に最も進歩したサムネイル表示アプローチでした。この小さなUIパズルにさらに数回戻ります。実際の例（およびサンプルマークアップ付きのソースコード）は、別のページにあります。

当時の通常のレイアウト


それが私たちが始めたところです、そしてそれは吸いました。いくつかのページで共通のスタイルが必要な場合、可能性は非常に限られており、コピーと貼り付けが唯一のツールでした。 Space Jam Webサイトは、他の多くのように、デザインの一貫性をまったく気にしないことを選択しました。

次にCSSが登場しました。それはだった本当の奇跡。これらのコードはすべてのページで繰り返され、すべて消えています。特定の色のすべてのトップレベルの見出しを作りたいですか？問題ない：

H1 {
    color: #FF0000;
}

バム！そして、それはすべてです。<h1>ドキュメントに見出しがいくつあっても、それぞれの見出しは赤く焦げており、もう一度考える必要はありません。さらに良いことに、このフラグメントを自分のファイルに入れ、この疑わしい審美的な選択をサイトのすべてのページにほとんど労力をかけずに適用できます。同じことがあなたの壮大なタイルの背景、リンクの色、テーブルのフォントサイズにも当てはまります。

（タグのコンテンツを<style>HTMLコメントで囲むことを忘れないでください。そうしないと、CSSをサポートしていない古いブラウザでは、タグがテキストとして表示されます）。

タグを連続してすべてに適用する必要はありません。 CSSは「クラス」と「ID」の概念を導入したため、スタイルは特別にマークされた要素にのみ適用されます。タイプセレクターP.importantクラスの段落にのみ適用されます<P CLASS="important">が#header、影響するだけです<H1 ID="header">（違いは、識別子はドキュメント内で一意でなければならず、クラスは何度でも使用できることです）。これらのツールを使用して効果的に独自のタグを作成し、サイトに固有のHTMLの個別のバージョンを取得します。

それは大きな前進でしたが、当時はレイアウトにCSSを使用することを考えていた人はいませんでした。CSS 1勧告が96年12月に公開されたとき、レイアウトに触れることはほとんどありませんでしたが、既存の HTML関数を、それらが添付されているタグから分離するだけでした。私たちが持っていたフォントの色や背景のおかげへ<FONT COLOR>と<BODY BACKGROUND>。少なくともページ上のオブジェクトの場所にリモートで影響を与えた唯一のことは、新聞のように、画像を横に引き、テキストの折り返しを含めたfloat同等のプロパティでした<IMG ALIGN>。それほど印象的ではありません。

しかし、これは驚くべきことではありません。 HTMLには、テーブルを除いて実際のレイアウトツールはありませんでした。テーブルのプロパティは、CSSで一般化したり、タグ構造に関連付けるのが難しいため、CSS 1に追加するものはありませんでした。たとえば、タグを使用して<FONT>、Webデザインを少しだけ自動化しました。面倒でエラーが発生しにくくなり、ページコードがより簡潔になり、保守が容易になりました。かなり良い前進であり、誰もが喜んでそれを受け入れましたが、テーブルはレイアウトのメインツールのままでした。

互換モード


ブラウザでの初期のCSS実装はすべてバグでいっぱいでした。おそらく最も有名なのは、コンテナモデル（ボックスモデル）のバグです。

ご覧のとおり、コンテナ（要素が占める長方形のスペース）にはいくつかの寸法があります。それ自体の幅と高さ、パディング、オプションのボーダー、そして隣接するコンテナからそれを分離するフィールドです。 CSSは、これらの距離がすべて重なり合うことを示しています。たとえば、次のスタイルのコンテナ：

    width: 100px;
    padding: 10px;
    border: 2px solid black;

...境界から境界まで、幅124ピクセルになります。

ただし、IE 4とNetscape 4は異なるアプローチを採用しました。幅と高さを境界線から境界線への寸法と見なし、そこから境界線とマージンを差し引いて要素自体の幅を取得しました。これらのブラウザの同じフィールドの幅は、境界線から境界線まで100ピクセルで、コンテンツには76ピクセルが残ります。

IE 6は、この仕様の矛盾を修正することを決定しました。残念ながら、変更を加えるだけでは、IEとNetscapeの両方で正常に機能する多くのWebサイトのデザインが壊れてしまいます。

したがって、IEチームは非常に奇妙な妥協案を思い付きました。彼らは古い動作（およびその他のいくつかの主要なエラー）を「互換性モード」（quirksモード、文字通り「quirksでの作業」）として宣言し、デフォルトでオンにしました。新しい「厳密」または「標準モード」に切り替えるには、ドキュメントの先頭のタグの前に「doctype」を指定する必要がありました<html>。それはこのようなものでした：

<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01//EN" "http://www.w3.org/TR/html4/strict.dtd">

長年にわたって、誰もがこの愚かな行を各HTMLドキュメントの先頭に挿入しなければなりませんでした（後のHTML5はそれをに簡略化します<!DOCTYPE html>）。考えてみれば、これは適切なCSS動作を選択するための非常に奇妙な方法です。 doctype宣言は90年代半ばからHTML仕様の一部となっていますが、誰も使用していません。 Microsoftのアイデアは、独自の拡張機能を導入せずに選択できるようにすることだったと思います。このような宣言を使用すると、最初は間違っていた動作を回避できます。 IEチームのための便利な出口！

最近のブラウザーには、1つの癖をエミュレートする「ほぼ標準」モードもあります。おそらくこれは2番目にスキャンダラスな機能です。表のセルに画像のみが含まれている場合は、下のスペースが削除されます。通常のCSSルールに従って、画像はテキストの（空の）行にあります。そのため、「y」などの文字の線間要素の場合、下にスペースが必要です。初期のブラウザはこれを正常に処理できませんでしたが、2000年以降に作成された一部のWebサイトはこの機能に依存していましたが、他のすべては完全に準拠していました。たとえば、大きな画像を分割して、テーブルのセルに相互に配置しました。標準に従って、空のスペースを追加すると、画像が崩れます。

しかし、過去に戻ります。規格は正式に普及しましたが、新しい問題が発生しました。IE 6がWebを支配し、DOCTYPE宣言はオプションであるため、ブラウザーの「厳密モード」でこれらの標準をサポートすることを考える必要はありませんでした。他のブラウザは最終的にそれをエミュレートし、カスタム動作は事実上の標準になりました。そのようなことを行ったWebデザイナー（私たちのおかげで、私たちの多くはそうでした）は、厳格モードを有効にするために注目度の高いキャンペーンを開始しました。
