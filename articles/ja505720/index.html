Docker Apache Kafka + Zookeeperイメージの選択


まず、コンテナのApache Kafkaイメージベンダーを決定する必要があります。特定の各プロジェクトの要件は、ソリューションの必要なセキュリティと信頼性のレベルで異なります。もちろん、場合によっては、独自のイメージを組み立てる必要がありますが、ほとんどのプロジェクトでは、Dockerハブで使用可能なイメージを選択するのが賢明です。2020年には、3つの主な人気ベンダー画像kafka + zookeperがあります。

  • コンフルエントイメージ [主に、大きなプラス-それらはApache Kafka自体の開発者によって提供され、情報セキュリティの面で最も信頼性が高い]
  • bitnamiからの画像 [さらに、合流性と比較して、画像は最小限の構成を必要とし、「箱から出して」開始する]
  • wurstmeister [ , github hub.docker, ]

Docker-compose


イメージを起動して構成するには、docker-composeを使用します。これにより、マルチコンテナーアプリケーションを簡単に実行できます。kafkaの最小構成では、多数の動物園管理者と少なくとも1つのブローカーが提供されるため、これなしでは実現できません。

omposeはmac / windows-Dockerディストリビューションにバンドルされているため、必要な環境をインストールすることは、docs.docker.com/engine/installからディストリビューションをダウンロードしてインストールすることになります。

ただし、Linuxユーザーは個別にインストールする必要があります。docker + composeインストールガイドについては、公式のdocs.docker.com/compose/installドキュメントを参照することをお勧めします。

Spring Boot上のJavaクライアント


kafkaからメッセージをパブリッシュおよび読み取る機能を備えた最小限のクライアントを作成します。これを行うには、Spring Initializrまたは使い慣れたその他のツールを使用して新しいプロジェクトを作成します。 1つの依存関係を接続するだけで十分です。Javaコードの最小限の構成で構成yamlファイルを使用して、



可能な限りSpring Bootアプリケーションを構成することを好みます。 Yaml形式の組み込み構造により、これらのファイルは.propertiesファイルよりも人間の認識にとってはるかに便利であり、モジュールごとに設定をグループ化できます。 .yml構成の唯一の欠点は、テストでサポートされていないことですが、組み込みのブートツールによって簡単にバイパスされます。
yamlで書き換えた場合、古い.propertiesファイルの認識がどのように変化するか（特に大規模なプロジェクトの場合）を確認することを強くお勧めします。これは、www.toyaml.com / index.htmlで数回クリックするだけで実行できます。
