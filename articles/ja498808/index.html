言語サーバー


言語サーバー（言語サーバー）は、特別なプロトコルである言語サーバープロトコル（LSP）を使用して相互にやり取りするエディターとIDEのプログラミング言語の可能性について話します。これは優れたアプローチです。これにより、1つの実装で多数のエディター/ IDEにデータを一度に提供できます。Gopls（Golangの言語サーバー）と、vimでの使用方法についてはすでに書きました。 YAMLでKubernetesのオートコンプリートを取得する手順は似ています。 vimを上記の方法で機能させるには、言語サーバークライアントをインストールする必要があります。私が知っている2つの方法は、LanguageClient-neovimとcoc.vimです。



。検討する記事でcoc.vimは、これが現時点で最も人気のあるプラグインです。vim-plugからインストールできます：

" Use release branch (Recommend)
Plug 'neoclide/coc.nvim', {'branch': 'release'}

" Or build from source code by use yarn: https://yarnpkg.com
Plug 'neoclide/coc.nvim', {'do': 'yarn install --frozen-lockfile'}

開始するにはcoc（したがってyaml-language-server）、node.jsをインストールする必要があります。

curl -sL install-node.now.sh/lts | bash

coc.vim構成し たらcoc-yaml、vimからサーバー拡張機能をインストールします。

:CocInstall coc-yaml



最後に、ほとんどの場合、例としてcoc-vim示した構成から始めます。特に、<Ctrl> +スペースの組み合わせをアクティブにして、オートコンプリートを呼び出します。

yaml-language-serverディスカバリーの構成


cocyaml-language-server を使用するには、YAMLファイルを編集するときにKubernetesからスキームをダウンロードするように依頼する必要があります。これは次のように編集して行いcoc-configます。

:CocConfig

構成では、kubernetesすべてのファイルを追加する必要がありますyaml。さらにの言語サーバーを使用するgolangので、私の一般的な構成は次のようになります。

{
  "languageserver": {
      "golang": {
        "command": "gopls",
        "rootPatterns": ["go.mod"],
        "filetypes": ["go"]
      }
  },

  "yaml.schemas": {
      "kubernetes": "/*.yaml"
  }
}

kubernetes- この定数で定義されたURLを使用してKubernetesスキームをダウンロードする必要があることを言語サーバーに通知する予約済みフィールド。yaml.schemas追加のスキームをサポートすることで拡張できます。詳細については、関連ドキュメントを参照してください。

これで、YAMLファイルを作成して、オートコンプリートの使用を開始できます。押すと、<Ctrl>キー+スペース（またはvimの中で構成された別の組み合わせ）は、現在のコンテキストに基づいて利用可能なフィールドとドキュメントを表示する必要があります：


<Ctrl>キー+スペースがここに動作しますが、私はそれを構成しているためinoremap <silent><expr> <c-space> coc#refresh()。まだ行っていない場合は、coc.nvim READMEで構成例を確認してください。

Kubernetes APIバージョンの選択


この記事の執筆時点では、yaml-language-serverにはKubernetes 1.14.0スキーマが付属しています。スキームを動的に選択する方法が見つからなかったため、対応するGitHubの問題を開きました。幸い、言語サーバーはtypescriptで記述されているため、手動でバージョンを変更するのは非常に簡単です。これを行うには、ファイルを見つけますserver.ts。

お使いのマシンでそれを見つけるには、単にvimでYAMLファイルを開き、でプロセスを見つけますyaml-language-server。

ps aux | grep -i yaml-language-server

joshrosso         2380  45.9  0.2  5586084  69324   ??  S     9:32PM   0:00.43 /usr/local/Cellar/node/13.5.0/bin/node /Users/joshrosso/.config/coc/extensions/node_modules/coc-yaml/node_modules/yaml-language-server/out/server/src/server.js --node-ipc --node-ipc --clientProcessId=2379
joshrosso         2382   0.0  0.0  4399352    788 s001  S+    9:32PM   0:00.00 grep -i yaml-language-server

プロセス2380は私たちに関連しています：YAMLファイルを編集するときにそれを使用するのはvimです。

ご覧のとおり、ファイルはにあり/Users/joshrosso/.config/coc/extensions/node_modules/coc-yaml/node_modules/yaml-language-server/out/server/src/server.jsます。これを編集して、値KUBERNETES_SCHEMA_URLをたとえばバージョン1.17.0 に変更するだけで十分です。

// old 1.14.0 schema
//exports.KUBERNETES_SCHEMA_URL = "https://raw.githubusercontent.com/garethr/kubernetes-json-schema/master/v1.14.0-standalone-strict/all.json";
// new 1.17.0 schema in instrumenta repo
exports.KUBERNETES_SCHEMA_URL = "https://raw.githubusercontent.com/instrumenta/kubernetes-json-schema/master/v1.17.0-standalone-strict/all.json";

使用するバージョンによってcoc-yaml、コード内の変数の場所は異なる場合があります。また、私はからリポジトリを変更したことに注意garethrしますinstrumenta。garethr彼はそこでサポート回路に切り替えたようです。

変更が有効になったことを確認するには、[Kubernetesの以前のバージョン]になかったフィールドが表示されるかどうかを確認します。たとえば、K8s 1.14の回路にはstartupProbeがありませんでした。
