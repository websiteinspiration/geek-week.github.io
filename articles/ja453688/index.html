Javaは流血型の企業であるだけでなく、遅延に敏感な高速アプリケーションでもあります

私はライファイゼンバンクでアルゴリズム取引をしています。これは銀行セクターのかなり特定の領域です。私たちは、低く予測可能な遅延で機能する取引プラットフォームを作ります。アプリケーションの成功は、とりわけアプリケーションの速度に依存するため、トレーディングに関与するスタック全体（プライベートネットワークチャネル、特別なハードウェア、OS設定、特別なJVM、そしてもちろんアプリケーション自体）に対処する必要があります。アプリケーション自体の最適化のみを停止することはできません。OSやネットワークの設定も同様に重要です。これには、データがスタック全体をどのように流れ、遅延が発生する可能性があるかを理解するための技術的な専門知識と経験が必要です。


すべての組織/銀行がこのクラスのソフトウェアの開発を可能にするわけではありません。しかし、そのようなプロジェクトがライファイゼンバンクの壁の中で立ち上げられたことは幸運であり、私は適切な専門分野を持っていました-私はインテルインテルコンパイラーラボラトリーでコードパフォーマンスを専門にしました。C、C ++、Fortran用のコンパイラを作成しました。ライファイゼンバンクでは、Javaに切り替えました。以前に多くの人々が使用するある種のツールを作成した場合、今はバリケードの反対側に移動し、コードだけでなくアプリケーションスタック全体のパフォーマンスの応用分析に従事しています。通常、パフォーマンスの問題を調査する方法は、カーネルやネットワークの設定など、コードをはるかに超えています。

Javaは高負荷用ではありませんか？


Javaは高負荷システムの開発にはあまり適していないと広く信じられています。
これには部分的にしか同意できません。 Javaは多くの面で美しいです。 C ++のような言語と比較すると、潜在的なオーバーヘッドが高くなる可能性がありますが、C ++の機能的に類似したソリューションの動作が遅くなる場合があります。 Javaでは自動的に機能するが、C ++では機能しない、またはその逆の最適化があります。 Java JITコンパイラーの後に続くコードの品質を見ると、パフォーマンスは、ピーク時に達成できるパフォーマンスよりも数倍も劣らないと信じたいと思います。しかし、同時に、私は非常に迅速な開発、優れたツール、および豊富な既製のコンポーネントを入手できます。

それに直面しよう：C ++の世界では、開発環境（IDE）はIntelliJとEclipseの背後に大幅にあります。開発者がこれらの環境のいずれかを使用する場合、デバッグ速度、バグの発見、および複雑なロジックの記述は桁違いに高くなります。その結果、Javaを適切な場所にファイリングする方が、最初からすべてをC ++で非常に長い時間実行するよりも速く動作するようになるので簡単であることがわかりました。最もおもしろいのは、競争力のあるコードを書くとき、JavaとC ++の両方での同期のアプローチが非常に似ていることです。それらはOSレベルのプリミティブ（たとえば、synchronized / std :: mutex）またはIronプリミティブ（Atomic * / std :: atomic <*>）のどちらかです。。そして、この類似性を確認することは非常に重要です。

一般的に、私たちは非高負荷アプリケーションを開発しています））

高負荷アプリケーションと低遅延アプリケーションの違いは何ですか？


高負荷という用語は、作業の詳細を完全に反映しているわけではありません。私たちは、遅延に敏感なシステムに取り組んでいます。違いはなんですか？負荷の高いシステムでは、ハードウェアリソースを最大限に活用して、平均的にかなり早く作業することが重要です。実際には、これは、システムへの100分の1、1000分の1、..

遅延のレベルが重要なシステムを扱っています。私たちの仕事は、システムが常に予測可能な応答を持つようにすることです。潜在的にレイテンシの影響を受けやすいシステムまれにしかイベントが発生しない場合は、負荷が高くない可能性がありますが、保証された応答時間が必要です。そして、それは彼らの開発を容易にしません。全く逆です！危険はどこでも待ち構えています。現代のハードウェアとソフトウェアの圧倒的多数のコンポーネントは、「平均して」良い仕事を志向しています。スループットのため。

少なくともデータ構造を取る。私たちはハッシュテーブルを使用しており、データ構造全体の再ハッシュがクリティカルパスで発生した場合、これにより、1つのリクエストで特定のユーザーに具体的なブレーキがかかる可能性があります。またはJITコンパイラ-最も頻繁に実行されるコードパターンを最適化し、ほとんど実行されないコードパターンを悲観化します。しかし、この特定のまれなケースの速度は、私たちにとって非常に重要です。

多分このコードは珍しい種類の注文を処理しますか？または迅速な対応を必要とするいくつかの異常な市場状況？これらの潜在的にまれなイベントに対するシステムの反応が、予測可能で、できれば非常に短い時間がかかることを確認しようとしています。

予測可能な反応時間を達成するには？


この質問に2つのフレーズで答えることはできません。最初の近似では、同期の種類（同期、リエントラントロック、またはjava.util.concurrentからの同期）があるかどうかを理解することが重要です。多くの場合、busy-spin'ahで同期を使用する必要があります。同期プリミティブの使用は常に妥協です。そして、これらの同期プリミティブがどのように機能するか、およびそれらがもたらすトレードオフを理解することが重要です。また、特定のコードが生成するガベージの量を評価することも重要です。ガベージコレクタを処理する最善の方法は、それをトリガーしないことです。ガベージの生成が少ないほど、ガベージコレクターを実行する頻度が少なくなり、システムが介入せずに長く動作します。

また、平均的な指標だけでなく、さまざまなツールを使用して分析することもできます。システムが100回に1回、1000回に1回の割合で動作する速度を非常に慎重に分析する必要があります。明らかに、これらの指標は中央値または平均よりも悪くなります。しかし、私たちがどれだけ知っているかは非常に重要です。これを示すために、たとえばGrafana、Prometheus、HDRヒストグラム、JMHなどのツールが役立ちます。

Unsafeを削除できますか？


多くの場合、謝罪者が文書化されていないAPIと呼ぶものを使用する必要があります。有名なUnsafeについて話している。安全でないことが、JavaマシンのパブリックAPIの事実上の一部になっていると私は思います。それを否定しても意味がありません。 Unsafeは、私たち全員が積極的に使用している多くのプロジェクトを使用しています。そして、それを拒否すると、これらのプロジェクトはどうなりますか？古いバージョンのJavaを使用するか、これらすべてを書き直すために多くのエネルギーを費やす必要があります。コミュニティはこの準備ができていますか？生産性の数十パーセントを失う可能性がありますか？そして最も重要なのは、何と引き換えに？

間接的に、私の意見は非常にきちんとした削除を確認しますUnsafeのメソッド-Java11では、Unsafeの最も役に立たないメソッドが削除されました。Unsafeを使用するすべてのプロジェクトの少なくとも半分が別のプロジェクトに移行するまで、Unsafeは何らかの形で利用可能になると思います。

意見があります：銀行+ Java =流血の骨化した企業？


私たちのチームにはそのような恐怖はありません。春には、おそらく私がさらに10行書いたでしょう））大きなテクノロジーを使わないようにしています。私たちはそれを実現し、制御し、必要に応じて変更できるように、小さくて、きちんと、そして速く行うことを好みます。後者は、非標準の要件の対象となるシステム（私たちのものなど）にとって非常に重要です。これは、フレームワークのユーザーの90％の要件とは異なる場合があります。また、大規模なフレームワークを使用する場合、コミュニティにニーズを伝えたり、独自に動作を修正したりすることはできません。

私の意見では、開発者は常に利用可能なすべてのツールを使用できる必要があります。私はC ++からJavaの世界にやって来ましたが、ランタイムを開発する人々へのコミュニティの分割には非常に衝撃を受けました仮想マシン/コンパイラまたは仮想マシン自体とアプリケーション開発者。これは、標準のJDKクラスコードで明確に見られます。多くの場合、JDKの作成者は別のAPIを使用します。潜在的に、これはピークパフォーマンスを達成できないことを意味します。一般に、標準ライブラリとアプリケーションコードの両方を作成するために同じAPIを使用することは、プラットフォームの成熟度を示す優れた指標であると思います。

他の何か


スタック全体ではないにしても、Javaコード、バイトコード、仮想マシンランタイムの内部とアセンブラ、ハードウェア、OS、ネットワークなど、スタック全体が機能する方法を知ることがすべての開発者にとって非常に重要だと思います。これにより、問題をより広く見ることができます。
パフォーマンスも言及する価値があります。平均に焦点を合わせず、常に中央値と高いパーセンタイル（最悪の10/100/1000 / ...測定値）を見ることが非常に重要です。
