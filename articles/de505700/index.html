Ich werde Java-Programmierer interviewen. Eine Reihe von Fragen für jedes Level

In meiner Karriere musste ich Entwickler und Manager sein und sogar einmal „CTO“ in einem kleinen Startup. Und natürlich musste ich eine große Anzahl von Interviews mit Cadidates führen. Deshalb habe ich für mich selbst solche Regeln für die Durchführung technischer Interviews entwickelt:

  • Stellen Sie allen Kandidaten die gleichen Fragen
  • Fragen sollten so sein, dass sowohl Junior als auch Senior sie beantworteten, aber auf unterschiedliche Weise,
  • Nach drei Fragen sollte das Niveau des Bewerbers verstanden werden.

Und so bekam ich diese drei Fragen für Java-Entwickler.

Erzählen Sie uns von HashMap


Ja, ja, die Standardfrage, mit der 50% der Interviews beginnen. Warum frage ich ihn? Weil es sehr, sehr unterschiedlich beantwortet werden kann.

  • Die ideale Antwort beginnt mit der Erklärung des Problems. Warum brauchen wir eine solche Struktur? Welche Art von Aufgabe löst es? Wie sieht die triviale Lösung für dieses Problem aus und warum ist es schlecht?
  • "Bonuspunkte" für einen historischen Ausflug, Kenntnis des Autors der Lösung, mindestens das ungefähre Jahr ihres Auftretens (natürlich nicht in Java)
  • Die Antwort von mindestens dem Junior muss Möglichkeiten zur Verwendung der Struktur enthalten. Welche Methoden unterstützt sie?
  • Der mittlere Entwickler muss das interne Gerät sowie eine Beschreibung der Verträge der Funktionen hashCode () und equals () sowie deren funktionale Anforderungen verstehen
  • Bonuspunkte für die Beschreibung nicht funktionaler Anforderungen für hashCode () und equals ()
  • Bonuspunkte für die Verwendung von HashMap in einer Umgebung mit mehreren Threads.
  • "Bonuspunkte" für die Prüfung von HashMap-Alternativen - Auflistung, Vergleich der Komplexität von Implementierung, Verwendung, Geschwindigkeit im Allgemeinen, Ideal- und Worst-Cases.
  • "Bonuspunkte" für HashMap-Alternativen in Form separater Systemkomponenten (z. B. Datenbanken und "externe" Caches).
  • "Bonuspunkte" zur Beschreibung, wie HashMap in Datenbanken verwendet wird und warum normalerweise binäre baumbasierte Indizes anstelle von HASH-Indizes verwendet werden.

Wie Sie sehen können, gibt es eine sehr große Anzahl von "Bonusoptionen", mit denen Sie mit senior'om sprechen können. Und über wie viele Punkte können Sie mindestens 10 Minuten sprechen?

Wenn Sie sich langweilen, ersetzen Sie HashMap durch TreeMap.

SQL schreiben ...


Ja, und wieder die "Standard" -Frage, die vor 10 Jahren bei jedem Interview gestellt wurde. Jetzt ist es weniger verbreitet, anscheinend aufgrund der Mode für nosql. Die Fähigkeit, mit Daten zu arbeiten, ist jedoch immer erforderlich, und Programmierer, die das kartesische Produkt verstehen , sind immer nützlicher (und teurer) als diejenigen, die sich auf das allgemeine Nacherzählen des Lehrbuchs beschränken.

Und die Frage ist:
Die Datenbank enthält zwei Tabellen: Schüler und Gruppen. Jeder Schüler hat eine Kennung für die group_id-Gruppe, zu der er gehört. Schreiben Sie eine SQL-Abfrage, ohne verschachtelt zu verwenden, und listen Sie die Gruppen (oder die Anzahl der Gruppen) ohne einen einzelnen Schüler auf

Die richtige Antwort enthält natürlich nicht nur JOIN, sondern LEFT JOIN, wenn der Wert mit NULL verglichen wird. Am interessantesten ist es, mit dem Kandidaten darüber zu sprechen, welche Fehler beim Schreiben einer Anfrage gemacht werden können und warum die Anfrage nicht funktioniert, wenn sie gestellt wird. Warum kann man nicht durch Gleichheit mit NULL vergleichen? Warum funktioniert ein einfacher JOIN nicht?

Ein Bonus für Indizes, die den Betrieb dieser Abfrage beschleunigen und SQL-Optionen in verschiedenen Dialekten (verschiedenen DBMS) berücksichtigen.

Finden Sie Fehler im Code


Dem Kandidaten wird die Aufgabe folgendermaßen geäußert:
Angenommen, Sie erhalten eine Pull-Anfrage (Merge Request) von einem anderen Entwickler für eine Codeüberprüfung. Diese Anfrage enthält eine große Anzahl verschiedener Dateien, von denen eine wie folgt aussieht (der gesamte Klassencode wird angegeben):

package ru.mycompany.utils;

/** Wrap around Lock to simplify interface */
class LockWrapper {
    private Lock lock;
    public wait() {
        this.lock.wait()
    }
}

Haben Sie Kommentare zu diesem Code und wenn ja, welche? Überspringen Sie die Anforderung mit einer solchen Datei weiter (zum Hauptzweig) oder benötigen Sie eine Wiederholung und warum?

Natürlich gibt es hier mindestens drei Ebenen.

  • Jeder Junior sollte auf die Syntaxfehler in diesem Code hinweisen. Der Entwickler muss angeben, warum der Code nicht kompiliert wird. Und das Problem liegt natürlich nicht nur in der Syntax.
  • Der Entwickler muss wissen, warum der Code auch dann nicht funktioniert, wenn er kompiliert wird. Und wie man diesen Code korrigiert, damit er funktioniert. Der Entwickler muss in der Lage sein, sowohl das Synchronisierungskonstrukt als auch die Klassen aus dem Paket java.util.concurrent zu verwenden und zu wissen, warum eine Sekunde nicht die dritte ist.
  • Ein idealer Entwickler sollte zunächst fragen, warum dieser Code benötigt wird, welches Problem er löst und ob er ihn lösen kann, selbst wenn er alle vorhandenen Probleme in diesem Code behebt.

Während der Beantwortung bringen wir zusammen mit dem Kandidaten den Code in die „ideale“ Form. Die endgültige Form hängt natürlich von der Qualifikation des Bewerbers ab. Die besten meiner Kandidaten haben hier 13 Fehler gefunden. Abgesehen vom „wichtigsten Fehler“.

Als Bonus können Sie darüber sprechen, welche Mittel der automatischen Codesteuerung verwendet werden sollten, um zumindest einen Teil der vorhandenen Fehler zu finden.

Anstelle einer Schlussfolgerung


Ein Rat, den ich allen geben möchte, die zukünftige Kollegen interviewen müssen. Versuchen Sie nicht zu antworten, ob die Person geeignet ist oder nicht. Formulieren Sie Ihre Meinung entweder in Form einer Note (falls Ihr Unternehmen sie angenommen hat) oder in Form des Geldbetrags, den Sie bereit sind, dem Kandidaten für die Arbeit in Ihrem Unternehmen zu zahlen. Dies vereinfacht zum einen die Entscheidung für Sie, da sie nicht schwarz oder weiß ist, und zum anderen für jemanden, der die endgültige Entscheidung trifft oder die Ergebnisse mehrerer Interviews vergleicht.
