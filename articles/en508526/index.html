In JavaScript, it is possible to use the operator to get the remainder of division to floating point numbers. It works like this: 



3.14 % 5 // 3.14
13.14 % 5 // 3.1400000000000006


At the same time, numbers are of type Number and are stored in 64 bits in accordance with the IEEE 754 standard. In .NET, this type has a twin brother System.Double or just double. If a numeric literal contains a floating point, then it defaults to double. The intention can be expressed explicitly by adding the suffix d or D to the number. The ability to divide double with the remainder is also available. So try. And ... Bingo! 



Console.WriteLine(3.14d % 5); // 3,14
Console.WriteLine(13.14d % 5); // 3,1400000000000006

In the following example, the condition is true. 



const x = { i: 1, toString: function() { return this.i++; } };
if (x == 1 && x == 2 && x == 3)
    document.write("This will be printed!");


This is achieved due to the fact that when checking for equality, the operands are cast to the same type. In this case, an overridden toString () method is called on x, which, in addition to returning something, changes the state of the object. We note to ourselves that pure functions are wonderful, but for the sake of interest we are trying to implement the concept from the example. 



Trickster x = new Trickster();
if (x == 1 && x == 2 && x == 3)
    Console.WriteLine("This will be printed!");


.NET allows you to override methods and casts, where possible, if the corresponding cast is defined. 



class Trickster
{
    private int value = 1;

    public override string ToString() =>
        value++.ToString();

    public static implicit operator int(Trickster trickster) =>
        int.Parse(trickster.ToString());
}


Hooray, it works. But it confuses that Trickster seems to adapt to int. Replace 



public static implicit operator int(Trickster trickster) =>
    int.Parse(trickster.ToString());


on the 



public static implicit operator double(Trickster trickster) =>
    double.Parse(trickster.ToString());


It works anyway. Now Trickster and int are cast to double.



Actually, redefinition of ToString () is not essential for achievement of the final result and is made for the maximum imitation of an example in JavaScript. It is enough to define a cast to double. Of course, not forgetting the importance of the side effect. 



class Trickster

void SetTimeout(Action action, int delay) =>

Nobody promised commutativity

Date() && {property: 1}; // {property: 1}
{property: 1} && Date(); // Uncaught SyntaxError: Unexpected token '&&'


In C #, operators are generally not applicable to user-defined types by default (exception == and! = For reference types). But some of them can be overloaded explicitly in type. Then the concern for commutativity rests with the developer. 



class Trickster
{

        null;

Crossing a hedgehog with a hedgehog



In JavaScript, you can perform mathematical operations together on strings and numbers. 



var a = "41";
a += 1; // "411"
var b = "41";
b -=- 1; // 42


In C #, this is only possible with addition. 



var a = "41" + 1; // 411

// Compilation Error:
// Operator '-' cannot be applied to operands of type 'string' and 'int'
var b = "41" - (-1);


But there are no obstacles to the patriots of the language. By adding a little sorcery, you can make even this code work ‚Äúcorrectly‚Äù: 



Trickster a = "41";
Console.WriteLine(a += 1); // 411
Trickster b = "41";
Console.WriteLine(b -=- 1); // 42


You just need to implement a pair of implicit type conversions and overload the pair of statements. 



class Trickster
{
    private string value;

    public static implicit operator Trickster(string s) =>
        new Trickster { value = s };

    public static implicit operator Trickster(int i) =>
        new Trickster { value = i.ToString() };

    public static string operator +(Trickster trickster, int i) =>
        trickster.value + i;

    public static int operator -(Trickster trickster, int i) =>
        int.Parse(trickster.value) - i;

    public override string ToString() =>
        value;
}


You can replace addition overload 



public static string operator +(Trickster trickster, int i) =>
    trickster.value + i;


to implicitly cast to string 



public static implicit operator string(Trickster trickster) =>	
    trickster.value;


and get the same result.



If you do not bother with returning the number from the subtraction operation and placing this number in Trickster, then you can replace 



public static int operator -(Trickster trickster, int i) =>	
    int.Parse(trickster.value) - i;


on the 



public static string operator -(Trickster trickster, int i) =>
    (int.Parse(trickster.value) - i).ToString();


and the int cast to Trickster is removed.



Juggling Bananas



In JavaScript, the string "banana" can be obtained in the following way: 



('b' + 'a' + + 'a' + 'a').toLowerCase(); // "banana"
('b'+'a'++'a'+'a').toLowerCase(); // Uncaught SyntaxError: Invalid left-hand side expression in postfix operation


Here, applying unary + to the second 'a' returns NaN, which is cast to the string 'NaN' for addition with the rest of the strings, and the result is 'baNaNa', and for the sake of beauty everything is reduced to lowercase.



In C # we create a class 



class Trickster
{
    private string value;

    public static implicit operator Trickster(string s) =>
        new Trickster { value = s };

    public static double operator +(Trickster trickster) =>
        double.TryParse(trickster.value, out double result) ? result : double.NaN;
}


and try 



//  double.NaN.ToString()  "NaN",   " "
Thread.CurrentThread.CurrentCulture = new CultureInfo("en-us");
Console.WriteLine(("b" + "a" + + (Trickster)"a" + "a").ToLower());


Unfortunately, despite the implicit casting, the compiler cannot build the string string -> Trickster -> double -> string, and you have to explicitly tell him. (If you think about it, such a cast would look more than strange.)



Trickster can be implemented differently: 



class Trickster
{
    private double value;

    public static implicit operator Trickster(string s) =>
        new Trickster { value = double.TryParse(s, out var d) ? d : double.NaN };

    public static double operator +(Trickster trickster) =>
        +trickster.value;
}


In this case, you can even replace the unary + overload with an implicit cast to double: 



public static implicit operator double(Trickster trickster) =>
    trickster.value;


As a bonus, if you do not put a space between the two pluses, then, as in the JavaScript example, we get a compilation error, not just one, but a whole bunch.
