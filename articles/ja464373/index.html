Androidのエッジツーエッジ：正しく実行する

過去のGoogle I / O 2019は多くのセンセーショナルな革新をもたらし、その多くは今後数年間でモバイル開発業界に影響を与えるでしょう。新たなトレンドをたどることは、やはり興味深いことでした。まず、歴史的に機械式の操作キーがダウンし、スマートフォンの画面が大きくなり、サイドフレームが目立たなくなりました。画面上のシステムボタンがジェスチャーに置き換わり、コンテンツを消費するためのスペースがますます増えています。アプリケーションは、ステータスバーとナビゲーションパネルの条件付き境界に制限されることなく、ディスプレイの下部から上部のフレームまで、表示されている表面全体に表示されます。私たちはエッジツーエッジ時代の瀬戸際にいます。



エッジツーエッジとは何ですか？文字通り、これは、ステータスバーや下部のナビゲーションボタンに制限されることなく、アプリケーションをディスプレイの下のフレームから上のフレームまでの表示面全体に表示する必要があることを意味します。

まず、Androidがシステムパネルをどのように描画するかを理解する必要があります。 Android 5.0以降、画面の水平エッジに沿ってシステムインデントを操作するための便利なAPIが提供されています。それらはWindowInsetsと呼ばれ、以下の図では赤く色分けされています。


また、Androidチームの開発者は、たとえばキーボードが表示されたときにこれらのインデントの変更をサブスクライブできるリスナーを追加しました。厳密に言うと、WindowInsetsは画面の端からのレイアウトファイルのマージンです。アクティビティ（分割画面モード、キーボードの外観）のサイズを変更すると、インセットも変更されます。したがって、エッジツーエッジをサポートするには、これらのインデントが存在しないことを確認する必要があります。WindowInsetsがnullの画面は次のようになります。


実装


この実装では、Windowとそのフラグを積極的に操作します。
すべての例はKotlinで記述されますが、拡張関数の代わりにユーティリティを使用して、Javaで簡単に実装できます。

ルートレイアウト要素で最初に行うことは、フラグを明示的に設定することです。

android:fitsSystemWindows="true"

これは、ルートビューがシステム要素の下に確実に描画されるようにするため、および変更をサブスクライブするときにインセットを正しく測定するために必要です。
次に、最も重要なことに目を向けます。画面の境界線を削除します。ただし、これは非常に慎重に行う必要があります。そしてそれが理由です：

  1. 下部のインセットをゼロにすると、キーボードの外観に対するウィンドウの反応が失われる危険があります。上部のインセットをリセットするためのStackOverflowには数十のヒントがありますが、下部のインセットは微妙に静かです。このため、NavigationBarが完全に透明になることはありません。下のインセットをリセットすると、adjustResizeフラグが機能しなくなります。

    解決策： インセットを変更するたびに、キーボードの下部の高さが含まれているかどうかを確認し、それ以外の場合にのみリセットしてください。
  2. インセットをリセットすると、ビューの表示部分がステータスバーとナビゲーションバーの下に表示されます。マテリアルデザイン（および常識）の概念によれば、アクティブな要素をシステム領域に配置することはできません。つまり、この領域には、ボタン、テキストを入力するためのフィールド、チェックボックスなどはありません。 

    解決策： リスナーにリスナーを追加します。これにより、WindowInsetsを変更するときに、システムインデントをアクティビティに変換し、ビューの適切なパディングとマージンを設定することで内部的に応答します。


この動作は許可しないでください（ステータスバーでツールバーがクロールします）。removeSystemInsets（）

関数   は次のようになります。

fun removeSystemInsets(view: View, listener: OnSystemInsetsChangedListener) {
    ViewCompat.setOnApplyWindowInsetsListener(view) { _, insets ->

        val desiredBottomInset = calculateDesiredBottomInset(
                view,
                insets.systemWindowInsetTop,
                insets.systemWindowInsetBottom,
                listener
        )

        ViewCompat.onApplyWindowInsets(
                view,
                insets.replaceSystemWindowInsets(0, 0, 0, desiredBottomInset)
        )
    }
}

calculateDesiredBottomInset（） 関数   は、現在のデバイス構成に応じて、キーボードの有無にかかわらず、下部のインセットを計算します。

fun calculateDesiredBottomInset(
        view: View,
        topInset: Int,
        bottomInset: Int,
        listener: OnSystemInsetsChangedListener
): Int {
    val hasKeyboard = isKeyboardAppeared(view, bottomInset)
    val desiredBottomInset = if (hasKeyboard) bottomInset else 0
    listener(topInset, if (hasKeyboard) 0 else bottomInset)
    return desiredBottomInset
}

isKeyboardAppeared（） メソッドを使用して、キーボードの高さを確認します  。 キーボードは画面の高さの4分の1未満しか占有できないという仮定に基づいています。必要に応じて、検証ロジックを好きなように変更できます。

private fun View.isKeyboardAppeared(bottomInset: Int) =
        bottomInset / resourdisplayMetrics.heightPixels.toDouble() > .25

 removeSystemInsets（） メソッド はリスナーを使用します。実際、これらはラムダ式の単なる型エイリアスです。その完全なコード：

typealias OnSystemBarsSizeChangedListener =

次のステップは、透明度をシステムバーに設定することです。

window.statusBarColor = Color.TRANSPARENT
window.navigationBarColor = Color.TRANSPARENT

上記のすべてをコンパイルしたら、次のメソッドを取得します。

fun Activity.setWindowTransparency(
        listener: OnSystemInsetsChangedListener = { _, _ -> }
) {
        InsetUtil.removeSystemInsets(window.decorView, listener)
        window.navigationBarColor = Color.TRANSPARENT
        window.statusBarColor = Color.TRANSPARENT
}

        // 
}

「エッジツーエッジ」効果は約100の異なる方法で実現できます（StackOverflow でのこのようなヒントの数はこれの明確な確認です）が、それらの多くは、Androidの異なるバージョンでの不正な動作につながるか、長時間表示する必要があるなどのパラメーターを考慮していませんリスト、またはキーボードを表示するときに画面のサイズ変更を中断します。



タールのスプーン


この記事で説明するソリューションは、現在のすべてのデバイスに適しています。実際は、Android Lollipop（5.0）以降のデバイスを意味します。彼らにとって、上記のソリューションは完全に機能します。ただし、Androidの古いバージョンの場合、当時はWindowInsetsについて何も知られていないため、独自の実装が必要になります。

Android KitKat（4.4）では、システムパネルの透明性が引き続きサポートされています。しかし、古いバージョンはそのような美しさをまったくサポートしていません。試すことすらできません。

Android 4.4でのインセットのシフトに焦点を当てましょう。これは、 fitSystemWindows（）メソッドで実行できます。。したがって、レイアウトのメイン要素は、現在のバージョンのAndroidの例のリスナーとまったく同じ実装を含む、オーバーライドされたfitSystemWindowsメソッドを持つコンテナーである必要があります。

class KitkatTransparentSystemBarsFrame @JvmOverloads constructor(
        context: Context,
        attrs: AttributeSet? = null,
        defStyleAttr: Int = -1
) : FrameLayout(context, attrs, defStyleAttr), KitkatTransparentSystemBarsContainer {

    override var onSystemInsetsChangedListener: OnSystemInsetsChangedListener = 
        { _, _ -> }

    override fun fitSystemWindows(insets: Rect?): Boolean {
        insets ?: return false
        val desiredBottomInset = InsetUtil.calculateDesiredBottomInset(
                this,
                insets.top,
                insets.bottom,
                onSystemInsetsChangedListener
        )
        return super.fitSystemWindows(Rect(0, 0, 0, desiredBottomInset))
    }

Android 4.4を搭載したデバイスでは、半透明フラグを設定することで部分的な透明性のみが機能します。

window.addFlags(
WindowManager.LayoutParams.FLAG_TRANSLUCENT_STATUS or
WindowManager.LayoutParams.FLAG_TRANSLUCENT_NAVIGATION
)

これらのフラグはシステムバーを半透明にし、わずかなグラデーションを追加しますが、残念ながら削除することはできません。ただし、このライブラリhttps://github.com/jgilfelt/SystemBarTintを使用すると、グラデーションを半透明のカラーバーに変換でき  ます。彼女は過去に一度以上私たちを救いました。5年前にライブラリに最新の変更が加えられたため、真のレトログラード都市にのみその魅力が明らかになります。

Kitkatのフラグ設定プロセス全体は次のようになります。

fun Activity.setWindowTransparencyKitkat(
        rootView: KitkatTransparentSystemBarsContainer,
        listener: OnSystemBarsSizeChangedListener = { _, _ -> }
) {
        rootView.onSystemBarsSizeChangedListener = listener
        window.addFlags(
                WindowManager.LayoutParams.FLAG_TRANSLUCENT_STATUS or
                        WindowManager.LayoutParams.FLAG_TRANSLUCENT_NAVIGATION
        )

}

これを念頭に置いて、どのバージョンのAndroidを搭載したデバイスでどのアプリケーションが実行されているかに関係なく、システムバーを透明（または少なくとも半透明）にすることができるユニバーサルメソッドを作成しています。

when {
        Build.VERSION.SDK_INT >= Build.VERSION_CODES.LOLLIPOP ->
        setWindowTransparency(::updateMargins)
        Build.VERSION.SDK_INT == Build.VERSION_CODES.KITKAT ->
        setWindowTransparencyKitkat(root_container, ::updateMargins)
        else -> { /*do nothing*/ }
}
