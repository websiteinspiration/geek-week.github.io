Linux 5.1のバグによりデータが失われました-修正パッチはすでにリリースされています

数週間前、SSDのデータ損失につながるバグがLinux 5.1カーネルバージョンで発見されました。最近、開発者はLinux 5.1.5パッチのパッチをリリースしました。これは「ギャップ」をパッチしました。

その理由について話し合います。


/ Unsplash / グレンキャリー

なんてバグ


年の初めに、開発者はLinux 5.1カーネルにいくつかの変更を加えました。その後、SamsungのSSDを搭載し、dm-crypt / LUKS暗号化とdevice-mapper / LVMを使用するシステムでエラーが発生し、データが失われました。しかし、問題は 5月中旬になって初めて知られるようになり、そのとき彼らはテーマ別フォーラムで積極的に議論を始めました。

それの-バグに直面して、少なくとも2人の意識パーティ郵送LKML最初のマイケル・ラス（マイケルLASS）、問題の報告、およびユーザー ArchLinuxを。

マイケル発売fstrimコマンド。マウントされたbtrfsボリュームで使用されなくなったデータブロックをドライブに通知します。次のシステムメッセージを受け取った後：

attempt to access beyond end of device
sda1: rw=16387, want=252755893, limit=250067632
BTRFS warning (device dm-5): failed to trim 1 device(s), last error -5

BTRFS warning (device dm-5): csum failed root 257 ino 16634085 off 21504884736 csum 0xd47cc2a2 expected csum 0xcebd791b mirror 1

その後、彼はbtrfsボリュームが破損し、物理デバイス上の残りの論理ボリュームが破壊されていることを発見しました。

ArchLinuxユーザーの場合、問題はLUKSの暗号化保護に影響しました。オペレーティングシステムを再起動してfstrimを実行した後、LUKSヘッダー（ボリュームの検索に使用される）が読み取り不能になり、暗号化された暗号化データを解読できませんでした。

理由は何ですか


問題はデバイスマッパー（DM）サブシステムで、そのタスクは仮想ブロックデバイスを作成することでした。LVM論理ボリュームマネージャー、ソフトウェアRAID、およびdm-cryptディスク暗号化システムを実装するために使用されます。

「fstrimチームは、max_io_len_target_boundary制限を考慮せずに一度に非常に多くのブロックをマークしました。その結果、まだ使用されているこれらのメモリセグメントは解放されます」と開発部門の責任者であるSergey Belkinはコメントしています1cloud.ru。「エラーはデバイスマッパーに関連していたため、理論上は、どのファイルシステムでもデータ損失が発生する可能性があります。」

パッチ


カーネル開発者は5月末にバグのパッチをリリースしました。drivers / md / dm.cファイルの4行のみが変更されました。対応するLinux 5.2カーネルにも対応する変更が加えられました（追加された行と削除された行はそれぞれ「+」と「-」でマークされます）：

 static int __send_changing_extent_only(struct clone_info *ci, struct dm_target *ti,
 				       unsigned num_bios)
 {
- unsigned len = ci->sector_count;
+  unsigned len;
 
@@ -1478,6 +1478,8 @@ static int __send_changing_extent_only(struct clone_info *ci, struct dm_target *
 	if (!num_bios)
 		return -EOPNOTSUPP;
 
+  len = min((sector_t)ci->sector_count, max_io_len_target_boundary(ci->sector, ti));
+

このパッチはArchLinux / ManjaroおよびFedoraディストリビューションの開発者によってすでに適用されています。Ubuntu 5.1ディストリビューションはLinux 5.1カーネルバージョンに翻訳されていなかったため、エラーには影響しませんでした。


/ Flickr / Andy Melton / CC BY-SA

パッチをインストールせずにデータ損失の状況を排除できます。次のコマンドを使用してfstrim.service / timerサービスを無効にするだけで十分です。

systemctl disable fstrim.timer
systemctl stop fstrim.timer

別のオプションは、fstabのマウント時にfstrim実行可能ファイルの名前を変更するか、破棄フラグを削除することです。dmsetupを介してLUKSで許可破棄を無効にすることもできます。ただし、これらの方法はすべて一時的なものであり、問​​題の本質を解決するものではありません。

初めてではない


Linuxカーネルでのコミットがメモリ破損の状況を引き起こすのは、これが初めてではありません。Linuxバージョン4.19でも同様のことが起こりました。その後、BLK-MQ I / Oスケジューラが原因でした。この問題は、デフォルトで設定されているCONFIG_SCSI_MQ_DEFAULT = yオプションを使用してカーネルをビルドしたときに明らかになりました。場合によっては、ボリュームデータが破損していました。

sed: error while loading shared libraries: /lib/x86_64-linux-gnu/libattr.so.1: unexpected PLT reloc type 0x00000107
sed: error while loading shared libraries: /lib/x86_64-linux-gnu/libattr.so.1: unexpected PLT reloc type 0x00000107

ほとんどの場合、問題はEXT4で明らかになりましたが、理論的には他のファイルシステムに影響を与える可能性があります。

次に、カーネルのメンテナの1人が、問題を解決する小さな修正を準備しました。ただし、同じバグが後でLinux 4.20ビルドで発見されました。2018年12月の終わりに、新しいグローバルアップデートでそれを取り除くことが可能になりました。
