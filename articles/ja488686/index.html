Flowmonネットワークを使用した分散アプリケーションおよびデータベースのパフォーマンスの監視



Dmitriy Andrichenkoが作成した記事| ロシアおよびCIS、セールスエグゼクティブ| Flowmon Networks

分散ネットワークアプリケーションとデータベースのパフォーマンスの監視に関する問題の解決に関する新しい記事のページへようこそ。この記事は、Flowmon Networksソリューションに関する一連の出版物の続きであり、特に、シグネチャレステクノロジーを使用した「ネットワークの監視と異常なネットワークアクティビティの検出」のレビューの続きです。
それでは始めましょうが、最初にFlowmon Networksとその問題について少し説明します。

読むのが面倒な人のために、Flowmon Networksソリューションに関するウェビナーがまもなく開催されます。

Flowmon Networks、


Flowmon Networksは、ガートナーの四角とレポートで強調表示されているヨーロッパのITメーカーであり、ネットワーク監視、情報セキュリティ、DDoS保護のための革新的なソリューションの開発、および現在の記事のトピックであるネットワークアプリケーションとデータベースのパフォーマンス監視に特化しています。

同社はチェコ共和国のブルノに本社を置いています。最終的な顧客にとって、これには1つの重要な利点があります。認定リストにある企業と連携できることです。 Flowmon Networksの詳細については、こちらまたはこちらをご覧ください。

しかし、Flowmonソリューションの革新的な点は何でしょうか。結局のところ、上記のどの分野も市場に新しいものではありません。ファイアウォールまたは侵入検知システムは長く成功して存在しており、監視トピック自体は新しいものではありません。すべてが真実ですが、いつものように、「悪魔は細部にあります」。

たとえば、ネットワーク情報セキュリティのトピックを検討してください。最初に何が思い浮かびますか？ファイアウォールまたはおそらくIDS / IPS？たぶんNGファイアウォール。そうです、これは定評のある古典ですが、2つの大きな欠点があります。

  • セキュリティ問題を特定するための限られた署名アプローチ、
  • ネットワークセグメントの境界レベルでのみポイント保護。

私たちは、ヒューリスティック分析テクノロジーと機械学習のアプリケーションについて話しています。言い換えれば、人工知能。利点は明白です。更新されて関連性がある場合にのみゼロデイ攻撃から保護する固定シグネチャはありません。
シグネチャレス分析により、非定型のアプリケーションレベルの攻撃、RFCからのプロトコル形式の逸脱、および管理者に毎日多くの頭痛の種となる他の多くの問題を記録できます。

さらに、2番目の主な利点は、セグメントまたは境界の「ジャンクション」でのポイントごとのトラフィック制御だけでなく、標準的な保護手段によって解決されるだけでなく、ネットワーク内の各ネットワーク接続の完全な制御と「透過性」です。

私たちは既存の防御を置き換えることを提案していません。絶えず進化するテクノロジーと潜在的な脅威がある現代の世界では、標準的な防御のセットではもはや十分ではないと言います。これについては先ほど、ここで書きました。

同様の状況は、データベースだけでなく、ネットワークアプリケーションの機能とパフォーマンスを監視するタスクでも発生します。ユーザーがビジネスアプリケーションの機能について不満を言う状況は誰もが知っていると思いますが、問題は解決されていません。ネットワーク管理者は、すべてがLANで正しく機能していると主張し、アプリケーション自体の問題に言及しています。アプリケーション管理者はサーバー、イベントログ、DBMSをチェックし、すべてがうまくいくことがわかりました。その結果、問題は診断されず、すべてのレベルで「すべてが順調」であり、管理者は互いに「うなずき」、エンドユーザーには何も機能しません。何をすべきか明確ではありません。それは起こった？今日はそのことについてお話します。

ソリューションアーキテクチャ


Flowmon Networksが分散アプリケーションとデータベースのパフォーマンス監視の問題を解決するために使用するアプローチとテクノロジーを正しく理解するには、分析全体がシステムに送信されるネットワークトラフィックに関する情報に基づいていることに注意してください。このアプローチの利点の1つは、ワークステーションとサーバーにエージェントソフトウェアがないことです。もちろん、ソリティアソリティアのパフォーマンスを測定することはできませんが、データベースを「ハング」させたSQLクエリ、またはアプリケーションがハングした後のボタンを特定することは十分に可能です。

では最後の記事Flowmon Networksの製品ポートフォリオとシステムをVMware EXSi仮想環境にインストールするプロセスについてはすでに検討しているので、ここでは繰り返しません。この場合の唯一の違いは、トラフィックの受信方法です。フロープロトコルのいずれも、ISO OSIモデルを使用してレベル7プロトコルの機能を分析する必要があるパケットの内容に関する情報を送信しないため、スイッチのミラーリングされたSPAN（Switched Port Analyzer）ポートを使用してデータを収集します。

この場合、ソリューションアーキテクチャは次のようになります。



スイッチは、必要なトラフィックを専用サーバー（Flowmonプローブ）にミラーリングします。専用サーバー（Flowmonプローブ）は、その処理とIPFIXリッチ形式への変換を担当し、ストレージ、相関、および分析のために中央ノード（Flowmonコレクター）に送信されます。ところで、SPANポートの代わりに、TAPトラフィックスプリッターを使用できます。



この導入オプションの利点は次のとおりです。

  • ネットワーク機器のモデルとメーカー（Cisco、Juniper、その他）からの独立、
  • 既存のネットワーク機器に追加の負荷がないこと
  • 会社のネットワークの既存の論理アーキテクチャの維持。

実際、システムの各コンポーネントは、専用のハードウェアサーバーまたは仮想マシンのいずれかです。 2番目のケースでは、Flowmonコレクターに統合Flowmonプローブが含まれますが、パフォーマンスは当然低下します。

セントラルノード（Flowmon Collector）は、モジュラーアーキテクチャの原則に基づいて構築されており、お客様のタスクごとに個別に構成されます



。FlowmonCollectorは、ネットワーク管理者がLAN内のトラフィックを監視するために必要なすべての機能を備えたシステムコア（ネットワーク可視化のトラブルシューティング）で構成されていますネットワーク接続、およびいくつかの追加の個別にライセンスされたモジュール：

  • 異常検出セキュリティ（ADS）モジュール-トラフィックのヒューリスティック分析と一般的なネットワークプロファイルに基づいて、ゼロデイ攻撃を含む異常なネットワークアクティビティを検出します。
  • アプリケーションパフォーマンス監視（APM）モジュール-「エージェント」をインストールしてターゲットシステムに影響を与えることなく、ネットワークアプリケーションのパフォーマンスを監視します。
  • トラフィックレコーダーモジュール（TR）-一連の事前定義されたルールまたはADSモジュールからのトリガーによってネットワークトラフィックのフラグメントを記録し、情報セキュリティインシデントのトラブルシューティングや調査を行います。
  • DDoS保護（DDoS）モジュール-ボリュームDoS / DDoSサービス拒否攻撃からのネットワーク境界の保護。

この記事では、ネットワーク可視性のトラブルシューティングとアプリケーションパフォーマンスの監視という2つのモジュールの例を使用して、すべてがどのように機能するかを見ていきます。

ソリューションのインストール


仮想マシンの導入についてはすでに説明しましたが、すべてはOVFテンプレートから非常にすばやく簡単に実行されます。ここでは繰り返しません。システムリソースの要件のみを思い出します



。FlowmonCollector側では、NetFlowモニタリングからのSPANトラフィックのモニタリングの主な違いは、データの受信方法です。以前にIP構成でNetFlowの管理インターフェイスを使用していた場合、SPANトラフィックを受信するには、サーバーシャ​​ーシに専用の物理ポートを持つハイパーバイザーに関連付けられた、実際にはL2インターフェイスである監視インターフェイスが必要です。



つまり、モニタリングインターフェイスは、Flowmon Collectorに組み込まれたFlowmonプローブです。

次のステップは、専用ポートが正しく構成され、Flowmon Collectorでトラフィックを受信する準備ができていることを確認することです。



私たちの場合、UDP / 2055ポートはネットワーク機器からIPFIX / NetFlowの下で占有されているため、Flowmonプローブを使用するトラフィックの場合、UDP / 3000ポートを使用します。トラフィックをポートごとに異なるソースから分離する必要はありませんが、監視とトラブルシューティングの点でより便利で簡単です。

次に、FlowmonプローブからFlowmonコレクターへのトラフィックのエクスポートを構成します。これを行うには、Configuration Center-> Monitoring Portsセクションで、現在の設定を確認します。主に、レベル7 ISO OSIの必要なアプリケーションの監視がオンになっていることを確認する必要があります。デフォルトではオフになっているためです。



理想的には、制御したいプロトコルのみを含めますが、すべてをオンにすることができます。

設定を保存して、再度構成センターのメイン画面に移動します。FlowmonプローブからのトラフィックがFlowmonコレクターに正しく送信されていることを確認する必要があります。



Flowmonモニタリングセンター->ソースセクションでも確認してください。



トラフィックが流れ始め、システムが機能していることがわかります。Application Performance Monitoring（APM）モジュールの構成に直接進むことができます。

アプリケーションパフォーマンス監視モジュール（APM）


何をどのように正確に制御するかを扱います。Flowmon APMが制御するパラメーターは何ですか？

  • 問題のあるHTTPおよびSQLクエリ、アプリケーションサーバーとデータベースの応答エラーコードの分析
  • クライアント/サーバー間の相互作用のほか、アプリケーションサーバー同士やデータベースサーバーとの相互作用中に発生する遅延とパケット損失
  • 各トランザクションに関する情報（数、サイズ、時間、IPアドレス、セッションID、ユーザー名...）、およびSLA違反を伴う問題のあるトランザクションのリスト
  • トランスポートレベルでのアプリケーション応答時間（最大、最小、平均、パーセンテージ...）と送信時間
  • 同時ユーザーセッションの数、...



Flowmon APMはどのプロトコルをサポートしていますか？

  • HTTP 1.1、HTTP 2.0、SSLおよびTLS、
  • SQL（MSSQL、Oracle、PostgreSQL、MySQL、MariaDBを含む）、
  • メール（SMTP、IMAP、POP3を含む）、
  • VoIP SIP、
  • DHCP、DNS、SMB（v1、v2、v3を含む）、AS、NBAR2、
  • SCADA / IoT（IEC 60870-5-104を含む）。

その結果、監視対象のアプリケーションまたはデータベースごとに、システムはAPMインデックスメトリックの値を計算します。これは0から100まで変化し、サービスの現在の状態に依存します。メトリック値が高いほど、優れています。

ます。1.最終日のアプリケーションとデータベースの全体的なパフォーマンス。



完全を期すために、アプリケーションパフォーマンスインデックスがデータベースパフォーマンスインデックスとともに表示されます。これは、問題が発生する特定の領域のトラブルシューティングと理解に十分便利です。

たとえば、今回のケースでは、データベースパフォーマンスインデックスは100のうち96.839ですが、WebEshopアプリケーションには明らかな問題があり、そのインデックスは100のうち63.761しかありません。

この評価の理由、つまりユーザーの要求に対する応答時間が長いことがすぐにわかります。平均時間は21.148秒で、最大時間は151.797秒です。あなたがオンラインアプリケーションの管理者である場合、ページが読み込まれるまで2.5分間待機するユーザーはほとんどいないことを理解しています。これはすでに問題です。

ダッシュボードの各要素はインタラクティブでクリック可能です。すべてがドリルダウンの原則に従います。チャートで興味深い領域を選択すると、1つ下のレベルで「失敗」して、より詳細な情報を取得できます。



問題が修正された時間間隔を選択すると、管理者は質問に対する回答をすばやく見つけることができます。
