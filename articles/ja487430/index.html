インターフェイスを作成するには、グラフィカルエディターとコードの2つの古典的な方法があります。これは、モバイル開発におけるハリウッドのトピックの1つです。グラフィカルエディターには開発の支持者がおり、誰かがコード以外を認識しません。ただし、両方の方法を試すことをお勧めします。あなたは確かにそれらに出くわすでしょう、それに加えて、たとえば、グラフィカルエディターで行う簡単なことと、すでにコード内にある複雑なものなど、アプローチを組み合わせることを禁じている人はいません。



そして最近、iOS開発者はこれら2つのアプローチを組み合わせようとしている新しいSwiftUIフレームワークを持っています。つまり、グラフィカルエディターで変更するとコードが変更され、その逆も同様です。おそらく、このフレームワークが普及すると、すべての論争は忘却に陥ります。

残念ながら、SwiftUIはiOS 13以降でのみ使用できるため、これまでのところ大規模なプロジェクトでは使用できません。



グラフィカルエディターには、非常にシンプルなインターフェイスがあります。既製のコンポーネントを取得して仮想画面にドラッグすると、すぐに何が得られるかを確認できます。このため、シンプルなUIコンポーネントの作成は非常に高速です。また、さまざまなデバイスの仮想画面のサイズを変更して、アプリケーションを起動せずにインターフェースをテストすることもできます。





一方、コードにはインターフェース設計があります。コードでインターフェイスを開発するときは、UIKitライブラリが便利です。UIViewと呼ばれる、このライブラリから小さなビルディングブロックを使用します。 UIViewオブジェクトは、画面に表示したコンポーネントにほぼ対応しています。グラフィカルエディターと同様に、ボタンやテキストなどの一般的に使用されるコンポーネント用の既成のサブクラスがあります。

コードで開発する場合、他のUIViewが自分自身でUIViewを作成できること、つまり、複雑なインターフェイスを構築できることを理解することが非常に重要です。その結果、いわゆるツリーが得られます。コードから変更する方が簡単です。つまり、非常に動的なユーザーインターフェイスがある場合は、コードで記述する価値があり、後で簡単に変更できます。

プラットフォームSDKは、Apple、つまりGoogleから提供されるモバイルアプリケーションを作成するためのツールとライブラリのセットです。

現在、SDKにはオペレーティングシステムと対話するための多数の低レベルライブラリがあり、モバイルアプリケーションのほとんどすべてのアイデアをそれほど労力なしで実装できます。

使用するライブラリは、アプリケーションの仕様によって異なります。たとえば、カメラを作成する場合は、おそらく2つのライブラリが必要になります。カメラを操作するためのAVFoundationと、画像処理のためのCoreImageです。

したがって、この段階ではすべてのライブラリを記憶する必要はなく、単に不可能です。より重要なのは、適切なツールを見つける能力です。そして残念なことに、公式ドキュメントは機能を完全にドキュメント化しているとは限りません。たとえば、ブログや別の開発者のTwitterで、興味深いことが見つかることがあります。情報フィールドをフォローする価値があります。

それで、翻訳者のためにSDKから何が必要ですか？



私たちのトランスレーターは古典的なクライアント/サーバーアプリケーションです。サーバーにリクエストを送り、レスポンスを受け取ります。次に、どういうわけかそれを変換してユーザーに表示します。 Foundationフレームワークは、この作業に役立ちます。日常のタスクを解決するための便利な抽象化が含まれています。

ここからURLSessionクラスを取得します。これは、httpプロトコルを使用してサーバーを操作するためのクラスです。JSONSerializationは、JSON形式からオブジェクトを変換するのに役立つクラスです。



それでは、この機能のためにどのコードを書く必要があるか見てみましょう。左側は、ユーザー入力が変更されるたびに実行されるメソッドです。まず、トランスレータサーバーのアドレスを使用してURLを作成し、リクエストを行います。さらに、回答を受け取った後、JSON形式から解析し、そこから目的のテキスト、つまり翻訳テキストを取得します。そして最後のステップ-入力フィールドで、受け取った回答を設定する必要があると言います。

どちらのアプリケーションでも、「hello」というテキストを入力すると、実際の翻訳者にすでに何度か翻訳が表示されていることがわかります。私たちの翻訳者では、翻訳は大きなハングで表示されます。

この問題を解決するには、モバイルアプリケーションのマルチスレッドモデルについて理解する必要があります。



デスクトップと同様に、モバイルアプリケーションはマルチスレッド環境で実行されます。ただし、重要な制限があります。モバイルアプリケーションには、メインスレッドまたはいわゆるUIスレッドがあります。また、UIを使用した操作、UIの変更はすべてこのメインスレッドで行う必要があります。ボタンの色を変更したり移動したりする場合は、すべてUIストリームにある必要があります。

一方、すべてのユーザーインタラクションもメインスレッドで行われます。つまり、ユーザーがボタンをクリックしました。メインスレッドでメッセージを受け取りました。私たちの場合と同様に、テキストを変更しました-メインスレッドでも取得しました。ストリームは1つしかなく、その中でUIを使用したすべてのアクションが実行されるため、それに気を配り、可能な限り少ない計算を行うことが非常に重要です。そうしないと、ユーザーアクションを長時間処理していて、他のユーザーアクションからのキューがある状況に陥るリスクがあります。その後、すべてがフリーズし始めます。

このため、デフォルトでは多くのシステムライブラリがバックグラウンドストリームで実行されます。たとえば、ネットワークへのトリップには平均100〜500ミリ秒かかります。これはメインストリームにとって非常に負荷の高い操作であるため、ネットワークとのすべてのやり取りはバックグラウンドストリームで行われます。

繰り返しになりますが、これは私たちにとって問題を引き起こします。サーバーから取得した結果をすぐに使用してユーザーインターフェースを変更すると、クラッシュにつながる可能性があるためです。

最後の行を思い出してください：

self.textOutputView.text = translation

その中で、サーバーの結果を取得し、出力フィールドに設定しました。したがって、最初のルールに違反しました。メインスレッドからではなく、UIを変更しました。標準ライブラリGrand Central Dispatchは、これを修正するのに役立ちます。メインスレッドに切り替えて、メインスレッドでこのアクションを実行するのに非常に簡単です。

DispatchQueue.main.async {
  self.textOutputView.text = translation
}

残念ながら、現在のプロジェクトでは、アーキテクチャの重要性を示すことはできませんが、それでもまだ非常に小さいです。



しかし、確かに、アーキテクチャはより良い生活から生まれたのではなく、開発者が何もする必要がなく、発火しないように抽象化を書いているという事実からではありません。アーキテクチャは、モバイル開発の課題に対する答えです。

これらの問題の主なものはスケーラビリティです。アプリケーションが大きくなりすぎて機能が異なる場合、理解、拡張、デバッグが容易であることが重要です。それ以外の場合、機能を追加すると古い機能がすべて変更されます。また、バグが発生するリスクが高まります。

アプリケーションが大きくなるまで待ちません。すでに最初の段階で、SOLIDや4つのギャングパターンなど、モバイルアプリケーションでも機能するソフトウェアを構築する基本的な方法を学ぶことができます。

モバイルアプリケーションのアーキテクチャには固有の特徴があります。それらのグローバルコンポーネントまたは大規模なコンポーネントは、どのオブジェクトがどのレイヤーに属すべきかを高レベルで示すアーキテクチャパターンに従って構築されます。最も一般的なのは、MVC、MVP、およびMVVMです。

そして、現実には建築は特効薬ではないことを忘れないでください。プロジェクトの詳細を考慮する必要があります。目に涙を浮かべて、ある種のアーキテクチャをプロジェクトに取り入れている場合、おそらく何かがうまくいかなかった。

理論を学ぶとき、それはあなたにとってひどく複雑に見えるでしょう。しかし、実際には、優れたアーキテクチャを備えたプロジェクトで作業すると、コードの記述が容易になったので、満足します。あなたはあなたが機能を持つために何をどこに追加するかを知っています。したがって、この段階では、アーキテクチャをよく理解するために、大規模なプロジェクトで作業またはインターンすることをお勧めします。そのような機会がない場合は、多くの企業がプロジェクトをオープンソース（Wikipedia、Firefox）に配置しています。そして、GitHubに行ってそこで見ることを禁じている人はいません。

これで、アプリケーションの準備が整いました。ユーザーがダウンロードできるようにパブリックドメインに配置しましょう。



ほとんどの場合、ユーザーはGoogle PlayとApp Storeからアプリを入手します。ただし、アプリケーションをストアに追加する前に、署名する必要があります。これは、オペレーティングシステムが信頼できる開発者からのアプリケーションのみを実行するためです。したがって、信頼できる開発者だけがモバイルデバイスで作業できるようになるため、モバイルプラットフォームでは、悪意のあるアプリケーションやウイルスははるかに少なくなります。

これを行うには、開発者証明書を発行する必要があります。これは実際には小さな官僚制であり、幸いにも、Xcodeの最新バージョンでは自動化されています。

また、アプリケーションと説明のスクリーンショットを撮る必要があります。アプリケーションのページは広告であるため、最後の2つのステップは無視しないでください。明るくない場合、誰もアプリケーションをダウンロードしないため、開発のすべてが無駄になっている可能性があります。

最もよく使用するスキルはデバッグです。



主なデバッグ方法は、古き良きconsole.logまたはprintfです。名前はたくさんありますが、意味は同じです。何か問題が発生した場合は、ロギングと印刷変数を追加します。しかし、残念ながら、この方法には重大な欠点もあります。

最初の欠点は、ソースコードの変更です。 printfを追加すると消えるバグがあります。そして、printfを削除します-再び発生します。これらのバグには別の名前（heisenbug）さえ付いています。

2番目の結果は、アプリケーションを再コンパイルして再度実行する必要があることです。大規模なプロジェクトでは、これには数分かかる場合があり、新しい各ログを追加するときに1分待つ必要がある場合は、合計で多くの時間を費やします。

そして、printfの最後の最も重大な欠点-残念ながら、それはすべてのバグに役立つわけではありません。

次に重要なスキルは、アプリケーションを最適化することです。



モバイルデバイスのリソースは限られているため、多くの場合、パフォーマンスの問題があります。開発時には、CPUの消費量などについて考慮する必要があります。ほとんどの場合、これはあなたが書くコードの複雑さです。ほとんどのユーザーはモバイルトラフィックでアプリケーションを使用する可能性が高いため、レンダリング速度、メモリ、ネットワークトラフィック。ユーザーを尊重しましょう。多分それは相互関係になるでしょう。

5番目のポイントはバッテリーで、最初の4つから続きます。

アプリケーションを最適化する際に最も重要なことは、問題のある領域を特定することです。仮定だけに依存している場合、多くの時間を失う可能性が高く、あまり利益を得ることができません。これは、プラットフォームツールに役立ちます。それらにはプロファイラーが含まれます-プログラムがコードのどこで最も時間を費やしているかを示すプログラムです。つまり、プログラムが最も頻繁にハングするメソッドが表示され、おそらくフリーズの理由がわかります。

アルゴリズムとデータ構造を知ることは、より効率的なソリューションを見つけるのに役立ちます。また、マルチスレッドの知識も役立ちます。現在、モバイルデバイスにはマルチコアプロセッサがあり、複数のスレッドにまたがって問題を並列化することにより、パフォーマンスがわずかにn倍向上します。

残念ながら、最初の2つのポイントが役に立たない場合があります。次に、オペレーティングシステムがどのように機能するかを理解し、メモリマップ（mmap）などのシステムコールに関する知識を習得します。私たちの場合、サードパーティのiOSキーボードには、RAM消費量に52メガバイトという制限があります。つまり、機能を追加し、美しいユーザーインターフェイスを作成したいのですが、52メガバイトに制限されています。これをどうするかは明確ではありません。

別の機能を追加すると、これが起こりました。私たちはしばしばこの制限を超え始め、それをどうするかわかりませんでした。その結果、メモリマップシステムコールが役に立ちました。辞書を含むファイルを取得し、そこからすべてのヒントを取得し、メモリマップの呼び出しを開始しました。したがって、ファイルを完全にRAMに読み込まずに、ファイルの一部を操作します。

最後に頻繁に発生する問題は開始時間です。ここで、コードが機能し始める前に何が起こるかを読むことをお勧めします。特に、静的リンクと動的リンクについて。



最後の便利なスキルは、継続的な統合または退屈なタスクの自動化です。

退屈な作業を好む人はいないので、自動化します。ほとんどの場合、最も簡単なことは、コミットごとにアプリケーションをビルドすることです。アプリケーションをビルドする頻度が高いほど、問題をより早く見つけることができます。テストがある場合は、それらも実行します。

前述のように、このパブリケーションは小規模な官僚機構に関連付けられているため、非常に自動化されており、プールリクエストごとにアセンブリをベータストアにリリースして、テスターがテストできるようにすることができます。 App Storeのスクリーンショットの生成を自動化して、手動で実行しないようにすることもできます。
