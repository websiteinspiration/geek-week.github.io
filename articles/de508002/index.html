Ehrgeizigere Änderungen sind auch in Bezug auf die Leistung, den architektonischen Ansatz zum Erstellen von Anwendungen und den deklarativen Konstruktor aufgetreten:

Übrigens verwendeten einige Steuerelemente in SwiftUI selbst den Lazy-Ansatz. Zum Beispiel die gleichen Listen und LazyVGrid / LazyHGrid (analog zu UICollectionView). 3. Jetzt können Sie eine 100% ige Anwendung auf SwiftUI-Komponenten erstellen.



Ja Ja. Zu diesem Zweck hat Apple entwickelt, wie auf AppDelegate, SceneDelegate und UIHostingViewController verzichtet werden kann.



Annotation verwendenMain und Protokolle App, Szene können Sie dies erreichen: 



@main
struct testmultiplatformApp: App {
  @SceneBuilder var body: some Scene {
        WindowGroup {
            MailViewer()
        }
        Settings {
            SettingsView()
        }
    }
}


Mainsignalisiert, dass dies der neue Einstiegspunkt Ihrer Anwendung ist. Die Implementierung des App-Protokolls ist erforderlich. Beachten Sie den Typ der body-Eigenschaft der App-Struktur. Eine Anwendung kann eine oder mehrere sogenannte Szenen haben, von denen jede das Szenenprotokoll implementiert. Jede Szene hat ihre eigene Stammansicht und ihren eigenen Lebenszyklus. Wenn Sie mehrere Szenen verwenden möchten (z. B. in einer Anwendung mit mehreren Fenstern), müssen Sie das Body-Attribut SceneBuilder festlegen. Dies ist im Wesentlichen der Kapselungsmechanismus von UISceneDelegate und seiner Logik.



WindowGroup wird verwendet, um einen einzelnen Bildschirm in voller Größe wie UIWindow / UIWindowScene zu erstellen.



Wir erinnern uns jedoch, dass SwiftUI ein Add-On für UIKit ist. Und UIKit blieb drinnen. Wenn wir überhaupt eine solche Vorlagenanwendung ausführen, sehen wir in der Ansichtshierarchie Folgendes:







Sowohl der vorhandene UIHostingViewController als auch die UIWindowScene. Aber drinnen. Ja, für die Initialisierung vereinfacht dies die Arbeit erheblich. Aber haben sie sich eine Lösung ausgedacht, damit sie UISceneDelegate nicht mit einem expliziten Job der gesamten Navigationsstruktur an die Anwendung zurückgeben müssen?



Das Verfolgen der Lebenszyklus-Szene wird auf eine Methode reduziert: 



public func onChange<V>(of value: V, perform action: @escaping (V) -> Void) -> some Scene where V : Equatable


So schlagen sie vor, diese Methode zu verwenden, um den Übergang der Anwendung in den Hintergrundzustand zu verfolgen: 



  ///
    /// Use this modifier to trigger a side effect when a value changes, like
    /// the value associated with an ``SwiftUI/Environment`` key or a
    /// ``SwiftUI/Binding``. For example, you can clear a cache when you notice
    /// that a scene moves to the background:
    ///
    ///     struct MyScene: Scene {
    ///         @Environment(\.scenePhase) private var scenePhase
    ///         @StateObject private var cache = DataCache()
    ///
    ///         var body: some Scene {
    ///             WindowGroup {
    ///                 MyRootView()
    ///             }
    ///             .onChange(of: scenePhase) { newScenePhase in
    ///                 if newScenePhase == .background {
    ///                     cache.empty()
    ///                 }
    ///             }
    ///         }
    ///     }
    ///
    /// The system calls the `action` closure on the main thread, so avoid
    /// long-running tasks in the closure. If you need to perform such tasks,
    /// dispatch to a background queue:
    ///
    ///     .onChange(of: scenePhase) { newScenePhase in
    ///         if newScenePhase == .background {
    ///             DispatchQueue.global(qos: .background).async {
    ///                 // ...
    ///             }
    ///         }
    ///     }
    ///
    /// The system passes the new value into the closure. If you need the old
    /// value, capture it in the closure.
