EF Core + Oracle：移行をべき等にする方法



通常、EF Coreフレームワークは、別のMicrosoft製品であるMS SQLと組み合わせて使用​​されます。ただし、これはドグマではありません。たとえば、CUSTISではC＃でビジネスロジックを記述し、Oracleを使用してデータベースを管理しています。EF Coreには素晴らしい移行メカニズムがありますが、私たちの場合、それらはべき等ではありません。実際、OracleやMySQLなどの他の多くのデータベースは、トランザクションDDLをサポートしていません。これは、移行が途中で失敗した場合、ロールバックまたはロールバックできないことを意味します。MS SQLなしでEF Coreへのべき等の移行を実装する方法

バックグラウンド


当社には、Oracleデータベースにパッチをインストールするためのかなり強力なツールがあり、これをいくつかのプロジェクトで使用しています。Liquibase、EF移行、およびその他のオープンツールがないときに書かれました。Patcherを使用すると、数百のデータベースの操作、インストール履歴の追跡、ログの表示、シークレットの保存などを行うことができます。データベースを変更するためのスクリプトは、SQLまたはm4マクロの形式で記述されます。彼らの助けを借りて、とりわけ、構造を変更することができます：テーブル、列、その他のオブジェクトを作成します。さらに、m4マクロはべき等です。つまり、たとえばテーブルを作成しようとした場合、スクリプトは失敗しませんが、スクリプトがすでに存在していることを確認して、作成をスキップします。

パッチをインストールするスクリプトが2つの操作で構成されているとします。

  1. テーブルAを作成します。
  2. テーブルBの作成

最初の操作後にスクリプトがクラッシュした場合、テーブルAはOracleに残ります。パッチを再適用すると正しく機能します。スクリプトはAが既に存在することを確認するため、すぐに2番目の操作に進みます。

利点に加えて、パッチャーにはまだ欠点があります-ツールは閉じており、CUSTISでのみ使用されます。開発者は彼と一緒に働くことを学ぶ必要があり、社外ではそのような経験はあまり価値がありません。さらに、パッチャーはCode Firstモードの操作をサポートしていないため、データベース構造を変更するためのすべてのスクリプトを手動で作成する必要があります。

パッチをインストールするための既製のメカニズムを試してみて、移行を選択しました。 2019年の終わりに、お客様向けに別のプロジェクトが開始されました。そこで、新しいアプローチをテストすることにしました。このメカニズムの主な問題は、移行の非等倍性でした。

問題


MS SQLでは、一連のDDLステートメントまたは移行は単一の複合トランザクションとして実行されます。中断した場合、操作は完全にキャンセルされます。Oracle DDLは非トランザクションであるため、移行が低下すると、データベースの状態が不整合になります。

テーブルAとBの作成という2つの操作で構成されるパッチに戻りましょう。最初の移行後にマイグレーターが落ちた場合、OracleはテーブルAに残りCREATE TABLEます。また、移行のロールバックにも失敗します。EFCoreは、移行が完了したことをシステムテーブルに書き込みますが、プロセスの最後にのみ行われます。EF Coreの観点からは、移行がまだ完了していない場合、ロールバックするものはありません。

決定


インターネットでOracleの既製のソリューションを検索しても、結果は得られませんでした。私が見つけたのは、EFで作業するときにパッチを作成して  インストールする方法に関する記事だけです。少し後、StackOverflowでIMigrationsSqlGeneratorを作成するというアイデアを思いつきました。このインターフェイスは、EF操作を処理するSQLコードの生成を担当します。Oracle.EntityFrameworkCore

パッケージには、OracleMigrationsSqlGeneratorが含まれ、IMigrationsSqlGeneratorが実装されています。たとえば、列を追加する場合、次のコードが生成されます。

ALTER TABLE MY_TABLE ADD (MY_COLUMN DATE)

次に、コードは他のクラスに渡され、データベースで実行されます。

まず、OracleMigrationsSqlGeneratorオペレーションのペアをオーバーライドしようとしました。タスクはかなり実行可能であることが判明し、私はべき等移民を書き始めました。これがCUSTIS.OracleIdempotentSqlGeneratorが生まれた方法です。

EFオペレーションの前に、マイグレーターはそれが以前に実行されたかどうかを確認します。たとえば、次のように列が追加されます。

DECLARE
    i NUMBER;
BEGIN
    SELECT COUNT(*) INTO i
    FROM user_tab_columns
    WHERE table_name = UPPER('MY_TABLE') AND column_name = UPPER('MY_COLUMN');
    IF I != 1 THEN
        EXECUTE IMMEDIATE 'ALTER TABLE MY_TABLE ADD (MY_COLUMN DATE)';  
    END IF;       
END;

使用する


パッケージの使用は非常に簡単です。IMigrationsSqlGenerator適切なコンテキストでパッケージを置き換えるだけです。

public class MyDbContext : DbContext
{
    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        optionsBuilder.ReplaceService<IMigrationsSqlGenerator, IdempotentSqlGenerator>();
    }
}

移行は、EF Coreの標準ツールによって形成および設定されます。

dotnet ef migrations add v1.0.1
dotnet ef database update

CUSTIS.OracleIdempotentSqlGeneratorに規定 されている一般的なアプローチは、MySQL、MariaDB、Teradata、AmazonAurora、およびDDLがトランザクションに対応していないその他のデータベース用に作成されたジェネレーターで実装できます。
