契約優先アプローチを使用する


契約とは何ですか？

Webサービスの作成者は、サービスプロバイダーと見なされます。Webサービスを使用するアプリケーションは、サービスのコンシューマです。契約は、サービスに関するプロバイダーとコンシューマー間の合意です。



サービスを正しく使用するには、サービスの利用者が契約を完全に理解する必要があります。契約には、次のようなサービスの多くの側面の詳細が含まれています。

  • Webサービスを呼び出す方法は？
  • どのような交通手段を利用していますか？
  • リクエストとレスポンスの構造は何ですか？

これは、サービス定義とも呼ばれます。

コントラクトファーストアプローチでは、最初にサービスコントラクトを定義してから、サービスのみを実装します。

WSDLとの最初の契約


たとえば、SOAP Webサービスを定義する場合、WSDLを使用してコントラクトを定義します。



WSDLは、サービスエンドポイント、パブリッシュする操作、および要求と応答の構造を決定します。

REST APIの組織標準を定義する


組織標準の重要なガイドラインはYARASです。



YARASは、Yet Another RESTful API Standardの略です。YARASは、RESTful Webサービスを開発する際に従う必要がある標準、ガイドライン、および規則を提供します。彼は次のことを推奨しています。

  • あなたはあなたのサービスにどんな名前をつけるべきですか
  • リクエストとレスポンスをどのように構成すべきか
  • フィルタリング、並べ替え、ページネーションなどのアクションを実装する方法
  • バージョン管理への取り組み方
  • APIドキュメントにアプローチする方法

サービス開発への統一されたアプローチ


RESTful Webサービスの設計を始める前に、多くの複雑な問題を解決する必要があります。上記のすべてを知る必要があります。

組織のリーダーシップは、さまざまなリソースを扱うチームがこれらのことに対してさまざまなアプローチを持つことを望まないでしょう。

たとえば、チームAがクエリパラメータに基づいてバージョニングを整理し、チームBがURIに基づいてバージョニングを使用することは望ましくありません。

したがって、RESTful Webサービスへのアプローチのための組織標準を明確に定義することが重要です。

組織のニーズに合わせたYARASのカスタマイズ


YARASの良い点は、組織固有のニーズに合わせてカスタマイズできることです。たとえば、次のことができます。

  • リクエストとレスポンスの本文がどのようになるかを設定します。
  • 特定のバージョン管理システムを選択する

YARASはかなり包括的な範囲をカバーしているため、1つの重要な決定を逃していないことを確信できます。

標準のエンタープライズフレームワークREST APIフレームワークの選択


Javaの世界でRESTful Webサービスを作成するために使用される典型的なフレームワークは、Spring MVC、Spring REST、およびJAX-RSです。

組織固有のフレームワーク/アーキタイプ/参照アプリケーションを、希望のREST APIプラットフォーム上で共通の組織標準に準拠して作成すると、チームは共通の標準に簡単に準拠できるようになります。

典型的な機能は次のとおりです。

  • リクエストとレスポンスの構造
  • エラー処理
  • 濾過
  • 探す
  • バージョン管理
  • 誤った回答のサポート
  • HATEOAS

標準フレームワークは、組織全体のサービスを設計および実装するための統一されたアプローチを提供します。

  • REST API標準の改善
  • REST APIフレームワークの構築/設計

HTTPの広範な使用


RESTful Webサービスについて考えるときはいつでも、HTTPについて考えてください。

HTTPには、優れたWebサービスの作成に役立つすべての機能があります。

正しいHTTPリクエストメソッドを使用する


使用する必要があるHTTP要求メソッドについて考えてください。操作の実装を検討するときは、操作を実行するリソースを決定し、適切なHTTP要求メソッドを見つけます。詳細を取得、何かを作成、既存のものを更新、または既存のものを削除しますか？

HTTPメソッドの使用：

  • GET to get
  • 作成するPOST
  • 更新するPUT
  • 削除するにはDELETE
  • 部分更新のパッチ

適切なHTTP応答ステータスを使用する


操作を実行するときは、正しい応答ステータスを返すことを確認してください。

たとえば、特定のリソースが見つからない場合は、サーバー例外をスローしないでください。代わりに、404などの応答メッセージで適切な応答コードを送信します。

本当にサーバー例外がある場合は、コード500を送り返します。

検証が失敗した場合は、無効なリクエストのコードを送信します。

パフォーマンスを重視


各リソースは、XMLまたはJSON形式の複数の表現を持つことができます。サービス利用者は、自分が選んだプレゼンテーションを選択できます。

適切なドキュメントを作成する


消費者は、サービスを最大限に活用する方法を理解する必要があり、顧客を支援する最良の方法は、優れたドキュメントを作成することです。

SOAP WebサービスはWSDL機能を使用できますが、RESTful WebサービスにはSwaggerオプション（現在はOpen APIドキュメント標準）があります。

形式の選択は、優れたドキュメントを作成するための一部にすぎません。また、消費者を助けるために適切な量の情報を提供することも重要です。

ドキュメントは完全で、次の点が含まれている必要があります。

  • リクエストとレスポンスの構造とは何ですか？
  • 消費者はどのように自分自身を認証する必要がありますか？
  • 使用の制限は何ですか？
  • サービスから期待できるすべてのタイプの応答メッセージと対応するステータスコードを示します。

バージョン管理方法の比較


これまでのところ、バージョンには4種類のメソッドがあります。

  • 異なるURIの使用
  • クエリパラメータの使用
  • ヘッダーの使用
  • Acceptヘッダー/メディアタイプの使用

どれが一番いいですか？真実は、この質問に対する単一の答えはないということです。問題の事実は、さまざまなインターネットの巨人がさまざまなタイプのバージョンを利用していることです。

  • 異なるURIの使用-Twitter
  • クエリパラメータの使用-Amazon
  • ヘッダーの使用-Microsoft
  • Acceptヘッダー/メディアタイプの使用-GitHub


特定のニーズに応じて、これらの4つのオプションを評価する必要があります。考慮すべき重要な要素がいくつかあります。

  • URI汚染：URIとクエリパラメータを使用してバージョンを制御すると、URIスペースが汚染されます。これは、URIのメイン文字列に接頭辞と接尾辞を追加するためです。ヘッダーを使用したバージョン管理はこれを回避します。
  • Misuse Of HTTP Headers ( HTTP). Media Type HTTP, .
  • Caching (): URI. , URI , , . HTTP-, URI .
  • ブラウザ要求の実行可能性：タイトルとメディアタイプに基づくバージョン管理には、Postmanなどのツールが必要です。ただし、サービスコンシューマが技術的に精通していない場合は、URIまたはクエリパラメータに基づくバージョン管理を使用することをお勧めします。
  • APIドキュメント：APIをドキュメント化する方法についても考える必要があります。URIとクエリパラメータに基づくバージョン管理は、他の2種類のバージョン管理よりもドキュメント化が簡単です。
