10年後に体系化。歴史的および技術的回顧

10年前にsystemdが発表され、Linuxディストリビューションシステムの管理に革命をもたらし、Linuxユーザーをいくつかの陣営に分けました。討論の質と性質は2012年から2014年の激しい戦争以来それほど改善されておらず、systemdはコミュニティによる細心の注意にもかかわらず、技術的および公的側面の両方からツールによって完全に理解され、研究されていません。

この投稿は、systemdの使用方法に関するものではありません。ここではむしろ、その発生の歴史、コンポーネント全体、および単純にPID 1として始まり、現代のLinuxディストリビューションのミドルウェアと呼ばれるものになったシステムを理解する方法について説明します。

あるいは、Arch wikiのブログ、チャンネル、記事からのさまざまな資料の非常に無料の翻訳のセットにすぎないかもしれません。あなたが決める。

私はユーザーであり、以前はシステム管理者でしたが、systemdに多くの優しい気持ちを抱いているため、この資料はかなりの偏見を持って書かれています。

しかし、systemdについて話を始める前に、initについてお話したいと思います。

初期化


UNIXの第7版（1979）では、initタスクは/ etc / rcの開始時に実行され、端末初期化プロセス（getty）とログイン（login）を作成します。

その後、徐々に新しいタスクが追加されました。たとえば、描画プロセスの傍受。または、/ tmpディレクトリをクリアします。または、ファイルシステムをマウントします。

したがって、今日のinitの主なタスクは、システムを「カーネルが実行中」の状態から「システムが動作する準備ができている」状態にすることです。つまり、ユーザースペースを実行します。

時が経ち、インターネットが登場し、それに伴い、アプリケーション、データベース、キャッシュ、Webサーバーなどが登場しました。サービスの概念は、1つの目標を満たす一連の密接に関連する悪魔またはシステム全体として具体化されます。

もちろん、スクリプトを含む/ etc / rcは必要なすべてを実行できますが、たとえば、ある種のデーモンを再起動する必要がある場合はどうなりますか？ 1983年に発行されたUNIXシステムVは、ランレベルまたはパフォーマンスレベルの概念を導入し、この問題を部分的に解決し、デーモン全体をグループで制御できるようにしました。

LinuxはUNIXの初期化インターフェース（sysvinit）を採用しており、一般的に誰もがすべてに満足していますが、それでもsysvinitが恥ずかしいユーザーもいました。そのため、Matthias S. Brinkmannは彼のブログに「なぜsysvinitが悪いのか」と書いています。 Matthiasによれば、「sysvinitはすでに死んでいる」ので、オリジナルは削除されましたが、インターネットで見つかった転載を簡単に自由に翻訳します。

後に、Busybox開発者のDenis VlasenkoはSysV initを彼のブログに投稿する必要があります。これは冗談であり、実際には投稿が競合形式ではなく対話形式で行われたという予約があったにもかかわらず、システムに関する不満がありました。

要約すると、sysvinitは30年以上前に21世紀の課題を解決しなくなった古い概念で開発されました。さらに、システムは開発されておらず、新しいアイデアも取り入れられていませんでした。

混沌としたカオスの雰囲気の中で、sysvinitの問題を解決するためにいくつかの試みが行われました。

  • 誰かがdbusを介したサービスの発見はクールだと思っていました。
  • 誰かがスクリプトを書いて、initスクリプト（???）の書き込みを自動化することさえしました。
  • 誰かが/ etc / rcにpythonスクリプトを適合させて並列実行しようとしました。
  • Fedoraメンテナもバイクを切りました...解決策を探していました。

しかし、今日、最も有名な代替手段（systemdを除く）は、おそらくUpstartです。

アップスタート


sysvinitが解決を志願した問題の一部は、Ubuntuの開発者であるCanonicalによって2006年にリリースされた代替の初期化システムであるUpstart（ReplacementInitとも呼ばれる）でした。その主な概念は、イベント駆動型のアプローチです。システムの起動はイベント、新しいデバイスはイベント、デーモンの起動はイベントです。プロセスはイベントをサブスクライブし、明らかに必要な場合にのみ実行できます。

Upstartはsysvinitスクリプトと互換性があるため、多くのLinuxディストリビューションにすぐに統合されました。それにもかかわらず、ほとんどのLinuxディストリビューションが何らかの理由でsystemdに切り替わった後、開発はうまくいかなくなりました。最後のリリースは2014年で、今日のUpstartはChrome OSでのみ使用されています。

新しいinit


「私たちは新しいinitシステムを実験していますが、楽しいです。」

2010年の春に、Lennart PotteringのブログがRethinking PID 1をリリースしました。そこで、彼は効果的な初期化システムのビジョンを説明しました。

彼の意見では、効果的なinitには次の特性があるはずです。

macOSブートプロセスから収集できる別のアイデア：シェルスクリプトは悪です。作成は簡単ですが、保守が難しく、実行に時間がかかります。1つのスクリプトでgrep / awk / sed / cutを呼び出すたびに、プロセスの開始、ライブラリのロード、出力ローカリゼーションなどの操作などが行われます。すべてのinitスクリプトについても同様です。システムを起動する場合、多くのコンソールコマンドを実行すると、プロセス全体の速度が低下します。

systemd


Rethinking PID 1で以前に 発表されたプロトタイプsystemd は、以前は " BabyKit "（プロセスベビーシッター-プロセスの乳母から）として知られていましたが、その軽さのためにすぐに支配的な地位を獲得しました。

現在、プロジェクトページではPID 1については説明されていませんが、Linuxシステムの基本ブロックのセットについて説明しています。

この下に隠されているものは何ですか？systemdはコンポーネントのコンストラクターと呼ばれるようになりました。各コンポーネントは、サービスまたはシステム全体に関連するタスクを担当します。

systemdはシステム管理システムです。これはサブシステムをシステムレイヤーに取り込んで統合しますが、これは一方では中心的な懸念事項ではなく、もう一方ではユーザーにとって興味深いものではありません。

systemdはLinuxディストリビューション（少なくともFedora、SuSE、およびArch）の事実上の標準になりつつあり、多くのパッケージで優れたサポートを受けています。これまでのところ、OpenRCを使用しているのはGentooだけであり、Upstartを使用しているのはUbuntuだけです。したがって、OpenRCを使用することは、独自のパッチの多くをサポートする必要があることを意味し、Upstartを使用することは、Ubuntuのアップストリームになることを意味します。

さらに、2013年12月、Debianのinitに関する状況についての声明Russ Alburyが「3.1。エコシステムの現実、「会話はもはや「システム化と代替案」ではなく、「システム化はどの程度か」についてであると述べた：

強調することが重要であると思われる議論の中で省略されたポイントの1つは、 Debianがsystemdの一部を受け入れることに一般的に同意していること。 Systemdは多くのコンポーネントを含む包括的なプロジェクトであり、それらのいくつかは他のものより重要です。これらのパーツのほとんどは、現在Linuxプラットフォームにあるすべてのものよりも明らかに優れており、後でディストリビューションで使用されます。

言い換えれば、この議論は通常の意味でのsystemdとupstartに関するものではありません。問題は、むしろ、初期化システムを含むすべてのコンポーネントを含むsystemdを受け入れるのか、それともそのサブシステムを採用するのか、ただし初期化システムとしてupstartを選択するのかということです。いずれの場合でも、udev、logind、timedated、およびその他のサービスが機能します。

これは、その主要な側面で議論の性質を変えると思います。 2つの競合するシステム間の選択については、実際には話していません。むしろ、システムのメインコンポーネントを、私たちが最も気に入っているコンポーネントに置き換えるかどうかについて話しているのです。

これが起こったことです。 Debian Jessieはsystemdをシステムマネージャーとして残し、Debianに続いて、systemdはUbuntu 15.04に含まれていました。

確かに、この投稿はsystemd戦争に関するものではありません。もっと読みたいけれども、メーリングリストを掘り下げるのが面倒なので、部分的にこの投稿に影響を与えた誰かの資料を保管してください。

journaldの履歴：syslog

長い間、syslogデーモンはLinuxディストリビューションの重要なコンポーネントでした。歴史を通じて、syslogインターフェースの多くの実装が行われてきましたが、一般に、それらは同様の方法で実装され、ファイルシステムでほぼ1つのデータ形式を使用しました。

syslogデーモンの目的は、その名前が示すように、システムプロセスをログに記録することです。デーモンは、比較的自由な形式でアプリケーションとサービスからメッセージを受信し、それらをディスクに保存します。原則として、唯一のメッセージメタデータは、ファシリティ（例：カーネル、メール、ネットワークなど）と優先度、およびタイムスタンプ、プロセスタグ、およびそのPIDです。これらのフィールドのほとんどはオプションであり、正確な構文は実装ごとに異なります。

syslogはすべてのフィールドに入力することを強制しないので、使いやすく、同時に拡張性の点で強力です。同時に、これは彼の欠点です。syslogは厳密なフォーマットなしでは手に余計なものがあり、そのような結果は解析が困難です。

30年間（journaldの登場時）のSyslogシステムは、シンプルであるため、管理者にとって不可欠なツールです。ただし、制限の数は堅実であり、時間の経過とともに顕著な問題になっています。

journald

journaldとの基本的な統合は基本的なものです。プロセスがstdout / stderrに書き込むだけで十分です。サブシステム自体が、サービス、ユーザーセッション、実行可能コマンド、その他の属性を把握します。

より緊密に統合するには、ライブラリを使用する必要があります。たとえば、Pythonの場合は、pystemdライブラリのpystemd.journalモジュールがあります。

あなたはより多くのjournaldと連携する方法について学ぶことができSelectelポストとで男journalctl。

機械加工


systemd-machinedは、systemd-nspawn、systemdコンテナープラットフォームの一部です。概念的には、systemdコンテナーはDockerなどとは異なります。Dockerのように、それが主にステートレスコンテナープラットフォームである場合、systemd-nspawnは、ステートフルコンテナーでsystemdを実行する方法にすぎません。 nspawnはDockerよりもはるかに単純です。RESTAPI、ラベル、ネットワーク管理、ボリュームなどはありません。一方で、これはsystemdコンテナのマイナスですが、一方で、それらは自分でネットワークと環境をセットアップすることに慣れている人に適しています。

systemd-nspawnの主な利点は、systemdエコシステムとの緊密な統合です。コンテナーごとにスライスとサービスが作成され、systemctl set-propertyを介してリソースを構成できます。コンテナー内では、systemdは問題なく機能します。

systemd-nspawnの詳細については、Arch wikiまたはSelectelの投稿。

ほぼ100のバイナリのセットをモノリシックと呼ぶのは簡単ではありません。ただし、すべてのsystemdリリースは単一のtarアーカイブで配布され、単一のリリースサイクルで同じgitリポジトリにあります。

systemdはUNIXでは

ありませんこれにはいくつかの真実があります。systemdソースコードには単一のUNIX行がありません。しかし、開発者はUNIXからインスピレーションを得て、それぞれがその役割を果たしている同じ数十のファイルを見てみましょう。また、プロジェクト管理方法（つまり、1つのgitリポジトリでの開発）は、BSDモデル（Linuxとは異なり、実際のUNIXです）と多くの点で似ています。

一般に、UNIXは誰にとっても異なることを意味します。 Lennart Potteringや他の開発者にとって、これはインスピレーションの源ですが、一部の宗教にとっては、それは他の世界の宗教と同様に、UNIXのさまざまな解釈と理解があります。それは、コードスタイル、コマンドとAPIのセット、および誰かとして理解できます。 UNIXは一連のロジックと動作であると考えています。
