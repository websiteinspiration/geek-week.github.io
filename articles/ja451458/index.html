ゲームについて


人類は、世界大戦、資源の枯渇、絶え間ない競争にうんざりしています。主要な勢力は、最新のテクノロジーを使用して単一のリーダーシップを選択することに同意しました。指定された時期に、世界有権者は、次の千年の間に惑星を支配する割合の選択を決定しなければなりません。主要派閥は「正直な」権力闘争に従事している。ゲームセッションでは、各プレーヤーは分数を表します。

このカードゲームは選挙についてです。各派閥には選挙レースを実施するための予算、予算を増やし、投票を開始する収入源があります。ゲームの開始時に、アクションカードが入ったデッキがミックスされ、4枚のカードが各参加者に発行されます。各ターンで、プレーヤーは最大2つのゲームアクションを実行できます。カードを使用するには、プレイヤーはそれをテーブルに置き、必要に応じて目標を指定し、カードの使用コストを予算から差し引きます。ラウンド終了後、プレイヤーは未使用のカードを1枚だけ保持できます。各ラウンドの開始時に、プレイヤーはデッキからカードを取得します。そのため、各ラウンドの開始時に、各プレイヤーは4枚のアクションカードを手にします。

ラウンド3、6、および9の最後に、投票数が最も少ないプレーヤーがゲームから削除されます。複数のプレーヤーが同じ最小投票数を持っている場合、この結果を持つすべてのプレーヤーがゲームから排除されます。これらのプレーヤーの声は選挙人の一般的なプールに行きます。

ラウンド12の終わりに、最も多くの票を獲得したのが勝者です。

ニーズのリストを見て、すぐに自分のバックエンドを最初の段階で作成するのは意味がないという結論に達し、他のオプションは何であるかをググり始めました。だから私はPlayFab（Microsoftが買った）とGameSparks（Amazonが買った）が際立っている特別なクラウドゲームバックエンドがあることを知りました。

一般に、これらは機能的に類似しており、基本的なニーズをカバーしています。同時に、それらの内部アーキテクチャは大きく異なり、同じタスクの解決方法が少し異なり、機能の明示的な対応を追跡することは困難です。以下に、各プラットフォームの良い点と悪い点、および選択したトピックに関する考慮事項を示します。

プレイファブ


肯定的な機能：

  • 異なるゲームのアカウントがマスターアカウントに統合されます
  • ゲームの経済は、別の仮想ストアへの価格設定を含め、コードを1行も記述せずに説明されています
  • フレンドリーなユーザーインターフェース
  • マイクロソフトが買収後に製品を買収
  • Indie Studioサブスクリプションによる制作の所有コストは$ 99（最大10万MAU）です。Professionalに切り替えると、1k MAUサブスクリプションは$ 8（最小アカウント$ 300）になります。

否定的な機能：

  • ゲームデータの保存は厳密に制限されています。たとえば、特定のゲームセッションのデータを保存する無料サブスクリプションでは（すべてを正しく理解している場合、エンティティグループがこれに使用されます）それぞれ500バイトの3つのスロットが利用可能です
  • マルチプレーヤーを整理するには、クライアントからのイベントを処理してゲームロジックを計算するサードパーティのサーバーを接続する必要があります。これは、ハードウェア上のPhotonまたはAzure Thunderheadのいずれかであり、サーバーを整理するだけでなく、サブスクリプションを少なくともIndie Studioにアップグレードする必要があります
  • オートコンプリートのないクラウドコードとモジュールに侵入する方法がない（または見つからなかった）ことに我慢する必要があります。
  • 通常のデバッガーはなく、CloudScriptでのログの書き込みと表示のみが可能です

ゲームパーク


肯定的な機能：

  • ゲームデータストレージ。データを保存できる多くの場所（一般的なゲームメタデータ、仮想グッズ、プレーヤープロファイル、マルチプレーヤーセッションなど）があるだけでなく、プラットフォームは、何にも接続されていない本格的なDatabase-as-a-Serviceを提供します。さらに、MongoDBとRedisの両方が、さまざまなタイプのデータにすぐに利用できます。開発環境では、戦闘で10 GBを10 MB格納できます。
  • マルチプレーヤーは、同時接続数10、1秒あたりのリクエスト数10に制限された無料サブスクリプション（開発）で利用できます。
  • テストとデバッグのための組み込みツール（Test Harness）を含む、CloudCodeによる便利な作業

否定的な機能：

  • アマゾン（2018年冬）での購入以来ツールが停滞していて、イノベーションはない気がする
  • Amazon , 10 000 MAU
  • $300 ( Standard)


まず、ゲームのコンセプトを確認する必要があります。これを行うには、金銭的な投資をせずにスティックとスコッチテープのプロトタイプを作成し、ゲームメカニズムのプレイテストを開始したいと考えています。したがって、最初に選択するときは、無料サブスクリプションでメカニックを開発してテストする機会を高めます。
GameSparksはこの基準を満たしますが、ゲームクライアントのイベントを処理するサーバーとインディースタジオレベルのサブスクリプション（$ 99）が必要になるため、PlayFabは満たしません。

同時に、AmazonがGameSparksを開発しないというリスクを受け入れます。つまり、Amazonは「死ぬ」可能性があります。これと、それでも本番環境での所有コストを考えると、別のプラットフォームまたは自分のバックエンドに移動する必要がある可能性を念頭に置いています。

開発の最初のステップ


接続と認証


そのため、プロトタイピングの段階でバックエンドとしてGameSparksを選択しました。最初のステップは、プラットフォームに接続し、プレーヤーを認証する方法を学ぶことです。重要な点は、ユーザーはゲームをインストールした直後に登録やSMSなしでプレイできることです。これを行うために、GameSparksは、DeviceAuthenticationRequestメソッドを呼び出して匿名プロファイルを作成するオプションを提供します。後で、匿名プロファイルに基づいて、たとえばGoogleアカウントに接続することにより、本格的なプロファイルを作成できます。

私はTDDの頭脳を持っているので、クライアントをゲームに接続するためのテストを作成することから始めました。今後、CloudCodeはJSで作成する必要があるため、mocha.jsとchai.jsを使用してJSで統合テストを行います。最初のテストはこれでした：

var expect = require("chai").expect;
var GameClientModule = require("../src/gameClient");

describe("Integration test", function () {
    this.timeout(0);

    it("should connect client to server", async function () {
        var gameClient = new GameClientModule.GameClient();
        expect(gameClient.connected()).is.false;

        await gameClient.connect();

        expect(gameClient.connected()).is.true;
    });
})

デフォルトでは、mocha.jsのタイムアウトは2秒ですが、テストは統合されているため、すぐに無期限にします。テストでは、まだ実装されていないゲームクライアントを作成し、サーバーへの接続がないことを確認し、コマンドを呼び出してバックエンドに接続し、クライアントが正常に接続したことを確認します。

テストを成功させるには、GameSparks JS SDKをダウンロードしてプロジェクトに追加し、その依存関係（crypto-jsとws）を接続する必要があります。もちろん、GameClientModuleを実装する必要があります。

var GameSparks = require("../gamesparks-javascript-sdk-2018-04-18/gamesparks-functions");
var config = new require("./config.json");

exports.GameClient = function () {
    var gamesparks = new GameSparks();

    this.connected = () => (gamesparks.connected === true);

    this.connect = function () {
        return new Promise(function (resolve, reject) {
            gamesparks.initPreview({
                key: config.gameApiKey,
                secret: config.credentialSecret,
                credential: config.credential,
                onInit: () => resolve(),
                onMessage: onMessage,
                onError: (error) => reject(error),
                logger: console.log
            });
        });
    }

    function onMessage(message) {
        console.log("GAME onMessage: " + JSON.stringify(message));
    }
}

ゲームクライアントの最初の実装では、クライアントアプリケーションから接続を作成するための技術的な承認に必要なキーが構成から読み取られます。 connectedメソッドは、SDKの同じフィールドをラップします。最も重要なことは、connectメソッドで発生します。接続メソッドは、成功した接続またはエラーのコールバックを含むpromiseを返し、onMessageハンドラーを同じコールバックにバインドします。 onMessageはバックエンドメッセージ処理マネージャーとして機能します。今のところ、メッセージをコンソールに記録します。

私は、各クライアントがバックエンドで独自のプロファイルを作成することを確認するために、2つのクライアントをすぐに確認することにしました。これを行うには、ゲームクライアントに、GameSparksの外部に特定の識別子を渡すことができるメソッドが必要です。次に、クライアントがプレーヤーのプロファイルにアクセスしたことを確認します。GameSparksポータルで事前に準備されたプロファイル。

GameClientでの実装には、以下を追加します。

this.playerId = undefined;

this.authWithCustomId = function (customId) {
    return new Promise(resolve => {
        var requestData = {
            "deviceId": customId
            , "deviceOS": "NodeJS"
        }
        sendRequest("DeviceAuthenticationRequest", requestData)
            .then(response => {
                if (response.userId) {
                    this.playerId = response.userId;
                    resolve();
                } else {
                    reject(new Error(response));
                }
            })
            .catch(error => { console.error(error); });
    });
}

function sendRequest(requestType, requestData) {
    return new Promise(function (resolve) {
        gamesparks.sendWithData(requestType, requestData, (response) => resolve(response));
    });
}

実装は、DeviceAuthenticationRequestリクエストを送信し、応答からプレーヤーの識別子を受け取り、それをクライアントのプロパティに配置することになります。すぐに、別のメソッドで、ヘルパーがプロミスのラッパーを使用してリクエストをGameSparksに送信しました。

テストでは、ゲームクライアントの新しいインスタンスを作成し、作成されたクライアントの配列に追加するファクトリメソッドヘルパーを追加しました。特別なmochaハンドラーでは、配列内のクライアントに対してテストを実行するたびに、disconnectメソッドを呼び出してこの配列をクリアします。コードの「マジックストリング」はまだ好きではないので、テストで使用されるカスタム識別子を含む辞書を追加しました。

ゲーム検索組織（マッチメイキング）


マルチプレイヤーにとって非常に重要なマッチメイキング機能を開始します。このシステムは、ゲームの[ゲームを探す]ボタンを押すと機能し始めます。彼女はライバル、チームメイト、またはその両方（ゲームによって異なります）を拾います。原則として、このようなシステムでは、各プレーヤーには数値インジケーターMMR（マッチメイキングレシオ）があります。これは、プレーヤーの個人的な評価であり、同じレベルのスキルを持つ他のプレーヤーを選択するために使用されます。

この機能をテストするために、次のテストを思いつきました。

3つのクライアントがゲームに接続され（将来的には、いくつかのシナリオをチェックするために最低限必要です）、ゲームを検索するために登録されます。 3番目のプレーヤーをサーバーに登録すると、ゲームセッションが形成され、プレーヤーはそれに接続する必要があります。同時に、クライアントの状態が変化し、同じ識別子を持つゲームセッションのコンテキストが表示されます。

まず、バックエンドを準備します。 GameSparksには、ゲームの検索をカスタマイズするための既製のツールがあり、「Configurator-> Matches」のパスで利用できます。新しいものを作成し、セットアップを続行します。試合のコード、名前、説明などの標準パラメーターに加えて、カスタムゲームモードに必要なプレーヤーの最小数と最大数が示されます。作成したマッチに「StandardMatch」というコードを割り当て、2から3までのプレーヤー数を示します。

次に、「しきい値」セクションでプレーヤーを選択するためのルールを設定する必要があります。各しきい値について、そのアクションの時間、タイプ（絶対、相対、パーセント）および境界が示されます。



MMRが19のプレーヤーが検索を開始するとします。上記の例では、最初の10秒がMMRが19から21の他のプレーヤーの選択になります。プレーヤーが選択されなかった場合、2番目の検索枠がアクティブになり、検索範囲が16から次の20秒間拡張されます（ 19-3）から22（19 + 3）。次に、3番目のしきい値が含まれ、14（19-25％）から29（19 + 50％）の範囲で30秒間検索が実行されますが、最小必要プレーヤー数が蓄積された場合、試合は完了したと見なされます（最小マークを受け入れる。プレーヤー）。

実際、特定の試合に参加したすべてのプレーヤーのMMRを考慮に入れるため、メカニズムはより複雑です。 （この記事ではなく）ゲームの評価モードを作成するときに、これらの詳細を分析します。 MMRを使用する予定がない標準のゲームモードでは、必要なしきい値は1つだけです。

すべてのプレーヤーを選択したら、ゲームセッションを作成してプレーヤーを接続する必要があります。 GameSparksでは、ゲームセッション関数は「チャレンジ」です。このエンティティの一部として、ゲームセッションデータが保存され、ゲームクライアント間でメッセージが交換されます。新しいタイプのチャレンジを作成するには、「Configurator-> Challenges」というパスに沿って進む必要があります。そこで、「StandardChallenge」というコードで新しいタイプを追加し、このタイプのゲームセッションがターンベースであることを示します。プレイヤーは同時にではなく、順番を取ります。 GameSparksは、一連の動きを制御します。

クライアントがゲームを検索するために登録するには、MatchmakingRequestタイプのリクエストを使用できますが、プレーヤーのMMR値がパラメーターの1つとして必要であるため、お勧めしません。これはゲームクライアントの側で詐欺につながる可能性があり、クライアントはMMRを知らないはずです。これはバックエンドです。ゲーム検索に正しく登録するために、クライアントから任意のイベントを作成します。これは「Configurator-> Events」セクションで行われます。属性なしでイベントFindStandardMatchを呼び出します。次に、このイベントに対する反応を構成する必要があります。これを行うには、クラウドコードのセクション「Configurator-> Cloud Code」に移動します。そこで、「Events」セクションに、FindStandardMatchの次のハンドラーを記述します。

var matchRequest = new SparkRequests.MatchmakingRequest();
matchRequest.matchShortCode = "StandardMatch";
matchRequest.skill = 0;
matchRequest.Execute();

このコードは、MMRが0のStandardMatchにプレーヤーを登録するため、標準のゲームを検索するように登録されたプレーヤーは、ゲームセッションの作成に適しています。レーティングマッチの選択では、このタイプのマッチのMMRを取得するために、プレーヤープロファイルのプライベートデータにアピールすることができます。

ゲームセッションを開始するのに十分なプレーヤーがいる場合、GameSparksは選択したすべてのプレーヤーにMatchFoundMessageメッセージを送信します。ここでは、ゲームセッションを自動的に生成して、プレーヤーを追加できます。これを行うには、「ユーザーメッセージ-> MatchFoundMessage」にコードを追加します。

var matchData = Spark.getData();
if (Spark.getPlayer().getPlayerId() != matchData.participants[0].id) {
    Spark.exit();
}
var challengeCode = "";
var accessType = "PRIVATE";
switch (matchData.matchShortCode) {
    case "StandardMatch":
        challengeCode = "StandardChallenge";
        break;
    default:
        Spark.exit();
}

クライアントアプリケーションの時代がやってきました。クライアントモジュールの変更：

exports.GameClientStates = {
    IDLE: "Idle",
    MATCHMAKING: "Matchmaking",
    CHALLENGE: "Challenge"
}

exports.GameClient = function () {
    this.state = exports.GameClientStates.IDLE;
    this.challenge = undefined;

    function onMessage(message) {
        switch (message["@class"]) {
            case ".MatchNotFoundMessage":
                this.state = exports.GameClientStates.IDLE;
                break;
            case ".ChallengeStartedMessage":
                this.state = exports.GameClientStates.CHALLENGE;
                this.challenge = message.challenge;
                break;
            default:
                console.log("GAME onMessage: " + JSON.stringify(message));
        }
    }
    onMessage = onMessage.bind(this);

    this.findStandardMatch = function () {
        var eventData = { eventKey: "FindStandardMatch" }
        return new Promise(resolve => {
            sendRequest("LogEventRequest", eventData)
                .then(response => {
                    if (!response.error) {
                        this.state = exports.GameClientStates.MATCHMAKING;
                        resolve();
                    } else {
                        console.error(response.error);
                        reject(new Error(response));
                    }
                })
                .catch(error => {
                    console.error(error);
                    reject(new Error(error));
                });
        });
    }
}

テストに従って、クライアントにいくつかのフィールドが表示されました-状態とチャレンジ。onMessageメソッドは意味のある外観を取得し、ゲームセッションの開始に関するメッセージと、ゲームを拾うことができなかったというメッセージに応答するようになりました。対応するリクエストをバックエンドに送信するfindStandardMatchメソッドも追加されました。テストはグリーンですが、マスターしたゲームの選択には満足しています。
