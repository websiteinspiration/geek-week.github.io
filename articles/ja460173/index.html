Dockerの開発。ローカル環境。パート2-Nginx + PHP + MySql + phpMyAdmin

次の資料をよりよく理解するには、まず前の投稿を読むことをお勧めします

。Nginx+ PHP + MySql + phpMyAdminの束で構成されるローカル環境を開発する例を考えてみてください。このバンドルは非常に人気があり、通常の開発者の多くの標準的なニーズを満たすことができます。

前の投稿と同様に、最も純粋な形式のdockerよりもdocker-composeユーティリティに重点が移ります。

じゃ、行こう！

別のプロキシフォルダにあるdocker-compose.ymlから始めましょう。

nginx-proxyのdocker-compose.yml
version: '3.0'

services:

  proxy:
    image: jwilder/nginx-proxy
    ports:
      - 80:80
    volumes:
      - /var/run/docker.sock:/tmp/docker.sock:ro
    networks:
      - proxy


networks:
  proxy:
    driver: bridge


提示されたファイルは、イメージイメージjwilder / nginx-proxyに基づいてプロキシという名前で1つのコンテナを作成し、同じ名前でネットワークを作成するための構成を説明しています。networksディレクティブは、コンテナが接続されているネットワークを示します。この例では、これはプロキシネットワークです。 ネットワークを作成する場合、driver：ブリッジディレクティブを省略できます。ブリッジドライバがデフォルトのドライバです。このコンテナーは、ネットワークを介して他のコンテナーと通信します。 jwilder / nginx-proxyイメージは基本であり、Docker Hubによって取得されます



そこでは、その使用に関するかなり広範囲で詳細な説明も示されています。 nginx-proxyの動作原理は非常にシンプルです。転送されたDockerソケットを介して実行中のコンテナーに関する情報にアクセスし、VIRTUAL_HOSTという名前の環境変数の存在を分析し、指定されたホストからこの環境変数が設定されているコンテナーにリクエストをリダイレクトします。

よく知られているdocker-compose up -dコマンドでプロキシを開始し、次の出力を確認します。

Creating network "proxy_proxy" with driver "bridge"
Creating proxy_proxy_1 ... done

この結論から、最初にproxy_proxyネットワークが作成され、次にproxy_proxy_1コンテナが作成されたことがわかります。ネットワーク名は、docker-compose.ymlファイルが配置されているフォルダーの名前から取得されました。プロキシと同じ名前のネットワーク名があります。docker network lsコマンド

を入力すると、システム内のdocker networkのリストが表示され、そのうちの1つがproxy_proxyである必要があります。 コンテナー名は、フォルダー名とサービスの名前、および同様の名前のコンテナーが重複しないようにする番号と同じ原則に基づいて構築されます。container_nameディレクティブを使用して、コンテナーの名前を明示的に指定できますが、これはかなり役に立たない関数です。これについては、以下の投稿で詳しく説明します。



次の内容で2番目のdocker-compose.ymlを作成します。

他のサービスのためのdocker-compose.yml
version: '3.0'

services:

  nginx:
    image: nginx
    environment:
      - VIRTUAL_HOST=site.local
    depends_on:
      - php
    volumes:
      - ./docker/nginx/conf.d/default.nginx:/etc/nginx/conf.d/default.conf
      - ./html/:/var/www/html/
    networks:
      - frontend
      - backend

  php:
    build:
      context: ./docker/php
    volumes:
      - ./docker/php/php.ini:/usr/local/etc/php/php.ini
      - ./html/:/var/www/html/
    networks:
      - backend

  mysql:
    image: mysql:5.7
    volumes:
      - ./docker/mysql/data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD: root
    networks:
      - backend


  phpmyadmin:
    image: phpmyadmin/phpmyadmin:latest
    environment:
      - VIRTUAL_HOST=phpmyadmin.local
      - PMA_HOST=mysql
      - PMA_USER=root
      - PMA_PASSWORD=root
    networks:
      - frontend
      - backend


networks:
  frontend:
    external:
      name: proxy_proxy
  backend:


ここで何が発表されますか？

nginx、php、mysql、phpmyadminの4つのサービスが表示されます。そして2つのネットワーク。フロントエンドと呼ばれる1つのプロキシネットワークは、外部ネットワークおよび新しい内部バックエンドネットワークとして宣言されます。そのドライバーは指定されていません。以前に書いたように、ブリッジタイプのデフォルトドライバーが使用されます。

nginx


ここですべてが明確になるはずです。Dockerハブでは基本的なイメージを使用します。環境変数は、プロキシが機能するために必要であり、コンテナにアクセスできる必要があるアドレスを指定します。depends_onオプションは、このコンテナのphpコンテナへの依存を示します。これは、phpコンテナが事前に起動され、その後に依存するnginxコンテナが起動されることを意味します。次に、nginxの構成を転送します。それは少し低くなり、フォルダーをhtmlでマウントします。また、コンテナが2つのネットワークに同時にアクセスできることにも注意してください。フロントエンドネットワークからのプロキシとバックエンドネットワークからのphpの両方を通信する必要があります。原則として、すべてのコンテナーを押し出し、1つのフロントエンドを1つのネットワークに押し込むことは可能ですが、そのような分離の方が正しいと私は主張します。

default.nginx
server {
    listen 80;
    server_name_in_redirect off;
    access_log  /var/log/nginx/host.access.log  main;

    root /var/www/html/;

    location / {
        try_files $uri /index.php$is_args$args;
    }

    location ~ \.php$ {
        try_files $uri =404;
        fastcgi_split_path_info ^(.+\.php)(/.+)$;
        fastcgi_pass php:9000;
        fastcgi_index index.php;
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        fastcgi_param PATH_INFO $fastcgi_path_info;
    }

    location ~ /\.ht {
        deny  all;
    }
}


default.nginxは、コンテナに転送されるnginxの設定です。ここで重要なのは、fastcgi_pass phpディレクティブ：9000です。FastCGIサーバーのアドレスを設定します。アドレスは、ドメイン名またはIPアドレス、およびポートとして指定できます。

php：9000-サービスの名前はFastCGIサーバーのアドレスです。Nginxがphpにアクセスすると、phpが実行されているコンテナのIPアドレスが取得されます。ポート9000は標準ポートであり、ベースコンテナーの作成時に宣言されます。このポートはネットワーク経由でnginxに使用できますが、転送されなかったため、ホストマシンでは使用できません。

php


ここでは画像が表示されないのは珍しい。代わりに、作成ファイルから直接独自のイメージを作成します。コンテキストディレクティブは、Dockerfileが配置されているフォルダーを指します。

Dockerfile
FROM php:7.3.2-fpm

RUN apt-get update && apt-get install -y \
        libzip-dev \
        zip \
	&& docker-php-ext-configure zip --with-libzip \
	&& docker-php-ext-install zip \
	&& docker-php-ext-install mysqli

COPY --from=composer:latest /usr/bin/composer /usr/bin/composer

WORKDIR /var/www/html


Dockerfileは、ビルドがベースのphpイメージを使用することを示しています：7.3.2-fpm、次にphp拡張機能をインストールするためのコマンドが実行されます。次に、composerが別のベースイメージからコピーされ、プロジェクトの作業ディレクトリが設定されます。アセンブリの問題については、他の投稿で詳しく検討します。

また、php.iniファイルとプロジェクトのhtmlフォルダーがコンテナー内にスローされます。

PHPはバックエンドネットワーク上にあり、たとえば、プロキシはアクセスできなくなります。

mysql


ベースのmysqlイメージは、mysqlバージョンを担当する5.7タグで取得されます。./docker/mysql/dataフォルダーは、データベースファイルを格納するために使用されます（作成する必要はなく、起動時に作成されます）。そして、環境変数を介して、パスワードがrootユーザー、またrootに設定されます。

データベースはバックエンドネットワーク上にあるため、phpとの通信を維持できます。標準イメージは標準ポート3306を使用します。phpのdockerネットワーク経由でアクセスできますが、ホストマシンでは使用できません。このポートに転送する場合、たとえば同じPHPSTORMからポートに接続できます。しかし、phpmyadminインターフェースで十分であれば、これを行うことはできません。

phpmyadmin


phpmyadminの公式画像。環境変数は、nginxと同様に、VIRTUAL_HOSTを使用してプロキシと対話します。データベースへのPMA_USERおよびPMA_PASSWORDアクセス。PMA_HOSTはホストベース自体です。しかし、これは通常のようにlocalhostではなく、mysqlです。それら。データベースとの通信は、そのサービスの名前、すなわち mysql。phpmyadminコンテナは、バックエンドネットワーク接続を備えているため、データベースに接続できます。
