国際化とローカライゼーションの状態について知っておく必要がある3つのことがあります。私の名前はロムロシントラで、金融アーキテクチャアプリケーションに携わっています。私はTC39の人々とたくさん話し、彼らがJavaScriptの世界をより良い場所にするためにどのように努力しているかを確認します。さらに、私はオープンソースの強力な支持者であり、私は自由な時間にバルセロナのスクールオブテクノロジーの教師をしています。

国際化の問題は非常に重要です。たまたま、私たちの地球には多くの異なる人々や言語がいます。今日、世界には約195の国と6,000の言語があります。これは私たちの仕事を非常に困難にします。他のことについて考えてください。私はロシア語ではなく英語で記事を書き、レポートを読みます。私たちはすでに国際化の問題を抱えています。英語を話せない人は、あなたとの会話から除外されます。これを防ぐために、国際化が発明されました。

英語の国際化はi18nと省略されます。数字の18は、この単語の文字iとnの間の文字数です。つまり、国際化とは、ローカリゼーションを可能な限り簡素化するためのソフトウェアの設計です。国際化のおかげで、ソフトウェアはローカル設定、言語、通貨などをサポートできます。国際化により、誰もがWebにアクセスしやすくなります。ここでは、テスト（テスト駆動開発）を通じて開発と並行することができます。最初にテストが記述され、次にコードが記述されます。国際化に伴い、同じことをする必要があります。通常、人々はコードが書かれた後に国際化について考えますが、それは間違っています。

i18nと同様に、l10nはローカライズの省略形であり、10は最初のlと最後のnの間の文字数です。ローカリゼーションとは、製品が配布される言語および文化的環境に製品を適応させることです。つまり、「Hello」を「Hello」に変換するだけでなく、現地通貨や小数点なども使用する必要があります。つまり、ソフトウェアをユーザーにとってより使い慣れたものにする必要があります。単なる翻訳ではありません。

あなたのウェブプロジェクトはいくつの言語をサポートしていますか？多くは2つ以上あります。 5人以上いる人はいますか？ 15は？約25言語をサポートしています。国際化は最善の方法で組織されていないため、私たちはあまり良いサポートがありません。レポートの中で、国際化を改善する方法を説明し、私たちが取っている措置について話します。

したがって、私はもう一度繰り返します。国際化とは、ローカリゼーションを簡素化し、それをアーキテクチャレベルでサポートすることを意味します。ローカリゼーションとは、ソフトウェアをローカルの現実に適応させることです。翻訳が原文と一致しないことがよくあります。映画「痛みと利得」の名前が「血と汗：アナボリックス」と翻訳された映画業界の例を見てみましょう。



または別の例：ロシア語の風呂の広告。英語の翻訳には「ロシア火葬場」（ロシア火葬場）と記載されています。



これが、少なくとも生きている顧客を引き付けるとは思えません。国際化とローカリゼーションは、ユーザーに伝えたいことを正確に伝えることができるため、非常に重要です。本質的に、国際化はアクセシビリティの提供です。なぜなら、使用するソフトウェアを理解できない場合、実際には選択肢が限られているためです。ソフトウェアの利用範囲が広がり、収入が増えるため、ソフトウェアへのアクセスが容易になることは誰にとっても有益です。それはまたウェブをより良くします。

コード例を考えてみましょう：

文字列オブジェクトを翻訳する必要があります。HELLO_WORLD各言語に対応する行を持つ変数があります。多くの言語（Javaなど）でのこのような翻訳には、MessageFormat形式が存在します。それが何であるかを理解してみましょう。まず、いくつかの基本的なテクノロジーについて少し説明します。まず、Unicodeから始めましょう。これは、異なる言語の文字用に単一のスペースを作成する標準です。チェスを例に取ってみましょう。各種類の駒は異なる形をとることができますが、ボード上のどこに配置するべきかは常にわかっています。もちろん、バイト数が異なるUnicodeフォーマットもあります。UTF-8、UTF-16、UTF-32です。現在、最も一般的に使用されているメタタグはUTF-8です。 Unicodeを使用すると、このタグの場合、ブラウザは特殊文字を表示できます`meta忘れてください、私たちがページにどんな種類のシンボルを持っているか誰も理解しません。

Unicodeに加えて、他の2つの重要なテクノロジーはCLDRとICUです。 CLDRは、アルファベット、国、通貨、タイムゾーンなどの一種のデータベースであり、世界のさまざまな言語で保存されています。世界の6千の言語すべてがそこに存在するわけではありません。このデータベースでの作業はまだ進行中です。最後の更新は昨年10月でした。もう1つの重要なプロジェクトはICUです。これは、ソート、正規化、フォーマットなどのメソッドの形式で提供される、さまざまな言語の単語、数字、記号の巨大なデータベースです。これらのライブラリは、多くのプログラミング言語で使用されています。 JavaScriptでは、ICUはIntl APIの中核です。しかし、ブラウザに表示する必要がある世界の言語には非常に多くの多様な資料があり、これらの標準にそれらを含める作業は完全とはほど遠いものです。

MessageFormatは、特定のキーを特定の言語の特定のメッセージに関連付けることができる形式です。場合によっては、変数をMessageFormatに渡すことができ、変数を定義して最終行に入力します。同じ問題が他の言語では少し異なる方法で解決されました。 Androidでは、MessageFormatは Java で実装されています。そこでは、この形式で作業するために特別なライブラリは必要ありません。Androidはそれ自体と対話できます。 iOSには、JavaScriptのAPIとよく似たAPIがあります。これはシステムに組み込まれており、そこに何もダウンロードする必要はありません。必要な行をこのAPIのメソッドに渡すだけです。

この問題はJavaScriptでどのように解決されましたか？未だに。しかし、ソリューションを提供する多くのライブラリがあります。

Intl.DateTimeFormat


ブラウザにはすでに国際化とローカライズのための多くの組み込みメカニズムがあり、それらのほとんどは単にそれらについて知らず、それらを使用しません。 Intl.DateTimeFormatについて聞いたことはありますか？このプロジェクトでは、常に新しいAPIを作成しています。Moment.js、Day.js、date-fnsが必要なくなった可能性があります。

const myDate = new Date(); 
new Intl.DateTimeFormat('ru', { timeStyle : 'short'}).format(myDate); 
// short → 19:49 
// medium → 19:49:17 
// long → 19:49:17 GMT+2
// full → 19:49:17  ,

数か月前に作成されたtimeStyleがあり、Moment.jsを使用せずに日付と時刻をフォーマットできます。さらに、formatRangeメソッドがあります。日付範囲の選択に関連するタスク（予約機能のあるサイトなど）は常に困難です。しかし、このためのメソッドはすでにブラウザに存在しています。そして、最も重要なのは、この方法が国際化をサポートする一方で、追加のライブラリをダウンロードする必要がないことです。

Intl.RelativeTimeFormat


このプロジェクトの第2部のドキュメントを作成しました。もし参加したい場合は、ロシア語への翻訳および標準への準拠についてサポートが必要です。カウントダウンを行う必要がある場合は、RelativeTimeFormatが必要です。

const myTime = new Intl.RelativeTimeFormat('ru', { style: 'narrow' }); 
myTime.format(2 , 'quarter'); 
//Style Narrow : +2 . → in 2 qtrs. → dentro de 2 trim. 
//Style Long :  2  → in 2 quarters → dentro de 2 trimestres

これを行うのは非常に簡単です。2日、2週間、四半期などで時間を指定できます。以前は、Webにそのようなフォーマットは存在しませんでした。 

const myTime = new Intl.RelativeTimeFormat('ru', { style: 'narrow' }); 
myTime.format(2 , 'day'); 
//Style Narrow : +2 . → in 2 days → dentro de 2 días 
//Style Long :  2  → dentro de 2 días myTime.format(-1 , 'day');
//Style Narrow : -1 . → 1 day ago → hace 1 día 
//Style Long : 1   → 1 day ago → hace 1 día //Numeric(auto) :  → yesterday → ayer

これはロシア語の例です。このコードはすでにブラウザにあるので、自分でテストできます。

const myTime = new Intl.RelativeTimeFormat('ru', { style: 'narrow' }); 
myTime.format(20 , 'seconds'); 
//Style Narrow : +20  → in 20 sec. → dentro de 20 s 
//Style Long :  20  → in 20 seconds → dentro de 20 segundos

この方法は非常に便利です。上記の短い形式で時間を与えることができます。これらすべてについて、サードパーティのライブラリを使用する必要がないことを強調します。

Intl.NumberFormat


次に共有したいのはIntl.NumberFormatです。3番目のステージについて説明しますが、いくつかの変更がまだ議論されているため、例では2番目のみを示します。Intl.NumberFormatは、単位およびレコードフォームで機能します。彼がユニットで何をするかに注意を払う価値があります：彼はあなたに異なるスタイルで働くことを許します。

new Intl.NumberFormat("ru", { 
style: "unit", 
unit: "liter", unitDisplay: "long" 
}).format(16); 
// → 16  → 16 liters → 16 litros

すべてのユニットはUTC 35から取得され、それらの多くがあります。合計で約140のフォーマット用ユニットがここに表示されます。そのため、国際化はかつてないほど容易になりました。行を変換するだけで、必要なダイナミクスはすべてブラウザにすでに含まれています。

const nbr = 987654321; 
new Intl.NumberFormat('ru', { notation: 'scientific' }).format(nbr); 
// → 9,877E8 → 9.877E8 (en-US) 
new Intl.NumberFormat('ru', { notation: 'engineering' }).format(nbr); 
// → 987,654E6 → 987.654E6 (en-US) 
new Intl.NumberFormat('ru', { notation: 'compact' }).format(nbr); 
// → 988  → 988M (en-US) → 9.9亿 (zn-CN) 
new Intl.NumberFormat('ru', { notation: 'compact', compactDisplay: 'long' }).format(nbr); 
// → 988  → 988 millions (fr)

録音フォーム用になりました。正直に言うと、あまり頻繁には使用しません。指数（科学的記録）を使用した記録の形式を使用せず、大きな数を提示する必要がないためです。ただし、必要な場合は、対応するAPIが特別に用意されています。

「分離」を指定すると、「または」の和集合が得られます。



最後に、関数は使用する言語とアルファベットを自動的に決定し、それに応じてリスト項目を並べ替えることができます。

Int.PluralRules


別の重要なAPIはIntl.PluralRulesです。このAPIは最も古いものですが、何らかの理由で誰も使用していません。



レースやサッカーでファイナリストのリストを見ると、数字は常に名前の横に表示されます。「1」、「2」、「3」などですが、これは、より近いと言う方法に対応していませんスピーチの場合は、「1st」、「2nd」、「3rd」と書きます。そして、これには特別なAPIがあり、それほど難しくありません。



たとえば、「1猫」、「0猫」、「0.5猫」、「1.5猫」というフレーズを書くと、APIは自動的に正しい複数形の末尾を選択します。

Intl.DisplayNames


国のリストを表示する必要があることが多いため、これは最も人気のあるAPIの1つです。たとえば、データベースやJSONなどの国のリストがあるとします。その後、言語を切り替えるたびに、国や通貨などの新しいリストを含む個別のJSONをロードする必要があります。これらのJSONは多すぎますが、どのように終了しますか？さまざまな言語のデータベースが組み込まれたマイクロサービスを作成し、そこからすべてのデータを抽出します。もちろん、国のリストの例では幸運であり、データを頻繁に更新する必要はありませんが、常にそうであるとは限りませんよね？すべての問題を一度に解決することはできませんが、DisplayNamesはそれらのいくつかを解決します。以下の例のようなAPIがあり、通貨のリストのみまたは国のリストのみをリクエストできます。

const currencyNames = new Intl.DisplayNames(['en'], {type: 'currency'}); currencyNames.of('USD'); // "US Dollar" 
currencyNames.of('EUR'); // "Euro" 
currencyNames.of('TWD'); // "New Taiwan Dollar" 
currencyNames.of('CNY'); // "Chinese Yuan"


const languageNames = new Intl.DisplayNames(['en'], {type: 'language'}); languageNames.of('fr'); // "French" 
languageNames.of('de'); // "German" 
languageNames.of('fr-CA'); // "Canadian French" 
languageNames.of('zh-Hant'); // "Traditional Chinese"

これはとても便利なことです。国や通貨だけでなく、月、曜日など、開発者として必要な多くの機能を使用できます。

これまで、既存のAPIについて説明してきました。将来の計画に移りましょう。私の母国語はポルトガル語です。私のサイトでは、少なくともポルトガル語と英語をサポートする必要があります。そして私たちはスペインにとても近いので、スペイン語も役に立ちます。ポルトガルは非常に小さな国であり、フランスもそれほど遠くないため、このリストにフランス語を追加するとよいでしょう。
