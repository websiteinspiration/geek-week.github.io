トップ10 C ++ CoreHard Autumn 2019カンファレンスペーパー



CoreHardは、春と秋に年2回ミンスクで開催されるC ++カンファレンスです。2015年以降、私たちは愛好家の小さなコミュニティから大規模な地域会議へと成長しました。私たちは、C ++の世界で認められたマスターの知識と地元のC ++ムーブメントの開発を組み合わせて、地元のベラルーシ、ロシア、ウクライナの企業から講演者を積極的に引き付けようとしています。

私たちは、最近騒々しい会議からの、焼きたてのトップ10ビデオをハブロフスクの住民と共有することを急いでいます。

では、視聴者が評価した上位10件のレポートを見ていきましょう。

10.卓越した記憶モデル-Alexey Tkachenko


コンピュータのメモリの概念は、コンピュータにインストールされた一連の「鉄」ダイナミックメモリチップの範囲をはるかに超えています。メモリは、メモリ管理モジュール（MMU）によって管理されるアドレス可能な空間として認識されてきました。このアプローチにより、リソース管理の単位としてプロセスの概念を使用するすべてのオペレーティングシステムでアクティブに使用される物理メモリと仮想メモリを区別することができます。

レポートでは、MMUに実装されている保護方法の珍しいアプリケーションについて説明します。つまり、レジスタファイルを介してハードウェアと中央プロセッサとの相互作用をシミュレートする方法について説明します。

レポートでは、WindowsとLinuxオペレーティングシステム、およびさまざまなアーキテクチャについて説明します。
このレポートは、主に組み込み（組み込み）ソフトウェアの開発者を対象としていますが、レポートに記載されている手法は、「大規模」コンピューターの開発者に役立ちます。


講演者について

Aleksey Tkachenkoは、Peleng社で12年以上にわたり、ハードウェアおよびソフトウェアテスト用のソフトウェアと宇宙技術製品のベンチコンプレックスを開発してきました。基本的な作業ツール：低レベルタスク用のC ++およびハードウェアとのやり取り、クライアントソフトウェア用のC＃/。NET。余暇には、控えめなマイクロコントローラーの組み込みが好きです。



9. C ++タクシートリック-アントンポルキン


大規模なフレームワークを作成する場合、解決策が古くから知られている一般的なタスクを処理する必要があります。ただし、これらのソリューションは十分に効果的ではなく、はるかに優れた方法を実行できます。

レポートでは、キャッシュフレンドリーなpimpl、ユーザータイプのロギングの最適化、JSON / XML / YAMLからユーザータイプへの変換、その他の興味深いテクニックについて説明します。


スピーカーについて

Anton Polukhin-標準化C ++に関するワーキンググループの国際会議でのISOのロシア代表。C ++言語標準のいくつかの受け入れられた提案の著者。Boostライブラリの作成者であるTypeIndex、DLL、StacktraceおよびアクティブなメンテナAny、Conversion、LexicalCast、Variant。彼はBoost C ++ Application Development Cookbook and Second Editionの著者です。



8. C ++ 20：小さなこと-Timur Doumler


C ++ 20は、モジュール、コルーチン、概念、宇宙船オペレーター、および多くの新しいライブラリーなど、言語に新しい優れた機能をもたらします。ただし、さらにC ++ 20は多くの小さな改善を追加し、C ++をより強力で表現力豊かにすると同時に、より安全で全体的なものにします。

このレポートでは、ラムダ式、CTAD、構造化バインディング、初期化など、プログラマーの生活を楽にするのに役立つ言語のコアの小さな改善点について見ていきます。


スピーカーについて

Timur Doumlerは、サウンド処理および合成技術を専門とするC ++開発者であり、ISO C ++標準化委員会の積極的なメンバーであり、＃include <C ++>コミュニティの活動家です。



7.究極の低レイテンシを目指して-Mateusz Pusz


このレポートでは、低レイテンシドメインを通じてC ++の世界を見ていきます。これは、動的なメモリ割り当てと例外が歓迎されない世界であり、十分なSTLコンテナがない可能性があり、開発者はアセンブラレベルに下げて、コードが本当に必要なものを正確に最適な方法で実行することを確認する必要があることがよくあります。


講演者について

Mateusz Puszは、建築家、エンジニア、セキュリティスペシャリスト、コンサルタント、教師、そして現代のC ++の伝道者です。C ++標準化委員会のワーキンググループWG21およびSG14のメンバー。



6. Windbg：プレイしないとき-Alexander Golovach


プログラミングの実践では、多くの場合、開発者側で再現および調査できない問題が発生します。このような状況では、マシンへのリモートアクセスを構成することさえ不可能である場合があります。このレポートでは、Windbgデバッガーを例として使用して、マルチスレッドアプリケーションでのクラッシュ、リソースリーク、同期エラーなどの一般的なタスクのデバッグに関するヒントを説明します。このレポートは、主にWindowsプラットフォームを使用する初心者開発者を対象としています。


講演者について

Alexander GolovachはWindowsシステムプログラマーであり、Checkpoint Endpoint Securityの開発者です。彼はAnti Exploit、Anti Ransomware、Endpoint Forensicsなどの製品の作成に参加しました。



5. C＃でのLLVMバックエンドの動作-Egor Bogatov


LLVMには多数の最適化が含まれており、多くのプログラミング言語のバックエンドとして適しています。しかし、管理された言語とJITスクリプトの場合は少し複雑になります。このレポートでは、YegorがLLVMバックエンドを実装するときにC＃開発者が直面する困難について話します。


講演者について

Egor Bogatovは、マイクロソフトの.NET / Monoチームで働いており、マイクロ最適化とLLVMに興味があります。



4.バッファリングに関する調査：非同期アラート、更新の複製、保留中のリクエストの統合-Vadim Vinnik


特定のアプリケーションを開発することは問題の陳述につながる可能性があり、成熟した思考では、それを多くの異なるアプリケーションに潜在的に役立つ抽象に分離するのに十分一般的であることが判明します。特定のアプリケーションのニーズを満たすソリューションとは対照的に、繰り返し使用するのに適したフォームにソリューションをもたらすには、タスクのさまざまな側面を強調し、言語を使用してそれらをモデル化するための追加の精神的な努力が必要です。このレポートは、3つの抽象化を実装した経験を示しています。共通のテーマであるマルチスレッド処理でのデータバッファリングによって統合されています。タスクの分析からC ++ 17言語の実装までの一連の考え方を示します。


講師について

Vadim Vinnikは、物理学および数理科学の候補者であり、タラスシェフチェンコ国立キエフ大学を含む、学生にプログラミングを教える10年の経験があります。バックアップおよびデータ復旧ドメインに重点を置いた、産業用プログラミング（主にC / C ++ / C＃）における10年以上の経験。 Vadimは「アルゴリズム言語とプログラミングの基礎：C言語」、「C ++のオブジェクト指向プログラミングの基礎」の著者です。プログラミング理論に関する29の科学出版物の著者。書籍の英語からロシア語への翻訳者「オブジェクト指向プログラミングの原則」（Anton Eliens）、「Combining Programming Theories」（Charles Hoar、Hee Jifeng）。関心のある分野：C ++および関数型言語でのプログラミング（Haskell）



3. C ++ 14の静的SQL-Evgeny Zakharov


講演者は、C ++ 14でSQLite3用のsqlite_orm ORMライブラリライブラリを開発することについて話し、テキストクエリを忘れることができるORMを作成する彼の経験と、テンプレートを使用してそれがC ++にどのように適応するかを共有します。

2.メタプログラミング：有限状態マシンの構築-Sergey Fedorov


スイッチケースまたは仮想関数を使用した継承を使用して手動でステートマシンを実装すると、非常に時間がかかり、エラーが発生しやすくなります。レポートでは、スピーカーがステートマシンのライブラリAFSM（Another Finite State Machine）について語っています。これにより、宣言型言語を使用してステートマシンを記述できます。また、このライブラリのコードからの例に関する小さなトリックやメタプログラミングのトリックについても説明します。


スピーカーについて

Yandex.Taxiの主要開発者である Sergey Fedorovは、タクシーバックエンド用の非同期フレームワークを開発しています。メタプログラミングとオープンソースの大ファン。



1. C ++の保護-Pavel Filonov


C ++はしばしばヌンチャクと比較されます。熟練した手には効果的なツールであり、初心者の手には自分の健康に危険があります。この比較は、安全でないプログラミング言語としてのC ++の意見に明らかに触発されています。一部の新しい言語でも、この意見を独自のプロモーションに使用しています。しかし、「安全でない」という言葉の背後に隠されている具体的な危険とは何でしょうか。詳細な答えは明らかにいくつかの方向性を引き上げますが、そのうちの1つはサイバーセキュリティです。このレポートでは、C / C ++コードの脆弱性を悪用する簡単な例について説明します。スタックでシェルコードを実行し、libcに戻り、vptrを書き換え、ヒープオーバーフローします。

しかし、主な重点はそのような攻撃に対する保護に置かれています：スタックのカナリア、ASLR、不可能な領域、テストとサニタイザー、ファジング、SDLプロセス。


講演者について

Pavel Filonovは、C ++開発者としてMax Patrol SIEMシステムの開発に参加し、データサイエンティストとしてKaspersky MLADに参加しました。C ++ロシア会議のプログラム委員会のメンバー。CI / ML /パフォーマンスのトピックを専門としています。

自動化-Kirill Tikhonov


クライアントとサーバーからC ++で最も単純なWebサービスを作成します。これでC ++の部分は終了し、環境とインフラストラクチャのセットアップが完了します。組み立てとテスト実行の決定可能性を保証します。依存ライブラリのその後の更新を容易にします。静的チェック、コード検証、テスト実行を自動化します。サービスの可用性を確保し、インフラストラクチャを構成し、負荷を分散し、自動スケーリングと手動スケーリングを追加します。最後に、リクエストの5分後にコードが本番環境に置かれるように継続的な配信を構成しますが、無効な変更やプログラマーのエラーがその作業に影響を与えることはありません。


まだ手でコードを書いていますか？その後、あなたのところへ行きます！-セルゲイ・サドヴニコフ


C ++開発の世界では、大量のボイラープレートコードを記述しなければならない多くのタスクがあります。それは、シリアライゼーション、RPC、ORM、スクリプト言語バインディングなどです。そしてもちろん、たとえば、シリアライズのために：手ですべてを書き、すべてをマクロに入れ、Hanaが率いるテンプレートをドラッグし、一方をもう一方と組み合わせます。しかし、最終的に他のすべてを置き換える必要があるアプローチが1つあります。それはコード生成です。 json、bson、xml、yaml、protobuf-データを保存できる形式には多くのオプションがあります。各フォーマットには独自のライブラリがあり、ライブラリはありません。ライブラリは、データを記述する独自の方法を永続的に提供します。一部のライブラリは、自分自身を適応させてコードを微調整する必要があるインフラストラクチャ全体を提供します。そして、これは常に便利なわけではありません、自由が欲しいからですソーステキストがデータのシリアル化方法にできるだけ依存しないようにしたいと思います。そして、コード生成の最新の手段はこの自由を提供することができます、あなたはそれらを正しく使う方法を知る必要があるだけです。


C ++ rvalueの生涯災害-ArnoSchödl


右辺値リンクは、C ++ 11以降、言語に存在しています。これらは、移動オブジェクトをより効率的にするために導入されました。このような最適化とともに、この革新は、すべての自尊心のあるプログラマーが知っておく必要がある言語に多くのニュアンスをもたらします。右辺値リンクの世界に飛び込んで、慎重に調べてみましょう！


コンパイラー最適化のプリズムによるC ++例外-Roman Rusyaev


21世紀以外では、setjmp / longjmpに基づくC ++例外処理の非生産的な実装は過去のものです。 gccやclangなどの最新のコンパイラは、ゼロコスト例外処理と呼ばれるC ++例外の高度な実装を使用します。しかし、このゼロコストはどれほど真実なのでしょうか？はい、バイナリファイルのサイズを増やし、実際に例外がスローされた場合、スタックの巻き戻しプロセスを開始したり、ローカルオブジェクトのデストラクタを呼び出したりすることで支払います。しかし、例外がスローされない場合、パフォーマンスが低下しますか？コンパイラの最適化に制限はありますか？このレポートでは、C ++例外がコンパイラの最適化にどのように影響するかについて説明します。関数が例外をスローできる場合に適用できない最適化そして、どの最適化を適用するのが難しくなっています。例外の実装とコンパイラの最適化の例は、LLVMに基づいて検討されています。このレポートは、-fno-exceptionsを使用してアプリケーションを構築することが理にかなっている場合、およびC ++例外を使用するすべての魅力を享受できる場合に、例外が無効になっているコードと比較してコードのパフォーマンスが大幅に低下していないことを理解することを目的としています。


ジェネレーター、コルーチン、その他の脳の拡張性-Adi Shavit


C ++ 20はコルーチンをもたらし、それにより、ジェネレーター、反復可能な型、および範囲を作成する機能をもたらしました。コルーチンが、より軽量な抽象化と一般化を備えた、よりクリーンで読みやすいコードの作成にどのように役立つかを見ていきます。


大きくて古いプロジェクトを窒息させない方法-Yuri Minaev


世界は変わりました。以前は機能していたものが機能を停止することはありませんでしたが、不十分になりました。ペアプログラミング、コードレビュー、単体テストは依然として重要かつ必要ですが、C ++プロジェクトの適切なレベルの品質と信頼性を提供できなくなりました。多くのプロジェクトが何百回も成長しています。成長は段階的であり、誰もが何が起こったのか理解していませんでした。大規模な古いプロジェクトは、異機種混在のレイヤー（地層）で構成されており、最も重要なこととして、誰もがそれがどのように機能するかをすでに知っています。DevSecOps、静的分析、動的分析、品質測定プラットフォームなど、コードの品質と整合性を維持するのに役立つツールと方法論の時代がやってきました。


どのように助け、どのようにコンパイラを防ぐか-Andrey Oleinikov


原則として、コンパイラーが結果のバイナリー・ファイルを最適化して、できる限り速く機能するようにすることができます。しかし、コンパイラーは、プログラムが起動されるデータとハードウェアを知りません。加えて、私はコンパイルが許容できる時間をかけて欲しいと思います。このため、結果は最適ではない可能性があります。LLVMの例を使用して、プログラムを最適化し、結果を改善する方法をコンパイラーに伝える方法を確認することをお勧めします。
