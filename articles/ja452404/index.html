チャットボット


自動化は、時間、お金、人的資源を節約します。自動化の最も有望な分野の1つは技術サポートです。

企業は、サポートスペシャリストの負荷の一部を担うチャットボットを導入しています-彼らは、最も一般的な質問への回答を提供しています。高度なシステムは、サポートに頻繁に問い合わせる顧客の行動を分析し、既製のソリューションを適応させることができます。

幅広い企業が同様の製品を開発しています。たとえば、ServiceNowです。 1つのソリューションであるServiceNow Virtual Agentは、IBM Watsonスーパーコンピューターの機能を音声認識に使用します。エージェントは、ユーザーリクエストのチケットを自動的に作成し、そのステータスをチェックして、ITインフラストラクチャコンポーネントのデータベースであるCMDBと連携します。 ServiceNowチャットボットはアルバータ大学で導入されました-2 週間で、システムは着信リクエストの30％を処理することを学びました（ボリュームを80％に増やす予定です）。

ガートナーは、来年のグローバル組織の4分の1が仮想アシスタントをテクニカルサポートの最初のラインとして使用すると語っています。この数には、ボットをチャットする政府機関が含まれます年間400億ドルを節約します（PDF、p。3）。しかし、これはヘルプデスクツールの全範囲に限定されるわけではありません。

開発自動化


柔軟な方法論は新しいものではなく、多くの企業がそれらをうまく使用しています。確かに、ワークフローの大幅な再構築なしでは、会議、スプリント、その他のアジャイルコンポーネントは役に立たない：従業員が開発の進捗状況を監視することがさらに困難になり、プロセス全体の効率が低下する。

ここで、ソフトウェア開発管理システムが助けになります-今年のもう一つの傾向です。プロトタイプからリリース、サポートから新しいソフトウェアバージョンのリリースまで、アプリケーションのライフサイクル全体を制御できます。

ITギルドで開発管理アプリケーションを提供しています。これは、システムSDLC（ソフトウェア開発ライフサイクル）です。これは、複数の開発方法論（ウォーターフォールやスクラムなど）を一度に組み合わせて、それらの作業への適応を容易にするソフトウェアツールです。

注目のIB


人的要因が、ITシステムの脆弱性の主な理由です。例は以下のようになり、状況、管理者は、従業員や代理店のプロジェクトのオープンアクセスデータに残されたときに、NASAのJira-サーバと。もう1つの例は、2017年のEquifaxのハッキングです。これは、組織が脆弱性をカバーするためにパッチを適時にインストールしなかったために発生しました。


/ Flickr / Wendelin Jacober / PD

SOAR（セキュリティオペレーション、分析、レポート）システムは、人的要因を減らすことができます。彼らはセキュリティの脅威を分析し、明確なグラフとチャートでレポートを生成します。彼らの主な仕事は、会社の専門家が効果的かつタイムリーな決定を下せるよう支援することです。

SOARシステムのヘルプ脆弱性の検出と対応にかかる時間を半分にします。したがって、ブログの記事で書いたServiceNow Security Operations は、このクラスの製品です。彼は独立してITインフラストラクチャの脆弱なコンポーネントを見つけ、リスクの程度に応じてビジネスプロセスへの影響を評価します。

ITSMはクラウドに移行します


今後数年間で、クラウドサービス市場はITの最も急成長しているセグメントになります。ガートナーによると、2019年の成長率は17.5％になるとのことです。この傾向の後に、ITインフラストラクチャを管理するためのクラウドソリューションが続きます。

ITギルドでは、クラウドベースのITSMシステムを提供しています。ローカルシステムとの主な違いは、企業が使用する機能（ITOM、ITFM、ITAMや。。など。）。クラウドソリューションには、構成済みのテンプレートと構成済みのツールが付属しています。組織は彼らの助けを借りて、多くの潜在的な問題を回避して作業環境をすばやく構成し、ITインフラストラクチャをクラウドに移行して、業界のベストプラクティスに依存することができます。

たとえば、Cloud ITSMはSPLAT によって実装されました。このシステムは、IT資産を監視し、そのパフォーマンスを評価するのに役立ちます。また、ユーザーからのリクエストはクラウドで受け入れられ、処理されます。リクエストを記録するための統合システムにより、ユーザーの実装に対する制御の度合いが向上しています。

ITIL 4の適応が進行中


以前のバージョンとは異なり、ITIL 4はサービス管理の基本原則と概念に重点を置いています。特に、ライブラリは柔軟なソフトウェア開発方法-アジャイル、リーン、DevOpsと統合されました。これらのアプローチがどのように連携するべきかについてのアイデアを与えます。

今年、IT管理にライブラリを使用する企業は、イノベーションがビジネスプロセスに与える影響を決定します。開発者がより理解しやすくするために試みたITILのドキュメントは、これに役立ちます。将来的には、4番目のバージョンは、ITILを自動化、DevOpsプラクティス、クラウドシステムなどの新しいトレンドに微調整するのに役立ちます。
