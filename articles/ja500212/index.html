負荷テストAtlassian Jira、Confluence、Bitbucketパート2

ここでは、この記事のパート1を読むことができます。
パート2では、dc-app-performance-toolkitツールを使用してストレステスト用のデータを準備する方法について説明します  。

テストデータの準備


テストデータを準備するには2つの方法があります。

  • アトラシアンが提供するバックアップを使用します。このオプションは、たとえば、独自のプラグインを作成していて、このプラグインが機能するデータが問題ではない場合に使用できます。
  • 独自のデータを使用してください。このオプションは、Jira、Confluence、またはBitbucketの独自のコピーがあり、標準機能、スクリプト、プラグインなどを使用して変更する場合に便利です。

バックアップアトラシアン


アトラシアンは、負荷テストに使用できるJira、Confluence、およびBitbucketのバックアップを提供します。

これらのバックアップをインストールするには、に行く必要がアプリ-DC-パフォーマンスツールキット/アプリ/ utilに/ JIRAフォルダあなたはJiraのをテストしている場合は、  アプリ-DC-パフォーマンスツールキット/アプリ/ utilに/合流あなたがConfluenceのをテストしている場合や  Bitbucketをテストする場合は、app-dc-performance-toolkit / app / util / bitbucket。

これらのフォルダーには、Atlassianバックアップをインストールするために実行する必要があるファイルが含まれています。これらのファイルの説明は次のとおりです。

  • index-sync.sh ( Jira Confluence) – atlassian-jira.log (atlassian-confluence.log) “indexes – 100%”. , . , , Jira/Confluence. , , , , , , .
  • populate_db.sh – Postgres Jira/Confluence/Bitbucket.
  • upload_attachments.sh – data.
  • index-snapshot.sh ( Confluence) – , .

上記のスクリプトはすべて、テストするインスタンスがインストールされているのと同じ仮想マシンで実行する必要があります。これらのスクリプトは同じ仮想マシンから実行されるため、インスタンスのバージョンを個別に判別し、バージョンのデータベースダンプと添付ファイルをダウンロードします。

さて、もしPostgresを使いたくない場合はどうでしょうか？結局、populate_db.shはPostgresデータベースに対してのみ機能します。

この場合、Jiraの場合のみ、xmlバックアップをダウンロードし、標準のJira機能を使用してそれを復元できます。このバックアップへのリンクは次のとおりです。https：//centaurus-datasets.s3.amazonaws.com/jira/$ {Jira_Verson} /large/xml_backup.zip-Jira 8.0.3、7.13.6、8.5.0のサポートされているバージョン。

アトラシアンはConfluenceとBitbucketのxmlバックアップを準備しませんでした。スクリプトpopulate_db.shを使用するか、データを自分で準備する必要があります。これについては、「独自のデータの使用」で後述します。

私たちは自分のデータを使用します。


自分のデータでインスタンスをテストする場合は、これも可能です。必要なのは、データがdc-app-performance-toolkitツールで使用できることを確認することだけです。

dc-app-performance-toolkitがテストを開始すると、Taurusはprepare-data.pyスクリプトを実行し、テストするインスタンスからデータを選択します。その結果、インスタンスのデータを含むいくつかのcsvファイルが作成されます。

prepare-data.pyが必要なデータを選択できない場合、テストは失敗します。したがって、必要なデータがコピーに含まれていることを確認する必要があります。

どのようなデータが必要かを理解するには、テストする製品のprepare-data.pyファイルを調べる必要があります。

これらのファイルを調べてみましょう。

ジラ


Jiraのprepare-data.py ファイル   は、  dc-app-performance-toolkit / blob / master / app / util / data_preparation / jira /フォルダーにあります。

このスクリプトは、Jiraからデータを選択し、次のファイルを作成します。

  • issues.csv-ishのリスト。
  • jqls.csv-jqlクエリのリスト。
  • kanban-boards.scv-かんばんパネルのリスト。
  • project-keys.csv-プロジェクトキーのリスト。
  • scrum-boards.csv-スクラムパネルのリスト。
  • users.scv-ユーザーのリスト。

prepare-data.pyファイルを完全に調査するのではなく、このメソッドを確認するだけです。

def __create_data_set(jira_api):
    dataset = dict()
    dataset[USERS] = __get_users(jira_api)
    software_project_keys = __get_software_project_keys(jira_api, PROJECTS_COUNT_LIMIT)
    dataset[PROJECT_KEYS] = software_project_keys
    dataset[ISSUES] = __get_issues(jira_api, software_project_keys)
    dataset[SCRUM_BOARDS] = __get_boards(jira_api, 'scrum')
    dataset[KANBAN_BOARDS] = __get_boards(jira_api, 'kanban')
    dataset[JQLS] = __generate_jqls(count=150)

    return dataset

この方法でわかるように、テスト中に必要なすべてのデータが選択されています。

dataset[USERS] = __get_users(jira_api)

Jiraインスタンスからユーザーを選択します。ユーザー名は「performance_」で始まる必要があります。jira.ymlファイルのconcurrencyパラメーターで同時ユーザーとして設定したのと同じ数のユーザーを選択する必要があります。そのようなユーザーが見つからないか、その数が不十分な場合は、ユーザーが作成されます。ユーザーには、プレフィックス「performance_」とパスワードpasswordが付けられます。

software_project_keys = __get_software_project_keys(jira_api, PROJECTS_COUNT_LIMIT)
dataset[PROJECT_KEYS] = software_project_keys

タイプソフトウェアのプロジェクトキーを選択します。

__get_issues(jira_api, software_project_keys)

Closedステータスではないソフトウェアプロジェクトから8000を超えないソフトウェアを選択します。

dataset[SCRUM_BOARDS] = __get_boards(jira_api, 'scrum')

選択するパンパネルは250までです。

dataset[KANBAN_BOARDS] = __get_boards(jira_api, 'kanban')

選択するかんばんパネルは250までです。

dataset[JQLS] = __generate_jqls(count=150)

次の内容のjqlクエリを生成します： 'text〜“ abc *””。

したがって、テストをエラーなしで実行するには、ishとscrumおよびkanbanボードを備えたソフトウェアプロジェクトの可用性を確保する必要があります。インスタンスにこれらのオブジェクトがすでにある場合は、何もする必要はありません（変更を加えなくてもテストは成功します）。そうでない場合は、不足しているオブジェクトを作成する必要があります。

合流


 Confluence のprepare-data.pyファイルを検討してください  。

主な方法は次のとおりです。

def __create_data_set(rest_client, rpc_client):
    dataset = dict()
    dataset[USERS] = __get_users(rest_client, rpc_client, CONFLUENCE_SETTINGS.concurrency)
    dataset[PAGES] = __get_pages(rest_client, 5000)
    dataset[BLOGS] = __get_blogs(rest_client, 5000)
    return dataset

 dataset[USERS] = __get_users(rest_client, rpc_client, CONFLUENCE_SETTINGS.concurrency)

Jiraの場合と同じロジックに従って、負荷テスト用のユーザーを選択または作成します。

dataset[PAGES] = __get_pages(rest_client, 5000)

次のcqlを使用してConfluenceからページを選択します：「type = page and title！〜JMeter and title！〜Selenium and title！〜Home」。

dataset[BLOGS] = __get_blogs(rest_client, 5000)

次のcqlによってConfluenceからブログを選択します：「type = blogpost and title！〜Performance」。

したがって、Confluenceのテストを開始するには、コピーにページとブログが必要です。

Bitbucket


次に 、Bitbucket のprepare-data.pyファイルを確認します  。

主な機能は次のとおりです。

def __create_data_set(bitbucket_api):
    dataset = dict()
    dataset[USERS] = __get_users(bitbucket_api)
    dataset[PROJECTS] = __get_projects(bitbucket_api)
    dataset[REPOS] = __get_repos(bitbucket_api)
    dataset[PULL_REQUESTS] = __get_prs(bitbucket_api)
    return dataset

dataset[USERS] = __get_users(bitbucket_api)

Jiraと同じロジックに従ってユーザーを選択または作成しますが、今回は「dcapt-perf-user」プレフィックスが必要です。また、ユーザーを作成するとき、パスワードはユーザー名と同じです。

dataset[PROJECTS] = __get_projects(bitbucket_api)

プロジェクトを選択します。

dataset[REPOS] = __get_repos(bitbucket_api)

リポジトリを選択します。

dataset[PULL_REQUESTS] = __get_prs(bitbucket_api)

リクエストプールを選択します。リクエストプールの数は、bitbucket.ymlファイルの同時実行パラメータ以上である必要があります  。

したがって、Bitbucketのテストを実行するには、インスタンスプロジェクト、リポジトリ、およびrequetのプールに、bitbucket.ymlファイルの同時実行パラメータ以上の量が必要です。
