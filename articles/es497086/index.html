CQRS y microservicios en desarrollo de productos

Cómo diseñar un producto para no enterrar dinero en el suelo


¿En qué etapa de la creación de un producto o sistema para conectar el diseño arquitectónico del sistema de modo que no sea tan doloroso para el dinero gastado? Cómo decidir si combinar CQRS y microservicios.


Este artículo es para representantes comerciales que solicitan el desarrollo de una solución de TI. Le mostraremos cómo lanzar un producto y evitar costos injustificados asociados con la arquitectura. Y también vea cómo el uso de CQRS ayudará con la implementación de la funcionalidad en diferentes clientes de aplicaciones, y si los microservicios son la panacea.


Brevemente sobre CQRS


CQRS (segregación de responsabilidad de consulta de comandos) : una plantilla utilizada en el desarrollo de sistemas, que establece que cualquier método del sistema puede ser una solicitud (sin cambiar el estado del sistema) o un comando (cambiar el estado del sistema). Como muestra la práctica, este es uno de los patrones más utilizados en el desarrollo de software. Se puede usar en diferentes niveles y en diferentes situaciones. Por ejemplo, la división clásica de sistemas en OLTP / OLAP, cuando los datos a menudo se escriben en el sistema OLTP y se leen desde el sistema OLAP, no es más que el uso de la plantilla CQRS en la arquitectura de la base de datos.

public class NinjectFactory : IHandlersFactory

        private readonly IResolutionRoot _resolutionRoot;

        public NinjectFactory(IResolutionRoot resolutionRoot)
        {
            _resolutionRoot = resolutionRoot;
        }

        public IAsyncCommandHandler<TCommand> CreateAsyncCommandHandler<TCommand>()
        {
            return _resolutionRoot.Get<IAsyncCommandHandler<TCommand>>();
        }

        public IAsyncQueryHandler<TQuery, TResult> CreateAsyncQueryHandler<TQuery, TResult>()

            return _resolutionRoot.Get<IAsyncQueryHandler<TQuery, TResult>>();
