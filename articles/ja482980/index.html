Duanは、人気のあるAmazon S3ストレージプラットフォームの例を示しています。 S3からのファイルの書き込みと取得を可能にするために、Amazonはサービスと対話するための包括的で詳細なAPIを開発しました。たとえば、保存されたファイルのリスト（ListObjects）を取得するには、ホストと型encoding-type、継続トークン、およびx-amz-date。 Amazon S3を使用するには、ソフトウェアでこれらおよび他の多くの特定のパラメーター名を正確に使用する必要があります。

Amazon S3 APIのコピーを提供する企業の中には、Oracle自体があります。互換性のために、Amazon S3互換性APIは、x-amzタグに至るまで、多数のAmazon API要素をコピーします。

API（ライブラリ、命令セット）のコピーと再利用は正しいだけでなく、そのようなプログラミング方法論はコンピューターサイエンスの規範で直接推奨されています。S3プログラミングインターフェースをコピーする前でも、Oracle自体はこれを何度も行っていました。さらに、Oracleビジネスはもともと、IBMが開発したSQLプログラミング言語の実装に基づいて構築されていました。Oracleの最初の主力製品はDBMSで、主にIBM System Rからコピーされました。ここでは、DBMSの「標準API」としてのSQLの再実装について説明しています。

APIに知的財産権をオーバーレイすると、誰もが苦しむ法的地雷原を作成できます。APIは他のクラウドサービスも実装します。 Wi-Fiやインターネットプロトコルなどの多くの技術標準には、APIが含まれています。プログラムインターフェイスは、インターネット上の各コンピュータとサーバーに何らかの形で必然的に再実装されます。 Oracle Copyright Theoryは、コンピュータで行うほとんどすべての行為を違法行為に変える可能性があります。

これらの広範囲にわたる結果を回避するために、オラクルと彼女の主張を支持した控訴裁判所は、著作権侵害を、オリジナルと「互換性のない」APIの再実装のみに制限しようとしました。しかし、部分的な再実装も一般的です。 S3 APIのコピーでさえ、Oracleは多数の「相違点」と元のAmazon APIとの非互換性を指摘しています。

オラクルの訴訟の主な危険は、それが小規模なテクノロジー企業がS3などの支配的なプラットフォームと互換性のあるシステムのバージョンを作成することを妨げる可能性があることです。そのような互換性がなければ、プログラマーはこの会社の申し出で事実上ブロックされます。
