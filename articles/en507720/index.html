How x86_x64 addresses memory

Today I‚Äôm going to talk about memory addressing: one would seem to be a small, but nevertheless surprisingly difficult element of the semantics of x86_64 architecture commands. In particular, I want to talk about the command mov



and how, through only this x86_64 command , various memory addressing methods become available to the user.



I will not talk about the rest of the memory-affecting commands (that is, thanks to CISC, almost everyone else), commands that write massive fragments of memory (this is about you,fxsave

Some limitations



Despite the nightmare of coding x86_64 commands, and maybe thanks to it, memory addressing has some limitations.



Let's start with the good:



  • At a fairly high level, there are only two addressing modes in the x86_64 architecture.
  • All registers in both addressing modes must be exactly the same size. In other words, we cannot strangely mix 64, 32 and 16-bit registers and get the current address - there simply is no place for such a maneuver in x86_64 encoding.


Otherwise, everything is not so rosy:



  • One of these two addressing modes is still absurdly complex.
  • The registers should be the same size, but not required to be the same bit depth as the processor mode. For example, if we add the prefix byte ( 0x67



    ) in our coding , we can use 32-bit registers instead of 64-bit ones.


Addressing ‚ÄúScale-Index-Base-Displacement‚Äù



I have no idea what to call this mode, so I call it ‚ÄúScale-Index-Base-Displacement‚Äù. As far as I understand, neither Intel nor AMD at all perceive it as a single mode and instead mention it as a typical set of connected modes with a wide range of different encoding methods.



But today we are not talking about coding, but about semantics, and semantically each of these three connected modes is reduced to a combination of several of the following four parameters:



  • Scale : 2-bit constant coefficient, usually 1, 2, 4, or 8.
  • Index : Any general register (rax, rbx, & c).
  • Base : Any general purpose register.
  • Displacement : Integral Shift. Usually, even in 64-bit mode, it is limited to 32 bits, but using some encoding methods it can also be 64-bit. We will talk more about this.


These four parameters can be combined in several different combinations, a complete list in increasing order of complexity can be seen below:



  • Displacement



  • Base



  • Base + Index



  • Base + Displacement



  • Base + Index + Displacement



  • Base + (Index * Scale)



  • (Index * Scale) + Displacement



  • Base + (Index * Scale) + Displacement





Let's look at each combination in order.

Base



Addressing through the base register adds another level of uncertainty over absolute addressing: instead of using displacement



the absolute address encoded in the command field , the address is loaded from the specified public register (And any such register! Hooray!)



This uncertainty allows us to perform absolute addressing with an arbitrary destination register by the following pattern:

More uncertainty! If you haven‚Äôt guessed yet, calculating the current address with both the register base

Some addressing modes, as we have already seen in the example Base + Index

RIP Relative Addressing



Above, we described the addressing mode, which is almost identical to its historical equivalent in x86_32, and differs primarily in the use of 64-bit GPR registers, and sometimes in 64-bit offsets. However, the main difference is the addition of a completely new addressing mode called ‚ÄúRIP-relative‚Äù addressing.



Why is this mode called RIP Relative? Because it encodes the offset relative to the value of the RIP register (in particular, the RIP is not the current, but the next command). Usually this is expressed by the syntax already familiar to us [Base + Displacement]



, but now instead of GPR, base



it is now a register rip



. Example:

I promised not to describe the reasons why this mode is used in this post: it is an address-independent code and code model. But we will make a small exception: RIP-relative addressing simplifies and reduces address-independent code and is ideal for ‚Äúsmall‚Äù (and basic) code models in which all code and data must be addressed within a 32-bit shift.

Finally: segmentation



The x86_64 architecture removed all segmentation, but only almost. Thanks to the flat address space, segment registers are no longer required, but still appear in places:



  • Linux (more precisely glibc) uses fs



    in user space to access TLS segments configured by the kernel. The specification of these segments can be found in the per-CPU GDT configuration ( link ). Assuming nothing uses glibc (or any of your libc) gs



    , you can use it freely.
  • In kernel space, Linux uses a gs



    per-CPU variable to store the primary address of the region. We can observe this in the macro definition PER_CPU_VAR



    ( link ):
    #define PER_CPU_VAR(var)  %__percpu_seg:var
  • In x86_64, the definition grows:
    %gs:var


So, unfortunately, we still have to keep an eye on segmentation. On the other hand, this is not so difficult, because it all comes down to adding a value from the segment register to calculating the entire address.
