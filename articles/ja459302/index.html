Angularで失敗したHTTPリクエストを再試行する

サーバーデータへのアクセスの編成は、ほとんどすべての1ページアプリケーションの基本です。このようなアプリケーションのすべての動的コンテンツは、バックエンドからダウンロードされます。

ほとんどの場合、サーバーへのHTTP要求は確実に機能し、望ましい結果を返します。ただし、状況によっては、要求が失敗することがあります。

誰かが国中を時速200キロの速度で移動する列車のアクセスポイントを介してWebサイトを操作する方法を想像してみてください。このシナリオのネットワーク接続は遅くなる可能性がありますが、サーバーの要求はこれにもかかわらず、その仕事をします。

しかし、列車がトンネルに入るとどうなりますか？インターネットへの接続が中断され、Webアプリケーションがサーバーに「到達」できない可能性が高くなります。この場合、列車がトンネルを出てインターネット接続が復元された後、ユーザーはアプリケーションページをリロードする必要があります。

ページをリロードすると、アプリケーションの現在の状態に影響を与える可能性があります。これは、たとえば、ユーザーがフォームに入力したデータを失う可能性があることを意味します。

特定の要求が失敗したという事実と単に調整するのではなく、それを数回繰り返し、ユーザーに対応する通知を表示する方が良いでしょう。このアプローチでは、ユーザーがアプリケーションが問題に対処しようとしていることに気付いた場合、ほとんどの場合、ページをリロードしません。



今日私たちが公開している翻訳は、Angularアプリケーションで失敗したリクエストを繰り返すいくつかの方法の分析に当てられています。

ドキュメントには、オペレーターの説明が含まれretryWhenています。ドキュメントでは、次のように説明されています。「を除いて、元のObservableを再生するObservableを返しますerror。元のObservableが呼び出す場合error、このメソッドはThrowableをスローしnotifierます。これによりエラーが発生し、Observableがから返されます。このObservableがcompleteまたはを呼び出す場合error、このメソッドは子サブスクリプションで呼び出しcompleteますerror。それ以外の場合、このメソッドは元のObservableに再度サブスクライブします。」

はい、定義は単純ではありません。同じことをよりアクセスしやすい言語で説明しましょう。

オペレーターretryWhenは、Observableを返すコールバックを受け入れます。返されるObservableはretryWhen、いくつかのルールに基づいてオペレーターがどのように動作するかを決定します。つまり、これは演算子の動作retryWhenです。

  • 返されたObservableがエラーをスローすると、動作を停止してエラーをスローします。
  • 返されたObservableが完了を報告すると終了します。
  • 他の場合では、Observableが正常に戻ると、元のObservableの実行を繰り返します

コールバックは、元のObservableが初めてエラーをスローしたときにのみ呼び出されます。

したがって、上記のコードスニペットに基づいて、独自のRxJsオペレーターを記述しましょう。

import {Observable, of, throwError} from 'rxjs';
import {delay, mergeMap, retryWhen} from 'rxjs/operators';

const getErrorMessage = (maxRetry: number) =>
  `Tried to load Resource over XHR for ${maxRetry} times without success. Giving up`;

const DEFAULT_MAX_RETRIES = 5;

export function delayedRetry(delayMs: number, maxRetry = DEFAULT_MAX_RETRIES) {
  let retries = maxRetry;

  return (src: Observable<any>) =>
    src.pipe(
      retryWhen((errors: Observable<any>) => errors.pipe(
        delay(delayMs),
        mergeMap(error => retries-- > 0 ? of(error) : throwError(getErrorMessage(maxRetry))
        ))
      )
    );
}

スヌーズする指数リクエスト


失敗したリクエストの遅延再試行のアイデアを発展させましょう。以前は、繰り返される各要求の実行を同時に遅延させていました。

ここでは、各試行後の遅延を増やす方法について説明します。リクエストを再試行する最初の試行は、1秒後に、2秒後に2秒、3秒後に3回行われます。この動作を実装

する新しいオペレーターを作成しますretryWithBackoff。

import {Observable, of, throwError} from 'rxjs';
import {delay, mergeMap, retryWhen} from 'rxjs/operators';

const getErrorMessage = (maxRetry: number) =>
  `Tried to load Resource over XHR for ${maxRetry} times without success. Giving up.`;

const DEFAULT_MAX_RETRIES = 5;
const DEFAULT_BACKOFF = 1000;

export function retryWithBackoff(delayMs: number, maxRetry = DEFAULT_MAX_RETRIES, backoffMs = DEFAULT_BACKOFF) {
  let retries = maxRetry;

  return (src: Observable<any>) =>
    src.pipe(
      retryWhen((errors: Observable<any>) => errors.pipe(
        mergeMap(error => {
            if (retries-- > 0) {
              const backoffTime = delayMs + (maxRetry - retries) * backoffMs;
              return of(error).pipe(delay(backoffTime));
            }
            return throwError(getErrorMessage(maxRetry));
          }
        ))));
}

この演算子をアプリケーションで使用してテストすると、新しい試行のたびに、繰り返し要求の実行の遅延がどのように増加するかを確認できます。

各試行の後、一定の時間待機し、リクエストを繰り返し、待機時間を増やします。ここでは、いつものように、サーバーがリクエストに対して正しい答えを返した後、アプリケーションウィンドウにメッセージを表示します。
