Angular：アプリケーションを確認する必要があるが、バックエンドの準備がまだ整っていない場合

フロントエンドの開発に携わっている場合、おそらく次のmise-en-sceneはおなじみです。プロジェクトの条件が急速に縮小している、管理者、または顧客、あるいはその両方が、偽のデータであっても、現在動作しているアプリケーションを目で確認したいと考えています。同時に、裏があるかもしれませんが、それは表との相互作用のapiレイヤーで、まったく単語がありません。

それで、最近私はそのような状況に直面し、私は角度のあるフロントエンドを開発しています（椅子に座っている人がだらしなく撫でてくれた、誰かが理解してうなずいた）。

真剣にやってみます。一方で、状況は珍しいことではなく、多くのソリューションを選択できます。

いくつかの解決策が思い浮かびました：

  1. コンポーネントレベルのデータをハードコードする
  2. リゾルバーサービスのレベルでデータをハードコードし、必要なルートにピン留めします
  3. データサービスプロバイダーレベルでのデータのハードコード
  4. APIをフラッシュし、合意された契約に従ってハードコードされたデータを返す

しかし、これらのオプションのいずれも無尽蔵の松葉杖のようであり、それぞれに重大な欠点がありました。

  1. 最初のオプションはすぐに姿を消しました-プロジェクトが発展するにつれ、まったく不便な未使用のソリューションです。すべてを書き直す必要があります。
  2. このソリューションは実行できますが、プロジェクトの構造とコンポーネントのロジックが書き直されます。
  3. 可能なオプションは、スタブデータを非同期で返し、サーバー呼び出しをシミュレートすることもできますが、問題に対する以前のソリューションと同様に、インターセプター（存在するが存在する場合）は動作しなくなり、シミュレーションが機能することがわかりますバッキング付きは不完全になります。
  4. 最後のオプションは、前のオプションにあった問題がないので、かなり受け入れられるように見えましたが、衛生上の理由から、バックエンドプロジェクトにハードコードを記述したくありませんでした。

その結果、別のオプションが選択されました。ルートとコントラクトに従ってデータを返す別のWebサーバーを起動し、アセンブリと角度の実行用に別の構成を構成することです。どちらも簡単にできることがわかりました。

export const environment = {
  production: false,
  backend: 'http://localhost:3000/'
}

ここでは、ご覧のとおり、Expressを実行するポート3000でAPIを探しています。

ここで、構成に応じて、呼び出しを正しいサーバーに戻すインターセプターが必要です。

@Injectable()
export class RequestInterceptor implements HttpInterceptor {
  baseUrl: string;

  constructor() {
    this.baseUrl = environment.backend;
  }

  intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {
    return next.handle(this.modifyRequest(req));
  }

  private modifyRequest = (req: HttpRequest<any>): HttpRequest<any> => {
    if (req.url.startsWith('api/')) {
      const url = this.baseUrl;
      req = req.clone({
        url: url + req.url
      });
    }
    return req;
  }
}
