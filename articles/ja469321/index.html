Cisco 200-125 CCNA v3.0のトレーニング。51日目。EIGRPのトラブルシューティング

今日は、EIGRPプロトコルの実装をトラブルシューティングする方法について説明します。これを行うには、このビデオの下に表示されているwww.nwking.org/product/resources-day-51-eigrp-troubleshootingでこのトピックに関する研究室の作業をダウンロードする必要があります。その後、一緒に作業します。

ICND2コースのセクション2.6のトピックは引き続き継続しており、前述のとおり、サイトにアクセスして、ラボ「Day 51：EIGRPトラブルシューティング」をダウンロードする必要があります。これは、Cisco Packet Tracerプログラムのアーカイブファイルです。



このラボの初期設定はすでに完了しているため、最も一般的なプロトコル構成の問題のトラブルシューティングを3か月前に検討できます。それ以来、このビデオチュートリアルを終える時間がなかったので、私たちが話している問題の種類を忘れていました。

このラボの目的は、PC1、PC2、PC3の3台のコンピューターが相互にpingできることを確認することです。これが今可能かどうか見てみましょう。 PC1コマンドラインターミナルに移動し、コマンドping PC2：ping 10.1.20.20を入力すると、宛先ホストが利用できないというシステム通知が表示されます。状況は、10.1.30.30でのPC3のpingと同様です。

次に、PC2から他のコンピューターにpingを実行します。 PC3と同様に、10.1.10.10のPC1は使用できません。これは、ルーターR2のルーティングテーブルで、必要なエントリが欠落していることを意味します。

PC3からpingを開始してみましょう。ご覧のとおり、PC1とPC2も使用できません。

コンピューター間の接続を確立できない理由を見てみましょう。バンガロール、ボストン、パリの3つのオフィスがあり、ネットワークはEIGRPプロトコルを使用して相互に接続されています。 R1ルーターのトラブルシューティングから始めましょう。 CLIコンソールに移動してshow ip eigrpコマンドを入力しますが、最初にこのデバイスでEIGRPが設定されていることを確認したいと思います。これを行うには、show runを使用します。 eigrpを開始します。ご覧のとおり、プロトコルは機能します。



EIGRPプロトコルが3つすべてのルーターで実行されていることを確認するために、R2とR3に対し​​て同様のチェックを実行します。プロトコルは機能しますが、pingは通過しません。

R1設定に移動し、show ip routeコマンドを入力します。 EIFRPのおかげで、ルーターはいくつかのルートを「学習」したことがわかりますが、これらはNull0とマークされた要約ルートです。



覚えていれば、ルートを自動合計すると、ルーターはこの機能の設定に従ってすべてのルートを集約し始めます。この場合、彼は実際には10.0.0.0ネットワークの一部ではない多くのサブネットのルートを含む、10.0.0.0 / 8ネットワークブロック全体を合計しました。

ほとんどすべてのネットワークプロトコルは、トラフィックのループを防ぐためにNull0の概念を使用しています。この概念を理解していただくために、1つの例を示します。 R1ルーター、R2ルーター、インターネットがあるとします。 R1とR2の間にデフォルトルートデフォルトルートがあります。つまり、最初のルーターは特定のルートを知らない場合、外部ネットワークに接続されているため、デフォルトルートに沿ってすべてのトラフィックをR2に送信します。

2つのネットワークが接続されているときに、R1がルートの合計を実行するとします：10.1.1.0/24および10.1.2.0/24。その結果、合計ルート10.0.0.0/8がR2に送信されます。



最初のルーターのメッセージを受信した後、R2はそのルーティングテーブルに、ネットワーク10.0.0.0/8へのルートがR1経由で実行されたことを記録します。 10.1.3.0/24ネットワークのトラフィックが2番目のルーターに到着するとします。このエントリにガイドされて、R2はこのトラフィックをR1に転送します。

10.1.3.0ネットワークのトラフィックを受信すると、R1ルーターはそのトラフィックをルーティングテーブルで見つけませんが、R2へのデフォルトルートがあるため、すべての不明なトラフィックに対処する必要があるため、受信したメッセージをR2ルーターに送信します。 R2はそれを再度R1に送信し、TTLパケットライフタイムカウンターが0に達するまで続くトラフィックループを取得します



。ルートを合計して作成されたルーティングループは、自動合計モードのルーターがNull0を使用して解決するという問題です。したがって、ルーティングテーブルにNull0が存在する場合、これは自動または手動のルート合計の証拠となります。

このように、R1はNull0疑似インターフェイスを使用するため、ルートの集約を使用することを理解しています。これは「ブラックホール」のように機能し、ルーティングテーブルに記載されている特定のサブネットのアドレスを除いて、10.0.0.0 / 8ブロックのすべてのアドレス宛てのトラフィックが消失します。私たちの場合、これらはネットワーク10.1.10.0/24および10.1.10.1/32です。同時に、これらのサブネットのトラフィックは対応するインターフェイスを通過し、ブロック10.0.0.0/8の残りのアドレスのトラフィックはNull0に送信されます。
簡単に言えば、あるネットワークでルーティングテーブルに特別なルートがない場合、そのアドレス宛のパケットは、トラフィックの「キラー」として機能する「ブラックホール」Null0にダンプに送信されます。これが、動的ルーティングプロトコルでのトラフィックのループを防止するメカニズムのしくみです。

現時点では、ルートの合計がコンピュータ同士がpingを実行できない理由であるとは言えませんが、ここでは何らかの合計があることに注意する必要があります。

テーブルには近隣のデバイスへのルートがないため、近隣があることはわかりませんでした。 R2設定に入り、show ip routeコマンドを入力します。 R2はルート合計を実行しないことがわかります。テーブルにNull0インターフェイスがないためです。



R3にも同じコマンドを使用していますが、Null0がテーブルに存在するため、ここでは何らかのタイプの合計も使用されていることがわかります。



回路図では、3つの事業所のネットワーク間の接続が赤色で表示されているため、問題があると判断できます。 show ip eigrp interfacesコマンドを使用して、R3にEIGRPプロトコルを使用するインターフェイスのステータスを表示するように依頼すると、R2ルータが接続されているGigabitEthernet0 / 0インターフェイスとSe0 / 3/0シリアルインターフェイスが表示されます。



show ip interface briefコマンドを入力すると、Se0 / 3/0がオフになっており、ダウン状態であり、このルータに接続しているPC3に問題がないことがわかります。
次に、R1ルーターの設定に同じコマンドを入力します。ここでも同様の状況があります。つまり、R1とR3を接続するSe0 / 3/0インターフェイスも無効になっています。おそらく、これがコンピュータにpingを実行できない理由です。



この問題の解決に努めます。技術的には、トポロジーに従って、ルータR1とR3の間にネイバー関係が形成されるはずです。



対応するシリアルインターフェイスが管理上のダウン状態であるため、論理レベルではルーターが接続されていないため、これは発生しませんでした。誰かが誤ってシャットダウンコマンドなどを使用した可能性がありますが、この問題の原因を気にする必要はありません。修正するだけです。したがって、R1ルーターの設定で、コマンドconfig t（erminal）、int（erface）se0 / 3/0、no shut（down）を順に入力します。このインターフェースをオンにすると、システムが構成を開始します。すべてが順調に進んだことがわかります。デバイス192.168.13.3を使用して新しい近隣が確立されたという記録があります。次に、show ip eigrp interfacesコマンドを入力して、R1で3つのインターフェイスがすべて有効になっていることを確認します。

show ip eigrp neighborsコマンドは、192.168.13.3ネットワークが接続されているse0 / 3/0インターフェイスがネイバーに接続されていることを示しています。



ルーターR1とR2の間の近隣が確立されなかったのはなぜですか？ EIGRPは、R2が接続されているSe0 / 3/1インターフェイスを含むすべてのR1インターフェイスで起動されますが、R1ルーターはそのネイバーを認識しません。このデバイスで実行されているすべてのネットワークプロトコルを表示するshow ip protocols testコマンドを使用してみましょう。 R2コンソールで同じコマンドを入力すると、両方の設定ウィンドウで情報を比較できます。



最初のケースでは、eigrp-1とAS1が機能し、2番目のケースではeigrp-10とAS10が機能します。近隣を確立するためには、ASが一致する必要があることを知っているため、問題の原因は正確にこれにあります。近隣を確立できないことの2番目の理由は、メトリックKの係数の異なる値である可能性がありますが、私たちの国ではそれらは一致しています。 CCNAコースでは、これらの係数の問題を解決する必要はありません。一致する必要があることに注意してください。そうしないと、ルーター間に近傍を確立することができません。

同様に、認証設定が使用された場合、デバイスのパスワードは一致する必要があります。認証もCCNAコースの一部ではないので、このルールを覚えておいてください。

もう1つの違いがあります。R1ルーターは自動ルート集約を使用しますが、R2はこの機能を無効にします。おそらく、これはある程度、問題を引き起こす可能性がありますが、この場合、合計は近隣関係の確立に影響を与えません。
R3ルーターのパラメーターには、近隣を確立しているため、eigrp-1も存在すると思います。問題の原因はR2ルーターのeigrp-10であるため、この設定をeigrp-1に変更してみましょう。これを行うには、R2コンソールでshow run |コマンドを入力します。 eigrpを開始し、ネットワークルートRouting for Networksのパラメータをコピーします。



次に、グローバルコンフィギュレーションモードを開始し、no router eigrp 10コマンドを入力してからrouter eigrp 1コマンドを使用した後、上記でコピーしたRouting for Networksパラメーターの3行を貼り付けます。ご覧のとおり、Serial0 / 3/1インターフェイスに接続されているデバイス192.168.23.3とのネイバー関係をすぐに確立しました。

show ip eigrp neighborsコマンドを入力すると、R2に2つのネイバーがあることがわかります。これは、アドレス192.168.12.1のR1ルーターとアドレス192.168.23.3のR3ルーターです。



R1コンソールで同様のコマンドを入力した後、シリアルインターフェースSe0 / 3/1に接続された新しいネイバー-R2ルーターも持っていることがわかります。



これらのルーターが近所を確立した今何が起こったのか見てみましょう。



show ip routeコマンドは、10.1.10.0 / 24および10.1.20.0/24ネットワークがR1ルーターに接続されていることを示していますが、10.1.30.0 / 24ネットワークはどこにありますか？ R2ルーターでも同様の状況が見られます。これは、10.1.20.0 / 24ネットワークのみを認識します。つまり、それ自体が10番目と30番目のネットワークに「気付かない」ことです。その理由は、ルートの自動合計により問題が発生したためです。

R2コンソールで、インターフェイス192.168.12.1を介した自動合計の結果として表示されます。およびネットワーク10.0.0.0/8の192.168.23.3ロードバランシング。



自動合計はR1-R2セクションで行われ、これはサブネット192.168.12.1であり、R2-R3セクションではこれは192.168.23.3サブネットです。



負荷分散モードでは、非常に興味深いことが起こります。 PC2コマンドラインターミナルに移動して、ping 10.1.10.10コマンドを入力します。つまり、PC1コンピューターにpingを実行します。最初に10.1.10.1からの応答を受け取り、続いて192.168.23.3のホストが利用できないというメッセージを受け取ります。



これは、ロードバランシングがどのように機能するかを示しています。R2ルーターは、R​​2-R1チャネルで1つのpingパケットを送信し、R1-R3チャネルで2番目のパケットを送信する、というように続きます。ルータR1は要求を受信して​​応答を送信し、ルータR3は宛先が10.1.10.10であるパケットを受信しましたが、ルーティングテーブルにないため、宛先が何であるかわかりません。

R3設定に入り、show ip routeコマンドを入力します。ルーターは10.1.10.10について何も認識していないことがわかりますが、Null0インターフェイスで合計ルート10.0.0.0/8を使用しています。



したがって、デバイスにアドレス指定されたすべてのトラフィックは、アドレスブロック10.0.0.0/8に存在しますが、ルートがルーティングテーブルに記述されていない場合は、ループを防ぐためにNull0「ダンプ」に送信されます。

同様の状況は、アドレス10.1.30.30のPC3コンピューターにpingを実行したときにも発生します。これにより、接続が確立され、インターフェイス192.168.12.1を介したpingが実行されます。つまり、PC1コンピューターへのパケットが通過せず、「宛先ホストが利用できません」というメッセージが表示されます。



したがって、ルートの自動合計が本当に問題を引き起こしていると確信しました。したがって、R1の設定に戻り、すべての自動合計をキャンセルします。これを行うには、コマンドconfig t、router eigrp 1を入力し、自動要約は入力しません。 R3ルーターについても同様です。

何が起こったのか見てみましょう。 PC2コマンドラインターミナルに移動して、ping 10.1.10.10コマンドを入力します。ご覧のとおり、PC1のpingは成功しています。同様に、10.1.30.30のPC3コンピューターに簡単にpingを実行します。

R2コンソールに移動し、show ip routeコマンドを入力します。 EIGRPプロトコルによってアナウンスされたすべてのサブネット10.1.10.0/24、10.1.20.0/24、および10.1.30.0/24のルートのルーティングテーブルエントリが表示されていることがわかります。これで、ルーターはすべてのサブネットに接続する方法を認識しました。



これで、すべてのデバイスが問題なく相互にpingするはずです。 PC3は、10.1.20.20のPC2と10.1.10.10のPC1に正常にpingすることがわかります。 PC2とPC3をpingするときの状況は、PC1と同様です。ご覧のとおり、すべてが非常に簡単です。 EIGRP構成のトラブルシューティングに使用する必要があるテストコマンドを含むスライドに戻りましょう。



トラブルシューティング時に最初に行うべきことは、接続の物理的な状態をチェックして、すべてのスイッチとルーターがオンになっていて、正常に動作していることを確認することです。これで問題が解決しない場合は、show ip protocolコマンドを使用します。これは、デバイスで実行されているプロトコルと、それらの設定の設定を判別するのに役立ちます。このチームは、eigrp 1とeigrp 10の不一致を検出し、この問題を修正するのに役立ちました。次に、一貫してshow ip eigrp interface、show ip eigrp neighbor、およびコマンドを使用して、インターフェイステーブル、ネイバーテーブル、およびトポロジテーブルを表示する必要があります。

show ip eigrp interfaceコマンドを使用すると、どのインターフェイスが機能しているかを確認できます。R1ルーターの例でこれを確認してみましょう。 2つのシリアルインターフェイスにネイバーがあることがわかります。



show ip eigrp neighborコマンドを使用すると、これらのネイバーが何であるかを確認できます。



show ip eigrp topologyコマンドは、FD情報とネットワーク全体のその他のトポロジパラメータを提供します。



これらのコマンドを使用する前に、ネットワークデバイスの物理的な状態を確認することをお勧めします。また、ルートを追加するときにNull0インターフェイスがトラフィックのループを防ぐのに役立つことを忘れないでください。自動加算自体は問題を引き起こしませんが、場合によってはそれが可能であることを知っておく必要があります。
