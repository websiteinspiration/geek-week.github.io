Goのハッシュテーブル。実装の詳細



ジェネリックスのない言語でのマップの実装について説明し、ハッシュテーブルとは何か、Goでの配置方法、この実装の長所と短所は何か、この構造を使用するときに注意する必要があるものについて検討します。

カットの下の詳細。


注意！Goのハッシュテーブルにすでに慣れている場合は、基本をスキップしてここに移動することをお勧めします。そうしないと、最も興味深い瞬間に飽きるおそれがあります。

ハッシュテーブルとは


まず、ハッシュテーブルとは何かを思い出させてください。これは、キーと値のペアを格納し、原則として関数を所有できるようにするデータ構造です。

  • マッピング： map(key) → value
  • インサート： insert(map, key, value)
  • 除去： delete(map, key)
  • 探す： lookup(key) → value

Go言語のハッシュテーブル


go言語のハッシュテーブルはキーワードマップで表され、以下のいずれかの方法で宣言できます（これらについては後で詳しく説明します）。

 m := make(map[key_type]value_type)
 m := new(map[key_type]value_type)
 var m map[key_type]value_type
 m := map[key_type]value_type{key1: val1, key2: val2}

主な操作は次のとおりです。

  • インサート： 

    m[key] = value

  • 除去： 

    delete(m, key)

  • 探す： 

    value = m[key]

    または 

    value, ok = m[key]

囲碁でテーブルを回る


次のプログラムを検討してください。

package main

import "fmt"

func main() {
	m := map[int]bool{}
	for i := 0; i < 5; i++ {
		m[i] = ((i % 2) == 0)
	}
	for k, v := range m {
		fmt.Printf("key: %d, value: %t\n", k, v)
	}
}

起動1：

key: 3, value: false
key: 4, value: true
key: 0, value: true
key: 1, value: false
key: 2, value: true

実行2：

key: 4, value: true
key: 0, value: true
key: 1, value: false
key: 2, value: true
key: 3, value: false

ご覧のように、出力は実行ごとに異なります。そして、すべてGoのマップは順序付けされていない、つまり順序​​付けされていないためです。これは、移動するときに順序に依存する必要がないことを意味します。その理由は、言語ランタイムのソースコードにあります。

マップは関数にどのように渡されますか？


渡された数値を変更しようとする関数があるとします。呼び出しの前後で何が起こるか見てみましょう：

package main

func foo(n int) { 
    n = 10 
}

func main() {
	n := 15
	println("n before foo =", n)
	foo(n)
	println("n after foo =", n)
}

例はかなり明白だと思いますが、それでも結論が含まれています。

n before foo = 15
n after foo = 15

ご想像のとおり、n関数は参照ではなく値で取得されているため、元の変数は変更されていません。

同様のマパトリックをやってみましょう：

package main

func foo(m map[int]int) { 
    m[10] = 10 
}

func main() {
	m := make(map[int]int)
	m[10] = 15
	println("m[10] before foo =", m[10])
	foo(m)
	println("m[10] after foo =", m[10])
}

そして見よ、見よ：

m[10] before foo = 15
m[10] after foo = 10

値が変更されました。「まあ、Mapaは参照渡しされますか？」番号。Goにはリンクがありません。たとえばC ++のように、1つのアドレスで2つの変数を作成することはできません。ただし、同じアドレスを指す2つの変数を作成できます（ただし、これらはポインターであり、Goにあります）。

マップmを初期化する関数fnがあるとします。メイン関数では、単に変数を宣言し、それを送信して初期化し、その後に何が起こったかを調べます。

package main

import "fmt"

func fn(m map[int]int) {
	m = make(map[int]int)
	fmt.Println("m == nil in fn?:", m == nil)
}

func main() {
	var m map[int]int
	fn(m)
	fmt.Println("m == nil in main?:", m == nil)
}

結論： したがって、変数mはvalueによって渡されたため、通常のintを関数に渡した場合と同様に、変更されませんでした（fnの値のローカルコピーが変更されました）。では、なぜm自体の値が変化するのでしょうか。この質問に答えるために、言語ランタイムのコードを検討してください。

m == nil in fn?: false
m == nil in main?: true



// A header for a Go map.
type hmap struct {
	// Note: the format of the hmap is also encoded in cmd/compile/internal/gc/reflect.go.
	// Make sure this stays in sync with the compiler's definition.
	count     int // # live cells == size of map.  Must be first (used by len() builtin)
	flags     uint8
	B         uint8  // log_2 of # of buckets (can hold up to loadFactor * 2^B items)
	noverflow uint16 // approximate number of overflow buckets; see incrnoverflow for details
	hash0     uint32 // hash seed

	buckets    unsafe.Pointer // array of 2^B Buckets. may be nil if count==0.
	oldbuckets unsafe.Pointer // previous bucket array of half the size, non-nil only when growing
	nevacuate  uintptr        // progress counter for evacuation (buckets less than this have been evacuated)

	extra *mapextra // optional fields
}

Go in Mapは、hmap構造へのポインタにすぎません。これは、マップが値によって関数に渡されるという事実にもかかわらず、その中の値自体が変化するという質問への答えです-それはすべてポインターの問題です。 hmap構造には次の要素も含まれます：要素の数、「バケット」の数（計算を高速化するために対数として表示）、ハッシュをランダム化するためのシード（追加を困難にするために-継続的な衝突が発生するようにキーを取得することを試みます）、あらゆる種類のサービスフィールドそして最も重要なのは、値が保存されているバケットへのポインタです。写真を見てみましょう：



図は、メモリ内の構造の概略図を示しています-Goでマップされるポインターであるhmapヘッダーがあります（varで宣言すると作成されますが、初期化されていないため、挿入しようとするとプログラムがクラッシュします）。バケットフィールドはキーと値のペアのリポジトリであり、そのような「バケット」がいくつかあり、それぞれに8ペアが含まれています。まず、「バケット」には追加のハッシュビット用のスロットが含まれています（e0..e7はeと呼ばれます- 余分なハッシュビットのため）。次は、最初にすべてのキーのリストとしてのキーと値であり、次にすべての値のリストです。

関数のハッシュに従って、どの「バケット」に値を入れるかが決定されます。各「バケット」内では最大8回の衝突が発生する可能性があり、各「バケット」の最後に、前の衝突がオーバーフローした場合、追加の衝突へのポインターがあります。

マップはどのように成長しますか？


ソースコードで次の行を見つけることができます： 

 // Maximum average load of a bucket that triggers growth is 6.5.

つまり、各バケットに平均6.5を超える要素がある場合、バケット配列の増加が発生します。同時に、配列は2倍以上割り当てられ、非常に大きな遅延が発生しないように、挿入または削除のたびに古いデータが小さな部分にコピーされます。したがって、データを退避するプロセスでは、すべての操作が少し遅くなります（検索する場合も、2か所で検索する必要があります）。避難が成功すると、新しいデータが使用され始めます。


マップ要素のアドレスを取得します。


もう1つの興味深い点-私がやりたかった言語を使用する最初の段階では、次のようにしました。

package main

import (
	"fmt"
)

func main() {
	m := make(map[int]int)
	m[1] = 10
	a := &m[1]
	fmt.Println(m[1], *a)
}

しかしGoは言う：「m [1]のアドレスをとることはできません」。値のアドレスを取得できない理由の説明は、データの退避手順にあります。値のアドレスを取得してから、mapaが大きくなり、新しいメモリが割り当てられ、データが退避され、古いメモリが削除され、ポインタが正しくなくなったため、このような操作は禁止されています。

ジェネリックなしでマップはどのように実装されますか？


空のインターフェースもコード生成もそれとは何の関係もありません;全体はコンパイル時にそれを置き換えることです。Goの使い慣れた機能が次のようになることを考慮してください。

v := m["k"]     → func mapaccess1(t *maptype, h *hmap, key unsafe.Pointer) unsafe.Pointer
v, ok := m["k"] → func mapaccess2(t *maptype, h *hmap, key unsafe.Pointer) (unsafe.Pointer, bool)
m["k"] = 9001   → func mapassign(t *maptype, h *hmap, key unsafe.Pointer) unsafe.Pointer
delete(m, "k")  → func mapdelete(t *maptype, h *hmap, key unsafe.Pointer)

アクセス用のmapaccess関数があり、それぞれmapassignとmapdeleteの書き込みと削除を行います。すべての操作はunsafe.Pointerを使用します。これは、それが指す型を気にしません。各値に関する情報は、型記述子によって記述されます。

type mapType struct {
    key   *_type
    elem  *_type ...}
type _type struct {
    size  uintptr
    alg   *typeAlg ...}
type typeAlg struct {
    hash func(unsafe.Pointer, uintptr) uintptr
    equal func(unsafe.Pointer, unsafe.Pointer) bool...}

MapTypeは、キーと値の記述子（_type）を格納します。型記述子の場合、比較、ハッシュの取得、サイズなどの操作（typeAlg）が定義されているため、それらの生成方法は常にわかっています。

これがどのように機能するかについてもう少し。v = m [k]（キーkからvの値を取得しようとする）を書き込むと、コンパイラーは次のようなものを生成します。

kPointer := unsafe.Pointer(&k)
vPointer := mapaccess1(typeOf(m), m, kPointer)
v  = *(*typeOfvalue)vPointer

つまり、キーへのポインターであるmapType構造体を取得し、そこからキーと値の記述子、hmap自体へのポインター（つまり、マップ）を見つけ、それらすべてをmapaccess1に渡します。これは、値へのポインターを返します。ポインターを目的の型にキャストし、逆参照して値を取得します。

ここで、ランタイムからの検索コードを見てみましょう（これは、読み取り用に少し調整されています）。

func lookup(t *mapType, m *mapHeader, key unsafe.Pointer) unsafe.Pointer {

関数はマップ内のキーを検索し、対応する値へのポインターを返します。引数はすでによく知られています。これは、キータイプと値の記述子を格納するmapTypeで、マップ自体（mapHeader）と、キーを格納するメモリへのポインターです。値を格納するメモリへのポインタを返します。

        if m == nil || m.count == 0 {
                return zero
        }

次に、マップヘッダーへのポインターがnullでないかどうかを確認し、要素が0の場合はnull値を返します（ある場合）。

    hash := t.key.hash(key, m.seed)     // hash := hashfn(key)
    bucket := hash & (1<<m.logB-1)   // bucket := hash % nbuckets
    extra := byte(hash >> 56)           // extra := top 8 bits of hash
    b := (*bucket)(add(m.buckets, bucket*t.bucketsize)) // b := &m.buckets[bucket]

キーハッシュを計算します（型記述子から特定のキーを計算する方法を知っています）。次に、どの「バケット」に移動して確認する必要があるかを理解しようとします（「バケット」の数で割った残りの部分、計算はわずかに加速されます）。次に、追加のハッシュを計算し（ハッシュの最上位8ビットを取得）、メモリ内の「バケット」の位置を決定します（アドレス計算）。

   for {
           for i := 0; i < 8; i++ {
                   if b.extra[i] != extra { // check 8 extra hash bits
                           continue
                   }
                   k := add(b, dataOffset+i*t.key.size) // pointer to ki in bucket
                   if t.key.equal(key, k) {
                           // return pointer to vi
                           return add(b, dataOffset+8*t.key.size+i*t.value.size)
                   }
           }
           b = b.overflow
           if b == nil {
                   return zero
           }
   }

見れば検索はそれほど複雑ではありません。「バケット」のチェーンをたどり、見つからない場合は次のチェーンに進みます。「バケット」での検索は、追加のハッシュの迅速な比較から始まります（そのため、それぞれの先頭にあるこれらのe0 ... e7は、迅速な比較のためのペアの「ミニ」ハッシュです。一致しない場合はさらに進んで、一致する場合はさらに慎重にチェックします。検索の疑いのあるキーがメモリのどこにあるかを判断し、それが要求されたものと等しいかどうかを比較します。等しい場合は、メモリ内の値の位置を特定して戻ります。ご覧のとおり、超自然的なものはありません。

結論


マップを使用しますが、それらがどのように機能するかを理解してください！いくつかの微妙な点を理解することで、レーキを回避できます。なぜ、値のアドレスを取得することが不可能であるのか、初期化せずに宣言中にすべてが落ちる理由、要素の数がわかっている場合（事前に回避を回避します）、事前にメモリを割り当てる方が良い理由などです。
