.NET Core 3.0の診断の改善

.NET Core 3.0では、パフォーマンスの問題の診断と解決を簡素化する.NETランタイム環境の新機能を利用する一連のツールを紹介します。

これらの機能は、次のような一般的な診断の質問に答えるのに役立ちます。

  1. アプリケーションは動作していますか？
  2. アプリケーションに異常な動作があるのはなぜですか？
  3. アプリケーションがクラッシュするのはなぜですか？



アプリケーションは動作していますか？


時間の経過とともに、アプリケーションでメモリリークが発生し、最終的にOutOfMemoryExceptionが発生する可能性があります。他の場合では、問題のあるコードによってプロセッサの負荷が急上昇する可能性があります。これらは、メトリックで積極的に特定できる問題のほんの一部です。

指標


指標は、一定期間の測定データです。これらのメトリックにより、システムの状態を高レベルで監視できます。Windowsの.NET Frameworkとは異なり、.NET Coreはパフォーマンスカウンターを生成しません。代わりに、EventCounter APIを使用して.NET Coreでメトリックを生成する新しい方法を導入しました。

EventCountersは、.NET Coreをサポートするすべてのオペレーティングシステムで使用されるようになったため、Windowsパフォーマンスカウンターよりも優れています。さらに、パフォーマンスカウンターとは異なり、低特権環境（xcopyの展開時など）でも使用できます。残念ながら、パフォーマンスモニター（perfmon）などのツールがないため、これらのインジケーターをリアルタイムで表示することは困難です。 dotnet

-counters
3.0-preview5では、.NET Coreアプリケーションによって生成されたメトリックをリアルタイムで監視するための新しいコマンドラインツールを提供します。

このツールをインストールするには、次のコマンドを実行します

dotnet tool install --global dotnet-counters --version 1.0.3-preview5.19251.2

以下の例では、Webアプリケーションへのロードを開始すると、アプリケーションのCPU使用率とメモリが増加することがわかります。



このツールの使用方法の詳細については、プロジェクトのdotnet-countersのreadmeファイルを参照してください。dotnet-countersの既知の制限については、GitHubの未解決の問題をご覧ください。

アプリケーションに異常な動作があるのはなぜですか？


メトリックは異常な動作の発生を識別するのに役立ちますが、何が問題だったかについてはほとんど理解できません。アプリケーションに異常な動作が発生する理由を尋ねるには、トレースを通じて追加情報を収集する必要があります。たとえば、トレースによってコンパイルされたCPUプロファイルは、コード内のホットパスを特定するのに役立ちます。

痕跡


トレースは、固定されたタイムスタンプ付きの離散イベントレコードです。トレースには、システムの運命をより適切に判断できるローカルコンテキストが含まれています。従来、.NET Framework（およびASP.NETなどのフレームワーク）は、Windowsのイベントトレース（ETW）を使用して、内部コンポーネントに関する診断トレースを生成していました。.NET Coreでは、これらのトレースはWindowsではETW、LinuxではLTTngで記録されました。 dotnet

-trace

バージョン3.0-preview5では、各.NET Coreアプリケーションは、EventPipe（* nixのUnixドメインソケットまたはWindowsの名前付きパイプ）という二重チャネルを開いて、イベントを送信できます。コントローラープロトコルの作業は継続していますが、dotnet-traceはこのプロトコルの予備バージョンを実装しています。

このツールをインストールするには、次のコマンドを実行します

dotnet tool install --global dotnet-trace--version 1.0.3-preview5.19251.2



上記の例では、CPUプロファイラーイベントとランタイム.NETイベントを含むデフォルトプロファイルでドットネットトレースを実行しています。

デフォルトのイベントに加えて、実行しようとしている調査に基づいて追加のプロバイダーを有効にすることができます。

dotnetトレースを実行した結果、.netperfファイルを取得します。このファイルには、perfviewで視覚化できるランタイムおよびCPUスタックフェッチイベントが含まれています。 Visual Studio（16.1）の次の更新では、これらのトレースを視覚化するためのサポートも追加されます。



OS XまたはLinuxを実行している場合は、.netperfファイルを.speedscope.jsonファイルに変換できます。Speedscope.app。
次のコマンドを実行して、既存のトレースを変換できます。

dotnet trace convert <input-netperf-file>

以下の画像は、先ほど受け取ったトレースを視覚化した図を示しています。



このツールの使用方法の詳細については、Readmeファイルを参照してください。dotnet-traceの既知の制限については、GitHubの未解決の問題をご覧ください。

アプリケーションがクラッシュするのはなぜですか？


プロセスを監視するだけでは、異常な動作の原因を特定できない場合があります。プロセス障害や、プロセスヒープ全体へのアクセスなどの追加情報が必要になる状況では、プロセスダンプの方が分析に適しています。

ダンプ分析


ダンプは、プロセスの作業仮想メモリの状態の記録であり、通常、予期せず終了したときにキャプチャされます。カーネルダンプ診断は、アプリケーションのクラッシュや予期しない動作の原因を特定するためによく使用されます。

従来は、アプリケーションがクラッシュしたとき（たとえば、Windowsエラーレポート）にオペレーティングシステムを使用してダンプを受け取るか、特定の起動条件が満たされたときにprocdumpなどのツールを使用してダンプをキャプチャしていました。

これまでのところ、Linuxで.NETを使用したダンプダンプの問題は、gcoreまたはデバッガーを使用したダンプダンプが非常に大きなダンプを引き起こしたことです。

さらに、これらのダンプを収集した後でも、デバッガーを購入し、それを.NETのデバッガー拡張機能であるsosをロードするように構成する必要があったため、分析が困難でした。 dotnet

-dump

3.0.0-preview5では、WindowsとLinuxの両方でプロセスダンプを収集して分析できる新しいツールを提供しています。

dotnet-dumpは現在も開発が進んでおり、下の表は、どのオペレーティングシステムで現在サポートされている機能を示しています。



このツールをインストールするには、次のコマンドを実行します

dotnet tool install --global dotnet-dump --version 1.0.3-preview5.19251.2

dotnet-dumpをインストールしたら、次のコマンドを実行してプロセスダンプをキャプチャできます。

sudo $HOME/.dotnet/tools/dotnet-dump collect -p <pid>

Linuxでは、次のコマンドで結果のダンプをロードすることにより、結果のダンプを分析できます

dotnet dump analyze <dump-name>

次の例では、ASP.NET Core Hosting Environmentダンプを定義しようとしています。



このツールの使用方法の詳細については、readmeファイルを参照してください。dotnet-dumpの既知の制限については、GitHubで未解決の問題を確認してください。
