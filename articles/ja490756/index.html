Jenkinsをコードとしてデプロイする

注意 perev。：これは、Jenkinsのような人気のあるCI / CDツールのコードとして構成を使用する方法について、Preplyのエンジニアリングブログの記事を翻訳したものです。

弊社では、「すべてがコードのようである」という慣行に従うよう努めています。これは、インフラストラクチャリソースだけでなく、監視、ジェンキンスの作業などにも当てはまります。この記事では、この手法を使用してJenkinsをデプロイおよびサポートする方法について説明します。これは、サーバーとエージェントのインフラストラクチャだけでなく、プラグイン、アクセス、作業、その他多くのことにも当てはまります。

さらに、この記事では、次のような質問に対する答えを見つけようとします。

  • 私たちのジェンキンスはより安定しましたか？
  • サーバーとジョブの構成を頻繁に変更できますか？
  • ジェンキンスのアップデートはまだ私たちにとって苦痛ですか？
  • すべての変更を制御できますか？
  • fakapが発生した場合にJenkinsをすばやく復元できますか？



前書き


通常、「DevOpsツール」というフレーズについて最初に頭に浮かぶのは、CI / CDシステムです。たとえば、毎日数百のタスクを実行し、それが数万のビルドであるため、Jenkinsを使用しています。Jenkinsで使用する一部の機能は、他のCI / CDシステムでは使用できないか、機能が制限されています。


インフラストラクチャ、構成、ジョブ、プラグインを含むコードからJenkinsを完全に制御したいと考えています。私たちは、KubernetesでJenkinsを実行しようとしましたが、それは私たちのニーズに適合しませんでした。さらに、そのアーキテクチャのため、拡張が容易ではありませんでした。

これは議論されます

AWSを使用し、Terraformと他のハッシュツール（PackerやVaultなど）を使用してインフラストラクチャ全体を構成します。

前述のように、KubernetesでJenkinsを使用してみましたが、PVC、リソース、およびあまりうまく設計されていないアーキテクチャのスケーリングでいくつかの問題に直面しました。

Dockerを使用してJenkinsとそのプラグインをインストールします


次に、Jenkinsとそのプラグインをインストールします。プラグインの更新に常に問題があったため、主な目標は、インストールされたプラグインとそのバージョンをコードに明確にキャストすることでした。
そして、ここではDockerが役に立ちます。既製のプレインストールされたDockerイメージを取得して、構成のベースとして使用できるからです。

FROM jenkins/jenkins:2.215

ENV CASC_JENKINS_CONFIG /jenkins_configs
USER root

#   
RUN apt update && \
    apt install -y python3 python3-pip && \
    pip3 install awscli jenkins-job-builder jjb-reactive-choice-param --no-cache-dir

USER jenkins
VOLUME /jenkins_configs
VOLUME /var/jenkins_home

#  
COPY plugins.txt /usr/share/jenkins/ref/
RUN /usr/local/bin/install-plugins.sh < /usr/share/jenkins/ref/plugins.txt

サーバー構成用のコード（JCaSC）プラグインとしてのJenkinsの構成


一般に、Jenkins Configuration as Code（JCasC）と呼ばれるプラグインがあり、これにより、サーバー構成を人間が読めるテキスト形式で保存できます。

このプラグインを使用して、セキュリティ構成、アクセス、プラグインの設定、エージェント、タブなどを記述できます。

設定はYAML形式で提示され、5つのブロックに分かれています。

  • 資格情報（システムシークレットの説明）
  • jenkins（承認とクラウドの設定、グローバル設定、エージェントの説明、一部のセキュリティ設定とタブ）
  • セキュリティ（許可されたスクリプトなどのグローバルセキュリティ設定）
  • ツール（git、alureなどの外部ツールの構成）
  • 未分類（Slackとの統合など、その他の設定）

このプラグインは、既存のJenkinsインストールからの構成のインポートをサポートしています。

さらに、プラグインはさまざまなシークレットプロバイダーをサポートしていますが、この例では環境変数のみを使用します。

私たちはフリースタイルのプロジェクトにジョブビルダーを使用しています


Jenkinsでフリースタイルジョブを作成するには、いくつかの方法があります。

  • Webインターフェースを使用する（最も簡単な方法は、ジャンプして続行した）
  • REST APIを直接使用する
  • Job DSLやJJBラッパーなどのプラグインを使用する

Jenkins Job Builder（JJB）では、YAMLまたはJSONを使用してジョブを構成できます。また、すべてのジョブを構成してその状態を条件付きgitに保存できるため、非常に便利です。つまり、実際には、JJBを使用してCI / CDツールのCI / CDプロセスを構築できます。

.
├── config.ini
├── jobs
│   ├── Job1.yaml
│   | ...
│   └── Job2.yaml
└── scripts
    ├── job1.sh
    | ...
    └── job2.sh
これは、FSでのジョブの構成の構造がどのように（簡略化されて）見えるかです。

- job:
    name: Job1
    project-type: freestyle
    auth-token: mytoken
    disabled: false
    concurrent: false
    node: jenkins_agent
    triggers:
      - timed: '0 3 * * *'
    builders:
      - shell:
          !include-raw: ../scripts/job1.sh
そして、これがファイル内のジョブがどのように見えるか、Job1.yamlスクリプト内のステップですjob1.sh。JJB

構成ファイルも単純に見えます。

$ cat config.ini
[job_builder]
ignore_cache=True
exclude=jobs/Job2

[jenkins]
url=https://jenkins.example.org
user=some_user
password=some_password

$ jenkins-jobs --conf config.ini test -r jobs/
$ jenkins-jobs --conf config.ini update -r jobs/
新しい変更の適用は、[それjenkins-jobs update

自体] コマンドを使用して簡単に起動できます。トークンが作成されたユーザーは、ジョブを作成および構成するための適切な権限を持っている必要があります。JenkinsのJJBを使用して変更を適用する、1つの初期化ジョブ（シードジョブ）を適用するだけです。

あまり人気のないプラグインはサポートされていないため、JJBは「特効薬」ではないことに言及する価値があります。ただし、マクロサポートを含め、ジョブをコードに保存するための非常に柔軟なツールです。
