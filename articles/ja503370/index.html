クラウドをバックアップするには？Veeamのオプション



クラウドにアップロードされた情報はどこにもなく、そこから失われることはありません！これは、最も人気のあるITの伝説の1つを策定する方法であり、クラウドの堅さを賞賛し、平凡なオンプレミスサーバーからの移行による普遍的な幸福の始まりを約束します。
ただし、実際にはすべてが完全に間違っており、クラウドでのデータ損失の脅威は、罪深い地球に保存されているものと同じくらい深刻に受け止められるべきです。今日はこれについてお話しします。なぜクラウドがデータを失っているのか、Azureの例、クラウドベースのベンダーが提供するバックアップの種類、Veeamの例でこれらのプロセスを改善する方法を見てみましょう。

神話の破滅


偶然または故意ですが、多くの場合、最初のクラウドごとに約束されたファイブナインの可用性を見て、このステートメントは、そこに置いたすべてのものの安全性の血の誓いと同じであると認識します。ただし、クラウドプロバイダーは、クラウドが機能することを約束するだけです。そこから展開したサービスがファイルをクラックしたり暗号化したりしないように、それは決して気になりません。または、従業員が自宅ですべての文字を消去した場合、または開発者がステージングとセールを混同し、今では戦闘基地がない場合。そのような「OR」は多数ありますが、結果は常に同じです-クラウドは機能しますが、クラウド内のデータは失われるか、不要なゴミに変わります。

したがって、私たちは「私たちの父」として繰り返します。クラウドプロバイダーは、クラウド内のデータではなく、クラウドの可用性を保証します。損失や損傷から顧客データを保護することは、クラウドではなく、顧客の責任です。

クラウドボックスから出てくるもの


私たちを取り巻く現実のほとんどは資本主義の法則の対象であるため、クラウドプロバイダーが独自のペニーを稼いでいる間、この問題を解決することをクラウドプロバイダーが提案しなかったとしたら奇妙です。

ここで、私たちは自分自身に重要な質問をしなければなりません。正確に何を保持する必要があるかについての整合性ですか。クラウドの機能を最大限に活用し、IaaSモデルに取り組んでいるとします。次に、2つの質問が私たちを悩ませています。インフラストラクチャ全体のレベルでバックアップを提供する方法と、個々の仮想マシンのバックアップをどのように提供するのでしょうか。

Azureの場合、一見すると、組み込みのAzure Backupで両方の問題を正常に解決できますが、詳しく見ていきましょう。

Azure Backupは本当に強力で優れたツールです。これは、IaaSレベルのバックアップインフラストラクチャ用のAzure Backupと、MARS-個々のマシンレベルのエージェントバックアップの2つのバージョンで機能する組み込みサービスです。どちらも標準ポータルに統合されており、RPOの違いを除いてほとんど同じです。つまり、Azure Backupには24時間（つまり、1日に1回しかバックアップされません）、MARSには8時間かかります。まあまあのオプション、ビジネスが1時間以内にRPOタスクを設定する場合。また、Azureポータルにアクセスできるすべてのユーザーがバックアップにアクセスできるため、卵の保存をバスケットの数に依存させることわざと直接矛盾します。これは、ソフト削除機能によって部分的に相殺されます。これにより、「削除」ボタンをクリックした後、14日間削除されたデータにアクセスできることが保証されます。しかし、問題があります-この機能をオフにして、誰にも言わないでください。

そして、おそらく、重要なニュアンス-Azureのデータは通常のストレージBLOBに保存され、圧縮はそれらに適用されますが、重複排除は適用されません。インスタンス+ GBあたりの支払いに関して、これは不愉快な経済的問題になる可能性があります。

代理店を紹介します


しかし、ネイティブのMarsエージェントの代わりにVeeam Agent for Windowsを使用するとどうなるでしょうか。これには技術的な制限がないため、このオプションを安全に実装できます。高度なオプションを使用することもできます。容量階層の役割で、スケールアウトリポジトリの一部としてAzure Blobを接続します。これにより、便利なことに加えて、誤って（そうではなく）バックアップファイルが削除されてしまう問題が解決されます。



チェーン全体は次のようになります。既存のVeeam B＆Rインストールでタスクを作成し、エージェントをターゲットマシンにインストールして、バックアップを作成します。設定に従って、バックアップはリポジトリに保存され、最後にバックアップファイルがAzure Blobにアップロードされます。データをオブジェクトストレージに送信する前に特別な方法で構造化する必要があるため、BLOBへの直接バックアップはまだ可能ではありません。これについて詳しくは、NASバックアップが実装された場所に関する最近の記事を参照してください。原則として、Veeam B＆RサーバーをAzureに直接デプロイする場合、バックアップを地上にドロップする必要さえありません。

はい、このオプションを使用すると、RPOが大幅に削減された、より柔軟なバックアッププロセスを構築する機会が得られますが、その代償としてここで失うことになります。そして、そのようなソリューションのアーキテクチャは、正直なところ、扱いにくいように見えます。したがって、次に進みます。

Veeam Backup for Microsoft Azure


4月末に、Azureの容量をバックアップするための本格的な製品をリリースしました。イデオロギー的には、Veeam for AWSに似ていますが、どちらもパブリッククラウド内のデータを気にする必要がないという事実に加えて、機能の類似性は終わりです。結局のところ、AzureはHyper-Vであり、AWSはKVMに基づいています。

しかし、今こそ言葉から行動に移るときです。そして、私たちの新製品を詳しく見てみましょう。

したがって、Veeam Backup for Microsoft AzureはAzure内で実行されるサービスであり、マシンとドライブをAzure BLOBに直接バックアップできます。現場では、Ubuntu 18.04 LTSが搭載されたVeeam制御マシンと、マシンのバックアップ時に追加のワーカーが含まれているように見えます。ホストマシンは、管理者のオーケストレーション、保持ポリシーのスケジューリング、追跡、およびバックアップ、スナップショットの作成、さらには仮想マシンの復元のプロセスを開始します。つまり、典型的な脳の中枢です。

労働者にも独自の特徴があります。彼らの主なタスクは、データをスナップショットからBLOBに転送して保存することだけでなく、トランザクションの数（したがって最終的な顧客チェック）が最小限になるように実行することです。それらの数は、バックアップポリシーの1つの仮想マシン（Veeam B＆Rからのバックアップジョブのアナログ）の原則に従ってスケーリングされます。このマシンのバックアップが作成されるまで、ワーカーは次のマシンに切り替えません。したがって、バックアップを開始するたびに新しいマシンの雪崩が発生しないようにするために、最大数と最小数は設定で明示的に設定されます。自動展開モードを使用する場合、システムは次のようになります。たとえば、最小-3、最大-10などの範囲を指定します。したがって、ポリシーには10人のワーカーが関与します。ポリシーの完了後、しばらくすると、制御マシン7は削除され、残りの3つはオフになります。労働者は手動で配置することもできます。ただし、いずれの場合も、ワーカーはバックアップ内のマシンと同じリソースグループに配置されます。

はい、バックアップはAzure BLOBにのみ保存されます。すぐに地面に着陸することはできません。

Veeamが適切に機能するには、Azureサービスアカウントレベルのアカウントを提供する必要があります。彼はAzure ADアプリケーションです。マシン構成、サブスクリプション、リソースグループ、ストレージアカウント、スナップショット、Azure Lighthouseなど、必要なすべてのリソースに関する情報を取得するために使用されます。技術的な制限により、使用できるサービスアカウントは1つだけで、管理サーバーが展開されているテナントにのみアクセスできます。

バックアップの作成プロセスについて


時間Xが到着すると、Veeamはマシン構成と各マシンのすべてのVHDを要求します。その後、ディスクスナップショットが作成され（乱れた時間にスナップショットがない場合）、マシン構成はVeeamマシン上のデータベース（興味がある場合はPostgreSQLデータベース）に保存されます。マネージドディスクを使用している場合（申し訳ありませんが、「マネージドディスク」という言葉は私を笑わせます）、それらのスナップショットはマシンのリソースグループに保存されます。また、管理されていないディスクがある場合、そのスナップショットは元のストレージアカウントに残ります。その後、データは圧縮、暗号化され、Azure Blob Storageに送信されて保存されます。
