セキュリティ-脅威からのオブジェクトと興味の保護。
これは、サイバーセキュリティと通常の物理の両方に当てはまります。

  • 競合他社。

モニタリング


さまざまなバリエーションで、誰もが監視しています。中小企業は、サイトの可用性と、複雑なインフラストラクチャの何百ものパラメータを監視しています。

監視には「2つの椅子」があり、2つの異なるアプローチがあります。 1つ目は、ITの監視と測定基準です。この「ジャンルの定番」は、DevOpsがなかった時代から別の方向に広がっており、その仕事はITと強く結びついていませんでした。これは、誰もが始めようとしていることです。リソース、チャネル、サービスエラーのロードを監視します-テクノロジーに関連するすべてのもの。



2番目のアプローチは「ビジネスから」です。企業は、空きメモリグラフィックス、損失、ミリ秒には関心がありません。彼にとって重要なビジネスプロセスが機能することが重要です。ビジネスメトリックには、たとえば、新しいユーザーの登録の監視や、ログイン確認付きのSMSが含まれます。何かがうまくいっていないという疑問がビジネスに出てくると、監視はこのレベルに移行します。

プロセスを追跡する


IT監視とセキュリティの類似点を描くのは簡単です。プロセスとしては、それらは類似しているため、追跡します。いくつかのレベルがあります。

基本的な監視は、「ネットワークには何があるのか​​」という質問に答えます。これは、すべての資産（サーバー、ネットワーク機器、ジョブ）の発見とインベントリです。すべてがどのように機能するかを理解するために、「内部」にあるすべてのものとプロセスに関与するものを監視します。

ソースのタイプ。これはセキュリティの次のレベルであり、ITでは「テクノロジースタック」と呼ばれます。これが私たちの「動物園」全体です。オペレーティングシステム、データベース、言語、追加のエージェント、リポジトリ、および新しいプロジェクトの開発のためのすべての入力です。

システム環境設定。安全上、このレベルは少し高いです-オペレーティングシステム構成標準。私たちは、OSレベルで世界に開かれているデフォルトのポートを探し、システムパラメータ、メモリ、デフォルトのパスワード、および安全でない可能性のある設定を調査します。

アプリケーション設定、データベース。セキュリティでは、これらはアプリケーションとサービスのセキュリティポリシーです。すべてが内部規制に従って構成されていることを確認します。単純なパスワード、デフォルトのパスワード、およびその他の安全でない構成の使用を監視することが重要です。このレベルで、技術監視は終了します。

次にビジネスレイヤー、つまりデータストリームが登場します。個人的な経験から、多くは彼らが彼らに適用されないと信じています。しかし、何かが機能しない、または危険にさらされている場合は、すべてがセットアップされた理由と方法を理解する必要があります。

QIWIで働いていたとき、処理のさまざまな部分を開発するチームリーダーを集めました。毎週、チームリーダーが気に入らないデータフロー図を作成しました。しかし、この会議により、インシデントへの対応にかかる時間を大幅に削減でき、当直シフトのスペシャリストが、私たちにとってどのように、どのように機能し、どこに問題があり、どのように正しく診断できるかを理解できました。しかし、これにより、通常モードで何が相互作用し、何が異常なフローと見なすことができるかをセキュリティに理解させることができました。

行動分析。これらはユーザーインタラクションのレベルでの指標であり、めったに使用されません。例：セキュリティの場合、これはUEBA-User EndPoint Behavior Analysisであり、ITの場合、サイトのヒートマップを構成する同じGoogleタグマネージャーです。

データストリームと行動分析は、ビジネスにとって重要です。これは、人々が深い技術知識なしに話す言語です。

脆弱性管理


脆弱性は成熟した企業に特に関係があります。通常、レガシーが表示されます-取り除くのが難しい「呼吸して更新できない非常に重要なシステム」。重要度に関しては、レガシーはデフォルトのログインとパスワードまたは安全でないデータベースと同等です。この「歴史的遺産」が十分にない場合は、定期的な脆弱性管理を開始するときがきました。

パッチの確認。手動または自動でパッチを適用する対象を調査することから、管理を開始します。

手動：私たちはメールリスト、セキュリティメーリングを購読しています。自動化にはセキュリティスキャナーを使用しています。多くの選択肢があります：Openvas、Nexpose、Nessus、MBSA、有料と無料の両方。有料アップデートは速く、無料は遅くなります。パッケージのバージョンを手動でコンパイルして更新を監視することも可能ですが、大規模なインフラストラクチャの場合、これはうまく機能しません。

テストグループを作成します。パッチを適用する対象のリストを作成したら、他のソフトウェアや変更と同様に、何も壊れないことを確認します。これを行うには、すべてを更新するテストグループを選択します。何も壊れず、サービスが機能し、パフォーマンスが低下していないことを確認し、次に進みます。

私たちは行くだけでなく、生産中、「すべてが分離されている」と「これはテスト環境です。」機密情報が消去された、データのみをテストするテスト環境の人はほとんどいません。本番環境が保護されており、テスト環境が全世界から（VPNを介してであっても）アクセス可能であり、そこにユーザーデータがある場合、これがリークの最初のポイントになります。

通常、テスト環境の監視ははるかに少なく（監視、ログ）、すべてにパッチを適用する必要があります。テストと本番環境は、パッケージのバージョンと外部接続に従って互いに一致する必要があります。もう一度、インベントリに戻ります。

監視 -変更がすべての場所に適用されたことを確認します。パッケージの最新バージョン、構成のバージョンを確認します。再起動する必要があるものを再起動します。

管理者権限の使用の制御


「特権」ユーザーのリストを作成します。特定のアクションのためだけに定期的にパスワードを変更したり、sudoを構成したりすることは、多くの人にとって不便です。実際には多くの例があります。たとえば、誰かが開発をプロダクションに導き、開発者やOpsエンジニアを常にそこに連れて行くことができます。同時に、パスワードは変更されず、結果は誰にとっても明白です。

これを回避するために、「誰ができるのか」という質問をします。問題は、生産システムだけでなく、カスタムシステムについてもです。「以前はプログラムを配置できたが、今はできない」など、実装には多くの問題と質問がありますが、これは重要です。
権限の統合、冗長アクセスの削除、不要な組み込みグループおよびユーザーは、資格情報の漏洩に関連する問題を最小限に抑えるのに役立ちます。
すべてのユーザーを識別します。あなたはそれが誰であるかという質問に対する答えを常に知っているべきです。これを行うには、システムアカウント（サービスとユーザー）のインベントリを取得します。特に、それらが標準に従って構成されていない場合は、仮名またはニックネームを使用します。 Vasya1992が誰であるかを明確に理解する必要があります。

私たちは定期的にユーザーと権利、システムユーザーの一覧を作成し、パスワードを変更します。パスワードを変更しないと、ある日、パスワードを知らないはずの誰かに送られます。 3か月に1回の定期的な間隔で変更するのが理想的です。

パスワードの複雑さの要件。少しの「キャプテンエビデンス」-「1111」の形式または辞書の文字で構成されるパスワードは、単純なパスワードの列挙によってすばやく選択されます。したがって、パスワードの複雑さを設定します：10文字以上の数字、特殊文字、大文字と小文字。このようなハッシュの列挙には長い時間がかかります（ただし、他の方法での傍受の可能性はキャンセルされません）。 3か月ごとにパスワードを変更すると、検索時までにパスワードが異なります。

私たちは責任者を見つけます。「だれがこれをしたのか」という質問に対する答えを常に知っている必要があります。誰かが何かを削除したとき、誰もが事件を起こしました。しかし、より厳しい事件の場合には、犯人を知ることをお勧めします。証拠書類を整理する-ログを別の場所に置き、アクセスを強化します。システムが危険にさらされてログが失われたり削除されたりすると、常に2倍になります。

ロギング、モニタリング、ログ分析


何時間も話すことができる私のお気に入りのトピック：監視、ログ、ログの読み取り。すべてを誓い、何も見逃さないようにしましょう。しかし、それはそれほど単純ではありません。

入力から始めます（上記を参照）。多くの例で、私は2つ使用します。アプリケーションについては、Open Web Application Security Projectガイド。アプリケーションログに書き込む内容（入力、出力、ユーザー、権限の昇格）について詳しく説明します。セキュリティのために、監視する必要があるイベントのタイプを示すチェックリスト。

ログの記録方法と保存場所。ログが多すぎる場合があります。大規模なシステムでログを記録すると、非常に多くのログが作成されるため、すべてのイベントに反応しても意味がありません。ログに記録するものを選択する前に、次の点に注意してください。

  • Syslogを使用する場合のメッセージの長さ。
  • 独自のアプリケーションを使用してログを転送する場合は、OWASPチェックリストを参照してください。

ログを少なくとも3か月間保持します。たとえば、支払いシステムの標準では、1年以上保管されます。インシデントがなくても、半年前に問題が始まったことを振り返ると、「監視しなかった」という監視だけが役立つ場合があります。

Kpi


Timlidsとマネージャーは、安全インジケーターを評価システムに統合する方法に関心があります。パフォーマンス指標を導入する際の一般的なルールは、「評価は高品質でなければなりません！」です。



不十分なKPI：

  • インシデントの数。これは悪いKPIであり、インシデントによる損害の見積もりを提供しません。必要に応じて、1週間に20件のインシデントがあると言えますが、企業はこれが何を意味するのか理解していません。
  • データソースの数。これはKPIではありません-これはインベントリです。
  • 脆弱性の数。私たちは2万の脆弱性を抱えています。このKPIは重要度の理解を提供しません。

品質KPI。

SLAまたはダウンタイムのコスト。 SLA-顧客向け、OLA-サービス品質に関する内部契約。 SLAはさまざまなバリエーションで検討され、ダウンタイムのコストは財務部門またはビジネスユニットが負担します。彼らは、2分間の簡単な登録、または30分の取得に失敗した場合の費用に答えます。

あなたのSLAを答えに当てれば、あなたはあなたが愛され、世話され、そして大事にする理解できるビジネス指標を得ます。同時に、「この未成年者を本番で発行した」影響力の価格はあなたには明らかです。

基準に準拠していない割合。 2台のサーバーがあり、そのうちの1台が保護されていない場合、非準拠の割合は50です。非常にまれに、割合が2未満になるため、5％はすでに良好で、2-3％は優れています。

インシデントと逸脱の定性的なプロセスチェックと定性分析（RCA-根本原因分析）。定期的に費やす-セキュリティだけでなくITにも役立ちます。インシデントの根本をチェックして明確にすることで、サービスのレベルが上がり、同僚が定期的に報告を行って作業していることに対する意識が高まります。

これは、すべてのインシデントに適用されます。多くの人がRCAを採点しますが、内部の定期的なプロセスの結果に基づいて重大な外部インシデントなしでそれを実施することで、システム上の問題を排除できます。それらはますます発生しません。
重要なルール：KPIは技術専門家だけでなく理解されるべきです。

安全は文化です


私の友人は、データリークについて@dataleak電報チャネルをリードしています。彼は定期的に恥ずべき事件を公開しています。ReStoreストアの保護されていないデータベースからの電子メールと電話のリスト、MegaFonコールセンターの記録、Intelのログインとパスワードで、オープンリポジトリに保存されています。



リークはITとはまったく関係がない場合があります。たとえば、ドキュメントをゴミ箱に捨てるか、ExcelドライブをGoogleドライブに制限なく配置した場合などです。



理想的には、他人の過ちから学び、事前にセキュリティに従事する必要があります。しかし、セキュリティを課すことはできません。データやビジネスを失うまで、セキュリティを確保することは不可能です。人々は、まだバックアップを作成していない人と、すでにバックアップを作成している人に分けられます。セキュリティについても同様です。企業は、まだ破られていない企業と、すでに破られている企業に分かれています。

しかし、これは一人の警備員の仕事ではありません-誰もが同じようにする方法を言って、チェックのために走る必要があります。優れた専門家はほとんどいませんが、さまざまな規模の企業が数多くあります。
セキュリティは皆の仕事です。それは文化ではなく、機能です。
