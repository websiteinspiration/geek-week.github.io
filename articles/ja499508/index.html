Laravelキューに対応できるように製品を準備する

特に「Framework Laravel」コースの学生向けに記事の翻訳が用意されました。



こんにちは、イタリアのソフトウェアエンジニア、バレリオです。

このガイドは、実際のユーザーとのオンラインアプリケーションをすでに持っているが、Laravelキューを使用してシステムにスケーラビリティを実装する（または大幅に改善する）方法を深く理解していないすべてのPHP開発者を対象としています。フレームワークの第5バージョンの開始時に2013年の終わりにLaravelについて最初に学びました。その後、私はまだ真剣なプロジェクトに関与している開発者ではありませんでした。特にLaravelで、最も理解しにくいと思われた最新のフレームワークの側面の1つはキューです。

ドキュメンテーションを読んで、私は可能性について推測しましたが、実際の開発経験がなければ、これらすべては私の頭の中の理論のレベルでのみ残っていました。
今日、私はクリエーターですInspector.dev- 毎時間数千のタスクを実行するリアルタイムダッシュボード、そして私はこのアーキテクチャを以前よりもずっとよく理解しています。



この記事では、サーバーリソースを節約しながら、キューとタスクを発見した方法と、大量のデータをリアルタイムで処理するのに役立つ構成について説明します。

前書き


PHPアプリケーションが着信httpリクエストを受信すると、リクエストが完了してクライアント（ユーザーのブラウザーなど）に応答が返されるまで、コードが段階的に実行されます。この同期動作は、真に直感的で予測可能であり、理解しやすいものです。私はエンドポイントにhttpリクエストを送信し、アプリケーションはデータベースからデータを抽出し、それを適切な形式に変換し、いくつかの追加タスクを実行してそれらを送り返します。すべてが直線的に発生します。

タスクとキューは、このラインフローに違反する非同期動作を導入します。そのため、最初はこれらの機能が少し奇妙に見えました。
ただし、着信HTTP要求によって、すべてのチームメンバーに電子メール通知を送信するなど、時間のかかるタスクのサイクルがトリガーされる場合があります。これは、6〜10通の電子メールを送信することを意味し、4〜5秒かかる場合があります。したがって、ユーザーが対応するボタンをクリックするたびに、アプリケーションを使用し続けるには5秒待つ必要があります。

アプリケーションが大きくなるほど、この問題は悪化します。

タスクとは何ですか？


ジョブは、非同期で実行するロジックを含むハンドルメソッドを実装するクラスです。

<?php

use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Foundation\Bus\Dispatchable;
use Illuminate\Queue\InteractsWithQueue;
use Illuminate\Bus\Queueable;


class CallExternalAPI implements ShouldQueue
{
    use Dispatchable,
        InteractsWithQueue,
        Queueable;
        
    /**
     * @var string
     */
    protected $url;

    /**
     *    .
     *
     * @param array $Data
     */
    public function __construct($url)
    {
        $this->url = $url;
    }
    

    /**
     *  ,   .
     *
     * @return void
     * @throws \Throwable
     */
    public function handle()
    {
        file_get_contents($this->url);
    }
}

上記のように、Jobでコードの一部を終了する主な理由は、時間のかかるタスクを完了することであり、ユーザーが完了するのを待つ必要はありません。

「面倒な作業」とはどういう意味ですか？


これは当然の質問です。メールの送信は、キューについて説明する記事で使用される最も一般的な例ですが、私が何をする必要があるかについての実際の経験についてお話ししたいと思います。

製品の所有者として、私はユーザーの旅行情報を弊社のマーケティングおよびカスタマーサポートツールと同期させることが非常に重要です。したがって、ユーザーのアクションに基づいて、サービスおよびマーケティングの目的で、API（または外部のhttp呼び出し）を介してさまざまな外部ソフトウェアのユーザー情報を更新します。
私のアプリケーションで最も一般的に使用されるエンドポイントの1つは、完了前に10個の電子メールを送信し、外部サービスに3つのhttp呼び出しを行うことができます。ユーザーがそんなに長い時間待つことはありません-ほとんどの場合、彼らはすべて私のアプリケーションの使用をやめます。

キューのおかげで、割り当てられたクラスにこれらのすべてのタスクをカプセル化し、作業の実行に必要な情報をコンストラクターに転送し、バックグラウンドで後日完了するようにスケジュールできるので、コントローラーがすぐに答えを返すことができます。

<?php

class ProjectController 
{
    public function store(Request $request)
    {
        $project = Project::create($request->all());
        
        //  NotifyMembers, TagUserActive, NotifyToProveSource 
        //   ,     
        Notification::queue(new NotifyMembers($project->owners));
        $this->dispatch(new TagUserAsActive($project->owners));
        $this->dispatch(new NotifyToProveSource($project->owners));
        
        return $project;
    }
}

答えを返す前に、これらすべてのプロセスの完了を待つ必要はありません。反対に、待機はそれらをキューに公開するのに必要な時間と等しくなります。そして、それは10秒と10ミリ秒の違いを意味します!!!

これらのタスクをキューに送信した後、誰がこれらのタスクを実行しますか？


これは、古典的なパブリッシャー/コンシューマーアーキテクチャです。コントローラーからキューにタスクを既に公開しました。ここで、キューがどのように使用されるかを理解し、最後にタスクが実行されます。



キューを使用するには、最も人気のある職人コマンドの1つを実行する必要があります。

php artisan queue:work

ドキュメントが言うように：
Laravelには、キューに入れられたときに新しいタスクを処理するキューワーカーが含まれています。

すごい！Laravelは、タスクをキューイングするための既製のインターフェースと、キューからタスクを取得してバックグラウンドでコードを実行するためのすぐに使用できるコマンドを提供します。

スーパーバイザーの役割


それは最初の「奇妙なこと」でした。新しいものを発見するのは普通だと思います。トレーニングのこの段階の後で、私はこれらの記事を書いて自分のスキルを整理し、同時に他の開発者が知識を広げるのを助けます。

例外のスロー中にタスクが失敗した場合、チームはqueue:work作業を停止します。

プロセスqueue:workを継続的に実行する（キューを消費する）には、スーパーバイザーなどのプロセスモニターを使用してqueue:work、タスクが例外をスローしてもコマンドが停止しないようにする必要があります。スーパーバイザーは、クラッシュ後にコマンドを再起動し、再び次のタスクから、スローする例外を別にします。

タスクはサーバーのバックグラウンドで実行され、HTTPリクエストに依存しなくなります。これにより、タスクコードを実装するときに考慮しなければならないいくつかの変更が導入されます。

以下は、私が注意を払う最も重要なものです。

タスクコードが機能していないことを確認するにはどうすればよいですか？


バックグラウンドで実行している場合、タスクがエラーの原因であることにすぐに気付くことはできません。
ブラウザーからhttp要求を行うときなど、即時のフィードバックはありません。タスクが失敗した場合、それは黙ってそれを行い、誰も気付かないでしょう。インスペクタ

などのリアルタイム監視ツールを統合して、すべての欠陥を明らかにすることを検討してください。

あなたはhttpリクエストを持っていません


HTTPリクエストはもうありません。コードはcliから実行されます。

スケーリング方法を理解する


残念ながら、多くの場合、これでは不十分です。1つの回線と消費者を使用すると、すぐに役に立たなくなる可能性があります。

キューはFIFOバッファです（「先入れ先出し」）。多くのタスクを計画している場合、おそらくタイプが異なる場合でも、完了する前に他のユーザーがタスクを完了するのを待つ必要があります。

スケーリングには2つの方法があります：

キューごとに複数のコンシューマーな



ので、一度に5つのタスクがキューから削除され、キューの処理が高速化されます。

特別な目的のキュー

各タイプのタスク用に特定のキューを作成して、各キューの専用コンシューマで起動することもできます。



したがって、各キューは、他のタイプのタスクが完了するのを待たずに、独立して処理されます。

地平線


Laravel Horizo​​nは、設定するキューの数、およびコンシューマーを整理する機能を完全に制御できるキューマネージャーであり、開発者はこれら2つの戦略を組み合わせて、スケーラビリティのニーズに適したものを実装できます。

起動は、php artisan queue：workではなく、php artisan horizo​​nを介して行われます。このコマンドは、構成ファイルhorizon.phpをスキャンし、構成に応じて一連のキューワーカーを開始します。

<?php

'production' => [
    'supervisor-1' => [
        'connection' => "redis",
        'queue' => ['adveritisement', 'logs', 'phones'],
        'processes' => 9,
        'tries' => 3,
        'balance' => 'simple', //   simple, auto  null

    ]
]

上記の例では、Horizo​​nは3つのキューを開始し、各キューを処理するために3つのプロセスが割り当てられます。Laravelのドキュメントで述べたように、Horizo​​nのコード駆動型のアプローチでは、私のチームが共同作業できるバージョン管理システムに構成を残すことができます。これは、CIツールを使用する完璧なソリューションです。

構成パラメーターの意味を詳しく調べるには、この素晴らしい記事をお読みください。

私自身の構成



<?php

'production' => [
    'supervisor-1' => [
        'connection' => 'redis',
        'queue' => ['default', 'ingest', 'notifications'],
        'balance' => 'auto',
        'processes' => 15,
        'tries' => 3,
    ],
]

インスペクタは主に3つのキューを使用します。

  • 外部アプリケーションからのデータを分析するためのプロセスを取り込みます。
  • 通知は、データの受信中にエラーが検出された場合に通知をすぐにスケジュールするために使用されます。
  • デフォルトは、取り込みおよび通知プロセスと混合したくない他のタスクに使用されます。

balance=autoHorizo​​nを 使用して、アクティブ化されたプロセスの最大数は15であり、キューのロードに応じて動的に分散されることを理解しています。

キューが空の場合、Horizo​​nはキューごとに1つのアクティブプロセスをサポートします。これにより、タスクがスケジュールされている場合、コンシューマーはすぐにキューを処理できます。
