PostgreSQLアンチパターン：Sisyphus JOIN配列

時々、SQLクエリにパラメーターとして渡された線形配列からの「列内」にある同じデータで、積分サンプルを「接着」するタスクが発生します。

それは時々どのように行われますか：

WITH T1 AS (
  SELECT
    row_number() OVER() rn
  , unnest v1
  FROM
    unnest('{1,2,3,4}'::integer[])
)
, T2 AS (
  SELECT
    row_number() OVER() rn
  , unnest v2
  FROM
    unnest('{5,6}'::integer[])
)
SELECT
  T1.v1
, T2.v2
FROM
  T1
LEFT JOIN
  T2
    USING(rn);

v1 | v2
-------
 1 |  5
 2 |  6
 3 |
 4 |

つまり、最初に各配列がサンプルに「拡張」されて番号が付けられ、次にこの番号がCTE接続キーとして使用されました...


[explain.tensor.ruを見てください]

普通に


時間の4分の1以上が2、3のWindowAggに費やされました。

しかし、9.4以上のバージョンのPGを使用する場合、WITH ORDINALITYを使用して、unnestを含む任意のSRFの結果に番号を付けることができます。

WITH T1 AS (
  SELECT
    *
  FROM
    unnest('{1,2,3,4}'::integer[])
    WITH ORDINALITY T(v1, rn)
)
, T2 AS (
  SELECT
    *
  FROM
    unnest('{5,6}'::integer[])
    WITH ORDINALITY T(v2, rn)
)
SELECT
  T1.v1
, T2.v2
FROM
  T1
LEFT JOIN
  T2
    USING(rn);

[explain.tensor.ruを見てください]。

したがって、ウィンドウ関数の使用は完全に取り除かれました。

複数引数UNNEST


しかし、効率の観点から見ると、これまでのところすべてが良好であるとは限りません。ほぼ半分の時間がHash Left Joinに費やされました。

そして、著者は最初の配列が正確に長いという仮定から明確に進みました。それが彼がLEFT JOINを使用した理由です。しかし、この仮定は常に正しいとは限らず、問題を引き起こす可能性があります。

それを回避するために、同じバージョン9.4から登場した複数の配列に対して同時にunnestを使用します。

SELECT
  *
FROM
  unnest(
    '{1,2,3,4}'::integer[]
  , '{5,6}'::integer[]
  ) T(v1, v2);

その結果、要求と計画にはほとんど何も残っていませんでした。

Function Scan on t  (cost=0.01..1.00 rows=100 width=8) (actual time=0.006..0.007 rows=4 loops=1)

したがって、ミスをする可能性ははるかに少なくなります。そして実行時間は数回改善されました-そしてより長い配列ではその効果はさらに顕著になります。
