ストリーミング処理の主な欠点：
  • 開発とサポートの複雑さ。まず、データのテスト、管理、および取得は、バッチと比較すると少し難しいです。2番目の難しさ（実際、これは最も基本的な問題です）はロールバックに関連しています。ジョブが機能せず、失敗があった場合、すべてが壊れた瞬間を正確に把握することは非常に困難です。また、問題を解決するには、バッチ処理よりも多くの労力とリソースが必要になります。

  2. ストリーミングソースはたくさんありますか？

小売業のための例1.株価分析を：
  • 商品の表示はリアルタイムで変化しません。
  • ほとんどの場合、データはバッチモードで配信されます。
  • 情報の損失は重要です。

この例では、バッチを使用することをお勧めします。

例2. Webポータルの分析：

  • 分析速度は問題への応答時間を決定します。
  • データはリアルタイムで届きます。
  • ユーザーアクティビティに関する少量の情報の損失は許容されます。

分析が、Webポータルへの訪問者があなたの製品をどのように使用しているかを反映していると想像してください。たとえば、新しいリリースを公開し、すべてのカスタム機能が壊れている場合、すべてが正常に機能しているかどうかを10〜30分以内に理解する必要があります。「注文」ボタンからのテキストがなくなったとしましょう。分析により、注文数の急激な減少に迅速に対応でき、ロールバックする必要があることをすぐに理解できます。

したがって、2番目の例では、ストリームを使用する方が適切です。

SPOD要素


データ処理エンジニアは、まさにこのデータをキャプチャ、移動、配信、変換、保存します（そうです、データストレージもアクティブなプロセスです！）。
したがって、ストリーミングデータ処理システム（SPOD）を構築するには、次の要素が必要です。

  1. データローダー（ストレージにデータを配信する手段）;
  2. データ交換バス（必ずしも必要ではありませんが、リアルタイムでデータを交換するシステムが必要なため、ストリーミングする方法はありません）;
  3. データストレージ（それがない場合と同様）;
  4. ETLエンジン（さまざまなフィルタリング、並べ替え、その他の操作を行うために必要）。
  5. BI（結果を表示するため）;
  6. オーケストレーター（プロセス全体をリンクし、多段階のデータ処理を整理します）。

今回のケースでは、最も単純な状況を考慮し、最初の3つの要素のみに焦点を当てます。

Apache Kafkaが最良のオプションですが、社内にRabbitMQまたはNATSがあり、分析を少し追加する必要がある場合、最初からKafkaをデプロイしてもあまり利益がありません。

それ以外の場合はすべて、Kafkaが最適です。実際、これは水平スケーリングと大きな帯域幅を備えたメッセージブローカーです。データを操作するためのツールのエコシステム全体に完全に統合されており、高負荷に耐えることができます。それは普遍的なインターフェースを持ち、私たちのデータ処理の循環システムです。

内部では、Kafkaはトピックに分割されています。同じスキームのメッセージ、または少なくとも同じ目的のメッセージからの特定の個別のデータストリームです。

次のニュアンスについて説明するには、データソースが多少異なる場合があることを覚えておく必要があります。データ形式は非常に重要です。



Apache Avroのデータシリアル化形式については、特筆に値します。システムはJSONを使用して、コンパクトなバイナリ形式にシリアル化されるデータ構造（スキーマ）を決定します。したがって、大量のデータが保存され、シリアライゼーション/デシリアライゼーションが安価になります。

すべてが問題ないように見えますが、異なるシステム間でファイルを交換する必要があるため、回路を備えた個別のファイルが存在すると問題が発生します。簡単なように思えますが、別の部署で作業すると、相手が何かを変えて落ち着くことができ、すべてが壊れてしまいます。

これらのファイルをすべてフラッシュドライブ、フロッピーディスク、岩絵に転送しないようにするために、スキーマレジストリという特別なサービスがあります。これは、Kafkaに対して書き込みと読み取りを行うサービス間でavro-schemesを同期するためのサービスです。

さて、何も残っていません：

1. まず、ClickHouseにKafkaに「署名」し、そこにデータが必要な形式を「彼に説明」します。

CREATE TABLE IF NOT EXISTS clickstream_topic (
    firstInSession UInt8,
    timestamp UInt64,
    location String,
    partyId String,
    sessionId String,
    pageViewId String,
    eventType String,
    userAgentString String
) ENGINE = Kafka
SETTINGS
    kafka_broker_list = 'kafka:9092',
    kafka_topic_list = 'clickstream',
    kafka_group_name = 'clickhouse',
    kafka_format = 'JSONEachRow';

2. 次に、最終的なデータを配置する実際のテーブルを作成します。

CREATE TABLE clickstream (
    firstInSession UInt8,
    timestamp UInt64,
    location String,
    partyId String,
    sessionId String,
    pageViewId String,
    eventType String,
    userAgentString String
  ) ENGINE = ReplacingMergeTree()
ORDER BY (timestamp, pageViewId);

3. 次に、これら2つのテーブル間のリンクを提供します。

CREATE MATERIALIZED VIEW clickstream_consumer TO clickstream
    AS SELECT * FROM clickstream_topic;

4. 次に、必要なフィールドを選択します。

SELECT * FROM clickstream;

その結果、ターゲットテーブルからの選択により、必要な結果が得られます。
