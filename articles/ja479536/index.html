HAProxyを使用したZimbra Open-Source Editionの負荷分散

大規模なZimbra OSEインフラストラクチャを構築する際の主なタスクの1つは、適切なロードバランシングです。サービスのフォールトトレランスを向上させるという事実に加えて、ロードバランシングなしでは、すべてのユーザーに同じサービスの応答性を提供することは不可能です。この問題を解決するために、サーバー間でリクエストを再分配するソフトウェアおよびハードウェアソリューションであるロードバランサーが使用されます。それらの中には、RoundRobinのようなかなり原始的なものもあり、次の各リクエストをリスト内の次のサーバーに送信するだけです。たとえば、HAProxyなどのより高度なものがあります。 HAProxyとZimbra OSEロードバランサーがどのように連携するかを見てみましょう。



したがって、タスクの条件に応じて、Zimbra OSEインフラストラクチャが与えられました。このインフラストラクチャには、2つのZimbraプロキシ、2つのLDAPサーバーとLDAPレプリカサーバー、それぞれに1000個のメールボックスを持つ4つのメールストア、および3つのMTAがあります。メールサーバーを扱っていることを考えると、バランスを取る必要がある3種類のトラフィックを受信します。WebクライアントをダウンロードするためのHTTPと、電子メールを送信するためのPOPとSMTPです。同時に、HTTPトラフィックはIPアドレス192.168.0.57および192.168.0.58のZimbraプロキシサーバーに送信され、SMTPトラフィックはIPアドレス192.168.0.77および192.168.0.78のMTAサーバーに送信されます。

すでに述べたように、サーバー間でリクエストを均等に分散するために、Ubuntu 18.04を実行しているZimbraインフラストラクチャの入力ノードで動作するHAProxyロードバランサーを使用します。このオペレーティングシステムへのhaproxyのインストールは、sudo apt-get install haproxyコマンドを使用して行われます。その後、/ etc / default / haproxyファイルで、ENABLED = 0パラメーターをENABLED = 1に変更します。ここで、haproxyが機能していることを確認するには、service haproxyコマンドを入力します。このサービスが機能する場合は、コマンドの出力から明確になります。

HAProxyの主な欠点の1つは、デフォルトでは、接続されたクライアントのIPアドレスを送信せず、独自のIPアドレスに置き換えることです。これにより、攻撃者が送信した電子メールをIPアドレスで識別してブラックリストに追加できない場合があります。ただし、この問題は解決できます。これを行うには、Postfixがインストールされているサーバーでファイル/opt/zimbra/common/conf/master.cf.inを編集し、次の行を追加します。

26      inet  n       -       n       -       1       postscreen
        -o postscreen_upstream_proxy_protocol=haproxy
 
466    inet  n       -       n       -       -       smtpd
%%uncomment SERVICE:opendkim%%  -o content_filter=scan:[%%zimbraLocalBindAddress%%]:10030
        -o smtpd_tls_wrappermode=yes
        -o smtpd_sasl_auth_enable=yes
        -o smtpd_client_restrictions=
        -o smtpd_data_restrictions=
        -o smtpd_helo_restrictions=
        -o smtpd_recipient_restrictions=
        -o smtpd_relay_restrictions=permit_sasl_authenticated,reject
        -o syslog_name=postfix/smtps
        -o milter_macro_daemon_name=ORIGINATING
        -o smtpd_upstream_proxy_protocol=haproxy
%%uncomment LOCAL:postjournal_enabled%% -o smtpd_proxy_filter=[%%zimbraLocalBindAddress%%]:10027
%%uncomment LOCAL:postjournal_enabled%% -o smtpd_proxy_options=speed_adjust
 
588 inet n      -       n       -       -       smtpd
%%uncomment SERVICE:opendkim%%  -o content_filter=scan:[%%zimbraLocalBindAddress%%]:10030
        -o smtpd_etrn_restrictions=reject
        -o smtpd_sasl_auth_enable=%%zimbraMtaSaslAuthEnable%%
        -o smtpd_tls_security_level=%%zimbraMtaTlsSecurityLevel%%
        -o smtpd_client_restrictions=permit_sasl_authenticated,reject
        -o smtpd_data_restrictions=
        -o smtpd_helo_restrictions=
        -o smtpd_recipient_restrictions=
        -o smtpd_relay_restrictions=permit_sasl_authenticated,reject
        -o syslog_name=postfix/submission
        -o milter_macro_daemon_name=ORIGINATING
        -o smtpd_upstream_proxy_protocol=haproxy
%%uncomment LOCAL:postjournal_enabled%% -o smtpd_proxy_filter=[%%zimbraLocalBindAddress%%]:10027
%%uncomment LOCAL:postjournal_enabled%% -o smtpd_proxy_options=speed_adjust

このため、HAProxyからの着信トラフィックを受信するポート26、466、および588を開きます。ファイルを保存した後、zmmtactl restartコマンドを使用して、すべてのサーバーでPostfixを再起動する必要があります。

その後、HAProxyをセットアップします。これを行うには、まずcp /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg.bakの設定でファイルのバックアップコピーを作成します。次に、ソースファイル/etc/haproxy/haproxy.cfgをテキストエディターで開き、必要な設定を徐々に追加し始めます。最初のブロックは、ログを削除するサーバーの追加、同時接続の最大許容数の設定、実行可能プロセスが属するユーザーの名前とグループの指定です。

global
    user daemon
    group daemon
    daemon
    log 127.0.0.1 daemon
    maxconn 5000
    chroot /var/lib/haproxy

5000の同時接続の図が理由で現れました。インフラストラクチャには4000個のメールボックスがあるため、それらすべてが同時に仕事用メールに移動する可能性を考慮する必要があります。さらに、その数が増加した場合に備えて、小さなマージンを残す必要があります。

次に、デフォルト設定でブロックを追加します。

defaults
        timeout client 1m
        log global
        mode tcp
        timeout server 1m
        timeout connect 5s

このブロックでは、クライアントとサーバーの最大待機時間は、接続が期限切れになったときに接続を切断するように設定されており、HAProxyモードも設定されています。私たちの場合、ロードバランサーはTCPモードで動作します。つまり、ロードバランサーは、内容を分析せずに単にTCPパケットを送信します。

次に、異なるポートでの接続のルールを追加します。たとえば、ポート25がSMTP接続とメール転送に使用されている場合、インフラストラクチャで使用可能なMTAにポート25への接続をリダイレクトすることは理にかなっています。接続がポート80にある場合、これはZimbraプロキシに転送する必要があるHTTPリクエストです。

ポート25のルール：

frontend smtp-25
bind *:27
default_backend backend-smtp-25
 
backend backend-smtp-25
server mta1 192.168.0.77:26 send-proxy
server mta2 192.168.0.78:26 send-proxy

ポート465のルール：

frontend smtp-465
bind *:467
default_backend backend-smtp-465

backend backend-smtp-465
server mta1 192.168.0.77:466 send-proxy
server mta2 192.168.0.78:466 send-proxy

ポート587のルール：

frontend smtp-587
bind *:589
default_backend backend-smtp-587
 
backend backend-smtp-587
server mail1 192.168.0.77:588 send-proxy
server mail2 192.168.0.78:588 send-proxy

ポート80のルール：

frontend http-80
bind    *:80
default_backend http-80
 
backend http-80
mode tcp
server zproxy1 192.168.0.57:80 check
server zproxy2 192.168.0.58:80 check

ポート443のルール：

frontend https
bind  *:443
default_backend https-443
 
backend https-443
mode tcp
server zproxy1 192.168.0.57:80 check
server zproxy2 192.168.0.58:80 check

TCPパケットをMTAに送信するためのルールでは、send-proxyパラメータがアドレスの横にあることに注意してください。これは、前にPostfix設定に加えた変更に従って、送信者の元のIPアドレスがTCPパケットと共に送信されるようにするために必要です。

HAProxyに必要な変更がすべて行われたので、service haproxy restartコマンドを使用してサービスを再起動し、使用を開始できます。
