CSSマージンについて知っておくべきことすべて

CSSを学んだときに私たちの多くが最初に学んだことの1つは、「CSSブロックモデル」と呼ばれる、CSSのブロックのさまざまなコンポーネントの特殊性でした。ブロックモデルの要素の1つはマージン（外側のマージン）です。これは、このブロックのコンテンツから他の要素をはじく、ブロックの周りの透明な領域です。プロパティmargin-top、margin-right、margin-bottomとmargin-left彼らは一緒にすべての4つのプロパティの同時インストールのため減少したマージンプロパティで、CSS1のさえに記載されています。

マージンは非常に単純なように見えますが、この記事では、マージンを使用する際に人々が遭遇するいくつかのポイントについて説明します。特に、マージンが互いにどのように相互作用するか、および外側のマージンの崩壊が実際にどのように機能するか。

CSS2.1仕様には、ブロックモデルを示すための図があり、さまざまなブロックを説明するために引き続き使用する用語も定義されています。この仕様では、コンテンツボックス、パディングボックス、ボーダーボックス、マージンボックスについて説明します。これらはそれぞれ、コンテンツ、パディング、ボーダー、マージンの境界によって定義されます。

マージンを折りたたむ


CSS1仕様では、マージンだけでなく、折りたたみも定義されています。この行動は多くの失望の原因となっています。当時、折りたたみマージンは、CSSがテキストドキュメントのマークアップ言語として使用されていたことを考えると理にかなっています。マージンの折りたたみとは、マージンが低い見出しの後にマージンが大きい段落が続く場合、これらの2つのインデントが加算されず、要素間に大きなギャップが生じることを意味します。

2つのマージンが崩れると、要素間のスペースは2つのインデントの大きい方に等しくなります。

次の状況では、小さいインデントは実際には大きいマージンの折りたたみの内側で終了します。

  • 隣接する兄弟（単一の親）
  • 空のブロック
  • 親と最初/最後の子

隣接する看護要素


説明は、隣接する兄弟間でマージンがどのように崩れるかを示すデモから始めます。以下に記載されている場合を除いて、通常のストリームに2つの要素が次々と表示される場合、最初の要素の下マージンは後続の要素の上マージンとともに折りたたまれます。

以下のCodePenの例では、3つのdiv 要素があります。最初の要素の場合、上部と下部のマージンは50px、2番目の要素は-20px、3番目の要素は-3emです。最初の2つの要素間のマージンは50pxです。これは、下の要素の下のマージンが上の要素の上のマージンによって吸収されるためです。2番目の要素の下マージンにある3emは20ピクセルより大きいため、2番目の2つの要素間のマージンは3emです。


空のブロック


ブロックが空の場合、その上部マージンと下部マージンが互いにつぶれる可能性があります。次のCodePenの例では、クラスの2番目の要素empty（空のため表示されない）の上下の余白は50pxですが、最初と3番目の要素の間のスペースは100ピクセルではなく50です。これは、2つの余白が折りたたまれた結果です。

ブロックに何かを追加すると（パディングも含む）、上下のマージンが使用され、折りたたまれなくなります。


親と最初/最後の子


このマージン崩壊シナリオは、直感的でないため、他の人よりも多くの人を困惑させます。次のCodePen divではクラスwrapper（ラッパー）を使用します。境界線が見えるように、このdivプロパティをoutline赤に設定します。3つの子すべてに50ピクセルのマージンがあります。ただし、最初と最後の要素はラッパー要素の境界に隣接しています。要素とラッパーの間に50ピクセルのインデントはありません。


これは、子のマージンが親のマージンとともに折りたたまれ、親の外側で終了するためです。これは、開発者のブラウザーパネルを使用して子をテストすると確認できます。黄色でハイライトされた領域はマージンです。



マージンのみを折りたたむ


最後の例では、マージンの縮小についても強調しています。CSS2では、垂直方向（上下）のマージンのみが縮小するように設計されていました。したがって、上記の例では、左と右のマージンは折りたたまれず、ラッパー内で終了します。

注：マージンは、段落間など、ブロックの方向にのみ崩れることを覚えておく価値があります。

崩壊防止


要素が絶対配置またはプロパティに設定されている場合、マージンは折りたたまれませんfloat。しかし、マージンが崩壊する状況にある場合、これをどのように防ぐことができますか？

間に何かが存在する状況では、折りたたみマージンは発生しません。

たとえば、ブロックがボーダーまたはパディングに設定されている場合、空のブロックの上下のマージンは折りたたまれません。以下の例では、1pxのパディングブロックを追加しました。ブロックの上部と下部に50pxのマージンがあります。


これが論理です。空のブロックに境界線もパディングもない場合、それは事実上見えなくなります。たとえば、CMSのマークアップに配置された空の段落である可能性があります。 CMSが冗長な段落要素を追加した場合、余白が考慮されるため、他の段落の間に大きなインデントを生じさせたくないでしょう。ブロックに何かを追加して、これらのインデントを取得します。

同様の動作は、マージンが親を超える最初または最後の子のマージンでも見られます。親に境界線を追加すると、子のマージンは内側のままになります。


そして再び、この振る舞いには論理があります。視覚的に表示されない意味上のラッパー要素がある場合、おそらく表示時に大きなインデントを作成したくないでしょう。これは、ほとんどの場合Webがテキストである場合に役立ちました。また、レイアウト要素を使用する場合、この動作はあまり役に立ちません。

「ブロックフォーマットコンテキスト」の作成


新しい「Block Formatting Context」（BFC）は、マージンが親要素から外れるのを防ぎます。マージンがラッピング要素の境界の外側にある最初と最後の子を使用して例をもう一度見て、ラッパーを設定しdisplay: flow-rootてBFCを作成すると、子のマージンは内側のままになります。


の詳細についてはdisplay: flow-root、私の記事「CSSレイアウトとブロックフォーマットコンテキストについて」を参照してください。プロパティ値overflowをに変更するautoと、新しいBFCも作成されるため、同じ効果が得られますが、場合によってはスクロールバーが不要になることがあります。

フレックスコンテナーとグリッドコンテナー


フレックスコンテナーとグリッドコンテナーは、子のフレックスとグリッドの書式設定コンテキストを設定するため、ブロックレイアウトの動作が異なります。これらの違いの1つは、マージンが崩れないことです。
フレックスコンテナーは、コンテンツの新しいフレックスフォーマットコンテキストを設定します。これは、ブロックマークアップの代わりにフレックスマークアップが使用されることを除いて、ブロックフォーマットコンテキストの設定と同じです。たとえば、フロートはフレックスコンテナーの内部では機能せず、フレックスコンテナーのマージンは子要素のマージンで折りたたまれません。

あなたのウェブサイトのマージン戦略


崩壊のため、良い解決策はあなたのサイトでマージンを扱う一貫した方法を提供することです。あなたができる最も簡単なことは、要素の上部または下部にのみマージンを設定することをルールにすることです。この場合、余白が設定されている辺は常に余白のない他の要素の辺に隣接するため、折りたたみの問題が頻繁に発生することはありません。

注：Harry Roberts は、折りたたみの問題を解決するためだけではなく、一方向にのみマージンを設定することが望ましい理由を詳述した優れた出版物を公開しています。

このアプローチでは、発生する可能性のある親を超えて子要素からマージンが外れるという問題は解決されません。通常、この特定の問題はあまり一般的ではなく、これが発生している理由を理解すると解決策を見つけるのに役立ちます。理想的なソリューションは、これを必要とするコンポーネントを設定することです。display: flow-root古いブラウザーのフォールバック（フォールバック）を使用overflowして、BFC（ブロックフォーマットコンテキスト）を作成できます。親をflexコンテナに変えます。またはパディングを1pxに設定します。ブラウザによるプロパティサポートのリクエストを使用できることを忘れないでください（サポート）プロパティがサポートされているかどうかを判断し、display: flow-root古いブラウザのみが最適でないソリューションを取得するようにします。

ほとんどの場合、マージンが崩壊する（または崩壊しない）理由を理解することが重要だと思います。これにより、各ケースでこれに対処する方法を決定できます。どの方法を選択する場合でも、この情報をチームと共有してください。多くの場合、マージンの崩壊は少し不思議なので、それに対処する方法は必ずしも明白ではありません。コード内のコメントが適切な場合があります。この記事へのリンクを含めて、マージンの縮小に関する知識を共有することもできます。

利ざや


CSSでパーセンテージを使用する場合、それは何かのパーセンテージでなければなりません。パーセントで設定されたマージン（およびパディング）は、常に親要素の幅を基準にして計算されます。つまり、パーセンテージを使用する場合、要素の周囲のすべての側面から常に同じマージンが得られます。

以下のCodePenの例では、幅が200pxのラッパーがあり、その中にマージン= 10％のブロックがあります。すべての側面で、マージンは20pxで、200pxの10％です。


流れに依存する世界のマージン


この記事では、垂直マージンについて説明しましたが、最新のCSSは、原則として、物理的な側面よりもストリームに対する要素の位置を考慮しています。したがって、垂直マージンについて説明する場合、実際にはブロックディメンションのマージンについて説明しています。これらのマージンは、横書きモードでは上または下、縦モードでは右または左にすることができます。

フローに関連する論理的な方向を操作すると、上部と下部よりもブロックの最初と最後について話すのが簡単になります。これを簡単にするために、ブールプロパティと値の仕様がCSSに導入されました。物理的特性を流れに関連する特性に置き換えます。

マージンについて話す場合、これは次の選択肢を提供します（テキストの方向を左から右にして英語またはその他の水平方向に記録されたモードで作業する場合）。

  • margin-top = margin-block-start
  • margin-right = margin-inline-end
  • margin-bottom = margin-block-end
  • margin-left = margin-inline-start

また、2つの新しい省略されたプロパティがあります。

  • マージンブロック
  • マージンインライン

次のCodePenの例では、ストリーム関連のキーワードを使用してから、ブロックスペルモードを変更しました。マージンは、物理的な側面に縛られるのではなく、テキストストリームにどのように従うかを確認できます。

結論として


これで、マージンについて知っておくべきことのほとんどがわかりました。簡単に：

  • マージンが崩れる場合があります。これがいつ発生し、いつ発生しないかを理解することは、彼らが作成する可能性のある問題の解決に役立ちます。
  • マージンを一方向にのみインストールすると、それらに関連する多くの問題が解決されます。
  • CSSの状況と同様に、チームで行った決定を共有し、コードにコメントします
  • ウェブが書き込みモードから独立する方向に向かっているため、物理的な側面、上、右、下、左の代わりにブロックとラインの測定を考慮に入れると役立ちます
