叙情的な紹介


私たちの組織のファイルダンプは、Windows Server 2016のVMware ESXi 6仮想マシンで回転しています。そして、これは単なるダンプではありません。これは、構造部門間のファイル共有サーバーです。ネットワークスキャナーからのコラボレーション、プロジェクトドキュメント、およびフォルダーがあります。一般に、これは全生産寿命です。

そして、すべての生産寿命のこのコンテナがハングし始めました。さらに、ゲストは残りの部分に影響を与えることなく、静かに自分を掛けることができました。ホスト全体、したがって他のすべてのゲストマシンがハングする可能性があります。ハングアップしてvSphereクライアントサービスを切断することもできます。つまり、他のゲストのプロセスは生きていて、マシンは正常に動作して応答していますが、ファイルの破損はなく、vSphereクライアントはホストに固執しません。一般に、システムは識別できませんでした。ハングは、負荷が低い日中に発生する可能性があります。無負荷時の夜間でした。夜間に差分バックアップと中程度の負荷がかかる可能性があります。週末に完全バックアップと高負荷が発生する可能性があります。そして、状況の明らかな悪化がありました。最初は年に1回、次に6か月に1回でした。我慢の終わりに、週二回。

RAMに罪を犯した。しかし、週末でもゴミを止めてMemtestを追い払うことはできませんでした。 5月の休日を待っていました。 5月の休暇にMemtestを運転してみましたが、エラーは見つかりませんでした。

私は驚いて、休暇に行くことにしました。私が休暇中-ごみ箱は1回のハングではありませんでした。そして月曜日に最初の日が仕事に行ったとき-ごみ箱がハングしました。完全バックアップを維持し、その最後にハングアップしました。休暇中のこのような温かい出会いは、ゲストドライブを別のホストに物理的にドラッグするという決断を迫られました。

そして、休暇後の最初の日は、仕事のためにずっと仕事に取り掛かったものの、深刻なことは何もできないことが長い間知られていましたが、次の凍結への憤りは私の心と気分を打ち砕き、誓いました...

物理ディスクが別のホストに再配置されました。ホット接続。ドライブは、[ ドライブ ]タブのストレージ設定に表示されます。 [ データストア ]タブでは、これらのディスク上のストレージはそうではありません。更新 -表示されません。もちろん、最初のラッシュはAdd Storageです。追加ウィザードは、それがサポートするものを教えてくれます。もちろん、VMFSもサポートしています。疑いませんでした。各ステップでのウィザードメッセージの概要：次へ、次へ、次へ、完了。彼の視線は、マスターの階段の1つのウィンドウの下部に感嘆符が付いた小さな黄色の円に近づくことさえできませんでした。

マスターの最後に、新しいデータストアがリストに表示されました。これにより、他の物理ディスクからのデータストアが表示されました。

新しく追加されたデータストアのナビゲーションに移りますが、空です。もちろん、またびっくりしました。午前8時、休暇後の最初の15分間、コーヒーの砂糖もまだ攪拌されていません。そしてここにあります。私が最初に思ったのは、「ネイティブ」ホストから間違ったドライブをプルしたことです。目的のデータストアが「ネイティブ」ホストに存在するかどうかを確認しました。いいえ、存在しません。 2番目の考えは次のとおりです。確かではありませんが、3番目、4番目、少なくとも5番目の考えは同じだったようです。

疑念を払拭するために、私はすぐに新しいESXiをサンプルにインストールし、左ドライブを取り、それを読んだ後、ウィザードの手順を実行しました。はい。ウィザードを使用してデータストアを追加すると、操作をロールバックしてデータを復元する機能がないと、ディスク上のすべてのデータが失われます。後で、私はフォーラムの1つでマスターによるそのようなデザインの評価を読んだ。そして今、私は本当に同意しました。

6番目から、思考はより建設的な方法で流れました。はい。 3Tbドライブでも初期化には数秒かかります。これは高レベルのフォーマットです。したがって、パーティションテーブルは単純に書き直されました。したがって、データはまだ残っています。それでは、いくつかのアンフォーマットと出来上がりを探しましょう。

プログラムの列挙は快適ではありません。GetDataBackとR.Saverは、せいぜい、ライブディレクトリ構造とライブファイル名を持つNTFSパーティションを見つけます。しかし、それは私には合いません。 2つのvmdkファイルが必要です。システムディスクとゴミ箱です。

そして、私はそれがWindowsをインストールしてファイルバックアップからロールアウトするようだと理解しています。同時に、私はそこにDFSルートを持っていたことを思い出します。また、ユニットのフォルダへのアクセス権のボリュームと分岐システムの面でも完全にワイルドです。オプションではありません。時間的に受け入れられる唯一のオプションは、システムとディスクの状態を、データとすべての権限で復元することです。

ディスクをスキャンすると、7つのノードが存在することがわかりました。ノードの数は、「驚くほど」VMFSリカバリによって検出された* -flat.vmdkファイルの数と一致しました。



ファイルサイズとノードサイズの比較では、バイトまで同じ一致が示されました。同時に、* -flat.vmdkファイルの名前と、それに応じて仮想マシンに属するファイルが復元されました。



一般に、ESXiの観点から見ると、vmdkディスクは2つのファイルで構成されています。データファイル（<マシン名> -flat.vmdk）と物理ディスクパーティションファイル（<マシン名> .vmdk）です。 * -flat.vmdkファイルをローカルマシンからDatastoreにアップロードすると、ESXiはそれを有効なディスクファイルとして認識しません。 VMwareナレッジベースに、ディスク記述子ファイルを手動で作成する方法に関する記事があります。kb.vmware.com / s / article / 1002511、ただしこれを行う必要はありませんでした。対応するファイルのコンテンツをファイルコンテンツのプレビュー領域からDiskInternals VMFSリカバリに
