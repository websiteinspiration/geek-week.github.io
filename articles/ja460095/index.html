gitlab.comでフォークdeepNudeの禁止をキャッチ

ハブで公開された記事の1つで、私はgitlab.comにdeepNudeのコピーがあることをコメントで読みました。純粋に好奇心からそれを見つけ、「念のため」フォークを作成しました。3時間後、コードを実行できませんでした。禁止令に飛び込んだ...



サポートとのやり取りの後、私は彼らのサービスのルールに違反し、ロックが解除されたことがわかりましたが、24時間以内にアカウントからdeepNudeを削除する必要があります。

しかし、おそらく含まれている2つの要素と誤ったロック解除が原因で、参加できませんでし



た。サポートからの次の応答を待っている間、「gitlab.comのコードを信頼できない状態にしておくと、ルールが変更される可能性があるため、すべての作業を行う必要があります。 「レジストリとCI / CDをバイパスしてデプロイすることもできません。」

しかし、実際にはgitlabが便利であり、CI / CDなしで2019年にどのように暮らすことができるのか本当にわかりません。そして、他のグッズ。一般的に、私は別のサーバーを持ち上げて、そこに転がりましたdocker / docker -compose / gitlab runner / Gitlab CE /レジストリは持ち上げませんでした（Amazon ECR）

それだけです。現在、導入はサービス管理ポリシーに依存していません。私は満足し、突然、今回は完全に封鎖解除されたという手紙が届きました。deepNudeを削除し、すべてのリポジトリ（および50を超えるリポジトリ）をワンクリックでGitlabのselfHostingバージョンに転送しました。

最終的に


プロから：

  1. ビルドとデプロイメントが2倍に加速しました。6分から3倍（製品の組み立て/テスト/導入）
  2. 管理パネルGitlab CEを通じてさまざまなプロセスを完全に制御できました
  3. サービスポリシーへの依存はありません。ここはあなた自身の上司です
  4. 主観的に、Gitlab CEはすべての面でクラウドバージョンよりもはるかに高速に動作します。

マイナスの：

  1. バックアップを作成する必要があります（DigitalOceanに組み込まれているバックアップメカニズムを使用しています）
  2. サーバー/コンテナのステータスを定期的に更新および監視する必要があります

一般的に、deepNudeが（まだそこにある場合は）クローンし、gitlabに移動する必要があります））。後悔はしません。おそらくこのケースがなかったとしたら、私がこれを行うことはなかったでしょう。

誰か興味があれば、Docker-compose構成をアップロードして、Gitlab CEおよびCI / CD構成の例を起動できます。
