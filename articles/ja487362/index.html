Angular 9がリリースされました-Ivyが到着しました

Angularのバージョン9がリリースされました。これは、フレームワーク自体、Angleマテリアル、CLIを含むプラットフォーム全体をカバーするメインリリースです。このリリースでは、アプリケーションはデフォルトでIvyコンパイラーとランタイムに切り替わり、コンポーネントをテストするための改善された方法を提供します。

これは3年間で最大のAngular更新の1つであり、開発チームは開発者がより優れたアプリケーションを作成し、Angularエコシステムに貢献できる機会に興奮しています。

アイビー


バージョン9は、デフォルトですべてのアプリケーションをIvyコンパイラーとランタイムに転送します。修正された数百のバグに加えて、Ivyコンパイラーとランタイムには多くの利点があります。

  • 小さいバンドルサイズ
  • より高速なテスト
  • 最高のデバッグ
  • 改善されたCSSクラスとスタイルバインディング
  • 型チェックの改善
  • ビルドエラーメッセージの改善
  • ビルド時間の改善、デフォルトでAOTが有効
  • 国際化の改善

いくつかの改善を詳細に検討しましょう

小さいバンドルサイズ


Ivyコンパイラーは、ツリーシェイクで使用されないAngularの一部を削除し、Angularコンポーネント用に生成されるコードの量を削減するように設計されました。

これらの改善により、アプリケーションのサイズを大幅に削減できます。

  • フレームワークの機能の多くを使用しない小さなアプリケーションは、ツリーシェイクから最も恩恵を受けることができます。
  • 多くのコンポーネントを含む大規模なアプリケーションでは、工場のサイズを小さくすることでメリットが得られます。
  • 中規模のアプリケーションは、ツリーヘイキングのメリットが少なく、多くのコンポーネントを使用せずに小さなファクトリーのメリットを大きく受けるため、ほぼ同じか、わずかに小さくなります。



より高速なテスト


Ivy TestBed実装も更新され、より効率的になりました。

以前は、TestBedは、コンポーネントに変更が加えられたかどうか（オーバーライドなど）に関係なく、各テストの実行と実行の間にすべてのコンポーネントを再コンパイルしました。
Ivyでは、コンポーネントが手動でオーバーライドされない限り、TestBedはコンポーネントを再コンパイルしません。これにより、ほとんどのテスト間の再コンパイルが回避されます。

この変更により、フレームワークのコアテストは約40％速く合格します。ユーザーは、40〜50％のレベルでアプリケーションのテスト速度が向上することに気付くでしょう。

スタイルとCSSクラスの改善。


Ivyは、スタイルを処理するための機能拡張を提供します。以前は、アプリケーションに競合するスタイル定義が含まれている場合、これらのスタイルは互いに置き換えられていました。 Ivyを使用すると、スタイルは予想どおりにマージされます。

  },...

バージョン9では、時間に依存しない明確な優先順位でスタイルを管理できます。最も具体的なスタイルが優先されます。たとえば、[style.color]へのバインディングは、交差するバインディング[style]をオーバーライドします。

型チェックの改善


Angularコンパイラは、より多くのアプリケーションタイプをチェックし、より厳しいルールを適用できます。これらの機能は、開発の初期段階でバグを特定するのに役立ちます。

標準のフラグに加えて、追加の型チェックのために2つのメインフラグがサポートされています。

  • fullTemplateTypeCheck-このフラグをオンにすると、テンプレートですべてをチェックするようコンパイラーに指示します（ngIf、ngFor、tp-templateなど）。
  • strictTemplates-このフラグをアクティブにすると、型チェックに最も厳しいルールが適用されます。

修正されたビルドエラー


新しいIvyコンパイラーはより高速であり、より高いタイプセーフティを提供するだけでなく、エラーメッセージの読み取りも容易にします。

ビルド時間が改善され、AOTコンパイラーがデフォルトで有効になります。


Ivyアーキテクチャーのおかげで、コンパイラーのパフォーマンスが大幅に向上しました。
TypeScriptアプリケーションの単純なコンパイルのオーバーヘッドに関して測定されたコンパイラパフォーマンス。ドキュメンテーションプロジェクト（angular.io）の場合、オーバーヘッドは0.8xから0に減少しました。Ivyを使用すると、xがほぼ40％向上しました。

これらは、AOTビルドが著しく高速になることを意味します。これのおかげで、開発モードでもAOTを使用できます。これは、「ng serve」がリリースビルドと同じ利点を持ち、Angularの開発エクスペリエンスを向上させることを意味します。

コンパイラとランタイムの変更により、entryComponentsも不要になりました。これらのコンポーネントは、使用時に自動的に検出およびコンパイルされます。

より信頼性の高いng更新


ng updateの動作に変更が加えられ、信頼性と情報量が向上しました。

  • 常に最新バージョンのCLIを使用してください。8.3.19 CLIから、アップデートのターゲットバージョンのCLIを使用するようになりました。今後、更新は常に最後のCLIによって処理されます。
  • 更新の進行が改善されました。ng updateは、内部で何が起こっているかについてより多くの情報を提供します。移行ごとに、それに関する情報が表示されます。
  • 更新のより簡単なデバッグ。デフォルトでは、ng updateはすべての移行を開始し、検証のために累積的な変更をディスクに残します。バージョン9のアップグレードでは、新しいフラグ--create-commitsも導入されています。ng update --create-commitsを実行すると、ツールは移行ごとにコードベースの状態をキャプチャするため、コードに送信された変更を段階的に実行してデバッグできます。

「providedIn」の新機能


Angularで@Injectableサービスを作成する場合、インジェクターに追加する場所を選択する必要があります。以前の「ルート」および「モジュール」オプションに加えて、2つの追加オプションが表示されました。

  • platform-providedIn： 'platform'表示は、ページ上のすべてのアプリケーションで共有される特別なシングルトンプラットフォームインジェクターでサービスを利用できるようにします。
  • any-各モジュール（遅延モジュールを含む）に一意のインスタンスを提供します

コンポーネントのバインド


コンポーネントのテストでは、コンポーネントを見つけてイベントをトリガーするために、CSSセレクターなどの実装の詳細の使用に依存してきました。つまり、コンポーネントライブラリが変更されるたびに、これらのコンポーネントに基づくすべてのテストを更新する必要がありました。

バージョン9では、テストの代替方法を提供するコンポーネントバインディングが導入されました。実装の詳細から抽象化することで、テストが正しく定義され、脆弱性が低下していることを確認できます。

Angular Materialのほとんどのコンポーネントは、これらのバインディングを使用してテストできるようになり、コンポジション（CDK）の作成者コンポーネントも使用できるようになりました。

これは前のテスト例です：


it("should switch to bug report template", async () => {
  expect(fixture.debugElement.query("bug-report-form")).toBeNull();
  const selectTrigger = fixture.debugElement.query(
    By.css(".mat-select-trigger")
  );
  selectTrigger.triggerEventHandler("click", {});
  fixture.detectChanges();
  await fixture.whenStable();
  const options = document.querySelectorAll(".mat-select-panel mat-option");
  options[1].click(); // Click the second option, "Bug".
  fixture.detectChanges();
  await fixture.whenStable();
  expect(fixture.debugElement.query("bug-report-form")).not.toBeNull();
});

そして今、同じこと：


it("should switch to bug report template", async () => {
  expect(fixture.debugElement.query("bug-report-form")).toBeNull();
  const select = await loader.getHarness(MatSelect);
  await select.clickOptions({ text: "Bug" });
  expect(fixture.debugElement.query("bug-report-form")).not.toBeNull();
});
