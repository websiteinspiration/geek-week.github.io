Android APIレベル、下位互換性と上位互換性

皆さんこんばんは。Androidデベロッパーの将来の学生のために役立つ翻訳を用意しました。上級コース。」この資料を共有させていただきます。



この記事を読んでいる場合、次のようなことに興味がある可能性があります。

  • APIレベルとはどういう意味ですか？
  • 使用方法compileSdkVersion、minSdkVersionまたはtargetSdkVersion？
  • OSのバージョンが異なるデバイスでアプリケーションが正しく動作することをどのように保証できますか？



これらの概念はすべて相互に関連しているので、この記事では簡単にしかし効果的に説明します。

これを行うには、違いを理解する必要がありSDKやAPIをして知っているAPIレベルである Androidのエコシステムに。

Androidでは、SDKとAPIの間に1対1の関係があることは事実であり、多くの場合、これら2つの用語は同義語として使用されますが、これは同じことではないことを理解することが重要です。

Androidのバージョンごとに、SDKと同等のAPI、およびこのAPIのレベルがあると言う方が正しいでしょう。

SDK


ソフトウェア開発キットの 略です。「キット」（kit）という単語に注意してください...開発者がAndroidアプリケーションを作成、デバッグ、実行するのに役立つ、さまざまなツール、ライブラリ、ドキュメント、サンプルのセットにすぎません。SDKにはAPIが用意されています。Android StudioでSDK Manager

を開くと、Android SDKの構成をより明確に確認できます。Platform SDK の最初のタブには、Androidの各バージョンのSDKが一覧表示されます。 下の画像に示すように、Android 9.0 SDK（Pieとも呼ばれます）には以下が含まれています。





  • Android SDK Platform 28（これはフレームワークAPIです）。
  • Android 28のソースコード（これはAPIの実装です。ご覧のとおり、これはオプションです...覚えておいてください）。
  • 他にもたくさんあります。たとえば、Androidエミュレータ用のさまざまなシステムイメージです。


Android Studio SDK ManagerのSDKの概要。SDKツール

API


これは、アプリケーションプログラミングインターフェイスの略です。これは単なるインターフェースであり、ソフトウェアの2つの異なる「部分」間のリンクを提供する抽象化レイヤーです。これは、プロバイダー（ライブラリーなど）とコンシューマー（アプリケーションなど）の間の契約のように機能します。

これは、クラス、メソッド、関数、モジュール、定数などの正式な定義のセットであり、他の開発者が独自のコードを作成するために使用できます。ただし、APIには実装が含まれていません。

APIレベル


APIレベルは、Androidプラットフォームが提供するフレームワークAPIのバージョンを一意に識別する整数値です。

通常、プラットフォームフレームワークAPIの更新は、APIの新しいバージョンが以前のバージョンとの互換性を保つように設計されているため、新しいAPIの変更のほとんどは追加的であり、APIの古い部分は廃止されますが削除されません。

そして今、誰かが疑問に思うかもしれません...

Android APIが実装を提供せず、SDK ManagerがSDKの一部としてオプションのダウンロード可能なAPIソースコードを提供する場合、対応する実装はどこにありますか？

答えは簡単です。デバイス上。

コンパイルプロセスは、JavaまたはKotlinで記述されたコードを、依存関係（コードを削減する理由の1つです！）を含めてDEXバイトコードに変換し、すべてをリソースとともにAPKファイルに圧縮します。この段階では、APIの実装は最終的なAPKに含まれていません。

compileSdkVersion


この値は、アプリケーションをコンパイルするSDKのバージョンをGradleに指示するためにのみ使用されます。これにより、開発者はに設定されたAPIレベルまで使用可能なすべてのAPIにアクセスできますcompileSdkVersion。

SDKの最新バージョンでコンパイルすることを強くお勧めします。

  • 高レベルのAPIにより、開発者は最新のAPIと新しいプラットフォームによって提供される機会を活用できます。
  • 最新バージョンを使用するには、バージョンSupportLibraryとcompileSdkVersion一致する必要がありSupportLibraryます。

たとえば、を使用するにはSupportLibrary-28.x.x、compileSdkVersion28と等しくなければなりません。

  • AndroidXに切り替えるか使用するには、compileSdkVersionを28以上に設定する必要があります。
  • Google PlayのターゲットAPIレベルの要件を満たす準備ができている。Googleは、GoogleマーケットでAndroidの新しいバージョンを迅速に配布するために、毎年、新しいアプリケーションとアップデートの最小APIターゲットレベルを設定すると発表しました。あなたはこことここでより多くの情報を見つけることができます。

通常、APIへの変更は追加的であり、古いAPIは廃止されても削除されないため、AndroidアプリケーションはAndroidプラットフォームの新しいバージョンと互換性があります。

つまり、直接的な互換性は主にプラットフォームによって保証されており、で指定されたものよりもAPIのレベルが高いデバイスでアプリケーションを実行すると、compileSdkVersion実行時に問題がなく、アプリケーションは期待どおりに動作します。プラットフォームの新しいバージョン。

例：

アプリケーション + compileSdkVersion = 26、およびAPIレベル26でxyz()導入されたAPIメソッドは、Android 8 Oreo（APIレベル26）を搭載したデバイスで機能します。

APIメソッドxyz()は引き続きAPIレベル28で使用できるため、同じアプリケーションはAndroid 9 Pie（APIレベル28）を備えたデバイスで動作できます。

minSdkVersion


この値は、アプリケーションを実行できる最小APIレベルを示します。これは最小要件です。指定しない場合、デフォルト値は1です。

開発者は、正しい値を設定し、このAPIレベルまでのアプリケーションの正しい動作を保証する必要があります。これは下位互換性と呼ばれます。

開発中は、でLint指定されたものより下のAPIを使用しようとすると、開発者にも警告が表示されminSdkVersionます。警告を無視して修正しないことが非常に重要です！下位互換性

を確保するには、開発者は実行時にプラットフォームのバージョンを確認し、プラットフォームの新しいバージョンで新しいAPIを使用し、古いバージョンで古いAPIを使用するか、場合によっては、下位互換性を提供するいくつかの静的ライブラリを使用できます。

また、Google Playストアはこの値を使用して、デバイスプラットフォームのバージョンとminSdkVersionアプリケーションを比較することにより、特定のデバイスにアプリケーションをインストールできるかどうかを判断することも重要です。

プラットフォームでは下位互換性が保証されていないため、開発者はこの値を選択するときは十分に注意する必要があります。

プロジェクトの「適切な」値を選択することも、アプリケーションの対象者の規模に影響を与えるため、ビジネス上の決定です。見るプラットフォーム配布。

例：

Application + compileSdkVersion = 26 + minSdkVersion= 22およびxyz()APIレベル26で提示されるAPI メソッドは、Android 8 Oreo（APIレベル26）を搭載したデバイスで機能します。

同じアプリケーションをAndroid 5.1 Lollipop（APIレベル22）の古いデバイスにインストールして実行できますが、APIメソッドxyz()は存在しません。開発者がランタイムチェックまたはライブラリのいずれかを通じて下位互換性を提供しなかった場合、アプリケーションはAPIメソッドにアクセスしようとするとすぐにクラッシュしますxyz()。

targetSdkVersion


この値は、アプリケーションが開発されたAPIのレベルを示します。

と混同しないでくださいcompileSdkVersion。後者はコンパイル時にのみ使用され、開発者が新しいAPIを使用できるようにします。逆に、1つ目はAPKの一部であり（minSdkVersionおよび）、ランタイムの動作を変更します。これは、開発者が直接互換性を制御できる方法です。

新しいランタイム環境で作業するときに、アプリケーションの動作に影響を与える可能性のあるAPIの変更がベースシステムにある場合があります。

ターゲットアプリケーション層には、プラットフォームバージョンに固有のランタイム動作が含まれます。アプリケーションが実行時の動作に対するこれらの変更をサポートする準備ができていない場合、失敗する可能性があります。

簡単な例は、Android 6 Marshmallow（APIレベル23）で導入されたランタイム権限です。

アプリケーションはレベル23 APIを使用してコンパイルできますが、新しいランタイムアクセス許可モデルをサポートする準備がまだ整っていない場合は、ターゲットレベル22 APIを使用できます。

したがって、アプリケーションは新しいランタイム動作を含まなくても互換性を保つことができます。

いずれの場合でも、すでに述べたように、GoogleはアプリケーションにターゲットAPIレベルの新しい要件を満たすことを要求しているため、この値を更新するには常に高い優先度を与える必要があります。

すべてをまとめると、明確な関係がわかります

minSdkVersion ≤ targetSdkVersion ≤ compileSdkVersion

最新のAPIレベルでコンパイルし、targetSdkVersion == compileSdkVersionを使用することを強くお勧めします。
