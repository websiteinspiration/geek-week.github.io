Roman Rusyaev：コンパイラー最適化のプリズムによるC ++例外


LLVMの例の講演者が最新のC ++のゼロコスト例外について話している興味深いレポート。

LLVM IRは、プログラムを制御フローグラフとして表示します。グラフのノードには、従わなければならない命令のブロックがあります。各ブロックの最後に、制御を次のブロックに移すターミネーターがあります。ターミネーターは、別のブロックへの条件付き遷移、戻り命令、または関数の呼び出しのセマンティクスを持つ特殊な呼び出し命令であり、成功した場合は、制御フローを1つのブロックに転送し、例外の場合は、処理先のブロックを示します。ブロック内の命令には、ブロックにヒットした場合にすべての命令を実行するか、このブロックにまったく陥らないという特性があります。単一のブロック内でのみ機能する最適化があります。小さなブロックの場合、それらのコンテキストは小さくなります。その結果、最適化は悪化します。
講演者は、最近のコンパイラーが、invokeステートメントを、もはや終端ではないcallステートメントに変換し、その結果、コンパイラーに最適化の余地を与える方法について話しました。ただし、コンパイラーに依存しないようにするには、noexcept関数を自分で（これが正しい場合）作成して、コンパイラーがすべての最適化を確実に実行できるようにします。

（レポートスライド）

Maxim Khizhinsky：俳優とハンドラーのための快適クラスの住宅


講演者は、並列プログラミングに関連する多くの問題を取り除くための目標を設定しました。

  • 共有データ
  • コンテキスト切り替え、
  • 同期
  • 短期的なニーズに合わせてオンザフライでフローを頻繁に作成します。

その結果、講演者はプログラムをコンポーネントの寿命に等しいコンポーネントに分割し、コンポーネントを「アパートメント」に配置することを提案しました。その数はスレッドの数と同じでなければなりません。その結果、コンポーネント自体はシングルスレッドであり、コンポーネント間の通信はメッセージパッシングを通じて行われる必要があります。これで問題が解決することに同意しますが、プログラムのすべてのカスタマイズを犠牲にして、コンパイル時に行う必要があります。少なくとも、コンパイル時に「アパートメント」間でコンポーネントを均等に分散する必要があります。これにより、負荷に応じてシステムが再調整されなくなります。結果として、私の意見では、ソリューション自体は十分に柔軟に見えません。

（レポートスライド）

Nikolay Beloborodov：高負荷ネットワークアプリケーションでのスラブアロケーターの適用


その名前はそれ自体が物語っています。講演者は、スラブアロケーターを使用してシステムパフォーマンスを大幅に向上させた方法を説明しました。スラブアロケーターはいくつかのエンティティで動作します：

  • スラブは、同じサイズのセクションに分割される連続したメモリのストレッチ（通常は固定サイズ）です。これらの領域は、同じサイズのオブジェクトを格納するために使用されます。
  • cache-同じスプリットのスラブのリスト、
  • スラブアロケータ-キャッシュのセット。

この構造のおかげで、同じサイズのオブジェクトがローカルに保存されます。割り当て解除は、特定のサイトが脆弱で再利用できることを示すマークとして設計されています。これにより、メモリの断片化が回避されます。

このスラブアロケーターの定義から、サイズが制限された範囲にあるオブジェクトのリリースを強調するのに適していることが明らかになります。たとえば、毎回大きいサイズを割り当てると、新しいキャッシュが作成され、古いキャッシュは再利用されません。

講演者は、このため、一部のコンテナを放棄し、他のコンテナを優先する必要があったと述べています。たとえば、ベクターはリストに、ハッシュマップはツリーに置き換えられましたが、それでもパフォーマンスは向上しました。

アントンポルキン：C ++タクシートリック


アントン・ポルキンの報告は常に興味深いものであり、彼が提供するソリューションは見栄えがよいです。今回は、Antonが動的割り当ての点でpimplパターンをどのように改善できるかを示しました。このため、実装オブジェクトのストレージをオブジェクト自体に配置する必要があります。古典的なpimplパターンは次のとおりです。

// Foo.h
struct Foo {
    Foo();

private:
    struct Foo_impl; //forward declaration
    std::unique_ptr<Foo_impl> impl;
};

// Foo.cpp 
//implementation
struct Foo::Foo_impl {
};

動的割り当てを取り除きたいので、Fooオブジェクト内に直接場所を事前に準備します。
// Foo.h
struct Foo {
    Foo();

private:
    struct Foo_impl; //forward declaration
    std::aligned_storage_t<sizeof(Foo_impl), alignof(Foo_impl)> impl;
};

// Foo.cpp 
//implementation
struct Foo::Foo_impl {
}

Foo.hのFoo_impl型に関する完全な情報がなく、コンパイルエラーが発生するため、このメソッドは機能しません。残っている唯一の解決策は、ストレージのサイズを事前に推測することです。

// Foo.h
struct Foo {
    Foo();

private:
    struct Foo_impl; //forward declaration
    constexpr std::size_t kImplSize = 32;
    constexpr std::size_t kImplAlign = 8;
    std::aligned_storage_t<kImplSize, kImplAlign> impl;
};

// Foo.cpp 
//implementation
struct Foo::Foo_impl {
}

ただし、サイズがまだ正しいかどうかのチェックを追加する必要があります。オブジェクトを不適切なバッファに配置する試みはUBであるため、これは必要です。

// Foo.h
struct Foo {
    Foo();
    ~Foo();

private:
    constexpr std::size_t kImplSize = 32;
    constexpr std::size_t kImplAlign = 8;
    struct Foo_impl; //forward declaration
    std::aligned_storage_t<kImplSize, kImplAlign> impl;
};

// Foo.cpp 
//implementation
struct Foo::Foo_impl {
}

struct Foo::~Foo() {
    static_assert(kImplSize==sizeof(Foo_impl),"Size and sizeof(T) mismatch");
    static_assert(kImplAlign==alignof(kImplAlign),"Alignment and alignof(T) mismatch");

    // call destructor of Foo_impl
}

cppファイルをチェックし、何かが間違って示されている場合は、コンパイルエラーで終了し、プログラマーが2回目の試行から推測できるように、構造の正しいサイズを出力します。

アントンは、ADL機能を忘れずに、さまざまな形式のシリアル化ライブラリを便利にする方法を示しました。関数の引数にテンプレートパラメーターがある場合、関数はこれらの引数のパラメーター名前空間で検索されます。
