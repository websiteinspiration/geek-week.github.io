パーソナライズされたインターフェイス。パート2.自動ナビゲーション設定



ユーザーの作業の性質に応じたインターフェースの自動構成は難しい作業ですが、実現可能と思われます。今日、私たちのAndroidチームがパーソナライズされたインターフェイスの開発を始めた理由、実験中に取得したデータ、作成した結論、およびこの知識が製品全体のさらなる開発にどのように影響したかをお伝えします。序文を見逃した場合、最初の部分はアイデアの利点と欠点についてです。

ステップ0。私たちが何を望んでいるかを理解します。


当社の製品であるWrikeは積極的に成長し、新しい市場に参入しています。間もなくモバイルアプリケーションが完全に許容できないほど機能的に過負荷になることを理解しています。

不要なものをすべて削除して簡素化する必要があり、次の2つのことから始める価値があります。

  • 外出先でのユーザーエクスペリエンス機能。Wrikeがオフィスの外で完全に使用されている場合です。たとえば、イベント管理の分野では、プロジェクトのメインフェーズが数時間続くとき、彼らは食べ物を持ち込み、小道具を持ち込み、機器を設置します。
  • モバイルアプリケーションでの臨時の作業中は、ユーザーの焦点が外れるため、不必要な刺激で邪魔しないでください。

タスクがアプリケーションインターフェイスを簡素化することであった場合、主要なメトリックは最初の重要なアクションまでの時間です。このメトリックは、アプリケーションが起動された瞬間から、テープのスクロール、タスクの説明の編集、または目的のプロジェクトへの移動など、意味のあるアクションまで、画面で何が起こっているのかをユーザーが理解する速さを示します。

私たちは、すべてのユーザーが彼の仕事のスタイルに関係なく遭遇する共通のコンポーネント、つまり製品のナビゲーションから始めました。サービスのほとんどすべてのアクションがログに記録されるため、分析を行って、アプリケーションのどのセクションがより頻繁にクリックされ、どのセクションがより頻繁にクリックされないかを確認しました。セクション間のナビゲーションのみを追跡することは、機能の一部にとって、いずれにしても十分ではありませんでした。製品の一部には価値がありません。それらを見るだけでは、それらとの対話を開始する必要があります。たとえば、[受信トレイ]タブにはユーザーにとって重要なメッセージが含まれており、リボンをスクロールするだけで済みます。[お気に入り]タブに移動すると、お気に入りのアイテムに移動できます。言い換えれば、「お気に入り」セクション自体には価値がない場合が多いですが、そこからの迅速なナビゲーションは重要なアクションです。

ステップ1.データ収集


製品のWebバージョンのユーザーは十分にいるため、統計的有意性に問題はありませんでした。アプリケーションの場合、状況はさらに複雑になりました。それらの活動は大幅に少なく、聴衆のサイズも同様です。その理由は、従業員がほとんどの時間をオフィスのコンピューターに費やしている企業がWrikeを購入することが多いためです。たとえば、ユーザーの間で最も人気のあるセグメントの1つはマーケティング代理店です。これらの人たちは、広告キャンペーンを実行し、コンテンツ（テキストとビジュアル）を準備し、承認を実行します。

まず、製品のWebバージョンでのさまざまなユーザーの作業に関する情報を収集しました。ユーザーがカレンダーを開く頻度、受信トレイで作業する頻度、タスクバーの変更を確認する、リクエストに記入するなどです。次に、モバイルアプリケーションで同じ人々の作業に関する統計を収集しました。結果は、顧客がWebバージョンとアプリケーションでサービスのほぼ同じセクションを使用することを示しました。したがって、「外出先でのユーザーエクスペリエンスはWebバージョンとは大きく異なる」という仮説は確認されていません。ただし、重要ではないにせよ、機能の違いにより、新しい仮説が生ま​​れました。ユーザーは、アプリケーションの存在を認識していないため、アプリケーションの一部のツールを操作しません。

最終的に、セクションごとに意味のあるアクションを決定し、係数も導入しました。まず、モバイルデバイスから収集されたデータは、Webバージョンからのデータよりも重みがありました。次に、パターンに気づきました。多くの場合、ツールはペアで積極的に使用されています。オーディエンスの70％がツールAとBを同時に使用しているとすると、ツールの1つをアクティブに使用すると、下部のナビゲーションで1秒の可能性が必然的に高くなります。

製品に精通しているユーザーから詳細なフィードバックを得ることが重要でした。すべてのアカウントのほぼすべてのユーザーに関するデータがありましたが、リスクを冒したくありませんでした。最初の打ち上げは社内でした。Androidチーム全体が参加し、非常にアクティブなユーザー、アクティブなユーザー、アクティブでないユーザー、アクティブでないユーザーの5人の代表者が参加しました。

ナビゲーションであろうとゲームキャラクターの外観であろうと、構成プロセスはユーザーがやりたいことだと思います。そして、この種の投資は多くの場合、ユーザーにとっての製品の価値を高めます。

もちろん、1つのアカウント内の統計的有意性について話すことはできません。まず、登録ユーザーのオーディエンス全体に対してA / Aを実行し、グループに違いがないことを確認しました。次に、A / Bステージに進み、エンジニアリングアナリスト部門にグループB

の各ユーザーのベースにパッチを適用するように依頼しました。以下の仮説がテストされました。

  • インターフェース構成が改善されたことで、作業がより速くなります。
  • インターフェイスのカスタマイズは保持にプラスの影響を与えます。
  • ユーザーは、インターフェイスをカスタマイズできる場合、アプリケーションに対する忠実度が高くなります。

各仮説の背後に隠されていたものと、それらをどのように確認したかを検討してください。

仕事のスピードについて


私たちは仕事管理サービスを行うので、私たちの目標はユーザーの生産性を高めることです。生産性は人が職場にもたらす「利益の量」であると私は言うでしょう。利点はさまざまな側面で表現できますが、私たちのサービスは仕事に関するものであることを忘れないでください。多くの場合、仕事そのものではありません。デザイナーがバナーやウェブサイトを描き、開発者がコードを書き、セキュリティ部門がより安全なことを行います。彼らの仕事は他のツールに集中しており、Wrikeはプロセスの透明性を提供し、誰もが一方向に移動できるようにします。したがって、セッションの長さを長くするという目標はありませんが、新しいコンテンツを生成し、システムに頻繁にアクセスできることをうれしく思います。私たちの仕事は、アプリケーションでの作業を非常に高速にして、ユーザーがコンテキストを作業用の通信に切り替えて、再びビジネスに戻ることが難しくないようにすることです。

ローンチ後2週間以内に、主要な指標（0.0を超える列）にドローダウンが見られました。



グラフについての簡単な説明：

行0.0はグループA-インターフェースに変更を加えていないユーザーです。 0.0を超えるものはすべて、重要なアクションまでの時間の増加を示し、それ以下はすべて-逆に、時間の減少を示します。列は、それぞれ5番目から95番目までのパーセンタイルを示します。

この2週間で、最初の重要なアクションまでの時間を増やしましたが、グラフが徐々に右に「移動」する傾向があります。インターフェースの強制変更によりユーザーの習慣が崩れ、ナビゲーションの新しい変更に慣れるまでに時間がかかると仮定しました。

保持指標は大幅に低下しなかったため、しばらく待つことにしましたが、それには十分な理由があります。実験の8週間後、ほとんどすべてのユーザーが新しいナビゲーションに慣れ、アプリケーションの作業速度が向上しました。



最終的に、最初の重要なアクションまでの時間を約18％短縮することができました。

保持について


私たちの信念は、アプリケーションでの快適な経験に基づいて構築されました。より便利なツールがクライアントのシステムへの復帰を刺激することを提案しました。すぐに言っておきますが、保持率は少し増加しましたが、統計的に有意とは言えないため、この仮説は確認されていません。

メインサービスとの関連で衛星製品であるモバイルアプリケーションを使用して、次の結論に達しました。衛星がどれほど便利であっても、製品のメインバージョンよりも常に劣っています。たとえば、6時間のうち2時間コンピューターから離れていて、チームとの接続が必要な場合、この2時間はWrikeアプリケーションを使用します。残りの時間は、大きなモニターとフルキーボードで作業する方が便利です。私たちの場合、これはコンピュータでの作業タスクに集中する方が簡単であるという事実によるものです。

忠誠について


理解可能なエンティティの単純な構成は、アプリケーションとの個人の接続を増やすことでした。クライアントがコンテンツから価値を受け取るだけでなく、気遣いを感じるとき、彼のワークスペースは特別になります。私たちは多くの肯定的なフィードバックを得ました。ユーザーは自分の作業スタイルに合ったインターフェースを見たいと思っています。そのため、グループAのユーザーからは、「快適でかっこいい」という理由で、このような機能も必要だとのことでした。

ステップ5.グループAへのB / C


グループBの方が成功していることがわかりましたが、アプローチBを使用してもサポートの面で費用のかかる作業がまだ1つあったため、実験を急いで完了できませんでした。ユーザーの行動の全体的な分析は、ナビゲーションの一般的な再設計により変更されようとしていたUIに基づいていました。会社全体でやらなければならないことがたくさんあり、各ユーザーのナビゲーションを構成するスクリプトは長い間サポートされずに放置されていました。

したがって、グループAを2つの部分に分けました。グループBは上記の実験を受け、グループCはオンボーディングを受け、構成を個別に構成する提案を受けました。しかし、ユーザーが何かをカスタマイズすることはめったにないため、グループCの10％だけが私たちの提案に従い、ナビゲーションで何かを変更しました。顧客はアプリケーションを開いて、サービスプロバイダーが必要とすることではなく、必要なことを実行します。最初の週から自分でインターフェースを設定した人が、最初の重要なアクション（8番目のグループBのように）まで速度を上げたことがわかりました。グループCは短時間で良好なパフォーマンスを示しましたが、オンボーディングが不十分だったため、エンゲージメントは低くなりました。

ステップ6.初めてログインするときのナビゲーションの構成


あるUXデザイナーが私に言ったことがあります。「顧客は2つのタイプに分けられます。更新を読む人と読まない人があります。オンボーディングとは何の関係もありません。」例外はこのフレーズにあります。ユーザーに何かを強制したり、何かを支払ったりすることですが、一般的に私は彼の発言に同意します。長い間、現在の分析モデルでサービスを維持できないことを理解していたため、より適切なオンボーディングを行うことが決定されました。

ユーザーに調査への参加を呼びかけました。調査のデザインは、明確で明確なものであり、相互作用のメカニズムは原始的でした。開発コストとクライアントのメリットのバランスをとって、次の決定を下しました。



非公式の訴えとして、ユーザーに最も頻繁に使用するものを尋ね、アプリケーションをユーザーのニーズに合わせてカスタマイズすることを通知します。もちろん、これは、ロードインジケーターを使用したアニメーションナビゲーションの変更ほど印象的ではありません。そこでサーバーに魔法をかけているのです。そして、それはユーザーからのより多くのアクションを必要とします。しかし、このようなシステムにより、ナビゲーションパネルの設定に適切に変換することができました。

合計


試しましたが、間違えられませんでした。モバイルアプリケーションでは、すべての顧客がすでにソリューションを使用しています。次のステップは、同様のパスに沿ってWebバージョンを移動することです。自動化は私たちに大きな成功をもたらしましたが、それを維持することは高価でしたので、調査の形でのオンボーディングはかなり良いソリューションであることが判明しました。

最初の部分で書いたように、パーソナライズされたインターフェイスは、むしろ、さまざまな観点から有益な開発ベクトルです。正しい方法は1つではないと思いますが、この方向での実験とセットアッププロセスの自動化により、業界に関する新しい知識が得られ、複雑な製品が簡単になり、ユーザーが幸せになると確信しています。

あとがき


現在、製品のWebバージョンでナビゲーションのアクティブな作業が行われているため、プロジェクトは一時停止しています。イノベーションはユーザーエクスペリエンスに大きな影響を与えると考えています。それでも、次のステップは何かという考えがあります。これは、モバイルデバイスに登録して初めてWrikeを見る人のための自動インターフェース設定です。このようなユーザーは一度にシステム全体を認識する準備ができていないため、システムを怖がらせないようにダンプする価値はありません。

製品のWebバージョンに既に慣れているユーザーの場合、次のステップはコンテンツの編成に移行することです。どの「リクエスト」ユーザーが他のユーザーより頻繁に使用し、どの「レポート」と定期的にやり取りし、どのリストが「タスクバー」でより頻繁に表示されるかを知っています。結局のところ、最近B2C製品でよく使われている、最近の、またはおすすめのようなことをしたいと思っています。
