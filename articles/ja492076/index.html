Javaの六角形アーキテクチャの例

この記事の翻訳は、特にJava開発者コースの受講生のために作成されました。




開発者として、私たちはしばしば維持するのが難しいレガシーコードを扱わなければなりません。大規模で複雑なスパゲッティコードの単純なロジックを理解するのがいかに難しいかご存知でしょう。コードの改善や新しい機能の開発は、開発者にとって悪夢になります。

ソフトウェア設計の主な目標の1つは、メンテナンスの容易さです。保守が不十分なコードは管理が難しくなります。スケーリングが難しいだけでなく、新しい開発者を引き付けることが問題になっています。

ITの世界では、物事は急速に進んでいます。新しい機能を緊急に実装するように求められた場合、またはリレーショナルデータベースからNoSQLに切り替えたい場合、最初の反応はどうなりますか？



テストカバレッジが優れていると、新しいリリースで問題が発生しないという開発者の信頼が高まります。ただし、ビジネスロジックがインフラストラクチャロジックと絡み合っている場合は、テストに問題がある可能性があります。

六角形アーキテクチャの概要


用語六角形のアーキテクチャ（六角形、六角形のアーキテクチャが）れたアリスター・コバーンによって2006年に造語します。このアーキテクチャスタイルは、ポートおよびアダプタアーキテクチャとも呼ばれます。簡単に言うと、アプリケーションのコンポーネントは多くのエンドポイント（ポート）を介して相互作用します。要求を処理するには、ポートに一致するアダプターが必要です。

ここで、コンピューターのUSBポートに例えを描くことができます。互換性のあるアダプター（充電器またはフラッシュドライブ）があれば、それらを使用できます。



このアーキテクチャは、中央（コア）にビジネスロジックがあり、相互作用するオブジェクトとそれを制御するコンポーネントで囲まれた六角形として概略的に表すことができ、入力データを提供します。

実際には、ユーザー、API呼び出し、自動化スクリプト、ユニットテストが相互に作用し、アプリケーションへの入力を提供します。ビジネスロジックがユーザーインターフェイスロジックと混在している場合、多くの問題が発生します。たとえば、データ入力をユーザーインターフェイスから単体テストに切り替えるのは困難です。

アプリケーションは、データベース、メッセージキュー、Webサーバー（HTTP API呼び出しを介して）などの外部オブジェクトとも対話します。必要に応じて、データベースに移行するか、データをファイルにアップロードすると、ビジネスに影響を与えることなくこれを実行できるはずです。論理。

「ポートとアダプタ」という名前が示すように、相互作用が発生する「ポート」があり、「アダプタ」はユーザー入力を処理してドメインの「言語」に変換するコンポーネントです。アダプターは、データベース、メッセージキューなどの外部システムとの対話のロジックをカプセル化し、ビジネスロジックと外部オブジェクト間の通信を容易にします。

次の図は、アプリケーションが分割されるレイヤーを示しています。



六角形のアーキテクチャは、アプリケーションの3つの層、ドメイン、アプリケーション、インフラストラクチャを区別します。

  • ドメイン。レイヤーにはコアビジネスロジックが含まれます。彼は外層の実装の詳細を知る必要はありません。
  • アプリケーション。レイヤーは、ドメインとインフラストラクチャのレイヤー間のブリッジとして機能します。
  • インフラ。ドメインと外界との相互作用の実装。内側の層は彼にとってブラックボックスのように見えます。

このアーキテクチャーによれば、2種類の参加者がアプリケーションと対話します。プライマリ（ドライバー）とセカンダリ（ドリブン）です。主要なアクターはリクエストを送信し、アプリケーションを管理します（たとえば、ユーザーまたは自動テスト）。二次的なものは、外部との通信のためのインフラストラクチャを提供します（これらはデータベースアダプタ、TCPまたはHTTPクライアントです）。
これは次のように表すことができます。



六角形の左側は、ドメインへの入力を提供するコンポーネント（アプリケーションを「制御」）で構成され、右側は、アプリケーションによって制御されるコンポーネントで構成されます。

例


映画のレビューを保存するアプリケーションを設計してみましょう。ユーザーは、映画の名前を含むリクエストを送信して、ランダムに5つのレビューを取得できる必要があります。

簡単にするために、RAMにデータストレージを備えたコンソールアプリケーションを作成します。ユーザーへの応答がコンソールに表示されます。

アプリケーションにリクエストを送信するユーザー（User）がいます。したがって、ユーザーは「マネージャー」（ドライバー）になります。アプリケーションは、あらゆるタイプのストレージからデータを受信し、結果をコンソールまたはファイルに出力できる必要があります。管理（駆動）オブジェクトは、「データウェアハウス」（IFetchMovieReviews）および「応答プリンター」（IPrintMovieReviews）になります。

次の図は、アプリケーションの主なコンポーネントを示しています。



左側は、アプリケーションへのデータ入力を提供するコンポーネントです。右側には、データベースとコンソールを操作するためのコンポーネントがあります。

アプリケーションコードを見てみましょう。

制御ポート

public interface IUserInput {
    public void handleUserInput(Object userCommand);
}

管理ポート

public interface IFetchMovieReviews {
    public List<MovieReview> fetchMovieReviews(MovieSearchRequest movieSearchRequest);
}

public interface IPrintMovieReviews {
    public void writeMovieReviews(List<MovieReview> movieReviewList);
}

管理されたポートアダプター

ムービーは、ムービーリポジトリ（MovieReviewsRepo）から取得されます。コンソールでの映画レビューの表示はクラスになりConsolePrinterます。上記の2つのインターフェースを実装しましょう。

public class ConsolePrinter implements IPrintMovieReviews {
    @Override
    public void writeMovieReviews(List<MovieReview> movieReviewList) {
        movieReviewList.forEach(movieReview -> {
            System.out.println(movieReview.toString());
        });
    }
}

public class MovieReviewsRepo implements IFetchMovieReviews {
    private Map<String, List<MovieReview>> movieReviewMap;

    public MovieReviewsRepo() {
        initialize();
    }

    public List<MovieReview> fetchMovieReviews(MovieSearchRequest movieSearchRequest) {

        return Optional.ofNullable(movieReviewMap.get(movieSearchRequest.getMovieName()))
            .orElse(new ArrayList<>());
    }

    private void initialize() {
        this.movieReviewMap = new HashMap<>();
        movieReviewMap.put("StarWars", Collections.singletonList(new MovieReview("1", 7.5, "Good")));
        movieReviewMap.put("StarTreck", Arrays.asList(new MovieReview("1", 9.5, "Excellent"), new MovieReview("1", 8.5, "Good")));
    }
}

ドメイン


このアプリケーションの主なタスクは、ユーザーのリクエストを処理することです。フィルムを入手して処理し、結果を「プリンター」に転送する必要があります。現時点では、機能は映画検索のみです。ユーザーのリクエストを処理するには、標準インターフェースを使用しますConsumer。

メインクラスを見てみましょうMovieApp。

public class MovieApp implements Consumer<MovieSearchRequest> {
    private IFetchMovieReviews fetchMovieReviews;
    private IPrintMovieReviews printMovieReviews;
    private static Random rand = new Random();

    public MovieApp(IFetchMovieReviews fetchMovieReviews, IPrintMovieReviews printMovieReviews) {
        this.fetchMovieReviews = fetchMovieReviews;
        this.printMovieReviews = printMovieReviews;
    }

    private List<MovieReview> filterRandomReviews(List<MovieReview> movieReviewList) {
        List<MovieReview> result = new ArrayList<MovieReview>();
        // logic to return random reviews
        for (int index = 0; index < 5; ++index) {
            if (movieReviewList.size() < 1)
                break;
            int randomIndex = getRandomElement(movieReviewList.size());
            MovieReview movieReview = movieReviewList.get(randomIndex);
            movieReviewList.remove(movieReview);
            result.add(movieReview);
        }
        return result;
    }

    private int getRandomElement(int size) {
        return rand.nextInt(size);
    }

    public void accept(MovieSearchRequest movieSearchRequest) {
        List<MovieReview> movieReviewList = fetchMovieReviews.fetchMovieReviews(movieSearchRequest);
        List<MovieReview> randomReviews = filterRandomReviews(new ArrayList<>(movieReviewList));
        printMovieReviews.writeMovieReviews(randomReviews);
    }
}

次に、CommandMapperModelコマンドをハンドラーにマップするクラスを定義します。

public class CommandMapperModel {
    private static final Class<MovieSearchRequest> searchMovies = MovieSearchRequest.class;

    public static Model build(Consumer<MovieSearchRequest> displayMovies) {
        Model model = Model.builder()
            .user(searchMovies)
            .system(displayMovies)
            .build();

        return model;
    }
}

制御ポートアダプタ


ユーザーはインターフェースを介してシステムと対話しますIUserInput。実装は実行を使用ModelRunnerおよび委任します。

public class UserCommandBoundary implements IUserInput {
    private Model model;

    public UserCommandBoundary(IFetchMovieReviews fetchMovieReviews, IPrintMovieReviews printMovieReviews) {
        MovieApp movieApp = new MovieApp(fetchMovieReviews, printMovieReviews);
        model = CommandMapperModel.build(movieApp);
    }

    public void handleUserInput(Object userCommand) {
        new ModelRunner().run(model)
            .reactTo(userCommand);
    }
}

次に、上記のインターフェースを使用するユーザーを見てみましょう。

public class MovieUser {
    private IUserInput userInputDriverPort;

    public MovieUser(IUserInput userInputDriverPort) {
        this.userInputDriverPort = userInputDriverPort;
    }

    public void processInput(MovieSearchRequest movieSearchRequest) {
        userInputDriverPort.handleUserInput(movieSearchRequest);
    }
}

応用


次に、コンソールアプリケーションを作成します。マネージアダプターは依存関係として追加されます。ユーザーはリクエストを作成してアプリケーションに送信します。アプリケーションはデータを受信し、処理して、コンソールへの応答を表示します。

public class Main {

    public static void main(String[] args) {
        IFetchMovieReviews fetchMovieReviews = new MovieReviewsRepo();
        IPrintMovieReviews printMovieReviews = new ConsolePrinter();
        IUserInput userCommandBoundary = new UserCommandBoundary(fetchMovieReviews, printMovieReviews);
        MovieUser movieUser = new MovieUser(userCommandBoundary);
        MovieSearchRequest starWarsRequest = new MovieSearchRequest("StarWars");
        MovieSearchRequest starTreckRequest = new MovieSearchRequest("StarTreck");

        System.out.println("Displaying reviews for movie " + starTreckRequest.getMovieName());
        movieUser.processInput(starTreckRequest);
        System.out.println("Displaying reviews for movie " + starWarsRequest.getMovieName());
        movieUser.processInput(starWarsRequest);
    }

}

改善、変更できるもの


  • 私たちの実装では、1つのデータストアから別のデータストアに簡単に切り替えることができます。ストレージ実装は、ビジネスロジックを変更せずにコードに挿入できます。たとえば、データベースアダプタを作成することで、メモリからデータベースにデータを転送できます。
  • コンソールへの出力の代わりに、データをファイルに書き込む「プリンター」を実装できます。このような多層アプリケーションでは、機能の追加やバグの修正が容易になります。
  • ビジネスロジックをテストするために、複雑なテストを作成できます。アダプターは単独でテストできます。したがって、全体的なテストカバレッジを増やすことが可能です。

結論


六角形のアーキテクチャの次の利点に注目できます。

  • 伴奏 -疎結合で独立したレイヤー。他のレイヤーに影響を与えることなく、1つのレイヤーに新しい機能を簡単に追加できます。
  • テスト容易性 -単体テストは簡単かつ迅速に実行されるように記述されています。依存関係をシミュレートするスタブオブジェクトを使用して、各レイヤーのテストを記述できます。たとえば、メモリにデータウェアハウスを作成することで、データベースへの依存関係を削除できます。
  • 適応性 -メインのビジネスロジックは、外部オブジェクトの変更から独立します。たとえば、別のデータベースに移行する必要がある場合、ドメインに変更を加える必要はありません。データベースに適切なアダプターを作成できます。
