MLデータの分野でDevOpsが必要な理由



本番環境で機械学習（ML）をデプロイすることは簡単な作業ではありませんが、実際には、従来のソフトウェアをデプロイするよりも桁違いに難しいものです。その結果、ほとんどの組織がMLを使用して製品を宣伝し、顧客にサービスを提供することをあきらめ、あきらめているため、ほとんどのMLプロジェクトは、日中の生産を見ることはありません。

私たちが見る限りでは、ほとんどのチームが本番環境でMLを作成し、予想される規模でデプロイするための基本的な障害は、DevOpsプラクティスをまだ導入できていないことです。機械学習で。MLモデルの作成とデプロイのプロセスは、すでにリリースされているMLOpsソリューションによって部分的に開示されていますが、MLの最も困難な側面の1つであるデータのサポートが不足しています。

この記事では、業界がMLデータ用のDevOpsソリューションを必要とする理由、およびMLデータの固有の困難がMLを実践し、本番環境にデプロイする取り組みをどのように妨げるかについて説明します。この記事では、現在のMLインフラストラクチャエコシステムの空白について説明し、機械学習用の集中型データプラットフォームであるTectonでそれを埋めることを提案します。Tectonのリリースの詳細については、こちらをクリックして、共同創設者Mikeの記事を読んでください。

Tectonは、Uber、Google、Facebook、Twitter、Airbnb、AdRoll、Quoraなどの企業向けの内部MLプラットフォームを作成したエンジニアのチームによって作成されました。これらの企業のMLへの多大な投資により、組織と製品でMLを広範囲に使用するためのプロセスとツールの開発が可能になりました。この記事で紹介するレッスンは、Tectonプラットフォーム自体と同様に、過去数年にわたる本番環境でのML導入チームの経験に主に基づいています。

ソフトウェアのリリースが長く苦痛だった時期を覚えていますか？


20年前のソフトウェア開発と展開プロセスと今日のMLアプリケーション開発プロセスには多くの共通点があります。フィードバックシステムは非常に長くなり、製品のリリースに至るまでに、初期要件と設計はすでに時代遅れになっています。そして、問題の終わりに、DevOpsの形でソフトウェア開発の一連のベストプラクティスが出現し、開発ライフサイクルを管理する方法を提供し、継続的かつ迅速な改善を実現できるようにしました。

DevOpsアプローチにより、エンジニアは明確に定義された共通コードベースで作業できます。段階的な変更を展開する準備ができると、エンジニアはバージョン管理システムを介して変更を検証します。継続的インテグレーションとデリバリー（CD / CI）のプロセスは、最新の変更を取り、ユニットテストを実施し、ドキュメントを作成し、インテグレーションテストを実施し、その結果、制御された方法で本番環境への変更をリリースするか、配布用のリリースを準備します。

  • プログラマーはコードを最初から最後まで所有します。彼らは、本番環境のすべてのコード行に権限を与え、完全に責任があります。この所有感により、一般にコードの品質が向上するだけでなく、プログラムの可用性と信頼性も向上します。

Tectonの紹介：日付機械学習プラットフォーム


Tecronでは、データサイエンスの分野で最も一般的で最も難しい問題を支援するために、機械学習用の日付プラットフォームを作成しています。

大まかに言うと、Tecronプラットフォームには次の機能があります。

  1. ソースデータを関数とラベルに変換する関数プロセス
  2. アーカイブされた関数とタグデータを格納するための関数リポジトリ
  3. 最新の関数値をプロダクションに発行するための関数サーバー
  4. トレーニングデータと関数プロセスの操作のためのSDK
  5. 機能、ラベル、データセットを監視および追跡するWeb UI
  6. データ品質またはドリフトの問題、およびアラートを判別するための監視エンジン
