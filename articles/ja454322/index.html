意見があります：ブラウザーのDANEテクノロジーが失敗しました

DANEがDNSドメイン名認証とは何か、そしてなぜそれがブラウザで広く使用されていないのかについて話します。


/アンスプラッシュ/パウリウス・ドラグナス

DANEとは


認定センター（CA）-ID暗号化SSL証明書に従事している組織です。彼らは彼らに彼らの電子署名をつけて、信憑性を確認します。ただし、証明書が違反して発行される場合、状況が発生することがあります。たとえば、Googleは昨年、セキュリティ侵害のため、Symantec証明書の「信頼を解除するための手順」を開始しました（このストーリーについては、ブログで1回または2回詳細に説明しました）。 このような状況を回避するために、数年前、IETF は DANEテクノロジーの開発を開始しました（ただし、ブラウザーでは広く使用されていませんでした。なぜそれが起こったのか、詳しく説明します）。



DANE（名前付きエンティティのDNSベース認証）は、DNSSEC（ネームシステムセキュリティ拡張機能）を使用してSSL証明書の有効性を制御できるようにする一連の仕様です。DNSSECは、スプーフィングに関連する攻撃を最小限に抑えるドメインネームシステムの拡張機能です。これら2つのテクノロジーを使用して、ウェブマスターまたはクライアントはDNSゾーンオペレーターの1つに連絡し、使用された証明書の有効性を確認できます。

実際、DANEは自己署名証明書として機能し（DNSSECはその信頼性の保証です）、CAの機能を補完します。

使い方


DANE仕様はRFC6698で説明されています。ドキュメントによると、新しいタイプがDNSリソースレコードに追加されました-TLSA。送信された証明書、送信されたデータのサイズとタイプ、およびデータ自体に関する情報が含まれています。Webマスターは、証明書のデジタルフィンガープリントを作成し、DNSSECで署名し、TLSAに配置します。

クライアントはインターネット上のサイトに接続し、その証明書をDNSオペレーターから受け取った「コピー」と比較します。それらが一致する場合、リソースは信頼できると見なされます。

DANE wikiページには、TCPポート443を介したexample.orgサーバーの次のDNSクエリの例が記載されています。

IN TLSA _443._tcp.example.org

その答えは次のようになります。

 _443._tcp.example.com. IN TLSA (
   3 0 0 30820307308201efa003020102020... )

DANEには、TLSA以外のDNSレコードと連携するいくつかの拡張機能があります。1つ目は、SSH接続のキー検証用のSSHFP DNSレコードです。これは、中に記述されているRFC4255、  RFC6594  および  RFC7479。2つ目は、PGPを使用した鍵交換用のOPENPGPKEYエントリ（RFC7929）です。最後に、3番目は、S / MIMEを介した暗号鍵交換用のSMIMEAレコードです（標準はRFCで作成されておらず、そのドラフトしかありません）。

DANEの問題


5月中旬、DNS-OARC会議が開催されました（これは、ドメイン名システムのセキュリティ、安定性、および開発を扱う非営利組織です）。パネルの1つで、エキスパートはブラウザのDANEテクノロジーが（少なくとも現在の実装では）失敗したと結論付けました。カンファレンスに出席した5人の地域インターネットレジストラーの 1つであるAPNICのシニアフェローであるGeoff Huston は、DANEを「死んだ技術」と語っています。

一般的なブラウザは、DANEによる証明書認証をサポートしていません。市場にはTLSAレコードの機能を明らかにする特別なプラグインがありますが、それらのサポートは段階的に廃止されています。

ブラウザでのDANEの配布に関する問題は、DNSSEC検証プロセスの期間に関連しています。システムは、初めてリソースに接続するときに、SSL証明書の信頼性を確認し、DNSサーバーのチェーン全体（ルートゾーンからホストドメインまで）を通過するために、暗号計算を実行する必要があります。


/ Unsplash / Kaley Dykstra

この欠陥は、TLSのDNSSECチェーン拡張を使用してMozillaで試されました。認証中にクライアントが確認しなければならないDNSレコードの数を減らすことになっていた。ただし、解決できない開発チーム内で意見の相違が生じました。その結果、2018年3月にIETFによって承認されましたが、プロジェクトは中止されました。

DANEの人気が低いもう1つの理由は、DNSSECの普及率が世界で低いことです。リソースの19％しか機能しません。専門家はこれはDANEを積極的に宣伝するには十分ではないと感じました。

ほとんどの場合、業界は別の方向に発展します。 DNSを使用してSSL / TLS証明書を検証する代わりに、市場プレーヤーは逆に、DNS-over-TLS（DoT）およびDNS-over-HTTPS（DoH）プロトコルを促進します。Habréに関する以前の資料の最後で言及しました。彼らは、DNSサーバーへのユーザー要求を暗号化して検証し、攻撃者がデータを偽装するのを防ぎます。年の初めに、DoTはすでに実装されていますパブリックDNSとしてGoogleで。DANEに関しては、テクノロジーが「サドルに戻る」ことができ、それでも大規模になることができるかどうかは、今後見られる必要があります。
