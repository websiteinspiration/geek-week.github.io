React Lazy / Suspenseテクノロジーが助けになる


問題の原因を発見したら、次はこの問題を解決します。アニメーションに必要なすべてのものを作業の最初にロードする必要がないことは明らかでした。実際、ユーザーの現在のセッション中に、彼はお祝いのウィンドウを表示する必要がないというかなりの可能性がありました。次に、最近導入されたReact Lazy / Suspenseテクノロジーを見て、おそらくそれらを試す良い機会があると思いました。

「レイジー」（レイジー）コンポーネントの概念に慣れていない場合は、その意味がアプリケーションを小さなコードに分割することであることを理解してください。これらのフラグメントのダウンロードは、必要な場合にのみ実行されます。私の場合、これは、おめでとうを示すのに責任のあるコンポーネントをプログレスバーのメイン機能から選択する必要があるという事実で表されました。このコンポーネントのダウンロードは、ユーザーが推奨された一連の手順を完了した場合にのみ必要でした。

React 16.6.0（およびそれ以降のバージョン）には、遅延コンポーネントをレンダリングするように設計されたシンプルなAPIがあります。これらはReact.lazyとReact.Suspenseです。例を考えてみましょう：

const OtherComponent = React.lazy(() => import('./OtherComponent'));

function MyComponent() {
  return (
    <div>
      <React.Suspense fallback={<div>Loading...</div>}>
        <OtherComponent />
      </React.Suspense>
    </div>
  );
}

要素を表示するコンポーネント<div>があり、その中にコンポーネントSuspenseをラップするコンポーネントがありますOtherComponent。この例の最初の行を見るとOtherComponent、コードに直接インポートされていないことがわかります。通常、同様のインポートはのようになりimport OtherComponent from './OtherComponent';ます。

代わりに、インポートコマンドはファイルパスを取得する関数として設計されています。 Webpackにはコード分離ツールが組み込まれているため、このメカニズムは機能します。同様の構造がコードに存在する場合、promiseが返され、ファイルをダウンロードした後、このファイルのコンテンツで解決されます。インポートチームは関数にラップされていますReact.lazy。

レンダリングする材料には、返されるものがMyComponentされているOtherComponentコンポーネントに包まれたReact.Suspense性質を持っていますfallback。私たちの場合、レンダリングがになるOtherComponentと、対応するコンポーネントのロードが開始されることがわかります。その間、プロパティに書き込まれたものがレンダリングされますfallback。この例では、これはtext Loading…です。それは、実際には、すべてです。これらのメカニズムはその役割を果たします。

確かに、Lazy / Suspenseで作業する際に考慮すべきいくつかの機能があります。

  1. 「遅延」方法でインポートされるコンポーネントには、コンポーネントのエントリポイントとなるデフォルトのエクスポートが含まれている必要があります。名前付きエクスポートはここでは使用できません。
  2. 関数でインポートしたコンポーネントをコンポーネントにラップする必要がありReact.lazyますReact.Suspense。コンポーネントにReact.Suspenseはプロパティを指定する必要がありますfallback。そうしないと、エラーが発生します。ただし、コンポーネントの遅延読み込みが完了するまで何もレンダリングしたくない場合は、このプロパティに何かを書き込む必要がないように巧妙にバイパスせずに、単にfallback値nullに書き込むことができます。

はい、役に立ちました！コード分​​割は驚くほどうまくいきました。Webpackを使用して新しいバンドルを分析した結果を見てみましょう。


コード分​​離後のWebpack Bundle Analyzerの結果

ご覧のとおり、バンドルのサイズは約50％減少し、96 Kbになりました。いいね！

それで、今、問題は解決されましたか？いいえ、残念です。ページを見ると、おめでとうとポップアップウィンドウが配置を失っていることがわかりました。


おめでとうウィンドウが必要な場所に表示されない

問題は、Reactコンポーネントの状態を変更してウィンドウを開くように「要求」したことです。その間、nullコンポーネントを使用して既にレンダリングしました（つまり、何もレンダリングしていません）React.Suspense。必要なデータを遅延読み込みした後、関連資料がDOMに追加されました。ただし、ポップアップの配置はすでに行われています。その結果、対応するコンポーネントのプロパティは変更されなかったため、このコンポーネントは、配置に関する問題を解決する必要があることを「知りませんでした」。ブラウザウィンドウのサイズを変更すると、対応するコンポーネントがプロパティの変更とウィンドウのサイズ変更のイベントを監視し、必要に応じて再配置を開始したため、ポップアップウィンドウが適切な場所に表示されました。

この問題を解決するには？解決策は「仲介者」を排除することでした。

最初に「レイジー」コンポーネントをロードし、それからウィンドウを開く必要があることを祝福してウィンドウに通知する状態に書き込む必要がありました。同じWebpackコード共有メカニズムを使用してこれを行うことができましたが、次のようにインポートを実装せずにReact.lazy：

async loadAndSetHappyMoment() {
  const component = await import(
    '../SidebarHappyMoment/SidebarHappyMoment.component'
  );
  this.SidebarHappyMoment = component.SidebarHappyMoment;
  this.setState({
    tooltipLevel: TooltipLevel.happyMoment,
  });
}

この関数は、私のコンポーネントがWebソケットのメカニズムを介して、お祝いのウィンドウを表示する必要があることを通知された後に呼び出されます。 Webpack関数import（コードの2行目）を使用しました。あなたが覚えているなら、私はこの関数がpromiseを返すことを上で言ったので、その結果、構文を使用することができましたasync/await。

コンポーネントの読み込みが完了したら、コンポーネントのインスタンスに書き込みます（コマンドthis.SidebarHappyMoment = component.SidebarHappyMoment;）。これにより、後でレンダリングするときに使用できます。名前付きエクスポートを使用できるようになりました。私の場合、上記の行で名前を使用しましたSidebarHappyMoment。そして最後に、他の何よりも重要なことですが、開く必要があるウィンドウに「通知」し、コンポーネントの操作の準備ができていることを知った後、それに応じて状態を変更します。

その結果、レンダリングコードは次の形式になります。

renderTooltip() {
  if (this.state.tooltipLevel === TooltipLevel.happyMoment) {
    return <this.SidebarHappyMoment />;
  }
  //  ...
}

コマンドreturn <this.SidebarHappyMoment />;が戻ることに注意してくださいthis.SidebarHappyMoment-以前にコンポーネントのインスタンスに書き込んだものです。これは通常の同期レンダリング関数で、100万回使用したものと同じです。そして今、おめでとうございますウィンドウが表示されるべき場所に正確に表示されます。そして、その内容は、その内容が使用できるようになった後にのみ開くということです。

製品はアーキテクチャを定義します


このセクションの見出しのアイデアがあなたの想像力に大きな疑問符を呼び起こした場合は、これが事実であることに注意してください。製品はアーキテクチャを定義します。

重要なのは、コンポーネントが最初にレンダリングされたときに、製品が何を表示してインタラクティブにするかを定義することです。これにより、開発者はメインコードから正確に何を分離し、必要に応じて後でロードできるかを理解できます。ユーザーがサイトをセットアップするための推奨手順を完了した後、またはユーザーがサイト管理者でない場合は、進行状況バーとそれが何であるかを示す必要がないことを考え、「覚えた」接続されています。現在、この情報を使用して、バンドルの分離を続けることができました。それが私がしたことです。


バンドルの継続的な分離

その後、バンドルのサイズはわずか38 Kbでした。そして、私たちは190 Kbから始めたことを思い出させます。バンドルのサイズが80％縮小されます。ちなみに、すでにコード分離の可能性は他にもあります。バンドルの最適化を続けるのが待ちきれません。
