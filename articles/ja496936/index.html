Kubernetesのcert-managerでSSL証明書を暗号化しましょう



この記事では、アドオンのcert-managerを使用して、KubernetesのIngressのLet's Encrypt（だけでなく）からの証明書の注文と更新を自動化する方法について説明します。しかし、問題の本質の簡単な紹介から始めましょう。

ちょっとした教育プログラム


前世紀の90年代初頭に開発されたHTTPプロトコルは、日常生活に浸透しすぎて、少なくとも1日を使用せずに想像することは困難です。ただし、それだけでは、ユーザーとWebサーバー間で情報を交換するときに最低レベルのセキュリティさえ提供しません。 HTTPS（「S」-セキュア）が助けになります。SSL/ TLSで送信データのパッケージを使用することで、このプロトコルは情報を傍受から保護することで証明され、業界によって積極的に宣伝されています。

たとえば、Googleは2014年を遵守しています「HTTPS everywhere」の位置になり、検索結果に含まれていないサイトの優先度が低下します。この「宣伝」は、通常の消費者も迂回しません。最新のブラウザーは、訪問したサイトのSSL証明書の存在と正確さについてユーザーに警告します。





個人サイトの証明書の費用は数十ドルから始まります。常にそれを購入することは正当化され、適切であるとは限りません。幸いにも、2015年の終わり以来、Let's Encrypt（LE）証明書の形式で無料の代替が利用可能です。この非営利プロジェクトは、ほとんどのWebサイトを暗号化してカバーするために、Mozilla愛好家によって作成されました。

認証局がドメイン検証済みの証明書を発行する（市場で入手可能なものの中で最も単純）有効期間は90日で、長年にわたり、いくつかのサブドメインに対していわゆるワイルドカード証明書を発行することが可能でした。

証明書を取得するために、サイトでは、Let's Encrypt専用に作成された自動証明書管理環境（ACME）プロトコルで説明されているアルゴリズムを使用します。これを使用する場合、ドメイン所有権の確認は、特定のHTTPコード（HTTP-01と呼ばれる）の配置またはDNSレコード（DNS-01）のインストールを介した要求によって実行されます。詳細については、以下で説明します。

認定マネージャー


CERT-マネージャは CustomResourceDefinitionsのセット（したがってあるKubernetesための特別プロジェクト、ある制限 CA構成（認証局）証明書との直接発注するための- V1.12 K8Sのサポートされる最小バージョンで）。クラスターへのCRDのインストールは簡単で、1つのYAMLファイルを使用することになります。

kubectl create ns cert-manager
kubectl apply -f https://github.com/jetstack/cert-manager/releases/download/v0.13.0/cert-manager.yaml

（Helmを使用してインストールする可能性もあります。）

注文手順を開始するには、認証機関（CA）のリソースをクラスターで宣言する必要があります：IssuerまたはClusterIssuer、CSR（証明書発行リクエスト）の署名に使用されます。最初のリソースと2番目のリソースの違いは範囲内です。

  • Issuer 同じ名前空間内で使用でき、
  • ClusterIssuer グローバルクラスタオブジェクトです。

cert-managerで練習する


1番。自己署名証明書


最も簡単なケースから始めましょう-自己署名証明書を注文します。このオプションは、たとえば、開発者向けの動的テスト環境や、SSLトラフィックを終了する外部バランサーを使用する場合など、非常に一般的です。

リソースIssuerは次のようになります。

apiVersion: cert-manager.io/v1alpha2
kind: Issuer
metadata:
  name: selfsigned
spec:
  selfSigned: {}

また、証明書を発行するには、リソースの説明が必要です。リソースのCertificate発行場所（issuerRef以下のセクションを参照）と秘密キー（フィールドsecretName）の場所が示されます。その後、Ingressはこのキーを参照する必要があります（セクションtlscを参照spec）。

---
apiVersion: cert-manager.io/v1alpha2
kind: Certificate
metadata:
  name: selfsigned-crt
spec:
  secretName: tls-secret
  issuerRef:
    kind: Issuer
    name: selfsigned
  commonName: "yet-another.website"
  dnsNames:
  - "yet-another.website"
---
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: app
spec:
  tls:
  - hosts:
    - "yet-another.website"
    secretName: tls-secret
  rules:
  - host: "yet-another.website"
    http:
      paths:
      - path: /
        backend:
          serviceName: app
          servicePort: 8080

これらのリソースをクラスターに追加してから数秒後に、証明書が発行されます。コマンドの出力で対応するレポートを確認できます。

kubectl -n app describe  certificate selfsigned-crt
...
  Normal  GeneratedKey  5s    cert-manager  Generated a new private key
  Normal  Requested     5s    cert-manager  Created new CertificateRequest resource "selfsigned-crt-4198958557"
  Normal  Issued        5s    cert-manager  Certificate issued successfully

シークレットリソース自体を見ると、次のものが含まれています。

  • 秘密鍵tls.key、
  • ルート証明書ca.crt、
  • 自己署名証明書tls.crt。

これらのファイルの内容はopenssl、たとえば次のようなユーティリティを使用して確認できます。

kubectl -n app get secret tls-secret -ojson | jq -r '.data."tls.crt"' | base64 -d | openssl x509 -dates -noout -issuer
notBefore=Feb 10 21:01:59 2020 GMT
notAfter=May 10 21:01:59 2020 GMT
issuer=O = cert-manager, CN = yet-another.website

一般的なケースでは、これを使用して発行された証明書は、Issuer接続されているクライアントから信頼されないことに注意してください。理由は単純です。CAがないためです（プロジェクトのWebサイトの注を参照）。これを回避するにはCertificate、それが含まれている秘密ファイルへのパスを指定する必要がありますca.crt。これは、企業のCA組織である可能性があります。他のサーバーサービス/情報システムのニーズにすでに使用されているキーを使用して、イングレスに発行された証明書に署名します。

2番。HTTP検証で証明書を暗号化しましょう


LE証明書を発行するには、前述のように、HTTP-01とDNS-01の2種類のドメイン所有権確認を使用できます。

最初のアプローチ（HTTP-01）は、別個のデプロイメントで小さなWebサーバーを起動することです。これhttp://<YOUR_DOMAIN>/.well-known/acme-challenge/<TOKEN>により、要求された一部のデータが認証サーバーからリンクを介してインターネットに送信されます。したがって、この方法は、ポート80での外部からのイングレスの可用性と、パブリックIPアドレスへのドメインのDNSレコードの解決を意味します。

発行された証明書（DNS-01）を確認する2番目のオプションは、APIからドメインレコードをホストするDNSサーバーへ。発行者は、示されたトークンを使用して、ドメイン上にTXTレコードを作成します。これは、ACMEサーバーが確認中に受信します。正式にサポートされているDNSプロバイダーには、CloudFlare、AWS Route53、Google CloudDNSなど、独自の実装（acme-dns）が含まれます。

注：Let's Encryptには、ACMEサーバーへのリクエストに対してかなり厳しい制限があります。長時間禁止にならないようにするために、デバッグにはletsencrypt-staging証明書タイプを使用することをお勧めします（違いはACMEサーバーのみです）。

したがって、リソースについて説明します。

apiVersion: cert-manager.io/v1alpha2
kind: Issuer
metadata:
  name: letsencrypt
spec:
  acme:
    server: https://acme-staging-v02.api.letsencrypt.org/directory 
    privateKeySecretRef:
      name: letsencrypt
    solvers:
    - http01:
       ingress:
         class: nginx
---
apiVersion: cert-manager.io/v1alpha2
kind: Certificate
metadata:
  name: le-crt
spec:
  secretName: tls-secret
  issuerRef:
    kind: Issuer
    name: letsencrypt
  commonName: yet-another.website
  dnsNames:
  - yet-another.website

ステージングサーバーのアドレスはservery acme（c Issuer） として指定されることに注意してください。後でバトルが可能になります。 この構成を適用して、注文パス全体を追跡します。



  1. 作成Certificateにより新しいリソースが生成されましたCertificateRequest：

    kubectl -n app describe certificate le-crt
...
Created new CertificateRequest resource "le-crt-1127528680"
  2. 彼の説明では-創造の印Order：

    kubectl -n app describe certificaterequests le-crt-1127528680
…
Created Order resource app/le-crt-1127528680-1805948596
  3. これは、Orderテストパスをパラメータとどのような現在のステータスが何であるかを説明します。このような検証は、リソースによって実行されますChallenge。

    kubectl -n app describe order le-crt-1127528680-1805948596
…
Created Challenge resource "le-crt-1127528680-1805948596-1231544594" for domain "yet-another.website"
  4. 最後に、このリソースの詳細には、スキャン自体のステータスに関する情報が含まれています。

    kubectl -n app describe challenges le-crt-1127528680-1805948596-1231544594
...
  Reason:      Successfully authorized domain                                                                                                                                                                      
...
  Normal  Started         2m45s  cert-manager  Challenge scheduled for processing
  Normal  Presented       2m45s  cert-manager  Presented challenge using http-01 challenge mechanism
  Normal  DomainVerified  2m22s  cert-manager  Domain "yet-another.website" verified with "http-01" validation

すべての条件が満たされている場合（つまり、ドメインが外部からアクセス可能であり、LEからのアクセスが禁止されていない場合...）-1分足らずで、証明書が発行されます。成功describe certificate le-crtすると、出力にレコードが表示されますCertificate issued successfully。

これで、サーバーアドレスを戦闘（https://acme-v02.api.letsencrypt.org/directory）に安全に変更し、すでに署名されているのではなくFake LE Intermediate X1、実際に署名された実際の証明書を並べ替えることができますLet's Encrypt Authority X3。

これを行うには、最初にリソースを削除する必要がありますCertificate。そうでない場合、証明書がすでに存在し、関連しているため、注文手順はアクティブになりません。シークレットを削除すると、次のメッセージがすぐに返されますdescribe certificate。

  Normal  PrivateKeyLost  44s                   cert-manager  Lost private key for CertificateRequest "le-crt-613810377", deleting old resource

それはのために、「戦闘」マニフェストを適用するために残ってIssuer、既に上述したものCertificate（それは変わっていません）：

apiVersion: cert-manager.io/v1alpha2
kind: Issuer
metadata:
  name: letsencrypt
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    privateKeySecretRef:
      name: letsencrypt
    solvers:
    - http01:
       ingress:
         class: nginx

メッセージを受け取ったらCertificate issued successfully、describeそれを確認します。

kubectl -n app get secret tls-secret -ojson | jq -r '.data."tls.crt"' | base64 -d | openssl x509 -dates -noout -issuer
notBefore=Feb 10 21:11:48 2020 GMT
notAfter=May 10 21:11:48 2020 GMT
issuer=C = US, O = Let's Encrypt, CN = Let's Encrypt Authority X3

ナンバー3。DNS検証付きのワイルドカードLE


証明書をすぐにサイトのすべてのサブドメインに書き込み、今回はDNS検証（CloudFlare経由）を使用して、タスクを複雑にします。

まず、APIを介して作業するためにCloudFlareコントロールパネルでトークンを取得します。

  1. プロファイル→APIトークン→トークンの作成。
  2. 次のように権限を設定します。
    • 権限：
      • ゾーン-DNS-編集
      • ゾーン-ゾーン-読み取り
    • ゾーンのリソース：
      • 含める-すべてのゾーン
  3. 保存後に受け取ったトークンをコピーします（例:) y_JNkgQwkroIsflbbYqYmBooyspN6BskXZpsiH4M。

このトークンが格納されるシークレットを作成し、発行者で参照します。

apiVersion: v1
kind: Secret
metadata:
  name: cloudflare-api-token
type: Opaque
stringData:
  api-token: y_JNkgQwkroIsflbbYqYmBooyspN6BskXZpsiH4M
---
apiVersion: cert-manager.io/v1alpha2
kind: Issuer
metadata:
  name: letsencrypt
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory 
    privateKeySecretRef:
      name: letsencrypt
    solvers:
    - dns01:
        cloudflare:
          email: my-cloudflare-acc@example.com
          apiTokenSecretRef:
            name: cloudflare-api-token
            key: api-token

---
apiVersion: cert-manager.io/v1alpha2
kind: Certificate
metadata:
  name: le-crt
spec:
  secretName: tls-secret
  issuerRef:
    kind: Issuer
    name: letsencrypt
  commonName: yet-another.website
  dnsNames:
  - "yet-another.website"
  - "*.yet-another.website"

（実験を行う場合は、ステージングを使用することを忘れないでください！）

ドメインの所有権を確認するプロセスを見ていきましょう。

  Normal  Presented  53s   cert-manager  Presented challenge using dns-01 challenge mechanism

TXTレコードがパネルに表示されます



。...しばらくすると、ステータスが次のように変わります。

Domain "yet-another.website" verified with "dns-01" validation

証明書がすべてのサブドメインで有効であることを確認します。

kubectl -n app get secret tls-secret -ojson | jq -r '.data."tls.crt"' | base64 -d | openssl x509 -dates -noout -text |grep DNS:
          DNS:*.yet-another.website, DNS:yet-another.website

ほとんどのDNSプロバイダーには、DNSレコードが変更された瞬間からすべてのプロバイダーのDNSサーバーの実際の更新までに経過した時間を示すデータ更新期間があるため、通常、DNSによる検証はすぐには行われません。ただし、ACME標準は2つの検証オプションの組み合わせも提供します。これを使用して、メインドメインの証明書の受信を迅速化できます。この場合、説明Issuerは次のようになります。

apiVersion: cert-manager.io/v1alpha2
kind: Issuer
metadata:
  name: letsencrypt
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    privateKeySecretRef:
      name: letsencrypt
    solvers:
    - selector:
        dnsNames:
        - "*.yet-another.website"
      dns01:
        cloudflare:
          email: my-cloudflare-acc@example.com
          apiTokenSecretRef:
            name: cloudflare-api-token
            key: api-token            
    - selector:
        dnsNames:
        - "yet-another.website"
      http01:
        ingress:
          class: nginx

この構成を適用すると、2つのリソースが作成されChallengeます。

kubectl -n app describe orders le-crt-613810377-1285319347
…
  Normal  Created  3m29s  cert-manager  Created Challenge resource "le-crt-613810377-1285319347-3996324737" for domain "yet-another.website"                 
  Normal  Created  3m29s  cert-manager  Created Challenge resource "le-crt-613810377-1285319347-1443470517" for domain "yet-another.website"

ナンバー4。Ingressの特別なアノテーションの使用


cert-managerで証明書を作成する直接パスに加えて、ingress-shimと呼ばれるコンポーネントを使用して、明示的にリソースを作成することはできませんCertificate。アイデアは、特別なIngressアノテーションを使用して、証明書に示されているものを使用して証明書が自動的に注文されるというものIssuerです。結果は、おおよそ次のIngressリソースです。

apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt
spec:
  tls:
  - hosts:
    - "yet-another.website"
    secretName: tls-secret
  rules:
  - host: "yet-another.website"
    http:
      paths:
      - path: /
        backend:
          serviceName: app
          servicePort: 8080

正しい操作のためには、ここでは発行者の存在だけで十分です。つまり、エンティティを1つ少なく作成します。

加えて、ある時代遅れ注釈KUBE-LEGO - kubernetes.io/tls-acme: "true"必要- 、Issuerデフォルトのジョブ（またはマネージャコンテナ起動オプションで）ヘルムパラメータを使用して証明書マネージャをインストールするとき。

当社ではこれらのオプションを使用せず、SSL証明書を注文するために使用されるアプローチが不透明であるため（そして同時に発生するさまざまな問題のため）、これらのオプションをアドバイスすることはできませんが、より完全な図のために記事で言及することにしました。

結論の代わりに


CRDを使用した簡単な操作を通じて、KubernetesクラスターのIngressの一部として起動されたウェブサイトドメインのLet's Encryptプロジェクトから、自動更新可能な自己署名の無料のSSL証明書を書き込む方法を学びました。

この記事では、私たちの実践で最も一般的な問題を解決する例を示します。ただし、cert-managerの機能は、上記の機能に限定されません。ユーティリティのWebサイトでは、他のサービスとの連携の例を見つけることができます。たとえば、Vaultとのバンドルや外部発行センター（発行者）の使用などです。
