Netflixでのローカリゼーションの仕組み-翻訳

こんにちは、ハブル！Netflixチームが内部ローカリゼーションプロセスとこのために特別に開発したプログラムについて書いた資料「Localization Technologies at Netflix」の翻訳を紹介します。



Netflixのローカリゼーションプログラムは、3つの原則に基づいています：非の打ちどころのない言語学、チームの調和のとれた雰囲気、高度なテクノロジー。

私たちは、ローカリゼーションで一般に受け入れられている規範に反対するために、新しいプロセスやツールを試したり試したりすることを恐れていません。これのおかげで、これまでのところここまで来ました！Netflixで働くということは、パイオニアであることを意味します。

この記事では、世界制覇につながる2つのテクノロジーについてお話します。

Netflixグローバル文字列リポジトリ


Netflixは、高品質のコンテンツを作成するためではなく、そのコンテンツを配信するために成功しませんでした。成功のほとんどは、直感的で使いやすく、ローカライズされたユーザーインターフェイス（UI）です。Netflixはさまざまなプラットフォームで利用できます：Webバージョン、Apple iOS、Google Android、Sony PlayStation、Microsoft Xbox、Sony TV、Panasonicなど。これらのプラットフォームにはそれぞれ、内部化のための独自の要件があり、これは私たちのチームにとって深刻な課題です。

UIローカリゼーションが必要な例をいくつか示します。

  • 新しい言語を追加
  • 新機能の追加
  • 既存のテキストとデータへの変更

UIのテキスト翻訳は自動化されたプロセスではありません。翻訳中、ローカリゼーションマネージャーは開発チームと協力して、この行またはその行が何に属しているか、どの言語に翻訳する必要があるか、いつまでにローカライズされたファイルを提供する必要があるかを明確に理解します。複数の機能が並行して開発され、異なるGitブランチで実行される場合は、さらに複雑になります。

翻訳が完了すると、アプリケーションがアセンブル、テストされ、プラットフォームに配置されます。一部のデバイスでは、サードパーティによる確認が必要です（たとえば、Appleから）。これらすべてが、締め切りに望ましくない遅延を引き起こします。特に不快なのは緊急事態の変化の場合です。

しかし、開発チームとローカライザの両方のために、ローカリゼーションプロセスをすべての利害関係者に公開するとどうなるでしょうか。テキストに変更を加えるたびにビルドを再構築する必要がなくなった場合はどうなりますか？

これらの問題を解決するために、グローバル文字列リポジトリと呼ばれるグローバルUI文字列リポジトリを開発しました。ローカライズされた文字列はここに格納され、コード実行のために環境に代入されます。 Global String Repositoryをローカリゼーションプロセスに統合したため、相互に補完し合っています。

グローバル文字列リポジトリは、ローカリゼーションパッケージと名前空間（プレースホルダー）を分離します。ローカリゼーションパッケージは、すべてのデータをすべての言語で1行ずつ格納します。プレースホルダーは、チームが作業しているパッケージのプレースホルダーです。開発時には、標準のプレースホルダーが使用されます。ワークフローは次のようになります。

  1. 開発者は、パッケージ内の（プレースホルダー名前空間内の）英語バージョンの文字列を変更します。
  2. 翻訳プロセスが自動的に開始されます
  3. 言語学者が翻訳を完了する
  4. 翻訳者はプレースホルダーキットを利用可能にします

グローバル文字列リポジトリとの統合が行われる場合、アプリケーションの動作には次の2つのタイプがあります。

  • 実行時：UIをすばやく変更できます
  • アセンブリ中：ローカライズのためにグローバル文字列リポジトリを個別に使用し、アセンブリ（ビルド）でデータパケットを使用する

グローバル文字列リポジトリは、REST APIを介してローカライズされたデータへのアクセスを提供することにより、ビルドフェーズ中の統合を可能にします。

Netflix APIを介してグローバル文字列リポジトリを開くため、他のAPIのメタデータと同じスケーリングと要件が適用されます。実行時に統合するアプリケーションの場合、これは重要な部分です。 Netflixをさまざまなデバイスで実行するユーザーは6,000万人いるため、グローバル文字列リポジトリが優先されます。

Netflixと同様に、Global String Repositoryにはマイクロサービスアーキテクチャがあります。マイクロサービスは、3つのAWSリージョンでホストされるJava Webアプリケーション（Apache CassandraおよびElasticSearchで実行される）です。 APIリクエストごとに統計を収集します。

グローバル文字列リポジトリインターフェイスは、Node.js、Bootstrap、Backboneで開発され、AWSでホストされます。

ユーザー側では、グローバル文字列リポジトリはREST APIを使用してデータを取得し、組み込みキャッシュを備えたJavaクライアントを提供します。

私たちは長い道のりを歩み、グローバル文字列リポジトリを積極的に開発しているという事実にもかかわらず、私たちは努力する必要があります。これが私たちが現在取り組んでいることです：

  • 数値変数を含む文字列と性別識別子を含む文字列のサポートを開発します
  • 技術ソリューションの堅牢性を開発します
  • スケーリングプロセスの改善
  • さまざまな形式（Android XML、Microsoft .Resxなど）へのエクスポートをサポートしています

グローバル文字列リポジトリはNetflixビジネスドメインに関連付けられていないため、オープンソースソフトウェアとしてリリースする予定です。

ハイドラ


Netflix-さまざまなデバイス/ UIで無数のさまざまな組み合わせで多くのロケールをサポートするグローバルサービス。この場合、手動テストは適切ではありません。以前は、ローカライザとUI開発者のチームが、コンソールからiOSやAndroidまで、さまざまなデバイスですべてを手動でテストしていました。これは、コンテキストとUIへの準拠についてすべての行をチェックした方法です（たとえば、テキストの「トリミング」があった場合）。

しかし、Netflixの哲学は、卓越性を追求することです。このアプローチにより、私たちがしていることを再考することができます。それで、ハイドラが生まれました。

Hydraのタスクは、必要な画面を正確に表示する一意の画面のすべての可能なオプションのカタログを作成することです（検索は、フィルターによって実行されます。たとえば、デバイスとロケールを選択できます）。たとえば、ドイツ語のローカリゼーションのスペシャリストとして、PS3、Webサイト、Androidの未登録ユーザーが通過するパス全体を表示するようにフィルタリングを構成できます。同じ画面を、ユーザーが自分のデバイスで開くペースで表示できます。

Hydraでの画面の操作


Hydraは画面を直接操作しません。それらをカタログ化して表示するのに役立ちます。 Hydraカタログから画面表示を取得するには、UI自動化モデルを使用します。 Jenkins CIを使用すると、サポートされているすべてのロケールでデータ駆動型テストが並行して実行されます。これにより、適切なメタデータ（ページ名、機能領域、UIプラットフォーム、および1つの重要なメタデータ-画面上の固有の定義）を使用してHydraに公開されるスクリーンショットが作成されます。

誤った一致なしに画面の完全なカタログをコンパイルするには、一意の画面定義が必要です。これにより、各画面の画像がそれ自体と比較されるため、長期的にはより多くの画面を比較できます。一意の画面の定義は、UIごとに異なります。ブラウザの場合、これはページ名、ブラウザ、解像度、ローカル環境、および開発環境の組み合わせです。

技術


Hydraは、AWSでホストされるフルスタックウェブアプリケーションです。 Javaバックエンドには2つの主要な機能があります。受信スクリーンショットを処理し、REST APIを介してバックエンドにデータを提供します。



UIオートメーションが画面をHydraに送信すると、画像ファイル自体がS3に書き込まれ、無限のストレージ（プラスまたはマイナス）が確保されます。さらに小さなメタデータがRDSデータベースに書き込まれるため、REST APIを介して後でリクエストできます。 RESTエンドポイント（RESTエンドポイント）は、MySQLクエリのクエリ文字列パラメーターを提供します。

例：

REST/v1/lists/distinctList?item=feature&selectors=uigroup,TVUI;area,signupwizard;locale,da-DK

このリクエストには、データベースから必要なデータを選択するためのパラメーターが含まれています。

select distinct feature where uigroup = ‘TVUI’ AND area = ‘signupwizard’ AND locale = ‘da-DK’

knockout.jsを使用するJavaScriptフロントエンドにより、ユーザーはフィルターを選択し、それらのフィルターに一致する画面を表示できます。フィルターの内容と、選択したフィルターに対応する画面は、上記のRESTエンドポイントへの呼び出しによって提供されます。

画面が変わったことを通知する関数を実装する必要があります。現時点では、行が変更されても、これについて自動的に通知されるものはありません。 Hydraは多かれ少なかれ作業キューになり、ローカリゼーションの専門家はシステムにログインして、変更された特定の一連の画面のみを確認できます。

もう1つの機能は、画面の表示先となるキーの個々の行をマップする機能です。これにより、翻訳者は行を変更し、キー検索を実行して、この変更の影響を受ける画面を確認できます。そのため、翻訳者は、この行がコンテキスト内でどのように変化するかを事前に確認できます。

複雑な問題を恐れずに解決します。Netflixはグローバルサービスになり、ローカリゼーションチームが拡大します。このような課題により、私たちは最も才能のある人々を引き付けることができ、不可能と考えられることを実行できるチームを作ります。
