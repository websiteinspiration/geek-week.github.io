依存性注入の形而上学



依存性注入は、コンポーネント接続を減らすように設計されたオブジェクト指向プログラミングで一般的に使用される手法です。正しく使用すると、この目標を達成するだけでなく、アプリケーションに真に魔法のような品質をもたらすことができます。他の魔法と同様に、この手法は一連の呪文として認識されており、厳密な科学論文ではありません。これは、現象の誤解につながり、結果として、アーティファクトの誤用につながります。筆者の資料では、読者がオブジェクト指向設計の適切な基礎から自動依存性注入の魔法まで、論理的方法を簡潔かつ本質的に段階的に進めることをお勧めします。

この資料は、前回の記事で触れたHypo IoCコンテナーの開発に基づいています。ミニチュアコードの例では、短い例を書くための最も簡潔なオブジェクト指向言語の1つとしてRubyを使用します。これは、他の言語の開発者が理解するのに問題を引き起こすべきではありません。

レベル1：依存関係の逆転の原則


オブジェクト指向パラダイムの開発者は、毎日オブジェクトの作成に直面しており、そのオブジェクトは他のオブジェクトに依存している可能性があります。これは、依存関係グラフにつながります。次の形式のオブジェクトモデルを扱っているとします。

-一部の請求サービス（InvoiceProcessor）および通知サービス（NotificationService）。請求処理サービスは、特定の条件が満たされたときに通知を送信します。このロジックは対象外とします。原則として、このモデルは、個々のコンポーネントがさまざまな責任を負うという点ですでに優れています。問題は、これらの依存関係の実装方法にあります。よくある間違いは、この依存関係が使用される依存関係を初期化することです：

class InvoiceProcessor
  def process(invoice)
    #     
    notificationService = NotificationService.new
    notificationService.notify(invoice.owner)
  end
end

これは、論理的に独立したオブジェクトの高い接続性（高結合）が得られるという事実を考慮した場合の誤りです。これは単一責任原則の違反につながります-依存オブジェクトは、その直接の責任に加えて、その依存関係を初期化する必要があります。また、依存関係コンストラクターのインターフェースを「知っている」ため、変更の理由がさらに追加されます（「変更の理由」、R。Martin）。依存オブジェクトの外部で初期化された、この種の依存関係を渡す方がより適切です。

class InvoiceProcessor
  def initialize(notificationService)
    @notificationService = notificationService
  end
  
  def process(invoice)
    @notificationService.notify(invoice.owner)
  end
end

notificationService = NotificationService.new
invoiceProcessor = InvoiceProcessor.new(notificationService)

このアプローチは、依存関係の逆転の原則と一致しています。これで、メッセージ送信インターフェイスを使用してオブジェクトを転送しています-通知サービスオブジェクトの作成方法を「知る」ために課金サービスを使用する必要がなくなりました。請求書処理サービスの単体テストを作成する場合、開発者は、通知サービスインターフェイスの実装をスタブに置き換える方法について悩む必要はありません。Rubyなどの動的型付けの言語では、notifyメソッドを満たす任意のオブジェクトを置き換えることができます。C＃/ Javaなどの静的型付けでは、モックを簡単に作成できるINotificationServiceインターフェイスを使用できます。依存関係の逆転の問題は、Alexander Bendyu（アレクサンドル）最近10周年を迎えた記事で！

レベル2：関連オブジェクトのレジストリ


依存関係の逆転の原則を使用することは、複雑な練習のようには見えません。しかし、時間の経過とともに、オブジェクトと関係の数が増えるため、新しい課題が生じます。NotificationServiceは、InvoiceProcessor以外のサービスでも使用できます。さらに、彼自身は他のサービスに依存している可能性があり、それが次にサードサービスに依存している、などです。また、一部のコンポーネントは単一のコピーで常に使用されるとは限りません。主なタスクは、「依存関係を作成するタイミング」という質問に対する答えを見つけることです。
この問題を解決するには、依存関係の連想配列に基づいてソリューションを構築してみます。彼の作業のインターフェースの例は次のようになります。

registry.add(InvoiceProcessor)
  .depends_on(NotificationService)

registry.add(NotificationService)
  .depends_on(ServiceX)

invoiceProcessor = registry.resolve(InvoiceProcessor)
invoiceProcessor.process(invoice)

これを実際に実装することは難しくありません。


container.resolve（）が呼び出されるたびに、依存関係インスタンスを作成するファクトリに戻り、レジストリに記述されている依存関係グラフを再帰的にバイパスします。`container.resolve（InvoiceProcessor）`の場合、次のようになります。

  1. factory.resolve（InvoiceProcessor）-ファクトリは、レジスターのInvoiceProcessor依存関係を要求し、NotificationServiceを受信します。これもアセンブルする必要があります。
  2. factory.resolve（NotificationService）-ファクトリは、レジスタでNotificationService依存関係を要求し、ServiceXを受け取ります。ServiceXもアセンブルする必要があります。
  3. factory.resolve（ServiceX）-依存関係はありません。作成し、呼び出しスタックに沿ってステップ1に戻り、InvoiceProcessorタイプのアセンブルされたオブジェクトを取得します。

各コンポーネントは他の複数のコンポーネントに依存している可能性があるため、「デザイナパラメータを結果の依存関係インスタンスと正しく一致させる方法」が問題になります。例：

class InvoiceProcessor
  def initialize(notificationService, paymentService)
    # ...
  end
end

静的型付けを使用する言語では、パラメーター型をセレクターとして使用できます。

class InvoiceProcessor {
  constructor(notificationService: NotificationService, paymentService: PaymentService) {
    // ...
  }
}

Ruby内では、規則を使用できます。snake_case形式でタイプ名を使用するだけです。これは、予期されるパラメーター名になります。

レベル3：依存関係のライフタイム管理


私たちはすでに優れた依存関係管理ソリューションを手に入れました。その唯一の制限は、呼び出しごとに依存関係の新しいインスタンスを作成する必要があることです。しかし、コンポーネントのインスタンスを複数作成できない場合はどうなるでしょうか。たとえば、データベースへの接続のプール。より深く掘り下げ、依存関係の制御された存続期間を提供する必要があるかどうか？たとえば、HTTP要求の完了後にデータベースへの接続を閉じます。

そして、論理的な解決策は、一連の戦略（Strategy、GoF）を使用してコンポーネントのインスタンスを取得することです。 Container :: resolveを呼び出すときに常に新しいインスタンスを作成するわけではないため、ファクトリの名前をResolverに変更することが適切です。 Container :: registerメソッドには新しいパラメーター-life_time（lifetime）があることに注意してください。このパラメーターはオプションです。デフォルトでは、その値は「transient」（一時的）で、以前に実装された動作に対応しています。シングルトン戦略も明白です。これを使用すると、コンポーネントのインスタンスが1つだけ作成され、毎回返されます。
スコープは少し複雑な戦略です。 「一時的なパス」と「長い」の代わりに、中間にあるもの、つまり別のコンポーネントの存続期間を通じて存在するコンポーネントを使用する必要があることがよくあります。同様の例は、HTTPアプリケーション、データベース接続、モデル集計などのオブジェクトの存在のコンテキストであるWebアプリケーション要求オブジェクトです。リクエストの存続期間を通じて、これらの依存関係を収集して使用します。依存関係が破棄された後は、それらもすべて破棄されることを期待しています。このような機能を実装するには、かなり複雑な閉じたオブジェクト構造を開発する必要があります。

レベル4：依存性注入


これまでは、依存関係のレジストリを決定する方法、および形成された関係のグラフに従ってコンポーネントを作成および破棄する方法について説明しました。そしてそれは何のためですか？これをRuby on Railsの一部として使用するとします。

class InvoiceController < ApplicationController
  def pay(params)
    invoice_repository = registry.resolve(InvoiceRepository)
    invoice_processor = registry.resolve(InvoiceProcessor)
    
    invoice = invoice_repository.find(params[:id])
    invoice_processor.pay(invoice)
  end
end

この方法で記述されるコードは、読みやすく、テスト可能でもなく、柔軟性もありません。 Railsにそのコンストラクターを介してコントローラーの依存関係を強制的に注入することはできません。これはフレームワークによって提供されません。ただし、たとえばASP.NET MVCでは、これは基本レベルで実装されます。自動依存関係解決メカニズムを最大限に活用するには、制御の反転（IoC、制御の反転）技術を実装する必要があります。これは、依存関係を解決する責任がアプリケーションコードの範囲を超え、フレームワークに依存するアプローチです。例を考えてみましょう。
Railsのようなものを一から設計しているところを想像してみてください。次のスキームを実装します。


アプリケーションが要求を受信すると、ルーターはパラメーターを取得し、この要求を処理するように適切なコントローラーに指示します。このようなスキームは、典型的なWebフレームワークの動作を条件付きでわずかに変更してコピーします-IoCコンテナーは依存関係の作成と実装に関与します。しかし、ここで問題が発生します。コンテナ自体はどこに作成されますか？将来のアプリケーションのできるだけ多くのオブジェクトをカバーするために、フレームワークはその操作の非常に早い段階でコンテナを作成する必要があります。明らかに、アプリビルダーアプリ以外に適した場所はありません。また、すべての依存関係を構成するのに最適な場所です。

class App
  #   - ,      .
  def initialize
    @container = Container.new

    @container
      .register(Controller)
      .using_lifetime(:transient) # ,    

    @container
      .register(InvoiceService)
      .using_lifetime(:singleton) # ,    

    @container
      .register(Router)
      .using_lifetime(:singleton) # 
  end

  #     -     ,
  #      .
  def call(env)
    router = @container.resolve(Router)
    router.handle(env.path, env.method, env.params)
  end
end

どのアプリケーションにも、mainメソッドなどのエントリポイントがあります。この例では、エントリポイントは呼び出しメソッドです。このメソッドの目的は、ルーターを呼び出して着信要求を処理することです。エントリポイントは、コンテナを直接呼び出す唯一の場所である必要があります。その瞬間から、コンテナは道端に移動する必要があり、その後のすべての魔法は「内部」で発生するはずです。このようなアーキテクチャー内でのコントローラーの実装は、非常に珍しく見えます。明示的にインスタンス化していないという事実にもかかわらず、パラメーターを持つコンストラクターがあります。

class Controller
  #   .
  #    .
  def initialize(invoice_service)
    @invoice_service = invoice_service
  end

  def create_invoice(params)
    @invoice_service.create(params)
  end
end

環境は、コントローラーインスタンスの作成方法を「理解」します。これは、Webアプリケーションの心臓部に埋め込まれたIoCコンテナーによって提供される依存性注入メカニズムのおかげで可能です。コントローラのコンストラクタで、その操作に必要なすべてのものをリストできるようになりました。主なことは、対応するコンポーネントがコンテナに登録されていることです。次に、ルーターの実装に移ります。

class Router
  #         - 
  #     
  #     .
  def initialize(controller)
    @controller = controller
  end

  def handle(path, method, params)
    #  ""-
    if path == '/invoices' && method == 'POST'
      @controller.create(params)
    end
  end
end

ルーターはコントローラーに依存することに注意してください。依存関係の設定を思い出すと、Controllerは短命なコンポーネントであり、Routerは常に存在します。どうすればいいの？答えは、コンポーネントは外部から見えるため、対応するクラスのインスタンスではないということです。実際、これらはファクトリメソッド（Factoryメソッド、GoF）インスタンスを持つプロキシオブジェクト（Proxy、GoF）です。それらは、割り当てられた戦略に従ってコンポーネントのインスタンスを返します。コントローラーは「一時的」として登録されているため、ルーターはアクセス時に常に新しいインスタンスを処理します。シーケンス図は、おおよその作業メカニズムを示しています。


それら。依存関係の管理に加えて、IoCコンテナーに基づく優れたフレームワークは、コンポーネントの有効期間の正しい管理も担当します。

結論


依存性注入技術は、かなり洗練された内部実装を持つことができます。これは、柔軟なアプリケーションの実装の複雑さをフレームワークのコアに移す代償です。このようなフレームワークのユーザーは、純粋に技術的な側面について心配することはできませんが、アプリケーションプログラムのビジネスロジックの快適な開発により多くの時間を費やすことができます。高品質のDI実装を使用して、アプリケーションプログラマーは最初にテスト可能で十分にサポートされたコードを記述します。Dependency Injectionの実装の良い例は、前回の記事Orthodox Backendで説明したDandyフレームワークです。
