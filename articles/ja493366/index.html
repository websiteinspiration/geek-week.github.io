位置パラメータ


スクリプトと関数では、値が$ args組み込み変数に書き込まれる位置パラメーター（引数）を渡すことができます。この1次元配列は事前の宣言を必要とせず、そのスコープはスクリプトまたは関数に限定されます。たとえば、最も単純なスクリプトを実行します。

Write-Host " :" $args.count
Write-Host ":" $args




関数では、定位置パラメーターは同じ方法で使用されます。

function Print-Args {
    Write-Host " :" $args.count
    Write-Host " 0:" $args[0]
    Write-Host " 1:" $args[1] 
}

Print-Args“ Zero”“ One”

Print-Argsを呼び出すときは、パラメーターの間にコンマを入れないことに注意してください。これは、関数に渡される配列ではなく、1次元配列$ argsに書き込まれる個々の値です-そのスコープは、関数の本体によって制限されます。



上記の方法では、スクリプトまたは関数に任意の数のパラメーターを渡すことができますが、呼び出すときは、それらのシーケンスの順序に従う必要があり、配列のインデックスによってのみアクセスできます-これは必ずしも便利ではありません。

パラメータブロック（）


スクリプトと関数では、名前付きパラメーターを使用する方がはるかに便利です。では前の記事、私たちはそれらを記述するための一つの方法について話しました：

function test ($arg0, ..., $argN)
{
       
}

同様の構文は開発者にはおなじみですが、関数が呼び出されると、パラメーター（存在する場合）はスペースで区切られ、括弧で囲まれません-特定の不協和音が発生します。これはシェル言語の特殊性です：対話モードでシェルを操作する場合、値の間のスペースがはるかに便利です。試験（$ value0）コールは、正しいが、この場合のパラメータは、括弧内の式全体であり、すなわち（$ value0）ではなく$ value0。この方法で渡しても、いくつかのパラメーターは機能しません。test（$ value0、$ value1）を呼び出した結果、関数は1つだけを受け取ります-値が$ value0と$ value1の2つの要素の配列。

Microsoft Corporationでは、Param（）ブロックの使用を推奨しています。この構文はより一般的であり、関数の引数だけでなくスクリプトのパラメーターも指定できます。

param (
    $arg0, $arg1
)

Write-Host $arg0 $arg1




関数の本体では、次のようになります。

function test {
     param (
           $arg0, $arg1
     )
}

関数の引数リストが小さい場合、Param（）ブロックはデザインを散らかすだけですが、多くの場合、コードを読みやすくし、とりわけ優れたプログラミングスタイルの要素になります。

追加のパラメーター属性


関数の引数またはスクリプトのパラメーターを記述する場合、それらの追加の属性を設定できます。最も単純な例は、次のタイプの強制インストールです。

param([int]$arg0)

または

function test ([int]$arg0) {
}

型キャストに加えて、[parameter（）]属性を使用できます。

param(
    [parameter(Argument1=value1, Argument2=value2)]
    $ParameterName
)

その助けを借りて、パラメーターを必須にすることは簡単です。いくつかの属性の同時使用に注意してください-この場合、それらは互いに続きます：

param([parameter(Mandatory=$true)][int]$arg0)

または

function test ([parameter(Mandatory=$true)][int]$arg0) { 
}

または

function test {
          parameter([parameter(Mandatory=$true)][int]$arg0)
}




Positionでは、パラメーターの順序を指定できます（デフォルトでは、宣言の順序に対応しています）。

param(
                    [parameter(Mandatory=$true, Position=0)]
                    [int]
                    $arg0,

                    [parameter(Position=1)]
                    [string]
                    $arg1,

                    [parameter(Position=2)]
                    [array]
                    $arg2
)

[Parameter（）]属性には他の引数があり、その完全なリストはMicrosoftのWebサイトで入手できます。そこには他の属性が記述されており、渡された値を検証したり、正規表現を使用してそれらをチェックしたりできます。次にいくつかを示します。

[Alias（）]はパラメータのエイリアスを設定します。

param(
    [parameter(Mandatory=$true)]
    [alias("ARG","ArgumentName")]
    [string[]]
    $arg0
)

キャスト演算子[string []]は、パラメーター値が文字列配列であることを意味します。

[AllowNull（）]は、必須パラメーターとして$ nullを許可します。

param(
    [parameter(Mandatory=$true)]
    [AllowNull()]
    [string]
    $arg0
)

[AllowEmptyString（）]は、必須パラメーターとして空の文字列を許可します。

param(
    [parameter(Mandatory=$true)]
    [AllowEmptyString()]
    [string]
    $arg0
)

[AllowEmptyCollection（）]は、必須パラメーターとして空の配列を許可します。

param(
    [parameter(Mandatory=$true)]
    [AllowEmptyCollection()]
    [string[]]
    $arg0
)

[ValidatePattern（）]正規表現を使用した検証：

param(
    [parameter(Mandatory=$true)]
    [ValidatePattern("[0-9][0-9][0-9][0-9]")]
    [string[]]
    $arg0
)

[ValidateLength（）]は、文字列パラメーターの長さをチェックします。

param(
    [parameter(Mandatory=$true)]
    [ValidateLength(1,10)]
    [string]
    $arg0
)

パイプラインを介してパラメーターを渡す


最初の記事シリーズ、我々は、パイプラインを介してコマンドレットにデータを転送する可能性について話しました。PowerShellでは、コマンドレットと関数はオブジェクトまたはオブジェクトの配列（ステートメントの結果）を返し、それらも入力で受け取ります。これを確認するには、Get-Helpを使用してコマンドレットの1つを準備します。

Get-Help Stop-Process -Parameter Name




パイプラインを介して、対応する属性が設定されているパラメーター値（ByValueおよび/またはByPropertyName）を受け入れることができます。最初のケースでは、パラメーターのタイプが期待されるものに対応している場合、パラメーターはパイプラインを介して受信したオブジェクトと一致します。2番目の値では、パラメーターは、このパラメーターの名前またはエイリアスに対応する名前を持つ受信オブジェクトのプロパティになります。属性を設定するには、ブール値の引数ValueFromPipelineおよびValueFromPipelineByPropertyNameで[parameter（）]を使用します。デフォルト値は$ falseです。

param(
    [parameter(Mandatory=$true,
    ValueFromPipeline=$true)]
    [string[]]
    $Name
)

または

param(
    [parameter(Mandatory=$true,
    ValueFromPipelineByPropertyName=$true)]
    [string[]]
    $Name
)

ValueFromPipelineByPropertyNameは通常、混乱が生じないようにいくつかのパラメーターを渡す必要がある場合に使用され、引数はValueFromPipelineと同時に使用できます。 

param(
    [parameter(Mandatory=$true,
    ValueFromPipeline=$true,
    ValueFromPipelineByPropertyName=$true)]
    [string[]]
    $Name
)

Write-Host $Name




ご覧のとおり、スクリプトはパイプラインを介してパラメーターを受け取ることもできますが、それでも、上記の属性の実際のアプリケーションは、以下で説明する高度な関数の可能性が高くなります。

関数本体構造


PowerShellでは、関数にステートメントブラケットで囲まれた3つのオプションのコードブロック（Begin、Process、End）を含めることができます。次のようになります。

function test
{
   param()
   begin {}
   process {}
   end {}
}

Beginブロックは最初に実行される最初のブロックであり、パラメーターがパイプラインを介して関数に渡された場合、最初のオブジェクトが処理のために到着する前にコードが開始されます。この場合、Beginブロックの$ _および$ PSItem変数には値が含まれません。明示的に指定されたパラメーターを使用して関数が呼び出された場合、パイプラインからオブジェクトが受信されるのを待つ必要がないため、それらはBeginブロックで使用できます。次に、Processブロックが実行されます。パラメーターがパイプラインを介して渡されると、オブジェクトごとに1つずつ起動されます。明示的に指定されたパラメーターの場合、Processブロックは1回だけ開始されます。関数は、Endブロックを1回実行することで終了します。明らかに、これらの構造の使用は、関数がパイプラインからオブジェクトを受け取ることができる場合にのみ正当化されます。

function test
{

    param(
        [Parameter(ValueFromPipeline)]
        [string[]]
        $Param1,

        [string]$Param2
    )

    begin
    {
        Write-Host " Begin"
        Write-Host "       ( pipeline):" $Param1
        Write-Host "       ( ):" $Param2
    }

    process {
        Write-Host " Process"
        Write-Host "       ( pipeline):" $Param1
        Write-Host "       ( ):" $Param2
    }

    end
    {
        Write-Host " End"
        Write-Host "       ( pipeline):" $Param1
        Write-Host "       ( ):" $Param2
    }
}

「1」、「2」、「3」| テスト-Param2 'four'



[CmdletBinding（）]属性と高度な関数


「高度な」関数（および厳密に言えばスクリプト）を作成するには、[ CmdletBinding（） ] 属性を使用できます。特に、.NET CoreクラスのクラスであるVisual Studioでコンパイルされたバイナリコマンドレットの機能を使用して、高度な関数を定義できます。この属性は主に関数で使用されるため、ここでは停止します。

function <Name>
{
    [CmdletBinding(ConfirmImpact=<String>,
    DefaultParameterSetName=<String>,
    HelpURI=<URI>,
    SupportsPaging=<Boolean>,
    SupportsShouldProcess=<Boolean>,
    PositionalBinding=<Boolean>)]

    Param ()

    Begin{}
    Process{}
    End{}
}

実際、[CmdletBinding（）]は、オプションの引数を渡すことができるコンストラクターを呼び出すことにより、CmdletBindingAttributeクラスの新しいインスタンスを初期化します。彼らの詳細な説明はマイクロソフトのウェブサイトにあります。 CmdletBinding属性を使用すると、高度な関数の追加機能を制御できます：-Confirmおよび-WhatIf（SupportsShouldProcessを介して）、-Force、-Verbose、および-Debugのサポートの追加、および位置パラメーターバインディングの無効化など。さらに、特別なパラメータの使用を分析します。-Force

パラメーターは、さまざまな操作の要求を抑制するために使用されます。

-仮に起動をエミュレートし、このパラメーターなしで関数（コマンド）を実行した結果に関する情報を表示する必要があります。関数が破壊的なアクションを実行できる場合に一般的に使用されます。

Remove-Item C：\ Windows \ notepad.exe -WhatIf



-Confirmは確認を必要とし、関数が破壊的なアクションを実行できる場合にも使用されます。

function Delete-File {
[CmdletBinding(
    ConfirmImpact = 'High',
    SupportsShouldProcess = $true
)]
    param(
        [string]$File,
        [switch]$Force
    )
    if ($Force -or $PSCmdlet.ShouldProcess($File,"Delete file")) {
        Remove-Item $File
    }
}




-WhatIfおよび/または-Confirmを処理するために、ShouldProcessメソッド（SupportsShouldProcess = $ true）が呼び出され、コマンドの実行をプロンプトまたはエミュレートします。 -Force処理を実装するには、最初にIF条件に置きます。最初に、-or演算子の左側にある式がチェックされ、trueの場合、テストが停止します。ShouldProcessメソッドは呼び出されません。また、[CmdletBinding（）]属性で、システムへのコードの影響レベルを決定し、-Confirmパラメーターハンドラーを含む、ConfirmImpact引数を指定しました。この引数には次の値を指定できます。

なしまたは指定なし --Confirmパラメーターが渡された場合でも確認メッセージは表示されません。

低 -この機能はシステムにわずかに影響し、データ損失の重大なリスクを引き起こしません。

中 -中程度の露出で、破壊的なアクションの結果としてデータが失われるリスクはほとんどありません。

高 -コードは、破壊的なアクションの結果としてデータ損失のリスクが高くなります。

PowerShellセッションのデフォルトでは、露出レベルは高と見なされます。現在の値は$ ConfirmPreference変数に格納され、コードがシステムに同じかそれ以上のレベルの影響を与える場合、確認要求が常に表示されます。

オプション-Verboseおよび-Debugデバッグ情報を表示するために必要です。それらの使用は良いプログラミングスタイルと見なされます（Write-Hostは忘れてください。これは高度な関数では必要ありません）。最初のパラメーターは進行状況に関する情報を表示し、2番目のパラメーターは詳細なデバッグ情報を表示します。また、ステップバイステップのコード実行に切り替えることもできます。-Verboseおよび-Debugの動作は次のように定義されています。

function Get-Something {
[CmdletBinding()]
    param()
    if ($PSBoundParameters.Verbose) {$VerbosePreference = "Continue"}
    if ($PSBoundParameters.Debug) {$DebugPreference = "Continue"}
    Write-Verbose "Type some verbose information"
    Write-Debug "Type some debug information"
}

特別なパラメーターを操作するために、$ PSBoundParameters変数を使用しました。デフォルトでは、$ VerbosePreferenceと$ DebugPreferenceの値は「SilentlyContinue」に等しいため、対応するパラメーターが指定されていても、デバッグ情報は表示されず、「Continue」状態に転送する必要があります。

スクリプトモジュールとコマンドレットの作成


独自のコマンドレットの作成を始めましょう。実際、これらは、いわゆるスクリプトモジュール -拡張子が.psm1のテキストファイル。それらは、環境変数PSModulePathで定義されたディレクトリに格納されます。次のコマンドを使用して、それらへのパスを表示できます

。Get-ChildItem Env：\ PSModulePath | Format-Table -AutoSize

標準セットは次のようになります

。C：\ Users \％UserName％\ Documents \ WindowsPowerShell \ Modules
C：\ Program Files \ WindowsPowerShell \ Modules
C：\ Windows \ System32 \ WindowsPowerShell \ v1.0 \ Modules

前のセクションの拡張Delete-File関数を使用してModuleName.psm1ファイルを作成した後、たとえば、C：\ Users \％UserName％\ Documents \ WindowsPowerShell \ Modules \ ModuleNameに保存する必要があります。スクリプトモジュールは別のサブディレクトリに保存する必要があることに注意してください。その名前は、.psm1ファイルのベース名（拡張子なし）と一致します。Import-Module ModuleNameコマンドを実行すると、ユーザーはDelete-File関数を使用できるようになります。これは高度な機能であるため、実用的には同じコマンドレットです。
