如何禁用有关长时间收听音频的危险的警告（Android）

可能，许多从Android设备（不仅是）听音乐的人都遇到了以下警告：


在本文中，我们将研究发生此警告的原因和时间，以及如何确保不再发生此警告。

仅当通过外部设备（耳机/扬声器）收听音频时，它才会出现。对于尚未满足此要求的人，请作一些解释：假设您正在用耳机听音乐，声音很大。突然，声音变得更安静。您正在尝试使用机箱上的按钮调高音量，但无法正常工作。从口袋里拿出设备并解锁，您会看到这样的警告。只有与他达成协议，才可以调高音量。

是的，警告是合理的，但它出现在一个不可预测的时刻，有时是最不适当的：当您在公共交通中/开车/冬天在街上/手脏时，等等。在这些情况下，卸下设备，释放锁，接受警告以及将设备放回原位不方便。并且如果连接了扬声器而不是耳机，则此消息并不完全合适。

为什么会出现


该警告不是该平台作者的倡议。关键是存在WHO-ETU标准的 “安全聆听”（安全聆听）。在欧洲和其他一些国家，其实施是强制性的。该标准描述了您可以听音频的时间（取决于音量），听力损失的风险最小。例如，对于成年人来说，安全的每周声音“剂量”为1.6 Pa 2 h，相当于以83 dB的音量收听20小时。


实作


根据mcc（移动国家/地区代码），可以打开或关闭安全监听模式。这取决于资源的价值R.bool.config_safe_media_volume_enabled。

如果该模式打开，则系统以不安全的音量（大于85 dB）计算收听时间，并将该值定期存储在变量中Settings.Secure.UNSAFE_VOLUME_MUSIC_ACTIVE_MS。当该值达到20小时时，将显示警告。接受警告后，将重置该值并重新开始计数。

这样的实现非常简单，并且没有考虑到用户收听这20个小时的时间：也许几天，或者六个月听6-7分钟（根据标准，这对听力没有威胁） ）

安全的监听逻辑集中在类AudioService.java中，您可以在其中看到相应的字段：

// mMusicActiveMs is the cumulative time of music activity since safe volume was disabled.
// When this time reaches UNSAFE_VOLUME_MUSIC_ACTIVE_MS_MAX, the safe media volume is re-enabled
// automatically. mMusicActiveMs is rounded to a multiple of MUSIC_ACTIVE_POLL_PERIOD_MS.
private int mMusicActiveMs;
private static final int UNSAFE_VOLUME_MUSIC_ACTIVE_MS_MAX = (20 * 3600 * 1000); // 20 hours
private static final int MUSIC_ACTIVE_POLL_PERIOD_MS = 60000;  // 1 minute polling interval

该字段mMusicActiveMs 包含自上次确认对话以来用户以不安全的音量收听的毫秒数。初始值是从变量加载的Settings.Secure.UNSAFE_VOLUME_MUSIC_ACTIVE_MS。在同一变量中，每分钟都会写入一个新的mMusicActiveMs值。

还有一个字段mSafeMediaVolumeState，它包含安全监听系统的当前状态：

  • DISABLED：关
  • ACTIVE：启用，并且已达到收听限制，这意味着您必须先同意警告，然后才能允许用户增加音量
  • INACTIVE：开启，尚未达到限制


// mSafeMediaVolumeState indicates whether the media volume is limited over headphones.
// It is SAFE_MEDIA_VOLUME_NOT_CONFIGURED at boot time until a network service is connected
// or the configure time is elapsed. It is then set to SAFE_MEDIA_VOLUME_ACTIVE or
// SAFE_MEDIA_VOLUME_DISABLED according to country option. If not SAFE_MEDIA_VOLUME_DISABLED, it
// can be set to SAFE_MEDIA_VOLUME_INACTIVE by calling AudioService.disableSafeMediaVolume()
// (when user opts out).
private static final int SAFE_MEDIA_VOLUME_NOT_CONFIGURED = 0;
private static final int SAFE_MEDIA_VOLUME_DISABLED = 1;
private static final int SAFE_MEDIA_VOLUME_INACTIVE = 2;  // confirmed
private static final int SAFE_MEDIA_VOLUME_ACTIVE = 3;  // unconfirmed
private Integer mSafeMediaVolumeState;

检查超出限制的方法如下所示：

private void onCheckMusicActive(String caller) {
   synchronized (mSafeMediaVolumeState) {
       if (mSafeMediaVolumeState == SAFE_MEDIA_VOLUME_INACTIVE) {
           int device = getDeviceForStream(AudioSystem.STREAM_MUSIC);

           if ((device & mSafeMediaVolumeDevices) != 0) {
               sendMsg(mAudioHandler,
                   MSG_CHECK_MUSIC_ACTIVE,
                   SENDMSG_REPLACE,
                   0,
                   0,
                   caller,
                   MUSIC_ACTIVE_POLL_PERIOD_MS);
               int index = mStreamStates[AudioSystem.STREAM_MUSIC].getIndex(device);
               if (AudioSystem.isStreamActive(AudioSystem.STREAM_MUSIC, 0) &&
                   (index > safeMediaVolumeIndex(device))) {
                   // Approximate cumulative active music time
                   mMusicActiveMs += MUSIC_ACTIVE_POLL_PERIOD_MS;
                   if (mMusicActiveMs > UNSAFE_VOLUME_MUSIC_ACTIVE_MS_MAX) {
                       setSafeMediaVolumeEnabled(true, caller);
                       mMusicActiveMs = 0;
                   }
                   saveMusicActiveMs();
               }
           }
       }
   }
}

如何关闭警告


要关闭安全监听，您需要确保mSafeMediaVolumeState在配置阶段为变量分配了一个值DISABLED。

让我们看一下该值的初始设置位置：

private void onConfigureSafeVolume(boolean force, String caller) {
   ...
   boolean safeMediaVolumeEnabled =
           SystemProperties.getBoolean("audio.safemedia.force", false)
                   || mContext.getResources().getBoolean(
                 com.android.internal.R.bool.config_safe_media_volume_enabled);

   boolean safeMediaVolumeBypass =
           SystemProperties.getBoolean("audio.safemedia.bypass", false);

   int persistedState;
   if (safeMediaVolumeEnabled && !safeMediaVolumeBypass) {
       persistedState = SAFE_MEDIA_VOLUME_ACTIVE;
       /*  ,  mSafeMediaVolumeState   ACTIVE,  INACTIVE */
...
   } else {
       persistedState = SAFE_MEDIA_VOLUME_DISABLED;
       mSafeMediaVolumeState = SAFE_MEDIA_VOLUME_DISABLED;
   }

我们看到，除了资源的值之外R.bool.config_safe_media_volume_enabled，还有两个属性可以启用/禁用安全监听系统：audio.safemedia.force和audio.safemedia.bypass。

要禁用警告，您需要在system / build.properties文件中设置audio.safemedia.bypass = true的值。但这需要root权限。如果不是，那么您需要进一步了解并寻找另一种方式。

如何在没有root的情况下禁用警告


让我们看看通过单击“确定”关闭警告对话框并尝试重现此警告时会发生什么：

@Override
    public void onClick(DialogInterface dialog, int which) {
        mAudioManager.disableSafeMediaVolume();
    }

disableSafeMediaVolume实例 方法称为AudioManager。

/**
* Only useful for volume controllers.
* @hide
*/
public void disableSafeMediaVolume() { … }

它带有注释@hide。这意味着尽管有public修饰符，该方法也不会包含在public API中。在Android 9之前，可以使用反射轻松解决此问题。但是，现在仍然可以调用这种方法，但是要借助一个称为double-reflection的技巧：

val audioManager = getSystemService(Context.AUDIO_SERVICE) as AudioManager
val getDeclaredMethod = Class::class.java.getDeclaredMethod("getDeclaredMethod", String::class.java, arrayOf<Class<*>>()::class.java)
val disableSafeMediaVolumeMethod = getDeclaredMethod.invoke(AudioManager::class.java, "disableSafeMediaVolume", arrayOf<Class<*>>()) as Method
disableSafeMediaVolumeMethod.invoke(audioManager)

调用以异常结束
java.lang.SecurityException: Only SystemUI can disable the safe media volume: Neither user 10307 nor current process has android.permission.STATUS_BAR_SERVICE.
。STATUS_BAR_SERVICE权限具有protectionLevel =“签名|特权”，将不起作用。

好吧，那就试试这个。我们将监视Settings.Secure.UNSAFE_VOLUME_MUSIC_ACTIVE_MS定期存储当前值的变量mMusicActiveMs。当值开始接近20小时时，我们将重置它。然后，您需要使AudioService从设置中读取新值。

您Settings.Secure.UNSAFE_VOLUME_MUSIC_ACTIVE_MS可以像这样读取值：

val unsafeMs = Settings.Secure.getInt(contentResolver, "unsafe_volume_music_active_ms")

使用adb的方法相同：

 adb shell settings get secure unsafe_volume_music_active_ms

并写入值，应用程序将需要权限android.permission.WRITE_SECURE_SETTINGS。

它具有protectionLevel =“签名|特权| 开发 ”，这意味着可以使用adb将其返回给应用程序：

adb shell pm grant com.example.app android.permission.WRITE_SECURE_SETTINGS

值本身可以这样写：

Settings.Secure.putInt(contentResolver, "unsafe_volume_music_active_ms"
, 1)

您可以使用adb进行相同操作：

adb shell settings put secure unsafe_volume_music_active_ms 1

最好将其重置为1，就像在AudioManager中所做的一样，但不要重置为0。因为0对应于ACTIVE状态。

该方法带有注释@hide。两次反射调用会引发异常：
java.lang.SecurityException：权限拒绝：用户的get / set设置要求以用户-2的身份运行，但正在从用户0调用；这需要android.permission.INTERACT_ACROSS_USERS_FULL
是的，@hide这里的注释也不是偶然。当然，我们无法获得此许可。它具有protectionLevel =“签名|安装程序”。

还有一种方法可以使AudioService读取新值-重新启动它。您不能只重启系统服务。您需要重新启动设备，或切换到另一个用户，然后再返回。

现在是检验理论的时候了。

安装unsafe_volume_music_active_ms = 71 990 000 （仅剩10秒，在此期间您可以大音量收听音乐）

adb shell settings put secure unsafe_volume_music_active_ms 71990000


我们重新启动设备（您可以改为切换到另一个用户，然后返回）：

adb reboot

我们连接耳机，然后大声打开音乐。一分钟后会出现对话。

现在，我们重复相同的操作，但分配unsafe_volume_music_active_ms =1。打开音乐，等待一分钟。该对话框不出现。
