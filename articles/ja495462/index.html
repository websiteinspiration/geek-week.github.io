.NET Core：仮想マシンのx86_64組み込み関数

私たちはx86アーキテクチャが支配的な時代に生きています。すべてのx86互換プロセッサは似ていますが、すべて少し異なります。製造元、周波数、コア数だけではありません。

x86アーキテクチャは、その存在（および人気）の間に、多くの主要な更新（たとえば、64ビットへの拡張-x86_64）と「拡張命令セット」の追加を経験しています。デフォルトでは、すべてのプロセッサーに可能な限り共通のコードを生成するコンパイラーも、これに適応する必要があります。しかし、拡張された命令の中には、多くの興味深い便利なものがあります。たとえば、チェスプログラムでは、ビットを操作するための命令がよく使用されます：POPCNT、BSF / BSR（またはより最近のアナログTZCNT / LZCNT）、PDEP、BSWAPなど。

CおよびC ++コンパイラでは、そのような命令への明示的なアクセスは、「このプロセッサの組み込み関数」を通じて実装されます。example1 example2

.NETとC＃にはそのような便利なアクセスがなかったため、そのような関数のエミュレーションを提供する独自のラッパーを作成したことがありましたが、CPUがそれらをサポートしている場合は、呼び出しコードで直接呼び出しを置き換えました。幸い、私が必要とするほとんどの組み込み関数は、CALLオペコードの5バイトに配置されていました。詳細については、このリンクのハブを参照してください。

それから何年も経ちましたが、.NETには通常の組み込み関数は登場しませんでした。しかし、状況が修正された.NET Coreが登場しました。最初にベクター命令が登場し、次にほぼすべての* System.Runtime.Intrinsics.X86のセットが完成しました。
*-「古くなった」BSFとBSRはなく

、すべてが便利で便利なようです。命令の各セットのサポートの定義は常に混乱していることを除いて（いくつかは個別のフラグがあるため、いくつかはセットにすぐに含まれています）。そのため、.NET Coreは、「許可された」セット間にいくつかの依存関係も存在するという事実により、さらに混乱しました。

これは、KVMハイパーバイザーを備えた仮想マシンでコードを実行しようとしたときに発生しましたSystem.PlatformNotSupportedException: Operation is not supported on this platform at System.Runtime.Intrinsics.X86.Bmi1.X64.TrailingZeroCount(UInt64 value)。エラーが発生しました。 System.Runtime.Intrinsics.X86.Popcnt.X64.PopCountの場合も同様です。しかし、POPCNTの場合、仮想化のパラメーターにかなり明白なフラグを設定することが可能であれば、TZCNTは私を行き止まりに導きました。次の図では、netcoreの組み込み機能（記事の最後にあるコードとバイナリ）とよく知られているCPU-Zの可用性をチェックするツール

ご覧のように、フラグInstructionSet_BMI1はまだ設定されています（他のいくつかは設定されていません）。

リポジトリでこのフラグを探すと、次のコードに出くわす可能性があります。

if (resultflags.HasInstructionSet(InstructionSet_BMI1) && !resultflags.HasInstructionSet(InstructionSet_AVX))
    resultflags.RemoveInstructionSet(InstructionSet_BMI1);

だから、彼女は私たちの中毒です！AVXが定義されていない場合、BMI1（および他のいくつかのセット）は無効になります。論理とは何なのか、まだはっきりしていませんが、それがまだ存在していることを願っています。cpu-zやその他のツールがAVXを認識する理由を理解する必要がありますが、netcoreは認識しません。
