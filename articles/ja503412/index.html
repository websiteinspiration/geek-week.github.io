Project Loom：Java仮想スレッドは間近です

数日前、Ron PresslerがState of Loomの記事を出産しました。記事は本当に良いです、そこには興味深いメタファーがたくさんあります。私はソースを参照せずに今ここで不用意に使用します。

私の側では、不注意でいくつかの懐疑論を表明することを許可しましたが、このプロジェクトで織機はようやく本当に取り組むことができます。文字通り1時間後、ロン自身から返事が来た-「そしてあなたは試す！」。えーと、試してみました。

実験に必要なもの：

  • Project Loomを含むJDK15
  • ターミナル、IntelliJとGradleがそのようなゲームでの作業を完全に拒否したため

まず、Kotlin Coroutineのドキュメントからのストリームを使用した古典的な例がどのようになるかを確認することにしました。この例はKotlinで記述されていますが、Javaで書き直すことは難しくありません。

public class Main {
    public static void main(String[] args) {
        var c = new AtomicLong();
        for (var i = 0; i < 1_000_000; i++) {
            new Thread(() -> {
                c.incrementAndGet();
            }).start();
        }

        System.out.println(c.get());
    }
}

最初に、前の例と同様に、例がまだハングしていることを確認します。

javac Main.java && java Main

次に、Project Loomが提供する仮想スレッドを使用して例を書き直します。

for (var i = 0; i < 1_000_000; i++) {
    Thread.startVirtualThread(() -> {
        c.incrementAndGet();
    });
}

for (var i = 0; i < 1_000_000; i++) {

しかし、これはすでに興味深いです！Project Loomでは、Thread.sleep（）の呼び出しで、通常のスレッドと仮想スレッドのどちらであるかを区別でき、動作も異なります。

これ自体はとてもクールです。しかし、もう少し深く掘り下げましょう。

var threads = new ArrayList<Thread>();
var cores = 10;
for (var i = 0; i < cores; i++) {
    var t = Thread.startVirtualThread(() -> {
        var bestUUID = "";
        for (var j = 0; j < 1_000_000; j++) {
            var currentUUID = UUID.randomUUID().toString();
            if (currentUUID.compareTo(bestUUID) > 0) {
                bestUUID = currentUUID;
            }
        }
        System.out.println("Best slow UUID is " + bestUUID);
    });
    threads.add(t);
}

for (var i = 0; i < cores; i++) {
    var t = Thread.startVirtualThread(() -> {
        var bestUUID = UUID.randomUUID().toString();
        System.out.println("Best fast UUID is " + bestUUID);
    });
    threads.add(t);
}

for (Thread t : threads) {
    t.join();
}

ここでは、10個の低速タスクと10個の高速タスクを実行します。速いタスクは遅いタスクよりも100万回速いので、早く完了すると考えるのが理にかなっています。

直感は、遅いタスクの数がCPUのコアの数よりも少ない限り機能します。

その理由は簡単です。現時点では、Project Loomは通常のForkJoinPoolを使用しています。その結果、ドキュメントとアイデアが仮想ストリームが「模範的」であることを示しているという事実にもかかわらず、現時点では、それらは「協調的」な方法で動作します。コトリンのコルーチンのように、実際には。

前述の記事で、ロンは、通常のスレッドの場合と同様に、強制的な予測動作についても考察していることに言及していることに注意してください。しかし、これまでのところ、これは実装されていません。何万ものスレッドが存在する可能性がある場合に、そのような動作がどのように役立つかは完全には明らかではないためです。ただし、Go 1.14では、強制的な偽装が静かに導入されました。

Goとは異なり、関数呼び出しではコンテキストスイッチは発生しません。Kotlinと同様に、Suspendも配信されませんでした。ただし、Thread.yield（）を使用するか、Java IO関数（System.out.println（ ""）など）を呼び出すことで実行できます。

計算は簡単です。ほとんどの実際のプログラムは、ブロッキングIOを頻繁に使用します。そして、そもそも解決しようとするのは、彼がProject Loomを使用することです。
