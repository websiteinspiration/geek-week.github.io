Lacmusプロジェクト：コンピュータービジョンが失われた人々を救う方法

みなさん、こんにちは！

オープンデータサイエンスコミュニティのソーシャルグッドの機械学習（＃ml4sg）イニシアチブについては、すでにご存じかもしれません。そのフレームワーク内で、愛好家は機械学習の方法を使用して社会的に重要な問題を無料で解決します。私たちLacmusプロジェクトチーム（#proj_rescuer_la）は、人口密集地域の外（森や野外など）で迷っている人々を見つけるための最新のディープラーニングソリューションの実装に取り​​組んでいます。



概算によると、毎年ロシアでは10万人以上が亡くなっています。それらの具体的な部分は、人間の居住から遠く離れた人々です。幸いなことに、失われたものの中には自分で選ばれたものもあり、ボランティアの捜索救急隊は他の人々を助けるために動員されています。おそらくそのような分遣隊の中で最も有名なのはリサアラートでしょうが、私は彼だけではないことに注意したいと思います。

21世紀の現在の主な探索方法は、技術的な手段を使用して徒歩で周囲をとかすことです。多くの場合、サイレンや騒々しい灯台ほど複雑ではありません。もちろん、このトピックは関連性があり、ホットであり、科学的および技術的進歩を利用して成果を追求するための多くのアイデアを生み出します。それらのいくつかは、プロトタイプの形で具体化され、特別に組織されたコンテストでテストされます。しかし、森は森であり、限られた資料のリソースと相まって、調査の実際の条件はこの問題を困難にし、完全な解決策からはまだ遠く離れています。

最近、救急隊員は無人航空機（UAV）を使用して地域の広い範囲を調査し、40-50mの高さから地形を撮影しています。1回の捜索救急活動で数千枚の写真が得られ、これまでボランティアが手作業で調べてきました。このような処理が長く非効率的であることは明らかです。そのような作業の2時間後、ボランティアは疲れて検索を続けることができなくなり、結局のところ、人々の健康と生活はその速度に依存しています。

UAVで撮影した画像で行方不明者を捜索するためのプログラムを開発し、捜索救助チームと協力しています。機械学習の専門家として、検索を自動的かつ高速にするよう努めています。

UAVを使用したすべての検索操作で数千枚の写真が表示される場合、蓄積されたデータの配列は膨大なものになるはずです。次の理由により、すべてがそれほど単純であるとは限りませんでした。

  • タグ付きデータ用の集中ストレージはありません。検索操作中に撮影された写真は、今後使用および処理されません。
  • 得られたデータは非常に不均衡です。人物が見つかった1枚のショットには、数千枚の「空白」の写真があります。スキャンした画像に関する情報はどこにも記録されないため、必要なものを見つけるために、「訓練された目」を持たない小さなチームの努力によって、膨大な量の作業を2回行う必要があります。

したがって、トレーニングサンプルの量と質を向上させる代わりに、ニューラルネットワークのトレーニングに貴重な情報を一切使用せず、ディスクやクラウドストレージのどこかで失われたり死んだりします。私たちは、とりわけ貴重な写真を私たちにアップロードすることを可能にするサービスを書いています（それについても以下にあります）が、いつものように、人よりも多くのタスクがあります。

さらに、これまでのところ、ネットワークにはUAVからの画像を含む優れた（オープン）データセットがほとんどありません。私たちが見つけた最も適切なものは、スタンフォードドローンデータセット（SDD）です。。大学構内の高さからの写真で、「Pedestrian」（歩行者）というクラスのマークされたオブジェクトと、サイクリスト、バス、車が写っています。同じような撮影角度にもかかわらず、写真に写っている歩行者とその周辺は、私たちの写真で起こっていることとほとんど共通点がありません。このデータセットに対して行われた実験は、私たちのデータに基づいてトレーニングされた検出器の品質メトリックが低い結果を示すことを示しました。その結果、トップレベルの属性を抽出する、いわゆるバックボーンをトレーニングするためにSDDを使用するようになり、ドメイン領域の画像で極端なレイヤーを完成させる必要があります。

そのため、森で迷子になった人が上からの写真にどのように見えるかを理解するために、さまざまな検索エンジンや救助者と長い間連絡を取り合いました。その結果、行方不明者が最も多く見られる24種類のポーズについて、ユニークな統計を収集しました。私たちは撮影し、独自のデータセットであるLacmus Drone Dataset（LaDD）をマークアウトしました。最初のバージョンでは、400以上の画像が含まれていました。撮影は主にDJI Mavic ProとPhantomの助けを借りて50から100メートルの高さで行われ、写真の解像度は3000x4000、平均的な人物のサイズは50x100 pxでした。現時点では、2千枚の画像を含むデータセットの4番目のバージョンがすでにあり、実際と「シミュレーション」の両方が行われています。データセットの補充作業は継続しており、5番目のバージョンは間もなく始まります。

データセットを補充するにつれ、画像を季節ごとに分ける必要が出てきました。事実は、冬の写真でトレーニングされたモデルは、夏または春のいずれかでデータセット全体でトレーニングされたモデルよりも優れた結果を示すということです。雪に覆われた背景にある標識は、騒々しい草よりも抽出されたほうがよいでしょう。

これらの問題を何らかの形で回避するために、さまざまなトレーニングスキーム、ネットワーク設定、トレーニングサンプルが使用されました。テストしたニューラルネットワークアーキテクチャの1つであるRetinaNetは、大規模なクラスの不均衡による悪影響を正確に低減することを目的としています。 RetinaNetの作成者は、1ステージ検出器（事前定義された長方形アンカーの高密度ネットワークで画像をカバーし、オブジェクトを最もよくカバーするものを精緻化する）の精度を、2ステージ検出器（候補領域を最初に見つけることを検討している検出器）に比べて向上させるように設計しました。 、次にそれらの位置を指定します）。 RetinaNetに関する記事の著者の観点から見ると、多数の空のアンカーによって引き起こされる不均衡のために、1ステージ検出器は正確に失われます。このアドバンテージを背景に、ResNet50バックボーンを備えたRetinaNetを優先して選択しました。

このネットワークのアーキテクチャは2017年に導入されました。RetinaNetの主な機能は、トレーニングにおけるクラスの不均衡の悪影響に対処できるようにするもので、元のFocal Loss損失関数です。

FL(pt)=−(1−pt)γlog(pt)pt={p,if y=1,1−p otherwise


ここで、pは、モデルによって推定された目的のオブジェクトの領域のコンテンツの推定確率です（簡単に言えば、間隔[0、1]に削減された場合のニューラルネットワークの出力）。

他のドメインドメインでは、通常、損失関数は、異常値である可能性が高い非定型のインスタンス（ハードな例）に対して耐性を持つ必要があります。ウエイトトレーニングへの影響を減らす必要があります。逆に、Focal Lossでは、頻繁に発生するバックグラウンド（インライア、簡単な例）の影響が軽減され、RetinaNetウェイトをトレーニングするときに、めったに見られないオブジェクトが最大の影響を与えます。これは、式のこの部分が原因で行われます。

(1−pt)γ


係数 γ指数の γは、全損失関数の厳密な例の「重み」を決定します。

RetinaNetのトレーニング中に、画像スケーリングのすべてのレベルから、候補領域（アンカー）のすべての考慮された方向について損失関数が計算されます。合計すると、1つの画像に約100kの領域があり、これは、ヒューリスティックサンプリング（RPN）や、ミニバッチごとに少数の領域（約256）を選択するレアインスタンスの検索（OHEM、SSD）アプローチとは大きく異なります。焦点損失値は、すべてのアンカーの関数値の合計として計算され、目的のオブジェクトを含むアンカーの数で正規化されます。アンカーの大部分は簡単に識別できるバックグラウンドであり、全体的な損失関数への寄与は小さいため、正規化は総数に対してではなく、それらに対してのみ実行されます。

構造的に、RetinaNetは、バックボーンと2つの追加の分類サブネットおよびボックス回帰サブネット定義で構成されています。



いわゆるFeature Pyramid Networkがバックボーンとして使用されます。、FPNは、一般的に使用される畳み込みニューラルネットワーク（ResNet50など）の上で実行されます。 FPNには、畳み込みネットワークの隠れ層からの追加の横方向出力があり、異なるスケールのピラミッドレベルを形成します。各レベルは「トップダウンパスウェイ」によって補完されます。小さいが、より大きな領域の領域に関する情報を含む、より高いレベルの情報。より「最小化された」フィーチャマップを現在のマップのサイズに（たとえば、単に要素を繰り返すことによって）人為的に増やしたように見え、要素ごとに要素を追加し、それらをピラミッドの下位レベルと他のサブネットの入力（つまり、分類サブネットとBox Regression Subnet）。これにより、元の画像からさまざまな縮尺の標識のピラミッドを選択できます。とても小さいオブジェクト。 FPNは多くのアーキテクチャで使用され、さまざまなサイズのオブジェクトの検出を向上させます：RPN、DeepMask、Fast R-CNN、Mask R-CNNなど。

FPNの詳細については、元の記事をご覧ください。

私たちのネットワークでは、元のように、5レベルのFPN、P3からP7。レベルPlの解像度は2l入力画像よりも 2 l倍小さくなります（ResNetのどのポイントから来たかについては詳しく説明しません-このくそレッグブレークでは）。ピラミッドのすべてのレベルには、同じ数のチャネルC = 256があり、アンカーの数Aは約1000です（画像のサイズによって異なります）。

アンカーには、ピラミッドのレベルごとに[16 x 16]〜[256 x 256]の領域があります。P3からP7それぞれ変位のステップ（ストライド）[8-128] px。このサイズにより、小さなオブジェクトや周囲のいくつかの環境を分析できます。たとえば、枝は、周囲の現実を考慮に入れなければ、うそをついている人に非常に似ています。

元のFPNは、アンカーの3つのアスペクト比（1：2、1：1、2：1）を使用します。 RetinaNetの作成者は、これらのアスペクト比のそれぞれに3つのスケールを追加して、より高密度のカバレッジを提供しました[ ]。各レベルで9種類のアンカーを使用すると、元の画像で長さ/幅が16〜400ピクセルのオブジェクトを見つけることができます。分類サブネットは、指定されたアンカー内のKクラスのそれぞれの存在確率を予測します。実際、これは各FPN層に接続された単純な完全接続ネットワーク（完全にConvNet、FCN）です。パラメータはピラミッドのさまざまなレベルで同じであり、そのアーキテクチャは非常に単純です。20,21/3,22/3

全体で、長さK x Aのベクトルがこのネットワークの出力で形成されます。ここで、Kは異なるクラスの数です。私たちの場合、使用されるクラスは1つだけです-これは歩行者です。

Box Regression Subnetを使用すると、4コンポーネントのアンカー座標ベクトルをオブジェクトの実際のサイズに調整できます。これは、FPNレイヤーのそれぞれに接続された、完全に接続された小さなネットワークであり、分類サブネットとは独立して動作します。それらのアーキテクチャはほとんど同じですが、トレーニング中にサイズ（4 x A）のベクトルが最小化されます-アンカーごとに：

IoU（Intersection over Union）がオブジェクトを含む実際の領域> 0.5の場合、アンカーはオブジェクトを含むと見なされます。この場合yiは1が割り当てられ、それ以外の場合は0が割り当てられます。このアプローチは、検出器のトレーニングの計算コストを削減します。

干渉（画像内のオブジェクトの検出）は、バックボーンと2つのサブネットの転送機能を計算することです。速度を上げるために、0.05を超える予測確率で最良の1k領域でのみ分類が実行されます。検出の最終段階では、ピラミッドのすべてのレベルの領域のみが残り、分類確率はしきい値= 0.5を超えます。

救助者の希望と要望に基づいて、デスクトップアプリケーションを開発しました。必要なライブラリを簡単にインストールできるように、デスクトップアプリケーションでの作業をできるだけ簡略化したいと考えました。 Nvidia CudaとCuDNNのインストールやTensorFlowのカスタムビルドの作成は、平均的なユーザーにとって簡単な作業ではありません。また、UAVパイロットがPythonをラップトップにインストールしたり、依存関係を設定したりすることを心配しないようにしたかったのです。ライブラリのセットアップとインストールの複雑さをすべてユーザーから隠す、コンパクトなものが必要でした。そして、私たちはそのような解決策を見つけました-これはDockerです。コンテナーには、ニューロンを搭載して依存関係をインストールしたローカルの小さなWebサーバーをデプロイします。突然モデルを更新したい場合は、新しいバージョンのDockerイメージをダウンロードする必要があります。また、このようにGUIとモデルを分離しています。そのため、たとえばユーザーが良好な通信チャネルを持っている場合、GUIはローカルサーバーではなくリモートサーバーにアクセスできます。さらに、Dockerは、GUIから直接自動的に制御できる便利なAPIを提供しているため、ユーザーはすべてを透過的に維持できます。さらに、Dockerには独自のリポジトリが用意されており、さまざまなバージョンのイメージを保存でき、モデルを持つサーバーについて心配する必要がありません。

インターネットで検索したところ、dotnext会議でこのレポートを見つけました。 「うーん！これは新しいものです！ある種の非常に若いフレームワークですか？触ってみるのも面白い」と思って負けなかった！その結果、アプリケーションのGUIはAvaloniaUI C＃フレームワークで記述されているため、64ビットオペレーティングシステムのWin10、Linux、Macで実行できます。

AvaloniaUIは若いですが、非常に強力で高速なフレームワークです。その概念はWPFと非常によく似ており、コードを実際に変更することなくアプリケーションをWPFに転送できます。これは高速で効率的で、2Dグラフィックスをより速く描画し、WPFよりも少ないリソースを消費します。元のWPFを強化するいくつかの利点もあります。

内部デバイスと同様に、ここでは、SkiaSharpライブラリを使用してグラフィックスとGTK（UNIXシステム用）をレンダリングします。 X11レンダーの開発も進行中です。これにより、コンソールバッファー（！）内でも、どこでも、どこでもインターフェイスを描画できます。 .Net CoreをBiosで起動できるとすれば、AvaloniaUIはクールなマザーボードと同じようにファッショナブルなゲームインターフェイスをそこに描画します。

AvaloniaUIは人気を博しており、オープンフレームワークですが、まだベータ版であり、このフレームワークにはバグがあります。しかし、開発者は絶えずそれを改善し、補足しており、2019年の終わりの時点で、小規模な商用ソリューションを作成することはすでに可能であると言えます。 WPFとCに精通している場合は、ぜひお試しください。利点は、インターフェース（エレクトロンについては言えない）による低リソース消費にも起因する可能性があります。これは、グリッドに数メガバイトのRAMを獲得することを意味します。
