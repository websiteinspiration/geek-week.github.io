Googleのセキュリティ診断はどのように行われましたか

ユーザーデータのセキュリティを確保するために、Googleは制限付きAPIスコープを使用し、Googleユーザーデータにアクセスできるすべてのアプリケーションを慎重にチェックします。少し前に、Snov.ioで検証プロセスを実行し、共有したいGoogleの承認を得ました。

アプリケーションの新しいルール


2018年10月9日、 Googleは、Googleの制限付きAPIスコープを使用するアプリケーションの新しいルールを発表しました。検証には2つの段階があります。

  • アプリケーションがGoogle APIのユーザーデータに準拠していることを確認する
  • 最小のセキュリティ要件を確認する

制限付きスコープアプリの検証では、Google APIのユーザーデータポリシーと、特定のAPIスコープの追加要件で概説されている一連の追加ポリシーへの準拠を確認します。まず、アプリケーションはGoogle APIサービス：ユーザーデータポリシーへの準拠について審査されます。その後、2019年の残りの期間は、安全な取り扱い要件の遵守を実証する必要があります。

最小のセキュリティ要件への準拠の検証には、15,000ドルから75,000ドルという多額の費用がかかります。
評価は、Googleが指定したサードパーティの評価者によって行われ、アプリケーションの複雑さによって15,000ドルから75,000ドル（またはそれ以上）の費用がかかる場合があり、開発者が支払う必要があります。この料金は、アプリが評価に合格したかどうかにかかわらず必要になる場合があります。

早ければ2019年1月9日、 Googleは、GoogleのAPIを使用する予定のアプリケーションのためのルールを締めます。以前にGoogle APIを使用していたアプリケーションについては、2月15日までに確認のためにアプリケーションを送信する必要がありました。それ以外の場合、2月22日以降、新規ユーザーはアプリケーションにアクセスできなくなり、3月31日以降、既存のユーザーはアプリケーションを使用できなくなります。

そのような開発の結果は不快でしょう。これは、かなりの数のユーザー（10万人以上）がGmailを使用しているためです。したがって、これらの顧客を失うだけです。

アクションが必要なプロジェクトの場合、2019年2月15日までに確認のためにアプリを送信する必要があります。送信しない場合、2019年2月22日に新規ユーザーのアクセスが無効になり、3月31日に一般ユーザー向けアカウントの既存の許可が取り消されます。 2019。

更新前にすべてがどのように行われましたか？


Snov.io プラットフォームは、2017年からGoogle APIを使用しています。私たちのアプリケーションは、いくつかの制限されたスコープを使用しました：受信メールを読むためと、下書きで作業するため。 

Googleは以前に、Google APIを使用するアプリケーションをテストしました。新しいAPIスコープを適用するには、それをコンソールに追加し、それが何のために使用されるかを示す必要がありました。Googleの従業員がアプリケーションを確認している間、ユーザーには画面に「このアプリは確認されていません」という通知が表示されました。 



通常、このチェックには2〜3営業日かかります（ただし、Googleからの書簡では、プロセスが最長7日間続くことが示されていました）。常に問題なく通過しました。ユーザーが「このアプリは確認されていません」という通知が表示されないように、Googleがアプリケーションをチェックするまで待ってから、機能を製品に追加しました。 

初段の通過


まず、検証の最初の段階、つまりアプリケーションがGoogle APIのユーザーデータポリシーに準拠することに焦点を当てることにしました。 

1月16日、 Googleコンソールに[ 確認のため送信 ]ボタンが表示され、アプリケーションが送信されました。出発前に、Google APIのユーザーデータポリシーのすべての点に準拠していることを確認しました。プライバシーポリシーを変更しました。特に、Googleユーザーデータセクションを追加しました。このセクションには、Google APIから受信したデータの種類、使用方法、および削除したときの詳細が記載されています。 

2月12日、提出されたアプリケーションに対する応答を受け取りました。ビデオを録画し、リクエストされた制限付きAPIスコープをどのように使用するかを示すように求められました。 

その結果、Google Cloudでの2つのプロジェクトを放棄し、1つにマージする必要がありました。テストモードで動作するテストサーバーの最初のプロジェクトを放棄し、製品と同じプロジェクトをテストに使用しました。また、Googleを介してシステムに認証するための2番目のプロジェクトを削除し、代わりにログインに検証が必要なプロジェクトを使用しました。 

Googleの代表は、2週間後にどこかで私たちの手紙に答えました。質問については、直接的な回答ではなく、見積もりを受け取りました。彼らには、アプリケーションをチェックする特別なスクリプトがあるように見えました。 

たとえば、1つのIDを持つアプリケーションを使用してシステムにアクセスし、メールアカウントに接続するときに、別のIDを持つアプリケーションを使用することを思い出しました。これらは2つの完全に異なる機能であるため、意図的にこれを行いました。ログインアプリケーションはチェックを必要としませんでした。また、検証アプリケーションを無効にするために制限されたAPIスコープでアプリケーション検証をパスするのに失敗したくありませんでした。



4月20日、ついにGoogleデータポリシーのコンプライアンスチェックの第1段階に合格しました。

セカンドステージの通過 


ステップ1.監査に合格する会社を選択する


アプリケーションのテストの第2段階までに、GoogleはBishop FoxとLeviathan Securityから選択する2つの企業の連絡先を送信しました。彼らだけでチェックにパスすることができました。締め切りは12月31日まででした。

5月20日、検証のために2人の独立した専門家に手紙を送りました。司教フォックスとリヴァイアサンセキュリティは、私たちのアプリケーションに関する質問をアンケートに送信しました。私たちが使用するGoogleデータの種類、各プログラミング言語用に書かれたコードの行数、インフラストラクチャ、ソフトウェアの展開プロセス、ホスティングプロバイダーに関する質問に答える必要がありました。私たちはすべてを記入し、価格の提示を待ち始めました。



会社の担当者との準備および予備的なコミュニケーションの間に、サーバーが配置されているホスティングがSOC2に準拠していないことがわかりました。そして、テストに合格するには、適切なSOC2ホスティングに移行する必要がありました。私たちはずっとAmazonへの移行を考えてきたので、並行して移行するプロセスを開始しました。 

また、多くのWebサイトやソフトウェア開発者が提供するバグバウンティプログラムが必要であることも学びました。これにより、人々はバグ、特にエクスプロイトや脆弱性に関連するものを発見したことに対する認識と報酬を受け取ることができます。 

9月、フォックスビショップとリヴァイアサンセキュリティからの2つの既成の契約がありました。誰と契約を結ぶかを決めなければなりませんでした。エキスパートを選択するための基準はわかりませんでしたが、わずかにコストがかかるという事実にもかかわらず、Leviathan Securityとの合意は私たちにはより透明に見えました。

ステップ2.契約に署名し、検証の準備をする


10月8日に、 Leviathan Securityと契約を結びました。契約締結時点では、まだAmazonへの移行手続きは完了していません。したがって、「ホスティング」列の契約ではギャップがあり、後で入力する必要がありました。

また、検証には次のものが含まれます。

  • 外部ネットワーク侵入テスト 
  • アプリケーション侵入テスト 
  • 導入レビュー 
  • ポリシーと手順のレビュー 

そして次のステップ：

  • 準備 
  • 評価
  • 検証と検証
  • 最終報告

チェックは約1か月続きます。その価格には、見つかった脆弱性を排除するための時間が含まれています。次に、2番目のチェックが実行されます。検証の結果、評価書（LOA）を受け取っているはずですが、それをGoogleの担当者に送る必要があります。ただし、専門家は、評価の100％結果としてLOAを受け取ることを保証しません。 

10月23日、 Leviathan Securityは自己問診（SAQ）を送信しました。彼女と一緒に、監査に合格するために必要なポリシーも提供しました。

  • インシデント対応計画：インシデント発生時の役割、責任、アクションを確立します
  • リスク管理ポリシー：望ましくないインシデントまたは結果を特定、削減、および防止します
  • 脆弱性開示ポリシー：外部の関係者が脆弱性を報告する手段を提供します
  • Information Security Policy: Ensures that all users comply with rules and guidelines related to the security of the information stored digitally at any point in the network
  • Privacy Policy: Ensures that users can delete their accounts and related user data by demonstrating an account deletion if relevant

11月8日、外部調整会議が開催され、組織のすべての問題について話し合い、コミュニケーションのメッセンジャー（Slack）を特定し、アプリケーションについて簡単に説明しました。ソースコードへのアクセスを提供する必要があると警告されました-これは私たちにとって問題ではありませんでした。 

集会では、KMSを使用してOauthトークンを暗号化する必要があることを学びました。チェックのおおよその時間についても説明しました。彼女が年末に任命され、それを通過する時間がなかった場合、Leviathan SecurityはGoogleと交渉して期限を延長することを保証しました。

11月14日12月下旬または1月上旬に予定されている検査の開始に関する情報を受け取りました。また、Leviathan SecurityはGoogleに、期限よりも遅くLOAを提供できると警告しました。 

11月16日、 Googleから締め切りが3月31日まで延期されたという確認を受けました。
