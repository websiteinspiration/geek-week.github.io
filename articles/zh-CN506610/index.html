WAL-G：PostgreSQL数据库备份和恢复

众所周知，在SQL转储中备份（使用pg_dump或pg_dumpall）不是一个好主意。要备份PostgreSQL，最好使用pg_basebackup命令，该命令可以生成WAL日志的二进制副本。但是，当您开始研究创建备份和还原的整个过程时，您将了解到，您至少需要编写两个三轮车，以便所有这些都能起作用，并且不会使您从上至下感到痛苦。为了减轻痛苦，开发了WAL-G。

WAL-G是用Go语言编写的工具，用于备份和还原PostgreSQL数据库（最近更新是MySQL / MariaDB，MongoDB和FoundationDB）它支持使用Amazon S3存储（以及类似的产品，例如Yandex Object Storage）以及Google Cloud Storage，Azure存储，Swift对象存储以及仅与文件系统一起使用。整个设置简化为简单的步骤，但是由于有关它的文章分散在Internet上，因此没有完整的操作方法手册，其中包括往返的所有步骤（哈布雷（Habré）上有几篇文章，但很多要点都在那里）。



撰写本文主要是为了使您的知识系统化。我不是DBA，可以在某个地方用非利士发展语言表示，因此欢迎进行任何更正！

另外，我注意到以下所有内容均与Ubuntu 18.04上的PostgreSQL 12.3相关且已验证，所有命令必须以特权用户身份执行。

安装


在撰写本文时，WAL-G的稳定版本为v0.2.15（2020年3月）。我们将使用它（但是，如果您想自己从master分支进行组装，那么github存储库将提供有关此操作的所有说明）。要下载并安装，您需要执行以下操作：

#!/bin/bash

curl -L "https://github.com/wal-g/wal-g/releases/download/v0.2.15/wal-g.linux-amd64.tar.gz" -o "wal-g.linux-amd64.tar.gz"
tar -xzf wal-g.linux-amd64.tar.gz
mv wal-g /usr/local/bin/

之后，您必须先配置WAL-G，然后配置PostgreSQL本身。

WAL-G设定


作为备份存储的示例，将使用Amazon S3（因为它离我的服务器更近并且使用非常便宜）。要使用它，您需要一个“ s3存储桶”和访问键。

以前有关WAL-G的所有文章都使用环境变量进行配置，但是从此发行版开始，这些设置可以位于postgres用户主目录中的.walg.json文件中。要创建它，请执行以下bash脚本：

#!/bin/bash

cat > /var/lib/postgresql/.walg.json << EOF
{
    "WALG_S3_PREFIX": "s3://your_bucket/path",
    "AWS_ACCESS_KEY_ID": "key_id",
    "AWS_SECRET_ACCESS_KEY": "secret_key",
    "WALG_COMPRESSION_METHOD": "brotli",
    "WALG_DELTA_MAX_STEPS": "5",
    "PGDATA": "/var/lib/postgresql/12/main",
    "PGHOST": "/var/run/postgresql/.s.PGSQL.5432"
}
EOF
#    :
chown postgres: /var/lib/postgresql/.walg.json

我将在所有方面进行一些解释：

  • WALG_S3_PREFIX-将要倒入备份的S3-bucket的路径（可以在根目录中也可以在文件夹中）；
  • AWS_ACCESS_KEY_ID – S3 ( – ReadOnly Policy! );
  • AWS_SECRET_ACCESS_KEY – S3;
  • WALG_COMPRESSION_METHOD – , Brotli ( /);
  • WALG_DELTA_MAX_STEPS – «» ( , , );
  • PGD​​ATA -路径与数据库中的数据的目录（您可以通过运行找出pg_lsclusters命令）;
  • PGHOST-与数据库的连接，使用本地备份，最好通过unix-socket进行连接，如本例所示。

其他参数可以在文档中找到：https : //github.com/wal-g/wal-g/blob/v0.2.15/PostgreSQL.md#configuration。

PostgreSQL设置


为了使数据库内的归档程序将WAL日志上传到云中并从中恢复（如果需要），您需要在配置文件/etc/postgresql/12/main/postgresql.conf中设置几个参数。刚开始时，您需要确保以下设置均未设置为其他任何值，以便在重新启动配置时DBMS不会掉线。您可以使用以下命令添加这些参数：

#!/bin/bash

echo "wal_level=replica" >> /etc/postgresql/12/main/postgresql.conf
echo "archive_mode=on" >> /etc/postgresql/12/main/postgresql.conf
echo "archive_command='/usr/local/bin/wal-g wal-push \"%p\" >> /var/log/postgresql/archive_command.log 2>&1' " >> /etc/postgresql/12/main/postgresql.conf
echo “archive_timeout=60” >> /etc/postgresql/12/main/postgresql.conf
echo "restore_command='/usr/local/bin/wal-g wal-fetch \"%f\" \"%p\" >> /var/log/postgresql/restore_command.log 2>&1' " >> /etc/postgresql/12/main/postgresql.conf

#     SIGHUP    
killall -s HUP postgres

设置参数说明：

  • wal_level-向WAL杂志写多少信息，“副本”-编写所有内容；
  • archive_mode-使用archive_command参数中的命令启用WAL日志的下载；
  • archive_command-用于归档完整的WAL日志的命令；
  • archive_timeout-仅在日志归档完成后才进行归档，但是如果您的服务器稍稍更改/向数据库中添加数据，则在此处设置以秒为单位的限制是有意义的，此后将强制执行归档命令（我每秒都要对数据库进行大量写入，因此我拒绝在Production中设置此参数）；
  • restore_command-用于从备份还原WAL日志的命令，如果“完全备份”（基本备份）中缺少最后的数据库更改，则将使用该命令。

您可以在官方文档的翻译中阅读有关所有这些参数的更多信息：https : //postgrespro.ru/docs/postgresql/12/runtime-config-wal。

设置备份时间表


不管喜欢与否，但是cron是最方便的启动方法。这就是我们将配置用于创建备份的内容。让我们从创建完整备份的命令开始：在wal-g中，这是运行backup-push的参数。但是首先，最好由postgres用户手动执行此命令，以确保一切正常（并且没有访问错误）：

#!/bin/bash

su - postgres -c '/usr/local/bin/wal-g backup-push /var/lib/postgresql/12/main'

开始参数指示数据目录的路径-我提醒您可以通过运行pg_lsclusters来识别它。

如果一切顺利，并且数据已上传到S3存储库，则可以在crontab中配置定期启动：
#!/bin/bash

echo "15 4 * * *    /usr/local/bin/wal-g backup-push /var/lib/postgresql/12/main >> /var/log/postgresql/walg_backup.log 2>&1" >> /var/spool/cron/crontabs/postgres
#       
chown postgres: /var/spool/cron/crontabs/postgres
chmod 600 /var/spool/cron/crontabs/postgres

在此示例中，备份过程每天早上4:15开始。

删除旧备份


最有可能的是，您不需要存储中生代绝对的所有备份，因此定期“清理”您的存储（“完整备份”和WAL日志）将很有用。我们还将通过cron任务完成所有操作：

#!/bin/bash

echo "30 6 * * *    /usr/local/bin/wal-g delete before FIND_FULL $(date -d '-10 days' '+%FT%TZ') --confirm >> /var/log/postgresql/walg_delete.log 2>&1" >> /var/spool/cron/crontabs/postgres
#          (        )
chown postgres: /var/spool/cron/crontabs/postgres
chmod 600 /var/spool/cron/crontabs/postgres

Cron每天早上6:30执行此任务，删除除过去10天的副本外的所有内容（完整备份，增量和WAL），但将在指定日期之前至少保留一个备份，以便该日期之后的任何时间落入PITR 。

从备份还原


保证基础健康的前提是定期恢复并验证内部数据的完整性已不是什么秘密。如何使用WAL-G恢复-我将在本节中介绍，稍后我们将讨论检查。

另外，值得注意的是，要在测试环境（不是生产环境）上进行恢复-您需要在S3中使用只读帐户，以免意外覆盖备份。对于WAL-G，您需要在组策略中为S3用户设置以下权限（效果：允许）：s3：GetObject，s3：ListBucket，s3：GetBucketLocation。并且，当然，不要忘记事先在postgresql.conf设置文件中设置archive_mode = off。这样您的测试基础就不想安静地备份。删除所有PostgreSQL数据（包括用户）后，

只需轻轻移动手指即可进行恢复，因此在运行以下命令时请格外小心。

#!/bin/bash

#     (, pgbouncer),    ,       
service pgbouncer stop
#   ,     (, monit),        (   pgsql12)
monit stop pgsql12
#    
service monit stop
#    
service postgresql stop
#       (!!!);     ,      
rm -rf /var/lib/postgresql/12/main
#      
su - postgres -c '/usr/local/bin/wal-g backup-fetch /var/lib/postgresql/12/main LATEST'
#      -   (. https://postgrespro.ru/docs/postgresql/12/runtime-config-wal#RUNTIME-CONFIG-WAL-ARCHIVE-RECOVERY ),        postgres
su - postgres -c 'touch /var/lib/postgresql/12/main/recovery.signal'
#   ,     
service postgresql start

对于那些想检查恢复过程的用户-下面准备了一小部分bash-magic，以便在恢复问题中-脚本崩溃时返回非零退出代码。在此示例中，以5秒的超时时间（仅恢复10分钟）完成了120次检查，以发现信号文件是否已删除（这意味着恢复成功）：

#!/bin/bash

CHECK_RECOVERY_SIGNAL_ITER=0
while [ ${CHECK_RECOVERY_SIGNAL_ITER} -le 120 ]
do
    if [ ! -f "/var/lib/postgresql/12/main/recovery.signal" ]
    then
        echo "recovery.signal removed"
        break
    fi
    sleep 5
    ((CHECK_RECOVERY_SIGNAL_ITER+1))
done

#        ,    
if [ -f "/var/lib/postgresql/12/main/recovery.signal" ]
then
    echo "recovery.signal still exists!"
    exit 17
fi

成功恢复后，请不要忘记重新启动所有进程（pgbouncer / monit等）。

恢复后的数据验证


确保在恢复后检查数据库的完整性，以免出现备份损坏/弯曲的情况。最好对创建的每个存档都执行此操作，但是在何处以及如何进行取决于您的想象力（您可以每小时增加一台服务器，也可以在CI中运行测试）。但是至少-您需要检查数据库中的数据和索引。

要检查数据，足以在转储中运行它们，但是最好在创建数据库时启用校验和（数据校验和）：

#!/bin/bash

if ! su - postgres -c 'pg_dumpall > /dev/null'
then
    echo 'pg_dumpall failed'
    exit 125
fi

为了检查索引-有一个amcheck模块，我们将从WAL-G测试中获取sql查询，并围绕以下内容构建一些逻辑：

#!/bin/bash

#  sql-       
cat > /tmp/amcheck.sql << EOF
CREATE EXTENSION IF NOT EXISTS amcheck;
SELECT bt_index_check(c.oid), c.relname, c.relpages
FROM pg_index i
JOIN pg_opclass op ON i.indclass[0] = op.oid
JOIN pg_am am ON op.opcmethod = am.oid
JOIN pg_class c ON i.indexrelid = c.oid
JOIN pg_namespace n ON c.relnamespace = n.oid
WHERE am.amname = 'btree'
AND c.relpersistence != 't'
AND i.indisready AND i.indisvalid;
EOF
chown postgres: /tmp/amcheck.sql

#          
# (       – )
cat > /tmp/run_amcheck.sh << EOF
for DBNAME in \$(su - postgres -c 'psql -q -A -t -c "SELECT datname FROM pg_database WHERE datistemplate = false;" ')
do
    echo "Database: \${DBNAME}"
    su - postgres -c "psql -f /tmp/amcheck.sql -v 'ON_ERROR_STOP=1' \${DBNAME}" && EXIT_STATUS=\$? || EXIT_STATUS=\$?
    if [ "\${EXIT_STATUS}" -ne 0 ]
    then
        echo "amcheck failed on DB: \${DBNAME}"
        exit 125
    fi
done
EOF
chmod +x /tmp/run_amcheck.sh

#  
/tmp/run_amcheck.sh > /tmp/amcheck.log

#         exit code  grep’ 
if grep 'amcheck failed' "/tmp/amcheck.log"
then
    echo 'amcheck failed: '
    cat /tmp/amcheck.log
    exit 125
fi
