2020年にJavaに何を期待しますか？

2020年はすでに本格化しています。今年はJavaの世界のどのような変化が待ち受けていますか。この記事では、JavaとJDKの主なトレンドをリストします。そして、私はコメントで読者からの追加に満足します。

すぐに、その記事が事実を明らかにする性格のものであることを予約してください。議論された各トピックの詳細は、対応するプロジェクトのWebサイトまたはオープンソースの出版物に記載されています。



それでは始めましょう。残念ながら、Javaのリリースサイクルにあまり従わないが、長いサポートプログラム（LTS）を待っている人はすぐに失望するでしょう。今年は短いサポートライフサイクル（STS）でのリリースを待っています。
最初に、3月中旬にリリースされるJDK 14の次期リリースを検討します。このリリースサイクルでは、最大16のJEPが要求されます。完全なリストは次のとおりです。

instanceofのパターンマッチング（プレビュー）


長いJEPがついにプレビューで登場します。何年もの間Javaコードを書いている実践的なプログラマーなら、この苦痛に何度か遭遇していると思います。

if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.toUpperCase());
}

Kotlinでもコードを作成または作成している場合は、Javaコードを表示することによる痛みが少し悪化します。アンバープロジェクトの参加者は、Javaでのパターンマッチングのビジョンを提示します。これにより、この痛みが軽減されます。Java 14の登場により、次のように例を書き直すことができます。

if (obj instanceof String s) {
   System.out.println(s.toUpperCase());
}

アドオンはそれほど価値がないようです-コードの行を保存します。しかし、次のことを実行するとします。

if (obj instanceof String) {
    String s = (String) obj;
    if (s.contains(“prefix_”)) {
       return s.toUpperCase();
    }
}
return null;

かさばって見えませんか？同じことを試してみましょうが、パターンマッチングを使用します。

return (obj instanceof String s) && s.contains(“prefix_”) ? s.toUpperCase() : null;

だから明らかにそれはより良いでしょう。ただし、この機能のステータスはプレビューです。時間とともに変化するものを見てみましょう。私にとって、それは間違いなく私の人生を良くするでしょう。

役立つNullPointerExceptions


2020年が間近に迫っていますが、NullPointerExceptionsが飛ぶように書いていますか？心配しないでください、あなただけではないでしょう。Goetz LindenmaierとRalf Schmelterは、NullPointerExceptionsを回避するための新しい方法を提案していません（オプションはまだあります）が、nullがどこにあるかを正確に理解するためにアプリケーションのデバッグプロセスを改善することを提案しました。したがって、コードを5時（もちろん夜間）に作成するとします。そして、私たちはこの関数を書きました：

public String getStreetFromRequest(Request request) {
   return request.getAddress().getStreet().toUpperCase();
}

悪くはありませんが、@ Nullableアノテーションと@Nonnullアノテーションを配置し、送信されたフィールドのアドレスを確認するのを完全に忘れていました。NullPointerExceptionが発生しました。例外は私たちに何を伝えますか？

Exception in thread "main" java.lang.NullPointerException
	at Program.getStreetFromRequest(Program.java:10)
	at Program.main(Program.java:6)

残念ながら、行、クラス、スタックのみが表示されます。nullはどこに戻りましたか？多分これはリクエストですか？たぶんgetAddress（）はnullを返しましたか？または多分getStreet（）？まあ、呼び出しチェーンは時々傷つきます。

JEPの作成者は解決策を提案します。例外をスローする場合、スタックをバイパスしてnullが返された場所を正確に判断し、変数/メソッドの名前を表示することになっています。-XX：+ ShowCodeDetailsInExceptionMessagesオプションを指定してJava 14を試してみましょう。始めて少し違います。

Exception in thread "main" java.lang.NullPointerException: Cannot invoke "String.toUpperCase()" because the return value of "Address.getStreet()" is null
	at Program.getStreetFromRequest(Program.java:10)
	at Program.main(Program.java:6)

これで、夜間のプログラミングはうまくいかないことがわかりました（ただし、タスクが予定どおりに完了することもあります）。プログラムでは、アドレスが必須フィールドではないことを忘れていました。

レコード（プレビュー）


まだgetter / setter / equals / hashCodeを生成していますか？その後、このJEPがあなたのところにやって来ます！
データクラスは、アプリケーションソフトウェア開発者の人生の最後の場所からはほど遠いです。お気に入りのIDEを使用してDataクラスのメソッドを生成するか、さまざまなコンパイル時プラグインを使用してlombokなどの必要なメソッドを生成する必要があります。

その結果、次のようなコードがたくさんあります。

public class Request {
    private Address address;

    public Address getAddress() {
        return address;
    }

    public void setAddress(Address address) {
       this.address = address;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Request request = (Request) o;
        return Objects.equals(address, request.address);
    }

    @Override
    public int hashCode() {
        return Objects.hash(address);
    }

    @Override
    public String toString() {
        return "Request{" +
                "address=" + address +
                '}';
    }
}

またはそのような：
@Data
@AllArgsConstructor
public class Request {
    private Address address;
}

Java 14では、Amberプロジェクトのメンバーがデータクラスを作成するための新しい構文を提案しています。これを行うには、新しいキーワードレコードを使用します。Recordの構文は、クラスの説明や列挙型の構文とは少し異なり、Kotlinと少し似ています。上記のコードは次のように書き換えることができます。

public record Request(Address address) {
}

すべてのレコードフィールドには、デフォルトでプライベート修飾子と最終修飾子があります。レコード自体は最終クラスであり、別のクラスから継承することはできませんが、インターフェースを実装できます。ボックスからのレコードクラスでは、gettersメソッド、パブリックコンストラクターがあり、そのパラメーターはすべて説明順にレコードフィールドであり、/ hashCodeとtoStringに等しくなります。不愉快なことから：クラス名の後に指定されたものを除いて、レコードにフィールドを追加することはできません。たとえば、次のコードはエラーになります。

public record Request(Address address) {
   private final String anotherParameter; // compilation error
}

テキストブロック（2番目のプレビュー）


テキストブロックはプレビューとしてJava 13にリリースされました。ユーザーはねじれ、向きを変え、フィードバックを返しました（プレビューでJava 13を既に使用していると私は心から信じています）。その結果、Java作成者はテキストブロックに小さな変更を加えました。まず、行にescapesequence \ sを挿入することで、行の終わりを明示的に示すことができます。次に例を示します。

public static void main(String[] args) {
        final var test = """
                This is the long text block with escape string \s
                that is really well done            \s
                """;
        System.out.println(test);
}

ここで、すべてのスペースを\ s文字に明示的に設定すると、すべてのスペース文字が\ s文字に保存されます。

次に、最後の行で\ n文字を受け取ることなく、テキストブロックの長い行を折り返すことができます。これを行うには、改行で\を追加するだけです。それはどのように見えますか：

public static void main(String[] args) {
        final var test = """
                This is the long text block with escape string \
                that is really well-done functional            
                """;
System.out.println(test);

実行後、次の行を取得します。「これは、実際にうまく機能しているエスケープ文字列を含む長いテキストブロックです」。

良い追加、それは私には思えます。この機能を標準に変換できることを本当に楽しみにしています。
私たちが検討したすべての機能は、今後の会議で広く議論される可能性があります。それらのいくつかはすでにJoker 2019で議論されています。CayHorstmannによるこのトピック「Java 13以降の機能の進化」に関するJoker 2019の講演を必ずチェックしてください。

そして、もう少し興味深いもの。


インキュベーターのJEPリストには2つの興味深い項目があります。まず、さまざまなオペレーティングシステム用のインストーラーを作成するユニバーサルツールを用意します（最後に、Windowsへのプログラムのインストールに慣れ親しんだ人たちに伝えたいと思います）。 Jpackerは、Windows用のmsi / exeインストーラー、macOSパッケージ、およびLinux用のrpm / debを作成できます。これから何が起こるのかを見てみましょう。しかし、デスクトップ用に何かをしたときに、インストーラーを組み立てるための通常のツールを持っていないという個人的な問題に私は苦しみました。

さらに有望なのは、「Foreign-Memory」にアクセスするための新しいAPIです。あらゆる種類のネイティブまたは永続メモリ。このAPIは、主にJavaデータベースの作成者や、Nettyなどのフレームワークの作成者に役立ちます。それらは、UnsafeとByteBufferを使用して、可能な限りオフヒープでメモリアクセスを最適化します。

リストには何が表示されますか？

最初に目を引くのはJSON APIです。問題はすぐに発生します-なぜですか？明確な答えは明らかにありません。動機に関するJEPのセクションでは、JSONはWebサービスの標準の1つになりました。JSONと対話するためにJava SEを適応させるときが来ました（JSONを解析するためのライブラリがたくさんありますが）。最も可能性の高い説明は、ソフトウェア開発者が小さなコアAPIを使用して、重いJacksonを自分自身にドラッグする必要なしにバンドルのサイズを縮小できることです。データバインディングさえもないので、他の説明はありません。

また、暗号化APIに関連する多くの改善も見られます。はじめに、JDK開発者はEVSSL証明書のサポートを追加して、SSL証明書の検証プロセスを拡張したいと考えています。このAPIをJavaで使用すると、SSL接続がEV証明書（拡張検証）によって信頼されているかどうかを判別できます。ガイドラインに従ったEVSSL証明書は完全にサポートされます。新しいEdDSA暗号化アルゴリズムも追加され、HSM暗号化の検証が改善されます。

言語のことから、プリミティブでのジェネリックスの実装を選び出します。 C＃でプログラミングしてJavaに切り替えたことがある人なら誰でも、おそらくプリミティブでジェネリック型を実行できないのかと疑問に思うでしょう。答えは簡単です-ジェネリックスはオブジェクトでのみ機能し、プリミティブはオブジェクトではなく、特にObjectクラスを継承しません。この問題で戦争が始まったのは最初の年ではなく、ブライアン・ゲッツは再びそれに戻ってきています。これまで説明する特別なことは何もありません。タスクは明確です。リストなどの構造をサポートすることです。しかし、現時点でも、この機能を実装する前に解決する必要がある13の未解決の問題があります。正直なところ、このシリーズはどのように終わるのでしょうか。

そして最後に触れておきたいのは封印されたタイプです。これは、パターンマッチングに向けた次のステップです。封印された型は、封印された（修飾子）を実装し、クラスまたはインターフェースのキーワードを許可する言語拡張です。

封印されたクラスを使用して、子の数を、許可（明示的な制限）または同じコンパイル単位（ファイル）で指定されたクラスのみに制限します。シールドクラスの説明の例：

// 
public abstract sealed class Base {
   public static class ChildA extends Base{}
   public static class ChildB extends Base{}
}

// 
public sealed interface BaseInterface permits ChildC, ChildD{
}

//  
public class ChildC implements BaseInterface {
}
//  
public class ChildD implements BaseInterface {
}

シールド修飾子は、子孫の限られた有限のセットのみが基本クラスを拡張したり、インターフェースを実装したりできるようにします。このプロパティは、これらのクラスのオブジェクトを処理するときに使用できます。そしてもちろん、これはパターンマッチングを使用したswitchステートメントで使用するのに最適です。

結論


今年は、JDKのさまざまなイノベーションを見ました。射撃する人もいれば射撃しない人もいます。しかし、何よりも新しいJDKでは、すべてのリリースで無料でプログラムを高速化する新しい（またはそれほどではない）最適化を期待しています。最後のJoker 2019にいて、Tagir Valeev Java 9-14：小さな最適化のレポートにアクセスした場合、おそらく、私のように、貢献者がJDKを最適化するために行った作業に感銘を受けました。その結果は一見して見えず、複数のJEPに反映されていませんが、毎日使用しています。

すべての人に良いJavaリリース。新しいプラットフォーム機能を探索し、カンファレンスに行き、トレンドをフォローしてください。
