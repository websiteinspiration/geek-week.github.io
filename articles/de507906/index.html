Wir verwenden DI in Angular maximal - das Konzept der privaten Anbieter

Angular verfügt über einen sehr leistungsstarken Abhängigkeitsinjektionsmechanismus. Sie können damit alle Daten über Ihre Anwendung übertragen, konvertieren und in den richtigen Teilen neu definieren.



Daher können wir die Anwendungsarchitektur einfacher und flexibler gestalten: klarer Datenfluss, minimale Codekohärenz, einfaches Testen oder Ersetzen von Abhängigkeiten.



Trotzdem wird DI in Anwendungen recht bescheiden eingesetzt. Dies ist in der Regel die Einführung von Diensten oder die Übertragung einiger globaler Daten von oben nach unten durch den Abhängigkeitsinjektionsbaum.



In diesem Artikel möchte ich eine alternative Arbeitsweise mit Daten zeigen, die von DI stammen. Zweck: Vereinfachung der Komponenten, Anweisungen und Dienste, die diese Daten verwenden.







Wie DI häufig in Angular verwendet wird



Ich führe tägliche Überprüfungen von Angular-Code bei der Arbeit und in Open Source durch. In den meisten DI-Anwendungen kommt es normalerweise auf die folgenden Funktionen an:



  1. Abrufen von Angular-Entitäten aus dem Abhängigkeitsbaum: ChangeDetectorRef, ElementRef usw.
  2. Lassen Sie den Dienst in der Komponente verwenden.
  3. Holen Sie sich eine globale Konfiguration für das Token, das irgendwo oben deklariert ist. Legen Sie beispielsweise das API_URL-Token im Root-Modul fest und rufen Sie es bei Bedarf an einer beliebigen Stelle in der Anwendung vom DI ab.


Weniger häufig sind Fälle, in denen Entwickler weiter gehen und ein vorhandenes globales Token in eine bequemere Form umwandeln. Ein gutes Beispiel für eine solche Konvertierung ist das WINDOW-Token aus dem Paket @ ng-web-apis / common .



Angular stellt das DOKUMENT-Token bereit, damit Sie das Seitenobjekt von überall in der Anwendung abrufen können: Ihre Komponenten sind unabhängig von globalen Objekten, einfach zu testen, nichts wird mit SSR beschädigt.



Wenn Sie regelmäßig Zugriff auf das WINDOW-Objekt benötigen, können Sie dieses Token schreiben: 



import {DOCUMENT} from '@angular/common';
import {inject, InjectionToken} from '@angular/core';

export const WINDOW = new InjectionToken<Window>(
    'An abstraction over global window object',
    {
        factory: () => {
            const {defaultView} = inject(DOCUMENT);

            if (!defaultView) {
                throw new Error('Window is not available');
            }

            return defaultView;
        },
    },
);


Wenn jemand das WINDOW-Token zum ersten Mal vom DI-Baum anfordert, wird die Token-Factory ausgeführt. Er erhält ein DOCUMENT-Objekt von Angular und erhält von ihm einen Link zum Fensterobjekt.



Ferner schlage ich vor, einen anderen Ansatz für solche Transformationen in Betracht zu ziehen - wenn sie direkt bei den Anbietern der Komponente oder Direktive durchgeführt werden, die das Ergebnis einspeisen.



Private Anbieter



In unserem Team verwenden wir DI aktiv, wenn wir mit Angular arbeiten, und stellten fest, dass wir häufig einige Transformationen der vom DI empfangenen Daten vornehmen müssen, bevor wir sie verwenden. Tatsächlich benötigt unsere Komponente einige Daten, und wir binden andere darin ein und führen die darin enthaltene Transformationslogik aus.







Schauen wir uns gleich ein solides Beispiel an. Erin Coglar zeigte in ihrem Bericht The Architecture of Components auf einer großen internationalen Konferenz Angular Connect ein Beispiel:





Wenn es für Sie nicht bequem ist, das Video zu öffnen und anzusehen, unterschreiben wir den Fall hier.



Wir haben:



  • Die Komponente, die für die Anzeige von Informationen zu einer bestimmten Entität verantwortlich ist - die Organisation.
  • Abfrageparameter der Route, der die ID der Organisation angibt, mit der wir gerade arbeiten.
  • Ein Dienst, der anhand seiner ID ein Observable mit Informationen zur Organisation zurückgibt.


Was möchten wir tun?



Nehmen Sie die Organisations-ID aus den Abfrageparametern, übergeben Sie sie an die Servicemethode und rufen Sie als Antwort einen Stream mit Informationen über die Organisation ab. Zeigen Sie diese Informationen in der Komponente an.



Betrachten Sie drei Möglichkeiten, um das gewünschte Ziel zu erreichen, und analysieren Sie sie.



Wie es geht ist nicht notwendig



Manchmal stoße ich auf diese Art der Arbeit mit Daten in Komponenten. Bitte mach das nicht: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent implements OnInit {
   organization: Organization;

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}

   ngOnInit() {
       this.activatedRoute.params
           .pipe(
               switchMap(params => {
                   const id = params.get('orgId');

                   return this.organizationService.getOrganizationById$(id);
               }),
           )
           .subscribe(organization => {
               this.organization = organization;
           });
   }
}


So verwenden Sie die empfangenen Daten in der Vorlage: 



<p *ngIf="organization">
   {{organization.name}} from {{organization.city}}
</p>

Lass es uns gut machen.



Erins Bericht aus dem Video, das ich oben angewendet habe, ist gut gemacht. Mit seiner Variante stellt sich so etwas heraus: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent {
   readonly organization$: Observable<Organization> = this.activatedRoute.params.pipe(
       switchMap(params => {
           const id = params.get('orgId');
           return this.organizationService.getOrganizationById$(id);
       }),
   );

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}
}


So verwenden Sie die empfangenen Daten in der Vorlage: 

<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


Dieser Code funktioniert einwandfrei und ist frei von den Nachteilen des vorherigen Ansatzes: Er sieht ordentlich aus und enthält keine zusätzlichen Felder. Wenn wir die Komponente mit einem ähnlichen Stream erweitern möchten, fügen wir einfach einen weiteren hinzu - um dies zu vereinfachen, da zum Hinzufügen eines neuen Streams der Code des vorherigen Streams nicht berührt werden muss.



Außerdem wird der Datenstrom transparenter: Wir haben nur den Strom, der beim Erstellen der Komponentenklasse erstellt wird. Wenn Daten ausgegeben werden, werden die Informationen in unserer Vorlage angezeigt.



Machen wir es noch cooler: private Anbieter



Schauen wir uns die vorherige Lösung genauer an.



Tatsächlich ist die Komponente unabhängig vom Router und sogar vom OrganizationService. Es hängt von der Organisation $ ab. In unserem Abhängigkeitsinjektionsbaum gibt es jedoch keine solche Entität, sodass wir gezwungen sind, Transformationen in der Komponente durchzuführen.



Was aber, wenn Sie diese Konvertierung durchführen, bevor die Daten in die Komponente gelangen? Schreiben wir Provider speziell für die Komponente, in der die erforderlichen Transformationen stattfinden werden.



Der Einfachheit halber platzieren wir die Anbieter in einer separaten Datei neben der Komponente und erhalten die folgende Dateistruktur: Der







Anbieter befindet sich in der Datei organisation.providers.ts für die Datenkonvertierung und ein Token für den Empfang in der Komponente: 



export const ORGANIZATION_INFO = new InjectionToken<Observable<Organization>>(
   'A stream with current organization information',
);
         :

export const ORGANIZATION_PROVIDERS: Provider[] = [
   {
       provide: ORGANIZATION_INFO,
       deps: [ActivatedRoute, OrganizationService],
       useFactory: organizationFactory,
   },
];

export function organizationFactory(
   {params}: ActivatedRoute,
   organizationService: OrganizationService,
): Observable<Organization> {
   return params.pipe(
       switchMap(params => {
           const id = params.get('orgId');

           return organizationService.getOrganizationById$(id);
       }),
   );
}


Definieren Sie ein Array von Anbietern für die Komponente. Der Wert für das ORGANIZATION_INFO-Token wird von der Fabrik bezogen, in der wir die erforderliche Datenkonvertierung durchführen.



Mit einem Hinweis zur Funktionsweise von DI - deps können Sie die erforderlichen Entitäten aus dem DI - Baum entnehmen und als Argumente an die Token Value Factory übergeben. In ihnen können Sie jede Entität von DI abrufen, einschließlich der Verwendung von DI-Dekoratoren, zum Beispiel: 



{
       provide: ACTIVE_TAB,
       deps: [
           [new Optional(), new Self(), RouterLinkActive],
       ],
       useFactory: activeTabFactory,
}


Anbieter in der Komponente deklarieren: 



@Component({
   ..
   providers: [ORGANIZATION_PROVIDERS],
})


Und wir sind bereit, die Daten in der Komponente zu verwenden: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
   providers: [ORGANIZATION_PROVIDERS],
})
export class OrganizationComponent {
   constructor(
       @Inject(ORGANIZATION_INFO) readonly organization$: Observable<Organization>,
   ) {}
}


Die Komponentenklasse wird mit dem Empfang der Daten auf eine Zeile reduziert.



Die Vorlage bleibt gleich: 



<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


Was gibt uns dieser Ansatz?



  1. Reine Abhängigkeiten: Die Komponente implementiert keine unnötigen Entitäten und speichert sie nicht. Es funktioniert nur mit den Daten, die es benötigt, während es selbst sauber bleibt und nur Logik zum Anzeigen von Daten enthält.

Nach der Implementierung dieses Ansatzes sahen unsere Komponenten und Richtlinien viel sauberer und einfacher aus. Die Trennung der Logik der Datentransformation und ihrer Anzeige beschleunigte den Prozess ihrer Verfeinerung und Erweiterung. Die Zeit zum Abfangen von Fehlern wurde ebenfalls verkürzt, da Sie den Problembereich sofort abschneiden können: Entweder liegt das Problem in der Datenkonvertierung oder in der Art und Weise, wie es dem Benutzer angezeigt wird.



Fazit



Der beschriebene Ansatz erspart Ihnen nicht alle Entwurfsprobleme. Das Hinzufügen von Anbietern zu kleinen Dingen lohnt sich ebenfalls nicht: Manchmal ist der Code klarer, wenn Sie die Konvertierung in der Methode verwenden oder Pipe verwenden.



Trotzdem hoffe ich, dass private Anbieter Ihnen helfen, Komponenten mit einer großen Anzahl von Abhängigkeiten zu vereinfachen oder eine Alternative für das schrittweise Refactoring großer logischer Elemente bereitzustellen.
