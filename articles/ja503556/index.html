基礎


これ以前は、C ++標準では、1つの変数に複数の型を格納する問題に対する多くの解決策が提供されていませんでした。もちろんを使用できますがvoid*、安全ではありません。

理論的にvoid*は、型を何らかの方法で格納できるクラスにラップできます。

class MyAny
{
    void* _value;
    TypeInfo _typeInfo;
};

ご覧のように、特定の基本的なフォームを取得しましたが、std::any型の安全性MyAnyを確保するために、追加のチェックが必要です。そのため、独自に決定するよりも、標準ライブラリのオプションを使用する方が適切です。

これがstd::anyC ++ 17のものです。間違ったタイプを指定してアクセスしようとすると、オブジェクトに何かを保存してエラーを報告（例外をスロー）することができます。

いつ使用するのですか？


void*私 はそれを非常に危険なものと考えていますが、使用範囲は非常に限られていますが、std::any完全にタイプセーフであるため、いくつかの良い使い方があります。

例えば：

  • ライブラリ内-ライブラリが一部のデータを格納または転送する必要があり、このデータがどのタイプかわからない場合
  • ファイルを解析するとき-サポートされているタイプを本当に判別できない場合
  • メッセージング
  • スクリプト言語の相互作用
  • スクリプト言語のインタープリターを作成する
  • ユーザーインターフェイス-フィールドは何でも格納できます

これらの例の多くでは、サポートされているタイプの限られたリストを強調表示できるので、それstd::variantがより良い選択かもしれません。しかしもちろん、それが使用される最終的な製品を知らずにライブラリを作成することは困難です。そこに格納されるタイプがわからないだけです。

デモではいくつかの基本的なことを示しましたが、次のセクションではについてさらに学習するstd::anyので、読み続けてください。

stdを作成::


タイプのオブジェクトを作成するには、いくつかの方法がありますstd::any。

  • 標準初期化-オブジェクトは空です
  • 値/オブジェクトによる直接初期化
  • オブジェクトのタイプを直接示す- std::in_place_type
  • 経由して std::make_any

例えば：

//  :
std::any a;
assert(!a.has_value());

//   :
std::any a2(10); // int
std::any a3(MyType(10, 11));

// in_place:
std::any a4(std::in_place_type<MyType>, 10, 11);
std::any a5{std::in_place_type<std::string>, "Hello World"};

// make_any
std::any a6 = std::make_any<std::string>("Hello World");

値を変更


現在保存されている値std::anyを変更する方法は2つあります：メソッドemplaceまたは割り当て：

std::any a;

a = MyType(10, 11);
a = std::string("Hello");

a.emplace<float>(100.5f);
a.emplace<std::vector<int>>({10, 11, 12, 13});
a.emplace<MyType>(10, 11);

オブジェクトのライフサイクル


セキュリティの鍵std::anyは、リソースの漏洩がないことです。これを達成するためにstd::any、新しい値を割り当てる前にアクティブなオブジェクトを破棄します。

std::any var = std::make_any<MyType>();
var = 100.0f;
std::cout << std::any_cast<float>(var) << "\n";

このコードは以下を出力します：

MyType::MyType
MyType::~MyType
100

オブジェクトはstd::anyMyType型のオブジェクトで初期化されますが、新しい値（100.0f）を割り当てる前に、デストラクタが呼び出されMyTypeます。

価値へのアクセス


ほとんどの場合、std::any- の値にアクセスする方法は1つだけstd::any_castです。オブジェクトに格納されている場合は、指定されたタイプの値を返します。

この機能には多くの使い方があるため、非常に便利です。

  • 値のコピーを返し、std::bad_any_castエラー時に終了します
  • 値へのリンクを返し、std::bad_any_cast エラー時に終了します
  • 値へのポインタ（定数かどうか）、またはエラーの場合はnullptrを返します

例を見る：

struct MyType
{
    int a, b;

    MyType(int x, int y) : a(x), b(y) { }

    void Print() { std::cout << a << ", " << b << "\n"; }
};

int main()
{
    std::any var = std::make_any<MyType>(10, 10);
    try
    {
        std::any_cast<MyType&>(var).Print();
        std::any_cast<MyType&>(var).a = 11; // /
        std::any_cast<MyType&>(var).Print();
        std::any_cast<int>(var); // throw!
    }
    catch(const std::bad_any_cast& e) 
    {
        std::cout << e.what() << '\n';
    }

    int* p = std::any_cast<int>(&var);
    std::cout << (p ? "contains int... \n" : "doesn't contain an int...\n");

    MyType* pt = std::any_cast<MyType>(&var);
    if (pt)
    {
        pt->a = 12;
        std::any_cast<MyType&>(var).Print();
    }
}

ご覧のとおり、エラーを追跡するには2つの方法があります。例外を使用する（std::bad_any_cast）か、ポインタを返す（またはnullptr）です。std::any_castポインタを返す関数はオーバーロードされ、としてマークされていnoexceptます。

パフォーマンスとメモリ使用量


std::any強力なツールのように見え、さまざまなタイプのデータを格納するために使用する可能性が高いですが、その価格はどのくらいですか？

主な問題は、追加のメモリ割り当てです。追加のメモリ割り当ては必要

std::variant ありstd::optionalませんが、これはオブジェクトに格納されているデータのタイプが事前にわかっているためです。 std :: anyにはそのような情報がないため、追加のメモリを使用できます。

これは常に発生しますか、それとも時々発生しますか？どのルール？これはintのような単純な型でも起こりますか？

標準の内容を見てみましょう。
実装では、小さい値に動的に割り当てられたメモリを使用しないでください。例：構築されたオブジェクトがintのみを保持している場合。このような小さなオブジェクトの最適化は、is_nothrow_move_constructible_vがtrueであるT型にのみ適用されます。
実装では、小さなサイズの格納データに動的メモリを使用しないでください。たとえば、intのみを格納するオブジェクトが作成される場合です。小さなオブジェクトに対するこのような最適化は、is_nothrow_move_constructible_vがtrueであるT型にのみ適用する必要があります。

その結果、彼らは実装にSmall Buffer Optimization / SBOを使用することを提案しています。しかし、これには価格もあります。これにより、型が大きくなり、バッファをカバーします。

サイズを見てみましょう。std::anyいくつかのコンパイラの結果を次に示します。

一般に、ご覧のとおり、std::anyこれは単純なタイプではなく、追加のコストがかかります。通常、SBOにより、16〜32バイト（GCCまたはclangの場合は...、MSVCの場合は64バイト）の大量のメモリが必要です。

メッセージング


主にCで記述されているWindows Apiには、メッセージデータを格納する2つのオプションパラメーターを持つメッセージIDを使用するメッセージングシステムがあります。このメカニズムに基づいて、ウィンドウに送信されたメッセージを処理するWndProcを実装できます。

LRESULT CALLBACK WindowProc(
  _In_ HWND   hwnd,
  _In_ UINT   uMsg,
  _In_ WPARAM wParam,
  _In_ LPARAM lParam
);

実際にはデータが保存されているということであるwParamかlParamの異なる形で。場合によっては、数バイトだけを使用する必要がありますwParam。

メッセージが何でも処理メソッドに渡せるようにこのシステムを変更するとどうなるでしょうか。

例えば：

class Message
{
public:
    enum class Type 
    {
        Init,
        Closing,
        ShowWindow,        
        DrawWindow
    };

public:
    explicit Message(Type type, std::any param) :
        mType(type),
        mParam(param)
    {   }
    explicit Message(Type type) :
        mType(type)
    {   }

    Type mType;
    std::any mParam;
};

class Window
{
public:
    virtual void HandleMessage(const Message& msg) = 0;
};

たとえば、ウィンドウにメッセージを送信できます。

Message m(Message::Type::ShowWindow, std::make_pair(10, 11));
yourWindow.HandleMessage(m);

ウィンドウは次のようなメッセージに返信できます。

switch (msg.mType) {
// ...
case Message::Type::ShowWindow:
    {
    auto pos = std::any_cast<std::pair<int, int>>(msg.mParam);
    std::cout << "ShowWidow: "
              << pos.first << ", " 
              << pos.second << "\n";
    break;
    }
}

もちろん、データ型がメッセージにどのように格納されるかを決定する必要がありますが、これで、数値によるさまざまなトリックの代わりに実際の型を使用できます。
