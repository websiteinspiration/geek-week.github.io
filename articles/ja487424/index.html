1から100,000ユーザーにスケーリングする方法

多くの新興企業がこれを経験しました。新しいユーザーの群れが毎日登録され、開発チームはサービスの作業をサポートするのに苦労しています。

これは快適な問題ですが、Webアプリケーションをゼロから数十万人のユーザーに正確にスケーリングする方法については、Web上に明確な情報はほとんどありません。通常、火災の解決策またはボトルネックの解消のいずれか（および多くの場合両方）があります。したがって、人々はかなりステレオタイプ化されたトリックを使用して、アマチュアプロジェクトを非常に深刻なものに拡張します。

情報を除外して、主な数式を書き留めてみましょう。新しいGraminsta写真共有サイトを1人から10万人まで段階的に拡大していきます。

聴衆を10、100、1000、1万、10万人に増やす場合に必要な具体的なアクションを書き留めます。

1ユーザー：1車


ウェブサイトであろうとモバイルアプリケーションであろうと、ほとんどすべてのアプリケーションには3つの主要なコンポーネントがあります。

  • API
  • データベース
  • クライアント（モバイルアプリケーションまたはWebサイト自体）

データベースは永続的なデータを保存します。APIは、このデータとその周辺のリクエストを処理します。クライアントはデータをユーザーに転送します。

アーキテクチャの観点から見ると、クライアントエンティティとAPIが完全に分離されていると、アプリケーションのスケーリングについて話す方がはるかに簡単であるという結論に達しました。

最初にアプリケーションの作成を開始するとき、3つのコンポーネントすべてを同じサーバーで実行できます。ある意味、これは私たちの開発環境を思い起こさせます。1人のエンジニアが同じコンピューター上でデータベース、API、およびクライアントを実行しています。

理論的には、次に示すように、DigitalOcean DropletまたはAWS EC2の1つのインスタンスのクラウドにデプロイできます。

そうは言っても、サイトに複数のユーザーがいる場合、ほとんどの場合、データベースレベルを強調することは理にかなっています。

10ユーザー：データベースを別のレベルに引き上げる


データベースをAmazon RDSやDigital Ocean Managed Databaseなどのマネージドサービスに分割することは、長い間私たちに役立ちます。これは、単一のマシンまたはEC2インスタンスでのセルフホスティングよりも少しコストがかかりますが、これらのサービスを使用すると、複数リージョンのバックアップ、リードレプリカ、自動バックアップなど、便利な拡張機能をすぐに利用できます。

これがシステムの外観です。

100ユーザー：クライアントを別のレベルに引き上げる


幸い、私たちのアプリは最初のユーザーを本当に気に入っていました。トラフィックはより安定しているので、クライアントを別のレベルに移動する時が来ました。ことに留意すべきであるエンティティの分離は、スケーラブルなアプリケーションを構築するための重要な側面です。システムの一部はより多くのトラフィックを受信するため、特定のトラフィックパターンに基づいてサービスのスケーリングを制御するように分割できます。

そのため、APIとは別にクライアントを表現するのが好きです。これにより、Web、モバイルWeb、iOS、Android、デスクトップアプリケーション、サードパーティサービスなど、いくつかのプラットフォームの開発について非常に簡単に話すことができます。これらはすべて、同じAPIを使用するクライアントにすぎません。

たとえば、現在、ユーザーはモバイルアプリケーションのリリースを頻繁に要求しています。クライアントエンティティとAPIを分離すると、より簡単になります。

1000ユーザー：ロードバランサーを追加する


物事は順調です。 Graminstaユーザーは、ますます多くの写真をアップロードしています。登録数も増えています。唯一のAPIサーバーでは、すべてのトラフィックを管理することが困難です。もっと鉄が必要！

ロードバランサーは非常に強力なコンセプトです。重要なアイデアは、バランサーをAPIの前に置き、トラフィックを個々のサービスインスタンスに分散することです。これは、水平スケーリングの方法です。つまり、同じコードでサーバーを追加し、処理できる要求の数を増やします。

Webクライアントの前とAPIの前に個別のロードバランサーを配置します。つまり、APIコードとWebクライアントコードを実行する複数のインスタンスを実行できます。ロードバランサーは、負荷の少ないサーバーにリクエストを転送します。

ここで、もう1つの重要な利点が得られます-冗長性。 1つのインスタンスで障害が発生した場合（過負荷またはクラッシュの可能性があります）、着信要求に応答するインスタンスが残っています。単一のインスタンスが機能した場合、障害が発生するとシステム全体が停止します。

ロードバランサーは、自動スケーリングも提供します。ピーク負荷の前にインスタンスの数を増やし、すべてのユーザーがスリープする時間を減らすように構成できます。

ロードバランサーを使用すると、APIレベルを無限にスケーリングできます。リクエストの数が増えると、新しいインスタンスが追加されるだけです。

10 000 : CDN


おそらくそれは最初から行われているはずです。リクエストの処理と新しい写真の撮影により、サーバーに負荷がかかりすぎています。

この段階では、画像、動画などの静的コンテンツ（AWS S3またはDigital Ocean Spaces）を格納するためにクラウドサービスを使用する必要があります。一般に、APIは画像のアップロードやサーバーへの画像のアップロードなどの処理を回避する必要があります。

クラウドホスティングのもう1つの利点は、そのCDNです（AWSでは、このアドオンはCloudfrontと呼ばれますが、多くのクラウドストレージサービスがそのまま使用できます）。 CDNは、世界中のさまざまなデータセンターに画像を自動的にキャッシュします。

私たちのメインデータセンターはオハイオ州に配置できますが、誰かが日本から画像を要求した場合、クラウドプロバイダーはコピーを作成し、それを日本のデータセンターに保存します。この画像を日本で次にリクエストする人は、画像をはるかに早く受け取ります。これは、地球全体にアップロードして送信するのに長い時間がかかる写真やビデオなどの大きなファイルを扱う場合に重要です。



100,000ユーザー：データレベルのスケーリング


CDNは本当に役立ちました：トラフィックはフルスピードで増加しています。彼らが言うように、有名なビデオブロガーであるメイドモブリックが私たちに登録し、彼のストーリーを投稿しました。ロードバランサーのおかげで、APIサーバーのCPUとメモリの使用率は低く保たれます（10個のAPIインスタンスが実行されています）が、リクエストに対して多くのタイムアウトを取得し始めています...これらの遅延はどこから来たのですか？

メトリックを少し掘り下げると、データベースサーバーのCPUに80〜90％の負荷がかかっていることがわかります。私たちは限界にいます。

データ層のスケーリングは、おそらく方程式の最も難しい部分です。 APIサーバーはステートレスリクエストを処理するため、APIインスタンスを追加するだけです。しかし、ほとんどのデータベースは成功しません。一般的なリレーショナルデータベース管理システム（PostgreSQL、MySQLなど）について説明します。

キャッシング


データベースのパフォーマンスを向上させる最も簡単な方法の1つは、キャッシュレベルという新しいコンポーネントを導入することです。最も一般的なキャッシュ方法は、RedisやMemcachedなどのキーと値のレコードをRAMに格納することです。ほとんどのクラウドには、AWSのElasticacheやGoogle CloudのMemorystoreなど、これらのサービスのマネージバージョンがあります。

キャッシュは、サービスが同じ情報を取得するためにデータベースに対して繰り返し呼び出しを行う場合に役立ちます。実際、データベースにアクセスするのは1回だけであり、情報をキャッシュに保存します。これ以上は操作しません。

たとえば、Graminstaサービスでは、誰かがMobricスタープロフィールページにアクセスするたびに、APIサーバーがデータベース内の自分のプロフィールから情報を要求します。それは何度も何度も発生します。Mobrickのプロファイル情報はリクエストごとに変わるわけではないため、キャッシュに最適です。30秒の有効期間で

、データベースからの結果をキーごとにRedisにキャッシュしますuser:id。さて、誰かがMobrickのプロファイルを入力すると、最初にRedisをチェックし、そこにデータがある場合は、Redisから直接転送します。現在、サイトで最も人気のあるプロファイルへのクエリは、実際にはデータベースをロードしません。

ほとんどのキャッシングサービスのもう1つの利点は、データベースサーバーよりもスケーリングが簡単なことです。Redisには、組み込みのRedisクラスタークラスターモードがあります。ロードバランサーのように1、Redisキャッシュを複数のマシン（必要に応じて数千のサーバー）に分散できます。

ほとんどすべての大規模アプリケーションはキャッシングを使用します;これは高速APIの絶対的に不可欠な部分です。リクエスト処理の高速化とコードの効率化-これらすべてが重要ですが、キャッシュがなければ、何百万ものユーザーにサービスを拡張することはほとんど不可能です。

さらなる行動


アプリケーションのスケーリングが進むにつれ、サービスを個別にスケーリングしてサービスを個別にスケーリングしていきます。たとえば、Websocketの使用を開始する場合、Websocket処理コードを別のサービスにプルするのは理にかなっています。独自のロードバランサーの背後にある新しいインスタンスに配置できます。これは、開いているWebsocket接続に応じて、またHTTPリクエストの数に関係なく、スケールアップおよびスケールダウンできます。

また、データベースレベルでの制限との戦いも続けています。データベースのパーティショニングとシャーディングを研究する時が来たのはこの段階です。どちらのアプローチでも追加のオーバーヘッドが必要ですが、データベースをほぼ無限に拡張できます。

また、New RelicやDatadogなどのモニタリングおよび分析サービスもインストールしたいと考えています。これは遅いクエリを識別し、改善が必要な場所を理解します。スケールするにつれ、ボトルネックを見つけて解決することに集中したいと思います。多くの場合、前のセクションのいくつかのアイデアを使用します。
