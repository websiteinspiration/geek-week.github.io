Warum schreiben wir keinen Code in Controller?

Ich denke, viele von Ihnen haben die Meinung gehört, dass die Controller keinen Code enthalten sollten, und daher wird ein Controller mit Methoden in einer Zeile als „Best Practice“ angesehen. Ich bezweifle wiederum, dass die Vorteile davon so groß sind. Wenn Sie ähnliche Gedanken hatten, frage ich unter kat.



Hallo alle zusammen! Ich möchte sofort sagen, dass meine Meinung nicht wahr ist, und der Zweck dieses Beitrags ist es, meine Meinung zu äußern und die Kommentare anderer zu hören. All dies gilt für die Implementierung der API. Wenn Sie MVC mit Ansichten haben, funktioniert dieser Fall nicht, da es in diesem Fall besser ist, nur die Logik mit Ansicht in die Controller zu schreiben

Was ist los?


Ich bin sicher, dass viele von Ihnen mit einer typischen CRUD-Anwendung mit 3-Layer-Architektur arbeiten (übrigens gibt es keine weiteren Layer, aber beim nächsten Mal mehr darüber). In dieser Architektur haben Sie eine Ebene für die Arbeit mit Daten (weitere DA), eine Ebene für Geschäftslogik (weitere BL) und eine Ansichtsebene (weitere VL).

Lassen Sie uns den Überschuss entfernen.





Oh Wunder, der Service-Code unterscheidet sich nicht vom Controller. Wir haben nur ein paar Attribute hinzugefügt und sind bereit, http-Anfragen abzufangen.

Ich glaube, dass ASP NET uns perfekt von der Arbeit mit HTTP abstrahiert hat. Wir haben den besten Ort, an dem wir mit unseren Typen arbeiten. Ich wiederhole, wenn Sie Ansichten haben, ist es in Controllern besser, Code nur für View zu schreiben, und in Diensten schreiben Sie wiederverwendete Methoden, um Daten für View abzurufen. In der heutigen Realität haben wir jedoch immer häufiger API + SPA.

Validierung?


Die ASP NET Core Pipeline ist sehr gut abgestimmt und bietet viele Lösungen. Schauen Sie sich FluentValidation an. Sie werden eine Validierung hinzufügen, ohne den Code in den Controllern zu ändern.

Willst du mehr Trennung?


Trennen Sie die Schnittstelle und wenn Sie auch eine Implementierung benötigen.



Als Bonus wird die Serviceschnittstelle zu einem Vertrag auf höchster Ebene, und im Rahmen eines Prozesses ist es nur ein direkter Aufruf des Codes vom Controller. Im Rahmen der Client-Server-Kommunikation wird eine einfache Implementierung derselben Schnittstelle mithilfe von HttpClient ersetzt.

Schließen Sie andere Kanäle an


Wenn sie mir sagen, dass wir einen anderen Kanal haben können, zum Beispiel über eine Warteschlange, kann ich einfach eine Instanz des Controllers abrufen und in einem anderen Kanal verwenden. Dieser Controller löst sich leicht von DI auf. Darüber hinaus ist ASP NET sehr flexibel und Sie können einigen Kanälen beibringen, es selbst zu verarbeiten, indem Sie die Pipeline erneut ändern.



Lustige Tatsache


Eine Testaufgabe in diesem Stil wird verworfen und abgelehnt. Sie haben jedoch viele Gründe, warum Sie dies getan haben. Wenn Sie jedoch zu einem Interview gehen und diskutieren, warum Sie den Code entwerfen, ist dies wahrscheinlich ein guter Ort, und Sie haben denselben Spezialisten, der das versteht Diese Programmierung ist vielfältig .

Gesamt


Sie können nur Dienste nach Repository-Typ, Cache, Client-API usw. wiederverwenden. Die Wiederverwendung der Logik zur Verarbeitung einer einzelnen Anforderung ist jedoch ein sehr seltener und höchstwahrscheinlich ein schlechter Fall.

Dieser Ansatz muss wie andere an den richtigen Stellen angewendet werden, spart Zeit und ist sehr praktisch für APIs (Microservices), die für den öffentlichen Zugriff geschlossen sind.

Ich glaube, dass die Controller der BL-Dienst oder Command \ QueryHandler sind, und in meinen Projekten übe ich diesen Ansatz und teile die Controller sehr gut. Ich empfehle, dass Sie es auch versuchen.
