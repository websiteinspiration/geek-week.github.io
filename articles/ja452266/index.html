サーバーレスラック


サーバーレスは、サーバーの物理的な不在についてではありません。これはコンテナの「キラー」ではなく、成功傾向でもありません。これは、クラウドでシステムを構築するための新しいアプローチです。今日の記事では、サーバーレスアプリケーションのアーキテクチャに触れ、サーバーレスサービスプロバイダーとオープンソースプロジェクトが果たす役割を確認します。最後に、サーバーレスについて説明します。

アプリケーションのサーバー側（オンラインストアでさえ）を記述したいと思います。これは、チャット、コンテンツを公開するためのサービス、またはロードバランサーにすることができます。いずれにしても、多くの頭痛の種があります。インフラストラクチャを準備し、アプリケーションの依存関係を決定し、ホストオペレーティングシステムについて考える必要があります。次に、残りのモノリスの作業に影響を与えない小さなコンポーネントを更新する必要があります。さて、負荷の下でスケーリングすることを忘れないでください。

しかし、必要な依存関係が既にプレインストールされていて、コンテナー自体が互いに、およびホストOSから分離されている一時的なコンテナーを取得するとどうなるでしょうか。モノリスをマイクロサービスに分割し、それぞれを他のサービスとは無関係に更新およびスケーリングできます。このようなコンテナーにコードを配置したら、任意のインフラストラクチャーで実行できます。もういい。

そして、コンテナを設定したくない場合はどうでしょうか？アプリケーションのスケーリングについて考えたくありません。サービスの負荷が最小限の場合は、実行中のアイドル状態のコンテナに料金を支払いたくありません。コードを書きたい。ビジネスロジックと市場の製品に光速で焦点を当てます。

そのような考えから、私はサーバーレスコンピューティングに移行しました。この場合のサーバーレスは、サーバーの物理的な欠如を意味するのではなく、インフラストラクチャを管理するための頭痛の欠如を意味します。

アイデアは、アプリケーションロジックが独立した機能に分解されるというものです。イベント構造があります。各関数は1つの「マイクロタスク」を実行します。開発者に必要なのは、クラウドプロバイダーによって提供されるコンソールに関数をロードし、それらをイベントのソースと関連付けることだけです。コードは自動的に準備されたコンテナでリクエストに応じて実行され、実行時間に対してのみ支払います。

アプリケーション開発プロセスがどのようになるか見てみましょう。

開発者から


以前、私たちはオンラインストアのアプリケーションについて話し始めました。従来のアプローチでは、システムのメインロジックはモノリシックアプリケーションによって実行されます。また、アプリケーションがインストールされているサーバーは、負荷がなくても常に実行されています。

サーバーレスに切り替えるには、アプリケーションをマイクロタスクに分割します。それぞれの下に、独自の関数を記述します。関数は互いに独立しており、状態情報を保存しません。異なる言語で書くこともできます。それらの1つがクラッシュしても、アプリケーション全体は停止しません。アプリケーションアーキテクチャは次のようになります。


サーバーレスでの機能の分割は、マイクロサービスでの作業に似ています。しかし、マイクロサービスはいくつかのタスクを実行でき、理想的には、関数がタスクを実行する必要があります。タスクが統計を収集し、ユーザーの要求に応じて表示することを想像してください。マイクロサービスアプローチでは、タスクは、書き込みと読み取りの2つのエントリポイントを持つ1つのサービスによって実行されます。サーバーレスコンピューティングでは、これらは相互接続されない2つの異なる機能になります。たとえば、統計がダウンロードよりも頻繁に更新される場合、開発者はコンピューティングリソースを節約できます。

サーバーレス機能は、サービスプロバイダーが決定する短時間（タイムアウト）で実行する必要があります。たとえば、AWSの場合、タイムアウトは15分です。したがって、要件を満たすために、長期間有効な機能（長期間有効）を変更する必要があります。このサーバーレスは、現在一般的な他のテクノロジー（コンテナーおよびサービスとしてのプラットフォーム）とは異なります。

各機能にイベントを割り当てます。イベントはアクションのトリガーです。
イベント 関数が実行するアクション
商品画像がストアにアップロードされました 画像を圧縮してカタログにアップロード
実店舗の住所がデータベースで更新されました 新しい場所を地図にアップロード
顧客が商品の代金を支払う 支払い処理を開始
イベントには、HTTPリクエスト、ストリーミングデータ、メッセージキューなどがあります。イベントのソースは、データの変更または発生です。また、タイマーで機能を実行できます。

アーキテクチャは機能し、アプリケーションはほとんどサーバーレスになりました。次に、サービスプロバイダーに行きます。

プロバイダーから


サーバーレスコンピューティングは通常、クラウドサービスプロバイダーによって提供されます。それらは、Azure Functions、AWS Lambda、Google Cloud Functions、IBM Cloud Functionsと呼んでいます。

プロバイダーのコンソールまたは個人アカウントを通じてサービスを利用します。機能コードは、次のいずれかの方法でダウンロードできます。

  • Webコンソールを介して組み込みエディターでコードを記述します。
  • コードを含むアーカイブをダウンロードし、
  • パブリックまたはプライベートのgitリポジトリで作業します。

ここでは、関数を呼び出すイベントを構成します。プロバイダーによって、イベントセットが異なる場合があります。



インフラストラクチャ上のプロバイダーは、サービスとしての機能（FaaS）システムを構築および自動化しました。

  1. 関数コードは、プロバイダー側​​のリポジトリーに到達します。
  2. イベントが発生すると、準備された環境を持つコンテナーがサーバーに自動的にデプロイされます。各関数インスタンスには、独自の独立したコンテナーがあります。
  3. ストレージから、関数はコンテナに送信され、計算され、結果を返します。
  4. 並列イベントの数は増加しています-コンテナーの数は増加しています。システムは自動的にスケーリングします。ユーザーが機能にアクセスしない場合、機能は非アクティブになります。
  5. プロバイダーはコンテナーのアイドル時間を設定します。この時間中に関数がコンテナーに表示されない場合、コンテナーは破棄されます。

したがって、サーバーレスをすぐに利用できます。従量課金制のモデルで、使用された機能と使用された時間に対してのみサービスの料金を支払います。

開発者にサービスを紹介するために、プロバイダーは最大12か月の無料テストを提供しますが、合計計算時間、1か月あたりのリクエスト数、費用、または電力消費を制限します。

プロバイダーと連携することの主な利点は、インフラストラクチャー（サーバー、仮想マシン、コンテナー）を気にする必要がないことです。プロバイダーは、自社開発とオープンソースツールの両方でFaaSを実装できます。これらについては後で詳しく説明します。

オープンソースから


ここ数年、オープンソースコミュニティはサーバーレスツールに積極的に取り組んできました。最大の市場参加者を含め、サーバーレスプラットフォームの開発に貢献しています。

  • Googleは開発者にオープンソースツールであるKnativeを提供しています。その開発には、IBM、RedHat、Pivo​​tal、SAPが関係していました。
  • IBMはServerless -platform OpenWhiskに取り組みました。これは後にApache Foundationプロジェクトになりました。
  • MicrosoftはAzure Functionsプラットフォームコードを部分的にオープンしました。

サーバーレスフレームワークの方向で開発も行われています。KubelessとFissionは事前に準備されたKubernetesクラスター内にデプロイされ、OpenFaaSはKubernetesとDocker Swarmの両方で動作します。フレームワークは一種のコントローラーとして機能します-リクエストに応じて、クラスター内でランタイムを準備し、そこでそこで関数を実行します。

フレームワークは、ニーズに合わせてツールを構成する余地を残します。したがって、Kubelessでは、開発者は関数のタイムアウトを構成できます（デフォルト値は180秒です）。フィッションは、コールドスタートの問題を解決するために、コンテナの一部を常に稼働させ続けることを提案しています（ただし、これにはダウンタイムのコストが伴います）。また、OpenFaaSは、HTTP、Kafka、Redis、MQTT、Cron、AWS SQS、NATなど、あらゆる好みと色に一連のトリガーを提供します。

開始するための手順は、フレームワークの公式ドキュメントに記載されています。それらを使用することは、プロバイダーを使用するよりも少しスキルが高いことを意味します。これは、少なくともCLIを介してKubernetesクラスターを起動する機能です。最大、他のオープンソースツール（たとえば、Kafkaのキューマネージャー）を含めます。

サーバーレスでの作業方法に関係なく-プロバイダーを通じて、またはオープンソースを使用して、サーバーレスアプローチには多くの長所と短所があります。

メリットとデメリットの観点から


サーバーレスは、コンテナーインフラストラクチャとマイクロサービスアプローチのアイデアを開発します。これにより、チームは1つのプラットフォームに縛られることなく、多言語モードで作業できます。システムの構築が簡素化され、エラーの修正が容易になります。マイクロサービスアーキテクチャを使用すると、モノリシックアプリケーションの場合よりもはるかに高速にシステムに新しい機能を追加できます。

サーバーレスは、開発者がアプリケーションのビジネスロジックとコーディングのみに集中できるようにすることで、開発時間をさらに短縮します。その結果、開発の市場投入までの時間が短縮されます。

おまけとして、負荷に合わせて自動スケーリングが行われ、使用されたリソースと使用された時間に対してのみ料金が発生します。

他のテクノロジーと同様に、サーバーレスには欠点があります。

たとえば、そのような欠点は、コールドスタート時間になる可能性があります（JavaScript、Python、Go、Java、Rubyなどの言語の場合、平均で最大1秒）。

一方で、実際には、コールドスタートの時間は多くの変数に依存します。関数が記述されている言語、ライブラリの数、コードの量、追加のリソース（同じデータベースまたは認証サーバー）との通信です。開発者はこれらの変数を制御するため、開始時間を短縮できます。しかし、その一方で、開発者はコンテナの起動時間を制御できません-それはすべてプロバイダーに依存します。

多くのアプリケーションでは、コールドスタートは問題になりません。ここでは、サービスのタイプとタスクに基づいて構築する必要があります。 1秒の遅延開始は、ビジネスアプリケーションにとって必ずしも重要ではありませんが、医療サービスにとって重要になる可能性があります。おそらく、この場合、サーバーレスアプローチは適切ではなくなります。

サーバーレスの次の欠点は、関数の存続期間が短いことです（関数が実行しなければならないタイムアウト）。

ただし、長期間のタスクを処理する必要がある場合は、ハイブリッドアーキテクチャを使用して、サーバーレスと他のテクノロジーを組み合わせることができます。

すべてのシステムがサーバーレススキームに従って動作するわけではありません。

一部のアプリケーションは、実行時にデータとステータスを保存します。一部のアーキテクチャはモノリシックのままであり、一部の機能は長寿命になります。ただし、（以前はクラウドテクノロジーでしたが、その後はコンテナーでした）サーバーレスは、大きな将来をもたらすテクノロジーです。

この流れで、サーバーレスアプローチの適用の問題にスムーズに移行したいと思います。

アプリケーション側


2018年、サーバーレスの使用率は1.5倍に増加しました。サービスにテクノロジーをすでに実装している企業の中には、Twitter、PayPal、Netflix、T-Mobile、Coca-Colaなどの巨大市場があります。同時に、サーバーレスは万能薬ではなく、特定の範囲のタスクを解決するためのツールであることを理解する必要があります。

50人が来るサービスがあるとしましょう。その下には、ハードウェアが弱い仮想マシンがあります。定期的に、サービスの負荷が大幅に増加します。その後、弱い鉄は対処できません。

たとえば、3つの仮想マシンに負荷を分散するバランサーをシステムに含めることができます。この段階では、負荷を正確に予測できないため、一定量のリソースを「予備」で実行し続けます。そして、ダウンタイムの過払い。

この状況では、ハイブリッドアプローチを使用してシステムを最適化できます。ロードバランサーの場合は、仮想マシンを1つ残し、関数を使用してサーバーレスエンドポイントへのリンクを配置します。負荷がしきい値を超えると、バランサーは要求処理の一部を担う関数のインスタンスを起動します。


したがって、あまり頻繁ではないが、多数の要求を集中的に処理する場合にサーバーレスを使用できます。この場合、いくつかの機能を15分間実行する方が、仮想マシンまたはサーバーを常に保持するよりも収益性が高くなります。

サーバーレスコンピューティングのすべての利点を活用するには、それを実装する前に最初に行うべきことは、アプリケーションロジックを評価し、サーバーレスが特定のケースで解決できるタスクを理解することです。
