カピバラページオブジェクトを使用したウェブサイトユーザー機能のテスト

ページオブジェクトは、技術的な実装からテストを抽象化（分離）する強力な方法として使用できます。機能（ウェブサイト）を簡単な方法でカプセル化することにより、テスト（ページオブジェクト）を使用して、テストの安定性を高め、DRYの原則を維持する（自分自身を繰り返さない）ことができます。

言い換えると


ページオブジェクトは、テスト環境からユーザーインターフェイスを抽象化（分離）し、ユーザーインターフェイスと対話するためのメソッドを提供し、必要な情報を抽出するクラスのインスタンスです。

用語


ページオブジェクト という用語は、一般的すぎる概念です。私の経験では、ページオブジェクトには次の3つのタイプが含まれます。

  • コンポーネントオブジェクトは、ユーザーインターフェイスの特定のコンポーネントまたはウィジェットを表します。例：テーブル、メニュー、記事、およびコンポーネントのグループを含むその他のブロック。
  • ページオブジェクトは、Webアプリケーションの特定の領域またはユーザーインターフェイスを記述します。複数のコンポーネントオブジェクトで構成され、このオブジェクトに含まれている抽象化と対話するための便利なメソッドが含まれている場合があります。
  • エクスペリエンスは、複雑な機能をグループ化するために使用されます。テストにはいくつかの手順が必要です。または、複数のページとのやり取りが必要です。私自身の経験から、この概念を使用してページの複雑な動作を抽象化しました（教育用ページのテスト、新しいユーザーの作成など）。

例


ブログを作成し、ページオブジェクトを使用しない 単純なRSpec Capybaraテストを考えてみます。

require 'feature_helper'

feature 'Blog management', type: :feature do
  scenario 'Successfully creating a new blog' do
    visit '/'

    click_on 'Form Examples'
    expect(page).to have_content('Create Blog')

    fill_in 'blog_title', with: 'My Blog Title'
    fill_in 'blog_text', with: 'My new blog text'

    click_on 'Save Blog'
    expect(page).to have_selector('.blog--show')

    expect(page).to have_content('My Blog Title')
    expect(page).to have_content('My new blog text')
  end

  scenario 'Entering no data' do
    visit '/'
    click_on 'Form Examples'

    expect(page).to have_content('Create Blog')

    click_on 'Save Blog'

    expect(page).to have_content('4 errors stopped this form being submitted')

    expect(page).to have_content("Title can't be blank")
    expect(page).to have_content("Text can't be blank")

    expect(page).to have_content('Title is too short')
    expect(page).to have_content('Text is too short')
  end
end

コードを詳しく見てみましょう。いくつかの問題があります。次のアクションがあります。対応するページへの切り替え、ページの操作、コンテンツの確認。コードの一部が重複していますが、これはDRYの原則に準拠することで修正できます。

テスト中のアプリケーションに変更がある場合、このコードを維持するのは難しいことを理解することが重要です。たとえば、要素のクラス、名前、識別子は変更される可能性があり、テストコードを定期的に更新する必要があります。

また、このコードには「セマンティックコンテキスト」がないため、コードのどの行が論理的にグループ化されているかを理解することは困難です。

ページオブジェクトの概要


用語セクションで説明したように、ページオブジェクトを使用して、プレゼンテーションレベルの抽象化を表すことができます。

前の例を取り上げ、Pageオブジェクトを使用して新しいブログを作成し、ブログを表示すると、前の例のコードをクリアできます。

技術的な実装に関する特定の情報を取り除いたので、最終結果（コード）は読み取り可能であり、ユーザーインターフェイスに関する特定の情報（ID、CSSクラスなど）を含むべきではありません。

require 'feature_helper'
require_relative '../pages/new_blog'
require_relative '../pages/view_blog'

feature 'Blog management', type: :feature do
  let(:new_blog_page) { ::Pages::NewBlog.new }
  let(:view_blog_page) { ::Pages::ViewBlog.new }

  before :each do
    new_blog_page.visit_location
  end

  scenario 'Successfully creating a new blog' do
    new_blog_page.create title: 'My Blog Title',
                         text: 'My new blog text'

    expect(view_blog_page).to have_loaded
    expect(view_blog_page).to have_blog title: 'My Blog Title',
                                        text: 'My new blog text'
  end

  scenario 'Entering no data' do
    new_blog_page.create title: '',
                         text: ''

    expect(view_blog_page).to_not have_loaded
    expect(new_blog_page).to have_errors "Title can't be blank",
                                         "Text can't be blank",
                                         "Title is too short",
                                         "Text is too short"
  end
end

契約に違反しないでください。ページオブジェクトに期待（期待）を含めないでください。ただし、必要に応じて、条件を処理するためにカピバラの組み込みメカニズムに依存するよりも、このアプローチを好みます。

たとえば、次のCapybaraコードでは、ページオブジェクト（この場合はself）内に'foo'が存在することを想定しています。

expect(self).to have_content 'foo'

ただし、次のコードでは：

expect(page_object.content).to match 'foo'

page_object.contentが条件に準拠しているかどうかがすぐにチェックされ、おそらくまだ宣言されていないため 、予期しないエラーが発生する可能性があります（フローティングテストが発生する可能性があります）。その他の例については、thoughtbotによる カピバラとの信頼性の高い非同期統合テストの記述を読むことをお勧めします。

メソッドの作成


1つのメソッドのフレームワークで、データを取得する場所（リージョン）を抽象化（記述）できます。

def visit_location
  visit '/blogs/new'
  # It can be beneficial to assert something positive about the page
  # before progressing with your tests at this point
  #
  # This can be useful to ensures that the page has loaded successfully, and any
  # asynchronous JavaScript has been loaded and retrieved etc.
  #
  # This is required to avoid potential race conditions.
  expect(self).to have_loaded
end

def has_loaded?
  self.has_selector? 'h1', text: 'Create Blog'
end

コンポーネントオブジェクト


この例ではNewBlogクラスを使用していますが、作成する実装はありません。

フォームを操作するので、このコンポーネントを表すクラスをさらに導入できます。

# ...

def create(title:, text:)
  blog_form.new.create title: title,
                       text: text
end

# ...

private

def blog_form
  ::Components::BlogForm
end

BlogFormのメソッド実装を非表示にできる場所：

module Components
  class BlogForm
    include RSpec::Matchers
    include Capybara::DSL

    def create(title:, text:)
      within blog_form do
        fill_in 'blog_title', with: title
        fill_in 'blog_text', with: text

        click_on 'Save Blog'
      end
    end

    private

    def blog_form
      find('.blog--new')
    end
  end
end

end

注：オブジェクトファイルの上部に手動でページオブジェクトを手動で作成しました。一部のRSpecテストでは、すべてのサポートファイルを自動的にダウンロードし、オブジェクトファイルでそれらにアクセスできるようにすると便利な場合がありますが、これにより、大きなコードを使用するときに過度のワークロードが発生する可能性があります。特に、これにより起動が遅くなり、意図しない循環依存が発生する可能性があります。
