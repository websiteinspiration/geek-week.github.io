Minecraftサーバー：WindowsとLinux


何かに対する何かに対する一連の記事を続けて、最後に何か有用なもの、つまりMinecraftサーバーに注目します。人類の最高のゲームをホストするには、どのオペレーティングシステムとどのJavaがさらに優れているかを検討してください。比較のために、Ubuntu 18.04 LTSとServer Core 2019が採用されました。OpenJDKはUbuntuにインストールされ、Oracle JavaとAdoptOpenJDKはWindowsにインストールされました。

他のすべての比較テストと同様に、仮想マシンにはネイバーがなく、ホスト上で常に実行されているVMは1つだけでした。

引数で起動したサーバー：

-Xmx8G -server

Windows VDSを使用した99ルーブルの イメージのように、Windows DefenderコンポーネントはWindows Server Coreから削除されました。対照的に、これは、電源を入れたままにすると失うものです。 



Javaディストリビューションごとに、最新の公開バージョンがインストールされました。つまり、

Oracle： "1.8.0_241"
AdoptOpenJDK： "1.8.0_242"
OpenJDK： "1.8.0_232"

ラウンドナンバー1、世界の世代


このテストでは、世界を生成します。ジェネレーターは、Biomes'O'Plenty、Dynamic Trees、PVG、worley caves、ICおよびBCがインストールされたGeographicraftでした。

世界は決して古典的ではなく、通常よりも著しく遅い速度で生成されます。 

2,704チャンクの世界が生成されました：


AdoptOpenJDKを搭載したWindowsは、競合他社から5秒間離脱します。

ラウンド番号2、サーバーの起動


測定は、仮想マシンごとに3つのパスで行われました。毎回、各サーバーは前の結果と比較して、1秒あたり1秒ごとにワールドのダウンロードを完了しました。


Windows上のOpenJDKとLinux上のOpenJDKは同じ結果を示します。

第3ラウンド、占有メモリ


プロセスはより多くのメモリを消費し始め、より多くのコアがインストールされます。以下は、ワールドがロードされていない空のサーバープロセスの占有メモリの表です。


Oracle JREは、偶数のコアで平均80〜100メガバイト多く消費しました。同じことがAdoptOpenJDKにも当てはまり、奇数のコアでのみ行われました。

Linuxはそのような奇妙さを示さなかった。

ラウンドナンバー4、32箱入りチキン2 in 2



シーンは、2 x 2の箱に32羽の雌鳥が衝突した計算です。シーンは事前に準備され、同じ世界がサーバーの周りに散在しているため、すべてが正直でした。

このテストでは、1つのコアがインストールされ、プロセスはリアルタイムの優先度に設定されました。


このシーンのOpenJDKワーキングセットは、ライバルよりも40メガバイト多くなっています。


OracleとAdoptOpenJDKでの平均プロセッサー消費量は同じですが、Oracleガベージがより頻繁かつ集中的に収集されるため、プロセッサーアクティビティのバーストが発生することがよくあります。

処理できるそのようなシーンの数を推定するには、サーバーのティックレートを増やしてみましょう。


高負荷テストでは、Ubuntu c OpenJDKがWindows c AdoptOpenJDkに追いつき、Oracleが追いつきました。


より高い負荷の下では、Windows上のOpenJDKはUbuntuよりも優れた結果をもたらしました。

UbuntuのOpenJDKサーバーが常に消去され、シーンがフリーズしました。Windowsは同じOpenJDKで少し悪かった。オラクルは、凍結回数を最小限に抑えて最善を尽くしました。


特に、Oracle SEはOpenJDKと同じ量のRAM内に保持されていました。

第5ラウンド、64 * 64チャンク、および動的ツリー



このシーンには、森と数十の暴徒が含まれています。何キロにもわたる木の成長と、ブロックの位置の更新。

各ツリーは個別のタイルですが、最初はティックレートが低く、20ゲームティックで1回だけティックします。以下は、サーバーティック率ごとのプロセッサ使用率のグラフです。


Ubuntu + OpenJDKおよびOracleを搭載したWindows Serverは、前述の引数でサーバーを起動できなかったため、スケジュールに参加しませんでした。

それでもサーバーを起動するには、フラグを次のように変更する必要がありました。

-Xms4g -Xmx8G -server -XX:+UseCompressedOops -XX:+AggressiveOpts

3つのインスタンスはすべて、最初はプロセッサの100％で稼働していましたが、Windows Server + AdoptOpenJDKのみがサーバーをドロップしませんでした。ガベージコレクション後、すべてが以下のスケジュールに戻りました。


Ubuntuでティックレートを60から70に切り替えると、プロセッサの負荷スケジュールが正弦波のように動作し始めたため、タスクの複雑さが増すため、平均CPU使用率が突然低下し始めました。このため、現在の場所でスケジュールを停止する必要がありました。

これはおそらくLinuxスケジューラとWindowsの違いです。

結論：

OSとJREのディストリビューションには客観的な違いがありますが、サーバーを維持するために客観的に優れている特定の推奨事項を提供することは不可能です。

この場合は、より慣れているオペレーティングシステムを選択する価値があります。
