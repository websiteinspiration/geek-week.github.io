新しい機能を開発したいが、ユーザーがそれを気に入ってくれるかどうか確信が持てないとし、それを簡単に隠す方法が必要であるとします。または、新しい大きな機能に取り組んでいて、モンスターのコミットを回避したいとします。または、サイトの動作を簡単に構成できるようにしたいだけです。これらの問題をすべて解決するにはどうすればよいですか、猫の下で読んでください。

feature switcher-


この問題は開発ではかなり一般的であり、上記のアプローチから機能トグルまたは機能スイッチャーを最大限に活用できるエレガントなソリューションがあります。

基本的に、機能スイッチャーはデータベースに格納されているブールフラグであり、機能を有効にするかどうかに関する情報が含まれています。このフラグの値は、キーによってデータベースから取得できます。機能スイッチャーを使用すると、アプリケーションを再デプロイしなくても、実行時にビジネスユーザーが管理パネルを使用して簡単に変更できるので便利です。

以下は、Javaで機能トグルを使用する例です。

if (configurationManager.getParameter("is.some.functionality.enabled")) {
    // do some stuff
} else {
    // do default logic
}

上記の例では、configurationManagerは、特定の機能スイッチャーの値をデータベースのキーで取得できるようにするクラスです。

また、機能スイッチャーを使用して、フロントエンドの特定の要素を表示/非表示にすることができます。これを行うには、以下のようにモデルにフラグ値を入れて、ビューに渡します。

@ModelAttribute("isSomeFunctionalityEnabled")
public void isSomeFunctionalityEnabled() {
    return configurationManager.getParameter("is.some.functionality.enabled");
}

次に、渡された値を使用して、これまたはそのHTMLコードをレンダリングします。

<c:choose>
  <c:when test="${isSomeFunctionalityEnabled}">
    <!-- Render some stuff -->
  </c:when>    
  <c:otherwise>
    <!-- Render some other stuff -->
  </c:otherwise>
</c:choose>

機能スイッチャーのタイプ


説明されている機能スイッチャーの使用の概念は、1つの使用例にすぎず、そのような機能スイッチャーはリリーストグルと呼ばれます。機能スイッチャーには、合計3つのタイプがあります。

  • リリーストグル -開発中に不完全に実装された機能を非表示にできます
  • 実験の切り替え-A / Bテスト用のスイッチ
  • 権限の切り替え -さまざまなユーザーグループの機能のオン /オフスイッチ

したがって、機能スイッチャーを使用すると、異なるデータベースと機能スイッチャーの異なるセットを使用して、同じコードベースで2つの異なるバージョンのサイトを構築できます。たとえば、ヨーロッパのサイトではGDPRに関連するすべての機能を含めることは理にかなっていますが、ロシアのサイトではこれを行うことはできません。



機能トグルの使用に関する問題


私は機能トグルが積極的に使用されているプロジェクトに取り組んでいるので、機能トグルを使用することの明らかな利点に加えて、それらに関連する問題に気づき始めました：

  • : , QA , , , feature switcher-. ,
  • : feature toggle- , , , «»
  • 予期しないサイトの内訳：古い機能スイッチャーの多くは、値を変更するときに何かを壊すという残念な性質があります（動作することを長い間確認していないため）。機能スイッチャーはデータベースに格納されており、ビジネスユーザーが管理パネルから簡単に変更できるため、値の変更により故障が発生することがよくあります。長期間使用されていない機能スイッチャーのパフォーマンスは、最初にテスト環境で確認する必要があります

説明されている問題のいくつかの解決策。


以下のアクションは、上記の問題の解決に役立ちます。

  • 利用可能な機能スイッチャーのドキュメント：特定の機能トグルがどのような効果を持っているか、およびデータベースでそれを探すためのキーを理解するには、すべての機能スイッチャーを説明する詳細なドキュメントを作成する必要があります。
  • 機能スイッチャーの定期的な改訂：デッドコードの出現を回避するために、廃止された機能スイッチャーと関連コードを定期的に削除します

概要


機能スイッチャーは非常にシンプルでありながら強力なメカニズムであり、巨大なコミットを回避したり、アプリケーションの動作を簡単に変更したり、機能トグルの異なる構成を使用して同じコードベースで複数の異なるアプリケーションをアセンブルしたりできます。

ただし、この開発パターンにはコードの読み取りや保守が困難になる欠点があるため、このパターンの過度の使用は避け、定期的に機能スイッチャーとそのリビジョンを文書化して未使用のものを削除することを忘れないでください。その結果、 「デッド」コードからプロジェクトをクリアします。
