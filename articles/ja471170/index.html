Maybeコンテナのファンクタ実装の例を考えてみましょう。これは「欠落している可能性のある値」の考え方です。


const compose = <A, B, C>(
   f: (a: A) => B,
   g: (b: B) => C,
 ): (a: A) => C => (a: A) => g(f(a));


//  Maybe:
type Nothing = Readonly<{ tag: 'Nothing' }>;
type Just<A> = Readonly<{ tag: 'Just'; value: A }>;
export type Maybe<A> = Nothing | Just<A>;

const nothing: Nothing = { tag: 'Nothing' };
const just = <A>(value: A): Just<A> => ({ tag: 'Just', value });

//    Maybe:
const fmap = <A, B>(f: (a: A) => B) =>
  (fa: Maybe<A>): Maybe<B> => {
    switch (fa.tag) {
      case 'Nothing':
        return nothing;
      case 'Just':
        return just(f(fa.value));
    }
};

//  1: fmap id === id
namespace Laws {
 console.log(
   fmap(id)(just(42)),
   id(just(42)),
 ); // => { tag: 'Just', value: 42 }


 //  2: fmap f ◦ fmap g === fmap (f ◦ g)
 const f = (a: number): string => `Got ${a}!`;
 const g = (s: string): number => s.length;
 console.log(
   compose(fmap(f), fmap(g))(just(42)),
   fmap(compose(f, g))(just(42)),
 ); // => { tag: 'Just', value: 7 }
}

メソッドfmapは2つの側面から見ることができます。

  1. 純粋な関数を「コンテナー化された」値に適用する方法として。
  2. 純粋な関数を「コンテナコンテキストに引き上げる」方法として。

実際、インターフェースの括弧がわずかに異なる場合、次の関数シグネチャを取得できますfmap。

const fmap: <A, B>(f: (a: A) => B) => ((ma: Maybe<A>) => Maybe<B>);

インターフェースを定義した後：

type Function1<Domain, Codomain> = (a: Domain) => Codomain;

fmap次の 定義が得られます。

const fmap: <A, B>(f: (a: A) => B) => Function1<Maybe<A>, Maybe<B>>;

この単純なトリックにより、ファンクターを「純粋な関数をコンテナーコンテキストに引き上げる」方法として考えることができます。これにより、さまざまな種類のデータを安全な方法で処理できます。たとえば、オプションのネストされた値のチェーンを正常に処理できます。データリストを変換する例外などを処理します。

前に説明したように、ファンクタを使用すると、関数を安全かつ不変の値に適用できます。モナドはファンクタに似ていますが、特別な場合に特別なロジックを委任できる点が異なります。ファンクター自身は、この関数を適用して結果をシェルにラップする方法を知っているだけで、追加のロジックはありません。

モナドは、シェルから値を抽出し、ネストからルールを定義するという原則によってデータを抽出するという原則によってデータ型全体を作成するときに発生します。ファンクタと同様に、モナドは処理された値がまったくわからない一連のステージの形式で計算を記述するために使用されるデザインテンプレートですが、コンポジションで使用されたときにデータフローを安全かつ副作用なしに制御できるようにするのはモナドです。モナドはさまざまな問題を解決することを目的とすることができます。理論的には、モナドは特定の言語の型システムに依存します。実際、多くの人々は、明示的なデータ型がある場合にのみ理解できると考えています。
