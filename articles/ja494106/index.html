健康な人のためのページオブジェクトクラスを作成するためのヒント

また会いましたね。Java QAエンジニアコースの開始を見越して、別の有用な記事の翻訳を用意しました。



これは新年の始まりであり、世界中の多くの人々が自分の健康にもっと注意を払うことを決めます。大丈夫だよ！実際、フロントエンドのテストにもそのようなソリューションが必要です。

Web UIテストコードベースで使用される最も一般的なデザインパターンは、ページオブジェクトモデル（POM）パターンです。このパターンには、テスト中のシステムの1ページを表すクラスモデリングが含まれます。このモデルに基づいて、クラスには、ユーザーインターフェイスページの要素とこれらの要素とやり取りするメソッドを表すプロパティが含まれます。

アプリケーションのこのログインページを例に、POMを使用してクラスを作成するためのヒントについて説明します。



クラス


このページを表すために、自動化環境で唯一のクラスを作成します。他の開発者が特定のユーザーインターフェイスページに関連付けられているクラスをすばやく見つけられるように、クラス名がアプリケーションのページと一致していることを確認することをお勧めします。

したがって、この例では、LoginPageというクラスを作成します。

public class LoginPage {
    
    private WebDriver driver;
 
    public LoginPage(WebDriver driver){
        this.driver = driver;
    }
}

プロパティ


LoginPageには、テストで使用されるページの各要素のプロパティが含まれている必要があります。これらのプロパティの値は、実際のページの要素へのポインターです。これらのロケーターを1か所で定義することにより、使用するすべての場所でロケーターをハードコーディングする必要がなくなり、重複を排除できます。

private By usernameField        = By.id("username");
private By passwordField        = By.id("password");
private By signInButton     	= By.id("log-in");
private By rememberMeCheckbox   = By.id("rememberMe");
private By twitterIcon          = By.id("twitterIcon");
private By facebookIcon     	= By.id("fbIcon");
private By linkedInIcon     	= By.id("linkedInIcon");
private By errorMessage     	= By.id("alert");

方法


プロパティに加えて、クラスには、入力フィールドへの入力やボタンのクリックなど、テストがアプリケーションと対話できるようにするメソッドも含める必要があります。テストで最適に使用するためにこれらのメソッドを開発するためのヒントをいくつか紹介します。

ゲッターとセッターを追加する


POMクラスの目的は、アプリケーションの状態を設定（設定）および取得（取得）することなので、これに適したメソッドが必要になります。

たとえば、ユーザー名とパスワードのフィールドを設定する必要があるため、それらのセッターを追加する必要があります。

public void setUsername(String username) {
    driver.findElement(usernameField).sendKeys(username);
}
 
public void setPassword(String password) {
    driver.findElement(passwordField).sendKeys(password);
}

public String getErrorMessage() {
    return driver.findElement(errorMessage).getText();
}

現在必要なものだけを追加してください


テストではそれらを必要としないため、各フィールドにゲッターとセッターを追加しなかったことに注意してください。現時点で必要なものだけを追加してください。新しいテストに必要な場合は、いつでも追加できます。これにより、未使用のコードを維持する必要がなくなります。

遷移は新しいオブジェクトを返す必要があります


ボタンやリンクをクリックするためのメソッドも必要です。ただし、クリックによってページが変更される場合、メソッドはクリックしたユーザーインターフェイスページを表すクラスへのハンドルを返す必要があります。

たとえば、ログインボタンを押すとアプリケーションのメインページが表示される場合、メソッドはこのメインページを表すクラスへのハンドルを返す必要があります。

public HomePage clickSignInButton() {
    driver.findElement(signInButton).click();
    return new HomePage();
}

この推奨事項のサポートのおかげで、すべての呼び出しテストは、このメソッドが呼び出されると遷移が発生し、テストがアプリケーションとの対話を継続するために必要なオブジェクトへのハンドルを持つことを認識します。

便宜上、メソッドを作成することを恐れないでください。


現在、アプリケーションに入るには、少なくとも3つのメソッド（setUsername、setPasswordおよびclickSignInButton）を呼び出す必要があります。これらは一般的に一緒に使用されるため、テストで1つのメソッドのみを呼び出すことができるように、クラスに便利なログインメソッドを提供することは理にかなっています。

public HomePage login(String username, String password) {
    setUsername(username);
    setPassword(password);
    return clickSignInButton();
}

このメソッドにはロジックを実装していませんが、代わりに個々のメソッドを呼び出していることに注意してください。これらのメソッドを個別に使用できるようにして、ネガティブテストで使用できるようにすることが重要です（たとえば、パスワードではなくユーザー名を設定します）。

条件を考慮する


チェックボックス要素（または他のスイッチ）の場合、それらをクリックするだけでは、目的のテストの設計に対応しない場合があります。たとえば、テストでチェックボックスをインストールしたいが、すでにインストールされている場合はどうなりますか？この場合、clickメソッドを呼び出すと、反対の結果が得られます。代わりに、ユーザーが必要な場合にのみチェックボックスをオンにするロジックを導入して、メソッドをより複雑にします。これを行うには、ブラインドクリックの代わりに、意図を示す引数を受け入れ（つまり、選択する必要があるかどうか）、それに応じて動作します。

public boolean isRememberUsernameChecked() {
    return driver.findElement(rememberMeCheckbox).isSelected();
}
 
public void setRememberUsername(boolean checkIt) {
    boolean alreadySelected = isRememberUsernameChecked();
 
    //,         
    if( (checkIt && !alreadySelected) || (!checkIt && alreadySelected) ) {
        driver.findElement(rememberMeCheckbox).click();
    }
}

POMクラスを中立に保つ


POMクラスは、アプリケーションに関する情報を操作および収集する中立オブジェクトである必要があります。メソッドをニュートラルに保ち、ページとやり取りしたいすべてのテストで使用できる十分な用途があることを確認してください。つまり、これらのメソッドにはステートメントを含めないでください。

たとえば、メッセージが表示されている場合、エラーメッセージを受信するメソッドもテストに失敗するとどうなるでしょうか。

//   .   
public String getErrorMessage() {
    String message = driver.findElement(errorMessage).getText();
    if(message.isDisplayed()){
        assert.fail(message);
    }
    return message;
}

ユーザー名またはパスワードが正しくないか指定されていない場合にエラーメッセージが表示されることを確認したいテストでは、この方法を再利用できません。これは、POMクラスが、エラーメッセージの表示がエラーであると判断するタスクを引き受けたためです。これらの場合ではそうではありません。代わりに、単に状態を返し、テストにそれが何を意味するかを判断させます。

public String getErrorMessage() {
    return driver.findElement(errorMessage).getText();
}

健全なコードが健全なテストに先行する


これらのヒントは、ページオブジェクトモデルのデザインパターンを実装する実用的なクラスを開発するのに役立ちます。このパターンを使用すると、権限の分離（テストや状態管理など）が容易になり、コードの重複が減少し、再利用可能なメソッドが提供されます。いいテストを！
