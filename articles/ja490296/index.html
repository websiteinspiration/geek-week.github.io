私たちの実践ではウィンドウ関数は確立されていますが、RANGEフレームとROWSフレームがどのように機能するかを知っている人はほとんどいません。

おそらくそれが、彼らがあまり一般的でない理由です。この記事の目的は、「誰がだれなのか」という質問が確実になくなるように、使用例を提供することです。と「それを適用する方法？」「なぜ？」という質問 記事は点灯しません。

フレームとは何か、そしてOVER（）句でORDER Byを使用して同様の効果を達成する方法を見てみましょう。

デモでは、単純なテーブルを使用して、コンパイラを使用せずに例を計算できるようにします。一般的に、私はそれを強くお勧めします-実行の結果が何であるかを見て考えてから、自分で確認してください-ウィンドウ関数の知覚に白い斑点を見つけます。これは、完成した結果を読んだときにはっきりしない場合があります。

SELECT sales_id, customer_id, count, 
SUM(count) OVER () as total,

最初の目立たないレーキを見てみましょう。コードを読み取るときに、cumとcum_uniqが同じであると考える開発者の数をどう思いますか？少し考えますか？おそらく、しかし、ここではそれが明らかであり、アプリケーションでコードを読み取るときに、また、ソートフィールドのそれほど一意ではない非明示性があったとしても、それは明らかです。

今私たちの素晴らしいウィンドウを開きます。

ウィンドウ、つまりフレームは、ROWSとRANGEの2種類です。最初にROWSを知ってください。
フレーム制限オプション：

  1. 現在の行/範囲の前のすべて、および現在の行の実際の値
    BUNWEEN UNBOUNDED PRECEDING
    BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  2. 現在の行/範囲、および現在の行
    とアンバウンドの次の間のすべて
  3. 含める前後の行数の指定（RANGEではサポートされていません）
    BETWEEN N先行AND N後続
    BETWEEN CURRENT ROW AND N後続
    BETWEEN N先行AND CURRENT ROW

しかし、実際のフレームを見てみましょう。

現在の行とそれ以前のすべての行で「ウィンドウ」を開きます。SUM関数では、ご覧のとおり、これはASCソートと一致しています。

SELECT sales_id, customer_id, cnt, 
SUM(cnt) OVER (ORDER BY customer_id, sales_id) AS cum_uniq,
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS UNBOUNDED PRECEDING) AS current_and_all_before,
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS current_and_all_before2
FROM sales
ORDER BY customer_id, sales_id;



同時に、ウィンドウ内（OVER（）句内）の並べ替え順序はクエリ自体の並べ替え順序とは関係がないことを思い出してください。例では、計算を確認して関数の動作を理解すると、計算を簡略化するために同じになります。

SELECT sales_id, customer_id, cnt, 
SUM(cnt) OVER (ORDER BY customer_id, sales_id) AS cum_uniq,
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS UNBOUNDED PRECEDING) AS current_and_all_before,
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS current_and_all_before2
FROM sales
ORDER BY cnt;



次に、後続のすべての行を含めたときのフレームの機能を見てみましょう。 

SELECT sales_id, customer_id, cnt, 
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING) AS current_and_all_frame,
SUM(cnt) OVER (ORDER BY customer_id DESC, sales_id DESC) AS current_and_all_order_desc
FROM sales
ORDER BY customer_id, sales_id;



ここでわかるように、逆ソートを使用すると、集約関数でも同じ結果が得られますが、ROWSの方が少し安定しています。これは、ソートフィールドが一意でない場合、同じ値の形式で驚きを得ることができるためです。

そして最後に、フレームに含める必要がある特定の行数を指定すると、並べ替えでは模倣できなくなるオプション

SELECT sales_id, customer_id, cnt, 
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS BETWEEN 1 PRECEDING AND CURRENT ROW) AS before_and_current,
cnt,
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS BETWEEN CURRENT ROW AND 1 FOLLOWING) AS current_and_1_next,
cnt,
SUM(cnt) OVER (ORDER BY customer_id, sales_id ROWS BETWEEN 2 PRECEDING AND 2 FOLLOWING) AS before2_and_2_next
FROM sales
ORDER BY customer_id, sales_id;



このオプションでは、範囲内にある行をすでに具体的に示していますが、私の意見では、結果から最も明白です。

ROWSとRANGEの違い


違いは、ROWSは行で動作し、RANGEは範囲で動作することです。確かに、これは名前から明らかですが、実際にはほとんど説明しませんか？

写真（記事の下部にあるソース）





を見てみましょう。注意深く見ると、並べ替えパラメーターの同じ値を持つ行が範囲と呼ばれることが明らかになります。

前述のように、ROWSは文字列に制限されていますが、RANGEはORDER BYウィンドウ関数で指定した一致する値の範囲全体をキャプチャします。

SELECT sales_id, customer_id, cnt, 
SUM(cnt) OVER (ORDER BY customer_id) AS cum_uniq,
cnt,
SUM(cnt) OVER (ORDER BY customer_id ROWS UNBOUNDED PRECEDING) AS current_and_all_before,
customer_id,
cnt,
SUM(cnt) OVER (ORDER BY customer_id RANGE UNBOUNDED PRECEDING) AS current_and_all_before2
FROM sales
ORDER BY 2, sales_id;



ROWS-並べ替えが一意でなくても、常に特定の行で動作しますが、RANGEは、期間を並べ替えフィールドの一致する値を持つ範囲に結合するだけです。この意味で、機能はSUM（）関数の動作と非常に似ており、一意でないフィールドでソートされます。別の例を見てみましょう。

SELECT sales_id, customer_id, price_per_item, cnt, 
SUM(cnt) OVER (ORDER BY customer_id, price_per_item) AS cum_uniq,
cnt,
SUM(cnt) OVER (ORDER BY customer_id, price_per_item ROWS UNBOUNDED PRECEDING) AS current_and_all_before,
customer_id,
cnt,
SUM(cnt) OVER (ORDER BY customer_id, price_per_item RANGE UNBOUNDED PRECEDING) AS current_and_all_before2
FROM sales
ORDER BY 2, price_per_item;




すでに2つのフィールドがあり、範囲は両方のフィールドの値が一致する範囲によって決定されます。

オプションは、現在の行からの後続のすべての行を計算に含める場合で、SUM関数の場合、逆ソートを使用して取得できる値と一致します。

SELECT sales_id, customer_id, price_per_item, cnt, 
SUM(cnt) OVER (ORDER BY customer_id DESC, price_per_item DESC) AS cum_uniq,
cnt,
SUM(cnt) OVER (ORDER BY customer_id, price_per_item ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING) AS current_and_all_before,
customer_id,
cnt,
SUM(cnt) OVER (ORDER BY customer_id, price_per_item RANGE BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING) AS current_and_all_before2
FROM sales
ORDER BY 2, price_per_item, sales_id desc;



ご覧のように、RANGEは一致するペアの範囲全体を再度キャプチャします。

ROWSとRANGEの機能は毎回新しいものではありませんが、その使用方法について疑問が生じます。この記事でROWSとRANGEの違いについての理解を深めていただければ幸いです。これで、このフレームまたはそのフレームが必要であることに疑いの余地はありません。
