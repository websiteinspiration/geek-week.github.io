Kubernetes 1.16：ハイライトの概要



今日、水曜日に、Kubernetes の次のリリースが開催されます -1.16。私たちのブログのために発展してきた伝統によると、10周年の時のために、私たちは新しいバージョンの最も重要な変更について話しています。

この資料を準備するために使用される情報は、Kubernetes拡張追跡テーブル、CHANGELOG-1.16および関連する問題、プルリクエスト、ならびにKubernetes拡張提案（KEP）から取得されます。じゃ、行こう！ ..

結び目


K8sクラスター（Kubelet）のノードの側には、（アルファバージョンステータスで）非常に多くの注目すべきイノベーションが提示されています。

最初に、ポッドでのデバッグプロセスを簡略化するために設計された、いわゆる「エフェメラルコンテナ」（Ephemeral Containers）が表示されます。新しいメカニズムを使用すると、既存のポッドの名前空間で始まり、短期間存続する特別なコンテナーを実行できます。それらの目的は、他のポッドやコンテナと対話して、問題とデバッグを解決することです。この機能のためにkubectl debug、本質的にkubectl exec次のように新しいコマンドが実装されました：（ケースのように）コンテナーでプロセスを開始する代わりにexec）ポッドでコンテナを起動します。たとえば、次のコマンドは新しいコンテナをポッドに接続します。

kubectl debug -c debug-shell --image=debian target-pod -- bash

エフェメラルコンテナーの詳細（およびその使用例）は、対応するKEPにあります。現在の実装（K8s 1.16）はアルファ版であり、ベータ版への移行の基準の1つは、「少なくとも2つのリリース[Kubernetes]のEphemeral Containers APIのテスト」です。

NB：本質や機能の名前さえ既存の似てkubectl-デバッグプラグイン私たちは、すでに書いたが。エフェメラルコンテナの出現により、個別の外部プラグインの開発が停止することが想定されています。

別のイノベーション-- ポッドのオーバーヘッドコストを計算PodOverheadするメカニズムを提供するように設計されています、使用するランタイムによって大きく異なります。例として、このKEPの作者は、ゲストカーネル、kataエージェント、initシステムなどの起動を必要とするカタコンテナーを引用しています。オーバーヘッドが非常に大きくなると、それを無視することはできません。つまり、追加の割り当て、計画などのためにオーバーヘッドを考慮する方法が必要になります。その実装のためにPodSpec、フィールドが追加されましたOverhead *ResourceList（RuntimeClass使用されている場合、のデータと比較）。

もう1つの注目すべきイノベーション- マネージャーノードトポロジ （ノードトポロジマネージャー）、Kubernetesのさまざまなコンポーネントのハードウェアリソースの分散を微調整するアプローチを統合するように設計されています。このイニシアチブは、高性能の並列コンピューティングと、CPUおよびハードウェアアクセラレーションの高度な機能を使用する操作の実行の遅延を最小限に抑えるための（電気通信、機械学習、金融サービスなどの）さまざまな最新システムの需要の高まりによって引き起こされます。 Kubernetesのこのような最適化は、これまでさまざまなコンポーネント（CPUマネージャー、デバイスマネージャー、CNI）のおかげで達成されており、アプローチを統合し、Kubelet側の新しい同様のコンポーネント（いわゆるトポロジ対応）の接続を簡素化する単一の内部インターフェイスを追加します。詳細は対応するKEPにあります。


トポロジマネージャーコンポーネントのレイアウト

次の機能は、起動時にコンテナーをチェックすることです（起動プローブ）。ご存知のように、長時間実行されるコンテナの場合、現在のステータスを取得することは困難です。実際の運用開始前に「強制終了」されるか、長時間デッドロックに陥ります。新しいチェック（と呼ばれる機能ゲートを介して有効化StartupProbeEnabled）は、ポッドが起動を完了する瞬間まで、他のチェックのアクションをキャンセルします。このため、この機能は元々はpod-startup liveness-probe holdoffと呼ばれていました。開始時間が長いポッドの場合、比較的短い時間間隔で状態調査を実行できます。

さらに、すぐにベータステータスになると、RuntimeClassの改善が追加され、「異種クラスター」のサポートが追加されます。RuntimeClassスケジューリング、ポッドのために、クラスタ・トポロジ考えずRuntimeClassを選択することができます。今、すべてのノードは、各RuntimeClassのサポートを持っている必要はありません。以前は、これを達成するために-ポッドが必要なすべてをサポートするノードに表示されるようにするには、適切なルールをNodeSelectorと容認に割り当てる必要がありました。KEPはもちろんの使用例と、実装の詳細を説明します。

その他の2つの機能がベータ版になりました：サーバー側の適用とブックマークの監視。

アルファバージョンでのみ有意な革新であった拒絶のSelfLink指定されたオブジェクトを表し、の一部であり、特別なURI ObjectMetaとListMeta（Kubernetes内の任意のオブジェクトの、すなわち部分）。なぜそれを拒否するのですか？シンプルな動機の音この分野が存続する本当の（乗り越えられない）理由がないからです。より正式な理由は、パフォーマンスを最適化（不要なフィールドを削除）し、generic-apiserverの作業を簡略化することです。このため、このようなフィールドを特別な方法で処理する必要があります（これは、オブジェクトがシリアル化される直前に設定される唯一のフィールドです）。現在の「陳腐化」（ベータ版）SelfLinkはKubernetesバージョン1.20と最後の1.21で発生します。
