redux-sagaのテストの変容

このフレームワークredux-sagaは、副作用を処理するための興味深いパターンを多数提供しますが、真の流血のエンタープライズ開発者として、コード全体をテストでカバーする必要があります。サガをテストする方法を考えてみましょう。



例として最も単純なクリッカーを取り上げます。データフローとアプリケーションの意味は次のとおりです。

  1. ユーザーがボタンを突きます。
  2. リクエストがサーバーに送信され、ユーザーがボタンを押したことが通知されます。
  3. サーバーは、行われたクリックの数を返します。
  4. 州は、行われたクリックの数を記録します。
  5. UIが更新され、クリック数が増加したことがユーザーに表示されます。
  6. ...
  7. 利益。

私たちの作業ではTypescriptを使用するため、すべての例はこの言語で記述されます。

ご想像のとおり、このすべてをヘルプで実装しますredux-saga。次に、sagasファイル全体のコードを示します。

export function* processClick() {
    const result = yield call(ServerApi.SendClick)
    yield put(Actions.clickSuccess(result))
}

export function* watchClick() {
    yield takeEvery(ActionTypes.CLICK, processClick)
}

この簡単な例ではprocessClick、アクションを直接処理するサガwatchClickと、サイクルを作成するサガを宣言していますaction’。

発電機


だから私たちは最も単純な物語を持っています。リクエストをサーバーに送信し( call)、結果を受け取ってレデューサーに渡し( put)ます。サガがサーバーから受け取ったものを正確に送信しているかどうかをテストする必要があります。始めましょう。

テストのために、サーバーの呼び出しをロックし、サーバーからの結果がリデューサーに入ったかどうかを確認する必要があります。

sagasはジェネレーター関数であるため、テストするための最も明白な方法next()は、プロトタイプジェネレーターにあるメソッドです。この方法を使用する場合、ジェネレーターから次の値を受け取り、ジェネレーターに値を転送する機会があります。したがって、私たちは箱から出して、ウェットコールを受ける機会を得ます。しかし、すべてがとてもバラ色ですか？これが私がベアジェネレーターで書いたテストです：

it('should increment click counter (behaviour test)', () => {
    const saga = processClick()

    expect(saga.next().value).toEqual(call(ServerApi.SendClick))
    expect(saga.next(10).value).toEqual(put(Actions.clickSuccess(10)))
})

テストは簡潔でしたが、何をテストしますか？実際、それは単にsagaメソッドのコードを繰り返すだけです。つまり、sagaの変更はテストを変更する必要があります。

このようなテストは開発に役立ちません。

Redux-saga-test-plan


この問題に遭遇した後、私たちはそれをグーグルすることに決め、突然私たちだけのものではなく、最初から遠いことに気付きました。開発者向けのドキュメントで直接、redux-sagaテストファンを満足させるために特別に作成されたいくつかのライブラリを紹介しています。

提案されたリストから私たちは図書館を選びましたredux-saga-test-plan。これが私が書いたテストの最初のバージョンのコードです。

it('should increment click counter (behaviour test with test-plan)', () => {
    return expectSaga(processClick)
        .provide([
            call(ServerApi.SendClick), 2]
        ])

        .dispatch(Actions.click())

        .call(ServerApi.SendClick)
        .put(Actions.clickSuccess(2))

        .run()
    })

のテストコンストラクターは、テストを記述するインターフェイスを返すredux-saga-test-plan関数expectSagaです。（processClick最初のリストの）テストサガが関数自体に渡されます。

このメソッドを使用して、provideサーバー呼び出しまたはその他の依存関係をブロックできます。の配列がit StaticProvider’に渡され、どのメソッドが返すかが記述されています。

ブロックActには、1つのメソッド-がありdispatchます。アクションがそれに渡され、サーガはそれに応答します。

ブロックassertはcall put、対応する効果がサガの作業中に引き起こされたかどうかをチェックするメソッドで構成されています。

それはすべてメソッドで終わりrun()ます。このメソッドは、テストを直接実行します。

このアプローチの利点：

  • 一連の呼び出しではなく、メソッドが呼び出されたかどうかをチェックします。
  • mokiは、どの関数が濡れて何が戻るかを明確に説明しています。

ただし、行うべき作業があります。

  • より多くのコードがあります。
  • テストは読みにくいです。
  • これは振る舞いのテストです。つまり、それはまだサガの実装と接続されています。

最後の2つのストローク


状態試験


まず、最後を修正します。動作テストから状態テストを作成します。この事実は私たちを助け、それtest-planは私たちがイニシャルstateと送信を設定することを可能にしreducer、それはputサガによって生成された効果に応答するはずです。次のようになります。

it('should increment click counter (state test with test-plan)', () => {
    const initialState = {
        clickCount: 11,
    

    return expectSaga(processClick)
        .provide([
            call(ServerApi.SendClick), 14]
        ])
        .withReducer(rootReducer, initialState)

        .dispatch(Actions.click())

        .run()
        .then(result => expect(result.storeState.clickCount).toBe(14))
})

このテストでは、エフェクトがトリガーされたことを確認しなくなりました。実行後の最終状態をチェックしているのは素晴らしいことです。

サーガの実装をなんとか取り除くことができたので、テストをより理解できるようにしましょう。これを次のように置き換えるthen()と簡単ですasync/await。

it('should increment click counter (state test with test-plan async-way)', async () => {
    const initialState = {
        clickCount: 11,
    }    

    const saga = expectSaga(processClick)
        .provide([
            call(ServerApi.SendClick), 14]
        ])
        .withReducer(rootReducer, initialState)

    const result = await saga.dispatch(Actions.click()).run()

    expect(result.storeState.clickCount).toBe(14)
})

統合テスト


しかし、リバースクリック操作（unclickと呼ぶことにしましょう）も取得し、サグファイルが次のようになった場合はどうなるでしょうか。

export function* processClick() {
    const result = yield call(ServerApi.SendClick)
    yield put(Actions.clickSuccess(result))
}

export function* processUnclick() {
    const result = yield call(ServerApi.SendUnclick)
    yield put(Actions.clickSuccess(result))
}

function* watchClick() {
    yield takeEvery(ActionTypes.CLICK, processClick)
}

function* watchUnclick() {
    yield takeEvery(ActionTypes.UNCLICK, processUnclick)
}

export default function* mainSaga() {
    yield all([watchClick(), watchUnclick()])
}

クリックアクションとアンクリックアクションが状態で呼び出されたときに、サーバーへの最後のトリップの結果が状態に書き込まれることをテストする必要があるとします。このようなテストは、次のものでも簡単に実行できますredux-saga-test-plan。

it('should change click counter (integration test)', async () => {
    const initialState = {
        clickCount: 11,
    }            

    const saga = expectSaga(mainSaga)
        .provide([
            call(ServerApi.SendClick), 14],
            call(ServerApi.SendUnclick), 18]
        ])
        .withReducer(rootReducer, initialState)

    const result = await saga
        .dispatch(Actions.click())
        .dispatch(Actions.unclick())
        .run()

    expect(result.storeState.clickCount).toBe(18)
})

現在mainSaga、個々のアクションハンドラではなく、をテストしていることに注意してください。

ただし、このテストをそのまま実行すると、コーニングが表示されます。



これは、効果が原因です。これはtakeEvery、アプリケーションが開いているときに機能するメッセージ処理サイクルです。したがって、それが呼び出されたテストは、takeEvery外部の支援なしでは作業を完了できずredux-saga-test-plan、テストの開始から250 ms後にそのような影響の動作を強制的に終了します。このタイムアウトは、expectSaga.DEFAULT_TIMEOUT = 50を呼び出すことで変更できます。
あなたは、複雑な効果を持つ各テストの1は、そのようなvoringsを受信したくない場合は、単に代わりの方法を使用run()する方法をsilentRun()。
