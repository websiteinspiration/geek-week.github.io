Docker API + Portainer API =利益


全てにおいて良い日！この記事では、スムージーモードでのプロジェクトデプロイメントにおけるPortainer APIとアプリケーションの機能について簡単に説明します。

いくつかのタスクを検討してください：

  • コントロール。特定のノード上のすべてのコンテナの操作を表示し、ステータスを管理および監視し、ログを読み取り、リソースを制御する機能があります。
  • 展開。ユーザーの参加を最小限に抑え、新しい場所での自動展開に適した条件を作成します。
  • gitlab-ciとの統合。開発プロセスを簡素化します。

コントロール


Dockerの経験があるため、docker-swarmへの切り替えは、Kubernetesに比べて比較的単純で開発のスピードが速くなっています。1つのリーダーノードと複数のワーカーについて、起動ステータス、ジョブ数、メモリ使用量統計、およびヘルスチェックを1つのウィンドウに表示する方法を決定する必要がありました。まず、タスクのニーズに基づいたグラフィカルインターフェイスを備えたターンキーソリューションが必要です。

Dockerを管理するためのGUIインターフェースの比較に関する多くの記事を見つけることができます。いくつか例を挙げましょう。こことここ。問題を解決するために必要な要件は、Portainerで見つかりました。

良い点：

  • 導入の容易さ
  • すぐに使用できる認証
  • API
  • 独自のレジストリを追加する機能
  • コンテナコンソールに接続する機能
  • ログを読む
  • 使用済みメモリの特性とコンテナのネットワークを追跡します。

Portainerは、Portainer ServerとPortainer Agentの2つの要素で構成されています。。どちらの要素もDockerコンテナーのように機能します-デプロイするのに十分です。一般的に、Portainerの作業は、すべてのノードにエージェントを配置し、httpリクエストを介して通信することに基づいています。ドキュメントから、Portainer AgentはDocker APIを制限してその環境を管理するための回避策であることが明らかになります。コンテナ、ネットワーク、ボリューム、イメージとの対話は、Docker APIが対象とする現在のノードに限定されているためです。これに対する解決策は、例えば、各ノードでポートを開いて、それを証明書で保護し、通信スキームを確立することなどです。つまり、便利ではありませんが、オプションです。エージェントのタスクは、以前に定義されたノードのクラスター指向のリソースと通信する機能を提供することです。クラスタ内の各ノードからこれらのすべてのリソースを取得するには、Docker APIリクエストを1つだけ完了するだけで十分です。



一般に、Swarmクラスターを管理するときのユーザーエクスペリエンスは向上します。もう少し、スクリーンショットを使用すると、ここまたはここに表示されます。

展開


（スクリプトとスタックへのリンクは下に残しておきます）。

いくつかの例を挙げて、配備の利便性を確認できます。Dockerでスタックをロードおよびアンロードするタスクを検討してください。ここで、次のコマンドで標準のDocker APIメソッドを使用できることに注意してください。

docker stack deploy -c "  "

この場合、スタックはロードされますが、Portainerでは編集のためにブロックされ、これもタスクのニーズを満たしません。次に、Portainer APIは、スタックをダウンロードする機能を備えています。便宜上、マネージャのUNIXマシンにhttpieおよびjqパッケージをインストールするか、クラシックカールを使用することをお勧めします。ログインとパスワードをスクリプトに渡してPortainerに入り、次のコマンドを実行します。

sh script.sh Username Password

まず、認証キーを取得します。

key=$(http POST 127.0.0.1:9000/api/auth Username="$1" Password="$2" | jq -r '.jwt')

swarm_id（スタックが飛ぶアドレス）を取得します。

swarm_id=$(http GET 127.0.0.1:9000/api/endpoints/1/docker/swarm "Authorization: Bearer $key" | jq -c '.ID')

次に、いくつかの二次情報を取得する方法に沿って、ブートプロセスを開始します。

for file in stacks/*  
	do
	dir=$(readlink -f "$file")
filename=$(basename "$file" .yml)
	curl -X POST "http://127.0.0.1:9000/api/stacks?method=file&type=1&endpointId=1" -H "Content-Type: multipart/form-data" -H "Accept: application/json" -H "Authorization: Bearer $key" -F "Name=$filename" -F "SwarmID=$swarm_id" -F "file=@$dir"

その後、編集可能なスタックがPortainerに表示されます。ユーザーがPortainerを使用してスタックに変更を加えた場合も、逆になります。これらの変更を取得するには、スタックをアンロードしてymlファイルに保存し、他のスタックがあるディレクトリに保存する必要があります。これを行う方法：

キーを取得します。

key=$(http POST 127.0.0.1:9000/api/auth Username="$2" Password="$3" | jq -r '.jwt')

すべてのスタックを実行し、名前とIDを取得して保存します。

for var in $(http GET 127.0.0.1:9000/api/stacks "Authorization: Bearer $key" | jq -c '.[]?')
  do
    name=$(echo $var | jq -r '.Name')
    id=$(echo $var | jq -r '.Id')
   http GET 127.0.0.1:9000/api/stacks/$id/file "Authorization: Bearer $key" | jq -r '.StackFileContent' > $name.yml
  done

誰でもPortainer APIツールのアプリケーションを見つけることができます。

テスト中


たとえば、gitlab-runnerが使用されました。次のgitlab-ciのアプリケーションを検討してください。service.shスクリプトで、上記の状況の実行を記録して実行します。

stages:
  - staging
stage-short:
  stage: staging
  image: [SOME_STAGE]
  only:
    changes:
      - [SOME_PROJECT]/*   
    refs:
      - master
  script:  
    - /scripts/service.sh $PORTAINER_ADMIN $PORTAINER_PASS
