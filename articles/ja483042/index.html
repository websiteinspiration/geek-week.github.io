9 Vueパフォーマンスのヒント

この記事には、Vueアプリケーションのパフォーマンスを向上させ、表示速度を上げ、バンドルのサイズを小さくするための9つのヒントがあります。

コンポーネントを操作する


機能部品


単純な小さなコンポーネントがあるとします。渡された値に応じて、彼はすべて特定のタグを表示します。

<template>
  <div>
    <div v-if="value"></div>
    <section v-else></section>
  </div>
</template>

<script>
export default {
  props: ['value']
}
</script>

このコンポーネントは、機能属性を追加することで最適化できます。機能コンポーネントは単純な関数にコンパイルされ、ローカル状態はありません。これにより、そのパフォーマンスははるかに高くなります。

<template functional>
  <div>
    <div v-if="props.value"></div>
    <section v-else></section>
  </div>
</template>

<script>
export default {
  props: ['value']
}
</script>

Vue v3.0での機能コンポーネントの例

子コンポーネントへの分離


複雑なタスクを実行するために必要なコンポーネントを表示するコンポーネントを想像してください。

<template>
  <div :style="{ opacity: number / 300 }">
    <div>{{ heavy() }}</div>
  </div>
</template>

<script>
export default {
  props: ['number'],
  methods: {
    heavy () { /*   */ }
  }
}
</script>

ここでの問題は、コンポーネントが再レンダリングされるたびに、つまり、props値が変更されるたびに、Vueがheavy（）メソッドを実行することです。

重いメソッドを子コンポーネントに分離すると、このようなコンポーネントを簡単に最適化できます。

<template>
  <div :style="{ opacity: number / 300 }">
    <ChildComp/>
  </div>
</template>

<script>
export default {
  props: ['number'],
  components: {
    ChildComp: {
      methods: {
        heavy () { /*   */ }
      },
      render (h) {
        return h('div', this.heavy())
      }
    }
  }
}
</script>

何のために？Vueは、依存データが変更されていないコンポーネントのレンダリングを自動的にスキップします。したがって、親コンポーネントで小道具が変更された場合、子は再利用され、heavy（）メソッドは再起動されません。

これは、子コンポーネントが親コンポーネントでデータ依存関係を持たない場合にのみ意味があることに注意してください。それ以外の場合、子は親とともに再作成され、この最適化は意味がありません。

ゲッターローカルキャッシュ


次のコンポーネントには、2番目の計算されたプロパティに基づいて、ある種の計算されたプロパティがあります。

<template>
  <div :style="{ opacity: start / 300 }">{{ result }}</div>
</template>

<script>
export default {
  props: ['start'],
  computed: {
    base () { return 42 },
    result () {
      let result = this.start
      for (let i = 0; i < 1000; i++) {
        result += this.base
      }
      return result
    }
  }
}
</script>

ここで重要なことは、基本プロパティがループで呼び出されることであり、これにより複雑さが生じます。Vueはリアクティブデータにアクセスするたびに、依存関係などを構築するためにアクセスするデータと方法を決定するロジックを実行します。これらの小さなオーバーヘッドは、例のように、呼び出しが多い場合に合計されます。

これを修正するには、baseに一度アクセスして、値をローカル変数に保存します。

<template>
  <div :style="{ opacity: start / 300 }">{{ result }}</div>
</template>

<script>
export default {
  props: ['start'],
  computed: {
    base () { return 42 },
    result () {
      const base = this.base // <--
      let result = this.start
      for (let i = 0; i < 1000; i++) {
        result += base
      }
      return result
    }
  }
}
</script>

v-showでのDOMの再利用


次の例を見てください。

<template functional>
  <div>
    <div v-if="props.value">
      <Heavy :n="10000"/>
    </div>
    <section v-else>
      <Heavy :n="10000"/>
    </section>
  </div>
</template>

ここに、v-ifとv-elseを使用して、ある種の子コンポーネントを切り替えるラッパーコンポーネントがあります。

v-ifのしくみを理解することが重要です。状態が切り替わるたびに、1つの子コンポーネントが完全に破棄され（destroyed（）フックが呼び出され、すべてのノードがDOMから削除されます）、2番目のコンポーネントが完全に作成されて再度マウントされます。また、これらのコンポーネントが「重い」場合、切り替え時にインターフェースがハングします。

より生産的な解決策はv-showを使用することです

<template functional>
  <div>
    <div v-show="props.value">
      <Heavy :n="10000"/>
    </div>
    <section v-show="!props.value">
      <Heavy :n="10000"/>
    </section>
  </div>
</template>

この場合、両方の子コンポーネントがすぐに作成およびマウントされ、同時に存在します。したがって、Vueは切り替え時にコンポーネントを破棄して作成する必要はありません。彼がすることは、CSSで1つを非表示にし、2つ目を表示することだけです。したがって、状態の切り替えははるかに高速になりますが、これによりメモリコストが高くなることを理解する必要があります。

<keep-alive>を使用する


したがって、単純なコンポーネントは、ルーターのラッパーコンポーネントです。

<template>
  <div id="app">
    <router-view/>
  </div>
</template>

問題は前の例と同様です。ルーター間のすべてのコンポーネントは、ルート間の遷移中に作成、マウント、破棄されます。

そして、ここでのソリューションは似ています-Vueにコンポーネントを破棄せずにキャッシュして再利用するように指示することです。これは、特別な組み込み<keep-alive>コンポーネントを使用して行うことができます。

<template>
  <div id="app">
    <keep-alive>
      <router-view/>
    </keep-alive>
  </div>
</template>

Vueはより多くの「生きている」コンポーネントをサポートするため、この最適化はメモリ消費の増加につながります。したがって、特に多くのルートがある場合は、すべてのルートでこのアプローチを無計画に使用しないでください。include属性とexclude属性を使用して、キャッシュする必要があるルートとキャッシュしないルートを設定できます。

<template>
  <div id="app">
    <keep-alive include="home,some-popular-page">
      <router-view/>
    </keep-alive>
  </div>
</template>

遅延レンダリング


次の例は、いくつかの非常に重い子コンポーネントを持つコンポーネントです。

<template>
  <div>
    <h3>I'm an heavy page</h3>

    <Heavy v-for="n in 10" :key="n"/>

    <Heavy class="super-heavy" :n="9999999"/>
  </div>
</template>

問題は、コンポーネント処理がメインスレッドでの同期操作であることです。また、この例では、親と子の両方のすべてのコンポーネントの処理が完了するまで、ブラウザには何も表示されません。また、子会社の処理に時間がかかる場合は、しばらくの間、インターフェイスまたは空白の画面で遅延が発生します。

子コンポーネントのレンダリングを延期することで、状況を改善できます。

<template>
  <div>
    <h3>I'm an heavy page</h3>

    <template v-if="defer(2)">
      <Heavy v-for="n in 10" :key="n"/>
    </template>

    <Heavy v-if="defer(3)" class="super-heavy" :n="9999999"/>
  </div>
</template>

<script>
import Defer from '@/mixins/Defer'

export default {
  mixins: [
    Defer()
  ]
}
</script>

ここで、defer（n）関数はfalse nフレームを返します。その後は常にtrueを返します。これは、テンプレートの一部の処理を数フレーム延期するために使用され、それによってブラウザにインターフェースを描画する機能を提供します。

使い方。上で書いたように、v-ifディレクティブの条件がfalseの場合、Vueはテンプレートの一部を完全に無視します。

アニメーションの最初のフレームでは、次のようになります。

<template>
  <div>
    <h3>I'm an heavy page</h3>

    <template v-if="false">
      <Heavy v-for="n in 10" :key="n"/>
    </template>

    <Heavy v-if="false" class="super-heavy" :n="9999999"/>
  </div>
</template>

Vueは単にヘッダーをマウントし、ブラウザーはそれを表示して2番目のフレームを要求します。この時点で、defer（2）はtrueを返し始めます

<template>
  <div>
    <h3>I'm an heavy page</h3>

    <template v-if="true">
      <Heavy v-for="n in 10" :key="n"/>
    </template>

    <Heavy v-if="false" class="super-heavy" :n="9999999"/>
  </div>
</template>

Vueは10個の子コンポーネントを作成してマウントします。ブラウザはそれらを表示し、defer（3）がtrueを返す次のフレームを要求します。

したがって、徐々に処理される重いコンポーネントを作成し、ブラウザーがテンプレートの既にマウントされた部分を表示できるようにしました。これにより、インターフェイスがハングしたように見えないため、UXが大幅に向上します。

ミキサー延期コード：

export default function (count = 10) {
  return {
    data () {
      return {
        displayPriority: 0
      }
    },

    mounted () {
      this.runDisplayPriority()
    },

    methods: {
      runDisplayPriority () {
        const step = () => {
          requestAnimationFrame(() => {
            this.displayPriority++
            if (this.displayPriority < count) {
              step()
            }
          })
        }
        step()
      },

      defer (priority) {
        return this.displayPriority >= priority
      }
    }
  }
}

遅延コンポーネント読み込み


次に、子コンポーネントのインポートについて説明します。

import Heavy from 'Heavy.js'

export default {
  components: { Heavy }
}

従来のインポートでは、ブラウザがインポート文に到達すると、子コンポーネントがすぐに読み込まれます。または、コレクターを使用する場合、子コンポーネントは一般バンドルに含まれます。

ただし、子コンポーネントが非常に大きい場合は、非同期でロードすることをお勧めします。
とても簡単です：

const Heavy = () => import('Heavy.js')

export default {
  components: { Heavy }
}

それだけで十分です。Vueは、コンポーネントの遅延読み込みをそのまま使用できます。彼は必要なときに自分でコンポーネントをダウンロードし、準備ができ次第表示します。このアプローチは、どこでも遅延読み込みに使用できます。

コレクターを使用する場合、Heavy.jsに関するすべてのものが別のファイルに取り出されます。したがって、最初のダウンロード中にファイルの重量を減らし、表示速度を上げます。

遅延読み込みビュー


遅延読み込みは、ルートに使用されるコンポーネントの場合に非常に役立ちます。ルートのすべてのコンポーネントを同時にダウンロードする必要がないため、常にこのアプローチを使用することをお勧めします。

const Home = () => import('Home.js')
const About = () => import('About.js')
const Contacts = () => import('Contacts.js')

new VueRouter({
  routes: [
    { path: '/', component: Home },
    { path: '/about', component: About },
    { path: '/contacts', component: Contacts },
  ]
})

これらの各コンポーネントは、ユーザーが最初に指定されたルートをリクエストしたときにのみ読み込まれます。そして前に。

動的コンポーネント


同様に、動的コンポーネントで遅延読み込みを簡単に使用できます。

<template>
  <div>
    <component :is="componentToShow"/>
  </div>
</template>

<script>
export default {
  props: ['value'],
  computed: {
    componentToShow() {
      if (this.value) {
        return () => import('TrueComponent.js')
      }

      return () => import('FalseComponent.js')
    }
  }
}
</script>

この場合も、各コンポーネントは必要なときにのみロードされます。

Vuexを使用する


大きなコミットを避ける


ストレージに大量のデータを格納する必要があると想像してください。

fetchItems ({ commit }, { items }) {
  commit('clearItems')
  commit('addItems', items) //   10000 
}

問題はこれです。すべてのコミットは同期操作です。つまり、大きな配列を処理すると、作業中はインターフェイスがブロックされます。

この問題を解決するには、配列を分割して一度に1つずつ追加し、ブラウザーに新しいフレームを描画する時間を与えます。

fetchItems ({ commit }, { items, splitCount }) {
  commit('clearItems')
  const queue = new JobQueue()
  splitArray(items, splitCount).forEach(
    chunk => queue.addJob(done => {
      //      
      requestAnimationFrame(() => {
        commit('addItems', chunk)
        done()
      })
    })
  )
  //     
  await queue.start()
}

ストレージにデータを追加する当事者が少なければ少ないほど、インターフェースはよりスムーズになり、タスクを完了するための合計時間が長くなります。

さらに、このアプローチでは、ユーザーにダウンロードインジケーターを表示することができます。これにより、アプリケーションでの経験も大幅に向上します。

不要な場合は反応をオフにします


そして今日の最後の例です。同様のタスクがあります。非常に大きなオブジェクトの配列を、ネストレベルの束とともにストレージに追加します。

const data = items.map(
  item => ({
    id: uid++,
    data: item, // <--   
    vote: 0
  })
)

事実は、Vueがすべてのネストされたフィールドを再帰的にトラバースし、それらをリアクティブにすることです。大量のデータがある場合、これはコストがかかる可能性がありますが、これははるかに重要です-不要です。

アプリケーションが、トップレベルのオブジェクトのみに依存し、数レベル下のどこかでリアクティブデータを参照しないように構築されている場合は、リアクティブを無効にして、Vueを不必要な一連の作業から節約できます。

const data = items.map(
  item => optimizeItem(item)
)

function optimizeItem (item) {
  const itemData = {
    id: uid++,
    vote: 0
  }
  Object.defineProperty(itemData, 'data', {
    //    "-"
    configurable: false,
    value: item
  })
  return itemData
}
