パブリックIPがない場合のスマートホームへの安全なアクセス（パート1）

前書き


意味を反映した大まかなタイトルを選ぶのは難しいので、自分で設定した課題をすぐに説明します。

「スマートホーム」があります。私の場合、これはioBrokerを備えたファンレスホームサーバーですが、これは重要ではありません。家のものに加えて、外部からセンサーをフックしたいと思います（たとえば、リモート温室からのESP32）。私はmqttを通じてこれを行うことにしました。インターネットからインターフェースへのアクセス。

いつものこと。しかし、ニュアンスがあります：

  • プロバイダーからパブリックIPアドレスを取得する方法がありません。そして、他のプロバイダーはありません。
  • 特定のクラウドサービスにバインドするのは好きではありません。外部サービスも終了する場合があります（gbridgeが最近通知を送信したため）。そして、障害が発生した場合に備えて、何をすべきかは明確ではありません。私は自分のものを好んでいます。それは転送でき、何かが起こった場合、ほとんど血を使わずにやり直します。
  • 安全は重要です。パラノイアではなく、インターネット上にioBrokerを置くこと。特に、いくつかのサービスが表示されていることを考えると（flot ...）。いや、本当に。

さらに、すぐに結果ではなく、プロセスを示したいと思います。それがどのように進んだか、ウィッシュリストがどのように変容したか、決定が変わった。一部のポイントをより正確に/効率的に解決できる可能性があります（私はシステム管理者ではなく、開発者ではありません）。あるいは、誰かがここまで進んで、暫定的な解決策を利用しないかもしれません。たとえば、私は自分にとって安全または便利ではないと感じました。実際、この部分で説明することはかなり有効なオプションですが、私にとっては「中級」です。

mqttのパブリックアドレス


自宅では、ルーターのパブリックIPは光りません（私は修正について話しているわけではありません。これはdyndnsとアナログで解決できます）。つまり、プロバイダーはオプションなしで10.x.x.xを提供します。だから、あなたは小さなVPSを借りて、それを通してプロブロを作る必要があります。

最も簡単な方法は、sshをトンネリングすることです。ホームサーバーで（iob.xxx.xxと呼びます）、次のコマンドを実行します。

ssh -N -T -R pub.xxx.xx:1883:127.0.0.1:1883 a@iob.xxx.xx

外部サーバーpub.xxx.xxのポート1883に接続すると、実際にはmiott（mosquito）コンテナーが実行されているホームiob：1883にいることがわかります。

当然、これは自動的に開始する必要があり、接続は障害後に復元されます。したがって、私はautosshを使用して、サービスとして設計しました。

/etc/systemd/system/ssh_mqtt.service：

[Unit]
Description=SSH Tunnel mqtt
After=network.target

[Service]
Environment="AUTOSSH_GATETIME=0"
ExecStart=autossh -M 0 -o "ServerAliveInterval 30" -o "ServerAliveCountMax 3" -NR pub.xxx.xx:1883:127.0.0.1:1883 -i /home/a/.ssh/id_rsa a@pub.xxx.xx

[Install]
WantedBy=multi-user.target

あらゆる種類のsystemctlの有効化/再起動など説明しません。

残念ながら、autosshにもかかわらず、私は絶え間ないフリーズに悩まされていました。そのため、エンティティを作成する必要がないと判断し、通常のsshで解決しました：

/etc/systemd/system/ssh_mqtt.service：

[Unit]
Description=SSH Tunnel mqtt
After=network.target

[Service]
Restart=always
RestartSec=20
User=anri
ExecStart=/bin/ssh -N -T -R pub.xxx.xx:1883:127.0.0.1:1883 -i /home/a/.ssh/id_rsa a@pub.xxx.xx

[Install]
WantedBy=multi-user.target

その後、ちなみに、このプロバイダーは非常に斜めであることがわかりました。まあ、または45ルーブル/月の最も安いレートなので、曲がって彼のために働きます。 ssh（MobaXterm）経由で接続されているだけでも、セッションは定期的にハングします。結局、私は自分に別のVPS（55ルーブル/月）を注文したところ、フリーズの問題は解消されました。

ちなみに、価格だけでなく、ping（10〜20ミリ秒）を考慮して、どこでVPSを使用するかを自分で選択しました。

一般に、このオプションは非常に正常です。特に、TLS経由でポート8883にのみインターネット接続を行ったことを考えると、特にそうです。それら。蚊のパスワードは暗号化されて送信されました。

その後、クライアント証明書を必須にしました。それら。まず、TLSレベルでクライアント証明書を提示し、次に蚊で指定されたパスワード名でログインする必要があります。それら。パスワード検索フェーズに到達するのはそれほど簡単ではありません。
そこで、最初はLetsEncryptのサーバー証明書を使用していましたが、クライアント証明書が必要なため、自己署名に切り替えました。

その過程でESP32のソフトウェアにも取り組んだので、VPN接続に問題があるとバッテリーの消費がはるかに速くなることに気づきました（または、もう覚えていません）。通常の動作中のサイクル：ウェイクアップ、センサーに電源を投入、WiFiに接続、mqttサーバーとの接続を確立、センサーの読み取り値を読み取り、mqttに転送、センサーの電源をオフにして、10分間ディープスリープ状態にします。

通常、このようなサイクルには約4秒かかります。1.5〜2秒-WiFiへの接続。TLS経由のmqttへの移行により、さらに1秒かかります。それは4秒間あなたに合います、それと同じですが、センサーは目を覚ますのに時間が必要です。しかし、VPNがダウンした場合（autosshが落ちたときに明確に表示されていた場合）、どうすればよいですか？もちろん、20秒後にシステムがスリープ状態になるように設定しました。しかし、4秒ではなく20秒が非常に目立ちます。

一般に、mqttサーバーを外部VPSに維持することをお勧めします。すべてが時計のように機能するようになったので、これが必要かどうかはわかりません。でも、作り直しても意味がありません。

Visのパブリックアドレス


VisはioBrokerで人気のある視覚化システムです。わざわざ、httpsで自分で構成して、同様にポートを転送することはできません。さらに、アプリケーションレベルでパスワードを要求できます。

しかし、それはクールではありません。特に仕事のためにそれが追加のサービスを接続することを考慮します。たとえば、私はフロットでグラフを描画します。比較的言えば、vis.xxx.xx：8082 / vis / index.html に接続しますが、内部にはvis.xxx.xxxグラフへのリンク：8082 / flot / index.htmlがあります。ある時点で、/ visに接続するときにパスワードが要求され、グラフインターフェイスにパスワードなしでアクセスできることが判明しました。

ある時点でそれは一般的に奇妙でした-私はvisで承認されました、私はグラフを見ますが、右下に半透明の「サーバーへの接続なし」ウィンドウがあります。このcssブロックを非表示にするために書き直しました。しかし、同じ画面でグラフを切り替えるためにフレームを使い始めた直後に、フレーム内のディスプレイでの私のオーバーラップが機能しないことがわかりました。 （後で判明したように、そうなるはずです）。私は認証をオフにします-すべてがうまくいきます。

そこで、同じ外部サーバーでnginxをリバースプロキシモードで発生させることにしました。そして、すでにそれを承認します。

ブラウザからそれは働いた。しかし、Play Marketのネイティブアプリケーションiobroker.visは、この方法でログインできませんでした。そしてそれを使いたかった。これは実際にはウィンドウ内のブラウザですが、いくつかの優れた機能があります。スケールを設定して（垂直モードで93％）、画像が収まるとしましょう。画面解像度が異なる別のデバイスでは、係数を選択するだけです。そして、ブラウザでは毎回調整する必要があります...

わかりました。パスワードではなく、URLにトリッキーなコードを追加します。 「vis.xxx.xx：8082 / <長いシーケンス> /vis/index.html」と入力します。多くの場合、このようなトリックが使用されます。

ほとんど稼いだ。しかし、グリッチにより、このWebアプリケーションが正しく作成されていないことが掘り起こされました。その内部のリンクの多くは相対ではなく、ルートからのものです。

さて、私はいくつかのリンクを見つけました、私はそれらのために書きました、彼らはリファラーがそのようなコードを含んでいるなら、それでも信頼し、URLを書き直すなどします。しかし、彼らは徐々に明らかになりました。だから私はこれが曲がっていて間違っており、別のアプローチが必要だと判断しました。

VPN


アクセスの普遍性を少し犠牲にすることにしました。私のラップトップ、スマートフォンからアクセスさせてください。しかし、他の人のデバイスから、インターネットカフェからは必要ありません、私が管理します。次に、VPNをインストールする小さなクライアントを配置できます。そして、その内部では、SSLも認証も必要ありません。同時に、リンクをやり直す必要はありません。

私にとって最も簡単な方法はZerotierでした。私のOS（Windows、Android、Linux）にはクライアントがあります。そして、Dockerには既製のものさえあります。はい。これの機能については、後ですべてをdockerで実行します。

クライアントをインストールし、ネットワークに固有のコードを入力してから、my.zerotier.comのWebインターフェイスで確認し、必要に応じて、プライベートプライベートネットワーク（la 10.20.30.0）から静的アドレスを設定します。これで完了です。接続されているすべてのクライアントはお互いを認識します。

私が少し対処しなければならなかった唯一のことは、「クライアントを起動せずに自宅のWiFiのデバイスからリモートサーバーに接続する方法」でした。まあ、私のホームサーバーは、それ自体をルーティングする場合でも、既にクライアントです。すべてが簡単であることがわかりました。ホームネットワーク192.168.x.0は、管理されたルートセクションのmy.zerotier.comに登録する必要があります。もちろん、これは私のゲートウェイとして指定します。これが私のホームサーバーです。さて、WiFiネットワークで、ルートを適切に構成します（WiFiルーターで、ホームサーバーの静的10.20.30.0）。

Zerotierクライアントを接続するときに、別のDNSサーバーを指定できます。それら。クライアントを接続しました。ドメイン名はパブリックアドレスではなくプライベートアドレスに解決されます。これは、DNSがホームサーバーを指しているため、dnsmasqがZerotierプライベートネットワークからの個々のIPレコードを取得するためです。

ゼロティアでさえ、接続のためのルートの効果的な選択に満足しています。自宅のWiFiでZerotierクライアントをアクティブ化した場合、自宅のコンピューター（Zerotierによって発行されたIPアドレス）へのpingは、クライアントなし（WiFi経由のみ）と同じ数ミリ秒です。それら。クラウドへの接続は最初の一瞬に過ぎません。その後のトラフィック交換は、クラウド経由ではなく直接実行されます。たとえばOpenVPNをVPSにインストールすると、同じトラフィックがクライアントからVPSに流れ、同じWiFiネットワークに戻ってホームサーバーに戻ります。

原則として、あなたのムーンサーバーを置くためのチップさえあります。ほぼインターネットから切り離されたネットワークでは、このすべての経済が展開されます。

結果はどうですか？


ESP32は、VPSにデプロイされたmqttサーバーにデータを送信します。 TLSでは、クライアント証明書が必要です。

Zerotierを介したVPNがホームサーバーと共にインストールされます。 Sonoff rfBridgeは、mqttからVPSへのこのホームサーバーを介してTasmotaファームウェアと通信します。クライアント証明書を使用してTLSを設定する方法はないため、通常のMQTTは1883に構成されます。とにかく、ホームサーバーは、Zerotierを使用してこのトラフィックを暗号化します。

さて、私はホームネットワークから直接visに接続し、Zerotierクライアントをアクティブにしてインターネットから接続します。あなたはそれを全くオフにすることができません、これはまた働きます。しかし、他のVPNクライアントが必要になることもときどきあります（たとえば、「ILV禁止」に移動します）。 1台のスマートフォン上の2つのVPNがすぐに友達になったわけではありませんが、理解できませんでした。

すべてが非常に簡単です。しかし、ワームは魂を飲み込みました。私は原子炉を持っていませんが、突然ですか？彼らがTeamViewer（会社、具体的にはクライアントソフトウェアではない）を破り、それらを通じて多くのアカウントへのアクセス権を獲得したという事例がありました。そして、一般的に、私は最初に自分のすべてが大好きだと書いています。
そこで次のステップとして、ZerotierからOpenVPNに切り替えました。すべてが私の手の中にあります。

唯一の「エイリアン」はプロバイダーのVPSです。さて、私は特別にすべてをDockerコンテナで起動して、即座に移動できるようにしています。
OpenVPNをどの程度処理しなければならないかを知っていれば、たぶん私はそうではないでしょう。公平に言えば、主な問題はまさにコンテナによるものでした。
