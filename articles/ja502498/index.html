みなさん、こんにちは。コース開始前日に、「C＃Developer」という便利な資料の翻訳を用意しました。読書を楽しむ。



私は最近C＃でCriteoのベストプラクティスのリストを作成したので、それを公開して共有するのは良いことだと思いました。この記事の目的は、疑わしいため、または単にうまく機能しないために、避けるべきコードテンプレートの不完全なリストを提供することです。リストは多少前後しているため、少しランダムに見えるかもしれませんが、すべての要素がコードのどこかで見つかり、本番の問題を引き起こしました。これが良い防止策となり、間違いを防ぐことを願っています。

また、Criteo Webサービスは高性能コードに依存しているため、非効率的なコードを回避する必要があることにも注意してください。ほとんどのアプリケーションでは、これらのテンプレートの一部を置き換えることによる明確な違いはありません。

そして最後に、重要な点として、いくつかの点（たとえば、ConfigureAwait）はすでに多くの記事で説明されているため、それらについては詳しく説明しません。目標は、注意を払う必要があるポイントのコンパクトなリストを形成することであり、ポイントごとに詳細な技術計算を行うことはありません。

非同期コードを同期的に待機


同期して終了していないタスクを期待しないでください。これは、以下に適用されますが、これらに限定されませんTask.Wait, Task.Result, Task.GetAwaiter().GetResult(), Task.WaitAny, Task.WaitAll。

一般化として：2つのプールスレッド間の同期関係は、プールの枯渇を引き起こす可能性があります。この現象の原因については、この記事で説明します。

ConfigureAwait


コードを同期コンテキストから呼び出すことができる場合はConfigureAwait(false)、それぞれのawait呼び出しに使用します。

これはConfigureAwait キーワードを使用する場合にのみ役立つawaitことに注意してください。

たとえば、次のコードは無意味です。

//  ConfigureAwait        
var result = ProcessAsync().ConfigureAwait(false).GetAwaiter().GetResult();

決して使用しないでくださいasync void。async voidメソッドでスローされた例外は同期コンテキストに伝播し、通常はアプリケーション全体がクラッシュします。

メソッドでタスクを返すことができない場合（たとえば、インターフェイスを実装しているため）、非同期コードを別のメソッドに移動して呼び出します。

可能な限り非同期を避ける


習慣がなかったり、筋肉の記憶が原因で、次のように書くことができます。

public async Task CallAsync()
{
    var client = new Client();
    return await client.GetAsync();
}

コードは意味的には正しいasyncですが、ここではキーワードを使用する必要はなく、負荷の高い環境ではかなりのオーバーヘッドが発生する可能性があります。可能な限り回避するようにしてください：

public Task CallAsync()
{
    var client = new Client();
    return _client.GetAsync();
}

ただし、コードがブロックでラップされている場合（ try/catchまたはなどusing）は、この最適化に頼ることができないことに注意してください。

public async Task Correct()
{
    using (var client = new Client())
    {
        return await client.GetAsync();
    }
}

public Task Incorrect()
{
    using (var client = new Client())
    {
        return client.GetAsync();
    }
}

間違ったバージョン（Incorrect()）ではGetAsync、usingブロック内のタスクがawaitによって予期されていないため、呼び出しが完了する前にクライアントが削除される可能性があります。

地域比較


地域比較を使用する理由がない場合は、常に序数比較を使用してください。内部の最適化により、これはen-USデータプレゼンテーションフォームではそれほど重要ではありませんが、他のリージョンのプレゼンテーションフォームでは比較が1桁遅くなります（Linuxでは最大2桁！）。ほとんどのアプリケーションでは文字列の比較が頻繁に行われるため、オーバーヘッドが大幅に増加します。

ConcurrentBag <T>


ConcurrentBag<T>ベンチマークなしで使用しないでください。このコレクションは非常に特定のユースケース（ほとんどの場合、アイテムをキューに入れたスレッドによってアイテムがキューから除外される場合）用に設計されており、他の目的で使用すると深刻なパフォーマンスの問題が発生します。スレッドセーフなコレクションが必要な場合は、ConcurrentQueue <Tを使用してください>。

ReaderWriterLock / ReaderWriterLockSlim <T ベンチマークなしで>
使用しないでください。ReaderWriterLock<T>/ReaderWriterLockSlim<T>リーダーとライターを操作するときにこの種の特殊な同期プリミティブを使用するのは魅力的な場合がありますが、そのコストは単純なMonitor（キーワードで使用するlock）よりはるかに高くなります。クリティカルセクションを同時に実行するリーダーの数がそれほど多くない場合、同時実行性は増加したオーバーヘッドを吸収するのに十分ではなく、コードはさらに悪くなります。

メソッドグループではなくラムダ関数を優先する


次のコードを検討してください。

public IEnumerable<int> GetItems()
{
    return _list.Where(i => Filter(i));
}
private static bool Filter(int element)
{
    return i % 2 == 0;
}

Resharperはラムダ関数なしでコードを書き直すことを提案しています。

public IEnumerable<int> GetItems()
{
    return _list.Where(Filter);
}
private static bool Filter(int element)
{
    return i % 2 == 0;
}

残念ながら、これにより各呼び出しに動的メモリが割り当てられます。実際、呼び出しは次のようにコンパイルされます。

public IEnumerable<int> GetItems()
{
    return _list.Where(new Predicate<int>(Filter));
}
private static bool Filter(int element)
{
    return i % 2 == 0;
}

負荷の高いセクションでコードが呼び出された場合、これはパフォーマンスに大きな影響を与える可能性があります。

ラムダ関数を使用するとコンパイラーの最適化が開始され、静的フィールドにデリゲートがキャッシュされ、割り当てが回避されます。これはFilter静的な場合にのみ機能します。そうでない場合は、デリゲートを自分でキャッシュできます。

private Predicate<int> _filter;

public Constructor()
{
    _filter = new Predicate<int>(Filter);
}

public IEnumerable<int> GetItems()
{
    return _list.Where(_filter);
}

private bool Filter(int element)
{
    return i % 2 == 0;
}

列挙型を文字列に変換する


呼び出しEnum.ToStringに.net反射が内側に変換するために使用されているので、非常に高価であり、パッケージ構造誘発する上で仮想メソッドを呼び出します。これは可能な限り回避する必要があります。

多くの場合、列挙は定数文字列で置き換えることができます。

//       Numbers.One, Numbers.Two, ...
public enum Numbers
{
    One,
    Two,
    Three
}

public static class Numbers
{
    public const string One = "One";
    public const string Two = "Two";
    public const string Three = "Three";
}

列挙型を本当に使用する必要がある場合は、変換された値をディクショナリにキャッシュしてオーバーヘッドを償却することを検討してください。

列挙の比較


注：これは.netコアでは関係ありません。バージョン2.1以降、最適化はJITによって自動的に実行されます。

列挙型をフラグとして使用する場合、メソッドを使用したくなるかもしれませんEnum.HasFlag。

[Flags]
public enum Options
{
    Option1 = 1,
    Option2 = 2,
    Option3 = 4
}

private Options _option;

public bool IsOption2Enabled()
{
    return _option.HasFlag(Options.Option2);
}

このコードは、割り当てを持つ2つのパッケージを誘発：変換のための1 Options.Option2へEnumの仮想呼び出しのための、および他のHasFlag構造のために。これにより、このコードは過度に高価になります。代わりに、読みやすさを犠牲にして、二項演算子を使用する必要があります。

public bool IsOption2Enabled()
{
    return (_option & Options.Option2) == Options.Option2;
}

インターフェースを備えた構造を使用する場合、不適切なパッケージ化を回避


次のコードを検討してください。

public class IntValue : IValue
{
}

public void DoStuff()
{
    var value = new IntValue();

    LogValue(value);
    SendValue(value);
}

public void SendValue(IValue value)
{
    // ...
}

public void LogValue(IValue value)
{
    // ...
}

作るIntValue構造は、動的メモリの割り当てを回避するために誘惑することができます。しかし、AddValueとSendValueは期待されるインターフェースであり、インターフェースは参照セマンティクスを持っているため、値はすべての呼び出しでパックされ、この「最適化」の利点が失われます。実際、IntValue値は呼び出しごとに独立してパッケージ化されるため、クラスの場合よりもさらに多くのメモリが割り当てられます。

APIを作成していて、一部の値が構造体であると予想される場合は、ジェネリックメソッドを使用してみてください。

public struct IntValue : IValue
{
}

public void DoStuff()
{
    var value = new IntValue();

    LogValue(value);
    SendValue(value);
}

public void SendValue<T>(T value) where T : IValue
{
    // ...
}

public void LogValue<T>(T value) where T : IValue
{
    // ...
}

これらのメソッドからユニバーサルへの変換は一見役に立たないように見えますが、実際にIntValueは、それが構造体である場合に、割り当てによるパッケージ化を回避することができます。

CancellationTokenサブスクリプションは常にインライン


をキャンセルするとCancellationTokenSource、すべてのサブスクリプションが現在のスレッド内で実行されます。これにより、計画外の一時停止や暗黙的なデッドロックが発生する可能性があります。

var cts = new CancellationTokenSource();
cts.Token.Register(() => Thread.Sleep(5000));
cts.Cancel(); //     5

この動作は回避できません。したがって、をキャンセルするときCancellationTokenSourceは、現在のスレッドのキャプチャを安全に許可できるかどうかを自問してください。答えが「いいえ」の場合は、呼び出しをCancel内部Task.Runにラップして、スレッドプールで実行します。

TaskCompletionSourceの継続はしばしばインライン


サブスクリプションと同様にCancellationToken、継続TaskCompletionSourceはしばしばインラインです。これは適切な最適化ですが、暗黙のエラーが発生する可能性があります。たとえば、次のプログラムを考えてみます。

class Program
{
    private static ManualResetEventSlim _mutex = new ManualResetEventSlim();
    
    public static async Task Deadlock()
    {
        await ProcessAsync();
        _mutex.Wait();
    }
    
    private static Task ProcessAsync()
    {
        var tcs = new TaskCompletionSource<bool>();
        
        Task.Run(() =>
        {
            Thread.Sleep(2000); //  - 
            tcs.SetResult(true);
            _mutex.Set();
        });
        
        return tcs.Task;
    }
    
    static void Main(string[] args)
    {
        Deadlock().Wait();
        Console.WriteLine("Will never get there");
    }
}

呼び出しtcs.SetResultにより、継続await ProcessAsync()が現在のスレッドで実行されます。したがって、ステートメント_mutex.Wait()は、呼び出す必要のある同じスレッドによって実行され、_mutex.Set()デッドロックが発生します。これは、パラメーターTaskCreationsOptions.RunContinuationsAsynchronouslyc を渡すことで回避できますTaskCompletionSource。

無視する正当な理由がない場合はTaskCreationsOptions.RunContinuationsAsynchronously、を作成するときに常にこのオプションを使用してくださいTaskCompletionSource。

注意：TaskContinuationOptions.RunContinuationsAsynchronously代わりTaskCreationOptions.RunContinuationsAsynchronouslyにを使用するとコードもコンパイルされますが、パラメーターは無視され、継続は引き続きインラインになります。これはオートコンプリートにTaskContinuationOptions先行するため、驚くほどよくある間違いTaskCreationOptionsです。

Task.Run / Task.Factory.StartNew


を使用する理由がない場合はTask.Factory.StartNew、常にTask.Runバックグラウンドタスクの実行を選択してください。Task.Runより安全なデフォルト値を使用し、さらに重要なことに、返されたタスクを自動的にアンパックします。これにより、非同期メソッドでの明らかでないエラーを防ぐことができます。次のプログラムを検討してください。

class Program
{
    public static async Task ProcessAsync()
    {
        await Task.Delay(2000);
        Console.WriteLine("Processing done");
    }
    
    static async Task Main(string[] args)
    {
        await Task.Factory.StartNew(ProcessAsync);
        Console.WriteLine("End of program");
        Console.ReadLine();
    }
}

その外観にもかかわらず、プログラムの終了は処理が完了する前に表示されます。これはTask.Factory.StartNewが返されるためでTask<Task>、コードは外部タスクのみを想定しています。正しいコードはawait Task.Factory.StartNew(ProcessAsync).Unwrap()、またはのいずれかawait Task.Run(ProcessAsync)です。

  • 専用スレッドでタスクを実行する（を使用TaskCreationOptions.LongRunning）。
