本「{JSのことはわからない}のタイプと文法構文」

JavaScriptプログラミングの経験が何であれ、おそらく言語を完全に理解していません。この簡潔なガイドでは、既存のすべての書籍よりもタイプについて詳しく説明します。タイプのしくみ、変換の問題、新しい機能の使用方法について学びます。「You Do n't Know JS」シリーズの

他の本と同様に、JavaScriptプログラマーが近づかない（または存在しないと想定する）ことを好む言語の重要な側面を示しています。この知識を武器に、真のJavaScriptを習得します。

抜粋。平等は厳格と非厳格です。


非厳密な等価性は==演算子によってチェックされ、厳密な-===演算子によってチェックされます。両方の演算子は、「等しい」という2つの値を比較するために使用されますが、形式（厳格/非厳格）を選択すると、特に等号の決定がどのように行われるかについて、動作に非常に重要な違いが生じます。

これらの2つの演算子に関してよくある誤解があります。「==値の等価性をチェックし、===は値と型の両方の等価性をチェックします。」妥当に
聞こえますが、不正確です。数え切れないほどの評判の良いJavaScriptの本やブログは、まさにそれを言っていますが、残念ながらそれらはすべて間違っています。

正しい説明は次のとおりです。「==等価性をチェックするときに型変換を許可し、===型変換を禁止します。」

同等性検証パフォーマンス


最初の（不正確な）説明が2番目の（正確な）説明とどのように異なるかを考えて停止します。
最初の説明では、===演算子は型もチェックする必要があるため、==よりも多くの機能を果たすことは明らかです。

2番目の説明では、==演算子はより多くの処理を実行します。これは、異なる型では型変換を行わなければならないためです。

多くの人が陥る罠に陥らないでください。これがプログラムの速度に何らかの影響を与えるとは考えないでください。==は大幅に遅くなります===。変換にはしばらく時間がかかりますが、マイクロ秒の時間がかかります（つまり、100万分の1秒）。

同じタイプの2つの値を比較する場合、==と===は同じアルゴリズムを使用するため、エンジンの実装における小さな違いを考慮しない場合、それら
は同じ仕事をする必要があります。

異なるタイプの2つの値を比較する場合、パフォーマンスは重要な要素ではありません。別のことを自問する必要があります。2つの値を比較している場合、型変換を実行するかどうかを指定します。

変換が必要な場合は、厳密でない等価==を使用し、変換が望ましくない場合は、厳密な等価===を使用します。

==と===の両方の演算子は、オペランドのタイプをチェックします。違いは、型の不一致に対する応答の仕方です。

抽象等価チェック


==演算子の動作は、ES5仕様のセクション11.9.3（「抽象等式チェッカーアルゴリズム」）で定義されています。以下は、詳細かつ単純なアルゴリズムです。各組み合わせに適用する必要のある型と型変換メソッド（必要な場合）のすべての可能な組み合わせを明示的にリストしています。

- () , « ». , , JS- , .

, — , , , , ( ), . « » . 11.9.3 ES5. , , .

実際、最初のセクション（11.9.3.1）は、2つの比較された値が同じ型である場合、それらは単純で自然な方法で比較されると述べています。たとえば、42は42だけで、文字列「abc」は「abc」だけです。

注意すべきいくつかのマイナーな例外：

  • NaNの値がそれ自体と等しくなることはありません（第2章を参照）。
  • +0と-0は互いに等しい（第2章を参照）。

セクション11.9.3.1の最後のセクションは、オブジェクト（関数と配列を含む）との==等価性の厳密なテストに当てられています。そのような2つの値は、両方がまったく同じ値を参照している場合にのみ等しくなります。型変換は行われません。

厳密な等価チェック===は、2つのオブジェクト値の規定を含め、11.9.3.1と同じように定義されています。この事実はほとんど知られていませんが、2つのオブジェクトを比較する場合、==と===はまったく同じように動作します。

11.9.3の残りのアルゴリズムは、非厳密等価==を使用して2つの異なるタイプの値を比較できることを示しています。一方または両方の値には
暗黙的な変換が必要です。変換の結果、指定は1つのタイプに削減され
ます。その後、値の単純な同一性によって、等しいかどうかを直接比較できます。

不平等の厳密でない検証の操作！=期待どおりに正確に決定されます。実際、演算==は完全に実装され、その後
に結果の否定の計算が続きます。不等式を厳密にチェックする操作も同様です！==。

比較：文字列と数値


==の変換を示すために、まずこの章の前半で行った文字列と数値の例を作成します。

var a = 42;
var b = "42";

a === b;     // false
a == b;       // true

予想どおり、変換が許可されておらず、値42と "42"が異なるため、チェックa === bは失敗します。

ただし、2番目の比較a == bでは、厳密でない等価性が使用されます。つまり、タイプが異なる場合、比較アルゴリズムは一方
または両方の値の暗黙的な変換を実行します。

しかし、ここではどのような変換が行われていますか？aの値、つまり42は文字列になりますか、それともb“ 42”の値は数値になりますか？セクション11.9.3.4–5のES5仕様は次のように述べています。

  1. タイプ（x）のタイプがタイプで、タイプ（y）のタイプがストリングの場合、比較の結果x == ToNumber（y）を返します。
  2. Type（x）のタイプがStringでType（y）のタイプがNumberの場合、ToNumber（x）== yの比較結果を返します。

仕様ではNumber型とString型の正式名を使用していますが、プリミティブ型の本では通常、記数と文字列が使用されています。仕様の数値記号のケースと組み込みの数値（）関数を混同しないでください。私たちの目的のために、型の名前の文字の大文字と小文字は関係ありません-それらは同じことを意味します。

仕様によると、値「42」は比較のために数値に変換されます。変換がどのように実行されるかについては、すでに説明済みであり、具体的には抽象操作ToNumberを説明するときに説明されています。この場合、
取得された42の2つの値が等しいことは明らかです。

比較：ブール値のあるもの


タイプ==の暗黙の変換で最も危険なトラップの1つは、値をtrueまたはfalseと直接比較しようとしたときに発生します。

例：

var a = "42";
var b = true;

a == b; // false

待って、ここで何が起こっているのですか？「42」が本当の意味であることはわかっています（この章の前半を参照）。厳密で
ない等号==演算子を使用してtrueと比較してもtrueにならないことはどうしてわかりますか？

その理由は単純であり、一見すると狡猾です。誤解しやすいですが、多くのJS開発者はそれを完全に理解しようと努力していません。

もう一度、仕様、セクション11.9.3.6–7を引用します。

  1. タイプ（x）のタイプがブールの場合、比較の結果を返しますToNumber（x）== y。
  2. タイプ（y）のタイプがブールの場合、比較の結果x == ToNumber（y）を返します。

ここに何があるか見てみましょう。最初の一歩：

var x = true;
var y = "42";

x == y; // false

タイプ（x）は実際にはブール型に属しているため、ToNumber（x）操作が実行され、trueが1に変換されます。条件1 ==“ 42”が計算されます。いずれにせよ型は異なるため、（実際には再帰的に）アルゴリズムが繰り返されます。前のケースと同様に、「42」は42に変換され、条件1 == 42は明らかにfalseです。

オペランドを入れ替えても、結果は同じままです。

var x = "42";
var y = false;

x == y; // false

今回は、Type（y）はブール型であるため、ToNumber（y）は0を返します。条件 "42" == 0は再帰的に42 == 0に変わりますが、これはもちろんfalseです。

つまり、値「42」は== trueでも== falseでもありません。一見すると、このステートメントはまったく考えられないようです。どのようにして意味が真でも偽でもないのですか？

しかし、これは問題です！あなたは間違った質問をしています。実際、それはあなたのせいではありませんが、あなたをだましているのは脳です。

値 "42"は確かにtrueですが、構成 "42" == trueは、脳が何を言っていても、ブール値/変換テストをまったく実行しません。 「42」はブール値に変換されません（true）。代わりに、trueが1に変換されてから、「42」が42に変換されます。

好きかどうかにかかわらず、ここではToBooleanはまったく使用されないため、==演算では「42」の真偽はまったく重要ではありません。 ==比較アルゴリズムがすべての異なるタイプの組み合わせでどのように動作するかを理解することが重要です。ブール値が片側にある場合、常に最初に数値に変換されます。

これがあなたにとって奇妙に思えるなら、あなたは一人ではありません。個人的には、決して、決して、決して、決して== trueまたは== falseを使用しないことをお勧めします。決して。

ただし、ここでは==についてのみ話していることを思い出してください。 === trueおよび=== falseの構文は型変換を許可しないため、非表示のToNumber変換から保護されます。

例：

var a = "42";

//  (  !):
if (a == true) {
    // ..
}

//   (  !):
if (a === true) {
    // ..
}

//   ( ):
if (a) {
   // ..
}

//  ( ):
if (!!a) {
    // ..
}

//   ( ):
if (Boolean( a )) {
    // ..
}

コードで== trueまたは== false（ブール値との緩やかな等価性）を回避する場合、この真実/偽造トラップについて心配する必要はありません。

比較：未定義のnull


暗黙的な変換の別の例は、null値と未定義値の間でlax ==等式を使用する場合に発生します。ここでも、ES5仕様の
セクション11.9.3.2–3 を引用します。

  1. xがnullを含み、yが未定義を含む場合、trueを返します。
  2. xが未定義を含み、yがnullを含む場合、trueを返します。

非厳密演算子==と比較した場合のnullとundefinedは互いに等しく（つまり、相互に変換されます）、言語全体で他の値はありません。

私たちにとって、これは、厳密でない等価性テスト演算子==を使用して相互の暗黙的な変換を許可する場合、nullとundefinedは比較目的で区別できないと見なすことができることを意味します。

var a = null;
var b;

a == b; // true
a == null; // true
b == null; // true

a == false; // false
b == false; // false
a == ""; // false
b == ""; // false
a == 0; // false
b == 0; // false

nullとundefinedの間の変換は安全で予測可能であり、そのようなチェックに対して他の値が偽陽性を与えることはありません。この変換を使用することをお勧めします。これにより、nullとundefinedがプログラムで異なり、単一の値として解釈されます。

例：

var a = doSomething();

if (a == null) {
 // ..
}

a == nullチェックは、doSomething（）がnullまたは未定義を返し、他の値（0、false、および ""を含む）で失敗した場合にのみ合格します。

このような型変換を禁止するこのチェックの明示的な形式は、（私の意見では）非常に醜く見え、少し効率が悪くなる可能性があります！

var a = doSomething();

if (a === undefined || a === null) {
 // ..
}

a == nullの形式は、暗黙的な変換によってコードが読みやすくなるが、確実かつ安全に実行される状況のもう1つの例だと思います。

比較：オブジェクトと非オブジェクト


オブジェクト/関数/配列が単純なスカラープリミティブ（文字列、数値、ブール値）と比較される場合、ES5仕様では次のように記述されています（セクション11.9.3.8–9）。

  1. Type（x）のタイプがStringまたはNumberで、Type（y）のタイプがObjectである場合、比較の結果x == ToPrimitive（y）を返します。
  2. Type（x）のタイプがObjectで、Type（y）のタイプがStringまたはNumberの場合、ToPrimitive（x）== yの比較結果を返します。

仕様のこれらのセクションでは、ブール値ではなく文字列と数値のみが言及されていることに気づいたかもしれません。上記のように、セクション11.9.3.6–7では、ブールオペランドが最初にNumberとして表されることを確認しています。

例：

var a = 42;
var b = [ 42 ];

a == b; // true

値[42]の場合、抽象演算ToPrimitiveが呼び出され（「抽象演算」を参照）、結果は「42」になります。この時点から、単純な条件“ 42” == 42が残ります。これは、既にわかったように、42 == 42に変わり、aとbは型変換まで等しくなります。

ご想像のとおり、この章の前半で説明した抽象的なToPrimitive操作のすべての機能（（toString（）、valueOf（））は、この場合にも適用できます。これは、複雑なデータ構造があり、そのための特殊なメソッドvalueOf（）を定義します。これは、等しいかどうかを確認するために単純な値を提供する必要があります。

第3章では、基になるプリミティブ
値（「abc」）を返すプリミティブ値（たとえば、新しい文字列（「abc」）など）の周りのオブジェクトラッパーの「アンパッキング」について検討しました。この動作は、==アルゴリズムのToPrimitive変換に関連しています。

var a = "abc";
var b = Object( a );            //  ,  `new String( a )`

a === b;                           // false
a == b;                             // true

a == bは、bがToPrimitive操作によって基本の単純なスカラープリミティブ値「abc」に変換（または「アンパック」）されるため、aの値と一致するため、trueになります。

==アルゴリズムの他のオーバーライドルールのために、そうではないいくつかの値があります。例：

var a = null;
var b = Object( a );           //  ,  `Object()`
a == b;                            // false

var c = undefined;
var d = Object( c );           //  ,  `Object()`
c == d;                            // false

var e = NaN;
var f = Object( e );           //  ,  `new Number( e )`
e == f;                            // false

null値と未定義の値はパックできません（同等のオブジェクトラッパーがありません）。そのため、Object（null）はObject（）と基本的に異なりません。どちらの呼び出しも通常の
オブジェクトを作成します。

NaNは、Numberの同等のオブジェクトラッパーにパッケージ化できますが、==がアンパックを引き起こす場合、NaNの値はそれ自体と等しくないため、比較NaN == NaNは失敗します（第2章を参照）。
