Redux Toolkitは不要ですか？

Reduxを使用する際の膨大な量のボイラープレートコードの問題は誰にでも知られています。そして、標準化された便利なものを見つける希望を失うことなく、異なるプロジェクトで異なる松葉杖と自転車を使用しました。1年ほど前、私たちは調査に絶望し、問題の解決策を真剣に検討しました。これについては、以下で説明します。

登場ストーリー


当社は世界中のスタートアップのためのプロジェクトを開発しています。そして、これらのクレイジーなインスピレーションを受けた男にとって、最も重要な（またはそれらの1つ）は時間です。以前に市場に参入した-生存の可能性が高い。したがって、品質を犠牲にすることなく開発時間を短縮し、ヒューマンファクターに起因するエラーの数を減らすように常に努めています。ほとんどの場合、標準的なものを選択し、多かれ少なかれ安定しています。この場合、準備ができているものは何も見つからなかったので、彼らは袖をまくり、ここで何が循環できるかを見始めました。



いつものように、問題の説明から始めて、次のものを得ました。

  • コードを書くとき、各開発者は独自の微妙なアクセントを持っています。「非個人化コード」の概念の実装を達成することは困難です。
  • 多くのコード（ボイラープレート、コピーと貼り付け、それに続くすべての問題）、CRUDに100行以上。
  • 基本的なものを書くのに多くの時間が費やされます。1つは忘れて、もう1つは、読み込みの設定、エラーの処理などです。
  • ストアの構造はプロジェクトごとに異なり、場合によってはプロジェクトのさまざまな部分で異なります。

正しく尋ねられた質問または定式化された問題は、解決策の半分です。少し考えた後、ストアの形成に関するルールを紹介し、その構造を説明し、コードをリファクタリングしてくしにすることにしました。私たちのコードベースは信じられないほど大きくなかったという事実にもかかわらず、変更が必要なコードの量にすぐに遭遇しました。

首相は1日のうちの何時間かをどこで得るかについて困惑していましたが、平凡なところまで簡単なアイデアを生成して実装した愛好家が何人かいました。はい、独創的なアイデアは、登場後は常にシンプルに見えます。アイデアはこうでした。手動でコードを書き換える代わりに、コードを個別に生成するだけです。スニペットを使用して出口でシートを取得したくありませんでした。変更はすべて災害につながり、リファクタリングが繰り返されるため、いくつかのパラメーターを入力として取り、レデューサー、アクション、およびサガを構築する関数をすぐに見つけました。それで、コミュニケーションの最初のバージョンが生まれました（@ axmit / redux-communications） 「コミュニケーション」という名前は、それ自体が何らかの形で生まれたものです。このライブラリは、sagaとコンポーネントを結び付けます。そして、それは行きました...

直後、幸福が来ました。まあ、またはほとんどすぐに。開発時間は約20％減少しました（計算したとおり-別の記事のトピック）。まず、クロールするバグの数が大幅に減少したためです。そしてこれは、開発者が不注意によって愚かな間違いをする可能性がはるかに低いという事実は言うまでもありません。



Linus Torvaldsはかつてこう言っています。コードを見せてください。そして、私は彼に完全に同意します。ドックを引用したり、書き直したり、ここにコードシートを提供したりはしません。ほんの小さな例を挙げます。ライブラリの完全な説明へのリンクと例を含むサンドボックスも、記事の最後にあります。

典型的なタスクを考えてみましょう-エンティティのCRUDを作成する必要があります。タスクを例にとってみましょう。標準バージョンを説明するのは無意味だと思います。それは多くのスペースを必要とし、エディターに出くわした誰もがそれがどのように見えるかおおまかに想像するでしょう。そして、コミュニケーションをとるためには、次のことを行う必要があり

ます。

const basePath = `/api/task`;

const taskTransport = {
   add: task => axios.post(`basePath`, task).then(r => r.data),
   get: id => axios.get(`${basePath}/${id}`).then(r => r.data),
   update: task => axios.put(`${basePath}/${task.id}`, task).then(r => r.data),
   delete: id => axios.delete(`${basePath}/${id}`, task),
   getCollection: () => axios.get(basePath).then(r => r.data)
};

実際、トランスポートとコンポーネントの両方を作成する必要があり、完全な通信コードは次のとおりです。

import { taskTransport } from './task.transport';
import { CRUDStrategy, buildCommunication, StoreBranch } from '@axmit/redux-communications';
 
const namespace = 'task';
 
const strategy = new CRUDStrategy({
   namespace,
   transport: taskTransport
});
 
export const taskCommunication = buildCommunication(strategy);


これで、完全に機能するCRUDを作成することができます。もっと複雑なことが必要な場合は、CRUD通信を拡張するか、BaseCommunicationを使用できます。極端な場合、内部的には、すべての機能を備えた昔ながらの昔ながらのエディターと同じです。柔軟性は影響を受けていません。トランスポートは別のレイヤーに配置され、その実装は何でもかまいません。プロジェクトとaxiosを使用した簡単なクエリに関するgraphQLがあり、この点で問題は発生していません。注意深い読者は、ライブラリがsagasをエクスポートすることに気づいたかもしれませんが、これはその最も重要な制限の1つです。何らかの理由でsagasを使用できない場合、残念ながらこのライブラリは適していません。

なぜ今なのか？


記事を書くという決定は、この記事を読んだ後に来ました。このツールを突くと、通信がはるかにシンプルで簡潔になり、ストアの構造がより明確になり、柔軟性に劣らないことに驚いた。1時間座って、サンプルからredux-toolkitまでのソースを1時間整理した後、コミュニケーション用に書き直しました。違いを追跡しやすくするために最小限の変更を加えようとしましたが、私の見解では、例の構造は非常に混乱しています。具体的にはコードにコメントを残して、コードがどのようになり、どのようになったかを比較しやすくしました。* .communication.tsファイルと、それらが置き換えるスライスに注意してください。



行数がはるかに少なく、コード自体がはるかに快適（主観的）に見えるという事実はそれほど重要ではありません。製品ではかなり重いコミュニケーションがあります。もう1つ重要な違いがあります。コードは宣言型です。何をどこで取得したいのか、データをどう処理するのか、そしてどのようにそれを実行するのかを決定するだけです-まったく気にしません。
結論- カスタマイズにはredux-toolkit を使用する必要があります。それ以外にはMasterCa ... @ axmit / redux-communicationsがあります。

要約する


  • コードは、すべてのプロジェクトとすべての開発者の間で一貫しています。同様の問題は均一に解決され、ソリューションは多くの場合、個別のパッケージで提供され、将来的に再利用されます。同時に、コードの量は大幅に減少しました。
  • 6月は明確で理解できるフローを受け取りました。
  • 高齢者は、大量の定型コードを書く必要がなく、コミュニケーションを改善する方法を他に考えていることを喜んでいます。
  • デバッグが簡素化され、ストアの構造がシンプルになり、社内のすべての開発者が理解できるようになりました。
  • プロジェクト間またはシステムの異なる部分間の移行は、頭痛の種にはなりません。
  • PMも幸せです。バグの数は減少しました-顧客は満足しています。書き込みが簡単になりました-開発者は満足しています。首相は幸福のために他に何が必要ですか？
  • 徐々にコミュニケーションに移行するか、ハイブリッドアプローチ（コミュニケーション+スライス）を使用することもできます。

もちろん、ライブラリにも欠点があります。

  • サンプルコードとドキュメントがなければ、それを理解することは非常に困難です。
  • 任意のストアの構造を変更しても機能しません。すべての自動化の基盤となっているのは、ストアの構造です。しかし、私たちの仕事では、これで問題が発生したことはありません。
  • sagasでのみ作業できます。サンクは私たちに合わなかったし、私たちは常にサガを使うので、これは私たちにとって問題ではありません。しかし、何らかの理由でサーガがあなたに合わない場合は、ライブラリを使用できなくなります。

私は、既存の欠点にもかかわらず、私たちのライブラリが誰かに役立つことを願っています。それらの改善に関する提案や質問がある場合は、書いてください。私は議論させていただきます。
