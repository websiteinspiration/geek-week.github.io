検索結果とパフォーマンスの問題を表示する

すべての使い慣れたアプリケーションの典型的なシナリオの1つは、特定の基準に従ってデータを検索し、読みやすい形式で表示することです。並べ替え、グループ化、ページ付けの追加の機会もあるかもしれません。理論的には、この作業は簡単ですが、多くの開発者はそれを解決するときに多くの間違いを犯し、その結果、パフォーマンスが低下します。この問題のさまざまな解決策を検討し、最も効果的な実装を選択するための推奨事項を策定してみましょう。



ページングオプション＃1


頭に浮かぶ最も簡単なオプションは、検索結果を最も古典的な形式でページ分割することです。


アプリケーションがリレーショナルデータベースを使用するとします。この場合、このフォームで情報を表示するには、2つのSQLクエリを実行する必要があります。

  • 現在のページの行を取得します。
  • 検索条件に一致する行の総数を計算します-これはページを表示するために必要です。

例として、テストMS SQLデータベースのAdventureWorks for 2016サーバーを使用した最初のクエリを考えてみます。この目的のために、Sales.SalesOrderHeaderテーブルを使用します。

SELECT * FROM Sales.SalesOrderHeader
ORDER BY OrderDate DESC
OFFSET 0 ROWS
FETCH NEXT 50 ROWS ONLY

上記のクエリは、追加日で降順にソートされたリストの最初の50件の注文、つまり最後の50件の注文を表示します。

テストベースで迅速に実行されますが、実行計画とI / O統計を見てみましょう。


Table 'SalesOrderHeader'. Scan count 1, logical reads 698, physical reads 0, read-ahead reads 0, lob logical reads 0, lob physical reads 0, lob read-ahead reads 0.

クエリランタイムでSET STATISTICS IO ONコマンドを実行すると、各リクエストのI / O統計を取得できます。

実行プランからわかるように、最もリソースを消費するのは、ソーステーブルのすべての行を、追加された日付で並べ替えることです。そして問題は、テーブルに表示される行が多いほど、ソートが「難しく」なることです。実際には、このような状況は回避する必要があります。そのため、追加日にインデックスを追加し、リソース消費が変更されたかどうかを確認します。


Table 'SalesOrderHeader'. Scan count 1, logical reads 165, physical reads 0, read-ahead reads 5, lob logical reads 0, lob physical reads 0, lob read-ahead reads 0.

明らかに、それはずっと良くなりました。しかし、すべての問題は解決されましたか？商品の総費用が100ドルを超える注文の検索リクエストを変更してみましょう。

SELECT * FROM Sales.SalesOrderHeader
WHERE SubTotal > 100
ORDER BY OrderDate DESC
OFFSET 0 ROWS
FETCH NEXT 50 ROWS ONLY



Table 'SalesOrderHeader'. Scan count 1, logical reads 1081, physical reads 0, read-ahead reads 0, lob logical reads 0, lob physical reads 0, lob read-ahead reads 0.

おかしな状況があります。クエリプランは前のプランよりも少し悪いですが、実際の論理読み取り数は、フルテーブルスキャンの場合のほぼ2倍です。解決策はあります。既存のインデックスから複合価格を作成し、商品の合計価格を2番目のフィールドとして追加すると、165回の論理読み取りが行われます。

CREATE INDEX IX_SalesOrderHeader_OrderDate_SubTotal on Sales.SalesOrderHeader(OrderDate, SubTotal);

この一連の例は長期間続けることができますが、ここで私が表現したい2つの主な考えは次のとおりです。

  • 検索クエリに新しい基準または並べ替え順序を追加すると、その実行速度に大きな影響を与える可能性があります。
  • しかし、データの一部だけを差し引く必要があり、検索条件に一致するすべての結果を差し引く必要がない場合は、そのようなクエリを最適化する方法はたくさんあります。

では、最初に述べた2番目のクエリに進みましょう。検索条件を満たすレコードの数を数えるクエリです。同じ例を見てください-100ドルを超える注文を見つけます。

SELECT COUNT(1) FROM Sales.SalesOrderHeader
WHERE SubTotal > 100

上記の複合インデックスが存在する場合、以下を取得します。


Table 'SalesOrderHeader'. Scan count 1, logical reads 698, physical reads 0, read-ahead reads 0, lob logical reads 0, lob physical reads 0, lob read-ahead reads 0.

SubTotalフィールドが最初の位置にないため、クエリがそれを使用できないため、クエリがインデックス全体を通過することは驚くに値しません。この問題は、SubTotalフィールドに別のインデックスを追加することで解決され、その結果、論理読み取りはすでに48回しかありません。

数量をカウントするリクエストの例をいくつか追加できますが、本質は同じです。データの一部を受信し、合計数量をカウントすることは、根本的に異なる2つのリクエストであり、それぞれに最適化のための独自の対策が必要です。一般的なケースでは、両方のクエリで同等に機能するインデックスの組み合わせを見つけることができません。

したがって、このような検索ソリューションを開発する際に明確にすべき重要な要件の1つは、見つかったオブジェクトの総数を企業が確認することが本当に重要かどうかです。それはしばしば起こりません。そして、私の意見では、ほとんどのページングシナリオは「次のページに移動する」ように見えるため、特定のページ番号へのナビゲーションは、範囲が非常に狭いソリューションです。

実際、特定のページ番号に移動する方法がないという事実だけが変更されており、この表では、表示できるページ数を知る必要はありません。しかし、問題が発生します-（「次へ」リンクを正しく表示するために）次のページのデータがあるかどうか、テーブルはどのように知るのでしょうか？

答えは非常に簡単です。データベースから、表示する必要があるレコードよりも1レコードだけ差し引くことができます。この「追加」レコードの存在により、次の部分があるかどうかがわかります。したがって、1ページのデータを取得するには、クエリを1つだけ完了する必要があります。これにより、パフォーマンスが大幅に向上し、そのような機能のサポートが容易になります。私の実務では、合計レコード数のカウントを拒否すると、結果の出力が4〜5倍に加速される場合がありました。

このアプローチの場合、ユーザーインターフェイスにはいくつかのオプションがあります。上の例のように、「戻る」コマンドと「進む」コマンド、表示された結果に新しい部分を単に追加する「さらに読み込む」ボタン「無限スクロール」は、「もっと読み込む」の原理で機能します。 「しかし、次の部分を取得するためのシグナルは、ユーザーが表示されたすべての結果を最後までスクロールすることです。視覚的なソリューションが何であれ、データサンプリングの原則は同じです。

ページングの実装のニュアンス


上記のクエリのすべての例では、クエリ自体が結果行と返される行の数を示すときに、「オフセット+数値」アプローチが使用されています。最初に、この場合のパラメーターの転送を整理するのに最適な方法を検討します。実際に、私はいくつかの方法に出会いました：

  • リクエストされたページのシリアル番号（pageIndex）、ページサイズ（pageSize）。
  • 返される最初のレコードのシリアル番号（startIndex）、結果としてのレコードの最大数（カウント）。
  • 返される最初のレコードのシリアル番号（startIndex）、返される最後のレコードのシリアル番号（endIndex）。

一見、これは初歩的で違いがないように見えるかもしれません。しかし、これはそうではありません-最も便利で普遍的なオプションは2番目（startIndex、count）です。これにはいくつかの理由があります。

  • +1 , , pageIndex pageSize . , 50 . , , . «+1» , , 1 51, — 51 101 .. 51 pageIndex, 52 102 .. , — «» , .
  • ほとんどのデータベースでクエリを実行するには、最後のレコードのインデックスではなく数量を転送する必要があるため、3番目のオプションはまったく意味がありません。endIndexと基本的な算術演算からstartIndexを減算するとしますが、ここでは不必要です。

ここで、「オフセット+数量」によるページングの実装の欠点を説明する必要があります。

  • 次の各ページの取得は、前のページよりもコストがかかり、時間がかかります。これは、データベースが検索と並べ替えの基準に従って「最初から」すべてのレコードを調べ、目的のフラグメントで停止する必要があるためです。
  • すべてのDBMSがこのアプローチをサポートできるわけではありません。

代替案はありますが、それらも不完全です。これらのアプローチの1つ目は「キーセットページング」または「シークメソッド」と呼ばれ、次のように構成されます。一部を受け取った後、ページの最後のレコードのフィールドの値を覚えてから、それらを使用して次の部分を取得できます。たとえば、次のリクエストを実行しました。

SELECT * FROM Sales.SalesOrderHeader
ORDER BY OrderDate DESC
OFFSET 0 ROWS
FETCH NEXT 50 ROWS ONLY

そして最後のレコードで、注文日 '2014-06-29'の値を取得しました。次に、次のページを取得するために、これを試すことができます。

SELECT * FROM Sales.SalesOrderHeader
WHERE OrderDate < '2014-06-29'
ORDER BY OrderDate DESC
OFFSET 0 ROWS
FETCH NEXT 50 ROWS ONLY

問題は、OrderDateが一意でないフィールドであり、上記の条件が必要な行の多くをスキップする可能性があることです。この要求を明確にするために、一意のフィールドを条件に追加する必要があります（75074が最初の部分の主キーの最後の値であると想定します）。

SELECT * FROM Sales.SalesOrderHeader
WHERE (OrderDate = '2014-06-29' AND SalesOrderID < 75074)
   OR (OrderDate < '2014-06-29')
ORDER BY OrderDate DESC, SalesOrderID DESC
OFFSET 0 ROWS
FETCH NEXT 50 ROWS ONLY

このオプションは正しく機能しますが、条件にOR演算子が含まれているため、一般的には最適化が困難になります。主キーの値がOrderDateの増加に伴って増加する場合、SalesOrderIDによるフィルターのみを残すことで条件を簡略化できます。ただし、主キーの値と結果の並べ替えの基準となるフィールドの間に厳密な相関関係がない場合-ほとんどのDBMSでは、このORは回避できません。私が知っている例外はPostgreSQLで、タプルの比較が完全にサポートされており、上記の条件は「WHERE（OrderDate、SalesOrderID）<（ '2014-06-29'、75074）」と書くことができます。これらの2つのフィールドを持つ複合キーがある場合、同様のリクエストはかなり簡単です。

2番目の代替アプローチは、たとえばElasticSearchスクロールAPIまたはCosmos DB-データに加えてクエリが、次のデータバッチを取得できる特別な識別子を返す場合。この識別子に無制限の有効期間がある場合（Comsos DBの場合など）、これはページ間の順次遷移によるページングを実装する優れた方法です（上記のオプション＃2）。考えられる欠点：すべてのDBMSがサポートされるわけではありません。次の部分の受信した識別子の有効期間は限られている可能性があり、これは一般にユーザー操作（ElasticSearchスクロールAPIなど）の実装には適していません。

複雑なフィルタリング


さらに作業が複雑になります。オンラインストアの誰もが知っている、いわゆるファセット検索を実装する必要があるとします。この場合、注文テーブルに基づく上記の例はあまり意味がないため、AdventureWorksデータベースからProductテーブルに切り替えます。


ファセット検索のアイデアは何ですか？その中で、各フィルター要素について、この基準に一致するレコードの数が、他のすべてのカテゴリで選択されたフィルターを考慮して表示されます。

たとえば、この例で「バイク」カテゴリと色「黒」を選択すると、テーブルには黒いバイクのみが表示されますが、同時に次のようになります。

  • 「カテゴリ」グループの各基準について、このカテゴリの製品の数が黒で表示されます。
  • 色グループの各基準について、その色の自転車の数が表示されます。

このような条件の結果を出力する例を次に示します。


カテゴリに「衣料品」のマーク付けに加えて、テーブルには利用可能な黒い衣服も表示されます。「色」セクションの黒の製品の数も新しい条件に従って再計算されます。「カテゴリ」セクションでのみ、何も変更されません...これらの例が、使い慣れたファセット検索アルゴリズムを理解するのに十分であることを願っています。

これをリレーショナルベースで実装する方法を想像してください。カテゴリや色などの基準のグループごとに、個別のリクエストが必要になります。

SELECT pc.ProductCategoryID, pc.Name, COUNT(1) FROM Production.Product p
  INNER JOIN Production.ProductSubcategory ps ON p.ProductSubcategoryID = ps.ProductSubcategoryID
  INNER JOIN Production.ProductCategory pc ON ps.ProductCategoryID = pc.ProductCategoryID
WHERE p.Color = 'Black'
GROUP BY pc.ProductCategoryID, pc.Name
ORDER BY COUNT(1) DESC



SELECT Color, COUNT(1) FROM Production.Product p
  INNER JOIN Production.ProductSubcategory ps ON p.ProductSubcategoryID = ps.ProductSubcategoryID
WHERE ps.ProductCategoryID = 1 --Bikes
GROUP BY Color
ORDER BY COUNT(1) DESC
