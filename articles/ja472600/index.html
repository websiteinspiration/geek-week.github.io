序文


Gitは最も人気のある分散バージョン管理システムです。[1] [2]

Gitの主な目的は、プロジェクトの継続的に改善された状態のスナップショットを保存することです（Pro git、2019）。

この記事は、少なくとも基本的な知識とgitを使用するスキルがあり、知識を広げたい人を対象としています。

ここではgitの技術的な側面のみを検討します。gitの哲学とその内部実装の詳細については、いくつかの役立つ本を読むことをお勧めします（推奨参考文献を参照）。

1. gitの構成


gitを使い始める前に、自分で構成する必要があります。

1.1構成ファイル


  • / etc / gitconfig-すべてのユーザーとリポジトリの一般設定
  • 〜/ .gitconfigまたは〜/ .config / git / config-特定のユーザー設定
  • .git / config-特定のリポジトリの設定

特別なチームがあります 

git config [<>]

これにより、必要に応じてgitの標準動作を変更できますが、構成ファイルを手動で編集できます（とても速いと思います）。

git config（--system、--global、--local）コマンドに渡すパラメーターに応じて、設定はこれらのファイルのいずれかに書き込まれます。これらの「レベル」（システム、グローバル、ローカル）はそれぞれ、前のレベルの値を再定義します！

インストールされている設定を確認するには、git config --list --show-originを使用します。

ファイルを無視する
gitでは、どのファイルをどのコミットに入れるかを決定しますが、特定のファイルをインデックスとコミットに含めず、追跡されていないリストにも表示しないようにすることができます。これを行うには、リポジトリに特別なファイル（.gitignore）を作成し、無視されたファイルのテンプレートをそこに書き込みます。各リポジトリにそのようなファイルを作成したくない場合は、core.excludesfileを使用してグローバルに定義できます（「便利な設定」を参照）。作業中のプログラミング言語用の完成した.gitignoreファイルをダウンロードすることもできます。
.gitignoreをカスタマイズするには、bash正規表現を使用します。

1.2デフォルト設定


サーバーとクライアントの両方に多数のgit設定があり、ここでは基本的なクライアント設定のみが考慮されます。

使用する

git config name value

ここで、nameはパラメーターの名前、valueは設定を設定するためのその値です。
例：

git config --global core.editor nano

デフォルトのエディターnanoがインストールされます。

git config --get [name]を使用して既存のパラメーターの値を確認できます。nameは、値を取得するパラメーターです。

便利な設定：

  • user.name-コミットの作成時に使用される名前
  • user.email-コミットの作成時に使用するメール
  • core.excludesfile-テンプレートが特定のファイルをグローバルに無視するために使用されるファイル
  • core.editor-デフォルトのエディター
  • commit.template-内容がデフォルトのコミットメッセージに使用されるファイル（「コミットの操作」を参照）。
  • help.autocorrect-1に設定すると、gitは誤って記述されたコマンドを実行します。
  • credential.helper [mode]-資格情報ストレージモードを設定します。[キャッシュ]-資格情報は一定期間保存され、パスワードは保存されません（--timeout [秒]データが削除されるまでの秒数、デフォルトは15分）。[ストア]-認証情報は無制限に平文で保存されます（--file [ファイル]はデータを保存するパスを示し、デフォルトでは〜/ .git-credentials）。

1.3エイリアス


Gitのすべてのコマンドを完全に出力したくない場合は、エイリアスを簡単に構成できます。エイリアスを作成するには：

git config alias.SHORT_NAME COMMAND

ここで、SHORT_NAMEは省略形の名前で、COMMANDは省略形のコマンドです。例：

git config --global alias.last 'log -1 HEAD'

このコマンドを実行した後、git lastを実行すると、現在のブランチの最後のコミットに関する情報を表示できます。

以下の略語を使用することをお勧めします（独自の略語も定義できます）。

  • st =ステータス
  • ch =チェックアウト
  • br =ブランチ
  • mg =マージ
  • cm =コミット
  • reb =リベース
  • lg = "git log --pretty = format： '％h-％ar：％s'"

構成設定を表示するには、git config --listを使用します。

2.2ファイルのステータス


リポジトリ内のファイルのステータスを表示するには、次を使用します。

git status [<>]

このコマンドは、現在どのブランチにいるか、およびすべてのファイルのステータスを表示できます。必須のオプションはありません。-sのみが有用なオプションと区別でき、ファイルの状態の概要が表示されます。

ファイルのライフサイクル
図からわかるように、ファイルの追跡を解除したり追跡したりできます。監視対象のファイルは、未変更（未変更）、変更（変更）、準備済み（段階的）の3つの状態になります。
「監視されていない」ファイルを（git addを使用して）追加すると、「準備済み」の状態になります。
ファイルを「変更なし」状態に変更すると、「変更済み」状態になります。変更されたファイルを保存すると（つまり、「変更」状態）、ファイルは「準備済み」状態になります。ファイルをコミットすると（つまり、「準備済み」の状態）、ファイルは「変更なし」の状態になります。
HEADと作業ディレクトリのファイルのバージョンが異なる場合、ファイルは「変更済み」状態になります。それ以外の場合（HEADと作業ディレクトリのバージョンが同じ場合）ファイルは「変更されていない」状態
になります。HEADのファイルのバージョンが異なる場合作業ディレクトリですが、インデックス内のバージョンと変わらない場合、ファイルは「準備済み」状態になります。

このループは、次のように表すことができます：
未変更->変更->ステージング->未変更
つまり、ファイルを変更してインデックスに保存し、コミットしてから、もう一度やり直します。

2.3インデックスの操作


gitリポジトリのライフサイクルがどのように見えるかを理解していただければ幸いです。次に、gitリポジトリ内のインデックスとファイルを管理する方法を見てみましょう。

インデックスは、最後のコミットと次のコミットの間の中間の場所です。インデックスにファイルを追加または削除できます。コミットすると、ワークスペースからではなく、インデックスからデータを取得します。

インデックスを表示するには、git statusを使用します。

インデックスにファイルを追加するには
git add [<>]

便利なgit addコマンドオプション：

  • -f、-force-無視されたファイルも追加します
  • -u、-update-追跡されたファイルを更新


インデックスからファイルを削除するには、2つのgit resetコマンドとgit restoreコマンドを使用できます。
git-restore-作業ツリーファイルを復元します。
git-reset-現在のHEADを指定した状態にリセットします。
実際、両方のコマンドで同じことを実現できます。

インデックスから一部のファイルを削除するには、次を使用します。

git restore --staged <file>

このようにして、最後のコミット以降にgit addが実行されなかったかのように、インデックスを復元します（または、インデックスから特定のファイルを削除します）。このコマンドを使用すると、作業ディレクトリを復元して、コミット後に変更が加えられていないように見せることができます。ただし、このコマンドの動作は少し奇妙です。新しいバージョンのファイルをインデックスに追加した場合、インデックスがHEADと異なるまでは作業ディレクトリを変更できません。したがって、最初にインデックスを復元し、次に作業ディレクトリを復元する必要があります。残念ながら、単一のコマンドでこれを行うことはできません。両方の引数を渡す場合（git restore -SW）は何も起こらないためです。同様に、インデックス内のファイルとHEADが異なる場合、-Wの転送も行われません。多分、これは、誤って作業ディレクトリを変更しないように保護するために行われました。しかし、この場合、なぜデフォルトで-W引数が渡されるのですか？一般に、なぜこれが行われたのか、なぜこのコマンドがまったく追加されたのか理解できません。私にとって、リセットはこのタスクに非常によく対応し、インデックスと作業ディレクトリを最後のコミットだけでなく他のコミットにも移動できるため、機能が豊富です。

しかし、開発者自身はgit restore -Sを使用してインデックスをリセットすることを推奨しています。gitの代わりにHEADをリセットしてください。

git statusを使用すると、どのファイルが変更されたかを確認できますが、ファイルの変更内容を正確に知りたい場合は、次のコマンドを使用します。

git diff [<options>]

したがって、引数なしでコマンドを実行することにより、インデックスを作業ディレクトリと比較できます。既にファイルをインデックスに追加している場合は、git diff --cachedを使用して、最後のコミット（または指定したコミット）と作業ディレクトリの違いを確認します。2つのコミットまたはブランチを引数として渡すことで、それらの違いを確認することもできます。例：git diff 00656c 3d5119は、コミット00656cと3d5119の違いを示します。

2.4コミットの操作


インデックスが正しい状態になったので、変更をコミットします。編集後にgit addを実行しなかったすべてのファイルは、このコミットに含まれないことに注意してください。実際、そこにはファイルがありますが、古いバージョン（存在する場合）のみです。

変更をコミットするには、次を使用します。

git commit [<>]

git commitコマンドの便利なオプション：

commit.templateを使用して、デフォルトのコミットメッセージを定義できます。構成ファイル内のこのディレクティブは、デフォルトのコミットに使用されるコンテンツのファイルを担当します。例：git config --global commit.template〜/ .gitmessage.txt。

2.5閲覧履歴


コマンドを使用する 

git log [<>] [<->]

リポジトリのコミット履歴を表示できます。特定のコミットをソートおよび検索するためのオプションも多数あります。

便利なgit logコマンドオプション：

  • -p-各コミットの違いを表示します。
  • --stat-コミットごとに変更されたファイルの統計を表示します。
  • --graph-ブランチとマージ履歴を含むASCIIグラフを表示します。

コミットを時間、数量などで並べ替えることもできます。

  • -（n）最後のn回のコミットのみを表示します。
  • --since、--after-指定した日付以降に行われたコミットを表示します。
  • --until、-before-指定した日付より前に行われたコミットを表示します。
  • --author-著者のエントリが指定された文字列と一致するコミットのみを表示します。
  • --committer-コミッターエントリが指定された文字列と一致するコミットのみを表示します。
  • --grep-メッセージに指定された文字列が含まれるコミットのみを表示します。
  • -S-コードの変更によって指定された行が追加または削除されたコミットのみを表示します。

次に例を示し
ます。git log --since = 3.weeks-過去2週間のコミットを表示し
ますgit log --since = "2019-01-14"-2019-01-14で行われたコミットを表示し
ますgit log --since = "2年1日前」-2年1日前に行われたコミットを表示します。

2.6リモートリポジトリの操作


gitは分散ハード通貨であるため、ローカルリポジトリだけでなく、外部リポジトリでも作業できます。

リモートリポジトリは、外部サーバーに保存されているプロジェクトのバージョンです。

外部リポジトリを操作するには、次を使用します。

git remote [<options>]

http URL経由でリポジトリのクローンを作成した場合は、すでに外部リポジトリへのリンクがあります。それ以外の場合は、

git remote add [<options>] <name> <adres>

-f、-fetchを使用して、すぐに外部ブランチを抽出できます（外部リポジトリのブランチの名前と状態を取得します）。 --mirror [=（push | fetch）]を使用してデータを送受信するようにリポジトリを設定することのみができます。タグを取得するには、-tagsを指定します。

接続された外部リポジトリを表示するには、引数なしのgit remoteを使用するか、git remote -vを使用して、リポジトリとの間でデータを送受信するためのアドレスを表示します。

ブランチを追跡するには、git branch -u <rep / br>を使用します。repはリポジトリの名前、brは外部ブランチの名前、branchはローカルブランチの名前です。または、gitブランチ--set-upstream local_br origin / brを指定して、外部ブランチを監視するローカルブランチを指定します。

ブランチが外部を追跡している場合、どのブランチ（ローカルまたは外部）が遅れているか進んでいるか、およびコミットの数によってわかります。たとえば、コミット後にgit pushを実行しなかった場合、ブランチは外側のコミットより1コミット先になります。これについてはgit branch -vvを実行して確認できますが、最初にgit fetch [remote-name]（--allを実行してすべてのリポジトリから更新を取得）して、外部リポジトリから最新のデータを取得します。ブランチトラッキングをキャンセルするには、git branch --unset-upstream [<local_branch>]を使用します。

git pull [rep] [branch]を使用して、外部リポジトリからデータをダウンロードします。ブランチが外部を追跡する場合、git pullを実行するときにそれらを指定することはできません。デフォルトでは、監視対象のすべてのブランチからデータを受け取ります。

ブランチを新しいブランチにアップロードするには、git checkout -b <new_branch_name> <rep / branch>を使用します。

サーバーにデータを送信するには、

git push [<rep>] [<br>]

ここで、repは外部リポジトリの名前、brは送信するローカルブランチです。このエントリgit push origin master：devを使用することもできます。したがって、ローカルマスターブランチをoriginにアップロードします（ただし、devと呼ばれます）。送信する権限がないと、外部リポジトリにデータを送信できません。また、データが外部ブランチに先行している場合は、データを外部ブランチに送信できません（通常、-fを使用して送信できます-この場合、外部リポジトリの履歴を書き換えます）。ブランチが外部を追跡している場合は、ブランチの名前を省略できます。

外部ブランチを削除するには

git push origin --delete branch_name

外部リポジトリに関する詳細情報（HEADで示される、送受信用のアドレス、外部ブランチ、git pull用に構成されたローカルブランチ、およびgit push用に構成されたローカルリンク） 

git remote show <remote_name>

外部リポジトリの名前を変更するには、次を使用します 

git remote rename <last_name> <new_name>

外部リポジトリへのリンクを削除するには、次を使用します 

git remote rm <name>

3. gitでの分岐


ブランチは強力なツールであり、gitの主な機能の1つです。リポジトリのさまざまなブランチをすばやく作成して切り替えることができるためです。分岐の主な概念は、開発のメインラインから離脱し、メインラインに干渉することなく、独立して作業を続けることができるということです。ブランチは常にその中の最後のコミットを指し、HEADは現在のブランチを指します（gitのポインターを参照）。

3.1基本的な操作


ブランチを作成するには、 

git branch <branch_name> [<start_commit>]

ここで、branch_nameは新しいブランチの名前で、start_commitはブランチが指すコミット（つまり、ブランチの最後のコミット）です。デフォルトでは、ブランチは親ブランチの最後のコミットになります。

Gitブランチオプション：

競合を解決するときは、2つのブランチからの変更のうち、残す部分を選択する必要があります。競合するファイルを開くと、次のものが含まれます。

>>>>>>>ここに名前マージするブランチ

競合を解決したら、コミットしてマージを完了する必要があります。

競合中、どのファイルにどのような違いがあるかを確認できます。
git diff --ours-マージ前とマージ後の
差分

マージを許可しない場合は、「私たち」のバージョン（つまり、現在のブランチにあるバージョン）を選択するか、マージされたブランチにある「自分の」バージョンを選択して、競合を修正せずに、異なるマージ戦略を使用します。それぞれgit merge --Xoursまたはgit merge --Xtheirsを実行します。

4. gitのポインター


gitにはHEADブランチのようなポインターがあります。実際、すべてが非常に単純ですHEADは現在のブランチを指し、ブランチはその中の最後のコミットを指します。しかし、理解するためには、HEADが最後のコミットを示していると想像するのが良いでしょう。

4.1ポインタの移動


Pro git bookは、リポジトリを管理する方法の非常に良い例を提供しているので、私もそれを使い続けます。Gitが3つの異なるツリーのコンテンツを管理していると想像してください。ここで、「ツリー」は「ファイルのセット」を指します。
通常の操作では、Gitは3つのツリーを管理します。

  • HEAD-次の親、最後のコミットのスナップショット
  • インデックス-次のコミットのスナップショット
  • 作業ディレクトリ-サンドボックス

実際、gitは3つのツリーすべてを操作するためのツールを提供しています。次に、リポジトリの3つのツリーを操作できるgit resetコマンドについて説明します。

このコマンドのさまざまなオプションを使用すると、次のことができます。

  • --soft-ヘッドのみをリセット
  • --mixed-ヘッドとインデックスをリセット
  • --hard-HEAD、インデックス、作業ディレクトリをリセットします

リセットとは、指定されたコミットに移動することを意味します。デフォルトは--mixedです。

例1. 3つの追加のコミットを行い、それぞれに小さな変更が加えられ、そのうちの1つを作成したいので、git reset --softを使用して、インデックスと作業ディレクトリをそのままにしてコミットしながらHEADポインターを移動できます。その結果、ストーリーは1回のコミットですべての変更が行われたように見えます。

例2.インデックスに追加のファイルを追加し、そこから削除したいとします。これにはgit reset HEAD <files ...>を使用できます。または、コミットファイルをいくつかのコミットのように見せたいですか？先に述べたように、最後のコミットまでしかリセットされないgit restoreとは異なり、インデックスを任意のコミットにリセットできます。混合オプションでのみ、指定したファイルにアクションを適用できます！

例3.プロジェクトの新機能の作業を開始しましたが、突然雇用主はそれが不要になり、怒りに満ちた状態でgit reset --hardを使用し始めていないときにインデックス、ファイル、およびHEADを返しました特徴。そして翌日、機能はまだ洗い流されるべきだと言われます。しかし、何をすべきか？3つのツリーすべてをロールバックしたため、git logを使用して履歴でそれらを見つけることができないため、先に進む方法。そして出口があります-これはgit reflogリンクログです。このコマンドを使用すると、HEADが指している場所を確認でき、コミット履歴だけでなく上にも移動します。このログは各ユーザーのローカルです。

一般的に、私よりも多くの例を考え出すことができると思います。結論として、私はgit resetで魔法をかけることができると言います...
