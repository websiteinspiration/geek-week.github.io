自動化テストソフトウェアQIWI-ターミナル

こんにちは、ハブル！

今日は特定のトピックについて話します。QIWIセルフサービス端末のソフトウェアテスト自動化です。

自動化のテストのトピックには、Webサービスのテストなど、数回上下にスキャンされる領域があります。そのような領域については、個別のツール、パターン、およびベストプラクティスがあります。あなたは何も発明する必要はありません、リスクは最小です、あなたは取り、実行します。

逆の状況があります。対象分野は特定であり、既製のソリューションをのぞき見することはできません。ツールはなく、製品の技術スタックは独特です。スティックやええと...他の即興の素材から自分で道具を作り、同時にさまざまなサイズと致命的な力を持つ多くの多くの熊手を集める必要があります。

今日はこういうことを話したかった。この記事は、銀行の端末やセルフサービスマシン用のソフトウェアの開発とテストに携わる人に適しています。また、「しかし、このような状況も発生します」という例を挙げて、技術的な視野を広げたい人にも。2020年のQIWIターミナル。バックグラウンドでは、その充填を確認できます。




問題


最初のQIWI端末は2004年に登場し、携帯電話のアカウントを補充することができました。それ以来、大量の水が流れ、ASO（セルフサービスマシン）は大きく変化しました。彼らの助けを借りて、銀行カードの補充、送金、さまざまなサプライヤー（プロバイダー）のサービスの支払い-住宅および公共サービス、交通警察の罰金、信用機関、独自のQIWI製品（QIWIウォレット、無利子分割払いカード良心）を利用できます。QIWI端末は、postamata（オンラインストアの自動配達ポイント）のモードでも動作します。

端末はどのようなものですか？


鉄の観点から見ると、ASOターミナルは、請求書アクセプターとコインアクセプター、レシートプリンター、ディスペンサー、POS端末（プラスチックカードを受け取る）などの特定の周辺機器を備えた通常のデスクトップです。

そして、これが最初の問題です。


この有名な写真はカリフォルニア州ソノマ郡で撮られました。彼らは何も処理せずに、これらは自然な色であると言います。ちなみに、アメリカ最南端のロシアの植民地があったのはソノマ郡の領土でした。

はい、それは彼女です。 2014年に廃止されたオペレーティングシステムはWindows XPです。 POSReadyバージョンは2019年まで引き上げられましたが、...それほど単純ではありません。

事実、端末はQIWIの所有物ではありません。彼らはパートナー、いわゆるQIWIでターミナルを購入し、ショッピングセンターのスペースを借り、ほとんど受動的な収入を得るエージェント、個人起業家、および法人。

したがって、QIWI端末のフリートの90％（およびロシアと海外では100k以上）は、さまざまなハードウェア（512 MBから4 GBのRAM、さまざまなCPU（Pentium 4の深層ゼロから年まで））でWindows XPを実行しています。最近のCore i5）と、インターネットの品質と速度の違い。時代の異なる端末。定期的にアップグレードされる端末もあれば、長期間アップグレードされていない端末もあります（動作します-触れないでください）。私たちの仕事は、最新のターミナルソフトウェアアップデート（MarATLと呼ばれます）を定期的に提供し、さまざまなトッピングや周辺機器との互換性を維持することです。

期限切れのサポートオペレーティングシステム


簡単なテスト自動化回路を想像してみてください。 TeamCityやJenkinsなどのCIサーバーがあります。私たちの端末ソフトウェアは、イベント（コミットまたはヒープ）によってrawから組み立てられ、収集されたバイナリファイルはどこかに配置されます。ソフトウェアが自動的にインストールされ、夜間の機能の自動テストが起動します...ええと、やめてください！ソフトウェアのインストールはどこに行っていますか？そしてどうやって？

上記で述べたように、端末の90％はWindows XPを実行しており、2014年に終了しました。つまり、このOSは長い間セキュリティポリシーに準拠しておらず、アップデートもリリースされておらず、動作する最新のソフトウェアもありません。ネイティブのMicrosoft Visual Studioでさえ、タンバリンまたはMSBuildコレクターのツールチェーンのバージョンで踊った後にのみ、C ++をコンパイルします。

一般に、Windows XPを搭載したターミナルまたは仮想マシンでテストや何らかのソフトウェアを実行することは、非常に悪い考えです。XPでBuildagent TeamCityを上げることはできません。Ansibleプレイブックでそのようなマシンを構成することはできません。そこにもコンテナーはありません。Windows XPの仮想化テンプレートもそれほど多くありません。情報セキュリティを考えている大企業では、Windows XPマシンは企業ネットワークから、またはドメインよりも遠ざけられます。つまり、端末（またはそのふりをする仮想マシン）とのすべての対話はリモートで行われる必要があり、端末自体には最小限のサードパーティソフトウェアが必要です。

決定


多くのOpenSSHとWindows XPについて聞いて驚いた人もいます。最新バージョンのMicrosoftでは、OSHサポートはシステムに直接含まれています。 not-so-modern（Windows 7）には、PowerShellを使用するSSHインストーラーがあります。 C Windows XP、状況は少し悪いですが、ほんの少しです。追加のソフトウェアなしで実行できるインストーラーの古いバージョンがあります。

SSHは、よく知られている技術である、ホストをリモート制御するための古くて優れた信頼できる方法です。複数の処理を並行して実行できるSSHコネクタのクラスを実装する必要があります。たとえば、オンラインモードで、ターミナルから新しいログをロードし、いくつかのコマンドを実行します（ファイルのコピー、スクリプトの実行、権限の発行、プロセスのリストの取得、ターミナルのシステム時間の監視など）。

システム時間の監視は特に興味深いものです。標準のコマンドラインツールでは、Windows XPは時間を時間、分、秒などの形式でのみ表示し、UNIX時間は表示しません。たとえば待ち伏せは、XPが長い間タイムゾーンの更新を受け取っていないことであり、ロシア政府は夏時間または冬時間をキャンセルする一定のゲームで知られています。

たとえば、モスクワタイムゾーンの標準のWindows XP SP3は、実際のモスクワ時間より1時間早い時間を示しています。したがって、いずれの場合も端末ログのタイムスタンプは、テスト回路の時刻と一致しません。

支払い-支払い-支払い...


QIWI端末インターフェースは、本質的にはブラウザーエンジンで実行されるWebであり、通常はかなり古いものです。よるとのWebSocketプロトコル、それはMarATL、実際の端末ソフトウェアと対話します。一連のコマンドを使用してターミナルで支払いを選択すると（たとえば、セルラー通信の支払い）、支払いプロバイダーが選択され、手数料の額、請求書の受け入れに関する制限が決定され、特定のプロバイダーに対して支払いが作成され、支払いターミナルの処理を通じて支払いに送られます。

Webインターフェースをリモートで操作することはまあまあのアイデアであり、特にインターフェーステストは別のタスクです。標準のindex.htmlではなく、テスト時間に基づくテストWebページを作成できます。このページは、ターミナルソフトウェアで動作するJSスクリプトを実行します。一方、テストが実行されているホストに向かって外向きに見えるWebソケットクライアントを作成します。

自動テストの側面では、ターミナルからのクライアント接続の開始を待機し、テストページを通じて変更されていないコマンドをターミナルソフトウェアに送信するWebソケットサーバーを実装する必要があります。ソフトウェアの応答は、テストWebソケットサーバーにも送信されます。

つまり、支払いテストの一連のコマンドをJSONの形式で記述できます。これは、送信する必要があるコマンドと、それらを待機するための応答の種類を連続的に記述します。

そして、お金はどこにありますか？


最も油断のならない技術的な作業は、端末の周辺機器をシミュレートすることです。レシートプリンターは簡単にシミュレートされ、データをファイルに出力する仮想プリンターがOSで作成されます。このチェックをターミナル（またはターミナルソフトウェア自体によって形成されたそのコピー）から取得し、デコードして、たとえば、このチェックに必要なすべてのフィールドが存在することを同時にチェックできます。

お金を受け入れるデバイスでより複雑-ビルのアクセプターとコインのアクセプター。支払いシナリオをテストする場合は、ターミナルの請求書受領者の存在を何らかの形でエミュレートし、請求書の受け入れ、返却、さまざまな問題のシミュレーション（たとえば、しわくちゃの請求書の返却など）の機能をサポートする必要があることは明らかです。

ほとんどの入金デバイスは、CashCode NETプロトコルを使用して動作します。このプロトコルは、請求書受領者とコントローラー間の相互作用のシナリオを定義します（この場合、コントローラーはターミナルソフトウェアです）。


CashCodeプロトコルを使用した紙幣のコントローラーとバリデーターの相互作用の例。特定の頻度で、コントローラーはデバイスのステータスを要求します。


CashCodeコマンドの最小サイズは6バイトです。コマンドの説明については、CashCode NETプロトコル仕様を参照してください。

標準の請求書受領装置は、COMポートを介して端末に接続されています。VSPEなど、マシン上に仮想シリアルポートを作成できるサードパーティのユーティリティがあります。 TCP接続を介してこのポートを転送するためのスクリプトもサポートされています。

私たちのケースはより単純です。WinAPIを使用してポートに固執することができ、SSHを使用して接続できる最も単純なstdin / stdoutなどの任意の便利なインターフェイスを持つツールが必要です。

別のストリームのTulaは、シリアルポートを介してコントローラーと通信し、必要に応じて、おそらく「お金」を受け取る状況を模倣します。また、テストアカウントまたはプロバイダーのプールの種類も必要です。支払いは実際には処理されませんが、ある時点で払い戻されます。または、より複雑なケースでは、お金は実際の処理を通過しますが、特別な口座に送られ、そこから再び支払いテストのために引き落とされます。

これが自然のお金の循環です。

年中無休


端末とATMのソフトウェアは、インターネットの不足、請求書受領者の問題など、さまざまな緊急事態に対して非常に耐性がある必要があります。このようなソフトウェアは非常に高い優先度と権限を持つ端末で実行されるため、基本的にすべてのOS制御を傍受します。単に折りたたんだり、Alt-F4で閉じたりすることはできません。ソフトウェアはそれ自体と端末を含めて再起動できます。ループで。このようなストレスの多いシナリオ、たとえば支払い処理との通信の欠如を確認することも必要です。端末はこれに定期的に再起動して応答します。端末ソフトウェアをロードするたびに、端末が正しく正しく動作することを確認する必要があります。

ゼロからのインストールと更新


インストールされた端末ソフトウェアは、支払い処理を使用して一元的に更新されます。処理では、特定の端末の更新を計画し、その端末の更新プロファイル、つまり、どのファイルとどこからロードするかを決定します。更新するには、処理データベースでいくつかの手順を実行し、端末を監視する必要があります。必要なファイルのダウンロードが開始され、アップデートが正常に完了し、ターミナルソフトウェアが起動していることをログで確認します。

ゼロからのクリーンインストールは困難です。通常、ユーザーは、インストールファイルをターミナルにコピーし、設定と認証データをインストーラーフォームに入力するだけでフィールドで実行します。インストーラーフォームは、通常、標準のWindowsコントローラー（TextBox、CheckBoxなど）を備えたWinAPIアプリケーションです。

クリーンインストールスクリプトの自動テストでは、Pythonスクリプトがターミナルにアタッチされ、インストーラーが起動し、pywinautoライブラリを使用してコントローラーとやり取りし、独自のインストールログを書き込みます。インストールの最初の段階が完了すると、スクリプトは終了し、ターミナルソフトウェアは処理中に承認を渡し、インストールプロファイルへのパスを受け取り、必要なすべてのファイルをダウンロードします。ここでは、リアルタイムで、SSHを介して端末のログを監視する必要があります。インストール中のすべての異常な状況（たとえば、不正な認証データ）がそこに書き込まれます。これらのログをオンラインで読み取り、必要に応じて、クリーンインストールテストが失敗したことを考慮してください。
