変成テスト：この有望な手法について誰も知らない理由


認めざるを得ません。私はACM Magazineを読んでいます。プログラマーの基準であっても、私は「オタク」になります。とりわけ、私はこの雑誌から「変成テスト」について学びました。私が尋ねたすべての人々のように、私は彼のことを以前に聞いたことがありません。しかし、このトピックに関する科学文献は驚くほど膨大です。完全に異なる研究分野でのその応用の信じられないほど多くの例があります。では、なぜ彼のことを聞いたことがないのでしょうか。科学コミュニティ以外の人向けの記事は1つだけです。今それらの2つがあるとしましょう。

簡単な背景


ほとんどの書かれたテストはoraclesを使用します。つまり、あなたは答えを知っており、計算が正しい答えを与えるかどうかを明示的にチェックします。

def test_dist():
    p1 = (0, 3)
    p2 = (4, 0)
    assert dist(p1, p2) == 5

oracleテストに加えて、手動テストもあります。テスターはコンピューターの前に座り、入力データを結果と比較します。システムがより複雑になるにつれて、手動テストはますます有用でなくなります。それぞれがはるかに大きな状態空間内の1つの点のみをチェックするため、状態空間全体を探索するものが必要です。

これは私たちを生成テストに導きます：状態空間のランダムなセットをカバーするテストを書くこと。生成テストの最も一般的なスタイルは、プロパティベースのテスト、つまりPBTです。関数の「プロパティ」を見つけて、入力値を生成し、出力値がこのプロパティと一致するかどうかを確認します。

def test_dist():
    p1 = random_point()
    p2 = random_point()
    assert dist(p1, p2) >= 0

PBTの利点は、より多くのスペースをカバーできることです。その欠点は、特異性が失われることです。これはもはやオラクルのテストではありません！私たちは答えがどうあるべきかを知りません、そして関数は誤っているかもしれませんが、同じ性質を持つ方法で。ここでは、ヒューリスティックに依存しています。

PBTの深刻な問題は、優れた特性を見つけることです。ほとんどの関数には、単純な一般的なプロパティと複雑な特定のプロパティがあります。一般的なプロパティは多数の関数に適用できますが、それらは多くの情報を提供しません。より具体的なプロパティはより多くの情報を提供しますが、それらを見つけるのはより困難であり、限られたタスクの領域にのみ適用できます。グラフが非循環かどうかを決定する関数がある場合、どのプロパティテストを記述しますか？それらは機能が正しいことをあなたに自信を与えますか？

動機


次に、より複雑なタスクについて考えます。英語用の音声テキスト変換（STT）コンバーターを作成するとします。サウンドファイルを受け取り、テキストを表示します。どのようにテストしますか？

手オラクルを使用する最も簡単な方法。文をディクテーションし、出力テキストがそれと一致するかどうかを確認します。しかし、これでは十分ではありません！人間の発話の範囲は広大です。 1,000または10,000の異なるサウンドファイルをテストすることをお勧めします。文字起こし付きの手持ち式のオラクルは高すぎるでしょう。つまり、代わりにプロパティベースのテストを使用する必要があります。

しかし、どのようにして入力を生成するのでしょうか？たとえば、ランダムな文字列を作成し、それらをテキスト音声変換（TTS）コンバーターに渡し、STTが同じテキストを生成することを確認できます。しかし、これでも人間の声の範囲は非常に限られています。 TTSはイントネーションを変更し、言葉を飲み込み、強いアクセントを模倣しますか？それらに対処できない場合、STTは特に役立ちますか？ラジオ、ポッドキャスト、オンラインビデオなどの任意のテキストを使用することをお勧めします。

ここで新しい問題が発生します。TTSを使用するときは、テキストから始めました。任意のサウンドファイルの場合、それはありません。同時に、手動で書き起こしたくありません。代わりに、プロパティの使用に制限されています。テストする必要があるプロパティは何ですか？最も単純なプロパティの例：「プログラムは受信データでクラッシュしません」（優れたプロパティ）または「アコースティック音楽を単語に変換しません」（たぶん？）。これらの特性は、プログラムの主要なタスクの検証を十分にカバーしておらず、その品質への信頼をわずかに高めています。

したがって、2つのタスクがあります。まず、音声による大量の入力が必要です。次に、手動で音声をオラクルに変換するのに長い時間を費やすことなく、それらを有用なテストに変換する方法を理解する必要があります。

変成テスト


これらすべてについて、出力は個別に考慮されます。それらをより広いコンテキストに埋め込むとどうなりますか？たとえば、サウンドクリップが出力outに転記される場合、次の場合は常に取得outする必要があります。

  • 音量を2倍に
  • 頻度の増加
  • ペースを上げる
  • 背景ノイズを追加する
  • 車両ノイズを追加する
  • 上記の任意の組み合わせ。

これらはすべて、簡単にテストできる「単純な」変換です。たとえば、「車両ノイズ」を使用したテストの場合、車のノイズのサンプルを10個取り、それらをサウンドクリップに配置して、11のバージョンすべての認識結果が一致するかどうかを確認できます。ボリュームを2倍にするか増やして、11バージョンを33バージョンにしてから、ペースを2倍にして66バージョンにすることができます。この原則は、データベース内の各サウンドクリップに適用でき、受信データのスペースを大幅に拡大します。

比較のための66バージョンの存在は非常に便利です。しかし、それだけではありません。出力がどうあるべきかを知る必要はありません。 66のコンバージョンがすべて返された場合out、その後、テストは成功しました。少なくとも1つが別のものを返す場合、テストは失敗します。何が含まれているかを確認する必要はありませんout。これは非常に重要です。したがって、人間の関与をほとんど必要とせずに、テストスペースを大幅に拡大します。たとえば、シリーズのエピソードをダウンロードし、変換を実行して、テキスト1への変換のすべての結果が一致するかどうかを確認できます。ボイスクリップを聞かなくても、便利なテストが行われました。これで、オラクルを使用せずに複雑で詳細なテストを生成できます。

2セットの入力データとそれらの出力データが相互に接続されます。着信/発信データのセットに関連するこのようなプロパティは、変成関係2と呼ばれます。。このプロパティを適用するテストは、変成テストと呼ばれます。複雑なシステムでは、興味深い変成関係は、個々の着信/発信データの興味深いプロパティよりも簡単に見つけることができます。

もう少し正式に述べてみましょう：xandがある場合f(x)、いくつかの変換xを実行してx2、とを取得できますf(x2)。 STTの場合、チェックするだけですf(x) = f(x2)が、2つのデータセット間の任意の関係を使用できます。タイプf(x2) > f(x)または「f(x2)/f(x)整数値」の変成関係がある場合があります。同様に、この原則は、f(x)およびを使用して、入力データのいくつかのセットに拡張できます。f(x3)。この例としては、フィルターを使用しない検索エンジンの結果と、1つまたは2つのフィルターを使用するエンジンの結果を比較する場合があります。私が読んだユースケースの説明の大部分では、クレイジーなバグを見つけるのに十分であるため、2セットの入力データのみが使用されています。
