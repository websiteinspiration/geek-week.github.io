Linuxでの高速ルーティングとNAT

IPv4アドレスが使い果たされると、多くの通信事業者はアドレス変換を使用してクライアントのネットワークへのアクセスを整理する必要に直面しています。この記事では、コモディティサーバーでキャリアグレードのNATレベルのパフォーマンスを取得する方法について説明します。

ちょっとした歴史


IPv4アドレス空間が不足するというトピックは、もはや新しいものではありません。ある時点で、待機リストがRIPEに表示され、その後、アドレスのブロックを取引し、賃貸料のために取引を締結する取引所がありました。徐々に、通信事業者はアドレスとポートの変換を通じてインターネットアクセスサービスを提供し始めました。誰かが各加入者に「白い」アドレスを与えるのに十分なアドレスを取得できなかった一方で、誰かが流通市場でのアドレスの購入を拒否することでお金を節約し始めました。ネットワーク機器メーカーはこのアイデアを支持し、この機能には通常、追加の拡張モジュールまたはライセンスが必要です。たとえば、MXルーターラインナップのジュニパー（最新のMX104およびMX204を除く）では、NAPTを個別のMS-MICサービスカードで実行できます。CiscoASR1kにはGNライセンスが必要です。Cisco ASR9kでは、個別のA9K-ISM-100モジュールとそのモジュールに対するA9K-CGN-LICライセンス。一般に、喜びには多くのお金がかかります。

イプテーブル


NATを実行するタスクは、特別なコンピューティングリソースを必要としません。たとえば、任意のホームルーターにインストールされている汎用プロセッサがそれを解決できます。キャリア規模では、この問題は、FreeBSD（ipfw / pf）またはGNU / Linux（iptables）を実行している汎用サーバーを使用して解決できます。FreeBSDは考慮しません。私はこのOSを長期間使用することを拒否したので、GNU / Linuxに焦点を合わせましょう。

アドレス変換をオンにすることはまったく難しいことではありません。最初に、natテーブルのiptablesにルールを記述する必要があります。

iptables -t nat -A POSTROUTING -s 100.64.0.0/10 -j SNAT --to <pool_start_addr>-<pool_end_addr> --persistent

オペレーティングシステムは、すべてのアクティブな接続を監視し、必要な変換を実行するnf_conntrackモジュールをロードします。いくつかの微妙な点があります。まず、キャリアの規模でのNATについて説明しているため、デフォルト値では変換テーブルのサイズが急速に壊滅的な値になるため、タイムアウトを厳しくする必要があります。以下は、サーバーで使用した設定の例です。

net.ipv4.ip_forward = 1
net.ipv4.ip_local_port_range = 8192 65535

net.netfilter.nf_conntrack_generic_timeout = 300
net.netfilter.nf_conntrack_tcp_timeout_syn_sent = 60
net.netfilter.nf_conntrack_tcp_timeout_syn_recv = 60
net.netfilter.nf_conntrack_tcp_timeout_established = 600
net.netfilter.nf_conntrack_tcp_timeout_fin_wait = 60
net.netfilter.nf_conntrack_tcp_timeout_close_wait = 45
net.netfilter.nf_conntrack_tcp_timeout_last_ack = 30
net.netfilter.nf_conntrack_tcp_timeout_time_wait = 120
net.netfilter.nf_conntrack_tcp_timeout_close = 10
net.netfilter.nf_conntrack_tcp_timeout_max_retrans = 300
net.netfilter.nf_conntrack_tcp_timeout_unacknowledged = 300
net.netfilter.nf_conntrack_udp_timeout = 30
net.netfilter.nf_conntrack_udp_timeout_stream = 60
net.netfilter.nf_conntrack_icmpv6_timeout = 30
net.netfilter.nf_conntrack_icmp_timeout = 30
net.netfilter.nf_conntrack_events_retry_timeout = 15
net.netfilter.nf_conntrack_checksum=0

2番目に、デフォルトでは、変換テーブルのサイズは通信事業者の条件で機能するように設計されていないため、大きくする必要があります。

net.netfilter.nf_conntrack_max = 3145728

すべての変換を保存するハッシュテーブルのバケット数を増やすことも必要です（これはnf_conntrackモジュールのオプションです）。 

options nf_conntrack hashsize=1572864

これらの単純な操作の後、完全に機能する構造が得られ、多数のクライアントアドレスを外部プールに変換できます。ただし、このソリューションのパフォーマンスは不十分です。 NATにGNU / Linuxを使用する最初の試み（2013年頃）で、サーバーあたり0.8Mpps（Xeon E5-1650v2）で約7Gbit /秒のパフォーマンスを得ることができました。それ以来、GNU / Linuxカーネルネットワークスタックでさまざまな最適化が行われ、同じハードウェア上の1台のサーバーのパフォーマンスが1.8〜1.9 Mppsでほぼ18〜19ギガビット/秒（これらは制限値でした）になりましたが、トラフィック量の必要性があり、単一のサーバーで処理され、はるかに速く成長しました。その結果、さまざまなサーバー用のロードバランシングスキームが開発されましたが、これによりセットアップが複雑になり、提供されるサービスの質の向上と維持

Nftables


現在、DPDKとXDPの使用は、ソフトウェア「パケット転送」におけるファッショナブルな方向性です。このテーマについては多くの記事が書かれており、さまざまなプレゼンテーションが行われ、商用製品が表示されます（たとえば、VasExpertsのSKAT）。しかし、通信事業者のプログラマーのリソースが限られている状況では、これらのフレームワークに基づいて何らかの「シェア」を削減することは非常に問題があります。将来的にこのようなソリューションを運用することははるかに困難になります。特に、診断ツールを開発する必要があります。たとえば、DPDKを使用した通常のtcpdumpは機能せず、XDPを使用してワイヤに送り返されたパケットを「確認」できません。ユーザー空間へのパケット転送を出力するための新技術についてのすべての話の中で、レポートと記事は気付かれなかったnftablesでのフローオフロードの開発について、iptablesのメンテナーであるPablo Neira Ayuso氏。このメカニズムをさらに詳しく見てみましょう。

主な考え方は、ルーターがストリームの両側で1つのセッションのパケットを渡した場合（TCPセッションがESTABLISHED状態になった場合）、このセッションの後続のパケットをすべてのファイアウォールルールに渡す必要がないためです。これらのチェックはすべて、パケットをルーティングにさらに転送することにより、すべて同じ目的で終了します。はい、実際にはルートの選択を実行する必要はありません。このセッション内でどのインターフェイスとどのホストがパケットを転送するかはすでにわかっています。この情報を保存し、パケット処理の初期段階でルーティングに使用するだけです。 NATを実行する場合、nf_conntrackモジュールによって変換されたアドレスとポートの変更に関する情報を追加で保存する必要があります。はい、もちろん、この場合、iptablesのさまざまなポリサーやその他の情報統計ルールが機能しなくなります。ただし、サービスはデバイス間で分散されるため、独立したNATまたは境界などのタスクの一部として、これはそれほど重要ではありません。

構成


この関数を使用するには、次のものが必要です。

  • 新しいカーネルを使用してください。機能自体が4.16カーネルに登場したという事実にもかかわらず、しばらくの間、それは非常に「未加工」であり、定期的にカーネルパニックと呼ばれていました。LTSカーネル4.19.90および5.4.5がリリースされた2019年12月頃にすべてが安定しました。
  • かなり最近のバージョンのnftablesを使用して、iptablesルールをnftables形式に書き換えます。バージョン0.9.0で正常に動作します

原則的に最初の段落ですべてが明らかな場合、主なことは、アセンブリ中にモジュールを構成に含めることを忘れないこと（CONFIG_NFT_FLOW_OFFLOAD = m）であり、2番目の段落で説明が必要です。nftablesルールの説明は、iptablesの場合とはかなり異なります。ドキュメントはほとんどすべてのポイントを明らかにし、iptablesからnftablesへの特別なルールコンバーターもあります。したがって、NATとフローオフロードの設定の例のみを示します。例の小さな凡例：<i_if>、<o_if>はトラフィックが通過するネットワークインターフェイスですが、実際には3つ以上存在する可能性があります。<pool_addr_start>、<pool_addr_end>-「ホワイト」アドレスの範囲の開始アドレスと終了アドレス。

NAT構成は非常に簡単です。

#! /usr/sbin/nft -f

table nat {
        chain postrouting {
                type nat hook postrouting priority 100;
                oif <o_if> snat to <pool_addr_start>-<pool_addr_end> persistent
        }
}

フローオフロードはもう少し複雑ですが、理解できます。
#! /usr/sbin/nft -f

table inet filter {
        flowtable fastnat {
                hook ingress priority 0
                devices = { <i_if>, <o_if> }
        }

        chain forward {
                type filter hook forward priority 0; policy accept;
                ip protocol { tcp , udp } flow offload @fastnat;
        }
}

これが実際の設定です。これで、すべてのTCP / UDPトラフィックがfastnatテーブルに送られ、はるかに速く処理されます。

結果


これが「はるかに高速」であることを明確にするために、同じハードウェア（Xeon E5-1650v2）を搭載し、同じLinuxカーネルを使用して、iptables（NAT4）とnftables（NAT5）。



スクリーンショットには1秒あたりのパケットグラフはありませんが、これらのサーバーの負荷プロファイルでは、平均パケットサイズは約800バイトであるため、値は最大1.5Mppsになります。ご覧のとおり、nftablesを使用するサーバーのパフォーマンスマージンは非常に大きくなっています。現在、このサーバーは3Mppsで最大30Gbit / sを処理し、CPUリソースを解放しながら、40Gbpsネットワークの物理的な制限に明らかに対応できます。

この資料が、サーバーのパフォーマンスを向上させようとしているネットワークエンジニアに役立つことを願っています。
