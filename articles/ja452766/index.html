プログレッシブストリーミングテクノロジー、またはフリーズなしでネットワーク経由で4kビデオを視聴する方法



今日のインターネットの速度が100メガビット/秒であることには誰も驚かないでしょうが、その使い方には問題があります。すべての基本操作はネットワークを完全にロードしません。同時に、オンラインで見たい4k〜8kの重いオーディオ形式とビデオ形式の方が人気があります。そして、インターネットの高速性を見ると、論理的な疑問が生じます-なぜですか？プロバイダーが提供するフルスピードをマスターする方法は？クライアント側とサービス側の両方。記事でこれらすべての問題を検討してください。

私は本当に良いソフトウェア製品を作ったので、そこですべてがどのように配置されているのかをもっと詳しく伝えたいと思います。多くの新しい技術があります。おそらく、この業界で長い間誰も何も発明しておらず、その時が来たからでしょう。この記事は開発者だけでなく、一般の人も対象としています。私はすべてをできるだけ簡単に説明しようとしました。

1.今日のデータ転送テクノロジーの基本的な基礎から始めましょう。
高速データ転送アルゴリズムに関する多くの人々の知識には大きなギャップがあります-10メガビット/秒以上...これらのギャップを埋めましょう：

スペース番号1。問題は、無線伝送技術の全体が、最良のシナリオとして、継続的なデータ伝送に集中していることです。

wi-fi \ 3G \ 4Gを介して送信されたすべてのもの、さらには有線で送信されたものはすべて、最大伝送速度よりも低速であっても、伝送が一定であれば最も速く伝送されます。これは、断続的に送信するよりもはるかに高速ですが、最大速度になります。

原因：

  • 通信を再開するには、より多くのサービス情報をトラフィックに追加します。
  • 通信を再開するとき、サーバーはクライアントの評価を下げて、データを低速で送信する可能性があります（おそらく他のクライアントの出現が原因）、またはまったく送信しない可能性があります。家庭のwi-fiネットワークでも、ルーターは、たとえばwi-fi掃除機が原因で、定格を下げる可能性があります。それら。これは、3つ以上のクライアントがあるすべてのネットワークに適用されます。一般的に、ほぼ全員に:-)

スペース＃2。データをコピーする場合、どのプログラムも、ソースからデータを引き出して宛先に転送する一種の「バケット」を使用します。したがって、このバケットは、データ転送速度に応じて、サイズが異なる必要があります。理解するのは簡単です。小さじ1杯の水の入ったバケツをすくうと、マグカップをすくうよりもはるかに時間がかかります。

スペース＃3。ネットワーク経由でデータをダウンロードする速度を上げるには、RAMを使用する必要があります。着信データを記録する際のわずかなミリ秒の遅延は、秒、分、時間に注がれます。これが起こらないようにするには、まずRAMにデータを書き込み、次に永続的なメディア（ハードディスク）上のより大きな「バケット」にデータを書き込む必要があります。そうしないと、データ転送速度が劇的に低下します。

ファイルをコピーするだけで十分です。

2.これらのギャップは氷山の一角にすぎません。通常のファイル転送ではこれで十分ですが、ファイルがマルチメディアであり、オンラインでの再生を開始する必要がある場合はどうでしょうか。最新のマルチメディアファイルはRAMに完全に収まらないため、ディスクへの保存を検討する必要があります。

最良の戦略は、スペースに関する上記から構築されます。

  • データの読み込みは、再生に関係なく継続的に行われる必要があります。
  • 確実に巻き戻すには、別のデータロードストリームを作成する必要があります。
  • 起動時のフリーズを克服するには、技術データ（コーデックなど）の受信により、プリロードが必要です。経験的に、次の式を見つけました。ファイル全体のサイズは* 0.002または0.2％です。

これはFlacファイルを再生するのに十分です。

3.それは合理的にそして完全に私たちの通信チャネルを使用するオンラインストリーミングのために判明し、RAMとディスク容量が必要です。これがなければ、チャネルは十分に活用されません。

そして、ここでデータ読み込みアルゴリズムの分岐が始まります！技術的な詳細がなければ、おおよその量でそれらを提供します：

  • データが2度ダウンロードされないようにするには、ストリームの衝突を起こす必要があります。それら。トラックを開始してすぐに中央に巻き戻した場合、最初のストリームが中央に到達すると、作業が中断されます。
  • データが2回ダウンロードされないようにするため、データが既にロードされている場合はストリームを作成しないでください。
  • 通常の再生では、プレーヤーストリームとデータをダウンロードするストリームの間の相互作用の複雑なロジックが必要です。

私はこれらすべてともう少しやってみました。それは絶対に素晴らしいことがわかりました。データは飛んで、再生は中断されません。 Flacファイルは、再生の1〜3秒で完全にダウンロードされました。フルHDビデオを再生するにはそれで十分でした。

4.問題は、メディアファイルのサイズが大きく異なることです。また、重量が約80〜120 GBの4k Blu-rayムービーでは、何も機能しませんでした。プレーヤーは最初に15個のスレッドを作成し、それらすべてが速度を共有しました。もちろん、これはプレーヤーが待っていたメインスレッドには不十分でした。データは最大速度で読み込まれました。そうです...彼らはチャンネル全体をダウンロードしましたが、4kビデオがハングし、スライドショーよりも遅く再生されました。多くのストリームは4kビデオにとって有害で​​あることが判明しましたが、フルHDのメリットが4kビデオで有害に変わった境界はどこにありますか？

その結果、すべてがチャネルの速度に依存していました。ストリームの動作を最適化するには、次の2つのことを知っておく必要があります。a

）計算に必要な再生に必要な速度：（ファイルサイズ/秒単位の継続時間）* 8.

b）プレーヤーが現在使用しているメインストリームからデータをダウンロードする速度。

データのロードを管理する場合、ダウンロード速度を測定する機会が確実にあります。これで、各スレッドはその速度（Mbps単位）を知ったので、パフォーマンスに追加のコストがかかることはありません。プレーヤーが現在取り組んでいるストリームを特定してください。すべては通常のマルチスレッドモードで開始します。ただし、プレーヤーがメディアファイルの持続時間を秒単位で計算してこのデータを送信するとすぐに、すべてのストリームが再生に必要な固定速度を取得します。この直後に、すべてのストリーム（メインストリームを除く）が必要な速度をメインストリームの速度と比較し、その速度が低い場合は一時停止します。さらに、メインスレッドはゆっくりと速度を上げ、必要な速度を2回超えると（私が行ったように）、すべてのセカンダリフローを一時停止から一時停止します。

観察によれば、しばらくすると、メインストリームの速度が再び低下し、すべてのセカンダリストリームが再び一時停止し、データがダウンロードされるか再生が完了するまで続きます。このような柔軟な戦略は、データの読み込みが並列化され、必要に応じて1つのストリームに圧縮されると、通信チャネルを完全にダウンロードすると同時に、再生に必要なデータの可能な限り高速なダウンロードを保証します。この戦略は、10 MBと100 GBの両方のファイルで同様に機能します。損失なしでネットワークを介してプレイするために、より良いものを思いつくことは不可能です。何か提案がありましたら、コメントで喜んで話し合います。

ビットレート4kビデオは大きいですが、空高くはありません。今日最も効率的なVP9コーデックは、15 Mbpsで4kビデオを圧縮します。音声付きで約15.5が出てきます。 100 GBのBlu-rayムービーのビットレートは約60 Mbpsです... 4kビデオを見たい人は誰でもこれらのインターネット速度を持っています。これは、4kの動画をオンラインで視聴できることを意味します。

上記のアルゴリズムは単純ですが、実装は非常に複雑に見えます。ビデオとオーディオの品質の低下を伴うストリーミング技術は、正しいデータ転送アルゴリズムの実装の欠如のためにコンテンツを台無しにせざるを得ません。私は、多くの人々や企業が（上記の）知識のギャップを抱えていること、そしてもちろん、このアルゴリズムを実装することが困難であることを想定しています。したがって、彼はこのストリーミング方法の理解を簡単にするための記事を書きました。

このアルゴリズムを著作権を使用して作成するのは少し難しいですが、一般的にはそれも可能です。ディスクに保存して読み込む場合は、データを暗号化する必要があります。これはユーザーに対する敵のアクションのように見えますが、何をすべきかです。一部の企業はこれを行っています。

このテクノロジーがどのように機能するかの例として、マルチメディアアプリケーションであるMedia Libraryプレーヤーを作成しました。すべてのフォーマットをサポートしています。実行するために必要な機能を実証するには。デモサイトを開いて、「Films（TOP）/ 4k movies」ディレクトリツリーに移動します。そこで、映画を選択して見ることができます。すべての形式をサポートするには、プレーヤーモジュールをExoPlayerからVlcPlayerに切り替えます。
このシナリオでは、上記のすべてのデータ取得アルゴリズムが使用されます。私のアプリケーションは次のプロトコルをサポートしています：nmdc \ http \ ftp \ samba、およびMega.nzクラウド。

100 GBの4kビデオファイルは、最小限の遅延で再生され、可能な限り高速でメモリにダウンロードされます。一時停止するだけで、より多くのデータをプリロードできますが、これは通常必要ありません。再生は常に安定しています。安定していない場合でも、チャネル速度は再生に必要な速度に近くなります。

上記のギャップのために、大企業のポリシーは完全に正しくありません。そのため、進行が遅くなり、多くの場合、利用可能なデバイスと通信チャネルを完全に使用できなくなります。

アプリケーションは外部メディアへのキャッシュの保存を提供しますが、Googleからの外部メディアへのアクセスの奇妙な複雑さのため、現在、主にSamsungデバイスとAndroidバージョン5.0以降でサポートされています。将来的には、外部USBメディアの完全なサポートが表示されます。

一般に、このアルゴリズムにより、ストリーミング時のデータのダウンロードの品質と速度が大幅に向上します。小さいファイルは30〜50 MB、大きいファイルは50〜120 GBです。

観点では、それを使用することができます：

  • 携帯電話、テレビ、その他のデバイスで4k〜8kのメディアコンテンツを高品質に再生するため。
  • バーチャルリアリティでボリュームシーンを高解像度で表示します。
  • トレントのようなピアツーピアプロトコルと組み合わせる。
  • どのデバイスでも高解像度で高品質の音楽を再生できます。実験的なテストでは、2 Mbpsの不安定な通信チャネルで十分であることがわかりました。
