注釈の技術：Javaに適したKotlinコードの記述

こんにちは、ハブル！

今日は、JavaとKotlinの相互運用性という重要なトピックに触れます。提案されたパブリケーションの作成者は、Javaで作成されたコードベースをKotlinに書き直すことができないと合理的に想定しています。したがって、JavaコードとKotlinコードの相互作用を保証する方が正しいです。アノテーションでこれを行う方法を読んでください。

次の理由でこの記事を開いたと思います。

  1. 最終的にKotlinを試すことにしました。
  2. あなたは彼を好きでした、しかし、それは驚くことではありません。
  3. どこでもKotlinを使用することを決定
  4. 厳しい現実に直面：Javaを完全に見捨てることはできません。

どうして？

Kotlinがとてもかっこいいなら、どこでも、いつでもそれを使ってみませんか？ここでは、これが不可能であるいくつかのシナリオ：

  1. コードベース全体をゆっくりとKotlinに転送しようとすると、コマンドをに適用するのが単純に怖いファイルがあることがわかりますConvert Java to Kotlin file。リファクタリングの時間がある場合は、実行してください！ただし、実際のプロジェクトでは、リファクタリングの時間が常に見つかるわけではありません。
  2. コードは、JavaとKotlinの両方を使用するプログラマーによって使用されます。特に両方の言語をサポートするために多くの努力を必要としない場合（当然、私は注釈について話している）、それらすべてに特定の言語を使用するように強制することはできません（またはすべきではありません）。

ここでは、JavaとKotlin間の相互運用性を提供するいくつかの注釈を見ていきます。

Java

JvmFieldアノテーション

  • 彼女がすること？このプロパティのゲッターとセッターを生成せず、フィールドとして提供するようKotlinコンパイラーに指示します。
  • 最も一般的な実際のケースは、コンパニオンオブジェクトフィールドを提供することです。

使い方？Kotlin

内object / companion objectでフィールドを定義するとします。

object Constants {
val PERMISSIONS = listOf("Internet", "Location")
}

この関数をJavaから呼び出そうとする場合は、次のように記述する必要があります。

Utils.INSTANCE.getPERMISSIONS()

単純なフィールドのためのたくさんのコード！コードをすっきりさせるために、アノテーションを追加して余分な部分を削除しましょう。

object Constants {
@JvmField
val PERMISSIONS = listOf("Internet", "Location")
}

これで、Javaコードは次のようになります。

Utils.PERMISSIONS;

同じことが修飾子でも実現できますが、そのような修飾子はプリミティブ型または文字列でのみ機能します。 

//Kotin
object Constants {

}


//Java

  • 最も一般的な実際のケース：コンパニオンオブジェクトからのメンバー（関数、プロパティ）の提供。

使い方？Kotlinで

関数を定義するとしますobject。

object Utils {
fun doSomething(){ ... }
}

この関数をJavaから呼び出そうとすると、次のように書く必要があります。

Utils.INSTANCE.doSomething()

INSTANCEこの関数を呼び出すときはいつでも オブジェクトにアクセスする必要があります。コードをすっきりさせるために、アノテーションを使いましょう@JvmStatic。

object Utils {
@JvmStatic
fun doSomething(){ ... }
}

この関数をJavaから呼び出すと、次のように書くだけです。

Utils.doSomething();

はるかに良いですね。状況は、関数が元々Javaで静的メソッドとして作成されたかのようです。

注釈はフィールドにも適用できます。

@JvmStatic

}

このコードをJavaから呼び出すと、次のように記述できます。

Utils.getValues();

注：JvmFieldはフィールドとしてメンバーJvmStaticを提供しますが、関数を提供しますget。

そして、これはフィールドなvarので、メソッドも生成されsetます：

}

ただし、この場合、呼び出し呼び出しは次のようになるため、注釈を使用することはお勧めできません。

いつ使えないの？

部材は、改質続いときJvmStaticで注釈を付けることができないopen、overrideまたはconst。

この状況では、コードはコンパイルされません：



JvmOverloads

  • 彼女がすること？Kotlinコンパイラーに、デフォルトのパラメーター値を置き換えるこの関数のオーバーロードを生成するように指示します。
  • 過負荷とは何ですか？Kotlinでは、関数にデフォルトのパラメーターがある場合があるため、同じ関数をさまざまな方法で呼び出すことができます。Javaで同じことを行うには、この関数の個々のバリエーションを手動で定義する必要があります。これらの自動的に生成されたバリエーションのそれぞれを「過負荷」と呼びます。最も一般的な使用例：クラスコンストラクターのオーバーロード。はい、この手法はデフォルト設定のある機能で機能します。

使い方？

デフォルトのパラメーターを持つコンストラクター（またはその他の関数）を持つクラスがある場合...

class User constructor (
val name: String = "Test",
val lastName: String = "Testy", val age: Int = 0
)

...次に、このような関数をKotlinからさまざまな方法で呼び出すことができます。

val user1 = User()
val user2 = User(name = "Bruno")
val user3 = User(name = "Bruno", lastName = "Aybar")
val user4 = User(name = "Bruno", lastName = "Aybar", age = 21)
val user5 = User(lastName = "Aybar")
val user6 = User(lastName = "Aybar", age = 21) val user7 = User(age = 21)
val user8 = User(age = 21, name = "Bruno")
...

ただし、Javaからコンストラクターを呼び出そうとすると、1）すべてのパラメーターを渡すか、2）すべてのパラメーターにデフォルト値がある場合のみ、パラメーターをまったく渡せません。

オーバーロードを作成する場合は、アノテーションを使用できますJvmOverloads。

class User @JvmOverloads constructor ( val name: String = "Test",
val lastName: String = "Testy", val age: Int = 0
)

現在、Javaを使用する場合、多くの可能性



があります。ただし、この場合、Kotlinには多くのオプションはありません。たとえば、姓のみ、または年齢のみを渡すことはできません。

アノテーションはJvmOverloads、関数がデフォルトのパラメーターを持っているのと同じ数のオーバーロードのみを生成します。

  • 関数がある場合は、としてマークできますJvmOverload。これを他の注釈、たとえばと組み合わせることもできJvmStaticます。
  • いつ使用すべきではありませんか？関数にデフォルトのパラメータがない場合、この注釈は役に立ちません。

ファイル：JvmName

  • 彼女がすること？この要素から生成されたJavaクラスまたはメソッドの名前を指定します。
  • 最も一般的なケース：Kotlinファイルにより美しい名前を付けます。ただし、この注釈は、ファイルだけでなく、関数や、プロパティにアクセスするためのメソッド（ゲッターおよびセッター）にも適用できます。

彼女はどのように働きますか？

関数が特権要素であるKotlinでは、クラスの外部に存在する関数を記述できます。たとえば、新しいKotlinファイルを作成して次のコードを記述した場合、問題なくコンパイルされます。

//file name: Utils.kt

fun doSomething() { ... }

このコードはJavaから呼び出すことができます。

UtilsKt.doSomething();

注：ファイルの名前はUtilsですが、名前は呼び出し中に使用されますUtilsKtが、これは理想的ではありません。これを修正するには、ファイルの上に注釈を追加しますJvmName。

//  : Utils.kt
@file:JvmName("Utils")

fun doSomething() { ... }

接頭辞がどのように使用されてfile:いるかに注意してください。おそらくご想像のとおり、これは、使用している注釈がファイルレベルで適用されていることを示しています。Javaから次のコードを呼び出すと、

Utils.doSomething();

関数に注釈を付けることもできます。

//  : Utils.kt @file:JvmName("Utils")

@JvmName("doSomethingElse")
fun doSomething() { ... }

Kotlinからこのコードを呼び出すときは、元の名前（doSomething）を引き続き使用しますが、Javaでは注釈で指定された名前を使用します。

//Java Utils.doSomethingElse();


//Kotlin Utils.doSomething()

この機能は特に便利とは思われませんが、署名の競合を解決するために使用できます。このスクリプトは公式ドキュメントでよく理解されています。

ここでは、プロパティにアクセスするためのメソッドを使用できます。

class User {
val likesKotlin: Boolean = true
@JvmName("likesKotlin") get() = field
}

この呼び出しが注釈付きと注釈なしのJavaでどのように見えるかを確認してください。

//  
new User().getLikesKotlin()

//  
new User().likesKotlin()

接頭辞を使用しても同じことができますget。

class User {
@get:JvmName("likesKotlin")
val likesKotlin = true
}

  • この機会はいつ利用できますか？プロパティにアクセスするためのファイル、関数、メソッド。ただし、必要に応じて、必ず必要な接頭辞を付けてください。
  • いつ使用すべきではありませんか？関数に代替名を任意に設定すると、多くの混乱を招く可能性があります。このアノテーションは慎重に使用してください。適用する場合は、一貫して使用してください。
