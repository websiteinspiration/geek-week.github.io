werf 1.0の安定版の紹介：GitOpsがwerf 1.0をどのように使用するか、ステータス、計画



werfは、アプリケーションを構築してKubernetesに配信するためのオープンソースの GitOps CLIユーティリティです。 werfは、Dockerfileまたは独自の組み込みコレクター（YAML構文に基づき、AnsibleサポートとGitに基づく増分再構築を使用）を使用したアプリケーションイメージのアセンブルをサポートします。 Helm互換の構成フォーマットを使用して、アプリケーションを配信します。アプリケーションコード、収集されたイメージの構成、およびアプリケーションのロールアウト構成は、1つのGitリポジトリに格納されます。

待望の安定リリース1.0は、関数によって完成したユーティリティの基本バージョンです（最初の安定リリースの正確なバージョン番号は1.0.6です）。。基本バージョンでは、werfはアプリケーションの配信と保守の全サイクルをサポートしています。これには、アプリケーションイメージのアセンブル、Kubernetesへのデプロイ、未使用のイメージのクリーンアップが含まれます。

バージョン1.0で、同じプロジェクト（上のすべての操作があることが重要であるbuild、deploy、cleanup）単一のホストで実行する必要があります。これは、CIシステムで固定ワーカーを使用する必要があることを意味します。同時に、タスクの並列性に制限はありません。werfはこの問題を完全に解決します。さまざまなプロジェクトをさまざまなワーカーに分散することもできます。

このリリースに特化したこの記事では、このバージョンが提供する機能と提供しない機能、および将来のバージョンの計画について詳しく説明します。ただし、「GitOps」という用語の理解と、継続的インテグレーションおよびアプリケーション配信（CI / CD）のプロセスにおけるwerfの役割を理解することから始めます。

werfがGitOpsである理由


では、GitOpsとは何を意味し、werfはどの領域をカバーするのでしょうか。

「GitOps」という用語は、約2.5年前にWeaveworksによって作られたものであり、最近、ブログのこの新しい現象の本質を明らかにするその著者の記事を翻訳しました。私たちの理解では、GitOpsの基本的な考え方と基本的な意味は、Gitが「単一の真実のソース」 （単一の真実のソース）であるということです。Gitストア：

  • アプリケーションコード
  • すべての依存関係;
  • コンテナの収集方法に関する情報。
  • Kubernetesにデプロイする方法に関する情報。
  • や。。など。

そして、Gitの変更に合わせて現実を変える「何か」があります。このアプローチは、Gitを監視するオペレーターをKubernetesにインストールするだけでなく、任意のCIシステムから呼び出すことができるコンソールユーティリティを使用して実装できます。さらに、私たちの観点から見ると、CLIユーティリティを使用したアプローチは不要な制限を課しません。CIを任意のツールと任意の数のニュアンスで実行して、「現実」（つまりKubernetes）をGitの状態と同期させるCLIユーティリティを呼び出すことができます。 。

WERFアセンブリのための基本的な手順を用いて高CLIインタフェースを提供し、画像、アプリケーション配信、及び画像の治療公開します：werf build-and-publish、werf deploy、werf dismiss、werf cleanup。これらの非常に基本的なコマンドは特定のCIシステムに組み込まれ、現実との必要な同期を提供するものと想定されています。さらに、werfは、さまざまなサブシステムを管理するための低レベルのCLIインターフェースも提供します。ドキュメントの低レベルの管理コマンドを参照してください。werfはどのモデルにも組み込むことができるため

、組み込みのCI / CDがプッシュモデルとプルモデルのどちらで動作するかは関係ありません（詳細については、こちらを参照してください）。同時に、werfは、やkubernetes api-server などの個別の低レベルユーティリティの操作などの問題を解決し、統合されたCI / CDアプリケーションを構成するための「欠けている部分」です。gitdocker

werf 1.0安定版とは


1.画像の組み立て、公開、クリーニング


アプリケーションでDockerイメージのビルドが必要な場合、werf 1.0を使用すると、次のことができます。

  • 単一の構成で画像を組み立てるためのルールを記述します（複数の場合があります）werf.yaml。
  • 画像を収集してDockerレジストリに公開する
  • カスタムポリシーのDockerレジストリを定期的にクリーンアップします。

werfは、既存のDockerfileへの接続とStapelコレクターへの指示の2つの方法でアセンブリを記述できます。 Stapelを使用したビルドには、Gitでアプリケーションコードを変更するとき、アセンブリにAnsible構文を使用するときなど、増分再ビルドを加速するなどの利点があります。このコレクタと構文の詳細についてはドキュメントをご覧ください。その使用例はマニュアルに記載されています。 収集された画像のタグ付け/バージョン管理のさまざまなスキームは、Gitコミット、ブランチ、タグを参照して利用できます。 イメージのアセンブルは、アプリケーションのデプロイメントのオプションの段階であり、独自のアセンブルされたイメージがない場合はスキップできます。werf.yaml





2. 1つのホストのみでストレージをステージングする


werfは、ステージストレージの概念を導入しています。基本的なwerfコマンドは、次のようにステージストレージを使用します。

  • アセンブリ結果の保存-ステージストア内のDockerイメージ
  • ステージストアからのイメージを、新しいイメージの再構築およびアセンブルのキャッシュとして使用します。
  • リポジトリを使用して、収集された画像に関する情報を取得して、それらをさらに使用します（たとえば、アプリケーションをKubernetesに配信するとき）。

単一のアプリケーションを展開する場合は、すべてのチーム（アセンブリ、パブリケーション、イメージクリーニング、アプリケーションの展開）に単一ステージのストレージを使用する必要があります。

バージョン1.0では、ローカルホストストレージ（コマンドの対応するパラメーター:)のみがステージストレージとして機能できます--stages-storage=:local。使用すると、:localステージはディスクに保存されます。この結果：werf 1.0は、単一のアプリケーションのデプロイメントを編成するために1つのホストでのみ使用できます。このホストは、werfが正しく機能するために、コマンドの起動と起動の間にデータを保存する必要があります。

バージョン1.0では、外部ストレージへのステージの格納はサポートされていません。これにより、分散アセンブリを編成できます。ただし、そのような関数はwerfの将来のバージョンで表示される予定です（詳細は以下を参照）。

3.アプリケーションをロールアウトして、可用性を確認します


アプリケーションをロールアウトするために、ユーザーはHelmと互換性のある形式であるKubernetesマニフェストとテンプレートパラメーターのセットでチャートを説明します。

werfはKubernetesでアプリケーションを起動し、アプリケーションのロールアウトプロセスが完了する前に起動して機能することを保証します。これには、コンポーネントのロギングと、問題が発生したときにロールアウトエラーに即座に応答することが含まれます。ロールアウトコマンドはゼロ以外のコードでドロップされます。したがって、CI / CDでwerfロールアウトを使用すると、アプリケーションからダウンロードされたかどうか、問題をデバッグおよび修正するための十分な情報（ソフトウェアのような問題を見つけるために他のユーティリティを実行する必要がない）から、ソフトウェアから適切なフィードバックが得られますkubectl。

werfは既存のHelm 2インストールと完全に互換性がありますが、werfにはいくつかの利点があります。たとえば、リソースを更新するためのメカニズムとして、Kubernetesは3方向マージパッチを使用します。また、アプリケーションがクラスターに配信されるときにフィードバックを受け取る可能性もあります。違いの完全なリストはこのページで説明されています。

4.収集された画像とアプリケーション配信プロセスの関係


werfは、収集した画像を単一のシステムに統合します。つまり、アプリケーションにKubernetesを配信するプロセスで、タグ付けとバージョン管理のプロセスを行います。werfが収集するイメージは、Kubernetesリソース記述テンプレートで使用できます。

これらの関数が原因で、werfは Helm、Dockerおよびその他のビルダーやユーティリティよりも高レベルのインターフェースを提供し、個別にデプロイできると言えます。このインターフェイスを使用すると、werfを既存のCI / CDシステムに単純に統合でき、使用するすべてのコンポーネントを組み合わせるという問題を解決できません。彼がこのタスクを担当します。

5.既存のCI / CDシステムとの統合


バージョン1.0では、自動統合はGitLab CIシステムでのみ使用できます。このためのチームが提供されますwerf ci-env。CI / CDシステムから必要な情報を受信し、CI環境で正しく機能するようにwerfを自動的に構成します。

CI / CDシステムとの統合がどのように機能するかについての詳細は、マニュアルで確認できます（レビュー、GitLab CIの詳細、他のシステムとの統合）。

6. Linux、Windows、およびmacOSのクロスプラットフォーム開発


werf 1.0は、DockerとHelmの両方のリリースで独立して動作する静的にリンクされたバイナリファイルです。ホストシステムへの外部依存関係：

  • ローカルDockerデーモン
  • gitユーティリティ。

werfは、GNU / Linux、Windows、またはmacOSの任意のオペレーティングシステムおよび環境で実行できます。さらに、アセンブリの結果は、使用されるシステムに関係なく同じになります。キャッシュのステージの同じシグネチャ、ステージの同じ充填、このステージが収集されたシステムに関係なく。異なるシステムで動作するように設定を変更する必要もありません。

したがって、werf 1.0は、アプリケーションを構築してKubernetesに配信するためのクロスプラットフォームツールを提供します。

また、werfはLinux環境で動作する標準のDockerイメージを収集しますが、Windowsコンテナーはバージョン1.0ではサポートされていないことにも注意してください。

7.テストで機能をカバーする


現在、werfコードの60％が統合e2eテストと単体テストでカバーされています。

werfは、サポートされているすべてのオペレーティングシステム（Linux、Windows、およびmacOS）上でGitHubアクションを使用してテストされ、起動を整理します。いくつかのテストの詳細は、Code Climateでも入手できます。

8. werfのバージョン管理


現在、バージョン1.0のリリースまでに、プロジェクトは約700回のリリースを行っています。

werfは、安定性チャネル（alpha、beta、ea（early access）、stable、rock-solid ）を備えた高度なリリースシステムを使用しています。この投稿は、安定版チャネルでの最初のバージョン1.0のリリースと同時にリリースされる予定です。バージョンに対する不安定な変更は、最初にチャネルのチェーンを通過し、最終的には堅実なものになります。リリースは頻繁に行われ（場合によっては1日に数回）、変更は「小さな部分」で継続的に配信されます。

安定性チャネルと多くの頻繁なリリースにより、新しい変更に関する継続的なフィードバック、それらを迅速にロールバックする機能、一般にソフトウェアの高い安定性とその許容可能な開発速度を確保できます。

重要な点は、バージョン1.0-> 1.1、1.1-> 1.2の間の移行中に、ユーザーによる手動の介入を必要とするwerfのそのような変更が可能であるということです（それは、移行スクリプトまたはリリースで説明されている手動実行の単なる指示である可能性があります）。1.0内のバージョン（1.0.1、1.0.2、... 1.0.6-alpha.1、1.0.6-beta.2など）を更新すると、そのような手動の変更が不要になります。

今後の計画


将来のバージョンでの主な作業領域は、次のようになり、それらの実装に関するおおよその用語は次のとおりです。

1. werfを使用したアプリケーションのローカル開発およびデプロイメント


主な目標は、複雑なアクションなしで、ローカルと本番環境の両方でアプリケーションをデプロイするための単一の統一された構成をすぐに実現することです。

Werfには、アプリケーションコードを編集して、動作中のアプリケーションから即座にフィードバックを受け取ってデバッグするのに便利な操作モードも必要です。

バージョン1.1、2020年1月から2月

2.コンテンツベースのタグ付け


画像のコンテンツのみに基づいて、公開時に画像にタグを付ける。Gitコミットにバインドするモードとは異なり、このモードは不要なリビルドを完全に取り除きます。Git-commit-idは、ワークツリーコンテンツの普遍的な識別子ではありません（ただし、それに依存します）。

アプリケーションコードは変更されていないが、新しいコミットが行われた場合、Gitコミットの現在のタグ付けモードでは、公開時に新しい名前でイメージが作成されます。これには、Kubernetesでこのイメージを使用するリソースのロールバックも伴います。同時に、画像自体の内容は変わりませんでした。

これらの問題を解決するために、werfは、アプリケーションコンテンツのチェックサムの計算に基づく新しいタイプのタグ付け- コンテンツベースのタグ付けを導入します。

バージョン1.1、2020年2月から3月

3. Helm 3への移行


これには、新しいHelm 3コードベースへの移行と、既存のインストールを移行するための実績のある便利な方法が含まれます。

バージョン1.1、2020年2月から3月

4.画像の並列アセンブリ


現在、werf 1.0はwerf.yaml、で宣言された画像とアーティファクトのすべての段階を順番に収集します。ステージの組み立てプロセスを並列化する機能が必要です。

バージョン：1.1、2020年1月〜2月

5.画像の分散アセンブリ


現時点では、werf 1.0は1つの専用ホストでのみ使用できます（1つのホストのみのステージストレージについては、上記のポイントを参照してください）。

分散アセンブリの可能性を開くには、アセンブリが複数のホストで起動され、これらのホストがアセンブリ間（一時ランナー）の状態を維持しない場合、Dockerレジストリをステージリポジトリとして使用する可能性を実装するためにwerfが必要です。

以前は、werfプロジェクトがまだdappと呼ばれていたとき、そのような機会がありました。ただし、werfでこの関数を実装するときに考慮する必要がある多くの問題が発生しました。

バージョン1.2：2020年3月から4月

6. Kubernetes構成を説明するJsonnet


werfは、Jsonnet形式のKubernetesの構成記述をサポートします。同時に、werfはHelmとの互換性を維持し、記述形式を選択することができます。

その理由は、多くの人々によると、Go言語パターンには大きなエントリしきい値があり、これらのパターンのコード明瞭度も低下するという事実です。

Kubernetes構成記述システム（Kustomizeなど）を実装するための他のオプションも検討されています。

バージョン1.1：2020年1月から2月

7. Kubernetes内で作業する


目的：Kubernetesのランナーを使用して、イメージのアセンブリとアプリケーション配信を確実にすること。それら。新しいイメージのアセンブリ、その公開、クリーニング、デプロイは、Kubernetesポッドから直接行うことができます。

この機能を実現するには、最初にイメージを分散ビルドする機能が必要です（上記の段落を参照）。

また、Dockerデーモンなしのビルドの操作モード（つまり、Kanikoに似たビルドまたはユーザースペースでのビルド）のサポートも必要です。

werfは、Dockerfileだけでなく、インクリメンタルリビルドを備えたStapelビルダーでもKubernetesでのアセンブリをサポートします。

バージョン1.2：2020年4月から5月
