SCRUM指南中作者的隐藏含义是什么。第1部分。关于过程

谈论魔术和独角兽SCRUM-a吗？


许多人试图在家中实施SCRUM，这并不是什么秘密，但并不是每个人都能成功，而且许多人不了解魔术的来源。

立即同意，SCRUM的唯一手册是Scrum Guides，它会进行更改和更新，因此，我建议您定期重新阅读它。

这一系列文章将不会代替阅读本手册，而是对指南的补充，其中包括作者的一些个人补充。

我们可能将从手册最后一页上写的内容开始：

Scrum的角色，事件，工件和规则是不可变的，尽管仅可能实现Scrum的一部分，但结果并非Scrum。

（Scrum角色，事件，工件和规则保持不变。尽管仅可能引入Scrum的一部分，但结果不会是Scrum。）

这告诉我什么，在公司进行了4次转换之后，我们正在攻击的耙子的前额上已经有一个老茧。

关于这一点-如果我们想获得A95汽油，那么我们可能需要严格遵守生产标准，将批准栏中的油加热到严格定义的温度，将蒸汽加热到一定的高度，不要“用肉眼”添加成分，而是要注意最后一点他的母亲！工艺流程。为了使最终结果为A95，而不是某种会毁坏您的汽车的体重。

但为什么？为什么突然决定在家中实施SCRUM的典型（经典）经理认为“技术过程”不在他的公司中。他的人民不同，他们的手不同或他们的腿不同，代码写在错误的地方？总的来说，我似乎并不关心管理方法发展的30年。有一百万人有理由在一百万人中第一次发明您的自行车，然后自豪地在轮毂上写它，甚至写一本关于您的“黑色混乱”的书。通过开发人员的痛苦，使“ bodygirl”流行起来，破坏了公司几十年来一直遵循的既定经典过程，结果，它没有用。

// SCRUM-易于理解


这就是我的意思。比SCRUM更简单的方法是：进行计划，在早上花5分钟，然后在2周后收集所有人，让他们展示结果（通常不再需要），然后等待能给每个人带来金钱，欢乐和自豪的产品！只是？经理说，让我们实施吧！

这个“钩子”通常会遇到年轻而不是经验丰富的人，因为有经验的（读者）已经知道它不是那么简单。

// SCRUM-难以掌握


您知道Jeff和Ken在指南的这19页中隐藏了什么吗？一个简单的事实是，经理/经理/主管对自己的团队“越在乎”越多，自组织的团队越差，他们工作的结果越差。

每个人都知道（团队正在贬低的）不良经理是：

  • 无法委托
  • 他自己分发作品，他接受结果
  • 日常监控
  • 它需要不间断的报告，文档记录，填写时限（状态时间表）
  • 不信任团队
  • 施加决定

（我希望这里没有人认出我自己）

这是同样的“过度关注”，或者是“长者”，“父母”，“最负责任的”，“我只需要它”的感觉。

使用命令，这必定会带来坏魔术：

  • 团队失去思考能力。（您的经理在这里停止哲学思考，但请告诉我该怎么做）
  • 团队会处理任务，而不是结果，有时会假装要活跃。（我们执行任务1，任务2，任务3，并且产品下降了，让管理员/开发人员了解。）
  • 团队忙于报告，记录文档，但无法正常工作。（我在星期一和星期五半天写报告，我不做任何工作。）
  • 团队拒绝任何创新。（什么是自动测试？让我们完成任务。）

这并不奇怪，但是SCRUM只是“限制”团队不受“高级父母”的这种“过度控制”。

需要证明吗？根据《 Scrum指南》保留所有内容：

站立，仅适用于开发团队！


每个Scrum主管都知道，一旦“经理”到达，甚至是“我们的朋友”产品负责人，站起来便立即变成“状态报告”。团队将不再讨论如何使用它来实现冲刺的目标，但是突然之间，他们开始“在长者面前跳舞”，以告诉我在所有细节上我做了什么以及做得如何。相信我，这将立即花费超过15分钟的时间，最可悲的是，这对于所有团队成员来说都是绝对无用的浪费时间。

在该指南中，它并非徒劳无功-
每日Scrum是开发团队的内部会议。如果有其他人员在场，Scrum Master会确保他们不会打断metin

（Daily Scrum是开发团队的内部会议。
还有其他人，Scrum主管确保他们不会干扰会议。）

但是通常在新流程中成为产品负责人的前经理在不邀请他们站起来时讨厌它。SCRUM工艺流程中发生的第一件事就是所有站起来的人都来访问它，因为“控制高于一切”。

在与产品负责人的会议上，团队有不超过10％的时间，且不超过一分钟


（我指的是存在PO的人员，团队可以在需要时随时求助于PO。）

因为进行了为期2周的冲刺，所以严格按照以下3个会议进行：

  • Sprint计划最多4个小时
  • 重新冲刺最多2次
  • 和1.5小时的Sprint复古

就是这样，根据规定，产品所有者在2周内只有7.5个小时，在此绝对没有时间进行“控制”。团队剩余90％的时间致力于冲刺目标（但您是否考虑过您的团队真正可以编写多少代码？）

当然，实际上，我们以前的“经理”不会容忍这一点，并且会通过几次中间报告和集会来打破这种“混乱” 。

团队应该实现客户的需求，而不是产品所有者的个人目标。


因为没有任何地方写产品负责人必须写产品本身的要求，但是写的是确定优先级并使每个人都能理解的要求。

一个优秀的Scrum主管知道，为了确保“透明度”，必须邀请优先与团队见面的User Story客户。与客户建立直接沟通。因为给了客户承诺，哦，团队有多大的动力。

SCRUM的3个原则：透明，研究，适应

但是，理智的“经理”将允许什么？事实证明，“管理真理”正在受到自我的质疑，这是团队进行“谈判”并打破所有职业发展和占领宇宙计划的机会。

不，SCRUM是经典经理的噩梦，而过去的人们则生活在“流程”中。

因此，他们通常试图破坏整个SCRUM过程，增加控制权，减少透明度，并且不适应和发展。

总结：掩埋SCRUM的最佳方法是任命产品负责人，以前的项目或您自己的经理。

  • PO，不是领导者。
  • PO必须是一个具有独特能力的人-看到堆垛机，而其他人看不到他。
  • PO必须了解更多/更少钱/价值所在并确定优先级。
  • PO应该能够将堆垛机带到团队，团队本身已经可以在这里销售产品了。
