「クラシック」PHPの場合、問題は発生しません。クエリ実行の最後に変数が$name破棄され、後続の各リクエストは期待どおりに機能します。 PHPをデーモンとして開始する場合、リクエストごとにこの変数に別の都市が追加されます。これにより、マシンでメモリが不足するまで、配列が無秩序に増加します。

一般に、メモリが終了するだけでなく、プロセスの終了につながる他のエラーが発生する場合があります。このような問題があると、「クラシック」PHPは自動的に処理します。 PHPをデーモンとして起動する場合、このデーモンを何らかの方法で監視し、クラッシュした場合は再起動する必要があります。

このタイプのエラーは不快ですが、効果的な解決策があります。エラーのためにスクリプトが落ちないが、一部の変数の値が予期せず変更される場合（たとえば、配列をクリアする場合）は、はるかに悪いです$cities。この場合、後続のすべてのリクエストは、正しくないデータで機能します。 

要約すると、「クラシック」PHP（PHP-FPM、Apache mod_phpなど）のコードを書く方が簡単です。これにより、多くの問題やエラーから解放されます。しかし、このためにパフォーマンスで支払います。

上記の例からわかるように、コードの一部では、PHPは「クラシック」リクエストの各リクエストの処理に費やすことができなかった（または一度無駄にした）リソースを費やしています。これらは次の領域です。

OPcacheにもかかわらずpersistent_compile_file、クエリの実行時間の5.84％がかかることがわかります。 

これが発生する理由を理解するために、zend_accel_load_scriptのソースを見ることができます。 OPcacheが存在するにもかかわらず、すべての呼び出しで、include/requireクラスと関数のシグネチャは共有メモリからワーカープロセスのメモリにコピーされ、さまざまな補助作業が行われます。そして、この作業はリクエストごとに実行する必要があります。リクエストの最後にワーカープロセスのメモリがクリアされるためです。



これは、通常1回のリクエストで行う多数のinclude / require呼び出しによって悪化します。たとえば、Symfony 4には、コードの最初の有用な行を実行する前に約310個のファイルが含まれています。これは暗黙的に発生することがあります。以下のクラスAのインスタンスを作成するために、PHPは他のすべてのクラス（B、C、D、E、F、G）を自動ロードします。そして特にこの計画では、Composer'a宣言関数に従って割り当てられます。これらの機能がカスタムコードの実行中に確実に使用できるようにするには、PHP は関数を自動ロードしていないため、Composerは使用に関係なく常に接続する必要があり、呼び出し時に読み込まれます。

プリロードの仕組み


プリロードには、1つのメイン設定opcache.preloadがあり、そこにPHPスクリプトへのパスが渡されます。このスクリプトは、PHP-FPM / Apache /などの起動時に一度実行され、このファイルで宣言されるクラス、メソッド、および関数のすべてのシグネチャは、実行の最初の行からリクエストを処理するすべてのスクリプトで使用できます（重要注：これは変数とグローバル定数には適用されません-それらの値は、プリロードフェーズの終了後にゼロにリセットされます）。 include / require呼び出しを実行し、共有メモリからプロセスメモリに関数/クラスシグネチャをコピーする必要はありません。これらはすべて不変として宣言されているため、すべてのプロセスはそれらを含む同じメモリ位置を参照できます。

通常、必要なクラスと関数は異なるファイルにあり、それらを1つのプリロードスクリプトに結合するのは不便です。しかし、これを行う必要はありません。プリロードは通常のPHPスクリプトなので、必要なすべてのファイルに対してプリロードスクリプトからinclude / requireまたはopcache_compile_file（）を使用するだけで済みます。さらに、これらのファイルはすべて一度読み込まれるため、クエリ時にこれらのファイルを個別に接続している間は実行できなかった追加の最適化をPHPで行うことができます。PHPは、各個別ファイルのフレームワーク内でのみ最適化を行いますが、プリロードの場合は、プリロードフェーズでロードされたすべてのコードに対して最適化を行います。

ベンチマークプリロード


プリロードの利点を実際に示すために、私は1つのCPUバウンドエンドポイントBadooを取り上げました。私たちのバックエンドは通常、CPUに依存する負荷によって特徴付けられます。この事実は、非同期フレームワークを考慮しなかった理由への回答です。CPUに依存する負荷の場合には利点がなく、同時にコードをさらに複雑にします（別の方法で記述する必要があります）。また、ネットワーク、ディスクなどを操作するためにも役立ちます。特別な非同期ドライバーが必要です。 

プリロードの利点を十分に理解するために、私は実験のために、作業中にテスト済みスクリプトに必要なすべてのファイルをダウンロードし、wrk2を使用して通常の本番負荷に似たものをロードしました-Apache Benchmarkのより高度なアナログですが、同じくらい簡単です。

プリロードを試すには、まずPHP 7.4にアップグレードする必要があります（現在はPHP 7.2です）。PHP 7.2、プリロードなしのPHP 7.4、およびプリロードありのPHP 7.4のパフォーマンスを測定しました。



結果は次のようになります。つまり、PHP 7.2からPHP 7.4への移行により、エンドポイントでのパフォーマンスが+ 10％向上し、プリロードによってさらに10％向上します。 

プリロードの場合、結果は接続されたファイルの数と実行可能なロジックの複雑さに大きく依存します。多くのファイルが接続されており、ロジックが単純な場合、プリロードはファイルが少なくロジックが複雑な場合よりも多くなります。

変更-再起動


すべてのプリロードファイルは起動時にのみコンパイルされ、不変としてマークされ、将来再コンパイルされないため、これらのファイルに変更を適用する唯一の方法は、PHP-FPM / Apache /などを再起動（リロードまたは再起動）する

ことです。リロードの場合、PHPは再試行します可能な限り正確に再起動します。ユーザー要求は中断されませんが、それでも、プリロードフェーズの進行中は、すべての新しい要求が完了するまで待機します。プリロードにコードが多くない場合、これは問題を引き起こさない可能性がありますが、アプリケーション全体をダウンロードしようとすると、再起動中の応答時間が大幅に増加します。

また、再起動（リロードでも再起動でも）には重要な機能があります。このアクションの結果として、OPcacheがクリアされます。つまり、それ以降のすべての要求はコールドオペコードキャッシュで動作し、応答時間がさらに長くなる可能性があります。

未定義の文字


プリロードでクラスをロードするには、依存するすべてのものをこの時点までに定義する必要があります。以下のクラスの場合、これは、このクラスをコンパイルする前に、他のすべてのクラス（B、C、D、E、F、G）、変数$someGlobalVar、および定数SOME_CONST が使用可能でなければならないことを意味します。プリロードスクリプトは通常のPHPコードなので、オートローダーを定義できます。この場合、他のクラスに接続されているすべてのものが自動的にロードされます。しかし、これは変数と定数では機能しません。私たち自身が、このクラスが宣言されたときにそれらが定義されていることを確認する必要があります。

class A extends \B implements \C {
    use \D;

    const SOME_CONST = \E::E1;
    private static $someVar = \F::F1;

    private $anotherVar = \G::G1;
    private $varLink = $someGlobalVar;
    private $constLink = SOME_CONST;
}

さいわい、プリロードには、何かが邪魔にならないかどうかを理解するのに十分なツールが含まれています。まず、これらはロードに失敗したものとその理由に関する情報を含む警告メッセージです。

クラスフィールド/定数の最適化


上記で書いたように、プリロードはクラスのフィールド/定数の値を解決して保存します。これにより、コードを最適化できます。要求の処理中、データは準備ができており、他のデータから派生する必要はありません。しかし、これは明白ではない結果をもたらす可能性があり、次の例がそれを示しています。

const.php:
<?php
define('MYTESTCONST', mt_rand(1, 1000));

preload.php:
<?php

include 'const.php';
class MyTestClass {
    const RAND = MYTESTCONST;
}

script.php:
<?php

include 'const.php';
echo MYTESTCONST, ', ', MyTestClass::RAND;
// 32, 154

結果は直観に反する状況です。定数の1つに他の値が割り当てられているため、定数は等しいように見えますが、実際にはそうではありません。これは、クラス定数/フィールドが解決および保存されている間、クラス定数/フィールドとは対照的に、プリロードフェーズの終了後にグローバル定数が強制的にクリアされるためです。これにより、クエリの実行中にグローバル定数を再定義する必要があり、その結果として別の値を取得する可能性があります。

someFunc（）を再宣言できません


クラスの場合、状況は簡単です。通常、明示的に接続するのではなく、オートローダーを使用します。これは、クラスがプリロードフェーズで定義されている場合、リクエスト中にオートローダーが実行されず、このクラスに2回接続しようとしないことを意味します。 

状況は関数によって異なります。明示的に接続する必要があります。これにより、必要なすべてのファイルをプリロードスクリプト内の関数に接続し、リクエスト中にもう一度これを試行します（典型的な例はComposerブートローダーです。常にすべてのファイルを関数に接続しようとします）。この場合、エラーが発生します。関数はすでに定義されており、再定義できません。 

この問題を解決する方法はいくつかあります。たとえば、Composerの場合、プリロードフェーズですべてを接続でき、リクエスト中はComposerに関連するものは何も接続しません。別の解決策は、ファイルを関数に直接接続するのではなく、たとえばGuzzle HTTP が行うように、function_exists（）をチェックしてプロキシファイルを介して接続することです。

ロードランナー


RoadRunnerはGoで記述されたデーモンであり、一方ではPHPワーカーを作成して監視し（必要に応じて開始/終了/再起動）、他方ではリクエストを受け入れてこれらのワーカーに渡します。この意味で、その動作はPHP-FPMの動作（ワーカーを監視するマスタープロセスもある）と同じです。しかし、まだ違いがあります。重要なのは、RoadRunnerがクエリの完了後にスクリプトの状態をリセットしないことです。 

したがって、「クラシック」PHPの場合にどのリソースが費やされるかのリストを思い出すと、RoadRunnerはすべてのポイントに対処できます（リロードは最初のポイントのみです）。

  • ファイル接続（include、requireなど）。
  • 初期化（フレームワーク、ライブラリ、DIコンテナなど）。
  • （メモリに保存するのではなく）外部ストレージからデータを要求します。

Hello World RoadRunnerの例は次のようになります。

$relay = new Spiral\Goridge\StreamRelay(STDIN, STDOUT);
$psr7 = new Spiral\RoadRunner\PSR7Client(new Spiral\RoadRunner\Worker($relay));

while ($req = $psr7->acceptRequest()) {
        $resp = new \Zend\Diactoros\Response();
        $resp->getBody()->write("hello world");
        $psr7->respond($resp);
}

プリロードでテストした現在のエンドポイントを試し、変更せずにRoadRunnerで実行し、ロードしてパフォーマンスを測定します。変更なし-それ以外の場合、ベンチマークは完全に正直ではありません。

Hello Worldの例をこれに合わせてみましょう。

まず、上で書いたように、エラーが発生したときにワーカーがクラッシュしないようにします。これを行うには、すべてをグローバルなtry..catchでラップする必要があります。次に、スクリプトはZend Diactorosについて何も認識していないため、答えを得るには結果を変換する必要があります。これにはob_-functionsを使用します。3番目に、スクリプトはPSR-7要求の性質について何も知りません。解決策は、これらのエンティティーから標準のPHP環境にデータを取り込むことです。そして4番目に、スクリプトは要求が終了し、状態全体がクリアされることを期待しています。したがって、RoadRunnerでは、これを自分で行う必要があります。

したがって、最初のHello Worldバージョンは次のようになります。

その主な部分はfread（）の呼び出しにあり、ほとんど何もすることがありません。しかし、\ Spiral \ RoadRunner \ PSR7Client :: acceptRequest（）に他のブランチがいくつかあります恐怖自体を除いて。あなたはソースコードを見ることによってそれらの意味を理解することができます：

   /**
     * @return ServerRequestInterface|null
     */
    public function acceptRequest()
    {
        $rawRequest = $this->httpClient->acceptRequest();
        if ($rawRequest === null) {
            return null;
        }

        $_SERVER = $this->configureServer($rawRequest['ctx']);

        $request = $this->requestFactory->createServerRequest(
            $rawRequest['ctx']['method'],
            $rawRequest['ctx']['uri'],
            $_SERVER

            ->withQueryParams($query)

RoadRunnerがシリアル化された配列を使用してPSR-7準拠のリクエストオブジェクトを作成しようとしていることが明らかになります。フレームワークがPSR-7クエリオブジェクトを直接操作する場合（たとえば、Symfony が機能しない場合）、これは完全に正当化されます。その他の場合、PSR-7は、リクエストがアプリケーションで動作できるものに変換される前に、追加のリンクになります。この中間リンクを削除して結果をもう一度見てみましょう

RoadRunnerのニュアンス


一般に、RoadRunnerの使用は、単にプリロードを有効にするよりも深刻な変更であるため、ここでのニュアンスはさらに重要です。

まず、RoadRunnerは実際、PHPコードをデーモンモードで実行します。これは、この記事の冒頭で書いたすべての問題の影響を受けることを意味します。発生する可能性のある新しいクラスのエラーが表示され、コードの記述とデバッグがより困難になります。 。

次に、RoadRunnerを最大限に利用したい場合は、「クラシック」コードを実行するだけでは不十分です。まず、そのためのコードを記述する必要があります。この場合、RoadRunnerとアプリケーションフォーマット間のリクエスト/レスポンス間隔によるハッキングを回避します。おそらく、リクエストの最後にクリーニングする必要がないように、すぐにコードを記述し、リクエスト間で共有メモリのより大きな容量を使用することもできます。

3番目に、すべての結果は特定のエンドポイントで得られたものであり、RoadRunnerで実行するのに最適な候補とは限りません。他のスクリプトでは、完全に異なる結果が可能です。

結論


そこで、「クラシック」PHPのアーキテクチャーを調べ、プリロードがそれをどのように支援できるか、RoadRunnerアーキテクチャーがそれとどのように異なるかを理解しました。

「クラシック」なユースケースのPHP（PHP-FPM、Apache mod_phpなど）は、開発を簡素化し、多くの問題を回避するのに役立ちます。バックエンドに特別なパフォーマンス要件がない場合、これは検討されている方法の中で最も効果的な開発方法です。さらに、このソリューションから最大のパフォーマンスを引き出すために多くの方法が考案されており、プリロードはすでに追加されており、JITはまもなく追加されます。

バックエンドが読み込まれることを最初に理解している場合は、すぐにRoadRunnerに目を向けることが理にかなっています。

もう一度、私は私たちの実験の過程で得られた結果を与えます（繰り返します：高い確率で、成長はアプリケーションによって異なります）：

  • PHP 7.2-845 RPS;
  • PHP 7.4-931 RPS;
  • 最適化なしのRoadRunner-987 RPS; 
  • PHP 7.4 +プリロード-1030 RPS;
  • 最適化後のRoadRunner-1089 RPS。 

Badooは、PHP 7.4に切り替えることを決定し、公式リリースの直後（十分に安定した時点）に切り替えるようにコードを調整しています。 

RoadRunnerの効果的なアプリケーションはまだ見つかっていませんが、どのように機能し、何を期待できるかがわかったので、必要に応じて将来的に適用することができます。
