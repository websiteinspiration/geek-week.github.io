Olya，测试和工厂-通往美丽架构和简洁代码的道路

在测试自动化中，有许多不同的方法，解决方案和方法。最常见和必不可少的是页面对象（屏幕对象）模式。使用这种模式时，我遇到了两种方法：为我的页面对象创建工厂，而没有它。在本文中，以自动测试为例，我们比较了这两种方法的优缺点。让我向您展示我们的页面对象工厂是什么样的。我还将讨论在工厂进行自动测试时遇到的问题以及如何解决这些问题。

我将在Swift中引用本文中的所有示例，但对于Android自动测试而言，所有操作均以相同的方式进行。


开始


嗨，我叫Olya。我是hh.ru的移动应用程序测试人员。连续两年，我们将90％的手动测试转移到了自动化。在这段时间里，我们设法多次遇到各种陷阱，争论了数百场纠纷，现在我想与世界分享我们的经验。

我们在android（Kotlin，Kaspresso）和iOS（Swift，XCUITest）上编写自动测试。我们尝试使UI测试变小，仅检查单个脚本。铁的好处就足够了，每个平台上约300-400个UI自动测试的回归平均需要30-40分钟。

在自动测试中，我们使用Page Object模式。关于他的文章已有数千篇，因此我们将不作更详细的介绍。 

从历史上看，当在iOS和Android上创建第一个自动测试时，我们选择了不同的方法来处理页面对象。在Android上，我们只是遵循这种模式，在iOS上，我们还创建了一个页面对象工厂。工厂是初始化所有页面对象的地方。使用它，在屏幕的方法中，我们可以传输其他页面对象，从而建立类似于应用程序行为的交互链。

工厂是否需要页面对象？ 


在创建了数百个测试用例之后，我们得出的结论是，使用页面对象的工厂对于编写自动测试的测试人员来说是一个问题。使用不同的方法时，性能和稳定性没有区别。但是为了避免将来遭受本机代码的困扰，您需要尽快决定是否在自动测试的体系结构中为页面对象的工厂提供场所。

为了清楚起见，我将给出同一测试的两个示例：有工厂编写和没有工厂编写。 

该测试的作用：在应用程序的主屏幕上，用户尝试转到配置文件选项卡。在配置文件中，他选择“登录”，然后选择登录方式并输入他的用户名和密码，然后单击“登录”按钮。

//   
class ExampleLoginTestSuit: BaseTestCase {
   let mainScreen = MainScreen()
   let profileScreen = ProfileScreen()
   let authorizationScreen = AuthorizationScreen()
   let loginScreen = LoginScreen()
 
   func testExampleLogin() {
        let user = userFixtureService.createUser()
 
        mainScreen.openProfileTab()
        profileScreen.goToAuthorization()
        authorizationScreen.goToLogin()
        loginScreen
           .enterLogin(user.login)
           .enterPassword(user.password)
           .logIn()   
   }
}


//   
class ExampleLoginTestSuit: BaseTestCase {
 
   func testExampleLogin() {
        let user = userFixtureService.createUser()
 
        pageObjectsFactory
            .makeMainScreenPageObject()
            .openProfileTab()
            .goToAuthorization()
            .goToLogin()
            .enterLogin(user.login)
            .enterPassword(user.password)
            .logIn()
    }
}

如果在此示例之后，您认为没有工厂的测试看起来不错，只需抓住并编写没有它的测试即可。如果您没有非常大的应用程序，并且具有不同的非重复内容和元素，那么此技巧特别有用。或者，如果您编写的测试不超过10-15个步骤。在这种情况下，工厂将无法正常工作。 

但是，如果不是这种情况，或者您的内心对工厂测试有所反应，那么本文适合您。 


我们的工厂是什么样的？


让我们弄清楚页面对象工厂的用途，外观和功能。

标准测试用例是一组步骤：
  1. 启动应用程序
  2. 打开配置文件屏幕
  3. 点击“登录”
  4. 使用登录名和密码选择登录方式
  5. 输入用户数据（用户名和密码）
  6. 点击“登录”按钮

截屏测试


测试用例的每个步骤都对应于自动测试中的一个步骤。您也可以说每个步骤都是相应屏幕的单独页面对象方法。 

如果我们想在步骤中准确地在自动测试中重复此测试用例，构建相同的不间断的方法调用链，我们需要每个方法返回下一个屏幕的页面对象。

创建页面对象时，需要对其进行初始化。事实证明，如果我们要在其他内部创建页面对象，则每个页面对象中将有许多相同的初始化。为了不执行此操作，所有页面对象的初始化都移至工厂-pageObjectFactory（或screenFactory等）。在每种方法的最后，我们要求工厂创建所需的屏幕。

final class ProfilePageObject: BasePageObject {
 
    func goToAuthScreen() -> AuthPageObject {
        openAuthButton.tap()
        return pageObjectsFactory
            .makeAuthPageObject()
    }
}

所有页面对象均从BasePageObject继承，BasePageObject包含主要参数。在其中，我们必须规定一个必需的构造函数，以便工厂可以创建其任何继承人。在我们的例子中，基类如下所示：

class BasePageObject {
    let pageObjectsFactory: PageObjectsFactory
    let application: XCUIApplication
 
    required init(pageObjectsFactory: PageObjectsFactory,
                       application: XCUIApplication) {
        self.pageObjectsFactory = pageObjectsFactory
        self.application = application
    }
}

结果，工厂中屏幕的初始化如下所示：

final class PageObjectsFactory {
 
    private func initializePageObject<PageObject: BasePageObject>(ofType type: PageObject.Type) -> PageObject {
        return type.init(pageObjectsFactory: self, application: application)
    }
 
    func makeAuthPageObject() -> AuthPageObject {
        return initializePageObject(ofType: AuthPageObject.self)
    }
}

结果，在任何页面对象的方法中，我们都可以调用创建用于初始化它的屏幕的工厂方法。

没有工厂的测试仪的使用寿命


1.从测试中始终可以清楚地知道在哪个屏幕上执行了操作


如果没有工厂，则仅在测试点的每一行中都会清楚地记录屏幕和在其上进行的操作。这样一来，您就无需从一开始就读取整个测试代码，例如，在调试和修复期间。缺少工厂会增加可读性，尤其是在一个屏幕上执行操作或检查时。

2.编写带有可能导致不同屏幕的操作的测试更加方便


在没有工厂的情况下编写测试时，您无需考虑以下事实：根据应用程序的状态，任何用户操作的行为都可能不同，您只需描述正在发生的逻辑：

//        
    vacancyScreen.tapResponseButton()
    authScreen.authUser(user)
    responseScreen.checkScreenIsOpened()

要么

//       	
    vacancyScreen.tapResponseButton()
    responseScreen.checkScreenIsOpened()

对于工厂页面对象，您必须考虑根据用户的授权如何在tapResponseButton（）方法中返回所需的屏幕（或复制诸如tapResponseButtonAndLogin（）之类的方法）

final class VacancyPageObject: BasePageObject {
    …
 
    func tapResponseButton() -> ResponseSendPageObject {
        responseButton.tap()
        return pageObjectsFactory
            .makeResponseSendPageObject()
    }
 
    func tapResponseButtonAndLogin(login: String) -> ResponseSendPageObject {
        responseButton.tap()
        pageObjectsFactory
            .makeAuthorizationPageObject()
            .goToLogin()
            .logIn(login)
        return pageObjectsFactory
            .makeResponseSendPageObject()
    }
}

尽管他们都点击了同一按钮，但我们在此屏幕上有6种相似的方法。

3.无需将每个操作和每个检查都包装在单独的方法中


没有工厂的页面对象的方法不需要分别返回下一个屏幕来继续链，对于单个操作，您不能创建单独的方法。 

例如，在没有工厂的测试中点击按钮可能看起来像这样：

    vacancyScreen.responseButton.tap()


4.不必考虑复杂的解决方案，还可以考虑页面对象的体系结构


以我的经验，在没有工厂的情况下编写自动测试时，页面对象几乎没有问题。描述一个新的页面对象是一个简单而快速的过程。

工厂及其功能增加了一些困难，我将在后面讨论。

工厂无疑的优势


1.编写测试时，无法跳过步骤


页面对象的所有方法都是一链的链接。每个新的链接（方法）应与以下内容互锁。这是因为所有方法都返回一个页面对象（它们自己或另一个），这就是为什么在编写测试时我们无法选择任何屏幕方法的原因，只有以下一种方法。 

首先，它极大地加快了由新测试人员编写第一次自动测试的速度。 IDE本身会告诉您可以进一步执行哪些操作以及在哪个屏幕上进行操作。

其次，这种自动测试的体系结构使您可以通过编写来识别应用程序，反之亦然。 （再次-是年轻测试人员的良好动力）。

第三，没有方法可以跳过步骤，因为所需的方法只是在正确传递脚本之后才会出现。在没有工厂的情况下编写测试时，这种遗漏是很常见的，并且只有在运行测试时才能发现它们。

2.如果更改页面对象方法的签名，IDE将强制您更改与该对象关联的所有测试


该项目接自前一项。所有页面对象方法都重复应用程序的逻辑和行为。如果应用程序更改了屏幕之间的转换逻辑或添加了新的屏幕，我们将更正所需的方法并更改其返回值。在工厂自测中，为了找到受此更改影响的所有测试，您不必运行所有测试。IDE本身将指向“链”断裂的所有位置。没有工厂的测试不会遵循此规定。

3.架构和代码的清洁度不会因创建多余的屏幕对象而受到影响 


如果您不使用工厂，并且测试通过脚本通过了数十个屏幕，那么在测试的类开始时，您将必须编写一个整个块来创建每个屏幕的对象。而且您需要每次针对每类测试执行此操作。如果在同一个类中编写多个测试，那就更糟了。然后，随着所有页面对象的创建，此“块”的大小将显着增加。  

真实测试中的一个示例，其中用户响应空缺，注销并登录到另一个用户。对于所有类似的测试，此块随类的不同而变化。

有一个选项可以使用延迟初始化对测试的基类中的所有页面对象进行初始化。然后，在每个测试中，所有屏幕均可用，并且不会创建不必要的对象。但是，如果我们需要编写将通过另一个屏幕的方法，则当我们想在页面对象本身中创建页面对象时，列出许多屏幕的问题将再次出现。

工厂需要在需要时初始化必要的页面对象。

4.一个很好的机会，可以更深入地研究应用程序代码，研究其体系结构，模块，它们之间的交互等。


拥有页面对象工厂需要将所有操作与屏幕包装在一起，该方法将返回下一个屏幕。因此，页面对象正在大大扩展，这使您想知道如何使所有内容更加美观和准确。为此，您正在尝试使用体系结构技术，观察应用程序代码中各个屏幕的实现，跟踪模块的交互，以便在自动测试中构建类似的系统，这将使使用愉快且方便。

另外，我们从内部获得有关应用程序工作方式的知识。至少，这对于整体开发很有用，并且在测试时通常会很有帮助。

5.测试代码看起来非常整洁。


没有任何评论，这真的很美味。关于您如何“更准确，更清洁”的决定可以通过第一个示例来决定。

陷阱//他们如何跌跌撞撞地走



我们可以得出结论，编写自动测试时最好使用页面对象的工厂。没人问为什么他们使用工厂，总会有一个答案：“而您尝试编写没有它的测试并进行比较。” 工厂实际上承担了很大一部分任务，负责自动测试顺序中没有错误，等等。这也打开了我上面写过的一些有趣的机会。 

但我想指出的是，使用这些功能并非总是事事如意。在工厂自动化过程中，我们遇到了非常不愉快的问题，但最终我们解决了这些问题。

整个应用程序共有的协议和元素


任何移动应用程序中都有可以从任何屏幕访问的元素。例如，您可以使用一个标签栏（菜单）。问题出现了：如何在不中断测试页面对象方法的调用链的情况下，在测试中的任何时间访问选项卡式方法？ 

最明显也是我们最初的决定是使用这些方法扩展页面对象的基类。

import XCTest
 
/*
      .
 */
extension BasePageObject {
 
    private lazy var tabBar = application.tabBars[Accessibility.TabBar.identifier].firstMatch
    var searchTab =  tabBar.buttons[Accessibility.TabBar.searchTab].firstMatch
    …
 
    func openSearchTab() -> MainScreenPageObject {
        searchTab.tap()
        return pageObjectsFactory.makeMainScreenPageObject()
    }
    …
}

在讨论过程中，我们意识到标签栏并不是所有屏幕上唯一需要的元素，并得出结论，通过添加越来越多的扩展，我们将迅速使基类变得混乱。

该解决方案的另一个缺点是该方法可用于所有页面对象，这是错误的。这将破坏后继类的契约，从理论上讲，后者仅应包含特定于他们的方法。例如，对于页面警报对象，不需要选项卡方法。 

我们的最终解决方案-我们通过Tabbar的页面对象（与Kotlin的界面类似）制作了TabBarUsable协议。他们编写了扩展名（扩展名，实现），该扩展名允许您不重复代码，同时用组合替换继承。

import XCTest
 
protocol TabBarUsable {
    var searchTab: XCUIElement { get }
 
    func openSearchTab() -> MainScreenPageObject
    …
}

extension TabBarUsable where Self: BasePageObject {
 
    private var tabBar: XCUIElement { application.tabBars[Accessibility.TabBar.identifier].firstMatch }
    var searchTab: XCUIElement { tabBar.buttons[Accessibility.TabBar.searchTab].firstMatch }
 
    func openSearchTab() -> MainScreenPageObject {
        searchTab.tap()
        return pageObjectsFactory.makeMainScreenPageObject()
    }
    …
}

对于所有带有标签栏的屏幕，我们都增加了对该协议的遵守（符合）。

final class VacancyPageObject: BasePageObject, TabBarUsable {
    …
}

因此，在所有这些屏幕上都可以使用使用标签栏的所有方法。 

class ExampleTabbarTestSuit: BaseTestCase {
 
    func testExampleOpenSearchTab() {
        pageObjectsFactory
            .makeMainScreenPageObject()
            .openVacanciesList()
            .openVacancy() //   .makeVacancyPageObject()
            .openSearchTab() //   TabBarUsable 
    }
}

同时，架构不会中断，标签栏的方法仅适用于我们需要的屏幕，并且相同方法的代码不可重复。

协议和代码修订


如前所述，使用页面对象工厂时，所有操作和所有检查都包装在方法中。一段时间后，创建下一个自动测试，您开始注意到您为每个页面对象编写了相同的方法，甚至在元素上也没有什么不同。此类方法的示例可以是零屏幕检查，在不同屏幕上使用相同的列表项等等。


有一个合理的想法：“我会让我更轻松吗？！”。如何停止重复代码？

我已经写过的第一个方法是将所有这些方法放入基类。可能，但是要非常小心。首先，这个过程非常有趣。看来基类中的一种方法不会破坏任何东西，但是基类迟早会变成无法忍受的，没有结构的，杂乱无章的怪物，总之，它变得完全不可行。我们尝试过，我们知道。摆脱这个怪物比立即正确做起来更加困难。

随着时间的流逝，我们意识到DRY（不要重复自己）原理是有原因的。我们开始寻找在许多页面对象中使用的相同方法。找到它们之后，他们讨论了很长时间，这些方法是否真的在所有地方都使用了相同的逻辑。最后，他们决定，如果在80％的情况下以相同的方式使用这些方法，则应在单独的协议中将其删除。 

最早呈现的方法之一是waitView（）方法。我们在协议扩展中描述了特殊的ViewWaitable协议及其实现，现在，为了使此方法在页面对象中可用，您只需添加对协议的一致性（conform）。由于所有屏幕的视图标识符都不相同，因此使用该协议的所有页面对象都必须自己声明一个视图。

protocol ViewWaitable {
    var view: XCUIElement { get }
}

extension ViewWaitable where Self: BasePageObject {
 
    @discardableResult
    func waitView() -> Self {
        testWaiter.waitForElementToAppear(view)
        return self
    }
}


final class VacancyPageObject: BasePageObject, TabBarUsable, ViewWaitable {
 
    lazy var view = application.otherElements[Accessibility.view].firstMatch
    …
}

hh.ru应用程序中的另一个示例：我们有一个空缺列表，可以在许多不同的屏幕上找到它们，唯一的区别是显示此列表的视图标识符。此列表关联了许多操作和大量检查。将它们乘以约10（按此列表显示的屏幕数），以估算代码重复的规模。 

解决方案还是协议。为了使协议更加易于理解和使用，我们将列表元素（VacancyListContainig），单元格元素（VacancyCellContainig），带有检查的方法（声明，检查）和交互方法（操作）进行了划分。这种分离完美地解决了大页面对象内容可读性的问题。最终的架构如下所示：


空缺列表的页面对象可以增加对VacancyListPageObject协议的符合性。因此，实现该协议的所有方法无需重复代码即可在屏幕上使用。

结果，由于遵循DRY原理，我们拥有非常整洁的页面对象，相同的方法名称以及缺少额外的代码。 

相同的警报在不同的屏幕上//源


系统错误，警报，底页是回归分析的组成部分，我当然想在自动测试中进行介绍。错误和警报发生在不同的屏幕上，在不同的情况下，在不同的应用程序状态下也不同，但是元素本身基本上具有相同的标识符和行为。显然，具有两个按钮的相同类型的警报（例如，确定/取消）=一个单独的页面对象。  

我们回到了不中断方法调用链的愿望。我们有-几十个屏幕，我们可以从中打开同一警报。关闭此类警报时，我们必须返回从其打开的页面对象。同时，我们不想在这样的警报的页面对象中编写同样的数十种重复方法，这些方法只会在返回的页面对象中彼此不同。

最简单的解决方案是对测试进行一次评分和打破。这也没有错。因此，生活要简单得多。

    }
}

我们的解决方案更加复杂，但是它有多美！我们添加了警报类型参数（通用）的页面对象，它将是我们从中打开警报的页面对象的类型。在这种情况下，此屏幕的页面对象将在初始化时传输到页面对象警报，并作为通用源变量存储。 

有关其工作原理的更多信息：当您打开（并初始化）警报时，我们在源通用变量中传递所需的屏幕。这是通过页面对象方法完成的，该方法将打开警报：

final class VacancyPageObject: BasePageObject {
    …
 
    func openAlert() -> AlertPageObject<VacancyPageObject> {
        button.tap()
        return pageObjectsFactory
            .makeAlertPageObject(from: self) //  source VacancyPageObject
    }
}

接下来，此源（在本示例中为VacancyPageObject）通过测试所需的所有方法，并最终到达最终的dismissAlert（）。结果，警报关闭方法返回了我们需要的屏幕，我们可以在该屏幕上立即继续进行测试。

final class AlertPageObject<Source: BasePageObject>: BasePageObject {
 
    func dismissAlert() -> Source {
        cancelButton.tap()
        return source //  ,     — VacancyPageObject
    }
}

就像我说的那样，为了使页面对象可以接收和返回必要的屏幕（源），它需要分配一个通用类型，该类型仅限于页面对象的基本类型，并使用页面对象对其进行初始化，这将需要在最后返回：

final class AlertPageObject<Source: BasePageObject>: BasePageObject {
 
    // generic-          
    private let source: Source
 
    init(pageObjectsFactory: PageObjectsFactory,
         application: XCUIApplication,
         source: Source) {
        self.source = source
        super.init(pageObjectsFactory: pageObjectsFactory, application: application)
    }
 
    //  BasePageObject   ,      
    required init(pageObjectsFactory: PageObjectsFactory, application: XCUIApplication) {
        fatalError("init(pageObjectsFactory:application:) has not been implemented. Use another init")
    }
}

final class PageObjectsFactory {
    …
    func makeAlertPageObject<Source: BasePageObject>(
        from source: Source
    ) -> AlertPageObject<Source> {
        return AlertPageObject(pageObjectsFactory: self,
                                          application: application,
                                          source: source)
    }
}

巨大的页面对象，如何理解


有屏幕（尽管有屏幕），其中包含许多不同的内容，各种界面元素和各个逻辑块。同时，它们都在同一屏幕上，我想通过描述该屏幕的页面对象与它们进行交互。 

例如，以我们的移动应用程序为主屏幕。


该屏幕可以分为三个完全独立的部分：搜索栏，搜索历史记录块和带有工作清单的标签。可以在一个页面对象e中描述所有内容（毕竟，这是一个单独的完整屏幕），但是使用它并不是很方便，因为每个部分都有自己的逻辑，自己的检查等。

同样，在扩展页面对象时，迟早会出现方法名称的问题。结果，有许多不同的单元格，标题，字幕等，因此，除了简单的名称外，还必须给出长名称，并阐明它们的确切含义。编写测试变得非常不便和困难。每次在测试中选择一种方法时，都需要打开页面目标代码并仔细研究，以免出错。

final class VacancyListPageObject: BasePageObject {
    …
 
    func assertHistoryListFirstCellTitleExists() -> Self { ...  }
    func assertHistoryListCellByTitleExists(title: String) -> Self { ...  }
    func assertRecommendationsListFirstCellTitleExists() -> Self { ...  }
    func assertVacancyNearbyListFirstCellTitleExists() -> Self { ...  }
    …  
}


最简单的解决方案是不看问题。无论页面对象有多大，您仍然可以使用它（尤其是如果您习惯了它）。但我不想习惯它，但我希望它既美观又舒适。

另外，这些“部分”可以是代码中的不同模块，这意味着它们可以在应用程序的不同屏幕上重复使用。如果我们使用页面对象构建类似的体系结构，我们也可以重用它们。

我们做出了一个坚定的决定-现在针对此类部分，我们正在编写不同类的页面对象。对于它们每个，我们向统一的空MainScreenSection协议添加对应关系：BasePageObject {}。

例如，在主屏幕上选择“搜索历史记录”部分。其页面对象将如下所示：

final class SearchHistoryPageObject: BasePageObject, MainScreenSection {
    // 
    // 
}

基本上，描述了主屏幕的MainScreenPageObject的页面对象，我们创建了一种方法，通过该方法可以从测试中说出现在要在哪个部分进行操作。看起来像这样。 

final class MainScreenPageObject: BasePageObject, TabBarUsable {
    …
    func section<Section: MainScreenSection>(_ section: Section.Type) -> Section {
        return section.init(pageObjectsFactory: pageObjectsFactory, application: application)
    }
}

在此方法中，我们传递节的类型，在内部使用通用工厂方法初始化页面对象。 

结果，我们得到了不间断的测试，没有了对大页面对象的不必要的初始化，以及代码的可读性。

总结


您可以在没有工厂页面对象的情况下编写自动测试。使用这种方法的页面对象可以非常快速，简单地编写，尤其是在不需要复杂的初始化的情况下。但是，另一方面，在编写测试时，您需要格外小心。测试人员负责测试中的步骤顺序，页面对象的初始化，缺少对某些操作的包装等。如果您的应用程序不是很复杂，则可能不值得复杂化，并且列出的所有问题通常都是不可见的。

如果从一开始就对工厂的创建工作有所了解，那么将来它将首先承担上述所有责任，其次它将使编写自动测试的过程非常简单，几乎没有出错的机会。


由于没有单一正确的决定，因此您需要自行决定。我们继续使用这两种方法，每个人都感觉很好。祝你好运，美丽的自动测试！
