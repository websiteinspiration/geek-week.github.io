大規模なPythonプロジェクトでの厳密なモジュールの使用：Instagramの経験。パート1

InstagramとPythonの連携に関するシリーズの次の記事の翻訳の最初の部分を公開します。このシリーズの最初の記事では、Instagramサーバーコードの機能、頻繁に変更されるモノリス、および静的型チェックツールがこのモノリスの管理にどのように役立つかについて説明しました。2番目の資料は、HTTP APIの入力に関するものです。ここでは、InstagramがプロジェクトでPythonを使用して発生した問題のいくつかを解決するためのアプローチについて説明します。資料の作成者は、Instagramの経験が同様の問題に遭遇する可能性のある人々に役立つことを期待しています。



状況の概要


次のモジュールを見てみましょう。一見すると、完全に無害に見えます。

import re
from mywebframework import db, route
VALID_NAME_RE = re.compile("^[a-zA-Z0-9]+$")
@route('/')
def home():
    return "Hello World!"
class Person(db.Model):
    name: str

誰かがこのモジュールをインポートすると、どのコードが実行されますか？

  • まず、正規表現に関連付けられたコードが実行され、文字列がテンプレートオブジェクトにコンパイルされます。
  • 次に、デコレータが実行され@routeます。表示内容に依存している場合、ここではおそらく、対応する表現がURLマッピングシステムに登録されていると想定できます。これは、このモジュールの通常のインポートにより、アプリケーションのグローバル状態がどこか他の場所で変化しているという事実につながることを意味します。
  • 次に、すべてのクラス本体コードを実行しますPerson。何でも入れることができます。基本クラスにModelはメタクラスまたはメソッド__init_subclass__が含まれる場合があり、これにはモジュールのインポート時に実行される他のコードが含まれる場合があります。

問題＃1：サーバーの起動と再起動が遅い


このモジュールのインポート時に（おそらく）実行されないコード行はthisだけreturn "Hello World!"です。確かに、これを確実に言うことはできません！その結果、8行で構成されるこの単純なモジュールをインポートすることで（プログラムで使用しない場合でも）、数百行または数千行のPythonコードが起動されることがあります。そして、これはこのモジュールのインポートがプログラムの他の場所にあるグローバルURLマッピングの変更を引き起こすことは言うまでもありません。

何をすべきか？私たちの前は、Pythonが動的に解釈される言語であるという事実の当然の一部です。これにより、メタプログラミング手法を使用してさまざまな問題を正常に解決できます。しかし、それでも、このコードの何が問題になっていますか？

実際、このコードは完全な順序になっています。これは、小規模なプログラマチームが作業する比較的小さなコードベースで誰かがそれを使用する限りです。このコードは、それを使用する人が、Python機能がどのように使用されるかについて一定レベルの規律を維持することが保証されている限り、問題を引き起こしません。しかし、何百人ものプログラマが取り組んでいるプロジェクトに数百万行のコードがあり、その多くがPythonに関する深い知識を持っていない場合、このダイナミズムのいくつかの側面が問題になる可能性があります。

たとえば、Pythonの優れた機能の1つは、段階的開発に必要なステップの速度です。つまり、コードを変更する結果は、コードをコンパイルする必要なしに、そのような変更を行った直後に文字通り見ることができます。しかし、数百万行のプロジェクト（およびこのプロジェクトのややわかりにくい依存関係図）について話している場合、このプラスのPythonはマイナスに変わり始めます。

サーバーの起動に20秒以上かかります。また、最適化に十分な注意を払わないと、この時間が約1分に増えることがあります。これは、開発者がコードに加えられた変更の結果を確認するのに20〜60秒必要であることを意味します。これは、ブラウザーに表示される内容、および単体テストの実行速度にも当てはまります。残念ながら、この時間は人が何かに気を取られて、彼が以前にしたことを忘れるのに十分です。この時間のほとんどは、文字通り、モジュールのインポートと、関数とクラスの作成に費やされています。

ある意味、他の言語で書かれたプログラムをコンパイルした結果を待つのと同じです。しかし、通常、コンパイルは段階的に行うことができます。ポイントは、変更されたものと、変更されたコードに直接依存するものだけを再コンパイルできることです。その結果、通常、プロジェクトに小さな変更を加えた後に実行されるプロジェクトのコンパイルは迅速です。ただし、Pythonで作業する場合、インポートコマンドには副作用がある可能性があるため、サーバーを徐々に再起動する信頼性の高い安全な方法はありません。同時に、変更の規模は重要ではなく、サーバーを完全に再起動して、すべてのモジュールをインポートし、すべてのクラスと関数を再作成し、すべての正規表現を再コンパイルする必要があります。通常、最後にサーバーを再起動した瞬間から、コードの99％は変更されませんが、変更を入力するには、同じことを何度も繰り返す必要があります。

これにより、開発者の速度が低下するだけでなく、大量のシステムリソースが非生産的に浪費されます。事実は、変更を継続的にデプロイするモードで作業しているということです。これは、本番サーバーのコードを常にリロードすることを意味します。

実際、これが最初の問題です。サーバーの起動と再起動が遅いです。この問題は、コードのインポート中にシステムが常に大量の反復アクションを実行する必要があるために発生します。

問題＃2：安全でないインポートコマンドの副作用


判明したように、モジュールをインポートするときに開発者がしばしば解決する別のタスクがあります。これは、構成のネットワークストレージから設定を読み込んでいます。

MY_CONFIG = get_config_from_network_service()

サーバーの起動が遅くなるだけでなく、安全ではありません。ネットワークサービスが利用できない場合、これにより、一部の要求を実行できないことに関するエラーメッセージが表示されるだけではありません。これにより、サーバーの起動に失敗します。

色を濃くして、重要なネットワークサービスの初期化を担当するモジュールに、誰かがインポート中に実行されるコードを追加したと想像してみましょう。開発者はこのコードをどこに追加すればよいかわからなかったため、サーバーの起動の初期段階でインポートされるモジュールにコードを配置しました。このスキームが機能することが判明したため、ソリューションは成功したと見なされ、作業が続行されました。

しかし、他の誰かがインポートチームをどこかに追加しましたが、一見すると無害でした。その結果、12個のモジュールの深さを持つインポートのチェーンを通じて、ネットワークから設定をロードするモジュールが、対応するネットワークサービスを初期化するモジュールにインポートされるようになりました。

これで、サービスを初期化する前に使用しようとしていることがわかりました。システムは自然にクラッシュします。最良のケースでは、相互作用が完全に決定論的であるシステムについて話している場合、これは、開発者が1〜2時間かけて、マイナーな変更がどのように何かの失敗につながったかを理解することにつながる可能性があります。接続されていないようです。しかし、より複雑な状況では、これによりプロジェクトが本番環境に「落ちる」可能性があります。それを使用する普遍的な方法はありません。そのような問題に対処するか、またはそれらを防ぐためにリンターします。

問題の根本は2つの要因にあり、その相互作用は壊滅的な結果をもたらします。

  1. Pythonでは、インポート中にモジュールに任意の安全でない副作用を持たせることができます。
  2. コードのインポート順序は明示的に設定されておらず、制御されていません。プロジェクトの規模では、一種の「包括的なインポート」は、すべてのモジュールに含まれているインポートコマンドで構成されています。この場合、使用するシステムの入力ポイントによって、モジュールのインポート順が異なる場合があります。
