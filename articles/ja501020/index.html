内部でsetTimeout / setIntervalを含むコードをテストする方法

私たちの開発者はユニットテストをとても気に入っています。その有用性は明らかです。そして、これらのテストが本当に有用であり、痛みをもたらさないようにするには、それらの安定性を確保する必要があります。


当社は、「Wasaby」インターフェースフレームワークを開発し、それに基づくクラウドやデスクトップアプリケーションの製品を販売しています。リリースサイクルはカレンダーと密接に関連しており、製品の品質を管理するために継続的な統合プロセスが設定されています。アセンブリにはJenkinsを使用し、Chaiと組み合わせてMochaを使用しますJavaScriptコードの単体テスト用。そして最近、アセンブリの監視で、落下の全ケースの約半分が不安定なJavaScriptユニットテストが原因であることが示され始めたという状況に直面しました。症状は同じです。セットとは別のテストを実行する時間がないか、期待どおりに間違った結果が返されます。また、ケースの分析では、ほとんどの場合、setTimeout関数またはsetInterval関数の呼び出しを含むテストが、それ自体またはテスト済みのコードでクラッシュするという事実が明らかになります。この状況で何をすべきかについては、さらに話します。


なぜそれが起こるのですか？


すべてが単純です。開発者がテストしているコードの内部では、非同期呼び出しが発生し、最後に環境の状態が何らかの形で変化します。また、このテストは、この状態が期待どおりであることを確認するように設計されています。しかしそれ以来 私たちは理想的な世界に住んでいないので、すでに戦闘で機能していて、もともとこの種のテストのために準備されていなかったコードに対してテストが作成されることがあります。そして、開発者は、そのようなコードをテストする方法を決定する必要があります。


簡単な例を考えてみましょう。テスト中のコードは、インスタンスの状態を非同期で変更するクラスメソッドです。そしてそれは完全に書かれていません：


export class Foo {
  propToTest: boolean = false;
  methodToTest(): void {
    setTimeout(() => {
      this.propToTest = true;
    }, 100);
  }
}
