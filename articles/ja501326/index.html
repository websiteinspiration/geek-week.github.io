なぜこれに悩むのですか？


バージョン管理とはどういう意味ですか？ある種のソフトウェアに取り組んでいて、必要に応じて機能を追加または削除して、常にソフトウェアに変更を加えているとします。多くの場合、最後の変更は基本的な変更であり、以前に作成されたすべてのものを壊します。あなたが古い学校の代表である場合、おそらく最初のファイルにawesome_code.p1という名前を付けます。

次に、変更を開始し、元に戻す必要がある場合に備えて、機能したものを保存する必要があります。したがって、ファイルの名前をawesome_code.12.25.2018.p1に変更します。

同じ日にいくつかの変更を行い、新しいファイルにawesome_code.GOOD.12.25.2018.p1という名前を付けるまで、すべてが正常に機能します。そして、良い仕事を続けてください。

ほとんどの場合、プロフェッショナル環境では、同じコードベースで共同作業を行う複数のチームが存在するため、このモデルにさらに違反します。言うまでもなく、このクレイジーな電車はすぐに脱線してしまいます。

ソースコード管理バージョン管理システム





SCCバージョン管理システムを使用します。これは、複数のチームが共通のコードベースで共同作業できる集中管理された場所にファイルを保存する方法です。このアイデアは新しいものではありません。私がなんとかSCCシステムについて言及したのは、1972年までさかのぼります。したがって、コードを1か所に集中させるべきだという考えは、絶対に時代遅れです。

ただし、生成されたすべての成果物をバージョン管理する必要があるという考えは比較的新しいものです。これは、変更の履歴を追跡、確認、記録して、本番環境に関連するすべてのものをバージョン管理システムに保存する必要があることを意味します。

さらに、「すべての製品アーティファクトはバージョン管理されなければならない」という法律の適用により、「自動化が最初」の原則によって問題にアプローチすることができます。

たとえば、AWS開発環境で複雑な問題をクリックするだけの場合、一時停止して「これはバージョン付けされたアーティファクトをクリックするだけですか」と考えることができます。もちろん、答えは「いいえ」です。

ユーザーインターフェイスを使用してプロトタイプをすばやく作成し、何かが機能しているかどうかを確認するのは通常の習慣ですが、作業は短期間で終了します。 Terraformまたは別のコードとしてのインフラストラクチャツールで覚えておいてください-バージョン管理されたアーティファクトを管理し、Gitリポジトリを使用してそれらを保存します。

Gitリポジトリ


Gitが登場する前は、SVNなどのSCCSを使用するのは面倒で、ユーザーフレンドリーではなく、一般的にかなり苦しい経験でした。 Gitとの違いは、分散バージョン管理を提供することです。簡単に言えば、変更に取り組んでいる間、集中化されたソースコードリポジトリの他のユーザーをブロックするのではなく、コードベースの完全なコピーを操作します。作業が完了すると、このコピーはマスターリポジトリに埋め込まれます。

上記は、これがどのように機能するかを大幅に簡略化したものであることに注意してください。そのような説明はこの記事の目的には十分ですが、Gitの詳細な知識は面倒ですがやりがいのある経験です。



とりあえず、GitはSVNのように機能しないことを覚えておいてください。これは、複数の開発チームが安全かつ確実に共通のコードベースで作業できる分散ソースコード（バージョン）管理システムです。

なぜこれを知る必要があるのですか？


Gitがどのように機能するかを知らなければ、プロのDevOps（クラウド）エンジニアになることはできません。それを勉強するには？ 「Gitチュートリアル」のGoogle検索結果には、通常、非常に肥大化した混乱するチュートリアルへのリンクが含まれています。それにもかかわらず、いくつかの本当に賢明な教科書がそれらの中に出くわします。

このような一連のチュートリアルの1つとして、実際に読んで学習し、実際に使用することをすべての人に推奨しているのが、アトラシアンのGitチュートリアルです。これらはすべて十分ですが、このリポジトリのワークフローを説明する1つのセクションGitワークフローは、世界中のプロのプログラマーによって使用されています。

もう1つの非常に優れたチュートリアルは、Learn Git Branchingリポジトリーのブランチを探索することです。アトラシアンのチュートリアルは「読み取りと学習」の原則に基づいて構築されており、Learn Git Branchingはインタラクティブなチュートリアルです。いずれにせよ、このリポジトリがどのように機能するかを理解していない限り、DevOpsの学習にそれほど遠くまでは行きません。



私は、Git分岐機能のしくみについての理解の欠如、またはGitflowが何であるかを説明できないことで、DevOps開発エンジニアの候補者の候補者の99％が面接中に溺れていることに注意します。これが重要なポイントです。面接に来ても、Terraformやその他の最新のファッショナブルなインフラストラクチャとしてのコードツールを知らなくてもかまいません。これは、プロセスで調査できるため、正常です。しかし、Gitの作業を知らないということは、ソフトウェア開発における最新のベストプラクティスの基本が欠如していることを示しています。DevOpsに関係するかどうかは問題ではありません。これは、あなたの学習曲線があまりにも不均一であることを採用マネージャーに知らせます。

逆に、Gitのベストプラクティスについて自信を持って話せる能力は、最初にソフトウェア開発を行うためのインストールを考え出したことを採用マネージャーに伝えます。これは、まさにあなたが投影したいイメージです。結論：この素晴らしいDevOpsジョブを取得するために、世界をリードするGitエキスパートになる必要はありませんが、Gitに何が起こっているのかについて自信を持って話をするために、しばらくGitを使いこなす必要があります。これを行うには、少なくとも、以下の方法を十分に理解している必要があります。

  • リポジトリのコピーを作成します（リポジトリをフォークします）。
  • Gitブランチを作成する
  • ストリームをリポジトリに、またはその逆に同期します。
  • プルリクエストを作成します。

次は何ですか？


Gitの入門レッスンを学んだら、GitHubで自分のアカウントを取得します。これは最も一般的なオープンソースのGitリポジトリであり、おそらくGitの残りのメンバーと一緒にいたいと思うでしょう。

GitHubにアカウントを作成したら、すぐにコードを入力してください。GitHubでの作業中にコードを記述する必要があるすべてを修正します。これは、バージョンの取り扱いに優れた規律を植え付けるだけでなく、独自のブランドを作成するのにも役立ちます。

注： git + GitHubの使用を検討するときは、プルリクエスト（またはクールになりたい場合はPR）に特に注意してください。

ブランド


急勾配と言えば、自分のブランドはあなたが何ができるかを世界に示す方法です。 1つの方法（これは現在、最良の方法の1つです！）は、GitHubをブランドのプロキシとしてしっかりとバインドすることです。最近のほとんどすべての雇用主は、何らかの方法でそれを求めます。したがって、きちんと監視されたGitHubアカウントを作成するように努める必要があります。これはあなたがあなたの履歴書に入れて誇りに思うことができるものです。



次の記事では、Hugo静的サイトジェネレーターを使用して、GitHubにシンプルでクールなサイトを作成する方法を示します。それは囲碁で書かれており、おそらく、今日はアナログの中で最速です。Hugoは6秒間に約5,000ページを収集します。これはMiddlemanの75倍の速度です。しかし今のところは、コードをGitHubに配置するだけです。
後で、経験を積んだら、2つのGitHubアカウントを持つことを検討する価値があります。1つは、独自に作成したアプリケーション、作業コードを保存するためのもので、もう1つは、他の人に見せたいコードを保存するためのものです。

したがって、上記を要約します。

  • gitを学ぶ;
  • あなたが学んだすべてをGitHubに投稿する
  • 2つのアカウントを使用して、これから
    利益を得ることを学んだすべてを保存およびデモンストレーションします。

GitOpsなど、この分野の最新の開発情報を常に入手してください。GitOpsは、これまでに説明したすべてのアイデアを新しいレベルに引き上げます。そこでは、すべてがgit、pullリクエスト、デプロイメントパイプラインを使用して行われます。

GitOpsや同様の開発はビジネスの側面について語っています。つまり、Gitはクールでファッショナブルなので、Gitを使用しているだけではないことを示しています。これを使用して、ビジネスの俊敏性を保証し、イノベーションを加速し、より高速な機能を提供します。つまり、最終的にはより多くのお金を稼ぐためです。
