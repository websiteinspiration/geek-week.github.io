Anatomy Unit Testing

Unit tests are a mandatory part of my projects. This is the base to which other types of tests are added. In the article Testing and project economics, I told why testing is beneficial for the project economy and showed that unit testing is leading from an economic point of view. The comments suggested that testing requires a lot of effort, and even unit testing is unacceptable because of this. One reason for this is the inexperience of the team in testing. To write the first thousand tests, the team spends a lot of time trying and analyzing various approaches.



In this article I will talk about the best practices that I have come to over more than 10 years of testing various projects. These practices will allow unit testing to begin without a noticeable decrease in programmer productivity.



I define unit testing as testing a single unit production in a fully controlled environment.



A production unit is usually a class, but there can also be a function and a file.

It is important that the unit conforms to the SOLID principle, in which case the unit tests will be concise. Units are highly specialized and very well perform one specific task for which they are created. In most cases, units interact with each other, delegating the implementation of specialized tasks.



Fully Controlled Environment- This environment imitates the environment in which the unit works in the program, but is completely open for testing and configuration. The environment behavior is set for a specific test case via a concise API and any behavior outside this case is not defined for it. There should not be other production units in this environment, otherwise the complexity of the tests will increase and from unit testing we will proceed to integration testing.



About inheritance



Try not to apply inheritance. Use dependency composition instead. Often inheritance is used to implement the DRY principle (don't repeat yourself), taking the common code to the parent, but thereby violating the KISS principle (keep it simple stupid), increasing the complexity of units.



AAA (Arrange, Act, Assert) pattern



If you look at the unit test, then for most you can clearly distinguish 3 parts of the code:



Arrange (setting) - in this code block we set up the test environment of the unit under test;

Act - execution or call of the test script;

Assert - checking that the test call is behaving in a certain way.

This pattern improves the structure of the code and its readability, however, you should always start writing a test with the Act element.

AAS (Act, Assert, Setup) pattern



AAS is the same AAA pattern, but with a reordered parts sorted by the Driven approach and renamed the Arrange part in Setup to distinguish them by name.



The first thing we do when creating a test is to create an Act. Usually this is creating an instance of the tested unit class and calling its function. On the one hand, this is the easiest step, and on the other, this is what business history dictates to us.



Second, we verify that Act is acting as expected. We are writing the Assert part, where we express the required consequences of the Act, including from the point of view of business history.



And so, when the first 2 parts are ready, we can stop and think about how our unit will perform the required actions. Yes, yes, right now we can think for the first time how to implement it.



Look, the look of Act itself and its signature are dictated by the previous step, here we have nothing to invent. As the previous step wants to call our unit, so it will call it. The expected actions are also dictated by the previous step and the business history itself.



So right now, when we will write the last part of the test, we can stop and think about how our unit will work and what runtime environment it needs for this. And here we go in more detail to the ‚Äúcontrolled environment‚Äù and unit design.



SOLID Principles



From the SOLID principle, from the point of view of unit testing, 2 principles are very important:



Single responsibility principle - allows you to reduce the number of test cases for a unit. On average, a unit should have from 1 to 9 test cases. This is a very good indicator of the quality of a unit - if there is more test cases or you want to group them, then you definitely need to divide it into two or more independent units.



Dependency inversion principle- makes it easy to create and manage complex environments for testing through IoC containers. In accordance with this principle, a unit should depend on abstractions, which allows you to transfer to it any implementation of its dependencies. Including, and non-production implementations created specifically for its testing. These implementations do not have any business logic in them and are created not only for a specific unit under test, but also for a specific scenario for testing it. Usually they are created using one of the libraries for mock objects, such as moq.



IoC containers allow you to automatically create an instance of the unit under test and instances of its dependencies, immediately implemented as mock objects. Using such an IoC container is a very important step towards reducing the cost of maintaining the code and its friendliness for automatic refactoring.



Code quality



By the way, a few words about the quality of the test code and production. The best quality code should be the test code. There is only one reason for this - its size. For 1 line of production code, on average, there are 2-3 lines of test code, that is, it is 2-3 times more than production code. Under these conditions, it should be well read, structured, well typed, and very friendly to automatic refactoring tools. These are goals that are worthy of individual events and efforts.



Test uniformity



Many applications are implemented in a distributed and modular architecture, where different parts are written in different languages, for example, client-server applications, where the client is written under the web in typescript and the server written in c #. An important goal for such projects will be to bring tests for any part, regardless of language, to a single approach. This means that all tests on the project use the AAA or AAS approach. All tests use mock libraries with a similar API. All tests use IoC. And all tests use the same metaphors. This allows you to increase the portability of successful practices to different parts of the project, to simplify the adaptation of new colleagues (learned once and apply everywhere).



Number of tests for one production unit



On average, there are 1-9 tests per production unit. If there are more tests or if you have a desire to group tests, then this is a clear signal to check the unit‚Äôs production code. It is possible that he needs decomposition.
