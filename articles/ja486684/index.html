TDDの価値が誇張されていると信じている人への私の答え

かつて、クライアント企業の開発者とソフトウェアについて話しました。対話者がソフトウェア開発者である私たちが幸運だったと私たちが幸運だったと語ったとき、会話がどこか間違っていたことを理解する必要があります。誰かがコードを書くのにどれだけ進んだかは関係ありませんが、ソフトウェア開発業界全体を詐欺師の集団のように話すことは価値がないと思います。

私はこれに焦点を合わせなかった、会話はアジャイルに行きました。一般的に、クライアントは新しい方法論をテストし、彼らの作業プロセスを改善するという考えにオープンでした。しかし、私がテスト（TDD、テスト駆動開発）による開発について言及するまでは。これに対する唯一の答えは、「TDD値は誇張されている」というフレーズでした。



それを聞くのは辛かっただけでなく、TDDが「都市の伝説」のようなアジャイル手法の1つであることを実感しました。これが私がこの資料を書くきっかけになったものであり、TDDの価値を疑う人々にアピールしたいと思います。

TDDとは？


TDDの定義から始めましょう。これはソフトウェア開発戦略であり、その実装中は、プログラムを作成するときに、テストの作成によって完全にガイドされます。実際、これはこの方法論の名前から理解できます。このアイデアは、ケントベックが2003年に執筆した著書「テストによる開発」で提案されました。TDDの使用には、次の3つのステップが含まれます。

  1. 小さな機能で失敗するテストを作成する。
  2. テストの合格につながる機能の実装。
  3. 古いコードと新しいコードをリファクタリングして、適切に構造化された読み取り可能な状態に維持する。

このプロセスは、「赤、緑、リファクタリング」サイクルとも呼ばれます。

TDDの簡単な使用例を次に示します。ここではC、ピタゴラスの定理によって変数を計算するように設計されたメソッドをPythonで記述します。

ここにテストコード（ファイルtest.py）があります：

#!/usr/bin/env python

"""
test.py
"""

import unittest

from main import *

                a, b = 3, 4
                self.assertEqual(calculate_side_c(a, b), 5)

if __name__ == '__main__':
    unittest.main()

テストは、リファクタリング（TDDサイクルのパート「リファクタリング」）を受けたコードに対して成功しました

。開発ブックがテストされてから17年が経過しました。しかし、今日でも2020年には、TDDの利点がこの方法論を実装するための時間と労力に値するかどうかについて多くの人々にとって非常に重要な質問に対する決定的な答えはまだありません。その結果、多くの開発者はテストを通して開発を試みさえしません。

TDDをサポートするために、私は開発者になぜこの方法論を使用しなかったのか尋ねました。以下は、私が最もよく聞いた4つの回答です。

  1. QA部門があります。テストを書くのが彼らの仕事です。
  2. mokiとスタブが必要になる可能性のあるテストを作成するには、多くの時間と労力がかかります。
  3. TDDには、従来の開発に勝る利点はありません。
  4. TDDが遅い。

これらのアイデアを分析してみましょう。

QA部門があります。テストを書くことが彼らの仕事です


なぜ誰かがTDDを使用していないのかという私の質問に対するこの回答は、いつも少し笑わせます。コードを書いた人が責任を負うとき、私は「あなたはそれを構築し、あなたはそれを実行する」という原則を強く支持しています。したがって、開発者が機能的なコードを作成することが唯一のタスクであるかのように振る舞うことがわかると、それは私を混乱させます。

私は、開発者が自分のコードが次の品質を備えていることを確認する責任があると確信しています。

  • 正当性-ビジネスニーズを満たす。
  • 透明度。
  • テスト容易性。
  • 拡張性。
  • シンプルさ。

テストを作成するタスクをQA部門に転送することは、開発者の職責のリストに属していないと思います。QA部門のスペシャリストには、より重要な問題があります。彼らは、主に「ブラックボックス」メソッドを使用したコードのテストに作業時間を費やすべきではありません。

モキとスタブが必要になる可能性のあるテストを作成するには、多くの時間と労力がかかる可能性があります。


そう言う人の気持ちがわかります。たとえば、3つの異なる入力値を受け取るメソッドを実装する必要があります。それらのそれぞれはオブジェクトであり、これらの各オブジェクトにはオプションの属性はありません。メソッドのさまざまなオプションをチェックして、これらすべてをテストする必要があります。これを行うには、スタブとmokasを構成し、大量の追加コードを記述する必要があります。これはすべて、1つのメソッドをテストするためのものです。私はすでに同じような状況にあります。しかし、私はこれらすべてを別の視点から見ています。

テストの作成に費やす労力と時間を削減することを考えると、SOLIDプログラミングとテストピラミッドの原則が思い付きます。ここから

SOLID を始めましょう。。ここで、略語ではSOLIDが文字Sで表されるという事実、つまり、単一の責任の原則（単一の責任の原則）について詳しく説明します。これは、どのメソッドとクラスが1つの問題のみを解決すべきかによるアイデアです。彼らは彼らの活動がシステムの他の部分に影響を与えないように機能しなければなりません。

エンタープライズソフトウェアのほとんどのメソッドは、これらのメソッドに渡されるいくつかの数値の合計を見つけるよりも難しいタスクを実行することを理解しています。しかし、プロジェクトの1つの問題を解決することを目的とした方法を使用すると、テストが非常に容易になるという事実を否定することはできません。

テストピラミッドについて話しましょう。


テストピラミッド（ここからの画像）

テストピラミッドは、プロジェクト内のさまざまな種類のテストの正しい比率を見つけるためによく使用されます。

ユーザーインターフェイステスト（UIテスト）とサービステスト（サービステスト）の実装には、かなりの時間が必要です。したがって、多数のテストを実行するのに数ミリ秒かかるため、テストの大部分はユニットテスト（ユニットテスト）で表す必要があります。これはもちろん、DevOpsの主要な目標の1つであるフィードバックループの改善に役立ちます。

ユニットテストを実行してシステムの一部をテストする場合も、システムの他の部分との統合レベルがはるかに低くなります。これにより、そのようなテストを簡単に作成できます。TDD方法論は、単体テストの使用専用に設計されています。

もちろん、言うことは言うより簡単です。ほとんどのテストを作成するには、単体テストを含めて、少し努力する必要があります。ただし、単体テストの作成に多くの労力がかかると思われる場合は、いくつか質問して自分でテストできます。

  • これはモジュール式テストですか、それともサービステストとして、より高いレベルのテストと見なす必要がありますか？
  • コードは適切に設計されていますか？各クラスとメソッドは単一責任の原則と一致していますか（コードは非常に密結合されていますか）？

テストの作成に時間と労力がかかりすぎることが判明した場合、ほとんどの場合、これはコードをリファクタリングすることができ、おそらくリファクタリングする必要があることを示しています。

TDDは従来の開発に勝る利点はありません


皮肉なことに、TDDが従来の開発よりも優れていないという事実は、主にTDDスタイルで作業を試みたことのないプログラマーによって言われています。しかし、何かを試すまで、これが良いか悪いか理解できません。コールドプレイバンドでさえ、「絶対に試さなければ、決して分からない」ことを知っています。

TDDには2つの主な長所があります。

TDDの最初の利点はかなり明白です。動作するコードを書く前に、誰かが常にこのコードのテストを書く場合、プログラマーは当然のことながら、常に自己テストコードを自由に使える。 Martin Fowlerがこれについて書いているのは、次のとおりです。「コードベースで一連の自動テストを実行でき、テストが正常に完了すると、コードに重大な欠陥がないことを確認できるセルフテストコードがあります。」

言い換えると、TDDを使用するということは、コードがプログラマーのニーズどおりに機能することを意味します。

これは、不利な変更に対するコードの安定性に深刻な信頼を与えるため、素晴らしい事実です。 TDDを使用すると、プログラマーは、コードのリファクタリングまたは拡張の結果としてコードベースの何かが劣化していないかどうかを即座に見つけることができます。さらに良いことに、システムにエラーがあるかどうかがわかります。

セルフテストコードにより、開発チームは継続的インテグレーションとコード導入システムのメリットを真に活用できます。

TDDの2番目の主な利点は、この開発方法論により、プログラマーは、機能するコードを書く前であっても、何を書こうとしているのか、どのように書こうとしているのかを考えさせられることです。

コードを書く前によく考えてみると、開発者は本当にビジネスのニーズの本質に没頭していて、適切なメカニズムの実装における境界線のケースや起こりうる困難を考慮に入れているという事実につながります。その結果、実際に機能するコードを作成する場合、必要なものだけに労力が費やされます。さらに、TDDの使用は、開発者が構造とアーキテクチャの観点から将来のデバイスシステムを計画しているという事実につながります。そのようなことが計画され、システムの作業の最初から考慮に入れられている場合、これにより、スケーリングと拡張の能力が大幅に向上します。

TDDが遅い


TDDが遅いと言う開発者、通常はしばらくの間この方法論を使用して作業し、コードの作成後に実行されるテストの記述に戻った開発者。一般的に、これは、誰かがTDDを使用しない理由について話すときに最もよく参照される理由です。

プログラマがそう思う理由も理解できます。書く前にコードについて考えるのは時間がかかります。通常の条件下ではテストが記述されていない可能性のあるメソッドであっても、実装が計画されているすべてのテストの作成に時間を費やしています。スタブとモブを効果的に使用する方法を見つけることも、最速で楽しい作業ではない場合があります。

プログラマーの生産性と効率は、仕事のスピードと完成品の作成に費やされる時間の点で評価されることが多いという事実によって、状況はさらに悪化します。これにより、プログラマーのチームは、可能な限り迅速に作業したいという欲求、厳しい作業スケジュールに適合したいという欲望に駆り立てられます。

では、この論文DevOpsチームの有効性を評価するための4つの指標のシステム：

  1. 展開頻度。
  2. 実行時間を変更します。
  3. サービス復旧時間。
  4. 変更による失敗の頻度。

特に、「展開の頻度」と「変更による失敗の頻度」の指標に興味があります。

TDDを使用するとき、私が言ったように、システムがTDDのみを使用して構築されたという仮定に基づいて、システムに変更を加えた直後に、開発チームはこの変更がいずれのパフォーマンスにも違反していないかどうかを知るでしょうシステムの一部から。さらに、TDDを使用すると、エラーの検出速度が向上します。テストがコードの非常に小さなフラグメントをテストするように設計されているという事実は、障害が発生したプロジェクトの場所を非常に迅速に見つけることを可能にします。これにより、すでに展開されているシステムの更新されたバージョンではエラーが検出されにくいため、「変更中の失敗率」の割合が減少します。

開発チームがプロジェクトの持続可能性とその正しい作業に高い信頼を置いているという事実は、テストが正常に完了したときにチームがプロジェクトを展開するかどうかを決定できることを意味します。誰かがオンデマンドでプロジェクトの展開を行いましたか？

フェニックスプロジェクトは4種類の仕事をカバーしています。それらの1つは予期しない作業です。このような作業は、開発者が自分のやっていることをやめ、何か他のことをする必要があるときに行う必要があります。通常、何らかの間違いの修正。プロジェクトがセルフテストコードからビルドされている場合、これによりエラーの量が最小限に抑えられます。これにより、「予期しない作業」の量を最小限に抑えることができます。そして、開発者の生活の中でこのような「驚き」が少ないほど、彼はより良く感じ、より生産的でよりよく働きます。

開発者が本番環境のバグについて心配する必要がない場合、これは、製品の新しい有用な機能の開発により多くの時間を費やすことができることを意味します。そして、開発者がコードを事前に考え、SOLIDのような有用な原則を適用し、常にリファクタリングに従事する場合、これは「技術的負債」の量を最小限に抑えます。チームがコードの機能と品質に自信を持っている場合、文字通り2つの点でコードを展開できます。これらすべてにより、作業速度が向上します。
