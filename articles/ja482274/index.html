理由＃1：GDI記述子


サーバーでSystem.Drawingを使用して問題が発生したことがある場合、これが最も当てはまる可能性があります。テストされていない場合、これは最も可能性の高い原因の1つです。

System.Drawingの大部分は、Windows GDI + APIの薄いラッパーです。多くのSystem.DrawingオブジェクトはGDI記述子でサポートされており、プロセッサとユーザーセッションに量的な制限があります。このしきい値に達すると、「メモリ不足」の例外やGDI +「一般的な」エラーが発生します。

問題は、.NETでは、ガベージコレクションとプロセスの終了により、負荷が軽い場合でも、制限に達するまでにこれらの記述子の解放が遅延する可能性があることです。そのような記述子を含むオブジェクトでDispose（）を呼び出すのを忘れた（または必要なものを知らなかった）場合、環境でそのようなエラーが発生する危険性が非常に高くなります。また、リソースの制限やリークに関連するほとんどのバグと同様に、この状況はテストに成功し、生産的な運用が可能になります。当然、これはアプリケーションに最大の負荷がかかっているときに発生するため、最大数のユーザーがあなたの恥を知ることになります。

CPUとユーザーセッションの制限オペレーティングシステムのバージョンに応じて、プロセッサの制限はカスタマイズ可能です。しかし、バージョンは関係ありません。GDI記述子はUSHORTデータ型によって内部的に表されるため、ユーザーセッションごとに65,536記述子の厳密な制限があり、十分に作成されたアプリケーションでも、十分な負荷がかかるとこの制限に達するリスクがあります。より強力なサーバーを使用すると、1つのインスタンスでより多くのユーザーに並行してサービスを提供できると考えると、このリスクはより現実的になります。そして、本当に、スケーラビリティのハードリミットがよく知られているソフトウェアを誰が作成したいのでしょうか？

理由＃2：同時実行性


GDI +には常に同時実行性の問題がありましたが、それらの多くはWindows7 / Windows Server 2008 R2のアーキテクチャの変更に関連していたものですが、それらのいくつかは新しいバージョンでも見られます。最も注目に値するのは、DrawImage（）操作中にGDI +によって調整されるプロセスロックです。 System.Drawing（またはそれをラップするライブラリ）を使用してサーバー上の画像のサイズを変更する場合、DrawImage（）メソッドがおそらくこのコードの基礎になります。

さらに、DrawImage（）を同時に複数回呼び出すと、すべてが完了するまで、それらすべてがブロックされます。それらは実行されません。応答時間が問題にならない場合でも（なぜでしょうか？ユーザーは嫌いですか？）これらの要求に関連付けられているメモリリソースと、これらの要求に関連付けられているオブジェクトが保持するすべてのGDI記述子はランタイムに関連付けられていることに注意してください。実際、サーバーに負荷をかけすぎて問題が発生することはありません。

もちろん、この特定の問題には回避策があります。たとえば、一部の開発者は、DrawImage（）操作ごとに外部プロセスを作成します。しかし、実際には、このような回避策は脆弱性を追加するだけであり、実際に行うべきではありません。

理由＃3：メモリ


チャートを生成するASP.NETハンドラーについて考えてみます。次のようになります。

  1. キャンバスとしてビットマップを作成する
  2. ペンやブラシを使用してビットマップに複数の形状を描画します
  3. 1つ以上のフォントを使用してテキストを描画する
  4. ビットマップをPNGとしてMemoryStreamに保存する

チャートのサイズが600 x 400ポイントであるとしましょう。これは合計240,000ドットであり、デフォルトのRGBAフォーマットのドットの場合は4バイトが乗算され、ビットマップの場合は合計960,000バイトに加えて、描画オブジェクトと保存バッファー用に少し追加されます。リクエスト全体で1MBとします。ほとんどの場合、このようなシナリオではメモリの問題は発生せず、何かに遭遇した場合は、画像、ブラシ、ペン、フォントに独自の記述子があるため、記述子の数に制限がある可能性が高くなります。

実際の問題は、System.Drawingがイメージングタスクに使用されるときに発生します。System.Drawingは主にグラフィックライブラリであり、グラフィックライブラリは通常、すべてがメモリ内のビットマップであるという考えに基づいて構築されています。ささいなことを考えると、これは素晴らしいことです。しかし、画像は非常に大きくなる可能性があり、毎日大きくなります。メガピクセルの多いカメラは常に安くなっています。

System.Drawingの単純なアプローチで画像を構築する場合、サイズ変更ハンドラーの場合は次のようになります。

  1. 宛先画像のキャンバスとしてビットマップを作成します。
  2. 元の画像を別のビットマップに読み込みます。
  3. DrawImage() «-» -, .
  4. JPEG .

対象画像は、前の例のように、600x400になると仮定し、その後、再び我々はのための1メガバイトを持っている対象画像とメモリストリーム。しかし、誰かが彼らの派手な新しいDSLRから24メガピクセルの画像をアップロードしたと仮定しましょう。RGB形式のデコードされたソースビットマップには、それぞれ（72mb）に3バイトの6000x4000ピクセルが必要です。また、見栄えが良いだけなので、System.DrawingからのHighQualityBicubicリサンプリングを使用します。次に、呼び出されたメソッド内で発生するPRGBA変換のために、それぞれ4バイトのその他の6000x4000ポイントを考慮する必要があります。これにより、使用済みメモリがさらに96MB増加します。単一の画像を変換する要求に対して、合計で169mb（！）が取得されます。

ここで、そのようなことをしている複数のユーザーがいると想像してください。ここで、すべてのリクエストが完全に実行されるまでリクエストがブロックされることを思い出してください。メモリ不足になるまでどのくらいかかりますか？利用可能なすべてのものを使い尽くすことを心配しなくても、ピクセルの束を保持するよりもサーバーのメモリをより有効に使用する方法がたくさんあることに注意してください。アプリケーション/システムの他の部分へのメモリプレッシャーの影響を検討します。

  1. ASP.NETキャッシュが再作成にコストのかかるアイテムをフラッシュし始める場合がある
  2. ガベージコレクターがより頻繁に起動し、アプリケーションの速度が低下する
  3. IISカーネルキャッシュまたはWindowsファイルシステムキャッシュは、有用な要素を削除する可能性があります
  4. アプリケーションプールがメモリ制限を超え、再起動される可能性があります
  5. Windowsがメモリをディスクにスワップし始め、システム全体の速度が低下する

あなたは本当にこれのどれも欲しくないのですか？

画像処理タスク専用に設計されたライブラリは、この問題にまったく異なる方法で取り組みます。ソースまたはターゲットのイメージ全体をメモリにロードする必要はありません。描画しないのであれば、キャンバス/ビットマップは必要ありません。これは次のように行われます：

  1. ターゲットイメージのJPEGエンコーダー用のストリームを作成する
  2. 元の画像から1行を読み込み、水平方向に圧縮します
  3. 必要な回数だけ繰り返して、ターゲットファイルの1行を形成します。
  4. 結果の行を垂直方向に圧縮します
  5. ソースファイルのすべての行が処理されるまで、手順2から繰り返します。

この方法を使用すると、合計で1 MBのメモリのみを使用して同じ画像を処理できます。さらに大きな画像でも、オーバーヘッドをわずかに増やす必要があります。

この原則によって最適化された.NETライブラリは1つだけ知っているので、ヒントを示します。これはSystem.Drawingではありません。

理由＃5：画像処理が一見複雑


パフォーマンスはさておき、さまざまな方法でSystem.Drawingを使用すると、画像が正しく処理されなくなります。System.Drawingを使用するということは、誤った出力に対処するか、ICCプロファイル、色の量子化、exifの向き、修正、その他多くの特定のことについてすべてを学ぶことを意味します。これはうさぎの穴であり、ほとんどの開発者は時間も探究心もありません。
