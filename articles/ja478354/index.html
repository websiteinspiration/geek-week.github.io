実用的な関数型プログラミング

こんにちは、ハブル！Robert C. Martin（Uncle Bob）による「プラグマティック関数型プログラミング」の記事の翻訳に注目します。

関数型プログラミングへの移行は、ほんの10年前に本格的に発展しました。 Scala、Clojure、F＃などの言語が注目されています。実際、それはプログラミングの大きな一歩でした。「ああ、かっこいい、新しい言語！」 -熱意...どうやら何か特別なことがあったようです。  

ムーアの法則によれば、コンピュータの速度は18か月ごとに2倍になります。この法律は1960年代から2000年まで施行されました。それから彼は立ち止まった。周波数は3 GHzに達し、その後完全にプラトーに上昇しました。光速に達しました！信号は、チップの表面を高速で伝播するのに十分な速さで伝播することはできません。 

これにより、機器エンジニアは戦略を変更するようになりました。スループットを向上させるために、プロセッサ（コア）を追加しました。そして、これらのコアのためのスペースを作るために、彼らはチップからキャッシングとパイプライン機器のほとんどを取り除きました。このため、プロセッサは以前よりもかなり遅くなっています。しかし、それらはもっとあります。その結果、スループットが向上しました。 

8年前に初めてデュアルコアコンピューターを手に入れました。 2年後、私はすでにクアッドコアのコンピューターを持っていました。これから核の拡散が始まりました。そして結局のところ、最も興味深いことは、これがソフトウェア開発に想像を絶するような影響を与えることを私たちは皆理解しているということです。 

私たちの答えの1つは、関数型プログラミング（FP）の研究でした。 FPでは、初期化後に変数の状態を変更しないことを強くお勧めします。これは同時実行性に大きな影響を与えます。変数の状態を変更できない場合、競合状態にすることはできません。変数の値を更新できない場合は、同時に更新しても問題ありません。

もちろん、これはマルチコア問題の解決策と見なされていました。しかし、核、並行性の急増により、それは明らかになりました-いいえ、同時性は深刻な問題になります。 FPは、1つのプロセッサで1024コアを使用する際の問題を軽減するプログラミングスタイルを提供する必要があります。

これに基づいて、全員がClojure、またはScala、またはF＃、またはHaskellの学習に取り掛かりました。彼らは貨物列車がやってくるのを知っていて、彼が到着するまでには準備ができていたからです。

しかし、貨物列車は到着しませんでした。先ほど言ったように、6年前にクアッドコアのラップトップを手に入れました。その瞬間から、同じノートパソコンをあと2台持っていました。次に受け取ったラップトップは、まるでクアッドコアであるかのように外側に見えました。どうやら別の高原（ムーアの法則）が見えますか？

だから、AFはかつて思ったほど重要なスキルではありません。多分私達はカーネルで散らかされることはないでしょう。おそらく、32,768コアのチップについて心配する必要はないでしょう。たぶん私たちはすべてリラックスして、変数の更新に戻ることができます。

これは間違いだと思います。そして、非常に大きな間違いです。これは後藤の蔓延する使用と同じくらい誤解されます。また、動的に送信することを拒否するのと同じくらい危険だという考えもあります。

なぜ聞くの？そもそも私たちに興味を持った理由から始めましょう。 FPは並行性をはるかに安全にします。多数のスレッドまたはプロセスでシステムを構築している場合、FPを使用すると、競合状態や同時更新で発生する可能性のある問題が大幅に減少します。

なぜ他に？たとえば、FPは記述、読み取り、テスト、および理解が容易です。今、私はあなたの何人かが手を振って画面で悲鳴を上げるのを想像します。あなたはAFを試みました、そしてあなたを見つけるのは簡単ではありませんでした。このすべてのマッピング、削減、およびすべての再帰、特に末尾再帰は、理解するのが容易ではありません。もちろん私はそれを理解しています。しかし、今のところ、それはデートの問題です。これらの概念に慣れるとすぐに、そしてこの知人を開発するのにそれほど時間はかかりませんが、プログラミングははるかに簡単になります。

なぜ簡単になるのですか？システムのステータスを監視する必要がないため。変数の状態は変化しません。したがって、システムの状態は変更されません。これは、追跡する必要のないシステムだけではありません。リストの状態、配列の状態、スタックの状態、またはキューを監視する必要はありません。これらのデータ構造は変更できないためです。 FP言語でスタックにアイテムをプッシュすると、新しいスタックが取得され、古いスタックは変更されません。これは、プログラマーが空中でより少ない数のボールを同時に操作しなければならないことを意味します。結局のところ、覚える必要はほとんどありません。追跡する必要の少ないもの。そしてそれが、コードの作成、読み取り、理解、テストがはるかに簡単になる理由です。

それでは、どのFP言語を使用する必要がありますか？個人的に、私のお気に入りはClojureです。その理由は、Clojureは非常に単純だからです。これは美しい言語であるLispの方言です。披露させて。

Javaの関数は次のとおりです。f（x）;

これをLispの関数にするには、最初のブラケットを左に移動します：（fx）。

95％Lispと90％Clojureを知っています。すごいでしょ？愚かな小さな括弧の構文は、これらの言語が持っているほとんどすべての構文です。とてもシンプルです。

あなたは以前にLispプログラムを見たことがありますが、これらすべての括弧が好きではありません。ほとんどの場合、CAR、CDR、CADRなどはまだ好きではありません。心配しないでください。 Clojureは、Lispよりも句読点が少し多いため、括弧が少なくなっています。 Clojureはまた、CAR、CDR、およびCADRを、first、rest、secondに置き換えました。 ClojureはJVM上に構築されており、Javaライブラリ全体、および使用するその他のJavaインフラストラクチャまたはライブラリへのフルアクセスを提供します。相互運用性は迅速かつ簡単です。さらに良いことに、ClojureはJVMのOO機能への完全なアクセスを提供します。

"ちょっと待って！"あなたが話しているのが聞こえます。 「そうです、FPとOOは相互に互換性がありません！」誰があなたにそれを言ったの？これはまったくナンセンスです。しかし、FPではオブジェクトの状態を変更できないという事実は真実です。だから何？スタックに整数を挿入すると新しいスタックが得られるのと同じように、オブジェクトの値を調整するメソッドを呼び出すと、古いオブジェクトを変更する代わりに新しいオブジェクトを取得します。慣れればすぐに扱えます。

しかし、OOに戻りましょう。ソフトウェアアーキテクチャレベルで最も役立つOOの機能の1つは、動的なポリモーフィズムです。また、ClojureはJava動的ポリモーフィズムへの完全なアクセスを提供します。この例は、これを最もよく説明するのに役立ちます。

(defprotocol Gateway
  (get-internal-episodes [this])
  (get-public-episodes [this]))

上記のコードは、JVMのポリモーフィックインターフェイスを定義しています。Javaでは、このインターフェースは次のようになります。

public interface Gateway {
	List<Episode> getInternalEpisodes();
	List<Episode> getPublicEpisodes();
}

JVMレベルでは、生成されるバイトコードは同じです。実際、Javaで作成されたプログラムは、Javaで作成された場合と同じ方法でインターフェースを実装します。同様に、ClojureはJavaインターフェースを実装できます。Clojureでは、次のようになります。

(deftype Gateway-imp [db]
  Gateway
  (get-internal-episodes [this]
    (internal-episodes db))

  (get-public-episodes [this]
    (public-episodes db)))

dbコンストラクターの引数と、すべてのメソッドがそれにアクセスする方法に注意してください。この場合、インターフェイスの実装は、dbを渡すことにより、一部のローカル関数を委任するだけです。

おそらく何よりも優れているのは、LispとClojureが同型であるということです。つまり、コードはプログラムが操作できるデータです。見やすいです。次のコード：（1 2 3）は、3つの整数のリストを表します。リストの最初の要素が（f 2 3）のように関数である場合、それは関数呼び出しになります。したがって、Clojureのすべての関数呼び出しはリストです。リストはコードで直接制御できます。したがって、プログラムは他のプログラムを作成して実行できます。

つまり、関数型プログラミングは重要であり、学ぶ必要があります。そして、もしあなたがどの言語がそれを学ぶのに使うほうが良いかに興味があるなら、私はあなたにClojureを提供します。
