Laravelでメモリ消費を増やすことなくN + 1クエリの問題を解決する

ORMでアプリケーションを作成するときの開発者の主な問題の1つは、アプリケーションでのN + 1要求です。 N + 1クエリの問題は、アプリケーションがオブジェクトの呼び出しごとにリクエストを生成するときに、データベースにアクセスする効率的な方法ではありません。この問題は通常、遅延負荷または積極的負荷を使用せずにデータベースからデータのリストを取得したときに発生します。幸いにも、ORM Eloquentを備えたLaravelは、便利な操作のためのツールを提供しますが、いくつかの欠点があります。

Class User extends Authenticatable
{
    public function articles()
    {
        return $this->hasMany(Aricle::class, 'writter_id');
    }

    public function first_article()
    {
        return $this->hasOne(Aricle::class, 'writter_id')->orderBy('created_at', 'asc');
    }

    public function scopeWithFirstArticle($query)
    {
        $query->addSelect(['first_article_id' => Article::select('id')
                ->whereColumn('writter_id', 'users.id')
                ->orderBy('created_at', 'asc')
                ->take(1)
        ])->with('first_article')
    }

}

public function getFirstArticleAtribute()
{
    if(!array_key_exists('first_article', $this->relations)) {
        $this->setRelation('first_article', $this->articles()->oldest()->first());
    }

    return $this->getRelation('first_article');
}


実際、動的な通信のために遅延読み込みを実装していません。クエリの結果を割り当てるだけで、ユーザーの最初の記事を受け取ります。これは、貪欲な読み込みと遅延読み込みの両方でfirst_articleプロパティにアクセスする場合にも同じように機能します。







Laravel 5.Xの動的接続



残念ながら、このソリューションはLaravel 6以降にのみ適用されます。Laravel 6以前のバージョンでは、addSelectの異なる実装を使用しています。フレームワークの古いバージョンで使用するには、コードを変更する必要があります。サブクエリを実行するには、selectSubを使用する必要があります。







public function scopeWithFirstArticle($query)
{
    if(is_null($query->toBase()->columns)) {
        $query->select([$query->toBase()->from . '.*']);
    }

    $query->selectSub(
                Article::select('id')
                ->whereColumn('writter_id', 'users.id')
                ->orderBy('created_at', 'asc')
                ->take(1)
                ->toSql(), 
            'first_article_id'
            )->with('first_article');
}
