PVS-Studio分析仪：确定潜在的Java SE API兼容性问题











就会议而言，2019年是非常繁忙的一年。我们的团队可能要出差数周。如您所知，会议是分享知识的时候。除了在我们的展台上进行演讲并讲了很多有趣的事情外，我们还通过与会议参与者和演讲者的交流学到了很多东西。因此，在2019年秋季Joker会议上，Dalia Abo Sheasha的一份报告“迁移到Java 8之外”启发了我们实施新的诊断规则，该规则可让您识别Java SE API在不同版本的Java之间的不兼容性。将对此进行讨论。



Java SE兼容性问题的相关性



Java SE 14已经发布，尽管如此，许多公司仍在继续使用Java的早期版本（Java SE 6、7、8，...）。由于时间在运行并且Java一直在更新，因此Java SE API的不同版本之间的兼容性问题每年变得越来越重要。



发行新版本的Java SE时，它们通常与较早版本向后兼容，例如，基于Java SE 8开发的应用程序应该可以在11种Java版本上运行而不会出现问题。但是，实际上，许多类和方法可能会出现一些不兼容的情况。这种不兼容之处在于某些API会发生变化：它们被删除，改变了行为，被标记为过时等等。



仅当您开始考虑将项目迁移到最新的Java SE时，此问题才会恶化。或者，当您的应用程序的技术支持收到越来越多的信函时，表明该应用程序的行为不正确或根本无法启动。



我认为这足以引起注意！



现有工具包



没有一种万能的解决方案可以从一种Java版本切换到另一种Java版本。而且，如果您的应用程序是使用各种非平凡的解决方案进行多年持续开发的结果，那么当您对过渡到最新版本的Java感到困惑时，您可能会遇到一个相当耗时的过程。



此过程包括确定有问题或潜在有问题的API，需要对其进行检查并用替代解决方案替换。这会严重影响业务逻辑，这可能需要大量时间来修复和测试。如果不仅需要切换到Java SE的新版本，而且还需要确保与多个版本的兼容性，那么此任务将变得更加复杂。



在Internet上讨论了识别不同Java SE之间的API不兼容性的话题之后，我只遇到了JDK附带的工具：javac，jdeps，jdeprscan。



该业务中没有第三方工具，除了我荣幸地在Joker 2019报告-Application Binaries迁移工具包中遇到的第三方工具。



javac



开发应用程序时，不要忘记编译器警告。每个人对警告的态度都不同：



  • 有人严格注视着他们，有一点东西立即响起了警报。也许有点偏执，但它很可能会减少未来的技术风险；
  • 有人将它们收集在一个单独的文件中，以为他一定会审查它们，如何释放他；
  • 通常有人会忽略警告。


第一种选择是好的，因为可以立即解决部分问题，例如，只需不使用标记为不推荐使用的方法或类即可。使用这样的API并不是一个阻塞性的问题，但是您应该注意它，因为使用新发布的Java版本时，您的应用程序的行为可能会有所不同甚至崩溃。



jdeps



如果您已经对将应用程序迁移到Java SE的更新版本感到困惑，那么jdeps会很乐意为您提供帮助。



jdeps是一个命令行工具，它对应用程序和库的依赖性进行静态分析，接受* .class文件或* .jar作为输入。从Java SE 8开始，它与JDK捆绑在一起。



在这里，我们感兴趣的事实是，如果您使用--jdk-internals选项运行此工具，它将告诉您每个类都依赖哪个内部JDK API。由于内部API不能保证它在Java的未来版本中不会更改，因此这一点非常重要。



让我们来看一个例子。假设您已经开发Java 8应用程序很长时间了，并且从未对Java SE API与较新版本的兼容性感到困惑。在将您的应用程序迁移到例如Java 11时，存在一个问题。在这种情况下，您可以沿着阻力最小的途径，立即使用Java 11启动应用程序。但是出了点问题，并假设您的应用程序开始崩溃。在这种情况下，可以通过设置应用程序的* .jar文件从Java 11运行jdeps 。



结果将大致如下：













输出显示将删除Java 11 中的依赖项sun.misc.BASE64Encoder。而且，在Java 11中首次启动应用程序时，其崩溃很可能是由于java.lang.NoClassDefFoundError错误。除了通知您无法满足要求之外，jdeps还可以为您提供替代性依赖项，该依赖项可以代替当前依赖项使用。在这种情况下，他建议将远程依赖项替换为java.util.Base64。



该工具的第二次警告表明我们仍在代码中使用另一个内部依赖项，但到目前为止是正确的。在Java的将来版本中，此API是否会有任何更改，但是必须考虑到这一点。



当然，jdeps可以做的不仅仅是测试内部JDK组件的使用。这不包括在本文的范围内，但是您可以自己在官方页面上熟悉它的功能。



jdeprscan



jdeprscan的目标与jdeps的目标完全相同，即帮助找到不需要的有问题的API。



jdeprescan是一种静态分析工具，可扫描* .jar文件（或* .class文件的其他集合）以使用过时的API元素。使用过时的API并不是一个阻塞问题，但是您应该注意它。从Java SE 9开始，它与JDK捆绑在一起。



还假设存在将应用程序迁移到Java 11的问题。在这种情况下，通过运行命令 



jdeprscan --release 8 app.jar


您将获得Java 8弃用的API列表，即在将来的Java版本中可能会删除的API。更正所有警告后，即可运行 

jdeprscan --release 11 app.jar


它将生成Java 11已经弃用的API列表。通过这种方式，您可以查找并修复（如果需要）所有非推荐的API。



应用程序二进制文件迁移工具包



此工具旨在帮助您快速评估用户应用程序是否存在潜在问题，然后再将其部署到不同的服务器（JBOSS，WebShere，Tomcat，WebLogic等）上。除了所有功能之外，该工具还允许您确定不同版本的Java SE API中的差异。



让我们快速看一下这个工具的全部含义。



该工具的快速启动如下所示： 



java -jar binaryAppScanner.jar yourApp.jar --analyzeJavaSE 
--sourceJava=oracle8 --targetJava=java11 ....


analyticsJavaSE 选项暗含各种参数的使用，您可以通过调用帮助来了解这些参数。



开始分析之后，您将很快在网络浏览器中看到一个报告：















屏幕上并非所有内容都适合=（。虽然您还没有尝试运行此工具，但我将用语言来描述它。



在报告中，您可以看到具有3个严重级别的规则触发：



  • 严重性级别较高的规则（远程API，行为更改，使应用程序无法运行且需要修复）；
  • 警告规则（更改可能导致问题的API行为，值得探讨）；
  • 信息规则（使用过时的API，API行为可能会稍微改变行为，但不会影响程序）。


可以扩展每个操作，并查看有关内容和方式的说明。信息性消息中有一条建议，他们说除了我们以外，还运行jdeps。他们说我们专注于应用程序的迁移，并且jdeps将另外帮助检测内部JDK软件包中的问题（除了发现的问题之外）。



此外，在报告的底部，您可以找到执行分析所依据的规则列表。



如果使用Eclipse IDE，则可以使用插件。有关更详细的研究，请访问他们的页面。



在PVS-Studio中实施



在研究了考虑过的工具之后，我们得出的结论是，对于不同版本的Java SE API进行潜在的兼容性问题的搜索对于静态分析而言是一项值得完成的任务。



所讨论的工具实际上将使迁移应用程序或搜索可能会干扰您在Java SE的最新版本上的性能的API（如果您不想迁移应用程序）更加容易。但是考虑到这些工具始终需要与开发过程分开在命令行上运行以识别问题，因此我们决定这样做不是很方便。从在开发的早期阶段就必须精确地进行静态分析以检测有问题或潜在有问题的代码这一事实开始，我们实施了诊断规则V6078，这将向您发出有关“问题” API的信号。



规则V6078会提前警告您，您的代码取决于某些Java SE API函数和类，这可能会给您将来的Java版本带来困难。而且您在实施特定功能的一开始就不会与该API绑定，从而减少了将来的技术风险。



在以下情况下，诊断规则将发出警告：



  • 如果在Java的目标版本中删除了方法/类/包；
  • 如果在Java的目标版本中方法/类/包被标记为已弃用；
  • 如果方法已更改签名。


该规则当前允许您分析Oracle Java SE从版本8到版本14的兼容性。为了使规则生效，必须对其进行配置。



IntelliJ IDEA



在IntelliJ IDEA插件中，您需要启用规则并在“设置”>“ PVS-Studio”>“ API兼容性问题检测”选项卡中指定参数，即：



  • 源Java SE-开发应用程序的Java版本；
  • 目标Java SE-您要用来检查应用程序中使用的API的兼容性的Java版本（源Java SE）；
  • 排除软件包-您要从兼容性分析中排除的软件包（软件包以逗号列出）。












Gradle插件



使用gradle插件，您需要在build.gardle中配置分析器设置： 



apply plugin: com.pvsstudio.PvsStudioGradlePlugin
pvsstudio {
    ....
    compatibility = true
    sourceJava = /*version*/  
    targetJava = /*version*/
    excludePackages = [/*pack1, pack2, ...*/]
}


Maven插件



使用maven插件，您需要在pom.xml中配置分析器设置： 



<build>
  <plugins>
    <plugin>
      <groupId>com.pvsstudio</groupId>
      <artifactId>pvsstudio-maven-plugin</artifactId>
      ....
      <configuration>
        <analyzer>
          ....
          <compatibility>true</compatibility>
          <sourceJava>/*version*/</sourceJava>
          <targetJava>/*version*/</targetJava>
          <excludePackages>/*pack1, pack2, ...*/</excludePackages>        
        </analyzer>
      </configuration>
    </plugin>
  </plugins>
</build>


直接使用内核



如果直接通过命令行使用分析器，则要激活所选Java SE API的兼容性分析，必须使用以下参数： 



java -jar pvs-studio.jar /*other options*/ --compatibility 
--source-java /*version*/ --target-java /*version*/ 
--exclude-packages /*pack1 pack2 ... */


分析仪操作



假设我们正在开发基于Java SE 8的应用程序，并且我们有一个包含以下内容的类： 



/* imports */
import java.util.jar.Pack200;
public class SomeClass
{
  /* code */
  public static void someFunction(Pack200.Packer packer, ...)
  {
    /* code */
    packer.addPropertyChangeListener(evt -> {/* code */});
    /* code */
  }
}


使用诊断规则的各种设置运行静态分析，我们将观察下图：



  • 源Java SE-8，目标Java SE-9
    • “ addPropertyChangeListener”方法将被删除。


  • 源Java SE-8，目标Java SE-11
    • “ addPropertyChangeListener”方法将被删除。
    • 'Pack200'类将被标记为已弃用。


  • 源Java SE-8，目标Java SE-14
    • “ Pack200”类将被删除。




首先，在Java SE 9中，删除了“ Pack200.Packer”类中的“ addPropertyChangeListener”方法。在版本11中，将类'Pack200'标记为已弃用的事实添加到了该版本中。在版本14中，该类已完全删除。



因此，以Java 11启动应用程序，将得到“ java.lang.NoSuchMethodError”，如果以Java 14运行，则将得到“ java.lang.NoClassDefFoundError”。



了解这些信息后，在开发应用程序时，您将考虑该任务的替代解决方案。



进一步发展的想法



在执行诊断规则期间，出现了有关扩展的想法：



  • 由于jdeprscan和jdeps无法警告使用反射来访问封装的API，因此修改规则是有意义的，以便它尝试找出它们尝试使用哪种API。结果可能并不完美，但是为什么不呢？
  • 有各种各样的JDK实现（来自Oracle，IBM，Red Hat等）。通常，它们彼此兼容。但是内部JDK API有多少不同？毕竟，开发人员可以绑定到它，这可能在将应用程序从一个JDK迁移到另一个JDK时导致潜在的问题。


这些都是研究问题，最终结果是什么，时间就会证明=）如果您知道此诊断的有趣场景，并希望在PVS-Studio中看到它们，请给我们写信。



结论



寻找潜在的不兼容错误与我们的思想完全一致-使用PVS-Studio在代码编写的早期阶段查找并修复错误。就像输入错误一样，可以随时在代码中输入对特定功能的调用，因此，在项目上定期运行PVS-Studio的效率现在提高了两倍。从版本7.08开始，V6078



诊断程序可在分析器中使用。您可以在下载页面上的项目中下载并尝试使用分析仪。
