2019年にSEGA Mega Drive / Genesis用のゲームを作成する


今日でも、人々はまだ古いコンソール用の新しいゲームを作成し続けています。私たちはそれらを自作と呼んでいます。時々これは、子供が遊んだコンソール用のゲームを作成するという子供の頃の夢を実現する方法です。しかし、それに加えて、デザイナーやゲーム開発者にとっても興味深い作業です。レトロアイアンには、作者の創造性に挑戦する多くの制限がありました。 90年代のプロの開発者にとって、これらの制限はよく知られていました。今日、私たちがより高度なツールを手に入れると、そのようなマシン用のゲームの作成ははるかに手頃な価格になりました。

Unity、Unreal Engine、Godotを使用できますか？


申し訳ありませんが、これらのエンジンはゲームをSEGA Mega Drive / Genesisにエクスポートできません（たぶん今のところでしょうか？）。ただし、心配しないでください。これらの人気のある最新のエンジンと同じくらい便利な別のツールがあり、同時にメガドライブ/ジェネシス専用に開発されたツール：SGDKもあります。これは、このマシンのコードをC言語で記述できるフレームワークであり、90年代に必須であったアセンブリ言語の学習よりもはるかに単純で高速です。しかし、それだけではありません。 SGDKには、グラフィックとサウンドの作成を簡素化するツールもあります。お気に入りの2Dグラフィックエディター（PhotoshopからGIMPまたはAsespriteまで）でグラフィックを作成できます。、ハードウェアの制限に準拠している場合は自動的に変換されます（これについては以下で詳しく説明します）。同じことがサウンドにも当てはまります。.wavファイルでサウンドエフェクトを作成し、音楽に別のトラッカーを使用できます。最後にもう1つ、SGDKには強力な「スプライトエンジン」があり、画面に動画を表示する技術的なタスクのほとんどを簡素化します。しかし、実際にはどれほど「単純」なのでしょうか。この記事では、自分の職歴についてお話します。

ブラスト処理がシーンに表示されます


古いシステムのプログラマーが言うように、アセンブラーの代わりにCでプログラムを作成すると、ゲームのパフォーマンスが低下します。そしてそれは本当です。しかし、さまざまなレトロコンソールのコストは大きく異なります。それらは、使用されているCPUと、そのようなヴィンテージプロセッサ用のコードを作成する最新のコンパイラの効率に依存します。幸いなことに、Mega Drive / Genesisは、C言語の機能を使用するのに非常に適したMotorola 68000プロセッサを使用しています。これは、他のCPU、たとえば65816 SNESコンソールと比較して特に当てはまります。それ自体は強力ですが、SNES CPUがCで記述されたプログラムを十分な速度で実行することは困難です。つまり、C自作の開発に関しては、「ブラスト処理」はマーケティングの神話ではなく、現実です。

グラフィカルな制限


計算能力は問題ではないので、自作の開発者にとってのGenesis / Mega Driveコンソールの主なボトルネックは、グラフィックの制限になっています。より具体的には、彼女が表示する色の数が限られています。一般に、コンソールは512の一意の色を生成できますが、同時に画面に表示できる色は64色までです。ああ、彼女。比較のために：SNESは、32768個のユニークな色のパレットから256色を同時に表示できました。つまり、グラフィックアーティストは、SEGAゲームを競合他社のコンソールと同じように見栄えよくするための優れた才能を必要としていました。そして、多くの場合、彼らは成功しました。色の制限にもかかわらず、Mega Drive / Genesisの古いゲームの多くは美しく見えます。コミックスゾーン、トールストーリー、ストリートオブレイジII、レンジャーX、ガンスターヒーローズ、ソニック1-2-3など

これを念頭に置いて、お気に入りのグラフィックエディターを使用して画像を作成できますが、色を選択するときは十分注意してください。装置でフラッシュされた512色のパレットと一致しない場合、これは問題ではありません。SGDKは画像をMega Drive / Genesisパレットに自動的に変換します。しかし、画面上で使用および表示される色の総数については、自分で行う必要があります。

オブジェクトを動かす


画面に何かを表示するには、ROMからビデオRAMにグラフィックデータをコピーする必要があります。GPUはROM内のデータに直接アクセスできませんが、RAMからデータを読み取ることができます。したがって、スプライトの背景やアニメーションを変更するには、新しいデータを定期的にROMからRAMにコピーする必要があります。また、ビデオRAM（VRAM）の合計数は64 KBに制限されています。ゲーム用ROMのサイズ（512 KBから8 MB）と比較すると、これは非常に小さいです。

ボトルネック＃1：VRAM転送


少数のグラフィック要素を使用して非常に小さなゲームを作成する場合、ゲームの開始時に一度にすべてをメモリにロードできます。ただし、ほとんどのゲームでは、さまざまなアニメーションを表示するために、VRAMのグラフィックを定期的に更新する必要があります。そして通常それは多くの問題を引き起こします！

各フレーム（1秒あたり60回）でROMからVRAMに転送できるグラフィックデータの量には制限があります。具体的には、フレームあたり7524バイト、つまり約7 KBです。ストリートファイターIIのように、巨大なアニメーションスプライトとアニメーションの背景を表示する必要がある場合、この制限が常に干渉します。そして、これは最初の問題にすぎません。

ボトルネック＃2：スプライトの制限


8/16ビットコンソールの場合、もう1つの問題は、機器が表示できるスプライトの総数にあります。通常、2つの制限があります。1つは画面上のスプライトの総数、もう1つは1つの水平行のスプライトの総数です。 NESやマスターシステムなどの8ビットコンソールの場合、これらの制限は非常に厳しく、水平線あたりのスプライト数は8以下で、総数は64スプライトを超えてはなりません。これは、ソニックまたはマリオを64回画面に表示できることを意味しますか？いいえ、「ハードウェアスプライト」を意味します。これは、通常スプライトと呼ばれるものよりも小さいものです。これはNES向けのSuper Mario Brosの例です。単一のMarioスプライトを表示するには、8x8ピクセルのハードウェアスプライトが必要です。


（出典：https : //nesdoug.com/2018/09/05/06-sprites/）

16ビットコンソールでは、スプライトの制限がわずかに緩和されています。Mega Drive / Genesisは、最大80個のハードウェアスプライトを表示できます。ただし、個別のハードウェアスプライトのサイズは、最大32x32（以前は8x8ピクセルのみ）でした。メガドライブ/ジェネシスで「ハードウェアスプライト」をカットする例をいくつか示します。NES上のマリオには8つのスプライトが必要ですが、Sonicは2つのハードウェアスプライトだけを使用していることに気づきましたか？


（出典：https : //www.patreon.com/posts/new-resource-and-26847248）

ボトルネック＃3：スプライトの優先順位


それほど複雑ではなかったかのように、Mega Drive / Genesisにはもう1つの固有の機能があります。それはスプライトチェーンです。 2つのスプライトを重ね合わせるとき、コンソールはどちらを上に表示する必要があるかを見つける必要があります。マスターシステムとほとんどの任天堂コンソール（NES、ゲームボーイ、SNES）では、スプライトは大きなテーブルに設定され、テーブル内のそれらの順序を使用して表示の優先順位が設定されていました。言うまでもありませんが、スプライトの優先順位を変更するには、この大きなテーブルを常に再定義する必要があるため、これはかなり使いにくいツールです。

Mega Drive / Genesisでは、SEGAはより柔軟でシンプルなシステムを使用していました。他のコンソールと同様に、すべてのスプライトには大きなテーブルのインデックス番号がありますが、スプライトの優先順位を設定するためには使用されません。代わりに、各スプライトは、表示される次のスプライトへのリンクを格納します。最終的に、コンソールはスプライト0から始まる一連のスプライトを作成します。各スプライトは、その後に続くものだけを認識します。これにより、すべてのスプライトを大きな「スプライトリンクテーブル」に配置しなくても、スプライトの優先順位を簡単に変更できます。しかし、率直に言って、そのようなシステムは使いやすいですが、スプライトの優先順位を頻繁に変更するゲームを作成するときは、たとえば、ビートアップなどで、それでも面倒です。

特に、SGDKには強力な「スプライトエンジン」もあり、開発者の生活を大幅に促進します。これは、スプライトの表示を最新のフレームワークで行う方法と同様に処理するための関数のセットです。まず、アニメーションの各フレームに必要なハードウェアスプライトの数を気にする必要がなくなりました。SGDKがこれを自動的に行います。さらに、スプライトの優先度を操作する場合、スプライトのチェーンを手動で設定する必要はありません。SGDKは、スプライトごとに設定できる単純な値「スプライトの優先度」を使用してこれを行うため、最新のフレームワークと同様です。最後に、SGDKはVRAM内のすべての送信も自動的に処理します。はい、あなたはそれを正しく理解しました。スプライトエンジンのおかげで、VRAMについて心配する必要がなくなりました。単に表示するようにSGDKに指示することができます「XスプライトYのアニメーション」と彼は技術的に困難なすべての作業を行います。これらのものを手動で操作しなければならなかった場合、それは魔法のようです-すべてがシンプルで明確です！

音


レトロなハードウェアのサウンドもそれ自体が実際のテストであり、各コンソールはこの点でユニークです。 Mega Drive / Genesisには、この作業用に独立したZ80プロセッサーがあり、オーディオ機器に直接接続されています。つまり、コンソール用のサウンドと音楽を作成するには、まず「オーディオドライバー」、つまりZ80プロセッサー用のプログラムを作成する必要があります。 90年代には、各ゲームスタジオが独自のオーディオドライバーを開発し、ゲームごとに調整しました。コンソールの寿命が近づくにつれ、標準化されたドライバーが登場しました。たとえば、欧米の開発者が約100のゲームで使用していた Sega of AmericaのGEMS です。

8/16ビットコンソール用の優れたサウンドを作成することは、ミュージカルだけでなくプログラミングスキルの問題でもあります。優れた音楽またはサウンド効果を備えた一部のゲームは、不良オーディオドライバーによって破損しています。最も有名な例はストリートファイターIIです。このゲームのオーディオドライバーは、すべての音声サンプルを歪めるバグの影響を受けます。その数年後、Stefan Dallongewil （ SGDKの作者であり、認められたウィザード）がこのオーディオドライバーをリバースエンジニアリングして、バグを修正しました。その結果、透明な叫び声「波動拳！」でストリートファイターIIをプレイできるようになりました。これが違いを聞くことができるビデオです：


しかし、自作の開発と最愛のSGDKに戻りましょう。彼は、ゲームの要件に合わせて選択できるオーディオドライバーを選択することができます。たとえば、非常に高品質（たとえば、音声サンプル）で一度に1つのサウンド効果を再生できるPCMドライバーがあります。ただし、音楽トラックと4つのサウンド効果を同時に再生できるXGMドライバーもあります。何よりも、ゲーム中にオーディオドライバーを切り替えることができます。たとえば、30年の任天堂では、スプラッシュスクリーンとゲームの終了時の音声サンプルは、PCMドライバーによって高品質で再生されます。しかし、ゲーム中に同時に複数の効果音を演奏する必要があったので、XGMドライバーを使用しました。

リソース自体については、最新のツールを使用して作成できます。上で述べたように、サウンドエフェクトの場合、自動的に.wavファイルを取得できます。このファイルは、自動的に目的の形式に変換されます。音楽の場合、標準ツールはDeflemaskです。これは、さまざまな8/16ビットコンソールの音響機器用の音楽を作成できるトラッカーです。私のゲームで使用されている作曲は、才能のあるチップチューンアーティストであるMinerscale（Break An Egg）とWarlord（MeteoRain）が書いたため、私は音楽制作の経験があまりありません。どちらもDeflemaskを使用し、作成物をVGM形式にエクスポートしました。SGDKは、独自のオーディオドライバーで使用するために変換できます。

メガドライブ/ジェネシスのゲームを60分で作成しますか？勝負を受けて立つ！


ゲームジャムは、ゲームデザインとゲーム開発のスキルを適用する優れた方法です。限られた時間（通常は48時間または72時間）で、最初からどれだけ優れたゲームを作成できますか？

時間に関しては、最も難しいジャムは0h Game Jamです。その名のとおり、毎年冬に移行する時期に開催されます。時計の夜の02:00にゲームの作成を開始します。 60分間作業した後、時計が03:00から02:00に戻った瞬間にオンラインでゲームをリリースします。昨年私はこのジャムに参加することを決めました。 Nintendonの30年を終えたばかりで、Mega Drive / Genesis用のゲームを作成した経験があり、SGDKのシンプルさに触発されました。しかし、SGDKを十分にシンプルにして、16ビットの自作をわずか60分で作成できるのでしょうか。ここに答えがあります：


60分の作業

正直に言うと、結果はまったく印象的ではありません。ゲームのバランスはひどいです、そして乱数発生器に巨大なバグがあります、それがゲームが5番目の卵の後に通れないようになる理由です。また、背景画像、音声などはありません。それでも、これはMega Drive / Genesisの新しいゲームで、60分でゼロから作成され、実際のコンソールで起動されます。どうだかわかりませんが、私にとっては、これはすでに一種の成果です。

さらに、基本的なゲームプレイのアイデアにはある程度の可能性があったので、私はさらに何時間もゲームに取り組み続けました。その結果、ここでプレイできるかなり面白いアーケードゲームを手に入れました。また、「Nintendon'tの30年間」カートリッジでも利用できます。 物理メディアでのこのリリースに固有の3番目のゲーム-MeteoRainとともに。
