JavaScriptアプリケーションのパフォーマンスを改善するための12のヒント

パフォーマンスは、Web開発者またはWebアプリケーションが直面する最も重要な問題の1つです。過度のロードによってクラッシュするアプリケーション、または古くからロードされているページに満足する人は誰もいません。ウェブサイトのユーザーは、ダウンロードを待ちすぎたり、ページを稼働状態にする準備ができていません。Kissmetricsによると、訪問者の47％は、Webサイトが2秒未満でロードされることを期待しています。読み込みに3秒以上かかると、訪問者の40％がサイトを離れます。



本日の翻訳を公開している資料の著者は、上記の数字を考慮すると、生産性はWeb開発者が常に覚えておくべきことであることが明らかになったと述べています。JSプロジェクトのパフォーマンスを改善するための12のヒントを次に示します。

1.ブラウザのキャッシュメカニズムを使用する


ブラウザを使用してデータをキャッシュするには、主に2つの方法があります。1つ目は、サービスワーカーによって処理されるCache JavaScript JavaScript APIの使用です。2つ目は、通常のHTTPキャッシュです。

スクリプトは、特定のオブジェクトへのアクセスを整理するためによく使用されます。頻繁にアクセスする必要があるオブジェクトへの参照を変数に格納し、オブジェクトへのアクセスを必要とする繰り返し操作でこの変数を使用すると、コードのパフォーマンスを向上させることができます。

2.実行する環境に合わせてコードを最適化する


プログラムに加えられた改善を適切に評価するために、測定を行うことができる一連の環境を形成することをお勧めします。

実際には、たとえばJSエンジンの既存のすべてのバージョンでコードパフォーマンスの調査を実行したり、実行可能なすべての環境に合わせてコードを最適化したりすることはできません。ただし、1つの環境でコードをテストすることもベストプラクティスではないことに注意してください。このようなアプローチでは、結果が歪む可能性があります。したがって、コードが実行される可能性が最も高い一連の環境を作成し、これらの環境でプロジェクトをテストすることが重要です。

3.未使用のJSコードを取り除く


プロジェクトから未使用のコードを削除すると、ブラウザーがスクリプトをロードするのにかかる時間だけでなく、ブラウザーがコードを分析してコンパイルする必要がある時間も改善されます。未使用のコードを取り除くには、プロジェクトの機能に注意を払う必要があります。したがって、ユーザーが操作できない機能を見つけた場合は、プロジェクトから削除することを検討すると同時に、それに関連付けられているJSコードを検討してください。その結果、サイトの読み込みが速くなり、ブラウザーでの作業の準備が速くなります。これは、ユーザーがサイトで体験する印象に有益な効果をもたらします。プロジェクトを分析するときは、たとえば、その構成に含まれている特定のライブラリが誤って含まれている可能性があることに注意してください。使用されていない可能性があります。それを取り除く価値があります。同じことが、最新のブラウザーに既に実装されているものを実装する特定の依存関係の使用にも当てはまります。その結果、この依存関係によって複製された標準のブラウザー機能に切り替えると、不要なコードを取り除くのに役立ちます。

4.メモリを節約する


Webプロジェクトがそのメモリのみを使用し、それがないと絶対に実行できないことを確認するように努力する価値があります。実際のところ、開発者は特定のデバイス上のアプリケーションがどれだけのメモリにアクセスできるかを事前に知ることができません。アプリケーションが不当に大量のメモリを使用すると、ブラウザのJSエンジンのメモリ管理メカニズムの負荷が増大します。特に、これはガベージコレクターに適用されます。ガベージコレクションを頻繁に呼び出すと、プログラムの速度が低下します。これは、プロジェクトのユーザビリティに悪影響を及ぼします。

5.マイナースクリプトに遅延読み込みメカニズムを使用する


ユーザーは、Webページをできるだけ早くロードしたいと考えています。しかし、ページの最初の表示にプロジェクトのJSコード全体が必要になることはほとんどありません。ユーザーが特定のコードをアクティブにするために何らかのアクションを実行する必要がある場合（たとえば、要素をクリックするか、アプリケーションのタブに移動する）、ページと最も重要なリソースを最初にロードした後で、このコードのロードを完了した後に延期することができます。

このアプローチを使用すると、ブラウザーが最初に大量のJSコードをロードしてコンパイルすることを回避できます。つまり、これらの操作を実行する必要が原因でページ出力が遅くなるのを回避できます。最も重要なすべてのダウンロードが完了したら、追加のコードのダウンロードを開始できます。その結果、ユーザーがこのコードを必要とする場合、そのコードはすでに利用可能です。RAILモデルによると、Googleでは、約50ミリ秒の遅延読み込みスクリプトセッションを実行することを推奨しています。このアプローチでは、コードの読み込み操作は、ユーザーによるページの操作に影響を与えません。

6.メモリリークを回避する


アプリケーションでメモリリークが発生すると、ロードされたページがブラウザからメモリを要求するようになります。その結果、このページのメモリ消費は、システム全体のパフォーマンスに悪影響を与えるレベルに達する可能性があります。あなた自身もおそらく同様の問題に直面しました（そしておそらくあなたはそれを気に入らなかったでしょう）。メモリリークが発生したページに、スライダーや「カルーセル」などの画像を表示する手段が含まれている可能性があります。

Chromeデベロッパーツールを使用すると、サイトでメモリリークを分析できます。これは、[パフォーマンス]タブを使用してインジケーターを調べることによって行われます。通常、メモリリークはページから削除されたDOMフラグメントが原因ですが、いくつかの変数に関連しています。これにより、ガベージコレクターがこれらの不要なDOMフラグメントによって占有されているメモリをクリアできなくなります。

7.重いコンピューティングを行う必要がある場合は、Webワーカーを使用します


MDNリソースの資料から、WebワーカーがWebアプリケーションのメインスレッドとは別のバックグラウンドスレッドでコードを実行できることがわかります。このアプローチの利点は、重い計算を別のスレッドで実行できることです。これにより、メインスレッド（通常はユーザーインターフェイスの維持を担当します）がブロックまたはスローダウンすることなく実行できます。

Webワーカーを使用すると、ユーザーインターフェイスストリームをブロックせずに、プロセッサを集中的に使用する計算を実行できます。このテクノロジーを使用すると、新しいスレッドを作成してタスクを割り当てることができます。これは、アプリケーションのパフォーマンスに有益な影響を与えます。このアプローチを使用すると、完了に時間がかかるタスクが他のタスクの実行を妨げることはありません。メインスレッドで同様のタスクを実行すると、他のタスクがブロックされます。

8. DOM要素に複数回アクセスする場合は、その要素へのリンクを変数に保存します


DOM要素への参照を取得すると、処理が遅くなります。要素に複数回アクセスする場合は、その要素へのリンクをローカル変数に保存することをお勧めします。ただし、ここでは、変数に格納されているリンクである要素が後でDOMから削除された場合、その要素へのリンクを変数から削除する必要があることを覚えておくことが重要です。たとえば、変数に値を書き込むことでこれを行うことができますnull。これにより、メモリリークが回避されます。

9.変数が使用されるのと同じスコープで変数を宣言するよう努めます。


JavaScriptは、変数にアクセスしようとする場合、最初にローカルスコープで変数を探します。そこに表示されない場合、可視性のローカル領域が埋め込まれている可視性領域で検索が続行されます。これは、グローバル変数がチェックされるまで発生します。変数をローカルスコープに保存すると、変数へのアクセスが速くなります。

特別な必要なしに、変数を宣言するときにキーワードを使用しないようにしてくださいvar。代わりに使用して、変数と定数、キーワードlet、およびそれぞれを宣言します。const。これらは、ブロックスコープと他のいくつかの便利な機能が異なります。関数内での変数の使用に注意を払い、関数内でアクセスする変数がローカルであることを確認してください。グローバル変数の暗黙の宣言が引き起こす可能性がある問題に注意してください。

10.グローバル変数を使用しないようにします


グローバル変数は、スクリプトの実行中ずっと存在します。ローカルスコープが破棄されると、ローカル変数も破棄されます。したがって、グローバル変数は本当に必要な場合にのみ使用してください。
