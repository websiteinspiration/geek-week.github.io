専用ハースメモリーとOOMキラー介入

また会いましたね！次の記事の翻訳は、特に今月から始まっているKubernetesベースのインフラストラクチャプラットフォームコースの学生向けに用意されています。



最近、いくつかのポッドが絶えずクラッシュし、OSのシステムログにOOM Killerがコンテナープロセスを破棄したというメモを残しています。これがなぜ起こっているのかを理解することにしました。

ハースメモリ制限とcgroupメモリ設定


K3sディストリビューションでテストしてみましょう。123 MiB（123 Mi）の特徴的なメモリ制限でアンダーを作成します。

kubectl run --restart=Never --rm -it --image=ubuntu --limits='memory=123Mi' -- sh
If you don't see a command prompt, try pressing enter.
root@sh:/#

別のコンソールで、uid囲炉裏を見つけます。

kubectl get pods sh -o yaml | grep uid
  uid: bc001ffa-68fc-11e9-92d7-5ef9efd9374c

それが実行されているサーバーでcgroup、uid必要な炉を指定してパラメーターを見つけます。

cd /sys/fs/cgroup/memory/kubepods/burstable/podbc001ffa-68fc-11e9-92d7-5ef9efd9374c
cat memory.limit_in_bytes
128974848

128974848は正確に123 MiB（123 * 1024 * 1024）です。状況は晴れています。Kubernetesでは、メモリ制限はcgroupを通じて設定されることがわかりました。割り当てられたメモリ制限を超えるとすぐに、cgroupはコンテナープロセスの破棄を開始します。

ストレステスト


コマンドコンソールセッションを開いて、炉床のストレステスト用のユーティリティをインストールしましょう。

root@sh:/# apt update; apt install -y stress

同時に、コマンドを使用してシステムログエントリを追跡しますdmesg -Tw。

まず、メモリに100 MBを割り当てて、ストレステストユーティリティを実行します。プロセスは正常に開始されました。

root@sh:/# stress --vm 1 --vm-bytes 100M &
[1] 271
root@sh:/# stress: info: [271] dispatching hogs: 0 cpu, 0 io, 1 vm, 0 hdd

次に、2番目のストレステストを実行します。

root@sh:/# stress --vm 1 --vm-bytes 50M
stress: info: [273] dispatching hogs: 0 cpu, 0 io, 1 vm, 0 hdd
stress: FAIL: [271] (415) <-- worker 272 got signal 9
stress: WARN: [271] (417) now reaping child worker processes
stress: FAIL: [271] (451) failed run completed in 7s

起動により、信号9によって最初のストレステストプロセス（PID 271）が即座に破壊されました。

ノードで使用可能なメモリ量を確認します。

kubectl describe nodes k3s | grep Allocatable -A 5
Allocatable:
 cpu:                1
 ephemeral-storage:  49255941901
 hugepages-1Gi:      0
 hugepages-2Mi:      0
 memory:             2041888Ki

要求されたメモリサイズが指定されていない場合、デフォルトでは制限と同じになります。値を代入すると、次の値が得られますoom_score_adj。1000–123*1024/2041888=938.32これは939、システムログで指定された値に非常に近い値です。（OOM Killerが正確な値939を取得する方法はわかりません。）

したがって、コンテナー内のすべてのプロセスは同じ値oom_score_adjを持っています。OOM Killerコンポーネントは、メモリ使用量に基づいてOOM値を計算し、oom_score_adjスコアに基づいて結果を調整します。そして、最終的には、メモリのほとんどを消費した最初のストレステストのプロセスを破壊します。これは、推定oom_score = 1718に対応する100 MBです。

結論


Kubernetesは、cgroupおよびOOM Killerコンポーネントを介して、ハースのメモリ制限を制御します。OOMオペレーティングシステムとOOMハースの状態を注意深く調整する必要があります。
