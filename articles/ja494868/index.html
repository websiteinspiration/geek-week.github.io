なぜJavaScript IIFEの使用をやめるべきか

JavaScriptの即時に呼び出される関数式（IIFE）関数は、関数を定義した直後に呼び出すことができる構造です。



この記事では、以前のメリットにもかかわらず、IIFEの使用をやめる価値がある理由を理解します。

自律ブロック内で変数を宣言できます


ES6標準が登場して以来、letとconstを使用してブロック内で変数と定数を宣言できます。この標準に加えて、外部からアクセスできない自律ブロックに変数と定数を割り当てることも可能になりました。

例えば：

{
 let x = 1;
}

そうすると、xは外部からアクセスできなくなります。これは明らかに次の点で優れています。

(()=>{
 let x = 1;
})();

ES6がほとんどすべての最新のブラウザーでサポートされるようになったので、変数を外部から分離するためにIIFEの使用を停止する必要があります。変数を分離するもう1つの方法は、モジュールを使用して、変数のサポートに問題がないようにすることです。それらをエクスポートするまで、他のモジュールでは使用できません。

ほとんどすべてのクロージャを取り除くことができます


クローズは、関数がその外部変数を記憶し、それらにアクセスできるメカニズムです。関数内で別の関数を作成して返す場合、返された関数は外部関数の内部にある外部変数にアクセスできます。

たとえば、ここでいくつかの副作用を得ることができます：

const id = (() => {
 let count = 0;
 return () => {
   ++count;
   return `id_${count}`;
 };
})();

繰り返しますが、データを分離するための自律ブロックとモジュールがあるため、この庭全体をフェンスで囲んでも意味がありません。これらすべてを独自のモジュールに入れるだけで、データアクセスについて心配する必要がなくなります。

ベストプラクティスと常識により、可能な限り回避するように指示されているため、クロージャーは副作用を引き起こします。この場合、クリーンではない関数をテストすることが難しくなります。

さらに、これを回避できる場合は、ネストされた関数を生成しないでください。これらの関数を使用すると、コードを使用しない場合よりもコードがわかりにくくなります。
最良の代替策は、それらをモジュールに置き換えることです。

let count = 0;
export const id = () => {
 ++this.count;
 return `id_${count}`
}

上記のコードでは、count変数に同じ宣言があり、id（）関数をエクスポートして、他のモジュールで使用できるようにします。したがって、IIFEを使用せずに、必要に応じてcountとopen id（）を非表示にします。その結果、ネストが減り、別の関数を定義して実行する必要がなくなります。

変数のエイリアスは別の方法で作成できます。


これを書くことができます：

window.$ = function foo() {
 // ...
};(function($) {
 // ...
})(jQuery);

しかし今では、エイリアスを作成するためにIIFEを使用する必要はありません。モジュールを使用すると、変数を別の名前でインポートするだけで、そのエイリアスを作成できます。

そして、ただ書いてください：

import { $ as jQuery } from "jquery";
const $ = () => {};

さらに、グローバルスコープを汚染するため、ウィンドウオブジェクトに新しいプロパティを追加しないでください。

グローバルオブジェクトを簡単に取得できます。


globalThisの登場により、さまざまな環境でグローバルオブジェクトが標準になるため、その名前を気にする必要はありません。

IIFEを使用してグローバルオブジェクトをキャプチャできます。

(function(global) {
 // ...
})(this);

しかし今、これは必要ありません。はい、これがなくても可能になる前は、次の行を記述するだけで済みます。

const globalObj = self || window || global;

より正確には、次のように書くことができます。

const getGlobal = () => {
 if (typeof self !== 'undefined') { return self; }
 if (typeof window !== 'undefined') { return window; }
 if (typeof global !== 'undefined') { return global; }
 throw new Error('unable to locate global object');
};

そして、IIFEの使用時に表示される関数呼び出しとネストを追加することはできません。

縮小を簡単にすることができます


JavaScriptモジュールを使用すると、ファイルを適切に縮小するために、残りのコードをIIFEから分離する必要がなくなります。

Webpack、Browserify、Parcel、Rollupなどはモジュールを適切に処理できるため、これらを使用してよりクリーンなコードを作成する必要があります。

結論


コードでIIFEの使用を停止する時が来ました。これにより、追加の機能と冗長なネストが追加されます。

さらに、これは時代遅れです。IIFEは、JavaScriptでモジュールが登場して広く使用されるようになる前から使用されていました。2020年には、モジュールと自律ブロックを使用してコードを分離する必要があります。

モジュール内の変数への外部アクセスを防ぐために、ブロックスコープを使用できます。
