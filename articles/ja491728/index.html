LaravelによるDRY原理

新しいユーザーの追加を担当する単純なモジュールを考えてみます。

そして彼の例によって、DRY原理のアプリケーションがどのような可能性を開くかがわかります。

私にとって、DRY（Do n't Repeat Yourself）の原則は、常に2つの基本的な定義に組み込まれています。

  1. 知識の複製は常に原則の違反です
  2. コードの重複は常に原則の違反ではありません

最小限のロジックを含むコントローラーから始めましょう。

class UserController
{
    public function create(CreateRequest $request)
    {
        $user = User::create($request->all());
        
        return view('user.created', compact('user'));
    }
}

class UserApiController
{
    public function create(CreateRequest $request)
    {
        $user = User::create($request->all());
        
        return response()->noContent(201);
    }
}

最初の段階では、このようなコードの繰り返しは無害のようです。
しかし、すでに知識の複製があり、知識の複製は禁止されています。
これを行うには、UserServiceクラスでユーザーの作成を一般化します。

class UserService
{
    public function create(array $data): User
    {
        $user = new User;
        $user->email = $data['email'];
        $user->password = $data['password'];
        $user->save();

        return $user;
    }
    
    public function delete($userId): bool 
    {
        $user = User::findOrFail($userId);
 
        return $user->delete();
    }    
}

モデルを操作するすべてのロジックをサービスに移動したので、コントローラー内の重複を取り除きます。しかし、別の問題があります。ユーザーを作成するプロセスを少し複雑にする必要があるとしましょう。

class UserService
{
    protected $blogService;
    
    public function __construct(BlogService $blogService)
    {
        $this->blogService = $blogService;
    }

    public function create(array $data): User
    {
        $user = new User;
        $user->email = $data['email'];
        $user->password = $data['password'];
        $user->save();
        
        $blog = $this->blogService->create();
        $user->blogs()->attach($blog);

        return $user;
    }
    
    //Other methods
}

次第に、UserServiceクラスは成長し始め、膨大な数の依存関係を持つスーパークラスを取得するリスクがあります。

CreateUserシングルアクションクラス


このような結果を回避するために、サービスを単一のアクションのクラスに分割できます。

このクラスの基本要件：

  • 実行するアクションを表す名前
  • 単一のパブリックメソッドがあります（__invokeマジックメソッドを使用します）
  • 必要なすべての依存関係があります
  • 内部のすべてのビジネスルールに確実に準拠し、違反した場合は例外を生成します。

class CreateUser
{
    protected $blogService;

    public function __construct(BlogService $blogService)
    {
        $this->blogService = $blogService;
    }

    public function __invoke(array $data): User
    {
        $email = $data['email'];
        
        if (User::whereEmail($email)->first()) {
            throw new EmailNotUniqueException("$email should be unique!");
        }
        
        $user = new User;
        $user->email = $data['email'];
        $user->password = $data['password'];
        $user->save();

        $blog = $this->blogService->create();
        $user->blogs()->attach($blog);

        return $user;
    }
}

CreateRequetクラスにはすでにメールフィールドチェックがありますが、ここにもチェックを追加するのが合理的です。これにより、ユーザー作成のビジネスロジックがより正確に反映され、デバッグも簡単になります。

コントローラは次の形式を取ります

class UserController
{
    public function create(CreateRequest $request, CreateUser $createUser)
    {
        $user = $createUser($request->all());

        return view('user.created', compact('user'));
    }
}

class UserApiController
{
    public function create(CreateRequest $request, CreateUser $createUser)
    {
        $user = $createUser($request->all());

        return response()->noContent(201);
    }
}

その結果、ユーザーを作成するための完全に分離されたロジックがあります。変更して拡張すると便利です。

次に、このアプローチがもたらす利点を見てみましょう。

たとえば、ユーザーをインポートするタスクがあります。

class ImportUser
{
    protected $createUser;
    
    public function __construct(CreateUser $createUser)
    {
        $this->createUser = $createUser;
    }
    
    public function handle(array $rows): Collection
    {
        return collect($rows)->map(function (array $row) {
            try {
                return $this->createUser($row);
            } catch (EmailNotUniqueException $e) {
                // Deal with duplicate users
            }
        });
    }
}

Collection :: map（）メソッドにコードを埋め込むことで、コードを再利用する機会を得ます。また、メールアドレスが一意でないユーザーのニーズにも対応します。

ドレッシング


それぞれの新しいユーザーをファイルに登録する必要があるとします。

これを行うには、このアクションをCreateUserクラス自体に埋め込むのではなく、Decoratorパターンを使用します。

class LogCreateUser extends CreateUser 
{
    public function __invoke(array $data)
    {
        Log::info("A new user has registered: " . $data['email']);
        
        parent::__invoke($data);
    }
}

次に、使用してLaravel IoCコンテナを、我々は関連付けることができLogCreateUserを持つクラスでCreateUserクラスを、最初は、私たちが第二のインスタンスを必要とするたびに実装されます。

class AppServiceProvider extends ServiceProvider
{

    // ...

    public function register()
    {
        $this->app->bind(CreateUser::class, LogCreateUser::class);
    }

また、構成ファイルの変数を使用してユーザー作成設定を行う機会もあります。

class AppServiceProvider extends ServiceProvider
{

    // ...

    public function register()
    {
         if (config("users.log_registration")) {
             $this->app->bind(CreateUser::class, LogCreateUser::class);
         }
    }

結論


これは簡単な例です。複雑さが増し始めるにつれて、真のメリットが現れ始めます。私たちは常にコードが1か所にあり、その境界が明確に定義されていることを知っています。

次のような利点があります。重複を防ぎ、テスト
を簡素化し、他の設計原則とパターンのアプリケーションへの道を開きます。
