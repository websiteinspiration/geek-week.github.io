ベストプラクティス


私たちのプログラムは、greetと呼ばれる単一のパッケージ内にあり、エンドポイントを実装するためにすべての基本機能を使用します。これは例なので、メモリDAOを使用して、返される値に3つのフィールドを使用します。また、入力に応じて、データベースへの呼び出しを置き換え、目的の挨拶を返すメソッドも用意します。

package greet
type Dao struct {
      DefaultMessage string
      BobMessage string
      JuliaMessage string
}
func (sdi Dao) GreetingForName(name string) (string, error) {
      switch name {
      case "Bob":
            return sdi.BobMessage, nil
      case "Julia":
            return sdi.JuliaMessage, nil
      default:
            return sdi.DefaultMessage, nil
      }
}

次はビジネスロジックです。これを実装するには、出力データを格納するための構造、データ検索レベルでビジネスロジックが探しているものを記述するためのGreetingFinderインターフェイス、およびビジネスロジック自体をGreetingFinderのフィールドで格納するための構造を定義します。実際のロジックは単純です。GreetingFinderを呼び出し、発生する可能性のあるエラーを処理するだけです。

type Response struct {
      Message string
}
type GreetingFinder interface {
      GreetingForName(name string) (string, error)
}
type Service struct {
      GreetingFinder GreetingFinder
}
func (ssi Service) Greeting(name string)(Response, error) {
      msg, err := ssi.GreetingFinder.GreetingForName(name)
      if err != nil {
            return Response{}, err
      }
      return Response{Message: msg}, nil
}

次に、Webレイヤーが来ます。この部分では、必要なすべてのビジネスロジックを提供するGreeterインターフェイスと、Greeterで構成されたhttpハンドラーを含む構造を定義します。次に、http.Handlerインターフェイスを実装するメソッドを作成します。これは、httpリクエストを分割し、greeter（ウェルダー）を呼び出し、エラーを処理し、結果を返します。

type Greeter interface {
      Greeting(name string) (Response, error)
}
type Controller struct {
      Greeter Greeter
}
func (mc Controller) ServeHTTP(rw http.ResponseWriter,
                               req *http.Request) {
      result, err := mc.Greeter.Greeting(
                               req.URL.Query().Get("name"))
      if err != nil {
            rw.WriteHeader(http.StatusInternalServerError)
            rw.Write([]byte(err.Error()))
            return
      }
      rw.Write([]byte(result.Message))
}

これで挨拶パッケージの終わりです。次に、「良い」Go開発者がこのパッケージを使用するmainをどのように作成するかを見ていきます。メインパッケージでは、実行する必要のあるプロパティを含むConfigと呼ばれる構造を定義します。次にメイン関数は3つのことを行います。

  • まず、シンプルなライブラリ（https://github.com/evil-go/good-sample/blob/master/config/config.go）を使用して設定ファイルからプロパティを読み込み、それらを配置するloadProperties関数を呼び出します設定のコピーで。構成のロードが失敗した場合、main関数はエラーを報告して終了します。
  • 次に、メイン関数はgreetパッケージのコンポーネントをバインドし、configからそれらに値を明示的に割り当て、依存関係を設定します。
  • 3 番目に、小さなサーバーライブラリ（https://github.com/evil-go/good-sample/blob/master/server/server.go）を呼び出し、アドレス、HTTPメソッド、http.Handlerをエンドポイントに渡します。リクエスト処理。ライブラリ呼び出しにより、Webサービスが起動します。そして、これが私たちのアプリケーション全体です。

package main
type Config struct {
      DefaultMessage string
      BobMessage string
      JuliaMessage string

func main() {
      c, err := loadProperties()
      if err != nil {
            fmt.Println(err)

例はかなり短いですが、Goがどのように書かれているかを示しています。あいまいなものもありますが、一般的に何が起こっているかは明らかです。一緒に動作するように特別に設定された小さなライブラリを接着します。何も隠されていません。誰でもこのコードを取得し、その部分がどのように接続されているかを理解し、必要に応じてそれらを新しいものにやり直すことができます。

黒点


次に、秋とアウトボーイのバージョンを検討します。まず、greetパッケージをいくつかのパッケージに分割します。それぞれのパッケージには、1つのアプリケーション層が含まれています。こちらがDAOパッケージです。それは私たちのDIフレームワークであるFallをインポートし、私たちは「悪」であり、インターフェースとの関係を逆に定義しているので、GreetDaoと呼ばれるインターフェースを定義します。注意してください-エラーへのリンクをすべて削除しました。何かがおかしいとしたら、パニックに陥ります。この時点で、私たちはすでに不十分なパッケージング、不良なインターフェース、不良なバグを抱えています。素晴らしいスタートです！

良い例から少し名前を変更しました。フィールドにstructタグが追加されました。これらは、Fallがフィールドに登録された値を設定するために使用されます。また、「悪の力」を蓄積するパッケージのinit関数もあります。パッケージのinit関数で、Fallを2回呼び出します。

  • 一度、構造タグの値を提供する設定ファイルを登録します。
  • もう1つは、構造体のインスタンスへのポインターを登録することです。Fallはこれらのフィールドに入力して、DAOを他のコードで使用できるようにします。

package dao
import (
      "github.com/evil-go/fall"
)
type GreetDao interface {
      GreetingForName(name string) string
}
type greetDaoImpl struct {
      DefaultMessage string `value:"message.default"`
      BobMessage string `value:"message.bob"`
      JuliaMessage string `value:"message.julia"`
}
func (gdi greetDaoImpl) GreetingForName(name string) string {
      switch name {
      case "Bob":
            return gdi.BobMessage
      case "Julia":
            return gdi.JuliaMessage
      default:
            return gdi.DefaultMessage
      }
}
func init() {
      fall.RegisterPropertiesFile("dao.properties")
      fall.Register(&greetDaoImpl{})
}

サービスパッケージを見てみましょう。そこで定義されたインターフェースにアクセスする必要があるため、DAOパッケージをインポートします。サービスパッケージは、まだ検討していないモデルパッケージもインポートします-データタイプをそこに保存します。そして、すべての「良い」フレームワークと同様に、Fallはどこにでも浸透するため、Fallをインポートします。また、Webレイヤーへのアクセスを提供するサービスのインターフェイスも定義します。繰り返しますが、エラー処理はありません。

私たちのサービスの実装には、ワイヤー付きの構造タグがあります。ワイヤーでマークされたフィールドは、構造が秋に登録されると、その依存関係を自動的に接続します。この小さな例では、このフィールドに何が割り当てられるかは明らかです。しかし、より大きなプログラムでは、このGreetDaoインターフェースがどこかで実装されており、Fallに登録されていることだけがわかります。依存関係の動作を制御することはできません。

次のサービスのメソッドは、モデルパッケージからGreetResponse構造を取得するようにわずかに変更され、エラー処理が削除されています。最後に、秋にサービスインスタンスを登録するパッケージにinit関数があります。

package service
import (
      "github.com/evil-go/fall"
      "github.com/evil-go/evil-sample/dao"
      "github.com/evil-go/evil-sample/model"
)
type GreetService interface {
      Greeting(string) model.GreetResponse
}
type greetServiceImpl struct {
      Dao dao.GreetDao `wire:""`
}
func (ssi greetServiceImpl) Greeting(name string)     model.GreetResponse {
      return model.GreetResponse{Message: ssi.Dao.GreetingForName(name)}
}
func init() {
      fall.Register(&greetServiceImpl{})
}

次に、モデルパッケージを見てみましょう。見るものは何もありません。モデルはそれを作成するコードから分離されており、コードをレイヤーに分割するだけです。

Webパッケージには、Webインターフェイスがあります。ここでは、FallとOutboyの両方をインポートし、Webパッケージが依存するサービスパッケージもインポートします。フレームワークはバックステージで統合された場合にのみ連携して機能するため、FallとOutboyが連携することを確認する特別なコードがFallにあります。また、構造を変更して、Webアプリケーションのコントローラーになるようにします。彼女には2つのフィールドがあります。

  • 1つ目は、Fallを介してサービスパッケージからのGreetServiceインターフェイスの実装に接続されています。
  • 2つ目は、唯一のWebエンドポイントのパスです。このパッケージのinit関数に登録されている構成ファイルからの値が割り当てられます。

HTTPハンドラーはGetHelloに名前が変更され、エラー処理が不要になりました。また、Initメソッド（大文字を使用）もあります。これは、init関数と混同しないでください。Initは、すべてのフィールドに入力した後、Fallに登録された構造に対して呼び出される魔法のメソッドです。Initでは、Outboyを呼び出して、Fallを使用して設定されたパスにコントローラーとそのエンドポイントを登録します。コードを見ると、パスとハンドラーが表示されていますが、HTTPメソッドは指定されていません。Outboyでは、メソッド名は、ハンドラーが応答するHTTPメソッドを決定するために使用されます。このメソッドはGetHelloと呼ばれているため、GET要求に応答します。これらの規則を知らないと、彼がどのような要求に答えるかを理解できなくなります。確かに、これは非常に悪党ですか？

最後に、秋にinit関数を呼び出して、構成ファイルとコントローラーを登録します。

package web
import (
      "github.com/evil-go/fall"
      "github.com/evil-go/outboy"
      "github.com/evil-go/evil-sample/service"
      "net/http"
)
type GreetController struct {
      Service service.GreetService `wire:""`
      Path string `value:"controller.path.hello"`
}
func (mc GreetController) GetHello(rw http.ResponseWriter, req *http.Request) {
      result := mc.Service.Greeting(req.URL.Query().Get("name"))
      rw.Write([]byte(result.Message))
}
func (mc GreetController) Init() {
      outboy.Register(mc, map[string]string{
            "GetHello": mc.Path,
      })
}
func init() {
      fall.RegisterPropertiesFile("web.properties")
      fall.Register(&GreetController{})
}

プログラムの実行方法を示すためだけに残っています。メインパッケージでは、空のインポートを使用してOutboyとWebパッケージを登録します。そして、メイン関数はfall.Start（）を呼び出してアプリケーション全体を起動します。

package main
import (
    _ "github.com/evil-go/evil-sample/web"
    "github.com/evil-go/fall"
    _ "github.com/evil-go/outboy"
)
func main() {
      fall.Start()
}

外皮の破壊


そして、彼女は、私たちの邪悪なGoツールをすべて使用して書かれた完全なプログラムです。悪夢。彼女はプログラムの各部分がどのように組み合わさるかを魔法のように隠し、彼女の仕事を理解することをひどく難しくしています。

それでも、FallとOutboyでコードを書くことに魅力があることを認めなければなりません。小さなプログラムの場合、それは改善だとさえ言えます。設定がいかに簡単かをご覧ください！依存関係を事実上コードなしで接続できます！名前だけを使用して、メソッドのハンドラーを登録しました。そして、エラー処理がなければ、すべてがとてもきれいに見えます！

それが悪の働きです。一見、本当に魅力的です。しかし、プログラムが変化し、成長するにつれて、このすべての魔法が干渉し始めるだけで、何が起こっているのかを理解することが難しくなります。悪に完全に取り憑かれているときだけ、振り返って、閉じ込められていることに気づきます。

Java開発者にとって、これはおなじみのように見えるかもしれません。これらの技法は、多くの一般的なJavaフレームワークで見つけることができます。先に述べたように、私は1996年の1.0.2から20年以上にわたってJavaを使用しています。多くの場合、Javaの開発者は、インターネット時代に大規模なエンタープライズソフトウェアを作成するときに最初に問題に直面しました。サーブレット、EJB、Spring、Hibernateが登場したときのことを覚えています。 Java開発者が当時行った決定は理にかなっています。しかし、何年にもわたって、これらのテクニックは年齢を示しています。 Goなどの新しい言語は、古い手法を使用するときに見られる問題点を解消するように設計されています。ただし、Java開発者がGoを学び、それを使ってコードを書くようになると、覚えておく必要があります。Javaからパターンを再現しようとすると、悪い結果につながります。

Goは、数百人の開発者と数十のチームにまたがるプロジェクトのために、真剣なプログラミングのために設計されました。しかし、Goがこれを行うには、最適な方法で使用する必要があります。私たちは悪か善かを選択できます。悪を選択した場合、Goを理解する前に、若いGo開発者にスタイルとテクニックを変更するように働きかけることができます。または、私たちは良いものを選ぶことができます。Go開発者としての私たちの仕事の一部は、若いGopher（Gopher）を教育し、彼らが私たちのベストプラクティスの根底にある原則を理解できるようにすることです。
