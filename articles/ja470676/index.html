毎日のカウンターを格納する構造
最初のアイデアは、毎日の注文数を格納する構造が必要だということかもしれません。この構造は、固定サイズ（最大出発日で決まる）の配列にすることができます。
入力データ：
-入口= [
1、3、5 ] -出発= [
2、6、10 ] -k = 1

この例では、配列のサイズは10になります（10日目の最後の出口のため）。この配列を満たすために、エントリーと出口のリストを調べ、対応する日のカウンターを増減します。擬似コードの例：

int[] counts = new int[maxDepartures(departures)]
for each arr in arrivals {
 counts[arr]++
}
for each dep in departures {
 counts[dep]--
}

その結果、次の配列が得られます。 配列がいっぱいになったら、配列を調べて、すべての要素が（部屋の数）を超えていないかどうかを確認するだけです。 前の例では、部屋の最大数は1でした。5日目には2つの予約があるため、払い戻しを行います。 このソリューションの時間の複雑さはO（n）で、nは予約の数、空間はO（m）で、mは最大出発日です。理論的には悪くありませんが、大部分の配列は使用されませんが、大きな配列に多くのメモリを割り当てる可能性があります。 例えば：

解決策は、これらのイベントを通過してカウンターを増減することです。ある時点でカウンターが大きくなったk場合は、戻りfalseます。ただし、繰り返し処理を行うには、このイベントのコレクションを並べ替える必要があります。

ここでどの構造を使用するのが良いですか？要件を要約しましょう：

  • 検索して、その日がすでに存在するかどうかを確認します
  • 新しい日を追加し、
  • ソートされた日ごとに繰り返す構造を表示します。

時間の複雑さの観点から結果を見てみましょう：

  • そのような日がすでに存在するかどうかを確認するために検索します。平均的なケースではO（log（n））、最悪のケースではO（n）、
  • 新しい日を追加：平均的なケースではO（log（n））、最悪のケースではO（n）、
  • ソートされた各日を反復するための構造を表示します。深さ検索を使用してO（n）。

各要素を並べ替えてツリーに挿入する必要があるため、アルゴリズムの複雑さは、平均的な場合はO（n log（n））、最悪の場合はO（n²）です。

別のオプションは、ハッシュテーブルを使用して、すべてのイベントを追加した後でキーをソートすることです。

  • そのような日がすでに存在するかどうかを確認するために検索します。平均的な場合はO（1）、最悪の場合はO（n）（確率は連想配列の容量に依存します）、
  • 新しい日を追加：平均的な場合はO（1）、最悪の場合はO（n）、
  • ソートされた日ごとに反復するための構造を表示します。ソートキーの場合はO（n log（n））、ソートの場合はO（n）です。

結局のところ、ソリューションにはO（n log（n））が中央にあり（ソート操作による）、最悪の場合はO（n²）です。このソリューションは、ツリーベースのソリューションと同じ複雑さを持っているようです。

ソートされた連想配列を使用して、Javaで可能な実装を見てみましょう。

public boolean hotel(ArrayList<Integer> arrivals, ArrayList<Integer> departures, int k) {
  //  
  Map<Integer, Integer> events = new HashMap<>();

  //  
  int n = arrivals.size();

  for (int i = 0; i < n; i++) {
    int arrival = arrivals.get(i);
    int departure = departures.get(i);

    //     
    Integer current = events.get(arrival);
    events.put(arrival, current == null ? 1 : current + 1);

    //     
    current = events.get(departure);
    events.put(departure, current == null ? -1 : current - 1);
  }

  //   
  Map<Integer, Integer> sortedEvents = new TreeMap<>(events);

  int count = 0;
  for (Map.Entry<Integer, Integer> entry : sortedEvents.entrySet()) {
    count += entry.getValue();

    //  count    
    if (count > k) {
      return false;
    }
  }

  return true;
}

一定の空間的複雑さ


アルゴリズムを最適化したい場合、これらすべてのイベントを保存することが本当に必要かどうかを考える必要がありますか？コレクションデータ（エントリと出口）を反復処理して、プロセスの予約制限を確認することはできませんか？

解決策は可能ですが、そのためには、事前にソートして入力データを単純化する必要があります。

両方のコレクションが並べ替えられている場合は、2つのポインタ（1つは入口、もう1つは出口）を使用して各要素を繰り返し処理し、その場で制限を確認できます。

ご覧のとおり、各反復中arrivals.get(indexArrival) departures.get(indexDeparture)に、更新する必要のあるポインターを見つけるために、最小値を確認する必要があります。

一般に、アルゴリズムは、ソート操作により、時間的および空間的に一定の複雑度O（n log（n））を持ちます。
