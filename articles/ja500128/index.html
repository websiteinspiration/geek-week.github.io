Androidアプリケーションのアーキテクチャと設計（私の経験）

こんにちは、こんにちは！

今日は、Androidアプリケーションで使用しているアーキテクチャについてお話ししたいと思います。私はクリーンアーキテクチャをベースとして、Androidアーキテクチャコンポーネント（ViewModel、LiveData、LiveEvent）+ Kotlinコルーチンをツールとして使用しています。添付されているのは、GitHubで入手できる架空のサンプルコードです。

免責事項


私の開発経験を共有したいのですが、私の解決策が唯一の真実で欠陥のないものであると偽ることはありません。アプリケーションのアーキテクチャは、特定の問題を解決するために選択する一種のモデルであり、選択したモデルでは、特定のタスクに適用する際の妥当性が重要です。

問題：なぜアーキテクチャが必要なのですか？


私が参加したほとんどのプロジェクトには同じ問題があります。アプリケーションロジックがAndroid環境内に配置されているため、フラグメントとアクティビティ内に大量のコードが含まれています。したがって、コードはまったく必要のない依存関係に囲まれているため、単体テストはほぼ不可能になり、再利用も不可能になります。時間の経過とともにフラグメントは神のオブジェクトになり、小さな変更でもエラーが発生し、プロジェクトのサポートは高価になり、感情的にも高価になります。

アーキテクチャをまったく持たないプロジェクトがあり（すべてが明確で、質問はありません）、アーキテクチャを主張するプロジェクトがありますが、いずれにしてもまったく同じ問題が発生します。AndroidでClean Architectureを使用するのはおしゃれです。Clean Architectureは、リポジトリと、これらのリポジトリを呼び出し、他に何もしないスクリプトの作成に限定されていることをよく見ました。さらに悪いことに、そのようなスクリプトは呼び出されたリポジトリからモデルを返します。そして、そのようなアーキテクチャではまったく意味がありません。そして、スクリプトは必要なリポジトリを呼び出すだけなので、ロジックはViewModelに依存していることが多く、さらに悪いことに、フラグメントとアクティビティに落ち着いています。これはすべて、混乱になり、自動テストに対応できなくなります。

建築とデザインの目的


アーキテクチャの目的は、ビジネスロジックを詳細から分離することです。詳細には、たとえば、外部API（RESTサービス用のクライアントを開発するとき）、Android-環境（UI、サービス）などを指します。コアでは、クリーンアーキテクチャを使用しますが、実装を前提としています。

設計の目的は、UI、API、ビジネスロジック、モデルを結び付けることです。これにより、これらすべてが自動テストに役立ち、疎結合され、簡単に拡張できます。デザインでは、Androidアーキテクチャコンポーネントを使用します。

コマンド

UIは、コマンドを使用してスクリプトの実行を開始します。私のプロジェクトでは、コマンドの独自の実装を使用していますが、それらはアーキテクチャコンポーネントやその他のものの一部ではありません。一般に、それらの実装は単純です。アイデアをより深く理解すると、reactiveui.netでのコマンドの実装を確認できますc＃のため。残念ながら、私は実際のコードをレイアウトすることはできず、例の簡略化された実装だけをレイアウトできます。

コマンドの主なタスクは、スクリプトを実行して入力パラメーターをスクリプトに渡し、実行後にコマンドの結果（データまたはエラーメッセージ）を返すことです。通常、すべてのコマンドは非同期で実行されます。さらに、チームはバックグラウンド計算方法をカプセル化します。私はコルーチンを使用していますが、RXで置き換えるのは簡単です。これを行うのは、コマンド+ユースケースの抽象的な束で行うだけです。おまけとして、チームはそのステータスを報告できます。現在実行されているかどうか、および原則として実行できるかどうかです。チームは、二重呼び出しの問題（操作の実行中にユーザーがボタンを数回クリックしたとき）や、表示とキャンセルの問題など、いくつかの問題を簡単に解決します。

例


実装機能：ログインとパスワードを使用してアプリケーションにログインします。
ウィンドウには、ログインとパスワードの入力フィールドと「ログイン」ボタンが含まれている必要があります。作業のロジックは次のとおりです。

  1. ユーザー名とパスワードが4文字未満の場合、「ログイン」ボタンは非アクティブになります。
  2. ログインプロセス中は、[ログイン]ボタンを非アクティブにする必要があります。
  3. ログイン手順中に、インジケーター（ローダー）が表示されます。
  4. ログインが成功すると、ウェルカムメッセージが表示されます。
  5. ログインやパスワードが正しくない場合は、ログインフィールドの上にエラーメッセージが表示されます。
  6. 画面にエラーメッセージが表示された場合、ログインフィールドまたはパスワードフィールドに文字を入力すると、次の試行までこのメッセージは削除されます。

この問題は、たとえばMainActivityにすべてを入れるなど、さまざまな方法で解決できます。
しかし、私は常に2つの主要なルールの実装に従います。

  1. ビジネスロジックは詳細から独立しています。
  2. UIは可能な限りシンプルです。彼は自分のタスクのみを扱います（彼に転送されたデータを提示し、ユーザーからのコマンドをブロードキャストします）。

アプリケーションは次のようになります



。MainActivityは次のようになります。

class MainActivity : AppCompatActivity() {

   private val vm: MainViewModel by viewModel()

   override fun onCreate(savedInstanceState: Bundle?) {
       super.onCreate(savedInstanceState)
       setContentView(R.layout.activity_main)

       bindLoginView()
       bindProgressBar()

       observeAuthorization()
       observeRefreshView()
   }

   private fun bindProgressBar() {
       progressBar.bindVisibleWithCommandIsExecuting(this, vm.loginCommand)
   }

   private fun bindLoginView() {
       loginEdit.bindAfterTextChangedWithCommand(vm.loginValidityCommand)
       passwordEdit.bindAfterTextChangedWithCommand(vm.passwordValidityCommand)

       loginButton.bindCommand(this, vm.loginCommand) {
           LoginParameters(loginEdit.text.toString(), passwordEdit.text.toString())
       }
   }

   private fun observeAuthorization() {
       vm.authorizationSuccessLive.observe(this, Observer {
           showAuthorizeSuccessMsg(it?.data)
       })
       vm.authorizationErrorLive.observe(this, Observer {
           showAuthorizeErrorMsg()
       })
   }

   private fun observeRefreshView() {
       vm.refreshLoginViewLive.observe(this, Observer {
           hideAuthorizeErrorMsg()
       })
   }

   private fun showAuthorizeErrorMsg() {
       loginErrorMsg.isInvisible = false
   }

   private fun hideAuthorizeErrorMsg() {
       loginErrorMsg.isInvisible = true
   }

   private fun showAuthorizeSuccessMsg(name : String?) {
       val msg = getString( R.string.success_login, name)
       Toast.makeText(this, msg, Toast.LENGTH_LONG).show()
   }
}

アクティビティは非常に単純で、UIルールが実行されます。bindVisibleWithCommandIsExecutingなどの簡単な拡張機能をいくつか作成して、コマンドをUI要素に関連付け、コードを重複させないようにしました。
