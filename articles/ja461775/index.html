DjangoアプリケーションでCeleryを使用する3つのケース



DjangoでWebアプリケーションを作成しています。基本的に、これらはビジネス向けのSaaSサービスです。これらのアプリケーションはすべて非同期タスクを必要とします。それらの実装には、Celeryを使用します。この記事では、Celeryを使用する状況について、コード例とともに説明します。

セロリは、タスクキューを管理するためのシステムです。基本的に2つのことができます。キューからタスクを取得し、スケジュールに従ってタスクを実行します。キューブローカーは通常、RabbitMQまたはRedisです。タスクはキューに入れられ、セロリワーカーがそこからタスクを取り出して実行します。

Celeryの場合、ほとんどすべてのアプリケーションでアプリケーションを考えることができますが、それでは、私が自分で使用する場合についてのみ説明します。

1.スケジュールされたタスク


多くの場合、特定の日時に完了する必要のあるタスクがあります。ユーザーにリマインダーを送信し、アカウントの試用期間を終了し、ソーシャルネットワークに投稿を公開します。

Celeryでは、タスクを呼び出すときにETAパラメーター（タスクを起動する必要がある時間）を指定できます。しかし、この方法でタスクを計画すると、非常に信頼性が低くなることがわかります。タスクが開始されない場合があり、キャンセルするのは不便です。

より信頼できる方法は、celerybeatスケジュールを使用することです。つまり、特定の頻度または特定の時間に開始されるタスクがあるスケジュールを作成します。たとえば、スケジュールに従ってソーシャルネットワークに投稿を公開する必要がある場合、このタスクは1分に1回起動されます。アカウントの試用期間を終了する必要がある場合は、1日に1回タスクを実行できます。

# schedule.py
from datetime import timedelta
from celery.schedules import crontab


CELERYBEAT_SCHEDULE = {
   'publish_post_starter': {
       'task': 'publish_post_starter',
       'schedule': timedelta(minutes=1),
   },
   'end_trial_starter': {
     'task': 'end_trial_starter',
     'schedule': crontab(hour=10, minute=21),
  },
}

タスクスターターでは、計画された時間がすでに経過しているすべてのインスタンスを取得します。インスタンスを確認し、それぞれについてメインタスクを呼び出します。引数として、不要なデータでキューを詰まらせないように、インスタンスのIDのみを渡します。すべてのインスタンスをすぐに通過してアクションを実行できますが、ほとんどの場合、インスタンスごとに個別のタスクを呼び出す方が適切です。したがって、実行を高速化し、エラーが発生した場合、影響を受けるのはタスクの1つだけです。

# tasks.py
@app.task(name='publish_post')
def publish_post(post_id):
   ...

@app.task(name='publish_post_starter')
def publish_post_starter():
   post_ids = list(
       Post.objects.filter(
           publish_dt__lte=timezone.now(),
           is_published=False
       ).values_list('id', flat=True)
   )

   for post_id in post_ids:
       publish_post.delay(post_id)

2. WSGIからの長いコンピューティングとAPI呼び出し


WSGIは、ユーザーからの要求が処理されるコンテキスト（要求/応答サイクル）を指します。非同期タスクのコンテキストとは対照的に-セロリ。

レスポンシブインターフェイスを作成するには、すべてのボタンが即座に応答し、残りのインターフェイスをブロックしないようにする必要があります。これを行うには、ボタンのクリックがブロックされた後、スピナーがその上に配置され、ajaxリクエストがサーバーに送信されます。リクエストの処理に数秒以上かかる場合は、計算をCeleryタスクに移動できます。

WSGIでは、タスクを呼び出して応答を返します。前面で、ボタンのロックを解除してスピナーを取り外します。アクションが実行されているというメッセージをユーザーに表示します。並行して、Celeryタスクが実行され、完了すると、Webソケットに応答が返されます。フロントで結果を受け取ったら、それをユーザーに示します。

# rest_views.py
from rest_framework import status
from rest_framework.views import APIView
from rest_framework.response import Response

from tasks import send_emails


class SendEmailView(APIView):
   def post(self, request):
       # this id will be used to send response with websocket
       request_uuid = request.data.get('request_uuid')
       if not request_uuid:
           return Response(status=status.HTTP_400_BAD_REQUEST)
      
       send_emails.delay(request.user.id, request_uuid)
       return Response(status=status.HTTP_200_OK)

これとは別に、外部API呼び出しとWSGIを区別できます。この場合、実行時間に関係なく、すべての呼び出しはCeleryタスクを通じて起動されます。これは愚か者からの保護です。一部の外部APIにアクセスできないために、ユーザーインターフェイスがフリーズするという状況はありません。

3.竜巻の課題


ソーシャルネットワーク、テレグラム、または支払いサービスと統合する場合、通知が送信されるWebhookのURLが必要です。リクエスト数は常に事前に計算できるとは限りませんが、ユーザーからのリクエスト数を超える可能性があります。これらの要求は、コード200の応答を受信するまで受信され

ます。非同期のTornadoフレームワークは、このような要求の処理に適しています。トルネードで処理を同期に変換しないようにするために、ブロッキング操作があってはなりません。これはセロリが必要な場所です。Tornadoハンドラーは要求を受信し、データを検証し、Celeryタスクを呼び出し、成功した応答を返します。

# tornado_handlers.py
from tornado import gen, escape
from tornado.web import RequestHandler

from tasks import handle_vk_callback


class VkCallbackHandler(RequestHandler):
   @gen.coroutine
   def post(self, *args, **kwargs):
       try:
           data = escape.json_decode(self.request.body)
       except ValueError:
           self.set_status(status_code=400, reason='Invalid data')
           return

       handle_vk_callback.delay(data)
       self.write('ok')
       return
