複数のクエリを使用する方法。組成、レジューサー、FP

こんにちは、ハブル。私の名前はマキシムです。私はFINCHのiOS開発者です。今日は、私たちの部署で開発した関数型プログラミングの使用方法をいくつか紹介します。

どこでも関数型プログラミングを使用するように勧めるわけではありません-これはすべての問題の万能薬ではありません。しかし、私には、FPが非標準の問題に対して最も柔軟でエレガントなソリューションを提供できる場合があるようです。

FPは人気のあるコンセプトなので、基本的な説明は省略します。私はあなたのプロジェクトですでにmap、reduce、compactMap、first（where :)および同様のテクノロジーを使用していると思います。この記事では、複数のクエリの問題の解決とレデューサーの使用に焦点を当てます。

複数クエリの問題


私はプロダクションのアウトソーシングで働いており、下請け業者と一緒にクライアントがバックエンドの作成を担当する状況があります。これは最も便利なバックエンドとはほど遠く、複数の並列クエリを実行する必要があります。

時々私は次のようなものを書くことができました：

networkClient.sendRequest(request1) { result in
            
    switch result {
          
    case .success(let response1):
                // ...
        self.networkClient.sendRequest(request2) { result in
                    // ...
             switch result {
                    
             case .success(let response2):
                   // ...  -     response 
                   self.networkClient.sendRequest(request3) { result in
                            
                       switch result {
                       case .success(let response3):
                           // ...  -    
                           completion(Result.success(response3))
                            
                       case .failure(let error):
                           completion(Result.failure(.description(error)))
                       }

                   }

             case .failure(let error):
                 completionHandler(Result.failure(.description(error)))
             }

         }

    case .failure(let error):
        completionHandler(Result.failure(.description(error)))
    }
}

嫌だよね？しかし、これが私が取り組む必要があった現実です。

承認のために3つの連続したリクエストを送信する必要がありました。リファクタリング中、各リクエストを個別のメソッドに分割し、完了時にそれらを呼び出すことで、1つの巨大なメソッドをアンロードすることをお勧めします。それは次のようなことがわかりました：


func obtainUserStatus(completion: @escaping (Result<AuthResponse>) -> Void) {
        
        let endpoint= AuthEndpoint.loginRoute
        
        networkService.request(endpoint: endpoint, cachingEnabled: false) { [weak self] (result: Result<LoginRouteResponse>) in
            
            switch result {
                
            case .success(let response):
                self?.obtainLoginResponse(response: response, completion: completion)

            case .failure(let error):
                completion(.failure(error))
                
            }
        }
        
    }

    private func obtainLoginResponse(_ response: LoginRouteResponse, completion: @escaping (Result<AuthResponse>) -> Void) {
        
       let endpoint= AuthEndpoint.login
        
        networkService.request(endpoint: endpoint, cachingEnabled: false) { [weak self] (result: Result<LoginResponse>) in
            
            switch result {
                
            case .success(let response):
                self?.obtainAuthResponse(response: response, completion: completion)

            case .failure(let error):
                completion(.failure(error))
                
            }
        }

private func obtainAuthResponse(_ response: LoginResponse, completion: @escaping (Result<AuthResponse>) -> Void) {
        
       let endpoint= AuthEndpoint.auth
        
        networkService.request(endpoint: endpoint, cachingEnabled: false) { (result: Result<AuthResponse>) in
            
    	completion(result)
        }

}

プライベートメソッドのそれぞれでプロキシする必要があることがわかります 

completion: @escaping (Result<AuthResponse>) -> Void

そして私は本当にそれが好きではありません。

次に、「なぜ関数型プログラミングに頼らないのか」という考えが浮かびました。さらに、swiftは、その魔法と構文上の糖質により、興味深いコードで個々の要素にコードを分解することができます。

構成および減力剤


関数型プログラミングは、構成の概念（混合、組み合わせ）と密接に関連しています。関数型プログラミングでは、コンポジションは個々のブロックの動作を組み合わせて、将来的にはそれで動作することを前提としています。

数学的な観点からの構成は次のようなものです：

func compose<A,B,C>(_ f: @escaping (A) -> B, and g: @escaping (B) -> C) -> (A) -> C {
    return { a in g(f(a)) }
}

内部的に出力および入力パラメーターを指定する関数fおよびgがあります。これらのインプットメソッドからある種の結果の振る舞いを得たいと思っています。

例として、2つのクロージャを作成できます。1つは入力数を1増やし、2つ目はそれ自体で乗算します。

let increment: (Int) -> Int = { value in
    return value + 1
}
 
let multiply: (Int) -> Int = { value in
    return value * value
}

結果として、これらの操作の両方を適用したいと思います。


let result = compose(multiply, and: increment)
 
result(10) //     101


残念ながら、私の例は連想的ではありません
（インクリメントを入れ替えて乗算すると、数値121になります）が、ここではこの瞬間を省略します。


let result = compose(increment, and: multiply)
 
result(10) //     121

PS私は具体的には、可能な限り明確になるように私の例をより単純にすることを試みます）

実際には、多くの場合、次のようなことをする必要があります：


let value: Int? = array
        
        .lazy
        .filter { $0 % 2 == 1 }
        .first(where: { $0 > 10 })

これが作曲です。入力アクションを設定し、出力効果を取得します。しかし、これはいくつかのオブジェクトの追加だけではありません-これは全体の動作の追加です。

次に、より抽象的に考えてみましょう:)


このアプリケーションでは、ある種の状態があります。これは、ユーザーが現在表示している画面、またはアプリケーションに保存されている現在のデータなどです。
さらに、アクションがあります。これは、ユーザーが実行できるアクションです（ボタンをクリックする、コレクションをスクロールする、アプリケーションを閉じるなど）。その結果、私たちはこれら2つの概念を操作し、それらを相互に接続します。つまり、結合します（これは以前聞いたところです）。

しかし、私の状態とアクションを組み合わせるだけのエンティティを作成するとどうなるでしょうか。

だから、Reducer


struct Reducer<S, A> {
    let reduce: (S, A) -> S
}

現在の状態とアクションをreduceメソッドの入力に渡し、出力でreduceの内部で形成された新しい状態を取得します。

この構造は、いくつかの方法で説明できます。新しい状態を定義するか、関数法を使用するか、または可変モデルを使用します。


struct Reducer<S, A> {
    let reduce: (S, A) -> S
}

struct Reducer<S, A> {
    let reduce: (S) -> (A) -> S
}

struct Reducer<S, A> {
    let reduce: (inout S, A) -> Void
}

最初のオプションは「クラシック」です。

2つ目はより機能的です。重要なのは、状態を返すのではなく、アクションを実行するメソッドです。このメソッドはすでに状態を返します。これは基本的に、reduceメソッドのカレーです。

3番目のオプションは、参照によって状態を操作することです。このアプローチでは、状態を発行するだけでなく、入ってくるオブジェクトへのリンクを操作します。このような（変更可能な）モデルは悪いため、この方法はあまり良くないように思えます。新しい状態（インスタンス）を再構築して返すことをお勧めします。ただし、単純化してさらに例を示すために、後者のオプションを使用することに同意します。

レデューサーの適用


Reducerの概念を既存のコードに適用します。RequestStateを作成し、それを初期化して設定します。


class RequestState {

    // MARK: - Private properties
    
    private let semaphore = DispatchSemaphore(value: 0)
    private let networkClient: NetworkClient = NetworkClientImp()
    
    // MARK: - Public methods
    
    func sendRequest<Response: Codable>(_ request: RequestProtocol, completion: ((Result<Response>) -> Void)?) {
        
        networkClient.sendRequest(request) { (result: Result<Response>) in
          
            completion?(result)   
            self.semaphore.signal()
         
        }
        
       semaphore.wait()
     
    }
    
}

同期されたリクエストのために、DispatchSemaphoreを追加しまし

た。ここで、たとえば3つのリクエストを含むRequestActionを作成する必要があります。


enum RequestAction {
    
    case sendFirstRequest(FirstRequest)
    case sendSecondRequest(SecondRequest)
    case sendThirdRequest(ThirdRequest)
    
}

次に、RequestStateとRequestActionを持つReducerを作成します。動作を設定します-最初、2番目、3番目のリクエストで何をしたいか。


let requestReducer = Reducer<RequestState, RequestAction> { state, action in
  
    switch action {
        
    case .sendFirstRequest(let request):
      
        state.sendRequest(request) { (result: Result<FirstResponse>) in
            
            // 1 Response
            
        }
      
    case .sendSecondRequest(let request):
        
        state.sendRequest(request) { (result: Result<SecondResponse>) in
            
            // 2 Response
            
        }
     
    case .sendThirdRequest(let request):
        
        state.sendRequest(request) { (result: Result<ThirdResponse>) in
            
            // 3 Response
            
        }
        
    }
    
}

最後に、これらのメソッドを呼び出します。最初の要求、2番目の要求、3番目の要求が来ていることは明らかな、より宣言的なスタイルになります。すべてが読みやすく、明確です。


var state = RequestState()

requestReducer.reduce(&state, .sendFirstRequest(FirstRequest()))
requestReducer.reduce(&state, .sendSecondRequest(SecondRequest()))
requestReducer.reduce(&state, .sendThirdRequest(ThirdRequest()))

結論


新しいことを学ぶことを恐れず、関数型プログラミングを学ぶことを恐れないでください。ベストプラクティスはテクノロジーの交差点にあると思います。さまざまなプログラミングパラダイムを組み合わせて、よりうまく活用してください。
