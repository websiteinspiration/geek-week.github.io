Forrester Research：トップ10のソフトウェア構成分析ベンダーの比較

2019年4月、Forrester ResearchはForrester Wave™四半期レポートをリリースしました：ソフトウェア構成分析、2019年第2四半期、10のソフトウェア構成分析（SCA）ソリューションプロバイダーの評価と比較。この記事は、レポートの主要な条項を翻訳したものです。

主な調査結果


  • WhiteSourceとSynopsysは市場のリーダー（リーダー）のリストのトップを占めました。
  • SnykとSonatypeは強力な出演者と呼ばれます。
  • WhiteHat Security、Flexera、およびVeracodeは候補と呼ばれます。
  • GitLab、FOSSA、およびJFrogはチャレンジャーです。

翻訳者メモ

ソフトウェア構成分析とは何ですか？ソフトウェア構成分析プロセスは、サイバーサプライチェーンリスク管理フレームワーク（NISTのWebサイトを参照）のより広範なコンポーネント分析プロセスの亜種であり、サードパーティおよびオープンソースソフトウェアを使用する潜在的なリスクを特定するように設計されています。

ソフトウェア構成分析（SCA）などのソフトウェアソリューションは、リスクを自動的に検出し、コードの脆弱性を排除し、外部（既製）ライブラリの使用を制御するように設計されています。

特にコンポーネント分析とソフトウェア構成分析の一般的なアイデアについては、OWASPプロジェクトWebサイトの資料をよく理解することをお勧めします。

Software Composition Analysis (SCA)


コードの品質を維持しながらアプリケーションに新しい機能を迅速に追加する必要がある状況では、開発者は、一般的なオープンソースコンポーネントに依存して、コードを構成および自動化するだけでなく、パブリックリポジトリにある作成済みのソフトウェアモジュールの使用にますます頼りにしています。最新のソース調査の1つは、調査で分析されたコードベースのオープンソースコンポーネントの数が年々21％増加していることを示しています。
これらのサードパーティのオープンソースコンポーネントには、それ自体がすでにビジネスにリスクをもたらす重大な脆弱性が含まれている可能性があることを理解することが重要です。最近の調査によると、オープンソースコンポーネントの8番目ごとに既知のソースセキュリティの脆弱性が含まれています。情報セキュリティの専門家では、脆弱性が公開されてから悪用されるまでの時間が45日間から3日間に短縮されているため、最近発見された脆弱性の特定と削除にかかる時間が少なくなっています。

Forrester Waveの方法論


Forrester Waveレポートは、テクノロジー市場でのソフトウェア取得オプションを検討しているバイヤー向けのガイドです。すべての比較参加者に同等のプロセスを提供するために、Forresterは参加ベンダーを評価する際にForrester Wave™方法論ガイドに従います。

レポートは、評価されたメーカーのリストを形成するための初期調査を実施します。ベンダーの初期プールは、選択基準に基づいて絞り込まれ、最終的なリストに含まれます。次に、製品や戦略に関する詳細な情報が、アンケート、デモンストレーション/ブリーフィング、レビュー/クライアントへのインタビューを通じて収集されます。この入力は、専門家の調査と市場に関する彼の知識とともに、各参加者を他の参加者と比較する評価システムによってベンダーを評価するために使用されます。
Forresterは、各レポートのタイトルにForrester Waveの発行日（四半期および年）を明確に示しています。このForrester Wave調査に参加しているベンダーは、2019年1月28日までForresterから提供された資料に基づいて評価されました。それ以降、追加の資料は受け入れられませんでした。

レポートを公開する前に、Forresterはベンダーに調査の結果を調査して調査の正確さを評価するように依頼します。Forrester Waveダイアグラムに参加していないとマークされた製造元は基準を満たし、レポートに含めることができますが、調査への参加を拒否または一部のみ参加しました。これらは、Forrester Wave™およびForrester New Wave™非参加型および不完全参加型ベンダーポリシーに従って評価され、その立場は参加ベンダーの立場とともに公開されます。

ベンダープロファイル


分析の結果、各ベンダーの長所と短所が明らかになりました。

リーダー


WhiteSourceは、優先順位付けにより、リスクを排除するために必要な時間を短縮します。 WhiteSourceは最近、静的スキャンを実行してコンポーネントの脆弱な部分がアプリケーションによって直接呼び出されるかどうかを確認することにより、脆弱性に優先順位を付ける機能を導入しました。そうでない場合、脆弱性の優先度は低くなります。比較的最近のもう1つの可能性は、コンポーネントを会社のポリシーに準拠するバージョンに更新するための要求（プル要求）の作成による脆弱性の自動排除です。

ユーザーは、多数の言語のサポートとカスタマーサポートの品質を高く評価していますが、この製品を使用すると、推移的な依存関係を視覚化することができます。 WhiteSourceソリューションには欠点がほとんどありませんが、プロジェクトで使用されるコンポーネントのリスト（部品表またはBOM）をコンパイルする機能は期待どおりに機能しません。 WhiteSourceは、SDLCの最も早い段階でスキャンする必要があり、脅威を優先して自動的に排除するソリューションを探している企業に適しています。

Synopsysは、バイナリスキャンとレポートを通じてBlack Duckを活用しています。Synopsysは、そのソリューションの機能とBlack Duck（Black Duck HubおよびProtecode SC）の機能を組み合わせました。ただし、ライセンス要件のコンプライアンスをより包括的に分析するために、Black Duck ProtexおよびBlack Duckを使用することもできます。たとえば、要求された検出済みライセンス間の違いを分析できますが、これには2つの異なるツールを使用する必要があります。 Synopsysには、インタラクティブスキャンと静的スキャンのための別の機能もあります。同社は最近、リリース前のスキャンツールからのすべてのデータを組み合わせるように設計されたPolarisプラットフォームをリリースしました。

ユーザーは、信頼性が高く、脅威を排除するための詳細な推奨事項を提供するSynopsysコードスキャン製品を信頼していますが、誤検知結果のレベルは高いと考えています。Synopsysは、優れたポリシー管理、SDLCソリューションとの統合、コンポーネントの変更が強調表示されているプロジェクト仕様（BOM）の比較機能など、信頼性の高い予防的な脆弱性管理の機会を提供します。ただし、他の主要ベンダーが提供する脆弱性を自動的に修正する機能に関しては、Synopsysは期待に応えていません。Synopsysは、アプリケーション開発チームに明確なSDLC統合要件があり、異なるタイプのアプリケーションに対して異なるポリシーが必要な企業に適しています。

強力なパフォーマー


Snykは、バージョンを更新して修正を提供する開発者スクリプトに焦点を当てています。この会社の目標は、開発者が脆弱性を修正できるようにすることです。その結果、更新リクエストを作成することで脆弱性を修正できるだけでなく、リポジトリにコンポーネントの受け入れ可能なバージョンがない場合にカスタム修正を使用できるようになります。 Snykはまた、関係と関連する脆弱性を表示するグラフの形式でリクエストを視覚化して、特定の修正が必要な理由を開発者が理解するのに役立ちます。

ユーザーは、シンプルなSDLC統合、手動のコード修正なしのカスタムパッチ適用を含む自動脆弱性修正、依存関係の視覚化など、開発者のニーズに対するSnykの焦点を高く評価しました。ただし、ソリューションが開発者のニーズを完全に満たすためには、Snykがプログラムコードの脆弱性を検出するだけでなく、プログラムコードの使用されているソフトウェアコンポーネントのすべてのバージョンに関する情報も提供する必要があります。 Snykは、開発者のニーズに多くの注意を払っており、情報セキュリティの専門家のニーズにはあまり注意を払っていないため、推奨事項として、Snykの組み込み監査およびレポート機能の機能を拡張する必要があることに注意してください。 Snykは企業に最適開発の段階でプログラムコードの脆弱性の自動除去を実装するタスクを持っている人。

Sonatypeは、パフォーマンスを向上させるためにNexusプラットフォームの開発を続けています。 SonatypeはNexusリポジトリを使用して、脆弱性データベースを充実させています。 Sonatypeの調査チームは、発見された新しい脆弱性に関連するデータ、それらを解決する方法に関する情報、設定の変更、コンポーネントの更新に関する推奨、プログラムコードの変更に関する推奨をさらに補足します。 Nexusプラットフォームには、DepShield、IQ Server、Nexus Auditor、Nexus Firewall、Nexus Lifecycle、Nexus Repository、Nexus Vulnerability Scannerのさまざまな機能のライセンスがいくつかあります。Nexusプラットフォーム製品を最大限に活用するには、これらのコンポーネントの適切な組み合わせを選択する必要があります。

このソリューションのユーザーは、低レベルの誤検知、Nexusリポジトリとの統合、および高レベルのユーザーサポートに注目しています。ただし、Sonatypeソリューションでは、推移的な依存関係を追跡するための機能の改善や、スキャン結果に基づくレポートフォームの提供に、より多くの注意を払う必要があることも強調されています。さらに、価格設定およびライセンスポリシーは、さまざまな企業製品から適切なソリューションを選択する際に製品ユーザーに困難をもたらします。

このソリューションは、Nexus製品を既に使用している企業や、非常に低レベルの誤検知スキャン結果を必要とする企業にとって興味深いものです。

候補者


WhiteHat Securityは、最大の速度を達成するために手動の介入なしにSCAソリューションを提供します。 WhiteHat Securityは、専門家が誤検出スキャン結果のレベルを下げるために、スキャン結果を分析してから、ユーザーに評決を送るという事実で知られています。 WhiteHatは、セキュリティチームの検証結果が残っているWhiteHat Sentinel SCA標準に加えて、Sentinel SCA Essentialsを使用して完全に自動化されたソリューションを提供できるようになりました。

WhiteHat Sentinel Essentialsは新しいソリューションであるため、ユーザーは軽微な欠陥を経験する可能性があります。役割ベースのアクセスシステムの弱点、脆弱な依存関係を特定することの難しさが指摘されていますが、脆弱性情報の詳細は高く評価されています。 WhiteHat Securityは、ユニバーサルスキャナー（SCA、SAST、DAST）として位置付けられ、製品の改善に積極的に取り組んでいます。

WhiteHat Securityは、開発者の成熟度レベルが異なる企業に最適です。スピードを必要とし、製品が提供する機能を個別に使用するのに十分な経験がある企業もあれば、ベンダー側のセキュリティチームによる応答の結果を手動で表示して追加の支援を必要とする企業もあります。

Flexeraは、安全性の研究によって競合他社と区別されています。 Flexera Research GroupのSecuniaは、National Vulnerability Database（NVD）に正式に記録される前に、脆弱性とその対処方法に関する情報をFlexeraユーザーに提供することにより、新しい脆弱性の早期調査を行っています。このグループの成功は、正確さと応答時間に基づいて測定され、すべての脆弱性レポートがNVDで受信され、それに公開されました。 Secuniaが検出した脆弱性はFlexera FlexNet Code Insightに表示され、Secuniaとして識別されます。
Flexeraユーザーは、ユーザーインターフェイスの柔軟性と便利な自動化機能を認めています。同時に、システムの使用方法に関するドキュメントとトレーニングは有益ではなく、テクニカルサポートサービスのレベルと実装中のシステムサポートによって補足する必要があることも報告されており、実装されたAPIの機能は制限されています。

Veracodeは、2つの製品を個別に使用できます。2018年、VeracodeはSourceClearを買収して、そのVeracodeソフトウェア構成分析ソリューションを補完しました。 SourceClearソフトウェア構成分析はエージェントベースのスキャンツールであり、Veracodeソフトウェア構成分析はクラウドベースのSaaSソリューションです。これらの製品の機能はまだ統合されていません。たとえば、SCAはSaaSソリューションを使用して実行できますが、スキャンエージェントは使用しません。 Veracodeは、SCA製品への追加として、静的および動的プレリリーススキャンも提供しています。

VeracodeがSourceClearとSoftware Composition Analysisを完全に組み合わせるまで、ユーザーは製品のライセンスを購入および更新するときに、機能の更新やさまざまなレベルのAPIサポートの遅延により不便を経験します。それにもかかわらず、ベンダーは引き続きソリューションの統合に取り組んでおり、Veracodeは製品間の残りの違い、特に共同言語サポートと共通のポリシーエンジンを取り除くために緊密に連携することが期待されています。

Veracodeは、すでにVeracode製品を使用していて、セキュリティプロバイダーの数を制限したい企業に最適です。現在のVeracodeユーザーは、SCAデータがVeracode製品からの他のスキャンデータを補完する真の安全なアプリケーション開発プラットフォームの約束を高く評価します。

Challengers ()


GitLab , , .GitLabは2017年からセキュリティ製品を提供しており、バイナリSCAに加えて静的および動的コード分析の機能が機能に追加されています。ただし、開発者中心のSCA機能の中には、セキュリティの専門家にとって不便なものもあります。たとえば、無視機能を使用すると、開発者はあらゆる重大度の脆弱性を無視できます。これにより、ISスペシャリストは、開発者が無視することを決定した脆弱性を注意深く監視する必要があります。さらに、GitLabはリリース段階で品質管理によりビルドを遅くする傾向はありません。代わりに、表示オプションを使用することをお勧めします。これにより、セキュリティ専門家は各アセンブリのステータスを手動で表示する必要があります。

GitLabは積極​​的かつ迅速に実装され、セキュリティ機能の開発を続けています。ただし、これらの機会の多くはまだ初期段階または計画段階です。ソリューションを評価するとき、ユーザーは、多数の言語のサポートの欠如、持続不可能な脆弱性の検出、および基本的なポリシー管理機能を強調しました。

開発チーム全体ではないにしても、ほとんどの場合、GitLabをGitLabオープンソースリポジトリとして使用する場合は、GitLabに注意を払う価値があります。

FOSSAは、オープンソースの識別と独自のライセンスで製品を強化しています。FOSSAコンプライアンスは、未加工の著作権ヘッダーを自動的に検出し、プライベートコード、サードパーティコード、著作権で保護された外部コードを区別できます。この発見技術に加えて、FOSSAはオープンソースライセンスの専門知識を持つ法律顧問と協力していると主張しています。 FOSSAは、ユーザーが製品のパフォーマンスを向上させ、新しい言語とプラットフォームのサポートを追加できるようにするオープンソースソリューションです。

ビルド時にFOSSAが提供する脆弱性評価のようにこのツールを使用する専門家は、脆弱性スキャンの結果を信頼できると感じています。一部のFOSSAユーザーはスキャン結果を公開し、FOSSAをソースとしてリストします。 FOSSA自体はオープンソースプロジェクトであるため、ユーザー、わずかなドキュメント、コンテナスキャンなどの高度な機能によって確認された一連のツールと見なすのが最適です。 FOSSAは、外部のプロジェクト参加者が新しい機能を作成する時期を予測できないため、詳細な長期ロードマップをオープンソースプロジェクトに実装することは困難です。

会社の特定の要件に合わせてSCA製品の開発とカスタマイズにリソースを投資する意欲と能力がある場合は、FOSSAを検討する必要があります。

JFrogはArtifactoryリポジトリにあるバイナリコードのスキャンに限定されています。 JFrog XRayを使用すると、観測用のオブジェクトを選択する機能（ウォッチ）を使用して、バイナリコード内でスキャンされる内容を詳細に決定し、ポリシーをスキャン対象に関連付けることができます。ポリシーは、すべてのバイナリコードおよびさまざまなJFrogリポジトリのアセンブリに適用できます。

ユーザーは平均してJFrogを評価し、機能の柔軟性の欠如、直感的なインターフェース、およびレポートの制限を欠点として指摘しました。 JFrog Artifactoryとの優れた統合、新機能の開発速度の速さ、検出された問題の修正がプラスとして示されました。

JFrog Xiは、JFrog Artifactoryがすでに使用されており、脅威の自動除去が必要な場合に検討する必要があります。
