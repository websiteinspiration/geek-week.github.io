私のホームKubernetesクラスターの構造

1年前、私は自分のKubernetesクラスターを作成したいことに気付きました。私はソフトウェア開発者です。通常、単一ノードのローカルクラスターを使用するか、リモートマルチノードクラスターを使用してプロジェクトをテストします。単一ノードクラスターで作業するときは、通常はMinikubeを使用しますが、クラスター内の複数のノードの存在をエミュレートできるKindプロジェクトなどの他のソリューションもあります。複数のノードのサポートがMinikubeに表示される場合があります。

したがって、上記の両方の環境の機能を自由に使えるようにしたいと思います。つまり、複数のノードで構成されるクラスターがあり、これらのクラスターを操作することで、リモート環境とのやり取りに典型的なネットワーク遅延が発生することはありません。シングルボードコンピューター を使用したKubernetesマルチノードクラスターの構築については、多くの優れたチュートリアルがすでに作成されています。これらのマニュアルの多くは、Raspberry PiをSBCとして使用しています。これを見て、抵抗が最も少ない経路をたどることにし、このコンピューターも選びました。 Raspberry Piプラットフォームは、安価で手頃な価格のソリューションとしての地位を確立しています。





このプラットフォームを選択すると、いくつかの妥協点が提供されることに注意してください。たとえば、BroadcomとRaspberry Pi Foundationは、ARMv8暗号化拡張機能のライセンスを取得していません（これは、AESサポートのハードウェアアクセラレーションに必要です）。別の論争の的となる解決策は、オペレーティングシステムがロードされる標準的な記憶媒体の形でのmicroSDカードの使用です。

今では、PicoClusterが電気部品なしのキットのバージョンを販売した方が良いと理解しています。このキットの別の代替手段は、独自のエンクロージャーの開発でした。ただし、これにはさらに時間がかかります。ベクター図面を作成する必要があります。アクリルシートを提供している会社でレーザー切断のサービスを使用する必要があります。理想的には、静電放電を散逸させるコーティングが施されたシートです。

作業中に、電圧不足に関する警告に遭遇しましたそして、問題はケースに付属のマイクロUSBケーブルが原因であることがわかりました。また、ケースと合わせて12V〜5V、30Wの降圧DC-DCコンバーターがあり、より強力なものに交換しました。これは、システムの電源投入に関するセクションで後述する理由により行いました。

このプロジェクトに取り組み始めた時点では、2019年にリリースされたRaspberry Pi 4ボードを使用する予定はありませんでした。これは、作業ノードとして機能する3つのRaspberry Pi 3ボードがまだあることを説明しています。他に2つのボードをRaspberry Pi 4に交換しました。これらのボードは、より多くのリソースを必要とするノードに使用されます。これはメインノードと作業ノードであり、特にデータバックアップの作成に使用されます。

Broadcom BCM2837 SoC（Raspberry Pi 3）とBCM2711 SoC BCM2711（Raspberry Pi 4）の両方に、PWM信号を生成できるハードウェアタイマーがあることに注意してください。その結果、制御入力信号を発行することは非常に簡単ですプロセッサに過負荷をかけずにNoctua NF-A4x20 PWMで期待されています。さらに、ファンがサポートしている速度の半分（約2500 rpm）で動作している場合、ファンはほとんど聞こえません。結局のところ、これは通常の負荷でシステム温度を45°C / 113°F以下に維持するのに十分です。

新しいエンクロージャには、統合電源が含まれています。さらに、彼は私が選択したものよりも高価です。このケースのこれらすべての機能が、今ケースを購入した場合でも、Pico 5Sキットを選択する理由です。それを思い起こさせるためにより多くの努力が払われるべきです、しかし私が別の建物を選んだならば、結局結局は結局は結局は結局は結局は結局は結局それよりも良かったように思えます。一般的に、それは努力と時間の価値があります。

ケースの左パネルには、ボードのGPIOポートがあります。ケーブルの1つは、メインユニットとファンを接続します-その回転速度のPWM制御を構成するため。別のケーブルがメインノードを回路基板モジュールに接続します。これは、作業ノードのオン/オフを切り替えるために使用されます。4本のケーブルが各稼働ノードをメインノードに接続します。オフにすると、アクティブレベルが高いGPIOピンを使用します。これにより、メインノードは、動作の最後に実行される一連のアクションが完了した後に、安全に作業ノードの電源をオフにできます。

ソフトウェア


私はプログラマーなので、他の質問よりもクラスターのソフトウェア部分を処理する方が簡単だと思いました。ランチャーチームによるk3sでの素晴らしい作業の結果は、ここで私を確かに助けました。ここでは、k3sを使用してRaspberry PiにKubernetesクラスターをセットアップする方法については、詳しく説明しません。興味のある方は、こちらのガイドを参考にしてください。ここでは、私が詳しく説明したい構成の主なポイントを示します。

開始するには、各ノードで次を有効にしcgroupsます。

$ sudo sed -i '$ s/$/ cgroup_enable=cpuset cgroup_memory=1 cgroup_enable=memory/' /boot/cmdline.txt
$ sudo reboot

次に、メインノードにk3sをインストールします。

$ curl -sfL https://get.k3s.io | sh -s - --write-kubeconfig-mode 644
#   
$ sudo systemctl status k3s

これで、メインノードへの作業ノードの接続を許可するトークンを取得します。

$ sudo cat /var/lib/rancher/k3s/server/node-token

次に-各作業ノードにk3sをインストールします。

$ curl -sfL https://get.k3s.io | K3S_URL="https://<MASTER_IP>:6443" K3S_TOKEN="<NODE_TOKEN>" sh -
#   
$ sudo systemctl status k3s-agent

デフォルトでインストールされるコンテナイメージの内部レジストリを使用する場合は、特別な方法で設定する必要があります。これにより、containerdイメージをロードできるようになります。

$ sudo sh -c 'REGISTRY=$(kubectl get svc -n kube-system registry -o jsonpath={.spec.clusterIP}); \
cat <<EOT >> /etc/rancher/k3s/registries.yaml
mirrors:
  "$REGISTRY":
    endpoint:
      - "http://$REGISTRY"
EOT'
$ sudo service k3s restart
