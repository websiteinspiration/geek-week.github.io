ゲートウェイAPIをWebアプリケーションおよびAPIの単一のエントリポイントとして使用する

記事の翻訳は、特にコース「高負荷アーキテクト」の学生のために準備されました。



前書き


高可用性、スケーラビリティ、復元力などのAWSの利点は、SaaSプロバイダー（Software-as-a-Service）にとっての価値がすでに証明されています。 AWSはSaaSアプリケーションの最新化により、外部アプリケーションへのAPIアクセスを提供することで、マイクロサービスアーキテクチャへのスムーズな切り替えを支援します。Amazon API Gateway

などのAPI管理ツールは、安全でスケーラブルな外部APIを提供するための自然な選択肢です。ただし、アプリケーションをクラウドに移動する場合、SaaSプロバイダーは多くの場合、サービスを複数の異なるクライアントに迅速に展開する必要に直面します。そしてもちろん、それぞれの特定の要件があります。

Gateway APIは、マルチテナントマイクロサービスアーキテクチャの作成に役立ちます。このようなアーキテクチャでは、マイクロサービスの単一のインスタンスを使用してさまざまなクライアントにサービスが提供されるため、リソースをより最適に使用してコストを最適化できます。この構成では、プロバイダーは顧客にサービスを提供するために、「ホワイトラベル」ドメインのサポートと、特定のビジネスロジックをバックエンドの特定のクライアントにバインドするクライアントドメインを識別する機能が必要です。

この記事では、リファレンスアーキテクチャについて説明します。これにより、ゲートウェイAPIをWebアプリケーションの単一のエントリポイントとして使用し、それぞれに異なるサブドメインを使用して、複数の外部クライアントを持つAPIベースのマイクロサービスを使用できます。

Amazon API Gateway-単一のエントリポイント


複数のWebアプリケーションとマイクロサービスに単一のゲートウェイAPIを使用してアーキテクチャを構築することは、コンポーネントの再利用とコストの最適化にとって重要な要素です。

Amazon API Gatewayは、RESTful APIおよびWebSocket APIを作成および公開するための非常にスケーラブルなソリューションを提供します。バックエンドのさまざまなテクノロジーを選択できます。AWSLambda機能、AWS Step Functions、またはAWS Elastic Beanstalk、Amazon EC2、またはAWSの外部でホストされているHTTPエンドポイントを呼び出します。

API Gatewayは、セキュリティ、キャッシング、スロットリング、モニタリングなどの一般的なAPI管理タスクを引き受けます。その主な目的は、内部APIとマイクロサービスの上に抽象的なレイヤーを構築することですが、バックエンドアプリケーションを簡素化したり、Amazon S3バケットに保存されている静的なウェブページやドキュメントへのアクセスを提供したりすることもできます。

上記に加えて、次の主要なAPIゲートウェイ関数は、ここで説明するアーキテクチャの作成に役立ちます。

1.カスタムドメイン名のサポート：


ゲートウェイAPIを使用してAPIをデプロイする場合、APIエンドポイントのデフォルトのドメイン名はエンドユーザーにとってあまり便利ではありません。 

https://api-id.execute-api.region.amazonaws.com/stage

  • api-id 生成されたゲートウェイAPI
  • region APIの作成時に指定されます。
  • stage APIのデプロイ時に指定します。

デフォルトのAPIエンドポイントは扱いにくい場合があります。しかし、SSL証明書に基づいてサブドメインをテストできるAWS Certificate Managerとの統合のおかげで、APIなどのユーザーに、より簡単で直感的なURLを提供できますcustomer1.example.com。ゲートウェイAPIを使用すると、複数のサブドメインを単一のAPIエンドポイントにマップできます。これにより、外部クライアントの要件に従って「ホワイトラベル」名を使用できます。

2. APIリクエスト/レスポンスの変更


ゲートウェイAPIを使用すると、APIエンドポイントアドレスの各部分に個別の処理を設定できます。このため、APIリクエストをバックエンドの個々のエンドポイントにルーティングすると同時に、APIリクエストをより柔軟に処理するためにリクエスト/レスポンスのヘッダーを変更することが可能です。
