Spring Cache：1分のキャッシュの接続からキャッシュマネージャーの柔軟な構成まで

私はキャッシュを恐れていました。私は本当に登ってそれが何であるかを知りたくありませんでした。私はすぐに、数学のオリンピックの勝者だけが理解できるエンジンコンパートメントのルートエンタープライズのことを想像しました。これはそうではないことがわかりました。キャッシングは、非常にシンプルで理解しやすく、どのプロジェクトにも非常に簡単に実装できることがわかりました。



この投稿では、私が理解しているのと同じくらい簡単にキャッシングについて説明しようと思います。 1分でキャッシュを実装する方法、キーでキャッシュする方法、キャッシュの有効期間を設定する方法、および作業中のプロジェクトに何かをキャッシュするように指示されていて、いじくりたくない場合に知っておく必要があるその他の多くのことを学びます。面。

なぜ「委託」と言うのですか？キャッシングは、原則として、1分あたり数万のリクエストがある、大規模で負荷の高いプロジェクトに適用するのが理にかなっています。このようなプロジェクトでは、データベースに過負荷をかけないようにするために、通常、リポジトリの呼び出しをキャッシュします。特に、あるマスターシステムからのデータが特定の頻度で更新されることがわかっている場合。私たち自身はそのようなプロジェクトを書いておらず、私たちはそれらに取り組んでいます。プロジェクトが小さく、過負荷の恐れがない場合は、もちろん、何もキャッシュしない方がよいでしょう。常に新しいデータは、定期的に更新されるデータよりも常に優れています。

通常、トレーニングポストでは、スピーカーは最初にボンネットの下を這い回り、テクノロジーの根底を探り始めます。それは読者を非常に悩ませ、その後、彼が記事のかなりの部分を読み、何も理解しなかったときに、それがどのように機能するかを説明します。私たちとすべてが異なります。最初に、私たちはそれを機能させ、できれば最小限の労力で、そしてそれから、興味があれば、キャッシュフードの下を調べ、ビン自体の内部を調べ、キャッシングを微調整することができます。しかし、そうしなくても（これはポイント6から始まります）、キャッシュはそのように機能します。

私が約束したキャッシングのすべての側面を分析するプロジェクトを作成します。最後に、いつものように、プロジェクト自体へのリンクがあります。

1.返された結果をキャッシュする


Spring Cacheは何をしますか？Spring Cacheは、特定の入力パラメーターの戻り結果をキャッシュするだけです。それをチェックしよう。@Cacheableアノテーションをget（）サービスメソッドに配置して、返されたデータをキャッシュします。この注釈に「ユーザー」という名前を付けます（これが個別に行われる理由をさらに分析します）。

    @Override
    @Cacheable("users")
    public User get(Long id) {
        log.info("getting user by id: {}", id);
        return repository.findById(id)
                .orElseThrow(() -> new EntityNotFoundException("User not found by id " + id));

2.キャッシュのキーを宣言する


いくつかのパラメータがキャッシュされたメソッドに送られる状況があります。この場合、キャッシュが発生するパラメーターを決定する必要がある場合があります。パラメータで組み立てられたエンティティをデータベースに保存するメソッドに例を追加しますが、その名前のエンティティがすでに存在する場合は保存しません。これを行うには、nameパラメータをキャッシュのキーとして定義します。次のようになります。

    @Override
    @Cacheable(value = "users", key = "#name")
    public User create(String name, String email) {
        log.info("creating user with parameters: {}, {}", name, email);
        return repository.save(new User(name, email));
    }

対応するテストを書いてみましょう：

    @Test
    public void create() {
        createAndPrint("Ivan", "ivan@mail.ru");
        createAndPrint("Ivan", "ivan1122@mail.ru");
        createAndPrint("Sergey", "ivan@mail.ru");

        log.info("all entries are below:");
        service.getAll().forEach(u -> log.info("{}", u.toString()));
    }

    private void createAndPrint(String name, String email) {
        log.info("created user: {}", service.create(name, email));
    }

3人のユーザーを作成し、そのうち2人は同じ名前にします

        createAndPrint("Ivan", "ivan@mail.ru");
        createAndPrint("Ivan", "ivan1122@mail.ru");

そしてそのうちの2つはメールが一致します

        createAndPrint("Ivan", "ivan@mail.ru");
        createAndPrint("Sergey", "ivan@mail.ru");

作成メソッドでは、メソッドが呼び出されたことをすべて記録します。また、このメソッドが返したすべてのエンティティを記録します。結果は次のようになります。

3.強制キャッシング。@CachePut


エンティティの戻り値をキャッシュしたい場合がありますが、同時にキャッシュを更新する必要があります。そのようなニーズのために、@ CachePutアノテーションがあります。アプリケーションがメソッドに渡されると同時に、戻り値のキャッシュが更新されます（すでにキャッシュされている場合でも）。

ユーザーを保存するいくつかのメソッドを追加します。1つは通常の@Cacheableアノテーションでマークし、2つ目は@CachePutでマークします。

    @Override
    @Cacheable(value = "users", key = "#user.name")
    public User createOrReturnCached(User user) {
        log.info("creating user: {}", user);
        return repository.save(user);
    }

    @Override
    @CachePut(value = "users", key = "#user.name")
    public User createAndRefreshCache(User user) {
        log.info("creating user: {}", user);
        return repository.save(user);
    }

最初のメソッドはキャッシュされた値を返すだけで、2番目のメソッドはキャッシュを強制的に更新します。キャッシュは、キー＃user.nameを使用して実行されます。対応するテストを作成します。

    @Test
    public void createAndRefresh() {
        User user1 = service.createOrReturnCached(new User("Vasya", "vasya@mail.ru"));
        log.info("created user1: {}", user1);

        User user2 = service.createOrReturnCached(new User("Vasya", "misha@mail.ru"));
        log.info("created user2: {}", user2);

        User user3 = service.createAndRefreshCache(new User("Vasya", "kolya@mail.ru"));
        log.info("created user3: {}", user3);

        User user4 = service.createOrReturnCached(new User("Vasya", "petya@mail.ru"));
        log.info("created user4: {}", user4);
    }

すでに説明したロジックによれば、「Vasya」という名前のユーザーが初めてcreateOrReturnCached（）メソッドを介して保存されると、キャッシュされたエンティティが受信され、アプリケーションはメソッド自体に入りません。 createAndRefreshCache（）メソッドを呼び出すと、「Vasya」という名前のキーのキャッシュされたエンティティがキャッシュで上書きされます。テストを実行して、コンソールに何が表示されるか見てみましょう。

creating user: User(id=null, name=Vasya, email=vasya@mail.ru)
created user1: User(id=1, name=Vasya, email=vasya@mail.ru)
created user2: User(id=1, name=Vasya, email=vasya@mail.ru)
creating user: User(id=null, name=Vasya, email=kolya@mail.ru)
created user3: User(id=2, name=Vasya, email=kolya@mail.ru)
created user4: User(id=2, name=Vasya, email=kolya@mail.ru)

user1がデータベースとキャッシュに正常に書き込んだことがわかります。ユーザーを同じ名前で再度記録しようとすると、最初の呼び出しのキャッシュ結果が返されます（IDがuser1と同じであるuser2は、ユーザーが書き込まれなかったことを示し、これは単なるキャッシュです）。次に、2番目のメソッドを介して3番目のユーザーを書き込みます。これは、キャッシュされた結果でも、メソッドを呼び出し、新しい結果をキャッシュに書き込みます。これはuser3です。ご覧のとおり、彼はすでに新しいIDを持っています。その後、user3によって追加された新しいキャッシュを取得する最初のメソッドを呼び出します。

4.キャッシュからの削除。@CacheEvict


キャッシュ内の一部のデータをハード更新する必要がある場合があります。たとえば、エンティティはすでにデータベースから削除されていますが、キャッシュからは引き続きアクセスできます。データの一貫性を維持するには、少なくとも削除されたデータをキャッシュに保存しないようにする必要があります。

サービスにいくつかのメソッドを追加します。

    @Override
    public void delete(Long id) {
        log.info("deleting user by id: {}", id);
        repository.deleteById(id);
    }

    @Override
    @CacheEvict("users")
    public void deleteAndEvict(Long id) {
        log.info("deleting user by id: {}", id);
        repository.deleteById(id);
    }

1つ目は単にユーザーを削除し、2つ目もそれを削除しますが、@ CacheEvictアノテーションを付けます。2人のユーザーを作成するテストを追加します。その後、1人は単純な方法で削除され、2人目は注釈付きの方法で削除されます。その後、get（）メソッドを使用してこれらのユーザーを取得します。

    @Test
    public void delete() {
        User user1 = service.create(new User("Vasya", "vasya@mail.ru"));
        log.info("{}", service.get(user1.getId()));

        User user2 = service.create(new User("Vasya", "vasya@mail.ru"));
        log.info("{}", service.get(user2.getId()));

        service.delete(user1.getId());
        service.deleteAndEvict(user2.getId());

        log.info("{}", service.get(user1.getId()));
        log.info("{}", service.get(user2.getId()));
    }

ユーザーが既にキャッシュされているため、キャッシュされているため、削除しても取得できなくなることはありません。ログを見てみましょう。

getting user by id: 1
User(id=1, name=Vasya, email=vasya@mail.ru)
getting user by id: 2
User(id=2, name=Vasya, email=vasya@mail.ru)
deleting user by id: 1
deleting user by id: 2
User(id=1, name=Vasya, email=vasya@mail.ru)
getting user by id: 2

javax.persistence.EntityNotFoundException: User not found by id 2

アプリケーションが両方とも安全にget（）メソッドにアクセスし、Springがこれらのエンティティをキャッシュしたことがわかります。次に、さまざまな方法でそれらを削除しました。通常の方法で最初の1つを削除し、キャッシュされた値が残ったため、ID 1でユーザーを取得しようとしたところ、成功しました。ユーザー2を取得しようとしたときに、メソッドがEntityNotFoundExceptionを返しました-そのようなユーザーはキャッシュにいませんでした。

5.グループ化設定。@キャッシング


1つの方法で複数のキャッシュ設定が必要になる場合があります。この目的のために、@ Cachingアノテーションが使用されます。次のようになります。

    @Caching(
            cacheable = {
                    @Cacheable("users"),
                    @Cacheable("contacts")
            },
            put = {
                    @CachePut("tables"),
                    @CachePut("chairs"),
                    @CachePut(value = "meals", key = "#user.email")
            },
            evict = {
                    @CacheEvict(value = "services", key = "#user.name")
            }
    )
    void cacheExample(User user) {
    }

これは、注釈をグループ化する唯一の方法です。のようなものを積み上げようとすると

    @CacheEvict("users")
    @CacheEvict("meals")
    @CacheEvict("contacts")
    @CacheEvict("tables")
    void cacheExample(User user) {
    }

するとIDEAはそうではないことを教えてくれます。

6.柔軟な構成。キャッシュマネージャー


最後に、私たちはキャッシュを見つけました、そしてそれは私たちにとって理解不能で恐ろしいものではなくなりました。次に、内部を見て、一般的なキャッシングを構成する方法を見てみましょう。

このようなタスクのために、CacheManagerがあります。 Spring Cacheがどこにあっても存在します。 @EnableCacheアノテーションを追加すると、そのようなキャッシュマネージャーがSpringによって自動的に作成されます。 ApplicationContextを自動ラップしてブレークポイントで開くと、これを確認できます。他のビンの中で、cacheManager Beanがあります。



2人のユーザーが既に作成され、キャッシュに入れられた段階で、アプリケーションを停止しました。 Evaluate Expressionを介して必要なBeanを呼び出すと、そのようなBeanが実際に存在し、「users」キーを持つConcurentMapCacheと、キャッシュされたユーザーがすでに含まれている値ConcurrentHashMapがあることがわかります。



次に、Habrとプログラマーを使用してキャッシュマネージャーを作成し、好みに合わせて微調整します。

    @Bean("habrCacheManager")
    public CacheManager cacheManager() {
        return null;
    }

キャッシュマネージャーはたくさんあるので、使用するキャッシュマネージャーを選択するだけです。私はすべてのキャッシュマネージャーをリストするわけではありません。そのようなものがあることを知っていれば十分でしょう。

  • SimpleCacheManagerは、最も簡単なキャッシュマネージャーで、学習とテストに便利です。
  • ConcurrentMapCacheManager-リクエストごとに返されたインスタンスを遅延して初期化します。また、キャッシュの操作方法のテストや学習、および私たちのようないくつかの単純なアクションにも推奨されます。キャッシュでの本格的な作業のために、以下の実装が推奨されます。
  • JCacheCacheManager、 EhCacheCacheManager、 CaffeineCacheManagerは、柔軟にカスタマイズでき、非常に幅広いアクションのタスクを実行する、真の「パートナーパートナー」キャッシュマネージャーです。

私の控えめな投稿の一部として、最後の3つのキャッシュマネージャーについては説明しません。代わりに、例としてConcurrentMapCacheManagerを使用してキャッシュマネージャーをセットアップするいくつかの側面を見ていきます。

それでは、キャッシュマネージャーを再作成しましょう。

    @Bean("habrCacheManager")
    public CacheManager cacheManager() {
        return new ConcurrentMapCacheManager();
    }

キャッシュマネージャーの準備が整いました。

7.キャッシュ設定。寿命、最大サイズなど。


これを行うには、かなり人気のあるGoogle Guavaライブラリが必要です。私は最後のものを取った。

compile group: 'com.google.guava', name: 'guava', version: '28.1-jre'

キャッシュマネージャーを作成するときに、createConcurrentMapCacheメソッドを再定義します。このメソッドでは、GuavaからCacheBuilderを呼び出します。このプロセスでは、次のメソッドを初期化してキャッシュマネージャを構成するよう求められます。

  • maximumSize-キャッシュに含めることができる値の最大サイズ。このパラメーターを使用すると、データベースの負荷とJVM RAMの間の妥協点を見つける試みを見つけることができます。
  • refreshAfterWrite-値をキャッシュに書き込んだ後の時間。その後、値は自動的に更新されます。
  • expireAfterAccess-最後の呼び出し後の値の存続期間。
  • expireAfterWrite-キャッシュへの書き込み後の値の存続期間。これが定義するパラメータです。

その他。

マネージャでレコードの有効期間を定義します。長く待たないように1秒に設定してください。

    @Bean("habrCacheManager")
    public CacheManager cacheManager() {
        return new ConcurrentMapCacheManager() {
            @Override
            protected Cache createConcurrentMapCache(String name) {
                return new ConcurrentMapCache(
                        name,
                        CacheBuilder.newBuilder()
                                .expireAfterWrite(1, TimeUnit.SECONDS)
                                .build().asMap(),
                        false);
            }
        };
    }
