プラグインでサードパーティのアセンブリを使用する方法

問題



私と、MS Dynamic CRMの多くの開発者（私の場合、これはオンプレミスの2016年）が問題に遭遇したと思います。サードパーティアセンブリを使用するには、それらをメインアセンブリと組み合わせてから、プラグイン登録ユーティリティに登録する必要があります。



例として、多くはILMergeまたはILRepackを使用しています。



探す



さまざまなサイトでソリューションオプションを検索しましたが、ILMergeまたはILRepackを使用する以外のすべての場所で、他のオプションは見つかりませんでした。アセンブリがデータベースと共に別のサーバーに移動し、すべてが問題なく機能するように、タスクを実装したいと思います。



発見なし



私が説明したオプションは特に新しいものをもたらすものではないと思いますが、一方で、それは私の友人であり同僚であるYuriy Yurkovskiyが私を見つけるのを助けた簡単な解決策ではありません。

彼は必要なアセンブリをデータベースに保存し、モジュールを使用してサイトレベルでそれらをロードすることを提案しました。



予定



  1. アセンブリのリポジトリを作成しています
  2. 必要に応じて、アセンブリをメモリにロードする実装

using System;

using System.Data.SqlClient;

using System.Reflection;

using System.Web;



namespace CrmExternalAssemblyLoader

{

public class AssemblyLoaderModule: IHttpModule

{

private static bool IsLoaded;

private readonly object _lock = new object();

public void Dispose()

{

throw new NotImplementedException();

}



public void Init(HttpApplication context)

{

lock (_lock)

{

if (!IsLoaded)

{

AppDomain.CurrentDomain.AssemblyResolve += CurrentDomain_AssemblyResolve;

IsLoaded = true;

}

}

}



private Assembly CurrentDomain_AssemblyResolve(object sender, ResolveEventArgs args)

{

var assemblyName = new AssemblyName(args.Name);

var assembly = GetAssembly(assemblyName.Name + ".dll");

if (assembly != null)

{

return Assembly.Load(assembly);

}

return null;

}



public byte[] GetAssembly(string assemblyName)

{

var sqlConnString = @«Data Source=[database]; Initial Catalog=[catalog]; Integrated Security=true;»;

var query = @«select Lib

from CrmExternalAssambly

where Name = name»;

using (var connection = new SqlConnection(sqlConnString))

{

using (var command = new SqlCommand(query, connection))

{

command.Parameters.Add(new SqlParameter("@name", assemblyName));

connection.Open();

var reader = command.ExecuteReader();

if (reader.HasRows)

{

while (reader.Read())

{

return (byte[])reader[0];

}

}

return null;

}

}

}

}

}



アセンブリをコンパイルして、binフォルダーC：\ Program Files \ Microsoft Dynamics CRM \ CRMWeb \ binにスローします。



モジュールを構成に追加します







。サイトをリロードします。



できました。



プラグインの実行が開始すると、CRMはメモリ内のアセンブリを見つけようとしますが、それを見つけられず、AppDomain.CurrentDomain.AssemblyResolveに到達します。ここで、アセンブリはデータベースから差し引かれ、ドメインメモリに読み込まれます。アセンブリは、サイトがリロードされるまで存続します。



長所短所



+アセンブリをプラグインにマージする必要がないため、アセンブリのサイズが増加せず、プラグインの起動時にSQLに大きな負荷がかかりません（特にプラグインがどれにも登録されていない場合）。

+アセンブリは既にドメインにロードされているため、アセンブリへのアクセス速度が速くなります。

+すべてのサードパーティアセンブリはデータベースにあり、ベースと共に移行します。

+異なるバージョンのアセンブリを保存する機能（この時点では十分にテストされていません。そのため、まだ確認



する必要があります）-おそらくCRMを更新するときに、web.config CRMにモジュール行を再度追加する必要があります。
