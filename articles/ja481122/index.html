PostgreSQLアンチパターン：セットと選択をSQLに渡す

時々、開発者は一連のパラメータをリクエストに、または「入力」の選択全体に渡す必要があります。この問題に対する非常に奇妙な解決策が時々出くわします。

「反対側から」見て、それがどのように価値がないのか、なぜ、そしてどうすればもっと上手にできるのかを見てみましょう。

リクエスト本文への値の直接挿入


通常は次のようになります。

query = "SELECT * FROM tbl WHERE id = " + value

... とか、ぐらい：

query = "SELECT * FROM tbl WHERE id = :param".format(param=value)

この方法については、記述され、豊富に描かれています。



ほとんどの場合、これはSQLインジェクションへの直接パスであり、ビジネスロジックへの追加の負荷であり、クエリ文字列を「接着」する必要があります。

このようなアプローチは、PostgreSQL 10以下のバージョンでセクショニングを使用してより効率的なプランを取得する必要がある場合にのみ部分的に正当化できます。これらのバージョンでは、スキャンされたセクションのリストは、送信されたパラメーターを考慮せずに、リクエスト本文に基づいてのみ決定されます。

$ n個の引数


パラメータプレースホルダの 使用は適切です。これにより、PREPARED STATEMENTSを使用できるようになり、ビジネスロジック（クエリ文字列が生成されて1回だけ送信される）とデータベースサーバー（各クエリインスタンスの再解析とスケジューリングは不要）の両方の負荷が軽減されます。

可変数の引数


不明な数の引数を事前に渡したい場合、問題は待機します。

... id IN ($1, $2, $3, ...) -- $1 : 2, $2 : 3, $3 : 5, ...

この形式でリクエストを残すと、潜在的なインジェクションから私たちを救いますが、それでもなお、引数の数から各オプションのリクエストを接着/解析する必要が生じます。毎回行うよりはすでに良いですが、それなしで行うことができます。配列のシリアル化された表現を

含むパラメータを1つだけ渡すだけで十分です。

... id = ANY($1::integer[]) -- $1 : '{2,3,5,8,13}'

唯一の違いは、引数を目的の配列型に明示的に変換する必要があることです。しかし、これは問題の原因にはなりません。どこで対処するかは、前もってわかっているからです。

サンプル転送（マトリックス）


通常、これらはデータベースに挿入するためのデータセットを「1回のリクエストで」転送するためのあらゆる種類のオプションです。

INSERT INTO tbl(k, v) VALUES($1,$2),($3,$4),...

リクエストの「再貼り付け」に関する上記の問題に加えて、これはメモリ不足やサーバーのクラッシュにつながる可能性もあります。その理由は単純です-引数の場合、PGは追加のメモリを予約し、セット内のレコード数は適用されたウィッシュリストビジネスロジックによってのみ制限されます。特に臨床の場合、9,000ドルを超える「番号付き」引数を確認する必要がありました-必要ありません。「2レベル」のシリアライゼーションを

適用して、リクエストを書き換えます。

INSERT INTO tbl
SELECT
  unnest[1]::text k
, unnest[2]::integer v
FROM (
  SELECT
    unnest($1::text[])::text[] -- $1 : '{"{a,1}","{b,2}","{c,3}","{d,4}"}'
) T;

はい、配列内の「複雑な」値の場合は、引用符で囲む必要があります。
この方法で、任意の数のフィールドで選択範囲を「拡張」できることは明らかです。

アンネスト、アンネスト、...


以前の記事で 述べたように、いくつかの「列配列」の「配列の配列」の代わりに、定期的に送信オプションがあります。

SELECT
  unnest($1::text[]) k
, unnest($2::integer[]) v;

この方法では、さまざまな列の値のリストを生成するときに間違いを犯し、サーバーのバージョンにも依存する完全に予期しない結果を得るのは非常に簡単です。

-- $1 : '{a,b,c}', $2 : '{1,2}'
-- PostgreSQL 9.4
k | v
-----
a | 1
b | 2
c | 1
a | 2
b | 1
c | 2
-- PostgreSQL 11
k | v
-----
a | 1
b | 2
c |

ジェイソン


バージョン9.3以降、PostgreSQLはjsonタイプを操作するための本格的な関数を導入しました。したがって、ブラウザーで入力パラメーターの定義が行われた場合、SQLクエリ用のjsonオブジェクトをすぐに作成できます。

SELECT
  key k
, value v
FROM
  json_each($1::json); -- '{"a":1,"b":2,"c":3,"d":4}'

以前のバージョンでは、各（hstore）に同じ方法を使用できますが、hstoreで複雑なオブジェクトをエスケープすることによる正しい「たたみ込み」が問題を引き起こす可能性があります。

json_populate_recordset


「入力」json配列からのデータが何らかのテーブルに入力されることが事前にわかっている場合は、フィールドを「逆参照」し、json_populate_recordset関数を使用して必要な型にキャストすることで、多くを節約できます。

SELECT
  *
FROM
  json_populate_recordset(
    NULL::pg_class
  , $1::json -- $1 : '[{"relname":"pg_class","oid":1262},{"relname":"pg_namespace","oid":2615}]'
  );

json_to_recordset


そしてこの関数は、テーブル形式に依存することなく、転送されたオブジェクトの配列を選択範囲に単純に「展開」します。
SELECT
  *
FROM
  json_to_recordset($1::json) T(k text, v integer);
-- $1 : '[{"k":"a","v":1},{"k":"b","v":2}]'
k | v
-----
a | 1
b | 2

一時テーブル


ただし、送信されたサンプルのデータ量が非常に多い場合は、1回の大量のメモリの割り当てが必要になるため、1つのシリアル化されたパラメーターにデータを投げることは難しく、時には不可能です。たとえば、外部システムからのイベントに関する大量のデータパケットを長期間にわたって収集する必要があり、それをデータベース側で1回処理する必要があるとします。

この場合、最善の解決策は一時テーブルを使用することです：

CREATE TEMPORARY TABLE tbl(k text, v integer);
...
INSERT INTO tbl(k, v) VALUES($1, $2); --  - 
...
--   -

この方法は、大量のデータのまれな送信に適しています。
データの構造を説明する観点から見ると、一時テーブルはpg_classシステムテーブルの 1つの機能、およびpg_type、pg_depend、pg_attribute、pg_attrdefなどの「通常の」一時テーブルとはまったく異なります。

したがって、それぞれに短期間の接続が多数あるWebシステムでは、このようなテーブルが毎回新しいシステムレコードを生成し、データベースへの接続を閉じて削除されます。その結果、TEMP TABLEの制御されていない使用はpg_catalogのテーブルの「膨張」につながり、それらを使用する多くの操作を遅くします。
もちろん、これは定期的にVACUUM FULLをシステムカタログテーブルに渡します。

セッション変数


前のケースのデータ処理が1つのSQLクエリに対して非常に複雑であると想定しますが、十分な頻度で実行したいとします。つまり、DOブロックで手続き型処理を使用したいのですが、一時テーブルを介したデータ転送を使用すると、コストがかかりすぎます。

無名ブロックへの転送に$ nパラメータを使用することもできなくなります。セッション変数とcurrent_setting関数は、この状況から抜け出すのに役立ちます。

バージョン9.2より前は、「your」セッション変数のcustom_variable_classes 名前空間 を事前設定する必要がありました。現在のバージョンでは、次のように書くことができます：

SET my.val = '{1,2,3}';
DO $$
DECLARE
  id integer;
BEGIN
  FOR id IN (SELECT unnest(current_setting('my.val')::integer[])) LOOP
    RAISE NOTICE 'id : %', id;
  END LOOP;
END;
$$ LANGUAGE plpgsql;
-- NOTICE:  id : 1
-- NOTICE:  id : 2
-- NOTICE:  id : 3

サポートされている他の手続き型言語は、他の解決策を見つけることができます。
