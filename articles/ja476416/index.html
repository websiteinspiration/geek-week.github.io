Windows管理センターからのWindowsサーバーの管理

この記事では、引き続きWindows Server Core 2019での作業について説明します。以前の投稿では、99ルーブルのServer Coreを使用した新しいVDS Ultralight料金の例を使用してクライアント仮想マシンを準備する方法について説明しました。その後、彼らは示した Windows Serverの2019 Coreおよび方法で動作するように、その上にGUIをインストールする方法。今日は、Windows管理センターでの管理について説明します。 

写真：TASS

セキュリティ上の理由から、おそらく頭痛の種はサーバーの役割を分けることでした。複数のマシンを起動して、ドメインコントローラーとファイルサーバーを物理的に分離します。
幸い、仮想化が助けになり、現在、いくつかのサービスは互いに分離して動作できますが、セキュリティ上の理由から、同じサーバーでは動作できません。仮想化は多くの利便性をもたらし、単一のテンプレートから仮想マシンを導入すると、専門家は時間を節約し、物理的には、すべてが強力なハードウェアを備えた1つのボックスに収められます。

マシンは少なく、サーバーも増えています。私でも、ファイルサーバー、Javaアプリケーション用のサーバー、および多数のWebサーバーの2つのドメインコントローラーが形成されたことを「確認」するために、Windowsサーバーを効率的に管理する方法について説明します。コーヒーから左手を持ち上げることなく。

-Powershellを使用する！

もちろんそうですが... ...いいえ。この製品は、巨大なインフラストラクチャを管理するための便利なツールとして位置付けられています。これが完全に当てはまるわけではないことは明らかです。そのような場合にはPowershell ISEとスクリプトがあるため、本当に役立つユーザーケースを検討したいと思います。共有したい独自の経験がある場合は、この記事に追加できます。

TL、DR


Windows管理センターは、ストックコンポーネントの管理に適しています。現在、インストールされた役割を管理できるのはRSATのみです。 

WACをゲートウェイとして使用すると、インフラストラクチャのセキュリティを向上させることができます。

プレビュー-WAC用のコンポーネントのベータバージョンのインストールは、アセンブリの一部ではありません。文字通りすべてのコンポーネントはRSATを使用してのみ制御されるため、すべてをリストする必要はありません。

ニュアンス


Windows管理センターのPowershellには、Powershell ISEのような独自のスクリプト環境がありません。
Windows Admin Centerは5.0未満のPowershellをサポートしていません。古いマシンでは、使用する場合は新しいPowershellをインストールする必要があります。

マイクロインスタンスにおけるWindowsアドミンセンターの主な欠点は、サーバーRAMの消費です。これは、それぞれ50〜60メガバイトの4つのセッションを作成し、各セッションはWindows管理センターが閉じた後も残ります。 

Enter-PSSessionを介したPowershellと同じ問題は、新しいセッションも作成します。ターミナルウィンドウを閉じるだけの場合、Exit-PSSessionを使用して終了する前に閉じないと、70メガバイトのセッションがリモートサーバーに残ります。またはRemove-Pssession。
Windowsアドミンセンターを使用する場合、これに我慢する必要があります。これにより、約170メガバイトのRAMが取り除かれ、RSATはこれに悩まされることはありません。


（wsmprovhost.exeを参照）

作業を簡素化


WACがインストールされているワークステーションがドメインにある場合、管理の利便性が最大になります。ログインしているユーザーの資格情報を取得し、サーバーへの接続はワンクリックで実行されます。

RSATのように、txtファイルを使用してサーバーリストをインポートし、サーバー名をラインフィードでリストできます。

また、以前にServer Core仮想マシンをADに統合するには、sconfigを使用してそれを実行する必要がありました。つまり、画面に直接アクセスする必要があります。ホスティングの場合、私はVNCを通じてこれをすべて行わなければなりませんでした。これで、メインページに移動すると、[コンピューターIDの変更]をクリックしてドメインに入ることができます。



ちなみに、Windows Server 2019をドメインに追加するために、SysprepもVNCを介して完了する必要があったため、Sysprepは不要になりました。

ここでネットワーク設定を変更するには、2回クリックする必要があります。サーバーに接続して変更します。

片手でWinRMと同じくらい速く出てきます。



セキュリティを強化する


現在、展開には4つのタイプがあります。ローカルで、ゲートウェイとして、実稼働サーバーの1つに、クラスターの一部としてインストールします。


* Microsoft Webサイトの画像

別のサーバーにゲートウェイとしてインストールする場合、最も安全で推奨されるオプションです。これは、管理へのアクセスが特定のIPアドレスまたはネットワークセクションからのみ利用可能な場合のVPNスキームに類似しています。

同意します。セキュリティで保護されたネットワークに入ったためにYouTubeへの接続が完全に失われるよりも、1つのタブにビデオとミームを、もう1つのタブにWindows管理センターを置く方がはるかに便利です。
すべてのNサーバーを保護する方法は？次のスクリプトを使用します。

##     ##
$servers = Get-Content -Path .\Servers.txt
##      ##
$rules = Get-Content -Path .\Rules.txt
## IP      WAC ##
$gate = "1.1.1.1"
 
$MySecureCreds = Get-Credential
 
foreach ($server in $servers.Split("`n")) {
    foreach ($line in $rules.Split("`n")) {
        Invoke-Command -ComputerName $server -ScriptBlock {
            Set-NetFirewallRule -Name $Using:line -RemoteAddress $Using:gate 
        } -Credential $MySecureCreds 
    }
}

#  , RULES.txt#
 
RemoteDesktop-UserMode-In-TCP
RemoteDesktop-UserMode-In-UDP
WINRM-HTTP-In-TCP
WINRM-HTTP-In-TCP-PUBLIC

#   , SERVERS.txt#
 
1.1.1.1, 1.1.1.2, 1.1.1.3

このスクリプトは、特定のIPアドレスからのみRDPおよびWinRMを使用できるように標準のファイアウォールルールを変更します。インフラストラクチャへの安全なアクセスを整理する必要があります。

調査結果


現在、WindowsアドミンセンターはRSATを置き換えることはできませんが、RSATにはない機能がすでに含まれています。古いスナップインが追加されましたが、ブラウザーでの管理にはそれほど便利ではありません。

開発の優先順位は奇妙です。最も積極的に追加されているのは、本当に便利な機能ではなく、MicrosoftがホストするAzureと統合された機能です。
残念ながら、現時点では、RDPを介して接続することによってWindowsサーバーのすべての機能を便利にしか管理できません。

すべての欠点にもかかわらず、Windowsアドミンセンターには独自のSDKがあり、独自のモジュールを作成し、それを介して独自のソフトウェアを管理できるため、いつかRSATよりも優れたものになります。
