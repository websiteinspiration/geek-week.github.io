コンテナは安全ですか？

ベターリッジの法則を聞いたことがあれば、コンテナーの安全性がどのようになっているのかすでにご存じでしょう。しかし、いずれにしても、単音節の回答では不十分です。分離されたコンテナーを作成し、コンテナー内部からの悪意のあるアクションからインフラストラクチャを保護するためのソリューションと、それらが実際に達成できることを考えてみましょう。

既製のレシピはほとんどありませんが、コンテナの安全性に関する独自の研究で何を基にするかを理解できます。


スピーカーについて：アレクサンダーカヨロフ（allexx）彼は10年間、主にインフラストラクチャに関連するWebプロジェクトを開発しており、現在はChainstackで開発を指揮しています。このポジションでは、さまざまな役割に挑戦し、すべてを実行する必要があります。従来の開発から、技術的な意思決定や人の管理までです。これにより、記事で説明するトピックなど、さまざまなトピックを最初の人からさらに探ることができます。

私はLinuxカーネルの開発者ではなく、情報セキュリティの専門家やDevSecOpsでもありません。私はコンテナ化技術の消費者です。チェーンスタックチームは、ブロックチェーン用のマルチクラウドプラットフォームを提供し、コンテナーベースのインフラストラクチャ、つまりKubernetesを構築します。したがって、テクノロジーの開発者や脆弱性の調査の専門家ではなく、テクノロジーの消費者の視点からコンテナのセキュリティを検討します。

セキュリティとコンテナのトピックは非常に広範囲であり、一般的であると主張することなく、私が実際に遭遇したことについてのみ話します。

クラスターのセキュリティ


クラスターのセキュリティについて語る場合、ほとんどの場合、外部からの攻撃に対する耐性を意味します。



ハッカーがクラスターに侵入しようとする場合、ハッカーはクラスターをロードするか、外部から利用可能なリソースを使用するか、インフラストラクチャ要素を攻撃することでこれを行います。この記事では、これについては説明しません。

クラスターに「悪い」コンテナー（内部からインフラストラクチャを攻撃できるコンテナー）がある場合にどうなるかについて話しましょう。他のコンテナとインフラ全体の両方に影響を与える可能性があります。



私たちの会社でこれから身を守ることが大切な理由を説明します。多くのユーザーがいるため、マルチテナンシーモードで作業します。さまざまな企業からの負荷が1つのクラスターにかかります。マネージドインフラストラクチャを作成し、複数のクライアント企業のアプリケーションが同じクラスターで動作します。

概略的に：ポッド、つまりランタイムが起動され、何らかのストレージが存在します。すべての負荷は計算ノードであり、状態を保存せずには実行できません。



Docker Hubを使用して、ランタイムに適したDockerイメージを見つけます。幸いにも、それらの多くは公式です。私たちが立ち上げて管理するソリューションのベンダーは、それらをパックしてDocker Hubに送信する方法をすでに学んでいます。それらをクラスターで実行してランタイムを取得します。ただし、拡張などの概念があります。たとえば、Cordaブロックチェーンプラットフォーム用にJavaで記述された追加のサードパーティアプリケーションを使用できます。

検証済みのR3から公式のC3dイメージをダウンロードできます。信頼できる可能性が高いです。このイメージをクラスターで実行すると、すべて正常に動作します。しかし、ユーザーは拡張機能をインストールできます。これは基本的に、スマートコントラクトを作成するときにCordaプラットフォームを使用するための主なシナリオです。また、これらはJavaアプリケーションであるため、基本的には何でも起こり得ます。

そのようなアプリケーションの監査を検討していますが、これは別の課題です。このアプリケーションの内容がわからないという事実から始めます。永続性または状態でアプリケーションをロードすることにより、ランタイムをリロードし、実際にランタイムでさまざまなアクションを直接実行する機会を与えます。したがって、クラスターに内部から影響を与える可能性があります。これがコンテナの安全性を調査する主な理由でした。

多くのインフラストラクチャで発生する典型的なシナリオを考慮して、問題が発生する可能性のある場所を理解しようとします。しかし、最初に、反対側に行き、どのような問題が発生する可能性があるのか​​を確認します。突然、これはすべて遠くまで行き渡り、そのようなシナリオは基本的に不可能です。

脆弱性とセキュリティの脅威


セキュリティやDevSecOpsは私の専門ではないため、見つかった脆弱性に関するすべてのニュースをフォローするわけではありません。私は2019年に何が起こっているのかを見ただけで、2つの脆弱性が私の注意を引いた。

2019-02-11 CVE-2019-5736- runC（Score 9.3）を介してDockerから脱却。脆弱性の詳細はこの記事で詳しく説明されており、私たちは消費者として、説明の次の部分に最も関心があります。
「...悪意のあるコンテナが（最小限のユーザー操作で）ホストruncバイナリを上書きできるため、ホスト上でルートレベルのコードが実行されます。「ユーザー対話のレベルでは、任意のコマンドを... rootとして実行できます...」。
つまり、いくつかの簡単な操作を実行すると、ホストマシンへのrootアクセスを取得できます。さらに、Debian、Docker、Red Hat、Ubuntu、AWS、GCP、Azureなど、多くのシステムが影響を受けました。その結果、ほとんどすべてのベンダー、オペレーティングシステム、クラウドプロバイダーがニュースレターをリリースし、特定の変更を加えました。

2019-08-28 CVE-2019-11245- コンテナーはuid 0（スコア7.8）として実行しようとします。それほど怖くはありませんが、最近の脆弱性は、KubernetesのrunAsUserが常にrootアクセスを持つことができることです。
設定に関係なく、「...明示的なrunAsUserを指定していないポッドの場合、コンテナの再起動時、またはイメージが以前にノードにプルされていた場合、uid 0（ルート）として実行されます。ポッドがmustRunAsNonRoot：trueを指定した場合、kubeletはコンテナーをルートとして開始することを拒否します。ポッドでmustRunAsNonRoot：trueが指定されていない場合、kubeletはコンテナをuid 0 "として実行します。
幸い、KVE-2019-11245の影響を受けるのは、選択されたバージョンのKubernetes v1.13.6およびv1.14.2のみです。

これはほんの一部にすぎませんが、コンテナ内部からのセキュリティリスクが存在し、深刻になる可能性があることを思わせることにもなります。

Dockerスタックビューに詳細を追加します。上部には、すべてを実行できる悪魔があり、下には雲または物理的な鉄があります。containerdを忘れないでください。shimとは何か、それがrunCと呼ばれるランタイムの種類を覚えておいてください。

それにもかかわらず、図の右上隅から4つすべての長方形（dokerdを含む）は、何らかの方法で変更できます。しかし、Linuxカーネルレイヤーとコアカーネルモジュールは決して変更されません。



cgroupsと名前空間は、コンテナ化が一般的に行われている2つのクジラです。私たちが何のために働いても、私たちは常にそれらを使用します。

  • 名前空間は分離を提供します。つまり、同じコア上で異なるプロセスを実行し、ホスト名を変更し、ファイルシステムを管理し、プロセス間通信を行うことができます。
  • cgroupはリソースを担当し、メモリ、CPU、ディスクなど、プロセス間でほぼすべてのタイプのリソースを分散できるようにします。

図で青で強調表示されているモジュールは、システム管理者のみが覚えている可能性が高いです。AppArmor（またはSeLinux-これらのツールには基本的な違いはありません）、seccomp、機能は、リソースを管理し、プロセスをフィルター処理する方法を学ぶために、分離を完全に分離する責任があります。

Linuxカーネルとこれらのモジュールは、コンテナーソリューションの基礎です：古い、モダン、改良、変更。スタックの上位のものは、何らかの方法でLinuxカーネル機能を使用します。

コンテナーインフラストラクチャアーキテクチャに障壁はありません。分離があります。同じコアで異なる負荷を実行して、相互に干渉しないようにすることができますが、これはプロセスが「セキュリティサンドボックス」に存在することを意味するものではありません。

コンテナーのセキュリティを強化する方法


歴史を研究した結果、1つのコアですべてが機能するようなアーキテクチャであるため、このコアのセキュリティを強化する必要があることに気付きました。

：私たちは、私たちのソリューションのための3つの主な要件識別セキュリティ、ゼロコンフィグ、または非常に単純な構成で明るさ。つまり、インフラストラクチャにとって安全であるようにコンテナーを起動する必要があり、コンテナーを構成しないか、構成を最小限に抑える必要があります。この要件は、SELinuxとAppArmorを積極的に構成および設定し、セキュリティに関連するすべてのものを慎重に構成する必要があることを知っていたシステム管理者の経験から生まれました。そして、軽さは誰もがコンテナを愛するものです。コンテナーの迅速かつ便利な起動と開始がなければ、誰もが仮想マシンを使い続けるでしょう。

現在のパラダイムとツールの開発状況で可能な限りすでに利用可能なソリューションを探し始めました。これは、コンテナーを保護するための2018年の終わりにありました。

gVisor


大規模な調査を実施したからではなく、gVisorを使用することにしましたが、その時点でGoogle Cloud会議が開かれ、gVisorがオープンソースでリリースされました。私たちが遭遇した多くの情報ノイズがありました。
GVisorのマーケティングの説明には、「コンテナ用のサンドボックスを提供する」と書かれています。
これがまさに私たちが欲しかったものです。ユーザー空間の独立したカーネルであるgVisorによって導入された変更により、コンテナーパラダイムにとどまり、少なくともサンドボックスのような環境を実装できるようになりました。



つまり、同じコンテナが起動され、インターフェースの観点からは何も変わりません。システムコールは残りますが、現在はそれらとコンテナーの間にgVisorがあり、ユーザースペースで機能し、ユーザースペースでEscapeが機能するため、Escapeは発生しません。したがって、中間コンポーネントが追加され、一種のプロキシがgVisorの形式で取得されます。これにより分離が行われます。

GVisorアーキテクチャ


gVisorアーキテクチャは何度か更新されています。徐々に、開発者はOCI互換性のためにそれを適合させました。構成を変更する必要がないため、これは中間ソリューションよりも実用的です。

それはすべて、クライアント（この場合、dockerクラスター）がOCI互換システムにアクセスするという事実から始まります。ランタイムはセキュアからrunscと呼ばれ、OCIランタイムの実装です。



そして興味深いことが起こります。runscはコンテナーを作成し、その隣に別のサービスとしてデプロイされます。Sentryは、Linuxカーネル（ユーザー空間にあるカーネルそのもの）をエミュレートします。一連のコンテナとエミュレートされたコアは、サンドボックスと呼ばれます。

カーネルレベルでは、すべてが以前と同じです-本物のLinuxカーネル。

gVisorは進化的に発展し、最初は単純な機能を実行しました。Sentryを使用してコンテナーからのシステム呼び出しをインターセプトし、ユーザー空間で実行しました。システムコールをインターセプトする最も簡単な方法は、ptraceです。ほとんどどこでも機能し、仮想化を必要としないという点で便利です。初期の段階では、これがgVisorのメインの操作モードでしたが、後に、KVMの使用、つまり仮想化を提供するより高度なバージョンが登場しました。ほとんどの実験では、ptraceオプションを使用しました。

したがって、カーネルレベルでは、KVMまたはクラシックな名前空間の束とseccompがあり、カーネルシステムコールをフィルタリングできます。

追加コンポーネントGofer（Golangとの関連付け方法は不明）は、ファイルシステムへのすべてのシステムコールを処理し、ディスクに関連しないすべてのコールは、Sentryによって処理されます。

SentryとGoferの間の9PはPlan 9-システムのさまざまな部分間のファイルシステムインターフェイスです。オペレーティングシステム自体は普及しませんでしたが、その一部は非常に有用で、40年後に使用され始めました。

はじめに


アーキテクチャの面倒な側面をすべて調べた後、最後にすべてを開始する方法について話しましょう。

私はmacOSユーザーです。手順はかなり簡単ではないので、説明を共有します。

まず、macOSでローカルにruncをビルドしないでください。 Runscはこれを目的としていないため、macOSではまったく実行しないでください。ただし、これは必須ではありません。macOSにはDockerがあり、Linuxで自然に実行されます。したがって、Linuxでアセンブルされたrunsc binarが必要です。それを受け取ってダウンロードしてください（はい、まだセキュリティについて話し合っています）$ wget https://storage.googleapis.com/gvisor/releases/nightly/latest/runsc。

次に、macOSでdockerデーモンのセットアップを見つけます。難しい方法は、UIの設定で「デーモン」を見つけたり、「詳細設定」を選択したりすることです。 JSONを読む方が簡単だと思います：$ cat ~/.docker/daemon.json (taskbar > Preferences > Daemon > Advanced。

このファイルに以下を追加する必要があります。

"default-runtime": "runc",
"runtimes": {
    "runsc": {
        "path": "/usr/allexx/foo/runsc"
    }
}

つまり、新しいランタイムを作成し、バイナリファイルへのパスを指定します。 macOSの場合、binフォルダーをdockerにマウントすることが重要です。奇妙なことに、ドッカーは問題なく別のランタイムを許可します。

結果はこのバンドルです。runscバイナリファイルをmacOSとともにLinuxで動作するdockerにアップロードし、構成に書き込みます。 Dockerが再起動しました。これを使用できます：。 runcはデフォルトではランタイムではないことに注意してください。 簡単な方法のようですが、上級ユーザー向けのオプションもあります。実際、テクノロジーはかなりフレンドリーなので、それほど複雑ではありません。GKE-Google Kubernetes Engine-gVisorを起動するのにフレンドリーです（もちろん考えていたかもしれませんが、2019年の秋にそうなりました）。 新しいノードプールを作成する必要があります。$ docker run --rm --runtime=runsc -it alpine







gcloud beta container node-pools create [NODE_POOL_NAME] \
  --cluster=[CLUSTER_NAME] \
  --node-version=[NODE_VERSION] \
  --image-type=cos_containerd \

残念ながら、これがないと、マネージドKubernetesを起動できません。新しいノードプールを作成するとき、タイプgVisorのサンドボックスパラメーターを指定する必要があります。

次のように新しいノードプールを確認できます。

$ kubectl get runtimeclasses
  NAME AGE
  gvisor 19s

すべてが機能し、独自のアプリケーションを作成したり、既存のマニフェストを変更したりできます。

kind: Deployment
metadata:
  name: httpd
spec:
  replicas: 1
  selector:
    matchLabels:
    app: httpd
  template:
    metadata:
      labels:
        app: httpd
    spec:
      runtimeClassName: gvisor
      containers:
      - name: httpd
      image: httpd


仕様runtimeClassNameに新しいパラメーターが追加されました。これにより、gVisorを指定できます。GKEサンドボックスの使用方法について詳しくは、Google Cloudのドキュメントをご覧ください。

これで十分に機能するようです。しかし、たとえば、別のサーバーを使用したため、ネットワークは機能しませんでした。デフォルトでは、ネットワークサポートはオフになっています。ポートと対話するには、Linux機能モジュールを思い出してください。

spec:
  containers:
  - name: my-container
    securityContext:
      capabilities:
        add: ["NET_RAW"]

モジュールcapabilitiesはを介して設定securityContextでき、システムコールのパラメータをフィルタリングできます。この場合、ネットワークサポートを提供します。

適用性とパフォーマンス

コンテナは安全ですか？


この記事で取り上げたコンテナの安全性のあらゆる側面にもかかわらず、タイトルの質問に対する答えは間違いなくありません。

ランタイムレベルのみを調査しましたが、たとえば、マウント可能なファイルシステム、または同じEscapeを簡単に生成できるシンボリックリンクがあることを思い出すために、他のほとんどすべての脆弱性を利用できます。



トピックは複雑であり、コンテナーを完全に安全にすることにはほど遠いです。ただし、実行するオプションはいくつかあります。

  • securityContextとその構成を詳しく見てみましょう：runAsUser、allowPrivilegeEscalation、readOnlyRootFilesystem。
  • すべてのソフトウェアを最新の状態に保ちます：OS、ランタイム、Kubernetes。
  • GKEなどのマネージドKubernetesと互換性のある負荷監視ツールであるFalcoを詳しく見てみましょう。
  • 高度なセキュリティポリシーを構成します：seccomp、AppArmor、SELinux。
  • 代替案を見てください：Kata ContainersとFirecracker microvirtuals。
  • 実際のサンドボックスが必要な場合-ファッショナブルなコンテナの代わりに、本格的な仮想マシンまたはIronサーバーを使用してください。
