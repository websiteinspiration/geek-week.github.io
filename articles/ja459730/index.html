QtをSTM32に移植する

こんにちは！で我々エムボックスのプロジェクトは、STM32F7-ディスカバリー上のQtを立ち上げ、それについてお伝えしたいと思います。以前に、OpenCVをどうやって起動したかについてはすでに説明しました。

Qtは、グラフィカルコンポーネントだけでなく、QtNetwork、データベースを操作するための一連のクラス、Qt for Automation（IoTの実装を含む）などの多くを含むクロスプラットフォームフレームワークです。 Qtチームの開発者は、組み込みシステムでのQtの使用を予測しているため、ライブラリはかなり適切に構成されています。ただし、最近まで、Qtをマイクロコントローラーに移植することを考えている人はほとんどいませんでした。おそらく、このタスクは複雑に見えます-Qtは大きく、MCUは小さいです。

一方、現時点では、マルチメディアで動作するように設計されたマイクロコントローラがあり、最初のペンティアムより優れています。約1年前、彼女はQtブログの投稿に登場しました。開発者は、RTEMSの下でQtポートを作成し、stm32f7を実行するいくつかのボードでウィジェットを使用してサンプルを起動しました。興味をそそられました。それは顕著であり、開発者自身がこれについてQtがSTM32F7-Discoveryで遅くなると書いています。ウィジェットを描画するだけでなく、アニメーションを開始しながら、Emboxの下でQtを実行できるかどうか疑問に思いました。

Qt 4.8はEmboxに長い間移植されているため、試してみることにしました。弾力のあるアニメーションの例であるmoveblocksアプリケーションを選択しました。

QEMUのQt moveblock


まず、可能であれば、アニメーションをサポートするために必要な最小限のコンポーネントセットでQtを構成します。これには、「-qconfig minimum、small、medium ...」オプションがあります。これには、Qtの構成ファイルに多くのマクロが含まれています-有効/無効にするもの。このオプションの後に、他のものを無効にする場合は、構成に他のフラグを追加します。これが構成の例です。

Qtが機能するためには、OS互換性レイヤーを追加する必要があります。 1つの方法は、QPA（Qt Platform Abstraction）を実装することです。その基礎は、Linux用のQPAが機能することに基づいて、Qtの既成のプラグインfb_baseでした。その結果、EmboxのQtフレームバッファーを提供する小さなemboxfbプラグインが作成され、何の助けもなくすでにそこに描画されます。

プラグインの作成は次のようになります
QEmboxFbIntegration::QEmboxFbIntegration()
    : fontDb(new QGenericUnixFontDatabase())
{
    struct fb_var_screeninfo vinfo;
    struct fb_fix_screeninfo finfo;
    const char *fbPath = "/dev/fb0";

    fbFd = open(fbPath, O_RDWR);
    if (fbPath < 0) {
        qFatal("QEmboxFbIntegration: Error open framebuffer %s", fbPath);
    }
    if (ioctl(fbFd, FBIOGET_FSCREENINFO, &finfo) == -1) {
        qFatal("QEmboxFbIntegration: Error ioctl framebuffer %s", fbPath);
    }
    if (ioctl(fbFd, FBIOGET_VSCREENINFO, &vinfo) == -1) {
        qFatal("QEmboxFbIntegration: Error ioctl framebuffer %s", fbPath);
    }
    fbWidth        = vinfo.xres;
    fbHeight       = vinfo.yres;
    fbBytesPerLine = finfo.line_length;
    fbSize         = fbBytesPerLine * fbHeight;
    fbFormat       = vinfo.fmt;
    fbData = (uint8_t *)mmap(0, fbSize, PROT_READ | PROT_WRITE,
                             MAP_SHARED, fbFd, 0);
    if (fbData == MAP_FAILED) {
        qFatal("QEmboxFbIntegration: Error mmap framebuffer %s", fbPath);
    }
    if (!fbData || !fbSize) {
        qFatal("QEmboxFbIntegration: Wrong framebuffer: base = %p,"
               "size=%d", fbData, fbSize);
    }

    mPrimaryScreen = new QEmboxFbScreen(fbData, fbWidth,
                                        fbHeight, fbBytesPerLine,
                                        emboxFbFormatToQImageFormat(fbFormat));

    mPrimaryScreen->setPhysicalSize(QSize(fbWidth, fbHeight));
    mScreens.append(mPrimaryScreen);

    this->printFbInfo();
}


そして、ここで再描画は次のようになります
QRegion QEmboxFbScreen::doRedraw()
{
    QVector<QRect> rects;
    QRegion touched = QFbScreen::doRedraw();

    DPRINTF("QEmboxFbScreen::doRedraw\n");

    if (!compositePainter) {
        compositePainter = new QPainter(mFbScreenImage);
    }

    rects = touched.rects();
    for (int i = 0; i < rects.size(); i++) {
        compositePainter->drawImage(rects[i], *mScreenImage, rects[i]);
    }
    return touched;
}


その結果、メモリサイズ-Oのコンパイラの最適化が組み込まれているため、ライブラリのイメージは3.5 MBであることがわかりましたが、これはもちろんSTM32F746のメインメモリには適合しません。OpenCVに関する他の記事で既に書いたように、このボードには次の機能があります。

  • 1 MB ROM
  • 320 Kb RAM
  • 8 MB SDRAM
  • 16 MB QSPI

QSPIからのコード実行のサポートはOpenCVにすでに追加されているため、最初にEmbox c Qtイメージ全体をQSPIにロードすることを決定しました。そして乾杯、それはすべてQSPIからほぼ即座に始まりました！しかし、OpenCVの場合と同様に、遅すぎることがわかりました。



したがって、これを行うことにしました。まずイメージをQSPIにコピーし、次にそれをSDRAMにロードして、そこから実行します。 SDRAMからは少し速くなりましたが、QEMUには程遠いです。



次に、浮動小数点を含めるという考えがありました。Qtがアニメーション内の正方形の座標の計算を行うためです。彼らは試みたが、ここでは目に見える加速は得られなかったが、記事ではQt開発者は、FPUがタッチスクリーン上の「アニメーションのドラッグ」の速度を大幅に向上させると主張しています。おそらくmoveblocksの浮動小数点計算は大幅に少なくなり、これは特定の例に依存します。

最も効果的なアイデアは、フレームバッファをSDRAMから内部メモリに転送することでした。これを行うには、画面サイズは480x272ではなく、272x272でした。また、色深度をA8R8G8B8からR5G6B5に減らし、1ピクセルのサイズを4バイトから2バイトに減らしました。フレームバッファーのサイズは272 * 272 * 2 = 147968バイトです。これにより、おそらく最も目立つ、大幅な加速が行われ、アニメーションはほぼスムーズになりました。

最後の最適化は、RAMからのEmboxコードの実行とSDRAMからのQtの実行でした。これを行うには、通常どおり、最初に通常どおりEmboxとQtを静的にリンクしますが、テキスト、データ、データ、およびbssライブラリのセグメントをQSPIに配置して、後でSDRAMにコピーできるようにします。

section (qt_text, SDRAM, QSPI)
phdr	(qt_text, PT_LOAD, FLAGS(5))

section (qt_rodata, SDRAM, QSPI)
phdr	(qt_rodata, PT_LOAD, FLAGS(5))

section (qt_data, SDRAM, QSPI)
phdr	(qt_data, PT_LOAD, FLAGS(6))

section (qt_bss, SDRAM, QSPI)
phdr	(qt_bss, PT_LOAD, FLAGS(6))

コードの実行により、ROMのEmboxも具体的な加速を受けました。その結果、アニメーションは非常にスムーズであることがわかりました。


すでに最後に、記事を準備してEmboxのさまざまな構成を試しているときに、Qt moveblockがSDRAMのフレームバッファーを使用してQSPIから正常に機能し、フレームバッファーのサイズがボトルネックであることがわかりました。どうやら、最初の「スライドショー」を克服するには、フレームバッファサイズの平凡な縮小により、2倍の加速で十分でした。しかし、Emboxコードのみをさまざまな高速メモリに転送することでこのような結果を達成することはできませんでした（2倍ではなく、約1.5倍の加速が得られました）。

STM32F7-Discoveryを使用している場合は、EmboxでQtを自分で実行できます。これを行う方法については、Wikiを参照してください。

結論


その結果、なんとかQt！をリリースすることができました。私たちの意見では、タスクの複雑さはやや誇張されています。当然、マイクロコントローラの詳細を考慮し、一般的にコンピューティングシステムのアーキテクチャを理解する必要があります。最適化の結果は、コンピューティングシステムのボトルネックがプロセッサではなくメモリであるというよく知られた事実を示しています。
