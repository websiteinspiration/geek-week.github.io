高度な視覚化


計画時間/実行時間


ここで、リクエストの実行時に余分な時間がかかった場所を確認することをお勧めします。



I / Oタイミング


場合によっては、リソースの読み書きが多すぎて、一部の実行時間が一貫していないように見える状況に対処する必要があります。

ここで、「ああ、おそらくサーバーのディスクがビジー状態だったため、長時間読み取られていたのです！」しかし、どういうわけかそれはあまり正確ではありません...

しかし、確実に判断できます。実際のところ、PGサーバーの構成オプションにはtrack_io_timing次のものがあります。
I / O操作の時間計測を可能にします。このオプションは、オペレーティングシステムから常に現在の時刻を要求する必要があるため、デフォルトでは無効になっています。これにより、一部のプラットフォームでの作業が大幅に遅くなる可能性があります。pg_test_timingユーティリティを使用して、プラットフォームでの時間測定の時間コストを見積もることができます。I / O統計は、pg_stat_databaseビュー、EXPLAIN出力（BUFFERSパラメーターが使用されている場合）、およびpg_stat_statementsビューから取得できます。
このパラメーターは、ローカルセッション内でも有効にできます。

SET track_io_timing = TRUE;

さて、今の最良の部分は、実行ツリーのすべての変換を考慮に入れて、このデータを理解して表示することを学んだことです。



ここでは、合計実行時間の0.790ミリ秒、データの1ページの読み取りに0.718ミリ秒、書き込みに0.044ミリ秒、そして全体に対して、残りの有用なアクティビティはわずか0.028ミリ秒しか使用されませんでした。
