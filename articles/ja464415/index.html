hh.ruを検索するための新しいヒントサービス

検索のヒントは素晴らしいです。アドレスバーに完全なサイトアドレスを入力する頻度はどれくらいですか。そして、オンラインストアでの製品の名前は？このような短いクエリの場合、検索ヒントが適切であれば、通常は数文字を入力するだけで十分です。そして、あなたが20本の指も信じられないほどのタイピング速度も持っていなければ、きっとそれらを使うでしょう。
この記事では、前回のSchool of Programmersで行った新しいhh.ru検索ヒントサービスについて説明します。

古いサービスにはいくつかの問題がありました：

  • 彼は厳選された人気のあるユーザークエリに取り組みました。
  • ユーザー設定の変更に対応できませんでした。
  • トップに含まれていないクエリをランク付けできませんでした。
  • タイプミスは修正されませんでした。


新しいサービスでは、これらの欠点を（新しいものを追加しながら）修正しました。

人気のクエリの辞書


ヒントがまったくない場合、ユーザーの上位Nクエリを手動で選択し、単語の正確な出現を使用して（順序の有無にかかわらず）これらのクエリからヒントを生成できます。これは適切なオプションです。実装が簡単で、プロンプトの精度が高く、パフォーマンスの問題が発生しません。長い間、私たちの無茶苦茶はそのように機能しましたが、このアプローチの重大な欠点は問題の完全性が不十分であることです。

たとえば、「JavaScript開発者」というリクエストはそのようなリストに該当しなかったため、「JavaScript時間」と入力しても何も表示されません。最後の単語のみを考慮してリクエストを補足すると、最初に「JavaScript便利屋」が表示されます。同じ理由で、Damerau-Levenshtein距離で最も近い単語を見つけることにより、標準のアプローチよりも難しいエラー修正を実装することはできません。

言語モデル


別のアプローチは、クエリの確率を評価し、ユーザークエリの最も可能性の高い継続を生成する方法を学ぶことです。これを行うには、言語モデル（一連の単語シーケンスの確率分布）を使用します。



ユーザーのリクエストは主に短いため、ニューラルネットワーク言語モデルを使用することもせず、n-gramに限定しました。

P(w1…wm)=∏i=1mP(wi|w1…wi−1)≈∏i=1mP(wi|wi−(n−1)…wi−1)



最も単純なモデルとして、確率の統計的定義を取ることができます。

P(wi|w1…wi−1)=count(w1…wi)count(w1…wi−1)



ただし、そのようなモデルは、サンプルになかったクエリの評価には適していません。「ジュニア開発者Java」を観察しなかった場合、次のことがわかります。

P(junior developer java)=count(junior developer java)count(junior developer)=0



この問題を解決するには、平滑化と補間のさまざまなモデルを使用できます。私たちはバックオフを使用しました：

Pbo(wn|w1…wn−1)={P(wn|w1…wn−1),count(w1…wn−1)>0αPbo(wn|w2…wn−1),count(w1…wn−1)=0


α=P(w1…wn−1)1−∑wPbo(w|w2…wn−1)


ここで、Pは平滑化された確率 w1...wn−1 （ラプラス平滑化を使用しました）：

P(wn|w1…wn−1)=count(wn)+δcount(w1…wn−1)+δ|V|


Vは辞書です。

オプションの生成


それで、特定のリクエストの確率を評価できますが、これらの同じリクエストをどのように生成するのですか？次のことを行うのが妥当です。ユーザーにクエリを入力させますw1...wn、それから私たちに適したクエリは条件から見つけることができます

w1…wm=argmaxwn+1…wm∈VP(w1…wnwn+1…wm)



もちろん、並べ替え |V|m−n,m=1…M受信リクエストごとに最適なオプションを選択することはできないため、ビーム検索を使用します。N-gram言語モデルの場合、次のアルゴリズムになります。

def beam(initial, vocabulary):
    variants = [initial]
    for i in range(P):
        candidates = []
        for variant in variants:
            candidates.extends(generate_candidates(variant, vocabulary))
        variants = sorted(candidates)[:N]
    return candidates 

def generate_candidates(variant, vocabulary):
    top_terms = []
    #         1, 2, ... n 
    for n0 in range(n):
        top_n = sorted(vocabulary, key=lambda c: P(|variant[-n0:])
        top_terms.extends(top_n)
    candidates = [variant + [term] for term in top_terms]
    #      
    candidates = sorted(candidates, key=lambda v: P(variant))[:N]
    return candidates



ここで、緑色で強調表示されているノードは、最後に選択されたオプションであり、ノードの前の数字です wn -確率 P(wn|wn−1)、ノードの後- P(w1...wn)。

それははるかに良くなりましたが、generate_candidatesでは、特定のコンテキストに対してN個の最良の用語をすばやく取得する必要があります。n-gramの確率のみを格納する場合は、辞書全体を調べ、考えられるすべての句の確率を計算してから、並べ替える必要があります。明らかに、これはオンラインクエリでは成功しません。

確率のためのホウ素


フレーズの続きのN個の最良の条件付き確率バリアントをすばやく取得するために、用語でホウ素を使用しました。ノード内w1→w2 格納された係数 α、値 P(w2|w1) 条件付き確率で並べ替え P(∙|w1w2) 用語のリスト w3 一緒に P(w3|w1w2)。特別な用語eosは、フレーズの終わりを示します。


しかし、ニュアンスがあります


上記のアルゴリズムでは、クエリ内のすべての単語が完了したと想定しています。ただし、これは、ユーザーが現在入力している最後の単語には当てはまりません。現在の単語を入力し続けるには、辞書全体をもう一度調べる必要があります。この問題を解決するために、シンボリックボロンを使用します。そのノードには、ユニグラム確率でソートされたM項が格納されます。たとえば、M = 3のjava、junior、jupyter、javascriptのborは次のようになります。



次に、ビーム検索の前に、現在の単語を継続するためのM個の最適な候補を見つけますwn そして、N個の最良の候補を選択します P(w1…wn)。

誤植


ユーザーリクエストに対して適切なヒントを提供できるサービスを構築しました。新しい言葉を受け入れる準備もできています。そして、すべてがうまくいくでしょう... しかし、ユーザーは注意し、hfcrkflreキーボードを切り替えないでください。

これを解決するには？最初に頭に浮かぶのは、1つの行から別の文字を取得するために必要な文字の挿入/削除/置換または2つの隣接する文字の置換の最小数として定義される、ダメラウ-レヴェンシュタイン距離の最も近いオプションを見つけることによる修正の検索です。残念ながら、この距離は特定の交換の可能性を考慮していません。したがって、導入された単語「sapper」については、「collector」と「welder」のオプションは同等であることがわかりますが、直感的には2番目の単語を念頭に置いていたようです。

2番目の問題は、エラーが発生したコンテキストを考慮しないことです。たとえば、クエリ「order sapper」では、「welder」ではなく「collector」オプションを優先する必要があります。

タイプミスを修正するタスクに確率論的な観点からアプローチする場合、ノイズの多いチャネルのモデルに到達するのはごく自然なことです。

  1. アルファベットセット Σ;
  2. すべての後続行のセット Σ∗ その上;
  3. 正しい単語である多くの行 D⊆Σ∗;
  4. 与えられた分布 P(s|w)どこ s∈Σ∗,w∈D。

次に、修正タスクは、入力sの正しい単語wを見つけることとして設定されます。エラーの原因に応じて、測定しますP さまざまな方法で構築できますが、この場合、タイプミスの確率を推定することをお勧めします（これを基本置換と呼びましょう）。 Pe(t|r)、ここでt、rはシンボリックnグラムであり、次に評価する P(s|w)最も可能性の高い基本置換によってwからsを取得する確率として。

させてPartn(x)-文字列xをn個の部分文字列（おそらくゼロ）に分割します。ブリルムーアモデルには確率の計算が含まれますP(s|w) 次のように：

P(s|w)≈maxR∈Partn(w),T∈Partn(s)∏i=1nPe(Ti|Ri)


しかし、私たちは見つける必要があります P(w|s)：

P(w|s)=P(s|w)P(w)P(s)=const⋅P(s|w)⋅P(w)


P（w | s）の評価を学習することで、同じダメラウレーベンシュタイン距離でオプションをランク付けする問題も解決し、タイプミスを修正するときにコンテキストを考慮することができます。

計算 Pe(Ti|Ri)


基本置換の確率を計算するために、ユーザークエリが再び役立ちます。単語のペア（s、w）を作成します。

  1. Damerau-Levenshteinに近い。
  2. 単語の1つが他のN回よりも一般的です。

そのようなペアについては、レーベンシュタインに応じて最適な配置を検討します。


sとwのすべての可能なパーティションを作成します（長さn = 2、3に制限しました）：n→n、pr→rn、pro→rn、ro→po、m→ ``、mm→mなど。各Nグラムについて、

Pe(t|r)=count(r→t)count(r)



計算 P(s|w)


計算 P(s|w) 直接取る O(2|w|+|s|)：wのすべての可能なパーティションをsのすべての可能なパーティションでソートする必要があります。ただし、プレフィックスのダイナミクスは、O(|w|∗|s|∗n2)ここで、nは基本置換の最大長です。

d[i,j]={d[0,j]=0j>=kd[i,0]=0i>=kd[0,j]=P(s[0:j] | w[0])j<kd[i,0]=P(s[0] | w[0:i])i<kd[i,j]=maxk,l≤n,k<i,l<j(P(s[j−l:j] | w[i−k:i])⋅d[i−k−1,j−l−1])


ここで、Pはk-gramモデルの対応する行の確率です。よく見ると、ウッコネンクリッピングを使用したワグナーフィッシャーアルゴリズムと非常によく似ています。すべてのステップでP(w[0:i]|s[0:j]) すべての修正を列挙する w[i−k:i] で s[j−l:j] 提供 k,l≤n そして最も可能性の高いものの選択。

戻る P(w|s)


したがって、計算できます P(s|w)。次に、最大化するいくつかのオプションを選択する必要がありますP(w|s)。より正確には、元の要求に対してs1s2…sn あなたは選ばなければなりません w1…wnどこ P(w1…wn|s1…sn)最大。残念ながら、正直な選択肢の選択は、応答時間の要件に適合しなかった（そしてプロジェクトの期限が近づいている）ため、次のアプローチに焦点を合わせることにしました。

  1. 元のクエリから、kの最後の単語を変更することでいくつかのオプションを取得します。
    1. 結果の用語の確率が元の用語よりも数倍高い場合は、キーボードレイアウトを修正します。
    2. Damerau-Levenshtein距離がdを超えない単語を見つけます。
    3. それらからトップNオプションを選択します P(s|w);
  2. 元のリクエストとともにBeamSearchを入力に送信します。
  3. 結果をランク付けするとき、取得したオプションを割引します ∏i=0k−1P(sn−i|wn−i)。

1.2節では、順方向および逆方向のプレフィックスツリーでのファジー検索に基づいて、FB-Trieアルゴリズム（順方向および逆方向トライ）を使用しました。これは、辞書全体でP（s | w）を評価するよりも速いことがわかりました。

クエリ統計


言語モデルの構築により、すべてが簡単です。ユーザークエリに関する統計情報を収集し（特定のフレーズに対してクエリを実行した回数、ユーザー数、登録ユーザー数）、クエリをn-gramに分割して、bursを構築します。エラーモデルの場合はさらに複雑になります。それを構築するには、少なくとも、正しい単語の辞書が必要です。上記のように、トレーニングペアを選択するために、そのようなペアはダメラウ-レヴェンシュタイン距離で近く、1つは他のペアよりも頻繁に発生するという仮定を使用しました。

しかし、データはまだノイズが多すぎます。xssインジェクションの試行、不適切なレイアウト、クリップボードからのランダムなテキスト、「プログラマcではなく1c」という要求を持つ経験豊富なユーザー、キーボードから渡された猫からの要求。

したがって、ソースデータを消去するために、以下を除外しました。

  • 低周波項;

彼らはまた、空席やオープン辞書のテキストからの単語に対してチェックされたキーボードのレイアウトを修正しました。もちろん、すべてを修正することは不可能でしたが、そのようなオプションは通常、完全に切り捨てられるか、リストの一番下にあります。
