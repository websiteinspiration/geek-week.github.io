オフィスでの遠隔作業。RDP、Port Knocking、Mikrotik：シンプルで安全

多くの国でcovid-19ウイルスが蔓延し、普​​遍的な検疫が行われることに関連して、多くの企業が作業を継続する唯一の方法は、インターネットを介した仕事へのリモートアクセスです。リモートでの作業には比較的安全な方法がたくさんありますが、問題の大きさを考えると、ユーザーがリモートでオフィスに接続するための簡単な方法が必要です。追加の設定、説明、面倒な相談、長い説明は必要ありません。この方法は、多くの管理者が好むRDP（リモートデスクトッププロトコル）です。 RDPを介して職場に直接接続することで、軟膏の大きなハエを1つ除いて、問題を解決できます。RDPポートをインターネット用に開いたままにしておくのは非常に危険です。したがって、以下では、シンプルだが信頼できる保護方法を提案します。

私はMikrotikデバイスがインターネットアクセスとして使用されている小規模な組織によく出くわすので、これをMikrotikに実装する方法を以下に示しますが、ポートノッキング保護方法は、入力ルーターと同じ設定で他の高級デバイスに簡単に実装されます。ファイアウォール

ポートノッキングについて簡単に説明します。インターネットに接続されたネットワークの理想的な外部保護は、すべてのリソースとポートがファイアウォールによって外部的に閉じられている場合です。また、このようなファイアウォールが設定されたルーターは、外部からのパケットにはまったく反応しませんが、それらをリッスンします。したがって、ルーターを構成して、異なるポートでネットワークパケットの特定の（コード）シーケンスを受信すると、パケットの送信元であるIPのルーター（ルーター）が特定のリソース（ポート、プロトコルなど）にアクセスできるようにします。

さてポイントに。Mikrotikのファイアウォール設定については詳しく説明しません。インターネットには、このための高品質の情報源がたくさんあります。理想的には、ファイアウォールはすべての着信パケットをブロックしますが、

/ip firewall filter
add action=accept chain=input comment="established and related accept" connection-state=established,related


確立された（関連する）接続からの着信トラフィックを許可します。
次に、Mikrotikでポートノッキングを構成します。

/ip firewall filter
add action=drop chain=input dst-port=19000 protocol=tcp src-address-list="Black_scanners" comment=RemoteRules
add action=drop chain=input dst-port=16000 protocol=tcp src-address-list="Black_scanners" comment=RemoteRules
add action=add-src-to-address-list address-list="remote_port_1" address-list-timeout=1m chain=input dst-port=19000 protocol=tcp comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=19001 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=18999 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=16001 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=15999 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="allow_remote_users" address-list-timeout=1m chain=input dst-port=16000 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
move [/ip firewall filter find comment=RemoteRules] 1
/ip firewall nat
add action=dst-nat chain=dstnat comment="remote_rdp" src-address-list="allow_remote_users" dst-port=33890 in-interface-list=WAN protocol=tcp to-addresses=192.168.1.33 to-ports=3389

さらに詳しく：

最初の2つのルール

/ip firewall filter
add action=drop chain=input dst-port=19000 protocol=tcp src-address-list="Black_scanners" comment=RemoteRules
add action=drop chain=input dst-port=16000 protocol=tcp src-address-list="Black_scanners" comment=RemoteRules

ポートのスキャン時にブラックリストに登録されているIPアドレスからの着信パケットを禁止します。

3番目のルール：

add action=add-src-to-address-list address-list="remote_port_1" address-list-timeout=1m chain=input dst-port=19000 protocol=tcp comment=RemoteRules

目的のポート（19000）を正しく最初にノックしたホストのリストにipを追加します。
次の4つのルール：

add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=19001 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=18999 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=16001 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules
add action=add-src-to-address-list address-list="Black_scanners" address-list-timeout=60m chain=input dst-port=15999 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules

ポートをスキャンしたい人のためのトラップポートを作成し、そのような試みが検出されたら、そのIPをブラックリストに60分間置きます。その間、最初の2つのルールにより、そのようなホストが正しいポートをノックするのを防ぎます。

次のルール：

add action=add-src-to-address-list address-list="allow_remote_users" address-list-timeout=1m chain=input dst-port=16000 protocol=tcp src-address-list="remote_port_1" comment=RemoteRules

2番目の正しいノックが目的のポート（16000）に対して行われるため、1分間（接続を確立するのに十分）許可されたリストにIPを追加します。

次のコマンド：

move [/ip firewall filter find comment=RemoteRules] 1

ルールをファイアウォール処理チェーンの上位に移動します。これは、おそらく、新しく作成されたルールが機能しないようにするために、すでに異なる禁止ルールが設定されているためです。 Mikrotikの最初のルールはゼロから始まりますが、私のデバイスでは組み込みのルールによってゼロが占有されていたため、移動することができませんでした。つまり、1に移動しました。そのため、設定を確認します。ここで、移動して目的の数を指定できます。

次の設定：

/ip firewall nat
add action=dst-nat chain=dstnat comment="remote_rdp_to_33" src-address-list="allow_remote_users" dst-port=33890 in-interface-list=WAN protocol=tcp to-addresses=192.168.1.33 to-ports=3389

ランダムに選択されたポート33890を通常のRDPポート3389と必要なコンピューターまたはターミナルサーバーのIPに転送します。必要なすべての内部リソースに対してこのようなルールを作成し、できれば非標準の（そして異なる）外部ポートを公開します。当然、内部リソースのIPは静的であるか、DHCPサーバーに対して保護されている必要があります。

これでMikrotikが構成されました。ユーザーが内部RDPに接続するための簡単な手順が必要です。私たちは主にWindowsユーザーを持っているので、単純なbatファイルを作成し、それをStartRDP.batと呼びます。

1.htm
1.rdp

したがって、1.htmには次のコードが含まれています。

<img src="http://my_router.sn.mynetname.net:19000/1.jpg">
       RDP
<img src="http://my_router.sn.mynetname.net:16000/2.jpg">

これには、アドレスmy_router.sn.mynetname.netにある架空の画像への2つのリンクが含まれています。このアドレスは、Mikrotikで事前に有効にすることにより、MikrotikのDDNSシステムから取得します。ルーターのDNS名をコピーします。ただし、これが必要なのは、ルーターの外部IPが動的であるか、複数のインターネットプロバイダーの構成が使用されている場合のみです。

最初のリンクのポート：19000は、2番目のポートをそれぞれノックする必要がある最初のポートに対応します。リンク間には、短いネットワークの問題が原因で突然接続が切れた場合の対処方法を示す短い説明があります。ページを更新すると、RDPポートが1分間開かれ、セッションが復元されます。また、imgタグ間のテキストは、ブラウザーのマイクロ遅延を形成します。これにより、最初のパッケージが2番目のポート（16000）に配信される可能性が低くなります。これまでのところ、2週間（30人）でこのようなケースはありません。

次は1.rdpファイルです。これは、ユーザーごとに1つずつまたは個別に構成できます（私はそうしました-数時間よりも15分を費やして、それを理解できなかった人に相談する方が簡単です）。

screen mode id:i:2
use multimon:i:1
.....
connection type:i:6
networkautodetect:i:0
.....
disable wallpaper:i:1
.....
full address:s:my_router.sn.mynetname.net:33890
.....
username:s:myuserlogin
domain:s:mydomain

ここで興味深い設定の1つはmultimonを使用しています：i：1-これには複数のモニターの使用が含まれます-いくつかはそれを必要としますが、彼らはそれ自体をオンにすることを考えません。

接続タイプ：i：6およびnetworkautodetect：i：0-インターネットの大部分が10 Mbpsを超えているため、接続タイプ6（ローカルネットワーク10 Mbps以上）をオンにして、ネットワーク自動検出を無効にします。デフォルトで（自動）になっていると、まれにネットワーク遅延により、セッションの過小評価された速度が自動的に永続的に設定されます。これにより、特にグラフィックプログラムでは、作業に顕著な遅延が生じる可能性があります。

壁紙を無効にする：i：1-デスクトップイメージの
ユーザー名を無効にします：s：myuserlogin-ユーザーの大部分がユーザー名を知らないため、ユーザー名を指定します
domain：s：mydomain-ドメインまたはコンピューター名を指定しますが

WindowsのRDPクライアントについても少し説明します。MSは、プロトコルとそのサーバーおよびクライアント側を最適化する長い道のりを歩んできました。ハードウェア3Dの操作、モニターの画面解像度の最適化、マルチスクリーンなど、多くの便利な機能を実装しています。もちろん、すべてが下位互換モードで実装されており、クライアントがWindows 7でリモートPCがWindows 10の場合、RDPはプロトコルバージョン7.0を使用して動作します。ただし、RDPバージョンをより新しいバージョンにアップグレードできるという利点があります。たとえば、プロトコルバージョンを7.0（Windows 7）から8.1にアップグレードできます。したがって、クライアントの利便性のために、サーバー側のバージョンを最大化し、リンクを破棄してRDPプロトコルクライアントの新しいバージョンにアップグレードする必要があります。

その結果、稼働中のPCまたはターミナルサーバーにリモート接続するためのシンプルで比較的安全なテクノロジーを利用できます。しかし、より安全な接続のために、チェック用のポートを追加することで、ポートノッキング手法は攻撃に対して数桁複雑になる可能性があります。3、4、5、6を追加できます。同じロジックであり、この場合、ネットワークへの直接侵入はほぼ不可能です。 。
