フィグミロ。Figma用のプラグイン（およびMiro用の少し）を作成する



3日目には、実績のある同志のアドバイスに基づいて、レイアウトをFigmaからMiroにPNG画像としてエクスポートするプラグインを作成することにしました。

私たちのチームは熱心にこのアイデアを現実に翻訳し始めました。完全にランダムな一連の状況により、 Miroからのプラグインコンテストが完了する直前にプラグインの作成に取り掛かったため、それはまともなジャックポットを手に入れる機会に特に熱くなりました。

公平に言えば、そのような統合のアイデアが私たちのデザイナーの心に長い間残っていることは注目に値します。少し前まで、設計チーム全体が一連のSketch / Zeplinから現在人気の高いFigmaプラットフォームに移行しました。転送-移動しましたが、古いツールを使用するための多くの使い慣れたシナリオは利用できなくなりました。

特に、2019年8月に、大規模なeコマース企業のサイトでロボットの作業が始まりました。 Sketchを使用する従来のシナリオは次のとおりです。Miroで特別なプラグインを使用してSketchからレイアウトをアンロードし、クライアントチームと話し合いました。プロジェクトのすべての資料のリポジトリとしてMiroを使用します。コメントを操作して、スクリーンマップの変更を収集すると便利です（矢印を配置します）。

このプロジェクトには約450のレイアウトがありました。編集は一度ではなく、1日に数回、マイナーな変更を加えてレイアウトをアンロードしました。必要なプラグインがないため、Figmaでのすべてのアンロードは手動で行われました：エクスポートされたPNG / JPG、保存、Miroでのドロップ、正しい順序でのレイアウト。
20分から1時間までのテキスト（1分、アンロード）を置き換えます。15のレイアウトで1つの（！）Wordを置き換えるために、設計者は1時間を費やしました。エクスポートを自動化する必要があることを認識し、デザイナーやアートディレクターの時間と神経細胞を節約しました。
そこで、typescript、react、mobx（クールなツールを使用する機会を与えてくれたFigma）で武装し、プラグインの作業を開始しました。

最初は、楽観的な仮定に従って、1週間、最大2週間の作業がありました。タスクは、Miro Plugin for Sketchのすべての機能をFigmaレールに転送することだけでした。

しかし、研究活動に関連するソフトウェア開発ではよくあることですが、楽観的な見積もりは現実とはかけ離れています。そのため、プラグインを作成する過程で、かなり多くの技術的な障害が発生しました。

障害物回：REST APIミロ


FigmaはWebアプリケーション（デスクトップでは電子を使用）であるため、プラグインがブラウザー環境で機能することがわかりました。したがって、MiroのREST APIに送信するすべてのリクエストには、CORSブラウザポリシーが適用されます。

つまり、本文がapplication / json形式（承認など）のリクエストを送信したい場合、サーバーがこのリクエストを「好き」にしないと、サーバーからの応答が「送信」されます。私たちの場合、それは起こりました。

したがって、node.jsにプロキシを書き込むことが決定されました。これは、ログイン/パスワードによって承認され、私たちの欲望に共鳴します。これを行うために、プロキシはExpressに有名なミドルウェアcorsを使用します。

障害2：再びREST API Miro


Miro REST APIの公式ドキュメントを調査し、それに基づいてFigmaからMiroに画像をエクスポートするというアイデアを実装しようとしたところ、Miroで使用可能なさまざまなウィジェットから（ウィジェットとは、ステッカー、形状、テキストなどを意味します） RESTインターフェースでは、興味のあるウィジェットを画像で作成することはできません。これは、FigmaレベルでPNGに正常に変換したレイアウトは、Miroの目的のボードに送信できないことを意味します。

しかし、彼らは、Sketch用の同様のプラグインについて覚えていました。Sketchは、そのようなエクスポートを行う方法を知っています。いわば、Miroの人たちがGithubにソースコードを投稿してくれたため、同僚の経験を引き継ぎ、sketch_pluginソースコードを確認することが決定されました。そして、それは一度にいくつかのことを明らかにしました：

  • 近い将来、Sketchのプラグインを作成する予定はありません。Sketchプラグインの開発に使用されたObjective-CおよびJSのフランケンシュタインは、ソースを表示するときに角括弧の数の目に波紋を投影するためです。



JSとObjective-Cの最高のものをすべて組み合わせたい場合は、クラスを配置します。
スクリーンショットのモデルはここにあります。

  • Sketchプラグインは、こちらから入手できる別のAPIを使用します。このAPIは、Miroで選択したボード上にイメージを作成する機能を提供します。これはプラスです。欠点は、ドキュメントが見つからなかったことです。

これらの啓示の後、私たちのチームは、Sketchプラグインから不可解なAPIをベースにすることを決定しました。このAPIの仕様がなくても、暗闇をさまよいながら、Miroに画像をインポートする問題を解決します。そして、どうやら、ここでエピックを終わらせて終わらせることは可能でしたが、そうではありません。

障害3：パフォーマンス


プラグインの最初のベータテスト中に、ユーザーがFigmaからMiroに画像をエクスポートする長いプロセスに不満を感じることが多いことがわかりました。たとえば、約3分で、30のレイアウトがアンロードされました。この3分間の時間の約70％は、リクエストの本文をサーバーに送信することによって行われました。

結局のところ、私たちのfakapは、プロキシアプリケーションでは、jsonが画像とメタデータの予想される形式として使用され、これらの画像自体がUInt8Arrayの形式で転送されたことです。当然、予想される形式をmultipart / form-dataに変更し、同じ30のレイアウトが許容可能な50秒でロードされました。もちろん、成長の余地はまだありますが、速度が3倍に速くなればなるほど、これは私たちの小さな勝利でした。

障害その4：エッジシナリオ


プラグインの使用シナリオは恥ずかしいほど単純であるように見えましたが、作業の過程で、プラグインの操作の未処理のエッジ状態が多数発生しました。

特定の機能（承認、レイアウトをFigmaから画像としてエクスポートするなど）をNとして処理するのに時間がかかる場合、この機能のエッジスクリプトの処理にもN時間かかります。そしてもちろん、この余分な時間はプラグインのリリース日を大幅にシフトしました。しかし、この作業により、優れたレベルのツールの復元力を実現できました。

障害5番：許可


突然Figmaのプラグインを作成したい場合は、Figmaのプラグインはiframeを介して機能し、oAuthプロトコルはのフローティングフレームを介した承認を優先しないため、サードパーティのサービス（google、facebook、slackなど）を介した承認が利用できないことに備えてください。特定のセキュリティポリシーの強さ（特に、クリックジャッキング攻撃を防ぐため）。

プラグインのコンテキストでこれはどういう意味ですか？これは、サードパーティのサービスを介してMiroに登録されているユーザーが、ユーザー名/パスワードを使用してプラグイン経由でログインし、レイアウトをFigmaからエクスポートしたいボードを選択できないことを意味します。また、サードパーティのサービスを介して登録する場合、Miroは将来のユーザーアカウントの電子メールのみを設定し、パスワードは設定しないため、これを行うことはできません。

私たちが選択した回避策は非常に簡単です。プラグインのユーザーに、Miroのサードパーティサービスを通じて登録されており、パスワードがない場合は、パスワードを受け取ったり復元したりするために、標準機能の「電子メールによるパスワードの回復」を使用する必要があることを警告します。



その結果、なんとかミロと対戦できませんでした。開発の2週間での楽観的な評価は実現せず、2か月に成長しました。

私たちが技術的な問題を解決している間にミロが受賞者は、Semyon VolkovとRoman Krasilnikovで、同様のFigma2Miroプラグインがありました。彼らのプロジェクトは6位になりました。 （UIとエクスポートを開始するプラットフォームの観点から）さらにFigmaプラグインであった私たちの実装とは異なり、Miroのプラグインを作成しました。

しかし残念ながら、彼らが作成したソリューションはロボットのニーズを満たしていなかったため、すぐに準備が整った独自のプラグインの開発を続けました。プラグインはFigmiroと呼ばれます。

現在、Figmiroのおかげで、設計者はシステム間でレイアウトをインポートする時間を大幅に節約できます。たとえば、Sergey Kutkovの見積もりからレイアウトをアンロードする1時間は、2（！）分になりました。

要約すると、フィグミロでの研究はより研究的な性質のものだったことに注意したい。2つのサードパーティシステムを何らかの形で統合する場合、問題を解決するためのフレームワークに情報が不足しているため、高度な不確実性に備える必要があります。私たちのケースでは、両方のプラットフォームのAPI機能の調査、仮説のテスト、および解決策の検索に膨大な時間を費やしました。

Figmiroプラグインができること


すべてはSketchで行ったのと同じように機能します。フレームを選択し、ボタンをクリックすると、Miroにレイアウトが表示されます。

  • 選択したフレームをMiroの選択したボードにアップロードします。
  • ボード上の位置を維持しながら、以前にアンロードされたレイアウトを上書きします。

将来的には、プラグインをサポートし、アンロードと同期のプロセスを高速化するようにプラグインの作業を最適化する予定です。MiroがそのAPIを更新した場合、プラグインを再構築します。
