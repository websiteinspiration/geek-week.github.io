Comment se faire des amis Zabbix avec Asterisk «out of the box»

Dans un article précédent, «Zabbix - Extension des bordures de macro», j'ai expliqué comment obtenir une session d'autorisation et la remplacer dans la macro hôte locale. Dans cet article, je vais vous expliquer comment se faire des amis Zabbix avec Asterisk sans scripts ni logiciels externes.



L'idée de «se faire des amis» de ces deux systèmes est née il y a longtemps, et sans installer de logiciels et de scripts supplémentaires. Google rapide a donné beaucoup d'options de solution, tout se résumait à télécharger des scripts (dans la chaleur, bash, python, etc.) sur le serveur, et vous serez heureux. Je voulais mettre en place une surveillance «prête à l'emploi» - sans scripts externes et installer un logiciel supplémentaire sur un serveur avec surveillance et central téléphonique automatique.



J'ai passé 4 jours ouvrables avec, mais le résultat en valait la peine. Travaillez via l'interface AMI, la détection de bas niveau, les déclencheurs et, plus important encore, il faut maintenant environ 15 minutes pour connecter le PBX et tous les autres paramètres.



Il existe un zabbiks 4.4, environ 100 pièces de la version Asterisk 13. Certains échanges sont livrés avec l'interface Web FreePBX, certains avec une console nue, un tas de trucs et l'intégration via dialplan.



Nous recevons les données du central téléphonique



Le premier et principal point qui doit être résolu est l'obtention de données sur les pairs et les enregistrements SIP. Pour ce faire, le PBX dispose des interfaces AGI, AMI, ARI et console SSH. Pour des raisons évidentes, je n'ai pas envisagé de modules supplémentaires.



Vous devez d'abord comprendre ce que sont ces agi, ami, ari ....



  • AGI - l'utilisation de scripts dans le plan de numérotation. Principalement utilisé pour gérer les appels.
  • AMI - peut fournir toutes les informations nécessaires, fonctionne via le port 5038 par analogie avec Telnet. Convient pour nous!
  • ARI - moderne, à la mode, JSON. Il existe de nombreuses possibilités, le format des données est compréhensible pour Zabbix, mais pour moi il n'y a pas de chose principale: vous ne pouvez pas contrôler l'enregistrement des gorgées. Encore un inconvénient, pour les pairs, il n'y a que deux états en ligne / hors ligne, bien qu'il y ait plus d'états et il est utile de les prendre en compte lors du diagnostic.
  • SSH - il peut tout faire, mais parfois il n'est pas donné pour des «considérations de sécurité». Les considérations peuvent être différentes, je ne les démonterai pas.


Cependant, malgré toutes ses lacunes, l'ARI couvre 90% de tous les besoins de surveillance.



Zabbix et Telnet - ma déception



Je connais bien AMI, à un moment donné, j'ai mis en œuvre le suivi des pertes dans les conversations avec la division des bureaux distants, la gestion des appels, etc. Avec Telnet, tout est également très clair: ouvrez une connexion, envoyez des commandes et lisez la réponse. Ce que j'ai fait, mais le résultat m'a déçu.



Le Telnet de Zabbix n'est pas le même que dans la console Linux, il est un peu plus simple et affiné par l'authentification par login / mot de passe standard. Si la logique d'autorisation est différente et qu'aucune paire de connexion / mot de passe n'est demandée, une erreur se produit. Après de vaines tentatives pour contourner l'exigence d'autorisation, il a été utile de consulter le code source du module Telnet.



J'ai réalisé que tant qu'il n'y aura pas de demande de connexion traditionnelle avec un mot de passe, je n'irai pas plus loin. Par intérêt, j'ai jeté tout ce qui concernait l'autorisation hors du code, tout reconstruit. Travaux! Mais cela ne correspond pas aux exigences. Passez…



Retour à la recherche



J'ai relu la documentation sur ARI, effectué des tests supplémentaires - il n'y a pas d'inscription SIP ici. Il y a des fêtes, il y a des conversations, il y a des culottes, il n'y a pas d'inscriptions. À un moment donné, j'ai même pensé, avons-nous vraiment besoin d'une gorgée d'inscription?



Par une drôle de coïncidence, à ce moment une autre demande arrive de l'utilisateur, avec le problème des appels sortants. Le problème était la suspension de la gorgée d'enregistrement et a été résolu par un redémarrage régulier du module. 



asterisk -rx "sip reload"


Ce serait génial d'utiliser AMI sur le web: je pensais que cela résoudrait tous les problèmes. Je commence à creuser dans cette direction, et littéralement la première ligne de résultats de recherche sur la documentation officielle de l'astérisque, qui indique que mes objectifs ont l'option fichier Web / abled / astisk / manager.conf , qui doit être défini sur OUI dans la section [général ]



Après cela, grâce à une demande Web régulière du formulaire http: // ats: 8089 / mxml? Action = SIPshowregistry, nous obtenons toutes les informations nécessaires.

Lorsque vous utilisez l'interface FreePBX, vous ne pouvez pas activer cette option via le Web, vous devez l'activer via la console, en apportant des modifications au fichier manager.conf. FreePBX ne l'efface pas lorsque des modifications de configuration sont effectuées via le Web.
Combien ont fonctionné avec différentes sortes d'intégrations Asterisk, je n'ai jamais vu que cette fonction était mentionnée quelque part. J'ai été surpris que personne ne décrive cette méthode d'interaction avec un central téléphonique. Il était même particulièrement utile de rechercher des informations sur ce sujet: il n'y a pratiquement rien ou a été utilisé pour des tâches complètement différentes.



WEB AMI - quel genre de bête?



Ajout d' options WebEnabled fichier manager.conf Offrant un accès complet à la gestion par un ASCT Web. Toutes les commandes disponibles via l'AMI standard sont maintenant sur le Web, vous pouvez écouter les événements du PBX via le socket. Le principe de fonctionnement n'est pas différent de la console AMI. Après avoir activé cette option, vous pouvez contacter le PBX aux adresses suivantes:



https: // ats: 8089 / manager - une page Web avec une interface simple pour les tests et l'envoi manuel des demandes. Toutes les réponses sont formatées dans un formulaire HTML lisible. Pour le suivi n'est pas très adapté.

https: // ats: 8089 / rawman - sortie texte uniquement, le format est similaire à la console AMI

https: // ats: 8089 / mxml - sortie texte uniquement, au format XML. Convient pour nous!







J'ai alors pensé: «La voici - une solution! Maintenant, tout sera prêt! Easy Pisi Lemon Squeeze, "mais il était trop tôt pour se réjouir. Pour obtenir les informations dont nous avons besoin, il suffit d'utiliser une requête GET avec l'action d' action nécessaire , qui en réponse renvoie xml avec une liste de toutes les inscriptions et leur statut. C'est très bien, mais vous avez besoin d'une autorisation pour vous souvenir de la session à partir du cookie. Lorsque vous testez dans un navigateur, vous ne pensez pas à ce processus.

Pour y travailler, vous avez besoin de mansession_id = " 6f5de42c ", c'est-à-dire cookie d'autorisation lui-même.

Le contenu doit seulement être vérifié pour une réponse acceptée par l'authentification . De plus, pour tous les appels vers le serveur PBX, nous devrons ajouter un cookie d'autorisation à la demande.

Il y a beaucoup de déchets dans la réponse, donc dans le prétraitement, nous le filtrons selon le modèle XPath : // réponse / générique [@host] La

partie la plus intéressante commence. Pour travailler avec la détection et créer dynamiquement des éléments, la réponse doit être au format JSON. XML pour la détection automatique n'est pas pris en charge.



Pour convertir XML en JSON, j'ai dû jouer un peu avec le remplacement automatique, pour lequel j'ai fait un script dans JS.







Un point intéressant: dans la réponse PBX, tous les paramètres sont entourés de guillemets simples, et après avoir appliqué le modèle // response / generic [@host], ils sont remplacés par des doubles.



Pour créer des éléments, nous utilisons des variables de la réponse XML (maintenant JSON).







Registre SIP



Pour les enregistrements SIP, nous utilisons trois variables: nom d'utilisateur , hôte , port . J'étais satisfait du nom de l'élément 111111@login.mtt.ru: 5060 , je n'ai pas trouvé de situations où je devais utiliser les cinq variables.



Élément principal qui reçoit des informations sur toutes les inscriptions, Asterisk est AMI SIPshowregistry . Une fois par minute, il fait une demande GET à https: // ats: 8089 / mxml? Action = SIPshowregistry, après quoi les données de réponse XML sont transmises à tous les éléments dépendants pour analyse. Je crée un élément pour chaque inscription qui en dépend. C'est pratique, car nous obtenons des informations pertinentes pour une demande, et non pour chacune séparément. Cette implémentation a un inconvénient important - la charge sur le processeur.

Je ne stocke pas les informations reçues dans l'élément principal. Premièrement, je ne vois pas la nécessité de cela, et deuxièmement, si la réponse est supérieure à 64 Ko, le zabbix le coupe.



Puisque nous utilisons la réponse XML complète pour l'élément dépendant, nous devons obtenir la valeur de cet élément lors du prétraitement. Cela se fait via XPath comme ceci:

string (// response / generic [@ event = "RegistryEntry"] [@ username = "{# SIP_REGISTRY_USERNAME}"] [@ host = "{# SIP_REGISTRY_HOST}"] [@ port = "{# SIP_REGISTRY_PORT} "] / @ state) Je

n'ai pas utilisé de statuts de texte pour les statuts d'enregistrement, mais je les ai convertis sous forme numérique à l'aide de JavaScript: 



switch(value) {
  case 'Registered':
    return 1;
  case 'Unregistered':
    return 0;
  default:
    return -1;
}


Sip pairs



Par analogie avec les enregistrements SIP, il y a l'élément principal d'Asterisk - AMI SIPshowregistry, auquel des modules complémentaires sont ajoutés.



Deux éléments dépendants sont créés ici:



  • Statut d'homologue sous forme de texte
  • Temps de réponse de l'appareil - si l'état est OK, le temps de réponse de l'appareil est écrit, sinon «-1»


Le chemin d'accès à l'élément lui-même est déjà un peu plus simple que XPath :



string (// response / generic [@objectname = "{# SIP_PEER_OBEJECTNAME}"] / @ status)



Pour le deuxième élément, j'ai utilisé JavaScript pour séparer le temps de réponse du statut d'homologue, car ils sont stockés ensemble: 



if(value.substring(0,2) == 'OK'){
