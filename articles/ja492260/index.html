プロキシ設定


プロキシサーバーの通常の（不透明な）ポートはブリッジアドレス（192.168.120.1）に固定されるため、次の行を/etc/rc.confに追加します。

rc_squid_need="net.br0"

この記事では、次のプロキシ設定は考慮されません。

  • 通常の（非透過）ポート上のユーザーのIDの確認。
  • データ漏洩防止デバイス（InfoWatchトラフィックモニターなど）と統合するICAP

ファイアウォールのセットアップ


nftablesおよびulogdサービスを「デフォルト」の実行レベルでセットアップします（後者はパッケージ通知に必要です）。

rc-config add nftables default
rc-config add ulogd default

ファイアウォール設定にブリッジがあるので、次の行を/etc/rc.confに追加します。

rc_nftables_need="net.br0"

通知を/etc/sysctl.confで機能させるには、次の行を追加する必要があります。

net.netfilter.nf_log.2 = nfnetlink_log

構成ファイルが/ etc / nftablesフォルダーにあるように、/ etc / conf.d / nftablesに次の変更を加えます。

NFTABLES_SAVE="/etc/nftables/rules-save"

一時的な変更が構成ファイルに保存されないようにするには、/ etc / conf.d / nftablesに別の変更を加えます。

SAVE_ON_STOP="no"

ファイアウォールに割り当てられたタスクについて説明します。

  1. ipフィルタテーブルのすべてのチェーンのデフォルトルールは、（出力のチェーンでも）パケットを削除する必要があります。同様に、netdevフィルターテーブルの場合。
  2. IPv4および関連するサービス交換（ARPなど）を除くあらゆる種類の交換は、できるだけ早く抑制されます（IPv4もIPSecも使用されません）。
  3. IPv4マルチキャストパケットもできるだけ早く削除されます。
  4. Androidスマートフォンを保護するには（hostapdアプリケーションを使用してブリッジに接続している場合）、Microsoft Windowsで実行されている仮想マシンによって行われたブロードキャスト交換をワイヤレスインターフェースに送信しないでください。
  5. 既知のMACアドレスを持つWi-Fiデバイスのみを使用できます。

消費者のカスタマイズ


PC自体で


透過プロキシサーバーのより徹底的なテストのために、PC自体のすべてのアプリケーション（一部例外はありますが、後で詳しく説明します）に使用することにしました。

プロキシサーバー証明書を共有ストアに追加します。

mkdir -p /usr/local/share/ca-certificates
cp /etc/squid/squid.pem /usr/local/share/ca-certificates/squid.crt

リポジトリを更新します。

update-ca-certificates

アプリケーション-FirefoxおよびChromium Webブラウザーは共通の証明書ストアを使用しないため、これらのアプリケーションの対応するストアに追加する必要があります。

次の行を含む構成ファイル/etc/env.d/38proxyを作成します。

http_proxy=http://192.168.120.1:3128
https_proxy=http://192.168.120.1:3128
ftp_proxy=http://192.168.120.1:3128

環境変数を更新します。

env-update

再入力後、設定はすべてのアプリケーションに適用されます。

例外について：

  1. 万が一に備えて、Chromiumウェブブラウザはそのままにしておきます。プロキシサーバーなしで常にネットワークに直接接続します。このため、/ etc / chromium /デフォルトの構成ファイルを次のように変更します。 

    CHROMIUM_FLAGS="--enable-seccomp-sandbox —no-proxy-server"
  2. xfreerdpアプリケーションは、https_proxy環境変数が存在する場合、プロキシサーバーを介して接続しようとするため、次の行をシェル構成ファイルに追加します。 

    alias xfreerdp='https_proxy= xfreerdp'
  3. おそらくPython証明書ストアの性質が原因で、youtube-dlアプリケーションは証明書を検証できません。永続的な解決策はまだ見つかっていないため、次の行をシェル構成ファイルに追加します。

    alias youtube-dl='youtube-dl --no-check-certificate'

Red Hat Enterprise Linux OS 6および7の仮想マシン


仮想マシンで、PEM証明書を有効にします。

update-ca-trust force-enable

PCから仮想マシンにプロキシサーバー証明書を書き換えます。

scp squid.crt root@192.168.120.66:/etc/pki/ca-trust/source/anchors/

仮想マシンで証明書を更新します。

update-ca-trust extract

異なるエディションのMicrosoft Windowsオペレーティングシステムを搭載した仮想マシン


証明書は、「信頼されたルート...」セクションで、マシンの一般的な証明書ストア（ユーザーではなく！）にアップロードされます。Firefox Webブラウザーは共通の証明書ストアを使用しないため、このアプリケーションの対応するストアに証明書を追加する必要があります。

Raspbian OSを搭載したRaspberry Pi PCの場合


証明書をインストールせずにアプリケーションのアップグレードが成功しました。

Android OS搭載のスマートフォン


証明書はデバイスにプリロードされ、証明書管理アプリケーションを使用してインストールされます。その後、ユーザーは、グラフィックキーまたはパスワードを使用して、デバイスに入るセキュリティを強化するよう求められます（まだ行っていない場合）。

結論


新しいLinuxファイアウォール（nft）は、無料のプロキシサーバーの事実上の標準であるSquidと相性の良い無料ソフトウェアの好例です。
