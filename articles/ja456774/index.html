Kotlin / Scalaを使用した後にJavaで見逃したもの

最近、Javaが時代遅れの言語になり、サポートされている大規模なアプリケーションを構築することが困難になるとよく耳にします。一般的に、私はこの見解に同意しません。私の意見では、この言語は高速でよく整理されたアプリケーションを書くのにまだ適しています。しかし、私が認めるのは、毎日コードを書くときに、「別の言語でこれを行うことでどれほどよいかが決まる」と思うこともあるということです。この記事では、自分の痛みと経験を共有したいと思いました。Javaのいくつかの問題と、それらがKotlin / Scalaでどのように解決できるかを見ていきます。あなたが同様の気持ちを持っている場合、または他の言語が何を提供できるのか疑問に思っている場合は、猫の下でください。



既存のクラスを拡張する


内部コンテンツを変更せずに既存のクラスを拡張する必要がある場合があります。つまり、クラスを作成した後、それを他のクラスで補足します。小さな例を考えてみましょう。2次元空間の点であるクラスがあるとします。コードのさまざまな場所で、JsonとXMLの両方でコードをシリアル化する必要があります。

Visitorパターンを使用してJavaでどのように見えるかを見てみましょう
public class DotDemo {

    public static class Dot {
        private final int x;
        private final int y;

        public Dot(int x, int y) {
            this.x = x;
            this.y = y;
        }

        public String accept(Visitor visitor) {
            return visitor.visit(this);
        }

        public int getX() { return x; }
        public int getY() { return y; }
    }

    public interface Visitor {
        String visit(Dot dot);
    }

    public static class JsonVisitor implements Visitor {
        @Override
        public String visit(Dot dot) {
            return String
                    .format("" +
                            "{" +
                                    "\"x\"=%d, " +
                                    "\"y\"=%d " +
                            "}",
                    dot.getX(), dot.getY());
        }
    }

    public static class XMLVisitor implements Visitor {
        @Override
        public String visit(Dot dot) {
            return "<dot>" + "\n" +
                    "    <x>" + dot.getX() + "</x>" + "\n" +
                    "    <y>" + dot.getY() + "</y>" + "\n" +
                    "</dot>";
        }
    }

    public static void main(String[] args) {
        Dot dot = new Dot(1, 2);

        System.out.println("-------- JSON -----------");
        System.out.println(dot.accept(new JsonVisitor()));

        System.out.println("-------- XML ------------");
        System.out.println(dot.accept(new XMLVisitor()));
    }
}

パターンとその使用についての詳細

かなりボリュームがありますね。言語ツールの助けを借りて、この問題をよりエレガントに解決することは可能ですか？ScalaとKotlinは肯定的にうなずきました。これは、メソッド拡張メカニズムを使用して実現されます。それがどのように見えるか見てみましょう。

Kotlinの拡張機能
data class Dot (val x: Int, val y: Int)

//     
fun Dot.convertToJson(): String =
        "{\"x\"=$x, \"y\"=$y}"

fun Dot.convertToXml(): String =
        """<dot>
            <x>$x</x>
            <y>$y</y>
        </dot>"""


fun main() {
    val dot = Dot(1, 2)
    println("-------- JSON -----------")
    println(dot.convertToJson())
    println("-------- XML  -----------")
    println(dot.convertToXml())
}


Scalaの拡張機能
object DotDemo extends App {

  // val is default
  case class Dot(x: Int, y: Int)

  implicit class DotConverters(dot: Dot) {
    def convertToJson(): String =
      s"""{"x"=${dot.x}, "y"=${dot.y}}"""
    def convertToXml(): String =
      s"""<dot>
            <x>${dot.x}</x>
            <y>${dot.y}</y>
      </dot>"""
  }

  val dot = Dot(1, 2)
  println("-------- JSON -----------")
  println(dot.convertToJson())
  println("-------- XML  -----------")
  println(dot.convertToXml())
}


それははるかに良く見えます。時々、これは豊富なマッピングや他の変換では本当に十分ではありません。

マルチスレッドコンピューティングチェーン


現在、誰もが非同期コンピューティングと実行スレッドのロックの禁止について話し合っています。次の問題を想像してみましょう：いくつかの数値のソースがあり、最初の数値は単に数値を返し、2番目は最初の数値を計算した後に答えを返します。その結果、2つの数値を含む文字列を返す必要があります。

概略的に、これは次のように表すことができます


最初にJavaで問題を解決してみましょう

Javaの例
    private static CompletableFuture<Optional<String>> calcResultOfTwoServices (
            Supplier<Optional<Integer>> getResultFromFirstService,
            Function<Integer, Optional<Integer>> getResultFromSecondService
    ) {
        return CompletableFuture
                .supplyAsync(getResultFromFirstService)
                .thenApplyAsync(firstResultOptional ->
                        firstResultOptional.flatMap(first ->
                                getResultFromSecondService.apply(first).map(second ->
                                    first + " " + second
                                )
                        )
                );
    }


この例では、結果を制御するために番号がOptionalにラップされています。さらに、すべてのアクションはスレッドでの便利な作業のためにCompletableFuture内で実行されます。主なアクションは、thenApplyAsyncメソッドで行われます。このメソッドでは、引数としてOptionalを取得します。次に、コンテキストを制御するためにflatMapが呼び出されます。受け取ったOptionalがOptional.emptyとして返された場合、2番目のサービスには進みません。

受け取った合計は？ CompletableFutureとオプション機能をflatMapとmapで使用して、問題を解決することができました。私の意見では、ソリューションは最も洗練された方法には見えませんが、問題を理解する前に、コードを読む必要があります。そして、2つ以上のデータソースで何が起こりますか？

言語はどういうわけか私たちが問題を解決するのに役立ちますか？そして再び、Scalaに目を向けてください。Scalaツールでそれを解決する方法は次のとおりです。

Scalaの例
def calcResultOfTwoServices(getResultFromFirstService: Unit => Option[Int],
                            getResultFromSecondService: Int => Option[Int]) =
  Future {
    getResultFromFirstService()
  }.flatMap { firsResultOption =>
    Future { firsResultOption.flatMap(first =>
      getResultFromSecondService(first).map(second =>
        s"$first $second"
      )
    )}
  }


おなじみですね。そして、これは偶然ではありません。これは、主にjava.concurrentのラッパーであるscala.concurrentライブラリーを使用します。さて、Scalaは他に何ができますか 実際のところ、flatMap、...、mapという形式のチェーンはforのシーケンスとして表すことができます。

Scalaでの2番目のバージョンの例
  def calcResultOfTwoServices(getResultFromFirstService: Unit => Option[Int],
                              getResultFromSecondService: Int => Option[Int]) =
    Future {
      getResultFromFirstService()
    }.flatMap { firstResultOption =>
      Future {
        for {
          first <- firstResultOption
          second <- getResultFromSecondService(first)
        } yield s"$first $second"
      }
    }


改善されましたが、もう一度コードを変更してみましょう。猫ライブラリを接続します。

Scalaサンプルの3番目のバージョン
import cats.instances.future._

  def calcResultOfTwoServices(getResultFromFirstService: Unit => Option[Int],
                              getResultFromSecondService: Int => Option[Int]): Future[Option[String]] =
    (for {
      first <- OptionT(Future { getResultFromFirstService() })
      second <- OptionT(Future { getResultFromSecondService(first) })
    } yield s"$first $second").value


OptionTの意味はそれほど重要ではありません。この操作がどれほど簡単で短いかをお見せしたいと思います。

しかし、コトリンはどうですか？コルーチンで同様のことをしてみましょう。

Kotlinの例
val result = async {
        withContext(Dispatchers.Default) { getResultFromFirstService() }?.let { first ->
            withContext(Dispatchers.Default) { getResultFromSecondService(first) }?.let { second ->
                "$first $second"
            }
        }
    }


このコードには独自の特徴があります。まず、コルチンのKotlinメカニズムを使用します。async内のタスクは、作業を盗むメカニズムを備えた特別なスレッドプール（ForkJoinではない）で実行されます。次に、このコードにはasyncやwithContextなどのキーワードを取得するための特別なコンテキストが必要です。

Scala Futureが好きでKotlinを使用している場合は、同様のScalaラッパーに注意を払うことができます。そのようなタイプ。

ストリームを操作する


前の例を拡大してみましょうしようとすると、上記でより詳細に問題を表示するには：私たちは、最も人気のJavaプログラミングツールに向ける- 原子炉スカラ座で、 - FS2。

ストリーム内の3行ごとのファイルを読み取ることを検討し、そこで一致を見つけてみてください。
これは、JavaのReactorでこれを行う最も簡単な方法です。

JavaでのReactorの例
    private static Flux<String> glueFiles(String filename1, String filename2, String filename3) {
        return getLinesOfFile(filename1).flatMap(lineFromFirstFile ->
                getLinesOfFile(filename2)
                        .filter(line -> line.equals(lineFromFirstFile))
                        .flatMap(lineFromSecondFile ->
                            getLinesOfFile(filename3)
                                .filter(line -> line.equals(lineFromSecondFile))
                                .map(lineFromThirdFile ->
                                    lineFromThirdFile
                            )
                )
        );
    }


最適な方法ではありませんが、目安です。ロジックを増やし、サードパーティのリソースを呼び出すと、コードの複雑さが増すと推測するのは難しくありません。for-comprehension構文sugarの代替を見てみましょう。

Scalaでのfs2の例
  def findUniqueLines(filename1: String, filename2: String, filename3: String): Stream[IO, String] =
    for {
      lineFromFirstFile <- readFile(filename1)
      lineFromSecondFile <- readFile(filename2).filter(_.equals(lineFromFirstFile))
      result <- readFile(filename3).filter(_.equals(lineFromSecondFile))
    } yield result


変更点は少ないようですが、見た目はずっと良いです。

ビジネスロジックとhigherKindおよび暗黙の分離


それでは、コードを改善する方法を見てみましょう。次の部分はすぐには理解できないかもしれないということを警告したいと思います。可能性を示し、今のところ実装方法をループの外に置いておきたいと思います。詳細な説明には、少なくとも別の記事が必要です。欲望/コメントがある場合-私はコメントに従って質問に答え、より詳細な説明で2番目の部分を書きます:)

では、開発中に発生する可能性のある技術的影響に関係なくビジネスロジックを設定できる世界を想像してみてください。たとえば、DBMSまたはサードパーティのサービスに対する後続の各リクエストを個別のスレッドで実行することができます。単体テストでは、何も起こらない愚かなmokを作成する必要があります。等。

おそらく一部の人々はBPMエンジンについて考えましたが、今日は彼についてではありません。この問題は、関数型プログラミングと言語サポートのいくつかのパターンの助けを借りて解決できることがわかりました。1つの場所で、このようなロジックを説明できます。

1つの場所で、このようなロジックを説明できます
  def makeCatHappy[F[_]: Monad: CatClinicClient](): F[Unit] =
    for {
      catId <- CatClinicClient[F].getHungryCat
      memberId <- CatClinicClient[F].getFreeMember
      _ <- CatClinicClient[F].feedCatByFreeMember(catId, memberId)
    } yield ()


ここでF [_]（「穴のあいたef」と読みます）は、型よりも型を意味します（ロシア語の文学では、これを種と呼びます）。リスト、セット、オプション、フューチャーなどです。それはすべて、異なるタイプのコンテナーです。

次に、コードのコンテキストを変更します。たとえば、prod環境では、次のようなことができます。

戦闘コードはどのように見えるでしょうか？
class RealCatClinicClient extends CatClinicClient[Future] {
  override def getHungryCat: Future[Int] = Future {
    Thread.sleep(1000) // doing some calls to db (waiting 1 second)
    40
  }
  override def getFreeMember: Future[Int] = Future {
    Thread.sleep(1000) // doing some calls to db (waiting 1 second)
    2
  }
  override def feedCatByFreeMember(catId: Int, memberId: Int): Future[Unit] = Future {
    Thread.sleep(1000) // happy cat (waiting 1 second)
    println("so testy!") // Don't do like that. It is just for debug
  }
}

ビジネスロジックは、使用したフレームワーク、httpクライアント、サーバーに依存しなくなりました。いつでもコンテキストを変更でき、ツールも変わります。

これは、higherKindや暗黙などの機能によって実現されます。最初のものを検討します。これについては、Javaに戻ります。

コードを見てみましょう
public class Calcer {
    private CompletableFuture<Integer> getCalc(int x, int y) {
    }
}


結果を返す方法はいくつありますか？足りる。減算、追加、入れ替えなどが可能です。ここで、明確な要件が与えられていると想像してください。最初の数値を2番目の数値に追加する必要があります。これを行う方法はいくつありますか？一生懸命頑張って自分をもっと洗練させたら...一般的にはたった一つ。

そこにいる
public class Calcer {
    private CompletableFuture<Integer> getCalc(int x, int y) {
        return CompletableFuture.supplyAsync(() -> x + y);
    }
}


しかし、このメソッドの呼び出しが隠されていて、シングルスレッド環境でテストしたい場合はどうでしょうか？または、CompletableFutureを削除または置換してクラスの実装を変更したい場合はどうでしょうか。残念ながら、Javaでは無力であり、メソッドAPIを変更する必要があります。Scalaの代替案を見てください。

特性を考慮する
trait Calcer[F[_]] {
  def getCulc(x: Int, y: Int): F[Int]
}


整数値のコンテナータイプを指定せずに、特性（最も近いものはJavaのインターフェースです）を作成します。

さらに、必要に応じて、さまざまな実装を簡単に作成できます。

そのようです
  val futureCalcer: Calcer[Future] = (x, y) => Future {x + y}
  val optionCalcer: Calcer[Option] = (x, y) => Option(x + y)


また、Implicitなどの興味深いものもあります。これにより、環境のコンテキストを作成し、それに基づいて特性の実装を暗黙的に選択できます。

そのようです
  def userCalcer[F[_]](implicit calcer: Calcer[F]): F[Int] = calcer.getCulc(1, 2)

  def doItInFutureContext(): Unit = {
    implicit val futureCalcer: Calcer[Future] = (x, y) => Future {x + y}
    println(userCalcer)
  }
  doItInFutureContext()

  def doItInOptionContext(): Unit = {
    implicit val optionCalcer: Calcer[Option] = (x, y) => Option(x + y)
    println(userCalcer)
  }
  doItInOptionContext()


valの前に単純化された暗黙-現在の環境に変数を追加し、関数の引数として暗黙的に指定すると、環境から変数が取得されます。これは、暗黙のクロージャを多少連想させます。

まとめると、サードパーティのライブラリを使用しなくても、戦闘とテストの環境を簡潔に作成できることがわかります。
しかし、コトリンはどうですか
実際、kotlinで実行できるのと同じような方法で：
interface Calculator<T> {
    fun eval(x: Int, y: Int): T
}

object FutureCalculator : Calculator<CompletableFuture<Int>> {
    override fun eval(x: Int, y: Int) = CompletableFuture.supplyAsync { x + y }
}

object OptionalCalculator : Calculator<Optional<Int>> {
    override fun eval(x: Int, y: Int) = Optional.of(x + y)
}

fun <T> Calculator<T>.useCalculator(y: Int) = eval(1, y)

fun main() {
    with (FutureCalculator) {
        println(useCalculator(2))
    }
    with (OptionalCalculator) {
        println(useCalculator(2))
    }
}

ここでは、コードの実行コンテキストも設定しますが、Scalaとは異なり、明示的にフラグを立てます。
