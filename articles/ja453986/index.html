Clean Swiftアーキテクチャについて

読者こんにちは！

この記事では、iOSアプリケーションのアーキテクチャーであるClean Swiftについて説明します。主な理論的ポイントを検討し、実際の例を分析します。



理論


まず、アーキテクチャの基本的な用語を分析します。でクリーンスウィフト、アプリケーションはすなわち、シーンで構成されてい 各アプリケーション画面は1つのシーンです。シーンの主な相互作用は、ViewController- > Interactor- > Presenterのコンポーネント間の順次サイクルを通過します。これはVIPサイクルと呼ばれます。

コンポーネント間のブリッジは、送信されたデータを格納するModelsファイルです。シーン間のデータの転送と転送を担当するルーターと、インタラクターのロジックの一部を引き継ぐワーカーも存在します。



見る


コードを通じて記述されたストーリーボード、XIB、またはUI要素。

ViewController


構成とViewとの対話のみを担当します。コントローラには、ビジネスロジック、ネットワークの相互作用、コンピューティングなどを含めないでください。
そのタスクは、InteractorでViewを使用してイベントを処理し、データを表示または送信することです（処理およびチェックなし）。

相互トラクター


シーンのビジネスロジックが含まれています。

ネットワーク、データベース、デバイスモジュールで動作します。

InteractorはViewControllerからリクエストを受け取り（データまたは空で）、それを処理し、必要に応じて新しいデータをPresenterに転送します。

プレゼンター


表示するデータを準備します。

例として、電話番号にマスクを追加するか、タイトルの最初の文字を大文字にします。Interactor
から受信したデータを処理し、それをViewControllerに送り返します。

モデル


VIPサイクルの コンポーネント間でデータを転送するための一連の構造。サイクルの各円には、3つのタイプの構造があります。

  • リクエスト-ViewControllerからInteractorに転送するためのデータ構造（TextFieldからのテキストなど）
  • 応答-InteractorからPresenterに転送するためのデータ（ネットワークなどからダウンロード）を含む構造
  • ViewModel - ViewControllerに戻すためにPresenterで処理されたデータ（テキスト形式など）を含む構造

ワーカー


Interactorを アンロードし、Interactorが急速に成長している場合にアプリケーションのビジネスロジックの一部を引き継ぎます。複数のシーンで機能が使用されて

いる場合は、すべてのシーンに共通のワーカーを作成することもできます。

例として、Workerでは、ネットワークまたはデータベースを操作するロジックを作成できます。

ルーター


シーン間の遷移とデータ転送に関与するすべてのロジックは 、ルーターで取り出されます。


VIPサイクルの全体 像を明確にするために、標準的な例である承認を示します。

  1. ,
  2. ViewController IBAction, , TextField’, (Models -> Request)
  3. fetchUser Interactor’e
  4. Interactor
  5. , (Models -> Response) presentUser Presenter’e
  6. Presenter (Models -> ViewModel) displayUser ViewController’e
  7. ViewControllerは受信したデータをユーザーに表示します。承認の場合、エラーが表示されるか、ルーターを使用して別のシーンへの移行がトリガーされることがあります

したがって、責任がコンポーネントに分散された、単一の一貫した構造が得られます。

練習


それでは、VIPサイクルがどのように進むかを示す小さな実際的な例を見てみましょう。この例では、シーン（画面）を開くときのデータの読み込みをシミュレートします。コードの主要セクションにコメントを付けました。VIPサイクル

全体がプロトコルに関連付けられているため、アプリケーションを中断することなくモジュールを交換できます。DisplayLogicプロトコルがViewController 用に作成されます。このリンクは、後で呼び出すためにPresenterに渡されます。Interactor用に2つのBusinessLogicプロトコルが作成され、ViewControllerからのメソッドの呼び出しとDataSourceを担当します。
、データを保存し、ルーターを介して別のシーンをInteractorに転送します。Presenterは、Interactorから呼び出すためのPresentationLogicプロトコルにサブスクライブします。これらすべての接続要素はModelsです。VIPサイクルのコンポーネント間で情報を交換するための構造が含まれています。それでコード分析を開始します。



モデル

Interactor


Interactor クラスのインスタンスには、PresentationLogicプロトコルへのリンクが含まれており、その下でPresenterが署名されています。fetchUser（request :)

メソッドには、任意のデータ読み込みロジックを含めることができます。たとえば、受信したと思われるデータを使用して、定数を作成しました。 同じメソッドで、Response構造のインスタンスが作成され、前に取得したパラメーターが入力されます。レスポンスはpresentUser（response :)メソッドを使用してPresentationLogicに渡されます。つまり、ここでは生データを取得して、処理のためにPresenterに渡しました。

Presenter


これには、ViewControllerが署名されているDisplayLogicプロトコルへのリンクがあります。ビジネスロジックは含まれていませんが、受信したデータを表示する前にフォーマットするだけです。この例では、電話番号をフォーマットし、ViewModel構造のインスタンスを準備し、データがすでに表示されているDisplayLogicプロトコルのdisplayUser（viewModel :)メソッドを使用してそれをViewControllerに渡しました。

///プレゼンテーションログプロトコル
プロトコル HomePresentationLogic：クラス {
/// Interactor'aで受信したデータをフォーマットする方法
func presentUser(_ response: HomeModels.FetchUser.Response)
}
final class HomePresenter: HomePresentationLogic {
/// View Controller'a
weak var viewController: HomeDisplayLogic?
func presentUser(_ response: HomeModels.FetchUser.Response) {
//
let formattedPhone = response.userPhone.replacingOccurrences(of: "-", with: " ")
// ViewModel View Controller
let viewModel = HomeModels.FetchUser.ViewModel(userPhone: formattedPhone, userEmail: response.userEmail)
// View Controller'a
viewController?.displayUser(viewModel)
}
}
