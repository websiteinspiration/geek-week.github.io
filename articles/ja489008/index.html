Hobotフレームワークを使用した複雑なチャットボットでのルーティング



数年前にTelegram用のボットの開発を始めて、コマンドラインインターフェースの特殊なケースとして、ボットと連携することのパフォーマンス、シンプルさ、柔軟性を発見しました。今日多くの人が利用できるこれらの特性は、Telegram APIを操作するための簡略化されたメソッドを提供する、人気の高いtelegraf.jsフレームワークなどによるものです。

同時に、プロジェクトのアーキテクチャは完全に開発者にかかっており、複雑で多機能なボットの数が少ないことから判断すると、この点についてはまだ成長の余地があります。

この記事では、小さなチャットボットルーティングフレームワークについてお話ししたいと思います。このフレームワークなしでは、プロジェクトの開発は不可能です。

基本的な情報


チャットボットとCLIでは、単一の論理アクションの実行は、多くの場合、いくつかの改良ステップまたは分岐ステップで構成されています。これには、ユーザーがフローのどこにいるかを記憶し、この座標に従ってコマンドを実行するために、プログラムが特定の座標を保存する必要があります。

最も単純な図は、npm initコマンドの実行です。このコマンドでは、package.jsonの1つまたは別のデータを順番に指定するようにプログラムから要求されます。

最初のステップでは、パッケージ名のテキスト入力を待機していることをユーザーに伝えます。ユーザーが次のコマンドで送信するものは、この待機が書き込まれている変数により、パッケージ名として保存されます。

これを可変パスと呼びます-これまたはそのコードが論理的に接続されているパス。ボットがナビゲーションやコマンドを数ステップで発行する場合は、常に必要です。

Hobotフレームワーク


ActualizeBotを 始めて、私たちはそれがどれほど大きいかをすでに想像しました、そして私たちの最初の仕事は開発の拡張性とスピードを保つことでした。

これを行うために、クライアントロジックをパスに割り当てられたコントローラーに分割し、これらのコントローラー間を移動してユーザーから受信したメッセージを処理するための小さな抽象化を記述しました。

これらすべては、大規模なプロジェクトに基づいてTypeScriptで記述されており、ナビゲーションパイプラインをほのめかしていることをほのめかすHobotというエレガントな名前が付けられています。

コントローラは、3つのプロパティの単純なオブジェクトです。

  • path-初期化されたパスの初期化とナビゲートに使用されるパスの文字列識別子
  • get — , , hobot.gotoPath(ctx, path, data?). — data ,
  • post — . , , . updateType — , : text, callback_query . updateType ctx

コントローラーの例：

const anotherController = {
    path: 'firstPath',
    get: async (ctx, data) => 
        await ctx.reply('Welcome to this path! Say "Hi"'),
    post: async (ctx, updateType) => {
        //     : text / callback_query / etc...
        if (updateType === updateTypes.text && ctx.update.message.text === 'Hi') {
            await ctx.reply("Thank you!");
            // hobot       this:
            this.hobot.gotoPath(ctx, 'secondPath', { userJustSaid: "Hi" });
        } else {
            //     ,       
            await ctx.reply('We expect "Hi" text message here');
        }
    }
}

初めより少し複雑に見えますが、パスが100または200の場合、難易度は同じままです。

内部ロジックは自明であり、誰もこれをまだ行っていないことは驚くべきことです：
これらのコントローラーは、パスプロパティの値であるキーを持つオブジェクトに追加され、ユーザーアクション中またはhobot.gotoPath（ctx、path、data？ ）。

ナビゲーションは別のメソッドで割り当てられ、変数パスやナビゲーションロジックに触れないようにしますが、ctx.session.pathはいつでも手で変更できますが、もちろんお勧めできませんが、ビジネスロジックのみを考慮してください。

構造が破壊されない新しいボートを機能させるために必要なのは、通常のテレグラフボットを起動し、それと設定オブジェクトをHobotコンストラクターに渡すことだけです。configオブジェクトは、初期化するコントローラー、デフォルトのパス、およびコマンドとコントローラーのペアで構成されます。
