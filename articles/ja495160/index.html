モノリスから分散システムへ

銀行間の競争が絶えず成長しているため、顧客のさまざまなカテゴリーに適応する必要があります。したがって、1つはサイトにアクセスしてオンラインで銀行の商品を申請する方が簡単ですが、他の人は銀行の担当者とのライブコミュニケーション中に直接新商品やサービスを選択することに慣れています。2019年9月 ホームクレジットバンクは、クライアントの新しいプロセスを開始することを決定しました。その主な目的は、「クライアント-銀行オペレーター」の連絡を維持し、オペレーターが銀行のオフィスまたはショッピングセンターのカウンターに物理的に接続することを拒否することでした。

パイロットプロジェクトの開始日は2019年12月上旬に設定されました。その実装のために、できるだけ早く、銀行の新規および既存の顧客向けに個人用および非個人用のデビットカードを登録する機能を備えたシステムを開発する必要がありました。

新しいプラットフォームへの道


彼らはタブレットソリューションに目を向け始めました。オペレーターがクライアントと連携するための当行の現在のフロントオフィスシステムのアーキテクチャに基づいたタブレットでの新しいプロセスの実装は、時代遅れの技術スタックのために、現在のフロントオフィスは、8年前にSilverlightで記述されたモノリシックWebアプリケーションです。アプリケーションのUI部分が過負荷になり、アダプティブレイアウトが不足しているため、タブレットの現在の前面を操作する試みは失敗しました。さらに、MicrosoftのSilverlightサポートの欠如は、現在のアプリケーションのライフサイクルが終わりに近づいており、根本的な再設計と新しいテクノロジーへの移行の瞬間があったことを微妙にほのめかしました。マイクロサービスアーキテクチャを実装することにしました。モノリスを放棄する必要があったのはなぜですか？まず、ソリューションのスケーラビリティ、改善された全体的なフォールトトレランス、および独立したコンポーネントの更新により、第2に、世銀では、この場合、製品チームとマイクロサービスアプローチに機能を分散する傾向があるため、チームの柔軟性と独立性が向上します。最初の段階では、パイロットプロジェクトにユーザー、クライアント、デビットカード、インシデントのドメイン（マイクロサービス）が割り当てられました。アプリケーションのバックパーツを作成するために、彼らは.Net Core 2.2プラットフォーム（最近3.0に切り替えられました）を使用し、ビジネスロジックの要素をシステムのレガシーから借用しました。反応を利用してフロントパーツを作ることにしました。

UI / UX


アーキテクチャの定義と並行して、アプリケーションインターフェイスとプロセスビジネスロジックについて説明しました。オペレーターがクライアントとやり取りする方法、必要なドキュメントのパッケージ、オペレーターが特定の各販売ステップで必要とする情報を決定する必要がありました。目標は、単に機能を新しいプラットフォームにコピーするのではなく、現在のビジネスプロセスを簡素化することでした。そのため、新しい決定では、デビットカードアプリケーションが開かれたときにフィールドの数が減り、クライアントへのアンケートが除外されました（クライアントがホームクレジットバンクについて知った方法に関する情報を収集しました）。主要なデビット商品の登録にすばやくアクセスするための機能が、顧客データとともにメインページに追加されました。そして承認の段階で、アプリケーションは製品の関税と条件に関する詳細な情報を表示し、それもオペレーターを助けました。

生産的な環境での8年間、顧客データと製品とサービスを接続するための機能を含むフロントページのホームページは、無限のアイコンとフィールドが原因で過負荷になっています。



インターフェースに「空気」を加えたかった。ここでは、優先順位を付けて、メインページでアクセスするためにオペレーターが最初に必要とする機能を決定する必要がありました。この機能は、サイドパネルに表示でき、無用のために破棄できます（そうです、そのようなフィールドが見つかりました）。チームがデザイナーと実りある仕事をした結果、顧客データを含むページの最初のサムネイルと、デビットカード用のアプリケーションを作成する段階を含むページが設計されました。



ドキュメンテーション


現在のフロントオフィスのナレッジベースは、SharePointでの共同編集機能を備えたWordの「機能」です。新しいプロジェクトでは、セルフドキュメンテーションのためにSwaggerと組み合わせてConfluenceで新しいドキュメンテーションプロセスをパイロットすることにしました。移行パス、選択したソリューションの長所と短所については、別の記事で説明します。要件管理ツールの選択というトピックはまだオープンであると言えますが、リソースを最小限に投資してドキュメントを最新の状態に保つための最適なソリューションを探している段階です。

道に行こう！


その結果、切望されたパイロットを開始する方法に関するロードマップは次のようになりました。



ビルトインのチームインタラクションシステムは、パイロットの期限を把握することができました。プロジェクトの開始時に、チームは滝を放棄し、アジャイルトレイルに足を踏み入れなければなりませんでした。これは、私たちのチームが採用したアジャイルでした。私たちは並列化しました。開発者は実装に関するチームとの合意の段階で開始しました。これは通常、開発の開始前の要件に反映され、プロセスのすべての参加者に同意しました。未解決の質問は会議で議論されました。決定はチームによって一緒に行われました。チームの各メンバーはいつでもプロジェクトのステータスについて話し合うことができ、全員がプロセスに関与していると感じ、作業の結果に応じてパイロットを楽しみにしていました。

パイロット


パイロットは、新年の前夜にモスクワで最大のショッピングセンターの1つで行われました。パイロットプロジェクトでは、バンキング商品を知らず、以前は現在のフロントオフィスシステムで作業したことがない新しい販売スペシャリストを採用しました。パイロットの結果は、新しいユーザーが新しいアプリケーションで必要な一連のアクションを簡単に理解できることを示しました。つまり、そのようなスキームを他のサイトで立ち上げ、最小限の要件で新しいスペシャリストを募集できるということです。

新しいアプリケーションでは、オペレーターと顧客のニーズに基づいて、製品の機能を常に更新することを目指しました。そのため、ローンチの最初の週に、パイロットの結果に基づいて25の小さな改善がリリースされました。たとえば、契約締結時のドキュメントのパッケージには、製品の関税を記載した印刷フォームが追加されました。ほとんどの顧客は、オペレーターからタブレットの関税を見て、「手元に」印刷された関税を受け取りたいという要望を表明しました。

新しいプラットフォームによる販売活動は、Graphanaのレポートを通じてオンラインで監視され、契約が新たに締結されるたびに喜ばれました。



間違い


製品エラーを分析するときに現在のフロントオフィスに問題があります。特に、プロセスのビジネスロジックに埋め込まれていないアプリケーション画面（識別された事後）間に予測できない遷移がある場合は特に、オペレーターの一連のアクションを復元できるとは限りません。ユーザーへの訴えは通常、決定的ではありません。クライアントのフローが大きいため、特定の時点で特定のクライアントを使用してアプリケーションで実行された一連のアクションを復元することはできません。新しいシステムもこれらの問題を解決したいと考えていました。クライアントとオペレーターのセッションのフレームワーク内のすべてのマイクロサービスのエンドツーエンドのロギング、オペレーターの各アクションのテキストメッセージのログへの追加記録（新しいクライアントの作成、契約への署名、接続されたSMS通知）とフルスクリーンショットは、ビジネスプロセスの全体像をすばやく復元し、GrayLogでログを表示するときの製品エラーの分析を高速化しました。


パイロットの目標は、タブレットソリューションを実装することだけでなく、ホームクレジットバンクフロントオフィスシステムの将来のプラットフォームの基礎を築くことでもありました。今、私たちは改善を続けています。銀行のローン商品を扱うための新しいモジュール「クレジット」の追加に取り組んでいます。また、新しいテクノロジーの使用を調査するプロセスにおいても（graphql、その目的は、余分なデータを後ろから前に移動することではありません。ビジネスプロセスの柔軟な構成のためのカマンダ、レポートを構築するためのビジネスメトリックを収集するためのPrometheus）。

将来的には、さまざまな種類の銀行商品の販売機能を追加することで、タブレットソリューションの進歩が見られます。現在の優先事項は、銀行のクレジットカードを発行するプロセスを作成することです。さらに、電気店や年金基金の駐在員事務所へのタブレットの適用範囲を拡大します。
