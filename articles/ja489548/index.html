6か月前に、Chris PennerがBeating C With 80 Lines Of Haskell：Wcを公開しました。序文によれば、


課題は、Haskellのガベージコレクションプログラムを使用して、お気に入りの高級言語でCに最適化された手動で実装されたスマートクローンを構築することです。簡単に聞こえますか？wc

Chrisは、ByteStringsを使用した単純な実装から、モノイド、組み込みモノイドまで、最後に上記の並列マルチコアバージョンに到達しました。これにより、実行時に4つのコアで小さな純粋なCコードを実行できました。


数日前にハブレに同じトピックに関する別のメモが投稿されました 0xd34df00d C Haskell: wc。著者は、慣用的なHaskellと20行のコードを使用してアルゴリズムを実装する可能性を証明しました。これは、Cでの慣用的な実装よりもほぼ10倍高速です。


一方、私はHaskell（実際には依存型のためIdrisでさえ）とCoqを使用して、コードベースで重要な概念を実際に証明することを推奨しています。しかし、生産に入るすべてのものは、驚異的なフォールトトレランスのため、100％Elixir / Erlang / OTPのままです。私はerlang構文が好きな頑固なramではないことを確認したかったので、同じタスクの慣用的なエリクサーで何ができるかを確認することにしました。


以下に、私が実行を徐々にスピードアップして許容可能なレベルにするために使用したいくつかのトリックを示します。私はすでに大人で、サンタクロースを信じていません。エリクサーがマシンコードにコンパイルされた言語を実際に打ち負かすことができるという奇跡の希望はありませんでした。ただ最後に、サークルの勝者に遅れをとらないようにしたかっただけです。

@acc %{bc: 0, wc: 1, lc: 0, ns?: 1}

@type acc :: %{
        bc: non_neg_integer(),
        wc: non_neg_integer(),
        lc: non_neg_integer(),
        ns?: 0 | 1
      }

@spec parse(<<_::_*8>>, acc()) :: acc()
def parse("", acc), do: acc

def parse(
      <<?\n, rest::binary>>,
      %{bc: bc, wc: wc, lc: lc, ns?: ns}
    ),
    do: parse(rest, %{bc: bc + 1, wc: wc + ns, lc: lc + 1, ns?: 0})

def parse(
      <<?\s, rest::binary>>,
      %{bc: bc, wc: wc, lc: lc, ns?: ns}
    ),
    do: parse(rest, %{bc: bc + 1, wc: wc + ns, lc: lc, ns?: 0})

def parse(<<_, rest::binary>>, acc),
  do: parse(rest, %{acc | bc: acc.bc + 1, ns?: 1})
