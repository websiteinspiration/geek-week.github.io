プログレッシブHTMLダウンロードテクノロジーを使用したサーバー開始のクライアントへのデータ送信


最初の部分では、プリロードメカニズムを使用して、ページ処理の初期段階でクエリの実行を開始する方法について説明しました。つまり、そのような要求を開始するスクリプトがロードされる前でもです。これを踏まえると、マテリアルのプリロード段階でこれらのリクエストを実行しても、クライアントでHTMLページがレンダリングされる前に実行が開始されなかったことがわかります。そしてこれは、クライアントがサーバーに要求を送信し、サーバーがこの要求に応答する前に要求を開始できなかったことを意味します（ここで、サーバーがクライアントにHTML応答を生成するのにかかる時間も追加する必要があります）。次の図では、GraphQLクエリの開始がかなり遅れていることがわかります。そして、これは-それを考えるとHTMLタグにあるコードを使用して、そのようなリクエストの実行を開始します。<head>、そしてこれが、データのプリロードツールを使用して解決する最初のタスクの1つであるという事実。


実行前のリクエストは顕著な遅延で始まります

理論的には、そのようなGraphQLクエリの開始は、対応するページをロードするリクエストがサーバーに送信された瞬間を理想的に見るでしょう。しかし、サーバーから少なくともいくつかのHTMLコードを受信する前でも、ブラウザーが何かをロードし始めるようにするにはどうすればよいでしょうか。答えは、サーバーの主導でリソースをブラウザーに送信することです。このようなメカニズムを実装するには、HTTP / 2サーバープッシュのようなものが必要になるようです。しかし、実際には、クライアントとサーバー間の相互作用の同様のスキームを実装できる非常に古いテクノロジ（多くの場合、忘れられる）があります。このテクノロジーは、普遍的なブラウザーサポートによって区別されます。その実装では、HTTP / 2サーバープッシュの実装に典型的なインフラストラクチャの複雑さを詳しく調べる必要はありません。Facebookは2010年からこのテクノロジーを使用しています（詳細については、BigPipe）、およびEbayのような他のサイトでは、さまざまな形式のアプリケーションも見つかります。しかし、シングルページアプリケーションのJavaScript開発者は、基本的にこのテクノロジーを無視するか、単に使用しないかのように感じています。HTMLを段階的にロードすることです。このテクノロジーは、「アーリーフラッシュ」、「ヘッドフラッシュ」、「プログレッシブHTML」など、さまざまな名前で知られています。次の2つのメカニズムの組み合わせにより機能します。

  • 1つ目は、HTTPチャンク転送エンコーディングです。
  • 2つ目は、ブラウザでのHTMLのプログレッシブレンダリングです。

チャンク転送エンコーディング メカニズムはHTTP / 1.1で登場しました。ストリーミングモードでブラウザに送信される多くの小さな部分にHTTP応答を分割することができます。ブラウザはこれらの部品が到着すると「固定」し、それらから完全な応答コードを形成します。このアプローチは、サーバーでのページの形成方法に大きな変化をもたらしますが、ほとんどの言語とフレームワークは、そのような回答を部分に分割して配信することをサポートしています。 InstagramのWebフロントエンドはDjangoを使用するため、StreamingHttpResponseオブジェクトを使用します。このようなメカニズムの使用が有益である理由は、ページのコード全体が準備されるのを待つのではなく、ページの個々の部分の準備が整ったときに、ページのHTMLコンテンツをストリーミングモードでブラウザに送信できるためです。これは、リクエストを受け取った直後にブラウザーのページタイトルをフラッシュできることを意味します（「早期フラッシュ」という用語）。ヘッダーの準備には、特に大きなサーバーリソースは必要ありません。これにより、サーバーがページの残りの部分の動的データの生成でビジーな場合でも、ブラウザーはスクリプトとスタイルの読み込みを開始できます。この手法がどのような影響を与えるかを見てみましょう。これは通常のページの読み込みの様子です。


アーリーフラッシュテクノロジーは使用されません。リソースのロードは、ページのHTMLコードが完全にロードされるまで開始されません。

ただし、サーバーがリクエストに応じて、ページタイトルをすぐにブラウザーに送信するとどうなりますか。


アーリーフラッシュテクノロジーを使用：HTMLタグがブラウザーにリセットされた直後にリソースの読み込みが開始されます。

また、HTTPメッセージを部分的に送信するメカニズムを使用して、準備が整ったデータをクライアントに送信できます。サーバーでレンダリングされるアプリケーションの場合、このデータはHTMLコードの形式で表示できます。しかし、instagram.comのような単一ページのアプリケーションについて話している場合、サーバーはJSONデータのようなものをクライアントに送信することもできます。これがどのように機能するかを確認するために、単一ページのアプリケーションを起動する最も簡単な例を見てみましょう。

最初に、元のHTMLマークアップが、ページのレンダリングに必要なJavaScriptコードを含むブラウザーに送信されます。このスクリプトを解析して実行した後、XHRリクエストが実行され、ページのレンダリングに必要なソースデータが読み込まれます。


ブラウザが必要とするすべてのものをサーバーから独立して要求する状況でページをロードするプロセス

このプロセスには、クライアントがサーバーに要求を送信し、サーバーからの応答を待機するいくつかの状況が含まれます。その結果、サーバーとクライアントの両方が非アクティブになる期間があります。サーバーがクライアントからのAPIリクエストを待つのではなく、HTMLコードが生成された直後にサーバーがAPI応答の準備に取り掛かったほうが効率的です。回答の準備ができると、サーバーはそれ自体でクライアントに毒を与える可能性があります。これは、APIリクエストの完了後に以前にロードされたデータを視覚化するために必要なすべてをクライアントが準備するまでに、このデータが準備されている可能性が高いことを意味します。この場合、クライアントはサーバーへの個別の要求を満たし、サーバーからの応答を待つ必要はありません。

このようなクライアント/サーバー相互作用スキームを実装する最初のステップは、サーバー応答を格納するように設計されたJSONキャッシュを作成することです。システムのこの部分は、ページのHTMLコードに埋め込まれた小さなスクリプトブロックを使用して開発しました。これはキャッシュの役割を果たし、サーバーによってキャッシュに追加されるリクエストに関する情報を含みます（これは、簡略化された形式で、以下に示されています）。

<script type="text/javascript">
  //      API,       , 
  //     ,       ,  
  //      
  window.__data = {
    '/my/api/path': {
        waiting: [],
    }
  };

  window.__dataLoaded = function(path, data) {
    const cacheEntry = window.__data[path];
    if (cacheEntry) {
      cacheEntry.data = data;
      for (var i = 0;i < cacheEntry.waiting.length; ++i) {
        cacheEntry.waiting[i].resolve(cacheEntry.data);
      }
      cacheEntry.waiting = [];
    }
  };
</script>

HTMLコードをブラウザーにリセットした後、サーバーは個別にAPIリクエストを実行できます。これらのリクエストに対する回答を受け取った後、サーバーはJSONデータをこのデータを含むスクリプトタグの形式でページにダンプします。ブラウザーがページのHTMLコードの同様のフラグメントを受信して​​解析すると、データがJSONキャッシュに分類されることになります。ここで最も重要なことは、ブラウザがページを段階的に表示することです。つまり、応答のフラグメントを受信すると（つまり、スクリプトの完成したブロックは、ブラウザに到着すると実行されます）です。つまり、対応するデータの準備ができるとすぐに、サーバー上で大量のデータを同時に生成し、スクリプトブロックをページにドロップすることが可能です。これらのスクリプトはすぐにクライアントで実行されます。これは、Facebookが使用するBigPipeシステムの基盤です。そこでは、多くの独立したページャーがサーバーに並行してロードされ、利用可能になるとクライアントに送信されます。

<script type="text/javascript">
  window.__dataLoaded('/my/api/path', {
    // JSON- API,      , 
    //    JSON-...
  });
</script>

クライアントスクリプトは、XHRリクエストを実行する代わりに、必要なデータをリクエストする準備ができたら、まずJSONキャッシュをチェックします。キャッシュにすでにクエリ結果がある場合、スクリプトはすぐに必要なものを取得します。リクエストが進行中の場合、スクリプトは結果を待っています。

function queryAPI(path) {
  const cacheEntry = window.__data[path];
  if (!cacheEntry) {
    //   XHR-  API
    return fetch(path);
  } else if (cacheEntry.data) {
    //      
    return Promise.resolve(cacheEntry.data);
  } else {
    //       ,
    //        
    //   
    const waiting = {};
    cacheEntry.waiting.push(waiting);
    return new Promise((resolve) => {
      waiting.resolve = resolve;
    });
  }
}

これらすべてが、ページをロードするプロセスが次の図と同じになるという事実につながります。


ブラウザーがクライアントのデータの準備に積極的に関与している状況でページをロードするプロセス

これをページをロードする最も簡単な方法と比較すると、サーバーとクライアントが並行してより多くのタスクを実行できることがわかります。これにより、サーバーとクライアントが相互に待機するダウンタイムが短縮されます。

この最適化は、システムに非常に良い影響を与えました。そのため、デスクトップブラウザでは、ページの読み込みが以前より14％速く完了し始めました。また、モバイルブラウザでは（モバイルネットワークの遅延が長いため）、ページの読み込みが23％速くなりました。
