展開前にスマートコントラクトのアドレスを決定する方法：暗号交換にCREATE2を使用する

ブロックチェーンの話題はあらゆる種類の誇大広告の源であるだけでなく、技術的な観点から非常に貴重なアイデアでもあります。したがって、彼女は日当たりの良い街の住民を迂回しませんでした。人々は、ブロックチェーンシステムへの従来の情報アプローチの専門知識をシフトすることを目指し、研究し、試みています。これまでのところ、ほんのポイントです。Rostelecom-Solar開発の1つは、ブロックチェーンに基づいてソフトウェアのセキュリティをチェックできます。そしてその過程で、ブロックチェーンコミュニティの適用された問題を解決することについていくつかの考えが生じます。これらのライフハックの1つ-CREATE2を使用して展開する前にスマートコントラクトのアドレスを決定する方法-今日、私はカットの下で共有したいと思います。


CREATE2オペコードは、今年2月28日にコンスタンチノープルのハードフォークに追加されました。EIPに示されているように、このオペコードは主に州のチャネルのために導入されました。ただし、別の問題を解決するために使用しました。

残高のある取引所にユーザーがいます。各ユーザーには、誰でもトークンを送信できるイーサリアムアドレスを提供する必要があります。これにより、アカウントが補充されます。これらのアドレスを「ウォレット」と呼びましょう。トークンがウォレットに到着したとき、それらを単一のウォレット（hotwallet）に送信する必要があります。

次のセクションでは、CREATE2を使用せずにこの問題の解決策を分析し、なぜそれを放棄したかを説明します。最終結果のみに関心がある場合は、「最終的な解決策」セクションで見つけることができます。

イーサリアムアドレス


最も簡単な解決策は、新しいユーザーのために新しいイーサリアムアドレスを生成することです。これらのアドレスはウォレットになります。トークンをウォレットからホットウォレットに転送するには、バックエンドからウォレットの秘密鍵を使用してtransfer（）関数を呼び出し、トランザクションに署名する必要があります。

このアプローチには次の利点があります。

  • それは簡単です
  • トークンをウォレットからホットウォレットに転送するコストは、transfer（）関数を呼び出す料金と同じです

ただし、このアプローチは、秘密鍵をどこかに保存する必要があるという重大な欠点があるため、放棄しました。そしてポイントは、それらが失われる可能性があるだけでなく、これらのキーへのアクセスを注意深く制御する必要があることです。それらの少なくとも1つが侵害された場合、特定のユーザーのトークンはホットウォレットに到達しません。



ユーザーごとに個別のスマートコントラクトを作成する


ユーザーごとに個別のスマートコントラクトを展開すると、プライベートウォレットキーをサーバーに保存する必要がなくなります。取引所はこのスマートコントラクトを呼び出して、トークンをホットウォレットに転送します。

また、スマートコントラクトを展開しないとユーザーにウォレットのアドレスを表示できないため、この決定も拒否しました（これは実際には可能ですが、他の欠点があるため、ここでは説明しません）。エクスチェンジでは、ユーザーは必要な数のアカウントを作成でき、誰もが自分のウォレットを必要とします。これは、ユーザーがこのアカウントを使用するかどうかさえわからなくても、契約の展開にお金を費やす必要があることを意味します。

オペコードCREATE2


前の方法の問題を修正するために、CREATE2オペコードを使用することにしました。CREATE2を使用すると、スマートコントラクトが展開されるアドレスを事前に決定できます。アドレスは次の式を使用して計算されます。

keccak256 (0xff ++ address ++ salt ++ keccak256 (init_code)) [12:]

どこ：
  • address -CREATE2を呼び出すスマートコントラクトのアドレス
  • ソルト -ランダムな値
  • init_code-デプロイメント用のスマートコントラクトバイトコード

これにより、ユーザーに提供するアドレスに確実に目的のバイトコードが含まれるようになります。さらに、このスマートコントラクトは、必要なときに展開できます。たとえば、ユーザーが初めて自分の財布を使用することを決定したとき。

さらに、スマートコントラクトのアドレスを保存する代わりに毎回計算できます。

  • address ,
  • salt — user_id
  • init_code ,


以前のソリューションには1つの欠点があります。それは、スマートコントラクトを展開するために支払う必要があることです。ただし、これは取り除くことができます。これを行うには、呼び出すことができます転送（）関数を、次に（）selfdestruct財布コンストラクタで。そして、スマートコントラクトの展開のためのガスが返されます。

一般的な考えに反して、CREATE2オペコードを使用して同じアドレスにスマートコントラクトを数回展開できます。これは、CREATE2がターゲットアドレスのナンスがゼロであることを確認するためです（コンストラクターの先頭で値「1」が割り当てられています）。この場合、selfdestruct（）関数は毎回nonceアドレスをリセットします。これにより、同じ引数を指定してCREATE2を再度呼び出すと、ノンスのチェックに合格します。

このソリューションはイーサリアムアドレスオプションに似ていますが、秘密鍵を保存する必要がないことに注意してください。ウォレットからホットウォレットへの送金のコストは、スマートコントラクトのデプロイに料金がかからないため、transfer（）関数を呼び出すコストとほぼ同じです。

最終決定




最初に準備されたもの：

  • user_idでソルトを取得する関数
  • 適切なソルトを使用してCREATE2オペコードを呼び出すスマートコントラクト（つまり、ウォレットファクトリ）
  • 次のコンストラクタとの契約に対応するウォレットバイトコード：

constructor () {
    address hotWallet = 0x…;
    address token = 0x…;
    token.transfer (hotWallet, token.balanceOf (address (this)));
    selfdestruct (address (0));
}

新しいユーザーごとに、計算してウォレットのアドレスを表示します

keccak256 (0xff ++ address ++ salt ++ keccak256 (init_code)) [12:]

ユーザーがトークンを対応するウォレットアドレスに転送すると、バックエンドには、_toパラメーターがウォレットアドレスと等しいTransferイベントが表示されます。この時点で、ウォレットをデプロイする前に、取引所でユーザーの残高を増やすことがすでに可能です。

ウォレットアドレスに十分な数のトークンが蓄積されると、それらを一度にホットウォレットに転送できます。これを行うために、バックエンドはスマートコントラクトファクトリの関数を呼び出し、次のアクションを実行します。

function deployWallet ( uint256) {
    bytes memory walletBytecode =…;
    // invoke CREATE2 with wallet bytecode and salt
}

したがって、スマートウォレットコントラクトのコンストラクターが呼び出され、すべてのトークンがホットウォレットアドレスに転送された後、自己破壊されます。

完全なコードはここにあります。ウォレットのバイトコードを最適化してオペコードに書き込んだため、これは本番のコードではないことに注意してください。
