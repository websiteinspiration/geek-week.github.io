Flask Webアプリケーション：循環インポートの処理方法

Flaskは最も人気のあるPythonフレームワークの1つですが、それを使用するときにいくつかのエラーが発生すると、特定の問題が発生する可能性があります。この記事では、プロジェクトで循環インポートが発生しないようにする方法について説明します。



フラスコと周期的インポート


フラスコを使用する開発者は、モジュール間の依存関係の問題に直面することがよくあります。ビューとモデルを宣言するために、開発者はメインモジュール（「エントリポイント」）で作成および初期化されたグローバルオブジェクトを使用します。同時に、彼は周期的な輸入を受けるリスクがあり、そのためプロジェクトの維持は困難になります。

この問題を解決するためのドキュメントと基本的なフラスコチュートリアルでは、プロジェクトの初期化コードを__init__.pyに配置することをお勧めします。これにより、Flaskインスタンスクラスが作成され、アプリケーションが構成されます。これにより、パッケージのスコープからすべてのグローバルオブジェクトにアクセスできます。

このアプローチを使用すると、構造は次のようになります。

.
├── app
│   ├── __init__.py
│   ├── forms.py
│   ├── models.py
│   ├── views.py
│   └── templates
├── config.py
└── migrations

app / __ init__.py

import flask
from flask_mail import Mail
# other extensions

app = Flask(__name__)
mail = Mail(app)
# configure flask app

from app import views, models

app / views.py

from app import app

@app.route('/view_name/'):
def view_name():
     pass

すべてのコンポーネントが強力に接続されているため、明らかに、このアーキテクチャはあまり成功していません。その後、1か所でコードを変更すると、他の12か所で変更が発生するため、このようなプロジェクトを完成させることは困難です。

原則として、この問題は次のように解決します。

  • 標準ルーティングを避けます。
  • 「ラッパー」なしのオリジナルバージョンのライブラリをお勧めします。
  • 依存性注入を使用します。

これを詳しく見てみましょう。

上品で働く


ドキュメントで説明されている標準のルーティング方法の代わりに、上品な方法を使用できます。このアプローチでは、ビューのルーティングを手動で記述する必要はありません。ルーティングは、クラスとメソッドの名前に基づいて自動的に構成されます。これにより、コードの構造を増やしたり、アプリオブジェクトなしでビューを宣言したりできます。その結果、循環輸入の問題を解決することが可能です。フラスコクラスフル

コードを検討するときは、ルートにあるapp.pyで初期化が行われることに注意する必要があります。アプリケーションは、ブループリントを使用して構成されたサブプロジェクトに分割され、アプリオブジェクトに1行として登録されます。

ライブラリのオリジナルバージョンの設定


上記のコードは、フラスコクラスフルが循環インポートの処理にどのように役立つかを示しています。従来のフラスコプロジェクトでのこの問題の原因は、ビュー宣言といくつかの拡張の両方である可能性があります。顕著な例の1つは、フラスコSQLALCHEMYです。

フラスコ-sqlalchemy拡張機能は、sqlalchemyとフラスコの統合を改善するように設計されていますが、実際には、プロジェクトの利点よりも多くの問題をもたらすことがよくあります。

  • 拡張機能は、モデルの宣言を含む、データベースとのやり取りのためのグローバルオブジェクトの使用を促進しますが、これも循環インポートの問題につながります。
  • 独自のクラスを使用してモデルを記述する必要があるため、モデルがフラスコプロジェクトに緊密にバインドされます。その結果、これらのモデルはサブプロジェクトや補助スクリプトでは使用できません。

これらの理由から、フラスコ-sqlalchemyは使用しないようにしています。

依存関係注入パターンの使用


上品なアプローチを実装し、フラスコ-SQLalchemyを放棄することは、循環インポートの問題を解決するための最初のステップにすぎません。次に、アプリケーション内のグローバルオブジェクトへのアクセスロジックを実装する必要があります。これには、依存性インジェクターライブラリに実装されている依存性インジェクションパターンを使用すると便利です。

dependency-injectorライブラリを使用したコードでパターンを使用する例：

app.py

import dependency_injector.containers as di_cnt
import dependency_injector.providers as di_prv
from flask import Flask
from flask_mail import Mail

from app import views as app_views
from app import routes as app_routes

app = Flask(__name__)
mail = Mail(app)

#  blueprints
app.register_blueprint(app_routes.app_blueprint)

#  providers
class DIServices(di_cnt.DeclarativeContainer):
    mail = di_prv.Object(mail)

# injection
app_views.DIServices.override(DIServices)


app /routes.py

from os.path import join

from flask import Blueprint

import config
from app import views

conf = config.get_config()

app_blueprint = Blueprint(
    'app', __name__, template_folder=join(conf.BASE_DIR, 'app/templates'),
    static_url_path='/static/app', static_folder='static'
)

views.AccountView.register(app_blueprint, route_base='/')

app / views.py

import dependency_injector.containers as di_cnt
import dependency_injector.providers as di_prv
from flask_classy import FlaskView
from flask_login import login_required

class DIServices(di_cnt.DeclarativeContainer):
    mail = di_prv.Provider()

class AccountView(FlaskView):

    def registration(self):
        #  
        msg = 'text'
        DIServices.mail().send(msg)

    def login(self):
        pass

    @login_required
    def logout(self):
        pass

記事に記載されている対策は、周期的なインポートを排除し、コードの品質を向上させることができます。上記のアプローチを使用して、フラスコプロジェクトがどのように見えるかを、Webアプリケーションの形式で作成されたゲーム「Bulls and Cows」の例を使用して確認します。
