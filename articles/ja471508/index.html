Armless admin = hyper convergence？




これは、サーバーハードウェアの分野では非常に一般的な神話です。実際には、ハイパーコンバージドソリューション（すべてが1つになっている場合）には、何のために多くのことが必要です。歴史的に、最初のアーキテクチャはサービスのためにAmazonとGoogleによって開発されました。次に、それぞれが独自のドライブを持つ同じノードのコンピューティングファームを作成するというアイデアでした。これらすべては、いくつかのシステム形成ソフトウェア（ハイパーバイザー）によって結合され、すでに仮想マシンに分割されていました。主なタスクは、1つのノードを維持するための最小限の労力と最小限のスケーリング問題です。同じサーバーをさらに数千または2つ購入し、近くに接続しました。実際には、これらは分離されたケースであり、多くの場合、少数のノードとわずかに異なるアーキテクチャについて話します。

しかし、プラスは変わりません-スケーリングと制御が驚くほど簡単になります。マイナス-異なるタスクは異なる方法でリソースを消費し、どこかに多くのローカルディスクがあり、どこかにRAMが少ないなど、つまり、異なるタイプのタスクでは、リソース使用率が低下します。

設定を簡単にするために、10〜15％高い金額を支払うことがわかりました。これがタイトルから神話を引き起こした。テクノロジーが最適に適用される場所を長い間探して見つけました。実際、Tsiskaには独自のストレージシステムがありませんでしたが、完全なサーバー市場が必要でした。そして、ノード上のローカルストレージソリューションであるCisco Hyperflexを作成しました。

そして、これは突然、バックアップデータセンター（災害復旧）にとって非常に優れたソリューションであることがわかりました。なぜ、どのように-今私は教えてくれます。そして、クラスターテストを表示します。

もともとはVMware向けに作成され、より多くの機能を備えているため、VMwareに特化したソリューションについて検討します。Hyper-Vは、競合他社に追いつき、市場の期待に応える方法に沿って追加されました。

ディスクでいっぱいのサーバーのクラスターがあります。データストレージ用のディスク（SSDまたはHDD-好みとニーズに応じて）、キャッシュ用のSSDディスクが1つあります。データがデータストアに書き込まれると、データはキャッシングレイヤー（専用SSDディスクとサービスVM RAM）に保存されます。並行して、データブロックがクラスター内のノードに送信されます（ノードの数はクラスターのレプリケーション係数によって異なります）。正常な記録についてすべてのノードからの確認後、記録の確認がハイパーバイザーに送信され、次にVMに送信されます。バックグラウンドで記録されたデータは重複排除され、圧縮されてストレージディスクに書き込まれます。同時に、大きなブロックは常にストレージディスクに順次書き込まれるため、ストレージディスクの負荷が軽減されます。

重複排除と圧縮は常にオンになっており、無効にすることはできません。データは、ストレージディスクまたはRAMキャッシュから直接読み取られます。ハイブリッド構成が使用されている場合、読み取りもSSDにキャッシュされます。

データは仮想マシンの現在の場所に関連付けられておらず、ノード間で均等に分散されます。このアプローチにより、すべてのドライブとネットワークインターフェイスを均等にロードできます。ローカルでのデータの可用性が保証されていないため、明らかなマイナス要素として、読み取り遅延を最小限に抑えることはできません。しかし、私はこれは受け取ったプラスと比較して取るに足らない犠牲だと思います。さらに、ネットワークの遅延は、実際には全体的な結果に影響を与えないような値に達しています。

ディスクサブシステムのすべてのロジックについて、Cisco HyperFlex Data Platformコントローラーの特別なサービスVMが責任を持ち、これは各ストレージノードに作成されます。私たちのサービスVM構成では、8つのvCPUと72 GBのRAMが割り当てられていますが、これはそれほど小さくありません。ホスト自体に28の物理コアと512 GBのRAMがあることを思い出してください。

サービスVMは、SASコントローラーをVMに転送することにより、物理ディスクに直接アクセスできます。ハイパーバイザーとの通信は、I / O操作をインターセプトする特別なIOVisorモジュールと、ハイパーバイザーAPIにコマンドを転送できるエージェントを使用して行われます。エージェントは、HyperFlexのスナップショットとクローンの操作を担当します。

ハイパーバイザーでは、ディスクリソースはNFSまたはSMBボールとしてマウントされます（ハイパーバイザーのタイプによって異なります）。内部的には、これは、シンボリュームの割り当て、圧縮と重複排除、リダイレクトオンライトテクノロジーを使用したスナップショット、同期/非同期レプリケーションなど、本格的な本格的なストレージシステムの機能を追加できる分散ファイルシステムです。

サービスVMは、HyperFlexサブシステムのWEB管理インターフェースへのアクセスを提供します。 vCenterとの統合があり、毎日のタスクのほとんどはそこから実行できますが、たとえばデータストアは、すでに高速HTML5インターフェイスに切り替えている場合や、完全統合の完全Flashクライアントを使用している場合は、別のWebカメラから切り出す方が便利です。サービスWebカメラでは、システムのパフォーマンスと詳細なステータスを確認できます。



クラスターには、計算ノードという別の種類のノードがあります。内蔵ドライブのないラックサーバーまたはブレードサーバーにすることができます。これらのサーバーでは、ディスクを持つサーバーにデータが保存されているVMを実行できます。データへのアクセスの観点からは、アーキテクチャにはデータの物理的な場所からの抽象化が含まれるため、ノードのタイプに違いはありません。計算ノードとストレージノードの最大比率は2：1です。

計算ノードを使用すると、クラスターリソースをスケーリングする際の柔軟性が向上します。CPU/ RAMのみが必要な場合は、ディスクを備えたノードを購入する必要はありません。さらに、ブレードバスケットを追加して、ラックサーバースペースを節約できます。

テストとアーキテクチャの研究の印象では、すべてのホスト間にデータを配置するアルゴリズムにより、スケーラブルな予測可能なパフォーマンスが得られますが、ローカルディスクなどからより多くを絞り込める可能性があるため、これは読み取り時の制限でもあります。たとえば、より生産的なネットワークを節約するために、40 GbpsのFIを利用できます。

実際の使用


バックアップデータセンターを編成するには、2つの方法を使用できます（リモートサイトにバックアップを配置することは考慮していません）。

  1. アクティブパッシブ すべてのアプリケーションはメインデータセンターでホストされます。レプリケーションは同期または非同期です。メインデータセンターで障害が発生した場合は、バックアップデータセンターをアクティブにする必要があります。これは手動で行うことができます/スクリプト/オーケストレーションアプリケーション。ここでは、レプリケーションの頻度に見合ったRPOを取得します。RTOは、管理者の反応とスキル、および切り替え計画の開発/デバッグの品質に依存します。
  2. アクティブアクティブ この場合、同期レプリケーションのみが存在し、データセンターの可用性は、厳密に3番目のサイトに配置されたクォーラム/アービターによって決定されます。RPO = 0であり、RTOは0（アプリケーションで許可されている場合）または仮想化クラスター内のノードをフェールオーバーする時間に到達できます。仮想化レベルでは、アクティブ/アクティブストレージを必要とする拡張（メトロ）クラスターが作成されます。

通常、メインのデータセンターにクラシックストレージを備えたアーキテクチャが既に実装されているクライアントを目にするため、レプリケーション用に別のアーキテクチャを設計します。前述したように、Cisco HyperFlexは非同期複製と拡張仮想化クラスターの作成を提供します。同時に、2つのストレージシステム上に高価なレプリケーションとActive-Activeデータアクセス機能を備えた専用のミッドレンジ以上のストレージシステムは必要ありません。

シナリオ1：VMware vSphereの仮想化プラットフォームであるプライマリデータセンターとバックアップデータセンターがあります。すべての生産性の高いシステムは主にデータセンターに配置され、仮想マシンのレプリケーションはハイパーバイザーレベルで実行されます。これにより、バックアップデータセンターでVMをオンのままにしておくことができなくなります。データベースと特別なアプリケーションを組み込みのツールで複製し、VMをオンのままにします。メインデータセンターに障害が発生した場合は、バックアップデータセンターでシステムを起動します。仮想マシンは約100台あると考えています。メインデータセンターが稼働している間、バックアップデータセンターでテスト環境やその他のシステムを起動できます。メインデータセンターが切り替えられた場合、テスト環境やその他のシステムを無効にすることができます。双方向レプリケーションを使用することも可能です。設備の観点からは、何も変わりません。

クラシックアーキテクチャの場合、ハイブリッドストレージシステムを各データセンターに配置し、FibreChannel、ティアリング、重複排除、圧縮（オンラインではない）、サイトあたり8台のサーバー、2台のFibreChannelスイッチ、イーサネット10Gを介してアクセスします。従来のアーキテクチャでのレプリケーションとスイッチング制御については、VMwareツール（レプリケーション+ SRM）またはサードパーティ製のツールを使用できます。これらのツールは、わずかに安く、時には便利です。

VMware Ent Plus

すべての計算において、ネットワークインフラストラクチャ、データセンターのコストなどは考慮していません。これらは、従来のアーキテクチャでもHyperFlexソリューションでも同じです。

コストでは、HyperFlexは5％高くなります。ここで注目に値するのは、CPU / RAMリソースの場合、Ciscoにバイアスがかかったことです。構成では、メモリコントローラーのチャネルが均等に満たされたためです。コストはわずかに高くなりますが、桁違いではありません。これは、ハイパーコンバージェンスが必ずしも「金持ちのおもちゃ」ではないことを示していますが、データセンターを構築する標準的なアプローチと競合する可能性があります。また、すでにCisco UCSサーバーとそれに対応するインフラストラクチャを持っている人にとっても興味深いかもしれません。

利点には、SANとストレージの管理コストがないこと、オンラインでの圧縮と重複排除、サポート（仮想化、サーバー、ストレージシステム）の単一のエントリポイント、スペースの節約（すべてのシナリオではない）、操作の簡素化などがあります。

サポートに関しては、ここでは1つのベンダー-Ciscoから入手します。Cisco UCSサーバーの経験から判断すると、私は気に入っています。HyperFlexで開く必要はありませんでした。すべてがそのように機能しました。エンジニアは迅速に対応し、一般的な問題だけでなく、複雑な境界ケースも解決できます。時々私は彼らに質問を向けます：「そうすることは可能ですか、それを留めますか？」または「ここで何かを設定しましたが、機能しません。助けて！" -彼らはそこに必要なガイドを辛抱強く見つけ、正しい行動を指摘し、答えます：「ハードウェアの問題のみを解決します」彼らはしません。
