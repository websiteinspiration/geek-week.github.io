クラウドプロバイダーの目を通してCisco UCS

こんにちは、ハブル！

クラウドプロバイダーであることは、常に新しい知識と専門知識を蓄積することを意味します。長年にわたり、私たちはサービスの最高レベルを保証するために遵守しようとするかなり多数の慣行を形成してきました。それらの1つは、Cisco Unified Computing Systemソリューションの使用です。私たちの意見では、UCSがプロバイダーにとって最高のソリューションの1つである理由をお伝えし、作業のいくつかの機能とシステムの使用例について説明します。

Cisco UCSが市場に出てから約8年が経過しました。これは、聴衆がテクノロジーの全体像を把握するのに十分な期間であり、マニュアル、レビュー、トレーニング記事から、膨大な本のボリュームを編集できます。ただし、このトピックに関するマーケティング記事からは、2巻になります。Cisco UCSについてできる限り客観的に説明するように努めます。ソリューションの主要な機能に焦点を当て、それらに基づいてクラウドサービスプロバイダーのメリットについて話し合い、事例を共有します。

初めは言葉でした


「収束」という用語は、約10年前にHPエンジニアによって使用されるようになりました。実際、HPは、HPコンバージドモジュールと呼ばれるHP BladeSystem c7000シャーシへのインストール用にリリースした最初の企業です。たとえば、特定のブレードサーバーに特定の帯域幅を割り当てることが可能になりました。これはコンバージェンスへの最初のステップでしたが、このソリューションはコンバージドシステムに必要なすべての機能を備えていませんでした。

念のために説明しましょう。集中型インフラストラクチャは、機器とソフトウェアの単一の複合体であり、複合体に含まれるすべての機器とオーケストラを管理するための単一のエントリポイントがあります。

Cisco UCSに関しては、このソリューションは、機器およびソフトウェアパッケージの一部に関する収束の定義と完全に一致しています。

ソリューションアーキテクチャ


上記のスキームを注意深く検討し、複雑な「トップダウン」の要素について簡単に説明します。



Cisco UCS Managerソフトウェア

図に示されているすべてのハードウェアコンポーネントを管理するための単一のエントリポイント、およびコンポーネントを手動またはREST APIを介して管理できるオーケストレータ。これは、複合体の一種の「頭脳」です。ファブリックインターコネクト内にインストールされます。例外なく、すべての機器の設定は、管理インターフェイス（GUIまたはCLI）またはUCS Manager APIを介して実行されます。

ファブリックインターコネクト

Cisco Nexusに基づくハードウェア統合スイッチ。複合システムのすべてのコンポーネントのネットワーク接続、および外部サーバーとのブレードサーバーの接続を提供します。コンプレックスには2つのファブリックインターコネクトが含まれます。最新バージョンのFI6332およびFI6454では、最大20台のシャーシ5108を接続でき、この場合のブレードサーバーの総数は次のようになります。

  • b480 M5-最大80台のサーバー。
  • b200 M5-最大160台のサーバー。

現在、これは単一のエントリポイント内で統合の機会を提供し、追加のToRスイッチやブレードサーバーと共にシャーシにインストールされたその他のモジュールを使用せずにシームレスなネットワーク接続をサポートするほぼ唯一のソリューションです。

シャーシc5108

FIと比較すると、これらはかなりシンプルなデバイスです。それらのレイアウトはブレードシステムの標準です：PSU、ファン、およびシャーシの主要コンポーネント-ブレードサーバーとFI間の接続を提供するFEXモジュール。執筆時点では、4ポートの40GbEモジュール2304と8ポートまたは4ポートの10GbEモジュール2204の両方がサポートされています。これらの特徴的な機能は、ポートをグループ化して、全体的な帯域幅を拡大できることです。

VIC（仮想インターフェイスカード）

ブレードサーバーにインストールされているインテリジェントアダプター。ハードウェアサーバーと仮想マシンの両方に仮想ネットワークリソースを割り当てることができます。EthおよびFC / FCoEデータ転送プロトコルをサポートします。

ソリューションデバイスがある程度明確になったので、主観的な観点から、Cisco UCSが市場で最も便利なソリューションの1つである理由について話しましょう。

Cisco UCSを選ぶ理由


ソリューションの構成が明確になったので、その利点について話しましょう。シスコのソリューションは、その「親戚」よりもどのように優れていますか-たとえば、同じHP Synergy この質問は、私たちの同僚からよく聞かれますが、答えは（私たちには）表面にあります。ポイントはこれです：

  • ユニバーサルソリューション、「統一」という用語の遵守⇒OPEXの低下。
  • 最小限の設備により、最大数のケース（詳細は以下を参照）、およびスケーリングの容易さ⇒CAPEXを下げることができます。
  • 優れたパフォーマンスと負荷分散、エンタープライズレベルの可用性。

事実、これら3つのポイントでは、ソリューションのすべての主要な要件がビジネス側とIT側の両方から集中しています。それにも関わらず、これらの利点は根拠のないものに見えないため、実際の例を挙げてさらに「解読」します。

実用化


この記事の冒頭で約束したように、このセクションでは、Cisco UCSのケーススタディについて説明します。私たちは経験のレビューから始めて、特定の状況にスムーズに進みます。

機器の試運転


Cisco UCSソリューションを使用している間、オンラインで8つのシステム（複雑とは、ファブリックインターコネクトと少なくとも1つのブレードシャーシのペアを意味します）を合計16 FI以上で稼働させる必要がありました。 2014年に運用を開始した最初の複合施設。このプロセスには3日かかりました。そのうち2日は、ドキュメントの調査と機器のロジックの理解に費やされました。シスコのドキュメントは、IBMの最高のRedBookのレベルで構成されていることに注意してください。よく知っている人であれば、比較を理解できます。

構成のロジックと基本原則を扱ったので、装置を簡単に組み立てて起動しました。次に、すべてのコンポーネントのファームウェアを更新し、サーバープロファイルテンプレートをセットアップして、プロファイルを作成しました。1営業日で。

標準のITIL変更管理手順の一部として追加の実装が実行され、必要なすべてのテンプレートとポリシーの作成と構成を含め、電源投入の瞬間からシャーシが完全に使用できるようになるまで、FIの各ペアと1つまたは2つのシャーシを展開するのに4時間もかかりませんでした。

REST APIおよびPowerToolsモジュールを使用すると、インストールプロセスを高速化できます。たとえば、500以上のVLANを新しいインストールにコピーするには、PowerToolsを使用して、次の2つの簡単なステップで実行します。

  • 本番インフラストラクチャからVLANリストを取得する
  • VLANリストを新しいコンプレックスにアップロードします。

インフラストラクチャのスケーリングは、ブレードサーバーを備えた新しいシャーシを取り付けられたFIペアに接続することによって実行されます（必要な数の空きポートがある場合）。手順は100％オンラインであり、Cisco UCS Managerインターフェイスから実行できます。正しいグローバル設定を使用すると、シャーシが接続されているFIポートが目的の動作モードに切り替わった直後に、これらのポートがポートチャネルに自動的に収集されます。次に、以下のフレームワーク内で、自動確認手順が起動されます。

  • すべてのシャーシコンポーネントを最新バージョンのFWに更新します。
  • 電力上限設定;
  • FEXのバックプレーンポートを、これらのポートに必要なファクトリおよびポートチャネルアセンブリにマッピングします。

繰り返しになりますが、これらはすべて、施設の試運転中に設定されたグローバルポリシーに基づいて、エンジニアの介入なしに行われたことを思い出します。

やがて、そのような手順は約1時間半かかります。新しいシャーシをFIに物理的に接続するには、FEXをFIのポートに切り替え、DACケーブルを最適に使用します。また、シスコからオリジナルのケーブルを入手する必要はまったくありません。

ポリシーは、サーバーのテンプレートプロファイルに関連付けられています。次に、すべてが簡単です。テンプレートを目的のブレードサーバーにバインドして初期化します。出力は、オペレーティングシステムのインストールの準備ができているサーバーです。サーバーの初期化にかかる時間は10〜20分以内で、必要な数のサーバーに対して同時に実行できます。合計で、わずか25〜35分で、OSのインストールの準備が完全に整った80〜160台のサーバーを取得できます。もちろん、インストールプロセスを自動化することもでき、Cisco UCS APIはこのタスクに役立ちますが、これは別の記事のトピックです。
