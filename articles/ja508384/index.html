暴力に頼らずに開発者をあなたのビジネスの価値に慣れる方法

著者から



トピックは物議を醸し、相互間のコミュニケーションのルールと方法について、特に開発者とクライアントのコミュニケーションについて考えていない開発者とマネージャーが非常に多くいます。



しかし、私と私のチームにとって、そのような障壁は克服され（そして2回以上）、優れた結果をもたらしたので、私の成功した経験を共有せざるを得ません：



そう





ビジネスはビジネス、開発は開発、そして...


猫と犬を友達にする方法の記事を見ました。母親の行動主義者によるライフハック：



「犬を横にして、猫を背中に背負ってください。彼らはお互いを見るのではなく、匂いだけを見ます。この演習は、動物への信頼を築くのに役立ちます。リラックスできる音楽で両方を撫で始めてください。」



おそらく、このソリューションはニッチ種間3種に適しています。私はそれを試しました-私はほとんど目と猫を失いました。



同じ損失と感情について、「ビジネス」と「開発」の導入の問題は私を引き起こします。



しかし、この文脈では、私のケースはそれほどトラウマではありません。私は、ビジネスと開発の導入は計量的で目立たないようにする必要があると思います。



翻訳の難しさ



多くの場合、開発者はあなたのビジネスやクライアントのビジネスを表面的に理解していません。次に、彼は指示



をオンにします。ステップ1.来て、RUSTのすべてを書き直すように要求します。



ステップ2. RUSTのすべてを書き直すことはできないので、バーンアウトします。



if (___RUST == false) then .({immediately: true})





では、なぜ開発者はあなたのビジネスを理解したくないのでしょうか？



意味と目的を求めて



他の人々 に利益をもたらすということは、どのビジネスの目標も同じです。この目的のために、彼らは法定のお金と社会資本で支払います。



私の経験では、開発者は多くの場合、他の人にもたらすメリットを理解していません。



したがって、彼が自分の貢献を評価することは困難です-彼はこのコードのコードとお金しか見ていません。



したがって、彼の動機（RUSTのすべてを書き直す）は「トップコードを書く」ことに基づいており、「ユーザー向けに機能をリリースする（「利点」の同義語）」ではありません。



説明する



通常、企業にはさまざまな部門があります。マーケティング、テクニカルサポート、分析、販売、戦略です。そして誰からも遠く離れて、「開発部門」と呼ばれる半自律的な予約があり、彼らは特別な外交官翻訳者（PM、チームリーダー）を通じて連絡します。



実際には、開発者がクライアントやビジネスにアクセスできるようにすることは慣例ではありません。開発者をビジネスプロセスから分離、分離しても、開発者はタスクをよりよく理解できません。正当化される場合もあれば、正当化されない場合もあります。



これこそが、アパルトヘイトが開発者と開発者の利益を享受する者を分けるものです。時間の経過とともに、開発者はクリーン開発の国でビジネスの現実から遠ざかっています。それはしばしば単純な人間の喜びと苦痛を無視します。



勧告



開発者がマタンとハスケルの混合を話す自閉症の専門家にならないように、徐々に顧客やビジネスに紹介します。時々、開発者が顧客から直接「ありがとう、これは火事です」と直接聞くことが重要です。そして「まあ、EPT、これは良くない」



ビジネスプロセスに必要かつ重要な参加者であることの喜びは、状況を再現し、内部インストールの重点を「トップコードの記述」から「メリット」にシフトする欲求を生み出します。これにより、コードではなく機能のリリースと品質が向上します。


（私が書きたかったチームには、信頼性、合理的な自律性、健全な微気候に関する他の何かがあります。しかし、今日、感傷性のない別の機会に。）



これを達成する方法



私はこのようにそれを得ました：



  1. 時々、私は開発者をビジネスとのミーティングに駆り立てます。
  2. 時々、お客様に電話することを勧めます。
  3. 毎月末に会議を開催し、どの指標が達成され、どのようなメリットがもたらされたかを説明します。
  4. 私はクライアント/ビジネスにプログラマーと直接仕事に取り組む機会を与えます。


注意することが重要です！私は常に責任者以外の人が開発者に直接タスクを設定できるよう反対していますが、上記の点はこのルールに矛盾しません。



もちろん、これはすべて営業時間中に発生し、タスクトラッカーで開発者の個人的なタスクとして示されます。このような活動は少なくとも週に1〜2回行う必要があり、初心者の場合はより頻繁に行うことができます。



開発者に、コードを支払うまたは書くだけではないことを楽しんでもらいましょう。しかし、それが他の人々にもたらす利益の実現からも。他の有能な人々と一緒にビジネス問題を解決するときに存在する避けられない話題と苦痛から。


はい、誰とも交流したくない自給自足の仲間がいます。彼らは困難な同僚とさえコミュニケーションをとります。そして、これは恥ずかしさの問題ではなく、積極的な孤立、甘い不快感についてです。有用な特性ですが、複雑なプロジェクトでのコラボレーションには適していません。



チームワークは個々の才能よりも重要であるため、私はそのようなプログラマーを採用しないことを好みます。
