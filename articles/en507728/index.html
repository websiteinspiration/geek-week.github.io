Yet Another Production Calendar on MS SQL. PO-> SHA-> GO-> IN

I was prompted to write my decision by a post on Habr√© on a similar topic in which this task was solved ‚Äúhead-on‚Äù - by a simple listing of days and a work / day flag for the entire life span of the system in which this calendar is used. In a similar situation, I decided to do a little trickier, which ultimately turns out to be much easier to support. If you are interested in how this was done - welcome under cat:



In addition to simplicity, the ‚Äúhead-on‚Äù method has two advantages - 1) the logarithmic speed of search / selection, because the date itself is a natural PC, and choosing the weekend attribute by a set of dates is an operation on the field of the (cluster) index, and 2) there is only a slight decrease in performance when checking a random set (in the storage range, of course) with respect to the continuous date sub-range.



As a result, if a) the date check can be in a wide range (even before the Great October Revolution), b) in the check predicate, the scope of the checked dates is not measured for a decade, and c) you do not want to bother with thousands of records in the table, which in 99% of cases are not carry useful information, it can be done differently, namely ...



... do not store all possible dates at all (so that there is no object to which index seek should be applied;), synthesize the axis of dates directly on the fly, over a given range, get the work / day attribute by the day of the week number (Mon ... Fri: work, Sat ... Su: day off), and store exceptions to this rule in a compact table.



Well,



let's get started: Create a date axis generator in a given range (let it be an inline table-valued function, for the possibility of reuse in different applications): 



CREATE FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq
	FROM DR
)

SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')

ALTER FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)

However, there is a small hitch: the day number returned by the built-in DATEPART function depends on the value of the DATEFIRST parameter, and its default value, in turn, depends on the regional settings of the server. Therefore, hoping that we have an American system, and the week starts on Sunday, is generally unpatriotic unreliable.



What can be done to get around this? For example, read the current value of DATEFIRST through the variable @@ DATEFIRST (which returns the number of the first day relative to the default presentation for the United States, in which day number 1 is Sunday (!)) And then derive the day numbers in the current notation based on the current value @@ DATEFIRST corresponding to Saturday and Sunday. Take a moment and try to do it in your mind.



Happened? I don‚Äôt have either. I decided not to take another minute of the club‚Äôs help, and did it differently - I received the days of the week in the current notation from the dates for which I know a priori that it is Saturday and Sunday - for example, January 1 and 2, 2000. And then he simply compared the numbers of issued days with these a priori known in the current context: 



ALTER FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq,
		CONVERT(BIT, CASE WHEN DATEPART(WEEKDAY, DR.DateSeq) IN (Q.DOW_Sat, Q.DOW_Sun) THEN 1 ELSE 0 END) AS IsHoliday
	FROM DR
	CROSS JOIN (
		SELECT TOP (1)
			DATEPART(WEEKDAY, '1/1/2000')	AS DOW_Sat,	--  
			DATEPART(WEEKDAY, '1/2/2000')	AS DOW_Sun	--  
	) Q

)


One may ask, why is all this CROSS JOIN / SELECT TOP (1) ... mixed into the request? But for what: since my TVF is inline, and the function body is essentially one SELECT, you cannot enter any variables - everything that we can rely mainly on SELECT on can either be hardcode (and this is what we just want to avoid ), or output to subquery.



Therefore, we use CROSS JOIN (you can INNER ... ON 1 = 1, as you like).



TOP (1) here we need more SQL engine than we ourselves (after all, we understand that in subquery there is always one line!), But they saythat for similar constructions TOP (1) tells the query compiler that the subquery needs to be executed once before the main one, and not like CROSS APPLY for each row of the main SELECT. I am not 100% sure of this, especially if the body is subquery non-deterministic (I will check it somehow), but let it be so for now.



We check: 



SET DATEFIRST 5	--          
SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')


Bingo! Even in the Sunny City, the weekend is Saturday and Sunday!







Now the triviality remains - to create a table for storing exceptions and correctly merge it. We will do everything under the assumption that we will automate the global business, and a holiday for one country can be quite a gray workday in another: 



CREATE TABLE CountryCalendarOverride (
	CountryID INT NOT NULL,
	CalendarDate DATE NOT NULL,
	IsHoliday BIT NOT NULL,
       CONSTRAINT PK_CountryCalendarOverride PRIMARY KEY CLUSTERED 
       (
	  CountryID,
	  CalendarDate
       )
)


But what if we want to leave the label for the holiday, so as not to forget that we are celebrating? Or even not redefine the sign of work / day off, but just leave a comment on the opposite of some date? Very simple: 



ALTER TABLE CountryCalendarOverride ALTER COLUMN IsHoliday BIT NULL
GO
ALTER TABLE CountryCalendarOverride ADD DateComment NVARCHAR(256) NULL


Why do we need nullable IsHoliday now? Then, that one day we may want to give a comment simply for a certain date, without knowing / not thinking about whether it is a day off or a working day, and not going to redefine this particular flag - but simply say that this date needs attention.



Now everything is ready for the final spurt: 



SET DATEFIRST 5	--           
SELECT 
	DX.DateSeq,
	ISNULL(CCO.IsHoliday, DX.IsHoliday) AS IsHoliday,
	CCO.DateComment
FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01') DX
LEFT JOIN CountryCalendarOverride CCO ON DX.DateSeq = CCO.CalendarDate 
                     AND CCO.CountryID = 1	-- ,     
ORDER BY DX.DateSeq
OPTION (MAXRECURSION 0)
