Kubernetesの外部ストレージとしてのGlusterFS


最適なストレージを見つけることはかなり複雑なプロセスであり、すべてに長所と短所があります。もちろん、このカテゴリのリーダーはCEPHですが、機能は非常に豊富ですが、かなり複雑なシステムです。マスターマスターモードで数テラバイトの単純な複製ストレージが必要であることを考えると、このようなシステムは冗長です。多くの資料を研究した結果、私たちが興味を持っている回路について、市場で最もファッショナブルな製品をテストすることが決定されました。そのような計画の既成の解決策が見つからなかったという事実のため、このトピックに関する私のベストプラクティスを共有し、展開プロセスで発生した問題について説明したいと思います。

ゴール


新しいリポジトリに何を期待したか：

  • 複製のために偶数のノードを操作する機能。
  • 簡単なインストール、セットアップ、サポート
  • システムは、成人向けであり、実績のあるユーザーである必要があります
  • ダウンタイムなしでストレージスペースを拡張する機能
  • ストレージはKubernetesと互換性がある必要があります
  • ノードの1つがクラッシュすると、自動フェイルオーバーが発生します

質問がたくさんあるのは最後のポイントです。

配備


導入のために、2つの仮想マシンがCentO 8に作成されました。それぞれの仮想マシンは、ストレージを備えた追加のディスクを介して接続されています。

GlusterFSの場合、システムに影響を与えないように、別のXFSドライブを割り当てる必要があります。

パーティションを選択します。

XFSでフォーマットしてマウントします。

$ mkfs.xfs /dev/sdb1
$ mkdir /gluster
$ mount /dev/sdb1 /gluster

さらに、エントリを/ etc / fstabにドロップして、システムの起動時にディレクトリを自動的にマウントします。

/dev/sdb1       /gluster        xfs     defaults        0       0

取り付け


インストールに関しては、多くの記事が書かれていますが、これに関連してプロセスを深く掘り下げることはせず、注意する価値があることだけを検討します。

両方のノードで、最新バージョンのglusterfsをインストールして実行します。

$ wget -P /etc/yum.repos.d  https://download.gluster.org/pub/gluster/glusterfs/LATEST/CentOS/glusterfs-rhel8.repo
$ yum -y install yum-utils
$ yum-config-manager --enable PowerTools
$ yum install -y glusterfs-server
$ systemctl start glusterd

次に、隣接ノードがどこにあるかをグラスターに伝える必要があります。ノードは1つだけです。重要なポイント：ドメインネットワークがある場合は、ドメインでサーバー名を指定する必要があります。そうでない場合は、将来的にすべてをやり直す必要があります。

$ gluster peer probe gluster-02.example.com

成功した場合は、両方のサーバーからのコマンドを使用して接続を確認します。

$ gluster peer status
Number of Peers: 1

Hostname: gluster-02.example.com
Uuid: a6de3b23-ee31-4394-8bff-0bd97bd54f46
State: Peer in Cluster (Connected)
Other names:
10.10.6.72

これで、書き込むボリュームを作成できます。

gluster volume create main replica 2 gluster-01.example.com:/gluster/main gluster-02.example.com:/gluster/main force

どこ：

  • メイン-名前ボリューム
  • レプリカタイプのボリューム（詳細は公式ドキュメントに記載されています）
  • 2-レプリカの数

ボリュームを実行し、そのパフォーマンスを確認します。

gluster volume start main
gluster volume status main

複製ボリュームの場合は、次のパラメーターを設定することをお勧めします。

$ gluster volume set main network.ping-timeout 5
$ gluster volume set main cluster.quorum-type fixed
$ gluster volume set main cluster.quorum-count 1
$ gluster volume set main performance.quick-read on

これらの簡単な手順で、GlusterFSクラスターを構築しました。接続してパフォーマンスをチェックするために残っています。Ubuntuはクライアントマシンにインストールされています。マウントするには、クライアントをインストールする必要があります。

$ add-apt-repository ppa:gluster/glusterfs-7
$ apt install glusterfs-client
$ mkdir /gluster
$ mount.glusterfs gluster-01.example.com:/main /gluster

Glusterは、いずれかのノードに接続されると、すべてのノードのアドレスを提供し、自動的にすべてに接続します。クライアントがすでに接続している場合、いずれかのノードで障害が発生しても停止することはありません。ただし、最初のノードが利用できない場合、セッションが中断した場合に接続できなくなります。これを行うには、マウントするときに、2番目のノードを示すbackupvolfileパラメーターを渡します。
mount.glusterfs gluster-01.example.com:/main /gluster -o backupvolfile-server=gluster-02.example.com

重要な点：glusterは、変更がマウントされたボリュームを介して行われた場合にのみ、ノード間でファイルを同期します。ノードに直接変更を加えると、ファイルは同期しなくなります。

Kubernetesに接続する


この段階で、「接続する方法は？」という質問が始まりました。そして、いくつかのオプションがあります。それらを考慮してください。

ヘケティ


最も人気があり、推奨されるのは、外部サービスheketiを使用することです。heketiは、kubernetesとglusterの間のレイヤーであり、httpを介してストレージを管理および操作できます。しかしheketiはその単一障害点になります。サービスはクラスター化されていません。このサービスの2番目のインスタンスは独立して動作できません。変更はすべてローカルデータベースに保存されます。このサービスをkubernetesで実行することも適切ではありません。データベースが保存される静的ディスクが必要です。この点で、このオプションは最も不適切であることが判明しました。

Kubernetesのエンドポイント


パッケージマネージャーを備えたシステムにKubernetesがある場合、これは非常に便利なオプションです。ポイントは、Kubernetes内のすべてのGlusteFSサーバーに対して、共通のエンドポイントが作成されることです。このエンドポイントでサービスがハングしており、すでにこのサービスにマウントされています。このオプションを機能させるには、各Kubernetesノードにglusterfs-clientをインストールし、マウントできることを確認する必要があります。Kubernetesで、次の構成をデプロイします。

apiVersion: v1
kind: Endpoints
metadata: 
  name: glusterfs-cluster
subsets:
  - addresses:
      #  ip  
      - ip: 10.10.6.71
    ports:
      #    1,    
      - port: 1
  - addresses:
      - ip: 10.10.6.72
    ports:
      - port: 1

---
apiVersion: v1
kind: Service
metadata:
  name: glusterfs-cluster
spec:
  ports:
  - port: 1

これで、簡単なテスト展開を作成し、マウントがどのように機能するかを確認できます。以下は、簡単なテスト展開の例です。

apiVersion: apps/v1
kind: Deployment
metadata:
  name: gluster-test
spec:
  replicas: 1
  selector:
    matchLabels:
      app: gluster-test
  template:
    metadata:
      labels:
        app: gluster-test
    spec:
      volumes:
      - name: gluster
        glusterfs:
          endpoints: glusterfs-cluster
          path: main
      containers:
      - name: gluster-test
        image: nginx
        volumeMounts:
        - name: gluster
          mountPath: /gluster

すべてのKubernetesノードにcontainer-linuxがあるため、このオプションは適していませんでした。パッケージマネージャーがないため、マウント用のgluster-clientをインストールできませんでした。この点で、3番目のオプションが見つかり、使用することが決定されました。

GlusterFS + NFS + keepalived


最近まで、GlusterFSは独自のNFSサーバーを提供していましたが、現在はNFSに外部サービスnfs-ganeshaが使用されています。これについてはかなり書かれています。これに関連して、それを構成する方法を理解します。

リポジトリは手動で登録する必要があります。これを行うには、ファイル/etc/yum.repos.d/nfs-ganesha.repoに以下を追加します。

[nfs-ganesha]
name=nfs-ganesha
baseurl=https://download.nfs-ganesha.org/2.8/2.8.0/RHEL/el-8/$basearch/
enabled=1
gpgcheck=1
[nfs-ganesha-noarch]
name=nfs-ganesha-noarch
baseurl=https://download.nfs-ganesha.org/2.8/2.8.0/RHEL/el-8/noarch/
enabled=1
gpgcheck=1

そしてインストール：

yum -y install nfs-ganesha-gluster --nogpgcheck

インストール後、/ etc / ganesha / ganesha.confファイルで基本設定を行います。

# create new
NFS_CORE_PARAM {
    # possible to mount with NFSv3 to NFSv4 Pseudo path
    mount_path_pseudo = true;
    # NFS protocol
    Protocols = 3,4;
}
EXPORT_DEFAULTS {
    # default access mode
    Access_Type = RW;
}
EXPORT {
    # uniq ID
    Export_Id = 101;
    # mount path of Gluster Volume
    Path = "/gluster/main";
    FSAL {
        # any name
        name = GLUSTER;
        # hostname or IP address of this Node
        hostname="gluster-01.example.com";
        # Gluster volume name
        volume="main";
    }
    # config for root Squash
    Squash="No_root_squash";
    # NFSv4 Pseudo path
    Pseudo="/main";
    # allowed security options
    SecType = "sys";
}
LOG {
    # default log level
    Default_Log_Level = WARN;
}

サービスを開始し、ボリュームに対してNFSを有効にし、オンになっていることを確認する必要があります。

$ systemctl start nfs-ganesha
$ systemctl enable nfs-ganesha
$ gluster volume set main nfs.disable off
$ gluster volume status main

その結果、ステータスは、ボリュームに対してnfsサーバーが起動したことを示しているはずです。マウントとチェックを行う必要があります。

mkdir /gluster-nfs
mount.nfs gluster-01.example.com:/main /gluster-nfs

ただし、このオプションはフォールトトレラントではないため、2つのノード間を移動するVIPアドレスを作成し、ノードの1つが落ちた場合にトラフィックを切り替えるのに役立ちます。

CentOへのkeepalivedのインストールは、パッケージマネージャーを介してすぐに実行されます。

$ yum install -y keepalived

/etc/keepalived/keepalived.confファイルでサービスを構成します。

global_defs {
    notification_email {
        admin@example.com
    }
    notification_email_from alarm@example.com
    smtp_server mail.example.com
    smtp_connect_timeout 30

    vrrp_garp_interval 10
    vrrp_garp_master_refresh 30
}

#C   ,   .    , VIP .
vrrp_script chk_gluster {
    script "pgrep glusterd"
    interval 2
}

vrrp_instance gluster {
    interface ens192
    state MASTER #     BACKUP
    priority 200 #      ,  100
    virtual_router_id 1
    virtual_ipaddress {
        10.10.6.70/24
    }

    unicast_peer {
        10.10.6.72 #        
    }

    track_script {
        chk_gluster
    }
}

これでサービスを開始して、ノードのVIPが表示されることを確認できます。

$ systemctl start keepalived
$ systemctl enable keepalived
$ ip addr
1: ens192: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc mq state UP group default qlen 1000
    link/ether 00:50:56:97:55:eb brd ff:ff:ff:ff:ff:ff
    inet 10.10.6.72/24 brd 10.10.6.255 scope global noprefixroute ens192
       valid_lft forever preferred_lft forever
    inet 10.10.6.70/24 scope global secondary ens192
       valid_lft forever preferred_lft forever

すべてがうまくいった場合は、PersistentVolumeをKubernetesに追加し、動作を確認するためのテストサービスを作成する必要があります。

---
apiVersion: v1
kind: PersistentVolume
metadata:
  name: gluster-nfs
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  nfs:
    server: 10.10.6.70
    path: /main

---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
 name: gluster-nfs
spec:
 accessModes:
 - ReadWriteMany
 resources:
   requests:
     storage: 10Gi
 volumeName: "gluster-nfs"

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gluster-test
  labels:
    app: gluster-test
spec:
  replicas: 1
  selector:
    matchLabels:
      app: gluster-test
  template:
    metadata:
      labels:
        app: gluster-test
    spec:
      volumes:
      - name: gluster
        persistentVolumeClaim:
          claimName: gluster-nfs
      containers:
      - name: gluster-test
        image: nginx
        volumeMounts:
        - name: gluster
          mountPath: /gluster

この構成では、メインノードが落ちた場合、タイムアウトでマウントが落ちてスイッチが切れるまで、約1分間アイドル状態になります。このストレージの1分間は単純ですが、これは通常の状況ではないため、それに遭遇することはめったにありませんが、この場合、システムは自動的に切り替わり、動作し続けるため、問題を解決し、単純なことを心配せずにリカバリを実行できます。

概要


この記事では、GlusterFSをKubernetesに接続するための3つの可能なオプションを検討しました。このバージョンでは、Kubernetesにプロビジョナーを追加できますが、まだ必要ありません。同じノード上でのNFSとGluster間のパフォーマンステストの結果を追加するために残っています。

1Mb上のファイル：

sync; dd if=/dev/zero of=tempfile bs=1M count=1024; sync
Gluster: 1073741824 bytes (1.1 GB, 1.0 GiB) copied, 2.63496 s, 407 MB/s
NFS: 1073741824 bytes (1.1 GB, 1.0 GiB) copied, 5.4527 s, 197 MB/s

1Kb上のファイル：

sync; dd if=/dev/zero of=tempfile bs=1K count=1048576; sync
Gluster: 1073741824 bytes (1.1 GB, 1.0 GiB) copied, 70.0508 s, 15.3 MB/s
NFS: 1073741824 bytes (1.1 GB, 1.0 GiB) copied, 6.95208 s, 154 MB/s

NFSはどのファイルサイズでも同じように機能し、速度の違いは、小さなファイルで非常にパフォーマンスが低下するGlusterFSとは異なり、特に顕著ではありません。しかし同時に、ファイルサイズが大きいと、NFSのパフォーマンスはGlusterの2〜3倍低くなります。
