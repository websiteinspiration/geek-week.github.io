JSフレームワークを選択するためのルール

TL、DR

  • この記事では、TOP-3リストのJSフレームワークについては説明しません。
  • TOP-3以外のJSフレームワークで開発する場合、開発の開始時に予想されるよりも桁違いに多くの技術的な問題を解決する必要があります。
  • 物語は実際の出来事に基づいています。

ストーリーは、もともとbackbone.jsおよびmarionette.jsライブラリに基づいて開発された1つのミニプロジェクトから始まりました。もちろん、これらは単一ページのアプリケーション開発の歴史を始めた素晴らしいライブラリです。しかし、その時でさえ、それらは実用的というよりは歴史的価値がありました。単純なテーブルを表示するには、1）モデルの説明を含むモジュール、2）コレクションの説明を含むモジュール、3）ビューモデルの定義を含むモジュール、4）ビューコレクションの定義を含むモジュール、4）テーブル行テンプレート、 5）テーブルテンプレート、6）コントローラモジュール。小さなアプリケーションに約10のエンティティがある-非常に初期の段階では、すでに50以上の小さなモジュールがありました。そして、これはほんの始まりにすぎません。しかし今はそれについてではありません。

アプリケーションの6か月後のある時点で、それはまだ検索結果に表示されませんでした。プロジェクトにprerender.ioを追加する（これはphantom.jsエンジンを使用しました）助けにはなりましたが、期待したほどではありませんでした。クラウドがアプリケーションと私の上に集まり始めた後、私は何かを非常に迅速かつ効率的に、できれば今日行う必要があることに気付きました。私が設定した目標は、サーバーレンダリングに切り替えることです。 backbone.jsとmarionettejsでは、これはほとんど不可能です。いずれにしても、backbone.jsのrendrプロジェクトは、Spike Brehm（同形/ユニバーサルアプリケーションのアイデアの作者）の指導の下で開発され、github.comで58人の寄稿者と4,184件のいいね！を収集していましたが、2015年に停止され、1日の電撃は意図されていませんでした。私は代替案を探し始めました。開発のための時間的余裕がなかったため、TOP-3 JSフレームワークをすぐに検討から除外しました。少し検索したところ、急速に開発されているJSフレームワークriot.jsが見つかりましたgithub.com/riot/riotは、現在github.comに13,704件のいいね！があり、私が望んだように、時間の経過とともに最初の位置に到達できたはずです（ただし、発生しませんでした）。

そして、はい、私はriot.jsを使用してアプリケーションをサーバーレンダリングに転送することで（同じ日ではなく、その日の2日間は）質問を閉じました。これが発生したのと同じ日に、サイトは検索結果の最初の10ページに移動しました。そして、1週間以内に最初のページに移動しました。最初のページは、数年前から出ていません。

これでサクセスストーリーは終わり、敗北の歴史が始まります。次のプロジェクトはもっと複雑でした。確かに、アプリケーション画面の99％がユーザーの個人アカウントにあるため、サーバーレンダリングの必要がないという肯定的な点がありました。 riot.jsを使用した最初の成功体験に触発されて、成功を統合し、フロントエンドにriot.jsを適用するという考えを促進し始めました。そして、riot.jsのドキュメントが約束したように、最終的に、シンプルさと機能性を組み合わせたソリューションが見つかったように思えました。なんて間違えたんだ！

HTMLレイアウトデザイナーに必要なすべての開発ツールを提供する必要があるときに最初に遭遇した問題。特に、コンポーネントのホットオーバーロード（ホットリロード）を含め、コンポーネントを配置してその結果をすぐに観察できるエンジンであるコードエディターのプラグインが必要でした。これらはすべて、近い将来に産業運転に対応できる形で提供する必要がありましたが、そうではありませんでした。その結果、従来のテンプレートエンジンの1つでアプリケーションのレイアウトが開始され、その結果、HTMLドキュメントをriot.jsに翻訳するという恩知らずの段階に至りました。

ただし、この段階で発生した主な問題は、レイアウトをテンプレート形式からriot.jsコンポーネントに変換することにも関係していません。突然、riot.jsがテンプレートのコンパイルエラーや実行時エラーに関する完全な情報メッセージを提供することが判明しました（これは、完全に再設計されたバージョン4.0までのriot.jsのすべてのバージョンに当てはまります）。エラーが発生した行に関する情報だけでなく、エラーが発生したファイルまたはコンポーネントの名前に関する情報もありませんでした。エラーを何時間も続けて検索することは可能でしたが、それでも見つかりませんでした。そして、すべての変更を最後の作業状態にロールバックする必要がありました。

以下はルーティングに関する問題でした。 riot.jsでのルーティングは、箱から出してほとんどありませんgithub.com/riot/route-いずれにしても、同じ開発者からのものです。これにより、彼は問題のない仕事を望みました。しかし、ある時点で、予測できないほど過負荷になっているページがあることに気付きました。つまり、新しいルートへの移行が1ページのアプリケーションモードで発生する場合もあれば、従来のWebアプリケーションで作業する場合と同じように、同じ移行がHTMLドキュメント全体をオーバーロードする場合もあります。もちろん、この場合、まだサーバーに保存されていないと、内部状態は失われます。 （現在、このライブラリの開発は中止されており、riot.js 4.0では使用されません。）

期待どおりに機能した唯一のシステムコンポーネントは、最小限のフラックスのような状態マネージャーgithub.com/jimsparkman/RiotControlでした。確かに、このコンポーネントを使用するには、状態の変化のリスナーを指定してキャンセルする必要がありました。

この記事の最初のアイデアはこれでした：riot.jsフレームワークでの私たち自身の経験の例を示すことで、開発者が解決しなければならないタスクを示し、TOP-3リスト以外のJSフレームワークでアプリケーションを開発することを決定（決定）しました。ただし、準備の過程で、riot.jsのドキュメントの一部のページを更新することにしました。その結果、riot.js 4.0の新しいバージョンがリリースされたことを知りました。これは、完全に（ゼロから）再設計され、riot開発者の記事にあります。 medium.comのjs：medium.com/@gianluca.guarini/every-revolution-begins-with-a-riot-js-first-6c6a4b090ee。この記事から、私が心配していた、この記事で説明しようとしていたすべての主要な問題が解消されたことがわかりました。特に、riot.jsバージョン4.0では：

  • コンパイラーは完全に書き直されました（または、エンジンが以前に正規表現を処理していたため、最初に作成されました）-これは、特にエラーメッセージの情報内容に影響しました
  • サーバーレンダリングに加えて、ハイドレートがクライアントに追加されました-これにより、ダブルレンダリングなしでユニバーサルアプリケーションの作成を開始できるようになりました（古いバージョンではハイドレート（）関数がないため、サーバー上で初めてクライアント上で）
  • ホットオーバーロードコンポーネントのプラグインを追加しましたgithub.com/riot/hot-reload
  • その他の多くの便利な変更

これは広告ではなく、背景です。実際、私の結論は、使用に関する推奨事項に関して非常にあいまいであり、記事は一般的に特定のフレームワークやライブラリではなく、開発プロセス中に解決する必要があるタスクに当てられています。

残念ながら、riot.js開発者が行った作業は、コミュニティによってまだ適切に評価されていません。たとえば、サーバーレンダリングライブラリgithub.com/riot/ssrは、開発の開始から6か月が経過し、github.comで3人の共同寄稿者と3つのいいね！を収集しました（1つは寄稿者と同じですが、すべてのいいね！が寄稿者によって作られたわけではありません）。

したがって、彼は演劇の過程で記事の方向を変更し、回想録ではなく、少しだけ知識と経験、ライブラリのより高度なバージョン、および無制限の自由時間を確保して、最後までやり直そうとしました。

さあ、行きます。たとえば、github.com / gothinkster / realworldアプリケーションが実装されました。このプロジェクトは、Habréで何度も議論されています。彼に詳しくない人のために、彼の考えを簡単に説明します。このプロジェクトの開発者は、さまざまなプログラミング言語とフレームワークの助けを借りて（またはそれらを使わずに）同じ問題を解決します。ブログエンジンの開発は、機能的にはmedium.comの簡易バージョンに似ています。これは、日常的に開発しなければならない実際のアプリケーションの複雑さとtodo.appの間の妥協であり、必ずしもライブラリやフレームワークでの作業に感謝することはできません。このプロジェクトは開発者から尊敬されています。上記の確認から、Rich Harris（sveltejsの主要開発者）github.com/sveltejs/realworldからの実装も1つあると言えます。

開発環境


もちろん、あなたは戦いに突入する準備ができていますが、あなたの周りの開発者について考えてください。この場合、同僚がどの開発環境で作業しているかという問題に焦点を当てます。主な開発環境のプラグインや、使用するフレームワークのプログラムコードエディターがない場合は、サポートされない可能性があります。たとえば、開発にはAtomエディターを使用します。彼のために、過去3年間更新されていないriot-tagプラグインgithub.com/riot/syntax-highlight/tree/legacyがあります。そして同じリポジトリには、崇高なgithub.com/riot/syntax-highlightのプラグインがあります-これは最新であり、riot.js 4.0の現在のバージョンをサポートしています。

プロジェクトアセンブリ


プロジェクトの構築に必要なほとんどの機能に既に慣れているし、何が違うのか考えさえしなくなっている。しかし、それ以外の場合もあります。また、新しいJSフレームワークを選択した場合は、最初にすべてが期待どおりに機能することを確認することをお勧めします。たとえば、すでに述べたように、古いバージョンのriot.jsで開発する際の最大の問題は、コンパイルエラーメッセージの欠如と、このエラーが発生したコンポーネントに関するランタイム情報に関連しています。コンパイル速度も重要です。原則として、コンパイル速度を上げるために、正しく構築されたフレームワークでは、変更された部分のみが再コンパイルされるため、コンポーネントのテキストの変更に対する反応時間が最小限になります。まあ、本当に良いコンポーネントのホットリロードがサポートされており、Webブラウザでページを完全にリロードしない場合。

ルーティング


Leo Tolstoyを少し言い換えると、JSフレームワークのすべてのエンジンは互いに類似していますが、それらに接続されているすべてのルーティングは独自の方法で機能します。ルーターを条件付きと宣言型の2つのタイプに分類することがよくあります。今、私はそのような分類がどのように正当化されるかを理解しようとします。

歴史をちょっと見てみましょう。インターネットの黎明期、URL / URIはサーバーでホストされているファイルの名前と一致しました。一度に数ページの歴史をめくり、Model 2（MVC）アーキテクチャの登場について学びます。このアーキテクチャでは、ルーティングの機能を実行するフロントコントローラが表示されます。最初にフロントコントローラーからルーティング機能を別のブロックで選択することを決定したのは誰なのかと思いました。ルーティング機能は、多数のコントローラーの1つにリクエストを送信しますが、まだ回答が見つかりません。彼らは一斉にそれを始めたようです。

つまり、ルーティングによって、サーバーで実行する必要のあるアクションと、コントローラーによって生成されるビュー（コントローラーを通じて推移的に）が決定されます。ルーティングをクライアント側（Webブラウザー）に転送するとき、アプリケーションのルーティング機能に関するアイデアはすでに形成されていました。そして、ルーティングがアクションを決定するという事実に主に焦点を合わせた人々は命令型ルーティングを開発し、最終的にルーティングがユーザーに表示されるべきビューを決定することに注意を払った人々は宣言型ルーティングを開発しました。

つまり、ルーティングのためにサーバーからクライアントに転送するときに、サーバーのルーティングに特徴的な2つの機能（アクションの選択とビューの選択）を「ハング」させました。さらに、新しいタスクが発生しました。これは、HTMLドキュメントを完全に再読み込みせずに1ページのアプリケーションをナビゲートする、訪問の履歴を操作するなどです。説明のために、メガポピュラーフレームワークのルーターのドキュメントからの抜粋を引用します

このオプションでは、ルーティングが機能で明らかに過負荷になり、クライアント側でのタスクを再考する必要があります。私は自分の仕事に適したソリューションを探し始めました。主な基準として、私はそのルーティングを考慮に入れました：

  1. ユニバーサル/同形Webアプリケーションの場合、Webクライアント側とWebサーバー側の両方で同じように動作するはずです。
  2. 任意の（私の選択したものを含む）フレームワークで、またはそれなしで動作するはずです。

そして、私はそのようなライブラリを見つけました、これはgithub.com/kriasoft/universal-routerです。このライブラリの考え方を簡単に説明すると、URLの文字列を入力で受け取り、解析されたURLを実際のパラメーターとして渡す非同期関数を出力で呼び出すルートを構成します。正直なところ、私は聞きたかったのですが、それですべてですか？そして、誰もがこれでどのように取り組むべきでしょうか？そして、medium.com medium.com/@ippei.tanaka/universal-router-history-react-97ec79464573で記事を見つけました。この記事では、おそらくpush（）履歴メソッドを書き換えたが、私は必要とし、それをコードから削除しました。その結果、クライアント側のルーターの動作は、おおよそ次のように定義されます。

const render = async (location) => {

アプリケーション状態管理


最初は、mobxライブラリーをプロジェクトに含めました。すべてが期待どおりに機能しました。それがタスクに完全に対応していないことを除いて-私が記事の冒頭で設定した研究。そこでgithub.com/zerobias/effectorに切り替えました。これは非常に強力なプロジェクトです。これは、100％のredux機能（大きなオーバーヘッドがない場合のみ）と100％のmobx機能を提供します（ただし、この場合はもう少しコーディングする必要がありますが、デコレーターなしのmobxと比較するとさらに少なくなります）ストア

の説明は次のようになります。

import { createStore, createEvent } from 'effector'
import { request } from '../agent'
import { parseError } from '../utils'

export default class ProfileStore {
  get store () {
    return this.profileStore.getState()
  }

  constructor () {
    this.success = createEvent()
    this.error = createEvent()
    this.updateError = createEvent()
    this.init = createEvent()
    this.profileStore = createStore(null)
      .on(this.init, (state, store) => ({ ...store }))
      .on(this.success, (state, data) => ({ data }))
      .on(this.error, (state, error) => ({ error }))
      .on(this.updateError, (state, error) => ({ ...state, error }))
  }

  getProfile ({ req, author }) {
    return request(req, {
      method: 'get',
      url: `/profiles/${decodeURIComponent(author)}`
    }).then(
      response => this.success(response.data.profile),
      error => this.error(parseError(error))
    )
  }

  follow ({ author, method }) {
    return request(undefined, {
      method,
      url: `/profiles/${author}/follow`
    }).then(
      response => this.success(response.data.profile),
      error => this.error(parseError(error))
    )
  }
}

このライブラリは本格的なレデューサーを使用します（それらはeffector.jsのドキュメントで呼ばれています）。多くの人がmobxに欠けていますが、reduxよりもコーディングの労力ははるかに少ないです。しかし、主なことはそれでもありません。reduxとmobxの機能の100％を受け取ったので、私はeffector.jsに固有の機能の10分の1のみを使用しました。そこから、複雑なプロジェクトでの使用が開発者の資金を大幅に増やすことができると結論付けることができます。
