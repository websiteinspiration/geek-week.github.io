Javaプログラマーにインタビューします。あらゆるレベルの1セットの質問

私のキャリアでは、開発者であり、マネージャーである必要があり、小規模なスタートアップではかつて「CTO」でした。そしてもちろん、私は多数の候補者とのインタビューを行わなければなりませんでした。したがって、私自身、技術面接を実施するためのルールを作成しました。

  • すべての候補者に同じ質問をする
  • 質問は、ジュニアとシニアの両方がそれらに答えるようなものでなければなりませんが、異なる方法で、
  • 3つの質問の後、候補者のレベルを理解する必要があります。

そして、私はJava開発者にこれら3つの質問をしました。

HashMapについて教えてください


はい、はい。インタビューの50％が始まる非常に標準的な質問です。なぜ私は彼に尋ねるのですか？非常に異なって答えられるからです。

  • 理想的な答えは、問題の説明から始まります。なぜそのような構造が必要なのですか？それはどのような特定のタスクを解決しますか？この問題の簡単な解決策は「真正面」のように見え、なぜそれが悪いのでしょうか。
  • 歴史的なエクスカーションの「ボーナスポイント」、ソリューションの作成者の知識、少なくともその出現のおおよその年（もちろん、Javaではありません）
  • 少なくともジュニアの答えは、構造を使用する方法を含める必要があります。彼女はどのような方法をサポートしていますか。
  • 中間開発者は、内部デバイス、およびhashCode（）関数とequals（）関数のコントラクトの説明、それらの機能要件を理解する必要があります
  • hashCode（）およびequals（）の非機能要件を説明するためのボーナスポイント
  • マルチスレッド環境でのHashMapの使用を検討した場合のボーナスポイント。
  • HashMapの代替案を検討するための「ボーナスポイント」-それらをリストし、実装の複雑さ、使用、一般的な速度、理想的なケースと最悪のケースを比較します。
  • 別個のシステムコンポーネント（たとえば、データベースと「外部」キャッシュ）の形式のHashMap代替の「ボーナスポイント」。
  • データベースでのHashMapの使用方法、およびHASHインデックスではなくバイナリツリーベースのインデックスが通常使用される理由を説明する「ボーナスポイント」。

ご覧のとおり、senior'omと話すことができる非常に多くの「ボーナスオプション」があります。そして、あなたは少なくとも10分間、いくつのポイントについて話すことができますか？

飽きたら、HashMapをTreeMapに置き換えてください。

SQLの記述...


ええ、そして再び「標準」の質問です。10年前はすべての面接にありました。現在、それはあまり一般的ではありませんが、明らかにnosqlの方法が原因です。しかし、データを扱う能力は常に必要であり、デカルト積を理解しているプログラマーは、教科書の一般的な読み直しに制限されているプログラマーよりも常に有用（そして高価）です。

そして問題は：
データベースには、学生とグループの2つのテーブルがあるとします。各学生は、彼が所属するgroup_idグループの識別子を持っています。入れ子を使用せずにSQLクエリを記述し、1人の生徒なしでグループ（またはグループの数）を一覧表示する

もちろん、正解にはJOINだけでなく、値をNULLと比較するLEFT JOINも含まれます。最も興味深いのは、リクエストを作成するときにどんなミスが起こり得るか、そしてリクエストが行われた場合にリクエストが機能しない理由について候補者と話すことです。なぜ等号を介してNULLと比較できないのですか？単純なJOINが機能しないのはなぜですか？

このクエリの操作を高速化するインデックスと、さまざまな方言（異なるDBMS）のSQLオプションを検討する場合のボーナス。

コードのエラーを見つける


候補者には次のようなタスクが発声されます。
コードレビューのために、別の開発者からプルリクエスト（マージリクエスト）を受け取ったとします。このリクエストには多数の異なるファイルがありますが、そのうちの1つは次のようになります（クラスコード全体が示されています）。

package ru.mycompany.utils;

/** Wrap around Lock to simplify interface */
class LockWrapper {
    private Lock lock;
    public wait() {
        this.lock.wait()
    }
}

このコードについてのコメントはありますか？そのようなファイルを使用して（masterブランチに）リクエストをスキップしますか、それともやり直しが必要ですか？その理由は？

もちろん、ここには少なくとも3つのレベルがあります。

  • 後輩はこのコードにある構文エラーを指摘する必要があります。開発者は、コードがコンパイルされない理由を示す必要があります。もちろん、問題は構文だけではありません。
  • 開発者は、コードがコンパイルされても機能しない理由を知っている必要があります。そして、それが機能するようにこのコードを修正する方法。開発者は、同期コンストラクトとjava.util.concurrentパッケージのクラスの両方を使用でき、1秒が3秒にならない理由を知っている必要があります。
  • 理想的な開発者は、このコードが必要な理由、彼が解決する問題、このコードの既存の問題をすべて修正しても解決できるかどうかを尋ねることから始める必要があります。

回答の過程で、候補者とともに、コードを「理想的な」形式にします。もちろん、最終的な形式は候補者の資格によって異なります。私の最高の候補者は、ここで13個のエラーを見つけました。「最も重要な間違い」は別として。

おまけとして、存在するエラーの少なくとも一部を見つけるために使用する必要がある自動コード制御の手段について説明できます。

結論の代わりに
