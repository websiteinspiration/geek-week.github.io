日常生活で溺れない方法、またはストレステスト中にAWRダンプを比較した経験

みなさん、こんにちは！私の名前はLyudmilaです。私は負荷テストに従事しています。OracleDBMSのデータベースから、顧客の1人と、負荷テストシステムの回帰プロファイルの比較分析を自動化した方法を共有したいと思います。

この記事の目的は、データベースのパフォーマンスを比較するための「新しい」アプローチを見つけることではなく、私たちの経験を説明し、得られた結果の比較を自動化し、
Oracle DBAへの呼び出し回数を減らすことです。



任意のデータベースの負荷テストを実行する場合、主に次のことに関心があります。

  • 新しいアセンブリを取り付けた後に何かが壊れましたか？
  • テスト中のデータベースのダイナミクス。

AWRレポートの比較だけでは、目標を達成するには不十分です。
AWRダンプの集中ストレージも良い方法です。AWRダンプは、すべての履歴ビュー（dba_hist）を保持します。

この慣行はすでにクライアントによって適用されています。

負荷テストの次のセッションの後で、結果を比較します。

  • 工業用ダンプ付きの現在のテストダンプ。
  • 現在のテストダンプと以前のテストダンプ。

まず、いくつかのクエリの実行速度の比較から、イベントの比較に興味がありました。フォームはすでに存在しています。次のステップでは、各イベントに関する詳細情報が必要でした。たとえば、イベントがインデックスの競合である場合、実際にどのインデックスがハングしているのかを理解する必要があります。

さらに、実装ではスケジュールに従って実行される多くのタスク（ジョブ）があり、すべての時点で継ぎ目で亀裂が発生していることを理解する必要があるため、これらのイベントのどの時点が最も多いかに興味がありました。

一般的に、ここに私が得たかったものがあります：

  1. 異なるテスト間のイベントの定量的な比較（追加のスクワットなし）。
  2. 分析に必要なすべての関連情報：sql_id、クエリテキスト、テスト中の分布、セッションが参照したオブジェクト、モジュール。
  3. 自分がそれらの変更点を確認するための便利なフィルター。
  4. GUI GUI、すべてがカラフルなのですぐに表示されます（開発側から関係者をスクリーニングできます）
  5. モジュールのグループ化：前に説明したように、16237モジュールですが、実行される機能の観点からは何倍も少ないです。

ポータルについて


そのため、AWRダンプがシステムで作成され、データベースに注がれ、ポータルで比較されます。

次のスタックを使用しました。

  1. Oracle DB-AWRダンプの保存用
  2. Python 2+



ポータルインターフェイスは次のようになります。ポータルで



は、比較するダンプのタイプ、テストテストまたはテストプロムを選択できます。

各ダンプには独自の一意の識別子（DBID）があります。

次のパラメータでフィルタリングすることもできます。

  1. インスタンス（インスタンス）-クラスタデータベースがありました。
  2. リクエスト（Sql_id）;
  3. 待機のタイプ（Wait_Class）;
  4. イベント

左上でダンプを選択し、右側で必要なフィルターを設定して目的のモジュールをすぐに選択できます。これにより、変更/改善された機能の問題を特定できるため、以前のバージョンでの劣化の問題がありません。

中央の表は、ダンプを比較した結果です。列見出しは、出力されているデータをすぐに示します。右の2つの列は、2つのダンプの違いを示しています。

  • 赤で強調表示されたイベントは、スナップショットの比較ダンプと比較した場合よりも多くなっています。
  • 黄色-新しいイベント。
  • 緑-元のダンプにすでにあったイベント。

私たちがどれだけうまくテストしたかはすぐにわかります。イベントが非常に頻繁に発生した場合、最も可能性が高い：

  1. システムの過負荷。
  2. . : , .

新しいイベント-黄色-がある場合、これはシステムになんらかの変更があったことを示しており、その結果を分析する必要があります。ここでは、スナップショットごとのイベントの分布を確認し、待機に関する詳細情報を表示できます。

かつてのケースがありました。非常にまれで、上位のイベントに含まれていなかった新しいイベントが発見されましたが、そのため、機能にスローダウンがあり、重要なSLAがありました。AWRレポートの上位クエリのみの分析では、これを明らかにできませんでした。

リクエストごとに、より詳細な情報を取得できます。



レコードごとに、次の情報も表示できます。

  1. SQLテキストのクエリ。
  2. スナップショット上のイベントの定量的な比率での分布、つまり どの時点でイベントが多かったり少なかったりしたか。
  3. 待機しているモジュールとオブジェクトがハングしています。

複数のクエリ：

イベントをスナップショットに分散：

SELECT S.SNAP_ID, COUNT(*) RCOUNT 
FROM DBA_HIST_ACTIVE_SESS_HISTORY S, V_DUMPS_LOADED  V. 
WHERE V.ID = :1 
AND S.DBID = V.DBID 
AND S.INSTANCE_NUMBER = :2 
AND S.SQL_ID = :3 
AND S.EVENT_ID = :4 
GROUP BY S.SNAP_ID 
ORDER BY S.SNAP_ID ASC

モジュールごとにグループ化（単一の論理グループであるモジュールはそれに結合されます）、ブロックされているオブジェクト：

SELECT MODULE, OBJECT_NAME, COUNT(*) RCOUNT 
(SELECT CASE (WHEN INSTR(S.MODULE, ‘   1’)>0 THEN ‘  1’
WHEN INSTR(S.MODULE, ‘   2’)>0 THEN ‘  2’
…
ELSE S.MODULE END) MODULE, O.OBJECT_NAME
FROM DBA_HIST_ACTIVE_SESS_HISTORY S, V_DUMPS_LOADED V,  DBA_HIST_SEG_STAT O
WHERE V.ID = :1 
AND S.DBID = V.DBID 
AND S.INSTANCE_NUMBER = :2 
AND S.SQL_ID = :3 
AND S.EVENT_ID = :4 
AND S.CURRENT_OBJ# = O. OBJ# (+) AND V. DBID =  O.DBID  
)
GROUP BY MODULE, OBJECT_NAME 
ORDER BY RCOUNT DESC

最終的に何を手に入れましたか？


ポータルにより、AWRダンプを比較する時間を節約できました。手動での比較には4〜6時間かかりましたが、現在は2〜3時間かかります。私たちは常に、さまざまなテストの結果を迅速に比較し、現在必要なフィルターを設定するだけでなく、産業用ダンプと比較することができます。つまり、現在の結果をオンラインで見るだけでなく、過去のデータを自分で簡単に比較できます。

以前は、各回帰の後、履歴テーブルの結果をデータベースへのクエリと比較し、AWRレポートを表示し、問題のある期待値（発生したモジュール、発生した時間、発生したオブジェクト）をローカライズして、最終的には適切な開発チームの欠陥につながる可能性がありました。そして、比較するダンプを選択し、フィルターを設定するだけで、比較の結果がすぐに準備できます。また、テストダンプのDBIDを示すポータルへのリンクを開発者に送信することもできます。開発者自身は、モジュールによってフィルタリングされます。

ポータルの一部がすでに準備されているため、ポータルの作成に2週間しかかかりませんでした。ダンプをデータベースにロードしています。もちろん、そのようなポータルソリューションは、Oracleベースのプロジェクトには必要ありません。異なる名前の多数のモジュールに分割されている製品に役立ちます。単純なシステム、またはモジュールへの入力を重要視しないシステムの場合、ポータルは冗長になります。

ポータルは特定の期間に1度撮影された画像を分析するため、一部のイベントが画像に入らない可能性があるため、ポータルはデータベースのオンライン監視から完全に除外されません。

これは、テスト結果に基づいて履歴データを分析するのに便利なツールですが、大量の画像が作成され、大量のデータを確認する必要がある他の状況でも役立ちます。フィルターとグラフの組み合わせのおかげで、通常のAWRレポート（ダンプと混同しないでください）ではグループ化された情報に非表示になるイベントのバーストをすぐに確認できます。比較するダンプを選択し、フィルターを設定するだけで十分です。比較結果はすぐに準備できます。または、テストダンプのDBIDを使用してポータルの開発者にリンクを送信できます。それらは、モジュールによってフィルターされます。

プロジェクトに同様のポータルを開発する場合は、適切なフィルターのセットを選択します。毎回異なる条件に従ってフィルタリングする場合は、これに適切なフィルターを作成する方がはるかに簡単です。

結果のソリューションは、たとえば次のように最終化することができます。

  1. リクエストの期間を比較する。
  2. クエリプランの比較;
  3. 同じプランで異なるテキストでリクエストを比較する;
  4. テストレポートへのアンロード（Word / Exelドキュメントとして実行）。

または、一般に、テスト済みのデータベースに接続して、履歴データだけでなく、メモリ内のビューを使用して同様の画像をオンラインで構築するようにポータルに指示します。そして、それらをデータベースに保存します。
