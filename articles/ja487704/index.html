SSRS 2014で動的レポートを作成する方法


我々はすでにについて話しました、我々は1つの製造会社は、企業研修や人材開発のプロセスを変革どのように役立ちましたか。紙のドキュメントやExcelのスプレッドシートに溺れていた顧客の従業員は、便利なiPadアプリケーションとWebポータルを受け取りました。この製品の最も重要な機能の1つは、マネージャーが「現場で」従業員の作業を判断するための動的レポートの作成です。これらは、数十のフィールドと3000 * 1600ピクセルの平均サイズを持つ巨大なドキュメントです。

この記事では、Microsoft SQL Server Reporting Servicesに基づいてこの美しさを展開する方法、そのようなバックエンドがWebポータルの悪役になる理由、およびそれらの関係を確立するのに役立つトリックについて説明します。ソリューションのビジネス部分全体は、以前の記事ですでに説明されているため、ここでは技術的な問題に焦点を当てます。始めましょう！


問題の定式化


数百人のユーザーが利用できるポータルがあります。それらは段階的な階層に配置されており、各ユーザーには上位の監督者がいます。ユーザーが下位の従業員とイベントを作成できるようにするには、この権限の区別が必要です。ステップを飛び越えることができます。ユーザーは、自分よりも低いランクの従業員から活動を開始できます。

ここではどのようなイベントを意味しますか？これは、スーパーバイザーが販売時に実施する商社の従業員のトレーニング、サポート、または認定です。このようなイベントの結果、iPadにアンケートが記入され、従業員の専門的な資質とスキルの評価が表示されます。

アンケートデータによると、たとえば次のような統計を作成できます。

  • Vasya Ivanovがこの種の部下と一緒に1か月に何回イベントを作成しましたか？それらの何が完了しましたか？
  • 満足のいく評価のパーセンテージは何ですか？マーチャンダイザーが最悪に答える質問は何ですか？どのマネージャーがテストを受けるのが悪いですか？

このような統計は、Webインターフェースを介してXLS、PDF、DOCX形式で作成および印刷できるレポートに含まれています。これらの機能はすべて、さまざまなレベルのマネージャー向けに設計されています。

レポートの内容とデザインはテンプレートで定義され、必要なパラメーターを設定できます。将来ユーザーが新しいタイプのレポートを必要とする場合、システムはテンプレートを作成し、変更可能なパラメーターを指定して、テンプレートをポータルに追加することができます。これらすべて-製品のソースコードと作業プロセスに干渉することなく。

仕様と制限


ポータルはマイクロサービスアーキテクチャで実行され、フロントはAngular 5で記述されています。リソースはJWT認証を使用し、Google Chrome、Firefox、Microsoft Edge、IE 11ブラウザーをサポートしています。

すべてのデータはMS SQL Server 2014に保存されます。SQLServer Reporting Services（SSRS）がサーバーにインストールされ、お客様はそれを使用して拒否しません。したがって、最も重要な制限は、SSRSへのアクセスが外部から閉じられるため、ローカルネットワークからのみNTLM認証を介してWebインターフェイスとSOAPにアクセスできることです。

注意、質問：リソースを最小限に抑え、顧客に最大のメリットをもたらす、手動の方法なしでタスクを完了する方法？

顧客は専用サーバーにSSRSを持っているので、SSRSにレポートの生成とエクスポートのすべての汚い仕事をさせてください。次に、独自のレポートサービスを作成して、モジュールをXLS、PDF、DOCX、HTMLおよび対応するAPIにエクスポートする必要はありません。

したがって、タスクはSSRSをポータルと友達にして、タスクで指定された機能の操作を確実にすることでした。それでは、これらのシナリオのリストを見ていきましょう。興味深い微妙さがほとんどすべての点で見つかりました。

ソリューションの構造


SSRSが既にあるので、レポートテンプレートを管理するためのすべてのツールがあります。

  • レポートサーバー-レポートの操作ロジック全体、それらの保存、生成、管理などを担当します。
  • レポートマネージャー-レポートを管理するためのWebインターフェイスを備えたサービス。ここでは、SQL Serverデータツールで作成されたテンプレートをサーバーにアップロードし、アクセス権、データソース、およびパラメーター（要求のレポート時に変更できるものを含む）を構成できます。ダウンロードしたテンプレートに関するレポートを生成し、XLS、PDF、DOCX、HTMLなどのさまざまな形式にアップロードできます。

合計： SQL Serverデータツールでテンプレートを作成し、レポートマネージャーを使用してレポートサーバーにテンプレートを入力し、構成します。これで準備が整いました。レポートを生成し、パラメータを変更できます。

SSRSからポータルへのレポート


上で述べたように、SSRSにはレポートにアクセスするための独自のAPIがあります。ただし、セキュリティとデジタル衛生上の理由からその機能を提供する必要はありません。適切な形式でSSRSにデータを要求し、結果をユーザーに送信するだけです。レポート管理は、特別に訓練された顧客スタッフによって処理されます。

SSRSへのアクセスはローカルネットワークからのみであるため、サーバーとポータル間のデータ交換はプロキシサービスを介して行われます。


ポータルとサーバー間のデータ交換

それがどのように機能し、なぜReportProxyがここにあるのかを見てみましょう。

したがって、ポータル側には、ポータルがレポートのためにアクセスするReportServiceがあります。サービスは、ユーザーの承認、ユーザーの権限のレベルをチェックし、SSRSからのデータを契約に基づいて目的の形式に変換します。

ReportService APIには2つのメソッドしか含まれておらず、これで十分です。

  1. GetReports-現在のユーザーが受け取ることができるすべてのテンプレートの識別子と名前を提供します。
  2. GetReportData（format、params） -指定されたパラメーターのセットを使用して、指定されたフォーマットで既製のエクスポートされたレポートデータを提供します。

SSRSと通信し、必要なデータを適切な形式でSSRSから取得するには、これら2つの方法が必要です。ドキュメントから、SOAP APIを使用してHTTP経由でレポートサーバーにアクセスできることがわかっています。パズルが発展しているようです...しかし、実際には、驚きがここに私たちを待っています。

SSRSは外部に閉じているため、NTLM認証を介してのみアクセスできるため、SOAPポータルから直接使用することはできません。私たち自身の願いもあります：

  • 必要な機能セットにのみアクセスを許可し、変更を禁止します。
  • 別のレポートシステムに切り替える必要がある場合は、ReportServiceでの編集を最小限に抑え、まったく必要ないほうがよいでしょう。

これは、ReportProxyが役立ちます。SSRSと同じマシン上にあり、ReportServiceからSSRSへのリクエストのプロキシを担当します。リクエスト処理は次のとおりです。

  1. サービスはReportServiceからリクエストを受け取り、JWT承認をチェックします。
  2. APIメソッドに従って、プロキシはSSRSのSOAPプロトコルを経由して必要なデータを取得し、途中でNTLMを介してログインします。
  3. SSRSから受信したデータは、要求への応答としてReportServiceに送り返されます。

実際、ReportProxyはSSRSとReportServiceの間のアダプターです。

public class BasicAuthenticationAttribute : AuthorizationFilterAttribute
{
    public override void OnAuthorization(HttpActionContext actionContext)
    {
        var authHeader = actionContext.Request.Headers.Authorization;

        if (authHeader != null)
        {
            var authenticationToken = actionContext.Request.Headers.Authorization.Parameter;
            var tokenFromBase64 = Convert.FromBase64String(authenticationToken);
            var decodedAuthenticationToken = Encoding.UTF8.GetString(tokenFromBase64);
            var usernamePasswordArray = decodedAuthenticationToken.Split(':');
            var userName = usernamePasswordArray[0];
            var password = usernamePasswordArray[1];

            var isValid = userName == BasiAuthConf.Login && password == BasiAuthConf.Password;

            if (isValid)
            {
                var principal = new GenericPrincipal(new GenericIdentity(userName), null);
                Thread.CurrentPrincipal = principal;

                return;
            }
        }

        HandleUnathorized(actionContext);
    }

    private static void HandleUnathorized(HttpActionContext actionContext)
    {
        actionContext.Response = actionContext.Request.CreateResponse(
            HttpStatusCode.Unauthorized
        );

        actionContext.Response.Headers.Add(
            "WWW-Authenticate", "Basic Scheme='Data' location = 'http://localhost:"
        );
    }
}

  1. フロントはReportServiceにhttpリクエストを送信します。
  2. ReportServiceは、httpリクエストをReportProxyに送信します。
  3. SOAPインターフェイスを介したReportProxyは、SSRSからデータを受信し、その結果をReportServiceに送信します。
  4. ReportServiceは、契約に従って結果をもたらし、クライアントに提供します。

問題番号1。巨大なテーブル


設計コンセプトに従って、このページには、ユーザーがエクスポートする前にレポートを表示できるように表示領域が必要です。レポートがウィンドウに収まらない場合は、水平および垂直にスクロールできます。同時に、一般的なレポートは複数の画面のサイズに達する可能性があります。つまり、行と列の名前を付けたブロックを貼り付ける必要があります。これがなければ、ユーザーは常に特定のセルの意味を思い出すために表の先頭に戻る必要があります。または、一般的には、レポートを印刷して必要な葉を常に目の前に置いておく方が簡単ですが、画面上のテーブルは単にその意味を失います。

一般的に、ブロックの固着は避けられません。また、SSRS 2014はMHTMLドキュメントの行と列を修正する方法を認識していません-独自のWebインターフェイスでのみ。

ここでは、最新のブラウザーがCSSのstickyプロパティをサポートしていることを思い出してください。位置を付けます：マークされたブロックにスティッキー、左側または上部にインデントを指定します（左、上部のプロパティ）。ブロックは、水平および垂直スクロール中も所定の位置に残ります。

CSSがキャッチできるパラメーターを見つける必要があります。 SSRS 2014がWebインターフェースでそれらをキャプチャできるカスタムセル値は、HTMLにエクスポートするときに失われます。 OK、私たちはそれらを自分でマークします-私たちは方法だけを理解します。

ドキュメンテーションを読んだり、同僚と話し合ったりして数時間経った後、選択肢がないように見えました。そしてここでは、プロットのすべての法則により、ToolTipフィールドが表示され、セルのツールチップを指定できます。これは、tooltip属性のエクスポートされたHTMLコードにスローされることがわかりました-SQL Serverデータツールのカスタムセルに属するタグに正確に。選択の余地はありませんでした-細胞に固定のマークを付ける別の方法が見つかりませんでした。

したがって、ToolTipを使用してHTMLでマーキングルールと転送マーカーを作成する必要があります。次に、JSを使用して、tooltip属性を指定されたマーカーのCSSクラスに変更します。

セルを固定するには、垂直方向（固定列）と水平方向（固定行）の2つの方法しかありません。コーナーセルに別のマーカーを配置することは理にかなっています。これは、両方向にスクロールしても固定されたままです。

次のステップは、UIを実行することです。HTMLドキュメントを受け取ったら、マーカーが含まれているすべてのHTML要素を見つけ、値を認識し、適切なCSSクラスを設定し、tooltip属性を削除して、マウスをポイントしたときに表示されないようにする必要があります。結果のマークアップはネストされたテーブル（テーブルタグ）で構成されることに注意してください。

calculateCellCascadeParams(cell: HTMLElement, fixationType: FixationType) {
    const currentTD: HTMLTableCellElement = cell.parentElement;
    const currentCellIndex = currentTD.cellIndex;

    //   
    currentTD.style.left = '';
    currentTD.style.top = '';

    const currentTDStyles = getComputedStyle(currentTD);

    //  
    if (fixationType === 'row' || fixationType === 'both') {
        const parentRow: HTMLTableRowElement = currentTD.parentElement;

        //        
        //    .
        //   ,    .
        let previousRow: HTMLTableRowElement = parentRow;
        let topOffset = 0;

        while (previousRow = previousRow.previousElementSibling) {
            let previousCellIndex = 0;
            let cellIndexBulk = 0;

            for (let i = 0; i < previousRow.cells.length; i++) {
                if (previousRow.cells[i].colSpan > 1) {
                    cellIndexBulk += previousRow.cells[i].colSpan;
                } else {
                    cellIndexBulk += 1;
                }

                if ((cellIndexBulk - 1) >= currentCellIndex) {
                    previousCellIndex = i;
                    break;
                }
            }

            const previousCell = previousRow.cells[previousCellIndex];

            if (previousCell.classList.contains(fixationType + '_fixed_parent')) {
                topOffset += previousCell.getBoundingClientRect().height;
            }
        }

        if (topOffset > 0) {
            if (currentTDStyles.top) {
                topOffset += <any>currentTDStyles.top.replace('px', '') - 0;
            }

            currentTD.style.top = topOffset + 'px';
        }
    }

    //  
    if (fixationType === 'column' || fixationType === 'both') {
        //       
        //     .
        //   ,    .
        let previousCell: HTMLTableCellElement = currentTD;
        let leftOffset = 0;

        while (previousCell = previousCell.previousElementSibling) {
            if (previousCell.classList.contains(fixationType + '_fixed_parent')) {
                leftOffset += previousCell.getBoundingClientRect().width;
            }
        }

        if (leftOffset > 0) {
            if (currentTDStyles.left) {
                leftOffset += <any>currentTDStyles.left.replace('px', '') - 0;
            }

            currentTD.style.left = leftOffset + 'px';
        }
    }
}

問題番号2。レポートのエクスポート


システムからレポートをプルするには、（1）BlobオブジェクトのReportServiceを介してSSRSにアクセスし、（2）window.URL.createObjectURLメソッドを使用してインターフェースを介してオブジェクトへのリンクを取得し、（3）リンクをタグに挿入してクリックのシミュレーションを行う必要がありますファイルのアップロード。

これは、Firefox、Safari、およびAppleを除くすべてのバージョンのChromeで機能します。 IE、Edge、Chrome for iOSもこの機能をサポートするため、頭を下げなければなりませんでした。

IEとEdgeでは、イベントはファイルをダウンロードするブラウザー要求をトリガーしません。これらのブラウザには、クリックをシミュレートするために、ダウンロードするユーザーの確認が必要なだけでなく、追加のアクションを明確に示す機能があります。ソリューションは、IEとEdgeの両方で使用できるwindow.navigator.msSaveOrOpenBlob（）メソッドで見つかりました。彼は、ユーザーに操作の許可を求め、次に何をすべきかを明確にする方法を知っています。そこで、window.navigator.msSaveOrOpenBlobメソッドが存在するかどうかを判断し、その状況に対処します。

iOS上のChromeにはそのようなハッキングはありませんでした。レポートの代わりに、空白のページしかありませんでした。Webをさまようと、同様のストーリーが見つかりました。iOS13ではこのバグを修正する必要があると判断しました。残念ながら、私たちはiOS 12の時代にアプリケーションを書き直したので、結局私たちはこれ以上時間を無駄にしないことに決め、iOS向けChromeのボタンをオフにしました。
次に、UIへの最終的なエクスポートプロセスについて説明します。Angularレポートコンポーネントには、一連のステップを起動するボタンがあります。

  • ハンドラーは、イベントのパラメーターを通じて、エクスポート形式の識別子（「PDF」など）を受け取ります。
  • 指定された形式のBlobオブジェクトを受け取るリクエストをReportServiceに送信します。
  • ブラウザがIEかEdgeかを確認します。
  • 回答がReportServiceからのものである場合：
    • IEまたはEdgeの場合は、window.navigator.msSaveOrOpenBlob（fileStream、fileName）;を呼び出します。
    • それ以外の場合は、this.exportDownload（fileStream、fileName）メソッドを呼び出します。fileStreamはReportServiceへのリクエストから取得したBlobで、fileNameは保存するファイルの名前です。このメソッドは、window.URL.createObjectURL（fileStream）へのリンクを含む非表示のタグを作成し、クリックをシミュレートしてタグを削除します。

これが整理されて、最後の冒険は残りました。

SSRSはすぐに、見栄えの良いページネーションを含む既製のPDFを提供します。これにより、以前のバージョンのすべての問題から解放されます。唯一の問題は、ブラウザーでPDFを印刷できるかどうかです。

PDFはサードパーティの標準であるため、ブラウザはさまざまなビューアプラグインを通じてPDFをサポートしています。プラグインも漫画もないので、ここでも別のオプションが必要です。

また、PDFを画像としてページに配置し、このページを送信して印刷する場合はどうでしょうか。そのようなレンダリングを提供するAngularのライブラリとコンポーネントはすでにあります。検索、実験、実装。

印刷したくないデータを扱わないようにするために、レンダリングされたコンテンツを新しいページに転送することが決定され、すでにwindow.print（）が実行されています。その結果、プロセス全体は次のようになります。

  1. レポートをPDF形式でエクスポートするようにReportServiceに要求します。
  2. Blobオブジェクトを取得し、それをURL（URL.createObjectURL（fileStream））に変換し、レンダリングのためにPDFビューアにURLを渡します。
  3. PDFビューアから画像を取得します。
  4. 新しいページを開き、そこに小さなマークアップ（タイトル、小さなインデント）を追加します。
  5. PDFビューアからマークアップに画像を追加し、window.print（）を呼び出します。

いくつかのチェックの後、JSコードもページに表示され、印刷前にすべてのイメージが読み込まれたことをチェックします。

したがって、ドキュメントの全体的な外観はSSRSテンプレートのパラメーターによって決定され、UIはこのプロセスを妨げません。これにより、可能なバグの数が減ります。画像は印刷のために転送されているため、レイアウトの損傷や変形に対する保証が付いています。

欠点もあります：

  • 大きなレポートは非​​常に重く、モバイルプラットフォームに悪影響を及ぼします。
  • デザインは自動的に更新されません。色、フォント、その他のデザイン要素をテンプレートレベルでインストールする必要があります。

今回のケースでは、新しいテンプレートを頻繁に追加することは想定されていなかったため、ソリューションは許容範囲内でした。モバイルのパフォーマンスは当たり前です。
