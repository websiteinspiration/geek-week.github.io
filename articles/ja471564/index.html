TP-Link TL-WN727Nを使用したRaspberryPiのフレンド

こんにちは、ハブル！

どういうわけか、私はラズベリーを空中でインターネットに接続することを考えました。

言うまでもなく、このために、有名なTP-Link会社の笛が最寄りのUSB Wi-Fiストアで購入されました。これは、なんらかのナノUSBモジュールではなく、通常のフラッシュドライブと同じくらいの大きさの寸法のデバイス（まあ、または成人男性の人差し指で好きな場合）だとすぐに言っておく必要があります。購入する前に、RPIとTP-Linkでサポートされているホイッスルメーカーのリストを少し調べました（後で判明しましたが、詳細には悪魔が含まれていることがわかっているため、微妙な点は考慮していません）。それで、私の不幸な冒険の冷たい話が始まります。3つの部分からなる探偵小説があなたの注意を引くように提供されています。猫の下で興味を持ってください。

記事WN727N WiFiアダプターをUbuntu / Mintに接続する 部分的に私を助けましたが、最初に最初に。

一般的に、ご存知のように、LinuxでのUSB Wi-Fiの使用は、（ロシアの寿司のように）痛くて少し味がありません。

ボックスには、ドライバーCDも含まれています。あまり期待せずに、私はその上にあるものを調べます-確かに、彼らは気にしませんでした。インターネットで検索するとメーカーのウェブサイトに行きましたが、v4デバイスを修正するためのLinuxドライバーしかなく、v5.21を手元に置いていました。さらに、カーネル2.6-3.16の非常に古いバージョンでは。最初の失敗に落胆し、私はすでにTL-WN727Nを取る必要があるとすでに考えていました（これは少し高価であり、私の場合は150 Mbpsに対して300Mbpsを実行できますが、これはラズベリーには無関係であることが判明しました。後で書きます）。しかし最も重要なのは、そのためのドライバーがすでに存在し、ファームウェア-ralinkパッケージによって単にインストールされることです。。通常、デバイスのケースのシリアル番号の横にあるステッカーで、デバイスのリビジョンを確認できます。

さらにグーグルで検索してさまざまなフォーラムにアクセスしても、特別なことは何もありませんでした。どうやら私より先にそのようなアダプタをLinuxに接続しようとした人はいません。うーん、私は溺れた男として幸運です。

いいえ、私は嘘をついていますが、フォーラム（主に英語）へのアクセスも報われました。一部のトピックでは、Wi-Fiアダプター用の多数のドライバーを書いたことで有名なlwfinger氏の話がありました。リンクの記事の最後にある彼のgitリポジトリ。そして私が学んだ2番目の教訓は、デバイスがどのドライバーで機能するかを理解するには、デバイスを特定する必要があるということです。

パート1. ボーンの識別


ポートでデバイスをオンにしても、LEDは点灯しません。そして、一般的に、何らかの方法で何かが機能するかどうかは明確ではありません。

まず、カーネルがデバイスを認識しているかどうかを確認するために、dmesgを調べます。

[  965.606998] usb 1-1.3: new high-speed USB device number 9 using dwc_otg
[  965.738195] usb 1-1.3: New USB device found, idVendor=2357, idProduct=0111, bcdDevice= 0.00
[  965.738219] usb 1-1.3: New USB device strings: Mfr=1, Product=2, SerialNumber=3
[  965.738231] usb 1-1.3: Product: 802.11n NIC
[  965.738243] usb 1-1.3: Manufacturer: Realtek
[  965.738255] usb 1-1.3: SerialNumber: 00E04C0001

判明したことが判明し、USBバス自体にRealtekチップとVID / PIDデバイスがあることも明らかです。

さらに進んでlsusbを確認すると、別の失敗が待っています。

Bus 001 Device 008: ID 2357:0111 
Bus 001 Device 003: ID 0424:ec00 Standard Microsystems Corp. SMSC9512/9514 Fast Ethernet Adapter
Bus 001 Device 002: ID 0424:9514 Standard Microsystems Corp. SMC9514 Hub
Bus 001 Device 001: ID 1d6b:0002 Linux Foundation 2.0 root hub

システムはこのデバイスが何であるかを認識せず、名前の代わりに恥ずかしそうに空のスペースを表示します（ベンダー= 2357は間違いなくTP-Linkです）。

この段階で、好奇心旺盛な読者はおそらく何か面白いことに気づいたでしょうが、私たちはそれを私たちの時まで延期します。

空の名前の問題を調査すると、既知のVID / PIDに関する情報が入力される識別子のあるサイトに私を連れて行きました。 2357：0111がありませんでした。後で判明したように、lsusbはファイル/usr/share/misc/usb.idsを使用します。これは、このサイトの識別子と同じリストです。表示の美しさのために、システムにTP-Linkベンダーの行を追加しました。

2357  TP-Link
        0111  TL-WN727N v5.21

さて、デバイスのリストの表示を修正しましたが、これではドライバーの選択に一歩近づくことができませんでした。ドライバーを選択するには、ホイッスルが作成されているチップを知る必要があります。インターネット上で発見する次の失敗した試みは、何も良いことにつながりませんでした。細いマイナスドライバーで武装して、アダプターキャップをそっとこじ開けると、Liaoおじさんの無垢なヌードの中に悪意のある子供がいるのが見えます。拡大鏡の下に、チップの名前-RTL8188EUSが表示されます。これはもういいです。いくつかのフォーラムで、そのrwfinger氏のドライバーがこのチップに適しているという投稿を見ました（RTL8188EUについてのみ述べているにもかかわらず）。

パート2.ボーンの覇権


ドライバーのソースをgitaからダウンロードします。Linuxが通常関連付けられていることを実行するために、ひどいもの

を再インストールする時が来ました-ソートから何かを組み立てます。ドライバーのアセンブリは、プログラムのコンパイルと少し異なることがわかりました。

make
sudo make install

カーネルモジュールをコンパイルするには、特定のバージョンのカーネルヘッダーファイルが必要です。

ストックリポジトリにraspberrypi-kernel-headersパッケージがありますが、これにはカーネル4.19.66-v7l +のファイルのバージョンが含まれているため、これは適していません。しかし、目的のバージョンのヘッダーを取得するために、必要なヘッダーをダウンロードできる便利なrpi -sourceツール（githubの最後にあるリンク）があります。リポジトリのクローンを作成し、スクリプトを実行可能にして実行します。最初の起動はエラーで失敗します-bcユーティリティがありません。幸い、それはリポジトリにあり、インストールするだけです。

sudo apt-get install bc

この後、ヘッダーを再起動してダウンロードする（そして何かを設定するが、今は覚えていません）にはしばらく時間がかかり、座ってすべての症状を改善することができます。

すべてのヘッダーがダウンロードされたら、/ lib / modules / 4.19.73-v7 +ディレクトリが表示され、その中のシンボリックリンクがダウンロードしたファイルがある場所を指していることを確認します（これは/ home / pi / linuxです）：

pi@raspberrypi:/home/pi/rtl8188eu# ls -l /lib/modules/4.19.73-v7+/
lrwxrwxrwx  1 root root     14 Sep 24 22:44 build -> /home/pi/linux

準備段階が完了し、組み立てを開始できます。モジュールのアセンブリには再び一定の時間がかかります。ラズベリーの獣は速くありません（32ビットの900Mhz Cortex ARM v7石があります）。
したがって、すべてがコンパイルされました。2番目のステップ（make install）でドライバーをインストールし、ドライバーが機能するために必要なファームウェアファイルをコピーします。

install:
        install -p -m 644 8188eu.ko  $(MODDESTDIR)
        @if [ -a /lib/modules/$(KVER)/kernel/drivers/staging/rtl8188eu/r8188eu.ko ] ; then modprobe -r r8188eu; fi;
        @echo "blacklist r8188eu" > /etc/modprobe.d/50-8188eu.conf
        cp rtl8188eufw.bin /lib/firmware/.
        /sbin/depmod -a ${KVER}
        mkdir -p /lib/firmware/rtlwifi
        cp rtl8188eufw.bin /lib/firmware/rtlwifi/.

パート3.ボーンアルティメイタム


港に笛を差し込んでも……何も起こらない。それはすべて無駄になりましたか？

私はプロジェクト内のファイルを調査し始め、それらの1つで問題が何であるかを見つけます。ドライバーは、提供できるVID / PIDの完全なリストを示します。デバイスがこのドライバーで動作するように、rtl8188eu / os_dep / usb_intf.cファイルに IDを追加しました

最後まで読んだ人へのボーナス


アダプタに記載されている最大速度は問題ではないと言った方法を覚えていますか？
したがって、（モデル4のリリース前の）ラズベリーでは、すべてのデバイス（イーサネットアダプターを含む）が同じUSBバス上にあります。いいでしょう？そして、USBバスの帯域幅は、その上のすべてのデバイス間で共有されます。イーサネットとusb wi-fi（1台のルーターに接続）を介して、空中とワイヤーの両方で速度を測定すると、20 Mbpsの領域で速度が測定されました。

PS一般に、この特定のアダプター用のこのドライバーコンパイルガイドは、RPIだけでなく有効です。それを繰り返し、デスクトップ上でLinux Mintを実行しました-すべてがそこでも機能しました。カーネルのバージョンに必要なヘッダーファイルをダウンロードする必要があるだけです。

UPD。知識のある人からの提案：カーネルのバージョンに依存しないようにするには、dkmsを使用してドライバーをビルドしてインストールする必要があります。ドライバーへのreadmeでは、このオプションもそこにあります
pi@raspberrypi:/home/pi# sudo dkms add ./rtl8188eu
pi@raspberrypi:/home/pi# sudo dkms build 8188eu/1.0
pi@raspberrypi:/home/pi# sudo dkms install 8188eu/1.0


UPD2。提案されたデバイスIDのパッチは、メインストリームリポジトリブランチlwfinger / rtl8188euに受け入れられました。
