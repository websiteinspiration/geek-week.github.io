ServiceLoader：聞いたこともないような組み込みのDIフレームワーク

敬礼、友達。すでに今週の金曜日は、コース「Java Developer」の新しいグループの最初のレッスンになります。現在の出版物が専念されるのはこのコースです。



多くのJava開発者は、Springを使用して依存関係を実装しています。一部のユーザーは、Google GuiceまたはOSGiサービスを試したことがあるかもしれません。しかし多くの人は、JavaにすでにDIが組み込まれていることを知りません。あなたはそれがJava 11または12に登場したと思いますか？いいえ、Java 6で使用できます。

ServiceLoaderは、インターフェースまたは抽象クラスの登録済みインスタンスを検索および作成する機能を提供します。Springに精通している場合、これはBeanおよびAutowiredアノテーションと非常によく似ています。SpringとServiceLoaderの使用例を見てみましょう。そして、類似点と相違点について話し合います。

春


まず、Springで簡単なDIを作成する方法を見てみましょう。シンプルなインターフェースを作成します。

public interface SimpleService {
   String echo(String value);
}

そしてインターフェースの実装：

import org.springframework.stereotype.Component;

@Component
public class SimpleServiceImpl implements SimpleService {
   public String echo(final String value) {
       return value;
   }
}

に注意してください@Component。このアノテーションは、SpringコンテキストでBeanとしてクラスを登録します。

そして私たちのメインクラス。

@SpringBootApplication
public class SpringExample implements CommandLineRunner {
    private static final Logger log = 
            LoggerFactory.getLogger(SpringExample.class);

   @Autowired
   List<SimpleService> simpleServices;

   public static void main(String[] args) {
       SpringApplication.run(SpringExample.class, args);

   }

   public void run(final String... strings) throws Exception {
       for (SimpleService simpleService : simpleServices) {
           log.info("Echo: " + simpleService.echo(strings[0]));
       }
   }
}

@Autowiredコンボボックスの 注釈に注意してくださいSimpleService。注釈は@SpringBootApplication、パッケージ内のBeanの自動検索を目的としています。その後、起動時に自動的にに注入されSpringExampleます。

サービスローダー


Springの例と同じインターフェイスを使用するため、ここでは繰り返しません。代わりに、すぐにサービスの実装を確認します。

import com.google.auto.service.AutoService;

@AutoService(SimpleService.class)
public class SimpleServiceImpl implements SimpleService {
   public String echo(final String value) {
       return value;
   }
}

実装では、アノテーションを使用してサービスインスタンスを「登録」します@AutoService。このアノテーションは、javacがそれを使用してサービス登録ファイルを自動的に生成するため、コンパイル時にのみ必要です（翻訳者注：を含むMaven依存関係の@AutoService場合、スコープを指定-を指定）：

META-INF/services/io.github.efenglu.serviceLoader.example.SimpleService

このファイルには、サービスを実装するクラスのリストが含まれています：

io.github.efenglu.serviceLoader.example.SimpleServiceImpl

ファイル名はサービス（インターフェース）の完全な名前。ファイルには、任意の数の実装を含めることができ、それぞれ別の行にあります。

実装では、存在しなければならないパラメータなしコンストラクタです。このようなファイルは手動で作成できますが、注釈を使用する方がはるかに簡単です。そしてメインクラス：

public class ServiceLoaderExample {

   public static void main(String [] args) {
       final ServiceLoader<SimpleService> services = ServiceLoader.load(SimpleService.class);
       for (SimpleService service : services) {
           System.out.println("Echo: " + service.echo(args[0]));
       }
   }
}

メソッドServiceLoader.loadはgetのためServiceLoaderに呼び出され、サービスのインスタンスを取得するために使用できます。ServiceLoaderインスタンスIterableはサービスタイプのインターフェースを実装するため、変数servicesをループで使用できますfor each。

だから何？


どちらの方法も比較的小規模です。どちらも注釈とともに使用できるため、かなり簡単に使用できます。では、なぜSpringの代わりにServiceLoaderを使用するのでしょうか。

私たちが提供する依存関係に注意を払っていない場合は、ServiceLoaderはありNOの依存関係を。そうです、彼はJavaだけを必要とします。

Springベースのアプリケーションを開発しているかどうかは重要ではありませんが、多くの異なるフレームワークで使用されるものを作成している場合、または小さなコンソールアプリケーションを持っている場合、これはすでに非常に重要です。

速度


コンソールアプリケーションでは、ServiceLoaderの起動時間があるMUCH春ブーツのAppよりも短いです。これは、ダウンロード可能なコードの数が少ないこと、スキャンの欠如、リフレクションの欠如、大きなフレームワークの欠如が原因です。

記憶


春はメモリを節約することで有名ではありません。メモリの使用が重要な場合は、ServiceLoader for DIの使用を検討する必要があります。

Javaモジュール


Javaモジュールの重要な側面の1つは、モジュール内のクラスをモジュール外のコードから完全に保護する機能でした。ServiceLoaderは、外部コードが内部実装に「アクセス」できるようにするメカニズムです。Javaモジュールを使用すると、境界を保持しながら、内部実装のサービスを登録できます。

実際、これは、Javaモジュールの唯一の公式に承認された依存性注入サポートメカニズムです。Springおよび他のほとんどのDIフレームワークは、リフレクションを使用してコンポーネントを見つけ、接続します。ただし、これはJavaモジュールと互換性がありません。リフレクションでさえモジュールを調べることはできません（許可しない限り、なぜ許可する必要があるのですか）。

結論


春は素晴らしいことです。これは、ServiceLoaderで提供されるよりもはるかに多くの機能を備えています。しかし、ServiceLoaderが正しい選択である場合があります。シンプルで小さく、高速で、いつでも利用できます。
