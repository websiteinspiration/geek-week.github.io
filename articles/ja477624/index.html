EXPLAINが何について沈黙しているのか、そしてそれについて話す方法

開発者がDBAまたはPostgreSQLコンサルタントであるビジネスオーナーに来るという典型的な質問は、ほとんどの場合同じように聞こえます。

従来の一連の理由：


  • 数万のレコードに対していくつかのCTEのJOINを作成する場合の非効率なアルゴリズム

  • 表内のデータの実際の分布が、ANALYZEが最後に収集した時刻とすでに大きく異なっている場合の無関係な統計
  • リソースに「プラグイン」し
    、CPUの割り当てられた計算能力がすでに不足している場合、ギガバイトのメモリが常にポンプアップされるか、ディスクがすべてのデータベースの「ウィッシュリスト」に追いついていない
  • 競合するプロセスからのブロック

そして、ロックをキャプチャして分析することが非常に難しい場合は、他のすべてについて、EXPLAIN演算子（もちろん、すぐにEXPLAIN（ANALYZE、BUFFERS）...）またはauto_explainモジュールを使用して取得できるクエリプランが必要です。 しかし、同じドキュメントに記載されているように、

「計画を理解することは芸術であり、それを習得するには、ある程度の経験が必要です...」
しかし、適切なツールを使用すれば、それなしで実行できます。

しかし、「シートから」というテキストで計画を読むことは非常に難しく、愛されています。

  • ノードは、サブツリーのリソースの合計を表示します。つまり
    、特定のノードの実行にかかった時間、またはテーブルからのこの読み取りがディスクからデータを生成した正確な量を理解するために、何らかの方法で一方を他方から減算する必要があります。
  • ノード時間はループで乗算する必要があります
    はい、減算は「心の中で」行う必要がある最も難しい操作ではありません-結局のところ、実行時間はノードの1回の実行の平均で示され、数百にも及ぶ場合があります
  • まあ、そしてこれらすべてが一緒になって主な質問に答えることを難しくします-それで、「最も弱いリンク」は誰ですか？

リクエストのパラメータの置換


リクエストだけでなく、ログのDETAIL行からそのパラメーターもプランに「アタッチ」した場合は、オプションの1つにさらにそれをコピーできます。

  • クエリでの値の置換と、
    そのベースでの直接実行およびさらなるプロファイリング

    SELECT 'const', 'param'::text;
  • PREPARE / EXECUTE
    を介して値を置き換え、パラメトリックパーツを無視できる場合（たとえば、パーティション分割されたテーブルで作業している場合）にスケジューラーをエミュレートする

    DEALLOCATE ALL;
PREPARE q(text) AS SELECT 'const', $1::text;
EXECUTE q('param'::text);
