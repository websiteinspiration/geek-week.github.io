BaseEntity:

@MappedSuperclass
abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

We do not use data classes . This seems like a clear advantage to Kotlin over Java (prior to version 14), and there is an explanation for this.

Why not use?


Data classes, in addition to being final on their own, have certain fields for all fields equals, hashCodeand toString. And this is unacceptable in conjunction with Hibernate.

Why? And also why hashCodeit is always equal to a constant - the answer is in the documentation of Hibernate itself . Specifically, we are interested in this section:

Although using a natural-id is best for equals and hashCode , sometimes you only have the entity identifier that provides a unique constraint.

It's possible to use the entity identifier for equality check, but it needs a workaround:

  • you need to provide a constant value for hashCode so that the hash code value does not change before and after the entity is flushed.
  • you need to compare the entity identifier equality only for non-transient entities.

That is, you need to compare either by natural id, or, as in our example, by primary key id. This will avoid a lot of problems when comparing the entity and save from its loss when using the entity as an element in Set.

The presence toString, defined by all fields, completely kills all laziness, for example, when journalizing an entity, since all fields will be initialized for output to a string.

Given the features of Hibernate, this Kotlin functionality is not suitable for us.

Null validity in fields only according to DB


As for the validity of the values nullin the fields, everything is not so simple. Earlier we plunged a bit into the depths of Hibernate: speaking of plugin.jpa, I mentioned the use of a constructor without arguments when initializing an entity.

When initializing fields, reflection is also used. And if a value was stored in the database in the corresponding column null, the class is quietly initialized with this field with the value null. When accessing it, we run the risk of getting NPE, although the field is marked as not nullable. To avoid this, it is necessary to monitor the synchronism of the structure of tables and classes.

If you look at what is described in the last two sections more comprehensively, then these rules apply not only to primitives, but also to a bunch of entities.
