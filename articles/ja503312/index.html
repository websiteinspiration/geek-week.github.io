この知識は、開発者が扱っているアプリケーションが特定のサイズに達したときに特に重要です。メモリリークに十分な注意を払わないと、開発者が「メモリリークを解消するためのチーム」に参加することになります（私はそのようなチームの一員である必要がありました）。 メモリリークはさまざまな理由で発生する可能性があります。ただし、Angularを使用すると、メモリリークの最も一般的な原因と一致するパターンが発生する可能性があると思います。このようなメモリリークに対処する方法があります。もちろん、最善のことは問題と戦うことではなく、問題を回避することです。





メモリ管理とは何ですか？


JavaScriptは自動メモリ管理システムを使用しています。メモリのライフサイクルは通常、3つのステップで構成されています。

  1. 必要なメモリの割り当て。
  2. 割り当てられたメモリを操作して、読み取りおよび書き込み操作を実行します。
  3. 不要になったメモリを解放する。

上MDNその自動メモリ管理は言う-それは混乱の潜在的な供給源です。これは、開発者がメモリ管理について心配する必要がないという誤った感覚を与える可能性があります。

メモリ管理をまったく気にしない場合、これは、アプリケーションが特定のサイズに成長した後で、メモリリークが発生する可能性があることを意味します。

一般に、メモリリークは、アプリケーションに割り当てられたメモリであり、もはや必要ではないが解放されていないと考えることができます。つまり、ガベージコレクション操作に失敗したオブジェクトです。

ガベージコレクションはどのように機能しますか？


ガベージコレクションの手順は非常に論理的ですが、「ガベージ」と見なせるものはすべて削除されます。ガベージコレクターは、アプリケーションで不要になったメモリをクリーンアップします。アプリケーションでまだ必要なメモリ領域を見つけるために、ガベージコレクターは「マークアンドスイープ」アルゴリズム（タグ付けアルゴリズム）を使用します。名前が示すように、このアルゴリズムは、マーキングフェーズとスイープフェーズの2つのフェーズで構成されています。

ほとんどの場合、コンポーネントが繰り返し再レンダリングされると、時間の経過とともにメモリリークが発生します。たとえば、ルーティングを介して、またはディレクティブを使用した結果として*ngIf。たとえば、一部の上級ユーザーがブラウザでアプリケーションページを更新せずに1日中アプリケーションを操作している状況で、

@Component({
  selector: 'app-root',
  template: `<app-sub *ngIf="hide"></app-sub>`

  constructor() {

プロセスのメモリ消費が時間の経過とともに増加する場合、これはメモリリークに直面していることを意味します。プロセスが多かれ少なかれ一定の量のメモリを使用する場合、これはメモリリークがないか、またはリークが存在しても、それがかなり明白な方法で現れないことを意味します。

▍メモリリークはいつ発生しましたか？


最初の3つのシナリオでは、メモリリークを作成できなかったことにお気づきかもしれません。これら3つのシナリオには共通点があります。すべてのリンクはコンポーネントに対してローカルです。

監視可能なオブジェクトをサブスクライブすると、サブスクライバーのリストが格納されます。コールバックもこのリストにあり、コールバックはコンポーネントを参照できます。


メモリリークなし

コンポーネントが破棄されたとき、つまり、Angularにリンクがなくなったとき（つまり、コンポーネントがルートノードから到達できない場合）、観測されたオブジェクトとそのサブスクライバーのリストにもルートノードから到達できません。その結果、コンポーネントオブジェクト全体がガベージコレクションされます。

コンポーネント内のみにリンクされている監視可能なオブジェクトにサブスクライブしている限り、問題は発生しません。しかし、サービスが始まると状況は変わります。


メモリリーク

サービスまたは別のクラスによって提供される監視可能なオブジェクトをサブスクライブするとすぐに、メモリリークが発生しました。これは、サブスクライバーのリストのため、監視されたオブジェクトが原因です。このため、Angularにはコンポーネントへの直接参照はありませんが、コールバック、つまりコンポーネントにはルートノードからアクセスできます。その結果、ガベージコレクターは対応するオブジェクトに触れません。

明確にします。そのような構造を使用することはできますが、それらを正しく操作する必要があります。

適切なサブスクリプション作業


メモリリークを回避するには、サブスクリプションが不要になったときに、監視対象のオブジェクトから正しくサブスクライブを解除することが重要です。たとえば、コンポーネントが破壊された場合などです。監視対象オブジェクトの登録を解除するには、多くの方法があります。

大規模な企業プロジェクトの所有者に助言する経験は、この状況では、オペレーターと組み合わせてdestroy$チームによって作成されたエンティティを使用するのが最善であることを示してnew Subject<void>()いますtakeUntil。

@Component({
  selector:'app-sub',
  // ...
})
export class SubComponent implements OnDestroy {

  private destroy$: Subject<void> = new Subject<void>();
  randomNumber = 0;

  constructor(private dummyService: DummyService) {
      dummyService.some$.pipe(
          takeUntil(this.destroy$)
      ).subscribe(value => this.randomNumber = value);
  }

  ngOnDestroy(): void {
      this.destroy$.next();
      this.destroy$.complete();
  }
}

ここでは、コンポーネントの破棄後にdestroy$and演算子を使用してサブスクリプションを解除しtakeUntilます。

コンポーネントにライフサイクルフックを実装しましたngOnDestroy。コンポーネントが破棄されるたびに、destroy$メソッドnextとを呼び出しますcomplete。

この呼び出しはcompleteからサブスクリプションをクリアするため、この呼び出しは非常に重要destroy$です。

次に、演算子を使用takeUntilしてストリームに渡しますdestroy$。これにより、コンポーネントが破棄された後、サブスクリプションが確実にクリアされます（つまり、サブスクリプションからのサブスクライブが解除されます）。

サブスクリプションをクリアすることを忘れないようにするには？


コンポーネントを追加したり、destroy$呼び出しを忘れnextたりcomplete、フックのライフサイクルを実行したりするのは簡単ngOnDestroyです。私がプロジェクトに取り組んでいるチームにこれを教えたにもかかわらず、私はしばしばそれを自分で忘れていました。

幸い、リンターの素晴らしいルールがあります。これは、サブスクリプションからの適切なサブスクリプション解除を確実にすることができる一連のルールの一部です。次のようなルールセットを設定できます。

npm install @angular-extensions/lint-rules --save-dev

次に、以下に接続する必要がありますtslint.json。

{
  "extends": [
    "tslint:recommended",
    "@angular-extensions/lint-rules"
  ]
}

この一連のルールをプロジェクトで使用することを強くお勧めします。これにより、メモリリークの原因を特定するためのデバッグにかかる​​時間を節約できます。
