Comme dans toute application avec plus d'une entité, nous créerons un ancêtre commun pour tout le monde entity.

BaseEntity:

@MappedSuperclass
abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

Pourquoi ne pas utiliser?


Les classes de données, en plus d'être définitives par elles-mêmes, ont certains champs pour tous les champs equals, hashCodeet toString. Et cela est inacceptable avec Hibernate.

Pourquoi? Et pourquoi est-elle hashCodetoujours égale à une constante ? La réponse se trouve dans la documentation d'Hibernate elle-même . Plus précisément, nous sommes intéressés par cette section:

Bien que l'utilisation d'un identifiant naturel soit préférable pour equals et hashCode , parfois vous n'avez que l'identifiant d'entité qui fournit une contrainte unique.

Il est possible d'utiliser l'identifiant d'entité pour la vérification de l'égalité, mais il a besoin d'une solution de contournement:

  • vous devez fournir une valeur constante pour hashCode afin que la valeur du code de hachage ne change pas avant et après le vidage de l'entité.
  • vous devez comparer l'égalité d'identifiant d'entité uniquement pour les entités non transitoires.

Autrement dit, vous devez comparer soit par natural id, soit, comme dans notre exemple, par primary key id. Cela évitera beaucoup de problèmes lors de la comparaison de l'entité et économisera de sa perte lors de l'utilisation de l'entité comme élément dans Set.

La présence toString, définie par tous les champs, tue complètement toute paresse, par exemple, lors de la journalisation d'une entité, car tous les champs seront initialisés pour une sortie dans une chaîne.

Compte tenu des caractéristiques d'Hibernate, cette fonctionnalité Kotlin ne nous convient pas.

Validité nulle dans les champs selon DB uniquement


Quant à la validité des valeurs nulldans les champs, tout n'est pas si simple. Plus tôt, nous avons plongé un peu dans les profondeurs d'Hibernate: en parlant de plugin.jpa, j'ai mentionné l'utilisation d'un constructeur sans argument lors de l'initialisation d'une entité.

Lors de l'initialisation des champs, la réflexion est également utilisée. Et si une valeur a été stockée dans la base de données dans la colonne correspondante null, la classe est initialisée discrètement avec ce champ avec la valeur null. Lateinit var


Une autre fonctionnalité de Kotlin que je voudrais considérer est lateinit var.

Ajouter une nouvelle classe ancêtre: BaseAuditEntity.

@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class BaseAuditEntity<T> : BaseEntity<T>() {

   @CreatedDate
   @Column(updatable = false, nullable = false)
   lateinit var created: LocalDateTime

   @LastModifiedDate
   @Column(nullable = false)
   lateinit var modified: LocalDateTime
}

Prenons l'exemple lateinit vardes champs d'audit ( created, modified).

lateinit var- Il s'agit d'un not nullchamp d'initialisation retardée. L'accès au champ avant son initialisation génère une erreur:

kotlin.UninitializedPropertyAccessException: lateinit property has not been initialized

En règle générale, nous nous tournons vers les champs created, et modifiedmême après, comme une entité a été stockée dans la base de données. Dans ce cas, les données de ces polyahprostavlyayutsya l'étape de stockage et ils sont not null, alors lateinit varnous plus que convenable.
