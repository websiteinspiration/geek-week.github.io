DSA（直接状態アクセス）


直接状態アクセス-状態への直接アクセス。OpenGLオブジェクトをコンテキストにスナップせずに変更するためのツール。これにより、アプリケーションのすべての部分で共有されるグローバル状態に影響を与えることなく、ローカルコンテキストでオブジェクトの状態を変更できます。また、オブジェクトの状態を変更する関数を明確に定義できるため、APIがオブジェクト指向に少しなります。これはOpenGL Wikiが伝えていることです。

ご存知のとおり、OpenGLは、多数のラジオボタン（glActiveTexture、glBindTextureなど）を備えたAPIです。

ここからいくつかの問題があります：

  • セレクターと現在の状態は、状態をより深く変更できます。
  • テクスチャのフィルタを設定するには、アクティブユニットをバインド/変更する必要がある場合があります
  • アプリケーションの複雑さが増大する結果、状態管理が問題になる
  • 不明な状態は追加の設定につながります
  • 状態を保存/復元しようとすると問題が発生する可能性があります

Khronosグループは何を提供し、DSAはどのように役立ちますか？

  • オブジェクトを直接操作する関数を追加します
  • 現在のテクスチャオブジェクトではなく、指定したテクスチャオブジェクトのテクスチャフィルタを設定します。
  • テクスチャをアクティブではなく特定のユニットにバインドします
  • 非常に多くの新機能を追加します
  • OpenGL 1.xまでをカバー
  • 追加機能を追加します。

理論的には、DSAは非レンダリングおよび状態変更操作の数をゼロに減らすのに役立ちます...しかし、それは正確ではありません。

次に、いくつかの新機能について簡単に説明します。パラメータについて詳しく説明するのではなく、Wikiにリンクを残します。

  • glCreateTexturesは、glGenTextures + glBindTexture（初期化）を置き換えます。
    そうだった：
    glGenTextures(1, &name);
glBindTexture(GL_TEXTURE_2D, name);
    それは次のようになりました：
    glCreateTextures(GL_TEXTURE_2D, 1, &name);
  • glTexParameterXと同等のglTextureParameterX
    glGenTextures(1, &name);
glBindTexture(GL_TEXTURE_2D, name);

glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_NEAREST);

glTexStorage2D(GL_TEXTURE_2D, 1, GL_RGBA8, width, height);
glTexSubImage2D(GL_TEXTURE_2D, 0, 0, 0, width, height, GL_RGBA, GL_UNSIGNED_BYTE, pixels);
    これを次のように記述します。
    glCreateTextures(GL_TEXTURE_2D, 1, &name);

glTextureParameteri(name, GL_TEXTURE_WRAP_S, GL_CLAMP);
glTextureParameteri(name, GL_TEXTURE_WRAP_T, GL_CLAMP);
glTextureParameteri(name, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
glTextureParameteri(name, GL_TEXTURE_MAG_FILTER, GL_NEAREST);

glTextureStorage2D(name, 1, GL_RGBA8, width, height);
glTextureSubImage2D(name, 0, 0, 0, width, height, GL_RGBA, GL_UNSIGNED_BYTE, pixels);
  • glBindTextureUnit置き換えglActiveTexture + glBindTexture
    ここで我々はそれをやった方法は次のとおりです。
    glActiveTexture(GL_TEXTURE0 + 3);
glBindTexture(GL_TEXTURE_2D, name);
    今：
    glBindTextureUnit(3, name);

変更はglTextureImageにも影響しました。これは使用されなくなったため、

glTexImageはかなり安全ではなく、関数が呼び出されたときに値をチェックしないため、ドライバーが描画中にこれを行うため、無効なテクスチャが非常に簡単に取得されます。それを置き換えるためにGlTexStorageが追加されました。

glTexStorageは、呼び出し中に実行されるチェックでテクスチャを作成する方法を提供し、エラーを最小限に抑えます。テクスチャリポジトリは、すべてではないにしても、可変テクスチャによって引き起こされる問題のほとんどを解決しますが、不変テクスチャの方が信頼性が高くなります。

  • glVertexAttribFormatとglBindVertexBufferの代わりglVertexAttribPointer

デバッグ


バージョン4.3以降、デバッグ用に新しい機能が追加されました。私の意見では、非常に便利で便利です。これでOpenGLはエラーとデバッグメッセージのコールバックを呼び出し、レベルを調整できます。



有効にするために呼び出す必要があるのは、2つの関数glEnableとglDebugMessageCallbackだけです。

glEnable(GL_DEBUG_OUTPUT);
glDebugMessageCallback(message_callback, nullptr);

次に、メッセージを取得するためのコールバック関数を記述します。

void callback(GLenum source, GLenum type, GLuint id, GLenum severity, GLsizei length, GLchar const* message, void const* user_param)
{
   auto source_str = [source]() -> std::string {
	switch (source)
	{
	    case GL_DEBUG_SOURCE_API: return "API";
	    case GL_DEBUG_SOURCE_WINDOW_SYSTEM: return "WINDOW SYSTEM";
	    case GL_DEBUG_SOURCE_SHADER_COMPILER: return "SHADER COMPILER";
	    case GL_DEBUG_SOURCE_THIRD_PARTY:  return "THIRD PARTY";
	    case GL_DEBUG_SOURCE_APPLICATION: return "APPLICATION";
	    case GL_DEBUG_SOURCE_OTHER: return "OTHER";
        default: return "UNKNOWN";
	}
   }();

   auto type_str = [type]() {
	switch (type)
	{
	   case GL_DEBUG_TYPE_ERROR: return "ERROR";
	   case GL_DEBUG_TYPE_DEPRECATED_BEHAVIOR: return "DEPRECATED_BEHAVIOR";
	   case GL_DEBUG_TYPE_UNDEFINED_BEHAVIOR: return "UNDEFINED_BEHAVIOR";
	   case GL_DEBUG_TYPE_PORTABILITY: return "PORTABILITY";
	   case GL_DEBUG_TYPE_PERFORMANCE: return "PERFORMANCE";
	   case GL_DEBUG_TYPE_MARKER:  return "MARKER";
	   case GL_DEBUG_TYPE_OTHER: return "OTHER";
        default: return "UNKNOWN";
	}
   }();

   auto severity_str = [severity]() {
	switch (severity) {
	   case GL_DEBUG_SEVERITY_NOTIFICATION: return "NOTIFICATION";
	   case GL_DEBUG_SEVERITY_LOW: return "LOW";
	   case GL_DEBUG_SEVERITY_MEDIUM: return "MEDIUM";
	   case GL_DEBUG_SEVERITY_HIGH: return "HIGH";
         default: return "UNKNOWN";
	}
   }();

   std::cout << source_str       << ", " 
                 << type_str     << ", " 
                 << severity_str << ", " 
                 << id           << ": " 
                 << message      << std::endl;
}

glDebugMessageControl を使用してフィルターを設定することもできます。フィルターは、ソース/タイプ/重要度またはそれらの識別子を使用するメッセージのセットによってフィルタリングモードで機能します。

特定のスコープでメッセージをフィルタリングします。

glPushDebugGroup( GL_DEBUG_SOURCE_APPLICATION, DEPTH_FILL_ID, 11, “Depth Fill”); // 
Render_Depth_Only_Pass(); // 
glPopDebugGroup(); 	      //

関数呼び出しの順序を決定したり、デバッグ時にスタック上のエラーの場所を見つけたりできるように、非同期呼び出しを無効にすることは非常に役立ちます。これは非常に簡単に行われます：

glEnable(GL_DEBUG_OUTPUT_SYNCHRONOUS);

ラップされた呼び出し関数でOpenGLまたはウィンドウ関数を呼び出すのは安全ではなく、コールバックは無料ではないため、リリースビルドに含めたままにしないでください。
これらおよびその他の小さなことの詳細-仕掛け、ここで読むことができます。

SSO（個別のシェーダーオブジェクト）


OpenGLが「固定パイプライン」として機能すると、可視化のために転送されたすべてのデータに事前にプログラムされた処理が適用されたことになります。次のステップは「プログラマブルパイプライン」でした。プログラマブルパーツは、GLSLで記述されたシェーダーを実装します。クラシックGLSLプログラムは、頂点シェーダーとフラグメントシェーダーで構成されていました。次のパートでお話します）。


SSOを使用すると、再リンクせずにオンザフライでシェーダーのステップを変更できます。デバッグせずに単純なソフトウェアパイプラインを作成および構成する方法は次のとおりです。


GLuint pipe = GL_NONE;
// Create shaders
GLuint fprog = glCreateShaderProgramv( GL_FRAGMENT_SHADER, 1, &text);
GLuint vprog = glCreateShaderProgramv( GL_VERTEX_SHADER, 1, &text);
// Bind pipeline
glGenProgramPipelines( 1, &pipe);
glBindProgramPipelines( pipe);
// Bind shaders
glUseProgramStages( pipe, GL_FRAGMENT_SHADER_BIT, fprog);
glUseProgramStages( pipe, GL_VERTEX_SHADER_BIT, vprog);

glCreateProgramPipelinesがハンドルを生成してオブジェクトを初期化するの を見ると、glCreateShaderProgramvは指定されたソースを使用してシェーダープログラムを生成、初期化、コンパイル、およびリンクし、glUseProgramStagesはプログラムステップをパイプラインオブジェクトにアタッチします。glBindProgramPipeline-パイプラインをコンテキストに関連付けます。

ただし、注意点が1つあります。シェーダーの入力パラメーターと出力パラメーターが一致する必要があります。入力/出力パラメーターを同じ名前で同じ順序で宣言することも、修飾子を使用してそれらの場所を明確に一致させることもできます。
私は最後のオプションをお勧めします。これにより、明確に定義されたインターフェースを構成できるようになり、名前と順序に関して柔軟になります。

より厳密なインターフェイスを提供するには、各ステージで使用する組み込みの入力ブロックと出力ブロックを宣言する必要もあります。

ビルトインブロックインターフェイスは次のように定義されています（wikiから）：
頂点：

out gl_PerVertex
{
  vec4 gl_Position;
  float gl_PointSize;
  float gl_ClipDistance[];
};

テッセレーションコントロール：
out gl_PerVertex
{
  vec4 gl_Position;
  float gl_PointSize;
  float gl_ClipDistance[];
} gl_out[];

テッセレーション評価：
out gl_PerVertex {
  vec4 gl_Position;
  float gl_PointSize;
  float gl_ClipDistance[];
};

ジオメトリ：
out gl_PerVertex
{
  vec4 gl_Position;
  float gl_PointSize;
  float gl_ClipDistance[];
};

組み込みモジュールを再宣言し、通常の頂点シェーダーで属性の場所を使用する例：

#version 450

out gl_PerVertex { vec4 gl_Position; };

layout (location = 0) in vec3 position;
layout (location = 1) in vec3 color;

layout (location = 0) out v_out
{
    vec3 color;
} v_out;

void main()
{
    v_out.color = color;
    gl_Position = vec4(position, 1.0);
}
