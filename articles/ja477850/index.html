React Native-すべての問題に対する特効薬？Profi.ruのクロスプラットフォームツールをどのように選択したか

みなさん、こんにちは。私の名前はジェヴォルグです。私はProfi.ruのモバイル部門の責任者です。React Nativeを使った実験のストーリーを皆さんと共有したいと思います。React Nativeでの開発の長所と短所をどのように評価したかを、理論上および実際に説明します。この記事は、クロスプラットフォームのモバイル開発に関心があるが、この方法を採用するかどうかまだ決定していない人には役立ちます。

最大加速度



それはすべて、会社レベルで開発を10倍高速化することに決めたという事実から始まりました。通常の出来事の地平線を超えて、新しいことに挑戦することは不可能という目標を設定しました。すべてのProfi.ru開発チームが実験を行いました。当時、同社には、私と2人のチームリーダーを含む13人のネイティブモバイルデベロッパーがいました。私のチームは2つのモバイルアプリに取り組みました。 1つは顧客が専門家を、2つ目は顧客の専門家を探しています。私にとって、この期間は理解できず、感情的にストレスがたまりました。私の気持ちによると、すべてがすぐに機能するように、私たちは多くのことをしました。

プロジェクト全体で共通のアーキテクチャを使用し、コードの純度を監視しました。すべてのモジュールファイルを作成するジェネレーターを使用しました。彼らはバックエンドのすべてのビジネスロジックを取り除こうとしました。CI / CD、およびE2Eテストでカバーされるアプリケーションをセットアップします。このため、一部のアプリケーションは週1回安定してリリースされました。どうすれば開発を2倍に高速化できるかわかりませんでした。そのため、10ではるかに重要です。したがって、私たちは何が重要かを判断しました。

  1. 統一されたコードベース。私はすべてのモバイル開発者に同じコードを書いてほしかった。1つの言語で、iOSとAndroidのプラットフォームの違いはありません。したがって、開発を半分にスピードアップできました。
  2. 新しいツールを学ぶのは簡単。そのため、チームを拡大するときに、再トレーニングや採用の問題は発生しません。
  3. クイックリリース。週に一度ではなく、毎日リリースできるように。
  4. インスタントアップデート。すべてのユーザーが更新をすぐに受信できるようにします。それは今ウェブ開発で起こっているように。

短いレビューの後、React Native、Flutter、Kotlin / Nativeの3つの候補が登場しました。FlutterもKotlin Nativeもすぐにリリースできません。そして、これがおそらく最も重要であると考えました。加えて、これらの技術は当時完全に未加工でした。React Nativeを採用しました-すぐにリリースできます。さらに、ほとんどの開発者は既にReactを使用しています。

一般に、私はクロスプラットフォームツールについて否定的でした-ほとんどのネイティブモバイル開発者のように。任意のモバイル会議に行き、それについて話します-あなたはすぐに石になります。私はこのことを自分で愛しています:-)したがって、恐れを確認または反駁するために、調査を行いました。

長所、リスク、問題


さまざまな企業でのReact Nativeの使用例を調査しました-成功し、あまり良くありません。開発責任者を務めるBorey Egorovは、30を超える記事やその他の資料を注意深く読みました。一部は各パラグラフについて議論しました。記事の最後-最も興味深いものへのリンク。私たちは、私たちを加速することができるポイント、起こり得るリスクと質問に注意しました。その後、3社の開発者と話をしました。男たちはそれぞれ、大量生産の製品を作り、少なくとも1年間はReact Nativeと協力しました。

プロがいると、すべてがかなり明白でした。

  1. 一般的なコードベース。
  2. 店頭を経由せずに無線、または無線で更新します。
  3. 最初の2つのポイントから、ユーザーへの機能の配信速度が向上することになります。
  4. - . - React, React Native. , -.

リスクのリストはより長くなりました:-)

リスク1。 1つのプラットフォームではなく、長期的には、Android、iOS、React Nativeの3つをサポートする必要があります。

時々、開発者の画面は次のようになります：



現実。対談者の1人がReact Nativeを既存のコードに挿入しました。はい、本格的な3番目のプラットフォームが表示されますが、ネイティブ開発からどこにも行けません。彼のチームは、React Nativeとネイティブコードの間で状態を同期する必要がありました。これには、さまざまなコード/さまざまなパラダイムとIDEの間で多くの切り替えが必要でした。したがって、彼らはゼロから新しいプロジェクトを作成し、React Nativeでフレームワークを作成し、必要な場所にすでにネイティブピースを挿入することにしました。よくなった。

2番目のリスク。React Native Black Box-バグが発生した理由を開発者が理解できない場合があります。 React Nativeコード内、製品のネイティブ部分内、またはReact Nativeプラットフォーム自体内のどこでも検索する必要があります。

現実。私たちが話し合った人たちは、ログとさまざまなツール（Crashlytics、Kibanaなど）でアプリケーションを包みました。問題は残っていますが、どこで発生するかが明らかになります。

3番目のリスク。 React Nativeは小規模なプロジェクトに適しているが、プラットフォーム機能を備えた大規模な製品には適していないことが記事でよく見られます。

現実。 React Nativeと連携する大企業が市場にあるかどうかを調べました。それは判明しました-数百ではないとしても、数十があります。エリア内のSkype、テスラ、ウォルマート、Uber Eats、キッチンを含みます。

4番目のリスク。 AppleまたはGoogleからオペレーティングシステムをアップグレードするたびに、プロジェクトが中断する可能性があります。

現実。リスクは完全に許容できると判断しました。同じリスクがネイティブ開発にも存在します。 iOSおよびAndroid用の新しいOSがリリースされたら、アプリケーションをそれに適合させます。

5番目のリスク。 Androidでは64ビットシステムはサポートされておらず、問題は2015年以降オープンしています。また、2019年8月以降、Google Playは32ビットシステムのみをサポートするビルドを受け入れていません。

現実。2018年の夏にReact Nativeチームが取り組んだ問題を調べました。彼らは64ビットシステムのサポートをまだ完全に修正していませんが、次のリリースでサポートを追加することを約束しました。これは非常に動揺しています。サポートは後で追加されましたが、一部のAndroidデバイスは移行後に分類されます。後でわかったように、パーセンテージは不十分ですが、それでも私にとって最も苦しい点でした。

6番目のリスク。明日、アップルまたはグーグルが彼らのOSの新しいバージョンをリリースして、React Nativeを壊す可能性。または、Profi.ruが原則としてサポートできない新しいテクノロジー。

現実。当社または他の多くの会社に対して保証はありません。リスクを認識してそれを行うか、別の方法を試してください。私たちはそうすることを決め、それらが利用可能になったときにすべての問題を解決することにしました。

7番目のリスク。 React Nativeがネイティブアプリケーションと比較してどれほど高速であり、どのようなパフォーマンスを示すかを前もって言うことはできませんでした。

現実。私たちの対話者の一人の文字通りの引用は、「スクロールが遅くなったときの可変高さの要素のリスト」です。実際に確認することにしました。少し先に進みます-アプリケーションの最初のプロトタイプを作成した時点では、そのような問題はありませんでしたが、本格的なアプリケーションを開発したとき、React Nativeのパフォーマンスについて多くの疑問がありました。

8番目のリスク。 React Nativeの開発者をどれだけ早く見つけることができるかは不明です。 HeadHunterで、iOSには15万人以上の開発者がいるにもかかわらず、約300の履歴書が見つかりました。

現実。彼らはReact開発者を複数回雇い、何に注意を払うべきかを知っていたため、ここには深く入りませんでした。最後の手段として、React開発者をReact Nativeで再トレーニングできると判断しました。

また、モバイルデベロッパーはこのテクノロジーを好まないため、誰かがチームを離れるリスクもありました。ちなみに私は正しかった。誰かが残しました:-(

私はReact Nativeを書くことにうんざりしているので、以下はRNです:-)

何が変化し、何が変化しないか


調査の結果については、会社の創設者であるセルゲイクズネツォフ氏とイゴールルディ氏と話し合い、実験の承認を得ました。

私たちは、既存の製品に挿入するのではなく、新しい製品を最初から作成することにしました。また、クライアントアプリケーションには触れないでください。それはかなり成熟していて、経済的に何かを根本的に変えることは意味がありませんでした。さらに、クライアントアプリケーションがiOSとAndroidの両方で独自のネイティブエクスペリエンスを備えていることが重要でした。

しかし、専門家向けのアプリケーションを根本的に変えたかったのです。クライアントとは異なり、iOSとAndroidアプリケーションでスペシャリストが同じインタラクションエクスペリエンスを使用できることは恥ずかしくありませんでした。さらに、専門家向けの製品では、アニメーションや視覚効果なしでできると信じています。しかし、チーム全体が新しいテクノロジーに切り替える前に、それがどのように機能するかを感じる必要がありました。

実験中




2018年12月、3人のチームを編成しました。2人のReact開発者と1人のネイティブ-私。私はAndroidの仕組みを理解しており、iOS開発にも精通しています。

実験の一環として、次の点を確認したいと思いました。

  • RNでインスタントリリースがどのように機能するか。
  • ネイティブコンポーネントとRNの間の相互作用はどうですか。
  • ネイティブコンポーネントを使用できますか。
  • RNがカメラ、プッシュ、ディップリンクでどのように機能するか。
  • RNでのナビゲーションと状態保存の仕組み。
  • RN pixel perfectでできる限り。
  • RNでの自動テストの仕組み。
  • ネイティブまたはReactの開発者がどれだけ早くテクノロジーを学ぶことができるか。
