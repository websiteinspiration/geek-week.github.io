Leroy Merlinに200人のロシア人開発部門が必要な理由

こんにちは！私はロシアのLM開発責任者であるValery Laptevです。 2年間、私は巨大な部門を上げる必要があり、それはかなり興味深い経験でした。

リロイ・マーリンが多くの国にいるという事実。フランスの親会社はADEOと呼ばれています。フランス、イタリア、スペイン、ロシア向けのコードを記述します。私たちのビジネスモデルは異なります。ロシア市場で最低価格を維持している場合（すべての競合他社の監視を下回っている場合）、ヨーロッパではすべてが異なります。実際、海は地域の特性から他の法律まで、異なっています。ロシアのインフラストラクチャー（ハバロフスクと同じ非常に大きな遅延）の特徴と、注文を行うための別のライフサイクルがあります。これらすべてが、巨大なIFブロックで構成されるこのような地獄のようなコードを生み出します。



2年前には、60の店舗と多くのウィッシュリスト機能がありました。彼らは約6か月で転がりましたが、常に正しくはありません。低優先度で拒否された一連の機能の後の最後のストローは、後で解析するためにフィールド文字列を順序で入力する要求でした。ロシアはLMが存在する他の国よりも大きいため、これはロシアでの配信の機能に必要でした。彼らは私たちを拒否しました、より正確には、それは7〜8ヶ月でどこかにあると彼らは言いました。

のんびりした親会社の半年サイクルが合わなかった。当然のことながら、独自のコードを記述してレビューに提出し、実装を待機することを提案しました...確かに、これで良いことは何もありませんでした。

機能の実装が遅いのはなぜですか？


ストーリーは非常にシンプルです。私たちの数十の店舗にもかかわらず、私たちは世界で最初のものではありません。フランス（親組織があるところ）のニーズがあり、他の国々のニーズがあります。企業グループの利益またはコスト削減の可能性を優先し、この順序で実行されます。つまり、非常に幸運である場合を除いて、私たちの機能はほとんど作成されていません。一部のスプリントでは、誰かが自分のパートを早く完了してバックログの下部を分解します。

これは、非常に大きなテストカバレッジ、長いプレリリース手順、そして急速に成長しているコードを維持することへの抵抗を意味します。したがって、機能が少ないほど良いです。しかし、あなたはそこを保持します。

全体としての立場は非常にわかりやすいので、親会社の現場でやればいいと思います。合理的だからです。

問題の解決策


最初のアプローチは額にありました：そこでプルリクエストを実行するためのコードを開くことを提案しました。ビジネスクリティカルな機能を迅速かつ迅速にコーディングし、フランス語にそれらを提供し、独自の手順でそれをテストする約10人の開発者がそこに座っていると想定され、すべてがうまくいくでしょう。

実際、これらの人々はすでにそうでした：私たちは、さまざまな累積割引や珍しい株（私たちのビジネスモデルでは単純に不可能）など、他の国から得た余分なビジネスロジックを切り捨て、ダミーをこの場所に置いていました。

ADEOのフランス人は、自分たちで新しいバージョンをローリングしてインストールするためのリリースサイクルを望んでいました。実験用の1つのブランチに関する提案を受け入れました。

アクセスを与え、レビューのためのコードを取り始めました。とにかく遅いことがわかりました。数か月の展開-これはそうではありません。まあ、彼らは数週間で勝利しましたが、それでもプロセスはうまくいきませんでした。

その後、6か月間、コンテンツ部分（クライアントに表示される製品データの管理）で私たちにとって重要な機能が出ませんでした。私達は解放なしで6か月間座りました。彼らの機能がうまくいかなかったか、テストに合格しなかったのなら、彼らは私たちが何を必要としているかを正確に理解していませんでした。その結果、彼らはアップデートなしで長い間重要なシステムに座っていました。正面にはNodeJS + PostgreSQL + Couchbase + Elasticsearch + Angularがありました。コードでは、多くの考古学的なレイヤーにより、SQLデータベースや非リレーショナルデータベースの誤用などの問題が発生しています。いずれかの場所で、大量のマスターデータが取得され、SQLデータベースの1つのフィールドに挿入されてから、NoSQLデータベースのエンティティに分割されました。商品が展示されているサイトの1ページに、数十から数百のそのようなリクエストがありました。この遺産をさらに読むと、体のさまざまな部分の髪の毛が動き始めました。わかりましたヘッドセクションが直面しているレガシーレイヤーの機能

自分の開発


最初のアイデアは、機能を一緒に実行することでした。その場で、つまりフランスで。私たち4人はフランスに行き、同僚の隣に座って、物事を整理し、必要に応じて行動し始めました。

うまくいきませんでした。すべて同じように、すべてが非常に遅かった。

2番目のアイデアは、すでに存在するものをすべてフォークして、徐々にそれを完成させることでした。私たちは建築委員会の前に座り、見通しを評価し、おおよその開発計画を計算し、別の方法を選択する必要があることに気付きました。具体的には、はい。フォークするが、既存のコードを開発するのではなく、モノリスを分割して、必要な場所にマイクロサービスを配置します。

つまり、ロジックのブロック全体をコードの形で書き直すことを計画しました。そしてこのために、彼らはサービスの一部を私たちでできることに切り替え始めました。開発者の募集を始めました。次に、親会社のスタックを完全に追跡しました-Java、Spring、およびすべてが近くにあります。Couchbaseの代わりにMongoがありました（これは同様のNoSQLベースです）。

プロジェクトが発展するにつれ、独自の方法で多くのことを行う必要があることに気づき（特に必要のないレガシーをサポートしないため、より速くて簡単であるため）、他のテクノロジーに拡大し始めました。次に、Java 7、Wildfly（旧称JBoss）、SVNがありました。 Java 8、GIT、Tomcatに移行しない理由を尋ねました。彼らは気にしなかったが、数年後。その間、親愛なる皆さん、古いスタックに書いてください。だから私たちは一言一句、完全に分離することにしました。ビジネスは、問題の長所と短所を整理し、完全にサポートしました。

ほとんどすぐにコードの約30％をスローし、マイクロサービスの多くを書きました。彼らが触れなかったという事実から、それはお金のためのビジネスプロセスのトランザクションのほぼ全体のコアです。

当然、私たちが最初に考えたのは、開発領域をどのように区別するかです。これについても個別に説明しますが、一般的には次のようなスキームです。



水平はビジネスエリアです。たとえば、顧客関係（最初の緑色のセル）に関連するものはすべて1つの領域であり、1つの部署からの一連のアプリケーションがあります。目的関数をこのように分離すると、さまざまな場所にコードの複製がいくつか作成され、適切な企業バス（ここでも別の話）が必要になりますが、目的を明確に見つけ、問題が解決するまで解決できます。ほぼ3年後にこれを見ると、アーキテクチャは正しく選択されたと言えますが、私が今知っていることを知っていれば、いくつかの変更を加えることになります。

一般的な構造には、大規模な製品チームを構成する機能チームが多数あるという結論に達しました。同時に、製品チームには開発者自身だけでなく、デザイナーやビジネス担当者も含まれます。企業のIT部門の最終的な目標は、ビジネス開発のスピードを上げるか、自動化によるコストを削減することであるため、これらのチーム内にビジネス担当者が必要です。小売はすべてITに関するものです。 「接近不可能」と呼ぶことができる単一のプロセスはありません。

機能チームは、少人数のグループです（通常、アナリスト、テスター、バックエンド開発者、フロントエンド開発者、オペレーション）。アナリストは通常​​、製品の所有者の役割を果たすか、ビジネスの人がこの場所に来ます。所有者には、バックログ、優先度、タスクがあります。彼は彼らの発展を指示します。開発者は実装オプションを自分で選択し、何をどのように行うかをチームで話し合います。タスクの評価、コーディング、意思決定についてチームのリーダーはいない。誰もがすべてを行います。通常、経験豊富なチームメンバーが、多くの人が信頼できる意見を述べます。しかし、誰でも決定を下すことができます。もちろん、2人の開発者が機能の実装に同意できない場合、競合が発生します。対立が個人に向けられた場合-あなたは頭へのエスカレーションが必要です。しかし、そのほとんどは実装のためのソリューションの選択です。次に、全員が立ち、ボードに近づいていきます。それは通常すぐに判明しますが、彼らが一日中議論したケースがありました。議論が行き詰まると、審判はしばしば呼ばれます-誰もが信頼する意見を持つ別のチームの人。彼らは問題の本質を説明し、彼は解決します。 6月や研修生でさえこの論争に参加できますが、私が知っているのはそのような場合のみです。通常、ジョーンズはメンターに耳を傾けています。

機能チームの例：請負業者と一緒に商品とともにサービスを購入できるサービスプラットフォームがあります。たとえば、私はドアを購入しました-バスケットと設置にすぐに入れることができるので、独立したマスターが私たちの基準に従って来て行動します。確認・お支払い・保証を致します。したがって、このチームは製品を作成します。このためにITソリューションを作成し、ビジネス上の意思決定を行い、店舗のプロセスを変更します。請負業者に同意します。そこでは-ビジネスの製品の所有者、店の店員、建築家、開発者、アナリスト、テスター。企業のAPIプラットフォームをすぐに使用して、すべてのデータをやり取りし、マイクロサービスを作成します。このようなアプローチ-開発者や小さなチームと連携してすぐに運用やビジネスを行う人々-を使用すると、製品をすばやく作成できます。でも後で教えたほうがいいと思う内側からどのように見えるか。

最初は、個別のテスターを実行したくありませんでした。開発者がTDD方法論に従うか、自分のコードを最後までテストする必要があるという傾向がありました。実際、あまり効果的ではありませんでした。最初は、すべてがうまくいきました：書いた-あなたが答えます。アプリケーションが正しく機能するためには、テストが必要です。しかし、その後、タスクとアプリケーションが増えるほど、それはより困難になりました。一部のアプリケーションは終了し、一部は製品版に移行し、数か月間変更されませんでした。テストの作成や保守などが難しくなりました。チームアナリストが変更されました。比較的最近、彼らは間違っていることに同意しました：テスターが必要です。しかし、開発者はテストの作成を止めず、場合によってはTDDを行います。機能をチェックするテスト（アプリケーションは正しく動作する）と、チェックするテストアプリケーションが問題のある状況で機能すること-これは別の人が行う必要があります。もう1つは、ユニットテストだけでなく、起こりうる奇妙なケースをカバーするテスターが必要です。

現在、60人の純粋な開発者がいます-バック、フロント、フルスタック。アナリスト、テスター、サポートもいます。さらに、さらに7つのdevopsが追加されます。しかし、それでも、タイトルから200人の予定者が採用されていないため、仕事の分野が非常に広いため、現在、新しい人を探しています。もしあれば、私のサークルに空席があります。つまり、開発により、現在では約200のうち74です。

InnerSource


ロシアだけに多くの独立したチームがあり、他の多くの国でも似たようなものを見たチームがまだあることを考えると、私たちはインソースの方向に進んでいます。これは、限られた人々のグループ内のオープンソースに非常に似ています。

すべての国の部門のADEOグループ内で、すべてのコードはクラウドgithubにあります。プロジェクトは、統一されたデザインルールに従って作成されます。スタイル、テクノロジー、ツールに制限はありません。オープンソースコードと明確な設計ルールがある場合、スタック上の開発者は誰でも貢献できます。それをコードにコピーするには、ドックを読み取り、リポジトリのクローンを作成し、編集を行い、プルリクエストを送信する必要があります。

現在、ブラジル、ロシア、フランスはこの共通基盤を非常に積極的に使用しています。

現在、請負業者のコード全体（および多くの異なる方向にある）をInnerSourceに転送しようとしています。分析では、2つの軸でマップを作成しました。1つの軸と2番目の軸の内側の円のモードへの移行の技術的な複雑さ、内側の円への移行が一般的にどれだけ役立つか。コードが一意であり、1か所でのみ必要な場合-触れる必要さえないかもしれません。しかし、多くのチームがこのサイトを使用している場合、それは間違いなく必要です。

これらすべては、一般的に開発文化を強化します。また、必要な機能のマージリクエストを作成できるため、複数のチームの速度が向上します。オーナーチームによって割り当てられ、寄稿者であるチームによってテストされます。重要な条件の1つは、自動テストが利用可能であることと、リポジトリ内のコードの説明です。

アセンブリはJenkinsで行われ、コードはSonar（より正確には、すでにSonarQube）で実行されます。ソナーが失敗しました-リリースなし。テスターは自動テスト、コード所有者-機能テストを作成します。データベースは、インフラストラクチャエンジニアによってサービスとして提供されます。テストベースの構造とメインの構造が異なるため、本格的な負荷テストはまれに行われます。本番環境ではランダムデータがあり、匿名化された実際のデータはありません（これは将来のステップの1つです）。
