Si vous utilisez moki, testez-vous au moins quelque chose?







Avez-vous ressenti que, pour des raisons de test, vous rendiez le code plus difficile à lire? Disons que vous avez du code qui n'a pas encore été testé. Il a un certain nombre d'effets secondaires et vous êtes invité à exécuter les tests en premier. Vous commencez à suivre des conseils comme passer des variables globales en tant que paramètres ou extraire des effets secondaires problématiques pour créer des talons dans les tests .



Mais plus vous allez loin, plus vous vous sentez à l'aise. Vous avez un tas de talons s'accumulant pour les états et les données qui vous permettent d'écrire des tests unitaires rapides et fiables . Par conséquent, vous avez le sentiment d'avoir créé une classe complètement différente qui ne reproduit pas avec précision le vrai code.



Vous vous arrêtez et pensez: «Est-il permis de changer les signatures de code pour des raisons de test? Suis-je en train de tester du vrai code ou une classe complètement différente dans laquelle ce qui est nécessaire ne se produit pas? » Vous pouvez avoir un dilemme. Êtes-vous sûr de devoir continuer à adhérer à cette approche? Ou est-ce une perte de temps?



Millionième question: avez-vous besoin d'écrire des tests unitaires ou des tests d'intégration pour du code obsolète?



Paradoxe



Vous pourriez être dans l'une de ces situations:















Je ne sais pas pour vous, mais ces situations sont tellement proches de moi. Ils sont drôles parce qu'ils sont vrais (et c'est désagréable).



On vous dit que vous devez faire des stubs pour les dépendances problématiques et écrire des tests unitaires. Mais à un moment donné, il y a le sentiment qu'une zone aveugle se développe dans vos tests. Quelles garanties avez-vous que le code se comportera correctement lors de l'utilisation de cette base de données au combat? N'auriez-vous pas encore plus de problèmes si vous vous intégrez à une vraie base de données?



J'ai également fait face à ce dilemme! Pendant longtemps, j'ai fait la promotion de Testing Pyramid . Ils soutiennent que davantage de tests unitaires doivent être écrits car ils sont plus rapides et plus fiables que les tests d'intégration.



J'ai aussi fait beaucoup de front-end. Ces dernières années, le mantra "Écrivez des tests. Pas trop. Surtout l'intégration », promu par Kent Dodds . Il est très intelligent et une autorité dans le test d'applications frontales. Il existe même des outils comme Cypress pour tester la plupart des applications Web avec des tests de bout en bout! Encore plus proche de l'utilisateur!



Comment résolvez-vous ce paradoxe? Comment sera-ce bien? Vaut-il la peine de faire des talons pour que les tests soient rapides et fiables? Ou vaut-il mieux écrire des tests d'intégration moins fiables mais qui posent plus de problèmes?



Avant de continuer, permettez-moi de dire: c'est formidable que vous ayez ce dilemme. Cela vous semble un problème, mais cela signifie que vous allez dans la bonne direction! C'est un obstacle naturel et vous le surmonterez. Cette énigme a une solution.



Un regard pratique sur les tests



J'ai plongé dans ce problème pendant un certain temps et j'ai essayé différentes approches. La décision est intervenue après une discussion mémorable avec JB Rainsberger .



J'ai réalisé qu'il n'y a pas de paradoxe. Il existe différentes définitions d'un «module».



Il me semble que le mot «module» prête à confusion. Les gens ont une compréhension différente de ce que c'est. En règle générale, les nouveaux arrivants à tester considèrent une fonction ou une méthode comme un module. Ils comprennent alors qu'il peut s'agir d'une classe entière ou d'un module. Cependant, la vérité, comme beaucoup plus dans la vie, dépend de la situation.









Tests isolés



Je pense que la division en tests «modulaires» et «d'intégration» n'est pas assez claire. Une telle catégorisation conduit à des problèmes et à des litiges, bien que nous ayons un objectif: faciliter le changement de logiciel à l'aide de commentaires rapides et précis lorsque vous cassez quelque chose.



Par conséquent, je préfère parler de tests "isolés".



Je ne sais pas si ce terme est meilleur. Mais je l'aime pour l'instant, car cela me fait poser une question extrêmement importante: isolée de quoi?



Ma réponse: isolée de ce qui est difficile à tester.



Ce qui est difficile à tester



Accident, heure, réseau, système de fichiers, bases de données, etc.



J'appelle toute cette «infrastructure» et le reste s'appelle un «domaine». La combinaison du domaine et de l'infrastructure ressemble à un aspect très utile et pratique de la conception de logiciels. Et pas seulement pour moi. Cette séparation est au cœur de nombreuses architectures maintenables, telles que l'architecture hexagonale (connue sous le nom de ports et adaptateurs), l'architecture d'oignon, l'architecture propre et le noyau fonctionnel. / coque impérative, etc.



Ce sont toutes les nuances qui disent une chose: il vaut mieux séparer le domaine et l'infrastructure. La programmation fonctionnelle favorise également cette idée: des effets secondaires isolés sur le périmètre de votre système. Les effets secondaires sont les infrastructures. Et c'est difficile à tester.



Oui, mais comment cela aide-t-il?



En bref, cela vous indique exactement ce qu'il faut retirer et remplacer par des bouchons. Ne transférez pas la logique métier à l'infrastructure, sinon vous serez tourmenté par les tests.



Un domaine isolé est facile à tester. Vous n'avez pas besoin de faire de stubs pour d'autres objets de domaine. Vous pouvez utiliser le même code que celui utilisé dans le prod. Il vous suffit de vous débarrasser de l'interaction avec l'infrastructure.



« Mais il peut y avoir des bugs dans le code d'infrastructure! »



Si vous le gardez suffisamment compact, les risques sont réduits. Mais en gros, vous réduisez la taille du code que vous devez tester lors de l'intégration avec une vraie base de données.



L'essentiel est le suivant: vous devriez pouvoir tester toutes sortes de comportements de votre application sans enregistrer les données dans PostgreSQL. Besoin d'une base de données en mémoire. Vous devez toujours vérifier l'intégration avec PostgreSQL, mais quelques tests suffisent , et c'est une différence notable!



C'était une théorie, revenons à la réalité: nous avons du code qui n'a pas encore été testé.



Revenir à un code obsolète



Je pense que cette vision des tests vous aidera. Pas besoin de vous laisser emporter par des talons, mais lorsque vous travaillez avec du code obsolète, vous devrez vous laisser emporter. Temporairement. Parce que ce code est une vinaigrette du fait qu'il est difficile de tester (infrastructure) et la logique métier (domaine).



Vous devez vous efforcer d'extraire le code d'infrastructure du domaine.



Ensuite, rendez l'infrastructure dépendante du domaine, et non l'inverse . Il devrait y avoir aussi peu d’infrastructure que possible dans la logique métier. Vous obtiendrez des tests rapides, fiables et isolés couvrant la majeure partie de la logique d'application, et plusieurs tests d'intégration pour tester le fonctionnement du code d'infrastructure avec une vraie base de données, une API externe, une horloge système, etc.



Avez-vous encore des talons? Oui et non.



Vous aurez des implémentations d'infrastructure alternatives, de sorte que vous pouvez toujours exécuter la logique de production sans utiliser une vraie base de données.



Comment savoir si l'implémentation de la base de données fonctionne correctement? Écrivez des tests d'intégration. Juste assez pour vérifier l'exactitude du comportement. JB Rainsberger l'appelle «Contrats tests». Je pense qu'il vaut mieux expliquer avec cet exemple .

Mais tout cela est important: tests, architecture et code obsolète. Toutes ces compétences sont interconnectées. En pratiquant dans l'un, vous grandissez dans un autre.
