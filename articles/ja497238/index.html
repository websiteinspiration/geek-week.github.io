彼女は偶然ここにいません。しかし後で役立つ。

DBAはperfmonが嫌い


PerfMonインターフェースはWindows NT 3.1（？）で変更されていないため、まったく違いはありません。ところで、たぶん誰かがこの奇妙な事実の説明を知っています。コントロールパネルは年に数回書き換えられます。電卓も書き直されました。しかし、perfMonではありません。

DBAはカバーワードで考えます。優れたDBA医師が病気のSQLサーバーに到着すると、彼はスーツケースを開きます。スーツケースには、忠実に対応するために非常に多くの四半期があります。彼は記憶からそれらの最短のものを取り出します。 DBAは、PerfMonのメトリック（物理ディスク、またはディスクキューの長さ）を調べることがあります。ただし、実際のDBAは、PerfMonではなく、たとえば次のようなパッチを使用して、独自のSQLサーバーメトリックを取得します。

select * from sys.dm_os_performance_counters 
  where object_name='SQLServer:Buffer Manager' 
    and counter_name like 'Page life expectancy%' 
	and instance_name=''

DBA領域から離婚したIT担当者が、「どのような指標を収集してほしいですか？」（まあ、あらゆる種類のsplunk、squared-upなどがあります）。次に、DBAはスクリプトについて考え始め、残りはperfmonメトリックについて考え始めます。同時に、原則として、DBAのすべてのメトリックをダンプしようとします。これが発生すると、画像はその輻輳で役に立たなくなります。そして、映画「飛行機」からの短いビデオクリップが私を助けます。

その結果、DBAはkeriを書き込み、統計情報をN分ごとにDBAtasksデータベースの一部のプレートに書き込みます。ご存知ですか？

悪魔の擁護者（perfmon）


それにもかかわらず、perfmonカウンタは標準ツールです。ログに記録してアラートを設定できますが、最も重要なのは、これらのメトリックを多くのマシンから中央のリポジトリに「マージ」して、それらを分析し、美しい（そしてperfmonではない）チャートを構築できるシステムが多数あることです。

SQLの世界とPerfMonの世界と友達になれることがわかりました！

以下にその方法を示します。

今度は詳細が行きます


例として、SQL自体がPerfMonに公開しないメトリックをエクスポートする方法を示します。たとえば、次のクエリ：

select
  convert(numeric(10,2),round(((sum(version_store_reserved_page_count)*1.0)/128.00),2)),
  convert(numeric(10,2),round(((sum(user_object_reserved_page_count)*1.0)/128.00),2)),
  convert(numeric(10,2),round(((sum(internal_object_reserved_page_count)*1.0)/128.00),2))
from
  tempdb.sys.dm_db_file_space_usage;

バージョンストア（スナップショット用）、ユーザーオブジェクト（#tabおよび##タブ）、およびソート/一時ストレージスペースによって使用されるtempdbスペースの量を返します。それらのメトリックを作成しましょう（18以上、チームは文書化されていません）

dbcc addinstance ('SQLServer:User Settable', 'TempDB version store KB')
dbcc addinstance ('SQLServer:User Settable', 'TempDB user store KB')
dbcc addinstance ('SQLServer:User Settable', 'TempDB sort store KB')

これにより、SQLServer-> UserSettable-> Queryに3つのメトリックが作成されます。メトリックは整数です-int（bigintではありません）。また、その意味はそのまま解釈されます。つまり、前の値からの差分が必要な場合は、これが問題になります。ただし、この場合は、メトリックに値を割り当てるだけです。

declare @TEMPDBver int, @TEMPDBuser int, @TEMPDBsort int
select

@TEMPDBver = convert(numeric(10,2),round(((sum(version_store_reserved_page_count) *1.0)/128.00),2)),
@TEMPDBuser = convert(numeric(10,2),round(((sum(user_object_reserved_page_count) *1.0)/128.00),2)),
@TEMPDBsort = convert(numeric(10,2),round(((sum(internal_object_reserved_page_count) *1.0)/128.00),2))
from tempdb.sys.dm_db_file_space_usage;

if @TEMPDBver is not null    
  dbcc setinstance ('SQLServer:User Settable', 'Query', 
    'TempDB version store KB', @TEMPDBver)
if @TEMPDBuser is not null   
  dbcc setinstance ('SQLServer:User Settable', 'Query', 
   'TempDB user store KB', @TEMPDBuser)
if @TEMPDBsort is not null   
  dbcc setinstance ('SQLServer:User Settable', 'Query', 
    'TempDB sort store KB', @TEMPDBsort)

実際、それだけです。このコードを定期的に呼び出すだけで、すべてが機能します。

デルタカウンター


次に、デルタカウンターを作成します。これはミリ秒単位のロックタイムアウトになります。

dbcc addinstance ('SQLServer:User Settable', 'LOCK ms per s')

ここでは、そのようなメトリックがすでに存在していることに反対することができます。

select * from sys.dm_os_performance_counters 
  where counter_name like 'Lock Wait Time (ms)%' 
    and instance_name='_Total'
    and object_name='SQLServer:Locks'

しかし、この指標は最終的には一貫しています。ご存じのように、陽子が不安定な場合、宇宙のデータベースは最終的に整合性を保ちます。つまり、ロックを作成してこのメ​​トリックの値を見ると、大きくなりません。そして、ロックの最後でのみ、その値は待機時間全体にわたって急激にジャンプします。グラフでは、予想される「プラトー」の代わりに、シャープなピークが得られ、Y軸に沿ってスケール

が損なわれます。kveryでも同じ問題が発生します。

select sum(wait_time_ms) 
  from sys.dm_os_wait_stats 
  where wait_type like 'LCK_%'

したがって、現在のロックの時間を「完了」ロックの時間に追加する必要があります。

declare @NEWlock bigint
select @NEWlock=sum(wait_time_ms) 
  from sys.dm_os_wait_stats 
  where wait_type like 'LCK_%' -- finished waits
select @NEWlock=@NEWlock+isnull(sum(waittime),0) 
  from master.dbo.sysprocesses 
  where blocked>0 and lastwaittype like 'LCK_%' -- waits in progress

このクエリは、すでにチャート上で正しい高原を与えています。

ここで、前の値からのデルタを考慮する必要があります。2つの方法があります。

  • 頻繁に（たとえば、1分に1回）呼び出されるジョブとしてプロセスを設計します。以前の値を格納するラベルを作成します。実際、ジョブ（オーバーヘッド+ジョブの実行履歴の競合）を呼び出すのはよくありません。追加のプレートなしで実行できます。
  • waitforを使用して無限ループで実行されるプロセスを作成します。プロセスは、変数の以前の値を記憶しています。起動時と1時間ごとの2つのスケジュールでジョブとしてプロセスを処理します（ジョブが落ちた場合は再起動します）。

2番目のケースでは、5秒ごと、またはそれ以上の頻度でサイクルを実行することが可能です。したがって、次のようになります。

declare @OLDlock bigint, @NEWlock bigint
declare @lock int, @seconds int = 5 -- must match WAITFOR
loop:
  set @OLDlock=@NEWlock -- shift new to old
  select @NEWlock=sum(wait_time_ms) 
    from sys.dm_os_wait_stats 
	where wait_type like 'LCK_%' -- finished waits
  select @NEWlock=@NEWlock+isnull(sum(waittime),0) 
    from master.dbo.sysprocesses 
	where blocked>0 and lastwaittype like 'LCK_%' -- waits in progress
  set @lock=(@NEWlock-@OLDlock)/@seconds -- this is delta
  if @lock is not null    
    dbcc setinstance ('SQLServer:User Settable', 'Query', 
	  'LOCK ms per s', @lock)
  waitfor delay '00:00:05' 
goto loop

初めてDBCCループを実行するときは、@ OLDlock（および ロック）はnullになります。
