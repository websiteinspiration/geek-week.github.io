Kubernetes Web Viewウェブインターフェースの発表（およびKubernetesの他のウェブUIの概要）

注意 ペレフ。：元の素材の作者はZalandoのHenning Jacobsです。彼はKubernetesを操作するための新しいWebインターフェースを作成しました。これは「kubectl for the web」として位置付けられています。新しいオープンソースプロジェクトが登場した理由と、既存のソリューションが満たさなかった基準については、その記事を読んでください。



この投稿では、さまざまなオープンソースのKubernetesウェブインターフェースを見て、ユニバーサルUIの要件を示し、同時に複数のクラスターのサポートとトラブルシューティングを容易にするために設計されたインターフェースである Kubernetes Web Viewを開発した理由を説明します。

ユースケース


Zalandoでは、多数のKubernetes（900+）およびクラスター（100+）ユーザーにサービスを提供しています。特殊なWebツールの助けが非常に役立つ典型的な使用例がいくつかあります。

  1. サポートしている同僚とのコミュニケーション。
  2. インシデントへの対応とその原因の調査。

サポート


私の経験では、サポートのフレームワーク内での通信は次のよう

になります。-ヘルプ、XYZサービスは利用できません。
-演じるときに何が見えますkubectl describe ingress ...か？

または、CRDに似たもの：

-認証サービスになんらかの問題があります...-

このようなコミュニケーションは通常kubectl、問題を解決するためにチームのさまざまなバリエーションを導入することになります。その結果、会話の両側で端末とWebチャットを常に切り替えざるを得なくなり、さらに異なる状況が観察されます。

したがって、Kubernetesウェブフロントエンドで以下を許可したいと思います。

  • ユーザーはリンクを交換して同じことを観察できます。
  • これは、サポートでの人為的エラーを回避するのに役立ちます。たとえば、コマンドラインで間違ったクラスターを入力したり、CLIコマンドでタイプミスをしたりします。
  • 同僚に送信するための独自のビューを生成できます。つまり、ラベル列を追加し、1つのページにさまざまなタイプのリソースを表示します。
  • 理想的には、このWebベースのツールは、YAMLの特定のセクションへのディープリンクを配置できるようにする必要があります（たとえば、クラッシュを引き起こす無効なパラメーターをポイントします）。

インシデント対応と分析


インフラストラクチャ内のインシデントに対応するには、状況認識、影響を評価し、クラスタ内のパターンを探す機能が必要です。いくつかの実際の例：

  • 重要な本番環境サービスに問題があり、問題を解決するには、すべてのクラスターですべてのKubernetesリソースを名前で検索する必要があります。
  • ノードはスケーリング時に落ち始め、問題の大きさを評価するために、すべてのクラスターでステータスが「保留中」のすべてのポッドを見つける必要があります。
  • 個々のユーザーがすべてのクラスターにデプロイされたDaemonSetの問題を報告しており、問題が全体であるかどうかを確認する必要があります。

そのような場合の私の標準的な解決策はのようなものですfor i in $clusters; do kubectl ...; done。もちろん、同様の機能を提供するツールを開発できます。

既存のKubernetesウェブインターフェース


Kubernetesウェブインタフェースのオープンソースの世界では、私が使用して、追加情報を収集しようとしたので、大きすぎ*ではないツイッターを：



「Kubernetesクラウドサービスと市場がためのものですので、通常、自分のフロントエンドを提供するベンダー：* KubernetesのWebインターフェースの限られた数の私の説明良い「無料のKubernetes UIは比較的小さいです。

私はつぶやいについてK8Dash、Kubernator、及び八分円。それらと他の既存のオープンソースソリューションを見て、それらが何であるかを理解してみましょう。

K8Dash


「K8DashはKubernetesクラスターを管理する最も簡単な方法です。」



K8Dashは見栄えがよく、速度も速いですが、上記の使用例には多くの欠点があります。

  • 1つのクラスターの境界内でのみ機能します。
  • 並べ替えやフィルタリングは可能ですが、パーマリンクはありません。
  • カスタムリソース定義（CRD）はサポートされていません。

Kubernator


「KubernatorはKubernetesの代替UIです。高レベルのKubernetesダッシュボードとは異なり、それは低レベルの制御とクラスター内のすべてのオブジェクトの優れた概要を提供し、新しいオブジェクトを作成、編集して競合を解決する機能を備えています。完全なクライアントアプリケーション（kubectlなど）であるため、Kubernetes APIサーバー自体を除いてバックエンドを必要とせず、クラスターにアクセスするためのルールも考慮に入れます。



これはKubernatorのかなり正確な説明です。悲しいかな、彼はいくつかの機能を欠いています：

  • 1つのクラスターのみを提供します。
  • リスト形式の表示モードはありません（つまり、「保留中」ステータスのすべてのポッドを表示することはできません）。

Kubernetesダッシュボード


「Kubernetesダッシュボードは、Kubernetesクラスターのユニバーサルウェブインターフェースです。ユーザーは、クラスター内で実行されているアプリケーションを管理およびトラブルシューティングしたり、クラスター自体を管理したりできます。」



残念ながら、Kubernetesダッシュボードはインシデントのサポートおよび対応活動にはあまり役立ちません。

  • たとえば、リソースをフィルターしたり、並べ替え順序を変更したりする場合、パーマリンクはありません。
  • ステータスでフィルタリングする簡単な方法はありません。たとえば、「保留中」ステータスのすべてのポッドを表示します。
  • サポートされるクラスターは1つだけです。
  • CRDはサポートされていません（この機能は開発中です）。
  • カスタム列はありません（例：タイプでラベル付けされた列kubectl -L）。

Kubernetesオペレーションビュー（kube-ops-view）


「K8s Cluster Space Observer System Panel」



でKubernetes作用図全く異なるアプローチ：このツールは、オブジェクトの任意のテキストの詳細がなく、WebGLのを使用してクラスタノードとpod'yを示しています。これは、クラスターステータスのオンライン概要（「ポッドが落ちる？」）*には最適ですが、上記のサポートやインシデント対応のケースには適していません。

* 注意 ペレフ。：この意味で、この記事で詳細に説明したgrafana-statusmapプラグインにも興味があるかもしれません。

Kubernetesリソースレポート（kube-resource-report）


「ポッドおよびKubernetesクラスターからのリソースリクエストに関する情報を収集し、それらをリソース消費と比較して、静的HTMLを生成します。」



Kubernetes Resource Reportは、クラスター内のチーム/アプリケーションごとのリソース使用量とコスト配分に関する静的HTML レポートを生成します。このレポートは、アプリケーションがデプロイされているクラスターをすばやく見つけることができるため、インシデントのサポートと対応にいくらか役立ちます。

注意 ペレフ。：クラウドプロバイダーからのリソースの分布とそのコストに関する情報を表示する場合、最近公開したKubecostサービスとツールも役立つ場合があります。

オクタント


「Kubernetesクラスターの複雑さをよりよく理解できるように設計された拡張可能なWeb開発プラットフォーム。」VMwareで作成された



Octantは、私が比較的最近知った新製品です。これを使用すると、ローカルマシン上でクラスターを調べるのに便利です（視覚化さえあります）が、サポートの問題とインシデントへの対応の問題に限定的にしか触れていません。オクタントの欠点：

  • クラスタによる検索はありません。
  • ローカルマシンでのみ機能します（クラスターにデプロイされません）。
  • オブジェクトをソート/フィルタリングできません（ラベルセレクターのみがサポートされています）。
  • カスタム列は指定できません。
  • 名前空間でオブジェクトをリストすることはできません。

また、クラスタ化されたZalandoのOctantの安定性にも問題がありました。

Kubernetes Web Viewの紹介


「Kubectl for the web」



Kubernetesで利用できるインターフェースオプションを分析した後、新しいものを作成することにしました。KubernetesWeb Viewです。実際、実際、私kubectlはWeb上のすべての力、つまり次のことだけが必要です。

  • ユーザーがkubectlの使用を好むすべての（読み取り専用）操作のアクセシビリティ。
  • すべてのURLは永続的であり、ページを元の形式で表示する必要があります。これにより、同僚がそれらを共有して他のツールで使用できるようになります。
  • あらゆるタイプの問題を解決するすべてのKubernetesオブジェクトのサポート。
  • リソースのリストは、今後の作業（スプレッドシート、CLIツールなどgrep）およびストレージ（たとえば、検死）のためにダウンロード可能でなければなりません。
  • ラベルによるリソースの選択のサポート（cと同様kubectl get .. -l）。
  • ( kubectl get all) (, );
  • «» , , , .. / ;
  • ( HTML), , , JavaScript;
  • (, URL);
  • (, / );
  • たとえば、同僚がリソースの説明の特定のセクション（YAMLの行）をポイントできるように、柔軟なリンクを作成してテキスト情報を強調表示する追加の機会。
  • 特定のクライアントの要件に適応する機能。たとえば、CRD用の特別な表示テンプレートを作成したり、独自のテーブルビューを作成したり、CSSスタイルを変更したりできます。
  • コマンドラインをさらに調査するためのツール（たとえば、kubectlコピー可能な本格的なコマンドを表示する）;

Kubernetes Web Viewによって解決された目標以外に、次のようなものがあります。

  • Kubernetesオブジェクトを抽象化します。
  • アプリケーション管理（たとえば、展開管理、Helmチャートなど）;
  • 書き込み操作（安全なCI / CDやGitOpsツールを使用して実行する必要があります）;
  • 美しいインターフェース（JavaScript、テーマなど）;
  • 視覚化（kube-ops-viewを参照）
  • コスト分析（kube-resource-reportを参照）。

Kubernetes Web Viewはインシデントのサポートと対応にどのように役立ちますか？

サポート


  • すべてのリンクは永続的であり、同僚との情報交換を容易にします。
  • 独自のビューを作成できます。たとえば、2つの特定のクラスターに特定のラベルを付けてすべてのDeploymentとポッドを表示できます（複数のクラスター名とリソースタイプをコンマで区切ってリンクで指定できます）。
  • オブジェクトのYAMLファイル内の特定の行を参照して、オブジェクトの仕様に潜在的な問題があることを示すことができます。
