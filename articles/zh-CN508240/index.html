在与功能安全相关的软件开发中使用编程语言C和C ++的功能





克里斯·霍布斯（Chris Hobbs）在其基础著作《安全关键系统的嵌入式软件开发》 [1]中引用了程序员的普遍观点，即对编程语言施加限制就像是命令毕加索制作一幅图片，同时禁止他使用黄色。但是，很难想象有一家企业认真地从事关键系统软件的开发，而无论采用书面或非书面标准，该企业都不会指出要使用哪种编程语言以及如何使用它。



本文致力于IEC 61508标准中定义的方法，该方法包括使用C和C ++编程语言的安全子集来开发关键软件。



IEC 61508分为七个部分，并正式翻译成俄文，具有俄联邦国家标准的地位。对于所讨论的主题，我们主要关注第3部分“软件需求” [2]，其中包含必要的基本原理，以及第7部分“方法和工具” [3]，其中提供了具体建议。此外，我们将考虑标准“ MISRA C：2012” [4]和“ MISRA C ++：2008” [5]，它们对程序员具有最直接的实用价值。



编程作为风险管理工具



IEC 61508针对在开发生命周期的各个阶段使用各种降低风险的工具和技术提供了结构化和系统的建议。编写代码，即编码是阶段之一。我敢说，即使采用最不确定的软件开发流程，从物理上讲也无法排除其他因素。



当然，降低风险的措施有其代价，以实施这些措施所需的工作时间表示。每个员工的工作时间都有一个非常具体的货币表示方式。

在航空电子软件会议上，我听到了有关一家大型国际公司开发的某些管理系统的故事。该产品必须满足66个国家/地区监管机构的要求，因此，仅在2015年，一百多个组织就开发人员和产品进行了审核。在过去的一年中，开发人员花费了3500个工作日来确保进行这些审核。



降低风险的要求越高，执行相关活动的成本就越高。因此，IEC 61508将确保功能安全的系统分为四个等级类别，即俄罗斯安全完整性等级（UPB）。最高级别为UPB4，最低级别为UPB1。英文原件使用术语安全完整性等级（SIL）。



选择编程语言的标准



IEC 61508 [3]第七节的附录A包含有关方法和工具选择的非常有指导性的指南，而附录C则对方法和工具进行了概述，并进行了简要说明并提供了有关方法和工具的信息源。



选择准则非常简单地制定：针对每个级别和每种方式都给出一个名称。这些名称称为“建议”。最严格的建议是HR（强烈建议）和NR（强烈不建议）。



当然，为了开发用于关键系统的软件，有许多专门的编程语言，例如Ada，D，FORTRAN 77或RUST，它们提供严格的数据类型并解决与编码有关的其他重要问题。它们都有自己的利基，从工程的角度来看非常有趣。但是，由于我们亲爱的读者是现实主义者，因此对于本文而言，选择取决于实际上最经常使用C语言编写关键系统程序的语言。而且由于诸如Qt之类的面向对象技术Mesa 3D和Mesa 3D在实时系统的开发人员中越来越受欢迎，您不能忽略使用C ++的问题。



IEC 61508对这些语言怎么说？



对于C，C ++语言的高级别安全完整性-UPB3和UPB4-为了防万一，我注意到Java给出了NR建议。但是，对于C和C ++，如标准中所述，“具有子集和编码标准，以及使用静态分析工具”，给出了HR建议。同时，例如，对于Java，即使对于具有禁用或确定性垃圾回收功能的子集，对于UPB3和UPB4的级别也显示建议NR。



“语言的子集”是做什么用的？答案很简单：消除潜在的危险构造，确保代码的可读性及其对静态分析程序进行处理的适用性。



让我们与您一起弄清楚哪些子集和编码标准可以应用于C和C ++。



语言C



定义C语言子集的最权威文档是称为“ MISRA C”的标准[4]。在此基础上或至少在其影响下，开发了各种组织的安全编码标准，例如JPL的编码标准[6]。



在引言部分，“ MISRA C”给出了C流行的原因。我们将列举一些原因：



  • 用C编写的程序可以编译成有效的机器代码；
  • 有现成的C编译器可用于多种硬件体系结构。
  • C得到大量静态分析和测试工具的支持；
  • 语言C [7]，[8]有一个国际标准；
  • 在关键系统的开发中使用C取得了丰富的经验。


同时，“ MISRA C”的开发人员注意到C语言的缺点，其中不正确的定义使编译器开发人员无法以不同的方式实现某些功能，以及可能引入从编译器的角度来看并非如此的错误-替换赋值符号的经典问题等号，反之亦然。同样，作为缺陷，这是有效代码编译的另一面，表明没有运行时错误控制，这要求程序员明确控制程序中的错误。



“ MISRA C”允许同时使用C99和C90。没有关于C11 / C18标准的建议，仅是因为在最终确定“ MISRA C”文本时，ISO / IEC工作组还没有时间批准新版本的C，并且提及未经批准的标准将是一个坏主意。因此，当然可以很好地使用C11 / C18。



该标准非常重视直接选择工具：编译器和静态分析器。同时（并非偶然）提到了IEC 61508，ISO 26262 [9]和DO-178C [10]标准。如果我们将标准EN 50128 [11]和IEC 62304 [12]添加到此列表中，这些标准也在“ MISRA C”中提及，并且与软件开发生命周期相关，那么通常情况下，这个圈子将会闭合。与软件功能安全有关的所有其他标准均包含所列出文档中所描述的原理和方法的说明或特殊版本。但是，我完全不想说没有必要研究其他文档-当然，诸如从应用某些工具和方法来解决特定类别的问题的角度来看，IEC 60880 [13]非常重要。



“ MISRA C”指令有两种类型：规则和指令。



规则是指此类指示，可以通过分析源文本来检查是否符合要求，而无需任何其他信息源。静态分析器应能够严格检查规则的执行情况。指令包括不能严格按照规则制定的指令。要分析代码是否符合该指令，仅代码是不够的：您可能需要研究设计文档。



静态分析器可以在检查过程中提供帮助，但是必须正确解释其工作结果。



所有方向-规则或指令-根据三个重要程度之一进行分类：强制性，必需和推荐。



如果声明用C编写的代码符合“ MISRA C”标准，则它必须毫无保留或例外地满足该标准的所有强制性指令。



同样，声明为符合“ MISRA C”的C代码应满足该标准的所有“必需”准则。不同之处在于，允许对此程度的指示进行偏差，每个偏差都必须清楚地记录在案。该标准的附录之一给出了偏差记录的示例。对于某些项目，由客户或开发人员决定的“所需”指导可能等同于“强制性”。



最后，如果没有书面说明，则可能违反“推荐”说明。但是，应记住，这些说明是根据丰富的经验制定的，因此在任何情况下都不应忽略它们。该标准建议起草一份文件，解释不遵守建议的原因。不仅如此，对于某些项目，由客户或开发人员决定的“推荐”方向可以等同于“要求”或什至“强制性”。



规则，无论其重要性如何，均可决定或不可决定。可解决的规则包括那些静态分析器可以准确告诉您的代码是否与规则匹配的规则。但是对于某些结构，要得出这样的结论，有必要拥有仅在代码执行时才能知道的数据。在这种情况下，该规则被认为是不可解决的，并且静态分析器对代码是否符合一个或另一个不可解决规则的看法也不可靠。因此，为了确定符合性，需要动态分析仪。



最后，要完全混淆您，最后是分类：根据分析的规模，规则分为模块式和系统式。它们被称为模块化规则，对于它们来说足以检查编译单元，即单独的文件。不难猜测，为了验证是否符合系统规模规则，有必要分析两个或多个编译单元，最常见的是分析项目的整个源文本。

它们范围内的规则分为22组，指令分为4组。有些指令与数据类型有关，与指针一起使用，与控制语句有关，与预处理器指令有关。



可确定的编译单位规模的“必需”规则的一个示例，该规则包含在“标准库”组中：不应使用标准C库的头文件<stdlib.h>中定义的内存分配和释放函数。该规则的目的非常明显：减少诸如内存泄漏，释放未分配的内存，在初始化之前访问分配的内存等常见编程错误。假设您仍然需要使用堆。由于该规则是“必需的”而不是“强制性的”，因此可以通过记录为什么这样做的理由来“违反”。但是在这种情况下，来自系统规模的“资源”组的不可解决的约束规则生效：



  • 必须显式释放使用标准C库动态分配的所有资源；
  • 仅当使用标准C库的功能分配存储区时，才应释放该存储区。


如您所见，与例如malloc（）函数相比，第二条规则在某种程度上缩小了free（）函数的使用范围。



在对“ MISRA C”标准进行简短回顾的总结中，我想强调指出，我认为，对于在开发关键应用程序时使用C语言的任何程序员，都必须进行本文档的研究。此外，无论此标准是否在您的组织中实施。至少，它将使您加深对C的了解并了解该语言的许多细微差别。由于您需要个人或公司许可才能阅读本文档，因此，请注意上面提到的在他的影响下开发的JPL编码规则[6]，以及Gerard J. Holzmann撰写的简短但有效且相当知名的十个规则集[十四]。



C ++语言



当我草拟本文的计划时，没有关于C ++的部分。我的考虑很简单：无需在有关功能安全的文章中提及这种不确定性技术，以鼓励使用这种不确定性技术。但是，在编写材料的过程中，我仍然认为我的观点是我的个人观点，将如此广泛的语言放在括号之外是错误的。没有有关C ++局限性的信息，本文显然是不完整的。



附录G中的标准GOST R IEC 61508-7-2012 [3]简要定义了（面向UPB3和UPB4的HR建议）与编程语言无关的编写面向对象代码的基本原理，例如：



  • 继承仅允许阐明基类；
  • 继承的深度应该受到限制；
  • ;
  • .




该标准强烈建议（HR）使用经过验证的设计模式（模板）和软件框架（框架）。实际上，线框和自动生成的代码可能是在关键系统中使用C ++的主要方法。自1990年代后期以来，随着使用UML模型生成代码（方便使用基于状态机的算法）的框架的日益普及。然后，当嵌入式系统的强大功能使其可以使用丰富的UX，高级通信和其他技术时，Qt框架也开始普及。不同设计阶段的模式专用于例如Bruce Paul Douglass的“实时设计模式” [15]。框架的示例包括IBM Rational的OXF和Willert Software Tools的RXF。



可能的是，汽车工业软件可靠性协会并没有脱离过程，而是通过发布MISRA C ++：2008标准[5]为C ++安全编码方法的发展做出了贡献。尽管几年前采用了C ++ 11标准[16]，但MISRA C ++仍基于C ++ 03 [17]，因为在批准之时它是当前版本。 MISRA C ++明确声明该文档的开发人员无意推广C ++语言。他们的目标是为关键软件开发中各种SIL所需的C ++子集推荐的IEC 61508定义。



再一次，当将框架与从一种或另一种类型的模型自动生成代码结合使用时，C ++特别重要。面向模型的开发的主要目标是提高开发复杂应用程序的抽象水平（从而提高生产率），并通过自动执行代码生成和分析来减少错误。



结论



最后，关于使用C和C ++编写的代码的静态分析，需要说几句话。事实上，MISRA相关标准的任务之一是提供适用于自动模式下静态分析的规则。从MISRA标准的角度来看，选择合适的分析仪实际上与选择编译器是不可分离的。



自由分配和商业静态分析仪都有大量。此外，某些编译器具有内置功能来分析限制性规则的实现。



在非商业产品中，我会选择流行的cppcheck实用程序，您可以在公共领域找到很多有用的信息，例如，有关Habrahabr资源的文章。至于对符合MISRA C和MISRA C ++的代码的分析，可以在相应的Wikipedia文章中找到可能的工具列表。
