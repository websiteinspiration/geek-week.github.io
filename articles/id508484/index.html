API Script Kompiler Script: Kendalikan kompiler





Pengembangan aplikasi typescript selalu memiliki fase pembangunan proyek. Biasanya, sistem build dan otomatisasi alur kerja digunakan untuk ini, seperti webpack atau tegukan, digantung dengan plug-in yang cukup, atau proses build tersebar dalam package.json perintah dan skrip shell menggunakan tsc asli atau perintah CLI yang digunakan dalam kerangka proyek. Semua solusi ini memiliki pro dan kontra. Seringkali selama proses perakitan, sesuatu yang non-standar perlu dilakukan, dan ternyata sistem perakitan yang digunakan tidak menyediakan fungsionalitas yang diperlukan dari kotak, dan plug-in yang ada tidak cukup dengan apa yang dibutuhkan. Pada saat-saat seperti itu, pekerjaan pada proyek bangkit, dan pengambilan panik dalam konfigurasi dimulai dan pencarian plug-in yang sesuai. Pada titik tertentu Anda menyadari bahwa dalam waktu yang dihabiskan mencari kruk yang cocok,Anda bisa menulis keputusan Anda.



Dalam banyak kasus, proses kritis dalam suatu proyek dapat diotomatisasi dengan skrip javascript, ekspresifitas dan fungsionalitas yang memungkinkan untuk menggambarkan alur kerja yang diinginkan dan memilih dari berbagai pustaka tanpa mengganggu kehadiran plug-in untuk mereka untuk sistem build tertentu. Keuntungan penting dari pendekatan ini adalah kontrol penuh atas proses dan fleksibilitas maksimum. Untuk proyek-proyek yang menggunakan TypeScript sebagai bahasa pengembangan utama, muncul pertanyaan tentang bagaimana menanamkan proses mengkompilasinya ke dalam alur kerja Anda. API Compiler ScriptScript datang untuk menyelamatkan di sini. Pada artikel ini, kita akan melihat bagaimana itu dapat digunakan untuk mengkompilasi proyek yang diimplementasikan pada Script, berinteraksi dengan kompiler pada berbagai tahapan pekerjaannya, dan menulis skrip untuk hot-reload server REST,dikembangkan oleh Nest.js.

Awalnya, saya mencoba mengimplementasikan solusi menggunakan webpack, seperti yang disarankan dalam dokumentasi Nest.js , tetapi ternyata start-server-webpack-plugin, yang digunakan di sana untuk mengimplementasikan restart, menggunakan strategi yang sama dengan CLI, yaitu. menjalankannya dalam proses terpisah dan memulai kembali sesuai kebutuhan. Kompilasi sederhana



Untuk memahami prinsip-prinsip API Compiler TypeScript, sebagai permulaan, mari kita coba mengkompilasi proyek dengannya. Proyek di mana saya akan bereksperimen memiliki struktur standar untuk nest.js: sumbernya terletak di folder src, titik masuknya adalah file ./src/main.ts.



Agar kompiler berhasil mengkompilasi proyek, ia harus melewati parameter kompilasi. Cara termudah adalah menguraikannya dengan benar dalam skrip. Buat file tsconfig.js: 



const {ModuleResolutionKind, ModuleKind, ScriptTarget} = require("typescript")

module.exports =  {
    moduleResolution: ModuleResolutionKind.NodeJs,
    module: ModuleKind.CommonJS,
    target: ScriptTarget.ES2019,
    declaration: true,
    removeComments: true,
    emitDecoratorMetadata: true,
    experimentalDecorators: true,
    sourceMap: true,
    outDir: "./dist",
    baseUrl: "./",
    allowJs: true,
    skipLibCheck: true,
  }


Harap perhatikan bahwa beberapa parameter, seperti moduleResolution, module, target, didefinisikan melalui enumerasi, jadi itu tidak akan bekerja secara langsung untuk mengambil parameter dari tsconfig.json, di mana mereka ditulis sebagai string, jadi saya tidak akan menggunakannya untuk mengimplementasikan contoh paling sederhana. Cara mengambil parameter dengan benar dari tsconfig.json akan kita pahami nanti.



Sekarang kita memiliki opsi kompilasi dan Anda dapat pergi ke langkah berikutnya. Untuk melakukan ini, buat file ts-compiler.js dengan konten berikut: 



const ts = require('typescript');

function compile() {
  const compilerOptions = require('./tsconfig');
  const program = ts.createProgram(['./src/main.ts'], compilerOptions);
  const emitResult = program.emit();
}

compile();


Jika sekarang Anda menjalankan skrip ini untuk dieksekusi, file program yang dikompilasi akan muncul di direktori ./dist. Kompilasi terjadi dalam dua tahap: pertama, perintah createProgram menganalisis file sumber dan membuat daftar file untuk kompilasi, setelah itu kami mengkompilasi dan menyimpan file yang dikompilasi menggunakan fungsi emit. Cukup mudah.



Untuk meningkatkan kegunaan, sekarang kita perlu mengimplementasikan pemuatan opsi kompilasi dari file tsconfig.json. Tambahkan fungsi berikut ke skrip: 




const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function getTSConfig() {
  const configPath = ts.findConfigFile('./', ts.sys.fileExists, 'tsconfig.json');
  const readConfigFileResult = ts.readConfigFile(configPath, ts.sys.readFile);
  if (readConfigFileResult.error) {
    throw new Error(ts.formatDiagnostic(readConfigFileResult.error, formatHost));
  }
  const jsonConfig = readConfigFileResult.config;
  const convertResult = ts.convertCompilerOptionsFromJson(jsonConfig.compilerOptions, './');
  if (convertResult.errors && convertResult.errors.length > 0) {
    throw new Error(ts.formatDiagnostics(convertResult.errors, formatHost));
  }
  const compilerOptions = convertResult.options;
  return compilerOptions;
}


Untuk bekerja dengan file konfigurasi, API menyediakan sejumlah fungsi yang memungkinkan Anda menemukan, memuat, dan mengonversi konfigurasi ke bentuk yang dipahami oleh kompiler. Di sini, fungsi findConfigFile digunakan untuk menentukan path ke file konfigurasi, kemudian dimuat sebagai json menggunakan readConfigFile, dan kemudian, jika tidak ada kesalahan, kami mendapatkan parameter kompilasi menggunakan convertCompilerOptionsFromJson.



Jika terjadi kesalahan saat memuat konfigurasi, fungsi kami melempar pengecualian menggunakan mekanisme transmisi pesan kesalahan yang tertanam dalam API. Perpustakaan mengembalikan semua pesan kesalahan dan peringatan sebagai objek dari kelas Diagnostik. Untuk membentuk pesan berdasarkan output untuk konsol, Anda dapat menggunakan fungsi pemformatan pustaka standar, dalam hal ini formatDiagnostik untuk satu pesan dan formatDiagnostik untuk larik. Agar fungsi pemformatan berfungsi dengan benar, objek formatHost diperlukan, yang berisi parameter penting untuk memformat pesan.

function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
  };
  return displayFunction;
}

Pengamat program inkremental DIY



Di bagian sebelumnya, kami melihat bagaimana API Kompiler TypeScript dapat digunakan untuk menyesuaikan proses kompilasi ScriptScript. Sekarang saatnya untuk melakukan sesuatu yang lebih mendesak dan berguna, yang benar-benar dapat meningkatkan kecepatan dan kemudahan pengembangan. Selama kompilasi program, kompiler harus menyelesaikan semua dependensi dalam program.



Untuk melakukan ini, ia secara berurutan membaca dan menganalisis file proyek dan file perpustakaan. Bahkan untuk mengkompilasi proyek yang relatif kecil seperti proyek saya pada tahap persiapan, kompiler perlu membaca, mengurai dan menganalisis lebih dari 2000 file. Ini menyebabkan perlambatan yang signifikan dalam proses kompilasi, misalnya, pada kompilasi laptop saya membutuhkan waktu sekitar 20 detik. Pada saat yang sama, selama pengembangan, biasanya hanya beberapa file yang berubah di antara peluncuran kompiler, jadi masuk akal bahwa mulai dari versi 2.7, Scripteks memiliki opsi kompilasi tambahan, yang, bersama-sama dengan mode tontonan, yang memungkinkan Anda untuk memantau perubahan pada file pada disk, secara signifikan meningkatkan kecepatan kompilasi.



Fungsi-fungsi ini juga digunakan oleh CLI Nest.js, dan Anda dapat mulai bekerja dalam mode ini menggunakan perintah 



nest start --watch


Peluang lain untuk memanfaatkan mode pengembangan ini adalah menjalankan tsc dengan sakelar --watch. Tapi, seperti yang sudah saya tulis di pendahuluan, jika kemampuan perintah standar pada satu titik tidak lagi mencukupi, sekarang saatnya untuk melihat di balik tudung Typecript dan mencari tahu cara kerjanya di sana.



Untuk mulai dengan, seperti pada kasus sebelumnya, buat ts-wather.js dengan konten berikut: 



const ts = require('typescript');

const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

async function watchMain() {
  const configPath = ts.findConfigFile(
    './',
    ts.sys.fileExists,
    'tsconfig.json',
  );

  const host = ts.createWatchCompilerHost(
    configPath,
    {},
    ts.sys,
    ts.createEmitAndSemanticDiagnosticsBuilderProgram,
    diagnostic =>
      console.log(
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
    diagnostic =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );
  ts.createWatchProgram(host);
}
watchMain();

Agar nyaman bagi kita untuk mengaitkan hal-hal yang berbeda di host, mari kita tulis pembantu kecil: 



function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function() {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}


Ini akan memungkinkan kita untuk melakukan tindakan yang diperlukan sebelum dan sesudah fungsi yang dipanggil di host, sambil menerima argumen yang diteruskan ke sana dan hasil dari eksekusi. Sekarang helper ini dapat digunakan untuk mengganggu kompiler pada berbagai tahap. Misalnya, mari kita tambahkan pesan ke konsol dalam persiapan untuk kompilasi:

Mari kita cari tahu apa yang terjadi di sini. Pertama, kita memerlukan variabel aplikasi, di mana kita menulis hasil dari fungsi bootstrap, untuk menutup server dengan benar setelah mengubah kode dan memulai kompilasi ulang. Dan ketiga - karena sistem modular menyimpan modul yang dimuat, setelah membuat perubahan pada kode, kita perlu membersihkan cache dari dependensi. Untuk melakukan ini, saya menulis sebuah fungsi pembantu kecil clearCache, yang sebelumnya kami panggil. 



function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      );
    });
}


Di sini cache dihapus dari dependensi yang dapat mencegah kita me-restart server dengan benar. Dalam kasus kami, kami harus menghapus semua modul proyek kami dari direktori dist dari cache. Saya juga menghapus modul typeorm, karena fitur implementasinya mencegah server dari restart (mengapa ini terjadi adalah cerita lain). process.exit = (code) => {
  console.log('!!!!!!!!!!!!!!!!!!!!!!! WARNING!!!!!!!!!!!!!!!!!!!!!!!!!');
  console.trace(`Process try to exit with code ${code}.`);
};


Anda seharusnya tidak melakukan ini secara umum, tetapi Nest.js memiliki kebiasaan buruk menyebutnya jika terjadi kesalahan selama proses peluncuran, jadi kami hanya mengganti fungsi keluar dengan pesan kesalahan. Dalam hal ini, skrip kami akan terus berfungsi setelah memanggil process.exit (). Ini harus dilakukan sebelum memanggil fungsi watchMain.



Jadi, sekarang jalankan skrip kami dan lihat apa yang terjadi:







Di sini saya membuat kesalahan dalam kode server, setelah itu server tidak memulai dan pesan kesalahan ditampilkan, setelah kesalahan diperbaiki, server mulai lagi berhasil. Di bawah ini adalah skrip yang dihasilkan:



ts-compiler.js 
const ts = require('typescript');
const { join, dirname } = require('path');
const { exit } = require('process');

const formatHost = {
  getCanonicalFileName: (path) => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function () {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}

function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      process.stdout.clearLine(); // clear current text
      process.stdout.cursorTo(0); // move cursor to beginning of line
      process.stdout.write(
        `Clearing cache ${Math.floor((index * 100) / arr.length + 1)}%`,
      );
    });
  process.stdout.write(' finished.\n');
}

function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  let counter = 0;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    counter++;
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    counter = 0;
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
    console.log(`${counter} times function was called`);
  };
  return displayFunction;
}

async function watchMain() {
  const configPath = ts.findConfigFile(
    './',
    ts.sys.fileExists,
    'tsconfig.json',
  );

  const host = ts.createWatchCompilerHost(
    configPath,
    {},
    ts.sys,
    ts.createEmitAndSemanticDiagnosticsBuilderProgram,
    (diagnostic) =>
      console.log(
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
    (diagnostic) =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );

  host.readFile = displayFilename(host.readFile, 'Reading');
  let app;
  on(
    host,
    'createProgram',
    () => {
      console.log("** We're about to create the program! **");
      app && app.close().then(() => (app = undefined));
      host.readFile.enableDisplay();
    },
    () => {
      host.readFile.disableDisplay();
      console.log('** Program created **');
    },
  );
  let currentProgram;
  on(
    host,
    'afterProgramCreate',
    (program) => {
      console.log('** We finished making the program! Emitting... **');
      currentProgram = program;
    },
    () => {
      console.log('** Emit complete! **');
      const onAppClosed = () => {
        if (app) {
          setTimeout(onAppClosed, 100);
        } else {
          clearCache();
          require('./dist/bootstrap')
            .bootstrap()
            .then((res) => {
              app = res;
            });
        }
      };
      if (currentProgram && currentProgram.getSemanticDiagnostics().length === 0) {
        onAppClosed();
      }
    },
  );

  ts.createWatchProgram(host);
}
process.exit = (code) => {
  console.log('!!!!!!!!!!!!!!!!!!!!!!! WARNING!!!!!!!!!!!!!!!!!!!!!!!!!');
  console.trace(`Process try to exit with code ${code}.`);
};
watchMain();
