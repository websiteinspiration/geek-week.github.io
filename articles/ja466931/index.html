1. ToArrayとToList


public IEnumerable<string> GetItems()
{
    return _storage.Items.Where(...).ToList();
}

同意する、産業プロジェクトの非常に典型的なコード。しかし、彼の何が問題になっていますか？ IEnumerableインターフェイスは、「移動」できるコレクションを返します。このインターフェイスは、要素を追加または削除できることを意味するものではありません。したがって、リスト（ToList）にキャストしてLINQ式を終了する必要はありません。この場合、配列（ToArray）にキャストすることをお勧めします。 ListはArrayのラッパーであるため、このラッパーが提供するすべての追加機能をインターフェイスで切り離しています。配列は消費するメモリが少なく、その値へのアクセスが高速です。したがって、なぜもっと支払うのか。上記の例では、IEnumerableインターフェースは例示のみを目的として紹介されています。コード内でLINQ式のToListを呼び出す場合は、リストの機能が本当に必要であることを確認してください。一方では、「一致時の最適化」と言われるように、この最適化は重要ではありませんが、これは完全に真実ではありません。実際のところ、サービスがプレゼンテーション層のモデルを返す一般的なアプリケーションでは、そのようなToList呼び出しが無数に存在する可能性があります。

異なるストリームからコレクションにアクセスした場合、ArrayとListは同等に機能せず、変更される可能性があるというコメントを予想しています。本当にそうです。しかし、開発者がこのようなシナリオを検討する場合、高い確率で、ArrayもListも適していません。

APIを開発するときは、入力としてファイルパスを取得するメソッドシグネチャを避けてください（後でメソッドで処理するため）。代わりに、バイト配列を入力に渡す機能を提供するか、最後の手段としてストリーム実際のところ、この方法は、ディスクからのファイルだけでなく、ネットワーク経由で転送されたファイル、アーカイブからのファイル、データベースからのファイル、メモリ内で動的に生成されるコンテンツなどのファイルにも適用できます。 e。「ファイルへのパス」という入力パラメーターを持つメソッドを提供することにより、APIのユーザーに、データを再度読み取る前にデータをディスクに保存するように義務付けます。この無意味な操作は、パフォーマンスに重大な影響を与えます。ドライブは非常に遅いものです。便宜上、メソッドに入力パラメータ「ファイルへのパス」を提供できますが、内部では常に、入力でバイトの配列またはストリームを使用してパブリックオーバーロードメソッドを使用します。追加のディスク書き込み/読み取り操作を見つけるのに役立つ「マーカー」があります。プロジェクトで標準の方法を使用してみてください。Path.GetTempPath()およびPath.GetRandomFileName()（System.IOから）。高い確率で、上記の問題または同様の問題の回避策が発生します。

注意深い経験豊富な読者であれば、たとえば非常に大きなファイルを処理している場合、ディスクへの書き込みが逆にパフォーマンスを向上させる場合があることに気付くでしょう。これは真実であり、これを考慮に入れる必要がありますが、これは特定の実装ではまれな状況であると思います。

5. ArrayとListクラスについてもう少し


ロジックに従って、Listは配列のラッパーであるため、配列の反復は常にListの反復よりも効率的です。また、ロジックに従うと、「for」はIEnumerableインターフェイスの実装に必要な多くのアクションを実行するため、「for」は常に「foreach」よりも高速です。ここではすべてが論理的ですが、間違っています！ベンチマークの結果を見てみましょう：



ベンチマークコード
public class IterationBenchmark 
{
	private List<int> _list;
	private int[] _array;

	[Params(100000, 10000000)] public int N;

	[GlobalSetup]
	public void Setup()
	{
		const int MIN = 1;
		const int MAX = 10;
		Random rnd = new Random();
		_list = Enumerable.Repeat(0, N).Select(i => rnd.Next(MIN, MAX)).ToList();
		_array = _list.ToArray();
	}

	[Benchmark]
	public int ForList()
	{
		int total = 0;
		for (int i = 0; i < _list.Count; i++)
		{
			total += _list[i];
		}

		return total;
	}

	[Benchmark]
	public int ForeachList()
	{
		int total = 0;
		foreach (int i in _list)
		{
			total += i;
		}

		return total;
	}

	[Benchmark]
	public int ForeachArray()
	{
		int total = 0;
		foreach (int i in _array)
		{
			total += i;
		}

		return total;
	}

	[Benchmark]
	public int ForArray()
	{
		int total = 0;
		for (int i = 0; i < _array.Length; i++)
		{
			total += _array[i];
		}

		return total;
	}
}


実際、配列を反復する場合、「foreach」はIEnumerable実装を使用しません。この特定のケースでは、配列の境界外をチェックせずに、インデックスによる最も最適化された反復が実行されます。これは、「foreach」構造がインデックスで動作しないため、開発者がコードを「混乱させる」機会がないためです。これはルールの例外です。したがって、コードの重要なセクションで、最適化のために「foreach」の使用を「for」に置き換えた場合は、足を踏みにじることになります。これは配列にのみ関連することに注意してください。StackOverflowには、この機能について説明するブランチがいくつかあります。

7.メソッドの埋め込み


コードは2つの理由で最も頻繁にメソッドに分割されます。1つのタスクが複数のサブタスクに分割されている場合は、コードの再利用と分解を確認してください。人にとっては簡単です。インライン化は、分解の逆のプロセスです。メソッドコードは、メソッドを呼び出す必要がある場所に埋め込まれているため、コールスタックと受け渡しパラメーターを保存します。すべてを1つの方法に「詰め込む」ことはお勧めしません。しかし、理論的に「インライン化」できるメソッドは、対応する属性でマークできます。

[MethodImpl(MethodImplOptions.AggressiveInlining)]

この属性は、このメソッドを埋め込むことができることをシステムに通知します。これは、この属性でマークされたメソッドが必ずしも組み込みであるという意味ではありません。たとえば、再帰的メソッドや仮想メソッドを埋め込むことはできません。また、埋め込みメカニズムが非常に「デリケート」であることも注目に値します。システムがメソッドの埋め込みを拒否する理由は他にもたくさんあります。ただし、.NET Coreに取り組んでいるMicrosoftチームはこの属性を積極的に使用しています。.NET Coreソースコードには多くの例があります。

8.推定容量


私（およびほとんどの開発者も）は反射を開発しました：コレクションを初期化しました-そのために容量を設定できるかどうかを考えました。ただし、コレクション要素の正確な数は常に事前にわかっているとは限りません。しかし、これがこのパラメーターを無視する理由ではありません。たとえば、コレクションに含まれるアイテムの数について話している場合、ぼんやりとした「数千」が想定されます。これは、容量を1000に設定する機会です。たとえば、リストのデフォルトでは、容量= 16なので、 1000に達すると、システムは1008（16 + 32 + 64 + 128 + 256 + 512）の要素の追加コピーを作成し、次のGC呼び出しに処理される7つの一時配列を作成します。それら。この作業はすべて無駄になります。また、式を容量として使用することを禁止する人はいません。コレクションのサイズが他のコレクションの3分の1であると推定される場合、CapacityをotherCollection.Count / 3に設定できます。Capacityを設定するときは、コレクションの可能なサイズの範囲と、その値がどの程度密に分布しているかをよく理解することは価値があります。害が及ぶ可能性は常にありますが、正しく使用した場合、推定容量はあなたに良い勝利をもたらします。
