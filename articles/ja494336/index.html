フロントエンドツールキット：開発をスピードアップするための便利なユーティリティと機能



フロントテンダーがメモ帳を開いて数行のコードを記述し、ブラウザーで確認してFTP経由でサーバーにアップロードするだけで十分だった時代は終わりました。現代のユーザーインターフェイス設計は、はるかに複雑になっています。 JavaScriptエコシステムは急速に成長および変化しているため、混乱しやすくなっています。この投稿では、Parallelsフロントエンドチームがパフォーマンスを最適化するために何を使用するかを説明します。

現在、世界のどこかで、2つのフロントエンドベンダーが、どのフレームワークがプロジェクトに適しているかについて真剣に議論していると思います。そして、3番目のものはそれらに接続されています-フレームの熱烈な反対者。これらのプログレッシブなプリブラッドはすべて、不必要な依存関係を持つコードを膨張させるだけであり、一般にプログラムを実行できない人のために設計されているためです。彼らの議論は何も終わらないだろう。 1時間後、全員が静かに仕事に分散します。誰もが自分の意見に留まり、以前と同じように働きます。

私の意見では、そのような論争は、以前に現れたもの、ニワトリや卵についての論争よりも深刻です。問題のないフロントエンド開発のための唯一の真の効果的なソリューションが存在しないためです。 Angular、React、Vue.jsなどのフレームワークを使用しなくても、すべての基本機能を実行できることに同意します。しかし、目標が協力であり、1つだけでなくチームとして大規模なアプリケーションを作成する場合、それらの方が簡単です。

もちろん、それらを使用せずに行うこともできますが、標準の作成に時間を無駄にせず、構造を設定し（常にどこにあるかを知っています）、ルーチンを容易にし、開発をスピードアップできます。言語がアルファベットであると考える場合、フレームワークは、コミュニケーションの構築を容易にする陳腐なダイアログを備えたフレーズブックと見なすことができます。

この投稿では、チームとユーザーインターフェイスを作成するときに使用する便利なツールを紹介します。共同開発とプロジェクトのサポートを大幅に簡素化します。

***はじめ
に、自分自身について少しお話ししましょう。 9年前の2011年にITの世界に入りました。彼は1つの小さな組織でフルスタックとしてスタートしました。その後、彼はハイブリッドモバイルアプリケーションに従事し、WebViewにあるロジックの一部を担当しました。そして数年後、彼はParallelsになりました。ここでは、会社のすべての製品のWebソリューションのプロバイダーであるクラウドチームで働いています。ビジネス機能の大部分は、フロントエンドタスクを熟考して実装することを私に要求します。現在、マイアカウントポータルの開発に焦点を当てています。

上記のテキストからおそらくすでに理解しているように、このWebアプリケーションのインターフェースの作成には、そのフレームワークが欠かせませんでした。Vue.jsを選択しました。彼との共同作業を快適にするために、追加のツールも導入しました。これが、開発プロセスを可能な限り最適化するのに役立ちました。

Vue CLI


このコマンドラインユーティリティには、多くの便利な機能が含まれており、プロジェクトをすばやく開発できるように設計されています。大まかに言えば、起動用の標準フレームワークを作成し、ツールやアセンブリの構成について初期段階で考えることなく、アプリケーションの作成に集中できるようにします。後でプロジェクトのニーズに合わせてカスタマイズできます。

Vue CLIは、コアWeb開発テクノロジーのサポートを提供します。すぐに使えるのは、Webpack、Babel、TypeScript、ESLint、Sassなどのツールです。さらに、MochaとNightwatchを使用した単体テストとエンドツーエンドテストの組み込みサポートがあります。

Vue devtools


アプリケーションをリアルタイムでデバッグできるブラウザの拡張機能。コンポーネントのプロパティとメソッド、すべてのイベントのリストにアクセスできます。Webページを介してアプリケーションの状態を完全に制御でき、パーティションが結果を確認するのを待たずに済みます。

ChromeとFirefoxで動作します。他のブラウザーの公式の拡張機能はありませんが、エレクトロンアプリを介してVue Devtoolsを実行できます。その後、ツールはどの環境でも使用できます。

ゼプリン


このサービスは、レイアウトの開発への転送を便利にします。他の大企業と同様に、UIコンポーネントのセットがあります。以前はPSDファイルに含まれていたため、アクセスが非常に無秩序でした。そして、Zeplinには、グローバルスタイルガイドと呼ばれるものがあります。

その中で、エンタープライズインターフェースを構築するためのすべてのルールを収集できます。各UIコンポーネントについて、CSSコードが自動的に生成されます-要素を構成する各ブロックの色、サイズ、インデント、およびその他のプロパティが示されます。つまり、スタイルを毎回最初から実装する必要がなくなり、作業が大幅にスピードアップします。PSD時代は終わりました。

さらに、すべての変更の履歴があります。最初の段階の状況と最後の状況を簡単に追跡できます。

依存関係としてのGit URL


これは、使用しているnpm機能です。社内の同僚がアクセスして必要に応じて使用できるように、UIコンポーネントのセットを共有する必要がありました。この問題の最も明白な解決策は、npmパッケージです。しかし、それをパブリックレジストリに入れたくなかったので、ローカルのnpmサーバーは良い考えではなかったので、git + npmバンドルを使用しました。

したがって、Parallelsの各従業員はUIコードベースにアクセスできます。これはgitなので、バージョン管理の問題も解決されます。アップデートによる問題はなく、壊れることはありません。

衛兵


クライアント側で発生したエラーとログを、苦情を待たずに収集するために、Sentryを接続しました。このツールは、カスタムブラウザーでのコード実行を追跡します。

バグが発生した場合、インシデントの本質とそれに先行するアクションのリストの両方を反映した完全なレポートとともにリクエストが自動的に送信されます。このデータから、理由を見つけてすべてを修正することは非常に簡単です。

また、テクニカルサポートの負荷を少し軽減し、同僚を不必要な電話や電話から解放しました。

中毒の危険性について


最後に、少しアドバイス。ユーティリティを自分で作成してみてください。最近、既製のソリューションを備えたライブラリが非常に人気を博しています。フロントエンドの入札者は、目を閉じて記述できる「単一行」の関数に対しても依存関係を作るようになります。これは何を脅かしていますか？失敗。

有益な話は4年前に起こりました。 1人がnpmレジストリから自分の11行のパッケージを削除し、約1000のプロジェクトを破りました。 「レフトパッド」と呼ばれる小さなモジュールがないため、世界中の開発者がアプリケーションをデプロイしようとすると、エラーメッセージを受け取り始めました。

ですから、怠惰にしないでください。コードを一連の依存関係に置き換えないでください。自分に依存し、小さなユーティリティを自分で作成し、既成のライブラリソリューションを複雑な関数にのみ使用してください。
