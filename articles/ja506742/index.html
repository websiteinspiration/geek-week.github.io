ピクセルの中心を（0.5; 0.5）にする必要がある理由

現在、カメラの「目」から実行されるレイトレーシングがますます一般的になっているとき、このレッスンを再学習する必要があります。ピクセルの中心が座標（0.5; 0.5）にあると、コードが改善され、寿命が短くなります。あなたがすべてを正しく行っていると確信しているなら、同じ流れで続けてください;あなたにとって、記事は新しいものではありません。これはよく読まれる。

ピクセルの中心を（0.5; 0.5）に配置することの感覚は、ポールハックバートのかわいい短編記事「ピクセル座標とは」によって最初に説明されました（少なくとも私にとって）。 1990年の本、グラフィックジェムズ、pp。246-248から。

今日、この記事を見つけるのは難しいので、その本質を簡単に説明します。幅と高さが1000の画面があるとします。X軸のみを考えてみます。行の左端のピクセルの中心に0.0、次のピクセルの中心に1.0などを割り当てたくなるかもしれません。 73.6および74.4の浮動小数点座標が74.0の中心に転送される丸めを使用することもできます。

ただし、検討する価値があります。この比較により、左端は座標-0.5になり、右端は999.5になります。このようなシステムで作業するのは不便です。さらに悪いことに、ピクセルの座標の値にabs()またはのような異なる演算子が適用されている場合mod()、そのような比較はエッジで重要でないエラーを引き起こす可能性があります。

0.0から1000.0までの間隔で作業する方が簡単です。この場合、各ピクセルの中心には0.5の小数部があります。たとえば、整数ピクセル43は、43.0から43.99999までのサブピクセルの値の美しい範囲の値を持ちます。以下は、Paulの記事からの抜粋です。


OpenGL、ピクセルの中心は常に小数部分（0.5 0.5）を有していました。当初、DirectXはこれに準拠していませんでしたが、DirectX 10では、それが気になりました。

整数座標から浮動小数点ピクセルの座標に変換する操作は、0.5を追加することです。 floatをintegerに変換するだけで十分floor()です。

しかし、これは長い話です。結局のところ、皆が今日それをやっていますよね？私の例では、レイトレーシング用のパースペクティブカメラの方向を生成するための（疑似）コードに出くわしたので、このトピックに戻りました。

 float3 ray_origin = camera->eye;
 float2 d = 2.0 * 
     ( float2(idx.x, idx.y) / 
       float2(width, height) ) - 1.0;
 float3 ray_direction =
     d.x*camera->U + d.y*camera->V + camera->W;

idx ベクトルはピクセルの整数位置であり、幅と高さは画面解像度です。ベクトルdは計算され、UとVの 2つのベクトルの乗算を使用してワールド空間にベクトルを生成するために使用されます。次に、ベクトルWが追加されます-ワールド空間でのカメラの方向。UとVは、目からの距離Wにおける表示面のX軸とY軸の正の方向を示します。上記のコードでは、すべてが美しく対称的です。ほとんどの場合です。


ベクトルdは、画面上のポイントの正規化されたデバイス座標（Normalized Device Coordinates、NDC）で-1.0〜1.0の値のペアを示す必要があります。ただし、コードはここで失敗します。例を続けましょう。整数ピクセル位置（0; 0）は（-1.0; -1.0）に転送されます。いいですね。ただし、最大の整数ピクセル位置は（999; 999）であり、これは（0.998; 0.998）に変換されます。合計の0.002の違いは、この誤ったオーバーレイによって画像全体が半ピクセルずれるという事実によるものです。これらのピクセルの中心は、各エッジから0.001の位置にある必要があります。

コードの2行目は次のようになります。

    float2 d = 2.0 *
        ( ( float2(idx.x, idx.y) + float2(0.5,0.5) ) / 
            float2(width, height) ) - 1.0;

次に、ピクセルの中心の正しいNDC間隔（-0.999〜0.999）を取得します。代わりに、浮動小数点（0,0; ​​0,0）および（1000,0; ​​1000,0）を使用して角度値をこの方法で変換する場合（既に浮動小数点を使用しているため、0.5を追加しません）、次に、エッジからエッジまで、-1.0から1.0までの完全なNDC間隔を取得します。これはコードの正確さを証明します。

0.5が不快で対称性に欠ける場合は、ピクセル内にランダムな値を生成します。各ピクセルを通してランダムな光線を放出することによってスムージングを実行するとき、このエレガントな定式化を使用できます。

    float2 d = 2.0 *
        ( ( float2(idx.x, idx.y) + 
                float2( rand(seed), rand(seed) ) ) /
            float2(width, height) ) - 1.0;

間隔[0.0,1.0）から乱数をピクセル位置の各整数値に追加するだけです。このランダム値の平均は0.5、つまりピクセルの中心になります。

だから私は簡単に言います：注意してください。ハーフピクセルを正しく実装してください。私の経験では、オートデスクでのラスタライザコードに関する長年の作業中に、このようなハーフピクセルエラーがさまざまな場所（カメラ、テクスチャサンプリングなど）でポップアップします。組み立てラインのさらに下には、痛みしかありません。注意しないと、レイトレーサーにも表示される可能性があります。
