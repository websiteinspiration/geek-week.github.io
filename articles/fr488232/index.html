Résoudre les problèmes des tests automatiques Espresso-Android dans le monde réel -
Alexey Tyurin, VKontakte



Souvent, les projets qui ont déjà pris de l'ampleur et qui ont une grande base de code, arrivent à la conclusion que vous devez écrire des autotests. Y compris UI-autotests. Dans cette situation, vous n'avez pas la possibilité de refaire de manière significative l'architecture du projet pour les tests. En conséquence, les tests créés deviennent instables et distraient l'équipe de développement de produits plus qu'ils ne l'aident.
Le but du rapport: montrer aux gens comment des problèmes spécifiques sont résolus dans l'automatisation des tests d'interface utilisateur d'une application Android (au cas par cas). De plus, ces solutions ne nécessitent pas de retravailler l'architecture de l'application ni de révision importante pour les autotests.

Évaluer la couverture des tests API sans supprimer la couverture du code -
Sergey Alekseev, camarades de classe



L'API est l'un des services les plus importants à Odnoklassniki, il est requis pour le travail des plates-formes utilisateur et d'un grand nombre de partenaires externes, donc tester l'API est une tâche primordiale qui nécessite une attention particulière et de nombreuses solutions non standard.
Dans le rapport, je parlerai de la façon dont les tests d'API sont organisés à Odnoklassniki, partagerai mon expérience dans le développement d'un outil interne qui nous aide à trouver les points faibles / faibles des auto-tests d'API, puis je montrerai comment vous pouvez mettre en place un analyseur pour couvrir les méthodes d'API avec des tests automatiques pour votre infrastructure. pour les exigences personnalisées, et à la fin, je porterai une attention particulière aux approches de collecte et d'analyse de la couverture des tests.

Comment utiliser OpenAPI-Generator pour tester l'API REST -
Viktor Orlovsky, Yandex.Verticals



Dans mon rapport, je vais vous expliquer comment nous utilisons Openapi-generator dans Yandex.Verticals pour écrire des autotests sur l'API REST, partager mon expérience sur la façon dont nous pouvons rapidement et facilement couvrir un grand nombre de nouvelles API de services avec des autotests, et aussi prêter attention à quels problèmes nous avons rencontrés en cours de route et comment les résoudre. 
Nous passerons des tests hérités sur le client http à l'API à la génération de vrais tests basés sur la spécification.
