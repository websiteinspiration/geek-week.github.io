1.プロパティを分解する


プロパティの分解（React英語の用語では、それらは「プロップ」と呼ばれます）は、コードをよりクリーンにし、サポート機能を改善するための良い方法です。これにより、エンティティが使用するもの（Reactコンポーネントなど）を明確に表現または宣言できるようになります。ただし、このアプローチでは、コンポーネントに関連付けられているプロパティの構成を見つけるために、開発者がコンポーネントの実装を読み取る必要はありません。

プロパティの破壊により、プログラマはデフォルト値を設定することもできます。これはかなり一般的です：

import React from 'react'
import Button from 'components/Button'

const MyComponent = ({ placeholder = '', style, ...otherProps }) => {
  return (
    <Button
      type="button"
      style={{
        border: `1px solid ${placeholder ? 'salmon' : '#333'}`,
        ...style,
      }}
      {...otherProps}
    >
      Click Me
    </Button>
  )
}

export default MyComponent

JavaScriptで構造化解除を使用することの最も楽しい結果の1つは、私が見つけることができましたが、パラメーターのさまざまなオプションをサポートできることです。

3.標準の命名規則を使用してコンポーネントに名前を付ける


コンポーネントに名前を付けるときに特定の標準を使用すると、プロジェクトの作成者ではない誰かがこのプロジェクトのコードを簡単に読み取ることができます。

たとえば、高次コンポーネント（HOC）の名前には通常接頭辞が付けられwithます。多くの開発者はこれらのコンポーネント名に慣れています：

import React from 'react'
import hoistNonReactStatics from 'hoist-non-react-statics'
import getDisplayName from 'utils/getDisplayName'

const withFreeMoney = (WrappedComponent) => {
  class WithFreeMoney extends React.Component {
    giveFreeMoney() {
      return 50000
    }

    render() {
      return (
        <WrappedComponent
          additionalMoney={[
            this.giveFreeMoney(),
            this.giveFreeMoney(),
            this.giveFreeMoney(),
            this.giveFreeMoney(),
            this.giveFreeMoney(),
            this.giveFreeMoney(),
            this.giveFreeMoney(),
          ]}
          {...this.props}
        />
      )

  WithFreeMoney.displayName = `withFreeMoney(${getDisplayName(

4.ブールトラップを回避する


一部の出力がいくつかのプリミティブな論理値に依存し、一部の決定がこれらの値の分析に基づいて行われる場合、プログラマーは細心の注意を払う必要があります。これは、コードの品質が低いことを示しています。これにより、開発者はコンポーネントやその他のメカニズムを実装するためのコードを読み取って、これらのメカニズムの結果が正確に何であるかを正確に把握する必要があります。

我々はコンポーネント作成するとTypography、以下のオプションを取ることができます：'h1'、'h2'、'h3'、'h4'、'h5」、 、'h6'、。'title' オプションが次の形式でコンポーネントに渡される場合、コンポーネントの出力に正確に何が影響しますか？'subheading'



const App = () => (
  <Typography color="primary" align="center" subheading title>
    Welcome to my bio
  </Typography>
)

React（またはJavaScriptを使用した経験）の経験がある人は、システムの性質上、オプションtitleがオプションをオーバーライドするとすでに想定しているかもしれませんsubheading。最後のオプションは最初のオプションを上書きします。

しかし、ここでの問題は正確に限りオプション何に私たちは、コードに見ず、言うことができないということであるtitleかのオプションが適用されますsubheading。

例えば：

.title {
  font-size: 1.2rem;
  font-weight: 500;
  text-transform: uppercase;
}

.subheading {
  font-size: 1.1rem;
  font-weight: 400;
  text-transform: none !important;
}

にもかかわらずtitle、それが勝利、CSSルールはtext-transform: uppercase適用されません。これはtext-transform: none !important、に存在するルールの特定性が高いためですsubheading。このような状況で注意を怠ると、スタイルでのこのようなエラーのデバッグが非常に困難になる可能性があります。特に、コードがコンソールに警告またはエラーメッセージを表示しない場合。これにより、コンポーネントの署名が複雑になる可能性があります。

この問題の考えられる解決策の1つは、コンポーネントのよりクリーンなバージョンを使用することですTypography。

const App = () => <Typography variant="title">Welcome to my bio</Typography>

これがコンポーネントコードですTypography：

import React from 'react'
import cx from 'classnames'
import styles from './styles.css'

const Typography = ({
  children,
  color = '#333',
  align = 'left',
  variant,
  ...otherProps
}) => {
  return (
    <div
      className={cx({
        [styles.h1]: variant === 'h1',
        [styles.h2]: variant === 'h2',
        [styles.h3]: variant === 'h3',
        [styles.h4]: variant === 'h4',
        [styles.h5]: variant === 'h5',
        [styles.h6]: variant === 'h6',
        [styles.title]: variant === 'title',
        [styles.subheading]: variant === 'subheading',
      })}
    >
      {children}
    </div>
  )
}

これでApp、コンポーネントをコンポーネントに渡すとTypography variant="title"、コンポーネントの出力のみが影響を与えることが確実になりますtitle。これにより、コンポーネントがどのように見えるかを理解するためにコンポーネントコードを分析する必要がなくなります。

プロパティを操作するには、シンプルなデザインを適用できますif/else。

let result
if (variant === 'h1') result = styles.h1
else if (variant === 'h2') result = styles.h2
else if (variant === 'h3') result = styles.h3
else if (variant === 'h4') result = styles.h4
else if (variant === 'h5') result = styles.h5
else if (variant === 'h6') result = styles.h6
else if (variant === 'title') result = styles.title
else if (variant === 'subheading') result = styles.subheading

ただし、このアプローチの主な長所は、次のクリーンな単一行設計を使用してこれに終止符を打つことができるということです。

const result = styles[variant]

6.独自のフックの外に独立した関数を配置する


一部のプログラマーが独自のフック内で関数を宣言する方法を見てきましたが、同時に、これらのフックはそのような関数を特に必要としません。これにより、フックコードがわずかに膨張し、読み取りが複雑になります。コードの読み取りが困難になるのは、フックが実際にフックに含まれている関数に依存しているかどうかについて、リーダーが質問し始める可能性があるためです。そうでない場合は、関数をフックの外に移動することをお勧めします。これにより、コードリーダーは、フックが依存するものと依存しないものを明確に理解できます。

次に例を示します。

import React from 'react'

const initialState = {
  initiated: false,
  images: [],
}

const reducer = (state, action) => {
  switch (action.type) {
    case 'initiated':
      return { ...state, initiated: true }
    case 'set-images':
      return { ...state, images: action.images }
    default:
      return state
  }
}

const usePhotosList = ({ imagesList = [] }) => {
  const [state, dispatch] = React.useReducer(reducer, initialState)

  const removeFalseyImages = (images = []) =>
    images.reduce((acc, img) => (img ? [...acc, img] : acc), [])

  React.useEffect(() => {
    const images = removeFalseyImages(imagesList)
    dispatch({ type: 'initiated' })
    dispatch({ type: 'set-images', images })
  }, [])

  return {
    ...state,
  }
}

export default usePhotosList

このコードを分析するremoveFalseyImagesと、実際にはfunctions がフックの内部に存在する必要がないことがわかります。これは、その状態と相互作用しないため、コードの外部に配置して、問題なくフックから呼び出すことができます。

7.コードを書くときは一貫している


コードを作成するための一貫したアプローチは、JavaScriptでプログラミングする人にしばしば推奨されるものです。

Reactの場合、以下の構造の使用に対する一貫したアプローチに注意を払う価値があります。

  1. チームのインポートとエクスポート。
  2. コンポーネント、フック、高次コンポーネント、クラスの命名。

コンポーネントをインポートおよびエクスポートするとき、次のようなものを使用することがあります。

import App from './App'

export { default as Breadcrumb } from './Breadcrumb'

export default App

しかし、私は構文も好きです：

export { default } from './App'
export { default as Breadcrumb } from './Breadcrumb'

プログラマーが選択するものは何でも、彼は自分が作成するすべてのプロジェクトで一貫してこれを使用する必要があります。これにより、このプログラマの作業と他の人による彼のコードの読み取りが簡単になります。

エンティティの命名規則に従うことは非常に重要です。

たとえば、誰かがフックに名前を付けた場合useApp、他のフックの名前がプレフィックスを使用して同様の方法で構築されることが重要ですuse。たとえば、このアプローチでの別のフックの名前は次のようになりuseControllerます。

このルールに従わない場合、特定のプロジェクトのコードは、最終的には次のようになる可能性があります。

//  #1
const useApp = ({ data: dataProp = null }) => {
  const [data, setData] = React.useState(dataProp)

  React.useEffect(() => {
    setData(data)
  }, [])

  return {
    data,
  }
}

//  #2
const basicController = ({ device: deviceProp }) => {
  const [device, setDevice] = React.useState(deviceProp)

  React.useEffect(() => {
    if (!device && deviceProp) {
      setDevice(deviceProp === 'mobile' ? 'mobile' : 'desktop')
    }
  }, [deviceProp])

  return {
    device,
  }
}

これらのフックのインポートは次のようになります。

import React from 'react'
import useApp from './useApp'
import basicController from './basicController'

const App = () => {
  const app = useApp()
  const controller = basicController()

  return (
    <div>
      {controller.errors.map((errorMsg) => (
        <div>{errorMsg}</div>
      ))}
    </div>
  )
}
