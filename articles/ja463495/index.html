Laravelキャッシング：基本とヒントとコツ

キャッシング手法を使用すると、よりスケーラブルなアプリケーションを作成し、一部のクエリの結果を高速のメモリ内ストレージに保存できます。ただし、不適切に実装されたキャッシュは、アプリケーションのユーザーの印象を大幅に低下させる可能性があります。この記事には、私が過去のいくつかのプロジェクトから学んだ、キャッシングに関する基本的な概念、さまざまなルール、タブーが含まれています。


キャッシングは使用しないでください。


プロジェクトは高速で実行され、パフォーマンスの問題はありませんか？
キャッシュについては忘れてください。真剣に:)


データベースからの読み取り操作は、何の利点もなく非常に複雑になります。


確かに、この記事の冒頭にあるMohamed Saidは、いくつかの計算を行い、場合によっては、アプリケーションをミリ秒単位で最適化すると、AWSアカウントのコストを大幅に節約できることを証明しました。そのため、プロジェクトの予測節約額が1.86ドルを超える場合は、おそらくキャッシュが適切です。


使い方？


アプリケーションがデータベースからデータを取得したい場合、たとえば、PostエンティティをIDで取得する場合、このケースの一意のキャッシュキーを生成し（'post_' . $id非常に適切）、このキーを使用して高速キー値ストレージ（memcache、redisなど）で値を見つけようとします。値が存在する場合、アプリケーションはそれを使用します。そうでない場合は、データベースから取得し、将来使用するためにこのキーによってキャッシュに格納します。



このPostエンティティは更新できるため、キャッシュにこの値を保持することは常に良い考えとは限りませんが、アプリケーションは常に古いキャッシュ値を受け取ります。
したがって、キャッシング関数は通常、この値を保存する時間を尋ねます。


この時間が経過すると、memcacheまたはredisはそれを「忘れ」、アプリケーションはデータベースから新しい値を取得します。


例：


public function getPost($id): Post
{
    $key = 'post_' . $id;

    $post = \Cache::get($key);
    if($post === null) {
        $post = Post::findOrFail($id);
        \Cache::put($key, $post, 900);
    }

    return $post;
}
