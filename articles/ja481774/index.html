セキュリティクリブ：仮想パッチ

今日の記事のトピックは仮想パッチです。

仮想パッチは、既知の脆弱性に対するエクスプロイトの悪用を検出および防止するために設計されたセキュリティポリシーレイヤーです。

仮想パッチはトラフィック分析の後に機能し始め、悪意のあるトラフィックが脆弱なアプリケーションに入るのを防ぎます。したがって、該当する場合、仮想パッチはアプリケーションのソースコードを変更せずに脆弱性の悪用を防ぎます。通常、仮想パッチはWebアプリケーション（WAF）のファイアウォールに実装されます。

なぜ仮想パッチなのか、なぜコードを更新しないのか？

もちろん、コードの更新は最初の推奨されるソリューションですが、組織内のリソースの不足（すべての開発者がすでに忙しく、緊急パッチに転送できない）、他の誰かのソフトウェアの使用（ここでコードを変更する方法）、または少しだけ書き直す必要がある場合は、常に可能ではありませんこのパッチのすべてのアプリケーションではない場合。
これは、仮想パッチとコードの更新が相互に排他的であることを意味しません！これらは通常、（これらのプロセスの詳細を考慮して）異なるコマンドによって実行されるため、パッチの適用と更新を並行して行うことができます。

仮想パッチの利点は次のとおりです。

  • 本格的なパッチの可能性がなくなるまで脆弱性を閉じるための迅速な（比較的）ソリューション。
  • Webアプリケーションが製品である場合、既存のビジネスプロセスに違反することはありません。たとえば、製品の次のパッチが1か月に計画されている場合は、スケジュール全体を中断する必要はありません。
  • 現在、開発者がこれを行うことができない場合に、別のチームのメンバーとの脆弱性を閉じる機能。
  • 適時性-既製のソフトウェアを使用している場合、製品のパッチがいつリリースされるかは不明であり、もちろん、脆弱性を残したくない。

仮想パッチの欠点：

  • これはまだ万能薬ではありません。これを使用する場合、すべての攻撃ベクトルをカバーできるわけではないため、操作の危険性が残ります。
  • すべての一時的なソリューションの欠点は、一時的なものが永続的になる可能性です。会社は、それがうまくいったら、それ以上は触れないで、そのままにしておくことを決定します。
  • これはアドオンソリューションであり、脆弱性自体は消えませんが、追加の保護層の背後に隠れます。

仮想パッチの準備は、次の手順に分けることができます。

  1. トレーニング
  2. 脅威の識別
  3. 分析
  4. 仮想パッチを作成する
  5. 実装とテスト
  6. 復旧・作業継続

脅威検出フェーズ

このフェーズは、組織がWebアプリケーションに脆弱性があることを発見したときに始まります。一般に、脆弱性を特定するには2つの異なるアプローチがあります-事前対応と事後対応です。

予防的なアプローチ

組織がセキュリティ作業を引き受け、次のタスクを実行する場合。

  • アプリケーションの動的評価-ペンテストまたは自動評価テスト、あるいはその両方が実行中のWebアプリケーションで実行され、欠陥を特定します。
  • ソースコードのレビュー-Webアプリケーションのソースコードを手動または自動で評価して、欠陥を検出します。

対処的アプローチ

脆弱性を特定するための3つの主要な対処的方法があります。

  • ベンダーからのメッセージ-ソフトウェアベンダーが発見された脆弱性に関する情報を公開したとき。
  • 公開公開とは、公開ソースで使用するソフトウェアに発見された脆弱性の公開です。この場合、脆弱性についてより多くの人々が知っているため、脆弱性に対するあなたの反応は、ベンダーからの報告よりも速くなるはずです。
  • セキュリティインシデント-これは、攻撃がすでに進行中であることを意味します。脆弱性を具体化して閉じるには、ただちに対処する必要があります。

仮想パッチ

の作成段階適切な仮想パッチを作成するプロセスは、次の2つの側面に関連しています。

  • 誤検知なし-状況にかかわらず通常のトラフィックをブロックしない
  • 誤検知なし-攻撃者が明確に検出を回避しようとした場合でも、攻撃を見逃すことはありません。

これらのルールの両方の影響を最小限に抑えるよう努力する必要があります。これらの両方のルールに従うことは（ほとんどの場合）不可能である可能性がありますが、仮想パッチはリスクを軽減することであることを覚えておくことは重要です。

積極的なアプローチ（ホワイトリスト、ホワイトリスト）

このアプローチは、Webアプリケーションの入力データの独立した検証を作成することです。このアプローチは、有効なデータの特性（文字セット、長さなど）を決定し、これらの条件に適合しないものはすべて禁止します。 Webアプリケーションの各ページの各パラメーターのルールを定義することにより、アプリケーションを、そのソースコードに関係なく、追加の保護層でラップします。

ModSecurityのホワイトリストに基づいて仮想パッチを作成する例
ホワイトリストの仮想パッチを作成するには、通常の期待値を確認できる必要があります。準備フェーズの一部として正しいロギングが事前構成されていた場合、ログの確認を通じて、予想される入力値の形式を理解できるはずです。

この場合、この仮想パッチを適用できるように、reqIDパラメータには整数のみを含める必要があります。

#
# ,     1    "reqID"
#
SecRule REQUEST_URI "@contains /wp-content/plugins/levelfourstorefront/scripts/administration/exportsubscribers.php" "chain,id:1,phase:2,t:none,t:Utf8toUnicode,t:urlDecodeUni,t:normalizePathWin,t:lowercase,block,msg:'Input Validation Error for \'reqID\' parameter - Duplicate Parameters Names Seen.',logdata:'%{matched_var}'"
  SecRule &ARGS:/reqID/ "!@eq 1"

#
# ,    reqID    
#
SecRule REQUEST_URI "@contains /wp-content/plugins/levelfourstorefront/scripts/administration/exportsubscribers.php" "chain,id:2,phase:2,t:none,t:Utf8toUnicode,t:urlDecodeUni,t:normalizePathWin,t:lowercase,block,msg:'Input Validation Error for \'reqID\' parameter.',logdata:'%{args.reqid}'"
  SecRule ARGS:/reqID/ "!@rx ^[0-9]+$"

この仮想パッチは、reqIDパラメータを分析し、入力に整数のみを許可します。ただし、攻撃ベクトルはほとんど単一ではなく、攻撃者は別の方法で運を試すことができることを覚えておく価値があります。

仮想パッチでの否定的なアプローチ（ブラックリスト、ブラックリスト）

このアプローチは、有効なトラフィックのみを許可するのではなく、特定の既知の攻撃を定義するルールのリストに基づいています。

ModSecurityのブラックリストに基づいて仮想パッチを作成する例

公開ソースからのPoCコードの例として：

http://localhost/wordpress/wp-content/plugins/levelfourstorefront/scripts/administration/exportsubscribers.php?reqID=1' or 1='1

負荷を分析すると、攻撃者が単一引用符を挿入し、SQLロジックを最後に追加していることがわかります。このデータに基づいて、単一引用符を禁止できます。

SecRule REQUEST_URI "@contains /wp-content/plugins/levelfourstorefront/scripts/administration/exportsubscribers.php" "chain,id:1,phase:2,t:none,t:Utf8toUnicode,t:urlDecodeUni,t:normalizePathWin,t:lowercase,block,msg:'Input Validation Error for \'reqID\' parameter.',logdata:'%{args.reqid}'"
  SecRule ARGS:/reqID/ "@pm '"

特定のエクスプロイトに対する仮想パッチの作成に注意してください。

仮想パッチを自動的に作成する

脆弱性の数が増えるため、手動でのパッチの作成は耐え難くなる可能性があり、自動化が必要なステップになります。自動化ツール（たとえば、脆弱性スキャナー）を使用して脆弱性が発見された場合、これらのツールのレポートをパッチに変換できる可能性があります。最も一般的なパッチ自動化ツールは、WAFへの直接インポート（当然、ソリューションがこれをサポートしている場合）、OWASP ModSecurityコアルールセット（CRS）スクリプト、およびThreadFix仮想パッチです。

  • ウェブブラウザ
  • 端末用のWebユーティリティ（curlやwgetなど）
  • ローカルプロキシ
  • ModSecurity AuditViewer
