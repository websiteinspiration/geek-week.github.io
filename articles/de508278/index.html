Praktische funktionale Programmierung







Der Text des Artikels stammt aus einer Präsentation, die ich 2016 auf LinkedIn gezeigt habe. In der Präsentation wurde versucht, die funktionale Programmierung zu erklären, ohne Konzepte wie „Monaden“, „Unveränderlichkeit“ oder „Nebenwirkungen“ zu verwenden. Stattdessen konzentriert sie sich darauf, wie das Nachdenken über Komposition Sie zu einem besseren Programmierer machen kann, unabhängig davon, welche Sprache Sie verwenden.



Vor 40 Jahren, am 17. Oktober 1977, wurde John Bakus der Turing-Preis für seinen Beitrag zur Entwicklung hochrangiger Programmiersysteme, vor allem der Programmiersprache Fortran, verliehen. Alle Turing-Preisträger haben die Möglichkeit, in dem Jahr, in dem sie den Preis erhalten, einen Vortrag zu ihrem gewählten Thema zu halten. Als Schöpfer der Fortran-Programmiersprache würde man erwarten, dass Bakus einen Vortrag über Fortrans Vorteile und zukünftige Entwicklungen in dieser Sprache hält. Stattdessen hielt er einen Vortrag mit dem Titel „Ist es möglich, die Programmierung von Neumanns Stil zu befreien?“. in dem er einige der wichtigsten Sprachen der Zeit, einschließlich Fortran, für ihre Mängel kritisierte. Er schlug auch eine Alternative vor: einen funktionalen Programmierstil.



Die Vorlesung kontrastiert traditionelle Programme und ihre „Unfähigkeit, leistungsfähige Kombinationsformen effektiv zu nutzen“ mit Funktionsprogrammen, die „auf der Verwendung kombinierter Formen basieren“. In den letzten Jahren hat die funktionale Programmierung aufgrund des Wachstums von skalierbarem und parallelem Computing neues Interesse geweckt. Der Hauptvorteil der funktionalen Programmierung besteht jedoch darin, ob Ihr Programm parallelisiert ist oder nicht: Die funktionale Programmierung ist in der Komposition besser.



Komposition ist die Fähigkeit, komplexes Verhalten durch Kombination einfacher Teile zusammenzusetzen. Im Informatikunterricht wird viel Wert auf die Abstraktion gelegt: ein großes Problem aufgreifen und in Teile teilen. Das Gegenteil wird weniger betont: Sobald Sie die kleinen Teile erkennen, wie Sie sie zusammensetzen. Es scheint, dass einige Funktionen und Systeme einfach zusammenzustellen sind, während andere viel komplizierter sind. Wir müssen jedoch einen Schritt zurücktreten und fragen: Welche Eigenschaften dieser Funktionen und Systeme erleichtern ihr Layout? Welche Eigenschaften erschweren das Layout? Nachdem Sie genügend Code gelesen haben, wird das Muster angezeigt. Dieses Muster ist der Schlüssel zum Verständnis der funktionalen Programmierung.



Beginnen wir mit einer Funktion, die wirklich gut verknüpft ist: 



String addFooter(String message) {
  return message.concat(" - Sent from LinkedIn");
}


Wir können es einfach mit einer anderen Funktion verknüpfen, ohne Änderungen an unserem Quellcode vornehmen zu müssen: 



boolean validMessage(String message) {
  return characterCount(addFooter(message)) <= 140;
}


Das ist großartig, wir haben ein kleines Stück Funktionalität genommen und es zusammengestellt, um etwas mehr zu tun. Funktionsbenutzer validMessage



müssen sich nicht einmal der Tatsache bewusst sein, dass diese Funktion aus einer kleineren Funktion erstellt wurde. Es wird als Implementierungsdetail abstrahiert.



Schauen wir uns nun eine Funktion an, die nicht so übersichtlich ist: 



String firstWord(String message) {
  String[] words = message.split(' ');
  if (words.length > 0) {
    return words[0];
  } else {
    return null;
  }
}


Und dann versuchen Sie es mit einer anderen Funktion zu komponieren: 



// “Hello world” -> “HelloHello”
duplicate(firstWord(message));


Trotz der Einfachheit auf den ersten Blick erhalten wir eine schreckliche Ausnahme, wenn wir den obigen Code mit einer leeren Nachricht ausführen NullPointerException



. Eine Möglichkeit besteht darin, die Funktion "Veröffentlichen" so zu ändern, dass die Eingabe manchmal wie folgt erfolgt null



: 



String duplicateBad(String word) {
  if (word == null) {
    return null;
  } else {
    return word.concat(word);
  }
}


Jetzt können wir diese Funktion mit der Funktion firstWord



aus dem vorherigen Beispiel verwenden und einfach einen Nullwert übergeben null



. Dies ist jedoch gegen Komposition und Abstraktion. Wenn Sie die Komponenten jedes Mal, wenn Sie etwas mehr tun möchten, ständig ändern müssen, ist dies für das Layout nicht geeignet. Idealerweise sollen die Funktionen wie Black Boxes aussehen, bei denen die genauen Implementierungsdetails keine Rolle spielen.



Nullobjekte lassen sich nicht gut zusammensetzen.


Schauen wir uns eine alternative Implementierung an, die den Java 8-Typ verwendet Optional



(auch aufgerufen Option



oder Maybe



in anderen Sprachen): 



Optional<String> firstWord(String message) {
  String[] words = message.split(' ');
  if (words.length > 0) {
    return Optional.of(words[0]);
  } else {
    return Optional.empty();
  }
}


Jetzt werden wir versuchen, es mit einer unveränderten Funktion zu komponieren dublicate



: 



// "Hello World" -> Optional.of("HelloHello")
firstWord(input).map(this::duplicate)


Es klappt! Der optionale Typ sorgt dafür, dass firstWord manchmal keinen Wert zurückgibt. Bei Optional.empty()



Rückgabe von überspringt die firstWord



Funktion .map



einfach den Start der Funktion dublicate



. Wir konnten Funktionen problemlos kombinieren, ohne sie ändern zu müssen dublicate



. Vergleichen Sie dies mit dem null



Fall, in dem wir eine Funktion erstellen mussten duplicateBad



. Mit anderen Worten: Nullobjekte sind schlecht kompiliert und Optionen sind gut.



Funktionale Programmierer sind besessen davon, Dinge zusammenzusetzen. Infolgedessen erstellten sie eine große Anzahl von Werkzeugen, die mit Strukturen gefüllt sind, die nicht zusammensetzbaren Code zusammensetzbar machen. Eines dieser Tools ist ein optionaler Typ für die Arbeit mit Funktionen, die nur zu einem bestimmten Zeitpunkt eine gültige Ausgabe zurückgeben. Schauen wir uns einige andere Tools an, die erstellt wurden.



Asynchroner Code ist bekanntermaßen schwer zusammenzusetzen. Asynchrone Funktionen erhalten normalerweise "Rückrufe", die ausgelöst werden, wenn der asynchrone Teil des Anrufs abgeschlossen ist. Zum Beispiel die FunktiongetData



kann einen HTTP-Aufruf an einen Webdienst senden und dann eine Funktion für die zurückgegebenen Daten ausführen. Aber was ist, wenn Sie gleich danach einen weiteren HTTP-Aufruf tätigen möchten? Und dann nochmal? Wenn Sie dies schnell tun, gelangen Sie in eine Situation, die liebevoll als Rückrufhölle bezeichnet wird. 



getData(function(a) {  
    getMoreData(a, function(b) {
        getMoreData(b, function(c) { 
            getMoreData(c, function(d) { 
                getMoreData(d, function(e) { 
                    // ...
                });
            });
        });
    });
});


In einer größeren Webanwendung führt dies beispielsweise zu einem sehr verschachtelten Spaghetti-Code. Stellen Sie sich vor, Sie versuchen, eine der Funktionen getMoreData



in einer eigenen Methode zu isolieren . Oder stellen Sie sich vor, Sie versuchen, dieser verschachtelten Funktion eine Fehlerbehandlung hinzuzufügen. Der Grund, warum es nicht verknüpft werden kann, liegt darin, dass für jeden Codeblock viele kontextbezogene Anforderungen gelten: Der innere Block erfordert den Zugriff auf Ergebnisse von a, b, c usw. usw.



Werte lassen sich leichter zusammenstellen als Funktionen


Schauen wir uns das Toolkit eines funktionalen Programmierers an, um eine Alternative zu finden: Promise



(manchmal Future



in anderen Sprachen aufgerufen ). Hier ist jetzt der Code: 



getData()
  .then(getMoreData)
  .then(getMoreData)
  .then(getMoreData)
  .catch(errorHandler)


Funktionen geben getData



jetzt einen Wert zurück, Promise



anstatt eine Rückruffunktion zu akzeptieren. Werte lassen sich leichter zusammenstellen als Funktionen, da sie nicht die gleichen Voraussetzungen wie der Rückruf haben. Dank der Funktionalität, die das Objekt uns bietet, ist es jetzt einfach, dem gesamten Block eine Fehlerbehandlung hinzuzufügen Promise



.



Ein weiteres Beispiel für nicht zusammensetzbaren Code, über den weniger gesprochen wird als über asynchronen Code, sind Schleifen oder allgemeiner Funktionen, die mehrere Werte zurückgeben, z. B. Listen. Schauen wir uns ein Beispiel an: 



// ["hello", "world"] -> ["hello!", "world!"]
List<String> addExcitement(List<String> words) {
  List<String> output = new LinkedList<>();
  for (int i = 0; i < words.size(); i++) {
    output.add(words.get(i) + “!”);
  }
  return output;
}

// ["hello", "world"] -> ["hello!!", "world!!"]
List<String> addMoreExcitement(List<String> words) {
  return addExcitement(addExcitement(words));
}


Wir haben eine Funktion kompiliert, die ein Ausrufezeichen zu einer Funktion hinzufügt, die zwei Zeichen hinzufügt. Dies funktioniert, ist jedoch ineffizient, da es die Schleife zweimal und nicht nur einmal durchläuft. Wir könnten zurückgehen und die ursprüngliche Funktion ändern, aber dies verletzt nach wie vor die Abstraktion.



Dies ist ein leicht erfundenes Beispiel, aber wenn Sie sich Code vorstellen, der über eine größere Codebasis verstreut ist, zeigt dies einen wichtigen Punkt: Wenn Sie auf großen Systemen versuchen, Dinge in Module zu zerlegen, werden Operationen an einem Datenelement nicht zusammenleben. Sie müssen zwischen Modularität oder Leistung wählen.



Mit zwingender Programmierung können Sie nur Modularität oder nur Leistung erhalten. Mit funktionaler Programmierung können Sie beides haben.


Die Antwort eines funktionierenden Programmierers (zumindest in Java 8) lautet Stream



: Stream



Der Standardwert ist faul, was bedeutet, dass die Daten nur bei Bedarf durchlaufen werden. Mit anderen Worten, eine „faule“ Funktion: Sie beginnt erst zu funktionieren, wenn sie nach dem Ergebnis gefragt wird (die funktionale Programmiersprache von Haskell basiert auf dem Konzept der Faulheit). Schreiben wir stattdessen das obige Beispiel mit Stream



: 



String addExcitement(String word) {
  return word + "!";
}

list.toStream()
  .map(this::addExcitement)
  .map(this::addExcitement)
  .collect(Collectors.toList())


Daher wird die Liste nur einmal durchlaufen und die Funktion addExcitement



für jedes Element zweimal aufgerufen . Auch hier müssen wir uns vorstellen, dass unser Code in mehreren Teilen der Anwendung mit denselben Daten arbeitet. Ohne eine solch träge Struktur würde der Stream



Versuch, die Produktivität durch Kombinieren aller Crawls von Listen an einem Ort zu steigern, die Zerstörung bestehender Funktionen bedeuten. Mit einem faulen Objekt können Sie sowohl Modularität als auch Leistung erzielen, da Bypässe bis zum Ende verzögert werden.



Nachdem wir uns einige Beispiele angesehen haben, kehren wir zur Aufgabe zurück, um herauszufinden, welche Eigenschaften das Erstellen einiger Funktionen einfacher machen als andere. Wir haben gesehen, dass Dinge wie Nullobjekte, Rückrufe und Schleifen nicht gut zusammengesetzt sind. Auf der anderen Seite stapeln sich optionale Typen, Versprechen und Streams wirklich gut. Warum ist das so?



Die Antwort ist, dass zusammengesetzte Beispiele klar unterscheiden, was Sie tun möchten und wie Sie es tatsächlich tun.



Alle vorherigen Beispiele haben eines gemeinsam. Eine funktionale Vorgehensweise konzentriert sich auf das, was das Ergebnis sein soll. Die iterative Vorgehensweise konzentriert sich darauf, wie Sie tatsächlich dorthin gelangen, und auf Implementierungsdetails. Es stellt sich heraus, dass das Schreiben iterativer Anweisungen zur Vorgehensweise nicht so gut aussieht wie allgemeine Beschreibungen der zu erledigenden Aufgaben.



Zum Beispiel im Fall von Promise: In diesem Fall wird ein HTTP-Aufruf gefolgt von einem anderen ausgeführt. Die Frage ist nicht relevant und wird abstrahiert: Vielleicht werden Thread-Pools, Mutex-Sperren usw. verwendet, aber das spielt keine Rolle.



Die funktionale Programmierung trennt das gewünschte Ergebnis von der Erzielung dieses Ergebnisses.


Das ist genau meine praktische Definition der funktionalen Programmierung. Wir wollen eine klare Trennung der Probleme in unseren Programmen. Der Teil „Was Sie wollen“ ist gut und kompositorisch und macht es einfach, große Dinge aus den kleineren zu erstellen. Irgendwann ist der Teil „Wie Sie es machen“ erforderlich. Durch Trennen entfernen wir jedoch Material, das nicht so zusammengesetzt ist, aus dem Material, das zusammengesetzter können



wir es in realen Beispielen sehen:



  • Die Apache Spark-API zum Durchführen von Berechnungen für große Datenmengen abstrahiert die Details, auf welchen Computern sie ausgeführt wird und wo die Daten gespeichert sind.
  • React.js beschreibt die Präsentation und macht die DOM-Transformation zu einem effizienten Algorithmus


Selbst wenn Sie keine funktionierende Programmiersprache verwenden, werden sie durch die Trennung von Was und Wie von Ihren Programmen komponierbarer.
