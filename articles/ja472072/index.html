1. JSONPathをサポートする


（リリースノートでは、SQL / JSONパス言語のサポートを追加するように聞こえます（Nikita Glukhov、Teodor Sigaev、Alexander Korotkov、Oleg Bartunov、Liudmila Mantrova）

このパッチ自体、JSONPath機能、および問題の履歴については、ハブの別の記事で詳しく説明しました JSONPathはPostgres Professionalの重大な成果であり、PostgreSQL 12の主要な革新の1つです

。2014年に、A。Korotkov、O。Bartunov、およびF. Sigaevがjsquery拡張機能を開発しました。これはPostgres Pro Standard 9.5（およびそれ以降のバージョン）に含まれていましたStandardおよびEnterprise）。json（b）を操作するための追加の非常に幅広い機会を提供します。

標準SQL：2016が登場したとき、そのセマンティクスはjsquery拡張機能のセマンティクスとそれほど変わらないことがわかりました。標準の作成者がjsqueryを一瞥して、JSONPathを発明した可能性もあります。私たちのチームは、私たちが既に持っているものと、もちろん多くの新しいものも、少し異なる方法で実装する必要がありました。

関数を含む特別なパッチはまだコミットされていませんが、JSONPathパッチには、JSON（B）を操作するための主要な関数がすでに含まれています。次に例を示します。

jsonb_path_query('{"a": [1,2,3,4,5]}', '$.a[*] ? (@ > 2)')  3, 4, 5
jsonb_path_query('{"a": [1,2,3,4,5]}', '$.a[*] ? (@ > 5)')  0

さらに、以前に JSONで動作していたいくつかの関数が最適化されました。これはNikita Glukhovによって首尾よく行われました。

例えば、オペレータは、#>>機能に対応jsonb_each_text()し、jsonb_array_elements_text()迅速に十分なテキストにJsonbValueを変換するために使用されるが、他のタイプでゆっくりと働きました。今、すべてが速く働いています。

2. SP-GiST (KNN)


（SP-GiSTインデックスの最近傍（KNN）検索のサポートを追加します。ニキータGlukhov、アレクサンダー・コロトコフ、ヴラドSterzhanov）

当社からニキータGlukhovとアレクサンダーコロトコフは（Quadrocube別名）ミンスクからヴラドSterzhanovが始めた仕事を続けました。 Postgresは、最も近い隣人（以前はOracleとMS、より直接的で便利な方法）を検索する最初のDBMSであり、これがOleg Bartunov氏と彼のチームのメリットでした。この検索のアイデアは、元のツリートラバーサルアルゴリズムにあり、ほとんどの場合、これは大きな利益をもたらします。最近傍の検索は多くの場所で使用されますが、GISでは特に一般的です。

Vladは、平面が固定サイズの正方形に分割されている場合、クワッドツリーの空間インデックスSP-GiSTを操作するためのKNN検索パッチを作成し、KDツリー（k次元ツリー）を作成しました。

VladのGSoCメンター（Google Summer of Code）のAlexander Korotkovは、Postgres Professional Nikita Glukhovの同僚と共同開発を続けました。機能が大幅に強化されました。ツリーをトラバースするときの内部データキャッシングが改善され、距離順の円とポリゴンの演算子クラスが追加されました。

アルゴリズムを使用して最近傍を見つけるには ORDER BY [, ]、と書くだけで、オプティマイザはこのアルゴリズムを自動的に接続します。例えば、


   SELECT * FROM polygons 
         ORDER BY poly <-> point '(0,0)';

Nikita Glukhovによるパッチはgithubで見ることができます。

（リリースノートでは、これはロックのオーバーヘッドを減らすことによるbtreeインデックス挿入の速度の向上です。アレクサンダーコロトコフ）

Postgres Professionalのチーフシステムアーキテクトであるアレクサンダーコロトコフは、Bツリーインデックスに挿入するときに、より合理的なロッキングアルゴリズムを思い付きました。このパッチを適用した後の効果は、挿入が多かれ少なかれ「連続して」発生する場合に顕著になります。72コアのサーバーで測定したところ、この場合、ゲインは50％に達しています。無秩序な挿入では、ゲインはそれほど目立ちません。

4.経済的なWAL


（GiST、GIN、およびSP-GiSTインデックス作成のWAL書き込みオーバーヘッドを削減します。アナスタシアルベンニコバ、アンドレイV.レピホフ）

この一連のパッチは、 GiST、GIN、およびSP-GiSTインデックスの作成時に生成されるWALトラフィックを削減します。このようなインデックスのページを一度だけログに記録できるようになりました-最後に、インデックスがすでに構築されているときに。また、WALでエントリのインデックスを作成するときにエラーが発生した場合、失敗した試行はまったく表示されません。以前は、これはBツリーとRUMを作成するときにのみ可能でした。パッチは、一般的なWALメカニズムを使用します。

サイズを確認するためのxlogスクリプトが含まれています。4M +レコードが4GBを占めるIMDBデータベース（JSON形式）でのテストでは、次のことが示されました。

CREATE INDEX ON imdb USING gin(jb jsonb_path_ops);

古い方法は205秒、WAL 3.2 GBを実行し、新しいアルゴリズムは133秒、WAL 0.4 GBを与えました。

9.ログはpg_ctlを介してローテーションできます


（pg_ctlによるログファイルのローテーションの制御を許可します。堀口京太郎、アレクサンダークズメンコフ、アレクサンダーコロトコフ）

つまり、ユーティリティpg_ctlは新しいオプションを取得しました。

pg_ctl logrotate [-D _] [-s]

このコマンドを実行すると、サーバーは、ログの構成に応じて、新しいログファイルに切り替えるか、既存のログファイルを再度開きます。これは、緊急時に、特に診断のために急成長する巨大なログファイルを転送する必要がある場合などに必要になることがあります。

（xlogreader：ファイルブロックを2回読み込まないでください。ArthurZakirov）
