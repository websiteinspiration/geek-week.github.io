JenkinsとOracle APEXサービスの統合で未来へ

大量のデータとその視覚化を管理するプロセスでは、リポジトリ内の機能管理にアクセスするための単一のツールを導入する必要性に直面しています。さらに、このツールは、データ管理部門のすべての従業員（管理者、アナリスト、開発者、テスター）にとって等しく便利で理解しやすいものでなければなりません。 私の名前はVitaliyです。私はRostelecomでテストの自動化とDevOpsプロセスの実装の方向性を担当しています。この記事では、Rostelecomの大規模データウェアハウスにおけるこの問題の興味深い解決策について説明します。 そして、これが私たちがこの問題を解決するためのそのような一連のツールにたどり着いた方法です。







手始めに、少し背景。以前、オブジェクトをデータウェアハウスにインストールするプロセスを自動化するために、Pythonで独自のモジュールを作成したことを書きました。これはJenkinsによって制御され、必要な起動パラメーターを入力してボタンで手動で、またはユーザーの介入なしにスケジュールに従って完全に自動的に、必要な機能を起動できます。また、当社はすでにOracle APEXを使用して記述された多数のアプリケーションを実装しています。

Oracle Application Express（Oracle APEX）とは何ですか？


Oracle APEX-以前のバージョンのアプリケーションでは、HTML DBと呼ばれていました。このツールを使用して、ブラウザーのみを使用し、PL / SQLやJavaScriptなどの言語でプログラミングの経験がある場合、開発、テスト、およびその後の本番環境での実装のために任意の回路に簡単に展開できる、高速でスケーラブルで安全なWebアプリケーションを開発できます。 。

データ入力フォームを作成するために、複雑な方法でインターフェースをプログラムする必要はありません。アプリケーションには、独自のソリューションを開発するために再利用できる既製のテンプレートが多数あります。また、エンドユーザーはブラウザからアプリケーションにアクセスするため、アプリケーションをコンピュータにインストールする必要がありません。上記のすべてのプラスで、リポジトリのメタデータ（使用されたデータに関する情報）はOracleデータベースにあるため、このツールは無視できません。Oracle REST Data Services（ORDS）


APEXアプリケーションアーキテクチャの例は、Java EEに基づいてOracle HTTPサーバーとmod_plsqlを置き換えるデータサービスです。

ジェンキンスについて少し


多くの人はおそらくそれがどんな種類の獣であるかをすでに知っていますが、このツールを見ているだけの人のために、それが何であるか、そしてリポジトリでそれをどのように使用するかを説明します。

Jenkinsは継続的インテグレーションツールであり、開発プロセスのさまざまな段階でのソフトウェア開発とアプリケーションのデプロイメントに最もよく使用されます。簡単に言えば、Jenkinsはプロセス自動化環境を提供します。その柔軟性とプラグインの大規模なセットにより、あらゆるアプリケーションやテクノロジーへの統合を可能にし、さまざまなバージョン管理システムでの作業をサポートします。

私たちのデータ管理部門では、自動テストの実行を伴うCI / CDプロセスが実装されています。また、Jenkinsはいくつかの自己作成モジュールを管理しているため、多くの手動プロセスが自動化されています。Jenkins Webクライアントインターフェイスの例、クリック可能




誤解の問題とその解決策


多くのタスクを実装し、それらを自動化した結果、問題が発生しました。DevOpsエンジニアとテストスペシャリストだけでなく、これらのプロセスを管理しなければならないだけでなく、開発プロセスに関与している人々も管理する必要があります。エンドユーザーはすべて部門の内部従業員でした。私の意見では、Jenkinsクライアントインターフェイスは非常にシンプルで便利ですが、私はDevOpsの目を通してそれを見ており、誰もが同じ方法でそれを見ることができるわけではありません。従業員がボタン上で実行しなければならない多くのタスクがあったため、ユーザーにとってより親しみやすいインターフェイスを考え出すことが必要になりました。実際、JenkinsにはBlue Oceanというプラグインがあり、ツールのUI表現を変更できます。


Webクライアントインターフェイスの例Jenkins（Blue Oceanプラグイン）、クリック可能

このプラグインは問題を解決できませんでしたが、代わりに、標準インターフェイスが機能しない場合は再構成できます。ほとんどの場合、モジュールをJenkinsに統合する必要があり、その逆は必要ありません。これは、解決される問題の関心です。そのソリューションの時点で、私が書いたように、Oracle APEXを使用して記述された多数のアプリケーションがすでにありました。APEXアプリケーションの1つのインターフェースの例（クリック可能） インターフェースが十分に使いやすく、管理できること、およびJenkinsで必要なタスクを起動するためのすべてのパラメーターがOracleデータベースに格納されていることを確認した後、APEXからいくつかのJenkinsタスクを起動するというアイデアが生まれました。






かなり時間がかかりました。アプリケーション間の相互作用は、RESTなどのサービス指向システムを構築するというアーキテクチャの原則に従って実装されました。 RESTアーキテクチャは、ネットワーク上の分散アプリケーションのコンポーネントの相互作用に関するルールを意味します。 APEXアプリケーションでは、このスタイルを使用でき、アプリケーション開発中にRESTタイプのHTTP / HTTPsリクエストを送信するプロセスを形成するための既製のテンプレートを提供します。その結果、この種のタスクを起動するためにアプリケーションをすばやく持ち上げ、起動用のデータをデータベースから直接取得するようになり、選択できる可能性があり、起動パラメーターの入力エラーの可能性がなくなりました。 Jenkinsでタスクを起動するためのパラメーターの送信は、POSTリクエストによって実行されます。その本文には、必要なパラメーターを含むJSONが含まれています。

最終的に


私たちの部門でのアプリケーションデータの統合は優れた結果を示しました。Jenkinsと直接作業する必要がない人々にツールキットを理解させることはもはや必要ありません。カスタム権利マトリックスを使用して、Jenkinsレベルでのタスクの実行を区別し、制御を残すことができたため、ユーザーはタスクを作成できましたが、編集することはできませんでした。

この実験は成功したと認識されたため、開発のアイデアは、Jenkinsや私たちと連携する他のアプリケーションとのやり取りを簡素化するための独自のオープンソースフレームワークを作成する形で現れました。しかし、これはまったく別の話であり、今後の記事で紹介しようと思います。
