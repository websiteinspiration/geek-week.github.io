製造中のマイクロコントローラーのプロジェクトにC ++を使用してから5年

この記事では、マイクロコントローラーのプロジェクトの管理からCからC ++に5年間以上働いていた会社をどのようにして移したのか、そしてそれから何が起こったのか（ネタバレ：すべてが悪い）について説明します。

あなた自身について少し


Pascalの学校での経験しかなく、Cマイクロコントローラーで書き始めた後、アセンブラーを研究し、さまざまなマイクロコントローラーアーキテクチャーとその周辺機器の研究に約3年間費やしました。次に、C＃とC ++での実際の作業の経験があり、それらの並行研究には数年かかりました。この期間の後、私は再びそして長い間、マイクロコントローラーのプログラミングに戻りました。すでに実際のプロジェクトに取り組むために必要な理論的基礎を持っています。

1年目


Cの手続き型スタイルには何も反対していませんでしたが、実際のプロジェクトで実際に実践を始めた企業は、「オブジェクト指向スタイルのCプログラミング」を使用していました。こんな感じでした。

typedef const struct _uart_init {
	USART_TypeDef *USARTx;
	uint32_t baudrate;
	...	
} uart_cfg_t;

int uart_init (uart_cfg_t *cfg);
int uart_start_tx (int fd, void *d, uint16_t l);
int uart_tx (int fd, void *d, uint16_t l, uint32_t timeout);

このアプローチには、次の利点があります。

  1. コードは引き続きCコードであり、次の利点があります。
    • 「オブジェクト」を制御する方が簡単です。誰が、どこで、何が、どのシーケンスで発生するかを追跡するのが簡単だからです（割り込みを除いて、この記事ではこれについてではありません）。
    • 「オブジェクトへのポインタ」を格納するには、返されたfdを覚えておいてください
    • 「オブジェクト」が削除された場合、それを使用しようとすると、関数の戻り値で対応するエラーを受け取ります。
  2. そこで使用されるHALを介したそのようなオブジェクトの抽象化により、独自の初期化構造からタスク用にカスタマイズ可能なオブジェクトを書き込むことが可能になりました（HAL機能がない場合は、「オブジェクト」内のレジスタへのアクセスを隠すことができます）。

インフラストラクチャ全体を構築するときにC ++が選択されなかった理由について尋ねられたとき、彼らは次のような答えを出しました。「まあ、C ++は強力な追加コスト、制御されないメモリコスト、そして実行可能なかさばる実行可能ファームウェアファイルにつながります。」おそらく彼らは正しかった。実際、設計の最初の段階では、GCC 3.0.5しかありませんでした。これは、C ++に対する特別な親しみやすさでは光りませんでした（QNX6でプログラムを書くには、まだそれを使用する必要があります）。 constexprとC ++ 11/14はありませんでした。グローバルオブジェクトを作成できます。グローバルオブジェクトは、本質的には.data領域のデータであり、コンパイル段階で計算され、そのためのメソッドです。

質問には、レジスタに書き込んでみませんか。「オブジェクト」を使用すると、同じタイプのアプリケーションを「1日で」設定できるという明確な答えが得られました。

これらすべてを理解し、今ではC ++はGCC 3.0.5と同じではないことに気づき、C ++の機能の主要部分を書き直し始めました。はじめに、マイクロコントローラーのハードウェア周辺機器を操作し、次に外部デバイスの周辺機器を操作します。実際、これは当時利用可能なシェルよりも便利なシェルにすぎませんでした。

2年目と3年目


私はC ++プロジェクトに必要なすべてのものを書き直し、C ++ですぐに新しいモジュールを書き続けました。ただし、これらはCの単なるシェルでした。C++を十分に使用していないことに気づき、テンプレート、ヘッダーのみのクラス、constexprなどの強みを使い始めました。すべてが順調でした。

4年目と5年目


  • すべてのオブジェクトはグローバルであり、（プロジェクトのアーキテクチャに従って）コンパイル段階で互いにリンクします。
  • すべてのオブジェクトには、レイアウトの段階でメモリが割り当てられます。
  • 各ピンのクラスオブジェクトごと。
  • すべてのピンを1つのメソッドで初期化するためにカプセル化するオブジェクト。
  • ハードウェアバス上にあるすべてのオブジェクトをカプセル化するRCCコントロールオブジェクト。
  • CAN <->お客様のプロトコルによるRS485コンバータプロジェクトには60個のオブジェクトが含まれています。
  • あるオブジェクトのHALまたはクラスレベルで何かがそのレベルにある場合、「問題を修正する」だけでなく、この修正がこのモジュールのすべての可能な構成で機能するように修正する方法も考慮する必要があります。 ;
  • 使用されるテンプレートとconstexprは、最終的なファームウェアのマップ、asm、binファイルを表示する前に（またはマイクロコントローラーでデバッグを起動する前に）計算することはできません。
  • テンプレートにエラーがある場合、GCCからのプロジェクト構成の3分の1の長さのメッセージが出力されます。そこから何かを読んで理解することは別の成果です。
