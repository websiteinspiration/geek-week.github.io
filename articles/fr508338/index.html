Arrêtez d'utiliser l'instruction if-else





Vous avez regardé d'innombrables tutoriels utilisant des instructions If-else. Vous avez probablement également lu des livres de programmation faisant la promotion de l'utilisation de If-else comme méthode de branchement réelle.



C'est peut-être même votre mode par défaut pour utiliser If-else. Mais finissons-en maintenant en remplaçant If-else par des objets d'état.



Notez que cette approche peut être utilisée si vous écrivez une classe avec des méthodes qui doivent changer ses implémentations en fonction de l'état actuel. Vous adopteriez une approche différente si vous ne vous occupiez pas de l'état changeant de l'objet.



Même si vous avez entendu parler du modèle d'état, vous vous demandez peut-être comment il est implémenté dans le code prêt à la production.



Pour ceux qui sont encore dans le noir, voici une très brève introduction.

Il est révolu le temps où le code ressemblait à celui illustré ci-dessous.







Vous avez bien sûr déjà écrit des branches plus complexes. Il y a quelques années, j'en étais sûr.



La logique de branchement donnée ci-dessus n'est même pas très compliquée - mais essayez d'ajouter de nouvelles conditions, et vous verrez que cela devient parfois plus compliqué.



De plus, si vous pensez que créer de nouvelles classes au lieu d'utiliser simplement des instructions de branchement semble ennuyeux, attendez de le voir en action. Cette méthode est concise et élégante.



Encore mieux, cela rendra votre base de code plus SOLIDE, à l'exception de la lettre "D."



"Eh bien, je pensais que si le reste est mauvais, montrez-moi maintenant comment éviter le code de branchement désordonné"



Nous verrons comment je remplace If-else dans un code prêt pour la production. Il s'agit d'un exemple fictif, mais l'approche est la même que celle que j'ai utilisée dans les bases de code pour les gros clients.



Créons une classe très simple Booking



qui a plusieurs états. Il aura également deux méthodes publiques: Accept()



et Cancel()



.



J'ai dessiné un tableau au mieux de mes capacités, qui montre les différentes conditions dans lesquelles la réservation peut être située.







La refactorisation de la logique de branchement de notre code est un processus en trois étapes:



  1. Créer une classe d'état de base abstraite
  2. Implémentez chaque état en tant que classe distincte héritant de l'état de base
  3. Laisser une classe Booking



    avoir une méthode privée ou interne qui prend une classe d'état de base comme paramètre


la mise en oeuvre



Tout d'abord, nous avons besoin d'une classe d'état de base dont tous les États hériteront.







Notez que cette classe a également deux méthodes, Accept et Cancel - bien qu'ici, elles soient marquées comme internes.



De plus, l'état de base a une méthode "spéciale" EnterState(Booking booking)



. Elle est appelée chaque fois qu'un nouvel état est attribué à une réservation.



Deuxièmement, nous créons des classes distinctes pour chaque état que nous voulons représenter.







Notez comment chaque classe représente l'état décrit dans beau tableau ci-dessus. De plus, CanceledState ne permettra pas à notre réservation de passer à un nouvel état. Cette classe est très similaire dans l'esprit au modèle d'objet zéro.



Enfin, la classe de réservation elle-même.







Voyons, la classe de réservation délègue-t-elle simplement le pouvoir d'accepter et de refuser?



Cela nous permet d'éliminer la plupart de la logique conditionnelle et permet à chaque État de se concentrer uniquement sur ce qui est important pour lui-même - l'état actuel a également la possibilité de transférer la réservation vers un nouvel état.



Comment faire face aux nouvelles conditions



Si une nouvelle fonction était généralement implémentée à l'aide d'une vérification conditionnelle, vous pouvez maintenant simplement créer une nouvelle classe d'état.



C'est très simple. Vous n'avez plus à gérer la lourde instruction if-else.



Comment enregistrer un objet d'état dans une base de données



L'objet d'état n'a pas d'importance lors de l'enregistrement de l'objet, par exemple, dans une base de données SQL ou NoSQL. Il est seulement important de connaître l'état de l'objet et comment il doit être affiché dans la colonne.

Vous pouvez faire correspondre l'état avec un nom, une énumération ou un entier lisible. Tout cela vous convient, tant que vous disposez d'un moyen de reconvertir la valeur stockée en objet d'état.
