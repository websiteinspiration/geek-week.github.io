インポートの大きな違い：Typescriptでのインポートによる不確実性の明確化

記事の翻訳は、コース「React.js Developer」の開始前に準備されました。


私はかなり前からtypescriptを使用しており、モジュールの整理と設定のアドバイスに多くの問題を抱えていました。名前空間 import * as React from 'react'、esModuleInteropなどしたがって、なぜすべての誇大広告が発生したのかを理解しましょう。

名前空間についてはtypescriptのモジュラーシステムとしては触れません。アイデアが（特に現在の開発ベクトルを考慮して）最良ではなかったため、現在誰も使用していないためです。

それで、彼らが現れる前の事はどうでしたesModuleInteropか？babelやブラウザとほぼ同じモジュールがあり、名前付きのインポート/エクスポートもありました。ただし、デフォルトでのエクスポートとインポートの問題では、typescriptには独自のオプションがありました：（import * as React from 'react'代わりにimport React from 'react'）を記述する必要があり、もちろん、ここでは、反応だけでなく、からのすべてのデフォルトのインポートについても話していますcommonjs。どうやってそうなった？

これを理解するために、commonjsとモジュールの一部のパターン間の互換性がどのように機能するかを見てみましょうes6。たとえば、キーとしてエクスポートするモジュールがfooありbarます。

module.exports = { foo, bar }

requireとdestructuringを使用してインポートできます：

const { foo, bar } = require('my-module')

名前付きインポートを使用して同じ原則を適用します（ただし、正直なところ、これは構造化されていません）。

import { foo, bar } from 'my-module'

ただし、commonjsのより一般的なパターンはconst myModule = require('my-module')（まだ構造化されていないため）ですが、どのように実行しますes6か？

インポート用の仕様を開発する際es6、重要な側面の1つはとの互換性でした。これはcommonjs、commonjs多くのコードがすでにその上に記述されているためです。これは、デフォルトのインポートとエクスポートが表示された方法です。はい、唯一の目標はとの互換性を確保することでした。これにより、まったく同じ結果をcommonjs記述 import myModule from 'my-moduleして取得できました。ただし、仕様からはこれは明白ではなく、互換性の実装はトランスパイラーの開発者の特権でした。そして、ここで大きな分裂が起こりました： import React from 'react'またはimport * as React from 'react'-それが問題です。

typescriptが後者を選択したのはなぜですか？トランスパイラー開発者の代わりに自分を置き、インポートをからes6に転送する最も簡単な方法は何commonjsですか？次のインポートとエクスポートのセットがあるとします。

export const foo = 1
export const bar = 2
export default () => {}
import { foo } from 'module'
import func from 'module'`

したがって、デフォルトでは、エクスポート用jsのキーdefaultを持つオブジェクトを使用します。

module.exports = {
  foo: 1,
  bar: 2,
  default: () => {}
}
const module = require('module')
const foo = module.foo
const func = module.default

クールですが、互換性についてはどうですか？デフォルトでインポートすると、名前の付いたフィールドが使用されることになります。その場合、default作成時にimport React from 'react'それは意味const { default: React } = require('react')しますが、機能しません。次に、アスタリスクを付けてインポートを使用してみてください。これで、ユーザーはimport * as React from 'react'コンテンツにアクセスするために書く必要がありますmodule.exports。

ただし、とは意味的に異なりcommonjsます。
Commonjs通常のJavaScriptのようなものでした。関数とオブジェクトだけで、必要ありません。一方、インポートes6では、require現在は仕様の一部であるため、myModuleこの場合、それは通常のjavascriptオブジェクトではなく、名前空間と呼ばれるもの（typescriptの名前空間と混同しないでください）であり、それに応じて特定のプロパティがあります。それらの1つは、名前空間を呼び出せないことです。そして、ここで問題は何ですか、あなたは尋ねるかもしれませんか？1つの関数をエクスポートとして、

ES6モジュールの仕様に準拠した環境でこれを実行しようとすると、エラーが発生します。

これは、名前空間がJavaScriptオブジェクトと同じではなく、各es6エクスポートを格納する個別の構造だからです。

しかし、Babelはそれを正しく理解し、私たちがimport React from 'react「互換性のあるオプションを提供して、それが機能するようになりました。 transpilationの間に、彼マークで特別なフラグを持つ各ES6モジュールmodule.exports、私たちはフラグがtrueの場合、それが返されることを理解していることmodule.exports、そしてそれがfalseの場合（それがライブラリである場合、たとえば、commonjs転置されていない）、その後、我々は現在のエクスポートをラップする必要があります{ default: export }のでその毎回使用できますdefault（こちらをご覧ください）。

Typescriptはアスタリスクを使用してインポートを行いましたが、最終的にあきらめてオプションを追加しましたesModuleInteropコンパイラに。一般に、このオプションはbabelと同じimport React from 'react'働きをします。これを有効にすると、通常のインポートをとして記述でき、typescriptがすべてを理解します。

問題は、新しいプロジェクト（実行時tsc --init）ではデフォルトで有効になっていますが、後方互換性がないため、（TypeScript 3にアップグレードしても）既存のプロジェクトでは機能しません。したがって、不要なアスタリスクのインポートをデフォルトのインポートに書き換える必要があります。これはまだ名前付きエクスポートのコレクションであるため、Reactはこれを通常どおり処理しますが、たとえば名前空間の呼び出しではこれを処理しません。ただし、エクスポートタイピングですべてが順調である場合も心配しないでください（ほとんどの場合、すべてが順調です。それらの多くは自動的に修正されるためです。）、TypeScript 3では、アスタリスク付きのインポートを標準にすばやく変換できます。

したがって、私はこのオプションの使用をesModuleInterop強く推奨します。これは、コードを少なくして読みやすくするだけでなく（たとえば、これらは単なる単語ではないため、ロールアップでは、このようなアスタリスクを使用したインポートを使用できないため）、コミュニティ間の不一致も減少するからです。 typescriptとbabel。

注意：以前はenableSyntheticDefaultImports、デフォルトで正しくないインポートについて不満を述べようとしたときにコンパイラをシャットダウンするオプションがあったため、commonjs（たとえば、WebpackDefaultImportPlugin）、しかし、たとえば、テストがある場合でも、そのような互換性を確保する必要があるため、問題がありました。esModuleInteropターゲットが<=ES5の場合にのみ、デフォルトで合成インポートが含まれることに注意してください。したがって、このオプションを有効にしてもコンパイラーが不平を言うimport React場合は、追跡している目的を理解してください。デフォルトのインポートを含めることがオプションになるでしょう（または、vscode / webstormを再起動してください）。
