さらに多くのオプションがあります。まず、ゲーム内メニューが表示され、画像のスケールを制御できるようになりました。例えば：

var scaleValue=0.5;
var scale = 'scale('+scaleValue+')';
document.body.style.webkitTransform = document.body.style.msTransform = document.body.style.transform = scale;

これにより、使用可能な任意のサイズで画像をブラウザウィンドウに合わせることができます。この関数は、ウィンドウのサイズ変更イベントと連動して機能し、必要なスケールは事前に計算されます。後者は手動で調整することもできます。以前は、私のコンピューターではこのゲームが小さすぎ、電話では大きすぎてスクロールバーが表示されました。最後に、この問題は解決されました。確かに、おそらく、おそらく、ゲームの2番目のインターフェイスを固定する必要があります。ポートレートのインターフェイスは、モバイルデバイスの場合、特に電話の場合...電話の画面の大部分を占めるブラウザーの行のサイズも適切ではありません。ただし、ゲームを全画面に拡大するためのボタンがあります。

技術部


すでに述べたように、ゲームはサードパーティのライブラリを使用せずに純粋なJavaScriptで記述されています（JQueryも使用されていません）。キャンバスタグにインターフェイスを表示する機能を実装するライブラリはmineだけです。実際、すべてのゲームコンテンツはインターフェイスの一部です。後者では裏付けのあるメニューというものがあります。たとえば、同じ名前の1つのボタンで構成される「終了」メニューのゲームテーブル自体を「背景」として使用できます。そして、このメニューを表示して、テーブルとボタンをキャンバスに表示するだけで十分です。さらに、ボタンは基板上にある必要はありません。近くに配置することもできます。あるいは、例えば、基質は全くないかもしれない。さらに、メニューごとに、ボタンではない関連するスプライトまたは「サテライト」を表示できます。ボタンと同様に、相対座標を設定します。

したがって、2つの下位層のみが永続的にオンのままになります。 m3d.ui.initHtmlで指定されたすべての要素は、resize関数によって処理され、ブラウザウィンドウのサイズが変更されると、それに合わせて調整されます。

アニメーションエンジンも独自のものです。その機能は、与えられた時間の間、与えられた初期値から最終値までパラメータを変更することです。これは、たとえば、2つの軸に沿った座標（3Dの場合は3つに沿った）と回転角度になります。一般に、1つのアニメーションのパラメーターの数に制限はなく、何かを変更するための式を考え出すことができます。最後に、スプライトが配置されているコールバック関数では、すべてのパラメーターの現在の値が、システムの処理能力に依存する頻度で送信されます。 Fpsは低下する可能性がありますが、アニメーションの合計時間は変わりません。上部のキャンバスは、クリックイベント（およびモバイルデバイスの手押し車）を受け入れ、インターフェイス上のどのボタンを押す必要があるかを座標で決定します。

アニメーションはこのように機能します。移動するオブジェクトが最下層から削除され（オブジェクトはオフになる「サテライト」である必要があります）、インターフェース全体、つまり開いているすべての「サブストレートのあるメニュー」が再描画されます（これにより、サテライトはなくなり、削除されます）、オブジェクトそれは上層に上昇し、そこで移動し、その後消えて最終的に下層に落下し、そこで衛星としての別の「基質のメニュー」に結合するか、さらには相対座標が異なります。最下層のメニューが再描画されます。レイヤーのコンテンツを組み合わせる機能はブラウザに任されていますが、これはネイティブで行われます。ゲーム中のDOMはまったく変化せず、実際には常に2つのキャンバスが重ね合わされています。ブラウザーウィンドウのサイズを変更する場合を除き、すべてのコンテンツが既に新しい縮尺で1回再描画されます。
