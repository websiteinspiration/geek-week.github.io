慣用句


これらの考慮事項は私を慣用的な囲碁に導いた。何かを「慣用的」と呼ぶ場合、これはある時間の特定のスタイルに対応します。慣用的ではないものは、支配的なスタイルに対応していません。それは流行ではありません。

さらに重要なのは、誰かのコードが慣用的ではないと私たちが言うとき、これは理由を説明していません。なぜ慣用的ではないのですか？答えは辞書に載っています。

イディオム（n。）：スピーチ革命。全体として使用され、それ以上分解されず、通常、それ自体の中で順列を許可しません。

イディオムは一般的な意味の特徴です。本はあなたに慣用的な囲碁を教えません;それはあなたがコミュニティーの一部になるときだけ知られています。

慣用的な囲碁のマントラはしばしば制限的であるため、心配しています。彼女は言う：「あなたは私たちと一緒に座ることはできません。」他の誰かの作品を「慣用的ではない」と批判するとき、それは私たちが意味することではありませんか？彼らはそれを間違った。これは正しくありません。これは時代のスタイルに合わない。

慣用的なGoは、良いコードの書き方を教えるには適していないと思います。本質的には、何か間違ったことをしたことを人々に伝えることを意味するからです。彼がこのアドバイスを受けたいと思っている瞬間に人を押しのけないようなアドバイスをする方が良いです。

パッケージレベルの状態を回避する


明示的は暗黙的よりも優れています。

禅のPython、エントリ2

ここでは、私の意見では、ピーターズは事実を守るというよりは夢を見る。 Pythonでは、多くは明示的ではありません。デコレータ、dunderメソッドなどです。間違いなく、これらは強力なツールであり、それらには理由があります。特に複雑な各機能の実装に関して、誰かが働いた。ただし、このような機能を積極的に使用すると、コードを読み取るときに操作のコストを評価することが難しくなります。

さいわい、Goプログラマはオプションでコードを明示的にすることができます。おそらくあなたにとって、顕現は官僚主義と冗長性と同義かもしれませんが、それは表面的な解釈です。構文だけに焦点を当て、行の長さと式へのDRY原則の適用に注意を払うのは誤りです。接続性と状態の観点から明示性を提供することは、私にとってより重要なようです。

接続性は、一方が他方に依存していることを示す尺度です。一方が他方と密接に関連している場合、両方が一緒に移動します。一方に影響を与えるアクションは、もう一方に直接反映されます。すべての車が一緒に接続されている、またはむしろ接続されている列車を想像してみてください。蒸気機関車が行くところに車があります。

接続性は、凝集性-凝集性という用語でも説明できます。これは、一方がもう一方にどれだけ属しているかの尺度です。はんだ付けされたチームでは、すべての参加者は、特別に作成されたかのように、お互いにとても適しています。

一貫性が重要なのはなぜですか？トレインの場合と同様に、コードの一部を変更する必要がある場合は、密接に関連する残りのコードを変更する必要があります。たとえば、誰かが新しいバージョンのAPIをリリースし、コードがコンパイルされなくなったとします。

APIは避けられないバインディングのソースです。しかし、それはより陰湿な形で提示することができます。 APIの署名が変更された場合、APIとの間で転送されるデータも変更されることは誰でも知っています。それはすべて関数シグネチャに関するものです。私は1つの型の値を取り、他の型の値を返します。 APIが別の方法でデータの転送を開始した場合はどうなりますか？設定を変更していなくても、各API呼び出しの結果が前の呼び出しに依存している場合はどうなりますか？

これは状態と呼ばれ、状態管理はコンピュータサイエンスの問題です。

package counter

var count int

func Increment(n int) int {
        count += n
        return count
}

ここに単純なパッケージがありますcounter。カウンターを変更するには、を呼び出しIncrementます。ゼロの値でインクリメントすると、値を取得することもできます。

このコードをテストする必要があるとしましょう。各テスト後にカウンターをリセットする方法は？そして、テストを並行して実行したい場合、これをどのように行うことができますか？そして、プログラムでいくつかのカウンターを使用したい場合、成功しますか？

もちろん違います。明らかに、解決策は変数variableを型にカプセル化することです。

package counter

type Counter struct {
        count int
}

func (c *Counter) Increment(n int) int {
        c.count += n
        return c.count
}

ここで、説明されている問題がカウンターに限定されず、アプリケーションの主要なビジネスロジックに影響を与えることを想像してください。個別にテストできますか？並行してテストできますか？複数のインスタンスを同時に使用できますか？すべての質問に対する答えが「いいえ」の場合、理由はパケットレベルの状態です。

これらの条件を避けてください。パッケージ変数を使用するのではなく、フィールドとして必要な依存関係をタイプに提供することにより、接続性と悪夢のリモートアクションの数を減らします。

成功ではなく失敗の計画を立てる


バグを黙って渡さないでください。

Zen of Python、エントリ10

これは、サムライスタイルの例外処理を奨励する言語について言われています。勝利して戻ってくるか、まったく戻って来ないでください。例外に基づく言語では、関数は有効な結果のみを返します。関数がこれを実行できない場合、制御フローは完全に異なる方法で行われます。

明らかに、チェックされていない例外は安全でないプログラミングモデルです。どの式が例外をスローできるかわからない場合、エラーが発生したときに信頼できるコードをどのように書くことができますか？ Javaは、チェック済み例外の概念を使用してリスクを軽減しようとします。そして、私が知る限り、他の一般的な言語では、このソリューションの類似物はありません。多くの言語には例外があり、Java以外のすべての場所でチェックされません。

明らかに、Goは別の道を歩みました。Goプログラマーは、信頼できるプログラムは、成功したパスを処理する前に障害を処理する部分で構成されていると信じています。言語がサーバー開発、マルチスレッドプログラムの作成、およびネットワーク経由で入力されるデータを処理するプログラムのために作成された場合、プログラマーは、予期しない損傷したデータ、タイムアウト、および接続障害の処理に集中する必要があります。もちろん、信頼できる製品を作りたいのなら。

エラーは明示的に処理する必要があると思います。これが言語の主な価値です。

ピーターバーゴン、GoTime＃91

私はピーターの言葉に参加します、彼らはこの記事の執筆への原動力となりました。Goの成功は明示的なエラー処理にあると私は信じています。プログラマーは主にクラッシュの可能性について考えます。まず、「もしも」のような問題を解決します。その結果、プログラムの実行中にエラーが発生するのではなく、コードの作成段階でエラーが処理されるプログラムが得られます。

パッケージAPIの動作をブロックするテストを作成する


この記事ではテストについて触れないことを望んでいたかもしれません。申し訳ありませんが、別の機会に。

テストは、プログラムが何をするか、何をしないかについての合意です。単体テストでは、パッケージレベルでAPIの動作をブロックする必要があります。テストは、パッケージが約束することをコード形式で記述します。入力変換ごとにユニットテストがある場合、ドキュメントではなく、コードの形式で、コードが何を行うかについての合意を定義しています。

この合意の承認は、テストを書くのと同じくらい簡単です。どの段階でも、変更前に人々が依存していた動作は、変更後も引き続き機能することを高い自信を持って述べることができます。

テストはAPIの動作をブロックします。パブリックAPIを追加、変更、または削除する変更には、テストの変更が含まれている必要があります。
