Linux仮想インターフェースの紹介：トンネル

Linuxは多くのタイプのトンネルをサポートしています。これは、テクノロジーの違いを理解し、特定の状況でどのトンネルを使用するのが適切であるかを理解するのが難しいと思う初心者を混乱させます。本日の翻訳は、Linuxカーネルの一般的に使用されているトンネリングインターフェースの概要です。 Linuxで使用するためのトンネルとオプションの一般的な機能のみを考慮して、このトピックについて深く掘り下げることはしません。 この資料の作成者は、ここで説明する内容は、コンピュータネットワークの管理に関係するすべての人にとって興味深いものになると考えています。トンネルインターフェースのリストと特定の構成に関する参照情報は、iproute2コマンドを使用して取得できます。



ip link help

ここでは、IPIP、SIT、ip6tnl、VTIおよびVTI6、GREおよびGRETAP、GRE6およびGRE6TAP、FOU、GUE、GENEVE、ERSPANおよびIP6ERSPANの一般的に使用されるインターフェイスについて説明します。

この記事を読んだ後、これらのインターフェースの機能について学び、それらの違いを見つけます。それらを作成する方法を学び、それらが最もよく使用される状況について学びます。

IPIP


IPIPトンネルは、その名前が示すように、IP over IPモード（RFC 2003）で動作するトンネルです。IPIPトンネルパケットヘッダーは次のとおりです。


IPIPトンネルパケットヘッダー

これらのトンネルは通常、パブリックIPv4ネットワーク（インターネット）を介して2つの内部IPv4サブネットを接続するために使用されます。 IPIPを使用すると、システムに追加の負荷が最小限に抑えられますが、このようなトンネルで実行できるのはユニキャストデータ送信のみです。つまり、このようなトンネルを構築すると、マルチキャストデータ伝送に使用できなくなります。

IPIPトンネルは、IP over IPおよびMPLS over IPモードをサポートします。

モジュールがロードされるipipとき、またはIPIPデバイスが最初に作成されるときに、Linuxカーネルはtunl0、local=anyおよび名前空間を使用して各名前空間にデフォルトのデバイスを作成することに注意してくださいremote=any。カーネルは、IPIPパケットを受信すると、特定の場合にそれらをtunl0デフォルトのデバイスとして。これは、カーネルの属性local/remoteがパケットの送信元アドレスと宛先アドレスにより厳密に一致する別のデバイスをカーネルが見つけられないときに発生します。

IPIPトンネルを作成する方法は次のとおりです。

サーバーAで：

# ip link add name ipip0 type ipip local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR
# ip link set ipip0 up
# ip addr add INTERNAL_IPV4_ADDR/24 dev ipip0
Add a remote internal subnet route if the endpoints don't belong to the same subnet
# ip route add REMOTE_INTERNAL_SUBNET/24 dev ipip0

サーバーB：

# ip link add name ipip0 type ipip local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR
# ip link set ipip0 up
# ip addr add INTERNAL_IPV4_ADDR/24 dev ipip0
# ip route add REMOTE_INTERNAL_SUBNET/24 dev ipip0

この構成を使用する場合は、実際のデータと一致させる必要があることに注意してください。具体的には、LOCAL_IPv4_ADDR、REMOTE_IPv4_ADDR、INTERNAL_IPV4_ADDRおよびREMOTE_INTERNAL_SUBNETお使いの環境で使用されるアドレスに置き換える必要があります。他の構成例についても同様ですが、これについては後で説明します。

座る


SIT（Simple Internet Transition）は、IPv4プロトコルを使用してインターネット経由で分離されたIPv6ネットワークを接続することを主な目的とするトンネリング技術です。

当初、SITはIPv6 over IPv4トンネルモードでのみ機能しました。しかし、何年もの開発の間に、それはいくつかのより多くの体制の支持を得ました。特に、それはipip（同じことがIPIPトンネルで起こった）ip6ip、mplsipおよびany。

このモードはany、IPおよびIPv6トラフィックを処理するために使用され、状況によっては役立つ場合があります。 SITトンネルはISATAPもサポートします。ここでの例で、この技術を使用して。

SITパケットのヘッダーは以下のとおりです。


SITトンネルパッケージヘッダー

モジュールが読み込まれるsitと、Linuxカーネルがデフォルトのデバイスを作成しますsit0。

SITトンネルを作成する方法は次のとおりです（これらの手順はサーバーAとBで実行する必要があります）。

# ip link add name sit1 type sit local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR mode any
# ip link set sit1 up
# ip addr add INTERNAL_IPV4_ADDR/24 dev sit1

Ip6tnl


ip6tnlインターフェースは、IPv6モードのIPv4 / IPv6で動作します。これは、SITトンネルのIPv6バージョンに似ています。これは、ip6tnlパッケージヘッダーのようになります。


ip6tnlトンネルパケットヘッダーip6tnl

トンネルサポートモードip6ip6、ipip6およびany。モードipip6はIPv4 over IPv6スキームで表され、モードip6ip6はIPv6 over IPv6です。モードanyは両方のスキームをサポートします。

モジュールがロードされるip6tnlと、Linuxカーネルは名前でデフォルトのデバイスを作成しますip6tnl0。

ip6tnlトンネルを作成する方法は次のとおりです。

# ip link add name ipip6 type ip6tnl local LOCAL_IPv6_ADDR remote REMOTE_IPv6_ADDR mode any

VTIおよびVTI6


Linux VTI（仮想トンネルインターフェース）インターフェースは、Cisco VTIインターフェースおよびジュニパーのセキュアトンネル（st.xx）の実装に似ています。

このトンネリングドライバーは、IPカプセル化を実装します。これは、xfrmで使用して安全なトンネルを作成し、そのようなトンネル上でカーネルレベルのルーティングトンネルを使用できます。

一般に、VTIトンネルはIPIPまたはSITトンネルのように機能します。例外は、fwmarkとIPsecカプセル化/カプセル化解除を使用することです。

VTI6は、VTIに相当するIPv6です。

VTIトンネルを作成する方法は次のとおりです。

# ip link add name vti1 type vti key VTI_KEY local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR
# ip link set vti1 up
# ip addr add LOCAL_VIRTUAL_ADDR/24 dev vti1

# ip xfrm state add src LOCAL_IPv4_ADDR dst REMOTE_IPv4_ADDR spi SPI PROTO ALGR mode tunnel
# ip xfrm state add src REMOTE_IPv4_ADDR dst LOCAL_IPv4_ADDR spi SPI PROTO ALGR mode tunnel
# ip xfrm policy add dir in tmpl src REMOTE_IPv4_ADDR dst LOCAL_IPv4_ADDR PROTO mode tunnel mark VTI_KEY
# ip xfrm policy add dir out tmpl src LOCAL_IPv4_ADDR dst REMOTE_IPv4_ADDR PROTO mode tunnel mark VTI_KEY

さらに、libsecwanまたはstrongSwanを使用してIPsecを構成できます。

GREおよびGRETAP


GRE（Generic Routing Encapsulation）テクノロジは、RFC 2784で説明されています。GREトンネリングは、内部および外部IPパケットのヘッダーの間に追加のGREヘッダーを追加します。

理論的には、GREは、有効なイーサネットタイプを持つ任意のレイヤ3プロトコルのパケットをカプセル化できます。これにより、GREテクノロジーは、IPパケットのカプセル化のみをサポートするIPIPテクノロジーと区別されます。GREテクノロジーを使用した場合のパッケージヘッダーは次のようになります。


GREトンネルパケットヘッダー

GREトンネルはマルチキャストを許可し、IPv6をサポートすることに注意してください。

モジュールが読み込まれると、greLinuxカーネルがデフォルトのデバイスを作成しますgre0。

GREトンネルを作成する方法は次のとおりです。

# ip link add name gre1 type gre local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR [seq] key KEY

GREトンネルはOSIレイヤー3で動作しますが、GRETAPトンネルはOSIレイヤー2で動作します。これは、それぞれのパケットの内部ヘッダーの1つがイーサネットヘッダーであることを意味します。


GRETAPトンネルパッケージヘッダーGRETAPトンネル

を作成する方法は次のとおりです。

# ip link add name gretap1 type gretap local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR

GRE6およびGRE6TAP


GRE6は、GREに相当するIPv6です。GRE6トンネルを使用すると、レイヤー3プロトコルをIPv6でカプセル化できます。GRE6パッケージヘッダーは次のようになります。


GRE6トンネルパケットヘッダーGRE6TAP

トンネルは、 GRETAPトンネルと同様に、内部パケットヘッダーの中にイーサネットヘッダーがあります。


GRE6TAPトンネルパッケージヘッダー

GREトンネルを作成する方法は次のとおりです。

# ip link add name gre1 type gre6 local LOCAL_IPv6_ADDR remote REMOTE_IPv6_ADDR
# ip link add name gretap1 type gretap6 local LOCAL_IPv6_ADDR remote REMOTE_IPv6_ADDR

ふう


トンネリングは、ネットワークスタックのさまざまなレベルで実行できます。 IPIP、SIT、およびGREトンネルはIP層に存在します。また、FOUトンネル（「foo over UDP」スキームに従って配置されている）は、UDPレベルで動作します。

IPトンネリングではなくUDPトンネリングを使用することには、いくつかの利点があります。実際、UDPプロトコルは既存のハードウェアインフラストラクチャで動作します。

たとえば、これはネットワークカードのRSS、スイッチのECMPです。これらは中央プロセッサの関与なしにチェックサムを計算するためのテクノロジーです。開発者に適切なFOU パッチを適用すると、SITおよびIPIPのパフォーマンスが大幅に向上します。

現在、FOUトンネルは、IPIP、SIT、およびGREに基づくプロトコルカプセル化をサポートしています。FOUパッケージのヘッダーは次のようになります。


FOUトンネルパッケージヘッダーFOUトンネル

を作成する方法は次のとおりです。

# ip fou add port 5555 ipproto 4
# ip link add name tun1 type ipip remote 192.168.1.1 local 192.168.1.2 ttl 225 encap fou encap-sport auto encap-dport 5555

最初のコマンドは、5555にバインドされたIPIPの受信FOUポートを構成します。GREを使用するには、を使用する必要がありますipproto 47。2番目のコマンドtun1は、FOUカプセル化用に設計された新しい仮想IPIP（）インターフェースを設定します。宛先ポートは5555です。FOU

トンネルはRed Hat Enterprise Linuxではサポートされていないことに注意してください。

ゲ


別のタイプのUDPトンネリングは、GUE（Generic UDP Encapsulation）テクノロジーによって導入されています。FOUとGUEの違いは、GUEには独自のヘッダーがあり、ヘッダーにはプロトコル情報とその他のデータが含まれていることです。

GUEトンネルは現在、内部IPIP、SIT、およびGREカプセル化をサポートしています。GUEパッケージのヘッダーは次のようになります。


GUEトンネルパッケージヘッダーGUEトンネル

を作成する方法は次のとおりです。

# ip fou add port 5555 gue
# ip link add name tun1 type ipip remote 192.168.1.1 local 192.168.1.2 ttl 225 encap gue encap-sport auto encap-dport 5555

これらのコマンドにより、5555番号にバインドされたIPIPの受信GUEポートと、GUEカプセル化用に構成されたIPIPトンネルが作成されます。

GUEトンネルはRed Hat Enterprise Linuxではサポートされていません。

ジュネーブ


GENEVEトンネル（汎用ネットワーク仮想化カプセル化）は、XLAN、NVGRE、およびSTTのすべての機能をサポートしています。 GENEVEテクノロジーは、これら3つのテクノロジーの識別された制限を回避するように設計されています。多くの人は、このテクノロジーが長期的に見て、これら3つの古いフォーマットを完全に置き換えることができると信じています。 GENEVEトンネルパケットヘッダーは次のようになります。


GENEVEトンネルパケットヘッダー

このヘッダーは、 VXLANパケットのヘッダーに似ています。2つの主な違いは、GENEVEヘッダーがより柔軟であることです。Type-Length-Value（TLV）フィールドを使用してヘッダーを拡張することにより、新しい機能を簡単に実装できます。

GENEVEの詳細については、こちらとこちらをご覧ください。

GENEVEは、 Open Virtual Network（OVN）SDNソリューションで標準のカプセル化ツールとして使用されます。GENEVEトンネルを作成する方法は次のとおりです。

# ip link add name geneve0 type geneve id VNI remote REMOTE_IPv4_ADDR

ERSPANおよびIP6ERSPAN


カプセル化リモートスイッチドポートアナライザー（ERSPAN）テクノロジは、GREカプセル化を使用して、ポートミラーリングの基本機能をレイヤー2からレイヤー3に拡張します。これにより、ルーティングされたIPネットワークを介してミラーリングされたトラフィックを転送できます。 ERSPANパッケージヘッダーは次のようになります。


ERSPANトンネルパッケージヘッダーERSPAN

トンネルを使用すると、LinuxホストはERSPANトラフィックソースとして機能し、ミラーリングされたERSPANトラフィックをリモートホストまたは宛先のERSPANに送信できます。これにより、Ciscoスイッチまたはその他のデバイスが生成するERSPANパケットを受信して​​処理します。 ERSPAN。同様のシステムを使用して、ネットワークを分析および診断し、悪意のあるトラフィックを検出できます。

Linuxは現在、ERSPANの2つのバージョン-v1（タイプII）およびv2（タイプIII）のほとんどの機能をサポートしています。

ERSPANトンネルを作成する方法は次のとおりです。

# ip link add dev erspan1 type erspan local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR seq key KEY erspan_ver 1 erspan IDX

これを行うこともできます：

# ip link add dev erspan1 type erspan local LOCAL_IPv4_ADDR remote REMOTE_IPv4_ADDR seq key KEY erspan_ver 2 erspan_dir DIRECTION erspan_hwid HWID

トラフィックを監視するtcフィルターを追加します。

# tc qdisc add dev MONITOR_DEV handle ffff: ingress
# tc filter add dev MONITOR_DEV parent ffff: matchall skip_hw action mirred egress mirror dev erspan1
