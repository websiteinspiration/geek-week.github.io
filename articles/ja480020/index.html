Blue-Green deployment?


Blue-Greenデプロイメントは、エンドクライアントが変更を認識しないようにアプリケーションを本番環境に配信する方法です。つまり、ダウンタイムなしでアプリケーションをデプロイします。

従来のBG配置スキームは、次の図に示すようなものです。



  • 最初は、コード、アプリケーション、プロジェクトがまったく同じ2つの物理サーバーがあり、ルーター（バランサー）があります。
  • ルーターは最初にすべての要求をいずれかのサーバーに送信します（緑）。
  • 再度リリースする必要がある現時点では、プロジェクト全体が現在要求を処理していない別のサーバー（青）で更新されています。

免責事項：以下の例は、私が使用するユーティリティ/方法を示しています。同様の機能を持つ代替手段をすべて使用できます。

ほとんどの例は、PHPとDockerを使用したWeb開発（これは驚きです）と何らかの形で重複します。

以下の段落には、特定の例での要因の使用に関する簡単で実用的な説明があります。このトピックについてさらに理論を知りたい場合は、上記のソースへのリンクを参照してください。

1.コードベース


FTPとFileZillaを使用して、一度に1つずつサーバーにファイルをアップロードします。本番サーバー以外の場所にはコードを保存しないでください。

プロジェクトには常に単一のコードベースが必要です。つまり、すべてのコードは1つのGitリポジトリから取得されます。サーバー（本番、ステージング、test1、test2 ...）は、1つの共通リポジトリーのブランチからのコードを使用します。このようにして、コードの一貫性を実現します。

2.依存関係


フォルダが含まれるすべてのライブラリをプロジェクトのルートに直接ダウンロードします。ライブラリの現在のバージョンが含まれているフォルダーに新しいコードを転送するだけで、更新を実行できます。必要なすべてのユーティリティを、別の20個のサービスが実行されているホストサーバーに直接配置します。

プロジェクトには常に明確に理解できる依存関係のリストが必要です（私は環境を依存関係としても理解しています）。すべての依存関係は明示的に定義および分離する必要があります。
例としてComposerとDockerを取り上げます。

Composerは、ライブラリをPHPにインストールできるようにするパッケージマネージャーです。 Composerでは、バージョンを厳密に指定することも厳密に指定しないこともでき、バージョンを明示的に定義することができます。サーバーは20の異なるプロジェクトを持つことができ、それぞれが他から独立したパッケージとライブラリの個人的なリストを持っています。

Dockerは、アプリケーションが実行される環境を定義および分離できるユーティリティです。したがって、composerと同様ですが、より徹底的に、アプリケーションが何と連携しているかを判断できます。PHPの特定のバージョンを選択し、プロジェクトが機能するために必要なパッケージのみを配置します。余分なものは何も追加しません。そして最も重要なことは、ホストマシンや他のプロジェクトのパッケージや環境と交差しないことです。つまり、Dockerを介して実行されるサーバー上のすべてのプロジェクトは、絶対に任意のパッケージセットと完全に異なる環境を使用できます。

3.設定


定数を含む構成をコードに直接格納します。テストサーバー用の個別の定数、本番用の個別の定数。 if elseコンストラクトを使用して、環境に応じたアプリケーションの作業をプロジェクトのビジネスロジックに直接結び付けます。

構成は、プロジェクトのデプロイメントで異なる必要がある唯一のものです。理想的には、構成は環境変数（env vars）を介して渡される必要があります。

つまり、いくつかの.config.prod .config.local構成ファイルを格納し、デプロイメント時にそれらを.config（アプリケーションがデータを読み取るメインの構成）に名前変更しても、この方法では適切ではありません。構成からすべてのアプリケーション開発者が公開して利用できるようになり、本番サーバーからのデータが侵害されます。すべての構成は、展開システム（CI / CD）に直接保存し、展開時に特定の環境に必要なさまざまな値を持つさまざまな環境で生成する必要があります。

4.サードパーティのサービス（バッキングサービス）


環境にしっかりと結び付け、特定の環境では同じサービスに異なる接続を使用します。

実際、このアイテムを利用できないと通常の構成データを作成できず、一般に構成の可能性がなくなるため、このアイテムは構成上のアイテムと強く交差します。

キューサーバー、データベース、キャッシュサービスなどの外部サービスへのすべての接続は、ローカル環境とサードパーティ/本番環境の両方で同じである必要があります。つまり、接続文字列を変更することで、アプリケーションコードを変更せずに、ベース＃1の呼び出しをベース＃2に置き換えることができます。または、例として前向きに考えて、サービスをスケールアップするときに、追加のキャッシュサーバーへの接続を特別な方法で指定する必要はありません。

5.アセンブリ、リリース、実装


サーバーにコードの最終バージョンのみを残し、リリースをロールバックする可能性はありません。ディスクスペースを詰まらせる必要はありません。エラーで本番用コードを実行できると考える人は誰でも悪いプログラマです！

展開のすべての段階は、それらの間で分割する必要があります。

ロールバックする機会があります。エラーが発生した場合に古いバージョンを復元できるように、アプリケーションの古いコピー（すでに組み立てられており、戦闘準備が整っている）にすばやくアクセスできるリリースを作成します。つまり、条件付きでリリースフォルダーと現在のフォルダーがあり、展開とビルドが正常に完了すると、現在のフォルダーは、リリース番号の条件付きの名前を持つリリース内の新しいリリースへのシンボリックリンクによってリンクされます。

ここでは、Blue-Greenデプロイメントを思い起こします。これにより、コードを切り替えるだけでなく、すべてのリソースを切り替えたり、すべてをロールバックする機能を備えた環境を切り替えることもできます。

6.プロセス


アプリケーション自体に直接アプリケーション状態データを格納します。アプリケーション自体のRAMでセッションを使用します。サードパーティのサービス間で可能な限り共有します。アプリケーションは1つのプロセスしか持つことができず、スケーラビリティを許可しないと想定します。

セッションに関しては、サードパーティのサービス（memcached、redis）によって制御されるキャッシュにのみデータを保存するので、20のアプリケーションプロセスが実行されている場合でも、それらのいずれかがキャッシュにアクセスすると、同じ状態のクライアントで引き続き作業できます。ユーザーは別のプロセスでアプリケーションを操作していました。このアプローチでは、使用するサードパーティサービスのコピーがいくつあっても、すべてが正常に機能し、データへのアクセスに問題がないことがわかります。

7.ポートのバインド


Webサーバーだけが、サードパーティのサービスと連携する方法を知っている必要があります。また、サードパーティのサービスをWebサーバー内で直接起動することをお勧めします。たとえば、ApacheのPHPモジュールとして。
すべてのサービスは、いくつかのアドレスとポート（localgost：5432、localhost：3000、nginx：80、php-fpm：9000）へのアクセスを通じて相互にアクセスできる必要があります。つまり、nginxからphp-としてアクセスできますfpm、postgres、php-fpmからpostgres、nginx、そして実際には各サービスから、別のサービスにアクセスできます。したがって、サービスの実行可能性は、別のサービスの実行可能性に関連付けられていません。

8.並行性


1つのプロセスで作業します。そうしないと、突然、複数のプロセスが互いにうまく連携できなくなります。

スケーリングの可能性を残します。これに最適なのは、Docker Swarmです。
Docker Swarmは、異なるマシン間でコンテナクラスタを作成し、同じマシン上の多数のコンテナを管理するためのツールです。

swarmを使用して、各プロセスに割り当てるリソースの数と開始する同じサービスのプロセスの数を決定できます。内部バランサーは、特定のポートにデータを取り込み、それをプロセスに自動的にプロキシします。したがって、サーバーの負荷が増加していることがわかったので、プロセスを追加して、特定のプロセスの負荷を減らすことができます。

9.使い捨て


キューを使用してプロセスとデータを操作しないでください。1つのプロセスを強制終了すると、アプリケーション全体の動作に影響を与えるはずです。1つのサービスが失敗すると、すべてが失敗します。

各プロセスとサービスはいつでもオフにすることができ、これは他のサービスに影響を与えないはずです（もちろん、サービスが別のサービスにアクセスできないということではなく、このサービスの後に他のサービスが切断されないということです）。すべてのプロセスは緩やかに終了する必要があるため、プロセスが完了するとデータは影響を受けず、次にシステムの電源がオンになったときに正しく動作します。つまり、異常終了が発生した場合でも、データが損傷することはありません（ここではトランザクションメカニズムが適切であり、データベース内のクエリはグループでのみ機能し、グループからのクエリが少なくとも1つ失敗または失敗した場合、グループからの他のリクエストは最終的に実行されません。実際には）。

10.アプリケーション開発/仕事のパリティ


アプリケーションの本番、ステージング、ローカルバージョンは異なる必要があります。本番環境では、Yii LiteフレームワークとローカルでYiiを使用しているため、製品でより高速に動作します。

実際、すべての展開とコードでの作業は、ほぼ同じ環境で行う必要があります（物理ハードウェアについては触れていません）。また、必要に応じて、従業員はコードを本番環境にデプロイできる必要があります。特別にトレーニングされた開発部門ではなく、その特別な力のおかげで本番環境でアプリケーションを引き上げることができます。

Dockerもこれに役立ちます。以前のすべてのポイントが確認された場合、dockerを使用すると、環境を本番環境とローカルマシンの両方にデプロイするプロセスで1つまたは2つのコマンドを入力できます。

11.ロギング


ログをファイルとobdに書き込みます！ログからのファイルとデータベースはクリーンではありません。9000ペットバイトのハードドライブと標準を購入するだけです。

すべてのログは、一連のイベントと見なされます。アプリケーション自体はログの処理に関与してはなりません。ログはstdoutで発行するか、udpなどのプロトコルを使用して送信する必要があります。これにより、ログを処理するアプリケーションで問題が発生しなくなります。グレイログはこれに適しています。すべてのudpログを受け入れるGraylog（このプロトコルは、正常なパケット受信に関する応答を待つ必要はありません）は、アプリケーションに干渉せず、ログの構造化と処理のみを扱います。アプリケーションロジックは、同様のアプローチで機能するように変更されません。

12.管理タスク


データbdとtdを更新するには、apiで個別に作成されたエンドポイントを使用します。これを2回続けて実装すると、すべてが複製される可能性があります。しかし、あなたは愚か者ではありません、あなたは2回押すことはありません、そして私たちは移行を必要としません。

すべての管理タスクは、リリースレベルで、すべてのコードと同じ環境で実行する必要があります。つまり、データベースの構造を変更する必要がある場合、視覚的なデータベース管理ツールを使用して列の名前を手動で変更したり、新しい列を追加したりすることはありません。そのような場合のために、個別のスクリプトを作成します。移行はどこでも、すべての環境で同じように実行され、共通で理解可能な結果が得られます。プロジェクトにデータを入力するなど、他のすべてのタスクには、同様の方法を適用する必要があります。

これで、任意のコマンドを使用できます。 （データベースとキャッシュを構成しなかったため、コマンドの半分はキャッシュとデータベースで動作するように設計されているため、正しく実行されないことに注意してください）。



11. 構成と12. アセンブリ、リリース、実装

この部分はBlue-Greenデプロイメントに専念したかったのですが、この記事では詳細すぎました。これについては別の記事を書きます。

一言で言えば、この概念はJenkinsやGitlab CIなどのCI / CDシステムに基づいて構築されています。どちらの場合も、特定の環境に関連付けられた環境変数を設定できます。したがって、この状況では、構成のあるアイテムが実行されます。

についてのポイントビルド、リリース、実行は、Pipelineと呼ばれる両方のユーティリティの組み込み関数によって解決されます。

パイプラインを使用すると、展開プロセスを多くの段階に分割して、アセンブリ、リリース、実行の段階を強調できます。また、Pipelineでは、バックアップを作成できます。無限の可能性を秘めたこのツール。

アプリケーションコードはGithubにあります。
このリポジトリのクローンを作成するときは、必ずサブモジュールを初期化してください。

PS：これらのアプローチはすべて、他のユーティリティやプログラミング言語で使用できます。主なことは、本質は変わらないということです。
