物語


誕生vi


Viコマンドと関数は、QEDエディターを始めとして、50年以上前から存在しています。ここに彼のタイムラインがあります：

  • 1966：バークレータイムシェアリングシステムのQED（Quick Editor）
  • 1969 : ( , )
  • 1969 : QED → ed AT&T
  • 1976 : ed → em («Editor for Mortals»)
  • 1976: em → ex («EXtended»)
  • 1977 : ex , vi —

QEDとexの マニュアルを読んだ場合、それらの類似点がいくつか見つかるかもしれません。どちらのエディターも同様の文法を使用して、行範囲を示し、操作します。

QED edやemなどのエディターはプリント端末用に設計されており、ほとんどの場合、モデムが接続された通常の電気タイプライターでした。このような「フルコピー」端末はコマンドを紙に表示し、入力後に修正を加えることは明らかに不可能でした。したがって、編集プロセスは、紙に書かれたユーザーコマンドを手動で編集し、それらを再入力することで構成されていました。

1976年までに、ビデオ端末、たとえばADM-3Aが登場しました。 Exエディターに「オープンモード」が追加され、単一のページ内でビデオ端末を介して編集できるようになりました。カーソルを使用して端末の行を方向付けるために、視覚モードが追加されました。ビジュアルモードは「vi」コマンドによってアクティブ化され、画面下部のコマンドラインの位置を維持しながら、画面に表示されるファイルを常に更新しました。興味深い事実：ADM-3Aの矢印のh、j、k、lキーに矢印がプロットされ、カーソルをviに移動できました。

このEdからex / viへの移行について詳しくは、Bill Joyへのインタビューをご覧ください。その中で、彼は彼がどのようにex / viを作成したか、そして最終的に彼を失望させたいくつかのことについて話します。

クラシックviは単なる変更です。どちらも同じバイナリファイルで表され、実行可能ファイルの名前に応じて、exモードとviモードの両方で実行できます。このストーリー全体の遺産は、ex / viを使用すると「展開」され、システムリソースをほとんど必要とせず、限られた帯域幅の条件で動作できることです。また、ほとんどの既存のシステムで使用でき、POSIXで完全に説明されています。

viからvim


edから派生したex / viエディターは、AT＆Tの知的財産でした。Unix以外のプラットフォームでviを使用するには、ソースコードベースが異なるクローンを作成する必要がありました。

それらの一部を次に示します。

  • nvi-1980（4BSDの場合）
  • calvin-DOSの1987
  • 下劣-1990年版DOS
  • stevie-Atari STの1987
  • elvis-1990年、Minixおよび386BSD
  • vim-Amigaの1991
  • viper-Emacsの1995
  • elwin-Windows用1995
  • レミー-2002 for Windows

リストの中央からのクローン、Vimに焦点を当てます。Bram Mulenaarは、Amigaでviを使用することを望んでおり、Atariとの移植とviクローンstevieの開発を開始しました。そして彼は自分のバージョンのポートを「Vi Imitation」と名付けました。このプロセスについて直接知りたい場合は、フリーソフトウェアマガジンのインタビューをご覧ください。

バージョン1.22では、Vimの名前は「Vi IMproved」に変更されました。これは、コピーがオリジナルよりも優れていることを示しています。以下は、いくつかの機能の説明を含む次のメジャーバージョンのグラフです。



各バージョンの詳細については、vim8などのヘルプが適切なオプションです。計画されている更新と既知のエラーのリストを表示するには、todo.txtを参照してください。

たとえば、8番目のバージョンには、NeoVimからのプロジェクトへの圧力のため、非同期ジョブのサポートが含まれていました。後者の開発者は、エディター内でWebスクリプトのデバッグとREPL を実行したいと考えていました。

一般的にVimはスーパーポータブルです。完全に異なるプラットフォームで動作するように、その存在の全歴史を適応させて、このエディターはコーディングの「簡単な」文化の枠組みの中にとどまることを余儀なくされました。 Vimは、OS / 390、Amiga、BeOSおよびBeBox、Macintosh Classic、Atari MiNT、MS-DOS、OS / 2、QNX、RISC-OS、BSD、Linux、OS X、VMSおよびMS-Windowsで動作します。あなたはどこでもVimに依存することができ、どんな機器を使ってもかまいません。

元のviパスの最後に、2002年に、ex / viソースコードがBSDフリーソフトウェアライセンスの下で公開されました。魔術師はex-vi.sourceforge.netで入手できます。

しかし、ビジネスに取り掛かりましょう。Vimの分析を始める前に、Vimが構成ファイルをどのように編成して読み取るかを知っておくと役に立ちます。

構成階層


私は、Vimがすべての設定とスクリプトを.vimrcファイルからしか取得しないと誤って信じていました。ランダムな「ドットファイル」のリポジトリを表示することは、この意見を強化するだけです。かなり頻繁に、人々は、エディタのあらゆる側面を制御することをタスクとする、本質的に単一の.vimrcファイルをアップロードします。これらの巨大な設定は、「vimディストリビューション」と呼ばれることもあります。

実際、Vimは.vimrcが多くの「エントリポイント」の1つにすぎない、きちんとした構造になっています。実際、ロードするスクリプトをVimに正確に尋ねることができます。これを行うには、マシン上のランダムプロジェクトのソースコードを編集し、ダウンロードして、コマンドを実行します。

:scriptnames

このリストは読む価値があります。これらのスクリプトの機能を推測し、スクリプトが配置されているディレクトリを書き留めてください。

リストは予想よりも長かったですか？多くのプラグインをインストールしている場合、エディターにはやるべきことがたくさんあります。次のstart.log作成コマンドを実行して、起動時に速度が低下することを確認します。

vim --startuptime start.log name-of-your-file

箱から出してすぐにVimが起動する速度を比較してください。

vim --clean --startuptime clean.log name-of-your-file

起動時またはバッファの読み込み時に読み込むスクリプトを決定するには、Vimランタイムパスを確認する必要があります。このパスは、コンマで区切られたディレクトリのリストで表されます。各ディレクトリには共通の構造が含まれています。 Vimは、各ディレクトリのこれらの構造をチェックして、起動スクリプトを見つけます。ディレクトリは、リストにある順序で厳密に処理されます。

次のコマンドを使用して、システムの実行パスを確認します。

:set runtimepath

一般に、ディレクトリは、start.logに書き込まれるのと同じ順序で処理されます。例外は、「後」にのみ作成されます。これは常にリストの最後にあり、最後に処理されます。

サードパーティのプラグイン


プラグインは単なるVimスクリプトなので、runtimepathの正しい場所にプラグインを配置するだけで十分です。一般に、インストールプロセスは非常に簡単です。ファイルをアップロードするだけです。問題は、プラグインによっては更新や削除が非常に難しいということです。プラグインはさまざまなサブディレクトリに分散しており、スクリプトによって実行パスが詰まっているためです。つまり、結局のところ、どのファイルがどのプラグインに属しているかを判別することは困難です。

この問題を解決するために、「プラグインマネージャ」が開発され始めました。 vim.orgには、少なくとも2003年まで（アーカイブが嘘をついていない場合）までプラグインのレジストリがありました。ただし、エンティティとしての「プラグインマネージャ」は、2008年までに流行しました。

これらのツールは、プラグインに特別なディレクトリを追加して、実行パスを追跡し、プラグインを追跡できるタグを配置します。ほとんどのマネージャーは、ネットワークからプラグインの更新をプルアップします。

プラグインの管理における最も抵抗の少ない方法は、組み込みのVim 8機能を使用することです。サードパーティのコードをプルアップする必要はありません。それを行う方法を見てみましょう。

まず、ランタイムパス内にoptとstartの2つのディレクトリを作成します。

mkdir -p ~/.vim/pack/foobar/{opt,start}

プレースホルダー「foobar」に注意してください（名前は変更できます）。内部に含まれるすべてのパッケージを完全に分類します。ほとんどのユーザーは、すべてのプラグインを1つのカテゴリにダンプするだけで、通常、これは正常です。任意の名前を選択してください。私はfoobarを使い続けます。理論的には、~/.vim/pack/navigationやなど、いくつかのカテゴリを作成することもできます~/.vim/pack/linting。 Vimはカテゴリ間の重複を認識せず、重複が存在する場合は2回ダウンロードすることに注意してください。

「start」のパケットは自動的にロードされますが、「opt」のパッケージは、コマンドを使用してVimから送信されるまでロードされません。:packadd。このオプションは、めったに使用されないパッケージに適しており、スクリプトを実行しなくても、箱から出してすぐにVimによってサポートされます。:packaddパッケージのアンロードに相当するものはないことに注意してください。

この例を確認するために、ctrlpファジー検索プラグインをoptに追加します。最新バージョンをダウンロードして解凍します。

curl -L https://github.com/kien/ctrlp.vim/archive/1.79.tar.gz \
	| tar zx -C ~/.vim/pack/foobar/opt

このコマンドは、フォルダーを作成し~ / .vim / pack / foobar / opt / ctrlp.vim-1.79、すぐに使用できるパッケージをダウンロードします。vimに戻り、新しいパッケージのヘルプタグインデックスを作成します。

:helptags ~/.vim/pack/foobar/opt/ctrlp.vim-1.79/doc

このコマンドは、「sorts」という名前のファイルをフォルダー内にパッケージsortsと共に作成し、Vim内部システムでテーマを表示できるようにします。別の方法：パッケージのダウンロード後にhelptags ALLを実行すると、コマンドがすべてのファイルとその実行パスを処理します。
パッケージを使用する場合は、パッケージをダウンロードして、この場合はタブを使用して終了するため、完全な名前を入力する必要がないことを覚えておいてください。

:packadd ctrlp.vim-1.79

Packaddのベースディレクトリはruntimepathにあり、プラグインとftdetectスクリプトを使用できます。ctrlpをロードした後、CTRL-Pコマンドを使用して、部分一致によるファイル検索を開くことができます。

〜/ .vimディレクトリを追跡し、gitを使用して各パッケージのバージョンを制御する人もいます。私の場合、tarアーカイブからパッケージを解凍し、リポジトリを介して手動で追跡します。頻繁な更新を必要としない十分に成熟したパッケージとスクリプトを使用する場合、それらは十分に小さく、gitの履歴を乱雑にすることはありません。

皮肉なことに、上記のコメントは、4番目の障害、および部分的には3番目のタイプの障害の理解のみを反映しています。スワップファイルとバックアップを拒否すると、段落1と2で説明されているケースで保護が失われます。

安全な運用のために推奨する構成の例を以下に示します。

" Protect changes between writes. Default values of
" updatecount (200 keystrokes) and updatetime
" (4 seconds) are fine
set swapfile
set directory^=~/.vim/swap//

" protect against crash-during-write
set writebackup
" but do not persist backup after successful write
set nobackup
" use rename-and-write-new method whenever safe
set backupcopy=auto
" patch required to honor double slash at end
if has("patch-8.1.0251")
	" consolidate the writebackups -- not a big
	" deal either way, since they usually get deleted
	set backupdir^=~/.vim/backup//
end

" persist the undo tree for each file
set undofile
set undodir^=~/.vim/undo//

これらの設定には不完全な録音のバックアップが含まれますが、バージョン管理、最適なバージョン管理、何とか何とか何とか何とかしているため、操作が正常に完了した後はファイルを保存しないでください。等mkdir〜/ .vim / {swap、undodir、backup}が必要になる場合があることに注意してください。そうでない場合、Vimは次の読み取り可能なフォルダーにアクセスします。スワップファイルとバックアップ履歴には機密情報が含まれている可能性があるため、ターゲットフォルダーでchmodコマンドを実行して、その内容を非公開にする必要がある場合もあります。

設定パスの特殊性は、常にスラッシュで終了することです。このスペルを使用すると、関数は、異なるディレクトリにある同じ名前のファイルのページングファイルとバックアップファイルのパスのあいまいさを排除できます。たとえば、/ foo / barのスワップファイルは〜/ .vim / swap /％foo％bar.swpに保存されます（パーセント記号でスラッシュをエスケープしました）。最近のVimパッチには、スラッシュがbackupdirで考慮されないようにするバグがあったため、それに対する保護を上記に示します。
