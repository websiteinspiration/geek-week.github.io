Linuxでのネットワーク問題の模倣

みなさん、こんにちは。私の名前はサーシャです。私はFunCorpのバックエンドのテストをリードしています。私たちは、多くの人と同様に、サービス指向アーキテクチャを実装しています。一方で、これにより作業が簡素化されます。各サービスを個別にテストする方が簡単ですが、ネットワーク上で発生することが多いサービス間の相互作用をテストする必要があります。

この記事では、ネットワークに問題が発生した場合のアプリケーションの動作を説明する基本的なスクリプトを確認できる2つのユーティリティについて説明します。



ネットワークの問題の模倣


通常、ソフトウェアは良好なインターネットチャネルを備えたテストサーバーでテストされます。過酷な条件では、制作がスムーズに行われない可能性があるため、接続の悪い場所でプログラムを確認する必要がある場合があります。 Linuxでは、tcユーティリティがそのような条件のシミュレーションに役立ちます。

tc（Traffic Controlの略）を使用すると、システムでのネットワークパケットの送信を構成できます。このユーティリティには優れた機能があり、詳細についてはこちらをご覧ください。私たちが使用しているため、我々は、トラフィックshedulingに興味がある：すぐに私はそれらのほんの数を検討するのqdiscを、私たちは不安定なネットワークをエミュレートする必要があるため、私たちはクラスレスなqdisc使用しますnetemを。

サーバーでエコーサーバーを実行します（このためにnmap-ncatを使用しました）。

ncat -l 127.0.0.1 12345 -k -c 'xargs -n1 -i echo "Response: {}"'

クライアント/サーバー対話の各ステップですべてのタイムスタンプを詳細に表示するために、エコーサーバーにTestリクエストを送信する簡単なPythonスクリプトを記述しました。

クライアントのソースコード
#!/bin/python

import socket
import time

HOST = '127.0.0.1'
PORT = 12345
BUFFER_SIZE = 1024
MESSAGE = "Test\n"

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
t1 = time.time()
print "[time before connection: %.5f]" % t1
s.connect((HOST, PORT))
print "[time after connection, before sending: %.5f]" % time.time()
s.send(MESSAGE)
print "[time after sending, before receiving: %.5f]" % time.time()
data = s.recv(BUFFER_SIZE)
print "[time after receiving, before closing: %.5f]" % time.time()
s.close()
t2 = time.time()
print "[time after closing: %.5f]" % t2
print "[total duration: %.5f]" % (t2 - t1)

print data

定数値に加えて、遅延には、偏差、分布関数、および相関（前のパッケージの値を使用）を指定できます。これは次のように行われます。

tc qdisc change dev lo root netem delay 500ms 400ms 50 distribution normal

ここでは、100から900ミリ秒の間隔で遅延を設定し、値は正規分布に従って選択され、前のパケットの遅延値と50％の相関関係があります。
最初のコマンドでaddを使用し、次にchangeを使用していることに気付くでしょう。これらのコマンドの意味は明白なので、構成を削除できるdelがまだあることを追加します。

パケットロス


では、パケットロスをやってみましょう。ドキュメンテーションからわかるように、これは3つの方法で行うことができます。ある確率でランダムにパケットを損失する、2、3、または4つの状態のマルコフチェーンを使用してパケット損失を計算する、またはElliot-Gilbertモデルを使用する。記事では、最初の（最も単純で最も明白な）方法と、他の方法についてここで読むことができます。

25％の相関でパケットの50％を損失させましょう：

tc qdisc add dev lo root netem loss 50% 25%

残念ながら、tcpdumpはパケット損失を示すことができず、実際に機能すると仮定するだけです。これを確認するために、client.pyスクリプトの増加した不安定なランタイムは、再送信されたパケットの増加数だけでなく、即座に、またはおそらく20秒で実行できます。

パケットの複製


netemで 他に何ができますか？たとえば、パケット損失の反対の状況をシミュレートするには-パケットの複製。このコマンドは、確率と相関の2つの引数も受け取ります。

tc qdisc change dev lo root netem duplicate 50% 25%

パッケージの再注文


パッケージを混在させるには、2つの方法があります。

最初の部分では、パケットはすぐに送信され、残りは指定された遅延で送信されます。ドキュメントの例：

tc qdisc change dev lo root netem delay 10ms reorder 25% 50%

25％（および50％の相関）の確率で、パケットはすぐに送信され、残りは10ミリ秒の遅延で送信されます。

2番目の方法は、N番目ごとのパケットが所定の確率（および相関）で瞬時に送信され、残りのパケットが所定の遅延で送信される場合です。ドキュメントの例：

tc qdisc change dev lo root netem delay 10ms reorder 25% 50% gap 5

25％の確率で5つおきのパケットが遅延なく送信されます。

帯域幅の変更


通常、それらはどこにでもTBFに送信されますが、netemを使用してインターフェース帯域幅を変更することもできます。

tc qdisc change dev lo root netem rate 56kbit

このコマンドは、ローカルホストのトリップを、ダイヤルアップモデムでインターネットをサーフィンするのと同じくらい苦痛にします。ビットレートの設定に加えて、データリンク層プロトコルモデルをエミュレートすることもできます。パケットのオーバーヘッド、セルサイズ、セルのオーバーヘッドを指定します。たとえば、この方法で、ATMと56 kbpsのビットレートをシミュレートできます。

tc qdisc change dev lo root netem rate 56kbit 0 48 5

接続タイムアウトのシミュレーション


ソフトウェア受け入れのテスト計画におけるもう1つの重要なポイントは、タイムアウトです。分散システムでは、サービスの1つがオフになると、他のサービスが時間どおりに他のサービスをフォローアップするか、クライアントにエラーを返す必要があるため、これは重要です。ハングアップしたり、応答を待ったり、接続を確立したりすることはありません。

これにはいくつかの方法があります。たとえば、応答しないモックを使用するか、デバッガーを使用してプロセスに接続し、適切な場所にブレークポイントを設定してプロセスを停止します（これはおそらく最も変な方法です）。しかし、最も明白なものの1つは、ファイアウォールのポートまたはホストです。Iptablesはこれに役立ちます。

デモのために、ポート12345をファイアウォールで囲み、クライアントスクリプトを実行します。送信者からこのポートへの発信パケット、または受信者への着信パケットをファイアウォールできます。私の例では、着信パケットはファイアウォールになります（チェーンINPUTと--dportオプションを使用）。このようなパッケージは、TCPフラグRSTを使用してDROP、REJECT、またはREJECTにすることができます。ICMPホストに到達できない場合は可能です（実際、デフォルトの動作はicmp-port-unreachableであり、応答としてicmp-net-unreachable、icmp-proto-を送信するオプションもあります。到達不能、icmp-net-prohibitedおよびicmp-host-prohibited）。

DROPのルールがある場合、パケットは単に「消失」します。

iptables -A INPUT -p tcp --dport 12345 -j DROP

拒否


次に、同様のルールを追加しますが、REJECTを使用します。

iptables -A INPUT -p tcp --dport 12345 -j REJECT

クライアントはエラー[Errno 111] Connection refusedで 1秒後に終了します。ICMPトラフィックを調べます。

tcp-resetでREJECT


--reject-with tcp-reset オプションを追加してみましょう：

iptables -A INPUT -p tcp --dport 12345 -j REJECT --reject-with tcp-reset

この場合、最初のリクエストでRSTパケットを受信したため、クライアントはすぐにエラーを残します。

icmp-host-unreachableでREJECT


REJECTの別の使用例を試してみましょう。

iptables -A INPUT -p tcp --dport 12345 -j REJECT --reject-with icmp-host-unreachable

クライアントはエラー[Errno 113] No route to hostで1秒で終了します。ICMPトラフィックでは、ICMPホスト127.0.0.1 unreachableが表示されます。

残りのREJECTパラメータを試すこともできます。これらに焦点を当てます:)

リクエストのタイムアウトをシミュレートします


別の状況は、クライアントがサーバーに接続できたが、サーバーに要求を送信できない場合です。フィルタリングがすぐに開始されないようにパケットをフィルタリングする方法は？クライアントとサーバー間の通信のトラフィックを見ると、接続を確立するときはSYNフラグとACKフラグのみが使用されていますが、データを交換するときは、最後の要求パケットにPSHフラグが含まれていることがわかります。バッファリングを回避するために自動的に設定されます。この情報を使用してフィルターを作成できます。PSHフラグを含むパケットを除くすべてのパケットを通過させます。したがって、接続は確立されますが、クライアントはサーバーにデータを送信できなくなります。
