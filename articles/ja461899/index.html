この記事では、PHP言語とLaravelフレームワークでイベントCQRSシステムを作成する基本について説明します。コマンドバスを使用した開発スキームに精通しており、イベント（特に、リスナーの配列に対するイベントの公開）について理解していることを前提としています。この知識を更新するには、Laracastsサービスを使用できます。また、CQRSの原則をある程度理解していることを前提としています。そうでない場合は、Mathias Verraes Workshop on Event GenerationとGreg YoungのCQRSとEvent Generationの 2つの講義を聞くことを強くお勧めし  ます。

ここで指定したコードをプロジェクトで使用しないでください！これは、CQRSの背後にあるアイデアを理解するための学習プラットフォームです。このコードは信頼性が高いとは言えず、テストも不十分です。さらに、私がインターフェイスをプログラミングすることはほとんどないため、コードの個々の部分を変更することははるかに困難です。使用できるCQRSパッケージのより良い例は、Qandidate Labが開発したBroadwayです。これはクリーンで疎結合のコードですが、イベントシステムに遭遇したことがない場合、一部の抽象化では完全には明確になりません。

そして最後-私のコードはイベントとLaravelコマンドバスに接続されています。Laravelでコードがどのように表示されるかを確認したかったのですが（通常、このフレームワークは小規模な代理店のプロジェクトに使用します）、自分で実装を作成する必要があると思います。私のコードがフレームワークを使用していない人にもわかりやすくなることを願っています。

githubでは、コードは  https://github.com/scazz/cqrs-tutorial.gitにあります。ガイドでは、ロジックを増やすためにそのコンポーネントを検討します。

サーフスクールの初期登録システムを作成します。その助けを借りて、学校のクライアントはクラスに登録できます。記録プロセスでは、次のルールを作成します。

  • 各レッスンには少なくとも1つのクライアントが必要です...
  • ... 3つ以下。

イベントベースのCQRSシステムの最も印象的な機能の1つは、システムに必要な各メトリックに固有の読み取りモデルの作成です。 ElasticSearchでの読書モデルの投影の例が見つかります。GregYoungは、複雑なイベントを処理するために、イベントストアにサブジェクト指向の言語を実装しています。ただし、簡単にするために、読み取りプロジェクションはEloquentで使用する標準のSQLデータベースになります。その結果、クラス用とクライアント用に1つのテーブルが作成されます。

イベント生成の概念では、イベントをオフラインで処理することもできます。ただし、この記事では、「従来の」開発モデルを最大限に遵守し（ここでも簡単にするため）、イベントがリポジトリに格納された直後に、読み取り予測をリアルタイムで更新します。

したがって、書き込みのたびにイベントを生成し、イベントを使用して不変条件を防ぐために必要な変更を記録するシステムが得られました。次のステップは、これらのイベントをストア（EventStore）に保管することです。まず、このイベントリポジトリを作成する必要があります。簡単にするために、Eloquentモデル、次のフィールドを持つ単純なSQLテーブルを使用します：* UUID（イベントを適用するエンティティを知るため）* event_payload（イベントを再作成するために必要なすべてを含むシリアル化されたメッセージ）* recordedAt -イベントがいつ発生したかを知るためのタイムスタンプ。コードを注意深く確認すると、イベントストレージテーブルを作成して破棄するための2つのコマンドが作成されていることがわかります。

  • php artisan eloquenteventstore：create（App \ CQRS \ EloquentEventStore \ CreateEloquentEventStore）
  • php artisan eloquenteventstore:drop (App\CQRS\EloquentEventStore\DropEloquentEventStore) ( App\Console\Kernel.php, ).

SQLをイベントストアとして使用しない非常に2つの理由があります。追加のみのモデルを実装しない（データを追加するだけで、イベントは不変でなければならない）ことと、SQLが一時データベースの理想的なクエリ言語ではないためです。後続の出版物でのイベントストアの置き換えを容易にするために、インターフェイスをプログラミングします。

イベントを保存するには、リポジトリを使用します。save()レコードモデルが呼び出されるたびに、uncommittedEventsリストがイベントストアに格納されます。イベントを保存するには、そのシリアル化と逆シリアル化のメカニズムが必要です。このためのシリアライザを作成します。イベントクラス（例App\School\Lesson\Events\LessonWasOpened:)やイベントペイロード（イベントの再構築に必要なデータ）などのメタデータが必要です。

これらはすべてJSON形式でエンコードされ、エンティティのUUIDとタイムスタンプとともにデータベースに書き込まれます。イベントをキャプチャした後に読み取りモデルを更新するため、リポジトリは保存後にすべてのイベントをトリガーします。シリアライザはイベントクラスの書き込みを担当し、イベントはペイロードのシリアル化を担当します。完全にシリアル化されたイベントは次のようになります。

 { class: "App\\School\\Lesson\\Events\\", event: $event->serialize() }

このレコーディングモデルでは、イベントストアでイベントが既に適用されているエンティティを「ロード」するメソッドを実装する必要があります。これは、エンティティのUUIDを参照するすべてのイベントを再生することで実現できます。一般的に、プロセスは次のとおりです。

  1. イベントストアから関連するすべてのイベントメッセージを受信します。
  2. メッセージごとに、対応するイベントを再作成します。
  3. 新しいエンティティレコードモデルを作成し、各イベントを再生します。

リポジトリロード関数は、再作成されたイベントの配列を返します。これを行うには、まずリポジトリ内のイベントに関するメッセージを検索し、次にそれらを渡してSerializer各メッセージをイベントに変換します。Serializerイベントからメッセージを作成deserialize()するため、逆変換を実行するメソッドを追加する必要があります。Serializerイベントデータ（クライアント名など）をシリアル化するために各イベントに渡されることを思い出してください。同じSerializableEvent方法で逆変換を実行しますが、メソッドを使用してインターフェイスを更新する必要がありますdeserialize()。すべてを適切に配置できるようにコードを見てみましょう。まず、ダウンロード機能EventStoreRepository：

public function load($uuid) {
   	 $eventMessages = EloquentEventStoreModel::where('uuid', $uuid)->get();
   	 $events = [];
   	 foreach($eventMessages as $eventMessage) {
   		 /*       event_payload,        . */
   		 $events[] = $this->eventSerializer->deserialize( json_decode($eventMessage->event_payload));
   	 }

で適切な逆シリアル化関数を使用するeventSerializer：

app/CQRS/Serializer/EventSerializer.php
public function serialize( SerializableEvent $event ) {
   	 return array(
   		 'class'   => get_class($event),
   		 'payload' => $event->serialize()
   	 );
    }
    public function deserialize( $serializedEvent ) {
   	 $eventClass = $serializedEvent->class;
   	 $eventPayload = $serializedEvent->payload;
   	 return $eventClass::deserialize($eventPayload);
    }
