Gitlab、Jira、およびConfluenceのPython統合によるリリースビルドの自動化

最近、あるスタンドアップで、同僚は合理的な提案を行いました。リリースのアセンブリを自動化し、Pythonで記述されたJiraと対話するための既成のすでに実践されている方法をベースにしています。

展開プロセスは次のとおりです。テスト済みの十分な数のタスクが蓄積されると、タスクの影響を受ける各プロジェクトで候補候補（RC）がリリースされ、RCの一部としてタスクがテストされます。その後、RCはステージングサーバーに注がれ、戦闘環境のすぐ近くで、まだテストされ、完全な回帰が実行されます。そして、必要な展開アクションの後、フレッシュリリースがマスターに注がれます。

最近まで、すべての組み立てプロセスは開発者の1人によって手動で行われていました。それは1時間、2時間以上かかり、私にとっては非常に興味深い職業ではなかったようです。これで、ほぼすべての準備が整うと、5つのプロジェクトに影響を与える20のタスクのリリースは1分もかからなくなります。もちろん、競合の解決、失敗したテストの実行などは残っていますが、これを考慮しても、誰かが最初に自分を解放してRCを作成するまで待つ必要がある開発者とテスターの時間は大幅に節約されます。

一般的に、私はその仕事に取り掛かりましたが、それは非常に面白くて魅力的であることがわかりました。そして、エキサイティングなプロジェクトでなければ、仕事の喜びのために他に何が必要ですか？

私はレガシーを研究しました：それはJira APIを直接使用することが判明しました、そしてそれは書かれました、それは最適ではないように私には思えました。たとえば、リリースタスクのリストは次のように取得されました。既存のすべてのリリースがJiraからダウンロードされ、目的のリリースが見つかるまで、リリースごとに名前で比較されました。

一般に、APIと直接やり取りすると、コードが読みにくくなります。そして私は自転車を発明したくなかった。 Githubでの最初の検索で、かなりシンプルで強力なライブラリであるJIRA Pythonライブラリにたどり着きました。当初は、GithubにもあるGitPythonライブラリを使用してマージ要求を作成することを計画していました。しかし、問題をさらに調査すると、すぐにGitではなくGitlabに関連する何かを見つけるというアイデアにつながりました。その結果、私は最も有名なソリューションであるPython GitLabに落ち着きました。

まず、リリースに適したステータスのタスクのリストを取得しました。以前のソリューションを大幅に変更するのではなく、目的のリリースのタスクのAPIをすぐに要求することで、より効果的にすることにしました。

fix_issues = jira.search_issues(f'fixVersion={release_input}')

fix_id = jira.issue(fix_issues.iterable[0]).fields.fixVersions[0].id

おそらく同じことが内部で起こりますが、それはより美しく最適であることがわかりましたが、それでもあまり読みやすくはありませんでした。さらに、受け取ったタスクから、リクエストをマージするためのリンクを収集する必要があります。見つかったマージリクエストをnamedtupleに保存することにしました、それらはこれに最適です：

Merge_request = namedtuple('Merge_request', ['url', 'iid', 'project', 'issue'])

マージリクエストもJira APIを使用して受信されました：

projects = set()

links_json = requests.get(url=REMOTE_LINK.format(issue_number),

                            auth=login,jira_password).json()

for link in links_json:

   url_parts = link['object']['url'].split('/')

   project = f'{url_parts[4]}'

   iid = url_parts[6]

   projects.add(project)

その後、見つけたライブラリをどこで使えるか決めました。次に、おそらく、これらの部分をリファクタリングします。

次に、突然、正しいRCブランチが既に存在していることを確認する必要があります。すでにビルドを試みていた場合は、それらを削除して新しいブランチを作成する必要があります。私はすでにPython GitLabライブラリを使用してこれを行いました：

gl = gitlab.Gitlab('https://gitlab...ru/', private_token=GITLAB_PRIVATE_TOKEN)

pr = gl.projects.get(project)

try:

   rc = pr.branches.get(f'{RC_name}')

   rc.delete()

   pr.branches.create({'branch': f'{RC_name}', 'ref': 'master'})

except gitlab.GitlabError:

   pr.branches.create({'branch': f'{RC_name}', 'ref': 'master'})

その後、Jiraアセンブリタスクでテーブルへの入力を開始できます。テーブルの情報は、次の列に含まれています：いいえ、タスク、優先度、RCのタスクからのマージリクエスト、マージリクエストのステータス（Gitlabで渡されたテストが競合しているかどうかにかかわらず、注がれるかどうか）。

このステップで、Gitlabの不快な欠陥に遭遇しました。変更が以前にマージリクエストの宛先ブランチで凍結されていた場合、Gitlab APIは、マージリクエストのステータスをリクエストすると、競合の存在について回答します。原則として、これは理解できます。注ぐものが何もない場合は機能しませんが、なぜこの場所で競合があると言うのでしょうか。 Gitlabフォーラムは数年前から人々に尋ねてきましたが、まだ回答はありません。

私の場合、これは誤った競合ステータスにつながり、手動でマージ要求をチェックします。まだ何も考えていません。

マージ要求を作成するときのロジックは次のとおりです。テストに合格しない場合、または競合がある場合、マージ要求は作成されますが、RCに流れず、対応するステータスがテーブルに置かれます。

テストの実行を確認するには、適切なパイプラインのリストを探します。Gitlabは必要に応じて、日付順にソートされた降順でそれらを出力します。私たちは最初のものを取ります-それが必要なものになります：

pipelines = project.pipelines.list(ref=f'{issue}')

if pipelines:

   pipelines = pipelines[0]

   if pipelines.attributes['status'] != 'success':

       status = '(x)   !, '

次に、マージ要求自体を作成します。これがビルドの最初の試みではない場合に備えて、オープンなものをチェックします。必要なマージ要求がない場合は、次のように作成します。

mr = project.mergerequests.list(

 state='opened',

source_branch=source_branch,         target_branch=target_branch)

if mr:

   mr = mr[0]

else:

   mr = project.mergerequests.create(

{'source_branch': source_branch,

           'target_branch': target_branch,

           'title': f"{(MR.issue).replace('-', '_')} -> {RC_name}",

           'target_project_id': PROJECTS_NAMES[MR.project],})

status = mr.attributes['merge_status']

url = mr.attributes['web_url']

return status, url, mr

MR.issue.replace（ '-'、 '_'）-タスクの名前を変更して、Jiraのタスクに関するGitlabaの情報のないコメントを削除します。

次に、受信したマージ要求のステータスを確認します。競合がない場合は、RCに注ぎます。競合がある場合は、適切なステータスを記録し、手動で確認するために残します。

さらに、類推により、すべてのプロジェクトに対してRCからステージングへ、およびステージングからマスターへのマージ要求を作成します。それらは、RCブランチのリリースビルド全体をチェックした後に注がれます。

Jiraタスクテンプレートに2つのフィールドが追加されました：プレプレポストアクションとポストデポストアクション用です。関連するすべてのタスクについて、アルゴリズムはそのようなアクションのリストを収集し、何も忘れないようにするためにアセンブリタスクに入ります。

そして最後に、Jiraの出力は、アセンブリタスクの作成です。

existing_issue = jira.search_issues(

f'project=PROJ AND summary ~ " {release_name}"')

if existing_issue:

   existing_issue = existing_issue[0]

   existing_issue.update(fields={

       'description': message,

       'customfield_xxx': message_before_deploy,

       'customfield_yyy': message_post_deploy,})

else:

   issue_dict = {

       "fixVersions": [{"name": release_name,}],

       'project': {'key': 'PROJ'},

       'summary': f" {release_name}",

       'description': message,

       'issuetype': {'name': 'RC'},  #      

       'customfield_xxx': message_before_deploy,

       'customfield_yyy': message_post_deploy,}

   new_issue = jira.create_issue(fields=issue_dict)

さらに考えは次のとおりです。JiraのWebフックを使用してGitlabでスクリプトを実行します。たとえば、Webhookを作成してアセンブリタスクを作成したり（特別なタイプのタスクを作成したり）、タイトルに「Assembly」という単語を含むタスクを作成したりできます。そして、このフックのGitlabは、プロセス全体を開始するbashスクリプトを実行するか、PythonでDockerイメージを生成してスクリプトを実行します。 2番目のオプションはすでに複雑すぎますが。はい、JiraとGitlabのテクニカルアカウントが必要です。一般に、まだ最終的な決定はありません。

RCブランチを作成したら、必要なブランチにテストベンチをデプロイして、回帰テストを実行することもできます。 Jenkinsはこれを処理できます。しかし、これは今のところ計画にもあります。

これは、テスターの作業を加速し、開発者を日常の作業から解放するためです。ただし、ここでの経済的意味も非常に具体的です。1日あたり8時間の作業時間で、150,000ルーブルの架空の給与を持つ平均的な開発者を（真空で）とります。 2年間、約700のリリースがありました。これは1日あたりのリリースについてです。開発者がリリースをビルドする時間の少なくとも1時間はなくなったと私は思います。つまり、このプロセスの自動化により、会社は毎月最低150,000 / 8 = 18,750ルーブルを節約できます。

途中で、私は別のスクリプトを自分で作成し、次のリリースに関する統計を示しました。タスクの数、影響を受けるプロジェクトなどです。Gitlabのどのプロジェクトにも開発者ステータスがない場合、マージ要求を作成するときにアクセスが拒否されるためです。また、展開アクションについて事前に知っているか、誤ってこのリリースに含まれるタスクをキャッチするのも便利です。リリース通知の配布は、SMTPおよび電子メールモジュールを使用して解決されました。

私はこの問題の解決策に満足していました。何か新しいことを学び、それを実践することは常に興味深いことです。この経験が誰かに役立つなら、それは素晴らしいことです。
