ROSの起動ファイルはノード用のプレハブホッジポッドの一種であり、1つまたは複数の起動ファイルにそれらを配置できます。リストから最初の起動を見てみましょう。

bringup.launch 
<launch>

    <node pkg="rosserial_python" name="rosserial_lino" type="serial_node.py" output="screen">
        <param name="port" value="/dev/linobase" />
        <param name="baud" value="57600" />
    </node>

    <!-- IMU Relay and Filter -->
    <include file="$(find linorobot)/launch/include/imu/imu.launch" />

    <!-- Publish Linorobot odometry -->
    <node pkg="linorobot" name="lino_base_node" type="lino_base_node"></node>

    <!-- Publish static transform from base_footprint to base_link -->
    <node pkg="tf2_ros" type="static_transform_publisher" name="base_footprint_to_base_link" args="0 0 0.065 0 0 0  /base_footprint /base_link"/>

    <!-- Odom-IMU Extended Kalman Filter-->
    <node pkg="robot_localization" type="ekf_localization_node" name="ekf_localization">

        <rosparam command="load" file="$(find linorobot)/param/ekf/robot_localization.yaml" />

ご覧のように、6つのノードが起動され、一部のノードは個別の起動ファイルに配置されます。

  • rosserial_lino-teenzyとの通信を担当するノード。
  • imuはジャイロスコープです。
  • lino_base_node-ロボットのベース。
  • base_footprint_to_base_link-ロボットを空間に「バインド」します（0.065m-床からロボットのベースまでの距離）。
  • ekf_localization-「浄化された」オドメトリを変換するノード。
  • レーザー-LIDAR。

この起動ファイルに関係するすべてのノードが重要です。しかし、それらはロボットのナビゲーションよりもロボットのオドメトリに関連しています。

Move_base


2番目に重要なコンポーネント。

move_baseの主な機能は、ロボットを現在の位置からターゲットの位置に移動することです。
rvizで「2D Nav Goal」を使用するたびに、ロボットの移動に使用されるトピックmove_base / Goalにメッセージが送信されます。

Move_baseは基本的に5つのトピックで構成される単純なアクションサーバーです（ROSではトピック、サービスの他にアクションもあります）。

•	move_base/goal (move_base_msgs/MoveBaseActionGoal)
•	move_base/cancel (actionlib_msgs/GoalID)
•	move_base/feedback (move_base_msgs/MoveBaseActionFeedback)
•	move_base/status (actionlib_msgs/GoalStatusArray)
•	move_base/result (move_base_msgs/MoveBaseActionResult)

これを知っていれば、メッセージ（トピックmove_base / Goal）を直接彼に送信し、rvizをバイパスしてロボットを動かすことができます。

*この一部は、前の記事ですでに触れました。

メッセージ形式は次のようになります。

rostopic pub /move_base/goal geometry_msgs/PoseStamped '{ header: { frame_id: "map" }, pose: { position: { x:  2.49339078005, y: 0.0666679775475, z: 0 }, orientation: { x: 0, y: 0, z: -0.999261709946, w: 0.0384192013861 } } }'

*ここでは、「corridor」というコードネームのポイントにロボットを送ります。

Move_baseは1つのノードで構成されていますが、「ネストされた人形」であり、パラメータを含むいくつかのファイルが含まれています。

  • costmap_common_params.yaml
  • local_costmap_params.yaml
  • global_costmap_params.yaml
  • base_local_planner_default_params.yaml
  • move_base_params.yaml


move_baseコードを見ると、これらのパラメーターとの関係を理解できます。

現在の2輪ロボットプロジェクトでは、move_baseは次のように始まります。

move_base_2wd.launch 
<launch>
    <node pkg="move_base" type="move_base" respawn="false" name="move_base" output="screen">
        <rosparam file="$(find linorobot)/param/navigation/costmap_common_params.yaml" command="load" ns="global_costmap" />
        <rosparam file="$(find linorobot)/param/navigation/costmap_common_params.yaml" command="load" ns="local_costmap" />
        <rosparam file="$(find linorobot)/param/navigation/local_costmap_params.yaml" command="load" />
        <rosparam file="$(find linorobot)/param/navigation/global_costmap_params.yaml" command="load" />
        <rosparam file="$(find linorobot)/param/navigation/base_local_planner_default_params.yaml" command="load" />
        <rosparam file="$(find linorobot)/param/navigation/move_base_params.yaml" command="load" />
    </node>
</launch>


rviz（またはスクリプト）へのトリップの目的が示され、move_baseが対応するメッセージを受信すると、それをグローバルプランナー（以降、GP）に送信します。次に、GPUが目的地への安全なルートを計算します。この経路はロボットが移動する前に計算されるため、この計画では、移動中にロボットのセンサーから取得されるデータは考慮されません。

GPが移動計画をコンパイルするたびに、この計画はトピック/ move_base / DWAPlannerROS / global_planで公開されます（echo引数を使用して何が得られるか見てみましょう）。



グローバル計画はどのように計算されますか？


GPはすべての「魔法」を担当します。ダイクストラアルゴリズム（通常）を使用して、開始位置（初期ポーズ）と終了点（ゴールポーズ）の間の最短経路を計算します。

GPにはいくつかの種類があります。

  • Navfn
  • にんじんプランナー
  • グローバルプランナー

コストマップ


旅行計画を作成するとき、GPはマップサービスが提供するエリアの静的マップを使用します。ただし、このカードに加えて、さらに2つのいわゆる「コストマップ」が戦闘に参加します。

これらは2つのカードです。

  • 静的マップマップから作成されたグローバルコストマップ。
  • ローカルコストマップ-ロボットのセンサーが到着すると、センサーから受信したデータから形成されます。

これらのカードの目的は、地図上の障害物がある場所とない場所にロボットを表示することです。これらのカードがなければ、ロボットは障害物を見ずに地図上を移動します。

簡単に言うと、グローバルコストマップはGP（グローバルプランナー、グローバルプランナー）に使用され、ローカルコストマップはローカルプランナーに使用されます。

グローバルコストマップ


前述のように、グローバルコストマップは、基本的にユーザーによって提供される静的マップ（map.pgm）から作成され、障害物のサイズやその他の情報を決定します。

パラメータはプロジェクトファイルで定義されています：

/param/navigation/global_costmap_params.yaml

global_costmap_params.yaml 
global_costmap:
  global_frame: /map
  robot_base_frame: /base_footprint
  update_frequency: 1.0 #before: 5.0
  publish_frequency: 0.5 #before 0.5
  static_map: true
  transform_tolerance: 0.5
  cost_scaling_factor: 10.0
  inflation_radius: 0.55


ここでは、どのフレームが使用されているか、マップが更新されている頻度、静的であると判断されている、プランナーが計画を作成する精度、障害物の半径を確認できます。

そしてここで、inflation_radiusを少なくとも0.2mに減らす必要があります。現在のレートとフルシチョフの敷地内の狭い通路では、プランナーは単純に計画を立てることができません。

ローカルプランナー


ローカルスケジューラ（以下「LP」）は、GP（グローバルスケジューラ）と連携して動作し、同様の機能とそれを区別する機能の両方を備えています。

彼はGPと同様にルートの作成を担当しますが、他のタスクは肩にかかります。GPが設定したルートをたどり、途中の障害物を回避する（障害物を回避する）ことです。

GPが道路計画を計算するとすぐに、この計画はLPで毒殺されます。次に、LPはこの計画の各セグメントを満たします。したがって、ルートとマップを持つLPは「移動コマンド」を送信してロボットを移動します。

GPとは異なり、LPは走行距離計、LIDARからのデータを監視し、障害物のないルートを選択します。

ローカルプランが計算されるとすぐに、トピック/ local_planで公開されます。さらに、LPはトピック/ global_planに従ってグローバルプランの一部を公開します。

ローカルプランナーとグローバルプランナーにはいくつかのタイプがあります。

  • base_local_planner
  • dwa_local_planner
  • eband_local_planner
  • teb_local_planner

このプロジェクトはdwa_local_plannerを使用します。

ローカルコストマップ


ローカル「コストマップ」は、ローカルパスを計算するためにLP（ローカル偏光子）によって使用されます。

グローバルコストマップとは異なり、ローカルコストマップはロボットのセンサーからのデータに基づいて構築されます。
コストカードの幅と高さ（ユーザー定義）が与えられると、移動時にドラッグによってロボットがこのコストカードの中心に保持されます。

GPとLPを比較すると、次のように両者の間に線を引くことができます。GPは、最初はマップ上にない一時的な障害物を認識しませんが、LPはそれを決定します。

ローカルコストマップパラメータ


このマップを定義するオプションは多くありません（linorobot / param / navigation / local_costmap_params.yaml）：

local_costmap_params.yaml 
 
 global_frame: /odom #  ,    ,   odom.  
  robot_base_frame: /base_footprint # .
  update_frequency: 1.0 #before 5.0 #     
  publish_frequency: 2.0 #before 2.0 #     
  static_map: false #     .   true,   
  .
  rolling_window: true #      .
  width: 2.5 #   
  height: 2.5 #   
  resolution: 0.05 #increase to for higher res 0.025 #   
  transform_tolerance: 0.5  
  cost_scaling_factor: 5
  inflation_radius: 0.55#


ここでは、高さを1メートルに減らす必要があります。私たちが持っているマップの大部分は2dであり、最大の高さは床から1 m以下のLIDARであるため、1メートルを超える計算は意味がありません。

また、解像度を下げて修正します。

また、インフレを下げます-inflation_radius：0.3メートル。グローバルコストマップの場合のように、完全に削除しないことをお勧めします。ロボットは障害物にしがみついて、それらを回避しようとします。

update_frequencyを減らして、予期しない障害物がすぐにマップに当たるようにすることもできます。

ローカルコストマップは自動更新されます。これは、パラメーター（update_frequency）で指定された頻度で発生します。

各更新サイクルには、次の手順が含まれます。

  • ロボットのセンサーからデータを受信しました。
  • マーキングおよびクリーニング操作が実行されます（各センサーは障害物があるかどうかを報告します）。
  • マップ上の各セルには特定の値が割り当てられています（ビジー/空き）;
  • 障害は「インフレ」で示されます。

したがって、静的マップに加えて、コストマップがあります。グローバルコストマップとローカルコストマップです。

ただし、両方のコストカードに適用されるパラメーターがまだあります。

一般的なコストマップパラメータ


パスlinorobot / param / navigation / costmap_common_params.yamlに沿って配置されています

costmap_common_params.yaml 
obstacle_range: 2.5
raytrace_range: 3.0
footprint: [[-0.24, -0.22], [-0.24, 0.22], [0.24, 0.22], [0.24, -0.22]]
inflation_radius: 0.55
transform_tolerance: 0.5

observation_sources: scan
scan:
  data_type: LaserScan
  topic: scan
  marking: true
  clearing: true

map_type: costmap


このファイルには、ロボットセンサーが障害物をどのように認識するかが記述されています。たとえば、interference_range：2.5は、2.5 mの距離で検出された場合、障害物が検出された場合、コストマップにプロットする必要があることを意味します。フットプリント-宇宙におけるロボットの寸法。
Observation_sources：scan-情報が処理されるソース（/スキャントピックでLIDARを公開します）。また、コストマップ上の障害をマークしてクリアします。

特にROSとlinorobotの全体的なナビゲーションプロセスは何ですか？


ロボットの現在の位置が決定されると（ロボットはローカライズされます）、rvizエディターまたはスクリプトを介してターゲットポイント（目標位置）に移動するコマンドをロボットに送信できます。このコマンドはmove_baseノードに移動します。次に、move_baseノードは目標位置をグローバルプランナーに送信し、グローバルプランナーは現在の位置からターゲットまでのパス（パスプランニング）を計算します。この計画では、map_serverから受け取った障害物を含むグローバルコストマップが考慮されます。

さらに、グローバルプランナーは計算されたパスをローカルプランナーに送信し、ローカルプランナーはグローバルプランの各セグメントを実行します。

ローカルスケジューラは、ライダーのスムーズな移動のための衝突のないローカルプランを作成するために、ライダーからオドメトリとデータも受け取ります。

ローカルスケジューラは、ルート沿いの障害物を監視するためのローカルコストマップに「関連付け」られています。

ローカルスケジューラーは速度コマンドを生成し、それらをベースコントローラーに送信します。基本コントローラは、これらのコマンドを直接ロボット制御コマンドに変換します。

ボーナス ロボットをmajordomoに接続する方法。


ロボットは、rvizエディターからだけでなく、スクリプトを直接使用して制御することもできます。
ウェブブラウザを介して制御コマンドを送信することが可能です。

オープンソース製品のwebvizは、これらの目的に非常に役立ちます。

あなたの家の自動化を新しいレベルに上げるために、それを別の有名なプロジェクト-majordomoと交差させてみましょう。

Majordomo-自己完結型のオープンソースホームオートメーションプロジェクトは、このリソースで何度も言及されています。そして、作者はこのプロジェクトを数年にわたって個人的な非利己的な目的のために観察して使用してきたので、それを選ぶことを提案します。
ROSトピックからの出力をmajordomoの静的ページに接続する方法について話しましょう。

手順1.

ロボットにrosbridgeパッケージをインストールします。

sudo apt-get install ros-kinetic-rosbridge-suite

* rosbridgeのインストールが失敗する可能性が高いので、このオプションを試してください：

cd ~/linorobot_ws/src
git clone https://github.com/RobotWebTools/rosbridge_suite.git
git clone https://github.com/GT-RAIL/rosauth.git
cd ..
catkin_make
sudo pip install tornado
sudo pip install pymongo
sudo apt-get install python-twisted

起動します（メインの起動ファイルを起動した後）：

roslaunch rosbridge_server rosbridge_websocket.launch

ブリッジがポート9090で機能することを確認します。
netstat -a | grep 9090

ローカルブラウザー上にある任意のPCからChromeブラウザーを介してロボットに接続しようとします。

ここで、ipはipロボットです。
