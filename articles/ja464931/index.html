アセンブラ挿入... Cで？

したがって、この話は3つの要因の偶然から始まりました。わたし：

  1. ほとんどがCで書かれました。
  2. それがどのように配置され、機能するかを大まかに想像しただけです。
  3. アセンブラに興味を持った。

この一見無害な混合が奇妙なアイデアを生み出しました。どういうわけかこれらの言語を組み合わせることが可能ですか？C ++と同様に、Cにアセンブラー挿入を実行する機能を追加します。

これがもたらす結果に興味がある場合は、猫へようこそ。



最初の困難


そのときでさえ、C＃コードからアセンブラコードを呼び出すための標準的なツールが存在することは非常にありそうにないことに気づきました。問題の表面的な調査（とりわけ、最初の直感を確認した後-「箱から出して」そのような可能性はない）の後、イデオロギーの問題に加えて、純粋に技術的な問題があることが明らかになりました。 CLR仮想マシンによってさらに解釈されます。そして、まさにここで私たちは非常に問題に直面しています：一方で、コンパイラー（以下、私はMicrosoftのRoslynを意味します。これは、事実上、C＃コンパイラーの分野の標準であるためです）。テキストビューからのアセンブラー命令を認識してバイナリ表現に変換することはできません。つまり、マシン命令をバイナリ形式で直接挿入として使用する必要があります。一方、仮想マシンには独自のバイトコードがあり、認識できません。そして、彼女に提供する一連のコマンドを実行します。

この問題の理論的な解決策は明白です。仮想マシンの解釈をバイパスして、バイナリ挿入コードがプロセッサによって実行されるようにする必要があります。頭に浮かぶ最も簡単なことは、バイナリコードをバイトの配列として格納することです。これには、適切なタイミングで制御が何らかの方法で転送されます。ここから最初のタスクが見えてきます。任意のメモリ領域に含まれているものに制御を移す方法を考え出す必要があります。

最初のプロトタイプ：配列を「呼び出す」


このタスクは、おそらく挿入に対する最も深刻な障害です。言語ツールを使用すると、配列へのポインターを簡単に取得できますが、C＃の世界ではポインターはデータ上にのみ存在し、たとえば関数へのポインターに変換して、後で呼び出すことができるようにすることはできません（まあ、少なくとも私は方法を理解できませんでした）する）。

幸いなことに（または残念なことに）月の下では何も新しくなく、Yandexで「C＃」および「アセンブラーの挿入」という単語をすばやく検索すると、]] [Aker]雑誌の2007年12月号に記事が表示されました。そこから関数を正直にコピーして私のニーズに合わせて、私は得ました

[DllImport("kernel32.dll")]

public void* InvokeAsm(void* firstAsmArg, void* secondAsmArg, byte[] code)
{
    int i = 0;
    int* p = &i;
    p += 0x14 / 4 + 1;
    i = *p;
    fixed (byte* b = code)
    {
        *p = (int)b;
        uint prev;
        VirtualProtect((int*)b, (uint)code.Length, 0x40, &prev);
    }
    return (void*)i;
}

このコードの主なアイデアはInvokeAsm()、スタック上の関数からの戻りアドレスを、制御を転送する必要があるバイト配列のアドレスに置き換えることです。次に、関数を終了した後、プログラムの実行を続けるのではなく、バイナリコードの実行を開始します。

で働く魔法に対処しましょうInvokeAsm()さらに詳細に。まず、ローカル変数を宣言します。これはもちろんスタックに表示され、次にそのアドレスを取得します（それによってスタックの先頭のアドレスを取得します）。次に、スタックの最上部を基準にした戻りアドレスのオフセットをデバッガーで入念に計算して得られた特定のマジック定数をそれに追加し、戻りアドレスを保存して、代わりにバイト配列のアドレスを書き込みます。戻りアドレスを保存することの神聖な意味は明らかです。挿入後もプログラムを実行し続ける必要があります。つまり、その後に制御を移す場所を知る必要があります。次に、kernel32.dllライブラリからWinAPI関数が呼び出されます-VirtualProtect()。挿入コードが配置されているメモリページの属性を変更するために必要です。もちろん、プログラムをコンパイルすると、データセクションに表示され、対応するメモリページには読み取りおよび書き込みアクセス権があります。また、そのコンテンツを実行する権限を追加する必要があります。最後に、保存されている実際の返信アドレスを返します。もちろん、InvokeAsm()このアドレスは原因となったコードには返されません。return (void*)i;挿入の「失敗」の直後に実行されます。ただし、仮想マシンが使用する呼び出し規約（最適化を無効にしたstdcallと有効にしたfastcall）は、EAXレジスタを通じて値を返すことを意味します。挿入から戻るには、push eax（コード0x50）とret（コード0xC3）の2つの指示に従う必要があります。

安全は安全ではない


もちろん、（この単語がC＃で使用されるという意味で）セキュリティについては何の問題もありません。ただし、上記の方法はInvokeAsm()ポインターで動作します。つまり、キーワードunsafeでマークされたブロックからのみ呼び出すことができるため、必ずしも便利ではありません。少なくとも/ unsafeスイッチ（またはVSのプロジェクトプロパティの対応するチェックマーク）を使用してコンパイルする必要があります。したがって、少なくともIntPtr（最悪の場合）を操作するシェルを提供することは論理的であるように思われ、理想的には、転送された型と返された型をユーザーが完全に指定できるようにします。まあ、それはジェネリックのように聞こえます、私たちはジェネリックを書きます、他に何が話したいと尋ねますか？実際には-何かがあります。

最も明白なのは、型が不明な引数へのポインターを取得する方法ですか？タイプ構造T* ptr = &argC＃では許可されていません。また、一般的に、理由を理解することは難しくありません。ユーザーは、マネージ型の1つを、型パラメーターとして取得できないポインターを使用する可能性があります。解決策は型パラメーターを制限することunmanagedですが、1つはC＃7.3でのみ表示され、2つ目は文字列と配列を引数として渡すことを許可しませんが、演算子fixedはそれらを使用できます（最初の文字またはそれぞれ配列要素）。まあ、それに加えて、ユーザーに制御された型で操作する機会も与えたいと思います-言語の規則に違反し始めたので、最後まで違反します！

管理オブジェクトへのポインターとポインターによるオブジェクトの取得


そして再び、あまり実りのない審議の後、私は最終的な解決策を探し始めました。今回はハブレに関する記事が役に立ちました。つまり、C＃ではなくILで直接、補助ライブラリを作成する方法が提案されています。そのタスクは、オブジェクト（実際にはオブジェクトへの参照）を仮想マシンスタックに置き、引数として渡してから、スタックから何か（たとえば、数値またはIntPtr。同じ手順を逆の順序で実行することにより、ポインター（例えば、アセンブラー挿入から戻された）をオブジェクトに変換できます。この方法は、発生するすべてのことが明確で透過的であるため、優れています。しかし、マイナス点があります。できるだけ少ないファイルで済ませたかったので、個別のライブラリを作成する代わりに、メインのライブラリにILコードを埋め込むことにしました。私が見つけた唯一の方法は、C＃でスタブメソッドを記述し、プロジェクトをビルドし、ildasmを使用してバイナリを逆アセンブルし、スタブメソッドコードを書き直して、すべてをilasmと一緒に戻すことです。これらはかなりの数の追加アクションであり、コードに変更を加えた後、ビルドするたびにそれらを実行する必要があることを考えると、一般に、すぐに飽きてきて、代替案を探し始めました。

ちょうどそのとき、素晴らしい本が私の手に渡りました。そのおかげで、私は自分のために多くのことを学びました-Jeffrey Richterによる「CLR via C＃」。その中で、20章のどこかで、オブジェクトと列挙の要素の1つを取るGCHandleメソッドAlloc()を持つ構造について話していましたGCHandleType。したがって、目的のオブジェクトとを渡してこのメ​​ソッドを呼び出すとGCHandle.Pinned、メモリ内のこのオブジェクトのアドレスを取得できます。さらに、呼び出しの前に、GCHandle.Free()オブジェクトは固定されています。ガベージコレクターの影響から完全に保護されています。ただし、特定の問題があります。まず、GCHandle「ポインタ→オブジェクト」の変換を実行することはまったく効果がなく、「オブジェクト→ポインタ」だけが役立ちます。使用することがより重要GCHandleType.Pinnedアドレスを受け取りたいオブジェクトのクラスまたは構造には属性が必要ですが、[StructLayout(LayoutKind.Sequential)]デフォルトではそれが使用されLayoutKind.Autoます。したがって、このメソッドは、一部の標準タイプと、これを念頭に置いて最初に設計されたカスタムタイプにのみ適しています。私たちが見つけたいと思っている普遍的な方法とは正確には違いますよね？

もう一度やり直してください。今それにもかかわらず、ロズリンによってサポートされている、2つの文書化されていない機能にの注意を払ってみましょう：__makeref()と__refvalue()。最初のメソッドはオブジェクトを受け取りTypedReference、オブジェクトとその型への参照を格納する構造のインスタンスを返します。2番目のメソッドは、送信されたインスタンスからオブジェクトを抽出しますtypedReference。これらの機能がなぜ私たちにとって重要なのですか？なぜならTypedReference-構造！ディスカッションのコンテキストでは、これは、それへのポインターを取得できることを意味します。これは、組み合わせて、この構造の最初のフィールドへのポインターになります。つまり、関心のあるオブジェクトへのリンクそのものが格納されます。次に、管理対象オブジェクトへのポインタを取得するには、返されるものへのポインタによって値を読み取り、__makeref()それをポインタに変換する必要があります。ポインタでオブジェクトを取得するに__makeref()は、必要なタイプの条件付きで空のオブジェクトから呼び出し、返されたインスタンスへのポインタを取得TypedReferenceし、そのオブジェクトへのポインタを書き込んでから呼び出す必要があり__refvalue()ます。結果は次のコードのようになります。

public static Tout ToInstance<Tout>(IntPtr ptr)
{
    Tout temp = default;
    TypedReference tr = __makeref(temp);
    Marshal.WriteIntPtr(*(IntPtr*)(&tr), ptr);
    Tout instance = __refvalue(tr, Tout);
    return instance;
}
