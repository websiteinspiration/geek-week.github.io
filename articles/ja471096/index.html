公開テスト：イーサリアムのプライバシーとスケーラビリティのソリューション

ブロックチェーンは、人間の生活の多くの領域の改善を約束する革新的なテクノロジーです。実際のプロセスと製品をデジタル空間に転送し、金融取引の速度と信頼性を保証し、コストを削減し、分散ネットワークでスマートコントラクトを使用して最新のDAPPアプリケーションを作成することもできます。

ブロックチェーンの多くの利点と多様な用途を考えると、この有望な技術がまだすべてのセクターに浸透していないのは奇妙に思われるかもしれません。問題は、現代の分散型ブロックチェーンにはスケーラビリティがないことです。 Ethereumは1秒あたり約20のトランザクションを処理しますが、これは今日の動的ビジネスのニーズを満たすには不十分です。同時に、ブロックチェーンテクノロジーを使用している企業は、ハッキングやネットワーク障害に対する高度な保護のため、あえてEthereumを放棄することはありません。

Opchainの開発チームであるスケーラビリティトリレンマを解決し、ブロックチェーンの分散化、セキュリティ、およびスケーラビリティを確保するため 作成されたプラズマキャッシュ-Node.jsに基づくスマートコントラクトとプライベートネットワークで構成される補助チェーンで、定期的にそのステータスをルートチェーン（Ethereum）に転送します。



プラズマキャッシュの主要プロセス


1.ユーザーは、スマートコントラクトの関数「deposit」を呼び出して、ETHの金額をそこに転送します。この金額は、Plasma Cashトークンに入れたいものです。スマートコントラクト関数は、トークンを作成し、それに関するイベントを生成します。

2.スマートコントラクトのイベントにサブスクライブしたプラズマキャッシュノードは、デポジットの作成に関するイベントを受け取り、トークンの作成に関するトランザクションをプールに追加します。

3。時々、特別なプラズマキャッシュノードがプールからすべてのトランザクション（最大100万個）を取得し、それらからブロックを形成して、マークルツリーとそれに応じてハッシュを計算します。このブロックは、検証のために他のノードに送信されます。ノードは、マークルハッシュが有効かどうか、トランザクションが有効かどうか（たとえば、トークンの送信者がその所有者かどうか）をチェックします。ブロックを検証した後、ノードはスマートコントラクトの「submitBlock」関数を呼び出します。この関数は、ブロックの番号とマークルハッシュをトレースチェーンに格納します。スマートコントラクトは、ブロックの正常な追加に関するイベントを生成します。トランザクションはプールから削除されます。

4.ブロックの送信に関するイベントを受け取ったノードは、ブロックに追加されたトランザクションを適用し始めます。

5。ある時点で、トークンの所有者（または非所有者）がプラズマキャッシュからトークンを引き出したいと考えています。これを行うには、彼は `startExit`関数を呼び出して、トークンの最後の2つのトランザクションに関する情報をそれに渡し、彼がトークンの所有者であることを確認します。スマートコントラクトは、Merkleハッシュを使用して、ブロック単位でトランザクションをチェックし、トークンを出力に送信します。これは2週間で行われます。

6.トークンの引き出し操作が違反して発生した場合（引き出し手順の開始後にトークンが消費されたか、トークンが引き出しの見知らぬ人であった場合）、トークンの所有者は2週間以内に引き出しに異議を申し立てることができます。



プライバシーは2つの方法で実現されます。


1.ルートチェーンは、子チェーン内で生成および転送されるトランザクションについては何も知りません。プラズマキャッシュとの間でETHを開始および撤退したユーザーに関する情報は残ります。

2.子チェーンを使用すると、zk-SNARKを使用して匿名トランザクションを整理できます。

障害物


プラズマキャッシュの開発中に、次の問題が発生しましたが、徐々に解決して解決しました

。1 .システムのさまざまな機能間の相互作用の競合。たとえば、トランザクションをプールに追加する機能はブロックの送信と検証をブロックし、逆もまた同様であり、速度の低下につながりました。

2.大量のトランザクションを送信し、同時にデータ転送のコストを最小化する方法はすぐには明らかになりませんでした。

3.高い結果を得るために、データをどこにどのように保存するかは明確ではありませんでした。

4. 100万トランザクションのブロックサイズは約100 MBを必要とするため、ノード間のネットワークを構成する方法は明確ではありませんでした。

5。 シングルスレッドモードで作業すると、長い計算（たとえば、マークルツリーの構築とそのハッシュの計算）が発生すると、ノード間の接続が切断されます。

これをどのように処理しましたか？


プラズマキャッシュノードの最初のバージョンは、トランザクションの受け入れ、ブロックの送信と検証、データにアクセスするためのAPIの提供など、すべてを同時に実行できる一種のコンバイナーでした。 NodeJSは最初はシングルスレッドだったので、重いマークルツリー計算機能がトランザクションの追加機能をブロックしていました。この問題を解決するための2つのオプションを確認しました

。1.それぞれが特定の機能を実行する複数のNodeJSプロセスを実行します。

2. worker_threadsを使用して、コードの実行をスレッドに入れます。

その結果、両方のオプションを同時に使用しました。1つのノードを論理的に3つの部分に分割しました。これらは別々に動作することができますが、同時に同期して

1です。トランザクションを受け入れてブロックを作成するノードを送信します。

2.ノードの有効性を検証する検証ノード。

3.ノードAPI-データにアクセスするためのAPIを提供します。

同時に、cliを使用してUNIXソケットを介して各ノードに接続できます。

マークルツリーの計算などの重い操作は、別のストリームで実行しました。

このようにして、すべてのプラズマキャッシュ機能の正常な動作を同時に、障害なく達成しました。

システムが機能するようになり次第、速度のテストを開始しましたが、残念ながら、1秒あたり5,000トランザクション、ブロック内で最大50,000トランザクションという、満足のいく結果が得られませんでした。何が正しく実装されていないのかを知る必要がありました。

まず、システムのピーク能力を確認するために、プラズマキャッシュとの通信メカニズムのテストを開始しました。以前に、Plasma CashノードがUNIXソケットインターフェイスを提供することを書きました。もともとはテキストでした。jsonオブジェクトは、 `JSON.parse（）`と `JSON.stringify（）`を使用して送信されました。

データベースへの記録


当初、Redisは、私たちの要件を満たす最も生産的なソリューションの1つとしてデータストレージ用に選択されました：キー値ストレージ、ハッシュテーブルの操作などです。redis-benchmarkを起動し、1パイプラインモードで1秒あたり約8万回のオペレーションを取得しました。

パフォーマンスを高めるために、Redisをより細かく調整しました。

  • UNIXソケット接続を確立しました。
  • 状態のディスクへの保存を無効にします（信頼性のために、レプリカを構成して別のRedisでディスクに保存することができます）。

Redisでは、1つのリクエストですべてのトランザクションを受信し、トランザクションを1つずつ削除する機能が必要なため、プールはハッシュテーブルです。通常のリストを使用しようとしましたが、リスト全体をアンロードすると、動作が遅くなります。

標準のNodeJSライブラリを使用して、Redisライブラリは毎秒18,000トランザクションのパフォーマンスを達成しました。速度が9回低下した。

ベンチマークは5倍以上の可能性を明らかに示したので、最適化を始めました。ライブラリをioredisに変更し、毎秒25kのパフォーマンスを得ました。 `hset`コマンドを使用して、トランザクションを1つずつ追加しました。したがって、Redisで多くのリクエストを生成しました。トランザクションをバンドルにマージし、1つのhmsetコマンドで送信するというアイデアがありました。結果は1秒あたり32kです。

以下で説明するいくつかの理由により、 `Buffer`を使用してデータを処理しますが、実際には、書き込む前にデータをテキスト（` buffer.toString（ 'hex'） `）に変換すると、追加のパフォーマンスを得ることができます。したがって、速度は1秒あたり35kに増加しました。現時点では、さらなる最適化を一時停止することを決定しました。

次の理由により、バイナリプロトコルに切り替える必要がありました

。1 .システムはハッシュや署名などを計算することが多く、そのために `Buffer。

2.サービス間で転送する場合、バイナリデータはテキストよりも軽量です。たとえば、100万トランザクションのブロックを送信する場合、テキスト内のデータは300メガバイトを超える可能性があります。

3.継続的なデータ変換はパフォーマンスに影響します。

したがって、私たちは、驚くべきライブラリー「binary-data」に基づいて開発された、データを格納および送信するための独自のバイナリプロトコルを基礎として採用しました。

通常のコマンド `BD.encode（block、Protocol）.slice（）;`および `BD.decode（buffer、Protocol）`によって、データを `Buffer`に変換してRedisに保存するか、別のノードに転送してデータを取得します。

また、サービス間でデータを転送するための2つのバイナリプロトコルがあります。

-UNIXソケットを介してプラズマノードと対話するためのプロトコル

  ```json
  {
    type: BD.types.uint8,
    messageId: BD.types.uint24le,
    error: BD.types.uint8,
    length: BD.types.uint24le,
    payload: BD.types.buffer(({node}) => node.length)
  }
  ```

どこ：

  • ` type`-実行するアクション。たとえば、1-sendTransaction、2-getTransaction;
  • `payload`-対応する関数に転送されるデータ。
  • `messageId`-応答を識別できるようにするためのメッセージID。

-ノード間の相互作用のプロトコル

  ```json
  {
    code: BD.types.uint8,
    versionProtocol: BD.types.uint24le,
    seq: BD.types.uint8,
    countChunk: BD.types.uint24le,
    chunkNumber: BD.types.uint24le,
    length: BD.types.uint24le,
    payload: BD.types.buffer(({node}) => node.length)
  }
  ```

どこ：

  • ` code`-メッセージコード、たとえば6-PREPARE_NEW_BLOCK、7-BLOCK_VALID、8-BLOCK_COMMIT;
  • `versionProtocol`-プロトコルバージョン。異なるバージョンのノードがネットワーク上で発生し、異なる方法で動作する可能性があるためです。
  • `seq`-メッセージ識別子;
  • 大きなメッセージを圧縮するために必要な「countChunk」と「chunkNumber」。
  • 「長さ」と「ペイロード」の長さとデータ自体。

事前にデータを入力したため、最終的なシステムは、Ethereumの「rlp」ライブラリよりもはるかに高速です。残念ながら、将来的にはスマートコントラクトを確定する必要があるため、まだ拒否することはできません。毎秒35,000トランザクションの

速度を達成できた場合、それらを最適な時間で処理する必要もあります。ブロックのおおよその形成時間は30秒かかるため、ブロックに1,000,000トランザクションを含める必要があります。これは、100 mbを超えるデータを送信することを意味します。

最初は、 `ethereumjs-devp2p`ライブラリを使用してノードを通信しましたが、あまり多くのデータを処理できませんでした。その結果、 `ws`ライブラリを使用して、websocketを介したバイナリデータ転送を設定しました。もちろん、大きなデータパケットを送信するときにも問題が発生しましたが、それらをチャンクに分割したので、そのような問題はありません。

また、マークルツリーの形成と1,000,000トランザクションのハッシュの計算には、約10秒の連続計算が必要です。この間、すべてのノードへの接続が壊れました。この計算を別のスレッドに転送することが決定されました。
