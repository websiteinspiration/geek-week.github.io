ルークかそうでないか-それが問題だ



今月5月3日、「Kubernetesの分散データウェアハウスの管理システム」のメジャーリリースが発表されました-Rook 1.0.0。 1年以上前に、Rookの一般的な概要をすでに公開しています。次に、実際に使用した場合の経験についてお話を伺うように依頼されました。今、プロジェクトの歴史におけるこのような重要なマイルストーンに間に合うように、蓄積された印象を共有できることを嬉しく思います。

つまり、Rookは、Ceph、EdgeFS、Minio、Cassandra、CockroachDBなどのストレージソリューションの導入、管理、自動復旧を完全に制御するKubernetes のオペレーターセットです。

現時点では、ほとんどの開発（そしてのみで安定した段階）溶液であるルーク・CEPHオペレータ。

注：Cephに関連するRook​​ 1.0.0のリリースの重要な変更点の中で、Ceph Nautilusのサポートと、CephFSまたはRGWバケットにNFSを使用する機能に注目できます。とりわけ、ベータレベルへのEdgeFSサポートの「成熟」が際立っています。

したがって、この記事では次のことを行います。

  • Rookを使用してCephをKubernetesクラスターにデプロイすることで得られる利点についての質問に答えてください。
  • 本番環境でRookを使用した経験と印象を共有します。
  • Rookに「はい！」と言う理由と、Rookの計画について説明します。

一般的な概念と理論から始めましょう。

「私は1人のルークの利点を持っています！」（未知のチェスプレイヤー）




Rookの主な利点の1つは、データウェアハウスとの対話がKubernetesメカニズムを通じて実行されることです。つまり、Cephを構成するためのコマンドをリーフレットからコンソールにコピーする必要がなくなります。

-CephFSクラスターにデプロイしたいですか？ yamlファイルを書くだけです！
- 何？ S3 APIを使用してオブジェクトストアをデプロイしたいですか？ 2番目のyamlファイルを書くだけです！

Rookは、一般的な演算子のすべてのルールによって作成されます。それとの相互作用は、必要なCephエンティティの特性を記述するCRD（カスタムリソース定義）を使用して行われます（これは唯一の安定した実装であるため、特に明記されていない限り、デフォルトで記事はCephについて説明します）。指定されたパラメーターに従って、オペレーターはセットアップに必要なコマンドを自動的に実行します。

オブジェクトストアを作成する例を使用して、詳細を見てみましょうCephObjectStoreUser。

apiVersion: ceph.rook.io/v1
kind: CephObjectStore
metadata:
  name: {{ .Values.s3.crdName }}
  namespace: kube-rook
spec:
  metadataPool:
    failureDomain: host
    replicated:
      size: 3
  dataPool:
    failureDomain: host
    erasureCoded:
      dataChunks: 2
      codingChunks: 1
  gateway:
    type: s3
    sslCertificateRef:
    port: 80
    securePort:
    instances: 1
    allNodes: false
---
apiVersion: ceph.rook.io/v1
kind: CephObjectStoreUser
metadata:
  name: {{ .Values.s3.crdName }}
  namespace: kube-rook
spec:
  store: {{ .Values.s3.crdName }}
  displayName: {{ .Values.s3.username }}

リストに示されているパラメーターはかなり標準的なものであり、コメントを必要とする可能性はほとんどありませんが、テンプレート変数で強調表示されているパラメーターには特別な注意を払う必要があります。

作業の一般的なスキームは、オペレータが必要なコマンドと戻り、我々は仕事を続けることができたと私たち「ではない本当の」秘密を、実行するためにYAMLファイルを通じて、私たち「順序」のリソース、その事実に帰着する（下記参照します）。そして、上記の変数から、コマンドとシークレットの名前が構成されます。

これはどんなチームですか？オブジェクトストレージ用のユーザーを作成する場合、ポッド内のRookステートメントは次のことを行います。

radosgw-admin user create --uid="rook-user" --display-name="{{ .Values.s3.username }}"

このコマンドの結果は、JSON構造になります。

{
    "user_id": "rook-user",
    "display_name": "{{ .Values.s3.username }}",
    "keys": [
        {
           "user": "rook-user",
           "access_key": "NRWGT19TWMYOB1YDBV1Y",
           "secret_key": "gr1VEGIV7rxcP3xvXDFCo4UDwwl2YoNrmtRlIAty"
        }
    ],
    ...
}

Keys-S3 APIを介してオブジェクトストレージにアクセスするために将来のアプリケーションが必要とするもの。Rookオペレーターは親切にそれらを選択し、名前付きのシークレットとして名前空間に入れますrook-ceph-object-user-{{ $.Values.s3.crdName }}-{{ $.Values.s3.username }}。

このシークレットのデータを使用するには、それらを環境変数としてコンテナーに追加するだけです。例として、ユーザー環境ごとにバケットを自動的に作成するジョブのテンプレートを示します。

{{- range $bucket := $.Values.s3.bucketNames }}
apiVersion: batch/v1
kind: Job
metadata:
  name: create-{{ $bucket }}-bucket-job
  annotations:
    "helm.sh/hook": post-install
    "helm.sh/hook-weight": "2"
spec:
  template:
    metadata:
      name: create-{{ $bucket }}-bucket-job
    spec:
      restartPolicy: Never
      initContainers:
      - name: waitdns
        image: alpine:3.6
        command: ["/bin/sh", "-c", "while ! getent ahostsv4 rook-ceph-rgw-{{ $.Values.s3.crdName }}; do sleep 1; done" ]
      - name: config
        image: rook/ceph:v1.0.0
        command: ["/bin/sh", "-c"]
        args: ["s3cmd --configure --access_key=$(ACCESS-KEY) --secret_key=$(SECRET-KEY) -s --no-ssl --dump-config | tee /config/.s3cfg"]
        volumeMounts:
        - name: config
          mountPath: /config
        env:
        - name: ACCESS-KEY
          valueFrom:
            secretKeyRef:
              name: rook-ceph-object-user-{{ $.Values.s3.crdName }}-{{ $.Values.s3.username }}
              key: AccessKey
        - name: SECRET-KEY
          valueFrom:
            secretKeyRef:
              name: rook-ceph-object-user-{{ $.Values.s3.crdName }}-{{ $.Values.s3.username }}
              key: SecretKey
      containers:
      - name: create-bucket
        image: rook/ceph:v1.0.0
        command: 
        - "s3cmd"
        - "mb"
        - "--host=rook-ceph-rgw-{{ $.Values.s3.crdName }}"
        - "--host-bucket= "
        - "s3://{{ $bucket }}"
        ports:
        - name: s3-no-sll
          containerPort: 80
        volumeMounts:
        - name: config
          mountPath: /root
      volumes:
      - name: config
        emptyDir: {}
---
{{- end }}

このジョブにリストされているすべてのアクティビティは、Kubernetesを超えることなく実行されました。YAMLファイルに記述されている構造は、Gitリポジトリーに折りたたまれ、繰り返し再利用されます。これには、DevOpsエンジニアとCI / CDプロセス全体にとって大きなプラスがあります。

ルークとラドスとの喜び


Ceph + RBDの束を使用すると、ポッドへのボリュームのマウントに特定の制限が課されます。

特に、名前空間には、Cephにアクセスするためのシークレットが必要です。これにより、ステートフルアプリケーションを機能させることができます。名前空間に2つから3つの環境がある場合は問題ありません。シークレットを手動でコピーできます。しかし、各開発者機能が独自の名前空間を持つ個別の環境を作成するとしたらどうでしょうか

この問題は、シークレットを新しい名前空間に自動的にコピーするshell-operatorを使用して解決しました（このようなフックの例については、この記事で説明しています）。

#! /bin/bash

if [[ $1 == “--config” ]]; then
   cat <<EOF
{"onKubernetesEvent":[
 {"name": "OnNewNamespace",
  "kind": "namespace",
  "event": ["add"]
  }
]}
EOF
else
    NAMESPACE=$(kubectl get namespace -o json | jq '.items | max_by( .metadata.creationTimestamp ) | .metadata.name')
    kubectl -n ${CEPH_SECRET_NAMESPACE} get secret ${CEPH_SECRET_NAME} -o json | jq ".metadata.namespace=\"${NAMESPACE}\"" | kubectl apply -f -
fi

ただし、Rookを使用する場合、この問題は存在しません。マウントプロセスは、FlexvolumeまたはCSI（まだベータ版）に基づく独自のドライバーを使用して行われるため、シークレットは必要ありません。

Rookは多くの問題を自動的に解決するため、新しいプロジェクトで使用することを奨励しています。

ルークの包囲


独自の実験を実施する可能性のために、RookとCephを配備して実用的な部分を完了します。この難攻不落の塔を襲撃するための方が簡単でした。開発者はHelmパッケージを用意しました。それをダウンロードしましょう：

$ helm fetch rook-master/rook-ceph --untar --version 1.0.0

ファイルには、rook-ceph/values.yamlさまざまな設定が含まれています。最も重要なことは、エージェントと検索の許容範囲を指定することです。テイント/許容メカニズムを使用できる理由については、この記事で詳しく説明します。

つまり、クライアントアプリケーションを備えたポッドを、データストレージ用のディスクが配置されているのと同じノードに配置したくありません。その理由は単純です。このようにして、Rookエージェントの作業がアプリケーション自体に影響を与えることはありません。

したがって、rook-ceph/values.yamlお気に入りのエディターでファイルを開き、次のブロックを最後に追加します。

discover:
  toleration: NoExecute
  tolerationKey: node-role/storage
agent:
  toleration: NoExecute
  tolerationKey: node-role/storage
  mountSecurityMode: Any

データストレージ用に予約されているノードごとに、対応する汚染を追加します。

$ kubectl taint node ${NODE_NAME} node-role/storage="":NoExecute

次に、次のコマンドでHelm-chartをインストールします。

$ helm install --namespace ${ROOK_NAMESPACE} ./rook-ceph

次に、クラスターを作成し、OSDの場所を指定する必要があります。

apiVersion: ceph.rook.io/v1
kind: CephCluster
metadata:
  clusterName: "ceph"
  finalizers:
  - cephcluster.ceph.rook.io
  generation: 1
  name: rook-ceph
spec:
  cephVersion:
    image: ceph/ceph:v13
  dashboard:
    enabled: true
  dataDirHostPath: /var/lib/rook/osd
  mon:
    allowMultiplePerNode: false
    count: 3
  network:
    hostNetwork: true
  rbdMirroring:
    workers: 1
  placement:
    all:
      tolerations:
      - key: node-role/storage
        operator: Exists
  storage:
    useAllNodes: false
    useAllDevices: false
    config:
      osdsPerDevice: "1"
      storeType: filestore
    resources:
      limits:
        memory: "1024Mi"
      requests:
        memory: "1024Mi"
    nodes:
    - name: host-1
      directories:
      - path: "/mnt/osd"
    - name: host-2
      directories:
      - path: "/mnt/osd"
    - name: host-3
      directories:
      - path: "/mnt/osd"

Cephのステータスを確認します-表示される予定HEALTH_OKです：

$ kubectl -n ${ROOK_NAMESPACE} exec $(kubectl -n ${ROOK_NAMESPACE} get pod -l app=rook-ceph-operator -o name -o jsonpath='{.items[0].metadata.name}') -- ceph -s

同時に、クライアントアプリケーションを含むポッドがCeph用に予約されたノードに到達しないことを確認します。

$ kubectl -n ${APPLICATION_NAMESPACE} get pods -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName

さらにオプションのコンポーネントが構成されます。それらの詳細については、ドキュメントに記載されています。管理のために、ダッシュボードとツールボックスをインストールすることを強くお勧めします。
