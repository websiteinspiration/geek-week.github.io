PostgreSQLからErlangへの論理レプリケーション

システムを開発するとき、メインの処理ロジックがアプリケーション（この場合はErlang）に集中しており、このアプリケーションのデータ（設定、ユーザープロファイルなど）がデータベース（PostgreSQL）にある場合の、かなり典型的なスキーム。Erlangアプリケーションは、永続的なリクエストを拒否することで処理を高速化し、データベースの負荷を軽減するために、ETSに設定をキャッシュします。この場合、このデータの変更は、別の（おそらく外部の）サービスを介して行われます。


このような状況では、キャッシュされたデータを最新に保つという課題が生じます。この問題を解決するには、さまざまな方法があります。その1つがPostgreSQLの論理レプリケーションです。彼については、以下で説明します。


ストリーム論理複製プロトコル


論理レプリケーションは、PostgreSQLストリーミングレプリケーションプロトコルを使用して、WALログを読み取り、必要なテーブルをフィルタリングし、これらの変更をサブスクライバーに送信することにより、PostgreSQLテーブルのデータ変更を取得します。このメカニズムは、スタンバイデータベースを作成するための物理レプリケーションに使用されるメカニズムに似ています。


論理複製には次の利点があります。


  • リアルタイムで遅延なく変更を受け取る。
  • 受信した変更をテーブルおよび操作（INSERT / DELETE / UPDATE）でフィルタリングする。
  • 加入者が受信したデータの完全性と完全性。サブスクライバーは、データベースで発生したのと同じ順序で変更を受け取ります。
  • サブスクライバーが一時的に停止してもデータは失われません。PostgreSQLはレプリケーションが停止した場所を記憶しています。

データベースの準備


論理レプリケーションを使用するには、サーバーからのWALレコードをより便利な形式でデコードするプラグインが必要です。
PostgreSQL 10より前のバージョンでは、/拡張pglogical_outputプラグインを使用できます。
PostgreSQL 10のpgoutputプラグインから始めます。

    {ok, ReplConn} = epgsql:connect(Host, User, Password, [{database, DBName}, {port, Port}, {replication, "database"}]),
