SwiftUIで考え方を変える



先週、コミュニティが実証済みの開発パターンをUIKitからSwiftUIに転送しようとしていることを確認しました。しかし、SwiftUIを使用して効率的なコードを記述する最善の方法は、UIKitに関するすべてを忘れて、ユーザーインターフェイス開発に関する考え方を完全に変えることだと私は確信しています。今週は、UIKitとSwiftUIを使用した開発の主な違いについて説明します。

違いの定義


UIKitは、iOSプラットフォームのユーザーインターフェイスを作成するための必須のイベント駆動型フレームワークです。これは、ビューの読み込み、ボタンのクリックなどのイベント中にすべての状態変更を処理する必要があることを意味します。このアプローチの大きな欠点は、ユーザーインターフェイスとその状態を同期することが難しいことです。状態が変化したら、ビューを手動で追加/削除/表示/非表示にして、アプリケーションの現在の状態と同期する必要があります。

スイフトイすべてのAppleプラットフォームのユーザーインターフェイスを作成するための宣言型フレームワークです。この場合のキーワードは宣言型です。宣言的とは、開発者が達成したいことを宣言する必要があることを意味し、フレームワークがこれを処理します。フレームワークは、ユーザーインターフェイスをレンダリングする最適な方法を認識しています。

UI = f(state)

SwiftUIでは、ユーザーインターフェイスはその状態の関数です。つまり、ビューの状態が変化するたびに、そのbodyプロパティが再カウントされ、新しいビューが生成されます。次の例を考えてみましょう。

struct ContentView: View {
    @ObservedObject var store: Store

    var body: some View {
        Group {
            if store.isLoading {
                Text("Loading...")
                    .font(.subheadline)
            } else {
                Image("photo")
                    .font(.largeTitle)
            }
        }
    }
}

上記の例では、「Loading ...」というテキストを表示し、ダウンロードの最後に画像を表示するビューがあります。ObserverObjectはこのビューの状態であり、変化するとすぐに、SwiftUIはbodyプロパティを再計算して新しいビューを割り当てます。 UIKitを使用した開発中は、プレゼンテーション階層の要素を手動で非表示/表示する必要がありますが、SwiftUIでは、ローディングインジケーターを追加/削除する必要はありません。 SwiftUIには、ビューの状態を説明する方法がいくつかあります。ビューの詳細については、「SwiftUIのプロパティラッパー」を参照してください。

次に、ビューの状態が変化したときに何が起こるかを詳しく見ていきます。SwiftUIには現在のビューの階層の痕跡があり、その状態が変化するとすぐに新しいビューを計算します。SwiftUIは比較アルゴリズムを使用して違いを理解し、必要なビューを自動的に追加/削除/更新します。デフォルトでは、SwiftUIは標準のイン/アウト遷移画像を使用してビューを表示/非表示にしますが、他のアニメーションへの遷移を手動で変更することもできます。SwiftUIの遷移とアニメーションの詳細については、私の記事「SwiftUIのアニメーション」を参照してください。

プレゼンテーション階層


ビューの階層と、SwiftUIが実際にビューの構造を表示する方法について話しましょう。最初に触れておきたいのは、SwiftUIはビューの構造を1対1のマッピングで表示しないことです。ビューコンテナーはいくつでも使用できますが、最終的には、SwiftUIはレンダリングに意味のあるビューのみを表示します。つまり、それらのロジックを小さな表現に抽出し、それらを作成してアプリケーションで再利用できます。心配しないでください。この場合のアプリケーションのパフォーマンスは低下しません。 SwiftUIでビューを作成する方法の詳細については、この記事をご覧ください。

SwiftUIの複雑な表現の階層を理解する最良の方法は、そのタイプを表示することです。SwiftUIは、静的型システムを使用して、違いを十分にすばやく判別します。まず、ビューのタイプをチェックしてから、ビューのコンポーネントの値をチェックします。作業コードでリフレクションを使用するのは好きではありませんが、学習プロセスでは非常に役立ちます。

print(Mirror(reflecting: ContentView(store: .init()).body))
