PVS-StudioがChocolateyになりました：Azure DevOpsからChocolateyをチェックしています


PVS-Studioをより便利に使用できるようにしていきます。アナライザーは、WindowsのパッケージマネージャーであるChocolateyで利用できるようになりました。これにより、特にクラウドサービスでのPVS-Studioの導入が容易になると確信しています。遠くまで行かないために、同じChocolateyのソースコードを確認してください。CIシステムはAzure DevOpsになります。

クラウド統合に関する他の記事のリストは次のとおりです。

  • PVS-Studioがクラウドに移行：Azure DevOps
  • PVS-Studioがクラウドに移行：Travis CI
  • PVS-Studioがクラウドに移行：CircleCI
  • PVS-Studioがクラウドに移行：GitLab CI / CD

Azure DevOpsとの統合に関する最初の記事に注意を払うことをお勧めします。この場合、重複しないようにいくつかのポイントが省略されているためです。

したがって、この記事のヒーロー：

PVS-Studioは、C、C ++、C＃、およびJavaで記述されたプログラムのエラーと潜在的な脆弱性を検出するように設計された静的コード分析ツールです。 Windows、Linux、およびmacOSの64ビットシステムで動作し、32ビット、64ビット、および組み込みARMプラットフォーム用に設計されたコードを分析できます。静的コード分析でプロジェクトを検証するのが初めての場合は、PVS-Studioからの最も興味深い警告をすばやく確認し、このツールの機能を評価する方法に関する記事を読むことをお勧めします。

Azure DevOps-開発プロセス全体を共同でカバーする一連のクラウドサービス。このプラットフォームには、Azure Pipelines、Azure Boards、Azure Artifacts、Azure Repos、Azure Test Plansなどのツールが含まれており、ソフトウェアの作成プロセスを加速して品質を向上させます。

Chocolateyは、Windows用のオープンソースパッケージマネージャーです。プロジェクトの目標は、Windowsオペレーティングシステムでのインストールから更新、アンインストールまでのソフトウェアライフサイクル全体を自動化することです。

Chocolateyの使用について


このリンクから、パッケージマネージャーのインストール方法を確認できます。アナライザーのインストールに関する完全なドキュメントは、「Chocolateyパッケージマネージャーを使用したインストール」セクションのリンクから入手できます。いくつかのポイントを簡単に繰り返します。

最新のアナライザーバージョンをインストールするコマンド：

choco install pvs-studio

PVS-Studioパッケージの特定のバージョンをインストールするコマンド：

choco install pvs-studio --version=7.05.35617.2075

デフォルトでは、アナライザーコア（コアコンポーネント）のみがインストールされます。他のすべてのフラグ（スタンドアロン、JavaCore、IDEA、MSVS2010、MSVS2012、MSVS2013、MSVS2015、MSVS2017、MSVS2019）は、-package-parametersを使用して渡すことができます。

Visual Studio 2019のプラグインを使用してアナライザーをインストールするコマンドの例：

choco install pvs-studio --package-parameters="'/MSVS2019'"

次に、Azure DevOpsでのアナライザーの便利な使用例を見てみましょう。

カスタマイズ

次に、アナライザーライセンスを含むファイルを作成する必要があります。ここで、PVSNAMEとPVSKEYは、設定で値を指定する変数の名前です。PVS-Studioのログインとライセンスキーを保存します。それらの値を設定するには、[変数 ] > [新しい変数 ]メニューを開きます。ログイン用のPVSNAME変数とアナライザーキー用のPVSKEYを作成します。PVSKEYの[ この値を秘密にしておく] チェックボックスを必ずオンにしてください。チームコード：

all "C:\Program Files (x86)\PVS-Studio\PVS-Studio_Cmd.exe" credentials 
–u $(PVSNAME) –n $(PVSKEY)

リポジトリにあるbatファイルを使用してプロジェクトをビルドしてみましょう。

all build.bat

アナライザーの結果を含むファイルが置かれるフォルダーを作成します。

all mkdir PVSTestResults

プロジェクト分析を実行します。

all "C:\Program Files (x86)\PVS-Studio\PVS-Studio_Cmd.exe" 
–t .\src\chocolatey.sln –o .\PVSTestResults\Choco.plog

Plogonverterユーティリティを使用して、レポートをhtml形式に変換します。

all "C:\Program Files (x86)\PVS-Studio\PlogConverter.exe" 
–t html –o \PVSTestResults\ .\PVSTestResults\Choco.plog

次に、レポートをアップロードできるようにタスクを作成する必要があります。

- task: PublishBuildArtifacts@1
  inputs:
    pathToPublish: PVSTestResults
    artifactName: PVSTestResults
    condition: always()

完全な構成ファイルは次のようになります。

検証結果


警告N1

アナライザー警告：V3005「プロバイダー」変数はそれ自体に割り当てられています。CrytpoHashProviderSpecs.cs 38

public abstract class CrytpoHashProviderSpecsBase : TinySpec
{
  ....

  ....

アナライザーはそれ自体への変数の割り当てを検出しましたが、これは意味がありません。最も可能性が高いのは、これらの変数の1つの代わりに、他の変数があるはずです。まあ、またはこれはタイプミスであり、不要な割り当ては単純に削除できます。

警告N2

アナライザー警告：V3093 [CWE-480]「＆」演算子は両方のオペランドを評価します。おそらく、代わりに短絡「&&」演算子を使用する必要があります。 Platform.cs 64

public static PlatformType get_platform()
{
  switch (Environment.OSVersion.Platform)
  {
    case PlatformID.MacOSX:
    {
      ....
    }
    case PlatformID.Unix:
    if(file_system.directory_exists("/Applications")
      & file_system.directory_exists("/System")
      & file_system.directory_exists("/Users")
      & file_system.directory_exists("/Volumes"))
      {
        return PlatformType.Mac;
      }
        else
          return PlatformType.Linux;
    default:
      return PlatformType.Windows;
  }
}

＆演算子と&&演算子 の違いは、式の左側がfalseの場合、とにかく右側が計算されるということです。この場合、system.directory_existsメソッドへの追加の呼び出しを意味します。

このフラグメントでは、これは小さな欠陥です。はい、この条件は＆演算子を&&演算子に置き換えることで最適化できますが、実際には、これは何にも影響しません。ただし、他の場合では、＆と&&の混乱により、式の右側が正しくない/無効な値で機能する場合に深刻な問題が発生する可能性があります。たとえば、V3093診断で検出されたエラーのコレクションには、次のようなケースがあります。

if ((k < nct) & (s[k] != 0.0))

インデックスkが正しくない場合でも、配列要素へのアクセスに使用されます。その結果、IndexOutOfRangeExceptionがスローされます。

警告N3、N4

アナライザー警告：V3022 [CWE-571]式「shortPrompt」は常にtrueです。 InteractivePrompt.cs 101
アナライザー警告：V3022 [CWE-571]式「shortPrompt」は常にtrueです。 InteractivePrompt.cs 105

public static string 
prompt_for_confirmation(.... bool shortPrompt = false, ....)

  ....

この場合、三項演算子の奇妙な論理があります。詳しく見てみましょう。番号1でマークされた条件が満たされている場合は、条件2に進みます。これは常にtrueで、これは行3が満たされていることを意味します。条件1がfalseであることが判明した場合は、番号4でマークされている行に進みます。常にtrue、つまり5行目が実行されることを意味します。したがって、コメント0でマークされた条件が満たされることはなく、プログラマーが期待した作業のロジックとは正確には一致しない可能性があります。

警告N5

アナライザー警告：V3123[CWE-783]たぶん、 '?:'オペレーターは、予期されたものとは異なる方法で動作します。その優先順位は、その状態で他のオペレーターの優先順位よりも低くなっています。Options.cs 1019

private static string GetArgumentName (...., string description)
{
  string[] nameStart;
  if (maxIndex == 1)
  {
    nameStart = new string[]{"{0:", "{"};
  }
  else
  {
    nameStart = new string[]{"{" + index + ":"};
  }
  for (int i = 0; i < nameStart.Length; ++i) 
  {
    int start, j = 0;
    do 
    {
      start = description.IndexOf (nameStart [i], j);
    } 
    while (start >= 0 && j != 0 ? description [j++ - 1] == '{' : false);
    ....
    return maxIndex == 1 ? "VALUE" : "VALUE" + (index + 1);

診断は行で機能しました：

while (start >= 0 && j != 0 ? description [j++ - 1] == '{' : false)

変数jは上の数行でゼロに初期化されているため、三項演算子はfalseを返します。この状態のため、ループ本体は1回だけ実行されます。このコードは、プログラマーの意図したとおりにはまったく機能しないようです。

結論


そこで、もう1つ小さなステップを踏みました。PVS-Studioの使用がさらに簡単で便利になりました。Chocolateyは、コードにエラーが少ない優れたパッケージマネージャーであり、PVS-Studioを使用するとさらに少なくなる可能性があることも言いたいです。PVS-Studio

をダウンロードしてお試しください。静的アナライザーを定期的に使用すると、チームによって開発されたコードの品質と信頼性が向上し、多くのゼロデイ脆弱性の防止に役立ちます。
