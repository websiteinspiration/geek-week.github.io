UPDATEコマンドで行が変更されると、DELETEとINSERTという2つの操作が実際に実行されます。ラインの現在のバージョン、XMAXはUPDATEを行うトランザクションの数に等しく設定されます。次に、同じ行の新しいバージョンが作成されます。そのxmin値は、以前のバージョンのxmax値と一致します。
このトランザクションの完了後しばらくしてから、古いバージョンまたは新しいバージョンは、どちらCOMMIT/ROOLBACKに応じて、テーブルを通過してクリーンアップされるときに「デッド」（デッドタプル）として認識されVACUUMます。



しかし、これはすぐには起こりませんが、「デッド」の問題はすぐに発生します。大きなテーブルのレコードを複数回または大規模に更新すると、VACUUMが支援できない状況に直面します。

＃1：動かしたい


ビジネスロジックメソッドがそれ自体で機能し、あるレコードのフィールドXを更新する必要があることに突然気付いたとします。

UPDATE tbl SET X = <newX> WHERE pk = $1;

次に、処理が進むにつれて、Yフィールドも更新する必要があることがわかります。

UPDATE tbl SET Y = <newY> WHERE pk = $1;

...そしてZも-なにかささいなことをするのですか？

UPDATE tbl SET Z = <newZ> WHERE pk = $1;

現在、このレコードのバージョンはいくつデータベースにありますか？うん、4枚！これらのうち、1つは関連性があり、3つは[自動] VACUUMを取得する必要があります。

このようにしないでください！1つのリクエストですべてのフィールドの更新を使用します -ほとんどの場合、メソッドのロジックは次のように変更できます。

UPDATE tbl SET X = <newX>, Y = <newY>, Z = <newZ> WHERE pk = $1;

＃2：IS DISTINCT FROMを使用してください、ルーク！


したがって、テーブル内の多数のレコードを更新する必要がまだあります（たとえば、スクリプトまたはコンバーターの使用中）。そして、次のようなものがスクリプトに飛び込みます。

UPDATE tbl SET X = <newX> WHERE pk BETWEEN $1 AND $2;

ほぼこの形式では、空の新しいフィールドに入力するのではなく、データのエラーを修正するために、クエリが頻繁に発生します。さらに、既存のデータの正確性はまったく考慮されていませんが、無駄です！つまり、レコードが私が望んでいたものと正確に一致していても、そのレコードは書き換えられています。なぜですか？正しい：

UPDATE tbl SET X = <newX> WHERE pk BETWEEN $1 AND $2 AND X IS DISTINCT FROM <newX>;

＃3：私は...ブロックすることで私の愛する人を認識します


2つの同一の並列プロセスを 実行します。各プロセスは、「動作中」であることを記録マークに向けています。

UPDATE tbl SET processing = TRUE WHERE pk = $1;

これらのプロセスが実質的に互いに独立しているものの、1つのIDのフレームワーク内であっても、この要求では、最初のトランザクションが完了するまで2番目のクライアントが「ロックアップ」します。

解決策＃1：タスクは、前の1に縮小されています。

ただ、再び追加しますIS DISTINCT FROM：

UPDATE tbl SET processing = TRUE WHERE pk = $1 AND processing IS DISTINCT FROM TRUE;

この形式では、2番目の要求はデータベース内の何も変更せず、すでに「必要なすべてのもの」があるため、ブロッキングは発生しません。さらに、レコードが「存在しない」という事実は、適用されたアルゴリズムですでに処理されています。
