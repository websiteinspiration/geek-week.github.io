OpenShiftでのJavaアプリケーションのデプロイ

これはそうですか？私たちは、fabric8-maven-pluginをプロジェクトに接続し、OpenShiftでアプリケーションをビルドして実行します。しかし私が勉強したとき、私はこのプロセスをより理解することを望み、それからOpenShiftでアプリケーションを構築およびデプロイするプロセスをより制御し、自由にしたかったのです。したがって、次のシナリオはそのような機能で判明しました。

  • 私は自分のアーティファクトを自分のツール（maven、gradleなど）で自分で構築します
  • Dockerfileを介してDockerイメージの作成を制御します
  • Openshiftでのビルドおよびデプロイメントプロセスは、テンプレートで構成されます。コンテナの仕様、ポッドはカスタマイズ可能です。
  • したがって、プロセス自体を外部のアセンブリ、展開システムに転送できます

もちろん、このために私はOpenShift自体で利用可能な豊富な機能を使用します。
そして、私は準備済みのJavaアーティファクトを用意してから、必要な特性と機能を備えたDockerfileを準備します

Dockerfile 

FROM openjdk

MAINTAINER Rylkov Alexander <arylkov@.>

COPY demo.springboot.mvn-0.0.1-SNAPSHOT.jar /deployments/app.jar

ENV JAVA_OPTS="$JAVA_OPTS -Xms500m -Xmx1024m"

EXPOSE 8080
EXPOSE 5005

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar /deployments/app.jar" ]

したがって、フォルダーにはアーティファクトとDockerfileの2つのファイルがあります。



ファイルの構成は条件付きです。たとえば、Dockerfileでdocker imageの追加ファイルが必要な場合は、ここに追加します。これで、Dockerfileで、基本的なJavaイメージと私のSpringブートアプリケーションが指定され、2つのポートが開いているなどです。

このステップでは、アプリケーションの構築とDockerイメージのアセンブルのためのツールに選択肢/自由があります。

次に、アプリケーション用のOpenShiftテンプレートを準備します。この操作は1回限りで、パラメーター化するだけで済みます。アプリケーションを作成、起動、操作するために可能なすべてのものが含まれている可能性があるため、これらのプロセスを管理します。私にとっては非常にシンプルで、最も必要な3つの構成で構成されます

  • 画像ストリーム
  • Buildconfig
  • DeploymentConfig

テンプレートjava-app-sample 
kind: Template
apiVersion: v1
metadata:
  labels:
    app: java-app-sample
  name: java-app-sample
  annotations:
    description: >-
      This example shows how to create a simple Java application in openshift
    tags: java
parameters:
  - name: APP_NAME
    description: application name
    value: app-sample
    required: true
objects:
  - kind: ImageStream
    apiVersion: v1
    metadata:
      name: "${APP_NAME}"
    spec:
      tags:
        - from:
            kind: DockerImage
            name: 172.30.1.1:5000/myproject/${APP_NAME}:latest
          name: latest
  - kind: BuildConfig
    apiVersion: v1
    metadata:
      name: "${APP_NAME}"
      labels:
        name: "${APP_NAME}"
    spec:
      source:
        binary: {}
        type: Binary
      strategy:
        type: Docker
      output:
        to:
          kind: DockerImage
          name: 172.30.1.1:5000/myproject/${APP_NAME}:latest
      postCommit: {}
      resources: {}
  - kind: DeploymentConfig
    apiVersion: v1
    metadata:
      name: "${APP_NAME}"
    spec:
      strategy:
        type: Rolling
        rollingParams:
          updatePeriodSeconds: 1
          intervalSeconds: 1
          timeoutSeconds: 120
        resources: {}
      replicas: 1
      selector:
        app: "${APP_NAME}"
      template:
        metadata:
          labels:
            app: "${APP_NAME}"
        spec:
          containers:
            - name: "${APP_NAME}"
              image: 172.30.1.1:5000/myproject/${APP_NAME}:latest
              ports:
                - containerPort: 8080
                  protocol: TCP
                - containerPort: 5005
                  protocol: TCP
              env:
                - name: TZ
                  value: Europe/Moscow
              resources:
                limits:
                  cpu: '0.5'
                  memory: 1000Mi
                requests:
                  cpu: '0.2'
                  memory: 500Mi
              imagePullPolicy: Always
          restartPolicy: Always
          dnsPolicy: ClusterFirst
    triggers:
      - type: ConfigChange
        imageChangeParams:
          automatic: true
          containerNames:
            - "${APP_NAME}"
          from:
            kind: ImageStreamTag
            name: 172.30.1.1:5000/myproject/${APP_NAME}:latest
      - type: ImageChange
    status: {}


テンプレートでは、いくつかのコンテナパラメータが例として示されています。たとえば、リソース/制限、TZ環境変数、1つのテンプレートパラメータが指定されています-アプリケーション名（APP_NAME）。レジストリDockerアドレスは、ミニシフトバージョン（開発用にローカルで使用する172.30.1.1/10000/myproject/）とその他に指定されていますが、

私のシナリオでは、最も重要なものがここに示されています

  - kind: BuildConfig
    spec:
      source:
        binary: {}
        type: Binary
      strategy:
        type: Docker

BuildConfigは、ソースがバイナリファイルであり、Dockerイメージの準備戦略を管理することを示してい

ます-Dockerfile（タイプ：Docker）OpenShiftでこのテンプレートを作成します

 oc create -f template.yaml


このテンプレートは、templatesディレクトリに表示されます（Web UIコンソールからプロセスを整理することはできますが、ここではできません）。



このテンプレートを使用して、アプリケーションを作成します（実際、これは単なる名前であり、これ以上はありません）

oc new-app java-app-sample -p APP_NAME=app-sample

テンプレート名とアプリケーション名パラメーターを指定します



3つの構成が作成されました。

コンテナー内でのアプリケーションの組み立てと展開のプロセスを開始します。

>oc start-build app-sample --from-dir . --follow

バイナリファイルとDockerfileが示されているフォルダーから実行することが重要です。--from-dirパラメーターは、Dockerレジストリでイメージをビルドおよびデプロイするためにフォルダー全体がOpenShiftでロードする準備ができることを示します。このコマンドには、アーカイブなどの他のパラメータがあります。



フォルダーの内容がOpenShiftに読み込まれ、Dockerファイルのイメージを準備するDockerプロセスが起動され、最終的にイメージがDockerレジストリに配置されました。
