Linuxネットワークアプリケーションのパフォーマンス。前書き

Webアプリケーションは現在広く使用されており、HTTPはすべてのトランスポートプロトコルの大部分を占めています。 Webアプリケーション開発の微妙な違いを調査すると、大多数はこれらのアプリケーションが実際に実行されるオペレーティングシステムにほとんど注意を払っていません。開発（Dev）と運用（Ops）の分離は、問題を悪化させるだけでした。しかし、DevOpsカルチャーの普及に伴い、開発者はクラウドでアプリケーションを起動する責任を負うようになったため、オペレーティングシステムのバックエンドを完全に理解することは非常に役立ちます。これは、システムを数千または数万の同時接続に展開しようとしている場合に特に役立ちます。

Webサービスの制限は、他のアプリケーションの制限とよく似ています。ロードバランサーであれデータベースサーバーであれ、これらのアプリケーションはすべて、高性能環境で同様の問題を抱えています。これらの基本的な制限とそれらを一般的に克服する方法を理解することは、Webアプリケーションのパフォーマンスとスケーラビリティを評価するのに役立ちます。

情報に詳しいシステムアーキテクトになりたいと考えている若い開発者からの質問に応えて、このシリーズの記事を書いています。 Linuxアプリケーションを最適化する方法を、オペレーティングシステムレベルでどのように機能するかの基本を掘り下げずに明確に理解することは不可能です。アプリケーションにはさまざまな種類がありますが、このシリーズでは、ブラウザーやテキストエディターなどのデスクトップアプリケーションではなく、ネットワークアプリケーションについて説明します。この資料は、LinuxまたはUnixプログラムがどのように機能するか、およびそれらを高パフォーマンスのために構造化する方法を理解したい開発者およびアーキテクトを対象としています。

Linuxはサーバーオペレーティングシステムであり、ほとんどの場合、アプリケーションはこの特定のOSで実行されます。私は「Linux」と言いますが、ほとんどの場合、一般的にすべてのUnixライクなオペレーティングシステムが意図されていると安全に想定できます。ただし、他のシステムでは付属のコードをテストしていません。したがって、FreeBSDまたはOpenBSDに関心がある場合、結果は異なる場合があります。 Linux固有のものを試してみると、それを指摘します。

取得した知識を使用してアプリケーションを最初から作成することができ、完全に最適化されますが、そうしない方がよいでしょう。組織のビジネスアプリケーション用にCまたはC ++で新しいWebサーバーを作成する場合、これが仕事の最終日になる可能性があります。ただし、これらのアプリケーションの構造に関する知識は、既存のプログラムの選択に役立ちます。プロセスベースのシステムとスレッドベースおよびイベントベースのシステムを比較できます。NginxがApache httpdよりもうまく機能する理由と、トルネードベースのPythonアプリケーションがDjangoベースのPythonアプリケーションよりも多くのユーザーにサービスを提供できる理由を理解し、理解するでしょう。

ZeroHTTPd：学習ツール


ZeroHTTPd  は、トレーニングツールとしてCで一から書いたWebサーバーです。 Redisへのアクセスなど、外部の依存関係はありません。独自のRedisルーチンを実行します。詳細については、以下を参照してください。

理論については長い間議論することができましたが、コードを記述して実行し、すべてのサーバーアーキテクチャを比較することほど優れた方法はありません。これは最も明白な方法です。したがって、プロセス、フロー、イベントに基づいて、各モデルを適用する単純なWebサーバーZeroHTTPdを作成します。これらの各サーバーをチェックして、お互いの動作を比較します。 ZeroHTTPdは単一のCファイルに実装されています。イベントベースのサーバーにはuthashが含まれています、単一のヘッダーファイルで提供される優れたハッシュテーブルの実装。他の場合では、プロジェクトを複雑にしないために依存関係はありません。

コードを整理するのに役立つコメントがたくさんあります。数行のコードで構成される単純なWebサーバーであるため、ZeroHTTPdは最小限のWeb開発フレームワークでもあります。機能は制限されていますが、静的ファイルと非常に単純な「動的」ページを生成できます。 ZeroHTTPdは、高性能のLinuxアプリケーションを作成する方法を学ぶのに最適です。概して、ほとんどのWebサービスはリクエストを待機し、チェックして処理します。これはまさにZeroHTTPdが行うことです。これは学習ツールであり、制作ツールではありません。彼はエラー処理が得意ではなく、最高のセキュリティプラクティス（そうそう、私はそれを使用しましたstrcpy）やCの厄介なトリックを自慢することはほとんどありませんが、彼がうまくやってくれることを願っています。

ゲストブック申し込み


最新のWebアプリケーションは通常、静的ファイルに限定されません。さまざまなデータベースやキャッシュなどとの複雑なやり取りがあります。そのため、訪問者が名前の下にエントリを残す「Guestbook」と呼ばれるシンプルなWebアプリケーションを作成します。ゲストブックには、以前に残されたエントリが保存されます。ページ下部にはビジターカウンターもあります。


ゲストブックWebアプリケーションZeroHTTPd

訪問者カウンターとゲストブックのエントリはRedisに保存されます。 Redisとの通信では、独自のプロシージャが実装されており、外部ライブラリに依存しません。公に利用可能で十分にテストされたソリューションがある場合、私は自作のコードをローリングするのはあまり好きではありません。しかし、ZeroHTTPdの目標は、Linuxのパフォーマンスと外部サービスへのアクセスを調査することですが、HTTPリクエストの処理はパフォーマンスに深刻な影響を与えます。各サーバーアーキテクチャでRedisとの通信を完全に制御する必要があります。 1つのアーキテクチャではブロッキング呼び出しを使用し、別のアーキテクチャではイベントベースのプロシージャを使用します。外部Redisクライアントライブラリを使用しても、そのような制御はできません。さらに、小さなRedisクライアントは、いくつかの機能（キーの取得、設定、増分、配列の取得と追加）のみを実行します。また、Redisプロトコルは非常にエレガントでシンプルです。彼は特別な訓練を受ける必要さえありません。プロトコルが約100行のコードですべての作業を実行するという事実は、それがどれほどよく考えられているかを示しています。

ゲストブックアプリケーションのメカニズム

ゲストブックページを発行する必要がある場合、テンプレートをメモリに読み込むためにファイルシステムを1回呼び出し、Redisを3回ネットワーク呼び出しします。テンプレートファイルには、上のスクリーンショットのページのほとんどのHTMLコンテンツが含まれています。コンテンツの動的な部分のための特別なプレースホルダーもあります：レコードと訪問者カウンター。それらをRedisから取得し、ページに挿入して、クライアントに完全な形式のコンテンツを提供します。Redisは、インクリメントされると新しいキー値を返すため、3回目のRedisの呼び出しを回避できます。ただし、非同期イベントベースのアーキテクチャを備えたサーバーでは、多くのネットワーク呼び出しがトレーニング目的の優れたテストです。したがって、ビジター数に関するRedisの戻り値を破棄し、別の呼び出しでリクエストします。

ZeroHTTPdサーバーアーキテクチャ


機能は同じだがアーキテクチャーが異なる7つのバージョンのZeroHTTPdを構築しています。

  • 反復
  • フォークサーバー（リクエストごとに1つの子プロセス）
  • フォーク前サーバー（フォーク前プロセス）
  • スレッドのあるサーバー（リクエストごとに1つのスレッド）
  • プレスレッド対応のサーバー
  • ベースのアーキテクチャ poll()
  • ベースのアーキテクチャ epoll

サーバーにHTTPリクエストをロードして、各アーキテクチャのパフォーマンスを測定します。ただし、並列処理の度合いが高いアーキテクチャを比較すると、要求の数が増加します。3回テストし、平均を考慮します。

パフォーマンスとスケーラビリティ


パフォーマンスとスケーラビリティに関する注意事項。理論的には、それらの間には関係がありません。応答時間が数ミリ秒と非常にうまく機能するWebサービスがあるかもしれませんが、まったくスケーリングしません。同様に、応答に数秒かかる、適切に実行されていないWebアプリケーションが存在する可能性がありますが、数万の同時ユーザーを処理するために数十にスケーリングされます。ただし、高性能とスケーラビリティの組み合わせは非常に強力な組み合わせです。高性能アプリケーションは通常、リソースを経済的に使用するため、サーバー上の同時ユーザーにより効率的にサービスを提供し、コストを削減します。

CPUおよびI / Oタスク


最後に、計算には常に2つのタイプのタスクがあります。I/ OとCPUです。インターネット経由のリクエストの受信（ネットワークI / O）、ファイルのメンテナンス（ネットワークとディスクI / O）、データベースとの通信（ネットワークとディスクI / O）はすべてI / Oアクションです。一部のデータベースクエリは、CPUを少し読み込むことができます（並べ替え、100万件の結果の平均の計算など）。ほとんどのWebアプリケーションは、可能な最大I / Oによって制限されており、プロセッサーが全容量で使用されることはほとんどありません。いくつかのI / Oタスクで多くのCPUが使用されている場合、これはおそらくアプリケーションアーキテクチャが不十分であることを示しています。これは、CPUリソースがプロセス制御とコンテキスト切り替えに費やされていることを意味している可能性があります。画像処理などをするとオーディオファイルまたは機械学習を変換する場合、アプリケーションには強力なCPUリソースが必要です。しかし、ほとんどのアプリケーションではこれはそうではありません。
