Androidフレームワークのさまざまなクラスのドキュメントを読むことにしたのです。私はメインクラス（View、Activity、Fragment、Application ）を実行しましたが、Application.registerActivityLifecycleCallbacks（）メソッドとActivityLifecycleCallbacksインターフェイスに非常に興味を持っていました。インターネットでの使用例のうち、アクティビティのライフサイクルをログに記録することほど優れたものはありません。それから私は自分でそれを実験し始めました、そして今私たちはYandex.Moneyで積極的にそれを使用して、外部からのActivityオブジェクトの影響に関連するさまざまなタスクを解決しています。

ActivityLifecycleCallbacksとは何ですか？


このインターフェースを見てください。API14に登場したときの様子は次のとおりです。

public interface ActivityLifecycleCallbacks {
    void onActivityCreated(Activity activity, Bundle savedInstanceState);
    void onActivityStarted(Activity activity);
    void onActivityResumed(Activity activity);
    void onActivityPaused(Activity activity);
    void onActivityStopped(Activity activity);
    void onActivitySaveInstanceState(Activity activity, Bundle outState);
    void onActivityDestroyed(Activity activity);
}

API 29以降、いくつかの新しいメソッドが追加されました。

このインターフェースはAndroid 4.0 ICSでのみ表示されたため、このインターフェースにはほとんど注意が払われていない可能性があります。ただし、ネイティブで非常に興味深いことを実行できるため、無駄です。すべてのアクティビティオブジェクトに外部から影響を与えます。しかし、その詳細については後で、最初にメソッドを詳しく見ていきます。

各メソッドは、アクティビティライフサイクルの同様のメソッドを表示し、アプリケーション内のアクティビティでメソッドがトリガーされたときに呼び出されます。つまり、アプリケーションがMainActivityで始まる場合、最初にActivityLifecycleCallback.onActivityCreated（MainActivity、null）呼び出しを受け取ります。

ActivityLifecycleCallbacksを機能させる方法は？


すべてのコールバックと同様に、最初にそれらを登録する必要があります。すべてのActivityLifecycleCallbacksをApplication.onCreate（）に登録するため、すべてのアクティビティとそれらを管理する機能に関する情報を取得します。

class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        registerActivityLifecycleCallbacks(MyCallbacks())
    }
}

少し余談ですが、API 29以降、ActivityLifecycleCallbacksをアクティビティ内から登録することもできます。これは、このアクティビティでのみ機能するローカルコールバックになります。

それで全部です。ただし、検索ボックスにActivityLifecycleCallbacksという名前を入力するだけでこれを見つけることができます。アクティビティのライフサイクルをログに記録する例はたくさんありますが、それは興味深いですか？アクティビティには多くのパブリックメソッド（約400）があり、これらすべてを使用して多くの興味深い便利なことを行うことができます。

これで何ができますか？


で、何がのぞみ？アプリケーションのすべてのアクティビティのテーマを動的に変更したいですか？setTheme（）メソッドはパブリックです。つまり、ActivityLifecycleCallbackから呼び出すことができます。

class ThemeCallback(
    @StyleRes val myTheme: Int
) : ActivityLifecycleCallbacks {
    override fun onActivityCreated(
        activity: Activity, 
        savedInstanceState: Bundle?
    ) {
        activity.setTheme(myTheme)
    }
}

このトリックを自宅でのみ繰り返します。
接続されたライブラリの一部のアクティビティは、カスタムテーマを使用できます。したがって、このアクティビティのテーマを安全に変更できると判断できるパッケージまたはその他の症状を確認してください。たとえば、次のようにパッケージをチェックします（Kotlinovsky =））：

class ThemeCallback(
    @StyleRes val myTheme: Int
) : ActivityLifecycleCallbacks {
    override fun onActivityCreated(
        activity: Activity,
        savedInstanceState: Bundle?
    ) {
        val myPackage = "my.cool.application"
        activity
            .takeIf { it.javaClass.name.startsWith(myPackage) }
            ?.setTheme(myTheme)
    }
}

この例は機能しませんか？アプリケーションまたはAndroidManifestのアプリケーションにThemeCallbackを登録するのを忘れている可能性があります。

別の興味深い例が必要ですか？アプリケーションの任意のアクティビティのダイアログを表示できます。

class DialogCallback(
    val dialogFragment: DialogFragment
) : Application.ActivityLifecycleCallbacks {
    override fun onActivityCreated(
        activity: Activity,
        savedInstanceState: Bundle?
    ) {
        if (savedInstanceState == null) {
            val tag = dialogFragment.javaClass.name
            (activity as? AppCompatActivity)
                ?.supportFragmentManager
                ?.also { fragmentManager ->
                    if (fragmentManager.findFragmentByTag(tag) == null) {
                        dialogFragment.show(fragmentManager, tag)
                    }
                }
        }
    }
}

このトリックを自宅でのみ繰り返します。
もちろん、すべての画面にダイアログを表示しないでください-ユーザーは私たちを気に入られません。ただし、特定の画面でこのようなものを表示すると便利な場合があります。

次に、別のケースを示します。Activityを実行する必要がある場合はどうでしょうか。しかし、Activity.startActivityForResult（）を呼び出した後、結果を待つ必要がある場合はどうでしょうか？私は1つのレシピを持っています：

class StartingActivityCallback : Application.ActivityLifecycleCallbacks {
    override fun onActivityCreated(
        activity: Activity,
        savedInstanceState: Bundle?
    ) {
        if (savedInstanceState == null) {
            (activity as? AppCompatActivity)
                ?.supportFragmentManager
                ?.also { fragmentManager ->
                    val startingFragment = findOrCreateFragment(fragmentManager)

                    startingFragment.listener = { resultCode, data ->
                        // handle response here
                    }

                    // start Activity inside StartingFragment
                }
        }
    }

    private fun findOrCreateFragment(
        fragmentManager: FragmentManager
    ): StartingFragment {
        val tag = StartingFragment::class.java.name
        return fragmentManager
            .findFragmentByTag(tag) as StartingFragment?
                ?: StartingFragment().apply {
                    fragmentManager
                        .beginTransaction()
                        .add(this, tag)
                        .commit()
                }
    }
}

この例では、Fragmentを単にドロップします。これにより、アクティビティが開始され、結果が取得され、処理が委任されます。注意：ここで、アクティビティがAppCompatActivityであることを確認します。これにより、無限ループが発生する可能性があります。他の条件を使用します。

しかし、それはまだ簡単です。これらはすべて、本当に興味深いトピックであるネイティブ依存性注入にあなたを導くためのものでした。はい、ダガー、コイン、ギス、コデインなどがあります。しかし、小さなプロジェクトでは、それらは冗長です。しかし、私には解決策があります...どれを推測しますか？

次のようなツールがあるとします。

class CoolToolImpl {
    val extraInfo = "i am dependency"
}

大人のプログラマーのように、インターフェースでそれを閉じます。

interface CoolTool {
    val extraInfo: String
}

class CoolToolImpl : CoolTool {
    override val extraInfo = "i am dependency"
}

そして今、ActivityLifecycleCallbacksのちょっとしたストリートマジック：この依存関係を実装するためのインターフェースを作成し、必要なアクティビティーに実装し、ActivityLifecycleCallbacksを使用してそれを見つけてCoolToolImpl実装を実装します。

interface RequireCoolTool {
    var coolTool: CoolTool
}

class CoolToolActivity : Activity(), RequireCoolTool {
    override lateinit var coolTool: CoolTool
}

class InjectingLifecycleCallbacks : ActivityLifecycleCallbacks {
    override fun onActivityCreated(
        activity: Activity,
        savedInstanceState: Bundle?
    ) {
        (activity as? RequireCoolTool)?.coolTool = CoolToolImpl()
    }
}

InjectingLifecycleCallbacksをアプリケーションに登録し、それを実行することを忘れないでください。

そしてテストすることを忘れないでください：

@RunWith(AndroidJUnit4::class)
class DIActivityTest {
    @Test
    fun `should access extraInfo when created`() {
        // prepare
        val mockTool: CoolTool = mock()
        val application = getApplicationContext<android.app.Application>()
        application.registerActivityLifecycleCallbacks(
            object : Application.ActivityLifecycleCallbacks {
                override fun onActivityCreated(
                    activity: Activity,
                    savedInstanceState: Bundle?
                ) {
                    (activity as? RequireCoolTool)?.coolTool = mockTool
                }
            })

        // invoke
        launch<DIActivity>(Intent(application, DIActivity::class.java))

        // assert
        verify(mockTool).extraInfo
    }
}

ただし、大規模なプロジェクトでは、このアプローチは適切に拡張されないため、DIフレームワークを誰からも奪うことはしません。取り組みを組み合わせてシナジーを達成する方が良い場合。Dagger2の例を紹介します。プロジェクトにAndroidInjection.inject（これ）などの基本的なアクティビティがある場合は、それを破棄します。代わりに、以下を実行します。 
  1. 指示に従って、アプリケーションにDispatchingAndroidInjectorを実装します。
  2. 各アクティビティでDispatchingAndroidInjector.maybeInject（）を呼び出すActivityLifecycleCallbacksを作成します。
  3. アプリケーションにActivityLifecycleCallbacksを登録します。


class MyApplication : Application() {
    @Inject lateinit var dispatchingAndroidInjector: DispatchingAndroidInjector<Any>

    override fun onCreate() {
        super.onCreate()
        DaggerYourApplicationComponent.create().inject(this);
        registerActivityLifecycleCallbacks(
            InjectingLifecycleCallbacks(
                dispatchingAndroidInjector
            )
        )
    }
}

class InjectingLifecycleCallbacks(
    val dispatchingAndroidInjector: DispatchingAndroidInjector<Any>
) : ActivityLifecycleCallbacks {
    override fun onActivityCreated(
        activity: Activity,
        savedInstanceState: Bundle?
    ) {
       dispatchingAndroidInjector.maybeInject(activity)
    }
}
