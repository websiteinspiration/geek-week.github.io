gitlab-ciでの統合テストのためのClickHouseの構成

私たちは、golang、別のトピックのカフカ、クリックハウス、gitlab-ciと落下するペイライン、腐ったsshキーなどのサービスを提供していました。休暇シーズン、街のひどい雨、ラップトップの破損、夜のアラート、ホットセール。このすべてがこの記事で必要だったわけではありませんが、テスターの典型的な日常生活を示したら、最後まで意図を示してください。気になったのはp0だけでした。世界でそれを逃したテスターほど絶望的で憂鬱で憂鬱なことはありません。しかし、すぐにそれに没頭することを知っていました。

なぜこれなのか


共通のサービスバンドル（何かを実行するサービス自体）と、これらの結果が書き込まれるデータベースがあります。これが直接発生することもあります。つまり、「サービス-ベース」です。私の場合、記録は仲介者、つまり「サービス-キュー-ベース」を介して行われます。 

全体として、いくつかの要素があり、これらの要素の境界（一方の出力と他方の入力）は、問題が発生するまさにその場所です。彼らはそこに現れないだけです。

鮮明な例：サービスでは、価格フィールドはfloat32として処理され、データベースでは10進数（18、5）として構成されます。テストケースとして、サービスの出力からデータベースへのfloat32の最大値を提供します-ああ、データベースは応答しません。またはもっと悲しい例-データベースはクラッシュしませんが、データベースのデータログにエラーはありません。データベースがもう注がれないだけです。または、記録は通過しますが、データ損失または歪みがあります。フィールドはfloat64としてサービスを終了し、float32として記録されます。 

または、サービスライフサイクルの過程で、このフィールドまたはそのフィールドのタイプを変更する必要があると判断しました。このフィールドは長い間製品に実装されてきましたが、ここでは編集する必要があります。もちろん、1か所だけ変更しました。ホバ、何かがまたうまくいかなかった。

仕事


これらすべての変更を追跡したくありません。落ちないようにしたいです。録音を正しく行いたい。

終了：統合テスト！ 

実装と困難 


どこで壊す？


開発環境があります。非常に不安定で、通常、開発者がサンドボックスとして使用します。ハードバックエンドにはカオスと無秩序の特徴があります。 

テスト環境またはqa-standがあります。それはより適切に構成され、開発者もそれを監視していますが、キックするまで何も起こりません。この環境は頻繁に更新されます。さらに多くの場合、そこで何かが壊れています。 

そして、製品があります-聖なるものの聖：そのようなものをその上で運転しない方が良いです。統合テストは、製品に到達する前に発見する必要があるバグの可能性を示唆しています。 

それでは、環境が不安定または戦闘の場合、環境をどうするのでしょうか。そうです、自分で作成してください！

ベースはどうするの？


データベースはいくつかの方法で起動できます。

上記で説明したように、この環境またはその環境の実際のベースには接続しません。

まず、必要な設定を使用して脆弱なクリックハウスサーバーを発生させ、必要なSQLをロールアウトし、クリックハウスクライアントを介して通信することができます。同様のベースを配置する最初の成功した試みで、ciは悲しかった。テストが点滅し、サーバーは動作せず、動作し続けました。ただ言ってみれば、なぜそれが始まったのか、それはまだ謎のままです。 （それ自体、私はそれとは何の関係もありません）。このオプションはお勧めしません。

箱から出してすぐに使える便利なオプションは、Dockerイメージの使用です。
目的のバージョンを車にダウンロードします。 Clickhouseはconfig.xmlを設定で開始する必要があります。詳細はこちら
再利用されるクリック画像の場合、正しいdockerfileを作成する必要があります。その中にconfig.xlをフォルダーにコピーすることを示し、他の必要な構成をドッキングします。必ずスクリプトをコピーして、ベースをデプロイしてください。 

外部から画像にアクセスするため、クリックハウスと通信するためのポートを開く必要があります。Clickはhttpでは8123で、tcpでは9000で動作します。

次のdockerfileを取得します。

CIで画像をスローする方法は？ 


どういうわけかciでDockerイメージを操作するには、なんとかしてそこに呼び出す必要があります。 

リポジトリ内のイメージをコミットして実行し、テストの実行の一部として必要なパラメーターを指定してdocker runを実行できます。ここでのみ、クリックのdocker-imageの重量は350MB未満です。そのようなファイルをgitに保存するのは不適切です。

さらに、異なるプロジェクトで同じDockerイメージが必要な場合（たとえば、異なるサービスが同じデータベースに書き込まれる場合）は、なおさらそうするべきではありません。dockerレジストリイメージストレージ
を使用できますが、プロジェクトですでに使用されていると考えられます。したがって、ログインし、Dockerイメージを収集してそこにプッシュします。 

docker build -t my_clickhouse_image .
docker login my_registry_path.domain.com

下車し、私たちのイメージがレジストリに飛んだ。組み立て時にタグを必ず指定してください！

ciをどうするか？


1ステップでサービスとデータベースの両方を起動する方法は？ 

すべては、サービスの開始方法と使用方法によって異なります。 Dockerイメージと同じようにサービスを操作し、実際に.gitlab-ci.yml全体がそれらでのみ機能する場合、すべてが簡単です。 
dind-docker -in- dockerがあります。これは、CIが動作するメインサービスとして示され、Dockerを完全に使用でき、まったく負担をかけません。

最新の画像を取り出し、必要なテスト手順をステージに追加し、一連のアクションを説明します。

image: docker:stable
services:
- docker:dind

stages:
- build
  …
- test-click
...
- test
- release
  …

test-click:
  variables:
    VERY_IMPORTANT_VARIABLE: “its value”
  before_script:
  - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
  - docker pull My_Service_Image
  - docker pull My_ClickHouse_Image
  - docker run -FLAGS My_ClickHouse_Image
  - docker run My_Service_Image /path/to/tests

公式のdocker dockerでは、dindの使用は推奨されていないことが示されていますが、本当に必要な場合は...

私のプロジェクトでは、バイナリの起動を通じてサービスをテストする必要があります。これが魔法の始まりです
これを行うには、データベースをサービスとして使用します。公式のgitlab-ciドキュメントでは、ciでのdockerコンテナーの最も一般的な使用例として、ベース付きのコンテナーの使用を挙げています。でも例のmysql、postress、およびRedisの設定が用意されています。しかし、簡単な方法を探しているのではなく、クリックハウスが必要です。

ベースを接続してください！必ずエイリアスを指定してください。指定しない場合、ランダムな名前とランダムなIPがデータベースに割り当てられます。つまり、彼女に連絡する方法が正確に不明になります。エイリアスにはそのような問題はありません-テストコードでは、たとえばhttpsのように呼び出しは次のようになりますhttp://my_alias_name:8123。

テストの場合も、データベースのイメージが必要です。これをレジストリに慎重に入れました。イメージをダウンロードするには、docker loginとdocker pullを行う必要があります。ciだけがdockerを認識していません-インストールする必要があります。

gitlab-ci.ymlのステップの結果コード：

Integration tests:
Services:
- name: my_clickhouse:latest
  alias: clicktest
Stage: tests
Variables:
Variables_for_my_service: “value”
Before_script:
- curl -ssl https://get.docker.com/ | sh
- docker login -u gitlab-ci-token -p $ci_build_token my_registry_path.domain.com
Script:
- ./bin/my_service &
- go test -v ./tests -tags=integration
Dependencies:
- build

利益


  • 私にはサービス基盤がたくさんあります。
  • 自動テストの一部として、データベースに簡単にアクセスできます-別名で。
  • セットアップテストの一環としてデータベースのレコードと設定をリセットし、サービスを呼び出し、データベースに書き込み、データベースに切り替え、データベースが落ちていないことを確認しました。到着したものを確認し、検証します。より多くのテストを投げます。
  • ペンでテストすることはできません！

結果


gitlab-ciでの数行のセットアップのようです。Dockerイメージの作成は簡単です。ローカルでのベースの起動は簡単です。私は1日で問題を発見した最初のテストと統合しました。しかし、ciでそれをローンチする試みは、1週間の苦痛と絶望の渦になりました。そして今、彼らがプログラミングしたすべてのものを修復しなければならない開発者の苦痛と絶望の数週間で。
