イベント指向アーキテクチャーのアンチパターン

また会いましたね！「ソフトウェアアーキテクト」コースの開始を見越して、別の興味深い資料の翻訳を用意しました。



ここ数年、マイクロサービスアーキテクチャの人気が高まっています。それを正しく実装する方法を教える多くのリソースがありますが、多くの場合、それはシルバープールとしてそれについて話します。マイクロサービスの使用には多くの議論がありますが、最も重要なのは、このタイプのアーキテクチャには不確実な複雑さがあり、そのレベルはサービスとチーム間の関係をどのように管理するかに依存します。あなたの場合、（おそらく）マイクロサービスが最良の選択ではない理由を説明する多くの文献を見つけることができます。

私たちletgoは、スケーラビリティのニーズを満たすためにモノリスからマイクロサービスに移行し、すぐにチームの作業に対するその有益な効果を確信しました。マイクロサービスを正しく使用すると、次のようないくつかの利点があります。

これに基づいて、私たちはサービス間の非同期通信に固執しようとし、同期は機能MVPなどの例外的なケースでのみ機能します。メッセージバス内の他のサービスによって発行されたドメインイベントに基づいて、各サービスが独自のエンティティを生成するようにするためです。

私たちの意見では、マイクロサービスアーキテクチャの実装の成功または失敗は、その固有の複雑さにどのように対処するか、およびサービスが相互にどのように相互作用するかに依存します。通信インフラストラクチャを非同期に転送せずにコードを分割すると、アプリケーションが分散モノリスに変わります。



letgoのイベント駆動型アーキテクチャ

今日は、letgoでドメインイベントと非同期通信を使用する方法の例を紹介します。Userエンティティは多くのサービスに存在しますが、その作成と編集は最初はUsersサービスによって処理されます。ユーザーサービスのデータベースには、名前、メールアドレス、アバター、国など、多くのデータが保存されています。チャットサービスにはユーザーの概念もありますが、ユーザーサービスのユーザーエンティティが持っているデータは必要ありません。ユーザーの名前、アバター、ID（プロファイルへのリンク）がダイアログのリストに表示されます。チャットでは、部分的なデータを含むUserエンティティのプロジェクションのみがあると言います。実際、チャットではユーザーについて話しているのではなく、「トーカー」と呼んでいます。このプロジェクションはチャットサービスを参照し、チャットがユーザーサービスから受信するイベントに基づいて構築されます。

リストについても同様です。Productsサービスでは、各リストの画像をn個保存しますが、ダイアログのリストではメイン画像を1つ表示するため、ProductsからChatへの投影には、nではなく1つの画像のみが必要です。


チャットでダイアログのリストを表示します。バックエンドの特定のサービスが情報を提供することを示しています。

ダイアログのリストをもう一度見ると、ユーザーとチャットのプロジェクションはChatが所有しているため、表示されるほとんどすべてのデータはChatサービスによって作成されたのではなく、Chatサービスに属していることがわかります。アクセシビリティと予測の一貫性の間にはトレードオフがありますが、この記事では説明しませんが、多くの小さなデータベースを1つの大きなデータベースよりもスケーリングする方が明らかに簡単であるとだけ言っておきます。

アンチパターン


いくつかの直感的な解決策はしばしば間違いになりました。以下は、ドメイン関連のアーキテクチャで遭遇した最も重要なアンチパターンのリストです。

1.厚いイベント
ドメイン値を失わないようにしながら、ドメインイベントをできる限り小さくするようにしています。大きなエンティティでレガシーコードベースをリファクタリングし、イベントアーキテクチャに切り替えるときは注意が必要です。このようなエンティティは私たちをファットイベントに導く可能性がありますが、ドメインイベントはパブリックコントラクトに変換されたため、それらをできるだけ単純にする必要がありました。この場合、リファクタリングは側面から見るのが最適です。まず、イベントストームテクニックを使用してイベントを設計します次に、サービスコードをリファクタリングして、イベントに適合させます。

また、「製品とユーザー」の問題にも注意する必要があります。多くのシステムは製品とユーザーのエンティティを使用します。これらのエンティティは、原則としてそれらの背後にあるすべてのロジックを引き出します。つまり、すべてのドメインイベントがそれらに関連付けられます。

2.意図としてのイベント
ドメインイベントは、定義上、すでに発生したイベントです。メッセージバスに何かを発行して、他のサービスで何が起こったかを要求する場合、ドメインイベントを作成するのではなく、非同期コマンドを実行している可能性があります。原則として、過去のドメインイベントであるser_registered、product_publishedを参照します。等。一方のサービスが他方について知っている数が少ないほど、より良い。イベントをコマンドとして使用すると、サービスがリンクされ、1つのサービスの変更が他のサービスに影響を与える可能性が高くなります。

3.独立したシリアル化または圧縮の欠如：
対象分野におけるイベントのシリアル化および圧縮のシステムは、プログラミング言語に依存すべきではありません。消費者向けサービスがどの言語で書かれているかを知る必要さえありません。そのため、たとえばJavaまたはPHPシリアライザーを使用できます。シリアライザの将来の変更は困難で時間がかかるため、チームでシリアライザの議論と選択に時間を費やしてください。私たちはJSONを使用していますが、パフォーマンスの良い他の多くのシリアル化形式があります。

4.標準的な構造の欠如
letgoバックエンドをイベント指向のアーキテクチャに移植し始めたとき、ドメインイベントの共通構造に同意しました。次のようになります。

{
  “data”: {
    “id”: [uuid], // event id.
    “type”: “user_registered”,
    “attributes”: {
      “id”: [uuid], // aggregate/entity id, in this case user_id
      “user_name”: “John Doe”,
      …
    }
  },
  “meta” : {
    “created_at”: timestamp, // when was the event created?
    “host”: “users-service” // where was the event created?
    …
  }
}

ドメインイベントに共通の構造があるため、サービスをすばやく統合し、抽象化を使用して一部のライブラリを実装できます。

5.スキーマ検証の欠如
シリアライズ中に、強い型付けなしでプログラミング言語の問題を経験しました。


{
  “null_value_one”: null, // thank god

イベントのシリアライゼーションを保証する確立されたテストカルチャー、およびシリアライゼーションライブラリの機能の理解は、これに対処するのに役立ちます。 letgoでは、AvroとConfluent Schema Registryに切り替えています。これにより、ドメインのイベントの構造を決定する単一のポイントが提供され、このタイプのエラーを回避し、廃止されたドキュメントも作成されます。

6.貧血ドメインのイベント
前に述べたように、またその名前が示すように、ドメインイベントにはドメインレベルの値が必要です。エンティティの状態の不整合を回避しようとするのと同じように、ドメインイベントではこれを回避する必要があります。これを次の例で説明します。システムの製品には、緯度と経度の位置情報があり、Productsサービスの製品テーブルの2つの異なるフィールドに格納されています。すべての製品を「移動」できるため、このアップデートを提示するドメインイベントがあります。以前は、このために、2つのイベントproduct_latitude_updatedとproduct_longitude_updatedがありました。これらは、チェス盤でルークでない場合はあまり意味がありませんでした。この場合、product_location_updatedイベントの方が意味があります。またはproduct_moved。


ルークはチェスの駒です。以前はツアーと呼ばれていました。ルークは、空いているフィールドをいくつでも縦または横にしか移動できません。

7.デバッグツールの欠如
letgoでは、毎分数千のドメインイベントを生成しています。これらすべてのイベントは、システムで何が起こっているのかを理解し、ユーザーアクティビティを登録し、イベント検索を使用して特定の時点でシステムの状態を再構築するための非常に有用なリソースになります。このリソースを上手に使用する必要があります。そのためには、イベントをチェックしてデバッグするためのツールが必要です。 「過去3時間にJohn Doeによって生成されたすべてのイベントを表示してください」などのリクエストも詐欺の検出に役立ちます。これらの目的のために、ElasticSearch、Kibana、S3でいくつかのツールを開発しました。

8.イベント監視の欠如
ドメインイベントを使用して、システムの状態をテストできます。何かをデプロイするとき（サービスによっては1日に数回行われます）、正しい操作をすばやく確認するためのツールが必要です。たとえば、Productsサービスの新しいバージョンを本番環境にデプロイし、product_publishedイベントの数が減少した場合20％、私たちは何かを壊したと言っても安全です。現在、InfluxDB、Grafana、およびPrometheusを使用して、派生関数でこれを実現しています。数学の過程を思い出すと、点xでの関数f（x）の導関数が、この点で関数のグラフに描かれた接線角度の接線に等しいことがわかります。サブジェクトエリア内の特定のイベントの速度を公開する機能があり、それから派生したものを使用する場合、この機能のピークが表示され、それらに基づいて通知を設定できます。これらの通知を使用すると、「1秒間に200未満のイベントを5分間パブリッシュした場合に警告する」などのフレーズを回避し、パブリケーションの速度の大幅な変化に集中できます。

10.ドメインイベントに関するドキュメントの欠如
ドメインイベントは、バックエンド上のすべてのシステムのパブリックインターフェイスになりました。 REST APIを文書化するのと同じように、ドメインイベントも文書化する必要があります。組織のすべての従業員は、各サービスによって発行された各ドメインイベントの更新されたドキュメントを表示できる必要があります。ドメインイベントのチェックにスキームを使用する場合、それらはドキュメントとしても使用できます。
