QuarkusおよびAMQ Onlineを使用したRed Hat OpenShiftクラウドベースのメッセージング

みなさん、こんにちは！これが彼です-Quarkusシリーズの最後の投稿です！ （ちなみに、私たちのウェビナーを参照してください- 「これはQuarkusあるKubernetesネイティブのJavaフレームワーク。」我々はゼロから始めるか、既製のソリューションを転送する方法を紹介します）



では、以前の私たちは、あなたがJava近代化の結果として作られた改善を定量化することが可能な適切なツールを見てポストアプリケーション。

バージョン0.17.0以降、Quarkusは、アプリケーションまたは組織間でビジネスメッセージを転送するためのオープンスタンダードであるAdvanced Message Queuing Protocol（AMQP）の使用をサポートしています。

Red Hat AMQ Onlineは、オープンEnMasseプロジェクトに基づいて構築されたサービスであり、Red Hat OpenShiftプラットフォームに基づくメッセージングメカニズムを実装しています。それがどのように動作するかの詳細情報を参照してください。ここで（ENへ）。今日は、AMQ OnlineとQuarkusを組み合わせて、メッセージ処理に関連する2つの新しいテクノロジーを使用してOpenShiftに基づく最新のメッセージングシステムを構築する方法を示します。

OpenShiftプラットフォームにAMQ Onlineをすでにデプロイしていることを前提としています（そうでない場合は、インストールガイドを参照してください）。

まず、Quarkusアプリケーションを作成します。これは、リアクティブメッセージングを使用した簡単な注文処理システムです。このアプリケーションには、固定間隔でメッセージキューに注文を送信する注文ジェネレーターと、キューからのメッセージを処理してブラウザーで表示できる確認を生成する注文プロセッサーが含まれます。

アプリケーションを作成したら、メッセージングシステム構成をアプリケーションに実装し、AMQ Onlineを使用してこのシステムで必要なリソースを初期化する方法を示します。

Quarkusアプリ


QuarkusアプリケーションはOpenShiftで実行され、amqp-quickstartの修正バージョンです。クライアント側の完全な例はここにあります。

注文ジェネレーター


ジェネレーターは5秒ごとに単調に増加する注文識別子を「注文」アドレスに送信します。

@ApplicationScoped
public class OrderGenerator {
 
    private int orderId = 1;
 
    @Outgoing("orders")
    public Flowable<Integer> generate() {
        return Flowable.interval(5, TimeUnit.SECONDS)
        .map(tick -> orderId++);
    }
}

注文ハンドラー


注文プロセッサはさらにシンプルで、確認識別子をアドレス「confirmations」に返すだけです。

@ApplicationScoped
public class OrderProcessor {
    @Incoming("orders")
    @Outgoing("confirmations")
    public Integer process(Integer order) {
        //       <img draggable="false" class="emoji" alt=":-)" src="https://s.w.org/images/core/emoji/11.2.0/svg/1f642.svg">
        return order * 2;
    }
}

確認リソース


確認リソースは、アプリケーションによって生成された確認をリストするためのHTTPエンドポイントです。

@Path("/confirmations")
public class ConfirmationResource {
 
    @Inject
    @Stream("confirmations") Publisher<Integer> orders;
 
    @GET
    @Produces(MediaType.TEXT_PLAIN)
    public String hello() {
        return "hello";
    }
 
 
    @GET
    @Path("/stream")
    @Produces(MediaType.SERVER_SENT_EVENTS)
    public Publisher<Integer> stream() {
        return orders;
    }
}

カスタマイズ


AMQ Onlineに接続するには、アプリケーションにいくつかの構成データ、つまり、Quarkusコネクタ構成、AMQPエンドポイント情報、およびクライアント資格情報が必要です。もちろん、すべての構成データを1か所に保持することをお勧めしますが、Quarkusアプリケーションをセットアップするための可能なオプションを示すために、それらを特別に分離します。

コネクター


コネクター設定は、アプリケーションプロパティファイルを使用して、コンパイル段階で提供できます。

mp.messaging.outgoing.orders.connector=smallrye-amqp
mp.messaging.incoming.orders.connector=smallrye-amqp

これを複雑にしないために、「注文」アドレスにのみメッセージキューを使用します。また、アプリケーションの「確認」アドレスは、メモリ内のキューを使用します。

エンドポイントAMQP


コンパイル段階では、AMQPエンドポイントのホスト名とポート番号は不明であるため、実装する必要があります。エンドポイントは、AMQ Onlineによって作成されるconfigmapで設定できるため、アプリケーションマニフェストの環境変数を使用して定義します。

spec:
  template:
    spec:
      containers:
      - env:
        - name: AMQP_HOST
          valueFrom:
            configMapKeyRef:
              name: quarkus-config
              key: service.host
        - name: AMQP_PORT
          valueFrom:
            configMapKeyRef:
              name: quarkus-config
              key: service.port.amqp

資格情報


サービスアカウントトークンを使用して、OpenShiftでアプリケーションを認証できます。これを行うには、最初にポッドファイルシステムから認証トークンを読み取るカスタムConfigSourceを作成する必要があります。

public class MessagingCredentialsConfigSource implements ConfigSource {
    private static final Set<String> propertyNames;
 
    static {
        propertyNames = new HashSet<>();
        propertyNames.add("amqp-username");
        propertyNames.add("amqp-password");
    }
 
    @Override
    public Set<String> getPropertyNames() {
        return propertyNames;
    }
 
    @Override
    public Map<String, String> getProperties() {
        try {
            Map<String, String> properties = new HashMap<>();
            properties.put("amqp-username", "@@serviceaccount@@");
            properties.put("amqp-password", readTokenFromFile());
            return properties;
        } catch (IOException e) {
            throw new UncheckedIOException(e);
        }
    }
 
    @Override
    public String getValue(String key) {
        if ("amqp-username".equals(key)) {
            return "@@serviceaccount@@";
        }
        if ("amqp-password".equals(key)) {
            try {
                return readTokenFromFile();
            } catch (IOException e) {
                throw new UncheckedIOException(e);
            }
        }
        return null;
    }
 
    @Override
    public String getName() {
        return "messaging-credentials-config";
    }
 
    private static String readTokenFromFile() throws IOException {
        return new String(Files.readAllBytes(Paths.get("/var/run/secrets/kubernetes.io/serviceaccount/token")), StandardCharsets.UTF_8);
    }
}

アプリケーションの組み立てと配備


アプリケーションは実行可能ファイルにコンパイルする必要があるため、GraalVM仮想マシンが必要です。このための環境をセットアップする方法の詳細については、 『Quarkusガイド』の関連する手順を参照してください。

次に、そこにある指示に従って、ソースをダウンロードし、アプリケーションをビルドしてデプロイする必要があります。

git clone https://github.com/EnMasseProject/enmasse-example-clients
cd enmasse-example-clients/quarkus-example-client
oc new-project myapp
mvn -Pnative -Dfabric8.mode=openshift -Dfabric8.build.strategy=docker package fabric8:build fabric8:resource fabric8:apply

これらのコマンドの後、アプリケーションはデプロイされますが、AMQ Onlineで必要なメッセージングリソースを構成するまで起動しません。

メッセージングの設定


アプリケーションがメッセージングシステムで必要とするリソースを設定する必要があります。これを行うには、以下を作成します。1）メッセージングシステムのエンドポイントを初期化するためのアドレススペース。2）アプリケーションで使用するアドレスを構成するためのアドレス。3）クライアントの資格情報を設定するメッセージングシステムのユーザー。

アドレス空間


AMQ OnlineのAddressSpaceオブジェクトは、接続エンドポイントと認証および承認ポリシーを共有するアドレスのグループです。アドレススペースを作成するときに、メッセージングシステムのエンドポイントを提供する方法を指定できます。

apiVersion: enmasse.io/v1beta1
kind: AddressSpace
metadata:
  name: quarkus-example
spec:
  type: brokered
  plan: brokered-single-broker
  endpoints:
  - name: messaging
    service: messaging
    exports:
    - name: quarkus-config
      kind: configmap

住所


アドレスはメッセージの送受信に使用されます。各アドレスには、セマンティクスを定義するタイプと、予約されたリソースの数を設定するプランがあります。アドレスは、たとえば次のように決定できます。

apiVersion: enmasse.io/v1beta1
kind: Address
metadata:
  name: quarkus-example.orders
spec:
  address: orders
  type: queue
  plan: brokered-queue

メッセージングユーザー


信頼できるアプリケーションのみができるアドレスにメッセージを送受信するには、メッセージングシステムでユーザーを作成する必要があります。クラスターで実行されているアプリケーションの場合、OpenShiftサービスアカウントを使用してクライアントを認証できます。ユーザー「serviceaccount」は、たとえば次のように定義できます。

apiVersion: user.enmasse.io/v1beta1
kind: MessagingUser
metadata:
  name: quarkus-example.app
spec:
  username: system:serviceaccount:myapp:default
  authentication:
    type: serviceaccount
  authorization:
  - operations: ["send", "recv"]
    addresses: ["orders"]

アプリケーションをカスタマイズするための権限


AMQ OnlineがAMQPエンドポイント情報の実装に使用したconfigmapを作成するには、ロールとロールバインディング（RoleとRoleBinding）を指定する必要があります。

---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: quarkus-config
spec:
  rules:
  - apiGroups: [ "" ]
    resources: [ "configmaps" ]
    verbs: [ "create" ]
  - apiGroups: [ "" ]
    resources: [ "configmaps" ]
    resourceNames: [ "quarkus-config" ]
    verbs: [ "get", "update", "patch" ]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: quarkus-config
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: quarkus-config
subjects:
- kind: ServiceAccount
  name: address-space-controller
  namespace: amq-online-infra

構成を適用する方法


次のようなメッセージング構成を適用できます。

cd enmasse-example-clients/quarkus-example-client
oc project myapp
oc apply -f src/main/resources/k8s/addressspace
oc apply -f src/main/resources/k8s/address

アプリケーション検証


アプリケーションが起動したことを確認するには、まず、対応するアドレスが作成されてアクティブになっているかどうかを確認します。

until [[ `oc get address quarkus-example.prices -o jsonpath='{.status.phase}'` == "Active" ]]; do echo "Not yet ready"; sleep 5; done

次に、アプリケーションルートURLを確認します（このアドレスをブラウザーで開くだけです）。

echo "http://$(oc get route quarkus-example-client -o jsonpath='{.spec.host}')/prices.html"

ブラウザは、メッセージがAMQ Onlineによって送受信されると、チケットが定期的に更新されることを示します。

要約する


したがって、メッセージングにAMQPを使用するQuarkusアプリケーションを作成し、このアプリケーションをRed Hat OpenShiftプラットフォームで動作するように設定し、AMQ Online構成に基づいてその構成を実装しました。次に、アプリケーションのメッセージングシステムを初期化するために必要なマニフェストを作成しました。

これで、Quarkusシリーズは終了ですが、これから新しくて興味深いことがたくさんあります。
