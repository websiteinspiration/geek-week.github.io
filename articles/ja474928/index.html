Vivaldiブラウザーの使い方を学ぶ



Vivaldiのユーザーが、ブラウザが接続するサービスについて考えたら、

彼はネットワークアクティビティスキャナーを起動し、ビバルディやGoogleサーバーへの多くの自動ブラウザー接続を恐怖で探します。

少し意識を取り戻した後、そのようなユーザーは、個人のユーザーデータが川のようにサードパーティのサーバーにどのように流れるかを説明するエラーメッセージを送信し始めます。

ただし、これらの好奇心の強いユーザーのほとんどは私たちに何も書き込みませんが、Vivaldi cornyが活動をスパイし、Googleと時々詳細を「マージ」することさえあると自分で結論付けます。まあ、なぜ驚かれるのでしょうか。結局のところ、他のブラウザーはユーザーのアクティビティを監視して詳細をブラウザーの開発者に送信することをためらっていません。

ほとんどの場合、そのような好奇心旺盛なユーザーは不正なブラウザーアクティビティの監視結果を取得します（記事の最後に、最も信頼できる結果を取得する方法の説明があります）。

実際、VivaldiブラウザーはChromeと同じエンジンを使用し、 Googleの一部の機能とセキュリティコンポーネントも使用しますが、ブラウザーでの作業の詳細はすべて非公開のままです-Vivaldi開発者とサードパーティサービスの所有者から。 Vivaldiはネットワーク上のアクティビティに関する詳細な統計情報を収集しません。また、他の一部のブラウザーとは異なり、日常の作業でどのブラウザー機能を使用するかについての情報もありません。

この記事では、外部サービスにアクセスするVivaldiの各ケースを検討し、これが必要な理由を説明します。快適かつ安全にネットワークで作業できるように、完全な作業に必要な情報とサービスをお知らせします。

ダウンロード可能なコンポーネント


Vivaldiの一部の機能は、Chromium / Googleデベロッパーが独自に開発および更新したコンポーネントに依存しています。これらのコンポーネントは、特別なVivaldiブラウザーの内部ページで表示できます

。vivaldi：//コンポーネント

これらのコンポーネントは、ブラウザーのビルディングブロックです。これらにより、Chromiumベースのブラウザーは、コードベースから新しい関数を追加することにより、機能を拡張できます。

これらの中で最も重要なのは、ネットワークセキュリティーを強化する証明書アシスタントとCRLset、およびDRMで保護されたオンラインビデオ（Netflixなど）の再生を担当するWidevineモジュールです。

これらのコンポーネントは、コンピューターでVivaldiを初めて起動した直後にダウンロードされ、ブラウザーにインストールされます。ブラウザは、バージョンの関連性をチェックし、現時点で最新のものをインストールします。

最初のダウンロードリクエストはサーバーのupdate.googleapis.comとclients2.google.comに送信されますが、将来的には、redirector.gvt1.comやr3---sn-8xouxav-vnas.gvt1.comなどのサーバーを介してデータが送受信される可能性があります。

SafeBrowsingのデータ


SafeBrowsingコンポーネントは、ウイルスやその他の感染を伴う特定のソフトウェアのダウンロードなどを提供する悪意のあるサイトからユーザーを保護するために使用されます。このコンポーネントは、Googleが作成および管理するブラックリストと連携して動作し、Vivaldiはsafebrowsing.google.comサーバー上の1日に数回このリストにアクセスします。

以前はSafeBrowsingはいわゆるBloomフィルターで機能していましたが、現在はより単純なシステムが使用されています。

これで、システムは、アクセスするWebサイトのURL（およびURLの一部）のハッシュ（暗号化チェックサム）を作成することによって機能します。このハッシュは、暗号化に使用される実際のコードを判別するために解読するのが非常に困難です。また、このハッシュを作成するためにいくつかの異なるコードが使用された可能性もあります（ただし、それらを見つけるのも非常に困難です。これがチェックサムの意味です）。システムは小さなハッシュを使用してメインリストを検索し、URLの少なくとも一部にブラックリストエントリがあるかどうかを確認します。肯定的な応答の場合、要求は一致ごとに送信され、サーバーはいずれかのプレフィックスと一致するすべてのフルサイズハッシュのリストで応答します。

これらの完全なハッシュのいずれかが生成されたハッシュのいずれかに一致する場合、このURLは悪意のあるWebサイトに属している可能性が高いため、ブロックされます。

URLのハッシュ（またはその一部）の名誉のみがSafeBrowsingサーバーに送信されるため、サービスは、アクセスしているWebサイトの完全なアドレスや、どの完全なハッシュがそれに一致するかを認識しません。

ダウンロード保護


たとえば、Windows用のインストーラーをダウンロードすると、コンピューターにとって危険なプログラム、トロイの木馬、またはその他の悪意のあるソフトウェアである可能性があります。このような「ギフト」からユーザーを保護するために、ChromiumとVivaldiブラウザー自体のSafeBrowsingコードは、URLのブラックリストへの不在やダウンロードのホワイトリストの存在を確認します。ダウンロードしたファイルのセキュリティをローカルで検証できない場合は、SafeBrowsingサーバーに詳細情報（ダウンロードしたファイルのURL、参照リンク、ダウンロードしたファイルのハッシュ、利用可能な証明書情報、サーバーからの応答）が送信され、ファイルが安全かどうかが判断されます。

残念ながら、マルウェアのベンダーはダウンロードURLの実際のシグネチャと実際のダウンロードデータの両方を変更できるため、この追加情報なしでファイルの評判を判断することは非常に困難ですが、原則として他の一部は同じままにする必要があります。検証済みデータのリストにURLと参照リンクが追加される前に、保護を回避する実際の攻撃がありました。

同じシステムがMozilla Firefoxでも使用されています。

ユーザー統計


最近、追跡の形式として一意の識別子の使用を認識する人がいるため、ユーザーのカウント方法を変更することにしました。現在、一意の識別子を必要としない新しいユーザーカウント方法を導入しています。

一意の識別子を完全に削除する前に、新しいコードが意図したとおりに機能し、このカウンターによって報告された数値を信頼できることを確認するために、いくつかの手順を実行します。詳細については、こちらの記事でユーザーに対する見方を説明しています。

不要な広告データベースへのアクセス


今年の初めに、誤解を招く広告やサイトを離れることを妨げる広告など、不当な広告慣行を行うサイトに対して、独自の広告ブロックサポートを導入しました。この機能は、Googleが提供するリストを使用して実装されます。このリストはGoogleから直接ダウンロードされません。代わりに、Googleの内部サーバーによってアップロードされたリストを簡単に前処理した後、独自のサーバーでホストされます。リストはブラウザによってサーバーから毎日自動的に更新され、毎回サーバーにアクセスする必要なく侵入型のWebサイトに適用されます。

自動更新を確認する


WindowsとMacのVivaldiバージョンは、ブラウザーの更新についてサーバーを常にチェックしています。必要に応じて、設定でオプションを無効にします。

ウェブサイト通知


サイト（ニュースサイトなど）による通知の送信を許可すると、サイトは通知の受信場所を示す小さなスクリプトをブラウザーにインストールします。これにはいくつかのサービスプロバイダーがあり、Webサイトがどちらを使用するかを決定します。それらの1つは、mtalk.google.comにあるGoogle Cloud Messaging（GCM）です。このシステムに対応する内部URLは次のとおりです：

chrome：//設定/コンテンツ/通知
chrome：//設定/ siteData
vivaldi：// gcm-internals

インストールされている拡張機能


ユーザーがインストールした拡張機能は、開発者自身のサイトに接続して、更新を要求したり、さまざまな使用状況データを報告したりできます。これは明らかに、Vivaldiが状況を制御できる場所ではありません。私たちは常にユーザーに拡張機能を慎重に選択し、拡張機能のサプライヤーが信頼できることを確認することを常にお勧めします。

ネットワークアクティビティスキャナーが全体像を示さない理由


ネットワークアクティビティスキャナーを実行するユーザーは、通常、ネットワークアクティビティモニターのみを使用します。この場合、逆DNSルックアップが実行され、その結果、「任意の」ホスト名が表示されます。

一方、たとえば、Vivaldiが接続するGoogleサーバーは、地理的に離れた複数のサーバー上の大きなサーバーパークにあり、これらの各サーバーは一意のホスト名を持っています。 IPアドレスの逆DNSインデックスに登録されるのは、この一意の名前です。

Google、Amazon、またはMicrosoftが使用するメインファームのこのIPアドレスでホストされているサーバーの完全なリストを受信したり、クライアントが実際に接続しているサーバーの名前が含まれたりすることはほとんどありません。

クライアントが接続しているサーバーを見つける最善の方法は、DNSクエリと応答をリッスンすることです。これにより、ブラウザが接続しているサーバーが正確にわかります。

何が起こっているかを確認するさらに良い方法は、ビバルディによって提供された情報を使用することです。引数--enable-logging = stderr --v = 1でVivaldiを起動します。その結果、データパケットが作成され、ユーザーデータインストールディレクトリのchrome_debug.logファイルに保存されます。表示されるデータには、NetworkDelegate :: NotifyBeforeURLRequest：の行が含まれます。これは、ブラウザによって要求されたURLを示します。

結論として、自動化されたサービスは、オンラインアクティビティに関するデータや統計をVivaldiやGoogleに送信しません。
