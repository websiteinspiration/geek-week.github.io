彼女が私を手に入れたら、私は台本を書きました。

私は既製のレシピを検索しようとしましたが、大部分はこれに要約されました。 アドレスのpingを取得し、デバッグをオンにして、「packets go、the answers not come。」という引用の最初の文で述べられていることを正確に確認しました。しかし、なぜ？ ポイント番号1は基本的に除外されました。ポイント番号2-あまりにも、ので同じIPアドレスからのテスト接続が異なる内部ホストが機能しました。ポイント番号4は完全に「zhost」です。しかし、3番目はすでに近いです。ブロックしないだけで、何らかの理由でホストがアスタリスクとパケットを交換することはできません。 そして、これは、PPPoEセッションが中断するポイント2以降の99％のケースで発生します。その後、セッションが復元され、LANからインターネットへのアクセスも可能になりますが、UDPセッションはスタックしています。 これらは、これらのセッションがどのように実行されるかをMikrotikが私たちに伝えたものです。

これは、すべてが機能しているときの通常モードのセッションの状態です。登録がない場合、メモリが正しく機能する場合、フラグS-見られた返信はありません。

次のコマンドを使用して、スタックしたセッションを削減しようとします。

[admin@MikroTik] > ip firewall connection remove [find where dst-address~":5060" and srcnat]

そして、なんと見れば、文字通り2番目のトランクが復元され、通話が続くと、明るい色で生命が咲きます！スクリプトを作成しています。

まず、アスタリスクからの情報に基づいてMikrotを制御する必要があります。したがって、スクリプトは、アスタリスクのあるホスト上で5分ごとにcronで実行され、証明書を使用してssh経由でMikrotを管理します。ここにすべてが完璧に描かれています-キーを作成してミクロティアの新しいユーザーに結び付ける方法。

まあ、スクリプト自体：

#!/bin/sh

if /usr/sbin/asterisk -x "sip show registry" | grep -q "Request Sent"; then
    echo "$(date) Resetting UDP connections on Mikrotik" >> /var/log/asterisk/mikrotik
    ssh -l admin-ssh -i /XXXX/.ssh/id_dsa 10.X.X.X 'ip firewall connection remove [find where dst-address~":5060" and srcnat]'
    sleep 2
    asterisk -x "sip reload"
fi

念のため、ハングアップしたセッションのリセットから2秒後に、登録を更新します-これで確実に機能します！

ログから判断すると、2019年9月4日以降、スクリプトは273回動作しました。彼が私にどれだけの時間を節約できたか、そしてマネージャーとそのクライアントの神経を計算できます。

PS同様の状況は、アスタリスクと一緒に出会ったすべてのルーターで発生します。そして、他のサービスに影響を与えずに問題を解決できるのはMikrotだけです。残りのルーターは愚かに過負荷にする必要があります。
