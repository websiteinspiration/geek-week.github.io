InstagramサーバーコードはPythonでのみ記述されています。まあ、基本的にそうです。私たちは少しのCythonを使用しており、依存関係には多くのC ++（C拡張と同様にPythonから操作できるコード）が含まれています。 私たちのサーバーアプリケーションはモノリスです。これは、数百万行で構成され、数千のDjangoエンドポイントを含む1つの大きなコードベースです（InstagramでのDjangoの使用についての話です）。これらすべてが1つのエンティティとして読み込まれ、維持されます。モノリスからいくつかのサービスが割り当てられていますが、私たちの計画には、モノリスの強力な分離は含まれていません。





私たちのサーバーシステムは、頻繁に変更されるモノリスです。毎日、何百人ものプログラマーが何百ものコードにコミットしています。これらの変更は継続的に展開され、7分ごとに実行されます。その結果、プロジェクトは1日に約100回本番環境にデプロイされます。コミットをマスターブランチに入れてから、対応するコードを本番環境にデプロイするまでに1時間もかからないように努めています（ここでは、 PyCon 2019で行われたこれについて説明します）。

この巨大なモノリシックコードベースを維持し、毎日何百ものコミットを行うと同時に、完全な混乱の状態にすることは非常に困難です。プログラマーが生産的になり、新しい便利なシステム機能をすばやく準備できる場所をInstagramにしたいと考えています。

この資料では、リンティングと自動リファクタリングを使用してPythonコードベースの管理を容易にする方法に焦点を当てています。

この資料で言及されているアイデアのいくつかを試してみたい場合は、最近、オープンソースプロジェクトLibCSTのカテゴリに移行したことを知っておく必要があります。

Linting：必要な場所に表示されるドキュメント


Lintingは、プログラマーがコードで気づかないと開発者自身が知らない問題やアンチパターンを見つけて診断するのに役立ちます。これは私たちにとって重要です。なぜなら、コードの設計に関する関連するアイデアは配布が難しくなるほど、より多くのプログラマーがプロジェクトに取り組むためです。私たちの場合、何百人もの専門家について話している。


Lintingの種類

Lintingは、Instagramで使用するさまざまな静的コード分析の1つにすぎません。

lintingルールを実装する最も原始的な方法は、正規表現を使用することです。正規表現は簡単に作成できますが、Pythonは「通常の」言語ではありません。その結果、正規表現を使用してPythonコード内のパターンを確実に検索することは非常に困難です（場合によっては不可能です）。

リンターを実装するための最も複雑で高度な方法について話す場合、 mypyや Pyreなどのツールがあります。。これらは、プログラムの詳細な分析を実行できるPythonコードの静的型チェック用の2つのシステムです。 InstagramはPyreを使用しています。これらは強力なツールですが、拡張やカスタマイズが困難です。

インスタグラムでリントについて話すとき、私たちは通常、抽象的な構文ツリーに基づいた単純なルールで作業することを意味します。サーバーコードのリンティングルールの基礎となるのは、まさにこのようなものです。

Pythonがモジュールを実行するとき、モジュールはパーサーを起動し、ソースコードをそれに渡します。これにより、解析ツリー（一種の具象構文ツリー（CST））が作成されます。このツリーは、入力ソースコードのロスレス表現です。コメント、角括弧、コンマなどの各詳細は、このツリーに保存されます。 CSTに基づいて、元のコードを完全に復元できます。


lib2to3によって生成されたPython解析ツリー（CSTのバリアント）

残念ながら、このアプローチは複雑なツリーの作成につながり、関心のある意味情報を抽出することが困難になります。

Pythonは解析ツリーを抽象構文ツリー（AST）にコンパイルします。この変換中に、ソースコードに関する一部の情報が失われます。コメント、角括弧、カンマなどの「追加の構文情報」について話している。ただし、ASTのコードのセマンティクスは保持されます。


astモジュールによって生成されたPythonの抽象構文ツリー LibCST

を開発しました。これは、CSTとASTの優れた機能を提供するライブラリです。 （CSTのように）コードに関するすべての情報が格納されているコードの表現を提供しますが、（ASTを使用する場合のように）コードのこの表現からコードに関する意味論的情報を簡単に抽出できます。


特定のLibCST構文ツリー

の表現リンティングルールは、LibCST構文ツリーを使用してコード内のパターンを検索します。この構文ツリーは、高レベルで簡単に探索でき、「不規則な」言語での作業に伴う問題を取り除くことができます。

特定のモジュールで、型のインポートが原因で循環依存関係があるとします。Pythonは、ブロックにタイプインポートコマンドを配置することでこの問題を解決しますif TYPE_CHECKING。これは、プログラムの実行中に何かをインポートすることに対する保護です。

#   
from typing import TYPE_CHECKING
from util import helper_fn

#   
if TYPE_CHECKING:
    from circular_dependency import CircularType

その後、誰かが別のタイプのインポートと別のセキュリティブロックをコードに追加しましたif。しかし、これを行った人は、そのようなメカニズムがモジュールにすでに存在していることを知らないかもしれません。

#   
from typing import TYPE_CHECKING
from util import helper_fn

#   
if TYPE_CHECKING:
    from circular_dependency import CircularType

if TYPE_CHECKING:  #   !
    from other_package import OtherType

リンターのルールを使用して、この冗長性を取り除くことができます！

コードで見つかった「保護」ブロックのカウンターを初期化することから始めましょう。

class OnlyOneTypeCheckingIfBlockLintRule(CstLintRule):
    def __init__(self, context: Context) -> None:
        super().__init__(context)
        self.__type_checking_blocks = 0

次に、対応する条件を満たすことで、カウンターをインクリメントし、コード内にそのようなブロックが1つだけあることを確認します。この条件が満たされない場合、コードの適切な場所に警告が生成され、そのような警告を生成するために使用される補助メカニズムが発生します。

def visit_If(self, node: cst.If) -> None:
    if node.test.value == "TYPE_CHECKING":
        self.__type_checking_blocks += 1
        if self.__type_checking_blocks > 1:
            self.context.report(
                node,
                "More than one 'if TYPE_CHECKING' section!"
            )
