シンプルでC ++。Userver Basics-非同期マイクロサービスを作成するためのフレームワーク

Yandex.Taxiはマイクロサービスアーキテクチャに準拠しています。マイクロサービスの数が増えるにつれ、開発者が定型文や典型的な問題に多くの時間を費やしていることに気づきましたが、ソリューションが常に最適に機能するとは限りません。

C ++ 17とコルーチンを使用して、独自のフレームワークを作成することにしました。これは、一般的なマイクロサービスコードの外観です。

Response View::Handle(Request&& request, const Dependencies& dependencies) {
  auto cluster = dependencies.pg->GetCluster();
  auto trx = cluster->Begin(storages::postgres::ClusterHostType::kMaster);

  const char* statement = "SELECT ok, baz FROM some WHERE id = $1 LIMIT 1";
  auto row = psql::Execute(trx, statement, request.id)[0];
  if (!row["ok"].As<bool>()) {
    LOG_DEBUG() << request.id << " is not OK of " << GetSomeInfoFromDb();
    return Response400();
  }

  psql::Execute(trx, queries::kUpdateRules, request.foo, request.bar);
  trx.Commit();

  return Response200{row["baz"].As<std::string>()};
}

そして、これが非常に効果的で速い理由です-カットの下でお知らせします。

Userver-非同期


私たちのチームは、経験豊富なC ++開発者だけで構成されているわけではありません。研修生、ジュニア開発者、さらにはC ++での記述に特に慣れていない人々もいます。したがって、userverの設計は使いやすさに基づいています。ただし、データ量と負荷があるため、非効率的に鉄の資源を浪費する余裕はありません。

マイクロサービスは、入力/出力の期待によって特徴付けられます。多くの場合、マイクロサービスの応答は、他のマイクロサービスおよびデータベースからのいくつかの応答から形成されます。効率的なI / O待機のタスクは、非同期メソッドとコールバックによって解決されます。非同期操作では、実行スレッドを生成する必要がないため、スレッドの切り替えに大きなオーバーヘッドがありません。コードのみを記述して維持するのは非常に困難です。

void View::Handle(Request&& request, const Dependencies& dependencies, Response response) {
  auto cluster = dependencies.pg->GetCluster();

  cluster->Begin(storages::postgres::ClusterHostType::kMaster,
    [request = std::move(request), response](auto& trx)
  {
    const char* statement = "SELECT ok, baz FROM some WHERE id = $1 LIMIT 1";
    psql::Execute(trx, statement, request.id,
      [request = std::move(request), response, trx = std::move(trx)](auto& res)
    {
      auto row = res[0];
      if (!row["ok"].As<bool>()) {
        if (LogDebug()) {
            GetSomeInfoFromDb([id = request.id](auto info) {
                LOG_DEBUG() << id << " is not OK of " << info;
            });
        }    
        *response = Response400{};
      }

      psql::Execute(trx, queries::kUpdateRules, request.foo, request.bar, 
        [row = std::move(row), trx = std::move(trx), response]()
      {
        trx.Commit([row = std::move(row), response]() {
          *response = Response200{row["baz"].As<std::string>()};
        });
      });
    });
  });
}

そして、ここでstackfull-coroutinesが助けになります。フレームワークのユーザーは、通常の同期コードを書いていると考えています。

  auto row = psql::Execute(trx, queries::kGetRules, request.id)[0];

ただし、内部では、次のようなことが起こります。

  1. TCPパケットが生成され、リクエストとともにデータベースに送信されます。
  2. View :: Handle関数が現在実行されているコルーチンの実行は中断されます。
  3. 私たちはOSのカーネルに次のように言います。
  4. 前のステップを待たずに、キューから実行可能な別のコルーチンを取得して起動します。

つまり、最初の例の関数は非同期で動作し、C ++ 20コルーチンを使用するコードに近いものです。

Response View::Handle(Request&& request, const Dependencies& dependencies) {
  auto cluster = dependencies.pg->GetCluster();
  auto trx = co_await cluster->Begin(storages::postgres::ClusterHostType::kMaster);

  const char* statement = "SELECT ok, baz FROM some WHERE id = $1 LIMIT 1";
  auto row = co_await psql::Execute(trx, statement, request.id)[0];
  if (!row["ok"].As<bool>()) {
    LOG_DEBUG() << request.id << " is not OK of " << co_await GetSomeInfoFromDb();
    co_return Response400{"NOT_OK", "Please provide different ID"};
  }

  co_await psql::Execute(trx, queries::kUpdateRules, request.foo, request.bar);
  co_await trx.Commit();

  co_return Response200{row["baz"].As<std::string>()};
}

これは、ユーザーがco_awaitとco_returnについて考える必要がないということだけです。すべてが「独自に」機能します。

私たちのフレームワークでは、コルーチン間の切り替えはstd :: this_thread :: yield（）を呼び出すよりも高速です。マイクロサービス全体のコストはごくわずかです。

I / Oバウンドタスクを解決するための上記の非同期アプローチは、Go開発者にはおなじみのはずです。ただし、Goとは異なり、ガベージコレクターからメモリとCPUのオーバーヘッドを取得しません。開発者は、一貫性、RAII、またはテンプレートの欠如に悩まされることなく、さまざまなコンテナーと高性能ライブラリーを備えた、よりリッチな言語を使用できます。

Userver-コード生成


例の最初の行に戻り、ResponseとRequestの背後にあるものを見てみましょう。

Response Handle(Request&& request, const Dependencies& dependencies);

userverでは任意のマイクロサービスを作成できますが、マイクロサービスにはAPIを文書化する必要があります（swaggerスキームで説明）。

たとえば、ハンドルの例の場合、swaggerダイアグラムは次のようになります。

さて、開発者はすでに要求と応答の説明を含むスキームを持っているので、それに基づいてこれらの要求と応答を生成しませんか？同時に、protobuf / flatbuffer / ...ファイルへのリンクも図に示すことができます-リクエスト自体からのコード生成はすべてを取得し、図に従って入力データを検証し、それを応答構造のフィールドに分解します。ユーザーはHandleメソッドで機能を記述するだけでよく、要求の解析と応答のシリアル化を伴うボイラープレートに気を取られることはありません。

同時に、コードの生成はサービスの顧客にも有効です。このようなスキームに従って動作するクライアントがサービスに必要であることを指定し、非同期リクエストの作成に使用できるクラスを取得できます。

Request req;
req.id = id;
req.foo = foo;
req.bar = bar;
dependencies.sample_client.SomeSampleBarPost(req);

このアプローチにはもう1つの利点があります。それは、常に最新のドキュメントです。開発者がドキュメントにないパラメータを突然使用しようとすると、コンパイルエラーが発生します。

Userver-ロギング


ログを書き込むのが大好きです。最も重要な情報のみをログに記録すると、1時間あたり数テラバイトのログが実行されます。したがって、ロギングに独自のトリックがある
ことは当然のことです。*非同期です（もちろん:-)）;
*遅いstd ::ロケールとstd :: ostreamをバイパスしてログに記録できます。
*（サービスを再起動せずに）ログレベルをオンザフライで切り替えることができます。
*ロギングにのみ必要な場合は、ユーザーコードを実行しません。

たとえば、マイクロサービスの通常の操作中、ログレベルはINFOに設定され、式全体が

    LOG_DEBUG() << request.id << " is not OK of " << GetSomeInfoFromDb();

計算されません。リソースを大量に消費する関数GetSomeInfoFromDb（）への呼び出しが含まれることはありません。

サービスが突然「だまされ」始めた場合、開発者は常に「DEBUGモードでログイン」という作業中のサービスに通知できます。この場合、「is not of」のエントリがログに表示され始め、GetSomeInfoFromDb（）関数が実行されます。
