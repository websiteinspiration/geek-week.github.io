iOSモバイルバンキングアプリケーションでのSwiftLint静的コードアナライザーの使用

リモートサービスチャネルコンピテンスセンター責任者Oleg Ivanov



みなさん、こんにちは！記事「ICD Mobile Bank：Development History」で約束したように、本日は、銀行のiOSモバイルアプリケーションにおける静的コードアナライザーとそのアプリケーションのエクスペリエンスについてお話したいと思います。

このツールキットを使用して達成したい目標を設定しましょう：

  • エラーと欠点の早期検出。
  • コードスタイル（コード設計標準は、プログラムコードを記述するときに使用される一連の規則と規則です）。

私たちの目標を達成する一般的な方法は、手動でコードを確認することです- コードレビュー。

テスターまたはテスターのグループは、テスト対象のコードを慎重に調査し、将来エラーになる可能性のあるエラーまたはコードのセクションを特定し、しばらくしてから（！）開発者によって分析および修正されるコードのコメントを通じて、改善のための推奨事項を提供します。ご覧のとおり、長くて高価なコード検証プロセスが出現しています。さらに、常に人的要因があります-一部のエラーは、レビュー担当者が単にスキップすることができます。

ここで静的コードアナライザーが役立ちます。

静的コードアナライザー-プログラムのソースコードのエラーや欠点を検出するための自動プロセスを開始します。

静的コードアナライザーは万能薬でも手動コードレビューの代わりでもありませんが、チェックを実行してパターン化されたエラーをすばやく見つけるのにかかる時間を短縮する優れたツールです。静的アナライザーの使用の関連性は、時間の経過とともに大きくなります。

iOSプロジェクトの静的コードアナライザーとして、SwiftLintを使用します。

SwiftLintは、プロジェクトのビルド段階で実行されるSwiftコード自動チェックユーティリティです。ユーティリティには、カスタムルールでこのセットを補足する機能を持つ一連のルールが含まれています。このツールは、コードスタイルに準拠するためにも使用されます。

アプリケーションでコードスタイルに従うことが重要なのはなぜですか？
プロジェクトの1人の開発者であれば、すべてが簡単です。独自のスタイルで記述し、このようにコードを購入する余裕があります。



ただし、大規模な開発チームで作業する場合は、重要な要素は、他の人のコードに改善を挿入する場所を理解して見つける速度です。
そして、ここですべてのチームメンバーは、コードを記述するための規則と規則を受け入れる必要があります。しかし、それらのコンプライアンスを確認するにはどうすればよいですか？ここでも、SwiftLint静的コードアナライザーが役立ちます。そして、私たちのコードはチームのすべてのメンバーが理解できる楽しい見た目になります：

SwiftLintのインストールは簡単です。

  1. ポッド「SwiftLint」をプロジェクトのポッドファイルに追加します
  2. 新しい「スクリプト実行フェーズ」をプロジェクトに追加します。

    "${PODS_ROOT}/SwiftLint/swiftlint"
  3. SwiftLint .swiftlint.yml
  4. .swiftlint.yml , SwiftLint

私たちは承認し難いため、ルールを作成しました印刷機能。印刷はかなり難しい操作です。設定.swiftlint.ymlファイルを介して：

custom_rules:
  disable_print:
    included: ".*\\.swift"
    name: "print usage"
    regex: "((\\bprint)|(Swift\\.print))\\s*\\("
    message: "Prefer os_log over print"
    severity: error

このルールにより、ロギング機能を（ロギングレベルで）作成するように求められました。このロギングは、開発者が迅速なデバッグのために使用します。たとえば、パラメータのあるログ、リクエスト/レスポンスの本文は、開発のためにエラーの分析に必要です。リリースの場合、.noneバージョンのログレベル。残りの印刷機能を使用すると、プロジェクトのビルドエラーが発生します。

func logApp(level: Constants.LogLevel, items: Any...) {
    if Constants.logLevel == .none {
        return
    }
    
    if level.rawValue <= Constants.logLevel.rawValue {
        // swiftlint:disable disable_print
        if let strings = items as? [String] {
            for string in strings {
                print(string)
            }
        } else {
            print(items)
        }
        // swiftlunt:enable disable_print
    }

しかし、print関数を使用するには、SwiftLintを使用してログ関数の呼び出しを閉じ、特別な命令でマークされたコードブロックに関する独自のルールを無視する必要がありました。

// swiftlint:disable disable_print
// swiftlunt:enable disable_print

SwiftLintには独自のルールを無視する機能があります。

// swiftlint:disable <rule1 [rule2 rule3…]>
<,   SwiftLint   rule1 [rule2 rule3…]>
// swiftlunt:enable <rule1 [rule2 rule3…]>

または

// swiftlint:disable all
<,   SwiftLint   >
// swiftlint:enable all

この機会を利用して、必要であることを十分に認識してください。

結論として、私は次のように述べています。私たちのチームでSwiftLintを使用すると、手動によるコードレビューのコストが20％削減されました。これで、レビュー担当者は一般的なエラー（フォースキャストなど）、コードスタイルに注意を払わず、新しいコードのチェックに集中できます。これにより、チーム全体の効率が大幅に向上しました（このようなエラーチェックを修正する必要はありません。これらは有能な従業員であり、時間は非常に重要です）。

すべて！今、SwiftLintはあなたと永遠に一緒です:)
