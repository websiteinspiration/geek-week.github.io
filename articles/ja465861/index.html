機密の「クラウド」。オープンなソリューションに代わるものを探しています

私はトレーニングでエンジニアですが、起業家や制作ディレクターともっとコミュニケーションをとっています。少し前、工業会社のオーナーがアドバイスを求めました。会社が大きく、90年代に設立されたという事実にもかかわらず、管理と会計はローカルネットワークで昔ながらの方法で機能します。

これは彼らのビジネスに対する懸念と国家による支配の増大の結果です。法規制は検査機関によって非常に広く解釈されます。例は、税法の改正であり、税法違反の制定法、銀行および監査秘密の実際の破壊を排除します。

その結果、企業の所有者は、情報を確実に保存し、ドキュメントを安全に転送するためのソリューションを探し始めました。仮想「安全」。

私たちは常勤のシステム管理者と一緒にタスクに取り組みました。既存のプラットフォームの詳細な分析が必要です。

  • サービスは、古典的な意味での曇りであってはなりません。第三者機関の施設で保存することなく。サーバーのみ。
  • 送信および保存されたデータの強力な暗号化が必要です。
  • ボタンをクリックするだけでデバイスからコンテンツを緊急に削除する機能は必須です。
  • ソリューションは海外で開発されています。


4番目のポイントを削除することを提案しました。ロシアのアプリケーションには公式の証明書があります。局長はそのような証明書で何をすべきかを明確に述べました。

オプションを選択


私は3つのソリューションを選択しました（選択肢が多いほど、疑問が増えます）。
  • オープンソースは、開発マニアのジェイコブボルグによってサポートされているSyncthing.netプロジェクトです。
  • Resilio.com、American Resilio Inc. （以前はBitTorrent Syncと呼ばれていました）。
  • Pvtbox.net同期アプリケーションからのPvtbox Project Electronic Safe。キプロス登録。

彼の最初の質問は次のとおりです。無料で何かを開発する意味は何ですか？Syncthingはすぐに放棄されました。引数は機能しませんでした。

数日後、顧客はサポートの欠如のためにResilio Syncを断念して却下しました。緊急時にどこを曲がればよいか明確ではありません。さらに、アメリカの企業登録に対する不信感。

さらなる分析のために、Pvtbox Electronic Safeは残りました。傍受、データの復号化、情報ストレージへの不正な入力の可能性に重点を置いて、このプラットフォームの完全な技術監査を実施しました。

監査プロセス


プログラムの開始時、作業中、穏やかな状態で化合物の分析を行いました。現代の標準によるトラフィックは元々暗号化されています。Linux（Xubuntu Linux 18.04）、Wireshark、Mitmproxyを使用して、MITM攻撃を実行し、その場で証明書を置き換えてみましょう。これを行うには、Pvtboxアプリケーションとpvtbox.netサーバーの間に仲介を実装します（https接続を介してpvtbox.netサーバーとのデータ交換があります）。

アプリケーションを起動して、プログラムとファイルの同期が機能することを確認します。 Linuxでは、ターミナルからプログラムを実行すると、ログをすぐに観察できます。


アプリケーションをオフにして、ファイル/ etc / hostsのホストアドレスpvtbox.netを置き換えますスーパーユーザー権限で。アドレスは、プロキシサーバーのアドレスに置き換えられます。


次に、ローカルネットワークのアドレスが192.168.1.64のコンピューターに対するMITM攻撃に備えてプロキシサーバーを準備します。これを行うには、mitmproxyパッケージバージョン4.0.4をインストールします。

ポート443でプロキシサーバーを実行します。
$ sudo mitmproxy -p 443

最初のコンピューターでPvtboxプログラムを実行し、mitmproxyの出力とアプリケーションログを確認します。


Mitmproxyは、クライアントがプロキシサーバーからのなりすまし証明書を信頼していないことを報告します。アプリケーションログでは、プロキシサーバーの証明書が検証に合格せず、プログラムが機能しなくなっていることもわかります。mitmproxy

プロキシ証明書をインストールします証明書を「信頼できる」ものにするために、Pvtboxアプリケーションがインストールされたコンピューターに接続します。コンピューターに、ca-certificatesパッケージをインストールします。次に、mitmproxy-ca-cert.pem証明書を、プロキシサーバーの.mitmproxyディレクトリから、/ usr / local / share / ca-certificatesディレクトリにあるPvtboxアプリケーションがインストールされているコンピュータにコピーします。

コマンド
を実行します。$ sudo openssl x509 -in mitmproxy-ca-cert.pem -inform PEM -out mitmproxy-ca-cert.crt
$ sudo update-ca-certificates


Pvtboxアプリケーションを起動します。証明書は再度検証に失敗しました。プログラムは機能しません。アプリケーションはおそらく、証明書固定セキュリティメカニズムを使用しています。

同様の攻撃がsignalserver.pvtbox.netホストで実行されました。また、ノード間のピア2ピア接続自体にも。開発者は、ピア2ピア接続を確立するためのアプリケーションが、DTLSv1.2プロトコルを使用したエンドツーエンドの暗号化を使用するオープンwebrtcプロトコルを使用することを示しています。

鍵は接続設定ごとに生成され、signalserver.pvtbox.netを介して暗号化されたチャネルを介して送信されます。

理論的には、WebRTCメッセージの提供を傍受し、WebRTCメッセージに応答し、そこで暗号化キーを置き換え、WebRTC経由で到着するすべてのメッセージを復号化することができます。しかし、signalserver.pvtbox.netにミット攻撃を実行することはできなかったため、signalserver.pvtbox.netを介して送信されたメッセージを傍受して置き換える可能性はありません。

したがって、ピア2ピア接続でこの攻撃を実行することはできません。

プログラムに付属の証明書ファイルも検出されました。このファイルは、パス/opt/pvtbox/certifi/cacert.pemにあります。このファイルは、mitmproxyプロキシサーバーからの信頼できる証明書を含むファイルに置き換えられました。結果は変更されていません-プログラムはシステムへの接続を拒否しました
。証明書で検証に合格しなかったという同じエラーがログで確認されました。

監査結果


トラフィックを傍受または交換できませんでした。ファイル名、さらにその内容が暗号化された形式で送信されるように、エンドツーエンドの暗号化が使用されます。アプリケーションは、盗聴や実装を防ぐいくつかの保護メカニズムを実装しています。

その結果、会社は情報に永続的にアクセスするために2つの専用サーバー（物理的に異なる場所にある）を購入しました。最初のサーバーは情報の受信、処理、保存に使用され、2番目のサーバーはバックアップに使用されます。

受け取った個々のクラウドには、ディレクターの作業用端末とiOS携帯電話が接続されていました。他の従業員は、通常のシステム管理者とPvtboxテクニカルサポートによって接続されていました。

過去から、友人からの苦情はありませんでした。私のレビューがHabrの読者にとって同じような状況で役立つことを願っています。
