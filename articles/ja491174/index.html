フロントエンドコンポーネントとは何か、なぜ必要なのか


HTMLタグ -条件付きで「ゼロ」レベルのコンポーネント。それぞれに独自の機能と目的があります。

CSSクラスは、小さなサイトを作成するときに通常到達する次のレベルの抽象化です。 CSSクラスにスタイルを適用するためのルールでは、要素の条件付きサブセットの一部であるすべての要素の動作について説明します。

CSSクラス、およびHTMLタグなどの他の要素に適用されるルールを使用すると、同じタイプの要素をいくつでも表示するためのルールを一元的に設定および変更できます。要素のスタイル（実際にはCSS、Sass、LESS、PostCSS、およびスタイルの適用方法）（BEM、SMACSS、Atomic CSS、CSSモジュール、スタイル付きコンポーネント）を操作するためのさまざまなツールがあります。

実際のコンポーネントは次のとおりです。

  • 同じスタイルと同じレイアウト（HTML）と動作（JS）の両方を持つ同じタイプの要素。
  • 互いにわずかに異なるスタイル要素と動作要素が似ています。

カスタムのHTMLタグを作成し、レイアウトにテンプレートのコードを含めることができるWebコンポーネントテクノロジーが開発されています。ただし、コンポーネントは、Angular、Vue、Reactなどの最新のフロントエンド開発フレームワークのおかげで広く使用されるようになりました。JavaScript機能により、コンポーネントを簡単に接続できます。

すべての主要なプロジェクトは、既製のコンポーネントのライブラリに来るか、既製のコンポーネントの1つを使用します。コードのコピーからコンポーネントの作成にいつ移行する必要があるかという問題は個別に決定されるため、明確なレシピはありません。

コードの記述だけでなく、そのサポートも覚えておく価値があります。同じレイアウトの単純なコピー/貼り付けとCSSクラスでのスタイルの分離により、特別なリスクなしにしばらく表示を作成できます。しかし、JSで記述された動作ロジックが各要素に追加された場合、特に以前に記述されたコードのサポートと変更に関しては、コードを再利用する利点は文字通り2〜3要素から感じられます。

再利用可能なReactコンポーネント


アプリケーションが十分に大きくなり、独自のコンポーネントライブラリを作成することにしたとします。これには、人気のあるReactフロントエンド開発ツールを使用することをお勧めします。その利点の1つは、組み込みコンポーネントを簡単かつ効率的に使用できることです。以下のコードでは、アプリの古いコンポーネントは3つのネストされたコンポーネントを使用しています：AppHeader、Article、AppFooter：

import React from "react";

            contacts : "8 800 100 20 30"
           firtsArticleTitle : "Welcome",
           secondArticleTitle : "Let's speak about..."
        }

    render() {
        return (
            <>

   }

注意：レイアウトで上級ラッピングタグを使用する必要はありませんdiv。通常はそうでした。 Modern ReactはFragmentツールを提供します<></>。これはその簡略記録です。これらのタグ内では、上記の例のように、フラットタグ階層を使用できます。

3つのライブラリコンポーネントを使用しました。そのうちの1つは1つのブロックで2回です。親アプリケーションからのデータはコンポーネントのプロップに転送され、プロパティを通じてその内部で利用できますthis.props。このアプローチはReactの典型的なものであり、典型的な要素からビューをすばやく組み立てることができます。特に、記事の内容（モデル）と機能のみが異なる多くの類似ページがアプリケーションにある場合。

ただし、ライブラリコンポーネントを変更する必要がある場合があります。たとえば、同じ機能を持つコンポーネントは、テキストコンテンツだけでなく、デザイン（色、インデント、境界線）も異なる場合があります。同じコンポーネントの異なる機能を提供することも可能です。

Reactコンポーネントを完成させる


ビューのみを担当するコンポーネントは上記で検討されました。ただし、ほとんどの場合、マッピングだけでなく標準のデータ処理ロジックもライブラリに送信する必要があります。

電話番号を入力するように設計された電話コンポーネントを見てみましょう。プラグインバリデーターライブラリを使用して、入力された番号をマスクし、電話が正しくまたは間違って入力されたことを上級コンポーネントに通知できます。

// Phone.js
import React from "react";
import Validator from "../helpers/Validator";
export default class Phone extends React.Component {
    constructor(props) {
        super(props);   
        this.state = {
            value : this.props.value || "",
            name : this.props.name,
            onceValidated : false,
            isValid : false,
            isWrong : true
        }
        this.ref = React.createRef();    
    };

    componentDidMount = () => {
        this.setValidation();
    };

    setValidation = () => {
        const validationSuccess = (formattedValue) => {
            this.setState({
            value : formattedValue,
            isValid : true,
            isWrong : false,
            onceValidated : true
           });
            this.props.setPhoneValue({
            value : formattedValue, 
            item : this.state.name, 
            isValid : true
            })
        }
        const validationFail = (formattedValue) => {
            this.setState({
            value : formattedValue,
            isValid : false,
            isWrong : true,
            });
            this.props.setPhoneValue({
            value : formattedValue, 
            item : this.state.name, 
            isValid : false
            })
        }
        new Validator({
            element : this.ref.current,
            callbacks : {
            success : validationSuccess,
            fail : validationFail
            }
        });
    }

    render() {
        return (
            <div className="form-group">
            <labeL htmlFor={this.props.name}>
                    <input 
                name={this.props.name}
                id={this.props.name}
                type="tel"
                placeholder={this.props.placeholder}
                defaultValue={this.state.value}
                ref={this.ref}
                />
            </label>
            </div>
        )
    }

}

このコンポーネントにはすでに内部状態があり、その一部を呼び出した外部コードと共有できます。上の例では、他の部分はコンポーネント内に残りonceValidatedます。したがって、コンポーネントのロジックの一部は完全にその中に含まれています。

一般的な動作は、アプリケーションの他の部分から独立していると言えます。たとえば、番号が検証されたかどうかに応じて、さまざまなテキストプロンプトを表示できます。ディスプレイだけでなく、データ処理ロジックも再利用可能な別のコンポーネントを取り入れました。

コンポーネントで実行されているアプリケーション全体の状態


上に示したように、アプリケーションは、パラメーターを渡すか、タイプセットすることによって、およびコールバックを使用することによって、コンポーネントを管理できます。

アプリケーションが正常に機能するためには、上位レベルで、ネストされた再利用コンポーネントの状態に関する意味のあるデータを受け取る必要があります。ただし、これはアプリケーション全体の最高レベルではない場合があります。

ログインとパスワードを入力するためのクライアント認証ブロックと再利用可能なコンポーネントがある場合、アプリケーション全体は、これらの単純なコンポーネントが常にどのような状態にあるかを知る必要はありません。むしろ、認証ブロック自体は、単純な再利用されたコンポーネントの状態に基づいて新しい状態を計算し、それを渡すことができます。認証ブロックは正しく入力されているかどうかがわかります。

コンポーネントのネストが大きい場合、「真の情報源」がどこにあるかを常に知るために、データを使用して作業の組織を監視する必要があります。

Reactでの非同期状態遷移に関連するいくつかの困難についてはすでに書いています。

再利用可能なコンポーネントは、コンポーネントの可能なブロックを管理するために必要なデータをコールバックを通じて常に渡す必要があります。ただし、DOMツリーの大部分を不必要に再描画したり、コンポーネントの変更を処理するコードを複雑にしたりしないように、余分なデータを転送する必要はありません。

データを編成する別のアプローチは、コンポーネント呼び出しコンテキストを使用することです。これはネイティブのReactメソッドです。createContext、バージョン16.3以降で利用可能React getChildContextです。以前のバージョンと混同しないでください！..

次に、コンポーネントのネストのツリーの下にあるコンポーネントの「厚さ」を介して小道具を渡す必要はありません。または、ReduxやMobxなどのデータ管理および変更配信のための専用ライブラリを使用します（Mobx + Reactバンドルに関する記事を参照）。

Mobxで再利用可能なコンポーネントのライブラリを構築する場合、そのようなコンポーネントのタイプごとに独自のストアがあります。つまり、アプリケーション全体のどこからでもエンドツーエンドでアクセスできる、コンポーネントの各インスタンスの状態に関する「信頼できる情報源」です。Reduxとその唯一のデータウェアハウスの場合、すべてのコンポーネントのすべての状態を1か所で利用できます。
