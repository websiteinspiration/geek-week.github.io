知らない人のために：Gettは、2011年にイスラエルで設立された国際的なタクシーサービスです。Gettは現在、イスラエル、イギリス、ロシア、米国の4か国で活動しています。当社の主な製品は、クライアントとドライバー向けのモバイルアプリケーション、自動車を注文できる企業向けクライアントのWebポータル、および従業員が関税計画を設定し、新しいドライバーを接続し、詐欺事件を監視するための一連の内部管理パネルです。2016年の終わりに、会社全体の利益のために活動するグローバルなR＆Dオフィスがモスクワに開設されました。

どうやって私たちが行きましたか


2011年、同社の主な製品はRuby on Rails上のモノリシックアプリケーションでした。当時、このフレームワークは非常に人気があったためです。 Ruby on Railsで迅速に開発および立ち上げられたビジネスの成功例があり、ビジネスの成功と関連しています。会社が開発していて、新しいドライバーとユーザーが私たちのところにやって来て、荷が増えました。そして、最初の問題が現れ始めました。

クライアントアプリケーションが車の位置を表示し、その動きが滑らかな曲線のように見えるようにするには、ドライバーが座標を送信する必要があります。したがって、ドライバーから座標を受信するエンドポイントは、ほとんどの場合、最も負荷が高くなります。そして、Ruby on RailsのWebサーバーフレームワークは、これをうまく機能させませんでした。広範囲にしか拡張できず、新しいアプリケーションサーバーを追加することは可能でしたが、これは高価で非効率的です。その結果、元々はJSで作成された別のサービスで座標の機能コレクションを取り出しました。しばらくの間、これで問題は解決しました。ただし、負荷が増大するにつれて8万RPMに達したとき、Node.js上のサービスは節約を停止しました。

次に、ハッカソンを宣言しました。会社のすべての従業員は、1日でプロトタイプを書く機会がありました。それはドライバーのコーディネートを収集することでした。そのサービスの2つのバージョンのベンチマークを以下に示します。prodで実行し、Goで書き直しました。


ほとんどすべての点で、囲碁のサービスは最高の結果を示しました。Node.jsのサービスはクラスターを使用しましたが、これはマシンのすべてのコアを使用するためのテクノロジーです。つまり、実験は正または負の正直でした。Node.jsにはシングルスレッドランタイムの欠点がありますが、結果には影響しません。

徐々に、私たちの製品の需要は高まりました。私たちはますます機能性を開発し、そのような問題に遭遇すると、ある場所にコードの一部を追加すると、プロジェクトが強く接続されている別の場所で何かが壊れる可能性があります。私たちは、サービス指向のアーキテクチャーに切り替えることで、この惨事を克服することにしました。しかし、結果としてパフォーマンスが低下しました。Rubyon Railsインタープリターがコードを実行しているときにランタイムリクエストが発生すると、そのリクエストはブロックされ、ワー​​カーはアイドル状態になります。そして、ネットワークI / O操作はますますなっています。

その結果、主要な開発言語の1つとしてGoを採用することにしました。

製品開発の特徴


まず、非常に異なる製品要件があります。私たちの車は3つの国を完全に異なる法律で運転しているため、非常に異なる機能セットを実装する必要があります。たとえば、イスラエルでは、旅行の費用をタクシーメーターで考慮することが法的に義務付けられています。これは、数年ごとに必須の認証を通過するデバイスです。運転手は、旅行を開始するときに「移動」ボタンを押し、終了したら「停止」ボタンを押して、タクシーメーターが示す価格をアプリケーションに入力します。

ロシアにはそのような厳しい法律はありません。ここで、価格ポリシーを自分で設定できます。たとえば、移動時間や距離に関連付けます。同じ機能を実装したい場合は、最初に1つの国で導入し、次に他の国で採用して導入します。

私たちの製品マネージャーは、製品ストーリーの形で要件を設定します。私たちは、そのようなアプローチに忠実に従うようにしています。これにより、自動的にテストにマークが付けられます。動作駆動型の開発方法論を使用して、入ってくる製品要件をテスト状況に投影できるようにします。プログラミングから離れている人にとっては、テスト結果を読んで何が何であるかを理解する方が簡単です。

また、一部の作業の重複を解消したいと考えました。結局のところ、ある種の機能を実装するサービスがあり、2番目のサービスを作成して、最初に解決したすべての問題を解決し、監視および移行ツールと再統合する必要がある場合、これは効果がありません。

フレームワーク


Ruby on RailsはMVCアーキテクチャに基づいて構築されています。移行の時点では、このフレームワークでのみプログラミングできる開発者の生活を楽にするために、それを放棄したくはありませんでした。ツールを変更しても快適さは増しません。アプリケーションのアーキテクチャも変更した場合、それはボートから泳ぐ方法を知らない人を押すのと同じです。この方法で開発者を傷つけたくなかったので、当時Beegoと呼ばれる数少ないMVCフレームワークの1つを採用しました。

Ruby on RailsのようなBeegoを使用してサーバーレンダリングを試みました。ただし、サーバーにレンダリングされたページはあまり好きではありませんでした。 1つのコンポーネントを破棄する必要がありましたが、今日のBeegoはバックエンドからJSONのみを生成し、すべてのレンダリングはフロントでReactによって実行されます。

Beegoでは、プロジェクトを自動的に構築できます。一部の開発者にとって、スクリプト言語からコンパイルの必要性に切り替えることは非常に困難でした。人がいくつかの機能を実装したときに面白い話がありました、そしてコードレビューによってのみ、または偶然にそれさえわかったとしても、結局、Goビルドを実行する必要があります。そして、タスクはすでにクローズされています。

Beegoでは、ルーターはコメントから生成され、開発者がコントローラーアクションへのパスを書き込みます。たとえば、ルータにタイプミスがあった場合、このアプローチに精通していない人が間違いを見つけるのは難しいため、この考え方にはあいまいな態度をとっています。数時間のエキサイティングなデバッグを行った後でも、人々はその理由を理解できなかった。

データベース


データベースとしてPostgreSQLを使用しています。そのような習慣があります-アプリケーションコードからデータベーススキーマを制御することです。これはいくつかの理由で便利です：誰もがそれらについて知っています。導入は簡単で、データベースは常にコードと同期しています。また、これらのパンを保持したいと考えていました。

複数のプロジェクトとチームがある場合、機能を実装するために、他の人のプロジェクトにクロールする必要があります。また、テーブルに列を追加するのは非常に魅力的です。この列には1000万のレコードがある場合があります。そして、このプロジェクトに没頭していない人は、テーブルのサイズに気付かないかもしれません。これを防ぐために、データベースの記録をブロックする可能性のある危険な移行に関する警告を発行し、開発者にこの警告を削除する手段を提供しました。

マイグレーション


パッチを当てたガチョウであるSwan を使用して移行することを決定し、いくつかの改良を加えました。これら2つは、多くの移行ツールと同様に、すべてを1つのトランザクションで実行する必要があるため、問題が発生した場合に簡単にロールバックできます。場合によっては、インデックスを作成する必要があり、テーブルがロックされていることがあります。 PostgreSQLには、これを回避するパラメーターがあります。問題は、PostgreSQLでthisのインデックスの作成を開始すると、トランザクションでもエラーが発生することです。最初は、トランザクションを開かないようにフラグを追加したかったのです。そして最後に、彼らはこれをしました：concurrentlyconcurrently

COMMIT;
CREATE INDEX CONCURRENTLY huge_index ON huge_table (column_one, column_two);
BEGIN;

これで、誰かがパラメータ付きのインデックスを追加すると、concurrentlyこのヒントが得られます。混乱しないように注意commitしてくださいbegin。このコードは、移行ツールが開いたトランザクションを閉じ、パラメータを使用してインデックスをロールconcurrentlyし、ツールが何かを閉じるように別のトランザクションを開きます。

モニタリング


人々がアクセスしている本番サービスがある場合、その作業を監視する必要があります。500個のエラーが発生しているか、リクエストが迅速に処理されているかなどです。 Ruby on Railsの世界では、NewRelicが非常によく使用されており、開発者の多くはそれをよく所有しています。彼らは、ツールがどのように機能するか、問題があるかどうかをどこで確認するかを理解しました。 NewRelicを使用すると、HTTP経由の要求の処理時間を分析し、データベースへの遅い外部呼び出しと要求を識別し、データフローを監視し、インテリジェントなエラー分析とアラートを提供できます。

NewRelicにはApdex集約関数があります。これは、回答の期間の分布のヒストグラムと、正常であると思われ、最初に設定されたいくつかの値に依存します。この機能は、アプリケーションのエラーのレベルにも依存します。 NewRelicはApdexを計算し、その値があるレベルを下回ると警告を発します。
NewRelicは、最近Goの公式エージェントを持つこともできます。これは一般的な監視の概要です。



左側はクエリ処理図で、それぞれがセグメントに分割されています。セグメントには、リクエストのキューイング、ミドルウェア処理、Ruby on Railsインタープリターでの滞在期間、リポジトリへのアクセスが含まれます。

まず、リクエスト処理の継続時間を測定したいと考えました。これは、Beegoが提供するフックを使用して行われました。

beego.InsertFilter("*", beego.BeforeRouter, StartTransaction, false)
beego.InsertFilter("*", beego.AfterExec, NameTransaction, false)
beego.InsertFilter("*", beego.FinishRouter, EndTransaction, false)

重要な点は、トランザクションの開始とその命名を共有したことだけです。なぜこれをしたのですか？ルーティングに費やされた時間を考慮して、リクエスト処理の期間を測定したいと思いました。同時に、リクエストが送信されたエンドポイントごとに集計されたレポートが必要です。ただし、トランザクションを開いた時点では、一致が発生するURLパターンはまだ定義されていません。したがって、リクエストが到着したときにトランザクションを開き、フックでコントローラーを実行して名前を付け、処理後に閉じます。したがって、今日のレポートは次のようになります。


抽象化を維持し、開発者に純粋なSQLの記述を強制したくなかったため、GORMと呼ばれるORMを使用しました。このアプローチには、利点と欠点の両方があります。 Ruby on Railsの世界には、人々を本当に甘やかすORM Active Recordがあります。開発者は、純粋なSQLを記述でき、ORM呼び出しでのみ操作できることを忘れます。

結論


今日、75％以上の本番サービスがGoで記述されています。Rubyでのアクティブな開発は行わず、サポートするだけです。これに関して、私は注意したい：

  • 開発速度が低下するのではないかとの懸念は確認されていない。プログラマーはそれぞれ独自のモードで新しいテクノロジーに注力しましたが、平均して、数週間の活発な作業の後、Goでの開発はRuby on Railsでの開発と同じくらい予測可能かつ高速になりました。
  • 負荷がかかった状態でのGoアプリケーションのパフォーマンスは、過去の経験に比べて驚くほど驚くべきものです。AWSでのインフラストラクチャの使用を大幅に節約し、使用するインスタンスの数を大幅に削減しました。

  • 今日はすでにBeegoとGormを離れました。これについては次の記事で詳しく説明します。
