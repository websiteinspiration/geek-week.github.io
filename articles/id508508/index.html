Ketergantungan injeksi - lebih mudah daripada kedengarannya?

Halo, Habr!



Kami sedang mempersiapkan untuk merilis edisi kedua dari buku legendaris oleh Mark Siman, "Ketergantungan Injeksi dalam. NET."







Oleh karena itu, hari ini kami memutuskan untuk secara singkat menyegarkan topik injeksi ketergantungan untuk spesialis .NET dan C # dan menawarkan terjemahan artikel Graham Downs, di mana paradigma ini dipertimbangkan dalam konteks inversi kontrol (IoC) dan penggunaan wadah.



Sebagian besar programmer mungkin tahu fenomena Injeksi Ketergantungan, tetapi tidak semua orang mengerti artinya. Anda mungkin harus berurusan dengan antarmuka dan wadah, dan kadang-kadang bekerja dengan mereka membuat Anda menemui jalan buntu. Di sisi lain, Anda mungkin baru saja mendengar sesuatu tentang injeksi ketergantungan, dan membuka artikel ini karena Anda ingin lebih memahami esensinya. Dalam artikel ini, saya akan menunjukkan betapa sederhananya konsep injeksi ketergantungan, dan seberapa kuat sebenarnya bisa.



Injeksi ketergantungan adalah pendekatan mandiri yang dapat digunakan sendiri. Di sisi lain, pendekatan ini dapat diterapkan baik dengan antarmuka dan dengan wadah untuk injeksi dependensi / manajemen inversi (DI / IoC). Saat menerapkan injeksi ketergantungan dalam konteks ini, Anda mungkin mengalami beberapa kebingungan, yang awalnya saya alami.



Sepanjang karir saya (saya berspesialisasi dalam pengembangan di Net / C #), saya biasa menggunakan injeksi ketergantungan dalam bentuk yang paling murni. Pada saat yang sama, saya menerapkan DI tanpa menggunakan wadah atau inversi kontrol sama sekali. Segalanya berubah baru-baru ini ketika saya diberi tugas di mana penggunaan wadah sangat diperlukan. Kemudian saya sangat meragukan semua yang saya tahu sebelumnya.



Setelah bekerja dengan gaya ini selama beberapa minggu, saya menyadari bahwa kontainer dan antarmuka tidak mempersulit pelaksanaan dependensi, tetapi, sebaliknya, memperluas kemampuan paradigma ini.


(Penting untuk dicatat di sini: antarmuka dan wadah hanya digunakan dalam konteks injeksi dependensi. Injeksi ketergantungan dapat diimplementasikan tanpa antarmuka / wadah, tetapi, pada dasarnya, satu-satunya tujuan antarmuka atau wadah adalah untuk memfasilitasi implementasi dependensi).



Artikel ini akan menunjukkan bagaimana injeksi ketergantungan dilakukan, baik dengan dan tanpa antarmuka dan wadah. Saya berharap bahwa setelah membacanya, Anda akan memahami dengan jelas prinsip injeksi ketergantungan dan akan dapat membuat keputusan berdasarkan informasi tentang kapan dan di mana harus menggunakan antarmuka dan wadah saat menerapkan dependensi.



Latihan



Untuk lebih memahami injeksi ketergantungan dalam bentuk paling murni, mari kita lihat contoh aplikasi yang ditulis dalam C #.



Pertama, perhatikan bahwa aplikasi seperti itu dapat ditulis tanpa suntikan ketergantungan. Kemudian kami akan memperkenalkan injeksi ketergantungan ke dalamnya, menambahkan fitur logging sederhana.



Saat Anda maju, Anda akan melihat bagaimana persyaratan untuk penebangan secara bertahap menjadi lebih rumit, dan kami memenuhi persyaratan ini menggunakan injeksi ketergantungan; sementara area tanggung jawab kelas Calculator

Program.cs: 



using System;
using System.Linq;

namespace OfferZenDiTutorial
{
    class Program
    {
        static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            var calc = new Calculator();
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }

        private static float GetNumber(string message)
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write(message);
                var input = Console.ReadLine();
                isValid = float.TryParse(input, out var number);
                if (isValid)
                    return number;

                Console.WriteLine("Please enter a valid number. Press ^C to quit.");
            }

            return -1;
        }

        private static char GetOperator()
        {
            var isValid = false;
            while (!isValid)
            {
                Console.Write("Please type the operator (/*+-) > ");
                var input = Console.ReadKey();
                Console.WriteLine();
                var operation = input.KeyChar;
                if ("/*+-".Contains(operation))
                {
                    isValid = true;
                    return operation;
                }

                Console.WriteLine("Please enter a valid operator (/, *, +, or -). " +
                                  "Press ^C to quit.");
            }

            return ' ';
        }

        private static float GetResult(Calculator calc, float number1, float number2, 
            char operation)
        {
            switch (operation)
            {
                case '/': return calc.Divide(number1, number2);
                case '*': return calc.Multiply(number1, number2);
                case '+': return calc.Add(number1, number2);
                case '-': return calc.Subtract(number1, number2);
                default:
                    //    ,        
                    throw new InvalidOperationException("Invalid operation passed: " + 
                                                        operation);
            }
        }
    }
}

Filelogger kelas



Tidak. Tentu saja, ini bukan bidang tanggung jawabnya. Oleh karena itu, agar prinsip tanggung jawab tunggal tidak dilanggar , segala sesuatu yang berkaitan dengan pencatatan informasi harus ada dalam file log, dan untuk ini perlu untuk menulis kelas mandiri yang terpisah. Ayo lakukan.



Pertama-tama, kita membuat kelas yang sama sekali baru, sebut saja itu FileLogger

FileLogger.csh: 



using System;
using System.IO;

namespace OfferZenDiTutorial
{
    public class FileLogger
    {
        private const string FileName = "Calculator.log";
        private readonly string _newLine = Environment.NewLine;

        public void WriteLine(string message)
        {
            File.AppendAllText(FileName, $"{message}{_newLine}");
        }
    }
}


Sekarang semua yang berhubungan dengan membuat file log dan menulis informasi untuk itu diproses di kelas ini. Selain itu, kami mendapatkan satu hal yang menyenangkan: tidak peduli apa yang dikonsumsi oleh kelas ini, tidak perlu menempatkan garis kosong di antara catatan yang terpisah. Catatan harus memanggil metode kami WriteLine

Untuk menggunakan kelas, kita membutuhkan objek yang instantiate. Mari kita selesaikan masalah ini di dalam kelas Calculator

Jelas, jawaban untuk pertanyaan terakhir adalah negatif!

Injeksi ketergantungan adalah elemen dari topik yang lebih besar yang disebut Pembalikan Manajemen, tetapi pembahasan terperinci darinya berada di luar cakupan artikel ini.



Dalam hal ini, Anda hanya perlu tahu bahwa kami membalikkan kontrol kelas logger, atau, secara metaforis, kami mendelegasikan masalah membuat file ke seseorang dengan FileLogger

Intinya, ini adalah injeksi ketergantungan. Baik antarmuka, maupun wadah untuk inversi kontrol, atau hal-hal seperti itu diperlukan. Pada dasarnya, jika Anda harus melakukan sesuatu seperti itu, maka Anda berurusan dengan injeksi ketergantungan. Keren kan?



Perluasan kemungkinan: kami akan membuat logger lain

Penafian kecil tentang antarmuka



Seperti yang Anda lihat, antarmuka adalah alat yang sangat kuat, tetapi mereka membawa tingkat abstraksi tambahan ke aplikasi, dan, karenanya, kompleksitas ekstra. Kontainer dapat membantu dalam memerangi kompleksitas, tetapi, seperti yang akan ditampilkan di bagian selanjutnya, Anda harus mendaftarkan semua antarmuka Anda, dan ketika bekerja dengan tipe tertentu, ini tidak diperlukan.



Ada yang disebut "kebingungan dengan bantuan abstraksi", yang, pada dasarnya, bermuara pada re-komplikasi proyek untuk mewujudkan semua tingkatan yang berbeda ini. Jika mau, Anda dapat melakukannya tanpa semua antarmuka ini, kecuali ada alasan spesifik mengapa Anda membutuhkannya. Secara umum, Anda tidak boleh membuat antarmuka seperti itu yang hanya memiliki satu implementasi.



Wadah Injeksi Ketergantungan



Contoh yang dikerjakan dalam artikel ini cukup sederhana: kita berhadapan dengan satu kelas yang hanya memiliki satu ketergantungan. Sekarang anggaplah kita memiliki banyak dependensi, dan masing-masing dikaitkan dengan dependensi lainnya. Bahkan ketika bekerja pada proyek-proyek yang cukup kompleks, kemungkinan Anda harus menghadapi situasi seperti itu, dan tidak hanya perlu diingat bahwa Anda perlu membuat semua kelas ini, tetapi juga ingat yang mana tergantung pada yang mana - terutama jika Anda memutuskan gunakan antarmuka.



Biasakan dengan wadah injeksi ketergantungan. Ini menyederhanakan hidup Anda, tetapi prinsip pengoperasian wadah seperti itu bisa sangat membingungkan, terutama ketika Anda baru mulai menguasainya. Sepintas, kesempatan ini bisa memberikan sedikit keajaiban.

Dalam contoh ini, kita akan menggunakan wadah dari Unity, tetapi ada banyak yang lain untuk dipilih, saya hanya akan menyebutkan yang paling populer: Castle Windsor , Ninject . Dari sudut pandang fungsional, wadah ini praktis tidak berbeda. Perbedaannya mungkin terlihat pada tingkat sintaksis dan gaya, tetapi, pada akhirnya, semuanya tergantung pada preferensi pribadi Anda dan pengalaman pengembangan (serta apa yang ditentukan di perusahaan Anda!).



Mari kita lihat lebih dekat contoh menggunakan Unity: Saya akan mencoba menjelaskan apa yang terjadi di sini.



Hal pertama yang perlu Anda lakukan adalah menambahkan tautan ke Unity. Untungnya, ada paket Nuget untuk ini, jadi klik kanan pada proyek Anda di Visual Studio dan pilih Kelola Paket Nuget:







Temukan dan instal paket Unity, fokus pada proyek Unity Container:







Jadi, kami siap. Ubah metode Main



file Program.cs

Program.cs 



 static void Main(string[] args)
        {
            var number1 = GetNumber("Enter the first number: > ");
            var number2 = GetNumber("Enter the second number: > ");
            var operation = GetOperator();
            //     
            var container = new UnityContainer();
            container.RegisterType<ILogger, NullLogger>();
            var calc = container.Resolve<Calculator>();
            var result = GetResult(calc, number1, number2, operation);
            Console.WriteLine($"{number1} {operation} {number2} = {result}");
            Console.Write("Press any key to continue...");
            Console.ReadKey();
        }
