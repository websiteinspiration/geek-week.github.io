JavaScriptパフォーマンス測定

関数の実行にかかる時間を測定することは、あるメカニズムの実装が別の実装よりも生産的であることを証明する良い方法です。これにより、コードに変更を加えた後、関数のパフォーマンスが低下していないことを確認できます。また、アプリケーションのパフォーマンスのボトルネックを探すのにも役立ちます。

Webプロジェクトに高いパフォーマンスがある場合、これはユーザーによる肯定的な認識に貢献します。そして、ユーザーがリソースでの作業を好んだら、彼らは戻るという性質を持っています。たとえば、このこの調査では、オンライン顧客の88％が、不便に遭遇したリソースに戻る可能性が低いことが示されています。これらの不便は、パフォーマンスの問題が原因である可能性があります。

そのため、パフォーマンスのボトルネックを見つけ、コードに加えられた改善の結果を測定するのに役立つツールがWeb開発で重要です。このようなツールは、特にJavaScript開発に関連しています。 JavaScriptはシングルスレッド言語であるため、JavaScriptコードのすべての行がDOMをブロックする可能性があることを知っておくことが重要です。 この記事では、関数のパフォーマンスを測定する方法と、測定結果をどう処理するかについて説明します。





一部の計算が重すぎてメインスレッドで実行できないと思われる場合は、それらをService WorkerまたはWeb Worker に移動することもできます。

Performance.now（）メソッド


Performance インターフェイスは、メソッドを介してDOMHighResTimeStamp型の値へのアクセスを提供しますperformance.now()。このメソッドは、ドキュメントが存在し始めてから経過した時間をミリ秒単位で示すタイムスタンプを返します。さらに、このインジケーターの精度は約5マイクロ秒（ミリ秒の端数）です。

メソッドを使用してコードフラグメントのパフォーマンスを測定するには、performance.now()2つの時間測定を実行し、これらの測定の結果を変数に保存してから、2番目の測定の結果から最初の結果を差し引く必要があります。

const t0 = performance.now();
for (let i = 0; i < array.length; i++) 
{
  // - 
}
const t1 = performance.now();
console.log(t1 - t0, 'milliseconds');

Chromeでは、このコードを実行すると、次のような結果が得られます。

0.6350000001020817 "milliseconds"

Firefoxでは、次のようになります。

1 milliseconds

ご覧のとおり、異なるブラウザーで得られた測定結果は非常に異なります。実際のところ、Firefox 60では、Performance APIによって返される結果の精度が低下しています。これについては、後で詳しく説明します。

パフォーマンスインターフェイスには、特定のタイムスタンプを返すだけではない、はるかに多くの機能があります。これには、パフォーマンスタイムライン API 、ナビゲーションタイミング、ユーザータイミング、リソースタイミングなど、このインターフェイスの拡張によって表されるパフォーマンスのさまざまな側面の測定が含まれます。これらのAPIについて詳しく知るための資料を以下に示します。

私たちのケースでは、関数のパフォーマンスを測定することについて話しているので、メソッドが与える十分な機会がありますperformance.now()。

Date.now（）およびperformance.now（）


ここでは、メソッドを使用してパフォーマンスを測定できると考えているかもしれませんDate.now()。これは確かに可能ですが、このアプローチには欠点があります。

このメソッドDate.now()は、Unixの時代（1970-01-01T00：00：00Z）から経過した時間をミリ秒単位で返し、システムクロックに依存します。これは、このメソッドがほど正確ではないことを意味するだけでなくperformance.now()、対照的にperformance.now()、特定の条件下で、誤ったクロックインジケーターに基づいている可能性のある値を返します。WebKitエンジンに関連するプログラマーであるRich Gentlekorが言うには、次のようになります。Dateシステム時間に基づいて、実際のアプリケーションを監視するための理想を呼び出すことは絶対に不可能です。ほとんどのシステムには、定期的に時刻を同期するデーモンがあります。システムクロックを15〜20分ごとに数ミリ秒調整することはよくあることです。このような周波数では、10秒間隔の測定の約1％のクロック設定が不正確になります。」

Console.time（）メソッド


このAPIを使用した時間測定は非常に簡単です。十分に、パフォーマンスを評価する必要があるコードの前にメソッドを呼び出し、console.time()このコードの後にメソッドを呼び出しますconsole.timeEnd()。この場合、1つのメソッドと他のメソッドが同じ文字列引数を渡す必要があります。1ページで、最大10,000個のそのようなタイマーを同時に使用できます。

このAPIを使用して行われる時間測定の精度は、Performance APIを使用する場合と同じですが、それぞれの特定の状況で達成される精度は、ブラウザーによって異なります。

console.time('test');
for (let i = 0; i < array.length; i++) {
  // - 
}
console.timeEnd('test');

このようなコードを実行すると、システムは自動的に経過時間に関する情報をコンソールに出力します。

Chromeでは、次のようになります。

test: 0.766845703125ms

Firefoxでは、次のようになります。

test: 2ms - timer ended

実際、ここにあるものはすべて、での作業中に見たものと非常によく似ていperformance.now()ます。

この方法の長所console.time()は、使いやすさです。つまり、そのアプリケーションは補助変数の宣言を必要としないという事実について話し、それらに記録されたインジケーター間の違いを見つけます。

時間精度の低下


上記のツールを使用して、さまざまなブラウザーでコードのパフォーマンスを測定した場合は、測定結果が異なる場合があることに注意してください。

これは、ブラウザーが時間ベースの攻撃やブラウザー識別メカニズム（ブラウザーフィンガープリンティング）からユーザーを保護しようとしているためです。時間測定の結果があまりにも正確であることが判明した場合、これは、たとえばユーザーを特定する機会を攻撃者に与える可能性があります。

Firefox 60では、すでに述べたように、時間測定結果の精度が低下します。これを行うには、プロパティprivacy.reduceTimerPrecision値を2 msに設定します。

functions関数を何度も実行する


配列を反復処理する関数があるとします。彼女は配列の各要素を使用していくつかの計算を実行し、その後、計算の結果を含む新しい配列を返します。この関数の最適化について考えると、状況に応じて何がより速く機能するかを知りたいと思います-ループforEachまたは通常のループfor。

この機能には2つのオプションがあります。

function testForEach(x) {
  console.time('test-forEach');
  const res = [];
  x.forEach((value, index) => {
    res.push(value / 1.2 * 0.1);
  });

  console.timeEnd('test-forEach')
  return res;
}

function testFor(x) {
  console.time('test-for');
  const res = [];
  for (let i = 0; i < x.length; i ++) {
    res.push(x[i] / 1.2 * 0.1);
  }

  console.timeEnd('test-for')
  return res;
}

関数をテストします。

const x = new Array(100000).fill(Math.random());
testForEach(x);
testFor(x);

コードを実行すると、次の結果が得られます。

test-forEach: 27ms - timer ended
test-for: 3ms - timer ended

サイクルはサイクルforEachよりもはるかに遅いように見えたfor。結局、テスト結果はこれを正確に示していますか？

実際、1回のテストでは、そのような結論を出すのは時期尚早です。関数を2回呼び出してみましょう。

testForEach(x);
testForEach(x);
testFor(x);
testFor(x);

以下を取得します。

test-forEach: 13ms - timer ended
test-forEach: 2ms - timer ended
test-for: 1ms - timer ended
test-for: 3ms - timer ended

それが使用される関数はforEach、2回目と呼ばれ、使用される関数と同じくらい高速であることがわかりforます。しかし、最初のforEach関数が呼び出されるという事実を考えると、関数の動作ははるかに遅く、それを使用する価値はまだないかもしれません。

この場合、FirefoxはforEach同様の入力により最適化されています。また、サイクルforはforEachChromeとFirefoxの両方よりも高速です。結果として、関数cのバリアントを詳しく検討する方が良いでしょうfor。

これは良い例であり、さまざまなブラウザでパフォーマンスを測定することの重要性を示しています。Chromeでのみ一部のコードのパフォーマンスを評価するforEachと、サイクルと比較して、サイクルforはそれほど悪くないという結論に達することができます。
