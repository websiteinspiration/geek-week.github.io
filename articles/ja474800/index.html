ToListとToArrayとサイクル


.NET Core 3.0のリリースでこの情報を準備する予定でしたが、彼らは私の前に出ました。他人の名声を盗んだり、他人の情報をコピーしたりしたくないので、比較が詳しく説明されている優れた記事を紹介します。

私自身、測定値と結果をお伝えしたいと思います。ループを作成する「C ++スタイル」の愛好家のために、それらに逆ループを追加しました。

コード：
public class Bench
    {
        private List<int> _list;
        private int[] _array;

        [Params(100000, 10000000)] public int N;

        [GlobalSetup]
        public void Setup()
        {
            const int MIN = 1;
            const int MAX = 10;
            Random random = new Random();
            _list = Enumerable.Repeat(0, N).Select(i => random.Next(MIN, MAX)).ToList();
            _array = _list.ToArray();
        }

        [Benchmark]
        public int ForList()
        {
            int total = 0;
            for (int i = 0; i < _list.Count; i++)
            {
                total += _list[i];
            }

            return total;
        }
        
        [Benchmark]
        public int ForListFromEnd()
        {
            int total = 0;t
            for (int i = _list.Count-1; i > 0; i--)
            {
                total += _list[i];
            }

            return total;
        }

        [Benchmark]
        public int ForeachList()
        {
            int total = 0;
            foreach (int i in _list)
            {
                total += i;
            }

            return total;
        }

        [Benchmark]
        public int ForeachArray()
        {
            int total = 0;
            foreach (int i in _array)
            {
                total += i;
            }

            return total;
        }

        [Benchmark]
        public int ForArray()
        {
            int total = 0;
            for (int i = 0; i < _array.Length; i++)
            {
                total += _array[i];
            }

            return total;
        }
        
        [Benchmark]
        public int ForArrayFromEnd()
        {
            int total = 0;
            for (int i = _array.Length-1; i > 0; i--)
            {
                total += _array[i];
            }

            return total;
        }
    }

スロー


1年前、私はレガシープロジェクトの会社で働いていました。そのプロジェクトでは、try-catch-throwコンストラクトを通じてフィールド検証を処理するのが普通でした。これはプロジェクトにとって不健全なビジネスロジックであることをすでに理解しているので、可能であればそのようなデザインを使用しないようにしました。しかし、そのような設計でエラーを処理するための悪いアプローチは何かを見てみましょう。2つのアプローチを比較し、各オプションの「ベンチ」を撮影するために、小さなコードを書きました。

コード：
        public bool ContainsHash()
        {
            bool result = false;
            foreach (var file in _files)
            {
                var extension = Path.GetExtension(file);
                if (_hash.Contains(extension))
                    result = true;
            }

            return result;
        }

        public bool ContainsHashTryCatch()
        {
            bool result = false;
            try
            {
                foreach (var file in _files)
                {
                    var extension = Path.GetExtension(file);
                    if (_hash.Contains(extension))
                        result = true;
                }
                
                if(!result) 
                    throw new Exception("false");
            }
            catch (Exception e)
            {
                result = false;
            }

            return result;
        }


.NET Core 3.0とCore 2.2の結果は同じような結果になります（.NET Core 3.0）：





キャッチを試行すると、コードの理解が複雑になり、プログラムの実行時間が長くなります。ただし、この構成が必要な場合は、エラー処理が予期されないコード行を挿入しないでください。これにより、コードの理解が容易になります。実際、システムをロードするのは、throw new Exceptionコンストラクトを介してエラー自体をスローするほどの例外処理ではありません。

例外のスローは、目的の形式でエラーを収集するクラスよりも低速です。フォームまたは一部のデータを処理していて、エラーがどうあるべきかが明確にわかっている場合は、なぜそれを処理しないのですか？

この状況が例外的でない場合は、新しいException（）をスローしないでください。例外の処理とスローは非常に高価です!!!

層のコンパイル


前回の記事でこの機能について簡単に説明しましたが、私の言葉を訂正して補足したいと思います。マルチレベルコンパイルはソリューションの起動時間を短縮しますが、コードの一部を犠牲にしてバックグラウンドでより最適化されたバージョンにコンパイルするため、わずかなオーバーヘッドにつながる可能性があります。 NET Core 3.0の登場により、層のコンパイルを有効にしたプロジェクトのビルド時間が短縮され、このテクノロジーに関連するバグが修正されました。以前は、このテクノロジーにより、ASP.NET Coreの最初のリクエストでエラーが発生し、マルチレベルコンパイルモードでの最初のビルド中にフリーズしました。現在、.NET Core 3.0ではデフォルトで有効になっていますが、必要に応じて無効にすることができます。チームリーダー、シニア、ミドル、または部門長である場合、プロジェクトの迅速な開発がチームの価値を高め、このテクノロジーにより開発者の時間とプロジェクトの時間の両方を節約できることを理解する必要があります。

.NETレベルアップ


.NET Framework / .NET Coreをアップグレードします。多くの場合、新しいバージョンごとにパフォーマンスが向上し、新しい機能が追加されます。

しかし、具体的にどのようなメリットがありますか？それらのいくつかを見てみましょう：

  • .NET Core 3.0では、.NET Coreアプリケーションの起動時間を短縮するR2Rイメージが導入されています。
  • バージョン2.2では、Tier Compilationが導入されました。これにより、プログラマーはプロジェクトの起動にかかる時間を短縮できます。
  • 新しい標準のサポート。NET標準。
  • 新しいバージョンのプログラミング言語のサポート。
  • 最適化は、新しいバージョンごとに、コレクション/構造体/ストリーム/文字列/正規表現などのベースライブラリによって改善されています。.NET Frameworkから.NET Coreにアップグレードすると、すぐにパフォーマンスが大幅に向上します。たとえば、.NET Core 3.0で追加された最適化の一部にリンクを添付します。https：//devblogs.microsoft.com/dotnet/performance-improvements-in-net-core-3-0/
