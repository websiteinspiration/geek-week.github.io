12 dicas para implementar o TypeScript em aplicativos React

TypeScript é o padrão do desenvolvimento moderno de front-end. De acordo com a pesquisa State of JavaScript , o TS é claramente de interesse dos programadores. De acordo com a pesquisa de 2019, quase 60% dos entrevistados experimentaram TS e continuam a usá-lo, 22% não tentaram e querem estudar.

Este artigo é uma coleção de dicas sobre como implementar e melhorar o uso do TypeScript. A primeira metade das dicas é geral sobre abordagens e infraestrutura. O segundo são alguns recursos de linguagem particularmente úteis.



Fico com o contexto de um aplicativo React, no entanto, algumas dicas serão relevantes em outros ecossistemas.

Dica 1: tolerância a erros


Os tipos corretos são confiáveis ​​no código escrito e facilitam a refatoração. Para que a digitação funcione para você, você deve aplicar estritamente às verificações de tipo:

  1. Verifique o modo dev para ver os erros imediatamente.
  2. Adicione validação ao IC para cada solicitação de pool. Isso garantirá que o ramo principal seja sempre do tipo correto.

Existem duas maneiras principais de compilar arquivos ts: o TSC original e o @ babel / preset-typescript. O método babel é interessante: as informações de tipo são cortadas dos arquivos .ts, o que torna a montagem rápida. Aqui está um artigo de introdução.

A verificação de tipo no caso de babel é adicionada adicionalmente usando o fork-ts-checker-webpack-plugin. No modo de desenvolvimento, as verificações devem ser feitas de forma assíncrona para não bloquear o assembly e não interferir na escrita do código. Nas solicitações de pool, a verificação de tipo, pelo contrário, deve interromper a construção se um erro for encontrado.

Dica 2: contratos de back-end


A comunicação com o back-end por meio da própria API implica em um contrato de troca de dados. É útil separar essas entidades em uma camada separada, criando interfaces separadas. Nós os chamamos de DTO - Data Transfer Object.

Para implementar essa camada, são necessários alguns acordos. Aqui está um exemplo das regras que você pode implementar em uma equipe:

  • O nome da interface possui o sufixo Dto: FooBarDto.
  • Localizado em uma pasta especial.
  • Só pode mudar com uma alteração de API.
  • Os dados desta interface são usados ​​apenas para leitura, a edição é proibida.

Vantagens dessa abordagem: a fonte de dados é imediatamente clara na interface, é mais fácil procurar por erros e código de depuração, verificar dados. Esses tipos são fáceis de documentar, fornecendo descrições e exemplos - colegas desenvolvedores o levarão na mão.

Se você usar ferramentas como o Swagger para descrever a API , veja como criar esses tipos automaticamente.

Dica 3: aliases em qualquer


Você e sua equipe escrevem códigos de tipo soberbo - sem problemas, sem erros e na hora certa.

Nesse caso, vá para o próximo ponto;) Na realidade, há tentações de omitir a digitação, especialmente ao implementar o TS em um projeto existente. Eu quero usar de forma explícita ou implícita. Existem razões adequadas: código complexo, tarefa urgente, falta de experiência - os esforços não pagam pelo resultado.

Em vez de ignorar o problema, tente legalizá-lo:

export type TODO_ANY = any;

Prós:

  • Você não trapaceia e não escreve muletas.
  • Você mostra conscientemente que, no futuro, vale a pena finalizar o código.
  • É fácil encontrar esse tipo pesquisando um projeto.
  • Qualquer um pode ser usado explicitamente onde o tipo não importa, por exemplo:

const getArrayLength = (array: any[]) => array.length

Limitações:

  • O uso muito frequente dessa técnica provavelmente não será benéfico.
  • Consciência e responsabilidade são necessárias para remover essas gags no futuro.

Você também pode achar útil: 

export type TODO_UNKNOWN = unknown;

No caso de qualquer, o compilador permitirá quase qualquer ação. No caso de desconhecido , pelo contrário, quase qualquer uso levará a um erro. Eles definem TODO_UNKNOWN até que o tipo do objeto fique claro → no futuro, eles usariam o objeto → obtiveram um erro → corrigiram o tipo → você é incrível.

Dica 4: tipos para bibliotecas


Se você estiver trabalhando com a popular biblioteca js, procure por tipos em DefinitelyTyped. Provavelmente, eles já estão escritos - nos conectamos e desfrutamos a vida.

Mas e se você tiver uma biblioteca interna na qual definitivamente não há tipos? Tente gerá-los automaticamente. Boas notícias para quem usa jsDoc: no TypeScript 3.7, o compilador tsc aprendeu a gerar tipos .d.ts a partir de arquivos .ts e .js. 

Outra boa notícia: existem conversores js -> ts. Por exemplo, o TypeStat me ajudou a lidar com componentes típicos do React.

A má notícia é que não é provável que um sistema de bom tipo funcione automaticamente. Conte com o "esqueleto", que posteriormente precisará construir a "carne".

Dica 5: tipos em cima de tipos


... e tipos perseguem!

Ocorre que os tipos de dependências externas do projeto não são adequados - eles são muito gerais (e você gostaria de ser mais específico), contêm um erro, a versão não se encaixa - ou os tipos simplesmente estão ausentes.

Normalmente, esses problemas surgem nas bibliotecas internas (que também são atualizadas com freqüência), enquanto nas bibliotecas externas é bastante raro. Se ocorrer um problema, tente definir / redefinir manualmente os tipos.

Nós descrevemos o tipo para o módulo de exemplo. Enfatizo que apenas o tipo é redefinido; a implementação não muda:

declare module "example" {
    const foo: { bar: string };
    export default foo;
}

É melhor mover essa substituição para um diretório e arquivo separados, por exemplo, typings / example.d.ts . Conecte todo o conteúdo das tipagens ao campo compilerOptions.typeRoots no tsconfig:

{
    "compilerOptions": {
        "typeRoots": ["node_modules/@types", "./typings"]
    }
}

Voila - tipos, por exemplo, são retirados da redefinição!

Dica 6: guia de estilo


Tipo ou interface? const ou enum? Como descrever tipos em um componente funcional? E no memoized?

Quando o TS está sendo introduzido em uma equipe, essas questões surgem constantemente. Um conjunto semelhante de perguntas aparece para iniciantes no projeto: a tempestade dura várias semanas e o estilo de codificação usual é resolvido.

A formalização de regras é uma ótima maneira de economizar tempo em disputas e holivares em solicitações de pool. Reúna uma equipe e crie seu guia de estilo.

  • Tome guias de estilo populares como base. Eu gosto destes: react-typescript-cheatsheet , react-redux-typescript-guide , modern-typescript-with-examples-cheat-sheet.
  • Linter e formatador de trabalho - deixe a máquina fazer o trabalho sujo.
  • Adicione as especificidades do seu projeto: modelos típicos, práticas recomendadas, práticas recomendadas.
  • Confirme essas decisões e consulte-as em uma revisão de código.
  • Ofereça à equipe a oportunidade de discutir e alterar essas decisões.

Um guia de estilo bem construído revela as maravilhas do CPDD ( Copy Paste Driven Development ): os desenvolvedores escrevem no mesmo estilo, gastam menos tempo graças aos clichês prontos. Guias de batalhas épicas x espaços (desculpe, tipos x interfaces) não são organizadas , a entrada de iniciantes no projeto é simplificada.

PS: Outra fonte de inspiração é a biblioteca de utilitários ts. Você encontrará não apenas auxiliares úteis, mas também o código dos profissionais na fonte. Por exemplo, tipos de utilitário , SimplyTyped .

Dica 8: tipo literal vs enum


O TypeScript apresenta uma estrutura de sintaxe conveniente - enum . Enum é usado para enumerações finais. Tendo dominado o enum, a princípio quero escrevê-lo com muita frequência. No entanto, o enum tem críticas justas em relação ao código gerado após a compilação.

Por outro lado, existem tipos literais - uma alternativa mais simples e conveniente para a enumeração final. Em alguns casos, o uso desses tipos é mais apropriado.

Um exemplo de vida é adereços de componentes simples:

//   enum
enum ColorEnum {
    yellow = "yellow",
    transparent = "transparent"
}

const ExampleEnum = (props: { color: ColorEnum }) => <div>{props.color}</div>;

//    
type ColorLiteral = "yellow" | "transparent";

const ExampleLiteral = (props: { color: ColorLiteral }) => <div>{props.color}</div>;

//   
const Example = () => (<>
        <ExampleEnum color={ColorEnum.yellow} />
        <ExampleLiteral color="yellow" />
 </>);

Neste exemplo, os dois componentes verificam o tipo de cor dos acessórios. A variante com enum acabou sendo mais redundante - este é um código adicional durante a compilação, além de importações adicionais no código-fonte e uma estrutura de escrita mais complicada. A variante com um literal de cadeia é mais concisa, além de conter uma verificação de tipo. Não faz sentido complicar o código da enumeração.

Prefira literais de sequência para verificações simples de tipo. Use enum para descrever uma lógica mais complexa. Nesse caso, é melhor dar preferência ao const enum (no entanto, essa estrutura não é suportada pelo babel , você precisará adicionar um plugin ).

Dica 9: afirmações const


Outro motivo para usar enum com menos frequência são as afirmações constantes.

como const informa ao compilador que esse objeto não será alterado. Portanto, os tipos podem ser deduzidos com mais detalhes.

const loadAction = {
    start: "LOAD_START",
    done: "LOADED",
    error: "LOAD_ERROR"
} as const;

type LoadActionType = typeof loadAction;
type LoadActionKey = keyof LoadActionType; // 'start' | 'done' | 'error'
type LoadActionValue = LoadActionType[LoadActionKey]; // 'LOAD_START' | 'LOADED' | 'LOAD_ERROR'
//  "as const" LoadActionValue   string

Você também pode aplicar como const às matrizes:

const DIRECTION_ASC = 1;
const DIRECTION_DESC = -1;
const sortDirectionTypes = [DIRECTION_ASC, DIRECTION_DESC] as const;
type SortDirection = typeof sortDirectionTypes[number]; // -> 1 | -1

É conveniente usar para transferências. Por exemplo, verifique se o dicionário descreve todas as opções:

const sortDirectionDescription: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " ",
    [DIRECTION_DESC]: " "
};
const sortDirectionDescription2: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " "
};
// Error: TS2741: Property '[-1]' is missing // in type '{ [DIRECTION_ASC]: string; }'

PS: Você pode ler mais sobre afirmações constantes neste artigo.

Dica 10: tipos de marca


A marca é um pequeno utilitário para a criação de tipos exclusivos ("com marca"). Normalmente, esses tipos são criados com base em primitivas - seqüências de caracteres, números.

Por exemplo, ao trabalhar com o ID de dados, é constantemente usado. Às vezes, é extremamente importante distinguir um tipo de ID de outro (por exemplo, o ID do usuário do ID de seu pedido na loja), embora ambos sejam strings. Com a Brand, você pode obter apenas dois tipos diferentes:

type Brand<T, U> = T & { __brand: U };

type UserId = Brand<string, "userId">; // "5435436" –  id 
type OrderId = Brand<string, "orderId">; // "42da-fet3-cdsl" –  id 

declare const userID = "1111" as UserId; // - 
declare function getOrderById(id: OrderId): any; // - 

getOrderById(userID);
// Error: TS2345
// Type '"userId"' is not assignable to type '"orderId"'.

Dica 11: tipos de união discriminados


O termo difícil união discriminada é fácil de entender na prática: cria uma união de vários tipos com uma parte comum (discriminante). Ao trabalhar com essa união, o compilador ts pode filtrar as opções e especificar o tipo.

O exemplo clássico de usar união discriminada se encaixa muito bem no modelo de redutor. Graças ao campo de tipo comum, o tipo de carga útil é exibido automaticamente:

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {
        case 'SET_ONE':
            return { ...state, one: action.payload}; // `payload` is string
        case 'SET_TWO':
            return { ...state, two: action.payload}; // `payload` is number
        default:
            return state;
    }
}

Dica 12: nunca


Nunca é um bom seguro contra casos brutos. Retire do parágrafo anterior, apenas sem 'SET_TWO' e com a adição de typeguard: never;

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {

        case 'SET_ONE':
            return { ...state,
                        one: action.payload}; // `payload` is string

        default:
            const typeguard: never = action;
            return state;
            // Error: TS2322: Type '{ type: "SET_TWO"; payload: number; }'
            // is not assignable to type 'never'.
    }
}

O compilador deduz o tipo de proteção tipográfica e chega a uma contradição. Por um lado - o tipo de que não é, nunca. Por outro lado, o tipo de saída é 'SET_TWO'. Tipos não convergem, obtemos um erro.

Outro caso interessante (mas raro) de usar nunca no React é proibir aninhamento ou aprovação de adereços:

type IconProps = {
    size: 's' | 'm';
    children?: never;
    foo?: never;
};

const Icon: React.FC<IconProps> = () => <div>Whatewer</div>;
    
const Example = () => (<>
    <Icon size="s"> <div /> </Icon> {/*  –  children */}
    <Icon size="s" foo="123"/> {/*  -  foo */} 
</>);

Conclusão


O uso do TypeScript em um projeto altera a maneira como você escreve código. Torna-se mais fácil estudar o código, realizar refatoração. Torna-se conveniente usar contratos no código: você não precisa ter em mente os detalhes da implementação das funções usadas e a forma dos objetos - deixe que as interfaces o descrevam. 

O projeto recebe um aumento de nível e, junto com novas vantagens, novas tarefas - suportando o sistema de tipos correto. Espero que essas dicas ajudem você a melhorar seu sistema.
