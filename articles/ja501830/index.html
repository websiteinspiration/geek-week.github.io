AngularプロジェクトをESLintに移植し、Prettier、Husky、lint-staged

こんにちは、ハブル！私の名前はボグダンです。PICDigitalフロントエンドチームリードで働いています。私たちはほとんどのプロジェクトをAngularで開発しており、最近、スタイルガイドを確認し、より便利な作業のための新しいツールを追加することにしました。

リンターとして、私はESLintを使用することに決めました。彼らはすぐにAngularをそこに転送する予定だからです。そして、この記事では、TSLintからESLintへの切り替えに関する説明を共有すると同時に、ESLintからPrettierを実行する方法、AirBnBスタイルガイドルールを追加する方法、およびVSコードとGitフックを設定してリントを便利で非表示にする方法を説明します。

きれいな＆ESLint


ESLintは静的コード分析のためのツールであり、ルールは2つのグループに分けられます。

  • 書式設定 -コードを1つのビューに表示します：行の長さ、コンマ、セミコロンなど。
  • コード品質 -問題のあるコードテンプレートの検索：不要なコード、エラー。

Prettierは自動コードフォーマットツールです。

私が興味を持った質問は、ESLintがコードをフォーマットできるのになぜPrettierを使うのかということでした。

答えは簡単です。Prettierはコードをより適切にフォーマットします。すべてのフォーマットを削除し、単一のスタイルでコードを完全に書き換えます。これにより、開発者はコードのフォーマットを忘れることができ、コードスタイルを検討するために時間を無駄にする必要がなくなります。たとえば、長いコード行があります



。ESLintを使用してフォーマットを変更しようとすると、エラーが発生します。

eslint example.ts --fix

output:
error    This line has a length of 97. Maximum allowed is 80

このような例は、リンターが常にコードのフォーマットに役立つとは限らないことを示しています。開発者は、個人的な考慮事項に基づいて、さまざまな方法でこのコードをフォーマットできます。

Prettierでファイルを保存またはフォーマットすると、行は次の形式になります



。Prettierはコードベース全体で統一されたスタイルを提供します。したがって、ESLintと一緒に使用できますが、相互に干渉しないように構成する必要があります。

ESLintセットアップ


ESLintを使用したリンティングの本質は、コードをAST（抽象構文ツリー）に変換するパーサーと、ルール（TypeScriptリンティングの推奨ルールやAirBnBコードガイドルールなど）を含むプラグインにあります。

取り付け


AngularアプリケーションをESLintに移行するには、次の依存関係が必要です。

  • eslint-リンター自体、
  • @ angular-eslint /  builder-標準のng lintコマンドを使用してAngularアプリケーションのESLintを実行するAngular CLI Builder
  • @ angular-eslint / eslint-plugin-Angular lintingの  ルールを含むプラグイン、
  • @ angular-eslint / template-parser  は、角度のある/コンパイラは、Angularテンプレートを裏打ちするためのルールを書くことを可能にします、
  • @ angular-eslint / eslint-plugin-template-  @ angular-eslint / template-parserと一緒に使用すると、Angularテンプレートを言語化するためのルールを起動するプラグイン、
  • @ typescript-eslint / parser -TypeScriptコードを解析するためのプラグイン、
  • @ typescript-eslint / eslint-plugin-リンティング用のTypeScriptルールを実行するプラグイン。

それらをインストールするには、次のコマンドを実行するだけです：

ng add @angular-eslint/schematics

執筆時点では、typescript-eslintとangular-eslintには、TSLintの標準的なCodelyzer構成のルールに相当するものはすべてありませんが、それらのほとんどは既に存在しています。Angular ESLintおよびTypeScript ESLintの単一リポジトリで、TSLintからESLintへのルール転送の現在のステータスを監視できます。

構成構成


Angularアプリケーションをリントするために必要なすべてのものをインストールしました。それでは、ESLintの構成に移りましょう。.eslintrc.jsファイルを作成して、Angular ESLintの推奨設定を追加しましょう。

module.exports = {
  extends: ['plugin:@angular-eslint/recommended'],
  rules: {
    '@angular-eslint/directive-selector': [
      'error',
      { type: 'attribute', prefix: 'app', style: 'camelCase' },
    ],
    '@angular-eslint/component-selector': [
      'error',
      { type: 'element', prefix: 'app', style: 'kebab-case' },
    ],
  },
  overrides: [
    //   ,       *.component.ts
    {
      files: ['*.component.ts'],
      parser: '@typescript-eslint/parser',
      parserOptions: {
        ecmaVersion: 2020,
        sourceType: 'module',
      },
      plugins: ['@angular-eslint/template'],
      processor: '@angular-eslint/template/extract-inline-html',
    },
  ],
};

構成は、JavaScript、JSON、またはYAMLファイルなど、さまざまな形式で記述できます。JavaScriptでコメントを残すことができます。

「プラグイン：@ angular-eslint / Recommended」には、3つのプラグインの設定が一度に含まれています。「@ typescript-eslint / eslint-plugin」、「@ angular-eslint / eslint-plugin」、「@ angular-eslint / eslint-plugin-template 」ここで彼が設定したルールを読むことができます。

ng lintコマンドの更新


また、angular.json構成では、ng lintコマンドを更新して@ angular-eslint / builderを実行する必要があります。

"lint": {
  "builder": "@angular-eslint/builder:lint",
  "options": {
    "eslintConfig": ".eslintrc.js",
    "tsConfig": [
      "tsconfig.app.json",
      "tsconfig.spec.json",
      "e2e/tsconfig.json"
    ],
    "exclude": [
      "**/node_modules/**"
    ]
  }
},

基本的なESLintセットアップの準備が整いました。これで、標準のng lintコマンドでESLintを開始できます。

追加のプラグインをインストールする


ESLintのプラグインをインストールするには、たとえばAngularでユニットテストをリンティングするために、Jasmineプラグインをダウンロードして設定に追加する必要があります。

npm install eslint-plugin-jasmine --save-dev

また、* .spec.ts拡張子を持つファイルの新しい設定ブロックを「overrides」プロパティに追加します。

overrides: [
  ...,
  {
    files: ['src/**/*.spec.ts', 'src/**/*.d.ts'],
    parserOptions: {
      project: './src/tsconfig.spec.json',
    },
    //   
    extends: ['plugin:jasmine/recommended'],
    //    
    plugins: ['jasmine'],
    env: { jasmine: true },
    //   'no-unused-vars'
    rules: {
      '@typescript-eslint/no-unused-vars': 'off',
    },
  }
],

類推すると、さまざまなファイル拡張子のために他のプラグインを追加できます。

スタイルガイドガイドの追加


コードベースの一貫性を高めるために、人気のあるスタイルガイドのいずれかのルールを選択してESLint構成に追加できます。

  • AirBnB：3つの必須の末尾のコンマとセミコロンの中で最も人気があり、最も厳密です。
  • Google：フォーマットの点ではAirBnBに似ていますが、それほど厳密ではない必須のJSDocコメントです。
  • StandartJS：末尾のコンマとセミコロンの使用を禁止します。

チームに適したスタイルガイドを選択してください。大規模なプロジェクトですべてのスタイルガイドを順番に試して、リンターが生成するエラーを確認し、これに基づいて選択することができます。

TypeScriptではJavaScriptルールが正しく機能しない可能性があるため、TypeScriptスタイルガイドの実装を選択してください。

例として、AirBnBスタイルガイドをESLint構成に追加してみましょう。これを行うには、TypeScriptのAirBnBルールを使用して構成をインストールし、インポート/エクスポート構文を操作するためのルールを使用してプラグインをインストールします。

npm install eslint-plugin-import eslint-config-airbnb-typescript --save-dev

トップレベルの設定を変更しないために、「overrides」プロパティにAirBnBスタイルガイドルールと作業に必要なTypeScriptパーサーを含む新しいルールブロックを作成します。

module.exports = {
  ...,
  overrides: [
    ...,
    {
      files: ['*.ts'],
      extends: [
        'airbnb-typescript/base',
      ],
      parser: '@typescript-eslint/parser',
      parserOptions: {
        ecmaVersion: 2020,
        sourceType: 'module',
      },
    },
  ]
}

別のスタイルガイドを追加するには、TypeScriptの一連のルールをインストールし、スタイルガイドのルールで「オーバーライド」に新しいルールのブロックを作成し、作業に必要なパーサーを指定する必要があります。

カスタマイズルール


ガイドのスタイルでいくつかのルールを無効化または再定義する場合は、 "rules"プロパティでこれを行うことができます。

module.exports = {
  ...,
  overrides: [
    ...,
    {
      files: ['*.ts'],
      extends: [
        'airbnb-typescript/base',
      ],
      parser: '@typescript-eslint/parser',
      parserOptions: {
        ecmaVersion: 2020,
        sourceType: 'module',
      },
      //  
      rules: {
        'import/no-unresolved': 'off',
        'import/prefer-default-export': 'off',
        'class-methods-use-this': 'off',
        'lines-between-class-members': 'off',
        '@typescript-eslint/unbound-method': [
          'error',
          {
            ignoreStatic: true,
          },
        ],
      },
    },
  ]
}


Prettierの構成


Prettierを構成に追加するには、Prettier自体をインストールする必要があります。これは、Prettierルールを備えたプラグイン、およびPrettierと競合する可能性のあるすべてのルールを無効にする構成です。

npm i prettier eslint-config-prettier eslint-plugin-prettier --save-dev

一番下のプロパティ "extends"に拡張子* .tsが付いたファイルのルールを含むブロックの "overrides"に、ルールとPrettier設定を追加します。

module.exports = {
  ...,
  overrides: [
    ...,
    {
      files: ['*.ts'],
      extends: [
        //   AirBnB
	'airbnb-typescript/base',
	//   Prettier
	'prettier/@typescript-eslint',
	'plugin:prettier/recommended',
      ],
      ...,
    },
  ]
}

Prettierと競合する可能性のあるルールを上書きするには、Prettierの構成を常にリストの一番下に置く必要があります。

`prettier / @ typescript-eslint`は、` @ typescript-eslint`ルールを無効にします。これはPrettierと競合する可能性があり、 `plugin：prettier / Recommended`は3つのことを行います：

  • eslint-plugin-prettier、
  • prettier / prettierルールのエラーを「エラー」としてコンソールに出力します。
  • Prettier eslint-config-prettierフォーマットルールを追加します。

Prettierの構成：


Prettierは設定なしでコードをフォーマットできますが、AirBnBスタイルガイドと一致させるには、いくつかの設定を追加する必要があります。アプリケーションのルートに.prettierrc.jsファイルを作成します。

module.exports = {
  trailingComma: "all",
  tabWidth: 2,
  semi: true,
  singleQuote: true,
  bracketSpacing: true,
  printWidth: 100
};

これらの設定は、VS Codeでファイルをフォーマットするために、または次のコマンドで使用する場合、ESLintとPrettierの両方で使用されます。

prettier "--write ."

VSコードの構成


VS Codeは、エラーを保存するときに、ESLintで検出されたエラーを強調表示して修正できます。これを行うには、VS Code用のESLintプラグインをダウンロードして、プロジェクト内にワークスペース.vscode / settings.jsonの設定を含むファイルを作成します。

  "eslint.validate": [ "javascript", "typescript", "html"],

  "eslint.options": {
    "extensions": [".js", ".ts", "html"]
  },

  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true,
  },

ここでは、拡張子が.js、.ts、.htmlのファイルを保存するときにエラーを強調して修正するようにESLintを構成します。

また、「shift + option + F」または「shift + alt + F」キーの組み合わせを使用してドキュメントをフォーマットするには、VS Code用のPrettierプラグインをダウンロードして、デフォルトのフォーマッターとして設定します。

gitフックの設定


Gitフックは、Gitが特定のイベント（コミット、プッシュ、受信）で呼び出すスクリプトです。

それらの助けを借りて、コミットを作成するときにコードをリンティングし始めて、リクエストプールに入るエラーを減らすことができます。Gitフックを使用してより便利に作業するには、Huskyをインストールし、コミットに追加されたコードのみをチェックします（これは、リンティングに時間がかかる大規模なプロジェクトで役立ちます）lint-staged：

npm i husky lint-staged --save-dev

これらのプラグインの設定をpackage.jsonに追加します。

"scripts": {
  ...
},
"husky": {
  "hooks": {
    "pre-commit": "lint-staged --relative"
  }
},
"lint-staged": {
  "*.{js,ts}": [
     "eslint --fix"
  ]
},

lint-stagedは、変更されたファイルの配列を呼び出されたコマンドに渡します。ng lintコマンドはファイルの配列を受け入れる方法を認識していないため、それを使用するには、追加のスクリプトハンドラーを作成する必要があります。または、この例のように、ESLintを呼び出すだけでもかまいません。このようなソリューションはプリコミットに使用でき、ng lintを実行して、たとえばCIパイプラインでプロジェクト全体をlintできます。

調査結果


Angularの将来のバージョンでは、基本的なルールを備えたESLintはそのまま使用できます。これで、ESLint構成プロセスにはいくつかの追加の手順が必要になり、ESLintにはTSLintの一部のルールに相当するものがなく、Angular ESLintはまだアルファ版です。したがって、今すぐESLintに切り替えるかどうかはあなた次第です。

ただし、ガイドコード、追加のルール、Prettier、Husky、lint-stagedは、自分で構成する必要があります。この記事が、これらすべてがどのように連携するかを理解するのに役立つことを願っています。

リンターの設定は簡単な作業のように思えるかもしれませんが、スタイルガイドの選択、さまざまなソリューションの相互同期など、いくつかの重要な組織上の問題が含まれています。

しかし、将来リンターを構成するために費やされる時間は、コードレビュープロセスでコードのスタイルとフォーマットを議論する時間を大幅に節約し、要求プールに分類されるエラーの数を減らし、コードベースの一貫性を保証します。

実装例はGithubにあります。

設定に誤りを見つけた場合、またはアドオンがある場合-書いてください！
