Quarkus-クラウドネイティブJavaの新しい外観

こんにちは、ハブル！

来年は、コンテナテーマであるCloud-Native JavaとKubernetesを本格的に開発する予定です。ロシア語でのこれらのトピックの論理的な続きは、Habréに関する優れた記事ですでに検討されているQuarkusフレームワークに関するストーリーになります。本日の記事では、Quarkusがエンタープライズにもたらす展望として、「サブアトミック超高速Javaデバイス」に焦点を当てません。

JavaとJVMは依然として非常に人気がありますが、サーバーレステクノロジーとクラウド指向のマイクロサービスを使用する場合、Javaとその他のJVM向けの言語は、メモリ領域を取りすぎて読み込みが遅くなり、Javaやその他の言語の使用に適さなくなります。寿命の短いコンテナで使用します。幸いなことに、この状況は現在、Quarkusのおかげで変わり始めています。

前書き


DevOps、コンテナー、サーバーレステクノロジーを実行すればするほど、軽量化されたコンテナーにコンテナー化されたコードを記述したり、PythonまたはJavaScriptでFaaSを 記述したりするようになります。 Javaは単純に起動するにはサーバーレスフレームワークで使用するには重すぎます。マイクロサービスに関しては、JavaScriptまたはPythonはより高速なロードとよりコンパクトなコンテナーを提供し、Javaをより効率的にします。



PythonとJavaScriptは、20年以上にわたってクラウド指向のマイクロサービス

Java言語を作成するのに最適な言語です、そしてその誕生の時、世界は今とは完全に異なっていました。 JVMの登場により、大きな問題が解決されました。コードを1回記述して、多くのプラットフォームとオペレーティングシステムで実行することができました。コンテナを使用すると、アプリケーション、ライブラリ、およびオペレーティングシステムのリソースを別々のコンテナにパックでき、そのような各コンテナはどこでも機能します。JVMが提供する移植性は、関連性が低くなりました。かつては、移植性を確保するために追加の費用を負担する準備ができていましたが、今ではこれらの時代が過ぎました。ここで、遅延を最小限に抑えて迅速に作業し、いつでも利用できる事後対応​​型アプリケーションが必要です。コンテナーおよびKubernetesなどのコンテナーオーケストレーションツール、プログラミング言語に関係なくそのような機能を提供します。

マイクロサービスアーキテクチャの使用に移行している企業は、Javaで記述されたSpringベースのサービスを利用して、それらを重いjarアーカイブにリンクし、JDKを追加して、Linuxを実行するコンテナーで実行します。このソリューションは機能しますが、サイズが500MBの重いコンテナを処理する必要があります。これらのコンテナは、それぞれ10〜30秒間アクセシビリティ状態になります。これは深刻な問題です。移行後、多くの企業はゆっくりとPythonの使用に切り替え、サーバー側のサービスをJavaのままにし、最終的にはFaaSに立ち止まります。

サーバーレステクノロジーとFaaSは、インフラストラクチャを気にすることなく関数の記述に集中できるため、今日非常に人気があります。いずれにせよ、それらはすべてコンテナーで動作しますが、クラウドプロバイダーはライフサイクルを管理します。最良の部分は、一定の時間が経過すると、プロバイダーはコンテナーを完全に停止し、次の呼び出しの後にのみ作業を再開することです。つまり、実際の作業時間に対してのみ支払います。最初の関数呼び出しは通常より少し長く続く可能性があり、これは有名なコールドスタートです。実は、コンテナは一次ロードが必要です。PythonやJavaScriptの使用はそれほど大きな問題ではありませんが、Javaの場合、初期読み込みには10〜15秒かかることがあります。これは文であり、Javaの人気が低下する理由の1つです。ここで、タスクを開始、完了、停止できるコードが必要です。多くのスレッドや長時間実行されるプロセスを処理するのではなく、非常に速くロードできる短期間のプロセスが必要です。

Quarkusの紹介


テクニカルブログを読んだり、ニュースをフォローしたりすると、サーバーレスパラダイムが世界を席巻していると思います。スタートアップは、JavaScriptを使用することで、関数を記述してクラウドでサービスとして提供できるようになりました。また、インフラストラクチャを管理する必要なく、数百万のユーザーをサポートするように関数をスケーリングすることもできます。確かに、シリコンバレーの外には現実の世界もあります。金融機関、政府、小売、その他の多くの業界が、何百万ものJavaラインによってサービスを提供しています。したがって、これらの業界では引き続き重いコンテナを使用する必要があるという事実を当然のことと見なさなければなりません。



Graalvmそして、特に、Substrate VMは、今日、Java言語の輝かしい長い将来への扉を開きます。 GraalVMは、JavaScript、Python、Ruby、R、およびJVM用の言語、特にJava、Scala、またはKotlinで記述されたアプリケーションを実行するためのユニバーサル仮想マシンです。最もクールなのは、GraalVMを使用すると、プログラムを（AOTモードで）ネイティブ実行可能ファイルにプリコンパイルできることです。。つまり、Javaコードを直接マシン固有のコードにコンパイルできます。結果のプログラムはJava HotSpot VMでは機能しませんが、必要なすべてのコンポーネント、特にメモリ管理、Substrate VMと呼ばれる仮想マシンの別の実装からのスレッドスケジューリングを使用します。基板VMはJavaで記述され、そのコードはネイティブ実行可能ファイルにコンパイルされます。結果として得られるプログラムは、Java VMに比べて起動が速く、したがってメモリオーバーヘッドが低くなります。これはすばらしいことですが、おそらくコンパイルは事前に行われますか？これは、JVMが作成された基本的な考え方、つまり、どこでも一度作成されたコードの使用と矛盾します。狂ってる ！！！ただし、自分で考えてください。これでコンテナーができました。JVMは必要ありません。普通Springブートを使用して作成されたコンテナーアプリケーションには、追加の抽象化レベルがあります。これは、Kubernetesが存在する世界では絶対に不要です。コンテナー内のJVMでJavaアプリケーションが実行されています。この製品は今日ではアプリケーションではなくコンテナーであるため、このコンテナーは変更されません。現在は、WARファイルではなく、コンテナをパッキングしています。したがって、コンテナー内のJVMでアプリケーションを使用することに関連するすべてのコストは無駄になり、アプリケーションをコンテナーにパックする場合、AOTは非常に論理的なソリューションになります。

確かに、AOTコンパイルはJavaの動的機能（実行時のクラスのロード、リフレクション、プロキシなど）を大幅に制限します。実際には、これはJavaエコシステムの90％が変更なしでは機能しないことを意味します。したがって、Javaエコシステムは適応する必要があります。良い知らせがあります：これの大部分は組み立て中に行うことができます！

これがQuarkusの力です。GraalVMを使用し、ビルド時にAOTコンパイルをサポートするエコシステムを提供します。したがって、Javaを使用して、ネイティブバイナリを作成できます。Quarkusのおかげで、GraalVM はJava開発者が利用できるようになりました。

Quarkusの例


このセクションでは、初心者向けガイドを簡単に見て、 Quarkusのパワーについての印象を与えましょう。



新しいQuarkusプロジェクトを作成する最も簡単な方法は、コマンドプロンプトウィンドウを開いて、次のコマンドを実行することです。

mvn io.quarkus:quarkus-maven-plugin:0.12.0:create \
    -DprojectGroupId=org.acme \
    -DprojectArtifactId=getting-started \
    -DclassName="org.acme.quickstart.GreetingResource" \
    -Dpath="/hello"

これにより、エンドポイント/ helloを提供するGreetingResuceを使用してMavenプロジェクトが生成されます。ネイティブファイルのDockerfile Dockerイメージとjvm（厚いjar形式の従来のイメージ）も生成されます。コードは非常にクリーンでシンプルです。

@Path("/hello")
public class GreetingResource {    @GET
    @Produces(MediaType.TEXT_PLAIN)
    public String hello() {
        return "hello";
    }
}

アプリケーションを開始するには、以下を使用します。./mvnw compile quarkus:dev
アプリケーションは./mvnwパッケージを使用してパッケージ化されます。結果は2つのjarファイルです。

  • getting-started-1.0-SNAPSHOT.jar-クラスとプロジェクトリソースのみが含まれます。これは、Mavenのアセンブリから生じる一般的なアーティファクトです。
  • getting-started-1.0-SNAPSHOT-runner.jarは実行可能なjarです。これは「uber-jar」ではないことに注意してください。ここに依存関係があり、それらはtarget / libディレクトリにコピーされます。

次を使用してアプリケーションを起動でき

ます。java -jar target /getting-started-1.0-SNAPSHOT-runner.jar 次に、GralVM をダウンロードしてインストールし、環境変数を設定する必要がありますGRAALVM_HOME。

これで、ファイルを使用してネイティブ実行可能ファイルを作成できます./mvnw package -Pnative -Dnative-image.docker-build=true。

イメージDockerを作成する方法は次のとおりdocker build -f src/main/docker/Dockerfile.native -t quarkus-quickstart/quickstart です。

これで、ミニシフトを使用する場合は、任意のコンテナーオーケストレーションエンジンを使用して起動できます。

kubectl run quarkus-quickstart --image=quarkus-quickstart/quickstart:latest --port=8080 --image-pull-policy=IfNotPresent

kubectl expose deployment quarkus-quickstart --type=NodePort

それで全部です！; これで、0.004秒で起動するJava RESTサービスを備えたコンテナーができました。
