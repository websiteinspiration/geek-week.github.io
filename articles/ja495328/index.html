Gitでのコミットの履歴を整理する方法

hackernoon.comで見つけた記事の翻訳を公開しています。著者のチアゴ・ミランダは、Gitをより便利かつ効率的に使用する方法について書いています。



Gitのいくつかの非常に便利なコマンドについて


先月、私はペアプログラミングを行っており、Gitで数百のコミットを作成しました。この記事では、この期間に学んだことの中で最も重要なことを共有します。コミットの履歴の操作、成功した手法、および12のチームを使用してそれらを実装する方法について話します。

1. Gitのストーリーは何ですか？




ファイルに加えられた変更を含むすべてのコミットの正確な記録。その中で、特定の変更とそれらが行われた時間を追跡したり、現在のバージョンを以前のバージョンと比較したりできます。ストーリーはどこにありますか？Git Bashからコマンドを入力します。

git log --oneline

コミットが多すぎる場合は、他のファイルと同じように、矢印キーまたはPage Up / Page Downキーを使用してコメントを切り替えることができます。終了するには、ホットキー（q）を押します。

2.コミットに関するコメントについて


意味のないコメントは書かないでください。簡潔で、説明は不要です。彼らの仕事は、あなたがコードに加えた変更とそれらが何に影響を与えるかを示すことです。

  • 方法：「固定index.html」
  • 方法：「ナビゲーションバーの速度の向上」

これが何のためにあるのか見てみましょう。
あなたがそのような大規模なプロジェクトに取り組んでいて、そのプロジェクトに採用されている専門家のほとんどが知らないことを想像してみてください。あなたは新しい機能を開発し、あなたの機器でそれをテストします、そしてすべてがうまくいきます。

しかし、プッシュした後、すべてが突然機能しなくなり、最初に確認するのは、この機能の開発中に行われた変更です。Gitログでは、「固定」スタイルのコメントを含む多くのコミットが見つかります。適切なものを見つけるために費やす時間を想像してみてください。

3.常にコミットする


コミット、コミット、そして再度コミット。関数の完成-コミットの追加、ブロック要素のスタイルの改善-コミットの追加など。理想的には、変更のたびにコミットを送信する必要があります。

おそらくあなたは考えています：なぜそれほど多くのコミットがあるのですか？主な理由は、新しい機能が他の人のコードと競合する可能性があることですが、常に小さな変更をコミットすると、それらを見つけて修正するのが簡単になります。これにより、同じコミットで変更された数十または数百の行を追跡することで時間を節約できます。

4.コミットに関する最後のコメントを修正する


ローカルリポジトリに小さなコミットを追加した後、タイプミスを修正したり、コミットについてより詳細にコメントしたい場合はどうなりますか？次のコマンドを使用すると、変更は非常に簡単です。

git commit -m “correct message” --amend

注意：すでにリモートリポジトリへのコミットを開始している場合は、このコマンドを使用しないことをお勧めします。

詳細については、公式ドキュメントを参照してください。

5.最後のXコミットを1つに結合する


状況：新しい機能へのコミットの送信、もう1つの小さな変更が必要であり、最小限の変更を加えて再度コミットする必要があることを理解している...その結果、5つのコミットはほぼ同じものです。それは起こった？このようなコミットは、Gitでのストーリーの一般的なビューから削除されますが、必要に応じて、次のコマンドを使用して簡単に修正できます。

git reset HEAD~3

HEAD〜3コマンドは、最新のものを含め、3つの上位コミットをロールバックします。この例では、最後の3つのコミットがログから削除されますが、コードの変更はそのまま残ります。次に、コミットする必要のあるコードを確認します。これを行うには、次のコマンドを入力します。

git stage --diff

履歴から削除したコミットへのすべての変更にインデックスが付けられ、今度は一度にコミットできるようになります。

HEADは通常、最後に追加したコミットを指すことに注意してください。不明な場合は、Gitログを確認してください。最後のコミットが凍結された場合（最も一般的なケースではありません）、HEAD〜1コマンドは凍結されたブランチからすべてのコミットを消去します。詳細については、ドキュメントをご覧ください。

6.変更を加えた最後のコミットを削除する


この方法では、ロールバックする機能なしにすべての変更が消去されるため、注意してください。結果が期待に合わない場合、通常はコードを実験した後に使用されます。まずサンドボックスリポジトリで試すことをお勧めします。

git reset –-hard HEAD~1

これで、最後のコミットと、コード内の関連するすべての変更が削除されました。

7.コミット履歴をクリアする


コミット履歴をクリアする最も効率的な方法は、rebaseコマンドを使用することです。注意：誤って間違ったキーを押すと、コミットが削除される可能性があります。そこで、次のコマンドを使用してインタラクティブにリベース（-iフラグ）を実行してみましょう。

git rebase -i HEAD~5



入力するとすぐに、ターミナル内に最後の5つのコミット（HEAD〜5）のリストが表示されます。すべてのコミットは日付でソートされます。つまり、一番上のコミットが最新になります（このリストは、ターミナル内でテキストファイルを編集できるデフォルトのGitテキストエディターであるVimを使用して開かれます）。そして、ここにいくつかの便利なコマンドを含む簡単な説明があります。ほとんどの場合、squashコマンドとrewordコマンドが必要になります。

pickコマンドをsquashコマンドで置き換えると、このコミットがログから削除され、コード内のすべての変更が、pickコマンドで強調表示された最後のコミットでグループ化されます。

コメントを調整する場合は、pickコマンドをrewordコマンドに置き換えて、コメントを書き直すことができます。

これで、次のウィンドウに進むことができます。ここでは、スカッシュを使用して接着するコミットのグループに対して1つのコメントを書き込む必要があります。続行するには、ESCキーを押して次のように入力します。

:wq!

(:)コマンドの送信(w)、変更の書き込み（保存）、(q)終了、および(!)コマンドの実行を示すには、 コロンが必要です。

各コミットグループがあなたのコメントを受け取ることに注意してください。結果はGitログで確認できます。

何らかの理由で操作を完了せずにこのウィンドウを離れた場合は、次のコマンドを使用していつでも戻ることができます。

git rebase --edit

変更を保存せずにウィンドウを終了するには、ESCキーを押して次のように入力します。

:q!

Vimは保存せずにファイルを閉じるコマンドを受け取ります。

8.インデックスの管理


通常、コミットは単一のファイルまたは関連ファイルのグループに送信する必要があります。しかし、インデックス作成中にすばやく変更するにはどうすればよいですか？

たとえば、3つのファイルがあり、そのうち2つだけをコミットする必要があるとします。この場合、次のコマンドを試すことができます。

git add .

次に、不要なファイルをインデックスから削除します。

git reset -- Readme.txt

そして結果を確認してください：

git status

CSSなどの拡張子からすべてのファイルを追加します。

git add *.css

誤ってすべてを追加しましたか？次に、次のコマンドを使用してインデックスをクリアします。

git reset --

より複雑な操作が必要な場合は、ダイアログモードを使用してファイルをインデックスに追加できます。

git add -i

最初に、適切な数を入力してオプション（3など）を選択し、アクションをロールバックします。

オプションを選択することにより、インデックスから削除するファイルのリストを次々と入力できます。

完了したら、（Enter）を押します。

ファイルは同じ方法で追加されます。オプション（4）を使用して、追跡されていないファイルを追加します。

終了するには、(q)オプションメニューに入ります。

9.まとめ


リモートリポジトリにpushする前、特にブランチをマージする前に行う主なことは、コミット履歴を整理することです。すべてをリモートリポジトリにプッシュした後は、何も修正できません。
