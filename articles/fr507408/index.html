Cas 1: Variables atomiques ordonnées pour une programmation sans verrouillage



La programmation sans verrouillage implique d'établir une communication et une synchronisation entre les threads à l'aide d'outils d'un niveau inférieur à celui des verrous mutuellement exclusifs. Dans le passé et aujourd'hui, il existe un large éventail de ces outils. Dans un ordre historique grossier, ils incluent des barrières explicites (clôtures / barrières explicites - comme mb () sous Linux), des appels API de commande spéciaux (comme InterlockedExchange sous Windows) et divers types de types atomiques spéciaux. Beaucoup de ces outils sont mornes et / ou complexes, et leur grande variété signifie qu'en fin de compte, le code sans verrouillage est écrit différemment dans différents environnements.



Cependant, au cours des dernières années, il y a eu une convergence significative entre les fournisseurs de matériel et de logiciels: l'industrie informatique s'unit autour de variables atomiques ordonnées séquentiellement comme norme ou la seule façon d'écrire du code sans verrou en utilisant les principaux langages et plates-formes du système d'exploitation. En un mot, ordonné atomiqueles variables peuvent être lues et écrites en toute sécurité dans plusieurs threads simultanément sans verrou explicite, car elles offrent deux garanties: leur lecture et leur écriture sont garanties d'être exécutées dans l'ordre dans lequel elles apparaissent dans le code source de votre programme; et chaque lecture ou écriture est garantie d'être atomique, «tout ou rien». Ils ont également des opérations spéciales, telles que compareAndSet , qui sont garanties d'être exécutées de manière atomique. Voir [1] pour plus d'informations sur les variables atomiques ordonnées et comment les utiliser correctement.

Quelques mots sur l'optimisation



Nous verrons comment les variables atomiques ordonnées limitent l'optimisation que les compilateurs, processeurs, effets de mise en cache et autres éléments de votre runtime peuvent effectuer. Jetons donc un coup d'œil à quelques règles d'optimisation de base.



La règle d'optimisation fondamentale dans presque toutes les langues est la suivante: les optimisations qui réordonnent ("transforment") l'exécution de votre code ne sont toujours légitimes que si elles ne changent pas la signification du programme, de sorte que le programme ne peut pas déterminer la différence entre l'exécution du code source et celle convertie. Dans certaines langues, cela est également connu comme la règle «comme si», qui tire son nom du fait que le code converti a les mêmes effets observables, «comme si» (comme si) le code source était exécuté sous la forme dans lequel il a été écrit à l'origine.



Cette règle a un double effet: premièrement, l'optimisation ne devrait jamais vous permettre d'obtenir un résultat qui était auparavant impossible, ou de violer les garanties sur lesquelles le code source était autorisé à s'appuyer, y compris la sémantique du langage. Si nous donnons un résultat impossible, au final, le programme et l'utilisateur pourront certainement remarquer la différence, et ce n'est plus «comme si» nous avons exécuté le code original non converti.



Deuxièmement, l'optimisation a permis de réduire l'ensemble des exécutions possibles. Par exemple, l'optimisation peut conduire au fait que certaines alternances potentielles (mais non garanties) (modification de l'ordre d'exécution des instructions - entrelacement) ne se produiront jamais. C'est normal, car le programme ne peut toujours pas compter sur eux pour se produire.



Variables atomiques ordonnées et optimisation



L'utilisation de variables atomiques ordonnées limite les types d'optimisations que votre compilateur, votre processeur et votre système de mise en cache peuvent effectuer. [3] Deux types d'optimisations méritent d'être notés:



  • Optimisation des opérations atomiques de lecture et d'écriture ordonnées.
  • Optimisation des opérations de lecture et d'écriture normales adjacentes.


Tout d'abord, toutes les opérations de lecture et d'écriture atomiques ordonnées dans un flux donné doivent être effectuées strictement dans l'ordre du code source, car c'est l'une des garanties fondamentales des variables atomiques ordonnées. Néanmoins, nous pouvons toujours effectuer certaines optimisations, en particulier, des optimisations qui ont le même effet que si ce thread fonctionnait toujours si rapidement qu'à certains moments il n'y aurait pas d'alternance avec un autre thread.



Par exemple, considérons ce code, où a est une variable atomique ordonnée: 



a = 1;  	// A
a = 2;  	// B


Est-il permis au compilateur, au processeur, au cache ou à une autre partie du runtime de convertir le code ci-dessus en ce qui suit, à l'exclusion de l'entrée redondante dans la ligne A? 



// A ': OK:    A 
a = 2;  	// B


La réponse est oui. Ceci est légitime car le programme ne peut pas déterminer la différence; c'est «comme si» ce thread a toujours fonctionné si vite qu'aucun autre thread fonctionnant en parallèle ne peut en principe alterner entre les lignes A et B pour voir une valeur intermédiaire. [4]



De même, si a est une variable atomique ordonnée et que local est une variable locale indivisible, il est permis de convertir 



a = 1;  	// C:   a
local = a;  // D:   a


dans 



a = 1;  	// C:   a
local = 1;  // D': OK,  " "


ce qui exclut la lecture d' un . Même si un autre thread essaie d'écrire dans un en même temps , c'est «comme si» ce thread fonctionnait toujours si vite que l'autre thread n'a jamais réussi à alterner entre les lignes C et D pour changer la valeur avant de pouvoir réécrire la nôtre en local .



Deuxièmement, les opérations de lecture et d'écriture ordinaires à proximité peuvent toujours être réorganisées autour des opérations atomiques ordonnées, mais avec certaines limitations. En particulier, comme décrit dans [3], les opérations de lecture et d'écriture ordinaires ne peuvent pas monter par rapport à la lecture atomique ordonnée (de "après" à "avant") et ne peuvent pas descendre par rapport à la lecture atomique ordonnée (de "à" À après). En bref, cela peut les faire sortir de la section critique du code, et vous pouvez écrire des programmes qui en bénéficient en termes de performances. Pour plus d'informations, voir [3].

Nous avons besoin d'un moyen de désactiver toutes les optimisations pour les lire et les écrire. ISO C et C ++ ont un moyen standard portable de dire au compilateur qu'il s'agit d'une variable si spéciale qu'elle ne devrait pas être optimisée: volatile .



Java et .NET n'ont pas de concept comparable. Après tout, les environnements gérés doivent connaître la sémantique complète du programme qu'ils exécutent, il n'est donc pas surprenant qu'ils ne prennent pas en charge la mémoire avec une sémantique «inconnaissable». Mais Java et .NET fournissent tous deux des passerelles d'urgence pour quitter l'environnement géré et invoquer du code natif: Java fournit la Java Native Interface (JNI) et .NET fournit la Platform Invoke (P / Invoke). Cependant, la spécification JNI [5] ne dit rien sur volatile et ni Java volatile ni C / C ++ volatile ; de même, la documentation P / Invoke ne mentionne pas l'interaction avec .NET volatile ou C / C ++ volatile. Ainsi, pour un accès correct à la région de mémoire non optimisable en Java ou .NET, vous devez écrire des fonctions C / C ++ qui utilisent C / C ++ volatile pour effectuer le travail nécessaire au nom du code appelant, afin qu'elles encapsulent et masquent complètement la mémoire volatile (t c'est-à-dire n'acceptez ni ne renvoyez rien de volatile ) et appelez ces fonctions via JNI et P / Invoke.



Variables non optimisables et (non) optimisation



Toutes les opérations de lecture et d'écriture de variables non optimisables dans un flux donné doivent être effectuées exactement comme écrites; aucune optimisation n'est autorisée, car le compilateur ne peut pas connaître la sémantique complète de la variable et quand et comment sa valeur peut changer. Ce sont des expressions plus strictes (par rapport aux variables atomiques ordonnées) qui doivent être exécutées uniquement dans l'ordre du code source.



Reprenons les deux transformations que nous avons considéré plus tôt, mais cette fois nous allons remplacer la ordonnée la variable atomique une avec la variable non optimisable (C / C ++ volatile) v : 



v = 1;  // A
v = 2;  // B


Est-il légitime de convertir comme suit afin de supprimer une entrée clairement redondante dans la ligne A ? 



// A ': ,   
v = 2;  // B


La réponse est non, car le compilateur ne peut pas savoir que l'exception de l'écriture de la chaîne A en v ne change pas la signification du programme. Par exemple, v peut être l'emplacement auquel l'équipement utilisateur accède qui s'attend à voir une valeur de 1 avant une valeur de 2 et sinon ne fonctionnera pas correctement.

Deuxièmement, qu'en est-il des opérations de lecture et d'écriture régulières voisines - peuvent-elles être réorganisées autour d'opérations non optimisables? Aujourd'hui, il n'y a pas de réponse portable pratique, car les implémentations du compilateur C / C ++ sont très différentes et il est peu probable qu'elles commencent bientôt à évoluer vers l'uniformité. Par exemple, une interprétation de la norme C ++ indique que les opérations de lecture normales peuvent se déplacer librement dans n'importe quelle direction par rapport à la lecture ou à l'écriture C / C ++ volatile , tandis qu'un enregistrement régulier ne peut pas bouger du tout par rapport à la lecture ou à l'écriture volatileC / C ++ - ce qui rend le C / C ++ volatile à la fois moins et plus restrictif que les opérations atomiques ordonnées. Certains fournisseurs de compilateurs prennent en charge cette interprétation; d'autres n'optimisent pas du tout la lecture ou l'écriture volatile ; et d'autres encore ont leur propre sémantique.
