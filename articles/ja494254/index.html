KubernetesのエンタープライズバージョンとしてのOpenShift

「KubernetesとOpenShiftの違いは何ですか？」-この質問は、うらやましいほどの不変性で発生します。実際には、車とエンジンの違いを尋ねるようなものです。類推を続けると、車は完成品であり、文字通りすぐに使用できます。座って行ったのです。一方、エンジンがあなたをどこかに連れて行くためには、最終的に同じ車を手に入れるために、最初に他の多くのものでそれを補わなければなりません。



したがって、Kubernetesはそのようなエンジンであり、その周りにOpenShiftブランドの自動車（プラットフォーム）が組み付けられており、それによって目標に到達します。

この記事では、以下の重要なポイントをより詳細に思い出して分析したいと思います。

OpenShift – Kubernetes 100% CNCF


OpenShiftはKubernetes Certifiedに基づいています。したがって、適切なトレーニングを行った後、ユーザーはkubectlの機能を賞賛します。また、KubernetesクラスターでOpenShiftに切り替えたユーザーは、kubeconfigをOpenShiftクラスターにリダイレクトした後、すべてのスクリプトが問題なく機能することをよく言っています。

OCと呼ばれるOpenShiftコマンドラインユーティリティについて聞いたことがあるでしょう。これはkubectlコマンドと完全に互換性があり、さらにいくつかのタスクを実行するときに役立ついくつかの便利なヘルパーを提供します。ただし、最初に、OCとkubectlの互換性についてもう少し説明します。

言い換えると、すべてのKubernetesのAPIはOpenShiftで完全にアクセスでき、100％の互換性があります。そのため、OpenShiftはCloud Native Computing Foundation（CNCF）によって認定されたKubernetesプラットフォームとして認識されます。 

OpenShiftは便利な機能でKubernetesをサポート


Kubernetes 'APIはOpenShiftで100％使用できますが、通常のKubernetes' kubectlユーティリティには機能性と利便性が明らかに欠けています。そのため、Red Hatは、OC（OpenShiftクライアントの略）やODO（OpenShift DO、このユーティリティは開発者向けです）などの便利な機能とコマンドラインツールを備えたKubernetesを追加しました。

1. OCユーティリティ-Kubectlのより強力で便利なバージョン


たとえば、kubectlとは異なり、新しい名前空間を作成し、コンテキストを簡単に切り替えることができます。また、コンテナイメージのビルドや、ソースコードやバイナリファイルからのアプリケーションの直接デプロイ（Source-to-image、 s2i）。

組み込みヘルパーとOCユーティリティの高度な機能が日常業務の簡素化にどのように役立つかの例を見てみましょう。

例1は名前空間の管理です。各Kubernetesクラスターには常に複数の名前空間があります。通常、これらは開発および本番環境の作成に使用されますが、たとえば、各開発者に個人的な「サンドボックス」を提供するために使用することもできます。実際には、kubectlは現在のスペースのコンテキストで機能するため、開発者はしばしば名前空間を切り替える必要があるという事実につながります。したがって、kubectlの場合、人々はヘルパースクリプトを積極的に使用します。ただし、OCを使用して目的のスペースに切り替える場合は、「oc project namespace_」と言うだけで十分です。

名前空間の名前を覚えていませんか？問題ありません。「oc get projects」と入力するだけで、完全なリストが表示されます。クラスタ上の名前空間の限られたサブセットにしかアクセスできない場合、これがどのように機能するのか疑問に思っていますか？まあ、kubectlはこれを正しく行うので、RBACでクラスター上のすべてのスペースを表示できる場合にのみ、大規模なクラスターでは、誰もがそのような権限を与えるわけではありません。したがって、私たちは答えます：OCの場合、これはまったく問題ではなく、そのような状況で完全なリストを簡単に提供します。このような些細なことから、Openshiftの企業焦点と、ユーザーとアプリケーションの観点から見たこのプラットフォームの優れたスケーラビリティが構成されています。

ここまで読んだ後、KubernetesがOpenShiftの主要コンポーネントであることに気づいたことを願っています。基本的なものですが、唯一のものではありません。つまり、Kubernetesをインストールしただけでは、エンタープライズクラスのプラットフォームは得られません。認証、ネットワーク、セキュリティ、監視、ログ管理などを追加する必要があります。さらに、利用可能な多数のツールから難しい選択をする必要があります（エコシステムの多様性を評価するには、CNCF図を見てください））そして、何らかの形で一貫性と一貫性を確保し、全体として機能するようにします。さらに、使用されているコンポーネントのいずれかの新しいバージョンがリリースされた場合は、更新と回帰テストを定期的に実行する必要があります。つまり、プラットフォーム自体の作成と保守に加えて、このすべてのソフトウェアを扱う必要があります。ビジネス上の問題を解決して競争上の優位性を達成するために、ここに多くの時間が残されることはまずありません。

しかし、OpenShiftの場合、Red Hatはこれらのすべての困難を乗り越え、Kubernetes自体だけでなく、Kubernetesを実際のエンタープライズクラスのソリューションに変えるために必要なオープンソースツールのセット全体を含む機能的に完全なプラットフォームを提供しますすぐにそして完全に落ち着いて本番稼働します。そしてもちろん、テクノロジースタックがある場合は、OpenShiftを既存のソリューションに組み込むことができます。

1.基本としての信頼できるOS：RHEL CoreOSまたはRHEL


Red Hatは20年以上にわたり、ミッションクリティカルなビジネスアプリケーション向けのLinuxディストリビューションの主要プロバイダーでした。この分野で得られた経験と絶えず更新されている経験により、私たちはコンテナの産業運用に真に信頼できる信頼できる基盤を提供することができます。RHEL CoreOSはRHELと同じカーネルを使用しますが、主にコンテナーの実行やKubernetesクラスターでの作業などのタスク用に最適化されています。サイズが小さく、不変性であるため、クラスターのインストール、自動スケーリング、パッチのデプロイなどが簡単になります。これらのすべての機能は、ベアメタルからプライベートクラウドやパブリッククラウドまで、さまざまなコンピューティング環境でOpenShiftを操作するときに同じユーザーエクスペリエンスを得る理想的な基盤になります。

3.開発者ツール


2011年以降、OpenShiftはPaaS（Platform-as-a-Service）プラットフォームとして利用できるようになりました。これは、開発者の生活を大幅に簡素化し、コードの作成に集中できるようにし、Java、Node.js、PHPなどのプログラミング言語の組み込みサポートを提供します。 Ruby、Python、Go、およびCI / CDの継続的統合と配信サービス、データベースなど。OpenShift4は、 Red Hatとパートナーが開発したKubernetesオペレーターに基づく100以上のサービスを含む広範なカタログを提供します。

Kubernetesとは異なり、OpenShift 4には特別なグラフィカルインターフェイス（開発者コンソール）。これにより、開発者はさまざまなソース（git、外部レジストリ、Dockerfileなど）からアプリケーションを名前空間に簡単にデプロイし、アプリケーションコンポーネント間の接続を視覚的に視覚化できます。


デベロッパーコンソールでは、Kubernetesで動作するアプリケーションコンポーネントおよび簡素化を可視化する。

また、OpenShiftには、Codeready開発ツールのセット提供しています Codereadyワークスペース OpenShiftの上に直接働き、完全にコンテナのWebベースのIDEおよび実装IDEのようなアプローチをサービス"。一方で、厳密にローカルモードで作業したい人のために、Codeready Containersがあります。これは、ラップトップにデプロイできるOpenShift 4の完全に機能するバージョンです。


Kubernetes / OpenShiftプラットフォームでの効率的な開発のための統合された「IDE as a service」。

箱から出してすぐに、OpenShiftは、コンテナー化されたJenkinsとパイプライン用のDSLプラグイン、またはKubernetes CI / CDベースのTektonシステム（現時点では）に基づく完全なCI / CDシステムを提供しますTechプレビュー版）。これらのソリューションはどちらもOpenShiftコンソールと完全に統合されているため、パイプライントリガーを実行したり、デプロイやログを表示したりできます。

4.アプリケーション用のツール


OpenShiftを使用すると、従来のステートフルアプリケーションと、マイクロサービスやサーバーレスなどの新しいアーキテクチャに基づくクラウドベースのソリューションの両方をデプロイできます。すぐに使えるOpenShift Service Meshソリューションは、Istio、Kiali、Jaegerなどのマイクロサービスツールをサポートするための鍵です。一方、OpenShiftサーバーレスソリューションには、Knativeだけでなく、Microsoftとの共同イニシアチブの一部として作成されたKedaなどのツールも含まれ、OpenShiftプラットフォームでAzure機能を提供します。


統合されたOpenShift ServiceMeshソリューション（Istio、Kiali、Jaeger）は、マイクロサービスの開発に役立ちます。
