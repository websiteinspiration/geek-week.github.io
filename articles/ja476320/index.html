新しいアプリケーションリリースをKubernetesに配信するプロセスでチームを実行する



私たちの実践では、クライアントアプリケーションをKubernetesで実行するように適応させるというタスクに直面することがよくあります。これらの作業を行う場合、いくつかの典型的な問題が発生します。最近、Kubernetesにアプリケーションを移植する際の記事「ローカルファイル」でそれらの1つを強調表示しました。もう1つはすでにCI / CDプロセスに関連付けられているため、この記事で説明します。

Helmとwerfを使用した任意のコマンド


アプリケーションは、ビジネスロジックとデータだけでなく、更新を成功させるために実行する必要がある任意のコマンドのセットでもあります。これらは、たとえば、データベースの移行、外部リソースの可用性の「ウェイター」、一部のトランスコーダーまたはアンパッカー、外部Service Discoveryのレジストラーなどです。異なるプロジェクトの異なるタスクに対応できます。

Kubernetesはそのような問題を解決するために何を提供していますか？Kubernetesはポッドとしてコンテナを起動するのに優れているため、標準的なソリューションはイメージからコマンドを実行することです。これを行うには、Kubernetesにジョブプリミティブがあり、アプリケーションコンテナーでポッドを実行し、このポッドの完了を監視できます。

Helmはもう少し進んで、展開プロセスのさまざまな段階でジョブを起動することを提案します。リソースマニフェストの更新前または更新後にジョブを起動するために使用できるHelmフックについて話しています。私たちの経験では、これは展開タスクの解決に使用できる優れたHelm機能です。

ただし、Helmでは、ロールアウト中にオブジェクトの状態に関する最新情報を取得できないため、werfユーティリティを使用しますこれにより、ロールアウト中にCIシステムからリソースの状態を直接監視し、障害が発生した場合に故障をより迅速に診断することができます。

結局のところ、これらの便利な機能であるHelmとwerfは相互に排他的な場合がありますが、解決策は常に存在します。リソースのステータスを監視し、移行の例で任意のコマンドを実行する方法を検討してください。

リリース前に移行を実行する


データベースアプリケーションのリリースの不可欠な部分は、データスキーマの更新です。別のコマンドを実行して移行を適用するアプリケーションの標準的な展開では、次の手順を実行します。

  1. コードベースの更新;
  2. 移行の開始。
  3. トラフィックをアプリケーションの新しいバージョンに切り替えます。

Kubernetes内では、プロセスは同じですが、必要に応じて調整する必要があります。

  1. コンテナを新しいコードで開始します。これには新しい移行のセットが含まれる場合があります。
  2. アプリケーションのバージョンを更新する前に、移行を適用するプロセスを開始します。

アプリケーションのデータベースがすでに実行されていて、アプリケーションをデプロイするリリースの一部としてデータベースをデプロイする必要がない 場合は、このオプションを検討してください。マイグレーションの適用には2つのフックが適しています。

  • pre-install -すべてのテンプレートを処理した後、Kubernetesでリソースを作成する前の、アプリケーションの最初のHelmリリースで機能します。
  • pre-upgrade-Helmリリースが更新および実行されたとき、およびpre-installテンプレートの処理後、ただしKubernetesでリソースを作成する前に機能します。

Helmと前述の2つのフックを使用したジョブの例：

---
apiVersion: batch/v1
kind: Job
metadata:
 name: {{ .Chart.Name }}-apply-migrations
 annotations:
   "helm.sh/hook": pre-install,pre-upgrade
spec:
 activeDeadlineSeconds: 60
 backoffLimit: 0
 template:
   metadata:
     name: {{ .Chart.Name }}-apply-migrations
   spec:
     imagePullSecrets:
     - name: {{ required ".Values.registry.secret_name required" .Values.registry.secret_name }}
     containers:
     - name: job
       command: ["/usr/bin/php7.2", "artisan", "migrate", "--force"]
{{ tuple "backend" . | include "werf_container_image" | indent 8 }}
       env:
{{ tuple "backend" . | include "werf_container_env" | indent 8 }}
       - name: DB_HOST
         value: postgres
     restartPolicy: Never

  • {{ tuple "backend" . | include "werf_container_image" | indent 8 }}必要なコンテナのイメージに置き換えます。
  • {{ tuple "backend" . | include "werf_container_env" | indent 8 }}キーで指定された行を削除しますenv。

そのため、このHelmテンプレート.helm/templatesは、残りのリリースリソースがすでに含まれているカタログに追加する必要があります。が呼び出されるとwerf deploy --stages-storage :local、すべてのテンプレートが最初に処理され、次にKubernetesクラスターにロードされます。

リリースプロセス中に移行を開始する


上記のオプションは、データベースがすでに実行されている場合の移行の使用を意味します。しかし、アプリケーションのブランチレビューをロールアウトする必要があり、1つのリリースでアプリケーションとともにデータベースがロールアウトされた場合はどうなるでしょうか。

注意：データベースのIPアドレスを含むエンドポイントでサービスを使用してデータベースに接続する場合、本番環境にロールアウトするときに同様の問題が発生する可能性があります。

この場合、アプリケーションは移行を適用しようとするため、フックpre-installとpre-upgradeは適切なデータベースではありません。したがって、リリース後に移行を行う必要があります。

Helmを使用すると、追跡しないため、このようなタスクを実行できます。アプリケーションの状態。Kubernetesにリソースを読み込んだ後、ポストフックは常に起動します。

  • post-install -最初のリリースですべてのリソースをK8にロードした後。
  • post-upgrade -リリースの更新時にK8のすべてのリソースを更新した後。

ただし、前述したように、werfにはリリース中にリソース追跡システムがあります。これについてもう少し詳しく説明します。

  • 追跡のために、WERFはの機能を使用していますkubedogのライブラリ我々はすでに、の話ブログでを。
  • werfのこの機能により、リリースのステータスを一意に決定し、CI / CDシステムのインターフェースで展開の成功または失敗に関する情報を表示できます。
  • Kubernetesクラスターでのリソースの正常な作成はステージの1つにすぎないため、この情報を受信しないと、リリースプロセスの自動化について話すことはできません。たとえば、設定が正しくないか、ネットワークの問題が原因でアプリケーションが起動しないhelm upgrade場合がありますが、後でこれを確認するには、追加の手順を実行する必要があります。

Helmポストフックでのマイグレーションのアプリケーションに戻ります。私たちが遭遇した問題：

  • 多くのアプリケーションは、何らかの方法で起動する前に、データベース内の回路の状態をチェックします。したがって、新しい移行を行わないと、アプリケーションが起動しない場合があります。
  • werfはデフォルトで、すべてのオブジェクトが確実にstate状態Readyになるので、postフックは機能せず、移行は失敗します。
  • 追加のアノテーションを使用してオブジェクトの追跡を無効にすることができますが、展開の結果に関する信頼できる情報を取得することはできません。

その結果、次のことが実現しました。

  • ジョブはコアリソースの前に作成されるため、移行にHelmフックを使用する必要はありません。
  • ただし、移行を伴うジョブはすべてのデプロイで実行する必要があります。これを実現するには、ジョブに一意の名前（ランダム）が必要です。この場合、Helmの場合、これは毎回リリースの新しいオブジェクトであり、Kubernetesで作成されます。
  • このようなリリースでは、すべてのジョブに一意の名前が付けられ、以前のジョブは新しいリリースで削除されるため、ジョブがマイグレーションで蓄積することを心配するのは意味がありません。
  • マイグレーションを伴うジョブには、データベースの可用性をチェックするinitコンテナーが必要です -そうでない場合、デプロイメントがドロップされます（ジョブはinitコンテナーに分類されます）。

結果の構成は次のようになります。

kind: Job
metadata:
name: {{ printf "%s-apply-migrations-%s" .Chart.Name (now | date "2006-01-02-15-04-05") }}

  activeDeadlineSeconds: 60
  backoffLimit: 0
  template:
    metadata:
      name: {{ printf "%s-apply-migrations-%s" .Chart.Name (now | date "2006-01-02-15-04-05") }}

      - name: job
        command: ["/usr/bin/php7.2", "artisan", "migrate", "--force"]

        env:
{{ tuple "backend" . | include "werf_container_env" | indent 8 }}

すべてのデプロイ操作のユニバーサルテンプレートの例


ただし、リリース中に実行する必要がある操作は、すでに説明した移行の開始以上のものになる場合があります。ジョブの実行順序は、フックのタイプだけでなく、アノテーションを使用して各フックに重みを割り当てることによっても制御できhelm.sh/hook-weightます。フックは昇順で重みでソートされ、重みが同じである場合はリソース名でソートされます。

ジョブが多数ある場合は、ジョブのユニバーサルテンプレートを作成し、構成をに移動すると便利values.yamlです。後者は次のようになります。

deploy_jobs:
  - name: migrate
    command: '["/usr/bin/php7.2", "artisan", "migrate", "--force"]'
    activeDeadlineSeconds: 120
    when:
      production: 'pre-install,pre-upgrade'
      staging: 'pre-install,pre-upgrade'
      _default: ''
  - name: cache-clear
    command: '["/usr/bin/php7.2", "artisan", "responsecache:clear"]'
    activeDeadlineSeconds: 60
    when:
      _default: 'post-install,post-upgrade'

...そしてテンプレート自体はこのようなものです：

{{- range $index, $job := .Values.deploy_jobs }}
---
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ $.Chart.Name }}-{{ $job.name }}
  annotations:
    "helm.sh/hook": {{ pluck $.Values.global.env $job.when | first | default $job.when._default }}
    "helm.sh/hook-weight": "1{{ $index }}"
spec:
  activeDeadlineSeconds: {{ $job.activeDeadlineSeconds }}
  backoffLimit: 0
  template:
    metadata:
      name: {{ $.Chart.Name }}-{{ $job.name }}
    spec:
      imagePullSecrets:
      - name: {{ required "$.Values.registry.secret_name required" $.Values.registry.secret_name }}
      initContainers:
      - name: wait-db
        image: alpine:3.6
        ommand: ["/bin/sh", "-c", "while ! nc -z postgres 5432; do sleep 1; done;"]
      containers:
      - name: job
        command: {{ $job.command }}
{{ tuple "backend" $ | include "werf_container_image" | indent 8 }}
        env:
{{ tuple "backend" $ | include "werf_container_env" | indent 8 }}
       - name: DB_HOST
         value: postgres
      restartPolicy: Never
{{- end }}

このアプローチにより、リリースプロセスに新しいコマンドをすばやく追加し、実行可能なコマンドのリストをより視覚的にすることができます。
