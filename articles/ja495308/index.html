キープアライブに向かう途中のレーキ

マイクロサービス間のデータ交換のアクティビティを増やすことは、最新のITソリューションのアーキテクチャーで問題になることがよくあります。最大を絞って、すべてのコストで生き残る-どの開発にとっても深刻な課題です。したがって、最適なソリューションの検索はノンストップのプロセスです。この記事では、高負荷のHTTPリクエストを使用するときに発生する可能性のある問題と、それをバイパスする方法について簡単に説明します。


この話は間違いから始まります。どういうわけか、負荷テストを実行しました。その主な要素は、多数の短いhttpリクエストの実行でした。netcore 2.2で作成されたクライアントが、ある時点から、System.Net.Sockets.SocketException：既に使用されているアドレスをスローします。ポートはクライアントで解放する時間がないことがすぐに明らかになり、ある時点でシステムは新しいポートを開くことを拒否されました。コードに移動すると、問題はHttpWebRequestクラスと構成で古いアプローチを使用することでした。


var request = WebRequest.CreateHttp(uri);
using(var resp = request.GetResponse()){ … }

リソースを解放しているようで、ポートは適時に解放する必要があります。ただし、netstatは、TIME_WAIT状態のポート数の急激な増加を示しています。この状態は、接続が閉じるのを待つことを意味します（失われたデータを受信する可能性もあります）。その結果、ポートは1〜2分間その中にある可能性があります。この問題は、多くの記事（にいくつかの詳細に議論されTIME_WAITキューの問題、TIME_WAITについての歴史）。それにもかかわらず、これはdotnetが「正直に」接続を閉じようとしていることを意味し、さらにシステムのタイムアウト設定の障害によってすでに発生しています。
