検索スケーリング。Elasticsearch インストールの利点と基本要件

こんにちは。私の名前はロマン・ラルチコフです。私はドキュメントビジョンのテクニカルサポートエンジニアです。この記事は、検索スケーリングの実装の技術的な詳細とElasticsearchの作業に精通している方を対象としています。この記事では、ESを使用する理由、システム要件、およびMS SQL Serverからの検索に対する利点について説明します。

プラットフォームの最新バージョンで検索をどのように拡大したかを知るためのより一般的な用語に興味がある場合は、同僚がこれについてWebセミナー「Docsvision ECM」で話しました。スケーリングElasticSearchを検索します。」

Elasticsearchを選ぶ理由




そもそも、Docsvision 5.5プラットフォームのバージョンは、モジュラーアーキテクチャの点で以前のバージョンとは根本的に異なっています。この点で、作業速度を維持しながら、システムをほぼ無制限にスケーリングできるようにする必要がありました。特に、高速な操作でもインデックスサービスを拡張できる必要がありました。

この点で、Docsvision 5.5のバージョンでは、外部（衛星）データベースを使用する可能性が導入されました。現在、1つのデータベースにすべてのデータを保存する必要はありません。集中的な作業中に毎日ボリュームが増大するため、データベースのサービスプロセスが複雑になり、クラッシュ時の回復速度が遅くなり、データベース自体の全体的な動作が遅くなります。



すべてに1つのデータベースを使用するのはよくありません。 Docsvision 5.5に実装されている個別の外部データベースにデータを転送する機能により、データベースを正しく再構築できます。したがって、検索について説明する場合、インデックス作成データは既にメインデータベースの外部に格納でき、そのサイズへの影響を排除できます。

カスタマイズのしやすさ、柔軟性、信頼性、スケーラビリティ、インデックス作成とオンライン検索の速度はすべて、Elasticsearchに関するものです。

Elasticsearchは非常にドキュメント指向です。インデックスを作成した後、列のデータ行ではなく、データの検索、並べ替え、フィルタリングを行うことができます。これは、データ取得に対する別のアプローチを示し、Elasticsearchが複雑な全文検索を実行できることを示しています。

ドキュメントはJSONオブジェクトとして表されます。同時に、シリアル化（データ構造をビットのシーケンスに変換するプロセス）JSONは、ほとんどのプログラミング言語でサポートされており、すでにNoSQLの標準形式です。

1. Elasticsearchの紹介


1.1。それは何ですか？


Elasticsearchは、Luceneライブラリーを使用し、Javaで書かれた、オープンソースのスケーラブルな全文検索エンジンです。このエンジンのすべての利点の説明は公式のウェブサイトで利用可能です。

ドキュメント/ファイルのデータベースでの複雑な検索を目的としています。 Elasticsearchデータベースでは、テーブルはインデックスと呼ばれ、ドキュメントをロードするプロセスはインデックスと呼ばれます。

これは、JSON形式のドキュメントの非リレーショナルリポジトリとLucene全文検索に基づく検索エンジンの両方と見なすことができます。公式クライアントは、Java、NET（C＃）、Python、Groovy、JavaScript、PHP、Perl、Rubyで利用できます。

ESは、Elastic Stack、Elasticsearch、Logstash、Beats、Kibanaと呼ばれる関連プロジェクトとともに、Elasticによって開発されています。

Elasticsearchは、データの格納と検索を担当します（以下、簡潔にするために、ESと呼びます）。

Docsvision 5.5では、使用する検索エンジンを選択できます。この記事では、Elasticsearch検索エンジンの使用に焦点を当て、MS SQL Serverからの検索に対するその利点について説明します。

  • ESはオープンソースプロジェクトであり、多くのグローバル企業がそれを使用して巨大なデータセットを検索しています。

2. Elasticsearchのインストールに必要なソフトウェアおよびシステム要件


ESは、高性能サーバーだけでなく、ラップトップにも展開できます。ただし、生産的な環境について話している場合は、別のサーバーを使用して、検討する価値のあるいくつかの推奨事項を順守する必要があります。

2.1。羊


ESの最も重要なリソースはRAMです。これは、おそらく最初に終了する主要なリソースです。最小許容サイズは8Gbで、推奨サイズは16〜64Gbです。本当に必要がある場合は、さらに許可されます。



並べ替えと集計は大量のメモリを消費する可能性があるため、十分なメモリを確保することが重要です。64 GBのRAMを搭載したマシンが理想的なソリューションですが、32 GBと16 GBを搭載したマシンも一般的です。マシンに8 GB以下がインストールされている場合、これは逆の結果になる可能性があります（最終的には、そのような「小さな」マシンがいくつか必要になる場合があります）。64 GBを超える使用には、独自の特性もあります。

2.2。CPU


原則として、ESにはプロセッサ要件があまりないため、その選択は他のリソースほど重要ではありません。



しかし、あなたはルールを守るべきです。複数のコアを備えた最新のプロセッサを選択する必要があります。通常、クラスター内のサーバーは2〜8コアのマシンを使用します。

より高速なプロセッサまたは複数のコアを持つプロセッサのどちらかを選択できる場合は、後者を選択する必要があります。複数のコアが提供する追加の並列処理により、わずかに高いクロック周波数よりも大きな結果が得られます。

2.3。ディスク


このディスクは、ESをすばやく操作するための重要なリソースでもあります。これは、クラスターを使用する場合に重要であり、大量のインデックス付きデータがあるクラスターでは二重に重要です。

ディスクはサーバーで最も遅いサブシステムです。つまり、記録が集中しているクラスターはディスクに高い負荷をかける可能性があり、それがサーバーのボトルネックになります。ソリッドステートドライブを使用できる場合は、それらを使用する必要があります。それらは回転媒体よりはるかに優れています。 SSDをサポートするホストでは、クエリとインデックス作成の両方のパフォーマンスが著しく向上しています。



ハードディスク（HDD）を使用する場合は、高性能サーバーディスク（スピンドル速度が15,000 rpmのディスク）を使用することをお勧めします。

RAID 0の使用は、回転ディスクとSSDの両方のディスク速度を向上させる効果的な方法です。高可用性はレプリカを介してESに組み込まれているため、ミラーリングまたはパリティーを備えたRAIDオプションを使用する必要はありません。

2.4。I / Oスケジューラ


ソリッドステートドライブを使用している場合は、オペレーティングシステムのI / Oスケジューラが正しく構成されていることを確認する必要があります。データがディスクに書き込まれているとき、I / Oスケジューラは、そのデータが実際にディスクに送信されるタイミングを決定します。ほとんどの場合、デフォルトのcfqスケジューラが使用されます（完全に正直なキュー）。

このスケジューラは、各プロセスにタイムスロットを割り当て、これらのさまざまなキューのディスクへの配信を最適化します。 HDDでの作業用に最適化されています。回転プレートの性質により、物理的な場所に応じてデータをディスクに書き込む方が効率的です。

ただし、回転プレートを使用しないため、ソリッドステートドライブには効果がありません。代わりに、締切またはnoopを使用してください。noopは単なるFIFOキューであるのに対して、デッドラインスケジューラは、記録が待機していた時間に応じて最適化します。

これらの単純な変更により、適切なスケジューラを使用して書き込みスループットを大幅に改善できます。

2.5。通信網


高速で信頼性の高いネットワークは、分散システムのパフォーマンスにとって重要です。低レイテンシにより、ノードはデータを簡単に交換でき、高スループットはデータの移動と回復に役立ちます。

最新のデータセンターネットワーク（1 GbE、10 GbE）は、大多数のクラスターに十分です。

複数のデータセンターにまたがるクラスターは、データセンターが近接していても、避ける必要があります。地理的に離れた距離にあるクラスターは絶対に避けてください。

ESクラスターは、すべてのノードが等しいと想定しています。長い遅延は分散システムの問題を悪化させ、デバッグと解決を困難にする傾向があります。

2.6。一般的な推奨事項


ESを起動するだけのコストを削減するために、「中」と「大」のマシンを優先し、低パフォーマンスのマシンを回避することは価値があります。同時に、本当に巨大なマシンは回避する必要があります。それらは、リソースの不均衡な使用につながることが多く（たとえば、すべてのメモリが使用されますが、中央プロセッサは使用されません）、マシンで複数のノードを実行する必要がある場合は、ロジスティックの複雑さを追加できます。
