In dem obigen Algorithmus gibt es so etwas wie ein Zertifikat. Im Rahmen dieses Beispiels müssen Sie eine kryptografische Theorie verstehen. Zu Habré gibt es einen ausgezeichneten Artikel zu diesem Thema .

In diesem Beispiel verwenden wir asymmetrische Verschlüsselungsalgorithmen. Um asymmetrische Algorithmen zu implementieren, benötigen Sie ein Schlüsselpaar und ein Zertifikat.

Ein Schlüsselpaar besteht aus zwei Teilen: einem privaten Schlüssel und einem öffentlichen Schlüssel. Ein privater Schlüssel muss, wie der Name schon sagt, geheim sein. Wir verwenden es, um Informationen zu entschlüsseln. Sie können den öffentlichen Schlüssel an jeden weitergeben. Dieser Schlüssel wird zum Verschlüsseln von Daten verwendet. Somit kann jeder Benutzer Daten mit einem öffentlichen Schlüssel verschlüsseln, aber nur der Besitzer des privaten Schlüssels kann diese Informationen entschlüsseln.

Ein Zertifikat ist ein elektronisches Dokument, das Informationen über den Benutzer, dem das Zertifikat gehört, sowie einen öffentlichen Schlüssel enthält. Mit einem Zertifikat kann der Benutzer alle Daten signieren und an einen Server senden, der diese Signatur überprüfen und die Daten entschlüsseln kann.

Um eine Nachricht mit einem Zertifikat korrekt zu signieren, müssen Sie sie korrekt erstellen. Dazu wird zuerst ein Schlüsselpaar auf Rutoken erstellt, und dann muss ein Zertifikat an den öffentlichen Schlüssel dieses Schlüsselpaars angehängt werden. Das Zertifikat muss genau den öffentlichen Schlüssel haben, der sich auf Rutoken befindet. Dies ist wichtig. Wenn wir einfach ein Schlüsselpaar und ein Zertifikat direkt auf der Clientseite der Anwendung erstellen, wie kann der Server dann diese verschlüsselte Nachricht entschlüsseln? Schließlich weiß er absolut nichts über das Schlüsselpaar oder das Zertifikat.

Wenn Sie sich eingehender mit diesem Thema befassen, finden Sie im Internet interessante Informationen. Es gibt einige Zertifizierungszentren, denen wir wissentlich vertrauen. Diese Zertifizierungsstellen können Benutzern Zertifikate ausstellen und diese Zertifikate auf ihrem Server installieren. Wenn ein Client danach auf diesen Server zugreift, sieht er dieses Zertifikat und stellt fest, dass es von einer Zertifizierungsstelle ausgestellt wurde. Dies bedeutet, dass diesem Server vertraut werden kann. Mehr darüber, wie man alles richtig einrichtet, das Internet ist auch voller Informationen. Damit können Sie beispielsweise beginnen .

Wenn wir zu unserer Aufgabe zurückkehren, scheint die Lösung offensichtlich. Sie müssen irgendwie Ihr Zertifizierungszentrum machen. Zuvor müssen Sie jedoch verstehen, auf welcher Grundlage das Zertifizierungszentrum dem Benutzer ein Zertifikat ausstellen sollte, da er nichts über ihn weiß. (Zum Beispiel sein Vor- und Nachname usw.) Es gibt so etwas wie eine Zertifikatsanforderung. Weitere Informationen zu diesem Standard finden Sie beispielsweise auf Wikipedia ru.wikipedia.org/wiki/PKCS.
Wir werden Version 1.7 - PKCS # 10 verwenden.

Beschreiben wir den Algorithmus zum Generieren eines Zertifikats für Rutoken (Primärquelle - Dokumentation ):

  1. Erstellen Sie auf dem Client ein Schlüsselpaar und speichern Sie es in Rutoken. (Speichern erfolgt automatisch)
  2. Wir erstellen eine Zertifikatanforderung auf dem Client.
  3. Vom Client senden wir diese Anfrage an den Server.
  4. Nach Erhalt einer Zertifikatanforderung auf dem Server stellen wir ein Zertifikat bei unserer Zertifizierungsstelle aus.
  5. Wir senden dieses Zertifikat an den Kunden.
  6. Auf dem Client speichern wir das Zertifikat für Rutoken.
  7. Das Zertifikat muss an das Schlüsselpaar angehängt werden, das im ersten Schritt erstellt wurde.

Jetzt wird klar, wie der Server die Signatur des Clients entschlüsseln kann, da er ihm selbst ein Zertifikat ausgestellt hat.
