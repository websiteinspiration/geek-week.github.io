クラウド内の仮想マシンクラスターでのAnsible +自動gitプル



良い一日


それぞれに多数の仮想マシンを持つクラウドクラスターがいくつかあります。このビジネスはすべてHetzner'eでホストされています。各クラスターには、マスターマシンが1つあり、そこからスナップショットが取得され、クラスター内のすべての仮想マシンに自動的に配布されます。

このスキームでは、通常のgitlab-runnersを使用できません。同じ登録済みランナーが多数表示されると多くの問題が発生し、回避策を見つけてこの記事/マニュアルを作成するように促したためです。

これはおそらくベストプラクティスではありませんが、このソリューションはできる限り便利でシンプルなように見えました。

チュートリアルでは猫をお願いします。

マスターマシンに必要なパッケージ：

  • python
  • ギット
  • sshキーを含むファイル

すべての仮想マシンに自動腸プルを実装する一般的な原則は、Ansibleをインストールするマシンが必要であることです。このマシンから、ansibleはgit pullコマンドを送信し、更新されたサービスを再起動します。これらの目的のために、クラスターの外側に別の仮想マシンを作成し、それにインストールしました。

  • python
  • ansible
  • gitlab-runner

組織的な問題から-gitlab-runnerを登録し、ssh-keygenを作成し、このマシンの公開sshキーを.ssh/authorized_keysマスターマシンにドロップし、マスターマシンでansibleの22番目のポートを開く必要があります。

今すぐansibleを設定します


私たちの目標は可能なことをすべて自動化することなので。ファイルで/etc/ansible/ansible.cfgはhost_key_checking = False、Ansibleが新しいマシンの確認を求めないように、行のコメントを外します。

次に、ansibleのインベントリファイルを自動的に生成する必要があります。そこから、git pullを実行する必要があるマシンのIPを取得します。

Hetzner APIを使用してこのファイルを生成します。AWS、Asure、データベースからホストのリストを取得できます（実行中のマシンを表示するためのAPIをどこかに持っていますよね？）。

インベントリファイルの構造はAnsibleにとって非常に重要です。その外観は次のようになります。

[]
ip-
ip-

[2]
ip-
ip-


そのようなファイルを生成するために、簡単なスクリプトを作成しましょう（それを呼び出しましょうvm_list）：

#!/bin/bash
echo [group] > /etc/ansible/cloud_ip &&
" CLI    IP    "  >> /etc/ansible/cloud_ip
echo " " >> /etc/ansible/cloud_ip
echo [group2] > /etc/ansible/cloud_ip &&
" CLI    IP     "  >> /etc/ansible/cloud_ip

ansibleが機能し、IPアドレスの受信者と友好的であることを確認する時が来ました： 

/etc/ansible/./vm_list && ansible -i /etc/ansible/cloud_ip -m shell -a 'hostname' group

出力には、コマンドが実行されたマシンのホスト名が表示されます。
構文に関するいくつかの単語：

  • /etc/ansible/./vm_list-マシンのリストを生成します
  • -i-インベントリファイルへの絶対パス
  • -m-シェルモジュールを使用するようにansibleに指示します
  • -aは引数です。ここに任意のチームを入力できます。
  • groupはクラスターの名前です。すべてのクラスターで行う必要がある場合は、グループをallに変更します

どうぞ-仮想マシンでgit pullを実行してみてください。

/etc/ansible/./vm_list && ansible -i /etc/ansible/cloud_ip -m shell -a 'cd /path/to/project && git pull' group

出力またはリポジトリからのアンロードですでに最新の状態になっている場合、すべてが機能します。

今、それは何のためにあったのか


gitlab'eのマスターブランチでコミットするときにスクリプトが自動的に実行されるように教えます

まず、スクリプトをより美しくして実行可能ファイルに入れます（exec_pullと呼びましょう）-

#!/bin/bash

/etc/ansible/./get_vms && ansible -i /etc/ansible/cloud_ip -m shell -a "$@"

gitlabに移動し、プロジェクトでファイルを作成します.gitlab-ci.yml
。

variables:
  GIT_STRATEGY: none
  VM_GROUP: group

stages:
  - pull
  - restart

run_exec_pull:
  stage: pull
  script:
  
   - /etc/ansible/exec_pull 'cd /path/to/project/'$CI_PROJECT_NAME' && git pull' $VM_GROUP
  
  only:
  - master

run_service_restart:
  stage: restart
  script:
 
   - /etc/ansible/exec_pull 'your_app_stop && your_app_start' $VM_GROUP
   
  only:
  - master

すべて準備ができています。今-

  • コミットする
  • すべてが機能することを喜ぶ

.ymlを他のプロジェクトに移植するときは、再起動するサービス名と、ansibleコマンドが実行されるクラスターの名前を変更するだけです。
