Kubernetesでのデプロイ戦略：ローリング、再作成、ブルー/グリーン、カナリア、ダーク（A / Bテスト）

注意 perev。：Weaveworksからのこのレビュー資料では、最も人気のあるアプリケーションロールアウト戦略を紹介し、KubernetesオペレーターのFlaggerを使用してそれらの最も高度なものを実装する可能性について説明します。シンプルな言語で書かれており、初心者のエンジニアでも問題を理解できるようにする視覚的な図が含まれています。


この図は、Container Solutionsによって行われたロールアウト戦略の別のレビューから取られたものです。

導入戦略


目標に基づいて使用できるいくつかの異なるタイプの展開戦略があります。たとえば、特定の環境を変更してさらにテストしたり、ユーザーやクライアントのサブセットを変更したり、一部の機能を公開する前にユーザーに対して限定的なテストを実施したりする必要があるかもしれません。

ローリング（段階的な「ローリング」デプロイメント）


これは、Kubernetesの標準のデプロイ戦略です。ポッドは1つずつ徐々に、アプリケーションの古いバージョンに置き換えられ、新しいバージョンのポッドに置き換えられます-クラスターのダウンタイムはありません。



Kubernetes は、古いポッドの折りたたみを開始する前に、新しいポッドの準備ができるまで待機します（準備テストで確認します）。問題が発生した場合、クラスター全体を停止することなく、このようなローリング更新を中断できます。デプロイメントタイプのYAMLファイルでは、新しいイメージが古いイメージを置き換えます。

apiVersion: apps/v1beta1
kind: Deployment
metadata:
  name: awesomeapp
spec:
  replicas: 3
  template:
    metadata:
      labels:
        app: awesomeapp
    spec:
      containers:
        - name: awesomeapp
          image: imagerepo-user/awesomeapp:new
          ports:
            - containerPort: 8080

ローリング更新のパラメーターは、マニフェストファイルで指定できます。

spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
       maxSurge: 25%
       maxUnavailable: 25%  
  template:
  ...

再作成（再作成）


この最も単純なタイプのデプロイでは、古いポッドが一度にすべて削除され、新しいポッドに置き換えられます。



対応するマニフェストは次のようになります。

spec:
  replicas: 3
  strategy:
    type: Recreate
  template:
  ...

ブルー/グリーン（ブルーグリーン展開）


青緑展開戦略（赤/黒、つまり赤黒とも呼ばれる）には、アプリケーションの古い（緑）バージョンと新しい（青）バージョンの同時展開が含まれます。両方のバージョンを配置すると、通常のユーザーは緑のバージョンにアクセスできますが、青のバージョンは、QAチームが別のサービスまたは直接ポート転送を介してテストを自動化するために利用できます。



apiVersion: apps/v1beta1
kind: Deployment
metadata:
  name: awesomeapp-02
spec:
  template:
    metadata:
      labels:
        app: awesomeapp
        version: "02"

青（新しい）バージョンがテストされ、そのリリースが承認されると、サービスはそれに切り替わり、緑（古い）は最小化されます。

apiVersion: v1
kind: Service
metadata:
  name: awesomeapp
spec:
  selector:
    app: awesomeapp
    version: "02"
...

Canary（カナリアデプロイメント）


カナリアロールは青緑色のように見えますが、より適切に管理され、段階的な段階的アプローチを使用しています。このタイプには、「非表示」の起動やA / Bテストなど、いくつかの異なる戦略が含まれます。

この戦略は、通常はアプリケーションのバックエンドで、いくつかの新しい機能を試す必要がある場合に使用されます。このアプローチの本質は、ほぼ同じ2つのサーバーを作成することです。1つはほぼすべてのユーザーにサービスを提供し、もう1つは新機能を備えたユーザーの一部のみを提供し、その後で結果を比較します。すべてが順調に進んだ場合、新しいバージョンは徐々にインフラストラクチャ全体に展開されます。

この戦略はKubernetesツールを使用してのみ実装でき、古いポッドを新しいポッドに置き換えますが、Istioのようなサービスメッシュを使用する方がはるかに便利で簡単です。

たとえば、Gitで2つの異なるマニフェストを作成できます。0.1.0タグが付いた通常のマニフェストと0.2.0タグが付いた「カナリア」マニフェストです。Istio仮想ゲートウェイのマニフェストの重みを変更することで、これら2つのデプロイメント間のトラフィックの分散を制御



できます。Istioを使用したカナリアデプロイメントを実装するためのステップバイステップガイドは、GitOpsワークフローとIstioの記事にあります。（注の翻訳：Istioのカナリアロールに関する資料もここで翻訳しました）。

Weaveworks Flaggerを使用したカナリアデプロイメント


Weaveworks Flaggerを使用すると、カナリアロールアウトを簡単かつ効率的に管理できます。

Flaggerはそれらの作業を自動化します。 IstioまたはAWS App Meshを使用してトラフィックのルーティングと切り替えを行い、Prometheusメトリックを使用して結果を分析します。さらに、カナリアデプロイメントの分析には、受け入れテスト、負荷テスト、およびその他の種類のチェックを実行するためのWebhookを追加できます。

：Kubernetesの展開と、必要に応じて、水平ポッドスケーリング（HPA）に基づき、フラガーはカナリア展開の分析と実装のためのオブジェクト（Kubernetesの展開、CLUSTERIPサービスとIstioやアプリのメッシュ仮想サービス）のセットを作成



の実装を制御ループ（制御ループ）Flaggerは、トラフィックをカナリアサーバーに徐々に切り替えながら、成功したHTTPリクエストの割合、平均リクエスト期間、ポッドヘルスなどの主要なパフォーマンスインジケーターを測定します。KPI（主要業績評価指標）の分析に基づいて、カナリア部分が成長または崩壊し、分析結果がSlackに公開されます。このプロセスの説明とデモは、App Meshのプログレッシブ配信にあります。



ダーク（隠し）またはA / B展開


ひそかな展開はカナリア戦略のもう1つのバリエーションです（ちなみに、Flaggerもそれと連携できます）。秘密のデプロイメントとカナリアデプロイメントの違いは、カナリアデプロイメントのように、バックエンドではなくフロントエンドを扱うことです。

これらのデプロイメントの別名は、A / Bテストです。新しい機能へのアクセスをすべてのユーザーに開放するのではなく、一部のユーザーにのみ提供されます。通常、これらのユーザーは、彼らがパイオニアテスターであることを知らない（したがって、「秘密の配備」という用語）。機能トグルの

使用 その他のツールを使用すると、ユーザーが新機能をどのように操作するか、それがユーザーを魅了するかどうか、新しいユーザーインターフェイスが混乱するかどうか、およびその他の種類のメトリックを監視できます。



フラッガーとA / Bの導入


Flaggerは、重み付けルーティングに加えて、HTTP設定に応じて、トラフィックをカナリアサーバーに転送することもできます。A / Bテストでは、HTTPヘッダーまたはCookieを使用して、特定のユーザーセグメントをリダイレクトできます。これは、サーバーのセッションアフィニティを必要とするフロントエンドアプリケーションで特に効果的です。詳細については、Flaggerのドキュメントを参照してください。
