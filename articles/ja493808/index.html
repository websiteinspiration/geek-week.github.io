Corutinによる非同期コードの線形化


コルーチンを使用してジェネレーターを作成するだけでなく、それらを使用して既存の非同期コードを線形化することもできます。小さな例でこれをやってみましょう。アクターフレームワークで記述されたコードを取り、このコードの1つの関数をコルーチンで書き換えます。プロジェクトをビルドするには、コルーチンブランチのgccを使用します。

私たちの目標は、麺からコールバックを取得することです。

    abActor.getA(ABActor::GetACallback([this](int a) {
        abActor.getB(ABActor::GetBCallback([a, this](int b) {
            abActor.saveAB(a - b, a + b, ABActor::SaveABCallback([this](){
                abActor.getA(ABActor::GetACallback([this](int a) {
                    abActor.getB(ABActor::GetBCallback([a, this](int b) {
                        std::cout << "Result " << a << " " << b << std::endl;
                    }));
                }));
            }));
        }));
    }));

並べ替え：

const int a = co_await actor.abActor.getAAsync();
const int b = co_await actor.abActor.getBAsync();
co_await actor.abActor.saveABAsync(a - b, a + b);
const int newA = co_await actor.abActor.getAAsync();
const int newB = co_await actor.abActor.getBAsync();
std::cout << "Result " << newA << " " << newB << std::endl;

それでは始めましょう。

俳優


まず、シンプルなアクターのフレームワークを作成する必要があります。本格的なアクターフレームワークを作成することは困難で大きなタスクであるため、ある種のフレームワークのみを実装します。
まず、基本クラスを作成します。

class Actor {
public:
    using Task = std::function<void()>;
public:
    virtual ~Actor();
public:
    void addTask(const Task &task);
    void tryRunTask();
private:
    std::queue<Task> queue;
    mutable std::mutex mutex;
};

アイデアは基本的にシンプルです。機能オブジェクトであるタスクをキューに入れ、tryRunTaskを実行すると、このタスクを完了しようとします。クラスの実装は私たちの意図を確認します：

Actor::~Actor() = default;

void Actor::addTask(const Task &task) {
    std::lock_guard lock(mutex);
    queue.push(task);
}

void Actor::tryRunTask() {
    std::unique_lock lock(mutex);
    if (queue.empty()) {
        return;
    }

    const Task task = queue.front();
    queue.pop();
    lock.unlock();

    std::invoke(task);
}

次のクラスは、アクターが所属する「スレッド」です。

class Actor;

class ActorThread {
public:
    ~ActorThread();
public:
    void addActor(Actor &actor);
    void run();
private:
    std::vector<std::reference_wrapper<Actor>> actors;
};

ここでもすべてが単純です。プログラムの最初に、addActorメソッドを使用してアクターをスレッドに「バインド」し、次にrunメソッドを使用してスレッドを開始します。

ActorThread::~ActorThread() = default;

void ActorThread::addActor(Actor &actor) {
    actors.emplace_back(actor);
}

void ActorThread::run() {
    while (true) {
        for (Actor &actor: actors) {
            actor.tryRunTask();
        }
    }
}

スレッドを開始すると、無限ループに入り、各アクターから1つのタスクを実行しようとします。最善の解決策ではありませんが、デモには役立ちます。

次に、アクタークラスの代表を見てみましょう。

class ABActor: public Actor {
public:
    using GetACallback = Callback<void(int result)>;
    using GetBCallback = Callback<void(int result)>;
    using SaveABCallback = Callback<void()>;
public:
    void getA(const GetACallback &callback);
    void getB(const GetBCallback &callback);
    void saveAB(int a, int b, const SaveABCallback &callback);
private:
    void getAProcess(const GetACallback &callback);
    void getBProcess(const GetBCallback &callback);
    void saveABProcess(int a, int b, const SaveABCallback &callback);
private:
    int a = 10;
    int b = 20;
};

このクラスはそれ自体に2つの数値-aとbを格納し、要求に応じてそれらの値を返すか、それらを上書きします。

コールバックとして、必要なパラメーターを持つ機能オブジェクトを受け入れます。しかし、異なるアクターが異なるスレッドで起動される可能性があるという事実に注目しましょう。したがって、作業の最後にメソッドに渡されたコールバックを呼び出すだけの場合、このコールバックは、メソッドを呼び出してこのコールバックを作成したスレッドではなく、現在の実行可能スレッドで呼び出されます。したがって、この状況を解決するコールバックのラッパーを作成する必要があります。

template<typename C>
class Callback {
public:
    template<typename Functor>
    Callback(Actor &sender, const Functor &callback)
        : sender(sender)
        , callback(callback)
    {}
public:
    template<typename ...Args>
    void operator() (Args&& ...args) const {
        sender.addTask(std::bind(callback, std::forward<Args>(args)...));
    }
private:
    Actor &sender;
    std::function<C> callback;
};

このラッパーは元のアクターを記憶しているため、自分で実行しようとすると、元のアクターのタスクキューに実際のコールバックが追加されるだけです。
その結果、ABActorクラスの実装は次のようになります。

void ABActor::getA(const GetACallback &callback) {
    addTask(std::bind(&ABActor::getAProcess, this, callback));
}

void ABActor::getAProcess(const ABActor::GetACallback &callback) {
    std::invoke(callback, a);
}

void ABActor::getB(const GetBCallback &callback) {
    addTask(std::bind(&ABActor::getBProcess, this, callback));
}

void ABActor::getBProcess(const ABActor::GetBCallback &callback) {
    std::invoke(callback, b);
}

void ABActor::saveAB(int a, int b, const SaveABCallback &callback) {
    addTask(std::bind(&ABActor::saveABProcess, this, a, b, callback));
}

void ABActor::saveABProcess(int a, int b, const ABActor::SaveABCallback &callback) {
    this->a = a;
    this->b = b;
    std::invoke(callback);
}

クラスのインターフェースメソッドでは、渡された引数をクラスの対応する「スロット」にバインドしてタスクを作成し、このタスクをこのクラスのタスクキューに配置します。タスクスレッドがタスクの実行を開始すると、適切な「スロット」を呼び出し、必要なすべてのアクションを実行してコールバックを呼び出します。コールバックは、実際のコールバックを、それを引き起こしたタスクのキューに送信します。

ABActorクラスを使用するアクターを書いてみましょう：

class ABActor;

class WokrerActor: public Actor {
public:
    WokrerActor(ABActor &actor)
        : abActor(actor)
    {}
public:
    void work();
private:
    void workProcess();
private:
    ABActor &abActor;
};

void WokrerActor::work() {
    addTask(std::bind(&WokrerActor::workProcess, this));
}

void WokrerActor::workProcess() {
    abActor.getA(ABActor::GetACallback(*this, [this](int a) {
        std::cout << "Result " << a << std::endl;
    }));
}

そして、すべてをまとめます：

int main() {
    ABActor abActor;
    WokrerActor workerActor(abActor);

    ActorThread thread;
    thread.addActor(abActor);
    thread.addActor(workerActor);

    workerActor.work();

    thread.run();

コードのチェーン全体を追跡してみましょう。

最初に、必要なオブジェクトを作成し、それらの間の接続を確立します。
次に、workProcessタスクをアクターのWorkerタスクキューに追加します。
スレッドが開始すると、キュー内のタスクを見つけて実行を開始します。
実行プロセスでは、ABActorクラスのgetAメソッドを呼び出して、対応するタスクをABActorクラスのキューに入れ、実行を完了します。
次に、スレッドはABActorクラスから新しく作成されたタスクを取得して実行します。これにより、getAProcessコードが実行されます。
このコードはコールバックを呼び出し、必要な引数をそれに渡します-変数a。しかし、彼が所有するコールバックはラッパーであるため、実際には、パラメーターが入力された実際のコールバックがWorkerクラスのキューに入れられます。
そして、サイクルの次の反復で、スレッドがWorkerクラスからコールバックを引き出して実行すると、「結果10」という行の出力が表示されます。

アクターフレームワークは、さまざまな物理ストリームに散在するクラスを互いに相互作用させるかなり便利な方法です。これを確信しているはずのクラス設計の特徴は、個々のアクター内ですべてのアクションが完全に単一のスレッドで実行されることです。ストリームの同期の唯一のポイントは、アクターフレームワークの実装の詳細で行われ、プログラマーには見えません。したがって、プログラマーは、ミューテックスのまとめや、競合状況、デッドロック、その他の頭痛の追跡について心配することなく、シングルスレッドのコードを書くことができます。

コルーチン。パート1.アウェイター


コルーチンの主なアイデアは、コルーチンを作成すると、ヒープ上にコルーチン用に別のスタックフレームが作成され、そこからいつでも「終了」できると同時に、現在の実行位置、プロセッサレジスタ、その他の必要な情報を維持することです。その後、中断されたコルーチンの実行にいつでも戻って、最後または次の中断まで完了することができます。

std :: coroutine_handle <>オブジェクトは、このデータの管理を担当します。このデータは、基本的にスタックフレーム（およびその他の必要なデータ）へのポインターを表し、再開メソッド（またはそのアナログ、演算子（））を持ち、コルーチンの実行に戻ります。

このデータに基づいて、まずgetAAsync関数を記述してから、一般化してみましょう。

それで、すでにstd :: coroutine_handle <> coroクラスのインスタンスがあるとしたら、何をする必要がありますか？

必要に応じて状況を解決する既存のメソッドABActor :: getAを呼び出す必要がありますが、最初にgetAメソッドのコールバックを作成する必要があります。

コールバックがgetAメソッドコールバック（getAメソッドの結果）に返されることを思い出してください。さらに、このコールバックはスレッドのワーカースレッドで呼び出されます。したがって、このコールバックから、ワーカースレッドからのみ作成され、アクションのシーケンスを実行し続けるコルーチンを安全に実行し続けることができます。しかし、また、コールバックで返された結果をどこかに保存する必要があります。当然、それはさらに便利です。

auto callback = GetACallback(returnCallbackActor, [&value, coro](int result) {
        value = result;
        std::invoke(coro);
 });
getA(callback);

したがって、ここで、どこかからcoroutine_handleオブジェクトのインスタンスと、結果を保存できるリンクを取得する必要があります。

将来的には、関数を呼び出した結果としてcoroutine_handleが渡されることがわかります。したがって、それを使用してできることは、他の関数に渡すことだけです。この関数をラムダとして用意しましょう。（コールバックの結果が格納される変数へのリンクを会社に渡します）。

auto storeCoroToQueue = [&returnCallbackActor, this](auto &value, std::coroutine_handle<> coro) {
    auto callback=GetACallback(returnCallbackActor, [&value, coro](int result){
        value = result;
        std::invoke(coro);
    });
    getA(callback);
};

この関数は次のクラスで保存します。

ここでは、機能オブジェクトに加えて、コールバックで待機する値のメモリ（変数値の形式）も保持します。

ここではメモリを値の下に保持しているので、このクラスのインスタンスをどこかにコピーまたは移動することはほとんどありません。たとえば、誰かがこのクラスをコピーし、クラスの古いインスタンスの変数値の下に値を保存して、新しいインスタンスから読み取ろうとしたとします。そして、保存前にコピーが行われたため、当然そこにはありません。不快。したがって、コンストラクターやコピーおよび移動演算子を禁止することで、この問題から身を守っています。

このクラスを書き続けましょう。次に必要な方法は次のとおりです。

    bool await_ready() const noexcept {
        return false;
    }

彼は私たちの意味が発行される準備ができているかどうかの質問に答えます。当然、最初の呼び出しでは、まだ値の準備ができていません。将来、誰もこのことについて質問することはないので、falseを返します。

coroutine_handleインスタンスはvoid await_suspendメソッド（std :: coroutine_handle <> coro）で渡されるので、準備されたファンクターを呼び出し、そこに値の下のメモリリンクも渡します。

    void await_suspend(std::coroutine_handle<> coro) noexcept {
        std::invoke(forwardCoroToCallback, std::ref(value), coro);
    }

関数の実行結果は、await_resumeメソッドを呼び出すことにより、適切なタイミングで尋ねられます。私たちは要求者に拒否しません：

    int await_resume() noexcept {
        return value;
    }

これで、co_awaitキーワードを使用してメソッドを呼び出すことができます。

const int a = co_await actor.abActor.getAAsync(actor);

ここで何が起こるかは、すでにおおまかに表しています。

最初に、タイプActorAwaiterSimpleのオブジェクトが作成され、co_awaitに渡されます。彼は最初に（await_readyを呼び出すことによって）誤って終了した結果（私たちにはない）があるかどうかを尋ね、次にawait_suspendを呼び出し、コンテキスト（実際には、現在のコルーチンスタックフレームへのポインター）を渡して実行を中断します。

将来、ABActorアクターが作業を完了して結果コールバックを呼び出すと、この結果（すでにWorkerスレッドスレッドにある）がActorAwaiterSimpleの唯一の（コルーチンスタックに残っている）インスタンスに保存され、コルーチンの継続が開始されます。

Corutinは実行を継続し、await_resumeメソッドを呼び出して保存された結果を取得し、この結果を変数aに渡します

現時点では、現在のAwaiterの制限は、int型の1つのパラメーターを持つコールバックでしか機能しないことです。Awaiterのアプリケーションを拡張してみましょう。

template<typename... T>
struct ActorAwaiter {

    std::tuple<T...> values;

    std::function<void(std::tuple<T...> &values, std::coroutine_handle<>)> storeHandler;

    ActorAwaiter(const std::function<void(std::tuple<T...> &values, std::coroutine_handle<>)> &storeHandler)
        : storeHandler(storeHandler)
    {}

    ActorAwaiter(const ActorAwaiter &) = delete;
    ActorAwaiter& operator=(const ActorAwaiter &) = delete;
    ActorAwaiter(ActorAwaiter &&) = delete;
    ActorAwaiter& operator=(ActorAwaiter &&) = delete;

    bool await_ready() const noexcept {
        return false;
    }

    void await_suspend(std::coroutine_handle<> coro) noexcept {
        std::invoke(storeHandler, std::ref(values), coro);
    }

    //   bool B  ,
    //   sfinae      
    template<
        bool B=true,size_t len=sizeof...(T),std::enable_if_t<len==0 && B, int>=0
    >
    void await_resume() noexcept {

    }

    //   bool B  ,
    //   sfinae      
    template<
        bool B=true,size_t len=sizeof...(T),std::enable_if_t<len==1 && B, int>=0
    >
    auto await_resume() noexcept {
        return std::get<0>(values);
    }

    //   bool B  ,
    //   sfinae      
    template<
        bool B=true,size_t len=sizeof...(T),std::enable_if_t<len!=1 && len!=0 && B, int>=0
    >
    std::tuple<T...> await_resume() noexcept {
        return values;
    }
};

ここでは、std ::タプルを使用して、一度に複数の変数を保存できるようにしています。

Sfinaeはawait_resumeメソッドに課されているため、すべてのケースでタプルを返すことはできませんが、タプルにある値の数に応じて、void、正確に1つの引数またはタプル全体を返します。

Awaiter自体を作成するためのラッパーは、次のようになります。

template<typename MakeCallback, typename... ReturnArgs, typename Func>
static auto makeCoroCallback(const Func &func, Actor &returnCallback) {
    return [&returnCallback, func](auto &values, std::coroutine_handle<> coro) {
        auto callback = MakeCallback(returnCallback, [&values, coro](ReturnArgs&& ...result) {
            values = std::make_tuple(std::forward<ReturnArgs>(result)...);
            std::invoke(coro);
        });
        func(callback);
    };
}

template<typename MakeCallback, typename... ReturnArgs, typename Func>
static ActorAwaiter<ReturnArgs...> makeActorAwaiter(const Func &func, Actor &returnCallback) {
    const auto storeCoroToQueue = makeCoroCallback<MakeCallback, ReturnArgs...>(func, returnCallback);
    return ActorAwaiter<ReturnArgs...>(storeCoroToQueue);
}

ActorAwaiter<int> ABActor::getAAsync(Actor &returnCallback) {
    return makeActorAwaiter<GetACallback, int>(std::bind(&ABActor::getA, this, _1), returnCallback);
}

ActorAwaiter<int> ABActor::getBAsync(Actor &returnCallback) {
    return makeActorAwaiter<GetBCallback, int>(std::bind(&ABActor::getB, this, _1), returnCallback);
}

ActorAwaiter<> ABActor::saveABAsync(Actor &returnCallback, int a, int b) {
    return makeActorAwaiter<SaveABCallback>(std::bind(&ABActor::saveAB, this, a, b, _1), returnCallback);
}

コルーチン。パート2.再開可能


C ++の観点からは、単語co_await、co_yield、またはco_returnを含む関数はコルーチンと見なされます。しかし、そのような関数は特定の型を返す必要もあります。関数の署名を変更しないことに同意しました（ここで、戻り値の型も署名を参照することを意味します）。

ラムダコルーチンを作成して、関数から呼び出しましょう。

void WokrerActor::workProcess() {
    const auto coroutine = [](WokrerActor &actor) -> ActorResumable {
        const int a = co_await actor.abActor.getAAsync(actor);
        const int b = co_await actor.abActor.getBAsync(actor);
        co_await actor.abActor.saveABAsync(actor, a - b, a + b);
        const int newA = co_await actor.abActor.getAAsync(actor);
        const int newB = co_await actor.abActor.getBAsync(actor);
        std::cout << "Result " << newA << " " << newB << std::endl;
    };

    coroutine(*this);
}

initial_suspend（）の後、一時停止はせずに進みます。プログラムの主要部分の実行を開始します。

co_awaitに到達すると、一時停止する必要があることがわかります。この状況については前のセクションですでに説明しました。戻って確認することができます。

実行を続行して結果を画面に表示した後、コルーチンの実行は終了します。final_suspendをチェックし、コルーチンのコンテキスト全体をクリアします。

void WokrerActor::workCoroProcess() {
    const auto coroutine = [](WokrerActor &actor) -> ActorResumable {
        const auto [a, b] = co_await actor.readAB();
        co_await actor.abActor.saveABAsync2(actor, a - b, a + b);
        const auto [newA, newB] = co_await actor.readAB();
        std::cout << "Result " << newA << " " << newB << " " << a << " " << b << std::endl;
    };

    coroutine(*this);
}

あとは、CoroTaskタイプを発明するだけです。考え直そう。最初に、co_returnはreadAB関数内で使用されます。つまり、CoroTaskはResumableインターフェースを満たす必要があります。ただし、このクラスのオブジェクトは、別のコルーチンのco_awaitを入力するためにも使用されます。これは、CoroTaskクラスもAwaitableインターフェイスを満たす必要があることを意味します。これらの両方のインターフェースをCoroTaskクラスに実装しましょう。

template <typename T = void>
struct CoroTask {
    struct promise_type {
        T result;
        std::coroutine_handle<> waiter;

        auto get_return_object() {
            return CoroTask{*this};
        }

        void return_value(T value) {
            result = value;
        }

        void unhandled_exception() {
            std::terminate();
        }

        std::suspend_always initial_suspend() {
            return {};
        }

        auto final_suspend() {
            struct final_awaiter {
                bool await_ready() {
                    return false;
                }
                void await_resume() {}
                auto await_suspend(std::coroutine_handle<promise_type> me) {
                    return me.promise().waiter;
                }
            };
            return final_awaiter{};
        }
    };

    CoroTask(CoroTask &&) = delete;
    CoroTask& operator=(CoroTask&&) = delete;
    CoroTask(const CoroTask&) = delete;
    CoroTask& operator=(const CoroTask&) = delete;

    ~CoroTask() {
        if (h) {
            h.destroy();
        }
    }

    explicit CoroTask(promise_type & p)
        : h(std::coroutine_handle<promise_type>::from_promise(p))
    {}

    bool await_ready() {
        return false;
    }

    T await_resume() {
        auto &result = h.promise().result;
        return result;
    }

    void await_suspend(std::coroutine_handle<> waiter) {
        h.promise().waiter = waiter;
        h.resume();
    }
private:
    std::coroutine_handle<promise_type> h;
};

では、ここで何が起こるか見てみましょう。

1.ラムダコルーチンに移動し、すぐにWokrerActor :: readABコルーチンを作成します。ただし、このコルーチンを作成した後は、コルーチンの実行を開始せず（initial_suspend == suspend_always）、コルーチンラムダに割り込んで戻ります。

2. co_awaitラムダは、readABの準備ができているかどうかを確認します。結果の準備ができていません（await_ready == false）。これにより、コンテキストがCoroTask :: await_suspendメソッドに渡されます。このコンテキストはCoroTaskに保存され、readAB

3 コルーチンの再開が開始されます。readABコルーチンが必要なすべてのアクションを完了すると、次の行に到達します。

co_return std::make_pair(a, b);

その結果、CoroTask :: promise_type :: return_valueメソッドが呼び出され、作成された数値のペアがCoroTask :: promise_type

4 内に保存されます。co_returnメソッドが呼び出されたため、コルーチンの実行が終了しました。つまり、CoroTask :: promise_type :: final_suspendメソッドを呼び出すときです。 。このメソッドは自己記述型の構造を返します（図を確認することを忘れないでください）。これにより、final_awaiter :: await_suspendメソッドを呼び出さなければならず、手順2で保存されたコルーチンラムダコンテキストが返されます。

ここでsuspend_alwaysを返せないのはなぜですか？結局のところ、このクラスのinitial_suspendの場合、成功しましたか？実際のところ、このコルーチンがラムダコルーチンによって呼び出され、それに戻ったため、initial_suspendで成功しました。しかし、final_suspend呼び出しに到達した時点で、コルーチンはおそらく別のスタック（特に、makeCoroCallback関数が準備したラムダ）から継続しており、suspend_alwaysをここで返すと、workCoroProcessメソッドではなく、それに戻ります。

5. final_awaiter :: await_suspendメソッドがコンテキストを返したため、プログラムは強制的に、返されたコンテキスト、つまりコルーチンラムダの実行を継続します。実行がポイントに戻ったので：

const auto [a, b] = co_await actor.readAB();

次に、CoroTask :: await_resumeメソッドを呼び出して、保存された結果を分離する必要があります。結果が受信され、変数aおよびbに渡されます。これで、CoroTaskインスタンスが破棄されます。

6. CoroTaskインスタンスは破棄されましたが、WokrerActor :: readABコンテキストはどうなりましたか？ CoroTask :: promise_type :: final_suspendからの場合、suspend_neverを返します（より正確には、await_ready質問にtrueが返されます）、その時点でコルーチンコンテキストがクリーンアップされます。しかし、私たちはそうしなかったので、コンテキストをクリアする義務は私たちに移されます。 CoroTaskデストラクタでこのコンテキストをクリアします。この時点ではすでに安全です。

7.コルーチンreadABが実行され、そこから結果が取得され、コンテキストがクリアされ、ラムダコルーチンが引き続き実行されます...

ふう、ちょっと整理した。メソッドABActor :: getAAsync（）などから、自己記述構造を返すことを覚えていますか？実際、getAsyncメソッドは、CoroTaskクラスとActorAwaiterクラスの実装から得られた知識を組み合わせて、次のようなものを取得することによって、コルーチンに変換することもできます。
