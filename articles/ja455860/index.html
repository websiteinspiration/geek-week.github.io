Camundaでのスタイリッシュでファッショナブルな若者向けBPM開発


BPMの開発は簡単ではありません。これは、技術的な観点からだけでなく、プロセスが顧客にとって読みやすく、理解可能でなければならないという事実によるものです。

すべてのビジネスプロセス開発ツールが、明確な説明と技術的機能との間の妥協点を見つけることを可能にするわけではありません。多くの高度な開発ツールとプロセスの説明には、もう1つの欠点があります。それらは非常にクールで強力、かつ複雑であるため、作成されている間、テクノロジーは大きな一歩を踏み出し、そのようなツールによる開発は無関係になりました。

2018年は、ビジネスプロセスの開発に対するアプローチを根本的に変えました。以下は、このアプローチがどのように進化し、どのように変わったかについてです。

プロローグの代わりに


私たちの部門は、ビジネスプロセスの開発に取り組んでいます-最小から最小まで、大規模で非常に収益性の高いものまで。最近まで、開発にはIBMの製品を使用していました。これにより、稼働中のビジネスプロセスを迅速に立ち上げることができました。

IBM BPMは、プロセス自体の説明、UIフォーム、統合モジュールなどの豊富な機能セットを含む強力なプラットフォームです。さらに、このプラットフォームのエントリーしきい値はかなり低く、初心者の開発者はすぐにプロジェクトに没頭できます。ただし、この製品には、開発速度を低下させない場合、速度と品質に貢献しないという欠点もあります。

これらの欠点は、純粋に技術的な開発の不便さに加えて、さらに別の問題を引き起こしました。おそらく、上記のすべてよりもはるかに深刻な問題です。 Java 12、Kotlin、マイクロサービス、その他のファッションのトレンドや作品の時代には、これらすべてのニュアンスがチームを非常にやる気にさせています。 2019年には、絶え間なく変化するJava 6用のIntegration Designerで開発する喜びを体験するのは困難です。



これらすべての制限があるため、浮かぶことは困難です。 1年弱前、ビジネスプロセスを記述するためにCamundaエンジンを試すという提案がありました。まず、非常に大規模ではありませんが、法人の端末を登録するためのかなり重要なプロセスが選択されました。

完全に書き直したため、古いコードはほとんどなく、事実上何も制限することができなかったため、開発言語としてKotlinを選択しました。主に肯定的なレビューで聞かれたこの新しい言語を試すのは興味深いことでした。私たちの部門の他のいくつかのプロジェクトでは、成功した実装経験がありました。最終的なスタックは次のようになりました：Camunda、Spring Boot 2、Kotlin、Postgre。

カマンダとは？




Camundaは、Javaで記述され、Javaを開発言語として使用するオープンソースのビジネスプロセスモデリングプラットフォームです。説明されているプロセスを実行できるようにするライブラリのセットです。 Camundaをプロジェクトに統合するには、いくつかの依存関係を追加するだけです。プロセスを保存するには、タスクに応じて、インメモリまたは永続的なDBMSを選択できます。私たちがPostgreを選んだのは、その話が「報告」のために私たちにとって重要だからです。デフォルトでは、プラットフォームはH2にデプロイされます。

開発は2つの部分で構成されます。特別なCamunda Modelerユーティリティでフロープロセスを作成することと、図で説明されているプロセスステップを処理するJavaコードを作成することです。プロセスからJavaコードを呼び出すには、JavaDelegateインターフェースを実装し、コンテキストでこのBeanを上げて（フルネームでdelagateを指定できますが、Beanを使用するとより便利で柔軟です）、目的のプロセスステップでそのIDを指定するだけです。Kotlinでは、デリゲートはさらに簡潔に見えます。デリゲートのロジックは非常に単純です。コンテキストから何かを差し引き、いくつかのアクションを実行して、コンテキストに戻しました。


Camunda Modelerポップアップウィンドウ

Javaデリゲートの例：


import org.camunda.bpm.engine.delegate.DelegateExecution;
import org.camunda.bpm.engine.delegate.JavaDelegate;

public class JavaExampleDelegate implements JavaDelegate {
   
    @Override
    public void execute(DelegateExecution execution)  {
        String someVar = (String) execution.getVariable("someVariable");
        // some actions
        execution.setVariable("otherVariable", "otherValue");
    }
}

Kotlinデリゲートの例：


import org.camunda.bpm.engine.delegate.DelegateExecution
import org.camunda.bpm.engine.delegate.JavaDelegate

class KotlinExampleDelegate: JavaDelegate {

    override fun execute(execution: DelegateExecution) {
        val someVar = execution.getVariable("someVariable")
        //some code
        execution.setVariable("otherVariable", "someValue")
    }
}

デリゲートでは、ビジネスロジック、統合、および心臓が望むすべてのものを記述できます。

ロジックを備えたビジネスコンポーネントの形でレイヤーを作成し、デリゲートをプロセスのフローへのリンクとしてのみ使用することで、コードとプロセスができるだけ混ざらないようにします。

ほとんどの場合、このアプローチは便利で、うまく機能します。プロセスとの相互作用は、DelegateExecutionを介して行われます。これにより、たとえば、コンテキスト、インシデントなどを操作できます。

それが私たちが欲しかったことですか？


当初、ツールを選択するときに、次の機能を備えたソリューションを探していました。

  • 障害が発生した場所から正確にプロセスを回復し、そのままで使用できることが望ましい。
  • 一般的にプロセスに何が起こるかを確認できるいくつかのGUI。
  • ロジックと統合だけでなく、プロセス自体の単体テストを作成する機能。
  • Java 8以降。
  • 開発されたコミュニティ。

Camundaはエラーの回復と分析で問題ありません。

読みやすいトレース、落下前にステップを実行する試行回数を設定する機能、落下時のカスタムハンドラー-たとえば、落下中にエンティティのステータスをエラーに変更したい場合。後者は、DefaultIncidentHandlerを実装するだけで簡単に実行できます。確かに、このハンドラーが機能する面白い瞬間があります。エラー回復コードは、プロセスステップに入るたびにトリガーされます。これはスーパーバグや問題だとは言えません。むしろ、開発時にこれを覚えて検討する必要があるだけです。

私たちはそれを次のように解決しました：

override fun resolveIncident(context: IncidentContext) {
        val incidentList = Context.getCommandContext().incidentManager.findIncidentByConfiguration(context.configuration)
        if (incidentList.isNotEmpty()) {
            //     
        }
}

CamundaにはGUIがあり、悪くありません。

ただし、プロセスバージョン間でのインスタンスの移行など、もう少し必要な場合は、すでに懸命に努力する必要があります。デフォルトのUIには最小限の機能しかありませんが、独自の管理パネルを作成できる非常に強力なRest APIがあります。

私たちが行ったのは、管理パネルのパスに沿ったものでした。私たちのビジネスプロセスアーキテクトは、すでに完了したプロセスの履歴を表示する機能、バージョン間の移行などの機能を含め、かなり短期間でそれを見落としました。

Camunda's Restは非常に強力であり、プロセスを使用してほぼすべてのことができます。たとえば、/ process-definition / key / aProcessDefinitionKey / startを使用して、次のような単純なリクエストでプロセスを開始できます。

{
  "variables": {
    "aVariable" : {
        "value" : "aStringValue",
        "type": "String"
    },
    "anotherVariable" : {
    "value" : true,
    "type": "Boolean"
    }
  },
"businessKey" : "myBusinessKey"
}

この例は、このAPIを使用するさまざまなケースの詳細な説明が含まれている公式ドキュメントから取られています。

ユニットテストでは、通常のJunitを使用します。さらに、プロセス自体をテストするためのかなり興味深いライブラリがあります-'org.camunda.bpm.extension'、名前： 'camunda-bpm-assert'。これにより、フロープロセスを検証するためのテストを記述できます。

多くの場合、コード内よりもフロー内のバグの問題を検索する方が難しいため、これは非常に便利です。このようなテストは、たとえば、不正確なリファクタリングから保護し、本当に何度か私たちを助けました。

多くのプロセスでKotlinを使用することでG8の必要性がなくなったため、Java 8の必要性は部分的になくなりました。 Kotlinはプロジェクトに非常によく適合しており、ビジネスロジックの記述のみに集中できます。信じがたいことですが、基本的にKotlinがクールさについて言っていることはすべて真実です。統合されたほとんどすべてのアプリケーションで知られている、多数のフィールドを持つエンティティは、それほど怖く見えなくなり、エンティティ間のマッピングがより読みやすくなりました。しばしば批判されているnullの安全性は実際に機能し、ほとんどの場合に役立ちます。

カムンダのコミュニティはかなり発達しています。これは、GitHubの新しいライブラリが常にテストとメトリックスに表示されているという事実によって証明されています。

CamundaがSpringと完全に統合されているのは素晴らしいことです。必要な依存関係、いくつかの注釈、およびいくつかの構成Beanを追加します-実際、それがすべての統合です！その結果、誰もが慣れている通常のSpringアプリケーションを記述して、ビジネスプロセスのフローを追加します。対話はJava APIを介して行われ、Javaコードからプロセスを操作できます。

たとえば、次の1つのコマンドだけでプロセスを開始できます。

runtimeService.startProcessInstanceByKey(
                    "MyTestProcess",
                    "MyBusinessKey",
                    mapOf(
                            "key1" to "value1",
                            "key2" to "value2",
                    )
)

ここで、MyTestProcessは、インスタンスではなく、プロセスのId-shnikです。MyBusinessKeyは、実行中のプロセスインスタンスの一意のキーです。通常、このフィールドにはビジネス値を使用します。これは、インスタンスと検索の間のナビゲーションを高速化するためです。

ほぼ同じように、「眠い」プロセスを起こすことができます。

私たちが遭遇した顕著なマイナスまたはいくつかの問題、特に思い出すことはできません。その結果、かなり短期間で、それは完全に機能するプロセスであり、安全に本稼働することがわかりました。他のプロセスがプラットフォームに実装されており、非常に成功しています。現在Camundaでは、約10万のプロセスが同時にスピンする約15のアプリケーションを起動しています。
