RATKing: New Campaign with Remote Access Trojans

In late May, we discovered a malware distribution campaign for the Remote Access Trojan (RAT) class, programs that allow attackers to remotely control an infected system.



The group under consideration was distinguished by the fact that it did not choose any specific RAT family for infection. In the attacks within the framework of the campaign several trojans were noticed at once (all in the public domain). With this trait, the group reminded us of the rat king - a mythical animal that consists of rodents with entwined tails.





The original is taken from the monograph by K. N. Rossikov ‚ÄúMice and mouse-like rodents, the most important in economic terms‚Äù (1908)



In honor of this creature, we named the group RATKing that we are considering. In this post, we will talk in detail about how attackers carried out the attack, what tools they used, and also share our thoughts regarding the attribution of this campaign.



Attack course



All attacks in this campaign were carried out according to the following algorithm:



  1. The user received a phishing email with a link to Google Drive.
  2. Through the link, the victim downloaded a malicious VBS script that prescribed a DLL to load the final payload into the Windows registry and ran PowerShell to execute it.
  3. The DLL library injected the final payload - in fact, one of the RAT used by cybercriminals - into the system process and wrote the VBS script into autorun to fix itself in the infected machine.
  4. The final payload was executed in the system process and gave the attacker the ability to control the infected computer.

Next, we will focus on the first three stages, since we are interested in precisely the delivery mechanism of malware. We will not describe in detail the mechanism of operation of the malware itself. They are widely available - either sold in specialized forums, or even distributed as open source projects - which means that they are not unique to the RATKing group.



Attack stages analysis



Step 1: Phishing Mailing



The attack began with the victim receiving a malicious email (attackers used different templates with text, one of the examples is given in the screenshot below). The message included a link to a legitimate repository drive.google.com



that allegedly led to a PDF download page.





An example of a phishing email



However, in fact, it was not a PDF document that was downloaded at all, but a VBS script.



When clicking on the link from the letter in the screenshot above, the file with the name was loaded Cargo Flight Details.vbs



. In this case, the attackers did not even try to disguise the file as a legitimate document.



At the same time, as part of this campaign, we discovered a script calledCargo Trip Detail.pdf.vbs



. It could already pass for a legitimate PDF, because by default Windows hides the file extension. True, in this case, the suspect could still be caused by its icon, which corresponded to the VBS script.



At this stage, the victim could recognize the fraud: just look for a second at the downloaded files. However, in such phishing campaigns, attackers often rely on an inattentive or rushing user.



Stage 2. VBS script operation



The VBS script, which the user could inadvertently open, registered the DLL library in the Windows registry. The script was obfuscated: the lines in it were written as bytes, separated by an arbitrary character.





An example of an obfuscated script



The deobfuscation algorithm is quite simple: every third character is excluded from the obfuscated string, after which the result is decoded from base16 to the original string. For example, from the value 57Q53s63t72s69J70r74e2El53v68m65j6CH6Ct



(highlighted in the screenshot above), a string was obtained WScript.Shell



.



To deobfuscate strings, we used a function in Python: 



def decode_str(data_enc):   
    return binascii.unhexlify(''.join([data_enc[i:i+2] for i in range(0, len(data_enc), 3)]))


Below on lines 9‚Äì10, the value is highlighted, during deobfuscation of which a DLL file was obtained. It was he who launched the next step using PowerShell.





A line with an obfuscated DLL



Each function in a VBS script was executed as the lines were deobfuscated.



After the script was launched, a function was called wscript.sleep



- with its help, deferred execution was performed.



Next, the script worked with the Windows registry. He used WMI technology for this. With its help, a unique key was created, and the body of the executable file was written into its parameter. The registry was accessed through WMI using the following command: 



GetObject(winmgmts {impersonationLevel=impersonate}!\\.\root\default:StdRegProv)




Record made in the registry with a VBS script

Examples of distributed malware with the same management server



Two things are noteworthy here.



Firstly, the fact that the attackers used several different RAT families at once. This behavior is not typical of well-known cybergroups, which often use approximately the same set of familiar tools.



Secondly, RATKing used malware, which are either sold in specialized forums for a small price, or even are open source projects.



A more complete list of the malware used in the campaign, with one important caveat, is provided at the end of the article.



About grouping



We cannot attribute the described malicious campaign to any known intruders. So far, we believe that these attacks were committed by a fundamentally new grouping. As we wrote at the beginning, we called it RATKing.



To create a VBS script, the group probably used a tool similar to the VBS-Crypter utility from the NYAN-x-CAT developer . This is indicated by the similarity of the script that this program creates with the script of attackers. In particular, they are both:



  • carry out deferred execution using the function Sleep



    ;
  • use WMI;
  • register the body of the executable file as a registry key parameter;
  • execute this file using PowerShell in its address space.


For clarity, compare the PowerShell command to run the file from the registry that the script created using VBS-Crypter uses: 



((Get-ItemPropertyHKCU:\Software\NYANxCAT\).NYANxCAT);$text=-join$text[-1..-$text.Length];[AppDomain]::CurrentDomain.Load([Convert]::FromBase64String($text)).EntryPoint.Invoke($Null,$Null);


with a similar command used by the attacker script: 



[System.Threading.Thread]::GetDomain().Load((ItemProperty HKCU:\/\/\/Software\/\/\/<rnd_sub_key_name> ).<rnd_value_name>);
[GUyyvmzVhebFCw]::EhwwK('WScript.ScriptFullName', 'rWZlgEtiZr', 'WScript.ScriptName'),0


Note that as one of the payloads, the attackers used another utility from NYAN-x-CAT - LimeRAT .



The addresses of C & C servers point to another distinguishing feature of RATKing: the group prefers dynamic DNS services (see the list of C&C in the table with IoC).
