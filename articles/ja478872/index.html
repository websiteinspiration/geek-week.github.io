DIY：倉庫監視を自動化する方法

X5は43の配送センターと4,029の自社トラックを管理しており、15,752の店舗で製品の供給を途切れることなく提供しています。この記事では、倉庫イベントを監視するためのインタラクティブなシステムをゼロから作成する経験を共有します。この情報は、幅広い製品を管理する数十の配送センターを持つ商社のロジスティクスに役立ちます。



原則として、ビジネスプロセスを監視および管理するためのシステムの構築は、メッセージとインシデントの処理から始まります。同時に、ビジネスイベントの発生そのものを自動化し、インシデントを記録する可能性に関連して、重要な技術的瞬間を逃しています。クラスWMS、TMSなどのほとんどのビジネスシステムには、独自のプロセスを監視するための組み込みツールがあります。ただし、異なるメーカーのシステムである場合、または監視機能が十分に開発されていない場合は、高価な改善を注文するか、追加の設定のために専門のコンサルタントに依頼する必要があります。

システムから指標を取得するために、情報源（表）の決定に関連するコンサルティングのほんの一部しか必要としないアプローチを検討してください。

当社の倉庫の特徴は、いくつかの倉庫管理システム（WMS Exceed）が同じロジスティクスコンプレックスで稼働しているという事実にあります。倉庫は、論理的にだけでなく、商品の保管のカテゴリ（乾燥、アルコール、冷凍など）に従って分類されます。単一のロジスティクスコンプレックス内には複数の個別の倉庫の建物があり、それぞれの操作は独自のWMSによって管理されています。



倉庫で発生するプロセスの全体像を形成するために、マネージャーは1日に数回各WMSのレポートを分析し、倉庫オペレーター（受信者、ピッカー、スタッカー）からのメッセージを処理し、情報ボードに表示する実際の稼働インジケーターを要約します。

管理者の時間を節約するために、倉庫イベントの運用管理のための安価なシステムを開発することにしました。新しいシステムは、倉庫プロセスの運用パフォーマンスの「ホット」インジケーターを表示するだけでなく、インシデントを修正し、タスクを監視するマネージャーが、特定のインジケーターに影響を与える原因を排除するのにも役立ちます。会社のITアーキテクチャの一般的な監査を実施した後、必要なシステムの特定の部分がすでに何らかの形で私たちのランドスケープに存在し、設定と必要なサポートサービスの両方の検査があることに気付きました。コンセプト全体を1つのアーキテクチャソリューションに縮小し、開発の範囲を評価するだけです。

新しいシステムを構築するために実行する必要がある作業量を評価した後、プロジェクトをいくつかの段階に分割することが決定されました。

最初の段階で、システムは準備された運用データのスライスをすべてのWMS複合システムから収集する必要があります。読み取りはほぼリアルタイムで行われます（間隔は5分未満）。重要なのは、システムをネットワーク全体に配備するときに、数十の倉庫のDBMSからデータを取得する必要があることです。取得された運用データは、システムコアロジックによって処理され、計画された指標からの偏差を計算し、統計を計算します。この方法で処理されたデータは、明確なグラフと図の形で、マネージャーのタブレットまたは倉庫の案内板に表示する必要があります。



最初の段階のパイロット実装に適したシステムを選択するときに、Zabbixを選択しました。このシステムは、倉庫システムのITパフォーマンスを監視するためにすでに使用されています。別のインストールを追加して、倉庫操作のビジネス指標を収集することにより、倉庫の状態の全体像を把握できます。

各WMSインスタンスは、監視システムのホストとして定義されています。メトリックは、準備されたSQLクエリを使用してスクリプトを実行することにより、データセンターネットワークの中央サーバーによって収集されます。データベースへの直接アクセスを推奨しないシステム（SAP EWMなど）を監視する必要がある場合は、ドキュメント化されたAPI関数へのスクリプト呼び出しを使用して、インジケーターを記述したり、単純なpython / vbascriptプログラムを記述したりできます。

Zabbixプロキシインスタンスは、メインサーバーからの負荷を分散するためにウェアハウスネットワークにデプロイされます。プロキシを通じて、すべてのローカルWMSインスタンスでの作業が提供されます。 Zabbixプロキシを備えたホスト上のZabbixサーバーによるパラメータの次のリクエストで、WMSデータベースからメトリックをリクエストするスクリプトが実行されます。

中央のZabbixサーバーに倉庫のグラフとインジケーターを表示するには、Grafanaをデプロイします。準備されたダッシュボードを倉庫のインフォグラフィックで出力することに加えて、Grafanaはインジケーターの偏差を制御し、自動アラートを倉庫サービスシステムに転送してビジネスインシデントを処理するために使用されます。

システムの主要コンポーネント（SQLcl、Zabbix、Grafana）のインストールと構成はさまざまなソースで説明されているため、ここでは繰り返さない。 SQLplusの代わりにSQLclを使用するのは、SQLcl（Javaで記述されたOracle DBMSコマンドラインインターフェース）ではOracleクライアントを追加でインストールする必要がなく、そのまま使用できるためです。

Zabbixを使用して倉庫のビジネスプロセスのパフォーマンスを監視する際に注意が必要な主なポイントと、それらを実装する方法の1つについて説明します。また、この投稿はセキュリティに関するものではありません。接続の安全性と提示された方法の使用には、パイロットソリューションを本稼働運用に移行するプロセスで追加の調査が必要です。

主なことは、そのようなシステムを実装するときに、システムが提供する設定を使用して、プログラミングなしで実行できることです。

Zabbix監視システムは、監視対象システムからメトリックを収集するためのいくつかのオプションを提供します。これは、制御されたホストの直接ポーリングと、低レベルの検出パラメーターの設定方法を含む、ホストのzabbix_senderを介してサーバーにデータを送信するより高度な方法の両方で実行できます。私たちの問題を解決するには、中央サーバーによる直接ホストポーリング方式が非常に適しています。これにより、メトリックの取得シーケンスを完全に制御できるようになり、制御対象の各ホストにそれらを配布する必要なく、設定/スクリプトの1つのパッケージを確実に使用できます。

システムのデバッグとチューニングの「実験的」として、WMSワークシートを受け入れ制御に使用します。

  1. 受け入れ時のTS、到着したすべて：「現在の時刻から-72時間」の期間のステータスを持つすべてのTS-SQLクエリ識別子：getCars。
  2. すべての車両ステータスの履歴：72時間到着したすべての車両のステータス-SQLクエリ識別子：carsHistory。
  3. 受入予定車両：ステータスが「予定」であるすべての車両のステータス。時間間隔は現在の時刻から「-24時間」と「+24時間」です-SQLクエリ識別子：carsIn。

したがって、一連のウェアハウスメトリックを決定したら、WMSデータベース用のSQLクエリを準備します。クエリを実行するには、メインデータベースではなく、その「ホット」コピー（スタンバイ）を使用することをお勧めします。

データ取得のためにスタンバイOracle DBMSに接続されています。テストベース192.168.1.106に接続するためのIPアドレス。接続パラメーターは、Zabbixサーバーの作業SQLclフォルダーのTNSNames.ORAに保存されます。

# cat  /opt/sqlcl/bin/TNSNames.ORA
WH1_1=
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = 192.168.1.106)(PORT = 1521))
    (CONNECT_DATA =
      (SERVER = DEDICATED)
      (SERVICE_NAME =  WH1_1)
    )
  )

これにより、ユーザー名/パスワードとデータベース名のみを指定して、EZconnectを介して各ホストに対してSQLクエリを実行できます。

# sql znew/Zabmon1@WH1_1

準備されたSQLクエリは、Zabbixサーバーの作業フォルダーに保存されます。 

/etc/zabbix/sql

サーバーのzabbixユーザーへのアクセスを許可します。

# chown zabbix:zabbix -R /etc/zabbix/sql

リクエストファイルは、Zabbixサーバーからアクセスするための一意の識別子名を受け取ります。 SQLclによる各データベースクエリは、いくつかのパラメーターを返します。クエリ内の1つのメトリックのみを処理できるZabbixの詳細を考慮して、追加のスクリプトを使用してクエリ結果を個別のメトリックに解析します。

メインスクリプトを準備しています。これをwh_Metrics.shと呼び、データベースへのSQLクエリを呼び出し、結果を保存し、データを取得するための成功指標を含む技術メトリックを返します。

#!/bin/sh 
##  </i>
export ORACLE_HOME=/usr/lib/oracle/11.2/client64
export PATH=$PATH:$ORACLE_HOME/bin
export LD_LIBRARY_PATH=$ORACLE_HOME/lib:/usr/lib64:/usr/lib:$ORACLE_HOME/bin
export TNS_ADMIN=$ORACLE_HOME/network/admin
export JAVA_HOME=/
alias sql="opt/sqlcl/bin/sql"
##      sql-    
scriptLocation=/etc/zabbix/sql
sqlFile=$scriptLocation/sqlScript_"$2".sql
##       
resultFile=/etc/zabbix/sql/mon_"$1"_main.log
##     
username="$3"
password="$4"
tnsname="$1"
##    
var=$(sql -s $username/$password@$tnsname < $sqlFile)
##       
echo $var | cut -f5-18 -d " " > $resultFile
##   
if grep -q ora "$resultFile"; then
    echo null > $resultFile
    echo 0
else
    echo 1
fi

Zabbix-proxy構成設定（デフォルトでは- / usr / local / share / zabbix / externalscripts）に従って外部スクリプトをホストするためのフォルダーに、スクリプトを含む完成したファイルを配置します。

スクリプトが結果を受け取るデータベースのIDは、スクリプトパラメータによって送信されます。データベース識別子は、TNSNames.ORAファイルの設定行に対応している必要があります。

SQLクエリの呼び出し結果は、mon_base_id_main.logという形式のファイルに保存されます。ここで、base_id =スクリプトパラメータとして取得されたDB識別子です。サーバーから複数のデータベースへ同時にリクエストがあった場合、データベース識別子による結果ファイルの分離が提供されます。クエリは、値の並べ替えられた2次元配列を返します。

次のスクリプト（getMetrica.shと呼びます）は、リクエストの結果を含むファイルから指定されたメトリックを取得するために必要です。

#!/bin/sh 
##      
resultFile=/etc/zabbix/sql/mon_”$1”_main.log
##      :
##    ,      (RSLT)   
## {1 1 2 2…}   ( IFS)
##         
IFS=’ ‘
str=$(cat $resultFile)
status_id=null
read –ra RSLT <<< “$str”
for i in “${RSLT[@]}”; do
if [[ “$status_id” == null ]]; then
status_id=”$I"
elif [[ “$status_id” == “$2” ]]; then
echo “$i”
break
else
status_id=null
fi
done

これで、Zabbixをセットアップして、倉庫受け入れプロセスのパフォーマンスの監視を開始する準備ができました。

Zabbixエージェントが各データベースノードにインストールおよび設定されます。

メインサーバーでは、すべてのサーバーをZabbixプロキシで定義します。設定については、次のパスに移動してください：

管理→プロキシ→プロキシの作成



制御されたホストの定義：

設定→ ホスト→ ホストの作成



ホスト名は、エージェント構成ファイルで指定されたホスト名と一致する必要があります。

ノードのグループ、およびデータベースのあるノードのIPアドレスまたはDNS名を示します。

メトリックを作成し、そのプロパティを指定します。

設定→ノード→ 'ノード名' →データ要素>データ要素の作成

1）データベースからすべてのパラメーターをリクエストするためのメインメトリックを作成し



、データ要素の名前を設定し、「外部チェック」のタイプを示します。 「キー」フィールドで、Oracleデータベースの名前、SQLクエリの名前、データベースに接続するためのログインとパスワードをパラメーターとして転送するスクリプトを定義します。リクエストの更新間隔を5分（300秒）に設定します。

2）各車両ステータスの残りのメトリックを作成します。これらのメトリックの値は、メインメトリックのチェック結果に基づいて形成されます。



データ要素の名前を設定し、「外部検証」のタイプを示します。 「キー」フィールドで、Oracleデータベースの名前とステータスコードを転送するスクリプトを定義します。その値はパラメーターとして追跡します。結果をファイルに書き込むことができるように、リクエストを更新する間隔をメインメトリック（310秒）よりも10秒長く設定しました。

メトリックを正しく取得するには、チェックがアクティブ化される順序が重要です。データ受信時の競合を回避するために、まず、スクリプトの呼び出しwh_Metrics.shでメインのGetCarsByStatusメトリックをアクティブにします。

設定→ノード→ 'ノード名'→データ要素→サブフィルター「外部チェック」。必要なチェックにマークを付け、「アクティブ化」をクリックします。



次に、1回の操作で残りのメトリックをアクティブにし、それらをすべて一緒に選択します。



Zabbixは現在、倉庫のビジネス指標の収集を開始しています。
