ロックフリープログラミングでは、相互に排他的なロックよりも低いレベルのツールを使用して、スレッド間の通信と同期を確立します。過去と今日の両方で、そのようなツールは多岐にわたります。大まかな歴史的順序では、明示的なバリア（明示的なフェンス/バリア-Linuxのmb（）など）、特別な順序付けAPI呼び出し（WindowsのInterlockedExchangeなど）、およびさまざまな種類の特別なアトミックタイプが含まれます。これらのツールの多くは退屈かつ/または複雑であり、それらの幅広い多様性は、結局、ロックフリーコードが異なる環境で異なる方法で記述されることを意味します。



ただし、過去数年にわたって、ハードウェアプロバイダーとソフトウェアプロバイダーの間で大きな収束がありました。コンピューティング業界は、標準として、またはメインのOS言語とプラットフォームを使用してロックフリーコードを記述する唯一の方法として、順番に合意された順序付けされた原子変数を中心に結束しています。一言で言えば、順序付けられた原子変数は2つの保証を提供するため、明示的なロックなしで複数のスレッドで同時に読み書きしても安全です。変数の読み取りと書き込みは、プログラムのソースコードに現れる順序で実行されることが保証されます。そして、すべての読み取りまたは書き込みは、「オールオアナッシング」のアトミックであることが保証されています。また、アトミックに実行されることが保証されているcompareAndSetなどの特別な操作もあります。順序付けされたアトミック変数とそれらを正しく使用する方法の詳細については、[1]を参照してください。

最適化について一言



順序付けされたアトミック変数が、コンパイラ、プロセッサ、キャッシング効果、およびランタイムの他の要素が実行できる最適化をどのように制限するかを見ていきます。それでは、最初にいくつかの基本的な最適化ルールを簡単に見てみましょう。



ほとんどすべての言語の基本的な最適化ルールは次のとおりです。コードの実行を並べ替える（「変換する」）最適化は、プログラムの意味を変更しない場合にのみ常に正当であり、プログラムはソースコードの実行と変換されたコードの実行の違いを判別できません。一部の言語では、これは「as-if」ルールとも呼ばれます。これは、変換されたコードが「あたかも」（あたかも）ソースコードが次の形式で実行された場合と同じ観察可能な効果があるためにその名前を取得しますもともと書かれていました。



このルールには2つの影響があります。1つ目は、最適化によって、以前は不可能であった結果を取得したり、言語のセマンティクスを含め、ソースコードが依存することを許可されていたりする保証に違反したりしてはならないということです。不可能な結果が出た場合、最終的にはプログラムとユーザーは間違いなくその違いに気付くことができます。これは、元の変換されていないコードを実行した場合とは異なります。



第二に、最適化により、可能な実行のセットを減らすことができました。たとえば、最適化により、潜在的な（ただし保証はされない）交替（命令の実行順序の変更-インターリーブ）が発生しないという事実につながる可能性があります。プログラムがまだそれらに依存することができないため、これは正常です。



順序付けされたアトミック変数と最適化



順序付けされたアトミック変数を使用すると、コンパイラ、プロセッサ、およびキャッシングシステムが実行できる最適化の種類が制限されます。[3]注目に値する2種類の最適化：



  • 順序付けされたアトミック読み取りおよび書き込み操作の最適化。
  • 隣接する通常の読み取りおよび書き込み操作の最適化。


まず、特定のストリーム内のすべての順序付けされたアトミック読み取りおよび書き込み操作は、ソースコードの順序で厳密に実行する必要があります。これは、順序付けされたアトミック変数の基本的な保証の1つだからです。それでもなお、一部の最適化、特にこのスレッドが常に非常に高速に実行され、ある時点で別のスレッドとの切り替えが行われないかのように同じ効果を持つ最適化を実行できます。



たとえば、aが順序付けられたアトミック変数である次のコードを考えてみます。 



a = 1;  	// A
a = 2;  	// B


コンパイラ、プロセッサ、キャッシュ、またはランタイムのその他の部分が、A行の冗長なエントリを除いて、上記のコードを次のコードに変換することは許可されますか？ 



// A ': OK:    A 
a = 2;  	// B


答えはイエスです。プログラムは違いを判別できないため、これは正当です。これは「あたかも」このスレッドが常に非常に高速に動作しているため、並行して実行されている他のスレッドが原則としてラインAとBを交互に使用して中間値を確認することはできません。[4]



同様に、aが順序付けられたアトミック変数であり、localが分割不可能なローカル変数である場合、変換することが許容されます 



a = 1;  	// C:   a
local = a;  // D:   a


に 



a = 1;  	// C:   a
local = 1;  // D': OK,  " "


からの読み取りを除外します。別のスレッドが同時にaに書き込もうとした場合でも、このスレッドは常に「あたかも」高速で動作していたため、他のスレッドが行CとDを交互に切り替えて値を変更できなかったため、ローカルに書き戻すことができませんでした。。



第2に、近くの通常の読み取りおよび書き込み操作は、順序付けられたアトミックな操作を中心に並べ替えることができますが、いくつかの制限があります。特に、[3]で説明されているように、通常の読み取りおよび書き込み操作は、順序付けられたアトミック読み取り（「後」から「前」）に関して上に移動できず、順序付けされたアトミック読み取り（「から」後へ）。つまり、コードの重要なセクションから外れる可能性があり、パフォーマンスの点でこれから利益を得るプログラムを作成できます。詳細については、[3]を参照してください。

このようなメモリの場所にある変数は、最適化できない変数です。コンパイラはそれらについてまったく想定できないためです。言い換えると、コンパイラーは、そのような変数が特定の型を持っている場合でも、通常の型システムに関与していないと言う必要があります。たとえば、プログラムの上記の例のメモリセルMまたはA1 / A2が「int」と宣言されている場合、これは実際に何を意味するのでしょうか。これが意味することができるほとんどは、それがintのサイズとレイアウトを持っているということですが、それがintのように動作することを意味することはできません-結局のところ、int書き込むときに自分自身を自動インクリメントしないでください。また、別のアドレスの別の変数に似たものに書き込むときに、不思議なことにその値を変更しないでください。



それらを読み書きするためのすべての最適化を無効にする方法が必要です。 ISO CとC ++には、最適化してはならない特別な変数であるvolatileであることをコンパイラーに伝える移植可能な標準的な方法があります。



Javaと.NETには同等の概念はありません。結局のところ、管理対象環境は、実行するプログラムの完全なセマンティクスを知っている必要があるため、「認識できない」セマンティクスでメモリをサポートしないことは当然のことです。ただし、Javaと.NETの両方が、管理対象環境を終了してネイティブコードを呼び出すための緊急ゲートウェイを提供します。JavaはJavaネイティブインターフェース（JNI）を提供し、.NETはプラットフォーム呼び出し（P /呼び出し）を提供します。ただし、JNI仕様[5]は、volatileについては何も述べておらず、Java volatileもC / C ++ volatileについても触れていません。同様に、P / Invokeのドキュメントでは、.NET 揮発性またはC / C ++ 揮発性との相互作用について言及していません。したがって、Javaまたは.NETの非最適化メモリ領域に適切にアクセスするには、C / C ++ volatileを使用するC / C ++関数を記述して、呼び出しコードに代わって必要な作業を実行し、揮発性メモリ（tつまり、揮発性のものを受け入れたり返したりしないでください）、JNIおよびP / Invokeを介してこれらの関数を呼び出します。



最適化できない変数と（非）最適化



特定のストリーム内の最適化不可能な変数のすべての読み取りおよび書き込み操作は、書き込まれたとおりに実行する必要があります。コンパイラーは変数の完全なセマンティクスとその値がいつどのように変化するかを知ることができないため、最適化はまったく許可されません。これらは、（順序付けされたアトミック変数と比較して）より厳密な式であり、ソースコードの順序でのみ実行する必要があります。



先ほど検討した2つの変換をもう一度考えますが、今回は順序付けされたアトミック変数aを最適化できない（C / C ++揮発性）変数vに置き換えます。

次に、隣接する通常の読み取りおよび書き込み操作についてはどうでしょう-最適化できないものを中心に並べ替えることはできますか？ C / C ++コンパイラーの実装は非常に異なり、すぐに統一に向けて動き出す可能性は低いため、今日、実用的な移植性のある答えはありません。たとえば、C ++標準の1つの解釈では、通常の読み取り操作は揮発性 C / C ++の読み取りまたは書き込みに対してどの方向にも自由に移動できますが、通常のレコードは揮発性の読み取りまたは書き込みに対してまったく移動できません。C / C ++- 揮発性の C / C ++を、順序付けされたアトミック操作よりも少なく制限しやすくします。一部のコンパイラプロバイダーはこの解釈をサポートしています。揮発性の読み書きをまったく最適化しないものもあります。さらに、独自のセマンティクスを持つ人もいます。
