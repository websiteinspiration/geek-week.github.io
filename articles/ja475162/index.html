Xマクロを最新のC ++コードで使用する

最近のC ++開発の傾向は、コード内のマクロを最大限拒否することを示唆しています。しかし、マクロがない場合もあり、特に醜い形では、マクロがないとさらに悪いため、実行できません。これと物語について。

ご存知のように、CおよびC ++のコンパイルの最初のステップは、マクロとプリプロセッサディレクティブをプレーンテキストに置き換えるプリプロセッサです。

これにより、たとえば次のような奇妙なことができます。

// xmacro.h
"look, I'm a string!"

// xmacro.cpp
std::string str = 
#include "xmacro.h"
;

プリプロセッサーが機能した後、この誤解は正しいコードに変わります。

std::string str =
"look, I'm a string!"
;

もちろん、このひどいヘッダーを他の場所に含めることはできません。そして、はい。このヘッダーを同じファイルに数回追加するため、＃pragmaは1回も、ガードも含めません。

実際に、マクロの助けを借りてさまざまなことを行うと同時に、ランダムな#includeを防ぐより複雑な例を書いてみましょう。

// xmacro.h
#ifndef XMACRO
#error "Never include me directly"
#endif
XMACRO(first)
XMACRO(second)
#undef XMACRO

// xmacro.cpp
enum class xenum {
    #define XMACRO(x) x,
    #include "xmacro.h"
};

std::ostream& operator<<(std::ostream& os, xenum enm) {
    switch (enm) {
        #define XMACRO(x) case xenum::x: os << "xenum::" #x; break;
        #include "xmacro.h"
    }
    return os;
}

これは依然として醜いですが、特定の魅力がすでに現れています。enumクラスに新しい要素を追加すると、オーバーロードされた出力ステートメントに自動的に追加されます。

ここでは、このメソッドの適用領域を公式化できます：1つのソースに基づいてさまざまな場所でコードを生成する必要性。

そして今、X-MacroとWindowsの悲しい物語。 Windowsパフォーマンスカウンターのようなシステムがあり、特定のカウンターをオペレーティングシステムに送信して、他のアプリケーションがそれらを取得できるようにします。たとえば、Zabbixは、パフォーマンスカウンターを収集および監視するように構成できます。これは非常に便利で、データのリターン/クエリでホイールを作り直す必要はありません。

新しいカウンターを追加することは、HANDLEカウンター= AddCounter（ "name"）のように見えると心から思いました。ああ、私がどれほど間違っていたか。

最初に、特別なXMLマニフェスト（例）を記述するか、Windows SDKのecmangen.exeプログラムを使用して生成する必要がありますが、何らかの理由でこのecmangen は Windows 10 SDKの新しいバージョンから削除されています。次に、XMLマニフェストに基づいてctrppユーティリティを使用して、コードと.rcファイルを生成する必要があります。言うまでもなくシステムに新しいカウンターを追加するには、専用のユーティリティのによってなされるLODCTR引数で私たちのXML-マニフェストを持ちます。

上記の要約：必要なカウンターを1つ追加するには：

  1. XMLマニフェストを変更する
  2. マニフェストに基づいて新しい.cおよび.rcプロジェクトファイルを生成する
  3. 新しいカウンターをインクリメントする新しい関数を書く
  4. カウンター値をとる新しい関数を書く

合計：4つの変更されたファイルをdiff-eで変更しました。これは、単一のカウンターとXMLマニフェストでの作業に悩まされ続けているためです。これは、plusコードの情報源です。これはマイクロソフトが提供するものです。

実際、発明されたソリューションは恐ろしいように見えますが、新しいカウンターの追加は1つのファイルの1行で正確に行われます。さらに、すべてがマクロと、残念ながら事前ビルドスクリプトを使用して自動的に生成されます。これは、XMLマニフェストがまだ必要なためですが、現在は主なものではありません。

私たちのperfcounters_ctr.hは、上記の例とほとんど同じに見えます。

#ifndef NV_PERFCOUNTER
#error "You cannot do this!"
#endif
...
NV_PERFCOUNTER(copied_bytes)
NV_PERFCOUNTER(copied_files)
...
#undef NV_PERFCOUNTER

前に書いたように、カウンターの追加はlodctr.exeを使用してXMLマニフェストをロードすることで行われます。私たちのプログラムからは、初期化と変更のみが可能です。

生成されたオープナーで対象となる初期化フラグメントは次のようになります。

#define COPIED_BYTES 0 //     0
#define COPIED_FILES 1 //     

const PERF_COUNTERSET_INFO counterset_info{
    ...
    2, //    XML- 
    ...
};

struct {
    PERF_COUNTERSET_INFO set;
    PERF_COUNTER_INFO counters[2]; //    
} counterset {
    counterset_info, { //    
        { COPIED_BYTES, ... },
        { COPIED_FILES, ... }
    }
}

合計：「カウンター名-増加するインデックス」という形式の対応が必要です。コンパイル段階では、カウンターの数を把握し、カウンターインデックスから初期化配列を収集する必要があります。ここで、Xマクロが役に立ちます。

カウンター名をその上昇インデックスに一致させるのは簡単です。

以下のコードは、内部インデックスが0から始まり、1ずつ増加する列挙型クラスになります。手で最後の要素を追加すると、保有している合計カウンターの数がすぐにわかります。

enum class counter_enum : int
{  
  #define NV_PERFCOUNTER(ctr) ctr,
  #include "perfcounters_ctr.h"
  total_counters
};

さらに、列挙型に基づいて、カウンターを初期化する必要があります。

static constexpr int counter_count = static_cast<int>(counter_enum::total_counters);

const PERF_COUNTERSET_INFO counterset_info{
    ...
    counter_count, 
    ...
};

struct {
    PERF_COUNTERSET_INFO set;
    PERF_COUNTER_INFO counters[counter_count];
} counterset {
    counterset_info, { //    
        #define NV_PERFCOUNTER(ctr) \
        { static_cast<int>(counter_enum::ctr), ... },
        #include "perfcounters_ctr.h"
    }
}

その結果、新しいカウンターの初期化は1行で済み、他のファイルに追加の変更を加える必要がありませんでした（以前は、再生成ごとに初期化時にのみ3つのコードが変更されていました）。

そして、カウンターをインクリメントするための便利なAPIを追加しましょう。精神の何か：

#define NV_PERFCOUNTER(ctr) \
inline void ctr##_tick(size_t diff = 1) { /*   counter_enum::ctr */ }
#include "perfcounters_ctr.h"

#define NV_PERFCOUNTER(ctr) \
inline size_t ctr##_get() { /*    counter_enum::ctr */ }
#include "perfcounter_ctr.h"

プリプロセッサは、美しいゲッター/セッターを生成します。これは、コードですぐに使用できます。次に例を示します。

inline void copied_bytes_tick(size_t diff = 1);
inline size_t copied_bytes_get();

しかし、まだ2つの悲しいことが残っています。XMLマニフェストと.rcファイルです（残念ながら、それは必要です）。

非常にシンプルにしました。カウンターを定義するマクロを使用して元のファイルを読み取り、「NV_COUNTER（ "と"） "の間の内容を解析する事前ビルドスクリプトです。これに基づいて、.gitignoreにある両方のファイルを生成します。差分を詰まらせないでください。

それは：XMLマニフェストに基づいた特別なソフトウェアで生成されたコーディングコードです。カウンターの追加/削除ごとにプロジェクトに多くの変更が加えられました。

それはだった：プリプロセッサと事前に作成スクリプトがすべてのカウンタ、XML-マニフェストと.RCファイルを生成します。カウンタを追加/削除するためのdiff-eの正確な1行。この問題の解決を助けてくれたプリプロセッサのおかげで、この特定のケースでは害よりも多くの利益を示しました。
