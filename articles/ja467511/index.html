AudioSwitcher-自動化する必要がないように見えるものの自動化


序文


私は自宅にソビエトの優れたスピーカーシステムをいくつか持っています。しかし、この手法は非常に古く、リモートコントロールからまたは自動的にオンにすることはできませんが、常にサウンドアンプに近づき、怠惰にオン/オフするだけです。私はこの問題を解決しました。当初、Arduinoが購入され、プロジェクトはその上で行われましたが、仕事の質は私には合わず、プロジェクトはSTM32F103C8でやり直されました。その結果、オーディオ入力が4つ、オーディオ出力が1つ、入力が220V、出力が220Vのデバイスを入手しました。少なくとも1つのアクティブなオーディオ入力がある場合、220 V出力に電圧が現れ、サウンドアンプが含まれ、アクティブなオーディオチャネルが出力に送信されます。

開発の課題


単純に見えるかもしれません。ADCが0を受信しない場合、チャネルをアクティブと見なします。ほとんどすべてですが、オーディオ信号のソースをオンにしてサウンドをオフにした場合にのみ機能します。オフにすると、デバイスの電源が完全にオフにならないため、デバイスによって干渉が異なります。また、音源が悪いと、マイクロコントローラーは、音がオフになったときにノイズを拾う可能性があり、非常に強力です。そして、これは正確にソース干渉であり、STMkaは私の外部オーディオカードに干渉を見ません。さらに、それからの静かな音は0です。

それを自分で行う方法は？


まず、必要なものを決定しましょう。コストは書きません。それはあなたの場所に大きく依存します。

私たちの必要なもの：

  • プリント回路基板
  • プログラマーST-Link v2
  • 1チップSTM32f103C8
  • 入力オーディオチャンネルを出力に切り替える4つのリレー
  • アンプをオンにするために220Vを切り替えるための1つのリレー
  • AC-DC降圧コンバーター220V-5V（古い電話充電から取得できます）
  • 私たちのデバイスとアンプに電流を供給するための電源ケーブルとコネクタ
  • ソケット
  • 抵抗器、コンデンサー、その他の小さなもの

当然、オーディオワイヤとソケット付きのミニジャックプラグが必要です。

リレーの選択に焦点を当てたい... 220Vリレーの選択ですべてが明確な場合、220V AC電圧を「切り替え」、3.3Vで制御できるはずです。サウンドリレーの選択はそれほど単純ではありません。すべてのリレーがソリッドステートリレーであっても、出力に干渉を与えるわけではありません。これは私たちにとって非常に重要です。私はミンスクに住んでいて、適切なものを手頃な価格で見つけることができなかったため、4つのPVT322Aリレーを有名な中国の店に注文しました。おそらくあなたの地域ではもっと安いものを見つけることができます。

スキームと配線

pdf GitHub



開始以来、ハードウェア機能の調査を続けます。Eagleフォルダーのリポジトリーにある図で、リレー用の電流制限抵抗（R4-7）を選択する必要があります。私の場合、それは30オームです。L1コイルもあります。高周波ノイズを平滑化するフィルターを選択してください。

PCBWAYまたはJLCPCBでプリント回路基板を注文できます。彼らの価格は低く、私はJLCPCBに注文し、彼らは私にたった2ドルを請求した。プリント基板を注文するときは、ガーバーファイルが必要です。同じフォルダからすべてを見つけるか、自分で生成できます。

ソフトウェア部分に移りましょう


プログラマーをコンピューターに接続する方法、プログラミング環境とドライバーをインストールする方法については触れません。これらの指示はたくさんあり、それらは非常にアクセスしやすいです。私の回路では、プログラマー向けの出力が提供されています。Visual Studio 2017 + VisualGDBを使用しました。同じリポジトリからプロジェクトをダウンロードした後、プロジェクトを開くことができます。すぐにSettings.cppファイルに切り替えます。

Settings.cpp
#define DEBUG0 0//init USART and send all measurement values
#define DEBUG1 1//init USART and send information about recognition music or not
#define DEBUG2 0//just init USART
#define MaxEqualToZeroValue 3 //the value which equal or less is equated to zero
#define MaxAvarageForNoise (float)0.4//this is max avarage of measurement values so that the sound is considered noise for NOT active channel
#define MaxAvarageForActiveNoise (float)0.06//this is max avarage of measurement values so that the sound is considered noise for active channel
#define CountOfConsecutiveZeroValueForNoise 250//if count of consecutive zero values bugger it that sound is equated to noise
#define MinCountOfZeroValue 550//it's minimum count of zero values to equate to music(not consecutive)
#define USE_LED 1
#define LED_GPIO_PERIPH RCC_APB2Periph_GPIOC
#define LED_GPIO_GROUP GPIOC
#define LED_GPIO_PIN GPIO_Pin_13
#define USE_AMP 1
#define AMP_GPIO_PERIPH RCC_APB2Periph_GPIOB
#define AMP_GPIO_GROUP GPIOB
#define AMP_GPIO_PIN GPIO_Pin_12


このファイルのすべての設定は文書化されていますが、各設定については詳しく説明します。

#define DEBUG0 0
#define DEBUG1 1
#define DEBUG2 0

ユニットをDEBUG0に割り当てると、デバイスは、UARTを使用してSerialPortPlotterが「ダイジェスト」できる形式でオーディオ入力から受信した値を出力することを除いて、何もしないで停止します。

ユニットDEBUG1を割り当てると、デバイスはすでに完全に動作していますが、UARTでの作業に関する情報が少し表示されます。これらはすべて、デバッグにのみ必要です。

DEBUG2を割り当てると、UARTの初期化のみが行われます。理由がわからない場合は、次のことは行わないでください:-)

#define MaxEqualToZeroValue 3

次に、対応する値以下がゼロと見なされるパラメーターがあります。先に述べたように、いくつかの音源は低品質で非常にうるさいです。

#define MaxAvarageForNoise (float)0.4

オーディオチャネルが現在非アクティブ（つまり、現在出力に切り替えられていないチャネル）であり、このチャネルの1つの測定サイクルの平均測定値がこのパラメーターの値よりも小さい場合、チャネルは音声なしと見なされます。

#define MaxAvarageForActiveNoise (float)0.06

このパラメーターは、前のパラメーターとほとんど同じですが、現在アクティブなチャネルのみが対象です。実際には、チャネルがアクティブでアンプが動作している場合、オーディオチャネルの電圧が低下します。また、この設定を無視すると、ワイヤーがどのデバイスにも接続されていなくても、デバイスは音があると見なします。

#define CountOfConsecutiveZeroValueForNoise 250

このパラメーターは、CPU消費を最適化するためだけのものです。デバイスが連続して所定の数のゼロを満たす場合、この信号は音ではないと見なされます。

#define MinCountOfZeroValue 550

そして、これは重要な設定です。一部のデバイスでは、電源を切ると奇妙なノイズが発生しますが、それらの共通の要因の1つを強調しました。それらのデバイスがゼロになることはほとんどありません。そのため、このパラメーターを入力する必要がありました。1つの測定サイクルのゼロ値の数が指定された値より少ない場合、信号はノイズと見なされます。

#define USE_LED 1
#define LED_GPIO_PERIPH RCC_APB2Periph_GPIOC
#define LED_GPIO_GROUP GPIOC
#define LED_GPIO_PIN GPIO_Pin_13
#define USE_AMP 1
#define AMP_GPIO_PERIPH RCC_APB2Periph_GPIOB
#define AMP_GPIO_GROUP GPIOB
#define AMP_GPIO_PIN GPIO_Pin_12

このブロックは、すでにマイクロコントローラをプログラミングしたことがある人にとっては非常に理解しやすいものです。彼は、LEDが配置されるピンとアンプ制御リレーへの出力を選択します。私のスキームを変更しない場合は、これらのパラメーターは必要ありません。

次の設定に移りましょう：

main.cppファイルを開くことにより、関数の最初に
int main()
一連の変数の定義があります。

これについて詳しく説明します。マイクロコントローラのハードウェア構成を担当する多くのパラメータがあります。それらには触れません。

const uint8_t channelsCount = 2;

使用する音声入力チャンネルの数です。

const uint8_t countOfIterationsForSwitch = 5;

アクティブ/パッシブ状態を変更するために必要な測定サイクルの数。

const uint8_t ADCSampleTime = ADC_SampleTime_239Cycles5;

このパラメータは、測定の品質を左右します。最大に設定されています。変更することはお勧めしません。

const uint16_t measurementsDuration = 2000;

これは、1つの測定サイクルが実行されるミリ秒単位の時間です。

const uint32_t measurementFrequencies[] = { 1000, 1000, 1000, 1000 };

理由はわかりませんが、チャンネルごとに異なる周波数の入力チャンネルを測定できる機能を実装しました。多分誰かがこの機能を必要とするでしょう。

結論


以上です。必要な設定をすべて説明しました。回路を組み立て、プロジェクトをコンパイルし、ファームウェアをマイクロコントローラーに入れて喜ぶだけです。

結論として、入力オーディオワイヤーを何も「動かない」状態のままにしておくことはできません。すべての端子が相互接続されているデバイスに挿入するか、ミニジャックジャックの形で差し込む必要があります。


音源がかなり良い場合は、低い設定を設定できますが、サウンド状態を切り替えるには、（コンセントからではなく）オフにする必要がある場合があります。多分いつか私はケースの3Dモデルへのリンクを追加しますが、今のところ私は3Dプリンターを持っていません、そして、ケースは現在そのようなものです。しかし、これは今のところのみです：3Dプリンターはすでに使用中です:-)
