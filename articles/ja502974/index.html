C＃.NETでの実装

こんにちは、ハブル！コース「C＃ASP.NET Core Developer」の開始を見越して、C＃でのキャッシュの実装に関する興味深い資料の翻訳を用意しました。読書を楽しむ。



ソフトウェア開発で最も一般的に使用されるパターンの1つはキャッシングです。シンプルでありながら、非常に効果的なコンセプトです。アイデアは、実行された操作の結果を再利用することです。時間のかかる操作の後、結果をキャッシュコンテナーに保存します。次にこの結果が必要になったときに、面倒な操作を再度実行する必要がなく、キャッシュコンテナーから結果を抽出します。

たとえば、ユーザーのアバターを取得するには、データベースから要求する必要がある場合があります。呼び出しごとにリクエストを実行する代わりに、このアバターをキャッシュに保存し、必要になるたびにメモリから抽出します。

キャッシュは、頻繁に変更されないデータに最適です。または、理想的には、それらは決して変更されません。現在の時刻など、絶えず変化するデータはキャッシュしないでください。キャッシュすると、誤った結果が得られる危険性があります。

ローカルキャッシュ、永続的なローカルキャッシュ、および分散キャッシュ


キャッシュには3つのタイプがあります。

  • インメモリキャッシュは、1つのプロセスでキャッシュを実装する必要がある場合に使用されます。プロセスが停止すると、キャッシュも一緒に停止します。複数のサーバーで同じプロセスを実行している場合は、サーバーごとに個別のキャッシュがあります。
  • 永続的なインプロセスキャッシュ -これは、プロセスメモリ外のキャッシュをバックアップする場合です。ファイルまたはデータベースに配置できます。メモリ内のキャッシュよりも複雑ですが、プロセスが再起動した場合、キャッシュはフラッシュされません。キャッシュされたアイテムの取得にコストがかかり、プロセスが頻繁に再起動する傾向がある場合に最適です。
  • 分散キャッシュ -これは、複数のマシンに共有キャッシュが必要な場合です。通常、これらは複数のサーバーです。分散キャッシュは外部サービスに保存されます。つまり、1つのサーバーがキャッシュ要素を保持している場合、他のサーバーもそれを使用できます。Redisのようなサービスはこれに最適です。

ローカルキャッシュ についてのみ説明します。

プリミティブ実装


Cで非常に単純なキャッシュ実装を作成することから始めましょう。

public class NaiveCache<TItem>
{
    Dictionary<object, TItem> _cache = new Dictionary<object, TItem>();
 
    public TItem GetOrCreate(object key, Func<TItem> createItem)
    {
        if (!_cache.ContainsKey(key))
        {
            _cache[key] = createItem();
        }
        return _cache[key];
    }
}

使用：

var _avatarCache = new NaiveCache<byte[]>();
// ...
var myAvatar = _avatarCache.GetOrCreate(userId, () => _database.GetAvatar(userId));

この単純なコードは重要な問題を解決します。ユーザーのアバターを取得するには、最初のリクエストのみがデータベースからの実際のリクエストになります。byte []リクエストの結果によるアバターデータ（）は、プロセスメモリーに保存されます。後続のすべてのアバター要求は、それをメモリから取得し、時間とリソースを節約します。

しかし、プログラミングにおけるほとんどのことのように、物事はそれほど単純ではありません。上記の実装は、いくつかの理由で良い解決策ではありません。一方、この実装はスレッドセーフではありません。複数のスレッドから使​​用すると、例外が発生する場合があります。さらに、キャッシュされたアイテムはメモリに永久に残りますが、これは実際には非常に悪いことです。

これが、キャッシュからアイテムを削除する必要がある理由です。

  1. キャッシュは大量のメモリを消費し始める可能性があり、最終的には不足とクラッシュのために例外が発生します。
  2. メモリの消費量が多いと、メモリが圧迫される可能性があります（GC圧迫とも呼ばれます）。この状態では、ガベージコレクターは必要以上に機能するため、パフォーマンスが低下します。
  3. データが変更されると、キャッシュの更新が必要になる場合があります。キャッシングインフラストラクチャはこの機能をサポートする必要があります。

これらの問題を解決するには、強制排除ポリシーのフレームワークに存在します（ポリシーの削除-立ち退き/削除ポリシーとも呼ばれます）。これらは、特定のロジックに従ってキャッシュからアイテムを削除するためのルールです。一般的な削除ポリシーには次のものがあります。

  • 一定の期間が経過した後でも、アイテムをキャッシュから削除する絶対有効期限ポリシー。
  • 一定期間アクセスされなかったアイテムをキャッシュから削除するSliding Expirationポリシー。つまり、有効期限を1分に設定した場合、30秒ごとに使用する間、アイテムはキャッシュに残ります。1分以上使用しないと、アイテムは削除されます。
  • サイズ制限ポリシー。キャッシュのサイズを制限します。

必要なものがすべてわかったところで、より良いソリューションに移りましょう。

gcへの圧力の最初の問題に関して：いくつかの方法とヒューリスティックによってgcへの圧力を制御することが可能です。この投稿はこれに関するものではありませんが、私の記事「C＃.NETでのメモリリークの発見、修正、防止：8つのベストプラクティス」を読んで、いくつかの便利な方法について学ぶことができます。

2番目の問題は簡単に解決できます。実際、MemoryCacheこれはそれを完全に解決する実装です：

public class WaitToFinishMemoryCache<TItem>
{
    private MemoryCache _cache = new MemoryCache(new MemoryCacheOptions());
    private ConcurrentDictionary<object, SemaphoreSlim> _locks = new ConcurrentDictionary<object, SemaphoreSlim>();
 
    public async Task<TItem> GetOrCreate(object key, Func<Task<TItem>> createItem)
    {
        TItem cacheEntry;
 
        if (!_cache.TryGetValue(key, out cacheEntry))//    .
        {
            SemaphoreSlim mylock = _locks.GetOrAdd(key, k => new SemaphoreSlim(1, 1));
 
            await mylock.WaitAsync();
            try
            {
                if (!_cache.TryGetValue(key, out cacheEntry))
                {
                    //    ,   .
                    cacheEntry = await createItem();
                    _cache.Set(key, cacheEntry);
                }
            }
            finally
            {
                mylock.Release();
            }
        }
        return cacheEntry;
    }
}

使用：

var _avatarCache = new WaitToFinishMemoryCache<byte[]>();
// ...
var myAvatar =
await _avatarCache.GetOrCreate(userId, async () => await _database.GetAvatar(userId));

この実装では、要素を取得しようとしたときに、同じ要素がすでに別のスレッドによって作成されている場合、最初のスレッドが完了するまで待機します。次に、別のスレッドによって作成された、すでにキャッシュされたアイテムを取得します。

コード解析


この実装は、要素の作成をブロックします。ロックはキーで発生します。たとえば、Alexeyのアバターを待っている場合でも、ZhenyaまたはBarbaraのキャッシュされた値を別のスレッドで取得できます。

辞書_locksはすべてのロックを保存します。通常のロックはで機能しないasync/awaitため、SemaphoreSlimを使用する必要があります。

ifの値がすでにキャッシュされているかどうかを確認する2つのチェックがあります(!_Cache.TryGetValue(key, out cacheEntry))。ロックされているのは、要素の唯一の作成を提供するものです。最適化のために、ロックの外側にあるもの。

いつ使うか WaitToFinishMemoryCache


この実装には明らかにオーバーヘッドがあります。それが関連する場合を見てみましょう。

次のWaitToFinishMemoryCache場合に使用：

  • アイテムの作成時間になんらかの価値があり、作成数をできるだけ最小限にしたい場合。
  • アイテムを作成する時間が非常に長い場合。
  • アイテムをキーごとに1つずつ作成する必要がある場合。

次のWaitToFinishMemoryCache場合は使用しないでください。

  • 複数のスレッドが同じキャッシュ要素にアクセスする危険はありません。
  • 要素を複数回作成することに断固として反対しているわけではありません。たとえば、データベースへの1つの追加クエリが何にも大きな影響を与えない場合です。

概要


キャッシングは非常に強力なパターンです。また、これも危険であり、落とし穴があります。キャッシュしすぎると、GCに圧力がかかる可能性があります。キャッシュが少なすぎると、パフォーマンスの問題が発生する可能性があります。また、探索するまったく新しい世界を表す分散キャッシングもあります。これはソフトウェア開発であり、習得できる何か新しいものが常にあります。
