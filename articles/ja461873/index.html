ViewPager 2-古いラッパーの新機能

ViewPagerは、Androidサポートライブラリで最も有名で広く使用されているコンポーネントの1つです。最も単純なカルーセル、オンボーディング、スライダーはすべてその上に作られています。2019年2月、AndroidX開発チームはViewPager2をリリースしました。これらの前提条件が何であったか、コンポーネントの更新バージョンが持つ利点を見てみましょう。



ViewPager 2


この投稿の執筆時点（2019年7月）では、ViewPager2のベータ版が利用可能です。これは、下記の問題を修正し、機能を改善および拡張できることを意味します。開発者は将来、TabLayoutのサポートを追加し（最初のバージョンでのみ機能します）、アダプターのパフォーマンスを最適化し、多くの小さな修正を行い、ドキュメントを完成させることを約束します。

統合


コンポーネントは標準パッケージには付属していませんが、個別に接続されています。これを行うには、モジュールのgradleスクリプトの依存関係ブロックに次の行を追加します：

implementation "androidx.viewpager2:viewpager2:1.0.0-beta02"

実装


良い知らせから始めましょう。最初のバージョンから2番目のバージョンへの移行はできるだけ単純で、要約するとインポートの変更です。古き良き構文には触れていません。getCurrentItem（）メソッドは現在のページを返します。ViewPager2.onPageChangeCallbackを使用すると、ページャーの状態の変更をサブスクライブできます。アダプターは、setAdapter（）を介して引き続きインストールされます。


最初と2番目のページャーにはインターフェース以外に共通点がないことが明らかになるので、深く掘り下げる価値があります。setAdapter（）メソッドの実装に慣れていれば、疑いの余地はありません。

はい、ViewPager2はRecyclerViewの単なるラッパーです。一方、これは大きなプラスであり、もう一方は頭痛の種です。PagerSnapHelperの登場により、RecyclerViewをリーフレットとして偽装することが可能になりました。このクラスは、スクロールの物理を変更します。ユーザーが指を離すと、PagerSnapHelperはリストの中心線に最も近いリストアイテムを計算し、滑らかなアニメーションで正確に中央に配置します。したがって、スワイプが十分に鋭い場合、リストは次の要素にスクロールします。それ以外の場合は、アニメーションが元の状態に戻ります。

PagerSnapHelperを使用する場合は、RecyclerView自体とそのすべてのViewHolderの幅と高さがMATCH_PARENTに設定されていることを確認してください。そうしないと、SnapHelperの動作が予測できなくなり、完全に予期しない場所でバグが発生する可能性があります。これにより、高さの小さい要素のカルーセルを作成することは可能ですが、かなり時間がかかります。

注目に値するのは、setPageTransformer（） メソッドです。これは、パラメーターとしてViewPager2.PageTransformerを取ります。これは、設定したコールバックを各ページの選択イベントのために、このページのために、独自のアニメーションを設定するのに役立ちます。コールバックは、現在のページのビューとその番号を入力として受け取ります。この方法に最も近い類似体であるItemAnimatorからRecyclerView。

これで楽しみの終わりではありません。 1つの更新で、ViewPager2はItemDecorationのサポートを受け取りました。子Viewを装飾するためのヘルパークラスです。このメカニズムは、要素、境界線、セルの強調表示の間のセパレーターを描画するために使用できます。

何年もの間、通常のRecyclerViewでの作業にうまく使用されてきたため、デコレーターの既製の実装はすでにたくさんあります。すべての開発がポケットベルに適用できるようになりました。箱から出して、ページャー要素セパレーターの標準実装が利用可能です：

        DividerItemDecoration(this, RecyclerView.HORIZONTAL)
)

2019年5月の次の更新に伴い、ViewPager2には別の重要なメソッドであるsetOffscreenPageLimit（Int）が追加されました。彼は中央の右側と左側にいくつの要素がページャーで初期化されるかを担当しています。RecyclerViewはデフォルトでビューのキャッシュと表示を担当しますが、このメソッドを使用すると、ロードするアイテムの希望数を明示的に設定できます。

確かにあなたは疑問を持つでしょう：正常に機能している最初のバージョンがあるのに、なぜフラグメントとそれらのためのアダプターを備えたセカンドページャーを使うのですか？大規模な動的データリストを操作する場合、ViewPagerは「特効薬」とは言えません。写真やバナーの静的なセットでカルーセルを作成するのに最適ですが、広告投稿の読み込み、フィルタリングを備えたページ付けされたニュースフィードは、ハードサポートされた醜いモンスターを生み出します。遅かれ早かれ、あなたはきっとRecyclerViewですべてを書き直したいという燃える欲望に出くわすでしょう。ページャー自体がページャーに変わったので、これを行う必要はありません。ダイナミックリストを処理するための強力な機能を利用しながら、それらを通常の構文でラップします。

PagerAdapter が提供できる唯一のものは、notifyDataSetChanged（）メソッドです。これにより、ViewPagerはすべてのレンダリングされたリスト項目を強制的に再描画します。既存の要素の位置のリストを格納し、それらの要素のgetItemPosition（）メソッドからPOSITION_UNCHANGEDを返すことを誰も妨げていないことに気づくかもしれません。それだけです。ただし、このソリューションは美しいとは言えず、煩雑であり、さらに、リストの要素が絶えず変化し、最後に追加されるだけではない場合に拡張することは困難です。FragmentStateAdapterには、RecyclerView.Adapterメソッドの完全な武器があります。ので、要素を再描画するためのロジックをより柔軟に構成できます。また、一緒にFragmentStateAdapter、あなたは使用することができますDiffUtilあなたはほぼ完全に変更の通知の作業を自動化することができます。

ViewPager2が 登場する主な理由は、ホイールを再発明することをためらうことです。一方では、AndroidX開発チームはすでに時代遅れのViewPagerを放棄する準備ができており、その機能の拡張に投資するつもりはありません。はい、そしてなぜですか？結局のところ、RecyclerViewはすでに必要なすべてを知っています。一方、このような広く使用されているコンポーネントのサポートを削除して中止しても、コミュニティの忠誠心が高まることはありません。

要約すると、ViewPager2は注目に値しますが、現時点では重大な欠陥がないわけではありません。
