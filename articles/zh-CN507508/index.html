如何在旧版项目中嵌入静态代码分析器而不影响命令











尝试使用静态代码分析器很容易。但是要实现它，尤其是在大型的老项目的开发中，则需要技巧。如果使用错误的方法，分析器可能会增加工作量，减慢开发速度并使团队失去动力。让我们简短地讨论将静态分析集成到开发过程中并作为CI / CD的一部分开始使用的正确方法。



介绍



最近，出版物“ 静态分析入门而又不压倒团队 ” 吸引了我的注意力。一方面，这是一篇不错的文章。另一方面，在我看来，它从未听过如何将静态分析轻松地引入具有大量遗留代码的项目中的完整答案。文章说，您可以忍受技术债务，只能使用新代码，但是以后没有解决该技术债务的方法的答案。



我们的PVS-Studio团队对此主题有自己的看法。让我们看看引入静态代码分析器的问题通常是如何产生的，如何克服这个问题以及如何不费力地逐步消除技术负担。



问题



通常，运行并查看静态分析器的工作原理并不困难[ 1 ]。您会在代码中看到有趣的错误，甚至看到潜在的可怕漏洞。您甚至可以修复某些问题，但随后许多程序员放弃了。



所有静态分析仪都会给出误报。这是静态代码分析方法的功能，因此无法采取任何措施。在一般情况下，这是一个无法解决的问题，赖斯定理[ 2 ] 证实了这一点。机器学习算法也无济于事[ 3 ]。如果甚至一个人都不能总是说出一个或另一个代码是错误的，那么您就不应该在程序中指望这一点了。



如果已经配置了静态分析器，则错误警报不是问题：



  • 禁用的规则集被禁用；
  • ;
  • C C++, , , - , ;
  • , ( memcpy printf) [4];

在这种情况下，可以期望假阳性率较低，约为10-15％[ 5 ]。换句话说，每10个分析仪警告中有9个将指示代码中的实际问题，或者至少表明“强烈气味代码”。您必须承认这种情况非常令人愉快，并且分析器是程序员的真正朋友。













实际上，在一个大型项目中，初始图景将完全不同。分析器会在旧代码上生成数百或数千个警告。不可能快速了解案件中的哪些警告，哪些不是。坐下来并开始处理所有这些警告是不合理的，因为这种情况下的主要工作将停止数天或数周。通常，团队无法承受这种情况。还有大量差异会破坏变更的历史。快速大量编辑代码中的许多片段将不可避免地导致新的错别字和错误。



最重要的是，在与警告的斗争中取得如此壮举毫无意义。同意由于该项目已经成功运行了许多年，所以其中的大多数关键错误已经得到修复。是的，这些修复非常昂贵，我必须进行调试，得到有关错误的负面用户反馈，等等。静态分析器即使在编写代码的阶段，也可以快速，廉价地修复许多这些错误。但是目前，这些错误已被固定，分析器主要检测旧代码中的非关键错误。此代码可能无法使用，很少使用，并且其中的错误可能不会导致明显的后果。也许某个地方的按钮阴影不是正确的颜色，但这不会打扰任何人使用该产品。



当然，即使是非严重错误也仍然是错误。有时，真正的漏洞可能隐藏在错误的背后。但是，放弃一切并处理几天/几周以来表现不佳的缺陷似乎是一个可疑的想法。



程序员看了看旧的工作代码中的所有这些警告……然后他们认为：我们可以不进行静态分析。让我们更好地编写新的有用功能。



以他们自己的方式，他们是对的。他们认为，对于初学者来说，他们必须以某种方式摆脱所有这些警告。只有这样，他们才能从代码分析器的常规使用中受益。否则，新警告只会陷入旧警告中，而没有人会注意它们。



这与编译器警告类似。并非没有道理，他们建议将编译器警告的数量保持为0。如果有1000条警告，那么当它们变为1001时，没有人会注意它，并且不清楚在哪里可以找到最新的警告。













这个故事中最糟糕的事情是，如果此时此刻最重要的人强迫您使用静态代码分析。这只会使团队失去动力，因为从他们的角度来看，只会增加其他官僚主义的复杂性。没有人会查看分析仪的报告，并且所有使用都只能是“纸上记录”。那些。正式地，分析已内置到DevOps流程中，但实际上，没有人从中受益。在会议参观者的展台上交流时，我们听到了详细的故事。这样的经历会永久（即使不是总是）阻止程序员接触静态分析工具。

解决现有技术债务并使用新警报



现代的商用静态分析仪仅允许您研究出现在新代码或修改代码中的新警告。这种机制的实现是不同的，但本质是相同的。在PVS-Studio静态分析仪中，此功能的实现如下。



为了快速开始使用静态分析，我们为PVS-Studio用户提供了使用批量警报抑制机制[ 6]。总体思路如下。用户启动了分析仪并收到许多警告。由于已经开发了多年的项目已经存在，可以发展并且可以带来收益，因此报告中很可能不会出现很多警告，表明存在严重缺陷。换句话说，关键错误已经以一种或另一种更昂贵的方式得到了修复，或者要归功于客户的反馈。因此，分析仪现在发现的所有东西都可以视为技术债务，试图立即消除是不切实际的。



您可以告诉PVS-Studio将这些警告视为无关紧要的（将技术债务推迟到以后），并且不再显示它们。分析器创建一个特殊的文件，在其中存储有关到目前为止无趣的错误的信息。现在，PVS-Studio将仅在新的或更改的代码上发出警告。此外，所有这些都可以巧妙地实现。例如，如果在源文件的顶部添加了空行，则分析器将理解为实际上没有任何更改，并且将保持沉默。该标记文件可以嵌入版本控制系统中。该文件很大，但并不吓人，因为通常放置它没有任何意义。



现在，所有程序员都将看到仅与新代码或更改代码有关的警告。因此，分析仪可以开始使用第二天的功能。以后可以再次归还技术债，逐步纠正错误并配置分析仪。



因此，解决了在大型旧项目中实施分析仪的第一个问题。现在让我们弄清楚如何处理技术债务。



错误修复和重构



最简单，最自然的方法是花一些时间来分析被大量抑制的分析仪警告，并逐步对其进行处理。在某处需要纠正代码中的错误，在某处进行重构以便告诉分析器该代码没有问题。一个简单的例子： 



if (a = b)


大多数C ++编译器和分析器都对这种代码起誓，因为他们真的很想编写（a == b）的可能性很高。但是有一个默契，并且通常在文档中指出，如果有其他括号，则可以认为程序员有意识地编写了此类代码，因此无需宣誓。例如，在用于V559诊断的PVS-Studio文档（CWE-481）中，明确规定以下行将被视为正确和安全： 



if ((a = b))


另一个例子。是否在此C ++代码中忘记了休息时间？ 



case A:
  foo();
case B:
  bar();
  break;


PVS-Studio分析仪将在此处生成警告V796（CWE-484）。也许这不是一个错误，然后您应该通过添加[[fallthrough]]属性或例如__attribute __（（fallthrough））为分析器提供提示： 



case A:
  foo();
  [[fallthrough]];
case B:
  bar();
  break;


我们可以说这样的代码更改不能修复错误。是的，是的，但是有两个有用的步骤。首先，分析仪报告消除了误报。其次，该代码对于参与其维护的人员而言更加易于理解。这非常重要！仅出于此目的，已经值得进行较小的重构，以使代码变得更易于理解和维护。由于分析器不了解是否需要“中断”，因此其他程序员也将无法理解。



除了纠错和重构之外，还可以精确地抑制明确的错误分析器警告。可以禁用某些不相关的诊断。例如，有人发现V550警告关于比较float / double值毫无意义。有人称他们为重要且值得学习的人[7 ]。由开发团队决定哪些警告相关，哪些不相关。



还有其他抑制误报的方法。例如，已经提到了宏标记。所有这些在文档中都有更详细的描述。最重要的是要了解，如果您逐步系统地处理误报，那么他们没有错。绝大多数无趣的警告在配置后都会消失，只有真正需要仔细研究并在代码中进行一些更改的地方。



此外，如果有任何困难，我们将始终帮助我们的客户设置PVS-Studio。此外，在某些情况下，我们自己消除了错误的警告并纠正了错误[ 8]。以防万一，我决定提一下这种扩大合作的选择也是可行的:)。



棘轮法



还有另一种有趣的方法可以逐步提高代码质量，从而消除静态分析器的警告。最重要的是，警告的数量只能减少。















静态分析器生成的警告数量是固定的。质量门的配置方式使得现在可以仅放置不增加操作数量的代码。结果，启动了由于分析仪设置和纠错而导致的操作数量逐渐减少的过程。



即使一个人想要稍微作弊并决定通过质量检查，不是通过消除新代码中的警告，而是通过改进旧的第三方代码，这是无所畏惧的。同样，棘轮沿一个方向旋转，缺陷的数量将逐渐减少。即使一个人不想编辑自己的新缺陷，他仍将不得不改进相邻代码中的某些内容。在某个时候，减少警告数量的简便方法便告一段落，这时将解决实际错误。



Ivan Ponomarev在一篇非常有趣的文章中详细介绍了这种方法，“ 在过程中实现静态分析，不要寻找任何错误。“我建议借此了解所有对提高代码质量感兴趣的人。
