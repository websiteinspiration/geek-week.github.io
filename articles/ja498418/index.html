今から約20年間、コンピュータのネットワーク帯域幅とコンピューティング機能により、ほぼリアルタイムモードでIPプロトコルを介したサウンドとビデオの圧縮とブロードキャストが可能になりました。この間、W3CやIETFなどの中央標準化組織、および多くの大小企業が、コンピュータおよびモバイルデバイスでオーディオビデオコンテンツを効率的に圧縮、パッキング、転送、同期、および再生するための何百もの標準とプロトコルを開発しました。第一に、IPはすべてのレベルで最も安価でアクセスしやすいこと、第二に、ビデオ会議とビデオ監視技術が不可欠であり、需要が高いため、IPを介したリアルタイムビデオキャプチャ、圧縮、およびブロードキャストに特別な注意が払われました。

Googleコードに重大な変更を加えることは、誇張なくしては不可能に近いです。結局のところ、これらすべてのオブジェクトは互いに非常に緊密に一致し、機能をカプセル化せず、特定の作業を実行する独立したブロックではありません。他のシナリオでそれらを変更せずに関与させることは不可能です。

Googleのコードはまったく使用せず、libsrtpなどのオープンライブラリを使用してすべてを自分で実装してください。おそらくこれは正しい方法ですが、これも大きな仕事であるという事実に加えて、実装がGoogleと完全に互換性がなく、したがって、機能しないか、すべての場合に機能しないという事実に遭遇する可能性があります。たとえば、許容できないクロムを使用します。その後、Googleの担当者と長い間議論し、基準に準拠していることを証明できますが、準拠していないと、何千回も正しいでしょう。しかし、彼らはせいぜい、「私たちはそれを修正するでしょう、おそらくどういうわけか後で」と言うでしょう。今すぐクロムに調整する必要があります。そしてポイント。

3.なぜすべてがそんなに悲しいのか

リアルタイムでブラウザにストリーミングするこの状況は、「ビジネス主導のテクノロジー」が時々もたらすことの非常に特徴的な例です。事業を動機とする技術は、事業に必要な方向に、また、事業に喜ばれる限りにおいて発展しています。ビジネスアプローチのおかげで、今ではパーソナルコンピューターと携帯電話を手に入れることができます。これらの消費者向け技術の開発と大衆への導入に政府や中央計画の省庁がそれほど関心を持つことはできません。所有者の個人的な利益に動機付けされた民間企業は、技術的な機会が発生するとすぐにこれを行いました。

本質的ではない消費財やサービスは、それがなくても安心して生活できるものである方が、民間事業によってよりよく開発され、エネルギー、道路、警察、学校教育など、人にとって不可欠なものは一元的に開発されるべきであることが長い間知られ、理解され、受け入れられてきました。国が管理する機関。

私たち、ソビエト連邦の子供たちと考え方は、「技術的に正しく強力なテクノロジーを作って、人々がそれを使用できるようにしましょう」と言うことはもちろん、計画されているソビエトシステム（政府が突然決定した場合）では、ストリーミングテクノロジーはリアルタイムIPは1年で開発および実装でき、ビジネスが20年で獲得したものよりも桁違いに優れています。しかし、そうなると、それが発展せず、時代遅れになり、結局、長期的には、商用の西側の技術が失われることも理解しています。

したがって、ストリーミングシミングなしでうまくやっていくことができるので、まさに民間の手に委ねられています。それは消費者の利益ではなく、彼ら自身の利益のためにそれを発展させます。消費者の利益にならないのはなぜですか？しかし、需給についてはどうでしょうか？消費者は何を必要とし、ビジネスはそれを提供しますか？しかし、それは提供していません。すべての消費者が叫んでいます-グーグル、WebRTCでAACオーディオをサポートしていますが、グーグルはそうするつもりはありませんが、それを行うことは決してありません。 Appleはまったく気にせず、ガジェットに待望のストリーミングテクノロジーを実装していません。どうして？はい、常にビジネスが消費者が必要とすることをするわけではないからです。彼は独占者であり、消費者が失うことを恐れていないとき、彼はこれをしません。その後、企業はその地位を強化することに忙しい。そのため、Googleは近年サウンドコーデックのメーカーを多数購入しました。そして今、それはOpusオーディオをプッシュし、すべてのテクノロジーが長い間AACオーディオに切り替わったため、全世界にAAC-> OpusをWebRTCに一致するようにトランスコードすることを強いています。 Googleは、AACは有料のテクノロジーであり、Opusは無料であるという事実により、これを正当化します。しかし実際には、これはそのテクノロジーを標準として確立するために行われます。 Appleがかつて私たちが好きになったその惨めなHLSでやったように、またはAdobeが以前の無責任なRTMPプロトコルでやったように。ガジェットとブラウザはまだ技術的に開発が難しいものです。ここから独占者が生まれ、ここから彼らは言うように、物事はそこにあります。また、W3CとIETFは同じ独占者によって後援されているため、「技術的に正しく強力なテクノロジーを作って、人々がそれを使用できるようにして、すべてがうまくいくようにする」という考え方はありません。しかし、そうだったはずです。

この状況から抜け出す方法は何ですか？どうやら、「適切な」ビジネス主導のテクノロジー、競争の結果、その他のあらゆる種類の素晴らしいことを待っているだけで、通常のインターネットで遠隔医療サービスを提供できるように、単純な田舎の医者にふさわしい民主的なものが最終的に生まれます。確かに、単純な田舎の医者ではなく、大金を支払うことができる人々のために修正を加える必要があります、ビジネスは長い間リアルタイムストリーミングソリューションを提供してきました。信頼性が高く、専用のネットワークと特別な機器が必要です。多くの場合、IPプロトコルでは機能しません。これは、このような悲しい状況のもう1つの理由であり、リアルタイムで作成されたものではなく、常に保証されているわけではありません。常にではありませんが、重要な状況ではありませんが、現時点では非常に適しています。では、WebRTCを試してみましょう。これまでのところ、すべての悪の中で、彼は最小で最も民主的です。結局のところ、あなたはグーグルに感謝を言う必要があります。

4. WebRTC

Wowza、Flashphoner、Kureto、Flussonic、Red5 Pro、Unreal Media Server を実装するメディアサーバーについて少し -これらはWebRTCをサポートするメディアサーバーの一部です。それらは、ブラウザーからサーバーへのビデオの公開を提供し、サーバーからWebRTCを介してブラウザーにビデオをブロードキャストします。

この記事で説明されている問題は、さまざまな方法でさまざまな程度で成功し、これらのソフトウェア製品で解決されています。それらのいくつか、たとえば、クレントやWowzaは、サーバーで直接オーディオビデオトランスコーディングを行い、他のもの、たとえば、Unreal Media Serverは、それ自体をトランスコードせずに、他のプログラムを提供します。WowzaやUnreal Media Serverなどの一部のサーバーは、WebRTC自体が各接続に個別のポートを割り当てるため、プロバイダーがファイアウォールで多くのポートを開く必要があり、セキュリティの問題が発生するため、1つの中央TCPおよびUDPポートを介したすべての接続でのストリーミングをサポートします。

これらのサーバーすべてにさまざまな方法で実装された多くのポイントと微妙な点があります。これがどれだけ消費者に適しているか、あなたを判断して、親愛なるユーザー。
