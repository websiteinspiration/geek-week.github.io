1.5.1。タワーモデリング


スタックは、後入れ先出し（LIFO）の原則に基づいてモデル化されたデータ構造です。スタックに取得される最後のものは、そこからフェッチされる最初のものになります。スタックの2つの主な操作は、プッシュ（put）とpop（extract）です。pushオペレーションは新しいアイテムをスタックにプッシュし、popはスタックからそれを削除して、最後に挿入されたアイテムを返します。リストをバックアップストレージとして使用して、Pythonでスタックを簡単にモデル化できます（リスト1.20）。


リスト1.20 hanoi.py

from typing import TypeVar, Generic, List
T = TypeVar('T')

class Stack(Generic[T]):

     def __init__(self) -> None:
          self._container: List[T] = []

     def push(self, item: T) -> None:
          self._container.append(item)

     def pop(self) -> T:
          return self._container.pop()

     def __repr__(self) -> str:
          return repr(self._container)

提示されたStackクラスは__repr __（）メソッドを実装します。これにより、タワーの内容を簡単に調べることができます。__repr __（）は、print（）関数がスタックに適用されたときに出力されるものです。

, . Generic Stack . T T = TypeVar ('T'). . Stack , Stack[int], T int. , — . , .

スタックは、ハノイタワーのチャレンジに最適です。ディスクをタワーに移動するには、プッシュ操作を使用できます。ディスクをあるタワーから別のタワーに移動するには、最初の（ポップ）から押して、2番目のタワー（プッシュ）に配置します。

タワーをStackオブジェクトとして定義し、最初のタワーをディスクで満たします（リスト1.21）。

リスト1.21 hanoi.py（続き）

num_discs: int = 3
tower_a: Stack[int] = Stack()
tower_b: Stack[int] = Stack()
tower_c: Stack[int] = Stack()
for i in range(1, num_discs + 1):
     tower_a.push(i)

1.5.2。ハノイの塔問題の解決


ハノイの塔の問題はどのように解決できますか？ドライブを1つだけ移動しようとしていると仮定します。次に、これを行う方法を知っていますか？実際、1つのディスクを移動することは、この問題の再帰的な解決策の基本的なケースです。複数のドライブの移動は再帰的なケースです。重要な点は、実際には、エンコードする必要がある2つのシナリオがあることです。1つのディスクの移動（基本ケース）と複数のディスクの移動（再帰ケース）です。

再帰的なケースを理解するために、特定の例を検討してください。タワーAに配置された上部、中央、下部の3つのディスクがあり、それらをタワーCに移動したいとします（その後、これは問題を概略的に説明するのに役立ちます）。最初に上部ディスクをタワーCに移動できます。次に-中央のディスクをタワーBに移動し、その後、上部のディスクをタワーCからタワーBに移動します。これで、下部のディスクが引き続きタワーAに配置され、2つの上部のディスクがタワーBに配置されました。あるタワー（A）から別のタワー（B）までドライブします。下のディスクをAからCに移動するのが基本的なケースです（1つのディスクを移動する）。これで、AからBと同じ手順で、2つの上部ディスクをBからCに移動できます。上部ディスクをAに移動し、中間ディスクをCに移動し、最後に上部ディスクをAからCに移動します。

コンピュータサイエンスのクラスでは、これらのタワーの小さなモデルがよく見られ、ピンとプラスチックディスクから構築されています。3本の鉛筆と3枚の紙であなた自身のモデルを作ることができます。おそらく、これはソリューションの視覚化に役立ちます。

3つのディスクがある例では、1つのディスクを移動する単純な基本的なケースと、一時的な第3タワーを使用して残りのディスク（この場合は2つ）を再帰的に移動するケースがありました。再帰的なケースを次のステップに分割できます。

  1. Cを中間タワーとして使用して、上部のn-1ドライブをタワーAからタワーB（一時）に移動します。
  2. 下のドライブをAからCに移動します。
  3. n-1台のディスクをタワーBからタワーCに移動します。タワーAは中間です。

驚いたことに、この再帰的アルゴリズムは3つだけでなく、任意の数のディスクに対して機能します。これをhanoi（）関数としてエンコードします。この関数は、3番目の一時的なタワーを使用して、あるタワーから別のタワーにディスクを移動します（リスト1.22）。

リスト1.22 hanoi.py（続き）

def hanoi(begin: Stack[int], end: Stack[int], temp: Stack[int], n: int) ->
       None:
      if n == 1:
          end.push(begin.pop())
      else:
          hanoi(begin, temp, end, n — 1)
          hanoi(begin, end, temp, 1)
          hanoi(temp, end, begin, n - 1)

hanoi（）を呼び出した後、タワーA、B、Cをチェックして、ディスクが正常に移動されたことを確認する必要があります（リスト1.23）。

リスト1.23 hanoi.py（続き）

if __name__ == "__main__":
     hanoi(tower_a, tower_c, tower_b, num_discs)
     print(tower_a)
     print(tower_b)
     print(tower_c)

ドライブが実際に移動されていることがわかります。ハノイのタワーの問題の解決策をコーディングするとき、いくつかのディスクをタワーAからタワーCに移動するために必要なすべてのステップを理解する必要はありません。任意の数のディスクを移動するための一般的な再帰アルゴリズムを理解し、それをシステム化して、コンピューターが残りの作業を行えるようにしました。これが、問題に対する再帰的な解決策を策定する力です。多くの場合、個々の行動の精神的表現にエネルギーを浪費することなく、解決策を抽象的に想像できます。

ちなみに、hanoi（）関数はディスク数に応じて指数関数的に実行されるため、64台のディスクでも問題の解決策は不適切です。変数num_discsを変更することにより、異なる数のディスクで実行することができます。ディスクの数が増えると、ハノイタワーのタスクを完了するためのステップ数が指数関数的に増加します。詳細は、多くのソースで確認できます。この問題の再帰的解決の背後にある数学について詳しく知りたい場合は、記事「ハノイの塔について」のカールバーチの説明を参照してください。

1.6。実際のアプリケーション


この章で紹介するさまざまな方法（再帰、メモ化、圧縮、およびビットレベルでの操作）は、最新のソフトウェアの開発において非常に広まっているため、これらがなければ、コンピューティングの世界を想像することはできません。タスクはタスクなしで解決できるという事実にもかかわらず、これらの方法を使用してタスクを解決する方が多くの場合、より論理的または適切です。

特に、再帰は多くのアルゴリズムだけでなく、プログラミング言語全体の基礎にもなっています。 SchemeやHaskellなどの一部の関数型プログラミング言語では、命令型言語で使用されるループが再帰によって置き換えられます。ただし、再帰的方法を使用して達成できるすべてのことを繰り返し実行することもできることを忘れないでください。

メモ化は、言語を解釈するプログラムであるパー​​サーの処理を高速化するために使用されています。これは、最近の計算の結果が再び要求される可能性が高いすべてのタスクで役立ちます。メモ化の別の領域は、プログラミング言語のランタイムです。これらのランタイムの一部（例えば、Prologバージョンの場合）は、関数呼び出しの結果を自動的に保存する（自動マッシュ）ため、同じ呼び出しで関数を次回実行する必要はありません。これは、fib6（）の@lru_cache（）デコレータに似ています。

圧縮により、帯域幅が制限されたインターネットの世界がより耐えられるようになりました。セクション1.2で説明したビット文字列メソッドは、1バイトでも冗長である可能性のある値の数が限られている現実世界の単純なデータ型に適用できます。ただし、ほとんどの圧縮アルゴリズムは、データセット内のパターンや構造を検索することで機能し、重複する情報を排除します。セクション1.2で説明されているよりもはるかに複雑です。

使い捨て暗号は、一般的な暗号化の場合には適していません。元のデータを復元するには、エンコーダーとデコーダーの両方にキーの1つ（この例ではダミーデータ）が必要です。これは煩雑で、ほとんどの暗号化スキームでは、キーを秘密に保つという目的を達成できません。しかし、「ワンタイム暗号」という名前は、冷戦中に架空のデータが記録された実際の紙のノートブックを使用して暗号化されたメッセージを作成したスパイによって考案されたことに興味があるかもしれません。

これらのメソッドはプログラムのビルディングブロックであり、他のアルゴリズムはそれらに基づいています。次の章では、それらがどの程度広く適用されているかを説明します。
