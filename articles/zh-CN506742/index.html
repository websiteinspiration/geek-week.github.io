为什么像素的中心应位于（0.5; 0.5）

如今，当从摄像机的“眼睛”执行光线追踪变得越来越流行时，需要重新学习这一课：如果像素的中心位于坐标（0.5; 0.5）上，代码将变得更好，并且生活变得更容易。如果您确定自己做的一切正确，那么请按照同样的方式继续；对于您来说，本文并不新鲜。阅读更好，这是。

保罗·赫伯特（Paul Huckbert）可爱的短篇文章“像素坐标是什么？”首先（至少对我来说）解释了将像素中心置于（0.5; 0.5）的感觉。摘自1990年的《Graphics Gems》第246-248页。

今天，这篇文章很难找到，因此我将简要概述其实质。假设我们有一个宽度和高度为1000的屏幕，仅考虑X轴，可能会尝试将0.0分配给行中最左边像素的中心，将1.0分配给下一个像素的中心，依此类推。您甚至可以使用舍入，将浮点坐标73.6和74.4转移到中心74.0。

但是，值得考虑。通过此比较，左边缘将在坐标-0.5处，右边缘将在999.5处。使用这种系统很不方便。更糟糕的是，如果将不同的运算符喜欢abs()或应用于像素的坐标值mod()，那么这样的比较会导致边缘处的微不足道的误差。

以0.0到1000.0的间隔工作更容易，其中每个像素的中心具有0.5的分数部分。例如，整数像素43的子像素值的范围在43.0至43.99999之间。这是保罗文章的图画：


在OpenGL中，像素中心始终具有小数部分（0.5; 0.5）。刚开始，DirectX并不遵循这一原则，但在DirectX 10中，它牢牢抓住了头脑。

从整数坐标转换为浮点像素坐标的操作将加0.5；否则，将转换为0。将float转换为整数就足够了floor()。

但这是一个漫长的故事。毕竟，今天所有人都在这样做，对吗？我回到这个主题是因为在我的示例中，我遇到了（伪）代码，用于生成透视相机的方向以进行光线跟踪：

 float3 ray_origin = camera->eye;
 float2 d = 2.0 * 
     ( float2(idx.x, idx.y) / 
       float2(width, height) ) - 1.0;
 float3 ray_direction =
     d.x*camera->U + d.y*camera->V + camera->W;

idx 向量是像素的整数位置，宽度和高度是屏幕分辨率。计算向量d，并使用两个向量U和V的乘积将其生成到世界空间中的向量。然后将向量W相加-相机在世界空间中的方向。U和V表示显示平面的X轴和Y轴的正方向与眼睛的距离为W。在上面的代码中，它们看起来都很漂亮且对称。所以在大多数情况下


向量d必须在屏幕上的点的标准化设备坐标（标准化设备坐标，NDC）中表示从-1.0到1.0的一对值。但是，代码在这里失败。让我们继续我们的示例：将整数像素位置（0; 0）传输到（-1.0; -1.0）。看起来不错吧？但是最大整数像素位置是（999; 999），相当于（0.998; 0.998）。 0.002的总差异是由于以下事实造成的：该错误的覆盖将整个图片移动了半个像素。这些像素中心应与每个边缘相距0.001。

第二行代码应如下所示：

    float2 d = 2.0 *
        ( ( float2(idx.x, idx.y) + float2(0.5,0.5) ) / 
            float2(width, height) ) - 1.0;

然后，我们获得了像素中心的正确NDC间隔，从-0.999到0.999。如果我们以这种方式用浮点（0,0; ​​0,0）和（1000,0; ​​1000,0）转换角度值（我们不加0.5，因为我们已经在使用浮点了），然后我们从边缘到边缘获得完整的NDC间隔，从-1.0到1.0；这证明了代码的正确性。

如果0.5困扰您并且您缺乏对称性，则在像素内生成随机值，即当通过每个像素发出随机光线进行平滑处理时，可以使用以下优雅公式：

    float2 d = 2.0 *
        ( ( float2(idx.x, idx.y) + 
                float2( rand(seed), rand(seed) ) ) /
            float2(width, height) ) - 1.0;

我们只需将间隔[0.0,1.0）中的随机数添加到像素位置的每个整数值。该随机值的平均值将为0.5，即像素的中心。

因此，我将简单地说：要小心。正确实现半像素。根据我的经验，在我从事Autodesk光栅化器代码工作的多年中，这种半像素错误会在许多不同的地方（相机，纹理采样等）弹出。在装配线的下游，他们带来的只有痛苦。如果我们不小心，它们也可能出现在光线跟踪器中。
