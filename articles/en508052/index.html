Snort or Suricata. Part 3: protect the office network

In a previous article, we talked about how to run the stable version of Suricata on Ubuntu 18.04 LTS. Configuring IDS on one node and connecting free rule sets is fairly straightforward. Today we‚Äôll look at how to protect the corporate network using Suricata installed on a virtual server, which is the most common type of attack. For this, we need VDS on Linux with two cores. The amount of RAM depends on the load: 2 GB is enough for someone, and for more serious tasks it may take 4 or even 6. Plus a virtual machine with the possibility of experimentation: you can start with a minimal configuration and increase resources as needed.

Removing IDS to a virtual machine may be needed primarily for testing. If you have never dealt with such solutions, you should not rush to order physical hardware and change the network architecture. It is better to run the system safely and at no extra cost in order to determine the need for computing resources. It is important to understand that all corporate traffic will have to be passed through a single external node: to connect a local network (or several networks) to VDS with IDS Suricata installed, you can use SoftEther , an easy-to-configure cross-platform VPN server that provides strong encryption. An office Internet connection may not have a real IP, so it‚Äôs better to upgrade it to VPS. There are no ready-made packages in the Ubuntu repository, you will have to download the software either withproject site , or from an external repository on the Launchpad service (if you trust him): 



sudo add-apt-repository ppa:paskal-07/softethervpn
sudo apt-get update


You can view the list of available packages using the following command: 



apt-cache search softether






We will need softether-vpnserver (the server in the test configuration is running on VDS), as well as softether-vpncmd - command line utilities to configure it. 



sudo apt-get install softether-vpnserver softether-vpncmd


To configure the server, a special command line utility is used: 



sudo vpncmd






We will not talk in detail about the setting: the procedure is quite simple, it is well described in numerous publications and does not apply directly to the topic of the article. In short, after starting vpncmd, select item 1 to go to the server management console. To do this, enter the name localhost and press enter instead of entering the name of the hub. The administrator password is set in the console with the serverpasswordset command, the virtual DEFAULT hub (hubdelete command) is deleted and a new one is created with the name Suricata_VPN, and its password is also set (hubcreate command). Next, go to the management console of the new hub using the hub Suricata_VPN command to create a group and user using the groupcreate and usercreate commands. User password is set using userpasswordset.



SoftEther supports two modes of traffic transfer: SecureNAT and Local Bridge. The first is a proprietary technology for building a virtual private network with its own NAT and DHCP. SecureNAT does not require TUN / TAP, as well as setting up Netfilter or another firewall. Routing does not affect the core of the system, and all processes are virtualized and work on any VPS / VDS, regardless of the hypervisor used. This leads to increased processor load and lower speed compared to the Local Bridge mode connecting the SoftEther virtual hub with a physical network adapter or TAP device.



The setup in this case is complicated because routing is done at the kernel level using Netfilter. Our VDSs are built on Hyper-V, so in the last step we create a local bridge and activate the TAP device with the command bridgecreate Suricate_VPN -device: suricate_vpn -tap: yes. After exiting the hub management console, we will see in the system a new network interface that has not yet been assigned an IP: 



ifconfig






Then you have to enable packet routing between interfaces (ip forward) if it is inactive: 



sudo nano /etc/sysctl.conf


Uncomment the following line: 



net.ipv4.ip_forward = 1


We save the changes to the file, exit the editor and apply them using the following command: 



sudo sysctl -p


Next, we need to define a subnet with dummy IPs for the virtual network (for example, 10.0.10.0/24) and assign an address to the interface: 



sudo ifconfig tap_suricata_vp 10.0.10.1/24


Then you need to spell out Netfilter rules.



1. If necessary, allow incoming packets to listening ports (proprietary protocol SoftEther uses HTTPS and port 443) 



sudo iptables -A INPUT -p tcp -m tcp --dport 443 -j ACCEPT
sudo iptables -A INPUT -p tcp -m tcp --dport 992 -j ACCEPT
sudo iptables -A INPUT -p tcp -m tcp --dport 1194 -j ACCEPT
sudo iptables -A INPUT -p udp -m udp --dport 1194 -j ACCEPT
sudo iptables -A INPUT -p tcp -m tcp --dport 5555 -j ACCEPT


2. Configure NAT from the 10.0.10.0/24 subnet to the main IP server 



sudo iptables -t nat -A POSTROUTING -s 10.0.10.0/24 -j SNAT --to-source 45.132.17.140


3. We allow passing packets from the subnet 10.0.10.0/24 



sudo iptables -A FORWARD -s 10.0.10.0/24 -j ACCEPT


4. We allow passing packets for already established connections 



sudo iptables -A FORWARD -p all -m state --state ESTABLISHED,RELATED -j ACCEPT


Automation of the process when the system is restarted using initialization scripts will be left to readers as homework.



If you want to issue IP clients automatically, you will also have to install some kind of DHCP service for the local bridge. This completes the server setup and you can go to the clients. SoftEther supports many protocols, the use of which depends on the capabilities of the LAN equipment. 



netstat -ap |grep vpnserver






Since our test router also works under Ubuntu, we will install the softether-vpnclient and softether-vpncmd packages on it from an external repository to use the proprietary protocol. You will need to start the client: 



sudo vpnclient start


To configure, we use the vpncmd utility, choosing localhost as the machine on which vpnclient is running. All commands are done in the console: you will need to create a virtual interface (NicCreate) and an account (AccountCreate).



In some cases, you need to specify the authentication method using the AccountAnonymousSet, AccountPasswordSet, AccountCertSet, and AccountSecureCertSet commands. Since we do not use DHCP, the address for the virtual adapter is manually set.



In addition, we need to enable ip forward (the parameter net.ipv4.ip_forward = 1 in the /etc/sysctl.conf file) and configure static routes. If necessary, on VDS with Suricata, you can configure port forwarding to use the services installed on the local network. On this, the union of networks can be considered complete.



The configuration we offer will look something like this:







Customize Suricata



In the previous article, we talked about two IDS operating modes: through the NFQUEUE queue (NFQ mode) and through zero copy (AF_PACKET mode). The second requires two interfaces, but differs in higher speed - we will use it. The parameter is set by default in / etc / default / suricata. We will also need to edit the vars section in /etc/suricata/suricata.yaml, setting the virtual subnet there as your home subnet.







To restart IDS, use the command: 



systemctl restart suricata


The solution is ready, now you may need to check it for resistance to the actions of attackers.



We simulate attacks



There may be several scenarios for the combat use of an external IDS service:



Protection against DDoS attacks (main purpose)



It is difficult to implement such an option within the corporate network, since the analysis packages must be sent to the system looking at the Internet interface. Even if IDS blocks them, spurious traffic can put a data channel. To avoid this, you need to order a VPS with a sufficiently efficient Internet connection that can skip all the traffic on the local network and all external traffic. This is often easier and cheaper than expanding the office channel. As an alternative, it is worth mentioning specialized services for protection against DDoS. The cost of their services is comparable to the cost of a virtual server, but it does not require laborious setup, but there are drawbacks - for the money the client receives only protection from DDoS, while their own IDS can be configured as you like.



Protection against external attacks of other types



Suricata is able to cope with attempts to exploit various vulnerabilities in corporate network services accessible from the Internet (mail server, web server and web applications, etc.). Usually, for this, IDS are installed inside the LAN after the border devices, but the take-out has the right to exist.



Protection against internal intruders



Despite the best efforts of the system administrator, computers on the corporate network may be infected with malware. In addition, hooligans sometimes appear in the LAN and try to perform certain illegal operations. Suricata can help block such attempts, although it is better to install it inside the perimeter to protect the internal network and use it together with a managed switch capable of mirroring traffic to one port. The external IDS in this case is also not useless - at least it will be able to catch attempts by malware living in the LAN to contact an external server.



First, we will create another test attacking VPS, and on the local network router we will raise Apache with the default configuration, after which we will forward the 80th port to it from the IDS server. Next, we will simulate a DDoS attack from the attacking node. For more serious tasks, you should use the Metasploit Framework . It is designed to conduct penetration tests and allows you to simulate a variety of attacks. Installation instructions are available on the project website. After installation, an update is required: 



sudo msfupdate


For testing, run msfconsole.







Unfortunately, in the latest versions of the framework there is no possibility of automatic hacking, so exploits will have to be manually sorted and run using the use command. First you need to determine the ports open on the attacked machine, for example, using nmap (in our case, it will be completely replaced by netstat on the attacked node), and then select and use the appropriate Metasploit modules . 



There are other means to test IDS resistance to attacks, including online services. For the sake of curiosity, you can arrange stress testing using the trial version of IP Stresser. To check the reaction to the actions of internal attackers, it is worth installing special tools on one of the machines on the local network.
