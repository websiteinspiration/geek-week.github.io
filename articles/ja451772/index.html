Clouderaの特別な点とその調理方法

統計によると、分散コンピューティングとビッグデータの市場は年間18〜19％の割合で成長しています。したがって、これらの目的のためにソフトウェアを選択するという問題は依然として関連しています。この投稿では、分散コンピューティングが必要な理由から始め、ソフトウェアの選択について詳しく説明し、HadoopとClouderaの使用について話し、最後にハードウェアの選択とそれがさまざまな方法でパフォーマンスに与える影響について説明します。


通常のビジネスで分散コンピューティングが必要なのはなぜですか？すべてが同時にシンプルで複雑です。単純-ほとんどの場合、情報の単位ごとに比較的単純な計算を実行するため。そのような情報がたくさんあるので難しいです。沢山の。その結果、テラバイト単位のデータを1000スレッドで処理する必要があります。したがって、使用シナリオは非常に普遍的です。さらに大きなデータ配列で多数のメトリックを考慮する必要がある場合はいつでも計算を適用できます。

最近の例：ピッツェリアチェーンのDodo Pizzaが特定された顧客の注文ベースの分析に基づいて、任意の充填のピザを選択するとき、ユーザーは通常、6つの基本的な材料セットといくつかのランダムなものだけで操作します。これに従って、ピッツェリアは購入を調整しました。さらに、注文段階で提供される追加の製品をユーザーに適切に推奨できるようになり、利益の増加に役立ちました。

別の例：コモディティポジションの分析により、H＆Mストアは販売レベルを維持しながら、個々のストアの範囲を40％削減できました。これは売れ行きの悪いポジションを排除することで達成され、計算には季節性が考慮されました。

ツールの選択


この種のコンピューティングの業界標準はHadoopです。どうして？ Hadoopは優れた、十分に文書化されたフレームワーク（同じHabrがこのトピックに関する多くの詳細な記事を発行している）であり、ユーティリティとライブラリのセット全体が付属しています。構造化データと非構造化データの両方の巨大なセットを入力でき、システム自体がそれらをコンピューティングパワー間で分散します。さらに、これらの同じ容量はいつでも増加または無効にできます-動作中の同じ水平スケーラビリティ。

2017年、影響力のあるコンサルティング会社のガートナーは、Hadoopはすぐに廃止されます。理由はかなり平凡です。アナリストは、企業がクラウドに大規模に移行すると信じています。なぜなら、そこでコンピューティングパワーを使用するという事実にお金を払うことができるからです。 Hadoopを「埋め込む」ことができると考えられる2番目の重要な要素は、作業の速度です。 Apache SparkやGoogle Cloud DataFlowなどのオプションは、基になるHadoopのMapReduceよりも高速であるためです。

Hadoopはいくつかの柱に依存しています。その中で最も注目すべきものは、MapReduceテクノロジー（サーバー間のコンピューティングのためのデータ分散システム）とHDFSファイルシステムです。後者は、クラスターのノード間で分散された情報を格納するように特別に設計されています。固定サイズの各ブロックを複数のノードに配置でき、レプリケーションにより、システムは個々のノードの障害の影響を受けません。ファイルテーブルの代わりに、NameNodeと呼ばれる特別なサーバーが使用されます。

以下の図は、MapReduceワークフローを示しています。最初の段階では、データは特定の特性に従って分割され、2番目の段階では、計算能力に従って分散され、3番目の段階では、計算が実行されます。


MapReduceはもともと、検索のニーズのためにGoogleによって作成されました。その後、MapReduceが無料のコードに入り、Apacheがプロジェクトを担当しました。さて、Googleは徐々に他のソリューションに移行しています。興味深いニュアンス：Googleは現在、Google Cloud Dataflowと呼ばれるプロジェクトを持っています。これは、Hadoopの次のステップとして位置付けられており、すぐに置き換えられます。

よく見ると、Google Cloud DataflowはApache Beamのバリエーションに基づいていることがわかりますが、Apache Beamには十分に文書化されたApache Sparkフレームワークが含まれているため、ほぼ同じ速度で意思決定を実行できます。まあ、Apache SparkはHDFSファイルシステムで正常に動作するため、Hadoopサーバーにデプロイできます。

ここでは、Google Cloud Dataflowに対するHadoopとSparkのドキュメントと既製のソリューションのボリュームを追加し、ツールの選択が明らかになります。さらに、エンジニアは、HadoopまたはSparkで実行するコードを自分で決定し、タスク、経験、資格に焦点を当てることができます。

クラウドまたはローカルサーバー


クラウドへの普遍的な移行の傾向により、Hadoop-as-a-serviceのような興味深い用語が生まれました。このシナリオでは、接続されているサーバーの管理が非常に重要になっています。悲しいことに、純粋なHadoopは多くのことを自分の手で行う必要があるため、その人気にもかかわらず、構成するのはかなり複雑なツールです。たとえば、サーバーを個別に構成し、そのパフォーマンスを監視し、多くのパラメーターを慎重に構成します。一般的に、アマチュアで働くことは、何かを台無しにしたり、何かを見逃したりする絶好の機会です。

そのため、当初は便利な展開ツールと管理ツールを備えたさまざまなディストリビューションが非常に人気を博しました。 Sparkをサポートし、物事を簡単にする最も人気のあるディストリビューションの1つはClouderaです。有料版と無料版の両方があり、後者ではすべての基本機能が利用可能で、ノードの数を制限しません。



セットアップ中に、Cloudera ManagerはSSH経由でサーバーに接続します。興味深い点：インストール中に、いわゆる小包：互いに動作するように構成されている必要なすべてのコンポーネントをそれぞれが含む特別なパッケージによって実行されることを示す方が良いでしょう。実際、これはパッケージマネージャーのこのような改善されたバージョンです。

インストール後、クラスター管理コンソールが表示されます。ここでは、クラスターごとのテレメトリ、インストールされているサービスを確認できます。さらに、リソースの追加/削除、クラスター構成の編集を行うことができます。
