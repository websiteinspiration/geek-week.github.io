Linuxでユーザーモードの使用を開始する方法

翻訳者からの紹介：さまざまな種類のコンテナーの私たちの生活への大規模な参入を背景に、一度すべてのテクノロジーを開始したことを知ることは非常に興味深く、有用な場合があります。それらのいくつかは今でも使用できますが、誰もがそのような方法を覚えているわけではありません（またはそれらが急速な開発中に見つからなかったかどうかを知っています）。そのようなテクノロジーの1つがユーザーモードLinuxです。オリジナルの作者はかなりうろたえ、古い開発の何がまだ機能し、何があまり機能していないかを理解し、自分で2k19で自家製のUMLを開始する方法について、段階的な指示のようなものを集めました。はい、元の投稿の著者をHabrに招待しましたカディなので、質問がある場合はコメントで英語で尋ねてください。



Linuxユーザーモードは、実際にはLinuxカーネルポートそのものです。このモードでは、本格的なLinuxカーネルをユーザープロセスとして実行でき、通常、開発者がドライバーをテストするために使用します。ただし、このモードは一般的な分離のためのツールとしても役立ちます。その原理は仮想マシンの操作に似ています。このモードは、Dockerよりも分離を提供しますが、KVMやVirtual Boxのような本格的な仮想マシンよりは劣ります。

一般に、ユーザーモードは奇妙で使いにくいように見えるかもしれませんが、それでも独自のアプリケーション領域があります。結局のところ、これは完全なLinuxカーネルであり、権限のないユーザーが操作します。この機能により、ホストマシンへの脅威なしに、信頼できない可能性のあるコードを実行できます。また、完全なカーネルであるため、そのプロセスはホストマシンから分離されています。つまり、ユーザーモード内で実行されているプロセスはホストから見えません。これは通常のDockerコンテナとは異なります。この場合、ホストマシンは常にリポジトリ内のプロセスを認識します。私のサーバーの1つからこのpstreeの断片を見てください：

Dockerコンテナーを操作するとき、ゲストシステムで実行されているプロセスの名前をホストから確認できます。 Linuxユーザーモードでは、これは不可能です。どういう意味ですか？つまり、Linux監査システム（Linuxの監査サブシステム）を介して動作する監視ツールは、ゲストシステムで実行されているプロセスを認識しません。ただし、状況によっては、この機能が両刃の剣になることがあります。

一般的に、以下の投稿全体は一連の研究であり、望ましい結果を達成するための失礼な試みです。これを行うには、さまざまな古代のツールを使用し、カーネルソースを読み取り、小学校にいる間に書かれたコードを集中的にデバッグし、必要なツールを探すために特別なバイナーを使用してHerokuビルドを選択する必要がありました。このすべての作業は、私のIRCのメンバーが私を魔術師（魔法）と呼び始めたという事実につながりました。この投稿が、信頼性の高いドキュメントを提供して、すべてを同じようにクランクアップできるようになりましたが、カーネルとOSのバージョンが新しいことを期待しています。

カスタマイズ


Linuxユーザーモードを設定するには、いくつかの手順を実行します。

  • ホストへの依存関係のインストール。
  • Linuxカーネルのダウンロード
  • カーネルアセンブリのセットアップ。
  • カーネルアセンブリ;
  • バイナリーインストール。
  • ゲストファイルシステムのセットアップ。
  • カーネル起動パラメーターの選択。
  • ゲストネットワークのセットアップ。
  • ゲストカーネルを起動します。

私が自分ですべてを上げようと決心した場合は、UbuntuまたはDebianのようなシステムで説明されているすべてを実行する可能性が高いと思います。上記のすべてを私のお気に入りの配布キットであるアルパインに実装しようとしましたが、どうやらLinuxカーネルがユーザーモードのドライバーのglibc-ismsにハードバインディングを持っていたため、何も起こりませんでした。やっと問題が判明したら、上流に報告するつもりです。

ホストへの依存関係のインストール


Ubuntuは、Linuxカーネルを構築するために、少なくとも以下のパッケージを必要とします（クリーンインストールを想定）。 次のコマンドを使用してインストールできます（root特権を使用するか、sudoを使用）。

- 'build-essential'
- 'flex'
- 'bison'
- 'xz-utils'
- 'wget'
- 'ca-certificates'
- 'bc'
- 'linux-headers'



apt-get -y install build-essential flex bison xz-utils wget ca-certificates bc \
                   linux-headers-$(uname -r)

Linuxカーネルのメニューセットアッププログラムを実行するには、インストールが必要ですlibncurses-dev。次のコマンドを使用してインストールされていることを確認してください（root権限を使用するか、sudoを使用）。

apt-get -y install libncurses-dev

カーネルのダウンロード


カーネルをロードして再構築する場所を決定します。この操作では、約1.3 GBのハードディスク領域を割り当てる必要があるため、1つあることを確認してください。

次に、kernel.orgにアクセスして、カーネルの最新の安定バージョンをダウンロードするためのURLを取得します。この記事の執筆時点：https : //cdn.kernel.org/pub/linux/kernel/v5.x/linux-5.1.16.tar.xz次

を使用してこのファイルをダウンロードします'wget'。

wget https://cdn.kernel.org/pub/linux/kernel/v5.x/linux-5.1.16.tar.xz

そしてそれを抽出し'tar'ます：

tar xJf linux-5.1.16.tar.xz

次に、tarballを解凍したときに作成されたディレクトリに入ります。

cd linux-5.1.16

カーネルのビルド構成


カーネルビルドシステムは、プロセスを自動化するための多くのカスタムツールとスクリプトを備えたMakefileのコレクションです。開始するには、オンラインセットアッププログラムを開きます。

make ARCH=um menuconfig

部分的に組み立てが完了し、ダイアログボックスが表示されます。ウィンドウの下部が ' [Select]'で強調表示されている場合、スペースまたはEnterキーを使用して設定を行うことができます。通常どおり、上矢印キーと下矢印キーを使用したウィンドウ内のナビゲーション。要素の選択は左または右です。

ビューポインタ--->は、Enterキーでアクセスするサブメニューにいることを意味します。それから抜け出す方法はを通じて明らかです。「[Exit]」

次のパラメーターを ' [Select]'に含め、その横に '[*]'記号があることを確認してください：

UML-specific Options:
  - Host filesystem
Networking support (enable this to get the submenu to show up):
  - Networking options:
    - TCP/IP Networking
UML Network devices:
  - Virtual network device
  - SLiRP transport

このウィンドウからすべてが順次選択することで終了できます。「[Exit]」ただ、最後にあなたが設定を保存して選択するように要求されていることを確認してください。「[Yes]」

この投稿を読んだ後、カーネルビルドオプションをいじってみることをお勧めします。これらの実験のおかげで、低レベルのカーネル力学の作業と、そのアセンブリに対するさまざまなフラグの影響を理解する上で多くのことを学ぶことができます。

カーネルアセンブリ


Linuxカーネルは、多くのことを行う優れたプログラムです。古い機器のこのような最小限の構成でも、組み立てにかなりの時間がかかる場合があります。したがって、次のコマンドでカーネルを構築します。

make ARCH=um -j$(nproc)

何のために？このコマンドは、ビルドプロセス中に使用可能なすべてのコアとプロセッサスレッドを使用するようにアセンブラーに指示します。コマンド$(nproc)ビルドの最後には、コマンドの出力に代入nprocの一部であり、coreutils標準のUbuntuのビルドを。

しばらくすると、カーネルが実行可能ファイルにアセンブルされます./linux。

バイナリーインストール


Linuxのユーザーモードでは通常のバイナリが作成されるため、他のユーティリティと同様にインストールできます。ここに私がそれをした方法があります：

mkdir -p ~/bin
cp linux ~/bin/linux

~/binあなたの中に 何があるかを確認することも価値があります$PATH：

export PATH=$PATH:$HOME/bin

ゲストファイルシステムのセットアップ


ゲストファイルシステムのディレクトリを作成します。

mkdir -p $HOME/prefix/uml-demo
cd $HOME/prefix

alpinelinux.orgを開き、ダウンロードセクションで最新のダウンロードリンクを見つけますMINI ROOT FILESYSTEM。この記事の執筆時点では、これは次のとおりでした。

http://dl-cdn.alpinelinux.org/alpine/v3.10/releases/x86_64/alpine-minirootfs-3.10.0-x86_64.tar.gz

wgetを使用してこのtarballをダウンロードします。

wget -O alpine-rootfs.tgz http://dl-cdn.alpinelinux.org/alpine/v3.10/releases/x86_64/alpine-minirootfs-3.10.0-x86_64.tar.gz

ゲストファイルシステムのディレクトリに入り、アーカイブを解凍します。

cd uml-demo
tar xf ../alpine-rootfs.tgz

説明されている手順により、小さなファイルシステムテンプレートが作成されます。システムの性質上、Alpine apk managerを介したパッケージのインストールは非常に困難です。しかし、このFSは一般的なアイデアを評価するには十分です。ゲストコアのゾンビプロセスによるメモリ消費を抑制

するためのtiniツールも必要です。

wget -O tini https://github.com/krallin/tini/releases/download/v0.18.0/tini-static
chmod +x tini

このバナーは、主な起動パラメーターを示しています。必要最小限のオプションのセットでカーネルを起動しましょう：

linux \
  root=/dev/root \
  rootfstype=hostfs \
  rootflags=$HOME/prefix/uml-demo \
  rw \
  mem=64M \
  init=/bin/sh

上記の行は、コアに次のことを伝えます。

  • ルートファイルシステムが疑似デバイスであるとし/dev/rootます。
  • ルートファイルシステムドライバーとしてhostfsを選択します。
  • 作成したゲストファイルシステムをルートデバイスにマウントします。
  • そして、はい、読み書きモードで。
  • 64メガバイトのRAMのみを使用します（使用する計画によってははるかに少なくすることができますが、64 MBが最適なサイズのようです）。
  • カーネルは自動的にプロセス/bin/shとして起動しinitます。

ネットワーク設定


次に、ゲストコアにネットワークをセットアップします。起動オプションの更新：

linux \
  root=/dev/root \
  rootfstype=hostfs \
  rootflags=$HOME/prefix/uml-demo \
  rw \
  mem=64M \
  eth0=slirp,,$HOME/bin/slirp \
  init=/bin/sh

次に、ネットワークをオンにします。

mount -t proc proc proc/
mount -t sysfs sys sys/

ifconfig eth0 10.0.2.14 netmask 255.255.255.240 broadcast 10.0.2.15
route add default gw 10.0.2.2

最初の2つの構成コマンドは、Slirpと通信するためのネットワークインターフェイスを確立する操作に/procも/sys必要ですifconfig。このコマンドrouteは、すべてのトラフィックがSlirpトンネルを強制的に通過するようにカーネルルーティングテーブルを設定します。これをDNSクエリで確認してみましょう。

nslookup google.com 8.8.8.8
Server:    8.8.8.8
Address 1: 8.8.8.8 dns.google

Name:      google.com
Address 1: 172.217.12.206 lga25s63-in-f14.1e100.net
Address 2: 2607:f8b0:4006:81b::200e lga25s63-in-x0e.1e100.net

動作します！

注：どうやら、元の投稿は有線ネットワークカード、または追加のドライバーを必要としないその他の構成を使用してデスクトップに書き込まれたようです。IntelのWiFi 8265を搭載したラップトップで、ネットワークを上げるとエラーが発生する

/ # ifconfig eth0 10.0.2.14 netmask 255.255.255.240 broadcast 10.0.2.15
slirp_tramp failed - errno = 2
ifconfig: ioctl 0x8914 failed: No such file or directory
/ #

どうやら、カーネルはネットワークドライバーと通信できません。残念ながら、ファームウェアをカーネルにコンパイルしようとしても、状況は修正されませんでした。公開時点では、この特定の構成で解決策を見つけることはできませんでした。より単純な構成（Virtualboxなど）では、インターフェースは正しく上昇します。

次のシェルスクリプトを使用してリダイレクトを自動化しましょう。

#!/bin/sh
# init.sh

mount -t proc proc proc/
mount -t sysfs sys sys/
ifconfig eth0 10.0.2.14 netmask 255.255.255.240 broadcast 10.0.2.15
route add default gw 10.0.2.2

echo "networking set up"

exec /tini /bin/sh

そして、実行可能としてマークします：

chmod +x init.sh

次に、カーネルコマンドラインに変更を加えます。

linux \
  root=/dev/root \
  rootfstype=hostfs \
  rootflags=$HOME/prefix/uml-demo \
  rw \
  mem=64M \
  eth0=slirp,,$HOME/bin/slirp \
  init=/init.sh

Dockerファイル


これらすべてを簡単に確認できるようにするために、説明されているほとんどの手順を自動化するDockerfileをコンパイルしました。これにより、機能する構成が提供されます。私は既製のカーネル構成も持っています。これには、投稿で説明されているすべてのものが含まれています。ただし、ここでは最低限の設定のみを示していることを理解することが重要です。


この投稿が、ゲストのコアを上げる方法を理解するのに役立つことを願っています。ある種の怪物であることが判明しましたが、このファミリのオペレーティングシステムの最新バージョンでのLinuxでのユーザーモードのアセンブリ、インストール、および構成に関する包括的なガイドとしてそれを公開するという考えでした。その後のアクションには、ゲストシステム内にすでにあるサービスやその他のソフトウェアのインストールが含まれます。 Dockerコンテナーイメージは単にtarballなので、イメージを抽出してdocker export、ゲストカーネルファイルシステムのルートでインストールパスを決定できるはずです。それでは、シェルスクリプトを実行します。
