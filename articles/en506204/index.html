Abstract data type. Part 2: Data Management

In the first part I described how to be guided in determining the Data Type (hereinafter referred to as TD). In this part I want to tell you what the Abstract Data Type (hereinafter referred to as ADT) is, how it helps to solve logically complex tasks (what is meant by logically complex tasks, see the first article) and what advantages we get.


Instead of a preface:

An Abstract Data Type is a collection that includes data and the operations that can be performed on them (S. McConnell, “Perfect Code,” Chapter 6.1. Class Basics: Abstract Data Types). In this series of articles, we consider just such a view of the ADT, extended by the data type.

Torvalds Linus - “Bad programmers worry about code. Good programmers worry about data structures and their relationships. ”
Calculation is the acquisition of new knowledge from input data.


ADT - is considered as a class containing data structures with which we work in the future and in this class. Data refers to sets of APs that were discussed in the first part .
ADT structures for data storage - means of data storage, i.e. it can be a stack, and a queue and any classical / non-classical data structure that is convenient for working with them, depends on the context of the task and on your imagination.


Why this approach allows you to organize complex logic:

  1. We only work with data (TD) without hesitation in obtaining it. We place the logic of data acquisition on “data providers” (see below). In essence, this is a struggle with complexity - we are separately thinking about how to obtain data that always has a certain type and separately we are thinking about how to work with data of this type.
  2. We can organize the ADT only with the data sets we need, performing the decomposition of the ADT if necessary (also a method of dealing with complexity).

The requirements for ADT are similar to the requirements for TD, with a small addition:

  1. ADT should be able to work only with its data.
  2. ADTs should not depend on the external state of the application.
  3. In order to receive data, use “data providers” and transmit their results to the ADT; do not use the ADT methods to obtain it.

Data Provider - this implies a class that receives the data we need from available sources, in various ways, and returns APs or sets of APs. How to organize data providers is a separate topic, depending on the context of the task and the imagination of the system developer.

Total:

  1. ADT stores aggregated data in a form convenient for us
  2. ADT should be able to work only with its data.
  3. ADT should not depend on the external state of the application.
  4. Do not use ADT methods to obtain data; for this, use “Data Providers” which return APs or sets of APs. Next, we transfer the received data to the ADT for calculations on them (we attach / apply the action to the AT).
