Swiftのエラー


この段落は、一般的なエラーのトップレベルの概要です。アプリケーションでのミスをすでに積極的に使用している場合は、安全に次の段落に進むことができます。

間違いとは？ある種の誤ったアクションまたは誤った結果。多くの場合、考えられるエラーを想定して、それらをコードに事前に記述できます。

この場合、AppleはタイプErrorを提供します。Appleのドキュメントを開くと、エラーは次のようになります（Swift 5.1に関連）。

public protocol Error {
}

追加要件なしの単なるプロトコル。ドキュメントは親切に説明しています-必須パラメーターがないため、Swiftエラー処理システムで任意の型を使用できます。そのような穏やかなプロトコルで、私たちは単に働きます。

enumを使用するという考えはすぐに頭に浮かびます。有限の既知の数のエラーがあり、それらにはある種のパラメーターがある可能性があります。これはAppleがやっていることです。たとえば、DecodingErrorの実装を検討できます。

Appleのベストプラクティスを活用してください。単純化された形式で考えられるネットワークエラーのグループを想像してください。

enum NetworkError: Error {

	case serverError
        //   ,

        //  ,   ,

これで、エラーが発生するアプリケーションの任意の場所で、Network.Errorを使用できます。

バグの扱い方は？ do catchメカニズムがあります。関数がエラーをスローできる場合、throwsキーワードでマークされます。これで、その各ユーザーは、do catch構文を介してアクセスする必要があります。エラーがなければ、エラーのあるdoブロックに陥り、catchブロックに陥ります。エラーの原因となる関数は、doブロック内の任意の数にすることができます。唯一の欠点は、catchでタイプErrorのエラーが発生することです。エラーを目的のタイプにキャストする必要があります。

別の方法として、オプションを使用できます。つまり、エラーが発生した場合はnilを取得し、かさばるデザインを取り除きます。オプションの変数を取得して、それにthrows関数を適用するとします。コードはif / guardブロックに1つに入れることができ、簡潔なままです。

以下は、throws関数の使用例です。

    }

    } catch {
    	//  NetworkError.serverError
    	print(error)

PS他の言語のdo catchと混同しないでください。 Swiftは例外をスローしませんが、エラーの値（発生した場合）を特殊レジスターに書き込みます。値がある場合はエラーブロックに進み、ない場合はdoブロックが続行されます。最も好奇心が強いソース：www.mikeash.com/pyblog/friday-qa-2017-08-25-swift-error-handling-implementation.html

このメソッドは、同期イベントを処理するのに適しており、長時間の操作（たとえば、ネットワークを介してデータを要求する）。これには時間がかかる可能性があります。その後、単純な補完を使用できます。

Swift 5の代わりとして、Resultが導入されました-成功と失敗の2つのオプションを含む準備された列挙型。それ自体では、エラーを使用する必要はありません。そして、それは非同期性と直接の関係はありません。ただし、このタイプを正確に完了に戻すことは、非同期イベントにとってより便利です（そうでない場合、2つの完了、成功と失敗、または2つのパラメーターを返す必要があります）。例を書いてみましょう：

func blah<ResultType>(handler: @escaping (Swift.Result<ResultType, Error>) -> Void) {
	handler(.failure(NetworkError.serverError)
}

blah<String>(handler { result in 
	switch result {
		case .success(let value):
			print(value)
		case .failure(let error):
			print(error)
	}
})

この情報は私たちが働くのに十分です。

もう一度、簡単に：

  • Swiftのエラーはプロトコルです
  • 列挙型の形式でエラーを表示すると便利です
  • エラーに対処するには、同期（キャッチする）と非同期（独自の競合または結果）の2つの方法があります。

ネイティブ実装？


少しインターネット検索すると、プロトコルRecoverableErrorが発生します。

// A specialized error that may be recoverable by presenting several potential recovery options to the user.
protocol RecoverableError : Error {
    var recoveryOptions: [String] { get }

    func attemptRecovery(optionIndex recoveryOptionIndex: Int, resultHandler handler: @escaping (Bool) -> Void)
    func attemptRecovery(optionIndex recoveryOptionIndex: Int) -> Bool
}

探しているようです：

  • recoveryOptions：[String]-回復オプションを格納するプロパティ
  • func attemptRecovery（optionIndex：Int）-> Bool-エラーから同期的に復元します。真-成功した場合
  • func attemptRecovery（optionIndex：Int、resultHandler：（Bool）-> Void）-非同期オプション、考え方は同じです

使用ガイドを使用すると、すべてがより控えめです。Appleサイトとその周辺を少し検索すると、Swiftの一般発表の前に書かれたエラー処理に関する記事が表示されます。

簡単に：

  • メカニズムはMacOsアプリケーション用に考えられており、ダイアログボックスを表示します
  • もともとはNSErrorを中心に構築されました。
  • RecoveryAttempterオブジェクトはuserInfoのエラー内にカプセル化されます。userInfoはエラーの状態を認識しており、問題に対する最適なソリューションを選択できます。オブジェクトはnilであってはなりません
  • RecoveryAttempterは非公式プロトコルNSErrorRecoveryAttemptingをサポートする必要があります
  • また、userInfoには回復オプションが必要です
  • そしてすべてがmacOS SDKにのみ存在するpresentErrorメソッドの呼び出しに関連付けられています。彼は警告を示しています

    			guard let `self` = self else { return }
    			switch result {
    			case .success:
    				break
    			case .failure(let error):
    				let notEnoughSpace = RecoveryOptions.freeSpace {
    					self.freeSpace()
    				}
    
    				let buyMoreSpace = RecoveryOptions.buyMoreSpace {
    					self.buyMoreSpace()
    				}
    				let options = [notEnoughSpace, buyMoreSpace]
    				let recovableError = RecoverableError(error: error, attempter: .cancalableAttemter(options: options))
    				self.presentAlert(from: recovableError)
    			}
    		}
    	}
    
    	func freeSpace() {
    		let alertViewController = createOKAlert(with: "Free space selected")
    		present(alertViewController, animated: true, completion: nil)
    	}
    
    	func buyMoreSpace() {
    		let alertViewController = createOKAlert(with: "Buy more space selected")
    		present(alertViewController, animated: true, completion: nil)
    	}

最小リクエスト時間


インターネットをオフにして再試行するとします。ローダーを実行します。答えは即座に来て、ミニゲーム「クリッカー」を取得します。点滅アニメーション付き。あまり良くない。



インスタントエラーをプロセスに変えましょう。アイデアはシンプルです-最小のリクエスト時間を作成します。ここでの実装は、ネットワークへのアプローチに依存します。Operationを使用するとします。私にとっては次のようになります。

final class DelayOperation: AsyncOperation {
	private let _delayTime: Double

	init(delayTime: Double = 0.3) {
		_delayTime = delayTime
	}
	override func main() {
		super.main()

		DispatchQueue.global().asyncAfter(deadline: .now() + _delayTime) {
			self.state = .finished
		}
	}
}

struct Task {
    	let closure: () -> Void
    
    	private var _delayTime: Double?
    
    	init(closure: @escaping () -> Void) {
    		self.closure = closure
    	}
    
    	fileprivate init(closure: @escaping () -> Void, time: Double) {
    		self.closure = closure
    		_delayTime = time
    	}
    
    	@discardableResult
    	func run() -> Self {
    		if let delayTime = _delayTime {
    			DispatchQueue.global().asyncAfter(deadline: .now() + delayTime) {
    				self.closure()
    			}
    			return self
    		}
    		closure()
    		return self
    	}
    
    	func delayedTask(time: Double) -> Self {
    		return Task(closure: closure, time: time)
    	}
    }
    
    //    
    func requestObject(completionHandler: @escaping ((Result<Bool, Error>) -> Void)) -> Task {
    		return Task {
    			completionHandler(.failure(NetworkError.internetError))
    		}
    			.delayedTask(time: 0.5)
    		.run()
    	}
