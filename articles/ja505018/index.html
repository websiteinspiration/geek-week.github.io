エピソード1-輸送


2020年1月の最後の週末にセレンの問題の1つに捧げられた対話の結果、誰かが問題を説明するために「なぜこれをやらないのか...」と言われたため、コマンドを説明する一連の記事を書くことにしましたSelenium WebDriverで、なぜ今日のデザインになったのか。

私はこのシリーズのすべてのエピソードでこれを繰り返します-多くの、時には狂気のポイントまで、多くの考えと考えがセレンのあらゆる細部の作品に入れられました。

どうして？

偶然にも、彼が意図したことをどれほどうまく行っているかによって、セレンは世界中の何百万人もの人々によって使用されています。これは、小規模な新興企業からMicrosoftやGoogleまでの幅広い企業が、サイトがすべてのブラウザで機能することを保証する方法です。

Selenium ?


Seleniumの何年にもわたって、私たちはブラウザーと対話する方法としてHTTPを使用することに焦点を当てることになりました。各クライアントバインディングを使用して同じ結果を得ることができるREST風のAPI（REST風-RESTの精神）を作成しました。

HTTPとREST風？マジ？

うん...

HTTPの部分から始めましょう。私たちが最初に始めたとき、私たちは、それぞれのブラウザーに対する最良のアプローチに基づいて、それぞれのブラウザーと対話するさまざまな方法をサポートする必要がありました。たとえば、Internet Explorerの場合、COMコードを記述しました。彼はうまく働いたが、彼からはまだ悪夢がある。 Firefoxについては、行ごとの読み取りモンスターを作成しました。幸いにも、Mozillaの「ブラウザを作成する」アプローチのおかげで、多くのことができました。 Operaでは、DevToolsプロトコルを使用して参加することができました。

したがって、これは特にWebDriverの初期の段階では、N：Mバインディングをサポートする必要があったことを意味します。ここで、Nは言語バインディング、Mはサポートするブラウザーです。この道は良い製品につながりません。私たちはすべての言語が理解できる何かが必要だと判断しました。また、かなり信頼できるものが必要でした。したがって、HTTPを選択することになり、を作成し始めましたJSONWireProtocol。

フレームワークでJSONWireProtocolは、JSONで通信するREST風のインターフェースを構築しました。 RESTっぽいのは、RESTのすべての原則に完全に準拠しているわけではなく、RESTを私たちのタスクの強力なツールにするのに十分な程度に具体化しているからです。

現状についてはどうですか？

Web、インターネット、そして世界は前進します。では、なぜセレンはそうではないのですか？

これは良い質問ですが、私たちが近代化しようとしているのは事実です。残念ながら、ネットワークは、実行されていない場合、機能しない状態を特徴としています。 HTTPはプロトコルとしてかなり信頼できます。また、多重化がどのように機能するかを気にすることなく、テスト用のクラスターを作成できます。これがSelenium Gridが作成された理由であり、複数のデバイスやコンピューターでのテストの整理に関しては、依然として適切な選択です。

ただし、ネットワークのように見えます。

したがって... Chromeを制御するためにChromeデバッグプロトコルを使用するツールがあります。彼らはSeleniumよりもいくつかのことを上手く行っています。これは、ブラウザとの通信方法を選択した結果です。残念ながら、これはChromeの独自のプロトコルであり、他のブラウザからアクセスできるようにすることはGoogleの利益にはなりません。

また、Googleチームからの興味深いデザインソリューションを無視すると、常にオープンな接続が必要になるという問題があります。この場合、WebSocketを使用していますが、インターネットが機能していない間はインターネットを利用できないという上記の私のコメントを思い出してください。 WebSocketは常に再接続します。このチャネルを上下するトラフィックの量の問題もあります。

これは、ローカルコンピューター上でのみ何かとやり取りする場合、操り人形師にとって問題ではありませんが、Circle CIまたはTravisCIなどのCIサービスとAWS Device Farm、Sauce Labs、またはBrowserStackなどのユーザーを統合する場合ランナーが突然インターネットに接続し、このデータが受信者に届くはずです。

ブラウザーベンダーとSeleniumのメンバーで構成されるW3Cブラウザーテストおよびツールワーキンググループは、奇妙なハッカーパッチやこれらの個人的な配信に頼ることなく、最初からブラウザー間の互換性を確保できるように、すべてがどのように見えるかを設計しようとしています。ブラウザ。

エピソード2-ナビゲーション
このエピソードでは、ナビゲーションに関連する膨大な量の作業を見ていきます。

from selenium import webdriver

driver = webdriver.Firefox()
driver.get("https://www.theautomatedtester.co.uk")


上に表示されているものは非常にシンプルに見えますが、ここにあります！

実際、これは私を無神経なインタビューに導きます。URLを入力してEnterキーを押したときにブラウザーで何が発生するかを説明するように求められた場合、ナビゲーションで何が発生しているのかを本人が知らない可能性が高くなります。とにかく... Seleniumとそのナビゲーションに戻ります。トランスポート層 を介してリクエストを送信するとすぐに、本当の楽しみが始まります。どこに行きたいかを見つけ、ブラウザにそこに行くように指示する必要があります。残念ながら、これは自動化に関する多くの問題の最初につながります。

driver.get



証明書


テクノロジーがメインではない大企業で働いたことがある場合は、開発チームがどのような痛みに直面するかを理解しています。これで、この苦痛が2倍になり、テスターや自動化エンジニアが対処しなければならない苦痛を想像できます。

これらの問題の1つは証明書です。企業はけちで、自己署名証明書やその他の変形をします。特に、Seleniumの初期の頃、Let's Encryptのようなサービスはありませんでした。そして今でも、ほとんどの開発者やQAグループがテスト環境やCIサーバーで構成変更にアクセスすることはほとんどありません。証明書を回避する方法を見つける必要がありました。（これは、Seleniumがセキュリティリスクと見なされる最初の理由の1つです。）

したがって、各ブラウザーは、自動化中に誤って構成された証明書をスキップします。彼らがこれに気づかなかった場合、多くのテスター/開発者は自分のサイトをテストする機会がありません。

ここで...最初の問題を回避したら、ページのロードに進む必要があります。

読み込み中


証明書を受け取り次第、ページをダウンロードします。幸いなことに、同等のものよりも複雑なことをする必要はありません。

location = "https://www.theautomatedtester.co.uk"; 
// 
window.location.href = "https://www.theautomatedtester.co.uk";


完了しました...


Seleniumがチームを「完了する」と、払い戻しが行われます。したがって、ページが読み込まれるのを待つだけです。完全に正直に言うと、「読み込み完了」とはどういう意味ですか？

ブラウザはいくつかの異なるイベントを発生させます。ページがいつ表示されるか、そしてそれが何であるかがわかりますreadyState。 Seleniumはこれをすべてテストし、待機しDOMContentLoadedます。

そして、ページにいて、そのページのアンカーにジャンプしようとすると、問題が発生します。次の例を見てみましょう。

from selenium import webdriver

driver = webdriver.Firefox()
driver.get("https://www.theautomatedtester.co.uk")
driver.get("https://www.theautomatedtester.co.uk#someAnchor")


ああ、見てください、ページ読み込みイベントはどれも発生しません！クラス？愚かなブラウザは効率的に動作し、アンカーまでスクロールするだけです。つまり、ページから発生するイベントだけに依存することはできません。

したがって、この場合の特別なコードと、この場合の「準備完了」の意味を理解する必要があります。間違えると、不安定なテストがたくさん作成されます。不安定なテストは人々を不機嫌にします、そして私たちは不機嫌な開発者を望んでいません、世界中にそれらの十分な数があり、不安定なテストなしに。

これらのチェックが完了したら、ページのロード戦略を使用してより速くロードしたい場合は、これらのイベントの見方を操作できます。。これは舗装ユーザー向けの機能なので、今は気にしませんが、ナビゲーションコマンドの速度に影響します。

JavaScriptフレームワークとナビゲーションについて


ここが「楽しさ」がすべて凝縮された場所です。多くのフレームワークは、ページに最初にロードされた後も多くをロードします。単一ページのアプリケーションで作業したことがある、または単にそれを使用したことがある場合、ロード時に多くの要素が表示されることを確認しました。残念ながら、これは、ナビゲーションコマンドの戻りだけに依存することはできないことを意味します。WebDriverWait以下に示すように、コードにコマンドを追加して、テストが正しい状態にあることを確認してから、必要な作業を行う必要があります。

from selenium import webdriver
from selenium.webdriver.support.wait import WebDriverWait

driver = webdriver.Firefox()
driver.get("https://www.theautomatedtester.co.uk")
element = WebDriverWait(driver, 10).until(lambda x: x.find_element_by_id(“someId”))


結論


ページをロードするとき、ページのロードが完了したときにSeleniumが戻ることに常に依存するとは限りません。ページの要素を確認する必要がある場合は、それを実行してください。最初のロード後に、ページ上のJavaScriptがそれをどのように変更できるかを理解してください。
