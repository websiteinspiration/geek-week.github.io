target-actionとdelegateをクロージャーに置き換える方法

Appleは、iOSアプリケーションでデータとイベントを処理するためのさまざまなオプションを提供しています。UIControlイベント処理は、target-actionパターンを通じて発生します。UIControlのドキュメントには次のように書かれています：
ターゲットアクションメカニズムにより、アプリのコントロールを使用するために作成するコードが簡素化されます
ボタンのクリックを処理する例を見てみましょう：

private func setupButton() {
    let button = UIButton()
    button.addTarget(self, action: #selector(buttonTapped(_:)), for: .touchUpInside)
}
// -   
@objc private func buttonTapped(_ sender: UIButton) { }

ボタンクリックの構成と処理は、コード内で互いに別々に配置されます。したがって、必要以上のコードを記述する必要があります。イベントとコントロールの数が増えると問題が発生します。

UITextFieldはデリゲートパターンを使用してテキストを編集および検証します。このパターンの長所と短所については詳しく説明しません。詳しくはこちらをご覧ください。

1つのプロジェクトでデータを処理するさまざまな方法を使用すると、コードの読み取りと理解が困難になることがよくあります。この記事では、便利なクロージャー構文を使用して、すべてを単一のスタイルにする方法を理解します。

なぜ必要なのか


最初は、GitHubで既製のソリューションを試し、しばらくの間クロージャーを使用していました。しかし、時間の経過とともに、メモリリークが発見されたため、サードパーティのソリューションを放棄する必要がありました。そして、その機能のいくつかは私たちにとって不快に思えました。その後、独自のソリューションを作成することが決定されました。

クロージャを使用して次のように書くことができれば、結果に非常に満足します。

textField.shouldChangeCharacters { textField, range, string in
    return true
}

基本的な目標：

  • クロージャで、元のタイプを維持しながら、textFieldへのアクセスを提供します。これは、たとえばボタンをクリックして型キャストせずにインジケーターを表示することで、クロージャー内のソースオブジェクトを操作するためです。
  • , . , .touchUpInside onTap { }, shouldChangeCharacters UITextField , .


主なアイデアは、すべてのメッセージをインターセプトしてクロージャを発生させるオブザーバオブジェクトを用意することです。

まず、オブザーバーを維持する方法を決定する必要があります。スウィフトは私たちに選択する力を与えます。たとえば、ディクショナリを格納する追加のシングルトンオブジェクトを作成できます。キーは監視対象オブジェクトの一意のIDで、値は監視者自身です。この場合、オブジェクトのライフサイクルを手動で管理する必要があり、メモリリークや情報の損失につながる可能性があります。オブジェクトを関連オブジェクトとして保存すると、このような問題を回避できます。

デフォルトの実装でObserverHolderプロトコルを作成し、このプロトコルに準拠する各クラスがオブザーバーにアクセスできるようにします。

protocol ObserverHolder: AnyObject {
    var observer: Any? { get set }
}

private var observerAssociatedKey: UInt8 = 0

extension ObserverHolder {
    var observer: Any? {
        get {
            objc_getAssociatedObject(self, &observerAssociatedKey)
        }
        set {
            objc_setAssociatedObject(
                self, 
                &observerAssociatedKey, 
                newValue, 
                .OBJC_ASSOCIATION_RETAIN_NONATOMIC
            )
        }
    }
}

次に、UIControlのプロトコルへの準拠を宣言する必要があります。

extension UIControl: ObserverHolder { }

UIControl（およびUITextFieldを含むすべての子孫）には、オブザーバーが格納される新しいプロパティがあります。

UITextFieldDelegateの例


オブザーバーはUITextFieldのデリゲートになります。つまり、オブザーバーはUITextFieldDelegateプロトコルに準拠する必要があります。UITextFieldの元の型を保存するには、ジェネリック型Tが必要です。そのようなオブジェクトの例：

final class TextFieldObserver<T: UITextField>: NSObject, UITextFieldDelegate {
    init(textField: T) {
        super.init()
        textField.delegate = self
    }
}

各デリゲートメソッドには、個別のクロージャーが必要です。そのようなメソッドの内部では、型をTにキャストし、対応するクロージャを引き起こします。TextFieldObserverコードを追加し、例では1つのメソッドのみを追加します。

var shouldChangeCharacters: ((T, _ range: NSRange, _ replacement: String) -> Bool)?

func textField(
    _ textField: UITextField,
    shouldChangeCharactersIn range: NSRange,
    replacementString string: String
) -> Bool {
    guard 
        let textField = textField as? T, 
        let shouldChangeCharacters = shouldChangeCharacters 
    else {
        return true
    }
    return shouldChangeCharacters(textField, range, string)
}

クロージャーを備えた新しいインターフェースを書く準備ができました：

extension UITextField {
    func shouldChangeCharacters(handler: @escaping (Self, NSRange, String) -> Bool) { }
}

何かがうまくいかなかった場合、コンパイラはエラーをスローします：
'Self'は、プロトコルで、またはクラスのメソッドの結果としてのみ使用できます。「UITextField」という意味ですか
空のプロトコルは私たちに役立ち、その拡張では、UITextFieldへの新しいインターフェイスを記述し、Selfを制限します。

protocol HandlersKit { }

extension UIControl: HandlersKit { }

extension HandlersKit where Self: UITextField {
    func shouldChangeCharacters(handler: @escaping (Self, NSRange, String) -> Bool) { }
}

コードがコンパイルされ、TextFieldObserverを作成してデリゲートとして指定するために残ります。さらに、オブザーバーがすでに存在する場合は、他のクロージャーを失わないように、オブザーバーを更新する必要があります。

func shouldChangeCharacters(handler: @escaping (Self, NSRange, String) -> Bool) {
    if let textFieldObserver = observer as? TextFieldObserver<Self> {
        textFieldObserver.shouldChangeCharacters = handler
    } else {
        let textFieldObserver = TextFieldObserver(textField: self)
        textFieldObserver.shouldChangeCharacters = handler
        observer = textFieldObserver
    }
}

これで、このコードは機能し、使用できるようになりましたが、改善することができます。別のメソッドでTextFieldObserverを作成および更新します。クロージャーの割り当てのみが異なり、ブロックの形式で渡します。拡張HandlersKitの既存のコードを更新します。

func shouldChangeCharacters(handler: @escaping (Self, NSRange, String) -> Bool) {
    updateObserver { $0.shouldChangeCharacters = handler }
}

private func updateObserver(_ update: (TextFieldObserver<Self>) -> Void) {
    if let textFieldObserver = observer as? TextFieldObserver<Self> {
        update(textFieldObserver)
    } else {
        let textFieldObserver = TextFieldObserver(textField: self)
        update(textFieldObserver)
        observer = textFieldObserver
    }
}

追加の改善


メソッドをチェーンする機能を追加します。これを行うには、各メソッドがSelfを返し、@ discardableResult属性を持っている必要があります。

@discardableResult
public func shouldChangeCharacters(
    handler: @escaping (Self, NSRange, String) -> Bool
) -> Self

private func updateObserver(_ update: (TextFieldObserver<Self>) -> Void) -> Self {
    ...
    return self
}

クロージャーでは、UITextFieldへのアクセスが常に必要であるとは限らないため、そのような場所では毎回 ` _ in`を記述する必要がないため、同じ名前で、必要なSelfを持たないメソッドを追加します。

@discardableResult
func shouldChangeCharacters(handler: @escaping (NSRange, String) -> Void) -> Self {
    shouldChangeCharacters { handler($1, $2) }
}

このアプローチのおかげで、より便利なメソッドを作成できます。たとえば、UITextFieldでテキストを変更すると、最終的なテキストがわかっている場合に便利な場合があります。

@discardableResult
public func shouldChangeString(
    handler: @escaping (_ textField: Self, _ from: String, _ to: String) -> Bool
) -> Self {
    shouldChangeCharacters { textField, range, string in
        let text = textField.text ?? ""
        let newText = NSString(string: text)
            .replacingCharacters(in: range, with: string)
        return handler(textField, text, newText)
    }
}

できた！示されている例では、1つのUITextFieldDelegateメソッドを置き換え、残りのメソッドを置き換えるには、同じ原理でクロージャーをTextFieldObserverとHandlersKitプロトコルの拡張に追加する必要があります。

target-actionをクロージャーに置き換える


target-actionの1つのオブザーバーとデリゲートをこのフォームに格納すると複雑になるので、イベントのUIControlに別の関連オブジェクトを追加することをお勧めします。イベントごとに個別のオブジェクトを保存します。そのようなタスクには辞書が最適です。

protocol EventsObserverHolder: AnyObject {
    var eventsObserver: [UInt: Any] { get set }
}

EventsObserverHolderのデフォルトの実装を追加することを忘れないでください。ゲッターにすぐに空の辞書を作成します。

get {
    objc_getAssociatedObject(self, &observerAssociatedKey) as? [UInt: Any] ?? [:]
}

オブザーバーは1つのイベントのターゲットになります。

final class EventObserver<T: UIControl>: NSObject {
    init(control: T, event: UIControl.Event, handler: @escaping (T) -> Void) {
        self.handler = handler
        super.init()
        control.addTarget(self, action: #selector(eventHandled(_:)), for: event)
    }
}

このようなオブジェクトでは、1つのクロージャを格納するだけで十分です。TextFieldObserverの場合と同様に、アクションが完了すると、オブジェクトのタイプが表示され、クロージャが発生します。

private let handler: (T) -> Void

@objc private func eventHandled(_ sender: UIControl) {
    if let sender = sender as? T {
        handler(sender)
    }
}

UIControlのプロトコル準拠を宣言します。

extension UIControl: HandlersKit, EventsObserverHolder { }

デリゲートを既にクロージャに置き換えている場合は、HandlersKitを再度照合する必要はありません。
UIControlの新しいインターフェイスを作成するために残っています。新しいメソッド内で、オブザーバーを作成し、キーevent.rawValueを使用してそれをeventsObserver辞書に保存します。

extension HandlersKit where Self: UIControl {

    @discardableResult
    func on(_ event: UIControl.Event, handler: @escaping (Self) -> Void) -> Self {
        let observer = EventObserver(control: self, event: event, handler: handler)
        eventsObserver[event.rawValue] = observer
        return self
    }
}

頻繁に使用するイベントのインターフェースを補足できます。

extension HandlersKit where Self: UIButton {

    @discardableResult
    func onTap(handler: @escaping (Self) -> Void) -> Self {
        on(.touchUpInside, handler: handler)
    }
}

概要


ばんざーい、target-actionとdelegatesをクロージャーに置き換えて、コントロール用の単一のインターフェースを取得しました。直接アクセスできるため、メモリリークについて考えたり、コントロール自体をクロージャにキャプチャしたりする必要はありません。

ここに完全なコード：HandlersKit。このリポジトリには、UIControl、UIBarButtonItem、UIGestureRecognizer、UITextField、UITextViewの例がさらにあります。

このトピックをより深く理解するために、EasyClosureに関する記事を読み、反対側から問題の解決策を検討することも提案します。
