HTTP / 3：基盤を壊し、勇敢な新しい世界

今から20年以上にわたり、HTTPプロトコルを使用してWebページを表示しています。ほとんどのユーザーはそれが何であり、どのように機能するかについてさえ考えていません。他の人は、HTTPのどこかにTLSがあり、その下にTCPがあり、その下にIPなどがあることを知っています。そして他の-異端者-TCPは最後の世紀であると信じ、彼らはより速く、より信頼性が高く、より安全な何かを望んでいます。しかし、彼らは新しい理想的なプロトコルを発明しようとして、80年代のテクノロジーに戻り、彼らに勇敢な新しい世界を築こうとしています。


少し歴史：HTTP / 1.1


1997年、HTTPバージョン1.1のテキスト交換プロトコルがRFCを獲得しました。当時、このプロトコルはブラウザで数年間使用され、新しい規格はさらに15年間続きました。このプロトコルは要求/応答ベースでのみ機能し、主にテキスト情報を送信することを目的としていました。

HTTPは、TCPプロトコル上で動作するように設計されており、宛先へのパケットの確実な配信を保証します。 TCPは、エンドポイント間の信頼できる接続の確立と維持、およびトラフィックのセグメント化に基づいています。セグメントには独自のシーケンス番号とチェックサムがあります。セグメントの1つが突然来ないか、間違ったチェックサムが付いている場合、失われたセグメントが復元されるまで送信は停止します。

HTTP / 1.0では、各リクエストの後にTCP接続が閉じられました。以来、それは非常に無駄だったTCP接続（3-Way-Handshake）の確立は迅速なプロセスではありません。 HTTP / 1.1では、キープアライブメカニズムが導入されました。これにより、単一の接続を複数のリクエストに再利用できます。ただし、簡単にボトルネックになる可能性があるため、同じホストへの複数のTCP / IP接続は、異なるHTTP / 1.1実装で許可されています。たとえば、Chromeおよび最近のバージョンのFirefoxでは、最大6つの接続が許可されます。

暗号化も他のプロトコルに任せることになっていたため、TCPを介してTLSプロトコルが使用されたため、データは確実に保護されましたが、接続の確立に必要な時間がさらに増加し​​ました。その結果、ハンドシェイクプロセスは次のようになります。

したがって、HTTP / 1.1にはいくつかの問題がありました。

  • 接続設定が遅い。
  • 1つのTCP接続が1つの要求に使用されます。つまり、残りの要求は別の接続を見つけるか、現在の要求が解放するまで待つ必要があります。
  • プルモデルのみがサポートされています。サーバープッシュに関する標準には何もありません。
  • 見出しはテキストで送信されます。

server-pushがWebSocketプロトコルを使用して何らかの方法で実装されている場合、残りの問題はより根本的に対処する必要がありました。

少し現代的：HTTP / 2


2012年に、SPDYプロトコル（「スピード」と発音）の作業がGoogleの内部で始まりました。このプロトコルは、HTTP / 1.1の基本的な問題を解決するように設計されており、同時に下位互換性を維持する必要がありました。2015年、IETFワーキンググループは、SPDYプロトコルに基づくHTTP / 2仕様を導入しました。HTTP / 2の違いは次のとおりです。

  • バイナリシリアル化。
  • 複数のHTTPリクエストを単一のTCP接続に多重化します。
  • サーバーからプッシュアウト（WebSocketなし）。

プロトコルは大きな前進でした。最初のバージョンを大幅に上回っており、複数のTCP接続を作成する必要がありません。1つのホストへのすべての要求が1つに多重化されます。つまり、1つの接続にいくつかのいわゆるストリームがあり、それぞれに独自のIDがあります。ボーナスは、ボックス化されたサーバープッシュです。

ただし、乗算は別の基本的な問題につながります。 1つのサーバーに対して5つのリクエストを非同期で実行するとします。 HTTP / 2を使用する場合、これらのすべての要求は同じTCP接続内で実行されます。つまり、要求のセグメントの1つが失われるか正しく到着しない場合、失われたセグメントが復元されるまで、すべての要求と応答の送信が停止します。明らかに、接続品質が悪いほど、HTTP / 2の動作は遅くなります。Daniel Stenbergによれば、失われたパケットが全体の2％を占める状況では、ブラウザのHTTP / 1.1は1つではなく6つの接続を開くため、HTTP / 2よりもパフォーマンスが優れています。

この問題は「ヘッドオブラインブロッキング」と呼ばれ、残念ながらTCPを使用して解決することはできません。

Daniel Steinbergによるイラスト

結果として、HTTP / 2標準の開発者は素晴らしい仕事をし、OSIモデルのアプリケーションレベルで実行できるほとんどすべてのことを行いました。トランスポートレベルまで下がって、新しいトランスポートプロトコルを発明する時が来ました。

新しいプロトコルが必要です：UDPとTCP


完全に新しいトランスポート層プロトコルを導入することは、今日の現実では解決できない課題であることがすぐに明らかになりました。事実は、グランドまたはミドルボックス（ルーター、ファイアウォール、NATサーバーなど）がトランスポートレベルを認識しており、新しいものを教えることは非常に難しい作業です。さらに、トランスポートプロトコルのサポートはオペレーティングシステムのカーネルに組み込まれており、カーネルの変更もそれほど進んでいません。

そして、ここであきらめて「もちろん、私たちは好みと遊女で新しいHTTP / 3を発明しますが、10〜15年で実装されます（この後、ほとんどの鉄の部品は交換されます）」と言うことは可能ですが、もう1つはありません。明白なオプション：UDPプロトコルを使用します。はい、そうです。90年代の終わりと0の初めにLANにファイルをスローしたのと同じプロトコルです。今日の鉄のほとんどすべては、それを扱う方法を知っています。

TCP上のUDPの利点は何ですか？まず、Ironが知っているトランスポートレベルのセッションはありません。これにより、エンドポイントのセッションを自分で決定し、そこで発生する競合を解決できます。つまり、（TCPのように）1つまたは複数のセッションに限定されず、必要なだけセッションを作成できます。第2に、UDPを介したデータ転送は、TCPを介するよりも高速です。したがって、理論的には、HTTP / 2で達成された今日の速度上限を打ち破ることができます。

ただし、UDPは信頼できるデータ転送を保証しません。実際、パケットを送信するだけで、相手側で受信されることを期待しています。受け取っていません？まあ、運はありません...これは成人向けのビデオを送信するには十分でしたが、より深刻なものには信頼性が必要です。つまり、UDPに何か他のものを巻き付ける必要があります。

HTTP / 2の場合と同様に、新しいプロトコルを作成する作業はGoogleで2012年に開始されました。つまり、SPDYの作業が始まるのとほぼ同時にです。2013年にジムロスキンドはQUICプロトコル（Quick UDP Internet Connections）を一般に公開し、2015年にはIETFを標準化するためにインターネットドラフトが導入されました。当時、GoogleでRoskindが開発したプロトコルは標準のものとは非常に異なっていたため、GoogleバージョンはgQUICと呼ばれていました。

QUICとは


まず、すでに述べたように、これはUDPのラッパーです。 QUIC接続はUDPの上に発生し、HTTP / 2と同様に、いくつかのストリームが存在する可能性があります。これらのストリームはエンドポイントにのみ存在し、個別に提供されます。パケット損失が1つのストリームで発生した場合でも、他のストリームには影響しません。

ダニエルスタインバーグのイラスト

第2に、暗号化は個別のレベルではなくプロトコルに含まれるようになりました。これにより、1つのハンドシェイクで接続を確立し、公開鍵を交換できます。また、トリッキーな0-RTTハンドシェイクメカニズムを使用して、通常、握手による遅延を回避できます。さらに、個々のデータパケットを暗号化できるようになりました。これにより、ストリームからのデータ受信の完了を待つのではなく、受信したパケットを個別に復号化できます。この操作モードは、TCPではまったく不可能でした。 TLSとTCPは互いに独立して機能し、TLSはTCPデータがどの部分に分割されるかを知ることができませんでした。そのため、TCPセグメントに1対1で適合し、個別に復号化できるように、セグメントを準備できませんでした。これらすべての改善により、QUICはTCPと比較してレイテンシを低くすることができます。

3番目に、ライトストリームの概念により、クライアントのIPアドレスから接続を解放できます。これは、たとえば、クライアントがあるWi-Fiアクセスポイントから別のWi-Fiアクセスポイントに切り替えて、IPを変更する場合に重要です。この場合、TCPを使用すると、時間のかかるプロセスが発生し、その間に既存のTCP接続がタイムアウトになり、新しいIPから新しい接続が作成されます。 QUICの場合、クライアントは単に、古いストリームIDを持つ新しいIPからサーバーにパケットを送信し続けます。なぜならストリームIDは一意になり、再利用されなくなりました。サーバーは、クライアントがIPを変更したことを理解し、失われたパケットを送信して、新しいアドレスへの通信を続行します。

第4に、QUICはオペレーティングシステムではなく、アプリケーションレベルで実装されます。一方で、これにより、プロトコルへの変更をより速く行うことができます。アップデートを取得するには、OSの新しいバージョンを待つのではなく、ライブラリをアップデートするだけです。一方、これはプロセッサの消費量の大幅な増加につながります。

そして最後に、見出し。ヘッダー圧縮は、QUICとgQUICで異なる点を参照するだけです。私はこれに多くの時間を費やす理由はないと思います。標準化のために提出されたバージョンでは、ヘッダー圧縮はHTTP / 2のヘッダー圧縮と可能な限り類似していると言えます。詳細については、こちらをご覧ください。

それはどれくらい速いですか？


これは難しい質問です。実際には、基準はありませんが、測定する特別なことは何もありません。おそらく、私たちが持っている唯一の統計は、2013年からgQUICを使用しており、2016年にChromeブラウザからサーバーに向かうトラフィックの約90％がQUICを使用しているとIETFに報告した Googleの統計です。同じプレゼンテーションで、gQUICを使用すると、ページの読み込みが約5％速くなり、ストリーミングビデオのフリーズがTCPに比べて30％少ないことが報告されています。

2017年に、Arash Molavi Kakhkiが率いる研究者グループは、TCPと比較してgQUICパフォーマンスの研究に関する大規模な研究を発表しました。
この調査では、ネットワークパケットの混合に対する不安定性、チャネル容量の不公平、小さい（最大10 kb）オブジェクトの転送が遅いなど、gQUICのいくつかの弱点が明らかになりました。ただし、後者は0-RTTを使用して補正できます。調査した他のすべてのケースでは、gQUICはTCPと比較して速度の向上を示しました。特定の数について話すのは難しいです。研究自体または短い投稿を読むことをお勧めします。

ここで、このデータは特にgQUICに関するものであり、開発中の標準には無関係であると言わなければなりません。QUICに何が起こるか：これまでのところ、謎は7つのシールの背後にありますが、gQUICによって識別された弱点が考慮されて修正されることが期待されます。

少し未来：HTTP / 3はどうですか？


そして、ここではすべてが非常に明確です。APIはまったく変更されません。すべては、HTTP / 2の場合とまったく同じままです。まあ、APIが同じである場合、HTTP / 3への移行は、バックエンドのQUICを介したトランスポートをサポートするライブラリの最新バージョンを使用して決定する必要があります。確かに、長い間、古いバージョンのHTTPにフォールバックし続ける必要があります。これでインターネットはUDPに完全に切り替える準備ができていません。

問題は何ですか


私たちは現実の世界に住んでいます。そこでは、単一の大きなテクノロジーが抵抗に会うことなく大衆に行き渡ることはできません。QUICも例外ではありません。

最も重要なのは、「https：//」が443番目のTCPポートにつながる事実ではなくなったことをブラウザーに何らかの方法で説明する必要があることです。 TCPがまったくない場合があります。これを行うには、Alt-Svcヘッダーを使用します。これにより、ブラウザは、このWebサイトが、そのようなアドレスで、そのようなプロトコルでアクセス可能であることも通知されます。理論的には、これは時計のように機能するはずですが、実際には、DDoS攻撃を回避するためにファイアウォールでUDPを無効にできるなどの事実に遭遇します。

ただし、UDPが禁止されていない場合でも、クライアントは、IPアドレスでTCPセッションを保持するように構成されているNATルーターの背後にある可能性があります。ハードウェアセッションがなく、NATが接続を保持しないUDPとQUICセッションを使用します。絶えず壊れます。

これらすべての問題は、以前はインターネットコンテンツの転送にUDPが使用されていなかったという事実に関連しており、ハードウェアメーカーはこれが発生することを予測できませんでした。同様に、管理者はまだQUIC用にネットワークを適切に構成する方法をまだ理解していません。この状況はゆっくりと変化し、どのような場合でも、そのような変化は新しいトランスポート層プロトコルの導入よりも時間がかかりません。

さらに、すでに説明したように、QUICはプロセッサーの使用率を大幅に向上させます。ダニエルステンバーグ氏は、プロセッサの成長を最大3回評価しました。
