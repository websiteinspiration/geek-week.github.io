あられのマルチモジュールプロジェクトの依存関係管理


前書き


記事はネイティブandroidプロジェクトを対象に書かれていますが、gradleはユニバーサルアセンブリシステムであるため、原則として、gradleが収集できる他のプロジェクトに適しています。アイデアは私のものではありません。私はJake Wharton SdkSearch -android sdkのドキュメントを検索するための一連のプログラムのgithubプロジェクトからそれを得ました。

問題


最近のアプリケーションは、ライブラリを使用せずに書くことはほとんど不可能（非現実的）です。依存関係の出現により、バージョン管理のタスクが発生します。おそらく、中規模の単一モジュールのandroidアプリケーションの場合、これは問題ではありません。コードを再利用する大規模なプロジェクトでは、問題が関係しています。特に、バージョン管理が合理的ではないサポート androidxライブラリのバージョン管理では、AndroidXのバージョンを確認するだけです。

決定


非常に多くの依存関係があるため、すべてのモジュールがライブラリの1つのバージョンに依存していることが非常に重要です。
Gradleはスクリプト言語としてgroovyを使用します。これは動的型付けと相まって、依存関係管理を体系化する便利な方法を提供します。

DSL gradでは、ext ExtraPropertiesExtensionオブジェクトを使用してプロジェクトにプロパティを追加できます。また、モジュールスクリプト内では、プロジェクトスクリプト（ルートモジュール）にアクセスできます。これにより、ルートスクリプトで依存関係のすべてのバージョンを宣言でき、内部の任意のモジュールからそれらを参照できます。

たとえば、Androidアプリケーションに次の依存関係を使用させます：kotlin、kotlin stdlib、kotlin junit、facebook sdk、androidx、google material

Root build.gradle：

いくつか のポイント：複数の依存関係が同じバージョン（たとえば、1つの大きなSDKの一部）を持っている場合、そのバージョンはKotlinやFacebookのバージョンなどのバージョンプロパティに追加されます。依存関係がgoogleマテリアルなどの1行の形式である場合、バージョンを作成することは現実的ではありません。 androidxライブラリのバージョンを取り出す必要もありません。 Googleは、個々のライブラリのリリースを高速化するために、バージョンを互いに揃えることを拒否しました。

この定義の結果、すべての子モジュールで、依存関係の宣言はバージョン管理のない簡潔な階層になります。現在、すべてのモジュールは同じライブラリバージョンに依存しています。

build.gradleのモジュール依存関係セクションの例

dependencies {
    implementation deps.kotlin.stdlib.jdk

    implementation deps.androidx.appCompat
    implementation deps.androidx.browser
    implementation deps.androidx.cardView
    implementation deps.androidx.constraintLayout
    implementation deps.androidx.ktx
    implementation deps.androidx.multidex
    implementation deps.androidx.recyclerView

    implementation deps.material

    implementation deps.fb.core
    implementation deps.fb.login
    implementation deps.fb.share

   testImplementation deps.kotlin.test.jdk
}

何が起こったのか-モジュールのライブラリバージョンが消え、ライブラリの階層が表示されました。
また、モジュールがルートスクリプトに記述されているすべてのライブラリに依存する必要はなく、必要なライブラリにのみ依存していることにも注意してください。
