イベントをサブスクライブする別のアプローチ、またはEventObjectは本当に必要ですか？

概要


この記事の目的は、別の視点から、イベント配信システムの説明を見る試みです。
 
この記事の執筆時点では、ほとんどの主要なphpフレームワークは、EventObjectの記述に基づいてイベントシステムを実装してい  ます。
 
これはphpの標準になりました。これは最近、psr / event-dispatcher標準の採用によって確認されました。
 
しかし、イベントオブジェクトの説明は、リスナーの開発にほとんど役立たないことがわかります。猫の下の詳細については。


何が問題ですか


開発でEventObjectを使用する人の役割と目標を見てみましょう。


  1. イベントを生成することにより、プロセスにサードパーティの指示を挿入する機能を提供する開発者（A）。


    開発者は、インターフェイスを介してEventObjectまたはその署名を記述します。


    EventObjectを説明する場合、開発者の目標は、他の開発者にデータオブジェクトの説明を提供し、場合によっては、このオブジェクトを介したメインスレッドとの相互作用のメカニズムを説明することです。


  2. 「リスナー」を説明する開発者（B）。


    開発者は、指定されたイベントをサブスクライブします。ほとんどの場合、リスナーの説明は呼び出し可能なタイプを満たす必要があります。


    同時に、開発者はリスナーのクラスやメソッドの命名に恥ずかしがり屋ではありません。ただし、慣例により、ハンドラーが引数としてEventObjectを受け取るという制限があります。

    namespace MyFramework\Events;

    class ListenerProvider {

        private $handlerAssociation = [];

        public function addHandlerAssociation(
            string $handlerInterfaceName, 
            string $handlerMethodName, 
            string $eventName
        ) {
            $this->handlerAssociation[$handlerInterfaceName] = [
                'methodName' => $handlerMethodName,
                'eventName' => $eventName
            ];
        }

        public function addHandler(object $handler) {
            $hasAssociation = false;

            foreach( $this->handlerAssociation as $handlerInterfaceName => $handlerMetaData ) {
                if ( $handler interfaceof $handlerInterfaceName ) {
                    $methodName = $handlerMetaData['methodName'];
                    $eventName = $handlerMetaData['eventName'];
                    $this->addListener($eventName, [$handler, $methodName]);

                    $hasAssociation = true;
                }
            }

            if ( !$hasAssociation ) {
                throw new \Exception('Unknown handler object');
            }
        }
    }
