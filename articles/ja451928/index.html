AMPページでウェブ解析を設定する方法

こんにちは、ハブル。私はWrike Marketingのデータアナリストです。すべての広告データ、LTVモデリング、およびチームがすべてのソースで最も効果的な広告を作成するのに役立つその他の技術的なタスクを収集して分析しています。最近、AMPページでデータ収集を設定する問題に遭遇し、トピックに関する情報がほとんど見つからなかったため、このタスクに対処する方法を説明することにしました。



Wrikeでは、ウェブサイトチーム、リードジェネレーション、マーケティングオペレーション、コンテンツマネージャー、マーケティング管理、経営幹部など、さまざまな関係者にサービスを提供するウェブ分析システムを構築しました。アナリストのチームがサイトで収集されたデータの完全性、一貫性、適時性を維持することは非常に重要です。これは、多数のレポートがそれらに基づいて作成され、予想収益が計算されて、広告キャンペーンの効果をリアルタイムで評価するためです。

クライアント側では、分析はGoogleタグマネージャー（GTM）で実行されます。このソリューションを使用すると、チームはウェブサイトの専門家が参加しなくても、分析スクリプトを簡単に追加および変更できます。インストールされたスクリプトは、データのエンドポイントごとに3つのグループに分類できます。

  1. グーグルアナリティクス
  2. , PostgreSQL;
  3. (LinkedIn, Twitter , ).

内部データベースは、Google Analyticsおよびサードパーティのサイトと通信します。交換中、通常、ユーザーはアナリティクスによって割り当てられたクライアントIDで識別されます。これは、データベースとGoogleアナリティクスの両方にユーザーパラメータとして保存されます。私たち自身のために、私たちはクッキーから価値を取り入れています_ga。



最近、ウェブサイトチームは企業のブログページをAMP標準に適合させました。つまり、AMP（Accelerated Mobile Pages）は、モバイルデバイス用のページを作成するための特別な標準であり、作業を大幅に高速化できます。ページが標準に準拠している場合、AMPプロジェクトはページをCDNにキャッシュし、ページはモバイルデバイスのGoogle検索に埋め込まれます。



検索結果でのブログのページ。リンクの左側にあるAMPアイコンは、ページが検索ページの直接フレーム（iframe）で開くことを意味します。

特に、AMPページの高速読み込みは、ページコードに実行可能なユーザーJSが存在しないことで実現され（scriptタイプapplication/ld+jsonまたはがない場合、タグは禁止されていtext/plainます）、ページ自体は事前に最適化されたタグの限られたセットから構築されます。Web分析の場合、これは成功したテストでした。

AMPページをエコシステムに追加するには、2つの課題があります。

  1. 従来のJSベースのデータ収集方法は機能しなくなりました。
  2. AMPはCookieではなくローカルストレージを使用し、クライアントIDは完全に異なるマスクによって生成されます。



問題は個別に検討します。

情報収集


GTMでは、AMPページに適した個別のコンテナを作成できますが、後者では、追加できるタグの種類が大幅に減少しています。最大の問題は、もちろん、通常スクリプトを配置するカスタムHTMLタイプがないことです。これと各データエンドポイントのその他の問題を解決する方法を見てみましょう。

グーグルアナリティクス


これは最も簡単なカテゴリです。AMPコンテナには、Googleアナリティクスをインストールできるユニバーサルアナリティクスタグタイプがあります。ページビューとイベントの送信に加えて、ユーザーパラメーターとインジケーターのみが分析に含まれます。GTMで通常のページに使用できるコンテンツグループ、eコマースデータ、およびその他のフィールドは、ここでは設定できません。

Yandex.MetricaをGoogleアナリティクスと一緒に使用すると、配信することもできますが、GTMでは機能しません。開発者を接続する必要があります。ここでは、詳細な手順を見つけることができます。


AMPのGTMを介してGoogleアナリティクスにページビューを送信する例

自分のログ


カスタム画像タグを使用して、リクエストを送信できます。バックエンドでは、GETリクエストパラメータからデータを書き込むピクセルを提供することが重要です。送信データを構成するには、要求されたページのアドレスに必要な変数を追加し、「キャッシュ無効化を有効にする」オプションを使用してピクセルがキャッシュされないようにすることを忘れないでください。
たとえば、ピクセルを使用してクライアントIDとリファラーを含むページの表示に関するデータを送信できます 。GTM変数へのリンクは中括弧で示され、GTMインターフェースの対応するセクションで設定できます。もちろん、組み込み変数のセットは制限されています。 JSが存在しないため、固有値を計算することも不可能です。必要に応じて、AMP変数を使用してクライアント側で必要な値を計算できます-それらはGTMからアクセスできます。

//www.your-site.com/log?event=pageview&page={{Page Path}}&ga={{Page Client ID}}&referrer={{Document Referrer}}



AMPは主にオーガニックトラフィックを受信するため、広告サイトに複雑なトラッカーをインストールする必要はありません。したがって、追跡を構成するのに十分な組み込み変数がありました。

その他のトラッカー


AMPと統合されていない他のトラッカーを適応させることは非常に困難です。何が適応可能で何が適応不可能かを理解するには、これらのトラッカーから送信されたリクエストを確認する必要があります。トラッカーから送信されたパラメータがGTMで利用できる場合は、同じカスタム画像を使用して必要なデータを送信できます。カウンターにnoscript部分がある場合、通常はGTMに転送できます。

多くのトラッカーを放棄しなければなりませんでしたが、AMPへの適応は、それがもたらす利点に比べて労働集約的すぎるように思われました。

クライアントIDの統合


AMPと組み合わせることで、ページをロードするためのオプションが1つではなく、3つまで可能になります。通常のバージョンと合わせて、4つが取得されます。

  1. 通常版（としましょうwww.your-site.com/blog/article-name/）
  2. サイトに直接ダウンロードします（ここでは、キャッシュのために記事を掲載していますwww.your-site.com/blog/article-name/amp/）。
  3. キャッシュされたバージョンAMP CDN（アドレスは次のようなものになります。www-your-site-com.cdn.ampproject.org/v/s/www.your-site.com/blog/article-name/amp/）
  4. 検索結果のiframe経由でダウンロード：iframeは段落3と同じアドレスを参照しwww.google.comますが、正式にはユーザーが閲覧します。

分析チームがAMPを分析システムに統合する措置を講じない場合、クライアントIDは、ドメインにある通常のバージョンとAMPバージョンの間でのみ転送されます。


デフォルトでは、ドメインにあるページのみがクライアントデータとセッションを交換します。

それ以外の場合は、クライアントIDは送信されません。したがって、自動化の観点から、検索結果の記事を読んでからサイトにアクセスしたユーザーは、2人のユーザーです。その結果、分析では「新規」ユーザーの急激な増加が見られ、ソースのデータは現実を反映しておらず、内部分析システムでは、最初の訪問からコンバージョンまでのユーザーの真の経路を作成する機会を失います。

解決策1.クライアントID API


Googleでは、クライアントID APIの使用を推奨しています。

このソリューションの実装は非常に簡単です。AMPページにオプトインメタタグを追加し、通常のサイトページのGoogleアナリティクスコードに構成を追加する必要があります。ただし、このソリューションには多くの欠点があります。

  1. それは、一対のGoogle検索でのみ機能します-あなたのサイト。
  2. これは一方向でのみ機能します。ユーザーが最初にAMPにログインした場合、通常のサイトではAMP識別子を使用します。ユーザーが既にサイトにアクセスしていて、AMPにログインしている場合は、新しい識別子が生成されます。
  3. AMP識別子を使用します。内部データベースが通常のクライアントIDで構築されている場合、これにより予期しない問題が発生する可能性があります。


Client ID APIを使用すると、ユーザーデータを検索結果からドメインに転送できますが、これは一方向でしか機能しません。

ソリューション2.カスタムamp-analytics構成


フルサイズバージョンとは異なり、AMPでは、TMはJSではなくJSONファイルを介して接続されます。設定ファイルには、すべてのタグ、トリガー、その他のGTM設定に関する情報が含まれています。トップレベルでは、ファイルは次のようになります。

{
   "requests":{...},
   "triggers":{...},
   "vars":{...},
   "transport":{...},
   "linkers":{...},
}

varsレベルには、クライアントIDに関する情報が含まれています。

   "vars":{
      "clientId":"CLIENT_ID(AMP_ECID_GOOGLE,,_ga)",
   }

GTMは、このフィールドで渡されるクライアントIDを使用します。したがって、このファイルをダウンロードしてメインドメインに配置し、クライアントIDをCookieに保存されているもの（メインドメインで利用可能）に置き換えることができます。これを行うには、メインドメインでJSONファイルジェネレーターを作成します。

  1. JSON- GTM https://www.googletagmanager.com/amp.json?id=GTM-XXXXXXX&gtm.url=SOURCE_URL ( GTM-XXXXXXX SOURCE_URL ;
  2. _ga. , client ID Google : , , (timestamp). , _ga. , GTM client ID:

   "vars":{
      "clientId":"111111111.111111111", //     
   }


インストールJSON GTMを独自のカスタムクライアントIDに置き換え、1人のユーザーの同じクライアントIDをすべての可能なAMPクライアントに転送します。

この構成では、表示ドメインに関係なく、JSON分析のすべてのリクエストがドメインを通過し、ドメインにクライアントIDが割り当てられます。したがって、AMPページをロードする4つの可能な方法はすべて、同じクライアントIDを受け取ります。
