Jsonnetで100 gitlabジョブを100行で説明する方法

Kubernetesのデプロイツールに関する前回の記事の続きとして、Jsonnetを使用して.gitlab-ci.ymlのジョブの説明を簡略化する方法についてお話します



与えられた


以下のようなmonorepaがあります：


  • 10のdockerfiles
  • 30説明されたデプロイ
  • 3つの環境：develの、ステージングと生産

仕事


パイプラインを設定します。


  • Dockerイメージのビルドは、バージョンを含むgitタグを追加することによって実行する必要があります。
  • 各展開操作は、環境ブランチにプッシュするときに、特定のディレクトリのファイルを変更することによってのみ実行する必要があります
  • 各環境には、その環境でのみデプロイを実行する個別のタグを持つ独自のgitlab-runnerがあります。
  • すべてのアプリケーションを各環境にデプロイする必要はありません。例外を作成できるように、パイプラインを記述する必要があります。
  • 一部のデプロイメントはgitサブモジュールを使用しており、変数セットを使用して実行する必要があります。 GIT_SUBMODULE_STRATEGY=normal

これらすべてを説明することは地獄のように思えるかもしれませんが、私たちは絶望せず、Jsonnetを武器に、簡単かつ自然にそれを行います。

local jobs = import 'lib/jobs.libsonnet';

//   
local ref(x) = { only+: { refs: [x] } };
local tag(x) = { tags: [x] };
local submodule(x) = { variables+: { GIT_SUBMODULE_STRATEGY: x } };

{
  // C docker-
  ['build:' + x]: jobs.dockerImage(x) + tag('build') + ref('tags')
  for x in [
    'dinner',
    'drunk',
    'fanatical',
    'guarantee',
    'guitar',
    'harmonious',
    'shop',
    'smelly',
    'thunder',
    'yarn',
  ]
}
+
{
  //         'prod'
  ['deploy:prod:' + x]: jobs.qbecApp(x) + tag('prod') + ref('prod')
  for x in [
    'dinner',
    'hall',
  ]
}
+
{
  //   git-submodule
  ['deploy:' + env + ':' + app]: jobs.qbecApp(app) + tag(env) + ref(env) + submodule('normal')
