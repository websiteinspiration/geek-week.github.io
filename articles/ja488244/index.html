C ++で例外を処理するときのオーバーヘッドを削減する方法



ランタイムエラーの処理は、ソフトウェアの開発時に遭遇する多くの状況で非常に重要です-誤ったユーザー入力から破損したネットワークパケットまで。ユーザーがPDFの代わりにPNGを突然ダウンロードした場合、またはソフトウェアの更新時にネットワークケーブルを切断した場合、アプリケーションはクラッシュしないはずです。ユーザーは、プログラムが何が起こっても動作し、緊急事態をバックグラウンドで処理するか、フレンドリーなインターフェイスを介して送信されるメッセージによって問題を解決するためのオプションを選択するように提案することを期待しています。

例外処理は混乱を招く複雑なタスクになる可能性があり、多くのC ++開発者にとって基本的に重要であり、アプリケーションの処理速度を大幅に低下させる可能性があります。ただし、他の多くの場合と同様に、この問題を解決するにはいくつかの方法があります。次に、C ++での例外処理プロセスについて詳しく説明し、その落とし穴に対処し、これがアプリケーションの速度にどのように影響するかを確認します。さらに、オーバーヘッドを削減するために使用できる代替手段を検討します。

この記事では、例外の使用を完全に停止するように勧めません。それらは適用されるべきですが、それを回避することが不可能であるときに正確に適用されます：たとえば、コンストラクター内で発生したエラーをどのように報告するか？実行時エラーを処理するために例外を使用することを主に検討します。これから説明する代替手段を使用すると、より信頼性が高く、メンテナンスが容易なアプリケーションを開発できます。

クイックパフォーマンステスト


プログラムの進行を制御するための通常のメカニズムと比較して、C ++の例外はどのくらい遅くなりますか？

例外は、単純な中断または復帰操作よりも明らかに低速です。しかし、どれほど遅いか調べてみましょう！

次の例では、ランダムに数値を生成し、生成された1つの数値をチェックすることに基づいて、エラーメッセージを表示する/表示しない単純な関数を記述しました。

エラー処理のためのいくつかの実装オプションをテストしました。

  1. 整数の引数で例外をスローします。これは実際には特に適用されていませんが、C ++で例外を使用する最も簡単な方法です。したがって、テストの実装における過度の複雑さを取り除きます。
  2. テキストメッセージを送信できるstd :: runtime_errorをスローします。このオプションは、前のオプションとは異なり、実際のプロジェクトでより頻繁に使用されます。2番目のオプションが最初のオプションと比較してオーバーヘッドコストの目に見える増加をもたらすかどうかを見てみましょう。
  3. 空の返品。
  4. Cスタイルのintエラーコードを返す

テストを実行するために、シンプルなGoogleベンチマークライブラリを使用しました。彼女は各テストをサイクルで繰り返し実行しました。次に、すべてがどのように起こったかを説明します。せっかちな読者はすぐに結果にジャンプできます。

テストコード

非常に複雑な乱数ジェネレータ：

const int randomRange = 2;  //     0  2.
const int errorInt = 0; 	//    ,    0.
int getRandom() {
	return random() % randomRange;
}

テスト機能：

// 1.
void exitWithBasicException() {
	if (getRandom() == errorInt) {
    	throw -2;
	}
}
// 2.
void exitWithMessageException() {
	if (getRandom() == errorInt) {
    	throw std::runtime_error("Halt! Who goes there?");
	}
}
// 3.
void exitWithReturn() {
	if (getRandom() == errorInt) {
    	return;
	}
}
// 4.
int exitWithErrorCode() {
	if (getRandom() == errorInt) {
    	return -1;
	}
	return 0;
}

これで、Googleベンチマークライブラリを使用できるようになりました。

// 1.
void BM_exitWithBasicException(benchmark::State& state) {
	for (auto _ : state) {
    	try {
        	exitWithBasicException();
    	} catch (int ex) {
        	//  ,    .
    	}
	}
}
// 2.
void BM_exitWithMessageException(benchmark::State& state) {
	for (auto _ : state) {
    	try {
        	exitWithMessageException();
    	} catch (const std::runtime_error &ex) {
        	//  ,    
    	}
	}
}
// 3.
void BM_exitWithReturn(benchmark::State& state) {
	for (auto _ : state) {
    	exitWithReturn();
	}
}
// 4.
void BM_exitWithErrorCode(benchmark::State& state) {
	for (auto _ : state) {
    	auto err = exitWithErrorCode();
    	if (err < 0) {
        	// `handle_error()` …  - 
    	}
	}
}

//  
BENCHMARK(BM_exitWithBasicException);
BENCHMARK(BM_exitWithMessageException);
BENCHMARK(BM_exitWithReturn);
BENCHMARK(BM_exitWithErrorCode);

//  !
BENCHMARK_MAIN();

ゼロコストモデルのおかげで、C ++のtryブロックのほとんどの実装では、追加のオーバーヘッドはありません。ただし、catch-blockの動作ははるかに遅くなります。簡単な例を使用すると、スローのスローと例外のキャッチがどのように機能するかを確認できます。そのような小さな呼び出しスタックでも！スタックの深さが増すと、オーバーヘッドは直線的に増加します。そのため、例外をスローしたコードにできるだけ近いところで例外をキャッチすることが非常に重要です。

まあ、私たちは例外がゆっくり動作することを発見しました。その後、それを止めるか？しかし、すべてがそれほど単純なわけではありません。

なぜ誰もが例外を使い続けるのですか？


例外の利点は、C ++パフォーマンスに関するテクニカルレポート（5.4章）に詳しく記載されています。

, , errorCode- [ ], . , . , .

繰り返しますが、主な考え方は、例外を無視して忘れることができないことです。これにより、例外は非常に強力な組み込みC ++ツールになり、Cから継承したエラーコードを介して困難な処理を置き換えることができます。

さらに、例外は、「ハードディスクがいっぱいです。 「または」ネットワークケーブルが破損している。」このような状況では、例外が理想的です。

しかし、プログラムに直接関連するエラー処理を処理する最良の方法は何ですか？いずれの場合も、エラーをチェックし、エラーが発生した場合は十分な情報を提供し、メッセージの形式またはその他の形式で送信する必要があることを開発者に明確に示すメカニズムが必要です。組み込みの例外に戻ってきたようですが、今は別の解決策について話します。

期待される


オーバーヘッドに加えて、例外にはもう1つの欠点があります。それらは順次に機能します。例外はスローされるとすぐにキャッチして処理する必要があり、後で延期することは不可能です。

それについて私たちは何ができますか？

方法があることがわかりました。 Andrei Alexandrescu教授がExpectedという特別なクラスを用意しました。これにより、クラスTのオブジェクト（すべて正常に処理された場合）またはクラスExceptionのオブジェクト（エラーが発生した場合）を作成できます。つまり、これかそれとも。またはまたは。
本質的に、これはデータ構造のラッパーであり、C ++ではunionと呼ばれます。

したがって、私たちは彼のアイデアを次のように説明しました：

template <class T>
class Expected {
private:
	//  union:  ,   .     
	union {
    	T value;
    	Exception exception;
	};

public:
	//    `Expected`   T,   .
	Expected(const T& value) ...

	//    `Expected`   Exception,  -   
	Expected(const Exception& ex) ...

	//    :    
	bool hasError() ...

	//   T
	T value() ...

	//        (  Exception)
	Exception error() ...
};

完全な実装は確かにより困難になります。しかし、期待の主な考え方は、同じオブジェクト（期待＆e）で、プログラムの通常の操作のデータ（タイプと形式がわかっているTと値を持つ）とエラーデータ（例外と例）。したがって、何が来たのかをもう一度簡単に確認できます（たとえば、hasErrorメソッドを使用）。

ただし、今すぐこの例外を処理するように強制する人は誰もいません。throw（）呼び出しやcatchブロックはありません。代わりに、必要に応じてExceptionオブジェクトを参照できます。

期待されるパフォーマンステスト


新しいクラスに対して同様のパフォーマンステストを記述します。

// 5. Expected! Testcase 5 

Expected<int> exitWithExpected() {
	if (getRandom() == errorInt) {
    	return std::runtime_error("Halt! If you want...");  //  : return,   throw!
	}
	return 0;
}

// Benchmark.


void BM_exitWithExpected(benchmark::State& state) {
	for (auto _ : state) {
    	auto expected = exitWithExpected();

したがって、組み込みの例外と比較して、Expectedは何倍も速く動作し、より柔軟なエラー処理メカニズムも提供します。さらに、意味的に純粋であり、エラーメッセージを犠牲にしてユーザーから品質の高いフィードバックを奪う必要がなくなります。

結論


例外は絶対的な悪ではありません。また、例外的な状況において、意図した目的のために非常に効果的に機能するので、時にはそれでもまったく良いことです。はるかに効果的なソリューションが利用可能な場合にのみ、問題が発生し始めます。

私たちのテストは、非常に単純であるにもかかわらず、例外をキャッチしない場合（スローキャッチブロック）、データを送信するだけで十分な場合（リターンを使用）、オーバーヘッドを大幅に削減できることを示しました。

この記事では、Expectedクラスを簡単に紹介し、それを使用してエラー処理プロセスを高速化する方法についても説明しました。Expectedを使用すると、プログラムの進捗状況を簡単に追跡できるようになり、さらに柔軟にユーザーや開発者にメッセージを送信できるようになります。
