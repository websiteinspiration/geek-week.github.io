今日は、DropboxがPythonコードの型制御にどのように関与しているかに関する資料の翻訳の最初の部分に注目します。



DropboxはPythonで多くのことを書いています。これは、バックエンドサービスとデスクトップクライアントアプリケーションの両方で、非常に広く使用されている言語です。 Go、TypeScript、Rustも大量に使用していますが、Pythonが主な言語です。私たちの範囲を考えると、何百万行ものPythonコードについて話していると、そのようなコードの動的型付けは理解を不必要に複雑にし、生産性に深刻な影響を与え始めることがわかりました。この問題を軽減するために、コードをmypyを使用した静的型チェックに徐々に変換し始めました。これはおそらく、Pythonで最も人気のあるスタンドアロン型チェックシステムです。 Mypyはオープンソースプロジェクトであり、その主な開発者はDropboxで働いています。

Dropboxは、Pythonコードに同様の規模で静的型チェックを実装した最初の企業の1つです。現在、mypyは何千ものプロジェクトで使用されています。彼らが言うように、このツールは「戦闘でテストされた」回数は無数です。私たちが今いる場所にたどり着くためには、長い道のりが必要でした。このようにして、多くの失敗した試みと失敗した実験がありました。この資料は、Pythonでの静的型チェックの歴史について語っています。私の科学的研究プロジェクトの一環であった非常に困難な初めから、Pythonで記述する無数の開発者が型チェックと型ヒントに慣れてきた現在まで。これらのメカニズムは現在、IDEやコードアナライザーなどの多くのツールでサポートされています。

型チェックが必要なのはなぜですか？


動的型付けされたPythonを使用したことがある場合は、静的型付けとmypyに関してこのような話題が最近なされた理由について、いくつかの混乱があるかもしれません。あるいは、動的な型付けのためにPythonが好きで、何が起こっているのか気分を害しているのかもしれません。静的型付けの価値の鍵は、決定の規模です。プロジェクトが大きくなるほど、静的型付けの傾向が強くなり、結局、それが本当に必要になります。

プロジェクトが数万行に達し、何人かのプログラマーがそれに取り組んでいるとしましょう。そのようなプロジェクトを考えると、私たちの経験に基づいて、そのコードを理解することが開発者の生産性をサポートするための鍵になると言えます。型注釈がないと、たとえば、関数に渡す必要がある引数や、関数が返すことができる型の値などを理解するのは簡単ではありません。タイプアノテーションを使用しないと答えることが難しい典型的な質問を次に示します。

  • この関数は戻ることができますNoneか？
  • この議論は何であるべきitemsですか？
  • 属性の種類は何ですかid：intそれは,, strまたはユーザタイプの、おそらくいくつかの種類は？
  • この引数はリストである必要がありますか？タプルを渡すことは可能ですか？

タイプアノテーションを備えた次のコードスニペットを見て、そのような質問に答えようとすると、これが最も簡単なタスクであることがわかります。

class Resource:
    id: bytes
    ...
    def read_metadata(self, 
                      items: Sequence[str]) -> Dict[str, MetadataItem]:
        ...

  • read_metadata戻らないNone戻り値の型ではないので、Optional[…]。
  • 引数itemsは文字列のシーケンスです。ランダムな順序で繰り返すことはできません。
  • 属性idはバイトの文字列です。

理想的な世界では、そのような微妙なことはすべて組み込みのドキュメント（docstring）に記述されると予想されます。しかし、経験上、作業しなければならないコード内のそのようなドキュメントがしばしば観察されないという事実の多くの例を示しています。そのようなドキュメンテーションがコードに存在していても、絶対的な正確さを当てにすることはできません。このドキュメントは不明確で不正確な場合があり、誤解の可能性がたくさん残っています。大きなチームや大きなプロジェクトでは、この問題は非常に深刻になる可能性があります。

Pythonはプロジェクトの初期または中間段階で非常にうまく機能しますが、ある時点で、Pythonを使用する成功したプロジェクトや企業は、「静的に型付けされた言語ですべてを書き直す必要があるか？」という重大な問題に直面する場合があります。

mypyのような型チェックシステムは、型を記述するための正式な言語を開発者に提供し、型の記述がプログラムの実装と整合していることをチェックする（そしてオプションで、それらの存在をチェックする）ことにより、前述の問題を解決します。一般的に、これらのシステムは、慎重にチェックされたドキュメントのようなものを提供してくれると言えます。

そのようなシステムの使用には他の利点があり、それらはすでに完全に重要です。

  • 型チェックシステムは、いくつかの小さな（そしてそれほど小さなものではない）エラーを検出できます。典型的な例は、値Noneやその他の特別な条件の処理を忘れた場合です。
  • , , . 100% , , , . , .
  • mypy . . . , , .

PyCharmやVisual Studio CodeなどのIDEとエディターは、型注釈機能を使用して、開発者にコードの自動補完、エラーの強調表示、一般的に使用される言語構成のサポートを提供します。そして、これらはタイピングがもたらす利点のほんの一部です。一部のプログラマーにとって、これはすべてタイピングの主な引数です。これは、実装直後のメリットです。この型の使用例では、mypyなどの個別の型チェックシステムは必要ありませんが、mypyは型注釈とコード間の一貫性の維持に役立つことに注意してください。

Mypyの背景


mypyの話は、私がDropboxに参加する数年前のイギリスのケンブリッジで始まりました。私は博士研究の一環として、静的型付け言語と動的言語の統合に取り組みました。 Jeremy SiekとWalid Tahaの段階的なタイピングに関する記事と、Typed Racketプロジェクトに触発されました。小さなスクリプトから数百万行からなるコードベースまで、さまざまなプロジェクトで同じプログラミング言語を使用する方法を見つけようとしました。同時に、どんな規模のプロジェクトでも妥協しすぎないようにしたいと思います。これらすべての重要な部分は、型付けされていないプロトタイププロジェクトから、包括的にテストされた静的型付けされた完成品への段階的な移行のアイデアでした。最近、これらのアイデアは、大部分が当たり前のことですが、2010年にはまだ積極的に検討されている問題でした。

型チェックの分野での私の最初の仕事はPythonを対象としていませんでした。代わりに、小さな「自家製」の言語Aloreを使用しました。ここに何が問題になっているのかを理解させる例があります（ここでは型注釈はオプションです）：

def Fib(n as Int) as Int
  if n <= 1
    return n
  else
    return Fib(n - 1) + Fib(n - 2)
  end
end

私たち自身のデザインの簡略化された言語を使用することは、科学研究で使用される一般的なアプローチです。これは、実験を迅速に行うことができるという事実だけでなく、研究とは何の関係も簡単に無視できるという事実によるものでもありません。実際に使用されるプログラミング言語は通常、複雑な実装を伴う大規模な現象であり、これにより実験が遅くなります。ただし、単純化された言語に基づく結果は少し疑わしいように見えます。結果が得られたときに、研究者は言語の実際的な使用に重要な考慮事項を犠牲にしている可能性があるためです。

私のAloreの型チェックツールは非常に有望に見えましたが、Aloreで記述されていない、実際のコードを試してテストしたかったのです。幸いなことに、Alore言語は主にPythonと同じアイデアに基づいていました。 Pythonの構文とセマンティクスで機能するように、型チェックツールを作り直すのは簡単でした。これにより、オープンソースのPythonコードで型チェックを試すことができました。さらに、Aloreで記述されたコードをPythonコードに変換するトランスポーターを作成し、それを使用してタイプチェックツールのコードを変換しました。これで、PythonのサブセットをサポートするPythonで記述された型チェックシステムができました。 （特定のアーキテクチャ上の決定、これはAloreにとっては理にかなっており、Pythonではうまく機能しませんでしたが、mypyコードベースの一部ではまだ目立っています。

実際、その時点で私の型システムでサポートされている言語は、Pythonとは呼ぶことができませんでした。Python3型の注釈の構文にいくつかの制限があるため、Pythonのバージョン

でした。JavaとPythonの混合のように見えました。

int fib(int n):
    if n <= 1:
        return n
    else:
        return fib(n - 1) + fib(n - 2)

当時の私の考えの1つは、型注釈を使用して、この種のPythonをCまたはおそらくJVMバイトコードでコンパイルすることにより、パフォーマンスを向上させることでした。コンパイラーのプロトタイプを作成する段階に進みましたが、型チェック自体が非常に便利に見えたので、この考えは残しました。

結局、サンタクララで開催されたPyCon 2013カンファレンスでプロジェクトを発表しました。私はまた、Pythonの寛大な生涯独裁者であるGuido van Rossumとも話しました。彼は私自身の構文を放棄し、標準のPython 3構文に固執することを私に確信させました。Python3は関数アノテーションをサポートしているので、私の例は以下のように書き直すことができ、通常のPythonプログラムを取得できます。

def fib(n: int) -> int:
    if n <= 1:
        return n
    else:
        return fib(n - 1) + fib(n - 2)

私はいくつかの妥協をする必要がありました（まず第一に、私はこの理由のために私自身の構文を発明したことに注意したいです）。特に、当時の言語の最新バージョンであるPython 3.3は、変数アノテーションをサポートしていませんでした。そのような注釈を構文化するためのさまざまな可能性を電子メールでGuidoと話し合いました。変数には型コメントを使用することにしました。これで目標を達成することができましたが、少し面倒に見えました（Python 3.6により、より快適な構文が得られました）。

products = []  # type: List[str]  # Eww

タイプコメントは、Python 2のサポートにも役立ちます。Python2には、タイプアノテーションの組み込みサポートがありません。

f fib(n):
    # type: (int) -> int
    if n <= 1:
        return n
    else:
        return fib(n - 1) + fib(n - 2)

実際、これらの（およびその他の）妥協点はそれほど重要ではなかったことが判明しました。静的型付けの利点により、ユーザーは完全ではない構文をすぐに忘れてしまいました。型が制御されているPythonコードでは特別な構文構造が使用されなかったため、既存のPythonツールとコード処理プロセスは正常に機能し続け、開発者による新しいツールの開発を大幅に促進しました。

グイドはまた、卒業を擁護した後、Dropboxに参加することを私に確信させました。ここから、mypyの歴史の中で楽しい部分が始まります。
