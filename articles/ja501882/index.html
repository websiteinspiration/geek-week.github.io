コンピュータービジョンアルゴリズムの使用方法：OpenCV.jsを使用したモバイルブラウザーでのビデオ処理

オンラインで人物を特定するための可能性はすでにすべてありますが、現在のところほとんど使用されていません。おそらく、私たちはユーザーに最適なシナリオを実装した最初の1人でした-スマートフォンからサイトにログインし、運転免許証またはパスポートの写真を撮って、データをシステムに送信します。

コンピュータービジョンアルゴリズムが、ビデオストリーム内のドキュメントをモバイルブラウザーで直接認識する方法を考えてみましょう。この記事では、SimbirSoftでOpenCV.jsをどのように使用したか、どのような問題が発生する可能性があるか、速度を確保し、速度を落とさずに「スムーズな」UXを実現する方法についての経験を共有します。




タスクは何でしたか


開発中のアルゴリズムのビジネスシナリオは以下の通りです。携帯電話からサイトにアクセスするユーザーは、自分のドキュメントを写真に撮ってシステムに送信し、さらに処理する必要があります。これは、サービスの使用を申請する際のIDプロセスの一部である場合があります。

このシナリオのWebアプリケーションは、その可用性と操作完了までの時間の短縮のため、モバイルアプリケーションよりも適しています。 Webページはインストールする必要がなく、ロード後すぐに動作する準備ができています。ユーザーは、リンクを受け取った直後に、追加のアクションに邪魔されることなく、必要なアクション（アプリケーションの送信）を実行できます。ビジネスの観点から、これらの要因はプロセスの変換と商業的有効性を高めます。

アーキテクチャの観点からは、ドキュメントの境界を直接検出し、画像内の余分な背景をトリミングするアルゴリズムが必要です。身元の確認、認証、不正チェックは、他のコンポーネントによって実装されます。ただし、少なくとも最小限のチェックを実行して、名刺、空の紙の長方形、および画像を処理するためのその他の明らかに無関係な画像の送信を除外することが望ましいです。

必要条件


私たちのプロジェクトの一環として、アルゴリズムには次の追加要件がありました。

  • リアルタイムで動作する機能：アルゴリズムの動作中にカメラからのビデオストリームが「スローダウン」しないようにする必要があります。
  • 幅広いコントラストと背景テクスチャで機能する能力：低コントラストとコントラスト、均一および不均一な背景。
  • 数年前にリリースされた予算モデルを含む、幅広いスマートフォンモデルのサポート。

最後に、プロジェクトには機械学習アルゴリズムをトレーニングするためのデータセットがなく、それを収集してマークアップする方法がありませんでした。Googleの検索結果からのテストサンプルはわずかしかありませんでした。

この問題の定式化を考慮して、opencvライブラリーの古典的なコンピュータービジョンアルゴリズムに基づいて開発することにしました。代替の可能性は、機械学習アルゴリズムとニューラルネットワークの使用でしたが、パフォーマンス要件のため、作業の初期段階ではすでに破棄されていました。適用すると、すべてのターゲットデバイスでリアルタイムフレーム処理を提供することができなくなります。

一般的なアプローチとアルゴリズムの構造


アルゴリズムの主な考え方は参照フレームであり、それに沿ってドキュメントを整列させる必要があります。その使用は一度にいくつかの目標を追求します。まず、ドキュメントをさらに処理するのに十分な適切な画像サイズを提供します。次に、後で説明するように、ドキュメントの境界線を検索するときの候補フィルターの1つとして使用できます。3番目に、ドキュメントの境界が見つからなかった場合に、画像をキャプチャしてトリミングするために使用できます。



図。1.アルゴリズムの一般的な構造

アルゴリズムの一般的な構造を図1に示します。1.ビデオストリームからのフレームがサイクルで処理され、その反復の間に、タイムアウトが目的のFPSに準拠するように設定されます。毎秒30フレームで停止しました。これにより、「スローダウン」を回避し、プロセッサの負荷とデバイスの消費電力を削減できます。

処理された各フレームは前処理を受け、その間に2つの主要な操作が実行されます。まず、640x480の固定サイズのフレームのコピーが作成されます。これにより、アルゴリズムの次のステップが機能します。元の画像も残り、検出されたドキュメントはその画像から切り取られます。これにより、最終的な画像の品質が保存されます。次に、作成されたコピーは灰色の濃淡に変換されます。処理中のドキュメントの色は、国によって、また国内のさまざまな地域でも異なる可能性があるため、アルゴリズムでは無視されます。例としては、米国の運転免許証があります。

文書を検出する最初のステップは、画像内の顔を検索することです。このヒューリスティックを使用すると、名刺やその他の明らかに無関係な画像のキャプチャがなくなります。検索は、標準のopencv'shash CascadeClassifier.detectMultiScale（）と事前トレーニング済みのカスケードhaarcascade_frontalface_defaultを使用して実行されます。検出された顔の最小サイズと最大サイズが制限されているため、計算コストを削減でき、画像内のドキュメントのスケールもさらに制限されます。顔が画像内で検出されたと見なされるのは、参照フレーム内の領域の一部（パスポートの場合、左または左下）です（図2）。これは、画像内のドキュメントの正しい配置を保証するための追加の手段です。

この記事の例には個人データは含まれていません。



図。 2.画像​​内の顔の予想される位置の領域。サポートフレームは赤で表示され、予想される顔の位置の領域の境界は緑で表示されます。

顔を検出したら、境界の検出に進みます。多くの場合、findContours（）がここで使用されます。ただし、この方法がうまく機能するのは、たとえば、暗い机の上に横たわっている紙の場合など、対照的な場合のみです。コントラストが低い場合、照明が悪い場合、または誰かがシートを手に持っていて、境界の一部を指で覆っている場合、検出された輪郭は個別のコンポーネントに分割され、重要なセクションが「失われる」か、まったく検出されません。

したがって、別のアプローチを採用しました。 2値化後、最初にCanny（）を使用して境界フィルターに画像を渡し、次にハフ変換HoughLines（）を使用してラインの結果の画像を確認します。しきい値パラメータは、検出された短いセグメントやその他の無関係なセグメントをフィルタリングするために、すぐに30に等しい十分な大きさに設定されます。

結果のラインのセットはさらにフィルターされ、参照フレームに近いラインのみが残ります。これを行うには、最初にフレームラインの方程式を極座標系（rho、theta）のポイントに変換します。thetaは常に0またはpi / 2であり、rhoはラインごとに一意です。その後、値のスケールの違いを考慮して、ユークリッド計量に従って、制御点の近くにあるものだけをハフ変換から取得した線から選択します。

参照フレームの4本の線に対応する4つのグループにフィルタリング後に得られた線のセットを分配し、グループ間の線のペアの交点を見つけ、平均し、4点の座標-検出されたドキュメントのコーナーを取得します（図3）。



図。 3.ラインのフィルタリングとドキュメントコーナーの定義。緑の線-フィルタリングの結果、黄色の点-文書のコーナーが検出された

次に、フレームの品質を確認する必要があります。これを行うには、フレームが最後に静止したままであることを確認します。これを行うには、absdiff（）を使用して現在のフレームから期間の最初のフレームを減算し、それをしきい値と比較します。減算の前に、ガウスフィルターGaussianBlur（）を使用して画像をさらに平滑化し、ノイズやその他のランダムな要因の影響を減らします。また、ラプラシアンラプラシアン（）を計算し、その分散を推定して、得られた値をしきい値と比較することにより、フレームのフォーカスを評価します。

すべてのチェックが成功した場合、最後の部分に進むことができます。検出された角度の座標を元の露出不足の画像の座標系に再計算し、roi（）メソッドを使用して結果の領域を切り取ります。ドキュメントは正常に検出されました。

実装機能


アルゴリズムの開発中、その主要コンポーネントはpythonスクリプトで組み立てられました。その後、アルゴリズムはopencv.jsとjavascriptに移植され、次にwasmに移植されました。このアプローチは、すべての段階での利便性を考慮して決定されます。 Pythonでは、チームがアルゴリズムのさまざまなバリアントを試して、大まかなパラメータ設定を実行する方が便利でした。 javascriptへの移植により、さまざまなデバイスやブラウザなど、ターゲットプラットフォームでのアルゴリズムの動作をテストできるようになりました。これらのチェックの結果に基づいて、アルゴリズムパラメーターの微調整が行われました。最後に、wasmでコードの重要なセクションを書き換えることで、パフォーマンスをさらに向上させることができました。

移行中に、OpenCV APIにいくつかの違いが発見され、その結果、実装に小さな変更が加えられました。たとえば、Pythonでのラプラシアンの分散は、単純にラプラシアン（）。Var（）と見なされます。 OpenCV.jsでは、NumPyを使用する方法はありませんが、var（）メソッドの代替実装は提供されていません。解決策：meanStdDev（）関数を標準偏差としてカウントします（リスト1）。

private isImageBlurry(image: cv.Mat): boolean {
		const laplacian = new cv.Mat();
		cv.Laplacian(image, laplacian, cv.CV_64F);
		const s_mat = new cv.Mat();
		cv.meanStdDev(laplacian, new cv.Mat(), s_mat);
		const s = s_mat.data64F[0];
		const v = Math.pow(s, 2);
		return (v < this.laplacianVarianceThreshold);
	}

リスト1. opencv.js（TypeScript）でのラプラシアンの分散による画像へのフォーカスの評価

もう1つの機能は、ライブラリーのサイズを小さくする必要があったことです。 OpenCV.jsの元の形式では、容量は7.9 MBです。インターネット経由でダウンロードすると、アルゴリズムの初期化が遅くなります。この問題の解決策は、ライブラリアセンブリプロセス中に未使用のモジュールを「トリミング」することです。これにより、出力ファイルのサイズを大幅に削減できます。1.8MBのサイズを達成することができました。アセンブリーに含まれるコンポーネントのリストは、platforms / js / opencv_js.config.pyという構成ファイルで構成できます（リスト2）。

white_list = makeWhiteList([core, imgproc, objdetect, video, dnn, features2d, photo, aruco, calib3d])

リスト2. javascriptのアセンブリーに含まれるopencvモジュールの元のホワイトリスト

最後に、必要なアルゴリズムのパフォーマンスを確保するための重要な貢献は、アルゴリズムをWebワーカーに移動することによって行われました。このステップとFPSの制限により、アルゴリズムの動作中にビデオストリームの「スローダウン」を取り除くことができました。これは、UXにプラスの影響を与えました。

結果


画像のキャプチャとトリミングの例を図4に示します。暗い均一な背景で最高品質のトリミングが達成され、明るい不均一な背景で最低品質が得られることがわかります。これは、さまざまな背景で取得され、ドキュメントの境界線を検出するために使用されるグラデーションに関連する予想される効果です。暗い背景に対して、グラデーションは明るい背景よりも大きく、均一な背景はグラデーション値の変動を少なくします。これにより、境界を確実に検出できるようになり、その結果、より適切なトリミングが可能になります。

結論


この記事では、モバイルブラウザーでの使用に適した、ビデオストリームからフレーム上のドキュメントを検出するアルゴリズムを紹介し、opencv.jsライブラリを使用したその実装の機能についても考察します。このアルゴリズムを使用すると、認証やID検証などのアルゴリズムでさらに使用するのに十分な品質でドキュメントの出力イメージを取得できます。結果として得られる実装の速度により、「スローダウン」やフレーム損失のない「スムーズな」UXを実現できます。
