それが古き良きAngularJSのすごい効果を引き起こしたものなので、「双方向バインディング」です。この魔法はすぐにAngularJSに夢中になり、退屈なページプログラミングと（ああ、ホラー！）Webフォームに関するすべてのアイデアを壊しました。データの変更は即座に画面に表示され、その逆も同様です。以前jQueryアプリケーションを開発した人たちは、バインディングをおとぎ話の中にあると感じていました。そして、ひげを生やしたモンスターは、jQueryの前に太ったクライアントを切り開いて、愚かに失われた人月を必死に数え始めました。

さらに、双方向バインディングの魔法は、特別な表記法や選択されたコンポーネントだけでなく、利用できました。独自のディレクティブとコンポーネントで（構成パラメーターを設定するだけで）簡単に使用できます。

Angular2 +では、作成者は組み込みの双方向データバインディング（ngModelを除く）を放棄しました。しかし、これは私たち自身のディレクティブで双方向バインディングを使用できないことを意味するわけではありません...それはただ無料が終わっただけで、今私たちは自分で何かをする必要があるということです。そして、できれば、それがAngularでどのように配置されているかを理解してください。

一言で言えば、双方向バインディング


A2 +では、1つのディレクティブのみが双方向のデータバインディングを実装します：ngModel。一見すると、これはAngularJSと同じ魔法です（表記が異なるだけです）。しかし、内部には何があるのでしょうか？

驚くべきことに、すべてが内部的には比較的シンプルで論理的です。双方向バインディングは、バインディングプロパティとバインディングイベントです。 1つの双方向ではなく、2つの一方向バインディング？では、2つにしましょう。

そしてすぐに例：

<input [(ngModel)]="username">
<p>Hello {{username}}!</p>

はい、はい。これは2009年のAngular2の美しく素晴らしいデモです。冗談ではない、美しい。フィールドを変更すると、ユーザー名の値がモデルに入り、フォームの挨拶にすぐに反映されます。

しかし、それはどのように機能しますか？Angular2の双方向バインディングはプロパティバインディングとイベントバインディングであることを思い出してください。そして、はい、1つのディレクティブで同時に使用できます。さらに、ngModelがなくても、双方向のデータバインディングを簡単に実装できました。たとえば、次のようになります。

<input [value]="username" (input)="username = $event.target.value">
<p>Hello {{username}}!</p>

出力{{username}}は明確ですが、入力には何が書き込まれていますか？理解しましょう：

  • [値] = "username" -角かっこ付きの表記で、ユーザー名の式をvalueプロパティにバインドします
  • (input) = «expression» – , expression input (-, ). :
    • username = $event.target.value – input
    • $event – Angular, :

はっきりしていますか？修正します。Angularモデルのユーザー名

プロパティをブラウザーの入力要素の値プロパティにバインドします（モデルからビューへの一方向のバインド）。 また、要素の入力イベントに式をバインドします。$ event.target.valueの値をモデルのユーザー名プロパティに割り当てます。$ event.target.value とは何ですか？すでに述べたように、$イベントにはイベントに関するさまざまな役立つ情報が満載です。この場合、それは、ターゲットプロパティが開始したDOM要素を参照するInputEventObjectです。



イベント（つまり、入力要素）。

したがって、基本的に行うことは、ユーザーが値を入力したときに入力要素（$ event.target）の内容（value）を読み取ることだけです。このユーザー名の値を割り当てると、ビューデータがモデルに送信されます。それで全部です。これは「一方向の双方向バインディング」です。美しさ？ しかし、ngModelはいつ登場するのでしょうか？入力要素を使用するシナリオは非常に一般的であり、需要があります。そして、なんらかの理由で、実装を隠し、余分なキーストロークから節約するディレクティブが必要です。





ngModelを理解する


ソースを見ると、ngModelにもプロパティとイベントへのバインディングがあることを確認できます。以下は、ngModelの例ですが、省略構文を使用していません。

<input [ngModel]="username" (ngModelChange)="username = $event">
<p>Hello {{username}}!</p>

ほとんどすべてが同じです。[ngModel]プロパティのバインディングは、入力要素の値の更新を処理します。イベントバインディング（ngModelChange）は、DOMで変更が発生していることを世界に通知します。

また、ハンドラー式では$ event.target.valueではなく、$ eventのみを使用していることに気付きました。ここで何か問題がありますか？どういたしまして。上記のように、$イベントはペイロードを運ぶ合成変数です。有用であると見なされるものの決定は、Angularが行います。言い換えると、ngModelChangeは内部の$イベントからtarget.valueを抽出しますそして、パッケージやタンバリンなしで、私たちに欲しいものを与えてくれます。技術的に正確に言うと、これらはDefaultValueAccessorのものです。データを取得してベースDOMオブジェクトに転送するのは彼です（ただし、考えられないだけです）。

最後に重要なことですが、ユーザー名とngModelを 2回書き込むことは依然として冗長であるため、Angularでは「ボックス内のバナナ」とも呼ばれる短縮構文[（）]を使用できます。これは前の例と同様であり、セクションの最初から例に戻りますが、ngModel実装を理解しています。同じ双方向バインディングを提供します。

<input [(ngModel)]="username">
<p>Hello {{username}}!</p>


独自の双方向データバインディングを作成する


これで、独自の双方向データバインディングを作成することができます。あなたがする必要があるのは単にngModelと同じルールに従うことです、すなわち：

  • プロパティバインディングを入力してください（例：[foo]）
  • 同じ名前とサフィックスChangeを持つイベントにバインドします（例：（fooChange））
  • イベントバインディングがプロパティの取得を処理するようにします（必要な場合）。

双方向のデータバインディングを作成するには、AngularJSよりも大幅に多くの作業が必要になることに注意してください。これは私たちを非常に動揺させるかもしれません...可能な限り独自の双方向バインディングを使用しようとするとします。実際には、双方向バインディングが必要かどうかを常に検討する必要があります。必要な場合は、ngModelを利用する方が簡単です。たとえば、後者は、カスタムフォームコントロールを作成するときに行われます。

しかし、カスタムカウンターコンポーネントを作成するとします（カスタムフォームコントロールを使用したくないとしましょう）。

@Component({
  selector: 'custom-counter',
  template: `
    <button (click)="decrement()">-</button>
    <span>{{counter}}</span>
    <button (click)="increment()">+</button>
  `
})
export class CustomCounterComponent {
  counterValue = 0;
  get counter() {
    return this.counterValue;
  }
  set counter(value) {
    this.counterValue = value;
  }
  decrement() {
    this.counter--;
  }
  increment() {
    this.counter++;
  }
}

カウンターの現在の値を表示するカウンター コンポーネントのプロパティがあります。双方向バインディングを提供するには、最初に、それを入力パラメーターに変換します。このため、@ Input（）デコレータは非常に便利です。

@Component()
export class CustomCounterComponent {
  counterValue = 0;
  @Input()
  get counter() {
    return this.counterValue;
  }
  ...
}

これにより、次のようにコンポーネントプロパティをコンシューマにバインドできます。

<custom-counter [counter]="someValue"></custom-counter>

ここで、 @ Output（）イベントに同じ名前（counter）とサフィックスChangeを設定する必要があります（counterChangeになります）。カウンタが変わるたびにこのイベントを発生させたい。 @Output（）プロパティを追加する理由 そして、いくつかのゲッターであるカウンターセッターで終了します。ここでは、値の変更をインターセプトし、現在のカウンター値を使用してイベントをスローします。

@Component()
export class CustomCounterComponent {

  ...
  @Output() counterChange = new EventEmitter();

  set counter(val) {
    this.counterValue = val;
    this.counterChange.emit(this.counterValue);
  }
  ...
}

これだよ！これで、双方向のデータバインディング構文を使用して、式をこのプロパティにバインドできます。

<custom-counter [(counter)]="someValue"></custom-counter>
<p>counterValue = {{someValue}}</p>

結論


Angularには、双方向のデータバインディングが組み込まれていません。 「ボックス内」の代わりに、プロパティとイベントのバインディングとして完全なバインディングを実装できるAPIがあります。

ngModelは、組み込みの双方向バインディングディレクティブFormsModuleとして提供されます（インポートのセクションClassified  @NgModule：a lane に追加することを忘れないでください）。カスタムフォームコントロールとして機能するコンポーネントを作成する場合は、ngModelを介したリンクをお勧めします。そうでなければ、それはすべてあなたの想像力に依存します。
