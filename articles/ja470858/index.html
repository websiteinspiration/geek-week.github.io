骨董品：386番目のプロセッサのベンチマークと90年代のダッシュ

2019年から、90年代前半のコンピューターは何もできないようです。ビデオは再生されません。 MP3の音楽はできません、またはできますが、気に入らないほどの品質です。クライシスは実行されません。しかし、約30年前にスペアパーツのビンテージシステムを組み立てて発売した後、私は疑問に思いました。これはどの単位で表されますか？そしてそれをどのように測定するのですか？

測定ではすべてが簡単ではありません。古代のコンピューターと現代のコンピューターを比較しようとすると、新しいPCで古いベンチマークの実行に成功すると、結果がおかしくなります。より最近のソフトウェアは古いコンピュータではうまくいきません。今日は多かれ少なかれ退屈です私が386番目のプロセッサの処理能力をPentium IIIベースのラップトップおよびIntel Xeon X5690ベースのデスクトップと比較する技術記事。スポイラー：90年代のパフォーマンスはゼロよりもはるかに速く成長しました。

キャラクター



したがって、私たちの主なテスト対象は、40メガヘルツの周波数を持つ80386DXプロセッサ、32メガバイトのRAM、80387コプロセッサ、およびハードドライブではなくフラッシュドライブを搭載したコンピューターです。これまでのところ、これはコンピュータではなく、古いビデオカードの下からボックスにインストールされたボードのコレクションです。このようなシステムは、1991年から1992年に大まかに起因している可能性があります。当時はより強力なコンピュータがありましたが、そのような386番目はかなり合理的な選択でした。



周波数900メガヘルツ、256メガバイトのメモリ、80ギガバイトのハードドライブを搭載したIntel Pentium IIIプロセッサをベースにしたThinkPad T22ラップトップを追加します。このノートパソコンについては、こちらで詳しく説明しました。 2001年に第386回から約10年後に発売されました。


その場合は、さらに10年先に時間を巻き戻すことが論理的です。この目的のために、 Xeon X5690プロセッサー（標準周波数3.3 GHz、動作-3.9）と32ギガバイトのメモリーを備えた、今年の初めに組み立てられたIntel X58チップセットをベースにした中程度のビンテージデスクトップが理想的です。印象的な特徴にもかかわらず、これはコンピュータ構築の「まだ」学校の代表です。ドライブをネイティブに接続し、タンバリンで踊ることなくMS-DOSで起動できます。このコンピュータのマザーボードは2009年にリリースされましたが、プロセッサは2011年に発売されました。

DOSでのベンチマーク


コンピューターのパフォーマンスを包括的に測定するための個別のプログラムとしての最新の形のベンチマークは、すぐには現れませんでした。 PC Magazineは、ほぼすべての80年代で最も基本的なパフォーマンスパラメータ（命令の実行、メモリ内のデータ処理の速度など）を測定しました。ほとんどの場合、操作の完了にかかった時間が測定されましたが、これは問題です。最近のPCでは、これまで数分かかっていたものが数秒で完了します。


初期のユニバーサルベンチマークの1つは、1981年以来開発されたLandMarkテストです。システムパフォーマンスはすでにプロセッサ周波数だけでなく、たとえばコプロセッサの可用性やキャッシュメモリの存在にも依存しているため、製造元はLandMarkにその結果を広告で書いています。計算方法は奇妙でした。パフォーマンスは、6メガヘルツの周波数の286番目のプロセッサーをベースにしたIBM PC / ATリファレンスシステムと比較されました。コンピューターのテストの結果は、286番目のプロセッサーの周波数の特定の理論値であり、この値で、調査中のシステムと等しくなります。たとえば、私の386番目のプロセッサは69.45メガヘルツで286番目のプロセッサと同じように動作します。 Pentium IIIラップトップは、286番目が8695メガヘルツの周波数で動作しているかのようです。 Xeonでテストを実行し、結果カウンターがオーバーフローする：



パーソナルコンピュータの開発の初期のこのような状況は頻繁に発生しました。開発者は、プロセッサの周波数、処理能力、またはメモリ容量がそのような巨大な値に達したことを想像できませんでした。あるいは、彼らは単に考えなかった：Y2K問題は同様のルーツを持っている。ちなみに、386番目のマザーボードは部分的に影響を受けます-1999年12月31日以降、1900年1月1日に始まります。年を直接設定すると、すべてが正常に機能します。そのため、ゲームSimCity 2000のDOSバージョンは32メガバイトのRAMに応答します。


一般に、LandMarkは、1993年以前のコンピューターを相互に比較するのに適しています。また、継続的に計算を実行します。そのおかげで、ターボボタンが原因でプロセッサ周波数が低下しているかどうかを確認できます。システムによっては、マザーボード上の対応する接点を閉じる必要があります。しかし、私は何か他のものを探す必要がありました。前の記事ですでに述べたSpeedsysベンチマークが役に立ちます。


ゼロからゼロまで開発され、その時代のプロセッサを問題なく認識します。最も重要なのは、Xeonでも正常に動作することです。結果は386（左下のチャートに興味があります）-7.07ポイントです。



PIII-900を搭載したラップトップでの結果は1039.51ポイントです。



Xeonでは、このベンチマークはネイティブでのみ起動できます。 DosBox、およびFreeDosを備えた仮想マシンでは、私には機能しませんでした。 DOSで起動可能なフラッシュドライブが役立ちました。結果は7540.05ポイントです。つまり、1991年から2001年にかけて、生産性は2001年から2011年にかけて7.2倍、147倍に増加しました。はい。ただし、このテストでは、Xeonに6つのプロセッサコアがあることを考慮していません。ちなみに、この奇妙な実験を繰り返すことを考えている場合は、BIOSですべての省電力システムをオフにすることをお勧めします。オフにしないと、結果にばらつきが生じます。



実際のアプリケーションをいくつか試す必要があります。ゲームはここが最適です。最初のDoomは1993年の終わりにリリースされ、386台目のコンピューターでは2007年のCrysisとほぼ同じ効果を生み出しました。つまり、たった今、最新のPCではなく、まともなPCがありました。しかし、ドゥームが出て、それを再生することはできません。より正確にはできますが、画像は切手のサイズになります。テストパラメータ（グラフィック設定、画像サイズ、Doom自体のバージョン）を正確に再現することが重要です。



かつて、YouTubeチャンネルのホストであるPhil's Computer Labは、便利なバッチファイルを使用して、DOS用の古代のベンチマークのパッケージを組み立てました。 Doomやその他のテストをそのアセンブリから開始しました。パッケージには、Doomベンチマークの2つのバージョンがあり、最小設定と最大設定があります。後者のオプションは、ゲームメニューを非表示にするため、実際のゲームではほとんど使用されませんでした。しかし、テストには適しています。ベンチマークは結果をその単位で書き込みます。単位は1秒あたりのフレーム数に変換する必要があります。 386番目は、「最低」で28.7 fps、最高の品質で7.12 fpsを示しました。最新のPCと比較してください。PentiumIIIでは461および123 fps、Xeon X5690では完全に軽薄な426および93 fpsです。 「Doom」は明らかに私の最も強力なシステムでは何かが好きではなく、Pentium IIIよりも低い結果を示しています。

Norton System Information 8.0ユーティリティでも同様の状況が見られます。統合プロセッサのベンチマークは、386で43.2ポイント、PIIIで344.7ポイント、Xeonで960ポイントを示しています。プロセッサの開発は過去15年間で鈍化していますが、それほどではありません。

Windows 95およびSuperPi


さて、6コアマシンでMS-DOSを実行することは、それでも変態です。32ビットWindowsで書かれたプログラムはどうですか？理論的には、3つのシステムすべてで実行できます。実際には、ソフトウェアとベンチマークのバージョンを慎重に選択する必要があります。開発者たちは、2000年代の半ばにすでにWin9xサポートを放棄し始めました。一部のソフトウェアはWindows 98で正常に動作しますが、私は1991〜1992年のコンピューターを使用しており、Windows 95は彼には大きすぎます。入れます。


OSR2によって修正されたWindows 95は1996年に発売されました。386歳のとき、彼女は仕事がゆっくりしているだけでなく、速くもありません。（これらの年の標準では）大量のRAMを節約しますが、標準のコントロールパネルでさえ、レトロコンピュータはゆっくりと描画します。ただし、これには1つの利点があります。最新のシステムとは異なり、Windows 95は予測どおりに速度が低下します。いずれにしても、このシステムのPCMarkまたはその他の包括的なベンチマークは開始されません。処理能力だけに依存する、非常に単純なものが必要です。たとえば、SuperPi。


Vogons フォーラムで、 Windows 95とWindows 98およびWindows 10の両方で実行されるSuperPiテストのバージョンを見つけました。Piの小数点以下100万桁を計算した結果：

  • Xeon X5690：10.531秒
  • Pentium III 900：3分3.786秒
  • 80386 DX40：16時間58分31.445秒

うーん。このプロセッサーは減速せず、仕事と私生活のバランスを維持します。2001年のプロセッサは、386番目よりも332倍速いPi数を考慮していることがわかります。XeonはPentium IIIより17.5倍高速で、386番目のプロセッサより5800倍高速です。順調です。

SuperPiの結果は、Speedsysと比較してパフォーマンスの向上に同様の調整を提供します。ただし、厳密にシングルスレッドであるため、6コアIntel Xeonのすべての機能を正確に評価することはできません。古いハードウェアで実行できるベンチマークと、複数のスレッドの新しいハードウェアで同時に実行できるベンチマークはどれですか。コメントでオプションを提案しますが、WinRARアーカイバが見つかりました。Windows 95以降のPCでは、2008年のバージョン3.80は問題なく動作しました。シングルスレッドモードとマルチスレッドモードでのXeonの結果は次のとおりです。

386番目のプロセッサではより困難です。組み込みのWinRarベンチマークは次のように機能します。データを継続的にアーカイブし、十分な量の統計を受け取った後、結果を示します-アーカイブの速度（キロバイト/秒）。最近のコンピューターでは、約25〜30メガバイトのデータを処理した結果が表示されます。386番目のプロセッサでこの数値に到達するのに1日以上かかりました！


テストの2日目の終わりに、私はあきらめました。WinRARは頑固に結果を表示しませんでしたが、上のスクリーンショットはパフォーマンスを正確に評価するのに役立ちます-毎秒約214バイト。これは、Pentium IIIの1210倍遅いです。シングルスレッドモードのXeonは、3番目のPentiumよりも約4倍高速です。マルチスレッドモードでは、15倍高速ですが、1つのスレッドといくつかのスレッドの違いはコアの数にわずかに対応していません。これはWinRARコードの制限である可能性があります。

調査結果


まず、ベンチマークの結果を見てみましょう。私の測定は、定義上、不正確です。同じWinRAR 386番目のシステムでは、大量のデータを（そのために）消化できず、プロセッサが思ったよりも頻繁にアイドル状態であると感じました。それにもかかわらず、3つの異なるベンチマーク（Speedsys、SuperPi、WinRAR）は、1991年と2001年のコンピューター間で数百倍のパフォーマンスの向上を示しました。次の10年間で生産性は最大10倍に向上しました。繰り返しますが、これは完全に正直な比較ではありません。古いベンチマークは新しい新鮮な鉄の命令を使用しない場合があります。

独立した情報源を見てみましょう。ここにコンピューティング能力に関するLinpackユニバーサルベンチマークの結果は、386番目から第3世代Core i7までのシステム（およびアーキテクチャの最適化、可能な場合は、SSEおよびAVX命令を使用）のシステムに対して与えられます。 386-40（0.53 Mフロップ）、Pentium III-1000（316.67）、およびCore i7-3930K（3112.94）の違いは、同様のレイアウトを示しています。 90年代の成長率はほぼ600倍で、ゼロの成長率は10倍に過ぎません。

最後の千年紀の終わりに、私たちはパソコンのパフォーマンスの爆発的な増加を観察しました。新世紀には、成長もあり、重要でしたが、そのような狂った進歩はもはやありませんでした。さらに、80年代には、PC互換PCの時代の初めから、開発もそれほど速くはありませんでした-8086は80386（同じランドマークテストの結果による）と500ではなく約50倍異なります。 ：まずはテキストエディターのLexiconとNorton Commanderから始め、ネットワーク経由のオーディオのストリーミング、3Dゲーム、スタジオ品質のビデオで終わりました。

6コアのXeonと386番目のプロセッサーのパフォーマンスが1000倍異なるにもかかわらず、この記事の半分はMicrosoft Word 6.0の古いシステムで問題なく入力されました。一部のタスクは、強力な最新システムをまったく必要としません。それらのパフォーマンスは、「便利さ」（クラウドとの同期など）とコードの記述方法によってかなり消費されます。WinRARで2日間のストレステストを行ったところ、私の386番目のシステムは最速ではないかもしれませんが、安定していて信頼性があるかもしれません。ですから、研究を続けていきます。
