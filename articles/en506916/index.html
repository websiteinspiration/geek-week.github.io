What is a Service Mesh?

Hello again! .. In anticipation of the start of the course “Software Architect”, we have prepared another useful translation.



Service Mesh is a configurable low-latency infrastructure layer that is needed to handle a large amount of network interprocess communications between application programming interfaces (APIs). Service Mesh provides fast, reliable, and secure communication between containerized and often ephemeral application infrastructure services. Service Mesh provides features such as service discovery, load balancing, encryption, transparency, traceability, authentication and authorization, as well as support for a circuit breaker pattern .
Service Mesh is typically implemented by providing each instance of the service with a proxy instance called Sidecar. Sidecar process communications between services, monitor and fix security problems, that is, everything that can be abstracted from individual services. Thus, developers can write, maintain and maintain application code in services, and system administrators can work with Service Mesh and run the application.

Istio from Google, IBM and Lyft is currently the most famous Service Mesh architecture. Kubernetes, originally developed by Google, is now the only container orchestration framework supported by Istio. Vendors are trying to create commercial supported versions of Istio. Interestingly, they will be able to bring new things to the open source project.

However, Istio is not the only option, as other implementations of Service Mesh are being developed. Patternsidecar proxyis the most popular implementation, as can be judged by the projects Buoyant, HashiCorp, Solo.io and others. Alternative architectures exist: Netflix technological tools are one of the approaches where the Service Mesh functionality is implemented through the Ribbon, Hysterix, Eureka, Archaius libraries, as well as platforms such as Azure Service Fabric.

The part of the Service Mesh application that manages network traffic between instances is called Data Plane . Creating and deploying a configuration that controls the behavior of the Data Plane is done using a separate Control Plane . Control Plane typically includes or is designed to connect to an API, CLI, or GUI to control an application.


The Service Plane's Control Plane distributes the configuration between Sidecar Proxy and Data Plane.

Often, the Service Mesh architecture is used to solve complex operational tasks using containers and microservices. Pioneers in the field of microservicesare companies such as Lyft, Netflix and Twitter, which provide stable services to millions of users around the world. ( Here you can find a detailed description of some of the architectural challenges Netflix faced ). For less demanding applications, simpler architectures are likely to be sufficient.

Service Mesh architecture is unlikely to ever answer all the questions related to the operation of applications and their delivery. Architects and developers have a huge arsenal of tools, and only one of them is a hammer, which among many tasks should solve only one thing - hammer in nails. Microservices Reference Architecture by NGINX, for example, includes several different models that provide a continuous range of approaches to solving problems using microservices.

Elements that are combined in the Service Mesh architecture, such as NGINX, containers, Kubernetes, and microservices as an architectural approach, can be equally productively used in implementations without Service Mesh. For example, Istio was designed as a full-fledged Service Mesh architecture, but its modularity suggests that developers can select and apply only the technology components they need. Keeping this in mind, it is necessary to form a clear understanding of the concept of Service Mesh, even if you are not sure that you will ever be able to fully implement it in the application.
