Utilisation d'événements personnalisés dans htmlix.js

Cet article examinera les principes de base de l'utilisation des événements utilisateur dans Htmlix et créera une petite application de quatre composants: un formulaire pour entrer un nom d'utilisateur, un message d'accueil, un bouton de sortie pour une application et un tableau avec les utilisateurs de l'application.



Un exemple prêt à l'emploi peut être cliqué ici .



Vous pouvez lire l'API htmlix ici.



Après avoir entré le nom d'utilisateur dans le formulaire, l'application affichera son nom dans le message d'accueil, quittera le bouton et créera un nouveau conteneur dans le tableau avec les utilisateurs, et enregistrera les données dans localStorage. Après avoir rechargé la page, il prendra le nom de localStorage. Un clic sur le bouton «Quitter le profil» supprimera les données de tous les composants et la variable de localStorage.



Créons quatre composants: le formulaire de connexion - formulaire, le message d'accueil - message d'accueil, le bouton de déconnexion - déconnexion et le tableau avec tous les visiteurs - tableau_utilisateurs, et voyons comment nous créerions une communication entre eux si nous n'utilisions pas les événements utilisateur.

Dans l'exemple ci-dessus, nous avons enregistré les données de la propriété "input" du composant de formulaire dans la variable "text". Nous devons maintenant les afficher dans les propriétés des composants d'accueil et de déconnexion, et également créer un nouveau conteneur dans le tableau users_array. Pour comprendre pourquoi des événements personnalisés sont nécessaires, essayons d'abord de le faire de différentes manières sans les utiliser.



1. manière - dans la méthode de clic de la fiche elle-même, accédez directement à chaque composant et définissez la propriété text: 




        event.preventDefault();
        var text = this.parent.props.input.getProp();
                //  
                this.rootLink.state["greeting"].props.user_name.setProp(text);
                this.rootLink.state["logout"].props.user_name.setProp(text);
                  
                  ///  
                 this.rootLink.state["users_array"].add({user_name: text});


L'inconvénient de cette approche est évident, que se passe-t-il si nous obtenons des données pour cette propriété ailleurs? Par exemple, en fonction du stockage local, ils se chargeront automatiquement.



2. La méthode consiste à créer une méthode dans l'objet stateMethods et à l'appeler lorsque les données changent: 




ststeMethods: {

         entryUser: function(text){

                this.state["greeting"].props.user_name.setProp(text);
                this.state["logout"].props.user_name.setProp(text);
                 this.state["users_array"].add({user_name: text});
       }
}

window.onload = function(){
             var name = window.localStorage.getItem('user_name');

             if(name)HM.stateMethods.entryUser.call.(HM, name); ///    = HM
}

// 
click: function(){
                  event.preventDefault();
		  var text = this.parent.props.input.getProp();
                  this.rootLink.stateMethods.entryUser.call.(this.rootLink, text); 
}


Dans ce cas, nous avons réussi à éviter la duplication de code, mais le problème est que les méthodes de travail avec les propriétés des composants ne sont pas à l'intérieur, mais dans des fonctions externes. Et si avant de définir une propriété, nous devons la formater d'une manière ou d'une autre, différemment pour chaque composant, ou si nous devons définir non pas une propriété, mais plusieurs, il est donc préférable lorsque les composants eux-mêmes fonctionnent avec leurs propriétés.



3. le moyen est de transférer toute la logique du travail avec les propriétés aux composants et d'y créer des méthodes avec lesquelles appeler certaines actions. 



	
	greeting: {
		
		container: "greeting",

                //     
		props: [ "user_name", ["greet_user", "aux"] ], 
		methods: {
                       
                     greet_user: function(name){
                                this.props.user_name.setProp(name+" !!!");
                     }
	        }
	},
	logout: { 		
		container: "logout",

		props: [ "user_name", ["set_name", "aux"]], //       
		methods: {

                     set_name: function(name){
                                this.props.user_name.setProp(name);
                     }
                          
		},	
	},
	users_array: { 
		arrayProps: [["entry_user", "aux"]],  ///      
                arrayMethods: {
                       entry_user: function(name){
                              this.add({user_name: name})
                       }
               }	
		container: "user",
		props: [ "user_name", ],

         entryUser: function(text){

                this.state["greeting"].methods.greet_user(text);
                this.state["logout"].methods.set_name(text);
                this.state["users_array"].methods.entry_user(text);
        }
   }
   //     entryUser


Maintenant, les composants sont encapsulés, cependant, nous accédons au composant par son nom. Que se passe-t-il si nous voulons plus tard modifier le nom du composant ou de la méthode, ou supprimer complètement le composant, ou nous aurons plusieurs de ces composants qui écoutent la même propriété. Ou nous aurons de nombreuses propriétés de ce type, nous devrons donc créer de nombreuses méthodes de ce type.



Pour faciliter cette tâche, vous pouvez utiliser des événements personnalisés. Le composant s'abonne à l'événement et, lorsqu'il se produit, fait quelque chose avec ses propriétés.



Modifions le code, en utilisant maintenant des événements personnalisés: 



var StateMap = {

   eventEmiters: {	
             //   -  	
             ["emiter-entry-user"]: {prop: ""}, 		
   },
   form: {
	container: "form",
	props: ["input", "click"],
	methods: {
			
		click: function(){
		    event.preventDefault();
		    var text = this.parent.props.input.getProp();

		     //        
		    this.rootLink.eventProps["emiter-entry-user"].setEventProp(text); 	
		    window.localStorage.setItem('user_name', text);							
		}			
	},		
},	
greeting: {
		
	container: "greeting",
	props: [ "user_name", ['listen_entry_user', "emiter-entry-user", "" ] ], //   "emiter-entry-user"
	methods: {
			
		listen_entry_user: function(){
		   //      
		   this.parent.props.user_name.setProp( this.emiter.getEventProp() );
		},	
	},		
},
logout: { 
		
	container: "logout",
	props: [ "user_name", ["listen_entry_user", "emiter-entry-user", "" ]], //   "emiter-entry-user"
	methods: {
			
		listen_entry_user: function(){
		     //       
		     this.parent.props.user_name.setProp( this.emiter.getEventProp() );
		},			
	},	
},
    //      , ..            
users_array: { 
		
	arrayProps: [ ['listen_entry_user', "emiter-entry-user", ""] ], //   "emiter-entry-user"
	   arrayMethods: {
			
		listen_entry_user: function(){
			//       
			this.parent.add( {user_name: this.emiter.getEventProp()} );
		},			
	},		
	container: "user",
	props: [ "user_name",   ],
	methods: {						
		}
   },        	


}
window.onload = function(){
	
		
	var HM = new HTMLixState(StateMap);
	var name = window.localStorage.getItem('user_name');
	
	if(name != null)HM.eventProps["emiter-entry-user"].setEventProp(name); ///  "emiter-entry-user"        	
}


Ainsi, nous nous sommes débarrassés de la méthode intermédiaire "entryUser" et soulevons simplement l'événement en lui passant de nouvelles données dans le formulaire et après le chargement de la page. Maintenant, si nous devons changer quelque chose dans le composant, ou le supprimer complètement, nous n'avons pas besoin de corriger le code dans chaque fonction, qui peut être n'importe quel nombre. Et pour vous désabonner de tout événement, vous pouvez simplement supprimer l'écouteur dans le composant, ou le désactiver temporairement en utilisant la méthode disableEvent () dans la propriété - abonné.
