ソースコードをいじるだけでライブシステムにアクセスできなくなる方法

実際の例でのセキュリティは、常により興味深いものです。

クライアントがペネトレーションテストを要求すると、彼は非常に多くの懸念の理由を持っていました。とりわけ、次のように聞かれました。これはどのように私を脅かすことができますか？ライブシステムへのアクセスは彼に与えられませんでした。」

ライブシステムで重大な穴になる可能性がある開発者のミスについては触れません。すべてがはるかに簡単です。ソースコード自体に直接の命令とアクセスが含まれている場合があります。

セキュリティについてテストしたさまざまなプロジェクトのうち、ソースコードのみでシステム自体に侵入できる実際の例を示します。これらすべての問題領域が修正され、スクリーンショットの追加情報は非表示になっています。

システム1.
gitaからクローンされたコードは最新バージョンだけでなく、すべての変更の履歴でもあります。通常、機密情報のソースコードはgittyleaksを使用して実行します。

このプロジェクトでは、コードから削除された秘密暗号化キーを見つけましたが、ライブシステムで引き続き使用されていました。このキーは認証に使用され、メカニズムを知っているため、管理者を含むすべてのユーザーの認証Cookieを生成できました。


図1.出力：gittyleaks --find-anything


システム2. gitユーティリティを
使用できます削除されたすべてのファイルを表示するように彼女に依頼します。このシステムでは、プロジェクトのルートにあり、SSHチャネル経由でプロジェクトをサーバーに自動的にデプロイするために使用されたdeployer.pemファイルが見つかりました。ライブシステムのSSHポートが開かれています。なぜ開いているのですか？開発者は、ビルドマシンが動的IPアドレスの背後にあると答え、ポートを閉じないことを決定しました-「とにかく、誰もSSHキーを取得することはありません」。 Gee-gee ...


写真2.出力：git log --diff-filter = D --summary

System 3。
このシステムにより、すべてがさらに簡単になりました。データベースのスクリプトを調べて、デフォルトでユーザーがどのように作成されるかを確認することは価値があります。通常、最も高い権限を持つ管理者を作成する最初のユーザー。スクリプトで、実際のパスワードからハッシュを生成してデータベースに書き込んだコードを見つけました。驚いたことに、5人のユーザーが作成されましたが、最も重要な管理者のパスワードは、驚いたことに、ライブシステムを通過しました。ちなみに、このコードは1年目ではなく、1人も使ったことはありません。


図3.データベーススクリプトと同様に、実際のパスワードを見つけることができます
