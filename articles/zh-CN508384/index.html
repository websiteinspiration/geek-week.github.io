如何在不诉诸暴力的情况下使开发人员熟悉您的业务价值

的话题这个话题是有争议的，并且有大量的开发人员和管理人员不考虑彼此之间的通信规则和方法，特别是关于开发人员和客户之间的通信。



但是对于我和我的团队来说，这样的障碍被克服了（并且不止一次）并且取得了优异的成绩，所以我不禁分享自己的成功经验：



所以





商业就是商业，发展就是发展，直到...


我看到了一篇有关如何结交猫狗朋友的文章。来自母亲行为主义者的生活骇客：



“将狗放在一边，然后将猫放在背上。他们不会看到对方，只会闻到气味。这项运动将有助于建立对动物的信心。开始用轻松的音乐抚摸两者。”



该解决方案可能适用于利基种间三人行。我尝试过-我几乎迷失了眼睛和猫。



关于同样的损失和感情，“商务”与“发展”相识的问题向我提出。



但是在这种情况下，我的案子受到的伤害较小。我认为，引入业务和发展应该是有条理的，不引人注目的。



翻译困难



通常，开发人员甚至对您的业务或客户的业务都不了解。然后他打开指令：



步骤1.来并要求重写RUST上的所有内容。



步骤2.精疲力尽，因为您不会愿意重写RUST上的所有内容。



if (___RUST == false) then .({immediately: true})





那么开发人员为什么不想要了解您的业务？



寻找意义和目的



任何企业的目标都是相同的- 使他人受益。为此，他们用法定货币和社会资本付款。



我的经验是，开发人员通常看不到带来其他人的好处。



因此，他很难评估自己的贡献-他只看到该代码和该代码的钱。



因此，他的动机（重写RUST上的所有内容）是基于“编写顶级代码”，而不是基于“为用户释放功能（“ benefit”的同义词）。



说明



通常，公司有不同的部门：市场营销，技术支持，分析，销售，策略。而且，与每个人相距甚远的是一个称为“发展部”的半自治保留区，该保留区是通过特殊的外交官翻译（PM，团队负责人）传达的。



在实践中，通常不允许开发人员来客户或公司。开发人员与业务流程的隔离和隔离使他们无法更好地理解任务。有时它是有道理的，有时不是。



种族隔离正是将开发者与受益者分离开来的地方。随着时间的流逝，在清洁发展的国家，开发人员越来越远离商业现实。这常常忽略了简单的人类欢乐和痛苦。



建议



这样一来，您的开发人员就不会变成说Matan和Haskell混合语言的自闭症专家，并逐步将其介绍给客户和企业。重要的是，开发人员应时不时地直接从客户那里听到“谢谢，这很着急！” 而且“嗯，EPT，这不好。”



成为业务流程中必要而重要的参与者的乐趣将引起人们重复这种情况的渴望，并将内部安装的重点从“编写顶级代码”转移到“受益”，这反过来将加快功能（而非代码）的发布和质量。


（关于信任，合理的自主权，我想写的团队中的健康小气候还有其他事情。但是今天，没有感伤的是另一次。）



如何做到这一点



我是这样的：



  1. 有时，我驱使开发人员参加与企业的会议。
  2. 我不时邀请您致电客户。
  3. 在每个月的月底，我都会安排一次会议，告诉您已经实现了哪些指标以及带来了哪些好处。
  4. 我给客户/企业提供了直接与程序员一起完成任务的机会。


重要的是要注意！我始终反对让负责人以外的其他人直接为开发人员设置任务，但是以上几点并不违反该规则。



当然，所有这些都在工作时间内发生，并在任务跟踪器中指示为开发人员的个人任务。此类活动应至少每周进行1-2次，对于初学者而言，这种活动可能会更频繁。



让开发人员知道他不仅可以支付或编写代码，还可以享受更多的乐趣。但也要从它带给他人的利益的实现中来。与其他有能力的人一起解决业务问题时不可避免的嗡嗡声和痛苦。


是的，有些不愿与任何人互动的同志。他们甚至与同事沟通困难。这不是关于害羞，而是关于积极隔离，甜蜜的厌恶行为。有用的性格特征，但不适用于复杂项目的协作。



我不想雇用这样的程序员，因为团队合作比个人才能更重要。

开发人员和企业-不能取悦于猫和狗。开始逐步介绍它们，您将获得快速的幸福。
