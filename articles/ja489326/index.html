効果的なエンタープライズテストに関する考察

こんにちは、ハブル！

最近、テストトピックの徹底的な調査に戻りました。予測可能な計画では、ユニットテストに関する優れた本も用意しています。同時に、このトピックでは他のどこよりもコンテキストが重要であると考えているため、本日、Java EEのスペシャリストであるSebastian Dashner のブログで公開された2つの出版物（1つにまとめたもの）の翻訳を提供します。効率的なエンタープライズテストについての考え。」

エンタープライズテストは、私たちが望むほど詳細にはまだ検討されていないトピックです。テストの作成、特にテストのサポートには多くの時間と労力がかかりますが、テストを中止して時間を節約しようとすることはオプションではありません。テストの有効性を高めるために、調査する価値のあるタスク、アプローチ、テストテクノロジーの量はどれくらいですか？

前書き


さまざまな種類のテストとその範囲に関係なく、一連のテストを準備する重要な点は、本番環境でアプリケーションが期待どおりに機能することをこの資料で確認することです。このような動機は、システムをユーザーの視点から考えると、システムがタスクを満たしているかどうかを確認する際の主な動機となるはずです。

アテンションスパンとコンテキストの切り替えは考慮すべきことであるため、テストが長期間実行およびテストされないこと、およびテスト結果が予測可能であることを確認する必要があります。コードを記述する場合、コードの迅速な検証（1秒以内に実現可能）が重要です。これにより、高い生産性と作業中の集中力が保証されます。
一方、テストのサポートを確保する必要があります。ソフトウェアは頻繁に変更され、機能テストでコードがかなりカバーされているため、製品コードの機能変更ごとにテストレベルでの変更が必要になります。理想的には、テストコードは、機能（ビジネスロジックなど）が変更されたときにのみ変更し、不要なコードをクリーンアップしてリファクタリングするときに変更しないでください。一般に、テストシナリオには、機能しない、構造的な変更の可能性を含める必要があります。

テストのさまざまな適用分野を検討すると、問題が発生します。これらの分野のうち、時間と労力を費やす価値があるのはどれですか。たとえば、マイクロサービスアプリケーションや、コードの配布と統合に関する重要な作業を提供するシステムでは、統合テストが特に重要であり、システムの境界を模索するのに役立ちます。したがって、このアプリケーションの環境と構造をできるだけ本番環境に近い形で維持しながら、ローカル開発中にアプリケーション全体をテストする効果的な方法が必要です。

ユニットテストは、単一のモジュール（通常はクラス）の動作をチェックしますが、モジュールの構造に関連しないすべての外部要因は無視またはシミュレーションされます。単体テストでは、個々のモジュールのビジネスロジックを検証する必要があります。それ以上の統合や構成をチェックする必要はありません。

私の経験では、ほとんどのエンタープライズ開発者は、単体テストのコンパイル方法についてかなりの知識を持っています。これを印象づけるために、私のコーヒーテストプロジェクトでこの例を見ることができます。。ほとんどのプロジェクトでは、JUnitをMockitoと組み合わせて使用​​して依存関係をシミュレートし、理想的にはAssertJを使用して読み取り可能なステートメントを効率的に定義します。ユニットテストは特別な拡張機能やスターターなしで実行できること、つまり通常のJUnitでこれを実行できることを常に強調します。説明は単純です。何百ミリ秒ものテストを数ミリ秒で実行する機能が必要なため、それはすべてランタイムに関するものです。

原則として、単体テストは非常に高速に実行され、複雑なテストスイートや特別なワークフローを簡単に組み立てることができます。それらは実行が簡単で、テストのライフサイクルに制限を課さないためです。

ただし、テストするクラスの依存関係をシミュレートする多数の単体テストがある場合、欠点が1つあります。それらは実装に密接に関連している（これは特にクラスとメソッドに適用されます）ため、コードのリファクタリングが困難です。つまり、本番コードのリファクタリングごとに、テストコードを変更する必要があります。最悪の場合、開発者はリファクタリングを部分的に拒否し始めます。これは単に負担が大きすぎて、プロジェクト内のコードの品質が急速に低下しているためです。理想的には、開発者が要素をリファクタリングおよび再配置できる必要があります。ただし、これにより、ユーザーが気付くようなアプリケーションの変更がない場合に限られます。ユニットテストは、必ずしも本番用コードのリファクタリングを単純化するわけではありません。

しかし経験から、ユニットテストは、簡潔なロジックで密に満たされたコードや、特定の関数（アルゴリズムなど）の実装を記述し、同時に他のコンポーネントとあまり活発に対話しないコードをチェックするのに非常に効果的であることが示唆されています。特定のクラスのコードの複雑さや密度が低いほど、その循環的複雑度は低くなり、他のコンポーネントとの相互作用が活発になるほど、このクラスをテストするときのユニットテストの効果は低くなります。特に、比較的少ないビジネスロジックが含まれているが、外部システムとの広範な統合が提供されているマイクロサービスの場合は、多くの場合、ユニットテストを使用する必要はほとんどありません。このようなシステムでは、個々のモジュール（まれな例外を除く）には通常、特殊なロジックはほとんど含まれていません。これを決定するときに考慮する必要があります時間と労力を費やすのにより適切なものは何か。

アプリケーション状況のテスト


テストと実装を緊密にリンクする問題に対処するために、少し異なるアプローチを試し、テストの範囲を拡大することができます。私の本では、より良い用語を見つけることができなかったため、コンポーネントテストについて書きました。しかし、本質的には、この場合、適用された状況のテストについて話している。

アプリケーション状況テストは、組み込みコンテナーを使用しない、コードレベルで動作する統合テストです。これらは、起動を高速化するために中止されました。彼らは、境界メソッドから、特定の実際のケースで通常使用される、よく調整されたコンポーネントのビジネスロジックをテストし、次に、それらに関連付けられているすべてのコンポーネントまでテストします。データベースなどの外部システムとの統合は、モックを使用して模倣されます。

コンポーネントを自動的に接続するより高度なテクノロジーを使用せずにそのようなシナリオを構築することは、大きな作業のように思えます。ただし、再利用可能なテストコンポーネントを定義します。これらは、テスト構成をシミュレート、接続、および追加することによってコンポーネントを拡張するテストの対応物でもあります。これはすべて、リファクタリングに必要な総作業量を最小限に抑えるために行われます。目標は、変更の影響の程度をテスト分野の単一のクラス（または複数のクラス）に制限する唯一の責任を作成することです。再利用を視野に入れてこのような作業を行うことで、必要な作業の総量が削減され、プロジェクトが大きくなるとそのような戦略が正当化されますが、各コンポーネントはわずかな修理のみで済み、この作業はすぐに償却されます。

これらすべてをよりよく想像するために、コーヒーの順序を記述するクラスをテストするとします。このクラスにはCoffeeShop、およびの2つのクラスが含まれていOrderProcessorます。



テストダブルのクラス、CoffeeShopTestDoubleそしてOrderProcessorTestDoubleそれらは*TD、彼らがコンポーネントを継承プロジェクトのテストエリアに位置しているCoffeeShopと、OrderProcessorプログラムの主なエリアに位置します。テストの担当者は、必要なシミュレーションと接続ロジックを設定し、このアプリケーションで必要なシミュレーションメソッドまたは検証メソッドを使用して、クラスのパブリックインターフェイスを拡張することができます。

次に、コンポーネントのテストダブルクラスを示しますCoffeeShop。

public class CoffeeShopTestDouble extends CoffeeShop {

    public CoffeeShopTestDouble(OrderProcessorTestDouble orderProcessorTestDouble) {
        entityManager = mock(EntityManager.class);
        orderProcessor = orderProcessorTestDouble;
    }

    public void verifyCreateOrder(Order order) {
        verify(entityManager).merge(order);
    }

    public void verifyProcessUnfinishedOrders() {
        verify(entityManager).createNamedQuery(Order.FIND_UNFINISHED, Order.class);
    }

    public void answerForUnfinishedOrders(List<Order> orders) {
        //     
    }
}

テストdoubleクラスは、CoffeeShop基本クラスのフィールドとコンストラクターにアクセスして、依存関係を確立できます。ここでは、テストツインの形式で、他のコンポーネントのバリアントも使用されます。特に、 OrderProcessorTestDouble実際のケースの一部である追加のシミュレーションまたは検証メソッドを呼び出すために必要です。

テストdoubleのクラスは再利用可能なコンポーネントであり、それぞれが各プロジェクトのスコープごとに1回記述され、その後多くの実際的なケースで使用されます。

class CoffeeShopTest {

    private CoffeeShopTestDouble coffeeShop;
    private OrderProcessorTestDouble orderProcessor;

    @BeforeEach
    void setUp() {
        orderProcessor = new OrderProcessorTestDouble();
        coffeeShop = new CoffeeShopTestDouble(orderProcessor);
    }

    @Test
    void testCreateOrder() {
        Order order = new Order();
        coffeeShop.createOrder(order);
        coffeeShop.verifyCreateOrder(order);
    }

    @Test
    void testProcessUnfinishedOrders() {
        List<Order> orders = Arrays.asList(...);
        coffeeShop.answerForUnfinishedOrders(orders);

        coffeeShop.processUnfinishedOrders();

        coffeeShop.verifyProcessUnfinishedOrders();
        orderProcessor.verifyProcessOrders(orders);
    }

}

コンポーネントテストでは、エントリポイントで呼び出されるビジネスロジックの特定のケース（この場合）を検証しCoffeeShopます。このようなテストは、すべての接続とシミュレーションが別々のテストツインで実行されるため、簡潔で読みやすく、後でなどの高度に専門化されたスクリーニング手法を使用できるため、簡単に読み取ることができますverifyProcessOrders()。

ご覧のように、テストクラスは本番クラスのスコープを拡張し、mokeeをインストールして、動作を確認するメソッドを使用できるようにします。このようなシステムのセットアップには多くの労力がかかるようですが、プロジェクト全体のフレームワーク内でコンポーネントを再利用できる多くの実用的なケースがある場合、これらのコストはすぐに償却されます。プロジェクトが大きくなるほど、このアプローチはより有用になります。特に、テストの完了にかかる時間に注意を払う場合はそうです。すべてのテストケースは引き続きJUnitを使用して実行され、最短の時間で数百回実行されます。

これがこのアプローチの主な利点です。コンポーネントテストは通常​​の単体テストと同じ速さで実行されますが、単一のコンポーネントまたはいくつかのコンポーネントのみを変更する必要があるため、製品コードのリファクタリングを促進します。さらに、サブジェクト領域に固有の表現力豊かなチューニングおよび検証方法を使用してテストの対応物を改善し、コードの可読性を高め、その使用を容易にし、テストスクリプト内のステレオタイプコードを取り除きます。
