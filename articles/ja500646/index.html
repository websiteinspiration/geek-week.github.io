JVMとバイトコードはWORAコンセプトの基礎であり、ニュアンスや各OSごとにコンパイルする必要性から解放されます。

JVMキッチン


JVMランタイムメモリを見てみましょう。



JVMは私たちのキッチンだと言えます。次に、残りの参加者を検討します。

メソッド領域-クックブック



メソッド領域には、各関数のコンパイル済みコードが格納されます。スレッドが関数の実行を開始すると、通常、この領域から命令を受け取ります。実際、それはスクランブルエッグからカタルーニャザルスエラまですべてを調理する方法を詳述した料理のレシピ本です。

スレッド1..N-チームクック



ストリームは、それらによって指定された指示（メソッド領域）に厳密に従います。このため、ストリームにはPCレジスタとJVMスタックがあります。各ストリームを、クックブックのレシピに正確に従って、与えられた割り当てを実行するクックと比較できます。

PCレジスタ-フィールドノート



Program Counter Register-ストリームのコマンドのカウンター。実行中の命令のアドレスを格納します。キッチンでは、これらは今私たちが料理本のどのページにあるかについてのいくつかのメモでしょう。

JVMスタック


フレームのスタック。フレームは関数ごとに割り当てられ、その中で現在のスレッドは変数とオペランドを処理します。ピクルスの準備との類似性の一部として、これはネストされた操作のセットになる可能性があります。

フレーム-デスクトップ



フレームは料理人のデスクトップとして機能し、その上にまな板と署名済みのコンテナがあります。

ローカル変数-署名されたコンテナー



これはローカル変数の配列（ローカル変数テーブル）であり、名前が示すように、ローカル変数の値、タイプ、およびスコープを格納します。これは、プロのアクティビティの中間結果を追加できる署名付きコンテナに似ています。

オペランドスタック-まな板



オペランドスタックは、JVM命令の引数を格納します。たとえば、加算操作の整数値、ヒープオブジェクトへの参照など。

私が提供できる最も近い例は、トマトとキュウリが1点でサラダになるまな板です。ローカル変数とは異なり、次の命令を実行するときに使用するものだけをボードに配置します。

ヒープ-分布表



フレームの操作の一部として、オブジェクトへのリンクを操作します。オブジェクト自体はヒープに格納されます。重要な違いは、フレームは1つのスレッドにのみ属し、ローカル変数はフレームが生きている間（関数が実行されている間）「生きている」ことです。ヒープは他のストリームにアクセスでき、ガベージコレクターがオンになるまで存続します。キッチンに例えると、それだけで一般的な分布表の例を示すことができます。そして、それはクリーナーの別のチームによって掃除されます。

JVMキッチン。内側からの様子。マルチスレッド


今、この例を考えてみましょう：

    fun add(ingredient: Any) {
        ingredientsCount = ingredientsCount.inc()
        //- 
    }

これは、スレッド問題の典型的な例です。成分数がありますingredientsCount。関数addは、成分を追加することに加えて、増分を実行しますingredientsCount。

逆アセンブルされたバイトコードは次のようになります。

    ALOAD 0
    ALOAD 0
    GETFIELD com/company/Solenya.ingredientsCount : I
    ICONST_1
    IADD
    PUTFIELD com/company/Solenya.ingredientsCount : I

命令が実行されるときのオペランドスタックの状態：


1つのスレッドで作業する場合、すべてが正しく実行されます。複数のスレッドがある場合、次の問題が発生する可能性があります。両方のスレッドが同時にフィールド値ingredientsCountを取得し、それをスタックに書き込んだとします。すると、オペランドスタックとフィールドの状態は次のingredientsCountようになります。


関数は2回（各スレッドで1回）実行され、値ingredientsCountは2に等しくなるはずです。しかし、実際には、スレッドの1つが古い値ingredientsCountで動作していたため、実際の結果は1（更新の喪失の問題）です。

状況は、料理にスパイスを加えるシェフのチームの並行作業に似ています。想像してみてください：

  1. 料理（ヒープ）が置かれている分布表があります。
  2. キッチンには2人のコックがい​​ます（スレッド* 2）。
  3. 各料理人は、スパイスの混合物を準備する独自のカッティングテーブルを持っています（JVMスタック* 2）。
  4. タスク：スパイスの2サービングを皿に追加します。
  5. 配布テーブルには、ユーザーが読み書きする用紙があり、どの部分が追加されたかが記載されています（ingredientsCount）。そして、スパイスを節約するために：
    • スパイスの準備を始める前に、料理人は追加したスパイスの数が十分ではないことを紙の上で読む必要があります。
    • スパイスを追加した後、料理人は彼の意見では、スパイスが皿に追加された数を書くことができます。

このような状況では、状況が発生する可能性があります。

  1. クック＃1は、スパイスの3サービングが追加されたことを読んだ。
  2. クック＃2はスパイスの3サービングが追加されたことを読んだ。
  3. どちらも自分の机に行き、スパイスの混合物を準備します。
  4. 両方のシェフが料理にスパイス（3 + 2）を追加します。
  5. クック＃1は、スパイスの4サービングが追加されたと書いています。
  6. クック＃2は、スパイスの4サービングが追加されたと書いています。

結論：商品が欠けていた、料理が辛いことが判明したなど。

このような状況を回避するために、ロック、スレッドセーフ関数などのさまざまなツールがあります。
