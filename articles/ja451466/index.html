graphql-落とし穴

おそらく完璧な技術はありません。Graphqlも例外ではありません。このテクノロジーの経験がない場合は、どのような問題が発生する可能性があるかを事前に把握し、事前に準備する必要があります。

まず、私は可能な限りgraphqlを使用する敵よりも支持者だと言います。そして、私は、このテクノロジーを使用することの適切さについて誰かを説得するつもりはありません。そして、それが私がgraphqlテクノロジーのフレームワークで未解決の問題に関連する質問を提起する理由です。

たとえば、誰かにとって、graphqlの各オブジェクトが少なくとも2回記述される必要があることは予期しない場合があります。1つはオブジェクトの戻り型として、もう1つはオブジェクトの入力型としてです（graphql.org/graphql-js/mutations-を参照） and-input-types）。しかし、私はこれを最初に言って、それを重大な欠点とは考えていません。今日は、graphqlテクノロジーを使用してアプリケーションを開発する際に、原則として解決する必要がある次のような問題について説明します。

  1. ユーザーとユーザーグループのアクセスの分離。
  2. エラー処理。
  3. SELECT N + 1問題

ユーザーとユーザーグループのアクセスの共有


graphqlは、ユーザーとグループのアクセスの共有については何も知りません。したがって、アクセスを共有するすべての作業は、アプリケーション開発者の責任です。3番目のパラメーターは、アプリケーションコンテキストオブジェクトをリゾルバー関数に渡します。したがって、たとえば、graphql JavaScript + expressの実装で作業する場合、コンテキストパラメータで、リクエストのexpress.jsオブジェクトから現在のユーザーを取得できます。ただし、アクセス制御に関する追加の作業は、各リゾルバーで直接実行する必要があります。

function(root, {id}, ctx) {
   return DB.Lists.get(id)
     .then( list => {
       if(list.owner_id && list.owner_id != ctx.userId){
         throw new Error("Not authorized to see this list");
       } else {
         return list;
       }
    });
}

当然、このアプローチはアクセス制御を複雑にします。宣言的な方法でアクセス権を設定する方法はなく、権利の制御は数十（大規模なシステムの場合は数千）のリゾルバー関数に分散されます。したがって、この問題を解決する多くのライブラリがあります。それらのいくつかは非常に人気があります（github.comの星の数から判断して）、たとえばgithub.com/maticzav/graphql-shield。

エラー処理


フロントエンドが入力検証と検証に合格しなかった各フィールドの詳細メッセージの生成を必要とする場合、graphqlでのエラー処理はおそらく柔軟性に欠けると思われます。たとえば、入力パラメーターが文字列として記述されていて、数値が受信された場合、エラーメッセージはほとんど役に立ちません。

{
  "errors": [
    {
      "message": "Expected type String, found 1.",
      "locations": [
        {
          "line": 2,
          "column": 15
        }
      ]
    }
  ]
}

入力パラメーターのタイプに重大なエラーがある場合、エラーメッセージが自動的に生成され、このプロセスを制御する方法はありません。入力パラメーターのタイプによる検証が成功した場合は、オブジェクトをスローすることにより、カスタムエラーメッセージをクライアントに送信できますnew Error ('custom message ...')。カスタムフィールドをエラーオブジェクトに追加することはできません（エラーのカスタマイズは、apollo-server-expressライブラリとapollo-errorsライブラリを一緒に使用すると実装されます）。もちろん、messageサーバーでオブジェクトを文字列にシリアル化し、クライアントでデシリアライズすることは常に可能です。しかし、そうする必要があるのでしょうか？

SELECT N + 1問題


この問題については、メッセージで詳しく説明されています。

graphqlはリゾルバー関数に基づいて構築されています。つまり、データベースからデータをフェッチすると、SELECT N + 1と呼ばれる問題が発生する可能性があります。リゾルバー関数で、このオブジェクトに関連付けられたデータが識別子（外部キー）で表されているオブジェクトのリストが取得されたとします。そのような識別子ごとに、独自の関数リゾルバーが呼び出され、（それぞれの）データベースへの追加の要求が行われます。したがって、単一のデータベースクエリ（SQL JOINを使用）ではなく、多数のクエリが実行され、データベースにクエリが過負荷になります。

この問題を解決するために、facebookはgithub.com/graphql/dataloaderライブラリを開発しました保留中のリクエスト戦略を使用します。関数リゾルバーで直接要求を実行する代わりに、配列に識別子（2次キー）を蓄積し、1つの要求でそれらをすぐに受け取ることが提案されています。
