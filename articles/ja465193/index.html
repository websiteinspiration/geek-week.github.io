DockerコンテナーでのAndroidプロジェクトの構築

Androidプラットフォーム用のプロジェクトを開発する場合、最も小さいプロジェクトでも、遅かれ早かれ開発環境に対処する必要があります。 Android SDKに加えて、Kotlin、Gradle、platform-tools、build-toolsの最新バージョンが必要です。そして、開発者のマシンでこれらの依存関係がすべてAndroid Studio IDEを使用して大幅に解決されている場合、CI / CDサーバーでは各更新が頭痛の種になる可能性があります。 Web開発でDockerが環境問題の標準的な解決策になった場合は、Android開発で同様の問題を解決してみてください...

Dockerが何であるかを知らない人のために-それが非常に単純であれば、これはいわゆるDockerを作成するためのツールです。最小限のOSカーネルと必要なソフトウェアのセットが含まれている「コンテナ」。環境を維持しながら、好きな場所に展開できます。コンテナーに何が含まれるかは、Dockerfileで定義されます。Dockerfileは、任意の場所で起動され、べき等プロパティを持つイメージにアセンブルされます。

Dockerのインストールプロセスと基本は、公式Webサイトで完全に説明されています。したがって、少し先を見て、ここにそのようなDockerfileがあります。

# ..     Android-  Gradle, 
#        Docker- 
#            Gradle
FROM gradle:5.4.1-jdk8

#       Android SDK  
#    
ENV SDK_URL="https://dl.google.com/android/repository/sdk-tools-linux-3859397.zip" \
    ANDROID_HOME="/usr/local/android-sdk" \
    ANDROID_VERSION=28 \
    ANDROID_BUILD_TOOLS_VERSION=28.0.3

#  ,   SDK   ,
#    
RUN mkdir "$ANDROID_HOME" .android \
    && cd "$ANDROID_HOME" \
    && curl -o sdk.zip $SDK_URL \
    && unzip sdk.zip \
    && rm sdk.zip \
#          
#  .  .  Android    
#          
#      
#   
    && mkdir "$ANDROID_HOME/licenses" || true \
    && echo "24333f8a63b6825ea9c5514f83c2829b004d1" > "$ANDROID_HOME/licenses/android-sdk-license" \
    && echo "84831b9409646a918e30573bab4c9c91346d8" > "$ANDROID_HOME/licenses/android-sdk-preview-license"    

#   SDK   build-tools, platform-tools
RUN $ANDROID_HOME/tools/bin/sdkmanager --update
RUN $ANDROID_HOME/tools/bin/sdkmanager "build-tools;${ANDROID_BUILD_TOOLS_VERSION}" \
    "platforms;android-${ANDROID_VERSION}" \
    "platform-tools"

Androidプロジェクトのあるフォルダーに保存し、次のコマンドでコンテナーのアセンブリを開始します

docker build -t android-build:5.4-28-27 .

-t パラメーターは、コンテナーのタグまたは名前を指定します。これは通常、名前とバージョンで構成されます。私たちの場合、それをandroid-buildと呼び、そのバージョンでは、gradle、android-sdk、およびplatform-toolsの一連のバージョンを指定しました。将来的には、このような「バージョン」を使用して、必要な画像を名前で検索することが容易になります。

アセンブリが渡された後、ローカルでイメージを使用できます。それをdocker pushコマンドでパブリックまたはプライベートのイメージリポジトリにアップロードして、他のマシンにダウンロードできます。

例として、プロジェクトをローカルで収集します。これを行うには、プロジェクトフォルダーでコマンドを実行します

docker run --rm -v "$PWD":/home/gradle/ -w /home/gradle android-build:5.4.1-28-27 gradle assembleDebug

それが何を意味するかを分析してみましょう：

docker run --
rmイメージ自体を起動するコマンド-コンテナーを停止した後、その存続中に作成されたすべてを削除します
-v "$ PWD"：/ home / gradle /-Androidで現在のフォルダーをマウントしますコンテナ/ホーム/のGradle /内部のフォルダに-Project
-w /ホーム/のGradleのセット作業ディレクトリ-
アンドロイドビルドコンテナ5.4.1-28-27：我々は構築されたことを私たちのコンテナの名前-
GradleのassembleDebugを -どの収集ビルドコマンド自体を、私たちのプロジェクト

すべてがうまくいけば、数秒/数分後に、8m 3sでBUILD SUCCESSFULのような画面が表示されます。！そして、フォルダーapp / build / output / apkがアセンブルされたアプリケーションになります。

同様に、他のGradleタスクを実行できます-プロジェクトのチェック、テストの実行など。主な利点は、他のマシンでプロジェクトをビルドする必要がある場合、環境全体をインストールする必要がないため、必要なイメージをダウンロードしてアセンブリを実行するだけで十分であることです。

コンテナーは変更を保存せず、各アセンブリは最初から開始されます。これにより、一方では起動された場所に関係なく、アセンブリのIDが保証されます。他方では、すべての依存関係をダウンロードしてすべてのコードを再度コンパイルする必要があるたびに、これにはかなりの時間がかかる場合があります。したがって、通常の「コールド」スタートに加えて、いわゆるメンテナンスを行いながらアセンブリを開始するオプションがあります。 「キャッシュ」。プロジェクトの作業フォルダーにコピーするだけで〜/ .gradleフォルダーを保存し、次のビルドの最初にそれを返します。すべてのコピー手順を別々のスクリプトに入れ、起動コマンド自体は次のようになりました

docker run --rm -v "$PWD":/home/gradle/ -w /home/gradle android-build:5.4.1-28-27 /bin/bash -c "./pre.sh; gradle assembleDebug; ./post.sh"

その結果、平均的なプロジェクトの組み立て時間は数倍に短縮されました（プロジェクトへの依存関係の数によって異なりますが、平均的なプロジェクトは5分ではなく1分で組み立てられました）。

このすべてがそれ自体で意味を持つのは、独自の内部CI / CDサーバーがあり、そのサポートを自分で行っている場合のみです。しかし、今ではこれらの問題がすべて解決され、心配する必要がない多くのクラウドサービスがあり、必要なアセンブリプロパティをプロジェクト設定で指定することもできます。
