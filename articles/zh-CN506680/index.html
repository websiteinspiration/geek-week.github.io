.NET Core与Node.js 论点与事实

本文是前一篇文章的续篇，前一篇文章提出了许多有趣的问题。当我开始为评论写答案时，我意识到我并没说太多。因此，我决定写一篇新文章，但从另一个角度来看：增加了个人经验和因果关系。如果您还没有阅读前一篇，那没关系。您可能没有看过。

其主要思想是，在中期的2017年，我开始在由开发后端应用的主要开发堆栈的变化我们公司从.net（C＃）来的Node.js（打字稿）。这是怎么发生的，我对所做的决定感到后悔，我不回头吗？关于它下面。

背景


首先，关于自己的简短故事。我是程序员，而不是经理。他在2003年以学生的身份在IDE Visual Studio 6中使用C ++编写了他的第一个应用程序，这并不令人ham愧。那是一个使用D3D，DirectSound和当时DirectX 9几乎所有其他模块的变形金刚游戏，但为什么“我是程序员”。



我的同学写了这个游戏大约一年。这正是我从小就梦想的程序-高科技，它利用封装，多态性和OOP的其他魅力，以及棘手的算法和不寻常的数据结构（例如斐波那契堆）。

然后，对于没有工作经验和明确投资组合的三年级学生来说，这是在GameDev时期非常受欢迎的一家公司找到工作的唯一机会。但是什么也没有。经过4小时的采访，其中包括在纸上写了一种使汇编器上的两行相交的算法的采访，但被拒绝了，理由是我们现在不准备联系学生。

这次失败的几个月后，我来到了一家大型（超过100位程序员）软件公司，随着C＃的出现，我转而使用它。



通过与经验更丰富，素质更高的同事进行交流，在实践中，我掌握了一些有趣的东西：

  • 面向方面的编程（AOP），用于从业务逻辑中消除实用程序代码；

主要缺点是有趣的任务数量很少，其中一个任务可以应用算法和数据结构以及各种模式的累积知识。好吧，我是一名程序员，没有它我无法生存！



我工作的部门开发了接口，来自DBMS的数据可用于其他系统。一年又一年，这个想法开始在我的脑海中浮现，在一个毫无生气的企业中，一切都只是那样，而面向对象的任务（例如，“抓住它，稍微扭动它，然后把它放回去”）就是一个想法。

在REST API中的关系DBMS和数据协定级别上，只有希望在OOP上并且仅在OOP上进行编写的开发人员的热情才能支持OOP。而且，就复杂性和连通性而言，中间层（从数据库获取并提供给消费者）之间的逻辑从来都不是GameDev，并且可以以功能样式轻松编写。

在2015年中期，凭借8年的C＃经验，我去了SoftMediaLab从事有趣的任务。



得益于在后端和前端获得的经验，我立即成为了一个新的相当大的项目的开发负责人-在线p2p和b2p贷款平台。后端包括C＃.Net Framework 4.6，微服务体系结构，WCF微服务通信，postgreSQL 9.5，ORMLite，用于历史表的ORM类代码生成以及用于滚动/重新加载DBMS对象的脚本，DBMS中约150个表，用于AOP的postharp 2.1，用于AOP的套接字基于ASP.Net MVC的SignalR，登录页面和WEB API。在前端，angular.js和JS自当时的打字稿尚未普及。 CI / CD，流复制到备份节点。

Node.js


在2017年初，打字稿在前端表现出最好的表现之后，尝试编写打字稿后端的想法对我来说似乎是合乎逻辑的。

当时，我对后端开发有以下要求：

  • 必须通过代码生成或ORM自动创建用于滚动/重新绘制DBMS的代码，包括触发器功能和Histor表。
  • 应该使用代码生成自动创建用于创建DTO模型和/到它们的映射器的代码；
  • REST API的详细描述应从源代码自动生成；
  • 应该有一个成熟的ORM系统，该系统允许使用MS SQL，PostgreSQL，mongoDb并为特殊情况编写自定义查询。
  • redis, rabbitMQ, socket, .. ;
  • — ;
  • ;
  • ;
  • IDE, , .

我花了一个月的时间研究与代码生成无关的项目，并找到了诸如typeorm，express.js，socket.io，redis，typescript-rest-swagger等有价值的解决方案。

在2017年中期，node.js已经是一种相当稳定的技术。在Google上的几个小时内，我发现了大约十二种服务在节点上实现的高实时负载条件下工作。我还了解到，需要在C ++中优化性能瓶颈。

为了研究代码生成的可能性，我聘请了一位明智的技术学院毕业生，他在C＃方面经验很少，而在JS / TypeScript中通常没有经验。除其他外，要检查输入TypeScript的阈值。

一个聪明的毕业生每月的工作成果是三个图书馆-grunt生成视图模型，grunt生成历史模型，grunt生成数据库，我们公司仍在项目中使用。仅前缀“ grunt-”不再相关。它从原始版本扩展为原始版本，当时它们还用作grunt项目构建器的插件。

在2017年夏季末，我召集了一次技术专家会议，在会议上我介绍了node.js的功能，并以多数票决定将下​​一个商业项目打包成一个node.js（TS）+ angular.io（TS）捆绑销售。

碰巧的是，这时我们公司的两个项目同时启动：

  • 一个允许办公室员工在一个窗口中通过Telegram，WhatsApp，FB，VK和Viber与客户进行通信的通信平台；
  • 和一个使我们的客户之一的内部业务流程自动化的项目。

这两个项目均已在新堆栈上成功实现。从那时起，我们一直在进行TS上的所有新项目。node.js的后端，react的前端（为什么要从angular.io切换到react是另一篇文章的主题）以及react-native上的移动应用程序。
