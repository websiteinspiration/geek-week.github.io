みなさん、こんにちは！

私の名前はアレクサンダーです。私はLinxdatacenterのネットワークエンジニアです。今日の記事では、トラフィックエクスチェンジポイント（インターネットエクスチェンジポイント、IXP）について説明します。その外観の前に何があったか、どのようなタスクで解決され、どのように構築されているかについて説明します。また、この記事では、EVP-NGプラットフォームとBIRDソフトウェアルーターを使用してIXPがどのように機能するかを示し、「内部」で機能する方法を理解できるようにします。

ちょっとした歴史


あなたが見れば、ここで、あなたはトラフィック交換ポイント数の急速な成長は1993年に始まったことがわかります。これは、当時存在していた通信事業者のトラフィックのほとんどが米国のバックボーンネットワークを通過したためです。したがって、たとえば、トラフィックがフランスのオペレーターからドイツのオペレーターに送信された場合、最初にフランスからアメリカに送信され、次にアメリカからドイツに送信されます。この場合のバックボーンネットワークは、フランスとドイツ間のトランジットとして機能しました。 1つの国のトラフィックでさえ、直接ではなく、アメリカの通信事業者のコアネットワークを通過することがよくあります。

このような状況は、トランジットトラフィックの配信コストだけでなく、チャネルの品質と遅延にも影響を与えました。インターネットユーザーの数が増加し、新しいオペレーターが登場し、トラフィック量が増加し、インターネットが成長しました。世界中のオペレーターは、オペレーター間の相互作用の組織化により合理的なアプローチが必要であることを認識し始めました。「なぜ、私はオペレーターAが、隣国の通りにあるオペレーターBに交通を届けるために、他の国を通過するために料金を払う必要があるのですか？」そのようなことは、当時の通信事業者から尋ねられました。それで、オペレーターの集中の点で世界の異なる部分で交通交換ポイントが現れ始めました：

  • 1994-ロンドンのLINX、
  • 1995-フランクフルトのDE-CIX、
  • 1995-MSK-IX、モスクワなど

インターネットと今日


概念的には、現代のインターネットのアーキテクチャは、一連の自律システム（自律システム、AS）とそれらの間の多数の接続（物理的および論理的）であり、AS間のトラフィックのパスを決定します。

ASは通常、通信事業者、インターネットプロバイダー、CDN、データセンター、エンタープライズセグメントの企業です。ASは、原則として、BGPプロトコルを使用して、AS間のピアリングを編成します。

自律システムがこれらの関係を編成する方法は、いくつかの要因によって決定されます。

  • 地理的
  • 経済的
  • 政治的
  • AS所有者間の取り決めと共通の利益
  • 等

もちろん、このスキームには特定の構造と階層があります。したがって、事業者は第1層、第2層、および第3層に分けられます。ローカルインターネットサービスプロバイダー（第3層）の顧客が通常は通常のユーザーである場合、たとえば、第1層事業者は他の演算子。ティア3のオペレーターは、サブスクライバーのトラフィック、ティア2のテレコムオペレーター、ティア3のオペレーター、ティア1-すべてのインターネットトラフィックを集約します。

トラフィック交換ポイントとは何ですか？


トラフィック交換ポイントは、相互トラフィック交換に関心のある参加者が相互ピアリングを編成する特別なネットワークインフラストラクチャがある場所です。トラフィック交換のポイントの主な参加者：通信事業者、インターネットプロバイダー、コンテンツプロバイダー、データセンター。トラフィック交換ポイントでは、参加者は互いに直接接続します。これにより、次の問題を解決できます。

  • 遅延を減らす
  • 通過トラフィックの量を減らし、
  • AS間のルーティングを最適化します。

IXPが世界の多くの大都市に存在することを考えると、これはすべてインターネット全体に好影響を及ぼします。

3つのクライアントがIXPと接続してトラフィックを交換したいと考えています。これらはインターネットプロバイダーだとしましょう。ルートサーバーを介してピアリングを整理したいと考えています。以下は、クライアント接続パラメーターの図です。

ISP #1 AS 100 1.1.0.0/16 50.50.50.10/24
ISP #2 AS 200 2.2.0.0/16 50.50.50.20/24

BGP :


router bgp 100
 no bgp enforce-first-as
 bgp log-neighbor-changes
 neighbor 50.50.50.254 remote-as 555
address-family ipv4
  network 1.1.0.0 mask 255.255.0.0
  neighbor 50.50.50.254 activate
  neighbor 50.50.50.254 send-community both
  neighbor 50.50.50.254 soft-reconfiguration inbound
  neighbor 50.50.50.254 route-map ixp-out out
 exit-address-family

ip prefix-list as100-prefixes seq 5 permit 1.1.0.0/16
route-map bgp-out permit 10
 match ip address prefix-list as100-prefixes
 set community 555:555

ここでの設定はbgp enforce-first-asではありません。デフォルトでは、BGPは、この更新を受信したピアのas bgp番号が、受信したBGP更新のas-pathに存在することを要求します。ただし、ルートサーバーはas-pathに変更を加えないため、その番号はas-pathに存在せず、更新は破棄されます。この設定は、ルーターにこのルールを無視させるために使用されます。

また、クライアントがこのプレフィックスにbgpコミュニティ555：555をインストールしたこともわかります。これは、ポリシーに従って、クライアントがこのプレフィックスを他のすべての参加者に通知したいことを意味します。

他のクライアントのルーターについては、固有の設定を除いて、設定は同じです。

BIRDの構成例：


define ixp_as = 555;
define ixp_prefixes = [ 50.50.50.0/24+ ];

template bgp RS_CLIENT {

以下は、martiansプレフィックスとIXP自体のプレフィックスを受け入れないフィルターについて説明しています。

function catch_martians_and_ixp()
prefix set martians;
prefix set ixp_prefixes;
{
  martians = [ 
  0.0.0.0/8+,
  10.0.0.0/8+,
  100.64.0.0/10+,
  127.0.0.0/8+,
  169.254.0.0/16+,
  172.16.0.0/12+,
  192.0.0.0/24+,
  192.0.2.0/24+,
  192.168.0.0/16+,
  198.18.0.0/15+,
  198.51.100.0/24+,
  203.0.113.0/24+,
  224.0.0.0/4+,
  240.0.0.0/4+ ];

  if net ~ martians || net ~ ixp_prefixes then return false;

  return true;
}

この関数は、前に説明したルーティングポリシーを実装します。

function bgp_ixp_policy(int peer_as)
{
  if (ixp_as, ixp_as) ~ bgp_community then return true;
  if (ixp_as, peer_as) ~ bgp_community then return true;

  return false;
}

filter reject_martians_and_ixp
{
  if catch_martians_and_ixp() then reject;
  if ( net ~ [0.0.0.0/0{25,32} ] ) then {
    reject;
  }
  accept;


}

ピアリングを設定し、適切なフィルターとポリシーを適用します。

protocol as_100 from RS_CLIENT {
  neighbor 50.50.50.10 as 100;
  ipv4 {
    export where bgp_ixp_policy(100);
    import filter reject_martians_and_ixp;
  }
}

protocol as_200 from RS_CLIENT {
  neighbor 50.50.50.20 as 200;
  ipv4 {
    export where bgp_ixp_policy(200);
    import filter reject_martians_and_ixp;
  }
}

protocol as_300 from RS_CLIENT {
  neighbor 50.50.50.30 as 300;
  ipv4 {
    export where bgp_ixp_policy(300);
    import filter reject_martians_and_ixp;
  }
}

結論


トラフィック交換ポイントは、通信事業者間の最適でないトラフィックフローの問題に対処するツールとしてインターネットの黎明期に発生しました。新しいグローバルサービスの登場とCDNトラフィックの増加に伴い、エクスチェンジポイントもグローバルネットワークの運用を最適化し続けています。世界のIXP数の増加は、サービスのエンドユーザーにとっても、通信事業者やコンテンツオペレーターなどにとっても有益です。IXP参加者にとっての利点は、外部のピアツーピアサービスの整理にかかるコストの削減、より多くのオペレーターにかかるトラフィック量の削減、ルーティングの最適化、およびコンテンツオペレーターとの直接的なインターフェースの機能に表れています。
