React Native architecture didesain ulang pada tahun 2020

Kerangka kerja React Native (RN) muncul pada tahun 2015. Ini dirancang untuk mengembangkan aplikasi mobile lintas platform menggunakan React library. Aplikasi ini mendukung kapabilitas asli platform tempat platform tersebut dibuat. Arsitektur Asli Bereaksi asli memiliki kekurangan tertentu. Namun terlepas dari ini, RN menerima dukungan masyarakat yang baik, popularitasnya berangsur-angsur tumbuh, tidak terkecuali - berkat reputasi tinggi React.



Proyek arsitektur React Native dibicarakan pada tahun 2018. Pekerjaan ini dilakukan oleh tim Facebook. Tujuan mendesain ulang adalah untuk membuat kerangka kerja lebih stabil, dan untuk memecahkan masalah paling umum yang telah terakumulasi dalam RN selama bertahun-tahun pengembangan. Bahan, terjemahan yang kami terbitkan hari ini, dikhususkan untuk mempertimbangkan bagaimana pemrosesan arsitektur RN dapat meningkatkan kinerja aplikasi dan kecepatan programmer.

Arsitektur lama


React Native adalah contoh solusi yang dapat disebut platform independent. Hasilnya, ternyata tujuan utama kerangka kerja ini adalah memungkinkan pengembang untuk menulis kode JavaScript menggunakan mekanisme Bereaksi. Tujuan dari React Native adalah untuk memungkinkan transformasi pohon komponen React menjadi sesuatu yang akan bekerja pada berbagai platform seluler. Ini berarti yang berikut:

  • Formasi antarmuka yang benar.
  • Memberikan akses ke kemampuan asli berbagai platform.

Biasanya, dalam ekosistem Android / iOS, mekanisme yang digunakan untuk menyelesaikan tugas-tugas ini terlihat seperti gambar berikut.

Untuk mengatur interaksi antara JS dan Native threads, modul Bridge digunakan. Modul ini ditulis dalam C ++ dan didasarkan pada antrian asinkron. Ketika sebuah jembatan menerima data dari salah satu pihak, ia membuat serialisasi data ini, mengubahnya menjadi bentuk string, dan meneruskannya melalui antrian. Ketika data tiba di tujuannya, itu deserialized.

Ini berarti bahwa semua utas mengandalkan pesan JSON asinkron yang dikirim melalui jembatan. Setiap pihak mengirim pesan-pesan ini, mengharapkan (tetapi tidak memiliki jaminan) bahwa suatu saat nanti, pesan-pesan ini akan dijawab. Dengan skema pertukaran data ini, ada risiko kelebihan saluran komunikasi.

Berikut adalah contoh yang biasanya diberikan untuk menggambarkan bagaimana skema pertukaran data dapat menyebabkan masalah dengan kinerja aplikasi. Misalkan pengguna aplikasi menggulir daftar besar. Ketika peristiwa onScroll terjadi di lingkungan asli, informasi secara serempak diteruskan ke lingkungan JavaScript. Tetapi mekanisme asli tidak menunggu sampai bagian JavaScript aplikasi melakukan tugasnya dan melaporkan kepada mereka tentang hal itu. Karena itu, ada penundaan, yang dinyatakan dalam penampilan dalam daftar, sebelum menampilkan isinya, ruang kosong.

Demikian pula, sebelum hasil perhitungan ulang tata letak mencapai layar, sistem harus melakukan beberapa sesi pertukaran data. Jadi, sebelum data tata letak masuk ke lingkungan asli, mereka perlu diproses dalam Yoga. Dan ini, tentu saja, berarti bahwa data seperti itu juga harus dikirim melalui jembatan.

Seperti yang Anda lihat, mengirim data JSON menggunakan mekanisme asinkron, serta membuat serial dan deserialisasi data ini, menyebabkan masalah kinerja. Mekanisme ini tidak sempurna. Tapi bagaimana lagi Anda bisa berinteraksi antara mekanisme asli dan JavaScript? Anda dapat menjawab pertanyaan ini menggunakan teknologi JSI.

Arsitektur baru


Selama mendesain ulang arsitektur React Native, fungsi jembatan dihapus, digantikan oleh mekanisme baru yang disebut JavaScript Interface (JSI).

Menggunakan JSI membuka beberapa peningkatan besar.

Peningkatan seperti pertama adalah bahwa bundel JS tidak lagi bergantung pada JSC. Dengan kata lain, mesin JSC sekarang dapat dengan mudah diganti dengan sesuatu yang lain, sangat mungkin menampilkan kinerja yang lebih tinggi. Misalnya - ke mesin V8.

Peningkatan kedua adalah yang mendasari arsitektur React Native baru. Itu terletak pada kenyataan bahwa, berkat penggunaan JSI, dalam JavaScript Anda dapat menyimpan tautan ke C ++ - objek (Host Objects) dan memanggil metode dari objek-objek ini. Sebagai hasilnya, ternyata bagian JavaScript dari aplikasi dan mekanisme asli akan tahu lebih banyak tentang satu sama lain daripada sebelumnya.

Dengan kata lain, JSI memungkinkan interaksi penuh antara semua utas. Melalui penggunaan konsep kepemilikan bersama, kode JavaScript dapat menjalankan metode asli langsung dari aliran JS. Tidak perlu menggunakan JSON untuk mentransfer data. Ini menghilangkan masalah khas menggunakan jembatan yang terkait dengan kemungkinan overflow antrian dan transfer data asinkron.


Arsitektur Baru React Native ( sumber )

Selain fakta bahwa skema di atas secara signifikan meningkatkan interaksi antara berbagai utas, arsitektur baru ini juga memungkinkan kontrol langsung modul asli. Ini berarti bahwa modul asli dapat digunakan kemudian, maka mereka diperlukan, dan tidak menginisialisasi semuanya ketika aplikasi dimulai. Ini mengarah pada peningkatan serius dalam kecepatan peluncuran aplikasi.

Mekanisme baru ini dapat membuka jalan bagi pengembangan proyek baru yang mampu melakukan apa yang tidak tersedia untuk aplikasi RN lama. Faktanya adalah bahwa sekarang yang kita miliki adalah kekuatan C ++. Ini berarti bahwa berdasarkan React Native, sekarang mungkin untuk membuat lebih banyak varietas aplikasi daripada sebelumnya.

Ringkasan


Selama bertahun-tahun pembangunan, banyak hal telah terakumulasi dalam infrastruktur Asli Bereaksi yang saat ini dapat dianggap usang dan tidak digunakan. Keinginan pengembang RN untuk menghapus dari itu tidak perlu dan meningkatkan kemampuan untuk mendukungnya, mengarah pada penghapusan beberapa kemampuannya. Sebagai contoh, kita berbicara tentang fakta bahwa modul kernelnya, seperti Webview dan AsyncStorage, secara bertahap dihapus dari kerangka kerja dan dibuat sebagai repositori yang dikembangkan oleh komunitas Asli Bereaksi.

Berkat inti kompak yang baru dan mekanisme interaksi yang mengesankan antara kode JavaScript dan kode asli, desain ulang arsitektur Native React harus mengarah pada fakta bahwa aplikasi RN akan menjadi lebih produktif, dan bahwa mereka akan lebih cepat dan lebih nyaman untuk dikembangkan.

React Native restrukturisasi diharapkan selesai pada Q4 2020. Saya senang dengan perspektif tentang kinerja aplikasi dan kegunaan yang datang dengan desain ulang arsitektur RN.

Jika Anda menggunakan React Native, beri tahu kami tentang kekurangan kerangka ini yang Anda temui.
