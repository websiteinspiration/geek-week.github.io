成長するリバースプロキシ-DNS Aレコードを変更せずにリモート保護とサイト最適化を実現するテクノロジー



過去1か月間、遠距離勤務とトレーニングへの広範な移行により、インターネットリソースの平均負荷は大幅に増加しました（「パンデミックとトラフィック-通信事業者からの見解」の記事を正確に読む方法を参照してください。）オンライン映画とゲーム、オンラインプラットフォームは大きな需要があります。 ..？そのような状況での食品配達の要求に応じたトレーニングサービス、DDoS攻撃によるものを含め、リソースの利用不能によって引き起こされる潜在的な経済的損害、特に高い

資料でプロジェクトを守るために選択するソリューション：

  • DNS Aレコードの変更を伴う従来のリバースプロキシによる保護の制限。これは、プロバイダーが頻繁に言及するものではありません。
  • これらの制限に関連するリスクを回避するためにどのソリューションを選択する必要がありますか？
  • Aレコードを移動および変更せずに大規模プロジェクトを保護する実際のケース。
  • インターネットリソースの保護を整理するための一般的な推奨事項。

だから、まず最初に。成長するプロジェクトをDDoS攻撃から保護することにしました。保護の基礎は、着信トラフィックの分析とフィルタリングです。ただし、トラフィックをクリアするには、まずクリーニングセンターに配送する必要があります。以下では、「保護決定」の下で、トラフィックの配信と浄化のためのテクノロジーの組み合わせを意味します。

ほとんどの場合、最初に出くわすソリューションは、DNS Aレコードの変更を伴うリバースプロキシテクノロジーに基づいています。したがって、まず、テクノロジーの原理、その機能（リバースプロキシに精通している場合は、これらの2つのセクションをスキップしてください）、およびトラフィックの配信に関連する制限（これはスキップしないでください）について検討します。次に、これらの制限を実際のケースの例で回避する方法を示します。最後に、インターネットリソースを保護する方法に関する一般的な推奨事項を示します。

リバースプロキシ-仕組み


ここでは、トラフィックを配信する手段としてリバースプロキシを取り上げます。彼女の作品のスキームは誰にでもよく知られていますが、ここで言及するのは言うまでもありません。



  1. DNS Aレコードの変更-WebサーバーのIPアドレスの代わりに、プロキシサーバーのIPアドレスが表示されます。世界中の変更の分布（DNS伝播）。
  2. 訪問者のOSは、サイトのドメイン名に対応するIPアドレスを要求します（DNS解決）。
  3. DNSサーバーは、プロキシサーバーのIPアドレスを報告することにより、要求に応答します。
  4. ビジターブラウザーはHTTP（s）セッションを開き、リクエストをプロキシサーバーに送信します。プロキシサーバーはターゲットWebサーバーへの接続を再確立し、リクエストをプロキシサーバーに渡します。

リバースプロキシ機能


Aレコードの変更を伴うリバースプロキシを介して機能するソリューションを提供する機会は何ですか？

  • リクエストを監視し、アプリケーションレベルでの攻撃からサイトを保護します。このテクノロジーにより、ターゲットサーバーに到着する各アプリケーションレベルのリクエストを処理できます。
  • ターゲットWebサーバーの負荷を減らし、サイトを高速化します。プロキシサーバーでは、データを圧縮してキャッシュできます。これは、コンテンツ配信ネットワーク（CDN）の作業の基礎です。
  • 複数のターゲットWebサーバー間の柔軟なロードバランシング。このテクノロジーにより、処理要求の負荷を複数のマシン間で分散できます。これにより、システムの復元力とパフォーマンスが向上します。
  • 接続が簡単。保護を有効にするには、DNS Aレコードを変更し、変更が有効になるのを待ちます。移転、新しいハードウェアとソフトウェアは必要ありません。
  • ターゲットWebサーバーの実際のIPアドレスを非表示にします。訪問者はプロキシサーバーのIPアドレスを使用して連絡し、そこから回答を受け取ります。これにより、ターゲットWebリソースの匿名性が保証されます。

逆プロキシの成長


私たちの実践からの実際の例で問題を見てみましょう。昨年、大規模なクライアントから保護サービスの要件の具体的なリストが提供されました。明確な理由で会社名を開示することはできませんが、クライアントのニーズは次のとおりです。

  • アプリケーションレベルの攻撃からサイトを保護します。
  • ターゲットWebサーバーから負荷の一部を取り除き、サイトのロードを高速化するために-クライアントには多くの静的コンテンツがあり、彼はCDNノードでのデータのキャッシュと圧縮に関心があります。
  • IPアドレス/ネットワークへの直接攻撃に対する保護を提供します（L3-L4 OSIレベルでのDDoS攻撃に対する保護）。 
  • サービスは、リソースの外部IPアドレスを変更せずに接続する必要があります。クライアントには独自のASおよびアドレスブロックがあります。
  • トラフィック処理サービスの管理とバックアップチャネルへの切り替えは、リアルタイムで行う必要があります。リソースの可用性のレベルは、クライアントにとって重要です。

DNS Aレコードを変更する「クラシック」リバースプロキシに基づくソリューションでは、リストの最初の2つの項目を閉じることができます。

安全なホスティング、仮想サーバー、専用サーバーなどのサービスを使用すると、L3-L7 OSIレベルの攻撃から身を守ることができますが、再配置が必要であり、単一のセキュリティプロバイダーを使用する必要があります。何をすべきか？

保護されたサービスによるネットワーク内のDDoS保護


ネットワークにフィルタリング機器を設置すると、L3-L7 OSIレベルでサービスを保護し、フィルタリングルールを自由に管理できます。このソリューションを選択すると、多額の資本（CaPex）と運用費用（OpEx）が発生します。これらは次の費用です。

  • トラフィックフィルタリング機器+ソフトウェアライセンス（CapEx）;
  • ソフトウェアライセンスの更新（OpEx）。
  • 機器のセットアップとその動作の監視（OpEx）の専任スペシャリスト。 
  • 攻撃を受けるのに十分なインターネットアクセスチャネル（CapEx + OpEx）;
  • 着信「ジャンク」トラフィック（OpEx）の支払い。

その結果、未処理のトラフィックの1メガビットあたりの実効価格は不当に高くなります。いずれにしても、そのようなソリューションのトラフィックをフィルタリングする機能は、専門のプロバイダーの機能よりも低くなります。さらに、サイトの速度を上げるためには、何らかの方法でCDNプロバイダーのサービスに頼らなければなりません。ソリューションの利点のうち、復号化されたトラフィックが保護されたネットワークの境界を離れることはありません。この問題については、後で詳しく説明します。

大企業であっても、そのようなソリューションは、中小企業は言うまでもなく、しばしば経済的に実現可能ではありません。また、クライアントに適していませんでした。

Aレコードを変更しないプロキシ。


そのようなお客様のニーズを満たすために、Webトラフィックを傍受する技術を開発し、Aレコードを変更せずにリモート保護サービスの一部として実装しました。ソリューションは原則に基づいています。クライアントのASとパブリックネットワーク間のすべての接続を保護する必要があります。クライアントは、BGPを介してIPアドレス/ネットワークのアナウンスを送信し、インターネット上でアナウンスします。



保護されたリソースを宛先とするすべてのトラフィックは、ネットワークを通過します。クライアントは、DDoS-GUARDネットワークからアナウンスを削除することにより、いくつかのバックアップ接続を残して、予期しない状況の場合にそれらを使用できます。通常モードでは、インターネットへのアクセスには接続のみを使用することをお勧めします。これにより、クライアントサービスの保護を保証できます。

結論


次に、開発されたDDoS-GUARDソリューションが「リバースプロキシの拡張」セクションの要件のリストを満たしているかどうかを確認します。

  • クライアントは、L3-L7 OSIレベルでの攻撃に対する保護を受け取ります。
  • コンテンツはCDNのノードに圧縮されてキャッシュされるため、ターゲットWebサーバーの負荷が軽減されます。
  • 保護管理はリアルタイムで行われます。クライアントは、個人アカウントまたはAPIを使用して、サブネット、IPアドレス、および個々のドメインのトラフィックフィルタリングルールを管理します。変更は1分以内に有効になります。緊急時には、クライアントはBGPアナウンスメントを削除するだけで、すべてのトラフィックをリダイレクトしてDDoS-GUARDネットワークをバイパスできます。
  • 同社が所有するIPアドレスは変更されていません。クライアントは、新しい機器やソフトウェアを購入したり、追加のスタッフを雇ったり、未処理のトラフィックの料金を支払う必要はありません。

さらに、非標準ポートで実行されているサービスとアプリケーションを保護することが可能です。
