パフォーマンスは単なるCPUではありません：Python用の独自のプロファイラーを作成する

あなたのPythonプログラムが遅いと仮定すると、これは一部にはプロセッサリソースの不足が原因であることがわかります。コードのどの部分がCPUに適用されないものを期待するように強制されているかを知るにはどうすればよいですか？ 今日公開している翻訳を読んだ後、Pythonコード用の独自のプロファイラーを作成する方法を学びます。特定のリソースの解放を待つ間、非アクティブなコード内の場所を検出するツールについて話している。特に、ここでは以下について説明します。





  • プログラムは何を期待できますか？
  • CPUリソースではないリソースの使用のプロファイリング。
  • 意図しないコンテキストスイッチのプロファイリング。

プログラムは何を期待していますか？


プログラムがプロセッサを使用した集中的な計算で忙しくない瞬間には、何かを待っているようです。これは、プログラムの無反応を引き起こす可能性があるものです。

  • ネットワークリソース。これには、DNSルックアップの完了の待機、ネットワークリソースからの応答の待機、一部のデータの読み込み完了の待機などが含まれます。
  • HDD。ハードドライブからのデータの読み取りには時間がかかる場合があります。ディスクへの書き込みについても同じことが言えます。読み取りまたは書き込み操作は、RAMにあるキャッシュを使用してのみ実行される場合があります。このアプローチでは、すべてがかなり迅速に行われます。ただし、プログラムがディスクと直接対話する場合、そのような操作はかなり遅いことが判明することがあります。
  • ロック。プログラムは、スレッドまたはプロセスのロック解除を待機する場合があります。
  • 作業の中断。場合によっては、プログラムが意図的に作業を一時停止することがあります。

パフォーマンスに悪影響を与える何かが発生したプログラムの場所を見つけるにはどうすればよいですか？

方法番号1：プログラムがプロセッサを使用しない時間の分析


組み込みのPythonプロファイラーcProfileは、プログラムの作業に関連するさまざまなインジケーターに関するデータを収集できます。これにより、プログラムがプロセッサリソースを使用しない時間を分析できるツールを作成するために使用できます。

オペレーティングシステムは、プログラムが使用したプロセッサ時間を教えてくれます。

シングルスレッドのプログラムをプロファイリングしていると想像してください。マルチスレッド化されたプログラムはプロファイリングが難しく、このプロセスを説明することも簡単ではありません。プログラムが9秒間実行され、プロセッサを7.5秒間使用した場合、スタンバイで1.5秒を費やしたことを意味します。

最初に、タイムアウトを測定するタイマーを作成します。

import os

def not_cpu_time():
    times = os.times()
    return times.elapsed - (times.system + times.user)

次に、今回分析するプロファイラーを作成します。

import cProfile, pstats

def profile_not_cpu_time(f, *args, **kwargs):
    prof = cProfile.Profile(not_cpu_time)
    prof.runcall(f, *args, **kwargs)
    result = pstats.Stats(prof)
    result.sort_stats("time")
    result.print_stats()

その後、さまざまな機能をプロファイルできます。

>>> profile_not_cpu_time(
...     lambda: urlopen("https://pythonspeed.com").read())

ncalls  tottime  percall  filename:lineno(function)
    3    0.050    0.017   _ssl._SSLSocket.read
    1    0.040    0.040   _socket.getaddrinfo
    1    0.020    0.020   _socket.socket.connect
    1    0.010    0.010   _ssl._SSLSocket.do_handshake
  342    0.010    0.000   find.str
  192    0.010    0.000   append.list

結果から、ほとんどの時間はソケットからのデータの読み取りに費やされたと結論付けることができますが、DNSルックアップ（getaddrinfo）、connectおよびTCPハンドシェイク（）とTLS / SSLハンドシェイクの実行には時間がかかりました。

プログラムの処理でプロセッサリソースを使用しない期間を調査するよう注意を払ったので、これはすべて純粋な待機時間、つまりプログラムが計算でビジー状態でない時間であることがわかります。

なぜそこに時間がために記録されるstr.findと、list.append？そのような操作を行う場合、プログラムは待つ必要がないので、説明はもっともらしいようであり、プロセス全体が実行されなかった状況を扱っているとしています。おそらく-他のプロセスの完了を待つか、またはスワップファイルからメモリへのデータのロードが完了するのを待っています。これは、プロセッサ時間の一部ではない、これらの操作の実行にいくらかの時間が費やされたことを示しています。

さらに、小さな否定的な時間の断片があるレポートを見たことがあることに注意したい。これは、経過時間とプロセッサー時間の間に一定の不一致があることを意味しますが、これがより複雑なプログラムの分析に大きな影響を与えるとは思いません。

方法番号2：意図的なコンテキストスイッチの数の分析


プログラムが何かを待機するのにかかる時間を測定する際の問題は、同じプログラムに対して異なる測定セッションを実行すると、プログラムの範囲外にある何かが原因で変化する可能性があることです。 DNSクエリが通常より遅くなることがあります。時々、通常よりゆっくりと、一部のデータが読み込まれることがあります。したがって、プログラムを取り巻く速度とは関係のない、より予測可能なインジケーターを使用すると便利です。

これを行う1つの方法は、待機する必要がある操作がプロセスを完了した回数を計算することです。つまり、待機期間の数を計算することであり、何かを待機するのに費やした時間ではありません。

プロセスは、次の2つの理由でプロセッサリソースの使用を停止できます。

  1. プロセスが即座に終了しない操作を実行するたびに、たとえば、ソケットからデータを読み取ったり、一時停止したりするなど、これはオペレーティングシステムに言うことと同じです。「作業を続けることができるときに起動してください。」これは、いわゆる「意図的なコンテキスト切り替え」です。プロセッサは、データがソケットに表示されるまで、またはプロセスがスタンバイモードを終了するまで、およびその他の同様の場合に、別のプロセスに切り替えることができます。
  2. 「意図しないコンテキスト切り替え」とは、オペレーティングシステムがプロセスを一時的に停止し、別のプロセスがプロセッサリソースを利用できるようになる状況です。

意図的なコンテキストスイッチをプロファイルします。

ライブラリを使用して、意図的なコンテキストスイッチをカウントするプロファイラーを作成しますpsutil。

import psutil

_current_process = psutil.Process()

def profile_voluntary_switches(f, *args, **kwargs):
    prof = cProfile.Profile(
        lambda: _current_process.num_ctx_switches().voluntary)
    prof.runcall(f, *args, **kwargs)
    result = pstats.Stats(prof)
    result.sort_stats("time")
    result.print_stats()

ここで、ネットワークで動作するコードを再度プロファイルしてみましょう。

>>> profile_voluntary_switches(
...     lambda: urlopen("https://pythonspeed.com").read())

ncalls  tottime  percall  filename:lineno(function)
     3    7.000    2.333  _ssl._SSLSocket.read
     1    2.000    2.000  _ssl._SSLSocket.do_handshake
     1    2.000    2.000  _socket.getaddrinfo
     1    1.000    1.000  _ssl._SSLContext.set_default_verify_path
     1    1.000    1.000  _socket.socket.connect

これで、待機時間データの代わりに、発生した意図的なコンテキスト切り替えの数に関する情報を確認できます。

予期しない場所で意図的なコンテキスト切り替えが発生する場合があることに注意してください。これは、メモリページエラーが原因でページファイルのデータが読み込まれているときに発生すると考えられます。

概要


ここで説明するコードプロファイリング手法を使用すると、システムに特定の追加の負荷がかかり、プログラムの速度が大幅に低下します。ただし、ほとんどの場合、プロセッサリソースの使用を分析していないため、結果が大きく歪むことはありません。

一般に、プログラムの作業に関連する測定可能な指標は、プロファイリングに役立ちます。たとえば、次のとおりです。

  • 読み取り（psutil.Process().read_count）および書き込み（psutil.Process().write_count）操作の数。
  • Linuxでは、読み書きされた合計バイト数（psutil。Process().read_chars）。
  • メモリ割り当てインジケーター（そのような分析を実行するには、多少の労力が必要です。これはjemallocを使用して行うことができます）。

このリストの最初の2つの項目の詳細は、psutilのドキュメントに記載されています。
