Comentarios de concesión de memoria en SQL Server 2019

¡Hola! En previsión del inicio del curso "MS SQL Server Developer" , hemos preparado otra traducción interesante para usted.





Si el optimizador no calcula correctamente la cantidad de memoria requerida para ejecutar la consulta, será un desperdicio de memoria que podría ser utilizado por otro proceso, o se derramarán datos en el disco (derrame de disco). Para resolver este problema, Microsoft ha agregado Comentarios de concesión de memoria. En este artículo, Greg Larsen explica cómo funciona esto.


Los comentarios de concesión de memoria en versiones anteriores de SQL Server (anteriores a SQL Server 2019 o 15.x) se implementaron solo para consultas que se ejecutan en modo por lotes. Las consultas en modo por lotes escanean y calculan hasta 900 filas a la vez, en contraste con las consultas en modo de fila, cuando solo se procesa una fila a la vez. En la versión 15.x, los comentarios de concesión de memoria se han ampliado para admitir consultas en modo de cadena.



¿Qué son los comentarios de concesión de memoria? Este es el proceso de ajustar el cálculo de la memoria necesaria para una consulta, teniendo en cuenta la cantidad de memoria utilizada en sus ejecuciones anteriores. Esto significa que si la consulta en caché usó demasiada memoria en la última ejecución, SQL Server reducirá la asignación de memoria en su próxima ejecución. O si SQL Server detecta una consulta que utiliza un disco debido a que la última vez que se le asignó memoria insuficiente, aumentará la memoria para la consulta. El objetivo de la retroalimentación sobre las concesiones de memoria es ajustar los requisitos de memoria cada vez que se ejecuta la consulta hasta que la consulta use la cantidad de memoria correspondiente al número de filas procesadas.



La funcionalidad Memory Feedback Feedback (en los modos de lote y cadena) es parte de la familia de funciones de Intelligent Query Processing (IQP, Intelligent Query Processing ). Por qué la estimación de la memoria requerida puede ser errónea



Hay muchos factores que afectan la cantidad de memoria requerida para una solicitud. Al calcular la memoria total requerida, el optimizador de consultas debe considerar los requisitos de memoria para cada operación en términos de ejecución. Estos son algunos parámetros que el optimizador tiene en cuenta al calcular la memoria necesaria para ejecutar la consulta:



  • La cardinalidad de las tablas involucradas en la consulta.
  • Altavoces seleccionables
  • Tamaño estimado de fila
  • ¿Es necesario ordenar y / o unir datos
  • ¿La consulta se ejecuta en paralelo?


Para calcular correctamente la memoria requerida, SQL Server debe tener en cuenta muchos aspectos de dónde podría necesitarse memoria al ejecutar una consulta. Si se asigna demasiada memoria, SQL Server desperdicia memoria cuando podría ser utilizada por otros procesos. O si no se asigna suficiente memoria, la solicitud fusionará los datos en el disco, lo cual es una operación costosa.



SQL Server no siempre es bueno para calcular la memoria la primera vez. Las estadísticas desactualizadas o inexactas son la razón más común por la que SQL Server calcula la cantidad incorrecta de memoria. Pero incluso si las estadísticas son correctas, SQL Server puede sobreestimar o subestimar la cantidad de memoria requerida por alguna otra razón. Una de estas razones es que SQL Server no evalúa correctamente el Tamaño de fila estimado para las columnas devueltas. En el siguiente ejemplo, veremos cómo el sobredimensionamiento de una fila hace que la consulta reciba una advertencia de concesión excesiva debido a tres columnas NVARCHAR (MAX), que siempre son NULL.



Pero ahora, si SQL Server calcula incorrectamente la memoria requerida la primera vez que se ejecuta la consulta, para las ejecuciones posteriores, se utiliza la retroalimentación de concesión de memoria. Cada vez que se ejecuta la consulta, se analiza la información sobre la memoria asignada para la ejecución anterior de esta consulta. Si en el inicio anterior la asignación de memoria era incorrecta, entonces el Motor de base de datos corrige la asignación de memoria.



Configuración para la primera ejecución



Para demostrar cómo la asignación de memoria se ajusta a la solicitud a lo largo del tiempo, es necesario realizar algunos trabajos preparatorios. El primero es cambiar el nivel de compatibilidad de la base de datos WideWorldImporters a 150. Debe hacer esto porque Memory Grant Feedback (Modo de fila) solo está disponible para bases de datos con nivel de compatibilidad 150. El código en el Listado 2 cambia el nivel de compatibilidad bases de datos en 150.

GO

Cuando la cantidad de memoria requerida no es estable



Hay solicitudes para las cuales cada ejecución requiere una cantidad diferente de memoria. Si esto sucede, SQL Server deja de intentar ajustar el tamaño de la memoria asignada y deshabilita los comentarios para dicha solicitud. Una de las situaciones en que esto sucede es el lanzamiento de un procedimiento almacenado, en el que, en cada ejecución, se pasan valores de parámetros diferentes.



Para reproducir esta situación cuando SQL Server deshabilita los comentarios sobre las concesiones de memoria, ejecutamos un procedimiento almacenado con diferentes parámetros varias veces. Por lo tanto, en cada ejecución, la consulta dentro del procedimiento requerirá una cantidad diferente de memoria debido a la diferente cantidad de registros devueltos, dependiendo del parámetro. El código de procedimiento almacenado se muestra en el Listado 4. 



USE WideWorldImporters;
GO
CREATE OR ALTER PROC TestProc (@KeyValue int)
AS
SELECT * FROM Sales.Orders
WHERE OrderID > 1 and OrderID < @KeyValue
ORDER BY OrderDate


Listado 4. Procedimiento almacenado de prueba El procedimiento



almacenado acepta un parámetro con un nombre @KeyValue



. El código para probar el procedimiento almacenado se muestra en el Listado 5. 



DECLARE @I INT = 1;
DECLARE @TestKeyValue INT;
WHILE @I < 35
BEGIN
	IF @I % 2 = 0
	 EXEC TestProc @KeyValue = 20000;
	ELSE
	 EXEC TestProc @KeyValue = 100;
	SET @I = @I + 1;
END


Listado 5. Código para llamar al procedimiento almacenado



Mirando el código en el Listado 5, ve que el procedimiento almacenado se ejecuta 35 veces. Cada vez que se inicia el procedimiento almacenado, el valor del parámetro @KeyValue



alterna entre 20,000 y 100. Cuando el parámetro se establece en 20,000, la consulta devuelve 19,998 filas, pero si el valor del parámetro es 100, solo se devuelven 98 filas. Como puede adivinar, cada ejecución requerirá una cantidad diferente de memoria, porque el número de líneas es muy diferente.



Ejecute el código en el Listado 5 en el SSMS y vea el plan de ejecución real. Se puede ver que para comienzos del segundo al 32, el valor de IsMemoryGrantFeedbackAdjust fue YesAdjusting. Pero en la ejecución 33, SQL Server dejó de intentar ajustar la memoria para el procedimiento almacenado y apagó los comentarios. Figura 8. La retroalimentación de asignación de memoria se deshabilita durante la ejecución 33.

Esto significa que el Motor de base de datos no pudo usar los comentarios para obtener una asignación estable de memoria para el procedimiento almacenado, y se decidió deshabilitar los comentarios para esta solicitud. Para todas las ejecuciones posteriores del procedimiento almacenado (después del inicio 33), la retroalimentación está deshabilitada.

Desactivar la memoria Otorgar comentarios



Si por alguna razón desea deshabilitar la retroalimentación de concesión de memoria en modo fila (Retroalimentación de concesión de memoria - Modo fila), hay varias formas de hacerlo. El primero es establecer el nivel de compatibilidad para la base de datos en menos de 150. Solo hay un problema con este método: todas las demás funciones que aparecieron en la versión 15.x también se deshabilitarán. O bien, puede deshabilitar los comentarios a nivel de base de datos a través de una opción ROW_MODE_MEMORY_GRANT_FEEDBACK



. 



USE WideWorldImporters;
GO
ALTER DATABASE SCOPED CONFIGURATION SET ROW_MODE_MEMORY_GRANT_FEEDBACK = OFF;
GO

datos Aquí se deshabilita la retroalimentación sobre las concesiones de memoria a nivel de la base de datos. Ahora las solicitudes de WideWorldImporters no utilizarán comentarios de cadena. Pero esta no es la única forma de desactivar los comentarios. También puede deshabilitar los comentarios para una solicitud específica utilizando HINT, como se muestra en el Listado 8. 



USE WideWorldImporters;
GO
SELECT * FROM Sales.Orders
ORDER By  OrderDate
OPTION
(USE HINT ('DISABLE_ROW_MODE_MEMORY_GRANT_FEEDBACK'));

Optimización del cálculo de la memoria necesaria para la solicitud.



La retroalimentación sobre las concesiones de memoria en modo por lotes ha existido durante bastante tiempo. Pero solo en la versión 15.x esta información estuvo disponible para consultas en modo fila. Todo lo que se necesita para habilitar la retroalimentación para las consultas en modo de cadena es establecer el nivel de compatibilidad de la base de datos en 150. Esto, por supuesto, funcionará si usa la versión actual de Azure SQL Database o SQL Server 2019. Cuando la memoria otorga la funcionalidad de retroalimentación está habilitada en modo de cadena, SQL Server usa información de asignación de memoria de ejecuciones anteriores de la consulta en caché para ajustar la asignación de memoria para la ejecución actual. Corregir la memoria requerida con retroalimentación cada vez que la consulta se ejecute tarde o temprano conducirá automáticamente al cálculo de la cantidad correcta de memoria.
