PHPマイクロサービスフレームワーク

こんにちは、私の名前はアレックスです。マイクロサービスを作成するためのPHPフレームワークを紹介します。それは3年前の私の実験から生まれました。それは後にペットプロジェクトになり、その後このフレームワークでいくつかの制作プロジェクトを作成しました。

私がそれを始めたとき、私は次のことを決定することに着手しました：

  • 既存のプロジェクトに簡単に統合できます。
  • 少なくとも機能するものをすばやく作成できます。
  • 最も簡潔で表現力豊かなデザイン。
  • 最新のPHPの機能を賢く使用しました。

では、どこから始めますか？もちろんソースから！githubでそれらを見ることができます。

さて、長い議論に入らないように、すぐに実際の例から始めましょう。
まず、.htaccessが必要です。ここで、いくつかのルールを構成します。

# use mod_rewrite for pretty URL support
RewriteEngine on
RewriteRule ^([a-z0-9A-Z_\/\.\-\@%\ :,]+)/?(.*)$ index.php?r=$1&%{QUERY_STRING} [L]
RewriteRule ^/?(.*)$ index.php?r=index&%{QUERY_STRING} [L]

次に、最初のサービスを作成できます。その中で、GETメソッドを処理し、すべてが正常であることを示すメッセージを返す1つのエンドポイントを作成します。一種の健康診断。
開始するには、フレームワークを接続する必要があります。

require_once ('vendor/service/service.php');

次に、マイクロサービスのクラスを作成します。

class TodoService extends ServiceBase implements ServiceBaseLogicInterface
{ /* class body */ }

ここにあります：

  • ServiceBaseは、最も基本的で最も実用的な機能を備えたサービスの基本クラスです。
  • ServiceBaseLogicInterfaceは、エンドポイントハンドラーを提供する場合にクラスが実装する必要があるインターフェースです。このインターフェイスはクラスに特別な要件を課しませんが。より厳密なタイピングのために作られました。

次に、最初のエンドポイントハンドラを開始します。

public function action_ping()
{
    return ('I am alive!');
}

次に、最初のマイクロサービスを起動します。

Service::start('TodoService');

すべてをまとめると、次のようになります。

/**
 * Service class
*/
class TodoService extends ServiceBase implements ServiceBaseLogicInterface
{
    /**
     * First endpoint
     */
    public function action_ping()
    {
        return ('I am alive!');
    }
}

Service::start('TodoService');

合理的な質問が発生する可能性があります-この機能はどのURLで利用できますか？実際のところ、action_ <name-part>プレフィックスを使用してメソッドを定義することで、それがURLハンドラー<name-part>であることをサービスに明らかにしましたこの例では、localhost / ping
のようになります。メソッド名の下線が-に変わります。それら。 action_hello_worldメソッドは、localhost / hello-world

URL で使用できます。

GUIアプリケーションの場合と同様に、マイクロサービスだけでなく、MVC（またはビジュアルコンポーネントとロジックを分離した別のパターン）を使用すると便利です。粉砕できるものは粉砕するのが一番です。

それら。今回のケースでは、サービスクラスが実用的な機能を実行してサービスを初期化し、目的のハンドラーを起動し、ロジックを別のクラスに配置します。これを行うには、コードを次のように変更します。

class TodoLogic extends ServiceBaseLogic
{
    /**
     * First endpoint
     */
    public function action_ping()
    {
        return ('I am alive!');
    }
}

class TodoService extends ServiceBase
{
}

Service::start('TodoService', 'TodoLogic');

次に、ロジックを持つクラスを取得しました。

class TodoLogic extends ServiceBaseLogic

基本クラスServiceBaseLogicから継承されます（最低限の機能があるため、後で詳しく説明します）。

TodoServiceクラスは、ServiceBaseLogicInterfaceインターフェースの実装を中止しました（実際、廃止されず、ServiceBaseLogicクラスのみが実装しました）。

ロジックを削除すると、TodoServiceクラスは空になり、安全に切り取ることができるため、コードをさらに削減できます。

class TodoLogic extends ServiceBaseLogic
{
    /**
     * First endpoint
     */
    public function action_ping()
    {
        return ('I am alive!');
    }
}

Service::start('ServiceBase', 'TodoLogic');

ここで、ServiceBaseクラスは、サービスの開始ではなく、サービスの開始を担当します。

さらに運転しました。

私のフレームワークを使用する過程で、ある時点で、巨大なサイズのロジックを持つクラスが判明し始めました。一方で私の美意識を苛立たせた一方で、ソナーは憤慨しました、他方では、メソッドを読み取りメソッドと書き込みメソッドに分割する概念（CQRSを参照）は、それを実装する方法が明確ではありませんでした。

したがって、ある時点で、ある属性または別のロジッククラスに従って別の属性に従ってエンドポイントハンドラーをグループ化し、必要に応じて、それらを同じサービス内で実行するか、簡単に異なるサービスに分散させることが可能になりました。

それら。 1つのサービス内ですべてのCRUDロジックを実行できます。そしてそれは2つのサービスに分けることができます：

  • 1つは読み取り方法を提供します。
  • もう1つはデータ変更方法を提供します。

エンティティ作成メソッドとエンティティリスト取得メソッドを例に追加してみましょう。

class TodoSystemLogic extends ServiceBaseLogic
{
    public function action_ping()
    {
        return ('I am alive!');
    }
}

/**
 * Read logic implementation
 */
class TodoReadLogic extends ServiceBaseLogic
{
    public function action_list()
    {
        return ('List!');
    }
}

/**
 * Write logic implementation
 */
class TodoWriteLogic extends ServiceBaseLogic
{
    public function action_create()
    {
        return ('Done!');
    }
}

Service::start('ServiceBase', [
    'TodoSystemLogic',
    'TodoReadLogic',
    'TodoWriteLogic'
]);

変更のみに触れます。

  • クラスTodoSystemLogic（システムメソッド）、TodoReadLogic（読み取りメソッド）、TodoWriteLogic（書き込みメソッド）が登場。
  • サービスを開始するときに、ロジックを持つ1つのクラスではなく、いくつかのクラスを転送します。
