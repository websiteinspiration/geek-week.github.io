i.MX6でのセキュアブート


組み込みシステム向けのプロジェクトを開発する場合、開発者は次の2つの問題を解決する必要があります。

  • 製品のなりすましからファームウェアを保護する方法。
  • ソフトウェアをコピーから保護する方法。

この記事では、i.MX6プロセッサを製品のブートローダーの変更から保護し、ファームウェアのコピープロセスを複雑にする方法について説明します。



前書き


ベアメタルプロジェクト[1]の場合に知的財産を保護することは、主にマイクロコントローラーに存在するハードウェアと外部の複雑なハードウェアを支援します。例は、評判の良いサイト[2]に記載されています。[3]などの追加のソフトウェア手法が使用されますが、それらは未知の保護メカニズムに基づいています。これは、デバイスのコンピューティング能力が低く、動的なファームウェア生成を実装するのに不便なためです。

LinuxなどのオペレーティングシステムでSoCが機能する場合、保護をいくつかのレベルで同時に構成する必要があります。

  • 終了アプリケーションレベル
  • オペレーティングシステムレベル
  • ブートローダーレベル

いずれかのレベルでアクセス権を取得する場合、さらにレベルの高いソフトウェアに影響を与え、レベル間の相互作用に変更を加えるのは簡単です。
最も単純な例：Linuxにインストールされたソフトウェアは、UARTプロトコルを介して（たとえば、/ dev / ttyACM0を介して）マイクロコントローラーと通信します。システムにアクセスする攻撃者は、同じUARTリソースを使用して、ランダムに送信されたパッケージを繰り返します。または、たとえば、ドライバーがカーネルに組み込まれている場合、攻撃者はUARTドライバーコードを別の方法で書き換えてカーネルを再コンパイルします。最終的なアプリケーションは何が起こっているのかわかりません。

セキュアブートとは何ですか。なぜ必要なのですか。


ターゲットシステム上のソフトウェアの最初の保護レベルは、セキュアブートです。以下では、i.MX6ファミリのプロセッサのセキュアブートについて説明します。

i.MX6のセキュアブートは、高保証ブート（HAB）を通じて実装され、次の2つの機能を実行できます。

  • セキュアブート
  • 暗号化されたダウンロード。

この記事では、プロセッサを安全にロードするためのプロセスについて説明します。I.MX6は、ブートローダー暗号化を提供するアルゴリズムも実装しています。ただし、プロセッサーごとに固有の暗号化機能を使用するため、その使用には、製品ごとに固有のブートローダーを生成する必要があります。これはシリーズで不便な手順なので、説明しませんでした。このトピックについて何か言うことがあれば、私は聞いてうれしいです。
セーフブートは、製品にロードしたカーネルを使用して、製品でブートローダーが確実に起動するようにします。

セキュアブートを与えた定義について。セキュアブートでは、製品をコピーする必要はありません。はい、これは手順をより困難にしますが、それほど難しくはありません。ブートローダーはFS全体に署名できないため、OSレベルおよび最終的なアプリケーションのレベルでの保護は提供されません。それはすべて古典的に機能します：非対称暗号化を介して。指でのプロセスについて説明しますが、[4]で詳しく読むことができます。ブートローダーはHABのスクリプトを示し、署名されたメモリ領域が示され、デジタル署名がどこにあるかを示します。公開キーは、ワンタイムプログラマブル（OTP）プロセッサメモリに事前に配置されています。ロード時に、スクリプトは、添付のデジタル署名と有線キーを使用して、指定されたメモリ領域をチェックします。テストが失敗した場合、プロセッサはダウンロードを開始しません。 HAB操作は、OTPメモリ領域にビットを設定することによってもアクティブにする必要があります。

セキュアブートの欠点：

  • HABがアクティブ化されていない場合でも、ファームウェアは起動します。これは、追加なしであることを意味します。ファームウェアが製品のコピーで実行される保護方法。これを防ぐために、ブートローダーは起動時に独立してHABをアクティブにする必要があります。しかし、USBフラッシュドライブを別の製品に挿入し、公開キーをフラッシュするのを忘れると、プロセッサを破棄できます。
  • プロセッサのメモリにアクセスするときに、公開鍵をコピーできます。これで製品をコピーできます。

このタスクは、統合されたSPLまたは個別のSPLとuboot（たとえば、rockoリリース）を使用してubootに適用できます。標準としてどこで使用されているかがわかるようにリリースを書いています。自分でubootをコンパイルすると、おそらくSPLがubootに統合されます。defconfigブートローダーの重要なポイントは、次のオプションを追加することです。

CONFIG_SECURE_BOOT=y
CONFIG_SYS_FSL_HAS_SEC=y 
#define CONFIG_SYS_FSL_SEC_COMPAT    4 /* HAB version */
#define CONFIG_FSL_CAAM
#define CONFIG_CMD_DEKBLOB
#define CONFIG_SYS_FSL_SEC_LE
#define CONFIG_FAT_WRITE

これらのオプションは、ハードウェアモジュールを操作するためのドライバーをアクティブにします。Linuxのmenuconfigと同じです。ブートローダーをコンパイルするときは、詳細モードを使用します（V = 1を削除します。このパラメーターを含むアセンブリはガイドに記載されています。ただし、自分でクリーニングしました）。この場合、エントリポイントのアドレスに関する情報や、デジタル署名の記録に必要なその他の情報がコンパイルログに追加されます。コンパイル後、次の出力ファイルが必要です。

  • u-boot-ivt.img
  • u-boot-ivt.img.log
  • SPL
  • SPL.log

スクリプトの結果によると、SPL_signed、u-boot-ivt.img_signedの2つのファイルを取得します。これらはターゲットシステムのディスクに書き込む必要があります。記録スクリプトの例をターゲットシステムに適用します。sudoなしで、誤って脚を撃たないようにします。

dd if=SPL_signed of=${DEVICE(/dev/sd*)} bs=1K seek=1; sync
dd if=u-boot-ivt.img_signed of=${DEVICE(/dev/sd*)} bs=1K seek=69; sync

空のメディアに書き込んだら、dtbとカーネルが見つからないため、コンソールにクラッシュするカーネルのないブートローダーを取得する必要があります。コンソールでは、実行した内容が実際に機能しているかどうかを確認できます。イベントの存在下で-何かがうまくいかなかった。重要：ブートローダーがHABサポートなしでコンパイルされている場合も、エラーは発生しません。
