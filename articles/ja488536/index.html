Docker-in-DockerをCIまたはテスト環境で使用する前に注意深く検討してください。



Docker-in-Dockerは、コンテナーイメージを構築するためにコンテナー自体で実行される仮想化されたDockerデーモンです。Docker-in-Dockerを作成する主な目的は、Docker自体の開発を支援することでした。多くの人がJenkins CIを実行するためにそれを使用します。最初はこれは正常に見えますが、Jenkins CIコンテナーにDockerをインストールすることで回避できる問題があります。この記事では、これを行う方法について説明します。詳細のない最終的なソリューションに興味がある場合は、「問題の解決」の記事の最後のセクションを読んでください。



Docker-in-Docker：良い


2年以上前、Dockerに–privileged フラグを挿入して、最初のバージョンのdindを作成しました。目標は、コアチームがDockerをより速く開発できるようにすることでした。Docker-in-Docker以前は、一般的な開発サイクルは次のとおりでした。

  • hackity hack;
  • アセンブリ
  • 実行中のDockerデーモンを停止します。
  • 新しいdockerデーモンを起動します。
  • テスト;
  • ループを繰り返します。

美しく、再現可能なアセンブリを（つまり、コンテナ内で）作成したい場合は、さらに複雑になります。

  • hackity hack;
  • 動作しているバージョンのDockerが実行されていることを確認してください。
  • 古いドッカーで新しいドッカーを構築します。
  • dockerデーモンを停止します。
  • 新しいdockerデーモンを起動します。
  • テストする;
  • 新しいDockerデーモンを停止します。
  • 繰り返す。

Docker-in-Dockerの登場により、プロセスが簡素化されました。

  • hackity hack;
  • 組み立て+ 1ステップでの起動。
  • ループを繰り返します。

そんなに良くないですか？



Docker-in-Docker：悪い


ただし、一般的な考えに反して、Docker-in-Dockerはスター、ポニー、ユニコーンで100％構成されているわけではありません。つまり、開発者が知っておく必要のある問題がいくつかあります。

それらの1つは、AppArmorやSELinuxなどのLSM（Linuxセキュリティモジュール）に関係します。コンテナが起動すると、「内部Docker」が「外部Docker」と競合または混乱するセキュリティプロファイルを適用しようとする可能性があります。これは、-privilegedフラグの元の実装を組み合わせるときに解決する必要があった最も難しい問題です。私の変更は機能し、すべてのテストは私のDebianマシンとUbuntuテスト仮想マシンにも合格しましたが、それらはMichael Crosbyマシンでクラッシュして燃焼しました（私が覚えている限り、彼はFedoraを持っていました）。問題の正確な原因は思い出せませんが、マイクはSELINUX = enforce（AppArmorを使用）を扱う賢い人物であり、私の変更ではSELinuxプロファイルが考慮されていなかったために発生した可能性があります。

Docker-in-Docker：怒り


2番目の問題は、Dockerストレージドライバーに関連しています。 Docker-in-Dockerを起動すると、外部Dockerは通常のファイルシステム（EXT4、BTRFSなど）の上で実行され、内部Dockerは書き込みから書き込みシステム（AUFS、BTRFS、デバイスマッパーなど）の上で実行されます。 、外部Dockerを使用するように構成されているものによって異なります）。この場合、機能しない組み合わせが多数あります。たとえば、AUFS上でAUFSを実行することはできません。

BTRFSの上でBTRFSを実行する場合、これは最初に機能するはずですが、サブキーが表示されるとすぐに、親サブボリュームを削除できません。デバイスマッパーモジュールには名前空間がないため、複数のDockerインスタンスが同じマシンでそれを使用する場合、すべてがお互いのイメージとコンテナーバックアップデバイスのイメージを参照（および影響）できます。これは悪いです。

これらの問題の多くを解決する回避策があります。たとえば、内部DockerでAUFSを使用する場合は、/ var / lib / dockerフォルダーを1つに変更するだけで、すべて問題なく動作します。 Dockerは、デバイスマッパーのターゲット名にいくつかの基本的な名前空間を追加したため、同じマシン上で複数のDocker呼び出しが行われた場合、それらは互いに「ステップ」しません。

ただし、GitHubのdindリポジトリにあるこれらの記事からわかるように、この設定は単純ではありません。

Docker-in-Docker：悪化


ビルドキャッシュはどうですか？これもかなり難しい場合があります。 「Docker-in-Dockerを実行する場合、すべてを内部のDockerに戻すのではなく、ホストにあるイメージを使用するにはどうすればよいですか？」

一部の企業家は、ホストからDocker-in-Dockerコンテナーに/ var / lib / dockerをバインドしようとしました。 / var / lib / dockerを複数のコンテナと共有する場合があります。


データを破壊したいですか？これがまさにあなたのデータを損傷するものだからです！

dockerデーモンは、/ var / lib / dockerへの排他的アクセスを持つように明確に設計されています。このフォルダー内のDockerファイルを「タッチ、ポーク、またはタッチ」するものは他にありません。

これはなぜですか？それは、dotCloudの開発から学んだ最も難しい教訓の1つの結果だからです。 dotCloudコンテナーエンジンは、/ var / lib / dotcloudに同時にアクセスするいくつかのプロセスで動作しました。 （適切な編集ではなく）アトミックファイルの置換、アドバイザリロックと必須ロックでコードを「停止」するなどのトリッキーなトリック、およびSQLiteやBDBなどのセキュアシステムでの他の実験は、常に機能するとは限りませんでした。最終的にDockerに変わったコンテナーエンジンを再設計したとき、主な設計上の決定の1つは、すべてのコンテナー操作を単一のデーモンの下で収集して、この同時アクセスの無意味さをすべて取り除くことでした。

誤解しないでください。いくつかのプロセスと最新の並列制御を含む、優れた信頼性の高い高速な処理を行うことはかなり可能です。ただし、Dockerを唯一のプレーヤーとして使用してコードを記述および保守する方が簡単で簡単だと思います。

つまり、複数のDockerインスタンス間で/ var / lib / dockerディレクトリを共有すると、問題が発生します。もちろん、これは特にテストの初期段階ではうまくいくかもしれません。 「聞いて、まぁ、ubuntuをドッカーとして実行できる！」しかし、たとえば2つの異なるインスタンスから同じ画像を引き出すなど、もっと複雑なことを実行してみてください。そうすれば、世界が燃えている様子がわかります。

つまり、CIシステムがアセンブリと再アセンブリを実行する場合、Docker-in-Dockerコンテナを再起動するたびに、核爆弾をキャッシュに落とすリスクがあります。これはまったくクールではありません！

問題の解決策


一歩下がろう。本当にDocker-in-Dockerが必要ですか、それともDockerを実行できるようにしたいですか？つまり、CIシステム自体がコンテナー内にあるときに、CIシステムからコンテナーとイメージをビルドして実行したいですか？

ほとんどの人は後者のオプションが必要だと思います。つまり、JenkinsのようなCIシステムでコンテナーを実行したいのです。そして、これを行う最も簡単な方法は、DockerソケットをCIコンテナーに挿入し、それを-vフラグに関連付けることです。

簡単に言うと、CIコンテナー（Jenkinsなど）を起動するときに、Docker-in-Dockerで何かをハッキングするのではなく、次の行から開始します。

docker run -v /var/run/docker.sock:/var/run/docker.sock ...

これで、このコンテナーはDockerソケットにアクセスできるようになり、コンテナーを起動できるようになります。ただし、「子」コンテナを起動する代わりに、「関連」コンテナを実行します。

公式のDockerイメージ（Dockerバイナリを含む）を使用してこれを試してください。

docker run -v /var/run/docker.sock:/var/run/docker.sock \
           -ti docker

Docker-in-Dockerのように見え、機能しますが、Docker-in-Dockerではありません。このコンテナが追加のコンテナを作成すると、最上位のDockerに作成されます。ネストの副作用は発生せず、ビルドキャッシュは複数の呼び出しで共有されます。

注：この記事の以前のバージョンでは、Dockerバイナリーをホストからコンテナーにバインドすることが推奨されていました。Dockerメカニズムが静的またはほぼ静的なライブラリに拡張されなくなったため、これは信頼性がなくなりました。

したがって、Jenkins CIのDockerを使用する場合、2つのオプションがあります。
基本的なイメージパッケージシステムを使用してDocker CLIをインストールする（つまり、イメージがDebianベースの場合は、.debパッケージを使用する）、Docker APIを使用します。
