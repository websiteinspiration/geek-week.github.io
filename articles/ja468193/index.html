このシリーズの記事では、Java仮想マシンがどのように機能するかについて説明します。今日は、JVMにクラスをロードするメカニズムについて説明します。

Java仮想マシンは、Javaテクノロジエコシステムの中心にあります。これにより、Javaプログラムが「どこでも一度だけ実行する」という原則を実装できるようになります。他の仮想マシンと同様に、JVMは抽象的なコンピューターです。 JVMの主なタスクは、クラスファイルをロードし、そこに含まれるバイトコードを実行することです。

JVMには、クラスローダー、ガベージコレクター（自動メモリー管理）、インタープリター、JITコンパイラー、フロー制御コンポーネントなどのさまざまなコンポーネントが含まれています。この記事では、クラスローダーについて見ていきます。

クラスローダーは、アプリケーションとJava APIの両方のクラスファイルをダウンロードします。プログラムの実行時に本当に必要なJava APIクラスファイルのみが仮想マシンにロードされます。

バイトコードは実行エンジンによって実行されます。



クラスローディングとは何ですか？

翻訳者注-このセクションでは、Java <9の動作について説明します。Java9以降では、以下に説明する小さな変更が加えられています。

Javaはクラスローディング委譲モデルを使用します。基本的な考え方は、各クラスローダーには「親」ローダーがあるということです。ときにクラスがあるの負荷独自にクラスを検索する前に、ローダー「代表者」、その親にクラス検索。

クラスローダー委譲モデルは、相互にロード要求を渡すローダーのグラフです。このグラフのルートは、ブートストラップブートローダーです。クラスローダーは、ロードを委任できる1つの親で作成され、次の場所でクラスを検索します。

  • キャッシュ
  • 親
  • ブートローダー自体

クラスローダーはまず、以前にクラスをロードしたことがあるかどうかを確認します。その場合、前回返されたのと同じクラス（キャッシュに格納されているクラス）が返されます。そうでない場合は、クラスをロードする機能が親に提供されます。これらの2つのステップは、再帰的に詳細に繰り返されます。親がnullを返した場合（またはClassNotFoundExceptionをスローした場合）、ローダーは独自にクラスを検索します。

最初にクラスをロードする権利は常に親ローダーに付与されるため、クラスはルートに最も近いローダーによってロードされます。これにより、ローダーは親または祖先によって個別にロードされたクラスのみを表示できます。子ローダーによって読み込まれたクラスを見ることができません。

Java SEプラットフォームAPIには、これまで2つのクラスローダーが定義されています。

ブートストラップクラスローダー（基本、プライマリローダー） -ブートストラップクラスパスからクラスをロードします。

システムクラスローダー（親ローダー） -新しいクラスローダーの親クラス。原則として、アプリケーションの読み込みと実行に使用されるクラスローダー。

JDK 9+クラスローダー


アプリケーションクラスローダー -通常、クラスパスからアプリケーションクラスをロードするために使用されます。また、ユーティリティを含む、またはユーティリティAPIをエクスポートする一部のJDKモジュールのデフォルトのブートローダーでもあります。 （翻訳者のメモ：たとえばjdk.jconsole、jdk.jshellなど）

プラットフォームクラスローダー -選択した（セキュリティ/権限に基づいて）モジュールとJava SE JDKをロードします。たとえば、java.sqlです。

ブートストラップクラスローダー -コアJava SEおよびJDKモジュールをロードします。

組み込みクラスローダーを表示するには、次のコードを使用できます。

package ru.deft.homework;

import java.sql.Date;

public class BuiltInClassLoadersDemo {

   public static void main(String[] args) {
       BuiltInClassLoadersDemo demoObject = new BuiltInClassLoadersDemo();
       ClassLoader applicationClassLoader = demoObject.getClass().getClassLoader();
       printClassLoaderDetails(applicationClassLoader);

       // java.sql classes are loaded by platform classloader
       java.sql.Date now = new Date(System.currentTimeMillis());
       ClassLoader platformClassLoder = now.getClass().getClassLoader();
       printClassLoaderDetails(platformClassLoder);

       // java.lang classes are loaded by bootstrap classloader
       ClassLoader bootstrapClassLoder = args.getClass().getClassLoader();
       printClassLoaderDetails(bootstrapClassLoder);
   }

   private static void printClassLoaderDetails(ClassLoader classLoader) {
       // bootstrap classloader is represented by null in JVM
       if (classLoader != null) {
           System.out.println("ClassLoader name : " + classLoader.getName());
           System.out.println("ClassLoader class : " + classLoader.getClass().getName());
       } else {
           System.out.println("Bootstrap classloader");
       }
   }
}

私にインストールされたAmazon Corretto 11.0.3でこのコードを実行すると、次の結果が得られます。

ClassLoader name : app
ClassLoader class : jdk.internal.loader.ClassLoaders$AppClassLoader
ClassLoader name : platform
ClassLoader class : jdk.internal.loader.ClassLoaders$PlatformClassLoader
Bootstrap classloader
