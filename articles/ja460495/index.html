コンテナーからパイプライン：OpenShift Container Platform 4のデフォルトがCRI-Oになりました

ナビゲーションのトピックは、Kubernetesとコンテナの操作の説明の類似点を見つけるのに非常に成功しているので、CoreOSとCRI-Oが解決するビジネス上の問題について、例としてリギングブロックの製造に関するBrunelの発明を使用して話してみましょう。。 1803年、マークブルネルは、成長するイギリス海軍のニーズに合わせて10万個の索具ブロックを製造する任務を与えられました。リフティングブロックは、ロープを帆に取り付けるために使用されるリグの一種です。 19世紀初頭までは、これらのブロックは手作業で作られていましたが、ブルネルは製造を自動化でき、機械を使用して標準化されたブロックの製造を開始しました。このプロセスの自動化により、結果としてすべてのブロックがほぼ同じになり、故障時に簡単に交換でき、大量に製造できるようになりました。

今度は、ブルネルが20の異なる船モデル（Kubernetesバージョン）と、まったく異なる海流と風を持つ5つの異なる惑星（クラウドプロバイダー）でこの作業を行う必要があると想像してください。さらに、航行中の惑星に関係なく、すべての船（OpenShiftクラスター）は船長（クラスターの操作を制御するオペレーター）の観点から同じように動作する必要がありました。海洋の類推を続けると、船長は船に使用されているリギングブロック（CRI-O）をまったく気にしません。主なことは、これらのブロックが強くて信頼性があることです。

OpenShift 4は、クラウドプラットフォームとして、非常によく似たビジネス上の課題に直面しています。新しいノードは、クラスターの作成時、いずれかのノードで障害が発生した場合、またはクラスターをスケーリングするときに作成する必要があります。新しいノードを作成して初期化するときは、CRI-Oを含む重要なホストコンポーネントをそれに応じて構成する必要があります。他の生産と同様に、「原材料」は最初に供給する必要があります。船の場合、金属と木材が原材料となります。ただし、OpenShift 4クラスターにコンテナーをデプロイするためのホストを作成する場合は、構成ファイルとAPIサーバーを入力で提供する必要があります。その後、OpenShiftはライフサイクル全体を通じて必要なレベルの自動化を提供し、エンドユーザーに必要な製品サポートを提供し、それによりプラットフォームへの投資を回収します。

OpenShift 4は、クラウドコンピューティング、仮想化プラットフォーム、さらにはベアメタルシステムのすべての主要サプライヤーに対して、プラットフォーム（バージョン4.Xの場合）のライフサイクル全体を通してシステムを便利に更新する機能を提供するように作成されました。これを行うには、交換可能な要素に基づいてノードを作成する必要があります。クラスターでKubernetesの新しいバージョンが必要な場合、CoreOSで対応するCRI-Oバージョンも受け取ります。 CRI-OバージョンはKubernetesに直接関連付けられているため、これらすべてにより、テスト、トラブルシューティング、またはサポートの順列が大幅に簡素化されます。さらに、このアプローチにより、エンドユーザーとRed Hatのコストが削減されます。

これは、Kubernetesクラスターの完全に新しい外観であり、新しい非常に便利で魅力的な機能を計画するための基礎となります。CRI-O（オープンコンテナープロジェクトコンテナーランタイムインターフェイス-オープンコンテナーイニシアチブ、略してCRI-OCI）は、OpenShiftでの作業に必要なノードの大量作成に最も成功した選択肢です。CRI-Oは以前に使用されていたDockerエンジンに置き換わり、OpenShiftユーザーに経済的で安定したシンプルで退屈な機能を提供します。

オープンコンテナの世界


世界は長い間、オープンコンテナに向かって動いています。 Kubernetesであろうと下位レベルであろうと、コンテナ標準の開発はあらゆるレベルで革新のエコシステムにつながります。

それはすべて、2015年6月の Open Containers Initiativeの作成から始まりました。この作業の初期段階で、コンテナーイメージ（イメージ）とランタイムの仕様が作成されました。これにより、ツールがコンテナイメージの単一の標準と単一のフォーマットを使用してそれらを操作できることを保証できるようになりました。配布仕様が後で追加され、ユーザーは簡単に共有できますコンテナ化されたイメージ。

次に、Kubernetesコミュニティは、Container Runtime Interface（CRI）と呼ばれる単一のプラグイン可能なインターフェース標準を開発しました。これにより、Kubernetesユーザーは、Dockerに加えて、コンテナーを操作するためのさまざまなエンジンを接続できました。

Red HatとGoogleのエンジニアは、CRIプロトコルを使用してKubeletからのリクエストを受け入れることができるコンテナーエンジンと、上記のOCI仕様と互換性のあるコンテナーを導入するという市場の需要を見ました。だから、OCIDがありました。しかし、すみません、この資料はCRI-Oに捧げられると言ったからですか？実際には、バージョン1.0のリリースだけでプロジェクトの名前はCRI-Oに変更されました。

そうです、OpenShift 4の登場により、個々のホストに接続してコンテナーエンジンをインストールしたり、ストレージを構成したり、サーバーを検索用に構成したり、ネットワークを構成したりする必要がなくなりました。 OpenShift 4は、オペレーターフレームワークを使用するために完全に再設計されましたエンドユーザーアプリケーションの観点からだけでなく、イメージの展開、システムの構成、更新のインストールなどの基本的なプラットフォームレベルの操作の観点からも。

Kubernetesは常に、ユーザーが目的の状態を定義し、コントローラーを使用して実際の状態を可能な限り特定の状態に近づけることでアプリケーションを制御できるようにしてきました。与えられた状態と実際の状態を使用するこのアプローチは、開発の観点からも、運用の観点からも、大きな機会を開きます。開発者は目的の状態を判断して渡すことができますYAMLまたはJSONファイルの形式でオペレーターに送信すると、オペレーターは運用環境で必要なアプリケーションインスタンスを作成でき、このインスタンスの運用状態は指定されたものに完全に対応します。

OpenShift 4は、プラットフォームでオペレーターを使用して、この新しいパラダイムを（セットと実際の状態の概念を使用して）RHEL CoreOSおよびCRI-Oの管理にもたらします。オペレーティングシステムとコンテナーエンジンの構成とバージョン管理のタスクは、いわゆるMachine Config Operator（MCO）を使用して自動化されます。MCOはクラスター管理者の作業を大幅に簡素化し、インストールの最終段階と、インストール後の後続の操作（2日目の操作）を本質的に自動化します。これにより、OpenShift 4は真のクラウドプラットフォームになります。これについては、後で詳しく説明します。

交換可能な要素の力の実証


前述のように、Machine Config Operatorを使用してOpenShift 4でコンテナーホストとコンテナーエンジンを管理すると、以前のKubernetesプラットフォームでは不可能だった新しいレベルの自動化が実現します。新しい機能を示すために、crio.confファイルに変更を加える方法を示します。用語で混乱しないように、結果に焦点を当ててみてください。

最初に、いわゆるコンテナーランタイム構成-Container Runtime Configを作成します。これを、CRI-Oの構成を表すKubernetesリソースと考えてください。実際には、これはMachineConfigと呼ばれるものの特別なバージョンであり、OpenShiftクラスター内のRHEL CoreOSマシンにデプロイされた構成です。

ContainerRuntimeConfigと呼ばれるこのカスタムリソースは、クラスター管理者がCRI-Oを簡単に構成できるようにするために開発されました。これはかなり強力なツールであり、MachineConfigPoolの設定に応じて特定のノードにのみ適用できます。これを、同じ目的を果たすマシンのグループと考えてください。

/etc/crio/crio.confファイルで変更する最後の2行に注意してください。これらの2行は、crio.confファイルの行と非常によく似ています。

vi ContainerRuntimeConfig.yaml

結論：

apiVersion: machineconfiguration.openshift.io/v1
kind: ContainerRuntimeConfig
metadata:
 name: set-log-and-pid
spec:
 machineConfigPoolSelector:
   matchLabels:
     debug-crio: config-log-and-pid
 containerRuntimeConfig:
   pidsLimit: 2048
   logLevel: debug

次に、このファイルをKubernetesクラスターに送信し、実際に作成されていることを確認します。作業は他のKubernetesリソースと同じ方法で実行されることに注意してください。

oc create -f ContainerRuntimeConfig.yaml
oc get ContainerRuntimeConfig

結論： 

NAME              AGE
set-log-and-pid   22h

ContainerRuntimeConfigを作成したら、MachineConfigPoolsの1つを変更して、クラスター内の特定のマシングループにこの構成を適用することをKubernetesに認識させる必要があります。この場合、マスターノードのMachineConfigPoolを変更します。

oc edit MachineConfigPool/master

結論（明確にするために、要点は残しておきます）： 

...
metadata:
 creationTimestamp: 2019-04-10T23:42:28Z
 generation: 1
 labels:
   debug-crio: config-log-and-pid
   operator.machineconfiguration.openshift.io/required-for-upgrade: ""
...

この時点で、MCOはクラスターの新しいcrio.confファイルの作成を開始します。この場合、完全に完成した構成ファイルは、Kubernetes APIを使用して表示できます。ContainerRuntimeConfigはMachineConfigの特殊なバージョンにすぎないので、MachineConfigsの行を見て結果を確認できます。

oc get MachineConfigs | grep rendered

結論： 

rendered-master-c923f24f01a0e38c77a05acfd631910b                  4.0.22-201904011459-dirty 2.2.0 16h
rendered-master-f722b027a98ac5b8e0b41d71e992f626                  4.0.22-201904011459-dirty 2.2.0 4m
rendered-worker-9777325797fe7e74c3f2dd11d359bc62                  4.0.22-201904011459-dirty 2.2.0 16h

マスターノードの結果の構成ファイルは、元の構成よりも新しいバージョンであることが判明したことに注意してください。それを表示するには、次のコマンドを実行します。ちなみに、これはおそらくKubernetesの歴史の中で最も優れた単一行スクリプトの1つであることに注意してください。

python3 -c "import sys, urllib.parse; print(urllib.parse.unquote(sys.argv[1]))" $(oc get MachineConfig/rendered-master-f722b027a98ac5b8e0b41d71e992f626 -o YAML | grep -B4 crio.conf | grep source | tail -n 1 | cut -d, -f2) | grep pid

結論： 

pids_limit = 2048

次に、構成がすべてのマスターノードに適用されていることを確認します。最初に、クラスター内のノードのリストを取得します。

oc get node | grep master

Output:

ip-10-0-135-153.us-east-2.compute.internal   Ready master 23h v1.12.4+509916ce1

ip-10-0-154-0.us-east-2.compute.internal     Ready master 23h v1.12.4+509916ce1

ip-10-0-166-79.us-east-2.compute.internal    Ready master 23h v1.12.4+509916ce1

次に、インストールされたファイルを確認します。ContainerRuntimeConfigリソースで指定した新しいpidおよびdebugディレクティブでファイルが更新されていることがわかります。エレガンス自体：

oc debug node/ip-10-0-135-153.us-east-2.compute.internal — cat /host/etc/crio/crio.conf | egrep 'debug||pid’

結論： 

...
pids_limit = 2048
...
log_level = "debug"
...

クラスター内のこれらすべての変更は、SSHを開始しなくても行われました。すべての作業は、Kuberentesマスターノードに連絡することによって行われました。つまり、これらの新しいパラメーターはマスターノードでのみ構成されました。同時に、作業ノードは変更されませんでした。これは、互換性のある要素を持つコンテナホストとコンテナエンジンに適用されるプリセット状態と現在の状態を使用するKubernetes手法の利点を示しています。

上記の例は、3つの作業ノードを持つ小さなOpenShift Container Platform 4クラスターまたは3000ノードを持つ巨大な本番クラスターに変更を加える機能を示しています。いずれの場合も、作業量は同じで非常に少量です。ContainerRuntimeConfigファイルを構成し、MachineConfigPoolの1つのラベルを変更するだけです。また、ライフサイクル全体を通じてKubernetesで使用されるOpenShift Container Platform 4.Xプラットフォームの任意のバージョンでこれを行うことができます。

多くの場合、テクノロジー企業は急速に発展しているため、基本コンポーネントに特定のテクノロジーを選択する理由を説明することができません。コンテナーエンジンは、これまでユーザーが直接操作するコンポーネントでした。コンテナーの人気は当然のことながらコンテナーエンジンの出現で始まったため、ユーザーはしばしばコンテナーエンジンに関心を示します。これが、Red HatがCRI-Oを選択したもう1つの理由です。コンテナーは今日のオーケストレーションに焦点を合わせて進化しており、OpenShift 4での作業時にCRI-Oが最高のエクスペリエンスを提供するという結論に達しました。
