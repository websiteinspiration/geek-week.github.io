リスト理解対マップ

こんにちは、ハブル。多くの場合、シーケンスを操作する場合、シーケンスの作成に関する問題が発生します。リスト内包表記の使用に慣れているようで、本では組み込みのマップ機能の必須の使用について叫んでいます。

この記事では、シーケンスを操作するためのこれらのアプローチを検討し、パフォーマンスを比較し、どのアプローチがより良いアプローチであるかを判断します。


リストの理解


リストの包含は、Pythonに組み込まれたPythonリストエンジンです。彼にはたった1つのタスクがあります-リストを作成することです。リストの包含は、反復可能な型からリストを作成し、入力値を変換（フィルタリング）します。

0から9までの数値の2乗のリストを生成するための単純なリストインクルードの例：

squares = [x*x for x in range(10)]

結果：

[0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

地図


mapは言語に組み込まれた関数です。最初のパラメーターとして関数を受け入れ、2番目のパラメーターとして反復可能なオブジェクトを受け入れます。ジェネレーター（Python 3.x）またはリスト（Python 2.x）を返します。私はPython 3を選択します

。0から9までの数値の2乗のリストを生成するmap関数を呼び出す例：

squares = list(map(lambda x: x*x, range(10)))

結果：

[0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

性能比較


関数なしでビルド


実験として、0から9,999,999の範囲の数値の2乗を検討します。

python -m timeit -r 10 "[x*x for x in range(10000000)]"
python -m timeit -r 10 "list(map(lambda x: x*x, range(10000000)))"

結果：

1 loop, best of 10: 833 msec per loop
1 loop, best of 10: 1.22 sec per loop

ご覧のとおり、リスト内包表記法は約32％高速に動作します。分解すると、map関数は「その作業の詳細を隠すように見える」ため、完全な回答を得ることができません。しかし、おそらくこれは、ラムダ関数が常に呼び出されているためであり、その中で二乗計算がすでに行われています。リスト内包の場合、二乗を計算するだけです。

機能で構築


比較のために、数値の2乗も考慮しますが、計算は関数の内部になります。

python -m timeit -r 10 -s "def pow2(x): return x*x" "[pow2(x) for x in range(10000000)]"
python -m timeit -r 10 -s "def pow2(x): return x*x" "list(map(pow2, range(10000000)))"

結果：

1 loop, best of 10: 1.41 sec per loop
1 loop, best of 10: 1.21 sec per loop

今回は状況が逆転した。mapメソッドは14％高速であることがわかりました。この例では、両方の方法が同じ状況にあります。どちらも、正方形を計算する関数を呼び出す必要があります。ただし、マップ関数の内部最適化により、より良い結果を表示できます。

地図は「安全」ですか？


多くの人がマップの使用を勧めるのはなぜですか。実際のところ、マップは実際にはリスト内包よりも安全な場合があります。

例えば：

symbols = ['a', 'b', 'c']
values = [1, 2, 3]
for x in symbols:
  print(x)
  squared = [x*x for x in values] #  . "x"   "for" 
  print(x)

プログラムの出力は次のようになります。

a
3
b
3
c
3

次に、mapを使用して同じコードを書き直します。

symbols = ['a', 'b', 'c']
values = [1, 2, 3]
for x in symbols:
  print(x)
  squared = map(lambda x: x*x, values) #  ,    
  print(x)

結論：

a
a
b
b
c
c

最も注意深いのは、mapを使用する構文から、これがPython 2であることにすでに気づいているかもしれません。実際、2番目のpythonでは、変数の上書きに関して同様の問題がありました。ただし、Python 3では、この問題は修正されており、関連性はなくなりました。

上記の例は同じ結果を示します。これは愚かな間違いであるように思われることもあり、そのような間違いを犯すことは決してありませんが、これは、内部ループを持つコードのブロックを別のブロックから単純に転送したときに発生する可能性があります。このような間違いは、修正に多くの時間と神経を費やす可能性があります。

結論


比較により、それぞれの方法がそれぞれの状況に適していることがわかりました。

  • 一度にすべての計算された値が必要ない場合（またはおそらくまったく必要ない場合）、マップを選択する必要があります。したがって、必要に応じて、大量のメモリを節約しながら、ジェネレーターにデータの一部を要求します（Python3。Python2では、マップがリストを返すため、これは意味がありません）。
  • すべての値を一度に計算する必要があり、関数を使用せずに計算を実行できる場合は、リスト内包の方向で選択する必要があります。実験の結果が示すように、パフォーマンスに大きな利点があります。
