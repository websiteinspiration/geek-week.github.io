ZuriHac：関数型プログラミングの練習

今年の6月、スイスの小さな町ラッパースヴィルで、ZuriHacと呼ばれるイベントが10回開催されました。今回は、初心者から言語の創始者まで500人以上のHaskell愛好家が集まりました。主催者はこのイベントをハッカソンと呼んでいますが、古典的な意味での会議やハッカソンではありません。その形式は、従来のプログラミングとは異なります。ラッキーなチャンスでZuriHacについて学び、それに参加しましたが、今では異常な発見について話すことが私たちの義務であると考えています！




私たちに関しては


この記事は、高等経済大学の応用数学およびコンピューターサイエンスプログラムの3年次の2人の学生（サンクトペテルブルク）によって作成されました：Vasily AlferovおよびElizaveta Vasilenko。私たち2人にとっての関数型プログラミングへの情熱は、大学2年次でのD.N. Moskvinによる一連の講義から始まりました。現在、VasilyはGoogle Summer of Codeプログラムに参加しており、Algaプロジェクトチームの指導の下、Haskell言語での代数グラフの実装に従事しています。エリザベスは、反統一アルゴリズムの実装に専念するコース作業で関数型プログラミングのスキルを適用し、その後型理論で使用しました。

パンドック


実際、Pandocはテキストドキュメントのユニバーサルコンバーターです。たとえば、docxからpdfへ、またはMarkdownからMediaWikiへ。著者のジョン・マクファーレンはカリフォルニア大学バークレー校の哲学教授です。一般に、パンドックは非常に有名であり、パンドックがハスケルで書かれていることを知って驚いた友人もいます。



Pandocがサポートするドキュメント形式のリスト。サイトにもグラフ全体がありますが、この画像は記事に適合しません。

もちろん、Pandocはフォーマットの各ペアに対して直接変換を実装していません。このような広範な変換セットをサポートするために、標準のアーキテクチャソリューションが使用されます。まず、ドキュメント全体が特別な内部中間表現に変換され、次にこの内部表現から異なる形式のドキュメントが生成されます。開発者は、抽象構文ツリーまたは抽象構文ツリーを表す内部表現「AST」を呼び出します。中間表現を非常に簡単に見ることができます。これを行うには、出力フォーマットとして「ネイティブ」を設定するだけです

$ cat example.html
<h1>Hello, World!</h1>

$ pandoc -f html -t native example.html
[Header 1 ("hello-world",[],[]) [Str "Hello,",Space,Str "World!"]]

Haskellで少なくとも少し作業した読者は、この小さな例から、PandocがHaskellで特別に記述されていることをすでに想定できます。このコマンドの出力は、通常Haskellで行われる方法と同様に作成されたPandocの内部構造の文字列表現です。たとえば、標準ライブラリです。

したがって、ここでは、内部表現が再帰構造であり、リストがある各内部ノードであることがわかります。たとえば、最上位レベルには1つの要素のリストがあります。「hello-world」、[]、[]属性を持つ最初のレベルのヘッダーです。このヘッダー内には、文字列「Hello」、スペース、および文字列「World！」のリストがあります。

ご覧のとおり、内部表現はHTMLと大差ありません。これはツリーであり、各内部ノードはその子孫のフォーマットに関するいくつかの情報を報告し、リーフにはドキュメントの実際のコンテンツが含まれます。

特定の実装のレベルまで下がると、ドキュメント全体のデータ型は次のように定義されます。

data Pandoc = Pandoc Meta [Block]

ここで、Blockは正確に上記の内部ピークであり、Metaはタイトル、作成日、作成者などのドキュメントに関するメタ情報です。これはフォーマットごとに異なり、Pandocはフォーマットからフォーマットに転送するときに可能な限りそのような情報を保存しようとします。

ヘッダーやパラ（段落）など、ブロックタイプのほぼすべてのコンストラクターは、属性と、下位レベルの頂点のリストを受け取ります。たとえば、SpaceまたはStrはInlineタイプのデザイナーであり、HTMLタグもその特別なInlineに変換されます。これらのタイプを完全に定義する理由はありませんが、ここで確認できます。

興味深いことに、Pandocタイプはモノイドです。つまり、ある種の空のドキュメントがあり、それらのドキュメントを積み重ねることができます。これは、リーダーを作成するときに便利です。ドキュメントを任意のロジックで部分に分割し、それぞれを個別に解析して、すべてを1つのドキュメントにまとめることができます。この場合、メタ情報はドキュメントのすべての部分から一度に収集されます。

たとえば、LaTeXからHTMLに変換する場合、まずLaTeXReaderと呼ばれる特別なモジュールが入力ドキュメントをASTに変換し、次にHTMLWriterと呼ばれる別のモジュールがASTをHTMLに変換します。このアーキテクチャーのおかげで、2次数の変換を書く必要はありません。新しいフォーマットごとにリーダーとライターを書くだけで、可能な変換のすべてのペアが自動的にサポートされます。

このアーキテクチャにも欠点があり、ソフトウェアアーキテクチャの分野の専門家が長い間予測していたことは明らかです。最も重要なのは、構文ツリーを変更するコストです。変更が深刻な場合は、すべてのリーダーとライターのコードを変更する必要があります。たとえば、Pandoc開発者が直面する課題の1つは、複雑なテーブル形式をサポートすることです。現在、Pandocは見出し、列、値を各セルに持つ最も単純なテーブルでのみ使用できます。 HTMLのcolspan属性が単に無視されるとしましょう。この動作の理由の1つは、すべての形式または少なくとも多くの形式で単一のテーブル表現スキームが欠落していることです。したがって、どの形式のテーブルを内部表現に格納する必要があるのか​​が明確ではありません。ただし、特定のビューを選択した後でも、テーブルの操作をサポートするすべてのリーダーとライターを完全に変更する必要があります。

Haskellは、関数型プログラミングに対する作者の愛情だけでなく、選ばれました。 Haskellは強力なワープロ機能で知られています。 1つの例は、parsecライブラリです。これは、関数型プログラミングの概念（モノイド、モナド、応用ファンクタ、代替ファンクタ）を積極的に使用して、任意のパーサを作成するライブラリです。Parsecの全機能は、単純な命令型プログラミング言語の完全なパーサーを解析するHaskellWikiの例で見ることができます。もちろん、ParsecはPandocでも積極的に使用されています。

簡単に言うと、モナドは、あるものが最初に来て次に別のものが来るときに、逐次解析に使用されます。たとえば、この例では：

whileParser :: Parser Stmt
whileParser = whiteSpace >> statement

最初にスペースを検討し、次にステートメントを検討する必要があります。これには、Parser Stmtタイプもあります。

解析が失敗した場合、代替ファンクタを使用してロールバックします。例えば、

statement :: Parser Stmt
statement = parens statement <|> sequenceOfStmt

括弧内のステートメントを読み込もうとするか、複数のステートメントを順番に読み込もうとする必要があることを意味します。

適用ファンクタは、主にモナドのショートカットとして使用されます。たとえば、tok関数にある種のトークンを読み取らせます（これはLaTeXReaderの実際の関数です）。そのような組み合わせを見てみましょう

const <$> tok <*> tok

彼女は2つのトークンを続けて読み取り、最初のトークンを返します。

Haskellには、これらすべてのクラスに対応する美しい記号演算子があり、プログラミングリーダーはASCIIアートのように見えます。このすばらしいコードに感心してください。

私たちのタスクはLaTeXReaderに関連していました。Vasilyのタスクは、\ mboxおよび\ hboxコマンドをサポートすることでした。これは、LaTeXでパッケージを作成するときに役立ちます。エリザベスは、LaTeXドキュメントでのエピグラフの実行を可能にするエピグラフチームのサポートを担当しました。

ハトレース


UNIXのようなオペレーティングシステムでは、ptraceシステムコールがしばしば実装されます。これは、プログラム環境のデバッグとシミュレーションに役立ち、プログラムが行うシステムコールを追跡できます。たとえば、非常に便利なstraceユーティリティは、ptraceを内部で使用します。

Hatraceは、Haskellでptraceのインターフェースを提供するライブラリーです。実際、ptrace自体は非常に洗練されており、特に関数型言語から直接使用することは非常に困難です。

起動時のHatraceはstraceのように実行され、同様の引数を受け入れます。 straceとの違いは、ptraceよりも単純なインターフェースを提供するライブラリでもあることです。

Hatraceは、Haskell GHCコンパイラーの不愉快なバグをすでに1つ検出しています。間違ったタイミングで強制終了されるため、不正なオブジェクトファイルが生成され、再起動時に再コンパイルされません。システムコールのスクリプティングにより、ランダムな強制終了によってエラーが約2時間で再現されたときに、1回の実行でエラーを確実に再現することが可能になりました。

ライブラリにシステムコールインターフェイスを追加しました。Elizabethがbrkを追加し、Vasilyがmmapを追加しました。作業の結果によると、ライブラリを使用する場合、これらのシステムコールの引数をより簡単かつ正確に使用できます。
