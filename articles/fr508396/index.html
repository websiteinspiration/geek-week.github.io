Transfert de dynamique moléculaire à CUDA. Partie I: Les bases

Le but de cet article est de soulever les questions de la parallélisation du code d'un programme de modélisation numérique utilisant la méthode de la dynamique moléculaire (MD) utilisant la technologie CUDA. Pourquoi est-ce nécessaire, car les progiciels MD existent déjà, y compris ceux qui fonctionnent sur CUDA? Le fait est que je développe mon propre concept de «champ de force non constant», qui n'est pas mis en œuvre dans les programmes de DM existants.



Refaire le code de quelqu'un d'autre pour ces besoins est une tâche plutôt ingrate, j'ai donc décidé de transférer mon code séquentiel déjà écrit et en même temps de partager quelques réflexions. De plus, c'est une réponse au commentaire qui scintille souvent ici sur des articles de CUDA comme celui-ci .



Alors, quelle est la dynamique moléculaire? Sur Habré il y a déjà plusieurs articles sur ce sujet, par exemple ici ou ici . En bref, MD est une méthode qui vous permet de simuler le mouvement de nombreuses particules (y compris les atomes, les ions, les molécules) et de calculer les propriétés collectives du système, en fonction de ce mouvement. Comment ça fonctionne? Supposons pour un ensemble de N particules des coordonnées initiales, des vitesses, des masses et (plus important!) Des lois d'interaction entre elles. Nous changeons les coordonnées en fonction des vitesses. Sur la base des lois d'interaction, nous calculons les forces agissant entre les particules. Puisque nous connaissons la force et la masse, nous connaissons l'accélération. Nous ajustons la vitesse en tenant compte de l'accélération. Et encore une fois, nous passons à changer les coordonnées. Et donc nous répétons mille fois jusqu'à ce que vous soyez fatigué Nous n’obtiendrons pas suffisamment de statistiques.







Dans les applications réelles, la séquence de calculs de la «coordonnée - vitesse - force» est un peu plus compliquée, basée sur l'expansion des fonctions de coordonnées des particules dans une série. Le plus courant est l'algorithme de Velocity Verlet . Le pseudocode du programme MD basé sur l'intégrateur Verlet est le suivant: 



init_md;
int iStep = 0;
while (iStep < nStep)
{
    reset_quantities;
    clear_cell_list;
    verlet_1stage;
    forces_calculation;
    verlet_2stage;
    temp_scale;
    termostat;
    some_output;
    iStep++;
}
final_md;


La boucle de programme principale exécute un nombre de fois spécifié par l'utilisateur ( nStep ). Une itération de cette boucle est appelée un pas de temps, ou simplement un pas de temps. Les fonctions INIT_MD , reset_quantites , some_output et final_md ne seront pas prises en compte, elles lisent les informations d'entrée, les valeurs mises à zéro , sortent les données, allouent et libèrent de la mémoire, etc. Du point de vue de la sortie d'informations, la dynamique moléculaire est idéale pour les calculs sur une carte vidéo: la plupart des calculs n'ont pas besoin d'être déchargés.



Fonctions verlet_1stage , verlet_2stage , temp_scale et termostatse rapportent à l'intégrateur des équations de mouvement et de température. La fonction forces_calculation calcule les forces agissant entre les particules, et la fonction clear_cell_list fait référence à la technique d'accélération des calculs (voir plus loin à ce sujet).



Considérez tous ces groupes de fonctions.



1 Intégration d'équations de mouvement



L'essence de la première étape de l'algorithme de Verlet peut être exprimée par les formules suivantes:







X=X+Vdt+Fmdt2











V=V+Fmdt2.







Seconde:





V=V+Fmdt2,







où X est la coordonnée des particules, V est sa vitesse, F est la force qui agit sur elle, m est sa masse et dt est le pas de temps. Tout cela s'applique à toutes les particules et, naturellement, se produit en 3 dimensions, donc:

La fonction put_periodic fait en sorte qu'une particule volant hors des frontières de la région simulée (boîte) apparaisse de son autre côté (conditions aux limites périodiques): 



__device__ void put_periodic(float3& xyz, float3 vel, float mass, int type, cudaMD *md)
{
   if (xyz.x < 0)
   {
      xyz.x += md->leng.x;

      atomicAdd(&md->specAcBoxNeg[type].x, 1); 
      atomicAdd(&md->negMom.x, mass * (-vel.x)); 
   }
   else
     if (xyz.x >= md->leng.x)
     {
        xyz.x -= md->leng.x;

        atomicAdd(&md->specAcBoxPos[type].x, 1);
        atomicAdd(&md->posMom.x, mass * vel.x);
     }
}


La fonction fonctionne pour une boîte avec la géométrie d'une boîte rectangulaire. Le code est pour mesurer x, le reste des mesures est complètement similaire. Les champs leng et revLeng sont la taille de la boîte dans la dimension correspondante et sa valeur inverse. En général, pour renvoyer une particule en boxe, vous pouvez vous passer d'opérateurs conditionnels: 



xyz.x -= floor(xyz.x * md->revLeng.x) * md->leng.x;


J'ai utilisé des opérateurs conditionnels dans la version série, car je devais compter le nombre de particules passant par la frontière de la boîte et la quantité de mouvement totale de ces particules (la pression peut en être exprimée). Ces compteurs sont également inclus dans le code ci-dessus, peut-être pour CUDA, vous devrez penser à comment vous en passer .

 int N = min(id0 + atPerThread, md->nAt);

 if (threadIdx.x == 0)

  {
    md->vls[i].x += md->rMasshdT[i] * md->frs[i].x;
    md->vls[i].y += md->rMasshdT[i] * md->frs[i].y;
    md->vls[i].z += md->rMasshdT[i] * md->frs[i].z;

    //  mv2
    kinE += (md->vls[i].x * md->vls[i].x + md->vls[i].y  * md->vls[i].y + md->vls[i].z * md->vls[i].z) * md->masses[i];

     atomicAdd(&md->engKin, shKinE);

Le code de structure est complètement similaire à la première partie de l'intégrateur. Nous mettons à jour les vitesses de chaque particule, résumons l'énergie cinétique.



Fonctions « de température » temp_scale et termostat . Les systèmes moléculaires diffèrent des systèmes mécaniques en ce qu'ils ont le concept de température. MD a hérité de la définition de la température de la théorie moléculaire-cinétique , où la température est une valeur proportionnelle à l'énergie cinétique moyenne des particules (il y a des raisons de croire que c'est une erreur). Pour maintenir la température souhaitée en de tels termes, vous pouvez simplement mettre à l'échelle les vitesses des particules, comme le fait la fonction temp_scale: 



__global__ void temp_scale(int atPerBlock, int atPerThread, cudaMD *md)
{
   float k;
   int i;
   int id0 = blockIdx.x * atPerBlock + threadIdx.x * atPerThread;
   int N = min(id0 + atPerThread, md->nAt);

   k = sqrt(md->teKin / md->engKin);
   for (i = 0; i < N; i++)
   {
      md->vls[i].x *= k;
      md->vls[i].y *= k;
      md->vls[i].z *= k;
   }

   if (!blockIdx.x)
    if (!threadIdx.x)
     {
        md->engKin = md->teKin;
     }
}


Le champ teKin contient l'énergie cinétique cible, elle est proportionnelle à la température réglée par l'utilisateur. La fonction multiplie simplement toutes les composantes des vecteurs vitesse par k, puis un flux définit l'énergie cinétique égale à la valeur donnée. Ici, la variable k est définie dans chaque thread, bien qu'il soit possible qu'elle doive être définie dans un (car c'est la même) pour ne pas faire de travail inutile, d'autant plus que la racine carrée est impliquée.Une telle échelle de température primitive est généralement utilisée dans les étapes préliminaires de la simulation (période d'équilibrage) avec une fréquence de plusieurs étapes. Des algorithmes plus sophistiqués de maintien en température (thermostats) simulent les collisions de particules avec un certain réservoir d'une température donnée, le plus populaire est le thermostat Nose-Hoover :

Pour résumer ce groupe de fonctions. Tous sont parfaitement parallélisés, non seulement parce qu'ils effectuent exactement les mêmes actions pour toutes les particules, mais aussi parce qu'ils ne nécessitent pas de références croisées. 2 Calcul des forces et liste des cellules



Dans les systèmes moléculaires, la principale source de forces agissant sur la particule (à l'exception des champs externes déjà mentionnés) sont d'autres particules. Le calcul de ces forces est la partie la plus complexe, longue et ambiguë du DM. L'une des approximations les plus simples est que toutes les interactions dans le système sont la somme des interactions entre chaque paire de particules (approximation par paire). À ces fins, un potentiel de paire est introduit - une dépendance fonctionnelle de l'énergie potentielle d'interaction de deux particules sur la distance entre elles. Les potentiels de couple les plus connus: le potentiel de Lennard-Jones (ou «6-12»), Buckingham et Born-Mayer-Huggins. Par exemple, je donnerai les expressions analytiques des deux premières:





U=4ϵ(σ6r6−σ12r12)





et





U=Aexp⁡(−rρ)−Cr6,







où U est l'énergie potentielle d'une paire de particules, r est la distance entre les particules et toutes les autres lettres sont des paramètres qui dépendent uniquement du type de particules en interaction et sont définis au stade de l'initialisation. Cependant, plus encore que l'énergie, nous nous intéressons aux forces d'interaction entre ces particules. Les forces sont triviales, grâce à la dérivée du potentiel de paire, pour laquelle il est également facile d'obtenir une expression analytique et de coudre dans le programme. Et, puisque la force d'action est égale à la force de réaction, nous obtenons immédiatement les forces agissant sur les deux particules l'une de l'autre, pseudocode: 



U = 0;
for (i = 0; i < N; i++)
	for (j = i + 1; j < N-1; j++)
	{
		dx = x[i] – x[j];
		dy = y[i] – y[j];
		dz = z[i] – z[j];
		r = sqrt(dx*dx + dy*dy + dz*dz);
		U += pair_pot(r, types[i], types[j]);
		F = force(r, types[i], types[j])
		fx[i] = F * dx;
		fx[j] = -F * dx;
		fy[i] = F * dy;
		fy[j] = -F * dy;
		fz[i] = F * dz;
		fz[j] = -F * dz;

	}


Le code est simple: nous trouvons la distance entre les particules, la substituons dans la fonction pour le potentiel de la paire et dans la fonction pour la force (les deux dépendent des types de particules en interaction) et calculons les projections des forces. Le cycle est effectué N (N-1) / 2 fois, où N est le nombre de particules. Pour réduire au moins en quelque sorte le nombre de calculs, nous pouvons profiter du fait que la valeur de ces potentiels diminue rapidement avec la distance. À des distances supérieures à 6-8 angströms, le potentiel ne peut pas être calculé, il sera presque nul. En ajoutant la condition de vérification de la distance (ou plutôt de son carré) nous réduirons le nombre de calculs, mais pas le nombre d'itérations du cycle. Pour les grands systèmes, la plupart des contrôles donneront faux. Les techniques suivantes peuvent réduire le nombre de vérifications en supprimant une partie des paires avec une grande distance interatomique connue.



Il existe deux méthodes principales pour réduire le nombre de recherches de paires de particules: une liste de voisins (aka liste Verlet ) et une liste de cellules(liste des cellules). Dans la première, pour chaque particule, nous nous souvenons de ses voisins qui se trouvent dans le rayon du potentiel. La liste des voisins doit être mise à jour avec une périodicité inconnue, ce qui est l'inconvénient de la méthode. Par conséquent, arrêtons-nous sur la liste des cellules, dont l'essence est de diviser le volume (boîte) simulé en (sous) cellules. Initialement, la taille des cellules a été choisie pour être supérieure ou égale au rayon de coupure. Ensuite, il suffit de trier les paires de particules à l'intérieur de chaque cellule et entre les cellules adjacentes. La taille des cellules peut être différente, l'essentiel est de déterminer quelles paires de cellules vous devez vérifier. En fait, c'est une tentative de décrire la sphère en cubes. Traditionnellement, pour implémenter une liste de cellules en code séquentiel, deux tableaux sont introduits, head et list. La taille de la première est égale au nombre de cellules, la seconde au nombre de particules. Le premier stocke l'indice d'une des particules dans une cellule donnée, les indices de la seconde correspondent aux indices des particules, et le contenu correspond à l'indice de la particule suivante dans la cellule (-1, s'il n'y a plus de particules dans la cellule), l'illustration est donnée ci-dessous.







Le pseudo-code pour parcourir une telle liste à l'intérieur d'une cellule avec l'index cell_index: 



i = head[cell_index];
while (i >= 0)
{
      j = list[i];
      while (j >= 0)
      {
              i-j;
           j = list[j];
      }
i = list[i]; 
}


Essayons maintenant d'implémenter une liste de cellules sur CUDA. Dans l'approximation zéro, nous distribuons simplement une plage de cellules à chaque flux, sinon, laissant un code séquentiel pour énumérer les paires de particules à l'intérieur de la cellule et les paires impliquant des particules de cellules voisines. Pour chaque cellule, un tableau de cellules voisines lstHNeig [iC] [jC] et le nombre de ces voisins, un tableau nHeadNeig [], doivent être définis :

À mon avis, une telle implémentation ne révèle pas pleinement le potentiel d'une carte vidéo, car un seul flux traite une cellule, ce qui, en règle générale, est nettement inférieur au nombre de particules. Deuxièmement, à l'intérieur de la fonction de traitement d'une paire de particules, il n'est possible de travailler que par des opérations atomiques. Essayons de paralléliser la tâche plus fortement, laissez chaque thread traiter une paire de cellules voisines (nous pensons qu'une liste de ces paires a déjà été créée):

{

Le tableau cellPairs [] contient le nombre de paires de cellules voisines (.x est la première cellule de la paire, .y est la seconde). Le code est plus simple et plus élégant, mais ici le traitement de paires de particules à l'intérieur d'une cellule disparaît, pour lequel une fonction distincte doit être écrite. Il y a un autre problème, l'organisation de la liste des cellules et son analyse sont inefficaces pour le calcul parallèle, car pour obtenir l'index de la prochaine particule dans la cellule, vous devez parcourir toutes les précédentes. Essayons de créer un code et une structure de données plus efficaces afin que quelques cellules puissent traiter plusieurs threads à la fois. Maintenant, laissez la liste des cellules être stockée dans une cellule de tableau à deux dimensions [] [] , où le premier index définit le numéro de la cellule, et le second - le numéro atomique dans la cellule.     {

        f = 0.0;

__device__ float cu_fe_lj(float r2, cudaVdW* vdw, float& eng)
// U = 4e[(s/r)^12 - (s/r)^6]
{
    float r2i = 1.0 / r2;
    float sr2 = vdw->p1 * r2i;
    float sr6 = sr2 * sr2 * sr2;

    eng += vdw->p0 * sr6 * (sr6 - 1.0);
    return vdw->p2 * r2i * sr6 * (2.0 * sr6 - 1.0);
}

__device__ float cu_fer_buck(float r2, float& r, cudaVdW* vdw, float& eng)
// U = A exp(-r/ro) - C/r^6
{
    float r2i = 1.0 / r2;
    float r4i = r2i * r2i;
    if (r == 0.0)    //  r , 
        r = sqrt(r2);

    eng += vdw->p0 * exp(-r / vdw->p1) - vdw->p2 * r4i * r2i;
    return vdw->p0 * exp(-r / vdw->p1) / r / vdw->p1 - 6.0 * vdw->p2 * r4i * r4i;
}


Les fonctions renvoient la force (qui doit être multipliée par la projection du vecteur distance) et ajoutent de l'énergie à la variable eng . Un pointeur vers l'une de ces fonctions est stocké dans la structure cudaVdW. D'une manière générale, pour obtenir des trajectoires de mouvement, l'énergie ne peut pas être calculée, elle est nécessaire pour différentes statistiques. Si nous supprimons les expressions d'énergie de ces fonctions, les calculs devraient s'accélérer. Et vous pouvez le faire: une fois quelques étapes pour appeler la fonction complète, et le reste du temps - seulement un analogique réduit. Une partie de l'interaction Coulomb est calculée par la fonction real_ewald : 



__device__ float real_ewald(float r2, float& r, float chprd, float alpha, float& eng)
{
    float ar; 
    float erfcar;
    float kqq = chprd * d_Fcoul_scale; // q[i]*q[j]*1/4pie0;

    r = sqrt(r2);
    ar = alpha * r;
    erfcar = erfc(ar);

    eng += kqq * erfcar / r;
    return kqq / r / r2 * (erfcar + 2 * ar / d_sqrtpi * exp(-ar * ar));
}

__global__ void all_pair(cudaMD* md)
{
    int i, j;
    int step = blockDim.x * gridDim.x;
    int ex = 0;  // exit flag
    int nAt = md->nAt;

    i = 0;
    j = blockIdx.x * blockDim.x + threadIdx.x + 1;
    while (1)
    {
        while (j >= nAt)
        {
            i++;
            if (i >= nAt - 1)
            {
                ex = 1;
                break;
            }
            j = i + 1 + j - nAt;
        }
        if (ex) break;
        global_pair(i, j, iStep, md);
        j = j + step;
    }
} 

__device__ void global_pair(int i, int j, cudaMD* md)
{
    float r, r2, f;
    float engCoul = 0;
    float engVdw = 0;
    cudaVdW* vdw;
    int ti, tj;

    float dx = md->xyz[i].x - md->xyz[j].x;
    float dy = md->xyz[i].y - md->xyz[j].y;
    float dz = md->xyz[i].z - md->xyz[j].z;

    //   
    if (dx > md->halfLeng.x)
        dx -= md->leng.x;
    else
        if (dx < -md->halfLeng.x)
            dx += md->leng.x;

    if (dy > md->halfLeng.y)
        dy -= md->leng.y;
    else
        if (dy < -md->halfLeng.y)
            dy += md->leng.y;

    if (dz > md->halfLeng.z)
        dz -= md->leng.z;
    else
        if (dz < -md->halfLeng.z)
            dz += md->leng.z;

    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)    
    {
        r = 0.0;
        f = 0.0;

        ti = md->types[i];
        tj = md->types[j];

        f += real_ewald(r2, r, md->chProd[ti][tj], md->alpha, engCoul);

        vdw = md->vdws[ti][tj];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw); 

        dx *= f;
        dy *= f;
        dz *= f;

        atomicAdd(&(md->frs[i].x), dx);
        atomicAdd(&(md->frs[j].x), -dx);
        atomicAdd(&(md->frs[i].y), dy);
        atomicAdd(&(md->frs[j].y), -dy);
        atomicAdd(&(md->frs[i].z), dz);
        atomicAdd(&(md->frs[j].z), -dz);

        atomicAdd(&(md->engCoul1), engCoul);
        atomicAdd(&(md->engVdW), engVdw);

    }
}
