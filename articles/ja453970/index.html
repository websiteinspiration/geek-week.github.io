PowerShellでReverse socks5プロキシを作成します。パート2

3つの部分での研究開発の物語。パート2-開発。
ブナはたくさんあります-さらに多くの利点があります。記事の

最初の部分では、リバーストンネルを整理するためのいくつかのツールについて知り、その長所と短所を調べ、Yamuxマルチプレクサーの動作メカニズムを調査し、新しく作成されたpowershellモジュールの基本要件を説明しました。RSocksTunリバーストンネルの既成の実装用のクライアントPowerShellモジュールの開発を開始する時が来ました。

まず、モジュールがどのモードで動作するかを理解する必要があります。明らかに、データのプリマ転送では、Windowsソケットメカニズムと、ソケットへの読み書きのストリーミングに.Net機能を使用する必要があります。しかし、その一方で、私たちのモジュールは複数のyamuxストリームを同時に処理する必要があるため、すべてのI / O操作がプログラムの実行を完全にブロックすることはありません。これは、モジュールがソフトウェアマルチスレッディングを使用し、yamuxサーバーで読み書き操作を実行し、異なるプログラムストリームで宛先サーバーへの読み書き操作を実行する必要があるという結論を示唆しています。もちろん、並列フロー間の相互作用のメカニズムを提供する必要があります。良い、powershellは、プログラムスレッドを開始および管理するための十分な機会を提供します。

仕事の一般的なアルゴリズム


したがって、クライアントの一般的なアルゴリズムは次のようになります。

  • サーバーへのSSL接続を確立します。
  • サーバーが私たちをセキュリティ担当者と区別できるように、パスワードを使用してログインします。
  • yamuxパッケージが新しいストリームをインストールするのを待ち、サーバーのキープアライブ要求に定期的に応答します。
  • yamuxパッケージが到着したらすぐに、新しいsocksScriptプログラムストリーム（ストリームと混同しないでください）を開始して、新しいストリームをインストールします。socksScriptの内部にはsocks5サーバーの機能が実装されています。
  • yamuxからのデータを含むパケットの到着時-12バイトのヘッダーから、データの対象となるストリームとそのサイズを理解し、yamuxサーバーからデータを読み取り、受信したデータを対応するストリーム番号を持つストリームに転送します。
  • 実行中の各socksスクリプトで、yamuxサーバー向けのデータの可用性を定期的に監視します。そのようなデータがある場合は、対応する12バイトのヘッダーを追加して、yamuxサーバーに送信します。
  • ストリームを閉じるyamuxパッケージが到着すると、対応するストリームに信号を送信してストリームを終了し、切断します。その後、ストリーム自体を完了します。

したがって、クライアントでは、少なくとも3つのプログラムフローを実装する必要があります。

  1. 接続を確立し、yamuxサーバーにログインし、そこからデータを受信し、yamuxヘッダーを処理し、生データを他のプログラムストリームに送信するメインのもの。
  2. ソックスサーバーでストリーミング。複数ある場合があります-各ストリームに1つ。彼らはsocks5機能を実装します。これらのフローは、内部ネットワークの宛先ポイントと相互作用します。
  3. 逆流。ソックスストリームからデータを受信し、それらにyamuxヘッダーを追加して、それをyamuxサーバーに送信します。

そしてもちろん、これらすべてのフロー間の相互作用を提供する必要があります。

そのような相互作用を提供するだけでなく、（ソケットと同様に）ストリーミング入出力の利便性を取得する必要もあります。最も適切なメカニズムは、ソフトウェアパイプを使用することです。 Windowsでは、パイプは、各パイプに独自の名前と匿名がある場合に登録されます。各パイプはハンドラーによって識別されます。もちろん、機密保持のために、匿名パイプを使用します。 （結局のところ、システム内の登録されたパイプを使用してモジュールを計算したくないのですよね？）。したがって、メイン/リターンフローとsocks-flowsの間の相互作用は、匿名パイプ（匿名パイプ）を介して行われ、非同期ストリームI / Oをサポートします。メインフローとリターンフローの間では、共有オブジェクトメカニズム（共有同期変数）を介して通信が行われます（詳細については、これらの変数は何であり、それらと一緒に暮らす方法を読むことができますこちら）。

ソックスストリームの実行に関する情報は、対応するデータ構造に格納する必要があります。この構造で靴下スレッドを作成するときは、次のように記述する必要があります。

  • yamuxセッション番号：$ ymxstream;
  • パイプ（チャネル）を操作するための4つの変数：$ cipipe、$ copipe、$ sipipe、$ sopipe。匿名チャネルはINまたはOUTのいずれかで機能するため、ソックスストリームごとに2つの匿名チャネルが必要で、それぞれに2つのエンド（パイプストリーム）（サーバーとクライアント）が必要です。
  • ストリームの呼び出しの結果は$ AsyncJobResultです。
  • ストリームハンドラ-$ Psobj。これにより、ストリームを閉じてリソースを解放します。
  • リバースストリーム（$ readjob）による匿名パイプからの非同期読み取りの結果。この変数は、対応するパイプからの非同期読み取りのために、戻りyamuxScriptストリームで使用されます。
  • 各靴下ストリームのデータを読み取るためのバッファ。

主流


したがって、データ処理の観点から、プログラムの作業は次のように構築されます。

  • サーバー側（rsockstun-Golangに実装）はsslサーバーを起動し、クライアントからの接続を待ちます。
  • クライアントから接続を受信すると、サーバーはパスワードをチェックし、正しい場合はyamux接続を確立し、socksポートを上げて、socksクライアント（プロキシチェーン、ブラウザーなど）からの接続を待ち、定期的にキープアライブパケットを交換しますお客様と一緒に。パスワードが正しくない場合-サーバーのインストール時に指定したページへのリダイレクトが実行されます（これは、情報セキュリティの警戒管理者向けの「法的」ページです）。
  • socksクライアントから接続を受信すると、サーバーはyamuxパケットをクライアントに送信して、新しいストリーム（YMX SYN）を確立します。

Yamuxヘッダーの取得と分析この

モジュールでは、最初にサーバーへのSSL接続を確立し、パスワードを使用してログインします。

$tcpConnection = New-Object System.Net.Sockets.TcpClient($server, $port)
$tcpStream = New-Object System.Net.Security.SslStream($tcpConnection.GetStream(),$false,({$True} -as [Net.Security.RemoteCertificateValidationCallback]))
$tcpStream.AuthenticateAsClient('127.0.0.1')

次に、スクリプトは12バイトのyamuxヘッダーを待ち、それを解析します。
微妙なニュアンスがあります...実践が示すように、ソケットから12バイトを読み取るだけです。

 $num = $tcpStream.Read($tmpbuffer,0,12)

必要なバイトの一部のみが到着した後に読み取り操作を完了することができるため、十分ではありません。したがって、ループ内で12バイトすべてを待機する必要があります。

     do {
            try { $num = $tcpStream.Read($tmpbuffer,0,12) } catch {}
            $tnum += $num
            $ymxbuffer += $tmpbuffer[0..($num-1)]
        }while ($tnum -lt 12 -and $tcpConnection.Connected)

ループが終了したら、$ ymxbuffer変数に含まれる12バイトのヘッダーを分析し、そのタイプと、Yamuxの仕様に従って設定されたフラグを確認します。

Yamuxヘッダーにはいくつかのタイプがあります。

  • ymx syn-新しいストリームをインストールします。
  • ymx fin-ストリーム補完。
  • ymxデータ-データに関する情報（データのサイズとストリーム）を表します。
  • ymx ping-キープアライブメッセージ。
  • ymx win update-データの一部の転送の確認。

上記のタイプのyamuxヘッダーに適合しないものは、例外的な状況と見なされます。そのような例外は10個ありますが、ここでは何か問題があると考えており、モジュールの作業を完了しています。（また、すべてのファイルを消去し、ディスクをワイプし、姓を変更し、新しいパスポートを

作成し、国をリストから削除します...）新しいsocks-streamの作成

新しいストリームを確立するためのyamux-packageを受け取ったら、クライアントは2つの匿名サーバーパイプ（$ sipipe、$ sopipe）、それぞれに基づいて、それらに基づいてクライアントパイプ（$ cipipe、$ copipe）を作成します。

$sipipe = new-object System.IO.Pipes.AnonymousPipeServerStream(1)
$sopipe = new-object System.IO.Pipes.AnonymousPipeServerStream(2,1)
$sipipe_clHandle = $sipipe.GetClientHandleAsString()
$sopipe_clHandle = $sopipe.GetClientHandleAsString()
$cipipe = new-object System.IO.Pipes.AnonymousPipeClientStream(1,$sopipe_clHandle)
$copipe = new-object System.IO.Pipes.AnonymousPipeClientStream(2,$sipipe_clHandle)

socksストリームの実行スペースを作成し、このストリーム（StopFlag）と対話するための共有変数を設定し、別のストリームにsocksサーバーの機能を実装するスクリプトブロックSocksScriptを実行します。

$state = [PSCustomObject]@{"StreamID"=$ymxstream;"inputStream"=$cipipe;"outputStream"=$copipe}
$PS = [PowerShell]::Create()

$socksrunspace = [runspacefactory]::CreateRunspace()
$socksrunspace.Open()
$socksrunspace.SessionStateProxy.SetVariable("StopFlag",$StopFlag)
$PS.Runspace = $socksrunspace
$PS.AddScript($socksScript).AddArgument($state) | Out-Null
[System.IAsyncResult]$AsyncJobResult = $null
$StopFlag[$ymxstream] = 0
$AsyncJobResult = $PS.BeginInvoke()

作成された変数は特別なArrayList構造に書き込まれます-Pythonのディクショナリの類似物

[System.Collections.ArrayList]$streams = @{}

追加は、組み込みのAddメソッドを通じて行われます。

$streams.add(@{ymxId=$ymxstream;cinputStream=$cipipe;sinputStream=$sipipe;coutputStream=$copipe;soutputStream=$sopipe;asyncobj=$AsyncJobResult;psobj=$PS;readjob=$null;readbuffer=$readbuffer}) | out-null

サーバーからsocksストリームを宛先とするデータを受信したら、yamuxストリームの数（このデータが対象とするsocksストリームの数）と12バイトのyamuxヘッダーからのバイト数を決定する必要があります。データ：

次に、ArrayListストリームから、ymxIdフィールドを使用して、このsocksストリームに対応するサーバーアウトパイプのハンドラーを取得します。

 if ($streams.Count -gt 1){$streamind = $streams.ymxId.IndexOf($ymxstream)}
        else {$streamind = 0}
 $outStream = $streams[$streamind].soutputStream

その後、ソケットからデータを読み取ります。ループを特定のバイト数読み取る必要があることを思い出してください。

            $databuffer = $null

受信したデータを対応するパイプに書き込みます。

を通知するパケットをyamixサーバーから受信すると、最初に12バイトのヘッダーからyamuxストリーム番号も取得します。

 $ymxstream = [bitconverter]::ToInt32($buffer[7..4],0)

次に、共有変数（またはフラグの配列。インデックスはyamuxストリーム番号です）を使用して、socksストリームに完了を通知します。

if ($streams.Count -gt 1){$streamind = $streams.ymxId.IndexOf($ymxstream)}
        else {$streamind = 0}
 
if ($StopFlag[$ymxstream] -eq 0){
            write-host "stopflag is 0. Setting to 1"
            $StopFlag[$ymxstream] = 1
        }

フラグを設定した後、socksストリームを強制終了する前に、socksストリームがこのフラグを処理するまで一定の時間待機する必要があります。これには200ミリ秒で十分です。

start-sleep -milliseconds 200 #wait for thread check flag

次に、このストリームに関連するすべてのパイプを閉じ、対応するRunspaceを閉じ、Powershellオブジェクトを強制終了してリソースを解放します。

$streams[$streamind].cinputStream.close()
$streams[$streamind].coutputStream.close()
$streams[$streamind].sinputStream.close()
$streams[$streamind].soutputStream.close()
    
$streams[$streamind].psobj.Runspace.close()
$streams[$streamind].psobj.Dispose()
$streams[$streamind].readbuffer.clear()

socksストリームを閉じた後、対応する要素をArrayListストリームから削除する必要があります。

$streams.RemoveAt($streamind)

最後に、.Netガベージコレクターに、スレッドで使用されているリソースを強制的に解放させる必要があります。それ以外の場合、スクリプトは約100〜200 MBのメモリを消費し、経験豊富で腐食性のあるユーザーの目を引く可能性がありますが、これは必要ありません。

[System.GC]::Collect()#clear garbage to minimize memory usage

 if ($stream.readjob -eq $null){

Yamuxウィンドウの更新


さらに、yamuxScriptは、yamuxサーバーから受信したバイト数を監視し、定期的にYMX WinUpdateメッセージを送信する必要があります。 Yamuxのこのメカニズムは、いわゆるウィンドウサイズ（TCPプロトコルと同様）を監視および変更する責任があります。これは、確認応答なしで送信できるデータバイト数です。デフォルトでは、ウィンドウサイズは256Kバイトです。つまり、このサイズより大きいファイルまたはデータを送受信するときは、windpw更新パッケージをyamuxサーバーに送信する必要があります。 yamuxサーバーから受信したデータの量を制御するために、特別な共有配列$ RcvBytesが導入されました。メインのストリームには、現在の値をインクリメントして、サーバーから受信した各ストリームのバイト数を記録します。設定したしきい値を超えると、yamuxScriptはパケットをWinUpdateサーバーに送信し、カウンターをリセットする必要があります。

            if ($RcvBytes[$stream.ymxId] -ge 256144){
                #out win update ymx packet with 256K size
                $outbuf = [byte[]](0x00,0x01,0x00,0x00)+ [bitconverter]::getbytes([int32]$stream.ymxId)[3..0]+ (0x00,0x04,0x00,0x00)
                $state.tcpstream.Write($outbuf,0,12)
                $RcvBytes[$stream.ymxId] = 0
            }

SocksScriptストリーム


次に、socksScript自体に直接移動します。
socksScriptは非同期で呼び出されることを思い出してください。

$state = [PSCustomObject]@{"StreamID"=$ymxstream;"inputStream"=$cipipe;"outputStream"=$copipe}
$PS = [PowerShell]::Create()
....
$AsyncJobResult = $PS.BeginInvoke()

呼び出し時には、ストリームに渡された変数$の状態に次のデータが存在します。

  • $ state.streamId-yamuxセッションの数。
  • $ state.inputStream-パイプを読み取ります。
  • $ state.oututStream-パイプを書き込みます。

パイプ内のデータは、yamuxヘッダーのない生の形式で提供されます。彼らは靴下クライアントから来た形で。

socksScriptの内部では、まず、socksのバージョンを判別し、それが5であることを確認する必要があります。

$state.inputStream.Read($buffer,0,2) | Out-Null
        $socksVer=$buffer[0]
        if ($socksVer -eq 5){ ... }

それでは、Invoke-SocksProxyスクリプトに実装されているとおりに行います。唯一の違いは、呼び出しの代わりに

$AsyncJobResult.AsyncWaitHandle.WaitOne();
$AsyncJobResult2.AsyncWaitHandle.WaitOne();

サイクリックモードで$ StopFlag配列のtcp接続と対応する終了フラグを監視する必要があります。そうしないと、socksクライアントとymuxサーバーの側から接続終了の状況を認識できません。

while ($StopFlag[$state.StreamID] -eq 0 -and $tmpServ.Connected ){
      start-sleep -Milliseconds 50
 }

接続先のtcpサーバーによって接続が終了した場合は、このフラグを2に設定します。これにより、yamuxscriptがこれを認識し、対応するymx FINパケットをyamuxサーバーに送信します。

if ($tmpServ.Connected){
       $tmpServ.close()
 }else{
       $StopFlag[$state.StreamID] = 2
 }

socksScriptが宛先サーバーに接続できない場合も、このフラグを設定する必要があります。

if($tmpServ.Connected){ ... }
 else{
     $buffer[1]=4
     $state.outputStream.Write($buffer,0,2)
     $StopFlag[$state.StreamID] = 2
 }
