証明書が腐っていて、クラスターがカボチャに変わったらどうしますか？

コマンドに応答して次の結果kubectl get podが得られた場合：

Unable to connect to the server: x509: certificate has expired or is not yet valid

その後、おそらく1年が経過し、kubernetes証明書が期限切れになり、クラスターコンポーネントがそれらの使用を停止し、それらの間の相互作用が停止し、クラスターがカボチャに変わりました。



クラスターを復元するにはどうすればよいですか？

まず、更新が必要な証明書がどこにあるかを理解する必要があります。

クラスターのインストール方法によっては、証明書ファイルの場所と名前が異なる場合があります。したがって、たとえば、クラスターを作成するとき、Kubebemはベストプラクティスに従って証明書ファイルを分解します。したがって、すべての証明書はディレクトリ/etc/kuberenetes/pkiにあり、拡張子が付い.crtたファイルにあります.key。秘密鍵はそれぞれファイルにあります。さらに、そこに/etc/kubernetes/ある.confユーザのアクセス設定を持つファイルは、マスタノードから管理者、マネージャコントローラ、shedulerとkubeletを占めています。.confファイル内の証明書は、base64エンコード形式のuser.client-certificate-dataフィールドにあります。

この小さなshcertスクリプトを使用して、証明書が発行され、証明書に署名された有効期限を確認できます。

shcert
#!/bin/bash

[ -f "$1" ] || exit

if [[ $1 =~ \.(crt|pem)$ ]]; then
 openssl x509 -in "$1" -text -noout
fi

if [[ $1 =~ \.conf$ ]]; then
 certfile=$(mktemp)
 grep 'client-certificate-data:' "$1"| awk '{ print $2}' | base64 -d > "$certfile"
 openssl x509 -in "$certfile" -text -noout
 rm -f "$certfile"
fi


APIでの認証に作業ノードでkubeletを使用する証明書はまだあります。 kubeadm joinを使用してノードをクラスターに追加した場合、ノードはTLSブートストラップ手順を使用して接続されている可能性が高く、この場合、オプションが指定されている場合、kubeletは証明書を自動的に更新できます--rotate-certificates。最近のバージョンのkubernetesでは、このオプションはデフォルトですでに有効になっています。
TLSブートストラップ手順を使用してノードが接続されていることを確認することは非常に簡単です。この場合、現在の証明書へのシンボリックリンクであるファイル/etc/kubernetes/kubelet.confは、通常、クライアント証明書フィールドのファイルで指定されます/var/lib/kubelet/pki/kubelet-client-current.pem。

kubeadmを使用してクラスターをインストールした場合は、朗報です。バージョン1.15以降、kubeadmは1つのコマンドでほぼすべてのコントロールプレーン証明書を更新できます

kubeadm alpha certs renew all

このコマンドは、/ etc / kubernetesディレクトリ内のすべての証明書を更新します。証明書の有効期限が切れており、すべてが壊れている場合でも同様です。

kubelet証明書のみが更新されません-これはファイルにある証明書です/etc/kubernetes/kubelet.conf！
更新：バージョン1.17以降のkubeadmには、すべてのノード（kubeadm initが実行された最初のウィザードでも）にキューレット証明書の自動更新が含まれています。チェックは非常に簡単です- /etc/kubernetes/kubelet.confファイルへのパスはクライアント証明書のフィールドに示されます/var/lib/kubelet/pki/kubelet-client-current.pem

この証明書を更新するには、create user accountコマンドを使用します

kubeadm alpha kubeconfig user --client-name system:node:kube.slurm.io --org system:nodes > /etc/kubernetes/kubelet.conf

システムにユーザーアカウントがある場合、このコマンドはこのアカウントの証明書を更新します。オプション--client-nameで正しいホスト名を指定することを忘れないでください。既存の証明書の[件名]フィールドでホスト名を確認できます。

shcert /etc/kubernetes/kubelet.conf

そしてもちろん、証明書を更新した後、コントロールプレーンのすべてのコンポーネントを再起動し、ノード全体を再起動するか、etcd、api、controller-manager、スケジューラコマンドdocker stopでコンテナを停止してから、kubelet を再起動する必要がありsystemctl restart kubeletます。

更新：kubeadmを直接ダウンロードできます。たとえば、バージョンkubeadm 1.17.4のコマンドは次のとおりです。

そして最後に、バージョン1.16から、コマンドを使用してクラスターを更新するときに、kubeadmはすべての証明書を更新する方法を学びましたkubeadm upgrade。したがって、クラスターを少なくとも年に1回定期的に更新する場合、証明書は常に有効です。

ただし、クラスターがkubeadmを使用してインストールされていない場合は、opensslを取得して、すべての証明書を個別に更新する必要があります。

問題は、証明書に拡張フィールドが含まれており、さまざまなクラスターインストールツールが独自のフィールドセットを追加できることです。さらに、openssl構成と証明書の内容の出力にあるこれらのフィールドの名前は、相互に関連付けられていますが、弱い相関関係があります。グーグルして選択する必要があります。

証明書の実際の属性と追加の名前は、コマンドを使用して表示できます

openssl x509 -in cert.crt -text

サーバーAPIの証明書を更新するときに問題が発生しました。更新された証明書が機能しませんでした。解決策は、過去1年間有効な証明書を発行することでした。

opensslでは、単純なコマンドで過去に有効な証明書を発行することはできません。コードは、証明書が現在からのみ有効であることを厳密に示しています。しかし、libfaketimeライブラリを使用してローカルで時間を遡ることができます

yum install libfaketime
LD_PRELOAD=/usr/lib64/faketime/libfaketime.so.1 FAKETIME="-365d" openssl x509 -req ...

次のアルゴリズムに従って拡張証明書を発行します。

既存の証明書に従ってCSRを作成し、構成ファイル内の拡張属性のリストで目的のセクションを指定します。

openssl x509 -x509toreq -in "node.cert" -out "node.csr" -signkey "node.key" -extfile "openssl.cnf" -extensions client

対応するルート証明書で署名し、1年前に時間をずらして、構成ファイル内の詳細属性のリストで目的のセクションを指定します

LD_PRELOAD=/usr/lib64/faketime/libfaketime.so.1 FAKETIME="-365d" openssl x509 -req -days 36500 -in "node.csr" -CA "kube-ca.pem" -CAkey "kube-ca-key.pem" -CAcreateserial -out "node.new.cert" -extfile "openssl.cnf" -extensions client

属性を確認し、コントロールプレーンのコンポーネントを再起動します。
