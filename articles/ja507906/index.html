私たちはAngularでDIを最大限に使用しています-プライベートプロバイダーのコンセプト

Angularには非常に強力な依存性注入メカニズムがあります。これにより、アプリケーションを介してデータを転送し、適切な部分に変換して再定義することができます。



したがって、理解しやすいデータフロー、最小限のコードコヒーレンス、テストの容易さ、依存関係の置き換えなど、アプリケーションアーキテクチャをよりシンプルかつ柔軟にすることができます。



それにもかかわらず、アプリケーションのDIはかなり控えめに使用されます。原則として、これはサービスの導入、または依存関係注入ツリーを介した上から下へのグローバルデータの転送です。



この記事では、DIから取得したデータを操作する別の方法を紹介します。目的：このデータを使用するコンポーネント、ディレクティブ、およびサービスを簡素化するため。







AngularでDIが一般的にどのように使用されるか



私は職場とオープンソースでAngularコードを毎日レビューしています。通常、ほとんどのDIアプリケーションでは、次の機能が使用されます。



  1. 依存関係ツリーからAngularエンティティを取得します：ChangeDetectorRef、ElementRefなど。
  2. コンポーネントで使用するサービスを取得します。
  3. 上記のどこかで宣言されているトークンのグローバル構成を取得します。たとえば、ルートモジュールでAPI_URLトークンを設定し、必要に応じてアプリケーションの任意の場所でDIから取得します。


開発者がさらに進んで、既存のグローバルトークンをより便利な形式に変換するケースはあまり一般的ではありません。そのような変換の良い例は、@ ng-web-apis / commonパッケージからのWINDOWトークンです。



AngularはDOCUMENTトークンを提供するので、アプリケーションのどこからでもページオブジェクトを取得できます。コンポーネントはグローバルオブジェクトから独立しているため、テストが簡単で、SSRで壊れることはありません。



WINDOWオブジェクトに定期的にアクセスする必要がある場合は、次のトークンを書き込むことができます。 



import {DOCUMENT} from '@angular/common';
import {inject, InjectionToken} from '@angular/core';

export const WINDOW = new InjectionToken<Window>(
    'An abstraction over global window object',
    {
        factory: () => {
            const {defaultView} = inject(DOCUMENT);

            if (!defaultView) {
                throw new Error('Window is not available');
            }

            return defaultView;
        },
    },
);


誰かがDIツリーから初めてWINDOWトークンを要求すると、トークンファクトリが実行されます-彼はAngularからDOCUMENTオブジェクトを受け取り、そこからウィンドウオブジェクトへのリンクを取得します。



さらに、そのような変換に対する別のアプローチを検討することを提案します-それらがコンポーネントまたはディレクティブのプロバイダーで直接実行され、結果を注入する場合。

便宜上、プロバイダーをコンポーネントの隣の別個のファイル







に配置し、次のファイル構造を取得します。プロバイダーは、データ変換用のorganization.providers.tsファイルに配置され、コンポーネントでそれらを受信するためのトークンになります。 



export const ORGANIZATION_INFO = new InjectionToken<Observable<Organization>>(
   'A stream with current organization information',
);
         :

export const ORGANIZATION_PROVIDERS: Provider[] = [
   {
       provide: ORGANIZATION_INFO,
       deps: [ActivatedRoute, OrganizationService],
       useFactory: organizationFactory,
   },
];

export function organizationFactory(
   {params}: ActivatedRoute,
   organizationService: OrganizationService,
): Observable<Organization> {
   return params.pipe(
       switchMap(params => {
           const id = params.get('orgId');

           return organizationService.getOrganizationById$(id);
       }),
   );
}


コンポーネントのプロバイダーの配列を定義します。ORGANIZATION_INFOトークンの値は、必要なデータ変換を行うファクトリから取得されます。



DI-depsの操作に関するメモでは、DIツリーから必要なエンティティを取得し、それらを引数としてトークン値ファクトリに渡すことができます。それらでは、DIデコレータの使用を含め、DIからエンティティを取得できます。次に例を示します。

   ..
   providers: [ORGANIZATION_PROVIDERS],

説明されているアプローチは、すべての設計上の問題からあなたを救うわけではありません。プロバイダーを小さなものに追加しても、それだけの価値はありません。メソッドで変換を使用したり、パイプを使用したりすると、コードが明確になる場合があります。
