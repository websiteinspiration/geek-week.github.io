安全なGoアプリケーションを開発するための6つの推奨事項

近年、ゴランはますます広がっています。 Docker、Kubernetes、Terraformなどの成功したプロジェクトは、このプログラミング言語に大きな賭けをしています。 Goは、コマンドラインツールを作成するための事実上の標準になっています。セキュリティについて話すと、Goのこの領域ではすべてが完璧な順序になっていることがわかります。つまり、2002年以降、CVEレジストリにはGolang脆弱性エントリが1つしかありません。

ただし、プログラミング言語に脆弱性がないという事実は、この言語で記述されたアプリケーションが完全に安全であることを意味するものではありません。開発者が特定の推奨事項に従わない場合、そのような言語でも保護されていないアプリケーションを作成する可能性があります。 Goの場合、OWASPの資料を参照すると、同様の推奨事項を見つけることができます。。



この記事の著者は、OWASPデータに基づいて、Goで安全なアプリケーションを開発するための6つの推奨事項に基づいて策定されました。

1.ユーザー入力を確認する


アプリケーションが正しく機能することを保証するだけでなく、ユーザーが入力したデータの検証も必要です。また、特別な方法で作成されたデータを使用してシステムを混乱させようとする侵入者と戦うことも目的としています。さらに、ユーザーデータを確認することで、一般的なエラーからユーザーを保護できるため、ユーザーはアプリケーションをより自信を持って操作できます。たとえば、ユーザーコマンドを分析することにより、そのようなアクションがシステムの誤動作につながる可能性がある状況で、複数のレコードを同時に削除する試みを防ぐことができます。

標準のGoパッケージを使用して、ユーザー入力を確認できます。たとえば、パッケージstrconvは文字列データを他のタイプのデータに変換するのに役立ちます。行くもサポート、ありがとうregexp正規表現。複雑なデータ検証スクリプトを実装するために使用できます。Goの開発環境では通常、標準ライブラリが優先されるという事実にもかかわらず、データのチェックを目的としたサードパーティのパッケージがあります。たとえば、validator。このパッケージにより、複雑なデータ構造または単一の値を簡単に検証できます。たとえば、次のコードは構造Userに含まれている電子メールアドレスが正しいかどうかをチェックします。

package main

import (
  "fmt"

  "gopkg.in/go-playground/validator.v9"
)

type User struct {
  Email string `json:"email" validate:"required,email"`
  Name  string `json:"name" validate:"required"`
}

func main() {
  v := validator.New()
  a := User{
    Email: "a",
  }

  err := v.Struct(a)

  for _, e := range err.(validator.ValidationErrors) {
    fmt.Println(e)
  }
}

2. HTMLテンプレートを使用する


XSS（クロスサイトスクリプティング、クロスサイトスクリプティング）は、深刻で広範囲にわたる脆弱性です。 XSSの脆弱性により、攻撃者はアプリケーションに生成されたデータに影響を与える可能性のある悪意のあるコードをアプリケーションに挿入することができます。たとえば、誰かがURLのクエリ文字列の一部としてJavaScriptコードをアプリケーションに送信する場合があります。アプリケーションがこのリクエストを処理すると、このJavaScriptコードを実行できます。その結果、アプリケーション開発者はこれを予期し、ユーザーからのデータをクリーンアップする必要があることがわかりました。

Goには、悪意のあるフラグメントから保護されたHTMLコードを生成できるhtml /テンプレートパッケージがあります。その結果、次のようなコードを実行する代わりに、攻撃されたアプリケーションを表示するブラウザ<script>alert(‘You’ve Been Hacked!’);</script>は、ハッキングされたことをユーザーに通知し、悪意のあるJavaScriptコードをプレーンテキストとして認識します。HTMLテンプレートを使用するHTTPサーバーは次のようになります。

package main

import (
  "html/template"
  "net/http"
)

func handler(w http.ResponseWriter, r *http.Request) {
  param1 := r.URL.Query().Get("param1")
  tmpl := template.New("hello")
  tmpl, _ = tmpl.Parse(`{{define "T"}}{{.}}{{end}}`)
  tmpl.ExecuteTemplate(w, "T", param1)
}
func main() {
  http.HandleFunc("/", handler)
  http.ListenAndServe(":8080", nil)
}

GoでWebアプリケーションを開発するときに使用できるサードパーティのライブラリがあります。これがGorilla Webツールキットであるとしましょう。このツールセットには、開発者を支援するライブラリが含まれています。たとえば、認証Cookieの値をエンコードします。そして、ここに別のプロジェクトがあります-nosurf。これは、CSRF攻撃の防止に役立つHTTPパケットです。

3. SQLインジェクションからプロジェクトを保護する


Web開発に慣れていない場合は、おそらくSQLインジェクション攻撃（任意のSQLコードをクエリに挿入する方法を使用）について知っているでしょう。対応する脆弱性は、OWASPトップ10ランキングで第1位にランクされています。 SQLインジェクションからアプリケーションを保護するには、いくつかの機能を考慮する必要があります。したがって、最初に確認するべきことは、データベースに接続するユーザーが制限された権限を持っていることです。さらに、すでに述べたように、ユーザーが入力したデータを消去するか、特殊文字をエスケープしてパッケージからHTMLEscapeString関数を使用することをお勧めしますhtml/template。

しかし、SQLインジェクションから保護する上で最も重要なことは、パラメーター化されたクエリ（準備された式）の使用です。 Goでは、接続用ではなくデータベース用に式が準備されます。パラメータ化されたクエリの使用例を次に示します。

customerName := r.URL.Query().Get("name")
db.Exec("UPDATE creditcards SET name=? WHERE customerId=?", customerName, 233, 90)

しかし、データベースエンジンが定義済みの式の使用をサポートしていない場合はどうでしょうか。しかし、これがクエリのパフォーマンスに影響を与えるとしたらどうでしょうか。このような場合は関数を使用できますdb.Query()が、最初にユーザー入力をクリアすることを忘れないでください。SQLインジェクションを使用した攻撃を防ぐために、sqlmapなどのサードパーティライブラリを使用できます。

SQL攻撃からアプリケーションを保護するためのあらゆる努力にもかかわらず、攻撃者が依然としてこれらの攻撃に成功する場合があることに注意してください。言う-外部アプリケーションの依存関係を介して。プロジェクトのセキュリティレベルを上げるために、適切なツールを使用してアプリケーションのセキュリティを確認できます。たとえば、Sqreenプラットフォームツール。

4.重要な情報を暗号化する


BASE64エンコーディングなどで特定の文字列を読み取れない場合でも、その中に隠されている情報が確実に保護されているわけではありません。したがって、重要な情報は暗号化して、暗号化されたデータにアクセスできる侵入者から保護する必要があります。通常、データベースパスワード、ユーザーパスワード、ユーザー個人データなどの暗号化された情報。

OWASPプロジェクトのフレームワーク内で、推奨される暗号化アルゴリズムに関していくつかの推奨事項が作成されています。例えば、それbcrypt、PDKDF2、Argon2、scrypt。さまざまな暗号化アルゴリズムの信頼できる実装を含むGoパッケージ、cryptoがあります。アルゴリズムを使用した例を次に示しますbcrypt。

package main

import (
  "database/sql"
  "context"
  "fmt"

  "golang.org/x/crypto/bcrypt"
)

func main() {
  ctx := context.Background()
  email := []byte("john.doe@somedomain.com")
  password := []byte("47;u5:B(95m72;Xq")

  hashedPassword, err := bcrypt.GenerateFromPassword(password, bcrypt.DefaultCost)
  if err != nil {
    panic(err)
  }

  stmt, err := db.PrepareContext(ctx, "INSERT INTO accounts SET hash=?, email=?")
  if err != nil {
    panic(err)
  }
  result, err := stmt.ExecContext(ctx, hashedPassword, email)
  if err != nil {
    panic(err)
  }
}

暗号化を使用する場合でも、サービス間の情報の安全な転送に注意する必要があることに注意してください。たとえば、暗号化された形式で保存されている場合でも、ユーザーのパスワードをプレーンテキスト形式で転送しないでください。インターネット上でデータを送信する場合、特定のシステム内で実行されたリクエストからデータを収集する攻撃者によって傍受される可能性があるという想定から始めることは価値があります。攻撃者は収集した情報を他のシステムから受信したデータと照合することができ、その結果、攻撃者は興味のあるプロジェクトを解読することができます。

5. HTTPSを適用する


今日では、ほとんどのブラウザーは、HTTPSをサポートするために、ブラウザーで開くサイトを必要としています。たとえば、Chromeでは、HTTPSを使用せずにサイトとデータを交換すると、アドレスバーに通知が表示されます。特定のプロジェクトをサポートする組織では、このプロジェクトを構成するサービス間でデータを安全に交換するために、セキュリティポリシーを適用できます。その結果、接続のセキュリティを確保するには、ポート443でリッスンするアプリケーションだけでなく、プロジェクトにも適切な証明書が必要です。攻撃者がHTTPデータ交換に切り替えないように、HTTPSの強制使用を整理する必要があります。

HTTPSを適用するアプリケーションの例を次に示します。

package main

import (
  "crypto/tls"
  "log"
  "net/http"
)

func main() {
  mux := http.NewServeMux()
  mux.HandleFunc("/", func(w http.ResponseWriter, req *http.Request) {
    w.Header().Add("Strict-Transport-Security", "max-age=63072000; includeSubDomains")
    w.Write([]byte("This is an example server.\n"))
  })
  cfg := &tls.Config{
    MinVersion:               tls.VersionTLS12,
    CurvePreferences:         []tls.CurveID{tls.CurveP521, tls.CurveP384, tls.CurveP256},
    PreferServerCipherSuites: true,
    CipherSuites: []uint16{
      tls.TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384,
      tls.TLS_ECDHE_RSA_WITH_AES_256_CBC_SHA,
      tls.TLS_RSA_WITH_AES_256_GCM_SHA384,
      tls.TLS_RSA_WITH_AES_256_CBC_SHA,
    },
  }
  srv := &http.Server{
    Addr:         ":443",
    Handler:      mux,
    TLSConfig:    cfg,
    TLSNextProto: make(map[string]func(*http.Server, *tls.Conn, http.Handler), 0),
  }
  log.Fatal(srv.ListenAndServeTLS("tls.crt", "tls.key"))
}

アプリケーションがポート443でリッスンすることに注意してください。HTTPSの強制使用を担当する行は次のとおりです。

w.Header().Add("Strict-Transport-Security", "max-age=63072000; includeSubDomains")

さらに、TLS構成でサーバー名を指定することには意味があります。

config := &tls.Config{ServerName: "yourSiteOrServiceName"}

Webアプリケーションが内部のチャット会社である場合でも、システムのパーツ間で送信されるデータの暗号化を常に使用することをお勧めします。攻撃者がネットワークを介して送信されたアプリケーションデータを傍受する可能性のある攻撃の結果を想像してください。可能な限り、プロジェクトへの攻撃の可能性に備え、攻撃者の生活をできるだけ複雑にするように準備することは価値があります。

6.エラー処理とロギングに注意してください。


このアイテムは私たちのリストの最後ですが、それは間違いなく重要性の最後から遠く離れています。ここでは、エラー処理とログについて説明します。

本番環境で発生する問題にタイムリーに対処するには、アプリケーションに適切なツールを装備する必要があります。同時に、ユーザーに表示されるエラーメッセージについては十分に注意する必要があります。何が問題だったかについてユーザーに詳細を伝えないでください。実際、攻撃者はこの情報を使用して、プロジェクトで使用されているサービスとテクノロジーを見つけることができます。さらに、ログは通常肯定的な見方で知覚されますが、これらのログはどこかに保存されることを覚えておく価値があります。また、アプリケーションログが悪用された場合、そこからの情報を分析すると、プロジェクトへの攻撃に役立ちます。

ここで最初に考慮すべきことは、Goに例外がないことです。つまり、Goで記述されたアプリケーションのエラーは、他の言語で記述されたアプリケーションのエラーと同じようには処理されません。通常は次のようになります。

if err != nil {
    //  
}

さらに、Goにはと呼ばれる標準のロギングライブラリがありlogます。これが最も簡単な使用例です。

package main

import (
  "log"
)

func main() {
  log.Print("Logging in Go!")
}

ロギングを整理するためのサードパーティライブラリがあります。例えば-ですlogrus、glog、loggo。これは小さな使用例logrusです：

package main

import (
  "os"

  log "github.com/sirupsen/logrus"
)

func main() {
  file, err := os.OpenFile("info.log", os.O_CREATE|os.O_APPEND, 0644)
  if err != nil {
    log.Fatal(err)
  }

  defer file.Close()

  log.SetOutput(file)
  log.SetFormatter(&log.JSONFormatter{})
  log.SetLevel(log.WarnLevel)

  log.WithFields(log.Fields{
    "animal": "walrus",
    "size":   10,
  }).Info("A group of walrus emerges from the ocean")
}

最後に、ログに含まれるデータを操作する場合は、すでに説明したセキュリティの推奨事項を使用してください。特に、そのようなデータをクリアして暗号化します。

概要


ここに記載されている推奨事項は、Goで記述されたプロジェクトで最低限必要なものです。ただし、問題のプロジェクトがコマンドラインユーティリティの場合、ネットワーク経由で送信されるトラフィックの保護を実装する必要はありません。残りのヒントは、ほとんどすべてのタイプのアプリケーションに適用されます。 Goで安全なアプリケーションを開発する問題を深く探求したい場合は、このテーマに関するOWASPブックをご覧ください。そして、ここでゴーアプリケーションのセキュリティを確保することを目的とした様々なツールへのリンクが含まれているリポジトリです。

アプリケーションのセキュリティ保護の分野で何をする場合でも、セキュリティは常に向上する可能性があり、攻撃者は常に新しい攻撃方法を見つけていることに注意してください。したがって、アプリケーションの保護は、ケースごとではなく常に行う必要がある作業です。
