API開発者は、REST APIの作成を呼び出す前に、次のルールに注意してください。

REST APIは、特定の通信プロトコルに依存すべきではありませんが、このプロトコルへのマッピングが成功するかどうかは、メタデータの可用性、メソッドの選択などに依存する可能性があります。一般に、URIスキームのプロトコルは、識別に任意のURIスキームを使用できるようにする必要があります。 [ここでのエラーは、識別が相互作用から分離されないことです。]

REST APIには、HTTP PATCHメソッドやリンクヘッダーフィールドなど、標準プロトコルの定義が不十分なビットの詳細を入力または修正する場合を除いて、通信プロトコルの変更を含めないでください。壊れた実装の回避策（HTMLが一連のHTTPメソッドを定義していると想定するほど馬鹿なブラウザなど）は、個別に、または少なくともアプリケーションでは、回避策が最終的に廃止されることを期待して決定する必要があります。 [ここでのエラーは、リソースインターフェイスが特定のものであり、普遍的ではないことです。]

REST APIは、リソースの表現に使用されるメディアのタイプの定義、アプリケーションの状態の管理、および既存の標準メディアタイプのハイパーテキストサポートによる拡張された関係とマークアップ名の定義に、ほとんどすべての記述的努力を費やすべきです。どのURIでどのメソッドを使用するかを説明するために費やした努力は、メディアタイプの処理ルール内で完全に定義する必要があります（ほとんどの場合、既存のメディアタイプによってすでに定義されています）。 [ここでのエラーは、リソースの外部の情報がハイパーテキストではなく対話を制御することです。]

REST APIは、固定リソース名または名前空間を定義するべきではありません（タイトなクライアント/サーバー接続）。サーバーは、独自の名前空間を自由に管理できる必要があります。代わりに、適切なURIを作成する方法（たとえば、HTMLフォームやURIテンプレートなど）をサーバーにクライアントに指示し、メディアタイプと参照関係でこれらの指示を定義します。 [ここでのエラーは、リソースの構造がこのリソースの外部に設定されていることです。たとえば、実際にはRPCの機能的関係の類似物であるドメイン固有の標準です。

REST APIには、クライアントにとって重要な「型付けされた」リソースがあってはなりません。仕様の作成者は、リソースタイプを使用してインターフェースの背後にあるサーバーの実装を記述することができますが、これらのタイプは無関係であり、クライアントからは見えない必要があります。クライアントにとって重要な唯一のタイプは、現在のビューのメディアタイプと標準化された関係名です。 [CM。高い]

REST APIは、初期URIとターゲットオーディエンスに適した標準化されたメディアタイプのセット（つまり、APIを使用できるクライアント側で期待される）以外の事前知識なしで入力する必要があります。この瞬間から、アプリケーション状態のすべての遷移は、受信したビューに存在するサーバーによって提供されるオプションの選択、またはこれらのビューを使用したユーザーの操作によって決定されます。移行は、メディアの種類に関するクライアントの知識と、オンザフライで改善できるリソース（たとえば、要求に応じたコード）との通信のメカニズムに関するクライアントの知識によって決定（または制限）できます。 [ここでのエラーは、リソースの外部の情報がハイパーテキストではなく対話を制御することです。]
