KubernetesでのDNSの問題。死後


日付： 2020年2月28日

著者： AMET U.、アンドレイS.、イゴールK.、アレクセイP.

ステータス：完了

ブリーフ： DNSの一部が利用できない（26分）クラスタKubernetesでいくつかのサービスの

影響： 15,000イベントはAサービスのために失われました、 BおよびC

根本的な原因： Kube-proxyがconntrackテーブルから古いエントリを正しく削除できなかったため、一部のサービスはまだ存在しない送信に接続しようとしました
E0228 20:13:53.795782       1 proxier.go:610] Failed to delete kube-system/kube-dns:dns endpoint connections, error: error deleting conntrack entries for UDP peer {100.64.0.10, 100.110.33.231}, error: conntrack command returned: ...

トリガー： Kubernetes-cluster内の負荷が低いため、CoreDNS-autoscalerはデプロイのポッド数を3から2に減らしました

：別のデプロイアプリケーションが新しいノードの作成を開始し、CoreDNS-autoscalerがクラスターサービスに追加のデッキを追加し、上書きコントラックテーブルの

検出を引き起こしました： Prometheusの監視で、サービスA、B、Cの多数の5xxエラーが検出され、Kibanaで当番エンジニアの5xxエラーの呼び出しが開始されました
