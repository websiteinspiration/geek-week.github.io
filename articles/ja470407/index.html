しかし、私はYandexタスクの実用的な無益性に関する論文に同意しません。日常的な開発でさえ、「いいえ」、「いいえ」ですが、いくつかの解決策があるタスクを設定します。長い間考える必要はありませんが、コードサイズ、パフォーマンス、表現力の点で最適ではありません。もう1つは正反対ですが、効率的で理解しやすいアルゴリズムを構築するには、プログラマーにある程度の経験が必要です。以下はインタビューの例です。

/*    getRanges,    : */
getRanges([0, 1, 2, 3, 4, 7, 8, 10]) // "0-4,7-8,10"
getRanges([4,7,10]) // "4,7,10"
getRanges([2, 3, 8, 9]) // "2-3,8-9"

このタスクについて、私は真剣に鈍く、そしてそれが最も美しい方法ではないと判断しました。解決策は残念ながら保存されていないため、候補の1つに解決策を提供します。

function getRanges(arr: number[]) {
  return arr.map((v, k) => {
    if (v - 1 === arr[k - 1]) {
      if (arr[k + 1] === v + 1) {
        return ''
      } else {
        return `-${v},`
      }
    } else {
      return v + ','
    }
  }).join('').split(',-').join('-')
}

マイナス：存在しないインデックスで配列にアクセスし、醜い文字列操作：join-split-join。このソリューションも正しくありません。例としてgetRanges（[1、2、3、5、6、8]）を使用すると、「1-3、5-6、8」が返され、最後にカンマを「キル」するには、ロジックを複雑にし、読みやすさを低下させることにより、条件をさらに改善します。

Yandexスタイルのソリューションは次のとおりです。
const getRanges = arr => arr
  .reduceRight((r, e) => r.length ? (r[0][0] === e + 1 ? r[0].unshift(e) : r.unshift([e])) && r : [[e]], [])
  .map(a => a.join('-')).join(',')

let n = 0
while (++n < 5) {
	setTimeout(() => console.log(n), 10 + n)

//     ?

これはJavaScript開発者にとって非常に重要なテストだと思います。ここでのポイントは、事前インクリメントと事後インクリメントの違いの閉鎖と理解ではありませんが、説明できない理由により、インタビュー対象者の4分の1は、サイクルが終了する前にconsole.logが実行されると考えています。誇張ではありません。これらの人々は少なくとも2年間の履歴書と実務経験があり、コールバックに関連付けられていない他のタスクを正常に解決しました。これは非同期で育った新世代のJavaScript開発者であるか、Promiseについて何か他のことを聞いたJavaScript開発者ですが、彼らのためのコールバック-現代のティーンエイジャー向けのディスクフォンのような-は初めてであっても番号をダイヤルしますが、理解しません仕組みと理由。

このタスクには続きがあります。console.logもsetTimeout内で実行されるようにコードを追加する必要がありますが、値は1、2、3、4がコンソールに表示されます。ここでは、「ライブ、ラーニング」と言うのが適切です。そのような解決策を提案しました：

setTimeout(n => console.log(n), 10 + n, n)

そして、setTimeoutとsetIntervalが3番目以降の引数をコールバックに渡すことがわかりました。はい、残念です。ちなみに、知識は役に立ちました。私はこの機能を2回以上使用しました。

Promiseでテストされたスキルを以下に示します。通常、私はこの問題を純粋な約束で解決してから、非同期/待機を使用するように依頼します。async / awaitを使用すると、ソリューションは直感的に簡単です。

function async fetchUrl(url) {
  for (let n = 0; n < 5; n++) {
    try {
      return await fetch(url)
    } catch (err) { }
  }
  throw new Error('Fetch failed after 5 attempts')
}

この決定に「ライブ、ラーニング」という言葉を適用することもできますが、私のYandexインタビュアーに関して、彼は非同期/待機を使用できる/使用できないことを指定していませんでした。また、このソリューションを書いたとき、彼は驚きました：「私は働きませんでした非同期/待機では、簡単に解決できるとは思いませんでした。」彼はおそらくこのようなものを見ることを期待していました：

この例は、人が約束をどれだけよく理解しているかを示すことができます。これは特に重要です。約束を完全に理解していない開発者のJavaScriptコードを見たら、次のようにsequelizeトランザクションの約束を準備しました。

const transaction = Promise.resolve()

そして彼はなぜ彼のトランザクションに一人のユーザーしか現れなかったのかと思った。Promise.allを使用することもできますが、Promise.prototype.thenが別のコールバックを追加するのではなく、新しいPromiseを作成することがわかります。これは次のようになります。

let transaction = Promise.resolve()
for (const user of users) {
  transaction = transaction.then(() => {
    await perform_some_operation...
    return some_action...
  })
}

このケースでは、約束を理解するタスクを複雑にすることを考えさせられましたが、問題を解決することを拒否し、文字通りくだらないと呼ばれる候補者に助けられ、彼は実際のコードでの作業に慣れていると言いました。私たちのプロジェクトのソースコードで、彼に実際のコードを与えました：
