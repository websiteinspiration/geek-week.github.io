可用性グループのセカンダリレプリカの統計の更新

整合性チェックやバックアップなど、セカンダリレプリカの優れた可用性グループ機能を気に入って使用しています。

実際、この情報をレプリカのデータベースに保存できないことは依然として頭痛の種です（さらにCDCのようなものについて考えると、さらに不快になります）。

しかし、文句を言うのはやめましょう。ここに主なアイデアがあります。親愛なるマイクロソフト、統計を更新するために私たちの手がかりを使ってみましょう。

常に*方法、または何かがある


*ほとんどの場合

、Enterprise Edition MS SQL Serverで可能なソリューションの既知の基本的な詳細をリストしましょう。

  • レプリカを読み取り可能にし、そこからデータを読み取ることができます（これを常に行う必要があったわけではありませんが、自分が何をしているのか本当にわかっている場合...）
  • オブジェクトをTempdb（そうです、マルチテラバイトのテーブルはおそらくそのような操作にはあまり適していません）、または書き込み可能な別のデータベースにコピーできます。
  • 両方のレプリカからアクセス可能な共有フォルダーに結果を書き込むことができます（ファイル共有内のテキストファイルにします）。
  • SQL Serverから統計をblobとしてエクスポートできます。
  • ダウンロードしたblobを統計にインポートできます。

そうしよう


SQL Server 2017（任意のバージョンを使用できます）を備えた仮想マシンのペアにテストAGがあり、統計を更新する単純なテーブルを作成します。

以下は、テーブルを作成して100万行を挿入するスクリプトです。

DROP TABLE IF EXISTS dbo.SampleDataTable;

CREATE TABLE dbo.SampleDataTable (
    C1 BIGINT NOT NULL,
	C2 BIGINT NOT NULL,
	CONSTRAINT PK_SampleDataTable PRIMARY KEY (C1)
);

INSERT INTO dbo.SampleDataTable WITH (TABLOCK)
SELECT t.RN, t.RN
	FROM
	(
		SELECT TOP (1000000) ROW_NUMBER()
			OVER (ORDER BY (SELECT NULL)) RN
		FROM sys.objects t1
		CROSS JOIN sys.objects  t2
		CROSS JOIN sys.objects  t3
		CROSS JOIN sys.objects  t4
		CROSS JOIN sys.objects  t5    
	) t
	OPTION (MAXDOP 1);

次に、列c2のST_SampleDataTable_C2統計を作成します。

CREATE STATISTICS ST_SampleDataTable_C2 ON dbo.SampleDataTable(C2);

次に、1000行を挿入します。これは非常に重要であり、そのため、統計を更新する必要があります。

set nocount on;

INSERT INTO dbo.SampleDataTable WITH (TABLOCK)
SELECT 10000000 + t.RN, 999999999
	FROM
	(
		SELECT TOP (1000) ROW_NUMBER()
			OVER (ORDER BY (SELECT NULL)) RN
		FROM sys.objects t1
		CROSS JOIN sys.objects  t2
		CROSS JOIN sys.objects  t3
		CROSS JOIN sys.objects  t4
		CROSS JOIN sys.objects  t5    
	) t
	OPTION (MAXDOP 1);

これで、列C2の値が999999999である1000のエントリがあります。これは間違いなく昇順キーの問題を意味し、計算でメインサーバーに負担をかけないようにレプリカで統計情報を更新する必要があります...彼が顧客にサービスを提供することを妨げた。

古き良きDBCC SHOW_STATISTICSコマンドを使用して、統計を調べてみましょう。

DBCC SHOW_STATISTICS('dbo.SampleDataTable', 'ST_SampleDataTable_C2')


私たちの王国ではすべてが完璧であり、統計は完璧な順序になっていますが、100万行しか考慮されておらず、最終的にこれらの統計の一部となる有害な1000行はありません。

また、DBCC SHOW_STATISTICSコマンドのSTATS_STREAMパラメータを使用して、統計ストリームを確認できます。

DBCC SHOW_STATISTICS('dbo.SampleDataTable', 'ST_SampleDataTable_C2')  WITH STATS_STREAM;



これは、ブログが長年にわたって書いている文字セットにすぎませんが、これが完全に文書化された機能であるかどうかはまだわかりません（ただし、ユーザーが使用するのを止めたことはありません）。

キューで


レプリカのテーブルをtempdbにコピーしてみましょう（私のAGは同期モードですが、非同期でも同じことができますが、データにわずかな遅延が生じる可能性があります）。

use TempDB;

DROP TABLE IF EXISTS dbo.SampleDataTable;

CREATE TABLE dbo.SampleDataTable (
    C1 BIGINT NOT NULL,
	C2 BIGINT NOT NULL,
	CONSTRAINT PK_SampleDataTable PRIMARY KEY (C1)
);

INSERT INTO dbo.SampleDataTable
SELECT C1, C2
    FROM AvGroupDb.dbo.SampleDataTable;

これで、レプリカのtempdbでフルスキャンを使用して統計を更新する準備ができました。
use TempDB;
UPDATE STATISTICS ST_SampleDataTable_C2 ON dbo.SampleDataTable(C2) WITH FULLSCAN;

メインサーバーに戻る


メインサーバーで、メインのSampleDataTableテーブルの統計ストリームを更新する前に、統計ストリームを格納する一時テーブルを作成する必要があります（実際には、このテーブルを多くのデータベース、テーブル、統計用に拡張できます）。

CREATE TABLE dbo.TempStats(
	Stats_Stream VARBINARY(MAX),
	Rows BIGINT,
	DataPages BIGINT );

テキストファイルから新しい一時テーブルにデータをインポートして、何をインポートしたかを見てみましょう。

BULK INSERT dbo.TempStats
   FROM '\\SharedServer\Tempdb\stats.txt'

SELECT *
   FROM dbo.TempStats;



レプリカで計算したのと同じデータを確認できますが、このデータはすでにメインサーバーにあり、あとはテーブルの統計を更新するだけです。この操作は、WITH STATS_STREAM = ...パラメータを使用したUPDATE STATISTICS操作を使用して実行できます。

DECLARE @script NVARCHAR(MAX)
SELECT @script = 'UPDATE STATISTICS dbo.SampleDataTable(ST_SampleDataTable_C2) WITH STATS_STREAM = ' + CONVERT(nvarchar(max), [Stats_Stream],1) 
	FROM dbo.TempStats

PRINT @script;
EXECUTE sp_executesql @script;

このスクリプトは、上記のインポートされた値を読み取り（はい、わかっています。この例は1つのテーブルに対して実行し、複数の統計、テーブル、データベースなどで問題がありませんでした）、UPDATE STATISTICSステートメントを生成して画面に表示し、最後に、それを満たします。

AGフィードバックチャネル


これは、レプリカからメインサーバーへのフィードバックのあるチャネルです。同期AGでトランザクションを誓約した後、メインサーバーはレプリカからの確認を待ちます。このチャネルは、この改善を実装するために使用できると思います。Simon Suが投稿で撮った写真を見てください。



これは、既存のフィードバックチャネルのメカニズム全体を表しています。レプリカは、手順12以降を使用して、情報が保存されたことをプライマリサーバーに確認します。同じチャネルを使用して、レプリカで再カウントした後、統計フローオブジェクトを送信できます。もちろん、この目的のためにtempdbを使用する必要はありませんが、永続的に格納されるべきではないデータベース内にインメモリオブジェクトを作成します（インメモリOLTPスキーマのみのテーブルを参照するか、OracleのNOLOGGINGテーブルを検討してください）。操作の最後に削除する必要があります-それは本当に素晴らしいでしょう。

これは、同期レプリカかどうかに依存すべきではありません。ほとんどの場合、統計情報は数秒ごとに更新されないため、アイデアの2番目の部分につながります。このようなパラメーターを使用して、メインサーバーの統計情報を更新する呼び出しを行います。
