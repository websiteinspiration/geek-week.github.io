指に抽象的な工場

この記事を書く理由は2つあります。最近、Abstract Factoryパターンに出会いました。彼らが言うように-「自分でそれを行う方法がわからない、友達に教える」。資料を統合する最良の方法の1つは、以前に研究した人に説明することです。 2番目の理由-このパターンを研究する過程で、私は個人的には抽象ファクトリー（少なくともハブレに関して）の本質を明確に述べる資料に出くわしませんでした。

それでは始めましょう。このパターンを学ぶことによって自分自身に答える必要がある最初の質問：「抽象ファクトリーとは」最も単純で最も正確な答えは、抽象ファクトリーは「工場の工場」であるということです。しかし、ここで2つ目の質問が表示されます。「なぜだれでも「工場の工場」が必要なのでしょうか？」これに答えるために、実際の例を考えてみましょう。

あなたが自動車市場を完全に支配することに決めたとしましょう。どうやってするの？車のブランドや制作、大規模な広告会社の運営などができます。しかし、この場合、トヨタやフォードなどの自動車市場の巨人と戦わなければなりません。この闘争から勝利を収めることは事実ではありません。より良い解決策は、これらすべての企業の工場を買収し、自社ブランドで自動車を生産し続け、利益をポケットに入れることです。私が間違っていなければ、経済におけるそのような構造は呼ばれています-保持。この持ち株は、アブストラクトファクトリーまたは「ファクトリーオブファクトリー」になります。このプログラムでは、抽象ファクトリ（保持）は、インターフェースまたは抽象クラスで表されます。持ち株に含まれる企業は、このインターフェースを実装するクラスによって表されます。

public interface CarsFactory { }

public class ToyotaFactory implements CarsFactory {}

public class FordFactory implements CarsFactory {}

次に、工場のマネージャーを集めて次のように言います。「これからは、セダンとクーペの2種類のボディを持つ車を作ります。たとえば、日本人はToyotaSedanとToyotaCoupeを行い、アメリカ人はFordSedanとFordCoupeを行います。そして、工場が正確に何をする必要があるのか​​を忘れないように、そしてたとえばSUVを開始しないようにするために、セダンとクーペの一般的な図面を目立つ場所にある私たちの所有するオフィスに置きます（特定の工場では、エンジニアは必要な車の作り方を理解します）。したがって、CarsFactoryインターフェースに2つのメソッドが表示されます。

public interface CarsFactory {
    Sedan createSedan();
    Coupe createCoupe();
}

したがって、CarsFactoryインターフェースの子クラスでは、これらのメソッドも実装する必要があります。

public class ToyotaFactory implements CarsFactory {
    @Override
    public Sedan createSedan() {
        return new  ToyotaSedan();
    }

    @Override
    public Coupe createCoupe() {
        return new ToyotaCoupe();
    }
}

public class FordFactory implements CarsFactory {
    @Override
    public Sedan createSedan() {
        return new  FordSedan();
    }

    @Override
    public Coupe createCoupe() {
        return new FordCoupe();
    }
}

メソッドの戻り値の型は、戻り値に共通の型であることに注意してください-セダンとクーペ。私たちの例えに戻ると、あなたは工場にセダンを作るように言いました-あなたはセダンを得ました。フォードセダンなどの機能は、興味のないものです。

上記のコードから簡単に推測できるように、特定のボディタイプ（セダンとクーペ）を記述するエンティティがプログラムに表示されます。これらのエンティティはインターフェイスになります。

public interface Sedan {}

public interface Coupe {}

そしてもちろん、これらの図面には、特定の工場で作成された自動車の形の特定の実施形態が必要です。

public class ToyotaCoupe implements Coupe {
    public ToyotaCoupe() {
        System.out.println("Create ToyotaCoupe");
    }
}

public class ToyotaSedan implements Sedan {
    public ToyotaSedan() {
        System.out.println("Create ToyotaSedan");
    }
}

public class FordCoupe implements Coupe {
    public FordCoupe () {
        System.out.println("Create FordCoupe");
    }
}

public class FordSedan implements Sedan {
    public FordSedan() {
        System.out.println("Create FordSedan");
    }
}

以上が、あらゆる種類の自動車を製造できる「工場の工場」です。将来的には、SUVの生産を開始するとよいと判断するかもしれません。別のインターフェースを作成し、SUVの図面を本社のオフィスに吊るす必要があります（必要なメソッドをCarsFactoryに追加し、それを子会社の工場に実装します）。また、市場の別の部分をキャプチャして、たとえばすべての日産プラントを購入することを決定することも可能です。つまり、CarsFactory-NissanFactoryを実装する別のクラスを作成し、このブランド（NissanCoupe、NissanSedanなど）で車の生産を開始する必要があります。

しかし、特定のユーザー（自動車購入者）は、私たちの持ち物とどのように相互作用しますか？バイヤーは一般的に、あなたが世界のすべての自動車工場を獲得したことを知りません。彼は開催地の小さなささやかなオフィスにやって来て、「車が必要です！「いいね！」-私たちは彼に言います-「あなたはそのアドレスに宛てた！工場の工場はあなたが必要とするものです！」

CarsFactory factory;

「この時間帯にどの社用車が好きですか？」と尋ねます。顧客がトヨタを購入したいとします。問題はありません！

factory = new ToyotaFactory();

「どのタイプの体がいいですか？」セダンとしましょう。"偉大な選択肢！"

factory.createSedan();

車の準備ができました、あなたは行くことができます！
