テラフォームの落とし穴


ループ、ifステートメント、デプロイメントテクニックに関連するいくつかの落とし穴、およびTerraform全体に関係するより一般的な問題を取り上げます。

  • countおよびfor_eachパラメータには制限があります。
  • ダウンタイムなしの導入制限
  • 良い計画でさえ失敗するかもしれません。
  • リファクタリングにはトリックがあります。
  • 据え置き整合性は、据え置きと整合性があります...

Countおよびfor_eachパラメータには制限があります


この章の例では、countパラメータとfor_each式がループと条件付きロジックでアクティブに使用されています。パフォーマンスは良好ですが、2つの重要な制限があるため、知っておく必要があります。

  • countおよびfor_eachでは、リソース出力変数を参照できません。
  • countおよびfor_eachはモジュール構成では使用できません。

Countおよびfor_eachはリソースの出力変数を参照できません


複数のEC2サーバーを展開する必要があり、何らかの理由でASGを使用したくないと想像してください。あなたのコードは次のようになるかもしれません：

resource "aws_instance" "example_1" {
   count             = 3
   ami                = "ami-0c55b159cbfafe1f0"
   instance_type = "t2.micro"
}

それらを順番に検討します。

countパラメータには静的な値が割り当てられているため、このコードは問題なく機能します。applyコマンドを実行すると、3つのEC2サーバーが作成されます。しかし、現在のAWSリージョン内の各アベイラビリティーゾーン（アベイラビリティーゾーンまたはAZ）に1つのサーバーをデプロイしたい場合はどうでしょうか？コードでaws_availability_zonesデータソースからゾーンのリストをロードし、各ゾーンを循環させ、countパラメーターを使用してEC2サーバーを作成し、インデックスによって配列にアクセスすることができます。

resource "aws_instance" "example_2" {
   count                   = length(data.aws_availability_zones.all.names)
   availability_zone   = data.aws_availability_zones.all.names[count.index]
   ami                     = "ami-0c55b159cbfafe1f0"
   instance_type       = "t2.micro"
}

data "aws_availability_zones" "all" {}

countパラメータは問題なくデータソースを参照できるため、このコードも同様に機能します。しかし、作成する必要のあるサーバーの数がリソースの出力に依存している場合はどうなりますか？これを実証するための最も簡単な方法は、名前から推測できるように、ランダムな整数を返すrandom_integerリソースを取得することです。

resource "random_integer" "num_instances" {
  min = 1
  max = 3
}

このコードは、1から3までの乱数を生成します。このリソースの結果出力をaws_instanceリソースのcountパラメータで使用しようとするとどうなるか見てみましょう。

resource "aws_instance" "example_3" {
   count             = random_integer.num_instances.result
   ami                = "ami-0c55b159cbfafe1f0"
   instance_type = "t2.micro"
}

このコードに対してterraformプランを実行すると、次のエラーが発生します。

Error: Invalid count argument

   on main.tf line 30, in resource "aws_instance" "example_3":
   30: count = random_integer.num_instances.result

The "count" value depends on resource attributes that cannot be determined until apply, so Terraform cannot predict how many instances will be created. To work around this, use the -target argument to first apply only the resources that the count depends on.

Terraformでは、リソースが作成または変更される前に、計画段階でcountおよびfor_eachを計算する必要があります。つまり、countとfor_eachは、リテラル、変数、データソース、さらにはリソースのリスト（それらの長さは計画中に決定できる場合）を参照できますが、リソースの計算された出力変数は参照できません。

countおよびfor_eachはモジュール構成では使用できません


countパラメータをモジュール設定に追加したくなるかもしれません：

module "count_example" {
     source = "../../../../modules/services/webserver-cluster"

     count = 3

     cluster_name = "terraform-up-and-running-example"
     server_port = 8080
     instance_type = "t2.micro"
}

このコードは、モジュール内でcountを使用して、webserver-clusterリソースの3つのコピーを作成しようとします。または、いくつかのブール条件に応じて、モジュール接続をオプションにして、countパラメーターに値0を割り当てることもできます。このコードはかなり妥当に見えますが、terraformプランを実行した結果、次のエラーが発生します。

Error: Reserved argument name in module block

   on main.tf line 13, in module "count_example":
   13: count = 3

The name "count" is reserved for use in a future version of Terraform.

残念ながら、Terraform 0.12.6がリリースされた時点では、モジュールリソースでのcountまたはfor_eachの使用はサポートされていませんでした。Terraform 0.12リリースノート（http://bit.ly/3257bv4）によると、HashiCorpは将来この機能を追加する予定です。そのため、この本を読んだ時期によっては、すでに利用可能になっている場合があります。確実に確認するには、こちらのTerraform変更ログを確認してください。


create_before_destroyブロックをASGと組み合わせて使用​​することは、1つのニュアンスを除いて、ダウンタイムがゼロのデプロイメントを整理するための優れたソリューションです。自動スケーリングルールはサポートされていません。または、より正確に言うと、これはすべてのデプロイメントでASGサイズをmin_sizeにリセットします。これは、自動スケーリングルールを使用して実行中のサーバーの数を増やす場合に問題になる可能性があります。

たとえば、webserver-clusterモジュールにはaws_autoscaling_scheduleリソースのペアが含まれており、午前9時にクラスター内のサーバー数が2から10に増加します。たとえば、午前11時に展開すると、新しいASGは10では起動せず、2つのサーバーのみで起動し、翌日の午前9時までこの状態を維持します。

この制限はいくつかの方法で回避できます。

  • aws_autoscaling_scheduleの繰り返しパラメーターを0 9 * * *（「午前9時に開始」）から0-59 9-17 * * *（「毎分午前9時から午後5時まで実行」）に変更します。ASGに既に10台のサーバーがある場合、この自動スケールルールを再実行しても何も変更されません。これが必要です。しかし、ASGグループが最近展開された場合、このルールにより、1分でサーバーの数が10に達することが保証されます。これは完全に洗練されたアプローチではなく、10台のサーバーから2台のサーバーへの大きなジャンプやその逆もユーザーに問題を引き起こす可能性があります。
  • AWS APIを使用してASG内のアクティブなサーバーの数を決定するユーザー定義スクリプトを作成し、外部データソース（249ページの「外部データソース」セクションを参照）を使用して呼び出し、ASGグループのdesired_capacityパラメーターをこのスクリプトによって返される値に設定します。したがって、新しい各ASGインスタンスは常に古いTerraformコードと同じ容量で実行され、そのメンテナンスが複雑になります。

もちろん、理想的には、Terraformにはゼロダウンタイムのデプロイメントの組み込みサポートが必要ですが、2019年5月の時点で、HashiCorpチームはこの機能を追加する予定はありませんでした（詳細はこちら）。

正しい計画が正しく実行されない可能性があります。


planコマンドを実行すると、完全に正しいデプロイメントプランが得られることがありますが、applyコマンドはエラーを返します。たとえば、第2章で作成したIAMユーザーに使用したのと同じ名前のaws_iam_userリソースを追加してみます。

resource "aws_iam_user" "existing_user" {
   #       IAM,
   #      terraform import
   name = "yevgeniy.brikman"
}

これで、planコマンドを実行すると、Terraformは一見、非常に合理的な展開計画を表示します。

Terraform will perform the following actions:

   # aws_iam_user.existing_user will be created
   + resource "aws_iam_user" "existing_user" {
         + arn                  = (known after apply)
         + force_destroy   = false
         + id                    = (known after apply)
         + name               = "yevgeniy.brikman"
         + path                 = "/"
         + unique_id         = (known after apply)
      }

Plan: 1 to add, 0 to change, 0 to destroy.

applyコマンドを実行すると、次のエラーが発生します。

Error: Error creating IAM User yevgeniy.brikman: EntityAlreadyExists:
User with name yevgeniy.brikman already exists.

   on main.tf line 10, in resource "aws_iam_user" "existing_user":
   10: resource "aws_iam_user" "existing_user" {

もちろん問題は、その名前のIAMユーザーがすでに存在していることです。そして、これはIAMユーザーだけでなく、ほとんどすべてのリソースで発生する可能性があります。誰かが手動で、またはコマンドラインを使用してこのリソースを作成した可能性がありますが、識別子が一致すると競合が発生する可能性があります。このエラーには、Terraformの初心者を驚かせることが多いフレーバーがたくさんあります。

重要な点は、terraform planコマンドはTerraformステータスファイルで指定されたリソースのみを考慮するということです。リソースが他の方法（たとえば、AWSコンソールをクリックして手動で）で作成された場合、それらはステータスファイルに入らないため、Terraformはplanコマンドの実行時にそれらを考慮しません。その結果、一見正しい計画は失敗します。

これから2つの教訓を学ぶことができます。

  • Terraformの使用をすでに開始している場合は、他に何も使用しないでください。インフラストラクチャの一部がTerraformを使用して管理されている場合、それを手動で変更することはできません。そうしないと、奇妙なTerraformエラーが発生するリスクが生じるだけでなく、コードがインフラストラクチャを正確に表現できなくなるため、IaCの多くのメリットが失われます。
