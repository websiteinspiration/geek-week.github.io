Combien de méthodes devraient être dans une classe?

C'est peut-être l'un des problèmes les plus importants rencontrés par les développeurs utilisant des classes. Nous avons les principes de SOLID, nous avons notre propre vision du beau et un grand nombre de divergences. Je n'espère pas donner une réponse exhaustive "combien de méthodes devraient être dans votre classe". Mais je partagerai mes réflexions sur les raisons pour lesquelles il devrait y avoir moins de méthodes dans les classes, beaucoup moins. Jusqu'à 1.

Dans une plus large mesure, mon raisonnement concerne les services (objets apatrides) qui, à mon avis, prévalent sur tous les autres types d'objets dans les projets réels. Une classe ne doit pas stocker en elle-même un méli-mélo de méthodes uniquement en raison de leur implémentation similaire!

Après avoir appris les bases de la POO, les débutants associent cours et objets aux chats et aux chiens. Autrement dit, à la première impression, les objets sont perçus comme des projections du monde réel ou certains modèles contenant un ensemble de méthodes nécessaires pour résoudre le problème. Et c'est une excellente opinion sur la POO, plus précisément qu'elle ne devrait l'être.

Mais après quelques projets commerciaux, la vue s'effondre. L'entreprise est régie par des services et des modèles anémiques. Je ne sais pas si c'est bon ou mauvais, mais c'est la réalité. Par conséquent, dans la pratique, on obtient souvent des classes épaisses qui n'ont aucune analogie avec le monde réel et ne se prêtent pas à une généralisation sans ambiguïté. On les appelle souvent des services, souvent ils forment des couches. Et la compréhension de l'objet tombe au niveau - un groupe de méthodes ayant un objectif similaire. Ou des méthodes qui fonctionnent avec des données similaires. Ou quelque chose d'autre similaire.

Chaque fois que je commence à écrire un service, je pense depuis longtemps au nom ... Encore et toujours un autre service vide de sens. Mais que faire s'il n'y a pas de nom? Comment le nom du service reflète-t-il le méli-mélo de demandes collectées à l'intérieur? Qui utilise généralement ces méthodes à la fois?

L'exemple de refactoring de l'oncle Bob ne me laisse pas partir longtemps [Clean Code, p. 177]:

class Sql {
    constructor(private table: string, private columns: Column[]) {}

    public create(): string;
    public insert(fields: object[]): string;
    public selectAll(): string;
    public select(criteria: Criteria): string;

    // ...
}

La classe Sql change lorsqu'un nouveau type de commande est ajouté. De plus, il changera lors du changement des détails de l'implémentation d'un type de commande existant ... Deux raisons du changement signifient que la classe Sql viole le principe de responsabilité uniforme.
Voici le résultat de la conversion.

Le Cheka a mentionné à plusieurs reprises que les classes devraient être compactes. Mais cet exemple plaît par sa spécificité, et c'est tout simplement incroyable!

J'ai pensé, parce qu'il y a tellement d'avantages à se séparer, pourquoi en réalité les classes gonflées prévalent? Parce que nous allons diviser les classes au niveau des fonctions et qu'il deviendra plus difficile de comprendre la situation dans son ensemble? (Mais FP-Schnicks est-il complexe à ce sujet?) Ou parce que l'encapsulation débordera de nos gros services-services? Il me semble qu'il y a un côté obscur qui garde le secret des grandes classes.

Qu'est-ce que la classe donne en termes de constructions de langage de programme, si nous rejetons la romance? La classe a un bon côté - les méthodes publiques qui forment automatiquement l'interface de l'objet. L'interface forme une abstraction, ce qui devrait simplifier le programme. C'est de la romance.

Mais si vous regardez de façon pragmatique, la classe ne collecte que les dépendances nécessaires via le constructeur, stocke les membres privés qui ne sont utilisés nulle part ailleurs et fournit aux clients un ensemble de fonctions collantes (publiques).

Et c'est le côté obscur - les champs privés, les méthodes privées et les dépendances. C'est ce que les développeurs sont le plus souvent guidés en combinant des méthodes dans le service. Parfois, il semble que si la classe contient tous les membres nécessaires, alors c'est le meilleur endroit pour une nouvelle méthode! Et le service se transforme en gardien des dépendances et des méthodes qui ont besoin de ces dépendances.

Depuis quand les dépendances communes doivent-elles unir quelque chose? La similitude de l'implémentation interne n'est pas la raison de la création de l'abstraction. En d'autres termes, en ajoutant une nouvelle méthode au service car elle est «très similaire à d'autres méthodes» dans son implémentation, nous violons l'encapsulation, SRP, OCP et ISP. Mais je rencontre constamment cela.

Ainsi, dans la pratique, l'architecture de service ne forme pas d'abstractions de haut niveau, mais représente simplement une analogie du procéduralisme. De plus, la pire analogie, puisque les procédures n'ont pas d'entités superflues sous forme de pseudo-généralisation des noms de service et ne tirent pas les dépendances inutilisées. (Quelque part sombre FPShniki).

Regardons un exemple de la documentation officielle Angular.

@Injectable({ providedIn: 'root' })
export class HeroService {
 private heroesUrl = 'api/heroes';
 private httpOptions = { headers: new HttpHeaders({ 'Content-Type': 'application/json' }) };
 
 constructor(private http: HttpClient, private messageService: MessageService) {}
 
 getHero(id: number): Observable<Hero> {
   const url = `${this.heroesUrl}/${id}`;
   return this.http.get<Hero>(url).pipe(
     tap((_) => this.log(`fetched hero id=${id}`)),
     catchError(this.handleError<Hero>(`getHero id=${id}`)),
   );
 }
 
 addHero(hero: Hero): Observable<Hero> {
   return this.http.post<Hero>(this.heroesUrl, hero, this.httpOptions).pipe(
     tap((newHero: Hero) => this.log(`added hero w/ id=${newHero.id}`)),
     catchError(this.handleError<Hero>('addHero')),
   );
 }
 
 // ...

   return (error: any): Observable<T> => {

@Injectable({ providedIn: 'root' })
export class HeroService {
 
 private heroesUrl = 'api/heroes';  // URL to web api
 
 httpOptions = {
   headers: new HttpHeaders({ 'Content-Type': 'application/json' })
 };
 
 constructor(
   private http: HttpClient,
   private messageService: MessageService) { }
 
 /** GET heroes from the server */
 getHeroes(): Observable<Hero[]> {
   return this.http.get<Hero[]>(this.heroesUrl)
     .pipe(
       tap(_ => this.log('fetched heroes')),
       catchError(this.handleError<Hero[]>('getHeroes', []))
     );
 }
 
 /** GET hero by id. Will 404 if id not found */
 getHero(id: number): Observable<Hero> {
   const url = `${this.heroesUrl}/${id}`;
   return this.http.get<Hero>(url).pipe(
     tap(_ => this.log(`fetched hero id=${id}`)),
     catchError(this.handleError<Hero>(`getHero id=${id}`))
   );
 }
 
 /* GET heroes whose name contains search term */
 searchHeroes(term: string): Observable<Hero[]> {
   if (!term.trim()) {
     // if not search term, return empty hero array.
     return of([]);
   }
   return this.http.get<Hero[]>(`${this.heroesUrl}/?name=${term}`).pipe(
     tap(x => x.length ?
        this.log(`found heroes matching "${term}"`) :
        this.log(`no heroes matching "${term}"`)),
     catchError(this.handleError<Hero[]>('searchHeroes', []))
   );
 }
 
 //////// Save methods //////////
 
 /** POST: add a new hero to the server */
 addHero(hero: Hero): Observable<Hero> {
   return this.http.post<Hero>(this.heroesUrl, hero, this.httpOptions).pipe(
     tap((newHero: Hero) => this.log(`added hero w/ id=${newHero.id}`)),
     catchError(this.handleError<Hero>('addHero'))
   );
 }
 
 /** DELETE: delete the hero from the server */
 deleteHero(hero: Hero | number): Observable<Hero> {
   const id = typeof hero === 'number' ? hero : hero.id;
   const url = `${this.heroesUrl}/${id}`;
 
   return this.http.delete<Hero>(url, this.httpOptions).pipe(
     tap(_ => this.log(`deleted hero id=${id}`)),
     catchError(this.handleError<Hero>('deleteHero'))
   );
 }
 
 /** PUT: update the hero on the server */
 updateHero(hero: Hero): Observable<any> {
   return this.http.put(this.heroesUrl, hero, this.httpOptions).pipe(
     tap(_ => this.log(`updated hero id=${hero.id}`)),
     catchError(this.handleError<any>('updateHero'))
   );
 }
 
 /**
  * Handle Http operation that failed.
  * Let the app continue.
  * @param operation - name of the operation that failed
  * @param result - optional value to return as the observable result
  */
 private handleError<T>(operation = 'operation', result?: T) {
   return (error: any): Observable<T> => {
 
     // TODO: send the error to remote logging infrastructure
     console.error(error); // log to console instead
 
     // TODO: better job of transforming error for user consumption
     this.log(`${operation} failed: ${error.message}`);
 
     // Let the app keep running by returning an empty result.
     return of(result as T);
   };
 }
 
 /** Log a HeroService message with the MessageService */
 private log(message: string) {
   this.messageService.add(`HeroService: ${message}`);
 }
}


Il s'agit d'un service typique. (Je dirais même «meilleures pratiques»). Et à première vue, tout va bien pour lui. HeroService fonctionne avec des entités Hero, rien de plus. Beaucoup de superflu! Aucun client n'utilise l'interface de manière cohérente - uniquement par parties.

Que faire lorsqu'une nouvelle méthode API pour Hero apparaît? Quand devez-vous arrêter de tout plier en un et de diviser la classe? Je suis sûr que tout le monde a posé ces questions. Mais la présence même de ce service dans le projet ne fait que confondre les développeurs, les poussant à écrire du code via une modification, pas une extension.

Voyons ce qui rend les méthodes intégrées à HeroService «similaires» et comment elles se décomposent.

  • Toutes les méthodes utilisent une URL commune - elle est résolue en mettant BASE_HERO_API_URL dans la constante.
  • Certaines méthodes utilisent les mêmes en-têtes http - il est décidé de les placer dans un intercepteur. Ou vous pouvez sélectionner un HttpClient spécial qui encapsulera les méthodes get, post, put, delete avec les options nécessaires.
  • handleError - est résolu en le plaçant dans une classe distincte, qui servira parfaitement de personne responsable du traitement des erreurs.

Il s'est avéré que tous les facteurs d'unification sont des détails facultatifs de la mise en œuvre, qui auraient pu être différents sans la volonté de circonstances aléatoires. Il n'y avait aucune raison externe à la fusion. Nous écrivons sur le Web, et c'est étrange quand un bouton appelle pour la première fois addHero, puis il utilise l'autre fonctionnalité HeroService.

@Injectable({ providedIn: 'root' })
export class GetHeroesOperation {
 constructor(
   private http: HeroHttpClient,
   private messageService: MessageService,
   private errorHandler: ErrorHandler,
 ) {}
 
 execute(): Observable<Hero[]> {
   return this.http.get<Hero[]>().pipe(
     tap((_) => this.messageService.add('fetched heroes')),
     catchError(this.errorHandler.handleError<Hero[]>('getHeroes', [])),
   );
 }
}
 
@Injectable({ providedIn: 'root' })
export class AddHeroOperation {
 constructor(
   private http: HeroHttpClient,
   private messageService: MessageService,
   private errorHandler: ErrorHandler,
 ) {}
 
 execute(hero: Hero): Observable<Hero> {
   return this.http.post<Hero>('', hero).pipe(
     tap((newHero: Hero) => this.logAddedHero(newHero)),
     catchError(this.errorHandler.handleError<Hero>('addHero')),
   );
 }
 
 private logAddedHero(newHero: Hero): void {
   return this.messageService.add(`AddHeroOperation: added hero w/ id=${newHero.id}`);
 }
}
 
@Injectable({ providedIn: 'root' })
export class ErrorHandler {
 constructor(private messageService: MessageService) {}
 
 handleError<T>(operation: string, result?: T) {
   return (error: any): Observable<T> => {
     // TODO: send the error to remote logging infrastructure
     console.error(error); // log to console instead
 
     // TODO: better job of transforming error for user consumption
     this.messageService.add(`${operation} failed: ${error.message}`);
 
     // Let the app keep running by returning an empty result.
     return of(result as T);
   };
 }
}
 
@Injectable({ providedIn: 'root' })
export class HeroHttpClient {
 private httpOptions = { headers: new HttpHeaders({ 'Content-Type': 'application/json' }) };
 
 constructor(@Inject(BASE_HERO_API_URL) private baseUrl: string, private http: HttpClient) {}
 
 get<T>(endpoint = ''): Observable<T> {
   return this.http.get<T>(this.getUrl(endpoint));
 }
 
 private getUrl(endpoint: string): string {
   return Location.joinWithSlash(this.baseUrl, endpoint);
 }
 
 post<T>(endpoint = '', data: unknown): Observable<T> {
   return this.http.post<T>(this.getUrl(endpoint), data, this.httpOptions);
 }
 
 put<T>(endpoint = '', data: unknown): Observable<T> {
   return this.http.put<T>(this.getUrl(endpoint), data, this.httpOptions);
 }
 
 delete<T>(endpoint = ''): Observable<T> {
   return this.http.delete<T>(this.getUrl(endpoint), this.httpOptions);
 }
}


Je n'essaie pas (et je ne veux pas) de faire des déclarations telles que: "Une classe ne peut pas avoir plus que la 1ère méthode!". Bien sûr, peut-être lorsqu'un objet a un état, d'autres règles entrent en vigueur. De plus, je ne suis pas opposé à l'inclusion de méthodes qui devraient être utilisées de manière cohérente dans la classe: isExecutable, validateParams, execute.

Ce que je dis, c'est plus sur les services de monstres. Puisque les services ne constituent le plus souvent pas une abstraction sensée. Et ce serait mieux si l'accent se déplaçait vers leur diminution. Pour ma part, j'ai formulé le principe suivant: Une

classe ne doit pas contenir un ensemble de méthodes uniquement en raison de leur implémentation similaire.

En suivant cette règle, vous pouvez détecter les éléments système perdus qui sont masqués dans les méthodes privées. Cette approche aidera à se conformer à l'OCP et à l'ISP littéralement. Lorsque vos fonctionnalités développent vraiment le programme, et n'ajoutez pas de méthodes dans 10 classes sur 10 couches.
