Hervorragend, fast 90% der Anwesenden. Der Docker-Workflow besteht also global aus drei Hauptkomponenten: dem Client-Teil, dem Docker-Host und der Docker-Registrierung. Angenommen, Sie möchten einen Container für ein bestimmtes Image ausführen. Sie sagen dem Kunden: "Laden Sie dieses Bild herunter und starten Sie die Container!". Für die Kommunikation mit dem Host verwendet der Client die Befehle Docker Build, Docker Pull und Docker Run. Er gibt dem Host einen Befehl und sagt: "Ja, ich habe dieses Image nicht!". Dann greift er standardmäßig auf die Registrierung des Docker-Hubs unter hubdocker.com zu, lädt das gewünschte Image herunter, speichert es und führt es in Containern aus. In diesem Sinne hat der Client absolut keine Zustände außerhalb des Arbeitsorts. Er sendet einfach einen Befehl wie CLI per Kabel. Er geht als REST-API an den Host und der Host versteht die API-Anforderungen.Lädt ein Image von einem Hub oder einer automatisierten Registrierung von Amazon ECR Docker-Containern herunter und startet dann Container basierend auf diesem Image. Dies ist ein vollständig statischer Status, der nur auf dem Host verwaltet wird. In diesem Fall haben wir einen Host, aber es ist einfach genug, eine Multi-Host-Lösung zu verwenden.

Genau dafür ist das Orchestrierungsframework gedacht. Sie möchten kein verteiltes Anwendungssystem auf einem einzelnen Host erstellen, da dies der schwächste Punkt ist, der im Falle eines Fehlers das gesamte System zum Erliegen bringt. Sie möchten die Aufgabe auf mehrere Hosts verteilen, um einen globalen Fehler zu vermeiden. Sie möchten beispielsweise eine Java-Anwendung ausführen, die einen Anwendungsserver, einen Datenbankserver, Caching und viele verschiedene Komponenten verwendet. Darüber hinaus müssen häufig mehrere Instanzen dieser Anwendung gleichzeitig ausgeführt werden, da eine typische Java-Anwendung eine Mehrcontaineranwendung ist, die Container auf mehreren Hosts verwendet. Daher benötigen Sie ein Orchestrierungsframework, nicht nur zum Planen der Containerisierung, sondern auch zum Erstellen der Fähigkeit zur Verwaltung Ihres Clusters.Lebenszyklen und andere notwendige Operationen.

In dieser Präsentation werde ich Ihnen erklären, wie sich die beiden wichtigsten Orchestrierungs-Tools für Anwendungen - Docker und Kubernetes - in ihren Funktionen, Vor- und Nachteilen unterscheiden. Ich werde diese Probleme aus der Sicht eines Entwicklers betrachten, der beginnt, eine Anwendung mit einem grundlegenden Schlüsselkonzept zu erstellen. Dieses Konzept umfasst Komponenten wie einen Cluster, einen einzelnen Container, viele Container, deren Interaktion untereinander, die Ermittlung von Diensten (oder die Katalogisierung von Diensten, mit denen sie sich gegenseitig kennenlernen können), den Lastenausgleich und den dauerhaften Speicher für zustandsbehaftete Container sowie das Wissen darüber, wie die lokale Entwicklung durchgeführt wird.



Aus der Dev-Perspektive gehen wir zur Ops-Perspektive über, zur architektonischen Seite des Systems: Wie multipliziert man den Master-Knoten, wie sollte der Scheduler-Algorithmus funktionieren, wenn auf mehreren Hosts gearbeitet wird, mit welchen Regeln und Einschränkungen kann ich den Container auf einem bestimmten Host ausführen, wie kann ich Führen Sie Überwachungs- und fortlaufende Updates durch, und verwenden Sie die Cloud-Unterstützung. Heute werden wir darüber sprechen.

Beginnen wir mit Docker Swarm - dies ist eine der Funktionen der Docker Engine, beginnend mit Version 1.2, einem Standard-Clustering-Tool. Es konvertiert eine Reihe von Docker-Hosts in einen einzelnen sequentiellen Swarm-Cluster. Wenn Sie zu docker.com gehen, können Sie eine von zwei Editionen herunterladen: Community Edition oder Enterprise Edition (heute heißen diese Versionen Stable und Edge). Der erste Build ist das, was ich verwende. Er ist stabil und wird alle 3 Monate aktualisiert, im Gegensatz zum neuesten Edge-Build, der jeden Monat aktualisiert wird. Ich verwende die Docker CE-Version für Mac. Wenn ich Docker CE auf meinen Computer herunterlade, markiere ich während der Installation den darin enthaltenen Schwarm. Danach kann ich sofort mit dem Erstellen von Clustern, dem Orchestrieren und Planen von Containern beginnen.

Beachten Sie, dass das Docker Swarm-Modul eine Kommunikation mit dem Host erfordert. Wenn Sie also eine Netzwerkkarte mit einem Multi-Host-System verwenden, müssen Sie eine bestimmte Docker-Hostadresse mit dem Befehl docker swarm unit - - listen-addr: 2377 abhören.
Ein gelbes Symbol zeigt standardmäßig einen einzelnen Masterknoten an. Der Assistent gibt mir ein Worker-Token-Steuerungs-Token. Ich nehme es und verwende es, um einen neuen Arbeitsknoten, eine physische oder virtuelle Maschine, anzuhängen. Es spielt keine Rolle.



Dieses Token gibt an, dass dieser Knoten dem ursprünglichen Master beitreten soll. Mit Worker-Token kann ich meinen eigenen Docker-Cluster bilden. Mit Docker ist es daher sehr einfach, einen Cluster aus mehreren Knoten zu erstellen. Die nächste Folie zeigt einen Cluster aus einem Masterknoten und 5 Arbeitsknoten.



Mit der gleichen Leichtigkeit können Sie Cluster mit mehreren Masterknoten erstellen - dem primären Masterknoten und dem sekundären Masterknoten - und diese mit Arbeitsknoten verknüpfen.



Wenn Sie auf der Clientseite eine Anforderung an den sekundären Master senden, leitet dieser als Proxy diese an den primären Master weiter, und dieser führt die Anforderung bereits aus und verteilt die Last auf die Arbeitsknoten. Standardmäßig kann Ihr Container auch in einem Manager-Knoten arbeiten. Beim Erstellen eines Clusters sollten Container jedoch nur in Arbeitsknoten platziert werden, und der Manager verwaltet nur den Cluster.

Das folgende Grundkonzept, ein Cluster, bedeutet eine Architektur aus mehreren Masterknoten und mehreren Arbeitsknoten, in denen Container ausgeführt werden. Dies ist ein streng konsistentes Schema, das auf dem Raft-Konsensalgorithmus basiert und aufgrund des Lesens aus dem RAM die höchste Geschwindigkeit aufweist.



Ich empfehle dringend, eine Vorlage mit dem Raft-Protokoll zu verwenden, da das System beim Aufteilen der Netzwerke nicht darüber nachdenken muss, wer jetzt die Rolle eines Assistenten spielt, da bei Raft die Auswahl automatisch erfolgt. Auf der untersten Ebene befinden sich Swarm-Arbeitsknoten, die über das Gossip-Protokoll miteinander kommunizieren. Sie sehen, dass rote Container auf drei Knoten und grüne Container auf zwei Knoten funktionieren. Dies legt nahe, dass Sie mit Swarm Worker mehrere verschiedene Aufgaben starten und ausführen können.

Wie funktioniert das Docker Replicated Service-Konzept? Da ich ein Java-Entwickler bin, verwende ich in diesem Fall den WildFly-Container.



Dazu gebe ich über die Docker-Befehlszeile den Befehl docker service create - - replicas 3 - - name web jboss / wildfly ein und gebe die Anzahl der benötigten Replikate an. Der Name des Dienstes lautet web und füge ein Wildfly-Image hinzu. Dieses Bild wird auf die 3 Docker-Hosts hochgeladen und der Container gestartet.

Ein weiteres wichtiges Konzept ist der Ausfall des Knotens Knotenfehler. Was geschieht?



Docker stellt fest, dass der aktuelle Status des Systems nicht mit dem gewünschten übereinstimmt. Ein Knoten ist aus dem Cluster herausgefallen. Dies bedeutet, dass Container von diesem Knoten auf einen anderen Host übertragen werden müssen. Wenn Sie also erklärt haben, dass drei Replikate funktionieren müssen, stellt Docker sicher, dass diese drei Replikate unter allen Umständen vorhanden sind. Eine Situation kann auftreten, wenn ein Container in einem Arbeitsknoten ausfällt. Ähnlich wie im vorherigen Fall verfolgt Docker die angegebene Anzahl von Containern. Wenn sie abnimmt, wird ein Scheduler aktiviert, der den Container sofort wiederherstellt und den aktuellen Status des Clusters in den gewünschten Status versetzt. Das heißt, wenn Sie 3 Replikate angekündigt haben, sollten alle diese Replikate zu einem bestimmten Zeitpunkt funktionieren.
