JavaとJVMは依然として非常に人気がありますが、サーバーレステクノロジーとクラウド指向のマイクロサービスを使用する場合、Javaとその他のJVM向けの言語は、メモリ領域を取りすぎて読み込みが遅くなり、Javaやその他の言語の使用に適さなくなります。寿命の短いコンテナで使用します。幸いなことに、この状況は現在、Quarkusのおかげで変わり始めています。

まず、話を始めましょう。


ストリームとコンテナ


バージョン8u131以降、エルゴノミクス機能の向上により、Javaでサポートされるコンテナーは多かれ少なかれあります。特に、JVMは、JVMが実行されているプロセッサコアの数を認識し、スレッドプール（通常はフォーク/結合プール）を適切に構成できます。もちろん、これはすばらしいことですが、HTTPサーブレットを使用し、Tomcat、Jettyなどで実行される従来のWebアプリケーションがあるとします。その結果、このアプリケーションは各要求に個別のストリームを与え、データベース、ファイル、その他のサービスにアクセスするときなど、I / O操作を待機している間、このストリームをブロックできるようにします。つまり、このようなアプリケーションのサイズは、使用可能なコアの数ではなく、同時リクエストの数に依存します。さらに、これは、コア数によるKubernetesの割り当てまたは制限がここでは特に役に立たないことを意味します。そして取引は駆け引きで終了します。

メモリ不足


ストリームはメモリです。また、コンテナ内のメモリ制限は万能薬ではありません。アプリケーションとスレッドの数を増やし始めるだけで、遅かれ早かれスイッチング周波数が大幅に増加し、結果としてパフォーマンスが低下します。さらに、アプリケーションが従来のマイクロサービスフレームワークを使用したり、データベースに接続したり、キャッシングを使用したり、何らかの方法でメモリを消費したりする場合は、JVMの内部を調べてメモリを管理する方法を確認し、強制終了しないツールを絶対に必要とするJVM自体（例：XX：+ UseCGroupMemoryLimitForHeap）。Java 9以降、JVMはcgroupを受け入れ、それに応じて適応することを学んだという事実にもかかわらず、メモリのバックアップと管理は依然としてかなり複雑な作業です。

割り当てと制限


Java 11では、CPUクォータのサポートが導入されました（PreferContainerQuotaForCPUCountなど）。 Kubernetesは、制限と割り当てのサポートも提供します。はい、これはすべて理にかなっていますが、アプリケーションが割り当てられた割り当てを再び超えると、サイズ（従来のJavaアプリケーションの場合と同様）はコアの数と各要求に対する個別のスレッドの割り当てによって決まるという結論に達します。これにはほとんど意味がありません。
さらに、Kubernetesの基礎となるプラットフォームの割り当てと制限、または水平（スケールアウト）スケーリングの機能を使用する場合、問題も解決しません。元の問題の解決により多くのリソースを費やすか、その結果、リソースを使用することになります。そして、それがパブリックパブリッククラウドの負荷の高いシステムである場合、実際に必要以上のリソースを使用するようになるでしょう。

そして、これをどうするか？


簡単な方法であれば、非同期でノンブロッキングのI / Oライブラリと、Netty、Vert.x、Akka などのフレームワークを使用します。それらは反応性があるため、コンテナでの作業にはるかに適しています。ノンブロッキングI / Oのおかげで、同じスレッドが同時に複数のリクエストを処理できます。 1つの要求がI / O結果を待機している間、その処理スレッドは解放され、別の要求に使用されます。そして、最終的にI / O結果が到着すると、最初の要求の処理が続行されます。同じストリーム内で要求処理を交互に行うことにより、スレッドの総数を減らし、要求を処理するためのリソース消費を減らすことができます。

ノンブロッキングI / Oでは、コアの数が並行して実行できるI / Oスレッドの数を決定するため、重要なパラメーターになります。これを正しく使用すると、コア間で効果的に負荷を分散し、少ないリソースでより高い負荷に対処できます。

どのように、そしてそれですべてですか？


いいえ、もう1つあります。リアクティブプログラミングは、リソースの有効活用に役立ちますが、代償も伴います。特に、非ブロッキングの原則に従ってコードを書き換え、入出力ストリームのブロックを回避する必要があります。そして、これは完全に異なる開発と実装のモデルです。そして、便利なライブラリはたくさんありますが、これは通常の考え方の根本的な変更です。

最初に、非同期で実行されるコードの記述方法を学ぶ必要があります。ノンブロッキングI / Oの使用を開始したらすぐに、リクエストに対する応答を受け取ったときに何が起こるかを明示的に規定する必要があります。ブロックして待機するだけでは失敗します。その代わりに、コールバックを渡したり、リアクティブプログラミングを使用したり、継続したりできます。しかし、それだけではありません。ノンブロッキングI / Oを使用するには、ノンブロッキングのサーバーとクライアントの両方が必要です。 HTTPの場合、すべてが単純ですが、データベース、ファイルシステムなどもあります。

完全なエンドツーエンドの反応性により最大の効率が得られますが、実際にはそのようなシフトを消化するのは困難です。したがって、リアクティブコードと命令コードを組み合わせる機能は、次のことを行うために必要な条件になります。

  1. ソフトウェアシステムの最も負荷の高い領域のリソースを効果的に使用します。
  2. 他の部分では、より単純なスタイルコードを使用します。

Quarkusの紹介


実際、これがQuarkusの本質です。1つのランタイム環境内で反応型モデルと命令型モデルを組み合わせることができます。

QuarkusはVert.xとNettyに基づいており、その上に、開発者を支援するために多くの反応フレームワークと拡張機能が使用されています。Quarkusは、HTTPマイクロサービスだけでなく、イベント駆動型アーキテクチャも構築するように設計されています。その反応的な性質により、メッセージングシステム（Apache Kafka、AMQPなど）で非常に効率的に動作します。

トリックは、命令コードと反応コードの両方に同じジェットエンジンを使用する方法です。



Quarkusは見事にそれを行います。命令型とリアクティブ型のどちらを選択するかは明らかです。一方と他方のリアクティブ型コアの両方を使用することです。そして、非常に役立つのは、イベントループスレッド（イベントループスレッド、別名IOスレッド）を通過するほとんどすべてを処理する高速なノンブロッキングコードです。しかし、従来のRESTまたはクライアント側アプリケーションがある場合、Quarkusは命令型プログラミングモデルの準備ができています。たとえば、QuarkusでのHTTPサポートは、非ブロッキングおよびジェットエンジン（Eclipse Vert.xおよびNetty）の使用に基づいています。アプリケーションが受信したすべてのHTTP要求は、最初にイベントループ（IOスレッド）を通過し、次に、要求を管理するコードの部分に送信されます。宛先に応じて、要求制御コードは別のスレッド（いわゆるワーカースレッド、サーブレットとJax-RSの場合に使用されます）内で呼び出すか、元の入出力ストリーム（リアクティブルートリアクティブルート）を使用できます。



Vert.xエンジン上で動作する非ブロッキングクライアントは、メッセージングシステムのコネクタに使用されます。したがって、メッセージングミドルウェアクラスシステムからのメッセージを効率的に送信、受信、および処理できます。
