小さい、はい。Firecrackerマイクロ仮想の開梱

Firecrackerマイクロ仮想化を記録します。マルチユーザーワークロードを分離するには、仮想マシンとコンテナという2つの一般的な方法を採用しています。私たちは両方のアプローチの最良のものを絞り込み、可能な限り単純化し、実際の高負荷でテストします。その結果、数百ミリ秒で起動できる仮想コンピュータを完全に分離できます。このソリューションはAWS LambdaとFargateの内部で機能し、毎秒数百万のサーバーレス機能とコンテナーをクラウドで実行します。それは爆竹と呼ばれています。



このマイクロ仮想化ツールはOpenSourceで利用できます。タスクでマルチテナントの分離が必要な場合（たとえば、独自のクラウドを作成することにした場合）、Firecrackerが必要です。アマゾンウェブサービスアーキテクトの

Vasily Pantyukhinは、Firecrackerアーキテクチャ、AWS Lambdaでのその使用方法について話し、代替ソリューションと比較して、統合の例を示します。

パブリッククラウドの自然な特徴


パブリッククラウドの基本的な特性の1つはマルチテナントです。

誰かがこの用語を「マルチテナンシー」または「マルチユーザー環境」と翻訳します。しかし、私の見解では、これは本質を反映していません。マルチテナンシーとは、異なるユーザーとそのワークロードが同じ物理インフラストラクチャ上に存在し、それをそれらの間で共有するが、お互いについて何も知らない場合です。マルチユーザーワークロードとは異なり、マルチテナンシーは、リソースの分離とそれらへのアクセスに関する根本的により厳しい要件によって区別されます。

パブリッククラウドに固有のもう1つのプロパティは、高負荷です。

信じてください、AWSの場合、これは大きな負荷です！スクリーンショットは実際のデータの例です。これらの数百万の「オウム」がどのくらいの期間、どの地域で測定されたかは言いませんが、これはある種の緊急事態ではなく、通常の操作モードです。



ある矛盾が判明しました。一方では、ユーザーをできるだけ隔離する必要があります。一方で、非常に高いレベルの生産性とリソースの利用を確保する必要があります。一方を改善すると、もう一方の制限につながることがよくあります。すべての面で最大値を取得するための妥協点を見つける方法、さらに良い方法は？

仮想マシンまたはコンテナー？


役立つ可能性のある2つの基本的なアプローチがあります。これらは仮想マシンとコンテナです。それぞれに長所と短所があります。



仮想マシンの主な欠点は、ロードに時間がかかることです。通常、マシンの起動には数十秒、場合によっては数分かかります。しかし、virtualokには長所があります。具体的な負荷を互いに絶縁します。そして、両方の観点から：

  • ある仮想マシンが別のマシンのデータにアクセスできない場合のセキュリティ。
  • リソース、8 GBのメモリを注文したとき、このRAMが私のものであり、私が支払ったリソースを誰も請求できないことを願っています。

コンテナーの場合は逆です。コンテナーは軽量であるため、非常に速くロードされます。水平方向に簡単に拡大縮小できます。しかし、私たちがパブリッククラウドプロバイダーとして生きていけない機能が1つあります。それらは共有カーネルを使用します-OSカーネルはすべてのコンテナー間で共有されます。



コンテナーは安全でないと言えますか？いいえ、深刻な脆弱性がありますが。

今日では、適切に「溶接された」コンテナが十分なレベルのセキュリティを提供すると思います。

問題は異なります-共有カーネルは、将来的に、テナントの分離を破壊する脆弱性が発生しないことを根本的に保証するものではありません。パブリッククラウドは、理論的にもそれを買う余裕はありません。

さらに別の矛盾があり、解決策を模索する必要があります。最も簡単な方法は、仮想マシンであるママから、そしてコンテナーであるパパからすべてを最大限に活用し、交差して何かを収束させることです。実際にやった。それは爆竹であることがわかりました。

爆竹はすでに生産されています。重要なサービスの同じ高負荷で、たとえば、AWS Lambda（サーバーレス機能）やAWS Fargate（サーバーレスコンテナー）。

古いAWS Lambdaの問題


AWS Lambdaはサーバーレス機能サービスです。Java、Go、Python、またはその他の言語で関数を受け取り、それをLambdaにスローすると、魔法のように実行されます。リソースを割り当てて管理する必要はありません。これは以前どのように実装されましたか？



AWSアカウントごとに、1つ以上の個別のEC2仮想マシンが割り当てられ、異なるテナントに属する機能を分離しました。このような仮想マシンは、何も実行しない場合でもリソースを消費します。関数を10分ごとに1回実行し、通常のLambdaのように200ミリ秒で実行するとします。EC2マシンは1時間以内に数秒しか使用されないことがわかりました。さらに、実行時であっても、使用可能なすべてのリソースを消費するわけではありません-ベースボードの下に処分します。これは猛烈に不採算です。

AWS Lambda Firecracker


AWS Lambdaの新しいバージョンで何が変更されましたか？エンドユーザーの観点からは、何も変わっていません。本番環境で更新されたアーキテクチャへの移行は、完全に透過的であり、消費者には見えません。ラムダ機能は短命です-簡単に実行できました。

近代建築はどのように見えますか？

現在、最低レベルは仮想マシンではなく、物理的なベアメタルです。

このようなサーバーでは、CPUのすべての機能（Intel VTなど）を完全に使用できます。これは、より高いレベルで仮想化を使用するときに追加の利点を提供します。



鉄の上には、カーネルにKVMモジュールを備えたホストOSがあります。独自のゲストOSを備えた爆竹マイクロ仮想マシンが上で起動されます。まあ、彼らはすでにAWS Lambdaサービス自体のコンポーネントを実装しています。

以前は、Lambda機能を使用する顧客ごとに、使用率の低い個別のEC2仮想マシンを割り当てていました。新しいアプローチでは、本当に必要なときにだけ軽量のマイクロ仮想を実行できます。Firecrackerのリソースを互いに分離することで、最大密度の一般的なハードウェアでこれを行うことができます。資源の利用は根本的に改善されました。

箱の中は何ですか？


ボクシングを約束したので、Firecrackerの主なコンポーネントを調べ、各コンポーネントを個別に検討してから、すべてをまとめます。



設計原則


私たちが最初に爆竹の開発について話し始めたとき、私たちは2つの原則に従うことに同意しました。

不要なものをすべて取り除きます。クラシック仮想マシンのロードが遅いのはなぜですか？特に、コードがオーバーロードされているためです。膨大な数のレガシーデバイスをサポートする必要があります。しかし、なぜ10年前に他の誰も使用していないデバイスをエミュレートするのでしょうか。したがって、不要なものはすべて取り除きます。特定の狭いタスクを解決し、この問題の解決に役立たない機能は不要と見なされます。

不要なものをすべて取り除き、1つのタスクに集中します。

残っているものを簡素化します。残されたものでさえ、可能な限りシンプルでなければなりません。

彼らは何を書いたのですか？


爆竹はトレンディな錆で書かれています：

  • 特にメモリに関して、より安全なコードを書くことができます。
  • パフォーマンスとオーバーヘッドは、最新のC ++に匹敵します。
  • Rust は長い間使用されており、2015年以来、多くの優れた高負荷ソリューションを書いています。

鉄


繰り返します-爆竹は実際のハードウェアにインストールする必要があります。参照構成として、i3.metalベアメタルマシンが選択されました。


注：レポートは2019年4月上旬のものでした。その時点では、Intelプラットフォームのみがサポートされていました。 5月にAMDのアルファサポートを追加し、6月にARMを追加しました。 AMDはIntelよりも少し安く、ARMサポートはマルチテナントIoTで作業する興味深い機会を提供します。 

AWSでi3.metalまたは他のベアメタルマシンを注文した場合、実験には強力で高価な構成になります。したがって、Firecrackerをインストールする場合は、実験後にこれらのマシンに支払いを行うことを忘れないでください。そうでなければ、まともなスコアが月末に来るでしょう。

より安いオプションはありますか？はい、仮想環境でFirecrackerを起動できます。しかし、AWSではもうありません。基本的に、EC2でのネストされた仮想化はサポートされていません。ただし、GCP、Azure、DigitalOceanで実行するか、Proxmox、Parallels、VMware Fusionを使用できます。ゲストOSカーネルのバージョンに応じて、特に要件に準拠している場合は、すべてが機能します。

芯


ソリューションの基本的な要素はKVMカーネルモジュールです。

念のため、余談としてKVMについて説明します。これはハイパーバイザー全体ではなく、一部にすぎません。その主なタスクは、仮想プロセッサ（vCPU）のセットアップと仮想マシンの起動です。



しかし、これは本格的な作業には十分ではありません。プロセッサに加えて、いくつかの他のデバイスが必要です。それらのエミュレーションはユーザー空間で行われます。

VMM


少なくとも基本的なデバイスを表示するには、別のコンポーネントであるVirtual Machine Manager（VMM）が必要です。 Firecracker以前は、VMMの主要なオプションはQEMUでした。



QEMUの使用の可能性を真剣に検討しましたが、独自の「自転車」を開発することにしました。これにはいくつかの理由がありました。

開発管理。 QEMUは、100万行を超えるコードを持つ、成熟した大規模な製品です。変更を加えるには、表示するソースコードが多すぎます。多くの人々が開発と開発に関する意思決定に参加しています。

安全性。 QEMUは多くのデバイスをエミュレートできます。そのため、攻撃範囲はかなり大きくなります。私たちの仕事は、マイクロバーチャルを作ることです。セキュリティの観点からは、攻撃対象を最小限に抑える必要があります。

新しい機能を実装する必要性。QEMUには適切なコードがあります。これは成熟した製品であり、明らかなバグはすべてすでに捕捉されています。しかし、その既存の形式では、QEMU機能はまだ私たちに適していませんでした-多くを追加する必要があります。この観点から見ると、QEMUと新製品の新しいコードは同じ品質になります。したがって、特別な勝利は機能しません。

デバイス


爆竹は、デバイスをエミュレートするために使用されるVMMを実装します。次のデバイスをエミュレートします。

  • コンソール。それはオフにすることができますが、便利で必要なもの。
  • キーボード。トリッキーなキーボードを作りました-「リセット」ボタンが1つだけです。Firecrackerで実行される可能性のあるオペレーティングシステムのソフトウェア「リセット」を信頼しません。したがって、彼らは鉄を作りました。
  • ディスクおよびネットワーク用のVirtioドライバー。Virtioは非常に原始的なデバイスです。これは基本的に「リングバッファ」です。ゲストシステムがバッファーに何かを書き込んで、呼び出しをクリックすると、ホストシステムがファイルを介してこのバッファーからデータを読み取ります。 



場合によっては、たとえば、コンテナーとの統合のために、通常のネットワークカードの基本的な機能を表すネットワークデバイスが必要です。Virtioはここには収まりません。単純すぎます。したがって、ネットワークには別のデバイス（Vsock）を使用します。

まあ、私たちはリソース消費を制御する能力も必要です。レートリミッターがこれを担当します。

デバイスがあります。しかし、マイクロ仮想化を管理および構成する方法は？ここでは管理APIが役立ちます。

コントロール


アマゾンにはいくつかの破れない基本原則があります。それらの1つは、すべてのサービスがAPIを介してのみ相互に通信することです。これらがユーザーとして使用する外部サービスであるか、Googleの内部サービスであるかは関係ありません。あるサービスが別のサービスのデータベースに直接アクセスすることはありません-禁止されており、罰せられます。Firecracker APIスレッドは、REST APIを介してマイクロ仮想化の設定と機能にアクセスするために使用されます。



Open API仕様に準拠しているため、Swaggerを使用できます。

メタデータ


そのような悪があります-ハードコーディング。これは、たとえば、別のリソースへのアクセスのログインとパスワードなど、一部のデータがコードに直接縫い付けられる場合です。もちろん、これは許されません。定期的に、マイクロ仮想マシン内でデータを転送する必要があります。これは、メタデータサービスを通じて行われます。



必要な情報をソケット経由でIMDSメタデータ取引に転送します。マイクロ仮想マシン内でこの情報を取得するには、REST APIを使用してhttp://169.254.169.254/latest/meta-dataをリクエストする必要があります。AWSユーザーはすでにこの魔法のIPを知っています。同様に、マイクロバーチャルは自身の構成に関する詳細情報を取得できます。

カタコンテナ

です。-コンテナー分離メカニズムはすでに発明されています-これはカタコンテナーです。なぜそれらを使用しなかったのですか？なぜ独自のものを開発するのですか？
-KataはQEMUと連携しますが、QEMUと連携したくありません。したがって、私たちは自分で調理することにしました。

しかし、それは興味深いことがわかりました。 Kata開発者はFirecracker開発者と面会し、統合することに同意しました。誰もがこれを大きなプラスと見なしているからです。Kata Containers v1.5はすでにQEMUとFirecrackerの両方をサポートしています。

私たちが競争するのではなく、調和して互いに補完し合うことがわかりました。 v1.12のKubernetesでは、runtimeClassNameをカタFCまたはカタQEMUとして定義し、適切な分離でコンテナを実行できます。



あなたの仕事に合った断熱材を​​選ぶ方法-爆竹かQEMU？私の個人的な見解では、アプリケーションがペットであるか牛であるかが重要です。

積荷が牛である場合、爆竹は適しています。ステートレスアプリケーションが元々柔軟な水平スケーリング用に設計されていて、1つ以上のコンテナーの障害が重大でない場合。信頼性の高い断熱を提供し、必要な数の新しいコンテナをすばやくロードして、故障したコンテナを交換できます。
