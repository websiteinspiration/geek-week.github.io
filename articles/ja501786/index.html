新しいアプリケーション「クラゲ」。なぜ羽ばたきますか？

テクニカルディレクターのBoris Goryachevが、メデューサが1年間どのように取り組んだか、なぜFlutterで書かれたのかについて語ります。


5月12日、新しいMedusaモバイルアプリケーション（iOS、Android）のリリースが行われました-書き直すことを決定してから約2年後。なぜそんなに長いの？なぜネイティブアプリではないのですか？なぜFlutterなのか？これらすべては、メドゥーサボリスゴリャチェフのテクニカルディレクターによって伝えられます。



以前のアプリケーションとは異なり、新しいアプリケーションをネイティブにしないことにしました。まず、同じコードを2回書くのは面倒です。第二に、それはうまくいかないので、異なる人々によって書かれた2つの異なるプロジェクトでは、すべてが同期していて同じです。通常、誰かが仕事を遅くし、誰かが休暇に行き、誰かが閉鎖する必要がある技術的な負債を抱えています。ネイティブアプリケーションの作成とサポートにおけるすべての経験が私たちを促しました。そして、私たちは自分のものを探し始めました。私たちは、React NativeとIonicを試し、Basecampのアプローチについて考えました。Webのすべてのもの+薄いネイティブレイヤーで、プログレッシブWebアプリにアクセスしてオンラインを維持しようとしました。

同じ頃、私はGoogle I / Oカンファレンスに参加し、そこでFlutterを作っている人々に会いました。私はすぐにダートを試してFlutterに少し取り組みましたが、そのときのテクノロジーはまだメデューサに対応していません。さまざまなインタラクティブなマテリアルを埋め込んだり、マテリアル内に埋め込んだりすることは不可能でしたが、これはメディアにとって重要です。だから私はフラッターが成長するまで待つことにしました。

待機中、ウェブ側で多くのことを行いました。サイトを書き直し、AMPの新しいバージョンを書きました。私たちはメデューサのすべてのプロジェクトを作成してui-kitに転送しました。これは、サイトが使用するコンポーネントの単一のライブラリであり、おかげで膨大な数のゲームメカニックが可能です。サイトのデスクトップバージョンとモバイルバージョンを分離し、配布ページ（メインページとセクションページ）が独自のルールに従って2つの異なる場所で形成されるようになりました。

サイトでの作業と並行して、新しいアプリケーションについて考えていました。そのナビゲーション、意味、画面、機能などを考え出しました。



並行して、技術部門の構造に重要な変更が加えられました。Googleカレンダーやその他の会議計画ツールを積極的に使用し始め、TrelloからBasecampに切り替えました。それは非常に明白であるため、それについて話すのは奇妙です。しかし、混乱を合理化するために多くの時間と労力を費やしました。会議の明確な議題、迅速なフォローアップ、失われないファイル、およびヒルチャートスコープにより、大量のタスクストリームに対処し、死ぬことはありませんでした。

なぜそれはまだフラッターです。ダートについて少し


人々がフラッターについて学ぶとき、彼らは必然的にダートについて学びます。これはFlutterの最大の欠点と考えられているようですが、書き込もうとするまでは。彼は本当にかっこいいです。以下のJavaScriptは特別です。

Dartは、Googleが開発したプログラミング言語です。それは2011年に発表されました、すなわち、それはまだ若い言語です。 （少なくとも今のところ）主要なプログラミング言語にはなっていませんが、同時に会社自体で非常に積極的に使用されています。 Google以外にも、Drartを使用してフルスタックを書き込むWrikeなどの大企業があります。

DartとFlutterはどちらも同じ会社でサポートされているため、言語をFlutterのニーズに合わせて変更できます。私の知る限りでは、両方のチームが積極的に相互に作用するため、チップは常に言語で表示され、Flutterでより快適なコードを記述できます。

Flutterの構成要素については説明しません。ウィキペディアはこのタスクにうまく対応できますが、著者の中にはChromiumでレンダリングした人がいると言っています。

2018年12月、FlutterチームはFlutterにWebview埋め込みライブラリをリリースしました。それは未加工であり、ちなみに、まだ開発者プレビューステータスを残していませんが、この事実は、将来のアプリケーションの構造を推定し始めるために止まりませんでした。数か月間、私は自由な時間に実験し、最終的な決定がなされました。

最初は、アプリケーション専用の新しいバージョンのAPIを書き始めました。このAPIのイデオロギーは次のように定式化できます。バックエンドで何かを実行できる場合、バックエンドでそれを行う必要があります。クライアントで何かをすることが難しいからではありません。実際のところ、App StoreとGoogle Playでのリリースは骨の折れる長いプロセスです。あなたは彼らの作業スケジュールに適応する必要があり、すべてのユーザーがすぐにアプリケーションを更新するわけではないことを覚えておいてください。サーバーでメインロジックが発生する場合、これは回避できます。タイトルを10ピクセル下に移動する必要がありますか？どういたしまして。コンポーネントをすばやく削除または追加しますか？問題はありません！



同じ理由で、モバイルアプリケーションのAPIには可能な限り単純なコンポーネントが含まれており、そこからほとんどすべてのマテリアルが再帰的に収集されます。 WebViewを通じてゲームを表示するため、「ほぼ」と言います。アプリケーションは、カード、ポッドキャスト、ニュース、「ビッグトップ」など、何を表示するかは関係ありません。すべてが1つのコードで処理され、アプリケーションのタスクはコンポーネントを取得してそれをレンダリングすることです（またはその子のリストに移動して再帰的に呼び出します）。

Flutterを支持するもう1つの重要な論点：開発者は「すべてのピクセルを制御する」。 Sketchのレイアウトのように、どこにでも正しい影があることを確認する必要がある場合、または曲線に沿って透明度を変更したい場合、またはフォントサイズとすべてのタイポグラフィをカスタマイズ可能にする場合、Flutterではすべてがシンプルで現実的です。

Flutterのもう1つの優れた機能は、開発の快適さです。私がWeb開発でよく使用しているホットリロードと、状態を失うことなくアプリケーションをリロードする高速性により、作業が可能な限り簡単になります。アプリケーションが再構築されるまで座って待つ必要はありません。その後、新しいコードが機能するまで待機し、作業している状態を繰り返す必要があります。すべてが本当に速くてクールです。

生態系


FlutterとDartはまだそれほど人気のあるテクノロジーではありませんが、ライブラリを見つけるのに問題はありませんでした。アプリケーションに必要なもののほとんどは、フレームワーク自体またはpub.devにあります。コミュニティはとても楽しく、いつでも手助けする準備ができています。このコミュニケーションは新鮮な空気の息吹であり、非常に協力的です。

また、Google自体もFlutterをサポートしています。私たちは、Firebaseを使用して、プッシュ、分析、​​プロファイル、およびユーザーデータ（読み取り履歴、エピソードの位置、ブックマーク）の保存を行い、すべてが「そのまま機能する」もちろん、これまでにネイティブアプリケーションを作成したことがない人としては、gradleファイルにアクセスしたり、info.plistにプロパティを配置したりするのが面倒なことがあります。しかし、通常はすべてがグーグル化されており、信じられないほど複雑なものはありません。

プラットフォームの違い


iOSとAndroidの違いを際限なく語ることができます。原則として、パターンはユーザーにとってなじみがあり、デザインガイドラインに従う必要があることをすぐに覚えています。私たちはこれに同意しますが、完全ではありません。コーポレートアイデンティティについて、常識について、そしてすでにあなたに馴染みのあるユーザーの行動論理について覚えておくことは重要です。

プラットフォーム自体の履歴を見ると、一部のパターンはデバイスによって変化し、他のパターンはシステムからシステムに流れていることがわかります。これらはタブレットではなく、まだ何も立っていません。企業がGoogleとAppleの推奨事項から離れ、正しいと思うことを実行する数十の素晴らしい例があります。



最も単純で最も鮮明な例は、iOSでスワイプして戻すことです。スワイプを受け入れる画面の領域は、デフォルトでは非常に小さく、約20ピクセルです。これは、iPhoneでメールアプリケーションが機能する方法であり、非常に不便です。 Twitterでは、この領域はわずかに拡大されていますが、Telegramでは単純に巨大です。 Androidの電報には、チャットリストからチャットへの独自の特別なタイプのナビゲーションがあり、iOSとAndroidのクロスです。はい、ガイドラインはすばらしいですが、iPhone XRでは、指をデバイスの左上端に到達させるのは非常に不便です。したがって、私たちは（もちろん、データに依存して）「Medusa」アプリケーションでは、iOSの「戻る」ボタンが下になることを決定しました。しかし、Androidではまったくありません。これを行うには、svaypas（私たちと新しいAndroid svaypasの両方）とシステムの戻るボタンがあります。

私の同僚、メドゥーサナスティアヤロヴァヤのアートディレクターは次のように述べています。

私は小さな手と大きな電話を持っています。ばかげているように見えますが、開発プロセスにおける2つの重要な決定に影響を与えました。最初に、スワイプゾーンを大幅に増やす（50％に増やす）ことを提案しました。これは、ガイドラインからの最初の逸脱です。次に、資料内のナビゲーションボタンと機能ボタンを操作するときに、下の一般メニューに戻るボタンを配置することをお勧めします。これは、インターフェイスの通常のパターンとは異なる点です。彼らは主にそれによってスクロールされるので、今、あなたはあなたの親指でそれに到達することができます。


私たちは、AppleとGoogleのいくつかの推奨事項に違反したという事実に間違いなく飛び立つことを知っています。しかし、どちらかと言えば、電話も使用しており、多くのパターンは、デバイス用に発明されたため、古くなっていることを理解しています。

少し練習：Flutterでの動作


Reactおよびその他のリアクティブフレームワークの後、数日でFlutterでプロトタイプを構築できます。 Flutterの一般的なアイデア-すべてがウィジェットです。 Flutterは2つ（実際には3つ）のパラダイムを提供します。マテリアルデザインのコンセプトに従うマテリアルウィジェットを使用できます。クパチーノウィジェット（iOSのように見え、動作するウィジェット）を使用できます。そして、あなたは自分ですべてを書くことができます。

大まかに言えば、開発者が対話するウィジェットには、ステートレスとステートフルの2つのタイプがあります。

ステートレスウィジェットとは、ウィジェットの状態を変更するロジックがウィジェット内に含まれないプラスまたはマイナスのウィジェットです。

ステートフルウィジェットにはsetState関数があります（hello、react！）。状態を変更すると、ウィジェットが再描画されます。

しかしもちろん、単純なsetStateでの大規模なアプリケーションはそれほど遠くないので、状態管理ソリューションが必要です。

Flutterにはいくつかのソリューションがあります。たとえば、データソースからのストリームとイベントが使用されるBloCパターンがあり、それらには新しいイベントの発生を誘発するビジネスロジックが含まれています。これらのイベントはウィジェットによって受信され、ウィジェットは変更に応答します。

Reactの後に来る人はReduxを試してみたいかもしれません。理論的には、このアプローチを使用してReactで記述した場合、問題はないはずです。

私は両方のアプローチを試し、最終的に3番目を選択しました。 Jellyfishアプリはプロバイダーを使用します。これはGoogleが作成したものではないライブラリですが、興味深いことに、Googleが自宅でプロバイダーを試して、そのBloCを放棄し始め、プロバイダーを別のライブラリーとして繰り返すことを決定しました。しかし、最終的に、彼は他の誰かのコードを繰り返すという考えを放棄し、オープンソースとして生まれたものを使用および維持し始めました。

プロバイダーデバイスは非常に単純です。状態のあるウィジェットです。この状態は、notifyListeners（）を内部的に呼び出す関数によって変更されます。変更に応答する必要のあるウィジェットはウィジェットツリーにあります。これらはウィジェットプロバイダーの直接の子またはその子の子です。 notifyListeners（）が呼び出されると、子はコンテキストを通じて新しい値を取得し、再構築が行われます。

通常、プロバイダーはアプリケーションの「最上部」でアドバタイズされ、一度に多くのプロバイダーを使用できます-それぞれが独自のビジネスロジックを担当します。

現在、アプリケーションのさまざまな場所で「リッスン」している8つのプロバイダーを使用しています。たとえば、ブックマークプロバイダーの仕組みを以下に示します。これにより、読者はお気に入りの素材を保存し、オフラインで、さまざまなデバイスの同じ場所から読むことができます。

読者がブックマークした資料へのキーを含むFirebaseからのドキュメントへの変更に対する反応がインスタンスに追加されます。このドキュメントの各onChangeについて、プロバイダー内の関連する変数が更新され、notifyListenersが呼び出されます。その結果、このプロバイダーをリッスンするすべてのウィジェットに正しいアイコンが表示されます。

プロバイダーを通じて、アプリケーションのテーマも切り替えられました。アプリケーションにはThemeProviderがあり、それ自体にテーマの2つのインスタンスが含まれています。実際、これは、色の名前と色自体を備えた単なるMapです。

その結果、正しい色が次のようにウィジェットに表示されます。

TextSpan(
	text: block['data']['first'],
	style: TextStyle(
		color: Provider.of<ThemeProvider>(context)
        .current
        .bodyFontColor,                            
  fontFamily: 'Proxima Nova',
));

簡略化した場合、プロバイダーコードは次のようになります。

class ThemeProvider extends ChangeNotifier {
  CustomThemeData current;
  
  CustomThemeData blackTheme = CustomThemeData(
    name: 'black',
    bodyFontColor: Color.fromRGBO(184, 184, 184, 1),
    ...
  );
  
  CustomThemeData lightTheme = CustomThemeData(
    name: 'black',
    bodyFontColor: Color.fromRGBO(184, 184, 184, 1),
    ...
  );
  
  ThemeProvider(withTheme, withFontMultiplier, withAutoTheme) {
    theme = withTheme;
    autoTheme = withAutoTheme;
    current = withTheme == 'light' ? lightTheme : blackTheme;
  }
}

したがって、現在の変数を変更すると、テーマから色を取得するすべてのウィジェットが例外なく正しく表示されます。
