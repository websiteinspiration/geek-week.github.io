Webプロジェクトのパフォーマンス専門家も、コード分​​離手法の使用を推奨しています。これらの手法により、JavaScriptコードを個別のバンドルに分割できます。このようなバンドルは、ブラウザーの要求に応じて、並行して、または必要になったときにのみ、ブラウザーでダウンロードできます。

特に最適なキャッシングテクニックに関連するコード分離の多くの利点の1つは、ソースコードで別のファイルに加えられた変更が、バンドル全体のキャッシュの無効化につながらないことです。つまり、開発者「X」によって作成されたnpmパッケージに対してセキュリティ更新が発行され、コンテンツがnode_modules開発者によってフラグメント化された場合、「X」によって作成されたパッケージを含むフラグメントのみを変更する必要があります。

ここでの問題は、これらすべてを組み合わせると、長期的なデータキャッシングの効率が向上することはほとんどないということです。

実際には、ソースコードファイルの1つを変更すると、ほとんどの場合、パッケージアセンブリシステムの複数の出力ファイルが無効になります。そして、これは、これらのファイルの内容のバージョンを反映するハッシュがファイル名に追加されているという事実によるものです。

問題解決アプローチ


技術的な観点から見ると、ファイル名に含まれるファイルの内容のハッシュに関する問題は、ハッシュが名前に含まれていることではありません。それは、これらのハッシュが他のファイル内に現れるという事実にあります。その結果、依存するファイルの名前のハッシュを変更すると、これらのファイルのキャッシュが無効になります。

上記の例の言語を話す、ということで、この問題の嘘を解決するには、それが可能フラグメントインポートする作りvendor断片dep2とdep3フラグメントファイル情報のバージョンを指定せずにvendor。この場合、およびのvendor現在のバージョンを考慮して、ダウンロードしたバージョンが正しいことを確認する必要がdep2ありdep3ます。

アプローチ＃1：カードをインポートする


インポートマップは、カスケードキャッシュの無効化の問題に対する最も簡単な解決策です。さらに、このメカニズムは実装が最も簡単です。ただし、残念ながらChromeでのみサポートされています（さらに、この機能は明示的に有効にする必要があります）。

それにもかかわらず、この決定が将来最も一般的になると確信しているので、インポートカードについての話から始めたいと思います。さらに、インポートカードの操作の説明は、問題を解決するための他のアプローチの特徴を説明するのに役立ちます。

インポートカードを使用して、カスケードキャッシュの無効化を防ぐには、3つのステップがあります。

▍ステップ1


プロジェクトをビルドするときに、ファイルの内容のハッシュを名前に含めないように、バンドラーを構成する必要があります。

ファイル名にその内容のハッシュを含めずに、前の例の図に示されているモジュールを持つプロジェクトをアセンブルすると、プロジェクト出力ディレクトリ内のファイルは次のようになります。

dep1.mjs
dep2.mjs
dep3.mjs
main.mjs
vendor.mjs

対応するモジュールのインポートコマンドにもハッシュは含まれません。

import {...} from '/vendor.mjs';

▍ステップ2


rev-hashなどの ツールを使用して、コンテンツのバージョンを示す名前にハッシュが追加された各ファイルのコピーを生成する必要があります。
作業のこの部分が完了すると、出力ディレクトリの内容は次のようになります（各ファイルに2つのオプションがあることに注意してください）。

dep1-b2c3.mjs",
dep1.mjs
dep2-3c4d.mjs",
dep2.mjs
dep3-d4e5.mjs",
dep3.mjs
main-1a2b.mjs",
main.mjs
vendor-5e6f.mjs",
vendor.mjs

▍ステップ3


ハッシュのある名前の各ファイルとハッシュのない名前の各ファイルの対応に関する情報を格納するJSONオブジェクトを作成する必要があります。このオブジェクトは、HTMLテンプレートに追加する必要があります。

このJSONオブジェクトはインポートマップです。これは次のようになります。

<script type="importmap">
{
  "imports": {
    "/main.mjs": "/main-1a2b.mjs",
    "/dep1.mjs": "/dep1-b2c3.mjs",
    "/dep2.mjs": "/dep2-3c4d.mjs",
    "/dep3.mjs": "/dep3-d4e5.mjs",
    "/vendor.mjs": "/vendor-5e6f.mjs",
  }
}
</script>

その後、ブラウザーは、インポートマップのキーの1つに対応するアドレスにあるファイルのインポートコマンドを見るたびに、キー値に一致するファイルをインポートします。

このインポートマップを例として使用すると、ファイルを参照するインポートコマンド/vendor.mjsが実際にリクエストを実行し、ファイルをダウンロードすることがわかります/vendor-5e6f.mjs。

//    `/vendor.mjs`,  `/vendor-5e6f.mjs`.
import {...} from '/vendor.mjs';

これは、モジュールのソースコードがハッシュを含まないモジュールのファイル名を非常に簡単に参照できることを意味し、ブラウザは常に名前にコンテンツのバージョンに関する情報が含まれるファイルをダウンロードします。また、モジュールのソースコードにはハッシュがない（インポートマップにのみ存在する）ため、これらのハッシュを変更しても、内容が実際に変更されたモジュール以外のモジュールは無効化されません。

おそらく、ファイル名を変更するだけでなく、なぜ各ファイルのコピーを作成したのか疑問に思われるかもしれません。これは、インポートマップを使用できないブラウザをサポートするために必要です。前の例では、そのようなブラウザはファイルのみを表示します/vendor.mjsそして、彼らは通常通りにこのファイルをダウンロードし、同様の構造に遭遇します。その結果、両方のファイルがサーバー上に存在する必要があることがわかりました。
