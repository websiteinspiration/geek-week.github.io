自動的にRealmオブジェクト構造に変換するSourcery

で、インターネット、さらに上Habré、レルムと連携する方法についての記事の束があります。このデータベースは非常に便利で、使用できる場合はコードを書くのに最小限の労力しか必要としません。この記事では、私がやってきた作業方法について説明します。

問題


コードの最適化


言うまでもなく、Realmオブジェクトの初期化コードを記述し、同じ関数を呼び出してオブジェクトを読み書きするのは不便です。あなたはそれを抽象的にラップすることができます。

データアクセスオブジェクトの例：

struct DAO<O: Object> {
    func persist(with object: O) {
        guard let realm = try? Realm() else { return }
        try? realm.write { realm.add(object, update: .all) }
    }
	
    func read(by key: String) -> O? {
        guard let realm = try? Realm() else { return [] }
        return realm.object(ofType: O.self, forPrimaryKey: key)
    }
}

使用：

let yourObjectDAO = DAO<YourObject>()
let object = YourObject(key)
yourObjectDAO.persist(with: object)
let allPersisted = yourObjectDAO.read(by: key)

たとえば、削除、同じタイプのすべてのオブジェクトの読み取り、並べ替えなど、DAOに多くの便利なメソッドを追加できます。これらのメソッドはすべて、どのRealmオブジェクトでも機能します。

不正なスレッドからアクセスした


レルムはスレッドセーフなデータベースです。これから生じる主な欠点は、Realm.Objectタイプのオブジェクトをあるスレッドから別のスレッドに転送できないことです。

コード：

DispatchQueue.global(qos: .background).async {
    let objects = yourObjectDAO.read(by: key)
    DispatchQueue.main.sync {
        print(objects)
    }
}

エラーが発生します：

Terminating app due to uncaught exception 'RLMException', reason: 'Realm accessed from incorrect thread.'

もちろん、1つのスレッドでいつでもオブジェクトを操作できますが、実際には、回避する方がよい特定の困難が生じます。

解決策として、Realm.Objectを異なるスレッド間で静かに転送される構造に変換することは「便利」です。

レルムオブジェクト：

final class BirdObj: Object {
    @objc dynamic var id: String = ""
    @objc dynamic var name: String = ""
    override static func primaryKey() -> String? { return "id" }
}

構造：

struct Bird {
    var id: String
    var name: String
}

オブジェクトを構造に変換するために、
トランスレータプロトコルの実装を使用します。

protocol Translator {
    func toObject(with any: Any) -> Object
    func toAny(with object: Object) -> Any
}

Birdの場合、次のようになります。

final class BirdTranslator: Translator {
    func toObject(with any: Any) -> Object {
        let any = any as! Bird
        let object = BirdObj()
        object.id = any.id
        object.name = any.name
        return object
    }
    func toAny(with object: Object) -> Any {
        let object = object as! BirdObj
        return Bird(id: object.id,
                    name: object.name)
    }
}

これで、レルムオブジェクトではなく構造を受け入れて返すように、DAOをわずかに変更する必要があります。

struct DAO<O: Object> {
    
    private let translator: Translator
    
    init(translator: Translator) {
        self.translator = translator
    }
    
    func persist(with any: Any) {
        guard let realm = try? Realm() else { return }
        let object = translator.toObject(with: any)
        try? realm.write { realm.add(object, update: .all) }
    }
    
    func read(by key: String) -> Any? {
        guard let realm = try? Realm() else { return nil }
        if let object = realm.object(ofType: O.self, forPrimaryKey: key) {
            return translator.toAny(with: object)
        } else {
            return nil
        }
    }
}

問題は解決したようです。これでDAOは、スレッド間を自由に移動できるBird構造を返します。

let birdDAO = DAO<BirdObj>(translator: BirdTranslator())
DispatchQueue.global(qos: .background).async {
    let bird = birdDAO.read(by: key)
    DispatchQueue.main.sync {
        print(bird)
    }
}

同じタイプのコードが大量にある。


スレッド間でオブジェクトを転送する問題を解決したので、新しい問題に遭遇しました。 2つのフィールドのクラスを持つ最も単純なケースでも、追加の18行のコードを記述する必要があります。たとえば10のように2つのフィールドがなく、そのうちのいくつかがプリミティブタイプではなく、変換する必要のあるエンティティであるとします。これにより、同じタイプのコードの行がたくさん作成されます。データベース内のデータ構造の些細な変更により、3つの場所に登る必要があります。

各エンティティのコードは、本質的には常に同じです。それの違いは構造の分野にのみ依存します。

Realm.ObjectとTranslatorをそれぞれ発行して構造を解析する自動生成を作成できます。Sourceryはこれに役立ちます。その助けを借りてモッキングについてのハブラに関する記事がすでにありました。

このツールを十分なレベルで習得するために、テンプレートタグとフィルター ステンシル（Sourceryの作成に基づく）とSourcery自体のドキュメントの十分な説明がありました。

特定の例では、Realm.Objectの生成は次のようになります。

import Foundation
import RealmSwift
#1
{% for type in types.structs %}
#2
final class {{ type.name }}Obj: Object {
        #3
        {% for variable in type.storedVariables %}
        {% if variable.typeName.name == "String" %}
        @objc dynamic var {{variable.name}}: String = ""
        {% endif %}
        {% endfor %}
        override static func primaryKey() -> String? { return "id" }
}

{% endfor %}

＃1-すべての構造を調べます。
＃2-それぞれについて、独自のクラスである子孫オブジェクトを作成します。
＃3-タイプ名== Stringの各フィールドに対して、同じ名前とタイプの変数を作成します。ここでは、Int、Date、およびより複雑なプリミティブなどのプリミティブのコードを追加できます。本質は明らかだと思います。

Translatorを生成するためのコードは似ています。

{% for type in types.structs %}
final class {{ type.name }}Translator: Translator {
    func toObject(with entity: Any) -> Object {
        let entity = entity as! {{ type.name }}
        let object = {{ type.name }}Obj()
        {% for variable in type.storedVariables %}
        object.{{variable.name}} = entity.{{variable.name}}
        {% endfor %}
        return object
    }
    func toAny(with object: Object) -> Any {
        let object = object as! {{ type.name }}Obj
        return Bird(
        {% for variable in type.storedVariables %}
        {{variable.name}}: object.{{variable.name}}{%if not forloop.last%},{%endif%}
        {% endfor %}
        )
    }
}
{% endfor %}

Sourceryは、依存関係マネージャーを介してインストールすることをお勧めします。バージョンを示して、作成したものがすべてのユーザーに同じように機能し、壊れないようにします。

インストール後、BuildPhaseプロジェクトで実行するbashコードを1行書くだけです。プロジェクトのファイルのコンパイルが始まる前に生成する必要があります。



結論


私が挙げた例はかなり単純化されています。大規模なプロジェクトでは、.stencilなどのファイルがはるかに大きくなることは明らかです。私のプロジェクトでは、それらは200行弱を占めていますが、4000を生成し、とりわけRealmでのポリモーフィズムの可能性を追加しています。
一般に、一部のオブジェクトを別のオブジェクトに変換することによる遅延は発生しませんでした。
私はどんなフィードバックと批判にも喜んでいます。
