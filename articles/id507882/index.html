Cara membuat Zabbix berteman dengan Asterisk ‚Äúout of the box‚Äù

Dalam artikel sebelumnya, "Zabbix - Memperluas Perbatasan Makro," saya berbicara tentang cara mendapatkan sesi otorisasi dan menggantinya dengan makro host lokal. Pada artikel ini saya akan memberi tahu Anda cara membuat Zabbix berteman dengan Asterisk tanpa skrip dan perangkat lunak eksternal.



Gagasan "berteman" dari kedua sistem ini lahir sejak lama, dan tanpa menginstal perangkat lunak dan skrip tambahan. Google cepat memberi banyak opsi solusi, semuanya berujung pada menempatkan skrip (di panas, bash, python, dll.) Di server, dan Anda akan senang. Saya ingin menerapkan pemantauan "di luar kotak" - tanpa skrip eksternal dan menginstal perangkat lunak tambahan pada server dengan pemantauan dan pertukaran telepon otomatis.



Saya menghabiskan waktu dengan itu dalam jumlah 4 hari kerja, tetapi hasilnya tidak sia-sia. Bekerja melalui antarmuka AMI, deteksi tingkat rendah, pemicu, dan yang paling penting, sekarang membutuhkan waktu sekitar 15 menit untuk menghubungkan PBX dan semua pengaturan lainnya.Ada



zabbix 4.4, sekitar 100 buah versi Asterisk 13. Beberapa pertukaran datang dengan antarmuka web FreePBX, beberapa dengan konsol kosong, banyak trik dan integrasi melalui dialplan.



Kami menerima data dari pertukaran telepon otomatis



Poin pertama dan utama yang perlu dipecahkan adalah mendapatkan data tentang pendaftaran rekan dan sip. Untuk melakukan ini, PBX memiliki antarmuka AGI, AMI, ARI dan konsol SSH. Untuk alasan yang jelas, saya tidak mempertimbangkan modul tambahan.



Pertama, Anda perlu mencari tahu apa agi, ami, ari ... ini.



  • AGI - penggunaan skrip dalam dialplan. Terutama digunakan untuk mengatur panggilan.
  • AMI - dapat memberikan semua informasi yang diperlukan, bekerja melalui port 5038 dengan analogi dengan Telnet. Cocok untuk kita!
  • ARI - modern, modis, JSON. Ada banyak kemungkinan, format data dapat dimengerti untuk Zabbix, tetapi bagi saya tidak ada hal utama: Anda tidak dapat mengontrol registrasi sip. Satu lagi minus, untuk rekan hanya ada dua status online / offline, meskipun ada lebih banyak negara dan berguna untuk memperhitungkannya saat mendiagnosis.
  • SSH - itu bisa melakukan segalanya, tetapi kadang-kadang itu tidak diberikan karena "pertimbangan keamanan". Pertimbangan mungkin berbeda, saya tidak akan membongkar mereka.


Namun, untuk semua kekurangannya, ARI mencakup 90% dari semua kebutuhan pemantauan.



Zabbix dan Telnet - Kekecewaanku



Saya mengenal AMI dengan baik, pada satu waktu menerapkan pelacakan kehilangan dalam percakapan dengan divisi di kantor jarak jauh, manajemen panggilan, dll. Dengan Telnet, semuanya juga sangat jelas: buka koneksi, kirim perintah dan baca jawabannya. Yang saya lakukan, tetapi hasilnya mengecewakan saya.



Telnet Zabbix tidak sama dengan di konsol Linux, itu sedikit lebih sederhana dan dipertajam oleh otentikasi login / kata sandi standar. Jika logika otorisasi berbeda, dan tidak ada permintaan untuk pasangan login / kata sandi, kesalahan terjadi. Setelah upaya yang sia-sia untuk menghindari persyaratan otorisasi, ada baiknya untuk melihat kode sumber modul Telnet.



Saya menyadari bahwa sampai ada permintaan login tradisional dengan kata sandi, saya tidak akan melangkah lebih jauh. Demi kepentingan, saya membuang semua yang terkait dengan otorisasi dari kode, membangun kembali semuanya. Bekerja! Tetapi tidak sesuai dengan persyaratan. Berpindah‚Ä¶



Kembali ke pencarian



Saya membaca kembali dokumentasi tentang ARI, melakukan tes tambahan - tidak ada registrasi sip di sini. Ada pesta, ada percakapan, ada celana, tidak ada pendaftaran. Pada titik tertentu, saya bahkan berpikir, apakah kita benar-benar memerlukan seteguk pendaftaran?



Secara kebetulan lucu, saat ini permintaan lain datang dari pengguna, dengan masalah panggilan keluar. Masalahnya adalah penangguhan sip registrasi dan diselesaikan dengan reboot modul secara teratur. 



asterisk -rx "sip reload"


Akan lebih baik menggunakan AMI di web: itu akan menyelesaikan semua masalah, saya pikir. Saya mulai menggali ke arah ini, dan secara harfiah baris pertama hasil pencarian pada dokumentasi resmi Asterisk, yang menyatakan bahwa tujuan saya memiliki opsi file /etc/asterisk/manager.conf webenabled , yang harus diatur ke YA di bagian [umum ] Setelah itu, melalui permintaan web reguler dari formulir http: // ats: 8089 / mxml? Action = SIPshowregistry kami mendapatkan semua informasi yang diperlukan.





Saat menggunakan antarmuka FreePBX, Anda tidak dapat mengaktifkan opsi ini melalui web, Anda harus mengaktifkannya melalui konsol, membuat perubahan pada file manager.conf. FreePBX tidak menghapusnya saat perubahan konfigurasi dilakukan melalui web.
Berapa banyak yang bekerja dengan berbagai macam integrasi Asterisk, saya tidak pernah melihat bahwa fungsi ini disebutkan di suatu tempat. Saya terkejut bahwa tidak ada yang menggambarkan metode berinteraksi dengan pertukaran telepon ini. Bahkan sangat berguna untuk mencari informasi tentang topik ini: praktis tidak ada atau digunakan untuk tugas yang sama sekali berbeda.



WEB AMI - binatang seperti apa?



Menambahkan opsi manajer file webenabled.conf Menawarkan akses penuh ke manajemen melalui ASCT web. Semua perintah yang tersedia melalui AMI biasa sekarang ada di web, Anda dapat mendengarkan acara dari PBX melalui soket. Prinsip operasi tidak berbeda dari konsol AMI. Setelah mengaktifkan opsi ini, Anda dapat menghubungi PBX di alamat berikut: https: // ats: 8089 / manager - halaman web dengan antarmuka sederhana untuk pengujian dan pengiriman permintaan secara manual. Semua jawaban diformat dalam bentuk HTML yang dapat dibaca. Untuk pemantauan sangat tidak cocok. https: // ats: 8089 / rawman - hanya keluaran teks, formatnya mirip dengan konsol AMI https: // ats: 8089 / mxml - hanya keluaran teks, dalam format XML.
