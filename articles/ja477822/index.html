その新機能は、Habréを含め、すでに繰り返し説明されています。これらは、矢印関数、型付きクラスのプロパティ、およびはるかに構文上の砂糖です。しかし何よりも、パフォーマンスのために新しいリリースを待っていました。バージョン7.4では、プリロードが表示されただけでなく、PHP自体がはるかに高速になりました。 悪い（または良い？）ニュース-PHP 7.4のリリースに伴い、PHP 7.2のアクティブなサポートは終了します。彼の最新リリースは12月中旬に予定されています。私たちは長い間PHP 7.4の実験を行ってきましたが、最近はサポートされていないバージョン7.2を使用しているため、PHP 7.4への移行に積極的に取り組んでいます。



待望のリリース、おめでとうございます！以下では、新しいバージョンにアップグレードする方法について少し説明します。

私たちは巨大なコードベースを持っているという事実にもかかわらず、PHPに13年間住んでいます。新しいバージョンに繰り返しアップグレードする必要があり、移行プロセスは十分に確立されています。 

大幅に簡略化すると、いくつかのステップを区別できます。

  • ユニットテストが新しいバージョンで問題なく通過し始めたことを確認します。
  • 新しいバージョンでのテスト実行をすべてのコード変更に必須にします（新しいコードは常に記述されているため、ステップ1を繰り返す必要がなく、互換性がない可能性があります）。
  • 新しいバージョンの開発プラットフォームに切り替えて問題を修正し、しばらくの間この状態で生活します。
  • ステージングのためにこれを繰り返します。
  • 異なる生産クラスターにスムーズに展開します。

PHPリポジトリでの編集


プリロードの問題（修正）


今回は、プロセスの一部が変更されました。プリロードを待っていたため、バージョン7.4.0beta1の7月に作業の一部を開始しました。その結果、PHP 7.4は完全に未加工だったため、デバッグにかなりの時間がかかっていました。しかし、その一方で、その結果、不愉快なバグを見つけて修正し、アップストリームに送りました。これは、コミュニティ全体に役立ちました。

次に、テストを実行します。

プライベートプロパティへのアクセスに関する問題（修正）


一般的にテストを実行するには、PHPUnit、SoftMocks、およびPHP-Parserをそれらの一部として更新する必要があります。私たちは大規模なコードベースを持っており、PHPUnitの更新でさえ、膨大な数のテストを書き直す必要がありました。

なんとかテストを実行した後、非常に奇妙なことがわかりました。次のエラーで多くのクラッシュが発生しました：

PHP Fatal error: Cannot access private property ClassLoader::$classMap in vendor/composer/ClassLoader.php

クラスのプライベートプロパティへのアクセスはクラス内でのみ実行されますが、PHPはエラーを報告します。呼び出しが別のクラスから来たかのようにプライベートプロパティにアクセスすることはできません。

不安定に再現されていたため、問題は複雑でした。 gdbを使用した長いデバッグでは、何らかの理由で、EG（fake_scope）には、プロパティにアクセスするクラスがなく、それに関連しないクラスがあることがわかりました。

最小限の再現ケース（オートローダーとリフレクションの3つのクラスが必要な場合があります）を見つけ、問題の原因を修正して上流で修正を開始したところ、この問題はPHP 7.3以降に存在していることが判明しました（おそらく、この後 変更）、全世界が1年間彼女と一緒に住んでいて、彼女は私たちの前に誰も気にしませんでした。

Badooコードの非互換性の修正


現在、コードとPHP 7.4の非互換性をすべて修正しています。私たちの非互換性の大部分（100を超える場所、すべての非互換性の80％以上）は、「null / bool / int型の値の配列オフセットにアクセスしようとしています」エラー（RFCに対応）の追加が原因です。他のデータ型の配列要素にアクセスする構文を使用すると発生します。 

次の例は、問題をよく示しています。

$a = false;
var_dump($a[‘somekey’]);

// PHP 7.3: 
// NULL
// 
// PHP 7.4:
// Notice: Trying to access array offset on value of type bool in Command line code on line 1
// NULL

実際にはこれは実際のコードでは発生しないはずですが、実際に示されているように、これはかなり一般的なケースです。たとえば、関数は通常の場合は配列を返し、エラーの場合はfalse / nullを返し、さらにfalse / nullに関するスタック情報を返すことができます。が失われ、このケースは個別に処理されません。 

これはやや弱い伝播ですが、PHPでの有用な変更です。コード内の多くの潜在的なエラーを見つけることができます。

持ち上がった問題に関する2番目の更新は、method_exists（）の動作方法の変更です。ちなみに、現時点では、リリースノートやアップグレードガイドにそれに関する情報はありません。その本質は次のとおりです。


class A1 { 
    private function priv() {} 
} 

class B1 extends A1 {} 

var_dump(method_exists(B1::class, 'priv'));

// PHP 7.3: bool(true)
// PHP 7.4: bool(false)

この機能も、実際のコードでは簡単に見つけることができません。しかし、結局のところ、私たちはこれを意図せず積極的にテストに利用しました。

もちろん、リフレクションに関連する多数の変更（例1、例2）、hexdec（）などへの変更、ArrayAccessオブジェクトの場合でもarray_key_exists（）の禁止など、他の非互換性とさまざまな程度で直面しています。 Composerを介して接続されたさまざまな依存ライブラリの非互換性、および必須になるなどのあらゆる種類のエキゾチックなものとの非互換性includeのストリームラッパーのstream_set_option（）。しかし、要するに、これらすべての変更に適応するためのコストは、非配列で配列の構文を使用する場合と比較できません。

現時点では、ユニットテストの処理が完了しています。PHP7.4では完全に合格しています。APIテストに取り組んでおり、年末までに異なるクラスターと環境間の切り替えを開始する予定です。
