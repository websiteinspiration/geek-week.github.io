3. RPC over RabbitMQ


RabbitMQは RPCの編成によく使用されますが、同様のMQシステムと同様に追加のオーバーヘッドが発生するため、その使用はあまり生産的ではありません。

RPCに「キュー」を使用する場合は、チャネルをクリーンアップする必要があります。ワーカーがしばらく落ち込んだ場合、再起動後、クライアントはこの間ずっとリクエストを送信し、さらには応答を無駄に待っていたため、無関係なタスクが大量に発生する可能性があります。労働者はアクティブではありませんでした。全体として、クライアントが以前に去ったとしても、クライアントと同じようにワーカーはタスクを受け取ります。クライアントのチャネルがカウントされない場合、RabbitMQでクライアントチャネルを閉じることは可能ですが、同時にパフォーマンスが劇的に低下します。

また、豚肉労働者が生きているかどうかを知る必要もあります。
さらに、RPCシステムではデータがワーカーに送信されるだけで、その逆の場合、リソースはチャネルの操作に費やされます。

2）「calc / sum」タスクのワーカーを登録し、タスクを待ちます。リクエストタイプ「get」、つまり -タスクを取得します。
$ curl localhost/calc/sum -H 'type: get'

3）クライアントはRPC計算/合計リクエストを行います：
$ curl localhost/calc/sum -d '{"id": 15, "data": "2+3"}'

4）ワーカーはタスク `{" id "：15、" data "：" 2 + 3 "}`を受け取ります-データは変更されていないので、結果を同じID、リクエストタイプ "result"に送信する必要があります：
$ curl localhost -H 'type: result' -d '{"id": 15, "result": 5}'

...そしてクライアントはそのまま結果を取得します `{"id": 15, "result": 5}`

5.1。Jsonrpc


JsonRPC 2はサポートされていませんが、いくつかの基本的な要素があります。たとえば、クライアントは（url / rpc / call）のようなリクエストを送信できます。
{"jsonrpc": "2.0", "method": "calc/sum", "params": [42, 23], "id": 1}
次のようなエラーを受け入れます。
{"jsonrpc": "2.0", "error": {"code": -32601, "message": "Method not found"}, "id": null}
ただし、需要があれば、JsonRPCサポートを改善できます。

5.2。Pythonクライアントとワーカーの例


# client.py
import requests
print(requests.post('http://127.0.0.1:8001/test/command', json={'id': 1, 'params': 'Hello'}).json())

# worker.py
import requests
while True:
    req = requests.post('http://127.0.0.1:8001/test/command', headers={'type': 'get'}).json()
    response = {
        'id': req['id'],
        'result': req['params'] + ' world!'
    }
    requests.post('http://127.0.0.1:8001/', json=response, headers={'type': 'result'})

そしてここでは、より生産的でコンパクトな「ワーカーモード」の例を見つけることができます。
