DevOpsのKubernetesを予約する

こんにちは、habrozhiteli！ Kubernetesは、現代のクラウドエコシステムの重要な要素の1つです。このテクノロジーは、コンテナー仮想化の信頼性、スケーラビリティー、および回復力を提供します。 John ArundelとJustin DomingusがKubernetesエコシステムについて話し、日常の問題に対する実証済みのソリューションを紹介します。ステップバイステップで、独自のクラウドベースのアプリケーションを構築し、それをサポートするインフラストラクチャを作成し、開発環境と継続的デプロイメントパイプラインを構成します。これは、次のアプリケーションで作業するときに役立ちます。

•コンテナーとKubernetesをゼロから始めましょう。このトピックを学ぶのに特別な経験は必要ありません。•独自のクラスターを起動するか、Amazon、GoogleなどのマネージドKubernetesサービスを選択します。•Kubernetesを使用して、コンテナーのライフサイクルとリソースの消費を管理します。•クラスターを最適化して、コスト、パフォーマンス、回復力、電力、およびスケーラビリティを実現します。•アプリケーションの開発、テスト、および展開に最適なツールを学びます。•安全と制御のために、現在の業界慣行を活用してください。•開発チームがより柔軟、迅速かつ効率的になるように、DevOps原則を会社に実装します。

誰のための本ですか？


この本は、サーバー、アプリケーション、サービスを担当する管理部門の従業員だけでなく、新しいクラウドサービスの構築や既存のアプリケーションのKubernetesとクラウドへの移行に関わる開発者にも最も関連しています。心配する必要はありません。Kubernetesとコンテナの操作方法を知っている必要はありません。すべてをお教えします。

経験豊富なKubernetesユーザーも多くの価値を見出します。RBAC、継続的なデプロイ、機密データの管理、観察可能性などのトピックについて詳しく説明します。あなたのスキルや経験に関係なく、この本のページがきっとあなたにとって興味深いものになることを願っています。

抜粋。構成オブジェクトと秘密オブジェクト


Kubernetesアプリケーションのロジックをその構成から（つまり、時間とともに変化する可能性のある値や設定から）分離する機能は非常に役立ちます。構成値には通常、特定の環境の設定、サードパーティサービスのDNSアドレス、認証用の認証情報が含まれます。

もちろん、これらすべてをコードに直接配置できますが、このアプローチは十分に柔軟ではありません。たとえば、構成値を変更するには、コードを再アセンブルしてデプロイする必要があります。より良い解決策は、構成をコードから分離し、ファイルまたは環境変数から読み取ることです。

Kubernetesは、構成を管理するいくつかの異なる方法を提供します。最初に、ポッドシェルの仕様で指定された環境変数を介してアプリケーションに値を転送できます（192ページの「環境変数」のセクションを参照）。次に、ConfigMapオブジェクトとSecretオブジェクトを使用して、構成データをKubernetesに直接保存できます。

この章では、これらのオブジェクトを詳細に検討し、デモアプリケーションを使用して構成と機密データを管理するためのいくつかの実際的なアプローチを検討します。

構成変更時のポッドシェルの更新


クラスターにデプロイがあり、ConfigMapのいくつかの値を変更したいとします。Helmチャートを使用する場合（「Helm：Kubernetesのパッケージマネージャー」セクション（102ページ）を参照）、構成の変更を検出し、ポッドシェルを1つの洗練されたトリックで再起動できます。デプロイメント仕様に次の注釈を追加します。

checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") .
       | sha256sum }}

これで、配置テンプレートに構成パラメーターのチェックサムが含まれます。パラメーターが変更されると、合計が更新されます。helm upgradeコマンドを実行すると、Helmはデプロイメント仕様が変更されたことを検出し、すべてのポッドシェルを再起動します。

Kubernetesの機密データ


ConfigMapオブジェクトが、クラスター内の構成データを格納およびアクセスするための柔軟なメカニズムを提供することはすでに知っています。ただし、ほとんどのアプリケーションには、パスワードやAPIキーなど、機密情報が含まれています。 ConfigMapに保存することもできますが、そのようなソリューションは理想的ではありません。

代わりに、Kubernetesは機密データを格納するために設計された特別なタイプのオブジェクト、シークレットを提供します。次に、このオブジェクトをデモアプリケーションに適用する方法の例を検討します。

まず、SecretオブジェクトのKubernetesマニフェストを確認します（hello-secret-env / k8s / secret.yamlを参照）：

apiVersion: v1
kind: Secret
metadata:
    name: demo-secret
stringData:
    magicWord: xyzzy

この例では、magicWord秘密鍵はxyzzy（en.wikipedia.org/wiki/Xyzzy_(computing））です。xyzzyという単語は、一般的にコンピュータの世界では非常に役立ちます。ConfigMapと同様に、多くのキーと値をシークレットオブジェクトに配置できます。ここでは、簡単にするために、キーと値のペアを1つだけ使用します。

シークレットオブジェクトを環境変数として使用する


ConfigMapと同様に、Secretオブジェクトは、環境変数またはそのディスク上のファイルとしてコンテナーで使用可能にすることができます。次の例では、環境変数にシークレットの値を割り当てます。

spec:
   containers:
       - name: demo
          image: cloudnatived/demo:hello-secret-env
          ports:
             - containerPort: 8888
          env:
             - name: GREETING
               valueFrom:
               secretKeyRef:
                  name: demo-secret
                  key: magicWord


デモリポジトリで次のコマンドを実行して、マニフェストを適用します。

kubectl apply -f hello-secret-env/k8s/
deployment.extensions "demo" configured
secret "demo-secret" created

前と同じように、ローカルポートをデプロイメントにリダイレクトして、ブラウザで結果を確認します。

kubectl port-forward deploy/demo 9999:8888
Forwarding from 127.0.0.1:9999 -> 8888
Forwarding from [::1]:9999 -> 8888

localhost アドレスを開くと、9999 /と表示されます。

The magic word is "xyzzy"

シークレットオブジェクトをファイルに書き込む


この例では、Secretオブジェクトをファイルとしてコンテナに添付します。コードは、デモリポジトリのhello-secret-fileフォルダーにあります。

シークレットをファイルとして接続するには、次のデプロイメントを使用します。

spec:
   containers:
       - name: demo
          image: cloudnatived/demo:hello-secret-file
          ports:
              - containerPort: 8888
          volumeMounts:
              - name: demo-secret-volume
                mountPath: "/secrets/"
                readOnly: true
   volumes:
      - name: demo-secret-volume
        secret:
           secretName: demo-secret

p。の「ConfigMapオブジェクトからの構成ファイルの作成」のサブセクションと同様です。240では、ボリューム（この場合はdemo-secret-volume）を作成し、volumeMounts仕様セクションのコンテナーに接続します。MountPathは/シークレットであるため、Kubernetesはこのフォルダーのシークレットオブジェクトで定義されたキーと値のペアごとに1つのファイルを作成します。

この例では、magicWordと呼ばれるキーと値のペアを1つだけ定義したため、マニフェストは読み取り専用のコンテナー内に機密データを含む単一の/ secrets / magicWordファイルを作成します。

前の例と同じ方法でこのマニフェストを適用すると、同じ結果が得られるはずです。

The magic word is "xyzzy"

秘密のオブジェクトを読む


前のセクションでは、kubectl describeコマンドを使用してConfigMapの内容を表示しました。シークレットでも同じことができますか？

kubectl describe secret/demo-secret
Name:          demo-secret

Namespace:      default
Labels:             <none>
Annotations:
Type:               Opaque

Data
====
magicWord: 5   bytes

データ自体は表示されませんのでご注意ください。Kubernetesの秘密オブジェクトのタイプは不透明です。つまり、それらのコンテンツはkubectl describe、ジャーナルエントリ、およびターミナルの出力に表示されないため、誤って機密情報を開示することはできません。

YAML形式で機密データのエンコードされたバージョンを表示するには、kubectl getコマンドを使用します。

kubectl get secret/demo-secret -o yaml
apiVersion: v1
data:
   magicWord: eHl6enk=
kind: Secret
metadata:
...
type: Opaque

base64


eHl6enk =とは何ですか？これは元の値とは完全に異なりますか？これは実際には秘密オブジェクトであり、base64エンコーディングで表されます。 Base64は、文字列としての任意のバイナリデータのエンコードスキームです。

機密情報はバイナリであり、出力にアクセスできない場合があるため（たとえば、TLS暗号化キーの場合）、シークレットオブジェクトは常にbase64形式で格納されます。

テキストbeHl6enk =は、秘密の単語xyzzyのbase64エンコードバージョンです。ターミナルでbase64 --decodeコマンドを実行すると、これを確認できます。

echo "eHl6enk=" | base64 --decode
xyzzy

したがって、Kubernetesがターミナルまたはログファイルに誤って機密データを表示しないように保護しているにもかかわらず、特定の名前空間のシークレットオブジェクトを読み取る権限がある場合、このデータをbase64形式で受信し、その後デコードすることができます。

テキストをbase64でエンコードする必要がある場合（たとえば、それをシークレットに配置する場合）は、引数なしでbase64コマンドを使用します。

echo xyzzy | base64
eHl6enkK

シークレットオブジェクトへのアクセス


誰がシークレットオブジェクトを読み取って編集できますか？これは、アクセス制御メカニズムであるRBACによって決定されます（258ページの「役割ベースのアクセス制御の概要」で詳しく説明します）。RBACシステムが存在しないか、有効になっていないクラスターを使用する場合、すべてのシークレットオブジェクトは、すべてのユーザーとコンテナーにアクセスできます（RBACがない産業クラスターがないことについては後で説明します）。

パッシブデータ暗号化


Kubernetesがすべての情報を保存しているetcdデータベースにアクセスできる人はどうですか？ APIを介してシークレットオブジェクトを読み取る権限なしに機密データを読み取ることはできますか？

バージョン1.7以降、Kubernetesはパッシブデータ暗号化をサポートしています。つまり、etcd内の機密情報は暗号化された形式でディスクに保存され、データベースに直接アクセスできるユーザーでも読み取ることができません。暗号化を解除するには、Kubernetes APIサーバーのみが持つキーが必要です。適切に構成されたクラスターでは、パッシブ暗号化を有効にする必要があります。

これを実行して、パッシブ暗号化がクラスターで機能するかどうかを確認します。

kubectl describe pod -n kube-system -l component=kube-apiserver |grep encryption
        --experimental-encryption-provider-config=...

experimental-encryption-provider-configフラグが表示されない場合、パッシブ暗号化は有効になっていません。Google Kubernetes Engineまたは他のKubernetes管理サービスを使用する場合、データは別のメカニズムを使用して暗号化されるため、フラグは存在しません。Kubernetesプロバイダーに問い合わせて、etcdのコンテンツが暗号化されているかどうかを確認してください。

機密データの保管


クラスタから削除してはならないKubernetesリソースがあります。たとえば、特に重要なシークレットオブジェクトです。Helm Managerによって提供されるアノテーションを使用して、リソースが削除されないように保護できます。

kind: Secret
metadata:
    annotations:
        "helm.sh/resource-policy": keep

秘密オブジェクト管理戦略


前のセクションの例では、機密データは、クラスターに格納された直後に不正アクセスから保護されていました。ただし、マニフェストファイルでは、プレーンテキストで格納されていました。

バージョン管理システムにあるファイルに機密情報を含めないでください。この情報をKubernetesクラスターに適用する前に安全に管理および保存するにはどうすればよいですか？
