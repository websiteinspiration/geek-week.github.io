Sailfish OS用のContinent-AP VPNプラグインの開発

前書き


Security Code社のモバイルプラットフォーム向けセキュリティ製品の開発およびテスト部門でプログラマーとして働いています。モバイル開発チームは、IOSとAndroidで既に正常に動作していたサブスクライバーステーションのContinent-APクロスプラットフォームライブラリの移植を任されました。主な問題は、Sailfish OSがAndroidやIOSほど文書化されていないことでしたが、ドキュメントを共有してくれたOpen Mobile Platformsの皆さんに感謝します。

Sailfish OS VPNアーキテクチャ


Sailfish OSのすべてのネットワーク接続について、ConnManサービスが責任を負います。

  • Wi-Fiネットワークとセルラーネットワークのスキャンと接続。
  • 共有接続（Wi-Fiホットスポット）;
  • 翻訳、飛行モードへの出力接続。
  • VPN接続管理。

最後のポイントについてもう少しお話します。ConnManには、いくつかのタイプの定義済みVPNプラグインがあります。接続を作成および構成するためのQMLウィジェットがファームウェアに統合されています。

VPNプラグイン


ConnMan配布パッケージに含まれていないすべてのサードパーティプラグインはライブラリを表し、パッケージマネージャーを介してインストールする場合は/ usr / lib / connman / plugins-vpn /に配置する必要があります。

プラグインは、バイナリファイルを実行するユーザーを指定し、その権限を規定する構成ファイルを持つことができます。ファイルはシステムのパス/etc/connman/vpn-plugin/continent.confに沿って配置されている必要があり、その名前はプロバイダーの名前（この場合はcontinent.conf）に対応している必要があります。

たとえば、ファイルの内容：

[VPN Binary]
User = nemo
Group = vpn
SupplementaryGroups = inet,net_admin

ConnManのプラグインcontinent-proto-plugin.soは、CONNMAN_PLUGIN_DEFINEマクロ（名前、説明、バージョン、init、exit）を使用して登録されます。この例では、マクロ呼び出しは次のようになります。

CONNMAN_PLUGIN_DEFINE(continent,
                "continent VPN plugin",
                CONNMAN_VERSION,
                CONNMAN_PLUGIN_PRIORITY_DEFAULT,
                continent_init,
                continent_exit);

名前（大陸）引数には引用符を付けないでください。continent_init、continent_exit関数は、RPMのインストール中にsystemctl restart connmanが呼び出されたときなど、プラグインがロードおよびアンロードされたときに呼び出されます。continent_init関数は、vpn_register関数とconnman_dbus_get_connection関数を呼び出します。

vpn_register(name, driver, binary_path)

name-登録されているプロバイダーの名前。この場合は「大陸」です。
driver-たとえば、DBusを介してプラグインにアクセスするときのコールバック関数へのポインターを含むstruct vpn_driver構造。
binary_path-バイナリファイルへのパス。この場合、「/ usr / sbin / continent」です。

connman_dbus_get_connection関数を使用すると、確立されたDBus接続、DBusConnection *接続を取得できます。

continent_exit関数は、ConnManでプラグインを登録解除し、DBus接続を閉じるために必要です。

VPNプロバイダーインスタンスは、DBusがnet.connman.vpn.Manager.Createメソッドを呼び出すときに作成され、/ var / lib / connman / provider _ $ {Hostasket_ {VPN.Domain}ディレクトリにそのインスタンスの設定ファイルが自動的に作成されます。プロバイダーは、net.connman.vpn.Manager.Removeを呼び出すことによって削除されます。 net.connman.vpn.Connection.Connectメソッドが呼び出されると、設定は、作成されたstruct vpn_provider *プロバイダーにロードされます。

また、struct vpn_driverのいくつかの関数についてお話ししたいと思います。それらのいくつかは実装に必要です。

connect-net.connman.vpn.Connection.ConnectがDBusを介してDBusオブジェクトの対応するアドレスで呼び出されたときに呼び出されるコールバックであり、シグネチャがあります。

static int continent_connect(
			struct vpn_provider *provider, 
			struct connman_task *task,
                	const char *if_name, vpn_provider_connect_cb_t cb,
                	const char *dbus_sender, void *user_data)

このコールバックの2番目の引数はstruct connman_task * taskです。バイナリファイルを実行しますが、たとえばサーバーのホストとポートなど、開始する前に引数を渡す必要があります。

connman_task_add_argument(task, "--host", value);
connman_task_add_argument(task, "--port", value);

プロバイダーオブジェクトインスタンスの構成ファイルにいくつかのパラメーターを格納します。これは上記で説明されており、たとえば、次のようにvpn_provider_get_string関数を呼び出して取得します。
char * value = vpn_provider_get_string(provider , “Host”)

ここで、プロバイダーはstruct vpn_providerのインスタンスです。

connman_task_add_argument(task, "--dev-name", if_name).

上記の行は、ConnMan-vpndが初期化し、VPNプロバイダーの現在のインスタンスに対して発生したTUNインターフェイスインスタンスからのIPパケットの読み取りと書き込みを提供する仮想インターフェイスの名前を示しています。バックグラウンドプロセスでは、デバイスを開いて読み書き用のファイル記述子を取得する必要があります。

短いメモ：プラグインの開発中に、TUNインターフェースがデフォルトのルートインターフェースとして生成されることが判明しました。

connman_task_add_argument(task, "--dbus-busname", dbus_bus_get_unique_name(connection));
connman_task_add_argument(task, "--dbus-interface", CONNMAN_TASK_INTERFACE);
connman_task_add_argument(task, "--dbus-path", connman_task_get_path(task));

バックグラウンドアプリケーションとVPNプラグイン間のフィードバックのために、ConnManTask DBusアドレスと現在のインスタンスへのパスを渡します。これには、初期化関数でconnman_dbus_get_connectionを呼び出す必要がありました。

バックグラウンドプロセスを開始します。

err = connman_task_run(task, continent_died, data, &data->stdin_fd, NULL, NULL);

continent_died-バックグラウンドプロセスが終了したときに呼び出されるコールバック。その中で、プロセスを終了するためのエラーコードを見つけ、メモリを展開し、追加されたルートを削除します。

notify-net.connman.Task.notifyがDBusを介して呼び出されたときに呼び出されるコールバックで、実行中のバックグラウンドアプリケーションからDBusメッセージを受信します。主なものは、ネットワークパラメータの送信です。TUNインターフェースのアドレス、仮想ネットワーク内のDNSサーバーなどです。ネットワークパラメーターは、辞書の形でDBusMessageにパックされ、ConnManタスクに転送されます。ConnManタスクでは、バックグラウンドアプリケーションの起動時にDbusパラメーターが送信されます。

通知機能でTUNインターフェースを初期化する例：

struct connman_ipaddress * ipaddress = connman_ipaddress_alloc(AF_INET);
connman_ipaddress_set_ipv4(ipaddress, address, netmask, remote_server_ip);
connman_ipaddress_set_peer(ipaddress, peer);
vpn_provider_set_ipaddress(provider, ipaddress);
vpn_provider_set_nameservers(provider, “8.8.8.8”);
return VPN_STATE_CONNECT;

たとえば、プロパティを変更するときにUIがnet.connman.vpn.Connection.GetPropertiesメソッドから学習できる現在の接続のプロパティに書き込むことによって、イベントのUIに通知する場合は、中間値も渡します。たとえば、プロパティを変更すると、ConnManはDBus-signal PropertyChangedを送信します。例：vpn_provider_set_string （プロバイダー、キー、値）。

disconnect-net.connman.vpn.Connection.DisconnectがDBus経由で呼び出されたときに呼び出されるコールバック
停止プロセスにはSIGTERMシグナルが送信され、3秒以内に切断が発生しない場合はSIGKILLシグナルが送信されます。

VPNプラグインの開発とデバッグ


Continent-AP VPNプラグインとそのコンポーネントのアセンブリは、Sailfish SDKの一部であるSailfish Build Engine OS仮想マシン（Virtual Box）で実行されます。プラグインをビルドするには、ライブラリが必要です：connman-devel、dbus-1、glibs-2.0。これらはssh経由でログインしてインストールします。

ssh -p 2222 -i ~/SailfishOS/vmshare/ssh/private_keys/engine/mersdk mersdk@localhost

クロスコンパイル用のツールキットであるsb2ユーティリティ（Scratchbox 2）を使用します。i486およびarmv7hlプラットフォームに必要なパッケージをインストールします。

sb2 -t SailfishOS-3.0.1.11-i486 -m sdk-install -R zypper -n in cmake patchelf chrpath connman-devel systemd-compat-libs systemd-devel 

sb2 -t SailfishOS-3.0.1.11-armv7hl -m sdk-install -R zypper -n in cmake patchelf chrpath connman-devel systemd-compat-libs systemd-devel

systemd-compat-libsとsystemd-develは、sd_journal_print関数を使用してシステムログに出力するために必要です。プロジェクトではCmakeを使用しているため、Cmakeをインストールします。これにより、さまざまなプラットフォームでのアセンブリが大幅に簡略化されます。

VPNプラグインとそのコンポーネントのアセンブリをsb2 sdk-buildから開始します。

sb2 -t SailfishOS-3.0.1.11-armv7hl -m sdk-build cmake . && make
sb2 -t SailfishOS-3.0.1.11-i486 -m sdk-build cmake . && make


次に、収集したバイナリファイルとライブラリをUIプロジェクトに配置します。UIプロジェクトには、Continent-AP SailfishディストリビューションRPMパッケージを生成するためのSPECファイルが含まれており、SPECファイルのデバイスのシステムフォルダーにファイルをインストールするセクションを少し調整します。次に例を示します。

%files
%defattr(-,root,root,-)
%{_sbindir}/continent
%{_libdir}/connman/plugins-vpn/continent-proto-plugin.so

プラグインは、Sailfish SDKの一部であるエミュレーターのUIとは別に開発されました。定数ログ出力モードで-fオプションを指定したgdbusおよびjournalctlは、デバッグツールとして非常に役立ちました。

たとえば、gdbusを使用してプロバイダーのインスタンスを作成します。

gdbus call --system --dest=net.connman.vpn --object-path /  --method net.connman.vpn.Manager.Create "{
'Type': <'continent'>, 
…	
}"

テストデバイスは、INOI R7モデル（電話）、INOI T8（タブレット）、およびVirtualBoxベースのエミュレーターでした。
