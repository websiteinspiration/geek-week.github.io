Mostruários de dados do DATA VAULT

Em artigos anteriores , introduzimos o básico do DATA VAULT, estendendo o DATA VAULT para uma análise de estado mais adequada e criando o BUSINESS DATA VAULT. É hora de concluir a série com o terceiro artigo.

Como anunciei em uma publicação anterior , este artigo se concentrará no tópico de BI, ou melhor, na preparação do DATA VAULT como fonte de dados para BI. Considere como criar tabelas e dimensões de fatos e, assim, criar um esquema em estrela.

Quando comecei a estudar materiais em inglês sobre o tema criação de data marts sobre o DATA VAULT, senti uma complexidade suficiente do processo. Como os artigos têm um volume impressionante, há referências a alterações nas formulações que apareceram na metodologia do Data Vault 2.0, a importância dessas formulações é indicada.

No entanto, tendo aprofundado a tradução, ficou claro que esse processo não é tão complicado. Mas talvez você tenha uma opinião diferente.

E então, vamos direto ao ponto.

Tabelas de dimensões e fatos no DATA VAULT


As informações mais difíceis de entender:

  • As tabelas de medição são construídas com informações de hubs e seus satélites;
  • As tabelas de fatos são construídas com as informações dos links e seus satélites.

E isso é óbvio depois de ler um artigo sobre o básico do DATA VAULT . Os hubs armazenam chaves exclusivas de objetos de negócios, seus satélites do estado dos atributos dos objetos de negócios vinculados ao tempo, os satélites vinculados a links que suportam transações armazenam as características numéricas dessas transações.

Nesta teoria, em princípio, termina.

Mas, no entanto, na minha opinião, é necessário observar alguns conceitos que podem ser encontrados em artigos sobre a metodologia DATA VAULT:

  • Data Marts brutos - janelas de dados brutos;
  • Information Marts - Information Storefronts.

O termo “data marts brutos” - refere-se a fachadas de lojas criadas com dados do DATA VAULT executando JOINs bastante simples. A abordagem "Raw Data Marts" permite expandir de forma flexível e rápida o seu projeto de armazém com informações adequadas para análise. Essa abordagem não implica a implementação de transformações complexas de dados e a implementação de regras de negócios antes de serem colocadas em uma loja, no entanto, os dados "Raw Data Marts" devem ser compreensíveis para o usuário comercial e devem servir de base para futuras transformações, por exemplo, com ferramentas de BI.

O conceito de "Information Marts" apareceu na metodologia do Data Vault 2.0 e substituiu o antigo conceito de "Data Marts". Essa mudança se deve à conscientização da tarefa de implementar um modelo de dados para a criação de relatórios como conversão de dados em informações. O esquema “Information Marts” deve, em primeiro lugar, fornecer à empresa informações adequadas para a tomada de decisões.

Definições suficientemente detalhadas refletem dois fatos simples:

  1. Os casos de exibição “Raw Data Marts” são construídos no DATA VAULT bruto, um repositório contendo apenas conceitos básicos: HUBS, LINKS, SATÉLITES;
  2. As fachadas das lojas Information Marts são construídas usando elementos do BUSINESS VAULT: PIT, BRIDGE.

Se observarmos exemplos de armazenamento de informações sobre um funcionário, podemos dizer que uma vitrine exibindo o número de telefone atual (atual de hoje) é uma vitrine do tipo "Dados brutos". Para criar essa vitrine, são usadas a chave comercial do funcionário e a função MAX () usada no atributo da data de upload do satélite (MAX (SatLoadDate)). Quando é necessário armazenar o histórico da alteração de atributo na montra - ela é usada, é necessário entender em que data o telefone estava atualizado, a compilação da chave comercial e a data de carregamento no satélite serão adicionadas a esta tabela como chave primária, a data do campo de expiração da data de validade também será adicionada.

Criar uma vitrine que armazene as informações relevantes de cada atributo de vários satélites incluídos no hub, por exemplo, número de telefone, endereço, nome, significa usar a tabela PIT, através da qual é fácil obter todas as datas relevantes. Este tipo de caso de exibição é conhecido como "Information Marts".

Ambas as abordagens são relevantes para medidas e fatos.

Para criar vitrines que armazenam informações sobre vários links e hubs, o acesso às tabelas BRIDGE pode ser usado.
