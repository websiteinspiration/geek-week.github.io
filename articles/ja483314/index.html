Redux-Thunkを使用して非同期Reduxアクションを実行する方法

あいさつHabr！あなたに私の存在の記事の翻訳- Reduxのサンク使用して非同期Reduxのアクション、著者- Alligator.io


デフォルトでは、Reduxアクションは同期的です。これは、サーバーAPIと対話したり、他の非同期アクションを実行したりする必要があるアプリケーションにとっては問題です。さいわい、Reduxは、アクションディスパッチとリデューサーの間にあるミドルウェアなどを提供します。 Reduxには、非同期アクション用の最も人気のあるミドルウェアライブラリが2つあります。これらは、Redux ThunkとRedux Sagaです。この投稿では、最初のものを検討します。

Reduxサンクは、アクションクリエーターを呼び出して、オブジェクトではなく関数を返すことができるミドルウェアライブラリです。関数はディスパッチメソッドを引数として取るため、非同期操作が完了した後、それを使用して関数の本体内の通常の同期アクションをディスパッチします。

興味があれば、関数を使用して操作を遅らせる場合、Thunkはプログラミングの世界での概念です。

インストールとセットアップ


まず、redux-thunkパッケージをプロジェクトに追加します。

$ yarn add redux-thunk
# ,   npm:
$ npm install redux-thunk

あなたが作成するときに、ミドルウェアを追加する店舗使用してアプリケーションをapplyMiddlewareは Reduxの提供：

index.jsを

import React from 'react';
import ReactDOM from 'react-dom';
import { createStore, applyMiddleware } from 'redux';
import { Provider } from 'react-redux';
import thunk from 'redux-thunk';

import rootReducer from './reducers';
import App from './App';

//  applyMiddleware,   thunk middleware  
const store = createStore(rootReducer, applyMiddleware(thunk));

ReactDOM.render(
  <Provider store={store}>
    <App />
  </Provider>,
  document.getElementById('root')
);

主な用途


通常、Redux-Thunkは、外部APIへの非同期リクエスト、データの受信または保存に使用されます。Redux-Thunkを使用すると、外部APIリクエストの「ライフサイクル」に従うアクションを簡単にディスパッチできます。

たとえば、通常のToDoアプリケーションがあります。「todoの追加」をクリックすると、通常、最初のアクションがディスパッチされ、新しいtodoの追加の開始を報告します。次に、todo要素が正常に作成され、サーバーから返された場合、別のアクションが新しいtodo要素でディスパッチされ、操作は成功します。何らかの理由でサーバーがエラーを返す場合、新しいToDoを追加する代わりに、操作が完了しなかったというエラーのアクションが表示されます。

Redux-Thunkを使用してこれを実装する方法を見てみましょう。コンポーネントでは、アクションは通常どおりディスパッチされます。

AddTodo.js

import { connect } from 'react-redux';
import { addTodo } from '../actions';
import NewTodo from '../components/NewTodo';

const mapDispatchToProps = dispatch => {
  return {
    onAddTodo: todo => {
      dispatch(addTodo(toto));
    }
  };
};

export default connect(
  null,
  mapDispatchToProps
)(NewTodo);

アクション自体では、状況ははるかに興味深いものです。ここでは、AjaxリクエストにAxiosライブラリを使用します。インストールしていない場合は、次のように追加します。

# Yarn
$ yarn add axios

# npm
$ npm install axios --save

-私たちは、アドレスにPOSTリクエストを行いますjsonplaceholder.typicode.com/todos：
アクション/ index.js
import {
  ADD_TODO_SUCCESS,
  ADD_TODO_FAILURE,
  ADD_TODO_STARTED,
  DELETE_TODO
} from './types';

import axios from 'axios';

export const addTodo = ({ title, userId }) => {
  return dispatch => {
    dispatch(addTodoStarted());

    axios
      .post(`https://jsonplaceholder.typicode.com/todos`, {
        title,
        userId,
        completed: false
      })
      .then(res => {
        dispatch(addTodoSuccess(res.data));
      })
      .catch(err => {
        dispatch(addTodoFailure(err.message));
      });
  };
};

const addTodoSuccess = todo => ({
  type: ADD_TODO_SUCCESS,
  payload: {
    ...todo
  }
});

const addTodoStarted = () => ({
  type: ADD_TODO_STARTED
});

const addTodoFailure = error => ({
  type: ADD_TODO_FAILURE,
  payload: {
    error
  }
});

addTodoアクション作成者が、オブジェクトの通常のアクションの代わりに関数を返すことに 注目してください。この関数は、ストアからディスパッチ引数を取ります。

関数本体内では、最初に通常の同期アクションをディスパッチします。これは、外部APIを使用して新しいタスクを追加し始めたことを示しています。簡単に言うと、リクエストはサーバーに送信されました。次に、実際にAxiosを使用してサーバーにPOSTリクエストを送信します。サーバーからの肯定応答の場合、サーバーから受信したデータを使用して同期アクションをディスパッチします。ただし、サーバーからエラーが発生した場合は、エラーメッセージとともに別の同期アクションをディスパッチします。JSONPlaceholderの

ように、本当に外部（リモート）であるAPIを使用する場合私たちの場合、サーバーからの応答が来るまでに遅延があることに気付くのは簡単です。しかし、ローカルサーバーで作業している場合、答えが早すぎる可能性があるため、遅延に気付くことはありません。したがって、便宜上、開発中に人為的な遅延を追加できます

。actions/ index.js（コードの一部）

export const addTodo = ({ title, userId }) => {
  return dispatch => {
    dispatch(addTodoStarted());

    axios
      .post(ENDPOINT, {
        title,
        userId,
        completed: false
      })
      .then(res => {
        setTimeout(() => {
          dispatch(addTodoSuccess(res.data));
        }, 2500);
      })
      .catch(err => {
        dispatch(addTodoFailure(err.message));
      });
  };
};

完全を期すために、リクエストの「ライフサイクル」全体を処理するためにtodoレデューサーがどのように見えるかの例を以下に示します：

reducers / todoReducer.js

import {
  ADD_TODO_SUCCESS,
  ADD_TODO_FAILURE,
  ADD_TODO_STARTED,
  DELETE_TODO
} from '../actions/types';

const initialState = {
  loading: false,
  todos: [],
  error: null
};

export default function todosReducer(state = initialState, action) {
  switch (action.type) {
    case ADD_TODO_STARTED:
      return {
        ...state,
        loading: true
      };
    case ADD_TODO_SUCCESS:
      return {
        ...state,
        loading: false,
        error: null,
        todos: [...state.todos, action.payload]
      };
    case ADD_TODO_FAILURE:
      return {
        ...state,
        loading: false,
        error: action.payload.error
      };
    default:
      return state;
  }
}

getState


Redux-Thunkを使用して非同期アクションクリエーターによって返される関数は、2番目の引数としてgetStateメソッドも受け入れます。これにより、アクションクリエーター内で直接状態を取得できます：

actions / index.js（コードの一部）

export const addTodo = ({ title, userId }) => {
  return (dispatch, getState) => {
    dispatch(addTodoStarted());

    console.log('current state:', getState());

    // ...
  };
};

このコードを実行すると、現在の状態がコンソールに出力されます。例えば：

{loading: true, todos: Array(1), error: null}

getStateの使用は、現在の状態に応じて別の方法で対応する必要がある場合に非常に役立ちます。たとえば、todo要素の最大数を4に制限した場合、この制限を超えた場合は単に関数を終了できます：

actions / index.js（コードの一部）

export const addTodo = ({ title, userId }) => {
  return (dispatch, getState) => {
    const { todos } = getState();

    if (todos.length >= 4) return;

    dispatch(addTodoStarted());

    // ...
  };
};
