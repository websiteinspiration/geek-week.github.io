すべてのMS SQL Serverデータベースを別のマシンに転送する

最近、すべてのデータベース（SQL Serverの1つのインスタンスで50を超える）を開発環境から別のハードウェアにあるSQL Serverの別のインスタンスに転送する必要がありました。手作業を最小限に抑え、できるだけ早くすべてを実行したいと考えていました。

免責事項


スクリプトは1つの特定の状況向けに作成されています。これは開発環境であり、単純な復旧モデルのすべてのデータベース、データファイル、トランザクションログが同じヒープ上にあります。

以下に記載されている内容はすべてこの状況にのみ当てはまりますが、自分で簡単に終了できます（条件）。

スクリプトは新しいSTRING_AGGなどの優れた機能を使用しないため、SQL Server 2008（または2008 R2、バックアップ圧縮がどこに現れたか覚えていません）以降、すべてが機能するはずです。古いバージョンの場合、WITH COMPRESSIONをバックアップコマンドから削除する必要がありますが、ファイルのコピーに時間差がない場合があります。

これは指示ではありません-そのような転送の「方法」。これは、動的SQLでメタデータを使用する方法のデモです。

もちろん、最も速い方法はディスクシェルフを新しいサーバーに再接続することですが、それは私たちの選択肢ではありませんでした。デタッチ-コピー-アタッチが考慮されましたが、チャネルが非常に狭く、データベースを非圧縮形式で転送するとかなり長い時間がかかるため、適合しませんでした。

その結果、新しいサーバーのボールを圧縮してバックアップを作成し、そこに復元することにしました。古い場所と新しい場所の両方のアイアンは悪くありません。バックアップも悪くありません。時間の増加も悪くありません。

したがって、「スクリプトジェネレータ」が作成されました。

DECLARE @unc_backup_path AS varchar(max) = '\\newServer\backup_share\'
	, @local_backup_path AS varchar(max) = 'E:\Backup\'
	, @new_data_path as varchar(max) = 'D:\SQLServer\data\';

SELECT name	
	, 'BACKUP DATABASE [' + name + '] TO DISK = ''' + @unc_backup_path + name + '.bak'' WITH INIT, COPY_ONLY, STATS = 5;' AS backup_command
	, 'ALTER DATABASE [' + name + '] SET OFFLINE WITH ROLLBACK IMMEDIATE;' AS offline_command
	, 'RESTORE DATABASE [' + name + '] FROM DISK = ''' + @local_backup_path + name + '.bak'' WITH ' 
		+ (
			SELECT 'MOVE ''' + mf.name + ''' TO ''' + 
				@new_data_path + REVERSE(LEFT(REVERSE(mf.physical_name), CHARINDEX('\', REVERSE(mf.physical_name))-1)) +
				''', '
			FROM sys.master_files mf
			WHERE mf.database_id = d.database_id
			FOR XML PATH('')
		) + 'REPLACE, RECOVERY, STATS = 5;' AS restore_command
FROM sys.databases d
WHERE database_id > 4 AND state_desc = N'ONLINE';

出力では、適切な場所にバックアップを作成し、データベースをオフラインで変換するための既製のコマンドを取得します。これにより、ユーザーは古いサーバーでスクリプトを操作したり、新しいサーバーで受信したバックアップを復元したりできません（すべてのデータファイルとトランザクションログを指定した場所に自動転送します）。 ）

これの問題は、誰かが座ってすべてのスクリプトを順番に実行する必要があるか（バックアップオフラインリカバリ）、または誰かが最初にすべてのバックアップを開始してからすべてのデータベースを切断してからすべてを復元する必要があるということです-アクションは少ないですが、座っている必要がありますそして追跡。

これらすべての操作を自動化したかったのです。一方では、すべてが簡単です。すでに既製のコマンドがあり、カーソルで囲んで実行します。そして、原則として、私はそれだけを行い、古いサーバーに新しいサーバーをリンクサーバーとして追加して起動しました。ローカルサーバーでは、コマンドはEXECUTE（@sql_text）を介して実行され、リンクサーバーではEXECUTE（@sql_text）AT [linkedServerName]を介して実行されました。

したがって、操作は順次実行されました-ローカルでバックアップし、ローカルデータベースをオフラインで変換し、リンクサーバーに復元しました。すべてが元気になりましたが、バックアップと復元を互いに独立して実行すれば、プロセスを少しスピードアップできるように思えました。
次に、発明されたカーソルは2つの部分に分割されました。カーソル内の古いサーバーでは、各データベースがバックアップされてオフラインで転送されます。その後、2番目のサーバーは新しいタスクが発生したことを理解し、データベースの復元を実行する必要があります。このメカニズムを実装するために、リンクサーバーのテーブルのレコードと無限ループ（停止基準を思い付くのが面倒だった）を使用しました。これは、新しいレコードがあるかどうかを確認し、何かがある場合は何かを復元しようとしています。

決定


古いサーバーでは、グローバル一時テーブル## CommandListが作成されてデータが入力され、すべてのコマンドが収集され、バックアップのステータスを同じ場所で追跡することができます。テーブルはグローバルなので、別のセッションからいつでも、そこで何が起こっているかを確認できます。

DECLARE @unc_backup_path AS varchar(max) = 'D:\SQLServer\backup\' --       
	, @local_backup_path AS varchar(max) = 'D:\SQLServer\backup\'	--        
	, @new_data_path as varchar(max) = 'D:\SQLServer\data\';		--      ,    

SET NOCOUNT ON;

IF OBJECT_ID ('tempdb..##CommandList', 'U') IS NULL
	CREATE TABLE ##CommandList (
		dbName sysname unique			-- 
		, backup_command varchar(max)	--   
		, offline_command varchar(max)	--        
		, restore_command varchar(max)	--       
		, processed bit				-- : NULL -  , 0 -  , 1 - 
		, start_dt datetime			--  
		, finish_dt datetime			--  
		, error_msg varchar(max)		--  ,  
	);

INSERT INTO ##CommandList (dbname, backup_command, offline_command, restore_command)
SELECT name	
	, 'BACKUP DATABASE [' + name + '] TO DISK = ''' + @unc_backup_path + name + '.bak'' WITH INIT, COPY_ONLY, STATS = 5;' AS backup_command -- INIT -      
	, 'ALTER DATABASE [' + name + '] SET OFFLINE WITH ROLLBACK IMMEDIATE;' AS offline_command
	, 'RESTORE DATABASE [' + name + '] FROM DISK = ''' + @local_backup_path + name + '.bak'' WITH ' 
		+ (
			SELECT 'MOVE ''' + mf.name + ''' TO ''' + 
				@new_data_path + REVERSE(LEFT(REVERSE(mf.physical_name), CHARINDEX('\', REVERSE(mf.physical_name))-1)) +
				''', '	
			FROM sys.master_files mf
			WHERE mf.database_id = d.database_id
			FOR XML PATH('')
		) + 'REPLACE, RECOVERY, STATS = 5;' AS restore_command	
FROM sys.databases d
WHERE database_id > 4 
	AND state_desc = N'ONLINE'
	AND name NOT IN (SELECT dbname FROM ##CommandList)
	AND name <> 'Maintenance';	--  linked server -    ,   ,    "linked server"

リンクサーバーが古いサーバー上に構成され、SQL Serverの新しいインスタンスを確認しました。この投稿で提供されているスクリプトは、私が自宅で作成したもので、新しいインスタンスに煩わされることはありませんでした。スクリプトを使用して、リンクされたサーバーとして自分に接続しました。したがって、ここでは同じパスとunc-pathローカルがあります。

これで、データベースをバックアップするカーソルを宣言して、データベースを切断し、リンクサーバーに復元するコマンドを記述できます。

DECLARE @dbname AS sysname
	, @backup_cmd AS varchar(max)
	, @restore_cmd AS varchar(max)
	, @offline_cmd AS varchar(max);

DECLARE MoveDatabase CURSOR
FOR 
SELECT dbName, backup_command, offline_command, restore_command
FROM ##CommandList
WHERE processed IS NULL;

OPEN MoveDatabase;

FETCH NEXT FROM MoveDatabase INTO @dbname, @backup_cmd, @offline_cmd, @restore_cmd;

WHILE @@FETCH_STATUS = 0
	BEGIN
		--    ,  :
		--  
		--   -      
		--    ,      
		--     

		--    
		UPDATE ##CommandList
		SET start_dt = GETDATE()
		WHERE dbName = @dbname;

		BEGIN TRY
			
			RAISERROR ('  %s', 0, 1, @dbname) WITH NOWAIT; --   messages   
			
			--  
			EXEC (@backup_cmd);

			RAISERROR ('    %s', 0, 1, @dbname) WITH NOWAIT;

			--    -  linked server
			INSERT INTO [(LOCAL)].[Maintenance].[dbo].[CommandList] (dbName, restore_command)
			VALUES (@dbname, @restore_cmd);

			RAISERROR (' %s  OFFLINE', 0, 1, @dbname) WITH NOWAIT;

			--    
			EXEC (@offline_cmd);

			--  ,    
			UPDATE ##CommandList
			SET processed = 0
				, finish_dt = GETDATE()
			WHERE dbName = @dbname;

		END TRY
		BEGIN CATCH
			
			RAISERROR ('    %s.   error_msg  ##CommandList', 0, 1, @dbname) WITH NOWAIT;

			--  -   ,      
			UPDATE ##CommandList
			SET processed = 1
				, finish_dt = GETDATE()
				, error_msg = ERROR_MESSAGE();

		END CATCH

		FETCH NEXT FROM MoveDatabase INTO @dbname, @backup_cmd, @offline_cmd, @restore_cmd;
	END

CLOSE MoveDatabase;

DEALLOCATE MoveDatabase;

-- 
SELECT dbName
	, CASE processed WHEN 1 THEN '' WHEN 0 THEN '' ELSE ' ' END as Status 
	, start_dt
	, finish_dt
	, error_msg
FROM ##CommandList
ORDER BY start_dt;

DROP TABLE ##CommandList;

各アクションはSSMSの[メッセージ]タブで「ログに記録」されます。そこで、現在のアクションを確認できます。RAISERRORでWITH LOGを使用する場合、原則として、すべてを何らかのジョブに入れてからログを確認できます。

実行時に、## CommandListにアクセスして、何がどのように行われているかを表形式で確認できます。

新しいサーバーで並行して、無限ループが回転していました。


SET NOCOUNT ON;

DECLARE @dbname AS sysname
	, @restore_cmd AS varchar(max);

WHILE 1 = 1	--   ,    
BEGIN
	SELECT TOP 1 @dbname = dbName, @restore_cmd = restore_command 
	FROM CommandList
	WHERE processed IS NULL; --    ,  

	IF @dbname IS NOT NULL 
	BEGIN
		--    
		UPDATE CommandList
		SET start_dt = GETDATE()
		WHERE dbName = @dbname;

		RAISERROR('  %s', 0, 1, @dbname) WITH NOWAIT;
		
		BEGIN TRY

			--  ,  -  ,  CATCH    
			EXEC (@restore_cmd);

			--   
			UPDATE CommandList
			SET processed = 0
				, finish_dt = GETDATE()
			WHERE dbName = @dbname;

			RAISERROR(' %s  ', 0, 1, @dbname) WITH NOWAIT;

		END TRY
		BEGIN CATCH

			RAISERROR('    %s', 0, 1, @dbname) WITH NOWAIT;

			UPDATE CommandList 
			SET processed = 1
				, finish_dt = GETDATE()
				, error_msg = ERROR_MESSAGE();

		END CATCH

	END
	ELSE	--   ,    
		BEGIN

			RAISERROR('waiting', 0, 1) WITH NOWAIT;

			WAITFOR DELAY '00:00:30';

		END
		
	SET @dbname = NULL;
	SET @restore_cmd = NULL;

END

彼が行うすべてのこと-CommandListテーブルを調べ、そこに少なくとも1つの未加工レコードがある場合-データベース名と復元するコマンドを取り、EXEC（@sql_text）;を使用してそれを実行しようとします。エントリがない場合は、30秒待ってから再試行してください。

カーソルとループの両方が各レコードを1回だけ処理します。うまくいきませんでしたか？エラーメッセージをテーブルに書き込み、ここには戻りません。

停止の条件について-私は実際には怠惰でした。入力中に、少なくとも3つの解決策を思い付きました-オプションとして、「リカバリの準備ができて\リカバリの準備ができていません\完了」フラグを追加し、古いサーバーの## CommandListに入力してカーソル内のフラグを更新するときに、データベースとコマンドのリストにすぐに入力します。「リカバリの準備ができたレコード」が残っていない場合は、作業の全量がすぐにわかるため、停止します。

調査結果


しかし、結論はありません。メタデータを使用して動的SQLを作成および実行する方法を見ると、誰かにとって便利で興味深いかもしれません。投稿で提供されているスクリプトはそのままでは製品版での使用には適していませんが、ログシッピング/データベースミラーリング/可用性グループの大規模なカスタマイズなどのために、それらをわずかに完成させて使用することができます。

ボールのバックアップを実行する場合、SQL Serverを実行しているアカウントには、そこに書き込む権限が必要です。

リンクサーバーの作成は投稿（GUIのマウスは数分で直感的に設定されます）と新しいサーバーへのログインの転送では公開されていません。ユーザーの移行を経験したことのある人は、SQLログインを再作成するだけではあまり役に立たないことを知っています。これは、データベースユーザーが接続するSIDを持っているためです。現在のパスワードと正しいsidでSQLログインを生成するためのスクリプトはmsdnにあります。
