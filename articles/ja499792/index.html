負荷テストAtlassian Jira、Confluence、Bitbucketパート1

みなさん、こんにちは！

Atlassian Jira、Confluence、およびAtlassian dc-app-performance-toolkitを使用したBitbucketストレステストでの経験を共有したいと思います。

これは一連の記事になるので、ストレステストの方法論については説明しませんが、ツールを使用したスト​​レステストの技術的な側面についてのみ説明します。

アトラシアン製品のコンテキストで負荷テストの方法論について詳しく知りたい場合は、Highload 2018でこれについて話しました。

それでは、dc-app-performance-toolkitに移りましょう。

このツールを使用すると、Jira、Confluence、BitbucketなどのAtlassianアプリケーションをロードできます。 Atlassian Jiraプラグインをテストしてデータセンターで認証する必要があるときにこのツールについて学びました。設定に何時間も費やす必要がなかったので、すぐにこのツールが気に入りました。すべてが箱から出してうまくいきました。

このツールは、Taurus、jmeter、Seleniumを使用しています。

このツールは、次の目的で使用できます。

  • Atlassian Marketplaceのプラグインを開発しています。この場合、このツールを使用して、データセンターのプラグインを認証できます。
  • Atlassian Jira, Confluence Bitbucket , , , . . , . - . , Atlassian, Atlassian, , , . , .

Atlassian Jira、Confluence、およびBitbucketをテストするために実行する必要がある手順はほとんどの場合同じであるため、Atlassian Jiraですべての作業を行います。製品に特定の手順がある場合は、これについて触れます。

取り付け


最初にすることはgithubでツールを入手することです：

git clone https://github.com/atlassian/dc-app-performance-toolkit.git

そして、作成したフォルダーに移動します。

cd dc-app-performance-toolkit

次に、ツールを使用するためのすべての依存関係をインストールする必要があります。これを行う方法は、path_to_dc_app_performance_toolkit / README.mdで確認できます。

ツールを使用する前に、path_to_dc_app_performance_toolkit / docフォルダーにあるこのツールのドキュメントをお読みください。

jira.yml


負荷テストを開始する前に、テストする製品に応じて、path_to_dc_app_performance_toolkit / appフォルダーにあるjira.yml、  confluence.yml、  または  bitbucket.ymlファイルに変更を加える必要があります  。これらはおうし座構成ファイルです。おうし座については、こちらをご覧ください。 jira.ymlについてコメントします。他の製品のファイルも同じ原理で構築されています。


settings:

settingsは、Taurus構成ファイルのセクションです。おうし座の最上位オプションが含まれています。このセクションの詳細については、  こちらをご覧ください。

  artifacts-dir: results/jira/%Y-%m-%d_%H-%M-%S

Artifacts-dir-ロードテストのアーティファクトを格納するために使用されるパステンプレート。アーティファクトの部分的なリストを次に示します。

  • bzt.log -bztコマンドのログ。このコマンドは、負荷テストを開始します。
  • error_artifacts -エラーで終了した、Seleniumのスクリーンショットとhtmlテストを保存するためのフォルダー。
  • jmeter.err -JMeterログ。
  • kpi.jtl -JMeterテストデータ。
  • pytest.out -Selenumテスト実行ログ。
  • selenium.jtl -セレンテストデータ。
  • results.csv -集計されたテスト結果。
  • jira.yml -テストに使用されたjira.yml。


  aggregator: consolidator

aggregator-テスト結果を収集し、その結果をレポートモジュールに転送するために使用されるアグリゲーター。アグリゲーターについて詳しくは、こちらをご覧ください。

  verbose: false

verbose-デバッグモードでTaurusを起動するためのパラメーター。この場合、このモードをオフにします。

  env:

envでは、環境変数を設定できます。詳しくはこちらをご覧  ください。

    application_hostname: localhost   # Jira DC hostname without protocol and port e.g. test-jira.atlassian.com or localhost
    application_protocol: http      # http or https
    application_port: 2990            # 80, 443, 8080, 2990, etc
    application_postfix: /jira           # e.g. /jira in case of url like http://localhost:2990/jira
    admin_login: admin
    admin_password: admin

上記のパラメータには、テストするインスタンスに関する情報が含まれています。コンピューターにあるJiraインスタンスのパラメーターを指定しました。これらのパラメーターは、JMeter、Selenium、およびスクリプトで使用されます。

    concurrency: 200

これらのパラメーターは実行セクションに渡されます。このセクションについて説明するときに、これらのパラメーターの意味を説明します。

execution:

実行-Taurus構成ファイルのセクション。テスト中に実行されるスクリプトが含まれています。詳細については、こちらをご覧ください。

  - scenario: jmeter
    concurrency: ${concurrency}
    hold-for: ${test_duration}
    ramp-up: 3m

これらは、JMeterのスクリプト実行オプションです。詳細については、  こちらをご覧ください。

同時実行性は仮想ユーザーの数です。このパラメーターは、JMeterがシミュレートする同時ユーザーの数を意味します。私たちのケースでは、200人の同時作業ユーザーをシミュレーションします。

ランプアップ-オーバークロック時間をテストします。同時に200人の同時稼働ユーザーに移動します。私たちのケースでは、3人のユーザーに3人で同時に働きます。

hold-for-同時実行パラメータで指定された数の同時ユーザーに到達した後のテストの期間。

  - scenario: selenium
    executor: selenium
    runner: pytest
    hold-for: ${test_duration}

まとめましょう。

jira.xmlファイルには6つのセクションがあります。
設定-最上位の設定とパラメーター。
このセクションでは、次のパラメータを変更する必要があります。

application_hostname: localhost   # Jira DC hostname without protocol and port e.g. test-jira.atlassian.com or localhost
    application_protocol: http      # http or https
    application_port: 2990            # 80, 443, 8080, 2990, etc
    application_postfix: /jira           # e.g. /jira in case of url like http://localhost:2990/jira
    admin_login: admin
    admin_password: admin

次のパラメータを変更する必要がある場合もあります。

concurrency: 5
test_duration: 5m

テストをすばやく実行できるように、これらのパラメーターを小さな値に変更しました。
test_durationと同時実行性を変更した場合は、実行セクションのランプアップパラメータを確認してください。あなたもそれを変更する必要があるかもしれません。
サービス-Taurusテストのライフサイクルの段階で実行されるスクリプト。
execution-テストスクリプトの実行パラメータ。最初にJMeterを実行し、次にSeleniumを実行します。
シナリオ-スクリプトパラメータをテストします。おそらく、1時間あたりの操作数とテストの割合を変更したいと思うでしょう。次のいずれかの部分でテストを追加または削除する方法について説明します。
modules-テストに使用されるモジュール。統合機能、JMeter、Seleniumを使用しています。Taurusは、テスト中にこれらのモジュールの可用性を提供します。
レポート-レポートモジュールのパラメーター。
良い。構成ファイルを変更し、テストを実行する準備ができました。

テストを実行


最初に、Python仮想環境をアクティブにし（これに関する情報はREADME.mdファイルにあります）、appフォルダーに移動してテストを実行します。

bzt jira.yml

私の場合、テストはエラーで失敗しました：

17:03:26 WARNING: Errors for python util/data_preparation/jira/prepare-data.py:
There are no software projects in Jira

すべてが正しいです。Jiraのコピーが空です。テスト用のデータを入力する必要があります。その方法については、パート2で説明します。
