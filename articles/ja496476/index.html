Windows Server Coreを使用してWindows Serverを低電力VPSにプッシュする


Windowsシステムの大食いのため、VPSは軽量のLinuxディストリビューション（Mint、Colibri OS、Debian、Ubuntu）によって支配されています。彼らが言うように、コンソールのみ、ハードコアのみ！実際、これは決して誇張ではありません。同じDebianが256 MBのメモリと1コアのクロックサイクル1 Ghzで、つまりほとんどすべての「切り株」で起動します。快適な作業のためには、512 MBから少し高速なプロセッサーが必要です。しかし、Windows VPSでもほぼ同じことができると言ったらどうでしょうか。 3〜4ヘクタールのRAMと、クロック速度が1.4 GHzの少なくとも2つのコアを必要とする重いWindowsサーバーをロールバックする必要がないのは何ですか。 Windows Server Coreを使用するだけで、GUIといくつかのサービスを削除できます。これを行う方法については、記事で説明します。

これはあなたのWindows Server Coreですか？


Mikesの公式WebサイトにもWindows（サーバー）コアが何であるかについての明確な情報はありません。つまり、すべてが混乱しているためすぐには理解できませんが、最初の参照はWindows Server 2008の時代にさかのぼります。実際、Windowsコアは動作可能なWindowsカーネルですサーバー（突然！）、独自のGUIのサイズとサイドサービスの約半分による「失われた重量」。

Windows Coreの主な機能は、低いハードウェア要件とPowerShellによる完全なコンソール管理です。

MicrosoftのWebサイトにアクセスして技術要件を確認した場合、Windows Server 2016/2019を起動するには、2ギガのRAMと少なくとも1つのコア（クロック速度1.4 GHz）が必要です。しかし、この構成ではシステムが起動することしか期待できず、OSの快適な操作ができないことは誰もが理解しています。このため、安価な仮想マシンではなく、一部のXeonで高価な物理マシンを提供しない場合、Windows Serverは通常、より多くのメモリとプロセッサから少なくとも2コア/ 4スレッドを割り当てられます。

同時に、サーバーシステム自体のコアは512 MBのメモリしか必要とせず、単に画面にレンダリングして多数のサービスを実行し続けるためにGUIによって消費されたプロセッサリソースは、より便利なものに使用できます。

ほとんどLinuxに似ていますが、彼は違います


私は本当にWindows Server CoreとLinuxディストリビューションを比較したいのですが、実際にはこれは完全に正しくはありません。はい、これらのシステムは、GUIと多くのサイドサービスの廃止によるリソース消費の削減の点では互いに似ていますが、操作とアセンブリへのいくつかのアプローチの点では、Windowsであり、UNIXシステムではありません。

最も単純な例-Linuxカーネルの手動アセンブリとそれに続くパッケージとサービスのインストールの助けを借りて、最軽量のLinuxディストリビューションでさえ、重いナイフに似たものに変えることができます（私は本当にPythonについてジャズプレイし、シリーズ「If Programming Languagesがあったら武器」、しかし私達はしません）。 Windows Coreでは、マイクロソフト製品を扱っているため、この自由度ははるかに低くなっています。

Windows Server Coreには既製のアセンブリが付属しており、そのデフォルト構成は上の表から見積もることができます。サポートされていないもののリストから何かが必要な場合は、不足している要素をコンソールからオンラインで追加する必要があります。ただし、機能オンデマンドと、コンポーネントを.cabファイルとしてデフレートする機能を忘れないでください。これらは、インストール前にアセンブリに追加できます。ただし、カットサービスの1つが不足していることがプロセスですでに見つかっている場合、このシナリオは機能しません。

ただし、コアバージョンとフルバージョンの違いは、システムを更新し、作業を停止せずにサービスを追加できることです。 Windowsコアは、再起動せずにホットパッケージのローリングをサポートしています。結果として、実際的な観察に基づいて、Windows Coreを実行しているマシンは、Windows Serverを実行しているマシンよりも6分の1の頻度で再起動する必要があります。

管理者にとっての素晴らしいボーナスは、RDPを使用せずにコンソールを介してシステムを意図したとおりに使用し、2台目のWindowsサーバーを作成しない場合、フルバージョンと比較して非常に安全になることです。実際、Windows Serverの脆弱性のほとんどは、RDPと、この同じRDPを介して実行してはならないことを実行するユーザーのアクションに依存しています。これは、ヘンリーフォードの話と車の色との関係に似ています。「顧客は、黒である限り、好きな色で車を塗装できます。」つまり、システムとの通信です。ユーザーはシステムと任意の方法で通信できます。主なことは、コンソールを通じてこれを行うことです。

Windows Server 2019 Coreのインストールと管理


前述のとおり、Windowsコアは実際にはGUIラッパーのないWindowsサーバーです。つまり、コアバージョンとしてほとんどすべてのバージョンのWindows Serverを使用できます。つまり、GUIを放棄できます。 Windows Server 2019ファミリーの製品の場合、これは4つのサーバービルドのうち3つです。コアモードは、Windows Server 2019 Standard Edition、Windows Server 2019 Datacenter、およびHyper-V Server 2019で使用できます。つまり、Windows Server 2019 Essentialsのみがこのリストに含まれません。

同時に、Windows Server Coreのインストールパッケージを探す必要はありません。 Microsoftの標準インストーラーでは、GUIを使用して手動でバージョンを選択する必要がある場合、コアバージョンが文字どおりデフォルトで提供されます。


システムを管理するための実際のオプションは、PowerShellで言及されているものよりも多くあります。PowerShellは、メーカーによってデフォルトで提供されています。Windows Server Coreで仮想マシンを管理するには、少なくとも5つの方法があります。

  • リモートPowerShell
  • リモートサーバー管理ツール（RSAT）;
  • Windows管理センター
  • Sconfig;
  • サーバーマネージャー

最初の3つのポジションは、最も興味深いものです。標準のPowerShell、RSAT、およびWindows管理センターです。ただし、いずれかのツールの利点を得ると、そのツールによって課せられる制限も得られることを理解することが重要です。

コンソールの機能については説明しません。PowerShellはPowerShellであり、明らかな長所と短所があります。RSATとWACはもう少し複雑です。 

WACを使用すると、レジストリの編集やディスクおよびデバイスの管理などの重要なシステム制御にアクセスできます。最初のケースのRSATは表示モードでのみ機能し、変更を加えることはできません。リモートサーバー管理ツールのディスクと物理デバイスを管理するには、GUIが必要ですが、これは私たちのケースではありません。一般に、RSATはファイルを操作できないため、レジストリの編集でプログラムの更新、インストール、アンインストールを行うことができません。

WACを使用するには、150〜180メガバイトのRAMを使用する必要があることに注意してください。接続すると、Windowsアドミンセンターはサーバー側に3〜4個のセッションを作成し、ツールが仮想マシンから切断されても強制終了されません。 WACは古いバージョンのPowerShellでも動作しないため、最低限のPowerShell 5.0が必要です。これはすべて、リソースの厳格さという私たちのパラダイムに逆行しますが、快適さを犠牲にする必要があります。私たちの場合、RAMによって。

Server Coreを管理するためのもう1つのオプションは、サードパーティのGUIをインストールして、インターフェイスと一緒に完全なアセンブリに含まれる大量のゴミをドラッグしないようにすることです。

この場合、2つのオプションがあります。元のエクスプローラーをシステムにロールアウトするか、エクスプローラー++を使用します。後者の代わりに、Total Commander、FAR Manager、Double Commanderなどの任意のファイルマネージャーが適しています。メモリーの節約が重要な場合は、後者が望ましいです。ネットワークフォルダーを作成してコンソールまたはスケジューラーから起動することにより、エクスプローラー++またはその他のファイルマネージャーを追加できます。

本格的なExplorerをインストールすると、UIを備えたソフトウェアの操作に関してより多くのオプションが提供されます。このために私達は連絡しなければなりませんサーバーMMC、Eventvwr、PerfMon、Resmon、Explorer.exe、さらにはPowershell ISEに戻るサーバーコアアプリ互換性機能オンデマンド（FOD）。ただし、WACの場合と同様に、この料金を支払う必要があります。explorer.exeやその他のサービスを貪欲に消費する約150〜200メガバイトのRAMが回復不能に失われます。マシンにアクティブなユーザーがいない場合でも。



これは、ネイティブエクスプローラーパッケージがある場合とない場合のマシンでのシステムのメモリ消費量のようです。

これは論理的な問題を提起します。PowerShell、FOD、ファイルマネージャーでこれらすべてが踊るのに、左右のいずれかのステップがRAM消費の増加につながるのなら、なぜですか？ Windows Server 2016/2019を転がして白人のように暮らすことができるのに、Windows Server Coreで快適に作業できるように、たくさんのツールで身を隠し、左右に恥ずかしがり屋になっているのはなぜですか？

Server Coreを使用する理由はいくつかあります。まず、電流はメモリ消費量のほぼ半分です。覚えていれば、この状態が最初の記事の基礎でした。比較のために、Windows Server 2019のメモリ消費量を、上記のスクリーンショットと比較してください。


そのため、Coreでは655 MBではなく、1146 MBの消費メモリ。 

WACが不要で、元のエクスプローラーの代わりにエクスプローラー++を使用している場合でも、Windows Serverを実行しているすべての仮想マシンでほぼ半分のヘクタールを獲得できます。仮想マシンが1つしかない場合、増加はわずかですが、5つある場合はどうでしょうか。ここでは、特に必要がない場合は、GUIの存在が重要です。 

第2に、Windows Server Coreのダンスは、Windows Serverの操作の主要な問題であるRDPとその安全性（より正確には、完全な不在）との戦いにあなたを導きません。Windows Coreは、FOD、RSAT、WACの形式であっても、RDPのないサーバーです。つまり、既存の攻撃の95％の影響を受けません。

バランスで


一般に、Windows Coreは、標準のLinuxディストリビューションよりもわずかに「太い」だけですが、はるかに機能的です。リソースを解放する必要があり、コンソール、WAC、およびRSATで作業する準備ができている場合は、完全なGUIの代わりにファイルマネージャーを使用してください。コアに注意する価値があります。さらに、これにより、Windows全体に追加料金を支払う必要がなくなり、節約したお金をVPSの混乱に費やして、たとえばRAMを追加することができます。便宜上、Windows Server Coreをマーケットプレイスに追加しました。
