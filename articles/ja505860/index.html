abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

なぜ使用しないのですか？


データクラスは、自分自身で最終的であることに加えて、すべてのフィールドの特定のフィールドを持っているequals、hashCodeとtoString。そして、これはHibernateと組み合わせて受け入れられません。

どうして？そして、なぜhashCodeそれが常に定数に等しいのか-答えはHibernate自体のドキュメントにあります。具体的には、このセクションに関心があります。

equalsとhashCodeにはnatural-idを使用するのが最適ですが、一意の制約を提供するエンティティIDしかない場合もあります。

等価性チェックにエンティティ識別子を使用することは可能ですが、回避策が必要です。

  • エンティティがフラッシュされる前後でハッシュコード値が変化しないように、hashCodeに定数値を提供する必要があります。
  • 非一時的なエンティティについてのみ、エンティティ識別子の同等性を比較する必要があります。

つまり、で比較するかnatural id、この例のようにで比較する必要がありますprimary key id。これにより、エンティティを比較するときの多くの問題が回避され、エンティティをで要素として使用するときの損失を防ぐことができSetます。すべてのフィールドが文字列への出力用に初期化されるため、たとえば、エンティティをジャーナル化するとき、すべてのフィールドによって定義される
