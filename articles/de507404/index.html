Erstellen Sie keine eigenen Benutzerauthentifizierungslösungen

Der Autor des Artikels, dessen Übersetzung wir veröffentlichen, schlägt vor, das Schreiben Ihres eigenen Codes für die Benutzerauthentifizierung einzustellen. Er glaubt, dass es an der Zeit ist, sicherere Lösungen zu implementieren, die Zeit und Geld sparen können, auch wenn nicht über ihre anderen Vorteile gesprochen wird.



Die meisten Anwendungen erfordern eine Art Benutzerauthentifizierungssubsystem. Vielleicht sind Sie ein Programmierer, der in einem großen Unternehmen an Geschäftsanwendungen arbeitet, in denen Sie den Zugriff auf bestimmte Materialien auf nicht autorisierte Mitarbeiter des Unternehmens beschränken möchten, und Sie müssen die Berechtigungen dieser Benutzer überprüfen. Möglicherweise schreiben Sie den Code für die neue SaaS-Plattform und benötigen ihn, damit die Benutzer Konten erstellen und verwalten können.







In beiden Fällen und in vielen anderen Fällen beginnen Sie höchstwahrscheinlich mit der Arbeit an der Anwendung, indem Sie Authentifizierungssubsysteme und Benutzerverwaltungstools erstellen. Erstellen Sie mindestens ein Registrierungsformular und eine Anmeldeseite. Authentifizierungssubsysteme sind genau das, was am häufigsten von Webentwicklern verlangt wird, die an bestimmten Projekten beteiligt sind. Trotzdem werden solche Subsysteme normalerweise auch nur sehr wenig beachtet.



Die Entwicklung eines sicheren Benutzerauthentifizierungssystems ist eine echte Herausforderung. Es ist viel größer als viele Leute denken. Dieses Problem ist sehr leicht falsch zu lösen. Noch schlimmer: Fehler beim Erstellen von Authentifizierungssubsystemen können katastrophal sein. Die Grundstruktur von Authentifizierungs- und Benutzerverwaltungssystemen umfasst nur wenige Formulare. Aus diesem Grund mag die Erstellung solcher Systeme recht einfach erscheinen. Aber wie Sie wissen, steckt der Teufel im Detail. Es erfordert viel Arbeit, um solche Systeme sicher zu machen (und, wenn möglich oder sogar notwendig, die Datenschutzanforderungen personenbezogener Daten zu berücksichtigen).



IDaaS



Glücklicherweise muss ein moderner Webentwickler kein eigenes System zur Authentifizierung und Verwaltung von Benutzern bereitstellen. Jetzt, im Jahr 2020, gibt es viele gute IDaaS-Lösungen (Identität als Service, Identifikation als Service). Die Verwendung solcher Lösungen vereinfacht die Ausstattung von Webprojekten mit Benutzerauthentifizierungsfunktionen erheblich.



Hier sind einige beliebte IDaaS-Projekte (in alphabetischer Reihenfolge): Auth0 , Azure AD , Google Identity Platform und Okta .



Darüber hinaus gibt es Anbieter von Identitätsinformationen, die auf Social-Media-Benutzerkonten basieren. Dies ist zum Beispiel Apples , Facebook, GitHub , Twitter . Für Entwickler ist es sehr einfach, die Funktionen dieser Systeme in ihre Projekte einzubeziehen. Jeder, der mit Authentifizierungsanbietern zusammenarbeitet, kann sehr schnell ein geeignetes Anwendungssubsystem erstellen, das Zugriff auf eine große Menge von Benutzerdaten hat. Manchmal kann sich die Interaktion des Projekts mit Anbietern jedoch negativ auf den Datenschutz der Benutzerdaten auswirken.



Tatsächlich kann man nicht den wahren Grund nennen, warum ein bestimmter Entwickler die Dienste eines Identitätsinformationsanbieters nicht nutzen sollte. Die Arbeit mit einem solchen Dienst spart viel Zeit, die für die Entwicklung der Anwendung selbst aufgewendet werden kann. Diese Dienste bieten dem Entwickler ohne nennenswerten Aufwand viele Möglichkeiten. Und das Wichtigste dabei ist, dass sie viel sicherer sind als eine selbst geschriebene Lösung.



Je mehr Benutzer, desto sicherer die Lösung.



Die meisten Identitätsanbieter bieten zusätzliche Sicherheitsfunktionen. B. Unterstützung für die Multi-Faktor-Authentifizierung (MFA, Multi-Faktor-Authentifizierung), Unterstützung für Sicherheitszertifikate oder -schlüssel (einschließlich U2F, FIDO2, WebAuthn usw.).



Unterschätzen Sie nicht die Bedeutung dieser Technologien. Laut einem Microsoft- Bericht kann die Einbeziehung von MFA bis zu 99,9% der Angriffe auf Konten verhindern.



Bei der Verwendung von Authentifizierungsdiensten von Drittanbietern anstelle ihrer eigenen Lösungen ist jedoch ein anderer Aspekt weniger bekannt. Wir sprechen nämlich von der Tatsache, dass sie aufgrund der sehr großen Anzahl von Benutzern, mit denen diese Dienste zusammenarbeiten, die Verhaltensmuster von Angreifern überwachen und Angriffe auf Konten leichter verhindern können.



Große Anbieter sammeln aufgrund der Tatsache, dass sie Millionen von Benutzern haben, die jeden Tag unzählige Anmeldungen durchführen, genügend Daten, um Modelle zu erstellen, die auf den Fähigkeiten der künstlichen Intelligenz basieren. Solche Modelle ermöglichen eine bessere Identifizierung verdächtiger Verhaltensmuster.



Angenommen, einer Ihrer in Kanada lebenden Benutzer meldet sich von zu Hause aus an. Und nach zwei Stunden erfolgt der Zugang zu demselben Konto aus der Ukraine. Die Dienste des Identitätsanbieters können dieses Verhalten als verdächtig betrachten und entweder die Anmeldung sofort verbieten oder dem Benutzer eine zusätzliche Überprüfung anbieten (z. B. mithilfe eines MFA-Tokens). Sie können außerdem angegriffene Benutzer oder Systemadministratoren und möglicherweise beide benachrichtigen.



Häufige Einwände gegen die Verwendung von Authentifizierungsdiensten von Drittanbietern



▍ Tatsächlich ist es nicht so schwierig, ein eigenes System zur Authentifizierung und Verwaltung von Benutzern zu erstellen



Formulare für die Registrierung im Projekt und für die Eingabe sind nur eine Seite der „Münze“ des Authentifizierungssystems. Der Entwickler, der ein solches System erstellt, muss viel mehr Aufgaben lösen als die Entwicklung eines Formularpaares.



Zunächst müssen Sie zusätzliche Aufgaben im Zusammenhang mit der Benutzerverwaltung lösen. B. Anwenden von Kennwortsicherheitsrichtlinien, Erstellen eines Systems zum Bestätigen von E-Mail-Adressen oder Telefonnummern, Entwickeln von Mechanismen zum sicheren Zurücksetzen von Kennwörtern. In Bezug auf die Passwortverwaltung fordere ich jedoch alle auf, NIST anzuhören und das System nicht mit einem Nicht-Trennungsmechanismus auszustatten, dessen Arbeit darauf abzielt, sicherzustellen, dass die Passwörter nach einer bestimmten Zeit ablaufen. Es ist nicht erforderlich, Benutzer zum Erstellen zu zwingen"Kreative" Passwörter mit Groß- und Kleinbuchstaben, Sonderzeichen usw.



Beim Entwurf solcher Systeme müssen Sie viele kleine Teile berücksichtigen. Es ist jedoch sehr leicht, Fehler zu machen. Riesige Unternehmen haben festgestellt, dass sie beispielsweise keine Passwörter in ihren Datenbanken hashen (oder falsch hashen), dass sie Passwörter versehentlich in Klartextdateien in Protokolldateien ablegen und dass ihre Mechanismen zurückgesetzt werden Passwörter sind mit Social-Engineering-Methoden sehr einfach zu knacken. Diese Liste geht weiter.



Die Tatsache, dass eine ordnungsgemäße Passwortverwaltung keine leichte Aufgabe ist, sollte niemanden überraschen. Wussten Sie, dass die Verwaltung von Benutzernamen ebenfalls schwierig ist? Zum Beispiel die Tatsache, dass Benutzernamen gleich aussehenbedeutet nicht, dass das System sie im Vergleich als solche betrachtet. In dieser Rede von 2018 wurden einige weitere interessante Ideen zum Ausdruck gebracht, was bei der Arbeit mit „einfachen“ Entitäten wie Benutzernamen schief gehen könnte.



Schließlich ist anzumerken, dass Webprojekte von den zusätzlichen Sicherheitsmaßnahmen der Identitätsanbieter in hohem Maße profitieren können. Dies umfasst beispielsweise Multi-Faktor-Authentifizierung und Sicherheitstoken.



▍Die Verwendung von Benutzerauthentifizierungsdiensten ist insbesondere bei großen Projekten nicht immer kostenlos



Wissen Sie, dass das, was sonst noch mit Sicherheit zu tun hat, nicht kostenlos ist? Die Notwendigkeit, Schaden durch Hacks zu beheben. Dieser Schaden kann in monetärer Form ausgedrückt werden. Wenn diejenigen, die gehackt wurden, einige Verluste erlitten haben, kann dies die Zeit sein, die dringend für die Fertigstellung ihres eigenen Projekts aufgewendet wird. Dies ist ebenso unangenehm wie die anderen Folgen von Hacks und der Verlust des Benutzervertrauens.



Und selbst wenn wir nicht über Hacks sprechen, ist die Implementierung eines sicheren Authentifizierungssystems und dessen Verwaltung, die Arbeit mit einer Benutzerdatenbank und die Lösung anderer ähnlicher Aufgaben eine Verschwendung von Zeit und Ressourcen. Dies sind die Ressourcen, die für die Entwicklung des Systems benötigt werden, und die Ressourcen, die erforderlich sind, um das System funktionsfähig zu halten.



▍ Ich bin ein sehr erfahrener Entwickler und weiß daher, wie man ein sicheres Authentifizierungssystem erstellt



Zunächst einmal herzlichen Glückwunsch. Tatsache ist, dass viel weniger Entwickler als Sie denken und was Sie erwarten können, um echtes Wissen darüber zu haben, wie sichere Authentifizierungssysteme erstellt werden.



Wenn Sie jedoch wirklich ein erfahrener Entwickler sind, ist es durchaus möglich, dass Sie Ihre Zeit besser für die Implementierung anderer Teile der Anwendung verwenden, die den Benutzern dieser Anwendung einige greifbare Vorteile bringen.



Wenn Sie wirklich Ihr eigenes Authentifizierungssystem erstellen möchten, können Sie sich einem Unternehmen wie Microsoft, Auth0 oder Facebook anschließen, um dessen Benutzeridentifikationssysteme zu verbessern.



▍ Ich möchte die Benutzer meines Projekts unter Kontrolle halten



In Anbetracht dieser Idee möchte ich Ihnen sofort eine Frage stellen: „Warum brauchen Sie das?“. Es ist möglich, dass Sie es wirklich nicht brauchen. Nun, es sei denn, Sie entwickeln ein neues Facebook. In diesem Fall sind Benutzerdaten Ihr wichtigstes Kapital. Je mehr solcher Daten Sie erfassen, desto besser.



Je mehr Benutzerdaten Sie sammeln, desto höher ist außerdem die Wahrscheinlichkeit, dass Sie Geld ausgeben müssen, um Ihr System mit bestimmten Regeln wie der DSGVO in Einklang zu bringen. Wenn Sie große Datenmengen über Benutzer speichern, kann dies die Folgen von Hacks verschlimmern und zusätzliche Kosten erfordern, um diese Folgen zu beseitigen.



Es ist zu beachten, dass die meisten der oben genannten spezialisierten Authentifizierungslösungen ziemlich detaillierte Informationen über Benutzer und deren Aktionen enthalten.



Wenn Sie erwägen, einen Authentifizierungsdienst zu verwenden, der von jemandem entwickelt wurde, den Sie hosten möchten, beachten Sie, dass die Verwendung solcher Dienste mit bestimmten Schwierigkeiten verbunden ist. Vielleicht möchten Sie einen ähnlichen Dienst nutzen, um sich die Möglichkeit zu geben, in Zukunft zu etwas anderem zu wechseln. Beachten Sie, dass solche Systeme schwer zu warten sind und dass die relativ geringe Anzahl der darin registrierten Benutzer häufig dazu führt, dass die Eigentümer solcher Systeme zusätzliche Sicherheitsmechanismen nicht effektiv nutzen können.



Wo soll ich anfangen?



Ich hoffe, ich habe Sie davon überzeugt, dass die Benutzerauthentifizierung eine Aufgabe ist, die am besten mit einem dedizierten Identitätsanbieter ausgeführt werden kann. Hier erfahren Sie, wie es geht.



Für den Anfang die guten Nachrichten. Es besteht darin, dass alle vier oben genannten Anbieter (Auth0, Azure AD, Google Identity Platform, Okta) und viele andere dieselben Protokolle verwenden: OpenID Connect / OAuth 2.0. Dies sind moderne Standardprotokolle, deren Client-Bibliotheken für fast alle Programmiersprachen und Frameworks zur Unterstützung existieren.



Gehen Sie wie folgt vor, um die Dienste eines Anbieters von Identifikationsinformationen zu nutzen, wenn Sie nicht auf Details eingehen:



  1. Registrieren Sie die Anwendung beim Identitätsanbieter. Nach der Registrierung erhalten Sie eine Kennung (Anwendungs-ID, Kunden-ID) und einen geheimen Schlüssel (Geheimschlüssel, Kunden-Geheimnis).
  2. Legen Sie die für Ihre Anwendung erforderlichen Berechtigungen fest. Zusätzlich zu der Tatsache, dass für die Anwendung ein Benutzerprofil verfügbar ist, können Sie je nach ausgewähltem Anbieter auf eine viel größere Datenmenge zugreifen. Dies kann beispielsweise den Zugriff auf Benutzernachrichten und den Zugriff auf deren Cloud-Speicher umfassen (z. B. über Office 365 oder G Suite).
  3. Integrieren Sie die Clientbibliothek des Authentifizierungsdienstes in Ihr Projekt.

Ein kryptografisch signiertes JWT-Token mit einer begrenzten Gültigkeitsdauer kann zum Verwalten von Benutzersitzungen verwendet werden. Das heißt, solange das Token gültig ist und die Anwendung es in der Anforderung empfängt, kann sie diese Anforderung als von dem Benutzer stammend betrachten, dem das Token gehört.



Das gleiche Token enthält einige Benutzerinformationen. Der Satz dieser Informationen hängt vom verwendeten Authentifizierungsdienst ab. Normalerweise ist dies der Benutzername, seine E-Mail-Adresse, seine Kennung.



Eine Anwendung kann diese Informationen verwenden, um einen Benutzer zu identifizieren. Sie können die Benutzer-ID verwenden, um auf die damit verbundenen Daten zuzugreifen, die in Ihrer Anwendung gespeichert sind.



Wie bereits erwähnt, sind JWT-Token kryptografisch signiert. Daher können Sie beim Überprüfen der Tokensignatur sicher sein, dass niemand die im Token enthaltenen Benutzerinformationen ersetzt hat.



Verwenden von OpenID Connect in Client-Server-Anwendungen



Wie genau der OpenID Connect-Mechanismus in einer bestimmten Anwendung verwendet wird, hängt stark von der Sprache ab, mit der die Anwendung erstellt wurde.



Auf jwt.io finden Sie eine vollständige Liste von Bibliotheken, mit denen JWT-Token überprüft werden können.



Für einige Technologie-Stacks können Sie außerdem übergeordnete Lösungen verwenden. Dies ist beispielsweise in Node.js / Express Express-JWT oder Pass .
