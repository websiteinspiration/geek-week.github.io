整理整頓


Gitのポイントは、作業を保存してコンテキストを切り替え、それ以外のことを行うことです。これは、コードのバックアップ、または複数の異なる関数を非同期で開発する機能です。最初のバージョンでエラーが見つかったという理由だけで、2番目のバージョンを破棄するのはひどいことです。 v1_final_bug_fixedのような名前でファイルを保存するのも恥ずかしいことです。ご存知のように、これは完全な混乱につながります。

同僚と共有できるGitブランチに更新がきちんと配置されると、生活がはるかに簡単になることは誰もが知っています。しかし、多くの場合、コンテキストを変更してから戻る場合があり、正しいブランチを見つけることができません。コミットはありましたか？多分彼は隠されているのですか？コミットが成功しなかったのかもしれませんが、今は全員が間違ったブランチに行っており、すべてが悪いので、私はひどく悪い仕事をしています！はい、誰もがそこにいて、そのような疑問を感じました。この状況に対処する方法があります。

簡単な方法があることがわかりました：

全編


たとえば、ステージングされた（準備）変更をコミットする前に誤って破棄してしまったなど、何かが完全にうまくいかない場合があります。それgit logが前の状態に戻ることができず、上記のヒントのいずれも役に立たない場合、それはgit reflogです。

参照によって内容を変更するGitのすべてのアクションHEAD@{}（たとえばpush/pull/branch/checkout/commit）は、reflog（参照ログ）に分類されます。実際、これはあなたがどのブランチにいるかに関係なく、すべてのアクションのストーリーです。これは、git log特定のブランチの変更を示すとの違いです。


git showコミットIDを使用して、特定の変更を確認 できます。これがあなたが探していたものである場合はgit checkout、目的のブランチに移動するか、上記のように特定のファイルを選択することもできます。

# See the reference log of your activity
git reflog --all

# Look at the HEAD at given point from reflog
git show HEAD@{2}

# Checkout the HEAD, to get back to that point
git checkout HEAD@{2}

コミットに失敗したファイルを準備する


極端な場合、git reflogファイルを元に戻せない場合（たとえば、中間ファイルでハードリセットを実行した場合）、もう1つのトリックがあります。

各変更は.git/objects、アクティブなプロジェクトのファイルで満たされたオブジェクト内に保存されるため、把握することはほとんど不可能です。ただし、と呼ばれるGitコマンドがありgit fsck、これはリポジトリ内の整合性（破損したファイルの存在）をチェックするために使用されます。フラグと一緒に使用し--lost-foundて、コミットに関連しないすべてのファイルを検索できます。このようなファイルは「ぶら下がりブロブ」と呼ばれます。

このコマンドでは、「ハングしているツリー」と「ハングしているコミット」を見つけることもできます。必要に応じて、フラグを使用できます--danglingが、利点は--lost-foundつまり、関連するすべてのファイルをフォルダに抽出します.git/lost-found。ほとんどの場合、アクティブなプロジェクトでは、そのような「ハング」しているファイルが多数あります。Gitには定期的に実行されてそれらを削除するごみ処理コマンドがあります。

したがって、--lost-foundすべてのファイルと作成日時が表示され、検索が非常に容易になります。個別のファイルはそれぞれ独立していることに注意してください。つまり、チェックアウトは使用できません。また、すべてのファイルにはわかりにくい名前（ハッシュ）が付けられるため、必要なファイルを別の場所にコピーする必要があります。

# This will find any change that was staged but is not attached to the git tree
git fsck --lost-found

# See the dates of the files
ls -lah .git/lost-found/other/

# Copy the relevant files to where you want them, for example:
cp .git/lost-found/other/73f60804ac20d5e417783a324517eba600976d30 index.html

チームワークのGit


Gitを単独で使用することは1つのことですが、通常はまったく異なる経験、スキル、およびツールを使用してチームで作業する場合、Gitは祝福または呪いになる可能性があります。これは、同じコードベースを共有し、コードレビューを実施して、チーム全体の進捗状況を監視するための強力なツールです。同時に、すべての従業員は、チームワークでそれを使用する方法について共通の理解を持つ必要があります。それが何であるかに関係なく：ブランチに名前を付ける、コミットで付随するメッセージをフォーマットする、またはコミットに含めるファイルを選択するという慣習では、適切なコミュニケーションを確保し、このツールの使用方法に同意することが重要です。

初心者向けのオンボーディングのシンプルさを確保し、会社が採用した原則や慣習を知らずにコミットを開始した場合にどうなるかを考えることは常に重要です。これは世界の終わりではありませんが、混乱を引き起こし、調整されたアプローチに戻るのに時間がかかる場合があります。

このセクションには、承認済みの合意をリポジトリ自体に直接統合し、宣言内のタスクの最大数を自動化して発行する方法に関するいくつかの推奨事項が含まれています。理想的なケースでは、新しい従業員はほとんどすぐに、チームの他のメンバーと同じスタイルで働き始めます。

同じ行末


デフォルトでは、WindowsはDOSの\r\n行末（CRLF）を使用し、MacとLinuxはUNIXの行末\n（LF）を使用し、古いバージョンのMacは\r（CR）を使用します。したがって、チームが成長するにつれて、行末に互換性がないという問題が発生しやすくなります。これは不便です。（通常）コードを壊すことはありませんが、そのため、コミットとプールリクエストはさまざまな無関係な変更を示します。多くの場合、人々は単にそれらを無視します。歩き回ってすべての間違った行末を変更するのは非常に面倒だからです。

解決策があります-すべてのチームメンバーに、自動回線補完のローカル設定を構成するように依頼できます。

# This will let you configure line-endings on an individual basis
git config core.eol lf
git config core.autocrlf input

もちろん、あなたはこの大会と初心者にサインアップする必要があります。これは忘れがちです。チーム全体でこれを行うにはどうすればよいですか？操作アルゴリズムに従って、Gitは.git / configリポジトリに構成ファイルが存在するかどうかを確認し、次にでユーザーのシステム全体の構成を~/.gitconfig確認し、次にでグローバル構成を確認します/etc/gitconfig。

これらはすべて良好ですが、これらの構成ファイルはいずれもリポジトリ自体を介してインストールできないことがわかりました。リポジトリ固有の構成を追加できますが、他のチームメンバーには拡張されません。

ただし、実際にリポジトリにコミットされるファイルがあります。これは.gitattributesと呼ばれます。デフォルトではありませんので、新しいファイルを作成して保存します*.gitattributes*。各ファイルの属性を設定します。たとえば、バイナリファイルの違いを計算する代わりに、git diffに画像ファイルのexifヘッダーを使用させることができます。この場合、ワイルドカードを使用して、設定がすべてのファイルに対して機能し、実際には、コマンド全体の共通構成ファイルとして機能するようにすることができます。

# Adding this to your .gitattributes file will make it so all files
# are checked in using UNIX line endings while letting anyone on the team
# edit files using their local operating system’s default line endings.
* text=auto

自動非表示


コンパイルされたファイル（などnode_modules/）を.gitignoreに追加して、ローカルに保存され、リポジトリにアップロードされないようにするのが慣例です。ただし、ファイルをアップロードしたいが、後でプールリクエストで毎回満たしたくない場合もあります。

この状況では（少なくともGitHubで）、マークされたパスを.gitattributesに追加linguist-generatedし、.gitattributesがリポジトリのルートフォルダーにあることを確認できます。これにより、プールリクエスト内のファイルが非表示になります。それらは「最小化」されます。完全なコードがなくても、変更の事実を確認できます。

コードレビュープロセスにおけるストレスと認知的負荷を軽減するすべてのものが、その品質を向上させ、時間を短縮します。

たとえば、リソースファイル（アセット）をリポジトリに追加したいが、後で変更および追跡する予定がないため、次の行を属性付きでファイルに追加できます。

*.asset linguist-generated

git blameをより頻繁に使用する


Harry Roberts の記事「Gitで行うのが好きな小さなこと」では、このチームをポジティブなアクションとして感じられるように（translation git blameを割り当てて）エイリアスを割り当てgit praise（translationを「English」から翻訳）することを推奨しています。もちろん、名前を変更してもチームの動作は変わりません。しかし、関数の使用について話し合うとgit blame、誰もが緊張します。もちろん私もそうです。非難（罪悪感）を否定的なものとして認識するのは自然なことですが、これは完全に間違っています！

強力な関数git blame （またはgit praise、必要に応じて）は、このコードで最後に作業した人を示します。私たちは彼を責めたり褒めたりするつもりはありませんが、状況を明らかにしたいだけです。どの質問を誰に対して行うかが明確になるため、時間を節約できます。

それはgit blame良いものとしてだけでなく、チーム全体が混乱を減らし、誰が何を知っているかを理解するのに時間を無駄にしないようにするコミュニケーション手段としても提示されるべきです。Visual Studioなどの一部のIDEは、この機能を注釈としてアクティブにします。各機能について、誰が最後に変更したか（したがって、誰と話し合ったか）を即座に確認できます。

手動自動化


Gitの主要な自動化機能の1つはgit bisectです。多くの人がそれを聞いたことがありますが、それを使用する人はほとんどいません。一番下の行は、Gitツ​​リー（コミット履歴）の処理と、エラーが入力された場所の検索です。

これを行う最も簡単な方法は手動です。を開始しgit bisect start、良いコミットと悪いコミットの識別子を設定し（バグがない場合とバグがある場合）、実行するgit bisect goodかgit bisect bad、コミットごとに実行します。

これは一見したところよりも強力な機能です。Gitログを線形に実行しないため、手動で反復プロセスとして実行できます。代わりに、最小限のステップでコミットを効率的に通過するバイナリ検索を使用します。

# Begin the bisect
git bisect start

# Tell git which commit does not have the bug
git bisect good c5ba734

# Tell git which commit does have the bug
git bisect bad 6c093f4

# Here, do your test for the bug.
# This could be running a script, doing a journey on a website, unit test etc.

# If the current commit has bug:
git bisect bad

# If the current commit does not have the bug
git bisect good

# This will repeat until it finds the first commit with the bug
# To exit the bisect, either:

# Go back to original branch:
git bisect reset

# Or stick with current HEAD
git bisect reset HEAD

# Or you can exit the bisect at a specific commit
git bisect reset <commit ID>

スチュアートハロウェイ

最初git bisectは面白くてかっこいい機能のように思えるかもしれませんが、結局はあまり役に立ちません。スチュワートのパフォーマンスの大部分は、これまでのようにデバッグすることは実際には逆効果であることを示しています。代わりに、テストに合格したかどうかにかかわらず、経験的事実に焦点を当てれば、作業バージョンから始めて、すべてのコミットでテストを実行し、慣れ親しんでいる「暗闇の中をさまよう」という感覚を減らすことができます。

では、どうやって自動化するのですかgit bisect？対応するコミットごとにスクリプトを渡すことができます。以前、bisectの各ステップでスクリプトを手動で実行できると述べましたが、実行するコマンドを渡すと、スクリプトは各ステップで自動的に実行されます。これは、この特定の問題またはテスト（モジュラー、機能、統合、あらゆるタイプのテスト）をデバッグするためのスクリプトにすることができます。したがって、回帰が繰り返されないことを確認するテストを作成し、以前のコミットでこのテストを実行できます。

# Begin the bisect
git bisect start

# Tell git which commit does not have the bug
git bisect good c5ba734

# Tell git which commit does have the bug
git bisect bad 6c093f4

# Tell git to run a specific script on each commit
# For example you could run a specific script:
git bisect run ./test-bug

# Or use a test runner
git bisect run jest
