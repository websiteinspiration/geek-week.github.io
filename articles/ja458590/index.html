iOSアプリケーションのアーキテクチャアプローチ

今日は、iOS開発におけるアーキテクチャのアプローチ、特定の実装におけるニュアンスと開発のいくつかについてお話します。どのアプローチを採用するかを説明し、詳細をもう少し詳しく説明します。


すべてのカードをすぐに公開します。MVVM-R（MVVM +ルーター）を使用しています。


実際、これは通常のMVVMであり、画面間のナビゲーションは別のレイヤー（ルーター）とデータを受信するためのロジック（サービス内）に配置されます。次に、各レイヤーの実装での成果を検討します。


なぜVIPERやMVCではなくMVVMなのですか？


MVCとは異なり、MVVMはレイヤー間で責任がかなり分かれています。画面のViewModelもプロトコルによって閉じられていますが、VIPERほど多くの「提供」コードはありません。このアーキテクチャはVIPERにいくらか似ており、PresenterとInteractorのみがViewModelで結合され、レイヤー間の接続は、リアクティブプログラミングとバインダー（ReactiveSwiftを使用）を使用して簡素化されます。

enum UserNetworkRouter: URLRequestConvertible {
    case info
    case update(userJson:[String : Any])
}

extension UserNetworkRouter: NetworkRequestParams {
    var path: String {
        switch self {
        case .info:
            return "/users/profile"
        case .update:
            return "/users/update_profile"
        }
    }

    var method: HTTPMethod {
        switch self {
        case .info:
            return .get
        case .update:
            return .post
        }
    }

    var encoding: ParameterEncoding {
        switch self {
        case .info:
            return URLEncoding()
        case .update:
            return JSONEncoding()
        }
    }

    var parameters: Parameters {
        switch self {
        case .info:
            return [:]
        case .update(let userJson):
            return userJson
        }
    }
}
