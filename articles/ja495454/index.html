近年、高性能なパッケージ処理が必要な場合にLinuxカーネルの制限を補うように設計されたプログラミングツールとテクニックが人気を博し始めています。この種の最も一般的な方法の1つは、カーネルのバイパス（カーネルバイパス）と呼ばれ、通過するネットワークレイヤーコアがユーザー空間からすべてのパケット処理を実行できるようにします。カーネルをバイパスするには、ユーザースペースからネットワークカードを管理する必要もあります。つまり、ネットワークカードを使用する場合、ユーザースペースドライバーに依存します。

ユーザー空間からネットワークカードの完全な制御をプログラムに移すことで、カーネルに関連するコスト（コンテキストの切り替え、ネットワークレベルの処理、割り込みなど）を削減します。これは、10 Gb /秒以上の速度で作業する場合に非常に重要です。カーネルバイパスに加えて、他の機能（バッチ処理）と正確なパフォーマンスチューニング（NUMAアカウンティング、CPU分離など）の組み合わせは、ユーザー空間における高性能ネットワーク処理の基本に対応しています。おそらく、パッケージ処理へのこの新しいアプローチのモデル例は、IntelのDPDK（データプレーン開発キット）です。）、シスコ（ベクターパケット処理）、ネットマップと、もちろん、からVPPを含む他のよく知られたツールや技術、ありますがSnabbが。

本質的に、ユーザースペースでネットワークの相互作用を整理する場合、パケット処理をカーネルからユーザースペースに転送することにより、生産性の向上が実現されます。 XDPは正反対です。ネットワークプログラムをユーザー空間（フィルター、コンバーター、ルーティングなど）からカーネル領域に移動します。 XDPを使用すると、パケットがネットワークインターフェイスに到着するとすぐに、カーネルのネットワークサブシステムに移動し始める前に、ネットワーク機能を実行できます。その結果、パケット処理速度が大幅に向上します。しかし、カーネルはどのようにしてユーザーがカーネル空間でプログラムを実行できるようにしますか？この質問に答える前に、BPFとは何かを見てみましょう。

BPFおよびeBPF

それほど明確ではない名前のBPF（Packet Filtering、Berkeley）にも関わらず、実際には仮想マシンモデルです。この仮想マシンは元々、パケットフィルタリングを処理するように設計されていたため、この名前が付けられました。

BPFを使用する最も有名なツールの1つはtcpdumpです。パケットをキャプチャするとき、tcpdumpユーザーは式を指定してパケットをフィルタリングできます。この式に一致するパケットのみがキャプチャされます。たとえば、「tcp dst port 80」という表現は、ポート80に到着するすべてのTCPパケットに適用されます。コンパイラは、この表現をBPFバイトコードに変換することで短縮できます。 上記のプログラムが基本的に行うことは次のとおりです。

BPF仮想マシンとそのバイトコードは、1992年後半にBSDパケットフィルター：ユーザーレベルでのパケットキャプチャの新しいアーキテクチャが1993年冬のUsenix会議で初めて紹介されたときに、Steve McCannとVan Jacobsonによって提案されました。

BPFは仮想マシンであるため、プログラムが実行される環境を定義します。バイトコードに加えて、バッチメモリモデル（ブート命令が暗黙的にパッケージに適用されます）、レジスター（AおよびX、バッテリーおよびインデックスレジスター）、スクラッチメモリストレージ、および暗黙的なプログラムカウンターも定義します。興味深いことに、BPFバイトコードはMotorola 6502 ISAをモデルにしています。 Steve McCannがSharkfest '11での本会議で回想したように、彼は高校からApple IIでプログラムしたときの6502ビルドに精通しており、この知識はBPFバイトコードの設計に関する彼の仕事に影響を与えました。

BPFのサポートはバージョン2.5以降のLinuxカーネルに実装されており、主にJay Schullistの努力によって追加されました。 BPFコードは、JITモードで動作するようにBPFインタープリターをEric Dumazettが再設計した2011年まで変更されていません（出典：パケットフィルター用のJIT）。その後、カーネルは、BPFバイトコードを解釈する代わりに、BPFプログラムをターゲットアーキテクチャ（x86、ARM、MIPSなど）に直接変換できます。

その後2014年に、Alexey StarovoitovがBPFの新しいJITメカニズムを提案しました。実際、この新しいJITは新しいBPFベースのアーキテクチャになり、eBPFと呼ばれています。しばらくの間、両方の仮想マシンが共存したと思いますが、現在、パケットフィルタリングはeBPFに基づいて実装されています。実際、最新のドキュメントの多くの例では、BPFはeBPFを意味すると理解されており、従来のBPFは現在cBPFとして知られています。

eBPFは、いくつかの方法でクラシックBPF仮想マシンを拡張します。

eBPF：例

LinuxカーネルソースにはeBPFの例がいくつかあります。これらは、samples / bpf /で入手できます。これらの例をコンパイルするには、次のように入力します。

$ sudo make samples/bpf/

私はeBPFの新しい例を自分で作成するのではなく、samples / bpf /にあるサンプルの1つを使用します。コードのいくつかのセクションを見て、それがどのように機能するかを説明します。例として、私はプログラムを選びましたtracex4。

一般に、samples / bpf /の各例は2つのファイルで構成されています。この場合：

  • tracex4_kern.c、カーネルでeBPFバイトコードとして実行する必要があるソースコードが含まれています。
  • tracex4_user.c、ユーザー空間のプログラムが含まれています。

この場合、 tracex4_kern.ceBPFをバイトコードにコンパイルする必要があります。現在gcc、eBPFのサーバー部分はありません。幸い、clangeBPFバイトコードを発行できます。オブジェクトファイルのコンパイルにMakefile使用clangしtracex4_kern.cます。

eBPFの最も興味深い機能の1つはカードであることを前述しました。tracex4_kernは1つのマップを定義します。

struct pair {
    u64 val;
    u64 ip;
};  

struct bpf_map_def SEC("maps") my_map = {
    .type = BPF_MAP_TYPE_HASH,
    .key_size = sizeof(long),
    .value_size = sizeof(struct pair),
    .max_entries = 1000000,
};

BPF_MAP_TYPE_HASH-eBPFが提供する多くのタイプのカードの1つ。この場合、それは単なるハッシュです。また、広告に気づいたかもしれませんSEC("maps")。SECは、バイナリファイルの新しいセクションを作成するために使用されるマクロです。実際、この例でtracex4_kernはさらに2つのセクションを定義しています。

SEC("kprobe/kmem_cache_free")
int bpf_prog1(struct pt_regs *ctx)
{   
    long ptr = PT_REGS_PARM2(ctx);

    bpf_map_delete_elem(&my_map, &ptr); 
    return 0;
}
    
SEC("kretprobe/kmem_cache_alloc_node") 
int bpf_prog2(struct pt_regs *ctx)
{
    long ptr = PT_REGS_RC(ctx);
    long ip = 0;

    //  ip-   kmem_cache_alloc_node() 
    BPF_KRETPROBE_READ_RET_IP(ip, ctx);

    struct pair v = {
        .val = bpf_ktime_get_ns(),
        .ip = ip,
    };
    
    bpf_map_update_elem(&my_map, &ptr, &v, BPF_ANY);
    return 0;
}

eBPFプログラムは、セクションに関連付けられたカードと機能を定義します。カーネルが特定のタイプのイベント（たとえば、tracepoint）をスローすると、バインドされた関数が実行されます。マップは、カーネルプログラムとユーザー空間プログラム間のデータ交換を提供します。
