最新のスクリプトの読み込み

各ブラウザに適切なコードを渡すことは簡単な作業ではありません。

この記事では、この問題を解決するためのいくつかのオプションを検討します。



最新のブラウザーで最新のコードを渡すと、パフォーマンスが大幅に向上します。JavaScriptパッケージは、よりコンパクトまたは最適化された最新の構文を含み、古いブラウザーをサポートすることができます。

開発者向けのツールの中で、モダンコードまたはレガシーコードの宣言的ロードのmodule / nomoduleパターンが支配的です。これにより、ブラウザーにソースが提供され、使用するソースを決定できます。

<script type="module" src="/modern.js"></script>  
<script nomodule src="/legacy.js"></script>

残念ながら、すべてがそれほど単純なわけではありません。上記のHTMLアプローチは、EdgeおよびSafariでスクリプトのリロードをトリガーします。

何ができますか？


ブラウザーによっては、コンパイル済みスクリプトのオプションの1つを提供する必要がありますが、いくつかの古いブラウザーはこれに必要なすべての構文をサポートしていません。

まず、Safari Fixがあります。Safari 10.1はnomodule、スクリプトの属性ではなくJSモジュールをサポートしているため、最新のコードとレガシーコードの両方を実行できます。ただし、beforeloadSafari 10および11でサポートされているカスタムイベントは、ポリフィルに使用できますnomodule。

方法1：動的ダウンロード


小さなスクリプトローダーを実装することで、これらの問題を回避できます。LoadCSSのしくみに似ています。ブラウザでのESモジュールと属性の実装を期待する代わりにnomodule、モジュールスクリプトを「リトマステストによるテスト」として実行し、その結果に基づいて、最新のコードまたはレガシーコードのダウンロードを選択できます。

<!-- use a module script to detect modern browsers: -->  
<script type="module">  
  self.modern = true
</script>

<!-- now use that flag to load modern VS legacy code: -->  
<script>  
  addEventListener('load', function() {
    var s = document.createElement('script')  
    if ('noModule' in s) {  // notice the casing  
      s.type = 'module'
      s.src = '/modern.js'
    } else {
      s.src = '/legacy.js'
    }
    document.head.appendChild(s)
  })
</script>

var s = document.createElement('script')

これはすぐに、最新のコードまたはレガシーコードをロードする関数に変換でき、それらの非同期ロードも提供します。

<script>  
  $loadjs("/modern.js","/legacy.js")
  function $loadjs(src,fallback,s) {
    s = document.createElement('script')
    if ('noModule' in s) s.type = 'module', s.src = src
    else s.async = true, s.src = fallback
    document.head.appendChild(s)
  } 
</script>

プレブート

ソリューションは完全に動的であるため、ブラウザーは、最新のスクリプトまたはレガシースクリプトを挿入するために記述したブートストラップコードを開始するまで、JavaScriptリソースを検出できません。通常、ブラウザーはHTMLをスキャンして、事前にダウンロードできるリソースを探します。この問題は解決されましたが、完全ではありません<link rl=modulpreload>。最新のブラウザーでパッケージの最新バージョンをプリロードできます。

残念ながら、これまでサポートして いるのはChromeだけmodulepreloadです。

<link rel="modulepreload" href="/modern.js">  
<script type="module">self.modern=1</script>  
<!-- etc -->

この手法が適している場合は、これらのスクリプトを埋め込むHTMLドキュメントのサイズを縮小できます。スプラッシュスクリーンやクライアントアプリケーションのダウンロードコードのようにペイロードが小さい場合、プリロードスキャナーをドロップしてもパフォーマンスに影響を与えることはほとんどありません。サーバー上で重要なHTMLを大量に描画してブラウザーに送信する場合、プリロードスキャナーは便利であり、説明されているアプローチは最適なオプションではありません。

このソリューションの使用例は次のとおりです。

<link rel="modulepreload" href="/modern.js">  
<script type="module">self.modern=1</script>  
<script>  
  $loadjs("/modern.js","/legacy.js")
  function $loadjs(e,d,c){c=document.createElement("script"),self.modern?(c.src=e,c.type="module"):c.src=d,document.head.appendChild(c)}
</script>

JSモジュールをサポートするブラウザのリストは、をサポートするブラウザとほとんど同じであることにも注意してください<link rl=preload>。一部のサイトでは、<link rl=preload as=script crossorigin>代わりに使用することが適切な場合がありますmodulepreload。の場合のように、従来のスクリプトのプリロードは時間の経過に伴う均一な解析を意味しないため、パフォーマンスが低下する可能性がありますmodulepreload。

方法2：ユーザーエージェントを追跡する


ユーザーエージェントの追跡は簡単な作業ではないため、適切なコード例はありません。しかし、その後、Smashing Magazineで優れた記事を読むことができます。

実際、<scrit src=bundle.js>すべてのブラウザで同じHTMLで始まります。bundle.jsがリクエストされると、サーバーはリクエスト元のブラウザのユーザーエージェント文字列を解析し、ブラウザがどのように認識されたかに応じて、返すJavaScript-モダンまたはレガシーを選択します。

アプローチは普遍的ですが、深刻な結果を伴います：

  • スマートサーバーが必要なため、このアプローチは静的展開環境（静的サイトジェネレーター、Netlifyなど）では機能しません。
  • これらのJavaScript URLのキャッシュは、非常に揮発性の高いユーザーエージェントに依存するようになりました。
  • UAの定義は困難であり、誤った分類につながる可能性があります。
  • ユーザーエージェントの行は、なりすましが簡単で、新しいUAが毎日表示されます。

これらの制限を回避する1つの方法は、module / nomoduleパターンをユーザーエージェントの区別と組み合わせて、パッケージの複数のバージョンを同じアドレスに送信しないようにすることです。このアプローチはページキャッシングを減らしますが、効率的なプリロードを提供します。HTMLを生成するサーバーはいつmodulepreload、いつ使用するかを知っていますpreload。

function renderPage(request, response) {  
  let html = `<html><head>...`;

  const agent = request.headers.userAgent;
  const isModern = userAgent.isModern(agent);
  if (isModern) {
    html += `
      <link rel="modulepreload" href="modern.mjs">
      <script type="module" src="modern.mjs"></script>
    `;
  } else {
    html += `
      <link rel="preload" as="script" href="legacy.js">
      <script src="legacy.js"></script>
    `;
  }

  response.end(html);
}

各リクエストに応じてサーバー上ですでにHTMLを生成しているサイトの場合、これは最新のスクリプトのダウンロードへの効果的な移行となる可能性があります。
