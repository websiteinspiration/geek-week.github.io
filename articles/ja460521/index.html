DDEとフィールド


その後、マイクロソフトは依然としてWordのDDEを無効にしましたが、その前に、この機能は単に誤用されたと述べました。彼らが何かを変えたがらないのは理解できる。私自身の経験から、私は、ドキュメントを開いたときにフィールドの更新が有効になっているが、WordマクロがITサービスによって無効にされている（ただし、通知がある）ような例を観察しました。ちなみに、対応するパラメーターはWordの設定セクションにあります。

ただし、フィールドの更新が有効になっている場合でも、上記のDDEの場合と同様に、Microsoft Wordは、フィールドが削除されたデータへのアクセスを要求すると、ユーザーに通知します。マイクロソフトは本当にあなたに警告しています。

しかし、ほとんどの場合、ユーザーはこの警告を見逃して、Wordでフィールド更新をアクティブにします。これは、危険なDDE機能を無効にするためにマイクロソフトに感謝するまれな機会の1つです。

オブジェクトの謎


このパッチをインストールした場合でも、MS Officeには他にもセキュリティホールがあり、ハッカーがWordで行ったのと非常によく似たことを実行できます。次のシナリオでは、コードを記述せずに、フィッシング攻撃の餌としてExcelを使用する方法を学習します。

このシナリオを理解するために、Microsoft Component Object Model、または略してCOM（Component Object Model）を思い出してみましょう。

COMは1990年代から存在しており、リモートRPCプロシージャコールに基づく「プログラミング言語に中立なオブジェクト指向コンポーネントモデル」として定義されています。 COM用語の一般的な理解については、StackOverflowに関するこの投稿をお読みください。

概して、COMアプリケーションは、ExcelまたはWordの実行可能ファイル、または起動可能なその他のバイナリファイルと考えることができます。

COMアプリケーションでもスクリプト（JavaScriptまたはVBScript）を実行できることがわかりました。技術的には、これはスクリプトレットと呼ばれます。おそらく、Windows上のファイルのsct拡張子に遭遇した可能性があります-これはスクリプトレットの公式の拡張子です。本質的に、これらはXMLラッパーにラップされたスクリプトのコードです。

<scriptlet>
<registration
description="test"
progid="test"
version="1.00"
classid="{BBBB4444-0000-0000-0000-0000FAADACDC}"
remotable="true">
</registration>
<script language="JScript">
<![CDATA[

var r = new ActiveXObject("WScript.Shell").Run("cmd /k powershell -c Write-Host You have been scripted!");

]]>
</script>
</scriptlet>
