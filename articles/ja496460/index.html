Unity Addressables：常に十分なメモリ


あなたは、美しいPS4 VRゲームをOculus Questに移植することに取り組んでいる数人のプログラマーとアーティストのチームを率いています。それを行うには6か月あります。あなたの最初の動きは何ですか？ Unity Addressablesを使ってみましょう。

あなたは、いくつかのかなり難しいタスクを同時に解決する必要があることを理解しています。一部の難易度は他の難易度よりも高くなりますが、それぞれの分野での経験に依存します。それらのうちどれが最も頻繁に睡眠を奪われたかを選択した場合、それは何になりますか？

私は次のように思います：読者の約70％がゲームをクエストに移植するときの最大の問題はCPU / GPUのパフォーマンスだと言う。私はこれに答えることができます：おそらくあなたは正しいです。生産性の向上は、VRゲームの最も難しい領域の1つです。このタイプの最適化には、製品の徹底的な調査が必要であり、これには時間がかかります。場合によっては、それ以上の最適化が不可能になることがあります。そのため、通常、ゲームプレイとグラフィックスのコストのかかる要素を取り除く必要があります。そして、プレーヤーを失望させることは危険です。

速度、速度、速度...この点について、クエストプラットフォームから何を期待できますか？それはどのくらい生産的ですか？実際のところ、開発の経験があれば、その機動性にもかかわらず、驚くほど強力であることがわかります。

「さあ、著者、なぜあなたは嘘をついているのですか？ 2番目のブラウザタブを開くとすぐに、電話の速度が低下し始めます。モバイルプラットフォームが生産的であると言えるのはなぜですか？」

大きな違いは、クエストのアクティブ冷却システムにあります。これは、他のモバイルプラットフォームでは提供されないCPU / GPUに大きな利点をもたらします。これは髪のほこりを吹き飛ばし、プロセッサーが顔に溶けないようにする強力なファンです。

さらに、より特殊なOSは、標準のAndroidよりも仮想現実（驚き）のレンダリングに最適化されています。過去数年にわたって、モバイルハードウェアは固定プラットフォームに急速に追いつき始めています。

しかし同時に、特に強力なプラットフォームからのVRゲームのポートでは、72 fpsの周波数での一定のレンダリングのタスクが困難になることも否定しません。 Oculus Questについて話すとき、画面、バッテリー、4つのカメラ、ファンを備えたSnapdragon 835を想像できます。

欠陥のように見えるものは、実際には利点として利用できます。このモバイルプラットフォームは、十分に研究された、弱い鉄を備えたデバイスです。 CPUとGPUの負荷を許容可能なレベルに迅速に削減するための既知のトリックは1,000あると言えます。興味がある場合は、その後の投稿で確認できます。そしてこの記事では、パフォーマンスを問題から除外します。

この問題で私たちが注目しているのは、PS4と比較して、Questには半分のRAM容量という1つのハードウェア特性があることです。つまり、ボリュームが8 GBから4 GBに減少しました。これは概算です。両方のプラットフォームで、オペレーティングシステムはそれらを完全に使用することを許可しないため、追跡できます。生態系が機能するために必要ないくつかのサブシステム。Questでは、最大約2.2 GBのRAMを使用できます。さらに多くの場合、カオスはすでに始まっています。

「しかし、カオスとはどういう意味ですか？」実際には、ゲームにとって正しいメモリ管理を実装することが重要です。これは、2つの制限があるために起こりました。

  • ハードメモリ制限：特定のしきい値を超えると、OSはゲームを強制終了します
  • ソフトメモリ制限

明らかに、ゲームで最初または2番目が発生することは望ましくありません。過去2時間のパスを失ったプレーヤーの怒りを想像できますか？はい、彼は間違いなくアプリケーションストアに行き、そこで楽しいことを言うことはありません。

もちろん、2.2 GBのRAMの保証された可用性はそれほど多くありません。通常、これは統計が最初から常に監視されている新しいプロジェクトでは問題になりませんが、ハードウェアがはるかに弱いポートでは明らかに困難になります。

過去に同様のポートを扱ったことがあれば、使用可能なRAMの量を半分にすることが非常に困難になることにすぐに気付くでしょう。それは、ゲームのアーキテクチャがそのような変更にどれだけうまく対応できたかに大きく依存しますが、ほとんどの場合、それは涙につながるだけです。

メモリ要件を削減するための最も一般的な戦略は、アセットの圧縮パラメーターの変更、スクリプトの最適化、シェーダーのバリエーションの削減などです。ほとんどの場合、最初の解決策はテクスチャのインポート設定を変更することですが、必要に応じて、メッシュ、アニメーション、サウンドの圧縮を使用できます。問題は、そのような手法は通常複雑で、独自の上限があることです。

すべてのプラットフォームが同じインポートパラメータをサポートするわけではありません。異なるデバイス用に開発する場合、QA、デザイングラフィックス、プログラミングの複雑さは言うまでもなく、アセンブリパイプラインのコストが大幅に増加します。たとえば、これはAndroid ASTCデバイスをサポートしますか、それともETC2（および一般にそれらのいずれか）のみをサポートしますか？ああ、それでも64ビットのビルドが必要ですが、同時に32ビットバージョンのプレーヤーを維持したいと考えています。ゲーム内のアップデートごとにAPKをいくつ作成してテストする必要がありますか？人生を簡素化したいなら、これらのテクニックだけに頼るべきではありません。

したがって、私たちはより深く行く必要があります。もちろん、特にポートを作成する場合は、すべてをできるだけ単純にする必要があります。パフォーマンスのためにゲームを完全にリサイクルすることは、単に移植しないことよりもさらに悪い選択肢です。記事のトピックの一部として、最も重要な利点の1つを示します。必要なメモリ量を数時間で半分にする方法を学びます。

すごくないですか？

さて、みましょう、私に聞いてみましょう：これはあなたのケースで本当に可能ですか？私は答えます：それは初期条件に依存しますが、私の経験では、答えはYESです。 Unity Addressablesは、ここで素晴らしいサービスを提供できます。トリックは何ですか？あなたは投資し、プロセスを習得する必要があります。しかし、そのようなワークフローにより、今月の従業員の称号を獲得することができます。

Unity Addressables：なぜ必要なのか？


この重要なセクションに注意を払う必要があります。私たちの仕事は、メモリ使用量を最適化し、すばやく実装するための簡単な方法を認識することです。これにはさまざまな方法がありますが、最も強力で同時に最も簡単な方法の1つは、最初のシーンをロードしてプロファイラーを起動することです。どうして？ゲームの最適化され

ていないアーキテクチャはゲームプレイのどの時点でも認識できるため、これを確認する最も速い方法は最初のシーンをプロファイリングすることです。この理由は、すべての資産へのリンクを含むシングルトン-Sのようなスクリプトの多くの場合、過度に積極的な利用ということです念のため。

つまり、多くのゲームには通常、アセットへのリンクの地獄を作成する全能スクリプトがあります。このコンポーネントは、現在使用されているかどうかに関係なく、すべてのアセットを常にロードし続けます。

それはどれほど悪いですか？

状況は異なります。あなたのゲームがメモリによって制限される可能性が高い場合？追加されるアセット数の増加に応じてゲームが適切に拡張されないため（たとえば、将来のDLCについて考えます）、これは非常に危険な決定です。 Androidなどの異種デバイス用に開発している場合、メモリ容量は1つではありません。各デバイスには独自の容量があるため、最悪のケースに依存する必要があります。ユーザーが突然Facebookのメッセージに返信するように切り替えた場合、OSはいつでもアプリケーションを強制終了することを決定できます。彼が戻ったとき、サプライズが彼を待っています-ゲームはすでに終了しています。

それ楽しいですか？

絶対違う。

状況を複雑にするのは、後でクロスプレイを維持しながらゲームを別の強力でないプラットフォームに移植することを決定した（または誰かがあなたに代わって決定した）場合、幸運を祈ることしかできないという事実です。あなたは間違いなくそのような技術的な問題に直面したくないでしょう。

一方、従来の資産管理が適切な状況はありますか？はい、もちろん。 PS4などの統一されたプラットフォーム向けに開発していて、ほとんどの要件が最初からわかっている場合、グローバルオブジェクトの利点は、改良されたメモリ管理システムのさらなる複雑さを上回る可能性があります。

あなたが認めなければならないので：私たちが必要とするすべてを格納する古き良きグローバルオブジェクトは、それがあなたに合っているなら簡単な解決策です。コードを簡略化し、参照されるすべてのアセットをプリロードします。

とはいえ、鉄のリソースを最大限に活用しようとする開発者にとって、従来のメモリ管理は受け入れられません。記事を読んでいるので、スキルを向上させたいと考えています。それでそれを行う時が来ました。

レベル1開発者：従来の資産管理


最も簡単な資産管理方法から始めます。今回のケースでは、コンポーネントのスカイボックスマテリアルへの直接リンクのリストを作成する必要があります。

これを私と一緒に行う場合、準備には3つの簡単なステップが必要です。

  1. gitからプロジェクトをダウンロードする
  2. Unityでプロジェクトを開く
  3. 再生ボタンをクリックしてください！

いいよこれで、ボタンをクリックしてスカイボックスを変更できます。だからオリジナル...と退屈。私が理解しているように、これまでのところUnity Addressablesはありません。

これらの退屈な瞬間を我慢する理由がすぐにわかります。

まず、私たちのプロジェクトはどのように構成されていますか？ 2つの主要なシステムに依存しています。 1つには、ゲームマネージャオブジェクトがあります。このコンポーネントは、スカイボックスマテリアルへのリンクを格納し、UIイベントに応じてそれらを切り替えるメインスクリプトです。とても簡単です。

using UnityEngine;

public class Manager : MonoBehaviour
{
    [SerializeField] private Material[] _skyboxMaterials;

    public void SetSkybox(int skyboxIndex)
    {
        RenderSettings.skybox = _skyboxMaterials[skyboxIndex];
    }
}

Managerは、RenderSettings APIを使用して特定のマテリアルをシーンに適用する機能をUIシステムに提供します。

次に、CanvasSkyboxSelectorがあります。このゲームオブジェクトには、垂直に配置されたボタンのセットをレンダリングするキャンバスコンポーネントが含まれています。各ボタンを押すと、前述のManager関数が呼び出され、ボタンのIDに応じて、レンダリングされたスカイボックスが置き換えられます。つまり、各ボタンのOnClickイベントは、ManagerオブジェクトのSetSkybox関数を呼び出します。簡単ですね。

資産レベル2の管理Unity Addressablesはメインウィンドウです

親愛なるリーダー、これは簡単な部分でした。そして今、楽しみが始まります。

前のセクションの私たちの古くからの友人、サーマネージャーを訪ねてください。チェックすると、アセットへの直接リンクがまだ保存されていることがわかります。これは必要ありません。

代わりに、間接リンクの使用方法をマネージャーに教えます。AssetReference（Unreal Engineでは、これらはソフトリファレンスと呼ばれます）。

コンポーネントをより美しくしましょう：

using UnityEngine;
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;

public class Manager : MonoBehaviour
{
    [SerializeField] private List<AssetReference> _skyboxMaterials;

    private AsyncOperationHandle  _currentSkyboxMaterialOperationHandle;

    public void SetSkybox(int skyboxIndex)
    {
        StartCoroutine(SetSkyboxInternal(skyboxIndex));
    }

    private IEnumerator SetSkyboxInternal(int skyboxIndex)
    {
        if (_currentSkyboxMaterialOperationHandle.IsValid())
        {
            Addressables.Release(_currentSkyboxMaterialOperationHandle);
        }

        var skyboxMaterialReference = _skyboxMaterials[skyboxIndex];
        _currentSkyboxMaterialOperationHandle = skyboxMaterialReference.LoadAssetAsync();
        yield return _currentSkyboxMaterialOperationHandle;
        RenderSettings.skybox = _currentSkyboxMaterialOperationHandle.Result;
    }
}

プロファイラーが満足しているときに私はそれが好きです。そして今、私たちは世界で最も幸せなプロファイラーを見ています。満足のいくプロファイラーとは、次のことを意味します。まず、Nokia 3210でもゲームをプレイできる満足のいくプレイヤーがいます。第二に、これらは満足のいくプロデューサーです。そしてあなたにとって、これはあなたの財布が満足されることを意味します。

  • 適切なメモリ管理。
