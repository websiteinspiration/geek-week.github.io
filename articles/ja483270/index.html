公共サービスの地域ポータルの例でのHighLoadアプリケーションの進化



「明日は20日です。つまり、再び嵐が来るでしょう。それを止めることは不可能です。準備をして、今度はそれが吹いて奇跡が起こり、私たちの湖のフェリーが海を征服することを期待してください。」このような考えは、数年前に地方自治体のサービスポータルのサポートに関与したチームを圧倒しました。どのようにしてこの状況に陥ったのか、そしてどのようにしてその状況から抜け出したのかを以下に説明します。

すべての始まり


遠い1990年代に、住宅および公益事業部門は開発のブームを経験し、新しい技術が導入され、自動化された情報システム、および新しい機器が購入されました。しかし、何かが実質的に変わらずに長い間残った、すなわちアパートの支払い。はい、アパートの同じ領収書は、支払いドキュメントに変換され、バーコードを取得しました。詳細な解読は、紙片のままでした。住宅および公共サービス企業または資源供給企業の決済センターの典型的な作業計画は次のとおりです。



徐々に、インターネットのインターネットはブロードバンドアクセスに置き換えられ、考えが浮上しました。オンライン決済の書類を電子形式で受け取ってみませんか？同時に、住宅および公共サービス部門は組織的な形で揺れ動いており、MPP住宅および公共サービス（住宅および公共サービスの地方自治体生産企業）は地方自治体の単一企業（地方自治体単一企業）、DEZy（単一顧客の監督）に置き換えられました。すべての変化の結果、住宅および公共サービス企業のIT部門は疎外され、あらゆる種類の決済センターがその基盤として生まれました。決済センターの本質は、実際には、家賃の計算と人口の情報サポートでした。

成長段階、00年代


その後、電子サービスを提供する最初の情報ポータルが誕生しました。最初のユーザーの数は数十人で測定され、誰もが電子決済ドキュメントを信頼したわけではありませんでした。情報ポータルの機能は、おなじみの情報システムの機能と大差はなく、もちろん、高負荷のアーキテクチャーになることはありませんでした。



数年が経過し、ポータルが改善され、計測デバイスの読み取り、証明書の生成などの機会が現れました。最初の「クラウド」が地平線上に現れたのはこの時であり、ますます多くのユーザーがポータルに登録し、データを要求し始めました。遠くに、接近する負荷の最初の波が現れました。

チーム（以下、チーム1と呼びます） 次の最適化対策を講じました：

  • PDFの支払いドキュメントのサイズを0.5 MBから0.2 MBに変更
  • 支払文書の形成のためのキューの作成
  • 繰り返されるリクエストのために生成された支払いドキュメントの保存
  • ポータルのニーズにのみ対応する、メインデータベースのレプリカの作成

数年間、状況は安定しているように見え、個々のポータルのユーザー数は数千人で測定されましたが、まだ嵐にはなりませんでしたが、大幅に変動し、決済センターの分散化が開発者の手に渡りました。



ビッグジャンプステージ


歴史の新たなマイルストーンは、地域レベルでの地方自治体サービスポータルの開発でした。共和国の居住者の単一のエントリポイントを含めることは魅力的なアイデアでした。さらに、これにより、州のサイトの評価が、地域の各居住者がさまざまな種類のサービスを受ける最高の商業サイトまたは銀行サイトのレベルにまで上昇します。最も人気のあるものの1つは、住宅と共同サービスの支払いとメーターの読み取り値の組み込みです。

したがって、次のステップは、決済センターの運用情報を支払いドキュメントに表示された情報から分離することでした。そのために、簡単なデータ転送フォーマットと情報を保存するためのデータベースを開発し、5年間の運用に必要な保存容量の計算を行いました。

この段階での主な決定：

  • 運用データベースの情報部門。
  • このフォーマットを保存するためのデータベースの開発;
  • 地域の決済センターのデータを単一のデータベースに結合する。
  • ポータルプロセスとの統合、サービスの開発。

この段階で、ポータルはユーザーに単一のインターフェースを提供したため、ソリューションは本格的なものからバックエンドに変わりました。ポータルには独自のチームがあり、決済センターの現在のソリューションに関係なく、ポータルを開発しました。私たちの歴史では、2番目のチーム（以下、チーム2と呼びます）と新しいベンダーが登場しています。後で見るように、これは問題の開発と解決に大きな影響を与えました。設計ソリューションの本質は次のとおりです。



データベースを設計する際に、データベースへのフォーマットの単純なマッピングが受け入れられ、PostgreSQL 9.3がDBMSとして選択されました（当時は非常に最新のバージョンでした）。形式は9つのフラットファイルで構成されており、それぞれがCOPYコマンド（非常に迅速に読み取る）でポータルデータベースの特定の請求センター（各請求センターには独自の登録番号がありました）の多くのテーブルに読み込まれました。一部の決済センターでは、支払文書の作成に必要なレコード数が1,000,000に達しました。1年でこれは12,000,000になり、5年以上-60,000,000を超えました。このデータベースへの要求数は、地区ポータルのすべてのユーザーの合計に増加し、数万を構成します。考えるべきことがありました。

そのような経験がないチーム1 潜在的な負荷を減らすために次の手順を実行しました：

  • テーブルは、決済センターと分割の月によって分割されました。
  • データの読み込みプロセスは、さまざまな請求センターで時分割されます。

急成長している問題


ポータルが開始され、準備された計画は現実に直面しました：

  • ユーザー数がすぐに見積もりを超えました。
  • クエリはマスターテーブルに行きましたが、DBMSはまだすべてのテーブルをポーリングしました
  • データベースサーバーの誤った負荷。このサーバーには、他に類を見ないほど大規模なデータベースがあり、その情報はランダムな瞬間に発生し、それをロードすると、利用可能なすべてのリソースが使用されました
  • 非効率的な実装が原因で、支払いドキュメントを作成するサービスが遅くなりました
  • ユーザーからの負荷は月間で均等に分散されませんでしたが、次の2つの期間に集中しました。


この瞬間は投稿の冒頭で説明されています。問題はすべての場所で同時に発生したため、問題の診断は困難でした。チーム1とチーム2は、その管理することにより均等に愛され、状況から抜け出すために措置を講じたが、お互いに実質的に何も連絡がありませんでした：



チーム2は、論理的かつ有用なステップを取るように見えた：ユーザーがシステムに入った後に支払文書の形成は、すぐに要求されるようになりました、彼がページを適切な場所に移動している間、PDはすでに形成されており、完成したドキュメントをすばやくプルアップできると期待しています。

この時点で、チーム1は毎月1か月に1つの問題を英雄的に解決し、毎回、問題の根本が隠れていることがそこにあることを顧客に確信させました。

  • 最適化されたSQL（時々パフォーマンスの向上を受け取りました）;
  • ポータルのデータベースサーバーを他のデータベースから分離しました。
  • 別のアプリケーションで支払いドキュメントの作成を実行し、それを最適化しました。
  • マスターテーブルの魅力を修正し、PostgreSQLのバージョンを変更しました。
  • もう一度、アピールを修正しました（マスターテーブルから要求された特定のパーティションは1つだけです-時にはアクセラレーションも）。

チームの英雄的な努力が地元の成功につながりました（2〜3か月間、問題は解決されたようです）。しかし、現実は常に新しい導入ノートを投げかけました：

  • データベースにはすでに数年が含まれており、1 TBを超えています。
  • ユーザーの数はすでに数十万人でした。

闘争が進行している間、チーム2は自動サービステストをセットアップしました。これにより、パフォーマンスの問題が数分で判明し、問題は電子メールで自動的に最高レベルの制御にエスカレートしました。

この時点でチーム1：

  • データベースのアーカイブ時間が許容制限を超え始めた（サービスが利用できなくなった）。
  • サービスへの1回の呼び出しですぐに何ヶ月も影響を受けたため、多くのリクエストが生成されました。
  • テーブル（パーティション）の数が増加したため、DBMSはクエリの実行が悪化しました。
  • 機器の読み取りのみを行いたいユーザーは、まだ支払い文書の作成を要求しました（チーム2の決定を思い出してください）。

技術的な観点から見ると、ソリューションが限界に達しており、プロセスを最適化したり、テクノロジーを根本的に変更したりするために、ユーザーの行動の分析を開始するときがきました。

分析の結果（ここでは別の記事のトピックです）、次のアクションが実行されました。

  • 前の期間に電話がなかったため、データは過去3年間まで切り捨てられました。それ以来、古い時代は毎年断たれています。
  • 特定のパーティションを直接参照するようにマスターテーブルを再アピールしました（DBMSの負荷を軽減）。

プレゼント


現在、システムは非常に安定しており、規制の時間枠と非機能的特性の要件に適合していますが、雲が再び地平線に現れました。

  • ユーザー数のさらなる増加。
  • 世帯数の増加;
  • 提供されるサービスの複雑さの増加;
  • ユーザーが利用できるデータの細分性の向上。

したがって、計画が作成され、次の対策の実施が準備されています。

  • ユーザーの行動の分析：実際に支払文書を見ている人の数、携帯電話のように単純に金額を支払う人の数。
  • 負荷が最も少ないときに通常支払文書を表示するユーザーのための予備支払文書の作成。
  • 代替テクノロジーへの移行（はい、この時点に達した人の多くは、携帯電話のリソースに関する最初のすべての問題を解決するはるかに高度なソリューションを持っています）。



これは大きな楽観的な見方をすることができるようです。すべてうまくいきました：すぐにうまくやった人とうまくやった人。しかし、明日は新しいHighloadプロジェクト、新しいなじみのない問題があります。それらを今どのように準備するか、まだデータがない場合に何を想定できますか？

問題を解決するための体系的なアプローチ


経験をHighloadプロジェクトの問題に取り組むための方法論に変えることはできますか？奇妙なことに、答えはYESです。すべてがすでに発明されており、制約理論TOCの枠組みの中にあります。たった5つの簡単なステップ：
  1. システムの制限を見つけます。
  2. システムの制限を最大限に活用する方法を決定します。
  3. この決定に他のすべてに従属します。
  4. システムの制限を拡張します。
  5. 注意！前の4つのステップで制限が解除されている場合は、ステップ1に戻りますが、慣性によってシステムが制限されないようにします。

この理論の説明とステップの本質は、記事の最後にある文献に非常によく説明されていますが、私は現在のプロセスの枠組みの中で私のビジョンを書きます。

  1. 制限：支払いドキュメントの作成時刻。
  2. 解決策：データをダウンロードするときに、すべての支払いドキュメントを事前に生成します。
  3. 決定の下位：ユーザーに連絡するときに、完成した文書を発行します。
  4. 制限を延長する：支払文書の形成のための時間を最適化します。
  5. すべての世帯の事前作成システムを放棄し、新しい制限を定義します。

このようなアプローチは、プログラマーの人件費を最小限に抑えながら、ソリューション時間を数年短縮します。提案されたアプローチではそれほど重要ではないため、ポストは発生したすべての問題を示しているわけではなく、解決策の詳細もありません。
