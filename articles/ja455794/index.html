Springの概要、またはプロジェクト全体で@Autowiredと@Componentがあり、それが何であるかを理解していない場合の対処方法

あいさつ、ハブル！

この記事は、すでにJavaの学習を始めており、Java Coreの理解にある程度成功した人にも役立ちます。今では、Springという言葉を聞きました。そして、おそらく、複数回：少なくとも、Spring Frameworkの知識は、javistの多くの空席の説明に表示されます。この記事は、最初のステップである、そのような一般的なフレームワークの一般的な考え方を理解するのに役立ちます。

遠くから始めましょう。ロシア語での制御の反転-略式の制御の反転-IoCのようなものがあります。 IoCは、コードをルーズに近づける原則の1つです。 IoCは、外部コンポーネントに対する責任の一部の委任です。

IoCアプローチにはさまざまな実装があり、そのうちの1つである依存性注入、依存性注入に関心があります。これは何ですか、名前はそれ自体が意味しているので、例を使用してそれを開示してみます。一連の店舗の運営を自動化するアプリケーションを作成しています。Shop（store）およびSeller（seller）というクラスがあります。クラスセラーには、タイプショップのフィールドがあります-ショップはセラーが働く店です。そのため、依存症に直面しています。売り手はショップに依存しています。ここで、ShopオブジェクトがどのようにSellerオブジェクトに入るかを考えてみましょう。オプションがあります：

  • コンストラクタを介してそれを実装し、すぐに、セラーを作成するときに、彼が働いているストアを指定します。

public class Seller {
    
    private Shop shop;

    public Seller(Shop shop) {
        this.shop = shop;
    }
}

  • セッターを作成し、彼の呼び出しを使用して、売り手のストアを設定します。

public class Seller {

    private Shop shop;

    public void setShop(Shop shop) {
        this.shop = shop;
    }
}

リストされている2つのメソッドは、依存性注入の実装です。そして最後に、私たちは春に到着しました。依存関係を注入する別の方法を提供します。

一般的に言って、Springは多くの場合に非常に幅広いライブラリーです。Webアプリケーションをすばやく作成するためのSpring MVC、アプリケーションに認証を実装するためのSpring Security、データベースなどを操作するためのSpring Dataがあります。しかし、Spring IoCは独立しています。これは、調査中のトピックである依存性注入を実装する基本的なタイプのSpringです。Spring IoCは、別の理由で、Springライブラリの学習の最初に注目に値します。他のタイプのスプリングを使用した実際の作業の過程でわかるように、他のすべてのスプリングでは、Spring IoCがフレームワークとして使用されます。

Spring IoCの紹介は主にBeanから始めます。最も単純な言葉で
Beanは、Springによって作成されたクラスオブジェクトで、フィールド値として別のオブジェクトに埋め込むことができます。

より複雑な単語が必要ですか？そしてどうぞ：
Beanはクラスオブジェクトであり、特定のビジネス関数または内部のSpring関数を備えた完全なプログラム要素であり、そのライフサイクルはビンコンテナによって制御されます。

すでにご存じのとおり、ショップがショップを実装できるようにするには、ショップがビンになる必要があります。どのオブジェクトにBeanと呼ばれる誇り高い権利があるかをアプリケーションに通知する方法はいくつかあります。それらすべてがApplicationContextの概念につながります。ApplicationContextは春の中心です。原則として、アプリケーションの最初（「ライズ」）に作成され、Beanのライフサイクルを制御します。そのため、ビンコンテナーとも呼ばれます。

主なものに向かっています。 Spring IoCとそのApplicationContextサーバントが、SellerオブジェクトのShopフィールドを置き換えるように、クラスをどのように書き直す必要がありますか？このような：

@Component
public class Shop {
}


@Component
public class Seller {

    @Autowired
    private Shop shop;
}

単に？はるかに簡単です！エレガントに？かなり。ここで次のことが起こりました。Componentアノテーションは、アノテーションを付けているクラスがビンであることを春に伝えました。 Annotation Autowiredは、Springに注釈を付けるフィールドの値を置き換えるように要求しました。この操作は注入と呼ばれます。正確にどのような値が置き換えられますか？これについては後で詳しく説明します。まず、クラスがどのようにBeanになるかを理解します。

アプリケーションの最初に、すべてのApplicationContext Beanの保護者を引き上げる必要があることはすでに知っています。彼はすべてのビンを一度に作成します。ほとんど全て。事実、デフォルトでは、すべてのBeanは値singletonにintraspringプロパティスコープを持っています。 Intraspringovoe、文字通りの意味では彼はシングルトンではありません。これはSpringのシングルトンです。コンテキストが発生すると、Springは指定されたクラスから1つのbinオブジェクトを作成します。この動作を変更したい場合は、Springを使用すると、1つのクラスのBeanの作成時間とその数を制御できますが、今はそうではありません。

したがって、ApplicationContextを発生させると、すべてのビンが作成されます。コンテキストがどこに存在するかを正確に調べてみましょう。最も重要なのは、どのクラスからビンを作成するかを決定する方法です。いくつかのオプションがありますが、簡単にするために、そのうちの1つについて説明します。xmlファイルを使用した構成です。以下にその例を示します。

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xmlns:beans="http://www.springframework.org/schema/c"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
           http://www.springframework.org/schema/beans/spring-beans-4.3.xsd
           http://www.springframework.org/schema/context
           http://www.springframework.org/schema/context/spring-context.xsd">
    <bean id="product" class="main.java.Product"></bean>
    <context:component-scan base-package="main"/>
</beans>

このファイルは、2つの方法でBeanを作成する方法を示しています。最初のものは、たとえば手動です。ご覧のとおり、クラスを示すBeanタグがあります。これは豆です。このファイルにbeanタグで登録されているものすべてから、beansが作成されます。

2番目の方法は冗長度が低くなります。クラスの上にComponentアノテーションを付けたことを思い出してください。このアノテーションが付けられたすべてのクラスのうち、Beanが作成されます。xmlファイルの次の行に感謝します。

<context:component-scan base-package="main"/>

彼女は春に言います：メインパッケージ全体をスキャンし、コンポーネントアノテーションが立つすべてのもの（またはコンポーネントの継承者である他のアノテーション）から、ビンを作成します。コンパクトですね。Beanを作成するためのクラスを含むパッケージを示し、これらのクラスに注釈を付けるだけです。

次のコード行でxmlファイルを使用してコンテキストを上げることができます。

ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");

ここで、beans.xmlは、前述のxmlニックネームへのパスです。

豆の作成を考え出した。 Springはセラーを作成するときにショップフィールドにどのように入力しますか？コンテキストを上げると、Shopクラスのbinオブジェクトが作成されます。 Sellerクラスのbinオブジェクトも作成され、Componentによって注釈も付けられます。 Autowiredによって注釈が付けられたショップタイプがあります。 AutowiredアノテーションはSpringに通知します。このフィールドにビンを挿入する必要があります。この場合、このロールに適したビンは1つだけです。つまり、そのタイプはフィールドのタイプと一致します。これはビンであり、Shopクラスのインスタンスです。必要に応じて、Sellerオブジェクトに挿入されます。今では質問がワームのように増えていることを理解しています：Springが目的のBeanを見つけられなかった場合、または適切なものを見つけた場合はどうなりますか（特に、クラスではなく、インターフェースを介してインジェクトでき​​ることを考えると）春は賢いですが、私たちにも同じことが必要です。システムにビンを1つだけ持っている必要があります。各Autowiredに適しているか、そのような競合への対処方法をSpringに教えます（これについてはここで説明しません。すでに疲れていて、固まっています。記事は終わりに近づいています）。

売り手もビンです。それがビンではなく、newを介して作成された場合、何も自動的に挿入されません。

おそらく今、あなたはなぜこれらすべての困難が必要なのかを考えています。しかし、私たちのアプリケーションが2つのクラスではなく、数桁も大きく、依存関係の管理がもはや最も簡単なタスクではないことを想像してください。

たぶん今、あなたはSpringが依存関係を実装することをどれほど美しく、簡単にそして簡潔にできるかを考えています。しかし、何かがうまくいかず、アプリケーションをデバッグする必要があると想像してください。そして、それはもはやそれほど簡単ではありません...

最後に、いくつかのヒント：

  • プロジェクトを実装していて、春からビンを取得して確認する方法がわからなくなった場合は、次のようにします。

ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");
Seller seller = (Seller) context.getBean(Seller.class);

これはビンを取得するための合法的な方法ですが、現代の現実ではこれは通常行われません。しかし、ケーススタディではできます。

  • Springはフレームワークなので、プロジェクトに含める必要があります。mavenを使用してアプリケーションを作成し、pom.xmlファイルにspring-coreおよびspring-context依存関係を追加します。

Spring IoCには、Beanを作成、構成、および注入するための多大な機会が含まれています。春を操作する方法の1つである氷山の先端のごく一部を調べましたが、それを鋭く見て、何が起こっているのかについての一般的な考えが得られたことを願っています。
