DNSSECによるより安全なSSH接続


SSHを使用するすべての人は、サーバーに初めて接続するときに、キーフィンガープリントを確認するメッセージが表示されることを知っています。さらに、キーはクライアント側に保存され、このメッセージは保存されたキーが変更されるまで表示されません。しかし、この手順の実際的な意味は何ですか？

実際には、MiTM攻撃の可能性について実際に考えずにサーバーのSSHキーのフィンガープリントを確認する人はほとんどいません。SSHFP DNSレコードの登場により、サーバーキーのフィンガープリントをDNSに保存し、DNSSECを使用して認証できるようになりました。この場合、最初の接続でキーを確認する必要さえありません。この記事では、SSHサーバーのSSHFPレコードを構成する方法について説明します。

テストサーバー


まず、サーバーが必要です。RuVDSパネルでは、SSHアクセスの詳細がサーバーカードにすぐに表示されます。接続用のIPアドレスとパスワードを保存します。



コントロールパネルから直接ファイアウォールを設定して、IPに対してのみSSHアクセスを許可することもできます。

SSHFPを構成するには、ドメインをサーバーのIPアドレスに転送する必要があります。このドメインのDNSレコードを構成します。

SSHでのキーの仕組み


例では、OpenSSHパッケージのみを検討します。これが最も一般的なオプションであるためです。

新しいサーバーをインストールすると、ランダムなSSHキーが生成されます。通常、これはOpenSSHパッケージのインストール時、または既製のシステムイメージが使用されている場合は最初の起動時に発生します。

キーファイルはここにあります：

$ ls -la /etc/ssh/ssh_host_*_key*

/etc/ssh/ssh_host_dsa_key
/etc/ssh/ssh_host_dsa_key.pub
/etc/ssh/ssh_host_ecdsa_key
/etc/ssh/ssh_host_ecdsa_key.pub
/etc/ssh/ssh_host_ed25519_key
/etc/ssh/ssh_host_ed25519_key.pub
/etc/ssh/ssh_host_rsa_key
/etc/ssh/ssh_host_rsa_key.pub

このリストには、dsa、rsa、ecdsa、ed25519など、さまざまなタイプのキーが同時にいくつかあります。これは、異なるSSHクライアントとの互換性のために行われます。接続段階で、クライアントとサーバーは、どの鍵アルゴリズムが使用されるかについて合意します。クライアントは、提案されたアルゴリズムがサポートされていない場合、別のアルゴリズムを使用するようサーバーに要求する場合があります。サーバーはその鍵の公開部分をクライアントに送信し、ユーザーは自分の指紋を手動で確認するように求められます。


サーバーは接続時に公開鍵のフィンガープリントをクライアントに送信します。

これが最初の接続である場合は、フィンガープリントを手動で確認するように求めるメッセージがクライアントに表示されます。

The authenticity of host 'example.com (123.45.67.89)' can't be established.
ECDSA key fingerprint is SHA256:7Q4nIqjuo/lSXWFkt9RaJYVHrT6LUAc6KWrdQ4/DDeA.
Are you sure you want to continue connecting (yes/no/[fingerprint])?

この段階では、通常、誰でもためらうことなく[はい]をクリックし、キーの指紋をファイルに保存します~/.ssh/known_hosts。これで、サーバーのキーが変更されると、MiTM攻撃の可能性に関するメッセージがクライアントに表示されます。初めて、クライアントが自分でキーチェックを実行し、その信頼性を確認したと想定されています。

このアプローチは非常に危険です。攻撃者が最初の接続の瞬間を捕捉できた場合、攻撃者はキーを滑らせて接続を傍受することができるためです。 c HTTPSの場合、証明書を使用してサーバーキーを確認するサードパーティがあります。 SSHと同じアプローチがWebで使用された場合、誰もが鍵をチェックするのではなく、常に同意するため、鍵検証メッセージによって絶えず苦しめられ、MiTM攻撃は至る所に発生します。

キープリントをDNSに保存します。SSHFPエントリとは


サーバーによって提案されたSSHキーが本当に本物であり、これがMiTM攻撃ではないことを確認する方法は？実際、サーバーを入力してキーを検証するには、最初にパスワードを入力する必要があります。しかし、攻撃者はすぐにサーバーをハックして、サーバー上のすべてのデータを置き換えることができるため、キャッチに気付くことはありません。したがって、実際の接続の前にキーの信頼性を検証する方法が必要です。

長い間、接続する前にサーバーのSSHキーを確認する唯一の方法は、別のチャネルを使用することでした。たとえば、サーバー管理者にサーバーキーのフィンガープリントを提供するよう依頼してください。これは不便なので、問題を無視して、ためらうことなく常に同意する方が簡単です。


SSHFPは、SSHFP DNS経由で接続する前にサーバーキー認証を許可します

-SSHキーを保存するためのDNSレコードのタイプ。SSHキーのフィンガープリントは、TXTレコードのようにDNSサーバーに配置され、DNSSECキーで署名されます。つまり、ホスト名を使用して初めてサーバーに接続するとき、クライアントはDNS経由でサーバーキーのフィンガープリントを事前に知ることができ、提案されたサーバーと一致する場合は警告なしに接続します。

DNSSECを構成する


SSHFPを使用するには、DNSSECが構成されたドメイン名が必要です。DNSSEC機能ですぐにDNSコントロールパネルを提供する多くのパブリックDNSサービスがあります。そのような最も人気のあるサービスはCloudFlareです。彼の例を使用して構成を検討してください。次の手順では、ドメインをCloudflare NSサーバーに委任する必要があります。

▍ステップ1-キーを生成する


[DNS ]パネルに移動し、[DNSSECを有効にする]をクリックします。

この時点で、ドメインゾーンに署名するためのキーが生成されます。公開鍵が表示されます。それらは、ドメインレジストラーの側に追加する必要があります。

▍ステップ2-レジストラへの公開鍵の追加


次に、ドメインレジストラーからの公開キーを含むDSレコードを作成する必要があります。
ご利用のレジストラによっては、DNSSECキー追加インターフェイスの外観が異なる場合があります。CloudFlareに表示される名前とは異なる名前が付けられたり、名前が異なったりする可能性があるため、値を混同しないようにすることが重要です。

▍ステップ3-追加されたDSレコードを確認する


レジストラの側面にDSレコードを追加した後、設定が正しいことを確認できます。CloudFlare側では、レジストラ側でDSレコードを追加することの正当性の検証に合格した場合にのみ、DNSレコードの署名がアクティブになります。


DSレコードの追加を待機して

います数分または数時間後、レコードが正しく追加された場合は、そのようなメッセージが表示されます。つまり、DNS応答はDNSSECを使用して署名されるようになりました。

SSHFPエントリの追加


サーバーでSSHFPレコードを生成します。この例では、アドレスmyserver.comのサーバーを管理しています。このドメイン名については、以前DNSSECを構成しました。

サーバーで次のコマンドを実行します。

#   SSHFP   SSH-
sudo ssh-keygen -r myserver.com

myserver.com IN SSHFP 1 1 057ecce168ace29d5a0099e3b63df2850e4c8e20
myserver.com IN SSHFP 1 2 52cd6099a304b9b8f57f2cd914e96a1b7477eb2f88c98c602
myserver.com IN SSHFP 2 1 42d677482e4450ee515d3aac94d96302a99bd4ec
myserver.com IN SSHFP 2 2 edda5fa445dc0da570c772a6df0d4012037e1a102840d29c4
...

/ etc / ssh / フォルダーからすべてのキーの指紋が生成されます。ファイルパスを指定することにより、特定のキーのフィンガープリントを選択的に生成できます。

ここで、これらすべてのレコードをDNSパネルに追加する必要があります（この場合はCloudflare）。


CloudflareパネルへのSSHFPレコードの追加

したがって、前のステップで取得したすべてのキーを追加する必要があります。これで、キーが追加されたことを確認できます。

dig SSHFP myserver.com

答えには、追加されたすべてのキーが表示されます。新しいエントリの署名には時間がかかる場合があるため、応答のキーがすぐに表示されない場合があります。これには通常、10〜15分以上かかりません。

サーバーに接続する


SSHクライアントがDNSを介してキーの有効性を確認するためには、設定でこれを有効にする必要があります。クライアント構成は、ユーザーのホームフォルダーにあります。そこで1行追加します。

#  
vi ~/.ssh/config

VerifyHostKeyDNS yes

これでサーバーに接続できます。実験の純粋さのために、〜/ .ssh / known_hostsからフィンガープリントを含む行を削除できます。明確にするために、-vオプションを追加できます

#   
ssh -v root@myserver.com

すべてが正しく構成されている場合は、サーバーに初めて接続するときに、キーのフィンガープリントを手動で確認するように求められません。これには、システムのDNSリゾルバーがDNSSEC検証をサポートすることも必要です。

SSHFPは、ドメイン名でサーバーに接続されている場合にのみ機能し、IPまたはSSHFPレコードを持たない別のドメインで接続されている場合には機能しないことに注意してください。
