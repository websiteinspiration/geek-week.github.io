2.ネットワークロードバランサー。高負荷に適しており、OSIレベル4（EKSおよびECSで使用可能）で動作し、TCP、UDP、TLSがサポートされています。

ネットワークロードバランサーは、Amazon VPCのターゲットにトラフィックを転送し、超低レイテンシで毎秒数百万のリクエストを処理できます。さらに、ワークロードが突然変化するトラフィックパターンを処理するように最適化されています。

3. Application Load Balancer。 7番目のレベルで動作し、Lambdaサポートがあり、ヘッダーとパスレベルでルールをサポートし、HTTPとHTTPSをサポートします。
マイクロサービスやコンテナーなどの最新のアーキテクチャに基づいて構築されたアプリケーションの配信に焦点を当てた高度なクエリルーティングを提供します。リクエストのコンテンツに基づいて、Amazon VPCのターゲットにトラフィックを転送します。

多くのユーザーにとって、HTTPとは異なりTCPはそれほど一般的ではないため、最初にApplication Load BalancerがClassic Load Balancerを置き換えました。

それも作成してみましょう。その結果、すでに2つのロードバランサーが存在します。



負荷分散コンポーネント


ロードバランスの共通コンポーネント（すべてのバランサーに共通）：

  • アクセスログポリシー
-ELBアクセスログ。設定を行うには、[説明]に移動して[属性の編集]ボタンを選択します。



次に、S3Bucket-Amazonオブジェクトストレージを指定します。



  • スキーム
-内部または外部バランサー。重要なのは、LoadBalancerが外部からアクセスできるようにするために外部アドレスを取得する必要があるか、それとも内部バランサーであるかです。

  • セキュリティグループ
-バランサーへのアクセス制御。これは基本的に高レベルのファイアウォールです。





  • サブネット
-VPC内のサブネット（それぞれ、およびアベイラビリティーゾーン）。サブネットは作成時に指定されます。VPCがリージョンによって制限されている場合、サブネットはアベイラビリティゾーンによって制限されます。ロードバランサーを作成するときは、少なくとも2つのサブネットに作成することをお勧めします（1つのアベイラビリティーゾーンに問題がある場合に役立ちます）。

  • リスナー
-あなたのバランサープロトコル。前述のように、Classic Load Balancerの場合は、HTTP、HTTPS、TCP、SSL、Network Load Balancerの場合-TCP、UDP、TLS、Application Load Balancerの場合-HTTPおよびHTTPSです。

Load Balancer v2のコンポーネント（ALBおよびNLB）


次に、Application Load BalancerとNetwork Load Balancerの第2バージョンのバランサーを詳しく見てみましょう。これらのバランサーには、独自のコンポーネント機能があります。たとえば、ターゲットグループ-インスタンス（および関数）などのコンセプトが登場しました。このコンポーネントのおかげで、トラフィックを転送するターゲットグループを指定する機会があります。





簡単に言うと、ターゲットグループでは、トラフィックが来るインスタンスを指定します。同じクラシックロードバランサーで強度をすぐにバランサーに接続する場合は、アプリケーションロードバランサーで最初に以下を実行します。

  • ロードバランサーを作成する
  • ターゲットグループを作成します。
  • 必要なバランサーポートまたはルールを目的のターゲットグループに送信します。
  • ターゲットグループで、インスタンスを割り当てます。

この作業のロジックはもっと複雑に見えるかもしれませんが、実際にはより便利です。

次のコンポーネントはリスナールールです。これは、Application Load Balancerにのみ適用されます。 Network Load Balancerでリスナーを作成し、特定のターゲットグループにトラフィックを送信する場合、Application Load Balancerではすべてがより楽しく便利になります。



次に、次のコンポーネントであるElastic IP（NLBの静的アドレス）について少し説明します。ルーティングのリスナールールがApplication Load Balancerにのみ適用される場合、Elastic IPはNetwork Load Balancerにのみ適用されます。

Elastic IPは、時間の経過とともにさまざまなEC2インスタンスに関連付けることができる単一のIPアドレスを提供します。EC2インスタンスにElastic IPアドレスがあり、このインスタンスが完了または停止している場合、新しいEC2インスタンスをElastic IPアドレスにすぐに関連付けることができます。ただし、実際のEC2が変更された場合でも、アプリケーションは同じIPアドレスを参照するため、現在のアプリケーションは機能を停止しません。Elastic IPが必要な理由に関する別のユーザー事例を

結論を出す


ELBは、着信トラフィックを複数のターゲット（コンテナー、Amazon EC2インスタンス、IPアドレス、およびLambda機能）に自動的に分散します。 ELBは、1つのアベイラビリティーゾーンと複数のアベイラビリティーゾーンの両方で、負荷が変化するトラフィックを分散できます。ユーザーは、高可用性、自動スケーリング、適切な保護を提供する3種類のバランサーから選択できます。これらすべては、アプリケーションの回復力を確保するために重要です。
