総当たり攻撃やDoS攻撃からのZimbra OSEの保護

Zimbra Collaboration Suite Open-Source Editionには、情報セキュリティを確保するための強力なツールがいくつかあります。その中でも、Postscreenはボットネットによる攻撃からメールサーバーを保護するためのソリューションです。ClamAVは、マルウェアの感染や、SpamAssassinの着信ファイルや文字をスキャンできるアンチウイルスです。-今日最高のスパムフィルターの1つ。ただし、これらのツールでは、ブルートフォースなどの攻撃からZimbra OSEを保護することはできません。最もエレガントではありませんが、特殊な辞書でのパスワードの列挙は非常に効果的ですが、ハッキングが成功してその結果すべてが発生する可能性があるだけでなく、Zimbra OSEを使用してサーバーのハッキングに失敗したすべての試みを処理するサーバーに大きな負荷が発生します。



原則として、標準のZimbra OSEツールを使用して、ブルートフォースから身を守ることができます。パスワードセキュリティポリシー設定を使用すると、パスワードの試行に失敗した回数を設定できます。その後、攻撃される可能性のあるアカウントがブロックされます。このアプローチの主な問題は、ブルートフォース攻撃によって1人以上の従業員のアカウントがブロックされ、何もする必要がない状況が発生し、その結果生じる従業員の単純な作業が会社に大きな損失をもたらす可能性があることです。そのため、ブルートフォースに対する保護のこのオプションは適用しないのが最善です。



総当たり攻撃から保護するには、DoSFilterと呼ばれる特別なツールがはるかに優れています。これはZimbra OSEに組み込まれており、HTTPを介してZimbra OSEから自動的に切断できます。つまり、DoSFilterの動作原理はPostScreenの動作原理に似ており、別のプロトコルでのみ使用されます。もともと1人のユーザーが実行できるアクションの数を制限するように設計されていたDoSFilterは、ブルートフォースからの保護も提供できます。 Zimbraの組み込みツールとの主な違いは、一定の回数の試行が失敗した後、ユーザー自身をブロックするのではなく、1つまたは別のアカウントにログインするために複数回試行されるIPアドレスです。これにより、システム管理者はブルートフォースから身を守るだけでなく、企業の従業員をブロックすることも回避できます。信頼できるIPアドレスとサブネットのリストに会社の内部ネットワークを追加するだけです。

DoSFilterの大きな利点は、1つまたは別のアカウントへのログインを何度も試行することに加えて、このツールを使用して、従業員の認証データを取得したサイバー犯罪者を自動的にブロックし、彼のアカウントに正常にログインして、何百ものリクエストをサーバーに送信し始めたことです。

DoSFilterは、Zextras Suite Pro拡張機能を使用するときに多くの問題を引き起こす可能性があることに注意してください。それらを回避するために、コマンドzmprov mcf zimbraHttpDosFilterMaxRequestsPerSec 100を使用して、同時接続数を30から100に増やすことをお勧めします。さらに、企業の内部ネットワークを許可リストに追加することをお勧めします。これは、コマンドzmprov mcf + zimbraHttpThrottleSafeIPs 192.168.0.0/24を使用して実行できます。 DoSFilterに変更を加えた後、必ずzmmailboxdctl restartコマンドを使用してメールサーバーを再起動してください。

DoSFilterの主な欠点は、アプリケーションレベルで機能するため、攻撃者がサーバー上でさまざまなアクションを実行する機能を制限できるだけであり、北への接続機能を制限できないことです。このため、認証の要求またはサーバーに送信されるレターの送信は、明らかに失敗ではありますが、そのような高レベルで止めることができない古き良きDoS攻撃です。

Zimbra OSEを使用して企業サーバーを完全に保護するには、Fail2banなどのソリューションを使用できます。これは、繰り返されるアクションについて情報システムログを常に監視し、ファイアウォール設定を変更することで侵入者をブロックできるフレームワークです。このように低いレベルでブロックすると、サーバーへのIP接続の段階で攻撃者を無効にすることができます。したがって、Fail2Banは、DoSFilterで構築された保護を完全に補完できます。 Fail2Ban Zimbra OSEと友達になり、ITインフラストラクチャのセキュリティを向上させる方法を見てみましょう。

他のエンタープライズクラスのアプリケーションと同様に、Zimbra Collaboration Suite Open-Source Editionは、その作業の詳細なログを維持します。それらのほとんどは、ファイルとして/ opt / zimbra / log /フォルダーに保存されます。以下はそのほんの一部です。

  • mailbox.log-Jettyメールサービスログ
  • audit.log-認証ログ
  • clamd.log-ウイルス対策ログ
  • freshclam.log-ウイルス対策更新ログ
  • convertd.log-添付ファイルコンバータログ
  • zimbrastats.csv-サーバーパフォーマンスログ

Zimbraのログは/var/log/zimbra.logファイルにもあり、PostfixとZimbra自体のログが保持されています。

システムをブルートフォースから保護するために、mailbox.log、audit.log、およびzimbra.logを監視します。

すべてが機能するためには、Fail2BanとiptablesがZimbra OSEを使用するサーバーにインストールされている必要があります。 Ubuntuを使用している場合は、dpkg -s fail2banコマンドを使用してこれを実行できますが、CentOSを使用している場合は、yum listがインストールされたfail2banコマンドを使用してこれを確認できます。。 Fail2Banがインストールされていない場合、このパッケージはほとんどすべての標準リポジトリにあるため、インストールしても問題ありません。

必要なソフトウェアがすべてインストールされたら、Fail2Banの構成に進むことができます。これを行うには、構成ファイル/etc/fail2ban/filter.d/zimbra.confを作成します。このファイルでは、不正なログイン試行に対応し、Fail2BanメカニズムをトリガーするZimbra OSEログの正規表現を記述します。認証試行が失敗したときにZimbra OSEによって生成されるさまざまなエラーに対応する一連の正規表現を含むzimbra.confの内容の例を次に示します。

# Fail2Ban configuration file
 
[Definition]
failregex = \[ip=<HOST>;\] account - authentication failed for .* \(no such account\)$
\[ip=<HOST>;\] security - cmd=Auth; .* error=authentication failed for .*, invalid password;$
;oip=<HOST>;.* security - cmd=Auth; .* protocol=soap; error=authentication failed for .* invalid password;$
;oip=<HOST>;.* security - cmd=Auth; .* protocol=imap; error=authentication failed for .* invalid password;$
\[oip=<HOST>;.* SoapEngine - handler exception: authentication failed for .*, account not found$
WARN .*;ip=<HOST>;ua=ZimbraWebClient .* security - cmd=AdminAuth; .* error=authentication failed for .*;$

ignoreregex =

Zimbra OSEの正規表現がコンパイルされたら、Fail2ban自体の構成の編集を開始します。このユーティリティの設定は、/etc/fail2ban/jail.confファイルにあります。念のため、コマンドcp /etc/fail2ban/jail.conf /etc/fail2ban/jail.conf.bakを使用してバックアップします。その後、このファイルを次の形式にします。

# Fail2Ban configuration file
 
[DEFAULT]
ignoreip = 192.168.0.1/24
bantime = 600
findtime = 600
maxretry = 5
backend = auto
 
[ssh-iptables]
enabled = false
filter = sshd
action = iptables[name=SSH, port=ssh, protocol=tcp]
sendmail-whois[name=SSH, dest=admin@company.ru, sender=fail2ban@company.ru]
logpath = /var/log/messages
maxretry = 5
 
[sasl-iptables]
enabled = false
filter = sasl
backend = polling
action = iptables[name=sasl, port=smtp, protocol=tcp]
sendmail-whois[name=sasl, dest=support@company.ru]
logpath = /var/log/zimbra.log
 
[ssh-tcpwrapper]
enabled = false
filter = sshd
action = hostsdeny
sendmail-whois[name=SSH, dest=support@ company.ru]
ignoreregex = for myuser from
logpath = /var/log/messages
 
[zimbra-account]
enabled = true
filter = zimbra
action = iptables-allports[name=zimbra-account]
sendmail[name=zimbra-account, dest=support@company.ru ]
logpath = /opt/zimbra/log/mailbox.log
bantime = 600
maxretry = 5
 
[zimbra-audit]
enabled = true
filter = zimbra
action = iptables-allports[name=zimbra-audit]
sendmail[name=Zimbra-audit, dest=support@company.ru]
logpath = /opt/zimbra/log/audit.log
bantime = 600
maxretry = 5
 
[zimbra-recipient]
enabled = true
filter = zimbra
action = iptables-allports[name=zimbra-recipient]
sendmail[name=Zimbra-recipient, dest=support@company.ru]
logpath = /var/log/zimbra.log
bantime = 172800
maxretry = 5
 
[postfix]
enabled = true
filter = postfix
action = iptables-multiport[name=postfix, port=smtp, protocol=tcp]
sendmail-buffered[name=Postfix, dest=support@company.ru]
logpath = /var/log/zimbra.log
bantime = -1
maxretry = 5

Fail2Ban設定でファイルを保存した後は、service fail2ban restartコマンドを使用してこのユーティリティを再起動するだけです。再起動後、メインのZimbraログは、正規表現への準拠について常に監視され始めます。これにより、管理者は、侵入者がZimbra Collaboration Suiteのオープンソース版のメールボックスに侵入する可能性を事実上排除するだけでなく、Zimbra OSE内で実行されているすべてのサービスを保護し、不正アクセスの試みに注意することができます。
