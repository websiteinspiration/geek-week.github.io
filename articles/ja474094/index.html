ブロック図エディター-Vue.jsとMxGraphの友情について


どのようにしてすべてが始まったのですか？


私はフロントエンドの開発者ですが、開発を目指して、フルスタックアプリケーションを作成し、大金持ちになって貴重な経験を積むことにしました。

したがって、私はバックエンドの計画を開始し、データストレージ用にMongoDBを選択し、構造とフィールドの関係を計画する準備が整いました。

しかし、NoSQLデータベース用の飾り気のない、シンプルで十分に機能するスキーマエディターの欠如に直面しました。

- 番号？それで、私はビジネスをして、ライブラリを見つけて、インターフェースを投入します！
Fullstackのアイデアが背景に取り入れられ、私はデータベーススキーマの最も単純なエディターに取り組み始めました。
-ナイーブ...-でも少し後で気づきました。

探す


まず、この種のライブラリはほとんどが支払われ、それらの多くを必要とします！
第二に、無料のものは機能性に優れていないか、バグでいっぱいです！

3番目のMxGraphは見つかりませんでした。優れたdraw.ioサービスが書かれていて、それ自体がデータを視覚化および編集するための非常に便利なツールですが、インターネット上ではほとんど言及されていません。

パフォーマンス


Vue.js については、誰もが聞いたことがあると思いますが、念のため-JavaScript-ジェットスタイルのユーザーインターフェイスを作成するためのフレームワーク。

MxGraphは、さまざまなプロセス、ワークフローとBPM、データベースの視覚化、マッピングネットワークとテレコミュニケーション、マッピングアプリケーションとGIS、UML図、電子回路、VLSI、CAD、財務およびソーシャルネットワーク、組織図などを視覚化および編集するためのライブラリです。

互換性


MxGraphはかなり古い（古いものではありません）ツールなので、Vueプロジェクトに単純にnpmをインストールしても、完全な互換性は得られません。したがって、ネットワークを掘り起こし、自転車の生産を開始する必要がありました。

苦しんでいる解決策が見つかりました


Vueコンポーネントのインポートと埋め込みは次のようになります。

<script>
 import mxgraph from 'mxgraph';

    //           
    const graphConfig = {
        mxBasePath: '/mx/', //the path in mxClient.basePath.
        mxImageBasePath: '/mx/images', // the path in mxClient.imageBasePath.
        mxLanguage: 'en', // the language for resources in mxClient.language.
        mxDefaultLanguage: 'en', // the default language in mxClient.defaultLanguage.
        mxLoadResources: false, // if any resources should be loaded.  Default is true.
        mxLoadStylesheets: false, // if any stylesheets should be loaded.  Default is true
    };

    const {
        mxClient, mxUtils, mxEvent, mxEditor, mxRectangle, mxGraph, mxGeometry, mxCell,
        mxImage, mxDivResizer, mxObjectCodec, mxCodecRegistry, mxConnectionHandler
    } = mxgraph(graphConfig);

    window.mxClient = mxClient;
    window.mxUtils = mxUtils;
    window.mxRectangle = mxRectangle;
    window.mxGraph = mxGraph;
    window.mxEvent = mxEvent;
    window.mxGeometry = mxGeometry;
    window.mxCell = mxCell;
    window.mxImage = mxImage;
    window.mxEditor = mxEditor;
    window.mxDivResizer = mxDivResizer;
    window.mxObjectCodec = mxObjectCodec;
    window.mxCodecRegistry = mxCodecRegistry;
    window.mxConnectionHandler = mxConnectionHandler;

    var editor;

    // CustomUserObject
    window.CustomUserObject = function (name, type) {
        this.name = name || 'New Name';
        this.type = type || 'New Type';
        this.clone = function () {
            return mxUtils.clone(this);
        };
    };

    export default {
     // vue logic
    }
</script>

なぜ変数を作成し、ウィンドウに登録したのですか？


実際のところ、webpackはアセンブリ中に変数の名前を変更し、その後mxgraphはそれらを見つけることができません。したがって、特にmxgraphの場合は、それらをグローバルオブジェクトに配置します。

また、構成についても忘れないでください。特に、組み込みのmxEditorインターフェイスを使用する場合は、ライブラリのコンテンツに問題が発生します。

そして、なぜエディタ変数はvueから外れたのですか？


操作中、mxgraphメソッドのコンテキストにより、常にvue変数にアクセスできるとは限りません。そして、それらをグローバル変数に取り込むことで、時間といくつかの神経を大幅に節約できます。

カスタムデータオブジェクトがウィンドウに表示される理由


Mxgraphは、プロトタイプを使用して新しいデータオブジェクトを作成します。また、ウィンドウに書き込む必要があります。そうしないと、組み込みのスキーマのインポート/エクスポートで問題が発生します。

さらなる開発が容易になり、オブジェクトへのアクセスに関する問題が発生しなくなりました。
私はこのアプローチが理想的だとは思いません。誰かがこの互換性ソリューションの代替案を提供してくれたらうれしいです。

そして何が起こった？


データベーススキーマをモデリングするための小さなアプリケーション：nosqldbm.ru
これは、将来のプロジェクトのために、おおよそのデータベーススキーマを視覚化するのに役立ちます。

私の最初の出版物を読んでいただきありがとうございます。興味を持っていただければ幸いです。
