Résolution du problème de requête N + 1 sans augmenter la consommation de mémoire dans Laravel

L'un des principaux problèmes des développeurs lorsqu'ils créent une application avec ORM est la requête N + 1 dans leurs applications. Le problème de requête N + 1 n'est pas un moyen efficace d'accéder à la base de données lorsque l'application génère une demande pour chaque appel à l'objet. Ce problème se produit généralement lorsque nous obtenons une liste de données à partir d'une base de données sans utiliser de chargement différé, de chargement ardent. Heureusement, Laravel avec son ORM Eloquent fournit des outils pour un fonctionnement pratique, mais ils présentent certains inconvénients.

Dans cet article, nous considérons le problème N + 1, les méthodes pour le résoudre et l'optimisation de la consommation de mémoire.







Regardons un exemple simple de la façon d'utiliser le chargement rapide dans Laravel. Disons que nous avons une application Web simple qui affiche une liste des en-têtes des premiers articles des utilisateurs de l'application. Ensuite, la connexion entre nos modèles peut ressembler à ceci:

Class User extends Authenticatable
{
    public function articles()
    {
        return $this->hasMany(Aricle::class, 'writter_id');
    }

    public function first_article()
    {
        return $this->hasOne(Aricle::class, 'writter_id')->orderBy('created_at', 'asc');
    }

    public function scopeWithFirstArticle($query)
    {
        $query->addSelect(['first_article_id' => Article::select('id')
                ->whereColumn('writter_id', 'users.id')
                ->orderBy('created_at', 'asc')
                ->take(1)
        ])->with('first_article')
    }

}

public function getFirstArticleAtribute()

    if(!array_key_exists('first_article', $this->relations)) {
        $this->setRelation('first_article', $this->articles()->oldest()->first());

    return $this->getRelation('first_article');

En fait, nous n'avons pas implémenté le chargement paresseux pour la communication dynamique. Nous avons simplement attribué le résultat de la requête pour recevoir le premier article de l'utilisateur. Cela devrait fonctionner aussi bien lors de l'accès à la propriété first_article pour le chargement gourmand et le chargement paresseux.







Connexions dynamiques dans Laravel 5.X



Malheureusement, notre solution ne s'applique qu'à Laravel 6 et supérieur. Laravel 6 et les versions précédentes utilisent une implémentation différente de addSelect. Pour une utilisation dans les anciennes versions du framework, nous devons changer notre code. Nous devons utiliser selectSub pour exécuter la sous-requête:







public function scopeWithFirstArticle($query)
{
    if(is_null($query->toBase()->columns)) {
        $query->select([$query->toBase()->from . '.*']);
    }

    $query->selectSub(
                Article::select('id')
                ->whereColumn('writter_id', 'users.id')
                ->orderBy('created_at', 'asc')
                ->take(1)
                ->toSql(), 
            'first_article_id'
            )->with('first_article');
}
