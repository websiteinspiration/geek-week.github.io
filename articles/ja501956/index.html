etcd Kubernetes-clusterのデータを直接操作した経験（K8s APIなし）

クラスター内のサービスにアクセスできるようにするために、Kubernetesクラスターへのアクセスを提供するように要求するリクエストがお客様から頻繁に寄せられています。データベースやサービスに直接接続したり、ローカルアプリケーションをクラスター内のアプリケーションに接続したりするために...



たとえば、接続する必要があります。ローカルマシンからサービスへmemcached.staging.svc.cluster.local。この機会に、クライアントが接続するクラスター内のVPNを提供します。これを行うには、ポッド、サービスのサブネットを発表し、クラスターDNSをクライアントにプッシュします。したがって、クライアントがサービスに接続しようとすると、memcached.staging.svc.cluster.local要求はクラスターDNSに送信され、それに応答してクラスターサービスネットワークまたはポッドアドレスからこのサービスのアドレスを受信します。

K8sクラスターはkubeadmを使用して構成します。サービスサブネットはデフォルト192.168.0.0/16で、ポッドネットワークは10.244.0.0/16です。通常、すべてがうまく機能しますが、いくつかの点があります。

  • サブネットは192.168.*.*、クライアントのオフィスネットワークで使用されることが多く、さらに多くの場合、開発者のホームネットワークで使用されます。そして、競合が発生します。ホームルーターはこのサブネットで動作し、VPNはこれらのサブネットをクラスターからクライアントにプッシュします。
  • いくつかのクラスター（本番、ステージ、および/またはいくつかの開発クラスター）があります。次に、デフォルトでそれらすべてにポッドとサービスの同じサブネットが存在するため、複数のクラスターのサービスを同時に操作することが非常に困難になります。

かなり長い間、1つのプロジェクトのフレームワーク内でサービスとポッドに異なるサブネットを使用するという慣行を採用してきました-一般に、すべてのクラスターが異なるネットワークを持つように。ただし、多くのサービスやステートフルアプリケーションなどを実行しているため、最初からロールバックしたくないクラスターが多数稼働しています。

そして、私たちは自分自身に尋ねました：既存のクラスターのサブネットをどのように変更しますか？

決定の検索


最も一般的な方法は、ClusterIPタイプですべてのサービスを再作成することです。あるいは、彼らはこれを助言することもできます：

次のプロセスには問題があります。すべての構成が完了すると、ポッドは/etc/resolv.confにDNSネームサーバーとして古いIPを表示します。
それでも解決策が見つからなかったため、kubeadm resetを使用してクラスター全体をリセットし、再度初期化する必要がありました。

しかし、これはすべての人に適しているわけではありません...ここでは、私たちのケースのより詳細な導入ノートを示します。

  • フランネルが使用。
  • 雲の中と鉄の上にクラスターがあります。
  • クラスタ内のすべてのサービスが繰り返しデプロイされるのを避けたいです。
  • すべてを最小限の問題で行う必要があります。
  • Kubernetesのバージョンは1.16.6です（ただし、以降のアクションは他のバージョンでも同様です）。
  • 主なタスクは、サービスサブネットを持つkubeadmを使用してデプロイされたクラスターに192.168.0.0/16置き換えること172.24.0.0/16です。

「：そして、ちょうどので、私たちはどのようにKubernetesにそれがetcdに格納され、何がすべてでそれで何ができるかと...私たちは思ったので見に長い時間のためにそれが面白かったということが起こった理由だけではなく、古いIPアドレス（サブネット）を置き換えることにより、etcd内のデータを更新しません新しいものに？」

etcdでデータを操作するための既製のツールを探したところ、タスクを完全に解決するものは見つかりませんでした。（ちなみに、etcdでデータを直接操作するためのユーティリティについて知っている場合は、リンクに感謝します。）ただし、OpenShift etcdhelper は良い出発点になりました（その作成者に感謝します！）。

このユーティリティは、証明書を使用してetcdとコマンドを使用してデータを読み出すために接続することができls、get、dump。

etcdhelperを追加する


次の考えは論理的です。「このユーティリティを追加して、etcdにデータを書き込む機能を追加することを妨げるものは何か」

これは、2つの新機能changeServiceCIDRと機能を備えたetcdhelperの修正バージョンに翻訳されていますchangePodCIDR。ここで彼女のコードを見ることができます。

新機能は何をしますか？アルゴリズムchangeServiceCIDR：

  • デシリアライザを作成します。
  • 正規表現をコンパイルしてCIDRを置き換えます。
  • クラスタ内のClusterIPタイプのすべてのサービスを通過させます。

    • 値をetcdからGoオブジェクトにデコードします。
    • 正規表現を使用して、アドレスの最初の2バイトを置き換えます。
    • 新しいサブネットからIPアドレスをサービスに割り当てます。
    • シリアライザを作成し、Goオブジェクトをprotobufに変換し、新しいデータをetcdに書き込みます。

機能changePodCIDRは基本的に同じchangeServiceCIDRです。サービス仕様を編集する代わりに、ノードに対して実行.spec.PodCIDRして新しいサブネットに変更します。

練習


serviceCIDRを変更する


タスクの実装計画は非常に単純ですが、クラスター内のすべてのポッドの再作成時にダウンタイムが発生します。主な手順を説明した後、理論的には、この単純な手順を最小化する方法についての考えも共有します。

準備アクション：

  • 必要なソフトウェアをインストールし、パッチを当てたetcdhelperをアセンブルします。
  • バックアップetcdおよび/etc/kubernetes。

serviceCIDRを変更するための短いアクションプラン：

  • apiserverおよびcontroller-managerマニフェストの変更
  • 証明書の再発行。
  • ClusterIPサービスはetcdで変更されました。
  • クラスタ内のすべてのポッドを再起動します。

以下は、完全な一連のアクションの詳細です。

1.データダンプ用のetcd-clientをインストールします。

apt install etcd-client

2. etcdhelperを収集します。

  • 私たちはgolangを置きます：

    GOPATH=/root/golang
mkdir -p $GOPATH/local
curl -sSL https://dl.google.com/go/go1.14.1.linux-amd64.tar.gz | tar -xzvC $GOPATH/local
echo "export GOPATH=\"$GOPATH\"" >> ~/.bashrc
echo 'export GOROOT="$GOPATH/local/go"' >> ~/.bashrc
echo 'export PATH="$PATH:$GOPATH/local/go/bin"' >> ~/.bashrc
  • 私たちは自分自身を保存しetcdhelper.go、依存関係をロードし、収集します：

    wget https://raw.githubusercontent.com/flant/examples/master/2020/04-etcdhelper/etcdhelper.go
go get go.etcd.io/etcd/clientv3 k8s.io/kubectl/pkg/scheme k8s.io/apimachinery/pkg/runtime
go build -o etcdhelper etcdhelper.go

3.バックアップetcdを作成します。

backup_dir=/root/backup
mkdir ${backup_dir}
cp -rL /etc/kubernetes ${backup_dir}
ETCDCTL_API=3 etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt --key=/etc/kubernetes/pki/etcd/server.key --cert=/etc/kubernetes/pki/etcd/server.crt --endpoints https://192.168.199.100:2379 snapshot save ${backup_dir}/etcd.snapshot

4. Kubernetesコントロールプレーンマニフェストでサービスサブネットを変更します。ファイル内/etc/kubernetes/manifests/kube-apiserver.yamlで/etc/kubernetes/manifests/kube-controller-manager.yaml、パラメータ--service-cluster-ip-rangeを新しいサブネットに変更します。172.24.0.0/16代わりに192.168.0.0/16。

5. kubeadmがapiserver（含む）の証明書を発行するサービスサブネットを変更しているため、再発行する必要があります。

  1. 現在の証明書が発行されているドメインとIPアドレスを見てみましょう。

    openssl x509 -noout -ext subjectAltName </etc/kubernetes/pki/apiserver.crt
X509v3 Subject Alternative Name:
    DNS:dev-1-master, DNS:kubernetes, DNS:kubernetes.default, DNS:kubernetes.default.svc, DNS:kubernetes.default.svc.cluster.local, DNS:apiserver, IP Address:192.168.0.1, IP Address:10.0.0.163, IP Address:192.168.199.100
  2. kubeadmの最小構成を準備します。

    cat kubeadm-config.yaml
apiVersion: kubeadm.k8s.io/v1beta1
kind: ClusterConfiguration
networking:
  podSubnet: "10.244.0.0/16"
  serviceSubnet: "172.24.0.0/16"
apiServer:
  certSANs:
  - "192.168.199.100" # IP-
  3. これがないと新しい証明書が発行されないため、古いcrtとキーを削除してみましょう。

    rm /etc/kubernetes/pki/apiserver.{key,crt}
  4. APIサーバーの証明書を再発行します。

    kubeadm init phase certs apiserver --config=kubeadm-config.yaml
  5. , :

    openssl x509 -noout -ext subjectAltName </etc/kubernetes/pki/apiserver.crt
X509v3 Subject Alternative Name:
    DNS:kube-2-master, DNS:kubernetes, DNS:kubernetes.default, DNS:kubernetes.default.svc, DNS:kubernetes.default.svc.cluster.local, IP Address:172.24.0.1, IP Address:10.0.0.163, IP Address:192.168.199.100
  6. API- :

    docker ps | grep k8s_kube-apiserver | awk '{print $1}' | xargs docker restart
  7. admin.conf:

    kubeadm alpha certs renew admin.conf
  8. etcd:

    ./etcdhelper -cacert /etc/kubernetes/pki/etcd/ca.crt -cert /etc/kubernetes/pki/etcd/server.crt -key /etc/kubernetes/pki/etcd/server.key -endpoint https://127.0.0.1:2379 change-service-cidr 172.24.0.0/16

    ! , pod' /etc/resolv.conf CoreDNS (kube-dns), kube-proxy iptables . .
  9. ConfigMap' kube-system:

    kubectl -n kube-system edit cm kubelet-config-1.16

    — clusterDNS IP- kube-dns: kubectl -n kube-system get svc kube-dns.

    kubectl -n kube-system edit cm kubeadm-config

    — data.ClusterConfiguration.networking.serviceSubnet .
  10. kube-dns, kubelet :

    kubeadm upgrade node phase kubelet-config && systemctl restart kubelet
  11. クラスター内のすべてのポッドを再起動する必要があります。

    kubectl get pods --no-headers=true --all-namespaces |sed -r 's/(\S+)\s+(\S+).*/kubectl --namespace \1 delete pod \2/e'

最小限のダウンタイム


ダウンタイムを最小限に抑える方法についての考え：

  1. コントロールプレーンのマニフェストを変更した後、たとえば名前kube-dns-tmpと新しいアドレスを使用して新しいkube-dnsサービスを作成します172.24.0.10。
  2. 作るifKUBE-DNSサービスを変更しないであろう、etcdhelperに。
  3. すべてのkubeletのアドレスをClusterDNS新しいアドレスに置き換えますが、古いサービスは新しいアドレスと同時に動作し続けます。
  4. アプリケーションを持つポッドが自然な理由で単独で、または合意された時間にローリングするのを待ちます。
  5. サービスkube-dns-tmpを削除serviceSubnetCIDRし、kube-dnsサービス用に変更します。

この計画により、サービスが削除され、サービスkube-dns-tmpのサブネットが置き換えられるまでのダウンタイムが最大で1分程度まで短縮されますkube-dns。

実際、podCIDRの変更は簡単にできます（たとえば、このように）。しかし、etcdでKubernetesオブジェクトを編集することが唯一の可能なオプションである場合があるため、etcdを直接操作する方法を学びたかったのです。（たとえば、ダウンタイムなしでサービス分野を変更することはできませんspec.clusterIP。）

合計


この記事では、etcd内のデータを直接操作する可能性について検討しています。Kubernetes APIをバイパスします。このアプローチでは、「トリッキーなこと」ができる場合があります。本文で説明されている操作は、実際のK8sクラスターでテストされました。ただし、広く使用するための準備状況はPoC（概念実証）です。したがって、クラスターでetcdhelperユーティリティーの変更バージョンを使用したい場合は、ご自身の危険とリスクで行ってください。
