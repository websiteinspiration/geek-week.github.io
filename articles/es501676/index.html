¿Comparar operadores C # ?: Vs if-else vs switch

Hoy, otra fábrica de animales voló a la revisión:

public static class AnimalsFactory
{
    public static Animal CreateAnimalByTernaryOperator(bool isCat)
    {
        return isCat ? (Animal)new Cat() : new Dog();
    }
}

Una vez más, me molestó que C # te obligara a lanzar el objeto Cat a Animal. Pero dejemos que la casta sea mejor, porque a través de la declaración if-else el código es aún más largo:
public static class AnimalsFactory
{
    public static Animal CreateAnimalByIfElseOperator(bool isCat)
    {
        if (isCat)
            return new Cat();

        return new Dog();
    }
}

Pasemos un momento de la revisión e intentemos resolverlo:

  • ¿El código IL será diferente en estos ejemplos?
  • ¿Se beneficiará uno de los ejemplos en rendimiento?


La respuesta a la primera pregunta es sí, el código IL es diferente, mostraré a continuación qué.

Pasemos al tema del rendimiento. Descargue el paquete nuget para los puntos de referencia de BenchmarkDotNet y escriba una prueba:

public class AnimalFactoryPerformanceTests
{
    [ParamsAllValues]
    public bool IsCat { get; set; }

    [Benchmark]
    public void CreateAnimalByTernaryOperator() =>
        AnimalsFactory.CreateAnimalByTernaryOperator(IsCat);

    [Benchmark]
    public void CreateAnimalByIfElseOperator() =>
        AnimalsFactory.CreateAnimalByIfElseOperator(IsCat);
}

Resultados de referencia:

|                          | IsCat |     |
|------------------------------ |------ |---------:|
| CreateAnimalByTernaryOperator | False | 1.357 ns |
| CreateAnimalByTernaryOperator |  True | 1.655 ns |
|------------------------------ |------ |---------:|
|  CreateAnimalByIfElseOperator | False | 1.636 ns |
|  CreateAnimalByIfElseOperator |  True | 1.360 ns |

Es sorprendente que el ternario “para un perro” funcione más rápido y “para un gato”, una declaración de si no.
Observamos el código IL del método con el operador ternario:

CreateAnimalByTernaryOperator(bool isCat)
{
    IL_0000: ldarg.0      // isCat
    IL_0001: brtrue.s     IL_0009
    IL_0003: newobj       instance void AnimalPerformance.Dog::.ctor()
    IL_0008: ret
    IL_0009: newobj       instance void AnimalPerformance.Cat::.ctor()
    IL_000e: ret
}

Al crear el objeto Dog , los comandos IL_0000 - IL_0008 se ejecutarán secuencialmente , mientras que al crear el objeto Cat , se produce un salto condicional ( IL_0001: brtrue.s IL_0009 ).

Como puede suponer, para la instrucción if-else, se genera un código IL que no requiere saltos condicionales para crear un objeto Cat . Mientras el objeto Perro se crea a través de una rama condicional:

CreateAnimalByIfElseOperator(bool isCat)
{
    IL_0000: ldarg.0      // isCat
    IL_0001: brfalse.s    IL_0009
    IL_0003: newobj       instance void AnimalPerformance.Cat::.ctor()
    IL_0008: ret
    IL_0009: newobj       instance void AnimalPerformance.Dog::.ctor()
    IL_000e: ret
}

Agregue la creación de Parrot y un nuevo método con la instrucción switch a la fábrica :

public static class AnimalFactory
{
    public static Animal CreateAnimalByTernaryOperator(AnimalType animalType)
    {
        return animalType == AnimalType.Cat
            ? new Cat()
            : animalType == AnimalType.Dog
                ? (Animal)new Dog()
                : new Parrot();
    }

    public static Animal CreateAnimalByIfElseOperator(AnimalType animalType)
    {
        if (animalType == AnimalType.Cat)
            return new Cat();

        if (animalType == AnimalType.Dog)
            return new Dog();

        return new Parrot();
    }

    public static Animal CreateAnimalBySwitchOperator(AnimalType animalType)
    {
        switch (animalType)
        {
            case AnimalType.Cat:
                return new Cat();
            case AnimalType.Dog:
                return new Dog();
            case AnimalType.Parrot:
                return new Parrot();
            default:
                throw new InvalidOperationException();
        }
    }
}

¿Qué método será más rápido?

Resultados de referencia

|------------------------------ |----------- |---------:|
| CreateAnimalByTernaryOperator |        Cat | 2.490 ns |
| CreateAnimalByTernaryOperator |        Dog | 2.515 ns |
| CreateAnimalByTernaryOperator |     Parrot | 2.333 ns |
|------------------------------ |----------- |---------:|
|  CreateAnimalByIfElseOperator |        Cat | 2.368 ns |
|  CreateAnimalByIfElseOperator |        Dog | 2.545 ns |
|  CreateAnimalByIfElseOperator |     Parrot | 2.735 ns |
|------------------------------ |----------- |---------:|
|  CreateAnimalBySwitchOperator |        Cat | 2.747 ns |
|  CreateAnimalBySwitchOperator |        Dog | 2.730 ns |
|  CreateAnimalBySwitchOperator |     Parrot | 2.722 ns |

Conclusión 1: Para sentencias ternarias y if-else, el tiempo de ejecución depende directamente del número de saltos condicionales que han pasado en el hilo de ejecución.

Conclusión 2: La declaración de cambio de C # se convierte en código IL a una declaración de cambio y, en promedio, funciona un poco más que las declaraciones de ramificación regulares.

Conclusión 3: Todo lo anterior es relevante solo para .NET Framework v.4.8. Después de ejecutar las mismas pruebas en .NetCore, obtuvimos resultados completamente diferentes que aún no se han interpretado de alguna manera.
