Zimbra Collaboration Suiteでメールストレージを最適化する

以前の記事の 1つ企業でのZimbra Collabortion Suiteの実装中のインフラストラクチャ計画に特化しており、このソリューションの作業における主な制限はメールストレージのディスクデバイスの入出力速度であると言われていました。実際、企業の数百人の従業員が同じメールストレージに同時にアクセスするとき、ハードドライブから情報を読み書きするためのチャネル幅は、応答性の高いサービスには十分でない場合があります。また、Zimbraの小規模なインストールでこれが特に問題にならない場合は、大企業やSaaSプロバイダーの場合、これらすべてが応答しない電子メール作業につながり、結果として従業員の効率が低下するだけでなく、SLA違反にもつながります。そのため、大規模なZimbraインストールを設計および運用する場合は、メールストレージ内のハードディスクの動作を最適化する問題に特に注意を払う必要があります。 2つのケースを見て、それぞれに適用できるディスクストレージの負荷を最適化する方法を見つけてみましょう。



1.大規模なZimbraインストールを設計する際の最適化

高負荷のZimbraインストールの設計段階で、管理者は使用するデータストレージシステムを選択する必要があります。この問題を決定するには、ハードドライブの主な負荷が、Zimbra Collaboration Suiteの一部であるMariaDB DBMS、Apache Lucene検索システム、およびブロブストレージによって作成されることを知っておく必要があります。そのため、これらのソフトウェア製品を高負荷で動作させるには、高速で信頼性の高い機器を使用する必要があります。

通常の状態では、ZimbraはハードドライブからのRAIDとNFSプロトコルを介して接続されたストレージの両方にインストールできます。非常に小規模なインストールの場合、Zimbraを通常のSATAドライブにインストールできます。ただし、大規模なインストールでは、これらのテクノロジーすべてに書き込み速度の低下や信頼性の低下などのさまざまな欠点があり、大企業、特にSaaSプロバイダーには受け入れられません。

そのため、大規模インフラストラクチャの状況では、ZimbraはSANを使用するのが最適です。現在、ストレージデバイスに最大の帯域幅を提供できると同時に、多数のキャッシュを接続できるため、その使用は企業にとって大きなリスクを伴いません。多くのSANで使用されているNVRAMを使用して、記録パフォーマンスを高速化することをお勧めします。ただし、ディスク自体に記録されたデータのキャッシュを無効にすることをお勧めします。電源障害が発生した場合、メディアに修復不可能な損傷を与え、データが失われる可能性があるためです。

ファイルシステムの選択については、Linux Ext3 / Ext4の標準を使用するのが最適です。ファイルシステムに関連する主なニュアンスは、-noatimeオプションを使用してマウントする必要があることです。このパラメーターは、ファイルへの最後のアクセス時刻を修正する機能を無効にします。つまり、読み取りと書き込みの負荷が大幅に軽減されます。一般に、Zimbraのext3またはext4ファイルシステムを作成するときは、mke2fsユーティリティの次のパラメーターを使用する必要があります。

-j — Create the file system with an ext3/ext4 journal.
-L — , /etc/fstab
-O dir_index —
-m 2 — 2%
-J size=400 —
-b 4096 —
-i 10240 — . ,

さらに、ブロブストレージ、Lucene検索メタデータストレージ、およびMTAキューストレージのディレクトリ同期を有効にすることをお勧めします。これは、通常Zimbraがfsyncユーティリティを使用して、データを含むBLOBをディスクに書き込むことを保証するためです。ただし、ZimbraメールボックスまたはMTAがメッセージ配信中に新しいファイルを作成する場合、対応するフォルダーで発生した変更をディスクに書き込む必要があります。そのため、fsyncを使用してファイルがすでにディスクに書き込まれている場合でも、ディレクトリへの追加に関する書き込みはディスクに書き込むことができず、その結果、突然のサーバー障害が原因で失われる可能性があります。dirsyncを使用すると、これらの問題を回避できます。

2.機能しているZimbraインフラストラクチャを使用した最適化Zimbra

を数年稼働させた後、ユーザー数が大幅に増加し、サービスの応答性が日々低下していることがよくあります。この状況からの脱出方法は明白です。サービスを以前と同じように高速に動作させるには、インフラストラクチャに新しいサーバーを追加する必要があるだけです。一方、速度を上げるためにインフラストラクチャに新しいサーバーをすぐに追加することは、常に可能とはほど遠いものです。多くの場合、IT管理者は経理部門やセキュリティ部門と長い間新しいサーバーの購入を調整する必要があります。さらに、サプライヤーは新しいサーバーを遅らせたり、間違ったものを提供したりすることができません。

もちろん、拡張のマージンを常に確保し、誰にも依存しないようにするために、Zimbraインフラストラクチャをマージンを持って構築するのが最善です。たとえば、ITマネージャーは、運用中に定期的にハードディスクにアクセスするLinuxシステムサービスを一時的に無効にすることで、生産性を少し向上させることができ、その結果、Zimbraの速度に悪影響を与える可能性があります。したがって、しばらくの間、以下を無効にすることができます。

これらのサービスを無効にした結果としてシステムリソースを節約することはそれほど重要ではありませんが、不可抗力に近い状況では非常に役立ちます。新しいサーバーをZimbraインフラストラクチャに追加した後、以前に無効にしたサービスを再度有効にすることをお勧めします。

また、syslogサービスを別のサーバーに移動して、操作中にメールストレージのハードドライブを読み込まないようにすることで、Zimbraの操作を最適化することもできます。これらの目的には、安価なシングルボードのRaspberry Piまで、ほとんどすべてのコンピューターが適しています。
