DBMS MS SQL Server Developer 2016およびPostgreSQL 10.5 for 1Cのパフォーマンスの調査

「1Cアカウンティング」をテストするための目標と要件


テストの主な目的は、他の同一条件下での2つの異なるDBMS上の1Cシステムの動作を比較することです。それら。1Cデータベースの構成と初期データの母集団は、各テストで同じでなければなりません。

テスト中に取得する必要のある主なパラメーター：

  • 各テストの実行時間（開発部門1Cにより削除）
  • テスト中のDBMSおよびサーバー環境への負荷は、DBMS管理者によって削除されるだけでなく、システム管理者によってサーバー環境によっても削除されます。

1Cシステムのテストは、クライアント/サーバーアーキテクチャを考慮して実行する必要があります。したがって、システム内のユーザーまたは複数のユーザーの作業をエミュレートして、インターフェイスでの情報の入力を処理し、この情報をデータベースに保存する必要があります。同時に、累積レジスターに合計を作成するには、大量の定期的な情報を長期間にわたってポストする必要があります。

テストを実行するために、1Cシステムへのテストデータのシリアル入力が実行される1Cアカウンティング3.0の構成用に、スクリプトテスト用のスクリプトの形式でアルゴリズムが開発されました。スクリプトを使用すると、実行するアクションのさまざまな設定とテストデータの量を指定できます。以下の詳細な説明。

テストされた環境の設定と特性の説明

備考


この分析は、異なるDBMSでの1Cパフォーマンスを比較するためだけに行われたものであることに注意してください。

この分析と結論は、上記の条件とソフトウェアバージョンでの1Cアカウンティングに対してのみ正しいものです。得られた分析に基づいて、他の設定やソフトウェアバージョン、および異なる1C構成で何が起こるかを正確に結論付けることは不可能です。

ただし、Gilevテストの結果は、適切な設定を使用した1Cバージョン8.3以降のすべての構成で、パフォーマンスの最大ドローダウンが、MS SQL DBMSに比べてPostgreSQL DBMSで15％以下になる可能性が高いことを示しています。正確な比較のための詳細なテストには、かなりの時間とリソースがかかることも考慮する価値があります。これに基づいて、私たちはより可能性の高い仮定を行うことができます1Cバージョン8.3以降は、MS SQLからPostgreSQLに移行でき、最大15％のパフォーマンス損失があります。移行に対する客観的な障害はありませんでした。これらの15％までは表示されない可能性があり、それらが現れた場合は、必要に応じてもう少し強力な機器を購入するだけで十分です。

また、テスト対象のデータベースが小さく、データサイズが100 GBを大幅に下回り、同時に実行されるスレッドの最大数が4であったことにも注意してください。これは、サイズが100 GBを大幅に超える大きなデータベース（たとえば、約1 TB）の場合です。 、およびアクセスが集中するデータベース（同時アクティブスレッドが数十、数百）の場合は、これらの結果が正しくない場合があります。

より客観的な分析のために、同じCentOS OSにインストールされているリリース済みのMS SQL Server 2019 DeveloperとPostgreSQL 12を比較したり、MS SQLが最新バージョンのWindows Server OSにインストールされている場合と比較したりすることは将来的に役立ちます。現在、誰もPostgreSQLをWindowsに配置していないため、PostgreSQL DBMSのパフォーマンス低下は非常に重要になります。

もちろん、Gilevテストは、一般的に1Cだけでなく、パフォーマンスについても話します。ただし、現時点では、十分な事実がないため、MS SQL DBMSは常にPostgreSQL DBMSよりもはるかに優れているとは言えません。このステートメントを確認または反駁するには、他のいくつかのテストを実行する必要があります。たとえば、.NETの場合、アトミックアクションと複雑なテストの両方を記述し、それらを繰り返してさまざまな条件で実行し、実行時間を修正して平均値を取得する必要があります。次に、これらの値を比較します。これは客観的な分析になります。

現時点では、そのような分析を行う準備はできていませんが、将来的には可能になる可能性があります。次に、PostgreSQLがMS SQLより優れている操作とその割合（％）、およびMS SQLがPostgreSQLより優れている場所（および割合）について詳しく説明します。

また、テストでは、ここで説明する MS SQLの最適化方法を適用しませんでした。おそらく、この記事ではWindows Disk Indexingをオフにするのを忘れただけかもしれません。

2つのDBMSを比較する場合、もう1つの重要な点に注意してください。PostgreSQLDBMSは無料でオープンであり、MS SQL DBMSは有料であり、ソースコードはクローズされています。

今、Gilevテスト自体を犠牲にして。テスト以外では、模擬テスト（最初のテスト）と他のすべてのテストのトレースが削除されました。最初のテストでは、主にアトミック操作（挿入、更新、削除、読み取り）と複雑な（複数のテーブルを参照し、データベース内のテーブルの作成、変更、削除）の両方を、さまざまな量の処理データでクエリします。したがって、Gilev模擬テストは、2つの環境（DBMSを含む）の平均統合パフォーマンスを相互に比較するための非常に客観的なものと見なすことができます。絶対値自体は何も言いませんが、2つの異なるメディアの比率は非常に客観的です。

他のGilevテストを犠牲にして。トレースは、スレッドの最大数が7であることを示していますが、ユーザー数に関する結論は50を超えていました。また、要求によって、他のインジケーターの計算方法が完全に明確ではありません。したがって、残りのテストは客観的ではなく、非常に多様で概算です。システム自体だけでなく、ユーザー自身の作業の詳細も考慮した特殊なテストのみが、より正確な値を提供します。
