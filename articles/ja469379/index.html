正式なUI仕様


適切なソフトウェアを作成するには、優れたユーザーインターフェイスが不可欠です。また、ユーザーがプログラムの使用に問題を抱えている場合、何かがおかしい可能性があります。私はユーザーインターフェイスで作業しないことを好みます-私はこれより優れているとは思いませんが、それは「私のものではありません」。視覚効果とインターフェースは私に懸念を引き起こし、これらすべてに対処できる人々は私にクレイジーな敬意を引き起こします。

私は形式的な方法が大好きです。私の友人であるケビン・リンナーは最近sketch.systemsをリリースしました 1、UIデザイナー向けの新しい正式な仕様ツール。さて、調べてみましょう-私の正式な方法への愛は私の恐怖を克服することができますか？

スナップショットでは、教師は生徒間で「投票」または「スナップ」を行うことができます。さらに、プログラムは調査結果を集約し、「要約」レポート、「学生別」、および「標準別」のセクションでいくつかのリアルタイムレポートを教師に提供します。さらに、プログラムには「学生」レポートから開くことができ、不正解に関する情報を提供する「回答」レポートが必要であると判断しました。

ユーザーが必要なボタンを押すことでレポート間を移動し、回答を除いて、すべてのレポートに他のレポートからアクセスできるようにすることを想定しました。トランジションのコンテキストで「利用可能」という言葉の意味はかなり曖昧です。つまり、「なんとかしてレポートに移動できる」ことを意味する場合もあれば、「ワンクリックで目的のレポートに直接移動すること」を意味する場合もあります。上記のすべてがユーザーのレポート全体のナビゲーションに加えて、独自のナビゲーションシステムを備えたアプリケーション全体のインターフェース。

システムが複雑になり始めたら、注意する必要があります。これは、仕様を書くことを意味します。では、これをどのように指定できますか？先生がアプリケーションの特定の画面にいて、別の画面に移動するアクションを実行できることがわかります。これは、教師の行動を有限状態機械と見なすことができるという考えに私を導きます。

状態機械


有限状態機械（FSM）は、抽象オートマトンの理論の最も単純な数学モデルの1つです。有限の状態のセット、状態間の一連の遷移、および遷移をトリガーする一連のイベント（トリガー）があります。各遷移はイベントに関連付けられているため、イベントが発生すると、状態が変化する可能性があります。3私たちのモデルでは、イベントは「教師がボタンを押す」です。以下は、現在のシステムのステートマシンモデルです。


このモデルは、ユーザーインターフェイスに2つの問題があることを示しています。まず、すべてのステートマシンには初期状態が必要ですが、それはありません。教師がレポートページにアクセスすると、どのレポートが最初に表示されますか？第二に、すでに表示しているレポートのボタンをクリックしたときに何が起こるかは示していません。いくつかの動作があるため、これはあいまいです。

  1. 概要レポートを表示している場合、概要ボタンは表示されないか、何も実行されません。
  2. 概要レポートを表示している場合は、概要ボタンでレポートをリセットします。

Harelステータスチャート


ほとんどの州には、いわゆるスーパーステートの共通のロジックがあります。4つの「レポート」すべてがシステムからログアウトするための同じロジックを持ち、主要な3つは同じ遷移を持っています。それらを「親の状態」のレポートにグループ化できる場合は、「出口」への遷移を決定し、この遷移を子の状態に分配するだけです。ロジックは継承に似ています。子ステートは親ステートの遷移を継承（または再定義）します。

ネストされた状態を持つ有限状態機械は階層状態機械と呼ばれ、それらを形式化するいくつかの異なる方法があります。UIのステートマシンモデルを記述する最も適切な方法-Harel状態図 （ハレルステートチャート）。 4彼女のルールは次のとおりです。

  1. すべての親の状態は抽象的です。各親状態は、デフォルトの子状態を定義する必要があります。
  2. 子の状態は自動的にすべての親の遷移を継承しますが、それらをオーバーライドすることもできます。
  3. 遷移は任意の状態を示す場合があります。親の状態に切り替える場合は、代わりにデフォルトの子の状態に移動してください。子の状態が親でもある場合、状態は再帰的に決定されます。

Graphvizで Harelステートチャートを設計し、グラフ化グラフの多くの頂点、エッジ、および喜びをその都度恐ろしくすることができます。Sketch.systemsのより優れたユーザーインターフェイスを使用します。

小切手


正式な仕様には2つの利点があります。それらの1つは暗黙的です。形式化に関する作業は、システムのより良い理解につながります。もう1つは明示的です。正式な仕様があれば、そのプロパティを確認できます。ユーザーインターフェイスにデッドロックはありますか？暗黙的または誤って指定された遷移はありますか？

Sketch.systemsは、Harel状態図の形式が正しいかどうかを確認できますが、モデルの動作を確認できません。状態マシンの状態を判別するためのツールは他にもあります（特にUML状態図）。ただし、それらはすべて、システムレベルの仕様ではなく、コードレベルの仕様に重点を置いています。彼らの目標は、最終的に状態図からCまたはJavaコードを生成することです。しかし、それらは抽象プロパティをテストするには低レベルであり、コードを生成するには高すぎます。正式なテストが必要な場合は、汎用仕様言語でモデルを記述する必要があります。

幸いなことに、この場合は非常に簡単です。これを行うには、Harel状態図の構造を最も正確に反映できる合金を使用します。5署名拡張を使用して、ネストされた状態を表すことができます。「標準」は「レポート」を拡張します。「標準」の各部分も「レポート」であることを意味します。これは、これが対応するハレル図の子状態であるというステートメントと同等です。これにより、遷移の定義が簡単になります。それらのそれぞれは、時間（t）、初期状態（開始）、および最終状態（終了）をとる単一の述語によって表され、時間tの状態がt.nextの開始から終了まで続くことを宣言します。親の状態は抽象的であるという事実にもかかわらず、それらを開始として使用して、親の遷移を利用することができます。

open util/ordering[Time]
sig Time {
    state: one State
}
abstract sig State {}
abstract sig Login extends State {}
abstract sig Reports extends Login {}

one sig Logout extends State {}
one sig Students, Summary, Standards extends Reports {}
one sig Answers extends Login {}

pred transition[t: Time, start: State, end: State] {
  t.state in start
  t.next.state in end
}

pred logout[t: Time] { transition[t, Login, Logout] }
pred login[t: Time] { transition[t, Logout, Summary] }
pred students[t: Time] { transition[t, Reports, Students] }
pred summary[t: Time] { transition[t, Reports, Summary] }
pred standards[t: Time] { transition[t, Reports, Standards] }
pred answers[t: Time] { transition[t, Students, Answers] }
pred close_answers[t: Time] { transition[t, Answers, Students] }

fact Trace {
  first.state = Summary
  all t: Time - last |
    logout[t] or
    login[t] or
    students[t] or
    summary[t] or
    standards[t] or
    answers[t] or
    close_answers[t]
}

これで、モデルのプロパティを確認できます。たとえば、「学生」レポートを経由せずに「回答」レポートを取得することは可能ですか？

check {all t: Time | t.state = Answers implies 
        t.prev.state = Students} for 7 // valid

また、誰かがログアウトして再度ログインするときの例をシミュレートすることもできます。

run {some disj t1, t2: Time | logout[t1] and login[t2]} for 7

値


これらすべてに価値がありますか？インタラクティブな状態図が英語の単なるメモよりも優れている理由は何ですか？確かに、スケーリングすると、ダイグラムの価値が高まります。スナップショットプロジェクトでは、約20の教師画面がいくつかの大きな階層にネストされていたことを覚えています。正式な仕様がなければ、作業をテストすることはできませんでした。私が覚えている限り、私たちが犯したいくつかの間違い：

  • 「回答報告の締めくくり」をイベントと考えるのを忘れ、「回答」が行き止まりに
  • 新しい調査の作成には、本当に必要ない郡のルートがたくさんありました。
  • 調査の作成後、ユーザーインターフェイスの動作を決定しなかったため、教師は調査作成画面に戻り、エラー中に調査は作成されなかったと考え、再作成しました
  • 複数の画面にアクセスするのは難しかったので、誰もその画面に行きませんでした。

正式な仕様があればとても助かると思います。上記で作成した例の記述には約5分かかり、完全なアプリケーションの仕様には2時間もかかりませんでした。設計段階で、リストされたエラーの少なくとも1つを見つけることができれば、後で多くの時間を節約できます。

結論


UIとのユーザーインタラクションの正式な仕様について説明しました。私の正式なメソッドへの愛は、UIへの私の恐怖を克服できますか？いいえ。Sketch.systemsへのリンクをたどった場合は、Javascriptのプロトタイプを状態図に添付できることを見たことがあるでしょう。それは魔法です！

私の恐れにもかかわらず、正式な方法にはいくつかの可能性があると思います。人々は通常、それらをNASAだけが使用する純粋に学術的なものと考えています。このブログの主なテーマは、正式なメソッドは日常の仕事のための強力なツールであることです。私はそれを楽しんでいるので、主にバックエンドと並列システムへの適用を検討します。しかし、それらの使用は私の好みだけに限定されません。形式的なメソッドは、ユーザーインターフェイスにとって特に重要です。ハレルの状態図は、可能な限り最良の表記法ではないと思いますが、これは良い最初のステップです。
