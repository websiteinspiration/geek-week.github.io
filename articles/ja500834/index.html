ブラウザにデータを保存するためのいくつかの技術があります。どっちがいいですか？

特定の場所では、インターネット接続が不十分であるか、存在しない場合もあります。したがって、オフラインサポートはプログレッシブウェブアプリケーションの主要な機能の1つです。高速接続でも、キャッシュやその他の技術を使用してユーザーエクスペリエンスを向上させることが賢明です。ファイル（HTML、JavaScript、CSS、画像など）とデータ（ユーザーデータ、ニュース記事など）を保存するには、いくつかの方法があります。しかし、どちらのソリューションを選択する方が良いでしょうか？そして、その耐久性をどのように保証するのですか？

何を使う？


次のことをアドバイスできます。

  • アプリケーションが機能してコンテンツを表示するために必要なネットワークリソースについては、Cache Storage API（Service Workerの一部）を使用します。
  • その他のデータについては、IndexedDBを使用します（promise からのラップを使用）。

IndexedDBとCache APIは、最新のすべてのブラウザでサポートされています。それらは非同期です。メインスレッドをブロックしないでください（コードの実行）。これらは、Windowオブジェクト、Webワーカー、およびServiceワーカーで使用できます。つまり、どこでも使えます。

少なくとも数百メガバイト、場合によっては数百ギガバイト。これはブラウザによって異なりますが、ストレージの容量は通常、ユーザーのデバイスで利用可能なメモリの容量に基づいています。

  • Chromeでは、最大60％のディスク容量を使用できます。StorageManager APIを使用して、制限（割り当て）を決定できます。
  • Internet Explorer 10以降では、最大250 MBのデータを保存できます。
  • Firefoxでは、最大2 GBのデータを保存できます。StorageManager APIを使用して制限を決定できます。
  • Safari（デスクトップとモバイルの両方）では、最大1 GBのデータを保存できます。制限に達すると、Safariはユーザーにクォータを200 MB増やす許可を求めます。

以前は、ストレージの制限に達したとき、ブラウザーはユーザーにメモリー量を増やす許可を要求しました。たとえば、50 MBの制限に達すると、ブラウザは割り当てを100 MBまで、つまり50 MBごとに増やすようにユーザーの許可を要求しました。

現在、ほとんどのブラウザーはそうではなく、クォータ内のストレージの量を自動的に増やしています。例外はSafariで、750 MBに達すると、制限を1.1 GBに増やす許可をユーザーに求めます。クォータを超えようとすると失敗します。

限度額の残高を確認するにはどうすればよいですか？


このため、多くのブラウザで StorageManager APIを使用できます。IndexedDBとキャッシュAPIで使用される合計バイト数を示し、残りを計算できます。

if(navigator.storage && navigator.storage.estimate) {
  const quota = await navigator.storage.estimate()
  // quota.usage ->   
  // quota.quota ->   
  const percentageUsed = (quota.usage / quota.quota) * 100
  console.log(`  ${ percentageUsed}% `)
  const remaining = quota.quota - quota.usage
  console.log(`   ${remaining} `)
}

StorageManager APIは、まだすべてのブラウザでサポートされているわけではないことに注意してください。ただし、サポートされている場合でも、エラーハンドラを提供する必要があります。場合によっては、クォータが実際のストレージ容量を超えることがあります。

エラーの処理方法は？


制限に達したときに何をすべきか？もちろん、QuotaExceededErrorやその他のエラーを処理します。次に、アプリケーションの設計に応じて、アプリケーションの処理方法を選択する必要があります。たとえば、古いコンテンツやデータをそれらのサイズに応じて削除したり、削除するものを決定する機能をユーザーに提供したりできます。

IndexedDBとCache APIは、クォータを超えるとDOMError QuotaExceededErrorをスローします。

Indexeddb


制限に達すると、IndexedDBへのデータの書き込みは失敗します。onabort（）メソッドは、イベントを引数として呼び出されます。イベントのerrorプロパティにDOMExceptionが含まれます。エラー名を確認すると、QuotaExceededErrorが返されます。

const transaction = idb.transaction(['entries'], 'readwrite')
transaction.onabort = function(event) {
    const error = event.target.error // DOMException
    if(error.name === 'QuotaExceededError') {
        // ...
    }
}

キャッシュAPI


制限に達したときにデータをキャッシュAPIに書き込む試みは、QuotaExceededError DOMExceptionで拒否されます。

try {
    const cache = await caches.open('my-cache')
    await cache.add(new Request('/sample1.jpg'))
} catch (error) {
    if(error.name = 'QuotaExceededError') {
        // ...
    }
}

ストレージのクリーンアップはどのように機能しますか？


Webリポジトリは、「スタンドアロン」と「管理」の2つのカテゴリに分類されます。スタンドアロンとは、ユーザーの介入なしにブラウザーでリポジトリーをクリアできることを意味しますが、重要なデータが存在する場合と同様に、長期間の使用には適応性が低くなります。管理対象の格納域は、いっぱいになっても自動的にクリーンアップされません。ユーザーはこのようなストレージを手動でクリアする必要があります（ブラウザーの設定を使用）。

デフォルトでは、Webストレージ（IndexedDB、キャッシュAPIなど）はスタンドアロンとして分類されます。つまり、手動制御がインストールされていない場合、ブラウザーは、特定の条件下（たとえば、データの入力時）にストレージを個別にクリアできます。

ストレージのクリーニングの条件は次のとおりです。

  • ストレージがいっぱいになると、Chromeは、オーバーフローの問題が解決されるまで、要求が最も少ない（使用時間が最も古い）ものからデータを削除します。
  • IE 10+はストレージをクリアしませんが、データを書き込む機能をブロックします。
  • FirefoxはChromeと同じことを行います。
  • Safariは以前に格納域を空にしていませんでしたが、最近7日間のデータ保持期間が追加されました。

iOSおよびiPad 13.4、macOS上のSafari 13.1以降、7日間のデータ保持期間があります。つまり、ユーザーが7日以内にデータにアクセスしない場合、データは削除されます。このポリシーは、ホーム画面に追加されたアプリケーションには適用されません。

おまけ：IndexedDBに対するプロミスラップ


IndexedDBは低レベルのAPIであり、使用する前にいくつかの構成が必要です。単純なデータを格納する必要がある場合は、これが不要な場合があります。最新のPromiseベースのAPIとは異なり、イベントベースです。idbなどのpromiseのラッパーは、このリポジトリのいくつかの強力な機能を隠しますが、さらに重要なことに、その複雑な内部メカニズム（トランザクション、バージョン管理）も隠します。

結論


制限されたストレージの時間と、制限を増やすためのユーザー許可要求は、忘却に陥っています。サイトは、作業に必要なすべてのリソースとデータを効果的に保存できます。StorageManager APIを使用して、使用されているメモリの量と残りの量を確認できます。また、ストレージを手動制御モードにすることで、データを削除から保護できます。
