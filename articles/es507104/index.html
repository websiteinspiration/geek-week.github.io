La gran división en la importación: aclarando la incertidumbre con la importación en Typecript

Se preparó una traducción del artículo antes del inicio del curso "React.js Developer"


He estado trabajando con el mecanografiado durante bastante tiempo, y tuve muchos problemas para ordenar sus módulos y asesorar configuraciones, y debo decir que realmente hay un montón de incomprensibles a su alrededor. Espacio de nombres import * as React from 'react', esModuleInteropetc. Por lo tanto, comprendamos por qué surgió todo el bombo.

No hablaré sobre los espacios de nombres como un sistema modular en mecanografiado, porque la idea no era la mejor (especialmente teniendo en cuenta el vector de desarrollo actual), y nadie lo está usando ahora.

Entonces, ¿cómo eran las cosas antes de que aparecieran esModuleInterop? Había casi todos los mismos módulos que tenían Babel o los navegadores, así como las importaciones / exportaciones con nombre. Sin embargo, en materia de exportaciones e importaciones por defecto, el mecanografiado tenía su propia opción: era necesario escribir import * as React from 'react'(en cambio import React from 'react'), y, por supuesto, aquí estamos hablando no solo de reaccionar, sino de todas las importaciones predeterminadas de commonjs. ¿Como paso?

Para entender esto, veamos cómo funciona la compatibilidad entre algunos patrones commonjsy módulos es6. Por ejemplo, tenemos un módulo que exporta fooy barcomo claves:

module.exports = { foo, bar }

Podemos importar usando require y desestructuración:

const { foo, bar } = require('my-module')

Y aplique el mismo principio utilizando importaciones con nombre (aunque, para ser sincero, esto no es una desestructuración):

import { foo, bar } from 'my-module'

Sin embargo, el patrón más común en commonjs es const myModule = require('my-module')(porque aún no ha habido una desestructuración), pero ¿cómo lo hace es6?

Al desarrollar la especificación para la importación, es6uno de los aspectos importantes era la compatibilidad commonjs, ya que ya había commonjsmucho código escrito en ella. Así es como aparecieron las importaciones y exportaciones predeterminadas. Sí, el único objetivo era garantizar la compatibilidad commonjs, de modo que pudiéramos escribir import myModule from 'my-moduley obtener exactamente el mismo resultado. Sin embargo, a partir de las especificaciones esto no era obvio, y además, la implementación de compatibilidad era prerrogativa de los desarrolladores del transpiler. Y aquí acaba de ocurrir el gran cisma: import React from 'react'o import * as React from 'react'esa es la pregunta.

¿Por qué mecanografiado eligió este último? Póngase en el lugar del desarrollador transpiler y te preguntas, ¿cuál es la forma más fácil de las importaciones de transporte es6a commonjs? Supongamos que tiene el siguiente conjunto de importaciones y exportaciones:

export const foo = 1
export const bar = 2
export default () => {}
import { foo } from 'module'
import func from 'module'`

Por lo tanto, usaremos el objeto jscon la clave defaultpara exportar de forma predeterminada.

module.exports = {
  foo: 1,
  bar: 2,
  default: () => {}
}
const module = require('module')
const foo = module.foo
const func = module.default

Genial, pero ¿qué pasa con la compatibilidad? Si importar de forma predeterminada significa que tomamos un campo con un nombre default, cuando lo escribamos import React from 'react'significará const { default: React } = require('react'), ¡pero no funciona! Luego, intente utilizar la importación con un asterisco. Ahora los usuarios tendrán que escribir import * as React from 'react'para acceder al contenido module.exports.

Sin embargo, hay una diferencia semántica de commonjs.
Commonjsera como JavaScript normal, no más. Solo funciones y objetos, sin ningún requerimiento . Por otro lado, en importación es6, requireahora es parte de la especificación, por lo tantomyModuleen este caso, no es solo un objeto javascript normal, sino lo que se llama un espacio de nombres (que no debe confundirse con espacios de nombres en mecanografiado), que, en consecuencia, tiene ciertas propiedades. Una de ellas es que no se puede llamar al espacio de nombres. ¿Y cuál es el problema aquí, puedes preguntar?

Probemos con otro patrón commonjs, con una función como exportación:

module.exports = function() { // do something }

Podemos usarlo requirey ejecutarlo:

const foo = require('my-module')
foo()

Aunque si intenta hacer esto en un entorno de queja de especificaciones con módulos ES6, obtendrá un error:

import * as foo from 'my-module'
foo() // Error

Esto se debe a que el espacio de nombres no es lo mismo que un objeto javascript, sino una estructura separada que almacena cada exportación es6.

Pero Babel lo hizo bien y proporcionó una opción de compatibilidad donde podemos escribir import React from 'react'y funcionará. Al traducir, marca cada módulo es6 con un indicador especial module.exports, de modo que comprendamos que si el indicador es verdadero, se devuelve module.exportsy si es falso (por ejemplo, si es una biblioteca commonjsque no se ha transpuesto), entonces tendremos que ajustar la exportación actual { default: export }para que podamos podría usar todo el tiempo default(echa un vistazo aquí ).

El mecanografiado se abrió paso a través de las importaciones con asteriscos, pero finalmente se rindió y agregó una opciónesModuleInteropal compilador. En general, esta opción hace lo mismo que babel, y si la habilita, puede escribir importación regular como import React from 'react', y mecanografiado lo entenderá todo.

El problema es que, aunque está habilitado de forma predeterminada en nuevos proyectos (en tiempo de ejecución tsc --init), no funcionará para proyectos existentes (incluso si actualiza a TypeScript 3) porque no tiene compatibilidad con versiones anteriores. Por lo tanto, tendrá que reescribir las importaciones innecesarias de asteriscos a las importaciones predeterminadas. React tratará esto normalmente, ya que todavía es una colección de exportaciones con nombre, pero no, por ejemplo, con una llamada de espacio de nombres. Pero no tenga miedo si todo está en orden con el tipo de exportación (y en su mayor parte, todo está en orden, ya que muchos de ellos se arreglan automáticamente), TypeScript 3 le permitirá convertir rápidamente la importación con asteriscos a estándar.

Por lo tanto, realmente recomiendo el uso de la opción esModuleInterop, aunque solo sea porque no solo le permitirá escribir menos código y facilitará la lectura (y estas no son solo palabras, por ejemplo, el resumen no le permitirá usar importaciones con asteriscos como este), sino que también reduce los desacuerdos entre comunidades mecanografiado y babel.

Precaución: solía haber una opción enableSyntheticDefaultImportsque cerraba el compilador cuando intentaba quejarse de una importación incorrecta de forma predeterminada, por lo que necesitábamos nuestra propia forma de garantizar la compatibilidad commonjs(por ejemplo,WebpackDefaultImportPlugin), pero fue problemático porque, por ejemplo, si tiene pruebas, aún debe garantizar dicha compatibilidad. Tenga en cuenta que esModuleInteropincluye la importación sintética de forma predeterminada solo si su objetivo es <=ES5. Por lo tanto, si habilita esta opción y los compiladores continúan quejándose de ella import React, entonces comprenda qué propósito persigue, y tal vez la inclusión de importaciones predeterminadas será su opción (o reinicie vscode / webstorm, quién sabe).

Espero que mi explicación aclare un poco la situación, pero si aún tiene preguntas, ¡puede hacerlas en Twitter !
