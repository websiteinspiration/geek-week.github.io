Elmアプリケーションで大きすぎるメッセージと戦う

Elm Architectureによると、すべてのアプリケーションロジックが1つの場所に集中しています。これはかなり単純で便利な方法ですが、アプリケーションが大きくなるにつれて、update長さが700行Msgで数百のコンストラクターがありModel、画面に収まらない関数が見られます。


このようなコードは習得が難しく、頻繁に維持することが困難です。アプリケーションの抽象化レベルを向上させる非常に単純な手法を示したいと思います。


簡単な例を見てみましょう。


まず、テキストボックスが1つだけの小さなアプリケーションを作成します。完全なコードはここにあります。


type alias Model =
    { name : String
    }

view : Model -> Html Msg
view model =
    div []
        [ input [ placeholder "Name", value model.name, onInput ChangeName ] []
        ]

 type Msg
    = ChangeName String

update : Msg -> Model -> Model
update msg model =
    case msg of
        ChangeName newName ->
            { model | name = newName }

アプリケーションは拡大しており、「自分自身について」の姓と「保存」ボタンを追加しています。ここでコミットします。


type alias Model =
    { name : String
    , surname : String
    , bio : String
    }

view model =
    div []
        [ input [ placeholder "Name", value model.name, onInput ChangeName ] []
        , br [] []
        , input [ placeholder "Surname", value model.surname, onInput ChangeSurname ] []
        , br [] []
        , textarea [ placeholder "Bio", onInput ChangeBio, value model.bio ] []
        , br [] []
        , button [ onClick Save ] [ text "Save" ]
        ]

type Msg
    = ChangeName String
    | ChangeSurname String
    | ChangeBio String
    | Save

update : Msg -> Model -> Model
update msg model =
    case msg of
        ChangeName newName ->
            { model | name = newName }

        ChangeSurname newSurname ->
            { model | surname = newSurname }

        ChangeBio newBio ->
            { model | bio = newBio }

        Save ->
           ...
