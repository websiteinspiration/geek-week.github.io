Kubernetesへの完璧なMongoDB移行



この記事では、最近の RabbitMQ移行資料を続け、MongoDBに焦点を当てます。多くのKubernetesおよびMongoDBクラスターにサービスを提供しているため、あるインストールから別のインストールにデータを移行し、ダウンタイムなしで移行するという自然な必要性に直面しています。主なシナリオは同じです。MongoDBを仮想サーバー/ ironサーバーからKubernetesに移植するか、MongoDBを1つのKubernetesクラスター内で（ある名前空間から別の名前空間に）移植します。

私たちのレシピは、Kubernetesでホストされているアプリケーションが機能する古いMongoDBクラスターが機能している（たとえば、3つのノードから、すでにK8sまたは古いサーバー上にある）場合を対象としています。



このようなクラスターをKubernetesの新しいプロダクションに転送するには？

理論


一般的な移行アルゴリズムは、RabbitMQの状況で説明されているものと似ています。

移動するには、MongoDBとKubernetesを備えたサーバーが同じネットワーク上にある必要があることに注意してください。 MongoDBクラスターのノードは、古いサーバー（古いMongoDBインストールが配置されている場所）のIPおよびK8のMongoDBからのポッドのDNS名を介して相互に通信します。したがって、（古いインストールの）鉄のサーバーでは、ルートをポッドに転送し、Kubernetesで実行されているDNSサーバーを使用するようにそれらを構成する必要があります（または必要な名前をに登録し/etc/hostsますが、一般的にはこの可能性を回避することをお勧めします）。

次のステップは、KubernetesポッドでMongoDBクラスターを上げることです。私たちの場合、データベースクラスターは3つのノードで構成され、各ノードは別々のpod'8 K8にありますが、それらの数は異なる場合があります。 ConfigMapでは、古いインストールのMongoDBマスターのアドレスを指定する必要があります。K8のポッドにあるMongoDBノードは、すぐに同期を開始します。

すべてのポッドが起動すると、6ノードのMongoDBクラスターが形成されます。



各ポッドが順番に起動し、起動時にウィザードからのデータを同期するため、ポッドが長時間起動することに注意してください。

その後、新しいMongoDBサーバーを使用するようにアプリケーションを切り替えることができます。



また、古いノードをMongoDBクラスターから削除するだけで、移動が完了したと見なすことができます。



このスキームを本番環境で使用することが多く、その使用の便宜上、アドオンオペレーターモジュールに実装しました（最近このユーティリティを発表しました）。これにより、典型的なMongoDB構成を多くのクラスターに分散できます。近いうちにモジュールを公開する予定ですが、現時点では、アドオンオペレーターを使用せずに、提案されたソリューションを実際に試すことができる個別の手順を示します。

実際にやってみる


必要条件


必要条件：

  • Kubernetesクラスター（minikubeも適しています）;
  • MongoDBクラスター（ベアメタルにデプロイでき、公式のHelmチャートからKubernetesで通常のクラスターとして作成できます）。

以下で説明する例では、MongoDBを含む古いクラスターに名前が付けられmongo-old、同じKubernetesクラスターにインストールされます。将来、新しいクラスターがインストールされます（mongo-new）。

古いクラスターの準備


1.記述されたスキームの動作を示す例として、「古い」（つまり移行の対象となる）MongoDBクラスターをKubernetesに直接作成します（実際には、K8の外の別のサーバーに配置することもできます）。これを行うには、Helmチャートをダウンロードします。

helm fetch --untar stable/mongodb-replicaset

...そして承認を設定して少し編集します：

auth:
  enabled: true
  adminUser: mongo
  adminPassword: pa33w0rd
  # metricsUser: metrics
  # metricsPassword: password
  # key: keycontent
  # existingKeySecret:
  # existingAdminSecret:
  # exisitingMetricsSecret:

values.yaml証明書などを設定 することもできます。

2.チャートを設定します。

helm install . --name mongo-old --namespace mongo-old

その後、MongoDBの「古い」テストインストールが起動します。

kubectl --namespace=mongo-old get pods



マスターと一緒にポッドに移動して、テストベースを作成します。

kubectl --namespace=mongo-old exec -ti mongo-old-mongodb-replicaset-0 mongo
use admin
db.auth('mongo','password')
use music
db.artists.insert({ artistname: "The Tea Party" })
show dbs



別のポッドに入ると、マスターがであることがわかりましたmongo-old-mongodb-replicaset-0。ただし、この問題のより便利な解決策として、Helm-chartをインストールした後、を判別する方法に関するコマンドが表示されますMASTER_POD。私の場合（mongo-old3ノードからの場合）は次のようになります：

for ((i = 0; i < 3; ++i)); do kubectl exec --namespace mongo-old mongo-old-mongodb-replicaset-$i -- sh -c 'mongo --eval="printjson(rs.isMaster())"'; done

これで、データが転送される古いMongoDBインストールの準備が整いました。

MongoDBクラスターの移行


次に、MongoDBの新しいインストールをデプロイします。これは、Kubernetesに配置され、本番環境のアプリケーションで使用されます。

注意：以前と同じバージョンのMongoDBを使用する必要があることに注意してください。それ以外の場合は、互換性の問題のリスクがあります。

前のセクション（「古い」MongoDBインストールをシミュレートした場所helm fetch）と同様に、前述のHelm-chart（コマンド）を使用して、承認と、使用されている場合はその他のパラメーターを構成します。さらにinit/on-start.sh、前の手順で取得した（または別のサーバーにMongoDBをインストールすることで既知の）ウィザードのアドレスを165行目に一時的に追加して、ファイルを修正します。

peers='mongo-old-mongodb-replicaset-0.mongo-old-mongodb-replicaset.mongo-old.svc.cluster.local:27017'

新しいMongoDBインストールを作成する準備ができました。

helm install . --name mongo-new --namespace mongo-new

すべてのポッドが開始するまで待機します（大量のデータがある場合、それらの起動には数時間かかる可能性があります）。



次にexec、新しいポッドを作成して、データベースのリストを確認します。

kubectl --namespace=mongo-new exec -ti mongo-new-mongodb-replicaset-0 mongo



2つのMongoDBクラスターが1つに結合され、6つのノードで構成されます。

現時点では、すでにアプリケーションを新しいクラスターに切り替えることができますが、移行を完了するにはいくつかの手順が残っています。新規インストールの

ファイルから、init/on-start.sh追加した行を削除します。

peers='mongo-old-mongodb-replicaset-0.mongo-old-mongodb-replicaset.mongo-old.svc.cluster.local:27017'

次に、クラスターの古いマスターに移動して、それを「破棄」します。次に、新しいマスターがクラスターに割り当てられます。MongoDBマスターと一緒にポッドに入ります。

kubectl --namespace=mongo-old exec -ti mongo-old-mongodb-replicaset-0 mongo
use admin
db.auth('mongo','password')

その後、ノードの優先順位を変更し、ウィザードを変更します。

cfg = rs.conf()
cfg.members[5].priority = 2
rs.reconfig(cfg)
rs.stepDown(120)

現在のノードはマスターでなくなりました-新しい選択が行われます。優先順位を変更すると、必要なノードがマスターになります。

注意：デフォルトでは、すべてのMongoDBノードの優先度は1です。上記の場合、必要なノードの優先度を2に上げます。したがって、新しいクラスターのメンバーは間違いなく共通のマスターになりつつあります。これらのメカニズムがMongoDBでどのように機能するかについては、ドキュメントを参照してください。

古いMongoDBインストールを無効にしてから、新しいウィザードに移動して古いノードを削除します。

rs.remove("mongo-old-mongodb-replicaset-0.mongo-old-mongodb-replicaset.mongo-old.svc.cluster.local:27017")
rs.remove("mongo-old-mongodb-replicaset-1.mongo-old-mongodb-replicaset.mongo-old.svc.cluster.local:27017")
rs.remove("mongo-old-mongodb-replicaset-2.mongo-old-mongodb-replicaset.mongo-old.svc.cluster.local:27017")

この後、移行は完了したと見なすことができます。古いMongoDBクラスターから新しいクラスターに正常に切り替えました！

概要


説明されているスキームは、MongoDBを移行する必要がある場合、または単に新しいクラスターに移動する必要があるほとんどすべての場合に適しています。

おそらく、転送中の主なニュアンスは、新しいポッドのIPアドレスを古いMongoDBインストールのサーバーに転送し、それがK8の外部にある場合、DNS（または/etc/hosts）でそれらに正しく名前を付ける必要があることです。この例では、同じKubernetesクラスターの異なる名前空間間で移行が行われたため、これらの手順は必要ありませんでした。
