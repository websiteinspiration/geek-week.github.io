GOツールを使用してDNSサーバーを最適化した方法

コースの新しいストリームの開始を見越して、「Developer Golang」は興味深い資料の翻訳を準備しました。



私たちの権威あるDNSサーバーは、何万ものWebサイトで使用されています。毎日何百万ものクエリに対応しています。今日、DNS攻撃はますます一般的になり、DNSはシステムの重要な部分であり、高負荷下でも適切に機能できるようにする必要があります。

dnsfloodは、多数のudpリクエストを生成できる小さなツールです。

# timeout 20s ./dnsflood example.com 127.0.0.1 -p 2053

システムモニタリングの結果、サービスのメモリ使用量が急増しているため、停止する必要がありました。そうしないと、OOMエラーが発生します。それはメモリリークの問題のようなものでした。goでの「類似」および「実際」のメモリリークにはさまざまな理由があります。

  • ぶら下げゴルーチン
  • 遅延とファイナライザーの誤用
  • サブストリングとサブカット
  • グローバル変数

この投稿では、さまざまなリークについて詳しく説明しています。

結論を出す前に、まずプロファイリングを行いましょう。

プロファイラーを有効にする


go tool pprof-データを分析およびプロファイリングするためのツール。構成には2つの方法がありますpprof。runtime/pprofたとえばpprof.StartCPUProfile()、コード内の関数を直接呼び出すか、net/http/pprofhttpリスナーをインストールしてそこからデータを取得します。それは持っているpprof、それは開発で安全に使用できますが、機密データが開示させることができるので、プロファイルのエンドポイントは、一般に公開すべきではないので、非常に小さなリソース消費を。

2番目のオプションについて行う必要があるのは、「net / http / pprof」パッケージをインポートすることだけです。

import (
  _ "net/http/pprof"
)

次に、httpリスナーを追加します。

go func() {
  log.Println(http.ListenAndServe("localhost:6060", nil))
}()

Pprofにはいくつかのデフォルトプロファイルがあります。

  • allocs：過去のすべてのメモリ割り当てを取得します
  • block：同期プリミティブのブロックにつながるスタックトレース
  • goroutine：現在のすべてのゴルーチンのスタックトレース
  • heap：ライブオブジェクトのメモリ割り当てをフェッチします。
  • mutex：競合するmutexホルダーのスタックトレース
  • profile：プロセッサプロファイル。
  • threadcreate：OSでの新しいスレッドの作成につながったスタックトレース
  • trace：現在のプログラムの実行を追跡します。

注：トレースエンドポイントは、他のすべてのエンドポイントとは異なり、pprofではなくトレースプロファイルです。代わりにgo tool traceを使用して表示できます。go tool pprof

割り当ての最適化


この時点で、ガベージコレクションがどのように機能するかを確認すると役立つ場合があります。Goはマークアンドスイープピッカーを使用します。選択されたすべてのものを追跡し、以前のサイズの2倍のサイズ（またはGOGCが設定されている他の値）に達するとすぐに、GCクリーンアップが開始されます。評価は3つの段階で行われます。

  • マーカー設定（STW）
  • マーキング（平行）
  • ラベリング終了（STW）

ストップザワールド（STW）フェーズはプロセス全体を停止しますが、通常は非常に短いですが、サイクルを継続すると継続時間が長くなる可能性があります。これは、現時点（go v1.13）のゴルーチンが関数呼び出しの時点でのみ取って代わられるため、GCがすべてのゴルーチンの停止を期待しているため、連続したサイクルでは、任意の長い休止時間が可能になるためです。

マーキング中にgcは約25％を使用しGOMAXPROCSますが、マークアシストに追加の補助機能を強制的に追加できます。これは、高速移動するゴルーチンがバックグラウンドマーカーを追い越してgcによって引き起こされる遅延を減らすときに発生します。ヒープの使用を最小限に抑える必要があります。

注意すべき2つの点：

  • 割り当ての数はサイズよりも重要です（たとえば、20バイトの構造からの1000のメモリ割り当ては、20,000バイトの単一の割り当てよりもはるかに多くの負荷をヒープに作成します）。
  • C / C ++のような言語とは異なり、すべてのメモリ割り当てがヒープ上で終わるわけではありません。goコンパイラーは、変数がヒープに移動するかどうか、またはスタックフレーム内に配置できるかどうかを決定します。ヒープに割り当てられた変数とは異なり、スタックに割り当てられた変数はgcをロードしません。

GoメモリモデルとGCアーキテクチャの詳細については、このプレゼンテーションを参照してください。

メモリ割り当てを最適化するには、goツールキットを使用します。

  • ホットディストリビューションを見つけるためのプロセッサープロファイラー。
  • 割り当てを追跡するためのメモリプロファイラ。
  • パターンのトレーサー。Gc
  • エスケープ分析により、割り当てが発生する理由を確認します。

maxmiddb 関数を使用して位置情報データを取得します。これらの関数はinterface{}パラメーターとして使用されます。これは、前に説明したように、ヒープシュートを引き起こす可能性があります。後で再利用

type MMDBGeoLocation struct {
  Coordinations struct {
     Latitude        float64 `maxminddb:"latitude"`
     Longitude       float64
     LongitudeOffset uintptr `maxminddb:"longitude"`
  } `maxminddb:"location"`
  Country struct {
     ISOCode string `maxminddb:"iso_code"`
  } `maxminddb:"country"`
}

type MMDBASN struct {
  AutonomousSystemNumber uint `maxminddb:"autonomous_system_number"`
}
func (g *GeoIp) GetGeoLocation(ip net.IP) (*MMDBGeoLocation, error) {
  if !g.Enable || g.CountryDB == nil {
     return nil, EMMDBNotAvailable
  }
  var record *MMDBGeoLocation = g.LocationPool.Get().(*MMDBGeoLocation)
  logger.Default.Debugf("ip : %s", ip)
  if err := g.CountryDB.Lookup(ip, &record); err != nil {
     logger.Default.Errorf("lookup failed : %s", err)
     return nil, err
  }
  _ = g.CountryDB.Decode(record.Coordinations.LongitudeOffset, &record.Coordinations.Longitude)
  logger.Default.Debug("lat = ", record.Coordinations.Latitude, " lang = ", record.Coordinations.Longitude, " country = ", record.Country.ISOCode)
  return record, nil
}

func (g *GeoIp) GetASN(ip net.IP) (uint, error) {
  var record *MMDBASN = g.AsnPool.Get().(*MMDBASN)
  err := g.ASNDB.Lookup(ip, record)
  if err != nil {
     logger.Default.Errorf("lookup failed : %s", err)
     return 0, err
  }
  logger.Default.Debug("asn = ", record.AutonomousSystemNumber)
  return record.AutonomousSystemNumber, nil
}
