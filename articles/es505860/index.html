Como en cualquier aplicación con más de una entidad, crearemos un antepasado común para todos entity.

BaseEntity:

@MappedSuperclass
abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

No utilizamos clases de datos . Esto parece una clara ventaja para Kotlin sobre Java (antes de la versión 14), y hay una explicación.

¿Por qué no usar?


Las clases de datos, además de ser definitiva por su cuenta, tienen ciertos campos para todos los campos equals, hashCodey toString. Y esto es inaceptable en conjunto con Hibernate.

¿Por qué? Y también por qué hashCodesiempre es igual a una constante: la respuesta está en la documentación de Hibernate . Específicamente, estamos interesados ​​en esta sección:

Aunque el uso de un id natural es mejor para equals y hashCode , a veces solo tiene el identificador de entidad que proporciona una restricción única.

Es posible usar el identificador de entidad para la verificación de igualdad, pero necesita una solución alternativa:

  • debe proporcionar un valor constante para hashCode para que el valor del código hash no cambie antes y después de enjuagar la entidad.
  • necesita comparar la igualdad del identificador de entidad solo para entidades no transitorias.

Es decir, debe comparar por natural id, o, como en nuestro ejemplo, por primary key id. Esto evitará muchos problemas al comparar la entidad y ahorrará su pérdida al usar la entidad como un elemento en Set.

La presencia toString, definida por todos los campos, mata completamente toda la pereza, por ejemplo, cuando se registra en una entidad, ya que todos los campos se inicializarán para la salida a una cadena.

Dadas las características de Hibernate, esta funcionalidad de Kotlin no es adecuada para nosotros.

Validez nula en campos de acuerdo solo con DB


En cuanto a la validez de los valores nullen los campos, no todo es tan simple. Anteriormente nos sumergimos un poco en las profundidades de Hibernate: hablando de plugin.jpa, mencioné el uso de un constructor sin argumentos al inicializar una entidad.

Al inicializar campos, también se usa la reflexión. Y si se almacenó un valor en la base de datos en la columna correspondiente null, la clase se inicializa silenciosamente con este campo con el valor null. Repositorios


Al usar Kotlin, los repositorios listos para usar ahora tienen una verificación de validación null. Entonces, si estamos seguros de que al buscar departmentpor nombre, el resultado será único y único, entonces el tipo de retorno puede especificarse como non nullable:

interface DepartmentRepository : JpaRepository<DepartmentEntity, Long> {

   fun findOneByName(name: String) : DepartmentEntity
}

lateinit var


Otra característica de Kotlin que me gustaría considerar es lateinit var.

Añadir una nueva clase ancestro: BaseAuditEntity.

@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class BaseAuditEntity<T> : BaseEntity<T>() {

   @CreatedDate
   @Column(updatable = false, nullable = false)
   lateinit var created: LocalDateTime

   @LastModifiedDate
   @Column(nullable = false)
   lateinit var modified: LocalDateTime
}

Considere el uso lateinit varde los campos de auditoría ( created, modified) como ejemplo .

lateinit var- Este es un not nullcampo de inicialización retrasada. El acceso al campo antes de su inicialización genera un error:

kotlin.UninitializedPropertyAccessException: lateinit property has not been initialized

Como regla general, pasamos a los campos created, e modifiedincluso después, como una entidad que se ha almacenado en la base de datos. En este caso, los datos en estos polyahprostavlyayutsya el paso de almacenamiento y son not null, entonces lateinit var, más que adecuado.
