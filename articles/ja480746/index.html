PHP 7.4でのプリロード：Composerとプリロードするファイルの選択

BadooはPHP 7.4に積極的に移行しており、新しいプリロード機能を使用する機会に非常に熱心です。少し前に私達は彼女と私達の実験について話しました。

どうやら、コミュニティは私たちと同じように興奮しています。フレームワーク開発者は、プリロードを導入する可能性について積極的に話し合っています（すでにサポートを行っている人もいます）。今こそ、Composerの依存関係マネージャーの番です。 



Italo Baezaが記事を書いて、 Composerがプリロードをどのように処理するかについての意見を述べました。このテキストの翻訳と彼の他の記事の翻訳を共有することにしました -Composerの開発者自身が提案に答えたこと、およびプリロードの処理を容易にする新しいツールについて。

PHP 7.4でComposerがプリロードする方法


プリロードは、より良いパフォーマンスを必要とする開発者にPHP 7.4が提供する重要な機能の1つです。この関数は、PHP 8で表示される（または表示される）JITエンジンを実装する前に「ウォーミングアップ」と呼ぶことができます。その前に、プリロードで十分です。この記事で

は、プリロード機能について説明しています。一番下の行は非常に単純です。php.iniは、プロセスの開始時にファイルがメモリにロード（プリロード）されるPHPスクリプトを指定します。OPCacheおよびオートローダーと組み合わせるComposerファイルは、一度コンパイルしてリンクすることもできます。その後、後続のすべてのリクエストで使用できるようになります。このおかげで、PHPはすべてのリクエストでファイルをダウンロードしてコンパイルする必要がありません。

ただし、Composerの開発者は、起動関数を提供することに加えて、プリロードをどのように支援するかについて同意しませんでした。事実は次のとおりです。

  • プリロードは、PHP 7.4で最初に発表されました。
  • ファイルのプリロードを支援するComposerディレクティブはありません。
  • プリロードするには、php.ini、つまりプロセス自体にアクセスする必要があります。
  • すべてのファイルをプリロードしても、最も要求されたファイルのみをプリロードする場合と比較して、必ずしもパフォーマンスが向上するわけではありません。

つまり、サーバーに完全にアクセスできる人だけがプリロードを使用できます。これには、共有サーバーと、php.iniの操作を必要としない一部のPaaSソリューションは含まれません。

では、これが革新的なものであるとすれば、Composerはどのようにしてプリロードを支援できますか？これが私の意見です。

プリロードの仕組み


プリロード操作のメカニズムは、起動時にロードされてメモリに保存されるファイルのリストに基づいている必要があります。これはリストであるため、ファイルの配列を操作し、各ファイルを手動でロードするのではなく、Composerにすべての作業を行わせる必要があります。

Composerは、アプリケーション（ルートプロジェクト）によって指定されたファイルのリストを取得し、PHPが問題なく使用できるファイルにすべてをコンパイルする必要があります。
同時に、プリロードメカニズムからパッケージを追加および削除する機能が必要です。
各パッケージの事前読み込みを有効または無効にするのは開発者の責任であるため、事前読み込みはパッケージレベルでは機能しません。

Composerでのプリロードはオプションです。開発者はそれを無効にして、PHPが独自のプリローダーを使用できるようにする必要があります。これはOPCache分析に基づいて機能します。これはアプリケーションのロードに依存し、単にすべてのファイルをプリロードするよりもはるかに効率的に機能します。

すべてpreload.jsonから始まります


システムを複雑にしないために、プロジェクトのルートにpreload.jsonファイルを置きます。Composerが選択できるプリロードファイルが一覧表示されます。これはJSONファイルなので、開発者は特別なコマンドを使用してファイルを生成することもできます。ComposerにそのようなスクリプトベースのJSONファイルを作成するためのユーティリティがあれば、すばらしいと思います。

{
    "pre-compile": [
        "my-script.php",
        "my-other-script.php"
    ],
    "extensions": [
        "php"
    ],
    "files": [
        "app/*",
        "config/",
        "helpers.php",
        "app/Models/*",
        "app/Controllers/*/Http/*",
        "app/Views/Compiled*.php"
    ],
    "namespace": [
        "App\\Models",
        "App\\Controllers\\",
        "App\\Views\\MainView",
        "Vendor\\Package\\*",
    ],
    "packages": {
        "symfony/http-client": true,
        "robert/*-client": true,
        "vendor/package": {
            "files": true,
            "namespace": true
        },
        "foo/bar": {
            "files": [
                "helpers.php",
                "loaders/*"
            ],
            "namespace": [
                "Foo\\Bar\\DynamicLoaders\\*",
                "Foo\\Bar\\Clients"
            ]
        }
    },
    "output": "preload-compiled.php"
}

preload.jsonを使用すると、プロジェクトにプリロードが含まれているかどうかをすばやく確認できます。ファイルがない場合、プリロードはサポートされていないか、望ましくありません。

キーの機能を見てみましょう。

プリコンパイル

これらのファイルはComposerによって実行されます。各スクリプトは、それらをプリロードリストに追加するために、絶対ファイルパスの配列を返す必要があります。これは、メインリストの役割を果たします。

"pre-compile": [
   "my-script.php",
   "my-other-script.php"
]

これらのファイルは、指定された順序で実行されます。

目標は、開発者が単一のJSONファイルに依存するのではなく、適切と思われるファイルのリストを作成することです。これらのファイルは最初に実行されます。そして、はい、このキーのみでpreload.jsonを実装できます。ここではPHPファイルについて説明しているため、配列をコンパイルするときに、他のファイルを追加することもできます。

extensions

これは、プリロードする必要があるファイル拡張子のリストです。デフォルトでは、php拡張子を持つファイルのみが取得されます。 

"extensions": ["php", "php5", "php7"]

たとえば、いくつかの便利なPHPファイルを含む* .phtmlファイルで満たされたディレクトリを追加できます。Composerはそれらのファイルのみを選択し、ディレクトリのコンテンツ全体は選択しません。
ご理解のとおり、このプロセスは、ファイルを手動で追加することで置き換えることができます。

files

このキーは、composer.jsonの場所に相対的なパスを持つリストからすべてのファイルをダウンロードするようにComposerに指示します。

"files": [
   "helpers.php",
   "app/Models/*",
   "app/Controllers/*/Http/*",
   "app/Views/Compiled*.php",
]

リストを見つけるのは簡単です：

  • 相対パスを使用してファイルとディレクトリを追加します。
  • ディレクトリから、それらに格納された子ファイルのみが追加されます（再帰的にではありません）。
  • 再帰パスは、アスタリスクの末尾（*）で示されます。
  • この記号を使用して、たとえば、特定のファイルやディレクトリを追加することもできます：src/Clients/*/Storesまたはsrc/Model*.php。

アプリケーション固有のスクリプトを手動で選択または作成せずに、マスクによってファイルを追加することは、大規模なアプリケーションを開発するときに特に役立ちます。Composer JSONファイルのautoloadキー

を使用してすべてのファイルをプリロードするだけの場合は、それに値を割り当てますtrue。
 
名前空間

このキーは次のように与えられた名前空間やクラス名を使用してロードファイルに作曲を伝えますfileかdirectory。同じメカニズムで、インストールされている他のパッケージからスペース名を動的に呼び出すことができます。

"namespaces": [
   "App\\Models",
   "App\\Controllers\\",
   "App\\Views\\MainView",
   "Vendor\\Package\\*",
]

これは、いつでも変更される可能性のあるファイルではなく、名前空間に依存する大きなアプリケーションで作業する場合にも便利です。 Composerは、名前空間に従ってファイルを自動的に抽出し、それらをリストに入れます。

packages

このキーを使用すると、名前空間に関連付けられた補助ファイルやクラスなど、外部パッケージから登録された他のファイルをロードできます。

"packages": {
   "symfony/http-client": true,
   "robert/*-client": true,
   "vendor/package": {
       "files": true,
       "namespace": true
   },
   "foo/bar": {
       "files": {
           "helpers.php",
           "loaders/*"
       },
       "namespace": [
           "Foo\\Bar\\DynamicLoaders\\*",
           "Foo\\Bar\\Clients"
       ]
   }
}

ここではすべてが非常に単純です。値がtrueの場合、このパッケージのcomposer.jsonファイルのautoloadキーのコンテンツ全体がロードされます。それ以外の場合は、プリロードの追加をより細かく制御できます。

キーの値がtrueの場合、に登録されてautoloadいるすべてのファイルがダウンロードされます。デフォルト値はfalseです。これはキーにも当てはまりますnamespace。

このルールを使用して、個々のファイルまたは名前空間を選択することもできます。ただし、この場合、キーautoloadは使用されません。

output

これは、コンパイルされたプリロードリストのファイル名です。

"output": "preload-compiled.php"

簡単な組み立て


プリロードリストの準備ができました。Composerを呼び出して、メインのプリロードスクリプトをコンパイルできます。

composer preload

その結果、PHPがプリロードする必要があるすべてのファイルを含むpreload-compiled.phpが作成されます。もちろん、ファイル名は自由に変更できます。 

また、preloadパラメーターでキーをオーバーライドする必要があります。

composer preload \
   --input=my-custom-preload-list.json \
   --output=my-preload.php

デフォルトで無効


preload.jsonがないプロジェクトは、プリロード用にファイルをコンパイルしようとするとエラーを返します。その理由は、Composerがプリロードする必要があると推測しない（そして推測しない）ためです。

プリロードがComposerの通常の機能を妨げることはありません。これはコンソールコマンドであるため、ローカル開発では、プリロードを完全に拒否できます。Composerのプリロードメカニズムに必要なのは、オートロードファイルだけです。これは、不在の場合に生成されます。さて、ほぼ2020年は庭の年です、PSR-4はどこでも使用されていますよね？

結果


次のようなphpファイルを取得する必要があります。

<?php
/**
 * Preloading @generated by Composer
 */
 
// Autoload the classes so those can be preloaded using `require_once`.
require_once __DIR__.'/../autoload.php';

// File list
$files = [
    '/var/www/app/Foo.php',
    '/var/www/app/Bar.php',
    '/var/www/helpers/basic.php',
    '/var/www/helpers/advanced.php',
    '/var/www/vendor/Foo/Bar/src/Class.php',
    '/var/www/vendor/Foo/Bar/helpers/helpers.php',
    '/var/www/vendor/Foo/Bar/config.php',
    // ...
];

// Preload all root project files
foreach ($files as $file) {
    require_once $file;
}

実際、これはComposerのオートローダー機能を使用してプリロードされるファイルのリストにすぎません。PHPはこのファイルを1回実行し、それが履歴になります。

私はComposerがハックを書かなくてもファイルをプリロードできるようになることを心から願っています。

上記の方法はComposerコアの一部ではないため、OPCache分析に基づいて、プリロード用の最も重要なファイルを選択して、必要の少ないファイルを変更する必要はありません。100 MBの容量を持つ1,500ファイルをプリロードする代わりに、元のパフォーマンスの99％を維持しながら、10 MBの容量を持つ150ファイルのみをダウンロードできると想像してください。

（...）説明します。近い将来、Composerへのプリロードに関連するものは追加されなくなります。

どうして？PHPでのプリロードは（依存関係ではなく）開発の問題であり、手動でphp.iniを編集することで解決します。開発者自身がPHPを管理している場合のみ、これを実行できます。

しかし、これでプロジェクトをプリロードするための独自のパッケージを作成することはできます。あなたもね。

プリロードとメトリック


プリロードは、深刻な処理をせずに生産性を簡単かつ安価に向上させるための優れたツールです。

しかし、問題はプリロードの方法ではなく、何であるかです。フレームワーク全体と数千のファイルをプリロードすると、メモリがすぐに使い果たされるため、少なくとも大規模なプロジェクトでは、盲目的に実行することはできません。最も要求されたファイルのみをダウンロードすることをお勧めします。しかし、それらをどのように定義しますか？

幸いなことに、OPCacheでは、opcache_get_status（）を使用して、最もアクセスされているファイルに関するデータを収集できます。最も需要の高いファイルを見つけるだけでなく、アプリケーションの起動後、ファイルがどれだけの時間を消費しているかを知ることができます。

1週間、またはOPCacheが特定の数のヒットを登録するまで待つことをお勧めします。それはすべてアプリケーションに依存しますが、要点はわかります。 
それでは、最も人気のあるファイルの統計に基づいてプリロードリストを作成しましょう。このパッケージを作りました。

想像してみてください...プリローダー！


このパッケージは、アプリケーションのプリロードリストを自動的に作成します。 OPCacheの使用統計を収集し、ファイルをヒット数でソートし、合計ファイルサイズが指定されたしきい値を超えないようにリストを作成します。



リストを作成するための最良の戦略を探して、長い間戸惑いました。そして、メモリ制限（パッケージのデフォルトでは32 MB）に達するまで、すべてのファイルを追加するのが最善であるという結論に達しました。ファイルはヒット数でソートされ、パッケージ自体は自動的に除外されます。

つまり、PHPはほとんどのリクエストを処理するときに、アプリケーションのパフォーマンスを向上させます。

そしてそれをどのように使うのですか？ Preloaderスクリプトを書き込む場所をComposer Autoloaderに指示すれば完了です。

use DarkGhostHunter\Preloader\Preloader;
Preloader::make()
   ->autoload('vendor/autoload.php')
   ->output('preload.php')
   ->generate();

もちろん、いつ生成するかを選択する必要がありますが、それがすべてのポイントです。これをランダムに実行して、たとえば100回のリクエストごとにリストを書き換えることもできます。

use DarkGhostHunter\Preloader\Preloader;
Preloader::make()
   ->whenOneIn(100)
   ->autoload('vendor/autoload.php')
   ->output('preload.php')
   ->overwrite()
   ->generate();

php.iniに配置できる既製のプリロードスクリプトを取得します。

<?php
/**
 * This file is generated automatically by Preloader.
 *

 * PHP at startup. Additionally, this file also includes information about Opcache.
 *
 *
 * Add (or update) this line in `php.ini`:
 *
 * opcache.preload=/www/app/vendor/preload.php
 *
 * --- Config ---
 * Generated at: 2019-11-20 15:20:49 UTC
 * Opcache
 *     - Used Memory: 130585 B
 *     - Free Memory: 294896 B
 *     - Wasted Memory: 347764 B
 *     - Cached files: 2675
 *     - Hit rate: 94%
 *     - Misses: 542
 * Preloader config
 *     - Memory limit: 32 MB
 *     - Overwrite: false
 *     - Files excluded: 0
 *     - Files appended: 0
 */

require_once '/www/app/vendor/autoload.php';

$files = [
    '/www/app/ClassFoo.php',
    '/www/app/ClassBar.php',
    '/www/app/ClassBaz.php',
    '/www/app/ClassQuz.php',
    '/www/app/ClassQux.php',
    '/www/app/vendor/author/package/src/Example.php',
    // ...
];

foreach ($files as $file) {
    require_once $file;
}
