PVS-Studio analyzer: mengidentifikasi potensi masalah kompatibilitas Java SE API











2019 adalah tahun yang sangat sibuk dalam hal konferensi. Tim kami bisa melakukan perjalanan bisnis selama berminggu-minggu. Dan seperti yang Anda tahu, konferensi adalah waktu untuk berbagi pengetahuan. Selain fakta bahwa kami membuat presentasi dan menceritakan banyak hal menarik di stan kami, kami juga belajar banyak dari berkomunikasi dengan peserta konferensi dan dari pembicara. Jadi, pada konferensi musim gugur Joker 2019, sebuah laporan oleh Dalia Abo Sheasha "Migrasi di luar Jawa 8" mengilhami kami untuk menerapkan aturan diagnostik baru yang memungkinkan Anda untuk mengidentifikasi ketidakcocokan dalam Java SE API antara berbagai versi Jawa. Ini akan dibahas.



Relevansi dengan Masalah Kompatibilitas Java SE



Java SE 14 telah dirilis. Meskipun demikian, banyak perusahaan terus menggunakan versi Java sebelumnya (Java SE 6, 7, 8, ...). Karena waktu berjalan dan Java diperbarui setiap saat, masalah kompatibilitas antara berbagai versi Java SE API menjadi semakin relevan setiap tahun.



Ketika versi baru Java SE dirilis, mereka biasanya kompatibel dengan versi sebelumnya, yaitu, misalnya, aplikasi yang dikembangkan berdasarkan Java SE 8 harus berjalan tanpa masalah pada versi 11 Java. Namun, dalam praktiknya, beberapa ketidakcocokan mungkin timbul dalam sejumlah kelas dan metode. Ketidakcocokan ini terletak pada kenyataan bahwa beberapa API mengalami perubahan: mereka dihapus, diubah dalam perilaku, ditandai sebagai usang dan banyak lagi.



Masalah ini hanya akan memburuk ketika Anda mulai berpikir tentang migrasi proyek Anda ke Java SE yang lebih baru. Atau ketika dukungan teknis dari aplikasi Anda akan menerima semakin banyak surat bahwa aplikasi berperilaku salah atau tidak dapat memulai sama sekali.



Saya pikir ini cukup untuk menarik perhatian!



Toolkit yang Ada



Tidak ada solusi satu ukuran untuk semua untuk beralih dari satu versi Java ke versi lain. Dan jika aplikasi Anda adalah hasil dari pengembangan berkelanjutan selama bertahun-tahun menggunakan berbagai solusi non-sepele, maka ketika Anda bingung dengan transisi ke versi terbaru Java, Anda cenderung menghadapi proses yang agak memakan waktu.



Proses ini terdiri dari mengidentifikasi API yang bermasalah atau berpotensi bermasalah yang perlu ditinjau dan diganti dengan solusi alternatif. Ini serius dapat mempengaruhi logika bisnis, yang dapat mengambil banyak waktu untuk diperbaiki dan diuji. Jika Anda tidak hanya perlu beralih ke versi Java SE yang baru, tetapi juga untuk memastikan kompatibilitas dengan sejumlah versi, maka tugas ini akan menjadi jauh lebih rumit.



Setelah mencari-cari di Internet tentang topik mengidentifikasi ketidakcocokan API antara Java SE yang berbeda, saya hanya menemukan alat yang datang dengan JDK: javac, jdeps, jdeprscan.



Tidak ada alat pihak ketiga dalam bisnis ini, kecuali yang saya punya kehormatan untuk bertemu di laporan Joker 2019 - Migration Toolkit for Application Binaries .



javac



Saat mengembangkan aplikasi, jangan lupa tentang peringatan kompiler. Setiap orang memiliki sikap berbeda terhadap peringatan:



  • Seseorang dengan ketat mengawasi mereka dan sesuatu kecil segera membunyikan alarm. Mungkin sedikit paranoid, tetapi mungkin mengurangi risiko teknis di masa depan;
  • Seseorang mengumpulkan mereka dalam file terpisah dengan pemikiran bahwa ia harus meninjau mereka, bagaimana ia akan dibebaskan;
  • Seseorang umumnya mengabaikan peringatan.


Opsi pertama adalah baik karena bagian dari masalah dapat diselesaikan tanpa penundaan, misalnya, tidak menggunakan metode atau kelas yang ditandai sebagai usang. Menggunakan API semacam itu bukan masalah pemblokiran, tetapi Anda harus memperhatikannya, karena sudah ada kemungkinan bahwa ketika menggunakan versi Java yang baru dirilis aplikasi Anda akan berperilaku berbeda atau bahkan crash.



jdeps



Jika Anda sudah bingung dengan masalah migrasi aplikasi Anda ke versi Java SE yang lebih baru, maka jdeps sedang terburu-buru untuk membantu Anda.



jdeps adalah alat baris perintah yang melakukan analisis statis dari dependensi aplikasi dan pustaka Anda, menerima file * .class atau * .jar sebagai input . Dimulai dengan Java SE 8, ia dibundel dengan JDK.



Dan di sini kami tertarik pada kenyataan bahwa jika Anda menjalankan alat ini dengan opsi --jdk-internals , ia akan memberi tahu Anda API JDK internal yang bergantung pada setiap kelas Anda. Ini adalah poin yang sangat penting karena fakta bahwa API internal tidak menjamin bahwa itu tidak akan berubah di versi Java yang akan datang.



Mari kita lihat sebuah contoh. Misalkan Anda telah mengembangkan aplikasi Java 8 Anda untuk waktu yang lama dan Anda tidak pernah bingung dengan kompatibilitas Java SE API yang digunakan dengan versi yang lebih baru. Ada pertanyaan tentang memigrasikan aplikasi Anda ke, misalnya, Java 11. Dalam hal ini, Anda dapat mengikuti jalur resistensi minimal dan segera memulai aplikasi menggunakan Java 11. Tetapi ada yang tidak beres, dan misalkan aplikasi Anda mulai macet. Dalam hal ini, Anda dapat menjalankan jdeps dari Java 11 dengan mengatur file * .jar aplikasi Anda.



Hasilnya akan kira-kira sebagai berikut:













Output menunjukkan bahwa dependensi sun.misc.BASE64Encoder di Java 11 akan dihapus. Dan crash aplikasi Anda ketika pertama kali diluncurkan di Java 11 kemungkinan besar disebabkan oleh kesalahan java.lang.NoClassDefFoundError . Selain memberi tahu Anda bahwa itu tidak dapat dilakukan, jdeps dapat menawarkan kepada Anda ketergantungan alternatif yang dapat digunakan sebagai pengganti ketergantungan saat ini. Dalam hal ini, ia menyarankan untuk mengganti ketergantungan jarak jauh dengan java.util.Base64 .



Peringatan kedua dari alat memberi sinyal bahwa kita masih menggunakan dependensi internal lain dalam kode, tetapi sejauh ini sudah benar. Apakah akan ada perubahan pada API ini di versi Java yang akan datang, tetapi ini harus diperhitungkan.



Tentu saja, jdeps dapat melakukan lebih dari sekedar menguji penggunaan komponen JDK internal. Ini tidak termasuk dalam ruang lingkup artikel ini, tetapi Anda dapat membiasakan diri dengan kemampuannya di halaman resmi sendiri.



jdeprscan



Tujuan jdeprscan persis sama dengan tujuan jdep, yaitu membantu dalam menemukan API yang tidak diinginkan dan bermasalah.



jdeprescan adalah alat analisis statis yang memindai file * .jar (atau kumpulan file * .class lainnya ) untuk menggunakan elemen API yang ketinggalan jaman. Menggunakan API yang sudah usang bukan masalah pemblokiran, tetapi Anda harus memperhatikannya. Dimulai dengan Java SE 9, ia dibundel dengan JDK.



Juga anggap ada pertanyaan tentang migrasi aplikasi ke Java 11. Dalam hal ini, dengan menjalankan perintah 



jdeprscan --release 8 app.jar


Anda akan mendapatkan daftar API yang sudah tidak digunakan lagi untuk Java 8, yaitu API yang mungkin dihapus di versi Java yang akan datang. Setelah mengoreksi semua peringatan, Anda dapat menjalankan 

jdeprscan --release 11 app.jar


yang akan menghasilkan daftar API yang sudah usang untuk Java 11. Dengan cara ini, Anda dapat menemukan dan memperbaiki (jika perlu) semua API yang tidak direkomendasikan.



Toolkit Migrasi untuk Binari Aplikasi



Alat ini dirancang untuk membantu Anda dengan cepat mengevaluasi aplikasi pengguna Anda untuk masalah potensial sebelum menempatkannya di server yang berbeda (JBOSS, WebShere, Tomcat, WebLogic, ...). Selain semua fungsi, alat ini juga memungkinkan Anda untuk mengidentifikasi perbedaan dalam Java SE API versi yang berbeda.



Mari kita lihat sekilas tentang apa alat ini.



Peluncuran cepat alat ini terlihat seperti ini: 



java -jar binaryAppScanner.jar yourApp.jar --analyzeJavaSE 
--sourceJava=oracle8 --targetJava=java11 ....


Opsi analyJavaSE menyiratkan penggunaan berbagai parameter, yang dapat Anda pelajari dengan menelepon bantuan.



Setelah memulai analisis, Anda akan segera melihat laporan di browser web:















Tidak semuanya cocok di layar = (. Dan sementara Anda belum mencoba menjalankan alat ini, saya akan menggambarkannya dalam kata-kata.



Dalam laporan, Anda dapat melihat pemicu aturan dengan 3 tingkat keparahan:



  • Aturan dengan tingkat keparahan yang tinggi (API jarak jauh, perubahan perilaku yang membuat aplikasi tidak dapat digunakan dan perlu diperbaiki);
  • Aturan peringatan (mengubah perilaku API yang dapat menyebabkan masalah dan mereka perlu ditelusuri);
  • Aturan informasi (menggunakan API yang sudah ketinggalan zaman, perilaku API yang mungkin sedikit mengubah perilaku, tetapi tidak memengaruhi program).


Setiap operasi dapat diperluas dan melihat deskripsi tentang apa dan bagaimana. Ada rekomendasi dalam pesan informasi, mereka mengatakan menjalankan jdeps selain kita. Mereka mengatakan kami fokus pada migrasi aplikasi, dan jdeps juga akan membantu mendeteksi masalah dalam paket JDK internal (selain apa yang mereka temukan).



Juga di bagian bawah laporan Anda dapat menemukan daftar aturan yang sesuai dengan analisis yang dilakukan.



Jika Anda menggunakan Eclipse IDE, maka Anda dapat menggunakan plugin . Untuk studi yang lebih rinci, Anda harus mengunjungi halaman mereka .



Implementasi di PVS-Studio



Setelah mempelajari alat yang dipertimbangkan, kami sampai pada kesimpulan bahwa pencarian potensi masalah kompatibilitas dari berbagai versi Java SE API adalah tugas yang layak untuk analisis statis.



Alat-alat yang dibahas benar-benar akan membuatnya lebih mudah untuk memigrasi aplikasi atau mencari API yang dapat merusak kinerja aplikasi Anda pada versi Java SE yang lebih baru (jika Anda tidak ingin melakukan migrasi aplikasi). Tetapi berpikir bahwa alat-alat ini sepanjang waktu yang Anda butuhkan untuk berjalan di baris perintah secara terpisah dari proses pengembangan untuk mengidentifikasi masalah, kami sampai pada keputusan bahwa ini sangat tidak nyaman. Dan mulai dari fakta bahwa analisis statis diperlukan untuk mendeteksi kode yang bermasalah atau berpotensi bermasalah pada tahap awal pengembangan, kami menerapkan aturan diagnostik V6078, yang akan memberi tahu Anda tentang API "masalah".



Aturan V6078 memperingatkan Anda sebelumnya bahwa kode Anda bergantung pada fungsi dan kelas Java SE API tertentu, yang dapat menyebabkan Anda kesulitan dalam versi Java yang akan datang. Dan Anda di awal implementasi fitur tertentu tidak akan terikat dengan API ini, sehingga mengurangi risiko teknis di masa depan.



Aturan diagnostik mengeluarkan peringatan dalam kasus berikut:



  • Jika metode / kelas / paket dihapus dalam versi target Java;
  • Jika metode / kelas / paket ditandai tidak digunakan lagi dalam versi target Java;
  • Jika metode telah mengubah tanda tangan.


Aturan saat ini memungkinkan Anda untuk menganalisis kompatibilitas Oracle Java SE dari versi 8 hingga 14. Agar aturan menjadi aktif, aturan harus dikonfigurasi.



IntelliJ IDEA



Di plugin IntelliJ IDEA, Anda harus mengaktifkan aturan dan menentukan parameter di Pengaturan> PVS-Studio> tab Deteksi Masalah Kompatibilitas API, yaitu:



  • Source Java SE - versi Java tempat aplikasi Anda dikembangkan;
  • Target Java SE - versi Java yang ingin Anda periksa kompatibilitas API yang digunakan dalam aplikasi Anda (Source Java SE);
  • Kecualikan paket - paket yang ingin Anda kecualikan dari analisis kompatibilitas (paket dicantumkan dengan koma).












Plugin untuk gradle



Menggunakan plugin gradle, Anda perlu mengonfigurasi pengaturan analyzer di build.gardle: 



apply plugin: com.pvsstudio.PvsStudioGradlePlugin
pvsstudio {
    ....
    compatibility = true
    sourceJava = /*version*/  
    targetJava = /*version*/
    excludePackages = [/*pack1, pack2, ...*/]
}


Plugin untuk Maven



Menggunakan plugin maven, Anda perlu mengonfigurasi pengaturan analyzer di pom.xml: 



<build>
  <plugins>
    <plugin>
      <groupId>com.pvsstudio</groupId>
      <artifactId>pvsstudio-maven-plugin</artifactId>
      ....
      <configuration>
        <analyzer>
          ....
          <compatibility>true</compatibility>
          <sourceJava>/*version*/</sourceJava>
          <targetJava>/*version*/</targetJava>
          <excludePackages>/*pack1, pack2, ...*/</excludePackages>        
        </analyzer>
      </configuration>
    </plugin>
  </plugins>
</build>


Menggunakan kernel secara langsung



Jika Anda menggunakan analisa secara langsung melalui baris perintah, maka untuk mengaktifkan analisis kompatibilitas dari Java SE API yang dipilih, Anda harus menggunakan parameter berikut: 



java -jar pvs-studio.jar /*other options*/ --compatibility 
--source-java /*version*/ --target-java /*version*/ 
--exclude-packages /*pack1 pack2 ... */


Penganalisa operasi



Mari kita asumsikan bahwa kita sedang mengembangkan aplikasi berbasis Java SE 8, dan kita memiliki kelas dengan konten berikut: 



/* imports */
import java.util.jar.Pack200;
public class SomeClass
{
  /* code */
  public static void someFunction(Pack200.Packer packer, ...)
  {
    /* code */
    packer.addPropertyChangeListener(evt -> {/* code */});
    /* code */
  }
}


Menjalankan analisis statis dengan berbagai pengaturan untuk aturan diagnostik, kami akan mengamati gambar berikut:



  • Sumber Java SE - 8, Target Java SE - 9
    • Metode 'addPropertyChangeListener' akan dihapus.


  • Sumber Java SE - 8, Target Java SE - 11
    • Metode 'addPropertyChangeListener' akan dihapus.
    • Kelas 'Pack200' akan ditandai sebagai usang.


  • Sumber Java SE - 8, Target Java SE - 14
    • Kelas 'Pack200' akan dihapus.




Pertama, di Java SE 9, metode 'addPropertyChangeListener' di kelas 'Pack200.Packer' telah dihapus. Dalam versi 11, fakta bahwa kelas 'Pack200' ditandai sebagai usang ditambahkan ke ini. Dan di versi 14, kelas ini benar-benar dihapus.



Oleh karena itu, memulai aplikasi di Java 11, Anda akan mendapatkan 'java.lang.NoSuchMethodError', dan jika Anda menjalankan di Java 14 - 'java.lang.NoClassDefFoundError'.



Mengetahui informasi ini, saat mengembangkan aplikasi Anda, Anda akan mempertimbangkan solusi alternatif untuk tugas tersebut.



Gagasan untuk pengembangan lebih lanjut



Selama implementasi aturan diagnostik, muncul ide tentang perluasan:



  • Karena fakta bahwa jdeprscan dan jdeps tidak dapat memperingatkan tentang menggunakan refleksi untuk mengakses API yang dienkapsulasi, masuk akal untuk memperbaiki aturan sehingga mencoba untuk mencari tahu jenis API apa yang mereka coba gunakan. Hasilnya mungkin tidak sempurna, tetapi mengapa tidak ?!
  • Ada berbagai implementasi JDK (dari Oracle, IBM, Red Hat, ...). Sebagai aturan, mereka kompatibel satu sama lain. Tetapi berapa perbedaan JDK API internal? Bagaimanapun, pengembang dapat terikat padanya, yang dapat menyebabkan masalah potensial saat memigrasi aplikasi dari satu JDK ke yang lain.


Ini semua adalah pertanyaan penelitian, dan apa yang akan muncul pada akhirnya, waktu akan memberi tahu =) Jika Anda tahu skenario yang menarik untuk diagnosis ini dan ingin melihatnya di PVS-Studio, kemudian menulis kepada kami .



Kesimpulan



Pencarian untuk potensi kesalahan inkompatibilitas sepenuhnya konsisten dengan ideologi kami - menggunakan PVS-Studio untuk menemukan dan memperbaiki kesalahan pada tahap awal penulisan kode. Seperti kesalahan ketik, panggilan ke fungsi tertentu dapat dimasukkan ke dalam kode kapan saja, oleh karena itu secara teratur menjalankan PVS-Studio pada proyek sekarang dua kali lebih berguna.



Diagnostik V6078 tersedia di penganalisis mulai dari versi 7.08 . Anda dapat mengunduh dan mencoba penganalisa pada proyek Anda di halaman unduhan .
