GraalVMを使用して、Spring Bootアプリケーションをネイティブにコンパイルします

「Spring Frameworkの開発者」コースの開始前に、記事の翻訳が準備されました。




春の恋人たち、こんにちは！Spring Tipsの次のリリースへようこそ。今日は、GraalVMでSpring Bootアプリケーションをコンパイルするために最近実装されたサポートについて説明します。Spring Fuのトピックに関する別のSpring Tipsリリースで、GraalVMとネイティブアプリケーションについてすでに説明しました。



GraalVMとは何かを思い出してみましょう。GraalVMは、OpenJDKの標準C1コンパイラの代替品です。GraalVMの使用について詳しくは、私のBootiful Podcastポッドキャストで、GraalVMの寄稿者でありTwitterエンジニアであるChris Thalingeと一緒に読むことができます。特定の条件下では、GraalVMを使用すると、通常のSpringアプリケーションをより高速に実行できます。少なくともこの理由から、注目に値します。

しかし、それについては触れません。他のGraalVMコンポーネント（ネイティブイメージビルダーとSubstrateVM）を見ていきます。 SubstrateVMを使用すると、Javaアプリケーションのネイティブ実行可能ファイルを作成できます。ちなみに、GralVMのこれと他の用途については、Oracle LabsのOleg Shelaevとのポッドキャストがありました。。ネイティブイメージビルダーは、妥協のテストです。 GraalVMに、実行時のアプリケーションの動作に関する十分な情報（動的にリンクされたライブラリ、リフレクション、プロキシなど）を提供すると、Javaアプリケーションを静的にリンクされたバイナリ（CやGolangのアプリケーションなど）に変換できるようになります。正直なところ、このプロセスは非常に骨の折れる作業です。しかし、そうすれば、信じられないほど高速になるネイティブコードを生成できます。その結果、アプリケーションのRAM使用量が大幅に減少し、1秒未満で実行されます。 1秒未満。かなり魅力的ですね。もちろん！

ただし、いくつかの点を考慮する必要があることを忘れないでください。結果のGraalVMバイナリはJavaアプリケーションではありません。通常のJVMでは実行されません。 GraalVMはOracle Labsによって開発されており、JavaチームとGraalVMチームの間にはなんらかの相互作用がありますが、私はそれをJavaとは呼びません。結果のバイナリはクロスプラットフォームではありません。アプリケーションはJVMを使用しません。 SubstrateVMと呼ばれる別のランタイムで実行されます。

したがって、ここには多くのトレードオフがありますが、それでもGraalVMには、特にスケーリングと効率が最優先であるクラウドベースのアプリケーションにとって、大きな可能性があると思います。

はじめましょう。 GraalVMをインストールします。ここからダウンロードするか、SDKManagerを使用してインストールできます。。Javaディストリビューションのインストールには、SDKManagerを使用するのが好きです。GraalVMはJavaの最新バージョンよりわずかに遅れており、現在Java 8および11をサポートしています。Java14または15以降（これを読んだときに存在するバージョン）のサポートがありません。

GraalVM for Java 8をインストールするには、次のコマンドを実行します。

sdk install java 20.0.0.r8-grl

Java 11にはまだ理解していない不明なエラーがあるため、Java 11ではなくJava 8を使用することをお勧めします。

その後、ネイティブイメージビルダーコンポーネントをインストールする必要があります。実行：

gu install native-image
gu-これはGraalVMのユーティリティです。

最後に、JAVA_HOMEGraalVMのポイントを確認します。私のマシン（SDKMANを搭載したMacintosh）では、JAVA_HOME次のようになります。

export JAVA_HOME=$HOME/.sdkman/candidates/java/current/

すべての設定が完了したので、アプリケーションを見てみましょう。Spring Initializrに移動し、Lombok、R2DBC、PostgreSQL、Reactive Webを使用して新しいプロジェクトを生成します。

100万回も同じようなコードを見たので、分解しないで、ここで説明します。

package com.example.reactive;

import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;
import lombok.extern.log4j.Log4j2;
import org.springframework.boot.ApplicationRunner;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.annotation.Id;
import org.springframework.data.r2dbc.core.DatabaseClient;
import org.springframework.data.repository.reactive.ReactiveCrudRepository;
import org.springframework.web.reactive.function.server.RouterFunction;
import org.springframework.web.reactive.function.server.ServerResponse;
import org.springframework.web.reactive.handler.SimpleUrlHandlerMapping;
import org.springframework.web.reactive.socket.WebSocketHandler;
import org.springframework.web.reactive.socket.WebSocketMessage;
import org.springframework.web.reactive.socket.server.support.WebSocketHandlerAdapter;
import reactor.core.publisher.Flux;

import java.time.Duration;
import java.time.Instant;
import java.util.Collections;
import java.util.stream.Stream;

import static org.springframework.web.reactive.function.server.RouterFunctions.route;
import static org.springframework.web.reactive.function.server.ServerResponse.ok;

@Log4j2
@SpringBootApplication(proxyBeanMethods = false)
public class ReactiveApplication {

    @Bean
    RouterFunction<ServerResponse> routes(ReservationRepository rr) {
        return route()
            .GET("/reservations", r -> ok().body(rr.findAll(), Reservation.class))
            .build();
    }

    @Bean
    ApplicationRunner runner(DatabaseClient databaseClient, ReservationRepository reservationRepository) {
        return args -> {

            Flux<Reservation> names = Flux
                .just("Andy", "Sebastien")
                .map(name -> new Reservation(null, name))
                .flatMap(reservationRepository::save);

            databaseClient
                .execute("create table reservation ( id   serial primary key, name varchar(255) not null )")
                .fetch()
                .rowsUpdated()
                .thenMany(names)
                .thenMany(reservationRepository.findAll())
                .subscribe(log::info);
        };
    }


    public static void main(String[] args) {
        SpringApplication.run(ReactiveApplication.class, args);
    }
}

interface ReservationRepository extends ReactiveCrudRepository<Reservation, Integer> {
}


@Data
@AllArgsConstructor
@NoArgsConstructor
class Reservation {

    @Id
    private Integer id;
    private String name;
}

ここで 完全なコードを見ることができます。

このアプリケーションの唯一の機能は、Spring Boot属性proxyBeanMethodsを使用して、アプリケーションがJDK以外のcglibまたは他のプロキシを使用しないようにすることです。GraalVMは非JDKプロキシをサポートしていません。JDKプロキシを使用する場合でも、GralVMがそれらについて学習できるように、いじくる必要があります。Spring Framework 5.2の新機能であるこの属性は、部分的にはGraalVMをサポートすることを目的としています。

それでは次に進みましょう。実行時にアプリケーションに存在する可能性のあるいくつかの点についてGraalVMに通知する必要があること、およびネイティブコードを実行するときに理解できない可能性があることを先に述べました。これらは、リフレクション、プロキシなどのようなものです。これを行うにはいくつかの方法があります。構成を手動で記述して、アセンブリに含めることができます。 GraalVMは自動的にそれを取得します。もう1つの方法は、アプリケーションの動作を監視するJavaエージェントを使用してプログラムを実行し、アプリケーションの終了後にすべてを構成ファイルに書き込んで、GraalVMコンパイラに転送できるようにすることです。

GraalVM機能を使用することもできます。（注意トランスレーター：「機能」-GraalVMという用語は、ネイティブコンパイル用のプラグインを示し、クラスファイルからネイティブバイナリを作成します）。 GraalVM機能はJavaエージェントに似ています。何らかの分析を行い、GraalVMコンパイラに情報を渡すことができます。この機能は、Springアプリケーションの動作を理解しています。彼女はいつ春の豆がプロキシになるか知っています。彼女は実行時にクラスを動的に作成する方法を知っています。彼女は、Springの仕組みを知っており、少なくともほとんどの場合、GraalVMが何を望んでいるのかを知っています（結局のところ、これは初期リリースです！）

GraalVMコンパイラには、アプリケーションの動作に関する情報を提供するために、Javaエージェント、GraalVM機能、コマンドラインパラメータなど、できるだけ多くの方法が必要です。これらすべてにより、アプリケーションを静的にコンパイルされたネイティブバイナリに正常に変換するための十分な情報がGraalVMに提供されます。長期的には、私たちの目標は春のプロジェクトです。そして、Spring GraalVM機能は、それらをサポートするために必要なすべてを提供します。

すべての設定が完了したので、アプリケーションを組み立てましょう。

  • 通常の方法でJavaアプリケーションをコンパイルします。
  • 情報を収集するには、Javaエージェントを使用してJavaアプリケーションを起動します。この段階で、アプリケーションが機能していることを確認する必要があります。考えられるすべてのユースケースを検討する必要があります。ちなみに、これはCIとテストを使用するための非常に良いケースです！誰もがアプリケーションのテストとパフォーマンスの向上について常に話し合っています。GraalVMを使用すると、両方を実行できます！
  • 次に、今回はアクティブなgraalプロファイルを使用してアプリケーションを再構築し、最初の起動時に収集された情報を使用してネイティブアプリケーションにコンパイルします。

mvn -DskipTests=true clean package
export MI=src/main/resources/META-INF
mkdir -p $MI 
java -agentlib:native-image-agent=config-output-dir=${MI}/native-image -jar target/reactive.jar

## it's at this point that you need to exercise the application: http://localhost:8080/reservations 
## then hit CTRL + C to stop the running application.

tree $MI
mvn -Pgraal clean package

すべてがエラーなしで成功した場合、ディレクトリtargetにはコンパイルされたアプリケーションが表示されます。それを実行します。

./target/com.example.reactive.reactiveapplication

これと同様の出力からわかるように、アプリケーションが起動します。

2020-04-15 23:25:08.826  INFO 7692 --- [           main] c.example.reactive.ReactiveApplication   : Started ReactiveApplication in 0.099 seconds (JVM running for 0.103)

悪くない？GraalVMネイティブイメージビルダーは、CloudFoundryやKubernetesなどのクラウドプラットフォームと組み合わせるのに最適です。アプリケーションを簡単にコンテナーにアセンブルし、最小限のリソースでクラウドで実行できます。
