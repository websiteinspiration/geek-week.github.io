Sim-simオープンまたはリバースエンジニアリングスマートインターコム

投稿を読んだ後 クルプニカmitmproxyに対処し、毎日使用されるモバイルアプリケーションのバックエンドがどのように配置されるかを確認するというアイデアがありました。選択はインターコムアプリケーションにかかっています。認証後、ドアを開けてビデオ通話に応答することができます。これから何が起こったのか、そして私が何とか見つけた穴はカットの下で教えます。



プロキシ設定


トラフィックを分析するには、中間者と呼ばれるアプローチがよく使用されます。これは、ローカルネットワークに接続されている場合、分析されたデバイスからのトラフィックは、まず復号化および分析されたコンピューターに送信され、次に暗号化されてサーバーに送信されるという事実にあります。トラフィックを復号化するために最も一般的なのは、mitmproxyプログラムです。 mitmproxyのインストールは大したことではありませんでした。

モバイルデバイスからのトラフィックを表示するには、電話とコンピューターから自宅のwifiに接続する必要があります。コンピューターにmitmproxyをインストールして実行します。電話のwifi設定で、ローカルアドレスとコンピューターポートをプロキシサーバーとして設定します。次に、電話からmitm.itに移動し、https要求を復号化できる証明書をインストールします。これらの手順の後、ブラウザからのリクエストが表示されるようになりました。やったー！しかし、次のステップは失望した：


グーグル 検索により、githubで未解決の問題が発生しました。APIレベル24以降、アプリケーションはユーザー証明書を信頼しないことが判明しました。幸い、apkを解凍してAndroidManifest.xmlに次の設定を追加することで、これを回避できます。

<network-security-config>  
     <debug-overrides>  
          <trust-anchors>  
               <!-- Trust preinstalled CAs -->  
               <certificates src="system" />  
               <!-- Additionally trust user added CAs -->  
               <certificates src="user" />  
          </trust-anchors>  
     </debug-overrides>  
</network-security-config>

さらに、これを自動的に実行する既製のスクリプトがgithub にあります。したがって、apk、パッチスクリプトをダウンロードし、adb installコマンドを設定して、すべてが機能するようにします。

トラフィック分析


次の形式のアドレスにリクエストが行われることがわかります。

https://{intercom-company-url}/api/

ヘッダーで2つのパラメーターが渡されます。 
'api-version': '2',
'authorization':  'Bearer your.jwt.token’

最初のパラメーターはapiバージョンで、2番目のパラメーターは認証トークンです。認証には、ヘッダー、ペイロード、署名の3つの部分で構成されるjson Webトークンが使用されます。
次のコマンドでデコードします。

pyjwt decode --no-verify your.jwt.token

ペイロードにaccount_idとexpが含まれていることを確認してください。expフィールドは、トークンが作成された時間に対応します。これにより、1つのアカウントからさまざまなデバイスのトークンを生成できます。

私たちにとって最も興味深いのは、利用可能なインターホンとドアを開けるリストのリクエストです。リクエストを送信するとき（私はPythonとリクエストライブラリを使用しました）に

https://{intercom-company-url}/api/customers/properties/{account_id}/intercoms

フィールド付きのドアホンのリストを含むjsonを取得します。

[
    {
        'id': ID,
        'mode': MODE, 
        'sip_account': {'ex_user': USER_ID, 'proxy': PROXY,  'password': PSWD}, 
        'video': [{'quality': 'low', 'source': 'rtsp://LINK }]
    }
]

json id-ドアフォン識別子、モード-開くことができるドア（可能な値はone_door、left_door、right_door）、SIPアカウント、およびビデオブロードキャストへのリンク。うわー！

セッション開始プロトコル（SIP）を使用して、ビデオ通話はインターコムを通じて行われます。 sip_accountフィールドには、アカウントがアクセスできるインターコムのIDのhnikiとパスワードを受け取りました。それらは内部ネットワーク上にあるため、それらをすべて同じように呼び出すことはできません。しかし、ビデオ分野は興味深いものです。カメラからビデオをブロードキャストするための外部リンクが含まれています。 vlcを開いてリンクをコピーするだけで十分です。ドアフォンのカメラで24時間体制で見回すことができます。このようなリンクをばらまくのは良くありません！

次に、ドアを開ける方法を考えます。ドアを開くボタンをクリックすると、アプリケーションはgetリクエストを送信します。

https://{intercom-company-url}/api/customers/intercoms/{intercom_id}/unlock?door=left_door&id={intercom_id}

idパラメータの有無は何の影響も受けないことが判明しましたが、2つのドアを備えたドアホンの場合、ドアによって、自分だけでなく、選択した次のドアも開くことができます。

シムシムオープン！


APIを理解したので、何か便利なことをすることにしました。結果は、音声コマンドを認識するAndroidアプリケーションであり、「sim-sim open」などの事前定義されたコマンドを認識すると、対応するドアを開く要求を送信します。

調査結果


スマートインターホンは優れていますが、安全でスマートなインターコムの方が優れています。
