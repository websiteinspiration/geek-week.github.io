Centosリポジトリミラーと最適なミラーの選択について

昨年、いくつかのLinuxディストリビューションのオンラインミラーをホストしました。これは複雑なプロセスではなく、Ubuntuのような大規模プロジェクトの場合、ほぼ完全に自動化されています。また、メーリングリストなどでプロジェクトに連絡して、希望を明示する必要がある場合もあります。


技術的には、これrsyncは通常スケジュールどおりです。このための誰かが、Fedoraのような既成のスクリプトのセットを提供し、誰かがこのサーバーと推奨されるパラメーターのセットからここで同期する必要があると言っています。最も高価なリソースはその場所です。最近4テラバイトに達しましたが、私たちの場合、これは利益を生み出さないという点で高価です。代わりに、使用しているディストリビューションのローカルでの可用性を受け取りました。これにより、必須のインターネットアクセスを排除することにより、サーバーの初期構成を簡素化することができました。そしてもちろん、これへの参加があまり目立たなくても、何か大きなものに参加できてうれしいです。

私たちのミラーは公開されており、誰もがミラーから更新を受け取ることができます。これは主にロシアですが、それだけではありません。それがどのように判明するか、そしてこの投稿の第7バージョンのCentosの例の更新で特定のサーバーが一般的にどのように選択されるかについて。

yumデフォルトでプラグインがインストールされて いるパッケージマネージャーについて説明しますfastestmirror。特定のミラーを選択するプロセスにのみ関心があります。

ミラーリスト


リポジトリのリストは、/etc/yum.repos.d/特に指定がない限り、ディレクトリ内のファイルで指定されていることがわかっています。これはBase、ファイルのリポジトリ設定は次のようになります/etc/yum.repos.d/Centos-Base.repo。

[base]
name=CentOS-$releasever - Base
mirrorlist=http://mirrorlist.centos.org/?release=$releasever&arch=$basearch&repo=os&infra=$infra
#baseurl=http://mirror.centos.org/centos/$releasever/os/$basearch/
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-CentOS-7

ここには、データを取得できる場所を指定する2つのオプションがあります。最初のbaseurlはミラーを直接示しています。ここでは選択の必要はありません。第二ミラーリストは、それがアクティブであるこのオプションで、選択が行われます、そこから10枚のミラーのリストが返されます、それを通してのリンクを示します。また、リンク内にいくつかの変数があり、それらはすべて最終的にリポジトリディレクトリツリー内の特定の場所を反映しています。

  • リリース -バージョン：6、7、8、8-stream。古いバージョンは既存のインフラストラクチャから削除されていますが、それらのリポジトリのスナップショットはhttp://vault.centos.org/にあります。
  • arch — , i386 x86_64. , . 7 x86_64, i386. ,
  • repo – os, updates , . 8 os baseos
  • インフラ - それがまだ使用されていないことを注意深く推測できます。少なくとも、ミラーのリストを作成するときにその処理を見つけることができませんでした。等しいstockですが、このパラメーターを省略すると、目に見える変化は起こりません
  • ccは国コードで、米国およびカナダの場合は州コードでもあります。リクエストされたIPアドレスによってリクエストされたときに国が計算されるため、上記のファイルの例にはありません。この変数を使用して、地理位置情報エラーを排除できます。ロシアのためにcc=ru

Centosの7番目のバージョンの場合、次の行を取得します。http：//mirrorlist.centos.org/？ release = 7＆arch = x86_64＆repo = os＆infra = stock これにより、次の種類の10個のミラーのリストが表示されます。

http://<   1>/7.xxx/os/x86_64/
http://<   2>/7.xxx/os/x86_64/
...
http://<   10>/7.xxx/os/x86_64/

各ミラーの特定のパスは、クエリ内の変数と完全に一致することに注意してください。以前のバージョンの代わりにフルバージョンが表示されますが、使用することhttp://< >/7/os/x86_64/もできます。また、最新バージョンへのシンボリックリンクを使用して実装されています。

実装


ここで、https：//github.com/CentOS/mirrorlists-codeにある実装について、ミラーのリストがどのように形成されるかに関心があります。パールスクリプトmakemirrorlists-combined.plがこれを行います。その主なタスクは、ファイルのハッシュrepodata/repomd.xmlをこのバージョンとアーキテクチャのリファレンスと比較して、ミラーの活力をチェックすることです。したがって、ファイルは、バージョン、リポジトリーのタイプ、およびアーキテクチャーのすべての使用可能な組み合わせに対して存在する必要があります。

検証は、リストに従って次の順序で実行されます（コードから引用します）。

# phase 1 -- this U.S. state or Canada province
# phase 2 -- this country
# phase 3 -- other nearby countries
# phase 4 -- mirrors from the same continent
# phase 5 -- a random mirror from each continent (for fallback only)
# note: phase 5 will be executed once for each continent (with "redo")
# phase 6 -- random mirrors from any continent (for fallback only)
# phase 7 -- centos servers from the same continent
# phase 8 -- centos servers from other continents (used also for fallback)

パラグラフ1と2はどちらか、または国コードだけでなく州コードもチェックするように機能します。本質的に、地理的に最も近いサーバーを選択する試みが行われます。ステップごとに、データベースからクエリが作成されます。次に例を示します。

SELECT $columns FROM mirrors WHERE type IN ('Direct', 'Indirect') AND status = 'Active' AND cc = '$cc' AND $commonqueryparams $skipregion $altarch_where ORDER BY RAND();

次に、上記のように、ハッシュを比較することにより、このリストのミラーの活力をチェックします。 10人が採用されれば、必要なタスクは完了し、作業は完了です。入力しなかった場合は、次の手順に進んで、一般的なリストを10にするか、すべてのオプションを使い尽くしてください。結果はファイルに保存され、ml.pyスクリプトを使用してPythonによって表示されるパーツのフロントエンドに転送されます。このタスクの目的は、最初に要求元の国をIPアドレスで見つけた後、目的のファイルを選択して提供することです。異なるリストが形成されるIPv4またはIPv6プロトコルのタイプも考慮されます。

を使用するクエリに戻りましょうORDER BY RAND()これはリストがランダムになるということですか？はい、DBMSでの実装はどれほどランダムであるかですが、これは各ステップ内の順序にのみ適用されます。つまり、特定の国で、目的のタイプのリポジトリとアーキテクチャの10を超えるミラーが入力された場合（ロシアの場合は17しかない）、結局、毎回、1つの国で10の異なるミラーのシャッフルリストが表示されます。数が少ない場合は、リストの一番上に常に 1つの国からのシャッフルされたリポジトリ、最も近い国からのシャッフルされたリポジトリなどが段階的に表示されます。その結果、いくつかのランダムな部分で構成される完全にランダムではないリストが得られます。 1つの国にそれほど多くのミラーが機能していない場合は重要です。たとえば、ロシアのIPv6ミラーは7つしかなく、常にリストの一番上にあります。

最速ミラー


合計では、ミラーのリストはせいぜい偶然に形成されたものであり、これはおそらくロードバランシングにとって悪くないことです。最悪の場合、リストから最初のミラーのみを選択することを期待できます。そのような選択肢はありません。さらに、私たちのような大きな国の場合、国の領土分割は、ペトロパブロフスクカムチャツキーhttp://mirror.vilkam.ruのミラーをリストの一番上にあるソチのクライアントに置くことでトリックを演じることができます。したがって、このリストからミラーの可用性を評価するとよいでしょう。これは、fastestmirrorプラグインによって行われます。その本質は単純なアクションに還元されます：

time_before = time.time()
sock.connect((self.host, self.port))
result = time.time() - time_before

なし、HTTPまたはICMPノードのソケットを開き、所要時間を考慮して、結果に適用しますsort()。前のステップで取得した10個のミラーすべてがURL、ノード名のみが使用される完全な形式で入力に入力されます。プラグインは個別に簡単に動作させることができます。52 yum行目と55行目をコメントアウトするだけです。

# from yum.plugins import TYPE_CORE

requires_api_version = '2.5'
# plugin_type = (TYPE_CORE,)

熟したアトラス


地域の選択により、すべてがある程度明確になりました。次に、国全体の状況を見てみましょう。なぜサービスhttps://atlas.ripe.net/を使用するのですか？RIPE NCCの後援の下でプローブ（プローブ）とそれらを操作するためのツールのセットです。これはすぐに警告します。これは最速で最も俊敏なサイトではありません。誰でもこのネットワークを使用するだけでなく、プローブを取得できます。ただし、テストを実行するには、登録済みでアクティブなデバイスから累積する内部通貨が必要です。ロシアでIPv6とIPv4を同時にサポートするデバイスをすべて選びましたが、そのうち82個が判明しましたが、測定結果によると、一部を取り除く必要があり、400個以上のうち 67 個が残っています。

実行できるテストオプションはいくつかありますが、もちろん使用されているものはありません。fastestmirrorでもいつもあるよping。フィルター処理され、テストの実行後mirror.linux-ia64.orgに表示ICMPされたミラーを除いて、すべてのミラーについて、mirror.axelname.ru選択したすべてのデバイスから、IPv6とIPv4で別々に、1週間に3時間ごとに10の要求が行われました。平均応答時間を記録しました。測定セット全体から、各ミラーについて約50回、中央値が取得され、このプローブの他のミラーと比較されました。応答時間が最も短いミラーが勝ちました。
