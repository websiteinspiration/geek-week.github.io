Umka：新しい静的型付きスクリプト言語


私が開発した静的に型付けされた埋め込み可能なスクリプト言語Umkaの最初のバージョンがリリースされました。これは、使い慣れたスクリプト言語の柔軟性とバイトコードへのコンパイルの段階での型エラーに対する保護を組み合わせることを目的としています。言語の主な概念- 明示的は暗黙的よりも優れている -は「Zen of Python」から借用されたものですが、ここではわずかに異なる、より明確な意味が得られるはずです。

言語の開発に着手するきっかけとなった印象がどれほど個人的で主観的であったとしても、その計画が素朴ではなかったことを願っています。カットの下では、言語の機能とその作成の動機について簡単に説明します。

動機


動的型付けの最初の利点は、通常、開発/デバッグサイクルの短縮とプログラマー時間の節約と呼ばれます。世間の不快感を引き起こす恐れがあるので、私自身の経験ではこれを決して確認していないことを認めざるを得ません。 Pythonでニューラルネットワークのトレーニングスクリプトを少し修正した後は常に、Python、NumPy、PyTorchが読み込まれ、ファイルから大量のデータを読み取り、GPUに転送して処理を開始するまで待つ必要があります。その後、PyTorchがサイズテンソル（1、1 、m、n、3）の代わりに（1、m、n、3）を使用します。

多くの人が個人的な理由で動的に型付けされた言語を好むことを私はすぐに認めます。動的タイピングに対する傾向や敵意は、オリーブやトマトジュースに対する態度と同じ程度の現象である可能性さえあります。この問題の客観的な研究を試みた結果、決定的な結果は得られなかったようです。

同時に、TypeScriptの人気、Pythonでの型注釈の導入、RedditとHabréに関する激しい議論により、動的に型付けされた言語によるスクリプト言語の実際の識別は、ドグマではなく偶然であり、静的に型付けされたスクリプト言語は存在するあらゆる権利を持っていると私たちは考えています。

それで、クマにちなんで名付けられた猫にちなんで名付けられた言語がありました。

舌


言語全体の構文は Goに触発されました。構文構造の例は、プロジェクトページにあります。変数を宣言するときは、型推論による省略表記を使用できます。注目すべきは、ポインターの構文で行われたGoルールからの逸脱です。 Goの作成者は、文字通り次の例Cが構文の不必要な複雑化であることが判明し、代わりにPascalのような後置参照解除演算子を導入する方が合理的であると不満を述べました。それはまさにウムカがやったことです。翻訳者p^*p

umkaはバイトコードにコンパイルされ、スタック仮想マシンによって実行されます。すべての型チェックは、コンパイル段階で行われます。スタック上のデータには、タイプ情報が含まれなくなりました。トランスレータは、独自のAPIと小さな「ラッパー」（実行可能ファイル）を備えたダイナミックライブラリの形式で提供されます。ソースコードはC99で記述され、さまざまなプラットフォームに移植されます。x86-64プロセッサー（WindowsおよびLinux）用のビルドがリリースされました。

メモリ管理これまでのところ、参照カウンターに基づいて行われました。言語が何らかの関心を引き起こし、それを使用する試みが行われた場合、より高度なガベージコレクターを配置することは理にかなっています。この言語は、スタックに配置された従来の複合データ型（配列と構造）、およびヒープに配置された動的配列をサポートしています。古典的な配列や構造体も、明示的な呼び出しでヒープに配置できますnew()。

ポリモーフィズムは、Goスタイルのインターフェースによって提供されます。クラス、オブジェクト、継承の概念はありません。

マルチタスクこれは「ファイバー」の概念に基づいています。同じ仮想マシン内で実行され、お互いを明確に引き起こす単純化されたフローです。本質的に、これはコルーチンと同義です。これらのコルーチンを使用するロジックはGoの伝統から少し逸脱し、LuaとWrenに近づくため、コードサンプルを提供することは理にかなっています。

fn childFunc(parent: std.Fiber, buf: ^int) {
    for i := 0; i < 5; i++ {
        std.println("Child : i=" + std.itoa(i) + " buf=" + std.itoa(buf^))
        buf^ = i * 3
        fibercall(parent)
    }
}

fn parentFunc() {
    a := 0
    child := fiberspawn(childFunc, &a)    
    for i := 0; i < 10; i++ {
        std.println("Parent: i=" + std.itoa(i) + " buf=" + std.itoa(a))
        a = i * 7
        if fiberalive(child) {
            fibercall(child)
        }
    }    
    fiberfree(child)
}
