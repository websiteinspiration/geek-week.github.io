メビウスバイクは、タリン向けに開発された自転車とスクーターのレンタルサービスです（現在、地理的な拡張が計画されています）。

最初のリリースの仮説は、「ヨーロッパの市場で自転車のレンタルアプリケーションが必要になる」というものです。2019年12月に主要な仮説が改訂され、次のように聞こえました。「エンドユーザーとフランチャイズ店の利便性のために、自転車とスクーターのレンタルサービスを最大化できますか？」この質問に答えるためには、以下を実装する必要がありました。

  • アプリケーションの構造（内部-バックエンド、外部-設計とフロントエンドの両方）の最適化に関する作業を実行して、新しいタイプのトランスポートと将来の可能なスケーリングを導入する
  • 管理パネルでレンタル条件を調整できるようにする

リファクタリングとGraphQLへの切り替え


再設計の主な目的は次のとおりです。

  • アプリケーションとのやり取りの外観や一般的なロジックを損なうことなく、新しい機能、たとえば別のタイプのトランスポートを簡単に追加できる、より柔軟でスケーラブルなコンポーネントシステムを作成する
  • インターフェイスを可能な限り軽量化および簡略化する
  • アプリケーションの外観を「更新」します


ただし、再設計の目に見える側面であるアプリケーションインターフェイスの更新に加えて、RESTful APIからGraphQLへの移行は非常に重要な部分でした。

簡単に言うと、GraphQLは、クライアント（電話/アプリケーション）が特別なクエリを使用してサーバーからデータを受信する方法を記述する構文です。要求に応じて、サーバーはこれまたはそのデータを提供します。

Artyom Bukhtoyarov-DevOps /バックエンド開発者


GraphQLには3つの主要な特性があります。

  • クライアントが必要なデータを正確に指定できるようにします
  • 複数のソースからのデータの集約を容易にします
  • 型システムを使用してデータを記述します


GraphQLは、アプリケーションインターフェイスと外部サービスの間のアダプターレイヤーとして機能します。これにより、さまざまな要求の処理を背面に転送できます。つまり、新しいトランスポートプロトコルを接続する場合、背面で接続する必要があり、レイアウトはGraphQLから統一された形式ですべてのデータを受け取ります。したがって、新しいタイプのトランスポートを追加するときに追加の組版作業の必要性を減らし、アプリケーションで処理される要求の数を減らしました。

さらに、リファクタリングのプロセスでは、GraphQLを使用することの次の利点を強調しました。

  • 開発者にとって便利なドキュメント
  • バックエンドとフロントエンドの両方でWebSocketに適したソリューション
  • GraphQLにより、開発者はコード構造を簡単にナビゲートできます
  • RESTful APIと比較してより柔軟な開発により、新しいものをより速く簡単に追加できます（この場合、これは新しいタイプのトランスポートです）。


物理的にはサンクトペテルブルクにあったものの、中国からの最初の自転車用ロックが中国での地理的位置を決定したとき、それはおかしかったです。そして、このロックのソフトウェアは暗号化されており、設定を変更するための復号化方法を探す必要がありました。

当初、トランスポートのタイプが1つだけだったとき、自転車を接続するためのサーバーはメインのバックエンドと連動していました。次に、自転車ロック用のサーバーを独立したサーバーとして展開できる独立したサービスとして取り出しました。これにより、メインサーバーの負荷を軽減し、スケーリングの機会を増やすことができました。サービス間でメッセージを転送するには、RabbitMQを使用します。そして、彼がmqtt（私たちのスクーターが動作するプロトコル）のプラグインを持っていることは非常に便利でした。これにより、新しいトランスポートモードを簡単に追加できました。

アプリケーションでの支払い。最初はアプリケーション自体に銀行カードを追加することを計画していましたが、最終的に銀行ページへのリダイレクトオプションを決定し、カードを追加するプロセス全体がすでにそこで行われています。この方法は、実装がはるかに簡単であることがわかりました。

フランチャイズと管理


サービスの所有者には、アプリケーションの管理パネルに関する特定の要件がありました。このサービスはフランチャイズとして販売されることが計画されているため、各フランチャイジーは、旅行の費用、料金パラメーター、サブスクリプションなどを編集できる必要があります。

したがって、新しい公園の所有者を接続できるように管理パネルを再設計しました。さまざまな役割が追加されました-公園の所有者とネットワークの所有者。時間ベースの料金、サブスクリプション、および罰金を追加および編集するためのセクション全体を作成しました。各都市では、公園の所有者が独自の料金を設定し、変更することができます。また、ネットワーク管理者による変更のモデレートも可能にしました。管理パネルの構造を拡張して、新しいタイプのトランスポートを追加しました。

フロントエンド


React Nativeでの開発の主な機能の1つは、コンポーネントのパフォーマンスの追跡です。fpsの低下を防ぐには、次のような価値があります。
  • レンダラー（再描画）の数を監視します。コンポーネントは、新しいデータが実際にインターフェースの状態を変更する必要がある場合にのみ再描画する必要があります
  • アニメーション構成またはReanimatedパッケージでuseNativeDriveキーを使用して、標準のReact NativeパッケージのネイティブAnimatedアニメーションを使用します。
  • プロジェクトに追加されたパッケージまたはコンポーネントのJSコードを確認します。利用可能なすべてのパッケージが最適化されているわけではなく、React Nativeの機能を正しく使用しているとは限りません。また、React Nativeの開発では、プラットフォームの違いを検討する価値があります。iOSとAndroidで同じコードを使用すると、まったく異なる方法で機能する可能性があります

React Nativeの強みは、その汎用性と言えます。iOSとAndroidに2つの異なる開発チームを配置する必要はありません。また、多くの場合、ネイティブプラットフォームでの開発が困難な設計概念を実装できます。

React Nativeの弱点は、その完全なネイティブ性ではありません。JSスレッドが過負荷にならないように、常に確認する必要があります。オートリンカーが導入されるまでのバージョン0.60までのネイティブパッケージの更新とインストールの痛み。

バックエンド


ほとんどの場合、GraphQLへの切り替えにより、APIの開発が簡素化されました。それは開発者とクライアントの両方にとってより明確になりました。

GraphQLの利点：
  • APIを自動的に文書化することが可能で、非常に便利です
  • APIを使用したより柔軟な作業を可能にします。クライアントは、現時点で必要なデータのみを選択します
  • すぐに使えるWebソケットのサポート。リアルタイムデータを頻繁に更新するため、これは非常に重要でした。
  • 必要に応じてスカラー型を簡単に記述し、後で再利用できます


GraphQLの弱点：
  • ページネーションのタイプによって機能を実行するのはより困難です。追加のテクノロジーを適用する必要があります
  • デフォルトでは名前空間はありません。APIの分離は自分で処理する必要があります。同時に、非推奨フィールドのサポートがあり、開発者の生活をさらに簡素化します
  • クエリのネストレベルを監視する必要があります。入れ子の多いリクエストを作成して、回答を長時間待つことができます

私たちはゲーミフィケーション（旅行したキロメートルに素敵なボーナスを得る）、暗いテーマを導入し、電話連絡先に接続し、近くに旅行している友人に通知し、新しい交通手段を導入する予定です。しかし最も重要なのは、再設計を正当化する必要があることです。これは、ビジネスに利益をもたらし、ユーザーのアプリケーションでの作業を簡素化するためです。
