Hyperledger Fabricチェーンコードの新しいソフトウェアモデル



少し前に、fabric-contract-api-goの最初のリリース がリリースされました-RFC 0001チェーンコード用の新しいソフトウェアモデルの実装。それが何であるか、そしてそれをどのように使うか見てみましょう。

ここでは、ピアが開発モードで実行される単純なファブリックネットワークを使用してリポジトリを準備しました。リポジトリーの指示に従って、ネットワークを開始して戻ります（5分以内に完了します）。

ネットワークが稼働し、チェーンコードがインストールされたので、新しいモデルで動作するチェーンコードの内部を見てみましょう。

でSimpleContract.go、私たちは、新しいAPIとモジュールをインポートします。

github.com/hyperledger/fabric-contract-api-go/contractapi

次に、我々は使用して私たちの契約を記述SimpleContract構造その中に契約の構造が埋め込まれているが。

type SimpleContract struct {
	contractapi.Contract
}


コントラクトを ビルドして、コントラクトがContractInterfaceインターフェースを満たすようにすることが不可欠です。ここで予約をして、契約書と言う必要があります！=チェーンコード。チェーンコードは、無制限の数の契約のコンテナーです。次のリストに示すように、チェーンコードは契約をマップに保存します。


type ContractChaincode struct {
	DefaultContract       string
	contracts             map[string]contractChaincodeContract
	metadata              metadata.ContractChaincodeMetadata
	Info                  metadata.InfoMetadata
	TransactionSerializer serializer.TransactionSerializer

マップコントラクトは、リクエストをルーティングするためにInvokeによって内部的に使用されます。

func (cc *ContractChaincode) Invoke(stub shim.ChaincodeStubInterface) peer.Response {

	nsFcn, params := stub.GetFunctionAndParameters()

	li := strings.LastIndex(nsFcn, ":")

	var ns string
	var fn string

	if li == -1 {
		ns = cc.DefaultContract
		fn = nsFcn
	} else {
		ns = nsFcn[:li]
		fn = nsFcn[li+1:]
	}
       ...
       nsContract := cc.contracts[ns]
       ...
       successReturn, successIFace, errorReturn = nsContract.functions[fn].Call(ctx, transactionSchema, &cc.metadata.Components, serializer, params...)
       ...
       return shim.Success([]byte(successReturn))
}

SimpleContractに戻ります。すべてのメソッドには、TransactionContextInterfaceインターフェースを満たすctxパラメータが必要です。デフォルトでは、すべてのメソッドが標準のTransactionContextを取得しますが、ほとんどの場合これで十分です。 このコンテキストにより、たとえば次のようにClientIdentityを操作できます。



func (sc *SimpleContract) Whois(ctx contractapi.TransactionContextInterface) (string, error) {
	return ctx.GetClientIdentity().GetID()
}

または、使い慣れたスタブ（shim.ChaincodeStubInterface）を取得して、元帳と対話するための通常のアクションをすべて実行します。

func (sc *SimpleContract) Write(ctx contractapi.TransactionContextInterface, key string, value []byte) error {
	return ctx.GetStub().PutState(key, value)
}

だが！デモリポジトリのコードでは、メソッドで完全に異なるコンテキストを確認できます。

func (sc *SimpleContract) Create(ctx CustomTransactionContextInterface, key string, value string) error {
	existing := ctx.GetData()

	if existing != nil {
		return fmt.Errorf("Cannot create world state pair with key %s. Already exists", key)
	}

	err := ctx.GetStub().PutState(key, []byte(value))

	if err != nil {
		return errors.New("Unable to interact with world state")
	}

	return nil
}

これはカスタムコンテキストです。それは非常に簡単に作成されます。リポジトリのcontext.goに注意してください

。1. contractapi.TransactionContextInterfaceと互換性のあるインターフェースを宣言します。

type CustomTransactionContextInterface interface {
	contractapi.TransactionContextInterface
	GetData() []byte
	SetData([]byte)
}

2. contractapi.TransactionContextを埋め込む構造

type CustomTransactionContext struct {
	contractapi.TransactionContext
	data []byte
}

3.宣言されたメソッドを実装します

// GetData return set data
func (ctc *CustomTransactionContext) GetData() []byte {
	return ctc.data
}

// SetData provide a value for data
func (ctc *CustomTransactionContext) SetData(data []byte) {
	ctc.data = data
}

これで、コントラクトを初期化するときに、この構造をハンドラーとして渡すだけです。

simpleContract := new(SimpleContract)

simpleContract.TransactionContextHandler = new(CustomTransactionContext)

そして今の代わりに、私たちの契約のすべてのメソッドCTX contractapi.TransactionContextInterfaceは受け入れCTX CustomTransactionContextInterfaceを。トランザクションフック

を通じて状態をプッシュするには、カスタムコンテキストが必要です。トランザクションフックは、contractメソッドを呼び出す前または後に起動するミドルウェアの美しい名前です。 メソッドを呼び出す前に、トランザクションの元帳からトランザクションで渡されたキーの値を取得するフックの例：SimpleContract.go




func GetWorldState(ctx CustomTransactionContextInterface) error {
	_, params := ctx.GetStub().GetFunctionAndParameters()

	if len(params) < 1 {
		return errors.New("Missing key for world state")
	}

	existing, err := ctx.GetStub().GetState(params[0])

	if err != nil {
		return errors.New("Unable to interact with world state")
	}

	ctx.SetData(existing)

	return nil
}

main.go
simpleContract.BeforeTransaction = GetWorldState

これで、メソッドで要求されたキーの値をもう少し簡潔に取得できます：

SimpleContract.go
func (sc *SimpleContract) Read(ctx CustomTransactionContextInterface, key string) (string, error) {
	existing := ctx.GetData()

	if existing == nil {
		return "", fmt.Errorf("Cannot read world state pair with key %s. Does not exist", key)
	}

	return string(existing), nil
}

メソッド呼び出し後のフックは、コンテキストに加えて空のインターフェースを受け入れることを除いて、ほとんど同じです（必要な理由は後でわかります）：

YetAnotherContract.go
func After(ctx contractapi.TransactionContextInterface, beforeValue interface{}) error {
	fmt.Println(ctx.GetStub().GetTxID())
	fmt.Println("beforeValue", beforeValue)
	return nil
}

このフックは、トランザクションIDと、メソッドがフックの前に返した値を表示します。このポストフックを確認するには、CLIコンテナーに移動して、contractメソッドを呼び出します。 コードが実行されているターミナルに切り替えます。出力は次のようになります。

docker exec -it cli sh
peer chaincode query -n mycc -c '{"Args":["YetAnotherContract:SayHi"]}' -C myc

beforeValueこんにちは＃前のメソッドの値

存在しない関数名でリクエストを処理したい場合はどうなりますか？このため、どの契約にもUnknownTransactionフィールドがあります：

unknown_handler.go
func UnknownTransactionHandler(ctx CustomTransactionContextInterface) error {
	fcn, args := ctx.GetStub().GetFunctionAndParameters()
	return fmt.Errorf("Invalid function %s passed with args %v", fcn, args)
}

main.go
simpleContract.UnknownTransaction = UnknownTransactionHandler

これは、CLIで確認することもできます 。

docker exec -it cli sh
peer chaincode query -n mycc -c '{"Args":["BadRequest", "BadKey"]}' -C myc

チェーンコードをピアで開始するには、すべてのコントラクトをチェーンコードに転送する前に、以前のようにStart（）メソッドを呼び出す必要があります：

main.go
cc, err := contractapi.NewChaincode(simpleContract, yetAnotherContract)

	if err != nil {
		panic(err.Error())
	}

	if err := cc.Start(); err != nil {
		panic(err.Error())
	}
