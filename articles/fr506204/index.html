Type de données abstrait. Partie 2: Gestion des données

Dans la première partie, j'ai décrit comment être guidé dans la détermination du type de données (ci-après dénommé TD). Dans cette partie, je veux vous dire quel est le type de données abstrait (ci-après dénommé ADT), comment il aide à résoudre des tâches logiquement complexes (ce que l'on entend par tâches logiquement complexes, voir le premier article) et quels avantages nous obtenons.


Au lieu d'une préface:

Un type de données abstrait est une collection qui inclut des données et les opérations qui y sont effectuées (S. McConnell, «Code parfait», chapitre 6.1. Principes fondamentaux des classes: types de données abstraits). Dans cette série d'articles, nous considérons une telle vue de l'ADT, étendue par le type de données.

Torvalds Linus - «Les mauvais programmeurs s'inquiètent du code. Les bons programmeurs se soucient des structures de données et de leurs relations. »
Le calcul est l'acquisition de nouvelles connaissances à partir des données d'entrée.


ADT - est considéré comme une classe contenant des structures de données avec lesquelles nous travaillerons plus tard dans cette classe. Les données se réfèrent à des ensembles de points d'accès qui ont été examinés dans la première partie .
Structures ADT pour le stockage de données - moyens de stockage de données, c'est-à-dire il peut s'agir d'une pile, et une file d'attente et toute structure de données classique / non classique qui est pratique pour travailler avec eux, dépend du contexte de la tâche et de votre imagination.


Pourquoi cette approche vous permet d'organiser une logique complexe:

  1. Nous travaillons uniquement avec des données (TD) sans hésitation à les obtenir. Nous attribuons la logique d'acquisition des données aux «fournisseurs de données» (voir ci-dessous). En substance, il s'agit d'une lutte avec la complexité - nous pensons séparément à la méthode d'obtention de données qui ont toujours un certain type et réfléchissons séparément à la façon de travailler avec des données de ce type.
  2. Nous pouvons organiser l'ADT uniquement avec les ensembles de données dont nous avons besoin, en décomposant l'ADT si nécessaire (également une technique pour combattre la complexité).

Les exigences pour l'ADT sont similaires à celles de la TD, avec un petit ajout:

  1. ADT ne devrait pouvoir travailler qu'avec ses données.
  2. Les ADT ne doivent pas dépendre de l'état externe de l'application.
  3. Pour recevoir des données, utilisez des «fournisseurs de données» et transmettez leurs résultats à l'ADT, n'utilisez pas les méthodes ADT pour les obtenir.

Fournisseur de données - cela implique une classe qui reçoit les données dont nous avons besoin des sources disponibles, de diverses manières, et renvoie des points d'accès ou des ensembles de points d'accès. Comment organiser les fournisseurs de données est un sujet distinct, selon le contexte de la tâche et l'imagination du développeur du système.

Cela dépend bien sûr du contexte de la tâche et de la volonté du développeur du système.
  2. Prévisibilité du résultat - le résultat sera toujours attendu, car en observant ces exigences, nous obtenons un analogue des fonctions pures du FP.

D'accord, il y a déjà beaucoup de paramètres et certains paramètres peuvent affecter les paramètres d'autres entités. Et puis nous rappelons qu'un type de réservoir peut avoir différents types de moteur, pour lesquels seul un certain type de carburant convient. Que faire, réécrire tout le code?

La solution complète utilisant l'approche décrite doit être donnée pendant longtemps, je vais donc essayer de décrire brièvement le concept lui-même:

  2. Dans ADT, nous gérons les données. Par exemple, nous chargeons dans l'ATD le terrain sur lequel la bataille a lieu et des chars de différents types avec un ensemble de données différent. Ensuite, nous gérons simplement les données en appelant certaines méthodes, par exemple, déplacer le char 1 vers le point B dans notre zone, ou tirer du char 2 au point C.Par conséquent, connaissant la vitesse du char, la vitesse du projectile, le type de dégâts du projectile, etc. on peut faire des calculs dans l'ATD où, au moment du coup du projectile lancé par le tank 2, le tank 1 sera localisé et si la coquille du tank 2 pénètre dans le tank 1 quels dommages lui seront causés.

Je ne prétends pas que cette approche est la meilleure solution, j'ai juste décidé de partager mes pensées, peut-être qu'ils aideront quelqu'un à résoudre leurs problèmes.

Total:

  1. ADT stocke les données agrégées sous une forme qui nous convient
  2. ADT ne devrait pouvoir travailler qu'avec ses données.
  3. ADT ne doit pas dépendre de l'état externe de l'application.
  4. N'utilisez pas de méthodes ADT pour obtenir des données, pour cela, utilisez des «fournisseurs de données» qui renvoient des points d'accès ou des ensembles de points d'accès. Ensuite, nous transférons les données reçues à l'ADT pour les calculs à leur sujet (nous attachons / appliquons l'action à l'AT).
