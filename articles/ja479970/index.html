Get best of the week

千のUIバグ、または開発者がUIの一般的なエラーを回避する方法

新機能のテストは、非常に独創的で興味深いプロセスのようです。しかし、インターフェイスのエラーが機能ごとに繰り返され、ほとんどの時間が小さなインターフェイスの問題の把握に費やされている場合はどうでしょうか。 



Badooでの4年間で、私が遭遇した1,000を超えるバグのうち、約20％がUIとUXに関連していました。それらの3分の1は製品規模では重要ではありませんが、それでもユーザーの忠誠度に直接影響するため、処理リソースを必要とします。このようなバグは手動でしか検出できません。さらに、特定の条件の特定のデバイスでのみ検出されることがよくあります。

新しい機能を設計する段階でもこれらのバグを防ぎ、テスト後のインターフェース処理を回避することは可能ですか？私の答えはイエスです！ 

この記事では、私の経験からの例を使用して、テストプロセスのルーチンを少なくし、同じバグの開始を停止する方法を説明し、Androidプラットフォームでのモバイルアプリケーションのインターフェイスの開発における最も一般的な間違いを示し、それらが最も頻繁に発生する場所について説明します。記事はハイゼンバグ会議での私の報告に基づいて書かれました、ビデオはここで見ることができます。

検索を好む人はいません。そのようなインターフェースのエラーを修正するのもそうです。



そして、ユーザーが目を引く小さなインターフェースの問題でいっぱいのアプリケーションを操作するのは、いらいらします。多くの場合、これらの問題は機能ごとに繰り返され、コンポーネントとともに新製品に継承されます。

Badooの私たちもこの状況に直面しました。開発プロセスは、これを回避するように設計されています。ただし、インターフェイスの問題は引き続き定期的に発生しました。



私たちのアイデアは、多くの段階を経て、新しい機能になります。さらに、インターフェースの問題の検索と防止で最も重要なことの1つは、品質管理の段階でテストを行うことではなく、見直すことです。この段階で、製品マネージャと設計者は、機能の新しいプロトタイプまたはアプリケーション全体を見て、それを取得したいかどうか、およびすべてが好きかどうかを評価します。これは、インターフェースの問題を見つけるのに役立つ最も便利な方法の1つです。必ずレビューに含めてください。レビューにより、プロセスのすべての参加者の時間を大幅に節約できます。

ただし、これだけでは不十分です。レビューはありますが、マイナーなインターフェースエラーや不便さの説明に費やした私の時間の約20％です。これらはどのような問題ですか？ 

UI / UXバグの最も一般的な原因 


Android開発チームと私が過去4年間で最も頻繁に発見した問題を分析した後、問題が発生した4つの主な理由を特定することができました



。

1.真空中の球形モデル


私たちが発見したエラーの半分以上は、デザイナーが作成したレイアウトが、プロトタイプを作成する過程で開発者が持っていた膨大な数の質問に答えなかった状況が原因です。 

私の経験から例を見てみましょう。左の写真はデザイナーのレイアウト、右の写真はプロトタイプの最初の反復です。



プロトタイプには非常に「美しい」データがあり、ノートブック内のユーザーの連絡先の名前が長いか写真がない場合の対処方法は明確ではありません。すべての場面でモックアップをレンダリングすることは時間のかかる作業です。全体像を完成させるには、すべてのボトルネックについて話し合う必要があります。

2.デザインの重要性を過小評価している


2番目に、開発者がデザインを無視して、レイアウトに関係なく独自の方法で何かを行った状況があります。 

人生の別の例。タスクは、ユーザーの生年月日入力画面を更新することでした。左側はデザイン、右側は開発者のプロトタイプです。 



日付選択コンポーネントはすでに他のいくつかの機能用に作成されており、開発者はそれを使用しましたが、デザイナーは特別に完全に新しいコンポーネントを描画して、ユーザーがアプリケーションに簡単に登録できるようにしました。つまり、生年月日を入力します。

3.ドキュメントのギャップ


人気の3番目の場所は、製品マネージャーへのドキュメントまたは質問のギャップです。たとえば、さまざまなインターフェイス要素の目的について、新規ユーザー向けのヒントを含む機能をテストしました。左の図はドキュメントのレイアウトで、右の図はインターネット接続が切断された場合に何が起こったかです。



接続に失敗した場合、いわゆるゼロケース、つまり接続がないことをユーザーに通知する画面が表示されます。彼のことはしばしば忘れられ、この場合、ヒントが引き続き表示されましたが、この状況は新機能のドキュメントには反映されていませんでした。

4. Android OSとファームウェアの機能


インターフェイスのエラーの最もまれな（しかし同時に定期的な）原因は、OSのアップデートまたは製造元からのファームウェアです。 

たとえば、Android 9では、強打の登場後、ユーザーの写真が次のように見えるようになったので、シェルキャラクターのゴーストのようになりまし



た。問題はUXだけではありませんでした。また、アプリ内通知がこの「フリンジ」の下に表示され始める場合もありました。

UX / UIエラーが許可される場合


これらのマイナーなインターフェースエラーに集中する必要がない場合はありますか？もちろん、はい。MVPを実行する場合、実行可能な製品は最小限であり、ユーザーのアイデアが全体として気に入っているかどうかを確認することが目標です。 

同意します。この場合、ごくわずかなバグを排除するのに時間を費やすのは意味がありません。これが報われるかどうかはわかりません。ただし、新機能が100％ではなく80％で作成されたからといって、ユーザーが新機能を気に入らなかったことを誰も保証できません。この場合、エラーの重要度は製品マネージャーによって決定されます。主なことは、ユーザーがプロジェクトを気に入っていて、MVPからさらに開発の段階に移行したことが明確になっている次の段階で、これらすべての小さな問題を忘れずに解消することです。 

これをどうするか 


インターフェイスエラーの上記の原因を取り除く方法、使用する方法は？Badooで使用する基本的な方法とコツについて見ていきましょう。最も時間がかかるものから始めましょう。

1.独自の設計システムを作成する


Badooでは、単一のCosmos設計システムを作成しました。これにより、設計者と開発者の相互作用が簡素化され、開発プロセスが大幅に加速しました。 



簡単に言えば、デザインシステムは、インターフェイスの1つまたは別のコンポーネントで何が発生する可能性があるか、どのような状態にあるか、テキストの長さに応じてどのように見えるかなど、すべての質問に回答します。上の図では、これはButtonコンポーネントの例として示されています。設計システムがある場合、すべての状況で新機能の詳細なレイアウトを描く必要はありません。 

設計システム開発は、さまざまなプラットフォーム上に多くの製品と複雑なインターフェースを備えた大企業の選択です。たとえば、Googleはマテリアルデザインを使用しています。あなたはそのようなシステムの開発にお金を費やす必要がありますが、将来的にはそれが多くの問題を回避するのを助けるでしょう。 

設計システムを開発する時間がない場合、またはそのような複雑な方法の使用を必要としない小さなアプリケーションがある場合はどうなりますか？コンポーネントまたは簡単なドキュメントで小さなライブラリを書くことができます。つまり、社内に存在する原則を簡単な内部ガイドラインまたはヘルプで説明します。 

コスモスの詳細については、同僚のクリスティアーノラストレッリの記事シリーズと記事の最後のリンクを参照してください。

2.視覚テストツールを使用する


市場に出回っているソリューション の数と同様に、視覚テストツールの人気は高まっています。CodeFest で私の同僚のKarl Crawfordのレポートで、当社でのVRTテストの使用について聞くことができます。ただし、写真を比較するだけでなく、ユーザースクリプトも保存する必要があったため、ここで停止しませんでした。そこで私たちは先に進み、クロスプラットフォームツールのLiveShotsを作成しました。 



LiveShotsにはさらに多くの機能があります。バージョン間だけでなく、iOSプラットフォームとAndroidプラットフォームの間でも、アプリケーションのインターフェイスを比較できます。自動テストに基づいて機能し、さまざまな言語をサポートするユーザースクリプトを収集するので、インターフェースへの最小限の変更でも気付かれることはありません。LiveShotsについて詳しくは、同僚のSasha Bayandinのレポートをご覧ください。

3.優れたテストベンチを構築する

 
より簡単に実装できるツールとメソッドに移ります。うまく組み立てられたテストベンチは、モバイルデバイスのさまざまなメーカーの断片化とファームウェア機能の問題を解決するのに役立ちます。品質テストと断片化に関連する問題の発見に必要なモバイルデバイスはいくつあると思いますか？さまざまなデバイスでのテストに多くの時間を費やすことなく、同時にユーザーの最も一般的な問題を見つけるには、5〜6台のデバイス（たとえば、Androidプラットフォーム）で十分です。Habréに関する私の記事で、テストベンチ用のデバイスを選択する方法の詳細を読むことができます。

4.補助ツールを使用する


インターフェイスの問題をテストおよび発見するための興味深いヘルパーアプリケーションは数多くあります。OS開発者は定期的にそのようなツールをデバイス設定セクションに直接追加します（開発者オプションを参照）。私の意見では、最も役立つのは、タップのクリックの表示とレイアウト境界の境界の表示です。 ポータブルレイアウトインスペクターと同様に、フォントサイズや色などのインターフェイス要素に関する詳細情報を表示



する開発者アシスタントと、デバイスモデル、画面解像度に関する詳細情報を含むスクリーンショットを撮ることができるデザイナーツールをお勧めします。バグレポートまたはどこかに保存します。
 

5.より頻繁に会う

 
それは明らかな方法のようです。しかし、会議で何を話し合うのですか？繰り返される間違いについて-これは、プロセスの参加者が責任とタスクについて異なる見解を持っていることを示しているためです。何度か発生した問題には、関係するすべての従業員による必須の分析が必要です。 

プレゼンテーションを行った会議で、インターフェイスの小さな問題について責任を負いたくない場合、つまりプロセスのすべての参加者がお互いをポイントする場合の対処方法について質問を受けました。テスト担当者は、インターフェイスの確認がタスクであると述べていますデザイナー、開発者-プロトタイプを彼に見せたとき、製品マネージャーはすべてに満足していて、デザイナーは最終的に製品が彼が作成したレイアウトとそれほど異なる理由を理解していません。最善の解決策は、すべての誤解を解決することです。つまり、一緒になってプロセスの改善に取り組み、すべての開発参加者の責任の領域について話し合い、明確にし、同じ間違いを見つけるのに時間を浪費しないことです。

6.ドッグフード


もうたくさん話されているもう一つの簡単な方法は、ドッグフード、つまり社内で自分の製品を使うことです。Facebookなどの大企業の代表はこれについて話すのが大好きです。もちろん、20,000人の従業員が自分の製品を自分で見るとき、テスターの軍隊は実際には必要ありません。実際、最も重要なことは、dogfoodは自分の製品をよりよく理解し、ユーザーのニーズを理解するのに役立つということです。したがって、過小評価しないでください。 

7.チェックリストを書く


分析した問題に基づいて、新しい機能やまったく新しいアプリケーションをすばやく確認し、モバイルアプリケーションのメモリボトルネックを回復する時間を節約するためのチェックリストを作成しました。これは、インターフェイスエラーや使いやすさの問題が最も一般的な場所です。このチェックリストは、プロジェクトで発生する最も一般的なバグに基づいて詳細を補足する場合に特に役立ちます。分解してみましょう。

私の実践からの非常に簡単な例があります：


  • すべてのテキストを確認してください。


ボタン名は長すぎてはいけません。このルールは、どの言語にも当てはまります。


  • 進行状況バーがあるかどうかを確認します。

 
Googleのガイドラインでさえ、画面の読み込みが3秒を超える場合は、たとえばアニメーションを表示するなどして、ダウンロードが行われていることをユーザーに通知する必要があると述べています。同様に、他の「重い」要素-ビデオと写真。 


  • データがない場合（ゼロの場合）に何が表示されるかを確認します。


データがない場合、たとえば、新しいメッセージを受信して​​いない新しいユーザーの場合、白い空白のページではなく、何が起こっているのか、およびアプリケーションの操作を続行する方法を説明するテキストが表示されることが重要です。 


  • ボタンとそのステータスを確認します。
 


ボタンが押されたことに応答することが重要であり、そうである場合、ボタンが非アクティブである理由は明らかです。 


  • インデントと配置をレイアウトと比較します。


あらゆる種類の断片化を提供することは不可能であるため、これは非常に辛い瞬間ですが、テストベンチはこれで多くのことを助けます。


  • 画像がない場合にプレースホルダーがあるかどうかを確認します。


繰り返しになりますが、画像や情報がない場合にユーザーに白い画面を表示しないでください。ただし、画像またはセクションがないことを説明してください。


  • 古い機能と新しい機能の相互作用を確認します。


ここでは、すべてが例に似ていますが、上で引用したユーザーへのヒントがあります。


  • オフライン作業;


この場合、ダウンロードインジケータを表示するだけでは不十分なので、切断された接続についてユーザーに通知することが不可欠です。


  • 新しいインターフェイスが小さなデバイスでどのように動作するかを確認します。


コメントはありません。写真を参照してください。


  • 画像が小型デバイス用に最適化されているかどうかを確認します。

 
小さな画面には常に十分なスペースがあるとは限らないため、最適化されていない画像が原因でレイアウト全体が表示される可能性があります。



  • OSの異なるバージョンとの相互作用を確認します。
 


OSの更新後にアプリケーションで発生した問題のリストを保持して、これらの変更によって同じ間違いを繰り返し行わないようにするとよいでしょう。

  • 異なるバージョンのファームウェアとの相互作用を確認します。


同様に、さまざまなデバイスのアプリケーションで発生した問題のリストを保持し、新しい機能が同様の条件でどのように動作するかを確認するとよいでしょう。 

  • アニメーションを確認します（特に小型で弱いデバイスの場合）。


画面の解像度が小さい弱いデバイスの場合は、アニメーションを破棄して静止画像に置き換えることをお勧めします。

開発段階でこのチェックリストを念頭に置いておくとよいでしょう。これは、モバイルアプリケーションインターフェイスを設計するときに開発者がすべての微妙な点を考慮に入れ、品質管理段階でテスターの時間を節約するのに役立ちます。
