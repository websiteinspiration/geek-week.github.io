自己文書化コードは（通常）ナンセンスです

みなさん、こんにちは！

今日の翻訳された出版物を見越して、このテキストは、最近のディスカッション資料「コード内のコメントで熱心に停止する」へのフォローアップとして意図されていることにすぐに注意します。そこで展開された議論と2019年7月19日の時点での189件のコメントに感銘を受け、Mediumポータル（Christopher Lane）から別の著者に発言権を与えることにしました。原作では、この記事は前の記事（5月16日と6月16日）より1か月遅れて発行されましたが、拍手のほぼ半分を集めました（翻訳の発行時の1.5 Kに対して706）。Habrで何が起こるか見てみましょう...



写真は著者のサイトrawpixels.comから取得されますペクセル

私はCindy Cheungによる技術文書に関する優れた記事と、開発者が独自のコードをよりよく説明する理由を注意深く読み、それに完全に同意する必要があります。

私はあなたのこのITで長い間地獄をやっており、私の経験では、開発者やエンジニアが単純に抵抗できないほど、私たちのビジネスには自己欺瞞があると示唆しています。
私のコードは自己文書化されています-誤った開発
理論的には、優れたエンジニアのコードは明確で読みやすいため、ドキュメントは必要ありません。

これはナンセンスです...原則として。

「自己文書化コード」が意味をなさないのはなぜですか？


Hemingwayが散文を書いたのと 同じくらいかっこいいコードを書いたとしましょう。おそらく、あなたの超重複コードは（別の開発者にとって）クリーンで明確です。結局のところ、このコードは技術者のために技術者によって書かれたものであり、コードがどれほど簡潔で簡潔に見えても、それはつぶやく可能性があるプログラマー以外の人が読むことを意図していません。意味？！」

なぜ自己文書化コードはすべてナンセンスだと思いますか？詳しく説明します。

理由1：自己文書化されていないプログラミングのトリックがたくさんあります

開発者を含むほとんどの人々が自動車ではないからといって。はい、おそらく私はあなたのコードを調べるつもりです。私はあなたのメソッドとクラスの名前を理解しますし、各メソッドであなたが正確に何をしているかも理解します。

しかし、コードはマシン用に書かれています。彼らは私たちよりもそれをどうするべきかをよく理解しており、それを説明するためにプログラミング言語を持っています。人々と一緒に、人間があなたのソフトウェアが何をしているかを理解できるように、より人間的な言語でコミュニケーションする必要があります。

「コードを読んで何が起こっているかを確認する」とドキュメントの間には非常に大きな違いがあります。コード内で行われるすべての詳細をコードに書き留めることは可能ですが、この場合は「自己文書化」と呼ぶことができますか？不可能だと誰もが理解していると思います。

次のC＃の単純なblobについて考えます。ファイルを読み取り、その内容を取得してから、StreamReaderを使用してファイルのエンコードを取得します。

var fileContents = “”;
Encoding fileEncoding; using (var reader = new StreamReader(filePath, Encoding.Default,  true))
 {
   reader.Peek(); 
   fileEncoding = reader.CurrentEncoding;
   fileContents = reader.ReadToEnd();
 }

StreamReaderで可能性のあるあいまいさを無視すると、このコードは非常に単純になりますよね。それから...慈悲を持っていますが、この行で何が行われていますか？

reader.Peek();

ファイルのエンコーディングを取得するには、リーダーがこのアクションを実行する必要があることがわかります。セルフドキュメンテーションはどこにありますか？ただし、コードをよりわかりやすくするには、10秒ほどで十分です。

reader.Peek(); //     ,    .

これはほんの一例にすぎません。コードがより複雑になると、そのような細かい詳細がいたるところに現れ始め、かつてはクリーンだったコードが次第に散らかっていきます。コードを読んでいる人がコードで何が起こっているのかを把握するのはますます困難になります。

理由2：

BASHまたはBATファイルを記述したことがある場合、難易度は本質的に自己文書化されていないため、そのようなファイルに記述されているアクションが順次実行されることがわかります。次々とタスク。ファイルは、最初の行から最後の行まで読み取られる短いストーリーに似ています。

ただし、プログラム、特にWebアプリケーションを作成している場合、すべてのWebサービスの初期ロードと構成のコードを除いて、このような一貫した履歴はありません。

最新のWebアプリケーションを構成するクラス自体は、順次実行されません。本質的には、クライアントとWebアプリケーションとの対話中に正確に呼び出されるWebまたはAPIコントローラーのコレクションです。各WebまたはAPIコントローラーは、新しいプロセスを分岐するためのスレッドを提供でき、メッセージが他のサービスに送信され、リスナーがWeb結果にそれらを使用できるように応答が期待されます。 「プロット」形式で説明することはほぼ不可能ではありません。あなたのすべての「自己文書化コード」から、初心者または非プログラマーは「どうやら、ここで何が起こっているのか理解しているようです」としか出ません。繰り返しになりますが、そのような「ドキュメント」を信頼する人は誰もいないでしょう。

アプリケーションが複雑になるほど、そのクラス、メソッド、およびフレームワーク全体が順次モードで動作しない可能性が高くなります。そのようなアプリケーションに遭遇した人は誰でも、コードからその中で何が起こっているのかを簡単に理解できると仮定すると、ますます滑りやすいトラックに入ります。

理由3：プログラミング言語の構文は、原則として読み取り可能とは言えないため

、APIエンドポイントを呼び出すこのjquery関数を見てください。

ふew ...

いいえ、構文に問題があるとは言いたくありません。 jqueryの呼び出しには、すべてが完全に受け入れられます。しかし、私は、初心者プログラマーまたは非プログラマーの目を通してそれを見ると、このリストはバイトコードほど明確ではないかもしれないことを強調します。それにはもう意味がありません。

構文上、プログラミング言語は、言語自体によって設定された制限を積極的に使用できるように設計されており、コードをコンパクトに保ち、必要に応じて簡単に変更できる便利なショートカットもあります。プログラミング言語は、誰もが理解できる独自の信頼できるコミュニケーション手段として考えられていません。言語自体、その構文とショートカットを知っている専門家向けに設計されています。

他のすべての人にとって、プログラミング言語は理解できません。

何をすべきか？

いくつかのトリックがあり、それを使用すると、専門家以外のユーザーがコードを理解するのに役立ちます。

ステージ1：ドキュメントを書いてみる


冒涜的ですね。ドキュメントを書く？！おかしなことを思いつくことはできませんでした！

真剣に、誰もあなたに戦争と平和を書くことを要求することさえありません。ただし、主要なアクション、検証、エラー処理をシンプルで一貫したスタイルで技術ドキュメントに記述してください。

  • クライアントがAPIエンドポイントを呼び出す /someurl/object/{id}
  • APIコントローラは{id}（type int）を使用してデータベース内のオブジェクトを検索します。
  • オブジェクトがを返す場合、nullAPIコントローラはクライアントにHTTP応答404（ファイルが見つかりません）を返します。APIコントローラはこれを警告として記録します。
  • 返されたオブジェクトが-のNOT null場合、APIコントローラはこのオブジェクトをJSON形式に変換し、HTTP応答200（OK）で呼び出し元に返します。

難しいことではありませんが、そのようなドキュメントを書くことで、誰かの生活を楽にします。自分勝手なモチベーションに近づいている場合は、それについて考えてください。常にヘルプや説明を求められている場合は、ドキュメントを参照するだけで、同じことを何度も何度も説明しないでください。

ステージ3：クラスとアクティビティに単一の言語（ユビキタス言語）で名前を付けます


ご存じのとおり、統一言語はDDD（サブジェクト指向の設計）の概念であり、チームとユーザーはすべてのクラスとその相互作用を記述する言語を開発する必要があります。このような言語は素人でも理解できるので、顧客、テスター、インストラクター、ビジネス担当者は、プログラムについて理解し、この分野のユーザーの問題をどのように解決するかを理解することができます。

統一言語が合意されたら、すべてのクラス、それらのメソッド、イベント、およびその他すべてが統一言語にできるだけ近い名前が付けられていることを確認する必要があります。

/// <summary>

これはほんの一部のコードですが、上記のコードは、ここで何が起こっているかを簡単で理解しやすい言語で記述しています。

ステージ4：コメントを書くだけ


上記のすべてがあなたにとっても重すぎると思われる場合は、コードに有益なコメントを付けてください。あなたは今それらを必要としないかもしれませんが（あなたは今あなたの頭でコードに没頭しています、すべてはあなたにすでに明らかです）、将来的にはそれらはあなたにとって非常に有用になるでしょう。

コメント付きの数行の正確な間隔の行、クラスまたはメソッドのコメントで、コードをよりわかりやすくすることができます。各行にコメントすることはお勧めしません（これはコードを複雑にするだけです）。コードの最も難しいセクションをコメントアウトして、コードを通過する人が誰でもこのコードが彼を導く場所を理解できるようにします。
