起動する準備はできましたか？


したがって、コードが作成され、パッケージ化され、どこかに展開されます。不変のマシンアーティファクト（EC2など）としてのコードのデプロイメントを無視し、コンテナーに焦点を当てることにしました。どうして？

ソースコードで不変のAMIを焼き付けるため、どこにでもコピーして実行するのは非常に困難です。これは優れたテンプレートですが、どうしても必要な場合に限ります。したがって、本当にそうする必要があるかどうかを検討することをお勧めします。そうでない場合は、マイクロサービスをコンテナまたはサーバーレス関数に再編成してみてください。

コンテナーが不可能な場合は、ソフトウェアをモノリシックアプリケーションとして作成、パッケージ化、および展開することにしたので、不変のAMIテンプレートを検討してください。独自の非クラウドアプリケーションまたは商用ソフトウェアをそのまま（そのまま）実行する必要がある場合も、同じことを行います。ただし、これはこの記事の主要なトピックではありません。

きちんと梱包されたコンテナがある場合、どのように起動しますか？

本当に働くコンテナ


あなたができる最も簡単なことは、単にdocker run myImageコマンドを実行してそれを終了することです。ただし、次の場合にはこのようなソリューションは機能しないため、これは良いアイデアではありません。

  • このコンテナは突然「死ぬ」でしょう。
  • 負荷を処理するには、複数のコンテナが必要です
  • ダウンタイムなしでデプロイメントを実装する必要があります。
  • マイクロサービスを完全に制御したい
  • CI / CDコンベヤを使用して、製品を顧客
    などにすばやく配送したい場合...

つまり、真の分散型エンタープライズレベルアプリケーションを作成する必要がある場合はどうなるのでしょうか。明らかに、これはプリミティブなdocker runコマンドが単純に処理できないほど複雑なプロセスです。

docker-composeコマンドテクノロジは、非常に類似した一連の問題に悩まされていることに注意してください。単一のコマンドで多くのサービスを実行できるDocker-composeは、本番環境でのデプロイを目的としていません。その目的は、ローカルプロトタイピング、迅速な機能テスト、または非常に小規模な（個人の家など）展開です。つまり、ビジネス収益を生み出さないユーザーワークロードのためのツールです。

だから、コンテナオーケストレーションは救助に急ぎます！

コンテナーオーケストレーションメソッドの概要


人生の他のすべてと同様に、問題を解決する方法は複数あります。最初で最も明白なのはKubernetesです。Google内部プロジェクトから生まれたKubernetesは、コンテナオーケストレーションの事実上の標準です。



また、アプリケーションを次の場所で実行する場合、これが唯一のオプションです。

  • プライベートデータセンター;
  • Google Cloud
  • Microsoft Azure
  • その他のパブリッククラウド。

ただし、AWSで作業する場合は、もう1つのオプション-ECSがあります。厳密に言えば、これは完全に真実ではありませんが、HashicorpのNomad（これらはTerraformを提供したのと同じ人々です）とDockerのDocker Swarmがあります。問題は、最小限の実装でニッチなプラットフォームがたくさんあるので、急速なキャリアの成長のためにそれらを無視することです。
とにかく、AWS Elastic Container Services（ECS）に戻ります。これは完全に管理されたコンテナーオーケストレーションサービスであり、簡単に開始でき、AWSエコシステムの他の部分と緊密に統合されています。彼はほんのいくつかのことを行いますが、彼はそれらをうまくやっています。要するに、これはKubernetesとは正反対であり、ECSがマクドナルドに十分対応できれば、おそらくあなたにも十分対応できるでしょう。

しかし、純粋に即時のキャリアの成長という点では、Kubernetesが最良の選択であることは間違いありません。 AWSで作業している企業の99％もECSを完全に満足させると私は確信しています。

だから今、あなたは選択をすることができます。あなたがこの分野の完全な初心者であれば、Kubernetesを試してみることができます。あなたの旅であなたをサポートできる志を同じくするDevOpsエンジニアのチームの外では簡単な仕事ではないからです！以下の画像は、将来のDevOpsエンジニアがKubernetesの役割に基づいてアクセスシステムを調査する方法を示しています。

まず、最先端のテクノロジーの印象にもかかわらず、クベルネテスの考え方は比較的古いものです。 2015年にGoogleがBorg（Kubernetesの前身）からラッパーを削除したとき、それはすでにかなり古い考えでした。

これを読んでください：「ボーグシステムのアーキテクチャと機能の簡単な説明、重要な設計上の決定、その政治的決定のいくつかの定量分析、およびそれを使った10年の経験から学んだ教訓の定性分析を提供します。」もう一度読んでください！ 2015年（！）、GoogleはKubernetesに似たシステムの立ち上げの詳細を共有しました。そのシステムは当時10年以上前のものでした。



しかし、彼ら自身は恥ずかしがり屋ではありません。 Kubernetesホームページの最初の文は次のとおりです。「Kubernetes（K8s）は、コンテナ化されたアプリケーションのデプロイ、スケーリング、管理を自動化するためのオープンソースシステムです。管理と検出を容易にするために、アプリケーションを構成するコンテナを論理ブロックにグループ化します。 Kubernetesは、Googleのワークロードに関する15年の経験と、コミュニティの最高のアイデアと実践を組み合わせて構築されています。」

したがって、次に誰かがKubernetesを新しい「ホット」なアイデアとして提示し、世界を引き継ぐ準備ができているのを聞くとき、彼が現在少なくとも15歳のテクノロジーを表していることを思い出してください。それは少し革新的ではありませんか？
次に、対象読者について考えます。 Googleは、GoogleとGoogleの問題を解決するためのツールを作成しています。繰り返しになりますが、Kubernetesホームページはこれについて非常に明確に述べています。

最後に、元のKubernetes開発者の1人と彼の最もアクティブなサポーターであるKelsey Hightowerもこの点を強調しています。「Kubernetesはプラットフォーム全体を構築する人々のためのものです。独自のプラットフォーム（AppEngine、Cloud Foundry、Herokuクローン）を作成している開発者は、Kubernetesが必要です。」

したがって、地球規模で作業する場合、または毎週数十億のコンテナを起動する場合、または他のユーザーのためにクラウドを作成する場合、Kubernetesが適切な選択です。そうでない場合、これで話は終わりではありません。そして、あなたの祖母が昼休み中にケルシーのツイートをたくさん読んでから、CI / CDとAutomated Canary Analysisを使用して、彼女のフラワーショップのウェブサイトを1週間でKubernetesに変換したことは気にしません。これは、製品の性質上、使用する必要がない限り、適切なツールではありません。

しかし、それは本当に重要ですか？ Kubernetes = $$$。レベルアップして、DevOpsの世界への旅を楽しみ、経験を私と共有してください。
