強いタイプ、または-結局のところ-強い誇大広告？

近年、インターネットは強力なタイピングを使用することで世界のすべての利点を約束する傾向があります。最も粘り強い講演者は、「型がある場合、テストは必要ない」などの結論に至ります。これは最も正確な約束ではないようです。明らかに、強いタイピングが初期段階で人為的エラーをキャッチするのに役立つ状況がありますが、文字どおりあらゆる場所に強制的に導入することは、良いことよりも多くの問題を引き起こします。


以下に書かれていることは、著者の謙虚な意見にすぎません（そして世界で最も賢い人ではなく、平凡な開発者でもあります）。しかし、この見解は、30年以上の専門的な経験と、この期間を通じて信頼性と耐障害性のあるビジネスソリューションを提供してきたキャリアに基づいています。さらに、強力な型指定のすべての伝道者に、まずパーサーmarkdownを作成して、型がまったく役に立たない状況を自分で確認するように強く勧めます。


強力な類型化の熟達者の標準的な議論は、「リファクタリングなどを検討する」のようなものになります。ここを見てください：開発者が整数識別子の代わりにget_user_by_idインスタンスuserを引数として渡してを呼び出そうとすると、コンパイラーは即座に文句を言うので、エラーはランタイムにリークしません。そしてそれは本当です！しかし、あなたは何を知っていますか？私は数十年にわたってコードを書いており、魂のないマシンが文字通りアプリケーション全体を書くとは思っていませんでした。関数の引数に無関係なゴミを渡さないようにする責任があると私は部分的に感じています。 「しかし、私たちの努力を最小限に抑えるのは素晴らしいことでしょうか？」 「私は文字通りその議論を聞いています。」ええと、はい、いいえ。


インスタンスとしての使用が意図されているものに識別子を割り当てることは、デバッグするのが最も難しいエラーではありません。タイムスタンプとすべきものにテキストを割り当てることは、より強力なタイプではなく、より多くの注意が必要であることを意味します。キャッチして修正するのが本当に難しい問題が、別の沼で私たちを待っています。いくつかの例を見てみましょう（構文に関する宗教的な議論を避けるために、意図的に疑似言語を使用しています）。


無効なブランチ


サードパーティサービスへのHTTP呼び出しを処理する関数があるとします。これを変更して、さまざまな回答オプションにカバレッジを追加する必要があります。このようなものがありました。


function call_3rd_party(uri) {
  case call(uri) {
    200 => handle_success()
    else => handle_error()
  }
}

ここで、障害をより適切に処理したいと考えています。


function call_3rd_party(uri) {
  case call(uri) {
    200 => handle_success()
    500 => handle_error(UNAVAILABLE_SERVER_ERROR)
    503 => handle_error(INTERNAL_SERVER_ERROR)
    else => handle_error(UNKNOWN)
  }
}
