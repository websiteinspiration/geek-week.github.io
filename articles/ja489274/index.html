テストでメッセージをアサートする

また会いましたね。「C＃開発者」コースの開始に先立ち、テストでのアサートメッセージに関する興味深い資料を翻訳しましたので、喜んで翻訳を共有します。



この投稿では、テストでAssertメッセージを使用する必要があるかどうかについて説明します。

読者の同僚から興味深い質問を受け取りました。これについて詳しく説明し

ます。アサートメッセージについて質問があります。メッセージパラメータを含むオーバーロードを使用して、アサートの失敗の理由を説明する文字列を送信する必要があります（「クレーム」） ？

この質問への答えは2つの側面に帰着します：

  • テストの読みやすさ-テストの内容を理解するのがいかに簡単か。
  • 診断のしやすさ-テストが失敗する理由を理解するのがいかに簡単か。

それぞれについて個別に説明しましょう

テストの読みやすさ


多くの場合、テストで何が行われているのかをチームメンバーと自分自身が理解できるようにするために、アサートメッセージを使用します。次の例を見てみましょう：

[Test]
public void Hiring_a_new_team_member()
{
    var company = new Company();
    var person = new Person(UserType.Customer);

    company.HireNewMember(person);

    Assert.AreEqual(UserType.Employee, person.Type); //  
    Assert.AreEqual(1, company.NumberOfEmployees); //  
}

裸のアサートの代わりに、テストアサートが何かを検証する理由を示すこともできます。

[Test]
public void Hiring_a_new_team_member()
{
    var company = new Company();
    var person = new Person(UserType.Customer);

    company.HireNewMember(person);

    Assert.AreEqual(UserType.Employee, person.Type, "Person must become an employee after hiring");
    Assert.AreEqual(1, company.NumberOfEmployees, "Number of employees must increase");
}

このような発言は役に立ちますが、代償が伴います。これらのメッセージはあなたを必要とします

  • それらを書く時間を費やす
  • 彼らを前進させ続ける

ここで、長所と短所のセットは、コードへのコメントと同じです。そして、コメントの場合のように、私はあなたに助言します：単に読みやすさの目的でアサートメッセージを書かないでください。テストがアサートメッセージなしでは明白でないと思われる場合は、代わりにリファクタリングしてみてください。長期的には、アサートメッセージとの同期を維持するよりも、テスト自体を説明する方が簡単です（これは、同期が解除されるまでの時間の問題です）。

アサートメッセージは、絶対に必要な場合（他の方法ではテストの可読性を改善できない場合）にのみ入力してください。しかしそれでも、それらを書かないことを選択する傾向があります。

テストを読みやすくするための最も簡単な方法は、読み取り可能なステートメントレコードに切り替えることです。たとえば、NUnitには特別な制約ベースのステートメントモデルがあり、次の方法でステートメントを記述できます。

[Test]
public void Hiring_a_new_team_member()
{
    var company = new Company();
    var person = new Person(UserType.Customer);

    company.HireNewMember(person);

    Assert.That(person.Type, Is.EqualTo(UserType.Employee));
    Assert.That(company.NumberOfEmployees, Is.EqualTo(1));
}

または、私のお気に入りのFluent Assertionsを使用することもできます。

[Test]
public void Hiring_a_new_team_member()
{
    var company = new Company();
    var person = new Person(UserType.Customer);

    company.HireNewMember(person);

    person.Type.Should().Be(UserType.Employee);
    company.NumberOfEmployees.Should().Be(1);
}

これらのステートメントは平易な英語で読み取られます。つまり、すべてのコードを正確に読み取る方法と同じです。私たち人間は、物語の形で情報を知覚することを好みます。すべてのストーリーはこのモデルに準拠しています。

アサートメッセージのもう1つの見方は、診断の容易さです。つまり、このテストのコードを調べなくても、テストが失敗した原因を簡単に理解できます。これは、CIアセンブリの結果を読み取るときに役立ちます。

診断の観点からは、このガイドに従ってください。ローカルで簡単にテストを再開できる場合、このテストはアサートメッセージを必要としません。これはすべての単体テストに当てはまります（アウトプロセスの依存関係では機能しないため）が、統合テストとエンドツーエンドテストではそれほどではありません。


ピラミッド

のテスト統合（特にエンドツーエンド）テストは遅く、自由に再実行できない可能性があるため、テストピラミッドを高く登るにつれて、より詳細な情報が必要になる場合があります。

ただし、統合テストやエンドツーエンドテストを使用しても、メッセージをアサートせずに診断を容易にする方法があります。

  • テストテストを1つの動作モジュールにする—テストが1つのことをチェックするとき、何が問題だったかを簡単に特定できることがよくあります。（これは、エンドツーエンドテストに常に適用できるわけではありません。これらのテストで、動作の複数のユニットがどのように密接に動作するかを確認する必要がある場合があるためです）。
  • ユニットテストに適切な名前を付けます -理想的なテスト名は、アプリケーションの動作をビジネス用語で説明し、プログラマー以外でも理解できるようにします。

そして、ユーザーステートメントがなくても、ユニットテスト環境が生成するメッセージが残っていることを忘れないでください。

たとえば、次のステートメントのエラー：

person.Type.Should().Be(UserType.Employee);

次のエラーメッセージが表示されます。

Xunit.Sdk.EqualException: Assert.Equal() Failure
Expected: Employee
Actual:   Customer

環境によって生成されたそのようなメッセージと人間が読めるテスト名の組み合わせにより、ユーザーのアサートメッセージの90％は、診断の容易さの点でも役に立たなくなります。唯一の例外は、長いエンドツーエンドのテストです。多くの場合、これらには複数段階のチェックが含まれているため、追加のアサートメッセージを使用して、失敗したステップを理解することは理にかなっています。ただし、そのようなエンドツーエンドのテストは多くありません。

もちろん、フレームワークによって生成されたエラーメッセージを利用するには、次のような一般的なブール比較を回避する必要があります。

(person.Type == UserType.Employee).Should().BeTrue();

次のエラーメッセージが表示されるためです。

Xunit.Sdk.TrueException: Assert.True() Failure
Expected: True
Actual:   False

これは診断にまったく役立ちません。

概要


ステートメントを書くことに関して、あなたはしばしば怠惰だと思いますか？さて、あなたはそれを正当化し、同僚にこの記事を送ることができます。

ジョークはさておき、ここに要約を示します。

  • アサートメッセージの使用には2つの側面があります。
    • テストの読みやすさ（テストの内容を理解するのがいかに簡単か）。
    • 診断のしやすさ（CIビルド中にテストが失敗する理由を理解するのがいかに簡単か）。
  • テストの読みやすさの点では、アサートメッセージはコードコメントです。それらに依存する代わりに、読みやすさを実現するためにテストをリファクタリングします。
  • 診断のしやすさの点で、メッセージをアサートする最善の代替策は次のとおりです。
    • テストによる単一の動作モジュールのテスト
    • ビジネス用語でのテストの命名
  • 唯一の例外は、長いエンドツーエンドのテストです。
