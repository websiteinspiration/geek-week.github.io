2020年3月30日、Stack Overflowの開発者は、ウェブサイトの訪問者にダークテーマのベータ版を使用する機会を与えました。私たちが公開している翻訳は、Stack Overflowのダークテーマがどのように作成されたのかを物語っています。


Stack Overflowのバナー

まず、少し皮肉です。個人的に、私は暗いインターフェイスのテーマのファンではありません。

暗いインターフェイスのコントラストレベルが低すぎることがよくあります。このようなインターフェイスのデザインでは、すべての色を使用することは困難です。シャドウを使用してボリュームをシミュレートする場合や、他の視覚効果を適用する場合も同様です。暗い背景にある明るいテキストを読むと、目が疲れます。明るいテーマを使用すると、同時コントラストなどの現象に対処することが難しく、暗いテーマを使用すると、すべてがより複雑になります。

しかし、私はスタックオーバーフローに関して暗いテーマを提起した人物です。

多くのユーザーがリソースにそのようなトピックを装備することを長い間求めてきたという事実にもかかわらず、私が話したい仕事は、暗いトピックを開発することを特に目的としていませんでした。しかし、ダークトピックに移行する過程で、多くの問題を解決する必要がありました。特に、Stack Overflowフロントエンドコードがアップグレードされ、コンテンツのアクセシビリティが改善されました。ダークトピックに取り組み、プロジェクトで設計システムをより広く使用するインセンティブを得ました。

ユーザーに暗いテーマを与えると同時に、プロジェクトのアクセシビリティを改善する方法を開くことができますか？私たちはこの質問に肯定的に答え、私が今話すことをすべてやった。

▍変数


静的でコンパイルされていない16進数の色値を、プログラムの実行中に使用するように設計されたカスタムCSSプロパティに変換する必要がありました。ポイントは、色の値はvar(--red-500)、静的なビュー構造を使用するのではなく、ビュー構造を使用して格納する必要があるということ@red-500です。これは、私たちのシステム設計とサイト全体の両方で興味深い作業でした。通常、のような単一の色値を取り@red-500、要素のさまざまな状態（要素が要素の上に移動したとき、またはフォーカスされたときに取る状態など）をハイライトまたは暗くして、背景色または要素の境界線の色として使用します。

たくさんある各ボタンの色、各ボタンの各状態の色について話す場合、これらの色は、単一のコンパイルされた色値の一連の変換中に得られたと言えます。映画「秋のゲーム」のワンシーンを思い出しました。そこで彼らは1000万分の1の投資を数十億ドルに変える可能性について話しました。このような「変容」が金融危機につながったことは当然のことです。

通常のCSS変数の問題は、それらにLess変換を適用できないことです。 CSS変数の値は実行時に計算されるため、このような構成darken(var(--red-500), 5%)はコンパイルエラーになります。

つまり、ボタンのスタイル設定などを担当するコードを作り直す必要がありました。最初、彼はこのように見えました：

▍ブラウザの互換性


CSS変数について話す場合、それらのアプリケーションはブラウザーによるサポートを考慮する必要があることを意味しました。特に、これらはInternet Explorer 11でサポートされていません。これは私たちが忘れたことのないブラウザーです。その結果、このブラウザーの機能を考慮に入れて、長年の開発でシステムに追加したCSSハックをすべて取り除き、IE11サポートを放棄することにしました。さらに、新しいブラウザーのインストールを提案する通知をIE11ユーザーに送信することを決定しました。この決定は私たちにとって容易ではありませんでした。その実装には数週間のリファクタリングが必要でした。

▍条件付きクラス


IE11で制限がなくなり、スタックの色を操作できました。要素にbodyクラスを追加することにしました.theme-system。これにより、適切なメディアクエリを使用して、明るい色を暗い色に置き換えることができました。さらに、bodyクラスに追加するだけで、メディアリクエストを完全に拒否でき.theme-darkます。これにより、ユーザーはシステム設定に関係なく、サイトの暗いテーマを使用できます。結果は次のとおりです。

body {
    --red-600: #c02d2e;
}

body.theme-system {
    @media (prefers-color-scheme: dark) {
        --red-600: #d25d5d;
    }
}

body.theme-dark {
    --red-600: #d25d5d;
}

完全な柔軟性を実現するために、スタックにより、暗いテーマが有効になっている場合にのみ適用されるアトミックカラークラスを操作できます。スタックでのCSSサポートの詳細については、こちらをご覧ください。たとえば、クラスを要素に追加します。d:bg-green-100、デザイナーは次の考えを表現します。「ダークモードでは、背景に緑を使用する必要がありますbg-green-100。」ダークモード用に設計された追加の条件付きクラスを使用すると、要素の境界線を削除し、背景を変更し、テキストの色を調整できます。で、このつぶやき、あなたは色調整の興味深い例を見ることができます。ダークモードでは、このような設定が必要になる場合があります。私がサイトのダークモードで作業する際のインスピレーションの源の1つがTailwind CSSフレームワークであったことに注意したい。

▍タグ

当初、見積もりは強力な黄色の背景色を使用してフレーム化されていたため、見積もり自体のコントラストが低下していました。さらに、黄色は暗い背景に表示されるときに特定の問題を引き起こします。その結果、細い灰色の垂直バーを使用して強調表示することで、一般的に受け入れられている見積もり表示方法に移行しました。

かなり理解しやすいスタックオーバーフローのページには、多数のコードが表示されます。構文を強調するために使用した色は、署名色とは関係がありませんでした。これらの色は、構文の強調表示のためにプロジェクトで使用された最初のライブラリの遺産であると強く疑っています。これらの色を分析した結果、構文のハイライトを徹底的に再設計することにしました。その結果、バックライトの色は当社の設計システムの色に置き換えられました。これは、明るいトピックと暗いトピックの両方に適用されます。そして、これは、起こったことが以前のことと根本的に異ならないようにするために行われます。

結果



3月30日にリリースされた明るいテーマと暗いテーマのベータ版。

暗いトピックへの移行中に行ったリファクタリング作業の量により、プロジェクトの外観に問題なくかなり大きな変更を加えることができました。私たちが現在持っているものに基づいて、たとえば、そのアクセシビリティを高めるハイコントラストのウェブサイトのテーマを開発することができます。

ダークテーマの作成は、スタックオーバーフロー設計へのアプローチの根本的な変化の結果でした。特に、私が1年前から推進してきたデザインシステムの活用について話しています。また、暗いテーマを作成することは、サイトの多くの部分を再構築する絶好の機会でした。これは、Stack Overflowの可用性を向上させる多くのプロジェクトの最初のものです。

IE11のサポートの拒否を考慮しない場合、ダークトピックに関する作業は、状況の調査を目的としたこの PRで2019年7月に始まりました。これに先立ち、2019年4月には、暗いトピックに関する議論がありました。2019年10月、ダークテーマの実験版が製品化されました。そして、少なくとも60のPRが行われた後、ダークテーマのベータ版がユーザーに提示されました。それは2020年3月30日に起こりました。
