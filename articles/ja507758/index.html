JavaScriptメモリ-リークなし









データの作成方法とアクセス方法は、アプリケーションのパフォーマンスに影響を与える可能性があります。方法を見てみましょう。



前書き



JavaScriptは非常に高水準の言語です。したがって、ほとんどの開発者は、データがメモリに格納される方法について考えていません。この記事では、データがメモリに格納される方法、それがプロセッサとメモリにどのように影響するか、データをJSに配布してアクセスする方法がパフォーマンスにどのように影響するかを見ていきます。



三角関係



コンピュータがいくつかの計算を実行する必要がある場合、プロセッサ（CPU）は処理するデータを必要とします。したがって、タスクに従って、彼はバスを介してデータサンプリングのリクエストをメモリに送信します。



これは次のようになります：







これが私たちのロマンチックな三角形です-プロセッサー->バス->メモリ



トリオで健全な関係を維持することは困難です



プロセッサはメモリよりもはるかに高速です。したがって、このプロセスはプロセッサ->バス->メモリ->バス->プロセッサは計算に時間を費やします。メモリのスキャン中、プロセッサはアイドル状態です。



ダウンタイムを防ぐために、システムにキャッシュが追加されました。キャッシュとは何か、またはキャッシュの種類については詳しく説明しませんが、キャッシュはプロセッサの内部メモリであると言えます。



CPUは、実行するコマンドを受け取ると、まずキャッシュ内のデータを探し、そこにデータがない場合は、バス経由で要求を送信します。



次に、バスは、要求されたデータとメモリの条件付き部分を転送し、すばやく参照できるようにキャッシュに格納します。



したがって、CPUはメモリの速度に対する不満が少なくなるため、CPUのダウンタイムが短くなります。



けんかはあらゆる関係の一部です。



特に大量のデータを処理するときに発生する可能性のある問題は、キャッシュミスと呼ばれる現象です。



キャッシュミスとは、計算中に、CPUがキャッシュに必要なデータがないことを検出し、通常のチャネルを通じてこのデータを要求する必要があることを意味します。低速のメモリが大量にあることに注意してください。







現金伝票のイラスト。アレイからのデータは処理されますが、キャッシュ制限外のデータも計算のために要求されます-これにより「キャッシュミス」が発生します。

フェア。例を見てみましょう。



これはBoomというクラスです。 



class Boom {
  constructor(id) {
    this.id = id;
  }
  setPosition(x, y) {
    this.x = x;
    this.y = y;
  }
}


このクラス（ブーム）には、id、x、yの3つのプロパティしかありません。



次に、xとyにデータを入力するメソッドを作成します。



データを設定しましょう： 



const ROWS = 1000;
const COLS = 1000;
const repeats = 100;
const arr = new Array(ROWS * COLS).fill(0).map((a, i) => new Boom(i));


次に、このデータをメソッドで使用します。 



function localAccess() {
  for (let i = 0; i < ROWS; i++) {
    for (let j = 0; j < COLS; j++) {
      arr[i * ROWS + j].x = 0;
    }
  }
}


localAccessが行うことは、配列を線形にトラバースし、xを0に設定



することです。この関数を100回繰り返すと（repeats定数を見てください）、実行にかかる時間を測定できます。 



function repeat(cb, type) {
  console.log(`%c Started data ${type}`, 'color: red');
  const start = performance.now();
  for (let i = 0; i < repeats; i++) {
    cb();
  }
  const end = performance.now();
  console.log('Finished data locality test run in ', ((end - start) / 1000).toFixed(4), ' seconds');
  return end - start;
}
repeat(localAccess, 'Local');

キャッシュミスの価格



さて、上記で学んだことによると、反復中に遠く離れたデータを処理する場合、キャッシュミスの問題を考慮することができます。削除されたデータは、隣接するインデックスにないデータです。これのための機能はここにあります： 



function farAccess() {
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      arr[j * ROWS + i].x = 0;
    }
  }
}


ここで発生するのは、各反復で、最後の反復からROWSの距離にあるインデックスに目を向けることです。したがって、ROWSが（この例のように）1000の場合、次の反復が得られます：[0、1000、2000、...、1、1001、2001、...]。



これを速度テストに追加しましょう： 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local');
}, 2000);


そして、これが最終結果です。







非ローカル反復は、ほぼ4倍遅くなりました。この違いは、データが増えると大きくなります。これは、キャッシュミスによりプロセッサがアイドル状態になるためです。

この問題にはさまざまな解決策がありますが、このパフォーマンス低下の理由がわかったので、おそらく自分で解決策を考え出すことができます。処理するデータをできるだけメモリに保存するだけです。

この手法は、データ局所性設計パターンと呼ばれます。
例を続けましょう。このアプリケーションで最も一般的なプロセスは、farAccess関数に示されているロジックを使用したデータ処理であるとします。最も一般的なforループで迅速に実行されるようにデータを最適化したいと思います。



次のデータを整理します。 



const diffArr = new Array(ROWS * COLS).fill(0);
for (let col = 0; col < COLS; col++) {
  for (let row = 0; row < ROWS; row++) {
    diffArr[row * ROWS + col] = arr[col * COLS + row];
  }
}


そのため、diffArrでは、元の配列のインデックス[0,1,2、...]にあるオブジェクトが次のように設定されます[0,1000,2000、...、1、1001、2001、...、2、1002 、2002、...]。数字はオブジェクトのインデックスを示します。これは、配列のソートをシミュレートします。これは、データ局所性設計パターンを実装する1つの方法です。



これを簡単に確認するために、farAccess関数を少し変更してカスタム配列を取得します。 



function farAccess(array) {
  let data = arr;
  if (array) {
    data = array;
  }
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      data[j * ROWS + i].x = 0;
    }
  }
}


スクリプトをテストに追加します。 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local')
  setTimeout(() => {
    repeat(() => farAccess(diffArr), 'Non Local Sorted')
  }, 2000);
}, 2000);
