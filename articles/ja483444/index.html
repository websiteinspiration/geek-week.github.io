2019 DORAレポート：DevOpsパフォーマンスを改善する方法



数年前、多くの組織はDevOpsをソフトウェア開発への基本的なアプローチではなく有望な実験と見なしていました。現在、DevOpsは、新製品のリリースを加速し、生産性を向上させる、実績のある強力な開発および導入プラクティスとツールのセットです。さらに重要なことは、DevOps効果は、ビジネスの全体的な成長と収益性の向上を目的としています。Mail.ru Cloud Solutions

チームは、DevOps Research＆Assessment（DORA）の専門家によって編集された、2019 Accelerate State of DevOps Reportから最も興味深いものを翻訳しました。この調査には、世界中の31,000人の専門家が参加しました。 2019年の業界の変化と、ビジネスがソフトウェア配信の効率を向上させる方法を見てみましょう。

業界と企業の規模がDevOpsに与える影響


この調査では、DevOpsのパフォーマンスと組織の業界との間のリンクは特定されませんでした。ただし、パフォーマンスがわずかに優れている小売業を除きます。これは特に、小売業者が需要と顧客のニーズの変動に迅速に対応する必要があるためです。調査によると、金融セクターや公共セクターを含むすべての企業がDevOpsで高いレベルを達成できます。

従業員数が5,000人を超える企業のDevOpsパフォーマンス指標は、従業員数が5,000人未満の企業よりも低かった。ほとんどの場合、これは、大規模な組織では、プロセスが大きくなり、制御が厳しくなり、ITシステムのアーキテクチャがより複雑になり、コードの開発と展開のプロセスに遅延が生じるためです。同時に、専門家は、会社の規模がDevOpsの構築の成功を妨げるものではなく、場合によってはより多くの労力が必要になると考えています。

レポートでは、DevOpsの有効性を評価するための4つの主要な測定基準が使用されました。ソフトウェア開発中の変更の完了時間、展開の頻度、障害の頻度、および回復の時間です。

この調査では、次の傾向が明らかになりました。高レベルの指標を持つチームの数はほぼ3倍になり、2018年の全回答者の7％から2019年には20％に増加しました。


パフォーマンスレベル別の開発チームの分布。

低パフォーマンスグループのチームと比較して、高パフォーマンスのDevOpsチーム：

  1. 208倍のコード展開を実行します。
  2. コードの展開に費やした時間は106分の1でした。
  3. 障害が発生する可能性が7倍少なくなります。
  4. 障害後の2.604倍高速なソフトウェアの復元。

さらに、高スコアのDevOpsチームは、低スコアのチームよりも組織のパフォーマンス指標を達成または超える可能性が2倍高くなります。

多くの専門家は、すべての指標を同時に増加させることは不可能であると考えています。そのため、リリースのロールアウトの速度を上げると、ソフトウェア配信プロセスとサービスの提供の信頼性に悪影響を及ぼす可能性があると考える人もいます。しかし、研究の結果、結果の速度と安定性は相互に排他的ではないことが示されています。

DevOpsプロセスを改善する方法


レポートには、DevOpsの改善に役立つ2つの領域が示されています。ソフトウェアの開発と配信の効率の向上と生産性の向上です。


各方向には独自のコンポーネントが含まれており、これにより、目的を達成できます。

レポートによると、デジタル変革の鍵は企業文化です。非常に効果的なDevOpsチームには、信頼と心理的安全の文化、作業結果の理解、明確な目標が必要です。このような環境により、チームメンバーは情報に基づいた決定を行い、意見を表明し、より創造的になることができます。

クラウドテクノロジー、継続的デリバリー、ディザスターリカバリーテスト、および変更管理も、ソフトウェアの開発とデリバリーの効率向上に役立ちます。使いやすいツールに投資し、技術的負債を減らす、つまり非効率的なコードと古いテクノロジーの割合を減らし、企業のナレッジベースを整理して外部ソリューションにアクセスすることで、労働生産性を向上させることができます。

DevOpsの方法論とイデオロギーは、これらのプロセスがクラウドや独自のハードウェアなどの外部条件に依存していないということです。クラウド自体は単なるツールにすぎず、どこかで役立つか、どこかでそれが妨げられるか、需要がなくなるでしょう。

DevOps


DevOpsの実装を希望する多くの組織は、一連の指示またはベストプラクティスを探しています。ただし、同じ会社はありません。したがって、どのプラクティスを選択するかは、ビジネスの現在の状態とその目標によって異なります。

同時に、DevOpsの有効性を向上させるのに役立つ一般的な方向性があります。それらのいくつかはチームレベルで開発され、他は組織レベルでの取り組みを必要とします。

この調査により、疎結合アーキテクチャがDevOpsのパフォーマンスに与えるプラスの影響が確認されました。

疎結合アーキテクチャとは、チームが、追加のサポート、リソース、承認なしで、フィードバックを少なくして、他のチームとは独立してオンデマンドでシステムを個別にテスト、展開、および変更できる場合です。これにより、より効率的に作業できますが、高度な組織と管理が必要になります。

このアプローチは、スタートアップといくつかの予約でのみ可能です。他の会社は異なる状況を持っているかもしれません。良い例：銀行/フィンテック。そこで使用できるのは専有ソリューションのみですが、DevOpsプラクティスが適用されます。

Denis Romanenko、フリーランスエキスパートMail.ruクラウドソリューション

成功したDevOpsチームがすべてを自動化


継続的インテグレーションとデリバリー（CI / CD）により、サービスとアプリケーションをより少ないコストとリスクで製品に提供し、組織の目標に従ってリリースを維持できます。

CI / CDが成功すると、チームはオンデマンドで本番環境への変更を実装でき、導入の品質に関するフィードバックを即座に確認でき、迅速に解決でき、次の導入サイクルを改善できます。

レポートは、成功したDevOpsチームが幅広いサポートプロセス、実践、ツールに投資していることを示しています。

  • 92％は自動組み立てツールを使用しています。
  • 87％は自動ユニットテストを使用しています。
  • 57％が自動化を受け入れテストに拡張します。
  • 72％はテスト環境での展開を自動化し、69％は本番環境での展開でも同じことを行います。
  • 69％がチャットボットを導入プロセスに統合しています。
  • 57％が監視ツールと統合されています。

適切なツールとテクノロジーを選択することが重要です。


複雑なシステムを構築して重要なビジネスインフラストラクチャを管理する場合、次のテクノロジーを選択することが重要です。

  • これは、最初の接続中と一定の操作の両方で簡単に使用できます。
  • あなたの目標を達成するのに役立ちます。

回答者の40％のみが、これらの方法の1つ以上を使用して毎年障害復旧テストを実施しています。同時に、災害復旧テストを実施する企業は、より高いレベルのサービス可用性を備えています。このレポートは、パフォーマンスの高いDevOpsチームが、ソフトウェア開発および展開プロセスで災害復旧テストデータを考慮する可能性が1.4倍高いことを示しています。

DevOpsチームが情報にアクセスできることが重要です


DevOpsチームのパフォーマンスを高いレベルに保つことで、問題を解決するための情報を簡単に見つけることができます。これは、複雑なシステムで構成される現代の技術環境で特に当てはまります。

このような情報のソースは、次の2つのグループに分類できます。

  1. 内部ソース：コードの作成と保守のための企業ドキュメント、企業ナレッジベース、リポジトリなど。内部の情報源を使用したDevOpsチームの生産性は1.73倍でした。
  2. 外部ソース：検索エンジンとスタックの補充。外部DevOpsチームの生産性は1.67倍でした。外部テクノロジーは、学習と成長、特にパブリッククラウドとオープンソースツールの使用に大きな利点をもたらします。

企業が技術的負債を減らすことは重要です


技術的負債には、既知のバグが修正されていないコードまたはシステムが含まれます。テストカバレッジが不十分。低品質のコードまたはデザイン。使用されていないが削除されていないアーティファクト。チームが効果的にサポートできない実装。時代遅れのテクノロジー; 不完全または古いドキュメント。

専門家は、技術的な負債がDevOpsのパフォーマンスに悪影響を及ぼすことを発見しました。技術的負債の多いチームは、生産性が1.6倍低下しました。レートの高いチームは、技術的負債が少ない可能性が1.4倍高かった。
