Yoctoプロジェクトの紹介


Yocto Projectは、組み込みシステム向けのディストリビューションの開発を簡素化するための共同オープンソースプロジェクトです。 Yoctoには、多数のテンプレート、メタデータ、ビルドツールが含まれています。 Yocto Projectでは、あらゆる種類のハードウェアプラットフォーム用の多数のBSP（プラットフォームサポートパッケージ）レイヤーを接続できます。

この記事の主な目的は、シングルボードRaspberry Pi 3Bコンピューター用の有名なマルチメディアセンターKodiバージョン17.6を例として使用して、Yoctoプロジェクトの典型的な配布パッケージのアセンブリを示すことです。

あなたの魂のどこかであなたがあなたがコレクターであると感じているが、あなたが何を集めたいかまだ決めていないなら、この記事はあなたのためです。すでにYocto Projectを日常業務で使用している場合は、この記事をスクロールできます。最後の章に進んで準備をしてください。

この記事は純粋に実用的であり、Yocto ProjectとOpenEmbeddedの成果をマルチメディアセンター「Kodi」のアセンブリに使用する可能性を示しています。Yoctoレイヤーは、GoogleのRepoユーティリティを使用して管理されます。このシリーズのドキュメントの記事。

Yocto Projectの配布ビルドエンジン


Yocto Projectでは、各プログラムユニットはアセンブリレシピを使用して記述されます。レシピ記述言語は「bash」を連想させるもので、Python言語でコードの一部を挿入できます。基本的な構文情報は、Yocto Projectのマニュアルから入手できます。目的に応じて、一連のアセンブリレシピを組み合わせて、個別のアセンブリレイヤーにすることができます。

レイヤーは、ハードウェア依存-BSPレイヤー、UIレイヤー（ユーザーインターフェイス）、特定のYoctoレイヤー、および特定の機能を実装するレイヤーに分割されます。
たとえば、OpenEmbeddedからのレイヤー=>マルチメディア、python、perl、ruby、ネットワーク、systemd、webserverなど。 。

YoctoプロジェクトでOpenEmbeddedを使用する


それでも、Yocto Projectを使用する場合は、確かに追加機能を備えたレイヤーが必要になります。すべての機会のためのレシピの大規模なセット。そして、そのようなセットがあります-これらはOpenEmbeddedからのレシピです。OpenEmbeddedは、組み込みLinux用のパッケージを構築するためのインフラストラクチャです。

OpenEmbeddedはYoctoプロジェクトと完全に互換性があります。これは、このプロジェクトがYoctoプロジェクトの基礎として使用されたためです。おそらくそれが、Yoctoプロジェクトの安定性、ドキュメント、サポートがわずかに優れている理由です（ただし、基本的にはOpenEmbeddedと同じです）。

プラットフォームサポートパッケージ（BSP）


ボードサポートパッケージは、プラットフォームのハードウェア機能を定義する特定のボード用の個別の専用レイヤーです。プロセッサの機能、割り込み、アドレス指定、ブートローダーの機能、ビデオアダプター（GPU）の機能など、ボードを区別する特定の機能を実装します。

この記事ではBSPレイヤー-meta -raspberrypiを使用します。
レイヤーリポジトリはgit.yoctoproject.org/git/meta-raspberrypiにあります。

RepoでYoctoレイヤーを管理する


Yocto Projectは、さまざまなサプライヤー（機器開発者）からの多数のレイヤーを使用できますが、これらすべてを何らかの方法で管理する必要があります。 12種類のボードがあり、各ボードには個別のBSP gitリポジトリが付属していると想像してください。これには、Yoctoプロジェクト自体のインフラストラクチャに加えて、OpenEmbeddedからの追加機能の可能性も含まれていません。

この状況では、個別の簡単なインストールスクリプトで済むことはありません。 Willy-nillyは、それをうまく実行できるツールを探す必要があります。さらに良い。この種の最高のツールの1つは、Googleユーティリティ-Repoです。

レポ-これは、Androidオペレーティングシステムとその大規模なコードベースのアセンブリ中にGITリポジトリを管理するためのメインツールです。Repoを使用すると、1つのプロジェクトで数百の個別のgitリポジトリを管理できます。そのバージョンは、マニフェストの 1つのxmlファイルで慎重に指定でき

、すべてのリポジトリのすべてのバージョンを正しく同期するには、1つのrepo syncコマンドを実行するだけです

配布ビルドマニフェスト


記事の一部として組み立てられているディストリビューション、それを何かと呼ぶ必要があります。Torvinという名前にします。コードネームはTorvinで、単一のプログラムのアセンブリを備えた最小限のLinuxディストリビューションが存在します。これは、1つのアプリケーションユーザープログラム-Kodiのみを意味し、それ以外は何もありません（他はすべてシステムレベルです）。マルチメディアセンターの場合、私の意見ではこれで十分です。

torvin-0.2.5.xml ファイルは、ディストリビューションレイヤーの管理に使用されます。

ポーキー変数の初期化


setup-environment 初期化スクリプトは、Freescale Communityプロジェクトから取得されました（yocto環境では、これは一般的なソリューションです）。スクリプトは、Pokyビルドシステムの変数の初期化を担当します。スクリプトは、非常によく分割されたディレクトリ構造を作成します。

  • build-ビルドディレクトリ
  • ソース -アセンブリソースコードをレシピ
  • download-プログラムコードをダウンロードするためのディレクトリ（gitデータベース、tar.gzアーカイブ）

setup-environmentスクリプトの内容は、次の場所で確認できます。

Shell.shスクリプトの内容
    #!/bin/bash

    MACHINE='raspberrypi3' source ./setup-environment build

    echo "you may try 'bitbake core-image-minimal'"
    bash


このルートスクリプトは、ビルド環境の構成変数を初期化するために使用され、通常、セッションの開始時に呼び出されます。

レポの初期化


リポジトリを初期化するには、次のコマンドを実行する必要があります。

mkdir torvin
cd torvin
repo init -u https:⁄⁄github.com/berserktv/bs-manifest -m raspberry/rocko/torvin-0.2.5.xml

ここで、-u https：⁄⁄github.com / berserktv / bs-manifestは、マニフェストプロジェクト

ノートへのGITパスを示します。- bcolor_nameを指定することもできます
（-bキーを指定しない場合、（デフォルトで）マスターブランチが想定されます）

パスはm raspberry / rocko / torvin-0.2.5.xmlを構成ファイルに追加すると、次のようになります。

  1. アセンブリが実行されているハードウェアプラットフォームの名前- ラズベリー
  2. Yocto / OpenEmbeddedのメインのブランチの名前はrockoです
  3. バージョンのコード名はtorvinです（0.2.xシリーズのすべてのバージョン）。
  4. 組み立てられているデジタルバージョン番号は0.2.5です。

リポジトリ同期


初期読み込みまたはその後の同期の場合は、次のコマンドを実行するだけです。

repo sync

これは、マニフェストファイルで指定されたGITプロジェクトの最新バージョンをすべてピックアップします（ブランチは通常示されます）。リビジョン属性にハッシュコミットまたはタグ名がある場合、このgitリポジトリのバージョンは変更されません。タグ名は次のように指定できます：revision = "refs / tags / v0.2.5"

local.confファイルの主な変数-次のことに注意する必要があります。

  • MACHINE-アセンブリが実行されるプラットフォームの名前
  • DISTRO-配布カテゴリー名
  • PACKAGE_CLASSES-ソフトウェアをインストールするためのパッケージ形式
  • LICENSE_FLAGS_WHITELIST-追加ライセンスの使用

Raspberry Piファミリーのボードに固有の設定

  • GPU_MEM = "128"-GPUビデオアダプター用のビデオメモリの量（RAMから割り当て）
  • GPU_MEM_256 = "112"-合計RAMサイズ= 256MBのボードのみ同じ
  • GPU_MEM_512 = "160"-合計RAMサイズ= 512MBのボードのみ同じ
  • GPU_MEM_1024 = "320"-合計RAMサイズ= 1024MBのボードのみに同じ

注：
たとえば、変数GPU_MEM = "128"のみを残した場合
、すべてのRPI、RPI2、RPI3ボードで は、ボード上
の実際のRAMの量に関係なく、
常に128 MB がGPUに割り当てられます
（RAMの合計サイズはこの値によって減少します）

すべての変数が指定されている場合、ディレクティブGPU_MEM_256、GPU_MEM_512、GPU_MEM_1024が優先されます。

マルチメディアセンターを組み立てるためのレイヤー


私の意見では、Yocto Projectは組み込みディストリビューションを作成するための産業用結合です。しかし、Buildrootビルドシステムを使用したことがある場合、Yoctoは扱いにくいように思えるかもしれません。ハードディスクの空き容量が膨大です。通常の操作では、Yoctoには約80〜100 GBの空き容量が必要です。これは通常、1つのプラットフォームのみのアセンブリを考慮しています。

Yoctoは、その主な目的に完全に対応します-可能な限り多くの異なるハードウェアプラットフォームをサポートします。これには、最も柔軟なアセンブリスイッチングメカニズムが必要です。そして、このメカニズムには時間と場所が必要です。 Yoctoでディストリビューションを構築するのは簡単なプロセスではありません。

したがって、別のレイヤーを使用して「マルチメディアセンター」を構築するためのすべての機能：

https://github/berserktv/berserk

（名前は、私のお気に入りの本、ハリーギャリソンのThe Hammer and the Crossから取られています。）
（Torvinもこの本の登場人物です。）

必要な機能を実現するために、拡張子が付いたファイルにある、レシピのいわゆるアドオンを使用します。 .bbappend
ファイルのbbappend では、たとえば、do_configure、do_compile、do_installメソッドなどに、通常のビルドレシピメソッドの独自のコマンド呼び出しを追加できます。

CONFIG_CFG80211_INTERNAL_REGDB = y

コディビルドレシピ追加


OpenEmbeddedレイヤーには、Kodiを構築するための通常のレシピがありますが、それは非常に一般的であり、Raspberry Piプラットフォーム用に少し安定したテスト済みバージョンを入手したいと考えています。

ソフトウェア開発者には、リバースポートパッチなどがあります。ソフトウェアは常に更新されており、新しいバージョンには新しい機能と既知のエラーの修正の両方が含まれています。リバースポーティングパッチを使用すると、プログラムの新しいバージョンの変更の一部を古いバージョンに転送して、より安定させることができます。しかし、これは非常に困難で骨の折れる作業であり、常に多くの開発者によって実行されます。

オープンソースコミュニティの世界では、Kodiを使用した有名なプロジェクトがいくつかありますが、そのうちの最高のもの（私の意見では）はLibreElec（OpenElec）です。LibreElecはRaspberry Piプラットフォーム向けの優れたビルドを備えています。ここでは、Kodiのリバースポートパッチを取得するための最良の方法があります。したがって、それについて学ぶことさえなくても、膨大な数の問題を取り除くことができます。

Kodiマルチメディアセンターは、「メディア」のコンテキストの再生に焦点を当てており、私の意見では、最も重要なコンポーネントは、KodiとFFmpegの組み合わせです。これらのプログラムの特定のバージョンの相互作用、他のライブラリは、YoctoおよびOpenEmbeddedレイヤーから残すことができます。

アセンブリには、Kodi 17.6の安定バージョンとFFmpeg 3.1.11を使用しました

IPTVでテレビを見る


Kodi Media Centerは、デジタルコンテンツを表示するための非常に強力なツールです。

「メディアセンター」を集めた主な機能は、IPTV（Internet Protocol Television）を使用してデジタルテレビを視聴する機能です。インターネットプロトコル上のテレビ。この機能を使用すると、インターネットプロバイダーのデジタルテレビを見ることができます。

これは、画質と追加機能の両方の点で最もモダンで最適なオプションです。提供するサービス。例えば、テレビチャンネルをアーカイブに提供することができ、そこでは、所望のビデオ録画が放送後しばらくの間利用可能である。

KodiでIPTVをサポートするために、いくつかのプラグインオプションがあります。pvr.iptvsimple

プラグインをビルドするには、次のレシピを使用します。

    └──kodiプラグイン
        └──kodi-pvr-iptvsimple.bb

コンソールシェルネットワーク構成拡張


この記事の一部として組み立てられた配布キットはデモなので、「ネットワークインターフェイス」をセットアップするための要件は最小限です。これには重いネットワークマネージャーをドラッグしたくありませんでした。これは私には非常に理解しづらく、非常に煩雑でした。そのため、通常の構成メカニズムの構成を補完する2つのシェルスクリプトを作成しました。

##############################################################
#     /etc/network/interfaces:
##############################################################
auto eth0
     iface eth0 inet manual
     up /etc/network/eth-manual $IFACE up
     down /etc/network/eth-manual $IFACE down

auto wlan0
     iface wlan0 inet manual
     up /etc/network/wlan $IFACE up
     down /etc/network/wlan $IFACE down

GUIを介してイーサネット/ WLANネットワークインターフェイスを便利に構成するために、別の小さなKodiプラグイン「script.berserk.network」を使用します。これは私が考え出したほとんど唯一のKodiプラグインですが、これを書くためにそれを書かなければなりませんでした。非常にコンパクトでミニマルで、Python言語で書かれています。

これらのコンポーネントは両方とも、レシピを使用して収集されます。

  • レシピ-berserk / bs-net / bs-net_0.1.3.bb
  • レシピ-mediacentre / kodi-plugins / script-berserk-network_0.2.5.bb

この時点で、詳細について詳しく説明します。したがって、Yoctoを使用することのすべての柔軟性は、レシピの異なるセットにあります。1セットのレシピを接続-最も単純なネットワークマネージャーが配布キットに追加され、別のセットが接続-たとえばsystemDなどを使用してお気に入りのネットワークマネージャーを追加

システムの起動時にWiFiアクセスポイントに自動的に接続するには、udevルールを使用します：/etc/udev/rules.d/80-wifi-start.rules

SUBSYSTEM=="net", ACTION=="add", DRIVERS=="?*", KERNEL=="wlan*", \
                  RUN+="/etc/network/wlan-runner $env{INTERFACE} up"

SUBSYSTEM=="net", ACTION=="remove", DRIVERS=="?*", KERNEL=="wlan*", \
                  RUN+="/etc/network/wlan-runner $env{INTERFACE} down"

wlan-runnerスクリプトは次のコマンドを実行するだけです：
/ etc / network / wlan $ IFACE up
または
/ etc / network / wlan $ IFACE down
