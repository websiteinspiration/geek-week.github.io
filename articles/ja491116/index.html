最新の識別標準：OAuth 2.0、OpenID Connect、WebAuthn

させるかさせないか？それが問題です…

現在、多くのサイトでソーシャルネットワークを使用して登録またはログインする機会があり、一部のサイトでは外部セキュリティキーまたは指紋の使用を提供しています。それは何ですか？適切に設計されたセキュリティ標準または独自の実装？これらのテクノロジーを信頼して、ウェブサイトの開発や日常生活で使用できますか？それを正しくしましょう。そのため、OAuth 2.0、OpenID Connect、WebAuthn、SAML 2.0、Credential Management APIなどのユーザーを識別するための標準とテクノロジーがいくつかあります。この記事では、OAuth 2.0、OpenID Connect、WebAuthnの3つの最も有望なプロトコルについて説明します。そして、それらをどのように実行するかを理解するために、3つの実験室での作業を行います。ユーザーを識別するためのプラットフォームとしてGitHubとGoogleを使用します。ほとんどのアカウントがあります。



OAuth 2.0


最も有名なOAuth 2.0プロトコルから始めましょう。2012年にRFC 6749：OAuth 2.0 Authorization Frameworkとして公開されました。

ユーザーはOAuth 2.0を使用して、特定のサイトがソーシャルネットワークからプライベートデータを受信することを許可しますが、ログイン/パスワードをサイトに転送しません。たとえば、Facebookを介してサイトに登録する場合は、このサイトにFacebookから名前、電子メールアドレス、その他のプライベートデータを取得する許可を与えるだけです。

技術的な実装を扱いましょう。簡単にするために、ソーシャルネットワークにユーザーの資格情報を保存するすべてのサイトを呼び出します。MySiteは、ソーシャルネットワークからユーザーデータを取得したいサイトまたはアプリケーションに名前を付けます。

Openid接続


OAuth 2.0については少し理解しました。次に、OAuth 2.0のアドオンであるOpenID Connectが必要な理由を調べてみましょう。

  • C OAuth 2.0 .. access token, . access token MySite. OpenID Connect — (identity). .
  • OpenID Connect “service discovery”. SSO (Single Sign-On), .

標準を技術面から見てみましょう。

OpenID Connect（OIDC）は、OpenID Foundationコンソーシアムによって開発されたオープンなOpenID標準です。OIDCは、OAuth 2.0を次の主要機能で拡張します。

  • 承認サーバーは、アクセストークンと更新トークンに加えて、 "IDトークン"（IDトークン）を返します。同じJWTに含まれています。IDトークンから抽出できる情報は、ユーザー名、ログイン時間、IDトークンの有効期限です。トークンIDは参加者間で転送できます。
  • OIDCは、ユーザーとその現在のセッションに関する情報を要求できるようにする追加のAPIを提供します。

OpenID Connectの相互作用図は、OAuthと同じように見えます。リクエストの内容の唯一の違い：

  • コードの最初のリクエストでは、追加の属性scope = openidが追加されています。
  • アルゴリズムの結果として、クライアントは、アクセストークンと更新トークンに加えて、トークンIDを受け取ります。

Webauthn


Web認証API（別名WebAuthn）：

  • この規格により、ユーザーはWebサイトやアプリケーションで外部セキュリティキー（USBキーなど）を使用して、または指紋、さらには他の生体認証データ（顔、網膜）を使用して自分自身を識別することができます。

  • W3C (World Wide Web Consortium) FIDO, Google, Mozilla, Microsoft, Yubico. W3C HTTP, HTML, XML . WebAuthn. WebAuthn : Chrome, Firefox, Edge, Safari.


OAuth 2.0と比較して、WebAuthnは以下の役割を追加します：

Authenticator：外部セキュリティキー（物理メディアまたは指紋スキャナー）。これにより、BlueTooth / NFC / USBなどのさまざまなテクノロジーを使用してユーザーを認証できます。に役立つ：

  • 公開鍵資格情報の生成（公開/秘密鍵のペア）。
  • オーセンティケータは秘密鍵をメモリに安全に保存します
  • 公開鍵を外部システムに渡します
  • 秘密鍵でデータに署名し、結果を外部システムに転送します

オーセンティケーターはCTAPプロトコル（クライアントからオーセンティケーターへのプロトコル）を使用してブラウザーと対話します。

証明書利用者：OAuth 2.0の「承認サーバー」と同じ機能を実行します。つまり、ユーザーのIDを確認します。OAuth 2.0でのみユーザー名/パスワードであり、WenAuthnでは公開鍵資格情報でした。

ユーザーエージェント：ブラウザーとネットワークアプリケーションを統合し、OAuth 2.0のクライアントと同じ目的を果たします。つまり、一方ではユーザーと対話し、ユーザーにGUIを提供します。他方では、ユーザーの資格情報を保存するシステムと対話します。

承認フロー


認証プロセスを開始する前に、OAuth 2.0と同様に、準備手順を実行する必要があります。OAuth2.0でのみアプリケーションを登録し、WebAuth 2.0ではユーザーを登録します。Web認証APIは2つの呼び出しを指定します。

  • navigator.credentials.create-ユーザーの資格情報を作成する
  • navigator.credentials.get-ユーザーの資格情報を確認します

したがって、登録するには、navigator.credentials.createを呼び出す必要があります。

この結果、ユーザーの認証システムは特定のサイトの秘密鍵を保存し、公開鍵は証明書利用者に保存されます。

SAML 2.0（セキュリティアサーションマークアップ言語）


2005年の成熟したプロトコルですが、SSOシステムを構築するためのシナリオのセットは限られています。XMLベースのデータ形式を使用します。詳細については、「SAML 2.0認証プロトコルを使用するユーザー」の記事をご覧ください。

資格情報管理API


開発はWebAuthn-W3Cと同じ組織によって行われます。資格情報管理標準を使用すると、次のことができます。

  • サブスクライバーのIDを保存します。これにより、ユーザーはパスワードを入力せずにサイトにアクセスできますが、ストアのパスワードを使用できます。
  • 特定のサイトを入力するために必要なアカウントを選択します。
  • あるデバイスで入力されたログイン/パスワードを他のデバイスで使用できるようにします。

Credential Management APIの一般的な実装例は、Googleのパスワードマネージャーであるpasswords.google.comです。
