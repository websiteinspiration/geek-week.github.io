RSトリガーへの別のアプローチ、TLA +

RSトリガーを完全同期回路としてモデル化しました。ただし、一部のアプリケーションでは、このようなモデルでは不十分であり、発生する可能性のある過渡現象を考慮する必要があります。TLA +は、並列非同期システムを分析するように設計されています。その助けを借りてパズルを解くことを練習したので、あなたはより深刻なタスクのためにこのツールを使い始めることができます。

RSトリガーは、再帰的にリンクされた2つのNOR要素（NOT OR）で構成されます。最初に、別のNORについて説明します。

NOR(a,b,c) == c' = ~ (a \/ b)

この要素は、回路内の3つのポイントを接続します。そのアクションは、NOT（OR（a、b））の次の瞬間にポイントcで信号を設定します。正確にこれがいつ起こるかはまだ示していません。

次に、2つのNORを1つのRSトリガーに結合します。

RS == (\/ (NOR(r, q, p) /\ UNCHANGED q)
       \/ (NOR(p, s, q) /\ UNCHANGED p)
       \/ (NOR(r, q, p) /\ NOR(p, s, q)))
      /\ UNCHANGED <<r,s>>

ここでは、電子回路の非同期性について明示的に説明します。2つのNORのいずれか、または両方を同時に機能させることができます。この場合、入力信号rとsは変化しないと仮定します。

入力にTRUEに等しいr信号とs信号の両方を送信しない場合、RSトリガーは正しく機能します。シミュレートされたデバイスが動作する回路は、この合意に従うものと想定しています。また、トランジェントがすでに完了しているときに入力信号の変化が発生すると仮定します。これは、同期回路でクロック周波数を選択するか、非同期回路で追加ロジックによって提供する必要があります。

Change == /\ p = ~ (r \/ q)
          /\ q = ~ (p \/ s)
          /\ IF r = FALSE /\ s = FALSE
             THEN \/ (r' = TRUE /\ s' = FALSE)
                  \/ (r' = FALSE /\ s' = TRUE) 
             ELSE r' = FALSE /\ s' = FALSE
          /\ UNCHANGED <<p,q>>

トリガーの定常状態が実行中の操作に対応していることを確認するために、変更前の信号値を保存する補助変数を導入します。

Check == /\ oldp' = p
         /\ oldq' = q
         /\ stable' = (r = FALSE /\ s = FALSE)

これで、遷移述語を定式化できます。

Next == Check /\ (RS \/ Change)

トリガーの定常状態の正確さに加えて、中間状態での出力の一貫性もチェックします-両方の出力が同時に値TRUEを受け入れることができないことを確認します（特に、これは、1つのトリガーの出力を別のトリガーに転送する権利があることを意味します）。

OutputOk == p /= TRUE \/ q /= TRUE

そして完全な不変式

Invariant == /\ (r = FALSE \/ s = FALSE)
             /\ (stable => oldq = q /\ oldp = p)
             /\ OutputOk

ただし、不変条件で記述できないもう1つの優れたプロパティは、トリガーのトランジェントが遅かれ早かれ終了することです。

RSok == /\ [] (r = TRUE ~> q = TRUE)
        /\ [] (s = TRUE ~> p = TRUE)

" r〜> q "は " r => <> q " の構文糖です。
" => "-論理演算子は「フォロー」します。
" <> "は時間演算子 "いつか"です。
「[]」は、「常に」時間演算子です。

時間演算子は不変条件での使用は禁止されていますが、PROPERTY（モデル全体のプロパティ）では使用できます。

RSokプロパティに小さな問題があります-実行されません！実際には、RSアクションを実行しても何も変更されない可能性があります。つまり、システムが仕様に基づいて永久にスピンできる状態グラフにループが形成されます。トリガーの前提条件を追加してこのループを削除することは難しくありません。

RS == (\/ ((p /= ~ (r \/ q)) /\ NOR(r, q, p) /\ UNCHANGED q)
       \/ ((q /= ~ (p \/ s)) /\ NOR(p, s, q) /\ UNCHANGED p)
       \/ (((p /= ~ (r \/ q)) /\ (q /= ~ (p \/ s)) /\ NOR(r, q, p) /\ NOR(p, s, q)))
      /\ UNCHANGED <<r,s>>

ただし、状況はさらに悪化します。モデルの一時的なプロパティを確認すると、TLA +は自動的に各ノードにループを追加します。これは、モデルを「構成」するために行われます。複数のモデルからモデルを構成する場合、1つのコンポーネントのアクションは、それに関連しないコンポーネントの状態で変更されません。したがって、コンポーネントのプロパティはループの存在に対して不変である必要があります。

どういうわけかそれと共存するために、TLA +は「公平性」をサポートしています。移行が可能であればいつかそれが起こると彼は言うことができます。

vars == <<r,s,p,q, stable, oldp, oldq>>
Spec == Init /\ [][Next]_vars /\ WF_vars(Next)
