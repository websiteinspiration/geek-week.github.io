現代世界における遺産の使用と保護



従来のインフラストラクチャは依然として多くの業界の企業の重要な部分です。さまざまなビジネスで、Solaris XPの古いサーバーで実行されるOracleデータベースで、Windows XPシステムをまだ使用している医療機関です。 Linux RHEL4システムを必要とするアプリケーション、および10年前に古くなったバージョンのWindowsシステムを備えたATM。多くの組織が依然としてWindows 2008システムでレガシーサーバーを使用している場合もあります。

継承されたインフラストラクチャは、データセンター、特に大企業では非常に一般的です。たとえば、AIX、医療機器、ショッピング用のターミナルシステムなど、AIXオペレーティングシステムを搭載した古いコンピューターが銀行やエンドポイントで大量のトランザクションデータを処理するための重要な操作をどのように実行するかをよく見ることができます。頻繁に使用されるオペレーティングシステムで、その耐用年数が長くなっています。このインフラストラクチャで使用されているアプリケーションのアップグレードは、継続的で難しいプロセスであり、通常は数年かかります。

安全でないレガシーシステムはデータセンターを危険にさらします


レガシーシステムの不適切な保護に関連する組織のリスクは非常に高く、これらのシステムのワークロードを超えています。たとえば、Windows XPシステムを実行しているパッチを適用していないデバイスを簡単に使用して、任意のデータセンターにアクセスできます。今月初めに、MicrosoftがWindows XPやWindows Server 2003などの古いオペレーティングシステムでリモートコード実行を許可する深刻なシステム脆弱性に対するセキュリティアップデートをリリースしたときに、同様のリスクの通知を受け取りました。

攻撃者がこのような保護されていないマシンにアクセスする場合（これは、最新の適切にパッチされたサーバーをハッキングするよりもはるかに簡単です）、横方向の動きを使用してネットワークにアクセスできます。データセンターがより高度になり、公的にアクセス可能なクラウドストレージサーバーを使用したり、「コンテナー」などの最新のテクノロジーを使用したりする可能性にまで拡大すると、ハッキングのリスクが高まります。異なるビジネスアプリケーション（継承されているかどうかに関係なく）間の相互依存性は、より複雑で動的になり、セキュリティの観点からトラフィックモデルの管理が困難になっています。これにより、攻撃者はインフラストラクチャのさまざまな部分を密かに移動できるようになります。

古いインフラストラクチャ、新しいリスク


レガシーシステムは何年にもわたって存在してきましたが、それらがもたらすセキュリティリスクは常に増大しています。組織がデジタル変革プロセスを経て、インフラストラクチャとデータセンターをアップグレードし、ハイブリッドクラウドストレージシステムに移行するにつれて、サイバー犯罪者は、ミッションクリティカルな内部アプリケーションにアクセスする機会が増えます。

以前はローカルにインストールされた少数のアプリケーションでのみ使用されていたレガシーシステムにローカルにインストールされたビジネスアプリケーションを、多数のオンプレミスアプリケーションとクラウドアプリケーションの両方で使用できるようになりました。アプリケーションと環境の数が増加しているレガシーシステムの使用により、潜在的なハッキングの領域が拡大しています。

だから問題は、このリスクをどのようにして減らすことができるかということです。新しいアプリケーションを今日のインフラストラクチャに迅速に展開できるようにしながら、レガシーでありながらビジネスクリティカルなコンポーネントのセキュリティをどのように維持しますか？

リスクの特定


最初のステップは、リスクを正しく特定して定量化することです。既存のインベントリシステムといわゆる「部族の知識」を使用するだけではおそらく十分ではありません。常に、環境の完全で正確な最新のビューを取得するよう努める必要があります。レガシーシステムの場合、組織内のこれらのシステムに関する知識は時間とともに減少する傾向があるため、適切な情報を取得することは特に困難な作業になる可能性があります。

セキュリティチームは、優れた分析および視覚化ツールを使用して、次の質問に答える計画を提供する必要があります。

  1. どのサーバーとエンドポイントがレガシーオペレーティングシステムを実行していますか？
  2. これらのワークロードはどの環境とビジネスアプリケーションに関連していますか？
  3. これらのワークロードは他のアプリケーションや環境とどのように相互作用しますか？どのポート？どのようなプロセスを使用していますか？どんなビジネス目的のために？

これらの重要な質問に答えることは、セキュリティリスクを減らすための出発点です。これらは、組織に最大のリスクをもたらすワークロード、攻撃者による攻撃中に損傷する可能性のあるビジネスプロセス、およびレガシーシステムと継承されていないシステムの間をクラウドストレージとデータセンターを通じて横に移動するときに攻撃者が使用できるネットワークルートを示します。継承されたマシンに予期しないデータストリームが送信されるのを見たり、データが突然送信されたりすると、ユーザーはしばしば驚かされ、セキュリティステータスとリスクについての質問が増えます。

優れた分析および視覚化ツールは、他の環境に移動する必要があるシステムを特定して分析するのにも役立ちます。最も重要なのは、情報フローのビジュアルマップにより、これらのリソースをセグメント化するための厳密なポリシーを簡単に計画および使用できることです。十分に計画されたポリシーは、これらの古いマシンがさらされるリスクを大幅に減らします。

マイクロセグメンテーションのリスク削減


ネットワークセグメンテーションは、データセンターとクラウドストレージのリスクを軽減する費用対効果の高い方法として広く使用されています。特に、ユーザーはマイクロセグメンテーションを使用して、攻撃者がワークロード、アプリケーション、および環境の間を横方向に移動する能力を大幅に制限する、堅固なモジュール式セキュリティシステムのポリシーを作成できます。

レガシーインフラストラクチャを使用すると、分析とマイクロセグメンテーションに適したツールの価値がさらに明確になります。 VLANなどの古いセグメンテーション手法は悪用が困難であり、多くの場合、同様のレガシーシステムをすべて1つのセグメントに配置し、1回のハッキングが発生した場合にグループ全体が攻撃にさらされることになります。さらに、レガシーVLANとデータセンターの他の部分との間のゲートウェイルールは維持が難しく、全体的なリスクを増大させる過剰認証ポリシーにつながります。レガシーワークロードと最新のワークロードの両方を適切に視覚化することで、セキュリティチームは、レガシーシステム間の特定のフローのみを許可するサーバーレベルのポリシーを計画できます。レガシー環境とモダン環境の間だけでなく、


マイクロセグメンテーションのソリューションを選択するときは、選択するソリューションがインフラストラクチャ全体で簡単に使用できることを確認してください。ソリューションは、データセンターまたはクラウドストレージのすべてのタイプのワークロードをカバーします。最新のアプリケーションをセグメント化し、レガシーシステムを無人のままにしておくと、インフラストラクチャに大きなセキュリティホールが残ります。

個人的には、セキュリティシステムプロバイダーは、顧客がこの増大する脅威に対処できるように支援するために、インフラストラクチャ全体をカバーするタスクを実行する必要があると考えています。一部のベンダーは最新のインフラストラクチャのみに焦点を当て、古いオペレーティングシステムのサポートを拒否していますが、優れた高度なセキュリティプラットフォームがインフラストラクチャのすべてのスペクトルをカバーする必要があると思います。


継承されたシステムは、組織にとって固有の問題を提起します。それらはビジネスにとって重要ですが、保守がより難しく、適切に保護されていません。組織がハイブリッドクラウドストレージに移行し、潜在的な攻撃の範囲を拡大するにつれて、レガシーアプリケーションを保護するために特別な注意を払う必要があります。これを行うには、セキュリティチームはレガシーサーバーを正確に特定し、他のアプリケーションや環境との相互依存関係を理解し​​、厳密なセグメンテーションポリシーを作成してリスクを制御する必要があります。主要なマイクロセグメンテーションプロバイダーは、他のタイプのインフラストラクチャを犠牲にすることなく、レガシーシステムをカバーできる必要があります。 Guardicore Centraプラットフォームは、分析、インフラストラクチャ全体の視覚化とマイクロセグメンテーション-新旧の機能により、死角を処理する負担を排除します。
