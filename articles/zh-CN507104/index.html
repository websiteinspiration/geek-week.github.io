导入方面的巨大分歧：用Typescript澄清导入中的不确定性

在课程“ React.js Developer”开始之前准备了文章的翻译。


我使用打字稿已经有一段时间了，在整理其模块和建议设置方面存在很多问题，我必须说，围绕它们确实有很多不可理解的地方。命名空间 import * as React from 'react'，esModuleInterop等等。因此，让我们理解为什么所有的炒作都出现了。

我不会将名称空间作为打字稿中的模块化系统来讨论，因为它不是最佳方法（特别是考虑当前的开发向量），并且现在没有人在使用它。

那么，事情在它们出现之前如何esModuleInterop？babel或浏览器拥有几乎所有相同的模块，以及命名的导入/导出。但是，在默认情况下的导出和导入方面，打字稿有自己的选择：有必要编写import * as React from 'react'（而不是import React from 'react'），当然，这里我们不仅在谈论react，而且还谈论从中的所有默认导入commonjs。这是怎么发生的？

为了理解这一点，让我们看看commonjs和模块中某些模式之间的兼容性是如何工作的es6。例如，我们有一个导出foo和bar作为键的模块：

module.exports = { foo, bar }

我们可以使用require和销毁来导入：

const { foo, bar } = require('my-module')

并使用命名的导入应用相同的原则（尽管，老实说，这不是解构）：

import { foo, bar } from 'my-module'

但是，commonjs中更常见的模式是const myModule = require('my-module')（因为尚未进行销毁），但是您如何使用它es6呢？

在开发导入规范时，es6重要的方面之一是与的兼容性commonjs，因为commonjs已经在上面编写了很多代码。这就是默认导入和导出的样子。是的，唯一的目标是确保与的兼容性commonjs，以便我们可以编写 import myModule from 'my-module并获得完全相同的结果。但是，从规范上看，这并不明显，此外，兼容性的实现是编译器开发人员的特权。这时发生了大分裂： import React from 'react'或者import * as React from 'react'-这就是问题。

为什么打字稿会选择后者？将您自己替换为transpiler开发人员，并问问自己，将进口货物从运es6到的最简单方法是什么commonjs？假设您具有以下一组进出口：

export const foo = 1
export const bar = 2
export default () => {}
import { foo } from 'module'
import func from 'module'`

因此，默认情况下，我们将使用js带有密钥的对象default进行导出！

module.exports = {
  foo: 1,
  bar: 2,
  default: () => {}
}
const module = require('module')
const foo = module.foo
const func = module.default

很酷，但是兼容性呢？如果默认情况下导入，则意味着我们使用一个具有名称的字段default，那么当我们编写import React from 'react'该字段时，它的意思是const { default: React } = require('react')，但它不起作用！然后改为尝试使用带星号的导入。现在，用户将不得不写import * as React from 'react'内容module.exports。

但是，的语义有所不同commonjs。
Commonjs就像普通的javascript，仅此而已。只是功能和对象，不需要任何要求。另一方面，在import中es6，它require现在是规范的一部分，因此myModule在这种情况下，这不仅是常规的javascript对象，而且是所谓的名称空间（不要与打字稿中的名称空间混淆），该名称空间因此具有某些属性。其中之一是无法调用名称空间。您可能会问，这是什么问题？

让我们尝试另一个模式commonjs，其中一个功能作为导出：

module.exports = function() { // do something }

我们可以使用require并执行它：

const foo = require('my-module')
foo()

尽管如果您尝试在具有ES6模块的规范投诉环境中执行此操作，则会收到错误消息：

import * as foo from 'my-module'
foo() // Error

这是因为名称空间与javascript对象不同，而是存储每个es6导出的单独结构。

但是Babel做到了这一点，并提供了一个兼容选项，我们可以在其中编写import React from 'react“，它将起作用。转换时，他在中为每个es6模块标记了一个特殊标志module.exports，以便我们理解，如果标志为true，则返回该标志module.exports，如果为false（例如，如果它是commonjs尚未转置的库），则需要包装当前导出，{ default: export }以便我们可以每次使用default（在这里看看）。

Typescript通过星号导入，但最终放弃并添加了一个选项esModuleInterop到编译器。通常，此选项与babel相同，如果启用它，则可以将常规导入编写为import React from 'react'，而打字稿将理解所有内容。

问题在于，尽管默认情况下在新项目中启用了它（在运行时tsc --init），但由于它不具有向后兼容性，因此它不适用于现有项目（即使您升级到TypeScript 3）。因此，您将不得不将不必要的星号导入重写为默认导入。 React会正常处理此问题，因为它仍然是命名导出的集合，但不包含名称空间调用。但是请不要担心导出类型是否一切正常（并且大多数情况下一切都正常，因为其中许多是自动修复的）），TypeScript 3将允许您快速将带有星号的导入转换为标准。

因此，我真的提倡使用该选项esModuleInterop，因为它不仅会允许您编写更少的代码并使其更易于阅读（例如，这些不只是单词，汇总将不允许您使用带有这样的星号的导入），而且还可以减少社区之间的分歧打字稿和通天塔。

注意：曾经有一个选项enableSyntheticDefaultImports可以在编译器默认抱怨导入错误时关闭编译器，因此我们需要我们自己的方式来确保与commonjs（例如，WebpackDefaultImportPlugin），但这是有问题的，因为例如，如果您进行了测试，则仍然需要确保这种兼容性。请注意，esModuleInterop默认情况下仅当目标为<=ES5 时才包括合成导入。因此，如果启用此选项，并且编译器继续抱怨它import React，然后了解您追求的目的，也许包括默认导入将是您的选择（或者重新启动vscode / webstorm，这是众所周知的）。

希望我的解释可以使情况有所澄清，但是如果您仍有问题，可以在Twitter上向我提问！
