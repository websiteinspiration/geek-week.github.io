RISC-Vの欠点

このドキュメントは、ARMの実行コア検証エンジニアとして数年前に最初に書いたものです。もちろん、私の意見は、さまざまなプロセッサのエグゼクティブコアを使用した詳細な作業に影響されました。だから割引でそれをしてください：多分私はあまりにもカテゴリー的です。

しかし、私はまだRISC-Vの作成者がもっと上手くできると信じています。一方、今日32ビットまたは64ビットのプロセッサを設計した場合、おそらく既存のツールを利用するために、そのようなアーキテクチャだけを実装したでしょう。

この記事では、RISC-V 2.0命令セットについて最初に説明しました。バージョン2.2では、いくつかの更新が行われました。

序文：個人的な意見


RISC-V命令セットは、最小限に抑えられています。命令数の最小化、コーディングの正規化などに多くの注意が払われています。このミニマリズムへの要望により、誤った直交性（遷移、呼び出し、戻りに同じ命令を再利用するなど）と、サイズと量の両方を増大させる必須の冗長性がもたらされました。指示。

たとえば、以下はCコードです。

int readidx(int *p, size_t idx)
{ return p[idx]; }

これは、非常に一般的な操作である配列のインデックス付けの単純なケースです。これはx86_64のコンパイルです。

mov eax, [rdi+rsi*4]
ret

またはARM：

ldr r0, [r0, r1, lsl #2]
bx lr // return

ただし、RISC-Vの場合、次のコードが必要です。

slli a1, a1, 2
add a0, a1, a1
lw a0, a0, 0
jalr r0, r1, 0 // return

簡素化RISC-Vは、より多くの命令を実行することにより、デコーダー（つまり、CPUフロントエンド）を簡素化します。しかし、パイプラインの幅のスケーリングは困難な問題ですが、わずかに（または強く）不規則な命令のデコードが適切に実装されています（主な困難は、命令の長さを判別することが難しい場合に発生します：これは、多数のプレフィックスを持つx86命令セットで特に顕著です）。

一連の命令の単純化を限界に近づけるべきではありません。レジスタメモリのシフトを伴うレジスタとレジスタの加算は、プログラムでは単純で非常に一般的な命令であり、プロセッサが効果的に実装するのは非常に簡単です。プロセッサが命令を直接実装できない場合は、命令をコンポーネントに分解することは比較的簡単です。これは、単純な操作のシーケンスをマージするよりもはるかに単純な問題です。

CISCプロセッサの「複雑な」特定の命令-複雑で、めったに使用されず、非効率的な命令-を、小さなシーケンスの操作を組み合わせるCISCおよびRISCプロセッサに共通の「機能的な」命令と区別する必要があります。後者は頻繁に使用され、高性能です。
