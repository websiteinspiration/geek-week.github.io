Migrações de banco de dados com Flyway

Uma tradução do artigo foi preparada antes do início do curso Java Developer .


1. Introdução


Este artigo descreve os principais conceitos do Flyway e um exemplo do uso dessa estrutura para alterar continuamente o esquema do banco de dados usando o banco de dados H2 na memória como um exemplo usando o plug-in do flyway maven.

O Flyway atualiza as versões do banco de dados usando migrações. As migrações podem ser gravadas em SQL (com sintaxe específica para um DBMS específico) ou em Java.

As migrações podem ser versionadas ou recorrentes. Os primeiros têm uma versão única e são aplicados exatamente uma vez. Os segundos não possuem um número de versão e são aplicados quando a soma de verificação é alterada.

Migrações repetidas em uma única execução sempre são aplicadas após a conclusão das migrações com versão. Migrações repetidas são aplicadas na ordem em que são descritas. Em uma migração, todas as operações são executadas em uma única transação do banco de dados.

Neste artigo, focaremos o uso do plug-in maven para migrações de banco de dados.

2. Plug-in do Flyway Maven


Adicione o plugin flyway maven a pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
</plugin>

A versão atual do plug-in pode ser visualizada no Maven Central .
Uma lista de parâmetros do plug-in pode ser encontrada na documentação . Os parâmetros do plug-in podem ser configurados de quatro maneiras diferentes.

2.1 Seção <configuration>Plugin Os
parâmetros podem ser especificados diretamente na tag na seção plugin em pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
    <configuration>
        <user>databaseUser</user>
        <password>databasePassword</password>
        <schemas>
            <schema>schemaName</schema>
        </schemas>
        ...
    </configuration>
</plugin>

2.2 Propriedades de Maven


Você também pode configurar o plug-in especificando parâmetros em <properties>:

<project>
    ...
    <properties>
        <flyway.user>databaseUser</flyway.user>
        <flyway.password>databasePassword</flyway.password>
        <flyway.schemas>schemaName</flyway.schemas>
        ...
    </properties>
    ...
</project>

2.3 Arquivo de configuração externa


Ou descreva a configuração em um .propertiesarquivo separado :

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=schemaName
...

O nome padrão é o arquivo de configuração flyway.properties. Este arquivo deve estar no mesmo diretório que o arquivo pom.xml. A codificação é configurada no parâmetro flyway.encoding(o padrão é UTF-8).

Se você usar um nome diferente para o arquivo (por exemplo customConfig.properties), precisará especificá-lo explicitamente ao chamar maven:

$ mvn -Dflyway.configFile=customConfig.properties

2.4 Propriedades do sistema


Finalmente, os parâmetros podem ser especificados como propriedades do sistema ao chamar o maven a partir da linha de comando:

$ mvn -Dflyway.user=databaseUser -Dflyway.password=databasePassword
  -Dflyway.schemas=schemaName

Se a configuração for especificada de várias maneiras, a prioridade será a seguinte:

  1. Propriedades do sistema
  2. Arquivo de configuração externa
  3. <propriedades da seção>
  4. Seção de <configuração do >plug-in

3. Exemplo de Migração


Nesta seção, consideraremos as etapas necessárias para migrar um esquema de banco de dados usando o banco de dados H2 na memória como um exemplo usando o plug-in maven. Para configurar o Flyway, usaremos um arquivo externo.

3.1 Alterações no POM


Para começar, adicione uma dependência no H2:

<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>1.4.196</version>
</dependency>

Aqui também podemos verificar a versão mais recente do driver disponível no Maven Central . Adicione um plugin para o Flyway, conforme descrito anteriormente.

3.2 Configurar o Flyway em um arquivo externo


Crie um $PROJECT_ROOTarquivo myFlywayConfig.propertiescom o seguinte conteúdo:

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=app-db
flyway.url=jdbc:h2:mem:DATABASE
flyway.locations=filesystem:db/migration

A configuração acima indica que os scripts de migração estão no diretório db/migratione para se conectar ao banco de dados usando H2 databaseUsere databasePassword.

Esquema do banco de dados para aplicativo app-db.

Obviamente, nos parâmetros flyway.user, flyway.passworde flyway.urlvocê deve especificar o nome, senha e URL do seu banco de dados.

3.3 Primeira migração


Sob convenções Flyway, os nomes dos scripts de migração devem estar no seguinte formato:

<Prefix<code>><</code>Version>__<code><</code>Description>.sql

Onde:

  • <Prefixo> - prefixo. Para migrações com versão, o padrão é "V". O prefixo é configurado através da propriedade flyway.sqlMigrationPrefix.
  • <Versão> - número da versão da migração. As versões principais e secundárias podem ser separadas por sublinhado. A versão deve sempre começar com 1.
  • <Descrição> - uma descrição textual da migração. A descrição deve ser separada do número da versão por dois sublinhados.

Exemplo: V1_1_0__my_first_migration.sql

Então, vamos criar um diretório db/migrationna $PROJECT_ROOTmigração de script V1_0__create_employee_schema.sqle SQL para criar a tabela employee:

CREATE TABLE IF NOT EXISTS `employee` (
 
    `id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name` varchar(20),
    `email` varchar(50),
    `date_of_birth` timestamp
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;

3.4 Realizamos migrações


Em seguida, $PROJECT_ROOTexecute o seguinte comando maven para aplicar as migrações do banco de dados:

$ mvn clean flyway:migrate -Dflyway.configFile=myFlywayConfig.properties

Nossa primeira migração deve ser concluída.

Agora, o esquema do banco de dados fica assim:
employee:

+----+------+-------+---------------+
| id | name | email | date_of_birth |
+----+------+-------+---------------+

Podemos repetir as etapas anteriores para realizar outras migrações.

3.5 Segunda migração


Para a segunda migração, crie um arquivo com o nome V2_0_create_department_schema.sqlcontendo as duas solicitações a seguir:

CREATE TABLE IF NOT EXISTS `department` (
 
`id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
`name` varchar(20)
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;
 
ALTER TABLE `employee` ADD `dept_id` int AFTER `email`;

Vamos executar a migração, exatamente como fizemos na primeira migração.

Agora, o esquema do nosso banco de dados mudou: employeeuma nova coluna foi adicionada e uma nova tabela foi criada department:

employee:
+----+------+-------+---------+---------------+
| id | name | email | dept_id | date_of_birth |
+----+------+-------+---------+---------------+

department:

+----+------+
| id | name |
+----+------+

Para verificar se as duas migrações foram bem-sucedidas, execute o seguinte comando maven:

$ mvn flyway:info -Dflyway.configFile=myFlywayConfig.properties

4. Desabilitando o Flyway na Inicialização da Primavera


Às vezes, pode ser necessário desativar as migrações do Flyway.

Isso pode ser necessário durante os testes quando um esquema de banco de dados é gerado com base em entidades. Nesse caso, você pode desativar o Flyway para o perfil de teste.

No Spring Boot, isso é muito simples.

4.1 Spring boot 1.x


Tudo o que precisamos fazer é definir a propriedade flyway.enabledno arquivo application-test.properties:

flyway.enabled=false


4.2 Spring boot 2.x


Nas versões posteriores do Spring Boot, essa propriedade foi alterada para 

spring.flyway.enabled:
spring.flyway.enabled=false

4.3 FlywayMigrationStrategy vazio


Se queremos desativar apenas a migração automática do Flyway na inicialização, mas queremos iniciar as migrações manualmente, o uso das propriedades acima não funcionará para nós.

Isso se deve ao fato de o Spring Boot não configurar automaticamente os compartimentos Flyway e, portanto, teremos que configurá-los nós mesmos, o que não é muito conveniente.

Nesse caso, podemos deixar o Flyway ativado e implementar um FlywayMigrationStrategy vazio :

@Configuration
public class EmptyMigrationStrategyConfig {
 
    @Bean
    public FlywayMigrationStrategy flywayMigrationStrategy() {
        return flyway -> {
            // do nothing 
        };
    }
}

De fato, isso desativará as migrações do Flyway quando o aplicativo for iniciado.

Mas ainda podemos iniciar a migração manualmente:

@RunWith(SpringRunner.class)
@SpringBootTest
public class ManualFlywayMigrationIntegrationTest {
 
    @Autowired
    private Flyway flyway;
 
    @Test
    public void skipAutomaticAndTriggerManualFlywayMigration() {
        flyway.migrate();
    }
}

5. Como o Flyway funciona


Para rastrear quando, por quem e quais migrações foram aplicadas, uma tabela especial com metadados é adicionada ao esquema do banco de dados. Esta tabela também armazena as somas de verificação das migrações e informações sobre se a migração foi bem-sucedida ou não.

A estrutura funciona da seguinte maneira:

  1. Verifica o esquema do banco de dados em busca de uma tabela de metadados (padrão SCHEMA_VERSION). Se a tabela de metadados não existir, ela será criada.
  2. Verifica o caminho de classe em busca de migrações disponíveis.
  3. Compara migrações para uma tabela de metadados. Se o número da versão for menor ou igual à versão marcada como atual, ele será ignorado.
  4. Marca todas as migrações restantes como pendentes. Em seguida, classifica-os em números de versão ascendentes e os executa na ordem especificada.
  5. À medida que as migrações se aplicam, atualiza a tabela de metadados.

6. Equipes


A Flyway possui as seguintes principais equipes de gerenciamento de migração:

  • Info . Exibe o status / versão atual do esquema do banco de dados. Informações sobre quais migrações são esperadas, quais foram aplicadas, o status das migrações concluídas e a data de sua conclusão.
  • Migrar . Atualizando o esquema do banco de dados para a versão atual. Examine um caminho de classe para encontrar as migrações disponíveis e aplicar as migrações pendentes.
  • Linha de base . Instalando a versão do esquema do banco de dados, ignorando migrações até e incluindo baselineVersion. A linha de base ajuda você a usar o Flyway em um banco de dados existente. Novas migrações são aplicadas como de costume.
  • A validação . Verificando o esquema atual do banco de dados em relação às migrações disponíveis.
  • Reparação . Restaurar tabela de metadados.
