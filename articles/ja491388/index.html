そのままjQueryからSvelteへ

みなさん、こんにちは。

これは、記事「jQueryからSvelteへの移行、苦労なし」の続きです。

以下、私が遭遇した困難について話します。それらの多くはなく、コミュニティのサポートなしでは対処できないほど根本的なものは1つだけでした。

前書き


フロントエンドをバラバラに書き直すつもりでしたが、まったくうまくいかなかったわけではなく、完全にうまくいかなかったので、大きなバラバラに書き直さなければなりませんでした。

まず、jQueryアプローチは必須であるため、Svelteアプローチは宣言型です。

次に、jQueryを使用するため、スコープは常にグローバルであるため、コードの任意の行からWebページのすべての要素にアクセスでき、IDまたはCSSセレクターによってそれらにアクセスできます。コンポーネント自体のみであり、外部要素や内部要素はなく、直接アクセスすることはできません。

Svelteを使用すると、実際のOOPを取得できます。自分で変更を加えることはできません。変更の必要性についてコンポーネントに伝えることができるだけです。これらの変更がどのように行われるかは、コンポーネント内のコードのみを知っています。

そしてそれは素晴らしいです:)

Svelteには、コンポーネントと通信するための2つのメカニズムがあります。

バインド変数（バインド、マッピング）


変数を宣言し、それをコンポーネント属性にマッピングします。

<script>
    import DetailTaskView from './DetailTaskView.svelte';
    let task = {};
    let isModal = false;
    function renderTask(data) {
        const isSuccess = data.hasOwnProperty(0);
        if (isSuccess) {
            isModal = true;
            task = data[0];
        }
    }
    import {fade} from 'svelte/transition';
</script>
{#if isModal}
    <div transition:fade>
        <DetailTaskView bind:isModal="{isModal}" {...task}/>
    </div>
{/if}

私たちは何をしましたか？


2つのローカル変数「task」と「isModal」を宣言しました。「task」はコンポーネントに表示される情報であり、データは表示されるだけで、変更されません。「isModal」はコンポーネントの可視性のフラグです。ユーザーがコンポーネントの十字をクリックすると、コンポーネントが非表示になります。 、クロスはコンポーネントに属しているため、クリックについては何も知りませんが、「isModal」変数の値が変更されたことがわかり、反応性のおかげで、この新しい値を計算します。

双方向バインディングが必要な場合は、「bind：」と記述します。コンポーネント内の値の変更は、「親」コンポーネントに報告されます。

コンポーネントに値を伝えるだけでよく、コンポーネントの属性名が変数名と一致する場合は、省略形を使用できます。 "{task}"と書くか、デストラクタ "{... task}"を使用できます。

便利です。

ただし、1つのコンポーネントに別のコンポーネントが埋め込まれていて、3番目のコンポーネントもある場合は、もちろん、ネスト階層を上下にスクロールするためのボラープレートがあります。

イベントバブリング


私は専門用語と間違えられるかもしれませんが、あまりキックしないでください。

親コンポーネントは子コンポーネントのイベントを処理できますが、子コンポーネントが報告するイベントのみを処理できます。

<!-- App.svelte   -->
<script>
    import SearchForm from './SearchForm.svelte';
    async function applySample(event) {
        const sample = event.detail.sample;
        if(sample){
            search(sample);
        }
        if(!sample){
            renderPage();
        }
    }
</script>
<div class="container">
    <h1>  </h1>
    <SearchForm on:search={applySample}/>
</div>

<!-- SearchForm.svelte    -->
<script>
	import { createEventDispatcher } from 'svelte';
	const dispatch = createEventDispatcher();
	let sample = '';
    function search(event) {
        event.preventDefault();
        dispatch('search', {
            sample: sample
        });
    }
</script>
<form class="form-horizontal" id="search"
on:submit={search}>
    <div class="form-group">
        <label class="control-label col-sm-2" for="sample">
        
        </label>
        <div class="col-sm-10">
            <input id="sample" class="form-control" type="search"
             placeholder="  "
             autocomplete="on" bind:value={sample}
             />
        </div>
    </div>
</form>

何が起きてる ？


親コンポーネントでは、「applySample」関数は子「SearchForm」コンポーネントの「on：search」イベントによって実行され、この関数はイベントオブジェクトからパラメーター（event.detail）を受け取って処理します。

コンポーネントはどうなりますか？


input要素の「value」属性が「sample」変数にマップされ、「on：submit」イベント（「form」要素の）によって「search」関数が実行され、「search」イベントが発生し、{sample：サンプルオブジェクトが「detail」プロパティに書き込まれます。 }-つまり、検索文字列の値です。

したがって、検索文字列の値は親コンポーネントに渡され、この値の処理方法を決定するのは彼です。

コンポーネントは入力フォームの表示と入力値の送信のみを担当し、コンポーネントは検索を実装して結果を表示しないため、責任を共有します。

美しさ！

命令型から宣言型への移行


ここでは、残念ながら、違いをはっきりと示すことはできません。つまり、次のように聞こえます。jQueryを使用するときに、htmlマークアップを作成して適切な場所に挿入した場合、Svelteを使用してコンポーネントの属性を含む配列を生成し、ループで事前計算された属性を使用してコンポーネントを追加します。

<!-- Paging.svelte        -->
<script>
    import {createEventDispatcher} from 'svelte';
    import OrdinalPlace from './OrdinalPlace.svelte';
    let pagingPlaces;
    //         pagingPlaces
    function addPlace(
        paging = [], index = Number.NEGATIVE_INFINITY,
        text = "", css = "") {
        paging.push({index:index,text:text,css:css});

        return paging;
    }
    const dispatch = createEventDispatcher();
    function browsePage(event) {
        const pageIndex = event.detail.index;
        dispatch('move', {
            index: pageIndex
        });
    }
</script>

{#if pagingPlaces.length}
    <table class = "table table-hover-cells table-bordered">
        <tbody>
            <tr>
            {#each pagingPlaces as place (place.index)}
                <OrdinalPlace on:move="{browsePage}"
                {...place}>
                </OrdinalPlace>
            {/each}
            </tr>
        </tbody>
    </table>
{/if}

<!-- OrdinalPlace.svelte     ""      -->
<script>
    export let index = -1;
    export let text = "";
    export let css = "";

    let number = index +1;
    function skip() {
        return !(text === "");
    }

    let letSkip = skip();
    let noSkip = !letSkip;

    import { createEventDispatcher } from "svelte";
    const dispatch = createEventDispatcher();
    function moveTo() {
        if(noSkip){
            dispatch("move", {index:index});
        }
    }
</script>
<td class="{css}" on:click="{moveTo}">
    {#if letSkip}
        {text}
    {/if}
    {#if noSkip}
        {number}
    {/if}
</td>

使い方 ？


Pagingコンポーネントを作成するとき、特定のページに移動するための「要素」の配列「pagingPlaces」を形成し、次にすべての要素を循環してコンポーネントを挿入し、1つのページング位置「OrdinalPlace」を表示します。

繰り返しになりますが、宣言型のアプローチでは、各位置を自分で形成するのではなく、そのような属性を持つ位置を表示する必要があることをコンポーネントに伝えます。

ここでは、イベントの発生の混乱したケースを示します。検索結果ページに移動するには、ユーザーが「OrdinalPlace」コンポーネントをクリックすると、このコンポーネントはページを読み込めないため、ページインデックスパラメーターを使用して「move」イベントを作成し、このイベントは親コンポーネント「Paging」を取得します。これにより、ページも読み込めません。 'move'イベントを発生させ、次の親コンポーネントがすでにそれを取得して何らかの方法で処理しています。

Svelteとコンポーネントアプローチにより、責任を共有し、SOLIDに従うことが求められます。

最大の待ち伏せ


上記の例は、ヒントがなければ処理できない基本的な問題の解決策を示しています。Svelteはすべてのコンポーネントをキャッシュするので、Svelteがこれらのコンポーネントの変更を追跡できるようにする必要があります。

問題のコードは次のとおりです。

            {#each pagingPlaces as place (place.index)}
                <OrdinalPlace on:move="{browsePage}"
                {...place}>
                </OrdinalPlace>
            {/each}

ページングでページのリストを表示するために、配列を実行し、Svelteがコンポーネントに各コンポーネントに配列インデックスを割り当てました。ここで、Svelteはこのインデックスに基づいてコンポーネントを再描画することを決定します。 、私は一日理解するように努めました、そして私はホールに助けを求めました、そしてホールで私はすぐにこの熊手に精通している人を見つけませんでした、しかし彼らは人々のおかげで私を助けました。

配列を操作するときは、次の点に注意してください。配列を通過する場合は、やはりインデックスを使用する必要があります。

            {#each pagingPlaces as place (place.index)}

「PagingPlaces as place（place.index）」-必ず使用してください。

もちろん、以前にReact / Vueを使用したことがある場合は、おそらくこの機能にすでに慣れていることでしょう。

視覚効果


私のアプリケーションはモーダルウィンドウを使用しました。このためのjQueryはマークアップ要件を設定します。これがないと、jQuery.modal（）メソッドは機能しません。

Svelteはこれを簡単にします：

{#if isModal}
    <div transition:fade>
        <DetailTaskView bind:isModal="{isModal}" {...task}/>
    </div>
{/if}

具体的には、「遷移：フェード」は、ページ上の要素の非表示/外観を担当します。
どのマークアップを使用するかはだれも指示しません。

それは良いです。

このアニメーションに加えて、スヴェルトはカップルより多く持って飛ぶとトゥイーン、チュートリアルのリンクから例を。

その他の


変数名


変数/パラメーター/属性の命名に関する問題、オブジェクトのプロパティとそこに書き込む変数の両方を呼び出すには1つの単語を使用する必要があります。電話でコードについて話す必要がある場合は、ハンドセットルールを使用する必要があります。そのため、最後で混乱することがなく、繰り返される名前などすべてを理解できます。違反。

アヤックス


これはSvelteには適用されませんが、jQueryの使用の拒否に関係しています。jQuery.ajaxはfetch（）で置き換えることができます。このような置き換えを行ったので、リポジトリで確認できます。

結論


jQueryの使用からSvelteの使用に移行するには、マークアップを作成するためのロジックを書き直す必要がありますが、特にコードがこれで罪を犯さない場合は、それほど難しくはなく、思えるかもしれません。

Svelteを使用すると、マークアップが簡素化され、JSコードが短縮されます。Svelteを使用すると、コードが再利用可能になり、ランダムエラーに耐性があります。

Svelteを使用してください。それはあなたとあなたの顧客にとって良いことです！
