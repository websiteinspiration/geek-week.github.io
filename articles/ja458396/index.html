サーバー側レンダリングでVue.jsでの開発を便利にする方法

みなさん、こんにちは！

少し背景から始めましょう。

私はVue.jsで新しいプロジェクトを試すことにしました。サーバー側レンダリング（SSR）、CSSモジュール、コード分割、その他の便利な機能が必要でした。もちろん、開発の生産性を向上させるには、ホットリブート（HMR）が必要でした。

Nuxt.jsなどの既製のソリューションを使用したくありませんでした。プロジェクトが成長するとき、カスタマイズできることが重要です。そして、すべての高レベルのソリューションは、原則として、これを行うこと、または与えることを許可しませんが、多大な努力を伴います（ReactにNext.jsを使用して同様の経験がありました）。

サーバーレンダリングとホットリブートを使用する場合のローカル開発の主な問題は、1つのwebpack-dev-serverを実行するだけでは不十分だったことです。。また、Node.jsが起動するソースを使用して何かを行う必要があります。そうしないと、次にページを再読み込みするときに、サーバーでは更新されなかったがクライアントでは更新されたコードが取得されます。

ドキュメントとインターネットに突入したので、残念ながら、既製の適切に機能する例とテンプレートが見つかりませんでした。したがって、私は自分で作成しました。



私は、あなたが快適な開発をリードできるように、私のテンプレートが何であるかを決めました：

  • Vuejs
  • SSR
  • Vuex
  • CSSモジュール
  • コード分​​割
  • ESLint、Prettier

ローカル開発では、これらすべてをブラウザで即座に更新し、サーバーコードも更新する必要があります。

本番モードでは、バンドルを縮小し、静的をキャッシュするためにハッシュを追加し、バンドルへのパスをHTMLテンプレートに自動的に設定する必要があります。

これはすべてGitHubのリポジトリに実装されています。コードを提供し、解決策を説明します。

Vue.jsにはサーバーレンダリングを設定するための非常に包括的なドキュメントがあるので、そこを見るのは理にかなっています。

サーバ側


したがって、Node.jsのサーバーとしてExpressを使用します。また、vue-server-rendererも必要です。このパッケージを使用すると、サーバーバンドル、html-template、およびクライアントマニフェストに基づいて、コードをhtml-stringにレンダリングできます。リソースの名前とパスがここに示されます。server.js

ファイルは、最終的には次のようになります。

const path = require('path');

const express = require('express');

const { createBundleRenderer } = require('vue-server-renderer');

const template = require('fs').readFileSync(
  path.join(__dirname, './templates/index.html'),
  'utf-8',
);

const serverBundle = require('../dist/vue-ssr-server-bundle.json');
const clientManifest = require('../dist/vue-ssr-client-manifest.json');

const server = express();

const renderer = createBundleRenderer(serverBundle, {
  //           ,    
  runInNewContext: false,
  template,
  clientManifest,
  inject: false,
});

//         nginx
server.use('/dist', express.static(path.join(__dirname, '../dist')));

server.get('*', (req, res) => {
  const context = { url: req.url };

  renderer.renderToString(context, (err, html) => {
    if (err) {
      if (+err.message === 404) {
        res.status(404).end('Page not found');
      } else {
        console.log(err);
        res.status(500).end('Internal Server Error');
      }
    }

    res.end(html);
  });
});

server.listen(process.env.PORT || 3000);

ご覧のとおり、vue-ssr-server-bundle.jsonとvue-ssr-client-manifest.jsonの 2つのファイルを使用しています。

これらは、アプリケーションのビルド時に生成されます。1つ目はサーバーで実行されるコードで、2つ目はリソースの名前とパスを含みます。

また、createBundleRendererオプションで、inject：falseパラメータを指定しました。つまり、リソースやその他のものをロードするためのhtmlコードの自動生成はありません。完全な柔軟性が必要です。テンプレートでは、このコードを表示する場所を個別にマークします。

テンプレート自体は次のようになります。

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    {{{ meta.inject().title.text() }}}
    {{{ meta.inject().meta.text() }}}
    {{{ renderResourceHints() }}}
    {{{ renderStyles() }}}
</head>
<body>
<div id="app"><!--vue-ssr-outlet--></div>
{{{ renderState() }}}
{{{ renderScripts() }}}
</body>
</html>

さらに詳しく考えてみましょう。

  • meta.inject().title.text() meta.inject().meta.text() -. vue-meta,
  • renderResourceHints() — rel=«preload/prefetch» ,
  • renderStyles() — ,
  • renderState() — , window.__INITIAL_STATE__
  • renderScripts() — ,

コメントの代わりに、アプリケーションのマークアップが置き換えられます。彼は必要です。

Vueサーバー側アプリケーションへのエントリポイントはentry-server.jsファイルです。

import { createApp } from './app';

export default context =>
  new Promise((resolve, reject) => { 
    //      Vue
    const { app, router, store } = createApp();

    // $meta - ,   vue-meta   Vue
    const meta = app.$meta();
    
    //     
    router.push(context.url);
    
    //  -  ,     
    context.meta = meta;

    router.onReady(() => {
      context.rendered = () => {
        //    ,     ,  window.__INITIAL_STATE__
        context.state = store.state;
      };

      const matchedComponents = router.getMatchedComponents();
      //    
      if (!matchedComponents.length) {
        return reject(new Error(404));
      }

      return resolve(app);
    }, reject);
  });

クライアント部分


クライアント側のエントリポイントはentry-client.jsファイルです。

import { createApp } from './app';

const { app, router, store } = createApp();

router.onReady(() => {
  if (window.__INITIAL_STATE__) {
    //    ,    
    store.replaceState(window.__INITIAL_STATE__);
  }

  app.$mount('#app');
});

//    HMR  ,  webpack-dev-server     hot
if (module.hot) {
  const api = require('vue-hot-reload-api');
  const Vue = require('vue');

  api.install(Vue);

  if (!api.compatible) {
    throw new Error(
      'vue-hot-reload-api is not compatible with the version of Vue you are using.',
    );
  }

  module.hot.accept();
}

でapp.js、私たちVueのインスタンスは、サーバーとクライアントの両方で使用され、作成されます。

import Vue from 'vue';
import { sync } from 'vuex-router-sync';

import { createRouter } from './router';
import { createStore } from './client/store';

import App from './App.vue';

export function createApp() {
  const router = createRouter();
  const store = createStore();

  sync(store, router);
  
  const app = new Vue({
    router,
    store,
    render: h => h(App),
  });

  return { app, router, store };
}

複数のリクエストが同じインスタンスを使用する状況を回避するために、常に新しいインスタンスを作成します。

App.vueは、ルートに応じて必要なコンポーネントを置き換える<router-view> </ router-view>ディレクティブを含むルートコンポーネントです。

ルーター自体は次のようになります

import Vue from 'vue';
import Router from 'vue-router';
import VueMeta from 'vue-meta';

import routes from './routes';

Vue.use(Router);
Vue.use(VueMeta);

export function createRouter() {
  return new Router({
    mode: 'history',
    routes: [
      { path: routes.pages.main, component: () => import('./client/components/Main.vue') },
      { path: routes.pages.about, component: () => import('./client/components/About.vue') },
    ],
  });
}

スルーVue.use：我々は2つのプラグイン接続ルータとVueMetaを。
ルートでは、コンポーネント自体は直接表示されませんが、

() => import('./client/components/About.vue')

これはコード分割に必要です。

状態管理（Vuexによって実装）に関しては、その構成は特別なものではありません。唯一のことは、サイドをモジュールに分割し、名前付きの定数を使用して、コードでナビゲートしやすくすることです。

次に、Vueコンポーネント自体のいくつかのニュアンスを検討します。metaInfo

プロパティは、vue-metaパッケージを使用してメタデータをレンダリングします。さまざまなパラメータを多数指定できます（more）。

metaInfo: {
    title: 'Main page',
}

コンポーネントには、サーバー側でのみ実行されるメソッドがあります。

serverPrefetch() {
    console.log('Run only on server');
}

また、CSSモジュールを使用したいと思いました。コンポーネント間で重複しないようにクラスの名前を気にする必要がない場合のアイデアが気に入っています。CSSモジュールを使用すると、結果のクラスは<クラス名> _ <ハッシュ>のようになります。

これを行うには、コンポーネントでスタイルモジュールを指定する必要があります。

<style module>
.item {
  padding: 3px 0;
}

.controls {
  margin-top: 12px;
}
</style>

テンプレートで属性を指定します：クラス

<div :class="$style.item"></div>

また、モジュールを使用することをwebpack設定で指定する必要があります。

アセンブリ


それでは、webpackの設定に移りましょう。

サーバー部分とクライアント部分の構成によって継承される基本構成があります。

const webpack = require('webpack');
const OptimizeCSSAssetsPlugin = require('optimize-css-assets-webpack-plugin');
const UglifyJsPlugin = require('uglifyjs-webpack-plugin');
const merge = require('webpack-merge');

const VueLoaderPlugin = require('vue-loader/lib/plugin');

const isProduction = process.env.NODE_ENV === 'production';

let config = {

最初は、すべてのCSS処理が基本構成に移動されました。クライアントとサーバーの両方で必要です。生産体制の縮小がありました。
しかし、サーバーがドキュメントであることが判明し、それに応じてエラーが発生するという問題に遭遇しました。これは、サーバーとクライアントのCSS処理を分割することで修正された mini-css-extract-pluginエラーであることが判明しました。

VueSSRServerPluginは、サーバーで実行されるコードを示すvue-ssr-server-bundle.jsonファイルを生成します。

ここで、クライアント構成を検討します。

const webpack = require('webpack');
const merge = require('webpack-merge');
const VueSSRClientPlugin = require('vue-server-renderer/client-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');
const path = require('path');

const baseConfig = require('./webpack.base.js');

const isProduction = process.env.NODE_ENV === 'production';

let config = merge(baseConfig, {
  entry: ['./app/entry-client.js'],
  plugins: [new VueSSRClientPlugin()],
  output: {
    path: path.resolve('./dist/'),
    filename: '[name].[hash:8].js',
    publicPath: '/dist/',
  },
  module: {
    rules: [
      {
        test: /\.css$/,
        use: [
          isProduction ? MiniCssExtractPlugin.loader : 'vue-style-loader',
          {
            loader: 'css-loader',
            options: {
              modules: {
                localIdentName: '[local]_[hash:base64:8]',
              },
            },
          },
        ],
      },
    ],
  },
});

if (!isProduction) {
  config = merge(config, {
    output: {
      filename: '[name].js',
      publicPath: 'http://localhost:9999/dist/',
    },
    plugins: [new webpack.HotModuleReplacementPlugin()],
    devtool: 'source-map',
    devServer: {
      writeToDisk: true,
      contentBase: path.resolve(__dirname, 'dist'),
      publicPath: 'http://localhost:9999/dist/',
      hot: true,
      inline: true,
      historyApiFallback: true,
      port: 9999,
      headers: {
        'Access-Control-Allow-Origin': '*',
      },
    },
  });
} else {
  config = merge(config, {
    plugins: [
      new MiniCssExtractPlugin({
        filename: '[name].[hash:8].css',
      }),
    ],
  });
}

module.exports = config;

注目に値することから、ローカル開発では、webPath-dev-serverを参照するpublicPathを指定し、ハッシュなしでファイル名を生成します。また、devServerには、writeToDisk：trueパラメータを指定します。 ここで説明が必要です。 デフォルトでは、webpack-dev-serverはリソースをディスクに書き込まずにRAMから分散します。この場合、クライアントマニフェスト（vue-ssr-client-manifest.json



）、ディスク上にあるため、無関係なリソースが示されます。更新されません。これを回避するために、ディスクに変更を書き込むように開発サーバーに指示します。この場合、クライアントマニフェストが更新され、必要なリソースが強化されます。

実際、将来的にはこれを解消したいと考えています。 1つの解決策は乙女です。/ distディレクトリからではなく、サーバーのURLからマニフェストに接続するためのserver.jsのモード。ただし、この場合は非同期操作になります。私はコメントで問題の素晴らしい解決策があればうれしいです。 Nodemonはサーバー側の再読み込みを担当し、2つのファイルを監視します：dist / vue-ssr-server-bundle.jsonとapp / server.js

それらを変更すると、アプリケーションが再起動します。server.jsを

変更したときにアプリケーションを再起動できるようにするために、このファイルをnodemonのエントリポイントとして指定するのではなく、server.jsを接続するnodemon.jsファイルを作成します。そして、nodemon.jsファイルがエントリポイントになります。 プロダクションモードでは、app / server.jsがエントリポイントになります。



結論


合計すると、設定といくつかのコマンドを含むリポジトリがあります。

ローカル開発の場合：

yarn run dev

クライアント側：webpack-dev-serverを起動し、Vueコンポーネントと単純なコードの変更を監視し、devサーバーへのパスを含むクライアントマニフェストを生成し、それをディスクに保存してコードを更新し、ブラウザーのスタイルをオンザフライで更新します。

サーバー側：監視モードでwebpackを起動し、サーバーバンドル（vue-ssr-server-bundle.json）を収集し、変更時にアプリケーションを再起動します。

この場合、コードはクライアントとサーバーで常に自動的に変更されます。
最初の起動時に、サーバーバンドルが見つからなかったというエラーが発生する場合があります。これは正常です。コマンドを再起動するだけです。

組立生産の場合：

yarn run build

クライアント側：jsとcssを収集して縮小し、名前にハッシュを追加して、相対リソースパスを持つクライアントマニフェストを生成します。

サーバー側から：サーバーバンドルを収集します。

また、server.jsを起動する別のyarn run start-nodeコマンドを作成しましたが、これは例としてのみ実行されます。本番アプリケーションで起動するには、PM2などのプロセスマネージャーを使用する価値があります。 説明した経験が、快適な作業のためのエコシステムを迅速にセットアップし、機能の開発に集中するのに役立つことを願っています。
