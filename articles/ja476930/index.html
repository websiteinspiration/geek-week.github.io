Dart（Flutter）の拡張機能

Dart 2.6の最近のリリースでは、この言語は新しい関数、静的拡張または静的拡張メソッドを導入しました。これにより、既存の型に新しいメソッドを追加できます。なぜ延長が必要なのですか？それらを使用する方法とそれらは何に適していますか？



前書き


まず、一般的に拡張とは何ですか？拡張は、クラス宣言モジュールとは異なる場所で既存のクラスを拡張する構文糖衣です。

プログラミングでは、拡張メソッドが古くから存在していたため、ダーツを使用するようになりました。拡張機能は、C＃、マニホールドを介したJava、Swift、Kotlinなどの言語で積極的に使用されています。

問題


catchErrorメソッドがあるとしましょう。これはひどいもので、新しいクールな関数に書き直す必要があります。厳密に型指定された関数または関数型チェックの代わりに、引数として任意の型の関数を使用するとします。これは、この機能を開発するときに8か月前にその時点で論理的だったために発生します。

最初に頭に浮かぶのは、この関数を書き直すことですが、ここではプロジェクトで頻繁に発生し、関数を変更するとプロジェクト全体が動作しなくなるという問題に直面しています。

まあ、最初のオプションが私たちのためではない場合。論理的な理由から、適切な場合は、すべての要件を満たす新しいFuture関数を実装できます。

abstract class Future<T> {
  ...
  /// Catches any [error] of type [E].
  Future<T> onError<E>(FutureOr<T> handleError(E error, StackTrace stack)) =>
      this.catchError(...   -  ...);
}
 ...
}

そして私はこのように彼女に電話します：

Future<String> someString = ...;
someString.onError((FormatException e, s) => ...).then(...);

残念ながら、この関数をFutureクラスに追加できません。これを行うと、それもFutureインターフェースに追加され、このインターフェースを実装する他のクラスは不完全になり、コンパイルできなくなります。

もう1つのオプションは、次のようなサードパーティの関数を実装することです。

Future<T> onFutureError<T, E>(Future<T> source, 
    FutureOr<T> handleError(E error, StackTrace stack)) => 
        source.catchError(... - ...);

そして、彼女の電話は次のようになります。

Future<String> someString = ...;
onFutureError(someString, (FormatException e, s) => ...).then(...);

すべてがうまくいきます。しかし、ひどく読まれ始めたのは悲しいことです。メソッドを使用します。これらはクラス内で実装されるため、-。doingSomething（）;と呼ばれます。このコードは理解できます。左から右に読んで、一連のイベントを頭の中で立ちます。ヘルパー関数を使用すると、コードが煩雑になり、読みにくくなります。

それでは、新しいクラスを実装して、ユーザーに古いインターフェースを改善された機能でラップさせることができます。

class CustomFuture<T> {
  CustomFuture(Future<T> future) : _wrapper = future;
  Future<T> _wrapper;

  Future<T> onError<E>(FutureOr<T> handleError(E error, StackTrace stack)) =>
      _wrapper.catchError(...-     ...);
}

呼び出しは次のようになります。 

Future<String> someString = ...;
CustomFuture(someString).onError((FormatException e, s) => ...).then(...);

素晴らしく見える！

拡張機能の問題を解決する


Pascalでのプログラミングを停止して2019年に戻ると、この機能の実装は次のサイズに削減されます。

extension CustomFuture <T> on Future<T> {
  Future<T> onError<E>(
      FutureOr<T> handleError(E error, StackTrace stack)) =>
          this.catchError(...something clever...);
}

そしてこれは呼び出しがどのようになるかです：

Future<String> someString = ...;
someString.onError((FormatException e, s) => ...).then(...);

それで全部です！この問題の解決に必要なコードはわずか5行です。君は。どんな種類の魔法があり、どのように機能するのか疑問に思うかもしれません。

実際には、ラッパークラスと同じように動作しますが、実際には単なる補助静的関数です。拡張機能により、明示的なラッパーの記述を手放すことができます。

これはラッパーではありません


拡張機能の設計は、既存のクラスの宣言のように見えますが、private _wrapperを使用したラッパーであるかのように機能します。ただし、ラッパークラスと比較して1つの利点があります。これは、_wrapperラッパークラスにアクセスするのではなく、クラス自体に直接アクセスすることです。

この機能は機能のために作られたものではありませんが、前に述べたように、拡張機能は実際に静的関数を呼び出すためのより便利な方法です。これは、ラッパーオブジェクトがないことを意味します。

それはすべて静的です


上記の「静的拡張メソッド」と言いましたが、それは理由があってやったのです！

Dartは静的に型付けされています。コンパイラーはコンパイル時に各式のタイプを知っているため、user.age（19）と記述し、ageが拡張である場合、コンパイラーは、呼び出し全体のタイプを見つけるために、指定されたオブジェクトにラップされているタイプを判別する必要があります。

どのような問題が発生する可能性がありますか？


拡張機能の問題の最も単純な例は、そのスコープに複数の拡張機能がある場合です。基本的に、勝者は、メンバーを呼び出す実際のタイプの式に最も近い拡張機能です。

問題を解決する最も簡単な方法は、必要な拡張を厳密に接続するか、拡張を明示的に使用することです。

...
List list = ...;
MyList(list).printlist();
SomeList(list).printlist();
...

extension MyList on List { 
  void printlist() {
    print(...- ...);
  }
}

extension SomeList on List {
  void printlist() {
    print(...-  ...);
  }
}

概要


  • dart言語には、既存の機能を拡張するための便利なツールがあります。
  • メソッド、演算子、セッター、ゲッターは拡張できますが、フィールドは拡張できません。
  • 拡張メソッドを明示的に呼び出すことができます。または、インターフェースメンバーまたは他の拡張と暗黙的に競合しない場合。
  • 暗黙的な呼び出しは、明示的な呼び出しと同じように機能します。
  • 拡張機能は静的です。それらに関するすべては、静的型に基づいて解決されます。

拡張機能の競合が原因で拡張機能の出力が失敗した場合は、次のいずれかを実行できます。

  1. 拡張機能を明示的に適用します。
  2. プレフィックスを含む競合する拡張機能をインポートします。これは、暗黙的な呼び出しに使用できないためです。
  3. 競合する拡張機能をまったくインポートしないでください。

それで全部です！拡張機能を最大限に活用できます。
