みなさん、こんにちは。私たちはこの作業に積極的に参加しており、1月には多くの強力なリリースを準備しています。特に、誰もが愛するLinux Administratorコースの新しいストリームのセットが発表されました。発売を見越して、有用な資料の翻訳を伝統的に共有しています。



ファイルのアクセス許可は、SUID実行可能ファイルに代わる安全な手段を提供しますが、最初は少し混乱するかもしれません。

SUID バイナリはセキュリティソリューションとしては不十分であることは誰もが知っています。幸い、アプリケーションでいくつかの制限された特権が必要な場合は、ファイルのアクセス許可と呼ばれるより効率的な方法があります。

上記の記事の詳細な閲覧を避けたい場合は、時間を節約します。基本的に、ファイルのアクセス許可は、rootとして実行されるプロセスを許可するため、何かを実行する権利があり、このリストによって制限される特定の機能を保存します特権をリセットし、特権のないユーザーが開始したとき。つまり、攻撃者がバッファオーバーフローまたは他のエクスプロイトによってプロセスを危険にさらした場合、プロセスが実際に必要とする特定の最小特権以外のものを利用することはできません。

アクセス許可は、通常は常にルートとして実行されるサービスに最適ですが、コマンドラインユーティリティについてはどうでしょうか。幸い、正しいユーティリティがインストールされていれば、これもサポートされます。たとえばUbuntuを使用している場合は、パッケージが必要になりますlibcap2-bin。また、古いバージョンのカーネル（バージョン2.6.24以降）を実行する必要があります。

これらの関数を使用すると、SUIDビットを設定するのと同じ方法でアクセス許可を実行可能ファイルに関連付けることができますが、アクセス許可の特定のセットのみが対象です。このユーティリティはsetcap、ファイルのアクセス許可を追加および削除するために使用されます。

最初のステップは、必要な権限を選択することです。この記事では、rawソケットtracewalkを使用できるはずのネットワーク診断ツールがあると想定しています。これには通常、アプリケーションをrootとして実行する必要がありますが、リストを表示すると、権限のみが必要であることがわかります。 バイナリがあるディレクトリにいるとすると、次のようにこの権限を追加できます。CAP_NET_RAW

tracewalk

sudo setcap cap_net_raw=eip tracewalk

とりあえず=eip、許可のためにサフィックスを無視してください、数秒でそれについて話します。権限名は小文字であることに注意してください。これで、次のようにして権限が正しく構成されているかどうかを確認できます。

setcap -v cap_new_raw=eip tracewalk

または、この実行可能ファイルに設定されているすべての権限を一覧表示できます。

getcap tracewalk

参考までに、実行可能ファイルからすべての権限を削除することもできます。

setcap -r tracewalk

この時点で、実行可能ファイルを非特権ユーザーとして実行できるはずです。また、彼はrawソケットを操作できるはずですが、rootユーザーが持っている他の特権を持っていないはずです。

この奇妙なサフィックスはどういう意味=eipですか？これには、権限の性質を少し理解する必要があります。各プロセスには、有効、継承、および許可（有効、継承、および許可）の 3つのアクセス許可セットがあります。

  • 有効な権限とは、プロセスが実際に実行できることを決定する権限です。たとえばCAP_NET_RAW、効率的なセットでない限り、生のソケットを処理できません。

  • 継承可能なアクセス許可は、子プロセスの使用可能なコレクションで継承できるアクセス許可です。動作中fork()またはclone()その瞬間に、それはまだ同じ実行可能ファイルを実行しているので、子プロセスは、親プロセスの権限のコピーは常に、与えられています。継承セットはexec()、実行可能ファイルを別のファイルで置き換えるために（または類似の）が呼び出されたときに使用されます。この時点で、利用可能なプロセスセットは継承されたセットによってマスクされ、新しいプロセスで使用される利用可能なセットが取得されます。

したがって、ユーティリティをsetcap使用すると、特定の実行可能ファイルに対してこれら3つのセットの権限を個別に追加できます。グループの意味は、ファイルのアクセス許可では少し異なります。

アクセス許可をsetcap3文字eで指定する場合、それぞれ、有効なセット、継承されたセット、アクセス可能なセットiをp参照してください。したがって、以前の仕様：

sudo setcap cap_net_raw=eip tracewalk

...は、解像度CAP_NET_RAWをアクセス可能で継承されたセットに追加し、有効ビットも設定する必要があることを示します。これにより、以前にファイルに設定された権限が置き換えられます。複数の権限を一度に設定するには、コンマ区切りのリストを使用します。

sudo setcap cap_net_admin,cap_net_raw=eip tracewalk

アクセス許可ガイドでは、これらすべてについてさらに詳しく説明していますが、この投稿でこの事件が少しわかりやすくなったと思います。ほんの少しの警告とトリックに言及するだけです。

まず、ファイル機能はシンボリックリンクでは機能しません。これらをバイナリ自体（つまり、シンボリックリンクのターゲット）に適用する必要があります。

第二に、これらは解釈されたスクリプトでは機能しません。たとえば、アクセス許可を割り当てたいPythonスクリプトがある場合、それをPythonインタープリター自体に割り当てる必要があります。明らかに、これは潜在的なセキュリティの問題です。これは、このインタープリターで実行されるすべてのスクリプトに指定されたアクセス許可があるためですが、これはSUIDを実行するよりもはるかに優れています。最も一般的な回避策は、明らかに、必要な操作を実行してスクリプトから呼び出すことができる別の実行可能ファイルをCまたはアナログで書き込むことです。これは、バイナリファイル/usr/bin/dumpcapを使用して特権操作を実行するWiresharkで使用されるアプローチに似ています。

$ getcap /usr/bin/dumpcap 
/usr/bin/dumpcap = cap_net_admin,cap_net_raw+eip
