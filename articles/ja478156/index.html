PostgRESTについて話し合う-Haskell上のオープンソースWebサーバー

PostgreSQLクエリで動作するようにAPIを設定できるツールについて説明します。ここでは、ユーティリティの機能、利点、欠点、および代替ソリューションについて説明します。


写真-  キャンペーンクリエイター -Unsplash

PostgRESTとは


APIでPostgreSQLデータベースを操作するためのオープンソースの Webサーバーです。著者によると、ソリューションは手動CRUDプログラミングの代替手段です。その助けを借りて、データベースの構造を複雑にし、オブジェクトリレーショナルマッピング（ORMマッピング）を実行するビジネスロジックを作成する必要がなくなります。これにより、命令型コードが出現します。

PostgRESTはHaskellで書かれており、無料のライセンスで配布されており、広範なコミュニティを備えています。gitterチャットで製品サポートを入手できます。今日、Webサーバーは積極的に使用されていますOracle-Moat、eコマースプラットフォームiAdvizeの分析ツール、URL画像チャートを使用してチャートを作成するシステムなど、本番環境でのプロジェクト。OAuth、websocket、nginxの

拡張機能もPostgREST用に開発されています。

可能性について少し


システムは、データベーススキーマのすべてのビューとテーブルへのクエリを処理できます。たとえば、peopleテーブルからデータを取得するには、次のコマンドを作成するだけで十分です。

GET /people HTTP/1.1

GETに 加えて、OPTIONS、POST、PATCH、DELETEなどのキーワードを識別できます。それらのすべては、データを操作するときにアカウントのアクセス権を考慮に入れます。ユーザーの権限が不十分なテーブルへのリクエストは拒否されます。公式ドキュメントには対応する例があります。データベースは、ToDoテーブルに新しいタスクを追加するためのAPIリクエストを受け取ります。

curl http://localhost:3000/todos -X POST \
     -H "Content-Type: application/json" \
     -d '{"task": "do bad thing"}'

応答として、PostgRESTは失敗を生成します。

{
  "hint": null,
  "details": null,
  "code": "42501",
  "message": "permission denied for relation todos"
}

Webサーバーは、ストアドプロシージャもサポートしています。それらは、PL / pgSQL、PL / Python、またはPLV8で書かれています。既製のプロシージャーは、POST / rpc / procedure_name（名前付きパラメーターを渡す場合）またはGET / rpc / procedure_name（GETパラメーターを使用する場合）を使用して呼び出すことができます。

コミュニティがツールについてどう思うか


PostgRESTシステムは軽量でリソースを必要としないという意見 があります（6つのインスタンスでは、メモリ消費量が70 MBを超えることはめったにありませんが、Node.js / Waterlineバンドルの場合、この数値は1ギガバイトに増えます）。同様の見解がハッカーニュースの居住者によって表明され、PostgRESTがHaskellで本番環境に展開した最初のツールであると指摘しました。


写真  -NeringaŠidlauskaitė -Unsplash

しかし、コミュニティにはPostgRESTを批判する人々がいます。あるユーザーは、Haskellのコードは読みにくいため、本番環境でのそのようなライブラリのデバッグは難しい場合があると指摘しました。さらに、実際には、ツールはバックエンドデータベースからのビジネスロジックを処理するタスク-これは10年以上前に行われました。これに関して、PostgRESTの代わりにプレーンSQLで制約を記述する方が良いという意見があります。

HNを使用している別のユーザーは、 Webサーバーは一時的で高性能なアプリケーションの展開にのみ適していると述べています。長期的には、データベース構造で問題が発生する可能性があります。データベーススキーマはAPIスキーマになるため、別のビューを構築する必要があります。
