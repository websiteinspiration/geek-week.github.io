MagicStringとは何ですか？これらの行はとても魔法ですか？

MagicStringはあまり知られていないライブラリです。これにもかかわらず、それは差し迫った問題の1つを解決します-その構造（AST-抽象構文ツリー）を使用してソースコードを変更します。

この記事では、MagicStringとは何か、そしてこれらの行が本当に「魔法」であるかどうかを学びます。これにより、Angularのドキュメントをいかに迅速に翻訳できたか、およびMarkdownと他の形式のファイルの両方のユニバーサルトランスレーターの作成にどのように役立つかを説明する次の記事を理解するのに役立ちます。





2週間前に、Angularのロシア語ドキュメント（angular24.ru）をリリースしました。この間、35の問題が追加され、テキストが修正され、2つのプルリクエストが追加されました。あなたがテキストを選択し、翻訳を提供し、GitHubで自動的に発行するシステムが機能することを心から疑いました。しかし、クラウドソーシングは機能します！ :)あなたはこの記事からこれについてもっと学ぶことができます。

リリース後、最もよく聞かれた質問の1つは「なぜですか」でした。質問は完全に正しいですが、それに答えるためには、まずMagicStringが何であるか、それがどのように機能し、どのように役立つかを理解する必要があります。

単純なソースコードがあるとします。

const a = 1;

constをvar に置き換えます。最も簡単な解決策は、constを通常のString.prototype.replaceでvarに置き換えることです。そして、このタスクでは、これが最も正しい解決策である可能性が最も高いです。しかし、グローバルスコープでのみconstをvarに置き換える必要がある場合はどうでしょうか。しかし、関数内でそれらを置き換えませんか？もちろん、より複雑な規則性を考えたり、トリッキーなコードを記述したりできますが、よりスケーラブルで柔軟な方法があります。

パーサーを使用して、AST-抽象構文ツリーを取得できます。 ASTに興味がある場合は、astexplorer.netにアクセスしてください。本質的には、コードの構造を正確に表示するツリーです。

さらに、このASTの各ノードには開始がありますそして、ソースコード内のこれらの要素の位置を示す終了インデックス。これらの座標を知っていて、ドキュメントの構造を手元に持っていると、ドキュメントの構造を維持したまま、複雑な置換や置換を行うことができます。



通常、置換はビジターパターンデザインといくつかのヘルパーを使用して行われます。これらは通常、「トランスフォーマーAPI」と呼ばれる単一のライブラリーにラップされます。各パーサーには、独自の「トランスフォーマーAPI」があります。

このようなライブラリは非常に使いやすいですが、いくつかの問題があります。それらの1つはパフォーマンスです。

ASTツリーの各（ほぼすべての）ノードには座標が含まれているため、1つのノードを変更する場合、ツリーの残りの座標を更新する必要があることがよくあります。ここでは、少しの血でできると主張できます-どこでも座標を更新するのではなく、構造に基づいてASTをテキストに戻すだけです。しかし、1つの問題があります。元のテキストの書式がすぐに失われるため、既存の行でconstをvarに置き換えるというタスクに矛盾します。実際、新しいフォーマットの新しい行を取得します。そして、これが小さな行で問題にならない場合は、constをvarで置き換えたためにフォーマットが完全に変更された1000行のファイルを想像してください。それは非常によく聞こえません。



そして、ここにMagicStringの魔法があります。彼らの存在を最初に知ったのは、バターナットと呼ばれるリッチハリスプロジェクトからでした。バターナットはJavaScriptの縮小版です。 butternutはUglifyJSの3倍、Babiliの10〜15倍高速であると述べられています。私は先に進み、プロジェクトが少なくとも3年前に銅の盆地で覆われていたと言います。しかし、それでも、そのパフォーマンスの秘密に興味をそそられました。それはMagicStringでした。

MagicStringの操作を見てみましょう。

var MagicString = require( 'magic-string' );
var s = new MagicString( 'const a = 1; const b = 2;' );

s.overwrite( 0, 5, 'var' );
s.toString(); // 'var a = 1; const b = 2;'

//

MagicStringのアルゴリズムは非常にシンプルです。元の文字列をオブジェクトにラップします。オブジェクトには文字列への変更を直接適用せず、将来の配列に座標と実行する必要があることを追加します。そして、誰かが結果の行を取得したい場合にのみ、蓄積された操作を実行するために1から1を開始します。例えば：

  1. インデックス0から始まり、インデックス5で終わるconstをvarに置き換えました
  2. 後続のすべての置換のインデックスは2未満である必要があります（varがconstより2文字少なく、1行短い）。
  3. すべての操作の座標を更新します
  4. 以下の操作等を適用します。




すべてがかなりシンプルに見えます。しかし、なぜMagicStringの方が速いのですか？答えは非常に簡単です。ツリーで実行する操作の数は、ASTノードの数よりも大幅に少なくなります。 ASTに必要なメモリの量、およびツリートラバーサル（ツリーの移動）は無料の操作ではなく、O（n + m）であることは言うまでもありません



さらに30分待つ準備ができている場合はどうなりますか？そして、これはMagicStringの2番目のプラスです。各パーサーは、変換用の独自のAPIを発明します。そして、これは非常に良いことです。そのようなAPIがある場合（すべてのパーサーがそれを提供するわけではありません）、ASTを使用してソーステキストを正常に置き換えることができないままになることがよくあります。しかし、MagicStringはソース文字列を変更するための単一のユニバーサルAPIです。使用したパーサーまたはパーサーの組み合わせは関係ありません。 MagicStringを使用すると、どのASTでも同じように作業できます。
