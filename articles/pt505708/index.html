Por que não escrevemos código nos controladores?

Eu acho que muitos de vocês ouviram a opinião de que não deveria haver código nos controladores e, portanto, um controlador com métodos em uma linha é considerado "Melhor Prática". Por sua vez, duvido que os benefícios disso sejam tão grandes. Se você teve pensamentos semelhantes, pergunto sob kat.



Olá a todos! Quero dizer imediatamente que minha opinião não é verdadeira, e o objetivo deste post é expressar minha opinião e ouvir os comentários de outras pessoas. Todas as opções acima se aplicam à implementação da API, se você tiver MVC com visualizações, esse caso não funcionará, porque nesse caso é melhor escrever apenas a lógica com View nos controladores

O que está errado?


Tenho certeza de que muitos de vocês estão trabalhando com um aplicativo CRUD típico com arquitetura de três camadas (a propósito, não há mais camadas, mas mais sobre isso na próxima vez). E nessa arquitetura, você tem uma camada para trabalhar com dados (DA adicional), uma camada de lógica de negócios (BL adicional) e uma camada de visualização (VL ​​adicional).

O BL pode ser feito de diferentes maneiras, encontrei duas opções:

Esses métodos têm várias regras próprias e uma delas é que os manipuladores não podem chamar outros manipuladores, eles devem ser completamente independentes, exatamente a mesma situação na implementação por meio de uma classe comum com vários métodos, os métodos estão tão ligados ao recurso que não podem ser usados. reutilizado.

Vamos remover o excesso.





Milagrosamente, o código de serviço não difere do controlador, apenas adicionamos alguns atributos e estamos prontos para capturar solicitações http.

Acredito que o ASP NET nos abstraísse perfeitamente do trabalho com HTTP, temos o melhor local para operar com nossos tipos. Repito, se você tiver visualizações, nos controladores é melhor escrever código apenas para o View, e nos serviços escrever métodos reutilizados para obter dados para o View. Mas nas realidades atuais, cada vez mais temos API + SPA.

Validação?


O ASP NET Core Pipeline está muito bem ajustado e possui muitas soluções, dê uma olhada no FluentValidation, você adicionará a validação sem alterar o código nos controladores.

Quer mais separação?


Separe a interface e se você precisar de uma implementação também.



Como um bônus, a interface de serviço se torna um contrato de nível superior e, dentro da estrutura de um processo, é apenas uma chamada direta do código do controlador, como parte da comunicação cliente-servidor, uma implementação simples da mesma interface usando o HttpClient é substituída.

Conecte outros canais


Se eles me disserem que podemos ter outro canal, por exemplo, através de uma fila, posso obter uma instância do controlador e usá-la em outro canal. Este controlador resolve facilmente a partir do DI. Além disso, o ASP NET é bastante flexível e você pode ensinar alguns canais a processá-lo, modificando novamente o pipeline.



Fato engraçado


Uma tarefa de teste nesse estilo será descartada e rejeitada, mas você tem vários motivos para fazê-lo, mas se for a uma entrevista e discutir por que está criando o código, provavelmente este é um bom lugar e você tem o mesmo especialista que entende essa programação é multifacetada .

Total


Você pode reutilizar apenas serviços por tipo de repositório, cache, API do cliente etc., mas reutilizar a lógica do processamento de uma única solicitação é um caso muito raro e provavelmente ruim.

Essa abordagem, como outras, precisa ser aplicada nos lugares certos, economiza tempo e é muito conveniente para APIs (microsserviços) fechados ao acesso do público.

Acredito que os controladores sejam o próprio Serviço BL ou Command \ QueryHandler e, em meus projetos, pratico essa abordagem e compartilho muito bem os controladores, recomendo que você também tente.

Não aumentamos a responsabilidade da estrutura, ela tinha uma tarefa:
