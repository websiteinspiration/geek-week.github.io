本番環境でのJavaScriptモジュールの使用：現在の状況。パート1

2年前、私は現在、一般にmodule / nomoduleパターンと呼ばれている技法について書きました。そのアプリケーションを使用すると、ES2015 +の機能を使用してJavaScriptコードを記述し、次にバンドラーとトランスパイラーを使用して2つのバージョンのコードベースを作成できます。それらの1つには最新の構文が含まれています（ビュー構文を使用して読み込まれ<script type="module">、2番目はES5構文です（それはを使用して読み込まれます<script nomodule>）。モジュール/ nomoduleパターンを使用すると、モジュールをサポートするブラウザーに、この機能をサポートしないブラウザーよりもはるかに少ないコードを送信できます。現在、このパターンはほとんどのWebフレームワークとコマンドラインツールをサポートしています。



以前は、最新のJavaScriptコードを本番環境に送信する機能、およびほとんどのブラウザーがモジュールをサポートしていたとしても、コードをバンドルで収集することをお勧めしました。

どうして？主に、ブラウザへのモジュールのロードが遅いと感じていたからです。 HTTP / 2などの最近のプロトコルは理論的には複数のファイルの効率的なロードをサポートしていますが、当時のすべてのパフォーマンス調査では、バンドルを使用する方がモジュールを使用するよりもさらに効率的であると結論付けています。

しかし、それらの研究が不完全であったことは認められなければならない。調査されたモジュールを使用したテストケースは、本番環境にデプロイされた非最適化および非最小化ソースコードファイルで構成されていました。最適化されたバンドルと、最適化されたクラシックスクリプトを使用したモジュールとの比較はありませんでした。

しかし、正直なところ、モジュールをデプロイするための最適な方法はありませんでした。しかし、今では、バンドルテクノロジーのいくつかの最新の改善により、静的および動的インポートコマンドの両方を使用して、ES2015モジュールの形式で本番コードをデプロイでき、同時に、利用可能なオプションを使用して達成できるよりも高いレベルのパフォーマンスを得ることができます。モジュールは使用されません。 元の資料が公開さ

れているサイト（今日翻訳の最初の部分）では、モジュールが数か月間本番環境で使用されていることに注意してください。

最適なコード構築戦略


本番用のコードを構築することは、常にさまざまなソリューションの長所と短所のバランスをとる試みです。一方では、開発者はコードをできるだけ速くロードして実行することを望んでいます。一方、彼はプロジェクトのユーザーが使用しないコードをダウンロードしたくありません。

さらに、開発者はコードがキャッシュに最適であることを確信する必要があります。コードバンドリングの大きな問題は、コードに変更を加えた場合でも、変更された行が1つでも、バンドル全体のキャッシュが無効になることです。数千の小さなモジュール（ソースコードに含まれている形式で正確に表されている）で構成されるアプリケーションをデプロイする場合、コードに小さな変更を安全に加えることができ、同時にアプリケーションコードのほとんどがキャッシュされることがわかります。 。ただし、すでに述べたように、このような開発アプローチでは、リソースへの最初のアクセス時にコードをロードするときに、従来のアプローチを使用する場合よりも時間がかかる可能性があります。

その結果、バンドルをパーツに分割するための適切なアプローチを見つけることが難しいタスクに直面します。マテリアルをロードする速度とそれらの長期キャッシングの間で適切なバランスを取る必要があります。

ほとんどのバンドラーは、デフォルトで、動的インポートコマンドに基づくコード分割技術を使用します。しかし、動的インポートにのみ焦点を当ててコードを分割しても、コードを十分に小さなフラグメントに分割することはできません。これは特に、リピーターが多いサイト（つまり、キャッシュが重要な状況）に当てはまります。

コードは可能な限り小さなフラグメントに分割する必要があると思います。フラグメントの数が大きくなりすぎてプロジェクトのダウンロード速度に影響が及ぶまで、フラグメントのサイズを小さくすることには価値があります。また、状況を個別に分析することをお勧めしますが、前述の調査で行った概算が信じられれば、モジュールのロード数が100未満の場合でも、ロード速度が著しく低下することはありません。個別のHTTP / 2パフォーマンス調査、ダウンロードしたファイル数が50未満の場合、プロジェクトの顕著なスローダウンは明らかになりませんでした。ただし、ここでは、ファイル数が1、6、50、および1000のオプションのみをテストしました。結果として、おそらく100ファイルは、ダウンロード速度を失うことを恐れずに簡単に移動できる値です。

それでは、コードを積極的に分割しすぎないようにするための最善の方法は何ですか？動的インポートコマンドに基づいてコードを分割することに加えて、コードをnpmパッケージに分割する方法を詳しく検討することをお勧めします。このアプローチでは、フォルダーからプロジェクトにインポートされるものはnode_modules、パッケージ名に基づいて完成したコードの個別のフラグメントに分類されます。

パッケージの分離


先に述べたように、バンドラーの最新の機能の一部により、モジュールに基づいてプロジェクトをデプロイするための高性能スキームを編成することが可能になります。私が話していたことは、2つの新しいロールアップ機能によって表されます。 1つ目は、動的コマンドによる自動コード分離ですimport()（v1.0.0で追加）。 2番目のオプションは、オプションに基づいてプログラムによって実行される手動のコード分離ですmanualChunks（v1.11.0で追加）。

これら2つの機能のおかげで、コードがパッケージレベルで分割されるビルドプロセスの構成が非常に簡単になりました。

オプションを使用した構成例を以下に示しますmanualChunksインポートされた各モジュールnode_modulesが、パッケージの名前に対応する名前の個別のコードに分類されるため（技術的には、フォルダー内のパッケージディレクトリの名前node_modules）：

export default {
  input: {
    main: 'src/main.mjs',
  },
  output: {
    dir: 'build',
    format: 'esm',
    entryFileNames: '[name].[hash].mjs',
  },
  manualChunks(id) {
    if (id.includes('node_modules')) {
      //   ,    `node_modules`.

      //   - ,       .

      const dirs = id.split(path.sep);
      return dirs[dirs.lastIndexOf('node_modules') + 1];
    }
  },
}

オプションmanualChunkは、単一の引数としてモジュールファイルへのパスを受け取る関数を受け入れます。この関数は文字列名を返すことができます。それが返すものは、現在のモジュールを追加する必要があるアセンブリのフラグメントを指します。関数が何も返さない場合、モジュールはデフォルトで使用されるフラグメントに追加されます。

ファイルの最後に、次のようなエクスポート式が表示されます。この式には、フラグメントに追加されたモジュールのエクスポートコマンドのみが含まれ、すべてのモジュールのエクスポートコマンドは含まれないことに注意してくださいlodash。
