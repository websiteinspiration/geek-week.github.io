これは、ジェスチャーコントロールに関するシリーズの3番目の記事です。あなたがそれらを逃したなら、あなたはそれらを最初と2番目の部分の翻訳に慣れることができます。

以前の記事では、画面の端から端までスペースを埋めるトピックについて説明しました。この記事では、アプリケーションのジェスチャーとAndroid 10の新しいシステムジェスチャーの間に発生する競合を処理する方法について説明します。

ジェスチャーの競合とはどういう意味ですか？例を見てみましょう。ユーザーがをドラッグアンドドロップして現在の曲をスクロールできる音楽プレーヤーがあるとしますSeekBar。



残念ながら、SeekBarホーム画面に戻るジェスチャー領域に近すぎるため、前のアプリケーション（QuickSwitch）にすばやく切り替えるジェスチャーが開始され、ユーザーに不便を与えます。

同じことが、ジェスチャ領域が配置されている画面の端でも発生する可能性があります。ナビゲーションドロワー（DrawerLayout）、カルーセル（ViewPager）、スライダー（SeekBar）、リスト内のスワイプなど、競合を引き起こす可能性のある多くの一般的な例があります。

「どうすればこれを修正できるのか」という質問が表示されます。この質問を容易にするために、状況に基づいて回答を提供するフローチャートを作成しました。

質問が説明を必要としないことを願っていますが、念のため、それぞれについて説明します。

1.アプリケーションでナビゲーションバーとステータスバーを非表示にする必要がありますか？


最初の質問は、アプリケーションの主なユースケースが必要かどうかで隠れて、ナビゲーションおよび/またはステータスバーを。非表示とは、これらのシステムパネルがまったく表示されないことを意味します。これは、「エッジツーエッジ」などの概念をアプリケーションに実装したことを意味するものではありません。

この質問に「はい」と答える考えられる理由：

  • WindowManagerフラグを使用していFLAG_FULLSCREENます。これは、テーマ属性を使用して行うこともandroid:windowFullscreen、テーマオプションの1つから拡張することもできますTheme.XXX.Fullscreen。
  • system-ui可視性フラグを使用していますSYSTEM_UI_FLAG_FULLSCREEN。
  • イマーシブモードでsystem-ui可視フラグの1つを使用します：SYSTEM_UI_FLAG_IMMERSIVEまたはSYSTEM_UI_FLAG_IMMERSIVE_STICKY。

この質問に「はい」と答えるべきアプリケーションの一般的な例は、ゲーム、ビデオプレーヤー、写真ビューア、描画アプリケーションです。

2. UIを使用するための主なシナリオは、ジェスチャー領域内/周辺でのスワイプを含みますか？


この質問は、ユーザーがスワイプする必要のあるジェスチャーエリア（「戻る」と「ホーム」の両方）内/横の要素がUIに含まれているかどうかを調べます。

ゲームは通常、

  • 画面上のコントロールは通常、画面の左端/右端と下部の近くにあります。
  • 一部のゲームでは、画面上の任意の場所にある要素をスワイプする必要があります。

ゲームに加えて、イエスと言わなければならないUIの一般的な例は次のとおりです。

  • UIは、ドラッグ可能なフレームが画面の左端と右端の近くにもある写真をトリミングします。
  • ユーザーが画面全体を覆うキャンバスに描画できる描画アプリケーション。

これがかなり単純な質問であることを願っています。これには、ジェスチャの領域をカバーし、画面の大部分、DrawerLayoutまたは大きな部分まで拡大するビューも含まれますViewPager。

私たちは戦術を少し変えて、個々の見方を検討し始めます。3番目の質問に対して肯定的に回答したビューのいずれかについて、少し説明します。ユーザーがスワイプ/ドラッグする必要がありますか？

4番目の質問に基づいて、ビューがジェスチャー領域に完全に配置されているのか、主に配置されているのかを明らかにします。

6.ビューの境界線が必要なジェスチャー領域と重なっていますか？


最後の質問は、ビューが必要なジェスチャー領域の下にあるかどうかを明確にします。あなたが私たちを思い出した場合は、以前の記事を、あなたがいることを思い出します必須のシステムジェスチャーの領域は、システムのジェスチャーが常に優先されます、画面の領域です。

Android 10では、画面の下部にある必須のジェスチャー領域が1つだけあり、ユーザーは家に戻るか、最新のアプリケーションを開くことができます。これはプラットフォームの将来のリリースで変更される可能性がありますが、今は画面下部のビューで作業するだけで済みます。

典型的な例は次のとおりです。

  • モードレスBottomSheetは、画面の下部で小さなドラッグアンドドロップビューに折りたたまれる傾向があるためです。
  • 画面の下部にある水平方向にスクロールするカルーセル（ステッカー付きのインターフェースなど）。

API


当社では、以前の記事で、我々はそれを言及し、「アプリケーションが画面の特定の部分のためのシステムのジェスチャーを除外する能力を持っています。」アプリケーションは、Android 10で最初に登場したジェスチャー除外APIを使用してこれを行います。

システムがジェスチャー領域を除外するために提供する2つの異なる機能がView.setSystemGestureExclusionRects()ありWindow.setSystemGestureExclusionRects()ます。何を使用するかはアプリケーションによって異なります。Androidを使用する場合View、システムはビューAPIを優先し、それ以外の場合はWindowAPIを使用します。

2つのAPIの主な違いは、ウィンドウAPI長方形はウィンドウの座標空間にあると想定します。ビューを使用する場合、通常は代わりにビュー座標空間で作業します。View APIは座標空間間の変換を処理します。つまり、ビューのコンテンツに関してのみ推論する必要があります。

例を見てみましょう。SeekBar画面の幅全体に配置された音楽プレーヤーの例を再び使用します。SeekBar前のセクションでホーム画面に戻るジェスチャーとの競合を修正しましたが、注意する必要があるジェスチャーの左側と右側の領域があります。

「スライダー」のときにユーザーが曲をスキップしようとするとどうなるか見てみましょうSeekBar（円形のドラッグ）がいずれかのエッジの近くにあります：


SeekBarは戻るジェスチャー領域と競合しています。

スライダーは右側のジェスチャー領域の下にあるため、システムはユーザーがジェスチャーを使用して戻って来ようとしていると考え、したがって戻る矢印を表示します。これは、ユーザーが現時点で戻りたくないので、ユーザーにとっては不便です。上記のジェスチャー除外APIを使用してこれを修正し、スライダーの境界を除外できます。

ジェスチャー例外APIは通常onDraw()、ビューがレンダリングされるときとonLayout()そうでないときの2つの場所から呼び出されます。あなたの見解List<Rect>除外されるすべての長方形を含みます。前述したように、これらの長方形は独自のビュー座標系にある必要があります。

制限事項


ジェスチャー除外APIは、すべてのジェスチャーの競合を解決するための完璧なソリューションのように思えるかもしれませんが、実際にはそうではありません。ジェスチャー除外APIを使用して、アプリケーションのジェスチャーがシステムの戻りアクションよりも重要であることを宣言します。これは強力なステートメントであるため、このAPIは、他に何もできない場合の緊急出口として設計されています。

ジェスチャー除外APIを使用して、アプリケーションのジェスチャーがバックトラックのシステムアクションよりも重要であることを宣言します。

APIが提供する動作は快適なユーザーエクスペリエンスに違反する可能性があるため、システムはその使用を制限します。アプリケーションが除外できるのは、エッジあたり最大200 dpまでです。

開発者がこれを聞いたときによくある質問をいくつか示します。

なぜ制限が必要なのですか？上記の説明が理由を示していることを願っています。ユーザーがサイドスワイプから一貫して戻ることができることが非常に重要であると考えています。 1つのアプリケーションではなく、デバイス全体で一貫して。この制限は制限が厳しすぎるように見えるかもしれませんが、画面の端全体を除いて1つのアプリケーションだけではユーザーに不便を引き起こし、アプリケーションの削除またはより根本的な何かにつながります。

言い換えれば、ナビゲーションシステムは常に一貫していて使いやすい必要があります。

なぜ200dpなのですか？200dpの議論は非常に簡単です。前述したように、ジェスチャー除外APIは最後の手段として使用するように設計されているため、この制限はいくつかの重要なタッチターゲットの倍数として計算されました。感覚的標的の最小推奨サイズは、48dp.4感覚的標的× 48dp = 192dpです。さらにインデントを追加すると、値が取得されます200dp。

エッジあたり200 dpを超える数を除外する必要がある場合はどうなりますか？システムは、リクエストした最低200dpのみを除外します。


システムは、画面外のマイビューの下端から数えて、合計200 dpの高さのリクエストを許可

しますが、これは制限と見なされますか？いいえ、システムは画面内にある除外された長方形のみを考慮します。同様に、ビューが部分的に画面上にある場合、要求された四角形の可視部分のみが考慮されます。
