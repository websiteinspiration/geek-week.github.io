Getメソッド


標準のget djangoメソッドに最初からエラーがあったことに気づく人はほとんどいません。 getメソッドは、1つのオブジェクトを返すか、複数のオブジェクトが見つかったことを警告するか、オブジェクトがないことを報告する必要があります。

1  def get(self, *args, **kwargs):
2         clone = self._chain() if self.query.combinator else self.filter(*args, **kwargs)
3         if self.query.can_filter() and not self.query.distinct_fields:
4             clone = clone.order_by()
5         limit = None
6         if not clone.query.select_for_update or connections[clone.db].features.supports_select_for_update_with_limit:
7             limit = MAX_GET_RESULTS
8             clone.query.set_limits(high=limit)
9         num = len(clone)
10        if num == 1:
11            return clone._result_cache[0]
12        if not num:
13            raise self.model.DoesNotExist()
14        raise self.model.MultipleObjectsReturned()

行9は、クエリセットで指定されたすべてのレコードのデータを受け取り、それらを一連のオブジェクトに変換します。ドキュメントにこれに関する警告があります。

バージョン3より前は、要求されたオブジェクトの数に制限はありませんでした。つまり、オブジェクトがたくさんあるという警告を出す前に、すべてのデータを完全に受信してオブジェクトに変換します。

その結果、メモリ内に数百万のオブジェクトを取得できましたが、1つ以上のオブジェクトが見つかったことがわかりました。現在、5、7、8行目があります。そして今、誇らしげに受け取るのはMAX_GET_RESULTS = 21個のオブジェクトだけで、それから複数のオブジェクトがあることがわかります
。
修復方法：django.db.models.query.pyのMAX_GET_RESULTSをオーバーライドします
GETをオーバーライドするか、GETを呼び出す前に、以下を使用します。

vars(queryset.query).update({'high_mark':2, 'low_mark':0})

queryset.query.set_limits(0,2)

Djangoモデルの__Init__メソッド


__init__組み込みメソッドのコードでの宣言は完全には明確ではありません 

_setattr=setattr

これはおそらく、関数リファレンスをローカルディクショナリに転送してコードを疑似的に高速化するためのものですが、これについてはそうではありません。いくつかの問題があり

ます。1. __init__モデルで追加の属性=値のペアを渡すと、「予期しないキーワード引数を取得」します。

この場合、__ init__メソッドに追加するのではなく、初期化後に属性を追加することをお勧めします。

obj = MyClass()
vars(obj).update({'attr':val, 'attr2':val2 ...})

2.新しいDjangoは、モデルの任意のフィールド（Field.descriptor_class）の記述子を再定義する機能を追加しました。ただし、オブジェクトが初期化されているかどうかは、1つの記述子ではわかりません。これは、たとえば、メインオブジェクトの初期化後にのみ表示されるprefetch_relatedオブジェクトのデータを記述子が使用する場合に必要です。

初期設定終了の信号は使いたくないので、チャンネル登録者が多くなる可能性があります。

この場合、__ init__をオーバーライドし、属性を追加して初期化を終了するよりも賢い方法は思いつきませんでした。

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._end_init = True

クエリセット


ハードコードされたEmptyQuerySet / DateQuerySetはクリーンアップされ、すでに良好です。ただし、クエリセットをマネージャーとして使用する状況は、イデオロギー的に好ましくありません。

マネージャーが作成したQuerySetクラスをオーバーライドする場合は、_queryset_class属性を追加します

class MyManager(models.Manager):
    _queryset_class = MyQuerySet

注意：これは古いバージョンでは機能しません。たとえば、次のようにできます。

class MyManager(models.Manager):
    def get_query_set(self):
        response = super(MyManager, self).get_query_set()
        response.__class__ = MyQuerySet
        return response

inlineFormset管理パネル


いくつかの問題があり

ます。1.メインフォームオブジェクトと直接通信しない標準のinlineFormsetレコードを表示することはできません。例：製品価格を編集するためのフォーム。真ん中にあるのは、「類似」商品の卸売価格を購入するための参照統計表形式のフォームです。

get_formsetインラインモデルメソッドをオーバーライドし、BaseInlineFormSetから継承した独自のMyFormSetを作成することで解決します

def get_formset(self, request, obj=None, **kwargs):
        kwargs['formset'] = MyFormSet
        super().get_formset(request, obj, **kwargs)

class MyFormSet(BaseInlineFormSet):
    pass

2.管理パネルでinlineformsetを使用してオブジェクトを編集しているときに、誰かが別のメカニズムを介してinlineformset内のオブジェクトレコードの1つを削除すると、エラーが発生し、オブジェクトを保存できなくなります。新しいブラウザウィンドウでのkopyペーストのみ。

これまでのところ、1つの解決策しか見つかりませんでした-inlineformsetは使用しないでください。

ContentType-djangoモデルレジストリ


50％の天才/ 50％の鈍さ。
どのモデルもデフォルトのモデルレジストリにアクセスできません。
私たちのプロジェクトでは、すべてのクラスにミックスインを追加することで解決しています：

from django.contrib.contenttypes.models import ContentType
class ExportMixin(object):
    @classmethod
    def ct(cls):
        if not hasattr(cls, '_ct'):
            cls._ct, create = ContentType.objects.get_or_create(**cls.get_app_model_dict())
            if create:
                cls._ct.name = cls._ct.model._meta.verbose_name
                cls._ct.save()
        return cls._ct

    @classmethod
    def get_model_name(cls):
        if not hasattr(cls, '_model_name'):
            cls._model_name = cls.__name__.lower()
        return cls._model_name

    @classmethod
    def get_app_name(cls):
        if not hasattr(cls, '_app_name'):
            cls._app_name = cls._meta.app_label.lower()
        return cls._app_name

    @classmethod
    def get_app_model_dict(cls):
        if not hasattr(cls, '_format_kwargs'):
            cls._format_kwargs = {'app_label': cls.get_app_name(), 'model': cls.get_model_name()}
        return cls._format_kwargs

これで、必要に応じてobj.ct（）を呼び出すことができます。

ユーザーモデル


ユーザーモデルをオーバーライドする機能は、バージョン1.5で登場しました。

しかし、バージョン3では、標準のUserCreationForm / UserChangeFormでモデル=ユーザーを修正していませんでした。
それは解決されました：

from django.contrib.auth.forms import UserCreationForm
from django.contrib.auth import get_user_model
class MyUserCreationForm(UserCreationForm):
    class Meta:
        model= get_user_model()
