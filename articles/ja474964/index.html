実際の実装とのモック同期

テスト戦略が議論されるたびに、同期の問題が発生します。基本的に-開発者のためにmokasが作成する追加の負荷と、mookが実際の依存関係から逸脱するリスクがあるためです。

では、mokaと実際の実装を確実に同期させる方が、どのように安くなるのでしょうか。

同期については、mokおよび実際の実装に対して同じチェックを実行するテストを作成できます。

それは次のようになります（DIなしで記述しますが、DIを使用するとよりシンプルで正確になります）。

public abstract class AbstractValidOrderDaoTest(){
    Dao dao;
    public abstract arrange();
    @Test
    public void whenValidOrderInDb_thenReturnValidOrder(){
        arrange();
        Order order = dao.retrieve();
        assertNotNull(order);
        assertNotNull(order.getCustomerName());
        //   
    }
}
public class ValidOrderDaoTest extends AbstractOrderDaoTest(){
    @Override
    public void arrange(){
        dao = new FakeValidOrderDao();
    }
}
public class OrderDaoTest extends AbstractOrderDaoTest(){
    @Override
    public void arrange(){
        dao = new RealOrderDao(new ValidOrderDataSource(url, user, pwd));
    }
}

OrderDaoTestは、基になるモックまたは実際の依存関係を持つ実際のオブジェクトに対して機能し、ValidOrderDaoTestはモックに対して機能します。

ValidOrderDataSourceが実際のベースである場合、OrderDaoTestは個別のパッケージに含まれ、統合テストの一部として実行されます。たとえば、データベースの更新時にクラッシュする場合があります。これはCI \ CDを妨害しないはずです。

ValidOrderDataSourceがモックデータベースの場合、OrderDaoTestは残りのユニットテストと共に実行されます。

mokの同期は実際のクラスのテストを意味するため、
実際のクラスでは、その基礎となる依存関係を吸収する必要があります。さらに、基になるMok依存症は、上にあるMokのシナリオに従って動作する必要があります。私たちの場合、これは
ValidOrderDataSourceです。

あなたがそれについて考えるならば、それは理にかなっています-より高いクラスのふるまいについてのどんな陳述も暗黙的に基礎となるもののいくつかのシナリオを意味します。コントローラがサービスから何かを返す場合、ベースがそれを提供できればよいでしょう。

逆に、上位のクラスは下位のクラスについて非現実的な考えを持っていることが多いため、不要なスクリプトを削除することは悪くありません。

再帰は、トップレベルのモックを同期させるために、外部の依存関係に至るまで、すべての基礎となるモックの同期を開始する必要があることを示唆しています。

これにより、より一般的で抽象的なシナリオがよりプライベートなシナリオに依存するようになるため、システム仕様がより透過的になります。

また、同期する必要のないmokaがあることに注意してください。それら。クロステストが必要になるような実際の実装はありません。これは、主要なエラーシナリオに適用されます。 EmptyResultException_Datasource例：これにより、必要なクロステストの数が大幅に削減されます。

同期は、キュー、外部サービス、データベースなどの実際の外部依存関係によって、特にそれらが取得して返すデータに関して、必ず必要です。

外部サービスが突然変化する場合、これは多くの場合開発段階ですが、同期テストを記述しないと、その動作を確認する方法がありません。

労働集約度の観点から。それ自体、すでにいくつかの任意のモック依存関係を備えた実際のクラステストがあります。非同期テストと比較して、いくつかのことを行う必要があります。

  • 抽象テストで行為を強調し、主張する
  • モカのために特定のテストをする
  • 実際のクラステストでモックの依存関係を修正する
  • 必要に応じて、外部依存関係まで完全に再帰的に繰り返します。
