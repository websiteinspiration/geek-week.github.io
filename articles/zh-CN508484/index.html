Typescript编译器API：控制编译器





Typescript应用程序开发始终有一个项目构建阶段。通常，为此使用构建系统和工作流自动化，例如webpack或gulp，挂有足够的插件，或者使用本机tsc或项目框架中使用的CLI命令将构建过程散布在package.json命令和shell脚本中。所有这些解决方案各有利弊。通常，在组装过程中，需要做一些非标准的事情，事实证明，所使用的组装系统无法从包装盒中提供必要的功能，并且现有的插件并不能完全满足需要。在这样的时刻，项目工作开始了，配置中的疯狂选择开始了，并寻找合适的插件。在某个时候，您意识到在花时间寻找合适的拐杖时，您可以写下您的决定。



在许多情况下，可以使用javascript脚本来自动化项目中的关键流程，该脚本的表现力和功能使其可以描述所需的工作流程并从各种库中进行选择，而不必为特定的构建系统而为它们使用插件。这种方法的一个重要优点是对过程的完全控制和最大的灵活性。对于使用Typescript作为主要开发语言的项目，出现了一个问题，即如何将其编译过程嵌入到您的工作流中。 Typescript编译器API可以在这里解决。在本文中，我们将看到如何将其用于编译在Typescript上实现的项目，如何在其工作的不同阶段与编译器进行交互，以及编写用于热重载REST服务器的脚本，由Nest.js开发。



我的空闲时间在一个节点上有一个项目。最初，它是专为从未取得成功的小型企业而设计的，但现在，我将其用作实验的训练场。该项目最初是在Nuxt.js上与Fastify一起构建的，已经经历了许多转换。在这里，转换为打字稿，并转换为单一存储库，并用Nest.js替换Fastify。由于该项目的负荷不应该很大，因此所有这些工作都应该在一个服务器上进行，所以我为REST服务器和通过Nuxt.js进行的前端交付使用了一个Express实例。为此，在Nest.js的后台运行的Express获得了Nuxt.js的渲染功能。



该解决方案在生产中效果很好，但是由于我只是在一个项目上工作，因此编写API函数和并行创建接口很方便。为此，在更改文件时重新启动Nest.js不应导致重新启动以HMR模式运行的Nuxt.js，否则它将重新开始构建过程，这毫无意义。但是这里出现了一个问题，事实是，尽管CLI Nest.js中有一个监视模式可以执行热重载，但它会在一个单独的进程中加载​​它，并在重新编译项目时重新启动。因此，以这种方式重新启动服务器将导致Nuxt丢失上下文，然后将开始其完整组装。



在所有这些变得清晰之后，我决定在项目中进行自己的热重装。最初，我尝试按照Nest.js文档中的建议使用webpack实施解决方案，但事实证明，用于实现重启的start-server-webpack-plugin使用与CLI相同的策略，即在一个单独的过程中运行它，然后根据需要重新启动。我重写了这个插件，以便它可以按我的需要工作，但是事实证明，使用Webpack解决方案非常困难，而且并非没有出现在这里和那里的问题，因此很难专注于项目的开发。



然后，我决定需要更改方法，我想直接使用tsc编译器中规定的机制，该机制始终像时钟一样快速，稳定地工作。我最终发现一篇不错的文章，展示了使用TypeScript Compiler API的示例。事实证明，内置在Typescript库中的工具使我能够实现所需的功能，并且使用起来非常方便。不幸的是，我发现TypeScript编译器API的信息很少，所以我决定分享自己设法找到的信息。我还注意到，在编写本文时，此API仍在开发中，有些地方可能会随着时间而改变。



简易编译



对于初学者来说，要了解TypeScript Compiler API的原理，让我们尝试使用它来编译项目。我将在其上进行实验的项目具有nest.js的标准结构：源位于src文件夹中，入口点是./src/main.ts文件。



为了使编译器成功编译项目，它需要将编译参数传递给它。最简单的方法是在脚本中正确解析它。创建tsconfig.js文件： 



const {ModuleResolutionKind, ModuleKind, ScriptTarget} = require("typescript")

module.exports =  {
    moduleResolution: ModuleResolutionKind.NodeJs,
    module: ModuleKind.CommonJS,
    target: ScriptTarget.ES2019,
    declaration: true,
    removeComments: true,
    emitDecoratorMetadata: true,
    experimentalDecorators: true,
    sourceMap: true,
    outDir: "./dist",
    baseUrl: "./",
    allowJs: true,
    skipLibCheck: true,
  }


请注意，一些参数（例如moduleResolution，module，target）是通过枚举定义的，因此无法直接从tsconfig.json中提取参数（它们以字符串形式编写），因此，我不会用它来实现最简单的示例。如何从tsconfig.json中正确选择参数，我们稍后会了解。



现在我们有了编译选项，您可以继续下一步。为此，请创建一个具有以下内容的ts-compiler.js文件： 



const ts = require('typescript');

function compile() {
  const compilerOptions = require('./tsconfig');
  const program = ts.createProgram(['./src/main.ts'], compilerOptions);
  const emitResult = program.emit();
}

compile();


如果现在运行此脚本来执行，则编译后的程序文件应显示在./dist目录中。编译分为两个阶段：首先，createProgram命令分析源文件并创建要编译的文件列表，然后我们使用emit函数编译并保存编译后的文件。很简单



为了提高可用性，我们现在需要从tsconfig.json文件中实现编译选项的加载。在脚本中添加以下功能： 




const formatHost = {
  getCanonicalFileName: path => path,
  getCurrentDirectory: ts.sys.getCurrentDirectory,
  getNewLine: () => ts.sys.newLine,
};

function getTSConfig() {
  const configPath = ts.findConfigFile('./', ts.sys.fileExists, 'tsconfig.json');
  const readConfigFileResult = ts.readConfigFile(configPath, ts.sys.readFile);
  if (readConfigFileResult.error) {
    throw new Error(ts.formatDiagnostic(readConfigFileResult.error, formatHost));
  }
  const jsonConfig = readConfigFileResult.config;
  const convertResult = ts.convertCompilerOptionsFromJson(jsonConfig.compilerOptions, './');
  if (convertResult.errors && convertResult.errors.length > 0) {
    throw new Error(ts.formatDiagnostics(convertResult.errors, formatHost));
  }
  const compilerOptions = convertResult.options;
  return compilerOptions;
}


为了使用配置文件，API提供了许多功能，可让您查找，加载配置并将其转换为编译器可以理解的形式。在这里，findConfigFile函数用于确定配置文件的路径，然后使用readConfigFile将其作为json加载，然后，如果没有错误，则使用convertCompilerOptionsFromJson获取编译参数。

我不想用数百行文件路径的覆盖物阻塞输出，因此，我将编写一个将输出路径输出为一行的函数： 



function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
  };
  return displayFunction;
}


现在让我们添加compile（）函数代码。 



function compile() {
  const compilerOptions = getTSConfig();
  const compilerHost = ts.createCompilerHost(compilerOptions);
  compilerHost.readFile = displayFilename(compilerHost.readFile, 'Reading');
  compilerHost.readFile.enableDisplay();

DIY增量程序监视程序



在上一节中，我们研究了如何使用TypeScript编译器API自定义Typescript编译过程。现在是时候做些更紧迫和有用的事情了，这确实可以提高开发的速度和便利性。在程序编译期间，编译器需要解决程序中的所有依赖关系。



为此，他顺序读取并分析了项目文件和库文件。即使要编译像我的相对较小的项目，在准备阶段，编译器也需要读取，解析和分析2000多个文件。这会导致编译过程明显变慢，例如，在我的笔记本电脑上，编译大约需要20秒。同时，在开发过程中，通常在编译器启动之间只有几个文件发生更改，因此从2.7版开始，Typescript具有增量编译选项是合乎逻辑的，该选项与监视模式一起使您可以监视磁盘上文件的更改，从而显着提高了重新编译的速度。



CLI Nest.js也使用这些功能，您可以使用以下命令以此模式开始工作 



nest start --watch


利用这种开发模式的另一个机会是使用--watch开关运行tsc。但是，正如我在引言中已经写过的那样，如果标准命令的功能在某个时候不再足够了，该是时候看看Typescript的内幕并了解它在那里如何工作了。

    './',
    ts.sys.fileExists,
    'tsconfig.json',

    configPath,

      console.log(

      ),
    diagnostic =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );
  ts.createWatchProgram(host);
}

为了方便我们在主机中挂接不同的东西，让我们编写一个小助手： 



function on(host, functionName, before, after) {
  const originalFunction = host[functionName];
  host[functionName] = function() {
    before && before(...arguments);
    const result = originalFunction && originalFunction(...arguments);
    after && after(result);
    return result;
  };
}


这将使我们能够在主机中被调用函数之前和之后执行所需的操作，同时接收传递给它的参数和执行结果。现在，可以使用该助手在不同阶段方便地干扰编译器。例如，让我们向控制台添加消息以准备编译：

让我们弄清楚这里发生了什么。首先，我们需要app变量，在其中写入引导函数的结果，以便在更改代码并开始重新编译后正确关闭服务器。它将包含用于调用服务器关闭命令的处理程序。其次，只有在编译过程中未在代码中发现任何错误时，服务器才会启动。为此，我们使用getSemanticDiagnostics检查诊断次数为零。第三，由于模块化系统缓存加载的模块，因此在更改代码后，我们需要清除缓存中的依赖项。为此，我编写了一个小的辅助函数clearCache，我们之前称之为。 



function clearCache() {
  const cacheKeys = Object.keys(require.cache);
  const paths = [
    join(__dirname, 'dist'),
    dirname(require.resolve('typeorm')),
    dirname(require.resolve('@nestjs/typeorm')),
  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      );
    });
}


此处清除了缓存中的依赖关系，这些依赖关系可能阻止我们正确地重新启动服务器。就我们而言，我们必须从缓存的dist目录中删除项目的所有模块。我还删除了typeorm模块，因为它们实现的功能会阻止服务器重新启动（这是另一个原因）。使用超时，我们等待直到服务器关闭并且app变量被清除。当编译器检测到文件更改并开始使用修改后的文件创建程序时，我们将关闭服务器。为此，我们将一个钩子添加到createProgram函数。

  ];
  cacheKeys
    .filter((item) => paths.filter((p) => item.startsWith(p)).length > 0)
    .forEach((item, index, arr) => {
      delete require.cache[item];
      process.stdout.clearLine(); // clear current text
      process.stdout.cursorTo(0); // move cursor to beginning of line
      process.stdout.write(
        `Clearing cache ${Math.floor((index * 100) / arr.length + 1)}%`,
      );
    });
  process.stdout.write(' finished.\n');
}

function displayFilename(originalFunc, operationName) {
  let displayEnabled = false;
  let counter = 0;
  function displayFunction() {
    const fileName = arguments[0];
    if (displayEnabled) {
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
      process.stdout.write(`${operationName}: ${fileName}`);
    }
    counter++;
    return originalFunc(...arguments);
  }
  displayFunction.originalFunc = originalFunc;
  displayFunction.enableDisplay = () => {
    counter = 0;
    if (process.stdout.isTTY) {
      displayEnabled = true;
    }
  };
  displayFunction.disableDisplay = () => {
    if (displayEnabled) {
      displayEnabled = false;
      process.stdout.clearLine();
      process.stdout.cursorTo(0);
    }
    console.log(`${counter} times function was called`);
  };
  return displayFunction;
}

async function watchMain() {
  const configPath = ts.findConfigFile(
    './',
    ts.sys.fileExists,
    'tsconfig.json',
  );

  const host = ts.createWatchCompilerHost(
    configPath,
    {},
    ts.sys,
    ts.createEmitAndSemanticDiagnosticsBuilderProgram,
    (diagnostic) =>
      console.log(
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
    (diagnostic) =>
      console.log(
        'Watch status: ',
        ts.formatDiagnosticsWithColorAndContext([diagnostic], formatHost),
      ),
  );

  host.readFile = displayFilename(host.readFile, 'Reading');
  let app;
  on(
    host,
    'createProgram',
    () => {
      console.log("** We're about to create the program! **");
      app && app.close().then(() => (app = undefined));
      host.readFile.enableDisplay();
    },
    () => {
      host.readFile.disableDisplay();
      console.log('** Program created **');
    },
  );
  let currentProgram;
  on(
    host,
    'afterProgramCreate',
    (program) => {
      console.log('** We finished making the program! Emitting... **');
      currentProgram = program;
    },
    () => {
      console.log('** Emit complete! **');
      const onAppClosed = () => {
        if (app) {
          setTimeout(onAppClosed, 100);
        } else {
          clearCache();
          require('./dist/bootstrap')
            .bootstrap()
            .then((res) => {
              app = res;
            });
        }
      };
      if (currentProgram && currentProgram.getSemanticDiagnostics().length === 0) {
        onAppClosed();
      }
    },
  );

  ts.createWatchProgram(host);
}
process.exit = (code) => {
  console.log('!!!!!!!!!!!!!!!!!!!!!!! WARNING!!!!!!!!!!!!!!!!!!!!!!!!!');
  console.trace(`Process try to exit with code ${code}.`);
};
watchMain();
