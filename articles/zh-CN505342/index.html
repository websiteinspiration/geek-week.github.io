依赖注入和依赖倒置原理是不相同的

本文的翻译是在“ C ++ Developer”课程开始之前准备的。



曾几何时，当我开始在LosTechies上写博客时，我写了几篇有关Dependency Inversion（DI），Dependency Injection（也是DI）的文章，当然还有关于我最终如何我开始理解依赖注入的本质。当时，我认为两个DI都是同一技术名称的变体，无论您称其为“注入”还是“倒置”。

注入！=反转


在这两个出版物发表大约一年之后，我为《 Code Magazine》撰写了有关SOLID软件开发原理的文章。在写作过程中，我问了我当时的合作者Derek Greer成为本文的审阅者。事实证明，这是我对SOLID的理解以来做过的最好的事情，因为Derek足够友好（并且对我的顽固耐心），向我展示了我对依赖反转的理解与正确的理解有何不同。他花时间指导我阅读Bob叔叔的原始文章，向我解释在哪里将依赖注入与反转混合使用，最后为我澄清了这个话题。结果，我得到了一篇很棒的文章，该文章仍然很受欢迎-并且我非常感谢Derek帮助纠正我的理解。

但是，直到今天，我仍然继续观察其他人如何犯同样的错误- 即使在红宝石中。因此，为了帮助其他人理解什么是依赖倒置和什么不是依赖倒置，我将在SOLID文章中重新发布有关此部分的内容。我了解，在本文其余部分的上下文之外，本文的一部分没有任何意义。但是，一般原则应该是显而易见的，并且您随时有机会按照文章的链接来阐明您的观点。

依赖倒置原则


依赖倒置原则（DIP）包含两个部分：

  • 上级模块不应依赖于下级模块。这些和其他都应该依赖抽象。
  • 抽象不应依赖细节。细节应取决于抽象。

上次要打开灯以照亮房间的一部分时刷新。是否有必要在墙壁上切一个洞，挖洞以寻找布线，将其裸露，然后将灯直接焊接到房屋的布线中？当然不是（至少我希望不是！）。电源插座为此提供了标准接口。在大多数工业化国家中，没有人期望传统的灯可以将其直接焊接到建筑物的电线中。另外，没有人期望只能将一盏灯插入插座。我们习惯于连接灯，计算机，电视，吸尘器和其他设备。标准的120伏，60赫兹的电源插座已成为美国社会的普遍部分。

相同的原则适用于软件开发。与其使用一组紧密互连的类，不如使用标准接口。此外，根据LSP（Liskov替代原理-Liskov替代原理），您想保证可以在不违反该接口期望的情况下替换实现。因此，如果您使用接口并希望能够替换它，则需要确保仅使用该接口，而不使用特定的实现。也就是说，依赖于接口的代码应该只知道有关该接口的详细信息。他不应该知道实现此接口的任何特定类。

策略，详细信息和抽象的关系


稍微解释一下反转的原理，可以说策略（高层）不应依赖于细节（实现），细节应取决于策略。更高级别的策略应该定义它将引用的抽象，在该抽象中该部分的某些实现将执行所请求的操作。这种观点有助于说明为什么它是依赖关系反转的原理，而不仅仅是依赖关系抽象的原理。

作为一个为什么零件对策略的依赖性与依赖性相反的示例，请考虑我们在FormatReaderService中编写的代码。格式化的文件阅读器服务是一项策略。它定义了IFileFormatReader接口应该执行的操作-这些方法的预期行为。这使您可以专注于策略本身，确定阅读服务的工作方式，而无需考虑细节的实现-各种格式的阅读器。因此，读者依赖于读者服务提供的抽象。最后，服务读取器和单个读取器都依赖于格式读取器接口的抽象。

通过反转依赖关系减少连接


您知道，没有依赖关系（零连接）对于一个类是不切实际的。如果连接为零，则将没有可用的一组类。但是，您还希望尽可能地摆脱直接的联系。您希望对系统进行划分，以便它可以更改其各个部分，而无需更改除这些部分以外的任何内容。依赖倒置的原理是此概念的关键。仅依靠抽象（例如接口或基类），可以减少系统不同部分之间的连接。这使您可以使用各种实现来重建系统。

考虑需要在清晰的层次结构中实例化的许多类，以便您可以获得所需的某些功能。最简单的方法是使用最高级别的类（您要调用的那个）来创建低一级类的实例，并使用它实例化低级的类，依此类推。图14显示了一个对象图，其中较高级别（策略）的对象依赖于该对象，并直接与较低级别（细节）的对象连接。


图14：与零件关联的策略。

这实现了必要的层次结构，但直接绑定了类。您必须先拖动Bar才能使用Foo。如果要分离这些类，则可以轻松地输入Bar将要实现的接口以及Foo所依赖的接口。图15显示了一个简单的IBar接口，您可以基于Bar类的公共API创建该接口。


图15：使用抽象进行分离。

在这种情况下，您可以通过引入接口将Bar的实现与其在Foo中的使用分开。但是，您只能通过从中选择接口来分离实现。您尚未反转依赖关系结构，还没有解决此架构中的所有连接问题。

在这种情况下您想要更改Bar会发生什么？根据您要进行的更改，您可以引发连锁反应，迫使您更改IBar接口。 Foo依赖于IBar接口，因此您还必须更改Foo的实现。您可能已经分离了Bar的实现，但是Foo依赖于Bar的更改。也就是说，政策仍然取决于细节。

如果要反转依赖关系的结构，以使Part变得依赖于策略，那么首先必须更改视角。使用该系统的开发人员应了解，您不仅从接口中提取了实现。是的，这种分隔是必要的，但还不够。您需要了解抽象属于谁-政治或详细信息。

依赖关系反转原则指出，细节必须依赖于策略。这意味着您必须具有定义并拥有该部件实现的抽象的策略。在脚本Foo-> IBar-> Bar中，您需要将IBar视为Foo的一部分，而不仅仅是Bar的包装器。尽管结构上没有任何变化，但所有权的观点已发生变化，如图16所示。


图16：政治拥有抽象。细节取决于政策。

如果Foo具有IBar抽象，则可以将这两个结构放在独立于Bar的程序包中。您可以将它们放在自己的名称空间，自己的程序集等中。这可以大大提高哪个类或模块取决于哪个类或模块的可见性。如果您看到AssemblyA包含Foo和IBar，而AssemblyB提供了IBar的实现，那么您会很容易注意到Bar的粒度取决于Foo定义的策略。

如果您正确颠倒了依赖项的结构，那么链对策略和/或详细信息更改的反应现在也将是正确的。当您更改Bar的实现时，您不再观察到向上的更改链。这是因为Bar需要符合Foo提供的抽象-细节不再指示更改策略。然后，当您更改Foo的需求并导致IBar接口发生更改时，您的更改将在整个结构中传播。条（详细信息）将根据策略更改而被强制更改。

电子邮件发送系统相关性的分离和反转


查看我们的代码库，您可能会注意到IFileFormatReader已经是依赖关系反转的示例。 FormatReaderService类拥有格式读取器接口定义。如果格式化文件阅读器服务的需求发生了变化，您很可能会看到一连串的变化，逐渐形成格式化阅读器。但是，如果特定文件格式的阅读器发生了更改，则您不太可能看到这些更改会影响格式化文件的阅读服务。这使您想知道还有哪些地方可以反转系统依赖性。

您要做的第一件事是分离接收日志消息和通过发送表单通过电子邮件发送日志的逻辑。您不介意这两个电子邮件阅读和发送服务的链接，但是何时以及什么原因引起的问题令人怀疑。您意识到该过程实际上是以以下形式重复的：一次是从文件发送，一次是从数据库发送。然后您回想起所有其他也使用此功能的部门，并开始怀疑复制过程的真实程度。另外，您的一些朋友最近谈论了“单元测试”。他们说，您必须确保要测试的过程的真实逻辑封装在没有引用外部系统的对象中。

考虑到所有这些，您决定创建一个名为ProcessingService的对象。经过数分钟的来回移动代码以尝试合并该过程之后，您意识到您不希望将处理服务直接连接到数据库读取器或文件读取服务。经过一番思考，您会注意到它们之间的一种模式：GetMessageBody方法。使用此方法作为基础，您可以创建一个新的IMessageInfoRetriever接口，并将其实现为数据库读取器和文件读取器。

public interface IMessageInfoRetriever
{ 
  string GetMessageBody(); 
}

public class FileReaderService: IMessageInfoRetriever
{
  public string GetMessageBody() {/* ... */}
}

public class DatabaseReaderService: IMessageInfoRetriever
{
  public string GetMessageBody() {/* ... */}
}

此接口允许您提供所需的处理服务的任何实现。然后，请注意电子邮件服务，该服务当前直接链接到处理服务。简单的IEmailService接口解决了此问题。图17显示了生成的结构。
