Djangoプロジェクト用のDebian、Nginx、およびGunicornの構成



全てにおいて良い日。

プロジェクトDjango 3.xのNginxでDebianサーバーを上げるタスクがありました。インターネット上のたくさんの情報を壊して、私はいくつかの異なるサイトからの推奨事項を組み合わせることによってこれをなんとかしました。Djangoプロジェクト用に最初のサーバーを設定する方法に興味がある場合は、ようこそ。

あなたが私のことを理解できるように、私はあなた自身について少しお話します。私は開発者でも、プログラマーでも、システム管理者でもないし、IT教育も受けていません。私はコンピュータサイエンスの教師です。しかし、職場では、コンピューターサイエンスの学校のコースから遠く離れているいくつかのポイントを生徒に説明する必要があります。その1つは、Djangoでのプロジェクトの開発です。

settings.pyの設定


本番サーバーにデプロイするときは、プロジェクトのデバッグを無効にし、いくつかの設定を変更する必要があります。これは、プロジェクトのsettings.pyで次のように行いました。

DEBUG = False
if DEBUG:
    ALLOWED_HOSTS = ['*']
else:
    ALLOWED_HOSTS = ['HOST IP', 'DOMAIN NAIM', 'localhost']
...
STATIC_URL = '/static/'
if DEBUG:
    STATIC_DIR = os.path.join(BASE_DIR, 'static')
    STATICFILES_DIRS = [
        STATIC_DIR,
        '/var/www/static/',
    ]
else:
    STATIC_ROOT = os.path.join(BASE_DIR, 'static/')
    STATICFILES_FINDERS = (
        'django.contrib.staticfiles.finders.FileSystemFinder',
        'django.contrib.staticfiles.finders.AppDirectoriesFinder',
    )
MEDIA_URL = '/media/'

でurls.py、私は静とメディアの設定を変更しました。

if settings.DEBUG:
    urlpatterns += static(settings.MEDIA_URL, document_root=settings.MEDIA_ROOT)
    urlpatterns += static(settings.STATIC_URL, document_root=settings.STATIC_ROOT)

これがどれほど適切な決定であるかはわかりません。おそらく、これらのファイルにより適切な変更が加えられています。しかし、この場合、開発と公開を切り替えるために、DEBUGの値を変更するだけで十分です。

Gunicornを構成する


カスタマイズのために開きましょう 

sudo nano /etc/systemd/system/gunicorn.socket

ファイルにいくつかの設定を書きましょう：

[Unit]
Description=gunicorn socket

[Socket]
ListenStream=/run/gunicorn.sock

[Install]
WantedBy=sockets.target

[Unit]セクションを作成してソケットを記述し、[Socket]セクションでソケットの場所を決定し、[Install]セクションで適切なタイミングでソケットをインストールする必要があります。

systemdユーティリティファイルを開いて、サービスを構成します。

sudo nano /etc/systemd/system/gunicorn.service

[Unit]
Description=gunicorn daemon
Requires=gunicorn.socket
After=network.target

[Service]
User=username
Group=www-data
WorkingDirectory=/var/www/djangoprojectname
ExecStart=/var/www/djangoprojectname/env/bin/gunicorn \
          --access-logfile - \
          --workers 5 \
          --bind unix:/run/gunicorn.sock \
          myproject.wsgi:application

[Install]
WantedBy=multi-user.target

ユーザー、プロジェクト名、および仮想環境を示すことを忘れないでください。

私に説明したように、ワーカーはプロセッサコア数* 2 + 1として計算されます。
次に、Gunicornソケットを起動してアクティブにします。

sudo systemctl start gunicorn.socket
sudo systemctl enable gunicorn.socket

そして、すべてが機能することを必ずテストしてください！

sudo systemctl status gunicorn.socket

実行中のサービスに関する情報を表示する必要があります。Q

キーを押して終了します（英語のレイアウトでは必須）。

file /run/gunicorn.sock

最後のコマンドは、ファイルに関するメッセージを表示する必要があります。

最初のsudo systemctl status gunicorn.socketコマンドが失敗した場合、または2番目のファイル/run/gunicorn.sockコマンドが、gunicorn.sockファイルがディレクトリにないことを報告した場合、Gunicornソケットを作成できませんでした。次のコマンドでGunicornソケットログを確認する必要があります。

sudo journalctl -u gunicorn.socket

間違いを確認して修正します。

ソケットのアクティブ化のテストに移りましょう。

sudo systemctl status gunicorn

おそらくあなたは記録を持っているでしょう：

Active: inactive (dead)

これまでのところ、すべてが順調です。

curlを介してソケットへの接続を確立します。

curl --unix-socket /run/gunicorn.sock localhost


Bad Request（400）エラーをスローすることがあります。localhostの

代わりにサーバーのIPを指定してみてください。問題がなければ、おそらくこれはまだ行っていないnginx設定が原因である可能性があります。ただ先に進みなさい。 また、ステータス出力をリクエストします。



sudo systemctl status gunicorn

Active: active (running)

レコードがこれである場合、+エラーに関する情報ではなく、まだ多くの情報がありますが、すべて問題ありません。

それ以外の場合は、ログのエラーを確認します

sudo journalctl -u gunicorn

そして、gunicornプロセスを再起動します

sudo systemctl daemon-reload
sudo systemctl restart gunicorn

NGINXセットアップ


サイトの新しいサーバーブロックを作成して構成し、アドレスバーでサイトにアクセスすると、サーバーが何をどこから取得するかをサーバーが理解できるようにします。

sudo nano /etc/nginx/sites-available/djangoprojectname

server {
    listen 80;
    server_name server_domain_or_IP;

    location = /favicon.ico { access_log off; log_not_found off; }
    location /static/ {
        alias /var/www/djangoprojectname/static/;
    }

    location /media/ {
        alias /var/www/djangoprojectname/media/;
    }

    location / {
        include proxy_params;
        proxy_pass http://unix:/run/gunicorn.sock;
    }
}

sever_nameセクションでは、複数のアドレスをスペースで区切って指定できます。

構文エラーをテストします。

sudo nginx -t

エラーがない場合は、サーバーを再起動して、ファイアウォールに必要な権限を与えます。

sudo systemctl restart nginx
sudo ufw allow 'Nginx Full'

ポート8000​​へのアクセスを削除することもできます

sudo ufw delete allow 8000

この時点で、すべてが機能するはずですが、httpsを構成するまではうまくいきませんでした。残念ながら、これがどのように接続されているのか説明できませんが、メディアファイルは正常に読み込まれていましたが、静力学は読み込まれていませんでした。

PostgreSQLのセットアップ


PostgreSQLを設定し、SQLiteからデータをインポートします。必要ない場合は、この手順をスキップするか、PostgreSQL設定のみを使用してください。

データベースとユーザーを作成します。

sudo -u postgres psql

CREATE DATABASE dbforproject;
CREATE USER projectdbuser WITH PASSWORD 'password';

Djangoプロジェクトの推奨事項に従います。

デフォルトのエンコーディングをUTF-8に設定し、デフォルトのトランザクション分離スキームを「コミットされた読み取り」に設定して、未確認のトランザクションからの読み取りをブロックします。タイムゾーンをGMT（UTC）に設定します。

ALTER ROLE projectdbuser SET client_encoding TO 'utf8';
ALTER ROLE projectdbuser SET default_transaction_isolation TO 'read committed';
ALTER ROLE projectdbuser SET timezone TO 'UTC';

その後、作成したユーザーに、新しいデータベースを管理するためのアクセス権を提供する必要があります。

GRANT ALL PRIVILEGES ON DATABASE dbforproject TO projectdbuser;

すべてのコマンドが完了したら、次のコマンドを使用してPostgreSQLダイアログを閉じることができます。

\q

これでPostgreSQLの設定が完了しました。これで、DjangoがPostreSQLで正しく機能するように設定します。

まず、SQLiteから古いデータをコピーします（必要な場合）。これを行うには、プロジェクトに移動して仮想環境を起動します。

cd /var/www/djangoprojectname
source ./env/bin/activate
cd mainappname/
python3.8 manage.py dumpdata > datadump.json

データベースに接続するための設定を変更します。

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql_psycopg2',
        'NAME': 'dbforproject',
        'USER': 'projectdbuser',
        'PASSWORD': 'password',
        'HOST': 'localhost',
        'PORT': '',
    }
}

データベースの移行を実行します。

python3.8 manage.py migrate --run-syncdb

このシェルの後：

python3 manage.py shell

そしてそれを書いてください：

>>> from django.contrib.contenttypes.models import ContentType
>>> ContentType.objects.all().delete()
>>> quit()

以前にアップロードしたデータをSQLiteからロードします。

python3.8 manage.py loaddata datadump.json
