Goアプリ用の最小限のDockerコンテナーの作成

こんにちは、ハブル！Meetspaceapp Nick Gauthierサービスの創設者、「Goアプリケーション用の最小限のDockerコンテナーの構築」による記事の翻訳に注目します。

読書時間：6分

さまざまなプログラミング言語（Goを含む）には、公式およびコミュニティでサポートされている多くのコンテナがあります。しかし、これらのコンテナは非常に大きくなる可能性があります。最初にGoアプリの標準的なコンテナー構築方法を比較してから、非常に小さな静的なコンテナー化されたGoアプリを作成する方法を紹介します

パート1：「アプリケーション」


テストには、小さなアプリケーションが必要です。google.comをフォークしてHTMLサイズを出力しましょう。

package main

import (
    "fmt"
    "io/ioutil"
    "net/http"
    "os"
)

func main() {
    resp, err := http.Get("https://google.com")
    check(err)
    body, err := ioutil.ReadAll(resp.Body)
    check(err)
    fmt.Println(len(body))
}

func check(err error) {
    if err != nil {
        fmt.Println(err)
        os.Exit(1)
    }
}

開始すると、一部の数しか取得できません。私は約17Kを得ました。わざとSSLを使うことにしましたが、その理由は後で説明します。

パート2：ドッキング


公式のGoイメージを使用して、「onbuild」Dockerfileを記述します。

FROM golang:onbuild

「Onbuild」イメージは、プロジェクトが標準の構造であり、標準のGoアプリケーションを作成することを前提としています。さらに柔軟性が必要な場合は、標準のGoイメージを使用して自分でコンパイルできます。

FROM golang:latest 
RUN mkdir /app 
ADD . /app/ 
WORKDIR /app 
RUN go build -o main . 
CMD ["/app/main"]

ここでは、アプリケーションのビルドに使用するMakefileなどを作成すると便利です。 CDNから一部のリソースをロードするか、別のプロジェクトからインポートするか、またはコンテナーでテストを実行
することができます... ご覧のとおり、Goのドッキングは非常に簡単です。特に、サービスとポートを使用しないことを考えると、接続する必要があるもの。しかし、公式画像には重大な欠陥が1つあります -それらは本当に大きいです。見てみましょう：

REPOSITORY SIZE     TAG         IMAGE ID            CREATED              VIRTUAL SIZE
example-onbuild     latest      9dfb1bbac2b8        19 minutes ago       520.7MB
example-golang      latest      02e19291523e        19 minutes ago       520.7MB
golang              onbuild     3be7ee2ec1ae        9 days ago           514.9MB
golang              1.4.2       121a93c90463        9 days ago           514.9MB
golang              latest      121a93c90463        9 days ago           514.9MB

ベースイメージは514.9MBを占有し、アプリケーションはさらに5.8MBを追加します。コンパイルされたアプリケーションに515MBの依存関係が必要なのはなぜですか？
実際、アプリケーションはコンテナ内でコンパイルされています。つまり、コンテナにはGoをインストールする必要があります。したがって、彼はGoの依存関係と、パッケージマネージャーと本当にOS全体が必要です。実際、golang：1.4のDockerfileを見ると、Debian Jessieに付属しており、GCCコンパイラーとビルドツールをインストールし、Goをダウンロードしてインストールします。したがって、Debianサーバー全体と、小さなアプリケーションを起動するためのGoツールキットが手に入ります。これについて何ができますか？

パート3：コンパイル！


通常のアプローチから少し離れることで状況を改善できます。これを行うには、作業ディレクトリでGoをコンパイルしてから、バイナリをコンテナーに追加します。つまり、単純なドッカーアセンブリは機能しません。マルチステップのコンテナーアセンブリが必要です。

go build -o main .
docker build -t example-scratch -f Dockerfile.scratch .

そして、単純なDockerfile.scratch：

FROM scratch
ADD main /
CMD ["/main"]

スクラッチとは？スクラッチは、Dockerの特別な空の画像です。サイズは0Bです。

REPOSITORY          TAG         IMAGE ID            CREATED              VIRTUAL SIZE
example-scratch     latest      ca1ad50c9256        About a minute ago   5.60MB
scratch             latest      511136ea3c5a        22 months ago        0B

その結果、コンテナーは5.6 MBしか占有しません。いいね！しかし、1つの問題があります。

$ docker run -it example-scratch
no such file or directory

どういう意味ですか？Goバイナリが実行されているオペレーティングシステムのライブラリを探していることを理解するのにしばらく時間がかかりました。アプリケーションをコンパイルしましたが、それでも起動する必要があるライブラリ（つまり、すべてのCライブラリ）に動的にリンクされています。残念ながら、スクラッチは空なので、ライブラリやロードパスはありません。すべての組み込みライブラリを使用してアプリケーションを静的にコンパイルするには、ビルドスクリプトを変更する必要があります。

CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main .

静的バイナリを提供するcgoを無効にします。また、OSとしてLinuxを指定しています（誰かがMacまたはWindowsでLinuxをビルドした場合）。-aフラグは、使用するすべてのパッケージを再構築することを意味します。これにより、cgoを無効にしてすべてのインポートが再構築されます。これで静的バイナリができました。実行してみましょう：

$ docker run -it example-scratch
Get https://google.com: x509: failed to load system roots and no roots provided

これは何ですか？このため、この例ではSSLを使用することにしました。これは、このようなシナリオでは実際によくあることです。SSL要求を完了するには、ルートSSL証明書が必要です。では、それらをどのようにコンテナに追加するのでしょうか。
オペレーティングシステムによっては、証明書が別の場所にある場合があります。多くのLinuxディストリビューションでは、これは/etc/ssl/certs/ca-certificates.crtです。したがって、最初に、ca-certificates.crtをコンピューター（またはLinux仮想マシン、またはオンライン証明書プロバイダー）からリポジトリーにコピーします。次に、DockerfileにADDを追加して、このファイルをGoが予期する場所に移動します。

FROM scratch
ADD ca-certificates.crt /etc/ssl/certs/
ADD main /
CMD ["/main"]

次に、イメージを再作成して起動します。動作します！アプリケーションのサイズを見てみましょう：

REPOSITORY          TAG         IMAGE ID            CREATED              VIRTUAL SIZE
example-scratch     latest      ca1ad50c9256        About a minute ago   6.12MB
example-onbuild     latest      9dfb1bbac2b8        19 minutes ago       520.7MB
example-golang      latest      02e19291523e        19 minutes ago       520.7MB
golang              onbuild     3be7ee2ec1ae        9 days ago           514.9MB
golang              1.4.2       121a93c90463        9 days ago           514.9MB
golang              latest      121a93c90463        9 days ago           514.9MB
scratch             latest      511136ea3c5a        22 months ago        0B

半分メガバイトを少し追加しました（そして、そのほとんどは静的ファイルからのものであり、ルート証明書からのものではありません）。私たちは本当に小さなコンテナを手に入れました-レジストリ間を移動するのはとても便利でしょう。

結論


私たちの目標は、Goアプリケーションのコンテナーサイズを小さくすることでした。Goの特徴は、アプリケーションを完全に含む静的にリンクされたバイナリファイルを作成できることです。他の言語でも同じことができますが、決してすべてではありません。他の言語でコンテナーサイズを縮小するために同様の手法を適用することは、最小要件に依存します。たとえば、JavaまたはJVMアプリケーションをコンテナの外部でコンパイルし、JVM（およびその依存関係）のみを含むコンテナに埋め込むことができます。しかし、それでもJDKのコンテナよりは少なくなります。
