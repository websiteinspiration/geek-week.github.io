ツールを理解するには、最新のWeb開発の課題を理解する必要があります。これらの記事では、数十年にわたって存在していた古いWeb開発の問題から始めて、それぞれを個別に検討します。

旧問題：JavaScriptの制限


最近まで、JavaScriptとWeb APIには多くの制限がありました（さまざまな理由により省略）。

それらの一部を次に示します。

  • モジュールはありませんでした
  • 定数はありませんでした
  • 約束はありませんでした/非同期
  • Array.includes（）がありません（!!）
  • 多くの一般的なプリミティブのぎこちない/不足している構文（for-of、テンプレートリテラルなし、矢印関数構文、テンプレートアンパック...）
  • （Web API）無数のDOM操作が無意味に複雑でした（例：クラス名の追加/削除、要素の非表示、要素の選択、要素の削除...）

ブラウザはJavaScriptしか実行できないため、別の言語を使用して制限を克服することはできません。

別に：JavaScriptとWeb APIの違いは何ですか？


先ほど「JavaScriptとWeb API」と言ったことにお気づきでしょうか。これらは2つの異なるものです！

WebページのJavaScriptを作成する場合、Webページ自体と対話するAPI呼び出しはWeb API（たまたまJavaScriptで作成されたもの）を表しますが、JavaScript言語の一部ではありません。

例：

  • ウェブAPI：documentみんなmethodでdocument。windowそしてそれぞれmethodに window; Event、XMLHttpRequest、fetchそして上のようにします。D.
  • JavaScript：関数、const/ let/ var、配列Promiseなど

たとえば、Node.jsでサーバーを作成する場合、JavaScriptで作成すると、たとえばpromiseを使用できますが、それを使用することはできませんdocument.querySelector（これを行うのは意味がありません）。

古いソリューション：jQueryとその仲間


2006年にjQueryライブラリがリリースされ、JavaScriptとWeb APIの多くの欠点を回避するのに役立ちました。

jQueryには、DOMの操作、非同期処理、ブラウザー間の不整合の解決、リソースのフェッチなど、一般的なWebの問題の解決に大きく役立つAPIが含まれています。

つまり、本質的には、これはすべて古いJavaScriptと古いWeb APIを使用して技術的に可能でしたが、プロセスは非常に煩わしく、退屈で、開発が難しいことが多かったのです。たとえば、JSONファイルをダウンロードして処理するための面倒なコードを記述する代わりに、jQueryライブラリをダウンロードして、優れたjQuery APIを使用することができます。

新しい解決策：JavaScript自体を改善しましょう


しかし、2006年からずいぶん時間が経ちました！

それ以降、JavaScriptとWeb API は大幅に改善されました。これには、jQueryや他の方法を示した人々の助けも含まれます。

JavaScriptは進化を続ける言語です。ソフトウェアが更新されるのと同じように、JavaScript言語自体もバージョンごとに更新されます。

「ES6」という言葉を聞いたことがあるかもしれません。これは「ECMAScript 6」を意味し、ECMAScriptの6回目の反復を指します。 ECMAScriptはJavaScriptの別名です。口語表現では、通常、「ECMAScript」を使用して仕様自体を参照し、「JavaScript」を使用してコードを参照します。

（ちなみに、別の混乱と私のお気に入りのカルス：JavaScriptがありませんECMAScriptの実装/方言;これは、「HTML」を「HTML仕様」の実装/方言と呼んでいます。とにかく、これは間違っています！ウィキペディア、あなたは間違っている！ JavaScriptとECMAScriptは同じものです）。

ES6（2015年にリリースされた）は、constモジュールやプロミス（およびES8が私のお気に入りの言語機能を導入した:) など、JavaScriptに多くの本当に素晴らしい言語機能を追加することで注目に値しますasync。

並行して、2006年以降、Web APIはdocument.querySelector、fetch、およびclassListやhiddenなどの小さなものを追加することで大幅に改善されました。

したがって、2019年のjQueryまたは他の同様のライブラリの代わりに、ほとんどの場合、JavaScriptとWeb APIを直接使用するだけで済みます。

…そんな感じ！

新旧の問題：ブラウザー間のサポート


JavaScript言語の更新がリリースされたら、ブラウザーも更新して新機能をサポートする必要があります（Web APIについても同様ですが、簡単にするために、JavaScriptのみを残してください）。

ただし、1）言語での新しい関数の定義の間には遅延があります。2）すべてのブラウザでの機能の実装。3）すべてのユーザーのブラウザを更新する（これは決して発生しない可能性があります）。


ジレンマ：古いまたは最新のJavaScriptで記述しますか？両方のアプローチには長所と短所があります（特定のコード例はここから取得されます）

開発者にはジレンマがあります。もちろん、JavaScript言語の最新の機能を使用したいと考えています。これらの機能強化は、特定の事柄のコーディングを大幅に容易にすることが多いためです。ただし、ブラウザを最後に再起動して更新したかどうかに関係なく、すべてのユーザーがWebサイトを機能させることも必要です。

この特定のジレンマはBabelによって解決されます。

Babelは、JavaScriptコードを...他のJavaScriptコードに変換するJavaScriptコンパイラです！特に、最新バージョンのJavaScriptを使用して記述されたJavaScriptコードを、より多くのブラウザーでサポートされている古いバージョンのJavaScriptを使用して記述された同等のコードに変換します。


Babelを使用すると、ブラウザーの互換性を気にすることなく、最新のJavaScriptでコーディングするメリットを享受できます。

免責事項：BabelはWeb APIをサポートしていません


たとえばfetch、JavaScriptで使用する場合、それfetch はWeb APIであり、JavaScript自体の一部ではないため、このサポートは提供されません（「ポリフィリング」と呼ばれます）（この問題は現在解決中です）。

したがって、別のWeb APIポリフィリングソリューションが必要になります。しかし、次の記事でこれに戻ります。

  1. サイトを通常の静的Webページとして設計します。

サイトをインターネットで公開する準備ができたら、すべてのブラウザーでサポートされていないJavaScript機能を使用したため、そこに「バニラ」JavaScriptファイルをアップロードしないでください。

代わりに、あなたは欲しい：

  1. JavaScriptをbabelでコンパイルしてブラウザー互換コードを取得します。



    これにより、別のフォルダーに新しいコンパイル済みJavaScriptファイルが作成されます。例：Babelは2つ目の「script.js」を作成します。



  2. コンパイルしたJavaScriptをHTMLおよびCSSとともにインターネットにアップロードします


    。サイトの外観と動作は*開発モードと同じですが、すべてのブラウザーでbabelによってコンパイルされたコードがユーザーに提供されます。

    • これはバベルのないプロジェクトです：site / script.js
    • これはbabelを使用したプロジェクトです：site / script.js

    （*私は願っています！デバッグとリリースのビルドに違いがある場合がありますが、これらはバグです！）

開発と本番のコードの違い




「開発」コードと「リリース」（本番）コードが分離されていることに注意してください。

  • 開発コード：Webアプリケーションの開発中に作成するコード。
  • 製品コード：ユーザーがにアクセスしたときに実行するコード。

以下の理由により、これらを意図的に分離したいと考えています。

  • 開発コードは開発者には良いがユーザーには悪い
  • 量産コードはユーザーにとっては良いが、開発者にとっては悪い

フロントエンド開発では、誰もがBabelを使用する、または使用するべきではありません。

だが！全体像は次のとおりです。

  • ユーザーに表示されない開発コードを書く、
  • 代わりに、ユーザーに表示する必要がある別の製品コードにコンパイルします

これは単なる一般的なものではありませんが、現代のフロントエンド開発にしばしば期待される状況です。

（個別のデバッグアセンブリとリリースアセンブリの存在は、ソフトウェアエンジニアリングでは一般的なテンプレートであり、Web開発には特別なものではないことに注意してください。しかし、普及とフロントエンドの大きな違いの両方のため、フロントエンドに特に関連しています。デバッグ/リリースコード）。

開発バージョンと製品バージョンの間のそのような分離が予想されるテクノロジの短いリスト：

  • npmモジュール
  • CSSプリプロセッサ
  • React / Vue / Angular /任意のWebフレームワーク

これは一般的なテンプレートなので、今すぐ注意してください。
