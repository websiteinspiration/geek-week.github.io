C＃とC＃の関係：REST、gRPC、およびその間のすべて

C＃クライアントとC＃サーバーの間で通信するには多くの方法があります。信頼できるものもあれば、信頼できないものもあります。非常に高速なものもあれば、そうでないものもあります。どのオプションが最適かを判断できるように、さまざまなオプションを理解することが重要です。この記事では、これまでで最も人気のある技術と、なぜそれらがそれほど広く使用されているのかについて説明します。REST、gRPC、およびその間のすべてについて話します。

クライアントとサーバー間の通信を現実の世界でどのように見せるかを見てみましょう。私はこのようなものを提示します：

従来のRESTアプローチ


REST APIは2000年代初頭に登場し、インターネットを征服しました。現在、これはWebサービスを作成する最も一般的な方法です。

RESTは、クライアントからサーバーへの要求に対して、GET、POST、PUT、およびDELETE操作の固定セットを定義します。リクエストごとに、ペイロード（通常はJSON）を含むレスポンスを取得します。リクエストには、リクエスト自体に、またはPOSTまたはPUTリクエストの場合はペイロード（通常はJSON）としてパラメーターが含まれます。次のルールを定義

するRESTful API標準があります（実際には必要ありません）。

  • GETはリソースを取得するために使用されます
  • PUTは、リソースの状態を変更するために使用されます。
  • POSTは、リソースを作成するために使用されます。
  • DELETEは、リソースを削除するために使用されます。

これまでRESTに慣れていない場合、上記の説明はおそらく役に立たないので、ここに例を示します。.NETにはRESTサポートが組み込まれています。実際、ASP.NET Web APIはデフォルトでREST Webサービスです。典型的なASP.NETクライアントとサーバーは次のように

なります。サーバー上：

[Route("People")]
public class PeopleController : Controller
{
    [HttpGet]
    public Person GetPersonById(int id)
    {
        Person person = _db.GetPerson(id);
        return person;//Automatically serialized to JSON
    }
}

クライアント上：

var client = new HttpClient();
string resultJson = await client.GetStringAsync("https://www.myserver.com/People/GetPersonById?id=123");
Person person = JsonConvert.DeserializeObject<Person>(resultJson);

RESTは便利ですが、最適なシナリオには適していません。では、もっと上手くできるかどうか見てみましょう。

再フィット


ReFitは RESTに代わるものではありません。代わりに、RESTの上に構築され、サーバーエンドポイントを単純なメソッドのように呼び出すことができます。これは、クライアントとサーバーの間でインターフェースを共有することによって実現されます。サーバー側では、コントローラーがインターフェースを実装します。

public interface IMyEmployeeApi
{
    [Get("/employee/{id}")]
    Task<Employee> GetEmployee(string id);
}

次に、クライアント側で同じインターフェイスを有効にし、次のコードを使用する必要があります。


var api = RestService.For<IMyEmployeeApi>("https://www.myserver.com");
var employee = await api.GetEmployee("abc");

とても簡単です。いくつかのNuGetパッケージを除いて、複雑な自動化を実行したり、サードパーティのツールを使用したりする必要はありません。

これは最適なシナリオにかなり近づいています。これで、IntelliSenseとクライアントとサーバー間の信頼できる契約ができました。しかし、いくつかの点でさらに優れている別のオプションがあります。

Swagger


ReFitと同様に、SwaggerもRESTの上に構築されています。OpenAPIまたはSwaggerはREST API仕様です。シンプルなJSONファイルでREST Webサービスを記述します。これらのファイルは、WebサービスAPIスキーマです。以下が含まれます：

  • APIのすべてのパス（URL）。
  • 各パスに期待される操作（GET、POSTなど）。各パスは異なる操作を処理できます。たとえば、同じパスmystore.com/Productは、商品を追加するPOST操作と商品を返すGET操作を受け入れることができます。
  • 各パスと操作に必要なパラメーター。
  • 各パスの予想される回答。
  • 各パラメーターと応答オブジェクトのタイプ。

従来のREST対Swagger対ReFit


選択する際に考慮すべきいくつかのポイントを次に示します。

  • 非常に単純なプライベートREST APIを使用している場合は、クライアントの生成と共通のインターフェースについて心配する必要がない場合があります。小さなタスクは追加の努力を正当化しません。
  • Swaggerは多くの言語をサポートしており、ReFitは.NETのみをサポートしています。Swaggerは、多くのツール、テスト、自動化ツール、およびユーザーインターフェイスツールの基盤でもあります。大規模なパブリックAPIを作成する場合は、これがおそらく最良の選択です。
  • SwaggerはReFitよりもはるかに複雑です。ReFitを使用すると、サーバープロジェクトとクライアントプロジェクトの両方に単一のインターフェイスを追加するだけです。一方、ReFitでは、コントローラーごとに新しいインターフェイスを作成する必要がありますが、Swaggerはこれを自動的に処理します。

しかし、何かを決める前に、RESTとは何の関係もない4番目のオプションを確認してください。

gRPC


gRPC（gRPC-Remote Procedure Call）は、Googleが開発したオープンソースのリモートプロシージャコールシステムです。これは、クライアントからサーバーにリクエストを送信する方法を提供するという意味で、RESTに少し似ています。ただし、これは非常に異なります。類似点と相違点は次のとおりです。

  • RESTと同様に、gRPCは言語に依存しません。C＃を含むすべての一般的な言語用のツールがあります。
  • gRPCは契約ベースで、.protoファイルを使用して契約を定義します。これは、Swagger swagger.jsonおよび一般的なReFitインターフェースに多少似ています。これらのファイルから任意のプログラミング言語のクライアントを生成できます。

.NETサーバー側のGRPC


これはガイドではなく、何を期待するかについての一般的な考え方です。以下は、gRPCでのコントローラの例です。


public class GreeterService : Greeter.GreeterBase
{
    public override Task<HelloReply> SayHello(HelloRequest request,
        ServerCallContext context)
    {
        _logger.LogInformation("Saying hello to {Name}", request.Name);
        return Task.FromResult(new HelloReply 
        {
            Message = "Hello " + request.Name
        });
    }
}

Startup.csファイルのConfigureに 以下を追加する必要があります。


app.UseEndpoints(endpoints =>
{
    endpoints.MapGrpcService<GreeterService>();
});

APIは、プロジェクトの一部である.protoファイルに記述されています。

syntax = "proto3";
 
service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}
 
message HelloRequest {
  string name = 1;
}
 
message HelloReply {
  string message = 1;
}

この.protoファイルは.csprojファイルに追加されます。

<ItemGroup>
  <Protobuf Include="Protos\greet.proto"/>
</ItemGroup>

.NETクライアント側のGRPC


クライアントは.protoファイルから生成されます。コード自体は非常に単純です。


var channel = GrpcChannel.ForAddress("https://localhost:5001");
var client = new Greeter.GreeterClient(channel);
 
var response = await client.SayHello(
    new HelloRequest { Name = "World" });
 
Console.WriteLine(response.Message);

gRPCとREST


GRPCはいい取引のようです。フードの下でより速く、より簡単です。それでは、全員をRESTからgRPCに切り替える必要がありますか？答えは、場合によります。ここにいくつかの考慮事項があります：

私の経験では、gRPCとASP.NETの操作はまだ小さいです。成熟したRESTのサポートを利用する方がよいでしょう。コントラクト通信に関する限り、これは良いことですが、RESTの代わりに、SwaggerとReFitについてすでに説明したRESTに似た方法があります。

最大の利点はパフォーマンスです。ほとんどの場合、これらの基準によると、gRPCははるかに高速です。特に、Protobufシリアル化が本当に重要な大きなペイロードの場合。これは、これが高負荷のサーバーにとって大きな利点であることを意味します。

大規模なASP.NETアプリケーションでRESTからgRPCに移行することは困難です。ただし、マイクロサービスベースのアーキテクチャを使用している場合、この移行は徐々に完了するのがはるかに容易になります。
