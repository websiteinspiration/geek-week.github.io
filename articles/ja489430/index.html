CI / CDに切り替えるときによくある7つの間違い


会社がDevOpsまたはCI / CDツールのみを実装している場合、それらを繰り返さないようにし、他の人のレーキを踏まないようにするために、最も一般的なエラーに精通することが役立つ場合があります。 Mail.ruクラウドソリューション

チームがいる翻訳の追加とジャスミンChokshiの記事でCI / CDへの移行時には避けてくださいこれらの陥りやすい落とし穴を。

文化やプロセスを変える準備ができていない


DevOps の循環図を見ると、DevOpsプラクティスでは、テストは継続的な作業であり、個々のデプロイメントの基本的な部分であることがわかります。


無限の循環図DevOps

テストと開発および提供プロセス中の品質保証は、開発者が行うすべての不可欠な部分です。これには、すべてのタスクにテストを含めるための考え方の変更が必要です。

テストは、各チームメンバーの日常業務の一部になります。進行中のテストへの移行は容易ではありません。これに備える必要があります。

フィードバックの欠如


DevOpsの効果は、絶え間ないフィードバックに依存します。連携やコミュニケーションの余地がないと継続的な改善はできません。

遡及的な会議を組織していない企業は、CI / CDに継続的なフィードバックの文化を導入することは困難です。各回の終わりに回顧会議が開催され、そこでグループメンバーは何がうまくいったか、何がうまくいかなかったかについて話し合います。回顧会議はスクラム/アジャイルの基盤ですが、DevOpsにも必要です。 

これは、遡及的な会議でレビューや意見を交換する習慣が植え付けられているためです。開始時に最も重要な瞬間の1つは、定期的なレトロミーティングの開催です。これにより、チーム全体が理解し、親しみやすくなります。

ソフトウェアの品質に関しては、すべてのチームメンバーがそれを維持する責任があります。たとえば、開発者は単体テストを記述できるだけでなく、テスト容易性に基づいてコードを記述できるため、最初からリスクを軽減できます。

テストに対する認識の変化を反映する最も簡単な方法の1つは、テスターに​​QAではなくソフトウェアテスターまたは品質エンジニアを呼ぶことです。この変更は単純すぎる、または愚かに思えるかもしれません。しかし、誰かが「ソフトウェア品質保証スペシャリスト」と呼ばれる場合、それは誰が製品の品質に責任があるかという誤った印象を与えます。アジャイル、CI / CD、およびDevOpsプラクティスでは、誰もがソフトウェアの品質に責任があります。

もう1つの重要な点は、チーム全体とそのメンバー、組織、および利害関係者のそれぞれにとって品質とは何かを理解することです。

ステージ完了の誤った理解


品質が継続的で一般的なプロセスである場合、ステージの完全性について共通の理解が必要です。ステージが終わったことをどのように理解するのですか？Trelloボードまたは他のかんばんボードでステージが完了としてマークされた場合はどうなりますか？

完了したステージの決定（DoD）は、CD DevOps / CIのコンテキストにおける強力なツールです。これは、チームが何をどのように構築しているかの品質基準をよりよく理解するのに役立ちます。

開発チームは、「終了」の意味を決定する必要があります。彼らは座って、各段階で満たされなければならない特性のリストを作成して、完全であると見なすことができるようにする必要があります。

DoDは、プロセスをより透明にし、CI / CDの実装を容易にします（すべてのチームメンバーが理解でき、相互に合意した場合）。

現実的で明確な目標の欠如


これは最も頻繁に引用されるヒントの1つですが、繰り返す価値があります。 CI / CDやDevOpsの実装など、深刻な取り組みを成功させるには、実際の目標を設定し、それらに対するパフォーマンスを測定する必要があります。 CI / CDで何を達成しようとしていますか？より良い品質のリリースをより早くリリースできるようにしますか？

目標は、透明性と現実性だけでなく、会社の現在の活動と一致している必要があります。たとえば、顧客はどのくらいの頻度で新しいパッチまたはバージョンを必要としますか？ユーザーに追加のメリットがない場合は、プロセスをオーバーロードしてリリースをより早くリリースする必要はありません。

さらに、必ずしもCDとCIの両方を実装する必要はありません。たとえば、銀行や医療クリニックなどの規制の厳しい企業は、CIとしか連携できません。

CIは、DevOpsを実装するすべての企業にとって良い出発点です。会社への導入に伴い、ソフトウェアの供給に対するアプローチは大きく変化しています。CIをマスターしたら、プロセス全体の改善、ロールアウト速度の向上、およびその他の変更について検討できます。

多くの組織にとって、単一のCIで十分であり、CDは、付加価値をもたらす場合にのみ実装する必要があります。

適切なダッシュボードとメトリックの欠如


目標を設定したら、開発チームはKPIを測定するためのダッシュボードを作成できます。その開発の前に、監視されるパラメータを評価する価値があります。

さまざまなレポートとアプリケーションが、さまざまなチームメンバーに役立ちます。スクラムマスターは、ステータスとリーチにより関心があります。上級管理職は専門家の燃え尽き率に関心があるかもしれませんが。

一部のチームは、赤、黄、緑のインジケーターが付いたダッシュボードを使用してCI / CDのステータスを評価し、すべてが正しく行われているか、エラーが発生したかを理解します。赤は何が起こっているかに注意を払う必要があることを意味します。

ただし、ダッシュボードが標準化されていない場合、誤解を招く可能性があります。誰もが必要とするデータを分析し、それが意味することの標準化された説明を作成します。グラフィック、テキスト、または数字など、利害関係者にとってより意味のあるものを見つけます。

手動テストなし


テスト自動化は、優れたCI / CDパイプラインの基礎を築きます。ただし、すべての段階で自動テストを実行しても、手動テストを実行すべきではありません。 

効率的なCI / CDパイプラインを構築するには、手動テストも必要です。人間による分析を必要とするテストには常にいくつかの側面があります。

手動テストの取り組みをパイプラインに統合することを検討する価値があります。一部のテストケースの手動テストが完了したら、展開フェーズに進むことができます。

テストを改善しようとしないでください。


効果的なCI / CDパイプラインでは、テスト管理であれ、統合と継続的な監視であれ、適切なツールへのアクセスが必要です。

強力で品質志向の文化を築くことは、テストの導入、導入後の顧客との対話の監視、改善の追跡を目的としています。 

簡単に実装できる実用的なヒントをいくつか紹介します。

  1. テストを記述しやすく、コードをリファクタリングするときに壊れないように十分な柔軟性があることを確認してください。
  2. テストプロセスに開発チームを含める必要があります。CIパイプラインでの検証に重要なユーザーの問題とクエリのリストを確認します。
  3. あなたは完全なテストカバレッジを持っていないかもしれませんが、常にUXと顧客の相互作用にとって重要なスレッドがテストされることを確認してください。

最後だが大事なことは


CI / CDへの移行は通常、ボトムアップで開始されますが、最終的には、企業側の管理、時間、およびリソースの参加を必要とする変革です。結局のところ、CI / CDは一連のスキル、プロセス、ツール、文化的再構築であり、そのような変更を体系的に導入することのみが可能です。
