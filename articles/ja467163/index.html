Kubernetesと自動化により2時間でクラウドに移行する方法



URUSは、さまざまな形式でKubernetesを試しました：Google Cloudでのスタンドアロンのベアメタルデプロイメント。その後、プラットフォームをMail.ru Cloud Solutions（MCS）に移動しました。Igor Shishkinは、彼らが新しいクラウドプロバイダーをどのように選択したか、そして2時間でこれまでどのようにしてそれに移行したかを説明します（t3ran）、Urus上級システム管理者。

Urusの機能


都市環境の質を改善する方法はたくさんありますが、そのうちの1つは環境に優しい方法にすることです。これはまさに、URUS-スマートデジタルサービス会社が取り組んでいることです。これは、企業が重要な環境指標を制御し、環境への悪影響を軽減するのに役立つソリューションを実装しています。センサーは、空気の組成、騒音レベル、その他のパラメーターに関するデータを収集し、それらを単一のプラットフォーム「Urus-Ekomon」に送信して、分析と推奨の準備を行います。

中の「ウルス」の作品はいかがですか


Urusの典型的なクライアントは、住宅地の中または近くにある会社です。工場、港湾、鉄道基地、またはその他の施設です。お客様がすでに警告を受けている、環境汚染の罰金を科されている、または騒音を減らし、有害な排出量を削減したい場合、彼は私たちのところにやって来て、私たちはすでに環境モニタリングのための既製のソリューションを提供しています。


H 2 S 濃度のモニタリンググラフは、近くの企業の定期的な夜間の排出量を示し

ています。URUSで使用するデバイスには、特定のガスの内容、ノイズレベル、および環境状況を評価するためのその他のデータに関する情報を収集するいくつかのセンサーが含まれています。センサーの正確な数は、常に特定のタスクによって決定されます。


特定の測定に応じて、センサーを備えたデバイスを建物の壁、柱、その他の任意の場所に配置できます。このような各デバイスは、情報を収集して集約し、データ受信ゲートウェイに送信します。そこで、長期保存とその後の分析のための前処理のためにデータを保存します。分析後に得られる最も単純な例は、大気質指数、別名AQIです。

同時に、他の多くのサービスがプラットフォームで動作しますが、基本的にはサービスの性質を持っています。たとえば、監視サービス（CO 2の含有量など）のいずれかが許容値を超えた場合、通知サービスは顧客に通知を送信します。

データをどのように保存するか。ベアメタル上のKubernetesのストーリー


URUSエコモニタリングプロジェクトには、いくつかのデータウェアハウスがあります。 1つには、デバイス自体から直接受け取った「生」データを保持します。このストレージは、古いカセットと同様に、すべてのインジケーターの履歴を持つ「磁気」テープです。 2番目のタイプのストレージは、前処理されたデータに使用されます-センサー接続およびデバイス自体の読み取りに関するメタデータが豊富なデバイスからのデータ、組織、場所などとの提携。この情報により、特定のインジケーターが一定期間にどのように変化したかを動的に評価できます。バックアップなどの生データストレージを使用し、必要に応じて前処理されたデータを復元します。

数年前にストレージの問題を解決する方法を模索したとき、プラットフォームを選択するためのオプションとして、KubernetesとOpenStackの2つがありました。しかし、後者はかなり怪しいように見えるので（これを見るには、そのアーキテクチャを見てください）、Kubernetesに落ち着きました。彼が好んだもう1つの論点は、比較的単純なプログラム制御であり、鉄のノードでさえもリソースにさらに柔軟にカットできることでした。

Kubernetes自体の開発と並行して、データを格納する方法も検討しましたが、すべてのストレージをKubernetesのハードウェアに保持しながら、優れた専門知識を得ました。私たちがKubernetesに住んでいたすべてのもの：ステートフルストレージ、モニタリングシステム、CI / CD。 Kubernetesはオールインワンプラットフォームになりました。

しかし、私たちはサービスとしてKubernetesと連携し、そのサポートと開発に従事したくありませんでした。さらに、ベアメタル上のコンテンツのコストが気に入らなかったため、常に開発が必要でした。たとえば、最初のタスクの1つは、Kubernetes Ingressコントローラーを組織のネットワークインフラストラクチャに統合することでした。これは、特にその時点でDNSレコードやIP割り当てなどのプログラムによるリソース管理の準備ができていないことを想像すると、面倒な作業です。その後、外部データウェアハウスの実験を開始しました。PVCコントローラーの実装には至りませんでしたが、それでも、これは大きな作業分野であり、個々の専門家を選抜する必要があることが明らかになりました。

Google Cloud Platform一時ソリューションへの移行


これ以上続行できないことに気付き、データをベアメタルからGoogle Cloud Platformに転送しました。実際、ロシアの会社にとって興味深いオプションはそれほど多くありませんでした。GoogleCloud Platformに加えて、Amazonだけが同様のサービスを提供していましたが、それでもGoogleのソリューションで解決しました。そして、Google自体が一種のPoC Kubernetes in Productionであるという事実は言うまでもなく、上流に近いほど経済的に収益性が高いように見えました。

最初の深刻な問題は、私たちの顧客基盤の拡大と並行して地平線上に現れました。個人データを保存する必要があったとき、Googleと協力してロシアの法律に違反するか、ロシア連邦で代替案を探しているかのどちらかを選択する必要がありました。一般的に、選択は予測可能でした。 :)

完璧なクラウドサービスの見方


検索を開始するまでに、将来のクラウドプロバイダーから何を取得したいかはすでにわかっていました。私たちが探していたサービス：

  • 高速かつ柔軟。新しいノードをすばやく追加したり、いつでも何かをデプロイしたりできるようにします。
  • 安価です。資源に限りがあったので、財政問題を非常に心配しました。Kubernetesを使用したいことはすでにわかっていましたが、今ではこのコストを最小限に抑えて、このソリューションの使用効率を上げるか、少なくとも維持することが課題でした。
  • 自動化。マネージャーや電話をかけたり、緊急モードで数十のノードを手動で起動する必要がない状況ではなく、APIを介してサービスを操作することを計画しました。ほとんどのプロセスは自動化されているため、クラウドサービスにも同じことが期待されています。
  • ロシア連邦のサーバーで。もちろん、私たちはロシアの法律と非常に152-FZに準拠することを計画しました。

当時、ロシアではKubernetesのaaSプロバイダーはほとんどいませんでした。プロバイダーを選択するときは、優先順位を妥協しないことが重要でした。作業を開始し、現在も作業を行っているMail.ruクラウドソリューションチームは、APIサポートを備えた完全に自動化されたサービスと、Horizo​​nがある便利なコントロールパネルを提供してくれました。これにより、任意の数のノードをすばやく増やすことができました。

2時間でMCSに移行する方法


そのような移転では、多くの企業が困難と失敗に直面していますが、私たちの場合はそうではありませんでした。幸運でした。移行を開始する前に既にKubernetesで作業していたため、MCSで3つのファイルを修正し、新しいクラウドプラットフォームでサービスを開始しました。その時までに、私たちはついにベアメタルを離れ、Google Cloud Platformに住んでいたことを思い出させてください。移動自体にかかった時間は2時間以内で、デバイスからデータをコピーするのに少し時間がかかった（約1時間）ためです。次に、Spinnaker（Continous Deliveryを提供するマルチクラウドCDサービス）をすでに使用しました。また、すぐに新しいクラスターに追加し、通常どおり機能しました。

開発プロセスの自動化とCI / CDのおかげで、URUSのKubernetesは1人の専門家（そして私です）によって占められています。ある段階で、別のシステム管理者が私と一緒に作業しましたが、メインルーチン全体をすでに自動化していることがわかりました。また、メイン製品の部分でますます多くのタスクがあり、リソースをこれに向けることは理にかなっています。

幻想なく協力を始めて以来、クラウドプロバイダーから期待通りの結果を得ました。インシデントがあった場合は、ほとんどが技術的なインシデントであり、サービスの相対的な鮮度によって簡単に説明できるものです。主なことは、MCSチームが欠陥をすばやく排除し、インスタントメッセンジャーの質問にすばやく対応することです。

エクスペリエンスをGoogle Cloud Platformと比較すると、フィードバックボタンがどこにあるかさえわからなかったのです。なぜなら、それは単に必要なかったからです。問題が発生した場合、Google自体が一方的に通知を送信しました。しかし、大きなプラスであるMCSの場合、地理的にも精神的にも、彼らはロシアの顧客にできるだけ近いと思います。

現在、同じクラウドプラットフォームであるサードパーティソリューションの使用は、若い企業にとって唯一の正しいソリューションであると考えています。通常、旅の最初の段階では、人的および経済的なリソースが限られているため、独自のクラウドまたはデータセンターの構築と維持には非常にコストと時間がかかります。クラウドプロバイダーは、これらのコストを最小限に抑えることができ、サービスが今ここで機能するために必要なリソースをすばやく取得し、実際にこれらのリソースに支払うことができます。 Urus社については、今のところ、クラウドでのKubernetesに忠実であり続けます。しかし、誰が知っているか、地理的に拡大するか、特定の機器に基づいたソリューションを実装する必要があるかもしれません。または、消費されたリソースの量が、古き良き時代のように、ベアメタル上の独自のKubernetesを正当化するかもしれません。 :)

クラウドサービスの経験から学んだこと


ベアメタルでKubernetesを使い始めましたが、それでも独自の方法で良好でした。しかし、その強みはクラウドのaaSコンポーネントとして正確に明らかにされました。目標を設定し、可能な限りすべてを自動化すると、ベンダーロックインを回避でき、クラウドプロバイダー間を移動するのに数時間かかり、神経細胞が残ります。他の会社に助言することができます：（クラウド）サービスを開始したい場合、限られたリソースと最大の開発速度で、今すぐクラウドリソースをレンタルして開始し、Forbesがあなたのことを書いた後にデータセンターを構築します。
