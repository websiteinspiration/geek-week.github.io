「インサイト主導」：機械学習モデルの高度な分析とライフサイクル管理

今日は、Insight-Drivenの概念と、DataOpsとModelOpsを使用してそれを実践する方法について説明します。洞察主導のアプローチは、最近作成されたデータ管理に関する有用な資料のライブラリで詳しく説明する包括的なトピックです（リンクは下にあります）。今日のハブラトピカでは、次のように機械学習モデルのライフサイクルの主要な段階に焦点を当てます これはコンセプト内の主要なトピックの1つです。



インサイト主導型アプローチの本質は何ですか


多くの専門家がデータドリブンの重要性について長い間話していましたが、もちろんこれは一般的に完全に正しいことです。なぜなら、このアプローチには、直観や個人的なリーダーシップの経験だけでなく、データを分析することによって管理上の意思決定をより効率的にすることが含まれるためです。アナリストフォレスターに注意してください彼らの活動にデータ分析に依存している企業は、より速く、競合他社よりも平均30％で成長すること。

しかし、会社がデータ自体の可用性からではなく、データと連携して機能すること、つまり、収益化できる洞察を見つけ、データを蓄積、処理、分析する価値があることから、会社が前進していることは誰もが理解しています。したがって、データ駆動型のより高度なバージョンとして、洞察駆動型アプローチについて具体的に話します。

ほとんどの場合、データを扱う場合、ほとんどのスペシャリストは主に企業内の構造化された情報を意味しますが、ビジネスの大部分が潜在的に利用可能なデータの約80％を使用しない理由についてはそれほど前に話しませんでした。Insight-Drivenは、画像を外部の非構造化情報で補足するための基礎を作成するだけでなく、それらの間の暗黙的な依存関係を検索するためのデータ解釈の結果も作成します。

DevOps + DataOps + ModelOps


インサイト主導のプラクティスは、DevOps、DataOps、ModelOpsに基づいています。これらの特定のプラクティスの組み合わせがアプローチの完全な実装を保証できる理由について話しましょう。



DevOps + DataOps。 DevOpsには、製品のリリースと更新の時間を短縮し、バージョン管理、継続的な統合、テストとモニタリング、リリース管理のためのツールを使用することで、サポートのコストを最小限に抑えることが含まれます。これらのプラクティスに加えて、社内のデータの理解、そのフォーマットと構造の管理方法、タグ付け、品質の追跡、変換、集約、および迅速な分析と視覚化の機能を備えている場合、DataOpsが得られます。。このアプローチの焦点は、意思決定支援、洞察検索、予測を提供する機械学習モデルを使用したシナリオの実装です。

ModelOps。会社が機械学習モデルを積極的に使用し始めるとすぐに、モデルの管理、品質指標の監視、再トレーニング、比較、更新、バージョン管理が必要になります。ModOpsは、そのようなモデルのライフサイクル管理を簡素化する一連のプラクティスとアプローチです。これは、ストリーミングサービスなど、ビジネスのさまざまな分野で多数のモデルを扱う企業によって使用されます。

企業に洞察主導のアプローチを実装することは、簡単なことではありません。しかし、彼と一緒に働き始めたい人のために、これを行う方法を説明します。

検索とデータ準備


インサイト主導のプラクティスの実装は、データの検索と準備から始まります。後でそれらは分析され、MOモデルを構築するために使用されますが、ケースは事前に決定されており、インテリジェントアルゴリズムが役立ちます。

タスクの定義。この段階で、会社はビジネス目標を設定します。たとえば、市場での利益の増加です。次に、新しい顧客の数の増加、平均小切手のサイズ、コンバージョンの割合など、ビジネス指標を決定してそれを達成します。したがって、関連データを検索することがすでに可能であるシナリオがあります。



ソーシングとデータ分析。データ取得の目標と方向が定義されたら、ソースを分析します。トレーニングテイクに関連するインテリジェントシナリオの開発のこれ以降の段階実装における企業の予算の70〜80％。実際のところ、データセットの品質は、設計された機械学習モデルの精度に影響を与えます。しかし、必要な情報は多くの場合、さまざまなシステムに「散在」しています。Hadoopプラットフォーム上のMS SQL、Oracle、PostgreSQLなどのリレーショナルデータベースやその他の多くのソースに存在する可能性があります。そして、この段階で、関連データがどこにあり、どのように収集するかを理解する必要があります。

多くの場合、アナリストはすべてを手動でアンロードして処理するため、プロセスが大幅に遅くなり、エラーのリスクが高まります。 SAPは、適切なソースに接続し、要求に応じてデータを収集するメタシステムを実装するようお客様に提供しています。
したがって、すべてのテーブル、非構造化データを含む外部プール、およびその他のソースをカタログ化し、タグ（階層タグを含む）を設定して、関連情報をすばやく収集できます。条件付きで、クライアントの情報が異なるデータベースにある場合、これらのエンティティを示すだけで十分です。次に「クライアントデータセット」が必要になったときに、既製のショーケースを選択します。

データソースが特定されたら、データ品質の追跡とプロファイリングに進むことができます。この操作は、ギャップの数、一意の値を理解し、データの全体的な品質を検証するために必要です。これらすべてについて、ルールを使用してダッシュボードを作成し、変更を追跡できます。

データ変換。次のステップは、タスクを解決する必要があるデータを直接処理することです。これを行うには、データをクリアします。チェック、重複排除、ギャップを埋めます。このプロセスは、フローベースのプログラミングで簡略化できます。この場合、一連の操作、つまりパイプラインを扱っています。その出力は、その後の作業のためにグラフィカルインターフェイスまたは他のシステムに送信できます。ここでは、データハンドラーがコンストラクターとして（シナリオによっては）組み立てられています。これは、定期的またはストリーミング処理、またはRESTサービスです。



フローベースのプログラミングの概念は、販売の予測やサービスの質の評価から顧客の流出の理由の発見まで、幅広いタスクを解決するのに適しています。 SAPでデータを検索および準備するための2つのツールがあります。最初-SAP Data Intelligenceデータ分析の専門家向け。同様のプラットフォームとは異なり、このソリューションは分散データで機能し、一元化を必要としません。モデルの実装、公開、統合、スケーリング、およびサポートのための単一の環境を提供します。2番目のツールはSAP Agile Data Preparationです。これは、アナリストとビジネスユーザーを対象とした小規模なデータ準備サービスです。データセットの収集、フィルタリング、処理、およびマップ情報を支援するシンプルなインターフェースを備えています。セルフサービスBI-分析シナリオを作成するためのセルフサービスシステムを転送するためのショーケースで公開できます（データサイエンスの分野での深い知識は必要ありません）。

モデル作成


準備が整ったら、機械学習モデルを作成する番です。ここで区別されます：研究、プロトタイピングと生産性。最後の段階には、モデルのトレーニングと適用のためのパイプラインの実装が含まれます。

研究とプロトタイピング。現在、多くのテーマ別フレームワークとライブラリが利用可能です。使用頻度のリーダーは、その人気を一年間ましたTensorFlowとPyTorch、ある成長 243パーセントで。 SAPプラットフォームでは、これらのフレームワークを使用でき、YandexのCatBoost、MicrosoftのLightGBM、scikit-learn、pandasなどのライブラリを柔軟に追加できます。HANA DataFrameを引き続き使用できますhanamlライブラリ内。このAPIはパンダを模倣しており、HANAでは「レイジーコンピューティング」を使用して大量のデータを処理できます。

プロトタイピングモデルについては、Jupyter Labを提供しています。これは、データサイエンスの専門家向けのオープンソースツールです。機能を拡張しながら、SAPエコシステムに組み込みました。 Jupyterラボはデータインテリジェンスプラットフォームで動作し、組み込みのsapdiライブラリにより、前のステップで接続された任意のデータソースに接続し、実験と品質メトリックを監視してさらに分析できます。

それとは別に、ノートブック、データセット、学習パイプライン、および推論は注目に値します、およびモデルの展開のためのサービスは一貫している必要があります。これらすべてのオブジェクトを組み合わせるには、MLスクリプト（バージョン付きオブジェクト）を使用します。

モデルトレーニング。 MLスクリプトの操作には2つのオプションがあります。まったくトレーニングする必要がないモデルがあります。たとえば、SAP Data Intelligenceでは、顔認識システム、自動翻訳、OCR（光学式文字認識）などを提供しています。それらはすべてそのまま使用できます。一方、トレーニングを受けて生産性を高める必要のあるモデルもあります。このトレーニングは、データインテリジェンスクラスター自体と、計算の間のみ接続されている外部コンピューティングリソースの両方で行うことができます。

SAP Data Intelligenceの「内部」はKubernetesプラットフォームであるため、すべてのオペレーターはDockerコンテナーに関連付けられています。モデルを操作するには、dockerファイルを記述して、使用するライブラリとバージョンのタグを添付するだけで十分です。

モデルを作成する別の方法は、AutoMLを使用することです。これらは自動化されたMOシステムです。このようなツールは、H2O、Microsoft、Googleなどの開発です。彼らはMITでこの方向に働いています。。しかし、大学のエンジニアは埋め込みと生産性に焦点を合わせていません。 SAPには、迅速な結果に焦点を当てたAutoMLシステムもあります。彼女はHANAで働いており、データに直接アクセスできます。データを移動したり変更したりする必要はありません。現在、モデルの品質に焦点を当てたソリューションを開発しています-後でリリースを発表します。

ライフサイクル管理。条件が変化し、情報が古くなるため、MOモデルの精度は時間とともに低下します。したがって、新しいデータが蓄積されたので、モデルを再トレーニングして結果を絞り込むことができます。たとえば、ある大手飲料メーカーは情報を使用していますスマートシステムの再トレーニングに関する200か国の消費者の好み。同社は、人々の好み、砂糖の量、飲み物のカロリー、さらには競合するブランドがターゲット市場で提供している製品さえも考慮に入れています。MOモデルは、企業が特定の地域で最も受け入れやすい数百の製品の中から自動的に決定します。


SAP Data Hubのオペレーターに基づくコンポーネントの再利用

しかし、新しいアルゴリズムとハードウェアコンポーネントの更新が利用可能になると、モデルのバージョン管理と更新も行う必要があります。それらの実装により、作業で使用されるモデルの精度と品質を向上させることができます。
