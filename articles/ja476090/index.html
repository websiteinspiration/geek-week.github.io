ブロックチェーン上の安全な2FAに移動します

SMSメッセージは、最も一般的な2要素認証（2FA）方式です。銀行、電子ウォレット、暗号ウォレット、メールボックス、あらゆる種類のサービスで使用されています。メソッドのユーザー数は100％に近づいています。

この方法は安全ではないので、私はそのような事件の発生がとんでもないです。あるSIMカードから別のSIMカードへの番号の再割り当ては、モバイル時代の初めに始まりました-これは、SIMカードが紛失したときに番号が復元される方法です。 「デジタルマネーウィーニングスペシャリスト」が実現：オプション「上書きSIMカード」は詐欺的なスキームで使用できます。結局のところ、SIMカードを制御する人は、他の人のオンラインバンキング、電子財布、さらには暗号通貨さえも管理できます。また、詐欺文書や偽造文書を使用して電気通信の従業員に賄賂を贈ることにより、別の人の番号を乗っ取ることができます。



何千ものSIMスワップエピソードが明らかになりました-これがこの詐欺スキームと呼ばれています。災害の規模は、世界がSMSを介して2FAを間もなく放棄することを示唆しています。しかし、これは起こりません- 調査によると、2FAメソッドを選択するのはユーザーではなく、サービスの所有者です。

ブロックチェーン経由でワンタイムコードを配信するセキュアな2FA方式を使用することをお勧めします。これをサービスオーナーに接続する方法を説明します。

SIMスワップの仕組み


スワップは交換を意味します。このようなすべてのスキームで、犯罪者は被害者の電話番号を割り当て、通常はSIMカードを再発行し、それを使用してパスワードをリセットします。理論上の典型的なSIMスワップは次のようになります。

  1. . : . (, ) — .
  2. . SIM- ; , , .
  3. , . - .

実際には、それはさらに深刻です。攻撃者は被害者を選択し、電話の位置を毎日監視します-加入者がローミングに切り替えたという情報の1つの要求は1〜2セントです。SIMカードの所有者が海外に出かけるとすぐに、彼らはコミュニケーションサロンのマネージャーと新しいSIMカードを発行することに同意します。それは約50ドルかかります（私は情報を見つけました-異なる国でそして異なるオペレーターから20ドルから100ドルまで）が、マネージャーは最悪の場合解雇されます-これに対する責任はありません。

これですべての侵入者がすべてのSMSを受信し、電話の所有者はそれについて何もできなくなります-彼は海外にいます。そして、悪者は被害者のすべてのアカウントにアクセスし、必要に応じてパスワードを変更します。

盗まれた商品を返品する可能性


銀行は時々被害者に向かって行き、彼らの口座から送金を引き出します。したがって、犯罪者が見つからなくても、法定金額を返すことができます。しかし、暗号通貨ウォレットを使用すると、技術的および合法的にすべてがより複雑になります。これまでのところ、交換/ウォレットは犠牲者を交換するための補償を支払っていません。

被害者が法廷でお金を弁護したい場合、彼らはオペレーターを非難します。彼は口座からお金を盗むための条件を作成しました。これは、スワップのために2億2,400万ドルを失ったマイケルターピンがしたこととまったく同じで、現在AT＆T通信会社を訴えています。



これまでのところ、暗号通貨の所有者を法的に保護するための仕組みが存在する州はありません。あなたの資本に保険をかけたり、その損失を補償することは不可能です。したがって、その結果に対処するよりも、スワップ攻撃を防ぐ方が簡単です。最も明白な方法は、2FAに対してより信頼性の高い「第2要素」を使用することです。

SMSを介した2FAの問題はSIMスワップだけではありません


SMSの確認コードも、技術的な観点から安全ではありません。Signaling System 7（SS7）の致命的な脆弱性が原因で、メッセージが傍受される可能性があります。SMSによる2FAは、安全でないと公式に認識されています（米国国立標準技術研究所は、デジタル認証ガイドでこれについて説明しています）。

さらに、2FAの存在は、多くの場合、ユーザーに誤ったセキュリティの感覚を与え、より単純なパスワードを選択します。したがって、このような認証は複雑ではありませんが、アカウントへの攻撃者のアクセスを容易にします。

また、SMSには長い遅延が発生したり、まったく送信されなかったりすることがよくあります。

ブロックチェーンの2FAをサービスに接続する方法


何人かの読者がサービスにブロックチェーン認証を追加することに興味を持っていただけたと思います。

私たちのメッセンジャーを例にしてこれを行う方法を説明します。類推すると、別のブロックチェーンを使用できます。2FAデモアプリケーションでは、postgresql10を使用してアカウント情報を格納します。

接続手順：

  1. 2FAコードを送信するブロックチェーンにアカウントを作成します。コードでメッセージを暗号化したり、トランザクションに署名したりするための秘密鍵として使用されるパスフレーズを受け取ります。
  2. 2FAコードを生成するスクリプトをサーバーに追加します。ワンタイムパスワード配信で他の2FAメソッドをすでに使用している場合、この手順はすでに完了しています。
  3. サーバーにスクリプトを追加して、ブロックチェーンメッセンジャーでユーザーにコードを送信します。
  4. 2FAコードを送信および入力するためのユーザーインターフェイスを作成します。ワンタイムパスワード配信で他の2FAメソッドをすでに使用している場合、この手順はすでに完了しています。

1アカウントの

作成ブロックチェーンでのアカウントの作成は、秘密鍵、公開鍵、およびそれから派生したアカウントアドレスの生成です。



最初に、パスフレーズBIP39が生成され、そこからSHA-256ハッシュが考慮されます。ハッシュは、ks秘密鍵とkp公開鍵を生成するために使用されます。同じSHA-256を反転させた公開鍵から、ブロックチェーン内のアドレスを取得します。

新しいアカウントから毎回2FAコードを送信する場合は、アカウントを作成するためのコードをサーバーに追加する必要があります。

adamant.createPassphraseHash = function (passphrase) {
  const seedHex = bip39.mnemonicToSeedSync(passphrase).toString('hex')
  return crypto.createHash('sha256').update(seedHex, 'hex').digest()

adamant.makeKeypair = function (hash) {
  var keypair = sodium.crypto_sign_seed_keypair(hash)

import crypto from 'crypto'

adamant.getAddressFromPublicKey = function (publicKey) {
  const publicKeyHash = crypto.createHash('sha256').update(publicKey, 'hex').digest()
  const temp = Buffer.alloc(8)
  for (var i = 0; i < 8; i++) {
    temp[i] = publicKeyHash[7 - i]
  }
  return 'U' + bignum.fromBuffer(temp).toString()
