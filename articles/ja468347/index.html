MetricKit。iOSアプリケーションのパフォーマンス分析



新しいおもちゃ


私たちは、WWDCで発表されたAppleの新しい素材に慣れ続けています。今回は、アプリケーションのパフォーマンスを監視するためのツールとして機能するまったく新しいフレームワークであるMetricKitについて考えます。

開発中のアプリケーションパフォーマンスの測定が簡単であることは誰もが知っています。 Xcodeは、使用されるRAMの量とプロセッサの負荷を示します。インストルメントを使用してシミュレータまたはテスト中のデバイスに接続し、独自のツールを作成することもできます（詳細については、カスタムツールパッケージに関する記事をご覧ください：パート1 / パート2）パフォーマンス調整の重要性を理解するだけでは、アプリケーションが行うほとんどすべてのことを測定することはできません。しかし、開発されたアプリケーションが実際のユーザーを対象としている場合、AppStoreについて話すと、状況はさらに複雑になります。アプリケーションをどれだけ徹底的にテストしても、実際には、パフォーマンスとユーザーエクスペリエンスに影響を与える驚きが常にあります。もちろん、さまざまなパラメーターを収集するための多くのツールがありますが、それらのほとんどは、iOS SDKと、アプリケーションの動作に対する実際のモニタリングの影響によって制限されます。

今年、アップルはこのギャップを埋めることを決定し、実際の環境でアプリケーションのパフォーマンスメトリックを収集および分析するのに役立つツールを開発者に提供しました。彼らは、アプリケーション設定を見つけることができるXcode 11オーガナイザーの別のタブのMetricKit（OSが提供するオプションへのアクセスを提供するフレームワーク）を発表しました。Xcodeでのパラメーターの表示は、AppStoreで既に公開されているアプリケーションでのみ機能するため、MetricKitで一時停止します。



MXMetricManager


フレームワークのアーキテクチャは非常にシンプルで簡単です。中央部分はMXMetricManagerクラスで占められています。これは、開発者にフレームワークAPIの大規模なセットを提供する単一要素の構造です。

一般に、ワークフローは3つの主要なステップで構成されます。

  1. MXMetricMnagerを初期化し、それにオブザーバーを割り当てます。
  2. 必要に応じて、Signpost APIを使用してアプリケーションに独自のメトリックを実装できます
  3. そして最後に、didReceivePayloadsメソッドで受信したデータを処理します。それらをバックエンドに送信して、さらに分析します。

パラメータは、MXMetricPayloadのインスタンスの配列として提供されます。ペイロードは、メタデータセットとタイムスタンプをカプセル化します。メトリックペイロードは、MXMetricをサブクラス化するための単純なラッパーです。パラメータのタイプごとに、それは別個です。

メトリックのタイプはAppleによってかなりよく文書化されているので、これについてあまり長く考えないでください。ただし、興味深いことに注意してください。MXMetricは、NSDictionaryまたはJSONでシリアル化するためのオープンAPIを提供しています。

MetricKitの内部コンポーネント。


外では、MetricKitはかなりシンプルに見えます。しかし、すべてが内部からどのように機能するかを見るのは常に興味深いものです。特定のタスクに直面している場合、より深いものに没頭することは常に興味をそそられます。そのため、MetricKitタグでパラメーターを渡し、いつでも更新されたメトリックを提供するようにしたいと思いました。もちろん、Xcodeで` Debug-> Simulate MetricKit Payloads`を使用できますが、idでは独自のデータを表示できません。確かに、これはあまり有用なチームではありませんが、研究の方向性を示してくれますし、とても面白く見えます;）

タスクを開始するには、明らかにMetricKit自体が必要です。 「バイナリファイルをライブラリにリンク」ダイアログで追加するとすぐに、フレームワークのリストにXcodeが表示するため、フレームワークのバイナリファイルを取得するのは簡単だと思うかもしれません。これは非常に楽観的な考えです。なぜなら、MetricKit.frameworkを開くと、MetricKit.tbdファイルが表示されるからです。サイズはわずか4kbです。明らかに、これは私たちが探しているものではありません。

では、ここで何が起こっているのでしょうか。

未定「テキストベースのdylibスタブ」の略で、実際には、文字とdylibバイナリへのパスをエクスポートするdylib記述を含むYAMLファイルです。 tbdファイルをリンクすると、バイナリのサイズが小さくなります。その後、実行時に、実際のdylibバイナリがtbdファイルで指定されたパスでOSからダウンロードされます。 Xcodeでファイルを開くと、ファイルは次のようになります



。tbdファイルからのパスを使用すると、MetricKitバイナリを簡単に取得してさらに調査できますが、さらに簡単な方法があります。

アプリケーションのバイナリには、Mach-Oヘッダーセクションに動的にリンクされた各ライブラリへのパスが含まれています。この情報は、-lフラグを使用するツールを使用して簡単にアクセスできます。

作成したテストプロジェクトの出力は次のとおりです。

→ otool -l ./Metrics | grep -i metrickit
name /System/Library/Frameworks/MetricKit.framework/MetricKit (offset 24)

tbdファイルで前に見たのと同じパスを見ることができます。バイナリフレームワークファイルがあると、内部要素を確認できます。このため、私は通常、ホッパー逆アセンブルを使用します。使い方は簡単ですが、バイナリファイルを注意深く調査するための非常に強力なツールです。

MetricKitバイナリファイルが開いたらすぐに、[Proc。]タブに移動します。 [タグ]リストを展開します。ここでは、エクスポートされたすべての文字を確認できます。そのうちの一つ（例えば、MXMetricManager）を選択すると、我々はすべてのメソッドが表示され、この方法を選択した、我々は右側にその内容が表示されます。



メソッドMXMetricManagerのリストを表示しているときに[ https://gist.github.com/deszip/88a258ae21d33dc75d7cbac9569c6ec1] _checkAndDeliverMetricReportsメソッドに気付くのは非常に簡単です。これは、MetricKitが更新をサブスクライバーに配信するために呼び出す必要があるもののようです。

残念ながら、彼に電話をかけようとしても、サブスクライバーは呼び出されませんでした。これは、これらのパラメーターが配信されないことを意味します。メソッドの実装を考慮して、いくつか興味深いことに注意します。/ Library / Caches / MetricKit / Reportsディレクトリの内容を列挙します。

次に、ディスク上の各アイテムのMXMetricPayloadインスタンスを解凍しようとします。そして最後に、登録されたサブスクライバーを反復処理し、データのリストを使用してdidReceiveメソッドを呼び出します。

問題はおそらく/ Library / Caches / MetricKit / Reportsにありますデータはありませんが、MXMetricPayloadのアーカイブされたインスタンスが必要であることはわかっています。それで、それらを作成してディスクに置いてから、「_checkAndDeliverMetricReports」を呼び出します。この場合も、MXMetricPayloadのインスタンスを作成し、任意のタイプのMXMetricを作成して追加し、データインスタンスをディスクにアーカイブする計画です。結局のところ、「_ checkAndDeliverMetricReports」メソッドを呼び出すと、スタブを引数としてサブスクライバーが呼び出されます。

ペイロードとメトリックに関するAppleのドキュメントを見ると、パブリック初期化子がなく、ほとんどのプロパティが読み取り専用であることがわかります。では、クラスをインスタンス化するにはどうすればよいのでしょうか。

ホッパーに戻り、MXMetricPayloadメソッドのリストを見てみましょう。



ここでは、パラメーターを割り当てるための初期化子とメソッドを確認できます。Objective-Cの動的な性質により、NSInvocationクラスと「performSelector」メソッドを使用すると、プライベートメソッドを簡単に呼び出すことができます。

例として、CPUのメトリックを作成し、ペイロードに追加します。このリンクを使用すると、完全なコードフラグメントを見つけることができます：[ https://gist.github.com/deszip/a0cf877b07cc2877129e0aaef2fed1e4 ]。

最後に、作成したすべてをアーカイブし、データを/ Library / Caches / MetricKit / Reportsディレクトリに書き込みます。

今度は ' _checkAndDeliverMetricReportsメソッドを呼び出します'、これは最終的に加入者への呼び出しにつながるはずです。今回は、スタブ化されたペイロードを含むデータを引数として渡します。

結論


MetricKitは、実稼働環境でのアプリケーションのパフォーマンスを管理するための、ユニークで不可欠なツールです。

残念ながら、WWDCセッションでのデモ中に表示されたものを除いて、現在Xcodeの「Metric」ユーザーインターフェイスを確認することはできません。



これは、コードのパフォーマンスの問題を排除することにより、ユーザーエクスペリエンスを次のレベルに引き上げるための非常に貴重なツールです。

このツールで今見ている1つの欠点は、各タイプの詳細がないことです。アプリケーションのバージョンのみが分離されており、特定のデバイスグループ、OSバージョン、リージョンなどのメトリックを表示できません。

しかし、もちろん、必要な重要な情報とともに、データを自分に送信してさらに処理する機会は常にあります。バグトラッカーなどのタスクに添付できます。でAppSpector、私たちのチームはから得られたデータを使用してパフォーマンス監視ツールの機能拡充に取り組んでいるMetricKitを。
