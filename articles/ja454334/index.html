モト。ウィープAWS

テストは開発プロセスの不可欠な部分です。また、変更がコミットされるまで、開発者はローカルでテストを実行する必要があります。
アプリケーションが使用している場合はAmazon Webサービスを、Pythonのモトライブラリがあり、このための理想的。


リソースのカバレッジの完全なリストはここにあります。
Githubのはありヒューゴピカードのカブ- モトサーバーを。準備ができたイメージ、実行して使用します。唯一のニュアンスは、ローンチ後、AWSリソースがまだ作成されていないことです。

まあ、これは修正するのに十分簡単です。

起動時にサービスのタイプ（変数MOTO_SERVICEのenv）を指定する必要があるため、リソースの作成について説明する必要があります。start.shを 少し変更します。 代わりに





moto_server $MOTO_SERVICE -H $MOTO_HOST -p $MOTO_PORT

インサート：

if [ -f /opt/init/bootstrap.py ]; then
  moto_server $MOTO_SERVICE -H $MOTO_HOST -p $MOTO_PORT & (sleep 5 && echo "Executing bootstrap script." && python /opt/init/bootstrap.py)
else
  moto_server $MOTO_SERVICE -H $MOTO_HOST -p $MOTO_PORT
fi
wait

結果のファイルが取得されます。

start.sh
#!/bin/sh

# validate required input
if [ -z "$MOTO_SERVICE" ]; then
  echo "Please define AWS service to run with Moto Server (e.g. s3, ec2, etc)"
  exit 1
fi

# setting defaults for optional input
if [ -z "$MOTO_HOST" ]; then
  MOTO_HOST="0.0.0.0"
fi

if [ -z "$MOTO_PORT" ]; then
  MOTO_PORT="5000"
fi

echo "Starting service $MOTO_SERVICE at $MOTO_HOST:$MOTO_PORT"

if [ -f /opt/init/bootstrap.py ]; then
  moto_server $MOTO_SERVICE -H $MOTO_HOST -p $MOTO_PORT & (sleep 5 && echo "Executing bootstrap script." && python /opt/init/bootstrap.py)
else
  moto_server $MOTO_SERVICE -H $MOTO_HOST -p $MOTO_PORT
fi
# Prevent container from exiting when bootstrap.py finishing
wait


新しいイメージを作成し、レジストリにプッシュします。

次に、AWSを操作するためのライブラリを使用して、リソース初期化スクリプト（SWFドメインなど）を記述します-boto3：

bootstrap_swf.py
import boto3
from botocore.exceptions import ClientError
import os

os.environ["AWS_ACCESS_KEY_ID"] = "fake"
os.environ["AWS_SECRET_ACCESS_KEY"] = "fake"

client = boto3.client('swf', region_name='us-west-2', endpoint_url='http://localhost:5000')

try:
    client.register_domain(
        name='test-swf-mock-domain',
        description="Test SWF domain",
        workflowExecutionRetentionPeriodInDays="10"
    )
except ClientError as e:
    print "Domain already exists: ", e.response.get("Error", {}).get("Code")

response = client.list_domains(
    registrationStatus='REGISTERED',
    maximumPageSize=123,
    reverseOrder=True|False
)

print 'Ready'


ロジックはこれです：

  • 起動時に/opt/init/bootstrap.pyにスクリプトをマウントします。
  • ファイルがマウントされている場合、それが実行されます。
  • ファイルがない場合は、裸のmoto-serverが単に起動します。

そして、1つのコンテナーを起動することで、リソース全体を軽くたたくことができます。

docker run --name swf -d \
    -e MOTO_SERVICE=swf \
    -e MOTO_HOST=0.0.0.0 \
    -e MOTO_PORT=5000 \
    -p 5001:5000 \
    -v /tmp/bootstrap_swf.py:/opt/init/bootstrap.py \
    -i awesome-repo.com/moto-server:latest

したがって、docker-compose.ymlを作成すると、変更のテストにかかる時間を節約できます。

docker-compose.yml
version: '3'
services:
  s3:
    image: picadoh/motocker
    environment:
      - MOTO_SERVICE=s3
      - MOTO_HOST=10.0.1.2
    ports:
      - "5002:5000"
    networks:
      motonet:
        ipv4_address: 10.0.1.2
    volumes:
      - /tmp/bootstrap_s3.py:/opt/init/bootstrap.py
  swf:
    image: picadoh/motocker
    environment:
      - MOTO_SERVICE=swf
      - MOTO_HOST=10.0.1.3
    ports:
      - "5001:5000"
    networks:
      motonet:
        ipv4_address: 10.0.1.3
    volumes:
      - /tmp/bootstrap_swf.py:/opt/init/bootstrap.py
  ec2:
    image: picadoh/motocker
    environment:
      - MOTO_SERVICE=ec2
      - MOTO_HOST=10.0.1.4
    ports:
      - "5003:5000"
    networks:
      motonet:
        ipv4_address: 10.0.1.4
    volumes:
      - /tmp/bootstrap_ec2.py:/opt/init/bootstrap.py
networks:                             
  motonet:                          
    driver: bridge                
    ipam:                         
      config:                       
        - subnet: 10.0.0.0/16


実際、開発者のラップトップだけでなく、このアプローチを使用できます。ビルド後にmoksを使用して予備テストを行うと、dev *環境で実行するときに起こりうる問題を取り除くのに役立ちます。
