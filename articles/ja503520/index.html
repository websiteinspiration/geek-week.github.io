GPSトラックを横断して感染者を特定する

コースの開始を見越して、PostgreSQLは興味深い記事の翻訳を準備しました。




COVID-19の大流行の時に、政府は感染した人々を特定して追跡するために厳しい対策を講じてきました。これには、携帯電話のデータを使用して感染者とその連絡先を追跡し、流行を抑えることが含まれます。この記事では、PostGIS関数を使用して、トラックの時空間分析を通じて感染した健康な人々の重複する経路を特定する方法について説明します。

今回は、パフォーマンスとチューニングに焦点を当てず、PostgreSQLの空間拡張とその機能性に関連して創造性を高めることを目指します。

記事の構造は次のとおりです。

  1. PostgreSQLでのデータ構造の設定。
  2. QGISを使用したトラックのサンプリング
  3. 個々のポイントを抽出するためのトラックのサンプルのセグメンテーション。
  4. 感染している可能性のある人と健康な人の交点を見つけ、可能性のある連絡先を特定します。

データ構造


まず、トラックとそのポイントを表すテーブルを定義します。mobile_tracks

テーブルは、QGISを使用して選択された人々のシミュレートされたトラックを格納するヘルパーテーブルです。mobile_points テーブルには、トラックをセグメント化した結果のポイントが格納されます。また、タイムスタンプも追加されました。タイムスタンプは、空間的交差に基づいて時間的交差を決定するために使用されます。 テストエリアはウィーン/オーストリアです。したがって、私はMGI /オーストリア34（EPSGは31286）を適切な予測として選択しました。





create table if not exists mobile_tracks
(
   gid serial not null
      constraint mobile_tracks_ok
         primary key,
   customer_id integer,
   geom geometry(LineString,31286),
   infected boolean default false
);
 
create index if not exists mobile_tracks_geom_idx
   on mobile_tracks using gist (geom);
 
create unique index if not exists mobile_tracks_customer_id_uindex
    on mobile_tracks (customer_id);
 
create table if not exists mobile_points
(
   gid serial not null
      constraint mobile_points_ok
         primary key,
   customer_id integer,
   geom geometry(Point,31286),
   infected boolean,
   recorded timestamp
);
 
create index if not exists mobile_points_geom_idx
   on mobile_points using gist (geom);

トラックとドット


最初に述べたように、私は以前にQGISを使用して変換したトラックに基づいてポイントを生成することにしました。図1は、感染した（赤）人と健康な（緑）人の足跡を示しています。


図1.健康で感染した人々のGPS追跡。

簡単な例として、次の仮定を行いました。

  • すべての人が同じ速度で移動します。
  • すべての人のトラックが同時に始まります。

トラックの個々のポイントを強調表示するために、PostGIS ST_Segmentize関数を次のように使用しました。

with dumpedPoints as (
    select (st_dumppoints(st_segmentize(geom, 1))).geom,
           ((st_dumppoints(st_segmentize(geom, 1))).path[1]) as path,
           customer_id,
           infected,
           gid
    from mobile_tracks),
     aggreg as (
         select *, now() + interval '1 second' * row_number() over (partition by customer_id order by path) as tstamp
         from dumpedPoints)
insert
into mobile_points(geom, customer_id, infected, recorded)
select geom, customer_id, infected, tstamp
from aggreg;

クエリはメーターごとにポイントを生成し、対応するタイムスタンプを1000ミリ秒ずつ順次増やします。


図2.受信したGPSポイント。

交点の決定。


今が分析を開始する時です。私たちの感染者は誰かに会いましたか？

簡単なアプローチから始めましょう。2秒の適切な時間間隔で、感染者から2メートル以内にいる健康な人のポイントを選択します。

SELECT distinct on (m1.gid) m1.customer_id infectionSourceCust,
                            m2.customer_id infectionTargetCust,
                            m1.gid,
                            m1.recorded,
                            m2.gid,
                            m2.recorded
FROM mobile_points m1
         inner join
     mobile_points m2 on st_dwithin(m1.geom, m2.geom, 2)
where m1.infected = true
  and m2.infected = false
  and m1.gid < m2.gid AND (m2.recorded >= m1.recorded - interval '1seconds' and
       m2.recorded <= m1.recorded + interval '1seconds')
order by m1.gid, st_dwithin(m1.geom, m2.geom, 2) asc

図3および4は、このクエリの結果を示しています。ここでは、個人の連絡先が青色で強調表示されています。前述のように、このクエリは、会った人を特定するための最も簡単なソリューションを表しています。

または、PostGIS関数を使用しST_CPAWithin ST_ClosestPointOfApproachて、この問題を同様の方法で解決することもできます。これを行うには、ポイントを軌跡としてモデル化する必要があります。

ソリューションを改善するために、空間的近接性と対応する移動時間のコヒーレントセグメントを定義しましょう。目標は、人々が実際に感染する可能性を十分に持っている期間を理解することです。完全なリクエストコードは記事の最後にあります。

ステップ1-aggregStep1


最初のクエリの結果に基づいて、各ポイントについて、その前任者までの時間間隔を計算します。最初の仮定を考えると、一貫性のない時間セグメントには1秒を超えるギャップがあります。


図5. GPSポイント間の時間間隔。

ステップ2-aggregStep2。


現在のポイントと前のポイントの間の時間間隔を示す新しい列を使用して、セグメントをクラスター化します。

ステップ3-aggregStep3。


各クラスターについて、最大タイムスタンプから最小値を差し引いて、ジョイントパス間隔を取得します。


図6.コヒーレントセグメント+感染した/健康な人のコパス時間。

ステップ4


最後に、感染した/健康な人の組み合わせごとに、最大ジョイントパス時間でセグメントが抽出され、st_makelineを使用してlineStringが生成されます（図7を参照）。


図7.感染した/健康な人の関節経路の最大時間。

結果はすでにさらなる分析の基礎として機能しているかもしれませんが、私たちのアプローチは、当初受け入れられた仮定によってまだ制限されています。

最近、MobilityDBと呼ばれる非常に有望なPostgreSQL拡張機能が公開されました。これは、広範囲の時空間問題を解決するための多くの機能を提供します。詳細については、この記事をご覧ください。

with points as
         (
             SELECT distinct on (m1.gid) m1.customer_id infectionSourceCust,
 
                                         m2.customer_id infectionTargetCust,
                                         m1.gid,
                                         m1.geom,
                                         m1.recorded	m1rec,
                                         m2.gid,
                                         m2.recorded
             FROM mobile_points m1
                      inner join
                  mobile_points m2 on st_dwithin(m1.geom, m2.geom, 2)
             where m1.infected = true
               and m2.infected = false
               and m1.gid < m2.gid AND (m2.recorded >= m1.recorded - interval '1seconds' and
                    m2.recorded <= m1.recorded + interval '1seconds') order by m1.gid, m1.recorded), aggregStep1 as ( SELECT *, (m1rec - lag(m1rec, 1) OVER (partition by infectionSourceCust,infectionTargetCust ORDER by m1rec ASC)) as lag from points), aggregStep2 as (SELECT *, SUM(CASE WHEN extract('epoch' from lag) > 1 THEN 1 ELSE 0 END)
                 OVER (partition by infectionSourceCust,infectionTargetCust ORDER BY m1rec ASC) AS legSegment
          from aggregStep1),
     aggregStep3 as (
         select *,
                min(m1rec) OVER w   minRec,
                max(m1rec) OVER w   maxRec,
                (max(m1rec) over w) -
                (min(m1rec) OVER w) recDiff
         from aggregStep2
             window w as (partition by infectionSourceCust,infectionTargetCust,legSegment)
     )
select distinct on (infectionsourcecust, infectiontargetcust) infectionsourcecust,
                                                              infectiontargetcust,
                                                              (extract('epoch' from (recdiff))) passageTime,
                                                              st_makeline(geom) OVER (partition by infectionSourceCust,infectionTargetCust,legSegment)
from aggregStep3
order by infectionsourcecust, infectiontargetcust, passageTime desc
