Spring Boot, Hibernate und Kotlin für Anfänger Schritt für Schritt


Hallo allerseits, mein Name ist Oleg, ich bin ein technischer Experte in DomKlik. In unserem Team besteht der Kern des Stacks aus Kotlin und Spring Boot. Ich möchte meine Erfahrungen mit der Interaktion und den Funktionen der Arbeit mit PostgreSQL und Hibernate in Verbindung mit Spring Boot und Kotlin mit Ihnen teilen. Am Beispiel eines Mikroservices werde ich außerdem die Vorteile von Kotlin und seine Unterschiede zu einer ähnlichen Java-Anwendung zeigen. Ich werde über die nicht so offensichtlichen Schwierigkeiten sprechen, auf die Neulinge bei der Verwendung dieses Stacks mit Hibernate stoßen können. Dieser Artikel ist nützlich für Entwickler, die zu Kotlin wechseln möchten und mit Spring Boot, Hibernate Java, vertraut sind.

Entitäten


Wie in jeder Anwendung mit mehr als einer Entität erstellen wir einen gemeinsamen Vorfahren für alle entity.

BaseEntity::

@MappedSuperclass
abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

Warum nicht verwenden?


Datenklassen, abgesehen davon, dass endgültig auf ihre eigenen, haben bestimmte Felder für alle Felder equals, hashCodeund toString. Und dies ist in Verbindung mit dem Ruhezustand nicht akzeptabel .

Warum? Und auch, warum hashCodees immer gleich einer Konstanten ist - die Antwort liegt in der Dokumentation von Hibernate selbst . Insbesondere interessiert uns dieser Abschnitt:

Obwohl die Verwendung einer natürlichen ID am besten für equals und hashCode geeignet ist , haben Sie manchmal nur die Entitätskennung, die eine eindeutige Einschränkung bereitstellt.

Es ist möglich, die Entitätskennung für die Gleichheitsprüfung zu verwenden, es ist jedoch eine Problemumgehung erforderlich:

  • Sie müssen einen konstanten Wert für hashCode angeben, damit sich der Hashcode-Wert vor und nach dem Löschen der Entität nicht ändert.
  • Sie müssen die Entitätskennungsgleichheit nur für nicht transiente Entitäten vergleichen.

Das heißt, Sie müssen entweder nach natural idoder wie in unserem Beispiel nach vergleichen primary key id. Dies vermeidet viele Probleme beim Vergleichen der Entität und schützt vor deren Verlust, wenn die Entität als Element in verwendet wird Set.

Das Vorhandensein toString, das von allen Feldern definiert wird, beseitigt alle Faulheit, beispielsweise beim Journalisieren einer Entität, da alle Felder für die Ausgabe in eine Zeichenfolge initialisiert werden.

Aufgrund der Funktionen von Hibernate ist diese Kotlin-Funktionalität für uns nicht geeignet.

Nullgültigkeit nur in Feldern nach DB


Was die Gültigkeit der Werte nullin den Feldern betrifft, ist nicht alles so einfach. Früher haben wir uns ein wenig in die Tiefen von Hibernate gestürzt: Als ich von plugin.jpa sprach, erwähnte ich die Verwendung eines Konstruktors ohne Argumente beim Initialisieren einer Entität.

Beim Initialisieren von Feldern wird auch die Reflexion verwendet. Und wenn ein Wert in der Datenbank in der entsprechenden Spalte gespeichert wurde null, wird die Klasse stillschweigend mit diesem Feld mit dem Wert initialisiert null. Beim Zugriff laufen wir Gefahr, NPE zu erhalten, obwohl das Feld als markiert ist not nullable. Um dies zu vermeiden, muss die Synchronität der Struktur von Tabellen und Klassen überwacht werden.

Wenn Sie sich die Beschreibung in den letzten beiden Abschnitten genauer ansehen, gelten diese Regeln nicht nur für Grundelemente, sondern auch für eine Reihe von Entitäten.

Repositories


Bei Verwendung von Kotlin haben sofort einsatzbereite Repositorys eine Validierungsprüfung null. Wenn wir also sicher sind, dass departmentdas Ergebnis bei der Suche nach Namen eindeutig und eindeutig ist, kann der Rückgabetyp wie folgt angegeben werden non nullable:

interface DepartmentRepository : JpaRepository<DepartmentEntity, Long> {

lateinit var


Ein weiteres Kotlin-Feature, das ich in Betracht ziehen möchte, ist lateinit var.

Fügen Sie einen neuen Klassenvorfahren hinzu : BaseAuditEntity.

@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class BaseAuditEntity<T> : BaseEntity<T>() {

   @CreatedDate
   @Column(updatable = false, nullable = false)
   lateinit var created: LocalDateTime

   @LastModifiedDate
   @Column(nullable = false)
   lateinit var modified: LocalDateTime
}

Betrachten Sie lateinit varals Beispiel die Verwendung von Überwachungsfeldern ( created, modified).

lateinit var- Dies ist ein not nullverzögertes Initialisierungsfeld. Der Zugriff auf das Feld vor seiner Initialisierung erzeugt einen Fehler:

kotlin.UninitializedPropertyAccessException: lateinit property has not been initialized

In der Regel wenden wir uns den Feldern zu createdund modifiedauch danach, wenn eine Entität in der Datenbank gespeichert wurde. In diesem Fall sind die Daten in diesen Polyahprostavlyayutsya der Schritt des Speicherns und sie sind not null, dann sind lateinit varwir mehr als geeignet.
