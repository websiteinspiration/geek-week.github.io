1- バージョン管理


おめでとう、あなたは受け入れられます！リポジトリからコードを抽出し、作業を開始します。何をやめる？

あなたが開発者だけであっても、バージョン管理はどのプロジェクトにも必要です。最も一般的に使用されているシステムは、GitとSVNです。

SVNは、集中型バージョン管理システムに基づいています。これは作業コピーが作成されるリポジトリであり、それらにアクセスするにはネットワーク接続が必要です。変更の承認は特定の方法で実行されます。システムは各ファイルを登録することによって変更を監視します。変更の完全な履歴はリポジトリでのみ表示できます。作業コピーには最新バージョンのみが含まれます。

ギット分散バージョン管理システムを使用します。作業できるローカルリポジトリが作成されます。ネットワーク接続は同期のためにのみ必要です。作業コピーが変更されると、ディレクトリ全体の状態が保存されますが、行われた変更のみが記録されます。リポジトリと作業コピーの両方に完全な変更履歴があります。

3- Objective-C対Swift


アプリケーションを作成するプログラミング言語を決定するときは、それぞれにどのような機能があるかを知っておく必要があります。可能であれば、Swiftを使用することをお勧めします。どうして？正直なところ、Objective-CにはSwiftに勝る利点はほとんどありません。ほとんどの例とチュートリアルはObjective-Cで記述されており、Swiftは更新のたびにパラダイムを調整します。ただし、これらの問題は最終的には解消されます。

Objective-Cと比較すると、Swiftは多くの点で飛躍します。読みやすく、自然な英語のように見えます。また、C上に構築されていないため、従来の規則を放棄することができます。 Objective-Cを知っている人にとっては、これはセミコロンがなくなることを意味し、メソッド呼び出しと式の条件を角括弧で囲む必要がなくなります。 XcodeとLLVMコンパイラは依存関係を検出してインクリメンタルビルドを自動的に実行できるため、Swiftは.hおよび.mファイルの代わりに.swiftファイルのみを必要とします。一般に、標準化されたコードを作成することについて心配する必要が少なくなり、少ない行数で同じ結果を達成できることがわかります。

まだ疑問がありますか？ Swiftはより安全で高速であり、メモリ管理を処理します（ほとんどの場合！）。初期化されていないポインタ変数を使用してメソッドを呼び出すと、Objective-Cで何が起こるか知っていますか？何もない。式は非アクティブになり、スキップされます。それはアプリケーションのクラッシュにつながらないので素晴らしいように聞こえますが、それはあなたがあなたの職業を変えることを考えたくなるでしょうから、多くのバグと不安定な振る舞いを引き起こします。真剣に。プロのドッグウォーカーになるという考えは、新しい色で輝いていました。同時に、Swiftカウンターはオプションの値で動作します。 nilが何であるかをよりよく理解し、nil値の使用を防ぐための条件を設定するだけでなく、nilオプションがまだ使用されている場合はランタイムクラッシュが発生するため、デバッグが簡単になります。ARC（自動参照カウント）は、Swiftでのメモリ管理を改善します。 Objective-Cでは、ARCは手続き型CまたはCore GraphicsのようなAPIでは機能しません。

4- 反応するか反応しないか？（それが問題です）


関数型反応プログラミング（FRP）が新たにヒットしました。非同期操作とイベント/データストリームのコンパイルを簡素化することを目的としています。Swiftの場合、これはObservableインターフェースを通じて表現される計算の一般的な抽象化です。

これを説明する最も簡単な方法は、小さなコードを使用することです。赤ちゃんティミーと妹のジェニーが新しいゲームコンソールを購入したいとします。ティミーは毎週両親から5ユーロを受け取ります。ジェニーも同じです。ただし、ジェニーは週末に新聞を配達することで、さらに5ユーロを稼ぎます。両方が1セントも節約できれば、コンソールが利用可能かどうかを毎週確認できます。節約額が変わるたびに、その合計値が計算されます。これで十分な場合、メッセージはisConsoleAttainable変数に格納されます。いつでも、購読してメッセージを確認できます。

// 
let timmySavings = Variable(5)
let jennySavings = Variable(10)

var isConsoleAttainable =
Observable
.combineLatest(timmy.asObservable(), jenny.asObservable()) { $0 + $1 }
.filter { $0 >= 300 }
.map { "\($0)    !" }

//  
timmySavings.value = 10
jennySavings.value = 20
isConsoleAttainable
   .subscribe(onNext: { print($0) }) //   

//  
timmySavings.value = 100
jennySavings.value = 200
isConsoleAttainable
   .subscribe(onNext: { print($0) }) // 300    !

5- 依存関係マネージャー


CocoaPodsおよびCarthageは、Cocoa SwiftおよびObjective-Cプロジェクトの最も一般的な依存関係マネージャーです。ライブラリを実装して最新の状態に保つプロセスを簡素化します。

CocoaPodsにはRubyを使用して構築された多くのライブラリがあり、次のコマンドを使用してインストールできます。

$ sudo gem install cocoapods

インストール後、プロジェクト用のPodfileを作成します。次のコマンドを実行できます。

$ pod install

または、次の構造でカスタムPodfileを作成します。

platform :ios, '8.0'
use_frameworks!

target 'MyApp' do
pod 'AFNetworking', '~> 2.6'
pod 'ORStackView', '~> 3.0'
pod 'SwiftyJSON', '~> 2.3'
end

作成したら、新しいモジュールをインストールします。

$ pod install

これで、プロジェクトの.xcworkspaceを開くことができます。依存関係をインポートすることを忘れないでください。

Carthageは、CocoaPodsとは異なり、分散型の依存関係マネージャーです。これの欠点は、ユーザーが既存のライブラリを見つけることがますます困難になることです。一方、このアプローチでは、サポート作業が少なくて済み、ストレージの集中管理による中断を回避できます。

インストールと使用の詳細については、GitHubプロジェクトをご覧ください。

6- データストレージ


アプリケーションのデータを保存する最も簡単な方法から始めましょう。NSUserDefaults。これは、アプリケーションが最初にロードされたときに表示されるデフォルトのユーザーデータを保存するために通常使用されるため、このように呼ばれます。このため、シンプルで使いやすくなっていますが、これにはいくつかの制限もあります。それらの1つは、このメソッドが受け入れるオブジェクトのタイプです。その動作は、同じ制限を持つプロパティリスト（Plist）に非常に似ています。保存できるオブジェクトには、次の6種類があります。

  • NSData
  • NSDate
  • NSNumber
  • NSDictionary
  • Nsstring
  • NSArray

Swiftとの互換性のために、NSNumberは以下のタイプを受け入れることができます。

  • UInt
  • Int
  • 浮く
  • ダブル
  • ブール

オブジェクトは次のようにNSUserDefaultsに保存できます（最初に、保存されたオブジェクトのキーを保存する定数を作成します）。

let keyConstant = "objectKey"

let defaults = NSUserDefaults.standardsUserDefaults()
defaults.setObject("Object to save", objectKey: keyConstant)

NSUserDefaultsからオブジェクトを読み取るには、次のようにします。

if let name = defaults.stringForKey(keyConstant) {
   print(name)
}

AnyObjectの代わりに特定のオブジェクトを受け取るNSUserDefaultsの読み書きには、いくつかの便利なメソッドがあります。

キーチェーンはパスワード管理システムであり、パスワード、証明書、秘密鍵、または個人的なメモが含まれる場合があります。キーチェーンには、2つのレベルのデバイス暗号化があります。最初のレベルでは、ロック画面ロックコードを暗号化キーとして使用します。 2番目のレベルは、デバイスで生成および保存されたキーを使用します。

どういう意味ですか？特にロック画面でパスワードを使用していない場合、これは完全に安全ではありません。デバイスに保存されているため、第2レベルで使用されるキーにアクセスする方法もあります。

最善の解決策は、独自の暗号化を使用することです。 （デバイスにキーを保存しないでください）

CoreData-これはAppleによって開発されたフレームワークであり、アプリケーションはオブジェクト指向の方法でデータベースと対話します。これにより、コードの量が減り、このセクションをテストする必要がなくなるため、プロセスが簡略化されます。

アプリケーションで永続データが必要な場合はCoreDataを使用する必要があります。これにより、データを保存するプロセスが大幅に簡略化され、データベースとの独自の通信方法を作成/テストできなくなります。

7 - CollectionViewsとTableViews


ほとんどすべてのアプリケーションには、1つ以上のCollectionViewsおよび/またはTableViewsがあります。それらがどのように機能するか、いつ使用するかを知ることで、将来のアプリケーションの複雑な変更を防ぐことができます。

TableViewは、単一の列に項目のリストを垂直に表示し、垂直スクロールによってのみ制限されます。各項目はUITableViewCellで表され、完全にカスタマイズできます。セクションと行でソートできます。

CollectionViewsは項目のリストも表示しますが、複数の列と行（たとえば、グリッド）を持つことができます。それらは水平および/または垂直にスクロールでき、各要素はUICollectionViewCellによって表されます。 UITableViewCellsと同様に、必要に応じてカスタマイズし、セクションと行で並べ替えることができます。

どちらも同様の機能を備え、再利用可能なセルを使用して移動性を向上させます。必要なものの選択は、リストに含めたい複雑さによって異なります。 CollectionViewは任意のリストを表すために使用でき、私の意見では、常に最良の選択です。連絡先リストを送信するとします。リストはシンプルで、単一の列で実装できるため、UITableViewを選択します。すべてが機能しています！数か月後、連絡先をリストではなくグリッド形式で表示するように設計者が決定します。これを行う唯一の方法は、UITableViewの実装をUICollectionViewの実装に変更することです。あなたのリストは単純かもしれませんが、おそらく設計変更の可能性が高い場合はUITableViewで十分かもしれませんが、UICollectionViewを使用してこのリストを実装するのが最善です。

どのような選択をしても、汎用のTableView / CollectionViewを作成することをお勧めします。これにより、実装が容易になり、大量のコードを再利用できます。

8- ストーリーボードVS Xibs VSプログラマブルUI


これらの各メソッドを個別に使用してユーザーインターフェイスを作成できますが、それらを組み合わせることを妨げるものはありません。

ストーリーボードは、プロジェクトのより広いビューを提供します。これはデザイナーにとって魅力的であり、アプリケーションのフローやウィンドウを確認することができます。欠点は、ウィンドウを追加すると接続が混乱し、ストーリーボードの読み込み時間が長くなることです。ユーザーインターフェイス全体が単一のファイルにあるため、マージの問題はより一般的です。また、解決するのがはるかに難しくなります。

Xibsウィンドウまたはウィンドウの一部を視覚的に表示します。利点は、再利用のしやすさ、ストーリーボードよりもマージの競合が少ないこと、各ウィンドウの内容を簡単に表示できることです。ユーザーインターフェイスを

プログラミングすることで、制御を強化し、マージの競合の頻度を減らします。マージの競合が発生した場合は、簡単に削除できます。不利な点は、視覚化が少なく、書き込みに必要な時間が増えることです。

ユーザーインターフェイスを作成するための上記のアプローチは、大きく異なります。しかし、私の主観的な意見では、最良のオプションは3つすべての組み合わせです。いくつかのストーリーボード（これでストーリーボードを切り替えることができます！）、メインウィンドウではないビジュアルオブジェクト用のXibs、そして最後に、特定の状況で必要な追加の制御のための小さなプログラミング。

9- プロトコル！


日常生活では、プロトコルが存在するため、特定の状況で私たちがどのように対応するかがわかります。あなたが消防士であり、緊急事態が発生したとします。各消防士は、対応を成功させるための要件を定めたプロトコルに従う必要があります。同じことがSwift / Objective-Cのプロトコルにも当てはまります。

プロトコルは、指定された機能のメソッド、プロパティ、およびその他の要件のスケッチを定義します。これは、クラス、構造、または列挙型によって採用でき、これらの要件の実際の実装を持ちます。

プロトコルの作成と使用の例を次に示します。

私の例では、消火に使用されるさまざまな種類の材料を列挙した列挙が必要です。

enum ExtinguisherType: String {

   case water, foam, sand

}

次に、緊急応答プロトコルを作成します。

protocol RespondEmergencyProtocol {

   func putOutFire(with material: ExtinguisherType)

}

次に、プロトコルに従う消防士クラスを作成します。

class Fireman: RespondEmergencyProtocol {

    func putOutFire(with material: ExtinguisherType) {

       print("Fire was put out using \(material.rawValue).")

    }

}

いいね！そして今、私たちは消防士を使用しています。

var fireman: Fireman = Fireman()

fireman.putOutFire(with: .foam)

結果は次のようになります。「泡を使用して消火した。」

プロトコルは委任でも使用されます。これにより、クラスまたは構造体が特定の関数を別の型のインスタンスに委任できるようになります。プロトコルは、その機能が適切なタイプであることを保証するために、委任された責任で作成されます。
小さな例！

protocol FireStationDelegate: AnyObject {

func handleEmergency()

}

消防署は消防士の緊急対応措置を委任します。

class FireStation {
   weak var delegate: FireStationDelegate?

   func emergencyCallReceived() {
      delegate?.handleEmergency()
   }
}

つまり、消防士もFireStationDelegateプロトコルに準拠する必要があります。

class Fireman: RespondEmergencyProtocol, FireStationDelegate {

   func putOutFire(with material: ExtinguisherType) {
      print("Fire was put out using \(material.rawValue).")
   }

   func handleEmergency() {
      putOutFire(with: .water)
   }

}

必要なことは、消防士が電話をかけたときに消防署の代表として任命され、受信した緊急電話を処理することだけです。

let firestation: FireStation = FireStation()
firestation.delegate = fireman
firestation.emergencyCallReceived()

その結果、「水を使って火が消されました。」

12-テスト


時間をかけてアプリケーションをテストすれば、順調に進んでいます。もちろん、これは万能薬ではありません。すべてのバグを修正することはできません。また、アプリケーションに問題がないことを保証することもできません。それでも、長所は短所を上回ると思います。ユニットテスト

のマイナス面から始めましょう：

  • 開発時間の増加。
  • コードの量を増やします。

長所：

  • モジュール式コードを作成する必要性（テストを簡略化するため）。
  • 明らかにリリース前にほとんどのバグを検出します。
  • サポートの簡素化。

ツール ユーティリティと組み合わせると、アプリケーションを柔軟にしてエラーやクラッシュなしに機能させるためのすべてが得られます。

アプリケーションのテストに使用できる多くのツールがあります。追跡する内容に応じて、1つ以上を選択できます。おそらく、最も一般的に使用されるツールがあるリーク、時間プロファイラ、および割り当て。

13-ジオロケーション


多くのアプリケーションでは、一部の機能でユーザーを見つける必要があります。そのため、場所がiOSでどのように機能するかについての一般的な考えがあると便利です。

Core Locationと呼ばれるフレームワークがあり、必要なものすべてにアクセスできます。

Core Locationフレームワークを使用すると、デバイスの現在の場所または移動方向を特定できます。フレームワークは、利用可能なハードウェアを使用して、ユーザーの位置と方向を決定します。このフレームワークのクラスとプロトコルを使用して、場所と方向に関連するイベントを構成およびスケジュールできます。コアロケーションを使用して、地理的領域全体の動きを追跡することもできます。iOSでは、Bluetoothビーコンまでの距離を確認することもできます*。

*私が理解しているように、それはiBeaconテクノロジーに関するものです

かっこいいですよね？フレームワークの機能をよりよく理解するために、Appleのドキュメントとその例を確認してください。

14- ローカライズ可能な文字列


どのアプリケーションにも何を実装すべきか。これにより、デバイスが配置されている地域に応じて言語を変更できます。アプリケーションの言語が1つしかない場合でも、将来的に新しい言語を追加する必要があるかもしれません。ローカライズされた文字列を使用してすべてのテキストを入力する場合は、新しい言語用のLocalizable.stringsファイルの翻訳バージョンを追加するだけです。

リソースは、ファイルインスペクタを介して言語に追加できます。NSLocalizedStringで文字列を取得するには、次のように記述する必要があります。

NSLocalizedString(key:, comment:)

残念ながら、ローカライズ可能なファイルに新しい行を追加するには、これを手動で行う必要があります。構造の例を次に示します。

{
"APP_NAME" = "MyApp"
"LOGIN_LBL" = "Login"
...
}

もう1つの言語（ポルトガル語）、ローカライズされたファイル：

{
"APP_NAME" = "MinhaApp"
"LOGIN_LBL" = "Entrar"
...
}

複数を実装する方法さえあります。
