Oracle JDKおよびJava Web StartからAdoptOpenJDKおよびOpenWebStartへの移行方法



良い一日。

この記事では、私が仕事をしている会社でのJava Web Startなどのツールの「近代化」について、または代替のオープンソースソリューションでの置き換えについて説明します。

自分自身について


私の名前はイルダーです。私はドイツの会社で働いています。多くのドイツの会社と同様に、古いテクノロジースタックを使用し、新しいスタックへの移行を試みています。

問題


問題の説明から始めます。当社は、システムの最も重要な部分を持っています。これは、Java 6で部分的にJava 8で記述されたレガシーアプリケーションです。Java8で起動されました。このアプリケーションを本番環境で使用するには、ユーザーがサイトからjnlpファイルをダウンロードして自宅で実行しますコンピューターで。

そのため、OracleはJavaのバージョン8のサポートを停止したことを発表し、Java Web Startテクノロジーを非推奨としてマークし、Java 11からそれを廃止することを決定するまで、会社は長い間幸せに存続していました。セキュリティアップデートを含めて受信しません。もちろん、これに同意できる人はほとんどいません。また、カラクンの愛好家がJWSの代わりを書くことを決めたと考えました。

私たちの会社では、少し前に、バックエンドにSpringブートを使用し、フロントエンドにReactを使用してアプリケーションを書き直し始めましたが、プロセスは高速ではなく、現在アップデートはありません。

解決策を探す


したがって、開発チームは代替ソリューションを見つけるという問題に直面しました。一般に、この問題には複数の解決策があることを認識する価値があります。最初に、設計者は2つのGetDownソリューションと同じOpenWebStartを選択しました。 OpenWebStartは最初のバージョンではリリースされていなかったため（一部の開発と計画しかなかったため）、最初の決定時には、選択肢は最初のオプションに当てはまりました。

各ソリューションには長所と短所がありましたが、同社はOpenWebStartのリリースを待たずに、GetDownに基づく概念実証を実装することにしました。

数週間を費やした後、開発者の1人がこのタスクに対処し、原則として、もう少し時間を費やすことで要件を満たす完全なソリューションを実装できることに気付きました。

その間に、他の緊急のタスクが到着し、開発チームはPoCステージからJava Web Start置換の完全な実装への移行から注意をそらしました。

締め切り


経営陣がすでに終了し、Java Web Startが置き換えられる時期だと気づくまで、誰もが幸せに暮らしていました。この瞬間まで、私はこのプロジェクトに関与していませんでしたが、ここでも彼らは私を結びつけました。
最初に、問題の解決策に関する情報をもう一度探すことにしました。それでは、ソリューションをゼロから始めましょう。このように、OpenWebStartの存在を発見しました。私はそれをローカルでテストし、いくつかの問題に遭遇し（その後、他の問題にも遭遇しました）、それらを解決しました。その結果、誰もがソリューションを気に入っていました。言うまでもなく、GetDownの場合のように開発に時間を費やす必要がないため、経営陣は特にそれを気に入っていました。しかし、最終的には、システムをOpenWebStartで動作させるために時間を費やしました。

OpenWebStartに関するクイック情報


OpenWebStartは、Java Web Start- IcedTea-Webの別の実装とJNLP仕様JSR-56に基づいています。執筆時点では、プロジェクトはバージョン1.1.4にあり、Java Web Startの主な機能を実装する予定です（進行状況はここで確認できます）。
可能なすべての設定を一覧表示する意味がわかりません。非常に時間がかかる場合があります。

以下はそのほんの一部です。

  • キャッシュ管理（サイズ、場所...）、
  • セキュリティ設定（ユーザーが証明書で署名されていないアプリケーションを実行できるようにする、警告ポップアップを表示するかどうか...）、
  • サーバーリストをホワイトリストに追加し、
  • リモートデバッグ設定、
  • セキュリティ証明書の管理
  • JRE / JDKバージョン管理、
  • その他の

OpenWebStartの機能と発生した問題


OpenWebStartをインストールする


OpenWebStartのインストールは非常に簡単です。ご使用のプラットフォーム用のインストールファイルをWebサイトにダウンロードし、インストーラーの指示に従うだけで十分です。

しかし、これは不運です。私たちの場合、ユーザーは約1万人のお客様で、当社のサポートサービスがこのツールをインストールする必要があります。解決策はすぐに見つかりました。いわゆるバックグラウンドインストールが利用可能です。

インストールファイルを各コンピューターにアップロードし、特別なコマンドを実行する必要があります（このため、会社には既にツールがあります）。

OpenWebStart_windows_Setup.exe -q -varfile response.varfile

ここで、response.varfileは、インストーラーによって事前に設定できるいくつかの設定を持つファイルです。たとえば、OpenWebStartなどをインストールするフォルダーです。

さて、この問題に対処しました。さらに、すべてのクライアントが特定のバージョンのAdoptOpenJDKをインストールしてOpenWebStartにリンクする必要があります。これはユーザーインターフェイスから簡単に実行できますが、これは私たちのケースではありません。



OpenWebStartが設定ファイルを取得するURLを指定できる設定で、必要なJREへのURLを指定できることがわかりました。また、設定ファイルを含むURL自体は、上記のresponse.varfileで指定できます（これは非常に複雑です）。

JREのバージョンが異なるJSON設定ファイル自体は次のとおりです。

JSONファイルで複数のJREが指定されている場合は、設定でJREのバージョンとベンダーを指定することもできます。1つだけを示し、JSONファイルをアップロードして、Amazon S3バケットに必要なJREでアーカイブしました。

テストの結果、ダウンロードしたJREバージョンではアプリケーションが起動しないことがわかりました。Javaの構造がアーカイブ内で少し異なることがわかりました。JREアーカイブの構造を再構築するバッチスクリプトを作成しました。

「まあ、今はすべてがうまくいくでしょう」と私たちは考えました。

アプリケーションを初めて起動すると、JREが自動的にロードされ、将来使用されます。しかし、私たちは次の驚きを待っていました。

不幸は一人で来ることはありません


しかし、通常は発生す​​るため、1つの問題に限定されませんでした。

私たちのアプリケーションには、たとえば、1つの機能があります。実際には2つあります。 1つのアプリケーション（1つのjnlpファイル）が2番目のアプリケーション（2番目のjnlp）を開始します。この場合、2番目のアプリケーションは起動しませんでした。ログでは、アプリケーションがデッドロックのままになっていることがわかります。ステルストレースから、その理由はOpenWebStartの内部ロギングメカニズムにあることが明らかになりました。

この問題は、OpenWebStartの内部ログを無効にすることで解決しました。もちろん、アプリケーションのログは残りました。



これらの設定は、バックグラウンドインストール中に使用されるresponse.varfileファイルでもオフになっています。

そして今、これらの障害や他のいくつかの障害を克服した後（私はすべての人に言及する必要はありません）、アプリケーションをリリースすることができました。現在、製品にリリースされる前にテストが行​​われています。テストしたように、OpenWebStartのバージョンは1.1.1から1.1.4にアップグレードされました。注目すべき変更点として、リモートでデビューする機能が追加されました。
