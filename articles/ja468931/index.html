コンテナーでsystemdを実行する

コンテナーでsystemdを使用するというトピックに長い間取り組んできました。 2014年に、セキュリティエンジニアのダニエルウォルシュは、Dockerコンテナー内でのsystemdの実行という記事を書きました。数年後、非特権コンテナーでのsystemdの実行という別の記事を書きました。非常に改善されました。特に彼は、「残念ながら2年後、Dockerシステムをググると、最初に現れるのは彼の古い記事と同じです。だから何かを変える時がきたのです。」さらに、Dockerとsystemdの開発者間の競合については、すでに一度話しました。



この記事では、過去の間に何が変化したか、およびこの問題でポッドマンがどのように役立つかを示します。

コンテナー内でsystemdを実行する理由はたくさんあります。

  1. マルチサービスコンテナー -多くの人々は、仮想マシンからマルチサービスアプリケーションを取り出し、コンテナーで実行したいと考えています。もちろん、そのようなアプリケーションをマイクロサービスに分割する方が良いでしょうが、誰もがまだそれを実行できるわけではないか、単に時間がないだけです。したがって、systemdがユニットファイルから起動するサービスの形でそのようなアプリケーションを起動することは、完全に理にかなっています。
  2. Systemdユニットファイル —コンテナー内で実行されているほとんどのアプリケーションは、以前は仮想マシンまたは物理マシンで実行されていたコードからコンパイルされます。これらのアプリケーションには、これらのアプリケーション用に作成されたユニットファイルがあり、それらの実行方法を理解しています。したがって、独自のinitサービスをハッキングするよりも、サポートされている方法を使用してサービスを開始する方が適切です。
  3. Systemdはプロセスマネージャーです。他のどのツールよりもサービスを管理（シャットダウン、サービスの再起動、またはゾンビプロセスのクロール）します。

コンテナーでsystemdを実行しない理由はたくさんあります。主なものは、systemd / journaldがコンテナーの出力を制御し、KubernetesやOpenShiftなどのツールがコンテナーがstdoutおよびstderrに直接ログを書き込むことを期待していることです。したがって、上記のようなオーケストレーションツールを使用してコンテナーを管理する場合は、systemdに基づくコンテナーの使用を真剣に検討する必要があります。さらに、DockerとMobyの開発者は、コンテナーでのsystemdの使用に強く反対しました。

ポッドマンの到来


状況がようやく地上に移ったことをお知らせいたします。 Red Hatでコンテナーの起動を担当するチームは、独自のコンテナーエンジンを開発することを決定しました。彼はPodmanという名前で、Docker'aと同じコマンドラインインターフェイス（CLI）を提供しています。また、ほとんどすべてのDockerコマンドは、Podmanで同じように使用できます。私たちはしばしばセミナーを開催しており、現在はDockerをPodmanに変更と呼ばれています。最初のスライドでは、登録することをお勧めしています：エイリアスdocker = podman。

多くはそうします。

私のポッドマンと私は、systemdベースのコンテナーに決して反対しません。結局のところ、SystemdはLinux init-subsystemとして最もよく使用されており、コンテナーで正常に機能しないようにすると、コンテナーの実行に何千人もの人が慣れている方法が無視されます。

ポッドマンは、systemdがコンテナーで正しく動作するために何をすべきかを知っています。彼女は/ fsと/ tmpにtmpfsをマウントするようなものが必要です。彼女は、「コンテナー」環境が有効になっているときに気に入っており、cgroupディレクトリーの自分の部分と/ var / log / journaldフォルダーへの書き込み許可を待っています。

initまたはsystemdが最初のコマンドであるコンテナーを起動すると、Podmanは自動的にtmpfsとCgroupを構成して、systemdが問題なく起動するようにします。この自動開始モードをブロックするには、-systemd = falseオプションを使用します。ポッドマンがsystemdモードを使用するのは、systemdまたはinitコマンドを実行する必要があると判断した場合のみであることに注意してください。

次に、Podmanを使用するときにコンテナーでsystemdを実行するDockerfileがどのように見えるかを確認します。

# cat Dockerfile

FROM fedora

RUN dnf -y install httpd; dnf clean all; systemctl enable httpd

EXPOSE 80

CMD [ "/sbin/init" ]

それで全部です。

次にコンテナを収集します。

# podman build -t systemd .

systemdがCgroupの設定を変更できるようにSELinuxに指示します：

# setsebool -P container_manage_cgroup true

ところで、多くの人はこのステップを忘れています。幸いなことに、これを行うのは一度だけで十分であり、システムを再起動した後に構成が保存されます。

次に、コンテナーを実行します。

# podman run -ti -p 80:80 systemd

ポッドマンとsystemdについてさらにクールなこと


ポッドマンはsystemdユニットファイルでdockerよりもうまく機能します


コンテナーをシステムの起動時に起動する必要がある場合は、適切なPodmanコマンドをsystemdユニットファイルに挿入するだけで、サービスを起動して監視できます。ポッドマンは、標準のfork-execモデルを使用します。言い換えると、コンテナープロセスはポッドマンプロセスと関連しているため、systemdはそれらを簡単に監視できます。

Dockerはクライアント/サーバーモデルを使用し、Docker CLIコマンドをユニットファイルに直接配置することもできます。ただし、DockerクライアントがDockerデーモンに接続すると、それ（クライアント）はstdinおよびstdoutを処理する別のプロセスになります。一方、systemdはDockerクライアントとDockerデーモンを実行しているコンテナーとの間の接続を認識していないため、このモデルでは、systemdは基本的にサービスを監視できません。

ソケット経由のSystemdアクティベーション


ポッドマンはソケットを介して正しくアクティベーションを実行します。Podmanはfork-execモデルを使用するため、ソケットをその子コンテナープロセスに転送できます。Dockerはクライアントサーバーモデルを使用しているため、方法を知りません。

Podmanがコンテナーを使用してリモートクライアントと対話するために使用するvarlinkサービスは、実際にはソケットを介してアクティブ化されます。 Node.jsで作成され、コックピットプロジェクトの一部であるcockpit-podmanパッケージを使用すると、Webインターフェースを介してPodmanコンテナーを操作できます。 cockpit-podmanが実行されているWebデーモンは、systemdがリッスンしているvarlinkソケットにメッセージを送信します。その後、systemdはPodmanプログラムをアクティブ化してメッセージを受信し、コンテナーの管理を開始します。ソケットを介してsystemdをアクティブ化すると、リモートAPIを実装するときに、常に動作しているデーモンなしで実行できます。

さらに、同じポッドマンCLIを実装し、コンテナーを起動するためにvarlinkを呼び出すpodman-remoteと呼ばれるポッドマン用の別のクライアントを開発しています。Podman-remoteはSSHセッションの上で動作するため、さまざまなマシン上のコンテナーと安全に対話できます。今後は、podman-remoteを使用してMacOSとWindowsをLinuxとともにサポートする予定です。これにより、これらのプラットフォームの開発者は、Podman varlinkを実行してLinux仮想マシンを実行し、コンテナーがローカルマシンで実行されていることを十分に感じることができます。

SD_NOTIFY


Systemdでは、必要なコンテナー化されたサービスが開始するまで、補助サービスの起動を遅らせることができます。Podmanは、SD_NOTIFYソケットをコンテナー化されたサービスに転送して、サービスがsystemdに作業の準備ができていることを通知できるようにします。また、Dockerはクライアントサーバーモデルを使用しているため、その方法を知りません。

計画中


特定のコンテナーを管理するためのsystemdユニットファイルを生成するpodman generate systemd CONTAINERIDコマンドを追加する予定です。これは、非特権コンテナのルートモードとルートレスモードの両方で機能するはずです。さらに、OCI互換のsystemd-nspawnランタイムを作成するリクエストもありました。

結論


コンテナーでsystemdを実行することは当然の必要です。そして、ポッドマンのおかげで、systemdに敵対的ではないが使いやすいコンテナーランチャー環境がついに完成しました。
