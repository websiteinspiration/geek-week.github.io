Windowsデスクトップ用.NET Core 3

9月に、WPFやWindowsフォームなどのWindowsデスクトップアプリケーションを構築するための.NET Coreサポートをリリースしました。それ以来、多くの開発者がデスクトップアプリケーションの.NET Coreへの移植に関するストーリーを共有することを嬉しく思っています。Windowsデスクトップアプリケーションの.NET開発者から、WPFとWindowsフォームを使用してビジネスをサポートする方法について、特にデスクトップが成功した場合の話を常に耳にします。

  • オフラインで動作する必要があるアプリケーション

.NET Core上のWindowsデスクトップはなぜですか？


.NET Core（および将来的には.NET Coreに基づいて構築された.NET 5）が.NETの未来になります。できる限り.NET Frameworkをサポートするように努めていますが、新しい機能は提供されず、.NET Core（最終的には.NET 5）にのみ追加されます。 Windowsデスクトップスタックを改善し、.NETデスクトップ開発者が将来のすべての更新からメリットを得られるようにするために、WindowsフォームとWPF for .NET Coreを導入しました。これらは、Windows APIと密接に関連しているため、引き続きWindowsのみのテクノロジです。しかし、.NET Coreには、クロスプラットフォームに加えて、デスクトップアプリケーションを改善できる他の多くの機能があります。

まず第一に、すべてのランタイムの改善と言語機能は.NET Coreでのみ追加され、将来的には.NET 5で追加されます。ここでの良い例は、C＃8です。これは.NET Core 3.0で利用可能になりました。さらに、.NET CoreバージョンのWindowsフォームとWPFは.NET 5プラットフォームの一部になります。また、アプリケーションを.NET Coreに移植すると、.NET 5に対応できるようになります。

    <PublishTrimmed>true</PublishTrimmed>
</PropertyGroup>

.NET Frameworkデスクトップと.NET Coreデスクトップの違い


デスクトップアプリケーションを開発する場合、.NET Frameworkと.NET CoreのWPFとWindowsフォームのバージョンに大きな違いはありません。私たちの取り組みの一部は、デスクトップPCの分野でこれらのプラットフォーム間で機能の類似点を提供し、将来的に.NET Coreの機能を拡張することでした。

WPFアプリケーションは.NET Coreで完全にサポートされており、マイナーな更新や改善に取り組んでいる間、それらを使用して作業を開始できます。 Windowsフォームの場合、ランタイムは.NET Coreに完全に移植されており、チームは現在Windowsフォームデザイナに取り組んでいます。 2020年の第4四半期に向けて準備する予定ですが、現時点ではVisual Studioでデザイナーの暫定バージョンを確認できます  16.4プレビュー3以降。[ツール]-> [オプション]-> [プレビュー機能]-> [.NET CoreアプリのWindowsフォームデザイナのプレビューを使用]のチェックボックスをオンにして、Visual Studioを再起動してください。まだ作業が進んでいるため、使用経験はまだ限られていることに注意してください。

重要な変更


.NET Frameworkと.NET Core にいくつかの大きな変更がありましたが  、WindowsフォームとWPF領域に関連するコードのほとんどは、同じ方法でCoreに移植されました。以前にWCFクライアント、コードアクセスセキュリティ、アプリドメイン、相互運用、リモーティングなどのコンポーネントを使用していた場合、.NET Coreに切り替えるにはコードをリファクタリングする必要があります。

覚えておくべきことの1つ：.NET Coreのデフォルトの出力パスは.NET Frameworkのパスとは異なるため、実行中のアプリケーションのファイル/フォルダー構造についてコードにいくつかの仮定がある場合、それはおそらくランタイムになります。

.NET関数の構成にも変更があります。 .NET Core machine.config はファイルではなくファイルを使用します <something>.runtimeconfig.jsonこれはアプリケーションに付属しており、同じ基本的な目的と同様の情報を持っています。system.diagnostics、、、  system.netまたは  などの一部の構成  system.servicemodelはサポートされていないため、これらのセクションのいずれかが含まれている場合、アプリケーション構成ファイルをロードできません。この変更は、System.Diagnostics以前は通常XML構成を使用して構成されていたトレースおよびクライアントWCFスクリプトに適用され  ます。.NET Coreでは、代わりにコードで構成する必要があります。再コンパイルせずに動作を変更するには、ソースMicrosoft.Extensions.Configuration またはからロードされた値を使用してトレースタイプとWCFをカスタマイズすることを検討してください  appSettings。

.NET Coreと.NET Frameworkの違いの詳細については、ドキュメントをご覧ください  。

.NET Frameworkから.NET Coreへの移植


まず、Portability Analyzer（Portability Analyzer）  を起動し、  必要に応じてコードを更新して、.NET Coreとの100％の互換性を確保します。ポータビリティアナライザーの使用手順は次の  とおりです。。アプリケーションに変更を加える前に、ソース管理を使用するか、コードのバックアップを作成することをお勧めします。これは、リファクタリングが希望どおりに失敗し、元の状態に戻ることにした場合に備えています。

アプリケーションが.NET Coreと完全に互換性がある場合、移植する準備が整います。開始点として、.NET Frameworkプロジェクトの.NET Coreへの変換を自動化するために作成したツールを試すことができます-try  -convert。

このツールは.NET Coreへの旅の出発点にすぎないことを覚えておくことが重要です。また、これはサポートされているマイクロソフト製品ではありません。移行の機械的な側面のいくつかは役立つ場合がありますが、すべてのシナリオまたはプロジェクトタイプを処理するわけではありません。ソリューションに、ツールが拒否するか変換できないプロジェクトがある場合、それらを手動で転送する必要があります。心配しないでください。これを行う方法について多くのレッスンを用意しています（記事の最後）。

try-convertツールは、古いスタイルのプロジェクトファイルを新しいSDKスタイルに転送し、対応するプロジェクトを.NET Coreで再構成しようとします。ライブラリについては、プラットフォーム（.NET Coreまたは.NET Standard）を選択する必要があります。の値を更新することにより、プロジェクトファイルでそれらの1つを指定できます。 <TargetFramework>。WPFやWindowsフォームなどの.NET Core固有の依存関係のないライブラリは、.NET Standardの恩恵を受けることができます。

<TargetFramework>netstandard2.1</TargetFramework>

多くの異なる.NETプラットフォームをターゲットとする呼び出し元が使用できるようにします。一方、ライブラリが.NET Coreを必要とする関数（たとえば、WindowsデスクトップUI API）を使用する場合、ライブラリは.NET Coreに重点を置く必要があります。

<TargetFramework>netcoreapp3.0</TargetFramework>

try-convertは、 コンピューターにインストールしてCLIから呼び出すことができるグローバルツールです  。

C:\> try-convert -p <path to your project>

または

C:\> try-convert -w <path to your solution>
