Feedback zur Speichergewährung in SQL Server 2019

Hallo! Im Vorfeld des Kursbeginns „MS SQL Server Developer“ haben wir eine weitere interessante Übersetzung für Sie vorbereitet.





Wenn der Optimierer die zum Ausführen der Abfrage erforderliche Speichermenge nicht korrekt berechnet, handelt es sich entweder um eine Speicherverschwendung, die von einem anderen Prozess verwendet werden könnte, oder es werden Daten auf die Festplatte verschüttet (Festplattenverschüttung). Um dieses Problem zu lösen, hat Microsoft Memory Grant Feedback hinzugefügt. In diesem Artikel erklärt Greg Larsen, wie das funktioniert.


Memory Grant Feedback in früheren Versionen von SQL Server (vor SQL Server 2019 oder 15.x) wurde nur für Abfragen implementiert, die im Batch-Modus ausgeführt werden. Abfragen im Stapelmodus scannen und berechnen bis zu 900 Zeilen gleichzeitig, im Gegensatz zu Abfragen im Zeilenmodus, wenn jeweils nur eine Zeile verarbeitet wird. In Version 15.x wurde das Feedback zur Speichergewährung erweitert, um das Abfragen im Zeichenfolgenmodus zu unterstützen.



Was ist Memory Grant Feedback? Hierbei wird die Berechnung des für eine Abfrage benötigten Speichers angepasst, wobei berücksichtigt wird, wie viel Speicher während der vorherigen Läufe verwendet wurde. Dies bedeutet, dass SQL Server die Speicherzuweisung bei der nächsten Ausführung reduziert, wenn die zwischengespeicherte Abfrage bei der letzten Ausführung zu viel Speicher verwendet hat. Wenn SQL Server eine Abfrage erkennt, die eine Festplatte verwendet, weil ihr beim letzten Zuweisen nicht genügend Speicher zugewiesen wurde, wird der Speicher für die Abfrage erhöht. Das Ziel der Rückmeldung zu Speicherzuweisungen besteht darin, die Speicheranforderungen bei jeder Ausführung der Abfrage anzupassen, bis die Abfrage die Speichermenge verwendet, die der Anzahl der verarbeiteten Zeilen entspricht.



Die Memory Grant Feedback- Funktion (sowohl im Stapel- als auch im Zeichenfolgenmodus) ist Teil der Funktionsfamilie Intelligent Query Processing (IQP, Intelligent Query Processing ). Abbildung 1 zeigt ein Diagramm mit allen IQP-Funktionen, die in Azure SQL Database und SQL Server 2019 vorhanden sind, sowie den Funktionen, die ursprünglich Teil der adaptiven Abfrageverarbeitung waren, die in früheren Versionen von Azure SQL Database und SQL Server 2017 enthalten war.

Warum die Schätzung des erforderlichen Speichers möglicherweise fehlerhaft ist



Es gibt viele Faktoren, die die für eine Anforderung erforderliche Speichermenge beeinflussen. Bei der Berechnung des insgesamt erforderlichen Speichers muss der Abfrageoptimierer die Speicheranforderungen für jede Operation in Bezug auf die Ausführung berücksichtigen. Hier sind einige Parameter, die der Optimierer bei der Berechnung des zur Ausführung der Abfrage erforderlichen Speichers berücksichtigt:



  • Die Kardinalität der an der Abfrage beteiligten Tabellen
  • Wählbare Lautsprecher
  • Geschätzte Zeilengröße
  • Müssen Daten sortiert und / oder verknüpft werden?
  • Läuft die Abfrage parallel?


Um den erforderlichen Speicher korrekt zu berechnen, muss SQL Server bei der Ausführung einer Abfrage viele Aspekte berücksichtigen, bei denen möglicherweise Speicher erforderlich ist. Wenn zu viel Speicher zugewiesen wird, verschwendet SQL Server Speicher, wenn er von anderen Prozessen verwendet werden kann. Wenn nicht genügend Speicher zugewiesen ist, werden die Daten bei der Anforderung auf der Festplatte zusammengeführt, was eine teure Operation ist.



SQL Server ist nicht immer gut darin, den Speicher beim ersten Mal zu berechnen. Veraltete oder ungenaue Statistiken sind der häufigste Grund, warum SQL Server die falsche Speichermenge berechnet. Aber selbst wenn die Statistiken korrekt sind, kann SQL Server die erforderliche Speichermenge aus einem anderen Grund überschätzen oder unterschätzen. Einer dieser Gründe ist, dass SQL Server die geschätzte Zeilengröße für zurückgegebene Spalten nicht korrekt auswertet. Im folgenden Beispiel sehen wir, wie eine Übergröße einer Zeile dazu führt, dass die Abfrage aufgrund von drei NVARCHAR (MAX) -Spalten, die immer NULL sind, eine übermäßige Grant-Warnung erhält.



Wenn SQL Server jetzt bei der ersten Ausführung der Abfrage den erforderlichen Speicher falsch berechnet, wird für nachfolgende Ausführungen das Feedback zur Speichergewährung verwendet. Jedes Mal, wenn die Abfrage ausgeführt wird, werden Informationen über den zugewiesenen Speicher für die vorherige Ausführung dieser Abfrage analysiert. Wenn beim vorherigen Start die Speicherzuordnung falsch war, korrigiert das Datenbankmodul die Speicherzuordnung.

Wenn die erforderliche Speichermenge nicht stabil ist



Es gibt Anforderungen, für die jede Ausführung eine andere Speichermenge erfordert. In diesem Fall versucht SQL Server nicht mehr, die Größe des zugewiesenen Speichers anzupassen, und deaktiviert das Feedback für eine solche Anforderung. Eine der Situationen, in denen dies geschieht, ist der Start einer gespeicherten Prozedur, in die bei jeder Ausführung unterschiedliche Parameterwerte übergeben werden.



Um diese Situation zu reproduzieren, wenn SQL Server das Feedback zu Speicherzuweisungen deaktiviert, führen wir eine gespeicherte Prozedur mit verschiedenen Parametern mehrmals aus. Daher benötigt die Abfrage innerhalb der Prozedur bei jeder Ausführung aufgrund der unterschiedlichen Anzahl zurückgegebener Datensätze je nach Parameter eine andere Speichermenge. Der gespeicherte Prozedurcode ist in Listing 4 dargestellt. 



USE WideWorldImporters;
GO
CREATE OR ALTER PROC TestProc (@KeyValue int)
AS
SELECT * FROM Sales.Orders
WHERE OrderID > 1 and OrderID < @KeyValue
ORDER BY OrderDate

gespeicherte Prozedur akzeptiert einen Parameter mit einem Namen @KeyValue

Deaktivieren des Memory Grant-Feedbacks



Wenn Sie aus irgendeinem Grund das Feedback zur Speichergewährung im Zeilenmodus (Feedback zur Speichergewährung - Zeilenmodus) deaktivieren möchten, gibt es verschiedene Möglichkeiten, dies zu tun. Die erste besteht darin, die Kompatibilitätsstufe für die Datenbank auf weniger als 150 festzulegen. Bei dieser Methode gibt es nur ein Problem: Alle anderen Funktionen, die in Version 15.x angezeigt wurden, werden ebenfalls deaktiviert. Sie können das Feedback auf Datenbankebene auch über eine Option deaktivieren ROW_MODE_MEMORY_GRANT_FEEDBACK



. 



USE WideWorldImporters;
GO
ALTER DATABASE SCOPED CONFIGURATION SET ROW_MODE_MEMORY_GRANT_FEEDBACK = OFF;
GO


Listing 7. Deaktivieren der Rückmeldung zum Zuweisen von Speicher für die Datenbank Deaktivieren der Rückmeldung zu Speicherzuweisungen



auf Datenbankebene ist hier deaktiviert. Anfragen für WideWorldImporters verwenden jetzt kein String-Feedback. Dies ist jedoch nicht die einzige Möglichkeit, Feedback zu deaktivieren. Sie können das Feedback für eine bestimmte Anforderung auch mit TIPP deaktivieren, wie in Listing 8 gezeigt. 



USE WideWorldImporters;
GO
SELECT * FROM Sales.Orders
ORDER By  OrderDate
OPTION
(USE HINT ('DISABLE_ROW_MODE_MEMORY_GRANT_FEEDBACK'));


Listing 8. Verwenden von TIPP zum Deaktivieren des Feedbacks



Listing 8 verwendet einen Hinweis DISABLE_ROW_MODE_MEMORY_GRANT_FEEDBACK



, der das Feedback zum Speichern von Speicher für eine einzelne Anforderung deaktiviert.



Optimierung der Berechnung des für die Anforderung erforderlichen Speichers



Rückmeldungen zu Speicherzuweisungen im Batch-Modus gibt es schon seit geraumer Zeit. Dieses Feedback wurde jedoch erst in Version 15.x für Abfragen im Zeilenmodus verfügbar. Um Feedback für Abfragen im Zeichenfolgenmodus zu aktivieren, muss lediglich die Datenbankkompatibilitätsstufe auf 150 festgelegt werden. Dies funktioniert natürlich, wenn Sie die aktuelle Version von Azure SQL Database oder SQL Server 2019 verwenden. Wenn die Feedbackfunktion für Speichergewährung aktiviert ist Im Zeichenfolgenmodus verwendet SQL Server Speicherzuordnungsinformationen aus früheren Ausführungen der zwischengespeicherten Abfrage, um die Speicherzuordnung für die aktuelle Ausführung anzupassen. Das Korrigieren des erforderlichen Speichers mit Rückmeldung bei jeder Ausführung der Abfrage führt früher oder später automatisch zur Berechnung der richtigen Speichermenge.
