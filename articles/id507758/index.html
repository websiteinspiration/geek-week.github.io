Memori JavaScript - tidak ada kebocoran









Cara Anda membuat dan mengakses data Anda dapat memengaruhi kinerja aplikasi Anda. Mari kita lihat caranya.



pengantar



JavaScript adalah bahasa tingkat yang sangat tinggi. Dengan demikian, sebagian besar pengembang tidak memikirkan bagaimana data disimpan dalam memori. Pada artikel ini, kita akan melihat bagaimana data disimpan dalam memori, bagaimana hal itu mempengaruhi prosesor dan memori, dan bagaimana cara Anda mendistribusikan data ke JS dan mengaksesnya mempengaruhi kinerja.



Cinta segitiga



Ketika komputer perlu melakukan beberapa perhitungan, prosesor (CPU) membutuhkan data untuk diproses. Dengan demikian, sesuai dengan tugasnya, ia mengirim ke memori permintaan untuk pengambilan sampel data melalui bus.



Ini terlihat seperti ini:







Jadi ini adalah segitiga romantis kami - Prosesor -> Bus -> Memori



Mempertahankan hubungan yang sehat dalam trio itu sulit



Prosesornya jauh lebih cepat daripada memori. Jadi, proses ini Prosesor -> Bus -> Memori -> Bus -> Prosesor "menghabiskan" waktu pada perhitungan. Saat memori sedang dipindai, prosesor tidak bekerja.



Untuk mencegah downtime, cache telah ditambahkan ke sistem. Kami tidak akan masuk ke perincian tentang apa itu cache atau apa jenis cache itu, tetapi cukuplah untuk mengatakan bahwa cache adalah memori internal prosesor.



Ketika CPU menerima perintah untuk dieksekusi, ia pertama-tama mencari data dalam cache, dan jika tidak ada data di sana, ia mengirimkan permintaan melalui bus.



Bus, pada gilirannya, mentransfer data yang diminta ditambah bagian bersyarat dari memori dan menyimpannya di cache untuk referensi cepat.



Dengan demikian, CPU akan memiliki ketidakpuasan yang lebih sedikit dengan betapa lambatnya memori dan, oleh karena itu, CPU akan memiliki lebih sedikit downtime.



Pertengkaran adalah bagian dari hubungan apa pun.



Masalah yang mungkin timbul - terutama ketika berhadapan dengan pemrosesan data dalam jumlah besar - adalah fenomena yang disebut cache miss.



Kehilangan cache berarti bahwa selama penghitungan, CPU mendeteksi bahwa CPU tidak memiliki data yang diperlukan dalam cache, dan CPU perlu meminta data ini melalui saluran reguler - Anda ingat bahwa ada banyak memori yang lambat.







Ilustrasi slip kas. Data dari array diproses, tetapi data di luar batas cache juga diminta untuk perhitungan - dan ini menciptakan "cache miss".



Oke, tapi saya pengembang JavaScript, mengapa saya harus peduli?



Pertanyaan bagus. Dalam kebanyakan kasus, Anda tidak memikirkannya. Namun, hari ini, karena semakin banyak data melewati server yang ditulis dalam Node.js, Anda lebih mungkin mengalami masalah cache cache ketika memilah-milah set data yang besar.



Saya akan percaya ketika saya melihatnya !!!

Ini adalah kelas yang disebut Boom. 



class Boom {
  constructor(id) {
    this.id = id;
  }
  setPosition(x, y) {
    this.x = x;
    this.y = y;
  }
}


Kelas ini (Boom) hanya memiliki 3 properti - id, x dan y.



Sekarang mari kita membuat metode yang mengisi x dan y.



Mari kita atur data: 



const ROWS = 1000;
const COLS = 1000;
const repeats = 100;
const arr = new Array(ROWS * COLS).fill(0).map((a, i) => new Boom(i));


Sekarang kita akan menggunakan data ini dalam metode: 



function localAccess() {
  for (let i = 0; i < ROWS; i++) {
    for (let j = 0; j < COLS; j++) {
      arr[i * ROWS + j].x = 0;
    }
  }
}


Apa yang dilakukan localAccess adalah secara linear melintasi array dan mengatur x ke 0.



Jika kita mengulangi fungsi ini 100 kali (lihat konstanta berulang), kita dapat mengukur berapa lama waktu yang diperlukan untuk mengeksekusi:

Harga untuk miss cache



Sekarang, sesuai dengan apa yang kami pelajari di atas, kami dapat mempertimbangkan masalah cache cache jika kami memproses data yang jauh selama iterasi. Data yang dihapus adalah data yang tidak ada dalam indeks yang berdekatan. Inilah fungsi untuk ini: 



function farAccess() {
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      arr[j * ROWS + i].x = 0;
    }
  }
}


Apa yang terjadi di sini adalah bahwa, pada setiap iterasi, kita beralih ke indeks, yang terletak pada jarak ROWS dari iterasi terakhir. Oleh karena itu, jika ROWS adalah 1000 (seperti dalam kasus kami), kami mendapatkan iterasi berikut: [0, 1000, 2000, ..., 1, 1001, 2001, ...].



Mari kita tambahkan ini ke tes kecepatan kami: 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local');
}, 2000);


Dan inilah hasil akhirnya:







iterasi nonlokal hampir 4 kali lebih lambat. Perbedaan ini akan meningkat dengan meningkatnya data. Ini karena prosesor idle karena cache miss.



Jadi berapa harga yang Anda bayar? Itu semua tergantung pada ukuran data Anda.



Oke, aku bersumpah aku tidak akan pernah melakukan itu!



Mungkin Anda tidak memikirkannya, tetapi ... ada beberapa kasus ketika Anda ingin mengakses array dengan logika yang tidak linier (misalnya, 1,2,3,4,5) atau tidak bersyarat (misalnya, untuk ( biarkan i = 0; i <n; i + = 1000)).



Misalnya, Anda mendapatkan data dari layanan atau basis data, dan Anda perlu memprosesnya, mengurutkannya atau memfilternya menggunakan logika kompleks. Ini dapat menyebabkan situasi yang mirip dengan yang ditunjukkan pada fungsi farAccess.



Berikut ini adalah ilustrasi yang (mudah-mudahan) mengklarifikasi situasi:







Data asli dalam memori vs data yang diurutkan dalam memori. Bilangan menunjukkan indeks objek dalam array sumber.



Melihat gambar di atas, kita melihat data dalam bentuk yang disimpan dalam memori (bar abu-abu atas). Di bawah ini kita melihat array yang dibuat ketika data berasal dari server. Akhirnya, kita melihat array yang diurutkan yang berisi referensi ke objek yang disimpan di berbagai posisi dalam memori.



Dengan demikian, iterasi pada array yang diurutkan dapat menyebabkan banyak kesalahan dalam cache, seperti yang ditunjukkan pada contoh di atas.

Perhatikan bahwa contoh ini adalah untuk array kecil. Kehilangan cache relevan untuk data yang jauh lebih besar.
Di dunia di mana Anda membutuhkan animasi yang halus di frontend dan di mana Anda dapat memuat setiap milidetik waktu prosesor di backend, ini bisa menjadi kritis.

Ada berbagai solusi untuk masalah ini, tetapi sekarang setelah Anda tahu alasan penurunan kinerja ini, Anda mungkin dapat menemukan solusi sendiri. Anda hanya perlu menyimpan data yang diproses sedekat mungkin dalam memori.

Teknik ini disebut pola desain Lokalitas Data .
Mari kita lanjutkan contoh kita. Asumsikan bahwa dalam aplikasi kita proses yang paling umum adalah pemrosesan data menggunakan logika yang ditunjukkan pada fungsi farAccess. Kami ingin mengoptimalkan data sehingga berjalan cepat di loop yang paling umum.



Kami mengatur data berikut: 



const diffArr = new Array(ROWS * COLS).fill(0);
for (let col = 0; col < COLS; col++) {
  for (let row = 0; row < ROWS; row++) {
    diffArr[row * ROWS + col] = arr[col * COLS + row];
  }
}


Jadi sekarang, dalam diffArr, objek yang ada dalam indeks [0,1,2, ...] dalam array asli sekarang ditetapkan sebagai berikut [0,1000,2000, ..., 1, 1001, 2001, ..., 2, 1002 , 2002, ...]. Angka-angka menunjukkan indeks objek. Ini mensimulasikan penyortiran array, yang merupakan salah satu cara menerapkan pola desain Lokalitas Data.



Untuk memeriksa ini dengan mudah, kami akan sedikit mengubah fungsi farAccess kami untuk mendapatkan array khusus: 



function farAccess(array) {
  let data = arr;
  if (array) {
    data = array;
  }
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      data[j * ROWS + i].x = 0;
    }
  }
}


Sekarang tambahkan skrip ke pengujian kami: 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local')
  setTimeout(() => {
    repeat(() => farAccess(diffArr), 'Non Local Sorted')
  }, 2000);
}, 2000);

Keluaran



Dalam artikel ini, kami menunjukkan pola desain Lokalitas Data. Bahkan, kami telah menunjukkan bahwa mengakses struktur data dengan cara yang tidak dioptimalkan dapat mengurangi kinerja. Kemudian kami mengoptimalkan data sesuai dengan cara kami mengolahnya - dan melihat bagaimana itu meningkatkan kinerja.
