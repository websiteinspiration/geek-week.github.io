.NETの非同期、Stack Overflowでの人気、「教会」ソフトウェア：Stephen Clearyへのインタビュー



スティーブンクリアリーは、スタックオーバーフローユーザーのトップ100に入っています。主に.NETでの非同期の回答に感謝します。彼の人生はプログラミングに限定されていません。Twitterでは、彼はまず自分自身について「クリスチャン」について書き、次に「開発者」について書きます。

現在、非同期ストリームの出現に関連して、彼の知識は特に関連しています。このトピックの報告者として、スティーブンの図はそれを頼みます。そしてすぐにDotNextで彼はそれについて教えてくれます。その間、私たちは彼にこのすべてについて尋ねました：宗教、スタックオーバーフロー、そして非同期性。

バイオグラフィー


-最初の質問は簡単です。あなたのプロの経歴について教えてください。

-私はかなり早くプログラミングに興味を持ちました、私は7-8歳でした。その後、学校でLogo言語で小さなプログラムを書くことができました。これが初めてのプログラミング体験でした。その後、12〜13歳のときに、最初のコンピューターに保存しました。そして、大学に入学したときにコンピューターサイエンスを選択したとき、それはかなり明白な決定でした。私は具体的な時間についてコンピューターにすでに興味を持っていました。

1998年に大学を卒業して、地元の会社で働き、産業用およびその他のファクトリオートメーション用の車両によって自動制御されるコンベアを扱っていました。 7年後、会社はデトロイトに移転しましたが、私はそこに引っ越したくなかったので、それ以来別の会社で働くことができました。 2013年以来、私は昨年、リモートでリモートで働いています-フェイスライフで。

-ロゴから始めて、今ではロゴスと呼ばれる主要製品を持つ会社にいるのはおかしいです。そして、あなたはそこで正確に何をしていますか？

-バックエンド：ロゴや会社の他の製品がアクセスするWebサービス。フェイスライフは教会関連の製品を幅広く取り揃えています。ロゴ-聖書研究用。そして説教をする人のために設計されたProclaimもあります（スライドなどの助け）。フェイスライフは歴史的に牧師のための製品を作りましたが、今では彼らは普通の人々を助けるものを加えています。

私はバックエンドを行っていますが、ASP.NET Web APIを頻繁に処理する必要があり、現在はDockerを使用しています。現在、私たちはクラウドとオンプレミスの一部に分かれており、何かをクラウドに転送するべきではありませんが、そこにいくつかのものを転送したいと思っています-私はそれをやっています。

宗教


-「教会」ソフトウェアを作成する会社に出会うことはあまりないので、はっきりさせておきたい：バックエンドでの作業はどこにでもあるように見えるのか、それとも具体性があるのか​​？そして、開発者は通常、信者であり、彼ら自身が会社の製品を使用するかどうか？

-作業に基本的な違いはありません。APIの適切な設計を他のあらゆる場所と同様に考える必要があります。一部の製品（特にデスクトップ製品）は非常に長い間使用されており、古いAPIを処理する必要があります。 Dockerの使用を開始するときは、下位互換性について考えることが重要です。一般的に、私たちは他の人とほぼ同じ懸念を抱いています。

フェイスライフは教会のために製品を作っていますが、それはキリスト教の会社ではありません。ここにたどり着くために、あなたはクリスチャンである必要はありません。第一に、そのような選択はアメリカの法律に違反します（雇用における宗教的差別になるでしょう）、そして第二に、会社はとにかくこれを望まないでしょう。

しかし、開発者が会社の製品を使用することに関しては、それはあらゆる面で奨励されています。たとえば、最近、ロゴの使用に関するトレーニングの日を過ごしました。社内ツールも多数あります。この場合、ツールを使用する部署で1日の仕事に出向き、これがどのように行われるかを直接確認することをお勧めします。

-バイオ分野のツイッターには、「クリスチャン」と「開発者」という言葉があります。あなたのアイデンティティのこれら二つの部分はどういうわけか交差していますか？

-別々だと思います。私はいつも仕事と生活を分離しようとしましたが、余暇には仕事をしていませんでした（常にうまくいくとは限りませんでした）。教会は私の性格の大きな部分を占めており、私の友人のほとんどはそこから来ています。仕事で友達はほとんどできませんでした。

「まあ、あなたの経歴はまだOSSを書くことに夢中になっている」とフリーソフトウェアを人々に与えることは利他的でキリスト教のように聞こえます。この場合、相関関係はありませんか？

-うーん、たぶん。哲学的な観点から見ると、オープンソースとキリスト教の関係が本当にわかると思います。どちらの場合も、あなたが言った「人々に与えること」に重点が置かれています。

「人道主義」タイプのオープンソースもあります。私自身はこれに関与していません;私は汎用ライブラリーを持っています。しかし、特に貧しい人々のいる場所で、他の生活に直接影響する何かに無料で取り組んだクリスチャンの開発者を知っています。たとえば、火災警報器が適切に機能していることを確認できるソフトウェア。ここにはもちろん、相関関係があります。

-あなたはStack Overflowでも活動しています-これはあなたがコミュニティに利他的に何かを与える方法でもありますか？

-それは私には言わないでください。私にとっては、それはより「教える」ように感じます。「教える」ことはキリスト教と関連することもありますが...私の家族には多くの教師がいて、私はこれを自分のやり方で続けています-職業としてではなく、スタックオーバーフローと会議を通して。だから私はこれに対する私のニーズを満たします。

スタックオーバーフロー


— Stack Overflow, . Windows Phone 8:



2019年に、ほとんど死んでしまったテクノロジーに関する回答のバッジを受け取ったのはどうしてですか。誰かがまだ彼女の質問をしていますか？

-バッジを取得するには、通常、少なくとも特定の数の回答と、これらの回答に対する特定の数の投票が必要です。私がこのバッジを取得したのは、何年か前に公開された私の回答の一部に投票したため、最終的に適切な数の投票を獲得したためです。 Windows Phone 8の質問を長い間見ていなかったので、これは面白いです！（笑い）

-SOでは、あなたの評判は320,000です-伝説のジョンスキートの評判の4分の1以上ですが、あなたは彼よりも1桁小さい答えを出しました。どうやってこれを手に入れたの？

-よくわかりません誰よりも先にサイトにアクセスした人の評判を得るのは簡単です。私はアーリーアダプターの1人でしたが、それでも多くの人より遅くに来ました。私は質問に答え始めました-最初は、1日1〜2回、これはジョン・スキートからめちゃくちゃ遠いです。私は非同期/待機、および同時実行性とマルチスレッド化のトピックに焦点を当てました-それが実際に私の専門分野になったからです。何年も前に私はすでに非同期に従事していたので、それは苦痛でした。 async / awaitが表示されたとき、私はそれらの意味がすぐにわかりました。私は適切なタイミングで適切な場所にいて、私の「教師の血」は人々が理解できる言語ですべてを説明するのに役立ちました。そして、Stack Overflowの場合、私はローカルの非同期/ awaitスペシャリストであることがわかりました。そしてジョン・スキート-ええと、彼はそれを直接言っていませんでした、しかし、非同期/待機の質問のほとんどは私に残っていると思います。しかし、もちろん、彼は私よりもはるかに多く答えて、追いつくことはできません！

-評判の数を見ると、「人が答えるのに半分の人生を費やしている」と考えるのは簡単ですが、実際にSOに費やす時間はどれくらいですか。

-あまりないです。 Stack Overflowを1日に数回確認し、通常は1〜3個の質問に答えます。そして、私は何年も前に残された回答に賛成票を投じます。これは、SOが先に来た人たちをどのように支援するかです。彼らは何年も前に最初に質問に答え、投票を受け取り、声がこれらの回答をより目に見える形にして、最終的には新しい投票を獲得します。これは、古い答えを促す雪崩効果です。

これはいくつかの問題を引き起こすように思われます。なぜなら、より新しいテクノロジーを参照しているため、より良い新しい回答がある場合があるためですが、古い回答は更新されません。しかし、一般的には、古い回答の投票の大半を獲得していますが、それでも毎日新しい回答に回答しようとしています。私はこれに多くの時間を費やさず、私は何年にもわたって常にそれをしています。

-John SkeetがDotNextで飛んできたとき、私たちはスタックオーバーフローの現在の状態と、サイトの問題であると彼が考えたものについて尋ねました。比較するのは興味深いです。これについてどう思いますか？

「特に昨年は、SOは改善していると思います。」彼らは今、質問の質を改善するために一生懸命に努力しています。多くの場合、このサイトで最初に何かを質問する人は、技術的な質問を正しく行う方法に慣れていません。

そして、これはインターネット上に常に存在する問題です。 90年代には、誰もがニュースグループに参加していたとき、彼女もそこにいました。 10年後、すべてがメーリングリストとGoogleグループで起こりました。 Stack Overflowプログラミングの質問に関する新しいリソース。しかし、プログラミングについては常に質問があり、良い質問をする方法は常に問題であり、友好的なコミュニティを維持する方法も問題でした。おそらく、これらの問題を完全に解決することはできません。私たちはそれに取り組むべきではないと言っているのではありません-私たちは間違いなくすべきです。しかし、過去を見ると、90年代でさえ、技術的な質問をする方法に関するチュートリアルをすでに書いていることがわかります。

Stack Overflowには新しい問題がいくつかあります。あなたは彼らの多くが最初の質問をする前に誰かに尋ねたことがないという事実から始めることができます。したがって、多くの場合、質問に回答するために提示する必要があるすべてのことを理解していないだけです。

そして、あなたがタスクに取り組んでいると想像してください。あなたはコードで頭を抱えています。あなたの頭はこれでいっぱいです。そして、あなたは決して役に立たないいくつかの特定のものを見ます。これは、大規模なシステムの小さな具体的な詳細であり、通常、「どうすればこの小さなことを行うことができますか？」と質問します。多くの場合、「Xを作成する方法」という質問の正解は「Xを実行しないで、Yを実行する」です。これは、最初の質問を書くときによく陥る罠です。彼らは、現在の形での彼らの答えが「無反応」であることを理解していません。

そして、質問の質の問題に加えて、特にスタックオーバーフローでは、誰もができるだけ早く答えようとして、ポイントを求めて苦労する傾向があります-質問をすぐに閉じるか、最も心地よい答えではない落書きをします。私は「悪意のある」という意味ではなく、率直に悪意のあるコメントをほとんど見ていません。むしろ、それらは過酷であり、質問の著者にとって、これは友好的ではないと解釈されます。

Stack Overflowはこれを修正するためにいくつかの簡単な手順を実行しました。さて、人々が最初に質問するとき、サイトはあなたがそれに含める必要があるものをあなたに伝えます。彼らは以前に「これらの同様の質問を見てください」と付け加えました、それは良い最初のステップでした。そして今、最初の質問のために完了する必要があるシステム全体があり、それはそれをうまく構成するのに役立ちます。

また、回答を書いた人にリマインダーを追加します。「これは初心者で、親しみやすい」のようなものであり、これはほとんどの人が初めて質問をうまく書けないことを思い出させる良い方法であり、これは完全に正常です。一般的に、彼らはそれに取り組んでいます。この問題は完全に解決されますか？私は疑う。しかし、進歩は可能です。

非同期性


-iPhoneの10周年に捧げられた投稿で、モバイルアプリケーションは応答性がなければならないため、その外観が非同期プログラミングに影響を与えたと書いています。最近開発を始め、非同期/待機なしで世界を見たことがない人のために、非同期の開発に関する一般的な歴史的背景を教えてください。

-ええと、非同期プログラミングは常に可能でした。 60年代には彼らはすでにそれをやっていたと思います。そして、彼には長い間同じ利点がありました。より応答性の高いUIと、よりスケーラブルなサーバー部分です。そして、サポートは常に問題でした。非同期コードを維持することは非常に困難でした。当初は、コールバックに基づいて構築されていました。

大きな一歩はプロミスの登場だったと思います。それはJavaScriptでそれを呼んだものであり、C＃ではTaskまたはValueTaskです。これは大きな前進でした。これで、操作を表すオブジェクトができました。進捗状況などを監視できます。そして、任意の言語でのasync / awaitは、基本的にはPromiseの構文糖衣です。

Promiseの登場は、非同期コードに最も影響を与えたイベントだと思います。また、非同期/待機の場合、ステートマシンが作成されていることが重要です。昔は、コールバックを処理するときに、自分でステートマシンを作成する必要がありました。そして、何らかの条件や移行を必然的に忘れ、何もうまくいかなかったため、それは常に困難でした。また、デバッグは非常に困難でした。 Async / awaitは、非同期コードを作成する機能を提供しませんでしたが、サポート可能なコードを作成する機能をもたらしました。

-そして今、あなたは状況が落ち着いたのを見ていますか、またはすぐに新しい変化が私たちを待っていますか？

-今はすべてがかなり安定していると思います。 Async / awaitは革新的なものに見えましたが、これまでに非同期プログラミングを行ったことがない人だけが対象です。そして働いた人にとって、それは非常に自然な感じがしました。しかし、一般的に、これは大きなイベントでした。

そして今、別のイベントが.NET Core 3に付属しています。これらは、非同期の列挙型ですが、誰もが非同期ストリームと呼ぶものを実行します。非同期ストリームとは何の関係もないStreamタイプがすでに存在するため、これは人々を混乱させると思います。一般的に、より多くの改善があります。 async / awaitが最初に自分自身を発表したときと同じくらい巨大なものを見ますか？私はそうは思いません。

新しいパラダイムシフトが必要な場合は、常にアクターベースのシステムの可能性があります。または、すべての非同期が暗黙であるgoroutineのようなもの。私の意見では、これらは多少似ています。問題は、.NETではこれを追加するのがそれほど簡単ではないことです。これを行うには.NETの制限が多すぎると思いますので、これは起こりそうにありません。同時実行へのアプローチが今日のマルチスレッド化と同じではない、アクター世界またはゴルーチン世界への大規模な移行が見られる場合、これには完全に新しい言語とランタイムが必要になります。そして.NETは価値がない。そして、プログラミング全体がそのような飛躍を遂げるとは思いません。間違いかもしれませんが、現在の立場は以下のとおりです。

-すべてが時間とともにどれだけ変化するかという問題の詳細。多くのプログラミングブックは急速に時代遅れになっていますが、変更が少ない場所では、より長持ちします。C＃Cookbookでの並行性はどうですか、どれだけの更新が必要ですか？

- 良い質問。最近、第2版を発行しました。第1版は2014年に発行されました。つまり、5年が経過し、すでに第2版になっています。私にとっては、急速な発展のように見えます。

それは本の書き方次第だと思います。なるべく古くならないように書きました。 Windows Phone 8などを参照しないようにする必要があるだけですが、とにかく時代遅れになってしまいます。非同期ストリームなどが出てきますので、本に載せたかったです。その結果、何かが古くなりましたが、ほとんどの資料は変更なしで新版に移行しました。

そしてもちろん、それは本が誰のためにあるかに依存します。もちろん、Visual Studio 2008の使用に関する本はすぐに期限切れになります。でも本物のクラシックの場所があると思います。私はコードコンプリートを世界で最も優れたプログラミングの本の1つと考えています。そして、それはどのくらい書かれましたか？私も知らない。数十年前。そして、これはまだ素晴らしい本です！その中の何かは時代遅れですが、全体的にはまだ素晴らしいです。

— async/await, — DTD :



彼は非同期を嫌い/重要性を待つ-彼の意見では、これはコード全体に広がり、スレッドのブロックにつながる可能性があります。これについてあなたはどう思いますか？他に何かを設計する価値はありましたか？

-はい、おそらく非同期/待機がプロジェクト全体に広まっているという事実が最も一般的な不満です。ここでいくつかのことを述べたいと思います。

まず、非同期コードは、真に非同期であるために、それを呼び出すすべての人から非同期を必要とします。そしてそれは動きません。どんなパラダイムを使用しても（古いものでも）、結局はそれに遭遇します。私は時系列ですべてのパラダイムを調べて、コードがどのようにすっきりとしていてより良くなったのかを示すレポートがありました。

非同期/待機の広範囲にわたる支配を回避するためのいくつかの異なるアプローチがあります。 1つは、すべての入力/出力を個別のオブジェクトに分離することです。 Ports＆Adaptersなどのデザインパターンを使用できます。これにより、ビジネスロジックの外部にすべてのI / Oを含めることができ、非同期/待機をまったく必要としません。最近、ビジネスロジックがI / Oを処理しないようにプロジェクトをリファクタリングすることによって、「どこでも非同期」を防ぐ方法についての全体的なレポートを見ました。このアプローチをお勧めします。

async / awaitの優位性に対処する別のアプローチがありますが、.NETでは実現できません。私の意見では、Goはこれをゴルーチンで実行しようとしました。実際、すべてがまったくそこにあります-それは確かな非同期/待機です。言語にasync / awaitを削除して、すべてに追加して暗黙的にすることができます。

他に方法はありません。 async / awaitが現れると、誰か（正確には誰なのか覚えていません）がゾンビウイルスのようなものだと言いました。コードの一部が感染するとすぐに配布が始まります。

-一部の開発者は、アクターモデルを使用し、フロー制御の単純さによってこれを説明しています（実際、アクターはシングルスレッドです）。ライブラリを適切に選択することで、開発者はプログラミングの並行性の複雑さを取り除くことができると思いますか？

「まあ、そうではない。」俳優のモデルでも問題が残っているからです。マルチスレッドプログラムのような競合状態は発生しませんが、友情が困難になります。

たとえば、メッセージの遅延や非同期メッセージの問題です。これらは通常、アクターモデルでのデッドロックを防ぐために必要です。ただし、非同期メッセージを使用するアクターモデルもデッドロックを引き起こす可能性があります。非同期メッセージも、独自の調整問題を引き起こす可能性があります。通常、このレベルでは、べき等性にも対処する必要があります。

また、非同期メッセージでアクターを使用すると、状態管理が完全にメッセージ内で発生しない限り、状態管理が非常に混乱する可能性があります。そして、この場合でも、結果の一貫性に問題があります。一般的に、私の見解では、アクターモデルはすべての問題を解決できるわけではありません。私はそれをこのように説明します。それは、問題が発生する正確な場所でのみ変更できます。

「あなたはドナーとして知られていますが、TypeScriptなどの他の言語を使用しています。」それらを試してみると、C＃と.NETをどのように見ることができますか？ C＃で見たいと思っているものがありますか？

-いい質問ですねC＃は他の言語から多くを取りました。彼が多くをとったそれらの1つはPythonです。そして私はそれが好きです。私は何年もPythonで書いていませんが、これは非常によく設計された言語だと思います。 C＃がPythonから借りた列挙子ブロックに本当に感謝しています。 Pythonは非同期ストリームが最初に登場した場所の1つだったので、C＃がそこから取得したと言えます。

異なる言語で私は異なるものが好きです。私は一般的に静的型付けを好むので、最近Pythonを使用していません。同じ理由で、JavaScriptの代わりにTypeScriptを使用しています。 JavaScriptは、その1つのスレッディングのためだけに利点があります。たとえば、Reactive Extensionsの関連性を見ると、.NETではこれが特に受け入れられていないことがわかります。そしてJavaScriptでは、どこでもRxを見ることができます。

-最後の2つの質問は、DotNextに関するレポートについてです。そこで非同期ストリームについて話します-.NET開発者に何を期待し、なぜこのレポートが役立つのか簡単に説明できますか？

-それで、非同期ストリームに関する私のレポート：なぜそれらが言語に追加されたのか、そしてそれらの使用の主なシナリオは何か。私はこれを実利的にアプローチし、内部で何が起こっているのかについての詳細はすべて説明しません。非同期ストリームを自分で適切に使用するために知っておく必要があることと、コードで非同期ストリームを消費するために知っておく必要があることがあり、シーンに入るときにこれを行う必要があります。

-そして最後の質問。会議に参加するのは非常に困難です。なぜDotNextに来ることにしたのですか。

-はい、私は常に会議に行く人ではありません。私は主に開発者です。私はそれが話者として私に役立つと思います。私は非常に実用的であり、レポートで私は自分自身が有用だと思うものを与えます。

プログラム委員会から1年半前に連絡がありましたが、最初は来ることに反対しませんでしたが、これが私のスケジュールに合うようにする必要がありました。そして今年は、非同期ストリームのリリースにより、すべての会議がそれらについてのレポートを求めているため、いつものように年に2回の会議ではなく、もう少し会議を行うことにしました。このおかげでDotNextにアクセスできてうれしいです。

DotNext 2019モスクワは11月6〜7日に開催されます。スティーブン・クリアリーのスピーチに加えて、寄付者のための他の何十ものレポートがあります-サイトでは、完全なプログラムを見ることができます。
