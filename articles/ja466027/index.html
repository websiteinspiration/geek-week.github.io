抜粋。並行してテストを実行する


テストスイートの実行には時間がかかる場合があります。これは、テストスイートの完了に数分かかる大規模なプロジェクトでよくあることです。デフォルトでは、pytestは特定の順序で順次テストを実行します。

ほとんどのコンピュータにはマルチコアプロセッサが搭載されているため、テストを分割して複数のコアで実行すると、速度が向上します。

このため、pytestには、pipを使用してインストールできるpytest-xdistプラグインがあります。このプラグインはpytestコマンドラインを––numprocesses（省略形–n）引数で拡張します。これは引数として使用されるコアの数を取ります。 pytest –n 4を起動すると、4つの並列プロセスでテストスイートが実行され、使用可能なコアの負荷間のバランスが維持されます。

コアの数が異なる場合があるため、プラグインはautoキーワードも値として使用します。この場合、使用可能なコアの数が自動的に返されます。

フィクスチャを使用してテストで使用されるオブジェクトを作成する


単体テストでは、テストの実行前後に一連の標準操作を実行する必要があることが多く、これらの指示には特定のコンポーネントが含まれます。たとえば、アプリケーション構成の状態を表すオブジェクトが必要な場合があり、各テストの前に初期化し、実行後に初期値にリセットする必要があります。同様に、テストが一時ファイルに依存している場合、このファイルはテストの前に作成し、後で削除する必要があります。このようなコンポーネントはフィクスチャと呼ばれます。これらはテスト前にインストールされ、実行後は表示されなくなります。

pytestでは、フィクスチャは単純な関数として宣言されます。フィクスチャ関数は目的のオブジェクトを返す必要があるため、それが使用されている場所のテストで、このオブジェクトを使用できます。

以下は簡単なフィクスチャーの例です：

import pytest

@pytest.fixture
def database():
    return <some database connection>

def test_insert(database):
    database.insert(123)

データベースフィクスチャは、リストにデータベース引数があるテストで自動的に使用されます。test_insert（）関数は、データベース（）関数の結果を最初の引数として受け取り、この結果を必要に応じて使用します。このようにフィクスチャを使用すると、データベース初期化コードを何度も繰り返す必要がなくなります。

コードテストのもう1つの一般的な機能は、フィクスチャが機能した後に過剰を削除する機能です。たとえば、データベース接続を閉じます。ジェネレーターとしてフィクスチャーを実装すると、検証済みオブジェクトをクリーニングする機能が追加されます（リスト6.5）。

リスト6.5 検証済みオブジェクトをクリアする


import pytest

@pytest.fixture
def database():
    db = <some database connection>
    yield db
    db.close()

def test_insert(database):
    database.insert(123)
ここでは、yieldキーワードを使用してデータベースからジェネレーターを作成したため、yieldステートメントの後のコードはテストの最後にのみ実行されます。このコードは、テストの最後にデータベース接続を閉じます。

他のテストがすでに開いている接続を使用する可能性があるため、各テストのデータベース接続を閉じると、計算能力が不当に浪費される可能性があります。この場合、スコープ引数をフィクスチャーデコレーターに渡し、そのスコープを示します。

import pytest

@pytest.fixture(scope="module")
def database():
    db = <some database connection>
    yield db
    db.close()

def test_insert(database):
    database.insert(123)

scope =“ module”パラメータを指定することで、モジュール全体に対して一度フィクスチャを初期化しました。これで、それを要求するすべてのテスト関数でオープンデータベース接続を使用できるようになります。

テストの前または後にいくつかの一般的なコードを実行して、フィクスチャを各テスト関数の引数として指定するのではなく、autouseキーワードで自動的に使用されるように定義できます。 True引数、autouseキーワードを使用してpytest.fixture（）関数を具体化すると、フィクスチャが宣言されているモジュールまたはクラスでテストを実行する前に、必ずフィクスチャが呼び出されます。

    curuser = os.environ.get("USER")
    os.environ["USER"] = "foobar"
    yield
    os.environ["USER"] = curuser

ダミーオブジェクトを使用したマネージテスト


ダミーオブジェクト（またはスタブ、モックオブジェクト）は、実際のアプリケーションオブジェクトの動作を模倣したオブジェクトですが、特別な制御された状態にあります。これらは、テストの条件を完全に説明する環境を作成するのに最も役立ちます。テストされたオブジェクトを除くすべてのオブジェクトをダミーオブジェクトに置き換えて分離し、コードテスト用の環境を作成できます。

1つの使用例は、HTTPクライアントを作成することです。考えられる値ごとにすべての状況とスクリプトを実行できるHTTPサーバーを作成することは、ほとんど不可能です（あるいは、信じられないほど難しい）。 HTTPクライアントは、エラーシナリオのテストが特に困難です。

標準ライブラリには、ダミーオブジェクトを作成するためのモックコマンドがあります。Python 3.3以降、mockはunittest.mockライブラリと統合されました。したがって、以下のコードスニペットを使用して、Python 3.3以前との下位互換性を提供できます。

try:
    from unittest import mock
except ImportError:
    import mock

モックライブラリは非常に使いやすいです。mock.Mockオブジェクトで使用可能な属性は、実行時に動的に作成されます。この属性には任意の値を割り当てることができます。リスト6.7では、mockを使用してダミー属性のダミーオブジェクトを作成しています。

リスト6.7 mock.Mock属性へのアクセス


>>> from unittest import mock
>>> m = mock.Mock()
>>> m.some_attribute = "hello world"
>>> m.some_attribute
"hello world"
リスト6.8のように、可変オブジェクトのメソッドを動的に作成することもできます。ここでは、常に42を返し、必要なものを引数として取るダミーメソッドを作成しています。

リスト6.8 mock.Mockダミーオブジェクトのメソッドを作成する


>>> from unittest import mock
>>> m = mock.Mock()
>>> m.some_method.return_value = 42
>>> m.some_method()
42
>>> m.some_method("with", "arguments")
42
ほんの数行で、mock.Mockオブジェクトにsome_method（）メソッドが追加されました。これは42を返します。これは、任意のタイプの引数を受け取りますが、引数の検証は行われません。

動的に生成されたメソッドには、（意図的な）副作用もあります。値を返すボイラープレートメソッドだけではなく、便利なコードを実行するように定義できます。

リスト6.9は、副作用のあるダミーメソッドを作成します。これは、文字列「hello world」を表示します。

リスト6.9 副作用のあるmock.Mockオブジェクトのメソッドを作成する


  >>> from unittest import mock
  >>> m = mock.Mock()
  >>> def print_hello():

❶ >>> m.some_method.side_effect = print_hello

関数全体をsome_method❶属性に割り当てました。技術的には、テストに必要なコードをダミーオブジェクトに含めることができるため、テストでより複雑なスクリプトを実装できます。次に、このオブジェクトを、それを期待する関数に渡す必要があります。

❷call_count属性は、メソッドが呼び出された回数を確認する簡単な方法です。

モックライブラリはアクションチェックパターンを使用します。つまり、テスト後に、ダミーに置き換えられたアクションが正しく実行されたことを確認する必要があります。リスト6.10はこれらのチェックを実行するためにassert（）メソッドをダミーオブジェクトに適用します。

モックライブラリを使用して、外部モジュールの関数、メソッド、またはオブジェクトを置き換えることもできます。リスト6.11では、os.unlink（）関数を独自のダミー関数に置き換えました。

mock.patch（）をコンテキストマネージャーとして使用すると、ターゲット関数が選択した関数に置き換えられます。これは、コンテキスト内で実行されるコードが修正されたメソッドを使用するために必要です。mock.patch（）メソッドを使用して、外部コードの任意の部分を変更し、アプリケーションのすべての条件をテストするように動作させることができます（リスト6.12）。

リスト6.12 mock.patch（）を使用して多くの動作をテストする


  from unittest import mock

  import pytest
  import requests

  class WhereIsPythonError(Exception):
      pass

❶ def is_python_still_a_programming_language():
      try:
          r = requests.get("http://python.org")
      except IOError:
          pass
      else:
          if r.status_code == 200:
              return 'Python is a programming language' in r.content
      raise WhereIsPythonError("Something bad happened")

  def get_fake_get(status_code, content):
      m = mock.Mock()
      m.status_code = status_code
      m.content = content

      def fake_get(url):
        return m

      return fake_get

  def raise_get(url):
      raise IOError("Unable to fetch url %s" % url)

❷ @mock.patch('requests.get', get_fake_get(
      200, 'Python is a programming language for sure'))
  def test_python_is():
      assert is_python_still_a_programming_language() is True

  @mock.patch('requests.get', get_fake_get(
      200, 'Python is no more a programming language'))
  def test_python_is_not():
      assert is_python_still_a_programming_language() is False

  @mock.patch('requests.get', get_fake_get(404, 'Whatever'))
  def test_bad_status_code():
      with pytest.raises(WhereIsPythonError):
          is_python_still_a_programming_language()

  @mock.patch('requests.get', raise_get)
  def test_ioerror():
      with pytest.raises(WhereIsPythonError):
          is_python_still_a_programming_language()


6.12実装のすべてのインスタンスのルックステストケースリストPythonはプログラミング言語である文字列でpython.org ❶を。選択したWebページ上でテストが特定の行を見つけられないオプションはありません。否定的な結果を得るには、ページを変更する必要がありますが、これは実行できません。しかし、モックの助けを借りて、トリックに行き、リクエストの動作を変更して、指定された文字列を含まない架空のページでダミーのレスポンスを返すことができます。これにより、python.orgに特定の文字列が含まれていないという否定的なシナリオをテストし、プログラムがそのようなケースを正しく処理できることを確認できます。

この例では、mock.patch（）デコレータバージョンを使用しています。ダミーオブジェクトの動作は変更されず、テスト関数のコンテキストで例を設定する方が簡単でした。

ダミーオブジェクトを使用すると、問題のシミュレーションに役立ちます。サーバーは404エラー、I / Oエラー、またはネットワーク遅延エラーを返します。コードが正しい値を返すか、それぞれの場合に正しい例外がスローされることを確認できます。これにより、コードの予期される動作が保証されます。

カバレッジでテストされていないコードを特定する


単体テストへのすばらしい追加は、カバレッジツールです[コードカバレッジは、テストで使用される尺度です。テストプロセス中に実行されたプログラムのソースコードのパーセンテージを表示します。]、テストされていないコードを検索します。コード分​​析と追跡ツールを使用して、実行された行を識別します。単体テストでは、コードのどの部分が再利用され、どの部分がまったく使用されなかったかを識別できます。テストを作成することが必要であり、テストでカバーするのを忘れたコードの部分を見つける機能は、このプロセスをより楽しくします。

シェルから使用できるように、pipを介してカバレッジモジュールをインストールします。

注意


OSのインストーラーを介してモジュールがインストールされている場合、このコマンドはpython-coverageと呼ばれることもあります。この例は、Debian OSです。


オフラインでのカバレッジの使用は非常に簡単です。起動して「デッドウェイト」になることのないプログラムの部分、つまりプログラムのパフォーマンスを変更せずに削除できないコードが表示されます。この章で前述したすべてのテストツールは、カバレッジと統合されています。

pytestを使用する場合は、pip install pytest-pycovを介してpytest-covプラグインをインストールし、いくつかのスイッチを追加して、テストされていないコードの詳細な出力を生成します（リスト6.13）。

--covオプションは、テストの終了時にカバレッジレポートの出力を有効にします。プラグインがレポートを適切にフィルタリングできるように、パッケージ名を引数として渡す必要があります。出力には、実行されていないコード行が含まれます。つまり、テストされていません。あとは、エディターを開いて、このコードのテストを書くだけです。

カバレッジモジュールはさらに優れており、HTML形式で明確なレポートを生成できます。 -–cov-report-htmlを追加するだけで、コマンドを実行したHTMLcovディレクトリにHTMLページが表示されます。各ページには、ソースコードのどの部分が実行されたか、実行されなかったかが表示されます。

さらに先に進むには、-cover-fail-under-COVER_MIN_PERCENTAGEを使用します。これにより、コードの最小パーセンテージをカバーしていない場合、テストスイートが失敗します。カバレッジの大きなパーセンテージが良い目標であり、テストツールはテストカバレッジのステータスに関する情報を取得するのに役立ちますが、パーセンテージ自体はあまり有益ではありません。図6.1は、カバレッジがパーセントのカバレッジレポートの例を示しています。

たとえば、コードをテストで100％カバーすることは価値のある目標ですが、これは必ずしもコードが完全にテストされていることを意味するわけではありません。この値は、プログラムのすべてのコード行が満たされていることを示すだけで、すべての条件がテストされたことを示しているわけではありません。

カバレッジ情報を使用してテストスイートを拡張し、実行されないコード用に作成することは価値があります。これにより、プロジェクトのサポートが簡略化され、全体的なコード品質が向上します。
