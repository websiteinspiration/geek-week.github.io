忙しいプロジェクトでCephを使用するためのヒントとコツ



さまざまなワークロードプロジェクトでCephをネットワークストレージとして使用すると、一見しただけでは簡単ではないようなさまざまなタスクが発生する可能性があります。例えば：

  • 新しいクラスター内の以前のサーバーを部分的に使用して、古いCephから新しいCephにデータを移行します。
  • Cephのディスクスペースの配分の問題の解決。

このようなタスクに対処するには、データを失うことなくOSDを正しく抽出する必要があります。これは、特に大量のデータに当てはまります。これについては、記事で説明します。

以下で説明する方法は、Cephのすべてのバージョンに関連しています。さらに、大量のデータをCephに保存できるという事実が考慮されます。データの損失やその他の問題を防ぐために、いくつかのアクションがいくつかの他のアクションに「分割」されます。

OSD序文


検討中の3つのレシピのうち2つはOSD（Object Storage Daemon）に特化しているので、実際の部分に入る前に、Cephの内容とその重要性について簡単に説明します。

まず第一に、Cephクラスター全体が多くのOSDで構成されているということです。 Cephの空きデータ量が多いほど、より多くの空きデータボリュームがあります。ここから、OSDの主な機能を簡単に理解できます。Cephオブジェクトデータをすべてのクラスターノードのファイルシステムに保存し、それらへのネットワークアクセスを提供します（読み取り、書き込み、その他の要求のため）。

同じレベルで、レプリケーションパラメータは、異なるOSD間でオブジェクトをコピーすることによって設定されます。そして、ここではさまざまな問題が発生する可能性があります。その解決策については後で説明します。

ケース番号1。データを失うことなくCephクラスターからOSDを安全に取得


OSDを削除する必要性は、クラスターからのサーバーの撤去（たとえば、それを別のサーバーに置き換えるなど）が原因で発生した可能性があり、それにより記事が執筆されました。したがって、操作の最終的な目標は、特定のサーバー上のすべてのOSDとマネーを抽出して、停止できるようにすることです。

コマンドの実行プロセスで、必要なOSDを誤って示す状況の便宜と排除のために、個別の変数を設定します。その値は、削除するOSDの数になります。私たちはそれを呼び出します${ID}-以下、そのような変数は私たちが作業するOSD番号を置き換えます。

作業を始める前の状態を見てみましょう：

root@hv-1 ~ # ceph osd tree
ID CLASS WEIGHT  TYPE NAME      STATUS REWEIGHT PRI-AFF
-1       0.46857 root default
-3       0.15619      host hv-1
-5       0.15619      host hv-2
 1   ssd 0.15619      osd.1     up     1.00000  1.00000
-7       0.15619      host hv-3
 2   ssd 0.15619      osd.2     up     1.00000  1.00000

OSDの削除を開始するには、OSD reweightをゼロまでスムーズに実行する必要があります。したがって、他のOSDとのバランスをとることで、OSDのデータ量を減らします。これを行うには、次のコマンドを実行します。

ceph osd reweight osd.${ID} 0.98
ceph osd reweight osd.${ID} 0.88
ceph osd reweight osd.${ID} 0.78

...ゼロまで続きます。

更新：記事のコメントにnorebalance+を使用した方法が記載されていbackfillます。解決策は正しいですが、最初に状況を確認する必要がありnorebalanceます。これは、OSDでネットワーク負荷を発生させたくないときに使用するためです。osd_max_backfillリバランスの速度を制限する必要がある場合に使用されます。その結果、リバランスが遅くなり、ネットワーク負荷が軽減されます。データを失わないように、

スムーズなバランス調整が必要です。これは、OSDに大量のデータが含まれている場合に特に当てはまります。コマンドの実行後にreweightすべてが成功したことを確認するには、それを実行するceph -sか、別のターミナルウィンドウで実行します。ceph -w変化をリアルタイムで観察するため。

OSDが「空」の場合は、標準操作を開始してOSDを削除できます。これを行うには、目的のOSDを次の状態にしdownます。

ceph osd down osd.${ID}

クラスターから「プル」OSD：

ceph osd out osd.${ID}

OSDサービスを停止し、FSでそのパーティションをアンマウントします。

systemctl stop ceph-osd@${ID}
umount /var/lib/ceph/osd/ceph-${ID}

CRUSHマップ からOSDを削除します。

ceph osd crush remove osd.${ID}

OSDユーザーを削除します。

ceph auth del osd.${ID}

そして最後に、OSD自体を削除します。

ceph osd rm osd.${ID}

注：Ceph Luminous以降を使用している場合、OSDを削除するための上記の手順は2つのコマンドに減らすことができます。

ceph osd out osd.${ID}
ceph osd purge osd.${ID}

上記の手順を実行した後、コマンドが実行されたceph osd tree場合、作業が実行されたサーバーには、上記の操作が実行されたOSDがなくなっていることがわかります。

root@hv-1 ~ # ceph osd tree
ID CLASS WEIGHT  TYPE NAME     STATUS REWEIGHT PRI-AFF
-1       0.46857      root default
-3       0.15619      host hv-1
-5       0.15619      host hv-2
-7       0.15619      host hv-3
 2   ssd 0.15619      osd.2    up     1.00000  1.00000

途中で、Cephクラスターの状態がに移行しHEALTH_WARN、OSDの数と使用可能なディスク領域の量も減少することがわかります。

次に、サーバーを完全に停止して、Cephからサーバーを削除する場合に必要な手順について説明します。この場合、サーバーをシャットダウンする前に、このサーバー上のすべてのOSDを抽出する必要があることに注意してください。

このサーバーにOSDが残っていない場合は、OSDを削除した後hv-2、次のコマンドを実行してサーバーをOSDカードから除外する必要があります。

ceph osd crush rm hv-2

別のサーバー（この場合は）で以下のコマンドを実行してmon、サーバーから 削除します。hv-2hv-1

ceph-deploy mon destroy hv-2

その後、サーバーを停止して、後続のアクション（サーバーの再デプロイメントなど）を続行できます。

ケース番号2。作成済みのCephクラスターでのディスク領域の割り当て


2番目のストーリーは、PG（配置グループ）の序文から始まります。CephでのPGの主な役割は、主にCephオブジェクトの集約とOSDでのレプリケーションです。必要なPGの数を計算できる式は、 Cephドキュメントの対応するセクションにあります。同じ場所で、この問題も特定の例で分析されました。

したがって、Cephの運用中によくある問題の1つは、Cephのプール間でOSDとPGの量が不均衡になることです。一般に、正しく選択されたPGの値は、クラスターの信頼できる操作の鍵となります。次に、反対の場合に何が起こり得るかを検討します。

  2. , , PG, .

実際には、より深刻な問題が発生します。OSDの1つでデータがオーバーフローします。これは、Cephが使用可能なデータ量を計算するときに（各プールMAX AVAILのコマンドの出力でceph df個別に確認できます）、OSDで使用可能なデータ量に依存するためです。少なくとも1つのOSDに十分なスペースがない場合、すべてのOSD間でデータが適切に分散されるまで、それ以上のデータの書き込みは機能しません。

これらの問題は、Cephクラスターの構成段階で大幅に解決されることを明確にしておく価値があります。使用できるツールの1つがCeph PGCalcです。その助けを借りて、必要なPGの量が視覚的に計算されます。ただし、Cephクラスターがすでにある状況では、これを使用できます。正しく構成されていません。ここで、修正作業の一環として、PGの数を減らす必要がある可能性が高く、この機能は古いバージョンのCephでは使用できないことを明確にする価値があります（Nautilusバージョンからのみ表示されます）。

それでは、次の画像を想像してみてくださいHEALTH_WARN。OSDの1つの場所の終わりに起因するクラスターのステータスがあります。間違いはこれを証明しますHEALTH_WARN: 1 near full osd。以下は、この状況を克服するためのアルゴリズムです。

まず、残りのOSD間で利用可能なデータを配布する必要があります。最初のケースでは、ノットを「排出」したときに既に同様の操作を実行しましたが、わずかに減らす必要がある唯一の違いがありreweightます。たとえば、0.95より前：

ceph osd reweight osd.${ID} 0.95

これにより、OSDのディスク領域が解放され、cephヘルスのバグが修正されます。ただし、すでに述べたように、この問題は主に初期段階での不適切なCeph設定が原因で発生します。将来的に表示されないように再構成することが非常に重要です。

私たちの特定のケースでは、すべてが頼りになりました：

  • replication_countいずれかのプールで重要すぎる
  • 1つのプールに多すぎるPGがあり、別のプールでは小さすぎる。

すでに述べた計算機を使用します。入力する必要があるものを明確に示しており、原則として複雑なものはありません。必要なパラメーターを設定すると、次の推奨事項が得られます。

ケース番号3。LVMからCeph RBDへの仮想マシンの移行


プロジェクトがレンタルベアメタルサーバーにインストールされた仮想マシンを使用する状況では、フォールトトレラントストレージに関する問題がよく発生します。また、このストレージに十分なスペースがあることも非常に望ましい...別の一般的な状況：サーバーにローカルストレージを持つ仮想マシンがあり、サーバーに空きディスクスペースがないため、ディスクを拡張する必要があるが、どこにも拡張できない。

問題はさまざまな方法で解決できます。たとえば、別のサーバーに移行するか（存在する場合）、サーバーに新しいディスクを追加します。しかし、これを常に実行できるとは限らないため、LVMからCephへの移行は、この問題の優れた解決策になる可能性があります。このオプションを選択すると、ローカルストレージをハイパーバイザー間で移動する必要がないため、サーバー間の移行プロセスをさらに簡略化できます。唯一の問題は、作業中はVMを停止する必要があることです。

以下に示すレシピとして、このブログの記事が採用され、その手順が実際にテストされました。ちなみに、そこにはスムーズな移行方法も記載されています。しかし、私たちの場合、それは単に必要ではなかったので、チェックしませんでした。これがプロジェクトにとって重要な場合は、コメントで結果をお知らせいたします。

実用的な部分に取り掛かりましょう。この例では、virsh、したがってlibvirtを使用しています。最初に、データの移行先のCephプールがlibvirtに接続されていることを確認します。

virsh pool-dumpxml $ceph_pool

プールの説明には、承認データを含むCeph接続データが含まれている必要があります。

次のステップは、LVMイメージをCeph RBDに変換することです。ランタイムは主に画像のサイズに依存します。

qemu-img convert -p -O rbd /dev/main/$vm_image_name rbd:$ceph_pool/$vm_image_name

変換後もLVMイメージは残ります。これは、VMをRBDに移行できず、変更をロールバックする必要がある場合に役立ちます。また、変更をすばやくロールバックできるように、仮想マシンの構成ファイルをバックアップします。

virsh dumpxml $vm_name > $vm_name.xml
cp $vm_name.xml $vm_name_backup.xml

...元の（vm_name.xml）を編集します。ディスクの説明が付いたブロックを探し（行<disk type='file' device='disk'>で始まり、で終わります</disk>）、次の形式にします。

<disk type='network' device='disk'>
<driver name='qemu'/>
<auth username='libvirt'>
  <secret type='ceph' uuid='sec-ret-uu-id'/>
 </auth>
<source protocol='rbd' name='$ceph_pool/$vm_image_name>
  <host name='10.0.0.1' port='6789'/>
  <host name='10.0.0.2' port='6789'/>
</source>
<target dev='vda' bus='virtio'/> 
<alias name='virtio-disk0'/>
<address type='pci' domain='0x0000' bus='0x00' slot='0x04' function='0x0'/>
</disk>

詳細のいくつかを見てみましょう：

  1. プロトコルsourceは、Ceph RBDのストレージへのアドレスを示します（これは、CephプールとRBDイメージの名前を示すアドレスで、最初の段階で決定されました）。
  2. ブロックsecretは、タイプとceph、それに接続するためのシークレットのUUIDを示します。そのuuidはコマンドで見つけることができますvirsh secret-list。
  3. ブロックhostにはモニターへのアドレスCephが示されています。

構成ファイルを編集し、LVMからRBDへの変換を完了したら、変更した構成ファイルを適用して仮想マシンを起動できます。

virsh define $vm_name.xml
virsh start $vm_name

仮想マシンが正しく起動したことを確認します。たとえば、SSH経由または経由で仮想マシンに接続することで確認できますvirsh。

仮想マシンが正しく動作していて、他に問題が見つからなかった場合は、使用されなくなったLVMイメージを削除できます。

lvremove main/$vm_image_name
