痛みのないIPFS（ただし、正確ではありません）



HabréにはIPFSに関する記事はすでに1つもなかったという事実にもかかわらず。

私はこの分野の専門家ではないことをすぐに明らかにしますが、このテクノロジーに何度も興味を示してきましたが、それをいじろうとすると、しばしば痛みが生じました。今日、私は再び実験を行い、共有したいいくつかの結果を得ました。手短に言えば、IPFSといくつかのチップをインストールするプロセスについて説明します（すべてがubuntuで実行されましたが、他のプラットフォームでは実行しませんでした）。

IPFSをお見逃しの場合は、habr.com / en / post / 314768に詳細が記載されています。

取り付け


実験の純粋さのために、ローカルモードとリモートでの作業に関するいくつかの落とし穴を考慮するので、すぐにいくつかの外部サーバーにインストールすることをお勧めします。その後、必要に応じて、それは長い間取り壊されません、多くはありません。

行く


公式ドキュメント。golang.org/dlで
現在のバージョンを確認してください注：最も頻繁に使用するユーザーに代わってIPFSをインストールすることをお勧めします

。実際、以下ではFUSEを介してマウントするオプションを検討しますが、微妙な点があります。

cd ~
curl -O https://dl.google.com/go/go1.12.9.linux-amd64.tar.gz
tar xvf go1.12.9.linux-amd64.tar.gz
sudo chown -R root:root ./go
sudo mv go /usr/local
rm go1.12.9.linux-amd64.tar.gz

次に、環境を更新する必要があります（詳細はgolang.org/doc/code.html#GOPATH）。

echo 'export GOPATH=$HOME/work' >> ~/.bashrc
echo 'export PATH=$PATH:/usr/local/go/bin:$GOPATH/bin' >> ~/.bashrc
source ~/.bashrc

goがインストールされていることを確認する

go version

IPFSをインストールする


私はipfs-updateを介してインストールする方法が一番好きでした。

次のコマンドでインストールします

go get -v -u github.com/ipfs/ipfs-update

その後、次のコマンドを実行できます

。ipfs-update versions-ダウンロード可能なすべてのバージョンを確認します。
ipfs-update version-現在インストールされているバージョンを確認します（IPFSがインストールされるまで、それはありません）。
ipfs-update install latest -IPFSの最新バージョンをインストールします。それぞれ最新ではなく、利用可能なリストから目的のバージョンを指定できます。

ipfsをインストールする

ipfs-update install latest

小切手

ipfs --version

一般的な用語ですべてを直接インストールします。

次の2行に注意してください。

ここで、IPFSをローカルにインストールした場合は、ローカルアドレスでIPFSインターフェースにアクセスし、すべてが利用可能になります（たとえば、localhost：5001 / webui /）。ただし、外部サーバーにインストールされている場合、デフォルトでは、ゲートウェイはインターネットに対して閉じています。2つのゲートウェイがあります。

  1. ポート5001の管理Webui（github）。
  2. ポート8080の外部API（読み取り専用）。

これまでのところ、実験では両方のポート（5001と8080）を開くことができますが、バトルサーバーでは、ポート5001をファイアウォールで閉じる必要があります。4001ポートがあり、他のピアがあなたを見つけられるようにするために必要です。外部からのリクエストに対してはオープンにしておく必要があります。

〜/ .ipfs / configを開いて編集し、次の行を見つけます。

"Addresses": {
  "Swarm": [
    "/ip4/0.0.0.0/tcp/4001",
    "/ip6/::/tcp/4001"
  ],
  "Announce": [],
  "NoAnnounce": [],
  "API": "/ip4/127.0.0.1/tcp/5001",
  "Gateway": "/ip4/127.0.0.1/tcp/8080"
}

127.0.0.1をサーバーのIPに変更し、ファイルを保存してから、ipfsを再起動します（実行中のコマンドCtrl + Cを停止して、再度実行します）。

サーバーを操作するためのWebインターフェースをセットアップする


これが約3時間を費やした最初の落とし穴です。

外部サーバーにIPFSをインストールしたが、IPFSをローカルにインストールまたは開始しなかった場合、Webインターフェースで/ webuiにアクセスすると、接続エラーが表示されます



。実際、webuiは非常にあいまいに動作しているようです。まず、インターフェースが開いているサーバーのAPIに接続しようとします（もちろん、ブラウザーのアドレスに基づいています）。そこで機能しない場合は、ローカルゲートウェイへの接続を試みます。また、ローカルでIPFSを実行している場合、Webuiは正常に機能します。外部サーバーでwebuiを開いても、外部IPFSではなくローカルIPFSでのみ機能します。次に、ファイルをアップロードしますが、何らかの理由で、外部サーバーにそのように表示されません...

ローカルで実行されていない場合、接続エラーが発生します。私たちの場合、エラーはCORSが原因である可能性が高く、これはwebuiでも示され、構成の追加を提案しています。

ipfs config --json API.HTTPHeaders.Access-Control-Allow-Origin '["http://ip_ :5001", "http://127.0.0.1:5001", "https://webui.ipfs.io"]'
ipfs config --json API.HTTPHeaders.Access-Control-Allow-Methods '["PUT", "GET", "POST"]'

ワイルドカードを登録しました

ipfs config --json API.HTTPHeaders.Access-Control-Allow-Origin '["*"]'

追加されたヘッダーはすべて同じ〜/ .ipfs / configにあります。私の場合は

  "API": {
    "HTTPHeaders": {
      "Access-Control-Allow-Origin": [
        "*"
      ]
    }
  },

ipfsを再起動し、webuiが正常に接続されていることを確認します（上記のように、外部からの要求に対してゲートウェイを開いた場合は、接続が確立されているはずです）。

これで、Webインターフェイスを介してフォルダーとファイルを直接アップロードしたり、独自のフォルダーを作成したりできます。

FUSEファイルシステムをマウントする


これはかなり興味深いトリックです。

ファイル（およびフォルダー）を、Webインターフェースだけでなく、ターミナルに直接追加することもできます。たとえば、

ipfs add test -r
added QmfYuz2gegRZNkDUDVLNa5DXzKmxxxxxxxxxx test/test.txt
added QmbnzgRVAP4fL814h5mQttyqk1aURxxxxxxxxxxxx test

最後のハッシュはルートフォルダーのハッシュです。

このハッシュを使用して、任意のipfsノード（ノードを見つけてコンテンツを取得できます）でフォルダーを開くことができます。ポート5001または8080のWebインターフェースで、またはipfsを介してローカルで実行できます。

ipfs ls QmbnzgRVAP4fL814h5mQttyqk1aUxxxxxxxxxxxxx
QmfYuz2gegRZNkDUDVLNa5DXzKmKVxxxxxxxxxxxxxx 10 test.txt

ただし、通常のフォルダと同じように開くことができます。

ルートに2つのフォルダーを作成して、ユーザーに権限を与えましょう。

sudo mkdir /ipfs /ipns
sudo chown USERNAME /ipfs /ipns

--mountフラグを使用してipfsを再起動します

ipfs daemon --mount

他の場所にフォルダーを作成し、それらのパスをipfsデーモン--mount --mount-ipfs / ipfs_path --mount-ipns / ipns_pathオプションで指定できます

。このフォルダーからの読み取りは、通常とは異なります。

ls -la /ipfs
ls: reading directory '/ipfs': Operation not permitted
total 0

つまり、このフォルダのルートに直接アクセスすることはできません。しかし、ハッシュを知っていれば、コンテンツを取得できます。

ls -la /ipfs/QmbnzgRVAP4fL814h5mQttyqxxxxxxxxxxxxxxxxx
total 0
-r--r--r-- 1 root root 10 Aug 31 07:03 test.txt


cat /ipfs/QmbnzgRVAP4fL814h5mQttyqxxxxxxxxxxxxxxxxx/test.txt 
test
test

さらに、パスを指定すると、フォルダー内のオートコンプリートも機能します。

上で述べたように、そのようなマウントには微妙な点があります。デフォルトでは、マウントされたFUSEフォルダーには現在のユーザーしかアクセスできません（システムのほかのユーザーは言うまでもなく、rootもこのフォルダーから読み取ることはできません）。これらのフォルダーを他のユーザーが利用できるようにする場合は、構成で「FuseAllowOther」：falseを「FuseAllowOther」：trueに変更する必要があります。しかし、それだけではありません。ルートとしてIPFSを実行する場合は、すべて問題ありません。そして、（sudoであっても）通常のユーザーに代わってエラーが発生した場合

mount helper error: fusermount: option allow_other only allowed if 'user_allow_other' is set in /etc/fuse.conf

この場合、＃user_allow_otherの行のコメントを外して/etc/fuse.confを修正する必要があります。

その後、ipfsを再起動します。

FUSEの既知の問題


マウントでipfsを再起動した後（またはその他の場合）、マウントポイント/ ipfsおよび/ ipnsが使用できなくなるという問題が2回以上認識されました。それらへのアクセスはありませんが、ls -la / ipfsは????を表示します。権利のリスト。

そのような解決策を見つけました：

fusermount -z -u /ipfs
fusermount -z -u /ipns

次に、ipfsを再起動します。

サービスを追加


もちろん、ターミナルでの起動は一次テストにのみ適しています。戦闘モードでは、システムの起動時にデーモンが自動的に起動するはずです。

sudoの代わりに、ファイル/etc/systemd/system/ipfs.serviceを作成し、それに書き込みます。

[Unit]
Description=IPFS Daemon
After=syslog.target network.target remote-fs.target nss-lookup.target

[Service]
Type=simple
ExecStart=/home/USERNAME/work/bin/ipfs daemon --mount
User=USERNAME
Restart=always

[Install]
WantedBy=multi-user.target

もちろん、USERNAMEはユーザーに置き換える必要があります（そして、おそらくipfsプログラムへのフルパスはユーザーによって異なります（フルパスを指定する必要があります））。

サービスを開始します。

sudo systemctl enable ipfs.service

サービスを開始します。

sudo service ipfs start

サービスのステータスを確認します。

sudo service ipfs status

実験の純粋さのために、将来サーバーを再起動してipfsが正常に自動的に開始することを確認することが可能になります。

私たちが知っているごちそうを追加します


外部サーバーとローカルの両方にIPFSノードがインストールされている場合の状況を考慮してください。外部サーバーでは、なんらかのファイルを追加し、CIDによってローカルでIPFSを介して取得しようとします。何が起こるか？もちろん、ローカルサーバーはほとんどの場合、外部サーバーについて何も認識しておらず、利用可能なすべてのIPFSピアに「要求」することで、CIDによってファイルを見つけようとします（既に「知る」ことができています）。それらは、順番に、他の人に尋ねます。そして、ファイルが見つかるまで。実際、公式のipfs.ioゲートウェイを介してファイルを取得しようとすると、同じことが起こります。。運が良ければ、ファイルは数秒で見つかります。そして、そうでなければ、それは数分で見つかりません、それは仕事の快適さに大きく影響します。ただし、このファイルが最初に表示される場所はわかっています。それでは、なぜローカルサーバーに「最初に探す」とすぐに伝えないのでしょうか。どうやら、これは行うことができます。

1.リモートサーバーに移動し、設定で〜/ .ipfs / configを探します

"Identity": {
    "PeerID": "QmeCWX1DD7HnPSuMHZSh6tFuxxxxxxxxxxxxxxxx",

2. sudo service ipfs statusを実行し、Swarmエントリを探します。次に例を示します。

Swarm announcing /ip4/ip__/tcp/4001

5.すべて問題なければ、ローカルの構成を開きます〜/ .ipfs / config、その中で「Bootstrap」を見つけます：[...
そして、受け取ったアドレスを最初にアレイに追加します。

IPFSを再起動します。

次に、ファイルを外部サーバーに追加し、ローカルサーバーで要求してみます。速く飛ぶ必要があります。

しかし、この機能はまだ不安定です。私が理解している限り、Bootstrapでピアアドレスを指定しても、操作の過程でipfsはピアとのアクティブな接続のリストを変更します。いずれにしても、一定のお祭りを指示する可能性について、これと希望の議論が行われ、ここで、できるように思われるはずipfs@5.0+にいくつかの機能を追加します。

現在のピアのリストは、webuiとターミナルの両方で表示できます。

ipfs swarm peers

あちこちで、ごちそうを手動で追加できます。 

ipfs swarm connect "/ip4/ip__/tcp/4001/ipfs/$PeerID"

この機能を改善するまでは、目的のピアとの接続を確認するツールを作成し、そうでない場合は接続を追加できます。

推論


すでにIPFSに精通している人の中には、IPFSに対する賛成論と反対論があります。原則として、昨日の話し合いの前日に、私は再びIPFSを掘り起こさなければなりませんでした。そして、上記の議論に関して、私は上記の議論のいくつかに強く反対するとは言えません（私は1人半のプログラマがIPFSを使用しているという事実だけに同意しません）。一般に、どちらも独自の方法で正しいです（特にチェックに関するコメント）。提供すると思います）。しかし、あなたが道徳的および法的評価を放棄した場合、この技術の技術的評価を与えるのは誰ですか？個人的には、「絶対にあるべきだと思います」というある種の内面的な感覚があります。しかし、正確には、明確な表現がないのはなぜですか。既存の集中型ツールを見ると、多くの点で非常に優れています（作業の安定性、作業速度、制御性など）。それにもかかわらず、私には1つの考えがあります。それは理にかなっているように思われ、そのような分散システムなしではほとんど実装できません。もちろん、私は本当に心を揺さぶりますが、私は次のようにそれを定式化します。インターネット上で情報を広めるという原則を変える必要があります。

説明します。よろしければ、「譲渡先の方が守り、意図されていない方に紛失・受信されないように」を原則に情報発信いたします。たとえば、さまざまなメールサービスやクラウドストレージなどを簡単に検討できます。そして、最終的に私たちは何を持っていますか？ HabréHubでは、情報セキュリティが最前線にあり、ほぼ毎日、別のグローバルリークに関するニュースを受け取っています。原則として、すべての最も興味深いものは、<irony> wonderful </ ironium>の記事に記載されています夏はもう終わりです。データ漏洩はほとんどありません。つまり、主要なインターネットの巨人はますます大きくなり、それらはますます多くの情報を蓄積しており、そのようなリークは一種の情報的な原子爆発です。これは決して起こらなかったし、ここでも。同時に、多くの人はリスクがあることを理解していますが、サードパーティ企業とのデータを信頼し続けます。第一に、特別な選択肢はありません。第二に、彼らはすべての穴にパッチを適用し、これが二度と起こらないことを約束します。

どのオプションが表示されますか？データは最初はオープンに配布されるべきだと私には思われます。ただし、この場合の開放性は、すべてが読みやすいことを意味するものではありません。保管と配布の開放性について話しているのですが、読書の全体的な開放性についてではありません。情報は公開鍵で配布する必要があると思います。結局のところ、公開鍵/秘密鍵の原則は、インターネットのようにすでに古くなっています。情報が機密情報ではなく、広範囲に設計されている場合は、公開鍵を使用して即座にレイアウトされます（ただし、暗号化された形式であり、誰でも既存の鍵で解読できます）。そうでない場合は、公開鍵なしでレイアウトされ、鍵自体がこの情報にアクセスする必要があるものに送信されます。同時に、それを読むべき人には鍵だけが必要であり、この情報をどこから入手できるのか、それは急上昇すべきではありません-それはネットワークからそれを引っ張るだけです（これはコンテンツではなく、アドレスによる配信の新しい原則です）。
