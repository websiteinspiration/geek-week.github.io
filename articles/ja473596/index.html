Python v3.x：コルーチンおよび同期関数の例外ハンドラー。一般的に、すべての

私の自由な時間に私は私の小さなプロジェクトに取り組みます。Python v3.x + SQLAlchemyで記述されています。多分いつかそれについて書くつもりですが、今日は例外を処理するための私のデコレータについて話したいです。関数とメソッドの両方に使用できます。同期および非同期。カスタム例外ハンドラを接続することもできます。

現在、デコレータは次のようになっています。
import asyncio

from asyncio import QueueEmpty, QueueFull
from concurrent.futures import TimeoutError


class ProcessException(object):

    __slots__ = ('func', 'custom_handlers', 'exclude')

    def __init__(self, custom_handlers=None):
        self.func = None
        self.custom_handlers: dict = custom_handlers
        self.exclude = [QueueEmpty, QueueFull, TimeoutError]

    def __call__(self, func, *a):
        self.func = func

        def wrapper(*args, **kwargs):
            if self.custom_handlers:
                if isinstance(self.custom_handlers, property):
                    self.custom_handlers = self.custom_handlers.__get__(self, self.__class__)

            if asyncio.iscoroutinefunction(self.func):
                return self._coroutine_exception_handler(*args, **kwargs)
            else:
                return self._sync_exception_handler(*args, **kwargs)

        return wrapper

    async def _coroutine_exception_handler(self, *args, **kwargs):
        try:
            return await self.func(*args, **kwargs)
        except Exception as e:
            if self.custom_handlers and e.__class__ in self.custom_handlers:
                return self.custom_handlers[e.__class__]()

            if e.__class__ not in self.exclude:
                raise e

    def _sync_exception_handler(self, *args, **kwargs):
        try:
            return self.func(*args, **kwargs)
        except Exception as e:
            if self.custom_handlers and e.__class__ in self.custom_handlers:
                return self.custom_handlers[e.__class__]()

            if e.__class__ not in self.exclude:
                raise e

順に分析していきます。__slots__少しメモリを節約するために使用します。オブジェクトが頻繁に頻繁に使用される場合に役立ちます。

__init__の初期化段階で、custom_handlersを保存します（それらを転送する必要がある場合）。念のため、彼はそこに辞書が表示されることを期待していると述べましたが、おそらく将来的には、いくつかのハードチェックを追加することは理にかなっています。self.excludeプロパティには、処理する必要のない例外のリストが含まれています。このような例外が発生した場合、デコレータを含む関数はNoneを返します。現時点では、私のプロジェクトのためにリストがシャープになっています。おそらく、別の構成に含めることは理にかなっています。

最も重要なことは__call__で起こります。したがって、デコレータを使用するときは、それを呼び出す必要があります。パラメータなしでも：

@ProcessException()
def some_function(*args):
    return None

それら。これは間違っており、エラーが発生します：

@ProcessException
def some_function(*args):
    return None

この場合、現在の関数を取得します。この関数は、非同期の程度に応じて、通常の同期またはコルーチンとして処理します。

読みやすさを向上させるために 2つある場合（そうです、コードはサディスティックな傾向のある人がサポートできます）、そして決定した場合に備えてself.custom_handlers .__ get __（self、self .__ class__）を実行@propertyクラスに格納されたハンドラ。

たとえば、次のようになります。

class Math(object):
    @property
    def exception_handlers(self):
        return {
            ZeroDivisionError: lambda: '   ,   '
        }
    
    @ProcessException(exception_handlers)
    def divide(self, a, b):
        return a // b

self.custom_handlers .__ get __（...）を実行しない場合、@ propertyのコンテンツの代わりに、<property object at 0x7f78d844f9b0>のようなものが取得されます。

実際、上の例はカスタムハンドラーを接続する方法を示しています。通常、これは次のように行われます。

@ProcessException({ZeroDivisionError: lambda: '   ,   '})
def divide(a, b):
    return a // b

クラスの場合（プロパティ/メソッドを渡す場合）、クラスデコレータを初期化する段階では、そのようなメソッドはなく、プロパティ/メソッドは単純な関数であることを考慮する必要があります。したがって、私たちは上で発表されたものだけを伝えることができます。したがって、@ propertyオプションは、コードの下位にあるすべての関数を自分自身で使用する機能です。まあ、自分が必要なければラムダを使うこともできます。

非同期コードの場合、上記の例はすべて当てはまります。

最後に、そのパスの例外がカスタムハンドラーに適合しなかった場合は、さらに発生するという事実に注意を向けたいと思います。
