NoVerify：高速に動作するPHPリンター

PHPには、静的分析ユーティリティとしてPHPStan、Psalm、Phan、Exakatがあります。リンターはうまく機能しますが、ほとんどすべてがPHP（またはJava）で書かれているため、非常にゆっくりです。個人的な使用や小さなプロジェクトではこれは正常ですが、何百万ものユーザーがいるサイトでは、これは重要な要素です。リンターが遅いと、CIパイプラインが遅くなり、テキストエディターまたはIDEに統合できるソリューションとして使用できなくなります。



数百万のユーザーがいるサイトはVKontakteです。新しい機能の開発と追加、バグのテストと修正、レビュー-これらすべては、厳しい納期の条件の下で、迅速に進む必要があります。したがって、5〜10秒で500万行のコードベースをチェックできる高速で高速なリンターは、かけがえのないものです。 

市場には適切なリンターがないため、Yuri Nasretdinov（youROCK）VKontakteから開発チームへの彼の助けを書きました-NoVerify。これはGoで書かれたPHPのリンターです。これは、対応するものよりも10〜30倍速く動作し、PhpStormが警告しない何かを見つけ、簡単に拡張して、静的分析について聞いたことがないプロジェクトにうまく統合できます。 イスカンデル・シャリポフ

がこのリンターについて話します。カットの下で：彼らがリンターをどのように選択し、独自に記述することを好んだか、NoVerifyがなぜ高速であり、内部からどのように配置されているか、なぜそれがGoで記述されているか、それが何を見つけ、どのように拡張するか、どのように妥協する必要があり、何に基づいて構築できるか。


イスカンデル・シャリポフ（準者）VKontakteのバックエンドインフラストラクチャで動作し、NoVerifyに精通しています。以前は、IntelチームのGoコンパイラーに従事していました。彼はPHPで記述していませんが、これは静的分析で彼が好む言語です-うまくいかないことがたくさんあります。

NoVerify技術詳細


プライベートチェックVKontakte。 Noverifyは次のように書かれています。 GitHubでは、NoVerifyリポジトリは2つの部分に分かれています。リンターの実装に使用されるフレームワークと、個別のチェックであるvklintsです。これは、リンターがサードパーティのチェックをロードするように行われます。Goで別のモジュールを作成して、フレームワークに登録することができます。 NoVerifyバイナリから開始すると、フレームワークは登録されているすべてのチェックセットをロードし、全体として機能します。 



NoVerifyは、ライブラリとバイナリ（リンター）の両方です。

チェックはvklintsと呼ばれます。彼らは、PhpStormとオープンソースのNoVerify-一般的な使用に適さない重要なエラーが表示されないことを発見しました。

vklintsとは何ですか？

特定の関数、クラス、さらには私たちの慣習に従わないグローバル変数の使用の詳細をチェックします。これは、スタイルガイドに記載されているさまざまな理由により、特別な場所では使用できないものです。

追加のスタイルチェック。それらは、PHPコミュニティーで受け入れられているものに対応しておらず、PHP標準勧告に記載されておらず、それと矛盾さえしていますが、私たちにとっては標準です。それらをフォローしたくないので、オープンソースに追加しても意味がありません。

一部のタイプの厳密な比較要件。たとえば、文字列を比較演算子で比較する必要があるチェックがあります===。特に、文字列を比較するには、関数を厳密に比較するためのフラグを渡す必要があります。

疑わしい配列キー。もう1つの興味深い間違い：開発者がコミットするときに、ファイルを保存する前にキーの組み合わせを押すことがあります。これらの文字は、文字列またはコードの一部に残る場合があります。かつて、配列のキーにはロシア語の文字「Y」がありました。ほとんどの場合、開発者はロシア語のレイアウトでCTRL-Sを押し、ファイルを保存してコミットしました。このようなキーが配列で見つかることがありますが、新しいエラーは発生しなくなります。

動的ルールは、PHPで説明されているより単純なNoVerify拡張メカニズムです。これについては別の記事が書かれています。Goコードを1行も書かずにNoVerifyにチェックを追加する方法です。

NoVerifyの仕組み


PHPを解析するには、パーサーが必要です。PHPでPHPパーサーを使用することはできません。遅いため、GoからはCのラッパーを介してのみ使用できます。したがって、Goでパーサーを使用します。

このパーサーにはいくつかの問題があります。残念ながら、彼はUTF-8でしか機能しないため、UTF-8 ではなくUTF-8を区別する必要があります。UTF-8に加えて、Windows-1251はロシアのPHPプロジェクトでよく見られます。そのようなファイルもあります。それらをどのように認識しますか？ 

このファイルにencodings.xmlは、UTF-8のファイルがあるすべてのパスがリストされています。これらのパスの外にあるファイルに遭遇した場合、その場で（事前に変換せずに）ストリームをストリーミングすることでUTF-8にストリーミングします。


解析と分析


数ステップで完了します。最初に、phpstorm-stubsからメタデータをロードします。これはPHPコードのように見えますが、実行されることはなく、標準関数からの入力/出力のタイプを表すデータです。phpStormメタデータには、リンター用の便利なオーバーライドディレクティブがあります。たとえば、型の配列を受け入れて型T[]を返すことを記述できます（関数に役立ちますarray_pop）。


Phpstorm-stubsが最初にロードされます。メタデータを最初のタイプ情報として使用します-基盤。この基礎はリンターに吸収され、ソースの分析を開始します。吸収前に

現在のマスターをロードします。2つのモードでコードをチェックします。

  • ローカルの変更：ベースラインに関して、コードに新しいエラーが見つかりました。
  • リビジョンの範囲を示します。最初と最後のリビジョン、そしてそれらの間にはすべてが含まれています-これは新しいコードであり、「以前」のものがすべて古いものです。

次は分析段階です。



AST分析。これでメタデータ、型情報ができました。すべてのPHPソース、parsimを取得し、ASTの真上で分析します。現時点では中間表現はありません。生のASTを分析することは、特にそれが表すライブラリとデータ型に依存している場合、あまり便利ではありません。 



分析結果はキャッシュに保存されます。これは再分析で使用され、はるかに高速です。

レポートとフィルタリング。次に、レポートまたは警告を 2回生成します。最初に、コードの古いバージョン（ベースラインの前）の警告を見つけ、次に新しいバージョンの警告を見つけます。レポートは比較（diff）によってフィルタリングされます-新しいバージョンのコードに表示された警告を探し、ユーザーに渡します。一部の静的アナライザでは、これは「ベースラインモード」と呼ばれます。



二重コード分析（差分モード）は非常に低速です。しかし、私たちはそれを買う余裕があります-NoVerifyは他のPHPリンカーよりも何十倍も高速です。同時に、少なくとも30％の追加の加速のための予備があります。

ファイルを分析するにはどうすればよいですか？PHPでは、関数を定義する前に呼び出すことができます。分析する前に、この関数に関する情報を知っておく必要があります。したがって、まずASTでファイル全体を調べ、インデックスを作成し、すべての関数のタイプを識別し、クラスを登録してから、それを分析します。 



分析は、ファイルの2番目のパスです。ほとんどのインタプリタとコンパイラは、2パス以上でも機能します。ファイルを2回「スキャン」しないようにするには、たとえばCのように、使用前に宣言を行う必要があります。

型推論


最も興味深い部分は、エラーが最も頻繁に発生することです。それはまだ正式に定義することが難しいPHP型システムの正確さに対応していません。

モデルはどのように見えますか。


セマンティックモデル（デモ）。

タイプのタイプ：

  • 期待されるのは、コメントで説明することです。プログラムにはいくつかの型が期待されますが、これは、それらが実際に使用されていることを意味するものではありません。
  • 実際 -プログラムにある実際のもの。たとえば、数値を何かに割り当てた場合、intまたはfloat（これが浮動小数点数の場合）が実際の型になることは明らかです。 

実際のタイプは「より強い」ように見えます-それらは本当の、本当です。しかし、注釈だけで型を取得できる場合もあります。

注釈（期待されるタイプ）は、信頼と不信の 2つのカテゴリに分類できます。たとえば、phpstorm-stubsは最初のカテゴリに属します。それらを使用する前に、（エラーなしで）モデレート済みと見なされます。エラーがある可能性があるため、信頼できないものは他の開発者が作成したものです。

実際の型は、値、アサーション、述語、PHP 7の機能を拡張する型ヒントなど、いくつかの部分に分けることもできますが、型ヒントでは解決できない問題があります。

// actual = Foo
// expected = static

書くときnew static()、クラスだけが返すことができるわけではありませんnew Foo。たとえば、Fooクラスがから継承されているbar場合、が存在する可能性がありますnew bar。したがって、少なくとも2つのタイプ情報が必要です。それらのどれも冗長ではありません-両方が必要です。

したがって、selfPHPインタープリターの場合、ここでの実際のタイプは-になります。しかし、IDEとリンターが機能するには、が必要staticです。継承クラスのコンテキストからこのコードを呼び出す場合、これが同じ基本クラスではなく、メソッドが多いという情報を知る必要があります。

class Foo {
    /** @return static */
    public function newStatic() : self {

    }
}

タイプヒント


静的型付けと型ヒントは同じものではありません。
機能の境界でしかチェックできないと聞いたことがあるかもしれません。境界で、入力と出力をチェックします。入力は関数の引数です。関数内では、どんな意味のないこともできます。foo値を割り当てますがint、それが何であるかを説明しましたT。宣言した型に違反していると文句を言うことができますが、PHPの場合はエラーは発生しません。 

declare(strict_types=1);
    function f(T $foo) {
        $foo = 10; //  int
        return $foo;
}

例はもっと難しいです-私たちは戻りfooますか？関数の開始時に、fooそれはTであると判断しました。戻りに関する情報はありません。 

declare(strict_types=1);
function f(T $foo) {
    $foo = 10; //  int
    return $foo;
}
// ? 1. f -> int
// ? 2. f -> T|int
// ? 3. f -> T

どのタイプが正しいですか？最初の2つでは、それらの違いを分析します。PhpStormとlinterは2番目のオプションを出力します。常に返されるという事実にもかかわらず、int型T|intは推定されます- 型の「結合」。これは、これらの値の両方を割り当てることができるタイプです。最初にタイプTに関する情報があり、次にを割り当てた10ので、変数のタイプはfooこれら2つのタイプと互換性がある必要があります。

注釈


コメントや注釈が存在する可能性があります。
以下の例では、数値を返すが文字列を返すことを記述しました。リンターが注釈とタイプヒントのレベルでのみ機能する場合は、常にを返すと見なしますint。しかし、Actual型はこれを回避するのに役立ちます。ここでは、Expected型はthis intであり、Actual型は文字列です。リンターは文字列が返されることを知っており、戻ると約束したことを警告するかもしれませんint。この分離は私たちにとって重要です。

/** @return int */
function f() { return "I lied!"; }

アノテーションの継承。ここで、ある種のインターフェースを実装するクラスにはメソッドがあるということです。メソッドには適切なコメント、ドキュメント、タイプがあります-インターフェースを実装するために必要です。ただし、実装にはコメントはあり@inheritdocません。まったくまたはまったくありません。

interface IFoo {
    /** @return int */
    public function foo();
}
class Fooer implements IFoo {
    /** @inheritdoc */
    public function foo() { return "10"; }
}

このメソッドは何を返しますか？ -インターフェースで記述されているものが返されるようですintが、実際には文字列です。これは良くありません。PHPはすべて同じですが、収束は重要です。

このコードを修正するには2つのオプションがあります。明らかなのは戻ることintです。しかし、おそらく別のタイプを返す必要があります。何をすべきか？文字列を返すことを書きます。この場合、コードを正しく分析するために、IDEとリンターの両方に明示的な型情報が必要です。

interface IFoo {
    /** @return int */
    public function foo();
}
class Fooer implements IFoo {
    /** @return string */
    public function foo() { return "10"; }
}

この情報は、人々がコメントを書いた場合、まったく必要ありません@inheritdoc。PhpStormがあなたのタイプを理解する必要はありません。ただし、型が正しく記述されていないと、問題が発生します。

PhpStormとlinterには、メタデータ（タイプ）に同じファイルを使用すると、一連のバラバラなバグがあります。JetBrainsリポジトリのphpstorm-stubsで必要なすべてを修正すると、IDEが壊れる可能性が高くなります。デフォルトのままにしておくと、すべてが正しく

機能するわけではないため、小さなフォーク  -VKCOM / phpstorm-stubsがあります。適合しないものを修正するために、いくつかのパッチが追加されました。PhpStormにはお勧めできませんが、リンターが機能するためには必要です。
