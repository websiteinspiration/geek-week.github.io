原因


そして、これについて話し合ったので、ネットワーク仮想化の前提条件について言及する価値があります。実際、このプロセスは昨日始まったわけではありません。

ネットワークは常にシステムの最も不活性な部分であると何度も聞いたことがあるでしょう。そして、これはあらゆる意味で真実です。ネットワークはすべての基盤となるものであり、ネットワークに変更を加えることは非常に困難です。ネットワークが存在する場合、サービスは許容されません。多くの場合、単一のノードを廃止すると、ほとんどのアプリケーションが追加され、多くのクライアントに影響を与える可能性があります。これがネットワークチームが変更に抵抗できる理由の1つです- どうやったらうまくいくのか（どうすればよいのかさえわからないため）ですが、ここでは新しいものを構成する必要があり、それがネットワークにどのように影響するかはわかっていません。

ネットワークプロバイダーがVLANを実行するのを待たず、各ネットワークノードでサービスを登録しないために、GRE、IPinIP、MPLS、MPLS L2 / L3VPN、VXLAN、GENEVE、MPLSoverUDP、MPLSoverGREなどのさまざまなオーバーレイ（重ね合わせたネットワーク）を使用することにしました。

彼らの魅力は、2つの単純なことにあります。

  • 構成されているのはエンドノードだけです-トランジットノードに触れる必要はありません。これによりプロセスが大幅にスピードアップし、新しいサービスを導入するプロセスからネットワークインフラストラクチャ部門を除外できる場合もあります。
  • 負荷はヘッダーの奥深くに隠されています-トランジットノードはそれについて、ホストでのアドレス指定、課されたネットワークのルートについて何も知る必要はありません。これは、テーブルに格納する必要がある情報が少ないことを意味するため、よりシンプルで安価なデバイスを使用してください。

この完全ではない問題では、考えられるすべてのテクノロジーを分析するつもりはありませんが、DCでのオーバーレイネットワークの動作のフレームワークについて説明します。

シリーズ全体では、同じサーバー機器が設置されている類似したラックの列で構成されるデータセンターについて説明します。

この機器は、サービスを実装する仮想マシン/コンテナ/サーバーレスを実行します。

アンダーレイは物理ネットワークです：ハードウェアスイッチとケーブル。アンダーレイのデバイスは、物理マシンに到達する方法を知っています。



標準のプロトコルとテクノロジーに依存しています。特に、ハードウェアデバイスは、チッププログラミングまたはプロトコルの実装をそれぞれ許可しない独自のソフトウェアで動作するため、他のベンダーとの互換性と標準化が必要です。
しかし、Googleのような誰かが独自のスイッチを開発して、一般に受け入れられているプロトコルを放棄する余裕があります。しかし、LAN_DCはGoogleではありません。
アンダーレイは、物理マシン間の基本的なIP接続であるため、変更されることはほとんどありません。アンダーレイは、その上で実行されているサービス、クライアント、テナントについて何も認識していません。あるマシンから別のマシンにパッケージを配信する必要があるだけです。
アンダーレイは次のようになります。

  • IPv4 + OSPF
  • IPv6 + ISIS + BGP + L3VPN
  • L2 + TRILL
  • L2 + STP

アンダーレイネットワークは、CLI / GUI / NETCONFという古典的な方法で構成されます。

オーバーレイ

クライアントデータは、共有ネットワークを介して送信するためのトンネリングヘッダーにカプセル化されます。

ただし、この場合、ToRスイッチはそれぞれ異なるサービスを共有できる必要があり、ネットワーク管理者はある程度、仮想マシンの管理者と協力して、デバイスの構成を（自動的に）変更する必要があります。

ただし、ToRでオーバーレイが開始されるVXLANファクトリは、十分に確立されたオーバーレイネットワーク設計の1つです。

ホストからのオーバーレイ

次に、この場合のToRスイッチは、コントロールプレーンとデータプレーンの両方の観点から、可能な限り単純なままにすることができます。実際、彼はSDNコントローラーと通信したり、接続されているすべてのクライアントのネットワークやARPを保存したりする必要はありません。また、物理マシンのIPアドレスを知っているだけなので、スイッチング/ルーティングテーブルが大幅に簡素化されます。

ケーススタディタングステン生地


各物理マシンには、vRouterが接続されており、それに接続されているネットワークとそれらがどのクライアントに属しているかを認識する仮想ルーターがあり、実際はPEルーターです。クライアントごとに、独立したルーティングテーブルを維持します（VRFを読み取ります）。そして実際にvRouterはオーバーレイトンネリングを行います。

vRouterについてもう少し詳しくは、記事の最後にあります。

ハイパーバイザーにある各VMは、TAPインターフェイスを介してこのマシンのvRouterに接続されます。

TAP-ターミナルアクセスポイント-ネットワークを可能にするLinuxカーネルの仮想インターフェイス。

1つのクライアントのすべてのネットワークは、1つのVRF（1つのテーブル）に配置されます。

VM0がVM2にパケットを送信しようとしています。今のところ、これは単一のクライアントVMであると想定します。

データプレーン


  1. VM-0のeth0インターフェイスにはデフォルトルートがあります。パケットはそこで送信されます。
    このeth0インターフェイスは、実際には、tap0 TAPインターフェイスを介してvRouter仮想ルーターに仮想的に接続されています。
  2. vRouterは、パケットが到達したインターフェイス、つまりパケットが属するクライアント（VRF）を分析し、このクライアントのルーティングテーブルで受信者アドレスを確認します。
  3. 同じマシンのレシーバーが別のポートの背後にあることを発見すると、vRouterは追加のヘッダーなしでパケットをそれに送信するだけです。この場合、vRouterにはすでにARPレコードがあります。



この場合のパケットは物理ネットワークに入りません-vRouter内でルーティングされます。

ゲートウェイは片足で、通常の仮想マシンのようにオーバーレイ仮想ネットワークを調べ、他のすべてのVMと対話できます。さらに、すべてのクライアントのネットワーク自体を終端することができるため、クライアント間のルーティングを実行できます。

もう一方の足で、ゲートウェイはすでにバックボーンネットワークを調べており、インターネットにアクセスする方法を知っています。




データプレーン


つまり、プロセスは次のようになります。

  1. デフォルトですべて同じvRouterにあるVM-0は、外界の宛先（185.147.83.177）を含むパケットをeth0インターフェイスに送信します。
  2. vRouterはこのパケットを受信し、ルーティングテーブルで宛先アドレスを検索します-トンネル1
    を介してVNGW1ゲートウェイを経由するデフォルトルートを見つけます。また、これはSIP 10.0.0.2およびDIP 10.0.255.2のGREトンネルであり、MPLSを最初に切断する必要があることも確認します- VNGW1が期待するこのクライアントのラベル。
  3. vRouterは、元のパケットをMPLS、GRE、および新しいIPヘッダーにパックし、デフォルトでToR1 10.0.0.1に送信します。
  4. アンダーレイネットワークは、パケットをVNGW1ゲートウェイに配信します。
  5. VNGW1ゲートウェイは、GREおよびMPLSトンネリングヘッダーを削除し、宛先アドレスを確認し、そのルーティングテーブルを調べて、インターネットに向けられていることを理解します。つまり、フルビューまたはデフォルトを介して行われます。必要に応じて、NAT変換を実行します。
  6. VNGWから国境に至るまで、通常のIPネットワークが存在する可能性がありますが、これはほとんどありません。
    これは、クラシックMPLSネットワーク（IGP + LDP / RSVP TE）にすることも、BGP LUで工場に戻すことも、VNGWからIPネットワークを介して境界までGREトンネルにすることもできます。
    いずれにせよ、VNGW1は必要なカプセル化を実行し、元のパケットを境界に向けて送信します。

Control Plane


VNGW1は、SDNコントローラーを使用してBGP近隣を確立し、そこからクライアントに関するすべてのルーティング情報を受信します。どのIPアドレス（vRouter'om）がどのクライアントであり、どのMPLSラベルがそれを識別するかです。

同様に、彼自身がこのクライアントのラベルが付いたデフォルトルートをSDNコントローラーに通知し、ネクストホップであることを示します。そして、このデフォルトがvRouter'yになります。

VNGWは通常、ルートまたはNATを集約します。

そして反対の方向に、彼はボーダーまたはルートリフレクターとのセッションにこの集約されたルートを正確に与えます。そして、それらからデフォルトルートまたはフルビュー、または何か他のものを受け取ります。

カプセル化とトラフィック交換の点では、VNGWはvRouterと同じです。
エリアを少し拡大すると、ファイアウォール、トラフィックの精製または強化ファーム、IPSなど、他のネットワークデバイスをVNGWおよびvRouterに追加できます。

また、VRFを連続して作成し、ルートを正しくアナウンスすることで、サービスチェーンと呼ばれるトラフィックループを希望どおりにループさせることができます。

つまり、SDNコントローラーは、VNGW、vRouter'ami、その他のネットワークデバイス間のルートリフレクターとして機能します。

ただし、実際には、コントローラーはACLとPBR（ポリシーベースルーティング）に関する情報も解放し、個々のトラフィックフローがルートの指示とは異なる方法で強制的に送信されるようにします。





よくある質問



なぜいつもGRE / UDPの発言をしているのですか？

まあ、一般的に、それはタングステン生地に固有であると言うことができます-あなたはそれを完全に無視することができます。

しかし、それを採用すると、OpenContrailでありながら、TF自体が両方のカプセル化をサポートしました。GREのMPLSとUDPのMPLSです。

ヘッダーのソースポートでは、元のIP +プロト+ポートからハッシュ関数をコーディングするのが非常に簡単で、バランスをとることができるため、UDPは優れています。

悲しいかな、GREの場合、カプセル化されたすべてのトラフィックで同じ外部IPヘッダーとGREヘッダーしかなく、バランスについての話はありません。パケットの奥深くまで見える人はほとんどいません。

しばらくの間、ルーターは、MPLSoGREでのみ、そしてごく最近になってMPLSoUDPで学習した動的トンネルの使用方法を知っていた場合。したがって、2つの異なるカプセル化の可能性について常にコメントする必要があります。

公平に言えば、TFがVXLANを使用したL2接続を完全にサポートしていることは注目に値します。


あなたはOpenFlowとの類似点を描くと約束しました。
彼らは本当に懇願する。同じOpenStackのvSwitchは、VXLANを使用して非常によく似た動作をしますが、それには、UDPヘッダーもあります。

データプレーンでは機能はほぼ同じですが、コントロールプレーンは大幅に異なります。 Tungsten FabricはXMPPを使用してルート情報をvRouterに配信しますが、OpenflowはOpenStackで動作します。


vRouterについてもう少し詳しく教えてください。
vRouter AgentとvRouter Forwarderの2つの部分に分かれています。

1つはホストOSのユーザースペースで実行され、SDNコントローラーと通信して、ルート、VRF、ACLに関する情報を交換します。

2つ目はデータプレーンを実装します-通常はカーネルスペースですが、SmartNICで起動することもできます-CPUと個別のプログラム可能なスイッチングチップを備えたネットワークカード。

vRouterがユーザースペースのDPDKアプリケーションである場合、別のシナリオが考えられます。

vRouter AgentはvRouter Forwarderの設定を削除します。


仮想ネットワークとは何ですか？
VRFに関する記事の冒頭で、すべてのテナントがVRFに関連付けられていることを述べました。そして、これがオーバーレイネットワークの動作を表面的に理解するのに十分である場合、すでに次のイテレーションで、明確にする必要があります。

通常、仮想化メカニズムでは、仮想ネットワークエンティティ（適切な名前と見なすことができます）は、クライアント/テナント/仮想マシンとは別に導入されます-これは非常に独立したものです。そして、インターフェースを介したこの仮想ネットワークは、すでに1つのテナント、別のテナント、2つのテナントに接続できますが、少なくともどこにでも接続できます。したがって、たとえば、サービスチェーンが実装されています。トラフィックを特定のノードを介して目的の順序で通過させる必要がある場合、仮想ネットワークを正しい順序で作成して接続するだけです。

したがって、仮想ネットワークとテナントの間には直接の対応関係はありません。



結論


これは、ホストとSDNコントローラーからのオーバーレイを使用した仮想ネットワークの操作の表面的な説明です。しかし、現在どの仮想化プラットフォームを使用していても、VMWare、ACI、OpenStack、CloudStack、Tungsten Fabric、Juniper Contrailなど、同じように機能します。カプセル化とヘッダーの種類、エンドネットワークデバイスに情報を配信するためのプロトコルは異なりますが、比較的単純で静的なアンダーレイネットワーク上で動作するソフトウェア調整オーバーレイネットワークの原則は変わりません。
これまでにプライベートクラウドを作成する分野では、オーバーレイネットワークに基づくSDNが優勝したと言えます。ただし、これはOpenflowが現代の世界で場所がないことを意味するものではありません。OpenStackeと同じVMWare NSXで使用されています。私の知る限り、Googleはこれを使用してアンダーレイネットワークを構成しています。
