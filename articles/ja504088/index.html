高度なWindowsロギング。mimikatzを探しています


みなさん、こんにちは。今日は、攻撃者がWindows Defenderをバイパスできたが失敗した例（セキュリティガード）について見ていきます。はい、これは再びmimikatzについてです。mimikatzを起動してWindows Defenderをバイパスする方法については、こちらをご覧ください。そして今日、私が約束したように、「青い」チームのために何かを考えてください。それが少なくとも誰かを助けるなら、それは無駄ではありません。じゃ、行こう。

ELKスタック（Elasticsearch、Logstash、Kibana）（Habréを含む）について多くのことが書かれていますが、今日はそれを使用します。正確には、ELKを脅威ハンティングのニーズに合わせてドープします。

狩猟ELK、またはHELK


HELKダイアグラムは次のようになります：



インストールに進みましょう。

HELKが配置されるサーバーとして、私は開発者の助言として選択しました：

tomhunter@helk:~$ cat /etc/lsb-release 
DISTRIB_ID=Ubuntu
DISTRIB_RELEASE=18.04
DISTRIB_CODENAME=bionic
DISTRIB_DESCRIPTION="Ubuntu 18.04.4 LTS"

インストールは簡単で、2つのコマンドで実行されます。

git clone https://github.com/Cyb3rWard0g/HELK
tomhunter@helk:~/HELK/docker$ sudo ./helk_install.sh

注：インストール用により多くのRAMを割り当てるように求められる場合があります。10 GBを使用します。

提案されたインストールオプションから、KAFKA + KSQL + ELK + NGNIX + ELASTALERTを選択しました。そして今、ただ待っています。そして、すべてを正しく行うと、大切なことがわかり



ます。サーバーが完成したら、クライアントを準備しましょう。これはWindows 10マシンになるので、Sysmonをインストールする必要があります。

設定ファイルも必要です：

 git clone https://github.com/olafhartong/sysmon-modular

構成を生成します。

 . .\Merge-SysmonXml.ps1
Merge-AllSysmonXml -Path ( Get-ChildItem '[0-9]*\*.xml') -AsString | Out-File sysmonconfig.xml

実行：

 PS C:\Users\Tomhunter\Documents\sysmon-modular> .\Sysmon64.exe -i .\sysmonconfig.xml

また、インストールされますwinlogbeatをコピーすることにより、winlogbeat.ymlファイルを削除し、新しいものを作成raw.githubusercontent.com/Cyb3rWard0g/HELK/master/configs/winlogbeat/winlogbeat.ymlをそれに


注：ここでも、IPアドレスを目的のアドレスに変更する必要があります（私の場合はホスト：["192.168.31.97:9092"]。  

インストール：

 PS C:\Users\Tomhunter\Desktop\winlogbeat-7.6.2-windows-x86_64> .\install-service-winlogbeat.ps1

注：サービスを手動で有効にするか、システムを再起動する必要があります。

すべて準備できました。何が得られるか見てみましょう。

そして、cybana（私の場合は192.168.31.97、デフォルトの資格情報：helk：hunting）に入ると、アラートが表示されます。未加工の形式では、次のようになります。



アラートは、電子メールやスラックなどで自分に送信できます。

しかし、私たちはこれがどのように機能するかにもっと興味があります、理解しましょう。

理論のビット


Windows 10でmimikatzを起動すると、次のメッセージが表示



されます。混乱を招くアラートを発行したルールは何ですか？（ネタバレ：mimikatz.exeファイル名ではありません）。ルール自体を見てみましょう：

sudo docker exec -it helk-elastalert bash
cat /opt/sigma/rules/windows/sysmon/sysmon_mimikatz_detection_lsass.yml



イベントID：10。これは、ProcessAccessです。

TargetImage：C：\ windows \ system32 \ lsass.exe。これは良いことですが、lsassに頼るのはmimikatzだけではありません。



GrantedAccess：0x1410または0x1010。

そして、ここにすべてが収まります。私が説明します。

mimikatzのソースコードを見てみましょう。つまり、kuhl_m_sekurlsa.cファイルとkuhl_m_sekurlsa_acquireLSA（）関数に興味があります



。MicrosoftのWebサイトを見ると、

PROCESS_QUERY_LIMITED_INFORMATION (0x1000)
PROCESS_VM_READ (0x0010)
PROCESS_QUERY_INFORMATION (0x0400)

ビット単位の「OR」を使用して、許可されたアクセスを取得します
注：以前のsigmaルールでは0x1410の検出しかありませんでした。これはおそらく、古いバージョンのmimikatzが両方のリクエスト（PROCESS_QUERY_INFORMATIONとPROCESS_QUERY_LIMITED_INFORMATION）を作成したためです。ただし、mimikatzは0x1010で渡されたため、ルールを自分で調整する必要がありました。現在、そのような問題はなく、すべてがそのまま機能します。

したがって、
kibanでフィルターevent_id：10 AND process_target_name： "* lsass.exe" AND process_granted_access： "0x1010" を設定した場合、mimikatzのみが存在します。



楽しいから：mimikatz_inmemory、mimikatz_trough_winrm、safetycatz（独自のチップを搭載したmimikatzの.NETバージョン）のルールがあります。

他に何ができますか？


もちろん、ルールはミミカッツに限定されず、たくさんあります。それらは、アプリケーション、クラウド、コンプライアンス、Linux、ネットワーク、プロキシ、Web、Windowsのカテゴリに分類されます。カテゴリはサブカテゴリに分かれています。たとえば、windowsは組み込み、マルウェア、その他、powershell、process_creation、sysmonに分かれています。

そして、おそらくGPOにPowershellログを含めることで、これらのルールに従ってアラートを受信することを知っています。
