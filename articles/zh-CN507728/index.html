在哈布雷（Habré）上有关类似主题的帖子中，我被提示写下我的决定，在该主题中，任务被“正面”解决-通过简单地列出日期以及使用该日历的系统整个生命周期的工作日标记。在类似的情况下，我决定做些棘手的事情，最终证明它更容易支持。如果您对这样做的方式感兴趣-欢迎在cat下使用：



除了简单之外，“正面”方法还有两个优点-1）搜索/选择的对数速度，因为日期本身就是一台自然的PC，按日期集选择Weekend属性是对（cluster）索引字段的操作，并且2）当检查一个连续集（当然是在存储范围内）时，相对于连续日期子范围，性能只会略有下降。



结果是，如果a）日期检查的范围很广（甚至在大十月革命之前），b）在检查谓词中，检查日期的范围不会十年，并且c）您不想打扰表中的成千上万条记录，而在99％的情况下都不会携带有用的信息，可以做不同的事情，即...



...根本不存储所有可能的日期（因此，没有对象应应用索引查找；），在给定范围内直接动态合成日期轴，按星期几的编号获取工作/天属性（星期一至星期五：工作，星期六... Su：请假），并将此规则的例外存储在紧凑表中。



好吧，



让我们开始吧：在给定范围内创建一个日期轴生成器（让它成为一个内联表值函数，以便在不同的应用程序中重用）： 



CREATE FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq
	FROM DR
)


我们检查： 



SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')

现在，我们将默认的周末符号粘贴到输出上： 



ALTER FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq,
		CONVERT(BIT, CASE WHEN DATEPART(WEEKDAY, DR.DateSeq) IN (7, 1) THEN 1 ELSE 0 END) AS IsHoliday
	FROM DR

现在已经很暖和了：







但是，有一个小问题：内置DATEPART函数返回的日期数字取决于DATEFIRST参数的值，而其默认值又取决于服务器的区域设置。因此，希望我们拥有美国系统，并且每周从星期日开始，通常是不可靠的。



如何解决这个问题？例如，通过变量@@ DATEFIRST读取DATEFIRST的当前值（该变量返回相对于美国默认表示的第一天的日期，其中第1天是星期日（！）），然后根据当前值@@ DATEFIRST得出当前符号中的天数。对应于星期六和星期日。花点时间尝试一下。



发生了吗我也没有。我决定不花一分钟时间去俱乐部的帮助上，而是采取了另一种方式-我收到了当前记法中的星期几，从我先验地知道它是星期六和星期日的日期开始，例如2000年1月1日和2日。然后，他仅将发布天数与当前情况下已知的先验天数进行比较： 



ALTER FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq,
		CONVERT(BIT, CASE WHEN DATEPART(WEEKDAY, DR.DateSeq) IN (Q.DOW_Sat, Q.DOW_Sun) THEN 1 ELSE 0 END) AS IsHoliday
	FROM DR
	CROSS JOIN (
		SELECT TOP (1)
			DATEPART(WEEKDAY, '1/1/2000')	AS DOW_Sat,	--  
			DATEPART(WEEKDAY, '1/2/2000')	AS DOW_Sun	--  
	) Q

)


有人可能会问，为什么所有这些CROSS JOIN / SELECT TOP（1）...混入请求中？但是要注意什么：由于我的TVF是内联的，并且函数主体本质上是一个SELECT，因此您不能输入任何变量-我们可以主要依赖SELECT的所有内容都可以是硬编码（这就是我们要避免的事情），或输出到子查询。



因此，我们使用CROSS JOIN（您可以根据需要将INNER ... ON 1 = 1）。



TOP（1）在这里，我们需要比我们自己更多的SQL引擎（毕竟，我们了解在子查询中总是只有一行！），但是他们说对于这种构造，TOP（1）告诉查询编译器子查询需要在主查询之前执行一次，而不是像主选择的每一行都执行CROSS APPLY一样。我不是100％确信这一点，特别是如果主体是不确定的子查询（我将以某种方式对其进行检查），但现在就这样吧。



我们检查： 



SET DATEFIRST 5	--          
SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')


答对了！即使在阳光明媚的城市，周末也是星期六和星期日！







现在微不足道了-创建一个表来存储异常并正确地合并它。我们将在使全球业务实现自动化的前提下做所有事情，在一个国家度假可能是另一个国家的灰色工作日： 



CREATE TABLE CountryCalendarOverride (
	CountryID INT NOT NULL,
	CalendarDate DATE NOT NULL,
	IsHoliday BIT NOT NULL,
       CONSTRAINT PK_CountryCalendarOverride PRIMARY KEY CLUSTERED 
       (
	  CountryID,
	  CalendarDate
       )
)


但是，如果我们想为假期留下标签，以免忘记我们正在庆祝，该怎么办？甚至不重新定义工作/休息日的迹象，而只是在某个日期的相反日期发表评论？很简单： 



ALTER TABLE CountryCalendarOverride ALTER COLUMN IsHoliday BIT NULL
GO
ALTER TABLE CountryCalendarOverride ADD DateComment NVARCHAR(256) NULL


为什么我们现在需要可为空的IsHoliday？然后，有一天我们可能只想针对某个日期发表评论，不知道/不考虑是休息日还是工作日，也不打算重新定义该特定标志-只是说这个日期需要注意。



现在一切准备就绪，可以进行最后冲刺： 



SET DATEFIRST 5	--           
SELECT 
	DX.DateSeq,
	ISNULL(CCO.IsHoliday, DX.IsHoliday) AS IsHoliday,
	CCO.DateComment
FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01') DX
LEFT JOIN CountryCalendarOverride CCO ON DX.DateSeq = CCO.CalendarDate 
                     AND CCO.CountryID = 1	-- ,     
ORDER BY DX.DateSeq
OPTION (MAXRECURSION 0)
