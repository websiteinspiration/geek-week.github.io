大規模プロジェクトをFlowからTypeScriptに変換した経験



JavaScriptは、動的型付けが可能な言語の1つです。このような言語は、迅速なアプリケーション開発に便利ですが、複数のチームが1つの大きなプロジェクトの開発を担当する場合は、最初から型チェックツールの1つを選択することをお勧めします。

TypeScriptコードの開発を開始するか、それをFlowプロジェクトに含めることができます。TypeScriptは、Microsoftが開発したJavaScriptのコンパイル済みバージョンです。TypeScriptとは異なり、フローは言語ではなく、コードを分析して型をチェックできるツールです。これらのアプローチに関するネット上の記事やビデオ、およびタイピングの使用を開始する方法についてのガイドを見つけることができます。この記事では、なぜFlowが適さなかったのか、そしてどのようにTypescriptに切り替え始めたのかを説明します。

ちょっとした歴史


2016年に、ECMシステム用のReact / ReduxベースのWebクライアントの開発を開始しました。次の理由により、タイピングをテストするためにフローが選択されました。

  1. ReactとFlowは同じ会社のFacebookの製品です。
  2. 流れはより活発に発達しました。
  3. フローはプロジェクトに簡単に統合できます。

しかし、プロジェクトが成長し、開発チームの数が増加し、Flowを使用するときにいくつかの問題が発生しました。

  1. バックグラウンドタイプチェックフローが使用したPCリソースが多すぎます。その結果、一部の開発者はこれをオフにし、必要に応じてチェックを開始しました。
  2. コードをFlowに合わせるのに、コード自体を書くのと同じくらいの時間がかかる状況がありました。
  3. コードがプロジェクトに表示され始め、フローテストに合格するためにのみ必要です。たとえば、nullの再確認：

     foo() {
    if (this.activeFormContainer == null) {
        return;
   }
   // to do something
   if (this.activeFormContainer != null) // only for Flow
          this.activeFormContainer.style.minWidth = '100px';
 }
  4. ほとんどの開発者はVisual Studio Codeコードエディターを使用しましたが、そこではFlowがTypeScriptほどサポートされていません。自動補完（IntelliSense）は開発中に常に機能するとは限らず、コードナビゲーションが不安定でした。Visual StudioでC＃を書くときと同じように簡単に開発できるようにしたいと思います。

一部の開発者は、TypeScriptに切り替えようとする考えを持っていました。移行のアイデアをテストし、管理を説得するために、プロトタイプを試すことにしました。

プロトタイプ


プロトタイプで2つのアイデアをテストしたかったのです。

  1. プロジェクト全体を翻訳してみてください。
  2. FlowとTypescriptの両方を同時に使用できるようにプロジェクトを設定します。

最初のアイデアでは、すべてのプロジェクトファイルを変換するユーティリティが必要でした。ネットワーク上でこれらの1つを見つけました。説明から判断すると、彼女はほとんどを翻訳できますが、変更の一部は自分で編集する必要があるか、ユーティリティ自体を追加する必要があります。少数のファイルでテストプロジェクトを変換することができました。しかし、実際のプロジェクトはコンパイルできませんでした。あまりにも多くのファイルを編集する必要があったでしょう。次のように、私たちはこの方向に進まないことにしました。

  1. やるべきことはまだたくさんありました！そして、プロジェクトを完成させる一方で、残りのチームは新しい機能の開発、バグの編集、テストの作成を続けます。また、ファイルのマージには時間がかかります。
  2. この方法でプロジェクトを翻訳した場合でも、テスターが実行しなければならない作業はどれくらいでしょうか。

私たちはこのオプションを放棄しましたが、それに関して有用な経験を得ました。各ファイルを翻訳するために必要な作業のおおよその量が明らかになりました。単純なReactコンポーネントの変換は次のようになります。 ご覧のとおり、多くの変更はありません。基本的には次のとおりです。





  • 削除// @フロー;
  • typeをより馴染みのあるインターフェースに置き換えます。
  • アクセス修飾子を追加します。
  • タイプをtsライブラリのタイプに置き換えます（図の例：イベントハンドラーとイベント自体）。

2番目のアイデアを実装すると、さらに開発が可能になりますが、既にTypeScriptを使用しており、バックグラウンドで既存のコードベースをゆっくりと変換できます。これにはいくつかの利点があります。

  1. 何かを見落とす心配がなく、翻訳が簡単です。
  2. テストが簡単。
  3. 変更を簡単にマージできます。

しかし、プロジェクトが2つのタイプのタイピングを並行して処理するように構成できるかどうかは完全には明らかではありませんでした。インターネットでの検索は具体的なものにつながらなかったので、彼らは自分たちでそれを整理し始めました。理論的には、Flowアナライザーはjs / jsx拡張子があり、コメントを含むファイルのみをチェックします

//@flow

/* @flow */

TypeScriptコンパイラの場合、ファイルには拡張子ts / tsxが必要です。そこから、タイピングへの両方のアプローチは同時に機能し、互いに干渉しないはずです。これを基にプロジェクト環境を整えます。最初のプロトタイプの経験を使用して、いくつかのファイルを翻訳しました。プロジェクトをコンパイルし、クライアントを起動しました-すべてが以前と同様に機能しました！

緑色の光


そして、ある晴れた日-スプリント計画の日である私たちのチームは、バックログにユーザーストーリー「TypeScriptへの切り替えを開始する」を持ち、次の作品のリストがあります。

  1. Webpackをセットアップします。
  2. tslintを構成します。
  3. テスト環境をセットアップします。
  4. ファイルをTypeScriptに変換します。

Webpackのセットアップ


最初のステップは、ts / tsx拡張子を持つファイルを処理する方法をwebpackに教えることです。これを行うために、構成ファイルのrulesセクションにルールを追加しました。元々使用されているts-loader：

// webpack.config.js
const rules = [
    ...
    {
      test: /\.(ts|tsx)?$/,
      loader: 'ts-loader',
      options: {
        transpileOnly: true
      }
    }
];

、アセンブリスピードアップ型のチェックをオフにするには、次のtranspileOnly: true理由 IDEは、コードの記述中に既にエラーを示しています。

しかし、私たちがReduxアクションの翻訳を始めたとき、それらが機能するためにbabel-plugin-transform-class-display-nameプラグインが必要であることが明らかになりました。このプラグインは、すべてのクラスに静的なdisplayNameプロパティを追加します。翻訳後、ts-loaderのみがアクションの処理を開始し、これによりそれらにbabelプラグインを適用できませんでした。その結果、ts-loaderを放棄し、以下を追加することでjs / jsxの既存のルールを拡張しましたバベル/ preset-typescript：

// webpack.config.js
const rules = [
    {
      test: /\.(ts|tsx|js|jsx)?$/,
      exclude: /node_modules|lib/,
      loader: 'babel-loader?cacheDirectory=true'
    },
    ...
];


// .babelrc.js
  const presets = [
    [
      "@babel/preset-env",
      {
        "modules": !isTest ? false : 'commonjs',
        "useBuiltIns": false
      }
    ],
    "@babel/typescript",
    "@babel/preset-react",
  ];

TypeScriptコンパイラが正しく機能するためには、tsconfig.json構成ファイルを追加する必要があります。これはドキュメントから取得されました。

Tslintを構成する


Flowを使用して記述されたコードは、eslintを使用してさらにチェックされました。TypeScriptには、対応するtslintがあります。最初は、すべてのルールをeslintからtslintに移行したいと思っていました。tslint-eslint-rulesプラグインを介してルールを同期する試みがありましたが、ほとんどのルールはサポートされていません。eslintを使用して、typescript-eslint-parserでtsファイルをチェックすることもできます。ただし、残念ながら、eslintに接続できるパーサーは1つだけです。すべての種類のファイルにts-parserのみを使用すると、jsファイルとtsの両方に多くの奇妙なエラーが表示されます。その結果、私たちは要件に拡張された推奨ルールセットを使用しました。

// tslint.json
  "extends": ["tslint:recommended", "tslint-react"]

ファイルをTypeScriptに変換する


これですべての準備が整い、ファイルの翻訳を開始できます。まず、プロジェクト全体で使用される小さなReactコンポーネントを転送することにしました。選択肢は「ボタン」コンポーネントにありました。



翻訳プロセス中に問題が発生しました：すべてのサードパーティライブラリにTypeScriptタイピング（たとえばbem-cn-lite）があるわけではありません。Microsoft のTypeSearchリソースで、そのタイプライブラリが見つかりませんでした。必要なライブラリのほとんどすべてについて、tsタイプライブラリを見つけて接続しました。解決策の1つは、requireを介して接続することでした。

const b = require(‘bem-cn-lite’);

しかし、同時に、型の欠如に関する問題は解決されませんでした。したがって、dts-genユーティリティを使用して、タイプ自体の「スタブ」を生成しました。

dts-gen -m bem-cn-lite

ユーティリティは、拡張子* .d.tsのファイルを生成しました。ファイルは@typesフォルダーに配置され、tsconfig.jsonを構成しました。

// tsconfig.json
    "typeRoots": [
      "./@types",
      "./node_modules/@types"
    ]

さらに、プロトタイプと同様に、コンポーネントを翻訳しました。プロジェクトをコンパイルし、クライアントを起動しました-それはすべてうまくいきました！しかし、テストは失敗しました。

テスト環境のセットアップ


アプリケーションをテストするには、StorybookとMochaを使用します。

Storybookは視覚回帰テストに使用されます（記事）。プロジェクト自体と同様に、webpackを使用してビルドされ、独自の構成ファイルがあります。したがって、ts / tsxファイルを操作するには、プロジェクト自体の構成と同様に構成する必要がありました。

ts-loaderを使用してプロジェクトをビルドしている間、Mochaテストの実行を停止しました。この問題を解決するには、テスト環境にts-nodeを追加します。

// mocha.opts
--require @babel/polyfill
--require @babel/register
--require test/index.js
--require tsconfig-paths/register
--require ts-node/register/transpile-only
--recursive
--reporter mochawesome
--reporter-options reportDir=../../bin/TestResults,reportName=js-test-results,inlineAssets=true
--exit

しかし、Babelに切り替えた後は、これを排除できます。

問題


翻訳プロセスでは、複雑さの度合いが異なる多数の問題が発生しました。これらは主に、TypeScriptの経験不足に関連しています。それらのいくつかを以下に示します。

  1. 異なるファイルタイプからコンポーネント/関数をインポートします。
  2. 高次コンポーネントの翻訳。
  3. 変更履歴の喪失。

異なるファイルタイプからコンポーネント/関数をインポートする


異なるファイルタイプのコンポーネント/関数を使用する場合、ファイル拡張子を指定する必要がありました。

import { foo } from ‘./utils.ts’

これにより、webpackおよびeslint構成ファイルに有効な拡張子を追加できます。

// webpack.config.js
resolve: {
   …
   extensions: [ '.tsx', '.ts', '.js' ]
 }


// .eslintrc.js
"import/resolver": {
  "node": {
    "extensions": [
      ".js",
      ".jsx",
      ".ts",
      ".tsx",
      ".json"
    ]
  }
}

高次コンポーネントの翻訳


すべてのファイルタイプの中で、高次コンポーネント（HOC）の翻訳が最も問題を引き起こしました。これは、入力でコンポーネントを取り、新しいコンポーネントを返す関数です。主にロジックの再利用に使用されます。たとえば、要素を選択する機能を追加する関数にすることができます。

const MyComponentWithSeletedItem = withSelectedItem(MyComponent);

または、Reduxライブラリからの最も有名な接続。そのような関数の型付けは簡単ではなく、型を操作するには追加のライブラリを接続する必要があります。あなたはネット上で主題に関する多くのマニュアルを見つけることができるので、私は翻訳プロセスを詳細に説明しません。要するに、問題はそのような関数が抽象的であるということです：プロパティのセットを持つコンポーネントは入力を受け入れることができます。タイトルとonClickプロパティを持つButtonコンポーネント、またはaltプロパティとimgUrlプロパティを持つPictureコンポーネントにすることができます。これらのプロパティのセットは、私たちに事前に知られているのではなく、関数自体が追加するプロパティだけが知られています。TypeScriptコンパイラがそのような関数の助けを借りて取得したコンポーネントを使用するときに誓わないようにするには、関数が戻り値の型から追加するプロパティを「カット」する必要があります。

これを行うには、次のものが必要です。

  1. これらのプロパティをインターフェイスに取り込みます。

    interface IWithSelectItem {
  selectedItem: number;
  handleSelectedItemChange: (id: number) => void;
}
  2. IWithSelectItemインターフェイスに入るすべてのプロパティをコンポーネントインターフェイスから削除します。これを行うには、ユーティリティタイプライブラリの操作Diff <T、U>を使用できます。

    React.ComponentType<Diff<TPropsComponent, IWithSelectItem>>

変更履歴の喪失


コードレビューなどのソースを操作するには、Team Foundation Serverを使用します。ファイルを翻訳するとき、1つの不快な機能に出くわしました。1つの変更されたファイルの代わりに、2つがリクエストプールに表示されます。

  • リモート-ファイルの古いバージョン。
  • 作成-新しいバージョン。


この動作は、ファイルが小さい場合など、ファイルに多くの変更（類似性<50％）がある場合に発生します。この問題を解決するために、以下を使用しようとしました。

  • git mvコマンド
  • 2つのコミットを実行します。1つ目はファイル拡張子の変更、2つ目は即時修正です。

しかし、残念ながら、両方のアプローチは私たちを助けませんでした。

概要


FlowまたはTypeScriptを使用する-誰もが自分で決める、どちらのアプローチにも長所と短所があります。私たちはTypeScriptを自分で選択しました。そして、あなたは自分の経験から確信を持っていました。アプローチの1つを選択し、3年たっても自分に合わないと突然気づいた場合は、いつでも変更できます。そして、よりスムーズな移行のために、並行して動作するようにプロジェクトを構成することができます。

執筆時点では、まだ完全にTypeScriptに切り替えられていませんが、主要部分であるプロジェクトの「コア」はすでに書き直されています。コードベースには、単純な反応コンポーネントから高次コンポーネントまで、あらゆる種類のファイルの変換の例があります。また、すべての開発チーム間でトレーニングが実施され、現在、各チームはタスクの一部として、プロジェクトの一部をこれらの職務に移しています。

年末までに移行を完了し、テストとストーリーブックを翻訳し、おそらくtslintルールのいくつかを書くことも計画しています。

私の個人的な感情によると、開発に時間がかからなくなり、システムをロードせずにオンザフライで型チェックが行われ、個人的にエラーメッセージが理解しやすくなったと言えます。
