単純すぎるタスク


NAは、大規模プロジェクトではアーキテクチャが重要であることを強調しています。しかし、これは小さなタスクへのChAアプローチの有用性を否定するものではありません。このタスクは単純すぎて、何らかの形式のアーキテクチャについて話すことはできないようです。また、アーキテクチャを介さない場合、変更を加える方法を指定するにはどうすればよいですか？インタラクティブインターフェイスの境界とコンポーネントを定義しないと、混乱しやすくなります。自分の仕事でフォームを変更するというタスクをどのように考えているかを覚えておいてください。

Angularでのフォームの実装の1つは次のようになります。

ContactFormGroupの実装（Angular）
export class ContactFormGroup {
    emailFormGroup = new EmailFormGroup();
    phoneFormGroup = new PhoneFormGroup();

    changes: Observable<unknown> = merge(this.emailFormGroup.changes, this.phoneFormGroup.changes);

    constructor() {}

    isValid(): boolean {
        return this.emailFormGroup.isValid() || this.phoneFormGroup.isValid();
    }

    getValue() {
        return {
            emails: this.emailFormGroup.getValue(),
            phones: this.phoneFormGroup.getValue(),
        };
    }
}

export class EmailFormGroup {
    emailControl = new FormControl();
    secondaryEmailControl = new FormControl();

    changes: Observable<unknown> = merge(
        this.emailControl.valueChanges,
        this.secondaryEmailControl.valueChanges,
    );

    isValid(): boolean {
        return this.emailControl.valid && !!this.emailControl.value;
    }

    getValue() {
        return {
            primary: this.emailControl.value,
            secondary: this.secondaryEmailControl.value,
        };
    }

    isSecondaryEmailVisible(): boolean {
        return this.isValid();
    }
}


したがって、3つのインターフェース（または重要ではないクラス）を取得します。これらのクラスを目立つ場所の別のファイルに配置して、インターフェースを単に見れば動く部分を理解できるようにする必要があります。我々は引っ張られ、特定され、問題のあるロジックを強調し、していると今、私たちはContactFormGroupの個々の部分の実装を組み合わせて、フォームの動作を制御します。また、さまざまなユースケースの要件を、個別のオブジェクトとして簡単に表すことができます。

これはMVCパターンの標準的な実装であり、それ以上のものではないようです。しかし、私は実際にはまったく尊重されていない基本的なことを却下しません。重要なのは、ビューからコードの一部を引き出したということではありません。重要なのは、変更される可能性のある重要な部分を強調し、その動作を説明して、それが単純になることです。

PS状態管理について


NAを理解する上で非常に大きな障害は、州の管理ライブラリへの取り組みです。実際、redux / mobxなどのライブラリは、変更についてコンポーネントに通知するタスクを同時に解決します。そして、一部の開発者にとって、状態マネージャーのないフロントは考えられないものです。ChAの原則は、州のマネージャーを使用してもしなくても適用できると思います。ただし、状態管理ライブラリに焦点を合わせると、必然的に柔軟性の一部が失われます。ChAとOOPの観点から考えると、状態管理の概念は一般的になくなります。状態管理なしの最も単純な実装はここにありますhabr.com/en/post/491684
