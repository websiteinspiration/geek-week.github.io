バージョン管理システムでのネットワーク機器とそのストレージの自動バックアップ

明確化：このソリューションは、D-Link DFL、cisco 29xx、およびWatchGuard Firebox向けに構成されていますが、ssh経由で接続するときにバックアップできるもの、および/またはftp / tftpサーバーにスケジュール/イベントでアップロードできるものすべてに適しています。

それはすべて、私の友人プログラマーが尋ねたという事実から始まりました：「ネットワーク構成をバージョン管理システムに保存してみませんか？」そして真実-私は思った-なぜですか？ほとんどの構成ファイルはテキスト形式でダウンロードできます（もちろん、これらはバイナリーですが、読み取り可能な情報がテキストエディターで開かれて表示されます）。

ネットワークには、約30の異なるD-Link DFL、1ダースのCisco 29xx、および2つのWatchGuard Fireboxがあります。各デバイスは、管理者レベル、常駐するITブランチの従業員、および本社のITにアクセスできます。これには、「6か月間バックアップを作成しておらず、すべてが壊れてしまいました。バックアップがありませんか？」そして、「最後のバックアップを焼き尽くした私たちの機器は前の管理者によって作成されました。私はどこにいるかわからないので、そのまま設定してください。」また、ciscoが保存されるたびにrunning-configを自動的にftpにリセットする場合、D-Linkの場合はそうではありません。また、バージョン管理システムは、「何をいつ変更したか覚えていませんが、月に一度使用するものが壊れている」などの問題を追跡するのに役立ちます。バックアップの規制は、人的要因からあなたを救うことも、あなたのためにバックアップすることもしないので、もつれを解くより警告する方が良いです。

私は正直に認めます。投稿する前に、悪臭や酸化については知りませんでした。しかし、彼らは両方とも私が完全に何が必要かを知りません。最初の1つはシスコ向けで、エキゾチックなものではあまり機能しません。2つ目はユーザー認証がありません（ただし、これはWebサーバーでの簡単な操作で実行できます）。最も重要なことは、どちらも冗長システムへのログインとパスワードが必要です。すべてのDリンクDFLはsshログインをサポートしています。シスコはバックアップ自体をマージし、通常はバックアップへの接続を必要としません。したがって、キーを使用してログインできる場合、パスワードを定期的に変更して保存する必要があるのはなぜですか。

自分に合ったことをすることにしました。

実際、自動バックアップは長い間行われていました。しかし、バージョン管理システム-それは新鮮でした。

つまり、タスク：ファイルのバージョン管理をサポートするネットワーク機器からすべての重要な構成を1か所に収集し、定期的にそれらを自動的に更新して、別のタイムゾーンの管理者が朝3に電話することなく簡単にダウンロードできるようにします。もちろん、これはすべて無料です。

すべてのDFLで、設定ファイルは同じ名前で呼び出されるため、コピーするときに明確に名前を付ける必要がありました。また、既存のrunning-config fillシステムをciscoでftpに再構成したくありませんでした。

まず、バージョン管理システムをセットアップします


私はsvn、tkを撮りました。 svnは、差分バイナリを保存する唯一の人気のあるファイルであり、履歴内のすべてのファイルではありません。小さなボリュームでは重要ではなかったが、すばらしい。一般に、バージョン管理システムが取る少量の構成ファイルを考慮します-違いはありません。

Visual SVNサーバーは、公式サイトからすぐに使える方法でインストールされます。私はそれをWindowsの下のサーバーに置いたので、同僚が管理（ユーザーの作成、権限の配布）を簡単に行えるようになりました。次に、リポジトリが作成され、必要なセクションが作成され、リポジトリに対する完全な権限を持つ唯一のユーザーが作成されます（現時点では）。 Windowsでは、アクションは完了しています。

自動バックアップは、1台のLinuxマシン（Debianの下）からのスクリプトで行われます。

まず、クライアントが存在するようにsvnを設定します。

sudo apt-get install svn

これで、プロジェクトのコピーが手動で作成されます（現在のディレクトリに作成されます）。

svn checkout https://servername.mydomain.ru/svn/DFL_BACKUPS/ DFL_BACKUPS --username go

これで、リポジトリに3つのセクションがあります。

  • Current_backup。D-LinkDFL構成が保存され、上書きされます。
  • Old_versions。ブランチ管理者が簡単にダウンロードできるように、日付が割り当てられた構成が保存されます。
  • シスコ、およびciscoとWatchGuard Fireboxの設定が保存されている場所（まあ、それは起こりました）。

DFL自動バックアップとDFL自体のセットアップから始めましょう


接続はsshになります。オープンな形式でパスワードを照らさないようにするために、公開鍵と秘密鍵のペアが使用されます。Privateは、サーバーの/ usr / dfl_scripts / sshkeys / dfl_keyに格納されています。

自動バックアップサーバーで公開鍵と秘密鍵のペアを生成します。

ssh-keygen –t rsa

オープンなものがすべてのDFLに注がれ、sshのアクセスルールが作成さ





れます。対応するリモート管理が許可されます。デフォルトからのポートは、たとえば2222に変更できますが、アクセスは特定のアドレスからのみ可能です。



D-Link DFLの現在の構成を含むファイルはルートにあり、config.bakと呼ばれます。scpを使用して自分にコピーするだけです。

したがって、スクリプト（ブランチのリストを切り取って、シートがないようにしました）：

#!/bin/bash
rm /usr/dfl_scripts/dflbackup/* #   
dfls=( "amur:10.28.10.254" # 
"arhangelsk:10.29.10.254"
"buratia:10.3.10.254"
"volgograd:10.34.10.254"
"voronezh:10.36.10.254"
"irkutsk_210:10.38.20.253"
"irkutsk_260:10.38.20.254"
"yaroslavl:10.76.10.254" )
rm /usr/dfl_scripts/log.txt # -
for dfl in "${dfls[@]}" ; do #   -
    filial="${dfl%%:*}" #delete the longest substr :* search from the end of string
    ip_addr="${dfl##*:}" #delete the longest substr *: search from the head of string
    echo $filial $ip_addr >> /usr/dfl_scripts/log.txt #    
    scp -P 2222 -i /usr/dfl_scripts/sshkeys/dfl_key admin@$ip_addr:config.bak /usr/dfl_scripts/dflbackup/$filial.bak 2>>/usr/dfl_scripts/log.txt #   ,     
done
FILES=$(ls /usr/dfl_scripts/dflbackup) #    
now=$(date +"%d_%m_%Y") #    
for f in $FILES ; do #      (    )
    cp /usr/dfl_scripts/dflbackup/$f /usr/dfl_scripts/DFL_BACKUPS/current_backup/$f #      
    cp /usr/dfl_scripts/dflbackup/$f /usr/dfl_scripts/DFL_BACKUPS/old_versions/${f%.bak}_$now.bak #        
done
cd /usr/dfl_scripts/DFL_BACKUPS #   svn 
svn add * --force -q >> /usr/dfl_scripts/log.txt #  .    .
svn commit -m "added backups $now" >> /usr/dfl_scripts/log.txt #    
/usr/bin/mail it@mydomain.ru < /usr/dfl_scripts/log.txt -s "   DFL" #    -

スクリプトはcronに詰め込まれ、毎日午前8時30分に実行されます。crontabは次のようになります。
30 08 * * * /usr/dfl_scripts/autobackup.sh
30 06 * * * /usr/dfl_scripts/ciscobackup.sh
00 06 * * 1 /usr/dfl_scripts/fireboxbackup.sh

シスコについて


デバイス自体では、次のようなものが構成されています。

archive
 log config
  logging enable
  logging size 500
  hidekeys
 path ftp://ftp.mydomain.ru/in/autolog/$H-$T
 write-memory
ip ftp username user
ip ftp password mypass
ip scp server enable

その結果、write-memoryコマンドは現在の構成をftpにリセットします。私のFTPは、自動バックアップの束とは別のサーバーにあります。2つの方法があります。そのサーバーにリポジトリのコピーを作成し、そこに構成を追加するか、隣接サーバーに構成をコピーします。基本的に、全体の違いはcpを実行するかscpを実行するかです。

configs with ciscoが落ちるディレクトリに、ファイルの外観を監視するためにincronを配置しました。ファイルが表示されるので、名前を変更し、一時ディレクトリの自動バックアップコレクタサーバーにコピーします。クリッピングにより、一時ディレクトリのファイルがsvnにアップロードされます。全体として、バージョン管理システムには、1日に1回、特定の時間の最新の構成があります。私の意見ではこれで十分ですが、ウィッシュリストに問題なくシステムを変更できます。

incrontab構成は次のようになります。

/var/ftp/cisco/in/autolog/ IN_CREATE /usr/ftp_scripts/incron_cisco.sh $# $%

スクリプト自体：

#!/bin/bash
if [[ $2 == "IN_MOVED_TO" || $2 == "IN_CREATE" ]]; then
  sleep 10
  filial="${1%%-*}" #delete the longest substr -* search from the end of string.       
  scp -P 2222 -i /usr/ftp_scripts/ssh_key_private /var/ftp/cisco/in/autolog/$1 go@10.10.10.220:/usr/dfl_scripts/ciscobackup/$filial.conf
fi

スケジュールされたスクリプトは、スケジュールに従って自動バックアップサーバーで起動されます。

#!/bin/bash
logfile=/usr/dfl_scripts/log_cisco.txt
rm $logfile
now=$(date +"%d_%m_%Y")
cd /usr/dfl_scripts/DFL_BACKUPS
cp /usr/dfl_scripts/ciscobackup/* /usr/dfl_scripts/DFL_BACKUPS/cisco/
svn add * --force –q >> $logfile
svn commit -m "added cisco backups $now" >> $logfile
/usr/bin/mail it@mydomain.ru < $logfile -s "   cisco  watchguard"

ご覧のとおり、ここではWatchGuard Fireboxについても説明しています。彼らのために別のディレクトリを作るのは率直に言って面倒だった。そのため、バックアップはciscoと同じ場所へのftpで行われます。30分前に、通常は週に1回、別のスクリプトでWatchGuardバックアップを実行します。構文がファームウェアバージョン12から変更されていることに注意してください。以前は、これは1つのバックアップイメージをftpコマンドで実行できましたが、エクスポートが追加されました。その上で、キーによるssh経由のログインを構成できなかったため、パスワードが使用されました。

#!/bin/bash
watchguards=( "10.10.10.20:password1"
"10.97.10.20:password2" )
now=$(date +"%Y_%m_%d")
for wg in "${watchguards[@]}" ; do
  ip_addr="${wg%%:*}"
  pass="${wg##*:}"
  sshpass -p $pass ssh backupuser@$ip_addr -p 4118 << ENDME
    backup image WG_$now.fxi
    export image WG_$now.fxi 123456qQ to ftp://user:mypass@ftp.mydomain.ru/in/autolog/WG_$ip_addr-$now.fxi
  ENDME
Done

WatchGuardバイナリは暗号化されてのみエクスポートされるため、テキストの外観と差分を使用すると、うまく機能しません。暗号化キーに注意してください：必須の大文字、小文字、数字、8文字以上。非準拠の場合、コマンドは単に実行されず、エラーの場所が明確に示されません。

だから、すべてが行われました。次に、クライアントをWindowsの作業用コンピューターに配置します。ブラウザからファイルと差分を表示するだけで、クライアントからのみ特定のリビジョンからバージョンをダウンロードできます。公式サイトから TorToSiteSVNを載せました。

サーバーで、ブランチに必要な権限（読み取り）を持つユーザーを作成します。

次に、コンテキストメニューから任意のファイルにクライアントを入力し、リポジトリに接続する必要があります。ファイルに変更を加える予定がない場合は、自分でリポジトリのコピーを作成する必要はありません。まあ、これには注意してください。ファイルシステムの一部だけでなく、バ​​ージョン管理システムを通じてコピーを操作する必要があります。

svn propset svn:mime-type text/plain current_backup/*.bak
