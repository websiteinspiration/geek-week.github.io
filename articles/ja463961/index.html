マッパーを所有するか、ExpressionTreesについて少し



今日は、AutoMapperの作成方法について説明します。はい、本当にお話ししたいと思いますが、それはできません。実際のところ、そのようなソリューションは非常に大きく、試行錯誤の歴史があり、アプリケーションにも長い道のりを歩んできました。私はこれがどのように機能するかについての理解を与えることができるだけであり、「マッパー」の作業のメカニズムを理解したい人のための出発点を与えます。私たちが自転車を書くと言うこともできます。

責任の否定


もう一度お知らせします。プリミティブマッパーを作成します。突然変更して製品で使用することにした場合は、これを行わないでください。この主題領域の問題のスタックを知っていて、それらを解決する方法をすでに知っている既製のソリューションを取ります。自転車マッパーを作成して使用する重要な理由はいくつかあります。

  • 特別なカスタマイズが必要です。
  • 自分の条件で最大のパフォーマンスが必要であり、コーンを充填する準備ができています。
  • マッパーの仕組みを理解したいとします。
  • あなたはサイクリングが好きです。

ワードマッパーは何と呼ばれていますか？


これは、オブジェクトを取得して別のオブジェクトに変換（値をコピー）するサブシステムです。一般的なタスク：DTOをビジネスレイヤーオブジェクトに変換します。最も原始的なマッパーは、データソースのプロパティを「実行」し、出力されるデータ型のプロパティと比較します。マッチング後、値がソースから抽出され、オブジェクトに書き込まれます。これが変換の結果になります。途中のどこかで、たぶん、この非常に「結果」を作成する必要があります。

コンシューマーにとって、マッパーは次のインターフェースを提供するサービスです。

public interface IMapper<out TOut>
{
     TOut Map(object source);
}

私が強調するのは、これが最も原始的なインターフェースであり、私の観点からは、説明に便利です。実際には、より具体的なマッパー（IMapper <TIn、TOut>）またはより一般的なファサード（IMapper）を処理する可能性が高く、それ自体が指定されたタイプの入出力オブジェクトの特定のマッパーを選択します。

素朴な実装


注：マッパーの単純な実装であっても、ReflectionとExpressionTreesの基本的な知識が必要です。リンクをたどらないか、これらのテクノロジーについて何も聞いていない場合は、それを読んでください。私は世界が同じになることは決してないと約束します。

ただし、独自のマッパーを作成しています。はじめに、出力されるデータ型のすべてのプロパティ（PropertyInfo）を取得します（以下、TOutと呼びます）。これは非常に単純です。TOut型でパラメーター化されたジェネリッククラスの実装を記述しているので、型がわかります。次に、Typeクラスのインスタンスを使用して、そのすべてのプロパティを取得します。

Type outType = typeof(TOut);
PropertyInfo[] outProperties = outType.GetProperties();

プロパティを取得するときは、機能を省略します。たとえば、セッター関数がないもの、属性によって無視されるものとしてマークされているもの、特別なアクセス権があるものなどがあります。最も簡単なオプションを検討しています。

さらに進みます。 TOut型のインスタンス、つまり受信オブジェクトをマップするオブジェクトそのものを作成できると便利です。 C＃では、これを行う方法がいくつかあります。たとえば、これを行うことができます：System.Activator.CreateInstance（）。または単に新しいTOut（）でさえ、これにはTOutの制限を作成する必要がありますが、これは一般化されたインターフェイスでは行いたくないものです。ただし、どちらもExpressionTreesについてある程度知っています。つまり、次のように実行できます。

ConstructorInfo outConstructor = outType.GetConstructor(Array.Empty<Type>());
Func<TOut> activator = outConstructor == null
   ? throw new Exception($"Default constructor for {outType.Name} not found")
   : Expression.Lambda<Func<TOut>>(Expression.New(outConstructor)).Compile();

なんでそうなの？ Typeクラスのインスタンスは、それが持つコンストラクターに関する情報を提供できることがわかっているため、データをコンストラクターに渡すようにマッパーを開発する場合に非常に便利です。また、ExpressionTreeについてもう少し学びました。つまり、それらはプラークがコードを作成およびコンパイルして、それを再利用できるようにします。この場合、実際には（）=> new TOut（）のような関数です。

次に、値をコピーするメインのマッパーメソッドを記述する必要があります。最も簡単な方法を説明します。入口に来たオブジェクトのプロパティを調べ、出力オブジェクトのプロパティの中から同じ名前のプロパティを探します。見つかった場合-コピーし、見つからない場合-先に進みます。

TOut outInstance = _activator();
PropertyInfo[] sourceProperties = source.GetType().GetProperties();

for (var i = 0; i < sourceProperties.Length; i++)
{
     PropertyInfo sourceProperty = sourceProperties[i];
     string propertyName = sourceProperty.Name;
     if (_outProperties.TryGetValue(propertyName, out PropertyInfo outProperty))
     {
          object sourceValue = sourceProperty.GetValue(source);
          outProperty.SetValue(outInstance, sourceValue);
     }
}

return outInstance;

したがって、BasicMapperクラスが完全に形成されました。ここで彼のテストに慣れることができます。ソースは、特定のタイプのオブジェクトまたは匿名オブジェクトのいずれかであることに注意してください。

パフォーマンスとボクシング


反射は素晴らしいですが、遅いです。さらに、その頻繁な使用はメモリトラフィックを増加させます。つまり、GCをロードします。つまり、アプリケーションをさらに遅くします。たとえば、私たちは使用PropertyInfo.SetValueとPropertyInfo.GetValue方法を。 GetValueメソッドは、特定の値がラップされたオブジェクト（ボクシング）を返します。これは、割り当てを最初から受け取ったことを意味します。

マッパーは通常、あるオブジェクトを別のオブジェクトに変換する必要がある場所に配置されます...いいえ、1つではなく、多くのオブジェクト。たとえば、データベースから何かを取得する場合です。この場所では、基本的な操作でメモリを失うことなく、通常のパフォーマンスを確認したいと思います。

私たちにできることExpressionTreesは私たちを再び助けてくれます。事実、.NETを使用すると、「オンザフライ」でコードを作成およびコンパイルできます。それをオブジェクト表現で記述し、何をどこで使用するかを言い、それをコンパイルします。ほとんど魔法はありません。

コンパイルされたマッパー


実際、すべてが比較的単純です。Expression.New（ConstructorInfo）ですでに新しいことを行っています。静的Newメソッドは、オペレーターとまったく同じように呼び出されることに気づいたでしょう。実際、ほとんどすべてのC＃構文は、Expressionクラスの静的メソッドの形式で反映されます。何かが足りない場合は、いわゆるいわゆる 「構文上の砂糖」

マッパーで使用する操作の一部を次に示します。

  • 変数宣言-Expression.Variable（Type、string）。Type引数は、作成される変数のタイプを示し、stringは変数の名前です。
  • — Expression.Assign(Expression, Expression). — , , — .
  • — Expression.Property(Expression, PropertyInfo). Expression — , PropertyInfo — Reflection .

この知識があれば、変数を作成し、オブジェクトのプロパティにアクセスし、オブジェクトのプロパティに値を割り当てることができます。ほとんどの場合、ExpressionTreeをFunc <object、TOut>形式のデリゲートにコンパイルする必要があることも理解しています。計画は次のとおりです。入力データを含む変数を取得し、TOut型のインスタンスを作成し、あるプロパティを別のプロパティに割り当てる式を作成します。

残念ながら、コードはそれほどコンパクトではないので、すぐにCompiledMapperの実装を確認することをお勧めします。ここには要点だけを持ち込んだ。

まず、関数のパラメーターのオブジェクト表現を作成します。オブジェクトを入力として受け取るため、オブジェクトはパラメーターになります。

var parameter = Expression.Parameter(typeof(object), "source");

次に、2つの変数と、割り当て式を順次追加する式リストを作成します。コンパイルされたメソッドを呼び出すときにコマンドが実行される方法であるため、順序は重要です。たとえば、まだ宣言されていない変数に値を割り当てることはできません。

さらに、単純な実装の場合と同様に、タイププロパティのリストを調べ、名前でそれらを照合します。ただし、すぐに値を割り当てるのではなく、値を抽出し、関連付けられている各プロパティに値を割り当てるための式を作成します。

Expression sourceValue = Expression.Property(sourceInstance, sourceProperty);
Expression outValue = Expression.Property(outInstance, outProperty);
                    
expressions.Add(Expression.Assign(outValue, sourceValue));

重要なポイント：すべての代入演算を作成したら、関数から結果を返す必要があります。これを行うには、リストの最後の式は、作成したクラスのインスタンスを含む式にする必要があります。この行の横にコメントを残しました。 ExpressionTreeのreturnキーワードに対応する動作が次のようになるのはなぜですか？これは別の問題だと思います。覚えやすいと思います。

最後に、作成したすべての式をコンパイルする必要があります。ここで何に興味がありますか？ body変数には、関数の「本体」が含まれています。 「通常の機能」には、体がありますね。さて、中かっこで囲みます。つまり、Expression.Blockはまさにそれです。中括弧もスコープであるため、そこで使用される変数（この場合はsourceInstanceとoutInstance）を渡す必要があります。

var body = Expression.Block(new[] {sourceInstance, outInstance}, expressions);
return Expression.Lambda<Func<object, TOut>>(body, parameter).Compile();

出力では、Func <object、TOut>を取得します。データをあるオブジェクトから別のオブジェクトに変換できる関数。なぜそんなに難しいのか、あなたは尋ねますか？最初に、ValueType値をコピーするときにボックス化を回避する必要があり、次に、PropertyInfo.GetValueメソッドとPropertyInfo.SetValueメソッドがやや遅いため、放棄したいと思いました。

なぜボクシングをしないのですか？コンパイルされたExpressionTreeは実際のILであり、ランタイムでは、コードと（ほぼ）似ています。なぜ「コンパイルされたマッパー」の方が速いのですか？繰り返しますが、それは単なるILだからです。ちなみに、BenchmarkDotNetライブラリを使用して速度を簡単に確認できます。ベンチマーク自体はここにあります。

比率の列では、「CompiledMapper」（CompiledMapper）は、AutoMapper（ベースライン、つまり1）と比較しても非常に良い結果を示しました。ただし、喜んではいけません。AutoMapperは、バイクに比べてはるかに優れた機能を備えています。このプレートでは、ExpressionTreesが「従来のリフレクションアプローチ」よりもはるかに高速であることを示したかっただけです。

概要


マッパーの作成が非常に簡単であることを示すことができたと思います。ReflectionとExpressionTreeは、開発者がさまざまなタスクを解決するために使用する非常に強力なツールです。依存関係の注入、シリアル化/逆シリアル化、CRUDリポジトリ、SQLクエリの作成、.NETアプリケーション用のスクリプトとしての他の言語の使用-これらはすべて、Reflection、Reflection.Emit、およびExpressionTreesを使用して行われます。
