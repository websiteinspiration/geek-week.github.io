ログは必要ありませんか？

近年、開発は大きく変化しています。モノリシックアプリケーションの代わりに、マイクロサービスと機能が登場しました。ユニバーサルインダストリアルモンスターのデータベースは、狭義のターゲットに退化しました。 Dockerは展開についての考え方を変えました。しかし、ログの考え方は変わりましたか？

Yandex.Verticalsの大きな問題の1つはログでした-1日あたり18 TB、1秒あたり250,000ログで、すべてがファイルに書き込まれます。 Scala、Java、Python、Goなどの多くの言語があるため、ログは異種混合です。次に、それらはFluent Bitによって収集され、Kafkaで書き込み、ハンドラーは1台の鉄の機械で動作し、Kafkaから組み立ててすべてをディスクに書き込みます。さらに、これはログの2番目のバージョンです。



その結果、長時間の探索という問題が発生する。これらのログは、grepを使用して検索されます。一部のサービスでは、grepが数時間に達することがあります。本番環境で問題が発生した場合、何時間もログを探す必要はありません。この問題を解決するために、Yandexは検索用にログ配信バイクを作成することにしました。これから何が起こったのか、アレクセイ・ダニロフ（Danevge）-Yandex.Verticalsのインフラストラクチャチームの開発者。auto.ruおよびYandex.Real Estateプロジェクトを開発、記述、サポートします。

免責事項 この記事では、最新の開発について説明しており、マイクロサービスアーキテクチャに適しています。ここではさまざまな製品が紹介されています-これらはYandex.Verticalsで使用されるツールです。他の条件下では、アナログはよりうまく機能しますが、ほとんど同じ機能を実行します。

Yandex.Verticalチームには300人がおり、そのうち約100人が開発者です。開発において、私たちは独自の製品ソリューションを作成するほとんどの企業と違いはありません。マイクロサービス、誰もがDockerに住んでいます。PHPのモノリスがHashicorp Nomadを介して展開された暗い隅にほこりを集めており、Scala、Java、Go、Node.js、Pythonなどの言語の動物園を維持しています。

Yandex.Verticalsの大きなインフラストラクチャの問題の1つは、アプリケーションログです。この問題に真剣に取り組んだとき、私たちはそれらのコレクションと処理の3番目のバージョンを使用しました。簡略化すると、次のように機能しました。

  • アプリケーションがファイルに書き込んだ。
  • Fluent Bitはファイルを読み取り、それらを行ごとにKafka filebeatに送信しました。
  • 専用のアイロンマシンでは、Kafkaトピックを読み取り、ディスク上のファイルに書き込むアプリケーションがありました。

暑い季節には、1日あたり18 TBのログ、つまり1秒あたり250,000行がありました。これは非常に大量であり、このデータでの作業が複雑になります。すべてがファイルに保存されるため、分析する唯一の方法はgrepです。大規模なアプリケーションの場合、分析には数時間かかる場合があります。生産上の問題のため、この時間はありません。

既製のソリューションは、価格、リソース、速度に適合しませんでした。彼らは私たちの流れを容認できるほど処理できませんでした。Elasticsearchをクックする試行回数を数えることすら難しい。どうやって作るか分からないのではないでしょうか。しかし、これは私たちが必要とするものではありません。それをログのリポジトリとして使用するには、特別な能力（スキル）が必要です。

このような状況で、ログを収集して分析する独自のシステムを実装することにしました。

Dockerログドライバー


ログを収集するために、 Go上のアプリケーションである独自のdockerログドライバーを作成しました。これは特別な方法で組み立てられ、Dockerプラグインコマンドによって配信され、レジストリに格納され、Dockerを実行する単一のインスタンスで実行されます。

ログドライバーの問題はすべての作業に悪影響を与える可能性があるため、最小限の実装を作成しようとしました。ドライバーはコンテナーのstdoutをリッスンし、すぐにログを近くのアプリケーションに渡します。それはすでに配信のより複雑な部分を扱っています。

問題


docker logドライバーのバージョンを更新する際の問題については別途触れます。


内部のGrafanaのスクリーンショット。

左側は、マシンに対するインストール済みバージョンの比率です。これで、3つのバージョンがすべてのハードウェアにインストールされました-車がどこにでも失われることはなく、不要なインストールはありません。右側は、このバージョンまたはそのバージョンを使用するコンテナーの数です。

Dockerドライバーはすぐに更新できません。これを行うには、すべてのコンテナとすべてのサービスを再起動する必要があるため、問題が発生する可能性があります。したがって、新しいバージョンをインストールするには、すべてのコンテナが更新されるのを待ちます。

一般的なスキーム


ログを収集および配信するための新しいシステムの一般的なスキームを検討してください。他の詳細はそれほど興味深いものではありません。



アプリケーションはログをJSON形式でstdoutに書き込みます。Dockerはコンテナーからのパイプをリッスンし、Dockerドライバーにリダイレクトします。Dockerドライバーは、Pusherのすべてを読み取り、非同期で再溶融します。

プッシャーはすべての鉄の車の上に立っています。ログを調理、飽和、反転し、Yandexメッセージキューにプッシュします。MQからのログストリームは、3種類のワーカーによって解析され、リポジトリに書き込まれます。

3つのロギングリポジトリがあります。

  • Yandex mapReduceは、長期間にわたってログを保存および分析します。これはHadoopの類似物です。
  • 最終日中にログを保存するためのClickHouse。
  • Humio（実験として）最終日のログを保存します。

利益


一般的な形式を使用すると、ログを書き込み、処理することができます。ログの収集はディスクを使用せずに自動で行われ、配信は数秒の領域で行われます。2秒から5秒までのキー検索。長期間にわたる保管と検索。

容量が少ない場合は、Humio、Splunk、Elasticなどの代替案を検討してください。最後の2つには公式のDockerドライバーがあります。AWSに住んでいる場合、それはAmazon CloudWatchです。

アマゾンクラウドウォッチ


Amazon CloudWatchは、メトリックス、イベント、およびログを処理します。彼は後者を探していません。スーパーサーチアイテムを提供しておらず、通常の形式でそれらを処理していません。Amazon CloudWatchは、ログを処理し、分析し、フィルターにかけ、グラフに表示します。


Amazon CloudWatchは、ログをメトリックスとグラフィックスに変換します。

ログをどうするか？


私たちの自転車に戻ります-それはすべてのケースを満たしますか？いいえ、私たちのソリューションではログを見つけることができますが、それらには、情報とそのタイプのはるかに多様なものが必要です。ログはより多くのケースで使用されます。

ログを収集するとすぐに、次の文が表示されます。「何かを解析し、何らかの方法で処理し、どこかに書き留めて、チャートやダッシュボードに表示し始めましょう。」これは地獄への道です。特に、一般的なツールについて話している場合はそうです。

ログを特定のカオスまたはデータのイベントログとして想像すると、それらは機能しません。

これは処理できない大きな情報の混乱になります。ゲームはログの形式化から始まります：「これらの行を特別な形式で記述して、後で解析するのに便利になるようにしましょう！」それも機能しません。信じて、試してみました。

タイピング


ログをタイプに分けて個別に処理すると、ログを扱いやすくするツールを見つけることができます。ログの場合とは異なり、有用なデータの場合と同様に機能します。このような作業はより透過的で便利です。一部のタイプのログは完全に破棄できます。

念のため


このタイプの「to be」ログは私のお気に入りです。これまたはその行が必要な理由を明確に答えることができない場合、これらはそれらです。このタイプは、「念のためログ」と呼ぶこともできます。

// validate customer
func Validate(customer Customer) {
    // ???
    log.debug(“Validate customer %v”, customer)
    …
    log.Error(“Customer not valid %v. Reason: %s”, ...)
    ….
}

ログは削除可能なコメントではありません。これはコードの一部であり、変更、維持、さらには削除がさらに困難です。

最良の場合、そのようなログはデバッグまたはトレースに変わる可能性があります。このタイプはコードを散らかします。ラッシュロギングのため、ユーザーの個人データ、パスワード、Cookieをそこに取得できます。

正しい方法は、それらを捨てて忘れることです。しかし、それから私たちは新たな問題に直面しています。エラーのある状況を解析する方法は？

致命的/重大なエラー


まず、重大なエラーのみを考慮します。これらは、ユーザーと開発者を苦しめるエラーです。最初-彼らが操作を実行できないとき。2番目-手動で修正する必要がある場合。

なぜログが合わないのですか？

迅速な対応なし。開発チームがサポートから、またはTwitterからユーザーからエラーについて学習した場合、何かを変更する時が来ました。

コンテキストはありません。エラーログの別の行は役に立ちません。コンテキストを少しずつ収集する必要があります。この場合でも、これはエラーではなくプロセスのコンテキストであるため、十分ではない可能性があります。

全体像はありません。質問に対する答えはありません：

  • このようなエラーが発生する頻度。
  • サービスの残りのレプリカで発生した。
  • 以前だった？

これらの問題を解決するには、適切なツール（sentry.ioなど）を使用してください。カスタマイズ可能なエラー情報を使用して、代表的な完全な（コンテキストを含む）エラー情報を処理できますalerting rule。歩哨のウェブサイトは、sentry.ioを使用した場合のログの違いについて説明しています。

重大なエラーではない


致命的で重大なエラーを投げましたが、今ではそれらはセントリーで書かれています。しかし、内部エラーが発生しました-さまざまなライブラリまたはサードパーティサービスからの回答。

良い例は、再試行の成功です。サービスAがサービスBに切り替えたが、ネットワークの問題により、応答が得られなかったとします。エラーの後、サービスAは再びサービスBに切り替わり、有効な応答を受け取りました。最初の呼び出しのエラーは重大ですか？番号。この場合、プロセスは正常に完了し、ユーザーはサービスを使用できました。

そのようなエラーがサービスの動作にとって重要ではなく、まれな繰り返しでユーザーに影響しない場合、これらのエラーはまったくエラーではありません。これはサービスの低下ですが、ユーザーへの応答は50ミリ秒後に来ました。このタイプのログは警告-警告を指します。

警告


 —   .

ここでは、重大なエラーに内在する同じ問題が確認されますが、予約されています。個々のイベントに対する反応は重要ではありません-時間の経過に伴うそれらの量が重要です。

サービスがキャッシュエントリを取得できず、コールドストレージにアクセスする例を考えてみます。これが1分間に1回発生する場合は、サービスの通常の操作に使用できます。レアエミッションは重要ではありません。

しかし同時に、全体像を見るためのツールが必要であり、リアルタイム分析が必要です。長期間にわたる変更を追跡するには、遡及分析も行うとよいでしょう。特定のレベル（しきい値）を超える劣化は、ユーザーに悪影響を及ぼす可能性があります- 深刻な劣化への対応が必要です。

警告とマークされたログではなく、低下の指標が必要です。

トレース


これは、ログを解析する必要があるパスを選択した場合、最初に開始します。呼び出しチェーンを構築し、要求内のデータ、呼び出しチェーンのエラー、応答時間、RPSの数を確認する必要があるため、ログのこの情報自体は役に立ちません。

トレースには素晴らしいツールがあります-JaegerまたはZipkin。どちらもサポートしているOpenTracingの使用をお勧めします。

3つのソースからトレースを収集できます。

  • 一般的なバランサーを使用する場合 -それらからログを解析し、Jaegerに送信します。
  • サービス自体。サービスディスカバリを通じてアドレスを受け取り、直接アクセスする場合。この場合、サービスからのトレースは直接Jaegerに送信されます。
  • スマートサービスメッシュ。彼は、トレースを収集して送信する方法を知っています（例：Istio）。


初期情報


この情報は、サービスAPI呼び出し、cron起動、データベースクエリ、または他のサービスへの呼び出しに関するものです。

{
    "_message": "Request: ...; request_id: ...,... ",
    "_level": "INFO",
    "_time": "2019-03-08T12:04:05.000+07:00",
    "_context": "r.y.a.w.v.c.Handler",
    "_tread": "785534"
}

この情報は「ジャストインケース」ブロックに属しますが、より一般的であるため、別のものです。この情報はエラーを解析するために必要であり、破棄する  ことができます。

内部メソッドへの呼び出しに関する情報が非常に重要であり、エラーが発生した場合に収集されたコンテキストでさえもそれなしでは実行できない場合、メソッド呼び出しをトレースとして計測する価値があります。

実行時間


この情報は、メソッド、API、データベースクエリ、またはその他のサービスの実行時間に関するものです。

{

    "_level": "INFO",

この情報を分析し、グラフに表示し、しきい値を設定する必要があるため、ログからの値はありません。このタイプのログは、Prometheusなどでメトリックに置き換える必要があり  ます。

ビジネス情報


この情報は、ビジネス分析、顧客行動の分析、財務計算に必要です。この場所では、歴史的に反対のアプローチである解析済みログを使用しました。しかし、これは、この方法でアプリケーションログを操作すると、アプリケーションログが退化する可能性があることの良い例です。

ビジネスデータを含むログの場合、TSKV形式の固定フィールドで合意が形成されました。これは分析に必要です。アプリケーションはビジネスログを専用ファイルに書き込みました。次に、ログが1行ずつ読み取られてMQに送信され、別のアプリケーションがそれらを処理してデータベースに書き込みました。これは、解析の結果の例です。

データが収束することを期待して、ログのストリーム全体を解析することはできません。

規約、フォーマット、ルール、信頼性の要件が出現しています。これはすでにアプリケーションログに少し似ています。この場合、ログはMQのすべての後続の要件を持つデータ配信キューになります。ここでは、ログ形式のミドルウェアが不要であることは注目に値します。

良い解決策は、このデータをMQに直接送信することです。すでにそこで処理され、適切なストレージに保存され、分析チームによって使用されます。たとえば、表示にはTableauを使用します。

パフォーマンス


このタイプのログはアプリケーションログにはほとんど見られず、メトリックとして収集されることがよくあります。それとは別に、言語固有の基本的なメトリックを収集するには、Prometheusライブラリを使用するだけで十分であることを追加します。彼女はデフォルトで到達したすべてのものを収集します。これらのメトリックを追加するコストはわずかです。



入力結果


ログをタイプ別にソートしたので、ログを操作するためのより強力なツールを選択できます。アマゾンのような複雑なシステムや宇宙技術はなく、明日上げることのできないものはありません。あなたはおそらくこれらのシステムまたは類似物のいくつかをすでに持っています：セントリーはどこかでほこりを集めています、プロメテウスはどこかで働いています。



問題は技術にあるのではなく、システムの状態を信頼性高く表現する手段としてログを信頼するときの認識の罠にあります。これはそうではなく、ログは無秩序なイベントのセットです。

例外があります-まれに使用できるデバッグログです。

デバッグログ


デバッグログは詳細情報である必要があります。これらは、上記で説明したシステムにすでに送信されているものと重複してはなりません。このタイプは、特殊なケースを解析するために存在します。たとえば、理解できないバグが本番環境で発生し、現在のところ、メトリックでは何が起こっているのか明確ではありません。

サービスを再起動せずに、ホットでデバッグログをオンにします。私たちはいくつかのサービスについて話しているので、それらの多くはありません。洗練されたインフラストラクチャは必要ありません。複雑な「準備」なしで十分なELKスタック。必要なすべてのコンテキストを使用して、Sentryにアラートを追加することも意味があります。

デバッグログは開発に使用できます。しかし、デバッグはそれらの優れた代替品です。

要約する
