Web SocketsとSocket.IOの違い



こんにちは！

WebソケットとSocket.IOはおそらく、リアルタイム通信（以下、ライブ通信と呼びます）の最も一般的な2つの手段です。しかし、それらはどう違うのですか？

ライブ通信用のアプリケーションを構築するとき、クライアントとサーバー間でデータを交換するためのツールを選択する必要があるときがあります。WebソケットとSocket.IOは、最新のWebでのライブ通信の最も一般的な手段です。どれを選ぶ？これらのテクノロジーの違いは何ですか？確認してみましょう。

Webソケット


Webソケットとは、単純なTCP接続を介した全二重通信チャネルを表すWeb通信プロトコルを意味します。簡単に言うと、このテクノロジーを使用すると、最小限のコストでクライアントとサーバー間の通信を確立でき、ライブ通信のすべての利点を利用するアプリケーションを作成できます。

たとえば、チャットを作成しているとします。できる限り迅速にデータを送受信する必要があります。これでWebソケットは問題なく動作します。TCP接続を開いて、必要なだけ開いたままにすることができます。

WebソケットはGoogle Chrome 4で2010年に登場しました。最初のRFC（6455）は2011年に公開されました

。Webソケットは次の場合に使用されます。

  • チャット
  • マルチプレイヤーゲーム
  • 共同編集
  • ソーシャル（ニュース）フィード
  • ロケーションベースのアプリケーション

Socket.io


Socket.IOは、Webソケットやその他のテクノロジーに基づいた（上に記述された）JavaScriptライブラリです。使用可能な場合はWebソケットを使用し、Webソケットが使用できない場合はFlash Socket、AJAX Long Polling、AJAX Multipart Streamなどのテクノロジーを使用します。簡単な例えは、Fetch APIとAxiosの比較です。

シンプルなWebソケットサーバー


Node.jsのプログラムでは、ポート3001で実行されるサーバーを作成します。クライアントが接続するたびに、一意のIDを割り当てます。クライアントにメッセージを送信するときに、成功を通知します：[<client-id>]：<message>

const WebSocket = require('ws')
const UUID = require('uuid')
const wss = new WebSocket.Server({ port: 3001 })

wss.on('connection', ws => {
  ws.id = UUID()

  ws.on('message', message => {
    ws.send(`[${ws.id}]: ${message}`)
  })
})

いいね！しかし、接続されている各クライアントにメッセージを送信したい場合はどうでしょうか。Webソケットは、デフォルトではメール配信をサポートしていません。これは次のように実装できます。

const WebSocket = require("ws")
const UUID      = require("uuid")
const wss       = new WebSocket.Server({ port: 3001 })

function broadcast(clientId, message) {
  wss.clients.forEach(client => {
    if(client.readyState === WebSocket.OPEN) {
      client.send(`[${clientId}]: ${message}`)
    }
  })
}

wss.on('conection', ws => {
  ws.id = UUID()
  ws.on('message', message => broadcast(ws.id, message))
})

簡単かつ簡単に！ご覧のとおり、WebSocket.Serverは接続されている各クライアントのレコードを保持しているので、繰り返し処理して全員にメッセージを送信できます。コンピューター（MacOS）またはブラウザー（Chrome）でコードをテストできます。

Socket.IO上のシンプルなサーバー


難しくありませんでした。Socket.IOでさらに簡単にできますか？同じサーバーをSocket.IOに書き込むにはどうすればよいですか？

const io = require('socket.io')
const server = io.listen(3002)

server.on('connection', socket => {
  socket.on('message', message => {
    socket.emit(`[${socket.id}]: ${message}`)
    socket.broadcast.emit(`[${socket.id}]: ${message}`)
  })
})

コードはほぼ半分短くなります！ご覧のとおり、broadcastメソッドは送信者に通知を送信しないため、手動で送信する必要があります。

問題があります。通常のWebソケットクライアントではコードをテストできません。これは、前述したように、Socket.IOは純粋なWebソケットを使用しないが、すべての可能なクライアントをサポートする多くのテクノロジーを使用しているためです。では、そのパフォーマンスをどのようにテストするのでしょうか。

// head
<script src="https://cdn.jsdelivr.net/npm/socket.io-client@2.3.0/dist/socket.io.slim.js"></script>

// body
<script>
  ioClient = io.connect('http://localhost:3002')
  ioClient.on('connect', socket => {
    ioClient.send('hello world')
    ioClient.on('message', msg => console.log(msg))
  })
</script>

特別なクライアントを使用する必要があります。上記の例では、CDNからロードします。このクライアントを使用すると、ブラウザーですばやく（ダーティー）テストを実行できます。

ご覧のとおり、例はそれほど異なりません。ただし、互換性について説明する場合、Socket.IOは独自のライブラリで動作し、Web開発に関連しない目的には使用できないことを忘れないでください。同時に、Webソケットを使用して、P2P通信、サーバー間のリアルタイムデータ交換などの幅広いタスクを解決できます。

メモについて


水平スケーリング。チャットの人気が高まり、リクエストを処理するために別のサーバーとロードバランサーを追加する必要があるとします。 「サーバー1」への接続を開くと、バランサーが「サーバー2」に切り替えると、「WebSocketハンドシェイク中にエラーが発生しました：予期しない応答コード：400」というエラーが発生します。 Socket.IOはCookieを使用して（または送信元アドレスに基づいて接続をルーティングすることによって）この問題を解決し、Webソケットにはそのようなメカニズムはありません。
パフォーマンス。前述のように、Socket.IOはWebソケットのトランスポート層の上にいくつかの抽象的な層を提供します。また、ここではデータをJSON形式にパッキングするため、バイナリデータをサーバーに送信することはできません（その逆も同様です）。このような機能が必要な場合は、目的の動作を保証するためにライブラリコードを「活用」する必要があります。Webソケットでは、このような問題は発生しません。

それで、何を選ぶべきですか？


あなたが決める。

Socket.IOは生活を楽にします。負荷分散、切断、またはメッセージの送信に関連する問題を心配する必要はありません...しかし、そのような機能は必要ですか？Socket.IOクライアントライブラリは、React、Redux、およびReact-Reduxパッケージを組み合わせた場合よりも重くなります。自分をWebソケットに限定できないと確信していますか？

留意すべきもう1つの重要な点は、サーバー側でSocket.IOを使用する場合、ほとんどのコードがこのライブラリによって提供される抽象化で記述されることです。Go、Elixir、Java、またはその他のプログラミング言語でNode.jsマイクロサービスを書き換える必要がある場合は、ほとんどすべてのロジックを書き換える必要があります。たとえば、Socket.IOにメッセージを送信するには、「ブロードキャスト」メソッドが使用されます（これはWebソケットに手動で実装されます）。したがって、リファクタリングするときは、このメソッドがどのように機能するかを理解する必要があります。この場合、Webソケットの方が適応が容易であるため、推奨されます。
