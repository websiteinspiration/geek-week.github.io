HTTP / 2イノベーション


はじめに、開発者の目にはどのHTTP / 2テクノロジーがGraphQLの価値に影響を与える可能性があるのか​​を考えてみましょう。HTTP / 2には多くの機能があります。これは、たとえば、新しいバイナリ形式と改善されたヘッダー圧縮です。しかし、私たちの場合、主な役割は、HTTP / 2を使用するときに要求と応答の配信がどのように処理されるかです。

TCP接続を開くことは、コストのかかる操作です。 HTTP / 1を使用するクライアントは、あまり頻繁に実行しない傾向があります。このため、システムに大きな負荷がかかるため、開発者は多くの場合、さまざまなテクノロジーに頼ってリクエストの数を制限しようとしました。たとえば、バッチクエリの実行、クエリ言語の使用、ページコードへのCSS / JSコードの埋め込み、個々の画像の代わりにスプライトシートの使用など。 HTTP / 1.1の場合。永続的な接続とパイプラインを使用してこれらの問題のいくつかを解決する試みが行われました情報処理。これらの2つのテクノロジーにより、ブラウザーは同じ接続内で複数の要求を送信し、それらに対する応答を受信できました。このようなデータ交換方式の欠点は、キューの先頭をブロックする問題（ヘッドオブラインブロッキング）が発生することでした。）この問題は、1つの遅い要求が、それに続くすべての要求の処理を遅くする可能性があるという事実で表されます。 HTTP / 2に取り組んだ専門家は、この問題を解決するさまざまな方法を提案しました。新しいバイナリプロトコルと共に、HTTP / 2は新しいデータ配信戦略も導入します。 HTTP / 2プロトコルを使用するシステムの相互作用中に、単一の接続が開かれ、その中で、各フレームがストリームの一部である場合にフレームを処理するように設計された新しいバイナリレベルを使用して、要求と応答の多重化が実行されます。このメカニズムを使用して、クライアントとサーバーは、フレーム内にあるそれらに関する情報に基づいて、要求フローと応答フローを再作成できます。これにより、HTTP / 2が非常に効率的になります。単一の接続内で実行される複数の要求の処理をサポートします。

しかし、それだけではありません。 HTTP / 2には、サーバープッシュと呼ばれる新しい概念があります。詳細には触れませんが、このテクノロジーにより、サーバーはクライアントにデータを事前に送信でき、クライアントがこのデータを要求する前にこれを行うことができます。この動作の最も顕著な例は、事前にスタイルシートとJavaScriptリソースをクライアントに送信することです。サーバーは、HTTPリクエストへの応答を生成するプロセスで、HTMLページをレンダリングするために特定のCSSファイルが必要であることを確認し、クライアントがこのファイルについてすぐに連絡することを事前に確認できます。これにより、クライアントが要求する前でも、サーバーは指定されたファイルをクライアントに送信できます。これが、前述のVulcainプロジェクトの仕組みです。このテクノロジーを使用して、関連するリソースの効率的なロードを整理します。

クライアントシステムとGraphQL開発


この記事では主にサーバーについて説明しますが、クライアント開発者はGraphQLテクノロジーを非常に気に入っていることを覚えておくことが重要です。GraphQLフラグメントを最新のフロントエンドフレームワークのコンポーネントアプローチと組み合わせると、完全に驚くべき抽象化が実現します。繰り返しますが、ここで定数クエリを追加すると、GraphQLによってクライアントシステムの開発者の作業が楽になると言えます。

GraphQLは、優れた機能を備えた全体的なシステムです。


GraphQLは、完全に独自の機能を持つものではありません。このシステムには代替手段があります。型付きスキーム？ OpenAPIについても同様です。さまざまなクライアントのユースケースをサポートするサーバーの抽象化ですか？これはさまざまな方法で実装できます。内省？ハイパーメディアを使用すると、クライアントはアクションを検出し、ルートエンティティの操作を開始できます。おいしいGraphiQLツール？ OpenAPIでも同様のものが作成されたと思います。 GraphQLの機能は、他のテクノロジーを使用していつでも再作成できます。ただし、GraphQLは全体的なシステムです。これが、このシステムを喜んで使用するGraphQLの開発者を非常に多く魅了するものです。これが、GraphQLの急速な普及と発展の理由の1つだと思います。それに、GraphQL-APIの構築は十分に文書化されているため、さまざまな言語用に設計されたGraphQLライブラリは、通常、高品質で人気があります。

HTTP / 2はGraphQLを助けることができますか？


したがって、GraphQLは開発者に多くの重要で便利なツールを提供しますが、HTTP / 2も優れたテクノロジーです。未来を見て、GraphQLシステムがHTTP / 2を使用することで得られるメリットについて考えてみましょう。たとえば、次のようになります。

概要


HTTP / 2は素晴らしいテクノロジーです（ここで示した例は、ある種の奇跡です）。GraphQLは、クライアント/サーバー通信セッションの数を減らす、または送信されるデータの量を最適化するのに役立つテクノロジーとしてのみ認識できます。もしそうなら、同様の観点からGraphQLを見る人は誰でも、HTTP / 2機能に基づくAPIを使用して非常に満足するでしょう。ただし、GraphQLで開発者に多くの便利な機能を提供するテクノロジーの組み合わせを見ると、GraphQLの能力はネットワークリソースの使用の改善とトラフィックの節約にまったく限定されていないことが明らかになります。
