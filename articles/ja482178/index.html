キューモニタリングシステムで大臣賞を受賞した韓国人学生の話

高校（2016年3月〜12月）の小学校の時、学校の食堂で起こった状況にとても悩まされました。

問題1：待機している列が長すぎる

どのような問題が発生しましたか？ここに1つあります



。多くの学生が配布に蓄積し、彼らは長い間（5〜10分）立っていなければなりませんでした。もちろん、これは一般的な問題であり、公正なサービススキームです。到着が遅いほど、サービスは遅くなります。なぜ待たなければならないのか理解できるでしょう。

問題2：待機の不平等な条件

しかし、もちろんこれだけではありませんでした。別のより深刻な問題を観察する必要がありました。最終的に私は状況から抜け出す方法を見つけることを試みることを決心したほど深刻でした。高校生（つまり、少なくとも1学年以上勉強するすべての人）と教師は、順番待ちをせずに配布に行きました。はい、そうです。そして、あなたは、小学生として何も話すことができませんでした。私たちの学校はクラス間の関係に関してかなり厳しい方針を持っていました。

そのため、私と友人は初めてでしたが、私たちは最初にダイニングルームに来て、食べ物を手に入れようとしていました。その後、高校生や教師が現れ、私たちを押しのけただけでした（より親切な人は、私たちをキューの場所に留まらせてくれました）。私たちは他の誰よりも早く到着しましたが、さらに15〜20分待たなければなりませんでした。

特にランチタイムが悪かった。日中は絶対にみんな（先生、生徒、スタッフ）が食堂に駆けつけてくれたので、中学生の私たちには喜びがありませんでした。

問題を解決する通常の方法

しかし、新規参入者には選択肢がないため、キューの最後に投げ込まれるリスクを減らすために2つの方法を考え出しました。 1つ目は、ダイニングルームに非常に早く（つまり、文字通り料理が提供される前に）来ることです。 2つ目は、ピンポンやバスケットボールで時間を潰して、長い遅延（夕食の開始から20分後）を伴うことです。

ある程度、これはうまくいきました。しかし率直に言って、誰もが足でダイニングルームに駆け寄り、食べられるようにしたり、残りの残り物を残りの後ろで食べたりすることを熱望していませんでした。ダイニングルームに人がほとんどいないときに知らせるソリューションが必要でした。

一部の占い師が私たちの未来を予測し、ダイニングルームにいつ行くべきかを正確に言って、私たちが長く待つ必要がないようにしたら、すばらしいでしょう。問題は、毎日すべてが異なる結果になるということでした。単純にパターンを分析して最高の瞬間を特定することはできませんでした。ダイニングルームで物事がどのように行われているかを確認する方法は1つしかありませんでした。徒歩でそこにたどり着くまでの道のりは、場所によっては数百メートルにもなります。だから、来て、ラインを見て、戻って来て、同じ精神でそれが短くなるまで続けるなら、あなたは多くの時間を失うでしょう。一般に、小学校のクラスはうんざりして住んでいた、そしてそれについて何もできなかった。

ユーレカ-食堂監視システムを作成するアイデア

そして突然、すでに次の学年（2017年）に、「キューの長さをリアルタイムで表示する（つまり、渋滞を検出する）システムを作成したらどうなるでしょうか」と私は思いました。私が成功できたとしたら、写真は次のようになります。小学生は電話を見て、現在の作業負荷のレベルに関する最新のデータを取得し、今行くのが理にかなっているかどうか結論を出します。

本質的に、このスキームは情報へのアクセスを通じて不平等を平滑化しました。その助けを借りて、若い学生は自分の最善を尽くすことを選択できます—長く行かなくても行けるか、時間をかけてより多くの利益を得て、後でより適切な瞬間を選ぶことができます。私はこの考えにとても興奮していました。

私は50個のメンブレンスイッチ、ESP8266に基づくWemos D1ミニボード、およびエナメル線が取り付けられる予定のいくつかのリングクリップのみを注文しました。

プロトタイピングと開発（2017年10月）

ブレッドボードから始めて、回路を組み立ててテストしました。材料の量に制限があったので、シールドを5つ持つシステムに制限しました。

私がC ++で作成したソフトウェアについては、次の目標を設定しました。

  1. 継続的に作業し、食事が提供される期間（朝食、ランチ、ディナー、午後のスナック）にのみデータを送信します。
  2. そのような周波数でのダイニングルームのキュー/渋滞の状況を認識して、データを機械学習モデル（たとえば、10 Hz）で使用できるようにします。
  3. 効率的な方法で（パケットサイズを小さくする必要があります）、短い間隔でサーバーにデータを送信します。

それらを達成するために、私は次のことをする必要がありました：

  1. RTCモジュール（リアルタイムクロック、またはリアルタイムクロック）を使用して継続的に時間を監視し、ダイニングルームで食事が提供される時間を決定します。
  2. データ圧縮方式を使用して、シールドの状態を1文字で登録します。データをバイナリ5桁コードとして扱い、データ要素を表すようにさまざまな値をASCII文字にバインドしました。
  3. ThingSpeak（分析およびオンライングラフ用のIoTツール）を使用し、POSTメソッドを使用してHTTPリクエストを送信します。

もちろん、いくつかのバグがありました。たとえば、sizeof（）演算子がchar *オブジェクトに対して4の値を返し、文字列の長さではないことを知りませんでした（これは配列ではないため、コンパイラーが長さを計算しないため）、HTTPリクエストに4文字しか含まれていない理由は非常に驚きましたURL全体から！

また、括弧を#defineステージに入れなかったため、予期しない結果が生じました。まあ、としましょう：

#define _A    2 * 5 
int a = _A / 3;

ここでは、Aが3（10/3 = 3）になることを期待する価値がありますが、実際には、2（2 * 5/3 = 2）のように計算が異なります。

最後に、私が対処したもう1つの注目すべきバグは、ウォッチドッグのリセットです。この問題で私は非常に長い間苦しみました。後で判明したように、ESP8266チップの低レベルレジストリに間違った方法でアクセスしようとしました（構造体へのポインターに誤ってNULL値を登録しました）。



私がデザインして組み立てたフットボード。写真が撮られたとき、彼はすでに5週間の踏みつけを経験していました。

ハードウェア（フットボード）

シールドが過酷な食事条件に耐えられるようにするために、私は次の要件を設定しました。

  • シールドは、常に人間の体重を支えるのに十分な強度が必要です。
  • シールドは、キューにいる人の邪魔にならないように薄くする必要があります。
  • スイッチは、攻撃的に動作することを確認する必要があります。
  • シールドは防水でなければなりません。ダイニングルームは常に湿っています。

これらの要件を満たすために、私は2層構造の設計に落ち着きました。アクリルレーザーカットがベースとトップカバーに行き、保護層としてコルク材を使用しました。

AutoCADで作成したシールドレイアウト。サイズ-400 x 400ミリメートル。



左側はうまくいったデザインです。右側には、レゴタイプの接続のオプションがあります。

ちなみに、このような固定システムではシールド間に40センチメートルの間隔が必要であることが判明したため、最終的に正しい設計を拒否しました。つまり、必要な距離（10メートル以上）をカバーできませんでした。





すべてのスイッチを接続するために、エナメル線を使用しました-それらはすべて70メートル以上かかりました！各シールドの中央に、メンブレンスイッチを配置しました。サイドスロットから外側、スイッチの左右に2つのクリップが突き出ていました。

さて、防水性のために、電気テープを使いました。電気テープがたくさん。

そしてそれはすべてうまくいった！

で、11月5日の午前8時（朝食時）に、システムはダイニングルームの状況に関する現在のデータを収集し始めました。私の目は信じられませんでした。ちょうど約2か月前に、私は家でパジャマを着て座っている一般的なスキームをスケッチしました、そして今、システム全体は問題なく問題なく動作しています...

テスト期間中のソフトウェアのバグ

もちろん、システムには十分なバグがありました。ここに私が覚えているものがあります。

プログラムは、クライアントをThingSpeak APIに接続しようとしたときに、利用可能なWi-Fiポイントをチェックしませんでした。エラーを修正するために、追加のステップとしてWi-Fiの可用性チェックを追加しました。

セットアップ機能では、接続が表示されるまで「WiFi.begin」を常に呼び出しました。後でESP8266ファームウェアが接続をインストールし、Wi-Fiのセットアップ時にのみbegin関数が使用されることがわかりました。セットアップ中に関数を1回だけ呼び出し始めたことで、状況を修正しました。

作成したコマンドラインインターフェイス（時間の設定、ネットワーク設定の変更を目的として設計されたもの）は、安静時には（つまり、朝食、昼食、夕食、午後の軽食の外では）機能しません。また、ロギングがない場合、内部ループが過度に加速され、シリアルデータの読み取りが速すぎることもわかりました。したがって、期待されるときに追加のコマンドが到着するのをシステムが待機するように、遅延を設定しました。

番犬へのオード

そして、その後、ウォッチドッグタイマーのその問題について-「フィールド」条件のテスト段階でそれを決定しました。誇張せずに、このことを4日間だけ考えました。休憩ごとに（10分間）新しいバージョンのコードを試すために、私はダイニングルームに急いで行きました。そして、配布が始まったとき、私はバグを捕まえるために床に1時間滞在しました。食べ物については考えなかった！良いウォッチドッグタイマーESP8266をありがとうございます。

テスト中のハードウェアの問題

もちろん、機器、つまりフットボードは理想とは程遠いものでした。予想通り、スイッチの1つがスタックしました。



11月7日の昼食時に、3番目のシールドのスイッチが動かなくなった

上で、ThingSpeak Webサイトのオンラインチャートのスクリーンショットを提供しました。ご覧のとおり、12：25頃に何かが起こり、その後シールド3が故障しました。その結果、実際には3番目のシールドに到達しなかった場合でも、キューの長さは3（値は3 * 100）として定義されました。修正は、スイッチをより広くするためにライニング（はい、電気テープ）を追加することでした。

時々、ワイヤーがドアにくっついたとき、私のシステムが文字通り根こそぎになった。トロリーがこのドアを通ってダイニングルームに運ばれ、パッケージが持ち込まれたので、ワイヤーがその後ろに運ばれ、閉じて、ソケットから引き出されました。そのような場合、私はデータの受信で予期しない障害に気づき、システムが電源から切断されていると推測しました。

私のプロジェクトで韓国文部省から最優秀賞を受賞した方法


ご存じのとおり、私はある種の賞や表彰を獲得しようとしていたので、このプロジェクトには取り組みませんでした。学校で遭遇した慢性的な問題を解決するために自分のスキルを活用したかっただけです。

しかし、私がプロジェクトの計画と開発をしているときに非常に親しくなった私たちの学校の栄養士であるミスOは、食堂のアイデアのためのコンテストについて知っているかどうか尋ねました。それから私はそれはある種の奇妙な考えだと思った-食堂の考えを測定すること。しかし、私は情報ブックレットを読み、プロジェクトが11月24日までに提出されなければならないことを知りました！まあまあ。私はすぐにコンセプト、データ、グラフを思いつき、アプリケーションを送信しました。

コンテストの元のアイデアへの変更

ちなみに、私が最終的に提案したシステムは、すでに実装されているものとは少し異なりました。実際、私は元の方法（リアルタイムでキューの長さを測定する）をはるかに大きい韓国の学校に適用しました。比較のために：私たちの学校には300人の生徒がいますが、他の生徒では1つのクラスだけで勉強する人もいます。システムのスケーリング方法を理解する必要がありました。

したがって、私はより手動制御に基づく概念を提案しました。現在、韓国の学校ではすべてのクラスに食事の順序を導入していますが、これは厳格に守られているため、「シグナル応答」タイプの別のフレームワークを作成しました。目の前のダイニングルームを訪問しているグループがキューの長さの特定の制限に達すると（つまり、キューが短くなると）、壁のボタンまたはスイッチを使用して手動で信号を送信すると想定されていました。信号はテレビ画面またはLED電球を介して送信されます。

国のすべての学校で生じた問題を本当に解決したかっただけです。 Miss Oの話を1つ聞いたとき、私はさらに強くなりました。今からお話しします。一部の大規模な学校では、誰もが適切にプロセスを構築できないため、路線がカフェテリアを超えて、冬であっても20〜30メートルの通りに出ていることがわかりました。そして、時には数分間、誰もダイニングルームにまったく姿を見せないことが起こります-これも悪いことです。生徒数の多い学校では、食べ物に割り当てられた時間の一部がなくなったとしても、スタッフがほとんどすべての人にサービスを提供することができません。したがって、最後までに分布に近づく人（通常、これらは小学生です）には、食べる時間がないだけです。

そのため、急いでアプリケーションを送信しなければならなかったとしても、それを広く使用するためにどのように適合させるかを非常に慎重に検討しました。

結論


ここにはいくつかの皮肉があります。私が意図的に登録したコンテストや科学フェアにいくら参加しても、良いことは何もありませんでした。そして、ここでこの機会が私を見つけ、良い結果をもたらしました。

プロジェクトを始めるように促した理由について考えさせられました。なぜ私は仕事を始めますか-「勝つ」ために、または私たちの周りの世界に本当に存在する問題を解決するために？もしあなたのケースに第二の動機が当てはまるなら、私はあなたにプロジェクトを放棄しないことを強く勧めます。このビジネスへの取り組みにより、途中で予期しない機会に遭遇する可能性があり、勝利の必要性のプレッシャーを感じることはありません-あなたの主な動機はあなたの仕事への熱意になります。

そして最も重要なのは、適切なソリューションを実装できれば、すぐに現実の世界で試すことができるということです。私の場合、学校はプラットフォームになりましたが、経験は蓄積され、誰が知っているのでしょう-おそらくあなたの国全体、あるいは世界全体があなたのアプリケーションを使用するでしょう。

毎回、この経験を思い出して、自分自身を誇りに思うようにさえなりました。その理由を説明することはできませんが、プロジェクトの実施プロセスは単に私に大きな喜びをもたらし、賞品は追加のボーナスでした。また、同級生の日常生活に支障をきたす問題も解決できてよかったです。ある学生が私に近づいてきて、「あなたのシステムでとても便利だ」と言ったことがあります。私は第七天国にいました！
賞がなくても、これだけでこの発展を誇りに思うと思います。多分これは他の人への助けが私にそのような満足をもたらしたのかもしれません...
