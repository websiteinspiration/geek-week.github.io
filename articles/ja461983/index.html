特定のクライアント向けに製品を変更する方法


そこで、クライアントにB2Bソフトウェア製品を販売しました。

プレゼンテーションでは、彼はすべてが好きでしたが、実装中にまだ何かが合わないことがわかりました。もちろん、「ベストプラクティス」に従って、自分を製品に変える必要があると言うことができます。これは、強力なブランドがある場合（たとえば、3つの大きな文字と3つの小さな文字すべてを送信できる場合）に機能します。そうでなければ、彼らは顧客が彼のユニークなビジネスプロセスのおかげですべてを達成したことをすぐにあなたに説明し、あなたの製品をより良く変えましょうか、それはうまくいきません。ライセンスを既に購入しているという事実を拒否して参照するオプションがあり、潜水艦から行く場所がありません。しかし、比較的狭い市場では、そのような戦略は長い間機能しません。

修正する必要があります。

アプローチ


製品の適応にはいくつかの基本的なアプローチがあります。

モノリス


変更は製品のソースコードに直接行われますが、特定のオプションに含まれています。そのような製品には、原則として、混乱することのないように、番号またはコードが割り当てられた設定を持つ巨大なフォームがあります。このアプローチの欠点は、ソースコードが大きなスパゲッティに変わることです。この場合、さまざまなユースケースが多すぎて、維持するのに非常に長く費用がかかります。後続の各オプションには、ますます多くのリソースが必要です。そのような製品の性能も望まれるべき多くのものを残します。そして、製品が書かれている言語が継承やポリモーフィズムのような現代の慣行をサポートしていない場合、すべてが非常に悲しくなります。

コピーする


クライアントには、製品のソースコード全体と、それを変更するためのライセンスが与えられます。多くの場合、このようなベンダーは顧客に製品自体を改造しないことを伝えます。これは高すぎるためです（サービスに連絡するよりもライセンスを販売するほうがはるかに有利です）。しかし、彼らには、第三国のどこかに比較的安価で高品質の従業員を雇い、彼らを助ける準備ができている、よく知られたアウトソーサーがいます。また、お客様のスペシャリストが直接改善を実施する状況もあります（スタッフがいる場合）。このような場合、ソースコードは開始点として扱われ、変更されたコードは元々のコードとは関係がなく、自分の人生を生きます。この場合、元の製品の少なくとも半分を安全に削除して、独自のロジックに置き換えることができます。

合併


これは、最初の2つのアプローチを組み合わせたものです。しかし、その中で、コードを修正する開発者は常に「マージが来る」ことを覚えておく必要があります。ソース製品の新しいバージョンがリリースされると、ほとんどの場合、ソースの変更と変更されたコードを手動でマージする必要があります。問題は、どのような衝突でも、特定の変更が行われた理由を覚えておく必要があることです。これは非常に昔のことかもしれません。また、元の製品でコードリファクタリングが実行された場合（たとえば、コードブロックが単純に再配置された場合）、マージには非常に時間がかかります。

モジュール性


論理的に最も正しいアプローチ。この場合の製品のソースコードは変更されず、機能を拡張する追加のモジュールが追加されます。ただし、このようなスキームを実装するには、製品にこの方法で拡張できるアーキテクチャが必要です。

説明


次に、例を挙げて、オープンで無料のlsFusionプラットフォームに基づいて開発された製品をどのように拡張するかを示します。

システムの重要な要素はモジュールです。このモジュールには、テキストファイルであるLSFの拡張が含まれ、lsFusionコードを。各モジュールでは、ドメインロジック（関数、クラス、アクション）とプレゼンテーションロジック（フォーム、ナビゲーター）の両方が宣言されています。原則として、モジュールは論理的な原則によって分割されたディレクトリにあります。製品は、その機能を実装し、別のリポジトリに格納されているモジュールのコレクションです。

モジュールは相互に依存しています。あるモジュールがそのロジックを使用する場合（たとえば、プロパティやフォームを参照する場合）は、別のモジュールに依存しています。

新しいクライアントが表示されると、そのクライアント用に別個のリポジトリー（GitまたはSubversion）が起動され、必要な変更が加えられたモジュールが作成されます。このリポジトリは、いわゆるトップモジュールを定義します。サーバーが起動すると、それらのモジュールのみが接続され、他のモジュールを介して直接または推移的に依存します。これにより、クライアントは製品のすべての機能ではなく、必要な部分のみを使用できます。
Jenkinsは、製品モジュールとクライアントモジュールを1つのjarファイルに結合するタスクを作成し、それを本番サーバーまたはテストサーバーにインストールします。

実際に発生する改善の主なケースをいくつか考えてみ

ます。製品にOrderモジュールがあり、標準の注文ロジックを記述しているとします。

MODULE Order;

CLASS Book '';
name '' = DATA ISTRING[100] (Book) IN id;

CLASS Order '';
date '' = DATA DATE (Order) IN id;
number '' = DATA STRING[10] (Order) IN id;

order '' = DATA Order (OrderDetail) NONULL DELETE;

book '' = DATA Book (OrderDetail) NONULL;
nameBook '' (OrderDetail d) = name(book(d));

quantity '' = DATA INTEGER (OrderDetail);
price '' = DATA NUMERIC[14,2] (OrderDetail);
sum '' (OrderDetail d) = quantity(d) * price(d);

FORM order ''
    OBJECTS o = Order PANEL
    PROPERTIES(o) date, number

    OBJECTS d = OrderDetail
    PROPERTIES(d) nameBook, quantity, price, NEW, DELETE
    FILTERS order(d) = o

    EDIT Order OBJECT o
;

FORM orders ''
    OBJECTS o = Order
    PROPERTIES(o) READONLY date, number
    PROPERTIES(o) NEWSESSION NEW, EDIT, DELETE
;

NAVIGATOR {
    NEW orders;
}

顧客Xは、注文明細に割引率と割引価格を追加したいと考えています。
最初に、新しいOrderXモジュールがカスタマーリポジトリに作成されます。ヘッダーでは、元のOrderモジュールに依存関係が配置されています。
REQUIRE Order;

このモジュールでは、テーブルに追加のフィールドが作成される新しいプロパティを宣言し、フォームに追加します。
discount ', %' = DATA NUMERIC[5,2] (OrderDetail);
discountPrice '  ' = DATA NUMERIC[14,2] (OrderDetail);

EXTEND FORM order
    PROPERTIES(d) AFTER price(d) discount, discountPrice READONLY
;

割引価格での録音はできません。初期価格または割引率のいずれかが変更された場合、別のイベントとして計算されます。
WHEN LOCAL CHANGED(price(OrderDetail d)) OR CHANGED(discount(d)) DO
  discountPrice(d) <- price(d) * (100 (-) discount(d)) / 100;

次に、注文明細の金額の計算を変更する必要があります（新しく作成した割引価格を考慮する必要があります）。これを行うには、通常、他のモジュールが動作を挿入できる特定の「エントリポイント」を作成します。Orderモジュールでのsumプロパティの最初の宣言の代わりに、以下を使用します。
sum '' = ABSTRACT CASE NUMERIC[16,2] (OrderDetail);
sum (OrderDetail d) += WHEN price(d) THEN quantity(d) * price(d); 

この場合、sumプロパティの値は1つのCASEに収集され、WHENは異なるモジュールに分散できます。モジュールAがモジュールBに依存している場合、モジュールBのすべてのWHENがモジュールAのWHENよりも後に動作することが保証されます。割引額を正しく計算するために、次の宣言がOrderXモジュールに追加されます。
sum(OrderDetail d) += WHEN discount(d) THEN quantity(d) * discountPrice(d);

その結果、割引が設定されている場合は金額が適用され、それ以外の場合は元の式が適用されます。

クライアントが、注文金額がある特定の金額を超えないように制限を追加したいとします。同じOrderXモジュールで、制約値が格納されるプロパティを宣言し、それを標準オプションフォームに追加します（必要に応じて、設定を含む別のフォームを作成できます）。
orderLimit ' ' = DATA NUMERIC[16,2] ();
EXTEND FORM options 
    PROPERTIES() orderLimit
;

次に、同じモジュールで、注文の量を宣言し、それをフォームに表示し、その過剰に制限を追加します。
sum '' (Order o) = GROUP SUM sum(OrderDetail d) IF order(d) = o;
EXTEND FORM order
   PROPERTIES(o) sum
;
CONSTRAINT sum(Order o) > orderLimit() MESSAGE '  ';

最後に、クライアントは注文編集フォームのデザインを少し変更するように依頼しました。注文ヘッダーを行の左側に区切り記号を付けて作成し、価格を常に2文字の精度で表示するように求めました。これを行うには、次のコードをモジュールに追加します。これにより、注文フォームの標準で生成されたデザインが変更されます。
DESIGN order {
    OBJECTS {
        NEW pane {
            fill = 1;
            type = SPLITH;
            MOVE BOX(o);
            MOVE BOX(d) {
                PROPERTY(price(d)) { pattern = '#,##0.00'; }
                PROPERTY(discountPrice(d)) { pattern = '#,##0.00'; }
            }
        }
    }
}
その結果、注文の基本ロジックが実装されている2つのOrderモジュール（製品内）と、必要な割引ロジックが実装されているOrderX（顧客側）が取得されます。

OrderX モジュールはOrderDiscountと呼ばれ、製品に直接転送できることに注意してください。そして、必要に応じて、各顧客が機能と割引を簡単に関連付けることが可能になります。

これは、プラットフォームが個々のモジュールの機能を拡張するために提供するすべての機会とはかけ離れています。たとえば、継承を使用して、レジスタロジックをモジュール式に実装できます。

依存モジュールのコードと矛盾する製品のソースコードの変更がある場合、サーバーの起動時にエラーが生成されます。たとえば、注文フォームがされて削除して注文モジュールは、起動時にエラーがあるだろうという点でOrderXのモジュール注文フォームが見つかりません。また、エラーはIDEで強調表示されます。また、IDEにはプロジェクト内のすべてのエラーを検索する機能があり、製品バージョンの更新により発生したすべての問題を特定できます。

実際には、（製品とすべての顧客の）すべてのリポジトリが1つのプロジェクトに接続されているため、製品が使用されている顧客モジュールのロジックを変更しながら、製品を静かにリファクタリングします。

結論


このようなマイクロモジュールアーキテクチャには、次の利点があります。

  • 各顧客は、彼が必要とする機能だけに接続されています。彼のデータベースの構造には、彼が使用するフィールドのみが含まれています。最終的なソリューションインターフェイスには、不要な要素は含まれていません。サーバーとクライアントは不要なイベントとチェックを実行しません。
