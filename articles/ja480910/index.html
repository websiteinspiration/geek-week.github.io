すべてのテストおよびJUnit RuleChainの基本クラス実装


前の記事（パート2.技術。アーキテクチャと技術スタック。実装の詳細と技術的驚き）に示されている自動テストの開発の概念は、すべての自動テストに一連のシステム機能が提供されるフレームワークの概念に基づいています-シームレスに統合され、開発者を有効にします自動テストは、テストクラスのビジネス実装の特定の問題に焦点を当てています。

フレームワークには、次の機能ブロックが含まれています。

  • ルール-WebDriverの初期化およびビデオテストの受信としてのテストインフラストラクチャコンポーネントの初期化およびファイナライズ。以下で詳しく説明します。
  • WebDriverHandlers-Javaスクリプトの実行やブラウザーログへのアクセスなど、Webドライバーを操作するためのヘルパー関数。静的なステートレスメソッドのセットとして実装されます。
  • WebElementsは、一般的なWeb要素またはそのグループのライブラリであり、必要なクロスファンクション機能と一般的な動作が含まれています。私たちの場合、この機能には、Webブラウザー側での非同期操作の完了の可能なチェックが含まれます。SeleniumおよびSelenideライブラリからのWeb要素の拡張として実装されます。

テスト環境の初期化。ルール


すべてのテストクラスのキークラスはBaseTestであり、そこからすべてのテストクラスが継承されます。BaseTestクラスは、以下に示すように、テストのJunit「ランナー」と使用されるRuleChainを定義します。適用されたテストクラスからルールクラスによって提供される関数へのアクセスは、ルールクラスの静的メソッドを介して実行されます。

FilterTestRunner.class - BlockJUnit4ClassRunnerの拡張で、特別なアノテーションの値によって正規表現に基づく実行可能テストの構成のフィルタリングを提供しますフィルタ（値= "some_string_and_tag"）。実装は以下のとおりです。

org.junit.rules.Timeout-テストの最大継続を制限するために使用されます。新しいブランチでテストを開始するため、最初にインストールする必要があります。

TestLoggerは、ELK分析で使用するためにテストがイベントをjson形式でログに記録できるようにするクラスです。 org.junit.runner.Descriptionからのテストデータでイベントを強化します。さらに、ELKのイベントをjson形式で自動的に生成し、その期間と結果

StandStateChecker（ Webドライバーを初期化する前にターゲットスタンドのWebインターフェイスの可用性をチェックするクラス）でテストを開始および完了します。スタンドが原則的に入手可能であることを簡単に確認できます。

WaitForAngularCreator-ウェブドライバーハンドラーを初期化して、角度の非同期操作の完了を制御するクラス。長い同期呼び出しで「特別な」テストをカスタマイズするために使用されます。

org.junit.rules.TemporaryFolder -Webブラウザーを介したファイルのアップロードおよびダウンロード操作用のファイルを保存するための一意の一時フォルダーを設定するために使用されます。

DownloaderCreatorは、ブラウザによってダウンロードされ、Selenoidビデオ機能によって記録されたファイルの一時ディレクトリへのアップロード操作をサポートするクラスです。

EnvironmentSaver- テスト環境に関する一般情報をAllureレポートに追加するクラス。SessionVideoHandler 

-ビデオテストファイルがあればアンロードし、それをアリュールレポートに適用するクラス。

DriverCreatorは、ローカル、ソレノイド、またはggr-selenoidなどの設定パラメーターに応じて、WebDriver（テストで最も重要なクラス）を初期化するクラスです。さらに、このクラスはテストに必要な一連のJavaスクリプトを実行します。Webドライバーにアクセスするすべてのルールは、このクラスの後で初期化する必要があります。

BrowserLogCatcher-ブラウザーのログから重大メッセージを読み取り、それらをELK（TestLogger）に記録し、それをAllureレポートに適用するクラス。

スクリーンシューター。 - WebDriverRunner.getWebDriver（）GetScreenshotAs（OutputType.BYTES）として失敗したテストのために、ブラウザ画面のスクリーンショットを取り、それを報告書に適用される、というクラス

AttachmentFileSaverあなたはアリュールレポートにビジネスロジックによって要求される任意のファイルのセットを添付することができますクラス-テスト。システムにアップロードまたはダウンロードされたファイルを添付するために使用されます。

失敗分類子-テストフォールの発生時に、このフォールがインフラストラクチャの問題が原因であるかどうかを判断しようとする特別なクラス。 「システムエラーNo.XXXXXXXXXXが発生しました」というタイプの特別なモーダルウィンドウ、およびタイプ404などのシステムメッセージの画面（落下後）の存在を確認します。フォールテストをビジネスクラッシュ（シナリオごと）またはシステムの問題に分割できます。拡張org.junit.rules.TestWatcherを介して機能します＃失敗したメソッド。

PendingRequestsCatcher-ドロップがアンギュラーとウェブフロントエンド間の不完全な、フリーズまたは非常に長いレストサービスによって引き起こされたかどうかをさらに分類しようとする別の特別なクラス。機能テストに加えて、負荷の高い状態で問題のあるフリーズレストサービスを特定したり、リリースの全体的な安定性を確認したりできます。これを行うために、クラスは、開いているWebドライバーを介してブラウザーで特別なjsを起動することにより、受信したフリーズされた残りのリクエストを含むすべてのイベントをELKにログインさせます。

FilterTestRunnerテストフィルタリングクラスの実装


以下は、任意のタグセットに基づいてテストをフィルタリングするためのBlockJUnit4ClassRunner拡張機能の実装です。

/**
 * Custom runner for JUnit4 tests.
 * Provide capability to do additional filtering executed test methods
 * accordingly information that could be provided by {@link FrameworkMethod}
 */
 
public class FilterTestRunner extends BlockJUnit4ClassRunner {
    private static final Logger LOGGER = Logger.getLogger(FilterTestRunner.class.getName());
    public FilterTestRunner(Class<?> klass) throws InitializationError {
        super(klass);
    }
    /**
    * This method should be used if you want to hide filtered tests completely from output
    **/
    @Override
    protected List<FrameworkMethod> getChildren() {
        //  ,        @Filter   
        TestFilter filter = TestFilterFactory.get();
        List<FrameworkMethod> allMethods = super.getChildren();
        List<FrameworkMethod> filteredMethod = new ArrayList<>();
        for (FrameworkMethod method: allMethods) {
            if (filter.test(method)) {
                LOGGER.config("method [" + method.getName() +"] passed by filter [" + filter.toString() + "]"  );
                filteredMethod.add(method);
            } else {
                LOGGER.config("method [" + method.getName() +"] blocked by filter [" + filter.toString() + "]" );
            }
        }
        return Collections.unmodifiableList(filteredMethod);
    }
 
 
    /**
    * This method should be used if you want to skip filtered tests but no hide them
    @Override
    protected boolean isIgnored(FrameworkMethod method) {

            return super.isIgnored(method);
        } else {
           return true;
        }}
