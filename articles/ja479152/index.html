npmとビルドなしのReactとVue

フロントエンドの最も人気のある2つのフレームワークと同様に、誰もがReactまたはVueに慣れ始めると、もちろんすべてのコマンドが魔法のコマンドを使用しますnpm install, npm build。そして、パパが「公衆」になった後でのみ、「必要な場所」に展開します。

しかし、npmのインストール/ビルドで「成長」したため、多くのフロントエンドの初心者開発者が知らない別の（私は変態と呼ぶでしょう）非標準的な方法があります。

しかし、ビルドする必要がなく、htmlにjsコードを挿入するだけの場合はどうでしょうか？




私の顔に唾を吐き、「I love node」という碑文で石を投げるのを待ってください...アセンブリがないと、単体テスト、速度、およびインポートコンポーネントの処理方法、ファイルの階層、そして一般的にある種のスパゲッティコードが失われることは確かです。それが判明 ...

では、なぜあなた（そして私たち）はそれを必要とし、どのように機能するのでしょうか。


先日、BPM / ERPシステムに機能を追加するための「ビジネス」からの注文がありましたが、これは本質的に高度に変更されたRedmineです。このすべての奇跡は長い間行われており、本番データやマシンなどを読み取るためのヘルパーマイクロサービスの束を備えたVPSサーバーでスピンしています。OSには触れられない...

Redmine-Ruby on Railsで記述されており、サーバーの前面全体を生成します。追加されたプラグインのすべてのフロントエンド対話はjqueryを介して行われました。これでRoRはwebpackでの作業を学習し、 "human npm"をねじ込むことができますが、これは最新バージョンであり、RubyまたはRailsの最新バージョンが存在しない古代のRubyおよびCentos 6があります。いくつかのリアクティブフォームを追加するために、ソースから収集してすべてを駆り立てるような気分ではないので、npmとアセンブリのない通常のJSのように、ReactまたはVueをRailsページテンプレートに追加する方法を探し始めました。
そしてすぐに見つかりました。

Vue npm なしのVue




Vueでは、すべてが非常にシンプルであることがわかりました。コンポーネントの開始はもちろん「奇妙」に見えますが、一般的には読み取り可能で「書き込み可能」です。

コンポーネントを含むVueの単純なページの例：
<html>
<head>
    <script src="https://unpkg.com/vue@2.6.10/dist/vue.js"></script>
    <title>Stranger Vue things</title>
</head>
<body>
<div id="vue-app"></div>
<script type="text/javascript">
const titleComponent = `<h1>{{ title }}</h1>`;

var app = new Vue({
    el: '#vue-app',
    template: titleComponent,
    data: function () {
        return {
            title: 'Stranger Vue things'
        };
    }
});
</script>
</body>
</html>

ブラウザーでダウンロードされたファイル：371 Kb、時間：590ミリ秒http-vue-loader

を使用して通常の.vue形式でVueコンポーネントをインポートし、スパゲッティコードを作成することもできます。依存関係から必要なvueが1つだけであることに非常に驚きました。これは朗報です。

React npm なしでReact


Reactを使用すると、すべてが少し複雑になりますが、それほど複雑ではありません。JSXを機能させるには、babelをインポートする必要があります。DOMを操作するには、react-domが必要です。上記がないと、reactは正常に動作しません。1つのインポートの代わりに、3つを実行する必要があります。

Reactとコンポーネントの簡単なページの例：

<html lang="en">
<head>
  <title>React Stranger Things</title>
  <script type="application/javascript" src="https://unpkg.com/react@16.0.0/umd/react.production.min.js"></script>
  <script type="application/javascript" src="https://unpkg.com/react-dom@16.0.0/umd/react-dom.production.min.js"></script>
  <script type="application/javascript" src="https://unpkg.com/babel-standalone@6.26.0/babel.js"></script>
</head>

<body>
  <div id="root"></div>

<script type="text/babel">
const rootElement = document.getElementById('root')    
class TitleComponent extends React.Component {    
    render() { 
        return (
            <h1> {this.props.title}</h1>
        );
    } 
}

function App(){
  return(
    <div>
        <TitleComponent title="React Stranger Things"/>
    </div>
  )
}

ReactDOM.render(
    <App />,
    rootElement
)
</script>
</body>
</html>

ブラウザでダウンロードされたファイル：542 Kb、時間：589ミリ秒

ここでは、Vueとは異なり、コンポーネントテンプレートを文字列として書き込む必要はありません。すべてを通常どおりに書き込むため、非常に便利で、不快感はありません。

そして、誰を選びましたか？


ブラウザがダウンロードしたjsインポートのサイズを採用した場合、勝者はVueです。しかし、これは一見しただけです。私たちは多くのコンポーネントを持っていなかったので、両方でタスクを実行しました。そして、Reactで書く方が便利でした。アセンブリを使用して書く場合、ほとんど違いがなく、インポートの違いはそれほど重要ではありません。

Preactはどうですか？


Preactは「ミニ」バージョンのreactで、わずかに高速で、重量はわずか3 KBです。私の仕事について聞いたとき、私が最初に覚えていたのはプレアクションでした。ドキュメントを開いたのは嬉しい驚きでした：React≠Preact。
PreactにはJSXがありません。コンポーネントはReactとは大きく異なります。ミニタスクの「正確な方法」で書くことを学ぶことは、非常に冗長で「高価」です。

これは松葉杖です！それらで彼を打つ


私たちはこれが松葉杖であることを知っています。しかし、タスクに基づいてそのようなソリューションを発明しなければならない場合もあります。通常の開発状況では、この実施形態でのフレームワークの使用のフロントエンドはひどい松葉杖であり、確かにまったく推奨されません。
