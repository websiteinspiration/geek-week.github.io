12 Tipps zum Implementieren von TypeScript in React-Anwendungen

TypeScript ist der Standard der modernen Front-End-Entwicklung. Laut dem Stand der JavaScript- Forschung ist TS eindeutig für Programmierer von Interesse. Laut der Umfrage für 2019 haben fast 60% der Befragten TS ausprobiert und verwenden es weiterhin, 22% haben nicht versucht und wollten studieren.

Dieser Artikel enthält eine Sammlung von Tipps zur Implementierung und Verbesserung der Verwendung von TypeScript. Die erste Hälfte der Tipps ist allgemein in Bezug auf Ansätze und Infrastruktur. Das zweite sind einige besonders nützliche Sprachfunktionen.



Ich halte mich an den Kontext einer React-Anwendung, aber einige Tipps sind in anderen Ökosystemen relevant.

Tipp 1: Fehlertoleranz


Richtige Typen sind das Vertrauen in den geschriebenen Code und die Erleichterung des Refactorings. Damit die Eingabe für Sie funktioniert, müssen Sie sich strikt auf Typprüfungen beziehen:

  1. Überprüfen Sie den Entwicklungsmodus, um Fehler sofort zu erkennen.
  2. Fügen Sie dem CI für jede Poolanforderung eine Validierung hinzu. Dadurch wird sichergestellt, dass der Hauptzweig immer die richtigen Typen hat.

Es gibt zwei Hauptmethoden zum Kompilieren von ts-Dateien: die ursprüngliche TSC und @ babel / preset-typescript. Die babel-Methode ist interessant: Typinformationen werden aus .ts-Dateien ausgeschnitten, wodurch die Assembly schnell erfolgt. Hier ist ein Einführungsartikel.

Die Typprüfung bei babel wird zusätzlich mit dem fork-ts-checker-webpack-plugin hinzugefügt. Im Entwicklungsmodus sollten Überprüfungen asynchron durchgeführt werden, um die Assembly nicht zu blockieren und das Schreiben von Code nicht zu beeinträchtigen. Bei Pool-Anforderungen sollte die Typprüfung im Gegenteil den Build stoppen, wenn ein Fehler gefunden wird.

Tipp 2: Backend-Verträge


Die Kommunikation mit dem Backend über die API selbst impliziert einen Datenaustauschvertrag. Es ist nützlich, diese Entitäten in eine separate Ebene zu unterteilen und separate Schnittstellen zu erstellen. Wir nennen sie DTO - Data Transfer Object.

Um eine solche Schicht zu implementieren, sind einige Vereinbarungen erforderlich. Hier ist ein Beispiel für die Regeln, die Sie in einem Team implementieren können:

  • Der Schnittstellenname hat das Suffix Dto: FooBarDto.
  • Befindet sich in einem speziellen Ordner.
  • Es kann sich nur mit einer API-Änderung ändern.
  • Die Daten dieser Schnittstelle werden nur zum Lesen verwendet, die Bearbeitung ist untersagt.

Vorteile dieses Ansatzes: Die Datenquelle ist auf der Schnittstelle sofort klar, es ist einfacher, nach Fehlern zu suchen und Code zu debuggen, Daten zu überprüfen. Diese Typen sind einfach zu dokumentieren und enthalten Beschreibungen und Beispiele. Mitentwickler werden Sie zur Hand haben.

Wenn Sie Tools wie Swagger verwenden , um die API zu beschreiben, erfahren Sie , wie Sie diese Typen automatisch erstellen können.

Tipp 3: Aliase für alle


Sie und Ihr Team schreiben hervorragend getippten Code - kein Problem, kein Fehler und pünktlich. Wenn ja, fahren Sie mit dem nächsten Punkt fort;)

In der Realität gibt es Versuchungen, die Eingabe zu unterlassen, insbesondere wenn TS in einem vorhandenen Projekt implementiert wird. Ich möchte explizit oder implizit welche verwenden. Es gibt ausreichende Gründe: komplexer Code, dringende Aufgabe, Unzweckmäßigkeit - Anstrengungen zahlen sich nicht für das Ergebnis aus.

Versuchen Sie, das Problem zu ignorieren, anstatt es zu ignorieren:

export type TODO_ANY = any;

Vorteile:

  • Sie betrügen nicht und schreiben keine Krücken.
  • Sie zeigen bewusst, dass es sich in Zukunft lohnt, den Code zu finalisieren.
  • Dieser Typ ist leicht zu finden, wenn Sie nach einem Projekt suchen.
  • Beliebig kann explizit verwendet werden, wenn der Typ keine Rolle spielt, zum Beispiel wie folgt:

const getArrayLength = (array: any[]) => array.length

Einschränkungen:

  • Eine zu häufige Anwendung dieser Technik ist wahrscheinlich nicht vorteilhaft.
  • Bewusstsein und Verantwortung sind erforderlich, um solche Gags in Zukunft zu beseitigen.

Sie können es auch nützlich finden: 

export type TODO_UNKNOWN = unknown;

In jedem Fall lässt der Compiler fast jede Aktion zu. Im Falle von Unbekanntem führt im Gegenteil fast jede Verwendung zu einem Fehler. Sie setzen TODO_UNKNOWN, bis der Typ des Objekts klar ist → in der Zukunft haben sie das Objekt verwendet → haben einen Fehler erhalten → haben den Typ korrigiert → Sie sind erstaunlich.

Tipp 4: Typen für Bibliotheken


Wenn Sie mit der beliebten js-Bibliothek arbeiten, suchen Sie in DefinitelyTyped nach Typen . Höchstwahrscheinlich sind sie bereits geschrieben - wir verbinden uns und genießen das Leben.

Aber was ist, wenn Sie eine interne Bibliothek haben, in der es definitiv keine Typen gibt? Versuchen Sie, sie automatisch zu generieren. Die gute Nachricht für Benutzer von jsDoc: In TypeScript 3.7 hat der tsc-Compiler gelernt, .d.ts-Typen aus .ts- und .js-Dateien zu generieren . 

Eine weitere gute Nachricht: Es gibt js -> ts Konverter. Zum Beispiel hat mir TypeStat geholfen, mit typischen React-Komponenten umzugehen.

Die schlechte Nachricht ist, dass ein gutes Typsystem wahrscheinlich nicht automatisch funktioniert. Verlassen Sie sich auf das "Skelett", das anschließend das "Fleisch" aufbauen muss.

Tipp 5: Typen über Typen


... und Typen jagen!

Es kommt vor, dass die Typen externer Abhängigkeiten des Projekts nicht geeignet sind - sie sind zu allgemein (und Sie möchten genauer sein), sie enthalten einen Fehler, die Version passt nicht - oder die Typen fehlen einfach.

Typischerweise treten solche Probleme in internen Bibliotheken auf (die auch häufig aktualisiert werden), während sie in externen Bibliotheken ziemlich selten sind. Wenn ein Problem auftritt, versuchen Sie, Typen manuell zu definieren / neu zu definieren.

Wir beschreiben den Typ für das Beispielmodul. Ich betone, dass nur der Typ neu definiert wird, die Implementierung ändert sich nicht:

declare module "example" {
    const foo: { bar: string };
    export default foo;
}

Es ist besser, eine solche Überschreibung in ein separates Verzeichnis und eine separate Datei zu verschieben, z. B. typings / example.d.ts . Verbinden Sie den gesamten Inhalt der Typisierung mit dem Feld compilerOptions.typeRoots in tsconfig:

{
    "compilerOptions": {
        "typeRoots": ["node_modules/@types", "./typings"]
    }
}

Voila - Typen zum Beispiel stammen aus der Neudefinition!

Tipp 6: Styleguide


Typ oder Schnittstelle? const oder enum? Wie werden Typen in einer Funktionskomponente beschrieben? Und im Auswendiglernen?

Wenn TS gerade in ein Team eingeführt wird, stellen sich solche Fragen ständig. Eine ähnliche Reihe von Fragen erscheint für Anfänger im Projekt: Der Sturm dauert mehrere Wochen, dann wird der übliche Codierungsstil festgelegt.

Das Formalisieren von Regeln ist eine großartige Möglichkeit, um Zeit bei Streitigkeiten und Holivars bei Poolanfragen zu sparen. Stellen Sie ein Team zusammen und erstellen Sie Ihren Styleguide.

  • Nehmen Sie beliebte Styleguides als Grundlage. Ich mag diese: React-Typoscript-Cheatsheet , React -Redux-Typoscript-Guide , Modern-Typoscript-mit-Beispielen-Cheat-Sheet.
  • Linter und Formatierer bearbeiten - lassen Sie die Maschine die Drecksarbeit erledigen.
  • Fügen Sie die Besonderheiten Ihres Projekts hinzu: typische Vorlagen, Best Practices, Worst Practices.
  • Übernehmen Sie diese Entscheidungen und beziehen Sie sich in einer Codeüberprüfung darauf.
  • Geben Sie dem Team die Möglichkeit, diese Entscheidungen zu diskutieren und zu ändern.

Ein gut erstellter Styleguide enthüllt die Wunder von CPDD ( Copy Paste Driven Development ): Entwickler schreiben im gleichen Stil und verbringen dank vorgefertigter Boilerplates weniger Zeit. Epische Kampfregisterkarten gegen Leerzeichen (sorry, Typen gegen Schnittstellen) sind nicht angeordnet , der Einstieg von Anfängern in das Projekt wird vereinfacht.

PS: Eine weitere Inspirationsquelle ist die ts-Utility-Bibliothek. In der Quelle finden Sie nicht nur nützliche Helfer, sondern auch den Code der Profis. Zum Beispiel Utility-Typen , SimplyTyped .

Tipp 8: Literal vs. Aufzählungstyp


TypeScript führt eine praktische Syntaxstruktur ein - enum . Enum wird für endgültige Aufzählungen verwendet. Nachdem ich enum gemeistert habe, möchte ich es zunächst sehr oft schreiben. Enum hat jedoch faire Kritik bezüglich des generierten Codes nach der Kompilierung.

Andererseits gibt es Literaltypen - eine einfachere und bequemere Alternative zur endgültigen Aufzählung. In einigen Fällen ist die Verwendung solcher Typen angemessener.

Ein Lebensbeispiel sind einfache Komponenten Requisiten:

//   enum
enum ColorEnum {
    yellow = "yellow",
    transparent = "transparent"
}

const ExampleEnum = (props: { color: ColorEnum }) => <div>{props.color}</div>;

//    
type ColorLiteral = "yellow" | "transparent";

const ExampleLiteral = (props: { color: ColorLiteral }) => <div>{props.color}</div>;

//   
const Example = () => (<>
        <ExampleEnum color={ColorEnum.yellow} />
        <ExampleLiteral color="yellow" />
 </>);

In diesem Beispiel überprüfen beide Komponenten die Art der Requisitenfarbe. Die Variante mit enum erwies sich als redundanter - dies ist zusätzlicher Code während der Kompilierung und zusätzliche Importe im Quellcode sowie eine umständlichere Schreibstruktur. Die Variante mit einem String-Literal ist prägnanter und enthält auch eine Typprüfung. Es macht keinen Sinn, den Enum-Code zu komplizieren.

Bevorzugen Sie String-Literale für einfache Typprüfungen. Verwenden Sie enum, um komplexere Logik zu beschreiben. In diesem Fall ist es besser, const enum den Vorzug zu geben (diese Struktur wird jedoch von babel nicht unterstützt , Sie müssen ein Plugin hinzufügen ).

Tipp 9: const Assertions


Ein weiterer Grund, Enum seltener zu verwenden, sind const-Behauptungen.

as const teilt dem Compiler mit, dass sich dieses Objekt nicht ändert. Daher können Typen detaillierter abgeleitet werden.

const loadAction = {
    start: "LOAD_START",
    done: "LOADED",
    error: "LOAD_ERROR"
} as const;

type LoadActionType = typeof loadAction;
type LoadActionKey = keyof LoadActionType; // 'start' | 'done' | 'error'
type LoadActionValue = LoadActionType[LoadActionKey]; // 'LOAD_START' | 'LOADED' | 'LOAD_ERROR'
//  "as const" LoadActionValue   string

Sie können Arrays auch als const anwenden:

const DIRECTION_ASC = 1;
const DIRECTION_DESC = -1;
const sortDirectionTypes = [DIRECTION_ASC, DIRECTION_DESC] as const;
type SortDirection = typeof sortDirectionTypes[number]; // -> 1 | -1

Es ist bequem für Überweisungen zu verwenden. Überprüfen Sie beispielsweise, ob das Wörterbuch alle Optionen beschreibt:

const sortDirectionDescription: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " ",
    [DIRECTION_DESC]: " "
};
const sortDirectionDescription2: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " "
};
// Error: TS2741: Property '[-1]' is missing // in type '{ [DIRECTION_ASC]: string; }'

PS: In diesem Artikel erfahren Sie mehr über const-Zusicherungen .

Tipp 10: Markentypen


Brand ist ein kleines Dienstprogramm zum Erstellen einzigartiger ("Marken") Typen. Typischerweise werden diese Typen auf der Basis von Grundelementen erstellt - Zeichenfolgen, Zahlen.

Beispielsweise wird beim Arbeiten mit Daten ständig die ID verwendet. Manchmal ist es äußerst wichtig, einen ID-Typ von einem anderen zu unterscheiden (z. B. die ID des Benutzers von der ID seiner Bestellung im Geschäft), obwohl beide Zeichenfolgen sind. Mit Brand können Sie nur zwei verschiedene Typen erhalten:

type Brand<T, U> = T & { __brand: U };

type UserId = Brand<string, "userId">; // "5435436" –  id 
type OrderId = Brand<string, "orderId">; // "42da-fet3-cdsl" –  id 

declare const userID = "1111" as UserId; // - 
declare function getOrderById(id: OrderId): any; // - 

getOrderById(userID);
// Error: TS2345
// Type '"userId"' is not assignable to type '"orderId"'.

Ein wenig mehr Theorie zu diesem Thema - über nominelle und strukturelle Typen - findet sich auf Habré und auch hier und hier.

Tipp 11: Diskriminierte Gewerkschaftstypen


Der schwierige Begriff diskriminierte Vereinigung ist in der Praxis leicht zu verstehen: Er schafft eine Vereinigung verschiedener Arten mit einem gemeinsamen Teil (Diskriminant). Bei der Arbeit mit einer solchen Union kann der ts-Compiler Optionen herausfiltern und den Typ angeben.

Das klassische Beispiel für die Verwendung einer diskriminierten Vereinigung passt gut auf die Reduzierschablone. Dank des allgemeinen Typfelds wird der Typ für die Nutzlast automatisch angezeigt:

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {
        case 'SET_ONE':
            return { ...state, one: action.payload}; // `payload` is string
        case 'SET_TWO':
            return { ...state, two: action.payload}; // `payload` is number
        default:
            return state;
    }
}

Tipp 12: Niemals


Nie ist eine gute Versicherung gegen Rohfälle. Nehmen Sie aus dem vorherigen Absatz, nur ohne 'SET_TWO' und mit dem Zusatz von typeguard: nie;

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {

        case 'SET_ONE':
            return { ...state,
                        one: action.payload}; // `payload` is string

        default:
            const typeguard: never = action;
            return state;
            // Error: TS2322: Type '{ type: "SET_TWO"; payload: number; }'
            // is not assignable to type 'never'.
    }
}

Der Compiler leitet den Typ für typeguard ab und kommt zu einem Widerspruch. Einerseits - dessen Typ es nicht ist, niemals. Der Ausgabetyp ist dagegen 'SET_TWO'. Typen konvergieren nicht, wir erhalten einen Fehler.

Ein weiterer interessanter (aber seltener) Fall der Verwendung von Never in React ist das Verbot des Verschachtelns oder Weitergebens von Requisiten:

type IconProps = {
    size: 's' | 'm';
    children?: never;
    foo?: never;
};

const Icon: React.FC<IconProps> = () => <div>Whatewer</div>;
    
const Example = () => (<>
    <Icon size="s"> <div /> </Icon> {/*  –  children */}
    <Icon size="s" foo="123"/> {/*  -  foo */} 
</>);

Auch nie vollständig wird für bedingte Typen ( bedingte Typen ) verwendet. Hier ist ein cooler Artikel zu diesem Thema, wenn Ihnen alle Beispiele zuvor zu einfach waren.

Fazit


Die Verwendung von TypeScript in einem Projekt ändert die Art und Weise, wie Sie Code schreiben. Es wird einfacher, den Code zu studieren und Refactoring durchzuführen. Es wird bequem, Verträge im Code zu verwenden: Sie müssen die Details der Implementierung der verwendeten Funktionen und die Form der Objekte nicht berücksichtigen - lassen Sie sie von den Schnittstellen beschreiben. 

Das Projekt erhält ein Level-up und zusammen mit neuen Vorteilen kommen neue Aufgaben - die Unterstützung des richtigen Typsystems. Ich hoffe, diese Tipps helfen Ihnen, Ihr System zu verbessern.
