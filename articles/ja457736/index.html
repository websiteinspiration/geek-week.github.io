Get best of the week

「ツールは、それが作成するシステムについて考える能力ほど重要ではありません。」マーティンクレップマンへのインタビュー



Martin Kleppmanは、ケンブリッジ大学の研究者で、CRDTとアルゴリズムの正式な検証に取り組んでいます。2017年に出版された彼の著書「データ集約型アプリケーションの設計」は、データの保存と処理のベストセラーになりました。 

ケビンスコット（MicrosoftのCTO）はかつてこう言っています。 「これは理論と実践を組み合わせたまれなリソースであり、開発者がインフラストラクチャとデータ処理システムの設計と実装についてより深く考えるのに役立ちます。」 Apache Kafkaの作成者でありCEO ConfluentのJay Krepsも同様のことを述べています。

そして、学術研究を始める前に、マーティンは業界で働き、Rapportive（2012年にLinkedInで購入）とGo Test It（RedGateで購入）の2つの新興企業を共同設立しました。

このhabrapostはマーティンとの詳細なインタビューです。ディスカッショントピックの例：

  • ビジネスから学術研究への移行。
  • 設計データ集約型アプリケーションを作成するための前提条件。
  • 人工的な誇大広告や広告ツールに対する常識。
  • CAP定理およびその他の業界エラーの不必要性。
  • 地方分権の有用性;
  • ブロックチェーン、Dat、IPFS、Filecoin、WebRTC;
  • 新しいCRDT。イザベルでの正式な検証。
  • イベントソーシングに関するディスカッション。低レベルのアプローチ。XAトランザクション 
  • Apache Kafka、PostgreSQL、Memcached、Redis、Elasticsearch;
  • 実際にそれをすべて使用する。
  • マーティンのレポートとHydra会議に参加するための基準。

インタビューは、Odnoklassnikiプラットフォーム会社のチームの主要開発者であるVadim Tsesko（@incubos）によって行われました。ヴァディムの科学的および工学的関心は、分散システムとデータウェアハウス、およびソフトウェアシステムの検証に関連しています。

ビジネスから学術研究まで


Vadim：個人的に私にとって非常に重要な質問から始めたいと思います。あなたはGo Test ItとRapportiveを設立し、LinkedInでの大規模システムの開発に長い間携わっていましたが、その後、商業開発を離れて大学で研究することにしました。これにあなたを駆り立てたきっかけを教えてください。大学で働くことの利点は何ですか、そしてあなたは何を犠牲にしましたか？

マーティン：それは非常に興味深い移行でした。私が理解しているように、かなりの数の人々が商業開発から学術研究に行くという事実のためにあなたは私の解決策に興味を持っています、そしてはるかに多くの場合反対の方向への動きがあります。大学の収益はビジネスの収益よりも大幅に低いため、これは理解できます。私は自分がどのトピックに取り組むかを自分で決めることができるという事実から研究者の地位に個人的に惹かれます。そして、トピックに取り組むことが次のトピックで利益を上げると約束していなくても、私にとって興味深く重要であると思われるものに基づいてこの選択をします6ヶ月。会社で働くすべてのものは、何らかの形で販売されなければなりません。現在、インターネットとソフトウェアの将来にとって重要なトピックに取り組んでいますが、それらについての私たちの理解は、完成した製品を作成するにはまだ十分ではありません。これまでのところ、これらのテクノロジーがどのように機能するかについての一般的な考えすらありません。これらの研究は基本的なものなので、会社ではなく大学で実施する方がよいと判断しました。大学の方が自由度が高く、10年間利益をもたらさないことを行うことができます。計画期間ははるかに広いです。

データ集約型アプリケーションの設計


Vadim：私たちは間違いなく研究のトピックに戻りますが、今のところ、あなたの本であるDesigning Data-Intensive Applicationsについて話しましょう。私の意見では、これは現代の分散システムに関する最良のガイドの1つであり、ほぼ百科事典です。今日存在する最も重要な成果がすべてリストされています。

マーティン：ありがとう、重宝してよかった。

Vadim：私たちの読者がまだそれに慣れていない可能性は低いですが、念のため、あなたが書いている分散システムの分野で最も重要な成果について説明しましょう。

マーティン：実際、この本を書いているとき、特定のテクノロジについて説明するという目標を設定していませんでした。むしろ、私はデータの保存と処理に使用されるシステムの世界のガイドを作りたかったのです。現在、膨大な数のデータベース、ストリームプロセッサ、バッチ処理ツール、あらゆる種類のレプリケーションツールなどが存在するため、この領域の全体像を作成することは非常に困難です。また、特定の問題を解決するためにデータベースが必要な場合、既存の多くの問題から1つを選択することは困難です。このようなシステムについて書かれた多くの本は、この場合、単に役に立たない。たとえば、Apache Cassandraに関する本では、Cassandraの素晴らしさについて書かれている場合がありますが、適切でないタスクについては何も言われません。したがって、私の本では、主な問題を特定しようとしますこれは、大規模なシステムを作成するときに自問する必要があります。これらの質問への回答は、現在の問題を解決するのに適している技術と、あまり良くない技術を判断するのに役立ちます。主なことは、すべてを行うことができる技術がないということです。私は、さまざまなコンテキストにおけるさまざまなテクノロジーの利点と欠点を示しています。

Vadim：実際、多くのテクノロジーには共通の特徴と機能があり、同じデータモデルを提供しています。同時に、広告を信用することはできません。システムの内部構造を理解するには、テクニカルレポートやドキュメントだけでなく、ソースコードも読む必要があります。

常識と人工的な誇大広告およびツール広告


マーティン：さらに、ドキュメンテーションはデータベースがあまり適切ではないタスクについて何も述べていないため、行の間を読む必要があることがよくあります。実際、どのデータベースにも独自の制限があるため、どのデータベースを常に知っておく必要があります。多くの場合、導入ガイドを読み、システムの内部操作を再構築する必要があります。

Vadim：はい、これは素晴らしい例です。この領域では、共通の語彙や単一の基準セットが十分になく、それを使用して1つのタスクのさまざまなソリューションを比較できると思いませんか？現在、同じものに異なる名前が使用されています。たとえば、トランザクション性の保証など、明確かつ明確に綴られるべき多くの側面はまったく言及されていません。

マーティン： ええそれはそうです。残念ながら、私たちの業界では、さまざまな楽器に非常に興奮しています。これらのツールは、自社製品の宣伝に関心のある企業によって作成されているため、理解できます。したがって、これらの企業は人々を会議に送り、本質的に、これらの製品が優れていることについて話します。これは技術レポートを装っていますが、本質的には広告です。私たちの製品の利点と欠点についてより正直であることは、業界として私たちを傷つけません。これに対する要件の1つは一般的な用語であり、これがないと、物事を比較することは不可能です。しかし、これ以外にも、さまざまなテクノロジーの長所と短所を分析する方法が必要です。

不必要なCAPの定理と他の業界の間違い


Vadim：私の次の質問はかなり敏感です。キャリアの中で遭遇した業界での一般的な間違いについて教えてください。たとえば、過大評価されたテクノロジーや、ずっと前に取り除くべきであった広く使用されているソリューションについて。これは最良の例ではないかもしれませんが、gRPC over HTTP / 2の代わりにJSON over HTTP / 1.1を使用することを思いついています。それとも、この見方を共有しないのですか？

マーティン：ほとんどの場合、システムを作成するとき、1つのことを達成するために、別のものを犠牲にする必要があります。ここでは、間違いについて話したくありません。 JSON over HTTP / 1.1と、たとえばProtocol Buffers over HTTP / 2を選択した場合、両方のオプションが存在する権利があります。プロトコルバッファを使用する場合は、スキームを定義する必要があります。これは、システムの動作を正確に判断するのに役立つため、非常に役立ちます。しかし、状況によっては、このスキームは特にデータ形式が頻繁に変更される開発の初期段階では、苛立ちを引き起こします。繰り返しますが、ここでは特定の目標を達成するために何かを犠牲にする必要があり、状況によってはこれは正当化されますが、そうでない場合もあります。本当に間違っていると言えるほど多くのソリューションはありません。しかし、これについて話しているので、CAPの定理について話しましょう-私の意見では、それからの利益は絶対にありません。システムの設計で使用する場合、CAPの定理の意味に誤解があるか、自明のステートメントがそれを利用して実証されます。非常に狭く定義された整合性モデル（線形化可能性）と非常に狭く定義されたアクセシビリティモデルを使用します。他のレプリカとの接続を確立できない場合でも、各レプリカは完全にアクセス可能でなければなりません。一方でこれらの定義は非常に正確ですが、他方では狭すぎます。多くのアプリケーションでは、一貫性やアクセシビリティのそのような定義は必要ありません。そして、アプリケーションがこれらの単語の異なる定義を使用する場合、CAPの定理は彼にとって役に立たない。だから私はそれを適用することにあまり意味がないと思います。ちなみに、私たちは業界の間違いについて話し始めたので、暗号通貨マイニングは完全に電力の浪費であることを正直に認めましょう。真剣にこれを行う方法がわかりません。

Vadim：さらに、ほとんどのストレージテクノロジーは現在、特定のタスク用にカスタマイズ可能です。障害が発生した場合、適切な動作モードを選択できます。

マーティン：そうです。さらに、テクノロジーの重要な部分は、CAPの定理の一貫性とアクセシビリティの厳密な定義に該当しません。つまり、これらはCPではなく、APではなくCAではなく、Pのみです。誰もこのソフトウェアについて直接書くことはありませんが、実際にはそれが可能です完全に合理的な開発戦略になります。これが、ソフトウェアを分析するときにCAPが害を及ぼすよりも害があると私が信じる理由の1つです。設計の決定の重要な部分は決して提示されておらず、非常に合理的な解決策である可能性がありますが、CAPではそれらを説明することはできません。

地方分権のメリット


Vadim：現在、データ集約型アプリケーションの開発で最も深刻な問題は何ですか？最も積極的に検討されているトピックは何ですか？私の知る限り、あなたは分散コンピューティングと分散データウェアハウジングの支持者です。

マーティン： はい。私の調査で証明した点の1つは、現時点ではサーバーと集中化に依存しすぎていることです。インターネットの初期の頃、それはARPANETから発展したとき、パケットをさまざまなルートに沿って送信できる非常に安定したネットワークとして設計されましたが、それらはまだ目標を達成しています。アメリカのいずれかの都市で核爆発が発生した場合、ネットワークの存続する部分は稼働し続け、故障したセクションを迂回するために代替ルートが使用されます。それは冷戦によって生み出された計画でした。しかしその後、すべてをクラウドに配置することを決定したので、今ではほぼすべてのものが米国東部のバージニア州のどこかにあるAWSセンターの1つを何とか通過しています。ある時点で、ネットワークのさまざまな部分を分散して使用するという理想を放棄し、現在すべてが依存しているサービスを特定しました。私は、データに対するより多くの制御がサービスではなくエンドユーザーに属する分散型のアプローチに戻ることが重要だと考えています。

分散化に関して言えば、銀行のような単一の中央機関が存在しない相互作用するエージェントのネットワークを持っているため、暗号通貨のようなものを意味することがよくあります。しかし、これは私が話している分散化ではありません。私の意見では、暗号通貨も非常に集中化されているためです。ビットコインとの取引を完了する必要がある場合は、ビットコインネットワークを通じて行う必要があるため、すべてがこのネットワークに集中化されています。ネットワーク構造は、単一の組織化センターを持たないという意味で分散化されていますが、各トランザクションはこのネットワークを介してのみ行う必要があるため、ネットワーク全体としては非常に集中化されています。これも一元化の一形態だと思います。暗号通貨の場合、これは避けられません。これは、2倍のコストが発生しないようにする必要があり、これはネットワークなしで達成することが難しいためです。これは、どのトランザクションが完了したかなどに関するコンセンサスを提供します。しかし、ブロックチェーンのようなシステムを必要としない多くのアプリケーションがあり、それらははるかに柔軟なデータモデルで動作できます。私が最も興味を持っているのは、これらの分散システムです。

Vadim：ブロックチェーンについて言及したので、分散システムの分野で有望またはあまり知られていない技術についてお話しいただけますか？私自身もIPFSで遊んでいましたが、この分野での経験ははるかに豊富です。

マーティン：実際、私はそのようなテクノロジーを積極的にフォローしていません。 IPFSについて少し読んだのですが、自分では使っていません。IPFSと同様に分散ストレージ技術であるDatを少し使用しました。違いは、Filecoin暗号通貨がIPFSに関連付けられていることです、データストレージの支払いに使用され、Datにブロックチェーンがアタッチされていません。 Datでは、P2Pベースで複数のマシンにデータを複製することしかできません。私たちが取り組んでいたプロジェクトにとって、Datは素晴らしいものです。ユーザーがドキュメント、データ、またはデータベースで共同作業するためのソフトウェアを作成しました。このデータの各変更は、データのコピーを持っているすべての人に送信されます。このようなシステムでは、DatをP2Pの原則に従って使用できるため、ネットワークレベルでの動作、つまりNATトラバーサルとファイアウォールの通過を確実に行うことができますが、これはかなり難しいタスクです。これに加えて、私たちはCRDTからレベルを作成しました。これにより、複数の人がドキュメントまたはデータセットを編集でき、編集をすばやく簡単に共有できるようになりました。 IPFSの上に同様のシステムを作成できると思います。Filecoinを無視し、P2Pレプリケーションのみを使用している間。 

Vadim：しかし、WebRTCがノードを相互に直接接続し、IPFSは分散ハッシュテーブルに近いため、このようなシステムの応答性は低下しませんでした。

マーティン：実は、WebRTCはわずかに異なるスタックレベルです。これは主にビデオ通話を目的としています。現在、通信に使用されているソフトウェアで使用される可能性が高いです。さらに、WebRTCは、任意のバイナリデータを送信できるチャネルを提供します。しかし、その上に複製システムを作成するのは難しい場合があります。しかし、DatとIPFSでは、このために何もする必要はありません。 

あなたは応答性について述べましたが、これは心に留めておくべき非常に重要な要素です。次のGoogleドキュメントを分散化したいとします。 Googleドキュメントでは、変更の単位は単一のキーストロークであり、新しい文字をそれぞれリアルタイムで同じドキュメントで作業している他のユーザーに送信できます。一方で、これにより迅速なコラボレーションが保証されます。つまり、大きなドキュメントを書き込む場合、何十万もの1文字の変更を送信する必要があり、多くの既存のテクノロジーはこの種のデータの圧縮に対応していません。キーストロークごとに100バイトだけを送信する必要があると仮定しても、100,000文字のドキュメントの場合、10 MBのデータを送信する必要がありますが、通常、このようなドキュメントは数十キロバイトしかかかりません。独創的な圧縮方法が発明されるまでは、そのようなデータの同期には莫大なリソースの追加コストが必要です。多くのP2Pシステムには、Googleドキュメントなどのシステムで使用できる状態のスナップショットを作成する効果的な方法がまだありません。私が現在取り組んでいるのはこの問題であり、複数のユーザーに対してより効率的なドキュメント同期のためのアルゴリズムを作成しようとしています。これは、あまりにも多くのリソースを必要とするため、すべてのキーストロークを保存しないアルゴリズムである必要があり、ネットワークをより効率的に使用できるはずです。これにより、Googleドキュメントなどのシステムで使用できるようになります。私が現在取り組んでいるのはこの問題であり、複数のユーザーに対してより効率的なドキュメント同期のためのアルゴリズムを作成しようとしています。これは、あまりにも多くのリソースを必要とするため、すべてのキーストロークを保存しないアルゴリズムである必要があり、ネットワークをより効率的に使用できるはずです。これにより、Googleドキュメントなどのシステムで使用できるようになります。私が現在取り組んでいるのはこの問題であり、複数のユーザーに対してより効率的なドキュメント同期のためのアルゴリズムを作成しようとしています。これは、あまりにも多くのリソースを必要とするため、すべてのキーストロークを保存しないアルゴリズムである必要があり、ネットワークをより効率的に使用できるはずです。

新しいCRDT、Isabelleでの正式な検証


Vadim：これについて詳しく教えてください。100倍を超えるデータ圧縮を達成できましたか？新しい圧縮技術や特別なCRDTについて話していますか？

マーティン：はい。これまでのところ、プロトタイプしかありません。まだ完全に実装されていません。それが実際にどれほど効果的であるかを見つけるために、追加の実験を行う必要があります。しかし、いくつかの方法は有望です。私のプロトタイプでは、1回の編集のサイズを100バイトから1.7バイトに減らすことができました。ただし、繰り返しますが、これは今のところ実験的なバージョンにすぎず、このインジケーターは少し変わる場合があります。いずれにしても、この領域には最適化の大きな機会があります。

Vadim：では、Hydraカンファレンスでのあなたのレポートはこれについてですか？

マーティン： はい。 CRDT、コラボレーションソフトウェア、およびこのコンテキストで発生するいくつかの問題について簡単に紹介します。それから私はこの分野で私たちが行っている研究について話します-彼らは多くの異なる問題を扱っています。アプリケーション側では、これらのアルゴリズムをJavaScriptで実装し、それに基づいて、アルゴリズムの動作をよりよく理解するための機能的なプログラムを作成します。同時に、これらのアルゴリズムのいくつかはかなり自明ではないため、これらのアルゴリズムの正確性を証明する正式な方法にも取り組んでおり、常に一貫した状態に到達するようにしたいと考えています。以前に開発されたアルゴリズムの多くは、一部の境界線のケースで一貫性を提供しません。これを回避するために、私たちは正当性を証明する正式な方法に目を向けました。

ヴァディム：このシステムにCoqやIsabelleなどの定理証明を使用していますか？ 

マーティン：はい、イザベル。

編集者メモ：マーティンは、ザストレンジループでイザベルに関する講演を読みます。

Vadim：この証拠を公開する予定はありますか？

マーティン：はい、1 年半前に公開した最初の証拠セットとCRDT検証フレームワーク。このフレームワークで3つのCRDTをテストしました。その中で最も重要なのは、テキストを共同編集するためのRGA（Replicated Growable Array）、CRDTでした。このアルゴリズムはそれほど複雑ではありませんが、むしろ自明ではありません。正しいかどうかはすぐには明らかではないため、正式な証明が必要でした。また、いくつかの既存のCRDTが正しいことを証明する作業にも取り組みました。この領域で最後に行ったのは、新しいデータモデル用の独自のCRDTを作成することでした。

ヴァディム：公式証明の量は、アルゴリズム自体のコードサイズよりどれだけ多いですか？これには時々困難があります。

Vadim：オンラインでのイベント処理に関するあなたの最近の記事についてお話ししたいと思います。私の理解する限り、あなたはイベントソーシングのサポーターです。現在、このアプローチは人気を博しており、プログラマーは、グローバルに順序付けられた操作ログの利点のために、どこにでもそれを適用しようとしています。そして、どのような状況でイベントソーシングが最善のアプローチではありませんか？人々はどこでもそれを使用しようとし、場合によってはうまく機能しないという事実のために、このテクノロジーに失望しないようにしたいと思います。

Vadim：これは素晴らしいアドバイスです。さらに、ほとんどのアプリケーションシステムは常に進化しており、その開発の方向は常に事前にわかっているわけではないため、どのようなクエリ、パターン、データストリームが現れるかを事前に知ることはできません。

マーティン：では、これらのイベント受信者の一部が既に更新されていて、他の人がまだ更新されていない場合の対処方法を知りたいのですか？

ヴァディム： はい。この状況では、ユーザーがサイトにアクセスして検索クエリを入力すると、たとえばこのエリアでアパートが利用可能であることがわかりますが、広告をクリックした後、データベースはまだイベントを受信できず、必要なドキュメントがそこにないため、404のコードを受け取ります。
