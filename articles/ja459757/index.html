Dartでのエラー処理への機能的アプローチ

新しいテクノロジーへの移行では、開発のための通常のツールが失われます。場合によっては、いくつかの技術的な制限のために欠席することを余儀なくされますが、可能であればツールを携帯します。Androidアプリケーションを開発するとき、私は例として基礎を取りましたフェルナンドセハスによって提案されたクリーンアーキテクチャ。Flutterで使用されているデザインパターンを理解した上で、このアーキテクチャを破棄してBLoCを採用することにしました。私はすぐにこのテンプレートに慣れました。これは、以前に使用したMVVMに非常に似ていますが、1つの詳細に我慢したくありませんでした。リポジトリメソッドを呼び出すとき、例外をキャッチし、それらをある型にキャストし、型に応じて必要な状態を作成する必要がありました。私の意見では、これは非常に乱雑で、フェルナンドに基づくAndroidプロジェクトで以前に使用されたEitherタイプを移植しました。


どちらも関数型プログラミング言語から来ました。これは、可能なタイプの1つの値を提供します。

Eitherの基本的な実装
/// Signature of callbacks that have no arguments and return right or left value.
typedef Callback<T> = void Function(T value);

/// Represents a value of one of two possible types (a disjoint union).
/// Instances of [Either] are either an instance of [Left] or [Right].
/// FP Convention dictates that:
///   [Left] is used for "failure".
///   [Right] is used for "success".
abstract class Either<L, R> {
  Either() {
    if (!isLeft && !isRight)
      throw Exception('The ether should be heir Left or Right.');
  }

  /// Represents the left side of [Either] class which by convention is a "Failure".
  bool get isLeft => this is Left<L, R>;

  /// Represents the right side of [Either] class which by convention is a "Success"
  bool get isRight => this is Right<L, R>;

  L get left {
    if (this is Left<L, R>)
      return (this as Left<L, R>).value;
    else
      throw Exception('Illegal use. You should check isLeft() before calling ');
  }

  R get right {
    if (this is Right<L, R>)
      return (this as Right<L, R>).value;
    else
      throw Exception('Illegal use. You should check isRight() before calling');
  }

  void either(Callback<L> fnL, Callback<R> fnR) {
    if (isLeft) {
      final left = this as Left<L, R>;
      fnL(left.value);
    }

    if (isRight) {
      final right = this as Right<L, R>;
      fnR(right.value);
    }
  }
}

class Left<L, R> extends Either<L, R> {
  final L value;

  Left(this.value);
}

class Right<L, R> extends Either<L, R> {
  final R value;

  Right(this.value);
}

実装は非常に基本的で、他の言語のソリューションよりも劣りますが、そのタスクに対応しています。私はすべてのリポジトリメソッドの結果としてこのタイプを使用し、例外処理はデータレイヤーに移動しました。これにより、コードを読みやすくするtry / catch構文が不要になります。


試す/キャッチする例
class ContactBloc {
  final ContactRepository contactRepository;

  ContactBloc(this.contactRepository);

  @override
  Stream<ContactState> mapEventToState(ContactEvent event) async* {
    if (event is GetContactEvent) {
      yield LoadContactState();
      try {
        var contact = contactRepository.getById(event.id);
        yield ContactIsShowingState(contact);
      } on NetworkConnectionException catch (e) {
        yield NetworkExceptionState(e);
      } catch (e) {
        yield UnknownExceptionState(e);
      }
    }
  }
}

abstract class ContactRepository {
  Future<Contact>getById(int id);
}

いずれかの例
class ContactBloc {
  final ContactRepository contactRepository;

  ContactBloc(this.contactRepository);

  @override
  Stream<ContactState> mapEventToState(ContactEvent event) async* {
    if (event is GetContactEvent) {
      yield LoadContactState();
      final either = contactRepository.getById(event.id);
      if (either.isRight) {
        final contact = either.right;
        yield ContactIsShowingState(contact);
      } else {
        final failure = either.left;
        if (failure is NetworkFailure) yield NetworkFailureState(failure);
        if (failure is UnknownFailure) yield UnknownFailureState(failure);
      }
    }
  }
}

abstract class ContactRepository {
  Future<Either<Failure, Contact>>getById(int id);
}

読みやすさについては、誰かが主張するかもしれません。おそらく誰かが試す/キャッチすることに慣れていて、彼自身のやり方で正しいでしょう、これは大部分がフレーバーです。追加の利点は、障害階層を定義して左側に戻ることができることです。抽象的な障害を作成し、すべてのServerFailure、NetworkFailure機能、および現在の機能の一部のContactFailure固有の機能を継承して共通にします。ブロックでは、予想される失敗を正確に把握します。


DartにFailureを実装することの欠点は、kotlinのようにシールされたクラスがないことです。それ以外の場合、キャストにifsはありません。言語は若く、活発に発展しており、ハンドラーをより簡潔に記述するためのツールが提供される時が来ることを願っています。


誰かがこの実装を好まないかもしれず、意味がないかもしれませんが、Dartでのエラー処理への機能的アプローチの可能性を紹介したかったのですが、他の言語ほどエレガントではありませんでした。
