Angular：カスタムのフォーム要素を作成し、それにフォームの状態を渡す

Webアプリケーションのさまざまなフォームは、多くの場合、同じレンガ要素から構築されています。コンポーネントフレームワークは、繰り返し可能なコードを取り除くのに役立ちます。次に、これらのアプローチの1つを検討します。したがって、Angularの慣習と同じです。

技術的なタスク：


  • コンポーネント「SNILSを入力するためのフォーム要素」を作成する必要があります。
  • コンポーネントは、入力された値をマスクでフォーマットする必要があります。
  • コンポーネントは入力を検証する必要があります。
  • コンポーネントは反応フォームの一部として機能する必要があります。
  • 不完全なフォームは、再起動の間、その状態を維持する必要があります。
  • ページをロードするとき、編集されたフォームはすぐにエラーを表示するはずです。
  • Angular Materialで使用されています。

プロジェクトの作成と依存関係のインストール


テストプロジェクトを作成する


ng new input-snils

サードパーティライブラリをインストールする


まず、Angular Material自体

ng add @angular/material

次に、チェックサムの計算規則に従って、SNILS自体をマスクおよびチェックする必要があります。

フォームデータの保存


localStorageを使用するためのサービスの準備


フォームデータはlocalStorageに保存されます。

LSを操作するためのブラウザーメソッドをすぐに使用できますが、Angularでは、普遍的なコードを作成して、すべての外部依存関係を制御下に置くことが慣例となっています。また、テストを簡素化します。

したがって、クラスがDIコンテナーからすべての依存関係を取得するときに、それは正しくなります。猫のMatroskinを思い出してください。インジェクターから不要なものを購入するには、まずインジェクターに不要なものを販売する必要があります。window.provider.ts

プロバイダーを作成する



import { InjectionToken } from '@angular/core';

export function getWindow() {
  return window;
}

export const WINDOW = new InjectionToken('Window', {
  providedIn: 'root',
  factory: getWindow,
});

何が起きてる？AngularのDIインジェクターはトークンを記憶し、トークンに関連付けられているエンティティを提供します。トークン-InjectionTokenオブジェクト、文字列、またはクラスを指定できます。これにより、新しいルートレベルのInjectionTokenが作成され、ブラウザと返されるファクトリと通信しますwindow。

ウィンドウができたので、LocalStorage storage.service.tsを操作するための簡単なサービスを作成しましょう


@Injectable({
  providedIn: 'root'
})
export class StorageService {
  readonly prefix = 'snils-input__';

  constructor(
    @Inject(WINDOW) private window: Window,
  ) {}

  public set<T>(key: string, data: T): void {
    this.window.localStorage.setItem(this.prefix + key, JSON.stringify(data));
  }

  public get<T>(key: string): T {
    try {
      return JSON.parse(this.window.localStorage.getItem(this.prefix + key));
    } catch (e) { }
  }

  public remove(key: string): void {
    this.window.localStorage.removeItem(this.prefix + key);
  }
}

StorageServiceはインジェクターからウィンドウを取得し、データを保存および読み取るための独自のラッパーを提供します。構成でモジュールを作成する方法の説明で記事をオーバーロードしないように、プレフィックスを構成可能にしませんでした。

FormPersistModule


フォームデータを保存するための簡単なサービスを作成します。

form-persist.service.ts

@Injectable({
  providedIn: 'root'
})
export class FormPersistService {
  private subscriptions: Record<string, Subscription> = {};

  constructor(
    private storageService: StorageService,
  ) { }

  /**
   * @returns restored data if exists
   */
  public registerForm<T>(formName: string, form: AbstractControl): T {
    this.subscriptions[formName]?.unsubscribe();
    this.subscriptions[formName] = this.createFormSubscription(formName, form);

    return this.restoreData(formName, form);
  }

  public unregisterForm(formName: string): void {
    this.storageService.remove(formName);

    this.subscriptions[formName]?.unsubscribe();
    delete this.subscriptions[formName];
  }

  public restoreData<T>(formName: string, form: AbstractControl): T {
    const data = this.storageService.get(formName) as T;
    if (data) {
      form.patchValue(data, { emitEvent: false });
    }

    return data;
  }

  private createFormSubscription(formName: string, form: AbstractControl): Subscription {
    return form.valueChanges.pipe(
      debounceTime(500),
    )
      .subscribe(value => {
        this.storageService.set(formName, value);
      });
  }
}

FormPersistServiceは、渡された文字列キーを使用してフォームを登録できます。登録とは、フォームデータが変更されるたびにLSに保存されることを意味します。
登録時に、LSから抽出された値も返されるため、フォームがすでに以前に保存されていることがわかります。

登録解除（unregisterForm）は、保存プロセスを終了し、LS内のエントリを削除します。

ストレージ機能を宣言的に説明したいと思います。コンポーネントコードで毎回説明するわけではありません。 Angularではディレクティブを使用して奇跡を起こすことができますが、今のところそうです。form-persist.directive.ts

ディレクティブを作成します


@Directive({
  selector: 'form[formPersist]', // tslint:disable-line: directive-selector
})
export class FormPersistDirective implements OnInit {
  @Input() formPersist: string;

  constructor(
    private formPersistService: FormPersistService,
    @Self() private formGroup: FormGroupDirective,
  ) { }

  @HostListener('submit')
  onSubmit() {
    this.formPersistService.unregisterForm(this.formPersist);
  }

  ngOnInit() {
    const savedValue = this.formPersistService.registerForm(this.formPersist, this.formGroup.control);
    if (savedValue) {
      this.formGroup.control.markAllAsTouched();
    }
  }
}

FormPersistDirectiveは、フォームにスーパーインポーズされると、ローカルインジェクターから別のディレクティブ（FormGroupDirective）を引き出し、そこからリアクティブフォームオブジェクトを取得して、FormPersistServiceに登録します。

登録キーはテンプレートから取得する必要があります。フォーム自体には固有の一意の識別子はありません。

フォームを送信する場合は、登録をキャンセルする必要があります。これを行うには、HostListenerを使用して送信イベントをリッスンします。

ディレクティブは、それを使用できるコンポーネントに配信する必要もあります。再利用されるエンティティごとに個別の小さなモジュールを作成することをお勧めします。

form-persist.module.ts

@NgModule({
  declarations: [FormPersistDirective],
  exports: [FormPersistDirective]
})
export class FormPersistModule { }

SNILSフォームの要素


どのようなタスクが割り当てられていますか？

まず、データを検証する必要があります。

snilsValidator


Angularを使用すると、バリデーターをフォームコントロールにアタッチできます。今度は独自のバリデーターを作成します。SNILSをチェックするために、外部のru-validation-codesライブラリーを使用すると、バリデーターは非常に単純になります。

snils.validator.ts

import { checkSnils } from 'ru-validation-codes';

export function snilsValidator(control: AbstractControl): ValidationErrors | null {
  if (control.value === ''  || control.value === null) {
    return null;
  }

  return checkSnils(control.value)
    ? null
    : { snils: 'error' };
}

問題は、このformControlが何であるかです。snilsErrors？これは、エラーを表示するためのカスタムパイプです。これから作成します。

snils-errors.pipe.ts

type ErrorType = 'required' | 'format' | 'snils';

@Pipe({
  name: 'snilsErrors',
  pure: false,
})
export class SnilsErrorsPipe implements PipeTransform {

  transform(control: AbstractControl, errorrType: ErrorType): boolean {
    switch (errorrType) {
      case 'required': return !control.hasError('required');
      case 'format': return !control.hasError('Mask error');
      case 'snils': return control.hasError('Mask error') || !control.hasError('snils');
      default: return false;
    }
  }
}

パイプはクリーンではありません。つまり、変更が検出されるたびに実行されます。

パイプはエラータイプパラメータを受け入れ、次の3つのタイプのエラーを検出します。

  • 「必須」-このエラーは、AngularディレクティブRequiredValidatorビルトインによるものです
  • 「Snils」-このエラーはバリデーターsnilsValidatorによるものです
  • 「マスクエラー」-ngx-maskライブラリのMaskDirectiveディレクティブからのこのエラー

そして、それはブール値を返します-そのようなエラーがあるかどうか。

次に、input-snils.component.tsコンポーネントのコード自体を見てみましょう


@Component({
  selector: 'app-input-snils',
  templateUrl: './input-snils.component.html',
  styleUrls: ['./input-snils.component.css'],
  encapsulation: ViewEncapsulation.None,
  providers: [
    {
      provide: NG_VALUE_ACCESSOR,
      useExisting: forwardRef(() => InputSnilsComponent),
      multi: true
    },
    {
      provide: NG_VALIDATORS,
      useExisting: forwardRef(() => InputSnilsComponent),
      multi: true,
    },
    {
      provide: STATE_VALUE_ACCESSOR,
      useExisting: forwardRef(() => InputSnilsComponent),
    },
  ]
})
export class InputSnilsComponent implements OnInit, ControlValueAccessor, StateValueAccessor, OnDestroy {
  public mask = '000-000-000 00';
  public formControl = new FormControl('', [snilsValidator]);
  private sub = new Subscription();

  @Input() readonly: boolean;
  @Input() placeholder = '';
  @Input() tabIndex = 0;
  @Input() required: boolean;

  private onChange = (value: any) => { };
  private onTouched = () => { };
  registerOnChange = (fn: (value: any) => {}) => this.onChange = fn;
  registerOnTouched = (fn: () => {}) => this.onTouched = fn;


  ngOnInit() {
    this.sub = this.linkForm();
  }

  ngOnDestroy() {
    this.sub.unsubscribe();
  }

  private linkForm(): Subscription {
    return this.formControl.valueChanges.subscribe(value => {
      this.onTouched();
      this.onChange(value);
    });
  }

  writeValue(outsideValue: string): void {
    if (outsideValue) {
      this.onTouched();
    }
    this.formControl.setValue(outsideValue, { emitEvent: false });
  }

  setDisabledState(disabled: boolean) {
    disabled
      ? this.formControl.disable()
      : this.formControl.enable();
  }

  validate(): ValidationErrors | null {
    return this.formControl.errors;
  }

  setPristineState(pristine: boolean) {
    pristine
      ? this.formControl.markAsPristine()
      : this.formControl.markAsDirty();

    this.formControl.updateValueAndValidity({ emitEvent: false });
  }

  setTouchedState(touched: boolean) {
    touched
      ? this.formControl.markAsTouched()
      : this.formControl.markAsUntouched();

    this.formControl.updateValueAndValidity({ emitEvent: false });
  }
}

ステータスをアイテムに渡す


ControlValueAccessorを使用する場合、次のようなニュアンスがあります。

反応型フォームは、元の状態（以下、単に「状態」）に触れています。

  • 初期値はtrueで、テンプレートからコントロール値が変更されるとfalseに変わります
  • タッチすると最初はfalseになり、コントロールがフォーカスを失ったときにtrueに変わります

それらは強制的に設定することもできますが、これはControlValueAccessor内のコントロールには影響しません。コンポーネントの場合、formControlです。

また、マットエラーエラーは、現在のコントロールに触れたときにのみレンダリングされます。復元されたフォームはすぐに検証エラーを表示する必要があるため、フォームの値がlocalStorageから読み取られた場合、FormPersistDirectiveはmarkAllAsTouchedを実行します。ただし、マットエラーはControlValueAccessorコンポーネント内にあり、formControlコントロールに依存しているため、表示されません。この独立したコントロールでは、タッチされた状態はまだfalseです。

これらの状態をスローするメカニズムが必要です。これを行うには、ControlValueAccessorの独自の類似物を作成できます。これをStateValueAccessorと呼びましょう。

まず、トークンとインターフェースを作成する必要があります。

state-value-accessor.token.ts

export const STATE_VALUE_ACCESSOR = new InjectionToken<StateValueAccessor>('STATE_VALUE_ACCESSOR');


state-value-accessor.interface.ts
export interface StateValueAccessor {
  setTouchedState?(touched: boolean): void;
  setPristineState?(pristine: boolean): void;
}

インターフェースは、それを実装するクラスが（オプションで）2つの指定されたメソッドを持つための要件を記述します。これらのメソッドはInputSnilsComponentに実装され、これらの状態を内部コントロールformControlに強制します。

次に、NgControlと、StateValueAccessorを実装するコンポーネントをバインドするディレクティブが必要です。フォームの状態が変化した瞬間を特定することはできませんが、フォームが変化するたびに、Angularはコンポーネントが変更検出サイクルを待機しているとマークすることを知っています。テストされたコンポーネントとその子孫は、ディレクティブが使用するngDoCheckライフサイクルフックを実行します。

FormStatusesDirective


ディレクティブform-statuses.directive.tsを作成し
ます。

const noop: (v?: boolean) => void = () => { };

@Directive({
  selector: '[formControlName],[ngModel],[formControl]' // tslint:disable-line: directive-selector
})
export class FormStatusesDirective implements DoCheck, OnInit {
  private setSVATouched = noop;
  private setSVAPristine = noop;

  constructor(
    @Self() private control: NgControl,
    @Self() @Optional()  @Inject(STATE_VALUE_ACCESSOR) private stateValueAccessor: StateValueAccessor,
  ) { }

  ngOnInit() {
    if (this.stateValueAccessor?.setTouchedState) {
      this.setSVATouched = wrapIfChanges(touched => this.stateValueAccessor.setTouchedState(touched));
    }

    if (this.stateValueAccessor?.setPristineState) {
      this.setSVAPristine = wrapIfChanges(pristine => this.stateValueAccessor.setPristineState(pristine));
    }
  }

  ngDoCheck() {
    this.setSVAPristine(this.control.pristine);
    this.setSVATouched(this.control.touched);
  }
}

FormStatusesDirectiveは、可能なすべてのコントロールに重ね合わされ、StateValueAccessorの存在を確認します。これを行うには、StateValueAccessorを実装するコンポーネントがチェックする必要があるインジェクターから、STATE_VALUE_ACCESSORトークンへのオプションの依存関係が要求されます。

トークンで何も見つからなかった場合、何も起こりません。setSVATouchedメソッドとsetSVAPristineメソッドは空の関数になります。

StateValueAccessorが見つかると、検出されたすべての状態変化に対して、そのsetTouchedStateメソッドとsetPristineStateメソッドが呼び出されます。form-statuses.module.ts

をエクスポートするためのモジュールをディレクティブに提供し
ます。

  declarations: [FormStatusesDirective],
  exports: [FormStatusesDirective]
})
