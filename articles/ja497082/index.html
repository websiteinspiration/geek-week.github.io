ロボット工学でのGazeboシミュレーターの使用


レポート作成者：Alexander Bogoslovsky

プレゼンテーション
スライドロボットを「ライブ」で操作できない場合があります。その場合は、シミュレータを使用する必要があります。 ROSについて話をしている場合は、最初にRazeで使用するために作成されたので、まず最初にGazeboシミュレーターについて説明します。ガゼボは、DARPAやNASAなどの著名な組織を含む、多くのロボット競技を開催しています。このシミュレーターは、ヒューマノイド、車輪付きロボット、ヘリコプター、水中ロボットなど、ほとんどすべてのソリッドステートロボットの実装に役立ちます。

Robots for GazeboはURDF（Unified Robotic Description Format）形式で作成されます-ロボットリンクとそれらの間の通信のノードを記述するXMLマークアップ。これはユニバーサル形式であり、実際のロボットをROSに配置するためにも必要です（たとえば、robot_state_publisherおよびjoint_state_publisherパッケージ）。元々はロボット全体を記述することを目的としていませんでした。したがって、モデルをGazeboにインポートすると、URDF形式はSDF（シミュレーション記述形式）に変換されます。これには、表面材料、色、センサーなど、モデリングに必要なすべての情報がすでに含まれている場合があります。ロボットをさらに配置するシミュレーションの世界には、基本的に同じSDF形式があります。そして、世界を説明するためのそのようなファイルの名前の拡張子は、「。world」の形式です。

Gazeboの多様性は、コミュニティによって作成された幅広いプラグインと、完全なオープンソースによるものです。タスクが具体的になりすぎた場合は、提供されたAPIに基づいて独自のプラグインを作成し、タスクを完了することができます。ユーザーの大規模なコミュニティは、シミュレータの公式ウェブサイトのガゼボに関する膨大な数の教育記事だけでなく、執筆にも役立ちます。ただし、独自のプラグインを作成する必要は常にありません。シミュレータの基本機能は非常に広範囲です。

このレポートでは、ガゼボの機能と特徴、およびプロジェクトでのこのシミュレーターの使用を決定するのに役立つその長所と短所を確認します。

ロボットアクションスクリプトを定義する方法としての動作ツリー


レポート作成者：Evgeny Safronov

プレゼンテーションスライド
多くの場合、ロボットの動作は複雑で多様であるだけでなく、可能な限り予測可能でなければなりません（できれば100％）。 MLメソッドに基づく統計的行動モデルでは、非常に複雑なアクションのシミュレーションが可能ですが、ロボットの行動の安全性と透明性を保証することは依然として困難です。そのような場合、エンジニアはさまざまな状態機械に頼ります。それらに代わるものは、ゲーム業界からロボット工学に登場した行動ツリー/ BTツリーです。

このビデオでは、ビヘイビアツリーの基本から始めます。ノードとリーフのタイプ、ツリートラバーサル（実行）ルールを検討し、BT作成の2つの主要なスタイル（命令型とリアクティブ型）を分析し、それらが純粋な形で使用されることはほとんどないことを確認します。機械学習の観点からBTとは何か、MLメソッドを手動で作成したツリー構造にどのように統合できるかについて説明します。OOPのクラスのようなノードテンプレートが、動作をカプセル化し、ツリーロジックの可読性を高め、個別のライブラリを作成するのに役立ちます。最後に-ROSを操作するための既製のフレームワークの概要。明確なリーダーはありません。基本的なセマンティクスからROSとの相互作用まで、多くの重要な側面に対してさまざまなアプローチがあります。ただし、BTをダウンロードしてロボット/シミュレータに接続できるようになりました！

あなたの袋の中には何がありますか？データに溺れずにバッグファイルを操作する方法


レポート作成者：Oleg Shipitko

プレゼンテーションスライド
ROSでは、bag-filesが主要なデータログ形式であることは明らかです。少なくとも一度ROSに遭遇したことのあるほとんどの開発者は、bag-fileとは何か、どのように扱うかを知っています。しかし、誰もが次のような問題について考えているわけではありません。
  • バッグファイルはどうですか？
  • その量を減らす方法は？
  • 録音と再生以外に、バッグファイルを使用して他に何ができますか？
  • メッセージの1つのタイプが変更されたが、以前に記録されたすべてのファイルを排出したくない場合はどうすればよいですか？

これらの質問やその他の質問に対する回答は、プレゼンテーションに特化しています。

私たちのスピーチでは、それぞれに答えようとしました。
レポートはいくつかの部分に分かれています。最初に、bag-filesとその内部構造を格納するためのフォーマットが検討されます。

レポートの2番目の部分では、記録ファイルの機能について説明します。 rosbag recordコマンドの自明ではない、めったに使用されない引数が強調表示されています。バッグイメージファイルへの書き込みの機能が考慮されます。 bag-fileの圧縮に使用されるアルゴリズムについて説明します。

記録されたファイルの単純化と検査に重点を置いた部分では、bag-fileの内容を表示し、それに含まれるトピックを見つけることができるコマンドが考慮されます。特に注意が必要なのは、rqt_bagファイル検査ツールとその主な機能です。

バッグファイルに記録されたデータは、さまざまな分類アルゴリズムのテストに使用できます。この場合、データをマークアップするタスクが発生します-データセットを、データセットに対して正しいクラスを定義する特定のタグと照合します。このような問題を解決するためのツールの例として、レポートではパッケージrosbag-sliding-windows-annotatorを検討します。これにより、数値データに注釈を付けるための基礎として視覚トピックのデータを使用できます。同時に、マークアップに必要なクラスがユーザーのjson-fileから読み込まれるため、パッケージは多くのマークアップタスクに合わせて構成可能になります。

多数のバッグファイルを保存する場合、いくつかの基準を満たすファイルをすばやく見つけるというタスクが発生します。モバイルロボットの場合、そのような基準は、たとえば、特定の天気、移動ルート、ファイル内の特定のイベントの有無（たとえば、ロボットのパスをブロックしている人）などです。必要な基準のセットを満たすファイルをすばやく検索するために、ファイルに注釈を付けることができます-クイック検索に必要なデータを含む特定の圧縮された量の情報を各ファイルにマップします。レポートでは、Visiletlect Service社が使用するバッグファイルのアノテーションへのアプローチについて説明します。各バッグファイルを画像にマッピングして、多数のファイルに関する情報をすばやく表示および比較できるようにします。

レポートの最後の部分では、bag-filesをROSに移行するための標準ツールの概要について説明します。ファイル内の1つ以上のメッセージが変更されたが、以前に記録されたバッグファイルを保存して使用したい場合は、移行が必要です。この場合、古いメッセージに新しいフィールドを追加する（または既存のものを削除する）ことで、古いファイルを上書きできます。このプロセスを簡素化するために、ROSは、レポートで簡単にカバーされる既成のツールを提供しています。
