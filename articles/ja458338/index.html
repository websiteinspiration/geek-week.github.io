アプリケーションセキュリティマネージャー。開発者かセキュリティか？

組織で最も成功した攻撃は、ソフトウェアの脆弱性とブックマークを通じて実装されます。幸いなことに、ソフトウェアの脆弱性スキャナーは、企業ではまだ何か珍しいものとは見なされていませんが、保護インフラストラクチャの必要な要素と見なされています。少量の開発で現状のスキャナーを使用できる場合、大量の場合はプロセスを自動化する必要があります。しかし、誰がそれを管理するべきでしょうか？リリースをチェックする頻度を決定しますか？脆弱性検証？リリースを拒否するかどうかを決定し、脆弱性を修正するためのコードを送信しますか？そして、他の多くの質問に答えます。これは、安全なソフトウェア開発のマネージャーであるApplication Security Managerが最前線に立つ場所です。



しかし、そのような珍しい鳥をどこで見つけるか、それを自分で育てる方法は？Raiffeisenbank JSCのアプリケーションセキュリティマネージャーであるArtem Bychkovと、Rostelecom SolarのソーラーappScreenerの責任者であるDaniil Chernovは、ロシア企業の開発慣行によって規定されたApplication Security Managerの要件について説明しています。

Application Security Managerとは


組織は遅かれ早かれ、そのような人をチームに雇う必要性に気づきます。特に、社内で利用できるスペシャリストがこの役割に直接ふさわしいわけではないためです。開発者？彼らの実務経験はソフトウェアの開発と関連しています-発見された脆弱性を情報セキュリティリスクに変換することは非常に困難であり、さらにはビジネスリスクに変換することは非常に困難です。警備員？複雑な開発に没頭することは彼らにとって問題です。脆弱性を検証するには、さまざまな言語のソフトウェアコードを理解できなければならず、深刻な開発経験が必要です。

安全な開発プロセスの実装中に発生する、アプリケーションセキュリティマネージャー（AFM）によって処理されるタスクを見てみましょう。

読者は、AFMの作業は、安全要件に準拠するためのソフトウェア開発の検証にのみ関連しているという意見を持っている可能性があります。しかし、セキュリティの問題は、設計から生産的な導入まで、システムのライフサイクルのさまざまな段階で発生します。安全な開発サイクルを構築するためのさまざまなモデル（ソフトウェアセキュリティタッチポイント、SDLCなど）と、これらのプラクティスをプロセスに組み込むためのさまざまな方法があります（使用するアプローチに応じて、ウォーターフォール、アジャイル）。ただし、すべての人が重要な点について同意します。システムのライフサイクルのすべての段階でセキュリティについて考える必要があります。

明らかに、多かれ少なかれ大規模なプロジェクトのフレームワークでは、1人の人がすべての段階で作業を実行できる可能性はほとんどありません。アプリケーションのセキュリティ要件を作成し、そのアーキテクチャのレビューを実行し、アナリストの作業の結果を検証し、コードセキュリティ監査を実施し、必要なアプリケーションのセキュリティテストがテスト中に実行され、システムが安全に展開され、正しく構成されていることを確認できるのは、だれもがまれです。さらに、多くの場合、これらの活動はさまざまなチームやユニットの代表者によって行われます。メカニズム全体が正しく機能するためには、プロセスの原動力がAFMである必要があります。 AFMのタスクは、単独で、または特定のタスクを専門のスペシャリストに委任することにより、安全な開発手法の実装を保証することです。しかし、私たちの実践に基づいて、担当者にタスクを配布し、AFMで月桂樹で休むだけでは機能しません。

AFMの要件は何ですか


まず、彼は彼が伴うプロジェクトを理解する必要があります。これは、ウォーターフォールモデルとは異なり、リリース前にレビューを実施する2か月がない場合に、アジャイル開発で特に重要です。それは、AFMに依存します。たとえば、設計段階で形成された要件がチームによってどのように解釈されるか、それらがアーキテクチャにどのように当てはまるか、それらが一般に実現可能かどうか、そして将来的に深刻な技術的問題を引き起こすかどうかによって異なります。ほとんどの場合、AFMは、自動化された機器と第三者による監査のレポートの主要な消費者、解釈者、および評価者です。無関係で誤った結果を除外し、リスクを評価し、例外の管理と代償措置の開発のプロセスに参加するのはAFMです。



これは人生の例です：監査またはソースコードスキャナーは、プロジェクトで安全でないハッシュ関数（MD5）の使用を明らかにしました。同社のポリシーでは、この機能は使用できないと正式に主張しており、ベンダーは3か月で大量の機能をより安全な機能に置き換えることに同意しています。ニュアンスは、この場合、関数が整合性を保護するために使用されなかったため、衝突に対するハッシュ関数の不安定性がシステムのセキュリティに影響を及ぼさなかったことでした。この場合の正式なアプローチと、ある機能を別の機能に置き換えることにより、プロジェクトが生産的で多大なコストになるまでの時間が大幅に遅れ、セキュリティの向上がゼロになりました。

第二に、AFMには、前者に加えて、さまざまな分野の知識が必要です。開発プロセスと情報セキュリティの原則を理解する必要があります。 「ハードスキル」も重要です。専門のスペシャリストと自動化ツールの作業結果を批評的に評価することは非常に難しいため、コードを読み取れないと、脆弱性を悪用する可能性のある方法を理解できません。ほとんどの場合、コード分析またはペンテストレポートに重大な脆弱性が現れる状況に直面していましたが、開発者はこれに同意せず（原則として、安全なシステムを作成することも望んでいます）、監査人がこれを操作できなかったことを示しています脆弱性。同様の状況で誰が正しいかを評価する方法は？技術的なスキルがなければ、論争を客観的に解決することは困難です。安全なソフトウェアを開発するプロセスが外部組織の手によって、および/またはサービスモデルに従って構築されている場合、「テクニカル」プラクティスのパフォーマンスを誰がどのように評価するか？

もう1つの人生の例：新しい開発ツールが導入され、そのパフォーマンスが参照プロジェクトで確認された後、商用運用に移されます。プロジェクトは徐々に接続され、美しい緑色のダッシュボードが描かれます...そしてここで情報セキュリティインシデントが発生します。結局のところ、使用された「穴」は動的解析の段階で検出されているはずです。しかし、これは起こりませんでした...誰も見ていなかったためですが、通常、優れた結果を生み出すこのトップエンドの脆弱​​性スキャナーは、新しいJavaScriptフレームワークを使用してSPAアプリケーションでどのように機能しますか。動的に生成された認証フォームを「確認」して必要なチェックを行うことができないことがわかりました。そして、すべてがうまくいったので、誰もそれに注意を払いませんでした。開発者はスキャナーの機能の詳細を掘り下げる必要はありませんでした。これに注意を引くために、そして警備員はウェブ開発への異なるアプローチの間に重大な違いを見なかった。

そのような専門家をどこで得るか




市場を研究した人々は、アプリケーションセキュリティの専門家の深刻な不足に直面しているに違いありません。通常、シナリオは次のとおりです。内部顧客が候補者の要件を作成し、それらをスタッフに転送します。要件が厳しい場合、無料の検索結果によると、既製のスペシャリストがパブリックドメインで履歴書を投稿することは非常にまれであるため、会社は候補をまったく受け取りません。彼らが転職した場合、これはほとんどの場合、既存の連絡先を通じて簡単かつ自然に起こります。になる方法
