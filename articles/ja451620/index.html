Stravaヒートマップカードの自動認証



この投稿は、最近の同様の記事への返信です。ここで説明されているアクションを自動化する方法を教えてみましょう。そして、このヒートマップをスマートフォンのナビゲーターに接続する方法。

この記事では、Stravaヒートマップマップが観光客やサイクリストにどのように役立つか、そしてオスマンドの例を使用してスマートフォンのナビゲーションアプリケーションをそれらに接続する方法を検討しました。実際のところ、マップの各部分をロードするには、許可データを使用してパラメーターを指定する必要があります。そのように：

GET https://heatmap-external-{abc}.strava.com/tiles-auth/all/hot/{z}/{x}/{y}.png?px=256&Signature={CloudFront-Signature}&Key-Pair-Id={CloudFront-Key-Pair-Id}&Policy={CloudFront-Policy}

このデータを取得するために、デスクトップブラウザからStrava Webサイトにログインすることが提案されました。その結果、Cookieがブラウザに表示されます。必要な行を見つけて、リクエストアドレスにコピーする必要があります。次に、手動でOsmandにリクエストを入力し、彼の助けを借りて地図をダウンロードできるようにします。

ただし、認証データはすぐに古くなり、上記の手順を数日ごとに繰り返す必要があるため、この方法はあまり便利ではありません。この記事では、上記の方法を自動化する方法を説明します。

中間リンクを追加する


ユーザーがCookieデータが期限切れになるたびにスマートフォンのリクエストURLを置き換える必要がないように、変更されていないURLを指定する必要があります。サーバーアプリケーションへのリンクを提供しました。このアプリケーションは、指定されたパラメーターに応じて、ユーザーをさまざまなアドレスにリダイレクトします。

https://anygis.herokuapp.com/Tracks_Strava_All/{x}/{y}/{z}

簡単に言うと、このサーバーアプリケーションの説明は省きます。彼の主な行動についてのみお話しします。

最大12のズームでタイルが要求された場合（Stavaは許可なしにタイルを提供します）、ユーザーは直ちにパブリックURLにリダイレクトされます。

https://heatmap-external-a.strava.com/tiles/all/hot/10/619/318.png

そうでない場合は、チェックが実行されます。迅速なアクセスのために、アプリケーションはデータベースに最新の有効バージョンのCookieを保存します。要求を受け取ると、このファイルを解析して、すべてのパラメーターが設定されたURLを作成します。

その後、このアドレスにHEADリクエストを送信して、利用可能かどうかを確認します。ステータスコード「200 Success」が返された場合、Cookieはまだ機能しています。アプリケーションはユーザーをこのアドレスにリダイレクトするだけで、地図が美しく読み込まれます。

ただし、コード「401 Unauthorized」が届いた場合、Cookieは古くなっているため、再度取得する必要があります。この場合、アプリケーションはスクリプトを実行して認証データを取得します。

自動ログイン


最初に頭に浮かぶのは、Strava APIを使用してログインすることです。残念ながら、そこから必要なパラメータを引き出すことができませんでした。そこで、回り道をすることにしました。

これを行うために、リモートコントロール付きのヘッドレスChromeブラウザーを使用して、承認ページを「物理的に」ダウンロードし、そこでユーザー名とパスワードを入力して、ボタンをクリックするようにスクリプトを作成しました。

これを行うには、Node.jsスクリプトを使用してブラウザーを制御できるPuppeteerライブラリがあります。構文はこの記事で詳しく説明されています。それに慣れることをお勧めします。

それを読んだ後、残っている唯一の問題は、スクリプトを実行する対象です。すでにNode.js開発者としての経験がある場合は、このセクションをスキップできます。残りについては、Apify.comの既製のサービスを使用することを提案できます。これにより、サーバーを作成して構成する必要がなくなります。私たちのタスクでは、無料のアカウントで十分です。

const Apify = require('apify');

Apify.main(async () => {

  // 

    //     JSON
    const input = await Apify.getInput();
    if (!input || !input.email || !input.password) throw new Error('Invalid input, must be a JSON object with the "email" and "password" field!');

    //    
    const browser = await Apify.launchPuppeteer();



  //  1 -    cookie

    //   
    const page1 = await browser.newPage();
    await page1.setViewport({width: 1280, height: 1024});

    //      
    await page1.goto('https://www.strava.com/login', {waitUntil: 'networkidle2'});

    //   html-  /     
    await page1.waitForSelector('form');
    await page1.type('input#email', input.email);
    await page1.type('input#password', input.password);

    //  ,  , 
    //   ,      
    await page1.waitFor(200);

    //       
    await page1.evaluate(()=>document
      .querySelector('button#login-button')
      .click()
    );

    //      cookie 
    await page1.waitForNavigation();
    const sessionFourCookie = await page1.cookies();



  //  2 -   cookie     

    //    
    const page2 = await browser.newPage();

    //      ,   .
    //  ,        cookie
    await page2.setCookie(...sessionFourCookie);
    await page2.goto('https://heatmap-external-a.strava.com/auth');

    //     cookie
    const cloudfontCookie = await page2.cookies();



  //  
    
    //

    //    cookie   
    return cloudfontCookie;

最終段階


スクリプトを実行すると、認証データを含むCookieが返されます。サーバーアプリケーションはそれをデータベースに保存し、それ以降のすべてのStravaカード要求に使用します。 Cookieが古くなり、この手順を繰り返す必要がなくなるまで。幸い、今ではすべてが自動的に行われます。ユーザー側の不要なアクションなし。

悲しいかな、すべてがそれほどスムーズではありません。この方法には弱点があります。これが作業速度です。実際のところ、サーバーの起動、リモートブラウザーの読み込み、2つのWebページの読み込み、およびそれぞれの承認には時間がかかります。私の見積もりによると、1分以上が出てきます。

この間、ナビゲーションアプリケーションは、タイムアウトのためにダウンロードを終了します。そのため、ユーザーにエラーコード401をすぐに返して、もう一度バッテリーを植えないようにすることをお勧めします。

これを行うために、サーバーアプリケーションは、Cookieデータが古いことを認識したときにフラグを設定します。その後、スクリプトを実行するリクエストを送信し、すぐにユーザーにエラーコードを返します。つまり、サーバーは1分間使用できなくなります。

また、新しいCookieを含む応答がスクリプトからのものである場合、アプリケーションはそれらをデータベースに保存し、フラグをクリアします。その後、サーバーは再び正常に動作し始め、マップをダウンロードするためのページにユーザーを即座にリダイレクトします。
