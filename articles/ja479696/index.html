テーブルの変更、またはそれを行わない方法に関する一言


これはおそらく記事ではありませんが、MySQLで大きなテーブルを操作するいくつかの機能についての短いメモです。

執筆の理由は、表に新しい列が毎日追加されているように思われたためです。しかし、すべてが予想されたほど単純ではないことがわかりました。

そこで、ある晩、大切なお客様の邪魔にならないように、テーブルに列を追加する必要がありました。

より明確にするために、テーブルとベースの特性：

  • テーブルサイズ110Gb
  • 行数：750万
  • ストレージエンジン：InnoDB
  • マスタースレーブスキームに従って接続された2つのSQLサーバーがあり、マスターはSSDにあり、スレーブはHDDにあります

列を追加するための明白な解決策であるように見えます-テーブルの変更。

alter table table_name add source varchar(32)

私たちはそれを使用しました（そう、私たちはそれが悪いことを理解していましたが、この特定のケースではリスクは最小限でした）。

結果はかなり不快でした：

  • ウィザードでは、列を追加するプロセスに約1時間かかりました（！）
  • スレーブでは、マスターでのプロセスの終了後に始まり、約8時間続きました（!!）
  • 変更テーブル中に、スレーブでデータ複製（!!!）が完全に停止しました

しかし、銀の裏地があります。小さなボーナスは、列を追加した後、テーブルのサイズが10％減少したことです。

バトルテーブルでこれを行う人々にはどのような問題が待ち受けていますか？

まず、Alter Tableの期間中、テーブルにデータを書き込むことはできません（ただし、読み取ることはできます）。実際、それはMySQLのバージョンに依存しますが、後者はそうではありませんが、トラブルを回避するために、バージョンが正確に何をすることができるかを理解する必要があります。

したがって、テーブルが大きい場合、利用不可の時間が大きくなります。（私たちと同じように、SSDを使用すると、1時間かかり、通常のディスクでは8時間かかりました）、これは顧客が期待することはほとんどありません。

次に、私たちの場合と同様に、Alter Tableの実行中、変更したテーブルだけでなく、すべてのテーブルの同期がスレーブで完全に停止しました。したがって、2番目のサーバー上のデータが重要であり、最新である必要がある場合は、更新を行わずにすべての結果を残すリスクがあります。

列を追加するときに発生したもう1つの明白でないポイント（ただし、これはまた別のタイミングでした）- 追加のディスク領域が必要です。

実際のところ、テーブルにいくつかの変更を加えると、テーブルが最初から再作成されるため、既存のテーブルと同じスペースが必要になります。大きなテーブルの場合、それぞれ、穏やかに置くために多くのスペースが必要です。ドキュメントによると、一時テーブルはオリジナルと同じディレクトリに作成されます。

さらに、すべての種類の変更テーブルの実行中に、すべての変更がログファイルに書き込まれるため、変更後、操作が実行されていた時間にデータをロールオーバーできます。また、ここでも不愉快な驚きが待っている可能性があります。テーブルが長期間変更され、操作のボリュームが大きい場合、ディスク領域が終了するだけでなく、SQL設定で指定されたファイルサイズの制限を超える可能性があります。いずれの場合も、「オンラインDDL操作は失敗し、コミットされていない同時DML操作はロールバックされます」と表示されます。

一時ファイルのディレクトリが小さいため、innodb_tmpdirを再定義する必要があったという事実に直面しました。

変数が現在指している場所を確認するには、次のようにします。

select @@GLOBAL.innodb_tmpdir;

一時ディレクトリのサイズには、テーブル+インデックスのサイズも必要になる場合があることに注意してください。一般に、スペースを確保してください。

ドキュメントを繰り返さないために、https：//dev.mysql.com/doc/refman/5.7/en/innodb-online-ddl-space-requirements.htmlで詳細を読んでください。

しかし、どのように行うのですか？実際、すべての場面に対応できる単一のレシピはありません。

更新に重要ではないテーブルに対して行うように、可能なオプションの1つ：

  • 目的の構造で新しいテーブルを作成します
  • 古いテーブルのフィールドに入力します
  • 古いテーブルを削除または名前を変更する
  • 新しい名前を変更

これは重要ではない更新テーブルでも機能することを繰り返します。同時に、レプリケーションのブロックを回避します。新しいテーブルへの入力は、レプリケーションを続行できるように行う必要があります。また、テーブルは連続して実行されるため、単一のSQL式では実行できないため、複数の小さなクエリに分割して、その間に各データを複製する必要があることに注意してください。他の場合では、他のオプションが可能です。おそらく誰かがコメントで共有します。

UPD。ユーザーArheopsレプリケーションの問題を解決するには、並列レプリケーション/ gtidを有効にすることをお勧めします。

さて、偶然にも、時には、テーブルの大きさとその中の行数を理解するために、教える方法を行う必要があります

select count(*) from table_name

ただし、ロードされた大きなテーブルでは、特に50万行以上の場合、これも最速の操作ではありません。

したがって、ボリュームのおおよその見積もりについては、次の方法を使用できます。

SHOW TABLE STATUS FROM express where name='table_name'

残念ながら、InnoDBエンジンでは、結果のサイズが50％異なる場合があります（この場合、上の表では、実際のレコード数は約750万であり、この方法では500万しか示されていません）。

これですべてです。このメモが、おそらく害のないSQLコマンドによる大きな問題を回避するのに役立つことを願っています。
