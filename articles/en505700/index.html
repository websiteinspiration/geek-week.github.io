I will interview Java programmers. One set of questions for any level

In my career I had to be a developer, and a manager, and even once ‚ÄúCTO‚Äù in a small startup. And, of course, I had to conduct a large number of interviews with Cadidates. Therefore, for myself, I have developed such rules for conducting technical interviews:

  • ask all candidates the same questions,
  • questions should be such that both junior and senior answered them, but in different ways,
  • after three questions, the candidate‚Äôs level should be understood.

And so I got these three questions for Java developers.

Tell us about HashMap


Yes, yes, the very standard question with which 50% of interviews begin. Why am I asking him? Because it can be answered very, very differently.

  • The ideal answer begins with the statement of the problem. Why do we need such a structure? What kind of task does it solve? What does the trivial solution to this problem look like ‚Äúhead on‚Äù and why is it bad?
  • "Bonus points" for a historical excursion, knowledge of the author of the solution, at least the approximate year of its appearance (not in Java, of course)
  • The answer of at least the junior must include ways to use the structure. What methods does she support.
  • The middle-developer is required to understand the internal device, as well as a description of the contracts of the hashCode () and equals () functions, their functional requirements
  • Bonus points for describing non-functional requirements for hashCode () and equals ()
  • Bonus Points for considering using HashMap in a multi-threaded environment.
  • "Bonus points" for considering HashMap alternatives - listing them, comparing the complexity of implementation, use, speed in general, ideal and worst cases.
  • "Bonus points" for HashMap alternatives in the form of separate system components (for example, databases and "external" caches).
  • ‚ÄúBonus points" for describing how HashMap is used in databases, and why binary tree-based indexes are usually used, rather than HASH indexes.

As you can see, a very large number of "bonus options", which you can talk with senior'om. And how many points can you talk about for at least 10 minutes?

If you get bored, replace HashMap with TreeMap.

Write SQL ...


Yeah, and again the "standard" question, which 10 years ago was at every interview. Now it is less common, apparently due to the fashion for nosql. But the ability to work with data is always necessary, and programmers who understand the Cartesian product are always more useful (and more expensive) than those who are limited to general retelling of the textbook.

And the question is:
Let there be two tables in the database: students and groups. Each student has an identifier for the group_id group to which he belongs. Write a SQL query without using nested, listing the groups (or the number of groups) without a single student

The correct answer will, of course, contain not just JOIN, but LEFT JOIN with comparing the value to NULL. The most interesting thing will be to talk with the candidate about what mistakes can be made when writing a request, and why the request will not work if made. Why can not be compared with NULL through equality? Why won't a simple JOIN work?

A bonus for indexes that will speed up the operation of this query and for considering SQL options in different dialects (different DBMSs).

Find errors in the code


The candidate is voiced the task like this:
Suppose you get a Pull Request (Merge Request) from another developer for a code review. This request has a large number of different files, but one of them looks as follows (the entire class code is given):

package ru.mycompany.utils;

/** Wrap around Lock to simplify interface */
class LockWrapper {
    private Lock lock;
    public wait() {
        this.lock.wait()
    }
}

Will you have comments on this code, and if so, which ones? Will you skip the request with such a file further (to the master branch), or will you require a redo, and why?

Of course, there are at least three levels here.

  • Any junior should point out the syntax errors that are in this code. The developer must indicate why the code does not compile. And the problem is not only in the syntax, of course.
  • The developer must know why even if the code compiles, it will not work. And how to correct this code so that it works. The developer must be able to use both the synchronize construct and the classes from the java.util.concurrent package, and know why one second is not third.
  • An ideal developer should start by asking why this code is needed, what problem he solves, and whether he can solve it even if he fixes all existing problems in this code.

In the process of answering, together with the candidate, we bring the code to the ‚Äúideal‚Äù form. The final form, of course, depends on the qualifications of the candidate. The best of my candidates found 13 errors here. Apart from the ‚Äúmost important mistake‚Äù.

As a bonus, you can talk about what means of automatic code control should be used to find at least part of the errors present.

Instead of a conclusion


One advice that I want to give to all those who have to interview future colleagues. Do not try to answer whether the person is suitable or not. Formulate your opinion either in the form of a grade (if your company has adopted it), or in the amount of money that you are willing to pay the candidate to work in your company. This, firstly, simplifies the decision for you, because it will not be black or white, and secondly - to someone who will make the final decision or compare the results of several interviews.
