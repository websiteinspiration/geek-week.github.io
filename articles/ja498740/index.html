インデックスの拡大


狡猾な開発者は、インデックスキーを一意にする必要があることを理解しています。最も簡単な方法は、意図的に一意のフィールドを使用してそれを拡張することです。PKは次の場合に最適です。

CREATE UNIQUE INDEX ON events(ts DESC, id DESC);

そしてリクエストは変化します：

SELECT
  ...
ORDER BY
  ts DESC, id DESC
LIMIT 26 OFFSET $1;

＃3 インデックスのクリーニング


ある日、DBAが非効率なインデックスの検索に関する記事を読んだところ、「最後ではあるが最低ではない」タイムスタンプが適切でないことに気付いたためです。そして、彼は再びあなたに来た-今、このインデックスは、それにもかかわらずに引き返すべきであると考えて(ts DESC)。

しかし、ページ間でレコードを「ジャンプ」するという最初の問題をどうするか？..そして、すべてが単純です-無制限の数のレコードを持つブロックを選択する必要があります！

一般に、「正確に26」ではなく「26以上」を読むことを禁じているのは誰ですか？たとえば、次のブロックに明らかに異なる値のレコードtsがあるように、ブロック間の「ジャンプ」に問題はありません！

方法は次のとおりです。

SELECT
  ...
WHERE
  ts < $1 AND
  ts >= coalesce((
    SELECT
      ts
    FROM
      events
    WHERE
      ts < $1
    ORDER BY
      ts DESC
    LIMIT 1 OFFSET 25
  ), '-infinity')
ORDER BY
  ts DESC;

ここで何が起こっているのですか？

  1. 25件のレコードを減らし、「境界」値を取得しますts。
  2. そこにすでに何もない場合は、NULL値をに置き換えます-infinity。
  3. 受け取った値tsとインターフェースから渡されたパラメーター$ 1（前の「最後の」描画された値）の間の値のセグメント全体を減算します。
  4. エントリが26未満のブロックが返された場合、それは最後のブロックです。
