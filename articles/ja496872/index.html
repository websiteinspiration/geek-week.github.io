新しいハイパーレジャーファブリック：Go SDKソフトウェアモデル



では前の記事、私たちはHyperledgerファブリックでcodecodeの新しいプログラミングモデルについて話しました。新しいGo SDKのリリースはまだ行われていませんが、今後何が待ち受けているかを確認したり、テストビルドでテストしたりすることさえできます。すべてがまだ実装されているわけではありませんが（コミットハンドラーなど）、ソフトウェアモデルの計画された設計は既に明確になっているので、詳しく見ていきましょう。

このリポジトリでは、前の記事で見たチェーンコードで始まるようにサンプルグリッドを少し変更し、新しいAPIを使用して記述された単純なクライアントを追加しました。リポジトリには、ネットワークを起動および構成するための簡単な手順があります。go.modで

確認できるように、新しいAPIを使用するには、リリースではなく（リリースはまだありません）、特定のコミットに対するすべての変更（これはリリースのリリース前の一時的な措置 です）を使用します。アプリケーションでは、app.goにゲートウェイパッケージをインポートします。 新しいAPIの本質すべてのSDK（go、node、java）に共通の抽象概念を提供し、開発者がプロ​​ジェクト間で転送するボイラープレートを削減します。そのような抽象化の1つがウォレットです。

require github.com/hyperledger/fabric-sdk-go v1.0.0-beta1.0.20200404205634-0e550fc1a918



"github.com/hyperledger/fabric-sdk-go/pkg/gateway"

。ウォレットの初期化は次のようになります：

wallet, err := gateway.NewFileSystemWallet("wallet")
if err != nil {
	fmt.Errorf(err.Error())
}

err = wallet.Put("admin", gateway.NewX509Identity("Org1MSP", string(cert), string(privKey)))
if err != nil {
	fmt.Errorf(err.Error())
}

WalletはWalletStoreインターフェースの実装です。

type WalletStore interface {
	Put(label string, stream []byte) error
	Get(label string) ([]byte, error)
	List() ([]string, error)
	Exists(label string) bool
	Remove(label string) error
}

ご覧のとおり、このインターフェイスはユーザーIDリポジトリの基本的な操作を説明しています。ファイルシステムに加えて、IDはメモリに格納でき、別の組み込みファクトリであるNewInMemoryWallet（）を使用するだけです。

ネットワークに接続します。

gw, err := gateway.Connect(gateway.WithConfig(config.FromFile("./../connection-org1.yaml")),
gateway.WithIdentity(wallet, "admin"),
gateway.WithDiscovery(true),
gateway.WithCommitHandler(gateway.DefaultCommitHandlers.OrgAny))

gateway.Connect（）は、ネットワーク接続を提供するオブジェクトであるGatewayへのポインタを返します。gateway.WithCommitHandler-コミットイベントを待機する戦略を設定するための機能オプション。これまでのところ、戦略は実装されていません（つまり、「古い」SDK のデフォルトの戦略である「すべての承認者からのコミットを待つ」が使用されます）が、以下が計画されています。

  • gateway.DefaultCommitHandlers.OrgAll-組織のすべてのピアでコミットを待機します（デフォルト）
  • gateway.DefaultCommitHandlers.OrgAny-組織のピアでコミットを待機しています
  • gateway.DefaultCommitHandlers.NetworkAll-チャネル内のすべての組織のすべてのピアでコミットを待機しています
  • gateway.DefaultCommitHandlers.NetworkAny-チャネルの任意のピアでコミットを待機しています

コミットを待つこともできませんが、後で詳しく説明しますが、まずこれに注意してください。

network, err := gw.GetNetwork("mychannel")
if err != nil {
	panic(err)
}

contract := network.GetContract("mycc")

最初に、チャネルから相互作用するピアのサブセットを取得し、次にコントラクトへのポインターを取得します。そのメソッドにより、チェーンコードとの相互作用が可能になります。

これで、コミットを待機するためにある種の戦略を使用するかどうかを決定する機会を得ました。まず、コミット待機中のトランザクションを見てみましょう。

result, err := contract.SubmitTransaction("Create", "b", "50")


ここで、ピアでトランザクションのシミュレーションを開始し、回答を取得し、提案応答を注文者に送信し、検証を待機して、コミットを待機するための上記の戦略に従ってコミットします。標準的で使い慣れた動作。さらに調べます。

result, err := contract.EvaluateTransaction("Read", "b")
	if err != nil {
		panic(err)
	}
fmt.Println(string(result))

この場合、クライアントは提案応答を受け取り、他には何もしません。コミットは期待していません。これは、状態を読み取るために使用する必要があります。

transient := make(map[string][]byte)
transient["someTransient"] = []byte("mytransientdata")
txn, err := contract.CreateTransaction("Update", gateway.WithTransient(transient))
if err != nil {
	panic(err)
}
result, err = txn.Submit("b", "100")
if err != nil {
	panic(err)
}

contract.CreateTransactionは、トランザクションをもう少し制御します。つまり、機能オプションを渡すことができます。

  • gateway.WithTransient（一時的） -プライベートデータを送信する
  • gateway.WithEndorsingPeers（ "peer0.org1.example.com"） -手動でターゲットを決定するため


app.goを 呼び出し、さまざまなキー、値、オプションをトランザクションメソッドに置き換えて、これがどのように機能するかを確認してください。

これで、新しいAPIの確認は終わりです。ちなみに、独自のウォレット、コミットハンドラー、IDを実装するのは非常に簡単なので、標準的な実装をタスクに適した実装に追加したり、コミュニティに独自のアイデアやコードを提供したりできます。
