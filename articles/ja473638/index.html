シェーダーとは何ですか？



出典：www.shadertoy.com/view/MsGSRd

実際、これはGPUで実行されているプログラムであり、その出力は異なる情報です。頂点シェーダーでは、これらはメッシュ頂点のパラメーターです。ピクセルシェーダーはピクセルごとに実行されます。

シェーダーのしくみを理解するには、グラフィックパイプラインとは何かを伝える必要があります。このトピックについてはかなり複雑な言葉で話すことがよくありますが、少しわかりやすくします。 OpenGLの例を見てみましょう。この点で、私はこの写真が本当に好きです。



照明等に関わる部分を省略した場合一般的に、同じUnlitシェーダーをhlslで作成するという観点から、本質は次のとおりです。シェーダーがあります

#pragma vertex vert
#pragma fragment frag

ここで、シェーダーの頂点部分はvert関数で、フラグメント部分はfrag関数で書き込まれると判断します。

シェーダーで説明する構造は、メッシュから取得するデータ、およびMeshRendererオブジェクトとMeshFilterオブジェクトでハングする頂点シェーダーで処理した後のデータを決定します。

          struct appdata
            {
                float4 vertex : POSITION;
                float2 uv : TEXCOORD0;
            };

            struct v2f
            {
                float2 uv : TEXCOORD0;
                UNITY_FOG_COORDS(1)
                float4 vertex : SV_POSITION;
            };

次に、頂点シェーダーはappdataデータを入力として受け取り、v2f構造の形式で結果を提供して計算し、フラグメントシェーダーに送信します。これにより、既にピクセルの色が計算されます。v2f情報は頂点（ピクセルよりも小さい）にのみ書き込まれるため、フラグメント部分のデータは補間されます。これはすべて、頂点が各頂点で独立して考慮されるという事実として表すことができます。次に、結果はフラグメントパーツに転送され、各ピクセルのフラグメントは独立していると見なされます。計算は並行して実行されるため、これらの部分には、近隣についての情報はありません（何らかの方法で巧妙に送信しない場合）。

詳細については、すべてのニュアンスと多くの例がUnity docs.unity3d.com/Manual/SL-Reference.htmlのドキュメントに記載されています

シェーダープログラミング言語



出典：www.shadertoy.com/view/WsS3Dc

他に忘れてはならない重要なこと。シェーダーが現在、単一性とは何の関係もない3つのプログラミング言語で記述されているという事実。CG、GLSL、HLSL。ユニットにシェーダーを書き込む最も簡単な方法はHLSLです。これは、.shader権限を持つシェーダーファイルが書き込まれる場所だからです。ユニタリコンテキストのシェーダーに関する情報が比較的少ない場合、HLSL、GLSL、およびCGに関する個別の情報は単なるトンです。シェーダーのドキュメントには、これらの言語で書かれたものをUnityに転送する方法が記載されています。したがって、これらのプログラミング言語に関する一般的なほとんどすべての情報が有効であることがわかります。3つの言語はすべてCに非常に似ていますが、それぞれに独自の特性があります。

さらに、シェーダーの研究の観点から、これらの言語がもはや問題を提起しない場合、「UnityCG.cginc」およびunityによって作成された他のライブラリーが、それらの作業を単純化するためにそれ自体で提供する可能性を確認できます。

シェーダーが悪い場合はどうしてですか？



出典：www.shadertoy.com/view/Md3cWr

シェーダーが鉄レベルで実行される方法と、シェーダーが非常に高速で、何百万もの操作を緊張せずに実行できる理由を理解することが重要です。

GPUの主なアイデアは、計算の最大並列処理です。ここでは、「波面」という概念を導入する必要があります。実際、非常にシンプルです。波面は、同じ一連の操作を実行するシェーダーのグループです。つまり、GPUの観点から見ると、最良のオプションは、同じ命令が同時に実行される場合です。実行の唯一の違いは入力です。分岐の問題は、シェーダーの単一グループで、シェーダーが異なる操作を呼び出す必要がある場合に発生する可能性があることです。これは、新しい波面の作成、データのコピーなどにつながります。そして、それは非常に高価です。

微妙な違いや例外はありますが、落ち着いて書けるようにするには、対象バージョンのグラフィックAPIでの動作を理解する必要があります。この点で同じOpenGL ES 2またはDX11は非常に異なるため。

ノードエディタがあるのに、なぜこれを知る必要があるのですか？




ノードエディタは主にテクニカルアーティスト向けのツールであることを理解することが重要です。これらは数学の専門知識を持つ専門家ですが、より多くのデザイナーです。ワイヤーフレームタイプのシェーダー（重心座標の理解が必要な場合）、またはトリッキーな投影に使用されるデカルト座標への変換は、物理的な材料の多くの数学モデルのように、コーディングがはるかに簡単です。同時に、シェーダープログラマーの観点からは、テクニカルアーティストが本物の魔法を作成するためのカスタムノードとツールを本質的に作成します。ノードエディタは、この観点から機能が制限されています。したがって、hlslなどの言語でシェーダーを記述できることが重要です。レンダリングの仕組みなどを理解する
