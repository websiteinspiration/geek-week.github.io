レルムでのページングライブラリの使用

Android部門の会議の1つで、私は、開発者の1人がRealmを使用するときに「無限」のリストを作成し、「遅延読み込み」と通知を維持するのに役立つ小さなlib をどのように作成したかを耳にしました。

私はドラフト記事を作成して書きましたが、ほとんど変わっていません。あなたと共有しています。彼は、問題が発生した場合はタスクをすくい取り、コメントすることを約束した。

無限のリストとターンキーソリューション


私たちが直面しているタスクの1つは、リストに情報を表示することです。それをスクロールすると、データが読み込まれ、ユーザーには見えないように挿入されます。ユーザーにとって、彼は無限のリストをスクロールするように見えます。

アルゴリズムはおよそ次のとおりです。

  • 最初のページのキャッシュからデータを取得します。
  • キャッシュが空の場合-サーバーのデータを取得し、リストに表示してデータベースに書き込みます。
  • キャッシュがある場合は、リストにロードします。
  • データベースの最後に到達した場合、サーバーにデータを要求し、リストに表示してデータベースに書き込みます。

簡略化：リストを表示するために、キャッシュが最初にポーリングされ、新しいデータをロードするための信号がキャッシュの終わりです。

無限スクロールを実装するには、既製のソリューションを使用できます。

  • RecyclerView.OnScrollListener, Github - , EndlessRecyclerViewScrollListener;

私たちはRealmをモバイルデータベースとして使用し、上記のすべてのアプローチを試した後、Pagingライブラリの使用をやめました。

一見すると、Android Paging Libraryはデータをダウンロードするための優れたソリューションであり、Roomと組み合わせてsqliteを使用する場合、データベースとして優れています。ただし、レルムをデータベースとして使用すると、遅延読み込みやデータ変更通知に慣れているものはすべて失われます。これらすべてをあきらめたくはありませんでしたが、同時にページングライブラリを使用しました。

多分私達はそれを必要とする最初ではありません


簡単な検索ですぐに解決策、レルム君主制ライブラリが見つかりました。簡単な調査の結果、このソリューションは適さないことが判明しました-ライブラリは遅延読み込みも通知もサポートしていません。私は自分で作成しなければなりませんでした。

したがって、要件は次のとおりです。

  1. レルムを引き続き使用します。
  2. レルムの遅延読み込みを保存します。
  3. 通知を保存します。
  4. ページングライブラリが提案するように、ページングライブラリを使用して、データベースからデータをロードし、サーバーからデータをページ分割します。

最初から、ページングライブラリがどのように機能し、気分を良くするために何をすべきかを考えてみましょう。

簡単に言うと、ライブラリは次のコンポーネントで構成されています

。DataSource-ページごとにデータをロードするための基本クラス。
PageKeyedDataSource、PositionalDataSource、ItemKeyedDataSourceの実装がありますが、それらの目的は今のところ重要ではありません。

PagedList -DataSourceからデータをチャンクでロードするリスト。ただし、Realmを使用しているため、データをバッチでロードすることは関係ありません。

リファレンス実装のソースコードで、回路がどのように機能するかを確認します。

1. getItem（int index）メソッドのPagedListAdapterは、PagedListのloadAround（int index）メソッドを呼び出します。

/**
* Get the item from the current PagedList at the specified index.
* <p>
* Note that this operates on both loaded items and null padding within the PagedList.
*
* @param index Index of item to get, must be >= 0, and < {@link #getItemCount()}.
* @return The item, or null, if a null placeholder is at the specified position.
*/
@SuppressWarnings("WeakerAccess")
@Nullable
public T getItem(int index) {
   if (mPagedList == null) {
       if (mSnapshot == null) {
           throw new IndexOutOfBoundsException(
                   "Item count is zero, getItem() call is invalid");
       } else {
           return mSnapshot.get(index);
       }
   }

   mPagedList.loadAround(index);
   return mPagedList.get(index);
}

2. PagedListは、voidメソッドtryDispatchBoundaryCallbacks（ブール型ポスト）をチェックして呼び出します。

/**
* Load adjacent items to passed index.
*
* @param index Index at which to load.
*/
public void loadAround(int index) {
   if (index < 0 || index >= size()) {
       throw new IndexOutOfBoundsException("Index: " + index + ", Size: " + size());
   }

   mLastLoad = index + getPositionOffset();
   loadAroundInternal(index);

   mLowestIndexAccessed = Math.min(mLowestIndexAccessed, index);
   mHighestIndexAccessed = Math.max(mHighestIndexAccessed, index);

   /*
    * mLowestIndexAccessed / mHighestIndexAccessed have been updated, so check if we need to
    * dispatch boundary callbacks. Boundary callbacks are deferred until last items are loaded,
    * and accesses happen near the boundaries.
    *
    * Note: we post here, since RecyclerView may want to add items in response, and this
    * call occurs in PagedListAdapter bind.
    */
   tryDispatchBoundaryCallbacks(true);
}

3.この方法では、データの次の部分をダウンロードする必要性がチェックされ、ダウンロード要求が発生します。

/**
* Call this when mLowest/HighestIndexAccessed are changed, or
* mBoundaryCallbackBegin/EndDeferred is set.
*/
@SuppressWarnings("WeakerAccess") /* synthetic access */
void tryDispatchBoundaryCallbacks(boolean post) {
   final boolean dispatchBegin = mBoundaryCallbackBeginDeferred
           && mLowestIndexAccessed <= mConfig.prefetchDistance;
   final boolean dispatchEnd = mBoundaryCallbackEndDeferred
           && mHighestIndexAccessed >= size() - 1 - mConfig.prefetchDistance;

   if (!dispatchBegin && !dispatchEnd) {
       return;
   }

   if (dispatchBegin) {
       mBoundaryCallbackBeginDeferred = false;
   }
   if (dispatchEnd) {
       mBoundaryCallbackEndDeferred = false;
   }
   if (post) {
       mMainThreadExecutor.execute(new Runnable() {
           @Override
           public void run() {
               dispatchBoundaryCallbacks(dispatchBegin, dispatchEnd);
           }
       });
   } else {
       dispatchBoundaryCallbacks(dispatchBegin, dispatchEnd);
   }
}

4.その結果、すべての呼び出しはDataSourceに分類され、データはデータベースまたは他のソースからダウンロードされます。

@SuppressWarnings("WeakerAccess") /* synthetic access */
void dispatchBoundaryCallbacks(boolean begin, boolean end) {
   // safe to deref mBoundaryCallback here, since we only defer if mBoundaryCallback present
   if (begin) {
       //noinspection ConstantConditions
       mBoundaryCallback.onItemAtFrontLoaded(mStorage.getFirstLoadedItem());
   }
   if (end) {
       //noinspection ConstantConditions
       mBoundaryCallback.onItemAtEndLoaded(mStorage.getLastLoadedItem());
   }
}
