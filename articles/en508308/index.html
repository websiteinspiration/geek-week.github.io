Context



To get the most out of what happened, let's first look at how Kubernetes works in a cloud-based environment.



When you create an instance of a Kubernetes cluster in such an environment, the cloud service provider is usually responsible for the operation of the management layer:





The management layer is located at the perimeter of the cloud provider, while Kubernetes nodes are at the perimeter of the client.



The mechanism for dynamically allocating volumes from the external storage -backend and comparisons with PVC (persistent volume claim, i.e. request on that).



Thus, after the PVC is created and attached to the StorageClass in the K8s cluster, the kube / cloud controller manager takes on the next steps to provide the volume (its exact name depends on the release). ( Note : we have already written here about CCM for an example of its implementation for one of the cloud providers .)



There are several types of provisioners supported by Kubernetes: most of them are included in the core of the orchestra , and others are controlled by additional provisioners, which are placed in pods in the cluster.



In our study, we focused on the internal mechanism for providing volumes, which is illustrated below:

The evolution of our research

This half-blind SSRF method was then used to scan the internal network of the cloud service provider and to poll various listening services (instance of Metadata, Kubelet, etcd, etc.) based on the responses of kube controller .







First, the standard listening ports of Kubernetes components (8443, 10250, 10251, etc.) were determined, and then the scanning process had to be automated.



Seeing that this method of scanning resources is very specific and not compatible with classic scanners and SSRF tools, we decided to create our own workers in a bash script that automate the entire process.



For example, to quickly scan the range 172.16.0.0/12 of the internal network, 15 workers were started in parallel. The above IP range has been selected solely as an example and can be changed to the IP range of a particular service provider.

Advanced Scenario # 3: CRLF + smuggling HTTP injection in ‚Äúold‚Äù versions of Kubernetes cluster



If, in addition to this, the provider offered customers old versions of the K8s cluster and opened them access to the kube-controller-manager logs, the effect became even more significant.



It is really much more convenient for an attacker to change HTTP requests at their discretion, designed to receive a full HTTP response.







To implement the latter scenario, the following conditions must have been met:



  • The user must have access to the kube-controller-manager logs (as, for example, in Azure LogInsights).
  • The Kubernetes cluster must use a version of Golang below 1.12.


We have deployed a local environment simulating the exchange of data between the GlusterFS Go-client and a fake target server (for now, refrain from publishing PoC).



A vulnerability was discovered that affected versions of Golang below 1.12 and that allowed hackers to conduct attacks such as HTTP smuggling / CRLF.



By combining the half-blind SSRF described above , along with this, we were able to send requests to your liking, including the replacement of headers, HTTP method parameters and data that kube-controller-manager and then processed.

As a result, an unsolicited response error occurs , a message about which is written to the controller logs. Thanks to the default ‚Äúverbosity‚Äù, the contents of the response HTTP message are also saved there.







This was our most effective ‚Äúbait‚Äù in the framework of the proof of concept.



Using this approach, we were able to conduct some of the following attacks in clusters of various managed k8s providers: escalating privileges with obtaining credentials on metadata instances, DoS masters using (unencrypted) HTTP requests on master instances etcd, etc.
