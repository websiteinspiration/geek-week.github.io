SSEGWSW：サービスワーカーによるサーバー送信イベントゲートウェイ

こんにちは！

私の名前はサーシャで、ティンコフビジネスで建築家として働いています。

この記事では、Service Workerを使用して、同じドメイン内で開いている長寿命のHTTP接続の数に対するブラウザーの制限を克服する方法について説明します。

必要に応じて、背景、問題の説明をスキップし、解決策を検索して、すぐに結果に進んでください。



バックグラウンド


むかしむかしTinkoff Businessで、Websocketで機能するチャットがありました。

しばらくして、彼は自分の個人アカウントのデザインに収まるのをやめ、一般に、長い間1.6から2+への書き換えを求めていました。それをアップグレードする時がきたと私は思いました。同僚のバックエンドは、チャットフロントエンドが変更されることを発見し、同時にAPIをやり直すことを提案しました。特に、トランスポートをwebsocketからSSE（サーバー送信イベント）に変更します。彼はこれを提案しました。NGINX構成を更新するとき、すべての接続が切断され、それを復元するのが面倒だったからです。

新しいソリューションのアーキテクチャについて説明し、通常のHTTPリクエストを使用してデータを送受信するという結論に達しました。たとえば、POSTメッセージを送信します。/ api / send-message、GETダイアログのリストを取得します：/ api / conversations-listなど。また、「対話者からの新しいメッセージ」などの非同期イベントがSSE経由で送信されます。そのため、アプリケーションのフォールトトレランスを向上させます。SSE接続が落ちても、チャットは機能しますが、リアルタイム通知を受信しません。

WebSocketでのチャットに加えて、「シン通知」コンポーネントのイベントを追跡しました。このコンポーネントを使用すると、たとえば、数分かかる場合があるアカウントのインポートが正常に完了したことなど、さまざまな通知をユーザーの個人アカウントに送信できます。WebSocketを完全に破棄するために、このコンポーネントを別のSSE接続に移動しました。

問題


1つのブラウザタブを開くと、2つのSSE接続が作成されます。1つはチャット用で、もう1つは微妙な通知用です。まあ、それらを作成しましょう。すみません、何ですか？申し訳ありませんが、ブラウザはごめんなさい！彼らは持っているドメインの同時持続的接続数の制限を。Chromeの価格はどれくらいですか？そう、6つ！3つのタブを開いた-接続プール全体をスコアリングしたが、HTTPリクエストを実行できなくなった。これは、HTTP / 1.xプロトコルにも当てはまります。HTTP / 2では、多重化によるそのような問題はありません。

この問題をインフラストラクチャレベルで解決するには、いくつかの方法があります。

  1. ドメインシャーディング。
  2. HTTP / 2。

多くのインフラストラクチャに影響を与える必要があるため、これらの方法はどちらも高価に思えました。

そのため、まず第一に、ブラウザー側で問題を解決しようとしました。最初のアイデアは、たとえばLocalStorageまたはBroadcast Channel APIを介して、タブ間で何らかのトランスポートを行うことでした。

その意味は次のとおりです。1つのタブでのみSSE接続を開き、残りのデータを送信します。Tinkoff Businessの個人アカウントを構成する50個のSPAをすべてリリースする必要があるため、このソリューションも最適に見えませんでした。50個のアプリケーションをリリースするのもコストがかかるので、他の方法を探し続けました。

決定


最近、サービスワーカーと協力して考えました。この状況でそれらを適用することは可能ですか？

この質問に答えるには、まずサービスワーカーが何ができるかを理解する必要がありますか？彼らはリクエストをプロキシできます、それはこのようなものです：

self.addEventListener('fetch', event => {
   const response = self.caches.open('example')
       .then(caches => caches.match(event.request))
       .then(response => response || fetch(event.request));

   event.respondWith(response);
});

HTTPリクエストのイベントをリッスンし、好きなように応答します。この場合、キャッシュから応答しようとしていますが、うまくいかない場合はサーバーにリクエストを送信します。

では、SSE接続をインターセプトして、それに答えてみましょう。

self.addEventListener('fetch', event => {
   const {headers} = event.request;
   const isSSERequest = headers.get('Accept') === 'text/event-stream';

   if (!isSSERequest) {
       return;
   }

   event.respondWith(new Response('Hello!'));
});

。要求は傍受されましたが、SSEはtext / plainの形式の応答を望んでいませんが、text / event-streamを望んでいます。今すぐストリームを作成する方法は？しかし、サービスワーカーからのストリームで応答することもできますか？ええと、見てみましょう



。Responseクラスは、本体としてReadableStreamを取ります。ドキュメントを読んだ後、ReadableStreamにenqueue（）メソッドを持つコントローラーがあることがわかります。これにより、データをストリーミングできます。ふさわしい！

self.addEventListener('fetch', event => {
   const {headers} = event.request;
   const isSSERequest = headers.get('Accept') === 'text/event-stream';

   if (!isSSERequest) {
       return;
   }

   const responseText = 'Hello!';
   const responseData = Uint8Array.from(responseText, x => x.charCodeAt(0));
   const stream = new ReadableStream({start: controller => controller.enqueue(responseData)});
   const response = new Response(stream);

   event.respondWith(response);
});



エラーはなく、接続は保留中の状態でハングし、データはクライアント側に到着しません。私の要求を実際のサーバー要求と比較すると、答えがヘッダーにあることがわかりました。SSE要求の場合、次のヘッダーを指定する必要があります。

const sseHeaders = {
   'content-type': 'text/event-stream',
   'Transfer-Encoding': 'chunked',
   'Connection': 'keep-alive',
};

これらのヘッダーを追加すると、接続は正常に開きますが、データはクライアント側で受信されません。ランダムなテキストを送信することはできないため、これは明らかです。何らかの形式が必要です。

javascript.infoでは、サーバーからデータを送信する際のデータ形式について詳しく説明しています。1つの関数で簡単に記述できます。

const sseChunkData = (data: string, event?: string, retry?: number, id?: number): string =>
   Object.entries({event, id, data, retry})
   .filter(([, value]) => ![undefined, null].includes(value))
   .map(([key, value]) => `${key}: ${value}`)
   .join('\n') + '\n\n';

SSE形式に準拠するには、サーバーは二重改行で区切られたメッセージを送信する必要があります\ n \ n。

メッセージは次のフィールドで構成されています。

  • data-メッセージ本文。行内の複数のデータは、改行で区切られた1つのメッセージとして解釈されます\ n;
  • IDは、 -再接続ラストイベントIDヘッダで送信されlastEventIdプロパティを更新します。
  • 再試行 -再接続までの推奨遅延（ミリ秒単位）。JavaScriptを使用して設定することはできません。
  • event-データの前に示されるユーザーイベントの名前。

必要なヘッダーを追加し、回答を目的の形式に変更して、何が起こるかを確認します。

self.addEventListener('fetch', event => {
   const {headers} = event.request;
   const isSSERequest = headers.get('Accept') === 'text/event-stream';

   if (!isSSERequest) {
       return;
   }

   const sseChunkData = (data, event, retry, id) =>
       Object.entries({event, id, data, retry})
           .filter(([, value]) => ![undefined, null].includes(value))
           .map(([key, value]) => `${key}: ${value}`)
           .join('\n') + '\n\n';

   const sseHeaders = {
       'content-type': 'text/event-stream',
       'Transfer-Encoding': 'chunked',
       'Connection': 'keep-alive',
   };

   const responseText = sseChunkData('Hello!');
   const responseData = Uint8Array.from(responseText, x => x.charCodeAt(0));
   const stream = new ReadableStream({start: controller => controller.enqueue(responseData)});
   const response = new Response(stream, {headers: sseHeaders});

   event.respondWith(response);
});

結果


これで、ブラウザーを超えずに、SSE要求を正常にインターセプトして応答できます。

最初は、サーバーとの接続を確立するというアイデアでしたが、1つだけです-そしてそこからデータをタブに送信します。やってみましょう！

self.addEventListener('fetch', event => {
   const {headers, url} = event.request;
   const isSSERequest = headers.get('Accept') === 'text/event-stream';

   //   SSE-
   if (!isSSERequest) {
       return;
   }

   //    SSE
   const sseHeaders = {
       'content-type': 'text/event-stream',
       'Transfer-Encoding': 'chunked',
       'Connection': 'keep-alive',
   };
   // ,    SSE
   const sseChunkData = (data, event, retry, id) =>
       Object.entries({event, id, data, retry})
           .filter(([, value]) => ![undefined, null].includes(value))
           .map(([key, value]) => `${key}: ${value}`)
           .join('\n') + '\n\n';
   //    ,   — url,  — EventSource
   const serverConnections = {};
   //   url            
   const getServerConnection = url => {
       if (!serverConnections[url]) serverConnections[url] = new EventSource(url);

       return serverConnections[url];
   };
   //         
   const onServerMessage = (controller, {data, type, retry, lastEventId}) => {
       const responseText = sseChunkData(data, type, retry, lastEventId);
       const responseData = Uint8Array.from(responseText, x => x.charCodeAt(0));
       controller.enqueue(responseData);
   };
   const stream = new ReadableStream({
       start: controller => getServerConnection(url).onmessage = onServerMessage.bind(null, controller)
   });
   const response = new Response(stream, {headers: sseHeaders});

   event.respondWith(response);
});
