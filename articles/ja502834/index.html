コードの配置

これらの違いを最小限に抑えることは、展開プロセス中だけでなく、展開後の実行中にも実行できる最善のことです。それでは、製品環境と非製品環境の違いをどのようにして削減または排除するのでしょうか。

最新の導入アプローチ


マシンにコードをデプロイすることは、1990年代と同じです。一連の本番マシンにコードをデプロイするときの最大の問題は、定義上、本番サーバー（コードが実行される）と開発サーバー（このコードが記述されている）が異なることです。展開直後、条件が変更されたため、これまで気付かなかった多くの問題が発生することは当然のことです。

したがって、デプロイする前に、デプロイメントアーティファクトがコードの一部ではなく、ランタイム全体であることを確認する必要があります。つまり、開発環境にコードを1回デプロイし、コードが実行されているマシン全体を複製して、必要な場所にコピーします。



これは「不変のデプロイメント」と呼ばれ、デプロイメント後の頭痛の時間を大幅に節約する非常に強力なテンプレートです。もちろん、コンテナーを実行する場合も同じ考え方が当てはまります。つまり、どこにでも同じコンテナーをデプロイします。次のように言うことができます。私の製品はdevとは異なります！」と、データベースのユーザー名/パスワード、接続文字列、S3ダンプスターの場所がすべて異なるため、そうです！はい、それらは非常に異なるものです。

この問題を解決するための方法は、12因子のアプリの原則を使用することで設定を。 12要素を適用すると、構成がenv varsまたはenv環境変数に格納されます。環境変数は、コードを変更せずにデプロイメント間で簡単に変更できます。構成ファイルとは異なり、誤ってそれらをコードリポジトリに格納する可能性は低く、カスタム構成ファイルやJavaシステムプロパティなどの他の構成メカニズムとは異なり、言語とオペレーティングシステムに依存しない標準です。したがって、構成全体を外部化し、環境変数としてコンピューターに転送する必要があります。

たとえば、AWSを使用している場合は、SSMをパラメーターの外部リポジトリとして使用します。これはCloudFormationと完全に統合されます。 AWS ssm cliコマンドから直接環境変数を設定することも非常に簡単です。もちろん、他のクラウドプロバイダーにも同様のメカニズムがあります。

また、問題が発生したときに製品マシンを「修正」するという衝動に抵抗してください。マシンは不変である必要があります。つまり、行う修正はすべてdevから行う必要があります。あなたの目標は、prodマシンへのアクセスをまったく防ぐことです。 ssh、scp、prodのいずれのアクセスも、誰にも、自分自身にも、初心者のハッカーにもアクセスできません。

しかし、問題を修正するためにログが必要な場合はどうなりますか？ご想像のとおり、ElasticSearch / Logstash / Kibana（ELK）スタックを使用するか、SumoLogicやDatadogなどの商用ソフトウェアを使用して、雑誌を外部化し、理想的には別の場所に送信する必要があります。

あなたが何をするにせよ、あなたの製品機械は「健康な牛」です。トラブルシューティングに何時間も費やして治癒するために世話をする必要のある「ペット」ではありません。

このアナロジーが頻繁に使用されていることは知っています。牛を本当に気にしている人から、これは完全に真実ではないという話を聞きましたが、本質は同じです。製品マシンを「修復」せずに、開発を修正して再デプロイするだけです。 。

コード配備の仕組み


だから、あなたは何をすべきかを知っていますが、あなたはその方法を知りません。残念ながら、これはジェンキンスが登場する場所です。わからない場合は、Jenkinsが最も人気のあるオープンソースのデプロイメント自動化サーバーの1つです。



ジェンキンス（および彼の前任者のハドソン）が私たちの近くに10年近くいるので、「残念ながら」と言っていますが、これは顕著です。セットアップは複雑で、メンテナンスがさらに困難です。疑わしい品質の何百万ものプラグインが付属しています。これらのプラグインは、原則として、最も不都合なときに中断し、他のすべてのプラグインを背後に落とします。実際、本当に安定したジェンキンスのワークショップはまれであり、通常は最大規模の組織でのみ見られます。

Jenkinsから始めることをお勧めするのはなぜですか？それは、そのすべての欠点にもかかわらず、それは依然として非常に人気があり、IT分野で広く使用されているためです。 Jenkinsを知ること、特にJenkinsfileがどのように構造化されているかを知ることは、求職者にとって大きな利点であり、見過ごすことはできません。 Jenkinsを探索するときは、廃止されたJenkinsジョブパイプラインではなく、新しいPipeline BlueOceanプラグインを使用してください。これは、CI / CDパイプラインをGitHub / GitLabリポジトリコード内に直接置きたい場合に重要です。したがって、パイプライン自体はバージョン管理されたコードです。



実際、非常に重要なので、もう一度繰り返す価値があります。すべてがコードです！アプリケーション、デプロイ方法、追跡方法、設定方法などはすべて、GitHub / GitLab / Anywhereに保存されている適切にバージョン管理されたコードスニペットです。この目標は、主要な開発者（関数型コードを作成するソフトウェアエンジニア）が矛盾のない環境を作成することです。

たとえば、次のことができるはずです。

  • 独自の小さなマイクロサービスを作成します。
  • 私が必要と考えるテストを追加します。
  • Jenkinsfileを追加
  • コードとしてのモニタリング設定を追加します。
  • env.yamlファイルでパラメーターを指定します。
  • これらすべてを1つのリポジトリに保存します。
  • Jenkinsが指定されたリポジトリを自動的に検出するようにします。
  • マイクロサービスを構築します。
  • 試して;
  • 展開します（カナリアまたはブルーグリーンの方法を使用）。
  • これが完了したら、メールにメッセージを送信するように手配してください！

これが目標であり、その達成はDevOpsエンジニアのコアミッションの本質です。

ジェンキンスの代替


先に述べたように、ジェンキンスは古くから存在しており、今日ではあまり人気のない代替品ではあるものの、私の意見では最良のものがあります。その1つがAWS CodeDeployです。これには制限がありますが、開発者はこの1年間で大幅な改善を行ってきました。AWSで作業している場合は、CodeDeployを試すことをお勧めします。

2番目の選択肢は、GitLab CI継続的統合モジュールです。組織がGitLabを実行している場合は、GitLabの他の部分とかなりうまく統合されているため、おそらくGitLabで作業を開始する必要があります。

最後に、GitHubは、アクションアプリケーションをデプロイするための独自のツールを発表しました。これは、独自の自動化によってサポートされています。

実際、ここではツールはそれほど重要ではないと思います。コードデプロイメントパイプラインを含め、すべてがバージョン化されたアーティファクトであり、開発者からのものでない限り、製品にはならないことを理解することが重要です。

いずれにしても、Jenkinsから始める場合は、コンテナとして設定してみてください。それほど難しくはなく、Jenkinsの拡張可能なコンテナ作業ノードを使用してJenkinsコンテナサーバーをデプロイする方法を学ぶ絶好の機会になります。実際、コンテナオーケストレーションなしで開始できます。これは次の記事の主題です。
