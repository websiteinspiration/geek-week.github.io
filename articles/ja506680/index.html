.NET CoreとNode.js。議論と事実

この記事は前の記事の続きであり、多くの良い興味深い質問がありました。コメントへの回答を書き始めたとき、あまり話していないことに気づきました。したがって、私は新しい本格的な記事を書くことにしましたが、別の角度から：個人的な経験と因果関係を追加しました。前のものを読んでいない場合は、それで問題ありません。読めないかもしれません。

主なアイデアは、2017年半ばに、当社が開発したバックエンドアプリケーションのメイン開発スタックを.Net（C＃）からNode.js（typescript）に変更したことです。これはどのようにして生じたのですか、私は決定を後悔し、私は戻って来ませんか？以下について。

バックグラウンド


最初に、あなた自身についての短い話。私はマネージャーではなくプログラマーとして生まれました。 2003年、学生としてIDE Visual Studio 6のC ++で、最初のアプリケーションを発表しました。それは、D3D、DirectSound、および当時DirectX 9の一部であった他のほとんどすべてのモジュールを使用したトランスフォーマーゲームでしたが、なぜ「私はプログラマーなのですか」。



私のクラスメートはこのゲームを1年ほど書きました。これはまさに私が子供の頃から夢見ていたプログラミングでした-非常にハイテクで、OOPのカプセル化、ポリモーフィズム、その他の魅力、トリッキーなアルゴリズム、フィボナッチヒープなどの異常なデータ構造を使用しています。

それから、3年生の実務経験と明確なポートフォリオのない学生にとって、これはGameDevの時に非常に人気のある会社で就職する唯一の機会であるように思われました。しかし、それは何も起こりませんでした。4時間のインタビューの後、とりわけ、アセンブラーで2行を交差させるためのアルゴリズムを紙に書いた後、生徒に連絡する準備ができていないという事実を理由に拒否されました。

C＃.Net


この失敗から数か月後、私は大規模な（100人を超えるプログラマー）ソフトウェア会社に行き、そこでC＃の登場により、私はそれに切り替えました。



より経験豊富で有能な同僚とのコミュニケーションの結果、実際には次のような興味深いことを習得しました。

  • ビジネスロジックからユーティリティコードを削除するアスペクト指向プログラミング（AOP）。

  • Data Load Options (DLO) , . - GraphQL, ;

主な欠点は、アルゴリズムとデータ構造の蓄積された知識、およびあらゆる種類のパターンを適用できる、非常に少数の興味深いタスクでした。ええと、私はプログラマーであり、それなしでは生きていけません！



私が担当した部門では、DBMSからのデータを他のシステムで使用できるようにするインターフェースを開発しました。年を重ねるごとに、頭の中でアイデアが高まり始めました。魂のない企業では、すべてがまさにそれであり、「これを実行し、少しひねって元に戻す」といったタスクのOOPはアイデアでした。

REST APIのリレーショナルDBMSおよびデータコントラクトのレベルでは、OOPは、OOPにのみ書き込みを行いたい開発者の熱意によってのみサポートされます。また、中間層のロジック（データベースから取得してコンシューマに提供する間の）は、複雑さと接続性の点で決してGameDevではなく、機能的なスタイルで簡単に記述できます。

2015年半ば、C＃で8年の経験があるため、興味深いタスクのためにSoftMediaLabに行きました。



バックエンドとフロントの両方で得た経験のおかげで、私はすぐに新しいかなり大きなプロジェクト-オンラインのp2pおよびb2p融資プラットフォームの開発の責任者になりました。バックエンドはC＃.Net Framework 4.6、マイクロサービスアーキテクチャ、WCFマイクロサービス通信、postgreSQL 9.5、ORMLite、DBMSオブジェクトのローリング/リロード用の履歴テーブルとスクリプトのORMクラスコード生成、DBMSの約150テーブル、AOPのpostharp 2.1、AOPのソケットSignalR、ランディングページ、およびASP.Net MVCに基づくWEB API。前のangular.jsとJS以来当時のtypescriptはまだ普及していませんでした。 CI / CD、バックアップノードへのレプリケーションのストリーミング。

私は一目でTypescriptが大好きでした。



もちろん、これはそれほど強く型付けされたC＃ではなく、プログラマーが使用しているオブジェクトの構造からの期待を説明する際の典型的なヒントでした。そして、私たちの期待、これは...まあ、あなたは覚えています...

Node.js


2017年の初め、typescriptが前面に出た後、typescriptのバックエンドを作成しようとする考えは、私には論理的に思えました。

当時、私はバックエンド開発のために次の要件を持っていました：

  • トリガー関数や履歴テーブルを含む、DBMSをローリング/再描画するためのコードは、コード生成またはORMによって自動的に作成する必要があります。
  • DTOモデルとそれらへ/からのマッパーを作成するためのコードは、コード生成を使用して自動的に作成する必要があります。
  • REST APIのSwagger記述は、ソースコードから自動的に生成されます。
  • MS SQL、PostgreSQL、mongoDbの操作や、特別な場合のカスタムクエリの記述を可能にする本格的なORMシステムが必要です。
  • redis, rabbitMQ, socket, .. ;

  • IDE, , .

私は1か月かけてコード生成に関連しない項目を調査し、typeorm、express.js、socket.io、redis、typescript-rest-swaggerなどの価値のあるソリューションを見つけました。

2017年半ば、node.jsはすでにかなり安定したテクノロジーでした。 Googleで数時間後に、ノードに実装された高リアルタイムの負荷条件の下で動作する約12のサービスを見つけました。また、パフォーマンスのボトルネックをC ++で最適化する必要があることも学びました。

コード生成の可能性を研究するために、C＃の経験がほとんどなく、JS / TypeScriptの経験がほとんどない技術研究所の賢明な卒業生を雇いました。特に、TypeScriptを入力するためのしきい値を確認します。

インテリジェントな卒業生の仕事の毎月の結果は3つの図書館でした-grunt-generate-view-model、grunt-generate-history-model、grunt-generate-database、私たちの会社はまだプロジェクトで使用しています。プレフィックス "grunt-"のみが関連しなくなりました。彼らは、うなり声プロジェクトビルダーのプラグインとしても働いたとき、それは元のバージョンから伸びています。

2017年の夏の終わりに、技術スペシャリストの会議を招集し、node.jsの機能を紹介し、投票の過半数を獲得して、node.js（TS）+ angular.io（TS）のバンドルで次の商用プロジェクトを作成することを決定しました。

たまたま、私たちの会社では、2つのプロジェクトが同時に始まりました。

  • オフィスの従業員がテレグラム、WhatsApp、FB、VK、Viberを介して顧客と1つのウィンドウで通信できるようにする通信プラットフォーム。
  • そして、ある顧客の内部ビジネスプロセスを自動化するプロジェクト。

どちらのプロジェクトも新しいスタックに正常に実装されました。それ以来、TSですべての新しいプロジェクトを行っています。node.jsのバックエンド、reactのフロントエンド（angular.ioからreactに切り替えられた理由は別の記事のトピックです）、モバイルアプリケーションはreact-nativeです。
