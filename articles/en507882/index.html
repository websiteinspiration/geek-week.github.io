There is a zabbiks 4.4, about 100 pieces of Asterisk 13 version. Some exchanges come with the FreePBX web interface, some with a bare console, a bunch of tricks and integration via dialplan.



We receive data from automatic telephone exchange



The first and main point that needs to be solved is obtaining data on peers and sip registrations. To do this, the PBX has interfaces AGI, AMI, ARI and SSH console. For obvious reasons, I did not consider additional modules.



First you need to figure out what these agi, ami, ari ... are.



  • AGI - the use of scripts in dialplan. Mainly used to manage calls.
  • AMI - can provide all the necessary information, works through port 5038 by analogy with Telnet. Suitable for us!
  • ARI - modern, fashionable, JSON. There are many possibilities, the data format is understandable for Zabbix, but for me there is no main thing: you can not control sip registration. One more minus, for peers there are only two online / offline states, although there are more states and it is useful to take them into account when diagnosing.
  • SSH - it can do everything, but sometimes it isn‚Äôt given due to ‚Äúsecurity considerations‚Äù. Considerations may be different, I will not disassemble them.


However, for all its shortcomings, ARI covers 90% of all monitoring needs.

It would be great to use AMI on the web: it would solve all the problems, I thought. I start to dig in this direction, and literally the first line of search results on the official documentation of the Asterisk, which states that my goals have the option webenabled file /etc/asterisk/manager.conf , which must be set to YES in the section [general ]



After that, through a regular web request of the form http: // ats: 8089 / mxml? Action = SIPshowregistry we get all the necessary information.

When using the FreePBX interface, you cannot enable this option via the web, you need to enable it through the console, making changes to the manager.conf file. FreePBX does not erase it when configuration changes are made through the web.
How many worked with various sorts of Asterisk integrations, I never saw that this function was mentioned somewhere. I was surprised that no one describes this method of interacting with a telephone exchange. It was even especially useful to look for information on this topic: there is practically nothing or was used for completely different tasks.



WEB AMI - what kind of beast?



Adding options webenabled file manager.conf Offering full access to management through a web ASCT. All the commands available through the regular AMI are now on the web, you can listen to events from the PBX through the socket. The principle of operation is no different from the console AMI. After activating this option, you can contact the PBX at the following addresses:



https: // ats: 8089 / manager - a web page with a simple interface for tests and manual sending of requests. All answers are formatted in a readable HTML form. For monitoring is not very suitable.

https: // ats: 8089 / rawman - only text output, the format is similar to the console AMI

https: // ats: 8089 / mxml - only text output, in XML format. It suits us!

There is a lot of garbage in the response, so in the preprocessing we filter it according to the XPath template : // response / generic [@host] The

most interesting part begins. To work with detection and dynamically create elements, the response must be in JSON format. XML for auto-detection is not supported.



To convert XML to JSON, I had to play around with auto-replacement a bit, for which I made a script in JS.







An interesting point: in the PBX response, all parameters are surrounded by single quotes, and after applying the template // response / generic [@host] they are replaced by double ones.



To create elements, we use variables from the XML response (now JSON).







SIP Registry



For sip registrations, we use three variables: username , host , port . I was satisfied with the name of the element 111111@login.mtt.ru: 5060 , I did not find situations where I need to use all five variables.



The main element that receives information about all registrations, Asterisk is AMI SIPshowregistry . Once a minute, it makes a GET request to https: // ats: 8089 / mxml? Action = SIPshowregistry, after which the XML response data is transmitted to all dependent elements for analysis. I create an element for each registration dependent on it. This is convenient, because we get relevant information for one request, and not for each separately. This implementation has a significant minus - the load on the processor.
