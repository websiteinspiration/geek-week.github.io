リサイクル可能な材料の分別収集のロジスティクス

参加する代わりに


ロシアで廃棄物の収集と処理のプロセスが完全に調整されたとき、言うのは容易ではありませんが、今は埋め立て地の補充に参加したくないです。したがって、多くの大都市では、何らかの方法で、特定の収集に従事するボランティア運動があります。

ノボシビルスクでは、このような活動はグリーンリスキャンペーンを中心に行われており、月1回、環境に配慮した町民が、蓄積されたリサイクル可能な家庭ごみを特定の時間に所定の場所に運びます。同時に、レンタルトラックがそこに到着し、収集および選別されたリサイクル可能なものをサイトに運び、そこからさまざまな処理企業間で再配布されます。この措置は2014年から存在しており、それ以降、リサイクル可能物の収集ポイントの数とその量は大幅に増加しています。トラックのルーティングでは、視線だけでは不十分で、輸送コストを最小限に抑えるための最適化モデルの開発に着手しました。これらのモデルの最初のものは、この記事の主題です。

セクション1では、アクションを構成するためのスキームを図解で詳しく説明します。さらに、セクション2では、輸送コストを最小化するタスクは、時間枠を使用して異種車両のフリート車両ルーティング問題をルーティングするタスクとして形式化されます。セクション3は、混合整数線形数学プログラミング問題GLPKを解くための自由に配布されたパッケージを使用してこの問題を解くことに専念します。

1.アクション「グリーンリス」


2014年以降、Living Earthイニシアチブグループは、ノボシビルスクでリサイクル可能なグリーンリスの分別収集のために毎月キャンペーンを実施しています。これを書いている時点では、PET、PE、PP、PS、ガラス、アルミニウム、鉄、家電製品、古紙などのラベルが付いたプラスチック廃棄物が多数あり、リサイクルが認められています。 50人以上のボランティアが行動の準備と実施に参加しています。この行為は営利目的ではなく、参加は無料であり、金銭的報酬を意味するものではありません。

ポイント

アクションは、15〜90分の間に車でカバーできる距離で互いに離れた都市の17地点で行われます。これらのポイントの1つは写真にあります。壁に沿って左側にあるのは、プラスチックのさまざまな部分を収集するためのバッグです。右側は、将来すべてが積み込まれるトラックで、中央には耳のあるボランティアです。



その時点でのすべての活動は、職務を遂行する準備ができている時間に制限があるキュレーターによって組織されます。アクションを計画するとき、キュレーターは、アクションがその時点で通過する必要がある時間間隔を報告します。

また、各地点で収集されたリサイクル可能な資源の平均量に関するデータもあります。

ルート

ポイントは、トラックで連続的に運転されるルートに編成されます。トラックの動きは、運行状況を監視し、特別なイベントの処理を決定するルートキュレーターによって監視されます。



トラック

は、貨物車の時間単位のレンタルの提案の枠組みの中で共通の基準でレンタルされます。プラスチックを所定の位置で圧縮することは不可能であるため、トラックを特徴付ける主なパラメータは車体の容積です。私たちの場合の収容能力は制限要因ではありません。

訴訟の主な費用はトラックのレンタル料の支払いに関連しているため、責任ある消費についての考えを形成するという意味で制度上の重要性を獲得する当社のシェアの存在と発展にとって、その削減は重要です。次に、この問題を解決するためのアプローチを、離散最適化手法、すなわち数理計画法の適用に基づいて説明します。

解決策を探す


最適化モデルに含まれるすべての式が変数の線形関数であることを簡単に検証できるため、正確な近似解を見つけるには、標準パッケージを使用して、Gurobi、CPLEX、Xpress、ORtoolsなどの混合整数プログラミング問題を解くことができます。SCIP、BLISなど
交通費を最小限に抑えるモデルをGMPL言語で記述します。これにより、無料のGLPKパッケージを目的に使用できるようになります。コードを記述してモデルをデバッグするには、すでにGLPKが構成に含まれているGUSEK開発環境をダウンロードすると便利です。

GUSEKの外観は次のとおりです。



左側にはモデルの説明が表示され、右側には計算の進行状況に関する情報を表示するウィンドウがあり、起動後にソルバーから提供されます。

モデルの完全な説明
param numOfPoints > 0, integer;	# 
param numOfTypes > 0, integer;	#  
param numOfRoutes = numOfPoints;#  

set V := 1 .. numOfPoints;	# 
set Vbar := V union {0};	#     ()
set T := 1 .. numOfTypes;	#  
set R := 1 .. numOfPoints;	# 


###############
param WDL >= 0, default 8;	#  
param B{i in Vbar} >= 0;	#  
param E{i in Vbar} >= 0;	#  
param L{i in Vbar} >= 0;	#  
param D{i in Vbar, j in Vbar} >= 0, <= WDL;	# 
param G{i in V}, >= 0;	# , 3
##########
param C{t in T}, >= 0;	# 
param P{t in T}, >= 0;	#   
param U0{t in T}, >= 0;	#  , 


/**********************************************
*	  
**********************************************/
var z{Vbar, R} binary;	# ,     ,     

s.t. pointToGroup 'point to group' {i in V}:  sum{r in R} z[i, r] == 1;
s.t. depotToGroup 'depot to group' {r in R}: z[0, r] == 1;
s.t. lexMinGroups 'lexicographycally minimal division' {i in V, r in R: r <= i}: 
	1 - z[i, r] 
	<= 
	sum{j in V: j <= i - 1}(1 - sum{k in R: k <= r - 1} z[j, k]) 
	+ 
	sum{k in R: k <= r - 1}z[i, k] ;

/**********************************************
*	  
**********************************************/
var x{Vbar, Vbar, R} binary;	# ,    c  r      i   j,   .

s.t. visitPoint 'visit point' {i in Vbar, r in R}: sum{j in Vbar} x[j, i, r] = z[i, r];
s.t. keepMoving 'keep moving' {i in Vbar, r in R}: sum{j in Vbar} x[j, i, r] = sum {j in Vbar} x[i, j, r];

var f{Vbar, Vbar, R} >= 0;		#,  .

s.t. flowFromDepot 'flow from depot' {r in R}: sum{i in V} f[0, i, r] == sum{i in V} z[i, r];
s.t. flowAlongActiveArcs 'flow along active arcs' {i in Vbar, j in Vbar, r in R}: f[i, j, r] <= numOfPoints * x[i, j, r];
s.t. flowConservation 'flow conservation' {i in V, r in R}: sum{j in Vbar} f[j, i, r] == sum{j in Vbar} f[i, j, r] + z[i, r];

var a{i in V} >= 0;	#    
var w{i in Vbar, r in R} >= 0;	#    

s.t. wait 'wait'{i in Vbar, r in R}: w[i, r] >= L[i] * z[i, r];
s.t. dontWait 'dont wait'{i in Vbar, r in R}: w[i, r] <= (E[i] - B[i]) * z[i, r];
s.t. arrivalTime 'arrival time' {i in V, j in V}: a[i] + sum{r in R}w[i, r] + D[i,j] <= a[j] + 3 * WDL * (1 - sum{r in R} x[i, j, r]);
s.t. arriveAfter 'arrive after' {i in V}: a[i] >= B[i];
s.t. departBefore 'depart before' {i in V}: a[i] + sum{r in R}w[i, r] <= E[i];

/**********************************************
*	  ,      
**********************************************/

var y{t in T, r in R}, binary;	# ,    t       r,   .
var u{t in T, r in R}, integer, >= 0;	#    t,     r.

s.t. assignVehicle 'assign vehicle' {r in R}: sum{t in T} y[t,r] == 1;
s.t. rentTime 'rent time' {r in R, t in T}: u[t, r] >= sum{i in V, j in Vbar}D[i, j] * x[i, j, r] + sum{i in Vbar}w[i, r] - WDL * (1 - y[t, r]);
s.t. minRentTime 'minimal rent time' {r in R, t in T}: u[t, r] >= U0[t] * (y[t, r] - sum{i in V}z[i, r]);
s.t. noRent 'no rent' {t in T, r in R}: u[t, r] <= WDL * y[t, r];
s.t. fitCapacity 'fit capacity' {r in R}: sum{i in V} G[i] * z[i, r] <= sum{t in T} C[t] * y[t, r];

minimize rentCost: sum{t in T, r in R} P[t] * u[t, r];

solve;

結論の代わりに


ルーティングの問題は計算が非常に複雑であり、訪問する必要があるポイントの数が増えると、ソルバーがソリューションを見つけてその最適性を証明するのに必要な時間が急速に増大しています。ただし、かなり小さな例の場合、妥当な時間内に、ソルバーは成功したルートのセットを構築でき、意思決定をサポートするために使用できます。モデルによって提案されたルーティングオプションの分析は、コスト削減の重要な機会を発見するのに役立ちました。これは、在庫の存在と開発にとって重要です。

私たちのさらなる努力は、ポイントで収集されたリサイクル可能な資源の量に不確実性を伴う作業に向けられました。トラックルーティングの戦略的および運用上の決定を行うために、いくつかの確率的プログラミングモデルを開発しています。トピックが関連性があり興味をそそられる場合は、私もこれについて書きます。これは、すぐに私たち全員が環境問題の解決にかなり徹底的に取り組む必要があるためです。
