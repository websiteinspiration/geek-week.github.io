Monitoring data at rest with a DCAP system

DLP systems do a good job of protecting against data leaks when it comes to data on the move. Difficulties begin when we want to cover the remaining 2 types: data at rest and data at work. What is wrong with monitoring data at rest using DLP?


Typically, DLP systems have an eDiscovery engine for working with data at rest. At SearchInform CIB, a module called Indexing Workstations (IRS) is responsible for this area. He is good, but for his task.

EDiscovery's job is to find sensitive content. Just to find. Do not reveal details about access rights or actions with this information. That is the difference. But information security specialists have problems not just a point search of documents. It is necessary to identify all employees who have access to files, determine which recent changes have been made, which revisions of critical files have been saved, etc. And here DCAP should come in.



DCAP in brief


It's time to decipher the abbreviation, already familiar in the text. DCAP-systems (Data-Centric Audit and Protection) are designed for automated audit of data in the file system, search for access violations and tracking changes in critical documents.

Basically, Gartner DCAP systems should focus on the following:


  • Data classification.
  • Storage of confidential data.
  • Data Security Management.
  • Monitoring and auditing data.
  • Protection of all data from unauthorized access and use.

FileAuditor. Past


Thinking about a new product at the beginning of 2018, we set such tasks for the alpha version:

  • For analytics, use your own engine that has proven itself in the flagship DLP system.
  • Support "agent" and "network" modes of operation.
  • At the same time be an independent product. That is, do not conflict with third-party solutions.
  • Be faster, higher, stronger than competitors ( FileAuditor was the first Russian DCAP solution). And in general, "write immediately without bugs."

The logic of work was laid as follows:

  • The system finds the file.
  • The file is checked according to the rules and receives a service tag.
  • If necessary, the file is copied to the repository.
  • All actions with files and folders are logged.
  • In subsequent checks, the system scans only modified and newly added files.
  • Access rights for files and folders are deducted.

If we talk about the wishes of customers, then first of all they wanted to know:

  • Where is what lies in the network and on the machines of employees?
  • Who has access to this?
  • Did the data leak?

The process of receiving responses began with the creation of a Data Control Rule. The Rule describes which objects it will be applied to, under what conditions the search will be performed, and what the system will do if the file falls under the Rule.



First of all, objects of interest are set in it. They can be both the file extension (determined by name or by signature), and the place. If you set several conditions, the system will combine them according to the logical "AND".

As can be seen from the screenshot, the system supports masks.



Next, the work flows to the "Search Terms" tab. They can also be combined. Here, the conditions for the contents of the file are set, because it is important to find not all files in a row, but files with specific contents. For example, with vultures.



And here analytical capabilities play a key role. At the time of going public release in August 2019, we supported 3 types of searches:

  • according to the text;
  • by file attributes;
  • by regular expressions.

Text Search


This is a full-fledged phrase search with a bunch of ‚Äúlotions‚Äù.



You can search for a phrase taking into account morphology, and by exact coincidence. Or by mask. Word order, transliteration, typos - all these options are designed to help create the most accurate search query and simplify version control of documents.

Separately, I would like to mention the search option by a sequence of characters, which allows you to search by non-indexed information (for example, by the characters No., @, \, /, ‚Äúand others). As a result, with a flick of the wrist you can find all the copies and versions of some important agreement.



If during the formation of the request we left only ‚ÄúContract No. 135 \‚Äù, the system would find all the documents containing this sequence.

File Attribute Search


At the moment, search is available for 4 attributes:

  • file name;
  • the size;
  • date of creation;
  • Date of change.



Thus, you can focus not only on the contents of the file, but also on its ‚Äúfreshness‚Äù. For example, search only for new documents.

The question may arise: why is the attribute "File Name" here, if it can already be set in the "Objects"? This is done in order to solve the problem of searching in attached documents. I will explain with an example. Initially, the object selected files with the extension .msg. We will search for the saved mail.



But we will not be interested in all letters in a row, but only those within which there is an attachment. Moreover, a certain type (.doc, .docx, .xls) and content (the phrase "For official use" is mandatory).



Although an abstract example, it shows that, if desired, the system can be configured to analyze data very precisely.

FileAuditor. The present


First, a few words about the improvements that were made to the release.

FileAuditor is a standalone product that does not conflict with third-party systems. Moreover, it integrates well and seamlessly with SearchInform CIB .

Both "agent" and "network" operating modes are supported. As in the case of DLP, each of the modes has its pros and cons. So, the agent can only work with the Windows family. At the same time, many repositories work on other OSs. And here the "network" mode comes to the rescue, for which neither the OS nor the file system are important. It is only important that the service can ‚Äúreach‚Äù the files via SMB. Also, in the network operation mode, it is possible to interact with cloud storage services (Dropbox, Yandex.Disk, etc.), databases, document management systems, and other IT infrastructure objects.

Intuitive scan statistics added. So that you can clearly see what the agents and the network service are doing now.



You can set both general data analysis rules and personal rules for a single machine.

A new type of search has been added - dictionary search, when the file searches for the mention of a certain number of words.



So, the rules are set, the services are started. It's time to take a look at the results of the work. They are visible in the Analytics Console.




The workspace consists of four blocks:

  1. Filter block. You can display only what is needed at the moment: display a list according to a certain rule, to a user, etc.
  2. Block search results. Folders and files that fit the configured filters fall into it.
  3. Event Log Block A log is maintained for each file. In the case of document version control by the system, switching between file versions is carried out precisely from the Event Log.
  4. View block. Combines 3 viewing modes. Therefore, we consider it separately.

The "View Files" mode allows you to look at a file in the "native" format. Actual for files, where to understand the content it is important to see the initial formatting (for example, Excel spreadsheets).

The "Text only" mode shows the processed (indexed) file. The main benefit is that the system highlights the places where the trigger occurred, and allows you to quickly "jump" from one trigger to another. Actual for large multi-page documents.

FAQ


Well, even though the post turned out to be voluminous, not all questions were reflected in it. Therefore, we list the most common of them in a question-answer blitz format.

1. Slows down the car?

No. You can configure it so that the employee does not feel any inconvenience (scheduled check, check only if the CPU load is less than N%, check only in the absence of active sessions, etc.)

2. How much space does the interception take?

It may not take at all if you only enabled data auditing. If you have enabled auditing and shadow copying, then only files that are subject to the rules will be copied to the repository. FileAuditor implements a deduplication system. If the file ‚Äúscattered‚Äù over the network, then only one file will be in the storage, not 50 copies.

3. In what form is the intercepted information stored?

All shadow copies are transmitted and stored only in encrypted form.

4. Does the storage have to be on the same server as the rest of the system?

No. You can specify any network folder if you need.

5. How long does the scan take?

The crucial point is whether this is the first scan. Recall that after the first scan, the agent already commits changes and new files. If the file has not changed, the agent sees this and does not waste time on it. If we take specific numbers, then on a real test the client has 70 TB for the first time, the system indexed about 10 days. The second scan of the same volume took about 30 minutes.

6. Do you work with all formats?

Yes (not really). Depends on the rule itself. If you need to check the contents of the file, then we are limited to those formats that SearchServer currently supports (about 150). If you need to control attributes (for example, the date of creation / file modification), then the format is any.
