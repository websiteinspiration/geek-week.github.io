Java + Selenium Webdriver + aShotでの視覚回帰テストのためのシンプルなソリューション

こんにちは。視覚的回帰テストの自動化における私の経験についての記事

を既に公開しています。 それ以来、プロジェクトは大幅に改善されました。構造が変更され、プロジェクトの設定と自動テストの作成がはるかに簡単になり、レポートが大幅に改善されました。VisualRegressionFrameworkは、小規模なプロジェクト向けのかなりシンプルなソリューションです。私が取り組んでいるプロジェクト用に約50の自動テスト（ページ+要素）が書かれました。







レイアウトの変更後にテストを実行します。変更はすぐに表示されます。つまり、「1つのページの要素のレイアウトが変更されたが、すべてが別のページで壊れた」などの問題は、長い間解決されています。

使用されている技術-Java、TestNG、Maven、Selenium Webdriver、aShot（画像を比較するためのライブラリ）。

機能：


  • ページのスクリーンショットを比較する
  • ページのスクリーンショットの比較+指定された要素（動的コンテンツなど）の無視
  • アイテムのスクリーンショットを比較する
  • ブレークポイント-幅の異なるページまたは要素のスクリーンショットを比較できます。デフォルトでは、3つのブレークポイントが使用されます-1920px、768px、360px
  • 予想されるスクリーンショットを自動生成
  • テストは通常​​の機能自動テストと同様に記述できます。制限はありません。スクリーンショットを撮る前に複雑なアクションを実行する必要がある場合があるため、これは重要です。

左側のサイドバーには、テストのリスト、エラーのあるログ、自動テストが実行されたブラウザーがあります。

ブレークポイントを切り替えたり、画像をクリックして拡大したりすることもできます。

フレームワーク構造


src / config / visual.properties-ここにすべての必要な設定があります

  • ブレークポイント-ブレークポイント、スクリーンショットが撮られるサイトの幅
  • allowedDiffSize-予想されるスクリーンショットと実際のスクリーンショットの許容される差異（ピクセル単位）
  • スクリーンショットのあるフォルダーへのパス
  • レポートのテンプレート、レポート自体、およびエラーログを含むファイルへのパス

report / REPORT.html-レポートの

スクリーンショット

  • 実際 -テスト中に作成された実際のスクリーンショット
  • 予想 -予想されるスクリーンショット。予想されるスクリーンショットがない場合は、テストを初めて実行するときに自動的に作成されます。
  • diff-不一致領域が赤で強調表示されている画像
  • gif-実際の画像、予想された画像、および差分画像から作成されたgif

src / main / framework-フレームワーククラス-レポートジェネレーター、スクリーンショット、gifなど

src / main / appはページオブジェクトアーキテクチャのほんの一例であり、使用する必要はなく、削除できます。

src / test / resources / testng.xml-ここで、A_BeforeAllTestsとA_AfterAllTestsの間に新しいテストクラスを追加する必要があります

src / test / java

  • A_BaseTest-すべての新しいテストクラスが継承する基本テストクラス
  • A_BeforeAllTests-テスト前の設定とブラウザーの初期化
  • A_AfterAllTests-テスト後のレポート生成
  • A_ErrorsLogListener-エラーのロギング
  • StaticGuestTest-テストクラスの例

クラスは特定の順序で実行されます。

  1. 最初に設定とブラウザの初期化-クラスA_BeforeAllTests
  2. その後、自動テストをテストします
  3. — A_AfterAllTests




    @Test
    public void indexPage()
    {
        app.open();
        app.preparePageForScreenshot();
        Comparer.comparePages("index_page");
    }

上記の例では、ページが開くだけで、スクリーンショット用にページが準備され、その後、ページの予想されるスクリーンショットと現在のスクリーンショットが比較されます。スクリーンショットが作成され、すべてのブレークポイントで比較されます。

関数preparePageForScreenshot（）が必要な理由：

デフォルトでは、ページは単に下にスクロールし、次にjs +まで上にスクロールします。2番目の遅延があります。これは、すべての画像、スタイルなどをロードできるようにするために必要です。

この機能では、不要な要素、たとえば広告付きの動的ブロックを非表示にすることもできます。私のプロジェクトでは、フッター、ヘッダー、サイドバーはすべてのページで同じなので、非表示にしています。これらの要素がないと、テストがより速く実行され、レポートの検証がより簡単になります。

ページ

すべてのブレークポイントでページのスクリーンショットを比較します。

Comparer.comparePages("test_name");

すべてのブレークポイントでページのスクリーンショットを比較し、要素を非表示にします。

Comparer.comparePages("test_name", new String[]{"css_locator_1", "css_locator_2"});

指定したブレークポイントでのみページのスクリーンショットを比較します。

Comparer.comparePagesWithBreakpoint("test_name", "1920");

指定されたブレークポイントでのみページのスクリーンショットを比較し、要素を非表示にします。

Comparer.comparePagesWithBreakpoint("test_name", "1920",  new String[]{"css_locator_1", "css_locator_2"});

要素

すべてのブレークポイントで指定された要素のスクリーンショットを比較します。

Comparer.compareElements("test_name", "css_locator");

指定されたブレークポイントでのみ指定された要素のスクリーンショットを比較します。

Comparer.compareElementsWithBreakpoint("test_name", "1920", "css_locator");
