Pythonおよび高速HTTPクライアント

現在、ある種のPythonアプリケーションを作成している場合は、HTTPサーバーと通信できるHTTPクライアントの機能を備えている必要があります。REST APIの遍在性により、HTTPツールは無数のソフトウェアプロジェクトの居住者に尊敬されています。そのため、プログラマーはHTTP接続での最適な作業を整理するためのパターンを所有する必要があります。 Pythonには多くのHTTPクライアントがあります。それらの中で最も一般的であり、さらに扱いやすいものは、リクエストと呼ばれます。今日、この顧客は事実上の標準となっています。





永続的な接続


HTTPを使用する際に検討する最初の最適化は、Webサーバーへの永続的な接続を使用することです。永続的な接続はHTTP 1.1以降標準になりましたが、多くのアプリケーションはまだそれらを使用していません。この欠陥は簡単に説明できrequestsます。ライブラリをシンプルモードで使用する場合（たとえば、メソッドを使用する場合get）、サーバーからの応答を受け取った後、サーバーへの接続が閉じられることがわかります。これを回避するには、アプリケーションで、Session開いている接続を再利用できるオブジェクトを使用する必要があります。

import requests

session = requests.Session()
session.get("http://example.com")
#   
session.get("http://example.com")

接続は接続プールに格納されます（デフォルトでは、デフォルトで10接続です）。プールのサイズはカスタマイズできます：

import requests


session = requests.Session()
adapter = requests.adapters.HTTPAdapter(
    pool_connections=100,
    pool_maxsize=100)
session.mount('http://', adapter)
response = session.get("http://example.org")

TCP接続を再利用して複数のHTTPリクエストを送信すると、アプリケーションに多くのパフォーマンス上の利点があります。

  • プロセッサーの負荷を減らし、RAMの必要性を減らします（同時に開く接続の数が少ないため）。
  • 次々に来るリクエストを実行するときの遅延を減らします（TCPハンドシェイク手順はありません）。
  • TCP接続を閉じるために、例外がスローされることはありません。

HTTP 1.1は、リクエストのパイプライン処理もサポートしています。これにより、以前に送信された要求への応答を待つことなく、同じ接続内で複数の要求を送信できます（つまり、「パケット」で要求を送信します）。残念ながら、ライブラリrequestsはこの機能をサポートしていません。ただし、リクエストのパイプライン処理は、並列処理ほど高速ではない場合があります。さらに、これに注意を払うことが適切です。「パケット」リクエストへの応答は、これらのリクエストを受信したのと同じ順序でサーバーから送信される必要があります。結果は、FIFOの原則（「先入れ先出し」-「先着順」）に従って、最も効率的な要求処理方式ではありません。

並列クエリ処理


requests加えて、もう一つの重大な欠点があります。これは同期ライブラリです。このようなメソッド呼び出しはrequests.get("http://example.org")、完全なHTTPサーバー応答が受信されるまでプログラムをブロックします。アプリケーションが待機し、何もしないといけないという事実は、サーバーとの対話のこの編成スキームのマイナスと見なすことができます。プログラムをただ待つのではなく、何か便利なことをさせることは可能ですか？

スマートに設計されたアプリケーションは、それが提供するようなスレッドプールを使用してこの問題を軽減できますconcurrent.futures。これにより、HTTPリクエストの並列化をすばやく整理できます。

from concurrent import futures

import requests


with futures.ThreadPoolExecutor(max_workers=4) as executor:
    futures = [
        executor.submit(
            lambda: requests.get("http://example.org"))
        for _ in range(8)
    ]

results = [
    f.result().status_code
    for f in futures
]

print("Results: %s" % results)

この非常に便利なパターンは、requests-futuresライブラリに実装されています。同時に、オブジェクトの使用はSession開発者に対して透過的です。

from requests_futures import sessions


session = sessions.FuturesSession()

futures = [
    session.get("http://example.org")
    for _ in range(8)
]

results = [
    f.result().status_code
    for f in futures
]

print("Results: %s" % results)

デフォルトでは、2つのスレッドを持つワーカーを作成しますが、プログラムは、オブジェクトFuturSession引数max_workersまたは独自のアーティストを渡すことで、この値を簡単にカスタマイズできます。たとえば、次のようになります。

FuturesSession(executor=ThreadPoolExecutor(max_workers=10))

リクエストによる非同期作業


すでに述べたように、ライブラリはrequests完全に同期しています。これにより、サーバーからの応答を待機している間にアプリケーションがブロックされ、パフォーマンスへの影響が少なくなります。この問題の1つの解決策は、HTTPリクエストを個別のスレッドで実行することです。ただし、スレッドを使用すると、システムに追加の負荷がかかります。さらに、これはプログラムに並列データ処理スキームを導入することを意味しますが、すべての人に適しているわけではありません。

Python 3.5以降、標準言語機能には、を使用した非同期プログラミングツールが含まれていますasyncio。aiohttpライブラリは、以下に基づく非同期HTTPクライアントを開発者に提供しますasyncio。このライブラリを使用すると、アプリケーションは一連のリクエストを送信して作業を続けることができます。同時に、別の要求を送信するために、以前に送信された要求への応答を待つ必要はありません。HTTPリクエストのパイプライン処理とは異なり、aiohttp複数の接続を使用してリクエストを並行して送信します。これにより、上記の「FIFOの問題」が回避されます。使用方法は次のaiohttpとおりです。

import aiohttp
import asyncio


async def get(url):
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as response:
            return response


loop = asyncio.get_event_loop()

coroutines = [get("http://example.com") for _ in range(8)]

results = loop.run_until_complete(asyncio.gather(*coroutines))

print("Results: %s" % results)

上記のすべてのアプローチ（Session、ストリーム、concurrent.futuresまたはを使用asyncio）は、HTTPクライアントを高速化するさまざまな方法を提供します。

パフォーマンス


以下は、HTTPクライアントがサーバーにリクエストを送信するサンプルコードhttpbin.orgです。サーバーは、とりわけ、リクエストへの応答に長い時間がかかるシステム（この場合は1秒）をシミュレートできるAPIをサポートしています。ここでは、前述のすべての手法が実装され、それらのパフォーマンスが測定されます。

import contextlib
import time

import aiohttp
import asyncio
import requests
from requests_futures import sessions

URL = "http://httpbin.org/delay/1"
TRIES = 10


@contextlib.contextmanager
def report_time(test):
    t0 = time.time()
    yield
    print("Time needed for `%s' called: %.2fs"
          % (test, time.time() - t0))


with report_time("serialized"):
    for i in range(TRIES):
        requests.get(URL)


session = requests.Session()
with report_time("Session"):
    for i in range(TRIES):
        session.get(URL)


session = sessions.FuturesSession(max_workers=2)
with report_time("FuturesSession w/ 2 workers"):
    futures = [session.get(URL)
               for i in range(TRIES)]
    for f in futures:
        f.result()


session = sessions.FuturesSession(max_workers=TRIES)
with report_time("FuturesSession w/ max workers"):
    futures = [session.get(URL)
               for i in range(TRIES)]
    for f in futures:
        f.result()


async def get(url):
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as response:
            await response.read()

loop = asyncio.get_event_loop()
with report_time("aiohttp"):
    loop.run_until_complete(
        asyncio.gather(*[get(URL)
                         for i in range(TRIES)]))

このプログラムを開始した後に得られた結果は次のとおりです。

Time needed for `serialized' called: 12.12s
Time needed for `Session' called: 11.22s
Time needed for `FuturesSession w/ 2 workers' called: 5.65s
Time needed for `FuturesSession w/ max workers' called: 1.25s
Time needed for `aiohttp' called: 1.19s

HTTPリクエストを実行するためのさまざまなメソッドのパフォーマンスの調査結果

最も単純な同期リクエストの実行方式が最も遅いことが判明したのは当然のことです。ここでのポイントは、接続を再利用することなく、クエリが1つずつ実行されることです。その結果、10個のクエリを完了するのに12秒かかります。

施設の使用Session、およびその結果、化合物の再利用により、時間を8％節約できます。これはすでに非常に優れており、これを達成することは非常に簡単です。パフォーマンスを気にする人は、少なくともオブジェクトを使用する必要がありますSession。

システムとプログラムでスレッドの操作が許可されている場合、これはスレッドを使用してクエリを並列化することを検討するのに十分な理由です。ただし、ストリームはシステムに追加の負荷をかけるため、いわば「無料」ではありません。彼らは作成され、実行される必要があり、あなたは彼らの仕事の完了を待つ必要があります。

高速非同期HTTPクライアントを使用したい場合は、古いバージョンのPythonで記述しているのでない限り、最も注意を払う必要がありますaiohttp。これは、最速のソリューションであり、スケーラブルです。数百の同時リクエストを処理できます。特に良い代替案ではない

代替案はaiohttp、数百のスレッドを並行して管理することです。

ストリームデータ処理


アプリケーションのパフォーマンスを向上させるのに役立つ可能性のある、ネットワークリソースの操作のもう1つの最適化は、ストリーミングデータの使用です。標準のリクエスト処理スキームは次のようになります。アプリケーションがリクエストを送信し、その後、このリクエストの本文が一度に読み込まれます。streamライブラリがサポートするパラメータとライブラリrequests属性を使用すると、このスキームから離れることができます。 ストリーミング処理組織が使用している様子は次のとおりです。contentaiohttp

requests

import requests

#  `with`         
#     .
with requests.get('http://example.org', stream=True) as r:
    print(list(r.iter_content()))

を使用してデータをストリーミングする方法はaiohttp次のとおりです。

import aiohttp
import asyncio


async def get(url):
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as response:
            return await response.content.read()

loop = asyncio.get_event_loop()
tasks = [asyncio.ensure_future(get("http://example.com"))]
loop.run_until_complete(asyncio.wait(tasks))
print("Results: %s" % [task.result() for task in tasks])

数百メガバイトのメモリが無用に割り当てられる可能性を防ぐ必要がある場合は、完全な応答コンテンツを即座にロードする必要をなくすことが重要です。プログラムが全体として回答にアクセスする必要がない場合、回答の個々のフラグメントを処理できる場合は、リクエストを使用してストリーミング処理する方法に頼ることがおそらく最善です。たとえば、サーバーの応答からのデータをファイルに保存する場合、それらを部分的に読み取りおよび書き込む方が、応答本文全体を読み取り、大量のメモリを割り当ててすべてをディスクに書き込むよりも、メモリ使用の点ではるかに効率的です。
