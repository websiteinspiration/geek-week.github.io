Jestを使用してJavaScriptコードをダミー用にテストします。パート1

こんにちは、ハブル！このガイドは、Jestのような優れたテストフレームワークに関する一連の記事の最初の部分です。この資料は、初心者や、テストに精通したばかりで、このフレームワークを学習したいユーザーに役立ちます。最初の部分で分析します：jestの使用を開始する方法、簡単なテストを作成する方法、およびチェックされる値を期待される値と比較するためにどのような方法がありますか。誰も気にしない-キャットへようこそ！

ジェストとは？


プロジェクトのホームページに示されているように：
Jestは、シンプルさを重視したすばらしいJavaScriptテスト環境です。
そして確かに、ジェストはとてもシンプルです。追加の設定が不要で、理解と使用が簡単で、ドキュメントも非常に優れています。Node、React、Angular、Vue、Babel、TypeScriptなどを使用するプロジェクトに最適です。
また、オープンソースであり、Facebookによってサポートされています。

取り付け


プロジェクトにJestをインストールするには、次のようにします。

npm install --save-dev jest

糸を使用している場合：

yarn add --dev jest

インストール後、package.jsonのスクリプトセクションを更新できます。

“scripts” : {
     “test”: “jest”
}

この単純な呼び出しで、すでにテストを実行できます（実際、jestには少なくとも1つのテストが存在する必要があります）。

グローバルにインストールすることもできます（ただし、私にとってモジュールのグローバルインストールは悪い習慣であるため、これを行うことはお勧めしません）。

npm install jest --global

そしてそれに応じて糸：

yarn global add jest

その後、コマンドラインから直接jestを使用できます。

プロジェクトのルートでjest --initコマンドを呼び出すと、いくつかの質問に答えた後、jest.config.js設定ファイルを取得できます。または、package.jsonに構成を直接追加できます。これを行うには、「jest」キーをjsonルートに追加し、対応するオブジェクトに必要な設定を追加します。オプション自体は後で分析します。追加の設定なしでjestをすぐに使用できるため、この段階では必要ありません。

最初のテスト


first.test.jsファイルを作成して、最初のテストを作成しましょう。

//first.test.js
test('My first test', () => {
    expect(Math.max(1, 5, 10)).toBe(10);
});

そして、npm run testを使用して、またはjestコマンドで直接テストを実行します（グローバルにインストールされている場合）。起動後、テストの合格に関するレポートが表示されます。

 <b>PASS</b>  ./first.test.js
  ✓ My first test (1 ms)

Test Suites: 1 passed, 1 total
Tests:       1 passed, 1 total
Snapshots:   0 total
Time:        0.618 s, estimated 1 s

テストを中断してjestを再度実行してみましょう：

//first.test.js
test('My first test', () => {
    expect(Math.max(1, 5, 10)).toBe(5);
});

ご覧のように、テストはテストに合格していません。 Jestは、問題が発生した場所、予想される結果、および代わりに得られた結果に関する詳細情報を表示します。

次に、テスト自体のコードを分析しましょう。テスト機能新しいテストを作成するために使用されます。これは3つの引数を取ります（この例では、2つの引数を使用して呼び出しを使用しました）。最初の行はテストの名前を含む行で、その冗談はレポートに表示されます。 2つ目は、テストのロジックを含む関数です。 3番目の引数-タイムアウトを使用することもできます。これはオプションであり、デフォルト値は5秒です。ミリ秒単位で設定されます。このパラメーターは、非同期コードを使用してテスト関数からpromiseを返すときに必要です。これは、jestが約束が解決するまでどれだけ待つ必要があるかを示します。この後、約束が許可されなかった場合-jestはテストが失敗したと見なします。非同期呼び出しの操作の詳細については、次の部分で説明します。または、テスト（）の代わりに（）を使用できます。。このような呼び出しに違いはありません。it（）はtest（）関数の単なるエイリアスです。

テスト関数内では、最初にexpect（）を呼び出します。彼に、チェックしたい値を渡します。私たちの場合、これはMath.max（1、5、10）を呼び出した結果です。expect（）は、結果の値を期待される値と比較するためのいくつかのメソッドを持つラッパーオブジェクトを返します。これらのメソッドの1つはtoBeです。

これらのメソッドの主なものを見てみましょう：

簡単なテストをいくつか書いてみましょう。まず、サークルを操作するためのいくつかのメソッドを含む単純なモジュールを作成します。

// src/circle.js
const area = (radius) => Math.PI * radius ** 2;
const circumference = (radius) => 2 * Math.PI * radius;
module.exports = { area, circumference };

次に、テストを追加します。

// tests/circle.test.js
const circle = require('../src/circle');

test('Circle area', () => {
    expect(circle.area(5)).toBeCloseTo(78.54);
    expect(circle.area()).toBeNaN();
});

test('Circumference', () => {
    expect(circle.circumference(11)).toBeCloseTo(69.1, 1);
    expect(circle.circumference()).toBeNaN();
});

これらのテストでは、面積と円周の 2つの方法の結果を確認しました。toBeCloseToメソッドを使用して、期待される結果を確認しました。最初のケースでは、チェックしたか、半径5の円の計算された面積がおよそ78.54に等しいのに対し、取得した値（78.53981633974483となる）との差は大きくなく、テストがカウントされます。2番目では、小数第1位までのチェックに関心があることを示しました。また、引数なしでメソッドを呼び出し、toBeNaNを使用して結果を確認しました。それらの実行結果はNaNになるため、テストは正常にパスします。

別の例を見てみましょう。製品の配列を価格でフィルタリングする関数を作成してみましょう：

// src/productFilter.js
const byPriceRange = (products, min, max) =>
         products.filter(item => item.price >= min && item.price <= max);
module.exports = { byPriceRange };

そしてテストを追加します：

// tests/product.test.js
const productFilter = require('../src/producFilter');

const products = [
    { name: 'onion', price: 12 },
    { name: 'tomato', price: 26 },
    { name: 'banana', price: 29 },
    { name: 'orange', price: 38 }
];

test('Test product filter by range', () => {
    const FROM = 15;
    const TO = 30;
    const filteredProducts = productFilter.byPriceRange(products, FROM, TO);

    expect(filteredProducts).toHaveLength(2);
    expect(filteredProducts).toContainEqual({ name: 'tomato', price: 26 });
    expect(filteredProducts).toEqual([{ name: 'tomato', price: 26 }, { name: 'banana', price: 29 }]);
    expect(filteredProducts[0].price).toBeGreaterThanOrEqual(FROM);
    expect(filteredProducts[1].price).toBeLessThanOrEqual(TO);
    expect(filteredProducts).not.toContainEqual({ name: 'orange', price: 38 });
});

このテストでは、byRangePrice 関数の結果を確認します。最初に、結果の配列と予想される配列の長さの対応を確認しました-2。次の確認では、要素に要素が含まれている必要があります-{name： 'tomato'、price：26}。配列内のオブジェクトとContainEqualに渡されるオブジェクトは、同じオブジェクトへの参照ではなく、2つの異なるオブジェクトです。しかし、toContainEqualは各プロパティを比較します。両方のオブジェクトが同一であるため、検証は成功します。次に、toEqualを使用して、配列全体とその要素の構造をチェックします。toBeGreaterThanOrEqualとtoBeLessThanOrEqual方法は、私たちは、アレイの第一及び第二の要素の価格をチェックするのに役立ちます。そして最後に、not.toContainEqualを呼び出します要素が配列に含まれているかどうかを確認します-{name： 'orange'、price：38}。これは、条件により、そこに存在してはなりません。

これらの例では、上記の検証関数を使用していくつかの簡単なテストを記述しました。次のパートでは、非同期コードの操作、チュートリアルのこのパートでカバーされなかったjest関数、その構成などについて説明します。
