Docker-compose コンテナの準備ができるまで待つ方法

前書き


コンテナーの実行とdocker-compose.ymlの作成については、多くの記事があります。しかし、私にとって長い間、別のコンテナーがその要求を処理する準備ができるまで、またはある量の作業を完了する準備ができるまで、いくつかのコンテナーを起動しない場合、どのように正しく処理するかが問題でした。

この質問は、個々のDockerを起動するのではなく、積極的にdocker -composeを使用し始めた後に関連しました。

それはなんのためですか？


実際、コンテナBのアプリケーションがコンテナAのサービスの可用性に依存するようにします。起動時に、コンテナBのアプリケーションはこのサービスを受け取りません。どうする？

2つのオプションがあります。

  • 最初は死ぬことです（できればエラーコードを使って）
  • 2番目は、コンテナーBのアプリケーションが割り当てられたタイムアウトに応答しなかった場合、待機してからとにかく停止することです。

コンテナーBが停止すると、Docker-compose（もちろん構成によって異なります）が再起動し、コンテナーBのアプリケーションがコンテナーAのサービスに到達しようとします。

これは、コンテナーAのサービスが要求に応答する準備ができるまで続きます。コンテナが常にリロードされていることに気づくまで。
そして実際、これはマルチコンテナアーキテクチャの通常の方法です。

しかし、特に、コンテナーAが起動してコンテナーBのデータを準備する状況に直面しました。コンテナーBのアプリケーションは、データの準備ができているかどうかを確認する方法がわからないため、すぐにデータの処理を開始します。したがって、データの準備に関する信号を自分で受信して処理する必要があります。

あなたはまだいくつかのユースケースを与えることができると思います。しかし、最も重要なのは、これを行う理由を正確に理解する必要があることです。それ以外の場合は、標準のdocker-composeツールを使用することをお勧めします。

少しのイデオロギー


ドキュメントを注意深く読むと、すべてがそこに書かれています。つまり、各
コンテナは独立したユニットであり、使用するすべてのサービス
が利用できるように注意する必要があります。

したがって、問題は、コンテナーを開始するかどうかではなく、コンテナー
内で必要なすべてのサービスの可用性を確認するかどうか
、そしてコンテナーアプリケーションに制御を転送した後でのみです。

これはどのように実装されていますか


この問題を解決するために、docker-composeの説明が非常に役立ちました。この部分
と、エントリーポイントとcmdの正しい使用法についての記事が役立ちました。 だから私たちが得る必要があるもの：



  • コンテナAでラップした付録Aがあります。
  • 開始し、ポート8000​​でOKの応答を開始します
  • また、コンテナーBから開始するアプリケーションBがありますが、アプリケーションAがポート8000​​で要求に応答し始めるまでに、アプリケーションBは動作し始めます。

公式ドキュメントには、この問題を解決する2つの方法があります。

1つ目は、コンテナに独自のエントリポイントを書き込み、すべてのチェックを実行してから、作業中のアプリケーションを起動することです。

2つ目は、既に作成されたバッチファイルwait-for-it.shを使用することです。
私たちは両方の方法を試しました。

独自のエントリーポイントを書く


エントリーポイント とは？

これは、DockerfileのENTRYPOINTフィールドにコンテナーを作成するときに指定する実行可能ファイルです。このファイルは、すでに述べたように、チェックを実行してから、コンテナーのメインアプリケーションを起動します。

つまり、取得できるの

はEntrypointフォルダを作成することです。

これには、container_Aとcontainer_Bの 2つのサブフォルダーがあります。その中にコンテナを作成します。

コンテナーAの場合、Pythonで単純なhttpサーバーを使用します。開始後、ポート8000​​でget要求への応答を開始します。

実験をより明確にするために、サーバーを開始する前に15秒の遅延を設定します。

コンテナAの 次のdockerファイルがわかります。

FROM python:3
EXPOSE 8000
CMD sleep 15 && python3 -m http.server --cgi

コンテナーBの場合は、コンテナーB 用に次のdockerファイルを作成します。

FROM ubuntu:18.04
RUN apt-get update
RUN apt-get install -y curl
COPY ./entrypoint.sh /usr/bin/entrypoint.sh
ENTRYPOINT [ "entrypoint.sh" ]
CMD ["echo", "!!!!!!!! Container_A is available now !!!!!!!!"]

そして、entrypoint.sh実行可能ファイルを同じフォルダーに配置します。こんな感じになります

#!/bin/bash

set -e

host="conteiner_a"
port="8000"
cmd="$@"

>&2 echo "!!!!!!!! Check conteiner_a for available !!!!!!!!"

until curl http://"$host":"$port"; do
  >&2 echo "Conteiner_A is unavailable - sleeping"
  sleep 1
done

>&2 echo "Conteiner_A is up - executing command"

exec $cmd

コンテナBで何が起こっているのか：

  • 起動するとENTRYPOINTが起動します。entrypoint.shを起動します
  • entrypoint.shは、curlを使用して、コンテナーAのポート8000​​のポーリングを開始します。これは、200応答を受信するまでこれを実行します（つまり、この場合、curlはゼロの結果で終了し、ループが終了します）。
  • 200を受信するとループが終了し、$ cmd変数で指定されたコマンドに制御が渡されます。そして、それは私たちがCMDフィールドのDockerファイルで示したもの、つまり echo "!!! Container_Aが利用可能になりました!!!!!!!!" なぜそうなのか、上記の記事で説明されています
  • 印刷- !!! Container_Aが利用可能になりました!!! そして結論。

すべてをdocker-composeから始めます。

docker-compose.ymlここにこれがあります：

version: '3'
networks:
 waiting_for_conteiner:
services:
 conteiner_a:
   build: ./conteiner_A
   container_name: conteiner_a
   image: conteiner_a
   restart: unless-stopped
   networks:
     - waiting_for_conteiner
   ports:
     - 8000:8000
 conteiner_b:
   build: ./conteiner_B
   container_name: conteiner_b
   image: waiting_for_conteiner.entrypoint.conteiner_b
   restart: "no"
   networks:
     - waiting_for_conteiner

ここで、conteiner_aでは、ポートを指定する必要はありません：8000：8000。これは、外部で実行されているhttpサーバーの動作を確認できるようにするために行われました。

また、コンテナーBはシャットダウン後に再起動しません。

ローンチ：

docker-compose up —-build

wait-for-it.shの使用


ドキュメントで説明されているように、この方法ではうまくいかなかったことはすぐに言及する価値があります。
つまり、ENTRYPOINTとCMDがDockerfileに書き込まれている場合、コンテナーが起動すると、ENTRYPOINTからのコマンドが実行され、CMDの内容がパラメーターとして渡されることがわかっています。Dockerfileで指定されたENTRYPOINTとCMDはdocker -compose.ymlで再定義できること

も知られています。wait - for - it.sh 起動形式は次のとおりです。



wait-for-it.sh __ -- ___

示すように続いて、記事では、我々は新しい定義することができますENTRYPOINTでドッカ-compose.ymlを、そしてCMDがされますから、置換Dockerfile。

したがって、

コンテナAのDockerファイルは変更されません。

FROM python:3
EXPOSE 8000
CMD sleep 15 && python3 -m http.server --cgi

コンテナーBのDockerファイル

FROM ubuntu:18.04
COPY ./wait-for-it.sh /usr/bin/wait-for-it.sh
CMD ["echo", "!!!!!!!! Container_A is available now !!!!!!!!"]

Docker-compose.ymlは次のようになります。

version: '3'
networks:
 waiting_for_conteiner:
services:
 conteiner_a:
   build: ./conteiner_A
   container_name: conteiner_a
   image: conteiner_a
   restart: unless-stopped
   networks:
     - waiting_for_conteiner
   ports:
     - 8000:8000
 conteiner_b:
   build: ./conteiner_B
   container_name: conteiner_b
   image: waiting_for_conteiner.wait_for_it.conteiner_b
   restart: "no"
   networks:
     - waiting_for_conteiner
   entrypoint: ["wait-for-it.sh", "-s" , "-t", "20", "conteiner_a:8000", "--"]

wait-for-it コマンドを実行し、コンテナーAが稼働するまで20秒待機するように指示し、別のパラメーター“-”を指定します。これにより、wait-for-itパラメーターが、完了後に起動するプログラムから分離されます。

やってみます！
そして、残念ながら、何も得られません。

我々は待ち、それを実行しているものを引数に確認した場合にのみ、私たちがで指定されていること、そして、我々が表示されますエントリポイントが渡され、CMDコンテナからは付属しません。

作業オプション


次に、1つのオプションのみが残ります。DockerfileのCMDで指定したものは、docker -compose.ymlのコマンドに転送する必要があります。 次に、コンテナB のDockerfileを変更しないでください。docker -compose.ymlは次のようになります。



version: '3'
networks:
 waiting_for_conteiner:
services:
 conteiner_a:
   build: ./conteiner_A
   container_name: conteiner_a
   image: conteiner_a
   restart: unless-stopped
   networks:
     - waiting_for_conteiner
   ports:
     - 8000:8000
 conteiner_b:
   build: ./conteiner_B
   container_name: conteiner_b
   image: waiting_for_conteiner.wait_for_it.conteiner_b
   restart: "no"
   networks:
     - waiting_for_conteiner
   entrypoint: ["wait-for-it.sh", "-s" ,"-t", "20", "conteiner_a:8000", "--"]

そして、このバージョンでは動作します。

結論として、私たちの意見では、正しい方法が最初であると言わなければなりません。これは最も用途が広く、あらゆる方法で準備チェックを行うことができます。Wait-for-itは、個別に、またはentrypoint.shに埋め込むことで使用できる便利なユーティリティです。
