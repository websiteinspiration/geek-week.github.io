TDD：仕様を正しく記述する方法（説明）

プログラミングがほとんどないため、仕様の記述（アサーションまたはテスト仮説）は、TDDコースでは通常バイパスされます。

ただし、それらは非常に重要です。

彼らはどのようにそしてどのようなテストが書かれるかを決定し、また、故障を修正するのがどれほど簡単になるかを決定します。実装にほとんど時間を要しません。それらは、ストーリーのユーザーとコードをつなぐ最初のものであり、テストコードの失敗を示す最初のものです。それらはまた、テスト自体のコードのエラーから私たちをさらに保護するものでもあります。

どうすればステートメントを改善できるでしょうか？

まず、仕様はユーザーストーリーに関連している必要があります。ストーリーのユーザーが「ログイン」という用語を使用している場合、assertはこの用語を変更して認証/承認/検証することはできません。ストーリーがうまく書かれていない場合は変更してください。

次に、仕様には、それらが記述するコンポーネントの名前を明示的に含める必要があります。これは、テスト領域を示し、突然アセンブリに予期しないコンポーネントが含まれることが多い場合、エラーを通知します。

有効なスクリプトは単一責任の原則とうまく組み合わせるので、コンポーネントの名前はテストの説明と一致する必要があります。多くの有効なシナリオがある場合、おそらくコンポーネントの責任が大きすぎます。

また、1つのシナリオは、原則として、UIイベントまたは何らかのパターンのいずれかを記述します。コンポーネント名の適切な候補には、パターンの名前（Validator、Strategy、Builder、Transformer、Controllerなど）を含めることができます。またはイベント名：送信者、ログイン、ReadonlyOrderViewなど。それら。コンポーネントの名前に応じて、クラスのメイン関数の入力値と出力値の両方を定義する必要があります。悪い名前：抽象的すぎる（サービス、コンポーネント、ヘルパー、ユーティリティ）および二重（ValidatorAndRenderer）。

3番目に、仕様では条件を明示的に記述する必要があります。

不完全に：

@Test public void testValidatePassword(){}


それは良いです：

@Test public void loginController_whenValidUsername_andValidPassword_shouldLogUserIn(){}


別の場所からこの関数を呼び出す必要がないため、エクストラロング名が有効です。テストの条件の数が行に収まらない場合は、コンポーネントの構造を変更する必要があります。

javascriptのテストも同じですが、そこに条件を置くことができれば、より便利になります。
describe('login UI component', () => {
    describe('when username provided', () => {
        describe('when valid password', () => {
            it('should log user in', () => {
                ...
            });
        });
    });
});


明示的な条件の方が読みやすく、欠落していることがわかります。

さらに、条件が書き込まれたテストコードと一致しない場合、コードは簡単に修正できます。

一般的に、適切な説明は、ドキュメントとコードのコメントの代わりです。それは抽象的で、フレームワークとコンパイラに依存せず、最後の2つと同じ深刻さで扱われるべきです。

たとえばGIVEN-WHEN-THEN-SHOULDなど、読みやすいように、条件自体がコード内で一貫している必要があります。

第4に、仕様を合理化する必要があります。それらを逆の順序で書くと便利です：

最初にエラー、最初にヌル、最後にハッピーパス。その場合、1つの有効なユーザーケースの後でテストを終了する必要はありません。 UIコンポーネントの場合：レンダリング、イベント。ステートフルコンポーネントの場合、すべての状態は一貫しています。

したがって、読みやすい仕様の構造は次のようになります。5-10エラー、1つの有効なスクリプト（または1つのスクリプトのいくつかのバリエーション）



仕様を書くときは、仕様を全文で書くことをお勧めします。その後、実際には、すでに実装の記述を開始できます。多くの場合、この形式では、タスクは疎外されます。まあ、誰か他の人がテスト実装を書いたとき。

第五に、テストの実装は、describeに記述されているものと一致する必要があります。
実装のいずれかの部分との一致を簡単に見つけるために、arrange-act-assertに分割することをお勧めします。アサーションと例外のテキストが存在する場合、それらもテストの説明と関連付ける必要があります。原則として、それらを単純に複製できます。

言うまでもなく、テストに条件またはアサーションを追加するときは、テストの説明も更新する必要があります。

したがって、仕様書を作成する順序は次のようになります。

  • ユーザーストーリーを読む
  • 名前コンポーネントと有効なスクリプト
  • 有効なシナリオの条件を策定する
  • 誤ったスクリプトを上に配置して、仕様を補足する
  • 隣人に行方不明者を読んで修正させましょう
  • red-green-refactorループを使用して、テストとコンポーネントを1つずつ実装します。
