サーバーレスの価格とコスト：AWS Lambda

こんにちは、ハブル！私は翻訳をより良くするよう努めており、あなたの批判を喜んで受けます。
（注：価格は2018年8月の時点で正確です）

AWS Lambdaには、固有の価格タグが付いた263のラインアイテムがあります。たとえば、「米国、ヨーロッパ、または日本以外のエッジロケーションを使用したAccelerated InterRegion Inbound、From EU（Paris）to EU（London）」と呼ばれ、17の地域のそれぞれに存在します。合計すると、これらはLambdaの完了時に請求できる4,471の個別のポジションです。

さらに、EC2サービスには635,035、Route53には12,261、APIゲートウェイには15,283のポジションがあります。合計すると、Amazon Web Servicesはさまざまなサービスに対して100万を超える個別料金を持っています。彼らの価格設定ポリシーは非常に複雑なので、価格を確認するためだけのAPI全体があります。これで、これらの数値がわかりました（ただし、ラムダは価格計算ツールに含まれていません）。

Lambdaの価格は非常に重要です。最初に、最初の100万回のリクエストの無料レベルが使用され、その後に続く100万回のリクエストごとに$ 0.20の価格タグが追加されます。これは、LambdaにHTTPリクエストを送信するためにGateway API（100万リクエストあたり3.50ドル）がかかります。Lambdaのデータ転送（GBあたり0.09ドル）、およびDynamo DBへのクエリに対しても課金されます。

「コンピューティング時間」とは何ですか？


Lambdaコンピューティング時間のGB秒あたり$ 0.00001667が支払われることがわかりました。 GB秒はちょっと変わった単位ですよね？ 1 GBのメモリを使用するときの秒数を意味する場合もありますが、実際には、計算が実行されたときの秒数に、これらの計算に割り当てられたGB数を掛けたものです。

測定の単位としてマイクロセンターを好まない人のために、100万リクエストあたり1〜6ドルでどのくらいのメモリとCPU時間を取得できるかを示すグラフに数値をまとめました。Amazon



は、基本料金0を含まない価格を公開しています、20ドルですが、計算に含めました。

そして、1ミリ秒のプロセッサ時間とは何ですか？ Apple IIでのミリ秒の計算は、3,000プロセッササイクルを表しています。私のIntel i7ラップトップでは、400万サイクルです。 Amazonの場合、使用するメモリの量に応じてプロセッサの速度が向上します。つまり、パフォーマンスはメモリ* CPUサイクルタイムであり、使用されるメモリに比例してCPUサイクルタイムが減少します。このCPUアクセラレーションは、1つのプロセッサコアのリソースがなくなるまで続きます。その後、アプリケーションのビルドによっては、2番目のコアのリソースが提供されます。

これらの詳細に基づいて、グラフを調整して、実際に得られている実際のCPU時間を示します。比較テストで使用した通常のサーバープロセッサに注目しました。



図からわかるように、Lambdaを1024 MBから100ミリ秒で実行しても、128 MBから800ミリ秒で実行しても、同じ量の計算が行われ、同じ料金がかかります。1024 MBを超えると、1つのプロセッサコアのパフォーマンスを達成し、2番目のコアの能力を享受し始めます（アプリケーションがマルチスレッドであり、それを使用できる場合）。
 

丸め


さらに、プロセッサー時間は最も近い100ミリ秒に丸められます。つまり、テストで現在の時間を返す単純なラムダでさえ、呼び出しには$ 0.41 /ミリオンコールがかかります。もちろん、Lambdaが100の倍数である正確なミリ秒数を使用することはめったにありません。より頻繁には、Amazonが丸めるミリ秒を使用して、支払う金額を計算します。それらは常に切り上げられるため、コールごとに平均50ミリ秒のCPUに対して過剰に支払うことになります。



APIゲートウェイ


AWS Lambdaは、Amazon内でさまざまなタスクを実行できます。しかし、まず第一に、サーバーなしでWebアプリケーションを作成する上で興味深いものです。そして、この目的のために、ラムダを直接使用することはできません。それは、リクエストを Lambda API の呼び出しに変換できる何らかのHTTPサーバーが必要だからです。これを行う最も簡単な方法は、Amazon Gateway APIを使用することです。

API Gatewayのコストは100万リクエストあたり$ 3.50で、これは多くの場合Lambda自体のコストを超えます。これが上のグラフですが、API Gateway用に調整されています。1、2、3


ドルの線がグラフから消えていることに気づくでしょう。 LambdaがCPUをほとんど消費しない場合、絶対最小値は$ 3.91 /ミリオンリクエストです。
キャッシュを使用する場合は、トラフィックと追加の時間料金も支払う必要があります。

ルート53


Amazon Route53 DNSも無料ではありません。最初の10億のクエリのコストは、100万のDNSクエリあたり$ 0.40です。これにより、総コストは$ 4.31 /百万になります。

その他の費用


すべてのコストを計算するのは面倒です。CloudWatchログのコストは、GBあたり0.50ドルです。同時に、GBあたり0.09ドルのトラフィックがアカウントに大きな影響を与えます。

ラムダ@エッジ


Amazonは、Cloudfront CDNサービスを通じてコードを実行する方法を提供しています。一定で変動するコストは、従来のラムダの3倍です。

HTTPSリクエストでLambda @ Edgeを使用するには、最初に100万ドルのリクエストでCloudfrontを購入する必要があります。また、簡単に主要なコスト項目になる可能性がある帯域幅に対しても支払います。サイトのサイズが1 MBの場合、Cloudfrontの帯域幅のみで最大$ 85/100万、南米などの地域では最大$ 250を支払うことになります。さらに、送信者に返送しなければならない100万リクエストごとに追加の20ドル。
