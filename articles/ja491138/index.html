衛星とAnsible Towerの統合

Red Hat SatelliteおよびRed Hat Ansible Automation Platformを使用していますか？ Satellite 6.3以降では、それらを相互に統合できるため、Ansible Towerの動的インベントリがSatelliteからホストリストをプルアップします。さらに、RHELホストがサテライト（プロビジョニングを意味する）を使用して初期化されている場合、Ansible Towerをこのプロセスに統合して、新しいホストで構成スクリプトを自動的に実行できます。



この投稿では、Ansible TowerでDynamic Inventoryを構成してSatelliteからホストをプルアップする方法と、例を使用してその使用方法を説明します。さらに、Satelliteから新しいホストを初期化した後、Ansible Towerへの自動通話を整理する方法も示します。

1. Inventory. Satellite, Dynamic Inventory Ansible Tower


Ansible Towerがホスト、ホストグループ、およびその他の関連情報のリストにアクセスできるようにするには、Satelliteのアカウントが必要です。このエントリには十分な最小限の権限があるため、Satelliteで新しいロールを作成し、Ansible Towerに必要な権限のみを付与してから、新しいアカウントを作成してこのロールを割り当てます。

Satellite 6.6以降には、Ansible Tower Inventory Readerの役割がすでに用意されているため、以下の手順をスキップして役割を作成できます。

Satellite 6.3-6.5では、ロールは手動で作成する必要があります。これを行うには、Satellite Webインターフェースに移動し、管理画面に移動し、ロールを選択して、ロールの作成をクリックします。

このロールにansible_tower_integration_roleという名前を付け、場所と組織を設定します。

サテライトに関連付けられた動的インベントリーの使用


そこで、SatelliteとAnsible Towerを同期しましたが、実際にどのように使用するかを検討します。

最も簡単な方法は、Ansible TowerテンプレートでInventoryソースとしてsatellite_inventoryを使用することです。hosts：allオプションがスクリプトで指定されている場合、スクリプトはすべてのサテライトホストで実行されます。

より高度なオプションは、スクリプトで自動的に作成されたインベントリグループを使用することです（作成方法-上記を参照）。つまり、目的のホストグループをhosts行に指定します。たとえば、screenパッケージがインストールされているこのシナリオのように：

---
- name: Install screen package
  hosts: "foreman_hostgroup_rhel6"
  tasks:
  - yum:
      name: screen
      state: installed

ここでは、ホストforeman_hostgroup_rhel6を登録しました。これにより、サテライトでrhel6ホストグループを形成するホストのリストが示されます。したがって、スクリプトはこれらのホストでのみ実行されます。

さらに、スクリプトでは、Ansible TowerがSatelliteとの同期中に受信する変数をhosts行で指定できます。たとえば、これを行うことができます：

---
- name: Install screen package
  hosts: "{{ hosts_var }}"
  tasks:
  - yum:
      name: screen
      state: installed

その結果、Ansible Towerのジョブテンプレートをオンザフライで変更できるようになり、外部変数を通じて在庫グループの1つを示します。



この例では、テンプレートはSatelliteのrhel7グループのメンバーであるホストでのみ実行されます。

さらに、ジョブテンプレートは、起動時にhosts_var変数の値をユーザーに要求するように構成できます（同時に、使用可能なインベントリグループをコメントの形式で表示します）。



上の画面は、テンプレートを開始するときに、スクリプトを実行するホストが属しているサテライトインベントリグループの名前を入力するように求められる状況を示しています。

さらに、同期中にSatelliteからプルされたホスト変数を使用できます。たとえば、これらの変数を参照する方法を示すスクリプトは次のようになります。

---
- name: Show Satellite variables
  hosts: all
  tasks:
  - name: Show subscription_status
    debug:
      msg: >
        Subscription Status: {{ foreman.subscription_status_label }}
  - name: Show Errata Counts
    debug:
      msg: >
        Bug fixes: {{ foreman.content_facet_attributes.errata_counts.bugfix }},
        Security: {{ foreman.content_facet_attributes.errata_counts.security }},
        Enhancement: {{ foreman.content_facet_attributes.errata_counts.enhancement }},
        Total: {{ foreman.content_facet_attributes.errata_counts.total }}

このスクリプトをAnsible Towerで実行すると、変数の値が表示されます。



そしてもちろん、これらの変数は条件付き構文でいつでも使用できるため、特定の条件下でのみタスクが起動されます。たとえば、ホストにセキュリティパッチがない場合や、ホストのサブスクリプションが無効な場合などです。

パート2.プロビジョニングコールバックによる新しいホストの自動構成


他の多くの機能に加えて、Satelliteはホストを初期化する、つまりプロビジョニングを実行することもできます。 Ansible Towerは、ホストを構成するように設計されています。統合により、Satelliteを使用して新しいRHELホストを初期化した後、Ansible Towerがこのホストに自動的に接続し、それに対応する構成スクリプトを実行することを確認できます。明らかに、このようなスキームはシステム管理者の時間を大幅に節約し、組織のニーズにより迅速に対応するのに役立ちます。

このスキームは、いくつかの条件が満たされた場合にのみ機能することに注意してください。まず、Ansible TowerでDynamic InventoryデータソースとしてSatelliteを使用する必要があります（前のパートでこの問題を取り上げました）。次に、ホストの初期化は、Satelliteだけでなく、ホストグループメカニズムを介して実行する必要があります（詳細については、プロビジョニングガイドを参照してください）。

以下では、ホストの初期化後にAnsible構成スクリプトが自動的に実行されるように、SatelliteおよびAnsible Towerを構成する方法を示します。

概観


Ansible TowerなどのIT自動化ツールは通常、2つのカテゴリーの1つに分類されます。プッシュ方式による作業とプル方式による作業があります。 Ansible Towerが属するプッシュシステムでは、オートメーションサーバーがホストへの接続を開始します。プルシステムでは、イニシエーターはホストであり、それ自体がオートメーションサーバーと通信します。

私たちの場合、新しいホストでの自動スクリプトでは、混合スキームが使用されます。新しく初期化されたホストは、オートメーションサーバーに「コールバック」して構成する要求を送信します。その後、オートメーションサーバーはこのホストに接続し、ホストから要求された構成スクリプトを実行します。したがって、Ansible Towerでは、このメカニズムはプロビジョニングコールバックと呼ばれ、「初期化コールバック」として変換できます。

プロビジョニングコールバックがまったく必要な理由を示すために、次の状況を検討してください。毎日午前0時に実行される特定のホスト構成スクリプトがあるとします。午前8時に、Satelliteを介して複数の新しいサーバーを一度に初期化するとします。同時に、SatelliteはAnsible Towerの動的インベントリデータソースとして使用されるため、作成されたホストは自動的にAnsibleに分類されます。ただし、自動化スクリプトの次の起動は、真夜中、つまり16時間後にのみ行われます。明らかに、これは理想とはほど遠いので、新しいホストの初期化の直後にスクリプトを実行したいと思います。この目的のために、プロビジョニングコールバックも考えられます。

一般に、プロビジョニングコールバックは次のように構成され、機能します。

Ansible Tower Provisioning Callback


/ etc / motdの内容を変更し、ある種のユーザーを作成し、特定のパッケージをインストールする構成スクリプトをAnsible Towerサーバーで作成することから始めましょう。次に、このスクリプトを使用して、初期化された直後に新しいホストを構成します。

実際には、自動化スクリプトは一部のバージョン管理システムに格納されている可能性が高いですが、簡単にするために、スクリプトはAnsible Towerサーバーに直接配置されていると想定しています。したがって、SSH経由でAnsible Towerサーバーに入り、次のコマンドを実行して、ディレクトリ/ var / lib / awx / projects / provisionを作成します。

# mkdir /var/lib/awx/projects/provision

次に、このディレクトリにスクリプトを作成し、provision.yamlと名付けて、次の内容を書き込みます。

---

- name: Provision new host

  hosts: all

  tasks:

  - name: Set content in /etc/motd 

    copy:

      content: Authorized use only!

      dest: /etc/motd

      mode: 644

      owner: root

      group: root


  - name: Create brian user account

    user:

      name: brian

      uid: 10000

      state: present


  - name: Install tmux package

    yum:

      name: tmux

      state: present

次に、TowerのWebインターフェイスに移動し、[資格情報]画面に移動します。緑色のプラス記号ボタンをクリックして、新しい認証情報エントリを作成し、名前をprovisioning_rootにして、認証情報タイプをマシンに設定します。ユーザー名フィールドにrootと入力し、パスワードフィールドに、Satelliteで対応するホストグループの新しいホストで使用するためにSatelliteで指定されているパスワードを指定します（その名前は、オペレーティングシステムタブのSatellite Webインターフェースにあります）。このエントリを使用すると、Ansible Towerは、Satelliteを介して初期化中に作成された新しいホストにログインできます。



その後、Ansible Towerインターフェースで[プロジェクト]画面に移動し、緑色のプラス記号をクリックして新しいプロジェクトを作成します。それをプロビジョニングと呼び、SCMタイプを手動に変更し、プロビジョニングをプレイブックディレクトリとして指定します。



次に、Ansible Towerインターフェースで[テンプレート]画面に移動し、緑色のプラス記号をクリックして新しいジョブテンプレートを作成します。これを「プロビジョニング」と呼び、「在庫」フィールドに「サテライト」、「プロジェクト」フィールドに「プロビジョニング」、「プレイブック」フィールドに「provision.yaml」、「資格情報」フィールドに「provisioning_root」と書き込みます。さらに、「プロビジョニングコールバックを許可する」チェックボックスを有効にし、「ホスト構成キー」フィールドの右側にある魔法の杖アイコンをクリックして、秘密鍵を生成する必要があります。このキーは、プロビジョニングコールバックをリクエストするために必要であり、誰でもない可能性がありますが、ホスト構成キーを知っている人だけが必要です。正しいキーがないと、Ansible Towerサーバーはリクエストに応答しません。



次に、将来のホスト構成キーの値と、このジョブテンプレートのURLから確認できるテンプレートIDを覚えておく必要があります。 

https://tower.example.com/#/templates/job_template/11

この例では、IDは11です。

コールバックをプロビジョニングするためのサテライトの構成


次に、サテライトのWebインターフェイスに移動し、[構成]画面に移動して、[ホストグループ]をクリックします。新しいホストを初期化するときに使用される既存のホストグループを選択します。この例では、これはRHEL 8です。

ホストグループの編集画面で、[パラメーター]タブに移動し、4つの新しいパラメーターを作成します：

ansible_host_config_key-ここで、Ansible Towerテンプレートからホスト構成キーを入力します。
ansible_job_template_id-ここでは、以前にテンプレートURLから見つけたテンプレートIDを記述します。
ansible_tower_fqdnは、Ansible Towerサーバーの完全修飾ドメイン名です。
ansible_tower_provisioning-trueに設定します。



すべての仕組みを確認する


次に、Satelliteから新しいホストを初期化し、プロビジョニングコールバックが機能し、そのホストでスクリプトが自動的に実行されることを確認します。

SatelliteのWebインターフェースで、[Hosts]画面に移動して[Create Host]をクリックし、4つの新しいパラメーターを作成したばかりのグループ、つまりRHEL 8グループが使用されていることを確認します。



ホストの初期化が完了したら、その上のテンプレートが正常に起動されたことを確認する必要があります。これを行うには、Ansible Tower Webインターフェースの[テンプレート]画面に移動し、リストでテンプレートを探して、インジケーターの左端の四角が緑色に変わるかどうかを確認します。



この緑色のボックスをクリックして、起動の詳細を表示します。今回の場合は成功しています。このタスクには制限（制限）があることに注意してください-初期化中に作成される新しいホストのみの起動。まあ、プロビジョニングコールバックメカニズムはテンプレートを要求するホストでのみテンプレートを実行するので、これは当然のことです。



次に、新しく作成したホストに移動し、スクリプトに記述されているものと一致するかどうかを確認します。

[root@provision-test-rhel8 ~]# rpm -qa | grep tmux
tmux-2.7-1.el8.x86_64

[root@provision-test-rhel8 ~]# id brian
uid=10000(brian) gid=10000(brian) groups=10000(brian)

[root@provision-test-rhel8 ~]# cat /etc/motd
Authorized use only!

トラブルシューティング


Ansible Towerテンプレートが、Satelliteを介して初期化された新しいホストで自動的に起動しない場合は、確認することがいくつかあります。

最初のステップは、Ansible Towerがプロビジョニングコールバックのリクエストを受信したかどうか、スクリプトがどのように実行されたか、およびスクリプトがまったく開始されたかどうかを確認することです。これを行うには、Towerインターフェースで[テンプレート]画面に移動し、テンプレート名の横にあるインジケーターの色を確認します。緑-起動に成功しました。赤-起動に失敗しました。



正方形が赤の場合は、クリックして追加情報を探し、起動が失敗した理由を理解します（これらはスクリプトの構文エラーとその他の問題の両方である可能性があります）。

ボックスが緑色の場合は、タスクが正常に実行されたことがレポートされても、実際にはホストで開始されなかったので、クリックして検索します。たとえば、スクリプトのhosts行に特定のホストとグループが示されているが、初期化中に作成されたホストが含まれていない場合、タスクは成功を報告しますが、追加情報には「スキップ：ホストが一致しません」と表示されます。

インジケーターの四角が緑でも赤でもない場合は、まずサテライトのホストグループのパラメーターを確認する必要があります。 Ansible TowerサーバーのURL、ホスト構成キー、テンプレートIDが正しく指定されていることを確認してください。

次に、新しく初期化されたホストに移動し、ファイル/etc/systemd/system/ansible-callback.service（RHEL 7または8システムの場合）またはファイル/root/ansible_provisioning_call.sh（RHEL 6）があることを確認します。

ファイルが存在しない場合は、ansible_tower_provisioningパラメーターがこのホストグループに対してtrueに設定されていること、およびサテライトサーバーの初期化パターンが変更されていないことを確認してください。
ファイル/etc/systemd/system/ansible-callback.serviceがホスト上に存在する場合は、それを開いて、プロビジョニングコールバック要求が送信されたURLを探します。

[root@provision-test-rhel8 ~]# cat /etc/systemd/system/ansible-callback.service
[Unit]
Description=Provisioning callback to Ansible Tower
Wants=network-online.target
After=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/bin/curl -k -s --data "host_config_key=aa5ebe82-491c-4fbb-bd36-a6657549451e" https://tower.example.com/api/v2/job_templates/11/callback/
ExecStartPost=/usr/bin/systemctl disable ansible-callback

[Install]
WantedBy=multi-user.target

curlを実行し、このファイルで指定されたコマンドを手動で複製して、プロビジョニングコールバックを手動で開始できます。

[root@provision-test-rhel8 ~]# /usr/bin/curl -k -s --data "host_config_key=aa5ebe82-491c-4fbb-bd36-a6657549451e" https://tower.example.com/api/v2/job_templates/11/callback/

無効なホスト構成キーが指定されている場合、次のエラーメッセージが表示されます。

[root@provision-test-rhel8 ~]# /usr/bin/curl -k -s --data "host_config_key=wrong-key-here" https://tower.example.com/api/v2/job_templates/11/callback/

{"detail":"You do not have permission to perform this action."}

テンプレートIDが正しく指定されていない場合（ここでは-11ではなく43）、答えは次のようになります。

[root@provision-test-rhel8 ~]# /usr/bin/curl -k -s --data "host_config_key=wrong-key-here" https://tower.example.com/api/v2/job_templates/43/callback/

{"detail":"Not found."}

次の例のように、curlコマンドでサイレント（-s）モードを無効にして、ホスト名の解決に関連するエラーを表示することもできます。 ：

[root@provision-test-rhel8 ~]# /usr/bin/curl -k  --data "host_config_key=wrong-key-here" https://ansible-tower.example.com/api/v2/job_templates/11/callback/

curl: (6) Could not resolve host: ansible-tower.example.com

さらに、エラーの原因を見つけるときに、Ansible Towerサーバーの/var/log/tower/tower.logファイルが役立つ場合があります。
たとえば、次のファイルは無効なホスト構成キーの使用を記録しています。
