1Cで使用した場合のIBM DB2 Express-C管理エクスペリエンス：エンタープライズ

IBM DB2で作業する機会がありました。 1CとDjango上のサーバーの両方がこのDBMSを一度に使用したため、OLAP要求は非常に迅速に処理されました（もちろん、応答が2秒以内になるように、インデックスとWebサーバーを手動で構成する必要がありました）。 2015年、私は忘れないようにこの短い説明を自分のために用意しました。そして、履歴書への追加として、おそらく誰かが紙で読んだかもしれません。 DB2での私の経験のいくつかの一般化。私はそれを少し修正し、ここを読んで私の視野を広げることをお勧めします。多分誰かが興味を持つでしょう。それ以来、私はDB2を使用していないので、すべてが忘れられているとすぐに言っておく必要がありますが、それでも何か覚えています。批判や説明は大歓迎ですが、私はもう仕事をしていないので、多分私には向かないかもしれませんが、他の誰かには役立つでしょう。

1C：EnterpriseとIBM DB2 DBMSの作業を整理する方法については、インターネット上に多くの説明があります。手始めに、これはまったく悪くはありませんが、実稼働での使用には十分ではありません。以前は、IBM DB2からBig Data University DB101RUトレーニングコースを始めることをお勧めしました。私はこのコースを受講し、2012年に試験に合格しました。非常に役立つと思います。資源がなくなったのは残念だ。新しいプラットフォームでは、そのようなものは見つかりませんでした。本番環境では、IBM DB2に追加の構成とメンテナンスが必要です。ここで最も必要なものについて簡単に説明します。 Windowsバージョン10.1fp2および10.5fp4のIBM DB2 Express-Cの無料版を検討しています（1つ目は1Cがテストモードで動作するためにサポートされ、2つ目は正式にサポートされています。新しいバージョンは有料のみです）。 64ビット10.5をそこにインストールすることは理にかなっています、固定サイズの長い文字列を含む複合型を使用する場合、RAM（速度のバッファサイズ）または記録サイズ（EXTENDED_ROW_SZ = ENABLE）の要件が高い場合。

実行する必要がある最初のことは、1C：Enterpriseの操作を中断せずにバックアップを作成するためにアーカイブされたジャーナルの使用に切り替え、いつでもデータベースを復元できるようにすることです（10.1fp2への復元には、未修正のため独自の特性があります）無料版のバグ-ログファイルの手動での移動が必要です）。 MS SQLとは異なり、ログのアーカイブは特定の所定の時点では実行されません（MS SQLでは強力ではなく、おそらく他に何かがある）。ログファイルが特定の所定のサイズに達した場合、復元操作の前にログをバックアップする必要はありません。ベースを非アクティブ化するだけです。ログを2方向にアーカイブすることは簡単に構成でき、その1つは例えばネットワークドライブ上にあります。さらに、ネットワークが短時間停止した場合、アクティブなログが占めるスペースの増加はそれほど大きくありません。アクティブログの場合、データベースをいつでも復元できるように、十分な空き領域を提供する必要があります。プログラマーが1Cベースで作業しているときに、さまざまな時点で頻繁なリターンが必要な場合、1つのバックアップで復元するのに十分であり、回復用のログファイルの選択は非常に簡単です。インスタンスの開始時に必ずデータベースをアクティブ化してください。そうしないと、暗黙的なアクティブ化を伴う多数の小さなログファイルが取得されます。もちろん、バックアップの保存期間を設定し（少なくとも2か月はログを保存する必要があるようです）、自動削除を構成する必要があります。ベースとバックアップ（ログ）は異なる物理ディスク上にある必要があります。極端な場合は、ローカルネットワーク上の別のコンピューターにバックアップできます。

別の理由でベースのアクティブ化も必要です。通常の操作では、Windowsをオンラインおよびオフライン保守でインストールする必要があります。このとき、ベースはアクティブになっているはずです。定期的に、データベースの履歴を調べて、オフラインウィンドウ中に実行されたアクションを確認する必要があります。現時点では、1Cの重いメンテナンスタスクがないため、オフラインメンテナンスウィンドウは22:00〜0:00の間に設定する必要があります。おそらく小さな基地では、1時間続くウィンドウで十分です。

定期的に、手動モードで再編成の必要性のチェックを実行し、テーブルとインデックスの状態を分析した後、個々のオブジェクトの再編成を実行する必要があります。数千のテーブルとインデックスを手動で再編成すると、時間がかかる場合があります。分析は、正規表現を使用した単純なフィルター（.jsなど）によって簡単に加速されます。

もちろん、AUTOCONFIGUREは定期的に実行する必要があります。たとえば、独自のログファイルサイズの設定など、個々の設定を手動で修正します。

1日後、データベースのオンラインバックアップを実行することが可能になり、頻繁に実行されます（シャットダウンやDBAの存在は必要ありません）。頻度は必要な回復時間に依存します。これは、最後のバックアップ後のアーカイブログファイルの数、つまりデータベースの負荷に依存します。中規模、大規模、高負荷のデータベースの場合、さまざまなタイプの増分バックアップを使用して、バックアップが占めるスペースを減らし、バックアップからのリカバリ時間を短縮することは理にかなっています。各バックアップの後、復元する前に、バックアップの状態を確認してください。バックアップの保存時間はDBAの裁量によるが、ログの保存時間以上である。

少なくとも月に1回、オフラインおよびオンラインのデータベースヘルスチェックを実行し（オフラインモードでは、データベースの操作は数分間中断されます）、必要に応じて修復を実行します（UPSのない「サーバー」にとって最も重要です）。データベースのオフラインバックアップを毎月実行します。DB2のバージョンを変更するとオンラインバックアップを展開できないため、長期間保存されるのはオフラインバックアップのみです。 「1C：Enterprise」が検証またはバックアップのためにオフラインでのデータベースのオフライン変換さえ許可しない場合、データベースの拡張コピーで示されたアクションを実行することが可能です。データベースは、バックアップから別の場所、たとえば別のサーバー上の別のディスクに簡単に復元できます。バックアップとして、アーカイブログは、DB2ツールを使用して圧縮することもできます（この場合、バックアップチェッカーは動作したままで、アーカイブログチェッカーは機能しません）。

データベースとオフラインバックアップをオフラインでチェックする前に、データベースとメンテナンスタスクをロックするように設定する必要があります。緊急時にはデータベースの安定化を行うことができますが、1C：Enterpriseサーバーを実行しているユーザーがSYSADM_GROUPグループに含まれているため、1Cをデータベースに誤って接続する機能がキャンセルされず、その結果、2番目が必要になります。ジョブの立ち上げ。

データベースが高速に動作していない場合は、統計を更新した後、最も困難なクエリの計画を取得し、データベースのコピーで1Cインデックスを試し、IBM Data Studioでクエリプランへの変更を追跡します（この場合、Eclipseを使用することは正当化されますが、他の場合はコマンドインターフェイスで十分です）行）、またはDB2設計アドバイザーの推奨事項を使用し、必要に応じて、1C外で手動で索引を作成します。同時に、DB2ツール（12を超えるSQLスクリプト）を使用して詳細なパフォーマンス情報の収集を開始し、システムモニターで負荷を分析します。ディスクシステムの負荷を軽減するには、十分なサイズの個別の高速ディスクにデータベースをインストールする必要があります（もちろん、通常のサーバーRAID 10アレイがある場合を除きます）。OSとともにSSDにアクティブログを配置することが可能です。おそらく、1C：エンタープライズサーバーのテンポの場所の変更も必要になります。ディスクの購入のみを計画している場合、小規模な組織では、データベースに単一の物理ディスクを一時的に使用することが許可されます。

db2diag.logを毎日確認して、エラーやデータベースでのアクションの結果を確認します。数十メガバイトのサイズに達したらアーカイブします。 Far Managerはログを表示するのに便利な方法です（データベースにはエラーがほとんどないと想定されています）。また、アーカイブログを手動で移動して特定の時点で復元する必要がある場合にも役立ちます。

IBM DB2の競争上の利点の1つは、私には思えますが、MS SQL Serverが通常の操作に64ビットの「1C：Enterprise」サーバーを必要とする場合、32ビットのIBM DB2で実行できると考えられます。
