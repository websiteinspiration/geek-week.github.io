PHP 7.4でのプリロード

この記事の翻訳は、特にバックエンドPHP開発者コースの受講生を対象としています。



PHP 7.4には、コードのパフォーマンスを大幅に改善できる機能であるプリロードが追加されています。

, Opcache


はい、プリロードはopcacheに基づいていますが、まったく同じというわけではありません。 OpcacheはPHPソースファイルを取得し、それらをopcodeにコンパイルして、コンパイルしたファイルをディスクに保存します。

オペコードは、実行時に簡単に解釈されるコードの低レベルの表現と考えることができます。したがって、opcacheを使用すると、ソースファイルをPHPインタープリターが実行時に実際に必要とするものに変換する段階をスキップできます。顕著な節約！

それにもかかわらず、さらに節約できます。でコンパイルopcashファイルは他のファイルについて何も知りません。クラスBの拡張であるクラスAがある場合でも、実行時にリンクする必要があります。さらに、opcacheはソースファイルが変更されているかどうかを確認し、変更が検出されるとキャッシュを無効にします。

そして、ここでプリロードが助けになります：ソースファイルをopcodeにコンパイルするだけでなく、依存するクラス、トレイト、インターフェースを接続します。これは、実行可能コード（つまり、PHPインタープリターが使用できるコード）のそのような「コンパイルされた」フラグメントをメモリーに保管します。

リクエストがサーバーに到着すると、メモリに既にロードされているコードベースの一部を、余分な時間をかけずに使用できます。

「コードベースのどの部分」について話しているのですか？


プリロードの練習


正しくプリロードするために、開発者はダウンロードするファイルをサーバーに通知する必要があります。これは単純なPHPスクリプトを使用して行われるため、何も心配する必要はありません。

複雑なことは何もありません。

  • プリロードスクリプトを提供し、php.iniファイルでへのリンクを指定しますopcache.preload。
  • プリロードする各PHPファイルopcache_compile_file()は、プリロードスクリプトから転送する必要があります。

ある種のフレームワークをプリロードしたいとします。Laravelにしましょう。この場合、スクリプトはディレクトリ内のすべてのPHPファイルを調べvendor/laravel、一度に1つずつ追加する必要があります。

このスクリプトをphp.iniに含める方法は次のとおりです。

opcache.preload=/path/to/project/preload.php


そしてここに実装例があります：

$files = /*  ,      */;
 
foreach ($files as $file) {
    opcache_compile_file($file);
}

代わりopcache_compile_fileにを使用できますinclude。ただし、執筆時点では2番目のオプションが機能しなかったため、ここにバグがあったようです。

関連のないクラスをプリロードできないという警告


警告がありますリンクされていないクラスをプリロードできませんか？実際には、ファイルをプリロードする前に、それらの依存オブジェクト（インターフェース、トレイト、親クラス）もプリロードする必要があります。

クラスの依存関係で問題が発生した場合は、サーバーの起動時に警告が表示されます。

Can't preload unlinked class 
Illuminate\Database\Query\JoinClause: 
Unknown parent 
Illuminate\Database\Query\Builder

opcache_compile_file()ファイルを解析 するだけで、実行はしないことに注意してください。つまり、クラスにプリロードされていない依存関係がある場合、クラス自体をプリロードすることはできません。

これは重要ではありません。サーバーは通常どおり動作しますが、プリロードしたいすべてのファイルを自由に使用できるわけではありません。

そのため、依存関係の問題を回避するために、事前ロードするファイルを慎重に選択する必要があります。手動で行うことは、恩知らずで時間のかかる作業であるため、開発者はすでに自動化ソリューションに取り組んでいます。

Composerのサポート


最も有望な自動化ソリューションは、コンポーザー開発者によって準備されており、最新のPHPプロジェクトですでに使用されています。

現在、みんながでプリロードを設定する機能に取り組んcomposer.jsonでいます。これにより、代わりにプリロードファイルが生成されます。それ自体をプリロードするように、この機能はまだ開発中です。ここでイベントの展開を追跡できます。
幸い、必要がない場合はプリロードファイルを手動で構成する必要はありません。Composerがこれを行うことができます。

サーバー要件


プリロードを使用する場合、開発者が留意すべき他の2つの重要なポイントがあります。プリロードを機能

さphp.iniせるためにエントリを作成する必要があることはすでに知っています。これは、共有ホスティングを使用している場合、PHPを好きなように設定できないことを意味します。

実際には、単一のプロジェクト用にプリロードされたファイルを最適化するには、専用（仮想）サーバーが必要です。これを覚えておいてください。

またphp-fpm、メモリ内のファイルを再ロードするたびに、サーバーを再起動する必要があります（使用する場合はこれで十分です）。ほとんどの場合、これは明白ですが、思い出すことは不必要ではありません。

パフォーマンス


さて、最も重要な質問です。プリロードは本当にパフォーマンスを向上させますか？

もちろん！ベンモレルは、上記で参照したcomposerの同じトピックにある比較テストの結果を共有しました。

そしてまた、それは興味深いです。必要に応じて、hot classesコードベースでよく使用されるいわゆるクラスのみをプリロードできます。ベンモレルのテストでは、これらのクラスを約100個だけロードすると、一度にすべてをプリロードするよりもパフォーマンスが向上することが示されています。前者の場合、生産性は17％増加し、後者の場合は13％増加します。

もちろん、プリロードするクラスの選択は、特定のプロジェクトによって異なります。できるだけプリロードすることから始めるのが賢明です。これらの数パーセントのパフォーマンスの違いが非常に重要である場合は、実行時にコードを制御する必要があります。

もちろん、これらの操作はすべて自動化できます。これは将来行われる可能性があります。

ここで、composerプリロードサポートを追加することが重要です。これにより、自分でファイルを作成する必要がなくなります。この機能は、サーバー上で非常に簡単に構成できます。
