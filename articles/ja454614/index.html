XXE：XML外部エンティティ


この記事では、外部XMLエンティティのインジェクションとは何かを説明し、いくつかの一般的な例を説明し、さまざまなタイプのXXEインジェクションを見つけて使用する方法を説明し、攻撃を防ぐ方法を一般化して説明します。

外部XMLエンティティの注入とは何ですか？


外部XMLエンティティ（XXEとも呼ばれます）の注入は、攻撃者がアプリケーションによるXMLデータの処理を妨害する可能性があるWebセキュリティの脆弱性です。多くの場合、攻撃者はアプリケーションサーバーのファイルシステム内のファイルを表示し、アプリケーション自体がアクセスできるサーバーや外部システムとやり取りすることができます。

状況によっては、XXEの脆弱性を利用してサーバー側のリクエストフォージェリ攻撃（SSRF）を実行することにより、攻撃者はXXE攻撃を利用してコアサーバーまたはその他の内部インフラストラクチャを侵害する可能性があります。

XXEの脆弱性はどのように発生しますか？


一部のアプリケーションは、XML形式を使用して、ブラウザーとサーバーの間でデータを転送します。これを行うアプリケーションは、ほとんどの場合、標準ライブラリまたはプラットフォームAPIを使用して、サーバーでXMLデータを処理します。XXEの脆弱性が発生するのは、XML仕様に潜在的に危険なさまざまな関数が含まれており、標準のパーサーがこれらの関数をサポートしているためです。

XML外部エンティティーは、カスタムXMLエンティティーの一種であり、特定の値は、それらが書き込まれているDTDファイルからロードされます。外部エンティティは、ファイルパスまたはURLのコンテンツに基づいてエンティティを定義できるため、セキュリティの観点から特に興味深いものです。

XXE攻撃にはどのような種類がありますか？


  • XXEを使用してファイルを抽出します。ファイルのコンテンツを含む外部エンティティが定義され、アプリケーションの応答で返されます。
  • XXEを使用してSSRF攻撃を実行します。外部エンティティは内部システムのURLに基​​づいて決定されます。
  • 制御チャネルでのXXE情報のブラインド抽出の使用。機密データは、アプリケーションサーバーから攻撃者が制御するシステムに転送されます。
  • 攻撃者が必要な情報を含むエラーメッセージをトリガーできる場合に、ブラインドXXEを使用してエラーメッセージからデータを返す。
  • XXEを使用してファイルを抽出する

サーバーファイルシステムから任意のファイルを抽出するXXEインジェクション形式の攻撃を実行するには、次の2つの方法のいずれかで表示されるXMLを変更する必要があります。

  • ファイルパスを含む外部エンティティを定義するDOCTYPE要素を入力（または変更）します。
  • 特定の外部エンティティを使用するように、アプリケーションの応答で返されるXMLのデータの値を変更します。

たとえば、ショッピングアプリケーションが次のXMLをサーバーに送信して商品在庫の数量をチェックするとします。

<?xml version="1.0" encoding="UTF-8"?>
<stockCheck><productId>381</productId></stockCheck>
     XXE,     XXE    /etc/passwd,    XXE:
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
<stockCheck><productId>&xxe;</productId></stockCheck>

このXXE構造は、値が/ etc / passwdファイルの内容である外部エンティティー＆xxeを定義し、値productIdを持つエンティティーを使用します。これにより、アプリケーションの応答にファイルの内容が含まれます。

Invalid product ID: root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
...

実際のXXEの脆弱性の場合、送信されたXMLには多数のデータ値が含まれることが多く、そのいずれもアプリケーションの応答で使用できます。 XXEの脆弱性を体系的にテストするには、原則として、特定のエンティティを使用してXMLの各データノードを個別にチェックし、応答のステータスを確認する必要があります。

機密データを抽出することに加えて、XXE攻撃を使用する別の方法は、それらを使用してサーバー側の要求（SSRF）を偽造することです。

これは潜在的に深刻な脆弱性であり、その助けを借りて、サーバーアプリケーションを使用して、サーバーがアクセスできる任意のURLへのHTTPリクエストを実行できます。

XXEの脆弱性を使用してSSRF攻撃を実行するには、アクセスするURLを使用して外部XMLエンティティを定義し、データ値を持つ特定のエンティティを使用する必要があります。アプリケーションの応答で返されるデータの値で特定のエンティティを使用できる場合、アプリケーションの応答でURLからの応答を表示して、サーバーシステムとの双方向のやり取りを行うことができます。そうでない場合は、ブラインドSSRF攻撃のみを実行できます（重大な結果をもたらす可能性もあります）。

XXEを使用した次の例では、外部エンティティがサーバーにシステムインフラストラクチャ内の内部システムへの内部HTTPリクエストを実行するように強制します。

<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://internal.vulnerable-website.com/"> ]>

XInclude攻撃


一部のアプリケーションは、クライアントから送信されたデータを受信し、それをサーバー側でXMLドキュメントに挿入して、解析します。この例は、クライアントデータがバックエンドSOAPリクエストに配置され、バックエンドSOAPサービスによって処理される場合です。

この場合、XMLドキュメント全体を制御できず、DOCTYPE要素を決定または変更できないため、従来のXXE攻撃を実行できません。ただし、代わりにXIncludeを使用できます。 XIncludeはXML仕様の一部であり、ネストされたドキュメントからXMLドキュメントを作成できます。 XInclude攻撃をXMLドキュメントの任意のデータ値に配置できるため、サーバー側のXMLドキュメントに配置された1つのデータ項目のみを管理している状況で攻撃を実行できます。

XInclude攻撃を実行するには、XInclude名前空間を参照し、関与するファイルへのパスを指定する必要があります。例えば：

<foo xmlns:xi="http://www.w3.org/2001/XInclude">
<xi:include parse="text" href="file:///etc/passwd"/></foo>

一部のアプリケーションでは、ユーザーがファイルをアップロードして、サーバー側で処理することができます。いくつかの一般的なファイル形式は、XMLを使用するか、XMLサブコンポーネントを含みます。XML形式の例としては、DOCXなどのOfficeドキュメント形式やSVGなどの画像形式があります。

たとえば、アプリケーションでは、ユーザーが画像をアップロードし、ダウンロード後にサーバーで画像を処理または確認することができます。アプリケーションがPNGまたはJPEG形式の受信を想定している場合でも、使用される画像処理ライブラリはSVG画像をサポートしている場合があります。SVG形式はXMLを使用するため、攻撃者は悪意のあるSVG画像を送信し、XXEの脆弱性を攻撃する可能性があります。

ほとんどのPOSTリクエストは、application / x-www-form-urlencodedなどのHTMLフォームによって作成されたデフォルトのコンテンツタイプを使用します。一部のサイトは、この形式でリクエストを受信することを期待していますが、XMLを含む他のタイプのコンテンツを許可しています。

たとえば、通常のリクエストに次のものが含まれているとします。

POST /action HTTP/1.0

アプリケーションがメッセージの本文にXMLを含む要求を受け入れ、コンテンツをXMLとして解析する場合、XML形式を使用するように要求を再フォーマットするだけでXXEを攻撃する機会を得ることができます。
