.wasmファイル内には何がありますか？wasm-decompileの紹介

.wasmファイルを作成して操作するために、多くのコンパイラやその他のツールを自由に使用できます。これらのツールの数は常に増加しています。場合によっては、.wasmファイルを調べて、その中身を把握する必要があります。 Wasmツールの開発者である場合もあれば、Wasmに変換するように設計されたコードを記述し、コードがどのように変換されるかを調べるプログラマーである場合もあります。そのような関心は、たとえば、パフォーマンスの考慮によって引き起こされる可能性があります。



問題は、.wasmファイルに、実際のアセンブラコードによく似たかなり低レベルのコードが含まれていることです。特に、たとえばJVMとは異なり、すべてのデータ構造はロード/ストア操作のセットにコンパイルされ、明確なクラスおよびフィールド名を持つものにはコンパイルされません。コンパイラーは、LLVMのように、取得したコードが近くに見えないように入力コードを変更できます。 

.wasmファイルを取得して、何が起こっているのかを知りたい人はどうですか？

逆アセンブルまたは...逆コンパイル？


wasm2watなどのツールを使用して、.wasmファイルをWasmコードの標準テキスト表現を含む.watファイルに変換できます（これはWABTツールキットの一部です）。この変換の結果は非常に正確ですが、結果のコードを読み取ることは特に便利ではありません。

ここに、例えば、Cで書かれた簡単な関数があります：

typedef struct { float x, y, z; } vec3;

float dot(const vec3 *a, const vec3 *b) {
    return a->x * b->x +
           a->y * b->y +
           a->z * b->z;
}

コードはファイルに保存されますdot.c。

次のコマンドを使用します。

clang dot.c -c -target wasm32 -O2

次に、起こったことを.watファイルに変換するために、次のコマンドを適用します。

wasm2wat -f dot.o

これは私たちに与えるものです：

(func $dot (type 0) (param i32 i32) (result f32)
  (f32.add
    (f32.add
      (f32.mul
        (f32.load
          (local.get 0))
        (f32.load
          (local.get 1)))
      (f32.mul
        (f32.load offset=4
          (local.get 0))
        (f32.load offset=4
          (local.get 1))))
    (f32.mul
      (f32.load offset=8
        (local.get 0))
      (f32.load offset=8
        (local.get 1))))))

コードは小さいですが、多くの理由で、読むことは非常に困難です。ここでは式が使用されていないこと、および全体的に冗長に見えることの他に、メモリからデータをロードするためのコマンドの形式で提示されたデータ構造を理解することは簡単ではありません。ここで、このようなはるかに大きなサイズのコードを分析する必要があると想像してください。このような分析は非常に困難な作業になります。

wasm2watを使用する代わりに、次のコマンドを実行してみましょう。

wasm-decompile dot.o

ここに彼女が私たちに与えるものがあります：

function dot(a:{ a:float, b:float, c:float },
             b:{ a:float, b:float, c:float }):float {
  return a.a * b.a + a.b * b.b + a.c * b.c
}

それはすでにはるかに良く見えます。すでに知られているプログラミング言語を連想させる式を使用することに加えて、逆コンパイラはメモリの操作を目的としたコマンドを解析し、これらのコマンドによって表されるデータ構造を再作成しようとします。次に、システムは各変数に注釈を付けます。この変数は、「組み込み」構造宣言を持つポインターとして使用されます。デコンパイラーは、名前付きの構造体宣言を作成しません。これは、それぞれ3つの浮動小数点値を使用する構造体の間に何か共通点があるかどうかがわからないためです。

ご覧のとおり、逆コンパイルの結果は、分解の結果よりも理解しやすいことがわかりました。

デコンパイラーによって書かれたコードはどの言語で書かれていますか？


wasm-decompileツールはコードを出力し、このコードをある種の「平均的な」プログラミング言語のように見せようとします。同時に、このツールはWasmから離れすぎないようにします。

wasm-decompilerの最初の目標は、読み取り可能なコードを作成することでした。つまり、逆コンパイルされた.wasmファイルで何が起こっているかを読者が簡単に理解できるようなコードです。このツールの2番目の目的は、ソースファイルで発生していることを完全に表すコードを生成することにより、.wasmファイルの最も正確な表現を提供することです。明らかに、これらの目標は常にお互いによく一致していることにはほど遠い。

wasm-decompilerの出力は、もともと実際のプログラミング言語を表すコードとは考えられていませんでした。現在、Wasmでこのコードをコンパイルする方法はありません。

データをロードおよび保存するためのコマンド


上記のように、wasm-decompileは特定のポインターに関連付けられたロードおよび保存コマンドを探します。これらのコマンドが連続したシーケンスを形成する場合、逆コンパイラーは「組み込み」データ構造宣言の1つを表示します。

すべての「フィールド」にアクセスしたわけではない場合、デコンパイラーは、構造を、メモリーを処理する特定の一連の操作と確実に区別できません。この場合、wasm-decompileはフォールバックオプションを使用してfloat_ptr（タイプが同じ場合）などの単純なタイプを使用するか、最悪の場合、などの配列の操作方法を示すコードを生成しますo[2]:int。そのようなコードは、それoがtypeの要素をポイントしていることを教えてくれintます。そして、3番目のそのような要素に目を向けます。

ローカルのWasm関数は変数ではなくレジスタの使用に重点を置いているため、この最後の状況は、あなたが考えているよりもはるかに頻繁に発生します。その結果、最適化されたコードでは、同じポインタを使用して、完全に無関係なオブジェクトを操作できます。

デコンパイラーは、索引付けへのインテリジェントなアプローチを求めており、などのパターンを識別でき(base + (index << 2))[0]:intます。このようなパターンのソースは、Cの通常のインデックス付け操作です（4バイト型を指すbase[index]場所など）base。 Wasmはデータのロードおよび保存コマンドで定数として定義されたオフセットのみをサポートするため、これはコードで非常に一般的です。 wasm-decompileによって生成されたコードでは、そのような構造はtypeに変換されbase[index]:intます。

さらに、逆コンパイラは、絶対アドレスがデータセクションを指すときを認識します。
