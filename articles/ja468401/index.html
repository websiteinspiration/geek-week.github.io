ASP.NET Core + SPAでJWTを交換する安全な方法

前書き


JWT認証（JSON Web Token）は、サーバーとクライアント間のかなり均一で一貫した認証および認証メカニズムです。JWTの利点は、状態をあまり管理せず、適切にスケーリングできることです。最新のWebアプリケーションで、承認と認証を使用することがますます使用されていることは驚くにあたりません。

JWTを使用してアプリケーションを開発する場合、多くの場合、問題が発生します。トークンをどこにどのように保存することが推奨されますか Webアプリケーションを開発している場合、最も一般的なオプションが2つあります。

  • HTML5 Webストレージ（localStorageまたはsessionStorage）
  • クッキー

これらの方法を比較すると、どちらもクライアントのブラウザーに値を格納していると言えます。どちらもかなり簡単に使用でき、キーと値のペアの通常のストレージを表します。違いはストレージ環境にあります。

Web Storage（localStorage / sessionStorage）は、同じドメイン内のJavaScriptを介してアクセスできます。これは、アプリケーション内のすべてのJavaScriptコードがWebストレージにアクセスできることを意味し、クロスサイトスクリプティング（XSS）攻撃に対する脆弱性を生み出します。ストレージエンジンとして、Web Storageはストレージおよび共有中にデータを保護する方法を提供しません。タブを更新（F5）したり閉じたりするときに保持したい補助データ（ステータスやページ番号、フィルターなど）にのみ使用できます。

トークンは、ブラウザのCookieを介して送信することもできます。httpOnlyフラグで使用されるCookie はXSSの影響を受けません。 httpOnlyは、サーバー上でのみCookieの読み取り、書き込み、削除にアクセスするためのフラグです。これらはクライアントのJavaScriptを介してアクセスできないため、クライアントはトークンを認識せず、認証はサーバー側で完全に処理されます。安全なフラグを

設定して、CookieがHTTPS経由でのみ送信されるようにすることもできます。これらの利点を考えると、私の選択はクッキーに落ちました。

この記事では、SPAと共にASP.NET Core Web APIでhttpOnlyセキュアCookie + JSON Webトークンを使用して承認と認証を実装する方法について説明します。サーバーとクライアントが異なるオリジンにあるオプションが考慮されます。

ローカル開発環境のセットアップ


HTTPSを介してクライアントとサーバーの関係を正しく構成およびデバッグするには、クライアントとサーバーの両方がHTTPS接続を持つように、ローカル開発環境をすぐに構成することを強くお勧めします。

これをすぐに行わず、HTTPS接続なしで関係を構築しようとすると、HTTPSを使用した本番環境での安全なCookieと追加の安全なポリシーが正しく機能しないと、非常に多くの詳細が生じます。

OS Windows 10サーバーでHTTPSを構成する例を示します-ASP.NET Core、SPA-React。プロジェクトの作成時に[HTTPS

の構成]フラグを使用してASP.NET CoreでHTTPSを構成できます。作成中にこれを行わなかった場合は、[プロパティ]で対応するオプションを有効にします。

SPAを構成するには、スクリプトを"start"に変更し、"set HTTPS = true"に設定する必要があります。私のセットアップは次のとおりです：

安全なトークン交換のアイデア


この部分はコンセプトです。次の2つのことを行います。

  1. httpOnlyおよびセキュアフラグを使用して、トークンをHTTPリクエストにトスします。
  2. HTTPリクエストからクライアントアプリケーショントークンを受信して​​検証します。

これを行うには、次のものが必要です。

  • ログイン時にトークンをhttpOnly Cookieに書き込み、ログイン時にそこから削除します。
  • Cookieにトークンがある場合は、後続の各リクエストのHTTPヘッダーのトークンを置き換えます。
  • Cookieにトークンがない場合、ヘッダーは空になり、リクエストは承認されません。

ミドルウェア


主なアイデアは、カスタムミドルウェアを実装して、受信HTTPリクエストにトークンを挿入することです。ユーザー認証後、Cookieを特定のキーの下に保存します（例：".AspNetCore.Application.Id"）。承認やトークンとは何の関係もない名前を設定することをお勧めします。この場合、トークンを含むCookie は、AspNetCoreアプリケーションのある種の目立たないシステム定数のように見えます。そのため、攻撃者が多くのシステム変数を見る可能性が高くなり、どの認証メカニズムが使用されているかを理解していなければ、さらに進んでしまいます。もちろん、彼がこの記事を読んでおらず、特にそのような定数を探していない場合。

次に、このトークンを後続のすべての着信HTTPリクエストに挿入する必要があります。これを行うには、数行のミドルウェアコードを記述します。これはHTTPパイプラインにすぎません。

SPAクライアントのセットアップ


クライアントとサーバーが異なるオリジンに配置されている場合、クライアントでも追加の構成が必要です。各リクエストはXMLHttpRequest.withCredentialsを使用してラップする必要があります。

メソッドを次のようにラップしました。

import axios from "axios";

const api = axios.create({ baseURL: process.env.REACT_APP_API_URL });
api.interceptors.request.use(request => requestInterceptor(request))

const requestInterceptor = (request) => {
   request.withCredentials = true;
   return request;
}
export default api;

リクエスト設定はどのような方法でもラップできます。主なことは、withCredentials = trueがあることです。XMLHttpRequest.withCredentials

プロパティは、Cookie、認証ヘッダー、TLS証明書などの資格情報を使用してクロスドメインリクエストを生成するかどうかを決定します。 このフラグは、応答で送信されたCookieを無視するかどうかを決定するためにも使用されます。このリクエストを作成する前にwithCredentialsフラグがtrueに設定されていない場合、別のドメインからのXMLHttpRequestは独自のドメインにCookieを設定できません。



つまり、この属性を指定しない場合、Cookieはブラウザに保存されません。Cookieをサーバーに送信できなくなり、サーバーはJWTで目的のCookieを見つけられず、HTTPパイプラインでベアラートークンに署名しません。

ASP.NET Core Serverを構成する


ミドルウェアサービス

XsrfProtectionMiddleware.cs
public class XsrfProtectionMiddleware
    {
        private readonly IAntiforgery _antiforgery;
        private readonly RequestDelegate _next;

        public XsrfProtectionMiddleware(RequestDelegate next, IAntiforgery antiforgery)
        {
            _next = next;
            _antiforgery = antiforgery;
        }

        public async Task InvokeAsync(HttpContext context)
        {
            context.Response.Cookies.Append(
                ".AspNetCore.Xsrf",
                _antiforgery.GetAndStoreTokens(context).RequestToken,
                new CookieOptions {HttpOnly = false, Secure = true, MaxAge = TimeSpan.FromMinutes(60)});

            await _next(context);
        }
    }



MiddlewareExtensions.cs
public static class MiddlewareExtensions
    {
        public static IApplicationBuilder UseXsrfProtection(this IApplicationBuilder builder, IAntiforgery antiforgery)
            => builder.UseMiddleware<XsrfProtectionMiddleware>(antiforgery);
    }



ConfigureServices


services.AddAntiforgery(options => { options.HeaderName = "x-xsrf-token"; });
services.AddMvc();


構成、設定

app.UseAuthentication();
app.UseXsrfProtection(antiforgery);
