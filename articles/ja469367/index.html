WindowsのSTM32CubeIDEでローカルにユニットテストを実行する

前書き


誰もがユニットテストの利点を知っています。まず、コードと同時にテストを作成すると、エラーを早期に検出でき、面倒な複雑なデバッグに時間を費やすことがなくなります。組み込み開発の場合、ユニットテストには、コードがデバイスの内部のどこかで実行され、操作が非常に難しいという事実に関連する機能があります。次に、コードがターゲットハードウェアに強く結び付けられています。


プロジェクトに機器から独立しているフラグメントがあり、同時に非常に複雑なロジックを実装している場合、ユニットテストを使用すると最大のメリットが得られます。たとえば、これは、ある種のデータ転送プロトコル、さまざまな計算、または制御状態マシンの実装です。


組み込みプラットフォームの単体テストを実行するには、3つの方法があります。


  1. ターゲットプラットフォームで直接起動します。この場合、デバイスの機器を操作でき、コードは戦闘条件とまったく同じように機能します。ただし、テストにはデバイスへの物理的なアクセスが必要です。さらに、デバイスにコードを常にダウンロードする必要があるため、テストサイクルは非常に長くなります。
  2. エミュレーターで実行しています。この方法は、ターゲットプラットフォームが利用できない場合でも（たとえば、まだ実行されていないため）作業ができるため、主に優れています。欠点は、鉄（および周囲の世界）の動作を再現する際の精度に限界があること、およびそのようなエミュレーターを作成することの難しさです。

#include "gtest/gtest.h"

extern "C" {
#include "led_counter.h"
}

// Test fixture
class LedCounterTest: public ::testing::Test
{
protected:
    void SetUp()
    {
        Led_Counter_Init();
    }
};

// Check initial value
TEST_F(LedCounterTest, TestInitialValue)
{
    Led_Counter_Init();
    ASSERT_EQ(Led_Counter_Get_Next(), 1);
}

// Check how value is incremented
TEST_F(LedCounterTest, TestIncrementValue)
{
    Led_Counter_Init();
    unsigned int val = Led_Counter_Get_Next();
    for(int i=0;i<1;i++)
    {
        ASSERT_EQ(Led_Counter_Get_Next(), ++val);
    }
}

// Check how value return to 0 after 3
TEST_F(LedCounterTest, TestZeroCrossing)
{
    Led_Counter_Init();
    for(int i=0;i<3;i++)
    {
        Led_Counter_Get_Next();
    }
    ASSERT_EQ(Led_Counter_Get_Next(), 0);
}
