Interacción de SwiftUI con Redux



Hola a todos. En este artículo hablaremos sobre el marco SwiftUI junto con Redux, este paquete nos permite crear aplicaciones de manera rápida y fácil. SwiftUI se utiliza para crear una interfaz de usuario de estilo declarativo, a diferencia de UIKit . Redux, a su vez, sirve para controlar el estado de la aplicación.

El estado es un concepto fundamental en SwiftUI y Redux. En nuestro caso, esto no es solo una palabra de moda, sino también una entidad que los conecta y les permite trabajar muy bien juntos. En este artículo intentaré demostrar que la tesis anterior es cierta, ¡así que comencemos!

Antes de sumergirnos en escribir código, primero comprendamos qué es Redux y en qué consiste.

Redux es una biblioteca de código abierto para administrar el estado de una aplicación. La mayoría de las veces se usa junto con React o Angular para desarrollar el lado del cliente. Contiene una serie de herramientas para simplificar significativamente la transferencia de datos de almacenamiento a través del contexto. Creadores: Daniil Abramov y Andrew Clark.

Para mí, Redux no es solo una biblioteca, ya es algo más, lo atribuyo a las decisiones arquitectónicas en las que se basa la aplicación. Principalmente debido a su flujo de datos unidireccional.

Flujo multidireccional o unidireccional.


Para explicar lo que quiero decir con flujo de datos, daré el siguiente ejemplo. Una aplicación creada con VIPER admite un flujo de datos multidireccional entre módulos:



Redux, a su vez, es un flujo de datos unidireccional y es más fácil de explicar en función de sus componentes constituyentes.



Hablemos un poco en detalle sobre cada componente de Redux.

El estado es la única fuente de verdad que contiene toda la información necesaria para nuestra aplicación.

La acción es la intención de cambiar de estado. En nuestro caso, esta es una enumeración que contiene información nueva que queremos agregar o cambiar en el estado actual.

ReductorEs una función que toma la acción y el estado actual como parámetros y devuelve un nuevo estado. Esta es la única forma de crearlo. También vale la pena señalar que esta característica debe estar limpia.

Store es un objeto que contiene State y proporciona todas las herramientas necesarias para actualizarlo.

Creo que la teoría es suficiente, pasemos a la práctica.

Implementación de Redux


Una de las formas más fáciles de conocer una herramienta es comenzar a usarla, como dijo mi maestro de programación, si desea aprender un lenguaje de programación, escriba una aplicación en él. Así que creemos una pequeña aplicación, que sea un simple diario de entrenamiento, tendrá solo cuatro opciones, la primera es mostrar una lista de entrenamientos, la segunda es agregar un entrenamiento completo, la tercera es eliminar y la cuarta es ordenar entrenamientos. Aplicación bastante simple, pero al mismo tiempo nos permitirá conocer Redux y SwiftUI.

Cree un proyecto limpio en Xcode, asígnele el nombre WorkoutsDiary y, lo más importante, seleccione SwiftUI para la interfaz de usuario, ya que usaremos SwiftUI para crear nuestra interfaz de usuario.

Después de crear el proyecto. Cree una estructura de entrenamiento que será responsable del entrenamiento que completamos.

import Foundation

struct Workout: Identifiable {
    let id: UUID = .init()
    let name: String
    let distance: String
    let date: Date
    let complexity: Complexity
}

Como puede ver, esta estructura no tiene nada de loco, el campo de identificación se requiere para cumplir con el protocolo identificable, y el campo de complejidad es solo una enumeración con la siguiente definición:

enum Complexity: Int {
    case low
    case medium
    case high
}

Ahora que tenemos todo lo que necesitamos para comenzar a implementar Redux, comencemos creando un Estado.

struct AppState {
    var workouts: [Workout]
    var sortType: SortType?
}

El estado es una estructura simple que contiene dos campos: entrenamientos y sortType . El primero es una lista de entrenamientos, y el segundo es un campo opcional que determina cómo se ordena la lista.

SortType es una enumeración que se define de la siguiente manera:

enum SortType {
    case distance
    case complexity
}

Para simplificar, clasificaremos por distancia y dificultad en orden descendente, es decir, cuanto mayor sea la complejidad de nuestro entrenamiento, mayor se mostrará en nuestra lista. Vale la pena señalar que sortType es un tipo opcional y puede ser nulo, lo que significa que la lista no está ordenada en este momento.

Continuaremos la implementación de nuestros componentes. Creemos una acción

enum Action {
    case addWorkout(_ workout: Workout)
    case removeWorkout(at: IndexSet)
    case sort(by: SortType)
}

Como podemos ver, Acción es una enumeración con tres casos que nos dan la capacidad de manipular a nuestro Estado .

  • addWorkout (_ workout: Workout) simplemente agrega un entrenamiento que se pasa como parámetro.
  • removeWorkout (at: IndexSet) elimina el elemento en el índice especificado.
  • sort (by: SortType) ordena la lista de entrenamiento por el tipo de clasificación especificado.

Creemos uno de los componentes más complejos, este es Reductor :

func reducer(state: AppState, action: Action) -> AppState {
    var state = state
    
    switch action {
    case .addWorkout(let workout):
        state.workouts.append(workout)
        
    case .removeWorkout(let indexSet):
        state.workouts.remove(atOffsets: indexSet)
    
    case .sort(let type):
        switch type {
        case .distance:
            state.workouts.sort { $0.distance > $1.distance }
            state.sortType = .distance
        case .complexity:
            state.workouts.sort { $0.complexity.rawValue > $1.complexity.rawValue }
            state.sortType = .complexity
        }
    }
    return state
}

La función que escribimos es bastante simple y funciona de la siguiente manera:

  1. Copia el estado actual para trabajar con él.
  2. En función de Action , actualizamos nuestro estado copiado .
  3. Devuelve el estado actualizado

Vale la pena señalar que la función anterior es una función pura, ¡y eso es lo que queríamos lograr! Una función debe cumplir dos condiciones para ser considerada "pura":

  • Cada vez, la función devuelve el mismo resultado cuando se llama con el mismo conjunto de datos.
  • No hay efectos secundarios.

El último elemento que falta en Redux es la Tienda , así que impleméntelo para nuestra aplicación.

final class Store: ObservableObject {
     @Published private(set) var state: AppState
     
     init(state: AppState = .init(workouts: [Workout]())) {
         self.state = state
     }
     
     public func dispatch(action: Action) {
         state = reducer(state: state, action: action)
     }
 }

En las implementaciones del objeto Store , utilizamos todas las ventajas del protocolo ObservableObject , que nos permite excluir la escritura de una gran cantidad de código de plantilla o el uso de marcos de terceros. La propiedad State es de solo lectura y utiliza el contenedor de la propiedad @Published , lo que significa que cada vez que se cambia, SwiftUI recibirá notificaciones. El método init toma un estado inicial como parámetro con un valor predeterminado dado en forma de una matriz vacía de elementos de entrenamiento. La función de despacho es la única forma de actualizar el estado: reemplaza el estado actual con el nuevo creado por la función reductora , basado enAcción , que se pasa como parámetro.

Ahora que hemos implementado todos los componentes de Redux, podemos comenzar a crear una interfaz de usuario para nuestra aplicación.

Implementación de la aplicación


La interfaz de usuario de nuestra aplicación será bastante simple. Y constará de dos pantallas pequeñas. La primera y principal pantalla es una pantalla que mostrará una lista de entrenamientos. La segunda pantalla es la pantalla de agregar entrenamiento. Además, cada elemento se mostrará en un color determinado, el color reflejará la complejidad del entrenamiento. Los glóbulos rojos indican la mayor dificultad del entrenamiento, el naranja es responsable de la dificultad promedio y el verde es el responsable del entrenamiento más fácil.

Implementaremos la interfaz usando un nuevo marco de Apple llamado SwiftUI. SwiftUI viene a reemplazar nuestro UIKit familiar. SwiftUI es fundamentalmente diferente de UIKit, principalmente porque es un enfoque declarativo para escribir elementos de UI con código. En este artículo, no profundizaré en todas las complejidades de SwiftUI y supongo que ya tiene experiencia con SwiftUI. Si no tiene conocimiento de SwiftUI, le aconsejo que preste atención a la documentación de Apple, es decir, mire sus varios tutoriales completos con la adición paso a paso y la visualización interactiva del resultado a la vista. También hay enlaces a proyectos de ejemplo. Estos tutoriales te permitirán sumergirte rápidamente en el mundo declarativo de SwiftUI.

También vale la pena señalar que SwiftUI aún no está listo para proyectos de producción, es demasiado joven y pasará más de un año antes de que pueda usarse de esta manera. Además, no olvide que solo es compatible con las versiones iOS 13.0+. Pero también vale la pena señalar que SwiftUI funcionará en todas las plataformas de Apple, ¡lo cual es una gran ventaja sobre UIKit!

Comencemos la implementación desde la pantalla principal de nuestra aplicación. Vaya al archivo ContentView.swift y cambie el código actual a esto.

struct ContentView: View {
    @EnvironmentObject var store: Store
    @State private var isAddingMode: Bool = false
    
    var body: some View {
        NavigationView {
            WorkoutListView()
                .navigationBarTitle("Workouts diary", displayMode: .inline)
                .navigationBarItems(
                    leading: AddButton(isAddingMode: self.$isAddingMode),
                    trailing: TrailingView()
                )
        }
        .sheet(isPresented: $isAddingMode) {
            AddWorkoutView(isAddingMode: self.$isAddingMode)
                .environmentObject(self.store)
        }
    }
}

La Vista de contenido es una vista estándar en SwiftUI. La parte más importante, desde mi punto de vista, es la línea de código que contiene la variable store. Crearemos @EnvironmentObject. Esto nos permitirá usar los datos de la Tienda donde sea necesario y, además, actualizará automáticamente nuestras vistas si se modifican los datos. Esto es algo así como Singleton para nuestra tienda.

@EnvironmentObject var store: Store

También vale la pena señalar la siguiente línea de código:

@State private var isAddingMode: Bool = false

EstadoEs un contenedor que podemos usar para indicar el estado de una Vista. SwiftUI lo almacenará en una memoria interna especial fuera de la estructura de Vista. Solo una vista vinculada puede acceder a ella. Una vez que el valor de la propiedadEstado cambios, SwiftUI reconstruye la Vista para tener en cuenta los cambios de estado.


Luego iremos al archivo SceneDelegate.swift y agregaremos el código al método:

func scene(
        _ scene: UIScene,
        willConnectTo session: UISceneSession,
        options connectionOptions: UIScene.ConnectionOptions
    ) {
        let contentView = ContentView().environmentObject(Store())
        
        if let windowScene = scene as? UIWindowScene {
            let window = UIWindow(windowScene: windowScene)
            window.rootViewController = UIHostingController(rootView: contentView)
            self.window = window
            window.makeKeyAndVisible()
        }
    }

Del mismo modo, cualquier @EnvironmentObject se puede pasar a cualquier representación secundaria en toda la aplicación, y todo esto es posible gracias al entorno. La variable IsAddingMode está marcadaEstadoe indica si la vista secundaria se muestra o no. WorkoutListView hereda automáticamente la variable de la tienda , y no necesitamos pasarla explícitamente, pero debemos hacer esto para AddWorkoutView , porque se presenta en forma de una hoja que no es un elemento secundario de ContentView . Ahora cree un WorkoutListView que heredará de View. Cree un nuevo archivo rápido llamado WorkoutListView .



struct WorkoutListView: View {
    @EnvironmentObject var store: Store
    
    var body: some View {
        List {
            ForEach(store.state.workouts) {
                WorkoutView(workout: $0)
            }
            .onDelete {
                self.store.dispatch(action: .removeWorkout(at: $0))
            }
            .listRowInsets(EdgeInsets())
        }
    }
}

Ver, que utiliza el elemento Lista de contenedores para mostrar una lista de entrenamientos. La función onDelete se utiliza para eliminar un entrenamiento y utiliza la acción removeWorkout , que se realiza mediante la función de envío proporcionada por la tienda . Para mostrar el entrenamiento en la lista, se utiliza WorkoutView.

Cree otro archivo WorkoutView.swift que será responsable de mostrar nuestro elemento en la lista.

struct WorkoutView: View {
    let workout: Workout
    
    private var backgroundColor: Color {
        switch workout.complexity {
        case .low:
            return .green
        case .medium:
            return .orange
        case .high:
            return .red
        }
    }
    
    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(workout.name)
                Text("Distance:" + workout.distance + "km")
                    .font(.subheadline)
            }
            Spacer()
            VStack(alignment: .leading) {
                Text(simpleFormat(workout.date))
            }
        }
        .padding()
        .background(backgroundColor)
    }
}

private extension WorkoutView {
    func simpleFormat(_ date: Date) -> String {
        let dateFormatter = DateFormatter()
        dateFormatter.dateFormat = "dd MMM yyyy"
        dateFormatter.locale = .init(identifier: "en_GB")
        return dateFormatter.string(from: date)
    }
}

Esta vista toma el objeto de entrenamiento como parámetro y se configura en función de sus propiedades.

Para agregar un nuevo elemento a la lista, debe cambiar el parámetro isAddingMode a verdadero para mostrar AddWorkoutView . Esta responsabilidad recae en AddButton .

Cree una vista para agregar un nuevo entrenamiento. Cree un nuevo archivo AddWorkoutView.swift :

struct AddWorkoutView: View {
    @EnvironmentObject private var store: Store
    
    @State private var nameText: String = ""
    @State private var distanceText: String = ""
    @State private var complexityField: Complexity = .medium
    @State private var dateField: Date = Date()
    @Binding var isAddingMode: Bool
    
    var body: some View {
        NavigationView {
            Form {
                TextField("Name", text: $nameText)
                TextField("Distance", text: $distanceText)
                Picker(selection: $complexityField, label: Text("Complexity")) {
                    Text("Low").tag(Complexity.low)
                    Text("Medium").tag(Complexity.medium)
                    Text("High").tag(Complexity.high)
                }
                DatePicker(selection: $dateField, displayedComponents: .date) {
                    Text("Date")
                }
            }
            .navigationBarTitle("Workout Details", displayMode: .inline)
            .navigationBarItems(
                leading: Button(action: { self.isAddingMode = false }) {
                    Text("Cancel")
                },
                trailing: Button(action: {
                    let workout = Workout(
                        name: self.nameText,
                        distance: self.distanceText,
                        date: self.dateField,
                        complexity: self.complexityField
                    )
                    self.store.dispatch(action: .addWorkout(workout))
                    self.isAddingMode = false
                }) {
                    Text("Save")
                }
                .disabled(nameText.isEmpty)
            )
        }
    }
}

Este es un controlador bastante grande que, como otros controladores, contiene la variable store. También contiene las variables nameText, distanceText, complexField e isAddingMode . Las primeras tres variables son necesarias para vincular TextField, Picker, DatePicker , que se puede ver en esta pantalla. La barra de navegación tiene dos elementos. El primer botón es un botón que cierra la pantalla sin agregar un nuevo entrenamiento, y el último agrega un nuevo entrenamiento a la lista, que se logra enviando la acción addWorkout. Esta acción también cierra la pantalla de agregar nuevo entrenamiento.


Por último, pero no menos importante, es TrailingView .

struct TrailingView: View {
    @EnvironmentObject var store: Store
    
    var body: some View {
        HStack(alignment: .center, spacing: 30) {
            Button(action: {
                switch self.store.state.sortType {
                case .distance:
                    self.store.dispatch(action: .sort(by: .distance))
                default:
                    self.store.dispatch(action: .sort(by: .complexity))
                }
            }) {
                Image(systemName: "arrow.up.arrow.down")
            }
            EditButton()
        }
    }
}

Esta vista consta de dos botones que se encargan de ordenar la lista de ejercicios y de activar o desactivar el modo de edición de nuestra lista de ejercicios. Las acciones de clasificación se llaman mediante la función de despacho, que podemos llamar a través de la propiedad de la tienda.


Resultado


La aplicación está lista y debería funcionar exactamente como se esperaba. Intentemos compilarlo y ejecutarlo.

recomendaciones


Redux y SwiftUI funcionan muy bien juntos. El código escrito con estas herramientas es fácil de entender y puede estar bien organizado. Otro buen aspecto de esta solución es su excelente capacidad de prueba de código. Sin embargo, esta solución no está exenta de inconvenientes, uno de ellos es una gran cantidad de memoria utilizada por la aplicación cuando el estado de la aplicación es muy complejo, y el rendimiento de la aplicación puede no ser ideal en algunos escenarios específicos, ya que todas las vistas en SwiftUI se actualizan al crear un nuevo estado. Estas deficiencias pueden tener un gran impacto en la calidad de la aplicación y la interacción del usuario, pero si las recordamos y preparamos el estado de manera razonable, el impacto negativo se puede minimizar o incluso evitar fácilmente.
