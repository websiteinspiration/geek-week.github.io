CPU


CPU使用率、％。特定の期間のCPU使用率を表示します。



分析するには？VMがCPUを90％安定して使用している場合、または100％までのピークがある場合は、問題があります。問題は、VM内のアプリケーションの「遅い」動作だけでなく、ネットワークを介したVMのアクセス不能にも表れます。監視システムがVMが定期的に落ちることを示している場合は、CPU使用率グラフのピークに注意してください。

仮想マシンのCPU負荷を示す標準のアラームがあります。



どうすればよいですか？CPU使用率がVMを常に圧倒している場合は、vCPUの数を増やすこと（残念ながら、これは必ずしも役に立たない）、またはより効率的なプロセッサを備えたサーバーにVMを移動することを検討できます。

MhzでのCPU使用率


vCenterの使用率グラフでは、％で表示できるのは仮想マシン全体のみであり、個々のコアのグラフはありません（Esxtopにはコアの値（％）があります）。コアごとに、使用量をMHzで確認できます。

分析するには？アプリケーションがマルチコアアーキテクチャ用に最適化されていない場合があります。100％使用するコアは1つだけで、残りは負荷なしでアイドル状態です。たとえば、デフォルトのバックアップ設定では、MS SQLは1つのコアのみでプロセスを開始します。その結果、ディスク速度が遅いため（これはユーザーが最初に不満を言っていたため）ではなく、プロセッサが処理できないため、バックアップが遅くなります。この問題は、パラメータを変更することで解決しました。バックアップは、いくつかのファイルで（それぞれ、いくつかのプロセスで）並行して実行され始めました。


核の不均一な負荷の例。

また、（上のグラフのように）コアの負荷が不均一で、一部に100％のピークがある状況もあります。 1つのコアのみをロードする場合と同様に、CPU使用率に関するアラームは機能しません（VM全体で発生します）が、パフォーマンスの問題があります。

何をすべきか？仮想マシンのソフトウェアがカーネルを不均等にロードする場合（1つのコアまたはカーネルの一部のみを使用）、それらの数を増やしても意味がありません。この場合、より効率的なプロセッサを備えたサーバーにVMを移動することをお勧めします。

サーバーBIOSで電源設定を確認することもできます。多くの管理者は、BIOSで高性能モードをオンにして、CステートとPステートの省電力テクノロジーをオフにします。最新のIntelプロセッサーはターボブーストテクノロジーを使用しており、他のコアが原因で個々のプロセッサーコアの周波数が増加します。ただし、有効になっている省エネ技術でのみ機能します。これらをオフにすると、プロセッサーはロードされていないカーネルの電力消費を削減できません。

ヴイエムウェアでは、サーバーの省エネテクノロジーをオフにするのではなく、ハイパーバイザーのエネルギー管理を最大化するモードを選択することをお勧めします。この場合、ハイパーバイザーのエネルギー設定で、高パフォーマンスを選択する必要があります。

増加したCPU周波数を必要とする別個のVM（またはVMコア）がインフラストラクチャにある場合、適切に構成された電力消費はそれらのパフォーマンスを大幅に改善できます。



CPU Ready（準備）


VMコア（vCPU）が準備完了状態の場合、有用な作業は行われません。この状態は、ハイパーバイザーが仮想マシンのvCPUプロセスを割り当てることができる空き物理コアを見つけられない場合に発生します。

分析するには？通常、仮想マシンコアが10％以上の時間の準備ができた状態である場合、パフォーマンスの問題が発生します。簡単に言えば、VMが物理リソースの可用性を待機する時間の10％以上。

vCenterでは、CPU Readyに関連付けられた2つのカウンターを確認できます。

  • 準備
  • いいよ

両方のカウンターの値は、VM全体と個々のカーネルの両方で表示できます。
準備はすぐにパーセントで値を示しますが、リアルタイム（過去1時間のデータ、測定間隔20秒）でのみ表示されます。このカウンターは、「熱心な追求」の問題を見つけるためにのみ使用するのが最適です。

レディカウンター値は、履歴の観点からも表示できます。これは、パターンを確立し、問題をより深く分析するのに役立ちます。たとえば、仮想マシンが特定の時間にパフォーマンスの問題を抱え始めた場合、ハングしたCPU Ready値の間隔をこのVMが動作するサーバーの合計負荷と比較し、負荷を減らすための対策を講じることができます（DRSが対応できなかった場合）。

準備は、準備とは異なり、パーセンテージではなくミリ秒単位で表示されます。これは合計タイプのカウンターです。つまり、測定期間中にVMコアが準備完了状態であった時間を示します。この値は、単純な式でパーセントに変換できます：

（CPUレディ合計値/（グラフのデフォルトの更新間隔（秒）* 1000））* 100 = CPUレディ％

たとえば、以下のグラフのVMの場合、仮想マシン全体の準備完了ピーク値は次のようになります。

(4286/(20∗1000))∗100=21



パーセンテージとしてレディ値を計算する場合、次の2つの点に注意する必要があります。

  • VM全体のReady値は、コア全体のReadyの合計です。
  • 測定間隔。リアルタイムの場合、これは20秒です。たとえば、日足チャートでは、300秒です。

アクティブなトラブルシューティングを使用すると、これらの単純なポイントが見落とされ、存在しない問題の解決に貴重な時間が費やされる可能性があります。

以下のグラフのデータに基づいて、準備完了を計算します。（324474 /（20 * 1000））* 100 = VM全体の1622％。コアを見ると、それほど怖くないことがわかります。1622/ 64 =コアあたり25％。この場合、トリックの検出は非常に簡単です。Ready値は非現実的です。ただし、複数のコアを持つVM全体で10〜20％について話している場合、各コアの値は通常の範囲内に収まる可能性があります。



何をすべきか？Readyの高い値は、サーバーに仮想マシンの通常の操作に十分なプロセッサリソースがないことを示します。この状況では、プロセッサ（vCPU：pCPU）のオーバーサブスクリプションを減らすだけです。明らかに、これは既存のVMのパラメーターを減らすか、VMの一部を他のサーバーに移行することで実現できます。

コストップ


分析するには？このカウンターには、Summationタイプもあり、Ready：

（CPU co-stop summation value /（chart default update interval in seconds * 1000））* 100 = CPU co-stop％などのパーセンテージに変換されます

ここでは、VMあたりのコア数にも注意する必要がありますおよび測定間隔。
costop状態では、カーネルは有用な作業を行いません。VMサイズとサーバーの通常の負荷を正しく選択すると、同時停止カウンターはゼロに近くなります。


この場合、負荷は明らかに異常です:)

どうすればよいですか？多数のコアを持つ複数のVMが同じハイパーバイザーで実行されており、CPUによるオーバーサブスクリプションがある場合、コストップカウンターが大きくなり、これらのVMのパフォーマンスに問題が生じる可能性があります。

また、ハイパートレッドが有効になっているサーバーの同じ物理コア上の1つのVMのアクティブなカーネルにスレッドが使用されている場合、コストップも増加します。このような状況は、たとえば、VMが動作するサーバー上に物理的に持っているよりも多くのコアを持っている場合、またはVMで「preferHT」設定が有効になっている場合に発生する可能性があります。この設定については、こちらをご覧ください。

コストップが高いことによるVMパフォーマンスの問題を回避するには、このVMで実行するソフトウェアメーカーの推奨事項と、VMが実行されている物理サーバーの機能に従って、VMサイズを選択します。

カーネルを予備で追加しないでください。これにより、VM自体だけでなく、そのサーバーの近隣のパフォーマンスの問題が発生する可能性があります。

その他の有用なCPUメトリック


実行 -vCPU測定期間のRUN状態にあった時間（ms）、つまり実際に有用な作業を実行した時間。

アイドル -vCPU測定期間の非アクティブな時間（ミリ秒）。高いアイドル値は問題ではなく、vCPUに「何もする必要がない」だけでした。

待機 -vCPU測定期間の待機状態の時間（ミリ秒）。 IDLEはこのカウンターに含まれているため、待機値が高いことも問題を示していません。ただし、高待機IDLEが低い場合、VMはI / O操作の完了を待機していたため、ハードディスクまたは一部の仮想VMデバイスのパフォーマンスに問題がある可能性があります。

最大制限-設定されたリソース制限が原因で、vCPU測定期間が準備完了状態であった時間（ms）。パフォーマンスがどうしても低い場合は、このカウンターの値とVM設定のCPU制限を確認すると便利です。VMには、実際には気づいていない制限がある場合があります。たとえば、これは、CPU制限が設定されたテンプレートからVMが傾斜している場合に発生します。

スワップ待機 -vCPUが測定期間中にVMkernelスワップでの操作を待機した時間。このカウンターの値がゼロを超える場合、VMには明らかにパフォーマンスの問題があります。RAMカウンターに関する記事では、SWAPについて詳しく説明します。

ESXTOP


vCenterのパフォーマンスカウンターが履歴データの分析に適している場合、問題のオンライン分析はESXTOPで行うのが最適です。ここでは、すべての値が完成した形式で表示され（何も翻訳する必要はありません）、最小測定期間は2秒です。

個々のVMコアのメトリックを表示するには、「e」を押して、関心のあるVMのGIDを入力します（下のスクリーンショットの30919）：



デフォルトで表示される列を簡単に確認します。 「f」を押すと、列を追加できます。

NWLD（ワールド数）-グループ内のプロセスの数。グループを展開して各プロセスのメトリック（たとえば、マルチコアVMの各コア）を​​表示するには、「e」を押します。グループに複数のプロセスがある場合、グループのメトリックの値は、個々のプロセスのメトリックの合計に等しくなります。

％USED-サーバーCPUがプロセスまたはプロセスのグループを使用しているサイクルの数。

％RUN-測定期間中にプロセスがRUN状態にあった時間。有用な作業を行いました。これは、ハイパースレッディング、周波数スケーリング、およびシステムタスクに費やされた時間（％SYS）を考慮しない点で、USEDとは異なります。

％SYS-システムタスクに費やされた時間。例：割り込みの処理、入力/出力、ネットワーク操作など。VMに多くの入力/出力がある場合、値は高くなる可能性があります。

％OVRLP-VMプロセスが実行されている物理コアが他のプロセスのタスクに費やした時間。

これらのメトリックは、次のように関連しています。

％USED =％RUN +％SYS-％OVRLP。

通常、％USEDメトリックはより有益です。

％WAIT-測定期間中にプロセスが待機状態であった時間。 IDLEを含みます。

％IDLE-測定期間中にプロセスがIDLE状態であった時間。

％SWPWT-測定期間中にvCPUがVMkernel Swapでの操作を待機した時間。

％VMWAIT-イベントの待機状態にあったvCPU測定期間の時間（通常はI / O）。 vCenterには同様のカウンターはありません。高い値は、VMへの入力/出力に問題があることを示します。

％WAIT =％VMWAIT +％IDLE +％SWPWT。

VMがVMkernel Swapを使用しない場合、パフォーマンスの問題を分析するときは、VMWAIT％を確認することをお勧めします。このメトリックは、VMが何もしなかった時間（％IDLE）を考慮しないためです。

％RDY-測定期間中にプロセスが準備完了状態であった時間。

％CSTP-測定期間中にプロセスがポスト状態にあった時間。

％MLMTD-設定されたリソース制限のためにvCPU測定期間中に準備完了状態であった時間。

％WAIT +％RDY +％CSTP +％RUN = 100％-VMコアは常にこれら4つの状態のいずれかになります。

ハイパーバイザー上のCPU


ハイパーバイザーのvCenterにはCPUパフォーマンスカウンターもありますが、興味深いものではありません。サーバー上のすべてのVMのカウンターの合計にすぎません。
[概要]タブでサーバーのCPUの状態を確認するのが最も便利です。



サーバーと仮想マシンには、標準のアラーム



があります。サーバーのCPU に高い負荷がかかると、サーバー上で実行されているVMでパフォーマンスの問題が発生し始めます。

ESXTOPでは、サーバーのCPU使用率データが画面の上部に表示されます。ハイパーバイザーには有益ではない標準のCPU負荷に加えて、さらに3つのメトリックがあります

。CORE UTIL（％） -物理サーバーのコアをロードします。このカウンターは、コアが測定期間中に作業を実行した時間を示します。

PCPU UTIL（％）-ハイパースレッディングが有効な場合、物理コアごとに2つのスレッド（PCPU）があります。このメトリックは、各スレッドが作業を行った時間を示します。

PCPU USED（％）はPCPU UTIL（％）と同じですが、周波数スケーリング（エネルギー節約のためにコア周波数を下げるか、ターボブーストテクノロジーによりコア周波数を上げる）とハイパースレッディングを考慮に入れます。

PCPU_USED％= PCPU_UTIL％*実効コア周波数/公称コア周波数。


このスクリーンショットでは、一部のコアについて、ターボブーストの動作により、コア周波数が公称より高いため、USED値は100％を超えています。

ハイパースレッディングがどのように考慮されるかについてのいくつかの言葉。サーバーの物理コアの両方のスレッドでプロセスが100％実行され、コアは公称周波数で実行されている場合、次のようになります。

  • カーネルのCORE UTILは100％、
  • 両方のスレッドのPCPU UTILは100％、
  • 両方のスレッドで使用されるPCEDは50％になります。

測定期間中に両方のスレッドが100％動作しなかった場合、スレッドが並行して動作した期間では、コアのUSED PCPUは半分に分割されます。

一般的なCPUパフォーマンスの問題


最後に、VMのCPUパフォーマンスに関する問題の一般的な理由について説明し、それらを解決するための簡単なヒントを示し

ます。コアクロック周波数が十分ではありません。 VMをより効率的なコアに転送する方法がない場合は、エネルギー設定を変更してTurbo Boostをより効率的に動作させることができます。

VMの誤ったサイジング（コアが多すぎる/少なすぎる）。少数のコアを配置すると、VM CPUに高い負荷がかかります。たくさんあれば、高いコストップをキャッチしてください。

サーバーのCPUの大規模なオーバーサブスクリプション。 VMの準備ができている場合は、CPUのオーバーサブスクリプションを減らします。
