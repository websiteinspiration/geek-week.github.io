Die große Spaltung beim Import: Klärung der Unsicherheit beim Import in Typescript

Vor Beginn des Kurses "React.js Developer" wurde eine Übersetzung des Artikels erstellt.


Ich arbeite seit einiger Zeit mit Typoskript und hatte viele Probleme beim Sortieren der Module und beim Empfehlen von Einstellungen. Ich muss sagen, es gibt wirklich viele Unverständliche. Namensraum import * as React from 'react', esModuleInteropusw. Lassen Sie uns deshalb verstehen, warum der ganze Hype entstanden ist.

Ich werde nicht über Namespaces als modulares System in Typoskript sprechen, da die Idee nicht die beste war (insbesondere unter Berücksichtigung des aktuellen Entwicklungsvektors) und niemand sie derzeit verwendet.

Wie war es also, bevor sie auftauchten esModuleInterop? Es gab fast alle Module, die Babel oder Browser hatten, sowie benannte Importe / Exporte. Bei Exporten und Importen hatte Typoskript jedoch standardmäßig eine eigene Option: Es musste import * as React from 'react'(stattdessen import React from 'react') geschrieben werden, und hier geht es natürlich nicht nur um Reaktionen, sondern um alle Standardimporte aus commonjs. Wie ist es passiert?

Um dies zu verstehen, sehen wir uns an, wie die Kompatibilität zwischen einigen Mustern in commonjsund Modulen funktioniert es6. Zum Beispiel haben wir ein Modul, das exportiert foound barals Schlüssel:

module.exports = { foo, bar }

Wir können mit erfordern und destrukturieren importieren:

const { foo, bar } = require('my-module')

Und wenden Sie dasselbe Prinzip mit benannten Importen an (obwohl dies, um ehrlich zu sein, keine Destrukturierung ist):

import { foo, bar } from 'my-module'

Das häufigere Muster in commonjs ist jedoch const myModule = require('my-module')(weil es noch keine Destrukturierung gegeben hat), aber wie machen Sie das es6?

Bei der Entwicklung der Importspezifikation war es6einer der wichtigsten Aspekte die Kompatibilität mit commonjs, da bereits commonjsviel Code darauf geschrieben wurde. So wurde der Standardimport und -export angezeigt. Ja, das einzige Ziel war es, die Kompatibilität mit sicherzustellen commonjs, damit wir schreiben import myModule from 'my-moduleund genau das gleiche Ergebnis erzielen können . Aus den Spezifikationen war dies jedoch nicht ersichtlich, und außerdem war die Implementierung der Kompatibilität das Vorrecht der Entwickler des Transpilers. Und hier ist gerade das große Schisma passiert: import React from 'react'oder import * as React from 'react'- das ist die Frage.

Warum hat Typoskript letzteres gewählt? Versetzen Sie sich in die Stelle des Transpiler Entwickler und fragen Sie sich, was ist der einfachste Weg , um Verkehr Einfuhren aus es6zu commonjs? Angenommen, Sie haben die folgenden Importe und Exporte:

export const foo = 1
export const bar = 2
export default () => {}
import { foo } from 'module'
import func from 'module'`

Also werden wir das Objekt jsmit dem Schlüssel defaultstandardmäßig für den Export verwenden!

module.exports = {
  foo: 1,
  bar: 2,
  default: () => {}
}
const module = require('module')
const foo = module.foo
const func = module.default

Cool, aber was ist mit Kompatibilität? Wenn Import von Standard bedeutet , dass wir ein Feld mit einem Namen nehmen default, dann , wenn wir schreiben import React from 'react'es bedeutet const { default: React } = require('react'), aber es funktioniert nicht! Versuchen Sie stattdessen, den Import mit einem Sternchen zu verwenden. Jetzt müssen Benutzer schreiben import * as React from 'react', um zum Inhalt zu gelangen module.exports.

Es gibt jedoch einen semantischen Unterschied zu commonjs.
Commonjswar wie normales Javascript, nicht mehr. Nur Funktionen und Objekte, ohne dass dies erforderlich ist . Andererseits ist es beim Import es6daher requirejetzt Teil der SpezifikationmyModuleIn diesem Fall handelt es sich nicht nur um ein reguläres Javascript-Objekt, sondern um einen sogenannten Namespace (nicht zu verwechseln mit Namespaces in Typoskript), der dementsprechend bestimmte Eigenschaften aufweist. Einer davon ist, dass der Namespace nicht aufgerufen werden kann. Und was ist das Problem hier, fragen Sie sich vielleicht?

Versuchen wir ein anderes Muster commonjsmit einer Funktion als Export:

module.exports = function() { // do something }

Wir können es verwenden requireund ausführen:

const foo = require('my-module')
foo()

Wenn Sie dies jedoch in einer Umgebung mit ES6-Modulen versuchen, wird eine Fehlermeldung angezeigt:

import * as foo from 'my-module'
foo() // Error

Dies liegt daran, dass der Namespace nicht mit einem Javascript-Objekt identisch ist, sondern mit einer separaten Struktur, in der jeder es6-Export gespeichert wird.

Aber Babel hat es richtig gemacht und eine Kompatibilitätsoption bereitgestellt, in die wir schreiben können, import React from 'reactund es wird funktionieren. Beim Transilieren markiert er jedes es6-Modul mit einem speziellen Flag in module.exports, damit wir verstehen, dass wenn das Flag wahr ist, es zurückgegeben module.exportswird und wenn es falsch ist (wenn es beispielsweise eine Bibliothek ist commonjs, die nicht transponiert wurde), müssen wir den aktuellen Export { default: export }so umschließen , dass wir könnte jedes Mal verwenden default(siehe hier ).

Typoskript gelangte durch Importe mit Sternchen, gab aber schließlich auf und fügte eine Option hinzuesModuleInteropan den Compiler. Im Allgemeinen funktioniert diese Option genauso wie babel. Wenn Sie sie aktivieren, können Sie den regulären Import als schreiben import React from 'react', und Typoskript versteht alles.

Das Problem ist, dass es, obwohl es in neuen Projekten (zur Laufzeit tsc --init) standardmäßig aktiviert ist, für vorhandene Projekte nicht funktioniert (selbst wenn Sie auf TypeScript 3 aktualisieren), da es keine Abwärtskompatibilität aufweist. Daher müssen Sie die unnötigen Sternchenimporte in die Standardimporte umschreiben. React behandelt dies normal, da es sich immer noch um eine Sammlung benannter Exporte handelt, jedoch nicht beispielsweise mit einem Namespace-Aufruf. Aber keine Angst, wenn beim Exportieren alles in Ordnung ist (und zum größten Teil alles in Ordnung ist, da viele von ihnen automatisch repariert werden), TypeScript 3 ermöglicht es Ihnen, den Import mit Sternchen schnell in Standard zu konvertieren.

Daher befürworte ich die Verwendung dieser Option wirklich esModuleInterop, schon allein deshalb, weil Sie damit nicht nur weniger Code schreiben und das Lesen vereinfachen können (und dies sind nicht nur Wörter, zum Beispiel können Sie beim Rollup keine Importe mit solchen Sternchen verwenden), sondern auch Meinungsverschiedenheiten zwischen Communitys reduzieren Typoskript und Babel.

Achtung: Früher gab es eine Option enableSyntheticDefaultImports, mit der der Compiler geschlossen wurde, als er versuchte, sich standardmäßig über falschen Import zu beschweren. Daher brauchten wir unseren eigenen Weg, um die Kompatibilität mit commonjs(z. B.WebpackDefaultImportPlugin), aber es war problematisch, denn wenn Sie beispielsweise Tests haben, müssen Sie diese Kompatibilität trotzdem sicherstellen. Bitte beachten Sie, dass esModuleInteropder synthetische Import standardmäßig nur dann eingeschlossen ist, wenn Ihr Ziel <=ES5 ist. Wenn Sie diese Option aktivieren und die Compiler sich weiterhin darüber beschweren import React, verstehen Sie daher, welchen Zweck Sie verfolgen, und möglicherweise ist die Einbeziehung von Standardimporten Ihre Option (oder starten Sie vscode / webstorm neu, wer weiß).

Ich hoffe, meine Erklärung hat die Situation ein wenig geklärt, aber wenn Sie noch Fragen haben, können Sie sie mir auf Twitter stellen !
