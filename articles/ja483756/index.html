HTTPリクエストを作成し、正常に機能を低下させます（単一のギャップではありません）



今日、Webアプリケーションが単一のXHR要求なしで実行できることを覚えている人はほとんどいません。AJAX（非同期JavascriptおよびXML）は、ページを再読み込みせずにデータを読み込むという優れた機能を提供します。このコンセプトは、ほとんどの最新のSPAの基礎となっています。

しかし、そのように与えられるものは何もありません。あなたはすべてを支払う必要があります。AJAXの概念は非常にシンプルに見えますが、サーバーからデータを要求するレベルでも、多くの問題に対処できます。

はじめに、AJAXを使用した最も単純なSPAアプリケーションを作成しましょう。

initApp();

function initApp() {
   document.body.innerHTML = `
           <h3>Employees list</h3>
           <ul id="employees-list"></ul>
           <button id="load-employees">Load employee</button>
   `;
   document.getElementById('load-employees').addEventListener('click', loadEmployee);
}

function loadEmployee() {
   fetch('http://dummy.restapiexample.com/api/v1/employee/1')
       .then(res => res.json()).then(({employee_name}) => addEmployee(employee_name));
}

function addEmployee(employeeName) {
   const employeeElement = document.createElement('li');
   employeeElement.innerText = employeeName;
   document.getElementById('employees-list').appendChild(employeeElement);
}

すべてが非常にシンプルです。ボタンをクリックすると、サーバーにデータをリクエストし、データを受信すると、リストに要素を追加します。

すでに述べたように、この段階では多くのことがうまくいかない可能性があり、トピックをさらに拡張するために、最初に少し理論を分析します。

フォールトトレラントインターフェイスを構築するための2つの哲学




優雅な劣化


これは、ユーザーに最初から可能な限り多くの機能が提供されるインターフェース設計哲学です。そして、システムのいずれかの部分で障害が発生した場合にのみ、それに依存する機能が無効になります。複雑に聞こえますが、以下では例を使って分析します-より明確になります。

プログレッシブエンハンスメント


代替/並列の哲学-プログレッシブエンハンスメントがあります。その中で、動きは反対方向に進みます。最初に、ユーザーには最小限の（または中程度の）機能セットが提供されます。そして、残りを初期化するために、彼らの仕事に必要なシステムの部分のサポートが最初にチェックされます。

通常、彼らがブラウザーアプリケーションのコンテキストで優雅な低下と漸進的な強化について話すとき、それらはブラウザー間の互換性または適応性を意味します。これらの概念を説明する人気のある例があります。アプリケーションにページ印刷機能があるとします。これを行う場合：

<body>
<a href="javascript:window.print()" id="print-btn">Print</a>
<script>
   const printButton = document.getElementById('print-btn');

   if (printButton && typeof window.print !== 'function') {
       printButton.parentNode.appendChild(document.createTextNode('  '));
       printButton.parentNode.removeChild(printButton);
   }
</script>
</body>

この場合、すぐに印刷ボタンが表示されるため、これは適切な低下ですが、ブラウザで印刷がサポートされていないことに気付いた場合、機能を削除します。

PS：元の例では、noscriptタグが優雅な低下を示すために使用されていましたが、非常に古くなっているようです。

if(typeof window.print === 'function') {

   document.body.appendChild(printButton);

予期しないHTTPステータスコード




最も単純なケースは、サーバーが予期した間違ったステータスコード（500など）を返す場合です。これは一般的なシナリオであり、おそらくそれを処理するためのツールがいくつかあります。たとえば、「サーバーエラーが発生しました」という通知をユーザーに表示します。これは明らかに低下ですが、どれほど優雅ですか？ここでプログレッシブエンハンスメントを適用することは可能ですか？いいえ、これは間違いなく進歩的な強化の場所ではありません-機能はすでに低下しています。あなたはこの問題を美しく打つことができるだけです：

  1. この状況が一般にクライアントで発生したことを確認するために、将来的にそれを防止します。これには、通常、エラーロガーが使用されます（例：sentry.io）。
  2. , . , . - , .
  3. . , «». .
  4. . HTTP- , , . , .
  5. , - , , 403. , . .




バックエンドを信頼しないでください！サーバーはコード200で応答できますが、応答の本文では必要な間違ったデータを返します。この状況では、予期しないステータスコードの場合と同じことができますが、答えが本当に無効であるかどうかを判断するのは困難です。

typescriptで書く場合は、typescript-json-schemaというすばらしいツールがあります。これを使用すると、typescriptインターフェースからJSONスキームを生成し、それらを使用して実行時にデータを検証できます。

長い答え




これは少数が期待する打撃です。エラーや無効な応答データを覚えている場合でも、タイムアウトを覚えていることはほとんどありません。原因はサーバーアプリケーションだけでなく、インターネットプロバイダーやクライアントデバイスである可能性もあります。

それを忘れないでください。画面上で回転する円と顔を合わせるよりも、リクエストが通常よりも長いことをユーザーに通知することをお勧めします。リクエストの実行に割り当てられた時間が経過すると、前の2つの状況と同様のシナリオを実行できます。

インターネットの欠如




Googleドキュメントにはオフラインモードがあることを知って、とても感動しました。インターネットのない飛行機で記事を書き終えることを決心したとき、それは私を大いに助けました。

もちろん、アプリケーションは異なり、それらの多くはインターネットなしでは実質的に役に立たない。しかし、これらのアプリケーションでも、接続が不足している場合に対処し、有益なメッセージを表示できます（ただし、「Chrome」でティラノサウルスをプレイするのも好きです）。

さらに、インターネット接続を接続または切断するイベントをリッスンできます。また、たとえば、ウィンドウのオンラインイベント中にデータを自動的に再読み込みします。

フォールトトレラントインターフェイスは容易ではない


全体として、HTTPリクエストを呼び出すときに実装する必要があるアクションのリスト：

  1. エラーをログに記録します。
  2. データをキャッシュして使用します。
  3. 失敗した要求を繰り返します。
  4. インターフェイスをブロックしないでください。
  5. 可能なすべてのサーバー応答を処理します。
  6. サーバーの応答を検証します。
  7. タイムアウトを設定します。
  8. オフラインモード（インターネットの欠如）。

最初は些細なことのように見えたものが、多くの問題を抱えた全体的な哲学に変わりました。もちろん、これはマストヘッドではありません。ただし、アプリケーションの成熟度が高く、本当に高品質のインターフェイスを作成したい場合は、これが開発に値する方向です。

この記事の目的は、HTTPリクエストを処理するときに起こりうる問題について話すことですが、特定のソリューションについては話しません。今日、たとえばAngularのHTTPインターセプターなど、これらの問題を解決することを目的とした多数のライブラリとフレームワークがあります。
考えられる問題について知っていると、インターネットでそれらの解決策を見つけるのがはるかに簡単になります。
