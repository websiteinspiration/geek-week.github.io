So beseitigen Sie blinde Flecken mit visuellen Tests

Vor Beginn des Python QA Engineer- Kurses wurde eine Übersetzung des Artikels erstellt .


Visuelle Fehler sind Fehler in der Darstellung der Anwendung. Sie werden während der gesamten Lebensdauer der Anwendung angezeigt und sehr häufig angezeigt, wenn die Anwendung auf den Bildschirmen von Mobilgeräten unterschiedlicher Größe (auf Laptops, Telefonen, Tablets, Uhren) angezeigt wird.

Das Schlimmste an visuellen Fehlern ist, dass sie mit herkömmlichen automatisierten Tests nicht "abgefangen" werden können. Dies liegt daran, dass die meisten automatisierten Testtools anhand des DOM (Dokumentobjektmodell) den Status der Anwendung melden.

Beispielsweise zeigt die Anwendung aus diesem Tweet nicht die letzte Ziffer des Preises an, der vollständige Preis befindet sich jedoch im DOM. Im ersten Fall beträgt der reale Preis 1250 Neuseeland-Dollar, wird jedoch als 125 Neuseeland-Dollar angezeigt. Sehr großer Unterschied!



Leider zeigen automatisierte Tests, dass alles in Ordnung ist und der Preis korrekt ist, auch wenn Benutzer ein anderes Bild sehen. Heute werden wir darüber sprechen, wie solche Fehler mithilfe visueller Tests erkannt werden können .

Was ist visuelles Testen?


Um visuelle Fehler zu erkennen, benötigen unsere Automatisierungsskripte „Augen“, dh die Fähigkeit, die Anwendung so zu sehen, wie Benutzer sie sehen, um ihr Erscheinungsbild zu überprüfen.

Mit visuellen Testwerkzeugen können wir genau das tun. Sie machen einen Screenshot Ihrer Anwendung, wenn sie den gewünschten Status annimmt, und machen dann jedes Mal, wenn Sie den Test ausführen, einen neuen Screenshot.

Screenshots von Regressionstests werden mit dem „richtigen“ Screenshot verglichen. Wenn signifikante Fehlanpassungen festgestellt werden, schlägt der Test fehl.

Einige dieser Tools verwenden künstliche Intelligenz, um Screenshots anzuzeigen, die eher wie Menschen aussehen. Sie sind viel effizienter als die grundlegenden pixelbasierten Bildvergleichsmethoden.

Soll ich noch einmal von vorne anfangen?


Wenn ich Ingenieuren von den Vorteilen visueller Tests erzähle, fragen sie mich oft, ob sie jetzt alle vorhandenen Tests „wegwerfen“ und von vorne beginnen müssen. Die Antwort ist nein.

Visuelle Tests können in Ihr vorhandenes Testsystem integriert werden. Viele visuelle Testtools unterstützen alle wichtigen Programmiersprachen sowie Automatisierungsframeworks wie Selenium WebDriver, Cypress, Appium und andere.

Trotz der Tatsache, dass visuelle Zusicherungen Ihre vorhandenen funktionalen Zusicherungen ergänzen, besteht der Vorteil des visuellen Testens darin, dass es auch viele Funktionstests enthält .

Im obigen Beispiel aus der Wellington Suite enthalten aktuelle Tests höchstwahrscheinlich Aussagen zu Preisen. Natürlich können Sie sie verlassen und visuelle Tests hinzufügen, um sicherzustellen, dass die Preise korrekt angezeigt werden. Ein Vergleich mit visuellen Testalgorithmen stellt bereits sicher, dass der Preis korrekt ist und korrekt angezeigt wird. Als Ergebnis stellt sich heraus, dass Sie für Tests weniger Code schreiben müssen. Je weniger Code, desto einfacher ist die Wartung.

Was ist, wenn ich keinen Screenshot des gesamten Bildschirms machen möchte?


Trotz der Tatsache, dass ein Screenshot des gesamten Bildschirms Ihnen helfen kann, die Fehler zu finden, an die Sie nicht einmal denken konnten, ist dieser Ansatz für visuelle Tests nicht immer wahr. Manchmal befindet sich der Bildschirm, den Sie testen, noch in der Entwicklung und ändert sich daher häufig, oder es werden Daten auf dem Bildschirm angezeigt, die für Ihre Tests nicht relevant sind.

Bereiche ignorieren


Mit einigen visuellen Testwerkzeugen können Sie bestimmte Bereiche des Bildschirms ignorieren. Diese Methode eignet sich für Bereiche, die nicht mit Tests verbunden sind, z. B. Statusleisten, Datumsangaben, Werbung usw.



Bereichsrahmen


Es kommt vor, dass es keinen bestimmten Bereich gibt, der ignoriert werden muss, aber es gibt einen Bereich, der getestet werden muss, und hier werden Sie auch visuellen Tests zu Hilfe kommen. Durch Kommentieren eines Screenshots oder Festlegen des Bereichslokators im Code können Sie nur einen begrenzten Bereich überprüfen.

Sei vorsichtig mit den Grenzen


Durch Überprüfen des gesamten Bildschirms wird sichergestellt, dass nichts übersehen wird, da eine solche Überprüfung tatsächlich die Dinge enthält, die Sie möglicherweise nicht mit Tests abdecken. Wenn Sie den Umfang der visuellen Tests einschränken, sollten Sie besonders auf Bereiche zwischen den Grenzen achten, die Sie möglicherweise übersehen haben.

Ich empfehle, den Testbereich so weit wie möglich zu erweitern (z. B. einen ganzen Abschnitt anstelle eines bestimmten Elements innerhalb eines Abschnitts) und bei Bedarf die visuelle Aussage mit den traditionellen zu kombinieren.

Was ist mit dynamischen Daten, die von Test zu Test variieren?


Möglicherweise denken Sie, dass visuelle Tests nicht anwendbar sind, wenn sich Daten von Test zu Test ändern, aber dies ist nicht der Fall. Mithilfe künstlicher Intelligenz können visuelle Tests die Muster erkennen, aus denen das Bildschirmlayout besteht, und anschließend einen anderen Vergleichsalgorithmus verwenden, mit dem überprüft wird, ob das Layout während der Regression unabhängig von den darin enthaltenen Daten unverändert bleibt.

Diese Methode ist ideal für Anwendungen wie Nachrichtenseiten, soziale Netzwerke und andere, bei denen die Daten nicht statisch sind.

Gilt das visuelle Testen nur für Webanwendungen?


Visuelle Tests sind nicht auf Webanwendungen beschränkt. Visuelle Tests können für mobile Anwendungen , Designsysteme wie Storybook, PDF-Dateien und einzelne Bilder verwendet werden.

Plattformübergreifende Tests können auch mithilfe von visuellen Gittern durchgeführt werden. Diese Raster machen visuelle Tests schneller und intelligenter und ermöglichen es Ihnen, Tests auf vielen Geräten mit unterschiedlichen Bildschirmgrößen durchzuführen, bei denen Fehler hauptsächlich verborgen sind!

Visuelle Raster unterscheiden sich von normalen darin, dass sie nicht für jede Konfiguration das gesamte Skript ausführen müssen. Stattdessen führt das visuelle Raster das Skript in einer Konfiguration aus, erfasst den Status Ihrer Anwendung und dupliziert diesen bestimmten Status in allen unterstützten Konfigurationen. Dies spart Zeit und Mühe beim Schreiben und Ausführen plattformübergreifender Tests.
