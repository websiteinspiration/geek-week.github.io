過負荷のサーバーを片付ける方法は？

本日翻訳する資料は、サーバーパフォーマンスのボトルネックの発見、問題の修正、システムパフォーマンスの向上、パフォーマンスの低下の防止に特化しています。ここでは、過負荷のサーバーの問題を解決する方法として、次の4つのステップを実行することを提案します。

  1. 状況の評価：サーバーパフォーマンスのボトルネックの特定。
  2. サーバーの安定化：状況を改善するための緊急対策の適用。
  3. システムの改善：システム機能の拡張と最適化。
  4. サーバーの監視：自動ツールを使用して問題の発生を防止します。

2.サーバーの安定化


システムに過負荷のサーバーがあると、システムの他の部分でカスケード障害がすぐに発生する可能性があります。その結果、サーバーが過負荷になっていることがわかった後、サーバーを安定させることが重要であり、それからシステムに深刻な改善を導入するために状況を調査することが重要です。

queryクエリ処理速度を制限する


リクエストの処理速度を制限すると、着信リクエストの数を制限してインフラストラクチャを保護できます。これは、サーバーのパフォーマンスが低下したときに非常に重要です。サーバーの応答時間が長くなると、ユーザーはページを積極的に更新する傾向にあり、サーバーの負荷がさらに増加し​​ます。

リクエストの処理を拒否することは簡単で効果的な方法ですが、外部システムを使用して受信するリクエストの数を制限することにより、サーバーの負荷を減らすのが最善です。これは、たとえば、ロードバランサー、リバースプロキシサーバー、またはCDNです。以下は、この種のいくつかのシステムを操作するための手順へのリンクです。

  • Nginx
  • HAProxy
  • Microsoft IIS

ここでは、リクエスト処理の速度を制限するためのさまざまなアプローチを使用してサーバーの負荷を軽減する方法について説明します。

▍HTTPキャッシング


コンテンツのキャッシュを改善する方法を探します。リソースをHTTPキャッシュ（ブラウザーキャッシュまたはCDNから）からユーザーに提供できる場合は、サーバーから要求する必要がないため、サーバーの負荷が軽減されます。

HTTPヘッダのようなのCache-Control、有効期限、およびETagのは、特定のリソースをキャッシュする方法を示しています。これらのヘッダーの監査と修正は、キャッシュの改善に役立ちます。Service

Workerのキャッシング機能を使用することもできますが、それらは別のキャッシュを使用します。これは、コアブラウザーキャッシングシステムへの支援であり、その代わりではありません。したがって、過負荷のサーバーの問題を修正するときは、HTTPキャッシュの最適化に注力する必要があります。

診断


Lighthouseを 起動し、効率的なキャッシュポリシーメトリックを使用して静的アセットを提供するを見て、キャッシング時間が短いまたは中程度のリソースのリスト（存続時間、TTL）を確認します。リストされているリソースを確認し、TTLを増やすことを検討してください。以下は、さまざまなリソースに適用できる推定キャッシュ用語です。

  • 静的リソースは長期間（1年間）キャッシュする必要があります。
  • 動的リソースは、短時間（3時間）キャッシュする必要があります。

キャッシュ設定


必要なキャッシュキャッシング時間（秒単位）をCache-Controlmax-ageヘッダー ディレクティブに書き込む必要があります。異なるシステムでこのヘッダーを設定する手順は次のとおりです。

  • Nginx
  • Apache
  • マイクロソフト

このディレクティブmax-ageは、キャッシングに影響を与える多くのディレクティブの1つにすぎないことに注意してください。キャッシュ機能に影響する他のディレクティブとヘッダーがあります。この問題をよりよく理解するために、この HTTPキャッシングガイドを読むことをお勧めします。

systemシステム機能の段階的な低下


システムの機能を徐々に低下させることは、サーバーから過度の負荷を取り除くことを目的とした機能を一時的に制限する戦略です。この概念は、さまざまな方法で適用できます。たとえば、本格的なアプリケーションではなく静的テキストページをクライアントに配信したり、検索を無効にしたり、通常より少ない検索結果を返したりします。これには、プロジェクトの基本的な機能に影響を与えない、リソースを集中的に使用する機能を無効にすることが含まれます。ここでの主な注意は、アプリケーションの主要な機能にあまり影響を与えずに破棄できる機能を無効にすることです。

3.システムの改善


CDCDNの使用


静的リソースを提供するタスクは、サーバーからコンテンツ配信ネットワーク（CDN）に転送できます。これにより、サーバーの負荷が軽減されます。

CDNの主な機能は、ユーザーの近くにある大規模なサーバーネットワークを使用して、資料をユーザーにすばやく配信することです。さらに、一部のCDNはパフォーマンス関連の追加機能を提供します。その中で-データ圧縮、ロードバランシング、メディアファイルの最適化。

CDN設定


CDNの利点は、ネットワークを所有する企業が世界中に分散したサーバーの大きなグループを持っている場合に明らかになります。したがって、独自のCDNサービスをサポートすることはほとんど意味がありません。典型的なCDN設定は、約30分かかるかなり迅速な手順です。これは、DNSレコードを更新して、CDNを指すようにすることです。

問題解決


CDNを使用してリソースがキャッシュされていない場合は、次の条件に該当するかどうかを確認してください。

  • リソースにはCache-Control：public headerがあります。
  • リソースにはCache-Control：s-maxage、Cache-Control：max-age、またはExpiresヘッダーがあります。
  • リソースには、Content-Length、Content-Range、またはTransfer-Encodingヘッダーがあります。

computingコンピューティングリソースのスケーリング


コンピューティングリソースをスケーリングする決定は、注意して行う必要があります。多くの場合、スケーリングを利用することで特定の問題を解決できますが、これは不都合なことですが、システムを不必要に複雑にし、そのサポートのコストを不当に増加させる可能性があります。

診断


最初のバイトまでの時間の指標（Time To First Byte、TTFB）が高い場合は、サーバーが限界に近づいていることを示している可能性があります。TTFB情報は、 Lighthouseレポートのサーバー応答時間の短縮（TTFB）セクションにあります。

状況をより深く調査するには、何らかの監視ツールを使用して、プロセッサの使用状況を分析する必要があります。現在または予測されたプロセッサ負荷の値が80％を超える場合は、サーバーの容量を増やすことを検討する必要があります。

問題解決


システムにロードバランサーを追加すると、複数のサーバー間でトラフィックを分散できます。ロードバランサーはサーバープールの前にあり、適切なサーバーにトラフィックを分散します。クラウドプロバイダーは、ユーザーロードバランサー（GCP、AWS、Azure）を提供しますが、HAProxyまたはNGINXを使用して独自のバランサーを使用することもできます。ロードバランサーが機能する準備ができたら、システムにサーバーを追加できます。

負荷分散に加えて、ほとんどのクラウドプロバイダーはコンピューティング能力の自動スケーリングを提供しています（GCP、AWS、Azure）。自動スケーリングは、負荷分散に関連付けられています。つまり、高負荷時のリソースの自動スケーリングでは、追加のリソースが割り当てられ、低負荷時には不要なリソースが無効化されます。ただし、これを考慮しても、自動スケーリングも普遍的なソリューションではないことに注意してください。サーバーの自動起動には時間がかかります。自動スケーリング構成には、深刻な構成が必要です。したがって、自動スケーリングの複雑なシステムを適用する前に、ロードバランサーを使用して比較的単純な構成を試す価値があります。

4.サーバーの監視


サーバー監視ツールは、コントロールパネルを使用したデータ収集と視覚化をサポートします。サーバーのパフォーマンスに関連するさまざまなイベントをユーザーに通知できます。これらのツールを使用すると、サーバーのパフォーマンスの問題を防止および軽減できます。

監視システムを設定するときは、可能な限りシンプルにするように努力する必要があります。過剰なデータ収集と頻繁な通知は、悪影響を引き起こす可能性があります。収集されるデータの範囲が広いほど、また収集される頻度が高いほど、それらを収集して保存するのにかかるコストは高くなります。そして、サーバーの状態の責任者が軽微なイベントに関するメッセージで攻撃された場合、結果として、彼はこれらのメッセージを無視します。

通知には、問題を一貫して正確に説明するメトリックを含める必要があります。たとえば、サーバーの応答時間（レイテンシ）はこれに特に適したメトリックです。これにより、多数の問題状況を識別でき、ユーザーがサーバーをどのように認識しているかに直接関係します。プロセッサ使用率レベルなどの低レベルのメトリックに基づく通知は、便利なアドオンの役割を果たすことができますが、問題のごく一部を示すことができます。さらに、通知は平均指標に基づくのではなく、95〜99パーセンタイルに対応する指標に基づいている必要があります。それ以外の場合、平均を分析すると、すべてのユーザーに影響を与えない問題を簡単にスキップできます。

監視設定

  • Aws
