Kaspresso：待っていた自動テストフレームワーク

Androidの自動テストを行ったことがある、または行っている人は誰でも、それがどんな苦痛かを知っています。
タスクや問題の量に飽きて、休暇が役に立たなくなる。人々は自動テストのためにやめた。

痛み、苦しみ、苦しみは必然的に何か新しい美の出現につながります。私たちは踏みならなければならなかったすべてのレーキをまとめようとし、AvitoとHHのメンバーと力を合わせ、自動テストとの関係を比類なくより良く、より実り多いものにするものを作成しました。

私はすぐに2つのかなりの高品質のビデオがあることに注意してネットワークに捧げKaspressoとAdbServer（Kaspressoための補助が、しかし同時に、強力かつ独立したプロジェクトで）：

は、ライブラリの作成の歴史を詳細に説明します。ある問題の解決から別の問題の解決へとどのように移行したか、そして最後に何が起こったかです。私はそれを強くお勧めします。

したがって、この記事では、このフレームワークの概要と解決するタスクをすばやく理解できるように、主要なポイントとチップについてのみ説明します。根性と詳細は、ビデオとドキュメントに記載されています。

そして、なぜ独自のフレームワークが必要なのでしょうか。


自動テストを書き始めるすべての開発者は必然的に質問をします：

  1. 自動テストの作成を開始するにはどうすればよいですか？
  2. 選択するツールは？
  3. 必要なツールがない場合はどうすればよいですか？
  4. ベストプラクティスは何ですか？

これらすべてに加えて、各チームがこれらの問題を何らかの方法で独自の方法で解決しようとしていることを追加します。結果として、それらに対する単一の答えはなく、正しい方法を調べる場所はありません。したがって、自動テストの作成とサポートは企業にとって高額であり、自動化自体の適切性に疑問を投げかけています。

一方、テストの自動化は良い目標を追求しています。それはあなたが常に緑でリリース可能なウィザードを持っていることを可能にします。これは、リリースをすばやくロールアウトできることを意味します。
唯一の問題は、上記の質問に対する答えを見つけることです。しかし、それらはすべてのチームでほぼ同じです。それでは、なぜソリューションを自動化しないのですか？

フレームワークに何を求めていますか？


フレームワークからの期待を少し明らかにしましょう。

読みやすさ


デフォルトでは、AndroidではEspressoライブラリのみを使用できます。もちろん、Appium + Cucumberもあり、理論的には2つのプラットフォームで同時にテストを作成できます。しかし、コミュニティは最初の楽器に自信を持って移行しています。上記のライブラリの長所と短所をすべて説明するわけではありません。ネットワークにはこれに関する情報が満載です。たとえば、ここは比較的最近のリンクの1つです。AppiumとEspressoのリンクです。何を選び、どのように使うか？

だから、エスプレッソ。悪いツールではありませんが、そのAPIは少し裏返されています。簡単なコードを見てみましょう：

@Test
fun espressoTest() {
    onView(allOf(allOf(withId(R.id.espresso),
        isDescendantOfA(withId(R.id.coffee_variates))),
        isDescendantOfA(withId(R.id.content))))
        .check(matches(withEffectiveVisibility(View.VISIBLE)))
}

あなたはそれを理解するために考えなければなりませんよね？

タイとオーストラリアの神々から、カカオライブラリが提供されました。これにより、テストで何が起こっているかが一目でわかります。以下は同じコードですが、Kakaoを使用しています。

@Test
fun kakaoTest() {
    mainScreen {
        myView.isVisible()    
    }
}

ずっといい。しかし、テストケース全体を自動化したとしましょう。コードはどのようになりますか？

@RunWith(AndroidJUnit4::class)
class OpenHomeScreenTest: TestCase() {
   @Test
   fun test() { 
      MainScreen() {
         homeButton.click()
      }

      HomeScreen() {
         title {
            isVisible()
            hasAnyText()
         }
      }
   }
}

主な問題は、このテストを見ると、自動化したテストケースと関連付けるのが難しいことです。もちろん、ログなどを追加することもできます。または、dslと入力して、テストの外観を即座に変換します。

@RunWith(AndroidJUnit4::class)
class OpenHomeScreenTest: TestCase() {
   @Test
   fun test() { 
      step(“1. Open Home screen”) {
         MainScreen() {
            homeButton.click()
         }
      }

      step(“2. Check Home title”) {
         HomeScreen() {
            title {
               isVisible()
               hasAnyText()
            }
         }
      }
   }
}

そうですね、見た目はまったく異なります。

安定


UIテスト用のライブラリはすべて失敗します。同じアクションを50回正常に実行でき、明らかな理由もなく51回目の休憩時間に実行できます。そして52回目の走行では、すべてが再び元気です。そして、そのような「フラッキング」はきちんとあなたの神経を台無しにすることができます。

私たちは、カカオエスプレッソのすべてのアクションをインターセプトしてみて、そのようなランダムエラーの処理を目的とした動作を既に追加していないのではないかと考えました。

これにより、カカオライブラリのバージョン2.1が誕生し、すべてのEspresso呼び出しに統合できるようになりました。

さらに、独自のインターセプターを作成しました。インターセプターを使用すると、ダイヤルピアでの動作を変更したり、たとえばログに記録したりできます。さらに、これらのインターセプターはカスタマイズ可能であるため、ニーズに合わせてカスタマイズできます。詳細については、ドック。

具体的には、不安定なテストとの戦いの一環として-アクションの一部が例外をスローした場合、Kaspressoは次のことを試みます。

  • スクロール。たぶん、あなたの見方は単に画面上に見えないでしょう。
  • 神から来る可能性のあるシステム対話を削除します。
  • 切断された通話を2秒間繰り返します。

これらの手順により、薄片状のテストで問題を完全に解決します！

すでに多くのテストがある場合のKaspressoの接続および構成方法


主な魅力は、すでにKakaoで多くのテストを作成していて、Kaspressoを実装したい場合は、何も書き直す必要がないことです。特別なTestCaseクラスからテストクラスを継承するだけです。そして、それだけです！

そうだった：

@RunWith(AndroidJUnit4::class)
class OpenHomeScreenTest {
   private val mainScreen = MainScreen()
   private val homeScreen = HomeScreen()

   @get:Rule
   val activityTestRule = 
      ActivityTestRule(MainActivity::class.java, true, false)

   @Test
   fun test() { ... }
}

それは次のようになりました：

@RunWith(AndroidJUnit4::class)
class OpenHomeScreenTest : TestCase() {
   private val mainScreen = MainScreen()
   private val homeScreen = HomeScreen()

   @get:Rule
   val activityTestRule = 
      ActivityTestRule(MainActivity::class.java, true, false)

   @Test
   fun test() { ... }
}

継承が気に入らない場合は、同様のTestRuleクラスを使用してください。

すでに述べたように、Kaspressoは非常に柔軟でカスタマイズ可能なフレームワークです。すべての設定は、同じ名前のKaspressoクラスを通じて使用できます。

デフォルト設定はdefaultです。何かをカスタマイズしたい場合は、次のようになります。

@RunWith(AndroidJUnit4::class)
class OpenHomeScreenTest : TestCase(
   Kaspresso.Builder.default().apply {
     viewBehaviorInterceptors.add(MyInterceptor())
     flakySafetyParams.timeoutMs = 1_000
  }
) {
   private val mainScreen = MainScreen()
   private val homeScreen = HomeScreen()

   @get:Rule
   val activityTestRule = 
      ActivityTestRule(MainActivity::class.java, true, false)

   @Test
   fun test() { ... }
}

つまり、TestCaseコンストラクターを通じてKaspresso.Builderが利用可能になり、必要なすべての設定を設定できます。コンフィギュレータの詳細はドキュメントに記載されています。

アップグレードスクリプトのテスト

残念ながら、純粋なEspressoではこれを行うことは不可能です。テスト済みのapkを再インストールすると、テストが失敗するためです。どういうわけかランナーをだまそうとすることができますが、そのような改善がどのように見え、どれほど安定するかを想像するのは難しいです。
したがって、Kaspressoでは、UiAutomatorに基づいてこの問題の解決策を準備しています。ただし、上部では、カカオと非常によく似ており、interseptingも同じようにサポートされている、おなじみのdslがすべて突き出ています。
