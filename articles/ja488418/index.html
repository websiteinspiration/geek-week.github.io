以前の記事では、RESTful Webサービスを作成しましたが、今度はセキュリティについて説明します

前書き


前回の投稿では、Java Spring BootフレームワークとMongoDBフレームワークを使用してREST APIを作成する方法について説明しました。ただし、APIは認証を必要としなかったため、おそらくまだ使用する準備ができていません。したがって、このガイドでは、Springの組み込みセキュリティ環境を使用して、このAPIに認証レベルを追加する方法を示します。

APIに認証が必要なのはなぜですか？


APIは内部データと対話するためのシンプルなインターフェースを提供するため、誰もこのデータにアクセスして変更することを望まないことは理にかなっています。認証により、信頼できるユーザーのみがAPIにアクセスできるようになります。

使い方


ユーザー名とパスワード を使用する基本的なHTTP認証を使用します。ユーザー名とパスワードは、次の形式でコロンで1行に区切られていますusername:password。

この行はBase64エンコードを使用してエンコードされるため、admin:p@55w0Rd次の行でエンコードされますYWRtaW46cEA1NXcwUmQ=（「p @ 55w0Rd」よりも強力なパスワードを使用することをお勧めします）。ヘッダーを追加することで、この認証をリクエストに添付できますAuthentication。前の例のこの見出しは、次のようになります（「基本」は、パスワードが基本HTTP認証を使用することを意味します）。

Authentication: Basic YWRtaW46cEA1NXcwUmQ=

パスワードハッシュ用のBCrypt


ハッシュは一方向の暗号化アルゴリズムです。実際、ハッシュした後、元のデータがどのようになっていたかを見つけることはほとんど不可能です。BCryptハッシュアルゴリズムは、最初にテキストをソルトし、次にそれを60文字の文字列にハッシュします。Java BCryptエンコーダーはmatches、文字列がハッシュと一致するかどうかをチェックするメソッドを提供します。たとえば、p@55w0RdBCryptでハッシュされたパスワードには意味がある場合があり$2b$10$Qrc6rGzIGaHpbgPM5kVXdeNZ9NiyRWC69Wk/17mttHKnDR2lW49KSます。matches暗号化されていないハッシュされたパスワードに対してBCrypt メソッドを呼び出すと、値が取得されますtrue。これらのハッシュは、Spring Securityに組み込まれたBCryptエンコーダーを使用して生成できます。

なぜパスワードをハッシュする必要があるのですか？


大企業からのパスワードの盗難を招いた最近のサイバー攻撃について聞いたことがあります。では、なぜハッキング後にパスワードを変更することのみが推奨されるのですか？これらの大企業は、パスワードが常にデータベースでハッシュされることを確認しているためです！

このようなデータハッキングの後でパスワードを変更することは常に価値がありますが、パスワードハッシュは一方向のアルゴリズムであるため、ユーザーの実際のパスワードを見つけるのが非常に困難になります。実際、適切にハッシュされた複雑なパスワードを解読するには数年かかる場合があります。これにより、パスワードの盗難に対する保護が強化されます。また、Spring Securityはハッシュを単純化するので、本当の質問は「なぜいけないのか」です。

認証サービスの作成


Springにユーザーデータの場所と、認証に必要な情報の場所を知らせる必要があります。これを行うには、認証サービス（Authentication Service）を作成します。まず、src/main/resources/java/[package name]呼び出されたサービスに新しいフォルダーを作成します。この構成フォルダーにという名前で新しいファイルを作成できますMongoUserDetailsService.java。

MongoUserDetailsS​​ervice.java


このクラスには1つのメインコンポーネントがあるため、ここではクラス全体を示し、以下で説明します。

package com.example.gtommee.rest_tutorial.services;

import com.example.gtommee.rest_tutorial.models.Users;
import com.example.gtommee.rest_tutorial.repositories.UsersRepository;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.core.userdetails.UsernameNotFoundException;
import org.springframework.stereotype.Component;

import java.util.Arrays;
import java.util.List;

@Component
public class MongoUserDetailsService implements UserDetailsService{
  @Autowired
  private UsersRepository repository;

  @Override
  public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
    Users user = repository.findByUsername(username);

    if(user == null) {
      throw new UsernameNotFoundException(“User not found”);
    }

    List<SimpleGrantedAuthority> authorities = Arrays.asList(new SimpleGrantedAuthority(“user”));

    return new User(user.getUsername(), user.getPassword(), authorities);
  }
}

このスニペットは、ファイルに必要なインポートから始まります。さらに、このセクションimplements UserDetailsServiceは、このクラスがユーザーを検索および認証するためのサービスを作成することを示しています。次に、このアノテーション@Componentは、このクラスを別のファイルに組み込むことができることを示しています（たとえば、SecurityConfigurationファイル。これについては、いくつかのセクションで説明します）。

注釈@Autowiredを超えますprivate UsersRepository repository。実装の例です。このプロパティはUsersRepository、作業用のインスタンスを提供します。アノテーション@Overrideは、このメソッドがデフォルトのUserDetailsS​​erviceメソッドの代わりに使用されることを示しています。まず、このメソッドは、で宣言しUsersたメソッドfindByUsernameを使用して、MongoDBデータソースからオブジェクトを取得しますUsersRepository。

次に、メソッドはユーザーが見つかったかどうかを確認します。次に、ユーザーに権限/役割が付与されます（これにより、アクセスレベルに認証レベルが追加されますが、このレッスンでは1つの役割で十分です）。最後に、このメソッドは、と認証されたユーザーUserを含むSpringオブジェクトを返します。usernamepasswordrole

セキュリティ構成の作成


データベースとハッシュアルゴリズムを使用するには、Springの組み込みセキュリティプロトコルの一部を再定義する必要があるため、特別な構成ファイルが必要です。それを作成するにはsrc/main/resources/java/[package name]、という名前configで新しいフォルダーを作成する必要があります。また、この構成フォルダーにという名前で新しいファイルを作成する必要がありますSecurityConfiguration.java。このファイルにはいくつかの重要な部分があるため、まずSecurityConfiguration基本クラスから始めましょう。

SecurityConfiguration.java


package com.example.gtommee.rest_tutorial.config;

import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;

@Configuration
@EnableConfigurationProperties
public class SecurityConfiguration extends WebSecurityConfigurerAdapter {
 @Autowired
 MongoUserDetailsService userDetailsService;
}

対処するのにすでに十分なので、上から始めましょう。注釈@Configurationは、クラスにJava Beanが含まれることを示しています。注釈@EnableConfigurationPropertiesは、特別な構成Beanとしてクラスに何が含まれるかを示します。次に、この命令はextends WebSecurityConfigurerAdapter親クラスWebSecurityConfigurerAdapterを構成クラスにマップし、セキュリティルールが確実に遵守されるようにするために必要なすべてのものをクラスに提供します。最後に、クラスはインスタンス（@Autowired）を自動的に挿入しMongoUserDetailsServiceます。これは後でこのファイルで使用できます。

認証ステップ


次に、Spring Securityにユーザー認証の処理方法を通知する必要があります。デフォルトでは、Spring Securityには事前定義されたユーザー名とパスワード、CSRF保護、およびセッション管理があります。ただし、ユーザーはユーザー名とパスワードを使用してデータベースにアクセスする必要があります。さらに、ユーザーはログインするのではなく、リクエストごとに再認証されるため、CSRF保護とセッション管理は必要ないためconfigure、デフォルトの認証スキームをオーバーライドする名前のメソッドを追加して、Springに正確にどのように伝えるかを指定できます。認証を処理するため、次のようになります。

@Override
protected void configure(HttpSecurity http) throws Exception {
 http
   .csrf().disable()
   .authorizeRequests().anyRequest().authenticated()
   .and().httpBasic()
   .and().sessionManagement().disable();
}

繰り返しになりますが、ここでは非常に多くのことが起こっているため、段階的に整理します。この注釈@Overrideはconfigure (HttpSecurity http)、デフォルトのSpring構成の代わりにメソッドを使用するようにSpring Bootに指示します。次にhttp、実際の構成が行われるオブジェクトの一連のメソッドを呼び出します。これらのメソッドは次のことを行います。

  • csrf().disable()：APIには必要ないため、CSRF保護を無効にします
  • authorizeRequests().anyRequest().authenticated()：エンドポイントへのすべてのリクエストは承認される必要があることを宣言します。そうでない場合は拒否されます。
  • and().httpBasic(): Springそのため、基本的なHTTP認証（前述）を想定しています。
  • .and().sessionManagement().disable()：APIには必要ないため、ユーザーのセッション情報を保存しないようにSpringに指示します

Bcryptエンコーダーの追加


ここで、Springにパスワードのハッシュ化と比較にBCryptエンコーダーを使用するように指示する必要があります。これは難しい作業のように思えますが、実際は非常に簡単です。クラスに次の行を追加するだけで、このエンコーダを追加できますSecurityConfiguration。

@Bean
public PasswordEncoder passwordEncoder() {
   return new BCryptPasswordEncoder();
}

そして、すべての仕事！この単純なBeanは、使用するPasswordEncoderがSpring BootであるBCryptPasswordEncoder()ことをSpringに通知し、パスワードハッシュをエンコードおよび比較します。Spring Bootには他にもいくつかのパスワードエンコーダーが含まれています-実験したい場合は、それらを試すことをお勧めします！

認証マネージャーを指定する


最後に、認証に（前のセクションで作成した）SecurityConfiguration使用することを指定する必要がありますMongoUserDetailsService。これを行うには、次の方法を使用します。

@Override
public void configure(AuthenticationManagerBuilder builder) 
throws Exception {
  builder.userDetailsService(userDetailsService);
}

このメソッドは、デフォルトの設定をオーバーライドするだけでAuthenticationManagerBuilder、代わりに独自のカスタムデータ転送サービスを置き換えます。

最終的なSecurityConfiguration.javaファイル


import com.example.gtommee.rest_tutorial.services.MongoUserDetailsService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;

@Configuration
@EnableConfigurationProperties
public class SecurityConfiguration extends WebSecurityConfigurerAdapter {
 @Autowired
 MongoUserDetailsService userDetailsService;

 @Override
 protected void configure(HttpSecurity http) throws Exception {
   http
     .csrf().disable()
     .authorizeRequests().anyRequest().authenticated()
     .and().httpBasic()
     .and().sessionManagement().disable();
 }

 @Bean
 public PasswordEncoder passwordEncoder() {
   return new BCryptPasswordEncoder();
 }

 @Override
 public void configure(AuthenticationManagerBuilder builder) 
throws Exception {
   builder.userDetailsService(userDetailsService);
 }
}
