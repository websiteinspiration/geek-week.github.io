Gradle + LLVM

この短い投稿は、ソースコードのダウンロードやフレームワークの構築に煩わされることなく、LLVMですぐに作業を開始したい場合に役立ちます。誰が期待どおりの結果を達成するためにあいまいなCMakeスクリプトをぶつけたくないのか、そして最後に、怠惰な人のために:)

Gradleビルドスクリプトのほんの数行で、優雅にそれを行う方法を説明します。

LLVMでコードを書く必要に直面して、私はその「ヒット」に気づきました。 :)「ヒット」はすでにプロジェクトのビルド構成段階にあります。 LinuxとWindowsの両方でプロジェクトを均一に構築する必要がありました。もちろん、プロジェクトを進めるたびに、LLVM全体を構築したくありませんでした。そして、最初は公式サイトのビルド済みバージョンのライブラリに依存していましたが、実際には、RTTIと例外なしでビルドされたLinuxバイナリしかありませんでした。 Windows用のインストーラーがありますが、clang内でのみLLVM-xxx-win64.exeと呼ばれています。

さて、ビルドの付属物には、Linuxのリリースバージョンをビルドするために構成段階で設定する必要があるCMakeシステムがあります。

cmake -DCMAKE_BUILD_TYPE=Release

Windowsでは、これは何らかの理由で十分ではなく、ビルドコマンド自体の構成をさらに変更する必要があります。 

cmake --build . --config Release

一般的に、私はGradleに目を向けて、ビルドを構成する手間を取り、不必要なルーチンから私を救い、最も必要なハイレベルなものだけを残しておく優れたプラグインを書くことにしました。

実際、以下の議論は彼についてのみです：cpp-llvm。
LLVMと統合するために必要な最小構成は次のとおりです。

plugins {
    id 'org.bitbucket.akornilov.cpp-llvm' version '0.1'
}

llvm {
    version = '9.0.0' // Required version.
}

仕事の要件：

  • インストール済みのJava（8以上）。
  • インストールされたGradle（記事の執筆時に最新バージョン5.6.2を使用しており、古いバージョンでプラグインをテストしていませんでした）
  • Linux上のGcc / CLang（少なくともC ++ 11をサポートするバージョン）。
  • Windows上のMSVC 2019（例：Community Edition）。
  • プラグインは「cpp-application」または「cpp-library」と組み合わせて使用​​する必要があります。
  • 最初の構築時にネットワークにアクセスしてLLVMバイナリファイルをダウンロードする（またはオフラインモード、以下を参照）。

ビルドに必要なすべてのLLVMバイナリはローカルキャッシュ（ユーザーディレクトリのgradleフォルダー内）に読み込まれ、他のビルドで再利用されます。

最初の開始では、建設の準備にはかなりの時間がかかる可能性があります。デバッグバージョンの場合は約300MB、リリースバージョンの場合は30MBがネットワークからダウンロードされます（公式バージョンの場合も約300MBです）。

静的リンクのみが使用されます。何らかの未知の理由により、ビルドLLVMバインディングはMSVCで動的ライブラリを作成できません。

その他のプラグイン構成


バイナリ（.tar.xzまたは.tar.gz）でアーカイブをダウンロードするための特定のアドレスを指定できます。


llvm {
    version = '7.0.1'
    serverUrl = 'http://releases.llvm.org/7.0.1/clang%2bllvm-7.0.1-x86_64-linux-gnu-ubuntu-18.04.tar.xz'
}

主なことは、指定されたバージョン（version = xxx）が、ダウンロード用に選択されたアーカイブ内の実際のバージョンと一致することです。そうでない場合、ビルド構成が正しくない可能性があります。

既成のLLVMバイナリ（ダウンロードまたは自分でビルドしたもの）を自律的に操作できます


llvm {
    version = '7.0.1'
    localPath = '<path to unpacked LLVM build>'
}

localPathは、プラグインがサブディレクトリ「include」と「lib」を参照することを期待しているローカルファイルシステム上の既存のフォルダーを指す必要があります。公式ビルドで使用されるディレクトリ構造。セルフチューニング後に別のフォルダー構造がある場合は、それを「くし」にする必要があります。:)

すでに述べたように、バージョンも前の段落で一致している必要があります。
この場合、LLVMに関連するものは何もネットワークからロードされません。

個々のコンポーネントを操作します。


llvm {
    version = '9.0.0'
    components = ['Engine', 'OrcJIT']
}

LLVMの特定の部分のみを使用する場合は、このパラメーターに注意を払う必要があります。たとえば、OrcJITだけが必要な場合、なぜ150個すべての静的ライブラリをリンクするのでしょうか。特定のコンポーネントを選択すると、リンカーの寿命が大幅に短縮され、時間を節約できます。

使用可能なすべてのコンポーネントのリストは、次のようになっています。

gradle llvmComponents

Linuxでは、このオプションを使用すると、デバッグバージョンのリンク時間を大幅に削減できます。


llvm {
    version = '9.0.0'
    forceReleaseLinux = true
}

この場合、ビルドのデバッグバージョンではLLVMリリースバージョンが強制的に使用されます。ほとんどの場合、LLVM自体をデバッグする必要がない限り、それを使用することをお勧めします。残念ながら、MSVC 2019では1つのビルドにリリースライブラリとリリースライブラリを混在させることができないため、このオプションはWindowsのアセンブリには影響しません。

さて、結論として、私はもちろん、このプラグインを他のプラグインと組み合わせて使用​​することをお勧めします:) 。これについては、この記事で説明しています。

cpp-build-tunerは、コンパイル時間と結果のバイナリのサイズを最適化するのに役立ちます。

cpp-ide-generatorを使用すると、いくつかのIDEと簡単に統合できます。その構成では、LLVMヘッダーへのパスがすでに存在し、IDEはそれらに正しくインデックスを付けることができます。



cpp-ide-generatorを使用するときに起こりうる問題：

FAILURE: Build failed with an exception.

* What went wrong:

A problem occurred configuring project ':llvm-app'.

> Cannot change dependencies of configuration ':llvm-app:cppCompileDebug' after it has been resolved.

これが表示された場合は、次のようにcpp-ide-generatorをcpp-llvmの後に追加する必要があることを意味します。

plugins {
    id 'cpp-application'
    id 'org.bitbucket.akornilov.cpp-build-tuner' version '0.7'
    id 'org.bitbucket.akornilov.cpp-llvm' version '0.1'
    id 'org.bitbucket.akornilov.cpp-ide-generator' version '0.5'
}
