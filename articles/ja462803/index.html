あるERPシステムから別のシステムへの移行


多くの場合、会社が新しいERPシステムに移行しても、購入と完了の財務コストは抑えられませんが、古いシステムから移行する必要があります。彼らには、1つのプログラムでビジネスプロセスを実行する数百または数千のユーザーがいて、どういうわけか、ビジネスを停止することなくすべてを別のプログラムに転送する必要があります。

近年、私たちはこの方法を数十回達成しており、これを実現するために最も痛みの少ない方法をすでに開発しています。

誰かが1つの製品を別の製品と交換するとき、最初に気づくのは、彼が彼から連れ去られたこと、そして彼が受け取ったものを理解するまでです。。したがって、多くの場合、最初の反応は次のとおりです。実際には、1人のユーザーが、古いシステムではフォーム間の5つの遷移で行われていた新しいアクションに不満を感じているという状況がありました。しかし、彼はそのような自動性に磨きをかけたため、人にとっては、2つの新しいがはるかに単純なアクションを実行するよりも、最初はそれを実行するほうが実際には高速でした。

もう1つの問題は、移行中に、顧客がビジネスプロセスをビジョンに変更することを好むことです。現在の「現状のまま」のプロセスが機能することが保証されているため、これは十分に注意して扱う必要があります。製品に組み込まれたプロセスは、他の顧客が使用しているため、原則として機能します。しかし、新しい発明は理論的には美しく見えるかもしれませんが、実際にはそれらはいくつかの些細なことに遭遇します。そのため、プロセス全体が停止する可能性があります。

マスターデータ


すべては、マスターデータのインポートから始まります。特に、小売業では、商品、商品、請負業者、および店舗のグループのディレクトリのインポートが最初に実現されます。段階的な移行では、マスターデータをインポートする方法が2つあります。

  • 1回限りのデータインポート、および新旧のシステムへの二重エントリ。
  • 古いシステムから新しいシステムへのデータの継続的な増分更新。

最初の方法を使用する場合もありますが、時間がかかりすぎ、ヒューマンファクターの影響を受けやすく、エラーが発生する可能性があります。したがって、常に2番目のオプションのみを使用します。

理想的には、古いシステムは変更を蓄積し、どうにかしてそれを再インポートする必要があることを伝えることができます。しかし、通常、古いシステムでは誰も何も変更または修正しません。したがって、使用する作業スキームは次のとおりです。

  1. 古いシステムのディレクトリにリクエストがあり、そこからすべてのデータを読み取ります。
  2. 読み取られたデータは、新しいシステムの電流と比較され、変更が検出されます。
  3. 変更は、新しいシステムへの単一トランザクションで書き込まれます。

製品ディレクトリをインポートするためのプラットフォーム上のコードは次のようになります。
importItems ' ' {
   NEWSESSION {
       LOCAL file = FILE ();
       //     
       READ 'jdbc:sqlserver://localhost;databaseName=items;User=import;Password=password@SELECT id, name FROM items' TO file;
  
       LOCAL id = STRING[20] (INTEGER);
       LOCAL name = ISTRING[100] (INTEGER);
       //    
       IMPORT TABLE FROM file() TO id, name;
      
       //  
       FOR id(INTEGER i) AND NOT item(id(i)) NEW b = Item DO {
           id(b) <- id(i);
       }
      
       //  
       FOR id(Item b) = id(INTEGER i) DO {
           name(b) <- name(i);
       }
      
       //  ,    
       DELETE Item b WHERE b IS Item AND NOT [ GROUP SUM 1 BY id(INTEGER i)](id(b));
      
       APPLY;
   }
}

実際、起動時のこのアクションは、古いディレクトリと新しいディレクトリを完全に同期します。増分情報は保存されないため、前回の交換がエラーで完了した場合でも、いつでも再起動できます。

すべてのアクションは反復なしでSQLクエリにコンパイルされるので、これはすべてかなり迅速かつ安全に行われます。通常、10万から20万件のレコードの商品のディレクトリの交換には数分かかりました。同時に、PostgreSQLはバージョン管理されているため、現時点ではユーザーの作業のブロックは発生しません。
同様に、サプライヤーの価格、品揃えマトリックス、注文スケジュール、および作業に必要なその他の情報は常に同期されています。ただし、ここでは、古いシステムのドメインロジックが新しいドメインロジックと一致しないという問題が発生する可能性があります。たとえば、システムには、マトリックスバージョンの履歴と、マトリックス内の製品レベルの概念があります。古いシステムで現在の品揃えがストアと商品のブール値としてフラットに保存されている場合、1つのマトリックスが各ストアに対して正確に1つのレベルで作成されます。

ほとんどの場合、ユーザーに手動で交換を開始する機会を提供しながら、すべてのディレクトリを古いシステムと1時間に1回同期するアクションを有効にします。

処理する


小売では、通常、翻訳プロセスを店舗間で垂直に分割し、同時に店舗プロセスからオフィスプロセスに水平方向に分割します。

最初のステップは、新しいシステムへの転送を開始するストアを1つ選択することです。古いシステムから現在の残高とストア価格を、対応する数量と価格を含む受信ドキュメントの形式でインポートします。

ソースコード例
importInventory ' ' {
    NEWSESSION {
        LOCAL file = FILE ();
        //     
        READ 'jdbc:sqlserver://localhost;databaseName=items;User=import;Password=password@SELECT idSku, quantity, idDoc, dateDoc, idDetail FROM inventory' TO file;
        
        LOCAL idSku = STRING[20] (INTEGER);
        LOCAL quantity = NUMERIC[16,3] (INTEGER);
        //  idDoc, dateDoc  idDetail -          
        //        ,       
        LOCAL idDoc = STRING[20] (INTEGER);
        LOCAL dateDoc = DATE (INTEGER);
        LOCAL idDetail = STRING[20] (INTEGER);
        //    
        IMPORT TABLE FROM file() TO idSku, quantity, idDoc, dateDoc, idDetail;      
        
        //    
        FOR [GROUP SUM 1 BY idDoc(INTEGER i)](id) AND NOT receipt(id) NEW r = Receipt DO {
            id(r) <- id;
        }
        
        //    
        FOR INTEGER i = [GROUP MIN INTEGER ii BY idDoc(ii)](id) AND id(Receipt r) = id DO {
            date(r) <- dateDoc(i);
        }
        
        //  ,   
        FOR idDetail(INTEGER i) AND NOT receiptDetail(idDetail(i)) NEW d = ReceiptDetail DO {
            id(d) <- idDetail(i);
        }
        

        //    
        FOR id(ReceiptDetail d) = idDetail(INTEGER i) DO {
            receipt(d) <- receipt(idDoc(i));
            quantity(d) <- quantity(i);
        }
        
        //     ,      
        DELETE Receipt r WHERE id(r) AND NOT [ GROUP SUM 1 BY idDoc(INTEGER i)](id(r));
        DELETE ReceiptDetail d WHERE id(d) AND NOT [ GROUP SUM 1 BY idDetail(INTEGER i)](id(d));
        
        //         
        APPLY;
    }
}

開店の数週間前に、POSシステムから実装のインポートがつながります。これは、作業の最初の日の注文の形成に関する履歴データがあるために必要です。

多くの場合、独自の生産プロセスは、メインプロセスより少し遅れて転送されます。この場合、原材料の移動に関するドキュメントの古いシステムへのアンロードおよびそこからの完成品のインポートが実行されます。

転送当日は夜間にインポートを開始し、お客様のカスタマーサポートスタッフと弊社の最初のサポートラインの両方が店舗に送られます。これらは、事前トレーニングを受けることなく、ユーザーが新しいシステムを使い始めるのに役立ちます。この場合、通常はしばらくの間、古いシステムでのドキュメントの変更は、人が誤解しているため終了せず、エラーを修正する機能が必要です。数週間後、残高が再インポートされ、古いシステムの変更はすでに禁止されています。

上記のすべての問題が最初のストアで識別および解決された後、約2週間後、さらに2〜3のストアが転送されます。繰り返しになりますが、問題の特定と修正の繰り返しです。そして、他のすべては、クライアントのカスタマーサポートサービスのリソースに応じて、非常に迅速に翻訳されます。この間、マスタデータは引き続き古いシステムに入力されるため、ストアは古いシステムで機能できます。

次に、すべてのストアが転送されると、徐々にまたはすべてのストアが古いシステムからのインポートをすぐにオフにし、ユーザーが新しいシステムにマスターデータを入力し始めます。

新旧のシステムからの要約情報は、経理部門またはBIシステムで取得され、両方のシステムが同時にアップロードされます。そこで、古いシステムと新しいシステムの稼働時間を含む、間隔の要約レポートを取得できます。

結論


あるシステムから別のシステムへの移行は明らかに複雑であり、このパスを数十回歩いていましたが、デバッグされたスキームでは、すべてが実際にはそれほど怖くないことがわかります。クライアントの従業員のほとんどが非常に保守的であり、ほんのわずかな機会で、「現状のまま」で行うことを要求するという事実により、さらに多くの問題が発生します。そしてここでは、お客様が新旧のプロセスを比較し、最適なプロセスを決定し、従業員を「破壊」できる十分な権限を持つ人物がいることが非常に重要です。または、「現状のまま」のスキームに従ってプロセスを変更します（逆の場合）。
