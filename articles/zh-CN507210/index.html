现代Java在处理大量数据时的性能，第2部分


仅供参考：第一部分。

基准批次输送机


批处理管道处理最终数量的存储数据。没有处理结果流，聚合功能的输出需要应用于整个数据集。这改变了性能要求：延迟-流处理中的关键因素-在这里不存在，因为我们实时处理数据。唯一重要的指标是输送机的总运行时间。

因此，我们选择并行。在测试的第一阶段，当在一个节点上工作时，此收集器确实显示出最佳的吞吐量（但仅在调整之后）。但是，这是以长时间停顿为代价的。如果群集节点之一在垃圾回收时停止，则将停止整个管道。而且由于节点在不同的时间收集垃圾，因此总构建时间会随着将每个节点添加到群集而增加。通过比较一个节点和三个节点的群集上的测试结果，我们分析了这种影响。

此外，在此阶段，我们没有考虑实验性的低延迟收集器。它们非常短的暂停不会影响测试结果，此外，这是由于带宽而实现的。

在单个节点上进行测试：传送带


对于单个节点上的批处理基准测试，我们使用了一条简单的管道。完整代码在这里。

p.readFrom(longSource)
 .rebalance() // Introduced in Jet 4.2
 .groupingKey(n -> n % NUM_KEYS)
 .aggregate(summingLong(n -> n))
 .filter(e -> (e.getKey() & 0xFF_FFFFL) == 0)
 .writeTo(Sinks.logger())

源是一个存根，它生成一个数字序列long。定义键函数的方式是groupingKey在键空间中循环：0、1、2，... NUM_KEYS，0、1、2，...这意味着在第一个循环中，管道将看到所有键并创建用于存储结果的固定数据结构聚合。在接下来的周期中，系统仅更新可用数据。这与考虑了不同世代的垃圾收集假设完全一致：对象要么经过所有计算，要么不存在很长时间，而在创建后不久就变成垃圾。

我们的源生成了4亿个元素，并且创建了1亿个单独的键，即四次遍历所有键。

操作员.rebalance()用一阶段聚合替换Jet的默认两阶段聚合。这使我们基准测试中的引擎行为更加可预测。

我们还测试了一个选项，其中聚合操作使用类型对象的实例作为状态，Long并在每次更新当前帐户时生成垃圾。在这种情况下，许多物体在老一代中经过相当长的时间后就会死亡。在这种情况下，我们必须将密钥的数量减少到7000万，因为在1亿的情况下，收集器上的负载太高。

我们不关注低延迟收集器，因为在批处理管道的情况下它们没有任何提供。由于我们已经看到JDK 14的工作原理几乎与JDK 11类似，因此我们进行了一项测试来确认这一点。然后，我们专注于JDK 8和JDK11。我们还将JDK 8中使用的默认Parallel收集器与G1进行了比较。

在单个节点上测试：结果


我们在配备16 GB RAM和6核Intel Core i7的笔记本电脑上运行基准测试。堆大小为10 GB。

最初，由于并行，性能很差，我们不得不调整垃圾回收。因此，我们强烈建议使用VisualVM和Visual GC插件。如果设置最大帧速率（10 fps），则可以非常详细地看到应用程序内存分配与垃圾收集器工作之间的关系。观看动画后，我们意识到主要问题是为新一代分配的内存太大。默认情况下，旧代与新代的比率仅为2：1，并且在执行期间不会动态变化。因此，我们决定应用该设置-XX:NewRatio=8，这改变了整体情况。现在Parallel表现最好。我们还应用-XX:MaxTenuringThreshold=2幸存者空间来减少空间之间的数据复制，因为临时对象会在管道中快速死亡。

现在介绍结果。批处理管道的唯一合适度量是任务持续时间。为了使结果可视化，我们取了它们的倒数，因此，该图以秒为单位显示了吞吐量。对于一个节点：



尽管有大量密钥，但不进行垃圾收集的聚合工作速度大约可提高30-35％。 G1和JDK 8捆绑包的工作效果最差，而Parallel和JDK 11捆绑包的调优效果最佳； G1和JDK 11捆绑包的损失不多，请注意，我们没有接触G1配置，这很重要。垃圾收集配置高度依赖于这种情况。例如，随着数据量的增加，结果可能会发生巨大变化。并且您需要针对整个集群配置特定的负载类型。

在三个节点上进行测试：传送带


为了正确地测试集群，我们必须使用更复杂的管道：

p.readFrom(longSource())
 .rebalance()
 .flatMap(n -> {
     Long[] items = new Long[SOURCE_STEP];
     Arrays.setAll(items, i -> n + i);
     return traverseArray(items);
 })
 .rebalance()
 .groupingKey(n -> n % NUM_KEYS)
 .aggregate(AggregateOperations.summingLong(n -> n))
 .filter(e -> e.getKey() % 1_000_000 == 0)
 .writeTo(Sinks.logger())
;

由于源代码无法并行工作，因此我们进行了一些优化，以使其不会成为系统的瓶颈。源生成数字0、10、20，...，并且我们并行化了flatMap内插缺失数字的阶段。同样在源和flatMap我们之间使用rebalance()，在整个集群中分布数据。在主要阶段（密钥聚合）开始之前，我们再次进行了平衡。汇总后，我们首先仅保留每百万个键值对，然后将它们传递给记录器。使用了十亿个数据元素和一组5亿个密钥。

就像在一个节点的情况下一样，我们使用没有垃圾的聚集以及带有垃圾的传送带来测试传送带。

在三个节点上进行测试：结果


我们在三个c5d.4xlarge实例的AWS集群上运行了该基准测试。他们拥有16个虚拟处理器核心和32 GB内存。通道带宽10 Gb / s。结果：



顺便说一句，我们注意到与单个节点相比，吞吐量总体提高了大约三倍。这是分布式处理的结果。对于汇编程序，G1和JDK 11的组合成为这两个测试的获胜者，另一个引人注目的结果是G1和JDK 8的组合几乎无法操作，但这有更深层次的原因影响其他方面。例如，Parallel在JDK 8和JDK 11上的明显优势。这是由于我们在一开始就注意到的效果：一旦任何节点暂停进行垃圾收集，整个集群的处理就会停止。而且，JDK 8上的G1会暂停很长时间，超过一分钟。这足以使集群中的故障检测器正常工作并确定节点已死亡。任务失败，集群重新组织，任务在两个节点上再次启动。这会导致更快地导致新的故障，因为现在每个节点都处理更多的数据。同时，弹出的节点再次加入，并且该任务在两个节点上又在其他节点上再次开始。重新启动任务无止境。

并行暂停的时间不至于破坏群集，但是在同一节点上的测试中，并行暂停的效果要差得多。在三个节点上，他损失了30％的G1和JDK 11。在更大的集群上，情况将更加糟糕。

如果您查看所有测试，令人惊讶的是，Parallel在JDK 8上运行的速度比在JDK 11上运行的速度快。但是，这是由于非常好的巧合：在这些测试运行中，所有节点上的完全暂停同步开始，这使收集器的工作并行化。显然，不能依靠这种效果。

即使在使用Parallel的测试配置中，我们也没有观察到长时间停顿对集群稳定性的灾难性影响，但这还是很可能的情况。在其他测试中，当我们增加堆和数据量，或保留相同的堆，但减少了操作余量时，Parallel的行为同样具有破坏性。但是，即使它没有导致故障，在图中，当在同一节点上工作时，它的优势也消失了。可以预料，随着向群集中添加新节点，情况将进一步恶化。

另一方面，一堆G1和JDK 11停顿了很短的时间，因此传送带没有停止。它具有减轻短暂停机时间的机制，虽然暂停时间不超过150毫秒，但垃圾回收仅对本地产生影响。
