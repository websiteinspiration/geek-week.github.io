Reactでフォームを再利用する

こんにちは！

BCSには管理パネルと多くのフォームがありますが、Reactコミュニティでは、一般的に受け入れられている方法はありません。それらを再利用できるように設計する方法です。公式Facebookガイドには、検証と再利用が必要な実際の条件でフォームを操作する方法に関する詳細情報は含まれていません。誰かがredux-form、formik、final-formを使用したり、独自のソリューションを作成したりしています。


この記事では、Reactでフォームを操作するためのオプションの1つを示します。スタックは次のようになります。React+ formik + Typescript。以下を示します。

  • コンポーネントが行うべきこと。
  • 小道具レベルでの構成、フィールド、および検証。
  • フォームを再利用可能にする方法。
  • レンダラーの最適化。
  • 私たちの方法よりも不便です。

新しいビジネスタスクでは、15〜20の同様のフォームを作成する必要があることを学びました。仮想的にはさらに多くのフォームが存在する可能性があります。 `store`からのデータを処理するconfigに1つの恐竜フォームがあり、` sagas`を介してリクエストを保存および実行するアクションを送信しました。彼女は素晴らしく、ビジネスの価値を果たしていました。しかし、それはすでに拡張不可能で再利用不可能であり、不十分なコードと松葉杖の追加があっただけです。

タスクは、フォームを何度でも再利用できるように書き直すことです。まあ、関数型プログラミングを思い出してください。関数型プログラミングには、外部データを使用しない純粋な関数があり、この場合は「redux」で、引数（props）で送信されるものだけです。

そして、それが起こったのです。

私たちのコンポーネントの考え方は、ラッパー（コンテナー）を作成し、そこに外界と連携するロジック（Reduxストアからデータを受信して​​アクションを送信する）を書き込むことです。このため、コンテナーコンポーネントは、コールバックを通じていくつかの情報を受信できる必要があります。フォームの小道具の全リスト：

Formikの使用


<Formik />コンポーネントを使用します。

render() {
  const {
    fields, validationSchema, validateOnBlur = true, validateOnChange = true,
  } = this.props;

  return (
    <Formik
      initialValues={fields}
      render={this.renderForm}
      onSubmit={this.handleSubmitForm}
      validationSchema={validationSchema}
      validateOnBlur={validateOnBlur}
      validateOnChange={validateOnChange}
      validate={this.validateFormLevel}
    />
  );
}

`validate`フォームのプロパティで、` this.validateFormLevel`メソッドを呼び出します。このメソッドでは、コンテナーコンポーネントにすべての変更されたフィールドを取得し、それらが有効かどうかを確認する機会を与えます。

private validateFormLevel = (values: FormFields) => {
  const { onChangeFields, validationSchema } = this.props;

  if (onChangeFields) {
    validationSchema
      .validate(values)
      .then(() => {
        onChangeFields(values, { isValid: true });
       })
      .catch(() => {
         onChangeFields(values, { isValid: false });
       });
   }
}

ここで、フィールドが有効かどうかをコンテナに明確にするために、もう一度検証を呼び出す必要があります。フォームを送信するときは、prop `onSubmit`を呼び出すだけです。

private handleSubmitForm = (): void => {
  const { onSubmit } = this.props;

  if (onSubmit) {
    onSubmit();
  }
}

小道具1-5ですべてが明確になるはずです。「config」、「fields」、「validationSchema」に移りましょう。

小道具「構成」


interface IFieldsFormMetaModel {
  /**   */
  sectionName?: string;
  sectionDescription?: string;
  fieldsForm?: Array<{
    /**    */
    name?: string; //          prop ‘fields’
    /**    checked */
    checked?: boolean;
    /** enum,      */
    type?: ElementTypes;
    /**    */
    label?: string;
    /**    */
    helperText?: string;
    /**      */
    required?: boolean;
    /**      */
    disabled?: boolean;
    /**  -    */
    minLength?: number;
    /**            */
    initialValue?: IInitialValue;
    /**      */
    selectItems?: ISelectItems[]; //   select, dropdown  
  }>;
}

このインターフェイスに基づいて、オブジェクトの配列を作成し、このスキームに従って「セクション」->「セクションフィールド」をレンダリングします。したがって、タイトルとメモが必要な場合は、セクションの複数のフィールドを、またはそれぞれのフィールドに一度に表示できます。レンダリングのしくみについては、後で少し説明します。
設定の短い例：

export const config: IFieldsFormMetaModel[] = [
  {
    sectionName: ' ',
    fieldsForm: [{
      name: 'subject',
      label: '',
      type: ElementTypes.Text,
    }],
  },
  {
    sectionName: '',
    sectionDescription: '  ',
    fieldsForm: [{
      name: 'reminder',
      disabled: true,
      label: '',
      type: ElementTypes.CheckBox,
      checked: true,
    }],
  },
];

ビジネスデータに基づいて、 `name`キーの値が設定されます。同じ値がpropの `fields`キーで使用され、ギミックの元の値または変更された値を送信します。

上記の例では、 `fields`は次のようになります。

const fields: SomeBusinessApiFields = {
  subject: '  ',
  reminder: 'yes',
}

検証のために、Yupスキーマを渡す必要があります。コンテナープロップを含むフォームにフォームを与え、リクエストなどの外部データとの相互作用を記述します。

フォームは、スキームに何らかの影響を与えることはできません。たとえば、

export const CreateClientSchema: (

レンダリングとフィールドの最適化


レンダリングでは、キーですばやく検索できるマップを作成しました。簡潔に見え、検索は「スイッチ」よりも高速です。

fieldsMap: Record<
  ElementTypes,
  (
    state: FormikFieldState,
    handlers: FormikHandlersState,
    field: IFieldsFormInfo,
  ) => JSX.Element
  > = {
    [ElementTypes.Text]: (
      state: FormikFieldState,
      handlers: FormikHandlersState,
      field: IFieldsFormInfo
    ) => {
      const { values, errors, touched } = state;

      return (
        <FormTextField
          key={field.name}
          element={field}
          handleChange={this.handleChangeField(handlers.setFieldValue, field.name)}
          handleBlur={handlers.handleBlur}
          value={values[field.name]}
          error={touched[field.name] && errors[field.name] || ''}
        />
      );
    },
    [ElementTypes.TextSearch]: (...) => {...},
    [ElementTypes.TextArea]: (...) => {...},
    [ElementTypes.Date]: (...) => {...},
    [ElementTypes.CheckBox]: (...) => {...},
    [ElementTypes.RadioButton]: (...) => {...},
    [ElementTypes.Select]: (...) => {...},
  };

各コンポーネントフィールドはステートフルです。別のファイルにあり、 `React.memo`でラップされています。不要なレンダラーを回避するために、すべての値は「子」をバイパスして小道具を介して送信されます。

結論


私たちのフォームは理想的ではありません。いずれの場合も、データを操作するためのコンテナーラッパーを作成する必要があります。それらを `store`に保存し、変換してリクエストを行います。あなたが取り除きたいコードの繰り返しがあります。小道具に応じて、フォームがフィールド、アクション、ダイアグラム、構成を含むストアから必要なキーを取得する新しいソリューションを見つけようとしています。次の投稿の1つで、何が起こったのかを説明します。
