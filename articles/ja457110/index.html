トレイトを使用したC ++およびモックインジェクションパターンの単体テスト

また会いましたね！コース"C ++ Developer"のグループでクラスを開始する前に、残り1週間未満です。この点に関して、私たちはこのコースの学生のために特別に翻訳された有用な資料を共有し続けます。



テンプレートを使用したコードの単体テストは、時々それ自体を思い出させます。（テンプレートをテストしていますよね？）テストしやすいテンプレートもあります。一部ではありません。テストされたテンプレートでのモックコード（スタブ）の実装について、最終的な明快さが欠けている場合があります。埋め込みコードが複雑になるいくつかの理由を観察しました。

以下に、コード実装の複雑さがほぼ増加する例をいくつか示しました。

  1. テンプレートは、型引数とコンストラクター内の参照によって同じ型のオブジェクトを受け取ります。
  2. テンプレートは型引数を取ります。コンストラクター引数のコピーを作成するか、単に受け入れません。
  3. テンプレートは型引数を取り、仮想関数なしで相互接続されたいくつかのテンプレートを作成します。

簡単なものから始めましょう。

テンプレートは、型引数とコンストラクター内の参照によって同じ型のオブジェクトを取ります


ユニットテストは、スタブ型のテストテンプレートのインスタンスを作成するだけなので、このケースは単純に見えます。一部のステートメントは、モッククラスをチェックできます。これですべてです。

当然、型引数を1つだけ使用してテストしても、テンプレートに渡すことができる残りの型の数については何もわかりません。同じことを上品に表現する方法：パターンは一般性の量指定子によって接続されているため、より科学的なテストを行うには、もう少し洞察を深める必要があるかもしれません。これについては後で詳しく説明します。

例えば：

template <class T>
class TemplateUnderTest {
  T *t_;
public:
  TemplateUnderTest(T *t) : t_(t) {}

  void SomeMethod() {
    t->DoSomething();
    t->DoSomeOtherThing();
  }
};

struct MockT {
  void DoSomething() { 
    // Some assertions here.
	  }
  void DoSomeOtherThing() { 
    // Some more assertions here.
  }
};

class UnitTest {
  void Test1() {
    MockT mock;
    TemplateUnderTest<MockT> test(&mock);
    test.SomeMethod();
    assert(DoSomethingWasCalled(mock));
    assert(DoSomeOtherThingWasCalled(mock));
  }
};


テンプレートは型引数を取ります。コンストラクター引数のコピーを作成するか、単に受け入れません


この場合、テンプレート内のオブジェクトにアクセス権がないため、アクセスできない場合があります。friend-classes を使用できます。

template <class T>
class TemplateUnderTest {
  T t_;
  friend class UnitTest;
public:
  void SomeMethod() {
    t.DoSomething();
    t.DoSomeOtherThing();
  }
};
class UnitTest {
  void Test2() {
    TemplateUnderTest<MockT> test;
    test.SomeMethod();
    assert(DoSomethingWasCalled(test.t_)); // access guts
    assert(DoSomeOtherThingWasCalled(test.t_)); // access guts
  }
};

UnitTest :: Test2 TemplateUnderTestの本文にアクセスでき、MockTの内部コピーのステートメントをチェックできます。

テンプレートは型引数を取り、仮想関数なしで相互接続されたいくつかのテンプレートを作成します


このケースでは、実際の例である非同期Google RPCを検討します。

C ++では、非同期gRPCにCallDataと呼ばれるものがあります。これは、名前が示すように、RPC呼び出しに関連するデータを格納します。CallDataテンプレートは、いくつかの異なるタイプのRPCを処理できます。したがって、テンプレートによって正確に実装されるのは当然です。

ジェネリックCallDataは、リクエストとレスポンスの2つのタイプの引数を受け入れます。次のようになります。

template <class Request, class Response>
class CallData {
  grpc::ServerCompletionQueue *cq_;
  grpc::ServerContext context_;
  grpc::ServerAsyncResponseWriter<Response> responder_;
  // ... some more state
public:
  using RequestType = Request;
  using ResponseType = Response;

  CallData(grpc::ServerCompletionQueue *q)
    : cq_(q),
      responder_(&context_) 
  {}
  void HandleRequest(Request *req); // application-specific code
  Response *GetResponse(); // application-specific code
};

CallDataテンプレートの単体テストでは、HandleRequestおよびHandleResponseの動作を確認する必要があります。これらの関数は、いくつかのメンバー関数を呼び出します。したがって、コールの正常性を確認することは、CallDataの正常性にとって最も重要です。ただし、トリックがあります。

  1. grpc名前空間の一部のタイプは内部で作成され、コンストラクターを介して渡されません。ServerAsyncResponseWriterそしてServerContext例えば。
  2. grpc :: ServerCompletionQueue引数としてコンストラクタに渡されますが、仮想関数はありません。仮想デストラクタのみ。
  3. grpc :: ServerContext 内部的に作成され、仮想機能はありません。

問題は、テストで完全なgRPCを使用せずにCallDataをテストする方法です。ServerCompletionQueueをシミュレートする方法は？それ自体がテンプレートであるServerAsyncResponseWriterをシミュレートする方法は？など...

仮想関数がないと、ユーザーの行動を置き換えることは複雑な作業になります。grpc :: ServerAsyncResponseWriterなどのハードコードされた型は、ハードコードされ、実装されていないため、モデル化できません。

それらをコンストラクタ引数として渡す意味はほとんどありません。これを行ったとしても、それらは最終クラスであるか、単に仮想関数を持たない可能性があるため、意味がない場合があります。

どうしようか？

解決策：特性




（オブジェクト指向プログラミングで行われるように）ジェネリック型から継承してカスタム動作を埋め込む代わりに、型を挿入します。これには特性を使用します。私たちは、プロダクションコードまたはユニットテストコードなど、コードの種類に応じて、さまざまな方法で特性を専門としています。

検討するCallDataTraits

template <class CallData>
class CallDataTraits {
  using ServerCompletionQueue = grpc::ServerCompletionQueue;
  using ServerContext = grpc::ServerContext;
  using ServerAsyncResponseWriter = grpc::ServerAsyncResponseWrite<typename CallData::ResponseType>;
};

これは、量産コードに使用される特性のメインテンプレートです。CallDataテンプレートで使用してみましょう。

/// Unit testable CallData
template <class Request, class Response>
class CallData { 
  typename CallDataTraits<CallData>::ServerCompletionQueue *cq_;
  typename CallDataTraits<CallData>::ServerContext context_;
  typename CallDataTraits<CallData>::ServerAsyncResponseWriter responder_;
  // ... some more state
public:
  using RequestType = Request;
  using ResponseType = Response;

  CallData(typename CallDataTraits::ServerCompletionQueue *q)
    : cq_(q),
      responder_(&context_) 
  {}
  void HandleRequest(Request *req); // application-specific code
  Response *GetResponse(); // application-specific code
};

上記のコードを見ると、アプリケーションコードがまだgrpc名前空間の型を使用していることは明らかです。ただし、grpc型をダミー型に簡単に置き換えることができます。下記参照。

/// In unit test code
struct TestRequest{};
struct TestResponse{};
struct MockServerCompletionQueue{};
struct MockServerContext{};
struct MockServerAsyncResponseWriter{};

/// We want to unit test this type.
using CallDataUnderTest = CallData<TestRequest, TestResponse>;

/// A specialization of CallDataTraits for unit testing purposes only.
template <>
class CallDataTraits<CallDataUnderTest> {
  using ServerCompletionQueue = MockServerCompletionQueue;
  using ServerContext = MockServerContext;
  using ServerAsyncResponseWriter = MockServerAsyncResponseWrite;
};

MockServerCompletionQueue mock_queue;
CallDataUnderTest cdut(&mock_queue); // Now injected with mock types.

特性により、状況に応じてCallDataに実装されているタイプを選択できました。機能を追加するために不要な仮想関数が作成されていないため、このメソッドは追加のパフォーマンスを必要としません。この手法は、最終クラスでも使用できます。
