時間測定とプロファイリング


コードの最適化を開始する前に、まずプログラム全体の速度を低下させるコードの部分を見つける必要があります。プログラムのボトルネックは明白な場合がありますが、プログラマーがどこにあるかわからない場合は、いくつかの機会を利用してそれを特定できます。

以下は、デモ用に使用するプログラムのコードです。これは、Pythonのドキュメントから引用しています。このコードは次eの能力を高めますx。

# slow_program.py
from decimal import *

def exp(x):
    getcontext().prec += 2
    i, lasts, s, fact, num = 0, 0, 1, 1, 1
    while s != lasts:
        lasts = s
        i += 1
        fact *= i
        num *= x
        s += num / fact
    getcontext().prec -= 2
    return +s

特定の機能の一時的なパフォーマンス指標の研究


これで、注意を向ける必要があるプログラムの場所がわかりました。したがって、他のプログラムコードをプロファイリングせずに、遅い関数を研究することにする場合があります。これを行うには、単純なデコレーターを使用できます。

def timeit_wrapper(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()  #       time.process_time()
        func_return_val = func(*args, **kwargs)
        end = time.perf_counter()
        print('{0:<10}.{1:<8} : {2:<8}'.format(func.__module__, func.__name__, end - start))
        return func_return_val
    return wrapper

このデコレータは、調査する関数に適用できます。

@timeit_wrapper
def exp(x):
    ...


print('{0:<10} {1:<8} {2:^8}'.format('module', 'function', 'time'))

ここで、測定する予定の正確な時間に注意を払う価値があります。対応するパッケージは、time.perf_counterおよびなどのインジケーターを提供しますtime.process_time。それらの違いはperf_counter、Pythonプログラムプロセスが実行されていない時間を含む絶対値を返すことです。つまり、このインジケーターは、他のプログラムによって作成されたコンピューターの負荷の影響を受ける可能性があります。メトリックprocess_timeはユーザー時間のみを返します。システム時間は含まれません。これは、プロセスの実行時間に関する情報のみを提供します。

コード加速


そして今、楽しい部分です。プログラムの加速に取り組みましょう。（ほとんどの場合）パフォーマンスの問題を魔法のように解決するあらゆる種類のハック、トリック、および不可解なコードをここに示すことはしません。基本的に、使用するとパフォーマンスに大きな影響を与える可能性のある一般的なアイデアと戦略についてお話ししたいと思います。場合によっては、コードの実行速度が30％向上することについて話しています。

▍組み込みのデータ型を使用する


組み込みデータ型を使用することは、コードを高速化するための非常に明白なアプローチです。ビルトインデータタイプは、特にツリーやリンクリストなどのカスタムタイプと比較すると、非常に高速です。ここでのポイントは主に、言語の組み込みメカニズムがCを使用して実装されていることです。Pythonを使用して何かを記述する場合、同じレベルのパフォーマンスを達成することはできません。

llru_cacheを使用してキャッシュ（メモ化）を適用する


キャッシュは、コードのパフォーマンスを向上させるための一般的なアプローチです。私はすでに彼について書いていますが、ここで彼について話す価値があると思います。

import functools
import time

#   12  
@functools.lru_cache(maxsize=12)
def slow_func(x):
    time.sleep(2)  #   
    return x

slow_func(1)  # ...  2    
slow_func(1)  #    -   !

slow_func(3)  # ...   2

上記の関数は、を使用して複雑な計算をシミュレートしtime.sleepます。パラメータを使用して初めて呼び出された場合1、2秒待機し、その後のみ結果を返します。彼女が同じパラメーターで再び呼び出されたとき、彼女の作業の結果はすでにキャッシュされていることがわかります。この状況での関数の本体は実行されず、結果はすぐに返されます。ここでは、より現実に近いキャッシングの例を見つけることができます。

プログラムの速度を低下させるもう1つのメカニズムは、ドット（.）演算子です。これは、オブジェクトの属性にアクセスするために使用されます。この演算子は、プロシージャを使用して辞書検索を呼び出します__getattribute__。これにより、システムに追加の負荷がかかります。このPython機能のパフォーマンスへの影響を制限するには？

strings弦に注意


文字列操作をループで実行すると、プログラムの速度が大幅に低下する可能性があります。特に、とを使用%sし.format()た文字列のフォーマットについて説明しています。それらを何かで置き換えることは可能ですか？Raymond Hettinger による最近のツイートを見ると、このような状況で使用する必要があるメカニズムはf-lineだけであることがわかります。これは、最も読みやすく、簡潔で、最速の文字列フォーマット方法です。ここでは、そのつぶやきに従って、文字列を操作するために使用できるメソッドのリストを示します-最も速いものから最も遅いものまで：

f'{s} {t}'  # !
s + '  ' + t
' '.join((s, t))
'%s %s' % (s, t)
'{} {}'.format(s, t)
Template('$s $t').substitute(s=s, t=t)  # !

generatorジェネレーターも速く動作できることを知ってください


ジェネレーターは、本質的に高速なメカニズムではありません。実際、これらは「遅延」計算を実行するために作成されており、時間を節約するのではなく、メモリを節約します。ただし、メモリを節約すると、プログラムの実行が速くなる場合があります。これはどのようにして可能ですか？実際には、ジェネレーター（イテレーター）を使用せずに大きなデータセットを処理すると、データがプロセッサーのL1キャッシュのオーバーフローを引き起こし、メモリー内の値を見つけるプロセスを大幅に遅くする可能性があります。

パフォーマンスに関しては、プロセッサーが処理するデータに迅速にアクセスできるように努め、プロセッサーが可能な限りそれに近づくようにすることが非常に重要です。これは、そのようなデータをプロセッサキャッシュに配置する必要があることを意味します。この問題はこれで対処されています レイモンドヘッティンガーによるパフォーマンス。
