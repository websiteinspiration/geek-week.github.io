CTO Meduza Boris GoryachevによるFlutter Devポッドキャスト：Meduzアプリケーションとメディア開発の主なもの

アプリケーションに10万を超えるDAUが含まれるFlutterプロジェクトについて話し合う機会があるとは限りません。新しいMeduzaアプリはまさにそのようなプロジェクトです。現在、これはロシアのストアで最も人気があり、議論されているFlutterアプリケーションです。 Boris GoryachevをFlutter Dev Podcastに招待して、チームがネイティブをクロスプラットフォームに任せ、Flutterを選択した理由を調べました。以下に、最も興味深いものを集めました。ネイティブWebと友達になる試み、クロスプラットフォームの最初の体験、メディア開発の機能、フォントゲーム、およびWebView /バックエンドドリブンUIの操作の難しさ。



「アイデア、サイト、2つのアプリケーション。」当初、2014年のMeduzaの発売時、チームは便利なモバイルアプリケーションに依存していたため、サイトはできるだけシンプルでした。後で判明したように、その瞬間の観客はまだウェブ上でニュースを放棄する準備ができていませんでした。このアプリケーションはユーザーを獲得しましたが、絶対数ではWeb（モバイルサイトとデスクトップの両方）より劣っていました。したがって、サイトの技術設備を改善することが決定されました。

「オーディエンスメドゥーザ。」内部の指標では、Meduzaは、サイトのモバイルユーザーとデスクトップユーザー、およびアプリケーションのユーザーを明確に区別します。後者は最も少ないですが、それらの活動は他のすべてのカテゴリよりも大幅に高くなっています。これがメドゥーサの聴衆の中核です。

「アプリケーションの最初のバージョンは何でしたか？」。メドゥーザはほんの数ヶ月で始まりました。賭けはブラウザからダウンロードされた美しいコンテンツにありました。メディアアプリケーションの作成の原則は誰にとっても同じです。ネイティブで、メインページとその他の配布ページが作成されます。人が資料をタップすると、WebViewが開き、さまざまなHTML、CSS、その他のモジュールが表示されます。

「開発アプローチは時間とともに変化しましたか？」。いいえ、メカニズムは同じままです。最初にネイティブビュー、次にユーザーがWebViewで「失敗」し、ブラウザのコンテンツがスクロールされます。このアプローチには欠点があります。コンテンツがブラウザーによって制御される場合、ネイティブコードとWebフォーム間の通信に多くの注意を払う必要があります。たとえば、美しい効果で表示され、ユーザーをフォローするある種のネイティブコンポーネントを描画します。このチームでは多くのコーンを満たしました。

「ネイティブとWebをやり取りする際の問題は何でしたか」。 Web開発者とネイティブ開発者は異なる言語を話します。開発中に、いくつかの間違ったオブジェクトが来た、間違った構造などが原因で、シリアライゼーション、デシリアライゼーション、Javaのクラッシュに問題がありました。スクロール位置によっては、要素の埋め込みにも問題がありました。たとえば、高さが制限されたWebViewがあり、ネイティブの一部と統合する必要があります。それ自体がスクロールする要素からコンポジションを作成する必要があります。その後、別のスクロールをインターセプトします。

「Meduzaのコンテンツをネイティブにレンダリングすることは、ブラウザが機能するようにブラウザを繰り返すようなものです。簡単な作業ではありません。チームはいくつかのマテリアルをネイティブレンダリングに転送できました。たとえば、「オンライン」カテゴリでは、資料は編集者によってリアルタイムで入力されます。送信後、情報はすぐに読者の画面に表示されます。すべてネイティブでやった。たとえば、クライアントは別の形式で動的データを受信します。クライアントはそれらを正しく処理し、落ちないようにする必要があります。すべてのニュアンスの調整に多くの問題がありました-先に進む方法を決定する必要がありました。

「最初のクロスプラットフォームエクスペリエンス-React Native。」 Meduzaのフロントエンド全体はReactで作成されています。ボリスは数週間で彼の最初のモバイルプロトタイプをReact Nativeで書きました。プロトタイプは正常に機能しました。表示、レンダリング、強制されたスタイルなど。 React Nativeでのアプリケーションの完全な開発が始まると、新しい問題が発生しました。たとえば、AndroidのJavaScriptはiOSのように機能しません。有効なJavaScriptがありますが、それは誓うものではありませんが、開いてからvoidが表示され、問題はJSXにあります。コンテナまたはテキストにマップするタグにも問題がありましたが、どういうわけかトリッキーな方法で閉じます。当初は、事後的にすべてのバグを修正する機会があったようです。しかし、遠くに行くほど、ささいなことでより多くの開発が行われました。

「彼らは2018年にGoogle I / OでFlutterについて学びましたが、延期することにしました。」Andrew Brogdan（Flutter）と話しました。Meduzaは、UIで何が起きるかをAPIが指示するためのバックエンド駆動型アーキテクチャを必要としていました。重要な点は、ウィジェットツリーに統合するWebViewがあることです。メディアアプリケーションは、ソーシャルからのビデオをユーザーに表示する必要があります。ネットワークとYouTube。残念ながら、当時、FlutterはWebViewをサポートしておらず、クロスプラットフォームは保留されていました。ネイティブアプリケーションは、ほぼ2年間サポートなしで使用されてきました。この間、バックエンド側でさまざまな編集が行われましたが、アプリケーションは多かれ少なかれ安定して動作しました。
こんにちは、私はテクニカルディレクターです。「人々が使用する2つのアプリケーションはありますか？まあ、彼らはサポートなしになるでしょう。」そして結局、彼らはほぼ2年間サポートなしで生活しましたが、それらを使用しました。

「2018年12月、GoogleはWebViewライブラリをFlutterに追加し、新しいMeduzaアプリケーションの開発が始まりました。」チームが数回アプリケーション全体を新しく発明したので、それだけでほぼ1年かかりました。画面、コンセプト変更、実験実施。 Flutterを使用すると、インターフェイスをすばやく作成できます。また、ホットリロードのおかげで、設計者は変更にすばやくコメントして編集し、多くの編集を行うことができます。合計で、アプリケーションコードとバックエンドの所要時間は6か月以内でした。同時に、ボリスはFlutterに完全に集中していませんでしたが、他のプロジェクトの開発と立ち上げに気を取られていました。
メディアは、すべてが常に失敗する人間の活動の一種です。世界では常に何かが起こっています。ジャーナリストは当局に拘束されており、政権は変わりつつあり、COVID-19の大流行は始まっています。このため、サービスの開発が妨げられ、会議が延期され、創造性が損なわれます。

「ロシアでのメディアアプリケーション開発の特徴。」私たちの国では、メディア開発者はほとんどいません。ワークショップには、ベストプラクティスから学び、借りることができる同僚はほとんどいません。この分野のプログラマーは別の考え方をしています。通常の専門家がプレーンテキストを見ると、メディアプログラマーは、タイポグラフィのない段落、長いダッシュではなく短いマイナス、引用符が「クリスマスツリー」ではなく「足」などに気づきます。このような詳細により、深刻な出版物がインターネット上のランダムなサイトと区別されます。
メディアの人々は常に「議題に疲れている」。おそらく、プラスまたはマイナスのメデューザのすべての人々がこれに苦しんでいます。毎日あなたは議題に没頭していますが、それは通常あまり楽しいものではありません。そして、これらの条件下では、さまざまな技術的な問題が発生します。その後、Googleはインデックスの一部を変更し、FacebookまたはVKontakte APIは拒否し、変更ログにそれを書き込みました。

「Flutterでフォントを微調整する機能は、Meduzaにとって非常に役立ちました。」出版物は長い間同じフォントのペア（Proxima NovaとPF Regal）を使用してきましたが、設計者はしばしばそれらに小さな変更を加えるため、自宅でのサポートは非​​常に困難でした。開発者はデフォルトのコンポーネントに手を加えたくなかったし、多くのデバイスでの変更を見たこともなかった。

「ライト・メディア・アプリケーションへのチームまたは1個のフラッタ開発者の十分に拡張する計画はあります。」このフレームワークを使用すると、プロジェクトをソロで作成できます。 Meduzaの場合、「ブロックAPI」が使用されます。バックエンドは、何を表示するかをUIに指示します。結論としては、Meduzaのマテリアルはすべて、入力されたブロックの配列であるということです。コンストラクターとして、そこから段落を組み立てることができますが、内部にはメディアプレーヤーなどがあります。プラスまたはマイナスの任意のブロックは、任意の構成で表示できます。アプリケーション全体がこれに基づいて構築されています。ある時点で、アプリケーションの任意の時点で何かを取得して置き換えることができます。他のテキスト、インデント、色などをバックエンド経由で送信します。

「アプリケーション開発計画：新しい形式が本番環境に現れる前の最適化とサポート」。近い将来、古いデバイス向けのパフォーマンスとポッドキャストの最適化。将来的には、バックエンドのみによる新しいフォーマットの立ち上げをサポートします。つまり、Flutter側で変更を加えることなく、変更や機能を実行できます。

この場合、フォーマットとは出版物のタイプを意味します：ニュース、カード、読み物など。それらのすべては、編集者によって積極的に使用されています。ネイティブ以降の主な問題は、アプリケーションでの新しいフォーマットのサポートでした。以前のWebviewが使用され、エディターがクールなフォーマットを思いついたときは常にありました。それはWebではサポートされていましたが、アプリケーションではサポートされていませんでした。私はユーザーが更新されるまで待たなければならず、その後、新しい資料がスマートフォンでユーザーに表示され始めました。これを回避するために、アプリケーションは製品計画よりも先に進み、まだ運用段階にない機能をサポートする必要があります。したがって、人々は更新する時間があり、すべてが同時にプラスまたはマイナスに実現されます。

「Flutterでは、新機能のリリースプロセスを可能な限り簡単に行うことができました」。メドゥーザでのプロセスは次のとおりです。バックエンドプログラマーがAPIを記述し、電話/シミュレーターでアプリケーションを開きます。次に、すべてが一致することを確認し、アプリケーションを作成します。新しいウィジェットを作成してカスタマイズする必要はありません。そのようなスキームには、パフォーマンスとトレードオフ、およびコードの可読性とのトレードオフがあります。 1つのコンポーネントまたはブロックのロジックが十分に深刻になると、それらは単にウィジェットからウィジェットを作成し、ブロックは新しいエンティティになります。たとえば、ポッドキャストプレーヤーはそのように実装されました。

「アプリケーションのどの部分がFlutterでレンダリングされ、どの部分がWebViewを介してレンダリングされるか」。ほとんどの要素はFlutterでレンダリングされます：コンテナー、テキスト、タイポグラフィ。例外は埋め込み要素（Twitter、YouTube、Facebook、TikTok）とWebViewを介して完全に作成されたゲームです。たとえば、Meduzaのテストや車のカスタムゲームなどです。それらはすべてゼロから行われます。 WebViewの高さがViewPortデバイスよりも高い場合、一部の電話のAndroidのWebViewがクラッシュすることを知っておくことが重要です。将来的には、WebViewから離れて「Flutterネイティブ」のものを優先し、JSONからのデータを使用して同じTwitterをレンダリングします。

「ポッドキャストの計画は何ですか、どのように改善する予定ですか」。最初はaudioplayersプラグインが使用されていましたが、その後、一連のjust_audioおよびaudio_serviceに切り替えることが決定されました。プレーヤーはプロセスに問題はなく、2番目のライブラリはポッドキャストに必要な通知をサポートしていました。たとえば、再生するものをオンにすると、停止をタップしたり、オーディオをシークしたりできるモジュールが上部に表示されます。最も難しいことは、UIを別のスレッドで発生するものと同期させること、そしてその瞬間をカーテンとスワイプで実現することでした。

「アーキテクチャの観点からのアプリケーション、またはプロバイダーに立ち寄った理由」。このような構造の操作を維持するために必要なアクションは少なくなります。グローバルな状態を変更する無限の定数や関数は不要です。検討されたオプションの中で、プロバイダーのロジックはMeduzaに最も近づきました。アプリケーションには8つあり、たとえば、1つはユーザーのブックマークを担当し、もう1つは閲覧履歴とポッドキャストを同期するためのものです。その助けを借りて、彼らはリスナーが停止した位置を覚えています。したがって、彼は別のデバイスに戻って、そこでポッドキャストを聞くことができます。別のプロバイダーがトピックに責任を負い、必要な色が必要な場合は代用するか、UIで必要な場合にそれらを提供します。

「何のために、アプリケーションはAndroidとiOSでUIの異なるパフォーマンスとレンダリングを示します」。最初の問題は、プロバイダーを介して実装された最初のデータフェッチでしたが、Computeには関与していませんでした。かなりの数の操作があります。APIでデータを取得し、かなり大きなJSONを取得して処理し、データに変換して、結果をモデルにプッシュします。そのため、アプリケーションの開始時に、FPSのドローダウンが観察されます。

もう1つの問題は、iOSでのFPSドローダウンが発生する広告の統合です。 Androidでは、広告はありません。アプリケーションのライブラリを選択するとき、公式のAdMobに立ち寄りました。しかし、結局のところ、そこでバナーをウィジェットツリーに統合することはできません。開発者は、ウィジェットツリーやフルスクリーン広告の上にバナーを使用することを望みません。これは、読者の忠誠心に大きな影響を与えるからです。

MeduzaチームはGoogleを利用してベータテストを行い、iOSで目的の広告フォーマットがすでにテストされています。現在のバージョンでは、トランジションがアニメーション化されると、AdMob広告が読み込まれます。ベンチマークは、パフォーマンスの低下があることを示していますが、どこにでもあるわけではありません。Flutterチームからの修正を待っています。道徳は単純です。MainIsolateから何かを削除する必要がある場合は、Computeを使用します。

その結果、すべての困難にもかかわらず、FlutterはMeduzaチームの期待に完全に応えました。ここでポッドキャストの完全版を探して、コメントで質問してください！
