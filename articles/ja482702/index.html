2020年には本当にTypeScriptが必要なのでしょうか？



TypeScriptは、現代のWeb開発者にとって不可欠なスキルの1つになりました。2019年、彼はGitHubで最も使用頻度の高い上位10言語を入力し、彼のサポートはCreate反応アプリに完全に追加されました。同時に、JavaやCなどの言語の地位は下がり続けています。

TypeScriptの利点について話すとき、通常、次のリストが思い浮かびます：

  • TypeScriptは静的型付けをサポートしています
  • TypeScriptを使用すると、コードが読みやすくなり、理解しやすくなります。
  • TypeScriptは、コードでの型チェックのおかげで、開発者が通常犯す多くの苦痛なバグを回避するのに役立ちます
  • TypeScriptは開発者にOOPのベストプラクティスに従うことを奨励します
  • 上記の結果として-TypeScriptは開発者の時間を節約します

興味深いことに、このリストのすべてのアイテムは批判的な目をせずに信仰に基づいて取られています。今日、私はあなたがこれらのポイントをより綿密に検討し、それらが私たちにとって本当にそれほど有用であるかどうかを知ることを提案します。

TypeScriptの主な目的はバグの数を減らすことなので、なぜこれがそれほど重要なのかを判断しましょう。答えは簡単です。バグが少ない-開発者やテスターに​​費やす時間が少ないため-価値のある製品を少ない費用で手に入れ、早期に収入を得ることができます。

それを念頭に置いて、TypeScriptが生産性と効率の向上にどのように役立つかを見てみましょう。

静的型付け-バグの魔法のタブレット？


TypeScriptの主な機能は、静的型付けのサポートです。開発者の間では、動的型付けがJavaScript開発者が抱えるほとんどすべての問題の原因であると広く信じられています。

悪いタイプの人がダイナミックタイピングで何を見つけるのかしら？このような批判の弾幕が、PythonやRubyなどの他の動的言語に当てはまらないのはなぜですか。この問題は、動的型付けではJavaScriptでの型キャストの場合ほどではないことを前提としています。実際、次のようなコードを驚かせるのは非常に不快な場合があります。


しかし、これはJavaScriptに関する知識が乏しく、言語自体ではなく問題です。あなたが強力なスポーツカーを持っていると想像してください。しかし、あなたの運転スキルは十分に平凡です。自動車メーカーに変更を加えて車の最高速度を下げたり、高度な運転を学び、優れた専門家になることを要求しますか？ TypeScriptが提供するのは、JavaScriptを適切に学習するのではなく、動的型付けの可能性を制限することです。

動的型付けの反対者に対するもう1つの質問は、静的型付けが非常に優れているのに、なぜJavaおよびCで記述されたコードでまだバグが発生するのかということです。はい、コンパイルの段階でこれらのエラーをキャッチできますが、正直なところ、これでは不十分です。高品質のコードを保証するには、SOLIDの原則とその他のベストプラクティスに従う必要があります。繰り返しになりますが、これはプログラミング言語よりも、プログラマーの知識と資格に関係しています。

TypeScriptコードは読みやすいですか？


Reduxサンクの2つの例を次に示します。

const getUsers = async dispatch => {
  //...
  try {
    const users = await APIService.get('/users')
    dispatch(successGetUsers(users))
  } catch(err) {
    dispatch(failedGetUsers(err))
  }
}

TypeScriptでも同じです：

const getUsers = (): ThunkAction<void, {}, {}, AnyAction> => 
  async (dispatch: ThunkDispatch<{}, {}, AnyAction>) => {
    //...
    try {
      const users: User[] = await APIService.get('/users')
      dispatch(successGetUsers(users))
    } catch(err) {
      dispatch(failedGetUsers(err))
    }
  }

これらのジェネリックすべてはどういう意味ですか？中に2つの空のオブジェクトがあるのはなぜですか？ドキュメントを読んですべてを理解するには、どれくらいの時間を費やす必要がありますか？まあ、それはRedux Thunkにとってまだ普通のことです。Reduxにとって非常に人気のあるミドルウェアであるため、優れたサポートチームとドキュメントを持っています。しかし、これをすべて備えていないコードを保守する必要がある場合はどうでしょうか。

微妙なバグはなくなりますか？


前の例では、TypeScriptがコードをより冗長にする方法を説明しました。ご存じのように、システムが大きく複雑になるほど、エラーの可能性が高くなります。JavaScriptコードで発生する可能性のある間違いに加えて、インターフェイスのタイプミスやジェネリックスの間違いなどをしないように注意する必要があります。公平に言えば、コンパイラがそのようなエラーをすばやく見つけて修正するのに役立つことは注目に値しますが、TypeScriptがなければ、私たちはそれらにまったく遭遇しなかっただろう。

OOPのベストプラクティスに従う


ご存知のように、高品質で簡単にスケーラブルで保守可能なコードを記述したい場合は、ベストプラクティスに従う必要があります。TypeScriptはこれを支援してくれます。いいですね、Expressの例を見てみましょう：

userRoute.js

router.get('/users', (req, res) => {
  //get users from DB
  res.json(users)
})
router.post('/users', (req, res) => {
  //create user 
  res.json(userCreated)
})

userRoute.ts

class UserRouter {
  public router = express.Router() 
  public address = '/users'
  constructor() {
    this.initRoutes()
  }
  initRoutes() {
    this.router.get(this.address, this.getUsers)
    this.router.post(this.addressm this.createUser)
  }
  getUsers(req: express.Request, res: express.Response) {
    //get users from DB
    res.json(users)
  }
  createUser(req: express.Request, res: express.Response) {
    //create user
    res.json(userCreated)
  }
}

あなたの目を引く最初のことは、より多くのコードが再び書かれているということです。クラスを使用してOOPスタイルで記述します。次に、これらのクラスをインスタンス化する必要があります。さらに、インターフェース、静的タイプ、およびその他の構成を使用できます。そして、これが完全な混乱にならないように、ベストプラクティスを使用せざるを得ません。そして、これがいわゆる「開発者にOOPのベストプラクティスを使用するように促す」です。

開発者がJavaScriptやPythonなどのシンプルで柔軟な動的言語に投票したとき、関数型プログラミングパラダイムがその利点といくつかの問題をよりエレガントかつ効率的に解決する能力を示したとき（そしてJavaScriptでは両方のアプローチを使用できます）、TypeScriptはOOP JavaとC＃で昔ながらのスタイルでコードを書く。

要約すると、TypeScriptは時間を節約するものではなく、多数のバグからユーザーを保護するものでも、生産性を向上させるものでもありません。さらに、より多くのコード、追加の構成、型定義ファイルを記述し、追加のドキュメントを読むのに時間を費やす必要があります。初心者はおそらく理想的ではないかもしれませんが、JavaScriptまたはRubyで動作するアプリケーションを作成しますが、すべての原則に従って優れたTypeScriptアプリケーションを作成します。そして、彼は私たちを雇って彼のスタートアップを書き直させます。TypeScriptを使用してすべてを正しく行うことができます。

どのように大規模で冗長で複雑な言語が消えて、より動的で単純な言語との競争に耐えられないのかを見るのはおかしいです。
