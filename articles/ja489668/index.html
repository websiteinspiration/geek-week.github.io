KubernetesのCPU制限とアグレッシブスロットリング

注意ペレフ。：ヨーロッパの旅行アグリゲーターであるオミオのこの警告的な物語は、読者を基本的な理論から、Kubernetes構成の魅惑的な実用的な複雑さへと導きます。このようなケースに精通することは、視野を広げるだけでなく、重要な問題を防ぐのにも役立ちます。



アプリケーションが所定の位置に「スタック」し、ヘルスチェックリクエストへの応答を停止し、この動作の理由を理解できなかったという事実に遭遇したことがありますか？考えられる理由の1つは、CPUリソースの割り当て制限です。この記事では彼について説明します。

TL、DR：
CFS割り当てエラーのあるLinuxカーネルバージョンを使用している場合は、KubernetesのCPU制限を無効にする（またはKubeletでCFS割り当てを無効にする）ことを強くお勧めします。そこのコアで過度のスロットルと遅延につながる深刻でよく知られたバグ。

Omioでは、インフラストラクチャ全体がKubernetesによって管理されています。ステートフルおよびステートレスのロードはすべて、Kubernetesでのみ機能します（Google Kubernetes Engineを使用しています）。過去6か月間に、ランダムなスローダウンが観察され始めました。アプリケーションがヘルスチェックにフリーズしたり応答を停止したり、ネットワークへの接続を失ったりした。そのような振る舞いは長い間私たちを困惑させ、そして最後に、私たちは問題を綿密に取り組むことにしました。

コンテナーとKubernetesに関する一言


実際、Kubernetesはインフラストラクチャの世界における最新の標準です。その主なタスクは、コンテナーのオーケストレーションです。

コンテナ


以前は、サーバーでのその後の起動のために、Java JAR / WAR、Python Egg、または実行可能ファイルなどのアーティファクトを作成する必要がありました。ただし、それらを機能させるためには、ランタイム（Java / Python）のインストール、必要なファイルの適切な場所への配置、オペレーティングシステムの特定のバージョンとの互換性の確保など、追加の作業を行う必要がありました。つまり、構成管理（開発者とシステム管理者の間で競合が発生することが多い）に細心の注意を払う必要がありました。

コンテナはすべてを変えました。これで、コンテナーイメージはアーティファクトとして機能します。これは、プログラムだけでなく、本格的なランタイム（Java / Python / ...）だけでなく、プレインストールされて実行の準備ができている必要なファイル/パッケージも含む、一種の拡張実行可能ファイルとして表すことができます。コンテナは、追加の手順なしでさまざまなサーバーにデプロイして実行できます。

さらに、コンテナは独自のサンドボックス環境で動作します。それらには、独自の仮想ネットワークアダプター、アクセスが制限された独自のファイルシステム、独自のプロセス階層、CPUおよびメモリに対する独自の制限などがあります。これはすべて、Linuxカーネルの特別なサブシステムである名前空間（名前空間）によって実現されます。

Kubernetes


前述のとおり、Kubernetesはコンテナオーケストラです。次のように機能します。マシンのプールを提供し、「Hey Kubernetes、2つのプロセッサと3 GBのメモリを搭載したコンテナのインスタンスを10個起動し、それらを操作可能な状態に維持します！」残りはKubernetesが処理します。彼は空き容量を見つけ、コンテナを起動し、必要に応じて再起動し、バージョンの変更時に更新をロールアウトします。実際、Kubernetesを使用すると、ハードウェアコンポーネントを抽象化して、さまざまなシステム全体をアプリケーションのデプロイと運用に適したものにすることができます。

Kubernetesのリクエストと制限とは


さて、コンテナーとKubernetesを見つけました。また、複数のコンテナを同じマシン上に置くこともできます。

あなたは共同アパートとの類似性を描くことができます。広々とした部屋（車/ノード）が取得され、いくつかのテナント（コンテナ）にリースされます。 Kubernetesは不動産業者として機能します。どうやってテナント同士が対立しないようにするのかという疑問が生じます。たとえば、そのうちの1人がバスルームを半日占有すると決めたらどうなるでしょうか。

これは、要求と制限が関係する場所です。 CPU リクエストは、計画のみを目的としています。これはコンテナの「ウィッシュリスト」のようなもので、最も適切なノードを選択するために使用されます。同時に、CPU 制限はリースと比較できます。コンテナのノードを取得するとすぐに、設定された制限を超えることはできません。そして、ここで問題が発生します...

Kubernetesでのリクエストと制限の実装方法


Kubernetesは、カーネルスロットリング（スキップクロック）メカニズムを使用してCPU制限を実装します。アプリケーションが制限を超えると、スロットルが有効になります（つまり、CPUサイクルが少なくなります）。メモリのリクエストと制限は異なる方法で構成されているため、検出が容易です。これを行うには、最後のポッドの再起動ステータス、つまり「OOMKilled」かどうかを確認するだけで十分です。CPUのスロットリングを使用すると、K8は使用可能なメトリックのみを作成し、cgroupは使用できないため、すべてがそれほど単純ではありません。

CPUリクエスト



CPUリクエストの実装方法

簡単にするために、4コアCPUを搭載したマシンの例を使用してプロセスを見てみましょう。

K8sはcgroupsメカニズムを使用して、リソース（メモリとプロセッサ）の割り当てを制御します。彼は階層モデルを使用できます。子孫は親グループの制限を継承します。配布の詳細は、仮想ファイルシステムに保存されます（/sys/fs/cgroup）。プロセッサの場合、これ/sys/fs/cgroup/cpu,cpuacct/*。

K8sは、ファイルcpu.shareを使用してプロセッサリソースを割り当てます。私たちの場合、ルートコントロールグループは、4096のCPUリソースシェアを受け取ります-利用可能なプロセッサ能力の100％（1コア= 1024;これは固定値です）。ルートグループは、で規定された子孫のシェアに応じてリソースを比例的に分配しますcpu.share、そしてそれらは、その子孫などにも同じことをします。通常、Kubernetesルートノードコントロールグループにはsystem.slice、user.sliceとの3つの子がありkubepodsます。最初の2つのサブグループは、重要なシステム負荷とK8の外部のユーザープログラムとの間でリソースを分散するために使用されます。最後の- kubepodsは、ポッド間でリソースを分散するためにKubernetesによって作成されます。

上の図は、最初と2番目のサブグループが1024のシェアを受け取り、4096のシェアがkuberpodサブグループに割り当てられていることを示しています。どうしてこれが可能か：結局のところ、ルートグループには4096のシェアしかなく、その子孫のシェアの合計はこの数（6144）？実際のところ、この値は論理的に意味があるため、Linuxスケジューラー（CFS）はこれを使用して、CPUリソースを比例的に割り当てます。私たちの場合、最初の2つのグループは680の実際のシェア（4096の16.6％）を受け取り、kubepodは残りの2736のシェアを受け取ります。ダウンタイムの場合、最初の2つのグループは割り当てられたリソースを使用しません。

幸い、スケジューラには未使用のCPUリソースの損失を回避するメカニズムがあります。 「アイドル」容量をグローバルプールに転送し、そこから追加のプロセッサ容量が必要なグループ間で分散されます（丸め損失を回避するために転送はバッチで行われます）。同様の方法が子孫のすべての子孫に適用されます。

このメカニズムにより、プロセッサー能力の公平な配分が保証され、プロセスが他のリソースを「盗む」ことがなくなります。

CPU制限


K8の制限と要求の構成は似ているという事実にもかかわらず、それらの実装は根本的に異なります。これは最も誤解を招きやすく、最も文書化されていない部分です。

K8sはCFSクォータメカニズムを使用して制限を実装します。それらの設定は、ファイルcfs_period_usとcfs_quota_uscgroupディレクトリで指定されます（ファイルもそこにありますcpu.share）。

対照的cpu.shareに、クォータは一定の期間に基づいており、利用可能なプロセッサ能力ではありません。cfs_period_us期間（時代）の持続時間を設定します-常に100,000μs（100 ms）です。 K8sにはこの値を変更する機能がありますが、現在はアルファ版でのみ使用できます。スケジューラーは時代を使用して、使用された割り当て量を再開します。 2番目のファイルcfs_quota_us、各時代の利用可能時間（クォータ）を設定します。マイクロ秒単位でも表示されることに注意してください。割り当ては時代の期間を超える場合があります。つまり、100ミリ秒を超えることがあります。

16コアマシン（Omioで最も一般的なタイプのコンピューター）の2つのシナリオを見てみましょう：


シナリオ1：2つのスレッドと200ミリ秒の制限。


シナリオ2を使用しない場合：10フローおよび200ミリ秒の制限。スロットリングは20ミリ秒後に開始され、プロセッサリソースへのアクセスはさらに80ミリ秒後に再開され

ます。CPU制限を2コアに設定したとします。 Kubernetesはこの値を200ミリ秒に変換します。これは、コンテナがスロットルなしで最大200ミリ秒のCPU時間を使用できることを意味します。

そして、ここから楽しみが始まります。上記のように、使用可能な割り当ては200ミリ秒です。12コアマシンで10個のスレッドが並行して実行されている場合（シナリオ2の図を参照）、他のすべてのポッドがアイドル状態のとき、割り当ては20ミリ秒（10 * 20ミリ秒= 200ミリ秒）で使い果たされ、すべてのスレッドこのポッドの80ミリ秒はスロットルです。すでに述べたスケジューラのバグにより状況が悪化します。これにより、過剰なスロットルが発生し、コンテナが既存の割り当てを処理することさえできません。

ポッドのスロットリングを評価するにはどうすればよいですか？


ポッドに移動して実行してくださいcat /sys/fs/cgroup/cpu/cpu.stat。

  • nr_periods -スケジューラの期間の総数。
  • nr_throttled-コンポジションのスロットル期間の数nr_periods。
  • throttled_time -ナノ秒単位の累積スロットル時間。

その結果、すべてのアプリケーションで高いスロットルが得られます。時にはそれは計算されたものより1.5倍強いです！

これにより、さまざまなエラーが発生します-準備チェックの失敗、コンテナーのハング、ネットワーク接続の切断、サービス呼び出し内のタイムアウト。最終的に、これは遅延の増加とエラーの増加につながります。

記事の冒頭で述べたように：

あなたは共同アパートとの類似を描くことができます... Kubernetesは不動産業者として機能します。しかし、どうすればテナント同士が対立しないようにできますか？たとえば、そのうちの1人がバスルームを半日占有すると決めたらどうなるでしょうか。

それが問題です。1つの過失コンテナが、マシンで利用可能なすべてのプロセッサリソースを吸収する可能性があります。インテリジェントなアプリケーションスタック（たとえば、JVM、Go、ノードVMが適切に構成されている）を使用している場合、これは問題ではありません。そのような状況で長時間作業できます。しかし、アプリケーションの最適化が不十分であるか、まったく最適化されていない場合（FROM java:latest）、状況は手に負えなくなる可能性があります。私たちOmioは、主要言語のスタックに適切なデフォルト設定を備えた自動化された基本的なDockerfileを持っているので、そのような問題はありませんでした。USE
