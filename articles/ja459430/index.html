自動フォーム生成を備えたモバイルアプリケーション：私たちのケース

私たちの開発者が気に入っているように、モバイルアプリケーションは必ずしも単純で簡潔であるとは限りません。他のアプリケーションは、複雑なユーザーの問題を解決するために作成され、多くの画面とスクリプトが含まれています。たとえば、テスト、アンケート、および調査を実施するためのアプリケーション-プロセスの多くのフォームに記入する必要がある場合はいつでも。このアプリケーションについては、この記事で説明します。



保険契約の現場登録に携わるエージェント向けのモバイルアプリケーションの開発を開始しました。アプリケーションの大きなフォームに顧客データ（車、所有者、ドライバーなどに関する情報）を入力します。各フォームには独自のセクション、セル、構造があり、各アンケート項目には固有のデータ型（文字列、日付、添付文書）が必要ですが、画面フォームは非常に似ていました。しかし、重要なのはその数です...同じ要素の視覚化と処理の繰り返しに何度も携わる人はいません。

フォームの作成に何時間もかかる手動作業を回避するには、少し工夫を凝らし、UIの動的な構築をたくさん行う必要があります。この記事では、この問題の解決方法を紹介します。

この問題をエレガントに解決するために、オブジェクトを生成するメカニズムであるViewModelを使用しました。これは、テーブルを使用してカスタムフォームを構築するために使用されます。



通常の作業では、開発者が画面に表示したい個々のテーブルごとに、個別のViewModelクラスを作成する必要があります。テーブルの視覚的コンポーネントを定義します。Enumフィールドによる構造の簡単な説明を使用して、1つ上のレベルに移動し、ViewModelとモデルを動的に生成することにしました。

使い方


すべては列挙型で始まりました。プロファイルごとに一意の列挙型を作成します-これらはプロファイルのセクションです。そのメソッドの1つは、このセクションのセルの配列を返すことです。

テーブルのセルには、セルのプロパティを説明する追加の関数が列挙されます。このような関数では、セルの名前、初期値を設定します。後で追加されたパラメータ

  • 表示チェック：一部のセルは非表示にする必要があります。
  • 「親」セルのリスト：このセルの値、検証、または表示が依存するセル、
  • セルタイプ：値を持つ単純なセル、スイッチ内のセル、要素を追加する機能を持つセルなど

特定のセクションへのバインディングを除外するために、すべてのセクションを一般的なQuestionnaireSectionCellTypeプロトコルにサブスクライブします。テーブル（QuestionnaireCellType）のすべてのセルに対して同じことを行います。

protocol QuestionnaireSectionCellType {
    var title: String { get }
    var sectionCellTypes: [QuestionnaireCellType] { get }
}

protocol QuestionnaireCellType {
    var title: String { get }
    var initialValue: Any? { get }
    var isHidden: Bool { get }
    var parentFields: [QuestionnaireCellType] { get }
	…
}

このようなモデルは非常に簡単に記入できます。すべてのセクションを実行し、各セクションでセルの配列を実行して、それらをモデル配列に追加します。

被保険者の画面の例（セクション付き列挙-InsurantSectionType）：

final class InsurantModel: BaseModel<QuestionnaireCellType> {
 
   override init() {
        super.init()
        initParameters()
   }
	
   private func initParameters() {
        InsurantSectionType.allCases.forEach { type in
            type.sectionCellTypes.forEach {
                if let valueModel = ValueModel(type: $0, 
                                               parentFields: $0.parentFields, 
                                               value: $0.initialValue) {
                    valueModels.append(valueModel)
                }
            }
        }
    }
}

できた！これで、初期値を持つテーブルができました。QuestionnaireCellTypeキーで値を読み取り、目的の配列要素に保存するメソッドを追加します。

一部のモデルにはオプションのフィールドがある場合があるため、オプションのキーを持つ配列を追加しました。モデルの検証中、これらのキーには値が含まれない場合がありますが、モデルは入力済みと見なされます。

さらに、便宜上、ValueModelのすべての値を共通のプロトコルStringRepresentableプロトコルにサブスクライブして、可能な値のリストを制限し、セルに値を表示するメソッドを追加します。

protocol StringRepresentable {
   var stringValue: String? { get }
}

機能性が高まり、他の多くのプロパティとメソッドがモデルに表示されました：モデルのクリーニング（一部のモデルでは初期値を設定する必要があります）、値の動的配列のサポート（値：配列）など。

このアプローチは、レルムを使用してデータベースに格納するのに非常に便利であることがわかりました。アンケートに記入するには、以前に保存した完成モデルを選択することができます。 CTPポリシーを拡張するために、エージェントはユーザーのドキュメント、それらに添付されているドライバー、新しいドキュメントのTCPデータに記入する必要がなくなります。代わりに、単に再利用して既存のものを埋めることができます。

テーブルを変更または補足するには、特定の画面に関連するViewModelを見つけ、目的のブロックを表示する必要がある列挙型を見つけ、いくつかのケースを追加または修正するだけです。すべて、テーブルは必要な形を取ります！

フォームにテスト値を入力することも非常に便利で迅速でした。これにより、テストデータをすばやく生成できます。また、プログラムがアンケートの特定の各フィールドに値を入力する場所から、初期データを含む個別のファイルを追加すると、初心者でも、特定のファイルを除いて、コードの残りの部分に移動して分解することなく、既成のアンケートを生成できます。

依存関係


開発プロセス中に解決した別のタスクは、依存関係の処理です。アンケートのいくつかの要素は相互に関連していた。そのため、このドキュメント自体のタイプを選択せず​​にドキュメント番号を入力することはできません。また、市区町村などを指定せずに家番号を指定することはできません。



すべての依存フィールドをクリアして、アンケートの値を更新しました（たとえば、ドキュメントタイプの削除または変更、「ドキュメント番号」フィールドをクリアします）：

func updateValueModel(value: StringRepresentable?, for type: QuestionnaireCellType) {
    guard let model = valueModels.first(where: { $0.type.equal(to: type) }) else {
       	return
    }
    
    model.value = value
    clearRelativeValues(type: type)
}

func clearRelativeValues(type: QuestionnaireCellType) {
    _ = valueModels.filter { $0.parentFields.contains(where: { $0.equal(to: type) }) } 
				  .compactMap { $0.type }
				  .compactMap { updateValueModel(value: nil, for: $0) }
}
