PHPとPHPUnitでの純粋なテスト


PHPエコシステムには、便利なPHPテストを提供する多くのツールがあります。最も有名なものの1つはPHPUnitです。これは、この言語でテストすることのほぼ同義語です。ただし、優れたテスト方法についてはあまり書かれていません。テストを作成する理由とタイミング、テストの種類など、さまざまなオプションがあります。ただし、正直に言うと、後で読むことができない場合にテストを作成しても意味がありません。

テストは特別な種類のドキュメントです。以前にPHPのTDDについて書いたように、テストは常に（または少なくともそうすべき）特定のコードのタスクが何であるかについて明確に話します。

1つのテストでこのアイデアを表現できない場合、テストは不良です。

私は、PHP開発者が読みやすく有用なテストを作成するのに役立つ一連の手法を用意しました。

基本から始めましょう


多くの人が疑問なく従う一連の標準的なテクニックがあります。それらの多くに触れ、なぜそれらが必要なのかを説明しようと思います。

1.テストに入出力操作を含めないでください


主な理由：I / O操作は遅く、信頼性が低くなります。

遅い：世界で最高のハードウェアを使用している場合でも、I / Oはメモリアクセスよりも遅くなります。テストは常に高速に動作する必要があります。そうしないと、実行される頻度が非常に低くなります。

信頼性が低い：一部のファイル、バイナリ、ソケット、フォルダ、およびDNSレコードは、テストしている一部のマシンで使用できない場合があります。I / Oのテストに依存するほど、テストがインフラストラクチャに関連付けられます。

I / Oに関連する操作：

  • ファイルの読み取りと書き込み。
  • ネットワーク呼び出し。
  • 外部プロセスの課題（助けを借りてexec、proc_openなど）。

入出力操作の存在により、テストをより速く記述できる状況があります。ただし、注意が必要です。このような操作が、開発、アセンブリ、およびデプロイメントのマシンで同じように機能することを確認してください。そうしないと、深刻な問題が発生する可能性があります。

I / O操作を必要としないようにテストを分離します。以下では、インターフェイス間で責任を共有することにより、テストがI / Oを実行しないようにするアーキテクチャソリューションを示しました。

例：

public function getPeople(): array
{
  $rawPeople = file_get_contents(
    'people.json'
  ) ?? '[]';

  return json_decode(
    $rawPeople,
    true
  );
}

この方法を使用してテストを開始すると、ローカルファイルが作成され、時々スナップショットが作成されます。

public function testGetPeopleReturnsPeopleList(): void
{
  $people = $this->peopleService
    ->getPeople();

  // assert it contains people
}

これを行うには、テストを実行するための前提条件を構成する必要があります。一見、すべてが合理的に見えますが、実際にはひどいです。

前提条件が満たされていないためにテストをスキップしても、ソフトウェアの品質は保証されません。これはバグを隠すだけです！

状況を修正します。インターフェイスに責任を移すことにより、I / O操作を分離します。

// extract the fetching
// logic to a specialized
// interface
interface PeopleProvider
{
  public function getPeople(): array;
}

// create a concrete implementation
class JsonFilePeopleProvider
  implements PeopleProvider
{
  private const PEOPLE_JSON =
    'people.json';

  public function getPeople(): array
  {
    $rawPeople = file_get_contents(
      self::PEOPLE_JSON
    ) ?? '[]';

    return json_decode(
      $rawPeople,
      true
    );
  }
}

class PeopleService
{
  // inject via __construct()
  private PeopleProvider $peopleProvider;

  public function getPeople(): array
  {
    return $this->peopleProvider
      ->getPeople();
  }
}

今、私JsonFilePeopleProviderはどんな場合でもI / Oを使用することを知っています。

代わりに、スタブを簡単に作成できるFlysystemファイルシステムのfile_get_contents()ような抽象化レイヤーを使用できます。 では、なぜそれが必要なのでしょうか。良い質問。そのためには、テストが必要です。アーキテクチャに挑戦して、不要なコードを削除します。

2.テストは意識的で意味のあるものでなければなりません。


主な理由：テストはドキュメントの形式です。それらは明確、簡潔、そして読みやすいものにしてください。

明快さと簡潔さ：混乱なし、スタブの千行なし、ステートメントのシーケンスなし。

読みやすさ：テストはストーリーを伝える必要があります。「与えられたとき、そのとき」の構造は、このために優れています。

優れた読みやすいテストの特徴：

  • 必要なメソッド呼び出しのみが含まれますassert（1つが望ましい）。
  • 彼は与えられた条件下で何が起こるべきかを非常に明確に説明しています。
  • メソッド実行の1つのブランチのみをテストします。
  • 彼はどんな声明のために宇宙全体のための書きかけをしません。

実装に条件式、遷移演算子、またはループが含まれている場合は、それらすべてをテストで明示的にカバーする必要があることに注意することが重要です。たとえば、初期の回答には常にテストが含まれるようにします。

繰り返しますが、ポイントはカバレッジではなく、ドキュメントにあります。

混乱するテストの例を次に示します。

public function testCanFly(): void
{
  $noWings = new Person(0);
  $this->assertEquals(
    false,
    $noWings->canFly()
  );

  $singleWing = new Person(1);
  $this->assertTrue(
    !$singleWing->canFly()
  );

  $twoWings = new Person(2);
  $this->assertTrue(
    $twoWings->canFly()
  );
}

「いつ与えられるか」という形式を採用して、何が起こるかを見てみましょう。

public function testCanFly(): void
{
  // Given
  $person = $this->givenAPersonHasNoWings();

  // Then
  $this->assertEquals(
    false,
    $person->canFly()
  );

  // Further cases...
}

private function givenAPersonHasNoWings(): Person
{
  return new Person(0);
}

「指定」セクションと同様に、「いつ」および「その後」はプライベートメソッドに転送できます。これにより、テストが読みやすくなります。

今、assertEquals無意味な混乱の中に。これを読んでいる人は、それが何を意味するのかを理解するために、ステートメントをたどる必要があります。

特定のステートメントを使用すると、テストが読みやすくなります。assertTrue()のような式ではなく、ブール変数を受け取る必要がありますcanFly() !== true。

前の例では、assertEquals間falseを$person->canFly()単純なものに置き換えていassertFalseます。

// ...
$person = $this->givenAPersonHasNoWings();

$this->assertFalse(
  $person->canFly()
);

// Further cases...

今、すべてが非常に明確です！翼がない人は飛べないはずです！詩として読んでください。

今度は、テキストに2回表示される「その他のケース」セクションは、テストがあまりにも多くのステートメントを作成していることを明確に示しています。この場合、メソッドはtestCanFly()まったく役に立ちません。

テストをもう一度改善しましょう：

public function testCanFlyIsFalsyWhenPersonHasNoWings(): void
{
  $person = $this->givenAPersonHasNoWings();
  $this->assertFalse(
    $person->canFly()
  );
}

public function testCanFlyIsTruthyWhenPersonHasTwoWings(): void
{
  $person = $this->givenAPersonHasTwoWings();
  $this->assertTrue(
    $person->canFly()
  );
}

// ...

実際のシナリオに合わせて、たとえばにテストメソッドの名前を変更することもできますが、testPersonCantFlyWithoutWingsとにかくすべてが私に適しています。

3.テストは他のテストに依存してはならない


主な理由：テストは任意の順序で実行され、正常に実行されるはずです。

テスト間に相互接続を作成する十分な理由がありません。最近、ログイン機能のテストを依頼されましたが、ここでは良い例として紹介します。

テストは：

  • ログイン用のJWTトークンを生成します。
  • ログイン機能を実行します。
  • ステータスの変更を承認します。

それはこのようなものでした：

public function testGenerateJWTToken(): void
{
  // ... $token
  $this->token = $token;
}

// @depends  testGenerateJWTToken
public function testExecuteAnAmazingFeature(): void
{
  // Execute using $this->token
}

// @depends  testExecuteAnAmazingFeature
public function testStateIsBlah(): void
{
  // Poll for state changes on
  // Logged-in interface
}

これはいくつかの理由で悪いです：

  • PHPUnitはこの実行順序を保証できません。
  • テストは独立して実行できる必要があります。
  • 並列テストはランダムに失敗する場合があります。

これを回避する最も簡単な方法は、指定されたwhen、thenスキームを使用することです。したがって、テストはより思慮深くなり、ストーリーを伝え、依存関係を明確に示し、テストされる機能を説明します。

public function testAmazingFeatureChangesState(): void
{
  // Given
  $token = $this->givenImAuthenticated();

  // When
  $this->whenIExecuteMyAmazingFeature(
    $token
  );
  $newState = $this->pollStateFromInterface(
    $token
  );

  // Then
  $this->assertEquals(
    'my-state',
    $newState
  );
}

また、認証などのテストを追加する必要があります。この構造は非常に優れているため、Behatがデフォルトで使用されます。

4.常に依存関係を実装する


主な理由：非常に悪いトーン-グローバル状態のスタブを作成するため。依存関係のスタブを作成できないため、関数をテストできません。

役立つヒント：静的なステートフルクラスとシングルトンインスタンスについては忘れてください。クラスが何かに依存している場合は、実装できるようにしてください。

ここに悲しい例があります：

class FeatureToggle
{
  public function isActive(
    Id $feature
  ): bool {
    $cookieName = $feature->getCookieName();

    // Early return if cookie
    // override is present
    if (Cookies::exists(
      $cookieName
    )) {
      return Cookies::get(
        $cookieName
      );
    }

    // Evaluate feature toggle...
  }
}

この初期の答えをどのようにテストできますか？

そのとおり。ありえない。

それをテストするには、クラスの動作を理解し、Cookiesそれに関連するすべての環境を再現できることを確認する必要があります。その結果、特定の答えが得られます。

こんなことしないで。

インスタンスCookiesを依存関係として実装すると、状況を修正できます。テストは次のようになります。

// Test class...
private Cookies $cookieMock;

private FeatureToggle $service;

// Preparing our service and dependencies
public function setUp(): void
{
  $this->cookieMock = $this->prophesize(
    Cookies::class
  );

  $this->service = new FeatureToggle(
    $this->cookieMock->reveal()
  );
}

public function testIsActiveIsOverriddenByCookies(): void
{
  // Given
  $feature = $this->givenFeatureXExists();

  // When
  $this->whenCookieOverridesFeatureWithTrue(
    $feature
  );

  // Then
  $this->assertTrue(
    $this->service->isActive($feature)
  );
  // additionally we can assert
  // no other methods were called
}

private function givenFeatureXExists(): Id
{
  // ...
  return $feature;
}

private function whenCookieOverridesFeatureWithTrue(
  Id $feature
): void {
  $cookieName = $feature->getCookieName();
  $this->cookieMock->exists($cookieName)
    ->shouldBeCalledOnce()
    ->willReturn(true);

  $this->cookieMock->get($cookieName)
    ->shouldBeCalledOnce()
    ->willReturn(true);
}

同じことがシングルトーンにも当てはまります。したがって、オブジェクトを一意にする場合は、（アンチ）シングルトンパターンを使用するのではなく、依存関係インジェクターを正しく構成します。それ以外の場合は、reset()またはのような場合にのみ役立つメソッドを記述しますsetInstance()。私の意見では、これはクレイジーです。

テストを容易にするためにアーキテクチャを変更することは完全に正常です！そして、テストを容易にするメソッドを作成することは正常ではありません。

5.保護された/プライベートメソッドをテストしない


主な理由：これらは、動作のシグネチャを定義することにより、関数をテストする方法に影響します。そのような条件下で、Aを入力すると、Bを取得することを期待します。プライベート/保護されたメソッドは、関数シグネチャの一部ではありません。

私はプライベートメソッドを「テスト」する方法を示したくありませんが、ヒントを示します。これは、リフレクション APIを使用してのみ実行できます。

リフレクションを使用してプライベートメソッドをテストすることを考えるときは、常に何らかの形で自分を罰します。悪い、悪い開発者！

定義により、プライベートメソッドは内部でのみ呼び出されます。つまり、それらは公開されていません。つまり、同じクラスのパブリックメソッドだけがそのようなメソッドを呼び出すことができます。

すべてのパブリックメソッドをテストした場合は、すべてのプライベート/保護メソッドもテストしました。そうでない場合は、プライベート/保護されたメソッドを自由に削除してください。

高度なヒント


あなたがまだ退屈していないことを願っています。それでも、基本について話す必要がありました。次に、開発プロセスに影響を与えるクリーンなテストと決定を記述することについての私の意見を共有します。

テストを書くときに忘れない最も重要なこと：

  • 調査。
  • 迅速なフィードバック。
  • ドキュメンテーション
  • リファクタリング
  • テスト中の設計。

1.終わりではなく最初にテストする


値：調査、迅速なフィードバック、文書化、リファクタリング、テスト中の設計。

これがすべての基礎です。リストされたすべての値を含む最も重要な側面。事前にテストを作成する場合、これは、最初に所定の方法を理解するのに役立ち、次に、スキームを構成する必要があります。その際、最初に文書化し、さらに重要なことに、要件を最も重要な側面として覚えて設定します。

実装の前にテストを書くことについて奇妙なことですか？そして、何かを実装し、テストするときにそれを見つけることがいかに奇妙であるかを想像してみてください。

また、このアプローチでは、2秒ごとに期待値が確認されます。あなたはできるだけ早くフィードバックを得ます。機能がどの程度大きいか小さいかに関係なく。

グリーンテストは、リファクタリングにとって理想的な領域です。主なアイデア：テストなし-リファクタリングなし。テストなしのリファクタリングは危険です。

最後に、「いつ、次に」与えられる構造を設定すると、メソッドに必要なインターフェースとその動作方法が明らかになります。テストをクリーンに保つと、さまざまなアーキテクチャ上の決定を常に行う必要があります。これにより、ファクトリー、インターフェースの作成、継承の中断などが必要になります。そうです。テストが簡単になります。

テストがアプリケーションの動作を説明するライブドキュメントである場合、それを明確にすることが不可欠です。

2.悪いテストよりもテストなしの方が良い


値：調査、ドキュメント、リファクタリング。

多くの開発者がテストをこのように考えています。私は機能を記述します。テストが特定の数の新しい行をカバーするまでテストフレームワークを駆動し、それらを運用に送ります。

新しい開発者がこの機能を使い始めるときは、状況にもっと注意を払う必要があるように思えます。テストはこの人に何を伝えますか？

名前が十分に詳細でない場合、テストはしばしば混乱を招きます。より明確なもの：testCanFly またはtestCanFlyReturnsFalseWhenPersonHasNoWings？

テストがフレームワークをより多くの行にカバーする厄介なコードであり、意味のない例が含まれている場合は、これらのテストを作成するかどうかを検討します。

変数名$aと$bの割り当て、または特定の用途に関連付けられていない名前の割り当てなどのナンセンスです。

覚えておいてください：あなたのテストはあなたのアプリケーションがどう振る舞うべきかを説明しようとするライブドキュメントです assertFalse($a->canFly())あまり文書化していません。そしてassertFalse($personWithNoWings->canFly())-すでにかなり。

3.侵入テストを実行する


値：調査、迅速なフィードバック、リファクタリング。

機能に取り掛かる前に、テストを実行します。ビジネスに取り掛かる前にエラーが発生した場合は、コードを記述する前にそのことがわかるので、気にならなかった壊れたテストのデバッグに貴重な時間を費やす必要がなくなります。

ファイルを保存した後、テストを実行します。何かが壊れていることを発見するのが早ければ早いほど、より早く修正して次に進むことができます。問題を解決するためのワークフローの中断が非生産的であると思われる場合は、問題について知らない場合、後で多くの手順に戻る必要があると想像してください。

同僚と5分間チャットしたり、Githubからの通知を確認した後、テストを実行します。彼らが赤面した場合は、どこで中断したかがわかります。テストが緑色の場合、作業を続行できます。
変数名を含めて、リファクタリングの後で、テストを実行します。

真剣に、いまいましいテストを実行してください。保存ボタンを押すたびに。
PHPUnit Watcherはこれを行うことができ、通知を送信することさえできます！

4.大きなテスト-大きな責任


値：テスト中の調査、リファクタリング、設計。

理想的には、各クラスは1つのテストを持つ必要があります。このテストは、このクラスのすべてのパブリックメソッド、および各条件式または遷移演算子をカバーする必要

があります... 次のようなものを読むことができます。

  • 1つのクラス= 1つのテストケース。
  • 1つの方法= 1つ以上のテスト。
  • 1つの代替ブランチ（if / switch / try-catch / exception）= 1つのテスト。

したがって、この単純なコードでは、4つのテストが必要になります。

// class Person
public function eatSlice(Pizza $pizza): void
{
  // test exception
  if ([] === $pizza->slices()) {
    throw new LogicException('...');
  }

  // test exception
  if (true === $this->isFull()) {
    throw new LogicException('...');
  }

  // test default path (slices = 1)
  $slices = 1;
  // test alternative path (slices = 2)
  if (true === $this->isVeryHungry()) {
    $slices = 2;
  }

  $pizza->removeSlices($slices);
}

パブリックメソッドが多いほど、必要なテストも多くなります。

長いドキュメントを読むのは好きではありません。テストもドキュメントであるため、サイズと意味が小さくても、品質と有用性は向上します。

また、クラスが責任を積み重ねていることも重要なシグナルであり、他のクラスに多数の関数を転送したり、システムを再設計したりして、クラスをリファクタリングするときがきました。

5.回帰問題を解決するための一連のテストをサポートする


値：調査、ドキュメント、迅速なフィードバック。

関数を考えてみましょう：

function findById(string $id): object
{
  return fromDb((int) $id);
}

誰かが「10」を送信していると思いますが、実際には「10バナナ」が送信されています。つまり、2つの値がありますが、1つは不要です。バグがあります。

最初に何をしますか？このような動作に誤りがあることを示すテストを作成してください!!!

public function testFindByIdAcceptsOnlyNumericIds(): void
{
  $this->expectException(InvalidArgumentException::class);
  $this->expectExceptionMessage(
    'Only numeric IDs are allowed.'
  );

  findById("10 bananas");
}

もちろん、テストは何も送信しません。しかし、あなたは彼らが送信するために何をする必要があるかを知っています。エラーを修正し、テストを緑にして、アプリケーションをデプロイして満足してください。

このテストを携帯してください。可能な限り、回帰問題を解決するために設計された一連のテスト。

それで全部です！迅速なフィードバック、バグ修正、ドキュメント、回帰耐性のあるコードと幸福。
