3CXテクニカルサポートの応答：PBXサーバーでのSIPトラフィックのキャプチャ

この記事では、3CX PBXによって生成されたSIPトラフィックのキャプチャと分析の基本について説明します。この記事は、初心者のシステム管理者またはテレフォニーの保守を担当する一般ユーザーを対象としています。トピックの詳細な調査については、3CX上級トレーニングコースを完了することをお勧めします。

3CX V16では、サーバーのWebインターフェースを介してSIPトラフィックを直接キャプチャし、標準のWireshark PCAP形式で保存できます。テクニカルサポートに連絡するときにキャプチャファイルを添付するか、自己分析のためにダウンロードできます。

3CXがWindowsで実行されている場合は、3CXサーバーにWiresharkを自分でインストールする必要があります。それ以外の場合、キャプチャしようとすると、次のメッセージが表示されます。


Linuxシステムでは、3CXをインストールまたはアップグレードすると、tcpdumpが自動的にインストールされます。

トラフィックキャプチャ


キャプチャを開始するには、インターフェースの[ホーム]> [SIPイベント]セクションに移動し、キャプチャするインターフェースを選択します。IPv6トンネリングインターフェイスを除くすべてのインターフェイスのトラフィックを同時にキャプチャすることもできます。



3CX for Linuxでは、ローカルホスト（lo）のトラフィックをキャプチャできます。このキャプチャは、3CXトンネルテクノロジーとセッションボーダーコントローラーを使用したSIPクライアント接続の分析に使用されます。

トラフィックキャプチャボタンは、WindowsではWireshark、Linuxではtcpdumpを起動します。この時点で、問題をすばやく再現する必要があります。キャプチャはプロセッサをロードし、十分なディスク領域を占有します。  


次の呼び出しパラメータに注意してください。

  • 他の番号/通話の参加者が呼び出した、呼び出された番号。
  • 3CXサーバーのクロックで問題が発生した正確な時刻。
  • 通話のルート。

「停止」ボタンを除いて、インターフェースの他の場所をクリックしないようにしてください。また、このブラウザウィンドウの他のリンクをたどらないでください。そうしないと、トラフィックのキャプチャがバックグラウンドで続行され、サーバーに追加の負荷がかかります。

キャプチャファイルを取得しています


停止ボタンは、キャプチャを停止し、キャプチャファイルを保存します。Wiresharkユーティリティで分析するためにファイルをコンピューターにダウンロードするか、このキャプチャやその他のデバッグ情報を含む特別なテクニカルサポートファイルを生成できます。テクニカルサポートパッケージにダウンロードまたは含めた後、キャプチャファイルはセキュリティ上の目的で3CXサーバーから自動的に削除されます。

3CXサーバーでは、ファイルは次の場所にあります。

  • Windows：C：\ ProgramData \ 3CX \ Instance1 \ Data \ Logs \ dump.pcap
  • Linux：/var/lib/3cxpbx/Instance/Data/Logs/dump.pcap

キャプチャ中のサーバー負荷の増加やパケット損失を回避するために、キャプチャ期間は200万パケットに制限されています。その後、キャプチャは自動的に停止します。より長いキャプチャが必要な場合は、以下で説明するように、別のWiresharkユーティリティを使用してください。

Wiresharkトラフィックキャプチャ


ネットワークトラフィックのより深い分析に関心がある場合は、手動でキャプチャしてください。こちらからお使いのOS用のWiresharkユーティリティをダウンロードしてください。ユーティリティを3CXサーバーにインストールした後、[キャプチャ]> [インターフェース]に移動します。OSのすべてのネットワークインターフェイスがここに表示されます。インターフェイスIPアドレスはIPv6で表示できます。IPv4アドレスを表示するには、IPv6アドレスをクリックします。



キャプチャするインターフェイスを選択し、[オプション]ボタンをクリックします。[無差別モードでトラフィックをキャプチャ]チェックボックスをオフにし、残りの設定は変更しないでください。



ここで、問題を再現する必要があります。問題が再現したら、キャプチャを停止します（メニュー[キャプチャ]> [停止]）。[テレフォニー]> [SIPフロー]メニューでSIPメッセージを選択できます。

トラフィック分析の基礎-SIP INVITEメッセージ


VoIPコールを確立するために送信されるINVITE SIPメッセージの主なフィールドを検討してください。分析の開始点です。通常、SIP INVITEには、SIP端末​​デバイス（電話、ゲートウェイ）および通信事業者が使用する情報を含む4〜6個のフィールドが含まれます。INVITEの内容とそれに続くメッセージを理解すると、問題の原因を特定するのに役立ちます。さらに、INVITEフィールドを知ることは、SIPオペレーターを3CXに接続するとき、または3CXを他のSIP交換と組み合わせるときに役立ちます。

INVITEメッセージでは、ユーザー（またはSIPデバイス）はURIで識別されます。通常、SIP URIはユーザーの電話番号+ SIPサーバーアドレスです。SIP URIは電子メールアドレスによく似ており、sip：x @ y：Portと記述されます。



リクエストラインURI：


Request-Line-URI-このフィールドには、呼び出しの受信者が含まれます。[宛先]フィールドと同じ情報が含まれますが、表示名は含まれません。

経由：


Via-INVITE要求が通過する各SIPサーバー（プロキシ）は、メッセージが受信されたIPアドレスとポートをViaリストの最上部に追加します。その後、メッセージはルートに沿ってさらに送信されます。最終受信者がINVITE要求に応答すると、すべての中継ノードがViaヘッダーを「調べ」、同じルートに沿って送信者にメッセージを返します。この場合、トランジットSIPプロキシはそのデータをヘッダーから削除します。

から：


From-ヘッダーは、SIPサーバーから見た要求の開始者を示します。ヘッダーは、電子メールアドレスと同じ方法で形成されます（user @ domain、userは3CXユーザーの内線番号、domainは3CXサーバーのローカルIPアドレスまたはSIPドメイン）。Toヘッダーと同様に、FromヘッダーにはURIが含まれ、オプションで表示名のユーザー名が含まれます。Fromヘッダーから、このSIPリクエストの処理方法を正確に理解できます。

SIP RFC 3261標準では、表示名が送信されない場合、IP電話またはVoIPゲートウェイ（UAC）は、表示名「匿名」を使用する必要があると規定しています。たとえば、From：「Anonymous」<sip：10000@10.172.0.2>

に：


To-このヘッダーは、リクエストの受信者を示します。これは、コールの最終受信者または中間リンクのいずれかになります。通常、ヘッダーにはSIP URIが含まれていますが、他のスキームも可能です（RFC 2806 [9]を参照）。ただし、機器の製造元に関係なく、すべてのSIP実装でSIP URIをサポートする必要があります。Toヘッダーには、表示名の表示名を含めることもできます（例：To：“ First Name Last Name” <sip：101@10.172.0.2>）。

通常、[宛先]フィールドには、リクエストを処理する最初の（次の）SIPプロキシを示すSIP URIが含まれています。これは、リクエストの最終的な受信者である必要はありません。

連絡先：


連絡先-ヘッダーには、INVITEリクエストの送信者に連絡できるSIP URIが含まれています。これは必須のヘッダーであり、1つのSIP URIのみを含める必要があります。これは、最初のSIP INVITE要求に対応する双方向通信の一部です。Contactヘッダーに、要求の送信者が応答を待機している正しい情報（IPアドレスを含む）が含まれていることが重要です。URI Contactは、通信セッションが確立された後の通信にも使用されます。

許可：


許可-フィールドには、コンマで区切られたパラメーター（SIPメソッド）のリストが含まれます。これらは、この送信者（デバイス）がサポートするSIPプロトコルの機能について説明しています。メソッドの完全なリスト：ACK、BYE、CANCEL、INFO、INVITE、NOTIFY、OPTIONS、PRACK、REFER、REGISTER、SUBSCRIBE、UPDATE。ここでは、その他のSIPメソッドについて説明します。
