典型的なOOPの誤解

こんにちは、ハブル！

今日、翻訳された出版物があなたを待っています。OOPとFIに関する新しい本に関連する検索をある程度反映しています。投票にご参加ください。



OOPパラダイムは死んでいますか？関数型プログラミングは未来だと言えるでしょうか？多くの記事がこれについて書いているようです。この見方に反対する傾向がある。話し合いましょう！

数か月ごとに、あるブログの投稿で、著者がオブジェクト指向プログラミングに対して一見根拠のある主張をし、その後彼女がOOPを過去の遺物であると宣言し、全員が関数型プログラミングに切り替えなければならない場合があります。

以前、私は OOPとFIが互いに矛盾しないことを書きました。さらに、それらを非常にうまく組み合わせることができました。

これらの記事の著者がOOPにそれほど多くの問題を抱えているのはなぜですか、なぜAFは彼らにとってそのような明白な代替手段のように見えるのですか？

OOPを教える方法


OOPを教えるとき、彼らは通常それが4つの原則に基づいていることを強調します：カプセル化、継承、抽象化、ポリモーフィズム。著者がPLOの衰退について推論する記事で通常批判されるのは、これらの4つの原則です。

ただし、OOPはFIと同様にツールです。問題を解決するため。それは消費される可能性があり、虐待される可能性もあります。たとえば、間違った抽象化を作成すると、OOPを悪用します。
したがって、クラスSquareはクラスを継承すべきではありませんRectangle。数学的な意味では、もちろんそれらは関連しています。ただし、プログラミングの観点からは、継承関係にはありません。実際には、正方形の要件は長方形の要件よりも厳しいです。長方形では2組の等しい辺がありますが、正方形ではすべての辺が等しくなければなりません。

継承


継承について詳しく説明しましょう。おそらく、継承されたクラスの美しい階層を持つ教科書の例を思い出してください。これらの構造はすべて、問題を解決するために機能します。ただし、実際には、継承は構成ほど頻繁には使用されません。
例を考えてみましょう。非常に単純なクラスであるWebアプリケーションのコントローラーがあるとします。最新のフレームワークのほとんどは、次のように操作することを前提としています。

class BlogController extends FrameworkAbstractController {
}

このようthis.renderTemplate(...)なメソッドはクラスから継承されるため、このようにすると、のような呼び出しが簡単になると想定されますFrameworkAbstractController。

このテーマに関する多くの記事に示されているように、多くの具体的な問題がここで発生します。基本クラスの内部関数は、実際にはAPIになります。彼女はもう変えることができません。ベースコントローラの保護された変数は、多かれ少なかれAPIに関連しています。

混乱することは何もありません。そして、コンポジションと依存性注入によるアプローチを選択した場合、次のようになります。

class BlogController {
    public BlogController (
        TemplateRenderer templateRenderer
    ) {
    }
}

ご存知のように、もはやある種の曖昧なものに依存しているのではなくFrameworkAbstractController、非常に明確に定義された狭い部分に依存していますTemplateRenderer。実際、BlogController動作を継承しないため、他のコントローラーから継承しません。

カプセル化


OOPの2番目によく批判される機能はカプセル化です。文学的言語では、カプセル化の意味は次のように定式化されます。データと機能は一緒に提供され、クラスの内部状態は外部の世界から隠されます。

この機会もまた、使用と乱用を可能にします。この場合の乱用の主な例は、漏洩状態です。

比較的言えば、クラスList<>に要素のリストが含まれていて、このリストは変更できると仮定します。次のように、注文バスケットを処理するクラスを作成してみましょう。

class ShoppingCart {
    private List<ShoppingCartItem> items;
    
    public List<ShoppingCartItem> getItems() {
        return this.items;
    }
}

ここで、ほとんどのOOP指向言語では、次のことが起こります。items変数は参照によって返されます。したがって、さらにこれを行うことができます。

shoppingCart.getItems().clear();

したがって、バスケット内のアイテムのリストを実際にクリアし、ShoppingCartはそれを知りません。ただし、この例を注意深く見ると、問題はカプセル化の原理ではないことが明らかになります。ここでは、ShoppingCart内部状態がクラスから流れているため、この原則に違反しています。

この特定の例では、クラスの作成者は不変性をShoppingCart使用して問題を回避し、カプセル化の原則に違反していないことを確認できます。

経験の浅いプログラマーは、カプセル化の原則に別の方法で違反することがよくあります。つまり、不要な状態を導入します。このような経験の浅いプログラマは、プライベートクラス変数を使用して同じクラス内のある関数から別の関数にデータを転送することがよくありますが、データ転送オブジェクトを使用して複雑な構造を別の関数に転送する方が適切です。このようなエラーの結果として、コードは非常に複雑になり、バグにつながる可能性があります。

一般的に、状態を完全に省くことは良いことです-可能な限りクラスに変更可能なデータを保存します。そうすることで、信頼できるカプセル化を確実にし、どこにもリークがないことを確認する必要があります。

抽象化


抽象化もまた、多くの点で誤って理解されています。コードに抽象クラスを詰め込んで、その中に深い階層を作成するべきではありません。

正当な理由なしにこれを行う場合、あなたは自分の頭のトラブルを探しているだけです。抽象クラスやインターフェイスとして、抽象化がどのように行われるかは関係ありません。いずれにしても、コードに余分な複雑さが現れます。この複雑さは正当化されなければなりません。
簡単に言うと、時間を費やして、それを実装するクラスに期待される動作を文書化する用意がある場合にのみ、インターフェースを作成できます。はい、あなたは私を正しく読みました。実装する必要がある関数のリストを作成するだけでは十分ではありません。また、それらがどのように（理想的には）機能するかを説明します。

ポリモーフィズム


最後に、ポリモーフィズムについて話しましょう。彼は、1つのクラスで多くの動作を実装できることを示唆しています。悪い教科書の例はSquare、多態性でそれが両方Rectangleになる可能性があることを書くことParallelogramです。上ですでに指摘したように、これらのエンティティの動作は異なるため、OOPでこれを行うことは明らかに不可能です。

ポリモーフィズムといえば、コードではなく動作を覚えておく必要があります。良い例はSoldier、コンピューターゲームのクラスです。彼は行動Movable（状況：彼が動くことができる）と行動Enemy（状況：あなたを撃つ）の両方を実現することができます。対照的に、クラスGunEmplacement は動作のみを実装できますEnemy。

だからあなたが書くならSquare implements Rectangle, Parallelogram、このステートメントは真になりません。抽象化はビジネスロジックに従って機能する必要があります。コードよりも動作について考える必要があります。

FPが特効薬ではない理由


それでは、OOPの4つの基本原則を繰り返したとき、関数型プログラミングの機能とは何かを考えてみましょう。コードを使用して、コード内のすべての問題を解決してみませんか？

FPの多くの支持者の観点から見ると、クラスは犠牲であり、コードは関数の形式で提示する必要があります。言語に応じて、データは関数から関数へ、プリミティブ型を使用して、または1つまたは別の構造化されたデータのセット（配列、辞書など）の形式で転送できます。

さらに、ほとんどの関数には副作用があってはなりません。つまり、バックグラウンドの予期しない場所でデータを変更するのではなく、入力パラメータを使用して出力を生成するだけです。

このアプローチはデータを分離します。機能 -一見、このFPは根本OOPは異なります。 FPは、このようにしてコードが単純なままであることを強調しています。あなたは何かをしたい、この目的のために関数を書きたい-それだけです。

問題は、一部の機能が他の機能に依存する必要があるときに始まります。関数Aが関数Bを呼び出し、関数Bがさらに5〜6個の関数を呼び出すと、最後に、ゼロ充填関数が検出され、壊れる可能性があります。これがうらやましくない場所です。

FPの支持者であると考えるほとんどのプログラマーは、その単純さのためにFPを愛しており、そのような問題を深刻だとは考えていません。これは、単にコードを渡すだけで、二度とそれについて考えないことがタスクである場合、かなり正直です。サポートに便利なコードベースを構築する場合は、純粋なコードの原則に準拠することをお勧めします。特に、実際にはFIもはるかに複雑になる依存関係の逆転を適用することをお勧めします。

OOPまたはFP？


OOPとFIはツールです。最終的には、どのプログラミングパラダイムを使用してもかまいません。このトピックに関するほとんどの記事で説明されている問題は、コードの編成に関連しています。

私の意見では、アプリケーションのマクロ構造ははるかに重要です。その中のモジュールは何ですか？彼らはどのようにして相互に情報を交換しますか？どのデータ構造が最も一般的ですか？彼らはどのように文書化されていますか？ビジネスロジックの観点から最も重要なオブジェクトはどれですか。

これらすべての問題は、使用されるプログラミングパラダイムとはまったく関係がなく、そのようなパラダイムのレベルでは、解決することすらできません。優れたプログラマーは、提供するツールを習得するためにパラダイムを研究し、タスクを解決するのに最適なものを選択します。
