Abletonは必要ありません：Ableton Push 2をVCVラックに接続します


最近の音楽の作成は、10年前の写真とほとんど同じです。それぞれに独自のDSLRアカウントとInstagramアカウントがあります。音楽業界はこれに非常に満足しています。なぜなら、そのような関心は多くのお金をもたらすからです。毎日、新しいVSTプラグインとアナログデバイスが登場し、テーマ別コースの数は急速に増えており、音楽制作に特化したブログはトップのYouTubeにアクセスしています。このような背景に対して、オープンソースプロジェクトは非常に興味深く見え、これに多額のお金を費やすことなくプロデューサーとして自分自身を試してみたい人は誰でも許可できます。そのようなプロジェクトの1つにVCVラックがあります。VCVラックは、最も高価なクラスの楽器（アナログモジュラーシンセサイザー）を誰でもアクセスできるようにすることを目的としています。最近、トラックのアイデアを思いついた時、手元にはLinuxラップトップしかなかったので、VCVですべてのトラックを作りたかったのです。そして途中で、利用可能なモジュールを使用して機能しないようにMIDIコントローラーを使用してシンセサイザーを制御したいという要望がありました。結局、Ableton Push 2をVCVラックに接続するためのプラグインを作成することにしました。この記事では、その原因と、VCV用の独自のモジュールを作成する方法について説明します。

VCVラックAPI


VCVの各モジュールは、オーディオとグラフィックの2つの部分で構成されています。オーディオ部分は、Moduleクラスから、サンプルごとに呼び出される、つまりサンプリング周波数で処理メソッドを継承します。ちなみに、VCVのサンプリング周波数は、標準の44.1 kHzから768 kHzまでさまざまで、十分な計算能力を持つモジュラーシンセサイザーのより正確なエミュレーションが可能です。ModuleWidget

タイプのオブジェクトは、基本構造からdrawメソッドを継承するグラフィックスを担当します。VCVはnanovgベクトルグラフィックスライブラリを使用します。drawメソッド内での描画これは、モジュールの境界内（過剰な部分はエンジンによって切り捨てられる）と、たとえばまだ使用しているフレームバッファー内の両方で発生する可能性があります。

それでは、VCV用のモジュールを作成するには何が必要ですか

環境のセットアップとRack SDKのインストール


最初のステップはドキュメントで十分に説明されており、問題は発生しません（少なくともLinuxでは）ので、ここでは詳しく説明しません。

プラグインテンプレートを生成する


Rack SDKには、このためのhelper.pyスクリプトがあります。彼はcreatepluginと言ってから、プラグインの名前と、オプションでプラグインに関する情報を指定する必要があります。

<Rack SDK folder>/helper.py createplugin MyPlugin

プラグインテンプレートが作成されると、次のコマンドでコンパイルおよびインストールできます。

RACK_DIR=<Rack SDK folder> make install

モジュールのフロントエンドを描画します


各プラグインには複数のモジュールを含めることができ、モジュールごとにメインパネルを描画する必要があります。このため、VCVのドキュメントでは、Inkscapeまたはその他のベクターエディターを使用するように指示されています。 VCVのモジュールは仮想ユーロラックスタンドに取り付けられているため、高さは常に128.5 mmで、幅は5.08 mmの倍数にする必要があります。
CV /ゲートソケット、ボタン、電球などの基本的なインターフェース要素は、ベクター形式でマークできます。これを行うには、対応する色の円をその場所に描画します（詳細はこちら）。これにより、helper.pyがこのマークアップのコードを生成します。個人的には、これはあまり便利な機能ではなく、コードから直接要素を配置する方が簡単だと思いました。画像とレイアウトの準備ができたら、もう一度helper.pyを実行する必要があります モジュールテンプレートを作成し、フロントパネルに関連付けます。

helper.py createmodule MyModule res/MyModule.svg src/MyModule.cpp

ボタンとツイストをつなぐ




ディスプレイとは別に、Ableton Push 2はコンピューターから通常のUSB-MIDIデバイスとして認識されるため、VCVラックとの接続を簡単に確立できます。これを行うには、モジュールのクラス内に入力MIDIキューと出力MIDIポートを作成します。

初期化コード
struct AbletonPush2 : Module {
    midi::Output midiOutput;
    midi::InputQueue midiInput;

    bool inputConnected;
    bool outputConnected;
}


接続されているMIDIデバイスの中からAbleton Pushを見つけて接続してみましょう。このモジュールはAbleton Pushでのみ機能するように設計されているため、ユーザーがデバイスを選択する必要がなく、名前で簡単に見つけることができます。

コントローラー接続コード
void connectPush() {
    auto in_devs = midiInput.getDeviceIds();
    for (int i = 0; i < in_devs.size(); i++){
        if (midiInput.getDeviceName(in_devs[i]).find("Ableton") != std::string::npos) {
            midiInput.setDeviceId(in_devs[i]);
            inputConnected = true;
            break;
        }
    }
		
    auto out_devs = midiOutput.getDeviceIds();
    for (int i = 0; i < out_devs.size(); i++){
        if (midiOutput.getDeviceName(out_devs[i]).find("Ableton") != std::string::npos) {
            midiOutput.setDeviceId(out_devs[i]);
            outputConnected = true;
            break;
        }
    }
}


これで、プロセスメソッドで、コントローラーが接続されているかどうかを定期的に確認し、メッセージが到着したかどうかをmidiキューに確認できます。ここでは、midi標準で一般的にエンコードされる内容と方法について言及する価値があります。実際には、メッセージには2つの主要なタイプがあります。これらは、ノートON / OFF、ノート番号と押圧力を送信すること、およびCC-コマンドコントロール、いくつかの可変パラメーターの数値を送信することです。midiの詳細については、こちらをご覧ください。

MIDIキューポーリングコード
void process(const ProcessArgs &args) override {
    if (sampleCounter > args.sampleRate / updateFrequency) {

        if ((!inputConnected) && (!outputConnected)) {
            connectPush();
        }

	midi::Message msg;
	while (midiInput.shift(&msg)) {
	    processMidi(msg);
	}
    }
}


ここで、コントローラーパッドを異なる色で光るように教えて、ナビゲートするのがより便利になるようにします。これを行うには、適切なmidiコマンドを送信する必要があります。たとえば、パッド番号36を考えます（これは左下のパッドです）。クリックすると、コントローラーはコマンド0x90（ノートオン）に続いてノート番号（36）と0〜127の番号（プレスの強さ）を送信します。逆に、コンピューターがコントローラーに同じ0x90コマンドと同じノート番号36を送信すると、3番目の番号は左下のパッドが光る色を示します。ボタンとパッドの数を上の図に示します。プッシュには、カラー、パレット、アニメーション、その他のバックライトパラメータを操作するための可能性がかなりあります。詳細については触れずに、デフォルトパレットのすべての可能な色をパッドに持ってきて、好きな色を選択しました。しかし、一般的なケースでは、ドキュメントによると、MIDIコマンドからLEDのPWM値への変換は次のようになります：



バックライト制御コード
void lightOn(int note, int color){
    midi::Message msg;
    msg.setNote(note);
    msg.setValue(color);
    msg.setChannel(1);
    msg.setStatus(0x9);
    midiOutput.sendMessage(msg);
}

void lightOff(int note){
    midi::Message msg;
    msg.setNote(note);
    msg.setValue(0);
    msg.setChannel(1);
    msg.setStatus(0x8);
    midiOutput.sendMessage(msg);
}


ディスプレイを接続します




ディスプレイを接続するには、USBを使用する必要があります。コントローラ自体は何かを描画する方法を知らず、グラフィックスについても何も知りません。彼が望んでいるのは、少なくとも2秒ごとに160x960ピクセルのサイズのフレームを送信することです。すべてのレンダリングはコンピュータの側面で行われます。ドキュメントで説明されているように、開始するには、USBデバイスを接続して開きます。

接続コードを表示
#ifdef _MSC_VER
#define _CRT_SECURE_NO_WARNINGS
#endif

#include <stdio.h>

#ifdef _WIN32

// see following link for a discussion of the
// warning suppression:
// http://sourceforge.net/mailarchive/forum.php?
// thread_name=50F6011C.2020000%40akeo.ie&forum_name=libusbx-devel

// Disable: warning C4200: nonstandard extension used:
// zero-sized array in struct/union
#pragma warning(disable:4200)

#include <windows.h>
#endif

#ifdef __linux__
#include <libusb-1.0/libusb.h>
#else
#include "libusb.h"
#endif

#define ABLETON_VENDOR_ID 0x2982
#define PUSH2_PRODUCT_ID  0x1967

static libusb_device_handle* open_push2_device()
{
  int result;

  if ((result = libusb_init(NULL)) < 0)
  {
    printf("error: [%d] could not initilialize usblib\n", result);
    return NULL;
  }

  libusb_set_debug(NULL, LIBUSB_LOG_LEVEL_ERROR);

  libusb_device** devices;
  ssize_t count;
  count = libusb_get_device_list(NULL, &devices);
  if (count < 0)
  {
    printf("error: [%ld] could not get usb device list\n", count);
    return NULL;
  }

  libusb_device* device;
  libusb_device_handle* device_handle = NULL;

  char ErrorMsg[128];

  // set message in case we get to the end of the list w/o finding a device
  sprintf(ErrorMsg, "error: Ableton Push 2 device not found\n");

  for (int i = 0; (device = devices[i]) != NULL; i++)
  {
    struct libusb_device_descriptor descriptor;
    if ((result = libusb_get_device_descriptor(device, &descriptor)) < 0)
    {
      sprintf(ErrorMsg,
        "error: [%d] could not get usb device descriptor\n", result);
      continue;
    }

    if (descriptor.bDeviceClass == LIBUSB_CLASS_PER_INTERFACE
      && descriptor.idVendor == ABLETON_VENDOR_ID
      && descriptor.idProduct == PUSH2_PRODUCT_ID)
    {
      if ((result = libusb_open(device, &device_handle)) < 0)
      {
        sprintf(ErrorMsg,
          "error: [%d] could not open Ableton Push 2 device\n", result);
      }
      else if ((result = libusb_claim_interface(device_handle, 0)) < 0)
      {
        sprintf(ErrorMsg,
          "error: [%d] could not claim interface 0 of Push 2 device\n", result);
        libusb_close(device_handle);
        device_handle = NULL;
      }
      else
      {
        break; // successfully opened
      }
    }
  }

  if (device_handle == NULL)
  {
    printf(ErrorMsg);
  }

  libusb_free_device_list(devices, 1);
  return device_handle;
}

static void close_push2_device(libusb_device_handle* device_handle)
{
  libusb_release_interface(device_handle, 0);
  libusb_close(device_handle);
}


フレームを送信するには、最初に16バイトのヘッダーを送信し、次に960個の16ビット番号の160行を送信する必要があります。同時に、ドキュメントによれば、文字列は1920バイトで送信されるべきではなく、ゼロが追加された2048バイトのパケットで送信されるべきです。

フレーム転送コード
struct Push2Display : FramebufferWidget {

    unsigned char frame_header[16] = {
          0xFF, 0xCC, 0xAA, 0x88,
          0x00, 0x00, 0x00, 0x00,
          0x00, 0x00, 0x00, 0x00,
          0x00, 0x00, 0x00, 0x00 
    };

    libusb_device_handle* device_handle;

    unsigned char* image;

    void sendDisplay(unsigned char * image) {
        int actual_length;
     
        libusb_bulk_transfer(device_handle, PUSH2_BULK_EP_OUT, frame_header, sizeof(frame_header), &actual_length, TRANSFER_TIMEOUT);
        for (int i = 0; i < 160; i++)
            libusb_bulk_transfer(device_handle, PUSH2_BULK_EP_OUT, &image[(159 - i)*1920], 2048, &actual_length, TRANSFER_TIMEOUT);
    }
}


現在は、フレームを描画してバッファに書き込むだけです。これを行うには、drawFramebufferメソッドを実装するFramebufferWidgetクラスを使用します。箱から出してすぐのVCVラックはnanovgライブラリーを使用しているため、ここでグラフィックを書くのは非常に簡単です。グローバル変数APPから現在のグラフィックコンテキストを学習し、その状態を保存します。次に、空の160x960フレームを作成し、それをdrawメソッドで描画します。その後、USB経由で送信される配列にフレームバッファーをコピーし、コンテキストの状態を返します。最後に、ダーティフラグを設定して、レンダリングの次の反復で、VCVエンジンがウィジェットの更新を忘れないようにします。

フレームレンダリングコード
void drawFramebuffer() override {

    NVGcontext* vg = APP->window->vg;

    if (display_connected) {
        nvgSave(vg);
	glViewport(0, 0, 960, 160);
        glClearColor(0, 0, 0, 0);
        glClear(GL_COLOR_BUFFER_BIT|GL_STENCIL_BUFFER_BIT);

	nvgBeginFrame(vg, 960,  160, 1);
        draw(vg);
        nvgEndFrame(vg);

        glReadPixels(0, 0, 960, 160, GL_RGB, GL_UNSIGNED_SHORT_5_6_5, image); 

        //  XOR   ,  
        for (int i = 0; i < 80*960; i ++){
            image[i * 4] ^= 0xE7;
	    image[i * 4 + 1] ^= 0xF3;
            image[i * 4 + 2] ^= 0xE7;
	    image[i * 4 + 3] ^= 0xFF;
        }
	    	
	sendDisplay(image);

	nvgRestore(vg);
    }

    dirty = true;
}


相互作用のロジックとパラメーターのマッピング


私のタスクでは、シーケンサーに記録されたパターンを切り替え、同時にパターンのグループごとに独自のパラメーターセットを制御できるようにしたいと考えました。マッピングすることで、あるモジュールのパラメーターと別のモジュールまたはMIDIコントローラーのパラメーターとのバインディングを理解します。美しくマッピングする方法についてはほとんど情報が見つからなかったので、それを実装するコードのほとんどをVCVに組み込まれているMIDIマップモジュールから取得しました。つまり、パラメータの束ごとに特別なParamHandleオブジェクトが作成され、松葉杖を通じて何が何に接続されているかをエンジンに伝えます。

結論


これが私が最後に得たモジュールです。MIDIからCVへの標準変換に加えて、パッドをグループ化し、色を割り当て、モジュールの任意のパラメーターをプッシュエンコーダーに関連付けて、コントローラーのディスプレイにそれらの名前と値を表示できます。下のビデオでは彼の概要を見ることができ、このビデオでは実際の動作を見ることができます。
