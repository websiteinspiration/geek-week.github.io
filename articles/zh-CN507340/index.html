我们如何通过网络延迟补偿算法胜过移动射击游戏的弹道计算机制





嗨，我是Pixonic的服务器开发人员Nikita Brigak。今天，我想谈谈对移动多人游戏的延迟补偿。



关于服务器延迟补偿的文章很多，包括俄语。这并不奇怪，因为自90年代末以来，该技术已被积极地用于创建多用户FPS。例如，您可以回忆起QuakeWorld mod，它是最早使用它的模型之一。



我们在移动多人射击游戏《恐龙小队》中使用了它。



在本文中，我的目标不是重复一千遍，而是要考虑到我们的技术堆栈和核心游戏玩法，告诉我们如何在游戏中实现滞后补偿。



简而言之，我们的核心和技术。



Dino Squad-网络移动PvP射击游戏。玩家控制恐龙，悬挂各种武器，并与6至6个团队作战



。Unity上的客户端和服务器均如此。对于射击者而言，该体系结构非常经典：服务器是专制的，客户端预测可以在客户端上进行。游戏模拟是使用内部ECS编写的，可在服务器和客户端上使用。



如果您第一次听说过滞后补偿，那么这里是对问题的简要介绍。



在多人FPS游戏中，通常在远程服务器上模拟比赛。玩家将他们的输入发送到服务器（有关按下的按键的信息），作为响应，服务器会考虑收到的数据向他们发送更新的游戏状态。通过这种交互方式，按下“前进”键与玩家角色在屏幕上移动的时间之间的延迟将始终具有更多的ping。



如果在本地网络上这种延迟（通常称为输入延迟）是不可见的，则在通过Internet进行游戏时，在控制角色时会产生“在冰上滑动”的感觉。这个问题对于移动网络来说是双重的，在这种情况下，播放器的ping为200毫秒的情况被认为是一个很好的连接。 Ping通常为350、500和1000毫秒。这样，在快速射击游戏中几乎不可能出现输入延迟。



解决此问题的方法是客户端仿真预测。在这里，客户端本身将输入应用于玩家的角色，而无需等待服务器的响应。当收到答案时，它仅比较结果并更新对手的位置。在这种情况下，按键和显示结果之间的延迟最小。



重要的是要了解细微差别：客户端总是根据他的最后输入来吸引自己，而敌人则像以前一样从服务器的数据中汲取了网络延迟。也就是说，在向对手射击时，玩家会看到他过去相对于自己。我们在前面写了更多有关客户预测的内容。



因此，客户预测解决了一个问题，但又产生了另一个问题：如果玩家在敌人过去的地点射击，则服务器在敌人的相同地点射击可能不在同一位置。服务器延迟补偿正试图解决此问题。发射武器时，服务器恢复玩家在射击时在本地看到的游戏状态，并检查他是否真的可以击中敌人。如果答案是肯定的，则即使服务器上此时不再有敌人，命中计数也算在内。



有了这些知识，我们就开始在Dino Squad中实施基于服务器的滞后补偿。首先，我必须了解如何在服务器上还原客户端看到的内容？到底需要恢复什么？在我们的游戏中，武器和技能的命中是通过rakecast和重叠（即通过与敌人的物理对撞机进行交互）来计算的。因此，玩家需要在本地“看到”这些对撞机的位置，我们需要在服务器上进行游戏。当时，我们使用Unity版本2018.x.物理API在此处是静态的，物理世界存在于单个副本中。没有机会保存他的病情，以便以后可以将他从盒子里恢复过来。那么该怎么办？

将所有这些元素放在一起，我们得到了一个“时间机器”，可以将物理世界的状态回滚到正确的时刻。代码很简单： 



public class TimeMachine : ITimeMachine
{
     //  
     private readonly IGameStateHistory _history;

     //    
     private readonly ExecutableSystem[] _systems;

     // ,      
     //    
     private readonly GameState _presentState;

     public TimeMachine(IGameStateHistory history, GameState presentState, ExecutableSystem[] timeInitSystems)
     {
         _history = history; 
         _presentState = presentState;
         _systems = timeInitSystems;  
     }

     public GameState TravelToTime(int tick)
     {
         var pastState = tick == _presentState.Time ? _presentState : _history.Get(tick);
         foreach (var system in _systems)
         {
             system.Execute(pastState);
         }
         return pastState;
     }
}


仍然需要了解如何使用此机器来补偿射门和能力的滞后。



在最简单的情况下，当机制是基于单个命中扫描时，一切似乎都很清楚：在玩家射击之前，您需要将物理世界回滚到所需的状态，进行rakecast，计算命中或未命中并使世界回到其原始状态。



但是在Dino Squad中，这样的机制很少！游戏中的大多数武器都是由弹丸创造的。弹丸是长寿命的子弹，可以在模拟过程中飞行多个刻度（在某些情况下为数十个刻度）。与他们做什么，他们应该在什么时候飞？



在一篇有关“半条命”网络堆栈的古老文章中，来自Valve的家伙问了同样的问题，他们的回答是：项目文件的滞后补偿是有问题的，最好避免这种情况。



我们没有这个选择：基于弹丸的武器是游戏设计的关键特征。因此，我们不得不发明一些东西。经过一番制动之后，我们制定了两个似乎可行的选择：



1.将项目与创建它的玩家的时间联系起来。对于每个玩家的每个子弹，每次服务器模拟时，我们都会将物理世界回滚到客户端状态，并执行必要的计算。这种方法可以使服务器上的负载分散，并可以预测项目文件的飞行时间。可预测性对我们尤为重要，因为我们可以在客户端上预测所有弹丸，包括对手的弹丸。





在图片中，第30滴答声中的玩家预先发射了导弹：他看到敌人朝哪个方向奔跑，并且知道了火箭的大致速度。在本地，他发现自己在第33滴答声中击中了目标。多亏了滞后补偿，它也将到达服务器

在我们的实现中，这两种方法在几行代码上确实有所不同，因此我们都将其视而不见，并且很长一段时间以来，我们使它们并行。根据武器的结构和子弹的速度，我们为每种恐龙选择一个或另一个选项。这里的转折点是机械师在游戏中的出现，例如“如果您在同一时间多次击中敌人，将获得如此高的奖励”。任何在玩家击中敌人时扮演重要角色的机制都拒绝与第二种方法成为朋友。因此，最后，我们确定了第一个选项，现在将其用于游戏中的所有武器和所有活动能力。



我们还应该提出绩效问题。如果您觉得这一切都会放慢脚步，我回答：就是这样。移动碰撞器，打开和关闭Unity的速度很慢。在Dino Squad中，在“最坏的情况”下，战斗中可能同时存在数百个项目文件。移动对撞机分别计算每个项目是一种不可允许的奢侈。因此，对我们来说，绝对必要的是最大程度地减少物理“回扣”的数量。为此，我们在ECS中创建了一个单独的组件，其中记录了玩家的时间。我们将其挂在所有需要滞后补偿的实体上（与项目相关，与能力有关）。在开始处理此类实体之前，我们先将它们聚类并一起处理，为每个聚类滚动一次物理世界。



在这个阶段，我们得到了整个工作系统。它的代码有些简化： 



public sealed class LagCompensationSystemGroup : ExecutableSystem
{
     // 
     private readonly ITimeMachine _timeMachine;

     //  
     private readonly LagCompensationSystem[] _systems;
     
     //  
     private readonly TimeTravelMap _travelMap = new TimeTravelMap();

    public LagCompensationSystemGroup(ITimeMachine timeMachine, 
        LagCompensationSystem[] lagCompensationSystems)
     {
         _timeMachine = timeMachine;
         _systems = lagCompensationSystems;
     }

     public override void Execute(GameState gs)
     {
         //      ,
         //     «».     ,
         //          .
         var buckets = _travelMap.RefillBuckets(gs);

         for (int bucketIndex = 0; bucketIndex < buckets.Count; bucketIndex++)
         {
             ProcessBucket(gs, buckets[bucketIndex]);
         }

         //       
         //  
         _timeMachine.TravelToTime(gs.Time);
     }

     private void ProcessBucket(GameState presentState, TimeTravelMap.Bucket bucket)
     {
         //      
         var pastState = _timeMachine.TravelToTime(bucket.Time);

         foreach (var system in _systems)
         {
               system.PastState = pastState;
               system.PresentState = presentState;

               foreach (var entity in bucket)
               {
                   system.Execute(entity);
               }
          }
     }
}


只剩下配置细节了：



1.了解在时间上限制最大移动范围的程度。



对我们来说，重要的是要在移动网络不佳的情况下使游戏尽可能易于访问，因此我们将故事限制为30个刻度（刻度频率为20 Hz）。这使玩家即使以很高的ping击中对手也可以。



2.确定哪些对象可以及时移动，哪些不能。



显然，我们正在移动对手。但是安装了能量屏蔽罩，例如，没有。我们决定，最好像网络射击手那样优先考虑防御能力。如果玩家已经将盾牌放在现在，则过去的经过滞后补偿的子弹不应飞过它。



3.决定是否有必要补偿恐龙的能力：咬人，打尾巴等。我们决定了所需的东西，并按照与子弹相同的规则对其进行处理。



4.确定如何处理执行滞后补偿的玩家对撞机。以一种很好的方式，他们的位置不应转移到过去：玩家必须在他现在在服务器上的同时看到自己。但是，我们也会回退射击运动员的对撞机，这有几个原因。



首先，它改善了聚类：我们可以对所有近距离ping的玩家使用相同的物理状态。



其次，在所有重复和重复的情况下，我们始终排除拥有能力或射弹的玩家的对撞机。在《恐龙小队》中，玩家按照射手的标准来控制具有相当不寻常几何形状的恐龙。即使玩家以不同寻常的角度射击，而且子弹的轨迹穿过玩家的恐龙对撞机，子弹也不会理会它。



第三，我们甚至在延迟补偿开始之前，根据ECS的数据计算恐龙武器的位置或能力的应用点。



结果，滞后补偿玩家的对撞机的真实位置对我们来说微不足道，因此我们走的是效率更高，同时更简单的方式。



网络延迟不能简单地消除；只能被掩盖。像任何其他伪装方法一样，服务器滞后补偿也有其自身的权衡。它改善了射击游戏者的游戏体验，但以牺牲被射击者为代价。但是，对于Dino Squad而言，这里的选择显而易见。



当然，对于程序员和游戏设计师来说，随着服务器代码整体复杂性的增加，我必须为此付出一切。如果早先的仿真是对系统的简单顺序调用，那么在进行滞后补偿的情况下，就会出现嵌套循环和分支。为了方便使用，我们还付出了很多努力。



在2019年版本中（或可能更早一些），Unity对独立的物理场景有正负完全支持。更新之后，我们几乎立即在服务器上实现了它们，因为我们想快速摆脱所有房间共有的物理世界。



我们为每个游戏室提供了自己的物理场景，因此无需在计算模拟之前从下一个房间的数据“清除”场景。首先，它大大提高了生产率。其次，如果程序员在添加新游戏元素时为了清除场景而在代码中犯了错误，它可以使我们摆脱一整类的错误。这样的错误很难调试，并且经常导致这样一个事实，即来自一个房间的场景的物理对象的状态“流入”另一个房间。



此外，我们对是否可以使用物理场景存储物理世界的历史进行了一些研究。也就是说，有条件的是，不为每个房间分配一个场景，而是为30个场景分配一个场景，并从中创建一个循环缓冲区，以存储历史记录。总的来说，该选项是可行的，但是我们没有介绍它：它并没有显示出某种疯狂的生产率提高，但是需要相当冒险的更改。很难预测在这么多场景的长时间工作中服务器的行为。因此，我们遵循以下规则：“ 如果没有损坏，请不要修复它。”
