Do not create your own user authentication solutions

The author of the article, the translation of which we are publishing, suggests stopping writing your own code for user authentication. He believes that the time has come to implement safer solutions that, even if not to talk about their other advantages, can save time and money.



Most applications require some kind of user authentication subsystem. Maybe you are a programmer working in a large company on business applications in which you want to restrict access to certain materials to unauthorized employees of the company and you need to check the permissions of these users. Perhaps you are writing the code for the new SaaS platform, and you need it so that its users can create and manage accounts.







In both of these cases, and in many others, you will most likely begin work on the application by creating authentication subsystems and user management tools. That is, at a minimum, create a registration form and a login page. Authentication subsystems are exactly what most often are asked to implement web developers engaged in certain projects. But, despite this, such subsystems are also something that very little attention is usually paid to.



Developing a secure user authentication system is a real challenge. It is much larger than many people think. This problem is very easy to solve incorrectly. Even worse: errors in creating authentication subsystems can be disastrous. The basic structure of authentication and user management systems includes only a few forms. Because of this, the creation of such systems may seem quite simple. But, as you know, the devil is in the details. It takes a lot of work to make such systems safe (and, when possible or even necessary, take into account the privacy requirements of personal data).



IDaaS



Fortunately, a modern web developer does not need to deploy their own system for authenticating and managing users. Now, in 2020, there are many good IDaaS solutions (Identity as a Service, Identification as a Service). The use of such solutions greatly simplifies equipping web projects with user authentication capabilities.



Here are some popular IDaaS projects (in alphabetical order): Auth0 , Azure AD , Google Identity Platform, and Okta .



In addition, there are providers of identity information based on something like social media user accounts. This, for example, Apple's , Facebook, Github , twitter . It's very easy for developers to incorporate the capabilities of these systems into their projects. Anyone working with authentication providers can very quickly create an appropriate application subsystem that has access to a large amount of user data. But sometimes the interaction of the project with providers can have a negative impact on the privacy of user data.



In fact, one cannot name the real reason why a certain developer should not use the services of an identity information provider. Working with such a service will save a lot of time that can be spent on developing the application itself. These services give the developer, without any significant effort on his part, many opportunities. And the most important thing here is that they are much safer than some self-written solution.



The more users, the safer the solution.



Most identity providers offer additional security features. Such as support for multi-factor authentication (MFA, multi-factor authentication), support for security certificates or keys (including U2F, FIDO2, WebAuthn , etc.).



Do not underestimate the importance of these technologies. According to a Microsoft report , the inclusion of MFA can prevent up to 99.9% of attacks on accounts.



However, in the use of third-party authentication services instead of their own solutions, there is another aspect, less known. Namely, we are talking about the fact that due to the very large number of users with whom these services work, they can monitor the behavior patterns of attackers and more easily prevent attacks on accounts.



Large providers, due to the fact that they have millions of users who perform countless logins every day, collect enough data to build models based on the capabilities of artificial intelligence. Such models allow better identification of suspicious patterns of behavior.



For example, suppose one of your users living in Canada logs in from home. And after two hours, the entrance to the same account is performed from Ukraine. The services of the identity provider can consider this behavior to be suspicious and can either immediately prohibit entry or offer the user to undergo additional verification (for example, using an MFA token). In addition, they can notify users who have been attacked, or system administrators, and possibly both.

Where to begin?



I hope I convinced you that user authentication is a task that is best accomplished through a dedicated identity provider. Here's how to do it.



For starters, the good news. It consists in the fact that all four of the above providers (Auth0, Azure AD, Google Identity Platform, Okta), and many others, use the same protocols: OpenID Connect / OAuth 2.0. These are modern standard protocols, client libraries for support of which exist for almost all programming languages ‚Äã‚Äãand frameworks.



In order to start using the services of any provider of identification information, if you do not go into details, do the following:



  1. Register the application with the identity provider. After registration, you will be given an identifier (Application ID, Client ID) and a secret key (Secret Key, Client Secret).
  2. Set the permissions required by your application. In addition to the fact that the application will have a user profile available, you, depending on the provider you choose, will be able to access a much larger amount of data. This may include, for example, access to user messages and access to their cloud storage (for example, through Office 365 or G Suite).
  3. Integrate the client library of the authentication service into your project.

A cryptographically signed JWT token with a limited validity period can be used to manage user sessions. That is, as long as the token is valid, if the application receives it in the request, it can consider this request as coming from the user who owns the token.



The same token contains some user information. The set of this information depends on the authentication service used. Usually this is the username, his email address, his identifier.



An application may use this information to identify a user. You can use the user ID to access the data associated with it stored in your application.



As already mentioned, JWT tokens are cryptographically signed. Therefore, when checking the token signature, you can be sure that no one has replaced the user information that is contained in the token.
