電子医療情報アーカイブが病気の診断をより効果的にする方法



IDCの予測によると、2025年までに、医療機関に保存されるデータの総量は2.3ゼタバイトに増加し、さまざまな医療画像が使用済みストレージ容量の最大80〜90％を占めるようになります。次の例では、医用画像の効率的な保存の重要性が重要です。

ツーソン（アリゾナ州）の病院の診断部門では、マンモグラフィ（X線、超音波、MRIに基づく乳癌の診断）にMRIスキャンを使用して、2匹の哺乳類のX線画像を最大40枚撮影しています。だけでなく、2つから5つの生検。 80％のケースでは、2年前に撮影された古い患者の写真がマンモグラフィの結果を解釈するために使用され、困難なケースでは、10年前に撮影された写真が必要になる場合があります。アーカイブから古い写真をすばやく取得するために、PACS（画像アーカイブおよび通信システム）デジタル病院システムが使用されます。

PACSに保存された古い画像を使用すると、悪性腫瘍の診断におけるエラーのリスクが大幅に軽減され、良性腫瘍の患者がマンモグラムを必要とせず、腫瘍が悪性でないことを確認するために生検さえも行われなくなります。同時に、古い画像と比較することで、悪性腫瘍の画像を誤って解釈するリスクを軽減し、患者に病理の適切な治療と追加の分析をすばやく割り当てることができます。

医療画像アーカイブの長期保存の特徴


PACSのストレージシステムとして使用される理想的なストレージシステムは何ですか？明らかに、大きなサイズの医用画像を考えると、数十年に渡って蓄積された各患者の分析結果を保存できるように、それは非常にスケーラブルでなければなりません。 2番目の要件は、アーカイブからのデータの迅速な検索と取得を確実にすることです。これを行わないと、古い画像を使用して新しい調査の結果を解釈することが不可能になります。

最後に、医用画像を保存するときは、「データ漏洩」、紛失、偶発的または意図的な削除または損傷の可能性を完全に排除する必要があります。データをPACSに保存する機能は、比較的穏やかなI / Oパフォーマンス要件です。分析結果がアーカイブに書き込まれるのは1度だけであり、決して変更されないことは明らかであり、限られた数のユーザーだけがアーカイブからこのデータを抽出するリクエストを送信できます。患者データは通常、年に1回だけ要求されます。



従来のエンタープライズクラスのストレージシステムは、主にデータストレージのコストが高すぎるため、PACSには適していません。これは主に、トランザクションアプリケーションにサービスを提供するときの医用画像アーカイブには不要な高性能によるものであり、安価なエントリーレベルのストレージシステムには、PACSアーカイブに必要なスケーラビリティがないためです。

おそらく医療画像を保存するための最良のソリューション


医療機関で医療画像やその他の非構造化コンテンツを保存するための主なソリューションは、高い拡張性とギガバイトあたりの低コストのストレージを備えたファイルおよびオブジェクトストレージシステムの使用です。このストレージセグメントのリーダーの1つであるScalityは、ソフトウェア定義ストレージであるScality RINGを宣伝しています。 Scality RINGの最初のバージョンは2010年にリリースされました。これは、ピアツーピア通信と標準のx86サーバーに展開されるシェアードナッシング分散アーキテクチャを使用するスケールアウトソリューションです。 Scality RINGは、S3およびSwiftオブジェクトデータアクセスプロトコル、シンプルなHTTP API、ファイルアクセスをサポートしています。昨年、Scalityはヘルスケアにおけるシステムの設置数を2倍に増やすことに成功しました。

Scality RINGソフトウェアは、3つのストレージノードの最小構成で構成されるクラスターに展開され、一連のインテリジェントデータアクセスサービス、およびデータ保護とシステム管理を実装します。上位レベルにはスケーラブルなデータアクセスサービス（コネクタ）があり、SMB、NFS、S3プロトコルを介してアプリケーションにデータを提供するほか、システムステータスを集中管理および監視するためのスーパーバイザを備えています。コネクタは通常、ストレージノードに直接インストールされますが、専用サーバーに展開することもできます。

中間レベルのストレージScality RINGは、いくつかのデータ保護メカニズム、システムの自己修復プロセス、システム管理および監視サービスを備えた分散仮想ファイルシステムです。スタックの最下位レベルは、仮想ストレージノードとI / Oデーモンが形成する分散ストレージレベルであり、物理ストレージサーバーとディスクインターフェイスを抽象化します。

ストレージ層の中心は、第2世代のピアツーピアルーティングプロトコルに基づくスケーラブルなKey-Value分散オブジェクトストレージです。ルーティングは、非常に多数のノードにわたる効率的な水平ストレージスケーリングと検索を提供します。これらのストレージサービスのソフトウェアは、ディスクサブシステムに必要な計算能力と容量を備えたすべてのサーバーに展開されます。 Scality RINGソフトウェアがデプロイされているサーバー（ノード）は、たとえば10/25/40/100ギガビットイーサネットを使用する標準のIPベースのネットワークファクトリーによって接続されます。

Scality RINGには、コネクタサーバー、メタデータを保存するための分散MESA DBMS、ストレージノード、I / Oデーモン、Webベースの管理ポータルなどのソフトウェアコンポーネントが含まれています。 MESAは、Scalityスケールアウトファイルシステム（SOFS）抽象化レベルで使用されるオブジェクトインデックスとメタデータ管理を提供します。

Scality RINGコネクタは、サーバーに保存されているデータへのアプリケーションアクセスを提供します。それらは、Representational State Transfer（REST）規格に基づくAmazon Web Services（AWS）S3オブジェクトや、NFS、SMB、FUSEファイルプロトコルなど、多くのデータアクセスプロトコルをサポートしています。 I / Oを水平方向にスケーリングしたり、多数のユーザーに並行してサービスを提供したりする必要がある場合は、単一のアプリケーションで複数のRINGコネクタを同時に使用してデータにアクセスできます。

ストレージノードは、キー（キースペース）RINGの分散キーハッシュの割り当てられた部分に関連付けられたオブジェクトを担当する仮想プロセスです。ストレージノードのデーモン（いわゆるbizoid）は、低レベルのローカルファイルシステムのディスクに格納されたデータの不変性を保証します。 6つの仮想ストレージノードが1つの物理サーバー（ホスト）に展開されます。各biziodは、特定の物理ディスク上の入出力操作を制御し、オブジェクトキーとこのディスク上の特定のオブジェクトのアドレスとの対応を維持する低レベルプロセスのインスタンスです。

オブジェクトストレージの高可用性（最大14ナイン）を確保するために、Scality RINGは、従来のRAIDテクノロジーの代わりに、ローカルおよび地理的に分散されたレプリケーションとイレージャーコーディングを含む、分散システム用に最適化されたさまざまなデータ保護メカニズムを使用します。 1つのコネクタでの消去コード。小さなオブジェクト（最大サイズ60 Kバイト）を格納する場合、レプリケーションはより費用対効果の高い保護ソリューションであり、大きなオブジェクトの場合-大きなデータセットを複製する必要がない消去コーディングです。複製中、サービスクラス（CoS）サービスクラスの6つのレベルが0から5まで使用されます。これは、オブジェクトの3-5レプリカの保存に対応し、すべてのレプリカは異なるディスクに保存されます。

イレージャーコーディングを使用する場合、オブジェクトのいくつかのレプリカを格納する代わりに、それを「データチャンク」に分割し、それをパリティコード（パリティチャンク）とともに書き込む、リードソロモンエラー訂正メカニズムを使用します。これらのピースはRINGノード間で分散されており、1つ以上のノードに障害が発生した場合、それらからデータを復元できます。また、RINGの高いフォールトトレランスは、メイン（「マスター」）ノードがなく、システム全体の障害につながる可能性のある非共有アーキテクチャによって保証されます。

保存された情報をさらに保護するために、ScalityはVEEAM、Commvault、Microfocus Data Protector、Cloudera、MAPRおよびWEKA.IO製品との互換性、および医療画像のアーカイブとしての医療におけるScality RINGの使用についてクラウドストレージを認定しているため、さまざまなバックアップソフトウェアパッケージを使用できます。 PACSシステムとの互換性に関する認証を提供します。

Healthcare Scality Ringのケーススタディ


現在、フランスだけでも、12を超える最大の病院が医療画像のアーカイブを保存するために400 TBから6 PBのScality Ringストレージを使用しています。たとえば、HPEとScalityの大規模な共同インストールは、4つのマルセイユ病院と3400のベッドを組み合わせ、フランスで3番目に大きいアシスタンスパブリケオピトードマルセイユ（AP-HM）で実装されています。 AP-HM病院では、2,000人の医師と8,500人のその他の医療スタッフが働いています。

2011年まで、AP-HMはEMC Centeraを使用してPACSイメージを保存していました。この時点で、医用画像の合計容量は60 TBでしたが、データを保護するために複製されたため、120 TBの容量を占めていました。毎年、病院のPACSはさらに20 TBの新しい画像を生成しました。 2011年にAP-HMはCenteraをNASシステムに置き換え、2017年までにイメージの量は320 TBに増加し、成長率は年間40 TBに増加しました。 NASの保証がなくなり、データボリュームが急速に増加したためにこのストレージのストレージ容量が十分ではなくなったため、AP-HM病院の管理者は再び交換することを決定しました。

新しいストレージシステムを選択するとき、病院で使用されるすべてのアプリケーションとの互換性を確保する必要がありました。これには、CIFSおよびNFSファイルプロトコルのサポート、数ペタバイトにわたるスケーリング、信頼性の高い保護およびデータセキュリティが含まれます。AP-HMはHPEとScality RINGを選択し、6つのHPE Apollo 4510ストレージサーバーと、METAメタデータDBMSを格納する2つのHPE ProLiant DL360サーバーから3つのデータセンターに分散するRINGクラスターを構築しました。主なアプリケーションは、GEヘルスケアのCarestream PACSシステムです。このシステムは、放射線学的研究の結果として得られた画像とゲノミクスアーカイブを記録します。医療画像およびその他のデータは、Commvaultソフトウェアを使用してバックアップされます。
