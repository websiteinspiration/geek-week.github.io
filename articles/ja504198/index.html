アニメーションの形状


アニメーションの一般的な仕組み


したがって、アニメーションの主なアイデアは、特定の個別の変化を連続的なプロセスに変換することです。たとえば、円の半径は100単位でしたが、50単位になりました。アニメーションがなければ、変化はアニメーションで瞬時に、スムーズに起こります。使い方？とても簡単です。スムーズな変更のために、「それは...となった」セグメント内のいくつかの値を補間する必要があります。半径の場合、98単位、95単位、90単位... 53単位、最後に50単位の半径を持ついくつかの中間円を描く必要があります。 SwiftUIはこれを簡単かつ自然に行うことができます。この変更を実行するコードをwithAnimation {...}でラップするだけです。それは魔法のように見えます...「hello world」よりも少し複雑なものを実装するまでは。

例に移りましょう。アニメーションの最も単純で理解しやすいオブジェクトは、フォームのアニメーションと見なされます。 SwiftUIの形状（形状フォームプロトコルに準拠した構造を引き続き呼び出します）は、これらの境界に適合するパラメーターを持つ構造です。それら。これは、関数本体（rect：CGRect）-> Pathを持つ構造です。ランタイムがこのフォームを描画するために必要なのは、必要なサイズ（関数パラメーターとして指定されたCGRectの四角形）に対して、アウトラインを要求することです（関数の結果はPathタイプのオブジェクト、実際はベジェ曲線です）。

形状は保存された構造です。これを初期化することにより、アウトラインを描画するために必要なすべてをパラメーターに格納します。このフォームの選択のサイズは変わる可能性があります。必要なのは、新しいCGRectフレームの新しいPath値を取得することだけです。

Circleのコピーを次のように作成しました。

struct CustomCircle: Shape{
    public func path(in rect: CGRect) -> Path{
        let radius = min(rect.width, rect.height) / 2
        let center = CGPoint(x: rect.width / 2, y: rect.height / 2)
        return Path(){path in
            if rect.width > rect.height{
                path.move(to: CGPoint(x: center.x, y: 0))
                let startAngle = Angle(degrees: 270)
                path.addArc(center: center, radius: radius, startAngle: startAngle, endAngle:  startAngle + Angle(degrees: 360), clockwise: false)
            }else{
                path.move(to: CGPoint(x: 0, y: center.y))
                let startAngle = Angle(degrees: 0)
                path.addArc(center: center, radius: radius, startAngle: startAngle, endAngle:  startAngle + Angle(degrees: 360), clockwise: false)
            }
            path.closeSubpath()
        }
    }
}

円の半径は、割り当てられた画面領域の境界線の幅と高さの半分の半分として計算されます。幅が高さよりも大きい場合は、上部境界線の中央（注1）から始め、時計回りの方向に丸全体を描き（注2）、これでアウトラインを閉じます。高さが幅よりも大きい場合は、右の境界線の中央から開始し、時計回りに完全な円を描き、輪郭を閉じます。

AnimatableModifier


ビュー内の変更をアニメーション化する方法


しかし、直接ビューに行きましょう。コンテナー内の要素の位置をアニメーション化したいとします。この例では、緑色で幅が10単位の単純な長方形になります。水平位置をアニメーション化します。

    @State var position: CGFloat = 0
    var body: some View{
        VStack{
            ZStack{

    public var animatableData: CGFloat {

次に、円の正しいアニメーションの場合のように、計算位置が単に0になり、中間ログがないことがわかります。どうして？

前の例のように、修飾子の中にあります。今回は、.offset（）修飾子が新しいインデント値を取得し、変更自体をアニメーション化します。それら。実際、アニメーション化しようとしたのは位置パラメータの変化ではなく、そこから派生した.offset（）修飾子のインデントの水平方向の変化です。この場合、これは無害な置換であり、結果は同じです。しかし、彼らが来たので、より深く掘り下げましょう。入力で位置（0から1まで）を受け取り、使用可能な領域のサイズを受け取り、インデントを計算する独自の修飾子を作成しましょう。

struct BorderPosition: ViewModifier{
    var position: CGFloat
    func body(content: Content) -> some View {
        GeometryReader{geometry in
            content
            .offset(x: self.getXOffset(inSize: geometry.size), y: 0)
            .animation(nil)
        }
    }
    func getXOffset(inSize: CGSize) -> CGFloat{
        let offset = -inSize.width / 2 + inSize.width * position
        print("at position  \(position) offset is \(offset)")
        return offset
    }
}

extension View{
    func borderIn(position: CGFloat) -> some View{
        self.modifier(BorderPosition(position: position))
    }
}

適切なアプローチは、修正をアニメーション化することです


Viewにいくつかの変更をアニメーション化するように依頼したときのアプローチ自体は正しくありませんでした。ビューの場合、フォームの場合と同じ方法を使用できません。代わりに、アニメーションを各モディファイヤに埋め込む必要があります。ほとんどの組み込みモディファイアは、すぐに使えるアニメーションをすでにサポートしています。独自のモディファイアのアニメーションが必要な場合は、ViewModifierの代わりにAnimatableModifierプロトコルを使用できます。また、上記で行ったように、形状の変化をアニメーション化するときと同じことを実装できます。

            return position
        }
        set {
            position = newValue
            print("setting position: \(position) at time \(Date().timeIntervalSince(startDate))")
        }
    }
    func body(content: Content) -> some View {
...
    }
    ...
}


今、すべてが正しいです。コンソール内のメッセージは、アニメーションが実際に機能し、モディファイヤ内の.animation（nil）がまったく干渉しないことを理解するのに役立ちます。しかし、それがどのように機能するかを正確に理解しましょう。

まず、修飾子とは何かを理解する必要があります。


ここにビューがあります。前の部分で述べたように、これは、パラメーターとアセンブリー命令が格納された構造です。この命令には、概して、一連のアクションは含まれていません。これは、非宣言的なスタイルで記述する通常のコードですが、単純なリストです。他のビュー、それらに適用された修飾子、およびそれらが含まれているコンテナが一覧表示されます。コンテナーにはまだ興味はありませんが、修飾子についてもっと話しましょう。

モディファイアは、格納されたパラメータとビュー処理命令を持つ構造体です。これは実際にはビューと同じ命令です。他の修飾子を使用したり、コンテナーを使用したりできます（たとえば、GeometryReaderを少し高く使用しました）。さらに他のビューも使用できます。しかし、入ってくるコンテンツしかなく、この命令を使用して何らかの方法で変更する必要があります。修飾子パラメーターは命令の一部です。しかし、最も興味深いのは、それらが保管されることです。

前回の記事では、命令自体は保存されず、ビューが更新された後に毎回スローされると述べました。すべてがそうですが、ニュアンスがあります。この命令の作業の結果として、先に述べたように、私たちはかなりの写真を得ることができませんでした-それは単純化でした。この命令の操作後、修飾子は消えません。親ビューが存在する間、それらはそのままです。

struct MyAnimatableVector: VectorArithmetic{
    static func - (lhs: MyAnimatableVector, rhs: MyAnimatableVector) -> MyAnimatableVector {
        MyAnimatableVector(position: lhs.position - rhs.position, height: lhs.height - rhs.height)
    }
    
    static func + (lhs: MyAnimatableVector, rhs: MyAnimatableVector) -> MyAnimatableVector {
        MyAnimatableVector(position: lhs.position + rhs.position, height: lhs.height + rhs.height)
    }
    
    mutating func scale(by rhs: Double) {
        self.position = self.position * CGFloat(rhs)
        self.height = self.height * CGFloat(rhs)
    }
    
    var magnitudeSquared: Double{
         Double(self.position * self.position) + Double(self.height * self.height)
    }
    
    static var zero: MyAnimatableVector{
        MyAnimatableVector(position: 0, height: 0)
    }
    
    var position: CGFloat
    var height: CGFloat
}

  • 明らかに+と-が必要な理由だと思います。
  • スケールはスケーリングの関数です。「それがあった-となった」という差を取り、アニメーションの現在のステージ（0から1）を掛けます。「それは+差分*（1-ステージ）になりました」と、animatableDataでパームオフする必要がある現在の値があります。
  • アニメーションに値が使用される新しいオブジェクトを初期化するには、おそらくゼロが必要です。アニメーションは最初は.zeroを使用していますが、正確な方法がわかりませんでした。しかし、これは重要だとは思いません。
  • itudeSquaredは、与えられたベクトルとそれ自体のスカラー積です。2次元空間の場合、これは2乗されたベクトルの長さを意味します。これはおそらく、2つのオブジェクトを要素ごとではなく、全体として比較するために使用されます。アニメーションには使われていないようです。

一般的に言えば、「-=」「+ =」関数もプロトコルサポートに含まれていますが、構造については、この形式で自動的に生成できます。

    static func -= (lhs: inout MyAnimatableVector, rhs: MyAnimatableVector) {
        lhs = lhs - rhs
    }

    static func += (lhs: inout MyAnimatableVector, rhs: MyAnimatableVector) {
        lhs = lhs + rhs
    }

なぜAnimatableDataは構造体にしかできない


もう1つポイントがあります。クラスをAnimatableDataオブジェクトとして使用することはできません。正式には、対応するプロトコルに必要なすべてのメソッドをクラスに記述することができますが、これはうまくいきません。その理由はここにあります。ご存じのとおり、クラスは参照データ型であり、構造は値ベースのデータ型です。別の変数に基づいて1つの変数を作成する場合、クラスの場合はこのオブジェクトへのリンクをコピーし、構造体の場合は既存の変数の値に基づいて新しいオブジェクトを作成します。この違いを示す小さな例を次に示します。

    struct TestStruct{
        var value: CGFloat
        mutating func scaled(by: CGFloat){
            self.value = self.value * by
        }
    }
    class TestClass{
        var value: CGFloat
        func scaled(by: CGFloat){
             self.value = self.value * by
        }
        init(value: CGFloat){
            self.value = value
        }
    }
        var stA = TestStruct(value: 5)
        var stB = stA
        stB.scaled(by: 2)
        print("structs: a = \(stA.value), b = \(stB.value))") //structs: a = 5.0, b = 10.0)
        var clA = TestClass(value: 5)
        var clB = clA
        clB.scaled(by: 2)
        print("classes: a = \(clA.value), b = \(clB.value))") //classes: a = 10.0, b = 10.0)

アニメーションでもまったく同じことが起こります。 「was」と「became」の違いを表すAnimatableDataオブジェクトがあります。画面に反映するには、この違いの一部を計算する必要があります。これを行うには、この差をコピーして、アニメーションの現在のステージを表す数を掛けます。構造の場合、これは違い自体には影響しませんが、クラスの場合には影響します。私たちが描く最初のフレームは「was」状態です。これを行うには、Steel + Difference * Current Stage-Differenceを計算する必要があります。クラスの場合、最初のフレームで差に0を掛けてゼロにし、その後のすべてのフレームは差= 0になるように描画されます。アニメーションは正しく描画されているように見えますが、実際には、アニメーションがないかのように、ある状態から別の状態に瞬時に移行しています。

おそらく、乗算結果用の新しいメモリアドレスを作成するある種の低レベルコードを記述できますが、なぜですか？あなただけの構造を使用することができます-それらはそのために作成されます。

SwiftUIが中間値を計算する方法を正確に理解したい人のために、どの操作で、どの時点で、メッセージがプロジェクトのコンソールにプッシュされます。さらに、アニメーション内でリソースを大量に消費する計算をシミュレートするために、0.1秒のスリープを挿入しました。楽しんでください:)

画面アニメーション：.transition（）


ここまで、修飾子またはフォームに渡される値の変化をアニメーション化することについて説明しました。これらはかなり強力なツールです。しかし、アニメーションを使用する別のツールもあります。これは、ビューの外観と非表示のアニメーションです。

前回の記事では、宣言型のif-elseでは、これが実行時のコードのフローを制御するのではなく、シュレーディンガーのビューであるという事実について話しました。これは同時に2つのビューを含むコンテナであり、特定の条件に従って表示するビューを決定します。 elseブロックを見逃した場合、2番目のビューの代わりにEmptyViewが表示されます。

2つのビュー間の切り替えもアニメーション化できます。これを行うには、.transition（）修飾子を使用します。

struct TransitionView: View {
    let views: [AnyView] = [AnyView(CustomCircleTestView()), AnyView(SimpleBorderMove())]
    @State var currentViewInd = 0
    var body: some View {
        VStack{
            Spacer()
            ZStack{
                ForEach(views.indices, id: \.self){(ind: Int) in
                    Group{
                        if ind == self.currentViewInd{
                            self.views[ind]
                        }
                    }
                }
            }
            HStack{
                ForEach(views.indices, id: \.self){(ind: Int) in
                    RoundedRectangle(cornerRadius: 10)
                        .fill(ind == self.currentViewInd ? Color.green : Color.gray)
                        .overlay(
                            Text("\(ind + Int(1))"))
                        .onTapGesture{
                            withAnimation{
                                self.currentViewInd = ind
                            }
                    }
                }
            }
                .frame(height: 50)
            Spacer()
        }
    }
}

それがどのように機能するか見てみましょう。まず、事前に、親ビューの初期化段階でも、いくつかのビューを作成して配列に配置しました。配列の要素は同じ型でなければならないため、配列はAnyView型です。それ以外の場合、要素はForEachで使用できません。前の記事の不透明な結果タイプ、覚えていますか？

次に、この配列のインデックスの列挙を規定し、それらのそれぞれについて、このインデックスによってビューを表示します。 ForEachを使用するには、SwiftUIがコレクションのコンテンツを反復できるように、各要素に内部識別子を割り当てる必要があるため、これを強制的に実行し、Viewをすぐに反復することはしません。別の方法として、各ビューでプロキシ識別子を作成する必要がありますが、なぜインデックスを使用できるのでしょうか。

コレクションの各ビューを条件にラップし、アクティブな場合にのみ表示します。ただし、if-elseコンストラクトはここに存在できないため、コンパイラーはそれをフローの制御のために取得します。そのため、これらすべてをグループで囲み、コンパイラーがビューとは何かを正確に理解できるようにします。 <View1、View2>。

現在、currentViewIndの値を変更すると、SwiftUIは前のアクティブビューを非表示にし、現在のビューを表示します。アプリケーション内のこのナビゲーションはどうですか？

                ForEach(views.indices, id: \.self){(ind: Int) in
                    Group{
                        if ind == self.currentViewInd{
                            self.views[ind]

ビューは同じアニメーションで表示および非表示になりますが、非表示になるのは逆の順序でスクロールするだけです。実際、ビューの表示と非表示の両方に個別にアニメーションを割り当てることができます。これには非対称遷移が使用されます。

                    Group{
                        if ind == self.currentViewInd{
                            self.views[ind]
                               .transition(.asymmetric(
                                    insertion: insertion: AnyTransition.scale(scale: 0.1, anchor: .leading).combined(with: .opacity),
                                    removal: .move(edge: .trailing)))
                        }
                    }

extension AnyTransition {
    static func spinIn(anchor: UnitPoint) -> AnyTransition {
        .modifier(
            active: SpinTransitionModifier(angle: -90, anchor: anchor),
            identity: SpinTransitionModifier(angle: 0, anchor: anchor))
    }
    static func spinOut(anchor: UnitPoint) -> AnyTransition {
        .modifier(
            active: SpinTransitionModifier(angle: 90, anchor: anchor),
            identity: SpinTransitionModifier(angle: 0, anchor: anchor))
    }
}


まず、通常のモディファイヤを記述します。ここでは、特定の数値を転送します。回転角度（度単位）と、この回転が発生する基準となる点です。次に、2つの関数でAnyTransition型を拡張します。 1つだったかもしれませんが、私にはもっと便利に思えました。ビュー自体で回転の角度を直接制御するよりも、それぞれに話す名前を割り当てる方が簡単だと思いました。

AnyTransitionタイプには、2つの修飾子を渡す静的修飾子メソッドがあり、1つの状態から別の状態へのスムーズな遷移を記述するAnyTransitionオブジェクトを取得します。 identityは、アニメーション化されたビューの通常の状態修飾子です。アクティブは、ビューの外観のアニメーションの開始状態、または非表示のアニメーションの終了状態です。セグメントのもう一方の端、内挿される状態。

つまり、spinInは、指定した点を中心に時計回りに回転することにより、画面（またはビューに割り当てられたスペース）の外側からビューを表示するために使用することを意味します。 spinOutは、ビューが同じように消え、同じ点を中心に時計回りに回転することを意味します。

私の考えでは、ビューの表示と非表示に同じポイントを使用すると、このポイントを中心に画面全体を回転させる効果が得られます。

すべてのアニメーションは、標準的なモディファイ機構に基づいています。完全にカスタムの修飾子を作成する場合は、TwoParameterBorderで行ったように、AnimatableModifierプロトコルの要件を実装するか、独自のデフォルトのアニメーションを提供する組み込みの修飾子を使用する必要があります。この場合、SpinTransitionModifierモディファイア内の組み込み.rotationEffect（）アニメーションに依存しました。

.transition（）修飾子は、アニメーションの開始点と終了点として何を考慮すべきかを明確にするだけです。アニメーションを開始する前にAnimatableData状態を要求する必要がある場合は、現在の状態のAnimatableDataモディファイアを要求し、差を計算してから、1から0への減少をアニメーション化し、.transition（）は元のデータを変更します。ビューの状態に縛られることはなく、それに基づいていません。明示的に初期状態と最終状態を自分で指定し、それらからAnimatableDataを取得して、差を計算し、アニメーション化します。次に、アニメーションの最後に、現在のビューが前面に表示されます。
