モニタリングを書いたとき

私は自分の話を共有することにしました。たぶん誰かが、よく知られた問題に対するこの予算の解決策が役立つと思うでしょう。

私は若くて熱く、どこにエネルギーを注ぐべきかわからなかったので、少しフリーランスにすることにしました。私はすぐに評価を得ることができ、サーバーを継続的にサポートするように依頼してきたいくつかの常連客を見つけました。

私が最初に考えたのは、監視の必要性です。私はそれを賢い人として行うことを決めました。ホイールを再発明するのではなく、MuninやZabbixなどの既成のオプションを検討することにしました。しかし、特に最初に携帯電話から開いた場合は、Webバージョンには適切なインターネット接続が必要であることがすぐにわかりました。都会から離れて自然の中でリラックスすると、安定したつながりを得ることは困難です。したがって、コンソール監視オプションが選択されました。

Atopとトップのログリーダーであるatopsarは、コンソールの監視として非常に役立ちました。彼らはすでにhabrについて言及されており、頂上は分解されていましたが、アトプサーについてはほとんど何も言われていませんでした。

取り付け


非常にシンプルなインストール、3つのチームのみ。

＃セントース

yum install atop

＃Debian / Ubuntu

apt-get install atop


次に、監視操作を自分で構成するか、デフォルト設定を使用できます。

＃Debian / Ubuntu / Centos

/etc/default/atop

標準ファイル：

 #cat /etc/default/atop
INTERVAL=60                    #,       ,    10 
LOGPATH="/var/log/atop"        #    
OUTFILE="$LOGPATH/daily.log"   #

スタートアップに追加
＃Debian / Ubuntu / Centos

systemctl enable atop


＃Debian / Ubuntu / Centos デーモンとして実行します

systemctl start atop

1つのチームに
集中して怠惰な場合#Centos

yum install atop && systemctl enable atop && systemctl start atop

＃Debian / Ubuntu

apt-get install atop && systemctl enable atop && systemctl start atop

アトプサー


atopとともに、atopsarもインストールされます。これは、atopデーモンによって実行されるバイナリログの便利なコンソールアナライザです。もちろん、ログを上から読むこともできますが、長い時間間隔でキャプチャする必要がある場合は、これはあまり便利ではありません。

アトプサーの仕事に関する小さな教育プログラム。

atopsarがキーなしで起動すると、今日のログを開き、各コアの負荷とすべてのカーネルのidl文字列を個別に表示します。

キーは、私が使用する：

-Aは =出力ログからのすべての情報
-Cプロセッサコアの負荷に=表示情報が、デフォルトのキーである
-m = RAMとスワップの負荷
-d =ディスク活性
-O =トップ3負荷プロセスCPU
-G= RAMの負荷の上位3プロセス
-D =ディスクの負荷の上位3プロセス
-N =ネットワークの負荷の上位3プロセス
-r =過去数日間の負荷を確認する必要がある場合に読み取るログのパスを指定
-b =出力を開始する時間
-e = 出力を終了する必要がある時間
-M =行の重要度がマークされた最後に追加の列を作成します（+負荷がある、*は重要な負荷）

監視のおかげで、サーバーの不正な動作の理由を理解できますどんなときも。

お知らせ


そのため、負荷の監視がありますが、それでも問題をすばやく見つけて解決することはできません。問題についての通知が必要です。

私はサーバーを監視しているだけなので、常にそれを見ることができる場所に通知し、少なくとも何らかの形でそれに対応する必要があります。

最初はSMSでした-高速で信頼性が高く、無料です。しかしその後、携帯電話会社はゲートウェイを介して無料のSMS配布を隠蔽しました。
メール-長い間、配送に問題がある可能性があります。
メッセンジャー-電話に置く必要があります。ボットを作成する必要があります。

検索の結果、テレグラムメッセンジャーは、電話とデスクトップでのシンプルさと便利なアプリケーションのために選択されました。botfather

を使用してボットを作成しました。
サーバー（IDL、smartctなどl）の負荷を追跡するいくつかのスクリプトをサーバーに配置した後、「oom killer」などのエラーの存在、バックアップ作成時のエラー、および制御が必要なその他の操作。

スクリプトは、たとえばLAをチェックし、Load Averadgeがサーバー上のコアの数を超えたことを通知するなど、bashで書かれたかなり単純なものです。

if [ ${LA[0]} -gt 2000 ] || [ ${LA[1]} -gt 3000 ] || [ ${LA[2]} -gt 4000 ]
    then
        wget -O /dev/null "https://api.telegram.org/$bot_id:$bot_key/sendMessage?chat_id=$chat_id&text=  $ip LA $LAd"
        wget -O /dev/null "https://api.telegram.org/$bot_id:$bot_key/sendMessage?chat_id=$chat_id&text=`top -b -n 1 | grep Cpu`"
        wget -O /dev/null "https://api.telegram.org/$bot_id:$bot_key/sendMessage?chat_id=$chat_id&text= 5  `top -b -n 1 | grep -A 5 'PID USER' | tail -5`"
    fi

構文が単純であるため、多くのユースケースが提供されます（そして、少なくとも少しのプログラミング言語を知っている人なら誰でも記述/追加できます）。

唯一の警告-サーバーがロシアにある場合（サーバーにIPv6がない場合）、プロキシを使用する必要があります。これを行うには、スクリプトの最初で、接続文字列をプロキシに登録する必要があります。

export https_proxy=http://:@IP.:

これは終わりではありません


バックパックを後ろに置いて静かに山の中を歩き、文明から一休みします。その後、電話が誤って接続をキャッチし、サーバーで発生した問題について通知をスローします。何をすべきか？穏やかな気分が風に吹き飛ばされた。妻に電話してコマンドを指示しますか？はは

インターネットを利用できない状態で問題をすばやく解決する方法を考え出すことが急務でした。ここで私は再びメッセンジャー（＃telegrammivi）によって救われました。私はボットに、他の人を無視して、自分だけと通信するように教えました。これで、問題の通知と一緒に、もう少し多くのデータが得られます。これに従って、問題の原因が誰であるかを理解し、リモートで解決を試みることができます。単にボットにメッセージを書いて、このメッセージが消えるように電話を高く投げてください。そうすれば、ボットはあなたの仕事をしに行きました。したがって、たとえば、不要なプロセスを強制終了したり、デーモンを再起動したり、IPをブロックしたりすることができます。

ここでは、クライアントからの将来の必要なリクエストも転送しました。たとえば、ユーザーにパスワードを緊急にリセットする（「ああ、サーバーにアクセスできないため、数百万を失う！」）、目的のフォルダーにアクセスできるユーザーを検索し、サイトのオンとオフを切り替えるなどです。 。もちろん、私はボットの機能を常に変更しています。顧客の想像力が予期せずポップアップし、リクエストによって提供されない場合があるからです。しかし、基本的なものは満足しています。
