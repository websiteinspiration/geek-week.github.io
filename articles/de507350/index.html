PVS-Studio Analyzer: Identifizieren potenzieller Java SE API-Kompatibilitätsprobleme











2019 war ein sehr arbeitsreiches Jahr in Bezug auf Konferenzen. Unser Team könnte wochenlang auf Geschäftsreise gehen. Und wie Sie wissen, ist die Konferenz die Zeit, um Wissen auszutauschen. Neben der Tatsache, dass wir an unserem Stand Präsentationen gehalten und viele interessante Dinge erzählt haben, haben wir auch viel aus der Kommunikation mit Konferenzteilnehmern und Rednern gelernt. Auf der Herbstkonferenz Joker 2019 hat uns ein Bericht von Dalia Abo Sheasha „Migration über Java 8 hinaus“ dazu inspiriert, eine neue Diagnoseregel zu implementieren, mit der Sie Inkompatibilitäten in der Java SE-API zwischen verschiedenen Java-Versionen identifizieren können. Dies wird diskutiert.



Relevanz von Java SE-Kompatibilitätsproblemen



Java SE 14 wurde bereits veröffentlicht. Trotzdem verwenden viele Unternehmen weiterhin frühere Versionen von Java (Java SE 6, 7, 8, ...). Aufgrund der Tatsache, dass die Zeit läuft und Java ständig aktualisiert wird, wird das Kompatibilitätsproblem verschiedener Versionen der Java SE-API von Jahr zu Jahr relevanter.



Wenn neue Versionen von Java SE veröffentlicht werden, sind sie normalerweise abwärtskompatibel mit früheren Versionen. Das heißt, eine auf Java SE 8 entwickelte Anwendung sollte beispielsweise problemlos auf 11 Java-Versionen ausgeführt werden können. In der Praxis kann es jedoch in einigen Klassen und Methoden zu Inkompatibilitäten kommen. Diese Inkompatibilität liegt in der Tatsache, dass einige APIs Änderungen unterliegen: Sie werden gelöscht, im Verhalten geändert, als veraltet markiert und vieles mehr.



Dieses Problem wird sich nur verschlimmern, wenn Sie über eine Migration Ihres Projekts auf eine neuere Java SE nachdenken. Oder wenn der technische Support Ihrer Anwendung immer mehr Briefe erhält, dass sich die Anwendung falsch verhält oder überhaupt nicht starten kann.



Ich denke, das ist genug, um Aufmerksamkeit zu erregen!



Vorhandenes Toolkit



Es gibt keine einheitliche Lösung für den Wechsel von einer Java-Version zu einer anderen. Und wenn Ihre Anwendung das Ergebnis jahrelanger kontinuierlicher Entwicklung mit einer Vielzahl von nicht trivialen Lösungen ist, werden Sie beim Übergang zur neuesten Version von Java wahrscheinlich auf einen ziemlich zeitaufwändigen Prozess stoßen.



Dieser Prozess besteht darin, eine problematische oder potenziell problematische API zu identifizieren, die überprüft und durch eine alternative Lösung ersetzt werden muss. Dies kann die Geschäftslogik ernsthaft beeinträchtigen, deren Behebung und Prüfung viel Zeit in Anspruch nehmen kann. Wenn Sie nicht nur auf die neue Version von Java SE umsteigen müssen, sondern auch die Kompatibilität mit einer Reihe von Versionen sicherstellen möchten, wird diese Aufgabe viel komplizierter.



Nachdem ich im Internet nach dem Thema der Identifizierung von API-Inkompatibilitäten zwischen verschiedenen Java SEs gesucht habe, bin ich nur auf die Tools gestoßen, die mit dem JDK geliefert werden: javac, jdeps, jdeprscan.



In diesem Geschäft gab es kein Tool von Drittanbietern, außer dem, das ich beim Joker 2019-Bericht - Migration Toolkit for Application Binaries - treffen durfte .



Javac



Vergessen Sie beim Entwickeln einer Anwendung nicht die Compiler-Warnungen. Jeder hat eine andere Einstellung zu Warnungen:



  • Jemand beobachtet sie streng und etwas löst sofort Alarm. Vielleicht ein bisschen paranoid, aber es kann in Zukunft die technischen Risiken verringern.
  • Jemand sammelt sie in einer separaten Datei mit dem Gedanken, dass er sie unbedingt überprüfen wird, wie er befreit wird;
  • Jemand ignoriert Warnungen im Allgemeinen.


Die erste Option ist gut, da ein Teil der Probleme unverzüglich gelöst werden kann, z. B. ohne eine Methode oder Klasse, die als veraltet markiert ist. Die Verwendung einer solchen API ist kein Blockierungsproblem, aber Sie sollten darauf achten, da es bereits wahrscheinlich ist, dass sich Ihre Anwendung bei Verwendung der neu veröffentlichten Java-Version anders verhält oder sogar abstürzt.



jdeps



Wenn Sie bereits verwirrt sind über das Problem der Migration Ihrer Anwendung auf eine neuere Version von Java SE, hat jdeps es eilig, Ihnen zu helfen.



jdeps ist ein Befehlszeilentool, das eine statische Analyse der Abhängigkeiten Ihrer Anwendung und Bibliotheken durchführt und * .class- Dateien oder * .jar als Eingabe akzeptiert . Ab Java SE 8 wird es mit dem JDK geliefert.



Und hier interessiert uns die Tatsache, dass, wenn Sie dieses Tool mit der Option --jdk-internals ausführen , Sie erfahren , von welcher internen JDK-API Ihre Klasse abhängt. Dies ist ein sehr wichtiger Punkt, da die interne API nicht garantiert, dass sie sich in zukünftigen Versionen von Java nicht ändert.



Schauen wir uns ein Beispiel an. Angenommen, Sie haben Ihre Java 8-Anwendung schon lange entwickelt und waren noch nie verwirrt über die Kompatibilität der Java SE-API, die mit neueren Versionen verwendet wird. Es gab eine Frage zur Migration Ihrer Anwendung auf beispielsweise Java 11. In diesem Fall können Sie den Weg des minimalen Widerstands beschreiten und die Anwendung sofort mit Java 11 starten. Es ist jedoch ein Fehler aufgetreten, und es wird angenommen, dass Ihre Anwendung abstürzt. In diesem Fall können Sie jdeps von Java 11 aus ausführen, indem Sie die * .jar- Dateien Ihrer Anwendung festlegen .



Das Ergebnis ist ungefähr wie folgt:













Die Ausgabe zeigt, dass die Abhängigkeit sun.misc.BASE64Encoder in Java 11 entfernt wird. Der Absturz Ihrer Anwendung beim ersten Start in Java 11 ist höchstwahrscheinlich auf den Fehler java.lang.NoClassDefFoundError zurückzuführen . Jdeps informiert Sie nicht nur darüber, dass dies nicht der Fall ist, sondern bietet Ihnen auch eine alternative Abhängigkeit, die anstelle der aktuellen verwendet werden kann. In diesem Fall schlug er vor, die Remote-Abhängigkeit durch java.util.Base64 zu ersetzen .



Die zweite Warnung des Tools signalisiert, dass wir noch eine andere interne Abhängigkeit im Code verwenden, aber bisher ist sie korrekt. Wird es in zukünftigen Versionen von Java Änderungen an dieser API geben, dies muss jedoch berücksichtigt werden.



Natürlich kann jdeps mehr als nur die Verwendung interner JDK-Komponenten testen. Dies ist nicht im Umfang dieses Artikels enthalten, aber Sie können sich auf der offiziellen Seite selbst mit den Funktionen vertraut machen.



jdeprscan



Die Ziele von jdeprscan sind genau die gleichen wie die von jdeps, nämlich die Suche nach einer unerwünschten und problematischen API.



jdeprescan ist ein statisches Analysetool, das eine * .jar- Datei (oder eine andere Sammlung von * .class- Dateien) nach veralteten API-Elementen durchsucht . Die Verwendung einer veralteten API ist kein Blockierungsproblem, aber Sie sollten darauf achten. Ab Java SE 9 wird es mit dem JDK geliefert.



Angenommen, es besteht die Frage, ob die Anwendung auf Java 11 migriert werden soll. In diesem Fall führen Sie den Befehl aus 



jdeprscan --release 8 app.jar


Sie erhalten eine Liste der APIs, die für Java 8 veraltet sind, dh die API, die in zukünftigen Java-Versionen möglicherweise entfernt wird. Nachdem Sie alle Warnungen korrigiert haben, können Sie ausführen 

jdeprscan --release 11 app.jar


Dadurch wird eine Liste der APIs erstellt, die für Java 11 bereits veraltet sind. Auf diese Weise können Sie alle nicht empfohlenen APIs finden und (falls erforderlich) reparieren.



Migration Toolkit für Anwendungsbinärdateien



Mit diesem Tool können Sie Ihre Benutzeranwendung schnell auf potenzielle Probleme untersuchen, bevor Sie sie auf verschiedenen Servern (JBOSS, WebShere, Tomcat, WebLogic, ...) bereitstellen. Zusätzlich zu allen Funktionen können Sie mit dem Tool auch Unterschiede in der Java SE-API verschiedener Versionen identifizieren.



Lassen Sie uns einen kurzen Blick darauf werfen, worum es bei diesem Tool geht.



Ein schneller Start des Tools sieht folgendermaßen aus: 



java -jar binaryAppScanner.jar yourApp.jar --analyzeJavaSE 
--sourceJava=oracle8 --targetJava=java11 ....


Die Option analyseJavaSE impliziert die Verwendung verschiedener Parameter, die Sie durch Aufrufen der Hilfe kennenlernen können.



Nach dem Start der Analyse wird in Kürze ein Bericht in einem Webbrowser angezeigt:















Nicht alles passt auf den Bildschirm = (. Und obwohl Sie noch nicht versucht haben, dieses Tool auszuführen, werde ich es in Worten beschreiben.



Im Bericht sehen Sie das Auslösen von Regeln mit drei Schweregraden:



  • Regeln mit hohem Schweregrad (Remote-APIs, eine Verhaltensänderung, die die Anwendung funktionsunfähig macht und behoben werden muss);
  • Warnregeln (Ändern des API-Verhaltens, das zu Problemen führen kann und deren Erkundung es wert ist, untersucht zu werden);
  • Informationsregeln (unter Verwendung veralteter APIs, API-Verhalten, das das Verhalten geringfügig ändern kann, das Programm jedoch nicht beeinflusst).


Jede Operation kann erweitert werden und eine Beschreibung dessen, was und wie. Es gibt eine Empfehlung in Informationsnachrichten, sie sagen, jdeps zusätzlich neben uns ausführen. Sie sagen, wir konzentrieren uns auf die Migration der Anwendung, und JDEPS wird zusätzlich dazu beitragen, das Problem in den internen JDK-Paketen zu erkennen (zusätzlich zu dem, was sie finden).



Außerdem finden Sie am Ende des Berichts eine Liste der Regeln, nach denen die Analyse durchgeführt wurde.



Wenn Sie die Eclipse-IDE verwenden, können Sie das Plugin verwenden . Für eine detailliertere Studie sollten Sie deren Seite besuchen .



Implementierung in PVS-Studio



Nachdem wir die betrachteten Tools untersucht hatten, kamen wir zu dem Schluss, dass die Suche nach potenziellen Kompatibilitätsproblemen verschiedener Versionen der Java SE-API eine wertvolle Aufgabe für die statische Analyse ist.



Die besprochenen Tools erleichtern die Migration einer Anwendung oder die Suche nach einer API, die die Leistung Ihrer Anwendung in neueren Versionen von Java SE beeinträchtigen kann (wenn Sie die Anwendung nicht migrieren möchten). Da wir jedoch der Meinung sind, dass diese Tools ständig getrennt vom Entwicklungsprozess auf der Befehlszeile ausgeführt werden müssen, um Probleme zu identifizieren, haben wir festgestellt, dass dies nicht sehr praktisch ist. Ausgehend von der Tatsache, dass eine statische Analyse erforderlich ist, um problematischen oder potenziell problematischen Code in einem sehr frühen Entwicklungsstadium zu erkennen, haben wir die Diagnoseregel V6078 implementiert, die Sie über die "Problem" -API informieren.



Regel V6078 warnt Sie im Voraus, dass Ihr Code von bestimmten Java SE-API-Funktionen und -Klassen abhängig ist, was in zukünftigen Java-Versionen zu Schwierigkeiten führen kann. Und Sie werden gleich zu Beginn der Implementierung einer bestimmten Funktion nicht an diese API gebunden sein, wodurch die technischen Risiken in Zukunft verringert werden.



Die Diagnoseregel gibt in den folgenden Fällen Warnungen aus:



  • Wenn die Methode / Klasse / das Paket in der Zielversion von Java entfernt wird;
  • Wenn die Methode / Klasse / das Paket in der Zielversion von Java als veraltet markiert ist;
  • Wenn die Methode die Signatur geändert hat.


Mit dieser Regel können Sie derzeit die Kompatibilität von Oracle Java SE von Version 8 bis 14 analysieren. Damit eine Regel aktiv wird, muss sie konfiguriert werden.



IntelliJ IDEE



Im IntelliJ IDEA-Plugin müssen Sie die Regel aktivieren und die Parameter auf der Registerkarte Einstellungen> PVS-Studio> API-Kompatibilitätsproblemerkennung angeben:



  • Source Java SE - die Version von Java, auf der Ihre Anwendung entwickelt wurde;
  • Ziel Java SE - die Version von Java, mit der Sie die Kompatibilität der verwendeten API in Ihrer Anwendung überprüfen möchten (Source Java SE);
  • Pakete ausschließen - Pakete, die Sie von der Kompatibilitätsanalyse ausschließen möchten (Pakete werden mit einem Komma aufgelistet).












Plugin für gradle



Mit dem Gradle-Plugin müssen Sie die Einstellungen des Analysators in build.gardle konfigurieren: 



apply plugin: com.pvsstudio.PvsStudioGradlePlugin
pvsstudio {
    ....
    compatibility = true
    sourceJava = /*version*/  
    targetJava = /*version*/
    excludePackages = [/*pack1, pack2, ...*/]
}


Plugin für Maven



Mit dem Maven-Plugin müssen Sie die Einstellungen des Analysators in pom.xml konfigurieren: 



<build>
  <plugins>
    <plugin>
      <groupId>com.pvsstudio</groupId>
      <artifactId>pvsstudio-maven-plugin</artifactId>
      ....
      <configuration>
        <analyzer>
          ....
          <compatibility>true</compatibility>
          <sourceJava>/*version*/</sourceJava>
          <targetJava>/*version*/</targetJava>
          <excludePackages>/*pack1, pack2, ...*/</excludePackages>        
        </analyzer>
      </configuration>
    </plugin>
  </plugins>
</build>


Den Kernel direkt verwenden



Wenn Sie den Analysator direkt über die Befehlszeile verwenden, müssen Sie die folgenden Parameter verwenden, um die Kompatibilitätsanalyse der ausgewählten Java SE-APIs zu aktivieren: 



java -jar pvs-studio.jar /*other options*/ --compatibility 
--source-java /*version*/ --target-java /*version*/ 
--exclude-packages /*pack1 pack2 ... */


Analysatorbetrieb



Nehmen wir an, wir entwickeln eine Anwendung, die auf Java SE 8 basiert, und wir haben eine Klasse mit den folgenden Inhalten: 



/* imports */
import java.util.jar.Pack200;
public class SomeClass
{
  /* code */
  public static void someFunction(Pack200.Packer packer, ...)
  {
    /* code */
    packer.addPropertyChangeListener(evt -> {/* code */});
    /* code */
  }
}


Wenn Sie eine statische Analyse mit verschiedenen Einstellungen für die Diagnoseregel ausführen, sehen Sie das folgende Bild:



  • Quell-Java SE-8, Ziel-Java-SE-9
    • Die Methode 'addPropertyChangeListener' wird entfernt.


  • Quell-Java SE-8, Ziel-Java SE-11
    • Die Methode 'addPropertyChangeListener' wird entfernt.
    • Die Klasse 'Pack200' wird als veraltet markiert.


  • Quell-Java SE-8, Ziel-Java-SE-14
    • Die 'Pack200'-Klasse wird entfernt.




Zunächst wurde in Java SE 9 die Methode 'addPropertyChangeListener' in der Klasse 'Pack200.Packer' entfernt. In Version 11 wurde die Tatsache hinzugefügt, dass die Klasse 'Pack200' als veraltet markiert wurde. In Version 14 wurde diese Klasse vollständig gelöscht.



Wenn Sie die Anwendung in Java 11 starten, erhalten Sie daher 'java.lang.NoSuchMethodError' und, wenn Sie in Java 14 ausführen, 'java.lang.NoClassDefFoundError'.



Wenn Sie diese Informationen kennen, werden Sie bei der Entwicklung Ihrer Anwendung alternative Lösungen für die Aufgabe in Betracht ziehen.



Ideen für die weitere Entwicklung



Während der Implementierung der Diagnoseregel entstanden Ideen zur Erweiterung:



  • Aufgrund der Tatsache, dass jdeprscan und jdeps nicht vor der Verwendung von Reflection für den Zugriff auf die gekapselte API warnen können, ist es sinnvoll, die Regel so zu ändern, dass versucht wird, herauszufinden, welche Art von API sie verwenden möchten. Das Ergebnis ist vielleicht nicht perfekt, aber warum nicht ?!
  • Es gibt eine Vielzahl von JDK-Implementierungen (von Oracle, IBM, Red Hat, ...). Sie sind in der Regel miteinander kompatibel. Aber wie stark unterscheidet sich die interne JDK-API? Schließlich können Entwickler daran gebunden sein, was zu potenziellen Problemen bei der Migration einer Anwendung von einem JDK zu einem anderen führen kann.


Dies sind alles Forschungsfragen, und was sich am Ende herausstellen wird, wird die Zeit zeigen =) Wenn Sie interessante Szenarien für diese Diagnose kennen und diese in PVS-Studio sehen möchten, schreiben Sie uns .



Fazit



Die Suche nach potenziellen Inkompatibilitätsfehlern entspricht voll und ganz unserer Ideologie - mit PVS-Studio können Fehler in den frühen Phasen des Code-Schreibens gefunden und behoben werden. Wie bei einem Tippfehler kann ein Aufruf einer bestimmten Funktion jederzeit in den Code eingegeben werden. Daher ist das regelmäßige Ausführen von PVS-Studio in einem Projekt jetzt doppelt so nützlich. V6078



Diagnostik ist im Gerät zur Verfügung ab Version 7.08 . Sie können den Analysator für Ihr Projekt auf der Download- Seite herunterladen und testen .
