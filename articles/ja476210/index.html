公開鍵標準に基づく暗号ワークステーション。PKCS＃11トークンの構成

もう一度、cryptoarmpkcsユーティリティの機能を確認したところ、主に暗号トークン/スマートカードPKCS＃11で機能しているため、その構成に組み込みの機能がないことがわかりました。トークンの初期化、PINコードの設定などについて話している。そして、この機能を追加することが決定されました。TclPKCS11パッケージの機能を拡張するために最初に必要なことは、ライブラリがCで書かれていることです。

TclPKCS11の新機能


パッケージに3つの新機能が追加されました。

::pki::pkcs11::inittoken <handle> <slotId> <SO-pin> <label for token>

::pki::pkcs11::inituserpin <handle> <slotId> <SO-pin> <USER-pin>

::pki::pkcs11::setpin <handle> <slotId> <so | user> <oldpin> <newpin>

最初の関数:: pki :: pkcs11 :: inittokenは、トークンを初期化することです。この関数が有効なトークンに適用されると、その上のすべてのオブジェクトが破棄されることに注意してください。

2番目の関数:: pki :: pkcs11 :: inituserpinは、ユーザーPIN（USER PIN）を初期化するためのものです。

ユーザーPINコードの初期初期化後、すぐにPINコードに変更する必要があります。これは将来使用されるため、SO-PINのように、覗き見しないようにしてください。セキュリティ上の理由から、トークンは、原則として、最初のPINコードを使用してトークンにログインすることを許可していません。

PINコードを変更するには、3番目の関数、つまり:: pki :: pkcs11 :: setpinを使用します。この機能により、ユーザーPINとSO-PINの両方を変更できます。

トークンは、原則として、元のSO-PIN（デフォルト）の復元を許可しないことにも注意してください。ただし、トークンを再初期化すると、再びデフォルトのSO-PINが取得されます。TclPKCS11パッケージの実装はgithubで見ることができます。

これで、新しい関数を備えたTclPKCS11パッケージが手元にあるため、これらの関数のGUIを実装することは難しくありませんでした。

PKI / PKIの今後のイノベーション


2019年11月7日、州下院は法律「電子署名について」の最初の改正を採択しました。電子署名の発行に関する規則が変更されるため、この修正はすべての組織および個々の起業家に適用されます。

私がすべてを正しく理解した場合、法人および個人の起業家は、連邦税務局およびロシア連邦中央銀行の金融機関でのみ資格証明書を受け取ることができます。そして、これらの証明書のみを使用して、
拡張された修飾電子署名（Cades-XLT1）を付けることができます。

テレコムおよびマスコミュニケーション省によって認定された認証センター（CA）は、このような証明書を個人にのみ発行することができます。

さらに、CAの要件は大幅に変化します。資本は700万から5億から10億に増加する必要があり、保険負債の額は3000万から3億から5億ルーブルに増加し、認定期間は5年から3年に短縮されます。

法案はすでに激しい批判を巻き起こしている。主な不満は、変更により商用CAの大部分が閉鎖され、CA連邦税務サービスの能力を拡張および維持するために多額の予算支出が必要になること、および単一のインフラストラクチャにすべてのタイプのリスクが集中することであり、サイバー攻撃の便利なターゲットになります。

もう1つの修正があります。1つではなく2つの拡張された適格な署名によって同時に会社のドキュメントに署名する義務です。組織の責任者は、法人の署名と個人としての個人のデジタル署名を文書に添付する必要があります。
