Azure Cosmos DBの利点と落とし穴

今日の多くのデータベースは、サポートの複雑さとコストを最小限に抑えながら、高いパフォーマンス、スケーラビリティ、および可用性を確保するためにあらゆることを行うよう努めています。 Azure Cosmos DBは、これらの品質を簡単に提供できるDBMSの優れた例です。この記事では、一見すると明らかではないかもしれないと同時に、システムを設計するときに考慮されない場合、将来的に深刻な問題になる可能性のある制限とともに、その機能について説明します。

設計段階で考慮しなかった場合に大きな問題になる可能性のある制限。

Cosmos DBは、SQL、Cassandra、MongoDB、Gremlin、Tableなど、いくつかのAPIをサポートしています。ここで、SQLはDocumentDBと呼ばれていたドキュメント指向のAPIを指し、通常のリレーショナルデータベースとは大きく異なります。この記事は、SQL APIの経験に基づいています。ストレージインスタンスを作成するときにAPIのタイプが選択されていることに注意してください。異なるAPIを介して同じインスタンスのデータを処理することはできません。

Cosmos DBドキュメントモデルは、アイテムで構成されるコンテナーにデータを格納します。ドキュメントの前半では、それぞれコレクションとドキュメントと呼ばれていました。スケーリング、帯域幅、インデックス作成のすべての設定は、コンテナレベルで示されます（いくつかの例外はありますが、これについては後で説明します）。概して、データベースは、名前付きのコンテナーの結合です。

オンザフライのスケーリング


Cosmos DBを使用すると、各コンテナーのパフォーマンスメトリック（帯域幅）を個別に管理できます。

スループットは、1秒あたりの要求の単位で測定されます（1秒あたりの要求単位または省略されたRU /秒）。リクエストユニットのおおよその同等物は、その識別子によってサイズが1 KBの要素を読み取ると考えることができます。たとえば、毎秒500の読み取りと1キロバイト要素の100のレコードを作成できるコンテナに必要なスループットは、およそ500 * 1 + 100 * 5 = 1000 RUになります。ただし、一般的なケースでは、要求の複雑さと要素のサイズが大きく異なるため、必要なスループットを正確に計算することは実際には不可能です。

データベースによって実行されるすべての操作には、RUに関して独自の「価格」があります。コンテナに示されるスループットは、Cosmos DBが1秒あたりに「費やす」ことができる制限です。ベースは1秒以内にリクエストの「価格」の合計を追跡し、制限にすでに達している場合、1秒あたりの量が指定された制限未満の値に戻るまで、後続のリクエストは受け入れられません。

コンテナーの可能な最小スループットは400 RUであり、1か月あたり約25ドルかかります。 RUの増加に伴ってコストは直線的に増加します。800RUのコンテナのコストはすでに月額約50ドルです。

スループットはコンテナレベルで示され、追加の手順なしでいつでも変更できます。

データベースレベルで帯域幅を指定することが可能です。この場合、すべてのコンテナが同じ「容量」を使用します。

「オートパイロット」モード（現在プレビュー段階）もあり、データベースの負荷に応じて、宣言された帯域幅を自動的に増減できます。その欠点は、構成できる最小値と最大値が1:10の比率であること、つまりピーク時にメトリックが40000 RUに設定され、休止時には400 RUになるようにすることはできません。

宣言された帯域幅を超えた場合、Cosmos DBは新しい実行リクエストを受け付けず、特別なステータスHTTP 429（「リクエストレートが大きすぎます」）を返します。

基本レベルでの帯域幅の構成とコンテナーレベルでの構成を切り替えるには、データベースを再作成する必要があります。スムーズな移行はまだ提供されていません。

無限のスケーラビリティのためのパーティショニング


Cosmos DBには、コンテナーの2つのオプションがありました：パーティションあり（パーティション）となし（非パーティション）。パーティションのないコンテナは、最大スループットが10,​​000 RU、サイズが10ギガバイトに制限されていました。現時点では、すべてのコンテナにパーティションを設定できるため、それらを作成するときに、パーティションキーを指定する必要があります。

重要なポイントは、論理パーティションと物理パーティションの違いを理解することです。：logicalは、同じパーティションキー値を持つ要素のセットで構成されますが、物理パーティションは、複数の論理パーティションを処理できる物理インフラストラクチャのノードであるCosmos DBの「コンピューティングユニット」です。データの量とパーティションキーによるレコードの分散に応じて、システムは新しい物理パーティションを作成し、それらの間で論理パーティションを再分散できます。

コンテナーに割り当てられるすべてのスループットは、その物理パーティション間で均等に分散されます。最初に、帯域幅が1000 RUのコンテナに1つのセクションがあると想像してください。サイズが10 GBの制限に達すると、システムはそれを2つの部分に分割します。つまり、最初にこのコンテナーで実行されているすべての要求が毎秒1000 RUで自由に使用できた場合、同じパーティションに関連する要求はすでに500 RUの帯域幅によって制限されています。

パーティションの数に制限はありませんが、1つの物理パーティションの最大データ量は10 GBで、最大スループットは10,000 RUです。これらの制限に達する可能性が最小になるように、区分化キーを選択する必要があります。同じ論理パーティションを複数の物理パーティションに分割することはできません。

自動インデックス


代わりに、個々のフィールドまたはその組み合わせのインデックスを作成する、コスモスDBは、あなたが設定することができますインデックスポリシーのためのオブジェクト内のパスを。ポリシーは属性のセットです。インデックスに含める方法、除外する方法、使用するインデックスのタイプなどです。

興味深い点は、他のほとんどのDBMSとは異なり、Cosmos DBは従来のBツリーの代わりに逆インデックスを使用するため、複数の条件で検索する場合に非常に効果的であり、複合インデックスを作成する必要がありません。複数のフィールドを検索します。このトピックの詳細については、このリンクの記事を参照してください。

インデックス作成ポリシーはいつでも変更できます。

2つの索引付けモードが使用可能です：整合（整合）および遅延（遅延）。Lazyを使用すると、記録は高速になりますが、書き込みと読み取りの一貫性が損なわれます。これは、この場合、書き込み操作が完了した後にバックグラウンドでインデックスが作成されるためです。インデックスが更新されている間、クエリは無関係なデータを返す場合があります。

複合インデックスを作成して、複数のフィールドにまたがるORDER BY操作を高速化できます。それ以外の場合、複合インデックスは役に立ちません。

空間インデックスのサポートがあります。

デフォルトのコンテナで作成されたインデックスポリシー（「すべてのフィールドにインデックスを付ける」）は、多数のフィールドを持つアイテムを書き込むときに、RUを大量に消費する可能性があります。

変更の追跡


Cosmos DBでの変更の追跡は、Change Feedと呼ばれるメカニズムのおかげで可能です。ある時点以降に変更されたドキュメントを、変更された順に返します。この初期の瞬間は柔軟に制御できます。変更のフロー自体の初期化の瞬間、または固定タイムスタンプ、またはコンテナが作成された瞬間のいずれかです。

変更は非同期でインクリメンタルに処理でき、結果は1つ以上のコンシューマ間で分散して並列処理できます。これにより、さまざまな統合シナリオで非常に高い柔軟性が得られます。

変更フィードに低レベルAPIを使用する場合は、パーティションが最大サイズに達したときに発生する可能性があるパーティション分割を必ず考慮してください。

このメカニズムは削除を追跡しないため、要素の削除に応答できるように、ソフト削除（ソフト削除）を使用することをお勧めします。

ストアドプロシージャとトランザクション


Cosmos DBは、JavaScriptで記述されたストアドプロシージャとトリガーをサポートしています。

JavaScript I / Oは完全に非同期であり、Cosmos DBではasync / awaitがまだサポートされていないため、多くのコールバックを記述する必要がありますが、コードの可読性には大きく貢献しません。

ストアドプロシージャから追加のエラー情報を返す便利な方法はありません。問題を解決する唯一の方法は、エラーメッセージに情報を追加し、クライアント上のメッセージからこのデータを「カット」することです。

トランザクションは、.NET SDKバージョン3.4以降で使用可能なストアドプロシージャと関数またはトランザクションパッケージ内でのみ機能します。

1つのトランザクションに含めることができるのは、同じ論理パーティションのドキュメントのみです。したがって、異なるコンテナへの書き込み操作はトランザクションで実行できません。

1つのストアドプロシージャの実行時間は制限されており（5秒）、トランザクションの継続時間がこれを超えると、キャンセルされます。複数のサーバー呼び出しを通じて「長期間」のトランザクションを実装する方法はいくつかありますが、それらは原子性に違反しています。

クエリ実行


MicrosoftはドキュメントAPIを「SQL」と呼んでいますが、その言語はSQLのようであり、リレーショナルデータベースで見られるものとは多くの違いがあります。

APIには、「高価な」クエリの実行を防止するのに役立つパラメーターがあります。EnableCrossPartitionQuery、EnableScanInQuery（ドキュメントのFeedOptionsクラスを参照）。条件にパーティションキーの固定値が含まれていない場合、パーティション間クエリが受信されます。クエリ条件に非インデックスフィールドが含まれている場合、データセットスキャンが発生する可能性があります。両方のパラメーターをfalseに設定することは、RUの過剰な消費に対処する良い方法です。ただし、場合によっては、複数のパーティションでクエリを実行すると役立つことがあります。

GROUP BY句は既にサポートされています（2019年11月に追加されました）。

集約関数MINおよびMAXはインデックスを使用していないようです。

JOINキーワードは言語に存在しますが、ネストされたコレクションを「展開」するために使用されます。通常のSQLのように、1つのクエリで異なるコンテナの要素を接続すると失敗します。

Cosmos DBは厳密なデータスキームを意味しないため、条件で指定されたフィールドが一部の要素に存在しない場合があります。したがって、オプションのフィールドに条件を課すクエリには、IS_DEFINED関数によるそれらの存在のチェックも含める必要があります。NULLを確認するだけでは不十分な場合があります。

クエリ関連のヒントは、Microsoftが公開しているチートシートにあります。
