High performance garbage collection for C ++

We already wrote about garbage collection for JavaScript, about the DOM, and how all this is implemented and optimized in the V8 JS engine. True, Chromium is not only JavaScript. Most of the browser and the Blink rendering engine where V8 is built in are written in C ++. JavaScript can be used to work with the DOM, and changes are displayed on the screen using the rendering pipeline.



Since the graph of C ++ objects related to the DOM is closely related to JavaScript objects, a couple of years ago the Chromium development team began using a garbage collector called Oilpan to manage the memory in which these objects are stored .. Oilpan is a C ++ garbage collector designed to manage C ++ memory that can be connected to V8. Memory management is carried out using cross-component garbage collection technology . In the framework of this technology, the graph of related C ++ / JavaScript objects is considered as a single heap. This material is the first Oilpan publication. Here we will review the basic principles underlying this garbage collector, as well as the C ++ API Oilpan. We will look at some of the features supported by Oilpan, talk about how the work of various subsystems of the garbage collector works. Here we will analyze the process of competitively freeing memory occupied by objects.











The most interesting thing is that the Oilpan system is part of Blink, but now it is being transferred to V8, where it will be presented in the form of a library for garbage collection . The purpose of all this is to facilitate access to C ++ - garbage collection mechanisms for all those who embed the V8 engine in their platforms. In addition, the fact that Oilpan will become a library will allow the use of this system for absolutely all C ++ programmers interested in it.



General information



Oilpan uses a garbage collection system that uses the Mark and Sweep algorithm . This algorithm divides the garbage collection process into two phases. The first phase consists in examining the heap, and in marking “live” objects that cannot be deleted from memory. The second phase is the sweep of the heap memory, which is occupied by unnecessary ("dead") objects.



We already wrote about the mark phase of our implementation of the algorithm . If we recall the main ideas of that material, it turns out that the search for “living” objects can be considered as a bypass of the graph in which the objects are nodes, and the pointers with which the objects refer to each other are edges of the graph. Walking begins with root entitieswhich are objects about which it is known in advance that they are “living”. This, for example, is a global object and active functions.



In this regard, C ++ from JavaScript is no different. True, unlike JavaScript objects, C ++ objects are statically typed. As a result, they cannot change their own view during program execution. When working with C ++ objects using Oilpan, this fact is taken into account and a description of pointers to other objects (graph edges) is provided using the Visitor pattern. The basic pattern used to describe Oilpan objects looks like this: 



class LinkedNode final : public GarbageCollected<LinkedNode> {
 public:
  LinkedNode(LinkedNode* next, int value) : next_(next), value_(value) {}
  void Trace(Visitor* visitor) const {
    visitor->Trace(next_);
  }
 private:
  Member<LinkedNode> next_;
  int value_;
};

LinkedNode* CreateNodes() {
  LinkedNode* first_node = MakeGarbageCollected<LinkedNode>(nullptr, 1);
  LinkedNode* second_node = MakeGarbageCollected<LinkedNode>(first_node, 2);
  return second_node;
}


In this example, the Oilpan controls LinkedNode



what the class LinkedNode



is indicating as an inheritor GarbageCollected<LinkedNode>



. When the garbage collector processes an object, it finds pointers to other objects by calling the object's method Trace



. A type Member



is an intelligent pointer, which, from a syntactic point of view, is similar, for example, to std::shared_ptr



that provided by Oilpan and is used to maintain a uniform state when traversing a graph of objects while marking objects. All this allows Oilpan to know exactly where the signs with which this system works are located.



Anyone who carefully read the above code may have noticed (and maybe it scared him) that first_node



andsecond_node



stored on the stack as regular C ++ pointers. Oilpan does not use additional abstractions to work with the stack. The garbage collector, processing the root objects on the heap that it manages, relies solely on a conservative stack scan when looking for pointers. All this works by word-by-word sorting of the stack and thanks to the interpretation of words in the form of pointers to entities that are in a managed heap. This means that using Oilpan does not degrade performance when accessing objects on the stack. Instead, the load is transferred to the garbage collection stage when a conservative stack scan is performed. Oilpan is integrated into the rendering subsystem and is trying to delay the start of the garbage collection process until the system reaches a state where there is definitely nothing interesting on the stack.Since the web is based on events, and the code is executed by processing tasks in event loops, Oilpan has enough convenient moments to start garbage collection.



Oilpan is used in Blink, and this is a large code base written in C ++, which contains significant amounts of mature code. Due to this, Oilpan, among other things, has the following features:



  • Multiple inheritance using mixins and links to similar mixins (internal pointers).
  • Support for invoking garbage collection when executing constructors.
  • Maintaining objects from unmanaged memory in a "live" state using smart pointers Persistent



    , which are considered as root entities.
  • Collections that are sequential (for example - vector) and associative (for example - set and map) containers. Ability to compress the data underlying collections.
  • Weak links, weak functions, and ephemeral data structures .
  • Finalizers are methods that are executed before deleting individual objects from memory.


Clearing memory for C ++



We’ll talk about how objects are marked during Oilpan’s work another time. Now we will proceed from the assumption that the marking of objects has already been completed and Oilpan has already detected all reachable objects using the object method Trace



. After the tagging phase is completed, the corresponding bit is set for reachable objects.



Cleaning is the phase of Oilpan's work when dead objects are destroyed (those that could not be reached in the phase of marking objects). The memory that they occupy is either returned to the operating system, or becomes available for allocation to other objects. Below we will talk about how Oilpan performs memory cleaning, consider the features of this process and its limitations. In addition, we will talk about how the Oilpan memory cleaning mechanism manages to achieve high speeds when destroying dead objects.



The memory cleaning system finds dead objects by sorting through the heap memory and checking the corresponding bits of the objects. In order to preserve the semantics of C ++, the memory cleaning system needs to call the destructor of each of the "dead" objects before releasing the memory it occupies. Nontrivial destructors are implemented as finalizers.



Here, from the point of view of the programmer, there is no predetermined order in which the destructors are called, since the search mechanism used by the memory cleaning system does not take into account the order of creation of the destroyed objects. This imposes restrictions on finalizers, according to which they cannot access other objects that are on the heap. We are faced with a task that is encountered quite often. It consists in the following: there is a platform that does not support specifying the order of finalization of objects (like Java), but at the same time, for this platform, you need to write code that requires a certain order of calling finalizers. Oilpan uses the Clang plugin, which, in static mode, provides a ban on access to heap objects during the destruction of objects (this is just one of Clang’s many features): 



class GCed : public GarbageCollected<GCed> {
 public:
  void DoSomething();
  void Trace(Visitor* visitor) {
    visitor->Trace(other_);
  }
  ~GCed() {
    other_->DoSomething();  // error: Finalizer '~GCed' accesses
                            // potentially finalized field 'other_'.
  }
 private:
  Member<GCed> other_;
};


An error occurs in this code when trying to access an object from the heap during the destruction of another object.



If you are interested in the issue of destroying objects, then know that Oilpan gives the programmer the callbacks that are called before the finalizer. This allows you to implement complex schemes for the destruction of objects when an object, before destruction, needs access to other objects. Such callbacks create a considerable additional load on the system during garbage collection. This load is greater than what destructors are capable of creating. Therefore, such callbacks are used relatively rarely in Blink.



Incremental and competitive memory flushing



Now that we’ve talked about the limitations of destructors in a managed C ++ environment, it's time to dwell in more detail on how memorypan implements and optimizes memory cleaning.



But, before moving on to this conversation, it is important to remember how programs run in a web environment. The execution of the code, namely the JavaScript code, as well as the cleaning of garbage, are initiated by the main thread by placing tasks in the event loop . The rendering system, like other environments for executing code, supports background tasks, which, together with the main thread, are executed in a competitive mode. This allows you to remove part of the load from the main stream.



At the very beginning, Oilpan used a memory-cleaning mechanism implemented according to the “stop-the-world” scheme. This meant that the execution of the application in the main thread was suspended during the memory cleaning procedure.









Pausing code execution during memory cleanup



For applications that do not have too strong restrictions regarding their real-time operation, delays are a determining factor when using the garbage collector. Clearing the memory using a stop-the-world mechanism can cause the program to pause for significant periods of time, which leads to delays noticeable to users. To reduce the time of these delays, garbage collection was improved by implementing incremental memory cleaning.









Incremental memory clearing



When using the incremental approach, the phases of marking and destruction of objects are separated from each other. In this case, the memory cleaning procedure is divided into several parts, represented by separate tasks that are performed in the main thread. In the best case, such tasks are performed only during downtime.systems, which avoids their conflict with tasks that represent the usual application mechanisms. The internal mechanisms of the garbage collector share one big task of cleaning memory into small tasks, this separation is based on the concept of “page”. Pages can be in two states. Some pages are in a state of waiting to be cleared (to-be-swept), while others are already cleared (already-swept). Memory allocation mechanisms only consider pages that have already been cleared and replenish local memory allocation buffers (Local Allocation Buffers, LABs) from the free memory list, which stores a list of available memory fragments. In order to get the memory, information about which is in the list of free memory, the application will first try to find a memory that refers to already cleared pages.Then the application will try to help the system in processing pages that are waiting to be cleaned, using, where it tries to allocate memory, a memory cleaning system. A new memory will be requested from the operating system only if the above actions did not result in the application receiving the memory it needs.



In Oilpan, the incremental memory cleaning mechanism has been used for several years. But, as applications, and hence the graphs of their objects, became more and more, memory clearing began to affect application performance. For the sake of improving garbage collection mechanisms, we began to use background tasks to implement competitive memory cleaning. In order to eliminate the “data race” between the background tasks that perform memory cleaning and the application that allocates memory for new objects, there are two rules:



  • The memory cleaning system only processes the memory of “dead” objects, which, by definition, are unattainable from the application.
  • An application allocates memory using only the memory that belongs to cleared pages. Such pages are no longer interested in a memory cleaning system.


Both rules make it possible to ensure that several entities do not fight with each other for the same object and its memory. Unfortunately, C ++ relies heavily on destructors that are implemented as finalizers. Oilpan runs finalizers in the main thread to help developers and eliminate data race in the application code itself. To solve this problem, Oilpan plans to defer the finalizer of the object in the main thread. More specifically, whenever a competitive memory cleaning system encounters an object that has a finalizer (destructor), it places it in the finalization queue, which will be processed at a special stage of finalization. The code at this stage is always executed in the main thread, in which, in addition, the application code is also executed. Here's how, in general terms,looks like a competitive memory clearing scheme.









Competitive memory cleaning using background tasks



Since finalizers may need access to everything that is contained in objects, adding the appropriate memory to the free memory list is delayed until the finalizer completes. If the object does not have a finalizer, then the memory cleaning mechanism that runs in the background thread immediately adds the freed memory to the free memory list.



results



The background memory cleanup mechanism was released in the Chrome M78 assembly. Our framework for testing Chrome under conditions close to real ones showed a decrease in the time spent in the main thread for operations related to cleaning the memory by 25-50% (on average - by 42%).
