ストレージのデータ品質

リポジトリ内のデータの品質は、貴重な情報を取得するための重要な前提条件です。品質が悪いと、長期的には連鎖反応が悪化します。
まず、提供された情報への信頼が失われます。人々はビジネスインテリジェンスアプリケーションの使用を減らし始めており、アプリケーションの可能性はまだ主張されていません。
その結果、分析プロジェクトへのさらなる投資が問われています。

定義と意味


品質とは、お客様の期待を裏付ける満足度です。

ただし、データ品質には定義が含まれていません。それは常に使用のコンテキストを反映しています。データウェアハウスとBIシステムは、データの取得元であるオペレーティングシステムとは異なる目的を実行します。

たとえば、オペレーティングシステムでは、クライアント属性は必須フィールドではない場合があります。リポジトリでは、この属性をディメンションとして使用でき、その入力は必須です。次に、デフォルト値を入力する必要が生じます。

データウェアハウスの要件は常に変化しており、通常はオペレーティングシステムよりも高くなっています。ただし、オペレーティングシステムからの詳細情報をストレージに保存する必要がない場合は、逆になる可能性があります。

データ品質を測定可能にするには、その基準を説明する必要があります。仕事に情報と数字を使用する人は、説明プロセスに関与する必要があります。この関与の結果はルールになる可能性があり、その結果、テーブルを一目見れば、エラーの有無を確認できます。このルールは、後続の検証のためにスクリプト/コードの形式で発行する必要があります。

データ品質の向上


データをリポジトリーにロードするプロセスで、すべての仮想エラーをクリーンアップして修正することは不可能です。良好なデータ品質は、すべての参加者の綿密な作業によってのみ達成できます。オペレーティングシステムにデータを入力するユーザーは、エラーにつながるアクションを見つける必要があります。

データ品質はプロセスです。残念ながら、多くの組織では継続的な改善のための戦略はありません。多くの人はデータの保存のみに制限しており、分析システムの可能性を最大限に活用していません。原則として、データウェアハウスを開発する場合、予算の70〜80％はデータ統合の実装に費やされます。制御と改善のプロセスは、たとえあったとしても、未完成のままです。

ツール


ソフトウェアツールの使用は、データ品質の改善と監視を自動化するプロセスに役立ちます。たとえば、フィールド構造、デフォルト値の存在、テーブルフィールド名の要件への準拠など、ストレージ構造の技術検証を完全に自動化できます。

内容の確認が難しい場合があります。ストレージ要件が変わると、データの解釈が変わる場合があります。ツール自体は、サポートを必要とする巨大なプロジェクトになる可能性があります。

ヒント


リポジトリが通常設計されているリレーショナルデータベースは、ビュー（ビュー）を作成する絶好の機会です。コンテンツの特徴がわかっている場合は、データをすばやく確認するために使用できます。データのエラーまたは問題を発見する各ケースは、データベースへのクエリの形式で記録できます。

これにより、コンテンツ知識ベースが形成される。もちろん、そのようなリクエストは高速でなければなりません。原則として、ビューのサービスは、テーブルで構成されたツールよりも人の手間がかかりません。ビューは常にチェックの結果を表示する準備ができています。
重要なレポートの場合、ビューには宛先の列が含まれる場合があります。同じBIツールを使用して、リポジトリ内のデータ品質のステータスをレポートすることは理にかなっています。

例


要求はOracleデータベース用に作成されます。この例では、テストは必要に応じて解釈できる数値を返します。値T_MINおよびT_MAXを使用して、アラームの程度を調整できます。REPORTフィールドは、電子メールを適切に送信する方法を知らない商用ETL製品のメッセージとしてかつて使用されていたため、rpadは「松葉杖」です。

大きなテーブルの場合、たとえば、AND ROWNUM <= 10を追加できます。エラーが10件ある場合は、これで十分です。

CREATE OR REPLACE VIEW V_QC_DIM_PRODUCT_01 AS
SELECT
  CASE WHEN OUTPUT>=T_MIN AND OUTPUT<=T_MAX
  THEN 'OK' ELSE 'ERROR' END AS RESULT,
  DESCRIPTION,
  TABLE_NAME, 
  OUTPUT, 
  T_MIN,
  T_MAX,
  rpad(DESCRIPTION,60,' ') || rpad(OUTPUT,8,' ') || rpad(T_MIN,8,' ') || rpad(T_MAX,8,' ') AS REPORT
FROM (-- Test itself
  SELECT
    'DIM_PRODUCT' AS TABLE_NAME,
    'Count of blanks' AS DESCRIPTION,
    COUNT(*) AS OUTPUT,
    0 AS T_MIN,
    10 AS T_MAX
  FROM DIM_PRODUCT
  WHERE DIM_PRODUCT_ID != -1 -- not default value
  AND ATTRIBUTE IS NULL ); -- count blanks
