Spring Frameworkの最も一般的な10のエラー

こんにちは、ハブル！Toni Kukurinによる「Spring Frameworkの最も一般的な10の間違い」の記事の翻訳を紹介します。

Springはおそらく最も人気のあるJavaフレームワークの1つであり、飼いならす強力な獣です。基本的な概念はかなり理解しやすいですが、強力なSpring開発者になるには時間と労力が必要です。

この記事では、Springの最も一般的なエラーのいくつか、特にWebアプリケーションとSpring Bootに関連するエラーについて説明します。Spring Bootウェブサイトが言うように、それは産業用アプリケーションをどのように構築すべきかという考えを課しているので、この記事ではこの考えを実証し、Spring BootでのWebアプリケーションの開発の標準プロセスにうまく適合するいくつかのヒントの概要を示します。
Spring Bootにあまり詳しくないが、前述のことをいくつか試したい場合は、この記事に付属するGitHubリポジトリを作成しました。記事のどこかが見当たらないと感じた場合は、リポジトリをローカルコンピューターに複製し、コードをいじることをお勧めします。

よくある間違い＃1：低すぎる


「ここで発明されていない」 症候群はソフトウェア開発の世界ではかなり一般的であるため、このよくある間違いに遭遇します。症状には、頻繁に使用されるコードのフラグメントを定期的に書き換えることが含まれ、多くの開発者がこれに苦しんでいるようです。

特定のライブラリの内部とその実装を理解することは、大部分が適切で必要です（そして優れた学習プロセスになる可能性があります）が、同じ低レベルの実装の詳細を常に解決することは、ソフトウェアエンジニアとしての開発にとって悪いことです。 Springのような抽象化やフレームワークが存在し、反復的な手工芸から厳密に分離して、より高いレベルの詳細（ドメインオブジェクトやビジネスロジック）に集中できるようになる理由があります。

したがって、抽象化を使用します。次に特定の問題が発生した場合は、最初にクイック検索を実行し、この問題を解決するライブラリがSpringに統合されているかどうかを判断します。現在、既存の適切なソリューションを見つける可能性が最も高いです。有用なライブラリーの例として、この記事の残りの例では、Lombokプロジェクトの注釈を使用します。Lombokは、テンプレートコードジェネレーターとして、また内部の遅延開発者として使用されますが、このライブラリの考え方に問題がないことを願っています。例として、Lombokでの「標準Java Bean」がどのように見えるかを見てください。

@Getter
@Setter
@NoArgsConstructor
public class Bean implements Serializable {
    int firstBeanProperty;
    String secondBeanProperty;
}

ご想像のとおり、上記のコードは次のようにコンパイルされます。

public class Bean implements Serializable {
    private int firstBeanProperty;
    private String secondBeanProperty;

    public int getFirstBeanProperty() {
        return this.firstBeanProperty;
    }

    public String getSecondBeanProperty() {
        return this.secondBeanProperty;
    }

    public void setFirstBeanProperty(int firstBeanProperty) {
        this.firstBeanProperty = firstBeanProperty;
    }

    public void setSecondBeanProperty(String secondBeanProperty) {
        this.secondBeanProperty = secondBeanProperty;
    }

    public Bean() {
    }
}

ただし、IDEでLombokを使用する場合は、プラグインをインストールする必要がある可能性が高いことに注意してください。IntelliJ IDEAのプラグインバージョンは、こちらで確認できます。

よくある間違い＃2：内部コンテンツの漏洩


内部構造を明らかにすることは常に悪い考えです。なぜなら、それはサービスの設計に柔軟性をもたらさないため、コーディングの習慣に悪影響を及ぼすからです。内部コンテンツの「リーク」は、特定のAPIエンドポイントからデータベース構造にアクセスできるという事実に現れています。例として、次のPOJO（「Plain Old Java Object」）がデータベース内のテーブルを表すとします。

@Entity
@NoArgsConstructor
@Getter
public class TopTalentEntity {

    @Id
    @GeneratedValue
    private Integer id;

    @Column
    private String name;

    public TopTalentEntity(String name) {
        this.name = name;
    }
}

TopTalentEntityデータにアクセスする必要があるエンドポイントがあるとします。TopTalentEntityインスタンスを返すのがどんなに魅力的であっても、より柔軟なソリューションは、APIエンドポイントにTopTalentEntityデータを表示する新しいクラスを作成することです。

@AllArgsConstructor
@NoArgsConstructor
@Getter
public class TopTalentData {
    private String name;
}

したがって、データベースバックエンドに変更を加える場合、サービスレイヤーで追加の変更を行う必要はありません。TopTalentEntityにパスワードフィールドを追加して、ユーザーパスワードハッシュをデータベースに保存するとどうなるかを考えてください。TopTalentDataなどのコネクタがないと、サービスの変更を忘れると、フロントエンドが非常に望ましくない秘密情報を誤って表示します。

よくある間違い＃3：職務分離の欠如


アプリケーションが成長するにつれて、コードの整理がますます重要な問題になります。皮肉なことに、ソフトウェア開発の優れた原則のほとんどはどこでも違反され始めています。特に、アプリケーションアーキテクチャの設計にほとんど注意が払われていない場合です。開発者が直面する最も一般的な間違いの1つは、コードの責任の混合であり、それを行うのは非常に簡単です。

通常職務分離の原則に違反するものなので、これは既存のクラスに新しい機能を「追加」するだけです。もちろん、これは優れた短期的な解決策です（手始めに、入力する必要が少なくなります）が、テスト中、メンテナンス中、またはその中間のいずれかに関係なく、将来的に問題になることは避けられません。リポジトリからTopTalentDataを返す次のコントローラーについて考えてみます。

@RestController
public class TopTalentController {

    private final TopTalentRepository topTalentRepository;

    @RequestMapping("/toptal/get")
    public List<TopTalentData> getTopTalent() {
        return topTalentRepository.findAll()
                .stream()
                .map(this::entityToData)
                .collect(Collectors.toList());
    }

    private TopTalentData entityToData(TopTalentEntity topTalentEntity) {
        return new TopTalentData(topTalentEntity.getName());
    }
}

最初は、このコードの一部に問題があることは目立ちません。TopTalentEntityインスタンスから取得されるTopTalentDataリストを提供します。ただし、よく見ると、実際にはTopTalentControllerがここでいくつかのことを行うことがわかります。つまり、特定のエンドポイントへのリクエストをマッピングし、リポジトリからデータを取得し、TopTalentRepositoryから取得したエンティティを別の形式に変換します。「よりクリーンな」ソリューションは、これらの責任を独自のクラスに分割することです。次のようになります。

@RestController
@RequestMapping("/toptal")
@AllArgsConstructor
public class TopTalentController {

    private final TopTalentService topTalentService;

    @RequestMapping("/get")
    public List<TopTalentData> getTopTalent() {
        return topTalentService.getTopTalent();
    }
}

@AllArgsConstructor
@Service
public class TopTalentService {

    private final TopTalentRepository topTalentRepository;
    private final TopTalentEntityConverter topTalentEntityConverter;

    public List<TopTalentData> getTopTalent() {
        return topTalentRepository.findAll()
                .stream()
                .map(topTalentEntityConverter::toResponse)
                .collect(Collectors.toList());
    }
}

@Component
public class TopTalentEntityConverter {
    public TopTalentData toResponse(TopTalentEntity topTalentEntity) {
        return new TopTalentData(topTalentEntity.getName());
    }
}

この階層のもう1つの利点は、クラス名を確認するだけで機能の場所を特定できることです。さらに、テスト中に、必要に応じてクラスをモック実装に簡単に置き換えることができます。

よくある間違い＃4：不整合と不十分なエラー処理


整合性のトピックは必ずしもSpring（またはJava）に限定されているわけではありませんが、Springプロジェクトで作業する場合に考慮すべき重要な側面です。コードの記述方法は議論の対象になる可能性があります（通常、チームまたは会社全体での合意事項です）が、共通標準の存在はパフォーマンスの大きな助けになります。これは特に、数人のチームに当てはまります。一貫性により、さまざまなクラスの責任に関する詳細な説明を維持または提供するために多くのリソースを必要とせずにコードを転送できます。

さまざまな構成ファイル、サービス、コントローラーを備えたSpringプロジェクトを考えてみましょう。名前が意味的に一貫しているため、簡単に検索できる構造が作成されます。この構造では、新しい開発者がコードの操作方法を制御できます。たとえば、サフィックスConfigが構成クラスに、サービスサフィックスがサービスに、コントローラーサフィックスがコントローラーに追加されます。

一貫性のトピックと密接に関連しているため、サーバー側のエラー処理には特別な注意が必要です。作成が不十分なAPIからの例外応答を処理する必要があったことがある場合は、例外を解析するのが難しいかもしれない理由をおそらく知っており、これらの例外が最初に発生した理由を特定することはさらに困難です。

API開発者は、すべてのユーザーエンドポイントをカバーし、それらを一般的なエラー形式に変換することが理想的です。これは通常、次の形式の言い訳だけでなく、一般的なエラーコードと説明があることを意味します。クライアント側の処理の複雑さに加えて）。
一般的なエラー応答形式の例は次のとおりです。

@Value
public class ErrorResponse {

    private Integer errorCode;
    private String errorMessage;
}

同様のものが通常最も人気のあるAPIにあり、簡単に体系的に文書化できるため、通常はうまく機能します。メソッドに@ExceptionHandlerアノテーションを付けることで、例外をこの形式に変換できます（アノテーションの例は、よくある間違い＃6にあります）。

よくある間違い＃5：不正なマルチスレッド


デスクトップやWebアプリケーションで見つかるかどうか、Springで見つかるかどうかに関係なく、マルチスレッド化は困難な作業になる可能性があります。並列プログラムの実行によって引き起こされる問題はとらえどころがなく、デバッグするのが非常に難しいことがよくあります。実際、問題の性質上、並列実行の問題を扱っていることを理解したら、おそらくデバッガを完全に破棄して開始する必要があります。エラーの原因が見つかるまで、コードを手動で確認してください。残念ながら、そのような問題を解決するためのテンプレートソリューションはありません。特定のケースに応じて、状況を評価してから、最良であると考える角度から問題に対処する必要があります。

もちろん、マルチスレッドのバグを完全に回避することが理想的です。繰り返しになりますが、これに対する単一のアプローチはありませんが、マルチスレッドエラーのデバッグと防止に関するいくつかの実際的な考慮事項があります。

グローバルステータスを回避する


まず、「グローバル状態」の問題を常に覚えておいてください。マルチスレッドアプリケーションを作成している場合は、グローバルに変更できるすべての要素を注意深く監視し、可能であれば完全に削除する必要があります。グローバル変数を変更可能にする必要がある理由がある場合は、同期を慎重に使用し、アプリケーションのパフォーマンスを監視して、新しい待機期間が原因で速度が低下していないことを確認してください。

可変性を避ける


これは関数型プログラミングから直接続き、OOPに従って、クラスの可変性と状態の変更は避けられるべきであると述べています。つまり、上記は、モデルのすべてのクラスにセッターとプライベートファイナルフィールドが存在することを意味します。それらの値は建設中にのみ変更されます。したがって、リソースの競合に問題がなく、オブジェクトのプロパティにアクセスしても常に正しい値が提供されることを確認できます。

重要なデータを記録する


アプリケーションが問題を引き起こす可能性のある場所を評価し、すべての重要なデータを事前に記録します。エラーが発生した場合は、受信したリクエストに関する情報に感謝し、アプリケーションの動作が悪い理由をよりよく理解できます。繰り返しますが、ロギングによってファイルのI / Oが増加するため、アプリケーションのパフォーマンスに深刻な影響を与える可能性があるため、ログを乱用しないでください。

既存の実装を再利用する


独自のスレッドを作成する必要がある場合（たとえば、さまざまなサービスに非同期要求を行う場合）は、独自のソリューションを作成するのではなく、既存の安全な実装を再利用してください。ほとんどの場合、これは、Java 8のきちんとした機能的なスタイルでExecutorServicesとCompletableFuturesを使用してスレッドを作成することを意味します。Springでは、DeferredResultクラスを介した非同期リクエスト処理も可能です。

よくある間違い＃6：注釈ベースの検証を使用しない


上記のTopTalentサービスに、新しいSuper Talentsを追加するためのエンドポイントが必要だと想像してみましょう。さらに、何らかの正当な理由により、新しい名前はそれぞれ正確に10文字でなければならないとします。これを行う1つの方法は次のとおりです。

@RequestMapping("/put")
public void addTopTalent(@RequestBody TopTalentData topTalentData) {
    boolean nameNonExistentOrHasInvalidLength =
            Optional.ofNullable(topTalentData)
         .map(TopTalentData::getName)
   .map(name -> name.length() == 10)
   .orElse(true);
    if (nameNonExistentOrInvalidLength) {
        // throw some exception
    }
    topTalentService.addTopTalent(topTalentData);
}

ただし、上記は（設計が不十分であることに加えて）実際には「クリーン」なソリューションではありません。複数の種類の有効性（つまり、TopTalentDataがnullでないこと、TopTalentData.nameがnullでないこと、TopTalentData.nameが10文字であること）を確認し、データが無効な場合も例外をスローします。

これは、SpringでHibernateバリデーターを使用することで、はるかにきれいに行うことができます。最初に、検証をサポートするためにaddTopTalentメソッドを書き換えます。

@RequestMapping("/put")
public void addTopTalent(@Valid @NotNull @RequestBody TopTalentData topTalentData) {
    topTalentService.addTopTalent(topTalentData);
}

@ExceptionHandler
@ResponseStatus(HttpStatus.BAD_REQUEST)
public ErrorResponse handleInvalidTopTalentDataException(MethodArgumentNotValidException methodArgumentNotValidException) {
    // handle validation exception
}

さらに、TopTalentDataクラスでチェックするプロパティを指定する必要があります。

public class TopTalentData {
    @Length(min = 10, max = 10)
    @NotNull
    private String name;
}

Springはリクエストをインターセプトし、メソッドを呼び出す前にそれを検証します-追加の手動テストを使用する必要はありません。

同じことを達成できるもう1つの方法は、独自の注釈を作成することです。カスタムアノテーションは通常、ニーズがHibernate組み込み定数セットを超える場合にのみ使用されますが、この例では、アノテーションを想像してみましょう長さ存在しません。チェック用とプロパティのアノテーション用の2つの追加クラスを作成して、文字列の長さをチェックするバリデーターを作成する必要があります。

@Target({ElementType.METHOD, ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Constraint(validatedBy = { MyAnnotationValidator.class })
public @interface MyAnnotation {

    String message() default "String length does not match expected";

    Class<?>[] groups() default {};

    Class<? extends Payload>[] payload() default {};

    int value();
}

@Component
public class MyAnnotationValidator implements ConstraintValidator<MyAnnotation, String> {

    private int expectedLength;

    @Override
    public void initialize(MyAnnotation myAnnotation) {
        this.expectedLength = myAnnotation.value();
    }

    @Override
    public boolean isValid(String s, ConstraintValidatorContext constraintValidatorContext) {
        return s == null || s.length() == this.expectedLength;
    }
}

これらの場合、職務を分離するためのベストプラクティスでは、プロパティがnull（isValidメソッドではs == null）の場合に有効としてマークし、アノテーションを使用する必要があることに注意してください。 Notnullこれがプロパティの追加要件である場合：

public class TopTalentData {
    @MyAnnotation(value = 10)
    @NotNull
    private String name;
}

一般的な間違い＃7：（まだ）XML構成を使用する


Springの以前のバージョンではXMLが必要でしたが、現在のほとんどの構成は、Javaコード/アノテーションでのみ行うことができます。 XML構成は、単に追加の不要な定型文を表しています。
この記事（および付随するGitHubリポジトリ）はアノテーションを使用してSpringを構成し、Springは接続する必要のあるBeanを知っています。

@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

この複合アノテーション（Springのドキュメントで詳細を確認できます）は、Beanを抽出するためにスキャンする必要があるパッケージに関するヒントをSpringに提供するだけです。私たちの特定のケースでは、これは、次のクラスがトップレベルのパッケージ（co.kukurin）から始めて、Beanを接続するために使用されることを意味します： @Configurationで注釈された追加のクラスがある場合、Java-もチェックされます構成。

  • @Component (TopTalentConverter, MyAnnotationValidator)
  • @RestController (TopTalentController)
  • @Repository (TopTalentRepository)
  • @Service (TopTalentService)



よくある間違い番号8：プロファイルを忘れる


サーバーの設計時によく発生する問題は、さまざまなタイプの構成（通常は工業用構成と開発構成）の違いです。テストからアプリケーションのデプロイメントに切り替えるたびにさまざまな構成パラメーターを手動で変更する代わりに、プロファイルを使用する方がより効率的です。

ローカル開発用のインメモリデータベースとPROMのMySQLデータベースを使用する場合を考慮してください。つまり、これは、異なるURLと（うまくいけば）異なる資格情報を使用して、それぞれにアクセスすることを意味します。2つの異なる構成ファイルを使用してこれを行う方法を見てみましょう。

ファイルAPPLICATION.YAML


# set default profile to 'dev'
spring.profiles.active: dev

# production database details
spring.datasource.url: 'jdbc:mysql://localhost:3306/toptal'
spring.datasource.username: root
spring.datasource.password:

ファイルAPPLICATION-DEV.YAML


spring.datasource.url: 'jdbc:h2:mem:'
spring.datasource.platform: h2

どうやら、コードをいじる間、産業用データベースで誤ってアクションを実行したくないので、devでデフォルトのプロファイルを設定することは理にかなっています。次に、サーバーで、JVMの-Dspring.profiles.active = prodパラメーターを指定して、構成プロファイルを手動でオーバーライドできます。さらに、OS環境変数を目的のデフォルトプロファイルに設定することもできます。

よくある間違い＃9：依存性注入を受け入れることができない


Springでの依存性注入の適切な使用は、必要なすべての構成クラスをスキャンすることにより、すべてのオブジェクトを一緒にバインドできることを意味します。これは、関係を切り離すのに役立ち、テストもはるかに簡単になります。次のようなことをして、クラスをハードリンクする代わりに：

public class TopTalentController {

    private final TopTalentService topTalentService;

    public TopTalentController() {
        this.topTalentService = new TopTalentService();
    }
}


Springにバインディングを実行させます。

public class TopTalentController {

    private final TopTalentService topTalentService;

    public TopTalentController(TopTalentService topTalentService) {
        this.topTalentService = topTalentService;
    }
}

GoogleトークのMisko Heveryが依存関係注入の「理由」を詳しく説明しているので、実際にこれがどのように使用されるかを見てみましょう。責任分担（よくある間違い＃3）では、サービスクラスとコントローラークラスを作成しました。TopTalentServiceが正しく動作していることを前提として、コントローラーをテストするとします。実際のサービス実装の代わりにモックオブジェクトを挿入して、別の構成クラスを提供できます。

@Configuration
public class SampleUnitTestConfig {
    @Bean
    public TopTalentService topTalentService() {
        TopTalentService topTalentService = Mockito.mock(TopTalentService.class);
        Mockito.when(topTalentService.getTopTalent()).thenReturn(
                Stream.of("Mary", "Joel")
		      .map(TopTalentData::new).collect(Collectors.toList()));
        return topTalentService;
    }
}

次に、設定プロバイダーとしてSampleUnitTestConfigを使用するようにSpringに指示することで、モックオブジェクトを埋め込むことができます。

@ContextConfiguration(classes = { SampleUnitTestConfig.class })

次に、コンテキストテストを使用して、ユニットテストにカスタムBeanを埋め込むことができます。

よくある間違い＃10：テストの欠如または不正確なテスト


単体テストのアイデアは長い間私たちと一緒にされてきたという事実にもかかわらず、多くの開発者はこれを行うことを（特にこれが必要でない場合）「忘れる」か、単に後で使用するために残しているようです。テストはコードの正確性を検証するだけでなく、アプリケーションがさまざまな状況でどのように動作するかに関するドキュメントとしても機能するため、これは明らかに望ましくありません。

Webサービスをテストするとき、例外的に「クリーン」な単体テストを行うことはめったにありません。通常、HTTPを介した対話では、DispatcherServlet Springを呼び出して、実際のHttpServletRequestを受信したときに何が起こるかを確認します（これにより、統合テストが行​​われます）。検証、シリアル化などを使用します）。安心してください-MockMVC上でRESTサービスを簡単にテストするためのJava DSLは、非常にエレガントなソリューションであることが判明しました。依存関係注入を伴う次のコードフラグメントを検討してください。

@RunWith(SpringJUnit4ClassRunner.class)
@ContextConfiguration(classes = {
        Application.class,
        SampleUnitTestConfig.class
})
public class RestAssuredTestDemonstration {

    @Autowired
    private TopTalentController topTalentController;

    @Test
    public void shouldGetMaryAndJoel() throws Exception {
        // given
        MockMvcRequestSpecification givenRestAssuredSpecification = RestAssuredMockMvc.given()
                .standaloneSetup(topTalentController);

        // when
        MockMvcResponse response = givenRestAssuredSpecification.when().get("/toptal/get");

        // then
        response.then().statusCode(200);
        response.then().body("name", hasItems("Mary", "Joel"));
    }
}

SampleUnitTestConfigは、TopTalentControllerのTopTalentServiceモック実装を有効にしますが、他のすべてのクラスは、Applicationクラスのパッケージにルートを持つパッケージをスキャンして取得した標準構成を使用して接続されます。RestAssuredMockMvcは、軽量環境を作成し、GETリクエストを/ toptal / getエンドポイントに送信するために使用されます。
