MonbMQパート1でのRabbitMQの使用

この記事は、キューとRabbitMQにまだ慣れていない方を対象としています。RabbitMQの操作方法をすでに知っていて、MonsterMQが提供する可能性を探りたいだけの場合は、github.com のプロジェクトページにアクセスすることをお勧めします。この記事の残りの部分では、RabbitMQがMonsterMQと連携する方法の基本について説明します。

RabbitMQはメッセージブローカーです。メッセージを受信、保存、および顧客に転送します。クライアントのメッセージの送信者と受信者は、たとえば、PHPで記述されたアプリケーションに対応できます。クライアントとRabbitMQ間の通信は、AMQP（Advanced Message Queuing Protocol）と呼ばれる特定のルールに従って行われます。たとえば、ブラウザとこの記事が置かれているWebサーバー間の通信も特定のルールに従っており、HTTP（ハイパーテキスト転送プロトコル）-ハイパーテキスト転送プロトコルと呼ばれています。 HTTPとAMQPはどちらもクライアントサーバープロトコルです。つまり、クライアントとサーバーの存在を意味します。 RabbitMQはAMQPサーバーであり、MonsterMQは、AMQPサーバーと通信するPHPでAMQPクライアントをすばやく作成できるライブラリです。

AMQPとRabbitMQは次の用語を使用します：作成

-メッセージを送信する手段、プロデューサー（サプライヤー）-メッセージを送信する人。



消費-メッセージを受信することを意味します。消費者-メッセージを受信する人。



キューはキューです。コンシューマに到達する前、およびサプライヤから送信された後、メッセージはRabbitMQ内のキューに格納されます。キューは、メッセージを格納するバッファです。多くのプロバイダーは1つのキューでメッセージを送信でき、多くのコンシューマーは1つのキューからメッセージを読み取ることができます。



プロバイダー、コンシューマー、ブローカーが同じホスト上にある必要はないことに注意してください。逆に、通常、それらは異なるホスト上にあります。

こんにちは世界


この記事の後半では、MonsterMQライブラリを使用してPHPで2つのプログラムを記述します。1つはメッセージを送信し、もう1つはそれらを受信して​​コンソールに出力します。下の図で、「P」はプロバイダー、「C」はコンシューマー、赤いボックスはメッセージバッファーであるキューです（各ボックスはメッセージです）。


（RabbitMQの公式Webサイトから取得した画像）

さらに作業を進めるには、MonsterMQをインストールする必要があります。これを行うには、コンソールで次のコマンドを入力します。

composer require goootlib/monster-mq:dev-master

（MonsterMQのインストールの詳細は、github.com のプロジェクトページにあります）
このコマンドは、MonsterMQを現在のフォルダーにインストールします。

出発


送信者をsend.php、受信者をreceive.phpと呼びます。送信者はサーバーに参加してメッセージを送信します。
サーバーに参加するには、send.phpに次のコードを記述します。

try {
   $producer = \MonsterMQ\Client\Producer();

   $producer->connect('127.0.0.1', 5672);
   $producer->logIn('guest', 'guest');
} catch(\Exception $e) {
   var_dump($e);
}

このコードでは、標準ポートのローカルサーバーで実行されているRabbitMQに参加します。RabbitMQが他のアドレスで実行されている場合は、他のIPアドレスとポートを指定します。接続後、標準のログインとパスワードでセッションを開始するlogIn（）メソッドも呼び出します。RabbitMQで異なるユーザー名とパスワードが設定されたアカウントを持っている場合は、ここでそれらを指定します。

メッセージを送信するには、最初にメッセージが置かれるキューを宣言する必要があります。キューがアナウンスされたら、メッセージを送信できます。

try {
   $producer = \MonsterMQ\Client\Producer();

   $producer->connect('127.0.0.1', 5672);
   $producer->logIn('guest', 'guest');

   $producer->queue('test-queue')->declare();
   $producer->publish('my-message', 'test-queue');
} catch(\Exception $e) {
   var_dump($e);
}

キュー宣言はべき等です。つまり、キューがすでに存在する場合、キューは作成されず、変更されません。セッションを完了して参加した後。

try {
   $producer = \MonsterMQ\Client\Producer();

   $producer->connect('127.0.0.1', 5672);
   $producer->logIn('guest', 'guest');

   $producer->queue('test-queue')->declare();
   $producer->publish('Test message', 'test-queue');

   $producer->disconnect();
} catch(\Exception $e) {
   var_dump($e);
}

メッセージを受信


送信者とは異なり、受信者は着信メッセージを聞くために実行を続けます。receive.phpコードでは、最初にsend.phpと同じことを行う必要があります。

try {
   $consumer = \MonsterMQ\Client\Consumer();

   $consumer->connect('127.0.0.1', 5672);
   $consumer->logIn('guest', 'guest');

   $consumer->queue('test-queue')->declare();
} catch(\Exception $e) {
   var_dump($e);
}

receive.phpではキューも宣言して いることに注意してください。これは、送信者の前に開始された場合に受信者がキューを宣言するためです。
サーバーからメッセージを受信することをサーバーに通知するため、また着信メッセージを待機して処理するサイクルを開始するために、次のコードを記述します。

try {
   $consumer = \MonsterMQ\Client\Consumer();

   $consumer->connect('127.0.0.1', 5672);
   $consumer->logIn('guest', 'guest');

   $consumer->queue('test-queue')->declare();

   $consumer->consume('test-queue');

   $consumer->wait(function ($message, $channelNumber) use ($consumer){
      echo "Message - {$message} received on channel {$channelNumber}";
   });
} catch(\Exception $e) {
   var_dump($e);
}

発売


これで、両方のスクリプトを受信者として実行できます。

php receive.php

だから送信者：

php send.php

受信者は、RabbitMQから受信するメッセージ（send.phpから受信する）と使用されているチャネル番号をコンソールに出力します。ターミナルを閉じるまで、またはCtrl + Cを押して実行を停止するまで、引き続き実行されます。別の端末で送信者を起動すると、メッセージが受信者に配信され、コンソールに表示される様子を観察できます。

現在、RabbitMQにキューが含まれているキューとその中のメッセージの数を確認する場合は、rabbitmqctlを使用して特権ユーザーとしてこれを実行できます。
sudo rabbitmqctl list_queues


Windowsではsudoを使用しない
rabbitmqctl.bat list_queues
