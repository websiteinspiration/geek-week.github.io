.pre-commitフックをDjangoプロジェクトに統合する

良い一日！

私の名前はAndrey Sobolevです。今日は、プロジェクトで.pre-commitフックをどのように準備したかをお伝えします。

前書き


まず、一般的なフックの概要と、フックが必要になる理由について簡単に説明します。Gitには、イベントが発生したときにスクリプトを実行できるツールが用意されています（たとえば、サーバーへのプッシュなど）。

事前コミットは、で説明されているスクリプトを実行するデフォルトのgit事前コミットフックへの便利なアドオンです。コミットする前に.pre-commit-config.yaml。理論的には簡単に聞こえますが、練習に移りましょう。

フレーク8とリンター一般についての私の意見を述べます。多数のレガシーコードを含む大規模なプロジェクトがすでにある場合は、安全にリンターを削除できます。「理想に近づく」ために費やされる費用は、当局には認められないでしょう。私たちは新しい（そして小さな）プロジェクトのためにリンターを置きます。繰り返しますが、これは私の個人的な見解であり、私は誰にもそれを課しません。

環境統合


開発環境のルートディレクトリに移動し、次のコマンドを実行します

$ pre-commit install
pre-commit installed at .git/hooks/pre-commit
$ pre-commit --version
pre-commit 2.4.0

.pre-commitがsqliteを誓う場合は、それをインストールし（たとえば、$ yum install sqlite）、pythonを再度コンパイルする必要があります。

テスト


コードのチェックとフォーマットに加えて、コミットを作成する段階でテストを実行します。これを行うには、pytest（https://docs.pytest.org/en/latest/）を使用して、ニーズに合わせて構成します。

私たちの環境のルートディレクトリで、pytest.iniファイルを作成します。
[pytest]
DJANGO_SETTINGS_MODULE = settings
python_files = tests.py test_*.py *_tests.py


次に、testsフォルダーを作成し、次のファイル
test_example_without_db.py、test_example_with_db.py

を配置して、便宜上、現在のデータベース（たとえば、バトルサーバーからのデータベースのコピー）を使用できるようにテストを構成し、毎回新しいデータベースを作成しないようにします。

簡単なテストtest_example_without_db.py

def inc(x):
    return x + 1

def test_answer():
    assert inc(3) == 4

簡単なテストでは、たとえばWebbotを接続してシステムのノードをバイパスし、テスターの手動作業を自動化できます。test_example_with_db.py

データベースを使用してテストする

import pytest
import settings
from chat.models import ChatRoom
from settings import POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD, \
                    POSTGRES_HOST, POSTGRES_PORT

@pytest.fixture(scope='session')
def django_db_setup():
    settings.DATABASES['default'] = {
        'ENGINE': 'django.db.backends.postgresql_psycopg2',
        'NAME': POSTGRES_DB,
        'USER': POSTGRES_USER,
        'PASSWORD': POSTGRES_PASSWORD,
        'HOST': POSTGRES_HOST,
        'PORT': POSTGRES_PORT,
    }   
    
    
@pytest.fixture
def db_access_without_rollback_and_truncate(request, django_db_setup, django_db_blocker):
    django_db_blocker.unblock()
    request.addfinalizer(django_db_blocker.restore)

def chat():
    return ChatRoom.objects.all().count()

@pytest.mark.django_db
def test_chat():
    assert chat() > 0

この例は非常に人為的なものであり、このメモ専用に作成されましたが、それでも、現在のデータベースにアクセスして、手動テストを超える複雑なテストを実行できます。

.pre-commitにテストを含めます


テストを接続するには、環境のルートディレクトリにtest.shと呼ぶシェルスクリプトが必要です。

source ../../python38_env/bin/activate && python -m pytest -v tests

その内容は非常に明白ですが、仮想環境のアクティブ化がコードに明示的に記述されていることに気付くかもしれません。チームが別のワークステーションで開発している場合（たとえば、ローカルマシンに環境をデプロイした人と、サーバーで開発している人）の場合、これは不便です。

この問題は、.env実装

$ git add .
$ git commit -m Sobolev:TestPreCommitHook

$ git commit -m Sobolev:TestPreCommitHook
