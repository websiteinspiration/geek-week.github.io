Kapacitorでのメトリック処理のトリック

おそらく、今日、なぜサービスメトリックを収集する必要があるのか​​という質問は誰にもありません。次の論理的なステップは、収集されたメトリックのアラートを構成することです。これにより、便利なチャネル（メール、Slack、テレグラム）のデータの逸脱を通知します。オンライン予約サービスでOstrovok.ruのホテル、当社のサービスのすべてのメトリックがInfluxDBに注がれており、Grafanaに表示され、基本的な警告もが設定されています。 「何かを計算してそれと比較する必要がある」などのタスクでは、Kapacitorを使用します。


Kapacitorは、InfluxDBからのメトリックを処理できるTICKスタックの一部です。彼はいくつかの測定値を結合（結合）し、受信したデータから有用なものを計算し、結果をInfluxDBに書き戻し、アラートをSlack / Telegram / mailに送信できます。

スタック全体にはクールで詳細なドキュメントがありますが、マニュアルで明示的に指定されていない有用なものが常にあります。この記事では、わかりにくいヒント（基本的なTICKscipt構文についてはこちらで説明します）をいくつか集めて、問題の解決例を使用して、それらをどのように適用できるかを示します。

float＆int、計算エラー


絶対に標準的な問題であり、カーストによって解決されます：

var alert_float = 5.0
var alert_int = 10
data|eval(lambda: float("value") > alert_float OR float("value") < float("alert_int"))

デフォルト（）を使用する


タグ/フィールドが入力されていない場合、計算でエラーが発生します：

|default()
        .tag('status', 'empty')
        .field('value', 0)

結合を埋める（内部vs外部）


デフォルトでは、joinはデータのないポイント（内部）をドロップします。
fill（ 'null'）を使用すると、外部結合が実行されます。その後、デフォルト（）を実行して空の値を入力する必要があります。

var data = res1
    |join(res2)
        .as('res1', 'res2)
        .fill('null')
    |default()
        .field('res1.value', 0.0)
        .field('res2.value', 100.0)

まだニュアンスがあります。上記の例でシリーズ（res1またはres2）の1つが空の場合、最後のシリーズ（データ）も空になります。このトピックに関するいくつかのチケットはgithubの（にあります1633、1871、6967、我々は修正を待っていると少し苦しむしています- ）。

計算での条件の使用（ラムダの場合）


|eval(lambda: if("value" > 0, true, false)

期間のパイプラインからの最後の5分間


たとえば、過去5分間の値を前週と比較する必要があります。2つの独立したバッチamiで2つのデータパケットを取得するか、より長い期間からデータの一部を取り出すことができます。

 |where(lambda: duration((unixNano(now()) - unixNano("time"))/1000, 1u) < 5m)

最後の5分間の代わりに、BarrierNodeノードを使用することもできます。これは、指定された時間の前にデータを遮断します。

|barrier()
        .period(5m)

Docker、ビルドおよびデプロイ


起動時に、kapacitorは[load]ブロックの構成で指定されたディレクトリからタスク、テンプレート、およびハンドラーをロードできます。

タスクを正しく作成するには、次のものが必要です。

  1. ファイル名-id /スクリプト名に展開されます
  2. タイプ-ストリーム/バッチ
  3. dbrp-スクリプトが機能するデータベース+ポリシーを示すキーワード（dbrp "supplier"。 "autogen"）

一部のバッチタスクでdbrpの行がない場合、サービス全体が開始を拒否し、正直にログに書き込みます。

それどころか、クロノグラフでは、この行はそうであってはなりません。インターフェースを介して受け入れられず、エラーがスローされます。

コンテナをビルドするときのハック：//.+dbrpの行がある場合、Dockerfileは-1で終了します。これにより、ビルドのビルド時にファイルの理由がすぐにわかります。

1対多に参加する


タスクの例：1週間あたりのサービスの稼働時間の95パーセンタイルを取得する必要があります。最後の10分の毎分をこの値と比較します。

ポイントのグループごとに1対多、最後/平均/中央値に参加することはできません。ノードをストリームに変換すると、「子の不一致エッジを追加できません：バッチ->ストリーム」というエラーが返されます。

ラムダ式の変数としてのバッチの結果も置換されません。

最初のバッチから目的の番号をudfを介してファイルに保存し、サイドロードを介してこのファイルをロードするオプションがあります。

どうやってやったの？


繰り返しになりますが、ドキュメントには優れた開始例（結合シリーズ全体のレートの計算）があり、同様のタスクの基礎として調べたり、基準にしたりできます。

結果として彼らがしたこと：

  • 2つのシリーズに数時間で参加し、チャンネルごとにグループ化します。
  • データがない場合は、グループごとにシリーズを記入します。
  • 最後の10分の中央値を以前のデータと比較します。
  • 何かを見つけたら叫ぶ。
  • 計算されたレートと発生したアラートをinfluxdbに書き込みます。
  • slackに役立つメッセージを送信します。

私の意見では、出力で取得したいすべてのものを可能な限り美しく管理しました（カスタムハンドラーを使用するとさらに多くのことです）。

結果のコードの例：
dbrp "supplier"."autogen"
var name = 'requests.rate'
var grafana_dash = 'pczpmYZWU/mydashboard'
var grafana_panel = '26'
var period = 8h
var todayPeriod = 10m
var every = 1m
var warnAlert = 15
var warnReset = 5
var reqQuery = 'SELECT sum("count") AS value FROM "supplier"."autogen"."requests"'
var errQuery = 'SELECT sum("count") AS value FROM "supplier"."autogen"."errors"'

var prevErr = batch
    |query(errQuery)
        .period(period)
        .every(every)
        .groupBy(1m, 'channel', 'supplier')

var prevReq = batch
    |query(reqQuery)
        .period(period)
        .every(every)
        .groupBy(1m, 'channel', 'supplier')

var rates = prevReq
    |join(prevErr)
        .as('req', 'err')
        .tolerance(1m)
        .fill('null')
    //   ,    
    |default()
        .field('err.value', 0.0)
        .field('req.value', 0.0)
    // if  lambda:  ,    
    |eval(lambda: if("err.value" > 0, 100.0 * (float("req.value") - float("err.value")) / float("req.value"), 100.0))
        .as('rate')

//     
rates
    |influxDBOut()
        .quiet()
        .create()
        .database('kapacitor')
        .retentionPolicy('autogen')
        .measurement('rates')

//     10 ,  
var todayRate = rates
    |where(lambda: duration((unixNano(now()) - unixNano("time")) / 1000, 1u) < todayPeriod)
    |median('rate')
        .as('median')

var prevRate = rates
    |median('rate')
        .as('median')

var joined = todayRate
    |join(prevRate)
        .as('today', 'prev')
    |httpOut('join')

var trigger = joined
    |alert()
        .warn(lambda: ("prev.median" - "today.median") > warnAlert)
        .warnReset(lambda: ("prev.median" - "today.median") < warnReset)
        .flapping(0.25, 0.5)
        .stateChangesOnly()
        //   message     
        .message(
            '{{ .Level }}: {{ index .Tags "channel" }} err/req ratio ({{ index .Tags "supplier" }})
{{ if eq .Level "OK" }}It is ok now{{ else }}
'+string(todayPeriod)+' median is {{ index .Fields "today.median" | printf "%0.2f" }}%, by previous '+string(period)+' is {{ index .Fields "prev.median" | printf "%0.2f" }}%{{ end }}
http://grafana.ostrovok.in/d/'+string(grafana_dash)+
'?var-supplier={{ index .Tags "supplier" }}&var-channel={{ index .Tags "channel" }}&panelId='+string(grafana_panel)+'&fullscreen&tz=UTC%2B03%3A00'
        )
        .id('{{ index .Tags "name" }}/{{ index .Tags "channel" }}')
        .levelTag('level')
        .messageField('message')
        .durationField('duration')
        .topic('slack_graph')

// "today.median"   "value",        (keep)
trigger
    |eval(lambda: "today.median")
        .as('value')
        .keep()
    |influxDBOut()
        .quiet()
        .create()
        .database('kapacitor')
        .retentionPolicy('autogen')
        .measurement('alerts')
        .tag('alertName', name)
