Java 14：レコードのプレビュー

まもなくJava 14で新しい構文機能- レコーディング（レコード）が提供されます。レコードの外観と「彼らが何を食べるか」について簡単に説明しているプレビューを調べた後、私はあえてドキュメントをHabrのロシア語に適合させました。誰も気にしない-猫へようこそ。


概要


エントリを使用すると、Javaの機能を拡張できます。これらは、定数の不変のデータセットの単純なキャリアであるクラスを宣言するための簡潔な構文を提供します。

理由と目標


「Javaは冗長すぎる」、「Javaでのセレモニー」が必要だという苦情はよくあることです。これは、特定のデータセットを格納するためだけに設計されたクラスが原因です。このようなクラスを正しく記述するには、コンストラクター、ゲッターとセッター、equals（）、hashCode（）、toString（）など、多くの形式が繰り返され、エラーが発生しやすいコードを記述する必要があります。開発者は時々チートを行い、equals（）とhashCode（）を再定義しないため、異常な動作やデバッグの問題が発生する可能性があります。または、開発者が別のクラスを宣言したくない場合は、「正しい形式」を持っているという理由だけで代替案を規定しますが、それはあまり適切ではありません。

開発環境は、ほとんどのコードをクラスに登録するのに役立ちますが、このコードを読んでいる開発者が数十行のボイラープレートコード間をすばやく移動し、このクラスが通常のデータキャリアであることを理解するのには役立ちません。Javaコードモデリング標準データセットは、記述、理解、および検証が簡単である必要があります。

一見すると、レコードはテンプレートコードの削減を目的としているように見えるかもしれません。「データをデータとしてモデル化する」（データをデータとしてモデル化する）という意味論的な目標を設定しました。セマンティクスが正しい場合、テンプレートコードは、開発者の参加なしですべてを実行します。結局のところ、永続的なデータセットの宣言は簡単、明確、かつ簡潔でなければなりません。

なかった目標


ボイラープレートコードに「宣戦布告」という目標を設定しませんでした。特に、JavaBeanコンポーネントの命名規則を使用して、可変クラスの問題を解決するつもりはありませんでした。プロパティ、メタプログラミング、およびアノテーションに基づくコード生成がこの問題の「解決策」として提案されることが多いにもかかわらず、これらの機能を追加することも私たちの目標ではありませんでした。

説明


レコードは、Javaにおける新しい種類の型宣言です。enumと同様に、書き込みは機能的に制限されたクラスです。ビューをアナウンスし、そのビューに基づいて構築するAPIを提供します。エントリはAPIをプレゼンテーションから分離せず、簡潔です。

エントリには、ステータスの名前と説明が含まれています。状態の説明は、このレコードのコンポーネントを宣言します。オプションで、レコードに本文を含めることができます。例えば：

record Point(int x, int y) { }

意味的にレコードは単純なデータキャリアであるため、自動的に標準要素を受け取ります。

  • 各状態コンポーネントのプライベート最終フィールド。
  • コンポーネントと同じ名前とタイプを持つ各状態コンポーネントのパブリック読み取りメソッド。
  • レコードの署名に一致するパブリックコンストラクタ。対応する引数から各フィールドを初期化します。
  • equals（）およびhashCode（）の実装。これは、2つのレコードが同じタイプで、同じ状態を含む場合に等しいと言います。
  • toString（）の実装。これには、すべての記録コンポーネントとその名前の文字列表現が含まれます。

つまり、レコードの表示は、完全に状態の説明に基づいています。また、レコードの状態に基づいて、equals（）、hashCode（）、およびtoString（）が形成されます。

制限事項


状態コンポーネントに対応するプライベート最終フィールドを除いて、レコードは他のクラスを継承できず、オブジェクトフィールドを宣言できません。他の宣言されたフィールドは静的でなければなりません。これらの制限により、状態の説明自体がビューを定義することが保証されます。

エントリーは最終的なものであり、抽象的なものにすることはできません。これらの制限は、レコードAPIが状態の説明によってのみ定義され、後で別のクラスまたはレコードで拡張できないことを示しています。

録音コンポーネントは最終版です。この制限は、データセットで広く使用されている「デフォルトで変更なし」の原則を実装しています。

上記の制限に加えて、レコードは通常のクラスのように動作します。レコードはトップレベルまたはネストとして宣言でき、ジェネリックにすることも、インターフェイスを実装することもできます。レコードは、new演算子を呼び出すことによって作成されます。書き込み本体は、静的メソッド、静的フィールド、静的初期化ブロック、コンストラクター、インスタンスメソッド、インスタンス初期化ブロック、およびネストされた型を宣言できます。レコードと個々の状態コンポーネントに注釈を付けることができます。レコードがネストされている場合、それは静的です。これにより、レコードに状態を自動的に追加する可能性があるネストされたインスタンスの状況がなくなります。

明示的に宣言されたエントリ


ゲッターの標準実装と、equals（）、hashCode（）、およびtoString（）メソッドは、ほとんどのユースケースで受け入れられますが、開発者は標準実装をオーバーライドするオプションを持っています。ただし、equals / hashCodeメソッドをオーバーライドする場合は特に注意が必要です。

署名はレコードの状態の説明と一致する正規コンストラクターの明示的な宣言に特に注意が払われます。コンストラクターは、パラメーターの正式なリストなしで宣言できます。この場合、状態の説明と一致すると見なされ、出力で対応する正式なパラメーター（this。X = x）からコンストラクター本体を標準で閉じることにより、レコードフィールドが暗黙的に初期化されます。これにより、正規コンストラクターはパラメーターのチェックと調整のみを行うことができ、明示的なフィールド初期化をスキップできます。例えば：

record Range(int lo, int hi) {
  public Range {
    if (lo > hi)  /* referring here to the implicit constructor parameters */
      throw new IllegalArgumentException(String.format("(%d,%d)", lo, hi));
  }
}

記録コンポーネントの注釈


アノテーションは、コンポーネント、パラメーター、フィールド、またはメソッドに適用される場合、記録コンポーネントに適用できます。これらのコンポーネントのいずれかに適用される広告注釈は、必要な要素の暗黙の宣言に適用されます。

記録コンポーネントのタイプを変更するタイプアノテーションは、必要な要素の暗黙の宣言（コンストラクターパラメーター、フィールド宣言、メソッドなど）のタイプに拡張されます。必須要素の明示的な宣言は、対応するレコードコンポーネントの型と完全に一致する必要があり、型注釈は含まれません。
