反射の加速に関する失敗した記事

記事のタイトルをすぐに説明します。当初、シンプルで現実的な例でリフレクションの使用を加速することについて、信頼できる適切なアドバイスを提供するように計画されていましたが、ベンチマーク中に、リフレクションが思ったほど遅く機能しないことが判明しました。しかし、結局、私も測定で間違いを犯したことがわかりました...カットの下のこのライフストーリーの詳細とコメント。この例はごく日常的であり、原則として企業で行われているように実装されているため、私には思えるように、それは人生のデモンストレーションである非常に興味深いものであることがわかりました。 「ストラップ」。

この記事の印象に基づいて作業を開始しました。リフレクションが遅いのはなぜですか。

ご覧のとおり、著者は、アプリケーションを大幅に高速化する優れた方法として、リフレクションタイプのメソッドを直接呼び出すのではなく、コンパイルされたデリゲートを使用することを提案しています。もちろん、IL放出はありますが、これはエラーを伴うタスクを完了するための最も労働集約的な方法であるため、回避したいと思います。

私は常に反省の速さについて同様の意見に固執していたことを考えると、著者の結論に特定の疑問を投げかけるつもりはありませんでした。

企業では、単純なリフレクションの使用に遭遇することがよくあります。タイプを取ります。プロパティ情報が取得されます。 SetValueメソッドが呼び出され、誰もが満足しています。値はターゲットフィールドに飛んだ、誰もが幸せです。非常に賢い人々-上級者とチームリーダー-は、あるタイプの「ユニバーサル」マッパーの別の単純な実装に基づいて、オブジェクトに拡張機能を記述します。これの本質は通常、すべてのフィールドを取得し、すべてのプロパティを取得し、それらを反復処理します。型のメンバーの名前が一致する場合は、SetValueを実行します。タイプの1つがいくつかのプロパティを見つけられなかったミスについては、例外を定期的にキャッチしますが、パフォーマンスを達成する方法もあります。試す/キャッチ。

バイクが機能する前にどのように発明されたかについての情報に完全に武装することなく、パーサーとマッパーを再発明している人々を見ました。私は人々が彼らの素朴な実装を戦略の後ろ、インターフェースの後ろ、注射の後ろに隠しているのを見ました。そのような実装から、私は私の鼻を向けました。実際、実際のパフォーマンスリークは測定しませんでした。可能であれば、手が届いた場合は単に実装をより「最適な」ものに変更しました。以下に述べる最初の測定のため、私はひどく恥ずかしかった。

リヒターや他のイデオロギーを読んでいるあなたの多くは、コードへの反映はアプリケーションのパフォーマンスに非常に悪い影響を与える現象であるというかなり公平な主張を思いついたと思います。

リフレクションの呼び出しにより、CLRは適切なアセンブリを探してアセンブリを移動し、そのメタデータをプルアップし、解析するなどを強制します。さらに、シーケンストラバーサル中のリフレクションにより、大量のメモリが割り当てられます。私たちは記憶を使い、CLRはHZを明らかにし、レースを凍結します。それは著しく遅いはずです、私を信じてください。最新の本番サーバーまたはクラウドマシンの大量のメモリは、高い処理遅延を回避できません。実際、メモリが多いほど、HZの動作に気づく可能性が高くなります。反射は、理論的には、彼のための余分な赤いぼろきれです。

それにもかかわらず、私たちはすべてIoCコンテナーと日付マッパーの両方を使用します。その原理もリフレクションに基づいていますが、通常、それらのパフォーマンスについての質問はありません。いいえ、依存関係の導入と外部限定コンテキストのモデルからの抽象化が非常に必要なため、いずれにしてもパフォーマンスを犠牲にする必要があるからではありません。すべてがシンプルです-パフォーマンスに大きな影響はありません。

実際、リフレクションテクノロジに基づく最も一般的なフレームワークは、あらゆる種類のトリックを使用して、より適切に動作するようになっています。これは通常キャッシュです。通常、これらは式と式ツリーからコンパイルされたデリゲートです。同じオートマッパーは、リフレクションを呼び出さずに相互に変換できる関数と型を照合して、競合する辞書を保持します。

これはどのように達成されますか？実際、これは、プラットフォーム自体がJITコードの生成に使用するロジックと同じです。最初にメソッドを呼び出すと、コンパイルされます（そうです、このプロセスは高速ではありません）。その後の呼び出しでは、制御は既にコンパイルされたメソッドに転送され、特別なパフォーマンス低下はありません。

私たちの場合、JITコンパイルを使用してから、コンパイルされた動作をAOTの対応物と同じパフォーマンスで使用することもできます。この場合、表現が助けになります。

簡単に言えば、問題の原理を次のように定式化できます。
リフレクションの最終結果は、コンパイルされた関数を含むデリゲートの形でキャッシュする必要があります。また、オブジェクトの外部に保存されている型のフィールド内の型に関する情報を含む必要なすべてのオブジェクト、つまりワーカーをキャッシュすることも意味があります。

これには論理があります。常識では、何かをコンパイルしてキャッシュできる場合は、これを実行する必要があります。

将来を考えると、式をコンパイルするために提案された方法を使用しなくても、リフレクションを処理する際のキャッシュには利点があると言えるでしょう。実際、ここで私は上記で言及した記事の著者の論文を単に繰り返しています。

次にコードについて。深刻な信用機関の本格的な生産で私が直面しなければならなかった私の最近の痛みに基づいた例を見てみましょう。すべてのエンティティは架空のものなので、誰も推測できません。

ある実体があります。連絡先にしましょう。パーサーとハイドレーターがこれらと同じ連絡先を作成する標準化された本文を持つ文字があります。手紙が届き、それを読んで、キーと値のペアを分解し、連絡先を作成して、データベースに保存しました。

これは初歩的です。連絡先に、プロパティの名前、年齢、連絡先番号があるとします。これらのデータは手紙で送信されます。また、ビジネスでは、エンティティのプロパティをレターの本文のペアにマッピングするための新しいキーをすばやく追加できるようにしたいと考えています。テンプレートに誰かが刻印された場合、またはリリース前に新しいパートナーからマッピングを早急に開始し、新しい形式に調整する必要がある場合。次に、新しいマッピング相関を安価なデータフィックスとして追加できます。つまり、人生の例です。

ご覧のように、プロパティセッターを持つ静的コレクションが使用されます-セッターエンティティを呼び出すコンパイルされたラムダ。次のコードによって生成されます：

        static FastContactHydrator()
        {
            var type = typeof(Contact);
            foreach (var property in type.GetProperties())
            {
                _proprtySettersMap[property.Name] = GetSetterAction(property);
            }
        }

        private static Action<Contact, string> GetSetterAction(PropertyInfo property)
        {
            var setterInfo = property.GetSetMethod();
            var paramValueOriginal = Expression.Parameter(property.PropertyType, "value");
            var paramEntity = Expression.Parameter(typeof(Contact), "entity");
            var setterExp = Expression.Call(paramEntity, setterInfo, paramValueOriginal).Reduce();
            
            var lambda = (Expression<Action<Contact, string>>)Expression.Lambda(setterExp, paramEntity, paramValueOriginal);

            return lambda.Compile();
        }

"遅い"（ベンチマークでは遅いプレフィックス）：

        protected override Contact GetContact(PropertyToValueCorrelation[] correlations)
        {
            var contact = new Contact();
            foreach (var property in _properties)
            {
                var correlation = correlations.FirstOrDefault(x => x.PropertyName == property.Name);
                if (correlation?.Value == null)
                    continue;

                property.SetValue(contact, correlation.Value);
            }
            return contact;
        }
