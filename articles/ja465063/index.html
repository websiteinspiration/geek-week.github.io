この記事では、Springでの作業中に発生する最も一般的な間違い、特にWebアプリケーションの開発とSpring Bootプラットフォームの使用に関連するいくつかの間違いについて説明します。Spring Bootのウェブサイトに記載されているように、Spring Bootは標準化されたすぐに使用できるアプリケーションを作成するためのアプローチ。この記事では、このアプローチを順守します。Spring Bootに基づく標準のWebアプリケーションの開発に効果的に使用できるいくつかの推奨事項を示します。

一般的な間違い＃6：注釈ベースのデータ検証を使用しない


前の例のTopTalentサービスには、新しいTopTalentデータを追加するためのエンドポイントが必要であると想像してください。また、いくつかの本当に重要な理由により、追加する名前はそれぞれ正確に10文字でなければならないものとします。これは、たとえば次のように実装できます。

@RequestMapping("/put")
public void addTopTalent(@RequestBody TopTalentData topTalentData) {
    boolean nameNonExistentOrHasInvalidLength =
            Optional.ofNullable(topTalentData)
         .map(TopTalentData::getName)
   .map(name -> name.length() == 10)
   .orElse(true);

    if (nameNonExistentOrInvalidLength) {
        //  
    }

    topTalentService.addTopTalent(topTalentData);
}

ただし、上記のコードは構造が不十分であるだけでなく、実際には「クリーン」なソリューションではありません。いくつかのタイプのデータ検証を実行し（つまり、オブジェクトがTopTalentDatanullでないこと、TopTalentData.nameフィールドの値がnullでないこと、フィールドの長さTopTalentData.nameが10文字であることを確認します）、データが正しくない場合も例外をスローします。

これはすべて、SpringのHibernateバリデーターを使用してより正確に行うことができます。まずaddTopTalent、データ検証のサポートを追加して、メソッドを書き直しましょう。

    topTalentService.addTopTalent(topTalentData);
}

@ExceptionHandler
@ResponseStatus(HttpStatus.BAD_REQUEST)

さらに、クラスで実行するプロパティ検証を指定する必要がありますTopTalentData。

public class TopTalentData {
    @Length(min = 10, max = 10)
    @NotNull
    private String name;
}

Springはリクエストをインターセプトし、メソッドを呼び出す前にそれを検証するため、追加の手動チェックは必要ありません。

独自の注釈を作成することで、目的を達成することもできます。実際の状況では、通常、ニーズが組み込みのHibernate制限のセットの機能を超える場合にのみ、独自のアノテーションを使用するのが理にかなっていますが、この例では、アノテーション@Lengthが存在しないとしましょう。文字列の長さをチェックするデータバリデーターを作成するには、2つの追加クラスを作成します。1つは検証用で、もう1つはプロパティに注釈を付けるためです。

@Target({ElementType.METHOD, ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Constraint(validatedBy = { MyAnnotationValidator.class })
public @interface MyAnnotation {

    String message() default "String length does not match expected";

    Class<?>[] groups() default {};

    Class<? extends Payload>[] payload() default {};

    int value();

}

@Component
public class MyAnnotationValidator implements ConstraintValidator<MyAnnotation, String> {

    private int expectedLength;

    @Override
    public void initialize(MyAnnotation myAnnotation) {
        this.expectedLength = myAnnotation.value();
    }

    @Override
    public boolean isValid(String s, ConstraintValidatorContext constraintValidatorContext) {
        return s == null || s.length() == this.expectedLength;
    }
}


これらの場合、責任の分離の原則を正しく適用するには、プロパティの値が(s == nullメソッドでnullの場合にプロパティを有効としてマークする必要があり、さらにこのプロパティに必要な場合isValidはアノテーションを使用する@NotNull必要があります。

public class TopTalentData {
    @MyAnnotation(value = 10)
  @NotNull
    private String name;
}

よくある間違い＃7：レガシーXMLベースの構成の使用


以前のバージョンのSpringで作業する場合、XMLの使用は必須でしたが、現在、ほとんどの構成タスクはJavaコードと注釈を使用して実装できます。 XML構成は、追加およびオプションのテンプレートコードとして機能します。

この記事では（コンパニオンGitHubリポジトリの資料と同様に）、Springを構成するためにアノテーションを使用します。また、SpringBootApplicationコンポジットアノテーションを使用してルートパッケージにアノテーションが付けられているため、SpringはどのJavaBeanコンポーネントをバインドする必要があるかを認識しています。

@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

この複合アノテーション（これについて詳しくは、Springのドキュメントを参照）は、JavaBeanコンポーネントを抽出するためにスキャンするパッケージをSpringプラットフォームに通知するだけです。私たちの特定のケースでは、これは次のco.kukurinサブパッケージがバインディングに使用されることを意味します：

  • 成分 （TopTalentConverter、MyAnnotationValidator）
  • @RestController（TopTalentController）
  • リポジトリー （TopTalentRepository）
  • サービス （TopTalentService）クラス

アノテーションが付いている追加のクラスがある場合は@Configuration、Java構成もチェックされます。

よくある間違い＃8：構成プロファイルを使用しない


サーバーシステムを開発するときの一般的な問題は、異なる構成間の切り替えです（原則として、これらは運用環境と開発環境の構成です）。テストモードと操作モードの間の遷移ごとにさまざまなパラメーターを手動で変更する代わりに、構成プロファイルを使用する方が効率的です。

ローカル開発環境でデータベースをRAMで使用し、アプリケーションの実際の操作の環境でMySQLデータベースを使用する場合を想像してください。これは基本的に、これらのデータベースのそれぞれにアクセスするために、異なるURLと、おそらく異なる認証情報を使用することを意味します。2つの構成ファイルを使用してこれを実装する方法を見てみましょう：

APPLICATION.YAML FILE

# set default profile to 'dev'
spring.profiles.active: dev

# production database details
spring.datasource.url: 'jdbc:mysql://localhost:3306/toptal'
spring.datasource.username: root
spring.datasource.password:


ファイルAPPLICATION-DEV.YAML

spring.datasource.url: 'jdbc:h2:mem:'
spring.datasource.platform: h2


コードを操作する場合、オペレーティング環境向けのデータベースでランダムなアクションを実行したくないと想定する必要があるため、開発環境（DEV）のプロファイルをデフォルトのプロファイルとして選択することは理にかなっています。その後、JVMのパラメーターを指定することにより、サーバー上の構成プロファイルを手動でオーバーライドできます-Dspring.profiles.active=prod。また、デフォルトの構成プロファイルは、オペレーティングシステムの環境変数で指定できます。

よくある間違い番号9.依存性注入メカニズムの使用の失敗


Springでの依存性注入メカニズムの適切な使用は、Springが必要なすべての構成クラスをスキャンすることによってすべてのオブジェクトをバインドできることを意味します。これは相互依存関係を緩和するのに役立ち、テストを非常に容易にします。次のようなクラスを密にリンクする代わりに：

public class TopTalentController {

    private final TopTalentService topTalentService;

    public TopTalentController() {
        this.topTalentService = new TopTalentService();
    }
}

... Springプラットフォームをバインドします。

public class TopTalentController {

    private final TopTalentService topTalentService;

    public TopTalentController(TopTalentService topTalentService) {
        this.topTalentService = topTalentService;
    }
}

Googleのテックトークチャンネルの講義Miskoヒーバーはあなたが依存性注入を使うべき理由を具体的に説明するが、ここでは、このメカニズムが実際に使用されている方法を参照してください。責任分担（「よくある間違い＃3」）では、サービスクラスとコントローラークラスを作成しました。クラスがTopTalentService正しく機能していると仮定して、コントローラーをテストするとします。別の構成クラスを作成することにより、現在のサービス実装の代わりにシミュレーターオブジェクトを挿入できます。

@Configuration
public class SampleUnitTestConfig {
    @Bean
    public TopTalentService topTalentService() {
        TopTalentService topTalentService = Mockito.mock(TopTalentService.class);
        Mockito.when(topTalentService.getTopTalent()).thenReturn(
                Stream.of("Mary", "Joel").map(TopTalentData::new).collect(Collectors.toList()));
        return topTalentService;
    }
}

その後、我々はコンフィギュレーション・ソースとして使用する必要が春のプラットフォームに示す、シミュレータオブジェクトを実装することができますSampleUnitTestConfig：
@ContextConfiguration(classes = { SampleUnitTestConfig.class })

その後、これは、私たちはユニットテストに埋め込むカスタムJavaBeanコンポーネントにコンテキストコンフィギュレーションを使用できるようになります。

よくある間違い番号10.テストの欠如または不適切なテスト


ユニットテストの考え方は決して新しいものではありませんが、多くの開発者は（特に必須ではない場合）それを「忘れる」か、あまりにも遅く費やしているようです。テストはコードの正しい動作を確認できるだけでなく、さまざまな状況でのアプリケーションの動作を示すドキュメントとしても機能するため、明らかにこれは誤りです。
Webサービスをテストするとき、例外的に「クリーン」な単体テストを実行することはめったにありません。HTTP接続の場合、通常はSpringサーブレットを使用してDispatcherServlet、実際のリクエストを受信したときに何が起こるかを確認する必要があるためですHttpServletRequest（つまり、統合が行われます）。検証、シリアル化などを使用するテスト）。エレガントで実績のあるソリューションは、MockMVCでRESTサービスのテストに便利なJavaライブラリであるREST Assuredを使用することです。依存関係注入を伴う次のコードフラグメントを検討してください。

@RunWith(SpringJUnit4ClassRunner.class)
@ContextConfiguration(classes = {
        Application.class,
        SampleUnitTestConfig.class
})
public class RestAssuredTestDemonstration {

    @Autowired
    private TopTalentController topTalentController;

    @Test
    public void shouldGetMaryAndJoel() throws Exception {
        // given
        MockMvcRequestSpecification givenRestAssuredSpecification = RestAssuredMockMvc.given()
                .standaloneSetup(topTalentController);

        // when
        MockMvcResponse response = givenRestAssuredSpecification.when().get("/toptal/get");

        // then
        response.then().statusCode(200);
        response.then().body("name", hasItems("Mary", "Joel"));
    }

}

SampleUnitTestConfigバインドクラスの代理実装TopTalentServiceするTopTalentControllerと、他のすべてのクラスは、Applicationクラスのパッケージに基づいてパッケージをスキャンした標準構成を使用してバインドされています。RestAssuredMockMvc軽量環境を指定し、GETエンドポイントにリクエストを送信するために使用されます/toptal/get。
