DeepCode：側面図


少し前まで、機械学習ベースの静的アナライザーであるDeepCodeは、CおよびC ++プロジェクトの検証のサポートを開始しました。そして、実際に、古典的な静的分析と機械学習に基づく静的分析の結果がどのように異なるかを実際に見ることができます。

記事「プログラムのソースコードの静的分析における機械学習の使用」では、DeepCodeについて既に説明しました。そしてすぐに彼らは、記事「DeepCodeがCおよびC ++のAIベースの静的コード分析サポートを追加する」を公開し、CおよびC ++で書かれたプロジェクトの分析のサポートを発表しました。

DeepCode分析の結果を確認するために、PhysXプロジェクトをチェックアウトすることにしました。ここでは、このプロジェクトでPVS-StudioまたはDeepCodeによって検出されたエラーの分析を提示するという目標はありませんが、サードパーティのプロジェクトをいくつか取り上げて、アナ​​ライザーがどのように機能するかの例を見てみました。

DeepCode分析の実行は非常に簡単です。 GitHubに投稿されたプロジェクトの分析が自動的に開始されます。問題はあるものの、様々なプロジェクトが含まれているものの、リポジトリ全体を一気にチェックしていたため、それらに対する警告が混同してしまいました。また、分析速度は、コードがコンパイルされておらず、アナライザーから多くのエラーが隠されている可能性があることを示しています。

さらに、PVS-Studio診断は、Pi番号または他の頻繁に使用される定数だけでなく、特定の定数（たとえば、Landau-Ramanujan定数）でも機能します。これらの誤った使用は、まれにしか使用されないため、大規模なトレーニングベースでさえも検出が困難な場合があります。

DeepCodeは3.1415という形式の定数に厳密に対応していたため、コード内の定数に小数点以下4桁（3.14159263）を1か所追加すると、操作が消えてしまいます。


PVS-Studioは、定数を丸めるためのさまざまなオプションで機能します。

PVS-Studioはこのコードについてより多くの不満を持っていました：

  1. V809 Verifying that a pointer value is not NULL is not required. The 'if (mFragmentShaderSource != 0)' check can be removed. Shader.cpp 178
  2. V809 Verifying that a pointer value is not NULL is not required. The 'if (mFragmentShaderSource != 0)' check can be removed. Shader.cpp 194
  3. V774 The 'mFragmentShaderSource' pointer was used after the memory was released. Shader.cpp 194
  4. V586 The 'free' function is called twice for deallocation of the same memory space. Inspect the first argument. Check lines: 179, 195. Shader.cpp 195

DeepCodeのトリガーは単純であることがわかり、PVS-Studioは多くの不要なトリガーを生成したと思うかもしれませんが、これはそうではなく、ここでの各トリガーには独自の意味があります。

最初の2つの操作は、free（）関数を使用するためにそのようなチェックが必要ないため、過度のポインターチェックに関連しています。 3番目の操作は、ポインターが解放された後にポインターを使用することが安全でないことを示しています。ポインタ自体が逆参照されずチェックされるだけであっても、それは依然として疑わしく、タイプミスを示していることがよくあります。さて、最後の操作は、DeepCodeが発見した同じ問題を指し示しているだけです。


DeepCodeの記事には、C / C ++サポートの開始に関するスクリーンショットがありました。警告は、バッファオーバーフローの可能性があることを示していますが、バッファの長さは、ホームの長さとラインの長さを考慮して割り当てられ、さらに+ 1が追加されます。したがって、バッファには十分なスペースがあります。

ここでの問題は、メモリが正常に割り当てられたことを確認できないことです。おそらく、この場合のDeepCode警告は何か他のことを示しています。警告の最初の文は、エラーが何であるか、アナライザーが実際に誓うものを理解するのに特に役立ちません。

この場合、おそらく私たちが書いた別の要因がトリガーされます-意味のある警告を作成する難しさ。訓練されたアナライザーの応答を分類した後の警告は、人が書いたものか、コミット/ドキュメントへのコメントから作成されたものです。ただし、エラーパターンが開発者ではなく機械学習によって推定された場合、適切な警告を生成することは困難です。

警告が消えるように、このコードで何を変更する必要があるのか​​と思っていました。しかし、奇妙なことに、新しいファイルの完全に類似したコードのセクションでは、そのような警告は発生せず、別の警告が表示されました。


おそらくポイントは、アナライザーが向けられていた環境がなくなったか、または私が何か間違ったことをしたことです。しかし、警告が不安定であること、つまりその表示が予測できないという事実は、私の意見では、ユーザーだけでなくアナライザー自体の開発者にも不便をもたらし、開発プロセスを複雑にする可能性があります。

一般に、DeepCode応答の大部分はまだ偽ではありませんが、その数は非常に少ないため、現時点での作業の結果は正確ではなく希少です。少数の偽陽性は、原則として少数の陽性を伴います。

結論

もちろん、DeepCodeはC / C ++のサポートを発表したばかりであり、開発が始まったばかりの製品の一部です。あなたはあなたのプロジェクトでもそれを試すことができます、彼らのサービスでは、あなたが誤った警告を与えられた場合に備えて便利なフィードバックウィンドウが実装されています。

ただし、静的分析で機械学習に伴う欠点を確認できます。そのため、静的アナライザーに機械学習を追加するという考えには懐疑的です。その利点は疑わしいからです。まったく同じアナライザーのサポート、ドキュメントの作成または編集、および診断自体の作業が必要です。さらに、開発中に遭遇する問題には、より複雑なソリューションと開発者からの特定のスキルが必要です。これにより、チームを拡大するときに潜在的な候補者の数が減ります。実際、この場合の開発者は、分析された言語の専門家であるだけでなく、機械学習の分野の知識も持っている必要があります。
