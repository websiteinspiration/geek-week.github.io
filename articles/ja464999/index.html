軽量アプリケーションサーバーの利点は何ですか？


以前は、エンタープライズJava EEアプリケーションサーバー（JBoss AS、Oracle WebLogic、IBM WebSphere ASなど）は、特に起動時間と配備時間について話す場合、重くて扱いにくい設計と見なされていました。しかし、クラウドテクノロジーは業界のますます大きな部分を占めており、アプリケーションサーバーの要件は変化しています。

そして今、フル機能を備えた、企業のアプリケーションサーバの代わりに、高速で来る、モジュラー、小さなアプリケーション・サーバは、特定のタスクに焦点を当て：Thorntail、ペーシュ・カショーロマイクロ -弟WildFlyとペーシュ・カショーロ。Meecrowaveは軽量のJAX-RS + CDI + JSONサーバーであり、KumuluzEEはNode.jsやGoなどを使用してJava EEを拡張できるサーバーです。

このリストには、OpenLiberty-WebSphere Libertyが稼働する最新のJava EE / Microprofile標準をサポートするオープンソースアプリケーションサーバー（EPL-1.0に従って配布）も含まれます。

速度：


速度は、クラウドアプリケーションにとって最も重要な指標です。クラウドアプリケーションが急速にスケーリング、管理、増大する負荷に対処するには、数秒で起動する必要があるためです。軽量のアプリケーションサーバーがこれを行うことができます。確認するには、OpenLibertyサーバーをダウンロードし、 bin / server run（コマンドの完全なリスト）を実行します。

モジュール性と柔軟性


ほとんどのアプリケーションは、全体としてJava EEを必要としませんが、エンタープライズアプリケーションで最もよく使用される専用の標準セットを必要とします。OSGIのおかげで、他のすべてを無視して、必要なJava EEおよび/またはMicroProfile標準のセットを選択できます。

たとえば、featureManagerブロックに数行を追加して、Java EEからJAX-RSを宣言し、MicroprofileからmpHealthを宣言します。usr / servers / serverName /server.xml

<?xml version="1.0" encoding="UTF-8"?>
<server description="new server">
 
   <!-- Enable features -->
   <featureManager>
       <feature>jsp-2.3</feature>
       <feature>mpHealth-1.0</feature>
       <feature>jaxrs-2.1</feature>
   </featureManager>
 
   <!-- To access this server from a remote client add a host attribute to the following element, e.g. host="*" -->
   <httpEndpoint id="defaultHttpEndpoint"
                 httpPort="9080"
                 httpsPort="9443" />
 
   <!-- Automatically expand WAR files and EAR files -->
   <applicationManager autoExpand="true"/>
</server>

画像サイズと組み立て


アプリケーションサーバーのサイズが大幅に減少し、
各アプリケーションを個別のアプリケーションサーバーにデプロイしてコンテナーにパッケージ化できるようになり、最大の柔軟性が得られます。さらに、画像はレイヤーで構成されており、アーティファクトを再構築および配布する際に、アプリケーションレイヤーのみがコピーされ、OS、ランタイム、およびアプリケーションサーバーがキャッシュされます。

Dockerhubには、構成済みのOpenLibertyサーバーを含むビルド済みのイメージがあります。それらの1つを使用して、Dockerfileを作成します。

FROM open-liberty
 
COPY usr/servers/defaultServer /opt/ol/wlp/usr/servers/defaultServer
 
ENTRYPOINT ["/opt/ol/wlp/bin/server", "run"]
CMD ["defaultServer"]

画像を組み立てましょう：

docker build -t app .

コンテナーとして実行します。

docker run -d --name app -p 9080:9080 app

結果http：// localhost：9080 / health /



コンテナを停止するには：

docker stop app

コンテナーを使用するシナリオは他にもたくさんありますが、一般的には、これは個々の記事のための機会なので、質問に戻りましょう。

仕事の特異性はどのように変化していますか？


装置


コンテナーイメージは1回だけ収集し、すべての環境で実行する必要があります。したがって、各アプリケーションをアプリケーションサーバーと一緒に収集することをお勧めします。これにより、アプリケーションのライフサイクルと展開が簡素化され、コンテナーテクノロジーの現代世界に完全に適合します。

アセンブリ


これで、異なる技術ブロックを個別のアーカイブにパックする必要がなくなりました。すべてのビジネスロジックは、Webサービスやエンドツーエンドの機能とともに、1つのwarファイルにパッケージ化されています。これにより、プロジェクトのインストールとアセンブリ手順が大幅に簡略化されます。アプリケーションをいくつかの階層レベルにパックしてから、再度1つのサーバーインスタンスにアンパックする必要はありません。

発売


ビルドプロセス中に、アプリケーションサーバーとアプリケーション自体の両方がイメージに追加されます。データソース、ドライブ、またはサーバーモジュールの潜在的な構成も、専用の構成ファイルを追加することにより、ビルドプロセス中に指定されます。構成のすべての違いは、アプリケーション内からではなく、外部から管理する必要があります。したがって、アプリケーションはすでに実行中のコンテナにデプロイするのではなく、コンテナの起動時にアプリケーションを起動するために、自動デプロイメントディレクトリのイメージアセンブリの段階でアプリケーションに追加する必要があります。

機能拡張


コンテナー、展開とスケーリングシステム、監視サービスとサービスグリッドにより、検出、監視、管理、認証、スケーリング、トレース、安定性をアプリケーションの上に構成し、大量のロジックを透過的に別のレベルに転送して、アプリケーションを容易にし、その開発を簡素化する機会が得られました。

アプリケーションサーバーをコンテナにパックする理由


まず、アプリケーションサーバーをコンテナーにパックすることで、各チームがアプリケーションサーバーを個別に構成し、機能の実装に集中できるようにし、開発者が手動操作、ミドルウェア、およびさまざまな承認に費やす時間を節約します。

おまけとして、あなたはコンテナとその周りに構築されたすべてのツールの利点を十分に楽しむ機会を得ます。アプリケーションは、ダウンタイムなしでアーティファクトの組み立てと展開を簡単に管理、スケーリング、更新、自動化できます。
