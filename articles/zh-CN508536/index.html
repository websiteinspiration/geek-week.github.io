NDC伦敦会议。微服务灾难预防。第1部分

您花了几个月的时间将您的整体转换为微服务，最后，每个人都聚集在一起进行切换。您转到第一个网页...什么都没有发生。重新启动它-再次没有任何好处，该站点是如此缓慢，以至于几分钟都没有响应。发生了什么？



吉米•博加德（Jimmy Bogard）在演讲中将对真正的微服务灾难进行“死后验尸”。他将展示他发现的建模，开发和生产中的问题，并告诉他的团队如何将新的分布式整体转变为最终的健全状况。尽管不可能完全防止项目错误，但是您至少可以在设计的早期阶段识别问题，以便最终产品变成可靠的分布式系统。







大家问候，我是吉米（Jimmy），今天您将听到在创建微服务时如何避免发生特大灾难。这是一家公司的故事，我在该公司工作了大约一年半，以防止他们的船与冰山相撞。为了正确地讲述这个故事，您将不得不回到过去，并谈论该公司如何开始以及它的IT基础架构如何随着时间增长。为了保护这场灾难中无辜者的名字，我将该公司的名称更改为Bell Computers。下一张幻灯片显示了这些公司在90年代中期的IT基础架构。这是用于计算机硬件商店的HP Tandem Mainframe大型通用容错服务器的典型体系结构。







他们需要为所有订单，销售，退货，产品目录，客户群建立一个管理系统，因此他们选择了当时最常见的大型机解决方案。这个庞大的系统包含有关公司的所有信息，所有可能的信息以及任何通过该大型机进行的交易。他们把所有的鸡蛋都放在一个篮子里，以为那是正常的。唯一未在此处输入的内容是邮购目录和通过电话下订单。



随着时间的流逝，系统变得越来越多，其中堆积了大量的垃圾。另外，COBOL不是世界上最具有表现力的语言，因此最终该系统变成了一大堆的整体垃圾。到2000年，他们看到许多公司都拥有与他们完全从事整个业务的站点，因此决定建立自己的第一个商业点站点。



初始设计看起来非常不错，并且由一个顶级网站bell.com和多个单个应用程序子域组成：目录catalog.bell.com，帐户account.bell.com，订单order.bell.com，产品搜索search.bell.com。每个子域都使用ASP.Net 1.0框架和自己的数据库，并且所有子域都与系统后端通信。但是，所有订单仍在唯一的大型机中继续处理和执行，大型机中仍留有所有垃圾，但前端是具有单独应用程序和数据库的单独网站。







因此，系统的设计看起来井井有条，合乎逻辑，但是真实的系统如下图所示。







所有元素都相互调用，访问API，并内置第三方dll库等。版本控制系统经常会抓住别人的代码，将其塞入项目中，然后一切都崩溃了。 MS SQL Server 2005使用链接服务器的概念，尽管我没有在幻灯片上显示箭头，但是每个数据库也可以互相通信，因为基于从多个数据库获得的数据构造表没有错。



由于他们现在在系统的各个逻辑区域之间有一定的隔离，因此变成了分散的污垢，最大的垃圾仍然留在大型机后端。







有趣的是，这个大型机是由Bell omputers的竞争对手建造的，仍然由其技术顾问提供服务。在确定其应用程序的操作不令人满意之后，该公司决定放弃它们并重新设计系统。



现有应用程序已投入生产15年，这是基于ASP.Net的应用程序的记录。该服务收到了来自世界各地的订单，每年从单个应用程序获得的利润就达到了十亿美元。利润的很大一部分来自bell.com网站。在“黑色星期五”，通过该站点发出的订单数量达到了几百万。但是，现有体系结构不允许进行任何开发，因为系统元素的刚性互连实际上不允许对服务进行任何更改。



最严重的问题是无法从一个国家/地区下订单，而在另一个国家/地区付款并发送给第三方，尽管这种交易方案在全球公司中非常普遍。现有的网站不允许进行此类操作，因此他们必须通过电话接受并下达此类订单。这导致以下事实：该公司越来越多地考虑更改架构，尤其是向微服务的过渡。



在研究了其他公司的经验之后，他们采取了合理的行动，以了解他们如何解决类似的问题。一种这样的解决方案是Netflix服务体系结构，它是一种通过API和外部数据库连接的微服务。



Bellomputers的领导者决定遵循这样的基本原则来构建这种架构。首先，他们使用数据库共享方法放弃了数据重复。相反，没有发送任何数据，每个需要该数据的人都必须转到集中式来源。其次是隔离和自治-每个服务都独立于其他服务。他们决定将Web API用于几乎所有内容-如果您想获取数据或对另一个系统进行更改，则所有这些操作都是通过Web API完成的。最后一件重要的事情是称为“ Bell on Bell”的新大型机，而不是基于竞争对手硬件的“大型机” Bell。



因此，在这些基本原则的指导下，他们用了18个月的时间创建了该系统，并将其带入了预生产阶段。周末之后恢复工作，开发人员聚在一起，打开了新系统所连接的所有服务器。 18个月的工作，数百名开发人员，最先进的Bell硬件-并没有取得积极的成果！这使很多人感到失望，因为他们反复在笔记本电脑上运行了该系统，一切都很好。



他们采取了合理的行动，把所有的钱都用来解决这个问题。他们安装了带有交换机的最现代的服务器机架，使用的千兆光纤，带有疯狂RAM的最强大的服务器硬件，将所有这些结合起来，进行设置-再也没有任何设置！然后他们开始怀疑原因可能是超时，因此他们进入了所有Web设置，所有API设置，并将整个超时配置更新为最大值，以便他们只能坐下来等待网站发生某些事情。他们等待了9分钟半，直到网站最终加载完毕。



之后，他们意识到目前的情况需要仔细分析，于是他们邀请了我们。我们发现的第一件事-在整个18个月的开发过程中，没有一个真正的“微型”产品被创造出来-一切都变得越来越大。在那之后，我们开始编写事后报告，也称为“追溯回顾”或“悲伤回顾”，又名“怪暴”（类似于暴风雨的“起诉暴风雨”），以了解灾难的原因。



我们有几个证据，其中之一是在API调用时完全流量饱和。使用服务的整体体系结构时，您可以立即了解到底出了什么问题，因为您可以跟踪到唯一报告所有可能导致崩溃的堆栈。如果一堆服务同时访问相同的API，则除了使用诸如WireShark之​​类的其他网络监视工具外，无法跟踪跟踪，因此您可以考虑一个单独的请求并了解其实施过程中发生了什么。因此，我们花了一个网页，将近两周的时间将马赛克的各个部分放在一起，对其进行了各种调用，并分析了每个马赛克的结果。

看这张图片。它表明一个外部请求导致服务进行许多返回的内部调用。事实证明，每个内部呼叫都希望能够独立地为该请求提供服务，因为它再也无法到达任何地方以获取必要的信息了。这张图片看起来像是毫无意义的级联调用，因为外部请求调用了附加服务，而附加服务又调用了其他附加服务，依此类推。







服务相互调用的半圆在该图中以绿色显示-服务A调用服务B，服务B调用服务C，然后该服务再次调用A，结果，我们得到了“分布式死锁”。单个请求进行了上千次网络API调用，并且由于系统没有内置的容错和环路保护功能，因此如果这些API调用中的至少一个失败，则请求将失败。



我们已经做了一些数学计算。每个API调用的SLA均不超过150毫秒，正常运行时间为99.9％。一个请求引起200个不同的呼叫，在最佳情况下，该页面可以在200 x 150 ms = 30秒后显示。自然，这是不好的。将99.9％的正常运行时间乘以200，我们得到了0％的可用性。事实证明，这种架构从一开始就注定要失败。



我们向开发人员提出了一个问题，他们在18个月的工作过程中如何看不到这个问题？原来，他们只读取他们启动的代码的SLA，但是如果他们的服务调用了另一个服务，则他们这次不会在SLA中进行计数。在同一过程中开始的所有内容都遵循150 ms的值，但是访问其他服务过程会多次增加总延迟。从中得出的第一课内容是：“您处置SLA还是SLA处置您”？在我们的情况下，第二个出来了。



我们发现的第二件事-他们知道由Peter Deutsch和James Gosling提出的关于分布式计算的误解概念的存在，但忽略了它的第一部分。它说“网络是可靠的”，“延迟为零”和“带宽是无限的”的说法是误解。误解是“网络是安全的”，“拓扑永远不会改变”，“管理员总是只有一个”，“数据传输价格为零”和“网络是同质的”。

他们犯了一个错误，因为他们在本地计算机上运行服务，并且从未“钩住”外部服务。在本地开发和使用本地缓存时，他们从未遇到过网络希望。在所有18个月的开发过程中，他们从未想过如果我们接触外部服务会发生什么。







如果您查看上图中的服务边界，您会发现它们都是错误的。关于如何定义服务边界的建议来源很多，其中大多数是错误的，例如下一张幻灯片中的Microsoft。







该图片来自MS博客，主题为“如何构建微服务”。它显示了一个简单的Web应用程序，一个业务逻辑块和一个数据库。请求是直接发出的，可能有用于Web的一台服务器，用于业务的一台服务器和用于数据库的一台服务器。如果您增加流量，图片将有所变化。







在这里，负载均衡器似乎可以在两个Web服务器之间分配流量，位于Web服务和业务逻辑之间的缓存以及位于业务逻辑和数据库之间的另一个缓存。这正是Bell用于其应用程序的体系结构-负载平衡和2000年代中期的蓝色/绿色部署。直到一段时间，一切都运转良好，因为该方案是用于整体结构。



下图显示了MS建议如何从单一服务切换到微服务-只需将每个主要服务划分为单独的微服务。在实施此方案期间，贝尔犯了一个错误。







他们将所有服务划分为不同的级别，每个级别都包含许多单独的服务。例如，Web服务包括用于呈现内容和进行身份验证的微服务，业务逻辑服务由用于处理订单和帐户信息的微服务组成，数据库被划分为一堆具有专用数据的微服务。 Web，业务逻辑和数据库是无状态服务。



但是，此图绝对是错误的，因为它没有映射公司IT集群以外的任何业务部门。该方案未考虑与外界的任何联系，因此例如如何获取第三方业务分析尚不清楚。我注意到他们还发明了一些服务，这些服务仅仅是为了发展个人雇员的职业，他们试图管理尽可能多的人，以便为此赚更多的钱。



他们认为，要切换到微服务，仅需采用N层物理层的内部基础架构并将Docker插入其中就足够了。让我们看一下传统的N层体系结构。







它包含4个级别：UI用户界面级别，业务逻辑级别，数据访问级别和数据库。更为先进的DDD（域驱动设计）或面向软件的体系结构，其中两个中间层是域对象和存储库。







我试图考虑此架构中的各个变化领域和职责范围。在典型的N层应用程序中，分类了各种变化区域，这些变化区域从上到下垂直渗透到结构中。这些是目录，在单台计算机上运行的Config设置，以及Checkout检查我的团队正在处理的内容。







该方案的独特之处在于，这些更改区域的边界不仅影响业务逻辑的级别，而且还影响数据库。



让我们看看成为服务的意义。服务定义具有6个特征属性-该软件可以：



  • 由特定组织创建和使用；
  • 负责系统内某种类型信息的内容，处理和/或提供；
  • 可以独立创建，部署和启动它来满足某些操作任务。
  • 与消费者和其他服务进行通信，在协议或合同保证的基础上提供信息；
  • 保护自己免受未经授权的访问，并保护其信息免受丢失；
  • 因故障而运行，以免导致信息损坏。


所有这些特性都可以用一个词“自治”来表达。服务彼此独立运行，满足某些限制，在人们可以接收所需信息的基础上定义合同。我没有提到具体的技术，其本身是暗含的。



现在考虑微服务的定义：



  • 微服务的规模很小，旨在解决一个特定问题；
  • 微服务是自治的；
  • 在创建微服务的体系结构时，将使用“城市规划”城市规划隐喻。此定义来自Sam Newman的书《创建微服务》。


有界上下文的定义来自Eric Evans的域驱动设计。这是DDD的主要模式，DDD是一个与体积体系结构模型一起使用的体系结构设计中心，将其划分为不同的有界上下文，并明确定义它们之间的交互。







简而言之，有界上下文是指可以在其中应用特定模块的区域。在此上下文中是一个逻辑上统一的模型，例如，可以在您的业务领域中看到。如果您从订购人员那里询问“客户是谁”，您将收到一个定义，如果您询问涉及销售的人员，您将得到另一个定义，而表演者将给您提供第三个定义。



因此，Bounded Context表示，如果我们不能对我们的服务使用者是一个明确的定义，那么请定义可以讨论该术语含义的边界，然后定义这些不同定义之间的过渡点。就是说，如果我们从下订单的角度谈论客户，那意味着这个和那个，如果从销售的角度来看，这个和那个。



微服务的下一个定义是封装任何类型的内部操作，以防止工作流的组件“泄漏”到环境中。接下来是``外部交互或外部关系的显式合同的定义''，这是由SLA产生的合同的概念表示的。最后一个定义是一个细胞或一个细胞的隐喻，这意味着将一组操作完全封装在微服务中以及其中存在与外界通信的受体。







因此，我们告诉贝尔计算机公司的家伙们：“我们将无法解决您造成的混乱中的任何事情，因为您根本没有足够的钱来解决这个问题，但是我们将仅修复一项服务以使其物有所值。” 从这一点开始，我将开始讲述我们如何解决唯一服务的问题，以便它比9分钟半后开始更快地响应请求。
