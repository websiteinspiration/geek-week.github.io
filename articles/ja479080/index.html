ゴールデンゲート：1時間あたりのテラバイトを複製する方法、またはVTBのゴールデンゲートでCDCを使用した経験

データをリアルタイムで分析するには多くの方法があります。VTBでは、Golden Gate Oracleツールに実装されているChange Data Capture（CDC）テクノロジーを使用しています。速度は非常に重要ですが、送信されるデータの量を減らし、ソースの負荷を減らしたいと考えています。このツールの主な範囲はOracleとMS SQLからのレプリケーションですが、CDCを長年にわたって使用してきた結果、プラットフォーム間のデータ移行やさまざまなタイプのDBMSなど、興味深いケースがいくつか蓄積されてきました。カットの下では、ゴールデンゲートとの経験を共有します。



CDC（Change Data Capture）が必要な理由


銀行カードの日常的な使用は古くからあり、一般に人々は、決済端末を使用するたびに特定の情報が銀行に迅速に転送されるとは考えていません。データ量は増え続けており、特別オファーをリアルタイムで送信するなど、できるだけ早く処理したいと思っています。なぜなら、彼らが言うように、夕食には良いスプーンだからです。また、従来のETLツール（抽出、変換、ロード-抽出、変換、ロード）は、オンザフライでデータを処理するにはあまり適していません。これは弱いリンクの1つです。ソースシステムのテーブルからデータを取得する場合、新しい行または変更された行のみを選択する必要があります。このアプローチは、ソースシステムをさらにロードし、送信されるデータの量を増やします。

CDCは、データベースログからリアルタイムで変更をキャプチャします。そのため、ソースの負荷が大幅に軽減され、送信されるデータ量が削減されます。このテクノロジーにより、トランザクションの変更が大量にあるシステムのコンピューティングリソースの必要性が減少しました。結局のところ、最も負荷の高い銀行システムでもデータをキャプチャするには、ソースに1〜2プロセッサコアで十分です。また、ETLを導入する場合、並列でデータを読み取るためにプロセッサ容量を購入する必要があります。

私たちはVTBで数年間、OracleのCDCツールであるOracle GoldenGateテクノロジーを使用してきました。その助けを借りて、オンラインデータウェアハウスを満たし、情報システムの機能を「ウォーム」ゾーンと「ホット」ゾーンに分散します。銀行のITランドスケープでのOGGの使用の90％は、OracleおよびMS SQLからのデータの読み込みですが、レプリケーションに加えて、他のタスクにも完全に対応しています。私たちの実践からいくつかの例を見てみましょう。



ケース1.オンラインレポート


2013年にGoldenGateに会いました。次に、カード処理コンプレックスで、トランザクションが同時に処理され、レポートが作成されました。 OLTPの負荷がDWH / DSSの負荷と混合され、大量の重いサンプルがデータベースメモリからキャッシュをフラッシュしました。その結果、高速トランザクションはハードドライブに移動する必要があり、重要なビジネスサービスの速度は低下していました。処理コアをオフロードするために、開発したすべての手順とレポートをOracle Exadataの「ウォーム」レプリカに取得しました。
GoldenGateを使用してデータを複製する方法については、ここで詳しく説明します。簡単に言うと、さまざまなタイプの負荷が混在している高負荷のシステムでは、それらを異なるOLTPサーバーとDWH / DSSサーバーに分散し、それらの間の同期にはGoldenGateを使用します。 「ウォーム」レプリカを強調表示するこのパターンは、他の多くの場合に役立ちます。たとえば、同じアプローチが私たちの不正防止システムで使用されました-すべての報告はOracle Exadata統合システムで行われ、データはGoldenGateを使用してそれらに複製されます。

故障のないシステムはありません。たとえば、開発者がレシーバーのデータを変更すると、データ使用エラーが発生し、GoldenGateプロセスが停止する可能性があります。同期していないデータを除外するために、独立したアービターとしてOracle GoldenGate Veridataを使用します。このツールは、ソースとレシーバーの間のデータを検証するだけではありません。主なことは、Veridataが違いを排除することです。レプリケーションを処理する場合、Veridataは正確なデータ比較と失われたレコードの検出を確実にすることが重要です。比較結果を含む完全なレポートを取得します。これは、信じられないほどの同僚に提示することができます。



ケース2.オンラインストレージでの統合されたレポートとステージング


別のケースは、運用ストレージの構築に関連しています。問題は、運用レポートに加えて、企業ストレージ（ステージング）のデータを準備していることです。さまざまなシステムから収集されたデータに基づいて、運用レポートを生成する必要がある場合があります。オンラインストレージのレベルでこれを行うのが最も便利です。リソースへの負荷を最小限に抑えて高速でデータを取得するために、もう一度GoldenGateを適用しました。

比較のために、以前に一部のシステムで変更の差分をどのように見つけたかを説明しましょう。システム自体がデルタの強調表示またはデータの遡及変更を許可しなかった場合、10 TBのソースのテーブルが、前日のレシーバーの10 TBのテーブルと比較されました。これらの10 TBは最初にソースでキャプチャする必要があり、ソースシステム、CPU、メモリだけでなく、データネットワーク、および比較に関与するシステムにも負荷がかかりました。そして、0.01％で新しいデータのデルタを見つけるためにこれすべて！

GoldenGateはソースに実質的に負荷をかけません。CDCはマガジンを読み取り、完成したデルタを生成するだけです。これにより、インフラストラクチャを大幅に節約できます。 Oracle、MSSQL、Teradata、または単にHadoopに基づく従来のリポジトリであるレシーバーが誰であるかは関係ありません。

この場合、ソースとレシーバーとしてOracleデータベースが使用されていることに注意してください。ソリューションはその有効性を示したので、すべての新しいシステムを共通のオンラインデータウェアハウスに接続し、オラクルだけではありません。GoldenGateのもう1つの利点は、銀行のITランドスケープで使用されているほとんどのデータベースからデータをダウンロードするのに適していることです。



ケース3.リアルタイムでのクライアントへのパーソナルオファー。


ストリーミング分析についてはすでに説明しました。つまり、リアルタイムのクライアントへのリアルタイムオファリング（RTO）の提供です。上級の同志は、銀行業での成功は直接あなたがあなたのクライアントをどれだけよく知っているか、そしてあなたが彼にオファーをすることができるか関連性があるかに直接依存すると言います。言い換えると、クライアントが銀行の提案を利用する可能性は、クライアントのニーズに対する銀行の反応率に反比例します。

使い方？たとえば、トランザクション履歴は、顧客が毎週金曜日に酒屋から購入したことを示しています。ジオポジショニングは、このネットワークのストアがあるショッピングセンターでそれを検出し、モバイルアプリケーションを介して、同じショッピングセンターのグルメストアへの割引の個人オファーを送信します。このようなケースは、銀行にとって最も興味深いものであり、共同ブランドや共同オファーを作成することができます。クライアントは個人および組織です。

オフラインとオンラインのパーツがあります。最初に、顧客はすべてのシステムのデータを使用して事前にセグメント化されます。アナリストとデータ科学者は行動、履歴データを研究し、いわゆるトラップを作成します。主なことは、アクワイアラートランザクション、モバイルアプリケーション、またはその他の利用可能なソースによって追跡できる重要なイベントをキャッチすることです。そしてすでにこのイベントはストリーミング分析によって処理されており、準備されたトラップに基づいてその瞬間に決定が下されます。

CDC GoldenGateの目標は、ソースシステムから分析プラットフォームへのリアルタイムのイベントデータフローを提供することです。GoldenGate for Big Dataライセンスには、Oracle Stream Analyticsも含まれています。その助けを借りて、データサイエンティストは独立してSpark Streamingでデータストリームを処理し、ビジュアル環境でアプリケーションを開発できます。

ケース4.新しいタイプの詐欺に対する運用上の反作用


不正防止システムは完全に閉鎖されており、当然のことです。詳細に専念する人が少ないほど、セキュリティは高くなります。彼らは標準的なケースを扱うのに素晴らしい仕事をしますが、時々標準的なスクリプトに適合しない状況があります。したがって、これらのモデルを非標準のシナリオで補足することが重要です。カードと通貨の取引、場所、決済システムの運用、モバイルアプリケーションでのアクション、ソーシャルネットワークの監視など、さまざまなシステムのイベントの相関に基づいて、常に新しいモデルを開発しています。モデルを変更するには、承認されたプロセス（ビジネスリクエスト、タスク設定、アプリケーションの実装のすべての内部ステージを通過するプロセス）に従う必要があります。
昨年、私たちは、Oracle GoldenGate for Big Dataを使用して、大量の小さなトランザクションがあった最も忙しい従来のシステムから、および詐欺防止システムからOracle Big Data Applianceのメインクラスターへのデータのアップロードをテストしました。HadoopとGoldenGateの両方が転送されたデータの量を処理しました-私たちは幾分驚きました。

1時間あたりのテラバイトおよびその他の調査結果


過去2年間で、GoldenGateにより、転送されるログの量が2倍になりました-1時間あたり約1 TBまで。これで、現時点で私たちのニーズはほとんどなくなります。残念ながら、私たちが遭遇する物理学があります。しかし、スループットを向上させるために、GoldenGate開発チームで積極的な作業が行われているため、これは限界からかけ離れています。同時に、他のベンダーのCDCソリューションを監視して試験運用していますが、Oracle GoldenGateから移行する理由は見つかりませんでした。現在、このテクノロジーは、市場で最も成熟しているテクノロジーであることが証明されています。
