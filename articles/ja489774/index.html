先に書いたように、私のアパートのすべての窓からは、明るいライトのあるにぎやかな通りを見渡せます。夜間や夏の明るい太陽からのランプの明るい光の問題を解決するために、電動ローラーブラインドを使用します。

最初は既成のキットを買いたかったのですが、価格が非常に高く、配色も気に入りませんでした。このアイデアは破棄され、いくつかのセットを自分で組み立てる必要がありました。

220 Vエンジンの選択は、フランスのソムフィまたは中国のドゥーヤのどちらかでした。 Somfyの方が静かですが、価格も高いため、Dooya DM35Sエンジンを選択しました。 「Aliexpress」の留め具と一緒に注文しました。エンジンには、最大位置と最小位置を調整する2つのリミットスイッチがあります。


マウント付きエンジン

次の問題はカーテンを巻くためのパイプでした。メールメッセージのルールによると、180 cm以下の小包を送ることができ、私が持っている窓の1つが2メートルを超えています。最寄りの建設市場で、目的の直径-50 mmのアルミパイプを見つけました。エンジンをパイプに取り付けた後、停電カーテンを見つけることは残っています。メカニズムなしで生地のみを送ってくれると売り手に同意した後、「Aliexpress」から私のサイズの生地を注文しました。

制御モジュール

モジュールはソケットに組み込まれています。そのようなモジュールの場合、モジュール、ワイヤー、スイッチを押すために、すぐに深いソケットを置くことをお勧めします。モジュールには、アラームへの応答、消費電力の計算、リミットスイッチによる校正など、十分な設定があります。


組み立てられたスイッチ

カーテンは各部屋とキッチンに設置されています。カーテンは、カーテンの横にあるスイッチ、またはインターフェースやスクリプトを使用して制御できます。

夕暮れ時にカーテンが自動的に下がり、街路灯が点灯します。彼らは平日と週末に各部屋のプログラムされた時間に従って上昇します。外が暗い場合は、カーテンを下げた部屋でスムーズに点灯し始めます。

稼働日と週末は、ダウンロードされた製造カレンダーに従って計算されます。電報コマンドを使用して、各部屋の翌日の時間を再構成できます。「目覚まし時計」に従ってカーテンを折り畳む前に、30分間確認、キャンセル、再スケジュールを行うことができる電文メッセージが届きます。アクションが従わない場合、カーテンは自動的に折りたたまれます。明るい日には、カーテンは部屋の照明と時間帯に応じて自動的に位置を設定し、ライトがオンになっているかどうかもチェックします。「休暇」コマンドを受け取ると、すべての部屋のすべてのカーテンが下げられます。カーテン管理シナリオカーテン管理シナリオ

マルチメディアシステム


リビングルームにはさまざまなリモコンを備えたマルチメディアデバイスがたくさんあり、それらを1つのボタンで制御したいと考えていました。

インターネットで、アンプ、テレビ、HDMIスプリッターのオンとオフを切り替えるための個別のIRコードを見つけました。pronto_broadlink Pythonスクリプトを使用して、base64にトランスコードされ、バイトの配列に変換されて、IRコードベースに書き込まれました。したがって、デバイスが現在どのような状態にあるかを知る必要はありません。単に信号を再度送信するだけです。

私のアンプはかなり古く、HDMI経由で4K画像を処理する方法がわからないので、HDMIスプリッターを購入しました。それはあなたがビデオとオーディオ信号を2つの別々のHDMIに分割することを可能にします：それらの1つはアンプに接続され、2番目はテレビに接続されます。


アンプおよびその他のデバイス

現時点では、デスクトップコンピュータとサーバーという2つのオーディオとビデオのソースがあります。インターフェイスから、1つのボタンですべてのデバイスを再生用にオンにすることができます。1つの場合のみ、信号の受信はサーバーからのスプリッターで構成され、2番目の構成ではコンピューターから構成されます。将来的には、ゲーム機をスプリッターに接続する予定です。

全員が家を出ると、すべてのマルチメディアデバイスがオフになります。メディアコントロールシーン




プレゼンスシステムは、6つのモーションセンサーとフロントドア開閉センサーで動作します。ドアを開閉した後、単一の占有センサーがトリガーされなかった場合、メッセージが確認とともに一定の間隔でテレグラムに送信されます。2分以内にアクションが実行されないか、「OK」ボタンが押されると、システムは「外出中」モードに切り替わります。このとき移動があった場合、コマンドをキャンセルしてメッセージを削除します。

サーバーを承認する


このモジュールは、許可サービスの単純な実装です。
2つのhttpメソッドがあります。1つはリクエストが内部ネットワークからのものかどうかをチェックし、そうであればJWTトークンを発行してユーザーを認証します。JWTトークンはRSA 256を使用して暗号化されます。秘密鍵はその隣のファイルに登録する必要があります。リクエストが内部ネットワークからのものでない場合、ユーザーはログインページにリダイレクトされます。そこでログインとパスワードを入力してログインする必要があります。ユーザーへの分離は想定されていないため、暗号化された形式のログインファイルとパスワード（SHA 512）が構成ファイルの近くにあります。

MQTTサーバー


MQTTサーバーは、aedesコンポーネントを使用して実装されます。

サーバーは2つのポートで立ち上がります。認証なしでサーバーと通信するための内部ポートと、WebSocketを実装する外部ポートです。すべてのフロントエンドクライアントはソケットを介して接続し、承認、トピックへのサブスクライブ、およびメッセージの発行中に認証を渡します。許可サーバーから受け取ったJWTトークンが入り、公開鍵を使用して検証されます。サーバーからのリクエストの場合、検証は行われません。

前面


自動化に巻き込まれているアパートの各アクティブなコンポーネントは、正面のモジュールです。各モジュールは自己完結型であり、レンダリングに必要なすべてのコンポーネントが含まれています。モジュールは、静的とZ-Waveの2つの集約に分けられます。モジュールは1つのインターフェースを実装する必要があります。

インターフェース
export interface INode {
    id: string;
    type: NodeTypes;
    name: string;
    addWidget: ComponentClass<AddProps> | StatelessComponent<AddProps>;
    model: INodeModelConstructor;
    service?: INodeServiceConstructor;
    smallComponent: ComponentClass<WidgetProps<any>> | StatelessComponent<WidgetProps<any>>;
    bigComponent: ComponentClass<WidgetProps<any>> | StatelessComponent<WidgetProps<any>>;
    dialogEditComponent: ComponentClass<WidgetProps<any>> | StatelessComponent<WidgetProps<any>>;
    dialogViewComponent: ComponentClass<WidgetProps<any>> | StatelessComponent<WidgetProps<any>>;
}


実装例
class Widget extends BaseWidget implements INode {
    model: INodeModelConstructor = Model;
    manufacturerid: string = '0x0086';
    producttype: string = '0x1a02';
    productid: string = '0x0064';
    addWidget: StatelessComponent<AddProps> = AddWidget;
    service: INodeServiceConstructor = Service;
    smallComponent: StatelessComponent<WidgetProps<Model>> = SmallComponent;
    bigComponent: StatelessComponent<WidgetProps<Model>> = BigComponent;
    dialogEditComponent: ComponentClass<WidgetProps<Model>> = DialogEditComponent;
    dialogViewComponent: ComponentClass<WidgetProps<Model>> = DialogViewComponent;
}
export default Widget;

結論


現在のところ、家の快適な気候と照明を維持するという私のニーズに合わせて、家のほぼ完全な自動化を実現しています。一部の自動化により、発熱体側と照明側の両方から、電力消費量を削減できました（自動化なしの同じ機器と比較して、冬は約1,500₽）。

もちろん、アルゴリズムにはまだバグがありますが、いつか私はそれらをキャッチします。少なくとも私はこの方向に進んでいます。システムは安定して動作します。Z-Waveプロトコルの速度が非常に遅いにもかかわらず、ブレーキは表示されません。

もちろん、システムは節約を犠牲にしてそれ自体で利益を得るわけではありません。快適さの問題があります。このような自動化は、カーテンの開閉、光の制御などの点で非常に湿っています。普通のアパートにいるときは、照明を消したり消したりするのを忘れたり、カーテンを開けたりしません。

手で気候システムを調整することはめったにありませんが、エアコンのオン/オフを切り替えたり、バッテリーの温度を調整したりする必要がある場合があります。一般的に、私は結果のシステムに満足しています。これにより、調整に費やす時間が短縮されます。コントローラーに障害が発生した場合、すべての機能はそのまま維持されます（オートメーションのみがオフになります）。止水機能も維持されます。アパートを売却することにした場合は、すべての自動化を簡単に削除します。またはそのままにして、もっと売る。
