サービスのAPIを作成する理由


数万人の起業家である私たちのクライアントは、銀行ソリューション、オンラインストア、製品会計、CRMなどのクラウドソリューションを積極的に使用しています。 1つに接続されている-と停止することはすでに困難です。そして今、5番目、8番目、10番目のサービスは起業家の仕事を容易にしますが、ユーザーはこれらのクラウドサービス間で手動でデータを転送します。仕事は悪夢に変わります。

明白な解決策は、ユーザーがクラウドサービス間でデータを転送できるようにすることです。たとえば、データをファイルとしてインポートおよびエクスポートし、目的のサービスにダウンロードできます。ファイルは通常、各サービスの形式に変更されます。これは多かれ少なかれ単純な手作業ですが、これらのサービスの数が増えると、実行するのがより難しくなります。

したがって、次のステップはAPIです。これにより、クラウドサービスは、複数のサービスを一度に接続することでメリットを得ます。そのようなエコシステムの出現は、追加の機会により新しい顧客を引き付けます。新しい機能を備えた製品は、より収益性と有用性が高くなります。

独自のプログラムインターフェースを作成すると、APIのおかげで製品について知っているプログラマーの形でサードパーティの売り手が集まります。彼らは提案されたAPIに基づいてソリューションを構築し始め、顧客のタスクを自動化することでお金を稼ぎます。

MyStoreの会計システムは、単純なプロセスで構築されています。主なものは、主要なドキュメント、商品を受け入れて発送する機能、および主要なドキュメントに基づいてビジネスレポートを受信する機能を使用することです。たとえば、クラウドアカウンティングへのデータ転送や、銀行システムや小売店からのデータの受け取りもあります。また、オンラインストアも利用しています。商品に関する情報を受け取り、残高に関するデータを送信します。



最初のMyStore API


MyStoreがAPIを使用して10年以上にわたって、データの交換、銀行との連携、支払い、外部電話の使用を可能にするあらゆる種類の統合を獲得してきました。

1年目は、XML形式のデータをアップロードできるようにしました。次に、ユーザーがデータをオフラインで保持することは、理解しやすく、親しみやすくなりました。ある種のクラウドではなく、これを提供しました。アンロードは、インターフェースからの手動エクスポートによって開始されました。つまり、APIをまだ呼び出すことができませんでした。

それから私たちはルサグロと協力し始めました-彼らはすでに「大人の」ERPを使って生産と販売を計画していましたが、工場での車の積み込みはMySkladで自動化されました。つまり、実際のAPIの最初の始まりです。サービスとERPの間の交換は、すべてのタイプのドキュメントのデータを含む大きなファイルを送信することによって行われました。

これはバッチデータ交換に適したオプションですが、ドキュメント、商品、請負業者、倉庫に関する情報など、依存関係を転送する必要がありました。このような埋め立ては、エクスポート中に生成することはそれほど難しくありませんが、すべての情報が1つのパッケージ（新しいドキュメントと既存のドキュメントの両方）で提供されるため、インポート中に分解することはかなり困難です。

最初のXML APIは長くは続かなかった-2年後、私たちはそれを再構築し始めました。彼の仕事を始めたときでさえ、ソフトウェアインターフェイスの構築時にいくつかの間違いを犯しました。

2010年、私たちはオンライン会計を使用した交換システムを構築しようとしました-BukhSoft。離陸しませんでした。しかし、統合の過程で、本格的なAPIが登場しました。REST交換サービスで、RPC呼び出しの形で操作を呼び出すような自由はありませんでした。 APIとのすべての通信は、残りの標準モードに移行しました。クエリ文字列にはエンティティの名前が含まれ、それを使用して実行される操作はhttpメソッドを使用して設定されます。エンティティの更新時にフィルタリングを追加しました。これで、ユーザーは自分のシステムでレプリケーションを構築できるようになりました。

同じ年に、倉庫と商品のバランスを降ろすためのAPIが登場しました。システムの最も重要な部分は、APIを通じてユーザーが利用できるようになりました-一次ドキュメントの交換と、商品の残高とコストに関する推定データ。

2015年12月、RetailCRMはAPIにアクセスする最初のサードパーティライブラリを公開しました。彼らはそれをかなり積極的に使用し始めましたが、サービス全体の人気は高まりましたが、APIの負荷はWebインターフェースの負荷よりも速く増加しました。かつて、成長は負荷の急上昇に変わりました。





そして、左の矢印で示されているこのジャンプは、APIを提供するサーバーの完全な驚きをもたらしました。 1週間、この負荷が正確に何を生成するかを理解しました。これらは、クライアントのフロントからAPIにブロードキャストされるまさにその要求であることがわかりました。約50人のお客様が食べました。そのとき、私たちは自分たちの過ちの1つである、限界の完全な欠如に気づきました。

その結果、同時リクエスト数に制限を設けました。1つのアカウントから同時に2つまでのリクエストを開くことが可能になりました。これは、バッチモードでデータを交換するためのレプリケーションモードで動作するのに十分です。また、ソフトウェアツールにいくつかのアカウントでの作業を導入したため、その瞬間から、私たちをバックエンドとして使用したいと考えていた人々は、関税の遵守を余儀なくされました。

今日のMyStore API


今日、MyStore APIは多くの問題を解決します：

  • オンラインストア、会計システム、銀行とのデータ交換。
  • 決済データ、レポートを受け取る。
  • クライアントアプリケーションのバックエンドとして使用-私たちのモバイルアプリケーションとデスクトップのキャッシュデスクはAPIを介して動作します

  • ロイヤルティシステム。
