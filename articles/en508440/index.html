10 ideas that all programmers should know about

I write in Python and Go, and in recent years I have been engaged in large-scale applications. It is about the fact that every day I and my team have to maintain systems responsible for ensuring the work of approximately two million users. This is not an easy task. Here I want to share a few valuable ideas that I have met over the years of work.







1. No need to solve security issues last



Application security systems are highly discouraged to implement in the last turn, or to reduce everything to the following thought: "Let's do security somehow later." In order to create a secure application, security issues must be on the agenda from the very first day of the project. If you leave the development of the security system to the last deadline, then, in fact, it will only increase the development time, since you will have to go back to the already written code and refactor it to bring it in line with the security needs. Or everything can turn out even worse: due to the lack of time for ‚Äúplugging‚Äù all the ‚Äúholes‚Äù, everything will end up in the production code that contains vulnerabilities. And this, as you can easily see, by analyzing the experience of different companies, such as Yahoo , can end very badly.

3. Not everyone needs microservices



Microservices are extremely attractive. I understand that. The idea that you can independently scale different parts of an application is amazing. It is able to justify the additional volume of work that is necessary for a certain project. But let's be honest with ourselves: you may not need microservices. Suppose we are trying to support the idea of ‚Äã‚Äãusing microservices with the following considerations: ‚ÄúI want to separate the opportunity code X from the opportunity code Y‚Äù, ‚ÄúI want the bad code from one part of the application not to penetrate into other parts‚Äù, ‚ÄúWe need to minimize the consequences of a possible hacking application.‚Äù But all this cannot be called reasons, guided by which it is necessary to switch to microservices. These, respectively, are symptoms of the use of bad approaches to development (do not go into other people's affairs, change only what is necessary),low standards code review (low-quality code has nothing to do in the master branch) and security flaws.



Do you need to independently scale different parts of the application? Do you have any problems with the speed of some components of the project, like mechanisms for entering the system? If so, then it is probably worth exploring the possibility of separating those components that need to be scaled into separate services.



Maybe your project uses architecture based on virtual servers, and you want to reduce costs? In this case, you do not need microservices. You, using them, at best, will remain at the same level of expenses. And in the worst case, you have to use additional servers. Suppose you have a monolith with five services. You broke it into microservices. You now have five applications. And this means that there are two possible scenarios. The first is if you have to run each of them on a separate server, which increases the number of your servers. The second option is if you will start all this on the existing infrastructure, spending more on supporting and administering the project.



4. A standardized development environment is good



If you lead a development team, then one of the most useful decisions you can make is the decision to use a single development environment. Moreover, I do not mean the need to build something like a containerized virtual environment. You can do this if you want, but even applying a simple rule, such as using the same version of a language, can work wonders. For example, you can seriously get on your nerves by trying to debug the error in Go 1.12, while your colleague, who turned to you for help, writes code for Go 1.11. Coordinating updates can be quite a difficult task, but if you approach the organization of this process correctly, everything will be done easily and pleasantly.



5. The implementation of the subsystem responsible for configuring the application may be more complex than it seems at first glance. Therefore, such things need to be planned.



Unlike what can be read on some popular sites, creating a subsystem responsible for setting up the application is a bit more complicated than ‚Äúputting everything into environment variables‚Äù. I believe that there must be at least four ways to configure applications: default values ‚Äã‚Äãspecified in the code, local configuration file, command line flags, environment variables, remote settings source (like HashiCorp Consul). I would call the remote settings source optional. But the other four mechanisms are simply necessary. If during the development process, just to start the application, you need to write 27 values ‚Äã‚Äãinto the environment variables, then, to put it mildly, this is not so convenient. Also, maybe you need a higher level of automation and a Makefile? If you provide for customizing the application through a local file, likeapplication.yaml



, this can greatly facilitate the description of the default configuration used during development. In addition, if the code contains default values, this means that you need to set these values ‚Äã‚Äãonly when you need something different from the standard settings. Command line flags are very useful in situations where applications are launched using various automation tools, like systemd



. This makes it easier to analyze the configuration when tracing processes. Of course, environment variables are extremely useful when running applications in containers, but there are some tasks for which they are not suitable. For example, environment variables should not be used to store passwords.



It is strongly recommended not to write something secret into environment variables: passwords, authentication tokens, certificates, in general - everything that you would not want to be made public. The problem here is that almost any process running on a computer can read the values ‚Äã‚Äãof environment variables. To store sensitive data, you need to use some specialized solution. Personally, I chose HashiCorp Vault for these purposes , but I do not urge to use this particular service. Choose what works best for your project.



6. Dependencies should be used when they are needed.



I suppose everyone remembers the left-pad apocalypse ? How did the 11-line package that was removed from the repository break essentially the entire Internet? This incident leads to one simple thought: ‚ÄúYou should not, without real need, add additional dependencies to the project.‚Äù Dependencies should be used only when there is a reasonable need for them. What dependencies are really needed in projects? For example, it could be a package that is an SDK for the platform you are working with (AWS SDK, for example). It may be some good abstraction over inconvenient standard libraries (which is why programmers use the Requests Python library instead urllib



). It could be some kind of widely used framework - like an Echo HTTP serverwritten in Go, or the Flask WSGI server written in Python. Some libraries that implement convenient methods that are not in the language, such as the Lodash library for JavaScript projects, also belong to ‚Äúreasonable‚Äù dependencies.



External dependencies should simplify development and should not force the programmer to write some auxiliary code. It is in the simplification of development that the strength of package managers lies. But, as in the case of the left-pad, using the package manager it is easy to fall into the trap when such an approach is used for any task: "There is a library for this, so I will use it." With every dependency imported into a project, the risk that the project may become unstable, unsafe, or simply unattended increases.



This risk also increases every package used by the new dependency. Such packages are called transitive dependencies. If you import only one dependency into a project, which, in turn, imports five packages, then your project also becomes dependent on these packages, which means that all the problems of these packages become its problems. And in my opinion, and in the opinion of other programmers, direct dependencies of projects should not have transitive dependencies. But this is not always possible. Although, if we are talking about very large packages, then they should have as few dependencies as possible. And if users of such packages need some additional functionality, they should be able to independently connect such functionality.



When making decisions about importing dependencies into my projects, I follow a simple rule: if I can write something in 10-15 minutes, then I do it. Otherwise, if possible, I will import a certain library into the project. This rule can save you from importing all kinds of junk into the application, but at the same time it is quite flexible and does not force the programmer to write his own HTTP server every time he needs to organize the work of the API.



7. There is no need to write abstractions until a real need arises.



It is very easy for a modern programmer to fall into the trap of abstracting everything and everyone. Thoughts that something may need to be reused can lead the developer into the dark and terrible jungle of OOP. And I managed to understand the reason for this. The principles of DRY (Don't Repeat Yourself, do not repeat) are deeply rooted in our minds. And not without reason. But there is a border, crossing which we spend too much time on abstracting entities, and too little on code writing. So here's a tip for you: ‚ÄúJust write the code!‚Äù If you are faced with the need to implement a function similar to something that you have already written, then you can return to the old code and start creating an abstraction. But, again, here you need to know the measure. I apply this rule here: if we are talking about a small function in three lines, then probablyit can simply be written, albeit repeated. But if the function contains only three lines of code, then I will probably wonder what, perhaps, this code should not be written as a function.



8. Sometimes your project is ‚Äúburned‚Äù and ‚Äúreborn from the ashes‚Äù



This is the scariest idea I am sharing here. This idea makes managers nervous. It spoils the mood of the product owners. She angers programmers. But ‚Äúrebirth from the ashes‚Äù is absolutely necessary. Starting from time to time is a good thing. This allows you to throw out any garbage from the code. This makes it possible to implement new ideas without rewriting half of the existing code base. This forces everyone who works on the project to overestimate it.



Imagine that your project is a huge forest. Each line of code is a mighty pine. As the forest grows old, it overgrows with undergrowth, it is covered with pine needles, cones, dry branches ... These are your problems, your technical duty. All this tends to accumulate. This process, in itself, never stops. Only some radical changes can stop it. In the forest, fires bring change. Fire cleans the forest of all unnecessary. Strong trees remain, and every little thing burns out. Although the fire may seem like a catastrophe that will destroy the forest, there is one secret: the forest is waiting for the fire. Waits patiently for years. The forest hopes that the fire will purify it and cause change. The fact is that while the flame is raging under the crowns of pines, seeds ripen in their cones, from which new powerful trees will grow.While the fire burns the forest litter, it makes room for the tender young trees, which will take their place next to the surviving forest veterans, scorched by fire. That is exactly what applications should be: their reliable, well-written parts will survive the ‚Äúfire‚Äù, and the place of what will be ‚Äúburned‚Äù will be replaced by new ideas and new code. And the project, like the Phoenix, will be reborn from the ashes.
