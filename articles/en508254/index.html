Isolated Tests



I believe that the division into ‚Äúmodular‚Äù and ‚Äúintegration‚Äù tests is not clear enough. Such categorization leads to problems and disputes, although we have one goal: to facilitate software change with the help of quick and accurate feedback when you break something.



Therefore, I prefer to talk about "isolated" tests.



Not sure if this term is better. But I like it for now, because it makes me ask an extremely important question: isolated from what?



My answer: isolated from what is difficult to test.



What is difficult to test



Accident, time, network, file system, databases, etc.



I call all this "infrastructure", and the rest is called a "domain." The combination of domain and infrastructure looks like a very useful and practical look at software design. And not only for me. This separation is at the core of many maintainable architectures, such as Hexagonal Architecture (known as Ports and Adapters), Onion Architecture, Clean Architecture, and Functional Core / imperative shell, etc.



All these are the nuances that say one thing: it is better to separate the domain and infrastructure. Functional programming also promotes this idea: isolated side effects on the perimeter of your system. Side effects are infrastructure. And it's hard to test.



Yes, but how does it help?



In short, this tells you exactly what to remove and replace with plugs. Do not transfer business logic to infrastructure, otherwise you will be tormented with testing.



An isolated domain is easy to test. You do not need to do stubs for other domain objects. You can use the same code that is used in the prod. You just need to get rid of the interaction with the infrastructure.



‚Äú But there can be bugs in the infrastructure code! ¬ª



If you keep it compact enough, then the risks are reduced. But basically, you reduce the size of the code that you need to test when integrating with a real database.



The bottom line is this: you should be able to test all kinds of behavior of your application without saving data in PostgreSQL. Need an in-memory database. You still need to check the integration with PostgreSQL, but just a few tests are enough , and this is a noticeable difference!



It was all a theory, let's get back to reality: we have code that has not yet been tested.



Going back to outdated code



I think this view of testing will help you. No need to get carried away with stubs, but when working with outdated code you will have to get carried away. Temporarily. Because this code is a vinaigrette from the fact that it is difficult to test (infrastructure) and business logic (domain).



You need to strive to extract infrastructure code from the domain.



Then make the infrastructure dependent on the domain, and not vice versa . There should be as little infrastructure as possible in business logic. You will get fast, reliable, isolated tests covering most of the application logic, and several integration tests to test the operation of the infrastructure code with a real database, external API, system clock, etc.



Do you still have stubs? Yes and no.



You will have alternative infrastructure implementations, so you can still execute production logic without using a real database.



How to find out that the database implementation is working correctly? Write some integration tests. Just enough to check the correctness of the behavior. JB Rainsberger calls it ‚ÄúContract Tests.‚Äù I think it‚Äôs best to explain with this example .
