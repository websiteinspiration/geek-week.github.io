WSUS Package Publisherの落とし穴

Package Publisherの展開方法と、インターネットで見つけられなかった問題について
ファイル\\ [serverName] \ UpdateServicesPackages \ AppName_abf10b91-bfa6-44ff-aa54-099e4bf1487d \ a7f3d4b2-02b6-4f0c-ab9b-e38c8de9c3f0_1.cabの署名を確認できません
ファイルのファイル署名の検証に失敗しました：\\ [serverName] \ UpdateServicesPackages \ AppName_abf10b91-bfa6-44ff-aa54-099e4bf1487d \ a7f3d4b2-02b6-4f0c-ab9b-e38c8de9c3f0_1.cab

どういうわけか、彼らは私に1000台のマシン用の小さなグリッドに更新サーバーをデプロイするタスクを設定しました。一般に、管理は私の主な仕事ではありません。過去2年間、私は非常に大きな休暇中にのみ目に見える窓を見てきました。しかし、最愛のITサービスは言った：「あなたは警備員のためにこれが必要なので、それをしてください。」

それで、私の意志を一握りにまとめて、WSUS展開マニュアルを読みに行きました。そして、すべてがシンプルで明確であり、発生する可能性のあるすべての問題がすでに誰かに遭遇し、フォーラムで長い間説明されてきた場合、Package Publisherに多くの質問がありました。

なぜ彼はまったく必要だったのですか？ Microsoftのシステムとアプリケーションだけでなく、サードパーティのもの、特にFirefoxも一元的に更新する必要があったためです。そして、それがすでにインストールされているマシン上でのみ。 （代替として、LUPも考慮されました。機能はほぼ同じですが、フォーラムの親切な人々は、サポートされなくなったため、WinServ2016との統合ははるかに難しいと述べています。）

そこで、WSUSが展開されました。 Windowsを愛すべき理由は、「次へ->次へ->完了、楽しいです」です。 Package Publisherの時間です。原則として、インターネット上で彼につながるすべてのリンクがここにつながります。インストールプロセスを詳細に説明しているgitへのリンクもあります。つまり、アーカイブをダウンロードして解凍し、「Wsus Package Publisher.exe」を実行します。

リヌカでは、githubでリポジトリのクローンを作成するのに慣れました。しかし、Windowsを愛すべきではないものについては、すべてがそこで機能するわけではありません。緑色のボタンをクリックするだけでリポジトリをダウンロードすると、恐ろしいことに、アーカイブにEXEファイルが存在しなくなります。真剣に、私は20分間、漁獲量とは何か、どこでそれを失ったのかを理解しようとしました。特定のリリースをダウンロードするだけでよいことがわかりました。

インストールは喜ばしい、またはむしろその不在。 EXE-shnikはインストールなしで起動し、WSUS（同じマシンにデプロイされた）を検出し、それに接続すると、証明書がないこと、およびアップデートを公開できないことに関するメッセージを表示します。

次のステップは、WSUS Package Publisher証明書（ツール->証明書）をフィードすることであると想定するのは当然です。自己署名を生成できます。しかし、私は本当にそれをしたくありませんでした。さらに、同僚は最近、ローカルの認証サーバーを展開しました。興味深いことに、証明書のダウンロードボタンは、パスフレーズを入力した後にのみアクティブになります。 "閉じる。" mmcコンソールで、必要な証明書が "WSUS"コンテナーにあること、および "信頼された発行元"と "信頼されたルート証明機関"に関連付けられているすべてのものを確認した後、WSUSを再起動した後、幸せになることを願っています。うん！

更新を作成するとき（Firefox でこれを行う方法についてはこちらをご覧ください）、最後のステップはエラーです。「ファイルのファイル署名の検証に失敗しました：
\\ [serverName] \ UpdateServicesPackages \ AppName_abf10b91-bfa6-44ff-aa54-099e4bf1487d \ a7f3d4b2-02b6-4f0c-ab9b-e38c8de9c3f0_1.cab "（Unable to verify the signature for the file ...）グーグルは、その理由は、「信頼されたルート認証局」コンテナに証明書が欠落しているためだと述べています。しかし、彼はそこにいました！そして彼だけでなく！私がそれを入れようとしなかったところ。無駄に。

失敗した試行の1時間半後、私はあきらめて、まだ自己署名WPP証明書を使用することにしました。 mmcコンソールを開いて見たものを信じないでください。コードに署名するための


特別な証明書

が生成されます。つまり、コードに署名するために特別に証明書を生成する必要があります。さらに重要なことに、秘密鍵はエクスポート可能でなければなりません！そして、それはGPOの助けを借りてネットワークマシン（ここでは既に秘密キーなし）に証明書のチェーンを配布することで技術的な問題であり、アプリケーションを一元的にインストールおよび更新できます。

そのため、ファイルエラーまたはその他の同様の理由でファイル署名の検証に失敗した場合：

  1. ローカルのコード署名認証局にPackage PublisherがインストールされているWSUSの証明書を生成します。秘密鍵はエクスポート可能でなければなりません。
  2. 秘密鍵とともに証明書をエクスポートし、秘密鍵を入力した後、それをPackage Publisherに追加します。WSUSを再起動しています。
  3. 秘密鍵なしでエクスポートし、クライアントマシンに配布します。
  4. すべてのアプリケーションを一元的に更新およびインストールし、人生を楽しんでいます。
