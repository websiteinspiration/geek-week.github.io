執筆時には、マーケティングチームと成長チームに焦点を当てましたが、他のチームにはこのプロセスが適していない可能性があることに注意してください。



アイデア≠仮説



メインプロセスに着手する前に、仮説に基づく思考をチームに導入する必要があります。アイデアは、元の形式ではまだ仮説ではないことを理解することが重要です。



仮説の主なプラスは、測定可能であることです。仮説のテストを開始した場合、どのメトリックを改善するかが正確にわかります。



仮説式



「アクション[X]は[N]なので、メトリック[Y]を[Z]ずつ増加させます。」



違いの例を見てみましょう。



アイデア：

「オンライン申請フォームを5ステップから4ステップに短縮する必要があります。登録への変換を増やすと思います。」



仮説：

「オンライン申請フォームを5ポイントから4ポイントに減らすと、登録への変換が2％から3％に1％ポイント増加します。分析システムでは、一部のユーザーがステップ4で離れていることがわかります。登録を完了せずに。」



アイデアを仮説の形にすることができない場合は、それを放棄する価値があります。正しく定式化された仮説は、いかなる方法でもメトリックに影響を与えない無意味なアクションを殺します。

仮説のバックログを形成します







私たちは仮説とは何か、そしてそれがアイデアとどのように異なるかを決定しました。

今度はバックログを仮説で満たす時です。



仮説のバックログは、単一のベースであり、仮説は順序付けられた形式になっています。



仮説テストには費用がかかる（開発者、アナリスト、マーケター、デザイナーなどの時間）ことに注意してください。仮説がメトリックにまったく影響しないことを理解している場合は、追加しないでください。



IT製品について話している場合は、すべての仮説をAAARRRファネルに入力できます（上の図では、それを概略的に示しています）。

目標到達プロセスの各ステップを分析することにより、各ステップのより多くの成長ポイントを見つけることができます。



たとえば、獲得ステップを実行し、チャネル/キャンペーン/グループ/クリエイティブなどによってすべての魅力のチャネルを分析します。分析中に、新しいユーザーの魅力を大幅に改善できる仮説を書き留めます。



成長ポイントを見つけるのに役立つ他のツール：



  • CJM（製品の顧客パス）。
  • JTBD。
  • 製品からの定量分析。
  • UX分析とCustDev。


トピックは非常に広範囲であり、個別の記事に値するので、それぞれの方法については詳しく説明しません。



バックログの推奨：



  • チームは、透明な作業とテイクの回避のために、仮説のバックログを1つ持つ必要があります。
  • AAARRRファンネルのどの部分に仮説が適用されるかを示します。
  • したがって、一見すると、それが影響を与えるファンネルのレベルがどれであるかがわかります。


ICEおよびPXLの仮説の優先順位付けとスコアリング



バックログを入力し、AAARRRファネルのステップに沿って仮説を配布しました。ここで、最初にテストしている仮説を理解することが重要です。



したがって、優先順位付けとスコアリングはスムーズに行われます。



部門のすべての従業員がバックログに仮説を追加し、誰もが彼の仮説をテストのキューの最初に配置したいと考えているとします。誤解や矛盾を避けるために、仮説に優先順位を付けてスコアリングするプロセスは、できる限り透明でなければなりません。これは、ICEスコアフレームワークに役立ちます。



ICEスコア-これ：

信頼性 -影響と実装の容易さの評価に自信を持っていることを示します。



簡易（実装の容易さ）は、このアイデアを実装するために必要な労力とリソースの評価です。



式のICE：

実験計画



仮説に優先順位を付けたので、今度は実験を設計する必要があります。



実験の設計は、仮説をテストするための詳細な技術的タスクであり、テストする対象、どのサンプルおよびどのメトリックを検討するかを理解するために必要です。



実験のデザインはそのようなテンプレートに基づいています：



  • 仮説。
  • 何してるの？
  • ユーザーがテストしているもの。
  • 実験を評価するための主要な指標。
  • 期待される効果。
  • 実験結果に応じたアクションプラン。

試験準備



実験を開始する前に、いくつかの準備手順を実行する必要があります。



  1. テストのタイプを決定します。
  2. 標本サイズ、信頼区間、検出力を計算します。
  3. 実験の期間を計算します。
  4. テストツールを準備します。
  5. 実験がきれいであることを確認してください。
  6. 分析システムをセットアップします。
  7. テスト用のセグメントを設定します。


すべての手順を順番に実行してみましょう。



1.テストのタイプを決定する



適切なテストオプションを選択してください。



ここにいくつかのオプションがあります：



  • A / Bテスト。
  • MVT（多変量テスト）。
  • A / Bテストの劣化。
  • A / A / Bテスト。


この例では、最も頻繁に使用されるオプションであるA / Bテストを検討します。



2.サンプルサイズ、信頼区間、検出力を計算する



開始する前に、最終的にデータが統計的に有意になるように、A / Bテストオプションのそれぞれに引き付ける必要があるユーザーの数を決定します。Evan Miller計算機を



使用してサンプルを計算します（下のスクリーンショットを参照）。



「ベースラインコンバージョン率」では、コントロールバージョンのコンバージョンを示します。この例では2％です。



「最小検出可能効果」では、コントロールバージョンの変換が何パーセント増加するかを示します。この例では、+ 1％増加します。



有意水準α（信頼水準）は、第1種のエラー（テストの制御バージョンが正しい場合はそれを拒否）で取るリスクのレベルであり、通常はα= 0.05です。



これは、5％のケースで、AとBの違いを見つけることを意味します。これは、実際には偶然によるものです。

有意水準が低いほど、偶然による違いを発見するリスクが低くなります。



統計的検出力1 −β（統計的検出力）は、第2種のエラーであり、実際に真である場合にサンプルのテストバージョンを受け入れる確率です（実際に影響がある場合、効果を検出する機会）。実験を計画するときは、制御バージョンからの逸脱を検出するために、パワーが適度に高い必要があることを覚えておく必要があります。インジケーターの何パーセントを表示すべきかわからない場合は、デフォルト値（80％）のままにします。



その結果、正しいテストを行うには、A / Bテストのバリアントごとに3,292人のユーザーが必要であることがわかりました。







次に、統計的に有意な結果を得るために、取得する必要のあるコンバージョンの数と、それらの間の違いを計算することができます。順次サンプリング



計算機を使用します。 正しい分析を行うには、170の変換が必要であり、A / Bテストのバージョン間の差は26を超える変換でなければなりません。











4.実験の期間を計算する



各バージョンをテストするための計算で得られたサンプルサイズを取得し、それを1日のトラフィックで割ると、テストに必要な日数が得られます。



次のように、少なくとも1週間をテストします。週末のデータは平日と異なる場合があります。

統計的に有意な実験に必要なトラフィックとコンバージョンの量を計算しましたが、テスト用のツールキットを選択する必要があります。

6.実験がクリーンであることを確認します。



テストを開始する前に、2つのテストバージョンに違いがないことを確認してください。

違いは実験の純度に影響するため、バージョンやトラフィックに技術的な違いはありません。



最高の清浄度のために、開始前にA / Aテストが行​​われます。A / Bテストとは異なり、A / Aでは2つの同一のページ（要素）をテストします。実験の目的は、パフォーマンスの違いを確認することではありません。



7.分析システムをセットアップする



分析システムを再確認し、実験に必要なすべてのメトリックが正しく機能することを確認します。



よくある間違いは、分析および測定システムをチェックせずにテストを実行することです。その結果、専門家は時間と予算を費やし、すべてが無駄に行われます。

実験

分析と統計的有意性



実験を行い、データを受け取りました。



ここで、バージョンAとBの間のメトリックで受け取った変更が、仮説または偶然に関連していることを理解する必要があります。



つまり、変化が統計的に有意であるかどうかです。



この例では、次のデータを受け取りました：



サンプルA：3,292セッション、67コンバージョン。

サンプルB：3,200セッション、99コンバージョン。計算機を



使用して統計的有意性を計算します。 P値= 0.000691、つまり、同じテストグループで観測された差異が見られる確率は非常に小さいです。これは、メトリックの成長と私たちの変化を高い確実性で関連付けることが可能であることを意味します。











A / Bテストは私たちの仮説を確認しました、テストバージョンは統計的にコントロールよりも有意に優れています。



統計的に有意な結果が得られたら、急いですべてのユーザーに拡大しないでください。チャンスをとらず、受信したデータを再確認してください。



サンプルと異常の交差がないことを確認してください。



同じユーザーが2つのサンプルに分類されるようなものであってはなりません。そうしないと、受信したデータの信頼性が低下するおそれがあります。
