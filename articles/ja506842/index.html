RPA：ロボット請求書登録プロセス。最も包括的なガイド

こんにちは！RPAのテーマはますます誇大宣伝になっています。ソフトウェアロボットの動作原理と、RPAと他の自動化方法の違いについては、テーマ別ポータルで説明されています。私はRPAプロセスのモデリングへのアプローチに焦点を合わせることにしました。たとえば、バイヤーにアカウントを登録するプロセスを選択しました。このプロセスは普遍的であり、あらゆる業界と規模の企業に関連しているため、私はそれに立ち止まりました。

要約されたアカウント登録は、次のように表すことができます。
  1. 営業マネージャーは、会計士にアカウントへのリクエストを送信します。
  2. 会計士は、会計システムで請求書を作成し、印刷された請求書フォームを作成して、マネージャーに送り返します。
  3. 買い手が電子文書を交換する場合、会計士はEDIシステムで請求書を作成して送ります。


いずれの場合も、このケースはさまざまな方法で整理されます。マネージャーはCRMシステムで作業できます。アカウントの申し込みは、BPMシステムを通じて送信できます。また、会計処理として、1C、SAP、またはOracleのさまざまな構成を使用できます。

その結果、請求書処理プロセスは、このシステムまたはそのシステムが機能する各段階で部分的にのみ自動化されます。しかし、人々はシステム間の接点で働いており、彼らの行動は退屈で単調なものです。彼らは、あるシステムからデータをコピーして別のシステムに貼り付けるだけです。システム間の統合についての疑問が生じます。

すでに標準構成になっている一部のシステムには、統合用の既製の機能が含まれている場合があります。難しさは、統合が実行されるシステムにサポートされていないバージョンがあるか、「あなたのために」非常に多く追加されている場合、機能しないことです。

この場合の解決策は、システム間の独自の統合を実装することです。このソリューションには欠点があります-それは高価で時間のかかるプロジェクトです。 APIまたはクローズドソースコードがないため、この統合ができない場合があります。たとえば、将来維持される予定のない古いシステムの場合など、実用的でない場合があります。

RPAテクノロジーは、フォームに入力してキーを押すことで簡単なデータ転送が必要な日常的なタスクで最もよく示されます。このような問題では、非常に明確なルールのシステムとして表すことができないロジックがあってはなりません。ロジックが単純であればあるほど、そのようなプロセスを実装するのは安価になります。それが会社のビジネスプロセスで繰り返される頻度が高いほど、その実装の利点はより明確になります。

ロボットは人の速度よりわずかに速い速度で処理を実行します。1台のロボットでは、一度に1つのタスクしか実行できません。したがって、自動化の助けを借りて、現在手動​​で実行されているロボット操作に切り替える予定の場合は、RPAが適切なオプションですが、データを高速で処理する必要がある場合は、他の方法を検討することをお勧めします。

私の例では、システム間の単純なデータ転送が必要であり、超人的な速度は必要ない場合、RPAによる統合が最適なソリューションです。

RPAでのプロセスモデリング


私がシミュレートした特定のアカウント登録ケースは次のとおりです。

  1. セールスマネージャは、BPMシステムでアカウントを作成するプロセスを開始します。
  2. アカウントの申請を受け取った会計士は、1Cでアカウントを作成し、アカウントの印刷フォームを作成して、タスクに適用します。
  3. アプリケーションに購入者との合意があったという情報が含まれていて、EDIを通じて作業している場合、会計士はCircuit Diadocでアカウントを作成し、印刷されたフォームを添付して、購入者に送信します。



ロボットが1Cでアカウントを作成してDiadocに送信するので、プロセスは完全に自動化されます。

もちろん、基本的にシミュレーションでは繰り返しのステップが実行されるため、ここでステップバイステップの説明を記述する意味はありません。そして、あなたの場合、実装はおそらく多少異なります。RPAプロセスを実装するための一般的なアプローチについて詳しく説明し、RPAデザイナーで最も頻繁に使用されるステップがどのように実行されるかを示します。

RPAプロセスは、主にユーザーインターフェイスとの対話に関するものです。ロボットはユーザーのアクションを繰り返す必要があります。また、ユーザーはほとんどの場合、対応するビジュアルインターフェイス要素をマウスでクリックして、アプリケーションウィンドウ内を移動します。ウィンドウを閉じる場合、ユーザーは「クロス」をクリックしてアクションを確認します-日付を入力する必要がある場合は「OK」ボタン-ドロップダウンカレンダーで検索することを好みます。このアクションを繰り返すために、目的のアイテムを検索するためのプロセスでポインターが書き込まれます。一般に、これは要素とその名前があるはずのウィンドウのタイトルです。アプリケーションはさまざまな方法で編成できますが、そのようなテキストポインターを選択できるとは限りません。この場合、ロボットは画像によってガイドされ、その過程で写真がポインターとして記録されます。

ピクチャポインタによるアイテムの検索は、テキストポインタによる検索よりも遅くなります。さらに、このような検索は信頼性が低くなります。要素の下の画面または背景の解像度を変更すると、ロボットが要素を見つけられない可能性があります。別のオプションがあります-キーを使用して移動します。彼らの助けを借りて、アプリケーションを開いたり、アイテム間を移動したり、ボタンをクリックしたりできます。検索は必要ありません。これは最速で最も信頼できるオプションであり、RPAに最も優先されるのは彼です。その後-テキストポインタと極端な場合-画像で検索します。

シミュレーションでは、ELMAから無料のRPAシステムを取得しました。

1Cに電話する


デザイナーで新しいプロセスを作成します。



ロボットが最初に行う必要があるのは、1Cアプリケーションを起動することです。キーを押すことによって実行される操作は、最も確実に機能します。したがって、「実行」コマンドを使用して1Cを起動します。

これを行うには、「キーボードショートカット」アクションをプロセスのグラフィカルモデルに移動し、どのキーを押すかを示します-Win +R。次に、「テキスト入力」アクションを移動し、1C実行可能ファイルのアドレスを指定します。たとえば、「C：\ Program Files \ 1cv8 \ 8.3.16.1063 \ bin \ 1cv8s.exe "。最後に、「キーボードショートカット」アクション。Enterキーを押す必要があります。



構成の選択


次に、1C構成ウィンドウが画面に表示されます。最も単純なケースでは、一連のキーストロークを使用して構成を選択することもできます。ただし、必要な構成がリストに表示される順序がわからない場合は、ポインターを使用して検索を提供する必要があります。

ポインターを検索するには、特別なデザイナーツールであるレコーダーを使用します。これにより、ユーザーアクションのシーケンスを記録し、ポインターを自動的に識別できます。レコーダーの実行中、マウスポインターが置かれているインターフェイス要素が認識され、緑色で強調表示されます。このようにして、記録されたアクションが適用される要素に移動できます。レコーダーは、アイテムのドラッグアンドドロップ、スクロール、マウスクリック、キーストロークなど、さまざまなユーザーアクションを記録できます。

レコーダーを起動し、目的の構成の名前を付けます。緑色で強調表示されたら、それをクリックします。次に入力します。記録を停止し、プロセスへのアクションの追加を確認します。



ログインパスワードを入力してください


次のステップは、アプリケーションを入力するためのログインとパスワードを入力することです。これらのパラメーターを呼び出し引数としてプロセスに渡します。それらをプロセスコンテキストに追加します。



次に、これらの変数の値を対応するフィールド1Cに入力する必要があります。これにはレコーダーを使います。録画モードでは、目的のアイテムにカーソルを合わせた後、レコーダーの特別メニューを呼び出して、アイテムに追加のアクションを適用できます。たとえば、コンテキスト変数から値を挿入します。



Userフィールドのクリックを記録し、Pasteアクションを適用して、ログイン変数を指定し、次にTab、PasswordフィールドのPaste関数を指定して、最後にEnterキーを入力します。

「サプライヤーのドキュメント」のフォームへの移行


ここでプロセスの再生を開始すると、デザイナーはコンテキスト変数のログインとパスワードの値を指定するように求め、次に1Cを開始して構成を選択し、アプリケーションにログインします。これでメインのアプリケーションメニューができました。



次のステップでは、「顧客のドキュメント」フォームを開く必要があります。最も可能性が高いのは、このために何をする必要があるかをすでに推測していることです。レコーダーを起動し、[販売]メニューをクリックして、[サプライヤーのドキュメント]をクリックします。

新しいドキュメント


次に、新しいドキュメントを作成する必要があります。1Cでは、Insertキーを押します。このアクションをレコーダーを使用してプロセスに追加するか、「キーボードショートカット」アクションをパレットからドラッグして手動で追加します。

ドキュメントタイプを選択


1Cは、ドキュメントのタイプを選択するように要求します。「購入者への請求書」が必要です。

多くの場合、テーブルとリストの操作は困難です。ポインタは常に定義されているわけではありません。または、ロボットが利用できる単純なアクションを使用して、多くのフィールド、多くの行があるテーブルの分析を必要とするプロセスを構築することは困難です。普遍的な方法はなく、タスクごとにそのアプローチが選択されます。

しかし、この場合、すべてが簡単です。目的のタイプのドキュメントのクリックを記録するか、フォームの上部にある検索バーを使用するか、必要なドキュメントの名前の入力を開始すると、1Cでハイライト表示されます。次にEnterを押すと、アカウントデータを入力する準備が整います。



アカウントヘッダー


アカウントのヘッダーには、アカウント、取引先、契約、倉庫の日付を入力します。これらの値は入力引数としてプロセスに渡す必要があるため、必要な変数をコンテキストに追加します。

1Cでは、適切なフィールドにこれらの値を入力し、ディレクトリのフォームを開いて、選択する必要はありません。レコーダーを起動し、クイックメニューを使用して、コンテキスト変数値の挿入を目的のフィールドに記録し、Tabキーを使用してフィールド間を移動します



アカウントライン


次に、勘定科目行を追加します。明らかに、別の行で作業をシミュレートする方法。ただし、特定のアカウントごとに、好きなだけ行を追加できます。プロセスに任意の数の行を転送する可能性を提供する必要があります。これは、複数のフィールドで構成され、受信した各行を1Cのフォームに追加します。

文字列のコレクションを転送するために、 "Array"タイプのコンテキスト変数を作成します。そのようなコレクションの別の行は、フィールド値をリストするテキスト行になります。

プロセスが行の別々のフィールドで機能する可能性があることから、「スクリプト」アクションを使用します。このアクションにより、プロセスロジックに任意のC＃プログラムコードを含めることができます。私たちの場合、プログラムはループで呼び出され、その後の実行ごとにコレクションの次の行を取得して解析し、フィールド値を対応する変数に割り当てます。ループを終了するには、「はい/いいえ」タイプの特別な変数を使用します。この値は、最後の行を解析するときにスクリプトコードで変更されます。

サイクルを編成するには、アクション「〜まで繰り返す」を選択します。それはあなたが繰り返されるべきアクションを置く必要があるコンテナです。



印刷、ファイルを保存


プロセスはほぼ準備ができています。上記で示したのと同じ手法を使用して、請求書の印刷フォームを作成してディスクに保存し、システムに請求書を保存してアプリケーションを閉じます。

Circuit Diadocに請求書を送信する


例1Cのすべての基本的なモデリング手法は、他のアプリケーションでの作業のシミュレーションにも適しています。

請求書処理の次の段階は、Diadocを介してクライアントに送信することです。 1Cでアカウントを作成して送信することは、2つの異なるプロセスとしてシミュレートすることをお勧めします。これにより、別々のマシンで個別にデバッグして実行できます。

ロボットはWebインターフェースを介してDiadocと対話します。すべてのブラウザがポインタを正しく認識できるわけではありません。 IEとFirefoxで最もよく定義されています。

Diadoc Webページを開くには、「実行」コマンドを使用します。パラメータとして、「iexplore diadoc.kontur.ru」次に、必要な入力パラメーターを作成します。さらに、レコーダーを使用して、システムへのログイン、アカウントファイルの読み込み、パラメーターの入力と送信をモデル化します。可能な場合はキーを押してアクションを実行し、できない場合はポインターを使用します。



要約する


プロセスには、ロボットとロボット管理者という2つの新しい責任領域が現れました。その過程で、ロボットはモデリング時に予期していなかった何かに遭遇する可能性があります。たとえば、1Cで予期しないエラーが発生したり、インターフェイスが更新後にCircuit Diadocで変更され、ロボットが必要なコントロールを見つけられなくなります。この場合、ロボットのサポートを担当するスペシャリストが、問題点を確認してロボットを復元する必要があります。私が説明したプロセスは、そのような状況に備えています。ロボットタスク「エラー」のステータスを受信すると、システムは管理者に適切なタスクを作成します。チェック後、ロボットは再起動します。
