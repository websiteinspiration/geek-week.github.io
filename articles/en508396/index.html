Transfer of molecular dynamics to CUDA. Part I: The Basics

The purpose of this article is to raise the issues of parallelizing the code of a program for numerical modeling using the molecular dynamics (MD) method using CUDA technology. Why is this necessary at all, since MD software packages already exist, including those that work on CUDA? The fact is that I am developing my own concept of a ‚Äúnon-constant force field‚Äù, which is not implemented in existing MD programs.



Remaking someone else‚Äôs code for these needs is a rather thankless task, so I decided to transfer my already written sequential code and at the same time share some thoughts. In addition, this is a response to the comment that often flickers here on CUDA articles like this .



So what is molecular dynamics? On Habr√© already there are several posts on this subject, for example here or here . Briefly, MD is a method that allows you to simulate the motion of many particles (including atoms, ions, molecules) and calculate the collective properties of the system, depending on this movement. How it works? Suppose for a set of N particles some initial coordinates, velocities, masses and (most important!) Laws of interaction between them are given. We change the coordinates according to the speeds. Based on the laws of interaction, we calculate the forces acting between the particles. Since we know strength and mass, we know acceleration. We adjust the speed taking into account acceleration. And again, we move on to changing the coordinates. And so we repeat a thousand times until you get tired We won‚Äôt get enough statistics.







In real applications, the sequence of calculations of the ‚Äúcoordinate - velocity - force‚Äù is somewhat more complicated, based on the expansion in a series of functions of the particle coordinate. The most common is the Velocity Verlet algorithm . The pseudocode of the MD program based on the Verlet integrator is as follows: 



init_md;
int iStep = 0;
while (iStep < nStep)
{
    reset_quantities;
    clear_cell_list;
    verlet_1stage;
    forces_calculation;
    verlet_2stage;
    temp_scale;
    termostat;
    some_output;
    iStep++;
}
final_md;


The main program loop executes a user-specified number of times ( nStep ). An iteration of this loop is called a time step, or simply a timestep. Functions INIT_MD , reset_quantites , some_output and final_md will not be considered, they read input information, zeroed values, outputs the data, allocate and release memory, etc. From the point of view of information output, molecular dynamics is ideal for calculations on a video card: most of the calculations do not need to be unloaded.



Functions verlet_1stage , verlet_2stage , temp_scale and termostatrelate to the integrator of equations of motion and temperature. The forces_calculation function calculates the forces acting between particles, and the clear_cell_list function refers to the technique of accelerating calculations (more on this later).



Consider all these groups of functions.



1 Integration of equations of motion



The essence of the first stage of the Verlet algorithm can be expressed by the following formulas:







X=X+Vdt+Fmdt2











V=V+Fmdt2.







Second:





V=V+Fmdt2,







where X is the particle coordinate, V is its velocity, F is the force acting on it, m is its mass and dt is the time step. All this applies to all particles and, naturally, occurs in 3 dimensions, so:

The input parameters atPerBlock and atPerTherad are equal to the number of particles (atoms) processed by one block and stream, respectively. Since the number of atoms in such calculations is usually constant, these variables can be calculated once and for all somewhere in the init_md function . The md parameter of type * cudaMD is a link to the structure, which contains links to all the variables necessary for the calculation, such as:



  • nAt is the number of particles;
  • xyz - an array of coordinates of type float3;
  • vls - an array of speeds of type float3 (3 components of the vector);
  • frs - array of forces (also 3 components of the vector);
  • the rMasshdT array immediately stores the expression 0.5 * dt * m for each particle;
  • types - an array of types of particles, because atoms can be different;
  • The spec array stores particle properties, mass, charge, etc. To get the properties of the ith particle, you need to invert the form spec [types [i]] .


I combined this all into a structure so as not to pass a bunch of parameters into the function. Perhaps this is not the best solution? Maybe it was worth declaring them global variables? Not in the sense that they are in global memory, but in the fact that they are visible in all functions of the program. We will discuss the keep_in_cell function in the section for accelerating calculations, and the put_periodic function puts a particle in a simulated volume (box) if it flies out of it (like in a pacman, as aptly put itDarein his post about MD). Or is the compiler able to optimize such things itself?



The put_periodic function makes it so that a particle flying out of the borders of the simulated region (box) appears on its other side (periodic boundary conditions): 



__device__ void put_periodic(float3& xyz, float3 vel, float mass, int type, cudaMD *md)
{
   if (xyz.x < 0)
   {
      xyz.x += md->leng.x;

      atomicAdd(&md->specAcBoxNeg[type].x, 1); 
      atomicAdd(&md->negMom.x, mass * (-vel.x)); 
   }
   else
     if (xyz.x >= md->leng.x)
     {
        xyz.x -= md->leng.x;

        atomicAdd(&md->specAcBoxPos[type].x, 1);
        atomicAdd(&md->posMom.x, mass * vel.x);
     }
}


The function works for a box with the geometry of a rectangular box. The code is for measuring x, the rest of the measurements are completely similar. The leng and revLeng fields are the box size in the corresponding dimension and its inverse value. In general, to return a particle to boxing, you can do without conditional operators: 



xyz.x -= floor(xyz.x * md->revLeng.x) * md->leng.x;


I used conditional operators in the serial version, because I had to count the number of particles passing through the box boundary and the total momentum of these particles (pressure can be expressed from it). These counters are also included in the code above, perhaps for CUDA you will need to think about how to do without if .

The structure code is completely similar to the first part of the integrator. We update the speeds for each particle, summarize the kinetic energy.



‚ÄúTemperature‚Äù functions temp_scale and termostat . Molecular systems differ from mechanical ones in that they have the concept of temperature. MD inherited the definition of temperature from molecular kinetic theory , where temperature is a value proportional to the average kinetic energy of particles (there is reason to believe that this is a fallacy). To maintain the desired temperature in such terms, you can simply scale the particle velocities, as the temp_scale function does: 



__global__ void temp_scale(int atPerBlock, int atPerThread, cudaMD *md)
{
   float k;
   int i;
   int id0 = blockIdx.x * atPerBlock + threadIdx.x * atPerThread;
   int N = min(id0 + atPerThread, md->nAt);

   k = sqrt(md->teKin / md->engKin);
   for (i = 0; i < N; i++)
   {
      md->vls[i].x *= k;
      md->vls[i].y *= k;
      md->vls[i].z *= k;
   }

   if (!blockIdx.x)
    if (!threadIdx.x)
     {
        md->engKin = md->teKin;
     }
}


The teKin field contains the target kinetic energy, it is proportional to the temperature set by the user. The function simply multiplies all components of the velocity vectors by k, and then one stream sets the kinetic energy equal to the given value. Here, the variable k is defined in each thread, although it is possible to define it in one (since it is the same) so as not to do unnecessary work, especially since the square root is involved.Such a primitive temperature scaling is usually used in the preliminary stages of the simulation (equilibration period) with a frequency of several steps. More sophisticated temperature maintenance algorithms (thermostats) simulate particle collisions with a certain reservoir of a given temperature, the most popular is the Nose-Hoover thermostat :

The difference from the previous function is only in a more complex definition of the correction factor.



To summarize this group of functions. All of them are perfectly parallelized, not only because they perform exactly the same actions to all particles, but also because they do not require cross-references. The stream, processing arrays with index i, does not access index j, there is no conflict of access to the same data.



2 Calculation of forces and list of cells



In molecular systems, the main source of forces acting on the particle (except for the already mentioned external fields) are other particles. The calculation of these forces is the most complex, time-consuming and ambiguous part of the MD. One of the simplest approximations is that all interactions in the system are the sum of the interactions between EACH pair of particles (pairwise approximation). For these purposes, a pair potential is introduced - a functional dependence of the potential energy of interaction of two particles on the distance between them. The most famous pair potentials: the potential of Lennard-Jones (or "6-12"), Buckingham and Born-Mayer-Huggins. For example, I will give the analytical expressions of the first two:





U=4œµ(œÉ6r6‚àíœÉ12r12)





and





U=Aexp‚Å°(‚àírœÅ)‚àíCr6,







where U is the potential energy of a pair of particles, r is the distance between particles, and all other letters are parameters that depend only on the type of interacting particles and are set at the initialization stage. However, even more than energy, we are interested in the forces of interaction between these particles. The forces are trivial, through the derivative of the pair potential, for which it is also easy to obtain an analytical expression and sew into the program. And, since the force of action is equal to the force of reaction, we immediately get the forces acting on both particles from each other, pseudocode: 



U = 0;
for (i = 0; i < N; i++)
	for (j = i + 1; j < N-1; j++)
	{
		dx = x[i] ‚Äì x[j];
		dy = y[i] ‚Äì y[j];
		dz = z[i] ‚Äì z[j];
		r = sqrt(dx*dx + dy*dy + dz*dz);
		U += pair_pot(r, types[i], types[j]);
		F = force(r, types[i], types[j])
		fx[i] = F * dx;
		fx[j] = -F * dx;
		fy[i] = F * dy;
		fy[j] = -F * dy;
		fz[i] = F * dz;
		fz[j] = -F * dz;

	}


The code is simple: we find the distance between the particles, substitute it in the function for the pair potential and in the function for the force (both depend on the types of interacting particles) and calculate the projection of the forces. The cycle is performed N (N-1) / 2 times, where N is the number of particles. To at least somehow reduce the number of calculations, we can take advantage of the fact that the value of these potentials rapidly decreases with distance. At distances greater than 6-8 angstroms, the potential can not be calculated, it will be almost zero. Adding the condition for checking the distance (or rather its square) we will reduce the number of calculations, but not the number of iterations of the cycle. For large systems, most checks will give false. The following techniques can reduce the number of checks by dropping part of pairs with a known large interatomic distance.



There are two main methods for reducing the number of searches for pairs of particles: a list of neighbors (aka Verlet list ) and a list of cells(cell list). In the first, for each particle we remember its neighbors that are within the radius of the potential. The list of neighbors needs to be updated with some, previously unknown, periodicity, which is the disadvantage of the method. Therefore, let us dwell on the list of cells, the essence of which is to split the simulated volume (box) into (sub) cells. Initially, the cell size was chosen to be greater than or equal to the cutoff radius. Then it is enough to sort out the pairs of particles inside each cell and between adjacent cells. The cell size may be different, the main thing is to determine which pairs of cells you need to check. In fact, this is an attempt to describe the sphere in cubes. Traditionally, to implement a list of cells in sequential code, two arrays are introduced, head and list. The size of the first is equal to the number of cells, the second to the number of particles. The first stores the index of one of the particles in a given cell, the indices of the second correspond to the indices of the particles, and the contents correspond to the index of the next particle in the cell (-1, if there are no more particles in the cell), the illustration is given below.







The pseudo-code for traversing such a list inside a cell with the index cell_index: 



i = head[cell_index];
while (i >= 0)
{
      j = list[i];
      while (j >= 0)
      {
              i-j;
           j = list[j];
      }
i = list[i]; 
}


Now let's try to implement a list of cells on CUDA. In the zeroth approximation, we simply distribute a range of cells to each stream, otherwise, leaving a sequential code for enumerating pairs of particles inside the cell and pairs involving particles of neighboring cells. For each cell, an array of neighboring cells lstHNeig [iC] [jC] and the number of these neighbors, an array nHeadNeig [], must be defined :

In my opinion, such an implementation does not fully reveal the potential of a video card, since a single stream processes one cell, which, as a rule, is significantly less than the number of particles. Secondly, inside the function of processing a pair of particles, it is possible to work only through atomic operations. Let's try to parallelize the task more strongly, let each thread process a pair of neighboring cells (we believe that a list of such pairs has already been created):

The cellPairs [] array contains the number of pairs of neighboring cells (.x is the first cell of the pair, .y is the second). The code is simpler and more elegant, but here the processing of pairs of particles inside one cell disappears, for which a separate function must be written. There is another problem, the organization of the list of cells and its crawl are inefficient for parallel computing, because in order to get the index of the next particle in the cell, you need to run through all the previous ones. Let's try to create more efficient code and data structure so that a couple of cells can be processed by several threads at once. Now let the list of cells be stored in a two-dimensional array cell [] [] , where the first index determines the number of the cell, and the second is the atomic number inside the cell. I also store the number of particles in a cell in the zero element of the array:cell [i] [0] , the value of cell [i] [j + 1] is equal to the general index of the particle in global memory. True, such an implementation requires knowing the maximum number of particles in a cell and will take up more memory. We will also divide our task into 2 subtasks. Variables of the form float3PerCell are the product of the maximum number of particles in a cell by the size of type float3. It makes sense to calculate them once during system initialization. The pair_in_cell and pair_between_cell functions calculate interactions between particles in the same and different cells, respectively. As parameters, they get the coordinates of the particles, their types and pointers where to write forces. Parameters for pair_between_cellalso include the magnitude of the shifts, to take into account the periodic boundary conditions. Codes for these functions will be given below. To sort out the pairs of particles inside the cell, I wanted to go the standard way - divide the number of pairs into equal intervals and give each stream an interval. However, to get the particle indices ij by the pair number, I came up with the quadratic equation and decided to do it differently: the first thread processes a pair of particles with indices 0-1, the second - 0-2, etc. Having processed the ij pair, the flow goes to the i- (j + N) pair, where N is the number of flows in the block and if it goes beyond the range limit it increases the index of the first particle in the pair, see the diagram:

    int* sBtypes = (int*)&sBids[md->maxAtPerCell];
    float3* sBxyz = (float3*)&sBtypes[md->maxAtPerCell];
    float3* sBfrs = (float3*)&sBxyz[md->maxAtPerCell];

    float3* sXyz;
    float3* sFrs;  
    int* sTypes;
    int* sIds;

    iPair = blockIdx.x + md->nPair1;
    if (threadIdx.x == 0)
    {
        engVdW = 0.0;
        engCoul = 0.0;
    }
    __syncthreads();

    iCA = md->cellPairs[iPair].x; 
    iCB = md->cellPairs[iPair].y;
    nA = md->cells[iCA][0];
    nB = md->cells[iCB][0];
    
    //    ,        
    if (!nA)
       return;
    if (!nB)
        return;

    //      shared_memory
    if (threadIdx.x < blockDim.x / 2)
    {
        iC = iCA;
        sXyz = sAxyz;
        sFrs = sAfrs;
        sTypes = sAtypes;
        sIds = sAids;
        deltId = 0;
    }
    else
    {
        iC = iCB;
        sXyz = sBxyz;
        sFrs = sBfrs;
        sTypes = sBtypes;
        sIds = sBids;
        deltId = blockDim.x / 2;
    }
    nOwn = md->cells[iC][0]; // nA or nB

    step = ceilf(2 * (double)nOwn / (double)blockDim.x);
    id0 = (threadIdx.x - deltId) * step;
    N = min(id0 + step, nOwn);
    for (i = id0; i < N; i++)
    {
        id = md->cells[iC][i + 1];
        sIds[i] = id;     
        sXyz[i] = md->xyz[id];    
        sFrs[i] = make_float3(0.0, 0.0, 0.0);    
        sTypes[i] = md->types[id];
    }
    __syncthreads();

    //       
    step1 = ceilf((double)nA / (double)blockDim.x);
    id1 = threadIdx.x * step1;
    N1 = min(id1 + step1, nA);
    for (i = id1; i < N1; i++)
        for (j = 0; j < nB; j++)
        {
            pair_between_cell1(sAxyz[i], &sAfrs[i], sAtypes[i], sBxyz[j], &sBfrs[j], sBtypes[j], md->cellShifts[iPair], md, eVdW, eCoul);

        }
    __syncthreads();

   //     global memory
    for (i = id0; i < N; i++)
    {
        atomicAdd(&(md->frs[sIds[i]].x), sFrs[i].x);
        atomicAdd(&(md->frs[sIds[i]].y), sFrs[i].y);
        atomicAdd(&(md->frs[sIds[i]].z), sFrs[i].z);
    }

    atomicAdd(&engCoul, eCoul);
    atomicAdd(&engVdW, eVdW);
    __syncthreads();
 
    if (threadIdx.x == 0)
    {
        atomicAdd(&md->engCoul1, engCoul);
        atomicAdd(&md->engVdW, engVdW);
    }
}


The code is almost similar to the previous function, only the remaining pairs are now processed (the index of the pair is shifted by md-> nPair1 ), the interaction is calculated only between the pair of cells (already counted inside the cells), and when copying to global memory, we use atomicAdd to summarize the results and not overwrite them. Function for calculating interactions between two different cells: 



__device__ void pair_between_cell1(float3 xyzA, float3* fxyzA, int typeA, float3 xyzB, float3* fxyzB, int typeB, float3 shift, cudaMD* md, float& engVdw, float& engCoul)
{
    float r, r2, f;
    float dx, dy, dz;
    cudaVdW* vdw;

    dx = xyzA.x - xyzB.x - shift.x;
    dy = xyzA.y - xyzB.y - shift.y;
    dz = xyzA.z - xyzB.z - shift.z;
    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)
    {
        r = 0.0;
        f = 0.0;

        f += real_ewald(r2, r, md->chProd[typeA][typeB], md->alpha, engCoul); 

        vdw = md->vdws[typeA][typeB];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw);


        fxyzA->x += f * dx;
        atomicAdd(&(fxyzB->x), -f * dx);
        fxyzA->y += f * dy;
        atomicAdd(&(fxyzB->y), -f * dy);
        fxyzA->z += f * dz;
        atomicAdd(&(fxyzB->z), -f * dz);
    }
}

I also give functions that calculate the forces and energies of pair potentials for the potential of Lennard-Jones and Buckingham: 



__device__ float cu_fe_lj(float r2, cudaVdW* vdw, float& eng)
// U = 4e[(s/r)^12 - (s/r)^6]
{
    float r2i = 1.0 / r2;
    float sr2 = vdw->p1 * r2i;
    float sr6 = sr2 * sr2 * sr2;

    eng += vdw->p0 * sr6 * (sr6 - 1.0);
    return vdw->p2 * r2i * sr6 * (2.0 * sr6 - 1.0);
}

__device__ float cu_fer_buck(float r2, float& r, cudaVdW* vdw, float& eng)
// U = A exp(-r/ro) - C/r^6
{
    float r2i = 1.0 / r2;
    float r4i = r2i * r2i;
    if (r == 0.0)    //  r , 
        r = sqrt(r2);

    eng += vdw->p0 * exp(-r / vdw->p1) - vdw->p2 * r4i * r2i;
    return vdw->p0 * exp(-r / vdw->p1) / r / vdw->p1 - 6.0 * vdw->p2 * r4i * r4i;
}


The functions return the force (which must be multiplied by the projection of the distance vector) and add energy to the variable eng . A pointer to one of these functions is stored in the cudaVdW structure. Generally speaking, to obtain trajectories of motion, energy can not be calculated, it is needed for different statistics. If we remove the expressions for energy from these functions, the calculations are likely to accelerate. And you can do this: once a few steps to call the full function, and the rest of the time - only a reduced analog. Part of the Coulomb interaction is computed by the real_ewald function : 



__device__ float real_ewald(float r2, float& r, float chprd, float alpha, float& eng)
{
    float ar; 
    float erfcar;
    float kqq = chprd * d_Fcoul_scale; // q[i]*q[j]*1/4pie0;

    r = sqrt(r2);
    ar = alpha * r;
    erfcar = erfc(ar);

    eng += kqq * erfcar / r;
    return kqq / r / r2 * (erfcar + 2 * ar / d_sqrtpi * exp(-ar * ar));
}


This differs from the familiar Coulomb law, but more on that in a separate article.

__global__ void all_pair(cudaMD* md)
{
    int i, j;
    int step = blockDim.x * gridDim.x;
    int ex = 0;  // exit flag
    int nAt = md->nAt;

    i = 0;
    j = blockIdx.x * blockDim.x + threadIdx.x + 1;
    while (1)
    {
        while (j >= nAt)
        {
            i++;
            if (i >= nAt - 1)
            {
                ex = 1;
                break;
            }
            j = i + 1 + j - nAt;
        }
        if (ex) break;
        global_pair(i, j, iStep, md);
        j = j + step;
    }
} 

__device__ void global_pair(int i, int j, cudaMD* md)
{
    float r, r2, f;
    float engCoul = 0;
    float engVdw = 0;
    cudaVdW* vdw;
    int ti, tj;

    float dx = md->xyz[i].x - md->xyz[j].x;
    float dy = md->xyz[i].y - md->xyz[j].y;
    float dz = md->xyz[i].z - md->xyz[j].z;

    //   
    if (dx > md->halfLeng.x)
        dx -= md->leng.x;
    else
        if (dx < -md->halfLeng.x)
            dx += md->leng.x;

    if (dy > md->halfLeng.y)
        dy -= md->leng.y;
    else
        if (dy < -md->halfLeng.y)
            dy += md->leng.y;

    if (dz > md->halfLeng.z)
        dz -= md->leng.z;
    else
        if (dz < -md->halfLeng.z)
            dz += md->leng.z;

    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)    
    {
        r = 0.0;
        f = 0.0;

        ti = md->types[i];
        tj = md->types[j];

        f += real_ewald(r2, r, md->chProd[ti][tj], md->alpha, engCoul);

        vdw = md->vdws[ti][tj];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw); 

        dx *= f;
        dy *= f;
        dz *= f;

        atomicAdd(&(md->frs[i].x), dx);
        atomicAdd(&(md->frs[j].x), -dx);
        atomicAdd(&(md->frs[i].y), dy);
        atomicAdd(&(md->frs[j].y), -dy);
        atomicAdd(&(md->frs[i].z), dz);
        atomicAdd(&(md->frs[j].z), -dz);

        atomicAdd(&(md->engCoul1), engCoul);
        atomicAdd(&(md->engVdW), engVdw);

    }
}
