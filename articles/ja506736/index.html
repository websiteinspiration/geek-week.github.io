クラスにはいくつのメソッドが必要ですか？

おそらくこれは、クラスを使用する開発者が直面する最も重要な問題の1つです。私たちはSOLIDの原則を持っています。私たちは、美しくて膨大な数の矛盾についての私たち自身のビジョンを持っています。 「クラスにはメソッドがいくつあるべきか」という完全な答えを出すつもりはありません。しかし、クラス内のメソッドを少なくする必要がある理由について、私の考えを共有します。最大1.

までの範囲で、私の推論はサービス（ステートレスオブジェクト）に関係しています。私の意見では、実際のプロジェクトの他のタイプのオブジェクトよりも優先されます。クラスは、実装が類似しているという理由だけで、それ自体がメソッドの寄せ集めを格納するべきではありません。

OOPの基本を理解した後、初心者はクラスとオブジェクトを猫と犬に関連付けます。つまり、第一印象では、オブジェクトは現実世界の投影または問題を解決するために必要な一連のメソッドを含むいくつかのモデルとして認識されます。そして、これはOOPについて、本来あるべきよりも正確な意見です。

しかし、いくつかの商業プロジェクトの後で、見解は崩壊します。企業はサービスと貧血モデルによって支配されています。それが良いか悪いかはわかりませんが、現実です。したがって、実際には、実世界との類似性がなく、明確な一般化が適用されないシッククラスが取得されることがよくあります。多くの場合、これらはサービスと呼ばれ、多くの場合、レイヤーを形成します。そして、オブジェクトの理解はレベルに低下します-同様の目的を持つメソッドのグループ。または、同様のデータを処理するメソッド。または他の同様のもの。

私はサービスを書き始めるたびに、その名前について長い間考えます...無意味なAnotherEntityServiceを何度も何度も繰り返します。しかし、名詞があってはならない場合はどうでしょうか。サービスの名前は、内部で収集されたリクエストの寄せ集めをどのように反映していますか？一般的に誰が一度にこれらの方法を使用しますか？

分割には非常に多くの利点があるので、なぜ実際に肥大したクラスが優勢になるのか、と私は思いました。クラスを関数のレベルに分解し、全体像を理解するのがより難しくなるからですか？ （しかし、FP-Schnicksはこれについて複雑ですか？）それとも、カプセル化が私たちの脂肪質のサービス-サービスからこぼれるからですか？大きなクラスの秘密を守るダークサイドがあるように思えます。

ロマンスを破棄した場合、プログラム言語の構成要素に関してクラスは何を提供しますか？クラスには明るい面があります。オブジェクトのインターフェースを自動的に形成するパブリックメソッドです。インターフェイスは、プログラムを簡略化する抽象化を形成します。これはロマンスです。

しかし、実際に見ると、クラスはコンストラクタを介して必要な依存関係のみを収集し、他では使用されないプライベートメンバーを格納し、クライアントに一連の固定（パブリック）関数を提供します。

そして、これはダークサイドです-プライベートフィールド、プライベートメソッド、依存関係。これは、サービス内のメソッドを組み合わせることによって開発者が最もよくガイドされるものです。場合によっては、クラスに必要なすべてのメンバーが含まれている場合、これが新しいメソッドに最適な場所であると思われることがあります。そして、サービスは依存関係とそれらの依存関係を必要とするメソッドのカストディアンに変わります。

共通の依存関係が何かを結合する必要があるのはいつですか？内部実装の類似性が抽象化を作成する理由ではありません。言い換えると、その実装は「他のメソッドと非常に似ている」ため、サービスに新しいメソッドを追加すると、カプセル化、SRP、OCP、ISPに違反します。しかし、私は常にこれに遭遇します。

したがって、実際には、サービスアーキテクチャは高レベルの抽象化を形成するのではなく、単に手続き主義のアナロジーを表します。さらに、最悪の例えは、プロシージャには疑似総称サービス名の形式の余分なエンティティがなく、未使用の依存関係を取得しないためです。（どこか暗いFPShniki）。

@Injectable({ providedIn: 'root' })
export class HeroService {
 
 private heroesUrl = 'api/heroes';  // URL to web api
 
 httpOptions = {
   headers: new HttpHeaders({ 'Content-Type': 'application/json' })
 };
 
 constructor(
   private http: HttpClient,
   private messageService: MessageService) { }
 
 /** GET heroes from the server */
 getHeroes(): Observable<Hero[]> {
   return this.http.get<Hero[]>(this.heroesUrl)
     .pipe(
       tap(_ => this.log('fetched heroes')),
       catchError(this.handleError<Hero[]>('getHeroes', []))
     );
 }
 
 /** GET hero by id. Will 404 if id not found */
 getHero(id: number): Observable<Hero> {
   const url = `${this.heroesUrl}/${id}`;
   return this.http.get<Hero>(url).pipe(
     tap(_ => this.log(`fetched hero id=${id}`)),
     catchError(this.handleError<Hero>(`getHero id=${id}`))
   );
 }
 
 /* GET heroes whose name contains search term */
 searchHeroes(term: string): Observable<Hero[]> {
   if (!term.trim()) {
     // if not search term, return empty hero array.
     return of([]);
   }
   return this.http.get<Hero[]>(`${this.heroesUrl}/?name=${term}`).pipe(
     tap(x => x.length ?
        this.log(`found heroes matching "${term}"`) :
        this.log(`no heroes matching "${term}"`)),
     catchError(this.handleError<Hero[]>('searchHeroes', []))
   );
 }
 
 //////// Save methods //////////
 
 /** POST: add a new hero to the server */
 addHero(hero: Hero): Observable<Hero> {
   return this.http.post<Hero>(this.heroesUrl, hero, this.httpOptions).pipe(
     tap((newHero: Hero) => this.log(`added hero w/ id=${newHero.id}`)),
     catchError(this.handleError<Hero>('addHero'))
   );
 }
 
 /** DELETE: delete the hero from the server */
 deleteHero(hero: Hero | number): Observable<Hero> {
   const id = typeof hero === 'number' ? hero : hero.id;
   const url = `${this.heroesUrl}/${id}`;
 
   return this.http.delete<Hero>(url, this.httpOptions).pipe(
     tap(_ => this.log(`deleted hero id=${id}`)),
     catchError(this.handleError<Hero>('deleteHero'))
   );
 }
 
 /** PUT: update the hero on the server */
 updateHero(hero: Hero): Observable<any> {
   return this.http.put(this.heroesUrl, hero, this.httpOptions).pipe(
     tap(_ => this.log(`updated hero id=${hero.id}`)),
     catchError(this.handleError<any>('updateHero'))
   );
 }
 
 /**
  * Handle Http operation that failed.
  * Let the app continue.
  * @param operation - name of the operation that failed
  * @param result - optional value to return as the observable result
  */
 private handleError<T>(operation = 'operation', result?: T) {
   return (error: any): Observable<T> => {
 
     // TODO: send the error to remote logging infrastructure
     console.error(error); // log to console instead
 
     // TODO: better job of transforming error for user consumption
     this.log(`${operation} failed: ${error.message}`);
 
     // Let the app keep running by returning an empty result.
     return of(result as T);
   };
 }
 
 /** Log a HeroService message with the MessageService */
 private log(message: string) {
   this.messageService.add(`HeroService: ${message}`);
 }
}

Heroの新しいAPIメソッドが表示されたらどうしますか？すべてを1つにまとめてクラスを分割するのをやめる必要があるのはいつですか。きっと皆がこれらの質問をしたことでしょう。しかし、プロジェクトにこのサービスが存在しているだけでは、開発者を混乱させるだけであり、拡張ではなく、変更を通じてコードを書くように促します。

HeroServiceに組み込まれているメソッドが「似ている」理由と、それがどのように機能しないかを見てみましょう。

  • すべてのメソッドは共通のURLを使用します-BASE_HERO_API_URLを定数に入れることで解決されます。
  • 一部のメソッドは同じhttp-headersを使用します-それはインターセプターに置くことによって決定されます。または、get、post、put、deleteメソッドを必要なオプションでラップする特別なHttpClientを選択することもできます。
  • handleError-エラーを処理する責任者として完全に機能する別のクラスに配置することで解決します。

すべての統一要素は実装のオプションの詳細であり、ランダムな状況の意志がない場合は異なる可能性があることが判明しました。合併の外部理由はありませんでした。私たちはウェブを書いていて、1つのボタンが最初にaddHeroを呼び出してから、他のボタンが他のHeroService機能を使用するのは奇妙です。

@Injectable({ providedIn: 'root' })
export class HeroHttpClient {
 private httpOptions = { headers: new HttpHeaders({ 'Content-Type': 'application/json' }) };
 
 constructor(@Inject(BASE_HERO_API_URL) private baseUrl: string, private http: HttpClient) {}
 
 get<T>(endpoint = ''): Observable<T> {
   return this.http.get<T>(this.getUrl(endpoint));
 }
 
 private getUrl(endpoint: string): string {
   return Location.joinWithSlash(this.baseUrl, endpoint);
 }
 
 post<T>(endpoint = '', data: unknown): Observable<T> {
   return this.http.post<T>(this.getUrl(endpoint), data, this.httpOptions);
 }
 
 put<T>(endpoint = '', data: unknown): Observable<T> {
   return this.http.put<T>(this.getUrl(endpoint), data, this.httpOptions);
 }
 
 delete<T>(endpoint = ''): Observable<T> {
   return this.http.delete<T>(this.getUrl(endpoint), this.httpOptions);
 }
}


「クラスには最初のメソッド以上のものを含めることはできません！」のようなステートメントを作成しようとはしていません（また、したくありません）。もちろん、オブジェクトに状態がある場合、他のルールが有効になります。また、クラスで一貫して使用する必要があるメソッドを含めることに反対していません：isExecutable、validateParams、execute。

私が言っているのは、モンスターのサービスについてです。ほとんどの場合、サービスはまともな抽象化を構成しないため。そして、重点が彼らの減少に移ったならば、それはより良いでしょう。私自身のために、私は次の原則を策定しました：

クラスはメソッドのセットを含むべきではありません。それらのメソッドの実装が似ているからです。

このルールに従って、プライベートメソッドに隠されている失われたシステム要素を検出できます。このアプローチは、文字通りOCPおよびISPに準拠するのに役立ちます。あなたの機能が実際にプログラムを拡張し、10個のレイヤーの10個のクラスにメソッドを追加しない場合。
