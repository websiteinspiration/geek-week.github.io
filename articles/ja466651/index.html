例としてJava / Spring / H2を使用してLiquiBaseスクリプトをローリングするためのXMLとSQLの選択

次のプロジェクトに取り組む過程で、チームはLiquibaseでのXMLまたはSQL形式の使用に異議を唱えました。当然、すでに多くの記事がLiquibaseについて書かれていますが、いつものように、私の見解を追加したいと思います。この記事では、データベースを使用した単純なアプリケーションの作成に関する小さなチュートリアルを紹介し、これらのタイプのメタ情報の違いについて検討します。

Liquibaseは、データベーススキーマの変更を追跡、管理、および適用するためのデータベースに依存しないライブラリです。データベースに変更を加えるために、バージョンを制御し、すべての変更を管理するメインファイル（* changeLog *）に接続されている移行ファイル（*チェンジセット*）が作成されます。データベースの構造と変更の説明として、XML、YAML、JSONおよびSQL形式。

データベース移行の基本的な概念は次のとおりです。



Liquibaseの詳細については、こちらまたはこちらをご覧ください。全体像がはっきりしているといいので、プロジェクトの作成に移りましょう。

テストプロジェクトでは、


  • Java 8
  • 春ブーツ
  • メイベン
  • H2
  • よくリキベース自体

プロジェクトの作成と依存関係


ここでは、Spring-bootの使用は条件付きではなく、ローリングスクリプトに対してMavenプラグインのみを実行できます。それでは始めましょう。

1. IDEでmavenプロジェクトを作成し、pomファイルに次の依存関係を追加します。

<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <dependency>
        <groupId>org.liquibase</groupId>
        <artifactId>liquibase-core</artifactId>
        <version>3.6.3</version>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-test</artifactId>
        <scope>test</scope>
    </dependency>
   <dependency>
        <groupId>com.h2database</groupId>
        <artifactId>h2</artifactId>
        <scope>runtime</scope>
        </dependency>
</dependencies>

2.リソースフォルダーで、application.ymlファイルを作成し、次の行を追加します。

spring:
  liquibase:
    change-log: classpath:/db/changelog/db.changelog-master.yaml
 datasource:
    url: jdbc:h2:mem:test;
    platform: h2
    username: sa
    password:
    driverClassName: org.h2.Driver
  h2:
    console:
      enabled: true

Liquibase行：change-log：クラスパス：/db/changelog/db.changelog-master.yaml-liquibaseスクリプトファイルの場所を示します。

3.リソースフォルダーに、パス/db/changelog/db.changelog-masterに沿って、次のファイルを作成します。

  • xmlSchema.xml-xml形式のスクリプトを変更
  • sqlSchema.sql-SQL形式の変更のスクリプト
  • data.xml-テーブルにデータを追加します
  • db.changelog-master.yml-変更セットのリスト

4.ファイルへのデータの追加：テストでは、2つの無関係なテーブルと最小限のデータセットを作成する必要があります。
既知のsql構文をsqlSchema.sqlファイルに追加します。

--liquibase formatted sql
--changeset TestUsers_sql:1

CREATE TABLE test_sql_table
(
    name VARCHAR NOT NULL,
    description VARCHAR
);

--changeset TestUsers_sql:2

CREATE TABLE test_sql_table_2
(
    name VARCHAR NOT NULL,
    description VARCHAR
);

チェンジセットとしてのSQLの使用は、簡単なスクリプトで制御されます。ファイルでは、誰もが通常のSQLを理解しています。

コメントは、変更セットを分離するために使用されます：--changeset
TestUsers_sql：1変更番号と姓
（パラメーターはここにあります。）

xmlSchema.sqlファイルで、liquibaseが提供するDSLを追加します。


<?xml version="1.0" encoding="UTF-8"?>

<databaseChangeLog
        xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
        xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-3.6.xsd">

    <changeSet id="Create table test_xml_table" author="TestUsers_xml">

        <createTable tableName="test_xml_table">
            <column name="name" type="character varying">
                <constraints primaryKey="true" nullable="false"/>
            </column>
            <column name="description" type="character varying"/>
        </createTable>

    </changeSet>

    <changeSet id="Create table test_xml_table_2" author="TestUsers_xml">

        <createTable tableName="test_xml_table_2">
            <column name="name" type="character varying">
                <constraints primaryKey="true" nullable="false"/>
            </column>
            <column name="description" type="character varying"/>
        </createTable>

    </changeSet>
</databaseChangeLog>

テーブルの作成を記述するこの形式は、さまざまなデータベースで共通です。Javaのスローガンと同じように、「一度記述すれば、どこでも機能します。」Liquibaseはxml記述を使用し、選択したデータベースに応じて、それを特定のSQLコードにコンパイルします。これは一般的なパラメータにとって非常に便利です。

各操作は個別のchangeSetで実行され、IDと作成者の名前を示します。xmlで使用される言語は非常に理解しやすく、説明する必要さえありません。

5.プレートにデータをアップロードします。これは必要ありませんが、プレートが作成されているため、何かをプレートに入れる必要があります。data.xmlファイルに次のデータを入力します。

<?xml version="1.0" encoding="UTF-8"?>
<databaseChangeLog
        xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
        xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-3.6.xsd">

    <changeSet id="insert data to test_xml_table" author="TestUsers">
        <insert tableName="test_xml_table">
            <column name="name" value="model"/>
            <column name="description" value="- "/>
        </insert>
    </changeSet>

    <changeSet id="insert data to test_xml_table_2" author="TestUsers">
        <insert tableName="test_xml_table_2">
            <column name="name" value="model"/>
            <column name="description" value="- "/>
        </insert>
    </changeSet>

    <changeSet id="insert data to test_sql_table" author="TestUsers">
        <insert tableName="test_sql_table">
            <column name="name" value="model"/>
            <column name="description" value="- "/>
        </insert>
    </changeSet>

    <changeSet id="insert data to test_sql_table_2" author="TestUsers">
        <insert tableName="test_sql_table_2">
            <column name="name" value="model"/>
            <column name="description" value="- "/>
        </insert>
    </changeSet>
</databaseChangeLog>

ローリングテーブルのファイルが作成され、テーブルのデータが作成されます。これらすべてを共通のローリング注文に組み合わせて、アプリケーションを起動する時が来ました。

sqlファイルとxmlファイルをdb.changelog-master.ymlファイルに追加します。

databaseChangeLog:
  - include:
      # schema
      file: db/changelog/xmlSchema.xml
  - include:
      file: db/changelog/sqlSchema.sql
      # data
  - include:
      file: db/changelog/data.xml

そして、これですべてが作成されました。アプリケーションを実行するだけです。コマンドラインまたはプラグインを使用してstart を実行できますが、ここではmainメソッドだけを作成して、SpringApplicationを実行します。

メタデータを表示


2つのスクリプトを実行してテーブルを作成および設定したので、databaseChangeLogテーブルを見て、何がロールアップされたかを確認できます。



XMLローリング結果：

  1. xmlファイルのidフィールドに、開発者がchangeSetを指すヘッダーが表示されます。個々のchangeSetは、タイトルと説明を持つデータベース内の個別の行です。
  2. 各変更の作成者が示されています。

SQLロール結果：

  1. SQLファイルのidフィールドには、changeSetに関する詳細情報はありません。
  2. 各変更の作成者は示されていません。

xmlを使用するもう1つの重要な結論は、ロールバックです。テーブルの作成、テーブルの変更、列の追加などのコマンドは、xmlを使用すると自動的にロールバックされます。SQLファイルの場合、各ロールバックは手動で書き込む必要があります。

結論


誰もが自分で何を使うかを選びます。しかし、私たちの選択はxml側にありました。詳細なメタ情報と他のデータベースへの簡単な移行は、誰もが好きなSQL形式のスケールを上回っていました。
