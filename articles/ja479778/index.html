＃メモ。ConcurrentHashMapでのアトミック操作の注意



太古の昔から、JavaにはすばらしいMapインターフェースとその実装、特にHashMapがありました。また、Java 5以降では、ConcurrentHashMapもあります。これらの2つの実装、それらの進化、およびこの進化が不注意な開発者につながる可能性があるものを検討してください。

警告：この記事では、GNU General Public Licenseバージョン2に基づいて配布されたOpenJDK 8ソースコードからの引用を使用しています。

Java 8より前の時代


最初にJava 7、次にJava 8（今とは異なり、6か月ごとに新しいバージョン）の長い長い待ち時間を見つけた人は、Mapでの操作が最も人気だったのを覚えています。それ：

  • V Map.put（Kキー、V値）
  • V Map.get（オブジェクトキー）

コレクションに値を入れる必要がある場合は、最初の方法を使用し、既存の値を取得するには2番目の方法を使用します。

しかし、遅延初期化が必要な場合はどうなりますか？次に、この種のコードが現れました：

String getOrPut(String key) {
    String result = map.get(key);  //(1)
    if (result == null) {          //(2)
        result = createValue(key); //(3)
        map.put(key, result);      //(4)
    }
    return result;
}

  1. キーで値を取得する
  2. 目的の値が見つかったかどうかを確認します
  3. 値が見つからない場合は、作成します
  4. キーによってコレクションに値を追加する

ちょっと面倒ですよね？さらに、単純なHashMapが使用されている場合、これは読み取りに不便なコードにすぎません。スレッド化されていません。しかし、ConcurrentHashMapの場合、追加機能がポップアップします。複数のスレッドが条件（1）をチェックして、そのうちの1つがコレクション（3）に値を書き込む前に、createValue（2）メソッドを数回呼び出すことができます。このような動作は、望ましくない結果をもたらすことがよくあります。

Java 8以前は、エレガントなオプションはありませんでした。複数の値の作成を回避する必要がある場合は、追加のロックを使用する必要がありました。
Java 8は物事をより簡単にしました。そうらしい…

Java 8が登場...


Java 8で私たちにもたらされた最も期待される機能は何ですか？そうです、ラムダ。そして、ラマだけでなく、標準ライブラリのすべてのさまざまなAPIにおけるそれらのサポート。マップデータ構造は無視されていません。特に、次のようなメソッドが登場しました。

  • computeIfAbsent
  • computeIfPresent
  • forEach
  • 等

これらのメソッドにより、前に示したコードを書き換えることがはるかに簡単になります。

String getOrPut(String key) {
    return map.computeIfAbsent(key, this::createValue);
}

コードを簡略化する機会を誰もあきらめないことは明らかです。さらに、ConcurrentHashMapの場合、computeIfAbsentメソッドもアトミックに実行されます。それら。createValueは、必要な値が存在しない場合にのみ1回だけ呼び出されます。

IDEも通過しませんでした。したがって、IntelliJ IDEAは、古いバージョンを新しいバージョンに自動置換します。




コードの簡略化とIDEヒントの両方が開発者にこの新しいAPIの使用を奨励していることは明らかです。その結果、同じcomputeIfAbsentがコードの非常に多くの場所に現れ始めました。

でも待って、どうやって？実際、ブロッキングに関するメソッドのドキュメントでさえ、それは更新の付録でのみ述べられています：
「指定されたキーがまだ値に関連付けられていない場合、指定されたマッピング関数を使用してその値を計算し、nullでない限り、このマップに入力します。メソッド呼び出し全体がアトミックに実行されるため、関数はキーごとに最大1回適用されます。他のスレッドによってこのマップに対して行われた更新操作の一部は、計算の進行中にブロックされる可能性があるため、計算は短く単純である必要があり、このマップの他のマッピングの更新を試みてはなりません。

実際、すべてがそうとは限りません。このメソッドのソースコードを見ると、2つの非常に厚い同期ブロックが含まれていることがわかります。

ConcurrentHashMap.computeIfAbsentの実装
public V computeIfAbsent(K key, Function<? super K, ? extends V> mappingFunction) {
    if (key == null || mappingFunction == null)
        throw new NullPointerException();
    int h = spread(key.hashCode());
    V val = null;
    int binCount = 0;
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & h)) == null) {
            Node<K,V> r = new ReservationNode<K,V>();
            synchronized (r) {
                if (casTabAt(tab, i, null, r)) {
                    binCount = 1;
                    Node<K,V> node = null;
                    try {
                        if ((val = mappingFunction.apply(key)) != null)
                            node = new Node<K,V>(h, key, val, null);
                    } finally {
                        setTabAt(tab, i, node);
                    }
                }
            }
            if (binCount != 0)
                break;
        }
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            boolean added = false;
            synchronized (f) {
                if (tabAt(tab, i) == f) {
                    if (fh >= 0) {
                        binCount = 1;
                        for (Node<K,V> e = f;; ++binCount) {
                            K ek; V ev;
                            if (e.hash == h &&
                                ((ek = e.key) == key ||
                                    (ek != null && key.equals(ek)))) {
                                val = e.val;
                                break;
                            }
                            Node<K,V> pred = e;
                            if ((e = e.next) == null) {
                                if ((val = mappingFunction.apply(key)) != null) {
                                    added = true;
                                    pred.next = new Node<K,V>(h, key, val, null);
                                }
                                break;
                            }
                        }
                    }
                    else if (f instanceof TreeBin) {
                        binCount = 2;
                        TreeBin<K,V> t = (TreeBin<K,V>)f;
                        TreeNode<K,V> r, p;
                        if ((r = t.root) != null &&
                            (p = r.findTreeNode(h, key, null)) != null)
                            val = p.val;
                        else if ((val = mappingFunction.apply(key)) != null) {
                            added = true;
                            t.putTreeVal(h, key, val);
                        }
                    }
                }
            }
            if (binCount != 0) {
                if (binCount >= TREEIFY_THRESHOLD)
                    treeifyBin(tab, i);
                if (!added)
                    return val;
                break;
            }
        }
    }
    if (val != null)
        addCount(1L, binCount);
    return val;
}


上記の例から、結果は6点でのみ形成でき、これらの場所のほとんどすべてが同期ブロック内にあることがわかります。かなり意外と。さらに、単純なgetには同期がまったく含まれていません。

ConcurrentHashMap.getの実装
public V get(Object key) {
    Node<K,V>[] tab; Node<K,V> e, p; int n, eh; K ek;
    int h = spread(key.hashCode());
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (e = tabAt(tab, (n - 1) & h)) != null) {
        if ((eh = e.hash) == h) {
            if ((ek = e.key) == key || (ek != null && key.equals(ek)))
                return e.val;
        }
        else if (eh < 0)
            return (p = e.find(h, key)) != null ? p.val : null;
        while ((e = e.next) != null) {
            if (e.hash == h &&
                ((ek = e.key) == key || (ek != null && key.equals(ek))))
                return e.val;
        }
    }
    return null;
}


じゃあ何をすればいいの？実際、2つのオプションしかありません：元のコードに戻すか、それを使用しますが、わずかに変更されたバージョンでは：

String getOrPut(String key) {
    String result = map.get(key);
    return (result != null) ? result : map.computeIfAbsent(key, this::createValue);
}

結論


一般に、一見平凡なリファクタリングによるそのような致命的な結果は、非常に予想外でした。ストレステストの存在によってのみ状況が救われ、劣化が明らかになりました。

幸い、新しいバージョンのJavaでこの問題が修正されました：JDK-8161372。

したがって、注意してください。魅力的なヒントを信頼したり、テストを作成したりしないでください。特にストレス。
