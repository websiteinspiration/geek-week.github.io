コンセプトベースのポリモーフィズムとは何ですか？なぜ必要なのですか？


C ++では、動的ポリモーフィズムは仮想関数と継承を使用して実装され、静的ポリモーフィズムはパターンを使用して実装されます。ここでは、これら2つのアプローチを組み合わせて、最善の方法を採用します。

継承を明示的に使用すると、コード接続が過剰になり、インターフェース分離（ISP）の原則に違反することがよくあります。これらの不利な点なしに動的なポリモーフィズムを実装する方法？

Sean Parentは、コンセプトベースのポリモーフィズムと呼ばれるイディオムを提案しました。継承は暗黙的であり、ユーザーから隠されています。これについての詳細は、彼のレポート「継承は悪の基本クラス」から学ぶことができます。彼は、Photoshopとアクション履歴を例として使用してアイデア全体を示し、「履歴ブラシ」が実際にどのように機能するかを学習します。

LLVMとそのデバイスについて少し


LLVMの例を使用して、このイディオムの利点を示します。LLVMはコンパイラを開発するためのインフラストラクチャです。以下は非常に高レベルのLLVMアーキテクチャで、記事の後半で使用されるエンティティのみをカバーしています。詳細については、公式ドキュメントを参照してください。



これはLLVMのアーキテクチャーであり、原則として、最近のコンパイラーのアーキテクチャーです。

主な部分は次のとおりです。

  • フロントエンドはプログラムのソースコードを取得し、中間表現（IR）に変換します。これにより、残りのコンパイラの作業が簡略化され、複雑なC ++コードを処理しなくなります。
  • ミドルエンド-最適化、分析、変換のセット。最も一般的な形式では、パス（パス）のセットです。すべてのパスは、PassManagerと呼ばれる特別なコンポーネントによって登録および起動されます。
  • バックエンドはターゲットコードを直接生成します。

コンパイラーは、いくつかの基本エンティティーの形式でプログラムを提示します。これは、モジュール（条件付きで.cppファイル）、関数、一連の命令を含む基本ユニットです。



LLVMにPassManagerの2つのバージョンが追加されました。 

  • LegacyPassManager、それは古典的なランタイム継承ベースのポリモーフィズムを使用します。継承階層には、モジュール、関数、ループなどで実行されるパスが含まれます。 
  • PassManager-コンセプトベースのポリモーフィズムに基づいた新バージョンで、LegacyPassManagerを置き換えることが提案されています。両方のバージョンが並行して存在し、独立して開発されます。

この記事のアイデアは、2つの異なる方法で実装される実際の製品コードの概念を示し、後者のアプローチの利点を示すことです。

LLVM PassManagerの概念ベースのポリモーフィズムの例


レガシーで実装されたとおり


まず、LegacyPassManagerですべてが古典的に配置される方法。PassManagerクラスがあり、Passクラスが1つあるとします。そのような階層があります。たとえば、クラスが継承するModulePassです。たとえば、定数伝播です。runOnModuleメソッドがあり、ここでは仮想です。したがって、通常の実行時ポリモーフィズムがあります。

/// ModulePass class - This class is used to implement unstructured
/// interprocedural optimizations and analyses. ModulePasses may do anything
/// they want to the program.
///
class ModulePass : public Pass {
...
/// runOnModule - Virtual method overriden by subclasses to process the module
/// being operated on.
virtual bool runOnModule(Module &M) = 0;
};

...

/// IPCP - The interprocedural constant propagation pass
///
struct IPCP : public ModulePass {
...
bool runOnModule(Module &M) override;
};

コードを見てみましょう。ここで問題は何ですか？この階層では、パッセージを開始する方法が、何を実行するか（関数-runOnFunction、モジュール-runOnModule、サイクル-runOnLoopなど）に応じて異なることがわかります。その結果、これにより、さまざまなIRエンティティで機能するパスのコレクションを単一の方法で処理することができなくなります（実際には、ポリモーフィズムを適用します）。明らかに、それを正しく実行する方法のように思われます。相続人で再定義される仮想実行メソッドが必要です。ただし、その後、問題が発生します。パラメータは常にそのタイプ（関数、モジュールなど）で渡されるため、後続クラスのrunメソッドには異なるシグネチャがあります。したがって、モジュール、関数などのダミー基本クラスを作成し、実行時にこのクラスへのポインターを渡し、メソッド内でダウンキャストする必要があります。このポインターに配置されているオブジェクトの種類によって異なります。そして、奇妙なことが始まります。新しい下位エンティティが出現すると、すべての設計原則に反する、上位のコードを毎回書き換える必要があります。

ケースごとにアダプタクラスを手動で作成できます。しかし、これは長くて面白くなく、実際にサービスを提供するコンポーネントのみの操作に便利なラッパークラスを書くという奇妙なアイデアです。テンプレートを使用してこのコードを自動的に生成することをお勧めします。これは、議論中のアプローチで対処することが提案されているものとまったく同じです。さらに、後で説明するように、さらに役立つ結果が得られます。

新しいバージョンで提案されたように


新しいPassManagerで次のことが起こります。多態性オブジェクトの概念は、次のように一般化されます。オブジェクトがメソッドを実装する場合は、ポリモーフィックにしたいメソッドのセットを導入します。また、このメソッドを実装するすべてのクラスはポリモーフィックであると言います。つまり、これらのクラスを交換可能なコンテキストで使用できます。これはPassManagerです。

LLVMのPassManagerクラスを検討してください。これは簡略版で、完全版はllvm / include / llvm / IR / PassManager.hにあります。IRテンプレートパラメーターは、渡すエンティティ（実行関数）に直接特化しています。これは、モジュール、関数、またはサイクルにすることができます。コードを見て、さらに説明します：

template <typename IR, typename... ArgTs> class PassManager {
public:
     void run(IR& ir, ArgTs... args) {
         for (auto& Pass : Passes) {
             Pass->run(graph, args...);
         }
     }

     template <typename PassT>
     void addPass(PassT Pass) {
         Passes.emplace_back(new detail::PassModel<IR, PassT, ArgTs...>(std::move(Pass)));
     }

private:
     std::vector<std::unique_ptr<detail::PassConcept<IR, ArgTs...>>> Passes;

};

次の基本エンティティを見てみましょう。

  • run , run
  • addPass ( ) PassT
  • Passes — , . , , type erasure,

それで、これはどのようなタイプであるべきですか？Passesベクトルには何が格納されていますか？

まず、PassModelとPassConceptが何であるかを見てみましょう。これらはPassManager内部のヘルパークラスです。どちらも詳細名前空間にあります。まず、PassConceptクラスがどのように見えるかを見てみましょう。これにも同じrunメソッドが含まれていますが、ここでは純粋に仮想的なメソッドです。 

namespace detail {

template <typename IR, typename... ArgTs> class PassConcept {
public:
 
     virtual ~PassConcept() = default;

     virtual void run(IR& ir, ArgTs... args) = 0;

};

2番目のクラスPassModelも定型です。PassConceptから継承されます。 

template <typename IR, typename PassT, typename... ArgTs> class PassModel final : public PassConcept<IR, ArgTs...> {
public:

     explicit PassModel(PassT Pass) : pass_(std::move(pass)) {}

     void run(IR& ir, ArgTs... args) final {
        pass_.run(ir, args...);
     }

private:

     PassT pass_;

};

} // end namespace detail

内容：

  • タイプPassTのプライベートpass_フィールド
  • 入力としてPassTタイプのオブジェクトを受け入れるコンストラクター。興味深いことは何もせず、動きのセマンティクスを使用してpass_を初期化するだけです
  • runメソッド。パスでrunメソッドを呼び出すだけです。それぞれ存在する可能性のあるすべての引数を渡します。

今どこから始めたか思い出します。次に、PassManagerはこれらすべてのパッセージを格納します。 PassConcept型要素からのベクトルパス。

つまり、全体像です。 PassManagerによって作成されます。 AddPassを使用して、モジュール、関数、ループなどに対して作成するパスを記録します。たとえば、インライン、一定の伝播、ループの展開などです。それら自体は誰からも継承されず、runメソッドのみが必要です。そして、この全体の概念だけがこれを提供します。どうやって？ 

インライン最適化があるとしましょう。 addPassでは、Inlineタイプのオブジェクトを渡します。したがって、パスでは、ベクターに、このインラインを既にPassConceptの形式で配置します。どうすればよいですか？ InlineはPassConceptクラスから継承されません。どのように要素をベクトルに入れるのですか？継承がないため、ベースタイプへのキャスト（アップキャスト）はできませんが、ここではそのようなトリックを行っています。インターフェイスを定義するこのPassConceptヘルパークラスがあります。彼の子孫はすべてrunメソッドを実装する必要があると彼は言います。 PassModelがあります。これは定型です。したがって、Inlineを配置すると、このPassModelはこのInline型でインスタンス化され、このオブジェクトはこのクラス内でコンパイルされます。 PassModel自体はrunを再定義しますが、それ自体はすでにこのパッセージのrunを呼び出しています。つまり、Inlineクラスから実行します。これはすべてコンパイル時に解決されます。Inlineがrunメソッドを定義していない場合、コンパイル時にエラーが発生します。 

したがって、この多型は継承なしで実現されます。問題が発生する可能性があります。継承ではないのですが、ここでは、PassModelがPassConceptから継承されているためです。回答：継承はありますが、それは内部的なものであり、突き出ていません。ユーザーはそれについて何も知りません。

私たちは概念的なレベルで話します。ここにユーザーがいて、特定のメソッドをオーバーライドしたいと考えています。同時に、彼は自分自身に余分な依存関係を引き込まないように、継承されることを望んでいません。どうやってするの？ PassConcept、PassModelを介して、継承を通じて実行時のポリモーフィズムを作成しますが、ユーザーはそれについて知りません。これらはすべて、これら2つのクラスの内部であり、名前空間で定義されています。

もう一度、これはどのように達成されますか？私はクラスを持っています、それを呼び出しましょう。コンパイラの観点から、それをインラインにします。Inlineをベクターにそれぞれ追加し、PassModelオブジェクトを作成します。このテンプレートパラメータのオブジェクトを受け取るコンストラクタがあります。したがって、PassManagerでrunメソッドを呼び出すと、すべてのパッセージを実行します。この場合、パスは1つだけで、Inline型です。PassConceptのrunCメソッドを呼び出します。PassModel内にある同じrunメソッドは、Inline型によってインスタンス化されます。すでにこのメソッドは、登録されたパッセージ（この場合はInline）のrunメソッドを呼び出しており、その結果、Inlineで実行されます。

アプローチの利点


これが、明示的に継承を使用せずに異なる動作をした方法です。これで、以前はLegacyPassManagerにあった明示的な依存関係がなくなりました。 

なんと珍しい再帰的なことがわかりました。 runメソッドをオーバーライドする任意のオブジェクトにポリモーフィズムを使用できます。 runメソッドはPassManager自体をオーバーライドするので、それ自体を登録することができます。つまり、自身をPassesパスベクトルに埋め込み、再度呼び出すことができます。 

すべてを組み合わせることができることがわかりました。 Legacyである古いPassManagerには明確な分離があります。モジュールで行われるモジュラー最適化があります。関数で発生する最適化があります。そして、ここではすべてがスムーズに行われます。 PassManagerを作成し、それを「モジュール」タイプでインスタンス化し、その中にインラインを配置します。他の何か、いくつかの他のモジュラー最適化を行います。次に、2番目のPassManagerを「関数」タイプでインスタンス化し、関数を最適化します。そして、モジュールによってインスタンス化されるPassManagerで、このPassesベクトルを介して、関数によってインスタンス化される別のPassManagerを置くことができます。 

PassManager<Module> MPM;
// ... register passes on module
MPM.addPass(GlobalDCEPass())
MPM.addPass(PGOInstrumentationGen());
//... register passes on function
PassManager<Function> FPM; 
FPM.addPass(CallSiteSplittingPass());
//... register all registered passes on function in module pass manager MPM.addPass(createModuleToFunctionPassAdaptor(std::move(FPM)));

フォローする時間がありますか？ 2つのPassManagerがあります。 1つはタイプIRモジュール、もう1つはタイプIR機能。モジュールのあるもので、いくつかのパスをすでに置いているとしましょう。次に、それらを関数で実行されるパスと混合します。何してるの？ addPassを呼び出して、PassManagerをPassとして渡します。これは、IRタイプ「関数」でインスタンス化されます（実際のコードでは、そこに配置されるのはPassManagerではなく、それをラップする特別なクラスですが、概念レベルでは重要ではありません）。 

したがって、異なるレベルの最適化を混在させることができます。PassManagerのネストのおかげで、モジュール、関数、サイクルなどへのパスを交互に実行できます。Legacy PassManagerでは、これはより複雑です。仮想関数runOnModuleを持つモジュール用の個別のクラス、仮想メソッドrunOnFunctionを持つ関数用の個別のクラスなどがあります。これらのクラスは両方ともPassの共通の祖先から継承されますが、互いに独立しており、異なるインターフェイスを持っているため、LegacyPassManagerを使用すると、さまざまなIRエンティティ（モジュール、関数、ループ）でパスを呼び出すのに不便になります
