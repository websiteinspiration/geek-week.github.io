Zimbraとサーバーの過負荷保護

電子メールは、ビジネスコミュニケーションの標準として定着しました。電子メールは経済効率が高く、テキストの引用や添付ファイルの添付に関連する多くの機能があるため、ドキュメント交換とビジネスコミュニケーションの普遍的な方法の役割に最適なのは電子メールです。これらの同じ機能が、スパマーが電子メールを非常に好む理由になっています。その結果、今日の電子メールは、ビジネスレターがたまにしか見られない海域で、スパムの巨大な荒れ狂う海になっています。そのため、メールサーバーの管理者の主なタスクの1つは、スパムメールからの保護です。 Zimbra Collaboration Suite Open-Source Editionでこれを使って何ができるかを見てみましょう。



無料のソリューションにもかかわらず、Zimbra OSEは、システム管理者に大量の非常に効果的なツールを提供して、不要な電子メールの受信の問題を解決できます。私たちはすでに、Amavis、SpamAssassin、ClamAV、cbpolicydなどのユーティリティについて書いています。これらのユーティリティを使用すると、スパムメールや感染したメールやフィッシングメールを除外することで、受信メールを確実にフィルタリングできます。ただし、それらの主な欠点は、すべてがすでに受信したメールメッセージで機能し、システムリソースを無駄なメッセージのフィルタリングに費やしていることです。しかし、企業が大量のボットネットの銃撃を受け、メールサーバーに絶え間なく大量の迷惑メールが送信されるため、それらをフィルタリングするだけでMTAサーバーの容量の大部分を占める場合はどうでしょうか。

理論的には、クラウドサービスを接続して受信メールをフィルタリングすることでこれから身を守ることができますが、実際にはこの保護方法はすべての企業に適しているわけではありません。この場合、サードパーティにスパムだけでなくビジネス通信の処理も委託する必要があるためです。安全で、多くの場合、企業のセキュリティポリシーに直接反する。さらに、クラウドスパムフィルターの信頼性に関連するリスクがあります。この状況から抜け出す方法は、それ自体でサーバー保護を構成することかもしれません。特にこれらの目的のために、Zimbraにはメールサーバーをロードせずにボットネットから送信された文字からメールサーバーを保護するように設計された組み込みのユーティリティPostscreenがありました。

Postscreenの作業の本質は、このユーティリティがメールサーバーへの接続要求をすべてスキャンし、サーバーに接続するのが疑わしいと思われるクライアントを許可しないことです。統計によると、世界中のスパムの約90％がボットネットによって送信されているため、Postscreenは多くの場合、不要なメール配信からメールサーバーを保護する最初の手段として使用されます。このおかげで、大規模なボットネットからの強力なスパム攻撃の状況下でも、メールサーバーは過負荷なしで安定して動作できます。

Postscreenの操作の原理は非常に単純です。ユーティリティは、受信メールを一連の簡単なチェックの前に実行してから、メールサーバーまたは他のサービスに転送してから、受信メールのより詳細で詳細な検証を行うことができます。それぞれのチェックに合格することはできますが、合格しないこともあります。小切手の各の結果に基づいて、Postscreenは、Zimbraの管理者の選択に3つのアクションのいずれかを適用することができます：ドロップは、無視または強制します。検証が失敗した場合、ドロップアクションはクライアントを強制的に切断し、無視アクションはでは、最終的な決定を行うときに検証の結果を無視できますが、同時にチェックに関する情報と統計を収集します。Enforceアクションを使用すると、最終的な決定を行うときに実行されたテストの結果を考慮することができますが、同時にシステム管理者が計画したすべてのテストを引き続き実行できます。

操作の単純な原理は、使いやすさと構成の容易さを意味しません。事実、Postscreenの設定を誤ると、企業にとって重要ないくつかのメッセージが宛先に届かなくなる理由になる可能性があります。そのため、Postscreenなどの強力なツールの設定に細心の注意を払い、特定の状況での動作を常にテストする必要があります。

ZimbraのPostscreenは最初から含まれていますが、多くの場合、初期設定に満足できない可能性があります。次に、セキュリティとリスクフリーの観点からPostscreenを構成するための最良のオプションを見ていきます。その本質は、チェックのいずれかが失敗した後、Postscreenはクライアントから切断せずに話すことなく、すべてのチェックを最後まで実行し、これらのチェックが失敗した場合はエラーメッセージを表示することです。これは、Postscreenがメッセージをスパムとして受け入れた場合、メッセージの配信不能を生きている送信者に通知します。これは、テスト設定で実施値を設定することによって実現されます。最初の失敗時にクライアントとの接続を切断することなく、開始されたチェックを最後まで完了することができるのはこの値ですが、同時に、完了時にスパムメールをブロックします。サーバーに配信せずに。

必要なチェックを有効にするには、次のコマンドを入力する必要があります

。 'zen.spamhaus.org = 127.0.0。[4..7] * 6' zimbraMtaPostscreenDnsblSites 'zen.spamhaus.org = 127.0.0.3 * 4' zimbraMtaPostscreenDnsblSites 'zen.spamhaus.org = 127.0.0.2 * 3'

このコマンド最も人気のある2つのパブリックスパムデータベースへの受信接続のDNSチェックを追加し、送信者アドレスが見つかったデータベースに応じてレターを評価できます。クライアントがペナルティを課せられた「スター」を取得するほど、スパマーである可能性が高くなります

zmprov mcf zimbraMtaPostscreenDnsblAction実施

このコマンドは、DNSチェックに合格した結果として実行されるアクションを決定します。この場合、テスト結果は記憶され、手紙自体はさらにテストを受け続けます。

zmprov mcf zimbraMtaPostscreenGreetAction実施

SMTPプロトコルでは、直接接続後、サーバーが最初にクライアントと通信するため、Postscreenはクライアントに挨拶を送信できます。多くのスパムクライアントは、挨拶の終了を待たずにコマンドの送信を開始するため、簡単に認識できます。このコマンドを使用すると、このテストの結果を考慮することができますが、同時にさらにテストを実行し続けます。

zmprov mcf zimbraMtaPostscreenNonSmtpCommandActionドロップ

このテストの一部として、Postscreenを使用すると、電子メールクライアントから送信されていない接続を除外できます。手紙を送らないので、心配することなく安全にサーバーから切断できます。

zmprov mcf zimbraMtaPostscreenPipeliningAction enforce

このチェックは、デフォルトでは、SMTPプロトコルでは、クライアントが一度に1つのコマンドしか送信できず、サーバーがこのコマンドに応答するのを待つことができるという事実に基づいています。ただし、多くのスパムボットは、サーバーからの応答を待たずに多数のコマンドを送信することにより、動作が異なります。これにより、事実上間違いなくスパムボットを特定できます。

原則として、これらのPostscreenのチェックは、スパムボットの大部分をサーバーから切り離し、メールサーバーの負荷を大幅に削減するのに十分なものです。同時に、生きている人々は手紙が届かなかったというメッセージを受け取ります。これにより、ポストスクリーンの設定が原因で重要な手紙を失うリスクが大幅に減少します。これが発生した場合は、信頼できる送信者をポストスクリーンのホワイトリストに追加できます。Postscreenのホワイトリストとブラックリストを作成するには、最初にファイル/ opt / zimbra / conf / postfix / postscreen_wblistを作成する必要があります。

その中に、許可および禁止されるIPアドレスとサブネットのリストをCIDRテーブルの形式で追加します。たとえば、サブネット121.144.169。*をブロックしますが、このサブネットから単一のIPアドレスへの接続を許可します。

＃ルールは指定された順序で評価されます。
＃ブラックリスト121.144.169。* 121.144.169.196を除く。
121.144.169.196/32許可
121.144.169.0/24拒否

エントリーの順序の重要性に注意を向けます。実際のところ、Postscreenは最初の一致までホワイトリストとブラックリストでファイルをスキャンし、ブロックされたサブネットが許可されたIPアドレスの前にある場合、このIPアドレスがホワイトリストと接続に追加されたというレコードにチェックが届かないサーバーは発生しません。

ホワイトリストとブラックリストを含むファイルを編集して保存したら、次のコマンドを使用して対応するチェックを有効にできます。

zmprov mcf zimbraMtaPostscreenAccessList "permit_mynetworks、cidr：/ opt / zimbra / conf / postfix / postscreen_wblist"
zmprov mcf zimbraMtaPostscreenBlacklistAction実施

これで、Postscreenは、既に設定したチェックに加えて、ホワイトリストとブラックリストでファイルにアクセスするため、管理者は信頼できる送信者をサーバーに接続できないという問題を簡単に解決できます。
