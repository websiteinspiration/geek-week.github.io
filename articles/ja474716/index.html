バグC ++ 20。ベルファストでの会議の結果

先日、ベルファストでC ++プログラミング言語の標準化委員会の会議がありました。C ++ 20に関する約400件のコメントが国の代表から委員会に流れ込み、その半分はなんとか対処することができました。

カットの下には、ロシアのコメント（そう、C ++ 20に関するあなたのコメント）の議論の結果、他の国からのコメント、そして適切なC ++ 23のニュース（Executors！）があります。

人々がサイトstdcpp.ru、チャット@ProCxx、Yandex.Taxiでの仕事、または会議で直接言及したC ++に関する問題はすべて、C ++ 20に関するコメントの形で設計しました。そしてそれから何が起こったのか...

std :: atomic <int> a {}; およびstd :: atomic <int> b;


奇妙に見えるかもしれませんが、変数aとbは 0に初期化されていません。アトミック変数をゼロに初期化するには、std :: atomic <int>を{0}で記述する必要がありました。

この振る舞いは完全に自明ではなく、多くの開発者が火傷しました。委員会は標準に関する私たちの意見を受け入れました。C++ 20では、std :: atomic_flagおよびstd :: atomicのデフォルトのコンストラクターは、それぞれ変数をクリア状態およびT（）で初期化します。

std ::コインランドリー


C ++ 17以降、標準には恐ろしい関数std :: launderがあります。委員会の人々は、標準のライブラリ開発者と一般ユーザーがそれに対処する方法を理解するだろうと考えました。

実際には、この機能を使用できない場合があります。それが原則的に必要であることは非常に明白ではありません：

struct C {
    const int c;
};

std::vector<C> v = {C{1}};
v.pop_back();
v.push_back(C{2});
assert(v.back().c == 2); //  ]:->

標準の文字に厳密に従っている場合、標準ライブラリの実装、コンパイラの機能、およびムーンフェーズによって、assertが成功または失敗することがあります。

問題について議論したところ、このような奇妙な動作を標準で説明するための最適化は、コンパイラーの1つにのみ実装され、具体的なパフォーマンスの向上をもたらさないことがわかりました。

したがって、C ++ 20以降では、リンクと定数フィールドを含む構造体をstd ::オプション、std ::バリアント、std ::ベクトル、std :: dequeなどに安全に格納できます。これで、新しい配置はstdロジックの一部を自動的に適用します。 ：洗濯。

*これはコンストラクタで


これは、失敗が待っていたコメントの1つです。

struct C {
    C(C&& other) noexcept
      : initial_(other.initial_)
      , secondary_(other.initial_) // O_O
    {}

    int initial_;
    int secondary_;
};

現在のルールによれば、O_Oのある行では、コンパイラーは＆other == thisであると想定する権利を持っているため、上の1行でother.initial_の値を書き直し、再度読み込む必要があります。

言い換えると、コンパイラーは、まだ作成されていないクラスがオブジェクトの構築元のパラメーターでエイリアスされていると想定する権利を有し、そのため、最適でないコードを生成する可能性があります。

一部のコンパイラー（GCCなど）は、ユーザーがそのような不名誉を書いていないと信じており、別名割り当ては不可能だと信じています。

委員会は、「エイリアシングが不可能であると仮定しよう」という私たちの発言を拒否しました。結局のところ、一部のコードベースには＆＆==これに恐ろしいハックがあり、人々はまだ彼らに別れを告げる準備ができていません。

残り


他のコメントによると、「Not in C ++ 20」という判定が出されました。
* constexprで__func__を使用することを望みました。
*概念が不完全なタイプを使用しないようにしました。そうしないと、複数のODR違反が発生するためです。しかし、その発言には予期しないプラスの効果がありました。requiresで不完全な型を使用すると、コンパイラが警告を発行するようになりました。
* std ::文字列の代入演算子を修正して、このような混乱が発生しないようにしました。

basic_string::operator=(charT c):
double d = 3.14;
std::string s;
s = d; // Compiles

委員会は、別の論文を書き、C ++ 20以降にこれを修正することを提案しました。
* std :: string_view：の一時的な行の割り当てを強制終了したかった。
std::string foo();

std::string_view sv;
sv = foo(); // Compiles... dangling reference

結果は前の発言と同じです：委員会は別の論文を書き、C ++ 20以降にこれを修正することを提案しました。
*次のコードのコンパイルを要求しました。

すべてが複雑で、C ++ 20を修復することは事実上不可能だと言われました。

他国からのコメント


重要な変更点：std ::コンストラクターの追加：range :: contiguous_rangeの概念を満たす型からのスパン。したがって、スパンはstd :: vectorおよびstd :: stringから暗黙的に作成できます。また、範囲:: contiguous_iteratorの概念を満たす2つの反復子からコンストラクタstd :: string_viewを追加しました。

楽しい編集は<compare>を期待していました。過去3年間で、演算子<=>は大きく変化しました。彼は等価性の比較に関与しなくなったため、<compare>の内容の3分の1は不要になりました。いくつかの国はこれに気づきました-彼らは基準を減らしました。

タイプではないテンプレートパラメータに忍び込んだ大きな変更。 C ++ 20では、ほとんどすべてのクラスをテンプレートパラメータとして渡すことが可能です（P1907を参照）。クラスに浮動小数点型または参照が含まれている場合でも、constexprデストラクタとパブリックメンバーを使用します。

また、欠落しているconstを標準のさまざまな部分に追加し、C ++ 20の新しい一部の関数の引数の名前と順序を変更しました。また、概念や範囲、標準テキストの省略形など、さまざまな編集が数多く行われています。

執行者


エグゼキュータは、C ++ 23の優先事項の1つです。これらは、リアクティブプログラミング、非同期プログラミング（ネットワーク、ディスク、プロセスなど）に必要であり、コルーチンシェルディングの基礎であり、サードパーティライブラリに単一のインターフェースを提供する必要があります。

同時に、エグゼキューターは内部実装用にアルゴリズムを最適化する必要があります。たとえば、コードの場合：

template <std::input_range Range>
void MySort(Range& data) {
    namespace stdr = std::ranges;
    std::sort(GetGlobalExecutor(), stdr::begin(data), stdr::end(data), 42);
}

GetGlobalExecutor（）関数は以下を返すことができます。
*シングルスレッドのエグゼキューター-通常のstd ::スレッドで実行する必要があります。
*マルチスレッドエグゼキュータ-並列ソートを実行する必要があります。
* GPU executor-データをビデオカードのメモリに移動し、そこでソートして、データを返す必要があります。
* NUMA executor-...
* ...

これまでのところ、このような機能を実装するには、恐ろしいカスタマイズポイントオブジェクト（CPO）を実行し、各アルゴリズムをそれらに変換する必要があります。委員会はそれを気に入らなかった...

しかし、少なくとも彼らは基本的なインターフェースであるP0443を予備的に承認した。エグゼキュータの後続のすべての文は、P0443のパッチの形式で記述する必要があります。
