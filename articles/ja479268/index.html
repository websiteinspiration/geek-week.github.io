イベントリスナーとWebワーカー

最近、Web Workers APIを見つけました。以前にこの十分にサポートされたツールに時間を割かなかったことは残念です。最新のWebアプリケーションは、メインのJavaScript実行スレッドの機能を非常に要求しています。これは、プロジェクトの生産性と、便利なユーザーエクスペリエンスを保証するプロジェクトの能力に影響します。最近のWebワーカーは、開発者が高速で便利なWebプロジェクトを作成するのに役立つものです。



すべてを理解した瞬間


Webワーカーには多くの肯定的な資質があります。しかし、特定のアプリケーションが複数のDOMイベントリスナーを使用する状況に遭遇したとき、私は本当にそれらの有用性に気付きました。フォームの送信、ウィンドウのサイズ変更、ボタンのクリックなどのイベント。これらのリスナーはすべてメインスレッドで動作する必要があります。メインスレッドが特定の操作で過負荷になり、実行に時間がかかる場合、ユーザーの影響に対するイベントリスナーの反応率に悪影響を及ぼします。アプリケーションは「スローダウン」し、イベントはメインスレッドの解放を待機しています。

イベントリスナーが私に大きな関心を示したのは、最初にWebワーカーが解決するように設計されているタスクを誤解したためです。最初は、コード実行の速度を改善するのに役立つと思いました。アプリケーションのコードの一部が並列に、別々のスレッドで実行される場合、アプリケーションは一定の期間内により多くのことができると思いました。しかし、Webプロジェクトコードの実行中、何かを始める前に、何らかのイベントを待つ必要があるという状況はよくあります。いくつかの計算が完了した後にのみDOMを更新する必要があるとしましょう。これを知って、私は、とにかく待機しなければならない場合、あるコードの実行を別のスレッドに転送しても意味がないと単純に信じていました。

ここにあなたがここで思い出すことができるコードの例があります：

const calculateResultsButton = document.getElementById('calculateResultsButton');
const openMenuButton = document.getElementById('#openMenuButton');
const resultBox = document.getElementById('resultBox');

calculateResultsButton.addEventListener('click', (e) => {
    // "    -, ,   , 
    //   DOM   ?"
    const result = performLongRunningCalculation();
    resultBox.innerText = result;
});

openMenuButton.addEventListener('click', (e) => {
    //      . 
});

ここでは、いくつかの計算が完了した後でフィールド内のテキストを更新しています。このコードが完了する前にDOMは更新されないため、このコードを別のスレッドで実行しても意味がないようです。その結果、もちろん、このコードは同期的に実行する必要があると判断しました。しかし、そのようなコードを見て、最初はメインスレッドがブロックされている限り、他のイベントリスナーが起動しないことを理解できませんでした。これは、「ブレーキ」がページに表示され始めることを意味します。

ページを「遅くする」方法


これは、上記を示すCodePenプロジェクトです。


ページが「遅くなる」状況を示すプロジェクト。

ボタンを押すFreezeと、アプリケーションが同期タスクの解決を開始します。これにはすべて3秒かかります（長い計算のパフォーマンスをシミュレートします）。同時にボタンをクリックした場合Increment-3秒経過するまで、フィールドの値はClick Count更新されません。このフィールドには、Increment3秒が経過した後にのみ、のクリック数に対応する新しい値が書き込まれます。一時停止中はメインストリームがブロックされます。その結果、アプリケーションウィンドウ内のすべてが機能していないように見えます。アプリケーションインターフェイスはフリーズされています。 「凍結」の過程で発生するイベントは、メインストリームのリソースを使用する機会を待っています。

クリックしFreezeて要素のサイズを変更しようとした場合resize me!、その後、再び、3秒が経過するまで、フィールドのサイズは変更されません。その後もフィールドのサイズは変わりますが、インターフェイスの「滑らかさ」について話す必要はありません。

イベントリスナーは、一見したところよりもはるかに大きな現象です。


前の例で示したように動作するサイトでの作業は、どのユーザーにとっても望ましくありません。ただし、ここではいくつかのイベントリスナーのみが使用されています。現実の世界では、完全に異なるスケールについて話しています。私はChromeでこのメソッドを使用することにgetEventListenersし、次のスクリプトを使用して、さまざまなページのDOM要素に接続されているイベントリスナーの数を調べます。このスクリプトは、開発者ツールコンソールで直接実行できます。そこで彼は：

Array
  .from([document, ...document.querySelectorAll('*')])
  .reduce((accumulator, node) => {
    let listeners = getEventListeners(node);
    for (let property in listeners) {
      accumulator = accumulator + listeners[property].length
    }
    return accumulator;
  }, 0);

このスクリプトを別のページで実行して、それらで使用されているイベントリスナーの数を調べました。私の実験の結果を次の表に示します。

Webワーカーによる「ブレーキ」の解消


上記のすべてを前提として、前の例を書き直してみましょう。これが彼の新しいバージョンです。見た目は古いものとまったく同じですが、内部の配置が異なります。つまり、メインスレッドをブロックするために使用されていた操作が、独自のスレッドに移動されました。この例で前の例と同じことを行うと、深刻な正の違いに気付くでしょう。つまり、ボタンをクリックした後でをFreezeクリックするIncrementと、フィールドClick Countが更新されます（Webワーカーの完了後、いずれの場合も、Click Count数値に1が追加されます）。要素のサイズ変更についても同様です。resize me!。別のスレッドで実行されるコードは、イベントリスナーをブロックしません。これにより、以前は単にページを「凍結」した操作の実行中であっても、ページのすべての要素が操作可能なままになります。

この例のJSコードは次のとおりです。

const button1 = document.getElementById('button1');
const button2 = document.getElementById('button2');
const count = document.getElementById('count');
const workerScript = `
  function pause(ms) {
    let time = new Date();
    while ((new Date()) - time <= ms) {}
               }

  self.onmessage = function(e) {
    pause(e.data);
    self.postMessage('Process complete!');
  }
`;
const blob = new Blob([
  workerScript,
], {type: "text/javascipt"});

const worker = new Worker(window.URL.createObjectURL(blob));

const bumpCount = () => {
  count.innerText = Number(count.innerText) + 1;
}

worker.onmessage = function(e) {
  console.log(e.data);
  bumpCount();
}

button1.addEventListener('click', async function () {
  worker.postMessage(3000);
});

button2.addEventListener('click', function () {
  bumpCount();
});

このコードを少し掘り下げると、Web Workers APIを配置してより快適にすることができますが、それを操作することに特に怖いものは何もないことに気付くでしょう。このコードは、シンプルですぐに作成されたデモであるため、恐ろしく見えます。APIの使いやすさを改善し、Webワーカーとの作業を容易にするために、いくつかの追加ツールを使用できます。たとえば、私は次の興味深いことに気づきました：

  • Workerize -Webワーカーでモジュールを実行できます。
  • Greenlet -Webワーカーで非同期コードの任意のフラグメントを実行できるようにします。
  • Comlink -Web Workers APIを抽象化する便利なレイヤーを提供します。

概要


Webアプリケーションが典型的な最新のプロジェクトである場合は、多くのイベントリスナーがある可能性が非常に高いことを意味します。また、メインスレッドで、他のスレッドで完全に実行できる多くの計算を実行することもできます。その結果、ユーザーとイベントのリスナーの両方に優れたサービスを提供し、「重い」計算をWebワーカーに任せることができます。

Webワーカーに対する過度の熱意と、Webワーカーへのユーザーインターフェイスに直接関連しないすべてのものを削除することは、おそらく最良のアイデアではないことに注意してください。このようなアプリケーションの処理には多くの時間と労力が必要になる場合があり、プロジェクトコードはより複雑になり、そのような変換のメリットは非常に小さくなります。代わりに、本当に「重い」コードを探し、それをWebワーカーに配置することから始める価値があるかもしれません。時間が経つにつれ、ウェブワーカーを使用するという考え方はより身近になり、インターフェイス設計の段階でもおそらくそれによって導かれるでしょう。

とにかく、Web Workers APIを理解することをお勧めします。このテクノロジーは非常に幅広いブラウザのサポートを享受しており、パフォーマンスのための最新のWebアプリケーションの需要が高まっています。したがって、ウェブワーカーのようなツールの研究を拒否する理由はありません。
