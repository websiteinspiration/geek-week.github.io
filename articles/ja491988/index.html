Red Hat OpenShift 4.2および4.3の新機能


OpenShiftの4番目のバージョンは比較的最近にリリースされました。現在のバージョン4.3は1月の終わりから利用可能であり、そのすべての変更は、3番目のバージョンにはなかった完全に新しい何か、またはバージョン4.1に登場したもののメジャーアップデートです。OpenShiftを使用しており、新しいバージョンへのアップグレードを計画しているユーザーを理解し、理解して考慮する必要があります。

OpenShift 4.2のリリースにより、Red HatはKubernetesを簡素化しました。コンテナ、CI / CDパイプライン、サーバーレスデプロイメントを作成するための新しいツールとプラグインが登場しました。イノベーションにより、開発者はKubernetesとの訴訟ではなく、コードの記述に集中する機会が与えられます。

実際、OpenShift 4.2および4.3のバージョンの新機能は何ですか？

ハイブリッドクラウドに移行


新しいITインフラストラクチャを計画するとき、または既存のITランドスケープを開発するとき、企業は、プライベートクラウドソリューションを実装したり、パブリッククラウドプロバイダーの力を利用したりするITリソースを提供するクラウドアプローチをますます検討しています。したがって、共通の管理システムを備えたオンプレミスリソースとパブリッククラウドリソースの両方が使用される場合、現代のITインフラストラクチャは、「ハイブリッド」クラウドモデル上で構築されることが増えています。Red Hat OpenShift 4.2は、ハイブリッドクラウドモデルへの移行を簡素化するように特別に設計されており、AWS、Azure、Google Cloud Platformなどのプロバイダーのリソースを、VMwareおよびOpenStackのプライベートクラウドとともにクラスターに簡単に接続できます。

新しいインストール方法


バージョン4では、OpenShiftのインストール方法が変更されました。 Red Hatは、OpenShiftクラスターをデプロイするための特別なユーティリティ-openshift-installを提供します。ユーティリティは、Goで記述された唯一のバイナリファイルです。 Openshit-installerは、デプロイに必要な構成を含むyamlファイルを準備します。

クラウドリソースを使用してインストールする場合、将来のクラスターに関する最小限の情報を指定する必要があります。DNSゾーン、ワーカーノードの数、クラウドプロバイダーの特定の設定、クラウドプロバイダーにアクセスするためのアカウント情報。構成ファイルを準備した後、1つのコマンドでクラスターをデプロイできます。

プライベートクラウド（vSphereとOpenStackがサポートされている）を使用している場合や、ベアメタルサーバーにインストールしている場合など、独自のコンピューティングリソースにインストールする場合は、インフラストラクチャを手動で構成する必要があります。コントロールプレーンクラスターの作成に必要な仮想マシンまたは物理サーバーの最小数を準備し、構成します。ネットワークサービス。この構成の後、1つのopenshift-installerユーティリティーコマンドでOpenShiftクラスターを同様に作成できます。

インフラストラクチャの更新


CoreOSとの統合


重要な更新は、Red Hat CoreOSとの統合です。現在、Red Hat OpenShiftマスターノードは新しいOSでのみ機能します。これは、コンテナ化されたソリューション用に特別に設計されたRed Hatの無料のオペレーティングシステムです。 Red Hat CoreOSは、コンテナーを実行するために最適化された軽量のLinuxです。

3.11でオペレーティングシステムとOpenShiftが別々に存在していた場合、4.2ではOpenShiftと密接にリンクしています。これは単一のアプライアンスであり、不変のインフラストラクチャです。


すべてのノードでRHCOSを使用するクラスターの場合、OpenShift Container Platformの更新はシンプルで十分に自動化されたプロセスです。

以前は、OpenShiftを更新するために、最初に製品が実行されていた基礎となるオペレーティングシステムを更新する必要がありました（当時はRed Hat Enterprise Linuxでした）。その後、ノードごとにOpenShiftを徐々に更新することが可能になりました。プロセスの自動化についての話はありませんでした。

これで、OpenShift Container PlatformはOSを含む各ノードのシステムとサービスを完全に制御するため、このタスクはWebインターフェースからボタンを押すことで解決されます。その後、OpenShiftクラスター内で特別なオペレーターが起動され、更新プロセス全体を制御します。

新しいCSI


2つ目は、ストレージインターフェイスコントローラーである新しいCSIです。これにより、さまざまな外部ストレージシステムをOpenShiftクラスターに接続できます。OpenShift用のストレージドライバーのプロバイダーの多くは、ストレージシステムの製造元自身が作成したストレージドライバーに基づいてサポートされています。サポートされているCSIドライバーの完全なリストは、このドキュメント（https://kubernetes-csi.github.io/docs/drivers.html）にあります。このリストには、主要メーカー（Dell / EMC、IBM、NetApp、Hitachi、HPE、PureStorage）、SDSソリューション（Ceph）、およびクラウドストレージ（AWS、Azure、Google）のディスクアレイのすべての主要モデルがあります。OpenShift 4.2は、CSIバージョン1.1仕様のCSIドライバーの操作をサポートしています。

RedHat OpenShift Service Mesh


Istio、Kiali、およびJaegerプロジェクト-Red Hat OpenShift Service Meshに基づいて、サービス間でリクエストをルーティングする通常のタスクに加えて、サービスを追跡して視覚化できます。これは、開発者がRed Hat OpenShift内にデプロイされたアプリケーションの相互作用、監視、および管理を簡素化するのに役立ちます。


Kiali

を使用したマイクロサービスアーキテクチャによるアプリケーションの視覚化 Service Meshのインストール、サービス、およびライフサイクル管理を簡素化するために、Red Hat OpenShiftは管理者に特別なオペレーター-Service Mesh Operatorを提供します。これはKubernetesオペレーターであり、再構成されたIstio、Kiali、およびJaegerパッケージをクラスターにデプロイして、アプリケーション管理の管理負担を最小限に抑えることができます。

Dockerの代わりにCRI-O


デフォルトのコンテナランタイムDockerはCRI-Oに置き換えられました。CRI-Oはすでにバージョン3.11で使用できましたが、4.2ではそれがメインになりました。良くも悪くもありませんが、製品を使用する際には覚えておく価値があります。

オペレーターとデプロイアプリケーション


演算子-4番目のバージョンに登場したRedHat OpenShiftの新しいエンティティ。これは、Kubernetesアプリケーションをパッケージ化、デプロイ、管理するための方法です。これは、Kubernetes APIとkubectlツールを使用して管理されるコンテナーにデプロイされたアプリケーションのプラグインと考えることができます。

Kubernetesオペレーターは、クラスターにデプロイするアプリケーションのライフサイクルの管理に関連するタスクを自動化するのに役立ちます。たとえば、オペレーターは更新の自動化、アプリケーションのバックアップとスケーリング、構成の変更などを行うことができます。オペレーターの完全なリストは、https：//operatorhub.io/にあります。

OperatorHubは、管理コンソールのWebインターフェースから直接利用できます。Red Hatが管理するOpenShiftのアプリケーションカタログです。それら。Red Hatによって承認されたすべてのオペレーターは、ベンダーサポートの対象になります。

ユニバーサルベースイメージ


これは、コンテナ内にアプリケーションを作成するために使用できるRHEL OSイメージの標準化されたセットです。最小限、標準、完全なセットがあります。それらはほとんどスペースを取らず、必要なすべてのインストール済みパッケージとプログラミング言語をサポートします。

CI / CDツール


RedHat OpenShif 4.2には、Tekton Pipelinesに基づいてJenkinsとOpenShift Pipelinesのどちらかを選択する機能があります。

OpenShift PipelinesはTektonに基づいています。Tektonは、コードとGitOpsが近づくにつれてPipelineにより適切にサポートされます。OpenShiftパイプラインでは、各ステップは独自のコンテナーで実行されるため、リソースはステップの実行中にのみ使用されます。これにより、開発者は、管理用の中央CI / CDサーバーなしで、モジュール配信パイプライン、プラグイン、およびアクセス制御を完全に制御できます。

OpenShift Pipelinesは現在、開発者向けのプレビュー段階にあり、OpenShift 4クラスターのオペレーターとして利用できます。もちろん、OpenShiftユーザーはRedHat OpenShift 4でJenkinsを使用できます。

開発者向けの管理アップデート


4.2では、OpenShiftは開発者と管理者の両方のWebインターフェースを完全に更新しました。

以前のバージョンのOpenShiftでは、全員が3つのコンソール（サービスカタログ、管理者コンソール、および作業コンソール）で作業していました。これで、クラスターは2つの部分（管理者コンソールと開発者コンソール）に分かれています。

開発者コンソールは、大幅なユーザーインターフェイスの改善を受けています。アプリケーションとそのアセンブリのトポロジを表示する方が便利です。これにより、開発者は、コンテナー化されたアプリケーションとクラスターリソースを簡単に作成、展開、視覚化できます。彼らにとって重要なことに集中することができます。

おど


Odoは、OpenShiftでのアプリケーション開発を簡素化する開発者向けのコマンドラインユーティリティです。このCLIは、Gitプッシュスタイルの相互作用を使用して、Kubernetesを初めて使用する開発者がOpenShiftでアプリケーションを構築するのに役立ちます。

開発環境との統合


開発者は、Microsoft Visual Studio、JetBrains（IntelliJを含む）、Eclipseデスクトップなどのお気に入りのコード開発環境を離れることなく、OpenShiftでアプリケーションを作成、デバッグ、デプロイできるようになりました。

Microsoft Azure DevOps用のRed Hat OpenShiftデプロイメント拡張機能


Microsoft Azure DevOps用のRed Hat OpenShift Deployment拡張機能が登場しました。このDevOpsツールセットのユーザーは、Microsoft Azure DevOpsから直接、アプリケーションをAzure Red Hat OpenShiftまたはその他のOpenShiftクラスターにデプロイできます。

3番目のバージョンから4番目のバージョンへの移行


私たちは更新ではなく新しいリリースについて話しているので、4番目のバージョンを3番目のバージョンの上に置くことはそれほど簡単ではありません。3番目から4番目のバージョンへの更新はサポートされません。

しかし、朗報があります。RedHatは、プロジェクトを3.7から4.2に移動するためのツールを提供しています。Cluster Application Migration（CAM）ツールを使用して、アプリケーションワークロードを転送できます。CAMを使用すると、移行を制御し、アプリケーションのダウンタイムを最小限に抑えることができます。

この記事で説明する主なイノベーションは、バージョン4.2で登場しました。最近リリースされた4.3では、変更はそれほど重要ではありませんが、まだ何か新しいものがあります。変更のリストは非常に広範囲であり、私たちは私たちの意見で最も重要なものを与えます：

Kubernetesのバージョンを1.16に更新します。


バージョンはすぐに2つのステップにアップグレードされ、OpenShift 4.2では1.14でした。

etcdでのデータ暗号化


バージョン4.3以降、etcdデータベース内のデータを暗号化することが可能になりました。暗号化を有効にすると、OpenShift APIおよびKubernetes APIリソースのシークレット、ConfigMap、ルート、アクセストークン、OAuth認証を暗号化できます。

プロジェクトダッシュボードの更新


新しいバージョンのプロジェクトダッシュボードでは、プロジェクトページに追加情報（プロジェクトのステータス、リソースの使用状況、プロジェクトの割り当て）が表示されます。
