分裂とルール


データベース（特にPostgreSQL）で作業しているとき、テーブルからデータを（Go GoPを使用して）並行して選択するというアイデアがありました。そして、「個々のゴルーチンのサンプル行をスキャンすることは可能ですか？」結局のところ

、func（* Rows）Scanをgourutinsで同時に呼び出すことはできません。この制限に基づいて、文字列のスキャンと並行して他のプロセス、特に結果のデータの準備を実行することにしました。

なぜなら スキャンはポインターに従ってデータをスタックし、2つのスライス（後で2つ理由を説明します）を作成することにしました。その間、スキャンを切り替えます。残りのグルテンは既に選択されたデータを処理します。

最初に、サンプル列の数を知る必要があります。

columns, err = rows.Columns()
count := len(columns)

次に、値とこれらの値へのポインターを使用して2つのスライスを作成します（行のスキャン中にデータを追加します）。

values := make([]interface{}, count)
valuesPtrs := make([]interface{}, count)
values_ := make([]interface{}, count)
valuesPtrs_ := make([]interface{}, count)

for i := range columns {
	valuesPtrs[i] = &values;[i]
	valuesPtrs_[i] = &values;_[i]
}

この例では、選択結果をマップ[文字列]文字列に追加します。キーは列名になります。タイプを示す特定の構造を使用できますが、この出版物の目的は、提案されたアプローチがどれほど実行可能であるかをhabrasocietyから見つけることです、マップの選択について詳しく見てみましょう。

次に、2つのゴルチンを分離します。そのうちの1つが結果のマップを形成します。

func getData(deleteNullValues bool, check, finish chan bool, dbData chan interface{}, columns []string, data *[]map[string]string) {
	lnc := len(columns)
	for <-check {
		row := make(map[string]string)
		for i := 0; i < lnc; i++ {
			el := <-dbData
			b, ok := el.([]byte)
			if ok {
				row[columns[i]] = string(b)
			} else {
				if el == nil {
					if deleteNullValues == false {
						row[columns[i]] = ""
					}
				} else {
					row[columns[i]] = fmt.Sprint(el)
				}
			}
		}
		*data = append(*data, row)
	}
	finish <- true
}

そして、2番目は、スキャンによって生成された値を持つ2つのスライスを切り替え、それらを前のgourutinのチャネルに送信します（結果を形成します）。

func transferData(values, values_ []interface{}, dbData chan interface{}, swtch, working, check chan bool) {
	for <-working {
		check <- true
		switch <-swtch {
		case false:
			for _, v := range values {
				dbData <- v
			}
		default:
			for _, v := range values_ {
				dbData <- v
			}
		}
	}
}

メインプロセスでは、ポインターのスライスを切り替えてデータを選択します。

for rows.Next() {
	switch chnl {
	case false:
		if err = rows.Scan(valuesPtrs...); err != nil {
			fmt.Printf("rows.Scan: %s\n%s\n%#v\n", err, query, args)
			return nil, nil, err
		}
	default:
		if err = rows.Scan(valuesPtrs_...); err != nil {
			fmt.Printf("rows.Scan: %s\n%s\n%#v\n", err, query, args)
			return nil, nil, err
		}
	}
	working <- true
	swtch <- chnl
	chnl = !chnl
}

データベースでは、32列のテーブルを作成し、10万行を追加しました。
テストの結果（データを50回サンプリングする場合）、次のデータを取得し
ました。費やした時間：1m8.022277124s-1つのスライスを使用して結果をサンプリング
費やした時間：1m7.806109441s-2つのスライスを使用して結果をサンプリング

反復回数を100に増やした場合：
時間費やした時間：2m15.973344023s-1つのスライスを使用した結果の選択
費やした時間：2m15.057413845s-2つのスライスを使用した結果の選択

差は、データ量の増加とテーブルの列の増加に伴って増加します。
ただし、データ量が減少するか、テーブルの列数が減少すると、反対の結果が観察されました。これは、原理的には理解できるためです。準備ステップのオーバーヘッドとグルティン部門が貴重な時間を「食い尽くす」ため、結果が平準化されます。

2つのスライスと2つのゴルチンについて：多数のスライスを使用してテストを実行しましたが、データベースから値をスキャンするよりもgetData関数とtransferData関数がデータを高速に処理するため、サンプリング時間が増加しました。したがって、より多くのコアがある場合でも、スキャン用の新しいスライスと追加のゴルーチン（非常にワイルドなデータボリュームを除く）を追加しても意味がありません。
