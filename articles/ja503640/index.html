本日の翻訳を公開している記事の著者は、JavaScriptの知識とスキルを常に向上させようとしています。彼は、プロとしての成長を目指して、自分の作品をどのように構築しているかについて話したいと思っています。



コードエディターの選択


最近、無数のコードエディタがあります。これは、プログラマーが生産的な作業を確実にするために何を選択すればよいかわからないという事実につながります。

主にVisual Studio Codeを使用しています。私は、Androidの開発、Android Studioを使用する開発、およびXcodeで作業するiOS開発の場合のみ例外を設けています。

VS CodeはMicrosoftプロジェクトです。はい、それはMicrosoftであり、私の見方では、非常に優れているようです。このエディターは、関連するほぼすべてのプログラミング言語をサポートし、膨大な数の拡張機能が作成されています。スマートヒントの出力をサポートし、優れたインターフェースを備えています。そして、それはとりわけ、システムに過度の負荷をかけません（もちろん、それは「簡単な」Sublime Textではありませんが、それでもなお、システム要件の点ではやや控えめです）。


VSコード

私は（そして、VSのコードは非常に人気がありませんでした）崇高テキストでのみ動作するために使用しました。ただし、多くの拡張機能（これについては後で説明します）を使用すると、作業中にマイナーエラーを自分で修正する必要がないため、多くの時間を節約できます。これらの拡張機能の中には、エラーを自動的に検出して修正するもの、コードをフォーマットするもの、Gitでの作業を簡素化するもの、端末を使用できるものなどがあります。

PHPで書くと、PhpStormが好きになります。Pythonで作業している場合は、PyCharmを確認する必要があります。間違いなく、これらのエディター（IDE）は強力なツールです。ただし、これらは1つの言語のみを対象としています。私はJavaScript、HTML、PHP、Node.js、React、Dockerを使用するフルスタック開発者です。したがって、私はVS Codeを選択しました。それは、優れた機能を備えたエディターであり、多くの拡張機能をサポートしており、特に気に入っているのは、インテリジェントなコード補完システムが含まれているためです。

ESLintアプリケーション


以前は、構文エラーが私の時間の多くを占めていました。さらに、彼らは私をとても怒らせました。これらのエラーには、宣言されていない変数や関数、値の外観null、コードの構造要素の欠落などがあります。プロジェクトのサイズが大きくなるにつれて、開発者は大量のコードを常に読み取らなければならないという事実に飽きてきます。彼はこのコードをより悪く認識し、プログラムのテキストを入力する際に​​タイプミスやその他の間違いを犯します。

ESLintを使い始めたとき、すべてが変わりました。これは、一般的な標準に従って、コードのエラーを探し、構文をチェックし、コードをフォーマットできるプラグインです。 ESLintを使用すると、コードエラーを減らし、見栄えのよいコードを作成できます。 ESLintを使用すると、通常のJavaScriptコードと、ReactやVueなどの一般的なライブラリやフレームワークを使用して記述されたコードの両方をチェックできます。

ESLintは、VS Codeと組み合わせると特に優れています。コードが入力されると、エラーと不正な構文構造がチェックされます。プログラマーには、関数と変数の最適な使用に関するヒントが与えられます。自動コードフォーマットをサポートしています。一般的に-私はESLintが本当に好きです。


VSコードでのESLintの使用

コードのフォーマットについて説明する場合、ESLintには代替のPrettierがあります。しかし、ESLintの機能は単純なコードのフォーマットよりもはるかに広いため、私はESLintを好みます。

最適なディレクトリ構造の選択


私が最近採用した1つのアイデアを共有したいと思います。「プロジェクトの作業の最初に、プロジェクトの構造を最適化しようとしないでください。」

以前は、プロジェクトの大小に関わらず、常にこのプロジェクトに最適なディレクトリ構造を見つけるために多くの時間を費やしていました。 「Node.jsプロジェクトのフォルダー構造に関する推奨事項」や「Reactアプリケーションの構造」という記事を検索して読みました。しかし、選択した構造の最適性を完全に確信することはできませんでした。自分のコードを正しく編成できるかどうかもわかりませんでした。その結果、すべてに長い時間がかかりました。

さらに、最初は最初に選択した構造に適合させようとしたが、それが「正しい」ものであると考えて、数日後、プロジェクトで混乱が始まったことに気づいた。私はシステム思考の分野にはあまり強くありません。そして、作業の最初のプロジェクトが非常にきれいに見えたからといって、しばらくしてから作業を続けてもすべてが残るというわけではありません。

プロジェクトアーキテクチャの選択について、最初からすべてをどのように編成するかについて、あまり長く考えないでください。最初に、一般的な方向性を選択し、必要なライブラリまたは必要なフレームワークを取り、プロジェクトの成長と発展に合わせてプロジェクトの構造を構築および改善することから作業を開始する必要があります。これは、プロジェクト構造への実用的なアプローチです。

プロジェクトを適切に構造化したい場合は、NestJS Node.jsフレームワークに注意することをお勧めします。私はこのフレームワークにかなり精通していることに注意したい。これは非常に優れたアーキテクチャーを持っています（AngularJSで使用されているアーキテクチャーに似ていますが、私は特にAngularが好きではありません）。

コードに問題があると疑われる場合のconsole.logの使用


JavaScriptでプログラミングするときに最も頻繁に使用する構造がメソッドであると確信していconsole.logます。この方法を適用する主な目的は、プログラムの特定の場所で使用されるデータを確認することです。

個人的には、プログラミングでは、使用する言語に関係なく、すべてが基本的にデータを中心に展開すると思います。したがって、期待どおりに機能しないコードを見つけた場合は、そのコードがを使用して機能するデータを確認することをお勧めしますconsole.log。

そのような状況では、表現の使用はより専門的に見えると多くの人が言いdebuggerます。はい、Google Chromeは優れたコードデバッグツールをサポートしていますconsole.log。しかし、たとえば、私はこれすべてを必要としません。使用するconsole.logより高度なツールを使用する場合と同様に、どのコード行がチェックされるかを知っています。私の見解では、console.log他のデバッグメカニズムよりも高速で便利です。さらに、大企業の著名なプログラマーが書いたチュートリアルを見てきました。そして、それconsole.logはそこにも適用されます。

私が注目したいのは、コードを理解して適用したconsole.log後、このデザインをコードから削除する必要があることです。からconsole.logリポジトリにコードを送信する必要はありません。コードリーダー（私は彼らに属しています）は通常、これを好まないでしょう。

タイムリーにコードにコメントする


プログラムを開発する過程で、長くて複雑なコードを書かなければならないことがよくあります。プログラマーがそのようなコードを作成するとき、彼は後でこのコードに出会ったので、彼は単にその中で何が起こっているのか理解しないだろうという正当な恐れがあります。同様の懸念は、プログラマーが作成したコードをいつか使用しなければならない人々の懸念によって決定されるかもしれません。これらの人々は、彼らが読むコードを理解する必要もあります。

コードコメントの習慣は非常に便利です。これは、何人かのプログラマーが取り組んでいるプロジェクトで特に重要です。コードがよくコメントされているプロジェクトに取り組んでいるチームメンバーは、同僚と話す時間を費やさず、プログラムの理解できない断片に会います。その結果、それらは自分自身に気を取られたり、忙しいかもしれない他のプログラマの気を散らしたりすることはありません。その結果、いわば、コード自体が説明されていれば、そのコードを使用する人はすぐに理解できます。時間を節約できます。

しかし、コメントはまず第一に関連性があり、第二に-見栄えが良くなければなりません。絶対にすべてについてコメントする必要はありません。コメントでやり過ぎると、コードの読み取りが複雑になります。


コメント付き

のコードコードの理解可能性について話す場合、通常、変数と関数の名前を選択して、これらの変数と関数の本質を説明できるようにしたいと思います。私は、関数とクラスが長くなりすぎないように努め、多くの問題を解決しています。他の極端な例のように、コードの読み取りが複雑になるため、コードを無理に実行したり、コードを小さすぎる部分に分割したりしないように、それらを小さな関数とクラスに分割します。何よりも、あなたがそれを読んだときにその意味がコメントなしで明確であるようにコードが書かれているなら これは理想的なオプションです。コードなしではどうしようもないと感じるときだけ、コードにコメントする必要があります。

約束とコールバックの拒否と非同期/待機への移行


promiseプロミスとコールバックの短所


プログラムを作成するときは、さまざまなAPIを使用して多くの作業を行う必要があります。たとえば、どこかに表示する必要があるデータを取得するためのAPIの呼び出しは次のようになります。

//   
axios.get('/users')
.then(response => {
    console.log(response)
})
.catch(error => {
    console.log(error)

そして、これは通常、別のAPIを呼び出す必要があるときに起こり、最初のAPIへの呼び出しが成功したときにのみこれを行います。

//   
axios.get('/users')
.then(response => {
   console.log(response)
   //   
   axios.get('/addresses')
    .then(response1 => {
        console.log(response1)
    })
    .catch(error1 => {
        console.log(error1)
    })

.catch(error => {
    console.log(error)

しかし、上記と同じ方法で多くのAPIを操作する必要がある場合はどうでしょうか。これは、外部APIへのリクエストを処理するプロセスがより複雑になったときに、プロジェクトサイズが大きくなる状況で起こります。

a非同期の強み/待つ


async / awaitデザインはES6標準に登場しました（2015年にリリースされました）。これは、非同期操作の処理におけるプロミスとコールバックの代替手段です。この構造の良いところは、入れ子になった構造のない、同期のように見える非同期コードの記述に役立つことです。

上記のコードをasync / awaitを使用して書き換えます。

async function test () {
    try {
        const users = await axios.get('/users')
        
        const addresses = await axios.get('/addresses')
    } catch (error) {
        console.log(error)
    }
}

test()

ここで、いくつかの点に注意を払う価値があります。

  • キーワードはawait常にキーワードと組み合わせて使用​​されasyncます。
  • 非同期関数を使用する場合、try / catch構文を使用して、エラーをインターセプトして処理します。
  • 本質awaitは、この構造は対応する約束で結果が返されることを期待することです。したがって、夢中になりすぎるとawait、アプリケーションの速度が低下する可能性があります。

async / awaitを使用すると、コールバックやプロミスとは異なり、try / catch構文を使用してエラーを処理できるという事実に注意を向けたいと思います。この構造は、非同期/待機エラーに加えて、対応するコードブロックで発生する他のエラーにも応答します。

TypeScriptを使用したコード品質の向上


最初はCでプログラミングしました。次にJavaでプログラミングしました。これらは、データ型に非常に厳密に関連している言語です。プログラマーは、使用する型（文字列、ブール値など）の明確で完全な説明、またはオブジェクトのフィールドにアクセスするための仕様（パブリック、プライベート、保護）を用意する必要があります。そして、これは本当に私を退屈させました。オブジェクトまたはその変数の型について、オブジェクトのプロパティを参照できるかどうかを常に考えなければなりませんでした。コードは、データ型に関連するエラーメッセージを表示するためにのみ実行できます。

JavaScriptに切り替えた後（PHPとPythonでも同じことが言えます）、すべてが大幅に簡素化されました。特定の変数に書き込まれる値のタイプは関係ありません。宣言するだけで十分であり、不必要に問題なく使用できます。

let x = 1

const test = 'This is a test'

const arr = [1, 2, 3, 4, 5]

これはJavaScriptの機能の1つであり、私はこの言語に一目惚れしました。言語の構文には不要な「偏見」が読み込まれていませんでした。コードはより正確であることがわかり、厳密に型指定された言語で記述されたコードよりも見た目がきれいでした。しかし、人生は私のJavaScriptの認識を調整しました。徐々に、多くのプログラマーがプロジェクトで作業している場合、またはプログラムを作成した後しばらくしてから読み取る必要がある場合、コードは私が書いたときに見たほどには明確で明確ではないことに気づき始めました。実際、変数を宣言する時点では、どのような値の値を取るかは不明であり、関数コードからどのタイプのデータが返されるかを理解するのは簡単ではありません...

const var2 = db.column2
const var3 = db.column3
const var4 = db.column4

このコードフラグメントを理解するにはどうすればよいですか？に頼らなければならないconsole.log：

console.log(var3) // -> number

楽しい経験ではありません。さらに、これには多くの時間がかかりました。コードは理解しにくいものであることが判明し、コードの難しさは、私にとっても、私の後にそれを扱う人々にとっても生じました。コードを理解するために常に使用する必要があるのは正常console.logですか？

それからTypeScriptが助けになりました。

私はTypeScriptをJavaScriptの改良版として認識しています。 TypeScriptのおかげで、明確に定義された型（文字列、ブール値、数値など）、アクセス修飾子（パブリック、プライベート）などがJSコードに表示されます。 TypeScriptで記述されたコードは、通常のJavaScriptコードが実行されるのと同じ場所で実行できるJavaScriptコードにコンパイルされます。このコードを実行するために必要な特別な環境はありません。

次に例を示します。

class Student {
    fullName: string;
    constructor(public firstName: string, public middleInitial: string, public lastName: string) {
        this.fullName = firstName + " " + middleInitial + " " + lastName;
    }
}

interface Person {
    firstName: string;
    lastName: string;
}

function greeter(person: Person) {
    return "Hello, " + person.firstName + " " + person.lastName;
}

let user = new Student("Jane", "M.", "User");

TypeScriptについて2年前に学びましたが、それが好きではなかったので、使いたくありませんでした。JavaScriptがもたらす自由にとても慣れています。 TypeScriptを試してみたいと思うこともありましたが、TSコードを常に操作し、複雑で乱雑な構造を常に確認していると、目を壊す危険があるように思えました。

しかし、少し前に、TypeScriptに切り替えて、すでに説明したJavaScriptの問題をすべて解決しました。さらに、TypeScriptへの動きはJavaScriptプログラマーのコミュニティ全体で観察されています。そして、プログラマーは通常TypeScriptに切り替えた結果に満足しています。

TypeScriptは現在、JavaScript開発環境で非常に人気があります。ライブラリとフレームワーク（Angular、React、Vue）の作成者は、プロジェクトがTypeScriptの使用をサポートしていることを確認します。たとえば、誰も知らない場合、Vue 3フレームワークはTypeScriptで完全に書き直されます。さらに、TypeScriptはMicrosoftによって開発されました。これにより、このプロジェクトの高品質が保証されます。TypeScriptを使用すると、優れたレベルのサポートを期待できます。

CI / CDの使用：プログラミング、テスト、デプロイメント


▍自動化テスト


プロジェクトで作業している場合は、遅かれ早かれエラーがプロジェクトに表示され始めます。それらを開発し、同時にそれらを改善するためにプロジェクトに取り組むときが最善です。コードの改善には、常に作業時間の約20％を費やすように努めるべきです。コードを改善するときに問題から身を守る唯一の方法は、テストを作成することです。少なくとも、テストは、改善中にコードに入るエラーができるだけ少ないことを確認するのに役立ちます。

ちなみに、テストは、コードを記述する前に記述して、それらを使用してチェックすることができます（これは、JavaScript開発だけではありません）。

▍CI/ CD-プロジェクトの継続的なテストと展開


CI / CD（継続的インテグレーション/継続的デリバリー）は、今日のトレンドです。これは、現代のプログラマーがコードを記述、テスト、およびデプロイするのに役立ちます。これは自動的かつ継続的に行われます。


リポジトリに表示されるコードに関する情報

実際、ほとんどすべてのCI / CDツールは、GitHub、GitLab、BitBucket、およびその他の同様のサービスと統合するように設計されています。その結果、これらすべての設定は非常に簡単です。そして、すべてが構成された後、コードがリポジトリーに送信されると、対応する自動化メカニズムが起動します。
