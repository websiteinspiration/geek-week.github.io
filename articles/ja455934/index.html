MS SQL Serverの新しいバージョンのバックアップデータを古いバージョンに転送する

バックグラウンド


かつて、バグを再現するために、生産拠点のバックアップが必要でした。

驚いたことに、私は次の制限に遭遇しました。

  1. データベースのバックアップはSQL Server 2016のバージョンで行われ、SQL Server 2014との互換性がありませんでした。
  2. 私の仕事用コンピューターでは、Windows 7が OSとして使用されていたため、SQL Serverをバージョン2016にアップグレードできませんでした
  3. サポートされている製品は、強力なレガシーアーキテクチャを備えた大規模なシステムの一部であり、他の製品やベースにも対応しているため、別のステーションに展開するのに非常に長い時間がかかる可能性があります。

上記を踏まえ、私は非標準的なソリューションの松葉杖の時が来ているという結論に達しました。

バックアップからのデータ復旧


Windows 10でOracle VM VirtualBox を使用することにしました（ここから Edgeブラウザーのテストイメージを取得できます）。SQL Server 2016が仮想マシンにインストールされ、アプリケーションデータベースがバックアップから復元されました（手順）。

仮想マシンでSQL Serverへのアクセスを構成する


次に、外部からSQL Serverにアクセスできるようにするために、いくつかの手順を実行する必要がありました。

  1. ファイアウォールの場合、ポート1433で要求をスキップするルールを追加します。
  2. サーバーへのアクセスは、Windows認証ではなく、ユーザー名とパスワードを使用してSQL経由で行うことが望ましい（アクセスを構成する方が簡単です）。ただし、この場合は、SQL ServerのプロパティでSQL認証を有効にすることを忘れないでください。
  3. SQL Serverのユーザー設定の[ユーザーマッピング ]タブで、復元したデータベースのユーザーロールdb_securityadminを指定します。

データ転送


実際、データ転送自体は2つの段階で構成されています。

  1. データスキーマの転送（テーブル、ビュー、ストアドプロシージャなど）
  2. データ自体を転送する

データスキーマの移行


次の操作を実行します。

  1. ポータブルデータベースの[ タスク]-> [スクリプトの生成]を選択します。
  2. 転送に必要なオブジェクトを選択するか、デフォルト値のままにします（この場合、データベース内のすべてのオブジェクトに対してスクリプトが作成されます）。
  3. スクリプトを保存するための設定を指定します。Unicodeでエンコードされた単一のファイルにスクリプトを保存するのが最も便利です。その後、障害が発生した場合、すべての手順を再度繰り返す必要はありません。

スクリプトを保存した後、元のSQL Server（古いバージョン）でスクリプトを実行して、必要なデータベースを作成できます。

注意：スクリプトを実行した後、バックアップからのデータベース設定と、スクリプトによって作成されたデータベースのコンプライアンスを確認する必要があります。私の場合、スクリプトにCOLLATEの設定がなかったため、データを転送し、タンバリンとダンスして拡張スクリプトを使用してデータベースを再作成すると、クラッシュが発生しました。

データ転送


データを転送する前に、以下に基づいてすべての制限のチェックを無効にする必要があります。

EXEC sp_msforeachtable 'ALTER TABLE ? NOCHECK CONSTRAINT all'

結論


このタスクは非常にまれであり、上記の制限のためにのみ発生します。最も一般的なソリューションは、SQL Serverをアップグレードするか、アプリケーションアーキテクチャで許可されている場合はリモートサーバーに接続することです。ただし、レガシーコードや低品質の開発の手に負えない人から誰も安全ではありません。この説明が不要であることを願っていますが、それでも必要な場合は、多くの時間と神経を節約するのに役立ちます。ご清聴ありがとうございました！
