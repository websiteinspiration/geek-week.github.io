オンザフライでmacOSのファイルをバックアップするための操作フックの使用

こんにちは、ハブル！私の名前はデニスコピリンです。今日は、macOSでのオンデマンドバックアップの問題をどのように解決したかについてお話ししたいと思います。実際、研究所で遭遇した興味深い仕事は、最終的にファイルシステムの操作に関する大規模な研究プロジェクトに発展しました。詳細はすべてカットの下にあります。



私は遠くから始めるつもりはありません。すべてはモスクワ物理学研究所のプロジェクトで始まりました。それは、私がアクロニスの基地部門で上司と開発しました。私たちは、リモートファイルストレージを整理する、つまりバックアップの現在のステータスを維持するという課題に直面しました。

データの安全性を確保するために、システム内のイベントに関する情報を収集するmacOSカーネル拡張を使用します。開発者向けのKPIにはKAUTH APIがあり、ファイルの開閉に関する通知を受け取ることができます-それだけです。 KAUTHを使用する場合、開発者はファイルへの書き込みイベントを利用できないため、ファイルを開いて書き込みを行うときに、ファイルを完全に保存する必要があります。そのような情報は私たちの仕事には不十分でした。実際、データのバックアップコピーを永続的に補足するには、ユーザー（またはマルウェア:)が新しいデータをファイルに書き込んだ場所を正確に理解する必要があります。



しかし、OSの制限に怯えている開発者は誰ですか？カーネルAPIで書き込み操作に関する情報を取得できない場合は、他のカーネルツールを介して独自の方法でインターセプトする必要があります。

最初は、コアとその構造にパッチを適用したくありませんでした。代わりに、通過するすべての読み取りおよび書き込み要求をインターセプトできるようにする仮想ボリューム全体を作成しようとしました。しかし、macOSの1つの不愉快な機能が判明しました。オペレーティングシステムには、1つではなく2つのUSBフラッシュドライブ、2つのディスクなどがあると信じています。そして、最初のボリュームで作業するときに2番目のボリュームが変更されるという事実から、macOSはドライブで正しく機能しなくなります。この方法には多くの問題があり、私はそれを放棄しなければなりませんでした。

別の解決策を見つける


KAUTHの制限にもかかわらず、このKPIを使用すると、すべての操作の前に、記録用のファイルの使用に関する通知を受け取ることができます。開発者には、コアBSDファイル抽象化であるvnodeへのアクセスが許可されます。奇妙なことに、vnodeへのパッチ適用はボリュームフィルタリングを使用するよりも簡単であることがわかりました。 vnode構造には、実際のファイルを処理する関数のテーブルがあります。したがって、このテーブルを置き換えるというアイデアがありました。



アイデアはすぐに良いアイデアと見なされましたが、Appleはその場所をどこにも文書化していないため、その実装にはvnode構造でテーブル自体を見つける必要がありました。これを行うには、カーネルのマシンコードを調べ、このアドレスに書き込んでシステムがその後停止しないようにすることができるかどうかを把握する必要がありました。

テーブルが見つかったら、メモリにコピーし、ポインタを置き換えて、新しいテーブルへのリンクを既存のvnodeに貼り付けます。これにより、ファイルを使用するすべての操作がドライバーを介して行われ、読み取りと書き込みを含むすべてのユーザー要求を登録できます。したがって、大切なテーブルの検索が私たちの主な目標となっています。

Appleがこれを本当に望んでいない場合、問題を解決するには、フィールドの相対的な場所のヒューリスティックを使用してテーブルの場所を「推測」するか、既知の関数を取得して逆アセンブルし、この情報からのオフセットを探す必要があります。

オフセットを探す方法：

簡単な方法vnodeでテーブルのオフセットを見つける最も簡単な方法は、構造内のフィールドの場所に基づくヒューリスティックです（Githubへのリンク））

struct vnode
{
  ...
  int (**v_op)(void *); /* vnode operations vector */
  mount_t v_mount; /* ptr to vfs we are in */
  ...
}

必要なv_opフィールドはv_mountからちょうど8バイトが削除されているという前提を使用します。後者の値は、パブリックKPI（Githubへのリンク）を使用して取得できます。


mount_t vnode_mount(vnode_t vp);

v_mountの値がわかると、「干し草の山の中の針」を探し始めます。vnode 'vp'へのポインターの値はuintptr_t *として認識され、vnode_mount（vp）の値はuintptr_tとして認識されます。その後、条件「haystack [i] == needle」が満たされるまで、iの「妥当な」値への反復が続きます。そして、フィールドの位置に関する仮定が正しい場合、オフセットv_opはi-1です。

void* getVOPPtr(vnode_t vp)
{
  auto haystack = (uintptr_t*) vp;
  auto needle = (uintptr_t) vnode_mount(vp);
  for (int i = 0; i < ATTEMPTCOUNT; i++)
  {
    if (haystack[i] == needle)
    {
      return haystack + (i - 1);
    }
  }
  return nullptr;
}

オフセットを探す方法：分解

単純ですが、最初の方法には大きな欠点があります。Appleがvnode構造内のフィールドの順序を変更すると、単純なメソッドが機能しなくなります。より普遍的ですが、それほど簡単ではない方法は、動的にカーネルを分解することです。

たとえば、macOS 10.14.6 の逆アセンブルされたカーネル関数VNOP_CREATE（Githubへのリンク）を考えてみます。私たちにとって興味深い指示には、矢印->が付いています。

アセンブラー命令をスキャンして、vnode dvpのシフトを見つけます。アセンブラコードの「目的」は、v_opテーブルから関数を呼び出すことです。これを行うには、プロセッサは次の手順に従う必要があります。

  1. 登録するdvpをアップロード
  2. v_opを取得するために逆参照する（23行目）
  3. vnop_create_desc.vdesc_offsetを取得します（24行目）
  4. 関数を呼び出す（26行目）

手順2〜4ですべてが明らかな場合、最初の手順で問題が発生します。どのレジスタdvpがロードされたかを理解するにはどうすればよいですか？これを行うには、目的のポインターの動きを監視する関数をエミュレートする方法を使用しました。 System V x86_64呼び出し規約によれば、最初の引数はrdiレジスタで渡されます。したがって、rdiを含むすべてのレジスターを追跡することにしました。私の例では、これらはrbxおよびrdiレジスタです。また、レジスタのコピーをスタックに保存できます。スタックはカーネルのデバッグバージョンにあります。

rbxレジスタとrdiレジスタがdvpを格納していることがわかっているので、23行目でvnodeを逆参照してv_opを取得していることがわかります。したがって、構造内の変位は0xd0であるという仮定が得られます。正しい決定を確認するために、スキャンを続け、関数が正しく呼び出されることを確認します（24行目と26行目）。

この方法の方が安全ですが、残念ながら欠点もあります。関数のパターン（つまり、上記で説明した4つのステップ）が同じであることを当てにする必要があります。ただし、関数パターンを変更する確率は、フィールドの順序を変更する確率よりも桁違いです。そこで、2番目の方法で停止することにしました。

テーブルのポインタを置き換えます


v_opを見つけた後、問題が発生します。このポインターの使用方法は？2つの異なる方法があります。テーブルの関数を上書きする（図の3番目の矢印）か、vnodeのテーブルを上書きする（図の2番目の矢印）。

最初は、1つのポインタを置き換えるだけなので、最初のオプションの方が収益性が高いようです。ただし、このアプローチには2つの重大な欠点があります。まず、v_opテーブルは、特定のファイルシステムのすべてのvnode（HFS +の場合はv_op、APFSの場合はv_opなど）で同じであるため、vnodeによるフィルタリングが必要になり、非常に負荷が高くなる可能性があります。次に、テーブルは読み取り専用ページに書き込まれます。この制限は、システムチェックをバイパスしてIOMappedWrite64を介した記録を使用する場合に回避できます。また、ファイルシステムドライバー付きのkextが同梱されている場合、パッチの削除方法を理解するのは困難です。

2番目のオプションは、より対象を絞って安全であることがわかります。インターセプターは必要なvnodeに対してのみ呼び出され、vnodeメモリーは最初に読み取り/書き込み操作を許可します。テーブル全体が置き換えられるため、メモリをもう少し割り当てる必要があります（1つではなく80個の関数）。また、テーブルの数は通常ファイルシステムの数と同じであるため、メモリ制限は完全に無視できます。

kextが2番目の方法を使用するのはそのためです。ただし、繰り返しますが、一見すると、このオプションの方が悪いようです。



その結果、ドライバーは次のように動作します。

  1. KAUTH APIはvnodeを提供します
  2. vnodeテーブルを置き換えます。必要に応じて、ユーザードキュメントなどの「興味深い」vノードの操作のみをインターセプトします
  3. 傍受するとき、記録しているプロセスを確認し、「私たち」を除外します
  4. 同期のUserSpaceリクエストをクライアントに送信し、クライアントは正確に何を保存する必要があるかを決定します。

どうした


今日、私たちは実験的なモジュールを持っています。これはmacOSカーネルの拡張であり、細分レベルでのファイルシステムへの変更を考慮に入れています。macOS 10.15で、Apple がファイルシステムへの変更の通知を受信するための新しいフレームワーク（EndpointSecurityへのリンク）を導入したことは注目に値します。アクティブ保護での使用が予定されているため、記事で説明されているソリューションは非推奨と宣言されました。
