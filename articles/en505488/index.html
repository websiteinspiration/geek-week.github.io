12 tips for implementing TypeScript in React applications

TypeScript is the standard of modern front-end development. According to State of JavaScript research , TS is clearly of interest to programmers. According to the survey for 2019, almost 60% of respondents tried TS and continue to use it, 22% did not try and want to study.

This article is a collection of tips on how to implement and improve the use of TypeScript. The first half of the tips is general regarding approaches and infrastructure. The second is a few particularly useful language features.



I stick with the context of a React application, but some tips will be relevant in other ecosystems.

Tip 1: error tolerance


Correct types are trust in the written code and facilitation of refactoring. In order for typing to work for you, you need to strictly apply to type checks:

  1. Make a check for dev mode to see errors right away.
  2. Add validation to CI for each pool request. This will ensure that the main branch is always the correct types.

There are two main ways to compile ts files: the original TSC and @ babel / preset-typescript. The babel method is interesting: type information is cut from .ts files, which makes the assembly quick. Here is an introduction article.

Type checking in the case of babel is added additionally using fork-ts-checker-webpack-plugin. In development mode, checks should be done asynchronously so as not to block the assembly and not interfere with writing code. In pool requests, type checking, on the contrary, should stop the build if an error is found.

Tip 2: backend contracts


Communication with the backend via the API itself implies a data exchange contract. It is useful to separate these entities into a separate layer, creating separate interfaces. We call them DTO - Data Transfer Object.

To implement such a layer, some agreements are needed. Here is an example of the rules that you can implement in a team:

  • The interface name has the suffix Dto: FooBarDto.
  • Located in a special folder.
  • It can only change with an API change.
  • The data of this interface is used for reading only, editing is prohibited.

Advantages of this approach: the data source is immediately clear on the interface, it is easier to look for errors and debug code, verify data. These types are easy to document, providing descriptions and examples - fellow developers will carry you on hand.

If you use tools such as Swagger to describe the API , see how you can create these types automatically.

Tip 3: aliases on any


You and your team write superbly typed code - no problem, no error, and right on time. If so, go to the next point;)

In reality, there are temptations to omit typing, especially when implementing TS in an existing project. I want to explicitly or implicitly use any. There are adequate reasons: complex code, urgent task, inexpediency - efforts do not pay for the result.

Instead of ignoring the problem, try legalizing it:

export type TODO_ANY = any;

Pros:

  • You do not cheat and do not write crutches.
  • You consciously show that in the future the code is worth finalizing.
  • This type is easy to find by searching for a project.
  • Any can be used explicitly where the type does not matter, for example like this:

const getArrayLength = (array: any[]) => array.length

Limitations:

  • Too frequent use of this technique is unlikely to be beneficial.
  • Consciousness and responsibility are needed to remove such gags in the future.

You may also find it useful: 

export type TODO_UNKNOWN = unknown;

In the case of any, the compiler will allow almost any action. In the case of unknown , on the contrary, almost any use will lead to an error. They set TODO_UNKNOWN until the type of the object is clear ‚Üí in the future they used the object ‚Üí got an error ‚Üí fixed the type ‚Üí you are amazing.

Tip 4: types for libraries


If you are working with the popular js library, look for types in DefinitelyTyped. Most likely, they are already written - we connect and enjoy life.

But what if you have an internal library in which there are definitely no types? Try to generate them automatically. Good news for those using jsDoc: in TypeScript 3.7, the tsc compiler has learned how to generate .d.ts types from both .ts and .js files. 

Another good news: there are js -> ts converters. For example, TypeStat helped me deal with typical React components.

The bad news is that a good type system is not likely to work out automatically. Count on the "skeleton", which subsequently will need to build up the "meat".

Tip 5: types on top of types


... and types chase!

It happens that the types of external dependencies of the project are not suitable - they are too general (and you would like to be more specific), they contain an error, the version does not fit - or the types are simply missing.

Typically, such problems occur in internal libraries (which are also frequently updated), while in external ones it is quite rare. If a problem occurs, try manually defining / redefining types.

We describe the type for the example module. I emphasize that only the type is redefined; the implementation does not change:

declare module "example" {
    const foo: { bar: string };
    export default foo;
}

It is better to move such an override to a separate directory and file, for example typings / example.d.ts . Connect all the contents from typings to the compilerOptions.typeRoots field in tsconfig:

{
    "compilerOptions": {
        "typeRoots": ["node_modules/@types", "./typings"]
    }
}

Voila - types for example are taken from redefinition!

Tip 6: styleguide


Type or interface? const or enum? How to describe types in a functional component? And in the memoized?

When TS is just being introduced into a team, such questions arise constantly. A similar set of questions appears for beginners in the project: the storm lasts several weeks, then the usual coding style is settled.

Formalizing rules is a great way to save time on disputes and holivars in pool requests. Gather a team and create your style guide.

  • Take popular style guides as a basis. I like these: react-typescript-cheatsheet , react-redux-typescript-guide , modern-typescript-with-examples-cheat-sheet.
  • Work linter and formatter - let the machine do the dirty work.
  • Add the specifics of your project: typical templates, best practices, worst practices.
  • Commit these decisions and refer to them in a code review.
  • Provide the team with the opportunity to discuss and change these decisions.

A well-built style guide reveals the wonders of CPDD ( Copy Paste Driven Development ): developers write in the same style, spend less time thanks to ready-made boilerplates. Epic battles tabs vs spaces are not arranged (sorry, types vs interfaces), entry of beginners into the project is simplified.

PS: Another source of inspiration is the ts utility library. You will find not only useful helpers, but also the code from the pros in the source. For example, utility-types , SimplyTyped .

Tip 8: literal vs enum type


TypeScript introduces a convenient syntax structure - enum . Enum is used for final enumerations. Having mastered enum, at first I want to write it very often. However, enum has fair criticism regarding the generated code after compilation.

On the other hand, there are literal types - a simpler and more convenient alternative to the final enumeration. In some cases, the use of such types is more appropriate.

A life example is simple component props:

//   enum
enum ColorEnum {
    yellow = "yellow",
    transparent = "transparent"
}

const ExampleEnum = (props: { color: ColorEnum }) => <div>{props.color}</div>;

//    
type ColorLiteral = "yellow" | "transparent";

const ExampleLiteral = (props: { color: ColorLiteral }) => <div>{props.color}</div>;

//   
const Example = () => (<>
        <ExampleEnum color={ColorEnum.yellow} />
        <ExampleLiteral color="yellow" />
 </>);

In this example, both components check the type of props color. The variant with enum turned out to be more redundant - this is additional code during compilation, and additional imports in the source code, and a more cumbersome writing structure. The variant with a string literal is more concise, while also containing a type check. It makes no sense to complicate the enum code.

Prefer string literals for simple type checks. Use enum to describe more complex logic. In this case, it is better to give preference to const enum (however, this structure is not supported by babel , you will have to add a plugin ).

Tip 9: const assertions


Another reason to use enum less often is const assertions.

as const tells the compiler that this object will not change. Therefore, types can be deduced in more detail.

const loadAction = {
    start: "LOAD_START",
    done: "LOADED",
    error: "LOAD_ERROR"
} as const;

type LoadActionType = typeof loadAction;
type LoadActionKey = keyof LoadActionType; // 'start' | 'done' | 'error'
type LoadActionValue = LoadActionType[LoadActionKey]; // 'LOAD_START' | 'LOADED' | 'LOAD_ERROR'
//  "as const" LoadActionValue   string

You can also apply as const to arrays:

const DIRECTION_ASC = 1;
const DIRECTION_DESC = -1;
const sortDirectionTypes = [DIRECTION_ASC, DIRECTION_DESC] as const;
type SortDirection = typeof sortDirectionTypes[number]; // -> 1 | -1

It is convenient to use for transfers. For example, check that the dictionary describes all the options:

const sortDirectionDescription: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " ",
    [DIRECTION_DESC]: " "
};
const sortDirectionDescription2: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " "
};
// Error: TS2741: Property '[-1]' is missing // in type '{ [DIRECTION_ASC]: string; }'

Tip 10: branded types


Brand is a small utility for creating unique (‚Äúbranded‚Äù) types. Typically, these types are created on the basis of primitive - strings, numbers.

For example, when working with data id is constantly used. Sometimes it is extremely important to distinguish one type of id from another (for example, the user id from the id of his order in the store), although both are strings. With Brand, you can just get two different types:

type Brand<T, U> = T & { __brand: U };

type UserId = Brand<string, "userId">; // "5435436" ‚Äì  id 
type OrderId = Brand<string, "orderId">; // "42da-fet3-cdsl" ‚Äì  id 

declare const userID = "1111" as UserId; // - 
declare function getOrderById(id: OrderId): any; // - 

getOrderById(userID);
// Error: TS2345
// Type '"userId"' is not assignable to type '"orderId"'.

Tip 11: discriminated union types


The complex term discriminated union is easy to understand in practice: it creates a union of several types with a common part (discriminant). When working with such a union, the ts compiler can filter out options and specify the type.

The classic example of using discriminated union fits nicely on the reducer template. Thanks to the common type field, the type for payload is automatically displayed:

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {
        case 'SET_ONE':
            return { ...state, one: action.payload}; // `payload` is string
        case 'SET_TWO':
            return { ...state, two: action.payload}; // `payload` is number
        default:
            return state;
    }
}

Tip 12: never


Never is good insurance against raw cases. Take from the previous paragraph, only without 'SET_TWO', and with the addition of typeguard: never;

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {

        case 'SET_ONE':
            return { ...state,
                        one: action.payload}; // `payload` is string

        default:
            const typeguard: never = action;
            return state;
            // Error: TS2322: Type '{ type: "SET_TWO"; payload: number; }'
            // is not assignable to type 'never'.
    }
}

The compiler infers the type for typeguard and comes to a contradiction. On the one hand - the type of which is not, never. On the other hand, the output type is 'SET_TWO'. Types do not converge, we get an error.

Another interesting (but rare) case of using never in React is to prohibit nesting or passing props:

type IconProps = {
    size: 's' | 'm';
    children?: never;
    foo?: never;
};

const Icon: React.FC<IconProps> = () => <div>Whatewer</div>;
    
const Example = () => (<>
    <Icon size="s"> <div /> </Icon> {/*  ‚Äì  children */}
    <Icon size="s" foo="123"/> {/*  -  foo */} 
</>);
