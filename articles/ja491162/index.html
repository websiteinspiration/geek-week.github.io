PostgreSQLアンチパターン：名前による検索の反復的改良、または「前後の最適化」に関する物語

全国の営業所の何千人ものマネージャーが、毎日何万人もの連絡先をCRMシステムに 記録しています。これは、潜在的な顧客とのコミュニケーションの事実、または顧客との取引実績です。そして、このクライアントの場合、最初に、できれば非常に迅速に見つける必要があります。そしてこれは最も頻繁に名前で起こります。 したがって、最も負荷の高いデータベースの1つである「自社のVLSIアカウント」で「重い」リクエストをもう一度分析すると、会社のカードを名前で「すばやく」検索するリクエストが「一番上」にあることがわかりました。 さらに、さらなる調査により、最適化の興味深い例と、パフォーマンスの低下が明らかになりました



いくつかのチームによるその後の改良で要求し、各チームは善意からのみ行動しました。

抽象開発者は何をしますか？

1.0：外部検索エンジン


ああ、検索は難しいです。私は何もしたくないのです。専門家に任せましょう！データベースに関連する外部検索エンジンをデプロイしてもらいます。Sphinx、ElasticSearchなど...

同期と変更の速度に関して、時間はかかりますが機能するオプションです。しかし、私たちの場合ではありません。検索は、クライアントごとにアカウントデータのフレームワーク内でのみ実行されるためです。また、データの変動性はかなり高く、マネージャーがカードを挿入した場合、' '5〜10秒後に、メールをそこに記載するのを忘れたことを覚えており、それを見つけて修正したいと考えています。

したがって、「データベースで直接検索」してみましょう。幸いなことに、PostgreSQLでは1つのオプションではなく、これを行うことができます。

1.1：「正直な」部分文字列


「サブストリング」という言葉にしがみついています。しかし、部分文字列（および正規表現でも）によるインデックス検索にぴったりです。優れたモジュールpg_trgmがあります。その場合にのみ、正しくソートする必要があります。

モデルを単純化するために、プレートを試してみましょう。

CREATE TABLE firms(
  id
    serial
      PRIMARY KEY
, name
    text
);

実際の組織とインデックスの780万件のレコードを記入します。

CREATE EXTENSION pg_trgm;
CREATE INDEX ON firms USING gin(lower(name) gin_trgm_ops);

行間検索の最初の10エントリを見てみましょう。

SELECT
  *
FROM
  firms
WHERE
  lower(name) ~ ('(^|\s)' || '')
ORDER BY
  lower(name) ~ ('^' || '') DESC --  " "
, lower(name) --   
LIMIT 10;

ええと、26ミリ秒、31MBの読み取りデータ、および1.7Kを超えるフィルタリング済みレコード-10人。オーバーヘッドが高すぎますが、どういうわけか効率を上げることは可能ですか？

1.2：テキスト検索？FTSです！


実際、PostgreSQLは、プレフィックス検索の可能性を含む、全文検索（全文検索）のための非常に強力なメカニズムを提供します。素晴らしいオプションです。拡張機能でさえインストールする必要はありません！やってみよう：

CREATE INDEX ON firms USING gin(to_tsvector('simple'::regconfig, lower(name)));

SELECT
  *
FROM
  firms
WHERE
  to_tsvector('simple'::regconfig, lower(name)) @@ to_tsquery('simple', ':*')
ORDER BY
  lower(name) ~ ('^' || '') DESC
, lower(name)
LIMIT 10;


[explain.tensor.ruを見てください]

ここでは、クエリ実行の並列化が少し役立ち、時間を半分に減らして11ミリ秒になりました。はい、読み取る必要が1.5倍少なくなりました-わずか20MB。そして、ここでは、小さいほど良いので、差し引く量が多いほど、キャッシュミスが発生する可能性が高くなり、ディスクから読み取られたすべての余分なデータページが、要求の「ブレーキ」になる可能性があります。

1.3：まだ好きですか？


前のリクエストはすべての人に適していますが、1日に10万回プルした場合にのみ、2TBの読み取りデータが実行されます。せいぜい-メモリからですが、運が悪ければ、ディスクからです。だから、もっと小さくしてみましょう。

ユーザーが最初に "...で始まる"を見たいことを思い出してください。だから、これはその純粋な形である前方一致検索でtext_pattern_ops！そして、必要なレコードが10個まで「不十分」である場合にのみ、FTS検索を使用してそれらを読み取る必要があります。

CREATE INDEX ON firms(lower(name) text_pattern_ops);

SELECT
  *
FROM
  firms
WHERE
  lower(name) LIKE ('' || '%')
LIMIT 10;

優れたパフォーマンス-わずか0.05msで100KBを少し超える読み取り！ユーザーが結果で迷子にならないように、名前で並べ替えるのを忘れていました。

1.4：「ファイルで変更」


しかし、範囲で検索できるインデックスがあり、ソートを使用するのが普通です- 通常のbtree！

CREATE INDEX ON firms(lower(name));

「手動で組み立てる」必要があるのは、そのリクエストだけです。

SELECT
  *
FROM
  firms
WHERE
  lower(name) >= '' AND
  lower(name) <= ('' || chr(65535)) --  UTF8,   - chr(255)
ORDER BY
   lower(name)
LIMIT 10;


[explain.tensor.ruをご覧ください]

すばらしい-選別作業とリソース消費はどちらも「微視的」であり、「純粋な」FTSより数千倍効率的です！単一の要求で収集する必要があります。

(
  SELECT
    *
  FROM
    firms
  WHERE
    lower(name) >= '' AND
    lower(name) <= ('' || chr(65535)) --  UTF8,    - chr(255)
  ORDER BY
     lower(name)
  LIMIT 10
)
UNION ALL
(
  SELECT
    *
  FROM
    firms
  WHERE
    to_tsvector('simple'::regconfig, lower(name)) @@ to_tsquery('simple', ':*') AND
    lower(name) NOT LIKE ('' || '%') -- " "    
  ORDER BY
    lower(name) ~ ('^' || '') DESC --    ,     btree-
  , lower(name)
  LIMIT 10
)
LIMIT 10;

2番目のサブクエリが実行されるのは、最初のサブクエリが最後のサブクエリが予期するLIMIT行数よりも少ない場合のみです。このクエリの最適化方法については、すでに書いています。

つまり、テーブルにbtreeとginが同時に存在するようになりましたが、統計的には、要求の10％未満が2番目のブロックに到達することがわかりました。つまり、このようなよく知られた典型的なタスクの制限により、サーバーリソースの総消費量を約1000倍削減することができました。

1.5 *：ファイルを省略


上記LIKEでは、間違った種類を使用することはできませんでした。しかし、USINGステートメントを指定することで、「真のパスに設定」できます。
デフォルトは暗黙的ASCです。さらに、特定のソート演算子の名前を文で指定できますUSING。ソート演算子は、Bツリー演算子の特定のファミリーよりも小さいまたは大きいメンバーである必要があります。ASC通常は同等でUSING <、DESC通常は同等USING >です。
私たちの場合、「少ない」とは~<~：

SELECT
  *
FROM
  firms
WHERE
  lower(name) LIKE ('' || '%')
ORDER BY
  lower(name) USING ~<~
LIMIT 10;

2：リクエストを「サワー」する方法


ここで、半年間または1年間「要求する」という要求を残しましたが、驚いたことに、5.5 TBのメモリ（バッファー共有ヒット）の1日の総「ポンピング」のインジケーターを備えた「トップ」に再び見つかりました。 いいえ、もちろん、私たちのビジネスは成長し、負荷は増加しましたが、それほどではありません！だからここで何かが汚い-それを理解しましょう。



2.1：ページングの誕生


ある時点で、別の開発チームは、同じであるが拡張された結果を使用して、簡単な添え字検索からレジストリに「ジャンプ」できるようにしたいと考えました。また、ページナビゲーションのないレジストリは何ですか？ねじ込みましょう！

( ... LIMIT <N> + 10)
UNION ALL
( ... LIMIT <N> + 10)
LIMIT 10 OFFSET <N>;

これで、開発者が「タイプページ」をロードして検索結果のレジストリを表示するのに負担をかけることなく可能になりました。

もちろん、実際には、データの次のページごとにどんどん読み取られています（以前はすべて破棄し、必要な「尾」も加えます）。つまり、これは明確なアンチパターンです。そして、インターフェースに格納されているキーから次の反復で検索を開始する方がより正確ですが、それについて-別の時に。

2.2：エキゾチックにしたい


ある時点で、開発者は別のテーブルからのデータを使用して結果の選択を多様化することを望みました。その目的のために、前のリクエスト全体がCTEに送信されました。

WITH q AS (
  ...
  LIMIT <N> + 10
)
SELECT
  *
, (SELECT ...) sub_query -- -    
FROM
  q
LIMIT 10 OFFSET <N>;

そして、そうであっても-悪くはありません、サブクエリは返された10レコードに対してのみ計算されるためです...

2.3：DISTINCTは無意味で無慈悲


そのような進化の過程のどこかで、2番目のサブクエリから条件が失わNOT LIKEれました。その後、いくつかのレコードを2回UNION ALL返し始めました-最初に行の先頭で見つかり、次にもう一度-この行の最初の単語の先頭で。制限では、2番目のサブクエリのすべてのレコードが最初のサブクエリのレコードと一致する可能性があります。 理由を検索する代わりに、開発者は何をしますか？..間違いなく！



  • 元のサンプルのサイズの2倍
  • 各行の単一のインスタンスのみを取得するようにDISTINCTを配置します

WITH q AS (
  ( ... LIMIT <2 * N> + 10)
  UNION ALL
  ( ... LIMIT <2 * N> + 10)
  LIMIT <2 * N> + 10
)
SELECT DISTINCT
  *
, (SELECT ...) sub_query
FROM
  q
LIMIT 10 OFFSET <N>;

つまり、結果は最終的にまったく同じであることは明らかですが、2番目のCTEサブクエリに「飛ぶ」可能性がはるかに高くなり、これがなければ、さらに多くのことが明確に読み取られます。

しかし、これは悲しいことではありません。DISTINCT特定ではなく、すぐにレコードのすべてのフィールドを選択するように開発者から依頼されたため、サブクエリの結果であるsub_queryフィールドも自動的に選択されます。ここで、実行するためDISTINCTに、データベースは10個のサブクエリを実行する必要はなく、すべて<2 * N> + 10を実行する必要がありました。

2.4：何よりも協力！


したがって、開発者は住んでいました-レジストリでは次の各「ページ」の受信が慢性的に遅くなり、重要なN値に「パッチ」されたため、開発者は気になりませんでした。

別の部門の開発者がやって来て、反復検索にそのような便利な方法を使用したくなくなるまで、つまり、サンプルから1つを取り出し、追加の条件でフィルタリングし、結果を描画してから、次の部分（この場合は、 N）を増やし、画面いっぱいになるまで続けます。

一般に、捕獲された標本では、Nはほぼ17000の値に達し、わずか24時間で少なくとも4Kのそのようなリクエストが「連鎖的に」実行されました。それらの最後はすでに大胆にスキャンされています各反復で1GBのメモリ ...
