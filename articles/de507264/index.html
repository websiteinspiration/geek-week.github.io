Wörterbuchkombination


Eine der neuen und bereits meine Lieblingsfunktionen mit Syntax. Wenn wir zwei Wörterbücher haben aund bdiese kombinieren müssen, verwenden wir jetzt die Gewerkschaftsoperatoren.

Wir haben den Zusammenführungsoperator " |":

a = {1: 'a', 2: 'b', 3: 'c'}
b = {4: 'd', 5: 'e'}
c = a | b
print(c)

[Out]: {1: 'a', 2: 'b', 3: 'c', 4: 'd', 5: 'e'}

Und die Update-Anweisung " |=", die das Quellwörterbuch aktualisiert:

a = {1: 'a', 2: 'b', 3: 'c'}
b = {4: 'd', 5: 'e'}
a |= b
print(a)

[Out]: {1: 'a', 2: 'b', 3: 'c', 4: 'd', 5: 'e'}

Wenn unsere Wörterbücher einen gemeinsamen Schlüssel haben , wird ein Schlüssel-Wert-Paar aus dem zweiten Wörterbuch verwendet:

a = {1: 'a', 2: 'b', 3: 'c', 6: 'in both'}
b = {4: 'd', 5: 'e', 6: 'but different'}
print(a | b)

[Out]: {1: 'a', 2: 'b', 3: 'c', 6: 'but different', 4: 'd', 5: 'e'}

Wörterbuchaktualisierung mit Iterationen


Ein weiteres interessantes Verhalten des Operators " |=" ist die Möglichkeit, das Wörterbuch mithilfe eines iterativen Objekts - beispielsweise einer Liste oder eines Generators - mit neuen Schlüssel-Wert-Paaren zu aktualisieren:

a = {'a': 'one', 'b': 'two'}
b = ((i, i**2) for i in range(3))
a |= b
print(a)

[Out]: {'a': 'one', 'b': 'two', 0: 0, 1: 1, 2: 4}

Wenn wir versuchen, dasselbe mit dem Standard-Union-Operator " |" zu wiederholen, erhalten wir TypeError, da nur Verknüpfungen zwischen Typen zulässig sind dict.

Neuer Parser


Dies ist eher eine versteckte Änderung, aber möglicherweise eine der wichtigsten Änderungen für die zukünftige Entwicklung von Python.

Python verwendet derzeit eine Grammatik, die hauptsächlich auf LL (1) basiert. Diese kann wiederum vom LL (1) -Parser analysiert werden, der den Code von oben nach unten, von links nach rechts analysiert und nur ein Token anzeigen kann.

Ich habe fast keine Ahnung, wie das funktioniert, aber ich kann Ihnen über einige tatsächliche Probleme in Python aufgrund der Verwendung dieser Methode berichten:

  • Python enthält Nicht-LL (1) -Grammatik; Aus diesem Grund verwenden einige Teile der aktuellen Grammatik Problemumgehungen, wodurch unnötige Komplexität entsteht.
  • LL(1) Python ( ). , ( SyntaxError):

      open("a_really_long_bar") as bar):

Alle diese Faktoren (und viele andere, die ich einfach nicht verstehen kann) haben einen großen Einfluss auf Python. Sie begrenzen die Entwicklung der Sprache.

Der neue PEG- basierte Parser bietet Python-Entwicklern viel mehr Flexibilität - und wir werden dies ab Python 3.10 bemerken.
