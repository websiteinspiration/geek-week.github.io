Streamlining the Check Point API with the Python SDK

All the power of interacting with the API is revealed when used in conjunction with program code, when it becomes possible to dynamically generate API requests and tools for analyzing API responses. However, the Python Software Development Kit (hereinafter referred to as the Python SDK) for the Check Point Management API , but in vain , remains invisible . It significantly simplifies the life of developers and automation lovers. Python has gained immense popularity lately and I decided to bridge the gap and give an overview of the main features of the Check Point API Python Development Kit . This article is an excellent complement to another article on the Habr√© Check Point R80.10 API. Management through CLI, scripts and more. We‚Äôll look at how to write scripts using the Python SDK and dwell on the new Management API functionality in version 1.6 (supported since R80.40). To understand the article, you will need basic knowledge of working with APIs and Python.

Check Point is actively developing the API and at the moment there are:

  • Check Point Management API (current version 1.6) - work with the management server through the API (and the ability to execute scripts on gateways under the control of the management server)
  • Check Point GAIA API (current version 1.4) - work with security gateways
  • Threat Prevention API 1.0 - Sandboxing in Check Point Cloud
  • Identity Awareness API - working with the Identity Awareness blade on gateways
  • Security Management Portal API - working with the SMB gateway management portal ( Learn more about SMB gateways )
  • IoT API - interaction with IoT controllers
  • CloudGuard Connect API - working with CloudGuard Connect (SD-WAN security solution)
  • Dome9 API - working with Dome9

The Python SDK currently only supports interaction with the Management API and the Gaia API . We will cover the most important classes, methods, and variables in this module.



Module installation


The cpapi module installs quickly and easily from the official Check Point repository on github using pip . Detailed installation instructions are in README.md . This module is adapted to work with versions of Python 2.7 and 3.7. In this article, examples will be given using Python 3.7. However, the Python SDK can be run directly from the Check Point (Smart Management) management server, but they only support Python 2.7, so the last section will provide code for version 2.7. Immediately after installing the module, I recommend looking at the examples in the examples_python2 and examples_python3 directories .

Beginning of work


In order for us to be able to work with the components of the cpapi module , at least two necessary classes must be imported from the cpapi module :

APIClient and APIClientArgs

from cpapi import APIClient, APIClientArgs

The APIClientArgs class is responsible for connecting to the API server, and the APIClient class is responsible for interacting with the API.

We determine the connection parameters


To define various parameters for connecting to the API, you need to create an instance of the APIClientArgs class . In principle, its parameters are predefined and you can omit them when running the script on the management server.

client_args = APIClientArgs()

But when starting on a third-party host, you must specify at least the IP address or host name of the API server (it is also a management server). In the example below, we define the server connection parameter and assign it the IP address of the management server as a string.

client_args = APIClientArgs(server='192.168.47.241')

Let's look at all the parameters and their default values ‚Äã‚Äãthat can be used when connecting to the API server:

Arguments to the __init__ method of the APIClientArgs class 
class APIClientArgs:
    """
    This class provides arguments for APIClient configuration.
    All the arguments are configured with their default values.
    """

    # port is set to None by default, but it gets replaced with 443 if not specified
    # context possible values - web_api (default) or gaia_api
    def __init__(self, port=None, fingerprint=None, sid=None, server="127.0.0.1", http_debug_level=0,
                 api_calls=None, debug_file="", proxy_host=None, proxy_port=8080,
                 api_version=None, unsafe=False, unsafe_auto_accept=False, context="web_api"):
        self.port = port
        # management server fingerprint
        self.fingerprint = fingerprint
        # session-id.
        self.sid = sid
        # management server name or IP-address
        self.server = server
        # debug level
        self.http_debug_level = http_debug_level
        # an array with all the api calls (for debug purposes)
        self.api_calls = api_calls if api_calls else []
        # name of debug file. If left empty, debug data will not be saved to disk.
        self.debug_file = debug_file
        # HTTP proxy server address (without "http://")
        self.proxy_host = proxy_host
        # HTTP proxy port
        self.proxy_port = proxy_port
        # Management server's API version
        self.api_version = api_version
        # Indicates that the client should not check the server's certificate
        self.unsafe = unsafe
        # Indicates that the client should automatically accept and save the server's certificate
        self.unsafe_auto_accept = unsafe_auto_accept
        # The context of using the client - defaults to web_api
        self.context = context


I believe that the arguments that can be used in instances of the APIClientArgs class are intuitive to Check Point administrators and do not need additional comments.

Connect via APIClient and context manager


The APIClient class is most conveniently used through the context manager. All you need to pass to an instance of the APIClient class is the connection parameters that were defined in the last step.

with APIClient(client_args) as client:

The context manager will not automatically make a login call to the API server, but it will make a logout call when it exits. If for some reason a logout is not required at the end of work with the API calls, you need to start work without using the context manager:

client = APIClient(clieng_args)


Connection check


The easiest way to check if a connection passes through the given parameters is by using the check_fingerprint method . If the verification of the sha1 hash sum for fingerprint of the server API certificate did not pass (the method returned False ), then this is usually caused by connection problems and we can stop the program (or allow the user to fix the connection data):

    if client.check_fingerprint() is False:
        print("Could not get the server's fingerprint - Check connectivity with the server.")
        exit(1)

Note that in the future, the APIClient class will check with each API call ( api_call and api_query methods , we will talk about them a little further) sha1 fingerprint certificate on the API server. But if an error is detected when checking sha1 fingerprint of the server API certificate (the certificate is unknown or has been changed), the check_fingerprint method will provide the ability to add / change information about it on the local machine in automatic mode. This check can be turned off altogether (but this can only be recommended if scripts are run on the API server itself, when connected to 127.0.0.1) using the APIClientArgs argument - unsafe_auto_accept (see details on APIClientArgs earlier in ‚ÄúDefining Connection Parameters‚Äù).

client_args = APIClientArgs(unsafe_auto_accept=True)

Login to the API server


In APIClient have as much as 3 login method on the API server, and each of them zaponimaet value sid (the session-id), which is used automatically in each subsequent API calls in the header (the name of the title in this setting - the X-CHKP-sid ), so that you do not need to further process this parameter.

Login method


The option using the username and password (in the example, the username admin and the password 1q2w3e are passed as positional arguments):

     login = client.login('admin', '1q2w3e')

The optional optional parameters are also available in the login method. I list their names and default values: 

continue_last_session=False, domain=None, read_only=False, payload=None

Login_with_api_key method


A variant using the api key (supported since management version R80.40 / Management API v1.6, ‚Äú3TsbPJ8ZKjaJGvFyoFqHFA ==‚Äù is the value of the API key for one of the users on the management server with the API key authorization method):

     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')

In the login_with_api_key method , the same optional parameters are available as in the login method .

Login_as_root method


The login option to the local machine with the API server:

     login = client.login_as_root()

Only two optional parameters are available for this method:

domain=None, payload=None

api_query


Immediately make a reservation that this method is applicable only to calls whose output involves offset (shift). This conclusion occurs when it contains or may contain a large amount of information. For example, this may be a request for a list of all created objects of the host type on the management server. For such requests, the API returns a list of 50 objects by default (you can increase the limit to 500 objects in the response). And in order not to pull the information several times, changing the offset parameter in the API request, there is an api_query method that does this work automatically. Examples of calls where this method is needed: show-sessions, show-hosts, show-networks, show-wildcards, show-groups, show-address-ranges, show-simple-gateways, show-simple-clusters, show-access- roles, show-trusted-clients, show-packages. Processing API Call Results


After that, you can use the variables and methods of the APIResponse class (both inside the context manager and outside). The APIResponse class has 4 methods and 5 variables predefined; we will dwell on the most important ones in more detail.



success


To begin with, it would be nice to make sure that the API call was successful and returned the result. There is a success method for this :

In [49]: api_versions.success                                                   
Out[49]: True

Returns True if the API call was successful (response code is 200) and False if not successful (any other response code). It is convenient to use immediately after the API call, so that depending on the response code, you can display different information.

if api_ver.success: 
    print(api_versions.data) 
else: 
    print(api_versions.err_message)

statuscode


Returns the response code after making an API call.

In [62]: api_versions.status_code                                               
Out[62]: 400

Possible response codes: 200,400,401,403,404,409,500,501 .

set_success_status


There may be a need to change the value of the success status. Technically, you can put anything there, even an ordinary string. But a real example would be to reset this parameter to False under certain concomitant conditions. Below, pay attention to an example when there are tasks performed on the management server, but we will consider this request to be unsuccessful (set the success variable to False , despite the fact that the API call was successful and returned code 200).

for task in task_result.data["tasks"]:
    if task["status"] == "failed" or task["status"] == "partially succeeded":
        task_result.set_success_status(False)
        break

Useful examples


The following are examples that use API calls that have been added to Management API 1.6.

To get started, consider the operation of add-host and add-address-range calls . Suppose we need to create all the ip addresses of the subnet 192.168.0.0/24 as objects of the host type, the last octet of which is 5, and write all the other ip addresses as address range objects. In this case, the subnet address and broadcast address should be excluded.

So, below is a script that solves this problem and creates 50 objects of the host type and 51 objects of the type range of addresses. To solve the problem, 101 API calls are required (not counting the final publish call). Also, using the timeit module, we calculate the time it takes to execute the script until the changes are posted.

Script using add-host and add-address-range 
import timeit
from cpapi import APIClient, APIClientArgs

start = timeit.default_timer()

first_ip = 1
last_ip = 4

client_args = APIClientArgs(server="192.168.47.240")

with APIClient(client_args) as client: 
     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')
     for ip in range(5,255,5):
         add_host = client.api_call("add-host", {"name" : f"h_192.168.0.{ip}", "ip-address": f'192.168.0.{ip}'})
     while last_ip < 255:
         add_range = client.api_call("add-address-range", {"name": f"r_192.168.0.{first_ip}-{last_ip}", "ip-address-first": f"192.168.0.{first_ip}", "ip-address-last": f"192.168.0.{last_ip}"})
         first_ip+=5
         last_ip+=5
     stop = timeit.default_timer() 
     publish = client.api_call("publish")
     
print(f'Time to execute batch request: {stop - start} seconds')


In my lab environment, it takes 30 to 50 seconds to execute this script, depending on the load on the management server.

Now let's see how to solve the same problem using the add-objects-batch call API , which support has been added to API version 1.6. This call allows you to create many objects at once in one API request. Moreover, these can be objects of different types (for example, hosts, subnets, and address ranges). Thus, our task can be solved in the framework of a single API call.

Script using add-objects-batch 
import timeit
from cpapi import APIClient, APIClientArgs

start = timeit.default_timer()

client_args = APIClientArgs(server="192.168.47.240")

objects_list_ip = []
objects_list_range = []

for ip in range(5,255,5):
    data = {"name": f'h_192.168.0.{ip}', "ip-address": f'192.168.0.{ip}'}
    objects_list_ip.append(data)
    
first_ip = 1
last_ip = 4


while last_ip < 255:
    data = {"name": f"r_192.168.0.{first_ip}-{last_ip}", "ip-address-first": f"192.168.0.{first_ip}", "ip-address-last": f"192.168.0.{last_ip}"}
    objects_list_range.append(data)
    first_ip+=5
    last_ip+=5

data_for_batch = {
  "objects" : [ {
    "type" : "host",
    "list" : objects_list_ip
}, {
    "type" : "address-range",
    "list" : objects_list_range
  }]
}


with APIClient(client_args) as client: 
     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')
     add_objects_batch = client.api_call("add-objects-batch", data_for_batch)
     stop = timeit.default_timer() 
     publish = client.api_call("publish")
     
print(f'Time to execute batch request: {stop - start} seconds')


And it takes 3 to 7 seconds to execute this script in my lab environment, depending on the load on the management server. That is, on average, on a 101 API object, a batch type call fulfills 10 times faster. On more objects, the difference will be even more impressive.

Now let's see how to work with set-objects-batch . Using this API call, we can massively change any parameter. Let's set the first half of the addresses from the previous example (to .124 hosts, and ranges too) the color sienna, and the second half of the addresses we will assign the color khaki.

Change the color of objects created in the previous example 
from cpapi import APIClient, APIClientArgs

    data = {"name": f'h_192.168.0.{ip}', "color": "khaki"}

data_for_batch_first  = {
  "objects" : [ {
    "type" : "host",
    "list" : objects_list_ip_first
}, {

Running Python scripts on a Check Point management server


Everything in the same README.md contains information on how to run Python scripts directly from the management server. This can be convenient when you cannot connect to the API server from another machine. I recorded a six-minute video in which I consider installing the cpapi module and the features of running Python scripts on the management server. As an example, a script is run that automates the configuration of a new gateway for a task such as auditing the Security CheckUp network . Of the features that I had to face: in Python 2.7, the input function has not yet appeared , so the raw_input function is used to process the information that the user enters. Otherwise, the code is the same as for starting from other machines, only it is more convenient to use the login_as_root function so as not to specify your own username, password and IP address of the management server again.


Script for quick setup Security CheckUp 
from __future__ import print_function

sys.path.append(os.path.abspath(os.path.join(os.path.dirname(__file__), '..')))
from cpapi import APIClient, APIClientArgs

def main():
    with APIClient() as client:
       # if client.check_fingerprint() is False:
       #     print("Could not get the server's fingerprint - Check connectivity with the server.")
       #     exit(1)
        login_res = client.login_as_root()

        if login_res.success is False:
            print("Login failed:\n{}".format(login_res.error_message))
            exit(1)

        gw_name = raw_input("Enter the gateway name:")
        gw_ip = raw_input("Enter the gateway IP address:")
        if sys.stdin.isatty():
            sic = getpass.getpass("Enter one-time password for the gateway(SIC): ")
        else:
            print("Attention! Your password will be shown on the screen!")
            sic = raw_input("Enter one-time password for the gateway(SIC): ")
        version = raw_input("Enter the gateway version(like RXX.YY):")
        add_gw = client.api_call("add-simple-gateway", {'name' : gw_name, 'ipv4-address' : gw_ip, 'one-time-password' : sic, 'version': version.capitalize(), 'application-control' : 'true', 'url-filtering' : 'true', 'ips' : 'true', 'anti-bot' : 'true', 'anti-virus' : 'true', 'threat-emulation' : 'true'})
        if add_gw.success and add_gw.data['sic-state'] != "communicating":
            print("Secure connection with the gateway hasn't established!")
            exit(1)
        elif add_gw.success:
            print("The gateway was added successfully.")
            gw_uid = add_gw.data['uid']
            gw_name = add_gw.data['name']
        else:
            print("Failed to add the gateway - {}".format(add_gw.error_message))
            exit(1)

        change_policy = client.api_call("set-access-layer", {"name" : "Network", "applications-and-url-filtering": "true", "content-awareness": "true"})
        if change_policy.success:
            print("The policy has been changed successfully")
        else:
            print("Failed to change the policy- {}".format(change_policy.error_message))
        change_rule = client.api_call("set-access-rule", {"name" : "Cleanup rule", "layer" : "Network", "action": "Accept", "track": {"type": "Detailed Log", "accounting": "true"}})
        if change_rule.success:
            print("The cleanup rule has been changed successfully")
        else:
            print("Failed to change the cleanup rule- {}".format(change_rule.error_message))
