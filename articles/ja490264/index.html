理論が言うこと


例として、テストピラミッドを使用した自動化アプローチを検討してください。ピラミッドのレベルとして、私たちが使用するテストのタイプを取り上げます。つまり、単体テスト、統合テスト、およびエンドツーエンド（E2E）テストです。


単体テスト（単体テスト）は、システムの個々の小さな部品（機能、メソッド）の動作を検証します。それらは小さく、迅速に通過し、大きな開発および保守コストを必要としません。

統合テストでは、外部コンポーネントを使用してアプリケーションの動作をテストします。たとえば、データベースやさまざまなサービスを使用します。テストの作成とサポート、および実行には、さらに時間がかかります。このようなテストでは、ほとんどの場合、何らかの環境を構成してサポートする必要があります。この場合、それらはしばしばより壊れやすくなります。環境が壊れやすい。

エンドツーエンドのテストでは、システム全体の動作を検証し、エンドユーザーのアクションをエミュレートします。ほとんどの場合、UIを介してシステムにアクセスします。ユーザーとしてボタンを押したり、フィールドに入力したりします。ただし、インターフェースのないシステムでも使用できます。これらのテストは、上記の検討よりもはるかに高価です。テスト自体と環境全体の両方を開発してサポートするためにより多くの時間が必要です。さらに、システムのあらゆる部分での変更の影響を受け、エミュレータまたはブラウザの予期しない動作が特定のテストの結果に影響を与える可能性があるため、非常に脆弱です。このようなテストの実行時間は、他のテストよりもかなり長くなります。個々のユーザーアクションを実行するには、さらに多くのアクションを実行する必要があります。たとえば、ログインするか、目的のメニュー項目に移動します。

ピラミッドの原理によれば、テストが簡単、高速、安価であるほど、テストの合計数に対する割合が高くなります。それら。スルーテストでは、1つのフィールドへの異なる値の割り当てを確認しないでください（これらの場合にUIの操作を確認する必要がない限り）。

これらすべてのタイプのテストについて、役割が割り当てられました。アプリケーションのライフサイクルのある時点で、1つまたは別のテスト自動化が必要になります。製品は単体テストのみで簡単に実行でき、エンドツーエンドのテストのニーズにすぐに対応できます。自動化を計画する際の主なことは、製品の現在のメリットを正確に理解し、時間を浪費する可能性があることを理解することです。

自動化を実装するときは、手動テストを忘れないでください。大量の自動化があっても、常に手動でチェックする必要があるものがあります。少なくとも、すぐにテストを書くことが合理的でない新機能。

しかし、私の意見では、すべての自動化は主に手動テストの量を減らすことを目的とすべきです。また、手動テストを計画するときは、どのケースがすでに自動化されているかを検討する価値があります。

自動テストはどのように編成されますか？


各製品には個別にユニットテストがあり、各PRで実行されます。それらは開発者の責任です。



統合テストでは、WebサービスとEDMSの相互作用を検証します。

これらは、クライアントアプリケーションの動作をシミュレートします。彼らは、EDSデータを取得および変更するためのWebサービスメソッドを使用します。

サーバー側の新しいバージョンをビルドするたびに実行します。



エンドツーエンドのテストは、エンドユーザーのパフォーマンスをシミュレートします。彼らはモバイルアプリケーションのインターフェイスのボタンを押し、アプリケーションはWebサービスを通じてEDMSと連携します。

私たちは長い間手動でモバイルアプリケーションをテストしてきました。これはほとんど機能のないアプリケーションでしたが、うまく対処できました。開発されたものはすべて1つの領域で±であり、手動テストは、アプリケーションのすべての機能を迅速かつ簡単にテストできるように編成できました。

Xamarin.UITestについて


これは、Xamarin.iOSおよびXamarin.Androidでのプロジェクトの自動UIテストのフレームワークです（Objective-C / SwiftおよびJavaのサポートも発表されています）。テストはNUnitを使用してC＃で記述されています。コンテンツプロジェクトは簡単に統合できます。

フレームワークの操作の原則は2つのコンポーネントに基づいています。要素の検索（クエリ）とそれを使用したアクション（アクション）の実行（スワイプの押下や実行など）。

たとえば、次は無効なログインを入力したときにエラー表示をチェックするテストです。

public void ShowErrIncorrectLoginOrPassword_IfLoginIsWrong()
  {
    var wrongLogin = TestsSettings.UserLogin + "1";
    app.EnterLoginAndPassword(wrongLogin, TestsSettings.UserPassword);
    app.WaitForElement(Resources.Identifiers.ErrorMessage, "Login is incorrect, alert message wasn't shown.", TestsSettings.Delay);
    Assert.AreEqual(CoreResources.ErrIncorrectLoginOrPassword, ErrorMessage);
  }


private string ErrorMessage => 
app.Query(x => x.Marked(Resources.Identifiers.ErrorMessage)).First().Text;

その中で使用される資格情報の入力方法：

public static void EnterLoginAndPassword(this AndroidApp app, string login, string password)
    {
      app.WaitForElement(Resources.Identifiers.LoginEdit, TestsSettings.Delay);
      app.EnterText(Resources.Identifiers.LoginEdit, login);
      app.EnterText(Resources.Identifiers.PasswordEdit, password);
      app.Tap(Resources.Identifiers.LoginButton);
    }

この例では、標準のフレームワークメソッドが使用されます-要素を待つ、テキストを入力する、要素をクリックする。

テストを開発およびデバッグするときは、組み込みのコンソールユーティリティREPL（read-eval-print-loop）を使用すると便利です。これにより、画面上にある要素のツリーを表示したり、標準のフレームワークメソッドを実行したりできます。

私たちにとって予期しないこと


インターフェイス要素に対処しても、期待した結果にならない場合がありました。

たとえば、アプリケーションの新しいウィンドウが同じアクティビティのフラグメントで開かれた場合、ユーザーには画面上にこの新しいウィンドウのみが表示されます。ただし、この場合、表示されている要素のツリーには、ユーザーが目で確認できない要素があります。このような「見えない」要素の検索は成功し、それを使ってアクションを実行できます。それは、ユーザーに表示されるウィンドウでクリックが実行されるだけです。それら。実際、それは誤検知になります。

ツリーには複数の同じ要素が含まれる場合があります。デフォルトでは、アクションはツリーの最初の適切な要素で実行されます。また、テストでラベルを介して要素にアクセスし、同じラベルの要素がどこかにある場合、テストは低下し始めます（または、必要な要素が最初の場合はそうではありません）。

筆記およびデバッグされたテストが最初の戦闘開始時に落ちた1つのケースがありました。テストが異なる画面サイズのデバイスで実行されたため。そして、このデバイスでクリックされた要素は、別のインターフェイス要素の下に表示されました。



送信トレイフォルダは[タスクの作成]ボタンの下にあり、この場合はフォルダをクリックするとボタンがクリックされました。これはユーザーにとって馴染みのある状況であり、画面をスクロールするだけです。しかし、要素のツリーでは、このオーバーレイは表示されません。

このような問題は小さな不都合を引き起こし、いくつかのテストをより注意深く検討するようになります。また、テストに干渉しないように、データをデータベースに合わせる場合もあります。結局のところ、この例の目的はフォルダーを開くことであり、その開きを妨げるものがないかどうかを確認することではありません。

別の驚きと失望は、Visual Studio for WindowsからiOSテストを実行およびデバッグできないことでした。この機能はまだ実装されていません。MacOS用のスタジオで働かなければなりません。
