欠点


私たちはこれまでずっと多くのKubernetesクラスターでloghouseを使用してきましたが、一般に、このソリューションは私たち自身と、アクセスを提供するさまざまなクライアントの両方に適しています。

その主な利点は、シンプルで直感的なインターフェイス、SQLクエリを実行できること、データベースにデータを挿入するときの圧縮率が高く、リソース消費量が少ないこと、およびストレージ中のオーバーヘッドリソースが少ないことです。運転中のログハウスで

ログハウス0.3.0の主な変更点


実際、十分な変更を蓄積しましたが、最も重要な変更を強調します。それらは3つの主要なグループに分けることができます：

  • ログストレージとデータベーススキーマの改善。
  • ログ収集の改善。
  • 監視の出現。

1.ログストレージとデータベース設計の改善


主な革新：

  • ログのストレージスキームが変更され、単一のテーブルで動作するように移行され、パーティションテーブルの拒否が完了しました。
  • 最新バージョンのClickHouseに組み込まれている基本のクリーニングメカニズムが適用され始めています。
  • これで、クラスターモードでも外部ClickHouseインストールを使用できます。

実際のプロジェクトで新旧の回路 のパフォーマンスを比較します。以下は、人気のあるオンラインリソースdtf.ruのアプリケーションログで一意のURLを検索する例です。

SELECT 
    string_fields.values[indexOf(string_fields.names, 'path')] AS path, 
    count(*) AS count
FROM logs
WHERE (namespace = 'kube-nginx-ingress') AND ((string_fields.values[indexOf(string_fields.names, 'vhost')]) LIKE '%foobar.baz%') AND (date = '2020-02-29')
GROUP BY string_fields.values[indexOf(string_fields.names, 'path')]
ORDER BY count DESC
LIMIT 20

選択は数千万件を超えるレコードで行われます。古いスキームは20秒で機能しました。

さらに、ClickHouseテーブルにTTLの使用を開始しました。これにより、指定した時間間隔より古いデータをデータベースから自動的に削除できます。

ログ収集の改善


主な革新：

  • 多数のログが表示されたときにバーストをスムーズにするように設計されたバッファーが追加されました。
  • アプリケーションから直接ログハウスにログを送信する機能を実装：TCPおよびUDPを介して、JSON形式で。

ログハウスのログハウスバッテリーはlogs_buffer、データベーススキーマに追加された新しいテーブルです。このテーブルはメモリ内にあります。RAMに格納されます（特殊なタイプのBufferがあります）。ベースの負荷を滑らかにするのは彼女です。追加のヒントをありがとう。ソビゴッド！

実装されたログをアプリケーションからログハウスに直接送信すると、netcatを介してもこれを実行できます。

echo '{"log": {"level": "info", "msg": "hello world"}}' | nc fluentd.loghouse 5170

これらのログは、ログハウスがストリームにインストールされている名前空間に表示することができますnet：



送信するデータの要件は最小限です：メッセージは、フィールドで有効なJSONでなければなりませんlog。次に、フィールドlogは文字列またはネストされたJSONのいずれかになります。

ロギングサブシステムの監視


重要な改善点は、Prometheusによるfluentdの監視でした。ログハウスにはGrafanaのパネルが付属し、次のようなすべての基本的なメトリックが表示されます。

  • 流暢な労働者の数;
  • ClickHouseに送信されたイベントの数。
  • 空きバッファサイズ（パーセント）。

Grafanaのパネルコードは、ドキュメントに記載されています。

ClickHouseのパネルは、既成の製品（f1yegorからのもの）に基づいて作成されています。

ご覧のように、パネルにはClickHouseへの接続数、バッファーの使用、バックグラウンドタスクのアクティビティ、およびマージの数が表示されます。これはシステムの状態を理解するには十分



です。fluentdのパネルには、fluentdのアクティブなインスタンスが表示されます。これは、ログをまったく望まない、またはまったく失うことができない人にとって特に重要です。



パネルには、ポッドのステータスに加えて、ログをClickHouseに送信するためのキューの負荷が表示されます。次に、ClickHouseが負荷を処理しているかどうかを理解できます。ログが失われない場合は、このパラメーターも重要になります。

パネルの例は、Prometheus Operatorの配信用にシャープになっていますが、設定の変数を使用して簡単に変更できます。
