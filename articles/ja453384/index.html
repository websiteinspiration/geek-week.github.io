セキュリティクリブ：REST



RESTは非常に人気のあるWebアプリケーションアーキテクチャです。サーバー上の関数を呼び出すには、指定されたパラメーターを持つ通常のHTTPリクエストが使用され（通常、パラメーターの構造化にはJSONまたはXMLが使用されます）、RESTアーキテクチャには厳密な標準がないため、柔軟性が追加されます（もちろん、多少の混乱が生じます）。

RESTを使用すると、セキュリティの問題に柔軟に取り組むことができます。また、多くの罪よりも、問題にまったく取り組む必要がありません。OWASPに基づいて、RESTアプリケーションのセキュリティを向上させるのに役立つヒントのリストを用意しました。

ここで必要とされるまれなケースの出発点として、PythonとDjangoを使用します。

ルール0


Https


設定するだけです。どういたしまして。送信されたデータの保護は誰にも害を与えていません。現時点では何も保護する必要がないと考えていても、常にそうであるとは限らず、HTTPSを構成する必要があります。だから、すぐにそれをよりよく構成し、誰もがうまくいくでしょう。

ルール1


認証


それは明白なアドバイスのように思われますが、多くの人は無視することを好みます。社内でのみ使用すると考え、どの従業員が認証を行っても違いがない場合でも、アプリケーションには常に認証が必要です。さらに、雑誌を追加すると、インシデントの調査と活動の分析が比類なく簡単になります。

現時点ではアクセストークンとして、JWTを使用するのが適切な形式と見なされています（JSON Webトークン）。整合性制御のために値{"alg"： "none"}のトークンを使用しないでください。トークンには常に署名またはMACが含まれている必要があります。MAC生成キーと検証キーが同じ（または互いに簡単に計算できる）ので、署名が望ましいです。つまり、サーバーがMAC値をチェックできれば、その値も生成できます。署名は、メッセージの整合性だけでなく、送信者の身元も確認します。

ルール2


使用しないHTTPメソッドを拒否する


使用するメソッド（GET、POST、PUTなど）のホワイトリストを使用してサーバーを構成し、このリストに当てはまらないものはすべて拒否します。サーバーが本番環境でTRACEリクエストを処理する必要がある可能性は低いです（この方法は、XSS攻撃とTRACEまたはTRACKメソッドで構成されるXST攻撃（クロスサイトトレース）の一部です。この攻撃により、たとえば、ユーザーのCookieを盗むことができますそれらはHttpOnlyとしてマークされます）。外部から利用できるインフラストラクチャに関する情報が少ないほど、優れています。

ルール3


アクセスの差別化


すべてのユーザーに、DELETEなどのすべてのメソッドが必要ですか？一部のユーザーが特定の操作を実行できないようにするには、アプリケーションでアクセス制御を構成します。たとえば、通常のユーザーは一部の機能ではGETメソッドのみにアクセスでき、マネージャーはGETやPOSTを使用できます。このため、便利なアクセス制御のためにユーザーに割り当てることができるロールをデータベースに設定することは価値があります。

その結果、各関数にはほぼこのタイプのチェックブロックがあります。

if request.POST and user.is_manager:
    do_stuff()

ルール4


クエリの数を制限することを検討してください


ユーザーが毎秒10万件のリクエストを送信してはならない場合は、この数を制限する必要があります。APIキーまたは他の便利なメカニズムを使用して、特定の期間に1人のユーザーから処理されるリクエストの数を追跡して制限します。たとえば、Djangoの場合、django-ratelimitはこの機能を提供します。この機能では、APIキーではなくIPアドレスではなく、制限の識別子としてさまざまなパラメーターを設定できます。

ルール6


必要以上の情報を提供しないでください


リクエストによってアプリケーションでエラーが発生した場合、またはなんらかの理由で単に利用できない場合は、応答に詳細を含めないで、最も抽象的なエラーメッセージを返します。一部のサーバーはエラー後にデフォルトでスタックトレースを返すため、このロジックを必ず再構成してください。

ルール7


常にログを保持する


各イベント（認証、エラー、要求など）をできるだけ詳細に記録します。必要に応じてそれらをより便利に検索できるように、それらを論理的に記録します。念のため、ログに記録する前に、記録されたデータをサニタイズしてください。

ルール8


Content-Typeを正しく示します-これは重要です！


ブラウザー（またはクライアント）が提供されたデータを正しく読み取るためには、提供されたデータに対応する正しく指定されたContent-Typeが重要です。ブラウザーの場合、ブラウザーが実際に送信されたもの以外のContent-Typeを検出しないようにするために、ヘッダーX-Content-Type-Options：nosniffを設定することも価値があります（XSS攻撃への対策）。

Cross-Origin Resource Sharing (CORS)


CORSは、クロスドメインクエリの操作を説明する標準です。アプリケーションがCORSをサポートしていない場合は、このタイプのヘッダーを無効にします。使用する必要がある場合は、アクセスポリシーをできるだけ具体的かつ厳密にする必要があります。

ルール11


URLのパラメーターを展開しない


すべての重要なデータ（パスワード、キー、トークン、ログイン、できればまた）は、リクエストの本文内またはヘッダー内にある必要がありますが、URLには表示されません。GETメソッドを通じて機密データを渡す必要がある場合は、それをヘッダーに入れます。

不可能：
example .com / controller / 123 / action？ApiKey = a53f435643de32

可能：
example .com / controller / 123 / action

ヘッダー：
ホスト：example.com
User-Agent：Mozilla ...
X-APIkey：a53f435643de32

ルール13


フレームワークを探索する


フレームワークを使用してアプリケーションにRESTを実装する場合は、よく行われているように、それを研究し、盲目的に使用しないでください。ドキュメント、特にセキュリティの部分をよくお読みください。デフォルト設定のままにしておいてください！特にセキュリティに関しては、各フレームワークに独自のニュアンスがあるため、それらを知ることは非常に重要です。
