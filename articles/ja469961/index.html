MicrosoftでのRust（またはSecurity Daemon Azure IoT Edgeの作成）



翻訳者からの注記：Rustは比較的新しいプログラミング言語であり、実用的な例はこの開発段階では常に興味深いものです。 Intel、Dropbox、Amazon、Facebook、GoogleなどがRustをプロジェクトで使用していることが知られています。 MicrosoftからのプロジェクトでのRustの使用については知られていましたが、詳細はありませんでした。おそらく、これはMicrosoftでのRustの使用に関する最初の公式出版物です。 （製品自体の技術的詳細は「現状のまま」提供されます。会社のWebサイトで詳細を確認してください）

Azure IoT EdgeMicrosoftのAzure IoT開発チームが作成したオープンソースのクロスプラットフォームソフトウェアプロジェクトです。 Azure IoT Edgeは、モノのインターネットセンターに基づくモノのインターネットサービスです。このサービスは、クラウドではなくエッジデバイスでデータを分析する必要があるお客様を対象としています。ワークロードの一部を境界領域に移動した結果、デバイスはクラウドにメッセージを送信する時間を短縮し、イベントにより迅速に応答します。」[公式のロシア語のドキュメントから取得（1）-約トランスレータ]

この投稿では、製品にセキュリティデーモンコンポーネントを実装するためのプログラミング言語としてRustを選択した理由のいくつかを説明しています。

セキュリティデーモンは、Azure IoT Edgeランタイムを読み込みます。これは、Azure IoT Edgeランタイムと、コンテナーランタイムやハードウェア暗号化ハードウェアセキュリティモジュール（HSM）やトラステッドプラットフォームモジュール（TPM）などの多くのホストサービスの間のメディエーターとして機能します。

セキュリティデーモンの技術スタックの選択


セキュリティデーモン（以下、エッジレットと呼びます）の作業を開始したとき、次の主要な設計目標を特定しました。
  • edgeletは、.NET CLRなどのランタイムを必要としないネイティブコンポーネントである必要があります。
  • エッジレットはデバイス上のHSM / TPM機器にアクセスするためのチャネルとして機能するため、保護する必要があります。
  • edgeletは、バイナリCアプリケーションインターフェイス（ABI）を介してHSM / TPM機器と通信するため、一般的なオブジェクト/ DLLのロードとC関数の呼び出しは簡単です。

これらのポイントによると、次のプログラミング言語のいずれかを選択する必要がありました。CまたはC ++、またはネイティブコードにコンパイルされるRust。ガベージコレクターのオーバーヘッドを負担したくないということは、囲碁が私たちには適していないことを意味しました。デーモンのセキュリティ要件により、メモリエラーと同時実行エラーを回避する必要がありました。結局のところ、これらの制限を考えると、Rustは私たちに最適です。このブログの以前の出版物（2）（「Microsoft Security Response Center」ブログ-翻訳コメントを意味する）では、プログラミング言語としてRustを選択することの利点が検討されました。

私たちは何が好きでした


Azure IoT Edgeを一般公開する前に、外部のセキュリティベンダーを招いてペネトレーションソフトウェアをテストしました。Rustのコードベースのセキュリティに問題が見つからなかったという事実は、私たちが選択した正当性の確認でした。最初から、コンパイラーはすべての警告を、clippyによって検証されたエラーを含むエラーとして処理することを決定しました。当社の継続的インテグレーションシステムは、rustfmtを通過しないリクエストのプールを拒否します。これにより、コードベース全体でコードの一貫したフォーマットが保証されます。

Rust and Cargoコンパイラの更新プロセス[アセンブリおよびパッケージ管理システム-約 翻訳者]Azure IoT Edgeでシームレスに動作します。コンパイラの新しいバージョンへのアップグレードは、ほとんどの場合、簡単な作業です。

さびの受け入れ


次のステップは言語を学ぶことでした。他の一般的なプログラミング言語と比較して、Rustにはかなり高いエントリしきい値があり、この研究がプロジェクトのタイムラインに与える影響を懸念しました。私たちのチームはC、C ++、C＃、Javaの開発者を経験しており、幸いにも、Rustに非常に熱心な開発者も何人かいました。

私たちは、Rustのワークショップを思い付きました。このワークショップでは、私たちの経験では、理解するのが非常に難しい言語の部分についてチームを作りました。これらの時間の消費は非常に役立つことが判明しました。言語の学習は、私たちが想像していたほど大きな問題ではないことがわかりました。

4〜6週間、チームのほぼすべてのメンバーがRustのソースコードに重要な貢献をしました。

困難


マイクロソフトでRustベースの製品を使用して製品を立ち上げる最初の経験が完全に成功したという事実にもかかわらず、いくつかの困難な点を指摘していました。
  • Rustエコシステムは、他の確立された言語と比較して、まだ比較的新しいものです。これは、それ以外の場合はおそらく作成する必要のないインフラストラクチャの一部を作成する必要があることを意味していました。
