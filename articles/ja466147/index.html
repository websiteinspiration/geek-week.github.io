反応データ表示マネージャー。前書き

これは、ReactiveDataDisplayManager（RDDM）ライブラリに関する一連の記事の最初の部分です。この記事では、「通常の」テーブルを操作するときに対処しなければならない一般的な問題と、RDDMについて説明します。




問題1. UITableViewDataSource


手始めに、責任の割り当て、再利用、その他のクールな言葉を忘れてください。テーブルを使った通常の作業を見てみましょう：

class ViewController: UIViewController {
    ...
}
extension ViewController: UITableViewDelegate {
   ...
}
extension ViewController: UITableViewDataSource {
    ...
}

最も一般的なオプションを分析します。何を実装する必要がありますか？正しくは、通常3つのメソッドが実装されていますUITableViewDataSource。

func tableView(_ tableView: UITableView, numberOfRowsInSection section: Int) -> Int
func numberOfSections(in tableView: UITableView) -> Int
func tableView(_ tableView: UITableView, cellForRowAt indexPath: IndexPath)

補助的な方法（numberOfSectionなど）に注意を払い、最も興味深い方法を検討するfunc tableView(tableView: UITableView, indexPath: IndexPath)

までは、製品の説明をセルにテーブルに入力すると、メソッドは次のようになります。

func tableView(_ tableView: UITableView, cellForRowAt indexPath: IndexPath) {
    let anyCell = tableView.dequeueReusableCell(withIdentifier: ProductCell.self, for: indexPath)
    guard let cell = anyCell as? ProductCell else {
        return UITableViewCell()
    }
 
    cell.configure(for: self.products[indexPath.row])
    return cell
}

すばらしいです。難しいことではありません。ここで、たとえば3つのセルタイプがあるとします。

  • 製品
  • 株式のリスト。
  • 広告。

簡単にするために、セルをメソッドに移動しgetCellます。

func tableView(_ tableView: UITableView, cellForRowAt indexPath: IndexPath) {
    switch indexPath.row {
    case 0:
        guard let cell: PromoCell = self.getCell() else {
            return UITableViewCell()
        }
        cell.configure(self.promo)
        return cell
    case 1:
        guard let cell: AdCell = self.getCell() else {
            return UITableViewCell()
        }
        cell.configure(self.ad)
        return cell
    default:
        guard let cell: AdCell = self.getCell() else {
            return UITableViewCell()
        }
         
        cell.configure(self.products[indexPath.row - 2])
        return cell
    }
}

どういうわけかたくさんのコード。設定画面を作りたいと想像してみてください。そこには何があるのでしょうか？

  • アバター付きのセルキャップ。
  • 「詳細」に遷移するセルのセット。
  • スイッチ付きのセル（たとえば、ピンコードによる入力の有効化/無効化）。
  • 情報を含むセル（たとえば、電話、電子メールなどがあるセル）;
  • 個人的なオファー。

また、順番が設定されています。大きな方法が判明します...

そして今、別の状況-入力フォームがあります。入力フォームでは、同一のセルの束。それぞれがデータモデルの特定のフィールドを担当します。たとえば、電話を入力するセルは電話などを担当します。
すべてがシンプルですが、「BUT」が1つあります。この場合でも、必要なフィールドを更新する必要があるため、さまざまなケースをペイントする必要があります。

6つの異なるタイプの入力フィールドを受け取り、フィールドの状態（可視性、入力タイプ、検証、デフォルト値など）に応じて、セルが大きく変化するように、バックエンドドリブンデザインの空想と想像を続けることができます。単一のインターフェースに導くことはできません。この場合、このメソッドは非常に不快に見えます。構成をさまざまな方法に分解したとしても。

ちなみに、その後、作業中にセルを追加/削除したい場合、コードがどのようになるか想像してみてください。ViewControllerデータに格納されているセルの一貫性とセルの数を個別に監視する必要があるため、見栄えがよくありません。

問題：

  • 異なるタイプのセルがある場合、コードはヌードルのようになります。
  • セルからのイベントの処理には多くの問題があります。
  • テーブルの状態を変更する必要がある場合の醜いコード。

問題2. MindSet


クールな言葉の時はまだ来ていません。
アプリケーションがどのように機能するか、または画面にデータがどのように表示されるかを見てみましょう。私たちは常にこのプロセスを順番に提示します。まあ、多かれ少なかれ：

  1. ネットワークからデータを取得します。
  2. 扱う;
  3. このデータを画面に表示します。

しかし、本当にそうでしょうか？番号！実際、これを行います：

  1. ネットワークからデータを取得します。
  2. 扱う;
  3. ViewControllerモデル内に保存します。
  4. 何かが画面の更新を引き起こします。
  5. 保存されたモデルはセルに変換されます。
  6. 画面にデータが表示されます。

数量に加えて、まだ違いがあります。まず、データを出力するのではなく、出力します。次に、データ処理プロセスに論理的なギャップがあり、モデルが保存され、プロセスはそこで終了します。その後、何かが起こり、別のプロセスが始まります。したがって、要素を画面に追加するのではなく、オンデマンドで要素を保存するだけです（ちなみにこれも問題です）。

そしてUITableViewDelegate、セルの高さを決定するためのメソッドが含まれていることを忘れないでください。通常は十分automaticDimensionですが、これでは不十分な場合があり、高さを自分で設定する必要があります（たとえば、アニメーションの場合やヘッダーの場合）。
次に、通常はセル設定を共有します。高さ構成のパーツは別の方法です。

問題：

  • データ処理とUIでの表示の間の明示的な接続は失われます。
  • セル構成はさまざまな部分に分かれています。

考え


複雑な画面にリストされた問題は、頭痛とお茶に行きたいという鋭い願望を引き起こします。

まず、デリゲートメソッドを常に実装する必要はありません。明白な解決策は、それを実装するオブジェクトを作成することです。次に、次のようにします。

let displayManager = DisplayManager(self.tableView)

いいよ これらのセルの構成を別の場所に移動する必要がある一方で、オブジェクトが任意のセルを操作できるようにする必要があります。

別のオブジェクトに構成を配置する場合は、構成を1つの場所にカプセル化します（スマートワードの時間です）。これと同じ場所で、データをフォーマットするためのロジック（たとえば、日付フォーマットの変更、文字列の連結など）を取り出すことができます。同じオブジェクトを通じて、セル内のイベントをサブスクライブできます。

この場合、2つの異なるインターフェースを持つオブジェクトがあります。

  1. インスタンス生成インターフェースUITableViewは、DisplayManager用です。
  2. 初期化、サブスクリプション、構成インターフェース-PresenterまたはViewController用。

このオブジェクトをジェネレーターと呼びます。それから、テーブルのジェネレーターはセルであり、それ以外のすべてのものです。UIにデータを表示してイベントを処理する方法です。

また、構成がジェネレーターによってカプセル化され、ジェネレーター自体がセルであるため、多くの問題を解決できます。上記のものを含みます。

実装


public protocol TableCellGenerator: class {
 
    var identifier: UITableViewCell.Type { get }
    var cellHeight: CGFloat { get }
    var estimatedCellHeight: CGFloat? { get }
 
    func generate(tableView: UITableView, for indexPath: IndexPath) -> UITableViewCell
    func registerCell(in tableView: UITableView)
}
 
public protocol ViewBuilder {
 
    associatedtype ViewType: UIView
 
    func build(view: ViewType)
}

このような実装により、デフォルトの実装を作成できます。

public extension TableCellGenerator where Self: ViewBuilder {
 
    func generate(tableView: UITableView, for indexPath: IndexPath) -> UITableViewCell {
        guard let cell = tableView.dequeueReusableCell(withIdentifier: self.identifier.nameOfClass, for: indexPath) as? Self.ViewType else {
            return UITableViewCell()
        }
 
        self.build(view: cell)
 
        return cell as? UITableViewCell ?? UITableViewCell()
    }
 
    func registerCell(in tableView: UITableView) {
        tableView.registerNib(self.identifier)
    }

小さな発電機の例を挙げます： 

final class FamilyCellGenerator {
 
    private var cell: FamilyCell?
    private var family: Family?
 
    var didTapPerson: ((Person) -> Void)?
 
    func show(family: Family) {
        self.family = family
        cell?.fill(with: family)
    }
 
    func showLoading() {
        self.family = nil
        cell?.showLoading()
    }
}
 
extension FamilyCellGenerator: TableCellGenerator {
    var identifier: UITableViewCell.Type {
        return FamilyCell.self
    }
}
 
extension FamilyCellGenerator: ViewBuilder {
    func build(view: FamilyCell) {
        self.cell = view
        view.selectionStyle = .none
        view.didTapPerson = { [weak self] person in
            self?.didTapPerson?(person)
        }
        if let family = self.family {
            view.fill(with: family)
        } else {
            view.showLoading()
        }
    }
}

ここでは、構成とサブスクリプションの両方を非表示にしました。これで、状態をカプセル化できる場所ができました（テーブルで再利用されるため、セルに状態をカプセル化することができないため）。また、セル内のデータを「オンザフライ」で変更する機会を得ました。

に注意してくださいself.cell = view。セルを覚えていたので、このセルを再ロードせずにデータを更新できます。これは便利な機能です。

しかし、私は気を取られました。どのセルもジェネレーターで表すことができるため、DisplayManagerのインターフェースをもう少し美しくすることができます。

public protocol DataDisplayManager: class {
 
    associatedtype CollectionType
    associatedtype CellGeneratorType
    associatedtype HeaderGeneratorType
 
    init(collection: CollectionType)
 
    func forceRefill()
    func addSectionHeaderGenerator(_ generator: HeaderGeneratorType)

    func clearCellGenerators()

これは実際にはすべてではありません。適切な場所にジェネレータを挿入したり、削除したりできます。

ちなみに、特定のセルの後にセルを挿入すると便利な場合があります。特に、データを徐々にロードする場合（たとえば、ユーザーがTINを入力した場合、TIN情報をアップロードして表示し、TINフィールドの後にいくつかの新しいセルを追加しました）。

合計


セルの動作は次のようになります。

class ViewController: UIViewController {
 
    func update(data: [Products]) {
        let gens = data.map { ProductCellGenerator($0) }
        self.ddm.addGenerators(gens)

またはここ：

class ViewController: UIViewController {
    func update(fields: [Field]) {
        let gens = fields.map { field
            switch field.type {
            case .phone:
                let gen = PhoneCellGenerator(item)
                gen.didUpdate = { self.updatePhone($0) }
                return gen
            case .date:
                let gen = DateInputCellGenerator(item)
                gen.didTap = { self.showPicker() }
                return gen
            case .dropdown:
                let gen = DropdownCellGenerator(item)
                gen.didTap = { self.showDropdown(item) }
                return gen
            }
        }
 
        let splitter = SplitterGenerator()
        self.ddm.addGenerator(splitter)
 
        self.ddm.addGenerators(gens)
 
        self.ddm.addGenerator(splitter)
    }
}

要素を追加する順序を制御できると同時に、データ処理とUIへの要素の追加の間の接続が失われません。したがって、単純なケースでは、単純なコードがあります。困難なケースでは、コードはパスタにならず、同時に問題ないように見えます。テーブルを操作するための宣言型インターフェースが登場し、セルの構成をカプセル化しました。これにより、異なる画面間の構成とともにセルを再利用できます。

RDDMを使用する利点：

  • セル構成のカプセル化。
  • コレクションからアダプタへの作業をカプセル化することにより、コードの重複を減らします。
  • コレクションを操作する特定のロジックをカプセル化するアダプターオブジェクトの選択。
  • コードがより明確になり、読みやすくなります。
  • テーブルを追加するために作成する必要があるコードの量が削減されます。
  • セルからのイベントを処理するプロセスが簡略化されます。
