オブジェクト指向プログラミング（OOP）は、C ++、Java、C＃、Visual Basic .NET、Ruby、Objective-C、さらにはSwift言語の中核です。 JavaScript、Python、PHPを使用しているため、Webテクノロジーオブジェクトなしでは実現できません。

そのため、Matt Weissfeldがオブジェクト指向の考え方を開発し、特定のプログラミング言語でオブジェクト指向の開発を進めるように助言しています。

この本は、開発者向けに開発者向けに書かれており、特定の問題を解決するための最良のアプローチを選択できます。継承と構成を正しく適用する方法、集約と関連付けの違いを理解する方法、およびインターフェースと実装の混乱を防ぐ方法を学びます。

プログラミング技術は常に変化し、開発されていますが、オブジェクト指向の概念はプラットフォームに依存せず、一貫して効果的です。この出版物は、OOPの基本的な基礎である、設計パターン、依存関係、およびコードを理解しやすく、柔軟性があり、適切に維持されるSOLIDの原則に焦点を当てています。

1. SRP：唯一の責任原則


唯一の責任の原則は、クラスに変更を加えるのに必要な理由は1つだけであると述べています。各クラスとプログラムモジュールには、1つのタスクを優先する必要があります。したがって、複数の理由でクラスを変更する可能性のあるメソッドを導入しないでください。クラスの説明に「and」という単語が含まれている場合、SRPの原則に違反している可能性があります。言い換えると、各モジュールまたはクラスはソフトウェア機能の一部を担当する必要があり、そのような責任はクラスに完全にカプセル化される必要があります。

図の階層を作成することは、継承を示す典型的な例の1つです。この例は、教育でよく見られ、この章全体（および本全体）で使用しています。この例では、CircleクラスはShapeクラスから属性を継承します。Shapeクラスは、サブクラスのコントラクトとして抽象calcArea（）メソッドを提供します。Shapeから継承する各クラスには、calcArea（）メソッドの独自の実装が必要です。

abstract class Shape{
     protected String name;
     protected double area;
     public abstract double calcArea();
}

この例では、Shapeクラスを継承するCircleクラスが、必要に応じてcalcArea（）メソッドの実装を提供します。

class Circle extends Shape{
     private double radius;

     public Circle(double r) {
           radius = r;
     }
     public double calcArea() {
           area = 3.14*(radius*radius) ;
           return (area);
     };
}

3番目のクラスCalculateAreasは、Shape配列に含まれるさまざまな形状の面積を計算します。Shape配列はサイズに制限がなく、正方形や三角形などのさまざまな形状を含めることができます。

class CalculateAreas {
     Shape[] shapes;
     double sumTotal=0;
     public CalculateAreas(Shape[] sh) {
           this.shapes = sh;
     }
     public double sumAreas() {
           sumTotal=0;
           for (inti=0; i<shapes.length; i++) {
           sumTotal = sumTotal + shapes[i].calcArea() ;
           }
           return sumTotal ;
     }
     public void output() {
           System.out.printIn("Total of all areas = " + sumTotal);
     }
}

CalculateAreasクラスは、問題を引き起こす可能性があるアプリケーション出力も処理することに注意してください。エリアカウント動作と出力動作は同じクラスに含まれているため、関連しています。

適切なテストアプリケーションTestShapeを使用して、このコードの機能を確認できます。

public class TestShape {
      public static void main(String args[]) {

            System.out.printin("Hello World!");

            Circle circle = new Circle(1);

            Shape[] shapeArray = new Shape[1];
            shapeArray[0] = circle;

            CalculateAreas ca = new CalculateAreas(shapeArray) ;

            ca.sumAreas() ;
            ca.output();
      }
}

これでテストアプリケーションを自由に使用できるようになったので、単独責任の原則の問題に焦点を当てることができます。繰り返しますが、問題はCalculateAreasクラスにあり、このクラスには、データを出力するだけでなく、さまざまな形状の領域を合計するための動作が含まれているという事実があります。

基本的な質問（および実際には問題）は、output（）メソッドの機能を変更する必要がある場合、形状の面積を計算するメソッドが変更されているかどうかに関係なく、CalculateAreasクラスを変更する必要があるということです。たとえば、プレーンテキストではなく、突然HTMLコンソールにデータを出力したい場合は、図形の領域を追加するコードを再コンパイルして再埋め込む必要があります。責任が関連しているからです。

唯一の責任の原則に従って、1つのメソッドを変更するタスクは他のメソッドに影響を与えず、再コンパイルする必要はありません。 「クラスには変更の理由が1つだけ必要です-変更する必要がある唯一の責任です。」

この問題を解決するには、2つのメソッドを別々のクラスに配置します。1つは元のコンソール出力用、もう1つはHTML出力用です。

class CaiculateAreas {;
     Shape[] shapes;
     double sumTotal=0;

     public CalculateAreas(Shape[] sh) {
           this.shapes = sh;
     }

     public double sumAreas() {
           sumTotal=0;

           for (inti=0; i<shapes.length; i++) {

                sumTotal = sumTotal + shapes[i].calcArea();

           }

                return sumTotal;
           }
}
class OutputAreas {
     double areas=0;
     public OutputAreas (double a) {
           this.areas = a;
     }

           public void console() {
           System.out.printin("Total of all areas = " + areas);
     }
     public void HTML() {
           System.out.printIn("<HTML>") ;
           System.out.printin("Total of all areas = " + areas);
           System.out.printin("</HTML>") ;
     }
}

ここでの要点は、必要に応じてさまざまな方向に結論を送信できるようになったことです。JSONなどの別の出力メソッドの可能性を追加する場合は、CalculateAreasクラスを変更せずに、それをOutputAreasクラスに追加できます。その結果、他のクラスに影響を与えることなく、CalculateAreasクラスを再配布できます。

2. OCP：オープン/クローズの原則


開放性/近接性の原則は、変更を加えずにクラスの動作を拡張できることを示しています。

数字の例に再び注目してみましょう。以下のコードには、Rectangleオブジェクトを取り、このオブジェクトの面積を計算し、値を返すShapeCalculatorクラスがあります。これは単純なアプリケーションですが、長方形でのみ機能します。

class Rectangle{
     protected double length;
     protected double width;

     public Rectangle(double 1, double w) {
           length = 1;
           width = w;
     };
}
class CalculateAreas {
     private double area;

     public double calcArea(Rectangle r) {

           area = r.length * r.width;

           return area;
     }
}
public class OpenClosed {
      public static void main(String args[]) {

            System.out.printin("Hello World");

            Rectangle r = new Rectangle(1,2);

            CalculateAreas ca = new CalculateAreas ();

            System.out.printin("Area = "+ ca.calcArea(r));
      }
}

このアプリケーションが長方形の場合にのみ機能するという事実は、開放性/閉鎖性の原理を明確に説明する制限につながります。CircleクラスをCalculateAreaクラスに追加したい場合（機能を変更）、モジュール自体に変更を加える必要があります。明らかに、これは、モジュールの動作を変更するためにモジュールに変更を加えてはならないことを示すオープン性/クローズ性の原則と矛盾します。

開放性/近接性の原則に準拠するために、抽象的なShapeクラスが作成され、図が抽象getArea（）メソッドを持つShapeクラスから直接継承する、すでにテスト済みの図を使用した例に戻ることができます。

現時点では、Shapeクラス（たとえばCircleクラス）を直接変更する必要なしに、必要な数の異なるクラスを追加できます。これで、Shapeクラスが閉じていると言えます。

以下のコードは、四角形と円のソリューションの実装を提供し、無制限の数の形状を作成できるようにします。

abstract class Shape {
      public abstract double getArea() ;
}
class Rectangle extends Shape
{

      protected double length;
      protected double width;

      public Rectangle(double 1, double w) {
            length = 1;
            width = w;
      };
      public double getArea() {
            return length*width;
      }

}
class Circle extends Shape
{
      protected double radius;

      public Circle(double r) {
            radius = r;
      };
      public double getArea() {
            return radius*radius*3.14;
      }
}
class CalculateAreas {
      private double area;

      public double calcArea(Shape s) {
            area = s.getArea();
            return area;
      }
}

public class OpenClosed {
      public static void main(String args[]) {

            System.out.printiIn("Hello World") ;

            CalculateAreas ca = new CalculateAreas() ;

            Rectangle r = new Rectangle(1,2);

            System.out.printIn("Area = " + ca.calcArea(r));

            Circle c = new Circle(3);

            System.out.printIn("Area = " + ca.calcArea(c));
}
}

この実装では、Shapeクラスの新しいインスタンスを作成するときにCalculateAreas（）メソッドを変更しないでください。

以前のコードの存在を気にすることなく、コードをスケーリングできます。開放性/近接性の原則は、元のクラスが編集を必要としないように、サブクラスを使用してコードを拡張する必要があることです。ただし、「拡張」自体の概念自体は、SOLIDの原理に関するいくつかの議論で議論の余地があります。端的に言えば、継承よりも構成を好む場合、これは開放性/近接性の原則にどのように影響しますか？

SOLID原則の1つに準拠している場合、コードは他のSOLID原則の基準を満たす場合があります。たとえば、公開性/近接性の原則に従って設計する場合、コードは単一責任の原則の要件に適合します。

3. LSP：Lisk置換の原則


Liskov置換の原則によれば、設計は、親クラスのインスタンスをいずれかの子クラスのインスタンスに置き換える可能性を提供する必要があります。親クラスが任意のタスクを実行できる場合、子クラスも実行できる必要があります。

一見正しいが、Lisk置換の原則に違反するコードを考えてみます。以下のコードには、汎用のShape抽象クラスが含まれています。次に、RectangleクラスはShapeクラスから属性を継承し、その抽象calcArea（）メソッドをオーバーライドします。次に、SquareクラスはRectangleを継承します。

abstract class Shape{
      protected double area;

      public abstract double calcArea();
}
class Rectangle extends Shape{
      private double length;
      private double width;

      public Rectangle(double 1, double w) {
            length = 1;
            width = w;
      }
      public double calcArea() {
            area = length*width;
            return (area) ;
      };
}
class Square extends Rectangle{
      public Square(double s) {
            super(s, Ss);
      }
}

public class LiskovSubstitution {
      public static void main(String args[]) {

            System.out.printIn("Hello World") ;

            Rectangle r = new Rectangle(1,2);

            System.out.printin("Area = " + r.calcArea());

            Square s = new Square(2) ;

            System.out.printin("Area = " + s.calcArea());
      }
}

これまでのところ、とても良いです。四角形は図のインスタンスであり、正方形は四角形のインスタンスなので、心配する必要はありません。

さて、哲学的な質問をしましょう：正方形はまだ長方形ですか？多くは肯定的に答えます。正方形は長方形の特殊なケースであると想定できますが、そのプロパティは異なります。長方形は正方形のような平行四辺形です（反対側は同じです）。同時に、正方形も菱形（すべての辺が同じ）ですが、長方形は菱形ではありません。したがって、違いがあります。

オブジェクト指向設計の場合、問題はジオメトリではありません。問題は、長方形と正方形を正確に作成する方法です。次に、Rectangleクラスのコンストラクタを示します。

public Rectangle(double 1, double w) {
      length = 1;
      width = w;
}

明らかに、コンストラクターには2つのパラメーターが必要です。ただし、Squareクラスのコンストラクターに必要なのは1つだけですが、親クラスのRectangleには2つ必要です。

class Square extends Rectangle{
      public Square(double s) {
      super(s, Ss);
}

実際、面積を計算するための関数は、これら2つのクラスのそれぞれで少し異なります。つまり、Squareクラスは、Rectangleを模倣して、同じパラメーターをコンストラクターに2回渡しています。このような回避策は非常に適しているように思えるかもしれませんが、実際には、コードに付随する開発者を誤解させる可能性があり、将来的に伴うと、落とし穴がかなりあります。少なくともこれは問題であり、おそらく、疑わしい設計上の決定です。あるコンストラクターが別のコンストラクターを呼び出すときは、休憩を取って構成を再検討することをお勧めします-おそらく子クラスが適切に構築されていません。

この状況から抜け出す方法を見つけるには？簡単に言えば、SquareクラスをRectangleに置き換えることはできません。したがって、SquareはRectangleクラスの子であってはなりません。それらは別々のクラスでなければなりません。

abstract class Shape {
      protected double area;

      public abstract double calcArea();
}

class Rectangle extends Shape {

      private double length;
      private double width;

      public Rectangle(double 1, double w) {
            length = 1;
            width = w;
      }

      public double calcArea() {
            area = length*width;
            return (area);
      };
}

class Square extends Shape {
      private double side;

      public Square(double s) {
            side = s;
      }
      public double calcArea() {
            area = side*side;
            return (area);
      };
}

public class LiskovSubstitution {
      public static void main(String args[]) {

             System.out.printIn("Hello World") ;

             Rectangle r = new Rectangle(1,2);

             System.out.printIn("Area = " + r.calcArea());

             Square s = new Square(2) ;

             System.out.printIn("Area = " + s.calcArea());
      }
}

4. ISP：インターフェース共有の原則


インターフェースの分離の原則は、いくつかの大きなインターフェースよりも多くの小さなインターフェースを作成する方が良いと述べています。

この例では、Mammalクラスのいくつかの動作、つまりeat（）とmakeNoise（）を含む単一のインターフェースを作成します。

interface IMammal {
     public void eat();
     public void makeNoise() ;
}
class Dog implements IMammal {
     public void eat() {
           System.out.printIn("Dog is eating");
     }
     public void makeNoise() {
           System.out.printIn("Dog is making noise");
     }
}
public class MyClass {
      public static void main(String args[]) {

            System.out.printIn("Hello World");

            Dog fido = new Dog();
            fido.eat();
            fido.makeNoise()
      }
}

Mammalクラスの単一のインターフェースを作成する代わりに
、すべての動作に対して個別のインターフェースを作成する必要があります。

interface IEat {
     public void eat();
}
interface IMakeNoise {
     public void makeNoise() ;
}
class Dog implements IEat, IMakeNoise {
     public void eat() {
           System.out.printIn("Dog is eating");
     }
     public void makeNoise() {
           System.out.printIn("Dog is making noise");
     }
}
public class MyClass {
      public static void main(String args[]) {

            System.out.printIn("Hello World") ;

            Dog fido = new Dog();
            fido.eat();
            fido.makeNoise();
      }
}

振る舞いをMammalクラスから分離します。継承（より正確にはインターフェース）を介して単一のMammalクラスを作成する代わりに、前の章で使用した戦略と同様に、構成に基づいた設計に移ることがわかりました。

簡単に言うと、このアプローチでは、コンポジションを使用してMammalクラスのインスタンスを作成でき、単一のMammalクラスに埋め込まれているビヘイビアーを使用せずに済みます。たとえば、食べずに皮膚から栄養素を吸収する哺乳類が発見されたとします。 eat（）動作を含むMammalクラスから継承する場合、この動作は新しい哺乳類では冗長になります。さらに、すべての動作が個別の単一のインターフェースに配置されている場合、意図したとおりに各哺乳類のクラスを構築することが判明します。
