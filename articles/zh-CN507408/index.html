情况1：用于无锁编程的有序原子变量



无锁编程涉及使用比互斥锁更低级别的工具在线程之间建立通信和同步。在过去和今天，都有各种各样的此类工具。按照粗略的历史顺序，它们包括显式障碍（显式栅栏/障碍-如Linux上的mb（）），特殊的订购API调用（如Windows上的InterlockedExchange）以及各种特殊的原子类型。这些工具中很多都是乏味和/或复杂的，它们的种类繁多意味着最终，无锁代码在不同环境中的编写方式有所不同。



但是，在过去的几年中，硬件和软件提供商之间已经出现了显着的融合：计算行业正在围绕顺序同意的有序原子变量作为标准或使用主要OS语言和平台编写无锁代码的唯一方法进行联合。简而言之，有序原子变量可以安全地同时在多个线程中读写，而无需任何显式锁，因为它们提供了两个保证：保证它们的读写按照它们在程序源代码中出现的顺序执行；并且保证每次阅读或写作都是原子的，“全有或全无”。它们还具有特殊操作，例如compareAndSet，可以保证自动执行。有关有序原子变量以及如何正确使用它们的更多信息，请参见[1]。

关于优化的几句话



我们将研究有序原子变量如何限制编译器，处理器，缓存效果和运行时其他元素可以执行的优化。因此，让我们首先快速了解一些基本的优化规则。



几乎所有语言的基本优化规则是这样的：对代码的执行进行重新排序（``转换''）的优化始终只有在不改变程序含义的情况下才是合法的，因此程序无法确定源代码的执行与转换后的代码之间的区别。在某些语言中，这也称为“按原样”规则，它的名称是由于转换后的代码具有相同的可观察效果，​​即源代码“好像”（好像）以以下形式执行：它最初是在其中编写的。



该规则具有双重作用：首先，优化绝不能使您无法获得以前不可能的结果，也不能违反允许源代码依赖的任何保证，包括语言语义。如果我们给出了不可能的结果，那么最终，程序和用户肯定会注意到差异，而这不再是“好像”我们已经执行了原始的未转换代码。



其次，优化允许减少可能执行的次数。例如，优化可能导致以下事实：永远不会发生某些潜在（但不能保证）的交替（更改指令的执行顺序-交错）。这是正常现象，因为程序仍然不能依靠它们来发生。



有序原子变量和优化



使用有序原子变量会限制编译器，处理器和缓存系统可以执行的优化类型。[3]两种类型的优化值得注意：



  • 优化有序原子读写操作。
  • 优化相邻的正常读写操作。


首先，必须严格按照源代码的顺序执行给定流中的所有有序原子读写操作，因为这是有序原子变量的基本保证之一。尽管如此，我们仍然可以执行某些优化，尤其是优化，其效果与该流始终总是执行得如此之快以致于在某些点上不会与另一个流交替发生的效果相同。



例如，考虑以下代码，其中a是有序原子变量： 



a = 1;  	// A
a = 2;  	// B


编译器，处理器，缓存或运行时的其他部分是否可以将上述代码转换为以下代码（不包括A行中的冗余条目）？ 



// A ': OK:    A 
a = 2;  	// B


答案是肯定的。这是合法的，因为该程序无法确定差异。好像“好像”该线程一直如此快地工作，以至于原则上没有其他并行运行的线程可以在A和B行之间交替以查看中间值。[4]



同样，如果a是有序原子变量，而local是不可分割的局部变量，则可以进行转换 



a = 1;  	// C:   a
local = a;  // D:   a


在 



a = 1;  	// C:   a
local = 1;  // D': OK,  " "


其不包括从读取一个。即使另一个线程尝试同时写入a，也似乎“好像”这个线程总是运行得如此之快，以至于另一个线程在设法将自己的值写回local之前，从未设法在C和D行之间交替更改值。



其次，附近的普通读写操作仍然可以在有序原子操作周围重新排序，但有一些限制。特别是，如[3]中所述，普通的读和写操作不能相对于有序原子读取（从“之后”到“之前”）向上移动，也不能相对于有序原子读取（从“到”之前的向下移动）降低。到之后）。简而言之，这可以使它们脱离代码的关键部分，并且您可以编写受益于此性能的程序。有关更多信息，请参见[3]。
