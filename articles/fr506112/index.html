Comment visualiser les dépenses quotidiennes sur les solutions cloud GCP


Les clients n'aiment pas payer plus que prévu - une justification détaillée des coûts fait partie intégrante et importante de la mise en œuvre des technologies cloud.

Google Cloud Platform propose différents plans tarifaires pour les ressources utilisées. Par exemple, le coût de GCE dépend de la configuration de l'ordinateur ( CPU , mémoire, modules réseau, disques durs). Les coûts de Google Kubernetes Engine (GKE) et de Google Cloud Dataproc sont basés sur tous les nœuds exécutés sur Google Compute Engine (GCE). Les coûts restants peuvent être calculés par une formule complexe et complexe. La planification d'un budget devient de plus en plus difficile, surtout si vous utilisez plusieurs technologies cloud. Le suivi et les rapports en temps opportun deviennent d'autant plus précieux que les dépenses en infrastructure augmentent.

La possibilité d'une vérification quotidienne des rapports de dépenses permettra également un ajustement en temps voulu des capacités réparties, et la facture finale en fin de mois ne sera pas une surprise.

Contexte


Les projets nécessitent souvent des rapports détaillés pour améliorer l'efficacité et suivre les coûts. Bien que GCP vous permette d'exporter les données de dépenses vers BigQuery pour une analyse plus approfondie, le traitement manuel de BigQuery n'est souvent pas suffisant. Idéalement, nous devrions pouvoir visualiser les données, les filtrer et les accumuler en fonction de paramètres supplémentaires.

Filtres communs:

  • Type de ressource;
  • Une période de temps;
  • Département et équipe;
  • Comparaison avec les périodes précédentes.

La première solution, basée sur la recommandation de Google, consiste à utiliser Google Data Studio . L'implémentation est très simple - il vous suffit de configurer la source de données. Des exemples de rapports et un tableau de bord sont préinstallés, mais la solution elle-même n'est pas suffisamment flexible. Lors de la création de tableaux et de graphiques dans Google Data Studio , vous ne pouvez pas saisir de formule, vous devez sélectionner tous les paramètres manuellement.

Grafana est un outil simple et compréhensible pour surveiller et analyser les données. Il est bien établi. Un tel tableau de bord serait idéal pour visualiser les données de paiement. La question reste de savoir comment connecter l'interface Web à BigQuery (BQ) . Plugin BQ, open source, trop bogué. En conséquence, le bouquet de BQ-Grafana n'est pas assez stable. De plus, il y a un autre point gênant: les requêtes BQ renvoient des données trop longtemps. Le téléchargement de

données vers PostgreSQL via CloudSQL s'est avéré être une bonne solution . Ce SGBD a le plugin officiel PostgreSQL pour Grafana . Et les résultats des tests ultérieurs ont montré que la méthode sélectionnée présente des avantages évidents en termes de vitesse.

Vue d'ensemble de la solution


Le workflow peut être décrit schématiquement comme suit. L'outil de tâche en attente dans le cluster Kubernetes exécute une tâche dans Cloud Dataflow toutes les 4 heures . Cette tâche, à son tour, démarre le processus de chargement des données de BigQuery vers PostgreSQL . Le déchargement a lieu uniquement sur les données trouvées après la dernière exportation de BQ . Après cela, les dernières données peuvent être vues dans Grafana , connecté à PostgreSQL .

Flux de données


Nous créons un compte avec accès à DataFlow et BigQuery . Cela permet de garantir que les tâches DataFlow peuvent recevoir des données de BigQuery .

export  project=myproject
gcloud iam service-accounts create "bq-to-sql-dataflow" --project ${project}
 
gcloud projects add-iam-policy-binding ${project} \
--member serviceAccount:"bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--role roles/dataflow.admin
 
gcloud projects add-iam-policy-binding ${project} \
--member serviceAccount:"bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--role roles/bigquery.dataViewer


Pour les tâches DataFlow , vous devez créer deux conteneurs: un pour temporaire et un pour la sortie.

gsutil mb gs://some-bucket-temp

Script pour charger les données


Un script est requis pour télécharger les données de BigQuery vers CloudSQL . Cloud DataFlow prend en charge Python et Javascript . Vous aurez également besoin de la bibliothèque Apache Beam , d'un fichier json (défini dans la variable d'environnement GOOGLE_APPLICATION_CREDENTIALS avec des privilèges de compte préconfigurés) et d'un fichier requirements.txt qui contient une liste de packages à installer (dans notre cas, nous n'avons besoin que d'un package beam-nuggets ). Le script Python est présenté ci-dessous:

Texte masqué
bq-to-sql :

args = parser.parse_args()
project = args.project
job_name = args.job_name + str(uuid.uuid4())
bigquery_source = args.bigquery_source
postgresql_user = args.postgresql_user
postgresql_password = args.postgresql_password
postgresql_host = args.postgresql_host
postgresql_port = args.postgresql_port
postgresql_db = args.postgresql_db
postgresql_table = args.postgresql_table
staging_location = args.staging_location
temp_location = args.temp_location
subnetwork = args.subnetwork
 
options = PipelineOptions(
        	flags=["--requirements_file", "/opt/python/requirements.txt"])
# For Cloud execution, set the Cloud Platform project, job_name,
# staging location, temp_location and specify DataflowRunner.
 
google_cloud_options = options.view_as(GoogleCloudOptions)
google_cloud_options.project = project
google_cloud_options.job_name = job_name
google_cloud_options.staging_location = staging_location
google_cloud_options.temp_location = temp_location
google_cloud_options.region = "us-west1"
worker_options = options.view_as(WorkerOptions)
worker_options.zone = "us-west1-a"
worker_options.subnetwork = subnetwork
worker_options.max_num_workers = 20
 
options.view_as(StandardOptions).runner = 'DataflowRunner'
 
start_date = define_start_date()
with beam.Pipeline(options=options) as p:
	rows = p | 'QueryTableStdSQL' >> beam.io.Read(beam.io.BigQuerySource(
                    	query='SELECT \
                        	billing_account_id, \
                        	service.id as service_id, \
                        	service.description as service_description, \
                        	sku.id as sku_id, \
                        	sku.description as sku_description, \
                        	usage_start_time, \
                        	usage_end_time, \
                        	project.id as project_id, \
                        	project.name as project_description, \
                        	TO_JSON_STRING(project.labels) \
                            	as project_labels, \
                        	project.ancestry_numbers \
                            	as project_ancestry_numbers, \
                        	TO_JSON_STRING(labels) as labels, \
                        	TO_JSON_STRING(system_labels) as system_labels, \
                        	location.location as location_location, \
                        	location.country as location_country, \
                        	location.region as location_region, \
                        	location.zone as location_zone, \
                        	export_time, \
                        	cost, \
                        	currency, \
                        	currency_conversion_rate, \
                        	usage.amount as usage_amount, \
                        	usage.unit as usage_unit, \
                        	usage.amount_in_pricing_units as \
                         	usage_amount_in_pricing_units, \
                        	usage.pricing_unit as usage_pricing_unit, \
                        	TO_JSON_STRING(credits) as credits, \
                        	invoice.month as invoice_month, \
                   	     cost_type \
FROM `' + project + '.' + bigquery_source + '` \
                        	WHERE export_time >= "' + start_date + '"',
                    	use_standard_sql=True))
	source_config = relational_db.SourceConfiguration(
          	              drivername='postgresql+pg8000',
                        	host=postgresql_host,
                        	port=postgresql_port,
                        	username=postgresql_user,
                        	password=postgresql_password,
                        	database=postgresql_db,
                        	create_if_missing=True,
                        	)
	table_config = relational_db.TableConfiguration(
                        	name=postgresql_table,
            	            create_if_missing=True
                        	)
	rows | 'Writing to DB' >> relational_db.Write(
    	source_config=source_config,
    	table_config=table_config
	)


Pour la cohérence des données, vous devez déterminer le temps d' export maximal , le temps d'exportation vers PostgreSQL , puis charger des enregistrements à partir de BigQuery qui commenceraient avec cette heure limite .

Une fois les données chargées, vous devez mettre à jour la vue matérialisée. Comme ce processus prend un certain temps, cela vous permettra de créer une nouvelle vue matérialisée avec une structure et des indices identiques, de supprimer l'ancienne et de renommer la nouvelle.

Création d'un fichier JSON avec des autorisations SA


Le compte que vous avez créé précédemment pour le workflow Cloud Dataflow est également utilisé dans les tâches Cron . Vous devez spécifier une commande qui créera un mot de passe de clé privée pour le compte, puis téléchargé vers le cluster Kubernetes , où il agira comme un mot de passe caché pour accéder à la tâche Cron .

gcloud iam service-accounts keys create ./cloud-sa.json \
--iam-account "bq-to-sql-dataflow@${project}.iam.gserviceaccount.com" \
--project ${project}

Déployer un mot de passe secret dans un cluster K8s:

kubectl create secret generic bq-to-sql-creds --from-file=./cloud-sa.json
