最新のLinuxバージョンでのFirebird 3のインストール：CentOS8とUbuntu 19

この記事では、新しいLinuxディストリビューションにFirebirdバージョン3.0を最適にインストールするために必要な最小限の手順について説明します。例として、CentOS 8とUbuntu 19が選択されました。

ターゲットシステムへのFirebirdディストリビューションの「配信」では、このガイドでは、プロジェクトの公式ウェブサイト（firebirdsql.org）からリンクからtar.gzアーカイブをダウンロードするオプションが選択されました。

最もせっかちな場合-すぐに戦闘に：

クイックインストール


次/etc/sysctl.confの行を追加してファイルを編集します。

vm.max_map_count = 256000

ファイルを保存して設定を適用します。

sudo sysctl -p /etc/sysctl.conf

CentOS 8とUbuntu 19では手順が異なりますが、配布パッケージをダウンロードするための公式のFirebirdプロジェクトWebサイトからのリンクと、起動プロセス中に配布パッケージが解凍されるディレクトリも示されています。
現在（2020年3月）Firebird 3.0.5リリースが関連しています（ここに64ビットバージョンへのリンクがあります）。

CentOS 8

sudo yum -y install epel-release
sudo yum -y makecache
sudo yum -y install libicu libtommath tar
ln -s libncurses.so.5 \
/usr/lib64/libncurses.so.5
ln -s libtommath.so.1 \
/usr/lib64/libtommath.so.0
curl -L |tar -zxC /tmp

Ubuntu 19

sudo apt-get -y install libncurses5 libtommath1
ln -s libtommath.so.1 \
/usr/lib/x86_64-linux-gnu/libtommath.so.0
wget -O- |tar -zxC /tmp

Firebird DBMSを実際にインストールする：

cd /tmp/
sudo ./install.sh

これらのアクションの目的をよりよく理解したい場合は、読み進めてください。

主要部分

小さなプリアンブル

OSが最小バージョンですでにインストールされており、パブリックリポジトリまたはローカルコピーへのアクセスが構成されていることを前提としています。

読者はLinuxとFirebird DBMSの基本的な知識があることを前提としています。

計画

一時ファイル（/tmp）、データベースファイル、およびローカルバックアップ用の個別のセクションをDBMSサーバーに割り当てることをお勧めします。

一時インクルードには、ロックファイル、ソートファイル、グローバル一時テーブル（GTT）の「マテリアライズ」ファイル、およびモニタリングテーブルが含まれます。ソートファイルとグローバル一時テーブルはにあり/tmp、mon $テーブルファイルとロックファイルはにあり/tmp/firebirdます。

ソートファイルはunlink作成直後に「削除」（）されるので、ディレクトリリストに「表示」することはできませんdeleted。

sudo ls -lhF /proc/`pgrep firebird`/fd

疑似/proc/…/fd/ディレクトリのリストでは、シンボリックリンクが表示され、ファイルに関する実際の情報は次のように表示されます。

sudo stat -L /proc/`pgrep firebird`/fd/

どこの関心のファイルの記述（記述）です。

「pgrep -」を呼び出す代わりに、対象のプロセスの識別子をすぐに置き換えることができます。

一時ファイルは非常に大きくなる可能性があるため/tmp、少なくとも20〜30 GBを割り当てることをお勧めします。ソートファイルのサイズは、リクエストで明示的または暗黙的にソートされるデータの量にのみ依存し、1人のユーザーがギガバイトの一時ファイルを「作成」できることに注意してください。

データベースファイルのセクションには、すべてのデータベースファイルを含める必要があります。さらに、少なくとも最大のデータベースファイルのコピー。数年前の将来のデータベースファイルの増加を考慮に入れる必要があります。

ローカルバックアップセクションには、すべてのベースのバックアップアーカイブと、最大のベースのバックアップが少なくとも1つ含まれている必要があります。このセクションには、最大のベースを復元する場所もあることが推奨されます。今後数年間のバックアップとバックアップアーカイブの増加を考慮に入れる必要があります。

予備的な準備

Firebird 3.0 DBMSサーバーは、システムメモリを動的に割り当てて解放するため、断片化につながる可能性があります。たとえば、多数のユーザーが一度にスーパーサーバーから切断すると、新しい接続でエラーが発生する場合があります。

メモリの断片化は、システムパラメータによって制御され、vm.max_map_countデフォルトでは64Kです。その値を4倍に増やすことをお勧めします。

sudo sysctl vm.max_map_count=256000

システムの再起動時に新しい値を設定するには、次の/etc/sysctl.conf行をファイルに追加します。

vm.max_map_count = 256000

このパラメーターを変更する理由がわかるように、コメントを付けることをお勧めします。最初にファイルを編集してから、ファイルに保存されている設定を適用できます。

sudo sysctl -p /etc/sysctl.conf

Firebird DBMS配布キットをダウンロードします。

Firebirdプロジェクトの公式Webサイト（firebirdsql.org）は、「公式」リリース（リリース）と「日次」アセンブリ（スナップショットビルド）のディストリビューションへのリンクを公開しています。

公式のLinuxリリースはアーカイブ（tar.gz）およびdeb / rpmパッケージとして入手でき、ビルドはアーカイブとしてのみ入手できます。「汎用インストーラー」（tar.gzの汎用インストーラー）を見ていきます。

アセンブリアーカイブをダウンロードして解凍する必要がありますが、これらのプロセスの両方を組み合わせます。解凍はで行われ/tmp、URLはダウンロード可能なアーカイブへのリンクを示します。

カール：

curl -L URL | tar -zxC /tmp

wget：

wget -O– URL | tar -zxC /tmp

デフォルトでcurlは、ロードされたデータを標準出力に送信しますが、リダイレクトは処理せず「‑L」を追加wgetしますが、逆に、リダイレクトは処理しますが、データをファイルに書き込み、「」を追加します‑O‑。以下のためtarの使用を示すgzipアンパックが実行されるにフィルタやディレクトリを。プロセスが完了すると、ディレクトリのようなものだろうFirebird‑3.0.5.33220‑0.amd64三つのファイルで：install.sh、buildroot.tar.gzとmanifest.txt。

Firebirdのインストール

事前準備中に、システムパラメータの値を調整しvm.max_map_count、可用性を確認して、ICU、ncurses、およびtommathライブラリをインストールしました。 ncursesとtommath（libncures.so.5およびlibtommath.so.0）のバージョンが正しいことを確認し、必要なシンボリックリンクを作成しました。

実際のインストールは非常に簡単です。Firebird配布キットのアーカイブが解凍されたディレクトリに移動し、チェックし、必要に応じてスクリプトの「実行可能」フラグを設定しますinstall.sh。

chmod +x install.sh

インストールスクリプトを実行します。

sudo ./install.sh

Enterキーを押して、インストールの開始を確認し、要求を受け取ったら、sysdbaパスワードを入力します。

インストールスクリプトは自動的にsystemdユニットfirebird-superserver（デフォルトのFirebird 3.0アーキテクチャ）を起動します。Firebirdサービスは、スーパーサーバーのデフォルトのパラメーターである2048ページ（ベースごと）のページキャッシュ、64 MB（共有）のソートバッファー、およびサードバージョンのクライアントのみの接続で動作します。表示オプションfirebird.conf：

grep -v ^# firebird.conf | grep -v ^$

からの新しい値はfirebird.conf、Firebirdサービスが再起動された後にのみアクティブ化されることに注意してください。

パラメータ値を選択するときは、ページキャッシュ（データベース用）、ソートバッファ（共有）、およびクライアント接続用にサーバーによって割り当てられたメモリの3つの主要な「コンシューマ」があることを考慮する必要があります。管理できるのは最初の2つだけです。クライアント接続のメモリ量は、キャッシュされたリクエストの数とテキスト、それらのプラン、およびリクエストに関連するデータベースオブジェクトによって異なります。クライアント接続のメモリ評価は経験的にのみ行われ、クライアントアプリケーションやデータベースオブジェクトが変更されると変更される可能性があります。

少量のメモリ（最大12〜16 GB）を備えたホスト上のスーパーサーバーでは、ページキャッシュとソートバッファーに合計RAMの3分の1〜4分の1以上を割り当てないでください。

データベースの数が固定されておらず、変化する可能性がある場合は、ページキャッシュの合計メモリを、サーバー上に配置できるデータベースの最大数で割る必要があります。ページキャッシュのサイズはページ単位で設定され、個別にバイトに変換する必要があります。

クラシックアーキテクチャに切り替えるには、少なくともを明示的に指定ServerModeしfirebird.conf、同じ場所のページキャッシュを減らし（2K以下）、ソートバッファー（すべてのソートの合計許容量を最大接続数で割った値）を減らし、ユニットを無効にして停止しfirebird-superserver、有効にして実行する必要があります。ユニットfirebird-classic.socket。

Firebird 3.0でスーパークラシックアーキテクチャを使用してもあまり意味がありません。「信頼性」-スーパーサーバーと同じように、一般的な並べ替えバッファーは同じです。一般的なページキャッシュはなく、それらの間の異なる接続を同期するための「損失」は、クラシックの場合と同じです。

Firebird 3.0では、一部のパラメータ（ページキャッシュ、ロックファイルサイズ、ハッシュテーブルなど）をdatabases.confデータベースごとに個別に設定できることに注意してください。スーパーサーバの場合、それは例えば、小さな値に設定する、便利であるDefaultDbCachePages中firebird.confで、必要なデータベースへと設定し、個々のページキャッシュをdatabases.conf。
