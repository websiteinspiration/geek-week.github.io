大人のポジティブチャレンジ

Habréには、新しいPosit浮動小数点形式に関する記事（1つ、2つ、2つ、および半分）がすでにいくつかありました。その著者は、あらゆる点で標準のIEEE 754 floatよりも優れていると述べています。新しいフォーマットは、批評家（た1、2 POSITの欠点は、その利点を上回ると主張します）。しかし、もし私たちが本当に新しい革命的なフォーマットを持っていて、批判が単に批判する人々の羨望と無能によって引き起こされたらどうでしょうか？まあ、見つけるための最良の方法は、あなた自身を取って計算することです。

前書き


新しい形式の利点は、類似した次数の数値の加算/乗算を使用した簡単な例で示され、1桁または2桁の精度が高くなります。ただし、実際の計算では、いずれにしても精度が制限されて計算されるため、単一の操作の誤差のプラス/マイナス1ビットは問題になりません。一連の操作の結果としてのエラーの蓄積は重要です。しばらくすると、下の桁のエラーが古い桁のエラーにつながり始めます。これが私たちが体験しようとすることです。

トレーニング


テストのために、私はここから pathos名でPosit実装を採用しました。Visual Studioでコンパイルするには、util.hファイルに#define CLZ（n）__lzcnt（n）という行を追加し、posit.cppファイルの0.f / 0.fをstd :: numeric_limits <float> :: quiet_NaNに置き換える必要がありました。 （）。ちなみに、基本的な数学関数（ルートを除く）の実装もこのライブラリにはありませんでした。これが、何かがおかしいと思われるもう1つの理由です。

テスト1.複素数の乗算


可能な限り、複雑な算術を使用して計算されたフーリエ変換が使用されます。最初は、フーリエ変換でPositをテストしたいと思いました。ただし、その精度は実装に大きく依存するため、適切なテストを行うには、すべての基本的なアルゴリズムを考慮する必要がありますが、これには時間がかかります。したがって、複素数の乗算という単純な演算から始めることができます。

特定のベクトルを取り、それを360度1度回転させると、最後に同じ元のベクトルが得られます。実際、エラーの蓄積により、結果はわずかに異なります。ターン数が多いほど、エラーは大きくなります。したがって、この単純なコードを使用して

complex<T> rot(cos(a), sin(a));
complex<T> vec(length, 0);
for (long i = 0; i < count; i++)
{
	vec *= rot;
}
cout << "error: " << stdev(vec.real() - length, vec.imag()) << endl;

異なるタイプのデータを含むベクトルを回転させ、エラーを元のベクトルからの結果のベクトルの平均二乗偏差と見なします（これは、差分ベクトルの長さと解釈することもできます）。

テスト2.有理多項式の計算


元のライブラリには数学関数がなかったので、私たちは自分で何かをしようとします。多くの関数は、テイラー級数で展開することで不十分に近似されます。また、有理多項式による近似により計算するのに便利です。この近似は、純粋に分析的なパデ近似を含むさまざまな方法で取得できます。さらに、係数を十分に大きくしてテストし、計算前に丸めが行われるようにします。

結論


残念ながら、奇跡は起こらず、革命は中止されました。単一の計算で実証されたPositの利点はトリックにすぎません。その代償として、「重い」実際の計算の精度が壊滅的に低下します。IEEE 754浮動小数点または固定小数点の代わりにPositを使用することが理にかなっている唯一の理由は、宗教的です。作成者の聖なる信仰によって正確さが保証されている魔法のフォーマットを使用すると、プログラムに多くの不思議をもたらす可能性があります。検証と批判のための
