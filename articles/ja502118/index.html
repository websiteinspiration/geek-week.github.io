システムレベルでの設計。パート3：System ComposerとMathWorksツールチェーンのリンク

では最初の 2チュートリアルの一部では、システムアーキテクチャの構築とシステムレベルでの設計を検討し、同時にSystem Composerを検討しました。システム自体のアーキテクチャは優れていますが、開発したシステムと接続されていることを確認する必要があります。ところで、System Composerを作成した理由は、SysMLまたはUMLを使用する従来のツールでこのような接続がないことでした。実際、多くの企業がすでに開発にモデル指向設計（MOS）パラダイムを使用しており、システムエンジニアリングにサードパーティのツールを使用する必要があり、不便でした。System Composerは、このギャップを埋めるために作成されました。チュートリアルのこの最後の部分では、モデル指向設計のためにMathWorksツールチェーンと組み合わせてSystem Composerを使用する方法を示します。

要件へのアーキテクチャのバインド


まず、要件を定義しましょう。要件は、システムが行うべきことです。技術仕様との違いは、要件はシステムの機能の説明であるということです。MATLAB / SimulinkにはSimulink要件管理ツールがあります。IBM DOORSなどの外部システムから要件をインポートし、独自の要件エディターでそれらを書き込むことができます。要件自体は、拡張子が* .slreqxの特別なファイルに保存されます。要件を作成し、ファイルAccessControl.slreqxに保存します。最初の部分からの推論から要件自体を定式化します。

  • RFIDタグの読み取りを提供する必要があります。
  • RFIDタグから取得したデータは、外部データベースに転送する必要があります
  • データベースの応答に基づいて、禁止または許可が生成されます
  • ユーザーにアクセス状況を通知する必要があります
  • アクセスステータスに基づいてロックが解除されます

これらの要件は、Simulink Requirementsに含まれている要件作成ツールである要件エディターで作成され、ファイルに保存されました。要件パースペクティブを使用してこのファイルをモデル自体で開くと、次のようになります。



要件をアーキテクチャ要素に添付するには、マウスで要件を目的のアーキテクチャ要素に転送するだけです。

また、設計の初期段階でよくあるように、要件が変更された場合はどうなりますか？アーキテクチャへの影響を分析するにはどうすればよいですか？さいわい、Simulink要件では、要件の変更を追跡し、これらの変更の影響を受けたアーキテクチャ要素にマークを付けることができます。



要件のあるアーキテクチャのカバレッジを分析するには、[要件]タブで[共有]を選択し、[トレーサビリティマトリックスの生成]を選択します。要件と要素の関係をグラフィカルに表示するトレーサビリティマトリックスが作成されます。このマトリックスは表であり、その列はアーキテクチャまたはモデルの要素であり、行は要件自体であり、セルには要件と要素の間の関係に関するグラフィックノートが含まれています。

また、[不足しているリンクを強調表示]ボタンをクリックすると、マトリックスで覆われていない要素が黄色で強調表示されます。



要件の完全性とそのトレーサビリティの分析は、安全に不可欠なシステムを作成する場合に非常に重要なプロセスであり、飛行機、自動車、原子炉では問題になりません。このようなシステムでは、要件でカバーされていない要素があってはなりません。これらのシステムがどのように開発され、MOSパラダイムでどのように開発されるかに興味がある場合は、トピックが非常に広範囲で長く、チュートリアルの範囲を超えるため、コメントを書き込んでください。

MATLABでのアーキテクチャ分析


System ComposerはMathWorksツールチェーンの一部であるため、アーキテクチャのプロパティの分析、レポートの作成などを行うことができます。アーキテクチャの分析により、システムの実装に必要な労働時間、最小重量と寸法、TDPなどを計算できます。また、一度ではなく体系的に分析を行うと、システム開発のダイナミクスを確認したり、問題のある領域を見つけることができます。

ACSについて、労働時間数を計算するとします。すべてのコンポーネントには共通のワークロードプロパティがあり、明らかにこれらのプロパティの値を合計する必要があります。これを行うには、分析モデルをクリックして分析用のアーキテクチャインスタンスを作成し、分析用のGenericComponentを選択します。



次に、インスタンス化ボタンをクリックして次の結果を取得します。



ここで、ワークロードプロパティの値を各要素に割り当て、[更新]ボタンをクリックして、アーキテクチャ自体のこれらの値を更新できます。また、インスタンスはアーキテクチャとは別に存在するため、クリックする必要はありません。プロパティの値をいじって、プロジェクトの妥協点を見つけたり、最適なプロパティ値を見つけたりできます。分析自体は、MATLABで作成された別の関数によって実行されます。たとえば、次のコードは「分析」関数のコードです。

function AccessControl_simple_analytics(instance,varargin)
if instance.isComponent()
workload = 0;    
    for child=instance.Components
        child_workload = child.getValue("GenericComponent.Workload");
        workload = workload + child_workload;
    end
instance.setValue("GenericComponent.Workload",workload);
end
end

関数が作成されたら、「分析」をクリックして、「関数の選択」メニューでそれを選択します。ここで、[分析]ボタンをクリックすると、[ワークロード]プロパティの値が合計されます。



これは、アーキテクチャ分析の非常に単純な例であり、実際の戦闘分析タスクでは、カーブフィッティング、回帰分析など、MATLAB分析機能の全スペクトルが使用されます。

ここでの主なことは、プロジェクトを開発するために、アーキテクチャの体系的な分析を実行できること、および実行する必要があることです。

コンポーネントとその実装の関係


最後に、System ComposerはSimulinkとは別に存在する独立したツールではありません。アーキテクチャを定義したら、既存のモデルを選択したり、モデルを自動的に作成したりしながら、各コンポーネントをSimulinkモデルに関連付けることができます。これにより、シミュレーションを実行して、システムの動作特性をSystem Composerで直接調べることができます。



最も重要なのは、モデルがコンポーネントから作成された場合、必要なインターフェースを備えた入力ポートと出力ポートがモデルに自動的に生成されることです。これは非常に重要な些細なことです。なぜなら、コンポーネントの統合は通常、一貫性のないコンポーネントインターフェイスのために遅くなり、事前に構築されたインターフェイスがこの問題を解決するためです。生成された各モデルは特定のアーティストに与えることができ、後でアーキテクチャに統合されるので落ち着くことができます

調査結果

システムレベルでの設計には、タスクのかなり詳細な分析が必要であり、直感的なツールが必要です。 System Composerは学習しやすいツールであり、システムアーキテクチャの作成と分析、およびMATLABの分析機能のモデル指向設計手法を最大限に活用します。

System Composerを使用して設計決定を徹底的に分析することで、コンポーネントの性質を理解し、システムの潜在的なボトルネックを特定することができます。データフロー分析から数値分析まで、分析にはさまざまな方法を使用できます。
