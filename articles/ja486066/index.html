アクセスエリア。ポイントからエリアまでの距離を検出し、逆ジオコーディングの要求を減らす



たとえば、モスクワの環状道路など、特定の地理的地点から地図上のエリアまでの距離を計算する機能を実装しなければなりませんでした。その結果、良い結果が得られた問題を解決する方法を2つ見つけ、現在は本番環境で定期的に使用しています。それらについては、記事の最初の部分で説明します。そして2番目では、ジオデータをキャッシュしてジオコーダーをより少なく使用する方法を示します。

パート1。地図上のポイントからエリアまでの距離を見つける2つの方法


モバイルアプリケーションが本当にモバイルである場合、デバイスの座標で動作します。ユーザー（およびデバイス）の場所は、配信コスト、複雑さの要因など、アプリケーションのさまざまなビジネス指標に影響します。
以下では、scipyとShapelyライブラリを使用したPythonでのアルゴリズムの実装例を示します。
ジオコーディングにはGoogleマップを使用します。機能性と使用コストの両方に適しています。この記事の執筆時点では、Googleは毎月最初の20,000のジオコーディングリクエストを無料で行うことを許可しています。

方法1.ポリゴンの頂点に基づいてルートを計算します


モスクワ地域のどこかからモスクワ環状道路までの距離を見つける必要があるとします。幾何学的なパスではなく、実際のパスが必要です。したがって、最初にモスクワ環状道路の出口ポイントから埋め立て地を建設しますが、それらはマップ上の道路の輪郭の頂点と一致しません。

exit_coordinates: List[Tuple[float, float]]
latitude: float
longitude: float

ジオメトリを操作するには、シェイプライブラリを使用します。その助けを借りて、関心のあるポイントがポリゴンの内側にあるかどうかを簡単に判断できます。そうであれば、距離は明らかに0です。

from shapely.geometry import Polygon, Point
polygon = Polygon(exit_coordinates)
polygon.contains(Point(latitude,longitude))

ポイントがポリゴンの外側にある場合に関心があります。出発点に最も近いオブジェクト（モスクワ環状道路からの出口）を見つける作業は、数学では非常に一般的です。通常は、KDツリーを使用して解決されます。じゃあやってみましょう。Pythonでは、ツリーはscipyライブラリに実装されています。モスクワ環状道路の出口から私たちのポイントに最も近いピークが見つかります。

from scipy.spatial import KDTree
kd_tree = KDTree(exits_coordinates)
dists, indexes = kd_tree.query((latitude, longitude), k=3)
nearest_coordinates = list()
for _, index in zip(dists, indexes):
		nearest_coordinates.append(exits_coordinates[index])

k の数-ピークの数-は、モスクワ環状道路からの最も近い出口が一時的に遮断される可能性があるため、小さすぎないようにしてください。または、川の向こう側にあり、私たちのポイントへの直接の経路がない場合があります。kが大きすぎても、私たちには役に立たない。適切な出口がいくつかの近くの山頂にあります。それでは、Googleマップサービスを見てみましょう。点の配列から特定の点までの距離を見つけるための機能、Distance Matrix APIがすでにあります。
Upd：Distance Matrix APIサービスは、最も近い各ピークへのルートを個別に請求します。したがって、この方法は、次に説明する2番目の方法よりも高価です。感謝ソヴェトニコフ

import googlemaps
gmaps_client = googlemaps.Client(key=settings.GOOGLE_MAPS_API_KEY)
gmaps_client.distance_matrix(
    origins=nearest_coordinates,
    destinations=(latitude, longitude),
    mode='driving',
)

解答しか解析できません。通常、複数のルートがあり、最初のルートが常に最短であるとは限りません。したがって、適切な値を選択します。

方法2.埋め立て地の中心からのルートを計算する


次に、ポリゴンの頂点に加えて、ポリゴンの内部に特定のポイントを定義し、そこからGoogleマップのすべてのルートを作成します。ルートAPIを使用してルートを作成し、最短のものを選択します。

start_point: Tuple[float, float],
destination: Tuple[float, float]
polygon: shapely.geometry.Polygon
gmaps_client = googlemaps.Client(key=settings.GOOGLE_MAPS_API_KEY)
driving_path = gmaps_client.directions(start_point, destination)

セグメントの最初の座標がポリゴンの内側になるまで（構文解析は省略されます）、最後から繰り返してパスセグメントの長さを追加します。

for step in reversed(driving_path):
    start_point = step['start_location']['lat'], step['start_location']['lng']

        end_point = step['end_location']['lat'], step['end_location']['lng']

ポリゴンの外にあるセグメントの一部しか追加できません。これを行うには、形状ライブラリを使用して、パスの始点と終点の座標間に幾何学的な線を作成し、その長さがポリゴンの外側にどれだけあるかを調べます。同じ長さのパーセンテージが、取得されたパスのセグメントに対して計算されます。
パスは、実際の曲率を持つ実際の道路上の地形ルートです。これが長い直線（大通りまたは高速道路）である場合、近似により、ルートをパーセンテージで正確に計算できます。
埋め立て地を横切る道路が十分にカーブしている場合、そのような概算は正しくありません。しかし、Googleマップルート自体の曲がった部分は通常、長さが短く、計算のエラーは結果に影響しません。

二部。リバースジオコーディングリクエストを保存


多くの場合、近くにあり、現実世界の1つのオブジェクトに対応するポイントのアドレスを決定する必要があります。毎回、外部ジオコーディングシステムへのリクエストはクールではなく、ここでキャッシュの問題が合理的に発生します。
通常、クライアントは、59.80447691、30.39570337などの過剰な精度で座標を送信します。しかし、小数部分の標識がいくつあるかが私たちにとって重要でしょうか？
怠惰で急いでいる場合、答えは4つです。それ以外の方は、以下の説明をご覧ください。

まず、少し地理。


  • 赤道の長さは40,075.696 kmで、緯度はゼロです。
  • 子午線は緯度の線に垂直な経度の線です。子午線の長さは40,008.55 km
  • 緯度-40,008.55 km / 360 = 111.134861111 km。したがって、100分の1は約1キロメートルの変化を与え、1万分の1は10メートルの変化を与えます。
  • , , , 60 ( -) 2 . — 40 075,696 * 0.5 / 360 = 55,66066888 , — 5 .

1/10000度のエラーは、5〜10 x 10メートルのエラーの長方形を示します。これにより、建物に正確に「入る」ことができます。建物はポイントよりもはるかに大きいです。エラーのために、ポイントが建物に該当しない場合でも、Googleはそれに最も近い場所を決定します。
3文字に切り上げるのは危険であり、精度はすでに大幅に低下しています。そして5つまで-スマートフォンのGPS精度は通常数メートルを超えないため、これは意味がありません。

したがって、「address：59.8045,30.3957」というキーを持つキャッシュに値がある場合、すべての座標は、4文字に丸められたときに、同じキーが取得され、1つのジオコーディングされた住所に対応します。リクエストの数が減ります-ジオコーディングサービスを使用することで、作業が速くなり、料金が安くなります。利益！
