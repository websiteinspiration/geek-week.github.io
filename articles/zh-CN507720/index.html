x86_x64如何寻址内存

今天，我将讨论内存寻址：x86_64体系结构命令的语义似乎很小，但是却非常困难。特别是，我想谈一谈该命令mov



，以及仅通过此 x86_64 命令如何使各种内存寻址方法对用户可用。



我不会谈论其余影响内存的命令（也就是说，感谢CISC，几乎所有其他命令），这些命令会写入大量的内存片段（这是关于您的，fxsave



）或其他相关问题（代码模型，与地址无关的代码和二进制重定位）。当x86_64处理器未处于64位模式（即除具有64位代码的长模式之外的任何模式）时，我也不会涉及历史寻址模式或活动模式。



一些限制



尽管编写x86_64命令是噩梦，但也许要感谢它，内存寻址还是有一些限制。



让我们从好的开始：



  • 在相当高的水平上，x86_64体系结构中只有两种寻址模式。
  • 两种寻址模式下的所有寄存器的大小都必须完全相同。换句话说，我们不能奇怪地混合使用64位，32位和16位寄存器并获得当前地址-x86_64编码中根本没有这样的作法。


否则，一切都不会那么乐观：



  • 这两种寻址方式之一仍然非常复杂。
  • 寄存器的大小应相同，但不必与处理器模式的位深度相同。例如，如果在编码中添加前缀字节（0x67



    ），则可以使用32位寄存器而不是64位寄存器。


解决“比例指数基准位移”



我不知道如何称呼这种模式，所以我称其为“ Scale-Index-Base-Displacement”。据我明白，无论是英特尔也不AMD 在所有感知为单一模式，而是提到它为一组典型的具有宽范围的不同的编码方法连接模式。



但是今天，我们不是在谈论编码，而是在谈论语义，在语义上，这三种连接模式中的每一种都简化为以下四个参数中的几个的组合：



  • 比例尺：2位常数系数，通常为1、2、4或8。
  • 索引：任何通用寄存器（rax，rbx和c）。
  • 基数：任何通用寄存器。
  • 排量：积分移位。通常，即使在64位模式下，它也被限制为32位，但是使用某些编码方法，它也可以是64位。我们将讨论更多。


这四个参数可以组合为几种不同的组合，完整列表按复杂度从高到低的顺序显示如下：



  • Displacement



  • Base



  • Base + Index



  • Base + Displacement



  • Base + Index + Displacement



  • Base + (Index * Scale)



  • (Index * Scale) + Displacement



  • Base + (Index * Scale) + Displacement





让我们按顺序查看每种组合。

最后：细分



x86_64体系结构删除了所有细分，但几乎删除了所有细分。由于地址空间平坦，因此不再需要段寄存器，但仍会出现在以下位置：



  • Linux（更确切地说是glibc）fs



    在用户空间中使用以访问由内核配置的TLS段。这些段的规范可以在每CPU GDT配置（链接）中找到。假设没有东西使用glibc（或您的任何libc）gs



    ，则可以自由使用它。
  • 在内核空间中，Linux使用gs



    每个CPU变量来存储区域的主地址。我们可以在宏定义PER_CPU_VAR



    （link）中观察到这一点：
    #define PER_CPU_VAR(var)  %__percpu_seg:var
  • 在x86_64中，定义增长：
    %gs:var


因此，不幸的是，我们仍然必须注意细分。另一方面，这并不是那么困难，因为全部归结为从段寄存器中添加一个值来计算整个地址。



顺便说一下，这是一个局部线程变量的示例：

int foo(void) {
    int *y = &x;
    return *y;
}


结果：

lea     rax, [rax + x@TPOFF]     ; calculate the effective address of x within the TLS
mov     qword ptr [rbp - 8], rax ; store the address of x into y
mov     rax, qword ptr [rbp - 8]
mov     eax, dword ptr [rax]
pop     rbp
ret
