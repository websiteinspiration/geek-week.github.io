SSHを使用しない方法


2004年に、誰かが公開鍵をファイルにコピーする方法を教えてくれましたauthorized_keys。それ以来、同じ古い公開鍵を無害に各サーバーにコピーし続けましたが、正しく構成するのを忘れていたため、これを初めて行うことはできませんでしたchmod。

一度だけ、2015年にOpenSSHが私のキータイプを非推奨と宣言したとき、私はキーを置き換えるのをためらっていました。今、数年前、私の公開鍵がまだどのサーバーにあるかを推測することしかできません。たぶん、以前使っていた古いスタートアップは？フリーランスのお客様はいますか？私の姪のクリプトファームでは？

組織の観点から見ると、ファイルはauthorized_keys厄介です。多分最初は誰かが私と同じように手動で書いたかもしれませんが、ホストとユーザーの数が増えると、そのような作業は悪夢に変わります。そしてある朝、あなたはインフラストラクチャーに引きずり込まれた公開鍵の沼地の真ん中で目を覚ます。

ほとんどの組織は通常、ファイル管理を自動化するためのハンドブックを作成します。authorized_keysただし、実行の合理性に関係なく、それらの収集、アンロード、およびメンテナンスは混乱のままです。

SSH証明書はTLS証明書よりも単純です。 SSH CAを使用すると、複数のホストを認証および承認する責任の一部を1つの集中型サービスに委任できます。

このプロジェクトでは、3つのプロバイダー（証明書を発行するための3つの方法）でCAをセットアップします。

  • ユーザー証明書については、Google OAuthアプリケーションに対応するOAuth OIDCプロバイダーを使用します。私たちはGoogleが信頼して人にアクセスし、認証されたユーザーにGoogleが署名した有効なOAuth認証トークンのみを発行することを信頼しています。CAはGoogleの公開鍵でトークンを検証します。
  • 新しいEC2ホストの場合、 AWSアカウントに一致するAWSプロバイダーが必要です。新しいホストは、このプロバイダーを通じてSSHホスト証明書を要求します。AWSは、Amazonの署名付きIIDの提供を信頼しており、CAは、Amazonの公開鍵を使用してそれらを検証します。
  • 最後に、毎週のホスト証明書の更新にSSHPOPプロバイダーを使用します。

最初のユーザーアカウントを設定しましょう。ローカルデバイスで以下を実行します。

その後、ユーザー証明書を自分で発行できます。

$ step ssh login [your email address] --provisioner "Google"

このチームは何をしますか？

  1. システムブラウザを起動し、Google OICD承認プロセスを起動します。
  2. 承認後、Googleから受け取ったトークンがCAに送信されます。
  3. CAはトークンを確認し、メールに対応するSSH証明書を発行します。
  4. 証明書がSSHエージェントに追加されます。それを見るには、それを使用しstep ssh list、それを解析して研究しstep ssh list --raw | step ssh inspectます。

最後に、CAを使用するようにSSHを構成しましょう。

$ step ssh config

このコマンドは次のことを行います。

  1. 最初に、彼女はCAホストの公開鍵を取得します-これはホスト証明書を検証するために使用されます-この鍵をSSH構成にインストールすると、次の結果が得られます（に保存されます~/.step/known_hosts）：

    @cert-authority * ecdsa-sha2-nistp256 AAAAE2VjZHNhLXNoYTItbmlzdHAyNTYAAAAIbmlzdHAyNTYAAABBBHyWTo9TLDwhyLlHq2ANkjtSGLyJ3xPtfL+7faiV+YA0k4kLUc8cJ5rWFHdUShbwtkOsLhkqeDDXoFnH/C5BG+4=
  2. 次に、SSH構成（c .step/ssh/config）にこの控えめなブロックをインポートします。

    Match exec "step ssh check-host %h"
		ForwardAgent yes
		User carl
		UserKnownHostsFile "/Users/carl/.step/ssh/known_hosts"
		ProxyCommand step ssh proxycommand %r %h %p --provisioner "Google"

Match exec上記のコマンドがtrueを返した場合にのみ、 ブロックが考慮されます。

このコマンドはCAに連絡し、SSH経由で接続しようとしているホスト名に適切な証明書があるかどうかを尋ねます。そうである場合、コマンドはtrueを返します。 CAが管理するホストがsconfig内の他のホストに干渉することは望ましくありません。

このディレクティブProxyCommandは、シングルサインオンプロセスのすべての魔法を実装します。接続を開始する前にstep ssh proxycommand、SSH証明書を検証し、SSH証明書がないか期限切れの場合は、OAuth OIDCプロセスを案内し、CAに新​​しい証明書を要求します。

質疑応答


CAに既存のホストを登録するにはどうすればよいですか？


Ubuntu 18.04 LTSホストの場合、このプロセスは新しいホストのロードに似ています。ルートの下でホスト起動スクリプトを実行するだけです。他のプラットフォームでは、スクリプトをニーズに移植する必要があります。

すべてのホストを起動して構成したらinstanceAge、CAのAWSプロバイダー設定を微調整したい場合があります。これらの設定はに保存され/etc/step-ca/config/ca.jsonます。あらゆる年齢のインスタンスは、デフォルトで自身を実行できます。たとえば、設定をに設定5mすると、追加の予防策が作成され、CAは事実上、5分未満のインスタンスからのIIDトークンのみを受け入れるようになります。

ノート：インスタンスIDスクリプトによって作成されたトークンは、ワンタイムトークンです。ホストを再登録することはできず、その証明書のみを更新できます。どうして？ホスト上のユーザーはいつでもIIDにアクセスでき、ユーザーがCAからホスト証明書を取得できないようにするためです。

別のOAuth OIDCプロバイダーを使用できますか？


もちろん！これを行うには、OAuth構成サーバーのネットワークアドレスを変更し、CAのユーザーデータスクリプトでOIDCプロバイダーの名前を変更するだけです。

AWSの代わりにGCPまたはAzureを使用できますか？


はい！step-ca3つすべてのプロバイダーがあり、CA構成とホストブートスクリプトを変更するだけで済みます。詳細については、サプライヤーのドキュメントを ご覧くださいstep-ca。

ホストの要塞（インストールサーバー）を使用できますか？


あなたはできる。まず、すべてのホストに証明書を発行し、次にknown_hostsホストキーをCA要塞に追加します。

$ CA_URL=https://[YOUR CA].compute.amazonaws.com
$ CA_FINGERPRINT=[CA FINGERPRINT]
$ step ca bootstrap \
      --ca-url $CA_URL \
      --fingerprint $CA_FINGERPRINT
$ mkdir -p ~/.ssh && echo "@cert-authority * $(step ssh config --host --roots)" > ~/.ssh/known_hosts

その後は、ssh要塞、そしてssh内部ホストに十分です。

SSH証明書の有効期限を変更できますか？


はい。デフォルトでは、これらは16時間有効ca.jsonで、オブジェクトの下のCA（）構成ファイルのclaimsOIDCプロバイダーで、対応する行を変更できます。次に例を示します。

"defaultUserSSHCertDuration": "720h",
"maxUserSSHCertDuration": "720h"

CAを切断すると、すべてのホストから切断されますか？


ただし、SSHおよびSHHDはCAと直接対話しません。CAは証明書を発行し、ホストデータベースを維持します。ユーザーはこのベースに基づいて、証明書認証を介してホストに接続する必要があるかどうかを判断します。

システムセキュリティのため、SSH証明書を使用してCAに直接接続しないでください。これは、キーペアを引き続き使用する必要がある唯一の場所です。

複数のAWSアカウントでホストを使用できますか？


もちろん。これを行うには、CA（/etc/step-ca/config/ca.json）構成ファイルにアカウントIDを追加し、CAサーバーを再起動します。

サポートを追加するにはどうすればよいsudoですか？


エージェント転送と認証プラグインを使用し pam_usshます。

CAで他に何ができますか？


たくさん！個人のCAができたので、TSL証明書を構成して、共有TLSを介してすべての内部トラフィックを暗号化できます。CA、ベンダー、構成オプションなどの詳細については、ステップ証明書のドキュメントをご覧ください。
