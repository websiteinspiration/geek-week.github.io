Reactは引き続きリードし、開発者はフックに情熱を傾けています


VueとAngularには専用のユーザーコミュニティがあります。 Vueは、GitHub上の星の数でReactよりも優れています。しかし、実際の使用に関しては、それがプログラマーの個人的なプロジェクトであろうと、彼らの仕事であろうと、Reactは自信を持って最初の地位を保持しています。

2018年の終わりに、Reactチームはいわゆるフックをリリースしました。 2019年、フックは文字通りReactの世界を捉えました。 React開発者の大多数は、コンポーネントの状態を管理し、ライフサイクルでイベントを処理するための主要なメカニズムとしてそれらを使用しています。 1年の間に、フックに関する無数の記事が書かれ、その使用パターンが明らかになり、最も重要なReactパッケージは、これらのパッケージでの作業に使用できるカスタムフックを提供し始めました。

フックは、プログラマーに、シンプルで簡潔な構文を使用して機能コンポーネントの状態とライフサイクルを管理する方法を提供します。さらに、Reactでは、開発者が再利用に適したコードを記述したり、一緒に使用できるロジックを作成したりできるカスタムフックを作成できます。これにより、高次コンポーネントやレンダリングプロパティが不要になります。

React開発チームは、プログラマーとツールの利便性に特に注意を払い、より生産的な作業を支援します。 React 16.8でフックが大声で登場した後、2019年にライブラリがバージョン16.4になったため、他のイノベーションはそれほど大きくありませんでした。

フックのリリース後のReact開発チームは、補助ツールを作成することによってプログラマーの利便性に焦点を合わせたと言えます。プログラマーの利便性、つまりReactアプリケーションの開発中に感じる感覚は、React Conf 2019カンファレンスのメイントピックになりました。ReactチームリーダーでReactチームマネージャーであるTom Ochchino氏は、Reactの使用に関する開発者の気持ちは3つのアイデアに基づいていると述べています：低入力障壁、高い生産性、ソリューションの拡張性。これらのアイデアをサポートするために、Reactチームによってリリースされた（またはリリースされる予定の）内容を見てください。

Svelteフレームワークは多くの注目を集めていますが、まだ広く使用されていません。


Svelteフレームワークは、すでに混雑しているフロントエンドフレームワークの世界でその位置を見つけることができました。ただし、すでに述べたように、これはまだこのフレームワークの広範な使用につながっていません。スヴェルトの本質を一言で表すと、これは「シンプルだが強力な」ツールであると言えます。プロジェクトのウェブサイトには、3つの重要な長所があります。

  • Svelte開発者は、より少ないコードを記述します。
  • フレームワークは仮想DOMを使用しません。
  • 彼は本当に反応的です。

Svelteは、ページの動作中にブラウザーで何かを行うのではなく、作業の大部分をコンパイル段階に移行しようとしています。 Svelteのコンポーネントベースのアーキテクチャは、その構造が純粋なHTMLとJavaScriptにコンパイルされています。これは、より少量のボイラープレートコードの必要性を約束します。フレームワークはリアクティブプログラミングのアイデアを使用しています。これにより、仮想DOMを変更する代わりにDOMに直接変更を加え、その変更を実際のDOMと調整することができます。

Svelteは、フロントエンドの開発者に何か新しくて興味深いものを提供します。2020年には、スヴェルトの成長と発展を観察することが興味深いでしょう。大規模プロジェクトでの活用例が出てくることを期待しています。これにより、より大きな競合他社-React、Vue、Angularと比較した場合の外観がわかります。

静的サイトはまだ関連性があり、開発者はJAMStackを導入しています


静的サイトは、それらがWebの一部であり、ポジションを放棄しないことを証明しています。この声明は、Gatsbyのようなフレームワークの使用の拡大、Netlifyのような静的サイトのホスティングの急速な成長、ユーザーインターフェイスなしでCMSを作成している無数の新しい企業によってサポートされています。

静的サイトは、すでにウェブ上にあったものを新しいツールやライブラリと組み合わせます。彼らはあなたが他のものとは比較にならないプロジェクトを作成することができます。 Reactのような最新のライブラリを使用してサイトを作成し、プロジェクトのビルド中にそれらを静的なHTMLページにコンパイルできます。これですべてのページが既製で保存されるため、クライアントがアクセスしたときにサーバーがリソースを費やしてデータをページに埋めることはありません。ページはクライアントに即座に配信でき、プロジェクトはCDNキャッシングの利点を最大限に活用できます。これにより、サイトのコンテンツを可能な限りユーザーに近づけることができます。

CSS-in-JSテクノロジーが人気を集めています


Web開発の発展を観察すると、テクノロジーがJavaScriptの兆候の下ですべてを完全に組み合わせる方法の邪魔になっているように感じるかもしれません。これは、JavaScript文字列に基づくスタイル構築手法を使用するCSS-in-JSテクノロジーの実装でも明らかです。

これにより、スタイルや依存関係を操作できますimport/export。たとえば、通常のJavaScriptコンストラクトを使用してそれらを共有できます。さらに、CSS-in-JSツールを使用してスタイル設定されたコンポーネントには、スタイルを説明する行に渡されたプロパティの値を含めることができるため、これにより動的スタイルの作成が簡素化されます。すでに述べたように、フェイスブックのビジョンではCSS-in-JSがスタイリングの未来である可能性が高いため、会社は独自のデザインの適切なライブラリをリリースする可能性があります。

以下は、クラシックCSSとCSS-in-JSを比較した例です。通常のCSSで動的なスタイルをサポートするには、クラス名を持つコンポーネントで作業し、コンポーネントの状態またはプロパティに基づいてそれらを更新する必要があります。さらに、スタイルのバリエーションごとに独自のCSSクラスが必要です。

// JS- 
const MyComp = ({ isActive }) => {
  const className = isActive ? 'active' : 'inactive';
  return <div className={className}>HI</div>
}
// CSS-
.active { color: green; }
.inactive { color: red; }

CSS-in-JSを使用する場合、CSSクラスを操作する必要はなくなりました。対応するプロパティを様式化されたコンポーネントに渡すだけで十分であり、コンポーネントは宣言構文によって動的なスタイルを提供します。動的なスタイリングの制御はコンポーネントに渡されるプロパティに基づいているため、コードははるかにクリーンで、スタイルとReactの機能が明確に分離されています。これで、スタイルコードは、React用に作成された通常のJSコードのように読み込まれます。

const Header = styled.div`
  color: ${({ isActive }) => isActive ? 'green' : 'red'};
`;
const MyComp = ({ isActive }} => (
  <Header isActive={isActive}>HI</Header>
)

styled-componentsと感情 ライブラリは、2つの主要なCSS-in-JSソリューションです。同時に、2019年の感情は、ダウンロード数の点でstyled-componentsよりも優れていました。これら2つのライブラリーは、人気の点で、CSS-in-JSテクノロジーを実装する他のツールを大幅にバイパスします。彼らの急成長が続く気がします。
