教堂



Chapel使用forall循环来并行化线程。还使用了每个元素的C指针，而不是对数组的标准引用，并且应用了由索引引导的迭代： 



proc calculateKernelMatrix(K, data: [?D] ?T)
{
  var n = D.dim(0).last;
  var p = D.dim(1).last;
  var E: domain(2) = {D.dim(0), D.dim(0)};
  var mat: [E] T;
  var rowPointers: [1..n] c_ptr(T) =
    forall i in 1..n do c_ptrTo(data[i, 1]);

  forall j in guided(1..n by -1) {
    for i in j..n {
      mat[i, j] = K.kernel(rowPointers[i], rowPointers[j], p);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


教堂代码最难以优化性能，并且需要进行最多的代码更改。



d



为了并行化D代码，使用了来自std.parallel包的taskPool流。D代码进行了最少的更改以优化性能-使用特定的编译器和选定的编译键（稍后讨论）非常有益。我的Matrix实现使我可以使用refColumnSelect通过引用选择列。 



auto calculateKernelMatrix(alias K, T)(K!(T) kernel, Matrix!(T) data)
{
  long n = data.ncol;
  auto mat = Matrix!(T)(n, n);

  foreach(j; taskPool.parallel(iota(n)))
  {
    auto arrj = data.refColumnSelect(j).array;
    foreach(long i; j..n)
    {
      mat[i, j] = kernel(data.refColumnSelect(i).array, arrj);
      mat[j, i] = mat[i, j];
    }
  }
  return mat;
}


朱莉亚



朱莉娅代码使用宏 线程数 用于并行化代码和宏 意见用于引用数组。与Julia中的数组混淆的唯一事情是它们的引用状态。有时，在这种情况下，数组将充当值对象，并且必须使用宏对其进行引用意见否则，它们将生成副本。在其他情况下，例如，当将它们传递给函数时，它们的行为类似于参考对象。处理起来可能有些棘手，因为您并不总是知道副本将生成哪组操作，但是它会在哪里发生意见提供了一个很好的解决方案。



对称类型节省了一些在矩阵中反映所需的额外工作。 



function calculateKernelMatrix(Kernel::K, data::Array{T}) where {K <: AbstractKernel,T <: AbstractFloat}
  n = size(data)[2]
  mat = zeros(T, n, n)
  @threads for j in 1:n
      @views for i in j:n
          mat[i,j] = kernel(Kernel, data[:, i], data[:, j])
      end
  end
  return Symmetric(mat, :L)
end


主要功能中 的@ bounds和@ simd宏用于禁用边界检查并将SIMD优化应用于计算： 



struct DotProduct <: AbstractKernel end
@inline function kernel(K::DotProduct, x::AbstractArray{T, N}, y::AbstractArray{T, N}) where {T,N}
  ret = zero(T)
  m = length(x)
  @inbounds @simd for k in 1:m
      ret += x[k] * y[k]
  end
  return ret
end


这些优化带来了明显的提高，但是非常易于使用。

性能优化



在所有三种语言中优化性能的过程非常不同，所有三个社区在此过程中都非常有用。但是有一些普遍的观点。



  • 内核函数的静态调度，而不是使用多态。这意味着在传输内核函数时，将使用参数（编译时）多态性，而不是动态（运行时）多态性，在这种情况下，使用虚拟函数进行调度会产生开销。
  • 使用视图/链接而不是在多线程模式下复制数据非常重要。
  • 计算的并行化非常重要。
  • 知道数组是行/列的基础，并在计算中使用它至关重要。
  • 边界检查和编译器优化有很大的不同，尤其是在Chapel和D中。
  • D和Julia中包含SIMD有助于提高性能。在D中，这是使用-mcpu = native标志完成的，而在Julia中，这是使用@ simd宏完成的。


从语言细节的角度来看，最困难的是在Chapel中过渡到高效的代码，而Chapel中的代码变化最大：从简单的数组读取操作到指针的使用和受控迭代的使用。但是在编译器方面，添加--fast相对容易，并且可以大大提高性能。



D代码变化很小，大多数性能是通过选择编译器及其优化标志获得的。LDC编译器具有丰富的性能优化功能。它具有8-O优化级别，但其中有些重复。例如，-O，-O3和-O5相同，并且还有无数其他影响性能的标志。在这种情况下，将使用标志-O5 --boundscheck = off -ffast-math，这些标志是积极的编译器优化，边界检查和LLVM的fast-math，而-mcpu = native可启用矢量化指令。



在Julia中，先前审查的更改中的宏显着提高了性能，但它们并不太令人困惑。我尝试更改优化级别-O，但这并没有提高性能。

互动性



程序员在开发过程中需要快速的编码/编译/结果周期，以便快速监视结果和结论，以便继续前进或进行必要的更改。为此，Julia解释器是最好的选择，它提供了一种平滑且多功能的设计，而D与此接近。即使编译少量代码，此编码/编译/结果循环也可能很慢。 D中有三种编译器：标准DMD编译器，LLVM LDC编译器和GCC GDC编译器。在此开发过程中使用了DMD和LDC编译器。 DMD编译速度非常快，这对开发非常方便。 LDC在创建快速代码方面做得很出色。与之相比，Chapel编译器非常慢。例如，在没有优化的情况下为DMD编译器和Chapel运行我们的矩阵代码的Linux时间。这给了我们D：

文档和示例



比较不同语言文档的一种方法是将所有文档与官方Python文档进行比较，这是编程语言的黄金标准。它以简单且用户友好的方式将示例与正式定义和说明结合在一起。由于许多程序员都熟悉Python文档，因此这种方法可以使他们了解如何进行比较。



Julia文档在质量上与Python文档最接近，并且为用户提供了一种非常流畅，详细且相对轻松的语言过渡。它还具有丰富的博客生态系统，并且可以轻松访问有关该语言许多方面的主题。 D的官方文档不是很好，并且可能会令人感到复杂和令人失望，但是有一本非常不错的免费书籍“ Programming in D”，这是对该语言的出色介绍，但没有哪本书能涵盖整个编程语言，并且没有很多高级主题的源代码示例。尽管提供的示例在可用性和质量方面各不相同，但Chapel文档足以胜任某些工作。程序员通常需要知道在哪里看。一个很好的比较主题是Chapel，D和Julia中的文件I / O库。 Chapel I / O库包含的示例太少，但相对清晰和简单； D输入/输出分布在多个模块中，文档更难遵循； Julia的I / O文档包含许多示例，并且清晰易懂。



影响Chapel采用率的因素之一可能是缺少示例-由于数组具有非标准接口，因此用户必须努力查找。尽管D语言中的文档在某些地方可能不是很好，但该语言与C和C ++有许多相似之处，因此它摆脱了更为稀缺的文档。



多维数组支持



这里的“数组”不是指D中可用的C和C ++样式的数组，而是数学数组。 Julia和Chapel带有数组支持，但D没有，但它确实有一个包含多维数组（ndslice）的Mir库。在计算矩阵核的实现中，我用D编写了矩阵对象，如果您了解原理，这很容易，但这不是用户想要做的。但是，D具有线性吕贝克代数库，它具有令人印象深刻的性能特征，并且与所有常见的BLAS实现都具有接口。数组Julia当然是最简单和最熟悉的。Chapel阵列在入门级上比Julia阵列更为复杂，但它们设计为使用单个或非常相似的代码在单核，多核系统和计算机集群上运行，这是一个很好的独特之处。



舌头力量



由于Julia是一种动态编程语言，因此有人可能会说：“嗯，Julia是一种动态语言，比静态编程语言更宽松，因此辩论结束了。”但它要复杂得多。静态类型的系统有其强大的功能。 Julia的类型系统本质上与静态语言的类型系统相似，因此您可以像使用静态语言一样编写代码，但是只能执行动态语言保留的操作。它具有用于常规和元编程的高级语法，以及功能强大的宏。它还具有非常灵活的对象系统和多个调度。功能的组合使Julia成为三种语言中功能最强大的一种。



D被认为是C ++的替代品，它从C ++中汲取了很多（并且也从Java借来了），但是D使得模板编程和编译时计算（CTFE）比C ++更加易于使用。这是一种具有单调度的语言（尽管有一个包含多种方法的程序包）。D具有宏，而不是宏，它具有用于类似目的的字符串和模式的“ mixin”。



Chapel通过单调度对OOP提供了通用支持和新生支持，没有宏观支持，在这些方面还不如D或Julia成熟。

C互操作性易于在所有三种语言中使用；教堂有很好的文献资料，但没有其他人欢迎。D文档更好，Julia文档最全面。但是，奇怪的是，没有一种语言文档具有编译您自己的C代码并将其与该语言集成所必需的命令，这是一个疏忽，特别是对于初学者而言。但是，在D和Julia中，很容易搜索和查找编译过程的示例。

用户可以在这三种语言的方便位置提问。对于Chapel来说，最简单的地方是Gitter，对于Julia来说，这是Discourse（尽管也有Julia Julia Gitter），对于D来说，这是官方论坛的站点。朱莉娅社区是最活跃的，其次是D，然后是教堂。我发现您从这三个社区中都得到了很好的答案，但是对于D和Julia来说，您可能会得到更快的答案。

摘要



如果您是编写数值算法并执行科学计算的初学者程序员，并且想要一种快速且易于使用的语言，那么Julia是您的最佳选择。如果您是在同一领域工作的经验丰富的程序员，Julia仍然是一个不错的选择。如果您希望使用带有所有“告密者”的更传统的，“工业”，静态编译的高性能语言，但是想要比C ++更高效，更安全且更省力的东西，那么D是您的最佳选择。您可以在D中编写“任何内容”，并借助其编译器获得出色的性能。如果需要在群集上计算数组，那么Chapel可能是最方便的解决方案。



就这项任务的总体表现而言，D是获胜者，显然证明了9项参考任务中有5项表现最佳。研究表明，朱莉娅作为一种高性能语言的标签不只是炒作-与竞争激烈的语言相比，它具有自己的优势。很难从Chapel获得竞争性能-Chapel团队需要大量研究才能提出当前的解决方案。但是，随着教堂语言的成熟，我们将能够看到进一步的改进。
