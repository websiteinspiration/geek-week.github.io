EcmaScriptのLexicalEnvironmentとClosures

こんにちは、ハブル！

私は長い間何も書いておらず、この数週間、プロジェクトに多くの作業をしていましたが、今は自由な時間があるので、新しい記事を紹介することにしました。

今日は、引き続きEcmaScriptの主要な概念を分析し、字句環境とクロージャについて説明します。字句環境の概念を理解することは、クロージャを理解する上で非常に重要であり、クロージャは、JSの世界（EcmaScript仕様に基づく）における多くの優れたテクニックとテクノロジーの基盤です。

それでは始めましょう。

字句環境（LexicalEnvironment、LO、LE）


公式のES6仕様では、この用語を次のように定義しています。
字句環境は、ECMAScriptコードのネストの字句構造に基づいて特定の変数および関数を検索するときに識別子名を解決するために使用される仕様のタイプです。字句環境は、環境のレコードと、場合によっては外部字句環境へのnull参照で構成されます。
詳しく見てみましょう。

字句環境は、コンテキスト識別子とその意味のつながりを格納する一種の構造体と想像します。これは、このコンテキストのスコープで宣言された変数、関数、クラスの一種のリポジトリです。

技術的には、LOは2つのプロパティを持つオブジェクトです。

  • 環境レコード（すべての広告が保存される場所です）
  • 生成コンテキストのLOへの参照。

現在のコンテキストの親コンテキストへのリンクを介して、必要に応じて、親コンテキストの「祖父コンテキスト」へのリンクを取得できます。グローバルコンテキストへのリンクも取得でき、その親への参照はnullになります。この定義から、字句環境はエンティティとそれを生成したコンテキストとの接続であることがわかります。関数の一種のScopeChainは、字句環境の類似物です。この記事では、ScopeChainについて詳しく説明しました。

let x = 10;
let y = 20;
const foo = z => {
  let x = 100;
  return x + y + z;
}

foo(30);// 150.   foo    {record: {z: 30, x: 100}, parent: __parent};
// __parent      {record: {x: 10, y: 20}, parent: null}

技術的には、識別子名を解決するプロセスは、ScopeChainと同様に発生します。LOループ内のオブジェクトの順次ポーリングは、目的の識別子が見つかるまで行われます。識別子が見つからない場合、ReferenceError。

字句環境は、コンテキストを作成する段階で作成され、満たされます。現在のコンテキストの実行が完了すると、コールスタックから削除されますが、そのレキシカル環境は、少なくとも1つのリンクが存在する限り存続します。これは、プログラミング言語の設計に対する最新のアプローチの利点の1つです。話す価値があると思います！

スタック構成と動的共有メモリ


スタック言語では、ローカル変数はスタックに格納され、関数がアクティブになると補充されます。関数が終了すると、そのローカル変数はスタックから削除されます。

スタック構成では、関数からローカル関数を返したり、関数をフリー変数に呼び出したりすることはできません。

フリー変数は関数で使用される変数ですが、この関数の仮パラメータでもローカル変数でもありません。

function testFn() {
  var locaVar = 10; //     innerFn
  function innerFn(p) {
    alert(p + localVar);
  }

 return innerFn; // 
}

var test = testFn();//  innerFn  
test();//

スタック構成では、外部LexicalEnvironmentでのlocaVar検索もinnerFn関数の戻りも不可能です。innerFnは、testFnのローカル宣言でもあります。testFnが完了すると、すべてのローカル変数がスタックから削除されます。

したがって、別の概念が提案されました-動的に割り当てられたメモリ（ヒープ、ヒープ）+ガベージコレクタ+参照カウントの概念。この概念の本質は単純です。オブジェクトへの参照が少なくとも1つある限り、オブジェクトはメモリから削除されません。詳細については、こちらをご覧ください。

クロージャは、コードブロックと、そのブロックが生成されるコンテキストのデータの組み合わせです。これは、LOまたはSopeChainのチェーンを介したエンティティと生成コンテキストとの関係です。このテーマに関する

非常に優れた記事を引用させてください。

function person() {
  let name = 'Peter';
  
  return function displayName() {
    console.log(name);
  };
}
let peter = person();
peter(); // prints 'Peter'

person関数が実行されると、JavaScriptは新しい実行コンテキストと関数の字句環境を作成します。この関数が完了すると、displayName関数が返され、変数peterに割り当てられます。

したがって、彼女の字句環境は次のようになります。

  environmentRecord: {
    name : 'Peter',
    displayName: < displayName function reference>
  }
  outer: <globalLexicalEnvironment>
}

  }

displayName関数には変数がなく、その環境レコードは空になります。この関数の実行中に、JavaScriptは関数の字句環境でname変数を見つけようとします。

displayName関数のレキシカル環境には変数がないため、外部レキシカル環境、つまり、メモリ内にあるperson関数のレキシカル環境を検索します。JavaScriptはこの変数を検出し、名前がコンソールに出力されます。
ESでのクロージャーの最も重要な特性は、静的スコープを使用することです（クロージャーを使用する他の多くの言語では、状況は異なります）。

例：

var a = 5;

function testFn() {
  alert(a);
}

(function(funArg) {
  var a = 20;
  funArg();//  5 ..  ScopeChain/LexicalEnvironment testFn   ,    = 5
})(testFn)

もう1つの重要なクロージャープロパティは次の状況です。

var first;
var second;

function testFn() {
  var a = 10;
  
  first = function() {
    return ++a;
  }

  second = function() {
   return --a;
  }

  a = 2;
  first();//3
}

testFn();

first();//4
second();//3

それら。いくつかの関数のクロージャに存在する自由変数は、それらによる参照によって変更されることがわかります。
