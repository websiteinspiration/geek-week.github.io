スレッドの優先順位

CPUハンガー理論は検証可能であるため、優れています。タスクマネージャーを使用してLockAppプロセスの優先度を上げることができました（UWPシステムによって中断されなかった短い期間のいずれかの間）。したがって、この投稿に使用した最後のトレースでは、LockAppが高い優先度で実行されました。通常のWindowsスレッドは、約8〜10の優先度で実行されます。通常の（非リアルタイム）Windowsスレッドを実行できる最高の優先度は15です。私のETWトレースは、LockAppが常に優先度13以上で機能することを示しました。

LockAppは最新のUWPアプリケーションであるため、スマートフォンアプリと同様の制限を受けます。とりわけ、これは、フォアグラウンドにないときに一時停止し、フォアグラウンドに戻ったときに「フリーズ解除」できることを意味します。James Forshawさんは、これに関するデータを得るためにMicrosoft-Windows-Kernel-Process ETWを記録することを提案しました。

イベントは、最大の混乱を引き起こすように設計されています。タスクの名前プロセスフリーズがされ、「解凍」と「凍結」、およびバージョンの両方に使用停止：勝利イベントプロセスは、（それが凍結停止している）の開始、およびのバージョンであることを意味勝利：スタートプロセスが停止する（フリーズし始める）ことを意味します。これはすべて非常に論理的ですが、非常に混乱します。イベント名をFreezeとThawに分割すると、混乱が少なくなります。

これらのイベントに関するドキュメントはありませんが、分析の結果、これらのイベントは常にバックグラウンドタスク/ブローカーインフラストラクチャサービスによって作成されると判断しました。対応するプロセスの名前とプロセスIDは、FrozenProcessIDフィールドに示されます。

残りのパズル


残念ながら、スリープをオンにする前にLockApp.exeがランク2に落ちる理由はまだはっきりしていませんが、LockAppは計画グループに属しており、おそらくアルゴリズムの1つが正しく動作しないと思います。しかし、これを調査するためのAPIが見つからず、時間がなくなりました。詳細がわかっている場合は、元の記事へのコメントを書き込んでください。ランクを計画決定の最も重要なコンポーネントとして使用するというまさにその原則は、システム内のほとんどのプロセスがそれに関与していない場合、必然的に失敗するはずです-計画グループのスレッドは常に、必要なリソースなしで残されるリスクを負います。ほとんどのスレッドが関与していない場合、動的リソース割り当て計画（DFSS）は失敗する運命にあります。

また、なぜ多くのアプリケーションがスリープ状態になった後もアクティブのままであるのかはわかりません。これは通常、「ラップトップが数時間スリープモードになっているときに多くのタイマーが終了する」という事実によって説明されますが、ラップトップが数秒しか夢中にならなかった場合、およびWPAレンダリング動作がウィンドウシステムで何かが発生したことを示している場合、この説明は適切ではありません。何か問題でも。その悪い動作のアプリケーションと待機サイクルドライバーに追加すると、すべてが時間の経過とともにCPUによってスタックされます。

CPUストームが停止し、同時にLockAppが起動するという事実は、明白な説明につながります。LockAppは、CPU需要が低下したときにのみ機能します。しかし、同様に説得力のある説明があります。LockAppが実行できるようになると（または、LogonUIがそれを取得すると）、CPU需要は低下します。どちらの説明も機能しますが、2番目の説明のほうがもっともらしいと思います。それ以外の場合、WPAの無限に見えるレンダリングが突然停止する理由を説明できないためです。

問題の解決策


LockApp.exeが起動に問題のある別のアプリケーションであり、優先度を上げても効果がないことに気づいたら、すぐに無効にしました。ファイルDisableLockScreen.regはこれを助けてくれました：

Windows Registry Editor Version 5.00
[HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\Personalization]
“NoLockScreen”=dword:00000001

ロック画面をオフにすると、カバーを開いた直後にラップトップが起動します。CPUのブレーキングやストームのどちらにも気づかなかったので、入るのに1ステップ少ない。最初のTwitterのポスト私は最初の問題に遭遇したとき、私は投稿は、誰かの役に立つかもしれ調査のためのタイムラインが含まれています。また、おかげさまでtwitterの頭のいい人がたくさん来てくれました。 記事に戻ると、ロック画面をオンにすると問題が解消したことがわかりました。単純に再起動しても問題は解決しませんでした。2月に何度か再起動しましたが、なぜ失われたのかはわかりません。
