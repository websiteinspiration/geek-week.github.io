RDPサービスに対するDDoS攻撃：認識して克服します。トゥチャの成功体験

「サードパーティ」がお客様の作業をどのように妨害しようとしたか、そしてこの問題がどのように解決されたかについてのクールな話をお伝えします。

すべての始まり


それはすべて、月の最終日である10月31日の朝に始まりました。

私たちのクラウドに、彼がサービスを提供するクライアントの複数の仮想マシンを保持しているパートナーの1人が、ウクライナのサイトで実行されているいくつかのWindowsサーバーが9:10から9:20に一度にリモートアクセスサービスとの接続を受け入れなかったと報告しました、ユーザーはデスクトップにアクセスできませんでしたが、数分後に問題は解決したようです。

通信チャネルの統計を上げましたが、トラフィックのバーストや障害は見つかりませんでした。コンピューティングリソースの負荷に関する統計を確認しました-異常はありません。そして、それは何でしたか？

次に、クラウドにさらに100台のサーバーを配置する別のパートナーが、一部のクライアントが指摘したのと同じ問題を報告し、一般にサーバーは使用可能（pingテストやその他の要求に正しく応答する）であることがわかりましたが、サービスこれらのサーバーへのリモートアクセスは、新しい接続を受け入れてから拒否しますが、異なるサイト上のサーバーの問題であり、トラフィックは異なるデータ伝送チャネルからのものでした。

そして、このトラフィックを見てみましょう。接続の確立を要求するパケットがサーバーに到着します。

xx:xx:xx.xxxxxx IP xxx.xxx.xxx.xxx.58355 > 192.168.xxx.xxx.3389: Flags [S], seq 467744439, win 64240, options [mss 1460,nop,wscale 8,nop,nop,sackOK], length 0

サーバーはこのパケットを受信しますが、接続は拒否します：

xx:xx:xx.xxxxxx IP 192.168.xxx.xxx.3389 > xxx.xxx.xxx.xxx.58355: Flags [R.], seq 0, ack 467744440, win 0, length 0

これは、問題が明らかにインフラストラクチャのいくつかの誤動作によって引き起こされたのではなく、他の何かによって引き起こされたことを意味します。多分すべてのユーザーがリモートデスクトップのライセンスに問題がありますか？たぶんいくつかのマルウェアがシステムに侵入できましたが、今日では数年前にXData や  Petyaと同様にアクティブになりました  か？

それらを整理している間、彼らはより多くのクライアントやパートナーから同様のリクエストを受け取りました。
そして、これらのマシンでは何が起こりますか？

イベントログには、パスワードの検索に関するメッセージが多数含まれています。



通常、このような試みは、リモートアクセスサービスに標準ポート（3389）が使用され、どこからでもアクセスが許可されているすべてのサーバーに記録されます。インターネットには、利用可能なすべての接続ポイントを常にスキャンしてパスワードを見つけるボットがたくさんあります（このため、「123」の代わりに複雑なパスワードを使用することを強くお勧めします）。しかし、その日のこれらの試みの強度は高すぎました。

どうやって進める？


多数のエンドユーザーの設定を変更して別のポートに切り替えるために多くの時間を費やすようにクライアントに推奨しますか？良い考えではありません、顧客は満足しません。 VPN経由でのみアクセスを許可することをお勧めしますか？ IPSec接続を発生させようと急いでパニックに陥っていますが、IPSec接続は発生しません。おそらく、そのような幸福はクライアントにも微笑まないでしょう。いずれにせよ、これは慈善の問題ですが、サーバーをプライベートネットワークに隠すことを常にお勧めします。設定をサポートする準備ができています。整理したい場合は、サイト間またはロードモードのクラウドでIPSec / L2TPをセットアップする手順を個別に共有します-warrior、そして誰かが自分のWindowsサーバーでVPNサービスを上げたい場合は、標準のRASまたはOpenVPNを上げる方法に関するヒントをいつでも共有できます。だが、ユーザーの負担を最小限に抑えて問題をできるだけ早く解決する必要があったため、私たちがどれほどクールであったとしても、これは顧客の間で教育作業を行うのに最適な時間ではありませんでした。

実装したソリューションは次のとおりです。通過トラフィックの分析を設定して、ポート3389へのTCP接続を確立するすべての試行を監視し、150秒以内にネットワーク上の16を超える異なるサーバーとの接続を確立しようとするアドレスを選択します-これらは攻撃のソースです（もちろん、クライアントまたはパートナーの1つが同じソースから非常に多くのサーバーとの接続を確立する必要がある場合は、いつでもそのようなソースを「ホワイトリスト」に追加できます。秒、32を超えるアドレスが検出された場合、ネットワーク全体をブロックすることは理にかなっています。ロックは3日間設定され、この間にこのソースからの攻撃がない場合、このソースは自動的に「ブラックリスト」から削除されます。ブロックされたソースのリストは300秒ごとに更新されます。



このリストは次のアドレスで入手できます：https : //secure.tucha.ua/global-filter/banned/rdp_ddos、それに基づいて独自のACLを構築できます。

そのようなシステムのソースコードを共有する準備が整いました。システムに非常に複雑なものはありません（これらは、「膝の上」で数時間以内に文字通り書かれたいくつかの単純なスクリプトです）同時に、このような攻撃から保護するだけでなく、特定するためにも適応および使用できます。ネットワークスキャンの試行をブロックする：  このリンクをたどります。

さらに、監視システムの設定にいくつかの変更を加えました。これにより、クラウド内の仮想サーバーのコントロールグループの反応を綿密に監視し、RDP接続を確立しようとしました。反応が1秒間続かなかった場合、これは注意を払う機会です。

このソリューションは非常に効果的であることが判明しました。顧客やパートナーからの苦情や監視システムからの苦情はもうありません。「ブラックリスト」には定期的に新しいアドレスとネットワーク全体が含まれています。これは攻撃が継続していることを示していますが、お客様の作業には影響していません。

数には安全があります


今日、私たちは他のオペレーターが同様の問題に直面していることを学びました。だれかがリモートアクセスサービスコードに変更を加えたのはMicrosoftであるとまだ信じており（覚えていれば、最初の日に同じことを疑ったが、すぐにこのバージョンを拒否した）、それを可能にするためにあらゆることを行うと約束しているすぐに解決策を見つけてください。誰かが問題を単に無視し、クライアントに自分自身を保護するようにアドバイスします（接続ポートの変更、プライベートネットワークでのサーバーの非表示など）。そして、初日には、この問題を解決しただけでなく、開発を計画しているよりグローバルな脅威検出システムの基盤も作成しました。



敵の死体がいつかそこに浮かぶと予想してすぐに問題に注意を向け、同じ日にそれを排除する機会を与えてくれたという事実を見越して沈黙せずに川岸に座らなかった顧客とパートナーに特に感謝します。
