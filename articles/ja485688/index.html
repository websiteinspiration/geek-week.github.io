ポイントは何ですか？


私がこれ以上興味を失うことがないように私が同様の記事を読んだら、コンポーネントコードの例を見たいと思います。そこで彼は：

import {HdElement} from '../../holiday/core/hd-element.js';

class MyComponent extends HdElement {
  constructor() {
    super();
    this.state = {
      imageURL: 'images/photo.jpg',
      firstName: 'John',
      secondName: 'Snow',
    };
  }
}

MyComponent.template = /*html*/ `
<style>
  :host {
    display: block;
    padding: 10px;
    background - color: #fff;
    color: #000;
  }
</style>
<img bind="src: imageURL" />
<div bind="textContent: firstName"></div>
<div bind="textContent: secondName"></div>
`;

MyComponent.is = 'my-component';

ここで、コンポーネントには「バインド」要素の奇妙な属性を持つ特定の状態とパターンがあることがわかります。これらの属性が主な魔法です。これらの属性を使用すると、テンプレートを解析し、結果のコピーをメモリに保存して、JSランタイムでテンプレート文字列をさらに処理することなく（複製）できます。これにより、各コンポーネントがインスタンス化されるときの初期レンダリング中に高いパフォーマンスが得られます。コンポーネントが担当するDOM部分が状態の関数によって決定される多くの最新のライブラリとは異なり、Holiday.jsテンプレートは初期構造を定義し、通常のDOM APIを使用して後でデータをバインドできるようにする半完成品です。それは非常に速く動作します。速度の問題に戻ります。

他の利点を挙げます。

  • ネイティブAPIへの最大の近接性、最小の追加の抽象化
  • ネイティブテンプレートの構文：新しいHTMLやCSSを学ぶ必要はありません
  • 開始するには、追加でインストールする必要はありません。プロジェクトにnode_modulesフォルダーがまったくない場合があります。
  • 容易さ：ユーザーにダウンロードを長時間待つことを強制しない
  • このキットには、グローバルな状態管理、ルーティング、UIライブラリが含まれています。砂漠にはいられません。

主なプロジェクトドキュメント：https : //holiday-js.web.app/

パターン


Holiday.jsのテンプレートは単なる標準のテンプレートリテラルであるため、プロジェクトにテンプレートを配置する方法とコンテンツを作成する方法は完全に自由です。それらを別々のモジュールに配置したり、コンポーネントコードと一緒に定義したり、データからオンザフライで生成したりできます。テンプレートでは、式と関数の結果を使用できます。これらはすべて、素晴らしい機会と表現の自由をもたらします。たとえば、一般的なスタイルをテンプレートにインポートし、変数を使用して、プリプロセッサを使用せずにCSSで値を計算し、非常に便利に既製の部分からテンプレートを組み立てることができます。

Webコンポーネントテンプレートの重要な部分はスロットです。"。スロットを使用すると、コンポーネントの"シャドウ "DOMの特別な場所を指定できます。その直接の子孫は、一般的なDOMツリーに表示されます。例：

<header>
  <slot name="header"></slot>
</header>

<div class="toolbar">
  <slot name="toolbar"></slot>
</div>

<div class="content">
 <slot></slot>
</div>

<footer>
  <slot name="footer"></slot>
</footer>

スロットへの配置：

<my-component>
  <div slot="header">Header content</div>
  <toolbar-component slot="toolbar"></toolbar-component>
  <div>Content of the default slot...</div>
  <div slot="footer">Footer content...</div>
</my-component>

ご想像のとおり、これにより、必要なモジュールをセルに簡単にソートできるコンポーネントレイアウトを作成できます。

上記のように、データとハンドラーを動的にバインドするには、bind属性を使用します。

<div bind="textContent: content; onclick: on.clicked"></div>

属性値をバインドするには、プロパティ名の先頭にある「@」記号を使用します。

<div bind="@style: style">Content...</div>

詳細については、ドキュメントの対応するセクションをご覧ください：https : //holiday-js.web.app/?templates

  constructor() {

    //       "setStateProperty":

パフォーマンス


フレームワークのベンチマークを書くのは簡単ではありません。代表的な比較の場合、比較可能なものを比較していることを確認し、副作用の影響をできるだけ最小限に抑える必要があります。したがって、彼らはそれぞれのケースで「ボンネットの下」で何が起こっているのかを明確に理解している必要があります。最初に、測定するパラメーターを決定する必要があります。私たちの場合、それは私たちにとって重要です：

  1. メインプログラムの実行開始時間（ダウンロード）
  2. コンポーネントの初期レンダリング時間
  3. これらのコンポーネントを更新して再描画する時間

残念ながら、ブラウザがいつプログラムを実行し、レンダリングが完全に終了したかを確認する信頼できる方法がありません。ブラウザレベルでは、独自の最適化の多くがあり、それは「ブラックボックス」のようなものであり、エンジンによって異なる場合があります。使用できるもの：Performance API、requestAnimationFrame、および非標準のrequestIdleCallbackメソッド。これらはChromeとFirefoxでのみ機能します。初めてで十分です。

Holiday.jsと何を比較しますか？私は2つのライブラリーから始めました：React（それがない場合）とLitElement（コアの類似テクノロジー）。これらのリポジトリでベンチマークコードを見つけることができます：

残念ながら、私は自動化のテストを完了していませんが、自動化だけでは副作用が発生しないため、これも簡単ではありません。現在、私はこの問題に取り組み続けており、Puppeteerを使用して適切なツールを準備しています（非常に便利なことをお勧めします）。それまでの間、手動モード（Chrome / Firefox）でテストを試して、ブラウザコンソールで結果を確認できます。 Holiday.jsはすべてのケースで自信を持ってリーダーであることがわかります。

Reactの場合、コンポーネントが表示されるときに追加のdivコンテナーが作成され、作成されたDOM要素の数を平準化することに気付くでしょう。Webコンポーネントは常にDOM要素とコンテナーであり、さまざまなテストでのDOMツリー構造の適合性が非常に重要であるため、これが当てはまると判断しました。さらに、Reactと同じように、Holiday.jsの過剰なネストを取り除き、データを表示することができます。ただし、この問題に問題があると思われる場合は、まず更新の速度に注意してください。

ブラウザのサポート


いいえ、Internet Explorerはサポートされていません。これは一定の努力（11番目のバージョンの場合）で達成できたかもしれませんが、サポートの拒否は原則的な立場です。第一に、IEは長​​い間死ぬべきだと私は信じています。次に、IEのサポートを主張する人々はしばしば深刻な論理的誤りを犯します。現在のカヌースデータによると、総トラフィックにおけるIEのシェアは、見つかったすべてのバージョンで約1.6％です。そして、他より一歩先を行く最も人気のあるブラウザはモバイルですクロム（〜35％）。したがって、IEをサポートする場合、アプリケーションに完全な適応性とモビリティを実装するために単に必要な多数のテクノロジーの使用を意識的に拒否します。つまり、少数派のために、大多数のユーザーの生活を複雑にします。または、製品の開発と保守のコストを大幅に増やします。第3に、Microsoftは最近、ユーザーにWindows 7を放棄するよう積極的に働きかけているため、近い将来のIEトラフィックのシェアはますます急速に減少します。 IEのサポートをバイパスできない場合があることは認めますが、そのようなケースは、私たちが以前考えていたよりもはるかにまれです。

今年1月15日、Microsoftは新しいEdgeのリリースバージョンをリリースしました、Chromiumエンジン上。この時点から、Holiday.jsは人気のあるブラウザーの現在のすべてのバージョンでネイティブにサポートされていると想定できます。古いEdgeでは、ポリフィルが必要です。ポリフィルを操作する機能はHoliday.jsに実装されていますが、スクリプト自体は個別に接続する必要があります。

ミニマリズム


Holiday.jsのコンセプトは、シンプルさとミニマリズムに基づいています。私がこのプロジェクトに取り組む際の重要な段階の1つは、機能の削除でした。これは急いで実装しましたが、後でそれなしでも実行できることに気付きました。したがって、何かが本当に欠けていると思われる場合は、自分で質問してください。これは本当にそうですか？私は次のように自分に答えました：他にひどく必要なものがある場合は、メインクラスから継承して、拡張クラスを個別に作成することをお勧めします。これまでのところ、このアプローチは失敗していません。現在の状態のHoliday.jsは戦闘でテストされており、非常にポジティブな動作をしていることが示されています。
