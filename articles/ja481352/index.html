DBA：PKなしでテーブルからクローンレコードをクリアする

ときの状況があり、主キーのないテーブル内または他のいくつかのユニークなインデックス、すでに既存のレコードの完全なクローンが誤って落ちます。



たとえば、時系列メトリックの値がPostgreSQLのCOPYストリームに書き込まれ、その後突然の障害が発生し、完全に同一のデータの一部が繰り返し表示されます。

データベースから不要なクローンを取り除く方法は？

PKがアシスタントではない場合


最も簡単な方法は、これがまったく起こらないようにすることです。たとえば、同じPRIMARY KEYをロールします。ただし、保存するデータの量を増やすことなくこれが常に可能であるとは限りません。

たとえば、元のシステムの精度がデータベースのフィールドの精度よりも高い場合：

metric   | ts                  | data
--------------------------------------------------
cpu.busy | 2019-12-20 00:00:00 | {"value" : 12.34}
cpu.busy | 2019-12-20 00:00:01 | {"value" : 10}
cpu.busy | 2019-12-20 00:00:01 | {"value" : 11.2}
cpu.busy | 2019-12-20 00:00:03 | {"value" : 15.7}

気づいてますか？00:00:02の代わりにカウントが1秒前にtsを使用してデータベースに書き込まれましたが、アプリケーションの観点からはかなり有効なままでした（結局のところ、データ値は異なります！）。

もちろん、PK（メトリック、ts）を実行できますが、有効なデータの挿入競合が発生します。PK（メトリック、ts、データ）

を作成できますが、これによりボリュームが大幅に増加するため、使用しません。 したがって、最も正しいオプションは、通常の非一意のインデックス（メトリック、ts）を作成し、問題が発生した場合は、事後に処理することです。



「間代戦争が始まった」


なんらかの事故が発生したため、テーブルからクローンレコードを破棄する必要があります。



ソースデータをシミュレートしましょう：

CREATE TABLE tbl(k text, v integer);

INSERT INTO tbl
VALUES
  ('a', 1)
, ('a', 3)
, ('b', 2)
, ('b', 2) -- oops!
, ('c', 3)
, ('c', 3) -- oops!!
, ('c', 3) -- oops!!
, ('d', 4)
, ('e', 5)
;

ここでは、手が3回震え、Ctrl + Vが動かなくなったので、

最初に、テーブルが非常に大きくなる可能性があることを理解しましょう。したがって、すべてのクローンを見つけた後、文字どおり「指で」特定のエントリを削除することをお勧めします。もう一度検索する必要はありません。

そして、そのような方法があります-これは、特定のレコードの物理的な識別子であるctidによってアドレス指定しています。

つまり、まず、テーブル行の完全なコンテンツのコンテキストでctidレコードを収集する必要があります。最も簡単なオプションは、行全体をテキストにキャストすることです。

SELECT
  T::text
, array_agg(ctid) ctids
FROM
  tbl T
GROUP BY
  1;


t     | ctids
---------------------------------
(e,5) | {"(0,9)"}
(d,4) | {"(0,8)"}
(c,3) | {"(0,5)","(0,6)","(0,7)"}
(b,2) | {"(0,3)","(0,4)"}
(a,3) | {"(0,2)"}
(a,1) | {"(0,1)"}

キャストしないことは可能ですか？
原則として、ほとんどの場合可能です。等価演算子なしでこのテーブルのタイプフィールドを使い始めるまで：

CREATE TABLE tbl(k text, v integer, x point);
SELECT
  array_agg(ctid) ctids
FROM
  tbl T
GROUP BY
  T;
-- ERROR:  could not identify an equality operator for type tbl

ええ、すぐにわかります。配列に複数のレコードがある場合、それだけがクローンです。それらだけを残しましょう：

SELECT
  unnest(ctids[2:])
FROM
  (
    SELECT
      array_agg(ctid) ctids
    FROM
      tbl T
    GROUP BY
      T::text
  ) T;

unnest
------
(0,6)
(0,7)
(0,4)

短い恋人
次のように書くことができます：
SELECT
  unnest((array_agg(ctid))[2:])
FROM
  tbl T
GROUP BY
  T::text;

シリアル化された文字列自体の値は興味深いものではないため、サブクエリの返された列から単純にそれを捨てました。

あとは、受け取ったセットをDELETEに使用させるだけです。

DELETE FROM
  tbl
WHERE
  ctid = ANY(ARRAY(
    SELECT
      unnest(ctids[2:])
    FROM
      (
        SELECT
          array_agg(ctid) ctids
        FROM
          tbl T
        GROUP BY
          T::text
      ) T
  )::tid[]);

自分で確認してください：


[explain.tensor.ruを見てください]

はい、すべてが正しいです。3つのレコードがテーブル全体の単一のシーケンススキャンに対して選択され、削除ノードは単一のパスを使用してTidスキャンを使用してデータを検索しました。

->  Tid Scan on tbl (actual time=0.050..0.051 rows=3 loops=1)
      TID Cond: (ctid = ANY ($0))

多数のレコードをクリアした場合は、VACUUM ANALYZEを実行することを忘れないでください。

より大きなテーブルと多くのテイクをチェックしましょう：

TRUNCATE TABLE tbl;

INSERT INTO tbl
SELECT
  chr(ascii('a'::text) + (random() * 26)::integer) k -- a..z
, (random() * 100)::integer v -- 0..99
FROM
  generate_series(1, 10000) i;


[explain.tensor.ruを見てください]

したがって、メソッドは正常に機能しますが、注意して適用する必要があります。削除されたレコードごとに、Tid Scanのデータページの読み取りが1つと、Deleteの読み取りが1つあるためです。
