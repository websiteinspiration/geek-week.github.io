U-Bootの質問へ

すべての理由を見つけると、あなたは多くを理解するでしょう。


最近、SPI実装の一部にあるU-Bootコードを見て、使用可能なデバイスのリストをバイパスするマクロに出くわしました。いくつかの移行後に、container_ofマクロにリセットされました。マクロテキスト自体が存在し、少し驚いたことに、以前に見たバージョンとは多少異なることに気付きました。小さな調査が行われ、興味深い結果が得られました。

マクロ自体は長い間知られており、やや奇妙な問題を解決します。ある構造（ptr）のフィールドへのポインターがあり、構造のタイプ（type）とフィールドの名前（メンバー）がわかっているため、全体として構造へのポインターを取得する必要があります。このようなタスクがどのように表示されるかはよくわかりません。おそらく、作者は「奇妙なことを望んでいた」かもしれませんが、タスクが設定されたら、解決する必要があります。解決策はよく知られています：

#define container_of(ptr, type, member) \
  ((type *)((char *)(ptr)-offsetof(type,member)))

すべては透明で明確ですが、発見された実装はやや複雑でした。

#define container_of(ptr, type, member) ({			\
  const typeof( ((type *)0)->member ) *__mptr = (ptr);	\
  (type *)( (char *)__mptr - offsetof(type,member) );})

2行目は最初のソリューションとほとんど同じですが、マクロの1行目は何をしますか？いいえ、それが行うことは理解できるだけです。構造フィールドへの定数ローカルポインターを作成し、マクロパラメーターの値（フィールドへのポインター）を割り当てます。しかし、これが行われる理由は不明です。

最初の行の目的に関する明らかな考慮事項は、マクロの最初のパラメーターをチェックして、それが本当にスタイル構造フィールドへのポインターであることを確認することです。

 int *x = (X)

これはこの場合、やや厄介な形をとります：

typeof( ((type *)0)->member ) *__mptr = (ptr)

検証に必要なタイプはその場で構築する必要があるためです。

さて、同意しましょう。これは便利ですが、コンパイラの警告を回避するために、結果のポインタを2行目に使用する必要がありました。

しかし、なぜ不変性修飾子-マクロはこの追加なしで正常に機能するはずです（私はすぐに試してみましたが、機能しました）。著者はそれを誤って置くことができませんでした。

そこを見る必要はありません
私は彼らが私に答えを促したことを認めなければなりません、私自身は推測しませんでした。

この修飾子は、定数構造のアドレスを見つけようとする場合、コンパイラが必要なので、単に必要であることがわかりますinvalid conversion from 'const xxx*' to `xxx*'。

興味深いことに、通常の構造内のフィールド自体の不変性は、マクロでの強制を必要としないだけでなく、定数構造、つまり次のような式の必要性を排除します。

struct s1 {
	int data;
	const int next;
};
	const struct s1 ss1 = {314,0};
container_of(&(ss1.next),struct s1,next);

エラーなしで、マクロテキストと式の修飾子なしでコンパイルします。

struct s1 {
	int data;
	int next;
};
	const struct s1 ss1 = {314,0};
container_of(&(ss1.next),struct s1,next);

必ず必要です。
