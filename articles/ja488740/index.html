Simo AhavaによるGoogleタグマネージャー用のカスタムHTMLタグガイド

1月の終わりに、Simo Ahavaは彼のブログに、GoogleタグマネージャーでカスタムHTMLタグを使用する可能性に関するレビューを投稿しました。カスタムHTMLタグは、サイトのコンテンツを変更する十分な機会を提供しますが、タグ機能とその処理は大きなリスクを伴うため、十分に注意する必要があります。 MediaGuruのアナリストTimur Ledenyovがこの便利なレビューを翻訳しました。

かなり長い間（2012年末以降）、最も重要なGTMオプションの1つはカスタムHTMLタグでした。この魔法のツールを使用すると、GTMはHTML要素をウェブサイトのページに追加できます。 2012年以降、Googleタグマネージャーは、カスタムタグテンプレートを備えた孤立した環境から、無制限のクライアント側コンテンツ管理ソリューションに進化しました。

この記事では、カスタムHTMLタグの原則とその適用の可能性について検討します。

カスタムHTMLタグの入力


名前が示すように、カスタムHTMLタグを使用すると、HTML要素をサイトページに配置できます。それを作成しましょう：



<script>
  console.log('Hello!');
</script>

<div>
  <span>Hello!</span>
</div>

このタグは、ページに3つの要素を追加します。

  • <script>、JavaScriptでコンパイルおよび実行されます。

注入


GoogleタグマネージャーでカスタムHTMLタグが呼び出されると、次のメカニズムが開始されます：

  1. ダミーが作成され<div>、.innerHTML属性を使用して、カスタムHTMLタグを表すコード化された文字列が追加されます。
  2. これにより、ブラウザはエンコードされた文字列をHTMLとして扱うようになり、カスタムHTMLに追加されたタグはHTML要素に変換されます。
  3. これらの要素は1つずつ削除され<div>、注入に進みます。
  4. 次に、各要素が最後の子document.body要素として追加されます。

このプロセスにはいくつかのニュアンスがあります：

  • Googleタグマネージャーがキュー内のonHtmlSuccessおよびonHtmlFailureと対話する方法
  • 要素<script>が注入前にすべてのユーザー属性をクリアする方法。

どういう意味ですか？カスタムHTMLタグに入力したものはすべて、挿入時の内容にかかわらず、本文の最後に追加されます。通常、これらはページフッターを意味しますが、最新のページレイアウトを考えると、これは必要ありません。



重要：新しい要素をページに追加すると、コンテンツの再配置が開始されます。通常、ブラウザは要素のサイズ、配置、レイアウト、属性を再計算する必要があります。埋め込み要素の前、囲み、またはネストされた要素。

それはそれほど簡単ではありません。追加する項目ごとに問題が悪化します。また、遷移間でDOMをリセットしない可能性がある単一ページのアプリケーションでは、1つのページで数百のそのような要素を観察できます。これらの要素はそれぞれ、パフォーマンスをますます悪化させます。結論としてこれに戻りますが、結論の前に私は言います：

ページ上の特定の場所にアイテムを追加する


カスタムHTMLタグの欠点は、要素が最後に埋め込まれること<body>です。何のために？要素が視覚的なコンポーネントである場合（つまり、画面に表示される必要がある場合）は、端<body>が見たい場所ではない可能性が高いです。これを回避するには、JavaScriptとそのDOM操作メソッドを使用する必要があります。

トリックは、ページ上に既にある要素を見つけて、それに関連する新しい要素を配置することです。

たとえば、ページに次のような小さな見出しを追加したいと

代わりに私は必要ですJavaScriptを使用して新しい要素を作成して、ページタイトルに、それは相対的な位置。

<script>
  (function() {
    //    <h3> 
    var h3 = document.createElement('h3');
    
    //

    var title = document.querySelector('h1');
    
    //      <h1>

      title.parentElement.insertBefore(h3, title.nextSibling);

  })();

スクリプト<head>をできるだけ高く配置する


これは一部は前のシナリオによるものですが、この状況が発生する頻度に応じて特に注意する必要があります。「スクリプトをの<head>ように上に配置する」と尋ねられる場合があります。

ただし、Googleタグマネージャーを使用すると、この利点は失われます。通常、GTMライブラリが読み込まれたとき、処理<head>はすでに終了しており、ブラウザはmayおよびmainでレンダリングしてい<body>ます。このため、スクリプトを<head>できるだけ高く追加しようとしても意味がなく、実際には最終結果に悪影響を与えるだけです。

どうして？カスタムHTMLを作成すると、要素が形成され、それが埋め込まれます<head>。まず、ブラウザーはカスタムHTML（パフォーマンスヒット）を追加し、次に新しい要素（別のパフォーマンスヒット）を作成し、最後に<head>（パフォーマンスヒット）に新しい要素を追加する必要があります。


代わりに、<script>HTMLを直接カスタムに追加する必要があります。このメソッドを使用すると、最後<body>に追加され、ブラウザはそれを可能な限り迅速に実行します。

サードパーティのJavaScriptコードのダウンロード


前のシナリオから実験を続けましょう。 JavaScriptコードをサイトにアップロードするサードパーティの開発者がいるとします。また<script>、カスタムHTMLを使用してページに要素を追加するだけでよいことに気付きました。

この場合、カスタムHTMLを使用する必要はありません。

代わりに、injectScript APIを使用してライブラリをロードするカスタムテンプレートを作成します。

カスタムテンプレートはJavaScriptインジェクションとロード用に最適化されており、安全な（より安全な）インジェクションのための権限とポリシーモデルを提供します。



将来的には、これが実装する最良の方法になるでしょう<script>。JavaScriptテンプレートの標準セットは非常に制限されているため、これは他のスクリプトには役立ちません。したがって、たとえばカスタムイベントリスナーを追加する場合は、カスタムHTMLタグが必要になります。
