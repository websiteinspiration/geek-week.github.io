解析の仕組み


パーサーは、コードのトークン化システムからトークンのリストを受け取り、トークンを1つずつ考慮してASTを構築します。トークンの使用方法を決定し、次にどのトークンが期待できるかを理解するために、パーサーは言語の文法の仕様を参照します。

文法の仕様は次のようになります。

...
ExponentiationExpression -> UnaryExpression
                            UpdateExpression ** ExponentiationExpression
MultiplicativeExpression -> ExponentiationExpression
                            MultiplicativeExpression ("*" or "/" or "%") ExponentiationExpression
AdditiveExpression       -> MultiplicativeExpression
                            AdditiveExpression + MultiplicativeExpression
                            AdditiveExpression - MultiplicativeExpression
...

式またはステートメントの実行の優先度について説明します。たとえば、式AdditiveExpressionは次の構成の1つを表す場合があります。

  • 式MultiplicativeExpression。
  • 式のAdditiveExpression後に演算子トークン " +"が続き、その後に式が続きますMultiplicativeExpression。
  • 式のAdditiveExpression後にトークンが-続き、その後に式が続きますMultiplicativeExpression。

その結果、式がある場合1 + 2 * 3、次のようになります。

(AdditiveExpression "+" 1 (MultiplicativeExpression "*" 2 3))

しかし、そうではありません：

(MultiplicativeExpression "*" (AdditiveExpression "+" 1 2) 3)

これらの規則を使用するプログラムは、パーサーによって発行されるコードに変換されます。

class Parser {
  // ...
  parseAdditiveExpression() {
    const left = this.parseMultiplicativeExpression();
    //    -  `+`  `-`
    if (this.match(tt.plus) || this.match(tt.minus)) {
      const operator = this.state.type;
      //    
      this.nextToken();
      const right = this.parseMultiplicativeExpression();

      //  
      this.finishNode(
        {
          operator,
          left,
          right,
        },
        'BinaryExpression'
      );
    } else {
      //  MultiplicativeExpression
      return left;
    }
  }
}

これは、Babelに実際に存在するものを非常に簡略化したバージョンです。しかし、このコードによって、何が起こっているのかという本質を説明できるようになれば幸いです。

ご覧のとおり、パーサーは本質的に再帰的です。最も優先度の低い設計から最も優先度の高い設計に移行します。たとえば、-をparseAdditiveExpression呼び出しparseMultiplicativeExpression、この構造は次のparseExponentiationExpressionように呼び出します。この再帰的プロセスは、再帰的降下解析と呼ばれます。

関数this.eat、this.match、this.next


あなたは、前の例でそのようないくつかの補助的な機能を、使用していることに気づいたかもしれませんthis.eat、this.match、this.nextなど。これらは、Babelパーサーの内部関数です。ただし、これらの関数はBabelに固有のものではなく、通常、他のパーサーに存在します。

  • この関数this.matchは、現在のトークンが指定された条件を満たすかどうかを示すブール値を返します。
  • この関数this.nextは、トークンのリストで次のトークンに進みます。
  • この関数this.eatは、関数が返すものと同じものを返しthis.match、関数が戻る場合this.matchはtrue、戻るthis.eat前にtrue呼び出しを実行しますthis.next。
  • この関数をthis.lookahead使用すると、次に進むことなく次のトークンを取得できます。これにより、現在のノードを決定するのに役立ちます。

変更したパーサーコードをもう一度見ると、コードの読み取りがはるかに簡単になっていることがわかります。

Babel用のプラグイン


パーサーが完成したので、Babelのプラグインを作成しましょう。

しかし、おそらく、今、特にプロジェクトの構築に使用するテクノロジースタックを考慮すると、私たちが独自のBabelパーサーをどのように使用するのかについていくつか疑問があります。

確かに、恐れることは何もありません。Babelプラグインは、パーサー機能を提供できます。関連ドキュメントはBabel Webサイトにあります。

babel-plugin-transformation-curry-function.js

import customParser from './custom-parser';

export default function ourBabelPlugin() {
  return {
    parserOverride(code, opts) {
      return customParser.parse(code, opts);
    },
  };
}

Babelパーサーのフォークを作成したので、これは、既存のすべてのパーサー機能と組み込みプラグインが引き続き完全に正常に動作することを意味します。

これらの疑問を解消した後、カレーをサポートするように関数を作成する方法を見てみましょう。

期待に応えられず、プロジェクトビルドシステムにプラグインを追加しようとした場合、カレーをサポートする関数が通常の関数にコンパイルされていることに気付くでしょう。

これは、コードを解析および変換した後、Babel @babel/generatorが変換されたASTからコードを生成するために使用するためです。@babel/generator彼は新しい属性について何も知らないのでcurry、単にそれを無視します。

カレーをサポートする関数がいつかJavaScript標準に入る場合、PRを実行して新しいコードをここに追加することができます。

関数がカレーをサポートするようにするために、より高次の関数にラップすることができますcurrying。

function currying(fn) {
  const numParamsRequired = fn.length;
  function curryFactory(params) {
    return function (...args) {
      const newParams = params.concat(args);
      if (newParams.length >= numParamsRequired) {
        return fn(...newParams);
      }
      return curryFactory(newParams);
    }
  }
  return curryFactory([]);
}

JSのカリー化機能のメカニズムの実装の機能に興味がある場合は、この資料をご覧ください。

その結果、カレーをサポートする関数を変換することで、次のことが可能になります。

//  
function @@ foo(a, b, c) {
  return a + b + c;
}

//   
const foo = currying(function foo(a, b, c) {
  return a + b + c;
})
