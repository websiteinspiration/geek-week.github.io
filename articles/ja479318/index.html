CLion 2019.3が登場！改善されたエディター速度と最も期待される新機能

こんにちは、ハブル！多くの人がすでに年末年始の準備を始めており、贈り物を買うために、誰かが長い年末年始の旅行を計画しています。そして、JetBrainsでは、製品リリースをリリースするのは今でも熱い時期です。本日、CおよびC ++向けのクロスプラットフォーム開発環境の最近リリースされたリリース-CLion 2019.3に関するニュースをお知らせします。



このリリースの主な目標は、それが聞こえるかもしれませんが哀れな、品質です。私たちは長い間ユーザーを悩ませてきた問題、まず第一に、エディターの生産性と応答性に焦点を当てることに決めました。次に、バグ、不完全性、非常に人気がありますが欠けている機能です。

まず、このリリースで最も重要なことについて簡単に説明します。

  • Clangdをベースにしたエンジンに実装された、エディターの速度と応答性の改善、主にオートコンプリート。
  • CMakeの忍者ジェネレーター、デフォルトのCMake設定、およびその他の設計モデルの改善。
  • デバッガーとの統合による更新。
  • ヘッダーファイルとサワーファイルを切り替える新しいアクション。
  • より正確なコード分析：仮想関数の新しいチェック、およびCMakeとDoxygenコメントでのスペルチェック。
  • C ++ 20標準からの概念のサポート。
  • コードカバレッジメトリック。
  • WSL2、Microsoftのフォーマットおよび命名規則、VCSサポートアップデートなど。

以下で詳しく説明しますが、今すぐ試す準備ができている場合は、サイトにアクセスしてビルドをダウンロードしてください。通常どおり、30日間の無料トライアルが利用できます。

エディターのパフォーマンス


多くのユーザーが送信することに同意している統計によると、CLionで最も一般的に使用されるアクションはオートコンプリートです。このリリースに焦点を当てて、応答性を高めることに決めたのは彼でした。これを行うために、すでにIDEにあるオートコンプリートプロバイダーに、Clangdに基づいてもう1つ追加しました。結論として、すべてのサプライヤーが並行して作業し、最初の結果が準備できるとすぐに、CLionはオプションのドロップダウンリストを表示し、必要に応じて他のオプションをロードし続けます。

もちろん、そのようなハイブリッドアプローチがメリットをもたらすかどうかを理解したかったのです。測定の結果、単純なプロジェクトでは、CLionベンダーとClangdベースのベンダーの速度に大きな違いはありませんでした。しかし、LLVM、Qt、Boost、Eigenなどの複雑なプロジェクトでは、Clangdの最初の100件の結果がはるかに速く表示されます。自分で判断する：




CLionの英語ブログの 別の記事で測定の詳細を読んでください。

他の重要な改善の中でも、IDEの起動時間のプラットフォームアクセラレーションに注目する価値があります。これは、最初に多くのプロセスを並列化し、最初にロードされたクラスを再編成し、macOSでのフォントのダウンロードを最適化することによって達成されました。加速する具体的な数値は、環境の設定、車、プラットフォーム、その他の要因によって異なります。

CLionでは、残念なことに、まだユーザーインターフェイスがハングしています。私たちは元の問題に従ってそれらをグループ化し、次々と問題を修正しようとします。したがって、このリリースでは、宣言に切り替えたとき、＃includeディレクティブの名前を変更したときに、使用状況ビューが開いている場合のフリーズを修正しました、他の場合と同様に、パンくずや安全な削除を使用する場合。ユーザーインターフェイスの一時停止は、私たちにとって最も緊急の問題であり、今後のリリースでもこの作業を継続していきます。

そして最後に、リファクタリングアクセラレーションの名前を変更します。このリファクタリングにより、コードでの名前の使用だけでなく、文字列リテラルおよびコメントでも名前を変更できます。しかし、これは必ずしも必要ではなく、名前の検索は、IDEユーザーが名前を変更する特定の用途を指定する前に実行されました。これは、人気のある名前の検索に長い遅延をもたらしました。これで、最初はコードのみで検索を選択し、次に検索自体を開始することができます。これを行うには、[設定]> [プリファレンス]に移動します。エディター| 全般| 「インプレースモードを有効にする」では、リファクタリングを無効にする必要があります。この場合、リファクタリングが呼び出されると（Windows / LinuxではデフォルトでShift + F6、macOSでは⇧F6）、CLionはすぐに[名前の変更]ダイアログを開き、検索パラメーターを指定できます。



CMake設計モデルの改善


ここでは、多くの人がMakefileのサポートの発表を待っているはずです。しかし、これまでのところ、コンパイルデータベースを介したそれらの統合への半自動アプローチしか利用できません。より多くのネイティブサポートはまだ開発中ですが、このリリースサイクルでは大きな進歩を遂げており、2020年3月下旬のいつか、次のリリース2020.1の新しいサポートを示すチャンスがあります。

しかし、私たちはCMakeをサポートするために最も待たれていた機会を利用しました-ユーザーが期待するNinjaジェネレーターを含む、任意のCMakeジェネレーターを指定する機能！以前は、生成されたファイルを解析したため、Makefileとそれに類似したジェネレーターを使用していました（より正確には、それらは-G“ CodeBlocks-Unix Makefiles”で、Windowsでは-G“ CodeBlocks-MinGW Makefiles”でした）および-G "CodeBlocks-NMake Makefiles"）。これで、CLionのCMakeプロファイルで任意のジェネレーターを指定できます。



これは、CMakeファイルAPIを介して実装され、WSL / WSL2のすべてのプラットフォームでリモートで機能するため、CMakeバージョン3.15以降を使用している場合にのみ可能です。

XcodeやVisual Studioなどのジェネレーターはマルチジェネレーターです。つまり、すべてのタイプのアセンブリ（デバッグ、リリースなど）のファイルを生成しますが、CLionは、 CMakeプロファイル。

このリリースでは、デフォルトのCMake設定も導入されました。つまり、CLionのすべての新しいプロジェクトで、CMakeジェネレーターやCMakeを生成するためのディレクトリなど、同じ事前定義された設定を使用できます。設定は、ファイル|ファイルで指定できます。その他の設定|新しいプロジェクトの設定...



CMakeプロジェクトモデルの重要な改善点のうち、プロジェクトに他の無効な構成が存在する場合でも、有効な構成を再読み込みする機能を指摘することは価値があります。これは、現在使用できないリモート構成をいくつか構成しているが、少なくともローカル構成を再ロードしたい場合に役立ちます。

デバッガー統合アップデート


デバッガでは、ユーザーに最も関係のある問題と欠陥を修正することにしました。まず、CLion はプロジェクトディレクトリから.gdbinit / .lldbinitを読み取ります。これは、特定のプロジェクトに対してのみ、デバッガ設定をカスタマイズしたり、プリティプリンタを追加したりする場合に役立ちます。以前は、ユーザーディレクトリのデバッガー構成ファイルでこれらの設定を指定する必要がありました。これで、プロジェクトに固有のこれらのパラメーターを構成できます。ただし、これを機能させるには、ユーザーディレクトリレベルの設定でデバッガ自体のこの動作を有効にする必要があります（GDBでこれを行う方法とLLDBを行う方法をお読みください）。



LLDBとの統合は、LinuxおよびmacOSで使用できます。新しいリリースでは、組み込みLLDBをバージョン9.0にアップグレードすると同時に、すべての組み込みプリティプリンターをグローバルに見直しました。これにより、両方のプラットフォームでの標準コンテナのパフォーマンスが大幅に向上しました。詳細な比較については、英語のブログをご覧ください。 macOSプラットフォームでは、libc ++ライブラリの処理がlibstdcxxよりはるかに優れていることに注意してください。これは、このプラットフォームでのこれらのライブラリの人気に対応しているようですが、macOSでlibstdcxxを使用している場合は、コメントでそのようなケースについて詳しく教えてください。

CLionにはいくつかのオプションがありますプロジェクトとデバッグのリモート作業。完全にリモートのオプションから、プロジェクトのアセンブリとデバッグの両方がリモートマシンで行われる場合から、プロジェクトのみがリモートでデバッグされるオプションまで。2つ目のケースは以上です。別の構成をCLion- Remote GDBサーバーに追加しました。長年のGDBリモートデバッグとは異なり（はい、名前はありませんでしたが、同意します）、CLionは新しい構成でプロジェクト自体をビルドし（おそらく、クロスコンパイル設定を構成する必要があります）、実行可能ファイルをリモートマシンにダウンロードします。指定したgdbserverでプログラムを実行します。古い構成は、コードが1台のマシンにあり、2番目のマシンにアセンブリがあり、3番目のマシンにデバッグがある場合、複雑なオプションに適しています。または、クロスコンパイルが不可能な場合。

ヘッダー/ソースファイルに移動


はい、起こりました！最後に、ヘッダーファイルとサワーファイルを切り替える別のアクションを追加しました。以前のプラットフォームとの主な違いは何ですか関連シンボルに移動しますか？事実は、一般的なプラットフォームアクションとして、[ 関連シンボルに移動]は非常に正確で最も適切なオプションを選択しようとすることです。これは長く、C ++の場合は常に正しいとは限りません。新しいGo to Header / Sourceアクションの動作は異なります。

  • 500ミリ秒以内に正しいオプションが1つもない場合は、適切なオプションのドロップダウンリストが表示され、そこからユーザーが最も適切なオプションを選択できます。
  • ユーザーが既にこのファイルから移動している場合は、移動先のファイルがリストの一番上に表示されます。
  • さらにリストには、同じ名前の同じディレクトリにあるファイルが表示されます。
  • そして、一致する宣言と定義の検索結果が続きます。

検索の特殊性に注意する価値があります（ユーザーはすでにそれに遭遇しているため）-検索は、直接インクルードされたファイルやインクルードファイルで実行されるようになりました。これは時間制限であり、異なるターゲットからの同じ名前の文字を混同しないようにするために必要です。

スペルチェッカーは、統合開発環境の不可欠なコンポーネントです。私たちは皆、スペルミスを犯し、同僚がコードやコメントを読むのが難しくなります。したがって、IDEがそのようなエラーを強調表示するとよいでしょう。このリリースでは、CMakeファイルとDoxygen形式のドキュメントコメントにスペルチェックが含まれています。



スペルミスが少なくなり、コードが読みやすくなります！

コードカバレッジメトリック


コードカバレッジは、CLionユーザーが待ち望んでいた機会です。前回のリリースでは、AppCodeチームがこの方向で作業を開始しました。この中で、CLionに既に含まれているクロスプラットフォームのC ++のケースを取り上げました。

CLionコードカバレッジメトリックは、llvm-cov / gcovツールとの統合を通じて機能します。この場合、単体テストと通常の構成の両方を実行でき、それらの実行に応じて、特定のコード行の実行頻度が計算されます。複数の起動の結果は、オプションで1つの一般にまとめることができます。

特別なカバレッジウィンドウで結果を直接確認できます。



...またはエディタの左側のガター。

さて、そして最も重要なことです。これが機能するには、特別なコンパイルオプションを渡す必要があります。

  • GCCの場合：-fprofile-arcs -ftest-coverageまたは--coverage。
  • Clangの場合：GCCの場合と同じフラグ、または-fprofile-instr-generate -fcoverage-mapping。違いは、カバレッジメトリックを組み立てる方法のみです。

CLion自体がこれらのパラメーターを渡さないのはなぜですか？ほとんどの場合、コンパイルプロセスに干渉しないというルールがあるためです。はい。CMakeではない場合に、これらのパラメータをどこに渡すかが常に明確であるとは限りません。しかし、現在、このようなケースを自動化するためのオプションについて検討しています（消毒剤に関しても同様の問題があります）。

コンパイルオプションの詳細とCLionでのメトリックの表示の詳細については、オンラインヘルプを参照してください。
