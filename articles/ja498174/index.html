心配を止めてA / Bテストを信じ始める方法

製品を開発する場合、新しいイテレーションごとに、メトリックが低下し、ユーザーが失われるリスクがあります。それにもかかわらず、時には、特に初期段階で、企業は無意識のうちにこのリスクを冒します-彼らは本能と仮説のみに依存して製品を変更します。

Badooの私たちは感情を信用していませんが、私たちは数字を信じています。私たちのサービスには合計5億人以上のユーザーがいて、長い間テストフレームワークを作成しました。 6年の間に2962テストが通過し、A / Bテストはその重要性、信頼性、および有効性を証明しました。



しかし、この記事では、システムの仕組みについては触れません。これには1つの記事では不十分です。さらに、多くのことは当社に固有のものであり、他の人には適していません。今日は、A / Bテストに関する私たちのアイデアの進化について話します。どのような種類のレーキをプロセスに取り入れ、テストの正しさをどのようにチェックしたかです。この記事は、まだテストを開始していないが、それについて考えている人や、テストシステムについてよくわからない人を対象としています。

A / Bテストとは何ですか？


ある状況を想像してみてください。新機能がリリースされる予定であり、私たちの製品マネージャーは、製品のわずかではあるが安定した収入を危険にさらす準備ができていません。いくつかの審議の後、A / Bテストを通じて機能を起動し、それが離陸してユーザーが競合他社に行かないかどうかを確認することを提案します。 

私たちは以前にA / Bテストを実施していなかったので、最初に学ぶことはそれらが何であるかです。ウィキペディアは次のように述べています。「A / Bテストはマーケティングリサーチ手法です。その本質は、要素のコントロールグループが、 1つ以上のインジケーターが変更されたテストグループのセットと比較され、改善目標の変更」対照グループ、少なくとも1つのテストグループ、およびターゲットインジケーターが必要な調査を実施する必要があることがわかりました。

テストグループについては、支払いページの代替バージョンを表示します。その上で、テキストを変更し、割引を強調表示し、画像を置き換えます。これはすべて、ユーラシアのユーザーのために行われるため、ユーザーはさらに購入できます。また、アメリカのユーザーは何も変更したくありません。 

ターゲットがあります。基本的なものを見てみましょう：

  • ARPU（ユーザーあたりの平均収益）-平均してユーザーから受け取る利益。 
  • CTA（行動を促すフレーズ）要素のクリック数-支払いページのボタンとリンク。 

技術的なタスク


ページの3つの要素（テキスト、ボタン、および割引情報付きの画像）を同時に変更する予定です。



同時にテストした場合、これらの変更のどれが結果に影響を与えたかを理解することは非常に困難です。おそらく、新しいテキストはユーザーを反発し、購入数の減少につながりますが、割引を強調するとそれが増加します。その結果、ゼロになります。開発リソースは無駄になり、作業仮説は拒否されます。 

したがって、これは行いません。テストのための変更は1つだけ残します-割引の割り当て。 


最初のテスト


計画は次のとおりです。ユーザーを2つの等しいグループに分け、結果を待ちます。受け取ったら、コントロールグループとテストグループのインジケーターを比較します。 

すべてが非常にシンプルに見えます。ユーザーを偶数と奇数に分け、統計を確認します。 

コードを書きます：

if (userId % 2 == 1) {
    showNewAwesomeFeature();
}

私たちは数週間を楽しみにしています：結果は素晴らしいです！ARPUが100％増加しました！人々はクリックして支払い、製品マネージャーは変更をすぐに展開するように求めます。 



オンにすると、さらに2週間ほど経ちますが、結果はありません。総利益は増加していません。 

何が悪いのでしょうか？

テストグループを選択します


ユーザーを2つの等しいグループに分けてテストを実行しただけですが、これは実行できません。結局のところ、ユーラシアのユーザーのためだけに何かが変わったのです。そして、アメリカのユーザーよりもはるかに少ないです。したがって、アメリカからのユーザーアクティビティの急激な増加が、実際には最高ではありませんでしたが、テストの結果に影響を与えていることがわかりました。

結論：テストグループでは、変更を実装するユーザーのみを常に選択してください。

コードを修正しましょう：

if (user.continent === ‘eurasia’ && userId % 2 == 1) {
    showNewAwesomeFeature();
}

これで、すべてが正常に機能するはずです。テストを実行します。数週間待っています。



起こりました！ARPUは80％上昇しています！変更をすべてのユーザーにロールアウトします。 

そして...同じ期間の後、グラフは再び私たちが計画したように見えなくなります。

統計的有意性の計算


「数週間後」にテストを停止することはできません。得られた結果は不正確な場合があります。私たちがフォローしているメトリックの変化が少なく、テストに参加する人が少ないほど、ランダムである可能性が高くなります。  

この確率は計算できます。それを表す値はp値と呼ばれます。それがどのように機能するかをお話します。

テスト中、ランダムな結果が得られる可能性があります。たとえば、サイトにアクセスしたユーザーが不均一に分散している場合があります-有料視聴者全体がいずれかのグループに分類されます。実際のテストでは、メトリックの差は通常それほど大きくありません。グラフで問題に気づくのは困難であり、不可能でさえあり、統計的テストは省略できません。特に、確率を修正する必要があります第1種および第2種のエラー -言い換えると、それらが存在しない場所で変更を見つける確率、および逆に、それらが存在する場合にそれらを見つけられない確率。メトリックの値と確立された確率に応じて、異なる数のユーザーが必要になります。オンライン計算機

を使用してそれを計算できます：監視対象メトリックの現在の値とターゲット値を指定します-テストに必要なユーザー数を取得できます（逆も同様）。現在の値に対して10％での変換および0.2％での変化の計算 これで、必要なすべてのデータを受け取り、テストを停止するタイミングがわかりました。これ以上の障害はありません。 A / Bテストを実行して結果を見てみましょう。










今回は、結果はより真実に似ていますが、それでも優れています。ARPUの成長率は55％です。 

テストを停止し、テストグループを適用します。そして...メトリックは再び低下しています。どうして？ 

ユーザー数を確認する


テストの進行中に実際にサイトにアクセスしたユーザーの数を調べてみましょう。ログから判断すると、テストグループの10％のみですが、これは考慮されていません。 

DAU（1日あたりのユニークユーザー数）が1000人の場合、これは、10日間で10,000人のユーザーがテストに参加することを意味しません。常にテストへの実際のヒット（テストヒット）を記録し、それらのみをカウントします。

単純なロジックを実装します。サイトにアクセスするユーザーごとに、A / Bテストおよびコントロールグループの名前を含むリクエストをサーバーに送信します。これにより、アクセスしたユーザーの数が正確にわかるので、間違いがなくなります。

A / Bテストを開始します。

結果は素晴らしいです。私たちは再びもっとお金を稼ぎます！私たちはすべての人に対してテストをオンにします-そして再び何かがうまくいきません：数週間後、メトリックは再び予測よりも低くなります。 

サイトを訪問したすべてのユーザーにヒットを送信し始めたことを覚えていますか？絶対にしないでください。ヒットは、リソースのテスト済み部分を操作したユーザーにのみ送信する必要があります。そして、あなたがそれらをより正確に定義するほど、より良いです。 

幸い、これは簡単に確認できます。これを行うには、A / A / Bテストを実行できます。 A / Bテストのように見えますが、この場合、2つのコントロールグループと1つのテストグループがあります。なぜこれが必要なのですか？ヒットを送信するタイミングが誤って選択された場合、サイトのテストされた部分を操作しなかったユーザーがテストに分類されます。この場合、グループAとAのメトリックには大きな違いがあり、何かが間違っていることをすぐに理解できます。 

A / A / Bテストを開始します。 

グループBでは、ユーザーの50％を残し、残りの50％をグループAとグループAの間で均等に分割します（これらをcontrolおよびcontrol_checkと呼びます）。はい、結果はより長く待つ必要がありますが、グループAとAの結果が収束するかどうかによって、ヒットが正しく送信されるかどうかがわかります。



結果は控えめ（20％の増加）ですが、現実的です。変更をまったくロールしましょう。 

すべてが完璧に動作します！ 

しかし、1か月後、指標は再び低下しました。

制御できるもののテスト


サードパーティの請求でもA / Bテストが実施され、結果に直接影響していることがわかりました。したがって、常に本番環境での結果に従い、管理しているものを完全にテストしてみてください。

合計


A / Bテストは製品の成長に役立ちます。ただし、テストを信頼するには、テストを正しく実施し、常に結果を確認することが重要です。このアプローチにより、製品を定性的にテストし、すべてのメトリックを落とす前に仮説をテストできます。

  • 常に対象グループを確認してください。
  • ヒットを送信します。
  • 適切なヒットを送信します。
  • 完全に制御するものをテストします。
