C ++的高性能垃圾收集

我们 已经 写了关于JavaScript的垃圾收集，关于DOM的内容，以及如何在V8 JS引擎中实现和优化所有这些内容。是的，Chromium不仅是JavaScript。内置V8的大多数浏览器和Blink渲染引擎都是用C ++编写的。可以使用JavaScript与DOM一起使用，并使用呈现管道将更改显示在屏幕上。



由于与DOM相关的C ++对象的图与JavaScript对象紧密相关，因此几年前，Chromium开发团队开始使用名为Oilpan的垃圾收集器来管理存储这些对象的内存。。 Oilpan是一个C ++垃圾收集器，旨在管理可以连接到V8的C ++内存。内存管理是使用跨组件垃圾收集技术执行的。在该技术的框架中，相关的C ++ / JavaScript对象的图形被视为单个堆。 该材料是Oilpan的第一本出版物。在这里，我们将回顾该垃圾收集器的基本原理，以及C ++ API Oilpan。我们将研究Oilpan支持的一些功能，并讨论垃圾收集器各个子系统的工作方式。在这里，我们将分析竞争性释放对象占用的内存的过程。











最有趣的是，Oilpan系统是Blink的一部分，但现在已被转移到V8，在这里它将以用于垃圾收集的库的形式呈现。所有这一切的目的是为所有将V8引擎嵌入其平台的用户提供便利的C ++垃圾收集机制。此外，Oilpan将成为一个库这一事实将允许所有对此感兴趣的C ++程序员使用此系统。



一般信息



Oilpan使用垃圾收集系统，该系统使用标记和清除算法。该算法将垃圾收集过程分为两个阶段。第一阶段包括检查堆，并标记无法从内存中删除的“活动”对象。第二阶段是清除堆内存，该堆内存被不必要的（“死”）对象占用。



我们已经写了算法实现的标记阶段。如果我们回想起这些材料的主要思想，事实证明，对“活动”对象的搜索可以看作是图的绕过，其中对象是节点，而对象彼此引用的指针是图的边缘。行走始于根实体这些物体是事先已知它们在“生存”的物体。例如，这是一个全局对象和活动功能。



在这方面，JavaScript的C ++没什么不同。的确，与JavaScript对象不同，C ++对象是静态类型的。结果，他们无法在程序执行期间更改自己的视图。使用Oilpan处理C ++对象时，将考虑到这一事实，并使用“访问者”模式提供了指向其他对象（图形边缘）的指针的描述。用于描述Oilpan对象的基本模式如下所示： 



class LinkedNode final : public GarbageCollected<LinkedNode> {
 public:
  LinkedNode(LinkedNode* next, int value) : next_(next), value_(value) {}
  void Trace(Visitor* visitor) const {
    visitor->Trace(next_);
  }
 private:
  Member<LinkedNode> next_;
  int value_;
};

LinkedNode* CreateNodes() {
  LinkedNode* first_node = MakeGarbageCollected<LinkedNode>(nullptr, 1);
  LinkedNode* second_node = MakeGarbageCollected<LinkedNode>(first_node, 2);
  return second_node;
}

作为常规C ++指针存储在堆栈中。 Oilpan不使用其他抽象来处理堆栈。垃圾收集器在处理其管理的堆中的根对象时，在寻找指针时仅依赖于保守的堆栈扫描。所有这一切都通过对堆栈进行逐字排序来完成，并且要感谢以托管堆中实体的指针形式对字进行解释。这意味着在访问堆栈中的对象时，使用Oilpan不会降低性能。而是在执行保守的堆栈扫描时将负载转移到垃圾回收阶段。 Oilpan已集成到呈现子系统中，并尝试推迟垃圾收集过程的开始，直到系统到达肯定没有任何有趣堆栈的状态为止。由于Web是基于事件的，并且代码是通过处理事件循环中的任务来执行的，因此Oilpan有足够的方便时间启动垃圾收集。



Oilpan用于Blink，这是用C ++编写的大型代码库，其中包含大量的成熟代码。因此，Oilpan除其他外具有以下功能：



  • 使用mixin和指向相似mixin的链接的多重继承（内部指针）。
  • 支持在执行构造函数时调用垃圾回收。
  • 使用智能指针将非托管内存中的对象保持在“活动”状态Persistent



    ，这些智能指针被视为根实体。
  • 顺序容器（例如，向量）和关联容器（例如，设置和映射）的集合。能够压缩基础集合的数据。
  • 链接薄弱，功能薄弱和短暂的数据结构。
  • 终结器是在从内存中删除单个对象之前执行的方法。


清除C ++的内存



我们将再谈一遍在Oilpan工作期间如何标记对象。现在，我们将基于以下假设：对对象的标记已经完成，并且Oilpan已经使用对象方法检测到了所有可到达的对象Trace



。标记阶段完成后，为可访问对象设置相应的位。



清理是油锅工作的阶段，当损坏的死物体（在标记物体的阶段无法达到的）被销毁时。它们占用的内存要么返回操作系统，要么可以分配给其他对象。下面我们将讨论Oilpan如何执行内存清理，并考虑此过程的功能及其局限性。此外，我们将讨论油盘内存清理机制在销毁死物时如何实现高速运行。



内存清理系统通过对堆内存进行排序并检查对象的相应位来查找死对象。为了保留C ++的语义，内存清理系统需要在释放其占用的内存之前调用每个“死”对象的析构函数。非平凡的析构函数被实现为终结器。



在这里，从程序员的角度来看，没有预定的析构函数调用顺序，因为内存清理系统使用的搜索机制没有考虑破坏对象的创建顺序。这对终结器施加了限制，根据终结器，终结器无法访问堆上的其他对象。我们面临着经常遇到的任务。它包含以下内容：有一个平台不支持指定对象的完成顺序（例如Java），但是与此同时，对于该平台，您需要编写要求一定顺序调用终结器的代码。 Oilpan使用了Clang插件，该插件在静态模式下禁止在销毁对象期间访问堆对象（这只是Clang的众多功能之一）： 



class GCed : public GarbageCollected<GCed> {
 public:
  void DoSomething();
  void Trace(Visitor* visitor) {
    visitor->Trace(other_);
  }
  ~GCed() {
    other_->DoSomething();  // error: Finalizer '~GCed' accesses
                            // potentially finalized field 'other_'.
  }
 private:
  Member<GCed> other_;
};


在销毁另一个对象期间尝试从堆中访问对象时，此代码中发生错误。



如果您对销毁对象的问题感兴趣，请知道Oilpan为程序员提供了在终结器之前调用的回调。当一个对象在销毁之前需要访问其他对象时，这使您可以实施复杂的销毁方案。这样的回调在垃圾回收期间给系统带来了相当大的额外负担。此负载大于析构函数能够创建的负载。因此，此类回调在Blink中很少使用。



增量和竞争性内存刷新



既然我们已经讨论了在托管C ++环境中析构函数的局限性，现在该更详细地介绍memorypan如何实现和优化内存清理。



但是，在进行此对话之前，重要的是要记住程序如何在Web环境中运行。主线程通过将任务放置在事件循环中来启动代码（即JavaScript代码）的执行以及垃圾的清除。与其他执行代码的环境一样，渲染系统支持后台任务，这些后台任务与主线程一起以竞争模式执行。这使您可以从主流中除去部分负载。



在一开始，Oilpan使用了根据“世界停止”方案实现的内存清理机制。这意味着在内存清理过程中暂停了主线程中应用程序的执行。









在内存清理过程中暂停代码执行



对于实时操作没有太大限制的应用程序，延迟是使用垃圾回收器的决定因素。使用世界停止机制清除内存可能导致程序在相当长的时间内暂停，从而导致用户注意到明显的延迟。为了减少这些延迟的时间，通过实现增量内存清理来改善垃圾收集。









增量内存清除



使用增量方法时，对象的标记和销毁阶段是相互分离的。在这种情况下，内存清除过程分为几个部分，分别由在主线程中执行的单独任务表示。在最佳情况下，此类任务仅在停机期间执行。系统，避免它们与代表常规应用程序机制的任务发生冲突。垃圾收集器的内部机制将清理内存变成一项小任务，这是一项大任务，这种分离基于“页面”的概念。页面可以处于两种状态。有些页面处于等待清除（待清扫）的状态，而另一些页面已经被清除（已经清扫）。内存分配机制仅考虑已被清除的页面，并从空闲内存列表中补充本地内存分配缓冲区（Local Allocation Buffers，LAB），该列表存储可用内存片段的列表。为了获取内存（有关空闲内存列表中的信息），应用程序将首先尝试查找引用已清除页面的内存。然后，应用程序将尝试使用试图在其中分配内存的内存清理系统来帮助系统处理等待清理的页面。仅当以上操作没有导致应用程序接收到所需的内存时，才会从操作系统请求新的内存。



在Oilpan中，增量内存清理机制已使用了几年。但是，随着应用程序及其对象图越来越多，内存清除开始影响应用程序性能。为了改善垃圾收集机制，我们开始使用后台任务来实现有竞争力的内存清理。为了消除执行内存清理的后台任务与为新对象分配内存的应用程序之间的“数据争夺”，有两个规则：



  • 内存清理系统仅处理“死”对象的内存，根据定义，这些对象无法从应用程序中获得。
  • 应用程序仅使用属于已清除页面的内存来分配内存。这些页面不再对内存清洁系统感兴趣。


通过这两个规则，可以确保多个实体不会为同一对象及其内存争斗。不幸的是，C ++严重依赖于作为终结器实现的析构函数。 Oilpan在主线程中运行终结器，以帮助开发人员消除应用程序代码本身中的数据争用。为了解决此问题，Oilpan计划推迟主线程中对象的终结器。更具体地说，每当竞争性内存清理系统遇到具有终结器（析构函数）的对象时，它将其放入终结队列中，该队列将在终结的特殊阶段进行处理。此阶段的代码始终在主线程中执行，此外，还执行应用程序代码。一般来说，这就是看起来像一个竞争性的内存清除方案。









使用后台任务进行竞争性内存清理



由于终结器可能需要访问对象中包含的所有内容，因此将适当的内存添加到可用内存列表将延迟到终结器完成之前。如果对象没有终结器，则在后台线程中运行的内存清理机制会立即将释放的内存添加到空闲内存列表中。
