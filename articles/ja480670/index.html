PM2：プロセス管理の問題に賢く取り組みます

ほんの数時間前、私はNode.JSが大規模なプロジェクトには遅すぎて、Golang、Rust、PHPなどを好むはずであるという事実についての議論を始めました。この議論の反対側の主な議論は、JavaScriptがシングルスレッドであるという事実でした。伝えられるところによると、アプリケーションを開発するとき、パフォーマンスは単にこの単一スレッド性に依存しており、もう何もできません-他の言語で書き直してください。ただし、一見したところよりも、NodeJSの方が少し優れています。このトピックを掘り下げる前に、私は各開発者が好きで、特定のタスクで好ましいと考えるプログラミング言語を使用する権利を尊重することを宣言したいと思います。

Habrでキーワード「PM2」で検索したところ、このプロセスマネージャに関する記事は見つかりませんでした。他のユーザーの記事の単一の参照のみ。Node.JSのバックエンド開発のこの暗いコーナーに光を当てて光を当てるという考えに興奮しました（強く語られました）（多くの人が知っていますが、そうです、私は知っています）。猫の下で興味を持っている皆さんにお願いします。



PM2自体について一言



PM2は、AGPL-3.0ライセンスに基づいてライセンスされたオープンソースプロセスマネージャーです。これを書いている時点では、NPMによると、毎週約35万回のダウンロードがあります。これは主に、NodeJSでアプリケーションを実行してそれを忘れる必要がある環境で使用されます（他の言語でも使用できますが、後で詳しく説明します）。これにより、アプリケーションをクラスター化し、プロセッサーコア間で負荷を柔軟に分散できます。GitHubのPM2リポジトリからの小さなクリッピング：

PM2は、ロードバランサーが組み込まれたNode.jsアプリケーションのプロダクションプロセスマネージャーです。アプリケーションを永遠に存続させ、ダウンタイムなしでアプリケーションをリロードし、一般的なシステム管理タスクを容易にすることができます。


開発時に、アプリケーションを本番サーバーに「ロールアウト」した後、「永久に」起動する方法がわからない場合、多くの新人が問題に直面します。彼らはSSHコンソールに書き込みset NODE_ENV=production && node app.js、すべて問題なく、アプリケーションは動作します。コンソールを閉じると、アプリケーションが機能しなくなります。StackOverflowの質問-node.jsアプリケーションを永続的に実行する方法は？すべての時間で23万7,000回を超えるスコアを記録しました。

PM2は1つのコマンドでこの問題を解決します。

pm2 start app.js


このコマンドは、NodeJSプロセスを「デーモン化」します（英語の「デーモン化」から）。そのメモリ消費を監視し、プロセッサの負荷を考慮します。

ラムに戻る



バックエンドへの負荷が高まるにつれ、状況に応じて使いやすいように、垂直方向と水平方向の両方にスケーリングする必要があります。ご存知のように、1つのプロセスが複数のプロセッサコアを使用できます。 NodeJSアプリケーションでは、フローは1つです。 PM2は、この状況を解決し、複数のプロセッサコア間で負荷を分散することができます。それでもコマンドは1つだけです。

pm2 start app.js -i max


この場合、maxパラメータはプロセッサコアの数に対応します。それら。8コアプロセッサ用に8つの個別のプロセスが作成されます。maxの代わりに-1を設定することもできます。その場合、プロセス数はコアの数から1を引いた数に対応します。すべての魅力は、HTTP（S）/ Websocket / TCP / UDP接続がこれらのプロセス間で均等に分散されることです。なぜ水平スケーリングしないのですか？PM2でのクラスタリングの詳細については、PM2クラスターモードを参照してください。



プロセスはいくつでも開始できますが、それでも「コアあたり1つのプロセス」という推奨に従うことをお勧めします。

記憶の尊重


PHPで開発しているときに、ある問題に遭遇しました。経験不足のため、彼は無意識のうちにシステムエンジンにバグを発生させました。これにより、特定の条件下で、プロセスが大量のRAMを消費し始めました。これに加えて、仮想マシンがハングしたばかりでプロセッサにアクセスできなかったため、プロセッサがロードされました。
PHP開発者が知っているように、PHP-FPMではプロセスの分散のタイプを指定できます（突然知らなかった場合、PHP-FPMでは新しいリクエストごとに新しいプロセスが作成されます）-静的、最小および最大しきい値が設定されている場合、動的-割り当て量必要な数の大きなプロセス。 8つのプロセスを開始し、それらすべてが多くのメモリを消費し始めた場合、PM2で何が起こりますか？そして、PM2はこの問題を解決することができます-コマンドラインでパラメーターを1つだけ使用します。

# Set memory threshold for app reload
pm2 start app.js -i max --max-memory-restart <200MB>


メモリ制限に達するたびに、PM2は自動的にプロセスを再起動します。メモリの分散はプロセスより簡単ですよね。8プロセス* 200メガバイト= 1.6ギガバイト。2番目のクラスの数学。

プロセスの再起動に加えて、N時間間隔後の再起動を構成することもできます。これがどのような場合に役立つかはまだわかりませんが、コメントでいくつかの例を示してください:)

また、仮想マシンを再起動するとどうなりますか？


サプライズサプライズ！この問題PM2も解決します。それでもコンソールにコマンドが1つしかない場合：

pm2 startup


PM2は、オペレーティングシステムの起動時に必要なすべてのプロセスを起動するスクリプトを生成します。ただし、注意が必要です。Node.JSのバージョンを更新すると、すべてが壊れる可能性があります。これを回避するには、Node.JSの新しいバージョンに正常にアップグレードした後、pm2 unstartupおよびを実行しpm2 startupます。詳細については、リンク-PM2起動スクリプトジェネレーターを参照してください。

変更を行う場合、クラスターを手動で再起動する必要がありますか？


もちろん違います！より正確には、もちろん、手動でアプリケーションを再起動できますが、なぜですか？あなたができるすべてを自動化し、力があなたと一緒に来るかもしれません！

pm2 start env.js --watch --ignore-watch="node_modules"


これは、ローカルリポジトリのマスターブランチをリモートリポジトリのマスターブランチとマージするときに使用できます。私のサイドプロジェクトでは、これは単純に行われます- git pull origin master && npm run build。server / buildおよびclient / buildフォルダー内のファイルを変更すると、プロセスが自動的に再起動されます。これは非常に単純な機能であり、このテキストで言及するに値しないことも理解しています。私はそれを深刻なもので希釈し、クラスタリングを使用すると、すべてのプロセスが順番に再起動されると書いています。はい、少なくとも1つは常に利用できるようにします。これはダウンタイムなしの展開です！

そして、あなたはプロセスを再起動することはできません。これにはリロードがあります（nginx reloadに似ています）：

pm2 reload all


チームが多すぎます！一般的に、私は設定を好みます


私はすでに面白いフレーズを思い付くのに飽きているので、それはシンプルで平凡です。エコシステムファイルがあります。サポートされている形式は、JSON、YAML、JSです。たとえば、サーバーとクライアントのフォルダー内のファイルを監視する必要がある場合：

module.exports = {
  apps: [{
    script: "app.js",
    watch: ["server", "client"],
    env_production : {
      "NODE_ENV": "production"
    }
  }]
}
