ProseMirrorを使用した単純なWYSIWYGエディターの作成

Sports.ruに独自のWYSIWYGエディターが必要な場合、ProseMirrorライブラリに基づいて作成することにしました。このツールの主要な機能の1つは、そのモジュール性とカスタマイズの幅広い可能性です。そのため、このツールを使用すると、エディターをプロジェクトに合わせて非常に細かく調整できます。特に、ProseMirrorはすでにNew York TimesとThe Guardianで使用されています。この記事では、ProseMirrorを使用してWYSIWYGエディターを作成する方法について説明します。



ProseMirrorの概要


ProseMirrorの作者はMarijn Haverbekeで、フロントエンドの開発者コミュニティで主に人気のある書籍Eloquent Javascriptの作者として知られています。私たちの作業の初め（2018年秋）には、このライブラリの操作に関する資料はありませんでした。ただし、著者による公式のドキュメントとチュートリアルを除きます。著者のドキュメントパッケージにはいくつかのセクションが含まれていますが、その中で最も役立つのは、ProseMirrorガイド（基本的な概念の説明）とリファレンスマニュアル（ライブラリの仕様）です。以下は、ProseMirrorガイドの主要なアイデアの要約です。

ProseMirrorは常にドキュメントの状態を独自のデータ構造に格納します。そして、すでにランタイムのこの構造から、対応するDOM要素がエンドユーザーが操作するページ上に生成されます。さらに、ProseMirrorは現在の状態（状態）だけでなく、必要に応じてロールバックできる以前の変更の履歴も保存します。状態の変更はトランザクションを通じて発生するはずです。DOMツリーでの通常の操作は、ここでは直接機能しません。トランザクションは、段階的な状態変更のロジックを記述する抽象概念です。彼らの仕事の本質は、州の管理のために、たとえばReduxやVuexなどのライブラリーでアクションを送信および実行することを連想させます。

ライブラリ自体は、必要に応じて無効化または追加できる独立したモジュールに基づいて構築されています。ほとんどの人が必要と考えるメインモジュールのリスト：

  • prosemirror-model-ドキュメントのすべてのコンポーネント、コンポーネントで実行できるプロパティとアクションを記述するドキュメントモデル。
  • prosemirror-state-ある状態から別の状態に移動するために選択されたフラグメントとトランザクションを含む、特定の時点での作成されたドキュメントの状態を記述するデータ構造。
  • prosemirror-view-ユーザーがドキュメントを操作できるように、ブラウザとツールでのドキュメントの表示。
  • prosemirror-transform-トランザクション履歴を保存する機能。どのトランザクションが実装されているかを利用して、以前の状態にロールバックしたり、1つのドキュメントを共同開発したりできます。

この最小限のセットに加えて、次のモジュールも役立つ場合があります。

  • prosemirror-commands-編集用の既製のコマンドのセット。原則として、自分でもっと複雑な、または個人的なものを書く必要がありますが、Marijn Haverbekeはすでにいくつかのことを行っています。たとえば、選択したテキストのフラグメントを削除します。

ProseMirror


エディターのアウトラインを作成することから始めましょう。 ProseMirrorのスキーマは、ドキュメントに含まれる可能性のある要素とそのプロパティのリストを定義します。各要素にはtoDOMメソッドがあり、WebページのDOMツリーでこの要素をどのように表すかを決定します。

WYSIWYGの原理は、スキームを作成するときに、ページ上に編集可能なコンテンツをどのように表示するかを完全に制御できるため、各要素にHTML構造を与え、表示するコンテンツと同じようにスタイルを設定できるという点で正確に実現されます。ノードを作成して、要件に合わせてカスタマイズできます。特に、段落、見出し、リスト、画像、段落、メディアが必要になる可能性が高いです。

テキストの各パラグラフは、レイアウトに必要なパラグラフスタイルのルールを決定するクラス「パラグラフ」を持つタグ「p」でラップする必要があるとします。この場合、ProseMirrorスキーマは次のようになります。

import { Schema } from "prosemirror-model";

const mySchema = new Schema({
    nodes: {
        doc: {
           content: 'block'
        },
        text: {},
        paragraph: {
            content: 'inline',
            group: 'block',
            toDOM: function toDOM(node) {
                return ['p', {class: 'paragraph'}, 0];
            },
        },
    },
});

最初に、コンストラクターをインポートしてスキーマを作成し、将来のエディターでノード（以下、ノードと呼びます）を記述するオブジェクトをスキーマに渡します。ノードは、作成されるコンテンツのタイプを記述する抽象概念です。たとえば、このようなスキームでは、テキストと段落のタイプのノードのみをエディターに含めることができます。 Docは、最上位ノードの名前で、ブロック要素のみで構成されます。この場合は、段落からのみです（他のものについては説明していないため）。

テキストはテキストノードであり、テキストDOMノードに多少似ています。 groupプロパティを使用すると、ノードをさまざまな方法でグループ化して、コードでより簡単にアクセスできるようにすることができます。グループは、都合のよい方法で設定できます。この例では、ノードをブロックとインラインのみに分割しました。テキストはデフォルトでインラインなので、これは明示的に省略できます。

まったく違うのは段落（段落）です。段落はインラインテキスト要素で構成され、DOMでも独自の表現を持つことを発表しました。DOMブロック要素とほぼ同じ意味で、段落はブロック要素になります。このスキームによれば、ページ上の段落は次のようにオンラインエディタに表示されます。

<p class="paragraph">Content of the paragraph</p>

これでエディター自体を作成できます。

import { EditorState } from "prosemirror-state";
import { EditorView } from "prosemirror-view";
import { Schema } from "prosemirror-model";

const mySchema = new Schema({
    nodes: {
        doc: {
            content: 'block+'
        },
        text: {
            group: 'inline',
            inline: true
        },
        paragraph: {
            content: 'inline*',
            group: 'block',
            toDOM: function toDOM(node) {
                return ['p', {class: 'paragraph'}, 0];
            },
        },
    },
});

/**
 * @classdesc  
 * @param {Object} el DOM-,    
 */
export default class Wysiwyg {
    constructor(el) {
        this.el = el;
    }

    /**
      * @description     
      * @param {Object} content ,    
      */
    setArticle(content) {
        const state = EditorState.fromJSON(
            {schema: mySchema},
            content,
        );
        const view = new EditorView(this.el, {state: state});
    }
}

最初に、いつものように、対応するモジュールから必要なすべてのコンストラクターをインポートし、上記のスキームを採用します。必要なメソッドのセットを持つクラスとしてエディターを作成します。 Webページでコンテンツを編集および作成できるようにするには、記事のレイアウトとコンテンツを使用して状態、状態を作成し、特定のルート要素の現在の状態のコンテンツを提示する必要があります。これらのアクションはsetArticleメソッドに配置しますが、これまでのところ何も必要ありません。

同時に、コンテンツはオプションです。そうでない場合は、空のエディターが表示され、コンテンツをその場で直接作成できます。次のマークアップを含むHTMLファイルがあるとします。

<div id="editor"></div>

ページ上に空のWYSIWYGエディターを作成するには、次のマークアップを使用してページで実行されるスクリプトの数行のみが必要です。

//     
import Wysiwyg from 'wysiwyg';

const root = document.getElementById('editor');
const myWysiwyg = new Wysiwyg(root);
myWysiwyg.setArticle();

このコードを使用すると、テキストを最初から作成できます。この時点で、ProseMirrorの動作を既に確認できます。

ユーザーがこのエディターにテキストを入力すると、状態トランザクションが発生します。たとえば、上記のスキームを使用してエディターで「これは私の新しいクールなWYSIWYGエディターです」というフレーズを入力すると、ProseMirrorは適切なトランザクションのセットを呼び出してキーボード入力に応答し、入力が完了すると、ドキュメントの状態のコンテンツは次のようになります。

content: [
    {
        type: 'paragraph',
        value: '    WYSIWYG-',
    },
]

たとえば、以前に作成した記事のコンテンツなどのテキストをエディターで開いて編集する場合、このコンテンツは以前に作成したスキームに対応している必要があります。次に、エディターの初期化コードは少し異なります。

//     
import Wysiwyg from 'wysiwyg';

const root = document.getElementById('editor');
const myWysiwyg = new Wysiwyg(root);
const content = {
    type: 'doc',
    content: [
        {
            type: 'paragraph',
            value: 'Hello, world!',
        },
        {
            type: 'paragraph',
            value: 'This is my first wysiwyg-editor',
        }
    ],
};
myWysiwyg.setArticle(content);

やったー！新しいコンテンツを作成し、既存のエディターを開いて編集するためのクリーンなページを作成できる、最初のエディターを作成しました。

次にエディターをどうするか


しかし、ノードの完全なセットがあっても、エディターには重要な機能（テキストや段落のフォーマット）がまだありません。このため、ノードに加えて、書式設定、マークの設定を含むオブジェクトも回路に転送する必要があります。簡単にするために、ブランドと呼びます。また、これらすべての要素の追加、削除、およびフォーマットを制御するには、メニューが必要です。メニューは、メニューオブジェクトのスタイルを設定し、特定のアクションを選択したときのドキュメントの状態の変化を記述するカスタムプラグインを使用して追加できます。

プラグインを使用すると、エディターの組み込み機能を拡張する任意のデザインを作成できます。プラグインの主なアイデアは、プラグインに対応するトランザクションを生成するために、特定のユーザーアクションを処理することです。たとえば、メニューのリストアイコンをクリックして新しい空のリストを作成し、カーソルを最初の要素の先頭に移動する場合は、対応するプラグインでこのトランザクションを説明する必要があります。

公式ドキュメントでフォーマット設定とプラグインの詳細を読むことができます。また、ProseMirror機能を使用する非常に小さな便利な例も非常に役立ちます。
