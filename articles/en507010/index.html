And again about GPG hardware keys for a penny

It all started with the fact that I read the article Hardware encryption key for $ 3 - is this possible? decided to gash such a thing. As a result, stlink came from the second time. The first was lost in the bowels of the mail.

Key Access Issue


gpg did not see the key. More precisely, I saw it, but only as root. A long study of the problem and googling followed. As a result, it turned out


  • I have no corny rule for user access to the device
  • Access to the map can be carried out by more than one method from gpg

The first problem is solved by adding the file /etc/udev/rules.d/60-gnuk.rules


SUBSYSTEMS=="usb", ATTRS{idVendor}=="234b", ATTRS{idProduct}=="0000",   \
ENV{ID_SMARTCARD_READER}="1", ENV{ID_SMARTCARD_READER_DRIVER}="gnupg"

And reboot udev


devadm control --reload-rules

This applies more to those who are not debian . It and its derivatives do not have this problem, but I have fedora. After that, the key was seen.


The second problem can be solved in two ways.


  • Disable pcscd service
  • Configure use of pcscd service in gnupg

You can disable a service by deleting it or by banally stopping the service


systemctl stop pcscd
systemctl disable pcscd

The second option is to configure the gnupg service to use pcscd


touch ~/.gnupg/scdaemon.conf
echo -e "pcsc-driver /usr/lib64/libpcsclite.so.1\ndisable-ccid]\n" > ~/.gnupg/scdaemon.conf

scdaemon gnupg pcscd.
