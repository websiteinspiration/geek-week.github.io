useSWR es mi nueva biblioteca React favorita

Se preparó una traducción del artículo antes del comienzo del curso "Desarrollador React.js" .


Los últimos meses he estado trabajando en una aplicación en NextJS. Cada semana se vuelve más y más. La aplicación usa axios para llamadas API y no-next-next para gestión de estado. Hay muchas llamadas a la API, pero no queremos que los usuarios vean un montón de pantallas de carga. Por lo tanto, almacenamos los resultados de las llamadas axios en repositorios no declarados .

Sin embargo, nos encontramos con un problema. Las bóvedas se están volviendo cada vez más complejas. Las páginas a veces requieren múltiples llamadas a la API, y a su vez dependen de los resultados de otras llamadas a la API. Los días se convirtieron en semanas y las semanas en meses, mientras que nuestras bóvedas no declaradas se volvieron cada vez más engorrosas. Nos encontramos con errores extraños mientras nuestra lógica de almacenamiento en caché casera luchaba por lidiar con casos extremos inesperados.

Entonces pensamos que el camino debería ser más simple.
Y aquí está él. E incluso es desarrollado por Vercel , creador de NextJS .

Conozca SWR


El nombre SWR viene de revalidar obsoleto , un método de almacenamiento en caché que ahora está ganando popularidad en el desarrollo frontend. Le permite descargar contenido en caché inmediatamente, y lo actualiza inmediatamente para manejar contenido nuevo en el futuro. En nuestro caso, obtuvimos el compromiso perfecto entre el rendimiento y la experiencia del usuario.

Cómo usar useSWR


useSWR es una biblioteca React hook desarrollada por Vercel . Le permite recuperar datos de una API u otra fuente externa, guardarlos en la memoria caché y luego representarlos.

Comencemos mirando un ejemplo de un componente React que recibe una lista TODO de un servidor JSON y lo procesa.

import React from "react";
import "./App.css";
const todosEndpoint = "http://localhost:3001/todos";
const TodoApp = () => {
  const [todos, setTodos] = React.useState([]);
  React.useEffect(() => {
    const getData = async () => {
      const response = await fetch(todosEndpoint);
      const data = await response.json();
      setTodos(data);
    };
     getData();
    }, []);
  return (
    <div className="App">
      {todos.map((todo) => (
        <div key={todo.id}>{todo.title}</div>
      ))}
    </div>
  );
};
export default TodoApp;

Ahora veamos el mismo componente, pero reescrito usando useSWR .

import React from "react";
import useSWR from "swr";
const todosEndpoint = "http://localhost:3001/todos";
const getData = async () => {
  const response = await fetch(todosEndpoint);
  return await response.json();
};
const TodoApp = () => {
  const { data: todos } = useSWR(todosEndpoint, getData);
  
return (
    <div>
      {todos && todos.map(todo => (
        <div key={todo.id}>{todo.title}</div>
      )}
    </div>
  );
};
export default TodoApp;

Como puede ver, es muy similar a la implementación anterior.

En este ejemplo, usamos useSWR (key, fetcher, options) para obtener nuestra lista TODO. Clave en uso SWR es necesaria para el almacenamiento en caché. En nuestro caso, usamos todosEndpoint . Como fetcher, pasamos una función asincrónica que recupera una lista TODO.

Cabe señalar que useSWR no tiene idea de cómo recuperar los datos. Puede usar cualquier estrategia de minería de datos asincrónica que desee. Puede usar fetch , axios o incluso GraphQL . Mientras su función está devolviendo datos asincrónicamente,useSWR será feliz.

La llamada useSWR devuelve los siguientes parámetros:

  • datos : datos clave obtenidos usando fetcher (no definido, si aún no está cargado);
  • error : error causado por el buscador ;
  • isValidating : un booleano que dice si una solicitud o revalidación está en curso;
  • mutate (data?, shouldRevalidate) : función para modificar datos en caché.

Ejemplo más complejo


Veamos un ejemplo más complejo que usa más parámetros devueltos por useSWR . En este ejemplo, extraeremos un elemento de la lista TODO y mostraremos la casilla de verificación para el estado completo de nuestra tarea de la lista. Cuando se hace clic en la casilla de verificación, enviamos una solicitud PUT a la actualización TODO, y luego llamamos a mutar para actualizar el useSWR caché .

import React from "react";
import useSWR from "swr";
import "./App.css";
const todosEndpoint = "http://localhost:3001/todos";
const getTodo = async (id) => {
  const response = await fetch(`${todosEndpoint}/${id}`);
  return await response.json();
};
const updateTodo = async (id, todo) => {
  const response = await fetch(`${todosEndpoint}/${id}`, {
    method: "PUT",
    headers: {
      "Content-type": "application/json; charset=UTF-8",
    },
    body: JSON.stringify(todo),
  });
  return await response.json();
};
const TodoApp = () => {
  const todoId = 1;
  const key = `${todosEndpoint}/${todoId}`;
  const { data: todo, mutate } = useSWR(key, () =>
    getTodo(todoId)
  );
  const toggleCompleted = async () => {
    const newTodo = {
      ...todo,
      completed: !todo.completed,
    };
    await updateTodo(todoId, newTodo);
    mutate(newTodo);
  };
  if (!todo) {
    return <div>Loading...</div>;
  }
  return (
    <div>
      <p>{todo.title}</p>
      <div>
        <input
          type="checkbox"
          id="completed"
          name="completed"
          value="completed"
          checked={todo.completed}
          onChange={toggleCompleted}
        />
        <label htmlFor="completed">Completed</label>
      </div>
    </div>
  );
};

Usar mutate es una excelente manera de aumentar el rendimiento visible de su aplicación web. Podemos cambiar los datos localmente y cambiar la vista, sin esperar hasta que se actualice la fuente de datos remota. useSWR incluso los revalidará y reemplazará con los últimos datos en segundo plano.

¿Por qué necesito una biblioteca para esto?


Puedes hacer esta pregunta. Tal vez ya tenga un mecanismo de administración de estado en su aplicación y no vea el punto de usar una biblioteca de terceros para algo tan simple como la extracción de datos y el almacenamiento en caché.

Y le responderé que el almacenamiento en caché de datos no es fácil. Y a medida que su aplicación crece, se vuelve aún más difícil. Encontrará casos extremos, y al procesarlos, obtendrá repositorios y proveedores complejos que son difíciles de entender y difíciles de mantener. Los errores se derramarán de la nada.
"En informática, solo hay dos problemas complejos: invalidación de caché e inventar nombres" - Phil Carleton

En lugar de implementar su propia solución de minería de datos, ¿por qué no confiar en una solución probada desarrollada por una de las compañías más respetadas en el ecosistema React?

Cómo nos ayudó useSWR


Portar el tema de la extracción de datos en nuestra aplicación para usar SWR ha dado muchas ventajas.

1. Nos permitió eliminar el código

Solicitudes de extracción que eliminan códigos innecesarios, pero no agreguen uno nuevo: mis favoritos. Mientras menos código haya en la aplicación, es menos probable que ocurran errores. Y esto está bien. A medida que envejezco, empiezo a apreciar la simplicidad cada vez más. useSWR nos permitió eliminar por completo el almacenamiento no declarado, lo que simplificó nuestra aplicación y facilitó su comprensión.

2. Simplificó la conexión de nuevos desarrolladores al proyecto

useSWRComo la mayoría de los proyectos de código abierto, cuenta con una excelente documentación. Implementar nuestra propia solución significaría que necesitamos escribir nuestra propia documentación y enseñar a los nuevos desarrolladores del proyecto cómo manejar la extracción de datos. Ahora que usamos useSWR , no necesitamos hacer esto.

3. Simplificó las cosas difíciles.

Nuestra aplicación, como la mayoría de las otras aplicaciones, contiene muchas llamadas API. Algunas de estas llamadas dependen de otras llamadas API. Con useSWR enlaces fáciles de escribir para consultas dependientes.

4. Se ha mejorado el rendimiento visible. La

aplicación se ve más rápido. Los usuarios lo calificaron y dieron buenos comentarios.

5. Actualiza datos obsoletos

useSWR actualiza los datos obsoletos en foco. Esto significa que los usuarios siempre tienen la versión más actualizada de los datos, excluyendo el tiempo de descarga.

Conclusión


useSWR tuvo un gran impacto en nuestra aplicación. Esta biblioteca ha simplificado nuestro código y ha mejorado la experiencia del usuario. Ni siquiera puedo imaginar otra biblioteca que sea tan fácil de implementar y tenga tantos beneficios como useSWR . Para obtener más información sobre useSWR , puede visitar el sitio o el repositorio en GitHub .

Bibliotecas alternativas


useSWR no es la única buena biblioteca para recuperar datos. Si bien funcionó bien en nuestro caso, quizás otra biblioteca sea más adecuada para la extracción de datos.

Consulta-reacción


react-query es una biblioteca de enlaces para recuperar, almacenar en caché y actualizar datos asincrónicos en React. Esta solución popular, escrita por Tanner Linsi , tiene un alcance muy similar al uso de SWR .

Apolo


El cliente Apollo es un cliente GraphQL que se encarga de recuperar y almacenar en caché sus datos, así como de actualizar la interfaz de usuario. Si usa GraphQL , Apollo será una gran solución.
