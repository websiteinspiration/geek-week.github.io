依存関係をテストする正当な理由：AGPLエディション

ここでは、BSD、MIT、Apache2のライセンスの下でコードを取得し、泡立て器を吹き飛ばさないでください。-2番目のシフト、および推移的な依存関係で、AGPLの下のコードが描画されます。私たちはこれを追跡するよう努めており、仕上がり不足ではなく追い越しを優先しています。



プロジェクトに新しい依存関係を追加する前に、常に基本的なチェックを行います。チェックするもの（標準チェックリスト）：

  • コードはどのライセンスの下で発行されますか？
  • 著者は誰ですか？
  • エラートラッカーに重大な未解決の問題はありますか？
  • エラートラッカーに重大なバグの履歴はありますか？
  • プルクエストのコードレビューはどのように行われますか？

その後、明らかに安全ではないものや悪意のあるものを探すために、コード自体をざっと調べます。これは、コードベース自体の品質を感じるために必要です。次に、「茶色のMとM」 -大きな問題を示す可能性のあるマイナーな詳細を見つけようとします。そして、推移的な依存関係で上記のすべてを再帰的に繰り返します。さらに、依存関係を更新するたびに、もう一度コードをざっと読みます。

これはかなりの量の作業ですが、event-streamなどの攻撃の犠牲にならないようにする必要があります。そして最近、依存関係を確認するもう1つの理由がわかりました。その時点で、DuoからWebAuthn on Go用にアクティブにアドバタイズされたライブラリのレビューを行っていましたが、ここにあります：github.com/duo-labs/webauthn。

それはすべて、いくつかの「茶色のMとM」に気付いたという事実から始まりました。

  • ライブラリであるにもかかわらず、データはstdoutに記録されました。
  • コードが蓄積されました。

もちろん、これらの小さな問題は何かの前触れにすぎませんでした。推移的な依存関係の1つ（github.com/katzenpost/core/crypto/eddsa）のレビューを開始したとき、AGPLv3ライセンスヘッダーに出会いました。

これは、Duo WebAuthnライブラリを使用したい人にとっては悪いニュースです。DuoがライブラリをBSDでライセンスしているという事実にもかかわらず、それを選択すると、アプリケーションをAGPLライブラリとリンクすることもできます。そして、（A）GPLによると、この方法でAGPLのセクション13の規則に該当する「変更された」製品を作成します。

つまり、公開ウェブアプリケーションでgithub.com/duo-labs/webauthnを使用した場合、ウェブアプリケーションはオープンソースになります。

この依存関係について最も非道なことは、golang.org / x / crypto / ed25519（Go の準標準「x」ライブラリの1つ）を複製することです。

実際、github.com / duo-labs / webauthnは、最初に使用されたgolang.org/x/crypto/ed25519と同じです。置換は、「COSEの事柄を自分の領域に統合する」と呼ばれる外部コラボレーションのプルクエスト中に発生しました。コードの一部をあるファイルから別のファイルに転送するプロセスで、このプルリクエストは実装をわずかに変更しましたOKPPublicKeyData.Verify。

ここでOKPPublicKeyData.Verify使用するものgolang.org/x/crypto/ed25519は：

// Verify Octet Key Pair (OKP) Public Key Signature
func (k *OKPPublicKeyData) Verify(data []byte, sig []byte) (bool, error) {
	f := HasherFromCOSEAlg(COSEAlgorithmIdentifier(k.PublicKeyData.Algorithm))
	h := f()
	h.Write(data)
	return ed25519.Verify(k.XCoord, h.Sum(nil), sig), nil
}

そしてここOKPPublicKeyData.Verifyでは、AGPLライセンスのgithub.com/katzenpost/core/crypto/eddsaを使用しています。

// Verify Octet Key Pair (OKP) Public Key Signature
func (k *OKPPublicKeyData) Verify(data []byte, sig []byte) (bool, error) {
	f := HasherFromCOSEAlg(COSEAlgorithmIdentifier(k.PublicKeyData.Algorithm))
	h := f()
	h.Write(data)
	var oKey eddsa.PublicKey
	err := oKey.FromBytes(k.XCoord)
	if err != nil {
		return false, err
	}
	return oKey.Verify(h.Sum(nil), sig), nil
}

この変更についての説明はありませんでした。プルリクエストのレビューはDuoの2人の従業員によって行われ、承認され、差し控えられました。

そのため、コードを移動するプルリクエストのリクエストを受け入れたくありません。コードの新しい構成が以前の構成よりも優れている場合でも、「新しいプルクエストで余分なことが行われていないか」を確認するのにかかる時間は、多くの時間を消費します。AGPLライセンスのあるライブラリの依存関係に関する警告

を投稿したところ、開発者は golang.org/x/crypto/ed25519を返しました。それにもかかわらず、github.com / duo-labs / webauthnを使用しないことにしました。ライブラリとその依存関係の大部分は、アテステーションと呼ばれるWebAuthnの誤った機能をサポートするように設計されています。非常にシンプルで証明のないライブラリを作成し終えたところです。これは、上記の10分の1未満です。すぐにソースコードを開きます。このライブラリの開発は、既存のWebAuthn Goライブラリを使用する責任よりも安価になりました。

このケースは、なぜGoでのプログラミングが好きなのかを思い出させてくれました。 Goの広範な標準および準標準の「x」ライブラリのおかげで、通常、プロジェクトには追加の依存関係はほとんどありません。 Goの評判と操作手順が優れているので、スチームバスを使用したり、コンパイラと標準ライブラリのソースコードを再確認したりする必要がありません。そして、私はRustを愛しているという事実にもかかわらず、それらの典型的なライブラリーの依存関係ツリーを見るたびに私は恐怖を感じます。これらの依存関係のすべてをチェックするには時間がかかりすぎるため、Rustの生産性はGoよりもはるかに低くなります。

最後に、Certificate Transparencyのような検証可能なデータ構造のファンとして、新しいGoチェックサムデータベースを気に入ってください。ただし、依存関係のチェックに時間を費やさないと、チェックサムデータベースは役に立ちません。残念ながら、私はすでに、チェックサムデータベースが依存関係管理のすべての問題を解決すると主張している非常に熱心なGoユーザーを見たことがあります。しかし、これはそうではありません。これを取り除く簡単な方法はありません。そして、プロジェクトに依存関係のレビューを行う必要があるという事実を理解する必要があります。
