VSでのデバッグ時にThreadPoolからスレッドに名前を付けない

2017年の一部では、VSでのデバッグ中に、プロジェクトの生産性が約80％低下し、ゲームがさまざまな非同期フレームのコレクションになりました。お祝いの原因は、プール内の関数SetThreadNameでした。なじみのない人-ThreadPoolは、同じコードの並列実行を担当する一種のマネージャーです。たとえば、サイクルを分散させることができます。その本質は、いくつかのスレッド（コアごとに1つのワーカー）を作成し、作業の完了時に待機/削除するために転送することです。

SetThreadNameは次のようになります。

void SetThreadName(DWORD dwThreadID, const char* threadName) {
    THREADNAME_INFO info;
    info.dwType = 0x1000;
    info.szName = threadName;
    info.dwThreadID = dwThreadID;
    info.dwFlags = 0;
#pragma warning(push)
#pragma warning(disable: 6320 6322)
    __try{
        RaiseException(MS_VC_EXCEPTION, 0, sizeof(info) / sizeof(ULONG_PTR), (ULONG_PTR*)&info);
    }
    __except (EXCEPTION_EXECUTE_HANDLER){
    }
#pragma warning(pop)
}

新しいWinSDKでは、RaiseExceptionを使用した古き良きハックをSetThreadDescriptionで置き換えることができるため、最終的に状況を保存できます。なぜなら RaiseExceptionはかなり遅い操作です。

状況からいくつかの最適な出口がありました：

  1. 原則として労働者に名前を付けないでください。
  2. インターネットでのみ提供し、フローを待機モードに保つ。実行中のプロセスに参加する場合、この方法は不適切です。名前を認識できなくなります。
  3. 労働者は破壊されません。各タスクは、以前に無効にされていた場合、接続されたデバッガの存在をチェックされます。IsDebuggerPresent
