Styling Android applications and the design system: how to do it and make friends with one another









Hello readers!  



At some point, any large application grows so that it is difficult to maintain the same design everywhere and dynamically respond to any changes and trends in design and UX requirements.  



Therefore, we decided to introduce a design system in our application and add support for several themes. 



Having studied various methods, we developed our approach to solving such a problem. I wanted to make the design system and style support reusable in my other projects. In accordance with this idea, components and themes were developed.



Design System Components



The design system and its components are designed to unify design and style unity throughout the application.



In our case, we will call the custom view components of the design system with the ability to adapt to several application styles. Components can be applied anywhere in the application (buttons, list items, headers, etc.).



Design, debugging and refinement of components of a design system



Design system components are ordered by designers. At the first stage, we coordinate with them the need for an element (reuse assessment) and its functionality. 



After coordination, it should be clear which options should be placed in the custom view attributes (text color, text, icon, icon tint color, etc.), and which to hide from changes from the outside (this allows you to protect the element from improper use by developers).



Next, designers draw the component in their environments and give it up for development. 



When implementing the component, you need to add support for themes (light or dark theme, etc.). I will describe below how the component supports several themes.



Best practices



  • Create a module with components of a design system. From the positive aspects: a separate module can be used in other applications, and modularity allows you to quickly navigate.
  • Create a test application with components of a design system. This speeds up development and debugging.


Ways to embed a theme in an application



I know two ways to support styles in Android:



  • Software (software repainting).
  • Standard style mechanisms in Android.


Software method



   We repaint the entire view hierarchy in runtime. We recursively go through it and, according to certain rules of transition from one topic to another, repaint the components. Those that should not be repainted are marked with android:tag



or android:contentDescription



. These components are not taken into account when parsing the screen hierarchy.



    You can repaint both before displaying the screen (for example, in onStart()



y Activity



), and when working with it.      



disadvantages



  • Requires additional resources, reduces performance. Stylization is applied after initialization of all components.
  • You need to be careful about the rules for moving from one topic to another. It takes a lot of rules to repaint, you can forget something. It turns out a long sheet from switch ‚Äî case



    (Java) or when



    (Kotlin). And in the appendage you need to take into account elements that do not need to be painted using the above tags.
  • You can not partially repaint in accordance with the themes. There are exceptions to any rule, and not everything in the application is always done according to the design system. It is not clear how to proceed if partial repainting of some elements is required.



The application of style comes down to the description of changes in specific elements: 



if (view is TextView) {

Standard constructor view



The standard view constructor provides extensive means for customizing an element. The appearance of elements can be changed through .xml attributes or through the default style definition in the standard view constructor.



Consider the standard constructor view using an example H1Component
