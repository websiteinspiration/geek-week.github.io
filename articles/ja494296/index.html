Xiaomi Zigbeeゲートウェイハック

この記事では、Xiaomiゲートウェイ（私がtake.ruを使用したユーロプラグを備えたバージョン）の解析における私のベストプラクティスと成功について説明します。代替ソフトウェアをインストールする方法、ワイプされたソフトウェアでゲートウェイを復元する方法、ワイプされたu-bootでゲートウェイを復活させる方法についても説明します。

---------たくさんの写真-------------



まず、ゲートウェイボードに接続してルートを取得する方法を説明します。

最初に、ゲートウェイに関するいくつかの詳細。

この記事では、imx6ullプロセッサで構築されたゲートウェイのバージョンについて説明します。

ボードのいくつかの技術仕様：

  • Wifi RTL8723BS-何らかの理由でそこからのドライバーが個別にロードされます。
  • CPU IMX6ULL
  • RAM 256 MB
  • ROM 256 MB
  • JN5169に基づくZIgbeeモジュール。

ボードには、サービスダイムP2およびP4が表示されます。理想的には、ポゴピンの下に作られますが、はんだ付けが簡単です。

P2は、すべてのサービス情報、ubootブートログ、システム管理が表示されるプロセッサーのWart 0です。任意のusb-uartアダプターを使用できます。

P4-USB OTG。ファームウェアをnandにアップロードするために使用されます。 （本格的なOTGとして、最後まで起動できなかったため、接続するとデバイスが検出されましたが、明らかに他に何かが不足しています）。



中国のUSBアダプターボードは、ニックネームP4に最適です。



すべてがはんだ付けされて接続されたら、最初にルートを取得する必要があります。

私はこの指示に従って（マニュアルへのリンク）しましたが、ポイント5の後の明確化のために、bootコマンドを入力する必要がありますそうでない場合、システムは座ってコマンドを待機します。指示はおそらく、誰もがLinuxを理解することを願って書かれたものです。

何らかの理由で、SSHサーバーは指示に従ってオンにならず、/ etc / init.d / dropbear startを開始するコマンドを手動で開始することによってのみオンになりました。

次のステップでは、ルートを受け取った後、WinScpを使用してシステムの内容をコピーしました（接続の作成時にSCPモードを選択します）。

さらに、実験で悲惨な結果が生じた場合に備えて、システム全体をバックアップする必要があります（私はそうしましたが、バックアップしませんでした）。

ゲートウェイのフラッシュメモリは4つのセクションに分かれています。



結局のところ、空き領域はそれほど多くありませんが、メモリ内にはさまざまな重複ファイルがたくさんあります。



数時間プレイした後、私はソフトウェアで何かを変更し、ゲートウェイはロードを停止しました:)

では、この鉄片を復活させて何かで埋める方法を考えてみましょう。
レンガがある場合にのみアクションを実行してください！

Imx6ULLプロセッサには、オープンフォームのすべての情報が含まれています。すべてのデータボードとリファレンスマニュアルは、登録後にメーカーのWebサイトで入手できます。

U-bootがまだロードされていました...しかし、回復しようとしていたときに、誤って余分なコマンドをコピーして、nandを完全に消去しました。そして結局のところ、u-bootはナンダにありました。そして、彼はレンガを手に入れました。

プロセッサーはさまざまなブートモードをサポートしています。リファレンスマニュアルには、すべて記載されています。



開発ボードには通常スイッチがありますが、そのようなものはありません。データシートには、どのような種類のプロセッサダイムであるかが記載されています。



そして中国人もとても素敵な贈り物をしました。彼らはそれらを非常に便利な場所に持ってきて、ほとんど署名しました）

プロセッサの隣のボードで、ブートモードピンがぴったり合う2つの抵抗がはんだ付けされています。



これらの2つの抵抗は、ブートモードのスイッチにすぎません。デフォルトではnandに設定されていますが、それぞれをサイドに移動すると、シリアルブートモードになります。

ゲートウェイをシリアルブートモードに転送しました。このモードは、さまざまなファームウェアのテストに便利です。

ファームウェアをアップロードするには、NXP mfgtoolsの専用ユーティリティを使用します。唯一のことは、プロセッサのプロファイルを準備することです。mfgtoolsを

ダウンロード。アーカイブは、必要なすべての設定を備えたバージョンがすでに準備されています。

ゲートウェイはUSB経由で接続します。シリアルブートモードでは、ゲートウェイはUSBインターフェイスを介してファームウェアをダウンロードしようとし、新しいHIDデバイスがコンピューターに表示されます。

Mfgtoolsは、新しいデバイスを検出したことを通知します。



時々コンピュータはそれを見ない、それが接続されているもので私は言うことができません。

ユーティリティがそれを確認したら、[開始]をクリックすると、ファームウェアのダウンロードプロセスが開始されます。
ユーティリティはnandをフォーマットし、新しいubut、カーネル、dtbおよびrootfsをフラッディングします。アーカイブのrootfsは友人のAleksandr Faronovによってコンパイルされ、彼はYoctoにクリーンなLinuxビルドをまとめました。 Uブート、カーネル、DTBセクションは、ゲートウェイのネイティブファームウェアから引き出されました。

このLinuxアセンブリでは、クリーンなシステムのみが起動し、それ以上はありません（zigbyもまだ機能しません。

私の場合、ユーティリティが機能した後、ゲートウェイは何もせず、ロードもしません。シリアルブートモードでプロセッサをリコールしているためです。

したがって、彼には起動可能なUブートイメージを与える必要があります。mfgtoolsのアーカイブには、ボードのRamにUブートのロードを実行する追加のユーティリティuuuがあります。それを含むフォルダーに移動します。
mfgtoolsを閉じ、ゲートウェイを新しいゲートウェイに再接続します。HIDデバイスが再度検出され、その後コマンドを実行します。

uuu u-boot-small.imx

ボードの読み込みが開始されますが、envがないため、U-bootの読み込み後に起動します。ゲートウェイコンソールで、2つのコマンドを実行し、ENVをロード用に登録します。

setenv bootargs 'console=ttymxc0,115200 ubi.mtd=3 root=ubi0:rootfs rootfstype=ubifs cma=96M mtdparts=gpmi-nand:3m(boot),7m(kernel),1m(dtb),-(rootfs)'

そしてboot bootを開始すると、ゲートウェイはnandaとアップロードしたファームウェアからロードを開始します。



概して、それだけです。mfgtoolsでは、\ Profiles \ Linux \ OS Firmware \ filesフォルダーにすべてのファームウェアファイルが配置され、互換性のあるシステムをロードすることができます。openwrtをダウンロードしてみました。



システムはロードされていますが、適切なコアをスリップする方法とそれをどこで取得するかはまだわかりません。

私は共同の取り組みにより、このゲートウェイを論理的な結論に導くことを期待しています）
