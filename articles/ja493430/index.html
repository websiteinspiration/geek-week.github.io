Webアプリケーションのネットアーキテクチャ

Webアプリケーションを含むアプリケーションの開発で長年使用してきたアプローチを皆さんと共有したいと思います。デスクトップ、サーバー、およびモバイルアプリケーションの開発者の多くは、このアプローチに精通しています。はこのようなアプリケーションを構築する上での基本ですが、このアプローチを使用したい人は確かに存在しますが、Web上での表現は非常に貧弱です。さらに、VS Codeエディターはこのアプローチで作成されています。



このアプローチを適用した結果、特定のフレームワークが取り除かれます。アプリケーション内のビューライブラリ（React、Preact、Vue、Mithrilなど）をビジネスロジックを書き換えることなく簡単に切り替えることができ、ほとんどの場合はビューも切り替えることができます。Angular 1上のアプリケーションがある場合、それをAngular 2 +、React、Svelte、WebComponents、さらにはプレゼンテーションライブラリに簡単に変換できます。Angular 2+でアプリケーションを使用しているが、専門家がいない場合、ビジネスロジックを書き直すことなく、アプリケーションをより人気のあるライブラリに簡単に転送できます。しかし、最終的には、フレームワークからフレームワークへの移行の問題を完全に忘れてください。これはどんな魔法ですか？

クリーンアーキテクチャとは


これを理解するには、Martin Robertの本"Clean Architecture»（Robert C.Martinによる" Clean Architecture»）を読むことをお勧めします。引用により記事に記載されている短い抜粋。

アーキテクチャに組み込まれた主なアイデア：

  1. フレームワークからの独立。アーキテクチャは、ライブラリの存在に依存しません。これにより、システムを制限することなく、フレームワークをツールとして使用できます。
  2. テスト容易性。ビジネスルールは、ユーザーインターフェイス、データベース、Webサーバー、またはその他の外部コンポーネントなしでテストできます。
  3. UIからの独立性。ユーザーインターフェイスは、システムの他の部分を変更することなく簡単に変更できます。たとえば、ビジネスルールを変更せずに、Webインターフェイスをコンソールに置き換えることができます。
  4. データベースからの独立。OracleまたはSQL ServerをMongoDB、BigTable、CouchDBなどに置き換えることができます。ビジネスルールはデータベースに関連していません。
  5. 外部サービスからの独立。実際、ビジネスルールは外の世界について何も知らないだけです。

この本で長年説明されているアイデアは、さまざまな分野で複雑なアプリケーションを構築するための基礎となっています。

この柔軟性は、アプリケーションをサービス、リポジトリ、モデルレイヤーに分割することで実現されます。Clean ArchitectureにMVCアプローチを追加し、次のレイヤーを取得しました。

  • 表示 -クライアントにデータを表示し、実際にロジックの状態をクライアントに視覚化します。
  • コントローラー -IO（入出力）を介してユーザーと対話する責任があります。
  • サービス -ビジネスロジックとコンポーネント間の再利用を担当します。
  • リポジトリ -データベース、API、ローカルストレージなどの外部ソースからデータを受信します。
  • モデル -レイヤーとシステム間でのデータ転送、およびこのデータを処理するロジックを担当します。

各レイヤーの目的については、以下で説明します。

純粋な建築とは


単純なjqueryスクリプトから大規模なSPAアプリケーションの開発まで、Web開発は長い道のりを歩んできました。そして今、Webアプリケーションは非常に大きくなり、ビジネスロジックの量はサーバー、デスクトップ、およびモバイルアプリケーションに匹敵するか、さらにはそれ以上になっています。

複雑で大規模なアプリケーションを作成し、ビジネスロジックをサーバーからWebアプリケーションに転送してサーバーのコストを節約する開発者にとって、Clean Architectureはコードを整理し、問題なく大規模に拡張するのに役立ちます。

同時に、タスクがランディングページのレイアウトとアニメーションだけの場合、Clean Architectureには挿入する場所がありません。ビジネスロジックがバックエンドにあり、データを取得してクライアントに表示し、ボタンのクリックを処理することがタスクである場合、Clean Architectureの柔軟性は感じられませんが、アプリケーションの爆発的な成長のための優れた出発点になる可能性があります。

すでに適用されている場所？


純粋なアーキテクチャは、特定のフレームワーク、プラットフォーム、またはプログラミング言語に関連付けられていません。何十年もの間、デスクトップアプリケーションの作成に使用されてきました。そのリファレンス実装は、サーバーアプリケーションのAsp.Net Core、Java Spring、NestJSのフレームワークにあります。また、iOやAndroidアプリケーションを作成する際にも非常に人気があります。しかし、ウェブ開発では、彼はAngularフレームワークで非常に失敗した形で現れました。

私自身はTypescriptだけでなく、C＃開発者でもあるので、たとえば、このアーキテクチャのリファレンス実装をAsp.Net Coreに取り入れます。

アプリケーションの作成を開始します


純粋なアーキテクチャーを、実際のWebアプリケーションに可能な限り近い架空のアプリケーションの例で検討します。これは保険会社のオフィスで、ユーザープロファイル、保険対象のイベント、提案された保険料率、およびこのデータを操作するためのツールが表示されます。



この例では、Functionalのごく一部だけが実装されますが、それから、Functionalの残りの部分をどこにどのように配置するかを理解できます。コントローラーレイヤーからアプリケーションの作成を開始し、最後にビューレイヤーを接続します。そして、作成の過程で、各レイヤーをより詳細に検討します。

コントローラーパターン


コントローラー -アプリケーションとのユーザー対話を担当します。これは、Webページ、デスクトップアプリケーション、モバイルアプリケーションのボタンをクリックするか、Linuxコンソールにコマンドを入力するか、ネットワーク要求、またはアプリケーションに着信するその他のIOイベントにすることができます。

クリーンアーキテクチャで最も単純なコントローラは次のとおりです。

クリーンなアーキテクチャのない通常のコントローラーを取得しました。React.Componentから継承すると、ロジックを備えた動作するコンポーネントが得られます。非常に多くのWebアプリケーション開発者が書いていますが、このアプローチには多くの重大な欠点があります。主なものは、コンポーネント間でロジックを再利用できないことです。結局のところ、推奨される料金は、個人のアカウントだけでなく、ランディングページやサービスにクライアントを引き付ける他の多くの場所にも表示できます。

サービスパターン


サービス -アプリケーションのビジネスロジック全体を担当します。コントローラーがデータを受信、処理、送信する必要がある場合、サービスを通じてこれを行います。複数のコントローラーが同じロジックを必要とする場合、それらはサービスで動作します。ただし、サービス層自体は、コントローラー層とビュー層、およびそれが動作する環境については何も認識してはなりません。

ロジックをコントローラーからサービスに移動して、コントローラーにサービスを実装してみましょう。

どのようなリポジトリが設定されているか、それらがまったく正しいかどうかを判断するときがきました。モデル層がこれを担当します。

モデル：DTO、エンティティ、ViewModels


モデル -アプリケーションが動作する構造の説明を担当します。このような説明は、新しいプロジェクトの開発者がアプリケーションが何と連携しているかを理解するのに非常に役立ちます。さらに、データベースの構築やモデルに保存されているデータの検証に使用すると非常に便利です。

モデルは、使用のタイプに応じてさまざまなパターンに分けられます。
  • エンティティ -データベースの操作を担当し、データベース内のテーブルまたはドキュメントを繰り返す構造です。
  • DTO（データ転送オブジェクト） -アプリケーションの異なるレイヤー間でデータを転送するために使用されます。
  • ViewModel-ビューでの表示に必要な事前準備された情報を含みます。


ユーザープロファイルモデルと他のモデルをアプリケーションに追加し、他のレイヤーに抽象オブジェクトではなく、非常に具体的なプロファイルで作業していることを知らせます。

依存性注入-依存性注入


依存関係の挿入 -依存関係をコントローラー、サービス、リポジトリの各レイヤーに挿入し、これらのレイヤーの外でこれらの依存関係を上書きできるようにします。

プログラムでは、コントローラー層はサービス層に依存し、リポジトリ層に依存します。現在の形式では、レイヤー自体がインスタンス化によって依存関係を引き起こします。そして、依存関係を再定義するために、レイヤーはこの依存関係を外部から設定する必要があります。これには多くの方法がありますが、最も一般的なのは、コンストラクターのパラメーターとして依存関係を渡すことです。

次に、すべての依存関係を持つプログラムを作成すると、次のようになります。
var programm = new IndexPageController(new ProfileService(new ProfileRepository()));

同意する-それはひどく見える。プログラムには2つの依存関係しかないことを考慮しても、すでにひどいように見えます。数百および数千の依存関係があるプログラムについて何を言うか。

問題を解決するには、特別なツールが必要ですが、そのためにはそれを見つける必要があります。Asp.Net Coreなどの他のプラットフォームの経験に目を向けると、プログラムの初期化段階で依存関係の登録が行われ、次のようになります。
DI.register(IProfileService,ProfileService);

そして、コントローラを作成すると、フレームワーク自体がこの依存関係を作成して実装します。

しかし、3つの重要な問題があります。
  1. JavaScriptでTypescriptをトランスパイルするとき、インターフェースの痕跡は残りません。
  2. クラシックDIに分類されたものはすべて永久に残ります。リファクタリング中にそれをきれいにすることは非常に困難です。そして、Webアプリケーションでは、すべてのバイトを保存する必要があります。
  3. ほとんどすべてのビューライブラリはDIを使用せず、コントローラーの設計者はパラメーターで忙しいです。


Webアプリケーションでは、DIはAngular 2+でのみ使用されます。Angular 1では、インターフェイスの代わりに依存関係を登録するときに文字列が使用されましたが、InversifyJSでは、インターフェイスの代わりにSymbolが使用されました。そして、これらすべてが非常にひどく実装されているため、このセクションの最初の例のように、これらのソリューションよりも多くの新しいものを用意する方が良いでしょう。

3つすべての問題を解決するために、私自身のDIが発明され、その解決策は、Java Springフレームワークとその自動配線されたデコレーターを見つけるのに役立ちました。このDIがどのように機能するかについての説明は、リンクの記事とGitHubリポジトリにあります。

結果のDIをアプリケーションに適用する時が来ました。

基本コンポーネントには、フレームワークのライフサイクルからの抽象化も含まれています。それらはすべてのフレームワークで異なりますが、すべてのフレームワークにあります。 AngularはngOnInit、ngOnChanges、ngOnDestroyです。 ReactおよびPreactでは、これはcomponentDidMount、shouldComponentUpdate、componentWillUnmountです。 Vueでは、これは作成、更新、破棄されます。ミスリルでは、oncreate、onupdate、onremoveです。 WebComponentsでは、これはconnectedCallback、attributeChangedCallback、disconnectedCallbackです。そして、すべての図書館で。ほとんどの場合、同じまたは同様のインターフェイスがあります。

さらに、ライブラリコンポーネントを独自のロジックで拡張して、すべてのコンポーネント間で再利用できるようになりました。たとえば、分析、モニタリング、ロギングなどのためのツールの導入。

UserProfileDto. 
import { Serializable, jsonProperty } from "ts-serializable";

export class UserProfileDto extends Serializable {

    @jsonProperty(String, null)
    public firstName: string | null = null;

    @jsonProperty(String, null)
    public lastName: string | null = null;

    @jsonProperty(Date, null)
    public birthdate: Date | null = null;

    public getAge(): number | null {
        if (this.birthdate) {
            const ageDifMs = Date.now() - this.birthdate.getTime();
            const ageDate = new Date(ageDifMs);
            return Math.abs(ageDate.getUTCFullYear() - 1970);
        }
        return null;
    }

    public getFullname(): string | null {
        return [
            this.firstName ?? "",
            this.lastName ?? ""
        ]
            .join(" ")
            .trim() || null;
    }

}

Reduxより優れている点は何ですか？


違いを理解するために、アプリケーションが大きくなったときのReduxの動作を見てみましょう。


図からわかるように、Reduxアプリケーションの成長が垂直方向に拡大すると、ストアとReducerの数も増加し、ボトルネックになります。そして、Storeを再構築して適切なReducerを見つけるためのオーバーヘッドの量がペイロードを超え始めています。

中規模アプリケーションのオーバーヘッドとペイロードの比率を簡単なテストで確認できます。
let create = new Function([
"return {",
...new Array(100).fill(1).map((val, ind) => `param${ind}:${ind},`),
"}"
].join(""));

let obj1 = create();

console.time("store recreation time");
let obj2 = {
    ...obj1,
    param100: 100 ** 2
}
console.timeEnd("store recreation time");

console.time("clear logic");
let val = 100 ** 2;
console.timeEnd("clear logic");

console.log(obj2, val);

// store recreation time: 0.041015625ms
// clear logic: 0.0048828125ms

100個のプロパティでストアを再作成するには、ロジック自体の8倍の時間がかかりました。 1000要素の場合、これはすでに50倍以上です。さらに、1つのユーザーアクションがアクションのチェーン全体を生成する場合があり、その呼び出しをキャッチしてデバッグするのは困難です。ストアを再作成するための0.04ミリ秒は非常に小さく、速度が低下しないことは確かです。しかし、0.04ミリ秒はCore i7プロセッサで1つのアクションに対してです。弱いモバイルプロセッサと、単一のユーザーアクションが数十のアクションを生成できるという事実を考えると、これはすべて、計算が16ミリ秒に収まらず、アプリケーションの速度が低下していると感じられるという事実につながります。

Clean Architectureアプリケーションの成長を比較してみましょう。


レイヤー間のロジックと責任の分離によりわかるように、アプリケーションは水平方向にスケーリングします。コンポーネントがデータを処理する必要がある場合、そのタスクに関連しないサービスに触れることなく、対応するサービスに切り替えます。データを受信すると、1つのコンポーネントのみが再描画されます。データ処理チェーンは非常に短く明白であり、最大4つのコードジャンプで、必要なロジックを見つけてデバッグできます。さらに、レンガの壁に例えれば、この壁の安定性に影響を与えることなく、この壁からレンガを取り外して別のレンガに置き換えることができます。

ボーナス1：フレームワークのコンポーネントの拡張機能


おまけとして、プレゼンテーションライブラリのコンポーネントの動作を補足または変更する機会を得ました。たとえば、少し変更を加えた場合、ビューでエラーが発生した場合の反応では、アプリケーション全体がレンダリングされません。
export class BaseComponent<P, S> extends React.Component<P, S> {

    ...

    public render(): React.ReactElement<object> {
        try {
            if (this.view) {
                return this.view(this, this.props);
            } else {
                return React.createElement("div", {}, "  ");
            }
        } catch (e) {
            return React.createElement(
                "div",
                { style: { color: "red" } },
                `    : ${e}`
            );
        }
    }

}

これで、反応はエラーが発生したコンポーネントのみを描画しません。同じ簡単な方法で、監視、分析、その他のニシキアキを追加できます。

ボーナス2：データの検証


データを記述し、レイヤー間でデータを転送することに加えて、モデルには大きな可能性があります。たとえば、class-validatorライブラリを接続すると、デコレータをぶら下げるだけで、これらのモデルのデータを検証できます。少し改良して、Webフォームを検証できます。
