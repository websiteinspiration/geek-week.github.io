Shopifyが1,000人を超える開発者とコラボレーションした方法



大規模なチームの共同作業を、特にShopifyなどの一般的なコードベースで編成することは困難です。モノリスは1日に40回交換されます。トランクベースのワークフローで開発を追跡し、マスターに毎日400件のコミットを注ぎます。安全な導入には3つのルールがありますが、開発の規模が大きくなるにつれて、それらを遵守することがますます難しくなりました。小さな競合によりメインブランチが壊れ、デプロイが遅いと本番と本番の間のギャップが広がり、重要な変更のデプロイ速度はラグプールリクエストにより遅くなりました。これらの問題を解決するために、マージキュー（メインブランチでのマージの速度を自動化および制御するためのツール）を更新しました。現在、GitHubと統合されており、メインブランチとマージする前に継続的インテグレーション（CI）を開始し、CIに含まれていないリクエストを削除します。展開速度が向上します。

メインブランチ（マスター）の安全な展開と保守のための3つの基本ルール：

  1. マスターは常に（CIを介して）緑である必要があります。これにより、いつでもマスターからデプロイできます。グリーンマスター-これは、メインブランチが常に正常にコンパイルされ、アセンブリのすべての段階を通過することを意味します。そうしないと、開発者はブランチに変更を注入できず、会社全体のプロセスが遅くなります。
  2. マスターは生産に近いはずです。先に行きすぎるとリスクが高まります。
  3. 緊急合併は迅速でなければなりません。緊急事態が発生した場合、迅速に修正できるはずです。

マージキューv1


2年前、私たちはオープンソースのShipit継続的デプロイメントツールでキューの最初の反復をロールアウトしました。私たちの目標は、メインブランチが本番環境から離れすぎないようにすることです。開発者は、マスターと直接マージする代わりに、プールリクエストをマージキューに追加します。Merge Queue v1 Poolリクエストはすぐにメインブランチとマージされませんが、キューに蓄積されます。 Merge Queue v1はキューのサイズを制御し、ウィザードでまだ運用環境にロールアウトされていない変更が多すぎる場合にマージを防止しました。これにより、クラッシュのリスクと起こり得る生産の遅れが減少しました。インシデントの間、私たちはキューをブロックし、緊急修正のためのスペースを提供しました。ブラウザ拡張Merge Queue v1









ブラウザーの拡張機能であるMerge Queue v1を通じて、開発者はGitHubインターフェイスのマージキューにプールリクエストを送信しました。また、キューをバイパスして、緊急時に迅速に修正をロールバックすることも可能にしました。

マージキューv1の問題


マージキューv1はプールリクエストを追跡しましたが、CIシステムはキューにあるプールリクエストで機能しませんでした。いくつかの不運な日—インシデントが展開を中断する必要があったとき— 50以上のプールリクエストがマージキューに蓄積されていました。このサイズのキューを組み合わせて展開すると、数時間かかる場合があります。また、キュー内のリクエスト間でソフトな競合が発生する可能性があるため（2つの独立したプールリクエストはCIを個別に通過しますが、一緒には通過しません）、キュー内のプールリクエストがメインブランチとのマージ後にCIを通過するという保証はありません。

主な問題はブラウザ拡張でした。新しい開発者はそれをインストールするのを忘れることがあり、プールへのリクエストをキューに送る代わりにメインブランチに直接マージすることもありました。これは、デプロイメントがすでにはるかに遅れているか、キューがインシデントのために一時停止されている場合に壊滅的な結果をもたらす恐れがありました。

マージキューv2


今年、キューの2番目のバージョンであるMerge Queue v2をリリースしました。キューのダウンタイムを減らしUIを改善し、ブラウザー拡張機能をより統合されたインターフェイスに置き換えることで、スループットの最適化に焦点を当てました。以前のバージョンのシステムでは解決できなかった問題も解決したかったのです。ウィザードを緑色に保ち、緊急修正をすばやく適用します。さらに、私たちのソリューションは、予測不可能な結果で失敗する信頼できないテストに耐えることでした。


Merge Queue v2は新しいインターフェースを実装しました。私は、GitHubに精通している開発者が直感的に使用できるようにしたかったのです。Terraformのインストールで既に使用しているAtlantisシステムからインスピレーションを得て、コメントベースのインターフェースを作成しました。


コメント

ベースのインターフェイスを備えたマージキューv2マージキューの使用方法を示すウェルカムメッセージが、各プール要求に対して発行されます。各マージはコメントで始まります/shipit。彼はシステムにWebフックを送信し、新しいプールリクエストを報告します。キューに追加する前に、プールリクエストがCIを通過し、レビュー担当者によって承認されていることを確認します。成功した場合、このコメントはGitHub GraphQLのaddReactionを通じて肯定的な絵文字で回答されます。

addReaction(input: {
  subjectId: $comment_id
  content: thumbs_up
})

他のプールリクエストコメントは、無効なブランチやレビューの欠落などのエラーを報告します。

addComment(input: {
  subjectId: $pr_id
  body: $error_message
})

キューをバイパスしてマスターに直接マージすると、全体的なスループットが低下するため、キュー統合プロセスの一部であるGitHubブランチ保護機能を使用してマスターに直接マージする機能をプログラムで無効にしました。

createBranchProtectionRule(input: {
        repositoryId: $repository_id
      	pattern: 'master'

      	# This is how we disable to merge pull request button for non-admins.
      	restrictsPushes: true

      	# Admins should be able to use the merge button in case merge queue is broken
      	# The app also depends on this to merge directly in emergencies
 	isAdminEnforced: false
 })

ただし、緊急事態が発生したときに、直接の合併がキューをバイパスする可能性は依然として必要です。これらの場合のために、/shipit --emergencyチェックをブロックする別のコマンドを追加し、コードはマスターに直接流れます。これは、直接的な合併は緊急事態のためにのみ予約されていることを開発者に伝えるのに役立ち、そのようなプール要求をそれぞれ確認する機会があります。

本枝を緑に保つ


メインブランチをグリーンに保つために、もう一度、いつどのように変更するかを見ました。マスターにマージする前にCIを実行すると、緑色の変更のみがマージされることが保証されます。これにより、壊れたマスターへの呼び出しの回数が減り、ビルドの失敗による遅延を心配することなく展開をスピードアップすることで、ローカル開発の品質が向上します。

ここでは、いわゆる「予測ブランチ」を作成することにしました。ここでは、プールリクエストが結合され、CIが起動されます。これはウィザードの将来のバージョンの可能性がありますが、このブランチは引き続き自由に操作できます。ステートフルシステムとリスク同期をサポートしないようにローカルチェックアウトを回避し、代わりにGraphQL GitHub APIを介してこのブランチと対話します。

GitHubの予測ブランチが目的の状態と一致するように、Reactの仮想DOMと同様のテンプレートを使用します。システムは、メモリ内に目的の状態の表現を作成し、開発したマッチングアルゴリズムを実行します。これにより、GitHub上の状態への必要な変更が実行されます。調整アルゴリズムは、2つのステップで目的の状態をGitHubと同期します。最初のステップは、廃止予定のマージコミットを破棄することです。これらは過去に作成されたコミットであり、ツリーの望ましい状態には不要になりました。 2番目のステップは、不足しているマージコミットを作成することです。それらが作成されると、対応するCIの起動が開始されます。

このスキームでは、キューを変更するときに目的の状態を自由に変更でき、非同期化に耐性があります。


マージキューv2は、コミットのキューでCIを起動します。

メインブランチを準備完了状態（緑）に保つには、後続のプールリクエストのカスケード障害を防ぐために、CIを渡さないキュープールリクエストも削除する必要があります。ただし、他の多くの大規模なコードベースと同様に、メインのモノリスShopifyは、信頼性の低いテストに悩まされています。このため、プールリクエストをキューから削除するかどうかという自信がありません。引き続きテストの改善を行っていますが、現状は現状であり、システムで対応する必要があります。

フォールトトレランスのしきい値を追加し、連続した障害の数がこのしきい値を超えた場合にのみプール要求を削除しました。アイデアは、実際の障害はその後の起動で持続し、誤警報は確認されないということです。しきい値を高くすると精度が向上しますが、時間がかかります。妥協点を見つけるために、信頼できないテストの結果でデータを分析できます。誤検知の確率が25％であるとします。複数の連続した偽陽性の確率を計算します。

フォールトトレランスのしきい値 確率
0 25％
1 6.25％
2 1.5％
３ 0.39％
4 0.097％

これらの図から、しきい値が高くなると確率が大幅に減少することは明らかです。正確にゼロになることはありませんが、しきい値3ですでに十分に近いので、これに近づきます。これは、4回目の連続した失敗で、CIを通過しないキューからプール要求を削除することを意味します。

帯域幅の増加


Merge Queue v2のもう1つの重要なタスクは、スループットを向上させることです。テストに合格したプールリクエストの最大数が各デプロイメントに確実に含まれるようにしながら、デプロイメントは継続的に続行する必要があります。

準備ができたプール要求の一定のストリームを保証するために、Merge Queue v2は、キューに追加されたすべてのプール要求に対してCIをすぐに起動します。この慎重さは、キューがブロックされているインシデント中に非常に役立ちます。 CIはメインブランチとマージする前に実行されるため、インシデントを解決してキューをアンロックする前でも、プールリクエストをデプロイできる状態になっています。次のグラフは、キューのブロック中にキュー内のプール要求の数が増加し、ロックが解除されて準備ができたプール要求がすぐにマージされると減少することを示しています。



各デプロイメントのプール要求の数を最適化するために、それらをキューに分割してパッケージにします。パッケージは、単一のデプロイで処理できるプールリクエストの最大数を指します。理論的には、大きなパケットはキューのスループットを増加させますが、リスクも増加させます。実際には、リスクを上げすぎるとスループットが低下し、分離が困難な障害が発生し、ロールバックの数が増加します。このアプリケーションでは、8つのプールリクエストのパッケージサイズを選択しました。これは、帯域幅とリスクのバランスの一種です。

各時点で、CIはキューからの3つのプール要求を処理します。パッケージの数を制限すると、CIリソースがプールリクエストのセット全体ではなく、すぐに必要なものにのみ使用されるようになります。これにより、コストとリソースの使用を削減できます。

調査結果


Merge Queue v2の導入により、使いやすさ、セキュリティ、および運用における展開スループットが向上しました。現在の規模ではすべての目標が達成されていますが、さらに成長するにつれて、モデルと仮定を改訂する必要があります。次のステップは利便性に焦点を当て、開発者にあらゆる段階で意思決定のコンテキストを提供します。Queue Merge Queue v2は、開発を継続する柔軟性をもたらしました。これは、展開を拡張する計画のほんの始まりにすぎません。
