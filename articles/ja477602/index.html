1つのDeepLinkでWebからアプリに



Androidアプリケーションはサイトやサービスを反映したものであり、多くの場合、便利なシェルで同様の機能を表します。このため、Web上のページとインストールされているクライアント間のナビゲーションの問題が緊急になります。この問題を解決するために、ディープリンクが発明されました。カットの下には、ユーザーがそれらを自宅に実装し、ユーザーがまだアプリケーションをインストールしていない場合のケースをどのように処理したかについての興味深い話があります。

ディープリンクは非常に前に発明されたので、それがないとアプリケーションを想像することは現在では困難です。テクノロジー自体に新しいAndroid APIは必要ありませんが、App Indexingを完了すると、API 17で機能するという事実に遭遇する場合があります。

ディップリンクに戻りましょう。それらの構成は、サポートされているリンクのパターンを記述する、アプリケーションマニフェストのインテントフィルターの一連の設定です。

例えば：

<intent-filter>

…

<data
  android:host="best.memes"
  android:pathPrefix="/memes"
  android:scheme="http" />

<data
  android:host="best.memes"
  android:pathPrefix="/jokes"
  android:scheme="https" />

…

</intent-filter>

これらの簡単な操作を行った後、フィルター設定に一致するリンクをクリックするたびに、ユーザーには、アプリケーションを含む複数のアプリケーションから選択できるようになります。次に、インテントフィルターを設定するアクティビティは、リンクを含むインテントを受け取ります。Intent＃getDataメソッドを使用して取得し、必要なパラメーターを解析した場合、ユーザーを関心のあるセクションにすぐに誘導できます。

実装後、妥当な質問が発生する場合があります。ユーザーがまだアプリケーションを持っていない場合はどうすればよいですか？答えは特別なディップリンクで、この場合は人を市場に導くことができます。十分な注意を払えば、そのようなリンクを自分で生成できますが、すべてのブラウザーとすべてのバージョンのAndroidで動作するという保証はありません。現在、これらの問題の少なくとも一部に対する解決策を提供するサービスがかなりあります。たとえば、OneLinkを使用するAppsFlyerやDynamicLinkを使用するFirebaseなどです。これらはすべてほぼ同じように機能します。DynamicLinkのみが、プリインストールされたGoogleサービスを使用してディップリンクを処理します。

Onelink


OneLink自体がAppsFlyerサーバーにつながります。ユーザーはどのデバイスからネットワークに入ったのかを判断し、適切なアドレスにリダイレクトします。デスクトップ、Android、iOSのリダイレクトを設定できます。 Androidアプリケーションがインストールされると、リンクは通常のディープリンクとしてIntentを介してリンクされます。アプリがない場合は、Google ChromeとGoogle Playが出てきます。

アプリケーションの存在はブラウザによってチェックされます。 Chromeには特別なリンク形式の仕様があり、それがインテントに変換されてシステムに送信されます。アプリケーションがインストールされていない場合に備えて、Google Playへのリンクを設定できます。詳細については、こちらをご覧ください。

一般に、Google Play上のアプリケーションへのリンクを転送して、インストールして起動した後、その一部をスキップすることができます。これは、クエリパラメータのURLを使用して実装されており、次のようになります。

play.google.com/store/apps/details?id=memes.best&url=https%3A%2F%2Fbest.memes%2Fjokes

この場合、best.memes /ディープリンクとしてインストールした後、ジョークはアプリケーションに入ります。デフォルトでは、AppsFlyerはこのように機能しません。ライブラリインターフェースを介してリンクを取得することを提案しています。ディープリンク自体は、サービスサーバーを通じてアプリケーションに転送されているようです。

AppsFlyerLib.getInstance().init(KEY, new AppsFlyerConversionListener() {
  @Override
  public void onInstallConversionDataLoaded(final Map<String, String> map) { }
 
  @Override
  public void onInstallConversionFailure(final String s) { }
 
  @Override
  public void onAppOpenAttribution(final Map<String, String> map) { }
 
  @Override
  public void onAttributionFailure(final String s) { }
}, mContext);

これは非常に不便です。最初に、一部のパラメーターを待つ必要があるのか​​、またはユーザーがアイコンをクリックしただけでパラメーターがないのかどうかを確実に理解できないからです。次に、不要なロックや期待をせずに、アプリケーションの目的のセクションをすぐに開きたいと考えています。また、AppsFlyerはメイン画面を開くことを提案し、パラメーターが到着したとき（および到着した場合）、リダイレクトします。このアプローチは適していなかったため、ユーザーがAndroidデバイスからディップリンクをクリックし、アプリケーションがない場合のケースのパラメーターを使用して、Google PlayにURLを生成しました。ライブラリを待たずにアプリケーションでディープリンクを取得できるように、Onelinkで設定します。

OneLinkは、Slackに取り込もうとするまでは問題なく動作しました。実際のところ、彼はChromeのカスタムタブを使用して、内蔵ブラウザーでリンクを開いています。要するに、これはアプリケーション中に開くブラウザタブであり、一般的なスタイルから抜け出さないようにカスタマイズできます（詳細はこちらを参照してください））この場合、Google Play Webバージョンが開き、インストール後のアプリケーションへのディップリンクは転送されません。同様に、OneLinkをアドレスバーにコピーしてリンクをたどると、ブラウザは動作します。Chrome開発者は、数件前のリリースノートでこのケースについて書いています。要するに、このアプローチでは、アプリケーションがインストールされていない場合、Google Playでのリダイレクトはブラウザーで機能せず、ユーザーはWebにとどまります。OneLinkはこの動作を克服できなかったため、DynamicLinkを使用しました。

ダイナミックリンク


システム内のGoogle Play開発者サービスの深い統合により、デバイス上のターゲットアプリケーションの存在の検証を最適化できます。これはどちらかと言えば閉じたエコシステムであるため、その作業の原理を完全に理解することはできませんでしたが、Chromeが、Google Play開発者サービスが所有する進行中のアクティビティを開いており、ディープリンクでの対処方法を決定していることを示しています。その後、Google Playまたはアプリケーションでリダイレクトが発生します。この場合、ディープリンクはインテントを介して、つまり追加のライブラリ松葉杖なしでアプリケーションに入ります。

主観的には、このアプローチはOneLinkよりも速く機能しませんが、Chromeカスタムタブでリンクを開いたときに機能します。これは、多くのアプリケーションがリンクを使用するため、大きな利点です。
