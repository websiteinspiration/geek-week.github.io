フォーム


フォームはプレゼンテーションロジックの最も重要な（そして実際にはほとんど唯一の）概念であり、ユーザーの操作とデータの印刷、エクスポート、インポートの両方に責任があります。

フォームは論理的に2つの部分に分けることができます。

  • フォームの構造によって、フォームに表示されるデータが決まります。
  • フォームの表示によって、このデータの表示方法が決まります。

形状構造


当然、フォームの構造から始めます。

オブジェクト


フォームを作成するときは、最初にフォームが表示するオブジェクトを決定する必要があります。フォームのオブジェクトとこれらのフォームオブジェクトに表示されるオブジェクトの間で、用語に少し混乱があるかもしれないことに注意してください。したがって、今後、これがコンテキストから明らかでない場合は、「フォームオブジェクト」（最初のケース）と「データベース内のオブジェクト」（2番目のケース）という用語を使用します。

フォームオブジェクトごとに、クラスを設定する必要があります。このクラスは、プリミティブ（組み込み）またはオブジェクト（カスタム）のいずれかです。
FORM currentBalances ' '
    OBJECTS s = Stock, i = Item //  ,  
;
フォームにオブジェクトを追加する順序に従って、オブジェクトの順序付きリストが形成されます。したがって、特定のオブジェクトセットの最後のオブジェクトは、このリストの最大のシリアル番号を持つ（つまり、最も遅い）このセットのオブジェクトと呼ばれます。

フォーム上の各オブジェクトには常に現在の値があります。その変化は、インタラクティブなプレゼンテーションでのユーザーの対応するアクションの結果として、または静的な表現でデータを読み取るプロセスで「実質的に」、プレゼンテーションに応じて発生します。

プロパティとアクション


フォーム上のオブジェクトを定義した後、プロパティとアクションを追加し、上記のオブジェクトを引数への入力として置き換えます。

アクションの追加はインタラクティブプレゼンテーションにのみ関連し、レポートとエクスポートでは無視されることに注意してください。また、フォーム上での表示の観点から見たプロパティとアクションの動作はまったく同じであるため、将来はプロパティという用語のみを使用します（アクションの場合、動作はまったく同じです）。

表示オブジェクト


各プロパティは、フォーム上の1つのオブジェクトにのみ表示されます（これを、このプロパティの表示オブジェクトと呼びます）。デフォルトでは、表示オブジェクトはオブジェクトであり、このプロパティの入力に渡されるオブジェクトセットの最後のオブジェクトです。たとえば、2つのオブジェクト（倉庫と商品、および3つのプロパティ）を持つ現在の残高の形式がある場合、倉庫と商品の名前、および倉庫内の商品の残高は次のようになります。
FORM currentBalances ' '
    OBJECTS s = Stock, i = Item //  ,  
    PROPERTIES name(s), name(i), currentBalance(s, i)
;
次に、倉庫の名前の場合、表示オブジェクトはs（倉庫）になり、商品の名前と残高の場合はi（商品）になります。

ただし、必要に応じて、開発者は表示オブジェクトを明示的に指定できます（たとえば、インタラクティブなビューで、商品ではなく倉庫のテーブルに残りのプロパティを表示します）。

フィルターと並べ替え


フォームごとに、開発者は、フォームで表示/選択できるオブジェクトのリスト、およびそれらが表示される順序を制限するフィルターと順序を設定できます。

フィルターを設定するには、フィルター基準として使用されるプロパティーを指定する必要があります。フィルターは、このプロパティの入力に渡されるオブジェクトのセットの最後のオブジェクトのテーブルに適用されます（つまり、プロパティ表示オブジェクトの定義と同様）。この場合、プロパティ値がNULLではないオブジェクト（シリーズ）のセットのみが表示されます。たとえば、上記のフォームにcurrentBalance（s、i）OR isActive（i）フィルターを追加すると、次のようになります。
FORM currentBalances ' '
    OBJECTS s = Stock, i = Item //  ,  
    PROPERTIES name(s), name(i), currentBalance(s, i)
    FILTERS currentBalance(s, i) OR isActive(i)
;

並べ替えは、注文オブジェクトを表示するフォーム上のプロパティのリストとして定義されます。それ以外は、すべてがフィルターに似ています。

オブジェクトのグループ


プラットフォームには、オブジェクトをオブジェクトのグループに結合する機能もあります。この場合、これらのオブジェクトの「デカルト積」がテーブル/リストに表示されます（つまり、2つのオブジェクト-すべてのペア、3つのオブジェクト-トリプルなど）。
FORM currentBalances ' '
    OBJECTS (s = Stock, i = Item) //       
    PROPERTIES name(s), name(i), currentBalance(s, i)
    FILTERS currentBalance(s, i) OR isActive(i)
;

したがって、フォームの単一のオブジェクトの代わりに、前後のほとんどすべての場所で、オブジェクトのグループを使用できます。

実際、これはドキュメントで行われました。「オブジェクトのグループ」というより一般的な用語はどこでも使用されていますが、この記事では物事が複雑にならないように（そして、複数のオブジェクトで構成されるオブジェクトのグループはそれほど頻繁に使用されていません）、オブジェクトのグループを忘れて検討することにしました。オブジェクトのグループは常に正確に1つのオブジェクトで構成されるため、より複雑な「オブジェクトのグループ」や「オブジェクトのセット」ではなく、どこでも「オブジェクト」という用語を使用します。

プロパティグループ


フォームのプロパティとオブジェクトは、グループに結合することもできます。グループは、インタラクティブ（デフォルトのデザイン）とフォームの階層表現（少し後で）で使用されます。デフォルトでは、グループへのプロパティのバインドはグローバルです（つまり、すべてのフォームのプロパティに一度に設定されます）。ただし、必要に応じて、個々のフォームに対してこのバインドを再定義できます。

列オブジェクト


デフォルトでは、プロパティはその表示オブジェクトに1回だけ表示されます。また、このプロパティの表示オブジェクト以外のオブジェクトの値（アッパーと呼びましょう）として、現在の値を使用しています。ただし、プラットフォームには1つのプロパティを数回表示する機能もあるため、一部の上位オブジェクトの値は現在の値ではなく、フィルターに適したデータベース内のすべてのオブジェクトで使用されます。このプロパティのマッピングにより、一種の「マトリックス」が形成されます-（表示オブジェクト）x（上位オブジェクト）。したがって、そのようなマトリックスを作成するには、フォームにプロパティを追加するときに、列を作成するためにどの上位オブジェクトを使用するかを示す必要があります（これらのオブジェクトを列内オブジェクトと呼びます）。
FORM currentBalances ' '
    //             ,
    //         
    OBJECTS s = Stock, i = Item
    //                 
    PROPERTIES name(i), currentBalance(s, i) COLUMNS (s) HEADER name(s)    
    FILTERS isActive(i), isActive(s)
;

印刷および構造化されたプレゼンテーションは静的です。つまり、フォームが開かれたときにすべてのデータを一度に読み取ります（インタラクティブなものではなく、必要に応じてデータを読み取ります）。

パフォーマンスの説明は、おそらく最も複雑なインタラクティブプレゼンテーションから始まります。

インタラクティブなプレゼンテーション


インタラクティブビューでは、フォームオブジェクトはテーブル形式で表示されます。このテーブルの行は、指定されたフィルターを満たすデータベース内のオブジェクトに対応し、列はプロパティに対応します。

ただし、必要に応じて、プロパティをテーブルの列の形式ではなく、つまりそのすべての行に対して表示することはできますが、フォーム上の個別のフィールドとして、つまりフォームオブジェクトの現在の値に対してのみ表示できます。例えば：
currentBalance ' ' (Stock s) = GROUP SUM currentBalance(s, Item i);
FORM currentBalances ' '
    OBJECTS s = Stock, i = Item
    // currentBalance(s)   ,        
    PROPERTIES name(s), currentBalance(s) PANEL, 
            name(i), currentBalance(s, i)
    FILTERS currentBalance(s, i)
;
フォームのスクリーンショット


フォームオブジェクトの現在の値の変更は、ユーザーがテーブルの現在の行を変更した結果、または特別な検索演算子（SEEK）を使用して作成されたアクションの結果として発生します。

原則として、パネルまたはテーブルにプロパティを表示する方法は、プロパティごとに個別に設定するのではなく、フォームオブジェクト全体に対して設定します。したがって、フォームオブジェクトがPANELとしてマークされている場合、そのすべてのプロパティがパネルに表示されます（つまり、現在の値に対して）。それ以外の場合（デフォルトでは）すべてのプロパティがテーブルに表示されます。パラメータとデフォルトアクションのないプロパティがパネルに表示されます。

対話型ビューのすべてのテーブルはデフォルトで動的です。つまり、データベース内の限られた数のオブジェクトのみが読み取られ、残りはテーブル内の現在のオブジェクトが変更されると読み取られます。この場合に表示されるオブジェクトの数は、テーブルの表示部分の高さに基づいて自動的に決定するか、フォームの作成時に開発者が明示的に設定できます。

また、インタラクティブプレゼンテーションのフォームは完全にリアクティブです。つまり、フォームに影響するデータが変更されると、フォーム上のすべてのデータが自動的に更新されます（Reactなど、一般的な場合のみ）。さらに、これはすべて（同じReactのように）完全な再計算ではなく、SQLサーバーで段階的に行われます。

一般的に、他のテクノロジーと比較して、上位3つの要件をタスクに含めようとすると、人々が宇宙に人を発射するように求められているかのように丸い目をすることがよくあります。通常のユーザーが2番目の要件を満たさなかった場合はバグとして分類されますが、1番目と3番目の要件は、データベースに少なくとも少しのデータ（たとえば、数万件のレコード）が表示されたときに、開発されたフォームが正常に機能することです。

インタラクティブプレゼンテーションは、Webクライアントモード（つまり、ブラウザのWebアプリケーション）とデスクトップクライアントモード（Javaアプリケーション）の両方でサポートされています。デスクトップクライアントは、他のネイティブクライアントと同様に、インターフェイスの応答性がやや優れていますが、最も重要なのは、（主にセキュリティの問題が原因で）機器を操作したり、ブラウザでは使用できないその他の操作を実行できることです。

ユーザーフォーム管理


システムの人間工学を改善するために（全員のフォームを作成しないことを含む）、フォームのインタラクティブなプレゼンテーションを設定するための操作の一部をユーザー自身が実行できます。たとえば、そのような操作は次のとおりです。

  • テーブルの設定（表示される列、順序、フォントなど）、
  • カスタムフィルターとソートの作成、
  • 列値によるデータのグループ化、
  • テーブルを印刷してExcelにアップロードします。

また、開発者は、ユーザーが個別にオン/オフを切り替えることができる、いわゆるフィルターグループを作成できます。例えば：
EXTEND FORM currentBalances //      
    FILTERGROUP stockActive //      ,      ,     / 
        FILTER '' active(st) 'F11' //      ,       F11
    FILTERGROUP bal
        FILTER '  ' currentBalance(st, sk) > 0 'F10'
        FILTER '  ' currentBalance(st, sk) < 0 'F9'
        FILTER ' ' currentBalance(st, sk) 'F8' DEFAULT
        FILTER ' ' NOT currentBalance(st, sk) 'F7'
;
これはユーザーがシステムをカスタマイズするためのすべての可能性ではありませんが、それらのほとんどはまだプレゼンテーションロジックと直接関係がないため、3番目の記事で他の可能性に戻ります。

上記の機能は、ERPプラットフォームの機能を参照している可能性が高いことに注意してください。これは、記事のタイトルとすでに完全に一致していません。一方、最初の記事で述べたように、将来的には、このクラスのプラットフォームを含めて、言語/プラットフォームが置き換えられると主張しているため、これらの機能についてまったく触れないのは誤りです。

オブジェクト演算子


フォームを操作する最も一般的なシナリオの1つは、オブジェクトの追加/削除、および新しいフォームでのオブジェクトの編集です。このようなシナリオを実装するために、プラットフォームには定義済みのオペレーターのセットがあり、フォーム作成オペレーターで必要なアクションを一言で作成できます。

  • NEW-オブジェクトを作成します
  • 編集-オブジェクトの編集
  • NEWEDIT-オブジェクトの作成と編集
  • DELETE-オブジェクトを削除します

また、これらのアクションを新しいセッションで実行する必要があることが非常に多いため（オブジェクトを作成するアクションを、これらのオブジェクトの作成元であるフォーム上のアクションから分離する必要がある場合）、プラットフォームは対応する構文シュガー（同じ名前の演算子と同様に機能するNEWSESSIONおよびNESTEDSESSIONオプション）をサポートします。アクションを作成しますが、オブジェクト自体を操作するオペレーター自体のように、開発者が新しいアクションを作成して名前を付ける必要はありません。例えば：
FORM teams
    OBJECTS t=Team
    //  /         
    PROPERTIES (t) NEWSESSION NEW, EDIT, DELETE
    OBJECTS p=Player
    FILTERS team(p)=t
    //          
    PROPERTIES (p) NEW, DELETE
;

フォームイベント


フォームイベントは、ドメインイベントに続く2番目の主要なメカニズムであり、アクションを実行するタイミングを決定します。

プラットフォームには、特定のユーザーアクションの結果として発生するさまざまなフォームイベントのセット全体がありますが、この記事では、最も頻繁に使用される1つ、CHANGEイベントのみを取り上げます。このイベントは、ユーザーがプロパティ変更/アクション呼び出しを開始したときに発生します。たとえば、キーボードのシステムキー以外のキーを押す、変更するプロパティのフィールド上にある、またはマウスでこのフィールドをクリックするなどです。

サブジェクト領域のイベントについては、フォームのイベントについては、処理を指定できます。指定されたイベントが発生したときに実行されるアクションです。ほとんどのフォームイベントには既にデフォルトの処理がいくつかあり、ユーザーが最も期待する動作をそのまま使用できます（たとえば、前述のCHANGEイベントでは、ユーザーからの入力を要求し、プロパティを入力した値に変更します）。ただし、実際には、フォームイベントで特定の処理を設定する必要がある場合でも、次のような状況が発生することがあります。
changeQuantity (Order o, Book b)  { 
    INPUT q = INTEGER DO { //  
        IF lastOrderDetail(o, b) THEN { // ,     
            IF q THEN //  
                quantity(OrderDetail d) <- q IF d = lastOrderDetail(o, b) WHERE order(d) = o AND book(d) = b; //        
            ELSE //   -  
                DELETE OrderDetail d WHERE order(d) == o AND book(d) == b;   
        } ELSE
            IF q THEN
                NEW d = OrderDetail { //   
                    order(d) <- o;
                    book(d) <- b;
                    quantity(d) <- q;
                }
    }
}  

EXTEND FORM order
    OBJECTS b = Book
    PROPERTIES name(b) READONLY, quantity(o, b) ON CHANGE changeQuantity(o, b)
;
ご覧のように、上記の処理には特定の魔法があります-その中にはサーバーとクライアントの両方を同時に操作します（実際、さらに魔法は、プラットフォームがこの処理のグループ調整と貼り付け操作をサポートするということですが、この機能はすでにこの記事の範囲を超えています）。このようなサーバーがクライアントに接続する可能性は、プラットフォームの重要な機能の1つであり、いくつかのフォームオペレーターで即座にサポートされます。そして、ここで私たちはすでにオペレーターを形成するためにスムーズに動いています。

フォームステートメント


フォームを操作するために、プラットフォームは、フォーカスの制御、ユーザーのフィルターと並べ替え、フォームオブジェクトの現在の値などを許可するいくつかの特別な演算子をサポートしていますが、最も重要なものは、おそらく前述のINPUT値を入力するための演算子です。

この演算子では、プリミティブ型、以前の値、入力が正常に完了した場合に実行する必要があるアクション、および入力がキャンセルされた場合に実行する必要があるアクションを指定できます。例えば：
FORM order
    OBJECTS o = Order
    PROPERTIES(o) customer ON CHANGE {
            INPUT s = STRING DO {
                customer(o) <- s;
                IF s THEN
                    MESSAGE 'Customer changed to ' + s;
                ELSE
                    MESSAGE 'Customer dropped';
            }
        }
;
したがって、プラットフォームがこのようなオペレーターを検出すると、自動的に次のようになります。

  • アクションを実行するプロセスを停止し、
  • ユーザーにリクエストを送信します（この場合、フィールドに値を直接入力します）。
  • 彼から答えを受け取り、
  • 受信した応答を考慮して、停止したアクションの実行を継続します。

INPUTだけがデータ入力演算子ではないことを言わなければなりません。これに加えて、メッセージ表示演算子（ASK）のダイアログフォームもデータ入力を担当します。
DELETE Order o WHERE selected(o);
ASK '   ' + (GROUP SUM 1 IF DROPPED(Order o)) + ' .  ?' DO {
    APPLY;
}
インタラクティブビューのフォームを開く演算子（DIALOG）のダイアログモードと同様ですが、残りのビューを検討した後、このモードとフォームを開く演算子自体に戻ります。

ここでインタラクティブビューを終了し、静的ビューに移ります。

静的ビュー


前述のように、静的表現は、開かれたときにすべてのフォームデータを読み取ります。したがって、これらの表現では、このデータを読み取る順序を決定する必要があります。そのためには、フォームオブジェクトを階層に編成する必要があります。階層内では、オブジェクトのデータが互いに「ネスト」されます。たとえば、オブジェクトAとBがあり、AがBの親である場合、静的ビューでは、Aのデータベース内の最初のオブジェクトのAのすべてのプロパティが最初に表示され、次にBのすべてのプロパティとペア（A、B）のすべてのプロパティが表示されます。 Bからのデータベース内のすべてのオブジェクト、Aからのデータベース内の2番目のオブジェクト、Bからのデータベース内のすべてのオブジェクトなどに同様の情報が表示されます（オブジェクトを相互にネストするこのメカニズムは、インタラクティブ表現の「フラット」ツリーの前述のメカニズムと非常に似ています。 ）

プラットフォームは、フォームの構造に基づいてオブジェクトの階層を自動的に構築しますが、この階層を構築するための完全なアルゴリズムが非常に明白であるとは言えません。

  • まず、オブジェクト間の関係は次のルールに従って構築されます。
  • 関係が構築された後、階層は、オブジェクトAの親が、Aが依存するオブジェクトのリストである最新のオブジェクトBを（直接的または間接的に）選択するように構築されます。

例えば：
FORM myForm 'myForm'
    OBJECTS A, B SUBREPORT, C, D, E
    PROPERTIES f(B, C), g(A, C)
    FILTERS c(E) = C, h(B, D) 
;


ただし、実際には、上記の例のような複雑なフォームは実際には見つかりません。通常、フォームオブジェクトの階層はその構造から非常に明白です。

印刷されたプレゼンテーション


読み取ったデータをグラフィック形式で表示するには、LGPLレポート生成ライブラリ-JasperReportsを使用します。

他のレポートライブラリと同様に、JasperReportsでは、各レポートにデータグループや他のサブレポートを含めることができます。したがって、オブジェクトの階層は次のようにレポートに変換されます。

  • オブジェクトの「チェーン」（つまり、O1、O2、O3、...オン、O2はO1の唯一の直接の子孫、O3はO2の唯一の直接の子孫など）はグループに変換されます。
  • オブジェクトに複数の子孫がある場合、そのような子孫ごとにサブレポートが作成されます。

必要に応じて、オブジェクトのチェーンを「分割」し、このオブジェクトにSUBREPORTオプションを設定することにより、オブジェクトのサブレポートを強制的に作成できます（通常、子孫オブジェクトにデータがない場合でもオブジェクトがデータを表示する必要がある場合に必要です）：

レポートのデザイン


すでに述べたJasperReportsはレポートのデザインを担当します（実際、これはlsFusionでの主な機能です）。フォームデザインと同様に、レポートデザインは通常、ゼロから作成されません。したがって、レポートの最初の起動時に、プラットフォームはフォームの構造に基づいて必要なファイルを自動的に作成し、開発者は特別なプログラムJasperSoft Studioを使用してプレビューフォームからこのデザインを直接変更できます。

ちなみに、残念なのは、lsFusionプラグインがIDEA用であり、Eclipse用ではないということだけです。これは、レポートデザインを開発するために別のプログラムをインストールする必要があるためです（Eclipseでは、JasperReportsサポートをプラグインとしてインストールできます）。一方、IDEAは、言語インジェクションなどのすばらしいものをサポートしています。これは、レポートが格納されているjrxmlファイル内で直接使用しました。その結果、たとえば、フォームのプロパティの使用と名前の変更の検索がサポートされます。それは私たちの命を二度以上救った。まあ、正直に言うと、Eclipseでオートコンプリートを備えたGrammarKitアナログ（私たちは自分で完成させる必要がありました）、スタブインデックス、レイジーカメレオンノードのサポート（本質的には2フェーズパーサー）を見つけませんでした。これはすべて、複雑なサポートにとって非常に重要です言語そして最も重要なのは、大きなファイルやプロジェクトでの効果的な作業のためです。しかし、これは別の問題です。

構造化ビュー


すべての構造化表現（フォーマット）は、条件付きで2つのタイプに分類できます。

  • 階層（XML、JSON）-1つのテキストファイル。オブジェクトの情報は、親オブジェクトの情報内にリスト（配列）の形式で配置されます。
  • フラット（DBF、CSV、XLS）-オブジェクトごとに1つのテーブルファイル。さらに、階層の深さが1より大きいオブジェクトごとに、親オブジェクトのテーブルの「最上位」の行の番号を含む、名前がparentの列がそのテーブルに存在する必要があります。

（追加の列をサポートする必要があるため）階層の深さが1より大きいフラット形式での作業はあまり便利ではないことに注意してください。したがって、フラット形式は通常、単純なフォーム（階層の深さが1未満）での作業にのみ使用されます。他の場合では、通常、階層形式が使用されます。したがって、それらから始めます。

フラットフォーマット


フラット形式では、フォームオブジェクトの各ファイルは次のようなテーブルです。

  • 行は、このフォームオブジェクトのベースにあるオブジェクトです。
  • 列は、表示オブジェクトがこのフォームオブジェクトと等しいプロパティです。

前述のように、フラット形式は通常、単純なフォーム（つまり、1つのオブジェクトで構成される）を扱う場合にのみ使用されます。したがって、そのようなフォームを明示的に作成しないために、プラットフォームでフォームを操作するための特別な構文糖があります。これにより、エクスポート/インポートの場所でこれらのフォームを直接作成できます。
run() {
    EXPORT XLSX FROM item = upper(name(Item i)), currentBalance(i, Stock s), 
            stock = name(s), barcode(i), salePrice(i)
            WHERE (name(i) LIKE '%%' OR salePrice(i) > 10) AND currentBalance(i, s);
}
フラット形式のエクスポートは、SQLのSELECTとよく似ています。さらに、非常に長い間、この演算子にも名前を付けたくなりましたが、それでも最終的には、フォームのエクスポートと均一性を維持することと、インポート（フォームとフラットインポートの両方）との対称性を維持することにしました。

全体として、詳細に進むことなく、フォームの表現を理解しましたが、これらの表現での発見に対処するために残されたものはほとんどありませんでした。

フォームオープニング


オブジェクトのいずれかのフォームを開くと、呼び出しコンテキストから値を渡すことができます。これは、フォームが開かれているビューに応じて、次のように使用されます。

  • インタラクティブビューでは、渡された値が現在のオブジェクトとして設定されます。
  • 静的ビューでは、追加のフィルターがインストールされます。オブジェクトは、送信された値と等しい必要があります。
run(Genre g) {
    SHOW booksByGenre OBJECTS g=g;
    PRINT booksByGenre OBJECTS g=g;
    EXPORT booksByGenre OBJECTS g=g;
}
オブジェクトの転送は、すべての表現でフォームを開くための主要な（そしてほとんど唯一の）一般的なメカニズムです。残りのメカニズムは、特定のビューに依存します。

インタラクティブビュー（SHOW、DIALOG）


フロー制御に関しては、インタラクティブビューでフォームを開くための2つのモードがあります。

  • 同期（WAIT）-ユーザーがフォームを閉じるまで待機します。その後、実行のすべての結果を記録した後で、制御を次のアクションに移します。
  • 非同期（NOWAIT）-クライアントでフォームを開いた直後に次のアクションに制御を移します。

デフォルトでは、フォームは同期モードで開きます。

フォームの場所の観点から見ると、開いたフォームは次の2つの方法で表示できます。

  • ウィンドウのように（FLOAT）-フォームはフローティングウィンドウとして表示されます。
  • ブックマークとして（DOCKED）-フォームはSystem.formsシステムウィンドウでブックマークとして開きます。

デフォルトでは、同期操作モードではフォームはウィンドウとして表示され、非同期モードではブックマークとして表示されます。

しかし、おそらくこのオペレーターの最も重要な機能は、いわゆる対話モード（DIALOG）です。このモードでは、演算子を使用して、特定のオブジェクト（または、必要に応じて複数のオブジェクト）の最後の現在の値を返すことができ、実際には値を入力できます。

したがって、フィールドの値の入力演算子（INPUT）だけでなく、この演算子のダイアログモードで、オブジェクトの初期値（オブジェクトを転送するメカニズムを介して）、および入力が正常に完了した場合に実行されるアクション（ユーザー[OK]をクリックした）、または逆にキャンセルした（ユーザーが[キャンセル]をクリックした）。
FORM booksByGenre
     OBJECTS g = Genre PANEL
     PROPERTIES (g) name
     OBJECTS b = Book
     PROPERTIES (b) name
     FILTERS genre(b) = g
;

EXTEND FORM ordersByGenre

                            book(o) <- b;
                    }    
;

ナビゲーター


ユーザーインターフェイスの最も単純なバージョンでは、ユーザーは、タブを切り替えて、使用するいくつかの固定フォームを開きます。ただし、ユーザーが操作するフォームが多数ある場合は、一度にすべてのフォームを開くのはあまり便利ではありません。このような場合は、いわゆるナビゲーターを使用してフォームの操作を制御できます。この場合、クライアントが起動すると、ナビゲーター自体のみがユーザーに表示され（フォームは開かれません）、ユーザーは必要に応じて自分でフォームを開くことができます。

ある意味で、ナビゲーターはフォームの非常に特殊な種類のインタラクティブなプレゼンテーションであり、大きなウィンドウ（デスクトップ）のモードで要素の階層をすばやく簡単に操作できるようになっています。したがって、ナビゲーターの際立った特徴は、パラメーターなしでアクションのみを表示できることです（最も一般的なのはフォームの開口部です）。さらに、ナビゲーターのデザインもフォームのデザインとは大きく異なります。

ナビゲーターデザイン


ナビゲーターでは、すべての要素が階層に配置されます。アクションに加えて、ナビゲーター要素は、共通機能を1つの場所にグループ化することを主な機能とするフォルダーにすることができます。

ナビゲーター自体のデザインは、多くのウィンドウ（デスクトップコンポーネント）で構成されており、それぞれにナビゲーターのいくつかの要素が表示されます。したがって、ナビゲーターの各要素は、その子孫が描画されるウィンドウに設定できます。したがって、ウィンドウごとに、そのウィンドウに表示されるナビゲーター要素のサブツリーのセットが一意に決定されます。



各ウィンドウは、デスクトップの事前定義されたセクションを占めます。



デスクトップ全体のサイズは100x100ピクセルです。ウィンドウを作成するときは、左上座標、ウィンドウの幅と高さをポイント単位で指定する必要があります。ウィンドウがデスクトップの領域全体を「覆う」ことが望ましい。これが発生しない場合、空き領域はウィンドウの1つに割り当てられます（これは保証されません）。 2つのウィンドウの座標とサイズがまったく同じであることも許可されています。この場合、それらは同じ場所に表示されますが、それらを切り替えるとタブが表示されます。

いつでも、各ウィンドウには、ユーザーが選択した1つの現在のナビゲーターフォルダーがあります。したがって、ナビゲーター要素の場合、その最上位階層フォルダーの1つが別のウィンドウにあり、選択されていない場合、このナビゲーター要素は表示されません。

上記のすべてに加えて、プラットフォームでも：
  • ウィンドウの場合、ツールバー、パネル、ツリー、メニューなどのナビゲーター要素を表示するために使用されるグラフィックコンポーネントのタイプを指定できます。
  • 組み込みのシステムウィンドウがいくつかあります-フォーム、ログ、ステータス、ルート、ツールバー、ツリー。さまざまな典型的なタスクで使用できます（たとえば、ルートは通常、アクションをモジュールに分割するために使用されます）
