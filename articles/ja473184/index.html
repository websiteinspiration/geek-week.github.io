Docker + php-fpm + PhpStorm + Xdebug

少し前に、チームリーダーは言った：みんな私たちの戦闘プロジェクトに同じ開発環境を持たせたい+神経と時間を節約するために、Webアプリケーション、APIリクエスト、およびコンソールスクリプトの両方をすべてデバッグできる必要があります。そして、このドッカーで私たちを助けてください。

否や言うほどない。カットの下の詳細。

ネットワーク上には多くのコンテナ化マニュアルがありますが、それらを実際の戦闘開発に適用するにはどうすればよいですか？プロジェクトごとに、独自のdocker-compose.yml？しかし、私たちのすべてのプロジェクトはapiを介して相互に通信し、それらはすべて標準のテクノロジースタックを使用します：nginx + php-fpm + mysql。

したがって、問題の状態を明らかにしましょう：

  1. 私たちは、会社で、チームで、いくつかの戦闘プロジェクトを行っています。私たちは全員、Ubuntu + PhpStormの下で作業しています
  2. ローカル開発では、各チームメンバーに同じ開発環境を提供するために、また新しい開発者が到着したときに迅速に作業環境を展開できるように、Dockerを使用したいと考えています。
  3. 私たちは快適に開発したいと考えています。Webアプリケーションとコンソールスクリプトの両方、APIリクエストなど、すべてをデビューさせたいと考えています。

ここでも、いくつかの作業プロジェクトをdockerに取り込みます。

バトルサーバーでは、標準のnginx + php-fpm + mysqlバンドルが使用されます。そして、何が問題なのですか？

まったく同じ環境とXdebugをローカルマシンにデプロイし、PhpStormでプロジェクトを構成して作業します。デバッグするには、PhpStormの「チューブ」をオンにします。すべてがそのまま機能し、すべて問題ありません。



これはすべて本当です-すべてが箱から出して動作します。しかし、私たちの作業環境の内部を調べてみましょう。

Nginx + php-fpmはソケットを介して通信し、xdebugはポート9000でリッスンします。PhpStormもデフォルトで、ポート9000でリッスンしてデバッグを行い、すべて正常に動作しているようです。また、PhpStormで複数のアプリケーションを開いており、複数のアプリケーションで盗聴（「チューブ」）が有効になっている場合はどうでしょうか。 PhpStormは何をしますか？ Xdebugの新しい接続が検出されたことを彼は誓います。無視しますか、それとも無視しますか？

つまり、PhpStormのデフォルト設定では、特定の時点で、1つのアプリケーションしかデビューできません。他のすべての開いているアプリケーションでは、デバッグをオフにする必要があります。くそー、それは不便です。デバッグのためにすべてのアプリケーションをリッスンしたいのですが、そのうちの1つにブレークポイントがある場合は、このアプリケーションの必要な行でPhpStormを停止します。

そして、これには何が必要ですか？ただし、各アプリケーションは、Xdebugの独自の設定で開始する必要があります。そのため、各アプリケーションはポートをリッスンし、サーバーを検索しますが、すべてが1つのヒープに共通しているわけではありません。

そして、これには素晴らしいドッカーがあります！ php：7.1-fpmなどの1つの共通イメージに基づいて、各戦闘アプリケーションを個別のコンテナーで実行できます。 dockerテクノロジーのおかげで、最小限のオーバーヘッドでアプリケーションを分離できます。

では、Dockerの下で戦闘プロジェクトを開始し、各プロジェクトを個別のコンテナで実行し、PhpStormで各プロジェクトを個別にデバッグするように構成します。すべてがうまくいくはずです。

そして、おっと、最初の問題：Dockerのコンテナーはrootとして実行され、ローカルでは通常uid 1000、gid 1000のユーザーとして機能します。アプリケーションは機能しており、各アプリケーションにすべてに対する777権限を与えることはオプションではありません。私たちのアプリケーションはgitの下にあり、ローカルで777の権利を与えると、gitはこれをすべて記録してバトルサーバーに転送します。

松葉杖、これがサンプルのphpイメージです：ビルドされる7.1-fpm。

このイメージからコンテナーを開始すると、www-dataユーザーはuidとgidを受け取り、これらは外部から転送されます。

また、コメントでは、トピックが提起されました：なぜwww-dataユーザーの権限をまったく変更するのか、標準の権限が適合しないのはなぜですか。そして毎回次のようなものを実行する必要があります

これらの変数がないと、php-fpmイメージは起動しません。質問：それらをdocker-compose.ymlに渡す方法は？回答：あなたにとってより便利だからです。起動時に次のことができます。
USER_ID=$(id -u) GROUP_ID=$(id -g) docker-compose up -d
