アプリケーションとトースト


常に便利で、プログラムのどの時点でも必要になる場合がある最初のことは、アプリケーションへのリンクです。これは、AndroidManifestに登録されているリンクへの単純なクラスによって解決されます。

class App : Application() {
  
   init {
      APP = this
   }
 
   companion object {
      lateinit var APP: App
   }
}

これのおかげで、常にアプリケーション全体のコンテキストへのアクセスがあり、どこからでもライン/リソースを取得できます。そして、少なくともこれは次の砂糖の粒に必要です：

fun Toast(messageId: Int) {
   Toast.makeText(App.APP, messageId, Toast.LENGTH_LONG).show()
}

fun Toast(message: String) {
   Toast.makeText(App.APP, message, Toast.LENGTH_LONG).show()
}

ささいなことですが、Kotlinとコンテキストにアクセスできるという事実のおかげで、アプリケーションの任意の場所からToastを短時間で簡潔に呼び出すことができます。メソッドがどこからでもアクセスできるようにするために、ルートクラスを指定せずに、ファイル内でマークアウトできます。

画面には誰がいますか？


1つのアクティビティを持つプログラムが人気を集めています。しかし、私たちのアーキテクチャでは、いくつかのアクティビティを使用することが決定されました。少なくとも承認ロジックとアプリケーションの主要部分を分離するため。時間の経過とともに、画面が表示されるかどうか、およびアプリケーションのどの部分であるかを理解する必要がありました。そして将来、アプリケーションのロケールで文字列を取得することも必要になりました。しかし、まず最初に。私たちのAPPの価値が一人で飽きないように、会社は彼を作るでしょう：

var screenActivity: AppCompatActivity? = null
var onScreen: Boolean = false

次に、AppCompatActivityから継承した基本クラスを作成します。

open class BaseActivity : AppCompatActivity() {
   override fun onStart() {
      super.onStart()
      App.onScreen = true
   }

   override fun onStop() {
      super.onStop()
      if (App.screenActivity == this) {
         App.onScreen = false
         if (isFinishing())
            App.screenActivity = null
      }
   }

   override fun onDestroy() {
      super.onDestroy()
      if (App.screenActivity == this)
         App.screenActivity = null
   }

   override fun onCreate(savedInstanceState: Bundle?) {
      App.screenActivity = this
   }

   override fun onRestart() {
      super.onRestart()
      App.screenActivity = this
   }

}

はい。新しいガイドについては、適切な場所でアクティビティを含むライフサイクルにサインアップできます。しかし、私たちは私たちが持っているものを持っています。

ローカライズされた文字列


怪しげなユーティリティの機能がありますが、TKはTKです。そのような機能に、システムの言語ではなくアプリケーションの言語の選択を含めます。長い間、プログラムで言語を置き換えることができるコードがあります。しかし、私たちは1つのバグに直面しました。このバグの本質は、アクティビティコンテキストではなくアプリケーションコンテキストを介して行を取得すると、その行がシステムロケールに戻ることです。また、アクティビティのコンテキストをスローすることが常に便利であるとは限りません。以下の方法が助けとなりました：

fun getRes(): Resources = screenActivity?.resources ?: APP.resources

fun getLocalizedString(stringId: Int): String = getRes().getString(stringId)

fun getLocalizedString(stringId: Int, vararg formatArgs: Any?): String = getRes().getString(stringId, *formatArgs)

これで、アプリケーションのどこからでも、正しいロケールで文字列を取得できます。

SharedPreferences


すべてのアプリケーションと同様に、いくつかの設定をSharedPreferencesに保存する必要があります。そして、人生を簡素化するために、それ自体が少しのロジックを隠すクラスが発明されました。まず、APP変数に新しい友達が追加されました。

lateinit var settings: SharedPreferences

アプリケーションの起動時に初期化され、いつでも利用できます。 

class PreferenceString(val key: String, val def: String = "", val store: SharedPreferences = App.settings, val listener: ModifyListener? = null) {
   var value: String
      get() {
         listener?.customGet()
         return store.getString(key, def) ?: def
      }
      set(value) {
         store.edit().putString(key, value).apply()
         listener?.customSet()
      }
}

interface ModifyListener {
   fun customGet() {}

   fun customSet() {}
}

もちろん、変数のタイプごとにそのようなクラスを生成する必要がありますが、シングルトンを必要なすべての設定で置き換えることができます。次に例を示します。

val PREF_LANGUAGE = PreferenceString("pref_language", "ru")

そして今、あなたは常にフィールドとして設定を参照することができ、リスナーを介したロード/保存と通信は隠されます。

日付形式


マルチスレッド。時にはそれは怖い言葉です。バックグラウンドで日付の行を形成すると、非常に奇妙なバグを見つけました。SimpleDateFormatはスレッドセーフではないことがわかりました。したがって、次のものが生まれました。

class ThreadSafeDateFormat(var pattern: String, val isUTC: Boolean = false, val locale: Locale = DEFAULT_LOCALE){
   val dateFormatThreadLocal = object : ThreadLocal<SimpleDateFormat>(){
      override fun initialValue(): SimpleDateFormat? {
         return SimpleDateFormat(pattern, locale)
      }
   }

   val formatter: SimpleDateFormat
      get() {
         val dateFormat = dateFormatThreadLocal.get() ?: SimpleDateFormat(pattern, locale)
         dateFormat.timeZone = if (isUTC) TimeZone.getTimeZone("UTC") else timeZone
         return dateFormat
      }
}

そして使用例（はい、これは再びシングルトンの中で使用されます）：

private val utcDateSendSafeFormat = ThreadSafeDateFormat("yyyy-MM-dd", true)
val utcDateSendFormat: SimpleDateFormat
   get() = utcDateSendSafeFormat.formatter

アプリケーション全体で、何も変更されておらず、スレッドの問題は解決されています。

キーボード


常に必要なもの。キーボードをすぐに表示する必要があり、その後、ある時点でキーボードを非表示にする必要があります。そして、次の2つの方法が必要です。2番目のケースでは、ルートビューを渡す必要があります。

fun showKeyboard(view: EditText){
    view.requestFocus();
    (App.APP.getSystemService(Context.INPUT_METHOD_SERVICE) as InputMethodManager?)
            ?.toggleSoftInput(InputMethodManager.SHOW_FORCED, InputMethodManager.HIDE_IMPLICIT_ONLY)
}

fun hideKeyboardFrom(view: View) {
    (App.APP.getSystemService(Activity.INPUT_METHOD_SERVICE) as InputMethodManager?)
            ?.hideSoftInputFromWindow(view.windowToken, 0)
}

そして、多分誰かが便利になるでしょう、トーストを表示しながら任意の行をクリップボードにコピーする機能：

fun String.toClipboard(toast: Int) {
    val clip = ClipData.newPlainText(this, this)
    (App.APP.getSystemService(Context.CLIPBOARD_SERVICE) as ClipboardManager?)?.setPrimaryClip(clip)
    Toast(toast)
}

RecyclerViewとTableLayout


そして、この小さな記事の最後に、私が最近解決しなければならなかったことを共有したいと思います。多分誰かが重宝するでしょう。

初期データは次のとおりです。

  1. 1k +表示するデータ。
  2. 各「セル」は約10のフィールドで構成されます。
  3. スワイプ、クリック、ダブルクリック、ロングクリックをキャッチする必要があります。
  4. そして…。データは300〜500ミリ秒ごとに更新されます。

最初のポイントを忘れた場合。次に、最も効果的なソリューションはTableLayoutです。なぜRecyclerViewではないのですか？そして、3点と4点のため。シート内には最適化があり、ビューを再利用しますが、常にではありません。また、新しいビューの作成時には、タッチハンドラーは存在しません。そして、これがスワイプにのみ影響する場合は問題ありませんが、時々問題は通常のタップで再現されます。ビューで直接データを更新しても、通知せずに役に立ちません。そのため、TableLayoutを使用することにしました。そして、データが100以下になるまで、すべてが順調でした。それから、フリーズの世界へようこそ。

私は解決する2つの方法、またはTableLayoutにセルを再利用してスクロール時に魔法をかける方法を教えました。または友達RecyclerViewと頻繁な更新を作ってみてください。そして、私は2番目の方法で行きました。タッチとスワイプ（主にスワイプが原因）はView.OnTouchListenerに基づく自己記述クラスによって処理されたため、dispatchTouchEventメソッドをオーバーライドすることでタッチ処理をRecyclerViewレベルに移動する効果的なソリューションであることがわかりました。

アルゴリズムは簡単です：

  • 触る
  • タッチが飛ぶ子をfindChildViewUnder（x、y）で決定する
  • LayoutManagerから要素の位置を取得します
  • MotionEvent.ACTION_MOVEの場合は、以前と同じ位置で作業しているかどうかを確認します
  • タッチする組み込みロジックを実行する

おそらく将来的にはこの方法で問題が発生する可能性がありますが、現時点ではすべてが機能しており、問題はありません。
