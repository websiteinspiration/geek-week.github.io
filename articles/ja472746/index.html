管理者用のターミナルサーバー。単一のSSHギャップではない



作業中にさまざまなサーバーへのSSHセッションを多数維持する必要がある場合は、別のWi-Fiに切り替えたとき、またはインターネットが一時的に失われたときに、セッションが簡単に中断することを知っているでしょう。しかし、これらすべての問題が長い間解決されており、セッションの中断と継続的な再接続を忘れることができると私が言ったらどうでしょうか？

ラップトップのふたを開くと、数十のSSHセッションがすべてすぐに利用可能になり、去ったときと同じ状態になっています。この資料では、システム管理者用のターミナルサーバーを構成する方法について説明します。このようなサーバーを使用すると、壊れたSSHセッションを忘れて、常に再接続してパスワードを入力することができます。



サーバーのチューニング


アイデアはシンプルで、投稿タイトルの画像に明確に示されています。すべてのSSH接続を特別なターミナルサーバーに保持します。このサーバーは、他のサーバーを管理するためのエントリポイントになります。同時に、最終的なサーバーで追加のソフトウェアを構成またはインストールする必要はありません。
ほぼすべての構成がターミナルサーバーに適していますが、各セッション内にコンソールログを格納するRAMを増やし、いつでも履歴を上にスクロールして、1か月前にセッションサーバーで実行した内容を確認できるようにすることをお勧めします。通常、1〜2 GBのメモリで十分です。

分布の選択


ターミナルサーバーでは、最も重要なことは稼働時間です。再起動の頻度が少ないほど、SSHセッションのライブ数が増えるためです。したがって、安定したdebianまたはubuntuブランチなど、最も保守的なLTS（長期サポート）ディストリビューションを選択します。プログラムの突然の再起動が予期しないものにならないように、自動更新（無人アップグレード）を構成します。

SSHサーバーのセットアップ


ターミナルサーバーはすべてのサーバーへのアクセスを一度に開くため、安全に保護できます。これを行うために、パスワードを使用した認証を禁止し、キーを使用したアクセスのみを残します。また、rootとしてログインすることも禁止します。
まず、システムに新しいユーザーを作成する必要があります。

/ etc / ssh / sshd_config
.....
#    root
PermitRootLogin no
#   ,  
ChallengeResponseAuthentication no
#   ,   PAM  
UsePAM no
....

SSHサーバーはパスワードでログインしようとすると接続を閉じるだけなので、このような構成はパスワードの大量列挙から保護するには十分です。多数の接続がある場合でも、サーバーに大きな負荷をかけることなく、接続はすぐに終了します。私の意見では、この構成では、fail2banのような追加のセキュリティ機能をインストールする必要はありません。

多くの場合、マニュアルの初心者は、SSHポートを変更し、22ではなく2222などの非標準のポートをインストールすることを勧めています。私の意見では、これはセキュリティを追加しない悪い習慣です。

Tmux —


Tmuxは、仮想端末を管理するための非常に便利なプログラムです。このプログラムがないと、自分の仕事を想像することはできません。最初は混乱して複雑に見えますが、自分を圧倒し、その使用方法を習得すると、それを拒否できなくなります。

tmuxが何かわからない人のために、タブ付きのWebブラウザーを想像してみてください。ただし、サイトの代わりにコンソールセッションがあります。無数のタブを開くことができ、各タブでプログラムを実行できます。同時に、サーバー上で実行されており、いつでも切断できます。実行中のすべてのタブとプログラムは元の場所に残り、それらに戻ることができます。

tmuxがまだインストールされていない場合は、インストールします。
apt install tmux

tmuxの用語では、個別のウィンドウのセットをセッションと呼びます。デフォルトでは1つのセッションのみを使用するため、セッション名はまったく使用しません。ただし、必要に応じて複数存在する可能性があることを知っておくことが重要です。

新しいセッションを作成します。
tmux new

この時点で、1つのウィンドウで新しいセッションを作成し、すぐにそれに接続しました。下に表示される緑色のステータスバーを確認できます。これは、ブラウザのタブ付きバーのようなものです。現在のタブとその隣、サービスメッセージが表示されます。

tmuxステータスバーにはウィンドウ（タブ）の名前が表示されます。
この時点で、SSH接続を閉じてサーバーに再接続しても、実行中のtmuxセッションは、最小化した場合と同じように、実行中のすべてのプログラムと同じ状態のままになります。tmuxセッション内で最上位のプログラムを実行して、切断します。明確にするために、ターミナルウィンドウを完全に閉じ、サーバーに再接続します。

サーバーに再接続した後、以前に起動したセッションに接続します。
tmux attach

また、実行中のプログラムが引き続き機能することを確認してください。この時点で、主要な原理を理解することが重要です。tmuxセッションを開始した後、接続しているかどうかに関係なく、サーバーのバックグラウンドで動作し続けます。

tmuxセッションでは複数の同時接続が許可されるため、これを使用してサーバー上の複数のユーザーが共同で同じコンソールをリアルタイムで見ることができます。これを行うには、全員が同じアカウントで同じサーバーに接続し、tmux attachと入力します。そこで、コマンドラインから直接チャットできます。これは、メッセンジャーでコンソールログを互いに投げ合うのではなく、1つの端末ですぐに機能するためによく使用されます。


Tmuxはウィンドウを複数に分割できます（タブ内の各ウィンドウはペインと呼ばれます）。これは、2つのコンソールを同時に表示する必要がある場合に便利です。たとえば、あるウィンドウでスクリプトを編集し、別のウィンドウでログを確認します。

tmuxを使用すると、1つのウィンドウ内に複数のウィンドウを作成してサイズを変更できます。

デフォルトでは、Ctrl + bを使用してtmuxを制御します。このコントロールホットキーを押した後、tmuxはメインコマンドを1文字で入力することを想定しています。

主なコマンドは次のとおりです
。Ctrl + b + c-（作成）新しいウィンドウ（タブ）を作成します
Ctrl + b + <数字> -タブ番号Nに移動します。数字は0から9までのキーです。ウィンドウの番号付けはゼロから始まります。
Ctrl + b + x-現在のウ​​ィンドウを閉じます。最後のウィンドウが閉じられると、tmuxセッションは終了します。
Ctrl + b + w-カーソルボタンを上下に移動できるすべてのウィンドウのリストを表示し、Enterキーを押して目的のウィンドウを選択します。
Ctrl + b + " -ウィンドウを水平に半分に分割して新しい
Ctrl + b +％を作成します-ウィンドウを垂直に分割して新しい
Ctrl + b +を作成します-現在のウ​​ィンドウの名前を変更します
Ctrl + b +下/上/左/右 -移動ウィンドウ内のペイン
Ctrl + b +ページ上/ページ下 -上にスクロールスクロール
Ctrl + b + / -vim 以下のように履歴で検索

これらは、私がtmuxを使用するのに10年以上必要だったすべてのホットキーです。実際、他にもたくさんありますが、最初はこれらにこだわるのが良いでしょう。

Tmuxの構成


Ctrl + bの ホットキーは不快だと思います。なんらかのアクションで3つのキーを押すのは多すぎるからです。tmux configsのテーマは、個別の味の領域であり、経験豊富なユーザーは、それを正しく便利に使用する方法について独自のビジョンを持っています。作者全員がtmuxの構成とテーマを選択することさえできます。

開始点として、私の構成の例を示します。これは、tmuxの迅速な開発を妨げているすべての問題を修正しているようです。設定は〜/ .tmux.confという名前のホームフォルダーにあります。

#  ctrl+b    .   macbook       
set-option -g prefix `

#     <`>  `+a
bind-key a send-prefix

#      
set -g base-index 1
set-option -g base-index 1
setw -g pane-base-index 1

# Lowers the delay time between the prefix key and other keys - fixes pausing in vim
set-option -sg escape-time 1

#     1000 .     
set -g history-limit 1000

#   

# default statusbar colors
set-option -g status-fg white
set-option -g status-bg default

# default window title colors
set-window-option -g window-status-fg default
set-window-option -g window-status-bg default




#       
#------------------

# Respawn windows when PANE IS DEAD
bind-key R respawn-window

#   default   local
new -d -s default -n local

#     irc   irssi
neww -d -n irc irssi

#     superserver   ssh root@superserver.com
neww -d -n superserver ssh root@superserver.com

#     anotherserver   ssh root@123.123.123.123
neww -d -n superserver anotherserver root@123.123.123.123


この構成では、起動時に複数のウィンドウを自動的に作成し、SSHセッションをすぐに開始できます。この場合、tmux newコマンドを使用して新しいセッションを手動で作成する必要はありません。常にtmux attachと入力するだけで十分です。セッションが以前に存在しなかった場合は、作成されます。

tmuxの自動起動


ターミナルサーバーに接続すると、サーバーが再起動されてtmuxセッションが閉じられた場合でも、すぐにtmuxに入ります。

これを行うには、〜/ .bashrcファイルの最後にtmux を追加します。このような設計は上記の構成でのみ機能することを覚えておくことが重要です。
if [ ! "$TMUX" ]; then
 tmux attach
fi

if [ "$TMUX" ]; then
 export TERM=screen
fi


この単純な条件は、tmuxを使用していない場合は、それに接続されていることを意味します。

これで、ターミナルサーバーのtmux構成が完了しました。これからは、新しいSSH接続ごとに、tmuxに個別のウィンドウを作成します。また、ターミナルサーバーへの接続が失われた場合でも、すべてのSSH接続はアクティブなままです。

モッシュ-休憩なし


次に、常にアクティブになるターミナルサーバーへの継続的な接続を提供する必要があります。ラップトップを数日間閉じて別のwifiネットワークで開いた場合でも、接続は自動的に復元されます。

Moshは通常のOpenSSHサーバー上のアドオンであり、切断を忘れることができます。 Moshは通常のSSHを使用してログインします。その後、別のUDPチャネルが生成され、外部IPアドレスが変更された場合でも、中断後に即座に復元されます。
ターミナルサーバーへの常時接続を維持する必要があるため、サーバーと作業中のコンピューターにのみmoshをインストールします。同時に、リモートサーバーへの接続はすでにtmuxで永遠に存続しているため、リモートサーバーに何もインストールする必要はありません。

サーバーにmoshをインストールします。
apt install mosh


稼働中のコンピューターにmoshをインストールします。すべての主要なオペレーティングシステムで使用できますが、ネイティブクライアントはUnixライクなオペレーティングシステムでのみ使用できます。Windowsバージョンは、CygwinまたはChromeアプリケーションを使用して実装されます。

私はmacOSを使用して、brewパッケージマネージャーからmoshをインストールします。
brew install mosh


ほとんどの場合、moshは追加の構成を必要とせず、そのまま使用できます。sshコマンドの代わりにmoshを書くだけで十分です：
mosh user@my-server.com


非標準構成の場合、コマンドは少し複雑に見えます。たとえば、ポートとキーパスを指定する必要がある場合：

mosh --ssh="ssh -p 2222 -i /path/to/ssh.key" user@my-server.com


Moshは標準のSSHクライアントとしてプライマリ認証を実行し、標準ポート22を承認します。同時に、moshサーバーは最初はポートをリッスンしません。また、元のOpenSSHデーモンを除いて、サーバー上で開いているポートはありません。TCP経由で接続した後、moshはユーザー空間のサーバーで実行され、UDP経由で追加のトンネルを開きます。


Moshプロトコルの動作スキーム

これで、クライアントで実行されているMoshセッションは、インターネットが表示されたときに常に復元されます。私のラップトップでは、再起動せずに1つのセッションを数か月間開いたままにしているため、ターミナルサーバーに常にログインし続ける必要はなく、常に機能します。

ターミナルサーバーに長い接続コマンドを毎回入力しないようにするために、接続コマンドのエイリアスを1文字から作成しました。

alias t='mosh --ssh="ssh -p 443 -i /path/to/ssh.key" user@my-server.com'


結論



この単純なスキームにより、時間と神経を大幅に節約でき、SSHが壊れたときに作業の結果を失うことはありません。私は常に、初心者の管理者が毎回サーバーにログインし始めて、スティッキーSSHセッションを強制終了する方法を確認する必要があります。
一見混乱するように見えるかもしれませんが、自分に打ち勝ち、それに慣れたら、SSH接続がまだ切断されている人たちに、後悔して後悔するようになることを保証します。
