オブジェクト間の特殊効果


ページ間の遷移と同様に、フィルターを使用して、同じDOMオブジェクトの2つの状態を切り替えることができます。これはリッチハリスのラムジェットに似ていますが、彼は2つの州の間で変形するのではなく、映画のようなスタイルでミックスします。

実際、これらのフィルターはCSSトランジションまたはアニメーション化されたCSSクロスフェード（）関数に似ています。

まず、要素に混合フィルターを適用します（600 ms）：

img.style.filter = 'blendTrans(duration=0.600)';

次に、オブジェクトに変更を加える前に、その現在の状態をフリーズします。

img.filters.blendTrans.apply();

最後に、画像ソースを変更してトランジションを呼び出します。

img.src = 'different-src.jpg';
img.filters.blendTrans.play();

CSS式


7番目のバージョンより前のInternet Explorerには、CSS式と呼ばれる優れた機能がありました。これは、「動的プロパティ」とも呼ばれるCSS式です。実際、これらはCSS関数でラップされたJavaScriptフラグメントであり、この関数での計算結果がCSSプロパティの値となりました。これらは、CSS Houdiniおよび機能calc()とその他の機能の前身の1つと見なすことができます。

たとえば、独自の関数を書くことができるmin()とmax()：

width: expression(Math.min(this.parentElement.clientWidth, 400) + 'px');

上記のコードは、要素の幅を400pxを超えるまで親の幅に設定します。彼女はこれで、おおよそどのように行動するかで止まりmax-widthます。キーワードthisは現在のアイテムを参照します。

IEは8番目のバージョンの疑似要素のみをサポートするため、CSS式を使用してそれらを「ポリフィル」できます。次に例を示します。

zoom: expression(
    this.runtimeStyle.zoom = '1',
    this.insertBefore(document.createElement('span'),(this.hasChildNodes()
                        ? this.childNodes[0]
                        : null)).className='before',
    this.appendChild(document.createElement('span')).className='after'
  );

上記のコードは、比較的無関係なCSSプロパティに割り当てられていますzoom。まず最初に、それは式のそれ以上の実行を無効にし、静的な値に置き換えます1。これにより、新しい起動ごとに新しい要素が作成されるのを防ぎます。次に<span>、クラス名.beforeとを使用して、コンテンツ領域の最初と最後に入力する要素を作成します.after。 Internet Explorer 8は疑似要素をサポートした最初のバージョンでしたが、同時にCSS式のサポートを拒否したため、上記のコードは疑似要素でブラウザーに悪影響を与えません。

IEはすぐに悪い評判を得たので、ほとんどすぐにCSS式を放棄しました。重要なのは、CSS式を使用すると、CPUに非常に簡単にロードでき、レンダリング速度が遅くなることです。CSS式の「問題」は、サイズ変更、スクロール、マウスの移動など、ブラウザーによって登録されたすべてのイベントの後に実行されることでした。次の例を見てください。

background: expression('#'+Math.floor(Math.random()*16777216).toString(16));

このコードはランダムな色を計算し、背景に配置します。次のビデオ（色の点滅によるてんかんの警告）でその様子を確認できます。


マウスを動かすたびに背景色がどのように変化するか確認してください。これはパフォーマンスにとって非常に悪いため、主要なWeb開発者はCSS式を使用したり、それらを実際のJavaScriptで置き換えたりしないようにアドバイスしています。しかし、当時私を含め、JavaScriptを書くのが得意だった人はほとんどいませんでした。今日、それはすべてコードの品質に依存すると言います。CPUと貧弱なJavaScriptコードもロードできます。この問題の1つの解決策が疑似要素のコードに示されています。最初の実行後に式が無効になり、静的this.styleまたはthis.runtimeStyle（CSSカスケードでインラインスタイルよりも優先度の高いスタイルオブジェクトを表す、Microsoft独自の発明です）。値を動的に保つ必要がある場合は、コードを変更し、必要な場合にのみ高価な計算を実行できます。

<script>

  window.updateWidth = false;

1998年に、Microsoftは、私たちが今日知っているCSS HoudiniおよびWeb Componentsに近い方法を提案しました。彼らは彼ら自身をHTMLコンポーネントと呼んだ：

« HTML . , HTML HTML- . HTML- ; HTML- ( HTC) , HTML, CSS JavaScript.

実際、Microsoft の研究は、Web Componentsの議論の初めに先行技術モデルとして認識されていました。

最初のHTMLコンポーネントは1999年にInternet Explorer 5によって実装され、最後は2010年にInternet Explorer 9によって実装されました。

HTMLコンポーネントには、添付動作、要素動作、デフォルト動作の3つのタイプがありました。

付属行動


アタッチされた動作の動作は、CSS Houdini Workletと同様に.htc機能し、アタッチされた要素に新しい機能を追加するファイル（）があります。添付ファイル自体はCSSを介して行われます。

img {
  behavior: url(roll.htc);
}

ファイル.htcは、外部との通信用の特別なXMLマークアップと、要素の動作を決定するスクリプトブロックで構成されていました。添付されたDOM要素はglobalとして導入されましたelement。以下の例の次のアタッチされた動作は画像要素に適合されており、マウスカーソルが置かれるたびにソースが変更されます（以下のすべての例の作成者であるJasper Pierreに助けてくれてありがとう）：

<public:attach event="onmouseover" onevent="rollover()" />
<public:attach event="onmouseout" onevent="rollout()" />
<script>
var originalSrc = element.src;
function rollover() {
    element.src = "rollover-" + originalSrc;
}
function rollout() {
    element.src = originalSrc;
}
</script>

要素の動作


要素の動作は少し進んで、.htc動作をファイルに転送するだけでなく、マークアップも転送して、カスタム要素を作成します。これは、Webコンポーネント標準のカスタム要素とよく似ています。また、外側（Light DOM）からは簡単に見えますが、内側からはShadow DOMの複雑な構造が隠されています。Shadow DOMのMicrosoftバージョンはViewlinkと呼ばれます。ビューリンクのサポートは明示的に指定する必要があります（オプトイン）。Shadow DOMと同様に、内部構造を操作しようとするドキュメントスタイルや外部スクリプトから保護します。

「Viewlinkは完全にカプセル化された動的HTML（DHTML）動作を記述し、信頼できるユーザー要素としてWebページにインポートできるようにする要素動作関数です。」

要素の動作については、CSSを使用してHTML要素を動作にバインドするだけでは不十分でした。代わりに、XMLの機能を使用して、カスタムコンポーネントの新しい名前空間を作成する必要がありました。

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN">

棺桶の蓋の最後の釘は、Internet Explorer 6のリリース後、新しいバージョンのInternet Explorerを新しいバージョンのWindowsにリンクするというMicrosoftの決定でした。したがって、彼らはInternet Explorerチームを解任し、Windows製品グループに統合しました。残念ながら、Windowsの次のバージョンであるWindows XPの後継であるコードネームLonghorn（後のWindows Vista）は大幅に遅延しました。開発は非常に無秩序だったので、プロセスを再開する必要さえありました。また、新しいInternet Explorerのリリースを延期し、バグを修正したり、既存のテクノロジを改善したりせずに、ブラウザを不安定なままにしました。マイクロソフトが5年後に目覚めたとき、それはすでに遅すぎました。W3Cは新しい標準を開発し、他のブラウザ開発者はそれらを実装しただけでなく、さらに革新をもたらしたWHATWGワーキンググループを設立しました。マイクロソフトは技術的リーダーシップを失い、市場シェアを失い、このショックから回復したことはありません。
