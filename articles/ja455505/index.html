1. CSS-in-JSの代わりにCSSを使用する


古いバージョンのアプリケーションでは、styled-componentsライブラリを使用しました。なぜこれが悪いのですか？問題は、プレーンCSSの方が高速で、占有するスペースが少ないことです。最近のブラウザーは、JavaScriptバンドルと並行してCSSコードをロードできます。さらに、通常のCSSを使用するために追加のライブラリは必要ありません。 styled-componentsの縮小版には約54 KB必要です。 styled-componentsの代わりに通常のCSSを使用すると、アプリケーションコードの読み込みが速くなり、スタイルを変更するときにシステムが実行する必要のある計算が少なくなるという事実につながりました。


styled-componentsライブラリを削除して通常のCSSに切り替えるだけで、サイトの読み込み時間を約0.3秒短縮

できます。CSSがCSS-in-JSスキームを実装するテクノロジーよりも優れたパフォーマンスを実現できる場合、開発者がこのテクノロジーを使用する理由に疑問を感じるかもしれません。コンポーネントの様式。CSS-in-JSを選択する理由の1つは、このテクノロジーによってスタイルの範囲を制限し、グローバルなスタイル設定を放棄できることです。アプリケーションテーマの操作に使用すると便利です。そして誰かが、たぶん、Reactアプリケーションをそのようにスタイル設定したいだけかもしれません。

2.大規模なCSSライブラリの回避



material-uiパッケージの分析

私はMaterial Designの大ファンです。material-uiと呼ばれる素晴らしいMaterialライブラリがReact用に書かれています。このライブラリの問題は1つだけです。これは彼女のサイズです。彼女はとても素晴らしいです。個別のコンポーネントのみを使用する場合でも、CSS-in-JSメカニズムの実装はバンドルに含まれ、これは約30 KBの縮小コードです。

代替案は何ですか？私は独自のコンポーネントを作成し、アプリケーションを作成する過程でそれらをスタイリングすることにしました。この選択の理由の1つは、CSSの知識を更新したかったことです。そして、私は長い間CSSコードを書いていません。ただし、他の可能性もあります。特に、CSSフレームワークについて話している。そのサイズは、material-uiのサイズよりもはるかに小さい。たとえば、これらはSpectreとBulmaで、GZIP圧縮後のコードはそれぞれ9および40 KBです。

3.ページの遅延読み込み


だから、あなたは多くのインポートされたページを持つルーターを持っています。いくつかのページについて話している場合、ここでは問題はありません。しかし、ページ数が増えると、サイトが最初に表示される時間が増えます。インポートコマンドは次のようになります。

import NotFound from "pages/NotFound";
import Projects from "pages/Projects";
import Project from "pages/Project";

それを改善するには？幸いなことに、Reactは遅延ページの読み込みを整理できます。同じことがコンポーネントコードにも当てはまり、必要に応じて読み込まれる小さなフラグメントに分割できます。これは次のようになります。

import React, { lazy, Suspense } from "react";

const load = (Component: any) => (props: any) => (
    <Suspense fallback={<Loader />}>
        <Component {...props} />
    </Suspense>
);

const NotFound = load(lazy(() => import("pages/NotFound")));
const Projects = load(lazy(() => import("pages/Projects")));
const Project = load(lazy(() => import("pages/Project")));

5.興味深いと思われるがあまりメリットのないパッケージを取り除く


元のプロジェクトでは、コンポーネントをマウントするときに、ページのロードをアニメーション化するアニメーションをたくさん使用しました。このすべてがページを遅くしただけではありません。これは彼女が彼女がすることができるよりもずっと遅くなった。私はサイトを見て、その可愛らしさに喜びを感じ、ある時点までパフォーマンスについて考えませんでした。しかし、サイトはアニメーション化されただけではありませんでした。他の同様の装飾がありました。たとえば、ページの上部に移動するためのボタン。たとえば、一部の要素のアニメーションによる読み込み。私はこれすべてが好きでしたが、よく調べてみると、たとえば、サイトは実際には最速のデバイスではなく遅くなることがわかりました。さらに、最初はラップトップでのみサイトをテストしたので、すぐにはわかりませんでした。

スライダーコンポーネントもありましたが、その「重量」をあまり考慮せずにページに追加しました。スライドショーに使いました。後で判明したように、このスライダーのコードだけが30 Kbの縮小形式で占有されていました。その後、スライドショーのコンポーネントを自分で作成することにしました。最終的に、そのミニファイされたコードは25 KBかかりました。このボリュームには、スライドショーだけでなくアプリケーションの他の部分でも使用できる、優れたアニメーションライブラリとジェスチャーを操作するためのシステムが含まれていました。そして、私がしたことはサードパーティのソリューションよりもはるかに優れているようです。
