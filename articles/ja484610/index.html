さよならクリーンコード



深夜だった。

私の同僚は、彼が1週間取り組んできたリポジトリにコードを書き込んだところです。次に、グラフィカルエディターを作成し、新しいコードで作業フィールドの形状を変更する機能を実装しました。長方形や楕円などの形状は、端にある小さなマーカーを移動することで変更できます。

コードは機能しました。

しかし、その中には同じタイプの多くの繰り返し構造がありました。各形状（同じ長方形または楕円形など）には、異なるマーカーセットがありました。これらのマーカーを別の方向に移動すると、形状の位置とサイズにさまざまな影響がありました。また、ユーザーがマーカーを移動しているときにShiftキーを押したままにした場合、さらに、サイズを変更するときにFigureの比率を維持する必要がありました。一般に、コードには多くの計算が含まれていました。

問題のコードは次のようになります。

let Rectangle = {
  resizeTopLeft(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeTopRight(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeBottomLeft(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeBottomRight(position, size, preserveAspect, dx, dy) {
    // 10   
  },
};

let Oval = {
  resizeLeft(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeRight(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeTop(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeBottom(position, size, preserveAspect, dx, dy) {
    // 10

let Header = {
  resizeLeft(position, size, preserveAspect, dx, dy) {
    // 10

    // 10   
  },  
}

let TextBlock = {
  resizeTopLeft(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeTopRight(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeBottomLeft(position, size, preserveAspect, dx, dy) {
    // 10   
  },
  resizeBottomRight(position, size, preserveAspect, dx, dy) {
    // 10   
  },

これらのすべての計算、これらすべてのほとんど同じ行は、私を強く魅了しました。

コードはきれいではありませんでした。

同じタイプの線の大部分は、同じ方向の図形の動きが指定された場所で観察されました。たとえば、メソッドにOval.resizeLeft()は、にあるようなコードが含まれていましたHeader.resizeLeft()。問題は、これらのメソッドの両方が、マーカーが左に移動したときに実行される形状の変更に責任があるということです。

同じ形の図形の方法は似ていた。たとえばOval.resizeLeft()、Figureの他のすべてのメソッドと同様でしたOval。これらの方法はすべて楕円形で機能しました。したがって、それらは類似しています。重複したコードはオブジェクトで見つけることができRectangle、HeaderおよびTextBlockテキストブロックので、長方形でした。

考えがある。

コードを異なる方法でグループ化することで、重複する構造を取り除くことができます。たとえば、次のようになります。

let Directions = {
  top(...) {
    // 5   
  },
  left(...) {
    // 5   
  },
  bottom(...) {
    // 5   
  },
  right(...) {
    // 5   
  },
};

let Shapes = {
  Oval(...) {
    // 5   
  },
  Rectangle(...) {
    // 5   
  },
}

次に、構成に頼り、これらの基本的な方法から必要なものを収集できます。

let {top, bottom, left, right} = Directions;

function createHandle(directions) {

let Rectangle = createBox(Shapes.Rectangle, fourCorners);
let Oval = createBox(Shapes.Oval, fourSides);

let TextBox = createBox(Shapes.Rectangle, fourCorners);

私がしたことは、私の同僚が書いたものの2分の1でした。私のバージョンのプログラムでは、繰り返しフラグメントはまったくありませんでした！最もクリーンなコード。特定の方向または特定の図に関連するシステムの動作を変更する必要がある場合は、いくつかのメソッドを書き直すのではなく、1か所で変更できます。

もう深夜だった（うんざりした）。私はリファクタリングの結果をブランチに注ぎ込み、master就寝しました。同僚のだらしのないコードをどのように「組み合わせた」かを誇りに思います。

翌朝


...期待どおりに行きませんでした。

マネージャーは私に密室で話すように誘い、その間彼は私の変更を元に戻すように丁寧に頼みました。それは私に衝撃を与えた。結局のところ、古いコードはごちゃごちゃです。そして私のコードはきれいでした。

私はしぶしぶ要求に応えましたが、指導者や同僚の正しさを理解するのに何年もかかりました。

これは、プログラマーの開発段階の1つです。


「クリーンなコード」への執着と重複を削除したいという欲求は、私たちの多くが通過したプログラマーの開発における段階の1つです。コードに自信がないときは、自尊心とプロとしての誇りについての認識を、測定可能なものに結び付けたいという欲求があります。一連の厳格なリンティングルール、エンティティの名前付けスキーム、プロジェクトファイルの構造、重複コードの最小化。

反復的なコードとの戦いを自動化することは不可能ですが、この戦いは、間違いなく、練習することで簡単になります。通常、変更のたびに、ほぼ同じ構造がプロジェクトに組み込まれたと言えます。その結果、重複するプログラムフラグメントを取り除くことが、コード品質の客観的な指標の改善として認識されています。さらに悪いことに、それはプログラマーの自己認識を歪めます：「私はクリーンなコードを書く人の一人です」。それは他の種類の自己欺瞞と同じくらい人に影響を与えます。

プログラマが抽象化を作成する方法を学ぶと、彼はこれに夢中になるかもしれません。重複するコードを見ると、そうでない場合でも抽象化を見つけるでしょう。そして、そのような慣行の数年後、重複コードは絶対にどこでも検出されます。抽象化は、プログラマにとって新しい才能になるでしょう。誰かが抽象化が美徳であると彼に言うならば、彼はそれを受け入れます。そして、彼は「純粋さ」に頭を下げない人々を非難し始めます。

私の「リファクタリング」と彼へのアプローチの仕方には、2つの重大な欠陥があることがわかりました。

  • -, , . , . ( ), — . . — .
  • -, . . . , . . «» .

汚いコードを書く必要があると言っていますか？番号。私は、それらが「クリーン」と「ダーティ」という用語によって何を意味するかをよく考えることだけを提案します。とんでもないことはありますか？あなたは何らかの形で正しさ、美しさ、優雅さを感じますか？プログラマーの作業の具体的な結果を説明するこのような概念を適用できると確信していますか？これらの概念は、コードの記述および変更方法にどのように影響しますか？

もちろん、そんなことは考えていませんでした。私はコードがどのように見えるかについて多くのことを考えましたが、人間とは何の関係もない人々で構成されるチームとともにコードがどのように開発されたかについては考えていませんでした。

プログラミングは進むべき道です。書き込んだコードの最初の行から現在の位置までどれだけ進んだかを考えてください。最初に関数を取得したり、クラスをリファクタリングしたりしたときに、これらのトリックが複雑なコードを単純化する方法に感心していたと思います。あなたの大義に誇りを感じているなら、あなたはコードの純粋さを追求したいと思うでしょう。しばらくこの欲望に屈してください。

しかし、そこで止まらないでください。クリーンなコードの熱狂的なフォロワーにならないでください。純粋なコードは目標ではありません。これは、私たちが扱っているシステムの非常に複雑なものを何らかの形で理解しようとする試みです。これは、特定の変更がプロジェクトのコードベースにどのように影響するかが不明であるが、疑わしい海のランドマークを探しているときにプログラマーが適用する保護メカニズムです。 

クリーンなコードのアイデアをガイドラインとしてください。そして-このアイデアをリリースします。

読者の皆様！「クリーンなコード」についてどう思いますか？
