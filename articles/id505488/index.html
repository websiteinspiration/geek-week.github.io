12 tips untuk mengimplementasikan TypeScript di Bereaksi aplikasi

TypeScript adalah standar pengembangan front-end modern. Menurut penelitian State of JavaScript , TS jelas menarik bagi para programmer. Menurut survei untuk 2019, hampir 60% responden mencoba TS dan terus menggunakannya, 22% tidak mencoba dan ingin belajar.

Artikel ini adalah kumpulan tips tentang cara menerapkan dan meningkatkan penggunaan TypeScript. Bagian pertama dari tips ini adalah umum mengenai pendekatan dan infrastruktur. Yang kedua adalah beberapa fitur bahasa yang sangat berguna.



Saya tetap dengan konteks aplikasi Bereaksi, tetapi beberapa tips akan relevan di ekosistem lain.

Kiat 1: toleransi kesalahan


Tipe yang benar adalah kepercayaan pada kode tertulis dan fasilitasi refactoring. Agar pengetikan berfungsi untuk Anda, Anda perlu menerapkan secara ketat untuk mengetik cek:

  1. Periksa mode dev untuk segera melihat kesalahan.
  2. Tambahkan validasi ke CI untuk setiap permintaan kumpulan. Ini akan memastikan bahwa cabang utama selalu jenis yang benar.

Ada dua cara utama untuk mengkompilasi file ts: TSC asli dan skrip @ babel / preset. Metode babel menarik: ketik informasi dipotong dari file .ts, yang membuat perakitan cepat. Berikut ini adalah artikel pengantar.

Jenis memeriksa dalam kasus babel ditambahkan tambahan menggunakan fork-ts-checker-webpack-plugin. Dalam mode pengembangan, pemeriksaan harus dilakukan secara tidak sinkron agar tidak memblokir perakitan dan tidak mengganggu penulisan kode. Di permintaan kumpulan, ketik pengecekan, sebaliknya, harus menghentikan pembangunan jika ditemukan kesalahan.

Tip 2: kontrak backend


Komunikasi dengan backend melalui API itu sendiri menyiratkan kontrak pertukaran data. Berguna untuk memisahkan entitas-entitas ini menjadi layer terpisah, menciptakan antarmuka terpisah. Kami menyebutnya DTO - Objek Transfer Data.

Untuk menerapkan lapisan seperti itu, beberapa perjanjian diperlukan. Berikut adalah contoh aturan yang dapat Anda terapkan dalam tim:

  • Nama antarmuka memiliki akhiran Dto: FooBarDto.
  • Terletak di folder khusus.
  • Itu hanya bisa berubah dengan perubahan API.
  • Data antarmuka ini hanya digunakan untuk membaca, pengeditan dilarang.

Keuntungan dari pendekatan ini: sumber data langsung jelas pada antarmuka, lebih mudah untuk mencari kesalahan dan kode debug, memverifikasi data. Jenis-jenis ini mudah didokumentasikan, memberikan deskripsi dan contoh - sesama pengembang akan membantu Anda.

Jika Anda menggunakan alat seperti Swagger untuk menggambarkan API , lihat bagaimana Anda dapat membuat jenis ini secara otomatis.

Kiat 3: alias pada apa saja


Anda dan tim Anda menulis kode yang diketik dengan sangat baik - tidak ada masalah, tidak ada kesalahan, dan tepat waktu. Jika demikian, lanjutkan ke poin berikutnya;)

Pada kenyataannya, ada godaan untuk menghilangkan pengetikan, terutama ketika menerapkan TS dalam proyek yang ada. Saya ingin secara eksplisit atau implisit menggunakan. Ada alasan yang memadai: kode kompleks, tugas mendesak, tidak dapat diandalkan - upaya tidak membayar untuk hasilnya.

Alih-alih mengabaikan masalah, cobalah melegalkannya:

export type TODO_ANY = any;

Pro:

  • Anda tidak menipu dan tidak menulis kruk.
  • Anda secara sadar menunjukkan bahwa di masa depan kode ini layak untuk diselesaikan.
  • Jenis ini mudah ditemukan dengan mencari proyek.
  • Apa pun dapat digunakan secara eksplisit jika jenisnya tidak penting, misalnya seperti ini:

const getArrayLength = (array: any[]) => array.length

Keterbatasan:

  • Terlalu sering menggunakan teknik ini tidak akan bermanfaat.
  • Diperlukan kesadaran dan tanggung jawab untuk menghilangkan lelucon seperti itu di masa depan.

Anda juga dapat menemukannya berguna: 

export type TODO_UNKNOWN = unknown;

Dalam hal apa pun, kompiler akan memungkinkan hampir semua tindakan. Dalam kasus yang tidak diketahui , sebaliknya, hampir semua penggunaan akan menyebabkan kesalahan. Tip 4: jenis untuk perpustakaan


Jika Anda bekerja dengan pustaka js populer, cari tipe di DefinitelyTyped. Kemungkinan besar, mereka sudah ditulis - kita terhubung dan menikmati hidup.

Tetapi bagaimana jika Anda memiliki perpustakaan internal di mana pasti tidak ada tipe? Cobalah untuk membuatnya secara otomatis. Kabar baik bagi mereka yang menggunakan jsDoc: di TypeScript 3.7, compiler tsc telah belajar cara membuat tipe .d.ts dari file .ts dan .js. 

Berita baik lainnya: ada konverter js -> ts. Sebagai contoh, TypeStat membantu saya menangani komponen Bereaksi yang khas.

Berita buruknya adalah bahwa sistem tipe yang baik kemungkinan tidak akan berfungsi secara otomatis. Mengandalkan "kerangka", yang selanjutnya perlu membangun "daging".

Tip 5: jenis di atas jenis


... dan ketik kejar!

Kebetulan jenis ketergantungan eksternal proyek tidak cocok - mereka terlalu umum (dan Anda ingin lebih spesifik), mereka mengandung kesalahan, versi tidak cocok - atau jenis tidak ada.

Biasanya, masalah seperti itu terjadi di perpustakaan internal (yang juga sering diperbarui), sedangkan di yang eksternal sangat jarang. Jika ada masalah, cobalah mendefinisikan / mendefinisikan ulang tipe secara manual.

Kami menjelaskan jenis untuk modul contoh. Saya menekankan bahwa hanya tipe yang didefinisikan ulang, implementasinya tidak berubah:

declare module "example" {
    const foo: { bar: string };
    export default foo;
}

Lebih baik untuk memindahkan override tersebut ke direktori dan file yang terpisah, misalnya mengetik / example.d.ts . Hubungkan semua konten dari pengetikan ke bidang compilerOptions.typeRoots di tsconfig:

{
    "compilerOptions": {
        "typeRoots": ["node_modules/@types", "./typings"]
    }
}

Voila - jenis misalnya diambil dari redefinisi!

Tip 6: styleguide


Jenis atau antarmuka? const atau enum? Bagaimana cara menggambarkan tipe dalam komponen fungsional? Dan di memoized?

Ketika TS baru saja diperkenalkan ke dalam tim, pertanyaan seperti itu muncul secara konstan. Serangkaian pertanyaan serupa muncul untuk pemula dalam proyek: badai berlangsung beberapa minggu, kemudian gaya pengkodean yang biasa diselesaikan.

Memformalkan aturan adalah cara yang bagus untuk menghemat waktu dalam perselisihan dan permintaan holivar. Kumpulkan tim dan buat panduan gaya Anda.

  • Ambil panduan gaya populer sebagai dasar. Saya suka ini: react-typescript-cheatsheet , react-redux-typescript-guide , modern-typescript-dengan-contoh-cheat-sheet.
  • Linter dan formatter kerja - biarkan mesin melakukan pekerjaan kotor.
  • Tambahkan spesifik proyek Anda: template khas, praktik terbaik, praktik terburuk.
  • Komit keputusan ini dan rujuk ke mereka dalam review kode.
  • Berikan kesempatan pada tim untuk mendiskusikan dan mengubah keputusan ini.

Panduan gaya yang dibangun dengan baik mengungkapkan keajaiban CPDD ( Copy Paste Driven Development ): pengembang menulis dengan gaya yang sama, menghabiskan lebih sedikit waktu berkat pelat boiler yang sudah jadi. Tab pertempuran epik vs spasi (maaf, tipe vs antarmuka) tidak diatur , entri pemula ke proyek disederhanakan.

PS: Sumber inspirasi lain adalah perpustakaan utilitas ts. Anda tidak hanya akan menemukan pembantu yang bermanfaat, tetapi juga kode dari pro di sumber. Misalnya, tipe utilitas , SimplyTyped .

Bukan saran - setidaknya TS


Sedikit pengetahuan untuk pengujian diri. Periksa apakah Anda mengerti apa yang sedang terjadi di sini:

const example = {
    a: "foo",
    b: "bar"
};

//  typeof  keyof
type Ex1 = typeof example; // { a: string; b: string }
type Ex2 = keyof typeof example; // 'a' | 'b'

//       
type Ex3 = Pick<typeof example, "a">; // { a: string };
type Ex4 = Omit<typeof example, "a">; // { b: string };
type Ex5 = Partial<typeof example>; // { a?: string; b?: string} 
type Ex6 = Record<'a' | 'b', number>; // { a: number; b: number };

//     
const func = (a: string, b: number) => true;
type Ex7 = ReturnType<typeof func>; // boolean
type Ex8 = Parameters<typeof func>; // [string, number]

Tip 8: tipe literal vs enum


TypeScript memperkenalkan struktur sintaks yang nyaman - enum . Enum digunakan untuk enumerasi akhir. Setelah menguasai enum, pada awalnya saya ingin menulisnya sangat sering. Namun, enum memiliki kritik yang adil mengenai kode yang dihasilkan setelah kompilasi.

Di sisi lain, ada tipe literal - alternatif yang lebih sederhana dan nyaman untuk penghitungan akhir. Dalam beberapa kasus, penggunaan tipe seperti itu lebih tepat.

Contoh kehidupan adalah alat peraga komponen sederhana:

//   enum
enum ColorEnum {
    yellow = "yellow",
    transparent = "transparent"
}

const ExampleEnum = (props: { color: ColorEnum }) => <div>{props.color}</div>;

//    
type ColorLiteral = "yellow" | "transparent";

const ExampleLiteral = (props: { color: ColorLiteral }) => <div>{props.color}</div>;

//   
const Example = () => (<>
        <ExampleEnum color={ColorEnum.yellow} />
        <ExampleLiteral color="yellow" />
 </>);

Dalam contoh ini, kedua komponen memeriksa jenis warna alat peraga. Varian dengan enum ternyata lebih redundan - ini adalah kode tambahan selama kompilasi, dan impor tambahan dalam kode sumber, dan struktur penulisan yang lebih rumit. Varian dengan string literal lebih ringkas, sementara juga berisi cek tipe. Tidak masuk akal untuk menyulitkan kode enum.

Lebih suka string literal untuk pemeriksaan tipe sederhana. Gunakan enum untuk menggambarkan logika yang lebih kompleks. Dalam hal ini, lebih baik memberikan preferensi ke const enum (namun, struktur ini tidak didukung oleh babel , Anda harus menambahkan plugin ).

Tip 9: pernyataan const


Alasan lain untuk menggunakan enum lebih jarang adalah pernyataan konstan.

sebagai const memberitahu compiler bahwa objek ini tidak akan berubah. Oleh karena itu, jenis dapat disimpulkan secara lebih rinci.

const loadAction = {
    start: "LOAD_START",
    done: "LOADED",
    error: "LOAD_ERROR"
} as const;

type LoadActionType = typeof loadAction;
type LoadActionKey = keyof LoadActionType; // 'start' | 'done' | 'error'
type LoadActionValue = LoadActionType[LoadActionKey]; // 'LOAD_START' | 'LOADED' | 'LOAD_ERROR'
//  "as const" LoadActionValue   string

Anda juga dapat menerapkan sebagai const ke array:

const DIRECTION_ASC = 1;
const DIRECTION_DESC = -1;
const sortDirectionTypes = [DIRECTION_ASC, DIRECTION_DESC] as const;
type SortDirection = typeof sortDirectionTypes[number]; // -> 1 | -1

Lebih mudah digunakan untuk transfer. Misalnya, periksa apakah kamus menjelaskan semua opsi:

const sortDirectionDescription: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " ",
    [DIRECTION_DESC]: " "
};
const sortDirectionDescription2: Record<SortDirection, string> = {
    [DIRECTION_ASC]: " "
};
// Error: TS2741: Property '[-1]' is missing // in type '{ [DIRECTION_ASC]: string; }'

PS: Anda dapat membaca lebih lanjut tentang pernyataan konstitusi dalam artikel ini.

Kiat 10: tipe bermerek


Merek adalah utilitas kecil untuk membuat tipe unik ("bermerek"). Biasanya, tipe ini dibuat atas dasar string - primitif, angka.

Misalnya, ketika bekerja dengan id data selalu digunakan. Terkadang sangat penting untuk membedakan satu jenis id dari yang lain (misalnya, id pengguna dari id pesanannya di toko), meskipun keduanya adalah string. Dengan Merek, Anda hanya bisa mendapatkan dua jenis:

type Brand<T, U> = T & { __brand: U };

type UserId = Brand<string, "userId">; // "5435436" â€“  id 
type OrderId = Brand<string, "orderId">; // "42da-fet3-cdsl" â€“  id 

declare const userID = "1111" as UserId; // -

getOrderById(userID);
// Error: TS2345
// Type '"userId"' is not assignable to type '"orderId"'.

Tip 11: jenis serikat yang didiskriminasi


Istilah sulit serikat terdiskriminasi mudah dipahami dalam praktiknya: ia menciptakan persatuan beberapa jenis dengan bagian bersama (diskriminan). Ketika bekerja dengan serikat seperti itu, kompiler ts dapat menyaring opsi dan menentukan jenisnya.

Contoh klasik tentang penggunaan serikat yang didiskriminasi sangat cocok dengan templat peredam. Berkat bidang tipe umum, jenis payload ditampilkan secara otomatis:

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {
        case 'SET_ONE':
            return { ...state, one: action.payload}; // `payload` is string
        case 'SET_TWO':
            return { ...state, two: action.payload}; // `payload` is number
        default:
            return state;
    }
}

Tip 12: tidak pernah


Tidak pernah ada asuransi yang baik terhadap kasus-kasus mentah. Ambil dari paragraf sebelumnya, hanya tanpa 'SET_TWO', dan dengan penambahan typeguard: never;

type Action =
    | { type: 'SET_ONE'; payload: string }
    | { type: 'SET_TWO'; payload: number };

export function reducer(state: AppState, action: Action): AppState {
    switch (action.type) {

        case 'SET_ONE':
            return { ...state,
                        one: action.payload}; // `payload` is string

        default:
            const typeguard: never = action;
            return state;
            // Error: TS2322: Type '{ type: "SET_TWO"; payload: number; }'
            // is not assignable to type 'never'.
    }
}

Compiler menyimpulkan tipe untuk typeguard dan menimbulkan kontradiksi. Di satu sisi - tipe yang tidak, tidak pernah. Di sisi lain, tipe keluarannya adalah 'SET_TWO'. Jenis tidak konvergen, kami mendapatkan kesalahan.

Kasus lain yang menarik (tetapi jarang) menggunakan Never in React adalah untuk melarang alat peraga bersarang atau lewat:

type IconProps = {
    size: 's' | 'm';
    children?: never;
    foo?: never;
};

const Icon: React.FC<IconProps> = () => <div>Whatewer</div>;
    
const Example = () => (<>
    <Icon size="s"> <div /> </Icon> {/*  â€“  children */}
    <Icon size="s" foo="123"/> {/*  -  foo */} 
</>);

Juga tidak pernah secara penuh digunakan untuk tipe bersyarat ( conditional-types ). Inilah artikel keren tentang hal ini jika semua contoh sebelumnya terlalu mudah bagi Anda.
