非同期関数とpromiseを返す関数の違い

単純にpromiseを返す関数と、キーワードを使用して宣言された関数の間には、小さいながらも重要な違いがありますasync。

次のコードスニペットを見てください。

function fn(obj) {
  const someProp = obj.someProp
  return Promise.resolve(someProp)
}

async function asyncFn(obj) {
  const someProp = obj.someProp
  return Promise.resolve(someProp)
}

asyncFn().catch(err => console.error('Catched')) // => 'Catched'
fn().catch(err => console.error('Catched')) // => TypeError: Cannot read property 'someProp' of undefined

ご覧のとおり、どちらの関数も同じボディであり、どちらの場合でも定義されていない引数のプロパティにアクセスしようとしています。2つの関数の唯一の違いはasyncFn、キーワードを使用して宣言されるものですasync。

つまり、JavaScriptはasnycFn、エラーが発生した場合でも、関数がプロミス（成功または失敗）を返すことを保証します。この場合、ブロック.catch()はそれをキャッチします。

ただし、関数の場合、fnエンジンは関数がpromiseを返すことをまだ認識していないため、コードの実行はブロック.catch()に到達せず、エラーはキャッチされず、コンソールに分類されます。

（私の開発環境では）ボディで発生するエラーがPromise.resolve({<body>})「キャッチ」されない1つの詳細に注意を向けたいのですが。

Promise.resolve((()=>{throw "oops"; })())
    .catch(e=>console("Catched ",e));
//  .catch()      ""

しかし、本体で発生するエラーnew Promise()（約transl。：元の「適切な約束」で）は「キャッチ」されます。

(new Promise((resolve,reject)=>{
    resolve((()=>{throw "oops"})())
}))
.catch(e=>console.log("Catched ",e));
// Catched  oops

このステートメントはどうですか：

async function fn() { <body> }

意味的に、このオプションはこれと同等です。

function fn() {
    return new Promise((resolve,reject)=>{
        resolve({ <body> })
    })
}

したがって、以下のコードは<body>内にある場合にエラーをキャッチしますnew Promise()（およそTransl。：元の「適切なPromise」内）：

function fn() {
    return Promise.resolve({<body});
}

したがって、記事の冒頭の例でどちらの場合もエラーを「キャッチ」するにはPromise.resolve()、ではなくを返す必要がありnew Promise()ます。

function fn(obj) {
  return new Promise((resolve, reject) => {
    const someProp = obj.someProp;
    resolve(someProp);
  });
}

async function asyncFn(obj) {
  return new Promise((resolve, reject) => {
    const someProp = obj.someProp;
    resolve(someProp);
  });
}

asyncFn().catch(err => console.error("Catched")); // => 'Catched'
fn().catch(err => console.error("Catched")); // => 'Catched'
