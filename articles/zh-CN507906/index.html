我们在Angular中最大程度地使用了DI-私有提供商的概念

Angular具有非常强大的依赖注入机制。它允许您通过应用程序传输任何数据，在正确的部分进行转换和重新定义。



因此，我们可以使应用程序体系结构更简单，更灵活：可理解的数据流，最小的代码一致性，易于测试或替换依赖项。



尽管如此，DI在应用程序中的使用还是相当有限的。通常，这是引入服务或通过依赖项注入树从上到下传输某些全局数据。



在本文中，我想展示一种处理从DI获得的数据的替代方法。目的：为了简化使用此数据的组件，指令和服务。







如何在Angular中通常使用DI



我在工作中和开放源代码中对Angular代码进行日常审查。通常，在大多数DI应用程序中，它可以归结为以下功能：



  1. 从依赖关系树中获取Angular实体：ChangeDetectorRef，ElementRef等。
  2. 获取服务以在组件中使用它。
  3. 获取令牌的一些全局配置，该配置在上方声明。例如，在根模块中设置API_URL令牌，并在必要时从应用程序中的任何位置的DI中获取它。


当开发人员走得更远并将现有的全局令牌转换为更方便的形式时，这种情况不太常见。这种转换的一个很好的例子是@ ng-web-apis / common包中的WINDOW令牌。



Angular提供了DOCUMENT令牌，因此您可以从应用程序中的任何位置获取页面对象：您的组件独立于全局对象，易于测试，SSR不会破坏任何对象。



如果您经常需要访问WINDOW对象，则可以编写以下令牌： 



import {DOCUMENT} from '@angular/common';
import {inject, InjectionToken} from '@angular/core';

export const WINDOW = new InjectionToken<Window>(
    'An abstraction over global window object',
    {
        factory: () => {
            const {defaultView} = inject(DOCUMENT);

            if (!defaultView) {
                throw new Error('Window is not available');
            }

            return defaultView;
        },
    },
);


当某人第一次从DI树中请求WINDOW令牌时，将执行令牌工厂-他将从Angular接收DOCUMENT对象，并从中获得到window对象的链接。



此外，我建议考虑对这种转换使用不同的方法-当它们直接在组件或指令的提供者中执行时，从而注入结果。



私人提供者

不必这样做。



有时我会遇到这种在组件中处理数据的方式。请不要这样做： 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent implements OnInit {
   organization: Organization;

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}

   ngOnInit() {
       this.activatedRoute.params
           .pipe(
               switchMap(params => {
                   const id = params.get('orgId');

                   return this.organizationService.getOrganizationById$(id);
               }),
           )
           .subscribe(organization => {
               this.organization = organization;
           });
   }
}

我们做得很好



，我上面应用的视频中Erin的报告做得很好。有了它的变体，结果是这样的： 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent {
   readonly organization$: Observable<Organization> = this.activatedRoute.params.pipe(
       switchMap(params => {
           const id = params.get('orgId');
           return this.organizationService.getOrganizationById$(id);
       }),
   );

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}
}


要在模板中使用接收到的数据： 

<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


该代码可以正常工作，并且没有以前方法的缺点：它看起来很整洁，没有多余的字段。如果要使用相似的流扩展组件，则只需再添加一个-使其变得更容易，因为要添加新的流，我们无需触摸前一个的代码。



此外，数据流变得更加透明：只有创建组件类时才创建的流。发布数据时，将显示我们模板中的信息。



让我们变得更酷：私人提供商



让我们仔细看看以前的解决方案。



实际上，该组件独立于路由器，甚至独立于OrganizationService。这取决于组织$。但是在依赖注入树中没有这样的实体，因此我们被迫在组件中执行转换。



但是，如果在数据进入组件之前执行此转换怎么办？让我们为将在其中进行必要转换的组件专门编写Provider。



为方便起见，我们将提供程序放置在组件旁边的单独文件中，获得以下文件结构：







提供程序将位于organization.providers.ts文件中以进行数据转换，并在组件中放置用于接收它们的令牌： 



export const ORGANIZATION_INFO = new InjectionToken<Observable<Organization>>(
   'A stream with current organization information',
);
         :

export const ORGANIZATION_PROVIDERS: Provider[] = [
   {
       provide: ORGANIZATION_INFO,
       deps: [ActivatedRoute, OrganizationService],
       useFactory: organizationFactory,
   },
];

export function organizationFactory(
   {params}: ActivatedRoute,
   organizationService: OrganizationService,
): Observable<Organization> {
   return params.pipe(
       switchMap(params => {
           const id = params.get('orgId');

           return organizationService.getOrganizationById$(id);
       }),
   );
}


定义组件的提供程序数组。ORGANIZATION_INFO令牌的值将从工厂进行获取，我们将在该工厂进行必要的数据转换。



关于DI操作的说明-deps允许您从DI树中获取必要的实体，并将其作为参数传递给令牌值工厂。在其中，您可以从DI获取任何实体，包括使用DI装饰器，例如： 



{
       provide: ACTIVE_TAB,
       deps: [
           [new Optional(), new Self(), RouterLinkActive],
       ],
       useFactory: activeTabFactory,
}


在组件中声明提供者： 



@Component({
   ..
   providers: [ORGANIZATION_PROVIDERS],
})


现在我们准备使用组件中的数据： 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
   providers: [ORGANIZATION_PROVIDERS],
})
export class OrganizationComponent {
   constructor(
       @Inject(ORGANIZATION_INFO) readonly organization$: Observable<Organization>,
   ) {}
}


组件类别随着数据的接收而减少到一行。

<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>

结论



所描述的方法不会使您摆脱所有设计问题。将提供程序添加到任何小事情上也不值得：有时，如果您在方法中使用转换或使用Pipe，则代码会更清晰。



不过，我希望私有提供者可以帮助您简化具有大量依赖性的组件，或者为逐步重构大型逻辑提供替代方法。
