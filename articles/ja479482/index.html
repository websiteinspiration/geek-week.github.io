SARIF SDKとそのエラー


今日、私たちはMicrosoftの別の高品質プロジェクトをテストしています。このプロジェクトでは、PVS-Studioを使用して英雄的にエラーを検索しています。 SARIFはStatic Analysis Results Interchange Formatの略です。これは、静的アナライザーの結果を他のツール（IDE、包括的なコード検証および分析ツール（SonarQubeなど）、継続的インテグレーションシステムなど）と対話および交換するための標準（ファイル形式）です。等SARIF SDKにはそれぞれ、SARIFをサポートするための.NET開発ツールとサポートファイルが含まれています。
SARIFはMicrosoftで開発され、現在はOASIS（オープンスタンダードに携わる非営利団体）によって開発された標準です。SARIFは、アナライザーの結果だけでなく、ツールに関するメタデータだけでなく、ツールの起動方法やタイムスタンプなどのデータも送信するように設計されています。この規格の詳細については、OASIS Webサイトを参照してください。SARIF SDKソースコードは、GiHubのリポジトリからダウンロードできます。プロジェクトのホームページはこちらから入手できます。

プロジェクトについて


SARIF SDKプロジェクトは小さく、799 .csファイルでした（約98,000の空でないコード行）。プロジェクトには、検証から常に除外するテストが含まれています。したがって、対象となるコードの部分は642 .csファイル（約79,000の空でないコード行）でした。もちろん、これでは不十分です。しかし、検証と分析は迅速かつ簡単で、記事の冒頭の写真に反映させようとしました。それにもかかわらず、いくつかの興味深いエラーが見つかりました。それらを見てみましょう。

V3125 [CWE-476]「v1StackFrame」オブジェクトは、nullに対して検証された後に使用されました。チェック行：1182、1171。SarifVersionOneToCurrentVisitor.cs 1182

internal StackFrame CreateStackFrame(StackFrameVersionOne v1StackFrame)
{
  StackFrame stackFrame = null;

  if (v1StackFrame != null)
  {
    stackFrame = new StackFrame
    {
      ....
    };
  }

  stackFrame.Location =
    CreateLocation(v1StackFrame.FullyQualifiedLogicalName,
                   v1StackFrame.LogicalLocationKey,
                   ....);

  return stackFrame;
}

伝統的に、その逆は真実です。最初に、v1StackFrameリンクのnull がチェックされ、次にチェックするのを忘れました。しかし、一つの重要な注意点があります：v1StackFrameとスタックフレーム変数は、論理的に関連します。かどうかを確認し、v1StackFrameは =であるnullの場合、あるStackFrameオブジェクトがします作成されていない、とのスタックフレームが残る=うヌル。この場合、ここにはチェックがないため、プログラムはstackFrame.Locationの呼び出しに依存します。つまり、v1StackFrameの危険な使用の前に分析者が指摘したように、問題は到達しません。このコードは動作します場合にのみ、非ゼロv1StackFrameの値がされているCreateStackFrameメソッドに渡されました。これは呼び出しコードで何らかの形で制御されていたのではないかと思いました。CreateStackFrameの呼び出しは次のようになります。

Frames = v1Stack.Frames?.Select(CreateStackFrame).ToList()

CreateStackFrameはセレクターとして使用されます。渡されたnull等価参照の検証はありません。おそらくどこかでFramesコレクションに入力するときに、これ（ゼロ参照を記録）が制御されますが、私はそれほど深く掘り下げませんでした。結論はすでに明白です-コードは作者の注意を必要とします。
