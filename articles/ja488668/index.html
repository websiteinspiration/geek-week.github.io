CI / CDチェーンとDocker自動化

90年代後半に最初のサイトを作成しました。次に、それらを作業状態にすることは非常に簡単でした。一部の共有ホスティングにApacheサーバーがあり、FTPを介してこのサーバーにログインし、ブラウザーの行にそのようなものを記述できましたftp://ftp.example.com。次に、名前とパスワードを入力し、サーバーにファイルをアップロードする必要がありました。他の時もあり、すべてが今より簡単だった。 過去20年間で、すべてが大きく変化しました。サイトはより複雑になり、運用にリリースする前に組み立てる必要があります。 1つの単一サーバーがロードバランサーの背後で動作する複数のサーバーになり、バージョン管理システムの使用が一般的になりました。





私の個人的なプロジェクトでは、特別な構成がありました。また、サイトを運用環境に展開する機能が必要であり、masterGitHubのブランチにコードを書き込むという1つのアクションのみを実行する必要があることも知っていました。さらに、小さなWebアプリケーションを確実に動作させるために、巨大なKubernetesクラスターを管理したり、Docker Swarmテクノロジーを使用したり、ポッド、エージェント、およびその他のあらゆる種類の問題を抱えたサーバーパークを維持したりしたくないことも知っていました。作業をできるだけ簡略化するという目標を達成するために、CI / CDについて理解する必要がありました。

小さなプロジェクト（ここではNode.jsプロジェクトについて話している）があり、リポジトリに格納されているものが正確に一致していることを確認しながら、このプロジェクトのデプロイメントを自動化する方法について学びたい場合プロダクションで機能するものについては、この記事に興味があると思います。

前提条件


この記事の読者は、コマンドライン領域とBashスクリプトの記述に関する基本的な知識を持っている必要があります。さらに、彼はTravis CIおよびDocker Hubアカウントを必要とします。

ゴール


この記事を無条件に「トレーニングマニュアル」と呼べるとは言わない。これはむしろ、私が学んだことについて話し、1つの自動化されたパスで実行される、自分に合ったコードをテストして本番環境にデプロイするプロセスを説明するドキュメントです。

これが私のワークフローの最終的な方法です。

リポジトリのブランチに送信されるコードを除いて、master次のアクションが実行されます。

  • Travis CIでプロジェクトをビルドします。
  • すべてのユニット、統合、エンドツーエンドのテストが実行されます。

master次の コードに当てはまる場合のみ：

  • 上記のすべてに加えて...
  • 現在のコード、設定、環境に基づいてDockerイメージをビルドします。

CI / CDとは何ですか？


略語CI / CDは、「継続的統合/継続的デプロイメント」-「継続的統合/継続的デプロイメント」の略です。

▍継続的な統合


継続的インテグレーションは、開発者がプロ​​ジェクトソースコードのメインリポジトリ（通常はブランチmaster）にコミットするプロセスです。同時に、自動テストを実施することにより、コードの品質が保証されます。

▍継続的な導入


継続的デプロイメントは、本番環境でのコードの頻繁な自動デプロイメントです。略語CI / CDの2番目の部分は、「継続的配信」（「継続的配信」）として開示されることがあります。これは一般に「継続的デプロイメント」と同じですが、「継続的デリバリー」はプロジェクトのデプロイメントプロセスを開始する前に変更を手動で確認する必要があることを意味します。

仕事の始まり


私がすべてをマスターしたアプリケーションはTakeNoteと呼ばれています。これは私が取り組んでいるWebプロジェクトで、メモを取るように設計されています。最初に、Netlifyが提供する標準のホスティングおよびデプロイメント機能を利用するために、JAMStackプロジェクトまたはサーバーなしのフロントエンドアプリケーションを作成しようとしました。アプリケーションの複雑さが増すにつれ、そのサーバー部分を作成する必要がありました。つまり、プロジェクトの自動統合と自動デプロイメントのために独自の戦略を立てる必要がありました。

私の場合、アプリケーションはNode.js環境で実行されるExpressサーバーであり、単一ページのReactアプリケーションを提供し、安全なサーバーAPIをサポートしています。このアーキテクチャは、このフルスタック認証ガイドにある戦略に従います。

私は自動化の専門家である友人に相談し、それを私が必要な方法で機能させるために何をする必要があるかを彼に尋ねました。この記事の「目標」セクションで概説されている、自動化されたワークフローがどのように見えるべきかについて彼は私に考えを与えました。このような目標を設定したという事実は、Dockerの使い方を理解する必要があることを意味しました。

Docker


Dockerは、コンテナー化テクノロジーのおかげで、Dockerプラットフォーム自体が異なる環境で動作している場合でも、アプリケーションを簡単に配布し、同じ環境にデプロイして起動できるツールです。まず、Dockerコマンドラインツール（CLI）を自由に使えるようにする必要がありました。Dockerのインストール手順は明確で理解しにくいものですが、インストールの最初のステップを実行するにはDockerデスクトップ（MacまたはWindows用）をダウンロードする必要があることがわかります。

Docker HubはgitリポジトリのGitHub、またはnpmレジストリとほぼ同じですJavaScriptパッケージの場合。これは、Dockerイメージのオンラインリポジトリです。Dockerデスクトップに接続します。

したがって、Dockerを使い始めるには、次の2つのことを行う必要があります。

  • Docker Desktopをインストールします。
  • Docker Hubにサインアップします。

その後、次のコマンドを実行してDockerのバージョンを確認することにより、Docker CLIが機能していることを確認できます。

docker -v

次に、尋ねられたらユーザー名とパスワードを入力して、Docker Hubにログインします。

docker login

Dockerを使用するには、イメージとコンテナの概念を理解する必要があります。

▍Dockerfile


ローカルで本番アプリケーションを実行する方法を知っています。既製のReactアプリケーションを構築するために設計されたWebpack構成があります。次に、ポートでNode.jsベースのサーバーを起動するコマンドがあります5000。次のようになります。

npm i         #  
npm run build #  React-
npm run start #  Node-

この資料のサンプルアプリケーションはありません。しかし、ここでは、実験のために、任意の単純なノードアプリケーションが適しています。

コンテナーを使用するには、Dockerに指示を与える必要があります。これはDockerfile、プロジェクトのルートディレクトリにあるというファイルを介して行われます。このファイルは、最初はかなりあいまいに見えます。

しかし、そこに含まれているのは、特別なコマンドを使用して、作業環境のセットアップに似たものについてのみ説明しています。これらのコマンドの一部を次に示します。

  • FROM-このコマンドはファイルを開始します。これは、コンテナが構築された基礎となる基本的なイメージを示します。
  • COPY-ローカルソースからコンテナにファイルをコピーします。
  • WORKDIR-次のコマンドの作業ディレクトリを設定します。
  • RUN-コマンドを実行します。
  • EXPOSE-ポート設定。
  • ENTRYPOINT-実行するコマンドを指定します。

Dockerfile 次のようになります。

#   
FROM node:12-alpine

#        app/
COPY . app/

#  app/    
WORKDIR app/

#   ( npm ci  npm i,     )
RUN npm ci --only-production

#   React-  
RUN npm run build

#   
EXPOSE 5000

#  Node-
ENTRYPOINT npm run start

選択したベースイメージによっては、追加の依存関係をインストールする必要がある場合があります。実際のところ、いくつかの基本的なイメージ（Node Alpine Linuxなど）は、可能な限りコンパクトになるように設計されています。その結果、彼らはあなたが期待しているいくつかのプログラムを持っていないかもしれません。

▍タグの割り当てと公開


作成されたイメージの1つを本番サーバーで使用するには、このイメージをDocker Hubからダウンロードできる必要があります。つまり、最初にDocker Hubでプロジェクトのリポジトリを作成する必要があります。その後、画像を修正できる場所を用意します。Docker Hubのユーザー名で名前が始まるように、イメージの名前を変更する必要があります。この後、リポジトリの名前になるはずです。名前の最後には任意のタグを付けることができます。以下は、このスキームを使用してイメージに名前を付ける例です。

これで、新しい名前でイメージを収集docker pushして、Docker Hubリポジトリに送信するコマンドを実行できます。

docker build -t <username>/<repository>:<tag> .
docker tag <username>/<repository>:<tag> <username>/<repository>:latest
docker push <username>/<repository>:<tag>

#     , , :
docker build -t user/app:v1.0.0 .
docker tag user/app:v1.0.0 user/app:latest
docker push user/app:v1.0.0

すべてがうまくいけば、イメージはDocker Hubで利用可能になり、サーバーに簡単にダウンロードしたり、他の開発者に転送したりできます。

トラビスci


Travis CIは、コードをテストおよびデプロイするためのツールです。 Travis CIのセットアップの複雑さについては説明しません。プロジェクトはそれぞれ固有であり、これはあまりメリットがありません。しかし、Travis CIを使用することに決めた場合に開始できる基本について説明します。選択するものは何でも-Travis CI、CircleCI、Jenkins、または他の何か、同様の構成方法がどこでも使用されます。

Travis CIでの作業を開始するには、プロジェクトのWebサイトにアクセスしますアカウントを作成します。次に、Travis CIをGitHubアカウントと統合します。システムのセットアップ中に、自動化するリポジトリを指定し、それにアクセスできるようにする必要があります。（私はGitHubを使用していますが、Travis CIはBitBucket、GitLab、および他の同様のサービスと統合できると確信しています）。

Travis CIが動作するたびに、対応するリポジトリブランチのデプロイメントを含む、構成ファイルで指定されたコマンドを実行するサーバーが起動されます。

▍導入


すべての自動テストが正常に完了したという前提に基づいて、オプションで、本番サーバーにコードをデプロイできます。これはブランチからのコードに対してのみ実行したいのでmaster、デプロイメント設定でシステムに適切な指示を与えます。後で検討するプロジェクトでコードを使用する前に、デプロイメントのために呼び出される実際のスクリプトが必要であることを警告します。

deploy:
  #  Docker-     Docker Hub
  provider: script
  script: bash deploy.sh
  on:
    branch: master

配備スクリプトは2つの問題を解決します。

  • CIツールを使用してイメージをビルドし、タグを付けてDocker Hubに送信します（この場合はTravis CIです）。
  • サーバーにイメージをロードし、古いコンテナーを停止して新しいコンテナーを開始します（この場合、サーバーはDigitalOceanプラットフォームで実行されます）。

最初に、イメージの組み立て、タグ付け、Docker Hubへの送信の自動プロセスを構成する必要があります。これは、画像に一意のタグを割り当ててログインを自動化するための戦略が必要であることを除いて、すでに手動で行ったものと非常に似ています。タグ付け戦略、ログイン、SSHキーのエンコード、SSH接続の確立など、展開スクリプトの詳細に問題がありました。しかし、幸いにも、私のボーイフレンドはbashやその他の多くのことを非常にうまく処理しています。彼は私がこのスクリプトを書くのを手伝ってくれました。

したがって、スクリプトの最初の部分は、イメージをDocker Hubに送信することです。これはとても簡単です。私が使用したタグ付けスキームには、gitハッシュとgitタグ（存在する場合）の組み合わせが含まれます。これにより、一意のタグの作成が可能になり、ベースとなるアセンブリの識別が簡単になります。DOCKER_USERNAMEそして、DOCKER_PASSWORDトラヴィスCIインターフェイスを使用して設定することができ、ユーザの環境変数です。 Travis CIは機密データを自動的に処理するため、悪意のある人物の手に渡ることはありません。

これがスクリプトの最初の部分ですdeploy.sh。

#!/bin/sh
set -e #     

IMAGE="<username>/<repository>"                             #  Docker
GIT_VERSION=$(git describe --always --abbrev --tags --long) # Git-  

#    
docker build -t ${IMAGE}:${GIT_VERSION} .
docker tag ${IMAGE}:${GIT_VERSION} ${IMAGE}:latest

#   Docker Hub   
echo "${DOCKER_PASSWORD}" | docker login -u "${DOCKER_USERNAME}" --password-stdin
docker push ${IMAGE}:${GIT_VERSION}

スクリプトの2番目の部分は、使用しているホストとそのホストへの接続がどのように構成されているかによって完全に異なります。私の場合、私はDigital Oceanを使用しているため、サーバーに接続するためにdoctlコマンドが使用されています。 Awsを使用する場合は、ユーティリティが使用さawsれます。

サーバーのセットアップは特に難しくありませんでした。そこで、ベース画像に基づいてドロップレットを設定しました。私が選択したシステムでは、Dockerを手動で1回インストールし、Dockerを手動で1回起動する必要があることに注意してください。私はUbuntu 18.04を使用してDockerをインストールしたので、Ubuntuを使用して同じことを行う場合は、この簡単なガイドに従うだけで済みます。

ここでは、サービスの特定のコマンドについては触れていません。この側面は、場合によって大きく異なる可能性があるためです。プロジェクトがデプロイされるサーバーにSSH経由で接続した後に実行されるアクションの一般的な計画のみを示します。

  • 現在実行中のコンテナーを見つけて停止する必要があります。
  • 次に、バックグラウンドで、新しいコンテナーを起動する必要があります。
  • ローカルサーバーのポートを値に設定する必要があります。80これによりexample.com、ポートを指定せずに、フォームのアドレスでサイトを入力でき、などのアドレスを使用できなくなりますexample.com:5000。
  • 最後に、古いコンテナとイメージをすべて削除する必要があります。

これはスクリプトの続きです。

#  ID  
CONTAINER_ID=$(docker ps | grep takenote | cut -d" " -f1)

#   ,  ,  
docker stop ${CONTAINER_ID}
docker run --restart unless-stopped -d -p 80:5000 ${IMAGE}:${GIT_VERSION}
docker system prune -a -f

TLS / SSLおよび負荷分散


上記で説明したことをすべて実行した後、私の前に発生した最後の問題は、サーバーにSSLがないことでした。 Node.jsサーバーを使用しているので、NginxとLet's Encryptのリバースプロキシを機能させるために、多くの工夫が必要です。

これらのすべてのSSL設定を手動で行うのは本当に嫌だったので、ロードバランサーを作成し、それに関する情報をDNSに記録しました。たとえば、DigitalOceanの場合、ロードバランサーで自己更新型の自己署名証明書を作成することは、シンプルで無料で高速な手順です。このアプローチには、追加の利点があります。つまり、必要に応じて、ロードバランサーの背後で動作している多くのサーバーでSSLを非常に簡単に構成できます。これにより、サーバー自体はSSLについてまったく「考えない」で、通常どおりポートを使用できます80。そのため、ロードバランサーでSSLを設定する方が、代替のSSL構成方法よりもはるかに簡単で便利です。

これで、サーバー上で、80ロードバランサーとの通信に使用されるポート22とSSH のポートを除く、着信接続を受け入れるすべてのポートを閉じることができます。その結果、これらの2つを除いて、任意のポートでサーバーに直接アクセスしようとすると失敗します。
