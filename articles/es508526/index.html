Cómo el desarrollador # С # aprendió mal de JavaScript





Recientemente me encontré con un artículo sobre Habré. Compara C # y JavaScript. En mi opinión, compararlos es como comparar la luna y el sol, que, según los clásicos, no están reñidos en el cielo. Este artículo me recordó a otra publicación . Se trata de escenarios de comportamiento inesperado y no obvio de JavaScript, y C # no se menciona de la palabra en absoluto, pero la curiosidad animada me animó a intentar repetir este comportamiento en otro idioma.



Atención



El material es de naturaleza de investigación y entretenimiento, y las técnicas descritas, como debe quedar claro en el título, no se recomiendan para su uso en el desarrollo de productos.



Los ejemplos están escritos en .NET Core 3.1.



¿Cuál es el resultado final?



En JavaScript, es posible usar el operador para obtener el resto de la división en números de coma flotante. Funciona así: 



3.14 % 5 // 3.14
13.14 % 5 // 3.1400000000000006


Además, los números son del tipo Número y se almacenan en 64 bits de acuerdo con el estándar IEEE 754. En .NET, este tipo tiene un sistema hermano gemelo. Doble o simplemente doble. Si un literal numérico contiene un punto flotante, el valor predeterminado es doble. La intención se puede expresar explícitamente agregando el sufijo do D al número. También está disponible la capacidad de dividir el doble con el resto. Así que inténtalo. Y ... ¡Bingo! 



Console.WriteLine(3.14d % 5); // 3,14
Console.WriteLine(13.14d % 5); // 3,1400000000000006


Digresión lírica
f F, float (System.Single), 32 IEEE 754. , . 



Console.WriteLine(3.14f % 5); // 3,14
Console.WriteLine(13.14f % 5); // 3,1400003


.NET decimal (System.Decimal), . 



Console.WriteLine(3.14m % 5); // 3,14
Console.WriteLine(13.14m % 5); // 3,14

¡Ven, inmundo, porque no hay verdad en ti!



En el siguiente ejemplo, la condición es verdadera. 



const x = { i: 1, toString: function() { return this.i++; } };
if (x == 1 && x == 2 && x == 3)
    document.write("This will be printed!");


Esto se logra debido al hecho de que al verificar la igualdad, los operandos se convierten al mismo tipo. En este caso, se llama a un método toString () anulado en x, que, además de devolver algo, cambia el estado del objeto. Nos damos cuenta de que las funciones puras son maravillosas, pero por interés estamos tratando de implementar el concepto a partir del ejemplo. 



Trickster x = new Trickster();
if (x == 1 && x == 2 && x == 3)
    Console.WriteLine("This will be printed!");


.NET le permite anular métodos y conversiones, cuando sea posible, si se define la conversión correspondiente. 



class Trickster
{
    private int value = 1;

    public override string ToString() =>
        value++.ToString();

    public static implicit operator int(Trickster trickster) =>
        int.Parse(trickster.ToString());
}


Hurra, funciona. Pero confunde que Trickster parece adaptarse a int. Reemplazar 



public static implicit operator int(Trickster trickster) =>
    int.Parse(trickster.ToString());


sobre el 



public static implicit operator double(Trickster trickster) =>
    double.Parse(trickster.ToString());


Funciona de todos modos. Ahora Trickster e int se lanzan al doble.



En realidad, la redefinición de ToString () no es esencial para el logro del resultado final y está hecha para la máxima imitación de un ejemplo en JavaScript. Es suficiente definir un elenco para duplicar. Por supuesto, sin olvidar la importancia del efecto secundario. 



class Trickster
{
    private int value = 1;

    public static implicit operator double(Trickster trickster) =>
        trickster.value++;
}


Y si recuerda la sobrecarga de operadores, puede implementar un objeto que sea igual y diferente a cualquier cosa 



Trickster x = new Trickster();
if (x == 1 && x == 2 && x == 3 && x == new[] { 1, 2, 3 } &&
    x != 1 && x != 2 && x != 3 && x != new[] { 1, 2, 3 })
    Console.WriteLine("This will be printed!");


de una manera tan simple

{

Agarre elegante



Este código JavaScript sale 3 veces. 



for (i = 0; i <= 2; ++i)
  setTimeout(() => console.log(i), 0);


Como no hay una función setTimeout lista para usar en C #, implementamos el análogo nosotros mismos y obtenemos exactamente el mismo resultado. 



void SetTimeout(Action action, int delay) =>
    Task.Run(async () =>
    {
        await Task.Delay(delay);
        action();
    });

for (int i = 0; i <= 2; ++i)
    SetTimeout(() => Console.WriteLine(i), 0);

//        
Console.ReadKey();

Nadie prometió conmutatividad



Los operadores en JavaScript a veces no son conmutativos. 



Date() && {property: 1}; // {property: 1}
{property: 1} && Date(); // Uncaught SyntaxError: Unexpected token '&&'


En C #, los operadores generalmente no son aplicables a los tipos definidos por el usuario de forma predeterminada (excepción == y! = Para tipos de referencia). Pero algunos de ellos se pueden sobrecargar explícitamente en tipo. Entonces la preocupación por la conmutatividad recae en el desarrollador. 



class Trickster
{
    //  C#  && 
    public static object operator +(Trickster trickster, object o) =>
        null;
}


Y no necesariamente se implementará. 



var a = new Trickster() + new object(); // OK

// Compilation Error:
// Operator '+' cannot be applied to operands of type 'object' and 'Trickster'
var b = new object() + new Trickster();


Cruzando un erizo con un erizo



En JavaScript, puede realizar operaciones matemáticas juntas en cadenas y números. 



var a = "41";
a += 1; // "411"
var b = "41";
b -=- 1; // 42


En C #, esto solo es posible con la suma. 



var a = "41" + 1; // 411

// Compilation Error:
// Operator '-' cannot be applied to operands of type 'string' and 'int'
var b = "41" - (-1);


Pero no hay obstáculos para los patriotas de la lengua. Al agregar un poco de brujería, puede hacer que incluso este código funcione "correctamente": 



Trickster a = "41";
Console.WriteLine(a += 1); // 411
Trickster b = "41";
Console.WriteLine(b -=- 1); // 42


Solo necesita implementar un par de conversiones de tipo implícito y sobrecargar el par de declaraciones. 



class Trickster
{
    private string value;

    public static implicit operator Trickster(string s) =>
        new Trickster { value = s };

    public static implicit operator Trickster(int i) =>
        new Trickster { value = i.ToString() };

    public static string operator +(Trickster trickster, int i) =>
        trickster.value + i;

    public static int operator -(Trickster trickster, int i) =>
        int.Parse(trickster.value) - i;

    public override string ToString() =>
        value;
}


Puede reemplazar la sobrecarga de adición 



public static string operator +(Trickster trickster, int i) =>
    trickster.value + i;


para lanzar implícitamente a una cadena 



public static implicit operator string(Trickster trickster) =>	
    trickster.value;


y obtener el mismo resultado.



Si no se molesta en devolver el número de la operación de resta y colocar este número en Trickster, puede reemplazar 



public static int operator -(Trickster trickster, int i) =>	
    int.Parse(trickster.value) - i;


sobre el 



public static string operator -(Trickster trickster, int i) =>
    (int.Parse(trickster.value) - i).ToString();


y el int intcast a Trickster se elimina.



Malabares, plátanos



En JavaScript, la cadena "banana" se puede obtener de la siguiente manera: 



('b' + 'a' + + 'a' + 'a').toLowerCase(); // "banana"
('b'+'a'++'a'+'a').toLowerCase(); // Uncaught SyntaxError: Invalid left-hand side expression in postfix operation


Aquí, la aplicación de unary + a la segunda 'a' devuelve NaN, que se convierte en la cadena 'NaN' para agregar al resto de las cadenas, y el resultado es 'baNaNa', y por el bien de la belleza, todo se reduce a minúsculas.



En C # creamos una clase 



class Trickster
{
    private string value;

    public static implicit operator Trickster(string s) =>
        new Trickster { value = s };

    public static double operator +(Trickster trickster) =>
        double.TryParse(trickster.value, out double result) ? result : double.NaN;
}


e intenta 



//  double.NaN.ToString()  "NaN",   " "
Thread.CurrentThread.CurrentCulture = new CultureInfo("en-us");
Console.WriteLine(("b" + "a" + + (Trickster)"a" + "a").ToLower());


Desafortunadamente, a pesar de la conversión implícita, el compilador no puede construir la cadena de cadena -> Trickster -> double -> string, y debe solicitarla explícitamente. (Si lo piensas bien, un elenco así se vería más que extraño).



Trickster se puede implementar de manera diferente: 



class Trickster
{
    private double value;

    public static implicit operator Trickster(string s) =>
        new Trickster { value = double.TryParse(s, out var d) ? d : double.NaN };

    public static double operator +(Trickster trickster) =>
        +trickster.value;
}


En este caso, incluso puede reemplazar la sobrecarga unaria + con una conversión implícita para duplicar: 



public static implicit operator double(Trickster trickster) =>
    trickster.value;


Como beneficio adicional, si no coloca un espacio entre las dos ventajas, entonces, como en el ejemplo de JavaScript, obtenemos un error de compilación, no solo uno, sino un montón.
