レオナルド番号


両方の問題を解決するために、ダイクストラはレオナルド数に基づく特別なバイナリヒープの使用を提案しました。

レオナルド数はフィボナッチ数とほとんど同じですが、それよりも優れています。
一連のレオナルド番号が再帰的に与えられます：

L 0 = 1
L 1 = 1
L n = L n-1 + L n-2 + 1

最初の20のレオナルド番号：
1、1、3、5、9、15、25、41、67 、109、177、287、465、753、1219、1973、3193、5167、8361、13529

絶対に任意の整数は、異なるシリアル番号を持つレオナルド番号の合計として表すことができます。

これは、私たちの場合に非常に役立ちます。nの配列要素は常にLeonardoの単一のヒープとして表すことができるとは限りません（nが Leonardo番号でない場合）。ただし、どの配列も常に、Leonardoの異なる数に対応するいくつかのサブ配列に分割できます。異なる順序のヒープになる。

これは、3つのLeonardヒープで構成される21番目の要素の配列の例です。各ヒープでは、ノードの数はLeonardoの任意の数に対応します。


知っておくべき重要なポイント：

  1. 各Leonardovパイルは、不均衡なバイナリツリーです。
  2. 各ヒープのルートは、対応するサブ配列の最後の要素です（通常のバイナリヒープのように、最初ではありません）。
  3. すべての子孫を持つノードは、小さいオーダーのレナードヒープでもあります。

ヒープの構築と解体


レオナルド数の漸化式では、

L n = L n-1 + L n-2 + 1は

、最後の単位に非常に満足しています。

そしてそれが理由です。隣接する2つのLeonardo番号に基づいて構築されたヒープに対応する2つの隣接するサブ配列があると仮定します。これらのサブ配列の直後の要素を使用して、これらのサブ配列を結合して、次のレナード数に対応する共通のヒープにすることができます。



配列の要素を調べて、Leonardヒープの束を作成します。要素を使用する場合は、前の2つのヒープを組み合わせることができます（これは、前の2つのヒープが2つの連続するLeonardo番号に対応する場合にのみ可能です）。その後、組み合わせます。結合できない場合（前の2つのヒープが2つの連続するLeonardo番号に対応していない場合）、現在の要素は、最初の（または最初の要素がその前に使用されている場合は2番目の）Leonardo番号に対応する1つの要素の新しいヒープを形成します。

アルゴリズムの第2段階では、逆のプロセスが発生します-ヒープを解析します。ヒープ内のルートを削除すると、以前の2つのLeonardo番号に対応する2つの小さいヒープが取得されます。

L n -1 = L n-1のため、これを行うことができます。+ L n-2

フィボナッチ数ではそのような有用な単位はないため、フィボナッチヒープは使用しません。

スムーズソート::スムーズソート


最終的なアルゴリズム：

  • I.配列からレナードヒープの束を作成します。各ヒープはソーティングツリーです。
    • I.1。配列の要素を左から右に繰り返します。
    • II.1。現在の要素を使用して、左の2つのヒープを既存のLeonardヒープのヒープに結合できるかどうかを確認します。
      • II.1.a. はいの場合、左端の2つのヒープを1つに結合します。現在の要素がこのヒープのルートになり、結合されたヒープをシフトします。
      • II.1.b. そうでない場合は、現在の要素を新しいヒープ（これまでのところ1つのノードで構成されている）として既存のLeonardヒープのヒープに追加します。
  • II. , :
    • II.1. . , .
    • II.2. ( ) ( ).

def leonardo_numbers(hi):

    a, b = 1, 1
    numbers = []
    while a <= hi:
        numbers.append(a)
        a, b = b, a + b + 1
    return numbers

def restore_heap(lst, i, heap, leo_nums):
    
    #      
    
    current = len(heap) - 1
    k = heap[current]

    while current > 0:
        j = i - leo_nums[k]
        if (lst[j] > lst[i] and
            (k < 2 or lst[j] > lst[i-1] and lst[j] > lst[i-2])):
            lst[i], lst[j] = lst[j], lst[i]
            i = j
            current -= 1
            k = heap[current]
        else:
            break

    # 
    
    while k >= 2:
        t_r, k_r, t_l, k_l = get_child_trees(i, k, leo_nums)
        if lst[i] < lst[t_r] or lst[i] < lst[t_l]:
            if lst[t_r] > lst[t_l]:
                lst[i], lst[t_r] = lst[t_r], lst[i]

            else:
                lst[i], lst[t_l] = lst[t_l], lst[i]

def get_child_trees(i, k, leo_nums):

    t_r, k_r = i - 1, k - 2
    t_l, k_l = t_r - leo_nums[k_r], k - 1
    return t_r, k_r, t_l, k_l

時間の複雑さ


ほとんど順序付けられた配列を入力として受け取る場合、視覚化により、そのような配列がはるかに速く処理される理由がわかります。



貯蓄はふるい分けのためにのみ発生します。ほとんど順序付けられたデータでは、ヒープが第2ステージで徐々に解散された後も含めて、ふるい分けシンクがツリー内で浅くなります。最初のランダムなデータでは、ヒープが最後のレベルまで低下することが多いため、シフトはより高価です。

合計時間の複雑さを推定してみましょう。

最初の段階では、n個の要素を反復処理し、左側に既に存在するヒープに追加します。ヒープ自体に追加するとO（1）程度かかりますが、ヒープの場合はふるいにかける必要があります。順序付けされたデータでは、浅いふるい分けは、ヒープに追加された1つの要素に対してO（1）のコストがかかることがよくあります。順序付けされていないデータでは、各追加のシフトはO（log n）で計算されます、ランダム性の結果として、ふるいにかけることは、しばしばツリーのレベルを一番下まで通過する必要があるためです。

したがって、最初の段階で最も時間的に複雑になるのは
、ほぼ順序付けられたデータの場合-O（n）、
ランダムデータの場合-O（n log n）です。

第2ステージの場合も同様です。次の最大値を交換するときは、それがあったルートでヒープをふるいにかける必要があります。そして、順序付けられたデータと無秩序なデータのふるい分けメトリックは異なります。

2番目の段階では、最適な時間の複雑さは最初の段階と同じ
です。ほぼ順序付けられたデータの場合-O（n）、
ランダムデータの場合-O（n log n）。

第1ステージと第2ステージの時間の複雑さを追加：
ほとんど順序付けされたデータの場合-O（2 n）= O（n）、
ランダムデータの場合-O（2 n log n）= O（n log n）。

一般に、スムーズなソートの最悪および平均時間の複雑さはO（n log n）です。
ダイクストラの彼の計算（私はあなたを退屈させません）は、最良の複雑さは、より順序付けられた入力データよりもスムーズにO（n）になる傾向があることを証明しました。したがって、名前-スムーズなソート。

余分なメモリの複雑さ


データを一連のLeonardヒープに分解するには、各手順でどのLeonardo番号が関係しているかを正確に覚えておく必要があります。これらの数を知ることで、ヒープ自体がアルゴリズム的に調整されます。この数列は非常に急速に増加するため、大規模な配列であっても、非常に小さなレナード数のセットが必要になります。

二項ヒープソート::二項ヒープソート


私たちがソートしたものと非常によく似たツリー構造があります - 二項ヒープです。これは、サイズの異なるヒープの集まりでもあり、それぞれのノードの数は2の累乗です。自然数は次数の異なる2つの和に分解されるため、任意の数の要素の配列をこのヒープに展開できます。

原則として、二項式に基づいてスムーズにソートできます。



より速く動作しますか？ほとんどありません。二項ヒープはバイナリではなく、前回の記事で、子孫の数を増やしても加速はしないが、画面が遅くなることがわかりました。さらに、2項式ヒープの方が長いブランチがあることに注意してください。これは、配列の隣接する順序付けられた領域が互いにゆっくりと接続する理由です。

ダイクストラの2項式ヒープが、そのアルゴリズムの可能な基礎として一般的に考えられていたかどうかは不明です。とにかく、Leonardovヒープの方がおそらく最適です。
