.NETがどこでも機能する場合、Windows 3.11およびDOSでも

.NET Coreはオープンソースであり、どこでも機能することをよく繰り返します。MonoGame、Unity、Apple Watch、Raspberry Piとマイクロコントローラー、12のLinux、Windowsなど。もうたくさん。

しかし、誰かはまだ十分ではありません。Michal Strehowskiは、本当にどこでも C＃を起動したいと考えています。


C＃on Windows 3.11

2つの「不可能」なシステムでC＃コードを起動し、「どこでも機能する」という私たちの定義を補完します。これらは楽しい実験です（本番環境では繰り返さないでください）が、Michalの技術的能力とベースプラットフォームの柔軟性の両方を強調しています。

Windows 3.11でC＃を実行する


で7人のつぶやき、ミハルは彼がWindowsの3.11の下でC＃のコードを実行するために管理方法について語っています。アプリケーションは単純です。ここでは、対応するダイアログボックスが表示されたMessageBoxA関数を呼び出すだけです。これは、Windowsの最初の日からです。関数を呼び出して結果を取得するには、DllImport / PInvokeを使用します。

かっこいいので、私は最初にこのアプリケーションをWindows 3.11に示しました。しかし、実際には、作者はDOSの実験が終わったところから始まりました。ネイティブCコードをコンパイルし、その後、ルールは存在しなくなります。

この例では、Win32ではなく、Win16プラットフォームで実行されます。しかし、1992年に（はい、私はその時に住んでプログラムし、プロジェクトで使用しました！）という特定の技術的橋がありました。Win32s：Windows 3.11に移植されたWindows NTのAPIのサブセット。したがって、いくつかの制限を考慮して、32ビットコードを記述し、Win16からWin32にアクセスできます。

Michalは、2020年にCoreRT AOTコンパイラーによって作成されたオブジェクトファイルは、1994年のVisual C ++ 2.0サンプルのリンカーによってアセンブルできることに気付きました。結果は、16ビットWindows 3.11で実行されているWin32sインターフェイスにリンクされたマシンコードです。マジック。ミカルを尊重してください。

DOSでC＃を8Kで実行する


以前にスタンドアロンの.NET Core 3.x実行可能ファイルについて書いたことがありますが、私はこれの大ファンです。私のアプリケーションはすでに28メガバイトに減少しています。 .NETランタイムと他の多くのリソースが含まれていることを考えると、これはかなりの量です。もちろん、可能な限り最小のプログラムのサイズでVM /ランタイムを判断するべきではありませんが、Michal はどのような制限に到達できるかを確認し、8000バイトのターゲットを設定したいと考えていました。

プログラムはテキストモードで動作します。これは私の意見では素晴らしいです。また、リソースの割り当てがないため、ガベージコレクターが不要になります。これは、どこでもnewを使用できないことを意味します。参照タイプはありません。

フィールドを使用して静的配列を宣言します。fixed char []：彼らはスタックに住んでいる必要があり、私たちのスタックは小さいです。

もちろん、ある種のスタンドアロン.NET実行可能ファイルを作成しようとすると、最初に65メガバイトのファイルが得られます。このファイルには、アプリケーション、ランタイム、標準ライブラリが含まれています。ILLinkerとPublishedTrimmedを

dotnet publish -r win-x64 -c Release

使用して.NET Core 3.xからのツリートリミングを最適化できますが、この方法ではファイルを25メガバイトにしか削減できません。 彼はMonoとmkbundleを使用してサイズを18.2メガバイトにしようとしましたが、エラーが発生しました。そして、ランタイムはまだそこにあります。 したがって、適切なランタイムはCoreRTだけで、仮想マシンは含まれていませんが、補助機能のみが含まれています。





dotnet publish -r win-x64 -c Release /p:Mode=CoreRT

そのため、彼は4.7メガバイトを獲得しましたが、それでもまだ多すぎます。一部の設定では、3メガバイトまで歩くことができます。リフレクションを完全に拡張して、1.2メガバイトに達することができます。これでフロッピーディスクに収まります。

dotnet publish -r win-x64 -c Release /p:Mode=CoreRT-ReflectionFree

この1メガバイトのサイズは、.NET SDKのみのハード制限のようです。

これは、Michal が標準ツールから離れるところです。これは、システムの基本的なタイプの再実装スタブを実行します。次に、いくつかの魔法のスイッチで再コンパイルして、実行可能ファイルのILバージョンのみがリリースされるようにします。

csc.exe /debug /O /noconfig /nostdlib /runtimemetadataversion:v4.0.30319 MiniBCL.cs Game\FrameBuffer.cs Game\Random.cs Game\Game.cs Game\Snake.cs Pal\Thread.Windows.cs Pal\Environment.Windows.cs Pal\Console.Windows.cs /out:zerosnake.ilexe /langversion:latest /unsafe

次に、これをCoreRTに渡してネイティブコードを取得します。

ilc.exe zerosnake.ilexe -o zerosnake.obj --systemmodule zerosnake --Os -g

そして、ここにいます。

「zerosnake.objが標準オブジェクトファイルになりました。CやC ++などの他のネイティブコンパイラによって作成されたオブジェクトファイルと同じです。最後のステップは、それを組み立てることです。」

さらにいくつかのトリック-出力は27 KBです！次に、リンカーからいくつかのスイッチを削除して、アセンブラー開発者が使用するのと同じメソッドを使用してさまざまなものを無効にして削除し、8176バイトを残しました。エピックスリラー。TwitterでMichal

link.exe /debug:full /subsystem:console zerosnake.obj /entry:__managed__Main kernel32.lib ucrt.lib /merge:.modules=.rdata /merge:.pdata=.rdata /incremental:no /DYNAMICBASE:NO /filealign:16 /align:16

をフォローし、彼を称賛します。
