STM32パート1：基本事項

それでは、始めましょう。前回の記事では、最初のポイントをすばやく確認しました。これは、2つのベクトル（スタック、Reset_Handler）を担当し、リンカースクリプトについて少し担当したstartup.cファイルです。今日は、初期化コードを補足し、スペアパーツのリンカーを分析して、すべてがどのように機能するかを調べます。


extern void *_estack;

void Reset_Handler();

void *vectors[] __attribute__((section(".isr_vector"), used)) = {
    &_estack,
    &Reset_Handler
};

void __attribute__((naked, noreturn)) Reset_Handler()
{
    while(1);
}


このコードがわかりにくい場合は、ここで読むことができます（この記事は泉ではありませんが、説明しようとしました）。

ここで何が欠けているのか、それを追加する方法を理解しましょう。明らかに、ミクロンには他のベクトルがあります。たとえば、Hard_Faultベクトルは、μによる不適切なアクションで呼び出されるベクトルです。たとえば、プロセッサが理解できない命令を実行しようとすると、Hard_Faultベクトルのアドレスにジャンプします。そのようなベクトルはたくさんあり、私たちのプログラムではすべてを使用することは事実ではありません。また、割り込みベクタも同じ表にあります。

（それでもベクトルがわからない場合は、これはアドレスへのポインタ、別名「ポインタ」です）。

コードを補足してから、私が行ったことを説明します。

extern void *_estack;

void Reset_Handler();
void Default_Handler();

void NMI_Handler()   __attribute__((weak, alias("Default_Handler")));
void HardFault_Handler()   __attribute__((weak, alias("Default_Handler")));
//     

void *vectors[] __attribute__((section(".isr_vector"), used)) = {
    &_estack,
    &Reset_Handler
    &NMI_Handler,
    &HardFault_Handler
};

void __attribute__((naked, noreturn)) Reset_Handler()
{
    while(1);
}

void __attribute__((naked, noreturn)) Default_Handler(){
    while(1);
}


ここで、新しい関数Default_Handler（）;を追加しました。すべての割り込みを絶対に処理します。しかし、助けを借り__attribute__((weak, alias("Default_Handler")));て、同じ名前の関数が宣言されている場合、それがこの割り込みのハンドラーになることに注意しました。つまり、現在のところ、これはDefault_Handlerの単なるニックネームです。そのため、リファレンスマニュアルのリストにあるすべてのベクターを追加できます。これは、割り込みを使用することにしたが、関数ハンドラーを作成するのを忘れた場合に、通常のDefault_Handlerが呼び出されるようにするためです。

私はこの時点で、ベクトルを使用して、セクターの初期化とリンカに進むことができると思います。まず、データの初期化とbssセクターをReset_Handler本体といくつかの外部変数に追加して、startup.cを再度更新します。

extern void *_sidata, *_sdata, *_edata, *_sbss, *_ebss;

void __attribute__((naked, noreturn)) Reset_Handler()
{


	void **pSource, **pDest;
	for (pSource = &_sidata, pDest = &_sdata; pDest != &_edata; pSource++, pDest++)
		*pDest = *pSource;

	for (pDest = &_sbss; pDest != &_ebss; pDest++)
		*pDest = 0;

	while(1);
}


つまり、新しい変数、そして再びそれらはリンカースクリプトで宣言されました。前のスクリプトを思い出してください。また、新しいものと比較してください。

MEMORY{
	ROM(rx) : ORIGIN = 0x08000000, LENGTH = 1M
	SRAM (rwx):     ORIGIN = 0x20010000, LENGTH = 240K
}

_estack = LENGTH(SRAM) + ORIGIN(SRAM);

SECTIONS{
	.isr_vector : {
	KEEP(*(.isr_vector))
	} >ROM
	
	.text : {
        . = ALIGN(4);
        *(.text)
    } >ROM
	
	_sidata = LOADADDR(.data);
        .data : {
		. = ALIGN(4);
		_sdata = .;
		*(.data)
		. = ALIGN(4);
		_edata = .;
	} >SRAM AT>ROM
	
	.bss : {
		. = ALIGN(4);
		_sbss = .;
		*(.bss)
		. = ALIGN(4);
		_ebss = .;
	} >SRAM
}


2つの新しいセクションが追加されました。これらは.dataと.bssです。なぜ必要なのですか？.dataはグローバル変数を残し、staticはbssに移動します。これらのセクションは両方ともRAMにあり、bssセクションですべてが単純な場合（これらは単なるゼロです）、データセクションでいくつかの問題があります。まず、データはコンパイル時に既知の初期化済みデータです。したがって、最初はデータセクションはフラッシュメモリのどこかにあります。次に、スクリプトをフラッシュメモリにあることをリンカに何らかの方法で示す必要がありますが、プログラムの実行中にRAMに転送されます。データセクションを記述するスクリプトの一部を見てみましょう。


        _sidata = LOADADDR(.data);
        .data : {
		. = ALIGN(4);
		_sdata = .;
		*(.data)
		. = ALIGN(4);
		_edata = .;
	} >SRAM AT>ROM


それは、_sidataの宣言と、現時点では明確ではない意味の割り当てから始まります。次に、セクション自体の説明が始まります。ここでALIGN関数についてお読みになることをお勧めしますが、簡単に言えば、プロセッサがデータまたは命令をそこからロードしようとするときに、より簡単に受け入れられる値をポイント（現在のアドレス）に割り当てます。

_sdata、_edata、およびセクションの最後も含まれています。しかし、このセクションの最後には、必要なものが揃っています。SRAM AT>ROM。この行は、セクションがSRAMにあることをリンカに伝えますが、このセクションはROMまたはフラッシュメモリにロードされます。 _sidata変数とLOADADDR関数に戻ります。この関数は、読み込みセクションのアドレスに等しい値を返します。つまり、セクションがフラッシュメモリに読み込まれていることを示しましたが、それをRAMで使用し、このセクションを転送するために、LOADADDR関数が返す、フラッシュメモリのアドレスを知る必要があります。一方、変数_sdataおよび_edataは、RAM内のセクションの場所を示すため、データセクションを適切なメモリにロードする機会が与えられます。これは、上記で補足した初期化コードが扱うものであることを思い出させてください。

なぜこのリンカが必要なのか、アプリケーションのセクションの配布にこのような困難があるのでしょうか。データセクションは、セクションをメモリに配置してシフトできるようにする必要がある理由のほんの一例です。以下は、私が使用するテキストセクションの例です。

.text (ORIGIN(ROM_ITCM) + SIZEOF(.isr_vector)): {
        . = ALIGN(4);
        *(.text)
    } AT>ROM_AXIM


このブロックの本質は、テキストセクションもフラッシュメモリにロードするが、異なるバスを使用して命令とARTアクセラレータ（通常のキャッシュのようなもの）にアクセスすることです。そして、すべてが機能するように、コード実行中にロードアドレスとアドレスを指定しましたが、セクションをオフセットする必要があることを指定しなかった場合は機能しません。このような発明は他にもたくさんありますが、すべてを示すことはしません。
