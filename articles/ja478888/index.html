モジュラーCSSとコードサポートの問題に関するいくつかの考え

質問は何ですか？


みなさん、こんにちは。今日、私は書体での私の個人的な経験をあなたと共有したいと思います。もっと簡単に言えば、サイトのcssを書く問題に私のビジョンを置きます。

技術開発の時代には、毎日新しいフレームワーク、プリプロセッサ、テンプレートエンジンなどに直面しても、問題はまったく発生しないはずです。賢い開発者は、強力で拡張可能なサイトを作成するための便利なクールな「機能」をたくさん考え出しました。しかし、CSSでこれらのツールを正しく使用する方法は？そして、困難は何ですか？

私は、いくつかの問題を解決するための私の方法とアプローチを提案することにより、これらの質問やその他の質問に答えるように努めます。

厳しい現実


初心者および経験豊富なフロントエンド開発者が、編集やデザインの再作成などを必要とする既成のプロジェクトをしばしば与えられることは誰にとっても秘密ではありません。私の実践では、他の誰かのコードを操作するとき、そのようなタスクがたくさんありましたが、それを理解し、理解する必要があります。また、既製のサイトが手に入る場合は、スタイルシートを開いて、数千行のコードが表示されるように準備してください。

もちろん、経験豊富な開発者は、Visual Studio Codeなどの何らかのIDEでそれを開き、検索と置換機能を少し手で振れば、任意のクラスを簡単に見つけて書き換えることができます。

しかし、新しいデザインの一部をいくつかのセクションに統合して、適切な量の編集を行う必要がある場合、これにどれだけの時間を費やしますか？

そうだと思いますたくさん。そして、あなたがするすべては穏やかに「役に立たない」でしょう。やり直したサイトが別の開発者に転送されたとき、またはしばらくしてそこに新しい変更を加えるように指示された状況を想像してみてください。

SASSが問題を解決します


このすべての情報を要約した後、典型的なWeb開発者は、CSSプリプロセッサーを使用して、状況がなくなったと言います。そこには、モジュール性と快適さなどがあります。

もちろん！しかし、経験の浅いプログラマーが手にするツールはすべて意味を失います。時々、サイトのスタイルを開いて、「sass」という名前の秘蔵フォルダを見ます。たいていの場合、奇跡を期待すると失望するだけです。

スタイルを個別のファイルに分解：header.sass、main.sass、animation.sassなど どれほど「素晴らしい」か。そして効果的ではありません。そのような次のファイルを開くと、同じスタイルの行の束に出くわします。

代替案は何ですか？


開発者がクラスを呼び出す方法や、スタイルのネストのチェーンについては、黙っておくのが良いでしょう。この分野で好転したすべての人は、BEMについて何も聞いたことがなく、クラスに正しく名前を付けることの重要性を理解していると思います。これにより、すべての問題を解決する価値があります。

私は特別なアプローチ、スタイルを開発するための特別な方法論を提供しますが、プリプロセッサーの助けが必要です。この方法を実装するには、いくつかの段階があります。

  • 最初のは、すでに理解したように、クラスの適切な名前です。
  • 2つ目は、ネストスタイルを取り除くことです。ここでBEM方法論が役立ちます。この素晴らしいドキュメントの基本を理解したら、次のステップに進むことができます。
  • SASS-このツールなしで。プロジェクトのモジュール性とパワーを保証するのは彼です。
  • プロジェクトの組み立て。最近のアセンブラについてはよく聞いたことがあると思います（インターネット上でプロジェクトを構築するための情報はたくさんあります）。ここでGulpが役立ちます。

これで準備が整いました。

今本質的に


フロントエンドの現代的な解釈では、かなりの数のJavaScriptフレームワークを見つけることができます。それらのほとんどは、サイトをより小さなパーツ、コンポーネントに分割しています。スタイルの書き方を考えるときも同じようにすることをお勧めします。各ボタン、メニュー項目、美しいリンク、画像、プレーンテキストでさえ、コンポーネントにすることができます。



たとえば、このサイトへの入力形式では、いくつかのコンポーネントを区別できます。

  1. ログインフォーム自体
  2. 入力フィールド
  3. 送信ボタン
  4. チェックボックス

そして、実践が示すように、サイトの送信ボタンに加えて、これらのボタンがさらに多くなり、それらのすべてのスタイルを1か所に保持するのが理にかなっています。入力フィールドについても同じことが言えます。それらは、サイトの別の場所にある場合があり、その外観を繰り返します。

これに基づいて、コンポーネントごとに個別のフォルダーを作成し、そこにのみスタイルを保持することが論理的であり、これにより、モジュール式のアプローチを実装できます。しかし、それだけではありません。

コンポーネント内部


先ほど述べたように、サイトにはさまざまな色、インデントのボタンが多数存在する可能性があります。これらはすべてデータであり、このデータをスタイルから分離することは非常に賢明です。私はそうすることを提案します。

各コンポーネントを3つのエンティティ（vars、maker、およびcreater）の形で提示することを提案します。簡単に言えば、データ、アクション作成者（このデータを管理する方法）、およびスタイル作成者です。さて、まず最初に。

  • データまたは変数。これらには、ボタンのテキストの色、ボタンの背景色、パディング、境界線の半径、フォントサイズなどが含まれます。このデータはすべて、変数または連想配列の形式で別のファイルに書き込まれます。
  • アクションの作成者または作成者。引数としてデータを取り、それらからスタイルを生成するさまざまな関数またはミックスインが含まれています。
  • スタイルメーカーまたはクリエーター。実際には、ここでデータとメソッドを自分用にインポートし、必要に応じてそれらを相互にバインドします。

次の図は、3つのエンティティすべての相互作用を示しています。



これらすべてを頭に入れるのは難しいですか？ボタンの基本コンポーネント「ボタン」を作成する例を考えてみましょう。SCSSをプリプロセッサーとして使用します。

すでにご存じのとおり、新しいボタンフォルダーには、_vars.scss、_creater.scss、_maker.scssの3つのファイルを作成します。

_vars.scss

$button_size:(
    sm:5px 20px,
    md:15px 20px
);
$button_color:(
   white:#ffffff
);
$button_back:(
    blue: #00e5e5
);
/* This is component config, it must be below other variables */
$button_config:(
    padding:$button_size,
    color:$button_color,
    background:$button_back
);

ご覧のように、データは連想配列の形式で表示されます。個々のスタイルはすべて、メインのbutton_configセットに含まれています。新しいスタイルを作成するには、開発者は新しい配列を作成し、それに必要な値を入力して、それをbutton_configに接続するだけです。

_creater.scss：

@import "vars";
@import "maker";

.button{
   @include InitialComponent($button_config);
}

ここでは、データとアクション作成者をインポートします。次に、makerに登録されているInitialComponent mixinを使用して、button_config変数を渡し、すべてのスタイルを初期化します。そして、これが魔法です。必要なクラスはすべて、BMの命名に従って作成されます。

実行の結果はスタイルになります：

.button_padding_sm {
  padding: 5px 20px; 
}

.button_padding_md {
  padding: 15px 20px;
 }

.button_color_white {
  color: #ffffff;
 }

.button_background_blue {
  background: #00e5e5;
 }

そして、ここでの成功とは何ですか？


このアプローチの利点はすぐには感じられません。しかし、プロジェクトが成長するにつれて、あなたはそのような方法論の便利さを理解します。異なるページのいくつかのボタンの色を変更する必要がある場合は、一連のスタイルでクラスを探す必要はありません。必要なのは、ボタンフォルダーを開き、それに応じてデータファイルを開き、そこで目的のスタイルを置き換えることだけです。

そして、もしあなたが新しいボタンを作る必要があるなら？

次に、さらに簡単です。必要なスタイルを既に準備されている配列に書き込むか、新しい配列を作成して、そこに構成で接続すると、クラスが作成されます。

これは非常に便利です。特に、コンポーネントをプロジェクト間で転送し、スタイルの書き込み速度が数倍になり、プロジェクトが完全に整っているという安心感と満足感だけが残る場合に便利です。

このアプローチは多くのプロジェクトで成功裏にテストされており、この方法論の実装はロシア語と英語のドキュメントですぐに無料で利用できるようになります。
