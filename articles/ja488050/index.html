Amazon Neptuneファーストインプレッション

敬礼、ハブロフスク。AWS for Developersコースの開始を見越して、興味深い資料の翻訳を用意しました。



私たちがbakdata としてお客様のサイトで見る多くのユーザーケースでは、たとえば、ユーザー間の関係、要素間の依存関係、センサー間の接続を分析するときに、関連情報がエンティティ間の関係に隠されています。このようなユーザーケースは、通常、グラフでモデル化されます。今年の初めに、Amazonは新しいNeptuneグラフデータベースをリリースしました。この投稿では、最初のアイデア、優れた実践例、および時間の経過とともに改善できる点を共有したいと思います。

Amazon Neptuneが必要な理由


グラフデータベースは、関連性の高いデータセットよりも高度に接続されたデータセットを適切に処理することを約束します。そのようなデータセットでは、関連情報は通常、オブジェクト間の関係に格納されます。 Neptuneのテストには、オープンデータのMusicBrainzを使用した素晴らしいプロジェクトを使用しました。 MusicBrainzは、アーティスト、曲、アルバムのリリースやコンサートに関する情報や、曲を作成したアーティストが誰とコラボレーションしたか、アルバムがどの国でリリースされたのかなど、音楽に関する考えられるメタデータを収集します。 MusicBrainzは、音楽業界に何らかの形で関連しているエンティティの巨大なネットワークと見なすことができます。

MusicBrainzデータセットは、CSVリレーショナルデータベースのダンプとして提供されます。合計すると、ダンプには157個のテーブルに約9,300万行が含まれます。これらのテーブルの一部には、アーティスト、イベント、レコード、リリース、トラックなどのマスターデータが含まれていますが、その他- リンクテーブル -アーティストとレコード間の関係、他のアーティストまたはリリースなどを格納しています...これらは、セットのグラフィック構造を示していますデータ。データセットをRDFトリプルに変換すると、約5億のコピーが得られました。

私たちが協力しているプロジェクトパートナーの経験と印象に基づいて、この知識ベースを使用して新しい情報を取得する設定を提示します。また、新しいリリースの追加やグループメンバーの更新などにより、定期的に更新されることを想定しています。

一括アップロード


大量のデータを海王星にロードするために提案された方法は、S3を介したバルクロードのプロセスです。triplesファイルをS3にアップロードした後、POSTリクエストを使用してダウンロードを開始します。私たちの場合、5億のトリプルで約24時間かかりました。私たちはそれがより速くなることを期待していました。

curl -X POST -H 'Content-Type: application/json' http://your-neptune-cluster:8182/loader -d '{
 
 
 "source" : "s3://your-s3-bucket",
 
 "format" : "ntriples",
 
 "iamRoleArn" : "arn:aws:iam::your-iam-user:role/NeptuneLoadFromS3",
 
 "region" : "eu-west-1",
 
 "failOnError" : "FALSE"
 
}'

Neptuneを起動するたびにこの長いプロセスを回避するために、これらのトリプルが既にロードされているスナップショットからインスタンスを復元することにしました。スナップショットから開始する方がはるかに高速ですが、Neptuneがリクエストに対応できるようになるまでには約1時間かかります。

Neptuneでトリプルを最初にロードするときに、さまざまなエラーが発生しました。

{
 
 
 "errorCode" : "PARSING_ERROR",
 
 "errorMessage" : "Content after '.' is not allowed",
 
 "fileName" : [...],
 
 "recordNum" : 25
 
}

上記のように、それらのいくつかは解析エラーでした。これまでのところ、現時点で正確に何がうまくいかなかったかはまだわかりません。もう少し詳細は間違いなくここで役立ちます。このエラーは、挿入されたトリプルの約1％で発生しました。しかし、Neptuneのテストに関しては、MusicBrainzからの情報の99％しか処理しないという事実を受け入れました。

SPARQLに慣れている人にとっては簡単なことですが、RDFトリプルには明示的なデータ型で注釈を付ける必要があるため、これもエラーを引き起こす可能性があることに注意してください。

ストリーミングダウンロード


上記のように、Neptuneを静的データウェアハウスとして使用するのではなく、柔軟で進化する知識ベースとして使用します。したがって、ナレッジベースを変更するとき、たとえば、新しいアルバムが公開されたときや派生したナレッジを具体化したいときなど、新しいトリプルを導入する方法を見つける必要がありました。

Neptuneは、SPARQLクエリを介して、生データとサンプルに基づく入力演算子の両方をサポートします。以下で両方のアプローチについて説明します。

私たちの目標の1つは、ストリーミングモードでデータを入力することでした。新しい国でのアルバムのリリースを検討してください。MusicBrainzの観点からは、これはアルバム、シングル、EPなどを含むリリースの場合、新しいレコードがリリースカントリーテーブルに追加されることを意味します。RDFでは、この情報を2つの新しいトリプルと比較します。

INSERT DATA { <http://musicbrainz.foo/release-country/737041> <http://musicbrainz.foo/release> <http://musicbrainz.foo/release/435759> };INSERT DATA { <http://musicbrainz.foo/release-country/737041> <http://musicbrainz.foo/date-year> "2018"^^<http://www.w3.org/2001/XMLSchema#int> };

別の目標は、グラフから新しい知識を得ることでした。各アーティストが自分のキャリアで発表したリリースの数を取得するとします。このようなクエリはかなり複雑で、Neptuneでは20分以上かかるため、この新しい知識を他のクエリで再利用するには、結果を具体化する必要があります。したがって、この情報を含むトリプルをグラフに追加し、サブクエリの結果を紹介します。

INSERT {
 
 
  ?artist_credit <http://musicbrainz.foo/number-of-releases> ?number_of_releases
 
} WHERE {
 
  SELECT ?artist_credit (COUNT(*) as ?number_of_releases)
 
  WHERE {
 
     ?artist_credit <http://musicbrainz.foo/rdftype> <http://musicbrainz.foo/artist-credit> .
 
     ?release_group <http://musicbrainz.foo/artist-credit> ?artist_credit .
 
     ?release_group <http://musicbrainz.foo/rdftype> <http://musicbrainz.foo/release-group> .
 
     ?release_group <http://musicbrainz.foo/name> ?release_group_name .
 
  }
 
  GROUP BY ?artist_credit
 
}

単一のトリプルをグラフに追加するには数ミリ秒かかりますが、サブクエリの結果を挿入するための実行時間は、サブクエリ自体の実行時間によって異なります。

頻繁に使用しなかったという事実にもかかわらず、Neptuneでは、情報の更新に使用できるサンプルまたは明示的なデータに基づいてトリプルを削除することもできます。

SPARQLクエリ


各アーティストのリリース数を返す前のサブサンプルの紹介では、Neptuneを使用して応答したい最初のタイプのリクエストをすでに紹介しています。Neptuneでクエリを作成するのは簡単です。次に示すように、SPARQLエンドポイントにPOSTリクエストを送信します。

curl -X POST --data-binary 'query=SELECT ?artist ?p ?o where {?artist <http://musicbrainz.foo/name> "Elton John" . ?artist ?p ?o . }' http://your-neptune-cluster:8182/sparql

Neptuneは、時間ごとと、I / O操作ごとに支払いを行います。テストでは、1時間あたり0.384ドルの最小のNeptuneインスタンスを使用しました。1人のアーティストのプロファイルを計算する上記のリクエストの場合、Amazonは数万のI / Oを請求します。これは0.02ドルのコストを意味します。

結論


まず、Amazon Neptuneはその約束のほとんどを守っています。マネージドサービスであるため、非常に簡単にインストールでき、多くの設定なしで起動できるグラフデータベースです。ここに5つの主要な発見があります。

  • 一括アップロードはシンプルですが遅いです。しかし、あまり役に立たないエラーメッセージのために複雑になる可能性があります
  • ストリームローディングは私たちが期待したすべてをサポートし、十分に高速でした
  • クエリは単純ですが、分析クエリを実行するほどインタラクティブではありません
  • SPARQLクエリは手動で最適化する必要があります
  • SPARQLクエリによってスキャンされたデータの量を見積もることが難しいため、Amazonの支払いを評価することは困難です。

それで全部です。トピック「負荷分散」に関する無料のウェビナーにサインアップしてください。
