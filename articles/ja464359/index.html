モバイルWebアプリケーションのテストの機能

こんにちは、ハブル。

モバイルWebのテストは、デスクトップWebのテストと多少似ています。一方、これらは、私たちが見慣れているHTML、CSS、JavaScript、およびその他の魅力です。同じ問題領域と典型的なバグ。一方、違いがあります。

この記事では、モバイルWebプロジェクトでテストするために重要な機能の小さなチェックリストをまとめました。リストは完全であるとは主張していないため、コメントの段落で補足してください。嬉しいだけです。唯一のルールは、アイテムがモバイルWebにのみ適用され、Webにはまったく適用されないことです。



モバイルWebプロジェクトをテストするには、少なくとも2つの方法があるという事実から始めたいと思います。1つ目は、Chrome DevTools（または開発ツールの他のブラウザー）を使用してモバイルブラウザーをエミュレートすることですが、これはあまり一般的ではありません）。2つ目は、実際のモバイルブラウザーを使用して実際のデバイスでテストすることです。

ほとんどの機能はデバイスなしで確認することは可能ですが、すべてではありません。したがって、私はチェックを2つの大きな部分に分割しました。PCでチェックするものと、モバイルデバイスを使用している場合のみです。

Chrome DevTools


したがって、前述のように、モバイルデバイスを使用せずにPCでモバイルテストを実行できます。Chromeブラウザはモバイルモードで動作します。

モバイルモード


Webページを表示するモバイルモードに切り替えるには、Chrome DevToolsを開き、この記号をクリックする必要があります。

ユーザーエージェント

すべてを正しく行うには、モバイルユーザーエージェントを設定して、Chrome DevToolsでネットワーク条件をさらに構成する必要があります。



次に、ページをリロードして、必要な結果を取得します。

ネットワークスロットリング


Chromeを使用すると、低速のインターネットと完全にオフラインの両方でアプリケーションをテストできます。これを行うには、同じ[ネットワーク条件]タブで、ネットワークスロットルパラメーターを選択できます。



これは、ユーザーがナビゲーターなどのインターネット環境の悪い場所でアプリケーションを頻繁に使用する場合に重要です。

Chrome DevToolsがすべてではありません。これは、デスクトップWebやモバイルWebを操作するための優れたツールです。もちろん、Googleの優れたドキュメント（英語）があります。

ロシア語のChrome DevToolsに関する2週間のオンラインコースがあります。プロフィールページのすべての情報。次へ。 :)

Chrome DevToolsでモバイルウェブをテストすると、いくつかの利点があります。シンプルで高速、実際のデバイスを必要とせず、最も明らかなアプリケーションのバグを特定できます。しかし、悲しいかな、すべてではありません。

パフォーマンス


モバイルデバイスを選択する必要がある最初の理由：弱いデバイスでアプリケーションをテストする必要性。

最新のWebアプリケーションは、あらゆる種類のアニメーション、クライアント側の複雑な計算などで過負荷になっています。デスクトップ上でこれらすべてがスムーズで美しく機能する場合（ただし、常にそうであるとは限りません）、一部のSamsung J-line（たとえば、J2）に遅延がある場合があります。

モバイルブラウザ


2番目の理由は、モバイルブラウザーです。これは、システムに組み込まれ、デフォルトで設定されているブラウザに関するものです。多くの人がそれらを使用し、モバイルChromeをインストールする手間を省きます。

そのようなブラウザーの代表の1つは、Samsung Internet Browserです。Webアプリケーションの動作を確認する価値があります。特に、「ユーザーが何に座っているか」を示す統計がない場合。そのような統計があり、これらのブラウザから誰もアプリケーションにアクセスしていないと主張する場合、おそらくそれをテストする必要はありません。でも…壊れただけで入ってこなかったら？:)

それについて考える価値があります。

バックグラウンドでの作業


モバイルWebアプリケーションはモバイルブラウザーで機能しますが、これは論理的です。同時に、アプリケーションがアクティブモードとバックグラウンドの両方になるようにモバイルデバイスが配置されます。たとえば、別のアプリケーションに切り替えた場合や、着信があった場合です。

ここで、なんらかの理由で、ユーザーが開いているアプリケーションでブラウザーをバックグラウンドに移動したとします。そして戻ってきた。次に、問題の原因の例をいくつか示します。たとえば、チャットアプリケーションがあり、通信の履歴全体が失われます。ここで、ページの再読み込みが必要です。不完全に？もちろん！または、ノートブックアプリケーションがあります。ユーザーは、着信によって中断されたときに何かを追加する時間がありませんでした。戻って、彼は彼が書いたものが消されたことを発見しました。今、あなたはもう一度すべてを書かなければなりません。それとも、そのようなアプリケーションを使用しない方が良いでしょうか？

バックグラウンドの後で、アプリケーションの主要な機能がどのように機能するかを必ず確認してください。

電子キーボード


ほとんどの場合、モバイルWebユーザーは電子キーボードを使用してテキストを入力します。表示されると、アプリケーションのレイアウトが壊れることがあります。通常、この問題は、画面の比率がどのように計算され、このレイアウトにどのように影響するかに関連しています。



登録ページ、いくつかのアンケートなど、ユーザーが多くのデータを入力する必要があるアプリケーションのページに特別な注意を払う価値があります。

画面の向き


モバイルアプリケーションの画面の向きには、縦向きと横向きの2種類があります。あなたはそれについてここで読むことができます。

デバイスを「自分の側に置く」と、ほとんどのモバイルアプリケーションが新しい画面の比率に「再描画」されます。これはモバイルブラウザにも当てはまります。この時点で、Webアプリケーションのレイアウトは、最も予測できない場所で壊れることもあります。



Chrome DevToolsは両方のモードをエミュレートできますが、それでもWebアプリケーションの切り替えと再描画のプロセスは、実際のモバイルブラウザーとは異なります。

再描画後のアプリケーションの外観は、ぜひチェックする価値があります。そして、できれば両方向に：ポートレートからランドスケープへ、そしてランドスケープからポートレートへ。

ネイティブアプリケーションのWebページ


別の一般的なケース：アプリケーションにモバイルWebバージョンとネイティブアプリケーションの両方がある場合。この場合、一部のページがネイティブアプリケーションに転送されず、WebViewとして表示されるだけです。

WebViewは、アプリケーションにWebページを埋め込むことができるコンポーネントです。組み込みのブラウザは、モバイルWebに表示されるものをアプリケーション内で単純にレンダリングします。多くの場合、これにより、ネイティブアプリケーションの開発時に時間を節約できます。

この場合、ブラウザだけでなくネイティブアプリケーション内でもモバイルWebの変更を確認する価値があります。もちろん、モバイルデバイスなしで行う方法はありません。

タップ加工


マウスをクリックするだけのデスクトップWebとは異なり、モバイルアプリケーションでは、タッチ、タップ、フリックなど、インターフェイスを操作する方法がいくつかあります。これについては、ここなど、多くの場所で読むことができます。

Chrome DevToolsは、これらのアクションの一部をエミュレートできます。しかし、最初に、すべてではありません。また、第2に、実際のデバイスを使用する場合とエミュレーション中の結果が常に同じになるとは限りません。

モバイルWebアプリケーションが特別なアクションに合わせて調整されていない場合でも、少なくともタップによるインターフェースの相互作用を確認する価値はあります。特にメニューやスイッチのような場所。一番下の行は、タップが特定の座標に該当しないということです。彼はエリア全体に落ちます。コントロールが近くにある場合、タップは一度に複数の要素に影響を与え、ユーザーに不便を与える可能性があります。

合計


モバイルWebプロジェクトのテストの主な機能について話しました。デスクトップWebをテストしている場合は、このリストが役立つ可能性があります。もちろん、これはこの問題に関して思い出すことができるすべてではありません。リストを補足したい場合は、必ずチェックをコメントに書き込んでください。
