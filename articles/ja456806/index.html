すべての心配事から1つのボット

「非人間の権利の保護について」という規約が採択されるまでは、これを使用してボットに作業ルーチンを提供する必要があります。今から始めるのは理にかなっており、5年後には車の蜂起が始まり、ボットの感情を退屈な仕事で侮辱する大規模な訴訟が人と機械の関係を規制するための法廷を埋めるでしょう。急いで。

保守的なルーチンと作業方法、確立されたパターンへの順調な奴隷は、機械的な習慣に変わりました。 6文字。

やりたくない仕事がありますが、やらなければなりません。この記事では、コードを含む大規模な挿入や、ボットを最初から作成する方法についての説明は行いません。Dodo Pizzaでリリースプロセスがどのように機能するか、C＃で記述されたボットに退屈なルーチンのいくつかを与えて、自動化と高速化の方法を説明した方がいいでしょう。ボットの機能とロジックに注目します。この資料があなたとあなたの同僚の生活を楽にするアシスタントを書くように促してくれるなら、それはすばらしいことです。

数年前、私たちは週に1回、1つのリリースをリリースしました。2018年5月、リリースの最大レートは147時間であるため、毎日リリースすることを目標に設定しました。今私たちの最小：リリースまで4時間ですが、これは頻繁には起こりません。レコードを修正して、プロセス全体を1つのボタンを押すだけに減らすことができるようにしたいと考えています。

リリースサイクル


現在Dodo ISでは、7つのチームが交互にリリースをリリースしています。チームの1人が「幸運」になります-relizmen。飛行機パイロットとしてのRelizmen：彼には次のアップデートをロールアウトするために巧みに使用する必要のあるレバーとデバイスがたくさんあります。 「オートパイロットを行う時が来ました。統計を退屈なテーブルに入力してオートテストの実行を追跡するよりも、もっとエキサイティングなことに時間を費やすほうがいいでしょう。」

したがって、リリスマンになるためには、チームに順番が来ることが必要です。すべてが明確で、誰も混乱しないように、私たちは壁にチームの名前が入ったステッカーを貼りました。リリースチームは、毎回ハンドルを使って移動する名誉冠を受け取ります。ブラックマーク

を受け取った後Relismanの王冠は、リリース手順のチェックリストがどこにあるかを覚えておくべきです。さらに：彼らは覚えている->発見->テンプレートからコピーを作成し、最終的に、Kaitenシステムでチェックリストを開きました。Kaitenは、開発中のタスクのステータスをカードの形で配置および監視する電子ボードです。新しい開発者にとって、この手順は全体として非常に明白ではありませんでした。そして、どのようにしてこのチェックシートを探すべきか、そして手掛かりがないときにどこから始めるべきかをどうやって知っていますか？

自分の意志をこぶしにまとめたので、我慢すれば十分だと判断しました。ボットに退屈なルーチンのいくつかを与える時が来ました。私たちでなければ誰ですか？いつですか？

完璧に！今、私たちは次に何をすべきか手がかりを持っています。チェックリストを開いて確認します。「Slackでリリースするチャネルを作成し、変更がリリースにあるすべてのチームを招待して、手動テストが必要かどうかを確認します。」これをボットに教えることは残っています。

Slack APIのドキュメントhttps://api.slack.comを開き、そこでチャネルを作成する方法を探します。



ご覧のとおり、Slackでは、他のツールと同様に、複雑なことは何もありません。これらはすべて、2つの必須パラメーターを含む1つのPOST要求を送信することになります（これらは、必須とは反対のパラメーターです）。リクエストでは、作成したチャネルの名前（nameパラメーター）と認証トークン（tokenパラメーター）を渡す必要があります。 「Work with：Token type-user、required scope（s）-channels：write」という行に注意してください。

Slackには2種類のトークンがあります。user-ユーザーに発行されるトークンとbot-アプリケーションに発行されるトークンです。トークンを発行するとき、所有者に付与する権利を決定します。チャネルを作成するには、チャネルを書き込む権限（チャネル：書き込み）を持つユーザートークン（トークンタイプ-ユーザー）が必要です。

Slackにメッセージを送信する際の1つのニュアンスに注意したいと思います。最初は、何か問題が発生した場合の対処方法については考えていませんでした。私たちはSlackでチームを募集し、それに投入したすべてのタスクを実行します。そして、チームの1つが落ちた場合はどうなりますか？私たちの場合、答えは「何もない」でした。そしてそれは悪いことです。私たちの解決策は、現在実行されているアクションをリリースチャットに書き込み、コマンドが完了しなかった場合は、チャットにエラーを報告し、エラーを記録することでした。

2番目に成功したソリューションは、コマンドのアクションの実行状況を保存するデータベースに接続することでした。これで、「/ startregress」コマンドを使用して新しいリリースを開始したので、何かが落ちることを恐れていません。もう一度呼び出すと、コマンドは最初からもう一度実行されます。毎回Slackで新しいチャネルを作成したり、プルリクエストを行ったりする必要はありません。 Slackでチャネルを作成しました-成功した実行のステータスをデータベースに記録しました。このアクションには戻りません。



そこで、1つのボタンをクリックしてリリースチャネルを作成し、タスクがリリースされる全員を招待します。

次に、GithubおよびTeamCityとの統合がありました。私たちはGitflowに取り組んでいます。リリースしたいときは、DEVブランチを取得し、テストに合格したworking = green =からブランチを作成します。

これを行うために、ボットはTeamCityに行き、そこを見てDEVブランチのテスト実行を開始します。自宅の芝生のようにテストが緑の場合は、GitHubに進んで、リリースブランチとプルリクエストを作成してください。

プルリクエストを作成するときは、ロールアウトする変更の説明を追加する必要があります。その後、回天が私たちを助けに来ます。ボットはリリースの名前で列を作成し、DEV列からタスクを取得してリリースに移動します。だから私たちは何が私たちと一緒になるのかを知り、見ています。移動後、ボットはカードの名前をコピーし、カード自体を参照してプルリクエストの説明に追加します。これで、リリースごとにどのタスクが実行されたかを確認できます。リンクからカードを開くと、これらのタスクのすべての詳細を確認できます。



リリースすることはほぼ可能ですが、変更を完全にテストするためだけに残っています。これを行うには、リリースブランチを本番環境（ステージと呼びます）に近い環境にデプロイし、デプロイメント後にテストします。デプロイとテストはTeamCityの1つのパイプラインで組み立てられます。私たちのタスクは、それを起動して、テストが正常に機能することを確認することです。ボットは、回帰の開始時にパイプラインを起動します。

これはすべて手動で行われたという事実から始めたことを思い出させてください。Relizmenはこぶしをこすり、リンクとツールを切り替えました。TeamCity、Kaiten、Slack、Githubなどすべてではありません。そして今、私たちはボットの最初のチームがすでに形成されているアクションのセット全体を持っています。リリースマンは「/ startregress」とダイヤルし、椅子に寄りかかってボットを監視します。

  • プルリクエストを作成し、その説明を入力します

  • 環境へのリリースブランチをリリースしてそこでテストを実行するパイプラインを起動します

リリースプロセス全体を分析し、リリースの各段階にかかる時間を書き留めます。これにより、開発とビジネスは、何が無駄になり、何がユーザーに新しい機能を提供するのを妨げているのかを理解することができます。2日間座ってテストを実行できませんか？したがって、私たちのテストに何か問題があります。もっと時間と注意を払う必要があります。以前は、チェックリストのアクションを実行するために、Googleスプレッドシートに少なくとも5回アクセスしました。そこで1つの日付を入力し、時間を設定します。


OK Google、私たちもあなたを自動化します！テーブルを簡単かつ自然に操作するために、nugetパッケージGoogle.Apis.Sheets.v4をボットのプロジェクトに接続します。そして、他のサービスと同様のスキームに従ってすべてが行われます。どのセルにどの値を挿入するかを言うリクエストを送信します。このクエリは次のようになります。

public void InsertEntry(string cellAddress, string value)
{
//           - _googleSettings.SheetName        - cellAddress
   var range = $"{_googleSettings.SheetName}!{cellAddress}";
   var valueRange = new ValueRange();

//        - value
   var objectsList = new List<object> {$"{value}"};

   valueRange.Values = new List<IList<object>> {objectsList};

//    Google.Apis.Sheets.v4           SpreadsheetId
  
   var insertEntryRequest =
       _service.Spreadsheets.Values.Update(valueRange, _googleSettings.SpreadsheetId, range);
   insertEntryRequest.ValueInputOption =
       SpreadsheetsResource.ValuesResource.UpdateRequest.ValueInputOptionEnum.USERENTERED;
   insertEntryRequest.Execute();
}

Googleとの統合を設定したので、ボットの2番目のコマンド「/ update」を準備しました。

  • 空の文字列を追加して、値を挿入します
  • GitHubに移動し、いつリリースブランチを作成したかを調べ、作成日をタブレットに追加します
  • TeamCityから、パイプラインの最初の開始に関するデータと、パイプラインが正常に終了したときの情報を取得します

次のステップは、リリースマンがリリースをロールすることです。これで、計算は手動で行われます。 1つの国を示し、リリースが戦闘でどのように動作するかを観察します。 KibanaとGrafanaのログと監視ツールに従ってすべてが良好であることを確認した後、次の国をポストします。このプロセスは自動化がそれほど簡単ではありません。しかし、ボットの助けを借りてではありませんが、ここには改善すべき点があります。たとえば、以前は、リリースが可能かどうかをインフラストラクチャチームに尋ねていました。これを行うと、他の作業がサーバーで行われる可能性があり、リリースが不適切になります。

リリースプロセスを最適化するためのミーティングを開催しました。解決策の1つは、リリースマンがSlackチャネルの1つでステータスを確認するだけで、インフラストラクチャが離陸許可を投稿することでした。これは常に同じことを尋ねるよりも便利であり、90％のケースで「できる」という答えを得ることができます。
