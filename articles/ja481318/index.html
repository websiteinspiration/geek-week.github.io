Unicodeによるハッキング（GitHubの例）

Unicodeは非常に複雑です。少数の人々は知っているすべてのトリックを：目に見えない文字と制御文字からサロゲートペアと組み合わせた絵文字に（2つの文字を追加すると、第三があった場合）。標準には、17 プレーンに2 16コード位置が含まれています。実際、Unicodeの学習は、別のプログラミング言語の学習と比較できます。 当然のことながら、Web開発者はニュアンスの一部を見落としています。一方、攻撃者は独自の目的でUnicode機能を使用できます。 セキュリティスペシャリストのJohn GracieがGitHubの使用を実演



忘れたパスワードを回復するためのメールアドレスのバグチェック。同様のバグが他のサイトで見つかります。

ジョングレイシーは、変換後の2つの異なる文字が同じ文字に変換される場合の「符号変換の衝突」について説明します。

この場合、彼はトルコ語の文字 'ı'（ドットなしの 'i'）を使用しました。これはラテン語の 'i'に変換されるため、John@Gıthub.com処理後の住所は次のようになりますJohn@Github.com。

'ß'.toLowerCase() // 'ss'
'ß'.toLowerCase() === 'SS'.toLowerCase() // true

// Note the Turkish dotless i
'John@Gıthub.com'.toUpperCase() === 'John@Github.com'.toUpperCase()

このような衝突は、すべてのUnicodeプレーンで見つかります。完全なリストは次のとおりです。

主にラテン文字に翻訳される文字に関心があります。オプションは11しかありません。表の3番目の位置は、ドットのないトルコ語の記号「i」です。

忘れたパスワードを回復するための手順が正しく機能しなかったため、GitHubにより、攻撃者は他の誰かのアカウントからパスワードを取得することができました。

この手順の一部として、入力された電子メールアドレスが、データベースに保存されているアドレスと比較されました。検証アルゴリズム：

  1. 入力された住所は、toLowerCaseメソッドを使用して小文字に変換されます。
  2. 入力された住所は、登録ユーザーのデータベース内の住所と比較されます。
  3. 一致が見つかった場合は、データベースからのパスワードがに送信され入力されたアドレス。

明らかに、メソッドを使用する場合、開発者はアドレス変換の衝突について知りませんでしたtoLowerCase。

この場合、エラーの修正は簡単です。入力されたアドレスではなく、データベースからのアドレスにパスワードを送信するだけで十分です。

もちろん、これは完全なバグ修正ではなく、単なるパッチです。より完全なソリューションは、確認するためにPunycodeに変換することです：John@Gıthub.com→ xn—john@gthub-2ub.com。 Punycodeは、ドメイン名を一意に一連のASCII文字に変換するように設計されています。メールアドレスは同じ方法で確認できますが、ほとんどのWebアプリケーションは確認できません。

見つかった脆弱性について、ジョングレイシーは現金報酬と格付けで2500ポイントを受け取りましたが、メインのgithubハッカーからはまだ遠いですAlexander Dobkin <img src = 404 onerror = alert（document.domain）>：このような変わった名前のユーザーは、GitHubページが生成されるGitHubサーバーで任意のコードを実行した場合を含め、すでに30,750ポイントを獲得しています。
