JSデザインパターン：オブジェクト作成パターン

それで、ハブロフスク市民、すでに木曜日のOTUSでコース「フルスタックJavaScript開発者」が始まります。そして今日は、コースの開始を見越して、別の著者の出版物を共有しています。



記事の作者：パベルヤクポフ

この記事では、Javascript言語の最も単純で最も一般的なデザインパターンについて説明し、どのパターンをいつ適用するかを説明します。
一般的にデザインパターンは何ですか？デザインパターンは、プログラムの作成における一般的な問題の一般的で頻繁に使用されるソリューションに使用される用語です。

簡単に言えば、設計パターンは十分に頻繁に使用される成功したソリューションです。ソフトウェアエンジニアリングの分野では、JavaScriptプログラミング言語が言及されていないときにも形成された概念としてのデザインパターンは、1994年にBig Fourによって詳細に説明されました。ただし、この記事では、JavaScriptプログラミング言語でのいくつかの単純なパターンの実装について説明します。

主なタイプのデザインパターンについて簡単に説明します。

作成デザインパターン


名前が示すように、これらのパターンは主に新しいオブジェクトを作成するために使用されます。これらには、コンストラクタ、ファクトリ、プロトタイプ、シングルトンなどのパターンが含まれます。

構造設計パターン


これらのパターンは、クラスとオブジェクトの構成に関連付けられています。これらは、システム全体の操作を中断することなく、オブジェクトまたはそのパーツを構造化または再構築するのに役立ちます。

行動設計パターン


このタイプのデザインパターンは、異なるオブジェクト間の通信を改善することを目的としています。行動設計パターンには、コマンダー、イテレーター、メディエーター、オブザーバー、状態パターン、戦略、およびテンプレートが含まれます。

この記事では、オブジェクトの作成に関連するデザインパターンのみを分析します。これは、初心者がto-doアプリケーション、コンソールでのテキストRPG、Canvas上の単純なゲームなどの作成を開始するのに少し簡単で適しているためです。

パターン「モジュール」


このデザインパターンは、いわばJavaScript言語で思わず思われてしまいました。当初、言語はグローバルスコープの汚染などの現象に対抗することを考えていませんでした（C＃やC ++などの他の言語では、この問題を解決する名前空間があります）。また、モジュールは、コードを再利用するために部分的に必要です。これは、モジュールを初めて作成して使用した後、他のチームのプロジェクトに工数を使用してより適切に接続できるためです。
モジュールmoduleは、その存在全体（すでにES5で広く使用されています）でIIFE（すぐに関数式と呼ばれます）を使用していました。

以下は、「モジュール」パターンの簡単な例です。

const first_Module = (function() {
let Surname = "Ivanov";
let Nickname = "isakura313";
  function declare_Surname() {
    console.log(Surname);
  }
  return {
    declare_Nickname: function() {
      alert(Nickname);
    }
  }
})();
first_Module.declare_Nickname();
console.log(Surname); //

「モジュール」パターンは、それに含まれるデータを完全にカプセル化します。パブリックメソッドを使用してのみアクセスでき、「すぐに」使用できるパブリックメソッドとプライベートメソッドのJavaScript実装がこの方法で実装されるまでは。

コンストラクタパターン


これは、オブジェクトを作成するために設計されたデザインパターンです。コンストラクタは、新しいオブジェクトを作成する関数です。ただし、JavaScriptでは、コンストラクター関数やクラス定義がなくても、オブジェクトをその場で作成できます。
コンストラクターは、最も一般的に使用されるデザインパターンの1つです。特定のタイプのオブジェクトを作成するために使用されます。

工場パターン


Factoryパターンは、クラスからオブジェクトを作成することに焦点を当てたもう1つのパターンです。その中で、サブクラスオブジェクトの作成を委任する共通のインターフェイスを提供します。

このデザインパターンは、異なる特性と同じ特性の両方を持つオブジェクトのコレクションを操作するために最もよく使用されます。
以下の例では、テキストクエスト用の敵クラス（EnemyFactory）を作成します。はい、例は非常に単純ですが、これは設計パターンの問題です。これらは「大規模」アプリケーションに必要です。ページをクリックして飛んでいる写真が3枚しかない場合、特に役立つデザインパターンはありません。

さて、コンソールにテキストRPGのコードを表示してください：

class EnemyFactory{
constructor() {
    this.createEnemy = function(type) {
      let enemy;
      if (type === 'goblin' || type === 'ork') enemy = new Infantry();
      else if (type === 'elf') enemy = new Archer();
      enemy.attack = function() {
        return `The ${this._type} is attack`;
      };
      return enemy;
    };
  }
}
class Infantry {
  constructor() {
    this._type = 'goblin';
    this.scream = function() {
      return 'AAAAAAAAA! Za ordu!!!';
    };
  }
}
 
class Archer {
  constructor() {
    this._type = 'elf';
    this.magic_attack = function() {
      return 'Magic fog around you! You cant see!!';
    };
  }
}
 
const enemy_army = new EnemyFactory();
let newGoblin = enemy_army.createEnemy('goblin');
let newElf = enemy_army.createEnemy('elf');
console.log(newGoblin.attack());
console.log(newElf.attack());
console.log(newGoblin.scream());
console.log(newElf.magic_attack());

プロトタイプパターン


ここでは、実際のオブジェクトのあるタイプの「スケルトン」を使用して、新しいオブジェクトを作成します。そしてプロトタイピングは、JavaScriptでOOPを構築する最もネイティブなタイプです。

プロトタイプパターンは、アプリケーションが何らかの形で機能を拡張または削減できる場合に役立ちます。

孤独なパターン


または、彼がよく知られているように、「シングルトン」。「loner」は、クラスのインスタンスが1つだけ存在できる特別なパターンです。パターンは次のように機能します。オブジェクトの初期化は、作成または返された単一のインスタンスがない場合に機能します。存在する場合、トリガーされたオブジェクトが返されます。
主人公のクラスを作成し、ジュマンジのように4ではなく1にしたいとします。

class Hero {
  constructor(name) {
    if (Hero.exists) {
      return Hero.instance;
    }
    this._name = name;
    Hero.instance = this;
    Hero.exists = true;
    return this;
  }
  getName() {
    return this._name;
  }
  setName(name) {
    this._name = name;
  }
}
//   
const smolder = new Hero('Smolder!');
console.log(smolder.getName()); // Smolder!
 
const ruby = new Hero('Ruby');
console.log(ruby.getName()); //  Smolder!
// ,       .         .
//
