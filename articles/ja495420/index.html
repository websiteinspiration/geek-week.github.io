そのようなクラスのアルゴリズムがあります-選択によるソート。一般的な考え方は、配列の並べ替えられていない領域に並べ替えられる最大の要素を検索するため、配列の順序付けられていない部分が削減されるということです。



通常の選択を並べ替えることは、総当たりです。最大値を求めて、配列を線形的に処理するのが簡単な場合、そのようなアルゴリズムの時間の複雑さはO（n 2）を超えることはできません。

束


配列の最高値を処理する最も効率的な方法は、データをヒープと呼ばれる特別なツリー構造に編成することです。これは、すべての親ノードが子孫ノード以上のツリーです。

ヒープの他の名前- ピラミッド、ソートツリー。

配列をツリー形式で表示することがどのように簡単かつほぼ自由であるかを見てみましょう。

配列の最初の要素を取り、それがツリーのルート（レベル1ノード）であると見なします。次の2つの要素は、2番目のレベルのノード、ルート要素の左右の子孫です。次の4つの要素は、3番目のレベルのノード、配列の2番目/ 3番目の要素の右/左の子孫です。次の8つの要素は、第4レベルのノード、第3レベルの要素の子孫です。等。この画像では、バイナリツリーのノードは明らかに配列内の対応する要素の真下にあります。



ただし、図のツリーはそのようなスキャンでより頻繁に描かれます。



この角度を見ると、束による並べ替えがピラミッド型の並べ替えと呼ばれているのは明らかです。ただし、これはチェスの象を将校、ルークをトゥラ、クイーンをクイーンと呼ぶのとほぼ同じです。i番目の要素の

子孫のインデックスは、基本によって決定されます（配列の最初の要素のインデックスが0に等しいと見なされる場合、大部分のプログラミング言語では慣例です）： 左側の子孫2× i + 1、 右側の子：2× i + 2 （チャート上の私アニメーションでは、従来、配列のインデックスは1で始まり、式は少し異なります。左の子：2× iと右の子：2× i + 1




、しかしこれらはすでに小さな算術ニュアンスです）。

これらの数式の結果の子孫がインデックスを超える場合、それはi番目のアイテムに子がないことを意味します。また、起こり得るI番目の要素が左子孫（要素の奇数の配列の最後の要素にあたる）であるが、権利がありません。

したがって、どの配列もツリーの形で簡単に表すことができますが、これはまだヒープではありません。これは、配列では、子孫要素の一部が親要素より大きい場合があるためです。

配列に基づいて作成されたツリーを束にするためには、適切にふるいにかける必要があります。

ふるい分け


束を分類する魂はふるいにかけられています。

要素のふるい分けは、それが不可分なチェーンで結合された子孫よりも小さい場合、この要素を可能な限り低く移動し、大きな子孫を1レベル上げる必要があるということです。

画像は、アイテムのふるい分けパスを示しています。青色は、ふるい分けが行われる要素を示します。緑-枝の下の大きな子孫。それらはスクリーンが作られている青い結び目よりサイズが大きいため、1レベル上に引き上げられます。一番上の青いノードからの要素自体は、緑のチェーンから最も下の子孫の場所に移動されます。



通常のツリーからソートツリーを作成し、この（ソート）状態のツリーをさらにサポートするには、シフトが必要です。

この画像では、配列の要素が再配布されているため、既にヒープに配置されています。配列はソートツリーに分解されますが、ツリー内のすべての子孫は親ノードよりも小さいですが、まだソートされていません（昇順または降順）。ただし、ソートツリーの最大要素は常にメインルートにあります。これは非常に重要です。

def HeapSort(data):

    #   -   
    # (   )       
    for start in range((len(data) - 2) / 2, -1, -1):
        HeapSift(data, start, len(data) - 1)

        #

        HeapSift(data, 0, end - 1)

#   ,      
def HeapSift(data, start, end):

    #   - ,     
    root = start 
    
    #      ,
    #   ,    
    while True:

        child = root * 2 + 1 #  
        #      -  
        if child > end: break 

        #       ,
        #      
        if child + 1 <= end and data[child] < data[child + 1]:
            child += 1

        #     ,   
        #       , 
        #       
        if data[root] < data[child]:
            data[root], data[child] = data[child], data[root]
            root = child
        else:
            break

アルゴリズムの複雑さ


単純なヒープが優れている理由-他のタイプのツリーとは異なり、個別に格納する必要はありません（たとえば、配列に基づくバイナリ検索ツリーを使用する前に作成する必要があります）。どの配列もすでにツリーであり、外出先ですぐに親と子孫を識別できます。追加メモリの複雑さはO（1）であり、すべてがすぐに発生します。

時間の複雑さについては、画面に依存します。O（log n）では、単一のシフティングがバイパスされます。最初に、配列から初期ヒープを構築するために、n要素のシフトを行います。この手順ではO（n log n）がかかります。第二段階では、nを取り出しますヒープからの現在の最大値、残りのソートされていない部分、つまり この段階でもO（n log n）がかかります。

合計時間の複雑さ：O（n log n）+ O（n log n）= O（n log n）。
さらに、ピラミッド型の分類は、退化したケースも、より良いケースもありません。アレイはまともな速度で処理されますが、パフォーマンスの低下や記録はありません。

ヒープの並べ替えは、平均して高速の並べ替えよりも少し遅くなります。しかし、クイックソートの場合は、コンピューターがハングするキラーアレイを選択できますが、ヒープソートの場合はできません。

:: Ternary heapsort


3値ヒープを見てみましょう。あなたはそれをバイナリから信じないでしょう、それは親ノードが最大で2つではなく3つの子孫を持つという点でのみ異なります。i番目の要素の3値ヒープでは、同様に計算された3つの子孫をコード化します（最初の要素のインデックス= 0の場合）：

左の子孫3× i + 1
中間の子孫3× i + 2
右の子孫3× i + 3

（Ifこの記事のアニメーションのように、インデックスは1で始まり、次にこれらの数式では1を引くだけです）。

選別プロセス：



一方で、ツリー内のレベル数はバイナリヒープに比べて著しく減少しています。つまり、平均すると、シフト中のスワップが少なくなります。一方、最小の子孫を見つけるには、より多くの比較が必要になります-子孫が2つではなく3つになるためです。一般的に、時間の複雑さの観点から-どこかで私たちが見つけ、どこかで失うものですが、一般的に同じことです。3値ヒープのデータは、2値データよりも少し速くソートされますが、この加速は非常に小さいです。ピラミッドソートのすべてのバリエーションで、アルゴリズムの開発者はバイナリオプションを採用することを好みます。これは、3進数の実装はおそらく困難であり（アルゴリズムに2〜3行追加することは「より困難」です）、速度の向上は最小限であるためです。

def NHeapSort(data):

    n = 3 #    

    #    
    #   -   
    # (   )       
    for start in range(len(data), -1, -1):
        NHeapSift(data, n, start, len(data) - 1) 

    #        
    #        .
    for end in range(len(data) - 1, 0, -1): 
        #       
        #    
        data[end], data[0] = data[0], data[end]
        #        
        #   
        #     
        NHeapSift(data, n, 0, end - 1)
    return data
    
#  -     N 
def NHeapSift(data, n, start, end):
    
    #   - ,     
    root = start 

    while True:
        
        #   (    )
        #   
        child = root * n + 1
        if child > end: 
            break 

        max = child
        
        #    
        for k in range(2, n + 1):
            current = root * n + k
            if current > end:
                break
                
            if data[current] > data[max]:
                max = current
        
        #     
        #        
        #  
        if data[root] < data[max]:
            data[root], data[max] = data[max], data[root]
            root = max
        else:
            break

ただし、それ以上が良いということではありません。状況を制限してN個の要素の配列のN個の子孫を取得すると、ヒープでの並べ替えは通常の選択による並べ替えに低下します。さらに、意味のないジェスチャーが実行されるため、選択による並べ替えの悪化したバージョンになります。ふるいにかけることで、配列の最初の位置に最大値が配置され、次に最大値が最後に送信されます（選択ソートでは、最大値はすぐに最後に送信されます）。

3進数のヒープが2進数を最小限に追い越す場合、4倍はすでに失われています。複数の子孫の中から最大の子孫を見つけることは、コストがかかりすぎます。
