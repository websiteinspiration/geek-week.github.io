文書化するときにGITを使用する

ドキュメンテーション自体だけでなく、それを処理するプロセスも重要になる場合があります。たとえば、プロジェクトの場合、作業の主な部分はドキュメントの準備に関連しており、誤ったプロセスはエラーや情報の損失につながり、その結果、時間と利益の損失につながる可能性があります。ただし、このトピックが作業の中心ではなく、周辺にある場合でも、正しいプロセスにより、ドキュメントの品質が向上し、時間を節約できます。

ここで概説するアプローチは、特定の実装の例とともに、低いエントリーしきい値を持っています。技術的には、明日は新しい方法で作業を開始できます。

問題の定式化


特定のドキュメントまたはドキュメントのセットを作成する必要があります。おそらく、これはプロジェクトのドキュメントやネットワークのロギング、またはもっと簡単なものです。たとえば、会社や部門のプロセスを説明する必要があります。一般的に、私たちは、テキスト、画像、プレートを含むすべてのドキュメントまたはドキュメントのセットについて話しています...

  1. この作業には、共同作業、1つまたは複数の従業員グループの取り組みが含まれます。
  2. 出力では、特定のテンプレートに従って作成された、企業スタイルの属性を持つ特定の形式のドキュメントが必要です。明確にするために、これはMS Word（.docx）であると想定します

10年前は、アプローチは明確でした。MSWord文書を作成し、何らかの形で変更に関する作業を整理しました。

そして、このアプローチはまだ有効です。大規模なインテグレーターがプロジェクトのドキュメントを作成するときにも使用されます。しかし、多くの編集やディスカッションを伴う集中的な作業を長時間続けてドキュメントを処理している場合、このアプローチはあまり便利ではないことは直感的に明らかです。

Git, Markdown


上記の例で説明した問題に直面して、私はこの問題を調査し始めました。ドキュメントを作成するときにMarkdownをGit
で使用することがますます一般的になっていることを知りました。 Gitは開発ツールです。しかし、それを文書化プロセスに使用しないのはなぜですか？この場合、マルチユーザー作業の問題が解決されます。しかし、Git機能を最大限に活用するには、ドキュメントのテキスト形式が必要です。MSWordではなく、別のツールを見つける必要があります。Markdownは、これらの目的に最適です。



Markdownはシンプルなテキストマークアップ言語です。通常のTXTファイルで美しくデザインされたテキストを作成するように設計されています。 Markdownでドキュメントを作成する場合、Markdown-Gitリンクは自然に見えます。

そして、すべてが問題なく、この場所では、2番目の条件がなければ、「出力では、特定のテンプレートに従って作成された企業スタイルの属性を備えた特定の形式のドキュメントが必要です」（そして私たちは最初に同意した）確かに、それはMS Wordになります）。つまり、Markdownを使用することにした場合は、このファイルを必要なタイプの.docxに変換する必要があります。Pandoc

など、さまざまな形式間で変換するためのプログラムがあります。
このプログラムを使用して、Markdownファイルを.docx形式に変換できます。
しかし、それでも、まず、MarkdownにあるすべてのものがMS Wordに変換されるわけではないことを理解する必要があります。次に、MS Wordは、細身でまだ小さい町Markdownに比べて、国全体です。 Wordには膨大な量のすべてがあり、Markdownには形式がありません。必要なPandocキーを含むMarkdown形式を、切望されているMS Word形式に変換することはできません。そのため、通常、変換後、受信した.docxドキュメントを手動で「調整」する必要があります。これもまた、時間がかかり、エラーにつながる可能性があります。

Pandocができなかったことを自動的に「終了」させるスクリプトを書くことができれば、それは理想的なソリューションです。

MS WordとMarkdownの機能は一般的に同じではないため、この問題を解決することは不可能だと思いますが、特定の状況、特定の要件に関連してこれを行うことはできますか？私の経験によれば、そうです。それは可能であり、多くの場合、あるいはほとんどの場合でさえ可能です。

特定の問題の解決策


したがって、私の場合、Pandocを使用してファイルを変換した後、手動で追加のファイル処理を行う必要がありました。

  • テーブルと画像のキャプションの自動番号付けでWordにフィールドを追加する
  • テーブルのスタイルを変更する

標準（Pandoc）または既知の手段を使用してこれを行う方法は見つかりませんでした。そのため、pywin32パッケージでPythonスクリプトを適用しました。その結果、完全に自動化されました。これで、1つのコマンドでMarkdownファイルを目的の形式のMS Wordドキュメントに変換できます。

詳細はこちら。

注

この例では、もちろん、抽象的なMarkdownファイルを変換しますが、「戦闘」ドキュメントにまったく同じアプローチが適用され、出力では、手動で書式設定して受け取ったものとほぼ同じMS Wordドキュメントを出力で受け取りました。

一般に、pywin32を使用すると、MS Wordドキュメントをほぼ完全に制御できるため、ドキュメントを変更して、企業の標準に必要な外観にすることができます。もちろん、VBAマクロなどの他のツールを使用しても同じ目標を達成できますが、Pythonを使用する方が便利でした。

このアプローチの簡単な公式：

Markdown + Git -- () --> MS Word

「何か」が何であるかはそれほど重要ではありません。私の場合、それはPandocとpywin32を使用したpythonでした。あなたは異なる好みを持っているかもしれませんが、重要なことはこれが可能であることです。そして、これがこの記事の主なメッセージです。

要約すると、このアプローチでは、このアプローチではMarkdownファイルのみを使用し、Gitを使用してコラボレーションとバージョン管理を整理し、必要な場合のみ（たとえば、クライアントにドキュメントを提供するため）、目的の形式のファイル（MS Wordなど）を自動的に作成します。 ）

処理する


多くの場合、上記の式は、ドキュメンテーションを使用するプロセスをどのように整理できるかを理解するには十分だと思います。それでも、私は通常ネットワークエンジニアに焦点を当てているので、プロセスがどのように見えるか、およびMS Wordファイルを編集するアプローチとの違いを概説します。

明確にするために、Gitを操作するためのプラットフォームとしてGitHubを選択します。次に、リポジトリを作成し、使用するMarkdownファイルをマスターブランチに配置します。

単純なgithubフローベースのプロセスを見ていきます。その説明は、インターネットとHabrの両方にあります。

4人がドキュメントに取り組んでいて、あなたもその1人だとします。次に、たとえば、これらの人々の名前で4つの追加のブランチが作成されます。それぞれがローカルで、独自のブランチで動作し、必要なすべてのgitコマンドで変更を行います。

いくつかの完成した作業を完了したら、プルリクエストを作成し、変更のディスカッションを開始します。おそらくディスカッションプロセスでは、何か他のものを追加または変更する必要があることがわかります。この場合、必要な変更を加え、追加のプルリクエストを作成します。最後に、変更が受け入れられ、masterブランチとマージされます（または拒否されます）。

もちろん、これはかなり一般的な説明です。詳細なプロセスを作成するには、開発者に連絡するか、知識のある人を見つけることをお勧めします。しかし、Gitに入るしきい値はかなり低いことに注意したい。これはプロトコルが単純であるという意味ではありませんが、単純なものから始めることができます。何もわからない場合は、数時間から数日かけて学習してインストールすれば、すぐに使えるようになると思います。

新しいプロセスに切り替える方法は？


記事の冒頭で、明日は新しい方法で作業を開始できると書きました。作品を新しい方向に翻訳するには？

以下に、実行しなければならない可能性が高い一連のステップを示します。

  • ドキュメントが非常に大きい場合は、分割してください
  • 各パーツをMarkdownに変換します（たとえば、Pandocを使用）
  • Markdownエディターの1つをインストールします（私はTyporaを使用しています）
  • ほとんどの場合、Markdownによって作成されたドキュメントのフォーマットを修正する必要があります
  • 前の章で説明したプロセスの適用を開始します
  • 並行して、タスクの変換スクリプトの変更を開始（または独自のタスクを作成）

Markdwon->出力ドキュメントタイプに必要な変換メカニズムを作成して完成させるのを待つ必要はありません。実際のところ、Markdownファイルの変換をすばやく完全に自動化できなくても、Pandocを使用して任意の形式で変換し、手動で最終的な形式にすることができます。通常、これを頻繁に行う必要はありませんが、特定の段階の最後にのみ行います。この手作業は不便ではありますが、私の意見では、デバッグ段階ではまだ十分に許容でき、プロセスの速度を大幅に低下させることはありません。

他のすべて（Markdown、Git、Pandoc、Typora）は準備ができており、作業を開始するために多くの労力や時間を必要としません。
