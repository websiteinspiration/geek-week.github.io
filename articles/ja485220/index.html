Webアプリケーションファイアウォールの進化：ファイアウォールから機械学習のクラウドベースのセキュリティへ

以前のクラウド関連の資料では、パブリッククラウドでITリソースを保護する方法と、従来のアンチウイルスがこれらの目的にまったく適していない理由について説明しました。この投稿では、クラウドセキュリティのトピックを継続し、WAFの進化と、ハードウェア、ソフトウェア、クラウドのどれを選択するかについて説明します。 



WAFとは何ですか？


ハッカー攻撃の75％以上がWebアプリケーションおよびサイトの脆弱性を狙っています。このような攻撃は通常、情報セキュリティインフラストラクチャおよび情報セキュリティサービスからは見えません。Webアプリケーションの脆弱性は、ユーザーアカウントや個人データ、パスワード、クレジットカード番号の侵害や詐欺のリスクをもたらします。さらに、Webサイトの脆弱性は、企業ネットワークへの侵入者のエントリポイントとして機能します。

Webアプリケーションファイアウォール（WAF）は、Webアプリケーションへの攻撃をブロックするファイアウォールです。SQLインジェクション、クロスライトスクリプティング、リモートコード実行、ブルートフォース、および認証バイパスです。ゼロデイ脆弱性を使用した攻撃を含みます。アプリケーションファイアウォールは、HTML、DHTML、CSSを含むWebページのコンテンツを監視し、潜在的に悪意のあるリクエストをHTTP / HTTPS経由でフィルタリングすることにより、保護を提供します。

最初の決定は何でしたか？


Webアプリケーションファイアウォールを作成する最初の試みは、90年代前半に行われました。この分野で働く少なくとも3人のエンジニアが知られています。 1つ目は、パデュー大学のコンピューターサイエンスの教授、ジーンスパフォードです。彼はプロキシを使用したアプリケーションファイアウォールのアーキテクチャについて説明し、1991年に「UNIX Security in Practice」という本でそれを公開しました。

2人目と3人目は、Bell Labsの情報セキュリティ専門家であるWilliam CheswickとMarcus Ranumでした。彼らは、アプリケーションファイアウォールの最初のプロトタイプの1つを開発しました。 DECはその配布に従事していました-製品はSEAL（Secure External Access Link）という名前でリリースされました。

しかし、SEALは完全なWAFソリューションではありませんでした。これは、高度な機能を備えたクラシックなネットワークファイアウォールでした。FTPおよびRSHへの攻撃をブロックする機能です。このため、今日の最初のWAFソリューションは、Perfecto Technologies（後にSanctum）の製品と見なされています。 1999年に彼女はAppShieldシステムを導入しました。当時、Perfecto Technologiesはeコマース向けの情報セキュリティソリューションを開発しており、オンラインストアは新製品のターゲットユーザーとなりました。 AppShieldは、動的な情報セキュリティポリシーに基づいて、HTTP要求を分析し、攻撃をブロックすることができました。

AppShieldとほぼ同じ時期（2002年）に、最初のオープンソースWAFが登場しました。彼らはModSecurityになりました。WAFテクノロジを普及させるために作成され、ITコミュニティ（GitHub上のリポジトリ）で引き続きサポートされています。ModSecurityは、正規表現（署名）の標準セット（パターンに対してクエリをチェックするためのツール）に基づいて、アプリケーションへの攻撃をブロックします-OWASPコアルールセット。

その結果、開発者たちはなんとか目標を達成することができました。ModSecurityに基づいて構築されたものを含め、新しいWAFソリューションが市場に登場し始めました。

三世代は歴史


テクノロジーの発展に伴って進化した3世代のWAFシステムを区別するのが慣例です。

初代。正規表現（または文法）で動作します。これにはModSecurityが含まれます。システムサプライヤーは、アプリケーションに対する攻撃の種類を調査し、正当で潜在的に悪意のある要求を表すパターンを生成します。 WAFはこれらのリストをチェックし、特定の状況で何をすべきかを決定します-トラフィックをブロックするかどうか。

正規表現の発見の例は、すでに述べたオープンソースのコアルールセットプロジェクトです。別の例はナクシです、これもオープンソースです。正規表現を使用したシステムにはいくつかの欠点があります。特に、新しい脆弱性が発見された場合、管理者は手動で追加のルールを作成する必要があります。大規模なITインフラストラクチャの場合、数千のルールが存在する可能性があります。非常に多くの正規表現を管理することは非常に困難であり、それらをチェックするとネットワークのパフォーマンスが低下することは言うまでもありません。

正規表現もかなり高いレベルの誤検知を持っています。有名な言語学者のノームチョムスキーは、文法の分類を提案しました。彼は、文法を4つの条件付きレベルの複雑さに分割しました。この分類によれば、正規表現は、パターンからの逸脱を意味しないファイアウォールルールのみを記述できます。つまり、攻撃者は第1世代のWAFを簡単に「だます」ことができます。これに対処する方法の1つは、悪意のあるデータのロジックには影響しないが、署名ルールに違反する特殊文字をアプリケーション要求に追加することです。



第2世代。 WAFのパフォーマンスと精度の問題を回避するために、第2世代のアプリケーションファイアウォールが開発されました。厳密に定義されたタイプの攻撃（HTML、JSなど）を識別する責任を持つパーサーが登場しました。これらのパーサーは、リクエストを記述する特別なトークン（たとえば、変数、文字列、不明、数値）で動作します。悪意のある可能性のあるトークンのシーケンスは、WAFシステムが定期的にチェックされる別のリストに配置されます。このアプローチは、Black Hat 2012カンファレンスで、SQLインジェクションを検出できるC / C ++ libinjectionライブラリの形式で最初に示されました  。

第1世代のWAFと比較して、特殊なパーサーはより高速に動作できます。ただし、新しい悪意のある攻撃が発生した場合、システムを手動で構成することに伴う問題は解決されませんでした。



三代目。第3世代の検出ロジックの進化は、保護されたシステムの実際のSQL / HTML / JS文法に検出文法をできる限り近づけることを可能にする機械学習手法のアプリケーションにあります。この検出ロジックは、再帰的に列挙された文法をカバーするようにチューリングマシンを適応させることができます。さらに、以前は、適応型チューリングマシンを作成するタスクは、ニューラルチューリングマシンの最初の研究が発表されるまで解決できませんでした。

機械学習は、第1世代の検出時に必要なシグネチャのリストを手動で作成することなく、またMemcached、Redis、Cassandra、SSRFなどの新しいタイプの攻撃用の新しいトークナイザー/パーサーを開発することなく、あらゆるタイプの攻撃をカバーするように文法を適応させるユニークな機会を提供します。第2世代の方法論で必要な場合。

3世代の検出ロジックをすべて組み合わせると、3番目の世代の検出が赤いアウトラインで表される新しい図を描くことができます（図3）。この世代には、WebアプリケーションとValarm APIの適応保護プラットフォームの開発者であるOnsekとともにクラウドに実装するソリューションの1つが含まれています。

これで、検出ロジックはセルフチューニングアプリケーションからのフィードバックを使用します。機械学習では、このフィードバックループは強化と呼ばれます。通常、このような補強には1つ以上のタイプがあります。

  • アプリケーション応答動作の分析（パッシブ）
  • スキャン/ファザー（アクティブ）
  • レポートファイル/インターセプター/トラップ手順（事後）
  • マニュアル（監督者により決定）

ハードウェア、ソフトウェア、またはクラウド-何を選択すればよいですか？


アプリケーションファイアウォールを実装するためのオプションの1つは、ハードウェアソリューションです。このようなシステムは、会社がデータセンターにローカルにインストールする特殊なコンピューティングデバイスです。ただし、この場合、独自の機器を購入し、インテグレータにセットアップとデバッグの費用を支払う必要があります（会社に独自のIT部門がない場合）。同時に、すべての機器は古くなり、劣化するため、ハードウェアを更新するための予算を立てる必要があります。

別のWAF展開オプションは、ソフトウェアの実装です。ソリューションは、任意のソフトウェアのアドオンとしてインストールされ（たとえば、ModSecurityはApacheの上に構成されます）、同じサーバー上で実行されます。通常、このようなソリューションは、物理サーバーとクラウドの両方に展開できます。それらのマイナス点は、拡張性とベンダーからのサポートが限られていることです。

3番目のオプションは、クラウドからWAFを構成することです。このようなソリューションは、サブスクリプションサービスとしてクラウドプロバイダーによって提供されます。同社は特殊なハードウェアを購入してセットアップする必要はなく、これらのタスクはサービスプロバイダーの肩にかかっています。重要な点-最新のクラウドベースのWAFは、プロバイダーのプラットフォームへのリソースの移行を意味するものではありません。サイトは、オンプレミスであっても、どこにでも配置できます。

なぜ今、ますます頻繁に曇ったWAFに目を向けるのか、さらに詳しく説明します。
