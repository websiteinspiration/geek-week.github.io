Steam Windowsクライアントローカル権限エスカレーション0day

何年も前から脆弱性を探していましたが、見たことが多かったようですが、慣れず理解できない部分があります。これは、脆弱性や問題に関する情報を受け入れることをベンダーが絶対に嫌うことです。あなたが間違いを犯したことを直接示しているのは非常に不快だと私は理解しています。問題が発生したこと、従業員が機能しなかったことをオープンソースで公に確認することは不愉快です。しかし、なぜ脆弱性情報を拒否する必要があるのか​​わかりません。


つまり、私たちの物語の主人公は、ValveのSteamソフトウェアです。また、その権限昇格の脆弱性により、NT AUTHORITY \ SYSTEMに代わってすべてのユーザーがコマンドを実行できるようになります。

脆弱性


脆弱性自体は非常に単純です。 Steamはその作業のために「Steam Client Service」サービスをインストールします。SDDLサービス記述子を見てください。

O:SYG:SYD:(A;;CCLCSWLOCRRC;;;IU)(A;;CCLCSWLOCRRC;;;SU)(A;;CCLCSWRPWPDTLOCRRC;;;SY)(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;BA)(A;;RPWP;;;BU)

ここでは、パート（A ;; RPWP ;;; BU）に注目しています。この場合、このエントリは、Usersグループのすべてのユーザーがサービスを開始および停止できることを意味します。
起動時にサービスが何をするか見てみましょう。それほど興味深いものではありませんが、異常に見える操作がいくつかあります。このサービスは、HKLM \ SOFTWARE \ Wow6432Node \ Valve \ Steam \ Appsセクションのコンテンツをリストし、すべてのサブセクションにいくつかの権限を設定します。



どんな権利があるのでしょうか？テストキーを作成し、サービス（上記のprocmonのログ）を開始して、何が正しいかを確認します。ここでは、セクションHKLM \ SOFTWARE \ Wow6432Node \ Valve \ Steamは、すべてのサブキーで継承されるすべてのユーザーにフルアクセス権を設定します。RegSetKeySecurityの呼び出しを通じて、HKLM \ SOFTWARE \ Wow6432Node \ Valve \ Steam \ Appsのすべてのサブセクションに同じ権限が設定されるという仮説が生ま​​れました。しかし、レジストリにシンボリックリンクがあり、ブランチHKLM \ SOFTWARE \ Wow6432Node \ Valve \ Steam \ Apps \ testが、たとえばHKLM \ SOFTWARE \ test2を指している場合はどうでしょうか。

対象のレジストリブランチにシンボリックリンクの場合の権限が登録されていることを確認します。



ターゲットブランチの権限をチェックし、SDDLフォーム（興味のないものをスキップ）で確認します。

(A;ID;KA;;;BU)(A;OICIIOID;GA;;;BU)

口頭の形式では、これはすべてのユーザーがフル（読み取りおよび書き込み）アクセスできることを意味します。これらは、Steamサービスがブランチに規定した権利です。

プリミティブがほとんどすべてのレジストリブランチを制御できるようになったので、PoCを簡単に導入できます。私はHKLM \ SYSTEM \ ControlSet001 \ Services \ msiserverブランチを選択しました。これは「Windowsインストーラー」サービスに対応します。これはユーザーが開始することもできますが、サービスはNT AUTHORITY \ SYSTEM特権で動作します。枝の制御取った後、HKLM \ SYSTEM \ ControlSet001の\ Servicesの\ msiserver 我々は実行ファイルのパスを変更するのImagePathと打ち上げサービスmsiserverを。私たちの実行可能ファイルは、最高の権限-NT AUTHORITY \ SYSTEMで起動されます。

上記で記述されたすべてのコードを収集し、SteamがインストールされているすべてのWindowsコンピューターで簡単な権限昇格を取得します。

脆弱性レポート


脆弱性レポートをHackerone経由でValveに送信しました。

ここで最初の驚きは私を待っていた-脆弱性を直接Valveに転送する前に、Valveは「Managed by HackerOne」機能を使用していたので、最初に私が本当に脆弱性レポートを持っていることをハッカーローンのスタッフに納得させなければなりませんでした。私は気にしません-私はNT AUTHORITY \ SYSTEMに代わってコンソールを呼び出すPoCを持っています-顔の証拠。そして、私は報告書で否定を得ます。その理由は、「ユーザーのファイルシステム上の任意の場所にファイルをドロップする機能を必要とする攻撃」のため、私のレポートが調査範囲外であることを示しています。 （攻撃には、任意のファイルシステムパスでファイルを見つける機能が必要です）。私の反応：「私は単一のファイルシステム操作さえしていませんが、その理由で本当に失敗しましたか？」

私はこれについてコメントを書いていますが、別の従業員が到着しましたが、その従業員が脆弱性のチェックを担当しています。それを確認し、Valveを渡します。やったー、目標は達成されました。か否か…？

時間が経過し、レポートは該当なしとして再びマークされます。理由：「ユーザーのファイルシステムの任意の場所にファイルをドロップする機能を必要とする攻撃」と「ユーザーのデバイスへの物理アクセスを必要とする攻撃」（攻撃にはユーザーのデバイスへの物理アクセスが必要）。次に、特権昇格攻撃は、Valveにとって単に興味深いものではないことに気付きました。

小さな統計：この脆弱性はWindows 8 x64、Windows 8.1 x64、Windows 10 x64でテストされました。Steamソフトウェアのバージョン番号の機能がわからないので、サービスコンポーネントのバージョンを修正します。

ちょっとした憶測


私は非常に失望しています。セキュリティが重要だという大げさな言葉を書いている真面目な会社は、あなたのコンピュータを開いて、あなたが走っているどんなプログラムにも最大限のアクセスができるようにします。

ランチャーは実際にはコンピューター上でサードパーティのプログラムを実行するように設計されているため、最大の特権を静かに取得できることはかなり皮肉なことです。未知の開発者が膝の上で作った無料ゲームが正直に動作することを確信していますか？ 90％割引で非表示の鉱山労働者を獲得できないと思いますか？もちろん、一部の脅威は管理者権限なしで機能し続けますが、悪意のあるプログラムの高い権限は神経を著しく損なう可能性があります-ウイルス対策を無効にし、システムのスタートアップに修正し、ほとんどすべてのファイル、ユーザーを変更します。

Steamの人気により、潜在的な犠牲者の数は非常に多くなっています。2015年、アクティブなSteamアカウントの数は1億2500万と推定されました。はい、すべてのSteamユーザーがWindows OSを持っているわけではありませんが、ほとんどのユーザーが確実にWindows OSを持っています。一部のユーザーは、同じマシン上にいくつかの「ライブ」アカウントを持っています。しかし、問題の大きさは依然として印象的です。
それとも、これはすべて意図的に残されているのでしょうか？たぶんSteamは一種の合法的なバックドアですか？これを正確に確立することは不可能ですが、事実を比較してみましょう：

  1. 悪用されやすい脆弱性があります（そして、Twitterではなく、Twitterのメッセージで判断します）。
  2. 検出するのは非常に簡単です。私が最初にそれを見つけたかどうかはわかりませんが、少なくとも最初の1つは公に説明されていました。
  3. そのような脆弱性および類似の脆弱性に関するレポートを受け入れることをためらう（特権の昇格がその範囲から外れるように、範囲は特別に選択されます）。

私はこれすべてが本当に好きではありません。Steamを削除することはお勧めしませんが、Steamの機能には細心の注意を払うことを提案します。セキュリティが危機に瀕しています。
