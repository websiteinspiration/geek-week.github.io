APEX開発コンソールからのアクティビティモニターへのリンク

私のお気に入りのページビューレポートは「加重ページパフォーマンス」です。

アクティビティモニターを使用すると、IRを簡単に操作できますが、さらに詳細が必要で、異なるワークスペースでレポートを実行する場合は、SQL Developerで次のクエリを使用して、できるだけ詳細にすることができます。

select workspace
      , application_name 
      , application_id, page_id
      , count(*) total_page_events
      , avg(elapsed_time) avg_elapsed_time
      , sum(elapsed_time) elapsed_time
from apex_workspace_activity_log
where view_date between to_date('201911190900','RRRRMMDDHH24MISS') and to_date('201911191200','RRRRMMDDHH24MISS')
group by workspace, application_name, application_id, page_id
order by 6, 7 asc

5. v（”）の呼び出しに注意


パフォーマンスの低いレポートを見つけた場合は、バインド変数を使用できるときにv（”）表記を使用したかどうかを確認できます。

select task_name
from tasks
where assigned_to=:APP_USER

または

select task_name
from tasks
where assigned_to=v('APP_USER')

大きなテーブルでは、v（”）は実際には関数呼び出しであるため、2つのステートメントの違いは非常に大きくなる可能性があります。これは、インデックスを利用せず、クエリによって全表スキャンが行われることを意味します。

ヒント：ビューでAPEXセッションの状態を参照する必要がある場合（バインディング変数を使用できない場合）、バインディング変数とほぼ同様に機能するスカラーサブクエリの使用を検討してください。見る：

select task_name
from tasks
where assigned_to = (select v('APP_USER') from dual)

私が参加したイベントの1つでこれを提供してくれたJohn Scottに感謝します。

6.可能であれば、クエリでの文字列置換を避けます


クエリの検索文字列に注意してください。

たとえば、分岐先のページを決定するためにクエリでデコードまたはケースステートメントが必要になる可能性がある状況を考えてみます。

select case when dept_no=20 then
          'f?p=&APP_ID.:3:&SESSION.::::P3_DEPTNO:'||deptno
       else
          'f?p=&APP_ID.:2:&SESSION.::::P2_DEPTNO:'||deptno
       end as link
       , deptno
       , dname
from dept

バインド変数の使用：＆SESSIONルックアップ文字列ではなく、SESSION。大きな違いを生み出し、Or​​acleの解析時間を大幅に節約できます。バインド変数バージョンにより、Oracleはクエリを再利用できます。

select case when dept_no=20 then
          'f?p=&APP_ID.:3:'|| :SESSION ||'::::P3_DEPTNO:'||deptno
       else
          'f?p=&APP_ID.:2:'|| :SESSION ||'::::P2_DEPTNO:'||deptno
       end as link
       , deptno
       , dname
from dept

14.さまざまなIR設定がAPEXパフォーマンスにどのように影響するかを理解します。


対話式レポートの実行が不十分なため、ユーザーが適用したさまざまな設定、パラメーター、またはフィルターは、悪い作業を悪化させる可能性があります（そうです、本当にそうです）。

前述のように、LEVEL9デバッグモードを使用して開始し、個別の実際のクエリを構成することをお勧めします。実行計画を調べ、インデックスを調べ、可能な場合は調整します。各ビュー（テーブル、グループ、グラフ、要約）は個別のクエリであり、カスタマイズが必要になる場合があることに注意してください。その後、検索フィルターまたは列ヘッダーフィルターを使用すると再び変更されます。

MaxRowCount設定も有効になるので、できるだけ小さくする必要があります。これに対する正しい答えはありません。ユーザーに有効なものを入手する前に、別の数字で遊ぶ必要があるかもしれません。

基本的に、いくつかのデフォルトの動作シナリオを削除または変更することを検討してください。ユーザーに対してどの機能を有効にするかに関して、開発者は多くのコントロールを持っています。さまざまな設定を試してみる価値はあります。ユーザーの要件を完全に理解していることを確認するために、ユーザーと相談することを忘れないでください。

最後に、IR がまだ遅すぎる場合は、2つの代替案を検討できます。

  1. パイプラインテーブル関数（select * from table（my_rpt_pipelined））->複雑なクエリでよりよく機能します。
  2. コレクションレポートの生成（APEX_COLLECTION）

これらの非常に役立つIRのヒントを提供してくれたKaren Cannellに感謝します。
