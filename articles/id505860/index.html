Spring Boot, Hibernate, dan Kotlin untuk pemula langkah demi langkah


Halo semuanya, nama saya Oleg, saya ahli teknis di DomKlik. Di tim kami, kernel tumpukan adalah Kotlin dan Spring Boot. Saya ingin berbagi dengan Anda pengalaman saya tentang interaksi dan fitur bekerja dengan PostgreSQL dan Hibernate bersamaan dengan Spring Boot dan Kotlin. Juga, dengan menggunakan contoh layanan microser, saya akan menunjukkan kelebihan Kotlin dan perbedaannya dari aplikasi Java yang serupa. Saya akan berbicara tentang kesulitan yang tidak terlalu jelas yang mungkin dihadapi pendatang baru saat menggunakan tumpukan ini dengan Hibernate. Artikel ini akan bermanfaat bagi pengembang yang ingin beralih ke Kotlin dan terbiasa dengan Boot Spring, Hibernate Java.

Plugin


Untuk aplikasi di Kotlin, sebagai pengumpul proyek, ambil Gradle Kotlin DSL. Daftar plugin yang terhubung akan menjadi standar untuk Spring Boot, dan untuk Kotlin dengan Hibernate kita akan memiliki beberapa yang baru:

plugins {
    id("org.springframework.boot") version "2.2.7.RELEASE"
    id("io.spring.dependency-management") version "1.0.9.RELEASE"
    kotlin("jvm") version "1.3.72"
    kotlin("plugin.spring") version "1.3.72"
    kotlin("plugin.jpa") version "1.3.72"
}

Pertimbangkan tiga yang terakhir.

kotlin(«jvm»)- Plugin Kotlin dasar untuk mengerjakan JVM. Tanpanya, tidak satu pun aplikasi pada Java stack akan mulai.

kotlin(«plugin.spring»)- karena kelas standar di Kotlin adalah final, maka plugin ini secara otomatis akan mengambil kelas ditandai dengan penjelasan @Component, @Async, @Transactional, @Cacheabledan @SpringBootTestterbuka untuk warisan, dan topik yang terkait dengan artikel ini, itu akan memungkinkan kelas ditulis dalam Kotlin akan proksi Spring proksi CGLIB.

Penting untuk dicatat bahwa entitas yang ditandai anotasi @Entity, @MappedSuperclassdan @Embaddabletidak akan menjadi opensetelah menghubungkan steker. Selain itu, get accessormereka juga akan final, dan kemudian kita akan kehilangan kesempatan untuk bekerja dengannya entity reference. Entitas


Seperti dalam aplikasi apa pun dengan lebih dari satu entitas, kami akan membuat leluhur bersama untuk semua orang entity.

BaseEntity:

@MappedSuperclass
abstract class BaseEntity<T> {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity<*>

       return this.id != null && this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}

DepartmentEntity:

@Entity
@Table(name = "department")
class DepartmentEntity(

       val name: String,

       @OneToMany(
               mappedBy = "department",
               fetch = FetchType.LAZY,
               orphanRemoval = true,
               cascade = [CascadeType.ALL]
       )
       val employees: MutableList<EmployeeEntity> = mutableListOf()
) : BaseAuditEntity<Long>() {

   fun addEmployee(block: DepartmentEntity.() -> EmployeeEntity) {
       employees.add(block())
   }

   fun setEmployees(block: DepartmentEntity.() -> MutableSet<EmployeeEntity>) {
       employees.clear()
       employees.addAll(block())
   }
}

Kami tidak menggunakan kelas data . Ini sepertinya keuntungan yang jelas bagi Kotlin di atas Jawa (sebelum versi 14), dan ada penjelasan untuk ini.

Kenapa tidak digunakan?


Kelas data, selain bersifat final sendiri, memiliki bidang tertentu untuk semua bidang equals, hashCodedan toString. Dan ini tidak dapat diterima sehubungan dengan Hibernate.

Mengapa? Dan juga mengapa hashCodeselalu sama dengan konstanta - jawabannya ada di dokumentasi Hibernate itu sendiri . Secara khusus, kami tertarik pada bagian ini:

Meskipun menggunakan natural-id adalah yang terbaik untuk equals dan hashCode , kadang-kadang Anda hanya memiliki pengenal entitas yang menyediakan kendala unik.

Dimungkinkan untuk menggunakan pengenal entitas untuk pemeriksaan kesetaraan, tetapi itu membutuhkan solusi:

  • Anda perlu menyediakan nilai konstan untuk kode hash sehingga nilai kode hash tidak berubah sebelum dan sesudah entitas yang memerah.
  • Anda perlu membandingkan kesetaraan pengidentifikasi entitas hanya untuk entitas yang tidak transien.

Artinya, Anda perlu membandingkan dengan natural id, atau, seperti pada contoh kami, oleh primary key id. Ini akan menghindari banyak masalah saat membandingkan entitas dan menyimpannya dari kerugian saat menggunakan entitas sebagai elemen dalam Set.

Kehadiran toString, ditentukan oleh semua bidang, benar-benar membunuh semua kemalasan, misalnya, ketika masuk entitas, karena semua bidang akan diinisialisasi untuk output ke string.

Mengingat fitur Hibernate, fungsi Kotlin ini tidak cocok untuk kami.

Validitas kosong di bidang hanya sesuai dengan DB


Adapun validitas nilai nulldi bidang, semuanya tidak begitu sederhana. Sebelumnya kami terjun sedikit ke kedalaman Hibernate: berbicara tentang plugin.jpa, saya menyebutkan penggunaan konstruktor tanpa argumen ketika menginisialisasi entitas.

Saat menginisialisasi bidang, refleksi juga digunakan. Dan jika nilai disimpan dalam database di kolom yang sesuai null, kelas diam-diam diinisialisasi dengan bidang ini dengan nilai null. Saat mengaksesnya, kami berisiko terkena NPE, meskipun bidang ditandai sebagai not nullable. Untuk menghindari ini, perlu untuk memantau sinkronisasi struktur tabel dan kelas.

Jika Anda melihat apa yang dijelaskan dalam dua bagian terakhir lebih komprehensif, maka aturan ini berlaku tidak hanya untuk primitif, tetapi juga untuk sekelompok entitas.

Repositori


Saat menggunakan Kotlin, repositori di luar kotak sekarang memiliki centang validasi null. Jadi, jika kami yakin bahwa saat mencari departmentberdasarkan nama, hasilnya akan unik dan unik, maka tipe pengembalian dapat ditentukan sebagai non nullable:

interface DepartmentRepository : JpaRepository<DepartmentEntity, Long> {

   fun findOneByName(name: String) : DepartmentEntity
}

Itu DepartmentEntityditunjukkan satu-satunya dan tidak bisa null. Jika karena alasan tertentu kami tidak menemukan apa yang kami cari department, maka kami tidak akan menangkap NPE, tetapi sesuatu yang lain:

org.springframework.dao.EmptyResultDataAccessException: Result must not be null!

Pemrosesan ini dicapai dengan menambahkan dukungan Kotlin khusus ke MethodInvocationValidatordan ReflectionUtilspada pegas data bersama.

lateinit var


Fitur Kotlin lain yang ingin saya pertimbangkan adalah lateinit var.

Tambahkan baru kelas leluhur: BaseAuditEntity.

@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class BaseAuditEntity<T> : BaseEntity<T>() {

   @CreatedDate
   @Column(updatable = false, nullable = false)
   lateinit var created: LocalDateTime

   @LastModifiedDate
   @Column(nullable = false)
   lateinit var modified: LocalDateTime
}
