We speak and show: how we created the VLC-based ITSkino synchronous video viewing service





The fact that self-isolation is not only “knock-knock-knock” into the coffin of the economy, but also new “horizons of opportunity”, many articles have already been written. True, many of them evoke Krylov’s fable about the fox and grapes. But still, in one respect, the quarantine coaches are right: this whole situation makes us invent — not necessarily some global things that will change the new world order. Sometimes - just small products that helped make self-isolation a little more enduring. We have already briefly written about one of them, the ITSkino player . Now we want to tell you how and what we did to give VLC the function of synchronously viewing streaming or local video in an unlimited number of people at the same time.



We now have two assemblies - for Windows and for MacOS. Source codes can be found at this link . Two of our developers talk about the nuances that were encountered in the process of creating assemblies.

The "center" of the entire system - ShareService - is a singleton that is initialized with the interface module.



Its main functionality:



  • “Intercepting” and processing playlist elements using source addresses starting with itsshare: //
  • processing changes in the state of playlist items (play / pause / stop);
  • handling of the “share” event;
  • registering the player as an itsshare protocol handler.

API :



  1. — api/stream/create
  2. — api/session/get
  3. Play — api/stream/play
  4. Pause — api/session/pause
  5. Stop — api/session/stop


The first time the application is launched, a unique user identifier is generated and the player is registered as the itsshare protocol handler.



If during the playback of a video that is not involved in a joint viewing session, the user clicks on the “share” button (selects the corresponding item in the ITSKino menu), then a request is api/stream/create



sent to the endpoint containing the source (local file name or link to the streaming video) and the current play time. In response, we get a hash of the created session and a link to it, which is automatically copied to the clipboard. This session hash associated with the source is stored for later use when interacting with the API.



When opening a link in a browser
https://itskino.ru/join?stream= <hash>
redirect to 
itsshare: // <hash>
Next, the VLC player opens (since it is the processor of this protocol), into which the corresponding URL is transmitted. ShareService captures the given URL, retrieves the session hash and accesses it at the endpoint api/session/get



to get the playback source. The source is associated with the session hash for later use when interacting with the API.



If you change the playlist element status (play / pause / stop) there is a check of whether there is an associated hash session - and if there is, then it is sent to the appropriate event endpoynt - api/stream/play



, api/session/pause



, api/session/stop



. The query api/stream/play



returns the current play time of the co-viewing session.



If all participants in the session sent a requestapi/session/pause



, then the session is suspended and resumed only at the first request api/stream/play



. When switching playlist items whose sources are associated with the session hash, access to api/session/stop



. A session ceases to exist if all participants in the process are sent api/session/stop



.
