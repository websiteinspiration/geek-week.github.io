EIGRPとは何ですか？これは、RIPなどの距離ベクトルプロトコルとOSPFタイプのリンク状態監視プロトコルの両方の便利な機能を組み合わせたハイブリッドルーティングプロトコルです。



EIGRPは、2013年に公開されたシスコ独自のプロトコルです。彼は、近隣を作成しないRIPとは異なり、チャネル状態追跡プロトコルから近隣を確立するためのアルゴリズムを採用しました。 RIPは他のプロトコル参加者ともルーティングテーブルを交換しますが、OSPFはそのような交換を開始する前に近隣を形成します。 EIGRPも同じように機能します。

RIPは30秒ごとに定期的に完全なルーティングテーブルを更新し、すべてのインターフェイスとすべてのルートに関する情報をそのすべてのネイバーに送信します。 EIGRPは、OSPFのようにHelloメッセージを送信するという概念を使用して、情報の定期的な完全更新を実行しません。数秒ごとに、彼はHelloを送信して、ネイバーがまだ「生きている」ことを確認します。

ルートを形成することを決定する前にネットワークトポロジ全体を調査する距離ベクトルプロトコルとは対照的に、RIPのようなEIGRPは噂に基づいてルートを作成します。私が「噂」と言うとき、私は隣人が何かを言うとき、EIGRPが無条件に同意することを意味します。たとえば、ネイバーが10.1.1.2を達成する方法を知っていると言った場合、EIGRPは尋ねずに彼を信じます。ネットワーク全体のトポロジについて教えてください！」

2013年までは、シスコのインフラストラクチャのみを使用していた場合、このプロトコルは1994年に作成されたため、EIGRPを使用できました。ただし、多くの企業は、シスコの機器を使用している場合でも、このパンクチャを処理することを望んでいませんでした。私の意見では、EIGRPははるかに使いやすいため、今日では最高の動的ルーティングプロトコルですが、人々はまだOSPFを好んでいます。これは、彼らがシスコ製品と「結びつき」たくないという事実によるものだと思います。ただし、このプロトコルはジュニパーなどのサードパーティのネットワーク機器をサポートするため、シスコはこのプロトコルを公開しました。シスコの機器を使用していない会社とチームを組んでも問題はありません。

ネットワークプロトコルの歴史を少し見てみましょう。



1980年代に登場したRIPv1プロトコルにはいくつかの制限があり、たとえば、希望の最大数は16でした。そのため、大規模ネットワークではルーティングを提供できませんでした。少し後、RIPよりもはるかに優れた内部IGRPゲートウェイルーティングプロトコルを開発しました。ただし、これはチャネル状態プロトコルよりも距離ベクトルプロトコルでした。 80年代の終わりに、オープンスタンダードが登場しました-IPv4プロトコル用のOSPFv2チャネルモニタリングプロトコル。

90年代初頭、シスコはIGRPを改善する必要があると判断し、改善された内部EIGRPゲートウェイルーティングプロトコルをリリースしました。 RIPとOSPFの両方の機能を組み合わせたため、OSPFよりもはるかに効率的でした。調査を始めると、EIGRPの構成がOSPFよりもはるかに簡単であることがわかります。シスコは、最速のネットワークコンバージェンスを保証するプロトコルを作成しようとしました。

90年代後半に、RIPv2プロトコルの更新されたクラスレスバージョンがリリースされました。 2000年代に、IPv6をサポートするOSPF、RIPng、およびEIGRPv6の3番目のバージョンが登場しました。世界はIPv6への完全な移行に徐々に近づいており、ルーティングプロトコルの開発者はこれに備える必要があります。

あなたが覚えているなら、距離ベクトルプロトコルとして最適なRIPルートを選択するとき、1つの基準のみがガイドされることを研究しました-希望の最小数、または宛先インターフェースへの最小距離。したがって、ルーターR1はルーターR3への直接ルートを選択しますが、このルートの速度-64 kbit / s-はルートR1-R2-R3の速度よりも数倍低く、1544 kbit / sに等しいです。 RIPプロトコルは、2ホップ長の高速ルートではなく、1ホップ長の低速ルートを最適と見なします。



OSPFはネットワークトポロジ全体を調べ、R2ルーターを経由するルートをR3ルーターとのより高速な接続として使用することを決定します。 RIPは希望数をメトリックとして使用し、OSPFメトリックはコストであり、ほとんどの場合、チャネルの帯域幅に比例します。

EIGRPはルートのコストにも焦点を当てていますが、そのメトリックはOSPFメトリックよりもはるかに複雑であり、帯域幅、遅延、信頼性、負荷、最大MTUパケットサイズなどの多くの要因に依存しています。たとえば、ノードの負荷が他のノードよりも多い場合、EIGRPはルート全体の負荷を分析し、負荷の低い別のノードを選択します。

CCNAコースでは、帯域幅や遅延などのメトリック形成要因のみを考慮し、メトリック式が使用するのは正確にそれらです。



RIPベクトルプロトコルは、距離と方向という2つの概念を使用します。 3つのルーターがあり、そのうちの1つが20.0.0.0ネットワークに接続されている場合、選択は距離に基づいて行われます-これらは希望、この場合は1ホップ、方向、つまりトラフィックを送信する方向（上または下）です。 。

さらに、RIPは情報の定期的な更新を使用して、30秒ごとに完全なルーティングテーブルをネットワーク全体に送信します。このアップデートは2つの機能を実行します。 1つ目は実際にルーティングテーブルを更新すること、2つ目はネイバーの生存率をチェックすることです。デバイスは、30秒以内にネイバーから応答テーブルの更新または新しいルート情報を受信しない場合、ネイバーへのルートが使用できなくなったことを理解します。ルータは30秒ごとに更新を送信して、ネイバーがまだ「生きている」かどうか、およびルートがまだ有効かどうかを確認します。

先ほど述べたように、スプリットホライズンテクノロジーはルーティングループを防ぐために使用されます。これは、更新が元のインターフェイスに送信されないことを意味します。ループを防止するための2番目のテクノロジーはルートポイズンです。図に示されているネットワーク20.0.0.0への接続が中断された場合、接続されたルーターは「汚染されたルート」をネイバーに送信し、このネットワークが16の希望で利用可能になったことを報告します。つまり、実際には到達不可能です。これがRIPプロトコルの動作方法です。

EIGRPはどのように機能しますか？ OSPFに関するレッスンで覚えている場合、このプロトコルは3つの機能を実行します。近隣を確立し、LSAを使用してネットワークトポロジの変更に応じてLSDBベースを更新し、ルーティングテーブルを作成します。近隣の確立は、多くのパラメータを使用するかなり複雑な手順です。たとえば、2WAY接続の確認と変更-一部の接続は双方向通信状態のままで、一部はFULL状態になります。 OSPFとは異なり、これはEIGRPプロトコルでは発生せず、4つのパラメータのみをチェックします。



OSPFと同様に、このプロトコルは10秒ごとに10個のパラメーターを含むHelloメッセージを送信します。 1つ目は、事前に構成されている場合の認証基準です。この場合、近隣が確立されるすべてのデバイスは、同じ認証パラメーターを持っている必要があります。

2番目のパラメータは、デバイスが同じ自律システムに属していることを確認するために使用されます。つまり、EIGRPプロトコルを使用して近隣を確立するには、両方のデバイスが同じ自律システム番号を持っている必要があります。 3番目のパラメータは、Helloメッセージが送信元IPソースと同じIPアドレスから送信されていることを確認するために使用されます。

4番目のパラメーターは、K値の変数係数が一致するかどうかを確認するために使用されます。 EIRGPプロトコルは、K1からK5までの5つの係数を使用します。覚えている場合、K = 0の値ではパラメーターは無視されます。K= 1の場合、パラメーターはメトリックの計算式で使用されます。したがって、異なるデバイスのK1-5値は一致する必要があります。 CCNAコースでは、デフォルトでこれらの係数の値を取ります：K1とK3は1に等しく、K2、K4とK5は0に等しいです。

したがって、これらの4つのパラメーターが一致する場合、EIGRPは近隣関係を確立し、デバイスはネイバーテーブルに相互に入力します。次に、トポロジテーブルが変更されます。



すべてのHelloメッセージはマルチキャストIPアドレス224.0.0.10に送信され、更新は、設定に応じて、ネイバーのユニキャストアドレスまたはマルチキャストアドレスに送信されます。このアップデートはUDPまたはTCP経由では届きませんが、RTP、Reliable Transport Protocol、またはReliability Transport Protocolと呼ばれる別のプロトコルを使用します。このプロトコルは、ネイバーがアップデートを受信したかどうかを確認し、その名前が示すように、その主要な機能は信頼できる通信を保証することです。更新がネイバーに到達していない場合は、受信するまで送信が繰り返されます。 OSPFには受信デバイスをチェックするメカニズムがないため、システムは隣接デバイスがアップデートを受信したかどうかを認識しません。
覚えていれば、RIPはネットワークトポロジ全体の更新を30秒ごとに送信します。 EIGRPがこれを行うのは、新しいデバイスがネットワークに表示された場合、または変更が発生した場合のみです。サブネットのトポロジが変更された場合、プロトコルは更新を送信しますが、完全なトポロジテーブルは送信せず、この変更を含むレコードのみを送信します。サブネットが変更された場合、そのトポロジーのみが更新されます。必要なときに発生する部分的な更新のように見えます。

ご存知のように、OSPFはネットワークに変更があったかどうかに関係なく、30分ごとにLSAを送信します。 EIGRPは、ネットワークに変更がなくなるまで、更新を長期間送信しません。したがって、EIGRPはOSPFよりもはるかに効率的です。

ルーターがサービスパックを交換した後、3番目の段階が開始されます。つまり、図に示す数式で計算されるメトリックに基づいたルーティングテーブルの形成です。彼女はコストを計算し、このコストに応じて決定を下します。
R1がHelloをR2に送信し、HelloをR1に送信したとします。すべてのパラメータが一致する場合、ルーターはネイバーのテーブルを作成します。このテーブルでは、R2がルータR1に関するエントリを書き込み、R1がR2に関するエントリを作成します。その後、R1ルーターは、それに接続されている10.1.1.0/24ネットワークに更新を送信します。ルーティングテーブルでは、これはネットワークのIPアドレス、ネットワークとの通信を提供するルーターのインターフェイス、およびこのインターフェイスを経由するルートのコストに関する情報のように見えます。覚えていると思いますが、EIGRPのコストは90です。次に、後で説明する距離値の距離値が示されます。



メトリックの完全な式は、係数Kの値とさまざまな変換を含むため、はるかに複雑に見えます。公式の完全な形式はシスコのWebサイトに記載されていますが、係数のデフォルト値を置き換えると、より単純な形式に変換されます-メトリックは（帯域幅+遅延）* 256に



なります。このような簡略化された形式の式を使用して、メトリックを計算します。キロビット単位の帯域幅は10 7を最小帯域幅の宛先ネットワークにつながるすべてのインターフェースの最小帯域幅で割ったものであり、累積遅延は宛先ネットワークにつながるすべてのインターフェースの数十マイクロ秒単位の合計遅延です。

EIGRPの調査では、Feasible Distance（可能な距離）、Reported Distance（アナウンスされた距離）、Successor（宛先ネットワークへのコストが最小の隣接ルーター）、Feasible Successor（バックアップ隣接ルーター）の4つの定義を学習する必要があります。それらの意味を理解するために、次のネットワークトポロジを検討してください。



まず、10.1.1.0 / 24ネットワークへの最適なルートを選択するために、ルーティングテーブルR1を作成します。各デバイスの近くに、帯域幅（kbit / s）と遅延（ms）が表示されます。帯域幅が100 Mbps、つまり1,000,000 kbit / sのGigabitEthernetインターフェイス、速度が100,000 kbit / sのFastEthernetインターフェイス、速度が10,000 kbit / sのイーサネット、速度が1544 kbit / sのシリアルインターフェイスを使用します。対応する物理インターフェースの特性についてルーターの設定を見ると、これらの値を見つけることができます。
シリアルインターフェイスの帯域幅はデフォルトで1544 kbit / sであり、64 kbit / sの回線がある場合でも、帯域幅は1544 kbit / sのままです。したがって、ネットワーク管理者は、正しい帯域幅値を使用していることを確認する必要があります。 bandwidthコマンドを使用して特定のインターフェイスに設定でき、delayコマンドを使用してデフォルトの遅延値を変更できます。 GigabitEthernetまたはEthernetインターフェースのデフォルトの帯域幅値について心配する必要はないかもしれませんが、シリアルインターフェースを使用する場合は回線速度を選択するときに注意してください。

この図では、遅延はミリ秒msのように示されていますが、実際にはマイクロ秒なので、マイクロ秒μsを正しく指定するための文字μがありません。

次の状況を注意深く検討してください。 show interface g0 / 0コマンドを入力すると、システムは、マイクロ秒だけでなく数十マイクロ秒の遅延を表示します。

この問題については、EIGRP構成に関する次のビデオで詳しく検討します。今のところ、式では単位ではなく数十マイクロ秒を使用しているため、式の遅延値を代入すると、回路から100μsが10に変わることに注意してください。

この図では、表示されているスループットと遅延が属するインターフェイスを赤い点で示しています。



まず、可能な距離Feasible Distanceを決定する必要があります。これは、数式で計算されるFDメトリックです。 R5から外部ネットワークへのセクションでは、10 7を10 6で除算する必要があります、その結果、10を取得します。次に、10マイクロ秒、つまり1ダースであるため、この帯域幅の値に1に等しい遅延を追加する必要があります。取得した値11に256を掛ける必要があります。つまり、メトリック値は2816になります。これは、このネットワークセクションのFD値です。

R5ルーターはこの値をR2ルーターに送信し、R2の場合は宣言されたReported Distance、つまりネイバーが通知した値になります。したがって、他のすべてのデバイスの宣言されたRD距離は、それについて通知したデバイスの可能なFD距離と等しくなります。

ルータR2はデータに基づいてFDを計算します。つまり、10 7を10 5で割ります。そして、100を取得します。次に、この値に、外部ネットワークへのルートの遅延の合計を追加します。R5遅延は1マイクロ秒、ネイティブ遅延は10テンです。合計遅延は11マイクロ秒になります。結果の100に加算して111を取得し、この値に256を掛けて、値FD = 28416を取得します。 R3ルーターは同じことを行い、計算後、値FD = 281856を受け取ります。ルータR4は値FD = 3072を計算し、それをRDとしてR1に送信します。

FDを計算するときのR1ルーターは、式の1,000,000 kbit / sの帯域幅を置き換えませんが、式は常に宛先ネットワークにつながるインターフェースの最小帯域幅を使用するため、R2ルーターのより低い帯域幅（100,000 kbit / sに等しい）に注意してください。この場合、R2ルーターとR5ルーターはネットワーク10.1.1.0/24の途中にありますが、5番目のルーターの帯域幅が大きいため、R2ルーターの帯域幅の最小値が式に代入されます。パスR1-R2-R5の合計遅延は1 + 10 + 1（数十）= 12で、スループットは100になり、これらの数値の合計に256を掛けると、値FD = 30976になります。

したがって、すべてのデバイスはそれらのインターフェイスのFDを計算し、R1ルーターには宛先ネットワークにつながる3つのルートがあります。これらは、ルートR1-R2、R1-R3およびR1-R4です。ルーターは可能な距離FDの最小値である30976を選択します-これはルーターR2へのルートです。このルーターは、後継者、つまり「後継者」になります。ルーティングテーブルは、フィジブルサクセサ（バックアップサクセサ）も示します。これは、R1とサクセサが切断された場合、ルートはバックアップルータのフィジブルサクセサを経由することを意味します。

フィージブルサクセサは、単一のルールに従って割り当てられます。このルータの宣言されたRD距離は、サイト上のルータのサクセサまでのFDより小さい必要があります。この例では、R1-R2のFD = 30976、R1-R3セクションのRDは281856、R1-R4セクションのRDは3072です。3072<30976なので、R4ルーターがフィジブルサクセサとして選択されています。

つまり、R1-R2ネットワークセクションで通信障害が発生した場合、10.1.1.0 / 24ネットワークへのトラフィックはR1-R4-R5ルートに沿ってルーティングされます。 RIPを使用しているときのルートの切り替えには数十秒かかり、OSPFを使用しているときは数秒かかります。EIGRPでは即座に発生します。これは、他のルーティングプロトコルに対するEIGRPのもう1つの利点です。
