継続的な監視-CI / CDパイプラインでのソフトウェア品質チェックの自動化

誇大広告のテーマはDevOpsです。CI / CDの継続的な統合と配信のコンベアは、それを好むすべての人が実装します。しかし、ほとんどの企業は、CI / CDパイプラインのさまざまな段階で情報システムの信頼性を確保することに常に十分な注意を払っていません。この記事では、ソフトウェアの品質チェックを自動化し、その「自己修復」の可能なシナリオを実装する私の経験についてお話ししたいと思います。

ソース

LANIT-Integrationの ITサービス管理部門のエンジニアとして働いています。。私のコア領域は、アプリケーションのパフォーマンスと可用性を監視するためのさまざまなシステムの実装です。 ITサービスの品質を監視するという局所的な問題について、さまざまな市場セグメントのIT顧客とよくやり取りします。主なタスクは、リリースサイクルタイムを最小限に抑え、リリースの頻度を増やすことです。もちろん、これはすべて問題ありません。リリースの増加-新機能の増加-ユーザーの満足度の向上-利益の増加。しかし、実際には、必ずしもすべてがうまくいくとは限りません。非常に高い展開率では、リリースの品質についてすぐに疑問が生じます。完全に自動化されたパイプラインを使用する場合でも、最大の問題の1つは、稼働時間やアプリケーションとのユーザーの対話に影響を与えずに、テストから本番にサービスを転送することです。

お客様との数多くの会話の結果に基づいて、リリースの品質管理、アプリケーションの信頼性の問題、およびCI / CDパイプラインのさまざまな段階での「自己回復」の可能性（たとえば、安定版へのロールバック）は、最も刺激的で関連性の高いトピックの1つであると言えます。


最近、私自身、オンラインバンキングアプリケーションサポートサービスの顧客側で作業しました。私たちのアプリケーションのアーキテクチャは、多数の自己作成マイクロサービスを使用しました。最も悲しいことは、すべての開発者が高い開発ペースに対応しているわけではなく、一部のマイクロサービスの品質が低下し、それが彼らとその作成者にとんでもないニックネームをもたらしたということです。これらの製品がどのような素材でできているかについての話がありました。


「問題の定式化」


リリースの頻度が高く、多数のマイクロサービスがあるため、テスト段階と運用段階の両方で、アプリケーション全体を理解するのは困難です。変更は常に発生し、適切な監視ツールなしにそれらを制御することは非常に困難です。多くの場合、朝の毎晩のリリース後、開発者はパウダー樽に座って何も壊れないのを待ちますが、テスト段階ではすべてのチェックは成功しました。

もう1つポイントがあります。テスト段階では、ソフトウェアの操作性がチェックされます。アプリケーションの主な機能の実装とエラーがないことです。パフォーマンスの定性的な見積もりが存在しないか、アプリケーションと統合レイヤーのすべての側面が考慮されていません。一部のメトリックはまったくチェックされない場合があります。その結果、実稼働環境で故障が発生した場合、テクニカルサポート部門は、実際のユーザーが不平を言い始めたときしかわかりません。低品質のソフトウェアがエンドユーザーに与える影響を最小限に抑えたいと思います。

ソリューションの1つは、CI / CDパイプラインのさまざまな段階でソフトウェア品質管理プロセスを実装し、事故が発生した場合にシステムを復元するためのさまざまなスクリプトを追加することです。DevOpsがあることも覚えておいてください。企業は、新製品をできるだけ早く受け取ることを期待しています。したがって、すべてのチェックとスクリプトは自動化する必要があります。

タスクは2つのコンポーネントに分かれています。

  • テスト段階でのアセンブリの品質管理（標準以下のアセンブリをキャッチするプロセスを自動化するため）;
  • 実稼働環境でのソフトウェア品質管理（自動問題検出メカニズムおよび自己回復の可能なシナリオ）。

メトリックを監視および収集するためのツール


タスクセットを実現するには、問題を検出してCI / CDパイプラインのさまざまな段階で自動化システムに転送できる監視システムが必要です。また、このシステムがさまざまなチーム（開発、テスト、運用）に役立つメトリックを提供する場合も、良い点になります。そして、もしビジネスならかなり素晴らしい。

メトリックを収集するには、さまざまなシステム（Prometheus、ELKスタック、Zabbixなど）を組み合わせて使用​​できますが、私の意見では、APM（アプリケーションパフォーマンスモニタリング）ソリューションがこれらのタスクに最適であり、これにより人生を大幅に簡略化できます。

護衛サービスでの仕事の一環として、DynatraceのAPMクラスソリューションを使用して同様のプロジェクトを開始しました。現在、インテグレーターとして働いている私は、監視システムの市場をよく知っています。私の主観的な意見：Dynatraceはそのようなタスクに最適です。
Dynatraceソリューションは、各ユーザー操作の水平表示を提供し、コード実行レベルまでの詳細を提供します。Webおよびモバイルアプリケーションのフロントエンドレベル、バックエンドアプリケーションサーバー、統合バスからデータベースへの特定の呼び出しまで、さまざまな情報サービス間の相互作用のチェーン全体を追跡できます。

ステップ1.モニター・システムのデプロイ

まず、テスト環境にエージェントをインストールする必要があります。同時に、Dynatraceソリューションには優れた機能があります。これは、OSインスタンス（Windows、Linux、AIX）にインストールされているOneAgentユニバーサルエージェントを使用して、サービスを自動的に検出し、それらのモニタリングデータの収集を開始します。プロセスごとに個別にエージェントを構成する必要はありません。同様の状況は、クラウドとコンテナのプラットフォームにも当てはまります。同時に、エージェントのインストールを自動化することもできます。 Dynatraceは、「インフラストラクチャとしてのコード」（インフラストラクチャとしてのコードまたはIaC）の概念に完全に適合します。）：すべての一般的なプラットフォーム用の既製のスクリプトと手順があります。エージェントをサービスの構成に組み込み、それをデプロイすると、すでに実行中のエージェントで新しいサービスをすぐに取得できます。

ステップ2.ソフトウェアの品質を評価するイベント

を決定する次に、サービスとビジネス・オペレーションのリストを決定する必要があります。サービスにとってビジネス上重要なユーザー操作を正確に検討することが重要です。ここでは、ビジネスおよびシステムアナリストに相談することをお勧めします。

次に、各レベルの検証に含めるメトリックを決定する必要があります。たとえば、ランタイム（平均、中央値、パーセンタイルなどによる分離）、エラー（論理、サービス、インフラストラクチャなど）、およびさまざまなインフラストラクチャメトリック（メモリヒープ、ガベージコレクター、スレッド数など）があります。

自動化と使いやすさのために、DevOpsチームは「コードとしてのモニタリング」の概念を導入しています。つまり、開発者/テスターは、ソフトウェア品質管理のインジケーターを定義する単純なJSONファイルを記述できるということです。

このようなJSONファイルの例を見てみましょう。 Dynatrace APIのオブジェクトは、キーと値のペアとして使用されます（APIのDynatrace APIの説明を参照してください）。

ファイルは時系列定義の配列です。

  • timeseriesId-チェックされたメトリック。たとえば、応答時間、エラー数、使用メモリなど。;  
  • 集計-メトリックの集計のレベル、この場合はavgですが、必要なもの（avg、min、max、sum、count、percentile）を使用できます。
  • タグ-監視システムのオブジェクトタグ、または特定のオブジェクト識別子を指定できます。
  • 重大および警告-これらのインジケーターはメトリックのしきい値を調整します。テストの値が重大なしきい値を超えると、アセンブリは失敗としてマークされます。

結論

結果として、継続的な統合のパイプラインで自動化された「サービスとしての監視」サービスが提供されます。開発者またはテスターは、JSONファイル内のメトリックのリストを決定するだけでよく、それ以外はすべて自動的に行われます。リリースの透過的な品質管理：パフォーマンス、リソース消費、またはアーキテクチャの回帰に関するすべての通知。

タスク2.本番環境でのソフトウェア品質管理の自動化


そこで、Pipelineのテスト段階で監視プロセスを自動化する方法の問題を解決しました。したがって、本番環境に到達する低品質のアセンブリの割合を最小限に抑えます。

しかし、もし悪いソフトウェアがまだ売りに出されていたり、何かが壊れている場合はどうすればいいのでしょうか。ユートピアについては、存在する問題を自動的に検出するメカニズムと、可能であればシステム自体が夜間でも回復するメカニズムを求めていました。

これを行うには、前のセクションと同様に、本番環境で自動ソフトウェア品質チェックを提供し、その下でシステムを自己修復するためのスクリプトを作成します。


コードとしての自動

修正ほとんどの企業には、プロセスの再起動、リソースのクリーンアップ、バージョンのロールバック、不適切な構成変更の復元、クラスター内のコンポーネントの数の増減、青または青など、さまざまなタイプの一般的な問題とそれらを修正するためのアクションのリストに関する知識ベースが蓄積されています。緑のアウトラインなど

これらのユースケースは、私が通信する多くのチームに長年知られているという事実にもかかわらず、少数の考えだけがその自動化に投資しました。

考えてみれば、アプリケーションの自己修復プロセスの実装はそれほど複雑ではありません。特定のケースごとに事前に記述したコードスクリプト（「コードとしての自動修正」の概念）の形式で、管理者の既知の作業シナリオを提示する必要があります。自動修復シナリオは、問題の根本原因に対処する必要があります。適切なインシデント対応アクションを自分で設定します。

監視システムからの任意のメトリックは、スクリプトを実行するためのトリガーとして機能できます。主なことは、これらのメトリックはすべてが悪いことを正確に判断するということです。私は生産的な環境で誤検知をしたくないのです。

Prometheus、ELK Stack、Zabbixなど、任意のシステムまたはシステムのセットを使用できます。しかし、私はAPMソリューションに基づいたいくつかの例を示します（Dynatraceも例になります）。これは、あなたの人生を楽にするのにも役立ちます。

まず、アプリケーションの観点から、操作性に関連するすべてのものがあります。このソリューションは、トリガーとして使用できるさまざまなレベルで何百ものメトリックを提供します。

  • ユーザーレベル（ブラウザー、モバイルアプリケーション、IoTデバイス、ユーザーの行動、変換など）;
  • サービスと操作のレベル（パフォーマンス、可用性、エラーなど）。
  • アプリケーションインフラストラクチャレベル（ホストOSメトリック、JMX、MQ、Webサーバーなど）。
  • プラットフォームレベル（仮想化、クラウド、コンテナーなど）。

最後に、私はDynatraceが最も好きな別の例を紹介します。

テスト環境でのアセンブリ品質管理の自動化に関する私の話の一部では、すべてのしきい値を手動モードで決定しました。これはテスト環境では正常です。テスター自身が、負荷に応じて、各テストの前にインジケーターを決定します。本番環境では、さまざまなベースラインメカニズムを考慮して、問題が自動的に検出されることが望まれます。

Dynatraceには、異常なメトリック（ベースライン）を決定し、すべてのコンポーネント間の相互作用マップを構築し、コンポーネント間でイベントを比較および相関し、サービスの動作の異常を特定し、各問題と根本原因に関する詳細情報を提供するメカニズムに基づいた興味深い組み込み人工知能ツールがあります。

Dynatraceは、コンポーネント間の依存関係を自動的に分析することにより、問題のあるサービスが主な原因であるかどうかだけでなく、他のサービスへの依存関係も判断します。以下の例では、Dynatraceがトランザクションの一部として各サービスの状態を自動的に監視および評価し、Golangサービスを主な理由として識別します。
