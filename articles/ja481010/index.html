LATEINIT


Kotlinの主要な機能の1つは、ゼロセキュリティへの取り組みです。lateinitステートメントは、ゼロのセキュリティを確保し、Androidの必要に応じて変数を初期化する簡単な方法を提供します。この機能は素晴らしいですが、Javaで作業した後は慣れる必要があります。1つのアイデアは、フィールドがnullになることができるように即座に宣言されることです。

var total = 0 
var toolbar: Toolbar? = null


Androidのレイアウトで作業するとき、彼らは存在する場所、それが明確ではないので、我々は、レイアウトが宣言される前にビューを宣言する方法を知らないので、この言語の関数はで、困難を引き起こす可能性がありますActivityかFragment。これは、私たちが相互作用するすべての場所でゼロ値の可能性に関する追加のテストによって補われますが、これはまだ痔です。したがって、修飾子を使用することをお勧めしlateinitます。

lateinit var toolbar: Toolbar

開発者は、実際に初期化されるまでツールバーを参照しないでください。これはバターナイフのようなライブラリと一緒に使うとうまくいきます：

@BindView(R.id.toolbar) lateinit var toolbar: Toolbar
override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        ButterKnife.bind(this)
        //       toolbar
        toolbar.setTitle("Hello There")
}

レバレッジレット


letを利用すると、オブジェクトの値がゼロ以外の場合にブロックを実行できます。これにより、nullチェックを回避でき、コードが読みやすくなります。Javaでは、次のようになります。

if (currentUser != null) {
    text.setText(currentUser.name)
}
  Kotlin   :
user?.let {
    println(it.name)
}

このコードははるかに読みやすく、それに加えて、ゼロ以外の値を持つ変数を自動的に作成します。

SNULLOREMPTY | ISNULLORBLANK


Androidアプリケーションの開発中にフィールドを何度もチェックする必要があります。Kotlinを使用せずにこれを管理した場合、おそらくAndroidのTextUtilsクラスについて知っています。TextUtilsクラスは次のとおりです。

if (TextUtils.isEmpty(name)) {
    //   
}

この例では、ユーザーがスペースだけをユーザー名として設定でき、テストに合格することがわかります。isNullOrEmptyそして、isNullOrBlankKotlin言語に組み込まれた、の必要性を排除TextUtils.isEmpty（someString）とスペースのみをチェックする追加の利点を提供します。必要に応じて、次のようなものを使用できます。

//       ...
if (number.isNullOrEmpty()) {
//    
}
//  ...
if (name.isNullOrBlank()) {
 //    
}

アプリケーションへの登録が必要な場合、フィールドへの入力が正しいことを確認することがよくあります。これらの組み込みメソッドは、フィールドをチェックし、何か問題がある場合にユーザーに警告するのに最適です。より複雑なチェックの場合は、たとえば、電子メールアドレスに対して拡張メソッドを使用できます。

fun TextInputLayout.isValidForEmail(): Boolean {
    val input = editText?.text.toString()
    if (input.isNullOrBlank()) {
        error = resources.getString(R.string.required)
        return false
    } else if (emailPattern.matcher(input).matches()) {
        error = resources.getString(R.string.invalid_email)
        return false
    } else {
        error = null
        return true
    }
}

上級者向けのヒント


ラムダ式を使用して、より簡潔で簡潔なコードを作成できることをご存知ですか？

たとえば、Javaで作業する場合、次のような単純なリスナークラスを用意するのが一般的です。

public interface OnClickListener {  
    void onClick(View v);
}

Kotlinの特徴は、Javaクラスに対してSAM（Single Abstract Method）変換を実行することです。次のようなJavaのクリックリスナー：

textView.setOnClickListener(new View.OnClickListener() {
    @Override
    public void onClick(View view) {
        // - 
    }
});
 Kotlin     :
textView.setOnClickListener { view ->
// - 
}

奇妙なことに、このような変換はKotlinで作成されたSAMインターフェースでは不可能です。これは、新しいKotlinユーザーを驚かせ、少しでも失望させるかもしれません。同じインターフェースがKotlinで定義されている場合、リスナーは次のようになります。

view.setOnClickListener(object : OnClickListener {
    override fun onClick(v: View?) {
       // - 
    }
})

このようなコードを短縮するには、次のようにクラスにリスナーを記述します。

private var onClickListener: ((View) -> Unit)? = null
fun setOnClickListener(listener: (view: View) -> Unit) {
    onClickListener = listener
}
//    
onClickListener?.invoke(this)

これにより、SAMの自動変換を可能にする単純なラムダ構文に戻ります。
