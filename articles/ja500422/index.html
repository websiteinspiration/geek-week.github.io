Jupyter用ビジュアルデバッガー

ほとんどのソフトウェアプロジェクトの進捗状況は、次に進む前に慎重に評価される小さな変更に基づいています。コード実行の高速な結果と高速な反復開発は、Jupyterの成功の主な理由の一部です。特に科学研究の分野で。

Jupyterのユーザーはノートブックを試すのが大好きで、対話型のコミュニケーション手段としてノートブックを使用しています。ただし、たとえば大規模なコードベースのリファクタリングなど、古典的なソフトウェア開発のタスクについて話す場合、そのような問題を解決するために、Jupyterは通常のIDEに置き換えられることが多いと言えます。


JupyterLab環境Jupyter

プロジェクトは、ノートブックと通常のIDEの間のギャップを埋めるために数年にわたって取り組んできました。これらの取り組みの大部分はJupyterLabプラットフォームによって表され、開発者はファイルマネージャー、テキストエディター、コンソール、メモ帳を含むより高度で便利なインターフェイスを利用できます。

確かに、最近まで、JupyterLabで何かが欠けていました。これは、ユーザーが他の環境への切り替えを余儀なくされた主な理由でした。Jupyterユーザーにはビジュアルデバッガーがありませんでした。ユーザー、特に通常のIDEに慣れているユーザーは、長い間この機能を求めてきました。

Jupyterのデバッガー


今日、開発の数か月後、Jupyterのビジュアルデバッガーの最初のパブリックリリースを発表できることを嬉しく思います！

これは最初のリリースにすぎませんが、デバッガでは既にノートブックセルとソースファイルにブレークポイントを設定できます。変数の内容を調べたり、コールスタックを表示したりできます。そして、彼の能力はこれに限定されません。

バックエンドでは、デバッガーが機能するために、Jupyter Debug Protocolが実装されているカーネルが必要です（これについては後で詳しく説明します）。これまでのところ、このプロトコルを実装する唯一のカーネルはxeus-pythonです。 Pythonプログラミング言語のJupyterコアです。 （プロジェクト開発計画には、ipykernelのデバッガプロトコルのサポートも含まれています。）次の

Jupyterデバッグプロトコルの詳細


ControlコントロールチャネルとIOPubチャネルの新しいメッセージタイプ


Jupyterカーネル（ユーザーコードを実行するインフラストラクチャの一部）は、十分に開発されたプロセス間通信プロトコルを使用してシステムの他の部分と通信します。

いくつかの通信チャネルがあります。

  • Shellリクエスト/レスポンススキームに従って機能し、たとえばコード実行のリクエストに使用されるチャネル。
  • チャネルIOPubは、コアからクライアントまで実行される一方向の通信チャネルです。たとえば、標準出力ストリーム（stdoutおよびstderr）をリダイレクトするために使用されます。
  • チャネルにControl似ていますShellが、別のソケットで動作するチャネル。このため、彼のメッセージはコード実行の要求と一緒にキューに入れられず、優先度が高くなっています。チャネルControlは既にクエリInterruptとに使用されていますShutdown。デバッガーに送信されるコマンドに使用することにしました。

2つの新しいメッセージタイプがプロトコルに追加されました。

  • debug_event, . IOPub.

▍ Debug Adapter Protocol


Jupyterアーキテクチャの主要な原則は、特定のプログラミング言語にバインドされていないことです。したがって、デバッガプロトコルがさまざまなカーネル実装への適応をサポートすることが重要です。

一般的なデバッグ標準は、Microsoftが開発したDebug Adapter Protocol（DAP）です。これは、たとえば、Visual Studio Codeデバッグサブシステムの基礎となるJSONベースのプロトコルです。このプロトコルは、さまざまな言語用に設計された多くのバックエンドをサポートしています。

結果として、最近追加されたメッセージdebug_[request/reply]やを通じてDAPメッセージを使用することは、完全に自然なことでしたdebug_event。

Jupyterカーネルプロトコルに対するこれらの拡張機能はすべて、公式仕様に含める候補として提案されています。関連するJupyter開発の提案（JEPyter Enhancement Proposal、JEP）は、ここにあります。

Xeus-pythonはデバッグをサポートする最初のJupyterカーネルです


XEUSは JupyterカーネルプロトコルのC ++実装です。これ自体は、コアではありません。これはカーネルの開発に役立つライブラリです。このライブラリは、CまたはC ++-API（Python、Lua、SQLなど）を備えた言語用のカーネルの開発に役立ちます。彼女は、 Jupyterメッセージングプロトコルを実装する時間のかかるタスクを担当します。これにより、カーネルの作成者は、言語構造の解釈に関連するタスク（コードの実行、その分析など）に集中できます。CERNで作成されたC ++ clingインタープリターに基づいて、C ++用の

人気のあるxeus-clingコアを含むいくつかのコアがxeusを使用して開発されました。xeus-pythonカーネルは、xeusに基づくipykernelの代替です。ここにxeus-pythonカーネルの最初のリリースに関する資料。

xeus-pythonカーネルは、いくつかの理由により、デバッガプロトコルの最初の実装に適しています。

  • 競争力のあるコード実行をサポートしているためControl、別のスレッドでチャネルを使用して作業を実行できます。
  • コンパクトなコードベースが特徴です。このおかげで、このカーネルは新しい「サンドボックス」であり、何か新しいものを実装するのに便利です。デバッガプロトコルの最初のバージョンをipykernelに実装するには、開発の初期段階で変更を調整するために、より深刻なリファクタリングとより多くの労力が必要になります。

▍Xeus-pythonカーネル開発計画


xeus-pythonの短期的な開発目標には、次のものがあります。

  • IPython機能のサポートを追加します。現在、これはipykernelと比較して、xeus-pythonの主な欠落機能です。
  • xeus-pythonのPyPI-「ホイール」の改善。

other他のコアについて


フロントエンドでのデバッグのサポートに関してすでに行われていることは、拡張カーネルプロトコルを実装するすべてのカーネルで使用できます。

今年は、可能な限り多くのコアでデバッグサポートを提供するよう努めます。

xeus-pythonカーネルと多くの共通点を持つ他のxeusベースのカーネルのデバッグサポートが間もなく登場します。たとえば、xeus-clingなどです。

デバッガフロントエンドアーキテクチャの詳細


JupyterLab のデバッガー拡張機能は、IDEに通常期待されるメカニズムをユーザーに提供します。

  • 変数の監視、ブレークポイントの表示、コードの表示、コールスタックの分析を実装するウィンドウを備えたサイドバー。
  • コード、特にメモ帳のセルとコンソールにあるコードに直接ブレークポイントを設定する機能。
  • コードの実行が停止した位置を示す視覚的なマーカー。

Jupyterノートブックを使用する場合、コード実行のステータスに関する情報はカーネルに保存されます。ただし、セルコードを実行できます。その後、このセルをノートブックから削除できます。ユーザーがデバッグ中にリモートコードブロックを入力したい場合はどうなりますか？

拡張機能はこの特定のシナリオをサポートし、以前に実行されたセルのコードを読み取り専用モードで表示できます。
