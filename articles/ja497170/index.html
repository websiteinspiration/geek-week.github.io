Webサーバー上のトラフィックを保存する方法



ロードされたWebプロジェクトは、テラバイトのトラフィックを消費します。多数の場合、10〜20％節約すると大幅にコストを節約でき、割り当てを超えないようにするのに役立ちます。
トラフィックがホスティング料金の限界に近づいたり、さらに超えたりした場合はどうすればよいですか？

この記事では、Webサーバー上のトラフィックを節約するのに役立つ基本的な手法を分析します。

絞って！


トラフィックを節約する最も簡単な方法は、それを圧縮することです。これにより、サーバープロセッサが読み込まれますが、より高速なデータをクライアントに送信できるため、クライアントのサイズが小さくなり、接続をより早く閉じることができます。主にDeflate互換のアルゴリズムが使用されますが、エキゾチックなアルゴリズムもあります。

Gzip


最も一般的な圧縮アルゴリズム。ロスレス圧縮、良好な圧縮率（1〜9、デフォルトは6で構成可能）および高速アンパッキング。シンプルで効果的で、ほとんどの場合に適しています。



Nginx

gzip            on;
gzip_min_length 1000;
gzip_proxied    expired no-cache no-store private auth;
gzip_types      text/plain application/xml;

Apache

AddOutputFilterByType DEFLATE text/plain
AddOutputFilterByType DEFLATE text/html
AddOutputFilterByType DEFLATE text/xml
AddOutputFilterByType DEFLATE text/css
AddOutputFilterByType DEFLATE application/xml
AddOutputFilterByType DEFLATE application/xhtml+xml
AddOutputFilterByType DEFLATE application/rss+xml
AddOutputFilterByType DEFLATE application/javascript
AddOutputFilterByType DEFLATE application/x-javascript

Zopfli


gzipに代わる最新の方法では、圧縮率が3〜8％向上しますが、はるかに遅くなります（クライアントで同じ速度で解凍されます）。Deflateで動作するため、zlibと100％互換性があり、ブラウザーでのサポートも完全です。

git clone https://code.google.com/p/zopfli/
cd zopfli
make

Nginx

gzip_static     on;

ブロトリ


Zopfliのように、Googleの腸で開発されました。静力学だけでなく、gzipのようにその場で圧縮することもできます。以前のアルゴリズムとは異なり、テキスト内の繰り返しを検索するだけでなく、多くのタグと標準コードフレーズを含む独自のディクショナリを即座にマップします。これは、html / css / js圧縮に非常に効果的です。Zopfliがgzip後に約8％の圧縮を行う場合、その後、Brotliはさらに10〜15％を投げることができ、他の誰かは23％を投げることができます。ただし、これはhttpsでのみサポートされており、zlib / deflateと互換性がありません。Caniuseは心強いです： 標準モジュールの形での



Nginx

サポートはPlusでのみ利用可能であり、通常のNginxはサードパーティモジュール（--add-module = / path / to / ngx_brotli）でビルドする必要があります：

git clone https://github.com/google/ngx_brotli.git
git clone https://github.com/bagder/libbrotli.git
./autogen.sh
./configure

ここではApacheすべてが簡単です。mod_brotliをインストールして、モジュールを構成します。

<IfModule brotli_module>
 BrotliCompressionLevel 10
 BrotliWindowSize 22
 AddOutputFilterByType BROTLI text/html text/plain text/css text/xml
 AddOutputFilterByType BROTLI text/css
 AddOutputFilterByType BROTLI application/x-javascript application/javascript
 AddOutputFilterByType BROTLI application/rss+xml

 </IfModule>

キャッシュしてください！


また、ユーザーとサーバー間のチャネルの負荷を軽減し、リソースをリロードする必要性を最小限に抑えることもできます。ファイルがキャッシュされている場合、次のリクエストで、ブラウザはそのコンテンツをローカルで受信します。

HTTPヘッダーのCache-control、Expires、Varyを使用すると、非常に柔軟なキャッシュポリシーを設計できますが、額のどこにでもmax-age = 2592000を置くことができます。

Nginx

location ~* ^.+\.(js|css)$ {
	expires max;
}

Apache

<ifModule mod_headers.c>
    <FilesMatch "\.(html|htm)$">
        Header set Cache-Control "max-age=43200"
    </FilesMatch>
    <FilesMatch "\.(js|css|txt)$">
        Header set Cache-Control "max-age=604800"
    </FilesMatch>
    <FilesMatch "\.(flv|swf|ico|gif|jpg|jpeg|png)$">
        Header set Cache-Control "max-age=2592000"
    </FilesMatch>
    <FilesMatch "\.(pl|php|cgi|spl|scgi|fcgi)$">
        Header unset Cache-Control
    </FilesMatch>
</IfModule>
<ifModule mod_expires.c>
    ExpiresActive On
    ExpiresDefault "access plus 5 seconds"
    ExpiresByType image/x-icon "access plus 2592000 seconds"
    ExpiresByType image/jpeg "access plus 2592000 seconds"
    ExpiresByType image/png "access plus 2592000 seconds"
    ExpiresByType image/gif "access plus 2592000 seconds"
    ExpiresByType application/x-shockwave-flash "access plus 2592000 seconds"
    ExpiresByType text/css "access plus 604800 seconds"
    ExpiresByType text/javascript "access plus 604800 seconds"
    ExpiresByType application/javascript "access plus 604800 seconds"
    ExpiresByType application/x-javascript "access plus 604800 seconds"
    ExpiresByType text/html "access plus 43200 seconds"
    ExpiresByType application/xhtml+xml "access plus 600 seconds"
</ifModule>

配布してください！


通常、ビジーなサイトに適したCDNは多額の費用がかかりますが、初めて、無料で十分です。CDNを介して重いリソースをダウンロードすると、トラフィックが数倍に削減されます。特にシェアウェアの場合は、この機会をお見逃しなく。インターネット上に無料のネットワークが上位にある1回限りの記事がたくさんありますが、Cloudflareが常に最初に来ます。



また、無料のService Workerも用意されており、好みや花魁を使って手動でCDNを上げることができます。例はいくつかありますが、チュートリアルは公式のSWポータルにあります。

結論


少なくともgzipを使用していない場合は、インターネットへようこそ。ここでは、80％以上のサイトがgzipを使用しています。十分な標準圧縮と-9がない場合は、Zopfliの形式のバックアップでBrotliを使用します（brotleyはまだ100％のカバレッジを持っていないため）。これにより、多くのトラフィックを節約できます。

  • gzip：コンテンツに応じて50〜95％の圧縮率。ウェブ平均65-80％
  • Zopfli：+平均でgzipに比べて3〜8％圧縮、ただし10％発生
  • Brotli：まれなショットが最大20％以上のコンテンツに応じて、gzipに比べて+ 10-15％圧縮



クライアントでデータをキャッシュします。これにより、選択したキャッシングポリシーとサイトでの変更に応じて、繰り返しアクセスする間のトラフィックが99％以下に減少します。コンテンツ配信と基本的なバランシングに

はCDNを使用します。主なトラフィックが数倍に削減される一方で、配信サーバーは大胆です。どの程度正確にネットワーク、負荷、および選択した動作モードに依存します。

上記のすべては可能な限り最短時間で行われ、サーバーアーキテクチャの複雑な再構築を必要としないため、適切な設計に時間をかけることができます。コストを圧縮、キャッシュ、配布、監視して、多くのお金がかからないようにします。
