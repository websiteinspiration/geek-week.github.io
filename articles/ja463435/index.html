モバイルアプリケーションのバックエンドを開発するためのクラウドサービスの概要

バックエンド開発は複雑で費用のかかるプロセスです。モバイルアプリケーションを開発するとき、彼はしばしば不当に多くの注意を受け取ります。モバイルアプリケーションの一般的なシナリオを実装する必要があるたびに、プッシュ通知を送信したり、アクションに関心のあるユーザーの数を調べたり、注文したりするため、正当化されません。二次的なものの実装で品質と詳細を失うことなく、アプリケーションにとって重要なことに集中できるようにするソリューションが欲しいのですが。そして、そのような解決策があります！

このようなサービスは、モバイルバックエンドアズアサービス（MBaaS）と呼ばれます。彼らの助けを借りてバックエンドを作成するためのプロセスは、「手動」の開発と比較して単純化されています。これにより、別のバックエンド開発者を雇う必要がなくなります。また、MBaaSプロバイダーがサーバーの安定性、ロードバランシング、スケーラビリティ、その他のインフラストラクチャの問題に関連するすべての問題を処理するという事実は、結果の品質に信頼を与え、そのようなサービスの主な利点です。

この記事では、Microsoft Azure、AWS Amplify、Google Firebase、Kumulosなど、いくつかの大規模で実績のあるサービスを検討します。



サービスについて検討するポイント：バックエンドと分析の機能、サービス統合の複雑さ、作業の信頼性と安定性、価格ポリシー。各サービスを見て、これらの基準に従って機能を確認してみましょう。

Microsoft Azure





Microsoft Azureは、完全なBaaS機能を含み、モバイルアプリケーションのバックエンドの作成に役立つInfrastructure-As-A-Service（IaaS）サービスです。

MBaaS


Microsoft Azureには、モバイルアプリケーションのバックエンドを作成するためのすべての機能があります。プッシュ通知、自動スケーリング、データ同期、ソーシャルネットワークとの統合などの処理。

Azureの重要な機能は、サーバーの地理的な場所です。それらは世界の54の地域にあり、遅延に適したサーバーを選択する可能性が高くなります。誤動作の場合、ほとんどの場合、特定の地域のみが影響を受けるため、地域が多いほど、その「非常に不安定な」地域に到達する可能性が低くなると想定できます。マイクロソフトによると、他のどのクラウドソリューションプロバイダーよりも多くの地域があります。これは間違いなくプラスです。



分析


このサービスは、アプリケーションの状態をリアルタイムで監視し、「クラッシュ」に関するレポートを収集する機会をリアルタイムで提供します。これにより、問題を即座に特定して解決することができます。

また、Azureでは、独自のライブラリを使用してアプリケーションの分析を収集できます。基本的なメトリック（デバイス情報、セッション情報、ユーザーアクティビティなど）を収集し、追跡用の独自のイベントを作成します。収集されたすべてのデータはすぐにAzureにエクスポートされ、便利な形式で分析作業を行うことができます

追加機能


実際のデバイスでのアプリケーションビルドのテスト、開発プロセスを自動化するためのCI / CD設定、ベータテスト用またはApp StoreまたはGoogle Play

Azureで直接アプリケーションビルドを送信するためのツールなどの興味深い機能もあり、そのまま使用できるように設計されたフレームワークをそのまま使用できますマップおよび地理空間データ。これにより、同様の形式での作業が簡単になります。

特に興味深いのは、人工知能を使用して問題を解決できる可能性です。これにより、さまざまな分析指標を予測し、コンピュータービジョン、音声認識などのすぐに使用できるツールを使用できます。

統合の複雑さ


Microsoft Azureサービスは、主要なモバイルプラットフォーム（iOSおよびAndroid）向けのSDKを提供しますが、クロスプラットフォームソリューション（XamarinとPhoneGap）の場合はそうではありません。 

一般に、ユーザーは複雑なインターフェースと高いエントリーしきい値に不満を持っています。これは、サービス統合に問題がある可能性を示しています。 

高いエントリしきい値はAzureの特別なケースではなく、IaaSの一般的な問題であることを理解することが重要です。たとえば、後で説明するアマゾンウェブサービスもこの病気にさらされます。

信頼性




マイクロソフトのサービスの安定性はまともです。少なくとも月に1回、さまざまな地域で短期的な誤動作が発生する可能性があることがわかります。この図は、サービスが十分に安定していることを示しています。特定の地域では誤動作がほとんど発生せず、非常に迅速に修正されるため、サービスは適切なアップタイムを維持できます。 



これは、Azureサーバーでの最近のインシデントのリストによって確認されます-それらのほとんどは短期間の警告であり、サーバーが最後に「横になった」のは5月上旬でした。統計は安定したサービスの状況を確認します。

費用


価格政策のMicrosoftのAzureサービスの支払いのために異なる関税があり、テストのために十分である一定の制限付きフリープランもあります。 AzureはIaaSサービスであることを覚えておくことが重要です。そのほとんどは、作業リソースの計算の特異性と複雑さのため、作業コストの予測が困難です。多くは困難に直面し、多くの場合、使用済み容量を正しく計算できないことさえあります。実際のスコアは、カウントしたスコアとは大幅に異なる場合があります。 



これらのプランに加えて、AzureにはApp Serviceドメイン、Azure App Service証明書、SSL接続などの個別の有料サービスがあります。これらはすべて、インフラストラクチャの管理に関連しているため、触れません。
多くのレビューで、ユーザーは難しい価格設定ポリシーとサービスのコストを予測できないことについて不満を言っています。マイクロソフトが提案する計算機はダメと呼ばれており、サービス自体が非常に高額です。

Azureの概要


MicrosoftのAzureサービスは、主要なMBaaSプロバイダーとして使用するための機能的で安定したツールです。サービスが最初に本格的なインフラストラクチャを提供するという事実は、モバイルアプリケーションのフレームワークの外でバックエンドをさらに開発する多くの機会を開きます。多数のサーバーとそれらが配置されている膨大な数の地域は、遅延に最も適したサーバーを選択するのに役立ちます。肯定的なユーザーレビューはこれを確認します。否定的な側面-参入の高いしきい値とサービスのコストを予測することの難しさ。

AWS Amplify




アマゾンウェブサービス（AWS）は、リストに載った2番目のIaaSです。これは膨大な数のサービスを表し、Microsoft Azureと同様に、AWS Amplifyと呼ばれる専用の機能セットがあり、これは本質的にモバイルバックエンドです。以前は、MBaaS機能を提供する主なサービスであるAWS Mobile Hubという名前を聞いたことがあるかもしれません。Amplifyは、Amazonの記述方法を変更および改善したMobile Hubであり、前身の基本的な問題を解決しました。



Amazonによると、AmplifyはNetflix、Airbnbなどの多くの大企業から信頼されています。

MBaaS




Amazonのモバイルソリューションでは、モバイルアプリケーションに必要なすべての機能をすばやく構成できます。サーバーロジック、データストレージ、ユーザー認証、またはコンテンツの処理と配信、通知、分析など。 

Amazonは、スケーリング、負荷分散など、必要なインフラストラクチャ条件もすべて提供しています。

分析


分析のために、オーディエンスをセグメント化し、さまざまなチャネル（プッシュ通知、SMSおよび電子メール）を介して大規模なターゲティングキャンペーンを実施して、ユーザーをサービスに引き付けることができる、別のAmazon Pinpointサービスが責任を負います。

Pinpointはリアルタイムのデータを提供し、動的なオーディエンスセグメントを作成し、その関与を分析し、このデータに基づいてマーケティング戦略を最適化できます。

追加機能


Amazon Amplifyは、実際のデバイスでアプリのビルドをテストするためのAWS Device Farmへのアクセスを提供します。このサービスでは、さまざまな物理デバイスでのアプリケーションの並行自動テストが可能で、手動テストも利用できます。

サービスAWS Amplify Consoleは、サーバーリソースとしてデプロイおよびホスティングするためのツールであり、開発プロセスを自動化するためのCI / CD設定を備えたWebアプリケーションです。

また、ユーザーと対話するためのインターフェースとして、音声およびテキストボットをそのままモバイルアプリケーションに導入する可能性もあります。Amazon Lexで動作します。

興味深いことに、AWS Amplifyは、React Nativeアプリケーション用の既製のUIコンポーネントの小さなライブラリも提供します。これは、開発プロセスのわずかな高速化として機能したり、プロジェクトのプロトタイプまたはMVPで使用したりできます。

統合の複雑さ


Amazon Amplifyは、iOS、Android、JavaScript、およびReact Native 用の SDK と、かなり詳細なドキュメントを提供します。RESTに加えて、サービスはGraphQLもサポートすることに注意することが重要です。

Azureの分析プロセスで述べたように、高いエントリしきい値はすべてのIaaSに共通の問題です。アマゾンも例外ではなく、その逆も同じです。これはおそらく理解するのが最も難しいサービスの1つです。これは、AWSが持つさまざまなツールが多数あるためです。AWSをゼロから学ぶにはかなりの時間がかかります。しかし、自分だけをAmplifyに限定すると、適切な時間枠で実際のソリューションを実装できます。

信頼性





Amazonの統計サービスは、Azureほど安定していないようです。しかし、それは少数の完全な停電（赤血球）を満足させます。基本的に、発生するすべてのことは、一部のサービスの動作における警告と不安定性です。

これは、AWSサーバーでの最近のインシデントのリストによって確認されます-それらのいくつかは、異なる期間（時には最大16時間）の警告であり、サーバーが最後に「横になっている」のは6月中旬でした。一般的に、非常に安定しています。



費用





一見したところ、 Amazon Web Services の価格設定ポリシーは非常に簡単です。無料の制限を超えて、使用した分だけを支払います。しかし、Microsoft Azureと同様に、使用するサービスが増えるほど、作業の総コストを予測することが難しくなります。



AWSを高すぎると呼ぶインターネット上の多くのレビューがあります。AWSの使用を特定の金額で最適化して、月々の請求額をできるだけ最小限に抑えようとする企業が長い間いる場合、私は何を言えるでしょう。 

Amazon Amplifyの概要


一般に、Amazon AmplifyのストーリーはAzureに似ています。多くの点で、MBaaSの同様の機能により、本格的なインフラストラクチャとバックエンドの開発の可能性が提供されます。アマゾンのマーケティングツール、特にピンポイントは前向きに際立っています。

否定的な側面の中でも、Azureよりも高いエントリしきい値と、コストの予測と同じ困難さを思い出します。これに安定性の低いサービスを追加し、レビューで判断すると、応答性の高いテクニカルサポートではない

Google firebase



Googleの Service Firebaseは、アプリケーションのサービスMBaaSとして最も興味深いオプションの1つです。それは長い間、有用なツールとしての地位を確立しており、Shazam、Duolingo、Lyftなどの多くの有名なアプリケーションに適しています。 


MBaaS




Firebaseは、モバイルアプリケーションに必要なすべてのことを処理します。このサービスは、データストレージ、同期、認証、クラウド機能（バックエンドコードを実行）などの本格的なバックエンド機能を組み合わせており、現時点では、ベータ版にはマシンに基づくさまざまな機能を実装するMachine Learning Kitが含まれています学習（テキストの認識、写真内のオブジェクトなど）。 

分析


Firebaseの重要な機能は、バックエンド機能に加えて、サービスがアプリケーション分析の幅広い機会を提供することです。組み込みのGoogleアナリティクス、ユーザーベースのセグメンテーション、プッシュ通知の操作。また2017年、Googleは広範なファブリックサービスを購入し、それをCrashlyticsと一緒にFirebaseに統合することで急激な買収をマークしました。

追加機能


Firebaseには、コンテンツへのダイナミックリンクを処理するためのFirebase Dynamic Linksツールが用意されています。このツールを使用すると、アプリケーションにつながるリンクを生成できます。インストールされていない場合は、ユーザーをApp StoreまたはGoogle Playに送信してインストールしてもらいます。また、そのようなリンクは、リンクが開いているデバイスに応じて機能します。コンピューターの場合は、ブラウザーでページが開き、デバイスがアプリケーションへの移行の場合です。

Googleでは、Firebase A / Bテストを使用してアプリケーションのA / Bテストを実施し、Remote Configツールを使用してリモート構成を構成することもできます。 

統合の複雑さ


このサービスは、アプリケーションに非常に多くの機能を組み合わせていることが明らかになります。Firebaseを統合するには、iOS、Android、JavaScript、およびC ++とUnityを含む必要なプラットフォームのSDKを使用する必要があります。これは、ゲームを開発している場合に非常に役立ちます。Firebaseにはかなり詳細なドキュメントと開発者の幅広いユーザーベースがあり、その結果、質問への回答であれ、記事のレビューであれ、ネットワーク上に多数のサポートコンテンツがあることに注意することが重要です。

信頼性


Googleに依存する必要があります-別の記事。一方では、安定性が高く機能しているプロバイダーがあり、他方では、Googleがいつこのサービスを終了するかはわかりません。グーグルがミッション「悪魔にならないでください」から削除されたことも不思議ではありません。 



プロバイダーがそのようなリソースを持っている場合、稼働時間は100％を目指す必要があるように見えますが、サービスの問題に関する多くのメッセージ（たとえば、ユーザーの1人からの引用）を見つけることができます。 Firebaseの場合、「稼働時間」が発生すると言うかもしれません。実際、Firebaseサービスでのイベントの統計を見ると、わずかなダウンタイムと5〜7時間の完全なシャットダウンの両方があることがわかりますが、これはサービスにとって重要な場合があります。



しかし、問題が数週間続くことが起こります。製品の重要で不可欠なコードがこれらのサービスで実行できることを忘れてはなりません。これらの統計は非常に満足していないようです。

費用




Firebaseの料金ポリシーは明確でシンプルです。Spark、Flame、Blazeの3つのプランがあります。彼らはイデオロギー的にお互いに異なっています。Sparkは、プラットフォームの機能の大部分をデプロイしてテストできる制限付きの無料プランです。FlameとBlazeのプランには、有料の使用が含まれます。Flameの固定コストは1か月あたり25ドルですが、基本的には大幅に大きな制限がある場合に限り、同じSparkを入手できます。 

ブレイズは他のものとは異なります。これにより、使用するリソースに比例して料金を支払いながら、プラットフォームの機能を無制限に使用できます。これは非常に柔軟なプランであり、使用する機能に対してのみ支払います。たとえば、プラットフォームをアプリケーションのテストにのみ使用する場合、無料のテスト制限を超えた場合にのみ料金が発生します。

一般に、Firebaseの料金は非常に透過的で予測可能です。このプロセスでは、この機能またはその機能にかかるコストを理解し、サービスのスケーリングまたは変更にかかるコストを計算します。

Firebaseの概要


GoogleのFirebaseサービスは、AWSとAzureが直接関連しているインフラストラクチャの問題を制限する本格的なMBaaSプロバイダーです。クラウドバックエンドの開発に必要なすべての機能が整っており、分析の十分な機会、比較的容易な統合、かなり低いエントリしきい値、および透明な価格設定です。 

否定的な側面-サービスの安定性に関する問題。残念ながら、これはどのような影響も受けません。Googleのエンジニアにのみ頼ることができます。

クムロス




Kumulosは、2011年に設立されたスタンドアロンのMBaaSサービスです。 

MBaaS


モバイルバックエンドとして、Kumulosは以前のサービスですでに見た多くの標準ツールを提供しています。スケジュールと地理位置情報、クラッシュの追跡と診断、Slack、Trello、Jiraとの便利な統合、データストレージ、ユーザー認証処理に基づいて、本格的なキャンペーンを作成することもできます。

Firebaseと同様に、このサービスは、負荷分散、スケーリング、その他のインフラストラクチャの問題に関するすべての問題を処理します。

分析




Kumulosには、定期的なレポートの生成、ユーザーのセグメンテーション、行動の詳細な分析、コホート分析など、広範な分析機能が組み込まれています。このプラットフォームはもともとビッグデータ用に作成されたものであり、多くのデータを処理する準備ができています。すべての分析はリアルタイムで表示されます。内部分析エンジンは、収集された統計に基づいてさまざまな洞察を予測します。

重要な機能は、Salesforce、Google BigQuery、Amplitude、Tableauなどの他のサービスにデータを保存およびエクスポートする機能です。

追加機能




興味深い、あまり見られない機能は、App Storeでのアプリケーションプロモーションを最適化するためのツールです。Kumulos App Store Optimizationは、アプリケーションのページを評価し、パフォーマンスを改善するソリューションを提供します。ユーザーの評価やさまざまな国のトップでのアプリケーションの位置など、アプリケーションの成功要因を監視し、このデータに基づいてレポートが生成されます。 



さまざまな顧客のアプリケーションデータを管理するための便利なインターフェイスを提供するモバイル開発スタジオ向けの特別なツールが利用できることは、非常に興味深いようです。また、特に顧客向けのレポートを生成します。

統合の複雑さ


Kumulosには、ネイティブツールとクロスプラットフォームツールの両方と統合するための幅広いSDKがあります。ライブラリは積極的に更新され、維持されています。



すべてのツールについて詳細なドキュメントが説明されており、プラットフォームを使用するためのチュートリアルや既製の例もいくつかあります。

信頼性


残念ながら、クムロスサービスのサーバーの安定性に関する統計は見つかりませんでした。

費用


無料の試用版に加えて、Kumulosには、スタートアップ、エンタープライズ、エージェンシーの3つの有料プランがあります。彼らは「私が使った分だけを支払う」という原則に基づいています。残念ながら、このサービスはパブリックドメインの価格表を提供していません。ニーズに基づいて個別に計算されているようです。

結論


アプリケーションまたはサービスの開発とその後の開発に劇的な影響を与えるため、モバイルバックエンド用のクラウドサービスの選択の問題を非常に真剣に考えることが重要です。 

この記事では、Microsoft Azure、AWS Amplify、Google Firebase、Kumulosの4つのサービスについて検討しました。その中には、2つの大きなIaaSサービスと2つのMBaaSがあり、これらは特にモバイルバックエンドに特化しています。そして、各オプションには、特定の問題と否定的な側面がありました。

完璧な解決策はないことを理解することが重要です。プロジェクトのテクノロジーの選択は、主要な要素の間の妥協です。私はそれらの上を歩くことをもう一度提案します：

機能性


直接選択する機能は、バックエンドに課すプラットフォームの機能によって異なります。たとえば、コストを節約するためにプッシュ通知を使用する場合でも、1つのエコシステム内に独自のインフラストラクチャを構築してバックエンドを一元化および均一化する場合でも、サービスを選択するときは、優先事項を常に明確に認識する必要があります。 

分析


分析なしで現代のサービスを想像するのは難しいです。結局のところ、サービスを改善し、ユーザーを分析し、最終的に大きな利益を生み出すことができるのはこのツールです。分析の品質と機能は、最終製品の品質を直接決定します。しかし、Firebaseの分析部分、YandexのAppMetricaなど、より適切なサードパーティの分析を接続することに煩わされる人はいません。

統合の複雑さ


統合の複雑さは、開発プロセスにおける金銭的および時間的リソースの両方のコストに直接影響します。もちろん、不人気またはツールキットへの進入に対する高いしきい値のために、開発者の検索プロセスが複雑になる可能性は言うまでもありません。

信頼性と安定性


サービスの信頼性と安定性は、最も重要な指標の1つです。そして、あなた自身のアプリケーションがプロバイダー側​​の問題に苦しんでいるとき、状況は快適ではありません。エンドユーザーは、そこで何が問題になっているか、サービスが機能していないという事実に責任があるかどうかを気にしません。彼は彼が計画したことを行うことができず、すべて、印象が台無しになり、彼はおそらく製品に戻らないでしょう。はい、理想的なサービスはありませんが、プロバイダー側​​で問題が発生した場合の損失を最小限に抑えるためのツールがあります。

価格方針


多くの人にとって、サービスの価格設定ポリシーは決定的な要素です。金融機能がプロバイダーの要求と一致しない場合、一緒に作業を続けることができないからです。製品が依存するサービスのコストを考慮して予測することが重要です。すべてのサービスの料金は異なりますが、送信される通知の数や使用されるストレージハードディスクのサイズなど、ほとんどの場合、使用するリソースに比例します。

ベンダーロック


これらのサービスを使用する場合、1つのソリューションに縛られないことが重要です。そうしないと、いわゆる「ベンダーロック」に完全に依存し、自分自身を破滅させてしまいます。これは、サービスに何かが発生した場合、所有者が変更され、開発の方向性または終了時に、新しいMBaaSプロバイダーを早急に探す必要があり、アプリケーションのサイズによっては、そのような移動にかなりの時間がかかり、結果として財務コストがかかることを意味します。バックエンドがMBaaSプロバイダーの固有の機能に関連付けられている場合、すべてのプロバイダーは異なり、すべてのプロバイダーが同じ機能セットを持たないため、特に恐ろしいでしょう。したがって、「無痛」で動くことができるのはまれです。
