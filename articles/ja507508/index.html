静的コードアナライザーをレガシープロジェクトに埋め込み、コマンドを非アクティブ化する方法











静的コードアナライザーを試すのは簡単です。しかし、それを実装するには、特に大規模な古いプロジェクトの開発では、スキルが必要です。間違ったアプローチでは、アナライザーは作業を追加し、開発を遅らせ、チームをやる気にさせる可能性があります。静的分析を開発プロセスに統合し、CI / CDの一部として使用する正しい方法について簡単に説明しましょう。



前書き



最近、「チームを圧倒せずに静的解析を開始する」という刊行物が私の注目を集めました。一方で、これは知っておくと良い記事です。一方、私には思えるように、多くのレガシーコードを含むプロジェクトに静的分析を簡単に導入する方法についての完全な答えは聞こえませんでした。この記事では、技術的負債に耐え、新しいコードでのみ作業できると述べていますが、この技術的負債を後でどうするかについての答えはありません。



PVS-Studioチームは、このトピックについて独自の見解を提供しています。静的コードアナライザーを導入することによる問題が一般的にどのように発生するか、この問題をどのように克服するか、そして技術的な負債を簡単に解消する方法を見てみましょう。



問題



静的アナライザーを実行してどのように動作するかを確認することは、原則として難しいことではありません[ 1 ]。コードの興味深いエラーや、ひどい潜在的な脆弱性を確認できます。何かを修正することもできますが、多くのプログラマーはあきらめます。



すべての静的アナライザーは誤検知を提供します。これは静的コード分析方法の機能であり、それを使用して何も実行できません。一般的なケースでは、これは解決できない問題であり、ライスの定理によって確認されています[ 2 ]。機械学習アルゴリズムは役に立ちません[ 3 ]。誰かが1つまたは別のコードが間違っているかどうかを常に言うことができない場合は、プログラムからこれを期待しないでください:)。

この場合、10〜15％程度の低いレベルの誤検知が予想されます[ 5 ]。言い換えると、10個中9個のアナライザー警告は、コードの実際の問題、または少なくとも「強いにおいコード」を示します。そのようなシナリオは非常に快適であり、アナライザーはプログラマーの真の味方であることを認めなければなりません。













実際には、大規模なプロジェクトでは、最初の状況はまったく異なります。アナライザーは、レガシーコードに対して数百または数千の警告を生成します。ケース内のこれらの警告のどれが何で何がそうでないかをすばやく理解することは不可能です。この場合の主な作業は数日または数週間停止するため、座ってこれらすべての警告に対処し始めるのは不合理です。通常、チームはそのようなシナリオに対応できません。変更の履歴を台無しにする膨大な数の差分も存在します。コード内の非常に多くのフラグメントをすばやく一括編集すると、必然的に新しいタイプミスやエラーが発生します。



そして最も重要なこととして、警告との戦いにおけるそのような偉業はほとんど意味がありません。プロジェクトが長年にわたってうまく機能して以来、その中の重大なエラーのほとんどはすでに修正されていることに同意します。はい、これらの修正には非常に費用がかかりました。デバッグしたり、バグに関するユーザーからの否定的なフィードバックを得たりする必要がありました。静的アナライザーは、コードを作成する段階でも、これらのエラーの多くを迅速かつ安価に修正するのに役立ちます。しかし現在のところ、何らかの方法でこれらのエラーは修正されており、アナライザーは主に古いコードの重要でないエラーを検出します。このコードは使用されない可能性があり、ごくまれに使用されます。また、コード内のエラーにより、重大な結果が生じることはありません。おそらく、ボタンの影が適切な色ではないかもしれませんが、これは製品を使用することに誰も気になりません。



もちろん、重大でないエラーであっても、エラーのままです。また、実際の脆弱性がエラーの背後に隠れることもあります。ただし、すべてをあきらめて、数日/数週間は十分に現れない欠陥に対処することは、疑わしい考えのように思えます。



プログラマーは、古い作業コードでこれらすべての警告を見て、見て、見て...そして彼らは考えています。静的分析なしで実行できるのです。新しい便利な機能を作成してみましょう。



彼らのやり方では、彼らは正しい。彼らは、初心者のために、これらの警告すべてを何らかの方法で取り除く必要があると信じています。そうして初めて、コードアナライザーを定期的に使用できるようになります。そうでなければ、新しい警告は単に古い警告に沈み込み、誰もそれらに注意を払うことはありません。



これは、コンパイラの警告と同じです。コンパイラの警告の数を0に保つことをお勧めするのは理由がないわけではありません。1000の警告がある場合、警告が1001になると誰もそれに注意を払わず、この最新の警告をどこで探すべきか明確ではありません。













このストーリーで最悪のことは、現時点でトップの誰かが静的コード分析を使用するように強制する場合です。彼らの観点からは、干渉するだけの官僚的な複雑さが加わるため、これはチームをやる気にさせるだけです。誰もアナライザーのレポートを見ることはなく、すべての使用は「紙の上」でのみ行われます。それら。正式には、分析はDevOpsプロセスに組み込まれていますが、実際には誰もこの恩恵を受けていません。カンファレンスの来場者からスタンドでコミュニケーションをとる際に詳しい話を聞きました。このような経験は、プログラマが静的分析ツールに連絡することを常にではないにせよ永久に阻止できます。



技術的債務の履行と解消



実際、大規模な古いプロジェクトでも静的分析を導入するのに複雑で怖いものはありません。



CI / CD



さらに、アナライザーは、継続的な開発プロセスの一部となることができます。たとえば、PVS-Studio配布キットには、必要な形式でレポートを簡単に表示したり、コードの問題のあるセクションを記述した開発者に通知したりするためのユーティリティがあります。CI / CDシステムの下からPVS-Studioを起動することに関心がある方は、ドキュメントの対応するセクションと一連の記事をよく理解しておくことをお勧めします。

しかし、コード分析ツールの実装の最初の段階での多数の誤検知の問題に戻ります。



既存の技術的負債を修正し、新しいアラートを処理する



最新の商用スタティックアナライザーでは、新しいコードまたは変更されたコードに表示される新しい警告のみを調査できます。このメカニズムの実装は異なりますが、本質は同じです。 PVS-Studioスタティックアナライザでは、この機能は次のように実装されています。



静的分析の使用を迅速に開始するために、PVS-Studioユーザーにマスアラート抑制メカニズム[ 6]。一般的な考え方は次のとおりです。ユーザーはアナライザーを起動し、多くの警告を受けました。長年にわたって開発されてきたプロジェクトが生きていて、発展し、お金を稼いでいるので、おそらく、重大な欠陥を示す警告がレポートにあまりありません。言い換えれば、重大なバグは、何らかの方法で、または顧客からのフィードバックのおかげで、すでに何らかの方法で修正されています。したがって、アナライザーが現在検出するすべてのものは、技術的な負債と見なすことができます。これは、すぐに排除しようとすることは現実的ではありません。



これらの警告を無関係と見なすようにPVS-Studioに指示することができ（後で技術的な負債を延期する）、警告は表示されなくなります。アナライザーは特別なファイルを作成し、これまでのところ興味のないエラーに関する情報を保管します。そして、PVS-Studioは、新規または変更されたコードに対してのみ警告を発行します。さらに、これらすべてがスマートに実装されています。たとえば、ソースファイルの先頭に空の行が追加された場合、アナライザは実際には何も変更されていないことを理解し、何も表示されません。このマークアップファイルは、バージョン管理システムに埋め込むことができます。ファイルは大きいですが、敷設しても意味がない場合が多いので、怖くはありません。



これで、すべてのプログラマーに、新規または変更されたコードのみに関連する警告が表示されます。したがって、アナライザーは翌日と呼ばれるものを使用し始めることができます。そして、後で技術的負債に戻って、エラーを徐々に修正し、アナライザーを構成することができます。



したがって、大規模な古いプロジェクトでのアナライザーの実装に関する最初の問題は解決されます。では、技術的負債をどうするかを考えてみましょう。



バグ修正とリファクタリング



最も単純で最も自然なのは、大幅に抑制されたアナライザーの警告を解析し、徐々に対処するのに少し時間をかけることです。どこかでコードのエラーを修正する必要があり、どこかでコードに問題がないことをアナライザーに伝えるためにリファクタリングを実行する必要があります。簡単な例： 



if (a = b)


ほとんどのC ++コンパイラーとアナライザーは、そのようなコードを信頼しています。なぜなら、彼らは本当に（a == b）と書きたかった可能性が高いからです。しかし暗黙の合意があり、追加の括弧がある場合、プログラマが意識的にそのようなコードを書いたと信じられ、誓う必要はないことがドキュメントに通常記載されています。たとえば、V559診断に関するPVS-Studioのドキュメント（CWE-481）には、次の行が正しく安全であると見なされることが明記されています。 



if ((a = b))


もう一つの例。このC ++コードの改行は忘れられていますか？ 



case A:
  foo();
case B:
  bar();
  break;


PVS-Studioアナライザーは、ここで警告V796（CWE-484）を生成します。おそらくこれは間違いではなく、[[fallthrough]]属性を追加するか、たとえば__attribute __（（fallthrough））を追加して、アナライザーにヒントを与える必要があります。 



case A:
  foo();
  [[fallthrough]];
case B:
  bar();
  break;


このようなコード変更はエラーを修正しないと言えます。はい、そうですが、2つの便利な手順があります。まず、アナライザーレポートは誤検知を排除します。第二に、コードはそのメンテナンスに関係する人々にとってより理解しやすくなります。そしてそれは非常に重要です！これだけのために、コードをより理解しやすく、保守しやすくするために、マイナーリファクタリングを実行する価値はすでにあります。アナライザーは「ブレーク」が必要かどうかを理解していないため、他のプログラマーにとっても理解が困難になります。



エラー訂正とリファクタリングに加えて、誤った誤ったアナライザー警告を正確に抑制することができます。一部の無関係な診断は無効にすることができます。たとえば、float / double値を無意味に比較することに関するV550警告が誰かに見つかります。そして誰かが彼らに重要で研究に値するものを紹介します[7 ]。関連する警告と関連しない警告を決定するのは開発チームの責任です。



誤ったアラートを抑制する方法は他にもあります。たとえば、マクロマークアップについてはすでに言及しています。このすべては、ドキュメントでより詳細に説明されています。最も重要なことは、誤検知を処理するために徐々に体系的にアプローチする場合、何も問題がないことを理解することです。興味のない警告の大部分は構成後に消え、本当に注意深く調査する必要のある場所とコードのいくつかの変更があります。



また、問題が発生した場合は、クライアントがPVS-Studioをセットアップするのを常に支援します。さらに、私たち自身が誤った警告を排除し、エラーを修正したケースがありました[ 8]。念のために、協力を拡大するためのそのようなオプションも可能であることに言及することにしました:)。



ラチェット方式



静的アナライザーの警告を排除して、コードの品質を徐々に改善する別の興味深いアプローチがあります。要するに、警告の数は減少するだけです。















静的アナライザーが生成する警告の数は固定されています。品質ゲートは、操作数を増加させないようなコードのみを配置できるように構成されています。その結果、アナライザーの設定やエラー修正により、操作回数が徐々に減るプロセスが開始されます。



新しいコードの警告を排除するのではなく、古いサードパーティのコードを改善することで、少し騙し、品質ゲートを通過することを決定した場合でも、これは恐ろしいことではありません。とにかく、ラチェットは一方向に回転し、徐々に欠陥の数が減っていきます。人が自分の新しい欠陥を編集したくない場合でも、隣接するコードで何かを改善する必要があります。ある時点で、警告の数を減らす簡単な方法は終わり、実際のエラーが修正されるときが来ます。



この方法論は、Ivan Ponomarevによる非常に興味深い記事で詳細に説明されています " プロセスに静的分析を実装し、それに関するバグを探しません「これにより、コードの品質を向上させることに関心のあるすべての人を知ることをお勧めします。
