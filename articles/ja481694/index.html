PostgreSQLのパーティション化パス



テーブルのサイズの制御をやめると、データを維持して利用できるようにするのは簡単な作業ではなくなります。本番環境ですでにこのような問題に遭遇しました。毎日より多くのデータがあり、テーブルがメモリに収まらず、サーバーが長時間応答しますが、解決策が見つかりました。

こんにちは、ハブル！私の名前はアルマズです。次に、パーティション分割の実装に役立つ方法を共有したいと思います。

PostgreSqlでのパーティション分割


パーティション分割（または、パーティションと呼ばれます）は、1つの大きな論理テーブルをいくつかの小さな物理セクションに分割するプロセスです。これが、データの管理に役立つものです。

例：「sales」テーブルがあり、1か月の間隔で分割されています。これらのセクションは、地域ごとにさらに小さなサブセクションに分割できます。


パーティション分割テーブルの「販売」のスキーム

このアプローチの短所：

-データベースの構造は複雑です。データベース定義の各セクションはテーブルですが、1つの論理エンティティの一部です。
-既存のテーブルをパーティションテーブルに変換したり、その逆を行うことはできません。 -Postgres
11では完全なサポートはありません。

プラス：

+パフォーマンス。場合によっては、テーブル全体を調べなくても、限られたセクションのセットで作業できます。大きなテーブルのインデックス検索でも遅くなります。データの可用性が向上しています。
+ ATTACH / DETACHコマンドでデータを一括アップロードおよび削除します。これにより、VACUUMの形でオーバーヘッドから解放されます。これにより、データベースをより効率的に保守できます。
+セクションにTABLESPACEを指定する機能。これにより、データを他のセクションに転送する機会が得られますが、同じインスタンス内で作業し、メインディレクトリのメタデータにはセクションに関する情報が含まれます（シャーディングと混同しないでください）。

PostgreSqlでパーティショニングを実装する2つの方法：



1.テーブルの継承（INHERITS）テーブルを
作成するとき、「別の（親）テーブルから継承する」と言います。同時に、表にデータ管理の制限を追加します。これにより、データ分割のロジックがサポートされますが、これらは論理的に異なるテーブルです。

ここで、テーブルの継承を介してパーティションを実装するPostgres Professional pg_pathmanによって開発された拡張に注意する必要があります。

 CREATE TABLE orders_y2010 (
      CHECK (log_date >= DATE '2010-01-01)
    ) INHERITS (orders);

2.宣言的アプローチ（PARTITION）

テーブルは宣言的に分割されたものとして定義されます。このソリューションはPostgreSqlのバージョン10で登場しました。

CREATE TABLE orders (log_date date not null, …) 
PARTITION BY RANGE(log_date);


私は宣言的アプローチを選択しました。これは大きな利点をもたらします-ネイティブ、より多くの機能がカーネルによってサポートされます。この方向でのPostgreSQLの開発を検討してください：


ソース

しかしPostgreSqlは進化し続けており、バージョン12ではパーティション分割テーブルへのリンクがサポートされています。これは大きな進歩です。

私のやり方


上記の場合、スクリプトはPL / pgSQLで作成され、既存のテーブルに基づいてパーティションテーブルを作成し、新しいテーブルへのすべてのリンクを「スロー」します。したがって、既存のテーブルに基づいてパーティション化されたテーブルを取得し、通常のテーブルと同様に引き続き使用します。
スクリプトは追加の依存関係を必要とせず、それ自体が作成する別の回路で実行されます。また、やり直しと元に戻す操作を記録します。このスクリプトは2つの主な問題を解決します。パーティション化されたテーブルを作成し、トリガートリガーを介してそれに外部リンクを実装します。

スクリプト要件：PostgreSql v.:11以降。

それでは、スクリプトを詳しく見ていきましょう。インターフェースは非常にシンプルです
。すべての作業を行う2つの手順があります。

1.主な課題-この段階ではメインテーブルを変更しませんが、セクション作成に必要なものはすべて別のスキームで作成されます。 

 call partition_run();


2.メインの作業中に計画された据え置きタスクを呼び出します。 

 call partition_run_jobs();


作業は複数のスレッドで起動できます。スレッドの最適な数は、パーティション化されたテーブルの数に近いです。

スクリプトの入力パラメーター（_ptレコード）



内部からのスクリプト、主なアクション：

-パーティションテーブルを作成します。
 perform _partition_create_parent_table(_pt);

-セクションを作成する 
 perform _partition_create_child_tables(_pt);

-セクションのデータをコピーします
 perform _partition_copy_data(_pt);

-制限を追加する（ジョブ） 
 perform _partition_add_constraints(_pt);

-外部テーブルへのリンクを復元する 
 perform _partition_restore_referrences(_pt);

-トリガーの復元
 perform _partition_restore_triggers(_pt);

-イベントトリガーを作成する 
 perform _partition_def_tr_on_delete(_pt);

-インデックスの作成（ジョブ） 
 perform _partition_create_index(_pt);

-ビュー、セクションリンク（ジョブ）を置き換える 
 perform _partition_replace_view(_pt);


スクリプトの実行時間は多くの要因に依存しますが、主なものは、ターゲットテーブルのサイズ、リレーションシップの数、インデックス、サーバーの特性です。私の場合、300Gbのテーブルは1時間未満で分割されました。


結果


何を手に入れましたか？クエリプランを見てみましょう。

 EXPLAIN ANALYZE 
select * from “sales” where dt BETWEEN '01.01.2019'::date and '14.01.2019'::date




通常のテーブルへのクエリと比較して、パーティションテーブルから結果をより速く取得し、サーバーのリソースをより少なく使用しました。

この例では、通常のテーブルとパーティションテーブルが同じベースにあり、約200Mのレコードがあります。これは、アプリケーションコードを書き直さずにアクセラレーションを得たので、良い結果です。他のインデックスのクエリも適切に機能しますが、注意してください。セクションを特定できるときはいつでも、結果が数倍速くなります。PostgreSqlはリクエストの計画段階で余分なセクションを破棄できます（enable_partition_pruningをonに設定）。

合計


多くの関係があるテーブルにパーティションを実装し、データベースの整合性を確保することができました。スクリプトは特定のデータ構造に依存せず、再利用できます。

PostgreSQLは、世界で最も先進的なオープンソースのリレーショナルデータベースです。
