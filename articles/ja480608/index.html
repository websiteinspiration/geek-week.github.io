RustがC ++をベンチマークゲームの結果で上回る

今月の初めに小さな勝利がありました。プログラミング言語のパフォーマンスを比較する人気のベンチマークの最新の結果によると、RustはCとC ++の間に2位にランク付けされています。



パフォーマンスを比較するために、10の異なるプログラムの実行時間の幾何平均が使用されます。最近の結果によると、Rustは10タスク中6タスクでC ++を上回っていました。

カットの下でいくつかの詳細と将来の展望に関する意見。

結果を正しく解釈するにはどうすればよいですか？


ベンチマークゲームは、割り当てられたタスクを解決するプログラムのコミュニティによって記述されたパフォーマンス測定に基づいてプログラミング言語が比較される人気のプロジェクトです。原則として、各タスクには、同じ言語で少なくともいくつかの十分に最適化された実装があります。全体的な結果を要約するために、最良の結果を示す各言語の実装が1つだけ選択されます。

一般的なプログラミング言語の場合、最善の解決策は言語自体のパフォーマンスの限界に近いことを期待する必要があります。ここ数か月の間に、C、C ++およびRustは非常に類似した結果を示しました。したがって、十分に最適化されたRustプログラムは、平均して、十分に最適化されたCまたはC ++プログラムよりもパフォーマンスが劣るべきではないと結論付けることができます。

しかし、ベンチマークは何も意味しません！


はいといいえ。

もちろん、特定のタスクの異なる実装間のパフォーマンスの違いは、多くの要因に依存し、確かにタスクごとに大幅に異なります。

たとえば、Rustは、正規表現を使用する必要があるソリューションのタスクについて、最悪の比較結果を表示します。これらの結果から、Rust（regex crate）での正規表現の実装は、問題を解決するときにPCRE2よりも遅いと結論付けることができます。一方、Rustは、積極的なSIMD操作を必要とするタスクでは、C ++よりも著しく優れています。

行われるべき一般的な結論：RustとC ++のどちらを選択するかについては、言語パフォーマンスがC ++を優先する理由にはなりません。少なくとも追加のテストなし。

現在の状況は変わりますか？


かなり。C、C ++およびRustは非常に近い結果を示しています。現在、その差はごくわずかなので、測定誤差は全体的な順位の最初の場所の分布に大きく影響します。来月RustとC ++が再び場所を変えるか、Cが最初にRustまたはC ++に取って代わる可能性があります。

Rustのパフォーマンスはさらに向上するでしょうか？


はい。 Rustコンパイラーは比較的新しいプロジェクトであり、かなりの数の可能な最適化がまだ完全に実装されていません。主な障害はLLVMです。これは、ユニバーサルトランスレータですが、正式にはC / C ++用に最適化されており、他のプログラミング言語から取得したバイトコード（IR）にはあまり適していません。

Rustは、CやC ++よりも厳密なプログラミング言語です。これは、典型的なプログラムの場合、Rustはトランスレータ（LLVM）にはるかに多くの保証を与えることができ、トランスレータはそれらを使用してより積極的な最適化を実行できることを意味します。

したがって、Rustが今後数年間でそのパフォーマンスを大幅に改善できることが期待されます。同時に、C / C ++コンパイラーは数十年にわたって開発されており、この間に機能の限界に達しています。それらのさらなる最適化は、コンパイラーの実装ではなく、主に言語機能によって制限されます。

ほんの数年前には、新しいプログラミング言語と（神は私を許してください）機能的なアプローチさえ使用して、C / C ++実装よりもパフォーマンスが優れたより効率の高いコードを書くことができると想像する人はほとんどいませんでした。今後数年で、Rustはその例で反対を証明し、さまざまなパフォーマンステストで自信のある最初の場所を占めるようになると思います。あなたも？
