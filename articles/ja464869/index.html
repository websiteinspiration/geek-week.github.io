前回の記事では、DublGISの内部製品と外部製品の開発の簡単な歴史を説明しました。今日は、製品の1つであるデータエクスポートの開発の詳細について説明します。プロジェクトのアーキテクチャと、プロジェクトを徐々に開発し、時間の経過とともに変化する要件に適応させるための個々の技術ソリューションについて説明します。

最後の記事の簡単な要約


大量の地図データ、組織のディレクトリ、広告、ユーザーフィードバック、レビュー、写真、さまざまな分析を収集する内部製品がいくつかあります。これらの製品は、データバスまたはRest Apiを介して相互に通信します。また、このすべてのデータをヒープに収集し、処理して、目的の形式に分解し、最終製品に配信するための既製の「バンドル」をパックして形成する、別のエクスポートプロセスがあります。配信は、PCおよびモバイルバージョンの更新サーバー、または実際には2GISのオンラインバージョンのオンラインバックエンドで行われます。



初期データ


ですから、入り口には次のものがあります。

  • 同じデータの複数のソース。
  • さまざまな配信方法（Firebird、バス、FTP、RestAPI）;
  • 同じオブジェクトの異なる構造;
  • データ構造の一定の変化。
  • さまざまな形式（データベースの生データ、XML、JSON）。

消費者の視点から：

  • 繰り返しになりますが、さまざまな形式（製品のさまざまなバージョンのデータ形式、販売用の別の形式）。
  • 一定のフォーマット変更;
  • 集計データ（さまざまなオブジェクトを1つに結合し、すべての支店から会社データを収集し、写真、レビュー、最寄りのストップなどへのリンクでそれらを補足する必要があります）;
  • 複雑な前処理と後処理（他のデータに基づいて一部のデータを更新する、データを変換する、欠落データを生成する、たとえば、建物にミニロゴを配置する、誤ったデータを削除または修正する）;
  • データの整合性と有効性の要件。
  • すべてのデータが必要です。

ここでは、最後の段落に焦点を当てる価値があります。ご存知のように、2GISの主な機能はオフライン作業です。つまり、PCバージョンとモバイルバージョンで表示されるデータのほとんどはデバイスにあります。しかし、これは巨大な配列です：数十万の地理オブジェクト（海、森、川、道路、建物、入口、ポーチ、署名、間取り図、3Dモデル）、数十万の企業とその連絡先、勤務時間、追加属性を持つ支店平均請求額やWi-Fiの可用性など。そしてもちろん、広告文や写真も。

そして、すべてが絶えず変化、追加、削除されています。

そして、この無限の変化の流れに溺れないようにするために、エクスポートアーキテクチャを開発するとき、いくつかの主要な領域に焦点を当てる必要がありました。

  • データソース;
  • 配送方法;
  • 処理アルゴリズム;
  • 消費者のためのデータフォーマット。

さまざまなソースとデータ形式から抽象化します


ソースが異なると、次のような問題が発生します。

  • 同じデータを異なる形式で提供します。
  • 単一のドメインオブジェクトに減らす必要があるエンティティまたは属性の異なるセットを持っている。

これはかなり標準的な問題であり、標準で解決されています。データを受信するためのインターフェースを作成するだけで、特定の実装が必要なところにすでに実装されており、必要な形式でデータを取得できます。



インターフェースの例：

public interface ISource : IDisposable
{
    ISourceReader GetDeletedRows();

    ISourceReader GetInsertedOrUpdatedRows();

    byte[] GetDataVersion();
}

public interface ISourceReader : IDisposable
{
    bool Read();

    object this[string columnName] { get; }
}

取得企業の実装例：

internal class FirmSetSource : ISource
   {
       public ISourceReader GetDeletedRows()
       {
           if(_lastDataVersion == null) return null;
           var query = DataContext.ExecuteObject<EsbFirmDeleted>(_lastDataVersion);
           return new DeletedIdsSourceReader<long>(
               query.Select(x => x.Id).GetEnumerator());
       }

       public ISourceReader GetInsertedOrUpdatedRows()
       {
           return new EnumeratorSourceReader(typeof(FirmSet), 
               GetNewOrChangedRows().GetEnumerator());
       }

       public virtual byte[] GetDataVersion()
       {
           return DataContext.ExecuteObject<EsbFirm>().Max(x => x.RowVersion);
       }       
   }

この抽象化により、ドメインモデルの違いに関する問題を部分的に解決できますが、完全には解決できません。重要な制限は、データを段階的に受信する必要があることです。つまり、更新のみを受信し、毎回すべてを吸い込む必要はありません。この場合、いくつかの集計を収集するためにデータ間の関係を追跡するのはかなり不便です。そして、すべてをエラーなしで行うことは比較的困難です。したがって、この段階ではソースからデータを1対1で抽出することを決定し、ドメインモデルの問題を別のレベルで解決します。

ドメインモデル


データセットの変更とデータソースの構造に依存しないようにするために、エクスポートデータベースは比較的安定したテーブルのリストを使用して作成され、最終的にドメインに分類されました。ソース1にエンティティA（次の図のデータオブジェクト）の一部の属性がない場合、それらはデフォルト値を受け取るか、オプションでした。また、エンティティBが何らかのソースデータの集合体または異なるソースでさえある場合は、各パーツを個別に取得し、次のステップで全体を組み立てることができます。



データ配信の方法から抽象化


実際、独自のデータベースをエクスポートし、ISourceReaderインターフェースを使用することで、この問題はすでに解決されています。しかし、1つの未解決のポイントがあります。わずかに異なるデータ取得モデルです。 1つのケースでは、現時点でのスナップショットをプルして取得します。もう1つは、バス上の変更のデルタ、3つ目は、リクエスト時の現在のステータスですが、前のリクエストの瞬間から削除されたオブジェクトに関する情報を含みます。

この動物園を統一するために、すべてのソースからのすべてのデータをマージするデータベースをもう1つ追加します。

あなたはそのような写真を得ます。



その結果、すべての都市のあらゆるチャネルから中央データベースにすべてのデータを読み取りました。ほとんどの場合、配信は増分的です。つまり、変更のみが行われます。古いDGPPは生きていましたが、代替の情報源でした。あるDBMSから別のDBMSにデータをポンプすることはできませんでした。

さらに、ISourceを介したエクスポートにより、DGPPまたはEMDBから都市データがその安定した同期データベースにプルされ、ドメインモデルに変換されました。

次に、それらを処理し、消費者の形式でアップロードするだけです。

データ準備アルゴリズムからの抽象化


そして、ここでもう一つの困難が生じます。まず、さまざまな消費者がそれぞれの形式でデータを求めています。さらに、彼らは異なるデータセットを望んでいます。また、付属資料では、オフラインデータは可能な限りコンパクトで構造化されているため、すばやく読み取ることができます。その結果、最終製品のチームによって開発されたバイナリ形式が得られます。そして、これらはまったく異なるテクノロジースタックで作業している人たちです。 .NETバックエンドと時々Javaを開発するためによく知られ、愛されています。彼らは主にC ++とpythonを持っています。

一般的に、テクノロジーの動物園。

急速な発展の黎明期、DGPPしかなかった（最後の記事を参照））およびPCバージョン2GISでは、最終データの形式はバイナリであり、C ++で記述され、COMオブジェクトにラップされた特別なライブラリによって準備されました。異種コードの統合ではないように思われます。参照を接続すると、.NETインターフェイスが生成されます-そして運転しました。そして、初めて。

しかし、いつものように、いくつかの問題が発生しました。

  1. 私たちのデータは急速に増大し始めました。新しいタイプのデータ、モスクワのような新しい大都市が登場しました。
  2. X64ビットOSが活発に普及し始めました。
  3. COMの問題はどういうわけかデバッグする必要がありました。

ポイントを見ていきましょう。

当社の製品で完全に必要とされるデータの増加により、その処理が大量のRAMを消費し始めたという事実につながりました。そして、COMライブラリを.NETプロセスx86に接続すると、x86プロセス、つまりアドレス空間が増加した最大3Gbの工作員が自動的に取得されます。チームはx64リソースのライブラリをサポートしていませんでしたが、ライブラリ自体はメモリの代わりにディスクを使用する機能を持っていたため、問題は多少緩和されました。

しかし、デバッグは依然として非常に困難でした。エクスポートを開始し、それがデータを準備するのを待って、これらのデータをライブラリに追加し始める必要がありました。そしてエラーが表示された後、何が問題だったかをログから理解し、プロセスを再度繰り返す必要があります。良くない、とても悪い。

解決策は通常通り表面上にあります。すべての外部コードを個別のプロセスに取り込み、単純なバイナリ形式またはテキスト形式の中間ファイルを介して通信を確立するだけで十分です。



その結果、元の.NETプロセスは完全に任意のCPUになりました。サードパーティコードのメモリリークや重大なエラーが影響を与えることはもうありません。エクスポートはデータを準備し、それを中間ファイルにアップロードし、ユーティリティに提供し、結果をファイルの形式でも受け取りました。サードパーティチームの担当者は、言語（C ++またはPython）でアルゴリズムを作成し、エクスポートを開始することなく、マシンでエラーが発生した場合に実際のデータでそれらをデバッグできました。

私たちがしなければならなかったのは、ランタイムで提供されたユーティリティインターフェイスで合意を形成し、必要なパラメーターの合意リストを持ち、情報メッセージとエラーを必要な形式でstdoutに表示することだけでした。
