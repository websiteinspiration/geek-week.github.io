PVS-Studio静的コードアナライザーを開発プロセスに導入する理由


PVS-Studioは、C、C ++、C＃、またはJavaで記述されたプログラムのソースコードのエラーと潜在的な脆弱性を見つけるためのツールです。PVS-Studioは、静的アプリケーションセキュリティテスト（SAST）ツールのクラスに属しています。アナライザーは、継続的インテグレーション（CI）の実践に焦点を当てており、修正にほとんどコストがかからない場合に、最も早い段階でエラーを特定できます。

静的コード分析


開発の過程にあるソフトウェアプロジェクトはますます増えています。例：

  • Linuxカーネル1.0.0：176,000行のコード
  • Linuxカーネル5.0：26,000,000行のコード
  • Photoshop 1.0：128,000行のコード
  • Photoshop CS 6：10,000,000行のコード

プロジェクトのサイズが大きくなると、その複雑さは線形よりも速く大きくなります。このため、コードベースのサイズが大きくなると、エラーの密度も大きくなります。複雑さの増大を補う1つの方法は、静的コード分析ツールを使用することです。

静的アナライザーは、プログラマーのアシスタントプログラムであり、コードの予備レビューを実行し、エラーが含まれている可能性が高いコードのフラグメントを示します。これにより、多くのエラーは、それらを修正するコストが最小である早い段階で修正できます。

静的分析は置き換えられませんが、コードレビュー、単体テスト、動的コード分析、回帰テスト、手動テストなど、他の欠陥検出方法を補完します。

たとえば、コードレビューについて話している場合、プログラムが単純なエラーを検出した方がはるかに優れています。次に、コードをレビューするプログラマーは、アルゴリズムのより有用な高レベルのチェックに集中でき、比較関数に強制的に読み込まれることはありません。さらに、実践が示すように、多くのエラーは「目で」探すのが非常に難しく、気づかれずにコードレビューフェーズを通過する可能性が非常に高くなります。

静的分析ツールのもう1つの利点は、エラーパターンの豊富なデータベースが存在することです。彼らはプログラマーが気づいていないかもしれない問題の存在を見つけることができます。いくつかの例：V698、V718、V1023。

反対意見の返信


時々、プログラマーは静的コード分析を開発プロセスに導入するという考えを否定的に認識し、静的分析方法論全体または具体的にはPVS-Studioツールを批判します。議論が始まると、批判は実証されておらず、確立された開発プロセスで何かを変更したくないという気持ちから生じていることがわかります。何も変更しないことの典型的な議論と、それらの弱点が何であるかを検討してください。

「静的分析は作業時間の一部になります」


「静的分析は作業時間の一部を占める」という文言は、文脈とは切り離して真実です。新規または変更されたコードに対して発行された静的アナライザー警告を定期的に確認するには、本当に時間がかかります。ただし、考えは継続する必要があります。ただし、これに費やす時間は、他の方法でエラーを識別するコストよりもはるかに少なくなります。

静的コードアナライザーの警告に時間を費やす必要があるという意見はどこから来たのですか？

コード分​​析方法にまだ慣れていないプログラマは、テストの実行と通常の使用を混乱させます。最初の起動時に、アナライザーは警告の膨大なリストを表示しますが、その多くは誤っています。その理由は、アナライザーがまだ構成されていないためです。調整されたアナライザーは、定期的な起動時に少数の誤検知を生成します。言い換えれば、通常の使用では、ほとんどの警告は実際の欠陥または臭気コードを検出します。この設定を行うことだけが重要です。

このトピックについては、「異論への取り組み：静的分析が稼働時間の一部になる」という記事で詳しく説明しています。

「静的アナライザーは非常にノイズが多い（多くの誤検知を生成する）」


上記のように、この結論は未構成のコードアナライザーを使用して行われます。 PVS-Studioをセットアップすると、誤検知の割合が10〜20％になると予想できます。つまり、5つの警告が発行された場合、4つは実際のエラーまたは将来エラーを引き起こす可能性が高いコードを示します。そのような設定の例は、記事「EFLコアライブラリの例を使用したPVS-Studioアナライザーの仕様、10〜15％の誤検知」で確認できます。

アナライザーのビューを歪めるもう1つの理由は、目的を理解せずに、できるだけ多くの警告を含めたいという要望です。たとえば、クラシックWindowsアプリケーションの組み込みシステムに焦点を合わせたMISRAルールセットを有効にすると、アナライザーは数十万の警告を生成します。それらには実用的な意味はありません。特に不要な診断は、ツールとの知り合いの段階で干渉し、その診断機能について誤解を招きます。「CおよびC ++コード用のPVS-Studioアナライザーが提供する興味深い警告をすばやく確認する方法」という記事は、これを回避するのに役立ちます。

「開発プロセスに静的分析を導入することは非常に難しく、長く、費用がかかります」

それほど怖くないです。大規模な古いプロジェクトでも静的分析を入念に実装できるようにする方法は少なくとも3つあります。

最初のアプローチ。 Ivan Ponomarevが彼の講演「Continuous Static Code Analysis」でよく語る「ラチェット法」。

2番目のアプローチ。静的解析の使用を迅速に開始するために、PVS-Studioのお客様に「マークアップデータベース」を使用するように提案します「一般的な考え方は次のとおりです。ユーザーはアナライザーを起動し、多くの警告を受けました。長年にわたって開発されたプロジェクトが存続し、開発されて資金をもたらすと、レポートには重大な欠陥を示す警告があまり表示されなくなります。つまり、クリティカルいずれにせよ、バグはより高価な方法で、またはクライアントからのフィードバックのおかげですでに修正されているため、アナライザーが見つけたものはすべて技術的負債と見なすことができ、すぐに修正するのは現実的ではありません。

PVS-Studioに、これらの警告を無関係と見なして（後で技術的な負債を延期する）、再度表示しないように指示できます。アナライザーは特別なファイルを作成し、そこにこれまでに興味のないエラーに関する情報を格納します。そして、PVS-Studioは、新規または変更されたコードに対してのみ警告を発行します。さらに、これらすべてが巧妙に実装されています。たとえば、特定の.cppファイルの先頭に空の行が追加された場合、アナライザは実際には何も変更されていないことを理解し、何も表示されません。このマークアップファイルは、バージョン管理システムに埋め込むことができます。ファイルは大きいですが、敷設しても意味がない場合が多いので、怖くはありません。

これで、すべてのプログラマーに、新規または変更されたコードのみに関連する警告が表示されます。したがって、アナライザーは翌日と呼ばれるものを使用し始めることができます。そして、後で技術的負債に戻って、エラーを徐々に修正し、アナライザーを構成することが可能になります。

3番目のアプローチ。あなたは私たちと契約を結び、静的分析の設定と統合の作業を私たちのチームに委任することができます。このプラクティスの例：「PVS-Studioチームがアンリアルエンジンのコードをどのように改善したか」

第二に、最大の効果は、さまざまなエラー検索技術を使用することの相乗効果により達成されます。一部のエラーは単体テストでよく検出され、一部は手動テストで検出されます。状況を想像してみてください。プロジェクトには10​​人のプログラマーがいて、多くの単体テストが書かれていますが、テスターはいません。プロジェクトの品質はユーザーに適しておらず、アイデアはテスターとしての仕事を開くことです。しかし、これは「別のプログラマーを雇う方が良い」という口実では行われません。ユニットテストをさらに増やしましょう！同意します、これは間違った決定です。明らかに、品質保証プロセスは一方的なものであり、手動テストの追加ははるかに大きなメリットをもたらします。同じ状況が静的分析の場合です。

「動的分析は静的分析よりも効率的です。」
