データ検証の何が問題になっており、Liskov置換の原則はそれとどのように関係していますか？



「この方法ですべてが私にとって良いのですか？」そして、「もし吹いたらどうなるか」と「念のためチェックした方がいい」のどちらかを選び、猫の下で歓迎します...

修正：前述のとおり lorc そして 0xd34df00d、以下で説明するものは、依存型と呼ばれます。あなたはそれらについてここで読むことができます。さて、以下は私がこれについて考えているソーステキストです。

開発中、多くの場合、アルゴリズムのデータの有効性を検証する必要があります。正式には、これは次のように説明できます。データ構造を取得し、その値を特定の範囲の許容値（ODZ）と照合して、それを渡します。その後、同じデータ構造を同じ検証にかけることができます。構造が変更されていない場合、その有効性を再確認することは明らかに不要なアクションです。

検証は本当に長くなる可能性がありますが、ここでの問題はパフォーマンスだけではありません。余分な責任ははるかに不愉快です。開発者は、構造の妥当性を再度確認する必要があるかどうかわかりません。逆に、不必要な検証に加えて、構造が以前に検証されたと誤って想定して、検証がないことを想定できます。

したがって、機能不全は、実証済みの構造を期待し、値が許容値の特定の範囲外にある構造では正しく機能しないメソッドで許可されます。

そこには明らかではないより深い問題があります。実際、有効なデータ構造は元の構造のサブタイプです。この観点から、有効なオブジェクトのみを受け入れるメソッドの問題は、架空の言語で次のコードと同等です。

class Parent { ... }
class Child : Parent { ... }

...

void processValidObject(Parent parent) {
    if (parent is Child) {
        // process
    } else {
        // error
    }
}

これで問題がより明確になることに同意します。私たちの前には、リスコフの代替原理の標準的な違反があります。それは、例えば、置換の原則に違反する悪い理由を読むにはこちら。

元のデータ構造のサブタイプを作成することにより、無効なオブジェクトを転送する問題を解決できます。たとえば、元の構造に従って有効なサブタイプオブジェクトまたはnullを返すファクトリを介してオブジェクトを作成できます。有効な構造を期待するメソッドのシグネチャを変更して、サブタイプのみを受け入れるようにすると、問題は解消されます。システムが正確に機能するという確信に加えて、コードの平方センチメートルあたりの検証の数は減少します。もう1つの利点は、そのようなアクションにより、データの検証の責任を開発者からコンパイラに移すことです。

Swiftでは、構文レベルで、nullをチェックする問題が解決されています。アイデアは、型をnull許容型と無効型に分離することです。同時に、それは砂糖の形で作られているので、プログラマーは新しいタイプを宣言する必要はありません。変数の型を宣言するとき、ClassNameは変数がゼロ以外であることを保証し、ClassNameを宣言するときは？変数はnullです。同時に、型間に共分散が存在します。つまり、ClassName型のオブジェクトをClassNameを受け入れるメソッドに渡すことができますか？

このアイデアは、ユーザー定義のDLDに拡張できます。タイプに保存されたDLDを含むメタデータをオブジェクトに提供すると、上記の問題が解消されます。このようなツールのサポートを言語で取得するとよいでしょうが、この動作は、継承やファクトリーを使用して、JavaやC＃などの「通常の」OO言語でも実装されています。

データ検証の状況は、OOP内のエンティティが現実の世界からではなく、エンジニアリングのニーズから取得されていることのもう1つの確認です。

UPD：コメントに正しく記載されているように、サブタイプを作成する価値があるのは、信頼性を高め、同一の検証の数を減らす場合のみです。

また、記事には例がありません。入り口でいくつかのファイルパスが表示されます。私たちのシステムは、すべてのファイルで機能する場合と、アクセスできるファイルのみで機能する場合があります。次に、それらを異なるサブシステムに転送します。これらのサブシステムは、アクセス可能なファイルとアクセスできないファイルの両方で機能します。さらに、これらのサブシステムはファイルをさらに転送しますが、ここでもファイルが使用可能かどうかは不明です。したがって、疑わしい場所では、アクセスチェックが表示されたり、逆に忘れられたりする可能性があります。このため、あいまいさとチェックが広範囲に及ぶため、システムはより複雑になります。しかし、これらのチェックはディスクをロードし、一般に重いです。このチェックはブールフィールドにキャッシュできますが、チェックが必要であるという事実から解放されません。チェックの責任を開発者からコンパイラに移すことをお勧めします。
