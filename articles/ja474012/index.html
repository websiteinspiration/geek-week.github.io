Dockerコンテナのセキュリティ

注意 ペレフ。：Dockerのセキュリティテーマは、おそらく今日のITの世界では永遠のテーマの1つです。したがって、さらに説明することなく、関連する推奨事項の次の選択の翻訳を提示します。この問題にすでに興味がある場合は、それらの多くはよく知っています。さらに、問題自体をさらに調査するために、有用なユーティリティのリストといくつかのリソースで選択自体を補足しました。



これがDockerセキュリティガイドです。フィードバックは、さまざまなリソースからの抜粋の集まりであり、すべてが完全にチェックされているわけではないため、歓迎されます。推奨事項は次の3つのカテゴリに分類されます。

  1. Dockerを使用する際にホストオペレーティングシステム内で必要な対策。
  2. アセンブリ構成ファイルとコンテナの作成に関連する手順。
  3. 特定のDocker Enterprise機能と統合できるセキュリティツール。

さまざまなリソースがリーダーシップの基礎になりました。その多くを以下に示します。網羅的とは言えませんが、すべての基本を網羅しています。追加情報はCISテストの説明（リンクはこのマニュアルの最後にあります）とDockerのドキュメントにあります。

Dockerセキュリティベンチマーク


Docker Bench for Securityは、最も一般的なベストプラクティスに準拠しているかどうか、Dockerを自動的にチェックします。スクリプトは優れたヒューリスティックセキュリティテストですが、包括的な分析ツールと見なすべきではありません。

ホストOS


明らかに、ホストシステム自体が保護されていないと、Dockerコンテナーを保護できません。したがって、オペレーティングシステムのセキュリティの分野におけるベストプラクティスに従う必要があります。さらに、以下の推奨事項に加えて脆弱性分析を実施することが賢明でしょう。

監査ルール


でDockerに関連付けられているファイルの監査ルールを作成して使用しますauditctl。たとえば、監査サービスを追加-w /usr/bin/dockerd -k dockerし/etc/audit.rulesて再起動できます。

FIPSモード


FIPSモードを有効にすると、暗号化ツールはFIPS （米国連邦情報処理標準 -約Transl。）で導入されたアルゴリズムに強制的に切り替えられるため、連邦および業界の標準と要件に準拠します。ホストOSがFIPSモードをサポートしている場合は、次のコマンドを実行して有効にすることができます。

sed -i 's/GRUB_CMDLINE_LINUX="/GRUB_CMDLINE_LINUX="fips=1 /g' /etc/default/grub
grub2-mkconfig -o /boot/grub2/grub.cfg && reboot

DockerエンジンでFIPSを有効にする必要もあります。

mkdir -p /etc/systemd/system/docker.service.d 2>&1; echo -e "[Service]\n Environment=\"DOCKER_FIPS=1\"" > /etc/systemd/system/docker.service.d/fips-module.conf; systemctl daemon-reload; systemctl restart docker

詳細については、DockerおよびRed Hatのドキュメントを参照してください。

Dockerシークレット


機密データは秘密として保管する必要があります。docker service createコマンドを使用して、対応するサービスを開始できます。

docker service create --label com.docker.ucp.access.label=/prod --name nginx --publish 443 --secret source=orcabank_prod_mobile.ca.pem.v1,target=ca.pem nginx

詳細については、ドキュメントを参照してください。

Docker構成ファイル


次の設定を構成ファイルに追加できます/etc/docker/daemon.json。

  • "icc":false -不要な情報漏えいを防ぐため、コンテナ間のデータ交換を無効にします。
  • log-level: "info" -デバッグ以外のすべてのログをキャプチャします。
  • {
  "log-driver": "syslog",
  "log-opts": {
    "syslog-address": "udp://1.2.3.4:1111"
  }
}

    -リモートログを接続し、指定したアドレスに転送します。syslogデーモンが実行されている場合にのみ機能します。TCPおよびUDPはオプションとして受け入れられます。特定のコンテナごとに接続することも可能です。これを行うには、Docker'a（--log-opt syslog-address=ADDRESS）の起動時に特別なフラグを設定します。
  • "userns-remap": "Your_User" -特定のユーザーの名前空間を分離することにより、特権の昇格を防ぎます。

輸送セキュリティ


TLSクライアント資格情報にアクセスできるユーザーのみが、Dockerデーモンに接続できるようにする必要があります（リモートアクセスが必要な場合）。

認可プラグイン


どのユーザーにどのコマンドの実行を許可するかを決定し、Docker用の適切な承認プラグインを作成します。次に、Dockerデーモンを実行してプラグインを追加します。

dockerd --authorization-plugin=PLUGIN_ID

承認プラグインの作成について詳しくは、ドキュメントをご覧ください。

デーモンオプション


Dockerデーモンは、一連のデフォルトパラメータで動作します。

  • --live-restore-このオプションは、システムをオフにするか再起動するときのコンテナーのダウンタイムを減らすのに役立ちます。最小限のダウンタイムでパッチを適用したり更新したりすることが容易になります。
  • --userland-proxy=false -ヘアピンNATが利用可能または使用されている場合、ユーザー空間のプロキシは冗長サービスになり、可能な攻撃ベクトルの数が増えるだけです。
  • --no-new-privileges -suidまたはsguidを使用するコンテナによる追加の特権の受け取りを防止します。
  • --seccomp-profile /path/to/profile-独自のseccompプロファイルがある場合は、このフラグを使用して適用できます。SeccompとDockerの詳細については、こちらをご覧ください。

コンテナーとアセンブリファイルの構成


ユーザー作成


コンテナーのユーザーが作成されていることを確認し、そのユーザーの下でコンテナーを実行します（ルートの下でコンテナーを実行しないでください）。

リモートアクセス


デーモンへのリモートアクセスを拒否します。それでも必要な場合は、証明書で保護してください。

ユーザー名スペースを分離


デフォルトではDockerのユーザー名スペースはホスト名スペースと共有されているため、Dockerのユーザー名スペースが分離されていることを確認することは特に重要です。場合によっては、これを使用して特権を昇格したり、コンテナの外に出たりすることもできます。構成ファイルを編集することで、ユーザーの名前空間を分離できます（前述のDocker構成ファイルのセクションを参照）。この問題についての追加の言及は、その重要性によるものです。

健康診断


Healthcheck（ヘルスチェック）は、コンテナーの整合性をチェックするための強力なツールです。手順に従ってDockerfileで構成されますHEALTHCHECK。Healthcheckを使用すると、コンテナが正常に動作していることを確認できます。以下の例では、サーバーが実行中の場合、ヘルスチェックは0で終了し、「クラッシュ」した場合は1で終了します。

HEALTHCHECK CMD curl --fail http://localhost || exit 1

SELinux


SELinuxがホストオペレーティングシステムでサポートされている場合は、SELinuxポリシーを作成またはインポートし、Dockerをデーモンモードで実行し、SELinuxを有効にします。

docker daemon --selinux-enable

この場合、Dockerコンテナはセキュリティ設定を使用して起動できます。次に例を示します。

docker run --interactive --tty --security-opt label=level:TopSecret centos /bin/bash

ネットワークインターフェース


デフォルトでは、Dockerはすべてのネットワークインターフェイスでリッスンします。ほとんどの場合、トラフィックはそのうちの1つだけで予想されるため、このようなアプローチは攻撃のリスクを不当に増大させます。したがって、コンテナーを開始するときに、そのポートをホスト上の特定のインターフェースにバインドできます。

docker run --detach --publish 10.2.3.4:49153:80 nginx

キャッシュされた画像のバージョン


イメージをダウンロードするときは、ローカルキャッシュがリポジトリのコンテンツと一致していることを確認してください。そうしないと、古いバージョンのイメージまたは脆弱性を含むイメージが表示される場合があります。

ネットワークブリッジ


デフォルトでは、ネットワークモデルdocker0はARPスプーフィングやMACフラッディングなどの攻撃に対して脆弱です。この問題を解決するには、ここで説明するように、仕様に従ってネットワークブリッジを作成します。

Dockerソケットの警告


コンテナ内にDockerソケットを投げないでください。それ以外の場合、コンテナはDockerコマンドを実行できるため、ホストオペレーティングシステムと通信して制御できます。こんなことしないで。

Docker Enterpriseの構成


Dockerの信頼


Docker Trustでは、画像の暗号化の整合性を検証するために使用できるキーを生成できます。Docker Trustキーを使用して、Dockerイメージにプライベートキーで署名できます。プライベートキーは、Notary Serverの公開キーによって検証されます。詳細はこちら。Enterprise EngineでDocker Trustを有効にする方法については、ドキュメントのこのセクションで詳しく説明しています。

脆弱性スキャン


Docker Enterpriseには脆弱性スキャナーが組み込まれており、CVEデータベースをダウンロードして、画像のオフラインスキャンの脆弱性を検出することができます。画像を定期的にスキャンすると、画像の安全性が向上します。ユーザーは、発見された脆弱性に関する警告をすぐに受け取ります。これを行う方法の詳細については、ここを参照してください。

注意 ペレフ。：Docker-imagesにはオープンソースの脆弱性スキャナーもあります。記事の最後にある例を参照してください。

LDAPとUCPの統合


Universal Control PlaneはLDAPと統合できます。その結果、不要な重複を回避するために認証システムが簡素化されます。詳しくは、LDAPディレクトリとの統合に関する記事をご覧ください。
