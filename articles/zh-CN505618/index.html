GRASP模板：创建者

哈Ha 我叫弗拉迪斯拉夫·罗丹（Vladislav Rodin）。目前，我是OTUS高负载架构师课程的负责人，并且还教授软件架构课程。

特别是为了开始新的课程“建筑与设计模式”，我写了一些材料，很高兴与大家分享。


介绍


在克雷格·拉曼（Craig Larman）的《应用UML和模式》第三版中描述了GRASP模式是GoF模式的概括，也是OOP原理的直接结果。它们补充了逻辑阶梯中缺少的步骤，使您可以从OOP原理中获取GoF模式。GRASP模板很可能不是设计模式（如GoF的），而是类之间责任分配的基本原则。实践表明，它们并不是很流行，但是，使用完整的GRASP模式集对设计的类进行分析是编写良好代码的前提。

GRASP模板的完整列表包含9个元素：

  • 信息专家
  • 创作者
  • 控制者
  • 低耦合
  • 高凝聚力
  • 多态性
  • 纯加工
  • 间接的
  • 受保护的版本

上次我们讨论了信息专家的原理。现在，我建议考虑一个与之相似的创作者。

创作者


措辞


这种模式解决了与其前身相同的典型问题：需要它们的类必须创建该类的实例。

违例


考虑订单和货物的相同问题。假设我们编写的代码符合Information Expert：

@Setter
@Getter
@AllArgsConstructor
public class Order {
    private List<OrderItem> orderItems;
    private String destinationAddress;
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }
}

@Setter
@Getter
@AllArgsConstructor
public class OrderItem {
    private Good good;
    private int amount;

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

尽管我们正在研究的原理看似琐碎，但在某些客户端代码中，您仍然可以找到以下内容：

public class Client {
    public void doSmth() {
        Good good = new Good("name", 2);
        OrderItem orderItem = new OrderItem(good, amount);
        List<OrderItem> orderItems = new ArrayList<>();
        orderItems.add(orderItem);
        Order order = new Order(orderItems, "abc");
        // client code 
    }
}

如果构建UML类图，则可以发现Client类现在取决于Order类及其所有内部元素：OrderItem和Good。因此，如果没有上述不需要的类，我们将无法重用Client类。实际上，由于Client类创建了所有对象，因此使遵循Information Expert的所有努力的结果无效。

在旧项目中，您经常可以看到一个类如何创建另一个对象，并通过5-6个类将其作为方法中的参数转发给该类，在该类中不使用该对象。这无非是从头开始添加一些依赖关系。

应用实例


让我们调整类之间的责任分配，以使分配不仅满足信息专家的要求，还满足创建者的要求：

@Setter
@Getter
public class Order {
    private List<OrderItem> orderItems = new ArrayList<>();
    private String destinationAddress;
    
    public Order(String destinationAddress) {
        this.destinationAddress = destinationAddress;
    }
    
    public int getPrice() {
        int result = 0;
        
        for(OrderItem orderItem : orderItems) {
            result += orderItem.getPrice();
        }
        
        return result;
    }

    public void addOrderItem(int amount, String name, int price) {
       orderItems.add(new OrderItem(amount, name, price));
   }
}

@Setter
@Getter
public class OrderItem {
    private Good good;
    private int amount;

    public OrderItem(int amount, String name, int price) {
        this.amount = amount;
        this.good = new Good(name, price);
    }

    public int getPrice() {
        return amount * good.getPrice();
    }
}

@Setter
@Getter
@AllArgsConstructor
public class Good {
    private String name;
    private int price;
}

现在，类之间的依赖关系数量将最小化。客户端代码有些简化，可能看起来像这样：

public class Client {
    public void doSmth() {
        Order order = new Order("address");
        order.addOrderItem(amount, name, price);
        // client code 
    }
}

结论


开发人员通常会忘记的创建者可以被视为Information Expert的特例，因为构造函数调用与方法调用相同。通过与Information Expert一起使用，可以使类之间的连接数最少，并可以提高重用性。
