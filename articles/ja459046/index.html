2つのステップで（存在の証明）を独立して実装する方法

みなさん、こんにちは！私はアララトと申します。アルゴリズム戦略のコンテストを開催しているQuantNetで働いています。最近、私の前に重要なタスクが発生しました-ユーザーの日付の不可侵性を保証することです（これは非常に重要です。戦略の有効性を正しく確認するためには、グローバル金融市場のデータをリアルタイムで使用する必要があるためです）。

そこで私はPoE（Proof of Existence）の概念に出会いました。これについてはインターネットで十分に書かれていますが、プラットフォームの詳細により、少し気が悪くなりました。したがって、私はこの記事を書き、PoEのアーキテクチャーと実装における私の経験を共有することにしました。フィンテックの人には特に関係があると思います。

私は記事を3つの主要なブロックに分けました：

  • PoEとは何か、いつ必要になるか
  • 実装アルゴリズム
  • 私の特定のケースの解決策

それでは、存在証明は何ですか？


存在証明（文字通り、存在の証明）は、ドキュメント、ファイル、またはデータが特定の日時に作成されたことを証明するのに役立ちます。最大のアプリケーションは特許登録です。しかし、私が見た限りでは、ほとんどの場合、金融とITが交差する領域に適用されます。

私のアルゴリズム取引の領域の例：次の2週間の株式の正しい予測の約70％を提供するアルゴリズムがあります。予測を他の市場プレーヤーに販売することにしました。ポイントは小さい-予測が正しいこと、および入札結果の前に作成されたものであることを購入者に納得させるため。特定の時間にそれらの実装を保証します。

これはどのように保証できますか？そう、PoEを実装します。

PoE実装アルゴリズム


最初に必要なもの：

  1. 特許を取得するファイルを準備します。（PDFを作成しましたが、他の任意のコンテナーを使用できます）。
  2. 指定されたファイルのハッシュを取得します（私はsha256形式を使用しました）。

念のため、ハッシュはファイルの個々の「指紋」であり、別のファイルのハッシュとの一致が（ほぼ完全に）ないことを保証します。ファイルからハッシュを受け取ったら、ブロックチェーンネットワークでトランザクションを生成するだけでよく、トランザクションの本文にドキュメントのハッシュが示されます。

順番に見ていきましょう：

パート1：スマートコントラクトとEthereumウォレットの準備

PoE自体は、最初にビットコインブロックチェーンに関連付けられました。しかし、実装にはイーサーブロックチェーンとスマートコントラクトを選択しました。スマートコントラクトは、将来の柔軟性、モジュール性、スケーラビリティを提供し、ブロックチェーンのハッシュのストレージとして機能します。

ハッシュを受け入れることができる単純なスマートコントラクトを選択しました。対応する日付を使用して、ハッシュ=>日付の形式です。また、呼び出されたときにハッシュ日付を返すメソッドがあると非常に便利です。さらに、契約の所有者だけが新しいハッシュを追加できることに注意してください。

スマート契約コード：

```
pragma solidity 0.5.9;

/**
 * @title Ownable
 * @dev The Ownable contract has an owner address, and provides basic authorization control
 * functions, this simplifies the implementation of "user permissions".
 */
contract Ownable {
    address private _owner;

    event OwnershipTransferred(address indexed previousOwner, address indexed newOwner);

    /**
     * @dev The Ownable constructor sets the original `owner` of the contract to the sender
     * account.
     */
    constructor () internal {
        _owner = msg.sender;
        emit OwnershipTransferred(address(0), _owner);
    }

    /**
     * @return the address of the owner.
     */
    function owner() public view returns (address) {
        return _owner;
    }

    /**
     * @dev Throws if called by any account other than the owner.
     */
    modifier onlyOwner() {
        require(isOwner());
        _;
    }

    /**
     * @return true if `msg.sender` is the owner of the contract.
     */
    function isOwner() public view returns (bool) {
        return msg.sender == _owner;
    }

    /**
     * @dev Allows the current owner to transfer control of the contract to a newOwner.
     * @param newOwner The address to transfer ownership to.
     */
    function transferOwnership(address newOwner) public onlyOwner {
        _transferOwnership(newOwner);
    }

    /**
     * @dev Transfers control of the contract to a newOwner.
     * @param newOwner The address to transfer ownership to.
     */
    function _transferOwnership(address newOwner) internal {
        require(newOwner != address(0));
        emit OwnershipTransferred(_owner, newOwner);
        _owner = newOwner;
    }
}

/**
 * @title HashStore
 * @dev The contract store the hashes and returns date by hash
 * Only the owner can add new hashes
 */
contract HashStore is Ownable {
    mapping(bytes32 => uint256) private _hashes;
    event HashAdded(bytes32 hash);

    function addHash(bytes32 rootHash) external onlyOwner {
        require(_hashes[rootHash] == 0, "addHash: this hash was already deployed");

        _hashes[rootHash] = block.timestamp;
        emit HashAdded(rootHash);
    }

    function getHashTimestamp(bytes32 rootHash) external view returns (uint256) {
        return _hashes[rootHash];
    }
}

```

すでにお気づきのとおり、OwnableとHashStoreの2つの別々の契約を使用しました。
HashStoreコントラクトは、ハッシュを格納し、要求に応じてハッシュ日付を発行する責任があります。所有可能契約は、新しいハッシュが契約所有者によって排他的に追加されたことを確認する責任があります。

ハッシュを追加するには、addHashメソッドを呼び出し、sha256という値をファイルの引数として渡す必要があります。契約内のハッシュが一致する場合、トランザクションは拒否されます。これにより、日付が異なる値の不要な重複がフィルター処理されます。ここでチェックされます：

require(_hashes[rootHash] == 0, "addHash: this hash was already deployed");

getHashTimestampメソッドを使用してハッシュトランザクションの日付を取得し、必要なトランザクション日付を引数としてハッシュを渡すことができます。getHashTimestampメソッドは、UNIX形式で時間を返します。より読みやすい形式に変換できます。

それで、契約を理解しました。次に、それをネットワークに展開する必要があります。これを行うには、次の2つが必要です。

  1. ブロックチェーンと相互作用するためのエーテルアドレス
  2. メソッドの呼び出しとコントラクトの展開のためのいくつかのエーテル。

テストでエーテルを無駄にしないために、私たちはropstenテストネットワークを使用できます。詳細はこちら。このアドレスはデプロイメントの開始者であり、将来新しいハッシュを追加する可能性があるため、契約の所有者です。その後、このアドレスを安全な場所に保存する必要があります。ウォレットの秘密鍵と契約のアドレスも保存する必要があります。将来的にはこれが必要になります。

パート2：Node.jsでのトランザクションと秘密キーの生成

したがって、この時点で、すでにイーサウォレット、その秘密キー、テストネットワーク内のコントラクト、およびデータハッシュがあります。ブロックチェーンとの相互作用を構成するために残ります。

これを行うには、web3.jsライブラリを使用して、ノードのトークンを作成します。私はinfura.ioサービスを使用して鉱山を作成しました。これは次のようになります。

ropsten.infura.io/v3/YOUR_TOKEN_HERE

ハッシュには、sha256パッケージを使用します。データ形式は任意ですが、例ではJSONのデータを使用しています。

PoEを使用してケースをどのように解決しましたか？


PoEに加えて、ブロックチェーンとトランザクション料金でユーザーに過負荷をかけないことが重要でした。たとえば、addHashメソッド（bytes32 rootHash）の呼び出しには0.2フィニー（0.0002 ETHまたは$ 0.06、2019年6月のレート）がかかります。



1日に約30の戦略ポジションの結論が出ました。つまり、2.1ドルかかります。もちろん、ユーザー数が100倍に増えてEtherのレートが上がると、当然コストも上がります。

私は1日、ブロックチェーンに1つのハッシュを保持することにしました。このハッシュは、毎日の戦略出力のハッシュとその内部識別子から生成されます。



機能的には劣らないが、はるかに安価なソリューションが得られます。スケーラビリティの問題は確認されていませんが、戦略の位置の出力をいつでもいくつかのブロックに分割し、いくつかのハッシュをブロックチェーンネットワークに保存できます。そして、参加者は他の隣接するハッシュを使用して別のハッシュを簡単に検証できます。
