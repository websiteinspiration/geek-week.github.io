Cでデータを処理するときに並列化を使用する＃



全てにおいて良い日！私は内部監査システムで働く技術専門家です。私の責任には、C＃プログラミング言語でETLツールを作成することが含まれます。

定期的に、データソースはxml、csv、json、またはその他の形式の緊密に構造化されたファイルです。時々それらの数は非常に大きくなり、絶えず増加します。たとえば、私のタスクの1つでは、ファイルの数が増え、1日あたりの平均更新頻度は約150,000ファイルでした。同時に1つのファイルの処理（ハードディスクからメモリへのバイト配列の読み取り、ダウンロードしたデータの変換、およびデータベースへの書き込み）に1秒かかる場合、すべてのファイルの処理に40時間以上かかることが明らかになります。この場合、ファイル数を増やす速度が処理速度よりも明らかに速いため、これらのファイルを最後まで処理することはできません。

この問題の1つの解決策は、互いに独立したスレッドのプールを作成するアプリケーションを開発することです。スレッドは、一般的なキューから選択されたファイルを処理します。ただし、この場合、相互ロックが発生する可能性が非常に高いため、作業スレッドの同期とリソース共有で問題が発生します。

これらの問題を回避するために、MicrosoftはTPLライブラリを.Netフレームワークに追加しました（バージョン4.0以降）。このライブラリを使用してこの問題を解決する方法を説明します。

したがって、最初の操作アルゴリズムは次のようになります。

ファイルストレージディレクトリがスキャンされ、すべてのファイルに関するデータを含むリスト（Listなど）が返されます。
次のファイルのデータがメモリに読み込まれ、必要に応じて変換されてデータベースに書き込まれるサイクルが開始されます（forまたはforeach）。

明らかに、最も時間のかかる操作は、ハードディスクからメモリへのデータの読み取りと、メモリからデータベースへのデータの書き込みです。

TPLライブラリを使用してアルゴリズムを最適化してみましょう。

ステップ1.

ファイルストレージディレクトリをスキャンして返されたリストを、ListからConcurrentQueueに変更します。
なぜこれを行うのですか？実際には、ConcurrentQueueクラスはスレッドセーフです。つまり、2つのスレッドが同時にこのリストからデータを抽出したり、データを書き込んだりしようとした場合、例外（Exception）はスローされません。
アルゴリズムのポイント1は次のようになります。ファイルストレージディレクトリがスキャンされ、すべてのファイルに関するデータを含むConcurrentQueueリストが返されます。

ポイント2：
ファイルからのデータを処理するサイクルを形成するデザインを変更してみましょう。 forをParallel.ForまたはParallel.ForEachに置き換えます。

新しい構造との違いは何ですか？すべてがシンプルで、基本的には言語構造の名前から明らかです。ループのすべての反復は、並列スレッドで実行されます。例として、Parallel.ForEachコンストラクトを使用したループの構成を示します。

Parallel.ForEach(listFiles, (currentFile) =>
       	  {
              	var dataFile = getDataFile(currentFile.FullName);
		TransformData(dataFile);
		WriteToDB(dataFile);
               });

ここで、

listFilesは、ディレクトリ内のファイルのリストを含むConcurrentQueueタイプのコレクションです。
currentFile-ForEach構文によって返されるlistFilesコレクションの要素。
dataFile-ファイルの内容をメモリに読み込むことによって取得される、メモリ内の条件付きのいくつかのデータ構造。
getDataFile-データ構造の形式でファイルの内容を返す条件付き関数。
TransformData-受信したデータを変換するための条件付きプロシージャ。
WriteToDBは、データベースにデータを書き込むための条件付きプロシージャです。

この例では、Parallel.ForEachコンストラクトを使用して、ループを編成します。このサイクルでは、並列ストリームで、データがハードディスクから読み取られ、変換されてデータベースに書き込まれます。同時に、並列フローの作業の編成に問題はありません。並列スレッドの数は、プロセッサコアの数とそのワークロードによって異なります。

提案されたアルゴリズムを使用して、ファイル処理を少なくとも2倍高速化します。もちろん、この数値はコアの数とプログラムが実行されるマシンのメモリによって異なります。

また、プログラムを高速化するには、メインのレコードに関係なく機能する別のストリームにデータベースのレコードを配置する必要があります。これは、データをキューに追加するときの競合を回避するために、ConcurrentQueueコレクションを使用して実行できます。

データベースへの書き込みの最適化を考慮して、上記の例を書き直します。
ファイルリーダーがDataTableでデータを返したとします）：

Parallel.ForEach(listFiles, (currentFile) =>
       	  {
              	DataTable dataFile = getDataFile(currentFile.FullName);
		TransformData(dataFile);
		threadWriteToDB.ListData.Enqueue(dataFile);
               });

ご覧のとおり、データベースの書き込みプロシージャを呼び出す行の代わりに、説明し、別のスレッドで初期化したConcurrentQueue ListDataコレクションに単純に追加します。このスレッドのインスタンスはループで使用されます。

データベースへの書き込みはすでに別のストリームにあります。データベースへの書き込みは、Parallel.Forおよび/またはParal-lel.Foreach構造を使用してファイルを操作する場合と同様に整理できます。

同等の数のファイルの処理が必要だった私のタスクでは、現在、平均して1日あたり200,000〜400,000のファイルを処理でき、データベースとデータチャネルの幅をロードすることによって速度が制限されています。
