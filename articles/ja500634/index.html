食品インフラストラクチャー：Tanukiのサポート方法



1つの有名な歴史上の人物を言い換えると、私たちにとってすべてのサービスの中で最も重要なのは配信です。家にロールパンやピザがない現在の状況ではどうしますか？想像したくありません。たまたま2年前、最大のネットワークの1つであるTanukiのサポートを受けました。サイトの毎月の聴衆は約100万人です。現在は2020年です。たぬきが私たちと協力し始めた2018年、それは2倍小さくなりました。新しいデータセンターへのスムーズな移行を確実にし、インフラストラクチャを完全に再構築しました。これにより、サイトとアプリケーションは増加した負荷に問題なく耐えることができます。

時々、最寄りのタヌキレストランから地理的に遠く離れていることを強く後悔します。それ以外の場合、すべての成功（および小さな不幸）は美味しいロールパンで詰まることになります。

一般的に、今日私たちはあなたに国内の「ホスピタリティ産業」の最大のウェブプロジェクトの一つのサポートの物語を伝えたいと思います。

2018年3月末に会いました。

国際女性の日は長い間過ぎましたが、男性たちはその結果に対処しました。すべてがかなり平凡です。3月8日の前夜、トラフィックは急激に増加し、サイトは長期間利用できませんでした。ほんの数時間ではなく、本当に長いです。トラフィックはメインサイトだけでなく、アプリケーション（AndroidおよびiOSで利用可能）とアグリゲーター（Yandex。Food、Delivery Club、Zaka-Zaka）からも流れたためです。

私たちが見たもの


技術的には、プロジェクトは非常に複雑であることがわかりました。

  • このサイトはSSR（サーバー側レンダリング）を備えた反応アプリケーションです。
  • モバイルアプリケーション-iOS / Android用。
  • API-すべてのアプリケーションで使用できます。
  • 注文処理を含む外部システム。


システムはリバースプロキシサーバーでした。それらへのトラフィックはDDoS攻撃に対する保護システムを通過し、そこからバックエンドサーバーに既に分散されていました。採用当時は古いサイトとモバイル版のAPIがあり、新しいサイトの開発が始まりました。新しいAPIの開発は、別々のサーバーで行われました。

データベースクラスターは、マスター/マスターレプリケーションを備えた2つのサーバーで構成され、フローティングIPが原因でネットワークレベルで障害が発生した場合の切り替えが行われました。すべての書き込みアプリケーションはこのIPで動作しましたが、読み取り用のMySQLスレーブは各バックエンドサーバーにありました-アプリケーションはそれに応じてlocalhostで動作しました。

レセプションでは、次のような問題がありました。

  • データベース構成での信頼性が不十分な分散メカニズム。マスターレプリケーションマスターは、頻繁な障害につながりました。
  • 各バックエンドのスレーブ-大量のディスク容量が必要でした。また、新しいバックエンドサーバーを操作したり追加したりするには、コストがかかりました。
  • アプリケーションの一般的な導入システムはありませんでした。Webを介した自己完結型の導入システムがありました。
  • ログを収集するためのシステムはありませんでした。その結果、最初に注文システムを使用してインシデントを調査することは非常に困難です。これは、特定の注文をどのように受け取ったかを判断する方法がないためです。
  • ビジネス指標の監視はありませんでした。注文の減少または完全な欠如をタイムリーに記録することはできませんでした。

監視のために受け入れられたサーバーの初期監査の後、運用ロードマップの形成から始めました。最初に、2つの主要な作業領域が特定されました。
  1. アプリケーションの安定化。
  2. 新しいAPIとサイトのための快適な開発環境の編成。

最初の方向での決定は、主にMySQLクラスタの安定化に関連していました。マスターレプリケーションマスターを拒否したくありませんでしたが、フローティングIPでの作業を続けることは不可能でした。ネットワーク接続の中断が定期的に観察されたため、クラスターの動作が中断しました。

最初に、nginxをMySQLのプロキシとして使用したため、フローティングIPを放棄して、マスター間でアップストリームが制御されるプロキシサーバーを優先することにしました。 2番目のステップは、スレーブ用の2つの別個のサーバーの割り当てです。彼らとの仕事もまた、プロキシサーバーを通して組織されました。また、再編成以来、データベースの操作に関連する問題を忘れていました。

次に、データベースのクエリレベルで注文を監視しました。標準からの逸脱は、多かれ少なかれ-すぐに調査を引き起こしました。次に、ログレベルで、特に注文管理システムとの外部のやり取りを監視するためのメトリックを作成しました。

同僚と一緒に、彼らの要求に応じて、すべてのシステムの追加調整を行い、安定して高速な動作を実現しました。 MySQLのチューニングとPHPのバージョンの更新の両方でした。さらに、同僚はRedisに基づくキャッシングシステムを導入しましたが、これもデータベースの負荷を軽減するのに役立ちました。

これらすべてが重要でした...しかし、ビジネスの主なものは売上の増加でした。そしてこの文脈で、会社のマネージャーは新しいサイトに高い期待を持っていました。開発者にとって、展開とアプリケーション制御のための安定した便利なシステムを入手する必要がありました。

まず、CI / CDアプリケーションの組み立てと配信のパイプライン、およびログを収集して操作するシステムについて考えました。

すべてのクライアントリポジトリは、自己ホスト型のbitbucketソリューションでホストされていました。パイプラインの構成には、ジェンキンスが選ばれました。

そもそも、開発環境にパイプラインを実装するのが慣習でした。これにより、開発速度を大幅に向上させることができました。次に、それは生産回路に導入され、自動展開により、通常は人的要因によって引き起こされる頻繁なミスが回避されました。

実装後、CI / CDはログの収集の整理とログの操作を開始しました。ELKスタックがメインスタックとして選択されたため、インシデント発生時にクライアントはより迅速かつ適切に調査を実施できました。その結果、アプリケーション開発はより速くなりました。

「2回の火災よりひどい...」


非常に複雑であるにもかかわらず標準的なタスクを解決した後、田貫は彼らが長い間言いたかったことを私たちに語りました：「移動しましょう！」

DCの変化は経済的要因によって引き起こされました。さらに、クライアントは、新しいDCに既に含まれていた追加サービスでインフラストラクチャを拡張しました。これも、移動する決定に影響を与えました。

複雑なシステムはもちろんのこと、あらゆるシステムの移行は、綿密な計画と大規模なリソースを必要とするプロセスです。

移動は繰り返し実行されました。最初の段階で、新しいDCにリバースプロキシサーバーが作成されました。また、パブリックIPしか持っていないため、管理者のシステムへのアクセスポイントとしても機能しました。

次に、ロギングとCI / CDのすべてのインフラストラクチャサービスを開始しました。領事クライアントアプリケーション間の対話の便利で、管理が容易で、かなり信頼できるサービスを整理することができます。

移行された次のデータベース、Redisおよびキューブローカー-RabbitMQ。ここでは、サービスディスカバリプロトコルに正しく登録され、DNSの動作を制御するようにすべてを整理することが重要でした。アプリケーションはデータベースを直接操作するのではなく、Haproxyを介して機能することに注意してください。Haproxyを使用すると、データベース間でバランスを取り、障害が発生した場合に切り替えることができます。

準備段階では、データセンター間のデータベースレプリケーションは増加しませんでした。バックアップを転送する必要がありました。次に、アプリケーションを直接構成し始めました。これは、内部DNSを介したすべての対話の編成です。これは、アプリケーションとデータベース/ Redis / RabbitMQ /外部サービス（注文サービスなど）の間の対話です。当然、同じ段階ですべてのCI / CDメカニズムが即座に接続され、次にアーキテクチャの2番目の変更が発生しました。以前は、インターフェースを介してアプリケーション設定を変更することはできませんでした-コンソールで直接ファイルを編集することによってのみ。すぐに、Webインターフェイスを介して設定を簡単に管理できるソリューションを導入しました。これはHashicorpボールト（Consulがそのバックエンドとして機能）に基づいていたため、環境変数を管理するための便利なメカニズムを構築できました。

次のステップは、サービスを新しいDCに切り替えることです。すべてのシステムの作業がhttpプロトコルを使用して編成され、すべてのドメインがDDoS攻撃に対する保護システムを通過したため、切り替えはこのシステムのインターフェースで直接アップストリームを操作するように削減されました。

以前は、必要なレプリカは古いDCから新しいDCに編成されていました。そして、合意された作業ウィンドウに切り替えられました。

現在のインフラストラクチャ





  • すべてのトラフィックはバランサーに送られます。APIへのトラフィックは、Tanukiアプリケーション（Android / iOS）から直接ではなく、Qratorを経由します。
  • 静的サーバーでは、tanuki.ruプロジェクトのメインサイトであり、ランディングページを備えたサーバーです。
  • バックエンドクラスターは、サーバー（フロントエンドサーバー、静的サーバー、アプリケーションサーバー）で構成されます。

注文の受け渡しの方式
受け取った注文はQratorを通過し（すぐに攻撃を除外します）、APIに到達します。次に、注文を配信するために雷電に行き、Redisに行き、nginxに行き、その後データベースに向かいます。

顧客にとって何が変わったか


  • システムの信頼性：問題は2019年7月に観察されました-注文は1時間以内に発行されませんでした。しかし、それは世界的な動きの前でした。その後大きな事件は観察されなかった。
  • 開発者の生活：彼らにはCI / CDという便利な開発環境があります。
  • フォールトトレランス：インフラストラクチャは大量のトラフィックに耐えられるようになりました。たとえば、休暇中のRPSのピークは550ユニットでした。

次は何ですか


現代の状況では、オンライン販売が前面に出ています。プロジェクトは、サービス顧客に信頼性とアクセシビリティを提供する必要があります。しかし、開発も非常に重要な要素です。製品のリリースは、エンドユーザーからできるだけ早く、見えないようにする必要があります。

もう1つの重要な問題は、リソースの利用とシステムの維持コストの削減です。

これらすべてにより、システム全体を確認する必要が生じます。最初のステップは、アプリケーションのコンテナー化を編成することです。次に、Kubernetesクラスターが計画されます。しかし、これについては次の記事で説明します。それまでの間-bon appetit :-)
