サードパーティのアプリケーションコンテキストからSpring Beanを正しく取得する

こんにちは、Khabrovites！

この記事では、Springフレームワークを使用するプロジェクトでよく発生する問題の1つについて説明します。この記事で説明する問題は、スプリング構成の一般的なエラーの1つが原因で発生します。構成でこのようなミスをする必要はないため、このエラーは非常に一般的です。

問題文


この記事で紹介する問題は、現在のアプリケーションコンテキストでのBeanの誤った設定に関連しており、他のアプリケーションコンテキストから取得されています。このような問題は、多数のjarで構成される大規模な産業用アプリケーションで発生する可能性があり、jarにはそれぞれ、スプリングBeanを含む独自のアプリケーションコンテキストがあります。

誤った構成の結果として、スコープがシングルトンであっても、予測できない状態のBeanのコピーがいくつか取得されます。さらに、豆の無知なコピーは、アプリケーションが任意のjarのすべての豆の12を超えるコピーを作成するという事実につながる可能性があり、これはアプリケーションのパフォーマンスの問題とアプリケーションの起動時間の増加を伴います。

現在の外部アプリケーションコンテキストからBeanを使用する例


他の多くのモジュールが存在するアプリケーションモジュールの1つで開発していて、各モジュールに独自のアプリケーションコンテキストがあるとします。このようなアプリケーションには、すべてのアプリケーションモジュールのアプリケーションコンテキストのインスタンスが作成されるモジュールが必要です。



いずれかの外部モジュールのアプリケーションコンテキストで、NumberGeneratorクラスのBeanのインスタンスが作成され、それをモジュールで受信するとします。また、NumberGeneratorクラスが、値を生成するいくつかのクラスを格納するパッケージorg.example.kruchon.generatorsにあるとします。



このBeanには状態（int countフィールド）があります。

package org.example.kruchon.calculators

public class NumberGenerator {
    private int count = 0;

    public synchronized int next() {
        return count++;
    }
}

このBeanのインスタンスは、GeneratorsConfigurationサブ構成で作成されます。 

@Configuration
public class GeneratorsConfiguration {
    @Bean
    public NumberGenerator numberGenerator() {
        return new NumberGenerator();
    }
    ...
}

また、外部アプリケーションのコンテキストには、外部モジュールのすべてのサブ構成がインポートされるメイン構成があります。

@Configuration 
@Import({GeneratorsConfiguration.class, ...})
public class ExternalContextConfiguration {
    ...
}

次に、NumberGeneratorクラスのシングルトンBeanが現在のアプリケーションコンテキストの構成で正しく構成されていない例をいくつか示します。

正しくない構成1.外部アプリケーションコンテキストのメイン構成のインポート


あり得る最悪の決定。 

@Configuration
@Import(ExternalContextConfiguration.class)
public class CurrentContextConfiguration {
    ...
}

  • 外部アプリケーションコンテキストからのすべてのBeanインスタンスがアプリケーションで再作成されます。つまり、外部モジュール全体のコピーが作成され、メモリの消費、パフォーマンス、およびアプリケーションの起動時間に影響します。
  • 現在のアプリケーションコンテキストでNumberGeneratorのコピーを取得します。NumberGeneratorのコピーには、CountGeneratorの最初のインスタンスと一致しない、countフィールドの独自の値があります。この不整合により、アプリケーションでデバッグが困難なエラーが発生します。

正しくない構成2.外部アプリケーションコンテキストのサブ構成をインポートする


2番目のオプションは正しくなく、実際に頻繁に発生します。

@Configuration
@Import(GeneratorsConfiguration.class)
public class CurrentContextConfiguration {
    ...
}

このオプションでは、外部モジュールの完全なコピーは作成されなくなりましたが、それでも、NumberGeneratorクラスの2番目のBeanを再び取得します。

不正な構成3. NumberGeneratorを使用するBeanへの直接注入を検索します


public class OrderFactory {
    private final NumberGenerator numberGenerator;

    public OrderFactory() {   
         ApplicationContext externalApplicationContext = getExternalContext();
         numberGenerator = externalApplicationContext.getBean(NumberGenerator.class);
    }

    public Order create() {
         Order order = new Order();
         int id = numberGenerator.next();
         order.setId(id);
         order.setCreatedDate(new Date());
         return order;
    }
    
    private ApplicationContext getExternalContext(){
        ...
    }
}

この方法では、スコープがシングルトンのBeanを複製する問題は解決されたと見なすことができます。結局のところ、ここでBeanを別のアプリケーションコンテキストから再利用し、再作成しません！

しかし、この方法：

  1. 開発されたクラスとその単体テストを複雑にします。
  2. 現在のモジュールのBean内のNumberGeneratorクラスのBeanの自動実装を除外します。
  3. 一般に、lookUpを使用してシングルトンBeanを注入することは一般的ではありません。

したがって、そのような解決策は、問題の合理的な解決策というよりも、不器用な回避策に似ています。

外部アプリケーションコンテキストからBeanを適切に構成する方法を検討してください。

解決策1.構成の外部アプリケーションコンテキストからBeanを取得する


このメソッドは、誤った構成の3番目の例と非常に似ていますが、1つだけ違いがあります。構成内の外部コンテキストからLookUpを作成することでBeanを取得し、直接Bean内ではありません。

@Configuration
public class CurrentContextConfiguration {
    @Bean
    public NumberGenerator numberGenerator() {
         ApplicationContext externalApplicationContext = getExternalContext();
         return externalApplicationContext.getBean(NumberGenerator.class);
    }

    private ApplicationContext getExternalContext(){
        ...
    }
}

これで、このBeanを独自のモジュールのBeanに自動的に埋め込むことができます。

解決策2.外部アプリケーションコンテキストを親にする


現在のモジュールの機能が外部の機能を拡張している可能性があります。外部モジュールの1つでアプリケーション全体に共通の補助Beanが開発され、他のモジュールではこれらのBeanが使用される場合があります。この場合、外部モジュールが前のモジュールの親であることを示すのは論理的です。この場合、親モジュールのすべてのBeanを現在のモジュールで使用できるため、親モジュールのBeanを現在のアプリケーションコンテキストの構成で構成する必要はありません。

parentパラメータを指定したコンストラクタを使用してコンテキストインスタンスを作成するときに、親関係を指定することができます。

public AbstractApplicationContext(ApplicationContext parent) { ... }

または、セッターを使用します。

public void setParent(ApplicationContext parent) { ... }

アプリケーションコンテキストがxmlで宣言されている場合は、コンストラクタを使用できます。

public ClassPathXmlApplicationContext(String[] configLocations,
   ApplicationContext parent) throws BeansException { ... }

結論


したがって、Spring Beanを構成するときは注意して、記事の推奨事項に従い、スコープがシングルトンのBeanをコピーしないようにしてください。ご質問にお答えいたします！
