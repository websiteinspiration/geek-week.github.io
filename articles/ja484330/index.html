[Nginx] response_status = 0に勝つ方法

「マージナルノート」のカテゴリーの記事。

TL：DR：
http2_max_field_size 8k; #  !


プロジェクトの1つで、いくつかの内部バックエンドロジックを変更した後、ログで奇妙なresponse_code、つまり0を確認し始めました。ログでは、次のようになります。

{
  "timestamp": "2020-01-17T08:41:51+00:00",
  "remote_addr": "zzz.zzz.zzz.zzz",
  "request_time": 0,
  "upstream_response_time": "",
  "upstream_header_time": "",
  "http_accept_language": "-language",
  "response_status": 0,
  "request": "",
  "host": "example.com",
  "upstream_addr": "",
  "http_referrer": "",
  "request_length": 5854,
  "bytes_sent": 0,
  "http_user_agent": ""
}

ドキュメントを読んでこのトピックをグーグルで検索しても、まったく何も起こりませんでした。この動作は、クライアントがヘッダーを渡さずに接続を閉じたときに発生すると主張されています。さて、wsgi_のバッファのサイズが異なるエキゾチックですが、今回のケースでは「ノーウェイ」という言葉に適合しませんでした。

一般的に、私たちのボリュームではそれは絶対に重要ではないという事実を考慮して、問題は問題ではないと判断しました。

私は、次のような問題で困惑して正確にまで：いくつかのケースでは、リンクは、http経由で問題なく開くが、完全にHTTPS経由で作業することを拒否し、奇跡的与える：接続＃0をホストexample.comにそのままに
カールを：から（52）空の返信サーバ

上記の例からわかるように、IPによってのみログでこのことを追跡することが可能でした。リクエストも他のデータもありませんでした。悪名高いステータスのみが0ですが、リクエストを中断しなかったことがわかります。私は何がうまくいかないのかを選び始めました。そして、すべてが非常に単純であることがわかりました：

443 ssl http2 backlog = 8192を聞いてください。

さて、ssl接続にhttp2を使用する場合、リクエストバッファを構成するだけでは不十分であり、ngx_http_v2_moduleでもそれらを構成する必要があります。

:	http2_max_field_size ;
:	http2_max_field_size 4k;
:	http, server

HPACKで圧縮された要求ヘッダーの最大サイズを制限します。制限は名前と値の両方に等しく適用されます。ハフマンコーディングを使用すると、名前と値のアンパックされた文字列の実際のサイズが大きくなる可能性があります。デフォルトの制限は、ほとんどのクエリに適しています。

一般的に、これはそれです。なぜ？リンクの長さが長いため-同じ4k以上。

たとえば、8 kb（または十分と思われる量）に入れて、問題を解決します。
だからそうなるのです。
