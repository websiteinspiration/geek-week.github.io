コンソールコマンドの複雑化、1979-2020

私の趣味は、あるモニターでマキロイの「UNIX哲学」を開き、別のモニターでマナを読むことです。

McIlroyの最初の原則は、「1つのことを行うが、うまくやる」と言い換えられることがよくあります。これは、「1つのことをうまく行うプログラムを作成する」という彼の言葉の略語です。新しい仕事のために、新しい「機能」を追加することによって古いものを複雑にするのではなく、新しいプログラムを作成してください。

McIlroyが例を示します。

UNIXコンパイラがリストを発行しないことは、部外者にとっては驚くべきことです。印刷はより適切に行われ、別のプログラムを使用してより柔軟に構成されます。

あなたがのためにヘルプを開くとls、それはで始まる

ls [-ABCFGHLOPRSTUW@abcdefghiklmnopqrstuwx1] [file ...]

それは、一文字のフラグでls、すべて小文字を含んを除き、{jvyz}14の大文字、@と1。これは、22 + 14 + 2 = 38の単一文字オプションのみです。

Ubuntu 17では、ヘルプlsは通常の概要を表示しませんlsが、58のオプション（--helpおよびを含む--version）があることがわかります。

何年にもわたって、オプションの数が劇的に増加していることがわかります。原則として、レコードは左から右に暗くなり（オプションが増える）、レコードが薄くなるケースはありません（オプションが少ない）。

皮肉なことに、コマンドラインオプションの数が増加している理由の1つは、McIlroyの別の言い回しです。「これはユニバーサルインターフェイスであるため、テキストストリームを処理するプログラムを作成します」（ls例を参照）。

構造化データまたはオブジェクトが送信された場合、フォーマットは最終段階に任せることができます。ただし、プレーンテキストの場合、フォーマットとコンテンツが混在しています。フォーマットはコンテンツを解析することによってのみ実行できるため、コマンドは通常、便宜上フォーマットオプションを追加します。ユーザーがの引数の知識をデータ構造の知識を適用し、「コードする」場合に加えて、フォーマットを行うことができますcut、awk、sedなど（ユーザーは、これらのプログラムがフォーマットでどのように機能するかについての知識も使用します。これは、プログラムによって異なるため、たとえば、2とのcut -f4 違いを知っている必要があります。awk '{ print $4 }）これは、シーケンス内の次のコマンドに1つまたは2つの引数を渡すよりもはるかに面倒です。これにより、ツールの複雑さがユーザーに伝達されます。

構造化データをサポートしたくないと言う人もいます。それは、普遍的なツールでは複数のフォーマットをサポートする必要があるためです。ただし、ユニバーサルツールを作成するには、すでにいくつかの形式をサポートする必要があります。一部の標準コマンドは、異なるフォーマットを使用しているため、他のコマンドからの出力を読み取ることができません。たとえば、wc -wUnicodeでは正しく処理されません。「テキスト」がユニバーサルフォーマットであると言うことは、「バイナリ」がユニバーサルフォーマットであると言うことと同じです。

コマンドラインツールのそのような複雑さの代替手段はないと言われています。しかし、それを言う人は、PowerShellのような代替手段を試したことがありません。PowerShellについては多くの不満がありますが、構造化データの転送と、メタデータを頭に置いておかなくても構造化データを簡単に操作できるため、パイプラインの適切な場所にある適切なコマンドラインツールに転送できます3。。

テキストを処理する際にプログラムは単純で互換性があるべきだと言われたとき、これらの人々はテキストデータが解析するための構造を持っていないと偽ります4。場合によっては、すべてをスペースで区切られた1つの行、または行と列の区切り文字を含むテーブルと考えることができます（もちろん、動作は他のツールとは異なります）。これは少し面倒です。データをプレーンテキスト形式にシリアル化すると、かなり複雑になる場合があります。データ構造のため、テキストへの単純なシリアル化では、意味のある方法でデータを再同化するために、かなりの解析作業が必要になるためです。

チームがより多くのオプションを使用できるようになったもう1つの理由は、複数のチームのパイプラインによって実装できる機能に便利なフラグが追加されたためです。この慣習は、Unix v7から続いています。lsソート順を変更するオプションが表示されました（これは出力をに渡すことで実行できますtac）。

時間の経過とともに、単に便宜上、追加のパラメーターが追加されました。たとえば、最初はパラメータmvがなかったコマンドがファイルを移動し、同時にバックアップコピーを作成できるようになりました（3つのオプション、バックアップを指定する2つの異なる方法、1つは引数を取り、もう1つは引数を取らない、環境変数から暗黙の引数を読み取るVERSION_CONTROL;別のオプションデフォルトのバックアップサフィックスを上書きできます）。今mvでもファイルを上書きしないか、新しいファイルのみを上書きするオプションがあります。

mkdir以前はオプションがなかった別のプログラム。現在、SELinuxまたはSMACKのセキュリティオプションを除くすべてのフラグ、ヘルプとバージョン番号が追加されています。これは、新しいディレクトリにアクセス許可を設定し、親ディレクトリが存在しない場合は作成することです。

最初は、tailオプションが1つしかなく-number、作業の開始点を示していました。次に、フォーマットの便宜のために、フォーマットとオプションの両方を追加しました。フラグ-zは、行区切り文字をに置き換えnullます。便利のために追加されたオプションの他の例を次に示します。-f新しい変更が表示-sされたときに印刷し、変更の確認とコード間のタイムアウト間隔を設定します> -f、および-retryファイルが利用できない場合はファイルへのアクセスを再試行します。

McIlroyは、これらすべてのオプションを追加したことで開発者を批判していますが、個人的には気分が良くなりました。それらのいくつかを使用したことはありませんが、他のものを使用することはめったにありませんが、これはコマンドラインパラメーターの美しさです。グラフィカルインターフェイスとは異なり、これらのパラメーターを追加してもインターフェイスが乱雑になることはありません。はい、マナとヘルプは膨らみますが、GoogleとStackoverflowの時代では、いずれにせよ、多くの人がマナを読むのではなく、質問をググってください。

もちろん、オプションを追加すると、メンテナの負荷が増加します。しかし、これは彼らがもたらす利益に対する公正な支払いです。メンテナーとユーザーの数の比率を考えると、後者ではなく前者に追加の負担をかけるのが合理的です。これは、パフォーマンスを50回リハーサルすることが賢明であるというGary Bernhardtの発言に似ています。聴衆が300人の場合、リハーサルに費やされた時間に対するパフォーマンスの視聴に費やされた時間の比率は6：1のままです。一般的なコマンドラインツールの場合、この比率はさらに極端になります。

これらの追加オプションのすべてがユーザーに余分な負担をかけると主張する人もいます。これは完全に間違っているわけではありませんが、この複雑さの負担は常に存在します。問題は正確にどこにあるかです。一連のコマンドラインツールとシェルを組み合わせて、誰もが新しいメソッドを記述できる言語を形成していると想像してください。人気がある場合、そのメソッドは標準ライブラリに効果的に追加され、標準は「テキストストリームを処理するためのプログラムを書く」という公理によって決定されます。インターフェース」の場合、言語を完全に理解すると、言語は一貫性のない書き込み専用の混乱に変わります。少なくとも、幅広いオプションと機能を備えたツールのおかげで、Unixユーザーは非常に一貫性のない巨大なツールのセットを、大規模なツールのセットだけに置き換えることができます。外部からは一貫性がありませんが、内部には一貫性があります。

McIlroyは、ツールボックスでの配慮の欠如を意味します。同様に、Unixの創始者たちは同じ部屋に座って、「並外れた単純さ」の一連の一連のツールを思いつくまで、慎重に考えるべきです。しかし、規模は拡大せず、Unix自体の哲学は必然的に私たちの混乱を招きました。それは誰かが長い間または一生懸命考えなかったということではありません。ポイントは、1つの部屋に収まる共通の文化的理解を持つ比較的小さなチームを超えて拡大しない哲学です。

「Unixの哲学」に基づいてツールを作成したい場合、「シンプルさ」の意味や「1つのこと」の原則について、人によって意見が異なります5。ツールが正しく機能する方法-一貫性の欠如は青々とした色で咲き、その結果、PHPなどの非常に一貫性のない言語と同様に、非常に複雑になります。人々はPHPとJavaScriptをさまざまな奇妙な点や矛盾点でからかっていますが、言語や標準ライブラリのように、人気のある* nixツールのコレクションを集めた人気のあるシェルを組み合わせて使用​​すると、同じLinuxディストリビューション内でも矛盾が生じるため、ランダムな複雑さが大幅に高まります。 。それ以外の場合はできません。 Linux、BSD、Solaris、AIXなどのディストリビューションを比較すると、システムを切り替えるときにユーザーが覚えておくべきランダムな複雑さの量は、PHPまたはJavaScriptの不整合を覆い隠します。最も広くばかげているプログラミング言語は、それらと比較して優れたデザインの実際の例です。

明確にするために、私や他の誰かが70年代の開発にうまく対処できると言っているわけではありません。当時得られた知識を考慮に入れて、当時の有用性と今日のエレガントさの両方を兼ね備えたシステムを作成します。もちろん、振り返って振り返って問題を見つけるのは簡単です。マッキロイのような一部のUnix愛好家のコメントに私は単に同意しません。McIlroyは、私たちが忘れた、または単純さの価値を理解していないことを示唆しています。または、Cは他の言語と同じくらい安全な言語であり、エラーを表示したくない場合は、エラーのないコードを記述するだけでよいと言うケントンプソン氏。この種のコメントは、長年にわたってほとんど変わっていないことを意味します。伝えられるところによると、70年代に私たちは今日と同じ方法でシステムを構築し、50年間の集団的経験の間、数千万人年は私たちに何も教えませんでした。そして、起源、つまりUnixの作成者に目を向ければ、すべてがうまくいくでしょう。すべての敬意をもって、私は同意しません。

アプリケーション：メモリ


McIlroyのバイナリの肥大化に対する不満はこの記事の範囲外ですが、2017年に16 GBのRAMを搭載したChromebookを300ドルで購入したことに注意します。 1メガバイトのバイナリは、標準のApple IIに4キロバイトのメモリが搭載されていた1979年に深刻な問題になる可能性があります。 Apple IIの価格は、1979年には1,298ドル、2020年には4,612ドルでした。今日では、メモリが400万倍もありながら、この価格の1/15未満の安価なChromebookを購入できます。メモリー使用量が1000倍に増えたという不満は、（ポータブル！）マシンのコストが桁違いに低く、400万倍のメモリーを搭載している場合、少しばかげているように見えます。

最適化が好きなので、ホームページを2つのパッケージに絞りました（CDNがbrotliを高レベルでサポートしている場合は1つあります）が、これは純粋に美的要件なので、楽しみのために行います。コマンドラインツールのボトルネックはメモリを使用しておらず、ツールのメモリを1メガバイトのサイズで最適化する時間は、ホームページを1つのパッケージに削減するようなものです。楽しい趣味かもしれませんが、それ以上のものはありません。

テーブルのコンパイル方法


コマンドの使用頻度は、githubのコマンド履歴の公開ファイルから取得されます。これは、必ずしも個人的な経験とは一致しません。 curl、git、gcc（後者には1000を超えるオプションがある）、wgetなどのインスタンスを除いて、「単純な」コマンドのみがカウントされました。シンプルさの概念は相対的です。内蔵シェルコマンドのようなcd、も考慮に入れていませんでした。

フラグの繰り返しは個別のオプションとは見なされませんでした。例えば、U git blame -C、git blame -C -C及びgit blame -C -C -Cものの異なる行動が、それらはすべて、一つの引数と見なされます-C -C、彼らは-C 実際には異なる引数です。

表のサブオプションは1つのオプションとしてカウントされます。たとえばls、以下をサポートします

--format=WORD across -x, commas -m, horizontal -x, long -l, single-column -1, verbose -l, vertical -C

。7つのオプションformatにもかかわらず、これは1つのオプションとしてカウントされます。

役に立たないと明示的に示されているオプションは、引き続きオプションと見なされます。たとえばls -g、は無視されます。

同じオプションの複数のバージョンは、1つのオプションと見なされます。例えば、-Aと--almost-allのためにls。

証明書にオプションが存在すると記載されているが、実際には存在しない場合、そのオプションは考慮されません。たとえば、v7 mvのヘルプには次のように書かれています。

マイクロソフトに対する一般的な不満は、多くの場合、技術以外の組織上の理由によるAPIの大規模な離職です（たとえば、リモートツイートへの応答で説明されているStephen Sinofskyのアクションを参照してください）。それは本当です。ただし、初心者ユーザーの観点から見ると、標準のWindowsソフトウェアは、原則として、* nixよりもはるかに優れた非テキストデータを送信します。 Windowsの非テキストデータのカバレッジは、1999年には少なくともCOMまで遡ります（1990年と1987年にリリースされたOLEとDDEもそれぞれ可能です）。

あなたはバイナリ形式をサポートしているのFooからコピーした場合たとえば、AおよびBフォーマットをサポートしていますバー、中BおよびCその後、バズにバー、サポートからコピーCし、D、FooとBazにサポートされている共通の形式がない場合でも、すべて正常に動作します。

何かをカットまたはコピーすると、アプリケーションは基本的に、データを提供できる形式のクリップボードを「通知」します。アプリケーションに貼り付けると、最終的なアプリケーションは、使用可能な任意の形式のデータを要求できます。データが既にクリップボードにある場合、Windowsが提供します。そうでない場合、Windowsはソースアプリケーションからデータを受信し、それをターゲットアプリケーションに転送して、しばらくの間コピーが保存されます。 Excelから「カット」すると、彼は何十ものフォーマットで利用可能なデータを持っていると「あなた」と言うでしょう。そのようなシステムは互換性の点でかなり優れていますが、単純またはミニマルとは言えません。
