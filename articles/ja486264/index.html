PostgreSQL 13：並列バキューム

先日、Amit Kapilaが沢田正彦パッチをコミットしました。これにより、並行してクリーニングを行うことができます。テーブル自体は1つの（先行する）プロセスによってクリアされますが、インデックスをクリーンアップするために、インデックスごとに1つずつバックグラウンドワークフローを開始できるようになりました。手動モードでは、これにより、複数のインデックスを持つ大きなテーブルのクリーニングを高速化できます。自動クリーニングはまだこの機能を使用していません。

あなたに知っている、テーブルの洗浄プロセスは、いくつかの段階で構成されています。

まず、テーブルがスキャンされ、不要な（「デッド」）バージョンの行へのリンクがメモリに収集されます。メモリは、maintenance_work_memパラメータによって制限されるため、すべてのリンクが一度に収まるとは限りません。

次に、すべてのインデックスが（以前のように）順番に並べ替えられ、見つかった無効なバージョンの文字列へのポインターがそれらから消去されます。

次に、すでに表示されているテーブルの部分が再びスキャンされ、行の無効なバージョンはすでにテーブルからクリアされています。

一度にすべての無効なバージョンの文字列を処理することができなかった場合は、メモリがクリアされ、最後に停止した場所からプロセス全体が再び繰り返されます。

このスキーム全体は変更されていませんが、インデックスを並行してクリアできるようになりました。これを行うために、リードプロセスは、既存のインデックスを解析して処理するいくつかのワークフローを開始します。1つのインデックスは1つのプロセスのみで処理されます。すべてのインデックスがクリアされた後、ワークフローは完了し、ファシリテーターは次のフェーズに進みます。

たとえば、ticket_flights demobaseのフライトテーブルを見てみましょう。インデックスは1つですが、さらにいくつか作成できます。

demo=# CREATE index on ticket_flights (fare_conditions);
demo=# CREATE index on ticket_flights (amount);

サイズがmin_parallel_index_scan_sizeパラメータの値を超えるインデックスのみが並列処理に関与します。私たちのインデックスは適切です：

demo=# SHOW min_parallel_index_scan_size;

------------------------------

(1 row)

demo=# SELECT pg_size_pretty(pg_relation_size(indexname::regclass))
FROM pg_indexes
WHERE tablename = 'ticket_flights';

----------------

 187 MB
 180 MB
(3 rows)

先行プロセスが2人のワーカーを起動し、1つのインデックスを取得したことがわかります。

ワークプロセスの数は明示的に指定できます（もちろん、いずれの場合も、max_worker_processesを超えないmax_parallel_maintenance_workersを超えることはありません）。特に、明示的な指示は、並行性をオフにするために使用できます。

demo=# VACUUM (PARALLEL 0, VERBOSE) ticket_flights;

このトピックがさらに発展することを願っています。テーブルは複数のプロセスで並行してスキャンできます（これは元のSavadaパッチに含まれていましたが、最後のパッチには含まれていませんでした）。各インデックスも並行してクリアできます。並列処理を使用するには、自動クリーニングを教えることも必要です。
