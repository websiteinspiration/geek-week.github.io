1.ウィンドウ関数


ウィンドウ関数がおそらく最も簡単な方法です。基本的なケース（グループのないテーブル）では、日付でソートされたデータを考慮します。

order by dt

...しかし、現在の行の前の行にのみ関心があります。

rows between unbounded preceding and current row

結局のところ、これらのパラメーターとの合計が必要です。

sum(val) over (order by dt rows between unbounded preceding and current row)

完全なリクエストは次のようになります。

select s.*,
       coalesce(sum(s.val) over (order by s.dt 
                rows between unbounded preceding and current row), 
                0) as total
from test_simple s
order by s.dt;

グループの累積合計（フィールドgrp）の場合、必要な編集は1つだけです。ここで、データをグループに従って「ウィンドウ」に分割したと見なし



ます。この分離を考慮するには、キーワードを使用する必要がありますpartition by 。

partition by grp

そして、それに応じて、これらのウィンドウの量を考慮してください： 

sum(val) over (partition by grp order by dt 
                rows between unbounded preceding and current row)

次に、リクエスト全体が次のように変換されます。

select tg.*,
       coalesce(sum(tg.val) over (partition by tg.grp order by tg.dt
                rows between unbounded preceding and current row), 
                0) as total
from test_groups tg
order by tg.grp, tg.dt;

ウィンドウ関数のパフォーマンスは、DBMSの詳細（およびそのバージョン！）、テーブルのサイズ、およびインデックスの可用性によって異なります。しかし、ほとんどの場合、この方法が最も効果的です。ただし、ウィンドウ関数は、DBMSの古いバージョン（まだ使用中）では使用できません。さらに、Microsoft AccessやSAP / Sybase ASEなどのDBMSにはありません。ベンダーに依存しないソリューションが必要な場合は、代替策を検討する必要があります。

2.サブクエリ


上記のように、ウィンドウ関数はメインDBMSの非常に遅い時期に導入されました。この遅延は驚くべきことではありません。リレーショナル理論では、データは順序付けされていません。リレーショナル理論の精神の多くは、サブクエリによる解決策に対応しています。

このようなサブクエリでは、現在より前の日付（現在を含む）の値の合計を検討する必要があります。dtrow≤dtcurrentrow。

コードでは次のようになります。

select s.*,
       (select coalesce(sum(t2.val), 0)
	from test_simple t2
	where t2.dt <= s.dt) as total
from test_simple s
order by s.dt;

もう少し効果的な解決策は、サブクエリが現在の日付までの合計（ただし、それを含まない）を考慮し、それを行の値と合計することです。

select s.*,
       s.val + (select coalesce(sum(t2.val), 0)
	        from test_simple t2
	        where t2.dt < s.dt) as total
from test_simple s
order by s.dt;

複数のグループの累積結果の場合、相関サブクエリを使用する必要があります。

select g.*,
	(select coalesce(sum(t2.val), 0) as total
	 from test_groups t2
	 where g.grp = t2.grp
	          and t2.dt <= g.dt) as total
from test_groups g
order by g.grp, g.dt;

条件g.grp = t2.grpは、グループへのエントリの行をチェックします（これは、原則としてpartition by grp、ウィンドウ関数での作業に似ています）。

3.内部接続


サブクエリと結合は交換可能なため、簡単に置き換えることができます。これを行うには、自己結合を使用して、同じテーブルの2つのインスタンスを接続する必要があります。

select s.*, 
       coalesce(sum(t2.val), 0) as total
from test_simple s
inner join test_simple t2
	 on t2.dt <= s.dt
group by s.dt, s.val
order by s.dt;

ご覧のとおり、サブクエリのフィルター条件はt2.dt <= s.dt結合条件になっています。また、集計関数を使用するには、sum()日付と値でグループ化する必要がありますgroup by s.dt, s.val。

異なるグループの場合にも同じことができますgrp：

select g.*, 
       coalesce(sum(t2.val), 0) as total
from test_groups g
inner join test_groups t2
	on g.grp = t2.grp
		and t2.dt <= g.dt
group by g.grp, 
	 g.dt, 
	 g.val
order by g.grp, 
	 g.dt;

4.デカルト積


サブクエリを結合に置き換えたので、デカルト積を試してみませんか？このソリューションでは、最小限の編集のみが必要です。

select s.*, 
       coalesce(sum(t2.val), 0) as total
from test_simple s,
     test_simple t2
where t2.dt <= s.dt
group by s.dt, 
	 s.val
order by s.dt;

またはグループの場合：

select g.*, 
       coalesce(sum(t2.val), 0) as total
from test_groups g,
     test_groups t2
where g.grp = t2.grp
	and t2.dt <= g.dt
group by g.grp, 
	 g.dt, 
	 g.val
order by g.grp, 
	 g.dt;

リストされているソリューション（サブクエリ、内部結合、デカルト結合）はSQL-92およびSQL：1999に対応しているため、ほとんどすべてのDBMSで使用できます。これらすべてのソリューションの主な問題は、パフォーマンスの低下です。結果を使用してテーブルを具体化する場合、これは大きな問題ではありません（ただし、速度を向上させる必要があります）。その他の方法の方がはるかに効果的です（特定のDBMSの仕様と、すでに指定されているバージョン、テーブルサイズ、インデックスに合わせて調整されます）。

5.再帰的なリクエスト


より具体的なアプローチの1つは、共通テーブル式での再帰クエリです。これを行うには、「アンカー」、つまり最初の行を返すクエリが必要です。

select dt,
       val, 
       val as total
from test_simple
where dt = (select min(dt) from test_simple)

次にunion all、再帰クエリの結果が「アンカー」に追加されます。これを行うには、日付フィールドを使用してdt、それに1日を追加します。

select r.dt,
       r.val,
       cte.total + r.val
from cte
inner join test_simple r
     on r.dt = dateadd(day, 1, cte.dt) -- + 1   SQL Server

1日を追加するコードは普遍的ではありません。たとえば、これはr.dt = dateadd(day, 1, cte.dt)SQL Serverやr.dt = cte.dt + 1 Oracleなどです。

「アンカー」とメインリクエストを組み合わせると、最終結果が得られます。

with cte (dt, val, total)
as
   (select dt, val, val as total
    from test_simple
    where dt = (select min(dt) from test_simple)
			
    union all
			
    select r.dt, r.val, cte.total + r.val
    from cte
    inner join test_simple r
    	on r.dt = dateadd(day, 1, cte.dt) -- r.dt = cte.dt + 1  Oracle,  ..
    )
select dt, val, total 
from cte
order by dt;

グループの場合の解決策はそれほど複雑ではありません。

with cte (dt, grp, val, total)
as
   (select g.dt, g.grp, g.val, g.val as total
    from test_groups g
    where g.dt = (select min(dt) from test_groups where grp = g.grp)
			
    union all
			
    select r.dt, r.grp, r.val, cte.total + r.val 
    from cte
    inner join test_groups r
    	on r.dt = dateadd(day, 1, cte.dt) -- r.dt = cte.dt + 1  Oracle,  ..
		and cte.grp = r.grp
    )
select dt, grp, val, total 
from cte
order by grp, dt;

6.関数を使用した再帰クエリ row_number()


以前の決定はdt、1日の連続的な増加による日付フィールドの連続性に基づいていました。row_number()行に番号を付けるウィンドウ関数を使用してこれを回避します。もちろん、これは不公平です。ウィンドウ関数の代替案を検討するからです。ただし、このソリューションは一種の概念実証である場合があります。実際には、行番号（レコードID）を置き換えるフィールドがある場合があります。さらに、row_number()ウィンドウ関数（を含むsum()）の完全なサポートが導入される前に、SQL Server に関数が登場しました。

したがって、row_number()私たちと再帰クエリを行うには、2つのSTEが必要です。最初は、行に番号を付けるだけです。

with cte1 (dt, val, rn)
as (select dt,
           val,
	   row_number() over (order by dt) as rn
    from test_simple)

...そして、行番号がすでにテーブルにある場合は、それなしで実行できます。次のクエリでは、次のようになりますcte1。

cte2 (dt, val, rn, total)
as
   (select dt, val, rn, val as total
    from cte1
    where rn = 1
			
    union all
			
    select cte1.dt, cte1.val, cte1.rn, cte2.total + cte1.val
    from cte2
    inner join cte1
    	on cte1.rn = cte2.rn + 1
    )

そして、リクエスト全体は次のようになります。

with cte1 (dt, val, rn)
as (select dt,
           val,
	   row_number() over (order by dt) as rn
	from test_simple),
cte2 (dt, val, rn, total)

    from cte2

    	on cte1.rn = cte2.rn + 1
    )
select dt, val, total 
from cte2
order by dt;

...またはグループの場合：

   from test_groups),

7.オペレーターCROSS APPLY/LATERAL


積算合計を計算する最もエキゾチックな方法の1つは、ステートメントCROSS APPLY（SQL Server、Oracle）または同等のステートメントLATERAL （MySQL、PostgreSQL）を使用することです。これらの演算子はかなり遅れて登場しました（たとえば、Oracleではバージョン12c以降のみ）。また、一部のDBMS（MariaDBなど）では、まったく機能しません。したがって、この決定は純粋に審美的な関心事です。

機能的に使用するCROSS APPLYか LATERAL、サブクエリと同じ：計算結果をメインリクエストに添付します。

cross apply (select coalesce(sum(t2.val), 0) as total
             from test_simple t2
	     where t2.dt <= s.dt
) t2

...これは次のようになります：

select s.*,
       t2.total
from test_simple s
cross apply (select coalesce(sum(t2.val), 0) as total
             from test_simple t2
	     where t2.dt <= s.dt
) t2
order by s.dt;

グループの場合の解決策は同様です。

select g.*,
       t2.total
from test_groups g
cross apply (select coalesce(sum(t2.val), 0) as total
             from test_groups t2
	     where g.grp = t2.grp
	            and t2.dt <= g.dt
) t2
order by g.grp,
         g.dt;

合計：プラットフォームに依存しない主なソリューションを確認しました。しかし、特定のDBMSに固有のソリューションがあります。ここには非常に多くのオプションがあるため、最も興味深いものをいくつか取り上げます。

8.オペレーターMODEL（Oracle）


MODELOracle のオペレーターは、最もエレガントなソリューションの1つを提供します。記事の冒頭で、累積合計の一般式を調べました。

{1=1,n=1n=n−1+n,n≥2



MODELこの式を文字通り1対1で実装できます！これを行うには、まずtotal現在の行の値をフィールドに入力します

select dt,
       val,
       val as total
from test_simple

...次に、行番号を次のように計算しますrow_number() over (order by dt) as rn（または、必要に応じて、完成したフィールドに番号を付けます）。最後に、最初の行を除くすべての行にルールを導入します total[rn >= 2] = total[cv() - 1] + val[cv()]。

この関数cv() は、現在の行の値を担当します。そして、リクエスト全体は次のようになります。

select dt, val, total
from
    (select dt,
            val,
            val as total
    from test_simple) t
model
    dimension by (row_number() over (order by dt) as rn)
    measures (dt, val, total)
    rules (total[rn >= 2] = total[cv() - 1] + val[cv()])
order by dt;

9.カーソル（SQLサーバー）


累計は、SQL Serverのカーソルが役立つだけでなく、他のソリューションよりも好ましい数少ないケースの1つです（少なくともバージョン2012までは、ウィンドウ関数が表示されていました）。

カーソルによる実装は非常に簡単です。最初に、一時テーブルを作成し、メインの日付と値を入力する必要があります。

create table #temp
      (dt date primary key,
       val int null,
       total int null
       ); 

insert #temp (dt, val)
select dt, val	   
from test_simple
order by dt;

次に、更新が行われるローカル変数を設定します。

declare @VarTotal int,
        @VarDT date,
        @VarVal int;

set @VarTotal = 0;

その後、カーソルを使用して一時テーブルを更新します。

declare cur cursor local static read_only forward_only
for select dt, val from #temp order by dt;

open cur;
fetch cur into @VarDT, @VarVal;

while @@fetch_status = 0
begin
	set @VarTotal = @VarTotal + @VarVal;
	
	update #temp
	set total = @VarTotal
	where dt = @VarDT;
	
	fetch cur into  @VarDT, @VarVal;
end;

close cur;
deallocate cur;

そして最後に、望ましい結果が得られます。

select dt, val, total
from #temp
order by dt;

drop table #temp;

10.ローカル変数による更新（SQL Server）


SQL Serverのローカル変数による更新は、文書化されていない動作に基づいているため、信頼できるとは言えません。それでも、これがおそらく最速の解決策であり、興味深い方法です。

2つの変数を作成してみましょう。1つは累積合計用で、もう1つはテーブル変数用です。

declare @VarTotal int = 0;
declare @tv table
      (dt date null,
       val int null,
       total int null
       );

まず、@tvメインテーブルのデータを入力します

insert @tv (dt, val, total)
select dt, val, 0 as total
from test_simple
order by dt;

次に、以下を@tv使用してテーブル変数を更新します@VarTotal。

update @tv
set @VarTotal = total = @VarTotal + val
from @tv;

...その後、最終結果が得られます。

select * from @tv order by dt;

概要：SQLで累積合計を計算する方法のトップ10を確認しました。ご覧のとおり、ウィンドウ関数がなくても、この問題は完全に解決可能であり、解のメカニズムは複雑であるとは言えません。
