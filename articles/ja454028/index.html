Nikita Popov：型付きプロパティなど。PHP 7.4の新機能




PHP自体の最もアクティブで著名なコア開発者の1人であるニキータポポフ氏は、当然のことながらカンファレンスの見出しに参加しました。彼のレポートは、作者のコメント（およびコード例）を含む変更ログと、言語の将来への先見的な遠足の間のクロスです。Nikitaは、一貫して簡潔に、次のバージョン以降のPHPで何が待ち受けているかを説明しました。

推定タイムラインによると、バージョン7.4のアルファリリースは2019年6月6日に行われ、7.4 GAのリリースは11月21日に展開されます。さて、その1年後の2020年12月には、噂に煽られた「8」がリリースされます。

PHP 7.4で実装される主なもの：

  • 型付きプロパティ。
  • FFI-Cで外部関数を操作するための外部関数インターフェイス。
  • 矢印関数;
  • -プリロード;
  • — ??=;
  • — ;
  • — (WeakReference);
  • — … .

おそらく、次期バージョンで最も待ち望まれていた革新は、クラスプロパティの型を設定する機能です。このようなプロパティは必ず明示的に初期化する必要があることは注目に値します。そのデフォルト値が指定されていない場合、プロパティはnullになりません-いいえ、そのような状況では例外がスローされます。また、特に、型付きプロパティは、交差型言語ではまだ欠けているようなPHPへの変換に役立ちます。つまり、引数にはいくつかの型を一度に指定します。

PHP 7.4の2番目の大きな変更は、矢印関数のデビューです。これらは単一行の匿名関数です-一部の開発者は確かに構文糖質ですが、役に立たないものもありますが、同じように便利なものもあります。

オペレーターの注意を奪うことも不可能です??=それは以前は言語から欠けていました。左側のパラメーターが式に存在しないかnullの場合、右側のパラメーター値が割り当てられます。

まだ議論されている可能性のある革新のうち、ジェネリックが言及されました。しかし、ニキータはすぐに、それらの完全な実装は簡単な作業ではないと予約しました。

バージョン7.4で廃止されたと見なされている機能については、いくつかありました。 3項演算子を左から右に処理すると、履歴のゴミ箱に入れられます。G8では、この構文はエラーを生成します。連結演算子の優先度も低くなります。これを加算演算子または減算演算子と一緒に使用すると、最初に処理されます。

Dmitry Stogov：PHP 8で最も興味深いこと





Nikita Popovが始めたこのラインは、PHPコアへの著名な貢献者であるZend TechnologiesのリードエンジニアであるDmitry Stogovによって継続されました。彼は言語のもう少し遠い未来について-秘跡バージョン8.0について話しました。近年では、PHPのパフォーマンス向上に重点的に取り組み、同じように強調してストーリーを構築しています。

10年半の間、言語は「加速された」。リリース5.1および7.0では、パフォーマンスが大幅に向上しました。セブンの時代には、スムーズな成長がありました。バージョン間でコード実行の速度を上げることは、急進的ではありませんでした。ただし、PHP 8.0は開発者にギフトバッグを約束します。彼らはたくさんの面白いことを待っています。

「おもしろい」と言います-JITのことですかあんまり。はい、確かに、ジャストインタイムの動的コンパイルはバージョン8.0でパフォーマンスの向上を約束しますが、重要ではありませんが、...合成テストでのみ、実際のアプリケーションでは、スローダウンさえ可能です。ただし、改善は可能です。

それから「肉」が始まりました：Dmitryはその中でPHPとJITの作業の基本を深く掘り下げ、同時に「JITの何が問題か」を説明し、JIT、プリロードメカニズム、およびFFIの方法を一般に公開しました。 PHPコードのパフォーマンスを向上させるためにそれらが呼ばれているためです。

上記のトライアドは、PHPの新しい用途を切り開きます。 PHPで直接拡張機能を記述し、サードパーティのCライブラリを接続し、追加の作業なしでプロトタイプを作成することが可能になります...

しかし、正直なエンジニアとして、Dmitryは言語の8番目のバージョンで利用できる目まいがする可能性を概説しただけでなく、その欠点についても警告しましたが、「非最適性」でした。もちろん、プリロードスクリプトがPHPの起動時に読み込まれるのは素晴らしいことですが、再起動しないとスクリプトを置き換えることはできず、一部のスクリプトは意図したとおりにプリロードで機能しない可能性があります。もちろん、FFIのおかげで、Cで定義されたデータ構造を操作でき、API自体がシンプルで快適であるという点は素晴らしいことです。FFIだけが、1000の新しい方法で新たな道を切り開き、JITなしのパフォーマンスは比較的低くなります。

同時に、G8の基盤にあるテクノロジーの組み合わせには、マイナスよりもプラスの方が多いです。さらに、Dmitryがレポートで言及した機能の多くは、バージョン7.4ですでに感じられます。
