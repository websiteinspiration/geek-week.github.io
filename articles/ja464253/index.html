日時の使用をやめる

特にPHPコースのバックエンド開発者の学生のために、人気のあるツールの副作用に関する興味深い記事の翻訳を用意しました。




PHPで日付と時刻を操作すると、コードに予期しないバグが発生するため、煩わしい場合があります。

$startedAt = new DateTime('2019-06-30 10:00:00');

$finishedAt = $startedAt->add(new DateInterval('PT3M')); 

var_dump($startedAt->format('Y-m-d H:i:s')); //2019-06-30 10:03:00 
var_dump($finishedAt->format('Y-m-d H:i:s')); //2019-06-30 10:03:00

両方の関数$startdateと$finishdate3分間のラッシュ。これは、などのメソッド add ()、sub()またはmodify()それらが呼び出されるDateTimeオブジェクトを返す前に変更するためです。もちろん、上記の例は望ましくない動作を示しています。

参照するオブジェクトを操作する前にコピーすることで、このエラーを修正できます。次に例を示します。

$startedAt = new DateTime('2019-06-30 10:00:00');

$finishedAt = clone $startedAt;
$finishedAt->add(new DateInterval('PT3M'));

PHPコードのクローンに出くわすたびに、誰かの失敗したコードアーキテクチャのハックのようなにおいがします。この場合、動作の変更を回避するためにクローンを使用しましたが、同時にコードが見苦しくなり、多くの不要なノイズが発生しました。

または、元のDateTimeインスタンスをDateTimeImmutable次のように変換することで問題を解決できます。

$startedAt = new DateTime('2019-06-30 10:00:00');

$finishedAt = DateTimeImmutable::createFromMutable($startedAt)->add(new DateInterval('PT3M'));

最初から使ってみませんDateTimeImmutableか？

DateTimeImmutableの妥協のない使用


日付/時刻オブジェクトを渡すときに予期しない変更を防ぐためにセキュリティメソッドを手動で適用する代わりにDateTimeImmutable、メソッドをカプセル化するものを使用して、コードの信頼性を高めます。

$startedAt = new DateTimeImmutable('2019-06-30 10:00:00');

$finishedAt = $startedAt->add(new DateInterval('PT3M'));

var_dump($startedAt->format('Y-m-d H:i:s')); //2019-06-30 10:00:00 
var_dump($finishedAt->format('Y-m-d H:i:s')); //2019-06-30 10:03:00

ほとんどの場合、日付の概念は値と見なされ、日付とその値を比較し、日付を変更すると、別の日付になります。これらはすべてValue Objectの概念と完全に相関しており、Value Objectの重要な特性の1つは不変であることです。

詳細なコーディングスタイル


不変性では、オブジェクトDateTimeImmutableを操作するたびにオブジェクトを明示的に再割り当てする必要があります。値が変更されることはなく、代わりにコピーが返されるためです。何年にもわたってDateTimeを扱い、可変性は多くの命令型プログラミング言語のデフォルト値であるため、それを使用する習慣を取り除き、再マッピングを促進する新しいスタイルのコードを作成することは困難です。

$this->expiresAt = $this->expiresAt->modify('+1 week');

PHPStanやその拡張機能の1つ などの統計分析ツールは、割り当てを省略して誤って使用した場合に警告を表示しDateTimeImmutableます。

ただし、たとえばプリミティブの値に対して算術演算を実行すると、変動に対するそのような認識バイアスが抑制されます$a + 3;。これ自体は明らかに再割り当てを欠い無意味な文として認識されています：$a = $a + 3;か$A += 3;。値オブジェクトの場合、このようなものを使用するのは素晴らしいことですよね？

一部のプログラミング言語には、演算子のオーバーロードと呼ばれる構文上の砂糖があります。を使用すると、カスタムタイプとクラスに演算子を実装して、プリミティブデータ型と同じように動作させることができます。PHPが他のプログラミング言語からこのトリックを借用したかどうかは気にしません。次のように書くことができます。

$this->expiresAt += '1 week';

ワンタイム計算


DateTime計算は同じ実行領域内で実行されるため、 パフォーマンスの観点から使用する方が良いと主張する人もいます。ただし、何百もの操作を実行する必要がなく、古いオブジェクトへのリンクがDateTimeImmutableガベージコレクターによって収集されることを覚えている場合、これは許容できます。ほとんどの場合、実際には、メモリの消費は問題になりません。

日付/時刻ライブラリ


Carbonは、PHPで日付/時刻APIを拡張し、豊富な機能セットを追加した非常に人気のあるライブラリです。より正確には、可変クラスのAPIを拡張しますDateTime。その使用は、この記事のトピックに反しています。

したがって、Carbonでの作業を楽しんでも不変性を好む場合は、Chronosに慣れることをお勧めします。これは、元々Carbonに基づいていたスタンドアロンライブラリで、不変のデフォルトの日付/時刻オブジェクトの提供に特に注意を払っていますが、必要に応じて変更可能なオプションも含まれています。
編集（07/05/2019）：Carbonには不変の日付/時刻オプションがあり、これは大きなプラスであることが判明しました。ただし、私がChronosを好む理由は、Carbonとは異なり、コードとドキュメントの両方でデフォルトの不変性を奨励および促進するためであり、これらはこの記事のコンテキストにおいて決定的な要素です。
