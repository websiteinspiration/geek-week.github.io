ゲーム「攻撃51％」：Obyteプラットフォームでシンプルなスタンドアロンゲームを作成する



前回の記事「自律エージェント」またはObyteオープン暗号プラットフォームでコードを実行する際に、自律エージェントとは何かについて話し、それらをイーサリアムスマートコントラクトと比較しました。 51％攻撃の例を使用して、最初の自律エージェント（AA）を作成しましょう。そして記事の最後で、それを改善する方法を分析します：プレーヤーを損失/資金の損失から保護する方法と、アルゴリズムを改善してゲームの結果に大量の堆積物を持つ「クジラ」の影響を減らす方法。

両方のスタンドアロンエージェントのオリジナルは、テンプレートの形でオンラインのOscriptコードエディタでいつでも利用できます。ドロップダウンメニューから「51％攻撃ゲーム」と「資金調達プロキシ」を選択するだけです。

OscriptでAAを書き始める前に、ドキュメントの「Getting Started Guide （eng）」を読んで、AAを書く基本原則にすぐに慣れることを強くお勧めします。

ゲームの本質


集められた資金のプール全体を集める権利を求めて、いくつかのチームが同時に競争します。チームプレーヤーはデポジットを行うことで、プールの総量を増やします。調達された資金の少なくとも51％を預金し、少なくとも1日はリーダーとして差し出したチームが勝利します。プールは、行われた貢献に比例して勝利チームのプレイヤー間で分配されます。これにより、各参加者は投資を倍増させることができます。

チームのいずれかがすべての資金の51％以上を収集するとすぐに、それは以前に勝者として宣言され、その参加者はもはや預金を行うことができません。しかし、他のすべてのチームには、優勝チームを追い抜くための24時間があります。これが発生するとすぐに、追い越しチームが勝者となり、タイマーが再びカウントダウンを開始します。

私たちの実装は、バイトと引き換えに1対1の比率で各預金者に「シェア」を発行します。これは、チームが勝った場合、参加者はウォンプールのシェアと交換できます。株式はObyteプラットフォームの資産であり、各チーム専用にリリースされています。彼らは、他の資産と同様に、他の人々に彼らの潜在的な賞金を売却して、譲渡/取引することができます。株価は、チームが勝つ可能性を評価する市場によって異なります。

誰でもチームを作成できます。チームの作成者は、勝利時にコミッションを設定することができます。勝利した場合は、各チームメンバーから請求されます。

また、チームの作成者に代わって、ゲームの2番目のAAを適用します。チームの作成者は、必要な金額を「クラウドに入金」し、それが達成された場合のみ（この例では、ゲームプールの> 51％に達したとき）、すべての資金をアドレスAAに送信しますゲーム、そうでなければお金は自由に取り戻すことができます。

Oscriptコードの作成


このAAのアドレスにトランザクションが到着するたびにAAコードが呼び出されることを思い出してください。トランザクションをトリガーします。実際、AAは、入力（トリガートランザクションのデータと状態変数に格納されているAA自体の現在の状態）に応答して出力（別の「応答」トランザクション）を生成するか、その状態を変更するコードです。私たちの仕事は、入力データに対するAA反応のルールをプログラムすることです。Obyteのトランザクションはメッセージのセットであり、ほとんどの場合、「支払い」メッセージまたは「データ」メッセージなどです。

    }`,
    messages: {
		cases: [
		]
    }
}

文字列$ bFinished = var ['finished']; 状態AAから終了した変数を読み取ります。
得られたメッセージのアレイが囲まする場合：{}ブロック通常と同様であり、スイッチ/ケース/デフォルトのブロック、子ブロック場合、一つだけのメッセージが選択され、または全く場合における条件に基づいた場合にブロックが返さ真、それはないであろう最後に選択されたメッセージ。
 

チームを作る


したがって、最初のブロックはトランザクションを処理して新しいコマンドを作成します。
 
// create a new team; any excess amount is sent back
if: `{trigger.data.create_team AND !$bFinished}`,
init: `{
	if (var['team_' || trigger.address || '_asset'])
		bounce('you already have a team');
	if (trigger.output[[asset=base]] < $team_creation_fee)
		bounce('not enough to pay for team creation');
}`,
messages: [
	{
		app: 'asset',
		payload: {
			is_private: false,
			is_transferrable: true,
			auto_destroy: false,
			fixed_denominations: false,
			issued_by_definer_only: true,
			cosigned_by_definer: false,
			spender_attested: false
		}
	},
	{
		app: 'payment',
		if: `{trigger.output[[asset=base]] > $team_creation_fee}`,
		payload: {
			asset: 'base',
			outputs: [
				{address: "{trigger.address}", amount: "{trigger.output[[asset=base]] - $team_creation_fee}"}
			]
		}
	},
	{
		app: 'state',
		state: `{
			var['team_' || trigger.address || '_founder_tax'] = trigger.data.founder_tax otherwise 0;
			var['team_' || trigger.address || '_asset'] = response_unit;
			response['team_asset'] = response_unit;
		}`
	}
]

預金を受け入れる


チームの作成が完了したら、次のブロックに進みます-チームメンバーからの預金の処理。
 
// contribute to a team
if: `{trigger.data.team AND !$bFinished}`,
init: `{
	if (!var['team_' || trigger.data.team || '_asset'])
		bounce('no such team');
	if (var['winner'] AND var['winner'] == trigger.data.team)
		bounce('contributions to candidate winner team are not allowed');
}`,
messages: [
	{
		app: 'payment',
		payload: {
			asset: `{var['team_' || trigger.data.team || '_asset']}`,
			outputs: [
				{address: "{trigger.address}", amount: "{trigger.output[[asset=base]]}"}
			]
		}
	},
	{
		app: 'state',
		state: `{
			var['team_' || trigger.data.team || '_amount'] += trigger.output[[asset=base]];
			if (var['team_' || trigger.data.team || '_amount'] > balance[base]*0.51){
				var['winner'] = trigger.data.team;
				var['challenging_period_start_ts'] = timestamp;
			}
		}`
	}
]

ここで出会う新しいものから、バイト単位でのデポジットと引き換えに、選択したチームのアセットのトークンをその参加者に発行します。 

asset: `{var['team_' || trigger.data.team || '_asset']}`,
outputs: [{address: "{trigger.address}", amount: "{trigger.output[[asset=base]]}"}

思い出すと、状態変数「team_」|| trigger.data.team || チームの作成段階で「_asset」を保存しました。これには、このチームのアセットを作成したユニットのハッシュ、つまりこのアセットの名前が格納されています。
 
同じブロックで、主な状態が51％確認されます。

if (var['team_' || trigger.data.team || '_amount'] > balance[base]*0.51){
	var['winner'] = trigger.data.team;
	var['challenging_period_start_ts'] = timestamp;
}

このトリガートランザクションの後、指定したチームの残高が51％を超えると、チームが勝者と宣言され、現在のUNIXタイムスタンプが記録されます（タイマーを開始）。
 
このタイムスタンプは、ゲームを終了しようとするトリガートランザクションを受け取ったときに、3番目のブロックでチェックされます。

// finish the challenging period and set the winner
if: `{trigger.data.finish AND !$bFinished}`,
init: `{
	if (!var['winner'])
		bounce('no candidate winner yet');
	if (timestamp < var['challenging_period_start_ts'] + $challenging_period)
		bounce('challenging period not expired yet');
}`,
messages: [
	{
		app: 'state',
		state: `{
			var['finished'] = 1;
			var['total'] = balance[base];
			var['challenging_period_start_ts'] = false;
			response['winner'] = var['winner'];
		}`
	}
]

賞金を支払う


そして最後のブロック、最も楽しいのは、勝者への全預金の支払いです。

// pay out the winnings
if: `{
	if (!$bFinished)
		return false;
	$winner = var['winner'];
	$winner_asset = var['team_' || $winner || '_asset'];
	$asset_amount = trigger.output[[asset=$winner_asset]];
	$asset_amount > 0
}`,
init: `{
	$share = $asset_amount / var['team_' || $winner || '_amount'];
	$founder_tax = var['team_' || $winner || '_founder_tax'];
	$amount = round(( $share * (1-$founder_tax) + (trigger.address == $winner AND !var['founder_tax_paid'] ? $founder_tax : 0) ) * var['total']);
}`,
messages: [
	{
		app: 'payment',
		payload: {
			asset: "base",
			outputs: [
				{address: "{trigger.address}", amount: "{$amount}"}
			]
		}
	},
	{
		app: 'state',
		state: `{
			if (trigger.address == $winner)
				var['founder_tax_paid'] = 1;
		}`
	}
]

ここで初期化ブロックは興味深いものであり、必要な値を事前に計算します。

 $share = $asset_amount / var['team_' || $winner || '_amount'];
$founder_tax = var['team_' || $winner || '_founder_tax'];
$amount = round(( $share * (1-$founder_tax) + (trigger.address == $winner AND !var['founder_tax_paid'] ? $founder_tax : 0) ) * var['total']);

チームのすべてのメンバーは、その作成者を除き、最初の貢献に比例した金額（トリガートランザクションで送信されたアセットトークンと引き換えに1行1バイト）が支払われます。作成者にはコミッションも支払われます（トリガートランザクションを送信した人のアドレスが、勝利チームのトリガーの作成者のアドレスと等しいことが確認されます。トリガー= address $ =勝者）。コミッションは1回だけ支払う必要があり、作成者は無限に多くのトリガートランザクションを送信できるため、フラグを状態AAに保存することを忘れないでください。

クラウドファンディング


そして今、約束された最適化に。現在の実装では、各プレーヤーはすぐにゲームのAAアドレスに送金し、チームを示します。同時に、チームがリーダーになることはなく、資金はすでに送金されています。お金を集めるプロセスを最適化し、決して勝者にならないチームにお金を送るという状況を回避できます。これは、2番目のAAを使用して行うことができます。資金のいわゆるクラウドファンディングを調整するには、ゲーム内の資金の51％に相当する資金調達の動的目標を設定します。

Oscriptでは、他のAAの状態を読み取ることができるため、クラウドファンディングAAに動的な目標を設定する機会があります。

アルゴリズムは次のようになります。チームの作成者は、ゲームのアドレスではなく、クラウドファンディング機能を実装するエージェントのアドレスに送金するようプレーヤーに要求します。このエージェントは、収集されたバイト数を自宅で保存し、ゲーム内のバイト数の51％以上を収集した場合にのみそれらをゲームに送信します。そしてすぐにこのチームがリーダーになります。必要な金額が集まらない場合、お金はプレイヤーに返却されます。資金調達の段階では、プレーヤーはチームのゲームトークンではなく、クラウドファンディングトークンを受け取ります。クラウドファンディングトークンは、成功すると払い戻しまたはゲームトークンと交換できます。

次の記事では、この機能を実装します。

バイトではなく、認証を収集します
