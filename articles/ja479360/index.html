数回クリックするだけで、Angularでオープンソースのプロジェクトを作成できます

私は最近、オープンソースに公開する前にnpmパッケージを素早く構成する方法に関する記事を公​​開しました。その中で、バージョニングのセットアップ、CI、ワークフローの構築、デモページへの便利なデプロイを整理しました。その記事のすべてのヒントは、Angularライブラリに適用されます。しかし、それでも、Angularの詳細では、すぐに方向転換することはできません。

Angularの6番目のバージョン以降、Angularワークスペースを使用しています。これはプロジェクトを整理するための優れたツールですが、オープンソースの初期設定を複雑にする可能性があります。それらを予測し、プロセスを自動化することにしました。



ライブラリのAngular Workspaceを整理する方法


ここに空のワークショップがあります。その中にライブラリ自体とデモ用のアプリケーション用のライブラリを生成できます。便利なプロジェクト構造が得られ



ます。デモに加えて、たとえば、ライブラリを使用してテストを実行するe2eテストを含むアプリケーションを追加できます。

ワークスペースのおかげで、1つの場所から複数のライブラリを管理できます。

  • 一般的なpackage.jsonには、個々のライブラリを操作するためのコマンドのセットが含まれています。たとえば、複数のパッケージの同時リリース、ビルド：すべてのコマンド、またはワークスペース全体のカバレッジ分析。
  • Angular Schematicsを使用すると、package.jsonに新しいコマンドを簡単に自動化できます。これにより、人的要因を最小限に抑えることができるため、複数のパッケージで構成されるプロジェクトに役立ちます。

デモをカスタマイズする


デモもワークスペースの一部ですが、アプリケーションタイプが含まれています。ライブラリのデモの最初のルールは、現在のバージョンのオンラインIDEで簡単に実行できることです。

これにより問題が発生する可能性があります。オンラインIDEは通常のAngularアプリケーションをデプロイできますが、ワークスペースには対応できません。彼らは自分でAngularアプリケーションへのエントリポイントを見つけようとし、準備したngサーバーを実行しないようにします。これは、IDEの起動方法によって異なりますが、さまざまな理由で見つかりません。

この問題は、すべてのオンラインIDEですぐに解決されます。angular.jsonとpackage.jsonをデモと一緒にプロジェクトフォルダーに配置します。まるで、独立したプロジェクトangular.jsonを設定しているよう

です。
{
    "$schema": "./node_modules/@angular/cli/lib/config/schema.json",
    ...
    "projects": {
        "demo": {
            "architect": {
                "build": {
                    ...,
                    "options": {
		   ...
                        "tsConfig": "tsconfig.demo.json",
                    }
...

tsconfig.demo.json
{
   "extends": "./tsconfig.json",
   "compilerOptions": {
       "outDir": "../out-tsc/app",
       "types": [],
       "paths": {}
   }
}

TS構成は変わりませんが、ワークスペース全体に依存せずにプロジェクトが開始するように、パスは空のオブジェクトで再定義されます。

デモアプリケーションは2つのバリエーションで開始できることがわかりました。

最初のオプションは開発用のデモ

で、通常はワークスペースのルートフォルダーからプロジェクトを起動します。これは、一般的な設定でコンパイルされ、隣接するフォルダーのライブラリからコンパイルされます。したがって、ライブラリを開発し、すぐにデモの変更を確認できます。

2番目のオプションは、オンラインIDEで

表示するためのデモです。デモのあるフォルダーからプロジェクトを開始します。次に、ローカルのpackage.jsonとangular.jsonが使用されます。デモは個別に起動し、NPMからライブラリの最新バージョンをインストールします。

オンラインIDEのデモにアクセスすると、実際のバージョンのパッケージを試すことができます。次の形式で、stackblitzまたはcodesandboxへのリンクを提供します。

https://stackblitz.com/github/[User|Organization]/[Repository]/tree/master/projects/demo
https://codesandbox.io/s/github/[User|Organization]/[Repository]/tree/master/projects/demo


ターンキーソリューション


私たちはすでに完全なスターターを収集し、それをgithubに置いています。

彼は、回路図を使用して新しいライブラリを構築し、package.jsonでコマンドを生成および更新する方法を知っています。これは、新しいライブラリと、すべてのユーザーを同時に操作するための両方です。たとえば、いつでも1つのコマンドでNPM内のすべてのライブラリーを公開したり、更新時にバージョンを自動的に同期したりできます。そして、カバーオールチームは各パッケージを個別に調べて、カバレッジに関するデータを収集し、その構造を維持


します。スターターのプロジェクトは、Travisで構成されたCIですぐにわかり、デモを公開する準備ができています。それでも、SSRを動作させるためのすべてのコマンドがあります。これにより、サーバーレンダリングモードでライブラリを確認できます。CODE_OF_CONDUCT.md

が含まれ、入力されている、およびgithub上の課題とプール要求のテンプレートと同様に。何も含める必要はありません。githubが自動的に取得します。

このスターターに基づいて、2つのライブラリが既に作成されています。

  • ng-dompurify（github、ハブの記事）。
  • ng-polymorpheus（github、ハブの記事）。

ライブラリを作成して自動化し、スターターを補完するためにライブラリを作成するときに発生するすべての問題に気づきます。問題が発生した場合、または不明な点がある場合は、プロジェクトのgithubの「Issues」に必ず書き込んでください。
