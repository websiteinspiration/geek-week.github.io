スケジュールされたハードウェアリソース管理

この記事では、絶対時間を参照してハードウェアリソースを管理するための自動化システムの実装例（電磁リレーを例として使用）について説明します。このようなシステムは、さまざまな機器のテスト自動化の問題を解決するのに非常に役立ちます。

たとえば、一部のテストラボでは、絶対スケールを参照して厳密に設定された時点で（たとえば、月曜日の午前10時に）、調査/テスト済みデバイスの電源リセットを自動化する必要があります。ただし、電源管理操作を実行する可能性に関する決定は、調査中のデバイスの他のハードウェアリソースの現在の状態（たとえば、出力GPIOラインの1つまたは別のレベル）の影響を受けるため、タスクは複雑です。

後者の状況では、ソリューションが多少複雑になり、リレー、リアルタイムクロック、入力GPIOラインなどの問題を解決するために必要なハードウェアリソースがサポートされている外部ハードウェアモジュールの使用を考えさせられます。

Arduinoプラットフォームを使用して行った最初の実験。しかし、このソリューションを安定した品質で大量にコピーする必要が生じたとき、彼らは既製のソリューションに目を向け始めました。その結果、イーサネット制御プロセスを監視する機能を備えた外部Laurent-5ハードウェアモジュールを使用してシステムを調整しました。



具体的には、テストしたデバイスの電源を毎日厳密に午前7時に厳密にリセットする必要がありました。ただし、デバイスが重要な動作を続けている場合は、リセットを実行しないでください。この場合、デバイスの出力ディスクリートGPIOラインに高ロジックレベル（+3.3 V）が設定されます。

Laurent-5モジュールを次のようにテストデバイスに接続しました。デバイスレディ信号が入力ディスクリートラインIO_1に送られました。そして、デバイスの電源は、通常閉じたリレー接点RELE_1を介して転送されました。リレーがオンになると、被試験デバイスへの電源が遮断されます。



システムを構成するには、まず、Laurent-5「入力」モジュールのGPIO IO_1の方向を変更する必要があります。設定する最も簡単な方法は、Webインターフェイスを使用することです（デフォルトのアドレスは192.168.0.101です）。メインコントロールパネルの「General Lines IO1-IO8」セクションに移動します。



ラインIO_1の「矢印」をクリックし、このGPIOラインの方向を「オン」状態に変更して、テスト中のデバイスの「準備」ラインの状態を分析します。



次に、「準備」ライン分析の自動化に役立ち、リレーを制御する論理CATルールを作成します。

CATセクションに移動し、[新しいイベントを作成]ボタンをクリックします。新しい論理ルールにID = 1が割り当てられるウィンドウが表示されます



。RTCイベントのタイプを選択します-タスクは指定した時間に実行されます。



イベント設定では、応答時間を示します-毎日午前7時。



Keコマンドを使用してこのイベントの発生に応答して、さらに作成するCATイベント2、3、および4の操作を有効にします。デバイスの信号の「準備」を分析し、準備ができていない場合に電源リセットを回避するには、追加の論理ルールが必要です。



わかりやすくするために、この論理ルールに記号名を付けましょう。



その結果、ID = 1の新しいイベントが論理ルールのリストに表示されます。



次のID = 2の論理ルールを追加します。これは、1秒あたり1回の頻度でタイマーで実行されます。





論理ルールが機能するために満たす必要がある追加の条件を示します。つまり、低論理レベルの必要性をIO_1に設定して、デバイスが電源リセットの準備ができていることを通知します。



すべての条件が満たされた場合、イベント2、3、4をオフにします。イベント3（下記参照）の応答カウンターをリセットし、リレーRELE_1を4秒間オンにした後、自動的に元の（オフ）状態に戻ります。



ただし、デバイスが「フリーズ」し、アラームが常に鳴る場合はどうすればよいですか？このために、ID = 3および4のイベントを使用します。ここで、指定された時間内にデバイスがハードウェアリセットの準備ができていることを信号で伝えなかった場合に、アラームメッセージの送信を伴うウォッチドッグタイマーの類似を実装します。

1秒に1回の応答頻度を持つ通常のタイマーに従って、ID = 3のイベントを作成してみましょう。このイベントは実際には何もせず、空の$ KEコマンドを送信するだけです。ただし、操作ごとに、このイベントの操作のカウンターは増加します。 ID = 4の論理ルールを使用して、この値を監視し、特定のしきい値（たとえば、5分間に相当する300操作）を超えた場合、操作を停止し、失敗した操作の数を後で分析するためにプログラム変数VAR_1の値をインクリメントします。

全体として、一連の論理ルールは次のようになります。システム全体を起動するには、ID = 1のイベント処理を有効にするだけで十分です。



そして、次のようになります。毎日07:00に、ID = 1の論理ルールがトリガーされます。この場合、ID 2、3、および4のイベント処理が反応として含まれます。テスト対象のデバイスが電源をリセットする準備ができている場合（論理レベル0〜信号線）-ルールID = 2のフレームワーク内で、イベント2〜4の処理がオフになり、何らかの理由で3番目のルールの操作カウンターがリセットされ、リレーが一時的にオンになってデバイスがリセットされます。

並行して、ガードタイマーのカウントダウンを1秒に1回のクロックで開始します。ID = 4ルールのフレームワーク内でウォッチドッグタイマーの値をチェックすることで、プログラム変数VAR_1をインクリメントすることで、その日の待機をクラッシュさせ、操作全体の失敗を通知できます。その値は、その後の分析のためにTCP / HTTP経由で要求できます。
