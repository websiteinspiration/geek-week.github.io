電報はHTTPSに変装することを学びました

Telegramクライアントのコードで、 HTTPS（TLS + HTTP / 2.0）に偽装する機能が追加されました。



この機能を使用するために、新しいシークレットプレフィックス「ee」を追加しました。さらに、16進数に加えて、プロキシサーバーアドレスの秘密をbase64としてエンコードする機能を追加しました。

Telegramクライアントとプロキシサーバー間のプロトコルでは、TCPを介したカプセル化の別のレイヤーが追加されました。 TCPを介してデータを送信する代わりに、データは次の形式のTLSレコードにラップされます。



作業の最初に、TLSハンドシェイクエミュレーションステップが追加されました。クライアントからプロキシサーバーへのパッケージの構造は次のとおりです。



ほとんどすべてのフィールドはTelegramクライアントには意味がなく、TLSのふりをするためにのみ必要です。最も重要な関数は、HMACの結果が配置されるランダムフィールドです。共有シークレットとパッケージ内のデータから。これにより、クライアントはシークレットを知っていることを証明できます。また、クライアントは、ランダムフィールドの最後の4バイトをunixtime形式の時刻で混乱させます。これにより、プロキシサーバーは、パケットがいつ生成されたかを判別できます。これは、リプレイ攻撃から保護するのに役立ちます。パケットがずっと前または将来生成された場合、プロキシサーバーはすぐにそれをドロップできます。

クライアントが接続すると、プロキシサーバーは送信されたHMACをチェックします。計算されたものと一致する場合、プロキシは次の構造のパケットで応答します。ランダム



フィールドこれもランダムではありませんが、共有シークレットからのHMACとパケット内のデータの結果であり、HMACを計算するときに、クライアントが送信したランダムな値はパケット自体のデータに起因します。データ自体を送信するとき、最初のメッセージはクライアントによって無視されます。これにより、ランダムな長さのデータを送信して、検出をさらに複雑にすることができます。
