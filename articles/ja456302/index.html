SASS（$変数）よりもCSS（-変数）を好むのはなぜですか？

2か月前のCodyHouse Frameworkのリリース以来、この環境ではSASSを使用していますが、SASS変数ではなくCSS変数を選択した理由を多くのユーザーから尋ねられました。この記事では、カスタムプロパティを使用する利点と、それらがワークフローで重要になった理由について説明します。



変数の定義


この記事では、カスタムCSSおよびSASSプロパティ（またはその他のCSSプリプロセッサ）の基本を理解していることを前提としています。そうでない場合は、基本的な例から始めましょう：

SCSSの場合：

$color-primary: hsl(220, 90%, 56%);

.link {
  color: $color-primary;
}

CSS：

:root {
  --color-primary: hsl(220, 90%, 56%);
}

.link {
  color: var(--color-primary);
}

カスタムプロパティを使用すると、CSSの拡張機能を使用せずに変数を定義できます（つまり、SASSを使用する必要はありません）。

彼らは同じですか？あんまり！SASS変数とは異なり、ユーザープロパティは1）宣言された要素に限定されます。2）カスケード、3）JavaScriptで制御できます。これらの3つの機能は、新しい可能性の世界全体を開きます。いくつかの実用的な例を紹介しましょう！

カラーテーマを作成して適用する


SASS変数を使用して2つの（簡略化された）カラーテーマを作成する方法の例を次に示します。

$color-primary: blue;
$color-text: black;
$color-bg: white;
/*  */
$color-primary-invert: red;
$color-text-invert: white;
$color-bg-invert: black;

.component {
  color: $color-text;
  background-color: $color-bg;

  a {
    color: $color-primary;
  }
}

.component--dark {
  color: $color-text-invert;
  background-color: $color-bg-invert;

  a {
    color: $color-primary-invert;
  }
}

上記の例では、「デフォルト」のテーマと「暗い」テーマがあり、背景とテキストの色が反転しています。暗いトピックでは、色変数を使用した各プロパティを調べ、それらを新しい変数で更新する必要があることに注意してください。

単純化された（非現実的な）例に固執する限り、問題は発生しません。しかし、要素が多いコンポーネントがある場合はどうでしょうか。色の変数が使用されているすべてのプロパティを書き換え、変数を新しいものに置き換える必要があります。また、メインコンポーネントを変更する場合は、すべての修飾子を再確認する必要があります。はい...それほど便利ではありません！

フレームワークを作成するときに、CSS変数に基づいた別のアプローチを思いつきました。まず、色の変数を定義しましょう：

:root, [data-theme="default"] {
  --color-primary: blue;
  /*   */
  --color-bg: white;
  --color-contrast-lower: hsl(0, 0%, 95%);
  --color-contrast-low: hsl(240, 1%, 83%);
  --color-contrast-medium: hsl(240, 1%, 48%);
  --color-contrast-high: hsl(240, 4%, 20%);
  --color-contrast-higher: black;
}

[data-theme] {
  background-color: var(--color-bg);
  color: var(--color-contrast-high);
}

[data-theme="dark"] {
  --color-primary: red;
  /*   */
  --color-bg: black;
  --color-contrast-lower: hsl(240, 6%, 15%);
  --color-contrast-low: hsl(252, 4%, 25%);
  --color-contrast-medium: hsl(240, 1%, 57%);
  --color-contrast-high: hsl(0, 0%, 89%);
  --color-contrast-higher: white;
}

注：上記の例では、data- *属性を使用してカラーテーマを適用していますが、これはCSS変数およびSASS変数とは関係ありません。さらに、「コントラストのレベル」に基づいた命名法を使用して、ニュートラル値のスケールを決定しました。

重要な点は、2番目、3番目、4番目...（暗い）テーマに対して新しい色変数を作成する必要がないことです。 SASSとは異なり、既存のカスタムプロパティの値を上書きできます。

コンポーネントに色変数を適用する方法は次のとおりです。

.component {
  color: var(--color-contrast-higher);
  background-color: var(--color-bg);
  border-bottom: 1px solid var(--color-contrast-low);

  a {
    color: var(--color-primary);
  }
}

コンポーネントのダークバージョンはどうですか？追加のCSSは必要ありません。変数をオーバーライドして置き換えないため、コンポーネントが最初に作成されたときにのみ、色に正しい変数を適用する必要があります。_colors.scssファイルでカラーテーマを設定し、カラー変数をコンポーネントの要素に適用した後、コンポーネントがどれほど複雑になるかは関係ありません。

非常に簡単な方法でカラーテーマを適用できます。

<section data-theme="dark">
  <div class="component">
    <div class="child" data-theme="default"></div>
  </div>
</section>

上記の例では、暗い色のテーマをセクションに適用し、デフォルトの色のテーマを.child要素に適用しました。そうです、色のテーマに投資できます！

これは、CSSプロパティを使用することで可能になります。これにより、このような興味深いことを最短時間で行うことができました。

フォントスケール管理


モジュラースケールは、活版印刷要素に適用される調和のとれた（次元の）値のセットです。SASS変数を使用してSCSSでフォントサイズを設定する方法は次のとおりです。

$text-xs: 0.694em;
$text-sm: 0.833em;
$text-base-size: 1em;
$text-md: 1.2em;
$text-lg: 1.44em;
$text-xl: 1.728em;

標準的なアプローチは、上記の例のように、サードパーティツールを使用してフォントスケールを作成（または数学演算を実行）してから、値をスタイルにインポートすることです。

フレームワークを作成するときに、スケール式全体を_typography.scssファイルに含めることにしました。CSS変数を使用してフォントサイズを設定する方法は次のとおりです。

:root {
  //   
  --text-base-size: 1em;
  
  //  
  --text-scale-ratio: 1.2;
  --text-xs: calc((1em / var(--text-scale-ratio)) / var(--text-scale-ratio));
  --text-sm: calc(var(--text-xs) * var(--text-scale-ratio));
  --text-md: calc(var(--text-sm) * var(--text-scale-ratio) * var(--text-scale-ratio));
  --text-lg: calc(var(--text-md) * var(--text-scale-ratio));
  --text-xl: calc(var(--text-lg) * var(--text-scale-ratio));
  --text-xxl: calc(var(--text-xl) * var(--text-scale-ratio));
  --text-xxxl: calc(var(--text-xxl) * var(--text-scale-ratio));
}

このアプローチの利点は何ですか？これにより、-text-base-size（ベースフォントのサイズ）と--text-scale-ratio（スケールファクター）の2つの変数のみを編集して、印刷システム全体を制御できます。

「変数sass»を使用して同じことを行うことはできませんか？メディアクエリの特定の時点でタイポグラフィを変更したい場合は、次のようにしません。

:root {
  @include breakpoint(md) {
    --text-base-size: 1.25em;
    --text-scale-ratio: 1.25;
  }
}

このスニペットは、私たちの柔軟なアプローチの要です。相対的なEms単位を使用するため、-text-base-size（ベースフォントのサイズ）を変更すると、タイポグラフィとスペースの両方に影響します。その結果、実質的にコンポーネントレベルでメディアクエリを設定する必要なく、すべてのコンポーネントのサイズを変更するシステムが得られます。

距離スケール制御


間隔スケールはタイプスケールと同等ですが、空間値に適用されます。繰り返しになりますが、構造にスケール式を含めることで、間隔システムを制御して応答性を高めることができました。

:root {
    --space-unit:  1em;
    --space-xxxxs: calc(0.125 * var(--space-unit)); 
    --space-xxxs:  calc(0.25 * var(--space-unit));
    --space-xxs:   calc(0.375 * var(--space-unit));
    --space-xs:    calc(0.5 * var(--space-unit));
    --space-sm:    calc(0.75 * var(--space-unit));
    --space-md:    calc(1.25 * var(--space-unit));
    --space-lg:    calc(2 * var(--space-unit));
    --space-xl:    calc(3.25 * var(--space-unit));
    --space-xxl:   calc(5.25 * var(--space-unit));
    --space-xxxl:  calc(8.5 * var(--space-unit));
    --space-xxxxl: calc(13.75 * var(--space-unit));
}

@supports(--css: variables) {
  :root {
    @include breakpoint(md) {
      --space-unit:  1.25em;
    }
  }
}

このアプローチは、上で説明したタイポグラフィ手法と組み合わせると特に強力になります。ほんの数行のCSS行でレスポンシブなコンポーネントが得られます。



この間隔システムでEmsモジュールを使用することで私が気に入っているもう1つの点は、間隔とタイポグラフィのサイズがメディアクエリの特定のポイントで正しく見える場合、ほとんど--space-unit値を更新しているという事実に関係なく、メディアクエリのすべてのポイントを正しく確認してください。したがって、ブラウザウィンドウのサイズを変更せずに設計できます（コンポーネントの動作を変更したくない場合を除く）。

コンポーネントレベルでの「垂直リズム」の編集


SASS変数とは異なり、CSS変数の値をオーバーライドできます。このプロパティを使用する1つの方法は、カスタムプロパティを他のカスタムプロパティに追加して、コンポーネントレベルで編集できる「コントロール」を作成することです。

以下に例を示します。テキストコンポーネントの垂直方向の間隔を設定する場合、おそらく要素の行の高さとマージンの下部を指定する必要があります。

.article {
  h1, h2, h3, h4 {
    line-height: 1.2;
    margin-bottom: $space-xs;
  }

  ul, ol, p, blockquote {
    line-height: 1.5;
    margin-bottom: $space-md;
  }
}

ただし、このペースは、テキストが使用される場所によって異なります。たとえば、テキストをより簡潔にする場合は、間隔のさまざまな値を適用するコンポーネントバリアント（修飾子）を作成する必要があります。

.article--sm {
  h1, h2, h3, h4 {
    line-height: 1.1;
    margin-bottom: $space-xxxs;
  }

  ul, ol, p, blockquote {
    line-height: 1.4;
    margin-bottom: $space-sm;
  }
}

...そして、毎回、「垂直リズム」を更新したい場合。

CSS変数に基づく代替アプローチを次に示します。

.text-component {
  --component-body-line-height: calc(var(--body-line-height) * var(--line-height-multiplier, 1));
  --component-heading-line-height: calc(var(--heading-line-height) * var(--line-height-multiplier, 1));
  --line-height-multiplier: 1;
  --text-vspace-multiplier: 1;

  h1, h2, h3, h4 {
    line-height: var(--component-heading-line-height);
    margin-bottom: calc(var(--space-xxxs) * var(--text-vspace-multiplier));
  }

  h2, h3, h4 {
    margin-top: calc(var(--space-sm) * var(--text-vspace-multiplier));
  }

  p, blockquote, ul li, ol li {
    line-height: var(--component-body-line-height);
  }
  
  ul, ol, p, blockquote, .text-component__block, .text-component__img {
    margin-bottom: calc(var(--space-sm) * var(--text-vspace-multiplier));
  }
}

--line-height-multiplierおよび--text-vspace-multiplierは、2つのテキストコンポーネント制御変数です。.text-componentクラスの修飾子を作成するとき、垂直間隔を編集するには、次の2つの変数を再定義するだけです。

.article.text-component { //     
  --line-height-multiplier: 1.13; //    
  --text-vspace-multiplier: 1.2; //   
}

行動コンポーネントの抽象化


コンポーネント値をオーバーライドする機能は、さまざまな方法で使用できます。一般に、1つ以上の変数でコンポーネントの動作を抽象化できるときはいつでも、このコンポーネントの編集が必要な場合（またはコンポーネントバリアントを作成する必要がある場合）に、作業が楽になります。

例では、私たちのある自動グリッドサイズの部品我々はギャラリーの要素が自動的にCSSで指定された最小幅に基づいて利用可能なスペースを埋めるレイアウトを作成するために、CSSのグリッドを使用し、その後、私たちは抽象の最小幅の変数に格納し、要素を。


最小幅の値は、グリッドコンポーネントの自動サイズ調整オプションを作成するときに変更する必要がある唯一のものです。

キャッチは何ですか？


簡単に言うと、ブラウザのサポート... PostCssプラグインを使用すると、この記事で説明するすべての方法でCSS変数を使用できます。古いブラウザ（<IE11）にはいくつかの制限があります。それ以外の場合は、無料で使用できます。今日のCSS変数の使用に関する制限について詳しくは、ドキュメントをご覧ください。

プリプロセッサでCSS変数を使用できますか？


はい！SASS（またはその他のプリプロセッサ）を使用すると、CSSでは実行できないことを実行できますが、必要な場合は、なぜそれを使用しないのですか？SASSは、ユーザーがサイトにアクセスするときにダウンロードする必要があるライブラリではありません。これはワークフローのツールです。たとえば、SASSを使用して、CSS変数で機能するカラー関数を定義します。

結論


この記事では、SASS変数よりもCSS変数を使用する利点を示すいくつかの例を検討しました。コンポーネントの変更プロセスをスピードアップする「コントロール」を作成したり、タイポグラフィや間隔に影響を与えるルールを設定したりする方法に焦点を当てました。私たちは多くのことを検討してきましたが、この投稿から何かを学び、あなたの仕事でそれが使用できることを願っています。
