Bagaimana kami melebihi mekanisme perhitungan balistik untuk penembak seluler dengan algoritma kompensasi penundaan jaringan





Hai, saya Nikita Brigak, pengembang server di Pixonic. Hari ini saya ingin berbicara tentang kompensasi untuk keterlambatan dalam multiplayer seluler.



Banyak artikel telah ditulis tentang kompensasi jeda server, termasuk dalam bahasa Rusia. Ini tidak mengherankan, karena teknologi ini telah secara aktif digunakan untuk membuat FPS multi-pengguna sejak akhir 90-an. Misalnya, Anda dapat mengingat mod QuakeWorld, salah satu yang pertama yang menggunakan mod itu.



Kami menggunakannya di penembak multipemain seluler Dino Squad kami.



Dalam artikel ini, tujuan saya bukanlah mengulangi apa yang telah ditulis ribuan kali, tetapi untuk memberi tahu bagaimana kami menerapkan kompensasi keterlambatan dalam permainan kami, dengan mempertimbangkan tumpukan teknologi kami dan fitur-fitur gameplay inti.



Dalam beberapa kata tentang inti dan teknologi kami.



Dino Squad - jaringan seluler PvP-shooter. Pemain mengendalikan dinosaurus, digantung dengan berbagai senjata, dan bertarung satu sama lain dengan tim 6 hingga 6.



Baik klien maupun server di Unity kami. Arsitekturnya cukup klasik untuk penembak: servernya otoriter, dan prediksi klien berfungsi pada klien. Simulasi game ditulis menggunakan ECS in-house dan digunakan baik di server maupun di klien.



Jika Anda pertama kali mendengar tentang kompensasi keterlambatan, berikut adalah kunjungan singkat ke masalah tersebut.



Dalam game FPS multipemain, pertandingan biasanya disimulasikan pada server jarak jauh. Pemain mengirim input mereka ke server (informasi tentang tombol yang ditekan), dan sebagai tanggapan, server mengirimi mereka status permainan yang diperbarui, dengan mempertimbangkan data yang diterima. Dengan skema interaksi ini, penundaan antara menekan tombol "maju" dan saat karakter pemain bergerak di layar akan selalu memiliki lebih banyak ping.



Jika pada jaringan lokal keterlambatan ini (secara populer disebut sebagai input lag) dapat tidak terlihat, maka ketika bermain melalui Internet itu menciptakan perasaan "meluncur di atas es" ketika mengendalikan sebuah karakter. Masalah ini sangat relevan untuk jaringan seluler, di mana kasus ketika seorang pemain memiliki ping 200 ms dianggap sebagai koneksi yang sangat baik. Ping sering 350, dan 500, dan 1000 ms. Maka hampir tidak mungkin untuk bermain dengan input lag dalam penembak cepat.



Solusi untuk masalah ini adalah prediksi simulasi sisi klien. Di sini, klien itu sendiri menerapkan input ke karakter pemain, tanpa menunggu respons dari server. Dan ketika jawabannya diterima, itu hanya membandingkan hasil dan memperbarui posisi lawan. Keterlambatan antara menekan tombol dan menampilkan hasil di layar minimal dalam hal ini.



Di sini penting untuk memahami nuansa: klien selalu menggambar dirinya sesuai dengan input terakhirnya, dan musuh - dengan penundaan jaringan, seperti sebelumnya, dari data dari server. Artinya, ketika menembaki lawan, pemain melihatnya di masa lalu relatif terhadap dirinya sendiri. Kami menulis lebih banyak tentang prediksi klien sebelumnya .



Dengan demikian, prediksi klien memecahkan satu masalah, tetapi menciptakan masalah lain: jika seorang pemain menembak pada titik di mana musuh berada di masa lalu, server mungkin tidak lagi berada di tempat yang sama ketika menembak pada titik yang sama dari musuh. Kompensasi kelambatan server sedang mencoba menyelesaikan masalah ini. Saat menembakkan senjata, server mengembalikan keadaan permainan yang dilihat pemain secara lokal pada saat tembakan dan memeriksa apakah ia benar-benar dapat mengenai musuh. Jika jawabannya adalah ya, klik itu diperhitungkan, bahkan jika musuh tidak lagi pada titik ini di server.



Berbekal pengetahuan ini, kami mulai menerapkan kompensasi lag berbasis server di Dino Squad. Pertama-tama, saya harus mengerti cara mengembalikan pada server apa yang dilihat klien? Dan apa sebenarnya yang perlu dipulihkan? Dalam permainan kami, serangan senjata dan kemampuan dihitung melalui rakecast dan tumpang tindih - yaitu, melalui interaksi dengan penumbuk fisik musuh. Menyatukan semua elemen ini, kami mendapat "mesin waktu" yang dapat memutar kembali keadaan dunia fisik ke saat yang tepat. Kode ini mudah: 



public class TimeMachine : ITimeMachine
{
     //  
     private readonly IGameStateHistory _history;

     //    
     private readonly ExecutableSystem[] _systems;

     // ,      
     //    
     private readonly GameState _presentState;

     public TimeMachine(IGameStateHistory history, GameState presentState, ExecutableSystem[] timeInitSystems)
     {
         _history = history; 
         _presentState = presentState;
         _systems = timeInitSystems;  
     }

     public GameState TravelToTime(int tick)
     {
         var pastState = tick == _presentState.Time ? _presentState : _history.Get(tick);
         foreach (var system in _systems)
         {
             system.Execute(pastState);
         }
         return pastState;
     }
}


Tetap mengerti bagaimana menggunakan mesin ini untuk kompensasi lag dari tembakan dan kemampuan.



Dalam kasus paling sederhana, ketika mekanik dibangun di atas satu hitcan, semuanya tampak jelas: sebelum pemain menembak, Anda perlu memutar kembali dunia fisik ke kondisi yang diinginkan, membuat rakecast, menghitung hit atau miss dan mengembalikan dunia ke keadaan semula.



Tetapi di Dino Squad ada sangat sedikit mekanik seperti itu! Sebagian besar senjata dalam gim ini dibuat oleh proyektil - peluru berumur panjang yang menerbangkan beberapa kutu simulasi (dalam beberapa kasus, puluhan kutu). Apa yang harus dilakukan dengan mereka, pada jam berapa mereka harus terbang?



Dalam sebuah artikel kuno tentang tumpukan jaringan Half-Life, orang-orang dari Valve mengajukan pertanyaan yang sama, dan jawaban mereka adalah ini: keterlambatan kompensasi untuk file proyek bermasalah, dan lebih baik untuk menghindarinya.



Kami tidak memiliki opsi ini: senjata berdasarkan proyektil adalah fitur utama dari desain game. Karena itu, kami harus menciptakan sesuatu. Setelah sedikit pengereman, kami merumuskan dua opsi yang menurut kami berhasil:



1. Kami mengikat proyek dengan waktu pemain yang menciptakannya. Setiap centang simulasi server untuk setiap peluru dari setiap pemain, kami memutar kembali dunia fisik ke status klien dan melakukan perhitungan yang diperlukan. Pendekatan ini memungkinkan untuk memiliki beban yang didistribusikan di server dan waktu penerbangan yang dapat diprediksi untuk file proyek. Prediktabilitas sangat penting bagi kami, karena kami memiliki semua proyektil, termasuk proyektil lawan, diprediksi pada klien.





Dalam gambar, pemain di tick 30 menembakkan rudal di muka: dia melihat ke arah mana musuh berlari, dan tahu perkiraan kecepatan roket. Secara lokal, ia melihat bahwa ia mencapai target pada centang ke-33. Berkat keterlambatan kompensasi, itu juga akan sampai ke server

Dalam implementasi kami, kedua pendekatan ini berbeda secara harfiah dalam beberapa baris kode, jadi kami berdua mengatasinya, dan untuk waktu yang lama mereka ada secara paralel dengan kami. Bergantung pada mekanisme senjata dan kecepatan peluru, kami memilih satu atau beberapa opsi untuk setiap dinosaurus. Titik balik di sini adalah penampilan dalam permainan seorang mekanik seperti "jika Anda memukul musuh berkali-kali dalam waktu seperti itu, dapatkan bonus seperti itu." Setiap mekanik di mana waktu di mana pemain menabrak musuh memiliki peran penting menolak berteman dengan pendekatan kedua. Oleh karena itu, pada akhirnya, kami menetapkan pilihan pertama, dan sekarang digunakan untuk semua senjata dan semua kemampuan aktif dalam permainan.



Kita juga harus mengangkat masalah kinerja. Jika menurut Anda semua ini akan melambat, saya jawab: apa adanya. Memindahkan colliders, menyalakan dan mematikannya Unity cukup lambat. Dalam Dino Squad, dalam skenario "kasus terburuk", beberapa ratus file proyek dapat ada secara bersamaan dalam pertempuran. Memindahkan colliders untuk menghitung setiap proyek secara individual, merupakan kemewahan yang tidak dapat diterima. Oleh karena itu, sangat penting bagi kami untuk meminimalkan jumlah "suap" fisika. Untuk melakukan ini, kami membuat komponen terpisah di ECS, di mana kami mencatat waktu pemain. Kami menggantungnya di semua entitas yang membutuhkan kompensasi lag (terkait proyek, kemampuan, dll.). Sebelum kita mulai memproses entitas seperti itu, kita mengelompokkannya pada saat ini dan memprosesnya bersama, menggulirkan dunia fisik satu kali untuk setiap cluster.



Pada tahap ini, kami mendapatkan seluruh sistem kerja. Kodenya agak disederhanakan: 



public sealed class LagCompensationSystemGroup : ExecutableSystem
{
     // 
     private readonly ITimeMachine _timeMachine;

     //  
     private readonly LagCompensationSystem[] _systems;
     
     //  
     private readonly TimeTravelMap _travelMap = new TimeTravelMap();

    public LagCompensationSystemGroup(ITimeMachine timeMachine, 
        LagCompensationSystem[] lagCompensationSystems)
     {
         _timeMachine = timeMachine;
         _systems = lagCompensationSystems;
     }

     public override void Execute(GameState gs)
     {
         //      ,
         //     ¬´¬ª.     ,
         //          .
         var buckets = _travelMap.RefillBuckets(gs);

         for (int bucketIndex = 0; bucketIndex < buckets.Count; bucketIndex++)
         {
             ProcessBucket(gs, buckets[bucketIndex]);
         }

         //       
         //  
         _timeMachine.TravelToTime(gs.Time);
     }

     private void ProcessBucket(GameState presentState, TimeTravelMap.Bucket bucket)
     {
         //      
         var pastState = _timeMachine.TravelToTime(bucket.Time);

         foreach (var system in _systems)
         {
               system.PastState = pastState;
               system.PresentState = presentState;

               foreach (var entity in bucket)
               {
                   system.Execute(entity);
               }
          }
     }
}


Tinggal mengkonfigurasi rincian:



1. Memahami berapa banyak untuk membatasi rentang gerakan maksimum dalam waktu.



Penting bagi kami untuk membuat game dapat diakses semaksimal mungkin dalam kondisi jaringan seluler yang buruk, jadi kami membatasi ceritanya dengan selisih 30 kutu (dengan tingkat kutu 20 Hz). Hal ini memungkinkan pemain untuk memukul lawan bahkan pada ping yang sangat tinggi.



2. Tentukan objek mana yang bisa dipindahkan dalam waktu dan mana yang tidak bisa.



Jelas, kami menggerakkan lawan. Tetapi pelindung energi yang dipasang, misalnya, tidak. Kami memutuskan bahwa di sini lebih baik untuk memprioritaskan kemampuan defensif, seperti yang sering dilakukan pada penembak jaringan. Jika seorang pemain telah menempatkan perisai di masa sekarang, peluru yang dikompensasi ketinggalan dari masa lalu tidak boleh terbang melaluinya.



3. Putuskan apakah perlu untuk mengimbangi kemampuan dinosaurus: gigitan, pukulan ekor, dll. Kami memutuskan apa yang dibutuhkan dan memprosesnya sesuai dengan aturan yang sama seperti peluru.



4. Tentukan apa yang harus dilakukan dengan colliders pemain untuk siapa kompensasi lag dilakukan. Dengan cara yang baik, posisi mereka tidak boleh digeser ke masa lalu: pemain harus melihat dirinya dalam waktu yang sama di mana dia sekarang berada di server. Namun demikian, kami juga memutar kembali colliders dari pemain tembak, dan ada beberapa alasan untuk ini.



Pertama, ini meningkatkan pengelompokan: kita dapat menggunakan keadaan fisik yang sama untuk semua pemain dengan ping dekat.



Kedua, di semua rakecast dan tumpang tindih, kami selalu mengecualikan colliders pemain yang memiliki kemampuan atau proyektil. Di Dino Squad, pemain mengendalikan dinosaurus yang memiliki geometri yang agak tidak biasa menurut standar penembak. Bahkan jika pemain menembak pada sudut yang tidak biasa, dan lintasan peluru melewati collider dinosaurus pemain, peluru akan mengabaikannya.



Ketiga, kami menghitung posisi senjata dinosaurus atau titik penerapan kemampuan sesuai dengan data dari ECS bahkan sebelum dimulainya kompensasi lag.



Akibatnya, posisi sebenarnya dari colliders dari pemain yang diberi kompensasi lag tidak signifikan bagi kami, jadi kami melanjutkan dengan cara yang lebih produktif dan pada saat yang sama lebih sederhana.



Latensi jaringan tidak bisa dihapus begitu saja, tetapi hanya bisa ditutup-tutupi. Seperti metode penyamaran lainnya, kompensasi server lag memiliki tradeoffnya sendiri. Ini meningkatkan pengalaman bermain game dari pemain menembak dengan mengorbankan pemain yang sedang ditembak. Namun, untuk Dino Squad, pilihan di sini sudah jelas.



Tentu saja, saya harus membayar semua ini dengan meningkatnya kompleksitas kode server secara keseluruhan - baik untuk programmer dan desainer game. Jika sebelumnya simulasi adalah panggilan sekuensial sederhana dari sistem, maka dengan kompensasi lag, loop bersarang dan cabang muncul di dalamnya. Untuk membuatnya mudah digunakan, kami juga menghabiskan banyak upaya.



Dalam versi 2019 (atau mungkin sedikit lebih awal), Unity memiliki dukungan penuh plus atau minus untuk adegan fisik independen. Hampir segera setelah pembaruan, kami menerapkannya di server, karena kami ingin dengan cepat menyingkirkan dunia fisik yang umum di semua kamar.



Kami memberi setiap ruang permainan adegan fisik masing-masing dan dengan demikian menyingkirkan kebutuhan untuk "menghapus" adegan dari data kamar sebelah sebelum menghitung simulasi. Pertama, memberikan peningkatan signifikan dalam produktivitas. Kedua, itu memungkinkan kami untuk menyingkirkan seluruh kelas bug yang muncul jika programmer membuat kesalahan dalam kode untuk membersihkan adegan ketika menambahkan elemen game baru. Kesalahan seperti itu sulit untuk di-debug, dan mereka sering mengarah pada fakta bahwa keadaan objek fisik dari pemandangan satu ruangan "mengalir" ke ruangan lain.



Selain itu, kami melakukan sedikit riset tentang apakah adegan fisik dapat digunakan untuk menyimpan sejarah dunia fisik. Artinya, secara kondisional, tidak hanya mengalokasikan satu adegan untuk setiap kamar, tetapi 30 adegan, dan membuat buffer melingkar dari mereka, di mana sejarah disimpan. Secara umum, opsi ini ternyata berfungsi, tetapi kami tidak memperkenalkannya: opsi itu tidak menunjukkan semacam peningkatan produktivitas yang gila, tetapi itu membutuhkan perubahan yang agak berisiko. Sulit untuk memprediksi bagaimana server akan berperilaku selama bekerja lama dengan begitu banyak adegan. Karena itu, kami mengikuti aturan: " Jika tidak rusak, jangan memperbaikinya ."
