ゲームにおけるキャラクター能力の柔軟なシステムの設計について

キャラクターの能力システムは、おそらくゲームで最も厳しい柔軟性です。設計段階で、最終バージョンまたはその後の更新でどのスペルが現れるかを予測することは不可能です。この投稿では、能力を実行するプロセスを抽象化する方法について説明します。

能力自体はアクションのセットにすぎません。最小能力のインターフェースは、「適用」という1つの方法で構成されていますが、カットの難しさについてはそれほど単純ではありません。



すべての能力は、適用できるかどうかを確認する一連のチェックから始まります。その中には、リチャージのチェック、マナの利用可能性、距離のチェックなど、通常のものがあります。すべての能力がすべてのチェックを必要とするわけではないことは、すでに明らかです。例えば、どんな距離でも適用される能力があります。つまり、能力が異なれば、実行前に異なるチェックセットが必要になります。ただし、多くのチェックが繰り返されることは明らかであり、多くの場合、多くの機能では同じチェックのセットが必要です。

全体として、チェックの一部は論理的に繰り返されます。つまり、チェックは合意に応じて、つまりすべての場所で即座に変更する必要があります。さらに、一般的な場合のチェックのパーツのセットは異なります。

チェックの一部が1つのインターフェイスを実装する個別のオブジェクトに選択され、単一リンクリストに組み込まれている場合、責任のテンプレートチェーンが得られます。



リンクのチェックが成功した場合、次のリンクのチェックが開始されます。次のリンクがない場合、チェック全体が成功したと見なされます。チェック自体に加えて、リンクにはエラーハンドラーも含まれる場合があります。たとえば、マナをチェックしているときに十分でないことが判明した場合、リンクはこれをプレーヤーに通知できます。

一連の職務を使用して、能力[強力なショット]に、キャラクターが弓を身に着けているかどうかをチェックする追加リンク、または能力[セカンドウィンド]のキャラクターのヘルスレベルが30％未満であることをチェックするリンクを簡単に挿入できます。

ロールバックして、多くの能力で同じチェックのチェーンがあることを思い出してみましょう。能力フルフィルメントリクエストの本質を強調し、テストチェーンの各タイプとそのクラスについて説明します。

リクエストは、一連の責任を作成し、それを起動して、プレーヤーが適切なコマンドを発行したときにキャンセルするためにのみ必要です。



クエリの実装ではすでにチェーンを作成します。

現在、私たちはすでに、能力を実行する能力の柔軟なテストを行う方法を学びました。さて、テストが成功した場合でも、あなたはまだ能力を果たす必要があります。

私はインターフェースを変更せずにこれを行うことを好み、機能が副作用として実行する最後の常に成功するリンクを追加しました。以下はその実装例です。

public class TerminalChecker: ICastChecker {
    CastChecker next { get; set; } 
    ISkill skill;
    public TerminalChecker(ISkill skill) {
        this.skill = skill;
    }
    public bool check() { 
        skill.cast();
        return true; 
    }
}

この実装により、リクエストを非同期にすることができます。これは、ユーザーからの追加情報が必要な場合に役立ちます。たとえば、プレイヤーがマウスで選択した領域に能力を適用する必要があります。もちろん、この時点でゲームを停止することはできません。

次に、リクエストとアビリティを一致させる必要があります。もちろん、これを行うには、能力インターフェースにプロパティを追加することにより、ポリモーフィズムを使用します。この段階で、このインターフェースに機能を拡張しました。



すべての仕事が終わったら、能力とは何かについて考えましょう。現在の実装では、これは一連のチェックが先行する一連のアクションです。高いレベルでは、特定のゲームロジックにまったく依存していないことに注意してください。呪文に適用される能力のシステムを説明する最初のアイデアにより、特定のルールに従って、任意のアクションを実行することを許可または許可しないシステムが得られました。

この特性により、このシステムはゲームの世界のあらゆる変更を説明できます。たとえば、販売トランザクションや建物の建設チームです。

一般的にすべてを見てみましょう



この例では、スプリント機能はターゲットのない通常の機能であり、そのような機能のリクエストを実装するクラスはNontargetCastRequestであり、ManaChecker、CooldownChecker、およびTerminalCheckerからの一連のチェックを形成します。

呼び出しコードは、このシステムの実装の詳細に依存しません。つまり、機能を追加または変更することによってゲームロジックを壊すことはありません。

これは最小限の形のキャラクター能力のシステムです。このモデルには、呼び出し元のコードに警告したり、機能をユーザーインターフェイスに転送したり、その他の生活の詳細を伝える手段がありません。あなたはそれらを自分で考えることができます。
