Camunda外部タスク-復元力のあるスケーラブルなアーキテクチャでアプリケーションを構築するための強力なツール



Tinkoffでは、ビジネスプロセス自動化システムを開発するためにCamunda + Springフレームワークを使用しています。BPMN（Business Process Management Notation）を使用してビジネスプロセスをフローチャートの形式で記述します。

図で最も一般的に使用される要素は、サービスタスク（歯車の長方形）です。Camundaは、サービスタスクを実行する2つの方法をサポートしています。

  1. Javaコードへの同期呼び出しの使用。
  2. 外部タスクを作成します。

2番目の方法では、外部システムを使用してタスクを実行できます。たとえば、あるカムンダアプリケーションを別のアプリケーションから呼び出す必要がある場合や、作業を外部システムに委任する必要がある場合などです。


BPMNダイアグラムの例

これは、複数のアプリケーションでロジックを再利用する場合に役立ちます。または、マイクロサービスアーキテクチャにこだわりたい場合。たとえば、ビジネスプロセスを処理するサービスを、レポートの生成やメール送信などの技術的なタスクを実装するサービスから分離します。

この機能に加えて、外部タスクはスケーラブルで復元力のあるアーキテクチャを提供します。これが発生する理由を理解するには、まず、BPMNおよびアプリケーションレベルで外部タスクがどのように機能するかを理解する必要があります。

外部タスクワーカー


外部タスク パターンは、HTTPリクエストを実行できるツールを使用して、任意の言語でタスク処理を実装できるという点で優れています。

以下はcamundaブログの例です。この例では、20秒ごとにcamundaに処理タスクのリストを要求する外部JavaScriptハンドラーを実装しています。タスクがある場合、タスクを送信し、タスクの完了についてcamundaに通知します。

const baseUrl = 'http://localhost:8080/my-app/rest';
const workerSettings = {
 workerId: 'worker01', // some unique name for the current worker instance
 maxTasks: 5,
 topics: [
   {
     topicName: 'sendEmail',
     lockDuration: 10000, // How much time the worker thinks he needs to process the task
     variables: ['video'] // Which variables should be returned in the response (to avoid additional REST calls to read data)
   }]};
const requestParams = {method: 'POST', headers: {contentType: 'application/json'}};

function pollExternalTasks() {
 return fetch(`${baseUrl}/external-task/fetchAndLock`, {
   ...requestParams,
   body: JSON.stringify(workerSettings)
 })
}

function processExternalTask(result = []) {
 return Promise.all(result.map(externalTask => {
   sendEmail(externalTask); // Here the actual work would be done

   return fetch(`${baseUrl}/external-task/${externalTask.id}/complete`, {
     ...requestParams,
     body: JSON.stringify({workerId: workerSettings.workerId}),
   })
 }));
}

setInterval(() => {
 pollExternalTasks().then(processExternalTask)
}, 20000);

上記のコードからわかるように、外部タスクを処理するための主要なメソッドはfetchAndLockとcompleteです。最初の方法はタスクのリストを要求し、それらの実装を保護し、2番目の方法はタスクの完了について通知します。これらの2つの方法に加えて、他にも方法があります。それらについては、公式ドキュメントで読むことができます。

Camunda外部タスククライアント




camundaは 、外部タスク処理を実装するために、数行で外部タスクハンドラーを作成できるJavascriptおよびJavaクライアントを提供しました。また、JavascriptとJavaの例を使用して、外部タスクの処理の基本原則を説明する詳細なガイドもあります。ExternalTask​​Client を使用した外部ハンドラーの実装例：



public class App {
   public static void main(String... args) {
       // bootstrap the client
       ExternalTaskClient client = ExternalTaskClient.create()
           .baseUrl("http://localhost:8080/engine-rest")
           .asyncResponseTimeout(1000)
           .build();

       // subscribe to the topic
       client.subscribe("sendEmail").handler((externalTask, externalTaskService) -> {
           try {
               String result = sendEmail(externalTask)
               Map<String, Object> variables = new HashMap<>();

               variables.put("result", result);
               externalTaskService.complete(externalTask, variables);
               System.out.println("The External Task " + externalTask.getId() + " has been completed!");
           } catch (e: Exception) {
               externalTaskService.handleFailure(externalTask, e.message, e.stackTrace.toString())
           }
       }).open();
   }
}

タスクで同期アクションを実行するだけでなく、プロセス全体を開始する必要がある場合は、たとえば、RuntimeServiceを介してプロセスを開始することで実行できます。

@Service
class EmailWorker(
   private val runtimeService: RuntimeService
) {
   val builder = ExternalTaskClientBuilderImpl().baseUrl("http://localhost:8080").workerId("myWorker")
   val taskClient = builder.build()
   val engineClient = (builder as ExternalTaskClientBuilderImpl).engineClient

   @PostConstruct
   fun init() {
       taskClient
           .subscribe("sendEmail")
           .lockDuration(10000)
           .handler { externalTask, externalService ->
               runtimeService.startProcessInstanceByKey(
                   "SendEmailProcess",
                   externalTask.getVariable("emailId"),
                   mapOf(
                       "text" to externalTask.getVariable("text"),
                       "email" to externalTask.getVariable("email")
                   )
               )
           }
           .open()
   }


   @PreDestroy
   fun destroy() {
       taskClient.stop()
   }
}

// Delegate from SendEmailProcess process
@Component
class EmailResultDelegate(private val emailWorker: EmailWorker) {
   fun doExecute(execution: DelegateExecution) {
       emailWorker.engineClient.complete(
           execution.readVar(EXTERNAL_TASK_ID),
           mapOf("result" to "Success")
       )
   }
}

この例では、外部のタスクハンドラ（EmailWorkerタスクが受信される）、開始SendEmailProcessのプロセス。

このプロセスがニュースレターを送信するために必要ないくつかのアクションを実行し、最後にEmailResultDelegateを呼び出し、次に外部タスクを完了するとします。

外部タスクのアーキテクチャ上の利点


別のcamundaアプリケーションでプロセスをより簡単な方法で開始する方法があることは注目に値します：POST：/ rest / process-definition / key / $ {id} / start REST

を使用する場合、トランザクションは保証されません。しかし結局のところ、私たちはRESTを介して外部タスクも処理しますが、その違いは何ですか？ 違いは、外部サービスを直接呼び出すのではなく、処理可能なタスクのみを公開することです。例を考えてみましょう： 一部の外部ハンドラーは、現在割り当てられているタスクを実行しますが、応答を受信すると、接続が切断されます。今カマンダの側に





外部ハンドラーが応答を受信しなかったため、処理されないタスクはブロックされます。しかし、これは恐ろしいことではありません。外部タスクのカムンダでは、タイムアウトによってタスクが再びキューに戻り、他の誰かがそれを処理できます。 次に、外部ハンドラーがタスクを受け取って完了し、ネットワークの問題のために失敗したcompleteメソッドを呼び出した場合を見てみましょう。これで、タスクがcamundaで正常に完了したかどうかを理解できなくなります。再試行できますが、ネットワークの問題が引き続き発生する可能性があります。



この場合、最善の解決策は問題を無視することです。タスクが正常に完了した場合、すべて順調です。そうでない場合は、タイムアウト後、タスクを再び処理できるようになります。ただし、これは、外部ハンドラーがべき等であるか、タスクを重複排除するロジックを含んでいる必要があることを意味します。

新しいプロセスを開始するときにも同様の問題が発生する可能性があるため、同じデータ（例えばbusinessKey）を持つ既存のインスタンスを確認する必要があります。
