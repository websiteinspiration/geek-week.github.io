Goに関するLLVM

コンパイラの開発は非常に困難な作業です。しかし、幸いなことに、LLVMのようなプロジェクトの開発により、この問題の解決策は大幅に簡素化され、1人のプログラマーでもCに近いパフォーマンスの新しい言語を作成できるようになりました。LLVMの操作は、このシステムが小さなドキュメントを備えた膨大な量のコードで表されているという事実によって複雑になっています。この欠陥を修正するために、今日公開している資料の作成者は、Goで記述されたコードの例を示し、最初にGo SSAに変換され、次にTinyGOコンパイラーを使用してLLVM IRに変換される方法を示します。。Go SSAおよびLLVM IRコードはわずかに編集されています。ここで説明に含まれていないものは、これらの説明をわかりやすくするために削除されています。


最初の例


ここで解析する最初の関数は、数値を追加するための単純なメカニズムです。

func myAdd(a, b int) int{
    return a + b
}

この関数は非常に単純で、おそらく思いつくのが簡単なことはありません。これは、次のGo SSAコードに変換されます。

func myAdd(a int, b int) int:
entry:
    t0 = a + b                                                    int
    return t0

この関数の表現では、データ型に関するヒントが右側に配置され、ほとんどの場合、それらを無視できます。

この小さな例では、すでにSSAの1つの側面の本質を見ることができます。つまり、コードをSSA形式に変換するとき、各式はそれが構成する最も基本的な部分に分割されます。私たちの場合、コマンドreturn a + bは実際には2つの演算です。2つの数値の加算と結果の返却です。

さらに、ここではプログラムの基本ブロックを確認できます。このコードには、入力（エントリブロック）の1つのブロックしかありません。以下のブロックについて詳しく説明します。

Go SSAコードはLLVM IRに簡単に変換できます。

define i64 @myAdd(i64 %a, i64 %b) {
entry:
  %0 = add i64 %a, %b
  ret i64 %0
}

ここでは他の構文構造が使用されていますが、関数の構造は基本的に変更されていません。 LLVM IRコードは、Cと同様にGo SSAコードよりわずかに強力です。ここで、関数宣言では、最初に返されるデータ型の説明があり、引数の型は引数名の前に示されます。さらに、IR解析を簡略化するために、グローバルエンティティの名前の前には記号を@付け、ローカルエンティティの名前の前には文字を付けます%（関数もグローバルエンティティと見なされます）。

注意する必要があるこのコードの機能の1つは、Go型の表現に関する決定です。intは、コンパイラとコンパイルの目的に応じて32ビットまたは64ビットの値で表すことができ、LLVM IRコードの作成時に受け入れられます。これは、LLVM IRコードがプラットフォームに依存しない多くの理由の1つです。あるプラットフォーム用に作成されたこのようなコードは、別のプラットフォーム用に単純に取得してコンパイルすることはできません（このタスクに細心の注意を払ってアプローチしない限り）。

注目に値するもう1つの興味深い点は、タイプi64-これは符号付き整数ではありません。数値の符号を表すという点で中立です。命令によっては、符号付きの数値と符号なしの数値の両方を表すことができます。加算演算の表現の場合、これは役割を果たさないため、符号付きまたは符号なしの数値での作業に違いはありません。ここで、Cでは、符号付き整数変数のオーバーフローが未定義の動作につながるため、Clangフロントエンドが操作にnsw（符号付きラップなし）フラグを追加することに注意してください。これは、追加されたときに、オーバーフローは発生しません。

これは、一部の最適化にとって重要な場合があります。たとえば、2つの値を追加するi1632ビットプラットフォーム（32ビットレジスタ付き）では、追加が完了した後、範囲内に留まるために文字拡張操作が必要ですi16。このため、レジスタのマシンサイズを考慮して整数演算を実行する方が効率的です。

このIRコードで将来何が起こるかは、今のところ私たちにとって特に興味深いことではありません。コードが最適化されます（ただし、このような単純な例の場合、まだ最適化されているものはありません）。その後、コードはマシンコードに変換されます。

2番目の例


これから検討する次の例は、もう少し複雑になります。つまり、整数のスライスを合計する関数について話しています。

func sum(numbers []int) int {
    n := 0
    for i := 0; i < len(numbers); i++ {
        n += numbers[i]
    }
    return n
}

このコードは、次のGo SSAコードに変換されます。

func sum(numbers []int) int:
entry:
    jump for.loop
for.loop:
    t0 = phi [entry: 0:int, for.body: t6] #n                       int
    t1 = phi [entry: 0:int, for.body: t7] #i                       int
    t2 = len(numbers)                                              int
    t3 = t1 < t2                                                  bool
    if t3 goto for.body else for.done
for.body:
    t4 = &numbers[t1]                                             *int
    t5 = *t4                                                       int
    t6 = t0 + t5                                                   int
    t7 = t1 + 1:int                                                int
    jump for.loop
for.done:
    return t0

ここでは、SSAの形式でのコードの表示に固有のその他の構造を既に確認できます。おそらくこのコードの最も明白な機能は、構造化されたフロー制御コマンドがないという事実です。計算のフローを制御するには、条件付きジャンプと無条件ジャンプのみがあり、このコマンドをフローを制御するためのコマンドと見なす場合は、returnコマンドです。

実際、ここでは、プログラムが中括弧を使用してブロックに分割されていない（Cファミリの言語のように）ことに注意してください。これは、アセンブリ言語に似たラベルで分割され、基本ブロックの形式で表示されます。 SSAでは、ベースブロックは、例えば、コードの連続配列ラベルで開始し、ベースブロックを完了するための命令で終わる、である、returnとjump。

このコードのもう1つの興味深い詳細は、命令phiです。指示はかなり珍しいです、それを理解するのにしばらく時間がかかるかもしれません。ことを忘れないでくださいSSAは、静的単一代入の略です。これは、コンパイラが使用するコードの中間表現であり、各変数には値が1回だけ割り当てられます。これは、myAdd上に示した関数のような単純な関数を表すのに最適ですが、このセクションで説明する関数のようなより複雑な関数の場合には適していませんsum。特に、サイクルの実行中の変数iと変化n。

SSAは、いわゆる命令を使用して、変数値の単一割り当てに関する制限を回避しますphi（その名前はギリシャ語のアルファベットから取られています）。事実は、Cのような言語用にコードのSSA表現が形成されるためには、いくつかのトリックに頼らなければならないということです。この命令を呼び出した結果は、変数（iまたはn）の現在の値であり、ベースブロックのリストがそのパラメーターとして使用されます。たとえば、次の指示を検討してください。

t0 = phi [entry: 0:int, for.body: t6] #n

その意味は次のとおりです。前のベースブロックがブロックentry（入力）の場合、t0これは定数0であり、前のベースブロックfor.bodyである場合t6、このブロックから値を取得する必要があります。これはすべて不可解に見えるかもしれませんが、このメカニズムのおかげで、SSAは保証されます。人間の視点から見ると、これはすべてコードの理解を複雑にしますが、各値が一度だけ割り当てられるという事実は、多くの最適化を大幅に簡略化します。

独自のコンパイラを作成している場合は、通常、そのようなことを処理する必要はありません。 Clangでもこれらの命令をすべて生成するのではなくphi、メカニズムを使用しますalloca（通常のローカル変数での作業に似ています）。次に、mem2regと呼ばれるLLVM最適化パスを実行すると、命令がallocaSSA形式に変換されます。ただし、TinyGoはGo SSAからの入力を受け取ります。GoSSAは、便宜上、既にSSA形式に変換されています。

この中間コードのもう1つの革新は、インデックスによるスライス要素へのアクセスが、アドレスを計算する操作と、結果として生じるポインターの逆参照操作の形で提供されることです。ここでは、IRコードに定数が直接追加されていることがわかります（例：-）1:int。関数をmyAdd使用した例では、これは使用されませんでした。これらの機能を理解したところで、LLVM IRフォームに変換するときにこのコードがどのようになるかを見てみましょう。

define i64 @sum(i64* %ptr, i64 %len, i64 %cap) {
entry:
  br label %for.loop

for.loop:                                         ; preds = %for.body, %entry
  %0 = phi i64 [ 0, %entry ], [ %5, %deref.next ]
  %1 = phi i64 [ 0, %entry ], [ %6, %deref.next ]
  %2 = icmp slt i64 %1, %len
  br i1 %2, label %for.body, label %for.done

for.body:                                         ; preds = %for.loop
  %3 = getelementptr i64, i64* %ptr, i64 %1
  %4 = load i64, i64* %3
  %5 = add i64 %0, %4
  %6 = add i64 %1, 1
  br label %for.loop

for.done:                                         ; preds = %for.loop
  ret i64 %0
}

ここでは、以前と同様に、他の構文構造を含む同じ構造を見ることができます。たとえば、呼び出しではphi値とラベルが交換されます。ただし、特に注意が必要な点もあります。

まず、ここでは、関数の完全に異なるシグネチャを確認できます。 LLVMはスライスをサポートしていないため、最適化という形で、この中間コードを生成したTinyGoコンパイラーがこのデータ構造の記述を複数の部分に分割しました。彼は、スライスの3つの要素を表す（可能性ptr、lenおよびcap）の構造（構造体）の形ではなく、3つの別々のエンティティの形でそれらを提示することは、いくつかの最適化が可能になります。他のコンパイラは、他の方法でスライスを提示する場合があります。それは、ターゲットプラットフォームの関数を呼び出すための規則に依存します。

このコードのもう1つの興味深い機能は、命令の使用ですgetelementptr（多くの場合、略してGEPと呼ばれます）。

この命令はポインターで機能し、スライス要素へのポインターを取得するために使用されます。たとえば、Cで記述された次のコードと比較してみましょう。

int* sliceptr(int *ptr, int index) {
    return &ptr[index];
}

またはこれと同等の次のように：

int* sliceptr(int *ptr, int index) {
    return ptr + index;
}

ここで最も重要なことは、命令getelementptrが逆参照操作を実行しないことです。既存のポインターに基づいて新しいポインターを計算するだけです。これは命令として撮影することができますmulし、addハードウェアレベルで。 GEPの手順の詳細については、こちらをご覧ください。

この中間コードのもう1つの興味深い機能は、命令の使用ですicmp。これは、整数比較の実装に使用される汎用命令です。この命令の実行結果は常に型の値i1-論理値です。この場合、slt以前に型で表された2つの数値を比較しているため、キーワード（より小符号付き）を使用して比較が実行されます。int。2つの符号なし整数を比較する場合、命令としてをicmp使用し、比較で使用されるキーワードはになりますult。浮動小数点数を比較するにfcmpは、同様に機能する別の命令を使用します。

概要


この記事では、LLVM IRの最も重要な機能を調べたと思います。もちろん、まだたくさんあります。特に、コードの中間表現には多くの注釈が含まれている可能性があり、他の方法ではIRで表現できない最適化パス中にコンパイラーに既知の特定の最適化機能を考慮に入れることができます。たとえば、これらはinboundsGEP命令のフラグ、または命令に追加できるnswand フラグです。同じことが、オプティマイザがマークした関数が現在のコンパイル単位の外部から参照されないことをオプティマイザに示すキーワードにも当てはまります。これにより、未使用の引数の除去など、多くの興味深い手続き間最適化を実行できます。 LLVMの詳細については、nuwaddprivate
