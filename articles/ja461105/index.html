5つの便利なwebpackプラグイン

こんにちは、ハブル！

webpackには、多くの人が知らない、プロジェクトで使用しない便利なプラグインがたくさんあります。カットの下で、私はこれらの5つを集めました、彼らはあなたの人生を大幅に簡略化することができます！



1.未使用のファイルwebpackプラグイン


大規模なフロントエンドプロジェクトでは、迷子になりやすいモジュールが常に多数あります。プロジェクトで未使用のモジュールを確認しないと、遅かれ早かれ、どこにも使用されていないジャンクファイルが蓄積されます。

プロジェクト内の未使用のファイルについて知るには、unused-files-webpack-pluginを設定に追加します。

npm i unused-files-webpack-plugin


/* webpack.config.js */

const { UnusedFilesWebpackPlugin } = require('unused-files-webpack-plugin');

module.exports = {
  /* ... */
  plugins: [
    /* ... */
    new UnusedFilesWebpackPlugin(),
  ],
};

インストール後、未使用のファイルに関する警告が表示されます。

src/
├── index.js
└── someFile.js ( )


WARNING in 
UnusedFilesWebpackPlugin found some unused files:
src/someFile.js

2.大文字と小文字を区別するパスwebpackプラグイン


OSXでの開発中に、モジュールをインポートするときにパスの小文字と大文字を混同することがよくあります。アセンブリはケシで組み立てられますが、他の大文字と小文字を区別するシステムでは落ちます。

この問題を解消するには、大文字と小文字を区別するパスのwebpack-pluginを配置する必要があります。

npm i case-sensitive-paths-webpack-plugin


/* webpack.config.js */

const CaseSensitivePathsPlugin = require('case-sensitive-paths-webpack-plugin');

module.exports = {
  /* ... */
  plugins: [
    /* ... */
    new CaseSensitivePathsPlugin(),
  ],
};

大文字と小文字を区別しないでモジュールをインポートすると、常にエラーが発生します。

/* index.js */

import someFile from './SOMEFILE';


/* someFile.js */

export default () => {
  console.log('Hello!');
};


ERROR in index.js
Module not found: Error: [CaseSensitivePathsPlugin] `SOMEFILE.js` does not match the corresponding path on disk `someFile.js`.

この問題は、インポート/未解決の eslintプラグインを使用して解決することもできます。

typescriptには、forceConsistentCasingInFileNamesという同様の作業オプションがあります。しかし、彼女は少し違ったふるまいをします：彼女は、異なる場所からのモジュールのインポートが場合によっては異ならないことをチェックします。レジスターの正確さはチェックされません。

3. Inspectpack


プロジェクトで、あなたとあなたのパッケージが同じライブラリの異なるバージョンを使用している状況が発生する可能性があります。次に、パッケージのいくつかのバンドルがバンドルに表示されます。これは、同じバージョンを指定することで簡単に修正できます。

Inspectpackは、この状況をすぐに見つけるのに役立ちます。

npm i inspectpack


/* webpack.config.js */

const { DuplicatesPlugin } = require('inspectpack/plugin');

module.exports = {
  /* ... */
  plugins: [
    /* ... */
    new DuplicatesPlugin(),
  ],
};

プラグインをインストールすると、パッケージのコピーが表示されたときに警告がポップアップ表示されます。

}


/* node_modules/one/package.json */

{
  /* ... */
  "name": "one",
  "dependencies": {
    "lodash": "3.0.0"
  }
}

4.循環依存プラグイン


開発中、依存関係の解決に問題が発生することがあります。2つのモジュールが相互にインポートし、循環依存関係が取得されます。これは、他のモジュールのチェーンを通じて暗黙的に発生する可能性があります。まれに、循環依存関係は正常ですが、これはおそらくプロジェクトのアーキテクチャに問題があることを示しています。

循環依存関係をすぐに確認して削除するには、circular-dependency-pluginを追加します。

npm i circular-dependency-plugin


/* webpack.config.js */

const CircularDependencyPlugin = require('circular-dependency-plugin');

module.exports = {
  /* ... */
  plugins: [
    /* ... */
    new CircularDependencyPlugin(),
  ],
};

ここで、循環依存関係が表示されると、警告が表示されます。

/* first.js */

import second from './second'

/* ... */


/* second.js */

import first from './first'

/* ... */


WARNING in Circular dependency detected:
first.js -> second.js -> first.js

WARNING in Circular dependency detected:
second.js -> first.js -> second.js

eslint のインポート/非サイクルルールとtslintのtslint-no-circular-importsは、同様の問題を解決します。ただし、後者には、型付けにのみ使用される型、インターフェース、およびクラスのインポートを無視する機能はありません。tslint：disableを作成する必要があることがよくあります。

5.スピード測定webpackプラグイン


数百のファイルで構成される大規模なプロジェクトでは、アセンブリに数分かかる場合があります。

Speed-measure-webpack-pluginは、すべてのビルドステップを測定し、問題を見つけるのに役立ちます。

npm i speed-measure-webpack-plugin


/* webpack.config.js */

const SpeedMeasurePlugin = require("speed-measure-webpack-plugin"); 
const smp = new SpeedMeasurePlugin();

module.exports = smp.wrap({
  /* ... */
});
