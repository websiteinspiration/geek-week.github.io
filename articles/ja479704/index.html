EA Origin Windowsクライアントでの権限昇格（CVE-2019-19247およびCVE-2019-19248）

脆弱性分析に関する私の新しい記事を読むことにしたすべての人への挨拶。前回は、3つの記事の短い直列に、私はスチームの脆弱性について話しました（1、2、および3）。この記事では、ゲームのランチャーでもある類似製品Originの脆弱性について説明します。発見された脆弱性は、CVE-2019-19247およびCVE-2019-19248によって取得されました。



今回はアンバンバナナとの試合はありません。 Electronic Arts Incのセキュリティ部門との通信の歴史は、当初は専門家レベルでした。連絡を受けたところ、登録番号が通知され、レポートは慎重に調査され、確認されました。私のメールはどれも無視されず、少しの議論のためにconfkallが編成されました。これらのレポートのメンテナンスは私にとって非常に簡単でした。そのため、私のレポートを扱ってくれたAdrian Stone、Elise Murphy、および他のEA従業員に感謝します。セキュリティに関するブログ投稿と助言。

次に、脆弱性について説明します。OriginのWindowsクライアントに、「特権の昇格」（lpe-ローカル特権の昇格またはeop-特権の昇格）などの2つの脆弱性が見つかりました。このタイプの脆弱性により、すべてのWindowsユーザーが、管理者によって最初に発行されたものよりも多くの権限を取得できるようになります。この場合、2つの「典型的な」拡張について話している-任意のユーザーからNT AUTHORITY \ SYSTEM（OSで最大の権限を持つアカウント）まで。最初の脆弱性はかなり退屈なので、次のセクションで簡単に説明します。しかし、2番目は非常に興味深いものでした。彼女のセクションでは、私がどのように彼女を探したかを正確に説明します。

CVE-2019-19248


この脆弱性は2つの部分で構成されています。

  1. 任意のパス上にフォルダを作成します（オールフルアクセス権）。
  2. NT AUTHORITY \ SYSTEM特権を取得するには、1節を使用します。

次に、最初のポイントについてさらに詳しく説明します。

環境の準備


Originクライアントを閉じて、Originクライアントサービスを停止する必要があります（理論的には、万が一に備えてクライアントを閉じると、サービス自体が停止します）。

フォルダ「C：\ ProgramData \ Origin」の場合、権限は「すべてフルアクセス」であり、その内容を完全に削除できます。

リンクビルディング


次に、いくつかのリンクを作成します。最初のリンクは、NTFS再解析ポイント（NTFSマウントポイント）タイプ-フォルダーからフォルダーへのリンクのタイプ：「C：\ ProgramData \ Origin」<->「\ RPC Control」です。再解析ポイントを作成するには、管理者権限は必要ありません。ソースフォルダーが空であり、ユーザーがそのフォルダーへの書き込み権限を持っていることだけが必要です（最後の手順でクリアし、そこで権限を確認します）。 \ RPCコントロールはファイルシステム内のフォルダーではなく、特別な種類のフォルダー-オブジェクトディレクトリです。通常のエクスプローラでは表示されませんが、Windowsの腸で使用される一般的な抽象化が原因で、再解析ポイントを作成できます。

ここで、通常のシンボリックリンク "\ RPC Control \ CatalogCache" <-> "C：\ Path \ To \ Target \ Folder"を作成します。ファイルシステムでは、管理者権限なしでシンボリックリンクを作成することは禁止されていますが、このルールはオブジェクトディレクトリには適用されません。したがって、リンクが正常に作成されます。これら2つのリンクの組み合わせの結果として、「C：\ ProgramData \ Origin \ CatalogCache」の呼び出しは「C：\ Path \ To \ Target \ Folder」にリダイレクトされます。

このようなリンクの詳細については、こちらをご覧ください。同じリポジトリで、リンクを操作するためのユーティリティをダウンロードできます。

発売


最後のステップで、クライアントを実行します。仕事の最初に、彼はOrigin Client Serviceを起動し、C：\ ProgramData \ Origin \ CatalogCacheフォルダーがないことを確認して、それを作成しようとします。シンボリックリンクを移動した結果、「C：\ Path \ To \ Target \ Folder」が作成され、このフォルダーに「すべてフルアクセス」権限が付与されます。

操作の最初のポイントで取得するために必要なもの。2番目に進みましょう。

任意のパスにフォルダーを作成する操作


ここでは、さまざまな方法で作業できます。

シンプルでかなり信頼できるのは、フォルダ「C：\ Windows \ system32 \ LogonUI.exe.local」を作成することです。"LogonUI.exe"（NT AUTHORITY \ SYSTEMから実行されるアプリケーションは、ユーザー選択画面とロック画面の操作を担当します） ".local-redirection"（ "dotlocal redirection"）メカニズムのおかげで、パス "C：\ Windows \ system32 \ LogonUI.exe.local \ amd64_microsoft.windows.common-controls_6595b64144ccf1df_6.0.17134.648_none_fb45a0e93062a6d2 \ comctl32.dll。 " 一般に、メカニズム自体は非常に一般的であるため、多くの目標を持つことができます。

長く興味深い方法は、特別な操作によって管理者パスワードのハッシュを差し引くことです。詳細はこちら。

合計


この脆弱性は非常に簡単に悪用されます。2番目のポイントで少し作業するだけで、ターゲットを見つけて適切なdllを作成できます。さらに、マットネルソンもこの脆弱性を報告しており、彼の執筆はこちらにあります。

CVE-2019-19247


これは私のお気に入りの脆弱性の1つです。これは、使用する暗号化とどの程度注意深く関連付ける必要があるかを示しています。

それはすべて、Originを通じてゲームをインストールしたことから始まりました。すべてがどういうわけかスムーズに進みました-数回のクリックとダウンロードの数分後、ゲームを起動できます。すぐにはわかりませんでしたが、問題は理解しました。ゲームは「C：\ Program Files \ GameName」のパスに沿ってインストールされましたが、UACを通じて1つの質問もしませんでした。私はすぐに権利を確認しました。すべてが標準でした-通常のユーザーは「C：\ Program Files」に書き込むことができませんでした。もう少し調査したところ、ゲームがOriginクライアント自体ではなく、Origin Client Serviceによって「規定」されていることがわかりました。

私は、何かが悪用される可能性があるかどうかを確認するために、クライアントがサービスに情報を送信する方法について想定を始めました。

情報を送信する方法は単純であることがわかりました-名前付きパイプ。これについてはインストールログから知りました。プレーンテキストでは、OriginClientServiceパイプがファイルとフォルダを操作するコマンドを受け付けていることが示されていました。

IDAを開き、そこにクライアントをアップロードしました。

* IDAで行われた作業：1 *

かなりすぐに、コマンドが一般にテキスト形式でパイプに送信されることがわかりました。近くでコマンドのリストを見つけ、それ以上の手間をかけずに、「copy」C：\ test \ payload.dll”“ C：\ Windows \ pwn.dll”のようなコマンドをパイプに送信しました。迅速な結果を見越して、「C：\ Windows」フォルダをチェックし、そこに新しいものを見つけませんでした。しかし、ログには新しい何かがあります-パイプのクライアントがデジタル署名の検証に合格しなかったという事実についてのいくつかの言葉。

IDAを開き、そこにサービスをアップロードしました。

* IDAで実施された作業：2 *

いずれにしても、チームは誰からも期待されていないことがわかりました。パイプに接続すると、サービスはパイプに接続しているユーザーを確認します。プロセスのpidが接続から抽出され、実行可能ファイルへのパスがpidから抽出され、ファイルの署名が正しいかどうかがチェックされ、EAによって発行されます。
音は聞こえますが、十分ではありません。正当な "Origin.exe"（クライアント実行可能ファイル）を取得して、フォルダーのどこかにコピーできます。このフォルダにインポートリスト「Origin.exe」からDLLをいくつか配置します。たとえば、version.dllが登場しました。私はこのアプローチを「逆dllインジェクション」と呼びました。通常のdllインジェクションでは、dllをexeファイルにコピーしましたが、今はその逆を行いました。私はすぐにversion.dllのプロキシdllを記述し、コマンドをパイプに送信するコードを追加します。ペイロードはまだコピーされていません。ログを読みます-「それはどういう意味ですか、コマンドを解読できませんでした！」暗号化をスキップしました。

IDAを開き、そこにクライアントをアップロードしました。

* IDAで行われた作業：3、署名バイパス：1 *

クライアントは彼の通常の作業で暗号化されたコマンドを送信するので、私はできます。そこで私が調べ、次に調べた結果は次のとおりです。AES暗号化、定数ベクトルの初期化、キーはファイルから読み取られます。この部分とIDAをコードに実際にコピーし、コンパイルして検証します。再び何も。しかし、ログは再び有用な情報を提供します-ターゲットパスとしてProgram Filesを指定することはできません。

IDAを開き、そこにサービスをアップロードしました。

* IDAで行われる作業：4、署名バイパス：1、暗号化バイパス：1 *

したがって、コマンドを取得するためのチェックがあることは事実ですが、ファイルはどこにでもコピーできないことがわかります。また、「\ .. \」のパスは記述できません。他のチームとは何かを調べます。
レジストリでの作業-再び多くの制限があります。しかし、ファイルの起動は面白そうです。少なくともチェックは特に見えません。コードを編集し、コマンド「ExecuteProcess“ C：\ test \ payload.exe”」を送信します。さて、あなたは理解しています...何もありません。

ログは再び署名について話します。ああ、もう勝ちました。コピーされたOrigin.exeを呼び出してプロキシdllを再度ロードしたが、システム権限が付与されていることをコードで示しています。チェックを追加し、コンソールを起動します。起動すると、NT AUTHORITY \ SYSTEM権限を持つコンソールが表示されます-ついにすべてが機能しました。

* IDAで行われる作業：4、署名のバイパス：2、暗号化のバイパス：1 *

したがって、再起動して最終的な実行を行い、最大の権限でコンソールを楽しむ必要があります。再起動して確認してください...何もありません。どう？うまくいきました。

診断により、Origin Client Serviceが開始されていないことが示されているため、開始します。しかし、それは始まりません。より正確には、開始しますが、すぐに終了します。Originクライアントを起動すると、サービスが正常に起動します。その後、エクスプロイトは再び正しく機能します。そこで停止することは可能ですが、これは私のやり方ではありません。私はエクスプロイトを完全に機能させるように強制したいのです。

IDAを開き、そこにサービスをアップロードしました。

* IDAで行われた作業：5、署名のバイパス：2、暗号化のバイパス：1 *

起動時に、サービスが開始したパラメーターを確認することがわかりました。そこに直接興味深いものは何もありません。ファイルの署名が検証されるプロセスの暗号化されたpidからのBase64。複雑に聞こえますが、暗号化と署名もすでにバイパスしています。私たちはいくつかのコードを書いており、完全なエクスプロイトの準備ができています。

合計


エクスプロイトは機能しています。作業はIDAで実行されました：5回、バイパス署名：3回、暗号化バイパス：2回。

結論


脆弱性が修正されました：EAの開発者がクライアントに特別な制限付き操作モードを導入しました。これにより、Originフォルダーとパイプの操作に深刻な制限が設定されます。
