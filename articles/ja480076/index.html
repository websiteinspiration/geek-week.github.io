さまざまなソースからのデータのマルチプロセッシングと調整

こんにちは、ハブル！

分散システムが多様であることを考えると、ターゲットストレージで検証済みの情報を利用できるかどうかは、データの一貫性の重要な基準です。

これには多くのアプローチと手法があり、ここでは調整の理論に焦点を当てます。私は、このシステムの実際の実装を検討することを提案します。

前書き


金融機関に複数の分散システムがあり、これらのシステムでトランザクションを確認し、調整済みのデータをターゲットストレージにアップロードするタスクに直面しているとしましょう。

データソースとして、大きなテキストファイルとPostgreSQLデータベースのテーブルを使用します。これらのソースのデータは同じトランザクションを持っているが、それらには違いがある可能性があるため、分析のために検証し、最終ストレージの検証済みデータに書き込む必要があるとします。

さらに、同じデータベースで複数の調整を並行して起動できるようにし、マルチプロセッシングを使用してシステムを大容量に適合させる必要があります。マルチプロセッシング

モジュールPythonでの操作の並列化に最適であり、ある意味で、特定のGILの欠陥を回避します。このライブラリの機能を以下で使用します。

開発中のシステムのアーキテクチャ



使用するコンポーネント：

  • ランダムデータジェネレーター -CSVファイルを生成し、それに基づいてデータベースのテーブルを埋めるPythonスクリプト。
  • データソース -PostgreSQLデータベースのCSVファイルとテーブル。
  • アダプター -この場合、ソース（CSVまたはDB）からデータを抽出し、中間データベースに情報を入力する2つのアダプターを使用します。
  • データベース-3つの部分：生データ、アダプターによって取得された情報を格納する中間データベース、および両方のソースからの調整済みトランザクションを含む「クリーンな」データベース。

最初のトレーニング


データストレージツールとして、DockerコンテナーでPostgreSQLデータベースを使用し、コンテナーで実行されているpgAdminを介してデータベースと対話します。

docker run --name pg -d -e "POSTGRES_USER=my_user" -e "POSTGRES_PASSWORD=my_password" postgres

pgAdminの実行：

docker run -p 80:80 -e "PGADMIN_DEFAULT_EMAIL=user@domain.com" -e "PGADMIN_DEFAULT_PASSWORD=12345" -d dpage/pgadmin4

すべてが開始した後、構成ファイル（conf / db.ini）でデータベースへの接続文字列を指定することを忘れないでください（トレーニングの例では、できます！）：

[POSTGRESQL]

CSV用アダプター


このアダプターは行をハッシュして、最初の列（トランザクションID）のみを変更せずに残し、受信したデータをdata / transaction_hashed.csvファイルに保存します。彼の作業の最後のステップは、COPYコマンドを使用してこのファイルをreconciliation_dbスキーマの一時テーブルにロードすることです。

最適なファイルの読み取りは、いくつかの並列プロセスによって実行されます。1行ずつ、5メガバイトずつ読み取ります。「5メガバイト」という数値は、経験的な方法で得られたものです。このサイズの1つのテキストで、仮想マシン上の大きなファイルを読み取るための最小の時間を得ることができました。このパラメーターを使用して環境を実験し、稼働時間がどのように変化するかを確認できます。

@m.timing
def process_wrapper(self, chunk_start, chunk_size):
    """ Read a particular chunk """
    with open(self.file_name_raw, newline='\n') as file:
        file.seek(chunk_start)
        lines = file.read(chunk_size).splitlines()
        for line in lines:
            self.process(line)

def chunkify(self, size=1024*1024*5):
    """ Return a new chunk """
    with open(self.file_name_raw, 'rb') as file:
        chunk_end = file.tell()
        while True:
            chunk_start = chunk_end
            file.seek(size, 1)
            file.readline()
            chunk_end = file.tell()

            if chunk_end > self.file_end:
                chunk_end = self.file_end
                yield chunk_start, chunk_end - chunk_start
                break
            else:
                yield chunk_start, chunk_end - chunk_start

@m.timing
def run_reading(self):
    """ The main method for the reading """
    # init objects
    pool = mp.Pool(mp.cpu_count())
    jobs = []

    m.info('Run csv reading...')
    # create jobs
    for chunk_start, chunk_size in self.chunkify():
        jobs.append(pool.apply_async(self.process_wrapper,
                                     (chunk_start, chunk_size)))

    # wait for all jobs to finish
    for job in jobs:
        job.get()

    # clean up
    pool.close()
    pool.join()

    m.info('CSV file reading has been completed')

テーブルのサイズに基づいて、処理に必要なプロセス数が計算され、各プロセス内で10個のタスクに分割されます。

def read_data(self):
    """
    Read the data from the postgres and shared those records with each
    processor to perform their operation using threads
    """
    threads_array = self.get_threads(0,
                                     self.max_id_num_row,
                                     self.pid_max)

    for pid in range(1, len(threads_array) + 1):
        m.info('Process %s' % pid)

        # Getting connection from the connection pool
        select_conn = self._select_conn_pool.getconn()
        select_conn.autocommit = 1

        # Creating 10 process to perform the operation
        process = Process(target=self.process_data,
                          args=(self.data_queque,
                                pid,
                                threads_array[pid-1][0],
                                threads_array[pid-1][1],
                                select_conn))

        process.daemon = True
        process.start()
        process.join()
        select_conn.close()


不一致を検索


2つのアダプターから受信したデータの検証に進みます。

SQL言語のすべての機能を使用して、データベースのサーバー側で調整（または不一致レポートの受信）が行われます。

SQLクエリは非常に複雑ではありません。これは、トランザクション識別子によって、アダプタからそれ自体へのデータをテーブルに結合するだけです。

sql_command = sql.SQL("""
    select
        s1.adapter_name,
        count(s1.transaction_uid) as tran_count
    from {0}.{1} s1
    full join {0}.{1} s2
        on s2.transaction_uid = s1.transaction_uid
        and s2.adapter_name != s1.adapter_name
        and s2.hash = s1.hash
    where s2.transaction_uid is null
    group by s1.adapter_name;""").format(sql.Identifier(self.schema_target),

「クリーンな」トランザクションをストレージテーブルに挿入するだけです。これは、さまざまなアダプタですべての点で（ハッシュによって）一致します。このプロセスは、次のSQLクエリによって実行されます。

sql_command = sql.SQL("""
    with reconcil_data as (
        select
            s1.transaction_uid
        from {0}.{1} s1
        join {0}.{1} s2
            on s2.transaction_uid = s1.transaction_uid
            and s2.adapter_name != s1.adapter_name
        where s2.hash = s1.hash
            and s1.adapter_name = 'postresql_adapter'
    )
    insert into {2}.transaction_log
    select
        t.transaction_uid,
        t.account_uid,
        t.transaction_date,
        t.type_deal,
        t.transaction_amount
    from {3}.transaction_log t
    join reconcil_data r
        on t.transaction_uid = r.transaction_uid
    where not exists
        (
            select 1
            from {2}.transaction_log tl
            where tl.transaction_uid = t.transaction_uid
        )
    """).format(sql.Identifier(self.schema_target),
                sql.Identifier(self.storage_table),
                sql.Identifier(self.schema_db_clean),
                sql.Identifier(self.schema_raw))
