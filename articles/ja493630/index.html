長い間、私は彼について何も書きませんでした。他のプロジェクトは気を散らしており、公開された最後の記事の評価から判断して、関心はすでに低下しています。記事を準備するのは大変な労力を要し、十分な数の読者が必要とする場合にのみ行うのが理にかなっています。

しかし、今は誰もが口先にいるため、誰かを助けることができる1つのことを共有したいという願望がありました。これは私たちの開発ではなく、オールハードウェアデバッグボードを使用したリモート作業に関する作業の一部として研究を行いました。ここで、その結果について説明します。USB / IPプロジェクトは多くの人に知られています。しかし、それは著者によって長い間削減されてきました。最新のドライバーはWIN7でした。今日は、WIN10のオプションをダウンロードする場所と、それをどのようにテストしたかを説明します。さらに、最新のアナログの開発者は、WindowsクライアントだけでなくWindowsサーバーも作成したことを保証します（ただし、このモードではテストを実行しませんでした。タスクにはそれが必要ではありませんでした）。しかし、これは誰かにとっても役立つかもしれません。

前書き


まず、USB / IPとは何かについての短い話。これは、ネットワーク経由でUSBデバイスを転送できるようにする一連のプログラムです。デバイス自体がサーバーに接続されています。クライアントは別のマシンにあります。同時に、クライアントマシン上に、ネットワークで動作するように完全には設計されていないアプリケーションがあります。実際のUSBデバイスが必要です。そして、このデバイスが接続されているという情報を受け取ります。このデバイスには標準ドライバがインストールされています。一般に、クライアントはローカルUSBデバイスで動作すると考えています。

誰かがセキュリティキーを投げています。 JTAGアダプターへのリモートアクセスの可能性を確認しました。

USB / IPプロジェクトは2013年まで活発に開発されていました。その後、Windowsブランチが停止しました。一般的に、バイナリ署名もリリースされました運転者。しかし、それはWindows 7の下にありました。Linuxブランチは開発を続け、このサービスはオペレーティングシステム自体に組み込まれました。少なくとも、それは間違いなくDebianビルドに組み込まれています。さらに、Linuxにはクライアントとサーバーの両方があり、Windowsには元々クライアントのみが作成されていました。Windowsでのサーバーは作られていない。Habréに関する非常に優れた記事

があり、このサービスを使用するためのリファレンスとして、およびこのサービスを使用するためのヒントとして使用できます。

Windowsの現在のバージョンのオプション


しかし、Windows 7がどんなに優れているとしても、それはすでに死んでいます。オールハードウェアに関する作業の一環として、問題の1つを解決するためのさまざまなオプションを検討し、「適合-適合しない」に基づいていくつかの選択肢を確認する必要がありました。チェックに多くの工数を費やすことは不可能でした。また、Windows 10でのドライバーの変更は、それ自体をドラッグする可能性があります。したがって、ネットワーク上で検索が行われ、usbip-winがプロジェクトに導入されました。発見時の最新バージョンは2020年2月23日であり、プロジェクトはライブです。 WIN7とWIN10の両方で構築できます。さらに、元のプロジェクトとは異なり、Windowsクライアントだけでなく、Windowsサーバーも構築できます。

私が確認したところ、プロジェクトは完全に組み立てられて設置されているので、彼とさらに作業が行われました。中のreadmeファイル 自分でビルドしたくない人のための既製のバイナリコードへのリンクがあります。

チェックの悲しい部分：サーバー側


まず、プロジェクトの一環として検証がどのように行われたかを説明します。それだけでは終わりませんでした。Reddコンプレックスの場合にインストールされているST-LINKアダプターを確認しました。これは、コンプレックスがDebianアセンブリのLinux OSを使用しており、このアセンブリには組み込みのUSB / IPサービスが含まれていることをすでに述べたためです。

記事によると、サービスをインストールします。

sudo apt-get update 
sudo apt-get upgrade 
sudo apt-get install usbip

記事の残りの部分では、サービスのロードプロセスを自動化する方法について詳しく説明します。Linuxを理解しているので、すでに何度も書いています。精通していない。私は他人のテキストをスマートな顔で引用する習慣はなく、本質をよく理解していません。したがって、すべてが伝えられているすばらしい記事へのリンクをもう一度思い出し、各OSの起動時に何をしたかを示します（幸い、すべてが確認のためだけに必要でした）。

sudo modprobe usbip-core
sudo modprobe usbip-host
sudo usbipd -D

上記の最初の2つのスペルの目的は不明ですが、これらがないと一部のディレクトリが作成されず、これらのディレクトリがないとUSBポートのエクスポートは行われません。ディレクトリは、システムが再起動されるまで作成されます。したがって、毎回作成する必要があります。3行目-すべてが明確になり、サービスが開始されます。

次に、デバイスの名前を確認します。

user@redd:~$ sudo usbip list -l

私たちはコマンドを与えます：

sudo usbip bind --busid=1-5.4.1.3

すべて、サーバーは動作する準備ができています。

チェックの悲しい部分：クライアントの部分


Windowsにドライバーをインストールします（これは一度だけ行い、その後は常にインストールされます）。これを行うには、引数install：usbip.exe installを指定してusbip.exeファイルを実行し、管理者として デバイスを使用できるかどうかを確認します。





usbip.exe list --remote=192.168.10.123

リストに載っていることを確認します。さて、それを接続してください：

usbip.exe attach --remote=192.168.10.123 --busid=1-5.4.1.3

新しいUSBデバイスがデバイスマネージャーに表示され、Keilはそれを完全に認識します...

しかし、これが楽しい終わりです。小さなプログラムがフラッシュドライブに約1分間注入されます。ラインをステップ実行しようとすると、ラインごとに5〜20秒かかります。これは受け入れがたい。一時停止中、毎秒約50キロビットのトラフィックが両方向に送られます。長く考え抜かれて行きます。

正直なところ、時間制限のために、すべてがそれほどひどい理由を推測するだけになりました。ネットワークを介して実行されているJTAGトラフィックがあると思います。そして、彼は両方向に小さなパケットで走っているため、問題があります。そのため、調査は完了し、「プロジェクトに適さない」という結果になりました。

楽しい部分：準備

結論


usbip-winプロジェクトは、USB / IPプロジェクトの最新の代替品です。彼は生きて成長します。同時に、Windowsのクライアント機能だけでなく、サーバー機能も提供します。 Linuxバージョンとの互換性が保存されます。

リモートUSBデバイスの安定性は予想外に驚くべきものでした。タイムアウトが発生することは確かでした。おそらくそれらはどこかで発生しますが、JTAGアダプターの場合、単一の障害はありませんでした。残念ながら、結果のシステムの速度が遅いため、すべてのUSBデバイスをネットワーク経由で転送できるわけではありません。ただし、JTAGアダプターの場合は、別の方法を検討できます。特に、ST-LINKの代わりにCMSIS-DAP。

両方のレビュー済みプロジェクト（usbip-winおよびCMSIS-DAP）は、ソースコードの形式でGitHubからダウンロードできます。

これが誰かが機器へのリモートアクセスを整理するのに役立つなら、私はうれしいです。Raspberry Piを使用すると、任意の場所に機器を落とすことができます。
