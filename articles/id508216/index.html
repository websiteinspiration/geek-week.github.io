Apa itu CI (Integrasi Berkelanjutan)

CI (Continuous Integration) - secara harfiah diterjemahkan sebagai "integrasi berkelanjutan". Ini mengacu pada integrasi masing-masing potongan kode aplikasi di antara mereka sendiri. Semakin sering kami mengumpulkan kode dan memeriksa:



  • Apakah dia akan pergi?
  • Apakah tes otomatis lulus?


Semua lebih baik! CI memungkinkan Anda melakukan pemeriksaan seperti itu secara otomatis. Ini digunakan dalam tim pengembangan lanjutan yang menulis tidak hanya kode, tetapi juga autotest. Dia ditanya di wawancara - setidaknya pemahaman tentang apa itu. Ya, bahkan dengan penguji.



Karena itu, saya akan memberi tahu dalam sebuah artikel tentang apa itu. Bagaimana CI bekerja dan bagaimana itu akan berguna untuk proyek Anda.

Apa itu CI?



CI adalah perakitan, penyebaran dan pengujian aplikasi tanpa campur tangan manusia. Sekarang saya akan jelaskan dengan sebuah contoh.



Katakanlah kita memiliki dua pengembang - Masha dan Vanya. Dan penguji Katya.



Masha menulis kodenya. Menambahkannya ke sistem kontrol versi (dari Sistem Kontrol Versi Bahasa Inggris , VCS ). Ini adalah sesuatu seperti dropbox untuk kode - lokasi penyimpanan tempat semua perubahan disimpan dan kapan saja Anda dapat melihat siapa yang mengubah apa, kapan.



Kemudian Vanya menyelesaikan fungsinya. Dan juga menyimpan kode dalam VCS.







Tapi ini hanya kode sumber - satu set file dengan ekstensi .java, atau yang lainnya. Jadi Katya dapat menguji perubahan, Anda perlu:



  1. Bangun build dari kode sumber
  2. Jalankan di mesin uji


Membangun build adalah ketika kita membuat satu file yang dapat dieksekusi dari satu set file kode sumber: Anda







dapat membangun build secara manual, tetapi ini adalah wasir yang tidak perlu: ingat bahwa dalam urutan apa untuk menjalankan, file mana yang saling bergantung, jangan membuat kesalahan dalam perintah ... Biasanya mereka menggunakan perintah khusus ... program. Untuk java, ini adalah Ant, Maven, atau Gradle. Dengan bantuan kolektor, Anda mengkonfigurasi proses build sekali, dan kemudian menjalankannya dengan satu perintah. Contoh peluncuran untuk Maven: 



mvn clean install


Ini semi-otomatisasi - Anda masih membutuhkan orang yang akan masuk ke tim dan merakit build dengan "pena". Katakanlah pengembang melakukan ini. Ketika Katya meminta build untuk pengujian, Vanya memperbarui versi dari repositori dan mengumpulkan build.







Tetapi membangun build ‚â† dapatkan aplikasi untuk pengujian. Itu masih perlu diluncurkan! Ini dilakukan oleh server aplikasi. Server berbeda: wildfly, apache, jetty ...



Jika ini wildfly, maka Anda perlu:



  1. Lampirkan direktori build to standalone / deployments
  2. Mulai server (setelah mengatur layanan sekali)


Dan ini lagi semi otomatisasi. Karena pengembang perlu menyalin arsip yang dihasilkan setelah perakitan ke bangku tes dan mengaktifkan layanan. Ya, ini dilakukan oleh beberapa tim, tetapi masih satu orang.







Tetapi jika kita menghapus seseorang dari skema ini, kita mendapatkan CI!







CI adalah aplikasi yang memungkinkan Anda untuk mengotomatisasi seluruh proses. Itu menarik perubahan dari repositori dengan kode. Dengan sendirinya! Ada dua opsi konfigurasi:



  • CI menyurvei repositori ‚ÄúHei, kukuk, apakah Anda ada perubahan ??‚Äù setiap N jam / menit, cara mengkonfigurasi.



  • Repositori melambaikan tangannya pada CI ketika melakukan: "Hai halo! Dan saya mendapat pembaruan di sini! ‚Äù (Apakah itu git hook atau yang setara di VCS Anda)







Ketika CI menerima perubahan, itu mulai membangun build dan autotests.



Jika perakitan gagal (tes jatuh, atau proyek tidak dapat dirakit), sistem menulis email ke semua pihak yang berkepentingan:



  • Manajer Proyek (untuk mengetahui apa yang sedang dilakukan!)
  • Pengembang yang membuat perubahan
  • Orang lain - saat Anda mengaturnya, itu akan terjadi.








Jika build berhasil, CI menyebarkan aplikasi pada mesin uji. Dan pada akhirnya Katka dapat menguji majelis baru!







Ya, tentu saja, begitu Anda harus mengatur semuanya - beri tahu server CI dari mana mendapatkan perubahan, autotest mana yang harus dijalankan, cara membangun proyek, di mana membangunnya nanti ... Tapi kemudian saya mengaturnya sekali - dan kemudian melakukannya!



Autotests juga harus ditulis sendiri, tetapi dilakukan dengan baik =)







Jika ada di jari Anda, maka sistem CI ( Continuous Integration ) adalah sejenis program yang memantau Kontrol Sumber Anda, dan ketika perubahan muncul di sana, secara otomatis menyatukannya, membangunnya, menjalankan tes otomatis (tentu saja, jika itu ditulis).



Dalam hal kegagalan, ini membuat ini diketahui oleh semua pihak yang berkepentingan, di tempat pertama sampai penglaju terakhir. (c) habr.com/en/post/352282






Program CI



Yang paling populer adalah Jenkins dan TeamCity.



Tetapi ada banyak variasi lain - CruiseControl, CruiseControl.Net, Atlassian Bamboo, Hudson, Microsoft Team Foundation Server.





Seperti apa bentuknya



Mari kita lihat tampilannya dari sudut pandang pengguna. Saya akan menunjukkan dengan contoh sistem TeamCity.



Ketika saya masuk, saya melihat semua tugas. Tugasnya berbeda:



  • Membangun membangun
  • Drive Tes Otomatis
  • Menyebarkan aplikasi di bangku tes
  • Jalankan tes GUI (atau tes tukang pos) pada dudukan ini
  • Beri tahu semua orang yang tertarik melalui email tentang hasil perakitan dan pengujian


Tugas dapat dikelompokkan. Di sini, katakanlah, kami memiliki proyek CDI. Masuk ke dalam, saya melihat tugas-tugas untuk proyek khusus ini:

  • CDI Core dengan tes - perakitan proyek dengan semua uji coba , yang, seperti dapat dilihat di layar, 4000+ buah. Tes membutuhkan waktu setengah jam, tetapi juga dijalankan untuk setiap komit.

Oleh karena itu, bahkan jika saya tidak berlangganan email peringatan tentang status majelis, saya dapat dengan mudah melihat dalam kondisi apa sistem sekarang. Anda membuka antarmuka grafis dari program dan melihat.





Bagaimana CI bekerja



Bagaimana dan di mana CI mengumpulkan pembuatan dan menjalankan tes otomatis? Saya akan menggunakan TeamCity sebagai contoh, tetapi sistem lain bekerja dengan cara yang sama.



TeamCity sendiri tidak mengumpulkan apa pun. Perakitan dan pelaksanaan autotest dilakukan pada mesin lain yang disebut "agen":







"Agen" adalah komputer sederhana. Sepotong besi atau mesin virtual, bukan itu intinya. Tetapi bagaimana komputer ini memahami apa yang perlu dilakukan?



TeamCity memiliki server dan klien. Server adalah aplikasi di mana Anda kemudian akan menyodok tombol dan menonton gambar yang indah "bagaimana semuanya berjalan dengan baik". Itu diinstal pada satu mesin.







Dan aplikasi "klien" diinstal pada mesin "agen". Dan ketika kita mengklik tombol "Jalankan" di server:







Server memilih klien gratis dan memberikannya semua petunjuk: apa yang sebenarnya perlu dilakukan. Klien mengumpulkan build, melakukan autests, mengumpulkan hasilnya dan kembali ke server: "On, hold, draw".







Server menampilkan hasilnya kepada pengguna plus mengirim email ke semua pihak yang berkepentingan.



Pada saat yang sama, kami selalu melihat agen tertentu yang dirakit:







Dan Anda dapat memilih sendiri di mana untuk menjalankan tes otomatis. Karena kebetulan autotest hanya dimiliki oleh satu agen bangunan. Ini berarti bahwa dia memiliki sesuatu yang salah dengan konfigurasi.

Kami juga dapat mengonfigurasi setiap agen daftar agen yang dapat dijalankannya. Untuk apa? Sebagai contoh, kami memiliki setengah dari agen Linux, dan setengah dari Windows. Dan kami melakukan perakitan hanya untuk satu sistem. Atau di Windows semacam bug mengambang merangkak keluar, tetapi memperbaikinya lama dan mahal, dan semua klien di Linux - baik, mengapa?



Dan itu juga terjadi bahwa agen dibagi antara proyek sehingga tidak ada pertarungan - proyek ini menggunakan Burana dan Apollo, dan bahwa Chip dan Dale. Masalahnya adalah bahwa hanya satu tugas yang dapat dilakukan pada satu agen. Oleh karena itu, tidak masuk akal untuk membeli mobil keren untuk agen, segera Anda tidak bisa mengendarai banyak tes di sana.



Pada akhirnya, cara kerjanya: pertama, admin membeli komputer di bawah "agen" dan menginstal aplikasi klien TeamCity pada mereka. Mereka tidak boleh terlalu keren, karena mereka tidak akan melakukan banyak tugas sekaligus.







Dalam hal ini, TeamCity Anda membayar jumlah lisensi untuk agen bangunan. Jadi semakin sedikit mereka, semakin baik.



Pada mesin yang terpisah, admin menginstal server TeamCity. Dan mengkonfigurasinya - mengatur rakitan, menunjukkan rakitan mana yang dapat didorong pada mesin mana, dll. Server membutuhkan tempat untuk menyimpan artefak - hasil perakitan.



Kami memiliki dua proyek - Klien dan Faktor Tunggal, yang saling berinteraksi. Penguji Klien Tunggal mungkin tidak mengumpulkan Faktor secara lokal. Dia meluncurkan build di TeamCity dan mengunduh build yang sudah selesai dari artefak!


Lebih lanjut, pengembang memilih rakitan mana yang ingin dia jalankan dan klik "Run". Apa yang terjadi pada saat ini:



1. Server TeamCity memeriksa daftar agen mana yang dapat dijalankan oleh majelis ini. Kemudian dia memeriksa agen mana yang saat ini gratis:







Ditemukan gratis? Saya memberinya teka-teki!







Jika semua agen sibuk, tugasnya akan antri. Antrian berfungsi sesuai dengan prinsip FIFO - masuk pertama, keluar pertama. Siapa yang pertama kali bangun - itu dan sandal.







Antrian dapat disesuaikan secara manual. Jadi, jika saya melihat bahwa antrian tersumbat dengan majelis yang diluncurkan sistem kontrol versi, saya akan menaikkan milik saya ke bagian paling atas. Jika saya melihat orang mulai berkumpul, itu artinya mereka juga penting, Anda harus menunggu.







Ini adalah praktik normal jika kapasitas agen tidak cukup di seluruh dan antrian dibuat. Anda melihat siapa yang meluncurkannya:



  • Robot? Jadi, ini hanya pemeriksaan rutin bahwa tidak ada lagi yang rusak. Ini bisa menunggu 5-10-30 menit, tidak apa-apa
  • Rekan? Majelis ini penting baginya, karena dia tidak menunggu peluncuran yang direncanakan. Kami berdiri dalam antrean, Anda tidak harus maju.


Kadang-kadang Anda bahkan dapat membatalkan pembuatan dari sistem kontrol versi, jika benar-benar panas, dan semua agen sibuk dengan tes per jam. Dalam hal ini, Anda dapat:



  • naikkan giliran Anda ke bagian paling atas sehingga dimulai pada agen bebas pertama
  • masuk ke agen, batalkan bangunan saat ini
  • restart dia! Meskipun akan jatuh ke bagian paling bawah dari garis, itu hanya jelek untuk membatalkan rakitan


2. Agen melakukan tugas dan mengembalikan hasilnya ke server







3. Server menarik hasilnya dalam antarmuka grafis dan menyimpan artefak. Jadi saya bisa masuk ke TeamCity dan melihat di artefak log lengkap dari autotests sebelumnya, atau mengunduh perakitan proyek untuk menyebarkannya secara lokal.



Sangat disarankan agar Anda melakukan pra-konfigurasi jumlah rakitan yang akan disimpan CI. Karena jika artefak berisi build dari 200+ mb dan akan ada banyak, maka peluncuran perakitan berikutnya akan gagal dengan kesalahan "kehabisan ruang disk":







4. Server melakukan email - itulah cara Anda mengaturnya. Dia dapat membuat buletin positif "majelis berhasil," dan dia dapat mengirim surat hanya jika terjadi kegagalan: "Oh, dia, ada yang salah!".





Integrasi dengan VCS



Saya berbicara tentang berbagai opsi untuk mengkonfigurasi integrasi CI-VCS:



  • CI menyurvei repositori ‚ÄúHei, kukuk, apakah Anda ada perubahan ??‚Äù setiap N jam / menit, cara mengkonfigurasi.



  • Repositori melambaikan tangannya pada CI ketika melakukan: "Hai halo! Dan saya mendapat pembaruan di sini! ‚Äù (Apakah itu git hook atau yang setara di VCS Anda)







Tetapi kapan yang digunakan?



Yang terbaik dari semuanya, tentu saja, adalah untuk sistem kontrol versi untuk memberi tahu server CI. Dan jalankan seluruh siklus untuk setiap perubahan: assemble, test, deploy. Kemudian setiap perubahan pada kode akan segera menuju ke lingkungan pengujian, yang akan serelevan mungkin.







Plus, setiap perubahan menjalankan uji otomatis. Dan jika tes gagal, itu segera menghapus komit siapa yang melanggarnya. Lagipula, itu digunakan untuk bekerja dan setelah pengeditan Vasya tiba-tiba pecah - itu berarti bahwa komitnya menyebabkan jatuh. Dengan pengecualian langka, saat musim gugur melayang.







Namun dalam kehidupan nyata, skema seperti itu jarang berlaku. Bayangkan saja - Anda dapat memiliki banyak proyek, banyak pengembang. Setiap melakukan sesuatu setidaknya sekali setiap setengah jam. Dan jika Anda menjalankan 10 majelis dalam waktu setengah jam untuk setiap komit, antrian di TeamCity tidak akan pernah tersapu!



Salah satu produk kami memiliki modul inti, dan ada 15+ Pelanggan. Masing-masing memiliki autotest sendiri. Perakitan pelanggan adalah fitur inti + pelanggan. Artinya, perubahan dalam proyek root dapat mempengaruhi 15 majelis berbeda. Jadi, mereka semua harus dijalankan ketika melakukan di core.



Ketika kami memiliki 4 agen build, all-all-all build dan tes untuk pelanggan ini diluncurkan pada Selasa malam. Dan pada jam 10 pagi, masih ada beberapa jam di TeamCity.


Pilihan lain adalah membeli banyak agen. Tapi ini harga untuk mobil itu sendiri + untuk lisensi di TeamCity, yang sudah jauh lebih mahal, dan bahkan membayar setiap bulan.



Oleh karena itu, mereka biasanya melakukan sebagai berikut:



1. Majelis yang sangat cepat dan penting dapat dibiarkan melakukan komitmen apa pun - jika dibutuhkan 1-2 menit, biarkan mereka mengejar.







2. Sisa build memeriksa untuk melihat apakah ada perubahan dalam VCS - misalnya, setiap 15 menit sekali. Jika ada, maka kami luncurkan.







3. Tes panjang (misalnya, tes kinerja) - sekali setiap beberapa hari di malam hari.











CI dalam pengujian



Jika kita berbicara tentang mengembangkan aplikasi kita, maka pengujian termasuk dalam siklus standar. Anda atau pengembang Anda menulis autotests, yang kemudian didorong oleh CI. Ini bisa berupa tes unit, api, gui atau stress.







Tetapi bagaimana jika Anda menguji kotak hitam? Ada aplikasi, tidak ada kode sumber. Ini adalah kenyataan pahit penguji integrasi - pemasok memberi Anda rilis baru aplikasi, yang perlu Anda periksa sebelum dimasukkan ke dalam produksi.



Di sini, katakanlah Anda memiliki tes API di tukang pos. Atau tes GUI di Selenium. Bisakah saya menyesuaikan loop CI untuk mereka?







Tentu saja Anda bisa!



CI tidak menetapkan batasan ketat seperti "Saya hanya bekerja di proyek dengan tes otomatis" atau "Saya hanya bekerja ketika saya memiliki akses ke kode sumber". Mungkin melihat ke dalam sistem kontrol versi, atau mungkin tidak melihat. Ini adalah kondisi opsional!



Menulis autotes? Beri tahu server CI cara menjalankannya sering - dan nikmati hasilnya =)









Total



CI - integrasi berkelanjutan. Inilah saat aplikasi Anda terus-menerus diuji: apakah semuanya baik-baik saja? Apakah tes lulus? Apakah perakitan akan berlangsung? Selain itu, semua cek dilakukan secara otomatis, tanpa campur tangan manusia.



Ini sangat penting untuk tim di mana beberapa pengembang bekerja pada kode satu aplikasi. Bagaimana ini bisa terjadi? Secara terpisah, bagian dari program ini bekerja, tetapi bersama tidak lagi. CI memungkinkan Anda untuk dengan cepat mendeteksi masalah seperti itu. Dan semakin cepat Anda menemukannya, semakin murah untuk memperbaikinya.







Karenanya nama - cek konstan integrasi potongan kode di antara mereka sendiri.

Tugas CI yang khas:



  • Periksa apakah ada pembaruan dalam kode
  • Membangun membangun
  • Drive Tes Otomatis
  • Menyebarkan aplikasi di bangku tes
  • Jalankan tes GUI (atau tes tukang pos) pada dudukan ini
  • Beri tahu semua orang yang tertarik melalui email tentang hasil perakitan dan pengujian








Dan semua ini - secara otomatis, tanpa campur tangan manusia! Yaitu, mereka mengaturnya sekali, dan kemudian itu sendiri.



Jika CI dikonfigurasi dalam proyek, Anda akan memiliki tegakan uji yang selalu terbaru. Dan jika ada yang rusak dalam kode, Anda akan segera mengetahuinya, server CI akan mengirim surat. Dan Anda bisa pergi ke antarmuka grafis dan melihat apakah semua build berhasil, dan tesnya hijau? Nilai gambar proyek dalam satu menit.
