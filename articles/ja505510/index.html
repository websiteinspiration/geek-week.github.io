Python SDKによるCheck Point APIの合理化

APIとの相互作用のすべての力は、API要求とAPI応答を分析するためのツールを動的に生成することが可能になるときに、プログラムコードと組み合わせて使用​​すると明らかになります。ただし、Check Point Management API用のPythonソフトウェア開発キット（以下、Python SDKと呼びます）は表示されません。開発者やオートメーション愛好家の生活を大幅に簡素化します。最近、Pythonが非常に人気を博しているので、そのギャップを埋めて、Check Point API Python Development Kitの主な機能の概要を説明することにしました。この記事は、HabréCheck Point R80.10 APIに関する別の記事を補足する優れた記事です。 CLI、スクリプトなどによる管理。Python SDKを使用してスクリプトを作成する方法を検討し、バージョン1.6（R80.40以降でサポート）の新しい管理API機能について詳しく説明します。この記事を理解するには、APIとPythonの操作に関する基本的な知識が必要です。

チェック・ポイントは積極的にAPIを開発しており、現時点では次のものが存在します。

  • チェック・ポイント管理API（現在のバージョン1.6） -API（および管理サーバーの制御下にあるゲートウェイでスクリプトを実行する機能）を介して管理サーバーと連携します。
  • Check Point GAIA API（現在のバージョン1.4） -セキュリティゲートウェイと連携
  • Threat Prevention API 1.0 -Check Point Cloudのサンドボックス
  • Identity Awareness API-ゲートウェイでIdentity Awarenessブレードを操作する
  • セキュリティ管理ポータルAPI -SMBゲートウェイ管理ポータルの操作（SMBゲートウェイの詳細）
  • IoT API -IoTコントローラーとの相互作用
  • CloudGuard Connect API - CloudGuard Connect（SD-WANセキュリティソリューション）との連携
  • Dome9のAPI -での作業Dome9

Python SDKは現在、Management APIおよびGaia APIとの相互作用のみをサポートしています。このモジュールでは、最も重要なクラス、メソッド、変数について説明します。



モジュールのインストール


cpapi モジュールは、pipを使用してgithub上の公式のCheck Pointリポジトリからすばやく簡単にインストールされます。詳細なインストール手順はREADME.mdにあります。このモジュールは、Python 2.7および3.7のバージョンで動作するように適合されています。この記事では、Python 3.7を使用して例を示します。ただし、Python SDKはCheck Point（スマート管理）管理サーバーから直接実行できますが、Python 2.7のみをサポートしているため、最後のセクションではバージョン2.7のコードを提供します。モジュールをインストールした直後に、examples_python2およびexamples_python3ディレクトリの例を確認することをお勧めします。

仕事の始まり


cpapiモジュールのコンポーネントと仕事への私たちのためには、我々は、インポートする必要が少なくとも二つ必要なクラスからcpapiのモジュール：

APIClientとAPIClientArgsを

from cpapi import APIClient, APIClientArgs

APIClientArgs クラスは APIサーバーへの接続を担当し、APIClientクラスは APIとの対話を担当します。

接続パラメータを決定します


APIに接続するためのさまざまなパラメーターを定義するには、APIClientArgsクラスをインスタンス化する必要があります。原則として、そのパラメーターは事前定義されており、管理サーバーでスクリプトを実行するときに省略できます。

client_args = APIClientArgs()

ただし、サードパーティのホストで開始する場合は、少なくともAPIサーバー（管理サーバーでもあります）のIPアドレスまたはホスト名を指定する必要があります。次の例では、サーバー接続パラメーターを定義し、管理サーバーのIPアドレスを文字列として割り当てます。

client_args = APIClientArgs(server='192.168.47.241')

APIサーバーに接続するときに使用できるすべてのパラメーターとそのデフォルト値を見てみましょう。

APIClientArgsクラスの__init__メソッドへの引数 
class APIClientArgs:
    """
    This class provides arguments for APIClient configuration.
    All the arguments are configured with their default values.
    """

    # port is set to None by default, but it gets replaced with 443 if not specified
    # context possible values - web_api (default) or gaia_api
    def __init__(self, port=None, fingerprint=None, sid=None, server="127.0.0.1", http_debug_level=0,
                 api_calls=None, debug_file="", proxy_host=None, proxy_port=8080,
                 api_version=None, unsafe=False, unsafe_auto_accept=False, context="web_api"):
        self.port = port
        # management server fingerprint
        self.fingerprint = fingerprint
        # session-id.
        self.sid = sid
        # management server name or IP-address
        self.server = server
        # debug level
        self.http_debug_level = http_debug_level
        # an array with all the api calls (for debug purposes)
        self.api_calls = api_calls if api_calls else []
        # name of debug file. If left empty, debug data will not be saved to disk.
        self.debug_file = debug_file
        # HTTP proxy server address (without "http://")
        self.proxy_host = proxy_host
        # HTTP proxy port
        self.proxy_port = proxy_port
        # Management server's API version
        self.api_version = api_version
        # Indicates that the client should not check the server's certificate
        self.unsafe = unsafe
        # Indicates that the client should automatically accept and save the server's certificate
        self.unsafe_auto_accept = unsafe_auto_accept
        # The context of using the client - defaults to web_api
        self.context = context


APIClientArgsクラスのインスタンスで使用できる引数は、チェックポイントの管理者にとって直感的であり、追加のコメントは必要ないと思います。

APIClientとコンテキストマネージャーを介して接続する


APIClient クラスは、コンテキストマネージャを通じて最も便利に使用されます。APIClientクラスのインスタンスに渡す必要があるのは、最後のステップで定義された接続パラメーターだけです。

with APIClient(client_args) as client:

コンテキストマネージャは、APIサーバーへのログイン呼び出しを自動的には行いませんが、終了するとログアウト呼び出しを行います。なんらかの理由でAPI呼び出しの終了時にログアウトが不要な場合は、コンテキストマネージャを使用せずに作業を開始する必要があります。

client = APIClient(clieng_args)


接続チェック


接続が特定のパラメータを通過するかどうかを確認する最も簡単な方法は、check_fingerprintメソッドを使用することです。サーバーAPI証明書のフィンガープリントのsha1ハッシュの検証に合格しなかった場合（メソッドがFalseを返した場合）、これは通常、接続の問題が原因であり、プログラムを停止できます（またはユーザーが接続データを修正できるようにします）。

    if client.check_fingerprint() is False:
        print("Could not get the server's fingerprint - Check connectivity with the server.")
        exit(1)

今後、APIClientクラスは、APIサーバー上のAPI 呼び出し（api_callおよびapi_queryメソッド、後でさらに説明します）sha1フィンガープリント証明書をチェックすることに注意してください。ただし、サーバーAPI証明書のsha1フィンガープリントを確認するときにエラーが検出された場合（証明書が不明であるか変更されている場合）、check_fingerprintメソッドは、ローカルマシンで自動モードでそれに関する情報を追加/変更する機能を提供します。このチェックは完全にオフにすることができます（ただし、スクリプトが127.0.0.1に接続されているAPIサーバー自体、上で実行されている場合にのみ、これはお勧めすることができます）APIClientArgs引数を使用して- unsafe_auto_accept （前述の「接続パラメーターの定義」のAPIClientArgsの詳細を参照してください）。

client_args = APIClientArgs(unsafe_auto_accept=True)

APIサーバーにログイン

ログイン方法


ログインとパスワードを使用するオプション（例では、ユーザー名adminとパスワード1q2w3eが位置引数として渡されます）：

     login = client.login('admin', '1q2w3e')

オプションのオプションパラメータは、loginメソッドでも使用できます。名前とデフォルト値をリストします。 

continue_last_session=False, domain=None, read_only=False, payload=None

Login_with_api_keyメソッド


apiキーを使用するバリアント（管理バージョンR80.40 /管理API v1.6以降でサポート、「3TsbPJ8ZKjaJGvFyoFqHFA ==」は、APIキー認証方法を使用する管理サーバーのユーザーの1つのAPIキーの値です）：

     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')

login_with_api_key メソッドでは、loginメソッドと同じオプションのパラメーターを使用できます。

Login_as_rootメソッド


APIサーバーを備えたローカルマシンへのログインオプション：

     login = client.login_as_root()

このメソッドで使用できるオプションのパラメーターは2つだけです。

domain=None, payload=None

API呼び出し結果の処理


その後、APIResponseクラスの変数とメソッドを使用できます（コンテキストマネージャーの内部と外部の両方）。APIResponseクラスには、 4つのメソッドと5つの変数が事前定義されています。最も重要なものについて詳しく説明します。



成功


まず、API呼び出しが成功し、結果が返されたことを確認してください。これには成功方法があります：

In [49]: api_versions.success                                                   
Out[49]: True

API呼び出しが成功した場合（応答コードは200）はTrueを返し、失敗した場合（その他の応答コード）はFalseを返します。API呼び出しの直後に使用すると、応答コードに応じてさまざまな情報を表示できるので便利です。

if api_ver.success: 
    print(api_versions.data) 
else: 
    print(api_versions.err_message)

ステータスコード


API呼び出しを行った後に応答コードを返します。

In [62]: api_versions.status_code                                               
Out[62]: 400

可能な応答コード：200,400,401,403,404,409,500,501。

set_success_status


成功ステータスの値を変更する必要がある場合があります。技術的には、通常の文字列であっても、何でも置くことができます。しかし、実際の例は、特定の付随する条件下でこのパラメーターをFalseにリセットすることです。以下では、管理サーバーで実行されるタスクがある場合の例に注意してください。ただし、このリクエストは失敗したと見なします（API呼び出しが成功し、コード200を返したにもかかわらず、success変数をFalseに設定します）。

for task in task_result.data["tasks"]:
    if task["status"] == "failed" or task["status"] == "partially succeeded":
        task_result.set_success_status(False)
        break

有用な例


以下は、Management API 1.6に追加されたAPI呼び出しを使用する例です。

はじめに、add-hostおよびadd-address-range呼び出しの操作を検討してください。サブネット192.168.0.0/24のすべてのIPアドレスをホストタイプのオブジェクトとして作成し、最後のオクテットが5であり、他のすべてのIPアドレスをアドレス範囲オブジェクトとして書き込む必要があるとします。この場合、サブネットアドレスとブロードキャストアドレスは除外する必要があります。

したがって、以下はこの問題を解決し、ホストタイプのオブジェクトを50個、アドレスのタイプ範囲のオブジェクトを51個作成するスクリプトです。この問題を解決するには、101のAPI呼び出しが必要です（最終的なpublish呼び出しは含まれません）。また、timeitモジュールを使用して、変更が送信されるまでスクリプトの実行にかかる時間を計算します。

add-hostおよびadd-address-rangeを使用したスクリプト 
import timeit
from cpapi import APIClient, APIClientArgs

start = timeit.default_timer()

first_ip = 1
last_ip = 4

client_args = APIClientArgs(server="192.168.47.240")

with APIClient(client_args) as client: 
     login = client.login_with_api_key('3TsbPJ8ZKjaJGvFyoFqHFA==')
     for ip in range(5,255,5):
         add_host = client.api_call("add-host", {"name" : f"h_192.168.0.{ip}", "ip-address": f'192.168.0.{ip}'})
     while last_ip < 255:
         add_range = client.api_call("add-address-range", {"name": f"r_192.168.0.{first_ip}-{last_ip}", "ip-address-first": f"192.168.0.{first_ip}", "ip-address-last": f"192.168.0.{last_ip}"})
         first_ip+=5
         last_ip+=5
     stop = timeit.default_timer() 
     publish = client.api_call("publish")
     
print(f'Time to execute batch request: {stop - start} seconds')


私のラボ環境では、管理サーバーの負荷に応じて、このスクリプトの実行に30〜50秒かかります。

次に、APIバージョン1.6にサポートが追加されたadd-objects-batch call APIを使用して同じ問題を解決する方法を見てみましょう。この呼び出しにより、1つのAPIリクエストで一度に多くのオブジェクトを作成できます。さらに、これらは異なるタイプのオブジェクト（たとえば、ホスト、サブネット、アドレス範囲）である可能性があります。したがって、1つのAPI呼び出しのフレームワークでタスクを解決できます。

add-objects-batchを使用したスクリプト

from cpapi import APIClient, APIClientArgs

objects_list_ip = []
objects_list_range = []

for ip in range(5,255,5):
    data = {"name": f'h_192.168.0.{ip}', "ip-address": f'192.168.0.{ip}'}
    objects_list_ip.append(data)
    
first_ip = 1
last_ip = 4


while last_ip < 255:
    data = {"name": f"r_192.168.0.{first_ip}-{last_ip}", "ip-address-first": f"192.168.0.{first_ip}", "ip-address-last": f"192.168.0.{last_ip}"}
    objects_list_range.append(data)
    first_ip+=5
    last_ip+=5

data_for_batch = {
  "objects" : [ {
