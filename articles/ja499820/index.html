ターゲット言語の1つとしてKotlinを選択した理由。パート2：Kotlinマルチプラットフォーム

開発プロセスへのKotlin言語の導入に関する一連の記事を続けます。ここで最初の部分を探してください。

2017年、Jetbrainsの野心的なプロジェクトは日の目を見て、クロスプラットフォーム開発に新しい視点を提供しました。 kotlinコードをさまざまなプラットフォームのネイティブコードにコンパイルする！私たちDomklikは、開発プロセスを最適化する方法を常に模索しています。コードを再利用するより良いことは何だと思いましたか？そうです-コードをまったく書かないでください。そして、すべてがあなたが望むように機能するように。しかし、これは起こりませんが。そして、多くの労力を費やすことなく、さまざまなプラットフォームに単一のコードベースを使用できるようにするソリューションがある場合、試してみませんか？

みなさん、こんにちは！私の名前はジェンナディヴァシルコフです。私はDomklikのAndroid開発者です。今日、モバイルデバイス用のKotlin Multiplatformの開発における私たちの経験を共有したいと思います。どのような問題が発生したか、どのように解決したか、そして何に至ったかを教えてください。このトピックは、Kotlin MPP（マルチプラットフォームプロジェクト）を試してみたい人、またはすでに試してみたが、まだ完成していない人にとっては、興味深いものになるでしょう。持ってきても、思い通りではない。開発したライブラリの開発と配信のプロセスをどのように調整するかというビジョンを伝えようと思います（それらの1つを例として使用して、Kotlin MPPでの開発パスの始まりを説明します）。

どのように私たちがそれをしたのか、物語はいかがですか？あります！



テクノロジーについて簡単に


まだ聞いたことがない、またはトピックに没頭していない人のために、Javaの世界でハングアップしてKotlinの世界に行きます（または行きませんが、のぞき見します）。

当然のことながら、開発はIntelliJ IDEAまたはAndroid StudioのKotlin言語で行われています。プラスは、すべてのAndroid開発者（少なくとも私たちと一緒に）が言語とこれらの素晴らしい開発環境の両方を知っており、愛していることです。

また、結果のコードを各プラットフォームに固有の言語（OBJ-C、JS、JVMですべてが明らか）にコンパイルする際の大きなプラスになります。

つまり、すべてがクールです。私の意見では、Kotlin Multiplatformはビジネスロジックをライブラリに取り込むのに理想的です。アプリケーションを完全に作成することも可能ですが、今のところ極端に見えますが、ライブラリ開発は私たちのような大規模プロジェクトに適しています。

私たちの経験


したがって、ある時点で、すべてがKotlin MPP（その後、Kotlin / Nativeとも呼ばれます）によって標準として受け入れられると判断し、共通のコードを取り出すライブラリを作成し始めました。最初はモバイルプラットフォーム専用のコードでしたが、ある時点で、jvmバックエンドのサポートが追加されました。Androidでは、開発されたライブラリの開発と公開は問題を引き起こしませんでしたが、iOSでは実際にいくつかの問題が発生しましたが、問題は解決され、フレームワークを開発および公開するための作業モデルが開発されました。

発生の背景


すべてのモバイルアプリケーションが一連の分析を収集することは秘密ではありません。イベントはすべての重要なイベントではなくハングアップします（たとえば、このアプリケーションでは、600を超えるさまざまなメトリックが収集されます）。そして、メトリック収集とは何ですか？簡単に言うと、これは特定のキーを持つイベントを分析エンジンの腸に送信する関数の呼び出しです。そして、それはfirebase、appmetricaなどのさまざまな分析システムに入ります。これにはどのような問題がありますか？ 2つのプラットフォームで同じ（プラスまたはマイナス）コードを常に複製する！はい、そして人的要因はどこにも見つかりません-開発者は、キーの名前とイベントで送信されるメタデータセットの両方で誤っている可能性があります。これは明らかに一度書かれ、各プラットフォームで使用される必要があります。一般的なロジックとテスト技術を作成するための理想的な候補（これはKotlin Mppのテストペンです）。

実装方法


イベント自体をライブラリに転送し（ワイヤードキーと引数に必要なメタデータのセットを持つ関数の形式で）、これらのイベントを処理するための新しいロジックを記述しました。イベント形式が統一され、ハンドラーエンジン（バス）が作成され、すべてのイベントがそこに注がれました。また、さまざまな分析システムについて、アダプターリスナーは次のように書いています（非常に便利なソリューションは、新しい分析を追加するときに、すべてのイベントを、または選択的に、新しい分析に簡単にリダイレクトできることです）。



実装プロセスに興味がある


iOS向けKotlin / Nativeでのストリーム操作の実装の特殊性のため、Kotlinオブジェクトをシングルトンとして取得して操作し、いつでもそこにデータを書き込むことはできません。スレッド間で状態を転送するすべてのオブジェクトはフリーズする必要があります（これにはフリーズ（）関数があります）。しかし、ライブラリーは、とりわけ、アプリケーションの存続期間中に変化する可能性がある状態を保管します。この状況を解決するにはいくつかの方法がありますが、最も簡単な方法で解決しました。

Common:

expect class AtomicReference<T>(value_: T) {
    var value: T
}

Android:

actual class AtomicReference<T> actual constructor(value_: T) {
    actual var value: T = value_
}

iOS:

actual typealias AtomicReference<V> = kotlin.native.concurrent.AtomicReference<V>

このオプションは、単純な状態の保存に適しています。私たちの場合、プラットフォーム構成は保存されています：

object Config {
    val platform = AtomicReference<String?>("")
    var manufacturer = AtomicReference<String?>("")
    var model = AtomicReference<String?>("")
    var deviceId = AtomicReference<String?>("")
    var appVersion = AtomicReference<String?>("")
    var sessionId = AtomicReference<String?>("")
    var debug = AtomicReference<Boolean>(false)
}

なぜこれが必要なのですか？アプリケーションの開始時に、モジュールがすでにロードされている必要があり、そのオブジェクトがすでにフリーズ状態にある場合、一部の情報は取得されません。この情報を構成に追加し、iOSのさまざまなストリームから読み取ることができるようにするには、そのような移動が必要です。

ライブラリの公開という問題にも直面しました。そして、Androidにこれで問題がなければ、組み立てられたフレームワークをiOSプロジェクトに直接接続するために当時公式文書によって提案された方法がいくつかの理由で私たちに適さなかったので、新しいバージョンをできるだけシンプルで透過的にしたかったのです。さらに、バージョン管理もサポートされています。
解決策は、ポッドファイルを介してフレームワークを接続することです。これを行うために、彼らはpodspecファイルとフレームワーク自体を生成し、それらをリポジトリーに入れて、ライブラリーをプロジェクトに接続することで、iOS開発者にとって非常にシンプルで便利になりました。また、開発の便宜上、いわゆるファットフレームワーク（実際にはすべてのアーキテクチャと、kotlinプラグインによって生成された追加のメタファイルが連携する太字のバイナリ）のすべてのアーキテクチャの単一のアーティファクトを収集します。スポイラーの下でのこのすべての実装：
