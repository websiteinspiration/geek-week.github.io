Java 14のJPackage用Mavenプラグイン

こんにちは！

つい最近、新しいJava 14がリリースされました。興味深い機能の中で、私はjpackage packer（インキュベーター中）に興味を持っていました。

この機能の意味は、アプリケーション、すべての依存関係とランタイムを取得し、配布、インストール、実行が簡単な1つのインストールファイルをアセンブルすることです。

ツールの使用方法の説明は記事にあります。

パッカーで遊んだ後、ペットプロジェクトに実装したいと思いました。プロジェクトはMavenを使用して作成されたため、プラグインを作成することが決定されました。私がMavenのプラグインを作成するのはこれが初めてなので、このタスクは私にとって二重に興味深いものになりました。

プラグインのトピックをググった後、プラグインを書くことは難しくないことが明らかになりました：AbstractMojoクラスから継承し、void execute（）メソッドを実装し、Mojoアノテーションをぶら下げ、プラグインの準備が整いました。

結果はそのような空白です：

@Mojo(name = "createpackage", defaultPhase = LifecyclePhase.VERIFY, threadSafe =true)
public class JpackageMojo extends AbstractMojo {

   @Override
   public void execute() {
        //    .
   }
}

「Createpackage」はターゲットの名前です。
「LifecyclePhase.VERIFY」 -検証時に実行されます。

.exeインストーラーをビルドするには、次のコマンドを実行する必要があります。

jpackage --name myapp --input lib --main-jar main.jar

つまり、出力でのファイル名、アプリケーションのあるフォルダー、およびmainメソッドでのdzharnikを指定する必要があります。このデータを取得するには、クラスに次の行を追加します。

@Parameter(defaultValue = "${project.build.directory}")
private File buildDirectory;
@Parameter( defaultValue = "${project}", readonly = true )
private MavenProject project;

ドキュメントには、ツールはjava.util.spi.ToolProvider.findFirst（ "jpackage"）から利用できると記載されていますが、使用できませんでした。おそらく、これはjpackageがインキュベーター内にあるためです。誰かが解決策を知っている場合は、コメントで共有してください。

まあ、それが美しくできない場合は、Runtime.getRuntime（）.Exec（コマンド）を使用して醜くします。簡単にするために、jpackage.exeを環境変数に追加することをお勧めします。

実行する結果のコマンド：

jpackage.exe --name name --input path --main-jar main --dest dest --win-menu --win-shortcut

いくつかのフラグが追加されています。--dest destは結果の場所を示し、-win-menu --win-shortcutは私とWindowsのデスクトップへのリンクを追加します。

その結果、私は以下を得ました：

@Mojo(name = "createpackage", defaultPhase = LifecyclePhase.VERIFY, threadSafe = true)
public class JpackageMojo extends AbstractMojo {

   public static final String ERROR_CREATE_PACKAGE = "Error create package";

   @Parameter(defaultValue = "${project.build.directory}")
   private File buildDirectory;
   @Parameter( defaultValue = "${project}", readonly = true )
   private MavenProject project;

   @Override
   public void execute() {
       var name = project.getArtifactId();
       var path = buildDirectory.getPath();
       var main = project.getArtifact().getFile().getName();
       var dest = path + "\\dest";
       try {
           String command  = "jpackage.exe --name " + name + " --input "
                   + path + " --main-jar " + main + " --dest " + dest
                   + " --win-menu --win-shortcut";
           getLog().debug(command);
           Process proc = Runtime.getRuntime().exec(command);
           getLog().info(Arrays.toString(proc.getInputStream().readAllBytes()));
           proc.waitFor();
           proc.destroy();
       } catch (IOException e) {
           getLog().error(ERROR_CREATE_PACKAGE, e);
       } catch (InterruptedException e) {
           getLog().error(ERROR_CREATE_PACKAGE, e);
           Thread.currentThread().interrupt();
       }
   }
}

プロジェクトでは、次の行をpom.xmlに追加して接続できます。
<plugin>
   <groupId>org.sterzhen</groupId>
   <artifactId>jpackage-maven-plugin</artifactId>
   <version>1.0-SNAPSHOT</version>
   <executions>
       <execution>
           <goals>
               <goal>createpackage</goal>
           </goals>
       </execution>
   </executions>
</plugin>

ペットプロジェクトでプラグインをテストするとき、もう1つの驚きがありました。つまり、パッカーがWindowsで機能するためには、WiX 3.0以降を追加でインストールする必要があります。

WiXをインストールした後、プラグインが機能し、小さな勝利の喜びを感じました。はい、それはプリミティブでWindowsでのみ機能しますが、これはほんの始まりにすぎません。
