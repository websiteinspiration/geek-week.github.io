パフォーマンスの問題Moment.jsの解決に関するストーリー

Moment.jsは、日付を解析およびフォーマットするための最も人気のあるJavaScriptライブラリの1つです。WhereToはNode.jsを使用しているので、彼らにとって、このライブラリの使用は完全に自然な流れでした。Moment.jsのサーバー使用に関する問題は予期されていませんでした。最後に、彼らは最初から、このライブラリをフロントエンドで使用して日付を表示し、その作業に満足していました。ただし、ライブラリがクライアントで適切に動作したという事実は、サーバーにも問題がないことを意味していません。 今日公開している翻訳は、パフォーマンスの問題Moment.jsを解決するストーリーに特化しています。





プロジェクトの成長と生産性の低下


最近、WhereToシステムによって返されるフライトレコードの数は約10倍になりました。その後、パフォーマンスが大幅に低下しました。 100ミリ秒もかからなかったレンダリングサイクルが、約5,000件の検索結果を表示するのに3秒以上かかることがわかりました。私たちのチームは研究に着手しました。いくつかのプロファイリングセッションの後、この時間の99％以上がと呼ばれる単一の関数に費やされていることに気付きましたcreateInZone。


createInZone関数の完了には約3.3秒かかります。

状況の調査を続けると、この関数がMoment.js関数によって呼び出されていることがわかりましたparseZone。なぜ彼女はとても遅いのですか？Moment.jsライブラリは一般的な使用シナリオ向けに設計されていると感じていたため、結果として、さまざまな方法で入力文字列を処理しようとします。多分それを制限する必要がありますか？ドキュメントを読んだ後、関数parseZoneが日付形式を指定するオプションの引数を受け入れることがわかりました。

moment.parseZone(input, [format])

最初に行ったのは、関数を使用しparseZoneて日付形式に関する情報を渡そうとしたことですが、これはパフォーマンステストで示されているように、何にもつながりませんでした。

$ node bench.js
moment#parseZone x 22,999 ops/sec ±7.57% (68 runs sampled)
moment#parseZone (with format) x 30,010 ops/sec ±8.09% (77 runs sampled)

関数parseZoneは少し速く機能しましたが、私たちのニーズではこの速度では明らかに十分ではありませんでした。

プロジェクト固有の最適化


Moment.jsを使用して、プロバイダーのAPI（Travelport）から取得した日付を解析しました。常に同じ形式でデータを返すことがわかりました。

"2019-12-03T14:05:00.000-07:00"

これを知って、私たちはMoment.jsの内部構造を理解し始めて（期待どおり）、同じ結果を生成するはるかに効率的な関数を記述しました。

parseZoneのより高速な代替手段を作成する


最初に、Moment.jsオブジェクトがどのように見えるかを理解する必要がありました。これは非常に簡単に理解できました。

> const m = moment()

次のステップは、コンストラクターを使用せずにモーメントをインスタンス化することでした：

export function parseTravelportTimestamp(input: string) {
  const m = {}
  // $FlowIgnore
  m.__proto__ = moment.prototype
  return m
}

これで、設定できるMomentインスタンスプロパティがたくさんあるように見えました（見つけた方法の詳細には触れませんが、Moment.jsのソースコードを見ると理解できるでしょう）。

const FAKE = moment()
const TRAVELPORT_FORMAT = 'YYYY-MM-DDTHH:mm:ss.SSSSZ'
export function parseTravelportTimestamp(input: string) {
  const m = {}
  // $FlowIgnore
  m.__proto__ = moment.prototype
  const offset = 0 // TODO

作業の最後のステップは、offsetタイムスタンプの値を解析する方法を見つけることでした。これは常に行内の同じ位置であることがわかりました。その結果、これを最適化することができました。

function parseTravelportDateOffset(input: string) {
  const hrs = +input.slice(23, 26)
  const mins = +input.slice(27, 29)
  return hrs * 60 + (hrs < 0 ? -mins : mins)
}

すべてをまとめると、次のようになります。

const FAKE = moment()
const TRAVELPORT_FORMAT = 'YYYY-MM-DDTHH:mm:ss.SSSSZ'

function parseTravelportDateOffset(input: string) {
  const hrs = +input.slice(23, 26)
  const mins = +input.slice(27, 29)

  return hrs * 60 + (hrs < 0 ? -mins : mins)
}

/**
 *     ISO-8601,    :
 * - "2019-12-03T12:30:00.000-07:00"
 */
export function parseTravelportTimestamp(input: string): moment {
  const m = {}
  // $FlowIgnore
  m.__proto__ = moment.prototype

  const offset = parseTravelportDateOffset(input)
  const date = new Date(input.slice(0, 23))

  m._isAMomentObject = true
  m._i = input
  m._f = TRAVELPORT_FORMAT
  m._tzm = offset
  m._isUTC = true
  m._locale = FAKE._locale
  m._d = date
  m._isValid = true
  m._offset = offset

  return m
}

性能試験


ベンチマークnpmモジュールを使用して、結果のソリューションのパフォーマンスをテストしました。ベンチマークコードは次のとおりです。

const FAKE = moment()
const TRAVELPORT_FORMAT = 'YYYY-MM-DDTHH:mm:ss.SSSSZ'

function parseTravelportDateOffset(input: string) {
  const hrs = +input.slice(23, 26)
  const mins = +input.slice(27, 29)

  return hrs * 60 + (hrs < 0 ? -mins : mins)
}

/**
 *     ISO-8601,    :
 * - "2019-12-03T12:30:00.000-07:00"
 */
export function parseTravelportTimestamp(input: string): moment {
  const m = {}
  // $FlowIgnore
  m.__proto__ = moment.prototype

  const offset = parseTravelportDateOffset(input)
  const date = new Date(input.slice(0, 23))

  m._isAMomentObject = true
  m._i = input
  m._f = TRAVELPORT_FORMAT

  m._isUTC = true

結局、タイムスタンプの分析を約64倍高速化することができました。しかし、これはシステムの実際の運用にどのように影響しましたか？これがプロファイリングの結果として何が起こったかです。


parseTravelportTimestampの合計実行時間は40ミリ秒未満です。

結果は驚くべきものでした。3.3秒から始めて、日付を解析し、40ミリ秒未満に到達しました。

概要


私たちのプラットフォームで作業を開始したとき、私たちはひどい量の問題を解決しなければなりませんでした。私たちは自分たちに繰り返し言っていることだけを知っていました。「最初に機能させましょうが、後で最適化を行うことができます」。

過去数年にわたって、私たちのプロジェクトは非常に複雑になりました。幸いなことに、これで「マントラ」の2番目の部分である最適化に進むことができるようになりました。

ライブラリソリューションは、プロジェクトが現在の場所に到達するのに役立ちました。しかし、私たちは「図書館」の問題に直面しています。それを解決して、ニーズに焦点を当てた独自のメカニズムを作成することにより、システムリソースの消費の観点からコードを「より簡単」にし、ユーザーにとって貴重な時間を節約できることを学びました。
