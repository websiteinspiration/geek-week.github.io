1Cとの迅速な統合：エンタープライズ

この記事では、サードパーティのアプリケーションを1C構成と統合する最も簡単な方法について説明します。この記事は、主に.Net Core、PHP、Pythonで作成する開発者を対象としています。

1Cと統合するには多くの方法があり、1C自体からの優れた記事はこれに専念しています。特に、1CがWebサービスメカニズムをサポートしていること、つまり、1C側で独自のサービスを実装でき、その結果、クライアントアプリケーション側で独自のORMライブラリを実装できることがわかります。これらのライブラリの1つについては後で説明します。

一般的な用語でのテクニックの説明


1C側


すべては、「ブロム」拡張機能が1C構成に追加され、新しいWebサービスが追加されることから始まります。拡張機能は無料で入手でき、ライセンスされています（MIT）。拡張自体は特定のデータモデルに関連付けられていないため、互換モード8.3.10以降をサポートする任意の構成にインストールできます。

拡張機能がインストールされたら、追加されたサービスにhttp（s）プロトコルを介してアクセスできるように、Webサービスメソッドにアクセスできるユーザーの権限を設定し、Webサーバーに設定を公開する必要があります。1C側では、これ以上何も必要ありません。

クライアント側


クライアント側では、Bromパッケージが接続されています。たとえば、.Net Coreの場合、これは次のコマンドで実行できます。

Install-Package Brom -Version 1.0.1-beta08

またはPythonの場合：

pip install brome

パッケージをインストールしたら、リモート1C構成との対話を実行するクライアントオブジェクトを作成するだけで十分です。以下、Cのコードを引用しますが、PHPとPythonでも同様です。したがって、1つのコマンドでクライアントを作成できます。

dynamic  = new (@"
		= http://mydomain.com/publication_name;
		= 1c_user_name;
			= 1c_user_pass
");

デザイナーでは、公開された1C構成のアドレスと、拡張機能へのアクセス権を発行したユーザーのデータを指定するだけで十分です。クライアントが作成されたら、最も興味深いものに進むことができます。

手続きと関数の呼び出し1C


作成された臭素クライアントを介して、1Cで定義されたプロシージャと関数を呼び出すことができます。この場合、呼び出されるメソッドはサーバー側であり、グローバルコンテキストまたはサーバーモジュール（汎用モジュールまたはマネージャーモジュール）のいずれかに含まれている必要があります。たとえば、グローバルコンテキスト「Number Prescription」の関数の呼び出しは次のようになります。

string  = .(2547, " = fr_FR");

2番目のパラメーターは、ローカライズ（フランス語）を示すフォーマット文字列です。パラメータは自然な形式で送信され、追加の変換やパッケージ化は必要ありません。

そして、これはディレクトリマネージャーモジュールのコードで検索機能の呼び出しがどのようになるかです：

var  = ...(840);

ここでは、モジュールマネージャーディレクトリ「Currencies」を通じて関数を呼び出しました。呼び出しの結果は、「DirectoryReference」タイプのオブジェクトになります。これで、オブジェクトへのリンクをパラメータとして別の関数に渡すことができます。

var  = ..(, DateTime.Today);

今回は、一般的なモジュール「通貨の使用」に移り、そのメソッド「通貨の取得」を助けました。

ライブラリは複雑なデータ型の操作をサポートしているため、リンクを取得または返すメソッドを呼び出すことができます：リンク、配列、構造体、値テーブル、値ツリー、システム列挙など... 1Cでの作業を簡略化するために、一部のクラスはクライアントライブラリに特別に実装されています。

リモートメソッドのスコープは、設定で指定することで柔軟に構成できます。したがって、たとえば、「Directories。*。Find By Code」領域を指定すると、すべての構成ディレクトリで「Find By Code」メソッドのみが使用可能になります。したがって、統合アプリケーションが使用できる方法を明確に示すことができます。

リンクを操作する


オブジェクトへの参照により、ポインタをオブジェクト1Cに転送できるだけでなく、オブジェクト自体からデータを受信することもできます。クライアント側のリンクの操作は、1Cと同じくらい簡単です。例えば：

var  = ...("00-000018", new Date(2017, 1, 1));

var 		= .;
var 		= .;
var 	= ..;

foreach (var  in .) {
	Console.WriteLine((., .));
}

ここで、マネージャーモジュールを介してドキュメントへのリンクを見つけ、ドキュメントのフィールドとその表形式の部分「製品」の値を取得しました。オブジェクトのフィールドへの最初のアクセス後、リンクがガベージコレクターによって削除されるまで、すべてのデータが1Cサーバーからダウンロードされ、クライアントに保存されます。

オブジェクトへのリンクは、サーバーにアクセスせずにクライアント側でも取得できます。これを行うには、オブジェクトの一意の識別子を知っているだけです。

var  = ...(new Guid("5a32b6ab-4661-11e9-912a-38d547755ef7"));

定義済みのコレクションアイテムへのリンクを取得することもできます。

var  = ...18_118;

クエリ実行


ほとんどの場合、1つのコレクションに格納されたデータが不十分になり、複雑なクエリによって生成されたデータを取得する必要があります。リクエストを満たすために、クライアントライブラリに特別なクラス「リクエスト」が用意されています。クライアント側でのリクエストの処理は、1C側の処理と非常によく似ています。

var  = .(@"
	
		.  ,
		.  ,
		.  
	
		.  
	
		. = &			
");
.("", "-0001");

var  = .();

ここでは、「Nomenclature」ディレクトリからデータを選択するパラメータを使用して簡単なクエリを作成しました。パラメータとして、要素の文字列記事を渡しました。一般的なケースでは、パラメーター値はリンク、システム列挙、さらには配列にすることもできます。クエリの結果、「Values Table」が返されました。このクラスはクライアント側で実装されています。この例では、ループを使用してテーブル行のフィールドを推測しました。

1C側では、すべてのリクエストがクエリビルダーを介して実行されるため、既製のリクエストをテキストとして指定できるだけでなく、ビルダーのマークアップを含むリクエストテンプレートも指定できます。

Runメソッドは、オプションのパラメーター「結果のクロールのタイプ」を受け入れます。クロールタイプとして「グループ化」または「階層を使用したグループ化」を指定した場合、値のテーブルではなく、メソッドは値のツリーを返します。

また、Runメソッドの代わりに、Run Packageメソッドを呼び出すこともできます（複数のリクエストを一度に実行するため）。この場合、走査のタイプに応じて、テーブルの配列またはツリーの配列が返されます。

コードスニペットの実行


一部の特殊なケースでは、特定のコードを1C側で直接実行する必要がある場合があります。これを行うために、Bromクライアントは「実行」メソッドを提供します。このメソッドは、実行可能コードと1つのオプションパラメーターを含む入力テキストを受け取ります。これらのパラメーターは、コード内の変数 "Parameter"で使用できます。

var  = .(@"
     = 0;
         
         =  + ;
    ;
", new double[] { 45, 67, 12.56, 11.9 });

この例では、配列内の数値を合計するコードを実行しました。配列自体がパラメーターとして渡され、「パラメーター」変数に配置されました。計算結果は変数「Result」に格納されました。この名前の変数が実行可能コードに入力されている場合、実行が終了したときの値が「Run」関数の結果として返されます。

コードフラグメントを実行する機能は、拡張機能の個別のアクセスロールによって制御されます。この役割はテスト目的でのみ有効にすることをお勧めします。次のように、ワーキングドラフトで必ず無効にしてください。これは明らかな脆弱性です。

利点欠点


説明されている方法の利点には、確かに次のものが含まれます。

  • . SOAP HTTP, ;
  • . 1;
  • . , 1;
  • . 1 ;
  • 1 . 1 , , ;
  • アプリケーションコンテキストへのアクセス。データベースデータにアクセスできるだけでなく、1C側で定義されたメソッドを呼び出したり、セッション状態にアクセスしたりすることもできます。

この手法の欠点も利用できます。

  • 低データレート。SOAPプロトコルはXMLシリアル化に基づいているため、大量のデータの転送には、過剰なトラフィックの交換や、データのパックとアンパックに時間が必要です。この状況では、COM接続を介した通信が望ましいようです。
  • 便利なクライアントはすべてのプラットフォームで利用できるわけではありません。.Net Core、PHP、またはPythonプログラマでない場合は、SOAPメカニズムを使用して自分で臭素クライアントを実装する必要があります。これは、一般的に言えば、時間がかかります。
  • クエリ言語の制限1C。データベースでのすべての作業は1Cクエリメカニズムを通じて行われるため、いくつかの制限があります。たとえば、データの従来のページネーションを実装することはできません。クエリ言語1Cでは、ページングのメカニズムはありません。
