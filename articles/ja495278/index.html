データベース、カード、チェックリスト、またはなぜビジネスナレッジマネージャー

ハリークラインはアメリカの心理学者であり、極端な状況における直感的な解決策の研究者です。著書 『Sources of Power』で、クラインは1つのケースを説明しました。消防隊が電話に来て、キッチンが住宅の建物で燃えていました。チームは部屋に入り、火を消し始めたが、消滅しなかった。ある時点で、司令官は不安を感じ、緊急の出発を命じました。誰もが家を出たとき、キッチンの床は壊れていました。地下室で火災が発生し、上部に拡大したため、台所では消火できなかった。



その後、ハリー・クラインは本の資料を収集するために司令官にインタビューし、彼の直感的な解決策を分解することに成功しました。司令官は状況は危険だと感じ、チームを撤退させた。これは、非常に深く隠され、実際の状況でのみ表示される「暗黙の知識」の例です。

緊急事態で特に役立つ「暗黙の知識」を引き出すことは、ナレッジマネジメントマネージャーのタスクの1つです。たとえば、本番環境が低下し、サーバーが機能しなくなりましたが、その理由は誰にもわかりません。同時に、6か月前に同様のことが起こり、Petyaが問題を解決しました。彼は「いたずらをした」、それをひねった、そしてそれはうまくいきました。確かに、ペティアは別の会社に辞任しましたが、遡及は行わず、知識ベースにインシデントとソリューションを追加しませんでした。ベースもないためです。しかし、チームにナレッジマネージャーがいれば、結果は少なくなり、問題はより早く解決されます。

ナレッジマネージャーはポジションではなく機能です


ナレッジマネージャーは、専門家の個々の知識と会社の経験が失われず、分析され、問題を効果的に解決するために使用されるように条件を整理する人です。たとえば、Microsoft Servicesにはこのための別の役職があります。別のユニットを管理するナレッジマネジメントのディレクター（最高知識責任者）です。ディレクターには多くの権限があります。彼はユニットのサービスを他の部門に販売したり、機器を購入したり、他の部門からソフトウェアを開発したり、別の予算を管理したりできます。 

Microsoft Servicesのナレッジマネジメントシステム（KM）の基礎は、2005年から働いている実践者のコミュニティです。 100以上のコミュニティと数千の専門家がいます。開発者やプロダクトマネージャーは、グーグルしたり、同僚から時間をかけたりする必要はありません。コミュニティですぐに、彼はトピックに関する専門家を見つけ、質問し、回答を得ます。

CROC-専門的に知識管理に従事しているIT会社-でも同様のシステムが機能します。コミュニティの助けを借りて、企業知識のほぼすべてのエコシステムが作成されます。これは作業の詳細に応じて必要です。CROCはコンサルティング会社であり、専門家の知識を販売しています。企業内の各活動（プロジェクト、プレゼンテーション、研究）ごとにグループが作成され、そこにすべての情報（資料、プロジェクトのアイデア、通信）が届きます。



Microsoft ServicesとCROCには優れた知識管理システムがあります。社内のナレッジマネジメントから個々のプラクティスを実装するために、大きなシステムは必要ありません。小さく始めれば十分です。たとえば、典型的なビジネスの問題を解決する5つの単純な（比較的）プラクティスがあります。

「多くのプロジェクトのチーフアーキテクトとして、他のイニシアチブの人々と一緒に、包括的なドキュメントシステムを作成しました。当初-SGMLでは、ソースコードはバージョン管理システムの下にあり、RTFとCHMの自動アセンブリは毎晩、後でwikiシステムに基づいて更新されます（約2005年）。

ナレッジベースでは、説明テンプレート、典型的な例、およびさまざまな図を追加しました。ドキュメンテーションについては、コメントを基礎として使用しました。ベースはこれまで社内で保存されただけでなく、デフォルトでプロジェクトチームによって維持および開発されています。

プロジェクト事務所


プロジェクトオフィスは、「拡張された」知識ベースです。承認された決定と承認されていない決定、およびそれらが選択または破棄された理由がオフィスに追加されます。

誰かが問題に遭遇し、その理由を見つけ、それを解決してすべてをデータベースに書き留め、次の従業員が同じ問題をより早く解決します。研究、テスト、仮説の時間が短縮され、time2marketが短縮されます。

プロジェクトオフィスには、Googleドライブなどのシンプルなドキュメント管理システムが適しています。その中で、ドキュメントを交換したり、アクセス権を発行したり、コメントを追加したりできます。

「プロジェクトオフィスは単なる知識ベースではありません。彼のために、私はプロジェクトで何が起こっているかを分析し、開発者へのインタビューを行い、一般的な経験を集めました。これに基づいて、私は記事、指示、および製品開発の実践について説明しました。

Svetlana Novikova
Ozonのテクニカルライターグループの責任者。 KnowledgeConf 2020のメンバーであるWrite the Docs Russiaコミュニティ活動家は、@ the_know_allテレグ​​ラムチャネルを運営しています。

プロジェクトに多くの遺産がある場合、プロジェクトオフィスは不可欠です。カスタマイズされたプロジェクトオフィスでは、特定のテクノロジーが選択された理由、すべてのコードまたは松葉杖が追加された理由を簡単に確認できます。コードの一部を更新した後は、すべての変更をデータベースに簡単に追加して、次世代の「世代」の開発者が利用できるようにすることができます。

「私がFlantの前にテクニカルディレクターを務めていたとき、開発部門をゼロからドキュメントまでトレーニングしました。APIによる最新の建築ドキュメントは、GitLab上の別のリポジトリに配置されました。必要に応じて、コードの各行が拡大する要件を簡単に把握できます。誰もがすべての決定の理由を理解したので、これは開発とQ＆Aの間の矛盾を排除するのに役立ちました。

彼はフラントでこのアイデアを拡張しました。彼はGitLabでドキュメントのストレージを整理し、それを使用する方法を人々に教え、知識を重要な指示の形で共有する原理を開発し、それを機能させました。」 ナレッジコン2020のメンバーである不明の

コンピテンシーマップ


カードまたは能力のマトリックスは、高度なトレーニングとその制御のためのツールです。通常、それは表のように見えます。行と列の従業員の名前は、彼らが知っているものです。


マトリックスの例は、記事「Dragons Live Here：The Matrix of Competencies as a Tool of Timlid」にあります。

表は、従業員トレーニングの失敗を明確に示しています。特定のスキルのメインキャリアである（それが他のスキルに転送される）人物と、通常のユーザーだけを表示します。コンピテンシーマップは、チームリーダーまたはテクニカルリードで構成して、チーム開発者のレベルを評価できます。

「Flantの前は、開発会社のCTOでした。チームが問題を成長させて解決するのをより困難にし、従業員がスキルを向上させることを望みました。これには、透明な成長ルール、フィードバックが必要でした。これは能力のマトリックスを助けました。

しかし、マトリックスはすべての人に適しているわけではありません。Flantでは、DevOpsエンジニアの仕事はしていませんでした。何十もの技術と何百ものニュアンスから、それは巨大なサイズに「膨らむ」からです。そこで、Performance Reviewでマトリックスをアップグレードしました。これはすでに、従業員の採用、適応、成長に関する体系的な問題を特定できるプロセスです。」

適応チェックリスト


新しい従業員を雇うことは常にお金の価値があります。費用は、人事または採用担当者、マネージャー（チームリーダー、技術専門家、CTO）、および採用に関与するその他すべての人の時間と給与で構成されます。従業員が会社に来ると、オンボーディングに時間と労力が費やされます。 

したがって、ナレッジマネージャーのタスクの1つは、テスト期間中に「迷子になる」ことがないように、新入社員を穏やかに苦痛を伴わずに紹介することです。たとえば、トヨタの新入社員は、経験豊富な仲間の監督の下で働いています。企業が新しい工場を開くと、最高の労働者と新人の分遣隊がそこに送られます。しばらくすると、経験豊富な従業員が戻ってきて、初心者は働き続けます。

「彼はナレッジマネジメントコミュニティを組織し、特にオンボーディングの問題を発見してソリューション戦略を練った同僚とのミーティングを開催しました。たとえば、全従業員を対象とした一般的な導入プログラムを導入しています。一般的な知識と実践：タスクトラッカー、Wiki、一般的なテクノロジーが含まれていました。このプログラムは会社の専門的な構造によって適用され、試用期間を過ぎた従業員の割合を増やすのに役立ちました。」

ナレッジマネジメントシステムの実装方法


今日、貴社で個々のナレッジマネジメントプラクティスの実装を開始してください。ナレッジマネジメントツールで解決できる特定のビジネス上の問題に基づいて構築します。たとえば、Googleドキュメントでローカル（チーム）ナレッジベースを整理します。

問題のローカルソリューションを提示します。データベースがどのようにしてプロジェクトを迅速に終了したか、または数字の言語で費用を抑えてそれを実行したかを教えてください。次に、ソリューションを他の部門に拡張するよう経営陣に依頼します。したがって、小さなステップで、ナレッジマネジメントを実装します。

このために個人を雇う必要すらありません。たとえば、CROCには「ナレッジアンドコーポレートコミュニケーションマネジメント」の責任者（Alexey Sidorin）がいますが、ナレッジベースを担当する個別のチームはありません。

社内には多くの社内システムがあります。各システムのコマンドは、データベースに個別に情報を追加します。各従業員は、ナレッジマネージャの機能の一部を実行します。これは2つの理由で便利です。

UZは長く、高価で、複雑です。これは、知識の収集、変更管理、社会学、分析、建築家、開発者、マネージャー、設計者に関する少しの知識を含むプロセスです。 1人ですべての専門家になることはできないため、すべての従業員を知識管理に含める方が簡単です。開発者はナレッジベースにドキュメントを追加し、マネージャ-プロジェクト履歴、テスター-見つかった問題とその解決方法を説明します。

中小企業の方が安い：個々の従業員に予算を割り当てる必要はありません。現在の機能と役割にいくつかの新しい機能と役割を分配するだけで十分です。マネージャーの機能を複数の人に「洗い流して」もらう方が便利です。誰かがビジネスプロセスを管理し、誰かが専門家として行動し、誰かがすべてを1つのデータベースに収集します。
