オプションのチェーン


オプションの連鎖は、個人的には、ES2020標準の最もエキサイティングな機能の1つです。この機能が非常に役立つ多くのプログラムを作成しました。 

オプションのチェーンを使用すると、各オブジェクトの存在を確認する必要なく、オブジェクトの深く埋め込まれたプロパティへの安全なアクセスを整理できます。この機能のしくみを見てください。

まず、オプションのチェーンが登場する前に書かなければならないコードを見てください。

nullおよび未定義の値のみの検証


nulland undefined（Nullish Coalescing） のみで値を確認することは、可能性が提案段階にあったときでも本当に嬉しかった機能の1つです。適切なチェックを実行するために専用の関数を作成する必要に出くわしました。

JavaScriptは「偽」と「真」の意味を持つことが知られています。これで、「ゼロ」値が追加されたと言えます。これらの値にはnullおよびが含まれundefinedます。 「偽」のJavaScriptの観点では、空の文字列、番号0、値でありundefined、null、false、NaN。つまり、たとえば、「falsehood」の値をチェックする特定の式は、空の文字列で機能し、値undefined、そして何がもっとたくさん。そして、それが「ゼロ」であるかどうかの値をチェックするための式はtrue、nullおよびに対してのみ返されundefinedます。個人的にはこの機会はそれほど素晴らしいものではないかもしれませんが、実際、非常に重要です。

いくつかの例を見てみましょう。

null nullおよび未定義でのみ値をチェックする機能までのコード


最近、明るいテーマと暗いテーマを切り替える機能を実装する必要があるプロジェクトに取り組んでいました。同時に、コントロールの状態を確認し、trueまたはの値に対応しているかどうかを確認する必要がありましたfalse。ユーザーが値を設定しなかった場合、デフォルトで、それは等しいはずですtrue。以下は、nullおよびでのみ値をチェックする可能性の前にこの問題を解決した方法ですundefined：

const darkModePreference1 = true
const darkModePreference2 = false
const darkModePreference3 = undefined
const darkModePreference4 = null
const getUserDarkModePreference = (darkModePreference) => {
  if (darkModePreference || darkModePreference === false) {
    return darkModePreference
  }
  return true
}
getUserDarkModePreference(darkModePreference1) 
// true
getUserDarkModePreference(darkModePreference2) 
// false
getUserDarkModePreference(darkModePreference3) 
// true
getUserDarkModePreference(darkModePreference4) 
// true

null nullおよび未定義の値のみをチェックする可能性のあるコード


この機能が言語で表示されるnullと、undefined十分な演算子が確認され??ます。あなたは条件文なしで行うことができますif：

const darkModePreference1 = true
const darkModePreference2 = false
const darkModePreference3 = undefined
const darkModePreference4 = null
const getUserDarkModePreference = (darkModePreference) => {
  return darkModePreference ?? true;
}
getUserDarkModePreference(darkModePreference1) 
// true
getUserDarkModePreference(darkModePreference2) 
// false
getUserDarkModePreference(darkModePreference3) 
// true
getUserDarkModePreference(darkModePreference4) 
// true

ここで次のことが起こります。変数darkModePreferenceに「ゼロ」値が含まれている場合、値が返されtrueます。これにより、コードの記述が容易になり、コンパクトで理解しやすくなります。

動的インポート


動的インポートは、より効率的なアプリケーションのパフォーマンスに貢献します。このテクノロジーにより、追加のツールを使用せずに、JSファイルをモジュールの形式で動的にインポートできます。また、モジュールが不要な場合はインポートされません。ES2020より前は、アプリケーションで使用されているかどうかに関係なく、モジュールがインポートされていました。

たとえば、特定のファイルをPDF形式でロードする機能を実装する必要があるとします。

通常どおり、この問題を解決するための新旧のオプションを検討してください。

dynamic動的インポートをサポートする前のコード


実際の状況に基づいて、PDF形式で資料をダウンロードする機能が、ページへのすべての訪問者によって使用されるわけではないことが予想されます。しかし、とにかく、対応するモジュールをコードにインポートする必要があります。これは、ページが読み込まれるときに、モジュールが必要かどうかに関係なく読み込まれることを意味します。

import { exportAsPdf } from './export-as-pdf.js'
const exportPdfButton = document.querySelector('.exportPdfBtn');
exportPdfButton.addEventListener('click', exportAsPdf);

これにより、システムに追加の負荷がかかります。これは、モジュールの遅延読み込みを使用することで軽減できます。これは、webpackや他のモジュールローダーで利用可能なコード共有技術の助けを借りて行うことができます（そして、それらの使用自体は、一定量のシステムリソースを費やすことを意味します）。

しかし今、ES2020のおかげで、モジュールを動的にロードする標準的な方法があり、バンドラーなしで実行できます。

dynamic動的インポートのサポートが出現した後のコード


const exportPdfButton = document.querySelector('.exportPdfBtn');
exportPdfButton.addEventListener('click', () => {
  import('./export-as-pdf.js')
    .then(module => {
      module.exportAsPdf()
    })
    .catch(err => {
      //      -  
    })
})

ご覧のとおり、適切なモジュールが必要な場合にのみロードすることで、モジュールの遅延ロードを編成できます。これにより、システムの負荷が軽減され、ページの読み込み速度が向上します。

Promise.allSettledを構築する


すべてのpromiseが正常に解決された場合にのみアクションを実行する必要がある場合は、メソッドを使用できますPromise.all()。確かに、この方法には欠点が1つあります。渡された少なくとも1つのpromiseが拒否された場合、メソッドはエラーをスローします。つまり、すべてのpromiseが正常に解決されるまで、必要なアクションは実行されません。

これは必要ないかもしれません。おそらく次のシナリオが適しています。「結果は私には重要ではありません。すべてのpromiseの完了後にコードを実行する必要があります。」この場合、メソッドは便利ですPromise.allSettled()。対応するプロミスは、他のプロミスの完了後にのみ正常に解決されます。彼らが成功したか失敗したかは関係ありません。

the Promise.allコンストラクトを使用したコード


const PromiseArray = [
    Promise.resolve(100),
    Promise.reject(null),
    Promise.resolve("Data release"),
    Promise.reject(new Error('Something went wrong'))];
Promise.all(PromiseArray)
  .then(data => console.log('all resolved! here are the resolve values:', data))
  .catch(err => console.log('got rejected! reason:', err))
//got rejected! reason: null

どうやら、Promise.all転送された約束の1つが拒​​否された後、エラーが発生します。

Pro Promise.allSettled構文を使用するコード


const PromiseArray = [
    Promise.resolve(100),
    Promise.reject(null),
    Promise.resolve("Data release"),
    Promise.reject(new Error('Something went wrong'))];
Promise.allSettled(PromiseArray).then(res =>{
console.log(res);
}).catch(err => console.log(err));
//[
//{status: "fulfilled", value: 100},
//{status: "rejected", reason: null},
//{status: "fulfilled", value: "Data release"},
//{status: "rejected", reason: Error: Something went wrong ...}
//]

そして、ここでは、いくつかの約束が拒否されましたが、Promise.allSettled転送されたすべての約束によって発行された結果が返されます。

その他の注目すべき機能


▍BigIntデータ型


新しいデータ型をBigInt使用すると、JavaScriptで表示される前に操作できる数よりも長い数を処理できます（pow(2,53)-1）。確かに、このデータ型は以前の言語にあったものとの下位互換性はありません。JavaScriptで数値を処理するための基礎であるIEEE 754標準は、使用可能な数値をサポートしていませんBigInt

▍String.prototype.matchAllメソッド


この方法String.prototype.matchAll()は正規表現に関連しています。これはイテレータを返し、グループを含め、正規表現を使用して文字列で見つかったすべての一致を処理できます。

▍グローバルプロパティglobalThis


グローバルプロパティglobalThisは、コードが実行される環境に対応するグローバルオブジェクトへの参照を保持します。ブラウザでは、グローバルオブジェクトはオブジェクトで表されますwindow。Node.jsでは、これはオブジェクトglobalです。Webワーカーでは、これはオブジェクトselfです。
