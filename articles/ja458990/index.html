コード内のコメントで熱心に停止する

こんにちは、ハブル！

今日、非常に物議を醸している記事があなたの注意のために提案されており、クリーンコード哲学の重要な側面に触れています。記事の作成者は、ほとんどの場合、コード内のコメントは有害であると断言する自由がありますが、コメントなしでいつ実行するかを示すことも忘れないでください。



したがって、私たちは注意深く読み、すべてにもかかわらず、コメントします。

純粋なコードはよく書かれた散文のように読むべきです-Robert Martin
原則として、不良コードとコメントでオーバーロードされたコードとの間には顕著な相関関係があります。コメントは乱雑なソースコードの最も一般的な機能です。

すべてのプログラマーは、コメントが不要なほどクリーンで表現力のあるコードを書くように努力する必要があります。各変数、関数、およびクラスの意味は、それらの名前と構造によって認識される必要があります。コメントを書く必要がある場合、これは通常、コードが十分に表現力がないことを意味します。コメントを書くときはいつでも、反省があなたに投げかけられなければなりません。

誰かがあなたのコードを読んだとき、このコードが何をしているのかコメントすることなく、彼にはっきりとわかるはずです。適切に名前が付けられたクラスと関数は、これがコードではなく優れた小説であるかのように、読者がイベントの発生を追跡するのに役立ちます。読者が新しい関数やクラスに遭遇したとき、その内容は彼を驚かせるべきではありません。覚えておいてください。プログラマーの作業時間の大部分は、コードの作成ではなく、理解する必要がある他の誰かのコードの読み取りに費やされています。

コメントマスク群れ


変数名や関数名に関するコメントをよく目にします。そのようなコメントは、コードが何をするか（またはすべきか）を説明します。このようなコメントは、プログラマーが十分に表現力のある名前を取得できなかったこと、またはこの関数が複数のことを行うことを明確に示しています。

コード内のエンティティの適切な命名は非常に重要です。必ず、各コードに正確に名前を付けて、他の開発者が初めて、そして明確にそれを理解できるようにしてください。

//    
List<Employee> find(Status status) {
  ...
}

この例では、名前findが十分な情報ではなかったため、関数の作成者は、この関数の機能を説明するコメントでこれを補う必要がありました。ただし、find関数がたまたま別のモジュールから呼び出された場合、それは何を見つけたかを推測するだけです。この場合、「検索」とはどういう意味ですか？彼女はどうやって何かを見つけますか？見つけたものを返しますか？ボブおじさんが著書 『Clean Code』で指摘しているように、コメントを書く必要がある場合は、コードで自分の考えを適切に表現できなかったことを意味します。

誰がそれが何をするかを理解するために、各関数の解説を掘り下げたいですか？

List<Employee> getEmployeesByStatus(Status status) {
  ...
}

この関数の目的はそのシグネチャからすでに明らかであり、コメントは冗長になります。それで、コメントが欠陥を伴う理由を説明する次の側面に行きます。

冗長なコメント


それらはコードを散らかすだけで、完全に不要です。コードに不要なコメントを多数入力すると、コードを読む人はすべてのコメントを1つにスキップすることを学習します。そのため、本当に重要なコメントは読まないでしょう。

//     
void sendEmail() {
  ...
}

//       
public class Employee {
  ...
}

/**
 * @param title   CD
 * @param author   CD
 * @param tracks    CD
 */
public void addCd(String title, String author, int tracks) {
  ...
}

最後の例は、上から冗長性が低下していることを示しています。多くの組織では、このように各関数とクラスに署名する必要があります。上司が同じことをする場合は、しないように頼みます。

無効な抽象化レベル


長い関数がある場合、またはコードを文書化して、実行されている部分を示す必要がある場合は、次のルールに違反している可能性があります。

  1. 各関数は正確に1つのことを行う必要があります。
  2. 関数は小さくなければなりません。

次に例を示します。

//    ,    , 
// -, ,   ,   
//     - 
public void doSomeThings() {

  //  
  ...
    ...
    ...
  
  //     -  
  ...
    ...
    ...
  
  // ,    
  ...
    ...
    ...
  
  //   -
  ...
    ...
    ...
}

コード内に独立した関数として区別できるフラグメントがある場合、リファクタリングが必要です。ロジックの各部分を個別の関数にカプセル化できた場合、コードは、埋め込まれたアクションの説明とまったく同じように読み取る必要があります。

リファクタリング後、次のようになります。

public void sendPromotionEmailToUsers() {
  calculatePrices();
  compareCalculatedPricesWithSalesPromotions();
  checkIfCalculatedPricesAreValid();
  sendPromotionEmail();
}

コードの各部分にコメントするのではなく、ロジックのどの部分も独自の関数に美しくカプセル化されていることを確認してください。

まず、このように読みやすさが向上します。コードを1行ずつ読み取る必要はありません。補助関数の名前を読み、それが何をするかを理解するだけで十分です。特定の関数の詳細をさらに詳しく理解する必要がある場合は、いつでもその実装を調べることができます。

第二に、これはテスト容易性を改善します。上記の例では、関数ごとに個別のテストを実行できます。これらの個別の関数をカプセル化しないと、より大きな関数のすべての部分をテストすることが困難になりますsendPromotionEmailToUsers()。関数の機能が異なるほど、テストが難しくなります。

最後に、そのようなコードのリファクタリングも単純化されています。ロジックの各要素を独自の関数にカプセル化すると、将来的には変更が容易になるだけでなく、この関数の動作にのみ影響します。関数の実行中に存在するローカル変数を持つ長い関数がある場合、そのような関数をリファクタリングすると、コードの強い一貫性のために別の場所で変更が引き起こされます。

コードをコメント化


コメント付きのコードは生命を脅かすものと見なされるべきです。覗き込んだり、匂いを嗅いだり、どこから来たのか尋ねたりしないでください。取り除くだけです。プログラムに長く留まるほど、プログラム全体のシャワーが長くなります。

/*
public void oldFunction() {
  noOneRemembersWhyIAmHere();
  tryToUnCommentMe();
  iWillProbablyCauseABuildFailure();
  haHaHa();
}
*/

そのようなコードのコメントを外そうとすると、コンパイルされるかどうか誰が知っていますか？このコードは他のコードを無効にしますか？削除してください。このコードが後で必要になった場合は、いつでもバージョン管理システムで取得できます。バージョン管理システムを使用しますよね？

TODOコメント


TODOコメントを書くのではなく、単に...何をしようとしていたのですか？原則として、そのようなコメントは単に忘れられ、後で無関係または偽になる可能性があります。別のコーダーがTODOコメントを見ると、この操作がすでに行われているかどうか、どのようにしてわかりますか そのようなコメントも削除してください。

TODOコメントが適切と思われる唯一のケースは、同僚からのマージを期待する場合です。そのようなコメントは、コードに修正を加えて修正できるようになるまでの間、コード内で長い間たわまないことが想定されています。
コメントを書く必要があると思われる場合は、まずコードのコメントが冗長になるようにリファクタリングを試してください。-マーティン・ファウラー

コメントうそ


コメントに関する別の問題は、コメントが間違っていることです。

ジミーが自分で書いたばかりの新機能についてコメントを残したとき、彼は後で自分のコードを読む他の開発者を助けていると思います。実際、彼は罠を仕掛けています。彼の解説は、それが凶悪な嘘になるまで、何ヶ月も何年も触れられないままになっているかもしれません。その後、何百回ものリファクタリングと要件の変更が行われた後、このコメントはいくつかの離れたモジュールによって否認されます。

コード行を書き換えるとき、コードの他の部分のコメントをキャンセルしていないかどうかはどのようにしてわかりますか？あなたはこれを知らないでしょう。したがって、コメントは根こそぎにする必要があります。

public class User {
  ...
  
  //       
  String name;
    
  ...
}

その後、開発者はとで除算nameすることにfirstNameしましたlastName。

//        
void processEmployees() {
  ...
  List<Employee> employees = findEmployees(statusList);
  ...
}

//     
List<Employee> findEmployees(List<String> statusList) {
  ...
}

ば！コメントが間違っています。修正を反映するようにコメントを更新することは可能ですが、変更後にすべてのコメントを手動で維持しますか？あなたはドキュメント作成者ではなく開発者です。

しかし、そのようなコメントは気づきやすく、問題なく変更できます。

別の例を考えてみましょう：

//      
void processEmployees() {
  ...
  List<Employee> employees = findEmployees(statusList);
  ...
}

//       
List<Employee> findEmployees(List<String> statusList) {
  ...
}

後で、誰かがfindEmployees関数を変更して、ステータスのリストではなく名前のリストで従業員を検索できるようにします。

//       
void processEmployees() {
  ...
  List<Employee> employees = findEmployees(statusList);
  ...
}

//       
List<Employee> findEmployees(List<String> nameList) {
  ...
}

まず、findEmployeesに関するコメントは無関係になり、変更する必要があります。これは問題ありませんよね？そうじゃない。

上記のコメントprocessEmployeesも無関係になり、変更する必要があります。このような小さなリファクタリングの結果、無効になっているコメントはあといくつありますか？

代替ソリューション：

void processEmployees() {
  ...
  List<Employee> employees = findEmployeesByName(nameList);
  ...
}

List<Employee> findEmployeesByName(List<Name> nameList) {
  ...
}

関数に正確かつ注意深く名前を付ければ、コメントは不要になり、コードは偽になりません。

コメントが正常な場合


開発者の間で、私はコードへの詳細なコメントの熱狂的な支持者をたくさん知っています。彼らと協力して、時々コメントは正常であることに同意する必要があります。コードにコメントを入力するたびに、しぶしぶコメントを入力しますが、コメントが必要な場合もあります...

複合式


コードに複雑なSQLステートメントまたは正規表現がある場合は、コメントを添えてください。このようなコマンドは通常、コードで明確かつ表現力豊かに表現することが困難です。このような式にコメントを付けると、他の開発者があなたのコードを理解するのに非常に役立ちます。

//    kk:mm:ss EEE, MMM dd, yyy
Pattern timePattern = Pattern.compile("\\d*:\\d*:\\d* \\w*, \\w*, \\d*, \\d*");

警告


副作用やトラブルの可能性について他の開発者に警告する必要がある場合、このコードはコメントを添付するのに適しています。このようなコメントは、プログラムの不可解な動作を警告するビーコンのように機能する可能性があります。したがって、それらを含むコードははるかに価値が高くなります。

意図の説明


クリーンなコードを記述できない場合は、それを認めてコメントを書き込んでください。あなたはあなたが書くすべてのコードに対して責任があるので、説明なしで悪いコードを決して残さないでください。

コメントを書く必要がある場合は、コメントが正しい場所にあることを確認してください。参照先のコードから離れた場所にあるコメントはすぐに嘘に変わるため、破棄する必要があります。関数または変数を説明するコメントは、そのすぐ上にあります。IDEがコメントの強調表示をサポートしている場合は、コメントがコードから目立つようにしてください。
