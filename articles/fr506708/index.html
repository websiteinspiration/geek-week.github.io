Authentification à deux facteurs VPN / Mikrotik - simple et évolutive

salut!

Pour la rédaction de cet article, j'ai été invité à lire le contenu similaire d'un article utilisateurnkusnetsov. Le nombre de vues montre que la communauté s'intéresse à ce sujet.

Par conséquent, j'ai décidé de partager avec vous ma propre solution, qui a été précédemment implémentée par moi et dans sa forme de base a:

  • Faible niveau d'entrée et simplicité du code (pour la compréhension / le débogage par un autre employé)
  • Les scripts ROS simples ne créent aucune charge et fonctionnent même sur hAP Lite
  • Évolutivité - la possibilité de connecter un grand nombre de passerelles VPN pour réduire la charge ou la répartition géographique
  • Possibilité d'utiliser Mikrotik CHR comme serveur VPN
  • «1xN» - 1 passerelle SMS pour un nombre illimité de routeurs avec possibilité d'extension avec une charge croissante
  • Possibilité de lier un routeur séparé à un modem "spécifique" (pour quoi? - plus d'informations à ce sujet plus tard)
  • Utiliser un seul script PHP sur un serveur distant
  • Peu importe quel appareil a initié la connexion VPN, l'autorisation par le lien depuis SMS
  • Possibilité de consigner les autorisations des employés sur le serveur
  • Augmentez la tolérance aux pannes et réduisez la charge du système en envoyant des SMS au hasard depuis plusieurs modems

Énoncé du problème et solution


Tache 1


Dans le processus de développement de cette solution, la tâche a été fixée pour minimiser le nombre de modems USB - réduire le coût de possession, simplifier l'administration, localiser les modems en un seul endroit et ainsi améliorer la maintenabilité du système.

Il a été décidé d'utiliser un routeur installé sur le «canal fiable» comme passerelle SMS de base. La seule chose que je ne savais pas, c'était la charge sur le système, les modems se débrouilleraient et si l'opérateur le bloquerait. Par conséquent, j'ai permis de connecter plusieurs modems et de diviser davantage le système en groupes. De plus, des modems supplémentaires peuvent être connectés à la passerelle de base via un simple hub USB. Mais pour un fonctionnement complet, avec des fonctionnalités minimales, vous avez besoin d'un modem, quel que soit le nombre de routeurs.

Tâche 2


Pour prévoir la possibilité d'utiliser des cartes SIM locales dans la zone d'installation du routeur.
Exemple: un vaste réseau d'agences avec plusieurs magasins au Kazakhstan. L'envoi d'un SMS depuis la Fédération de Russie coûtera assez cher. Cette solution permet aux employés du Kazakhstan de recevoir des SMS d'un numéro local.

Mais en cours de réflexion, il s'est avéré que la solution avait déjà été trouvée et mise en œuvre dans la tâche 1.

Tâche 3


Autorisation de tunnel depuis un appareil mobile sans avoir à se trouver physiquement sur l'appareil autorisé.

L'objectif est la possibilité d'autoriser non seulement les tunnels utilisateur, mais aussi toutes les connexions vpn: Mikrotik-> Mikrotik, Server-> Mikrotik, etc. Dans ce cas, l'utilisateur responsable de ces tunnels doit simplement suivre le lien du message SMS, dans lequel également quel tunnel veut se connecter s'affiche.

Pour pouvoir autoriser de cette manière, il est devenu nécessaire de transférer le script vers une ressource externe - un serveur accessible de n'importe où et d'utiliser le code d'autorisation pour marquer la liste d'adresses dans le pare-feu et déverrouiller ensuite.

La solution à ce problème impliquait déjà sans ambiguïté l'utilisation de l'API RouterOS (ou SSH). L'API a remporté l'option la plus simple.

Logiques


  1. L'utilisateur se connecte avec un nom d'utilisateur et un mot de passe pré-ajoutés
  2. La passerelle VPN place son ip dans la liste d'attente d'adresses et envoie une requête POST au serveur
  3. Le serveur reçoit des données, vérifie, génère un code d'autorisation et envoie un SMS au numéro de contact du formulaire: "Pour autoriser la connexion utilisateur 79001112233 ouverte - http_: //synome.ru/? Ruid = vrG7yYMbZ6 & auth = YU6zc"
  4. L'utilisateur suit le lien depuis n'importe quel appareil
  5. Le serveur fait une demande au routeur, vérifie la présence dans la liste d'adresses de l'entrée avec le code d'autorisation dans le commentaire. Si un enregistrement est trouvé, le supprime et l'utilisateur affiche une page de connexion réussie

Dans tous les autres cas, si quelque chose ne passe pas le test, l'utilisateur recevra une erreur de demande.

Configuration et code


Passons maintenant de la partie conceptuelle à la configuration de Mikrotik, du code et de leur description
Explication: ces scripts et code php sont fonctionnels pour tout type de serveur VPN. Choisir un type de serveur (PPP / L2TP / SSTP), configurer un profil et un tunnel - vous choisissez "à votre goût", mais je vais donner un exemple général de configuration rapide.

Ajouter à Mikrotik:


Pare-feu


Assurez-vous de créer une liste de ressources autorisées, parmi lesquelles: propre adresse MT, tout DNS public, adresse de serveur sur laquelle l'autorisation aura lieu 
/ip firewall address-list
add address=10.10.0.1 list=Allow-list
add address=8.8.8.8 list=Allow-list
add address=synome.ru list=Allow-list


Ajouter une règle bloquant le trafic des utilisateurs VPN de la liste d'attente vers tout hôte autre que celui autorisé 
/ip firewall raw
add action=drop chain=prerouting dst-address-list=!Allow-list src-address-list=VPN-blocked disabled=no


Profil PPP


Créez un profil dans lequel nous définissons le délai d'expiration de la connexion. La durée doit être inférieure à celle spécifiée dans le script On Up suivant, sinon, si l'autorisation n'a pas été effectuée, puis après avoir supprimé la liste de la liste d'adresses, l'utilisateur aura accès pendant une durée égale à la différence (délai d'inactivité moins délai d'expiration de la liste d'adresses)

Ajouter un profil 
/ppp profile
add dns-server=10.10.0.1 idle-timeout=59m local-address=10.10.1.1 name=2F-VPN use-compression=no use-encryption=no use-mpls=no


Ensuite, sur le dernier onglet du script, ajoutez:

On up 
:global ruid "#ROUTERLOGIN#";
:global pass "#ROUTER_PASSWORD#";
:local userip [/ppp active get [find name=$user] address];
# if phone number stored in comment
#:local userphone [/ppp secret get [find name=$user] comment];
# if phone number = username
:local userphone $user;

:local authkey [/tool fetch http-method=post http-data="ruid=$ruid&pass=$pass&tel=$userphone" url="https://yourwebsite.ru/" mode=https as-value output=user];

/ip firewall address-list remove [find address=$userip];
/ip firewall address-list add address=$userip list=VPN-blocked timeout=1h comment=($authkey->"data");

:log info message="User connect:";
:log info message=$userphone;
:log info message=$userip;
:log info message=($authkey->"data");


On down 
:local userip [/ppp secret get [find name=$user] remote-address];
/ip firewall address-list remove [find address=$userip];
:log info message="User disconnect:";
:log info message=$user;
:log info message=$userip;




: , . ( ) ip-. POST- . ip- address-list VPN-blocked - 1 . .

: ip- , address-list . .

PPP Secrets


/ppp secret
add comment="70001112233" name=70001112233 password=testuser profile=2F-VPN remote-address=10.10.1.100


Le numéro de téléphone peut être spécifié directement dans le nom, mais si nous voulons pouvoir spécifier un numéro pour plusieurs comptes (pour l'autorisation de plusieurs tunnels), nous indiquons le numéro dans le commentaire, tandis que dans le script On Up, vous devez changer le commentaire des lignes

:local userphone [/ppp secret get [find name=$user] comment];
