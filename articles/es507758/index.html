Memoria JavaScript: sin fugas









La forma en que crea y accede a sus datos puede afectar el rendimiento de su aplicación. A ver cómo.



Introducción



JavaScript es un lenguaje de muy alto nivel. Por lo tanto, la mayoría de los desarrolladores no piensan en cómo se almacenan los datos en la memoria. En este artículo, veremos cómo se almacenan los datos en la memoria, cómo afecta el procesador y la memoria, y cómo la forma en que distribuye los datos a JS y accede a ellos afecta el rendimiento.



Triángulo amoroso



Cuando una computadora necesita realizar algunos cálculos, un procesador (CPU) necesita datos para procesar. Por lo tanto, de acuerdo con la tarea, envía a la memoria una solicitud de muestreo de datos a través del bus.



Se ve así:







así que este es nuestro triángulo romántico - Procesador -> Bus -> Memoria



Mantener una relación sana en el trío es difícil



El procesador es mucho más rápido que la memoria. Por lo tanto, este proceso Procesador -> Bus -> Memoria -> Bus -> El procesador "pasa" tiempo en los cálculos. Mientras se escanea la memoria, el procesador está inactivo.



Para evitar el tiempo de inactividad, se ha agregado un caché al sistema. No entraremos en detalles sobre qué es un caché o qué tipos de caché son, pero basta con decir que el caché es la memoria interna del procesador.



Cuando la CPU recibe un comando para ejecutar, primero busca datos en la memoria caché y, si no hay datos allí, envía una solicitud a través del bus.



El bus, a su vez, transfiere los datos solicitados más la parte condicional de la memoria y los almacena en la memoria caché para una referencia rápida.



Por lo tanto, la CPU tendrá menos insatisfacción con la lentitud de la memoria y, por lo tanto, la CPU tendrá menos tiempo de inactividad.



Las peleas son parte de cualquier relación.



El problema que puede surgir, especialmente cuando se trata de procesar grandes cantidades de datos, es un fenómeno llamado pérdida de caché.



Una falta de memoria caché significa que durante los cálculos, la CPU detecta que no tiene los datos necesarios en la memoria caché y necesita solicitar estos datos a través de canales regulares; recuerde que hay mucha memoria lenta.







Ilustración de recibo de efectivo. Los datos de la matriz se procesan, pero los datos fuera del límite de la memoria caché también se solicitan para el cálculo, y esto crea una "pérdida de memoria caché".



Ok, pero soy un desarrollador de JavaScript, ¿por qué debería importarme?



Buena pregunta. En la mayoría de los casos, no lo piensas. Sin embargo, hoy, a medida que más y más datos pasan a través de los servidores escritos en Node.js, es más probable que encuentre un problema de pérdida de caché al ordenar a través de grandes conjuntos de datos.



¡Lo creeré cuando lo vea!



Justa. Veamos un ejemplo.



Aquí hay una clase llamada Boom. 



class Boom {
  constructor(id) {
    this.id = id;
  }
  setPosition(x, y) {
    this.x = x;
    this.y = y;
  }
}


Esta clase (Boom) tiene solo 3 propiedades: id, x e y.



Ahora creemos un método que complete x e y.



Vamos a configurar los datos: 



const ROWS = 1000;
const COLS = 1000;
const repeats = 100;
const arr = new Array(ROWS * COLS).fill(0).map((a, i) => new Boom(i));


Ahora usaremos estos datos en el método: 



function localAccess() {
  for (let i = 0; i < ROWS; i++) {
    for (let j = 0; j < COLS; j++) {
      arr[i * ROWS + j].x = 0;
    }
  }
}


Lo que hace localAccess es atravesar linealmente la matriz y establecer x en 0.



Si repetimos esta función 100 veces (mira la constante de repeticiones), podemos medir cuánto tiempo lleva ejecutar: 



function repeat(cb, type) {
  console.log(`%c Started data ${type}`, 'color: red');
  const start = performance.now();
  for (let i = 0; i < repeats; i++) {
    cb();
  }
  const end = performance.now();
  console.log('Finished data locality test run in ', ((end - start) / 1000).toFixed(4), ' seconds');
  return end - start;
}
repeat(localAccess, 'Local');

Precio por caché miss



Ahora, de acuerdo con lo que aprendimos anteriormente, podemos considerar el problema de pérdida de caché si procesamos datos que están muy lejos durante la iteración. Los datos eliminados son datos que no están en el índice adyacente. Aquí está la función para esto: 



function farAccess() {
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      arr[j * ROWS + i].x = 0;
    }
  }
}


Lo que sucede aquí es que, en cada iteración, recurrimos al índice, que se encuentra a una distancia de FILAS de la última iteración. Por lo tanto, si las FILAS son 1000 (como en nuestro caso), obtenemos la siguiente iteración: [0, 1000, 2000, ..., 1, 1001, 2001, ...].



Agreguemos esto a nuestra prueba de velocidad: 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local');
}, 2000);


Y aquí está el resultado final: la







iteración no local fue casi 4 veces más lenta. Esta diferencia aumentará con el aumento de datos. Esto se debe a que el procesador está inactivo debido a una falta de caché.



Entonces, ¿qué precio pagas? Todo depende del tamaño de sus datos.



Bien, ¡juro que nunca haré eso!



Quizás no lo piense, pero ... hay casos en los que desea acceder a una matriz con alguna lógica que no es lineal (por ejemplo, 1,2,3,4,5) o no condicional (por ejemplo, para ( sea ​​i = 0; i <n; i + = 1000)).



Por ejemplo, obtiene datos de un servicio o base de datos, y necesita procesarlos, ordenarlos o filtrarlos utilizando una lógica compleja. Esto puede conducir a una situación similar a la que se muestra en la función farAccess.



Aquí hay una ilustración que (con suerte) aclara la situación: los







datos originales en la memoria frente a los datos ordenados en la memoria. Los números indican los índices de los objetos en la matriz fuente.



Mirando la imagen de arriba, vemos los datos en la forma en que se almacenan en la memoria (barra gris superior). A continuación vemos la matriz que se creó cuando los datos provienen del servidor. Finalmente, vemos una matriz ordenada que contiene referencias a objetos almacenados en varias posiciones en la memoria.



Por lo tanto, iterar sobre una matriz ordenada puede provocar múltiples errores en la memoria caché, como se muestra en el ejemplo anterior.

Tenga en cuenta que este ejemplo es para una pequeña matriz. Los errores de caché son relevantes para datos mucho más grandes.
En un mundo donde necesita animaciones fluidas en la interfaz y donde puede cargar cada milisegundo de tiempo de procesador en el servidor, esto puede volverse crítico.



¡Oh no! ¡Todo está perdido!



No, en absoluto.



Hay varias soluciones a este problema, pero ahora que conoce la razón de esta disminución del rendimiento, probablemente pueda encontrar una solución por su cuenta. Solo necesita almacenar los datos que se procesan lo más cerca posible en la memoria.

Esta técnica se denomina patrón de diseño de Localidad de datos .
Sigamos nuestro ejemplo. Suponga que en nuestra aplicación el proceso más común es el procesamiento de datos utilizando la lógica que se muestra en la función farAccess. Nos gustaría optimizar los datos para que se ejecuten rápidamente en el bucle for más común.



Organizamos los siguientes datos: 



const diffArr = new Array(ROWS * COLS).fill(0);
for (let col = 0; col < COLS; col++) {
  for (let row = 0; row < ROWS; row++) {
    diffArr[row * ROWS + col] = arr[col * COLS + row];
  }
}


Entonces, ahora, en diffArr, los objetos que están en los índices [0,1,2, ...] en la matriz original ahora se establecen de la siguiente manera [0,1000,2000, ..., 1, 1001, 2001, ..., 2, 1002 , 2002, ...]. Los números indican el índice del objeto. Esto simula la clasificación de una matriz, que es una forma de implementar el patrón de diseño de Localidad de datos.



Para verificar esto fácilmente, modificaremos ligeramente nuestra función farAccess para obtener una matriz personalizada: 



function farAccess(array) {
  let data = arr;
  if (array) {
    data = array;
  }
  for (let i = 0; i < COLS; i++) {
    for (let j = 0; j < ROWS; j++) {
      data[j * ROWS + i].x = 0;
    }
  }
}


Ahora agregue el script a nuestra prueba: 



repeat(localAccess, 'Local');
setTimeout(() => {
  repeat(farAccess, 'Non Local')
  setTimeout(() => {
    repeat(() => farAccess(diffArr), 'Non Local Sorted')
  }, 2000);
}, 2000);

Salida



En este artículo, demostramos el patrón de diseño de Localidad de datos. De hecho, hemos demostrado que acceder a una estructura de datos de una manera que no está optimizada puede reducir el rendimiento. Luego optimizamos los datos de acuerdo con nuestra forma de procesarlos y vimos cómo mejoraba el rendimiento.



El patrón de diseño de Localidad de datos es común en el campo del desarrollo del juego, donde a menudo tienes que lidiar con muchas entidades que se repiten muchas veces. En los resultados anteriores, vemos que la localidad de datos sigue siendo importante incluso en lenguajes de alto nivel como JavaScript, y no solo en los juegos.



Hoy, dada la cantidad de datos transferidos entre servidores o descargados a los navegadores, los desarrolladores de aplicaciones deben tener en cuenta los patrones de diseño que solían ser cotidianos para los desarrolladores de juegos, tanto del lado del servidor como del lado del cliente.
