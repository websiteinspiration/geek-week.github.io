＃1。.toBeと.toEqual


最初は、これらのステートメントはすべて私にとっては正常に見えました。

expect('foo').toEqual('foo')
expect(1).toEqual(1)
expect(['foo']).toEqual(['foo'])

等価ステートメント（to.equal） にchaiを使用すると、これは自然なことです。実際、ジェストは文句を言うことはなく、これらの発言は通常通り通過します。

ただし、Jestには.toBeと.toEqualがあります。 1つ目はObject.isを使用して同等性をアサートするために使用され、2つ目はオブジェクトと配列の詳細な比較を提供するために使用されます。 .toEqualは、プリミティブ値の等式をアサートするなど、深い比較が不要であることが判明した場合にObject.isを使用することへのフォールバックを備えています。これにより、前の例がうまく機能した理由がわかります。

expect('foo').toBe('foo')
expect(1).toBe(1)
expect(['foo']).toEqual(['foo'])

この方法では、テストしている値がすでにわかっている場合に使用if-elseして.toEqual、のすべてをスキップでき.toBeます。
よくある間違いは.toBe、プリミティブ値の同等性を主張するために使用するものです。

expect(['foo']).toBe(['foo'])

あなたが見れば、ソースコード .toBeがクラッシュしたとき、それはあなたが本当にすることによって、このエラーをしたかどうかを判断しようとする関数を呼び出す .toEqual使用していること。これは、テストを最適化するときにボトルネックになる可能性があります。

プリミティブ値を使用していることが確実な場合は、最適化のためにコードをそのように再編成できます。

expect(Object.is('foo', 'foo')).toBe(true)

詳細はドキュメントをご覧ください。

＃2 より適切な比較


技術的には、.toBe任意の値を検証するために使用できます。Jestを使用すると、特定の比較ツールを使用して、テストを読みやすく（場合によっては短く）することができます。

// 
expect([1,2,3].length).toBe(3)

// 
expect([1,2,3]).toHaveLength(3)

const canBeUndefined = foo()

// 
expect(typeof canBeUndefined !== 'undefined').toBe(true)

// 
expect(typeof canBeUndefined).not.toBe('undefined')

// 
expect(canBeUndefined).not.toBe(undefined)

// 
expect(canBeUndefined).toBeDefined()

class Foo {
  constructor(param) {
    this.param = param
  
}

// 
expect(new Foo('bar') instanceof Foo).toBe(true)

// 
expect(new Foo('bar')).toBeInstanceOf(Foo)

これらは、ドキュメント内のJest比較の長いリストから選択したもののほんの一部です。残りは自分で確認できます。

＃3 ユーザーインターフェイスなしの要素のスナップショットテスト


Jestでのスナップショットテスト について聞いたことがあるかもしれません。これは、ユーザーインターフェイス要素の変更を追跡するのに役立ちます。ただし、スナップショットを使用したテストはこれに限定されません。

この例を考えてみましょう：

const allEmployees = getEmployees()
const happyEmployees = giveIncrementByPosition(allEmployees)

expect(happyEmployees[0].nextMonthPaycheck).toBe(1000)
expect(happyEmployees[1].nextMonthPaycheck).toBe(5000)
expect(happyEmployees[2].nextMonthPaycheck).toBe(4000)
// ...etc

あなたがますます多くの従業員を主張すると、それは疲れるでしょう。さらに、各従業員にさらに多くのクレームが必要であることが判明した場合は、新しいクレームの数に従業員の数を掛ければ、アイデアがわかります。
スナップショットテストでは、これはすべて次のように簡単に実行できます。

const allEmployees = getEmployees()
const happyEmployees = giveIncrementByPosition(allEmployees)

expect(happyEmployees).toMatchSnapshot()

回帰が発生するたびに、ノード内のどのツリーがイメージと一致しないかが正確にわかります。

しかし、利便性には代償が伴います。この方法はエラーが発生しやすくなります。画像が実際に正しくないことを知らない可能性があり、最終的にはとにかくそれを撮ることになります。そのため、スナップショットを自分の承認コードであるかのように再確認してください（そうであるため）。

もちろん、テストはスナップショットに限定されません。ドキュメントの完全な情報をお読みください。

＃4。describe.eachおよびtest.each


これに少し似たテストを書いたことがありますか？

describe('When I am a supervisor', () => {
  test('I should have a supervisor badge', () => {
    const employee = new Employee({ level: 'supervisor' })

    expect(employee.badges).toContain('badge-supervisor')
  })

  test('I should have a supervisor level', () => {
    const employee = new Employee({ level: 'supervisor' })

    expect(employee.level).toBe('supervisor')
  })
})

describe('When I am a manager', () => {
  test('I should have a manager badge', () => {
    const employee = new Employee({ level: 'manager' })

    expect(employee.badges).toContain('badge-manager')
  })

  test('I should have a manager level', () => {
    const employee = new Employee({ level: 'manager' })

    expect(employee.level).toBe('manager')
  })
})

これは単調で日常的なことですよね？多くのケースでこれを行うことを想像してください。
を使用するdescription.eachとtest.each、次のようにコードを圧縮できます。

const levels = [['manager'], ['supervisor']]
const privileges = [['badges', 'toContain', 'badge-'], ['level', 'toBe', '']]

describe.each(levels)('When I am a %s', (level) => {
  test.each(privileges)(`I should have a ${level} %s`, (kind, assert, prefix) => {
    const employee = new Employee({ level })

    expect(employee[kind])[assert](`${prefix}${level}`)
  })
})

＃5。グローバル機能の単一の模倣


ある時点で、特定のテストケースでグローバル関数に依存するものをテストする必要があります。たとえば、Date JavaScriptオブジェクトを使用して現在の日付に関する情報を受け取る関数や、それに依存するライブラリなどです。難しさは、それが現在の日付になると、正しい説明を得ることができないということです。

function foo () {
  return Date.now()
}

expect(foo()).toBe(Date.now())
//  This would throw occasionally:
// expect(received).toBe(expected) // Object.is equality
// 
// Expected: 1558881400838
// Received: 1558881400837


最後に、一貫性があり管理しやすいように、グローバルDateオブジェクトを再定義する必要があります。

function foo () {
  return Date.now()
}

Date.now = () => 1234567890123

expect(foo()).toBe(1234567890123) //

ただし、テスト間で再定義が維持されるため、これは悪い習慣と見なされます。Date.nowに基づく他のテストがない場合、これに気付くことはありませんが、リークも発生します。

test('First test', () => {
  function foo () {
    return Date.now()
  

  Date.now = () => 1234567890123

  expect(foo()).toBe(1234567890123) // 
})

test('Second test', () => {
  function foo () {
    return Date.now()
  

  expect(foo()).not.toBe(1234567890123) //  ???
})

以前はリークしないように「クラック」していました。

test('First test', () => {
  function foo () {
    return Date.now()
  

  const oriDateNow = Date.now
  Date.now = () => 1234567890123

  expect(foo()).toBe(1234567890123) // 
  Date.now = oriDateNow
})

test('Second test', () => {
  function foo () {
    return Date.now()
  

  expect(foo()).not.toBe(1234567890123) //  as expected
})

ただし、これを行う方法は、ハッカーによる方法ではありません。

test('First test', () => {
  function foo () {
    return Date.now()
  

  jest.spyOn(Date, 'now').mockImplementationOnce(() => 1234567890123)

  expect(foo()).toBe(1234567890123) // 
})

test('Second test', () => {
  function foo () {
    return Date.now()
  

  expect(foo()).not.toBe(1234567890123) //  as expected
})

したがって、jest.spyOnグローバルDateオブジェクトに従うと、1回の呼び出しに対してのみnow関数の実装を模倣します。これにより、残りのテストではDate.nowがそのまま残ります。
