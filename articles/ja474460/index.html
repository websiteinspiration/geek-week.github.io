TableGenを使用したLLVMのプロセッサアーキテクチャの説明

現在、LLVMはすでに非常に人気のあるシステムになり、多くの人々がさまざまなコンパイラーやアナライザーなどを作成するために積極的に使用しています。このトピックに関する有用な資料は、ロシア語を含めてすでに多数書かれています。これは朗報です。ただし、ほとんどの場合、記事の主なバイアスはフロントエンドとミドルエンドのLLVMに基づいています。もちろん、LLVMオペレーションの完全なスキームを説明する場合、マシンコードの生成はバイパスされませんが、基本的にこのトピックは、特にロシア語の出版物でさりげなく触れられます。同時に、LLVMには、プロセッサアーキテクチャを記述するためのかなり柔軟で興味深いメカニズムがあります。したがって、この資料は、LLVMの一部である、多少無視されているユーティリティTableGenに当てられます。

コンパイラが各ターゲットプラットフォームのアーキテクチャに関する情報を持つ必要がある理由は非常に明白です。当然、各プロセッサモデルには独自のレジスタセットや独自の機械命令などがあります。また、コンパイラーは、有効で効率的なマシンコードを生成できるようにするために、それらに関する必要なすべての情報を持っている必要があります。コンパイラーは、レジスターの配布など、プラットフォーム固有のさまざまなタスクを解決します。さらに、LLVMバックエンドは、実際の命令により近いマシンIRまたはアセンブラー命令自体ですでに最適化を実行します。このような最適化では、命令を置き換えて変換する必要があるため、命令に関するすべての情報を利用できる必要があります。

プロセッサアーキテクチャを記述する問題を解決するために、LLVMはコンパイラに必要なプロセッサプロパティを決定するために単一のフォーマットを採用しました。サポートされているアーキテクチャごとに、フォーマットファイル.tdには特別な正式言語の説明が含まれています。.incLLVMの一部であるTableGenユーティリティを使用してコンパイラをビルドすると、フォーマットファイルに変換されます。実際、結果のファイルはCソースですが、ほとんどの場合、これらの自動生成されたファイルを簡単に区別してフィルタリングできるように、別個の拡張子が付いています。 TableGenの公式ドキュメントがここにあり、必要なすべての情報を提供します。また、言語の正式な説明と一般的な紹介もあります。

もちろん、これは非常に広範なトピックであり、個々の記事を書くことができる多くの詳細があります。この記事では、すべての機能の概要がなくても、プロセッサーを説明する基本的なポイントを簡単に検討します。

.tdファイルのアーキテクチャの説明


そのため、TableGenで使用される正式な記述言語は、通常のプログラミング言語と同様の機能を備えており、アーキテクチャの特性を宣言的なスタイルで記述できます。そして、私が理解しているように、この言語は一般にTableGenとも呼ばれます。それら。この記事では、TableGenは正式言語自体の名前と、それから結果のアーティファクトを生成するユーティリティの両方を使用します。

現代のプロセッサは非常に複雑なシステムであるため、その説明が非常に膨大であることは驚くに値しません。したがって、構造を作成してサポートを簡素化するために、.tdCプログラマーが使い慣れたディレクティブを使用してファイルを相互に含めることができます#include。このディレクティブを使用すると、ファイルは常に最初にインクルードされますTarget.td必要なすべてのTableGen情報を提供するために実装する必要のあるプラットフォームに依存しないインターフェースが含まれています。このファイルはすでに含まれている.td、それは主に次のような基本的なクラスが含まれてそれ自体で説明するLLVMのintrinsikovを持つファイルをRegister、Instruction、ProcessorLLVMに基づいてそのコンパイラのためのアーキテクチャを作成するために継承する必要があり、そのためその上を。前の文から、TableGen言語には、すべてのプログラマーがよく使用するクラスの概念があることがわかります。

一般に、TableGenには、クラスと定義という 2つの基本エンティティしかありません。

クラス


TableGenクラスも、すべてのオブジェクト指向プログラミング言語と同様に抽象化ですが、より単純なエンティティです。

クラスはパラメータとフィールドを持つことができ、他のクラスを継承することもできます。
たとえば、基本クラスの1つを以下に示します。

// A class representing the register size, spill size and spill alignment
// in bits of a register.
class RegInfo<int RS, int SS, int SA> {
  int RegSize = RS;         // Register size in bits.
  int SpillSize = SS;       // Spill slot size in bits.
  int SpillAlignment = SA;  // Spill slot alignment in bits.
}

山括弧は、クラスプロパティに割り当てられる入力パラメーターを示します。この例から、TableGen言語が静的に型指定されていることにも気づくでしょう。タイプ利用可能TableGen： bit（値は0と1とのブール型のアナログ）int、string、code（コードスニペット、従来の意味でTableGenない方法および機能は、コードの行が書き込まれていないという理由だけで、このタイプ[{ ... }]）、ビット<N>リスト<タイプ>（値は角括弧を使用して設定されている[...] Pythonといくつかの他のプログラミング言語のように）class type,, dag。

ほとんどのタイプは理解する必要がありますが、質問がある場合は、記事の冒頭にあるリンクから入手できる言語仕様ですべて詳細に説明されています。

継承は、でかなりよく知られた構文によっても記述されます:。

class X86MemOperand<string printMethod,
          AsmOperandClass parserMatchClass = X86MemAsmOperand> : Operand<iPTR> {
    let PrintMethod = printMethod;
    let MIOperandInfo = (ops ptr_rc, i8imm, ptr_rc_nosp, i32imm, SEGMENT_REG);
    let ParserMatchClass = parserMatchClass;

この場合、作成されたクラスはもちろん、キーワードを使用して、基本クラスで指定されたフィールドの値をオーバーライドできますlet。また、前の例で提供された説明と同様に、フィールドのタイプを示す独自のフィールドを追加できます。

定義


定義はすでに具体的なエンティティであり、すべてのオブジェクトに馴染みのあるものと比較できます。定義はキーワードdefを使用して定義され、クラスを実装し、上記とまったく同じ方法で基本クラスのフィールドを再定義し、独自のフィールドを持つこともできます。

マルチクラス


当然のことながら、プロセッサ内の多数の命令は同様のセマンティクスを持っています。たとえば、2つの形式を取るとの3つのアドレスの命令セットがある場合が“reg = reg op reg”あり“reg = reg op imm”ます。ある場合では、値はレジスターから取得され、結果もレジスターに保管され、もう1つの場合では、2番目のオペランドは定数値（imm-即値オペランド）です。

すべての組み合わせを手動でリストするのはかなり退屈で、間違いを犯すリスクが高まります。もちろん、簡単なスクリプトを記述することで自動的に生成できますが、TableGenにはマルチクラスなどがあるため、これは必要ありません。

multiclass ri_inst<int opc, string asmstr> {
     def _rr : inst<opc, !strconcat(asmstr, " $dst, $src1, $src2"),
                 (ops GPR:$dst, GPR:$src1, GPR:$src2)>;
     def _ri : inst<opc, !strconcat(asmstr, " $dst, $src1, $src2"),
                 (ops GPR:$dst, GPR:$src1, Imm:$src2)>;
}

マルチクラスの内部では、キーワードを使用して、可能なすべての形式の命令を記述する必要がありますdef。しかし、これは生成される命令の完全な形式ではありません。同時に、フィールドを再定義して、通常の定義で可能なすべてのことを実行できます。マルチクラスに基づいて実際の定義を作成するには、キーワードを使用する必要がありますdefm。

// Instantiations of the ri_inst multiclass.
defm ADD : ri_inst<0b111, "add">;
defm SUB : ri_inst<0b101, "sub">;
defm MUL : ri_inst<0b100, "mul">;

その結果、defm実際に提供されるそのような定義ごとに、複数の定義が構築されます。これらは、メイン命令とマルチクラスで記述されたすべての可能な形式の組み合わせです。結果として、次の手順は、この例で生成されます：ADD_rr、ADD_ri、SUB_rr、SUB_ri、MUL_rr、MUL_ri。

マルチクラスには、cの定義だけdefでなく、ネストされたものも含めるdefmことができるため、複雑な形式の命令を生成できます。このようなチェーンの作成を示す例は、公式ドキュメントにあります。

TableGenの命令のプロパティとLLVMバックエンドコードの関係


クラスと定義のプロパティを使用すると、アーキテクチャ機能を正しく生成および設定できますが、LLVMバックエンドソースコードからそれらに直接アクセスすることはできません。ただし、場合によっては、コンパイラコードで直接、命令のいくつかのプラットフォーム固有のプロパティを取得できるようにする必要があります。

TSFlags


これを行うために、基本クラスにInstructionはTSFlags64ビットの特別なフィールドがありMCInstrDesc、TableGenはTableGenの説明から受け取ったデータに基づいて生成されたC ++クラスオブジェクトのフィールドに変換します。情報を格納するために必要なビット数を指定できます。これは、たとえば、スカラーALUを使用していることを示すためのブール値である場合があります。

let TSFlags{0} = SALU;

または、命令のタイプを保存できます。もちろん、1つ以上のビットが必要です。

// Instruction type according to the ISA.
IType Type = type;
let TSFlags{7-1} = Type.Value;

その結果、バックエンドコードの命令からこれらのプロパティを取得することが可能になります。

bool isSALU = MI.getDesc().TSFlags & SIInstrFlags::SALU;

プロパティがより複雑な場合は、自動生成された列挙に追加されるTableGenで説明されている値と比較できます。

(Desc.TSFlags & X86II::FormMask) == X86II::MRMSrcMem


関数述語


また、関数述語を使用して、命令に関する必要な情報を取得できます。彼らの助けを借りて、バックエンドコードで使用できるように関数を生成する必要があることをTableGenに示すことができます。このような関数定義を作成できる基本クラスを以下に示します。

// Base class for function predicates.
class FunctionPredicateBase<string name, MCStatement body> {
    string FunctionName = name;
    MCStatement Body = body;

X86のバックエンドで使用例を簡単に見つけることができます。そのため、必要な関数定義がすでに作成されている独自の中間クラスがあります。

    : FunctionPredicateBase<Name, body>, MCInstPredicate;

// Used to generate the body of a TII member function.
def IsThreeOperandsLEABody :
    MCOpcodeSwitchStatement<[LEACases], MCReturnStatement<FalsePred>>;

最適化中の命令の変換。命令マッピング


コンパイルの後の段階で実行される多数の最適化中に、あるフォームの命令のすべてまたは一部のみを別のフォームの命令に変換するというタスクがしばしば発生します。最初に説明したマルチクラスのアプリケーションを考えると、同様のセマンティクスとプロパティを持つ多数の命令を持つことができます。もちろん、コードでは、これらの変換は大規模な形式で記述できます。switch-case各命令について、変換でそれに対応して押しつぶされた構造。部分的には、これらの巨大な構造は、よく知られている規則に従って、望ましい命令名を形成するマクロの助けを借りて減らすことができます。ただし、この方法は非常に不便です。すべての命令名が明示的にリストされているため、維持するのが困難です。新しい命令を追加すると、非常に簡単にエラーが発生する可能性があります。関連するすべてのコンバージョンに必ず追加してください。LLVMはこのアプローチに悩まされて、ある形式の命令を別の形式の命令に効率的に変換するための特別なメカニズムを作成しましたInstruction Mapping。

アイデアは非常にシンプルです。命令を変換するための可能なモデルをTableGenで直接説明する必要があります。したがって、LLVM TableGenには、そのようなモデルを記述するための基本クラスがあります。

class InstrMapping {
  // Used to reduce search space only to the instructions using this
  // relation model.
  string FilterClass;

  // List of fields/attributes that should be same for all the instructions in
  // a row of the relation table. Think of this as a set of properties shared
  // by all the instructions related by this relationship.
  list<string> RowFields = [];

  // List of fields/attributes that are same for all the instructions
  // in a column of the relation table.
  list<string> ColFields = [];

  // Values for the fields/attributes listed in 'ColFields' corresponding to
  // the key instruction. This is the instruction that will be transformed
  // using this relation model.
  list<string> KeyCol = [];

  // List of values for the fields/attributes listed in 'ColFields', one for
  // each column in the relation table. These are the instructions a key
  // instruction will be transformed into.
  list<list<string> > ValueCols = [];
}

ドキュメントに記載されている例を見てみましょう。最終的なテーブルでは2つの列しか取得されないため、ソースコードにある例はさらに簡単になりました。バックエンドコードでは、命令マッピングを使用して記述された、古い形式から新しい形式の命令への変換、mmdspのdsp命令などを見つけることができます。実際、このメカニズムは今のところそれほど広く使用されていません。それは、ほとんどのバックエンドが表示される前に作成され始めたためであり、それが機能するためには、命令の正しいプロパティを設定する必要があるため、切り替えが常に簡単であるとは限らないため、いくつか必要になる場合がありますリファクタリング。

したがって、たとえば。述語のない形式の命令と、述語がそれぞれtrueとfalseである命令があるとします。フィルターとして使用するマルチクラスと特殊クラスの助けを借りてそれらを説明します。パラメータと多くのプロパティなしの簡略化された記述は、このようなものかもしれません。
