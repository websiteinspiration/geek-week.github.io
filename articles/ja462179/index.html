F文字列、またはコードをもう少し速く読みやすくする方法



Pythonには文字列をフォーマットする3つの方法があり、そのうちの1つは他の方法より優れています。しかし、先に進んではいけません。どのような書式設定について話しているのでしょうか。ユーザーに名前で挨拶するたびに、テンプレート文字列に名前の付いた文字列を挿入する必要があります。最も有用なログエントリには、変数値も含まれています。そしてここに例があります：

integer = 42
string = 'FORTY_TWO'

print('string number %s, or simply %d' % (string, integer))
print('string number {}, or simply {}'.format(string, integer))
print(f'string number {string}, or simply {integer}')

最初の方法である％演算子を使用したフォーマットは、CからPythonで行われました。これは、printf関数を模倣しています。このメソッドはpythonの最初のメソッドであり、Pythonバージョン2.5以下では（記事で説明されている）唯一のメソッドです。

2番目の方法は、組み込みの文字列クラスに属するstr.formatメソッドです。 Python 3.0に付属しており、バージョン2.6に移植されています。このメソッドは、より豊富な構文を持つため、推奨されています。

3番目のメソッドf-stringは、Pythonバージョン3.6で登場しました。PEP-0498で説明されているように、文字列をフォーマットする新しい方法の作成は、既存のメソッドの欠点によって動機付けられました。
このPEPは、Pythonで文字列をフォーマットするより簡単な方法を持ちたいという願望によって推進されています。既存のフォーマット方法は、エラーが発生しやすく、柔軟性に欠けるか、扱いにくいものです。
したがって、1つの問題を解決するには3つの方法があります。しかし、これはおそらく個人的な好みと好みの問題でしょうか？おそらく、しかし、コードのスタイル（特に、多数の参加者がいるプロジェクトのコード）は、統一性から確実に利益を得ます。最良のケースでは、文字列をフォーマットする1つの方法を使用する必要があります。そうすると、コードが読みやすくなります。しかし、どの方法を選択するのですか？そして、コードのパフォーマンスに違いはありますか？

パフォーマンスに関する質問に実験的に答えてみましょう。

import timeit

setup = """
integer = 42
string = 'FORTY_TWO'
""".strip()

percent_stmt ="'Number %s or simply %d' % (string, integer)"
call_stmt = "'Number {} or simply {}'.format(string, integer)"
fstr_stmt = """f'Number {string} or simply {integer}'"""


def time(stmt):
    return f"{timeit.timeit(stmt, setup, number=int(1e7)):.3f}"

print(f"Timing percent formating:    |  {time(percent_stmt)}")
print(f"Timing call formating:       |  {time(call_stmt)}")
print(f"Timing f-string formating:   |  {time(fstr_stmt)}")

Python 3.7を搭載したMacBookでの結果：

Timing percent formating:    |  2.025
Timing call formating:       |  2.943
Timing f-string formating:   |  1.348

違いは重要です。では、「。format」で正規表現検索を実行し、数百の式を書き換えますか？原則として、タスクは単純ですが時間がかかります。さらに、以前に機能していたコードに間違いを犯してバグを追加するチャンスもあります！自動化の余地があるようです。実際、ほとんどの式をf文字列に変換できるライブラリがあります：flynt、pyupgrade。

また、数行を占める式を変換したり、加えられた変更に関する統計を収集したりする可能性も注目に値します。フラグ--line_length XXは、変換後の行の長さの制限を定義します。 Flyntでは、-upgradeフラグを指定してpyupgradeを呼び出すことができます。

Pyupgradeは、より多くの機能を備えており、多くのPython 2アーティファクトからあなたのコードをきれいにすることができる-このような超にクラス名を指定して、オブジェクトからの継承など、および大いに多く。 Pyupgradeは、コミット前にコードを自動的に変更するユーティリティであるpre-commitで使用するように設計されています。

ソースをgitaまたは他のバージョン管理で変換することをお勧めします。テストを実行して、自分で変更を確認する価値があります（git diffまたはPyCharm環境を使用）。私たちが気になる人々の間で暮らしている限り、コードが数文字短くなったので、プロアクティブな変換も時間を節約します。結局のところ、遅かれ早かれ誰かがユーティリティになることができるものを自分の手で始めます。 F文字列はPython 3.6以降でのみ機能しますが、他のバージョンは廃止されるため、間もなくこれは問題になりません。

従来の.formatメソッドを完全に破棄すると失敗することに注意してください。同じテンプレートを使用して、コード内のさまざまな場所にさまざまな変数を含むメッセージを作成する場合は、このテンプレートを変数に保存して使用する必要があります。「自分自身を繰り返さない」という原則は、文字列のフォーマットからナノ秒を獲得するよりもはるかに重要です。

結論：

Python 3.6以降で利用できる3つの文字列フォーマット方法とその簡単な履歴を見て、パフォーマンスを比較しました。また、コードを文字列をフォーマットするための新しいメソッドに自動的に変換するための既存のオープンソースユーティリティと、それらの追加機能も調べました。コードの簡単なこと、そして頑張ってください！
