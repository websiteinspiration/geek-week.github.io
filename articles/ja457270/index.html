MVPを迅速に作成するためのエンジンとしてのPrisma-CMS

確かに多くの人がMVP（Wiki Minimum Viable Product ）の概念を聞いています。 MVPに関するMVPに関する記事はたくさんありますが、基本的には、MVPの概要とその目的、またはさまざまな成功例であり、ストーリーではありません。しかし、MVPが行ったことを説明する記事は1つも見つかりませんでした。しかし、個人のブログは1つのエンジンで実行したり、オンラインストアを別のエンジンで実行したりするほうが簡単だと考えられています。 （これらの目的のために、それぞれがお気に入りのエンジンをその名前に置き換えます）。しかし、なぜMVPのより便利なエンジンが定義されていないのでしょうか。この質問に対する明確な回答はしませんが、MVPを作成するために正確に私のPrisma CMSが何であるかについて私の考えを共有します。これについては、数か月前にここで書きました。猫の下で誰が気にかけてください。

興味深いことに、概して、MVPはより高度なプロトタイピングwikiと比較できます。同時に、プロトタイピング用の特別なソフトウェアがあり、たまに同じAxureを使用することもありました。しかし、Axureには何が欠けていたのでしょうか。なぜそれをすぐに放棄したのですか。（わかりません、おそらく彼はより高度になり、これらの問題は解決されましたが、当時はありませんでした）。

  1. 実際のデータを扱う作業はありません。つまり、プロトタイプにUsersセクションがありますが、APIを介してユーザーデータを取得してテンプレートのループに表示することができません。そして一般的には、原則として、実際のデータを操作する（レコードを作成し、読み取る）方法はありません。
  2. プロトタイピング段階を通過して直接開発に移る場合、プロトタイプで作成されたテンプレートを使用することはできません。つまり、プロトタイプが作成されて顧客と合意した後、プログラミングに切り替えたとき、プロトタイプを目で見るだけでそれ以上何もできませんでした。そして、私はプロトタイプを投げて、それを開発で使用したかったのです。

まだ瞬間はありましたが、この2つが最も重要です。プロトタイピングとプログラミングの段階は、それぞれ独自の生活を送っていて、お互いに影響を与えていないことがわかりました。ただし、プロトタイピングの段階から直接プログラミングにスムーズに移行してほしい。そして、現在の記事の枠組みの中で、プロトタイピングがMVPの作成に流れ込み、成功すれば、MVPを最終的な本格的な製品に開発することを提案することもできます。確かに、あなたがそう思うなら、MVPを作成しても、将来、より完全な製品が存在することは必ずしも保証されません。結局、MVPを作成する意味は何ですか？ 1つ目は、最小限のコストでアイデアを実装して、実際に試してみることができるようにすること、2つ目は、需要を調査し、それを投資して本格的な製品に開発する価値があるかどうかを自分で判断することです。そして、私たちはツールが必要であることがわかりました、これにより、ラピッドプロトタイピング/ MVPが可能になります。一方、MVPが成功した場合は、特別な制限なしにプロジェクトをさらに発展させることができます。

実際のところ、Prisma CMSを見ると、そのようなエンジンだけが表示されます。プロジェクトのドラフトをすぐに投げ出すにはたくさんあります：

  • ほとんどすべての作業は、独自のWYSIWYGフロントエンドエディターを介してフロントで行われます。
  • URLはreact-routerに実装されているため、サーバーではなく、エディターの前面に直接アドレスを書き込むことができます。
  • APIはGraphQLに実装されているため、サーバーではなくフロントでクエリを記述できます。同時に、Graph i QL が統合され、クエリの作成プロセスが簡素化されます。
  • さらに、生成されたフィルター、ページング、その他の便利なパンがあります。

そして、これはすべてオープンソースです。つまり、SaaSではありません。そのため、常に（すべてを自由に利用できるようにすることなく）支払う必要があります。これはサーバーにデプロイして、自分用にカスタマイズできます。

サーバーにインストールするプロセスとカスタマイズプロセスを説明する前に、サイトに別のセクションを作成するプロセスの記録を確認することをお勧めします。約15分ですが、最初の4分だけを見てください。これは、Prisma CMSのアイデアを理解するのに十分です。興味があれば、トピックをさらに読むか、おそらく自宅でエンジンをデプロイしてみてください。


そのため、サーバーにPrisma CMSをインストールします（私はubuntuを使用し、できれば少なくとも4GbのRAMとSSDを使用しています）。

node-js、npm / yarn、react、graphqlに既に慣れていることを前提としています。

1.必要なソフトウェアをインストールします


端末にすべてを一度に挿入すると、実行が中断する可能性があるため、1行ずつ実行することをお勧めします。

これは、情報提供のみを目的としたアクセスレベル設定などのない最小限のインストールです。つまり、新しいクリーンなサーバーを試すと、すべてがルートから実行されます（サイトの起動を含む）。私たちの実験では、これはまったく重要ではありません。

sudo apt update
sudo apt install mc
sudo apt install git

curl -sL https://deb.nodesource.com/setup_10.x | sudo -E bash -
sudo apt-get install -y nodejs

sudo npm i -g yarn

sudo apt-get install software-properties-common python-software-properties
sudo apt-key adv --keyserver hkp://p80.pool.sks-keyservers.net:80 --recv-keys 58118E89F3A912897C070ADBF76221572C52609D
sudo apt-add-repository 'deb https://apt.dockerproject.org/repo ubuntu-xenial main'
sudo apt-get update

sudo apt-get install -y docker-engine

sudo curl -L https://github.com/docker/compose/releases/download/1.18.0/docker-compose-`uname -s`-`uname -m` -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose

2.リポジトリのクローンを作成します


mkdir /var/www
cd /var/www
git clone https://github.com/prisma-cms/boilerplate
cd boilerplate
yarn --ignore-engines

3. MySQLとPrismaを使用してDockerコンテナーを起動します（Prismaは私の製品ではなく、別の製品ですが、サーバー側が強く接続されています。github.com /prisma/prismaを参照してください）。

sudo docker-compose -f src/server/scripts/docker/prisma/docker-compose.yml up -d

phpMyAdminを介してデータベースを調べたい場合は、それもインストールします。

sudo docker run -d --link prisma_mysql_1:db --network prisma_default -p 8090:80 phpmyadmin/phpmyadmin

ポート8090でハングします。デフォルトでは、ユーザー名/パスワードはroot / prismaです。

4.展開スキーム


endpoint=http://localhost:4466/my-project/dev yarn deploy

my-project / dev、それぞれ、独自のサーバーを作成し、同じサーバー上に多数のプロジェクトを作成できます。

重要！本番環境では、ポート4466はファイアウォールで閉じる必要があります。外部から直接アクセスできないようにする必要があります。

5. APIサーバーを起動します


APP_SECRET=MyStrongSecret endpoint=http://localhost:4466/my-project/dev yarn start-server

起動後、server-ip：4000 APIを開くことができます。

以前の記事で書いたように、これは生成されたロジック（ポート4466で実行されます）の上に独自のロジックを実装する中間サーバーです。

6.フロントを起動します


ターミナルをもう1つ開き、同じフォルダで実行します 

yarn start

フロントはポート3000から開始します。これで、フロントの「プログラミング」に直接進むことができます。server-ipを開きます：3000。入力すると、前面に保存されたテンプレートが1つもありませんが、承認ボタンが表示されます。ロジックは、登録してサイトの設計を開始することです。誰が最初です- それとスニーカーのサイト。これは次のようになります。


7.スクリプトをビルドしてSSR（Server-Side Rendering）を実行します


ヤーンスタートによる開始は、最初の開始（すべてが機能することの確認）およびJSパーツの直接プログラミングに適した開発モードです。そしてもちろん、バトルには戦線を集める必要があります。Ctrl + Cで糸のスタートを中断し、スクリプトをビルドします。

yarn build

タバコを吸ってコーヒーを注ぐことができますが、このプロセスは迅速ではありません。まれに、建物が最初にバラバラになり、もう一度実行するだけです。その場合、原則として、建物は正常に完了し、より速く完了します。

建物が終わったら、組み立てられたフロントを実行します。

yarn start-ssr

起動がはるかに速くなり、スクリプトが小さくなりました。

結論


ビデオは、ユーザビリティに非常に大きな問題がある場所を示していますが、プロジェクトはまだ開発中であり、これらの問題は徐々に解消されています。しかし、多くのことを正面で行うことができます。すでに本当に必要とされていないすべてのものが蓄積されているため、ソースコードの深刻なクリーニングはまだ行われていませんが、単にレガシーとして拡張されています。その結果、フロントははるかに簡単になります。そして、それはより速く組み立てられ、ページをより速くロードします。

いずれにせよ、私は私の考えでは、MVPのエンジンがどのように非常にうまく適合できるかを繰り返します-すばやく何かを投げてクライアントに見せます。また、最終製品の開発に進むと、フロントが不満でも、完全に捨てて独自の方法で書き直すことができ、データベースやサーバーロジックはどこにも行きません。結局のところ、これはヘッドレスcmsです。しかし、時間の経過とともに、正面が完全に許容できるレベルに到達することを願っています。

コミュニティが関心を持っている場合は、前面をカスタマイズする方法（前面エディタの独自のブロックを追加）とサーバーをカスタマイズする方法（スキームを拡張する、独自のエンティティを追加する、リゾルバを追加するなど）について、個別にトピックを書きます。そのようなタスクを迅速に完了するために、かなり多くのツールがすでに蓄積されています。
