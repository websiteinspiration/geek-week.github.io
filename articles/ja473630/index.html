iOSアプリケーションを開発するためのベストプラクティスとツール

モバイルアプリケーションを開発するとき、プロジェクトを最初から何度も作成する必要がありました。同時に、私のチームと私は、サードパーティツールの統合、プロジェクト構造の設定、基本クラスの作成、外部ライブラリの統合など、メインプロジェクトのセットアップに常に多くの時間を費やしていました。

プロジェクトの立ち上げにかかる時間を短縮でき、プロセスの主要部分を自動化できると判断しました。同時に、使用したベストプラクティスとツールを収集し、プロジェクトテンプレートを準備しました新しいプロジェクトを起動するときに使用します。このようなテンプレートを使用すると、開発者が新しいプロジェクトを構成する時間を節約できるだけでなく、すべてのチームメンバーが使い慣れた共通のプロジェクト構造を提供できるため、開発者が新しいプロジェクトの構造を考えて検討する必要がなくなります。今では常に同じです。

テンプレートに追加されたツールまたはアプローチはそれぞれ個別の記事に値しますが、各段落を要約して、なぜこの記事にそれらを含めたのかについて簡単に説明したいと思います。

ココアポッド


Cocoapodsは提出が必要だとは思いません。これは、 iOSプロジェクトの外部依存関係を管理するためのライブラリです。それは長い間存在しており、信頼性が高く、数千の（数百万ではないにしても）プロジェクトでフィールドで実証されています。たとえば、Carthageなどの代替の依存関係マネージャーもありますが、サポートされているオープンソースプロジェクトの範囲が最も広いため、Cocoapodを続行することにしました。 Cocoapodsの使用は非常に簡単で、いつでも便利なパッケージを簡単に見つけることができる検索インデックスが付属しています。

テンプレートプロジェクトには、SwiftlintとR.swiftを含むシンプルなPodfileが用意されています。テンプレートには、gemfileも含まれています。依存関係の管理に使用されるCocoapodのバージョンを管理します。このソリューションは開発者によって無視されることがよくありますが、チームの開発者が異なるバージョンのCocoapodを使用して依存関係をインストールするときに発生する問題を防ぎます。Gemfileは、チーム全体で同じバージョンのCocoapodsを強制的に使用します。

スウィフトティント


Swiftlintは、同じチームの各開発者に特定のルールとライティングスタイルを適用するための非常に便利なツールです。このシステムは、フォースキャスト、フォーストライなどの危険な瞬間について開発者に警告する自動コード検証システムと見なすことができますが、上記に加えて、このシステムは一般的なスタイルのコードを記述し、監視しますすべての開発者が「コードスタイル」に関連する同じ規則、たとえばインデントまたは間隔に従うようにします。このアプローチには、コード検証の時間を節約するだけでなく、プロジェクトファイルを認識可能にする大きな利点があり、読みやすさが向上し、結果として、開発チームのすべてのメンバーによる理解が深まります。このリンクすべてのルールのリストが提供されます。Swiftlintテンプレートでは、Cocoapodsを使用してインストールされ、「コンパイルフェーズ」ステップに含まれるため、プロジェクトがコンパイルされるたびに開発者に警告が表示されます。

R.スウィフト


R.swiftは、画像、フォントセグメント、ローカリゼーションなど、厳密に型指定され、自動的に入力されるリソースを取得するためのツールです。プロジェクトをスキャンし、リソースを取得するために必要なクラスを作成することにより、上記のすべてのアクションを実行します。このライブラリの最大の利点は、リソースを使用するときにプログラムコードを作成できることです。

  • 完全に型付け -キャストが少なく、返されるメソッドに関する仮定が少ない
  • コンパイル時間のチェック -コード実行中にアプリケーションが機能を停止する原因となる無効な行がなくなりました
  • オートコンプリート -名前image / nib / Storyboardを再度推測する必要はありません

公式の文字列APIを使用する次のコードを検討してください。

let icon = UIImage(named: “custom-icon”)

画像の名前を間違えると、ゼロになります。チームのメンバーが画像リソースの名前を変更した場合、このコードはゼロを返すか、画像を強制的に拡張すると実行が停止します。R.swiftを使用すると、次のようになります。

let icon = R.image.customIcon()

これで、アイコンが実際に存在することを確認でき（存在しない場合は、コンパイル時のチェックにより、コンパイラーから警告が表示されます）、オートコンプリートを使用するため、アイコンの名前にタイプミスがないことを確認できます。

R.swiftはCocoapodsを介してインストールされ、コンパイルフェーズでテンプレートに統合され、コンパイルごとにSwiftシェルクラスを生成します。これは、ファイル、画像、ローカリゼーション、フォント、カラー、ペンなどを追加した場合、プロジェクトをコンパイルした後、R.swiftからすべて利用できることを意味します。

テスト用の別のAppDelegateファイル


多くの場合、テストの実行時に別のTestAppDelegateクラスを使用することを忘れています。なぜこれが良い考えなのですか？通常、AppDelegateクラスは、アプリケーションの起動時に多くの作業を行います。ウィンドウ構成ロジック、アプリケーションユーザーインターフェイスの基本表示の構成、通知を受信するための登録ロジックの実行、データベース接続セットアップコードの使用、さらにはサーバーAPI呼び出しを行うこともできます。単体テストには副作用があってはなりません。単体テストを実行するためだけに、ランダムなAPI呼び出しを行い、アプリケーションのユーザーインターフェイス構造全体をカスタマイズしたくないのですか？

TestAppDelegateは、テストスイートの実行中に1回だけ実行するコードを取得するための適切な選択でもあります。これには、モックオブジェクト、スタブネットワークリクエストなどを生成するコードを含めることができます。

テンプレートには、アプリケーションのメインエントリポイントであるmain.swiftファイルが含まれています。このファイルには、アプリケーションの環境をテストするメソッドがあり、それがテスト環境の場合は、TestAppDelegateを呼び出します。

コンパイラパフォーマンスプロファイリングフラグ


Swiftは優れた言語であり、Objective-C（IMO）よりも使いやすく安全です。しかし、それが最初に導入されたとき、1つの大きな欠陥がありました-コンパイル時間です。私がSwiftでプロジェクトに取り組んでいた当時、約4万行のSwiftコード（中規模プロジェクト）がありました。コードは非常に重く、設定と型推論があり、クリーンなアセンブリのコンパイルには約5分かかりました。小さな変更を加えた場合でも、プロジェクトは再コンパイルされ、変更を確認するには約2分かかります。これは私が今まで経験した中で最悪の経験の1つであり、このため、Swiftの使用をほぼ中止しました。

次に、唯一の解決策は、プロジェクトのコンパイル時間のプロファイルを作成し、コンパイラーがより速く動作するようにコードを変更することです。このような問題を解決するために、Appleは非公式のコンパイラフラグを導入し、メソッド本体のコンパイルや式のタイプの決定に時間がかかりすぎることを開発者に警告しています。これらのフラグをテンプレートプロジェクトに追加して、アプリケーションのコンパイル時間が長いことを警告しました。

今日では、コンパイル時間が大幅に短縮されており、開発者がコンパイル時間を短縮するためだけにコードを最適化する必要はほとんどありません。ただし、プロジェクトが大きくなりすぎたときにこの問題を解決するよりも、事前にすべてを把握しておく方が適切です。

Readme


各プロジェクトにはReadmeファイルが含まれている必要があります。このファイルには、依存関係のインストールとプロジェクトの開始に関する指示が少なくとも含まれています。また、プロジェクトのアーキテクチャとモジュールの説明も含まれている必要があります。残念ながら、開発者はドキュメンテーションを書くのが好きではなく（readmeファイルはこの作業の一部です）、何ヶ月も開発されているプロジェクトを見て、Readmeファイルさえありました。readmeファイルを作成する負担を取り除くために、テンプレートには、プロジェクトのインストールと構造をカバーする標準のreadmeファイルが含まれています。テンプレートを使用して新しいプロジェクトを設定すると、Readmeファイルが自動的に含まれます。

ギティニョーレ


現在、ほとんどのプロジェクトはバージョン管理システムとしてGITを使用しています。GITを使用する場合、開発者はプロジェクト内の一部のファイルまたはフォルダー（ビルドフォルダーや派生データフォルダーなど）を無視したくありません。開発者がiOSプロジェクトに適したgitignoreファイルを見つける必要がないように、テンプレートにはGithubのメンバーが提供する標準のgitignoreが含まれています。

外部リンクと通知を処理するための基本クラス


今日では、ほとんどすべてのアプリケーションが外部リンクと通知を処理する必要があります。これを行うには、開発者はAppDelegateクラスにいくつかの標準コードを記述する必要があります。このテンプレートには、これを行う方法の説明が含まれています。また、外部リンクと通知の操作を容易にする基本クラスも含まれています。

結論


要約すると、説明されているテンプレートには最良のアプローチが含まれ、有用なサードパーティツールが統合されています。これにより、新しいプロジェクトのセットアップに費やされるチームの開発時間を節約できるだけでなく、プロジェクトの残りの部分に共通の強固な基盤を提供できます。このテンプレートを使用して、あなたとあなたのチームに利益をもたらしましょう！
