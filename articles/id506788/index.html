Migrasi Database dengan Flyway

Terjemahan artikel disiapkan sebelum dimulainya kursus Pengembang Java .


1. Perkenalan


Artikel ini menjelaskan konsep-konsep kunci Flyway dan contoh menggunakan kerangka kerja ini untuk terus mengubah skema database menggunakan in-memory H2 database sebagai contoh menggunakan plugin flyway maven.

Flyway memperbarui versi database menggunakan migrasi. Migrasi dapat ditulis dalam SQL (dengan sintaks khusus untuk DBMS tertentu) atau di Jawa.

Migrasi dapat diversi atau diulang. Yang pertama memiliki versi unik dan diterapkan tepat sekali. Yang kedua tidak memiliki nomor versi, dan mereka berlaku ketika checksum mereka berubah.

Migrasi berulang dalam sekali jalankan selalu diterapkan setelah migrasi berversi selesai. Migrasi berulang diterapkan sesuai urutan yang dijelaskan. Dalam satu migrasi, semua operasi dilakukan dalam satu transaksi basis data tunggal.

Pada artikel ini, kami akan fokus menggunakan plugin pakar untuk migrasi basis data.

2. Flyway maven plugin


Tambahkan plugin flyway maven ke pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
</plugin>

Versi plugin saat ini dapat dilihat di Maven Central .
Daftar parameter plugin dapat ditemukan di dokumentasi . Parameter plugin dapat dikonfigurasi dalam empat cara berbeda.

2.1. Bagian <configuration>plugin
Parameter dapat ditentukan secara langsung di tag di bagian plugin di pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
    <configuration>
        <user>databaseUser</user>
        <password>databasePassword</password>
        <schemas>
            <schema>schemaName</schema>
        </schemas>
        ...
    </configuration>
</plugin>

2.2. Sifat Maven


Anda juga dapat mengkonfigurasi plugin dengan menentukan parameter di <properties>:

<project>
    ...
    <properties>
        <flyway.user>databaseUser</flyway.user>
        <flyway.password>databasePassword</flyway.password>
        <flyway.schemas>schemaName</flyway.schemas>
        ...
    </properties>
    ...
</project>

2.3. File konfigurasi eksternal


Atau jelaskan konfigurasi dalam .propertiesfile terpisah :

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=schemaName
...

Nama default adalah file konfigurasi flyway.properties. File ini harus berada di direktori yang sama dengan file tersebut pom.xml. Pengkodean diatur dalam parameter flyway.encoding(standarnya adalah UTF-8).

Jika Anda menggunakan nama lain untuk file tersebut (misalnya, customConfig.properties), maka Anda harus menentukannya secara eksplisit saat memanggil pakar:

$ mvn -Dflyway.configFile=customConfig.properties

2.4. Properti sistem


Akhirnya, parameter dapat ditentukan sebagai properti sistem saat memanggil pakar dari baris perintah:

$ mvn -Dflyway.user=databaseUser -Dflyway.password=databasePassword
  -Dflyway.schemas=schemaName

Jika konfigurasi ditentukan dalam beberapa cara, prioritasnya adalah sebagai berikut:

  1. Properti sistem
  2. File konfigurasi eksternal
  3. <properti bagian>
  4. Bagian <konfigurasi >plugin

3. Contoh migrasi


Pada bagian ini, kami akan mempertimbangkan langkah-langkah yang diperlukan untuk memigrasi skema database menggunakan database H2 di-memori sebagai contoh menggunakan plugin maven. Untuk mengkonfigurasi Flyway, kami akan menggunakan file eksternal.

3.1. POM berubah


Untuk memulai, tambahkan ketergantungan pada H2:

<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>1.4.196</version>
</dependency>

Di sini kita juga dapat memeriksa versi driver terbaru yang tersedia di Maven Central . Tambahkan plugin untuk Flyway, seperti yang dijelaskan sebelumnya.

3.2. Konfigurasikan Flyway dalam file eksternal


Buat $PROJECT_ROOTfile myFlywayConfig.propertiesdengan konten berikut:

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=app-db
flyway.url=jdbc:h2:mem:DATABASE
flyway.locations=filesystem:db/migration

Konfigurasi di atas menunjukkan bahwa skrip migrasi ada di direktori db/migration, dan untuk terhubung ke database menggunakan H2 databaseUserdan databasePassword.

Skema basis data untuk aplikasi app-db.

Tentu saja, dalam parameter flyway.user, flyway.passworddan flyway.urlAnda harus menentukan nama, kata sandi, dan URL basis data Anda.

3.3. Migrasi pertama


Di bawah konvensi Flyway, nama skrip migrasi harus dalam format berikut:

<Prefix<code>><</code>Version>__<code><</code>Description>.sql

Dimana:

  • <Awalan> - awalan. Untuk migrasi berversi, standarnya adalah "V". Awalan dikonfigurasi melalui properti flyway.sqlMigrationPrefix.
  • <Versi> - nomor versi migrasi. Versi utama dan kecil dapat dipisahkan dengan menggarisbawahi. Versi harus selalu dimulai dengan 1.
  • <Description> - deskripsi tekstual dari migrasi. Deskripsi harus dipisahkan dari nomor versi oleh dua garis bawah.

Contoh: V1_1_0__my_first_migration.sql

Jadi, mari kita buat direktori db/migrationdalam $PROJECT_ROOTmigrasi skrip V1_0__create_employee_schema.sqldan SQL untuk membuat tabel employee:

CREATE TABLE IF NOT EXISTS `employee` (
 
    `id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name` varchar(20),
    `email` varchar(50),
    `date_of_birth` timestamp
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;

3.4. Kami melakukan migrasi


Selanjutnya, $PROJECT_ROOTjalankan perintah pakar berikut untuk menerapkan migrasi database:

$ mvn clean flyway:migrate -Dflyway.configFile=myFlywayConfig.properties

Migrasi pertama kami harus diselesaikan.

Sekarang skema database terlihat seperti ini:
karyawan:

+----+------+-------+---------------+
| id | name | email | date_of_birth |
+----+------+-------+---------------+

Kami dapat mengulangi langkah-langkah sebelumnya untuk melakukan migrasi lainnya.

3.5. Migrasi kedua


Untuk migrasi kedua, buat file dengan nama yang V2_0_create_department_schema.sqlberisi dua permintaan berikut:

CREATE TABLE IF NOT EXISTS `department` (
 
`id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
`name` varchar(20)
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;
 
ALTER TABLE `employee` ADD `dept_id` int AFTER `email`;

Mari lakukan migrasi, sama seperti yang kami lakukan untuk migrasi pertama.

Sekarang skema database kami telah berubah: employeekolom baru telah ditambahkan dan tabel baru telah dibuat department:

employee:
+----+------+-------+---------+---------------+
| id | name | email | dept_id | date_of_birth |
+----+------+-------+---------+---------------+

department:

+----+------+
| id | name |
+----+------+

Untuk memverifikasi bahwa kedua migrasi berhasil, jalankan perintah pakar berikut:

$ mvn flyway:info -Dflyway.configFile=myFlywayConfig.properties

4. Menonaktifkan Flyway di Spring Boot


Terkadang Anda mungkin perlu menonaktifkan migrasi Flyway.

Ini mungkin diperlukan selama tes ketika skema database dihasilkan berdasarkan entitas. Dalam hal ini, Anda dapat menonaktifkan Flyway untuk profil pengujian.

Di Spring Boot, ini sangat sederhana.

4.1. Boot musim semi 1.x


Yang perlu kita lakukan adalah mengatur properti flyway.enableddi file application-test.properties:

flyway.enabled=false


4.2. Boot musim semi 2.x


Di versi Spring Boot yang lebih baru, properti ini telah diubah menjadi 

spring.flyway.enabled:
spring.flyway.enabled=false

4.3 Strategi Jalur Terbang Kosong


Jika kami ingin menonaktifkan hanya migrasi Flyway otomatis saat startup, tetapi ingin memulai migrasi secara manual, maka menggunakan properti di atas tidak akan bekerja untuk kami.

Ini disebabkan oleh fakta bahwa Spring Boot tidak akan secara otomatis mengkonfigurasi nampan Flyway dan, oleh karena itu, kami harus mengonfigurasinya sendiri, yang sangat tidak nyaman.

Dalam hal ini, kita dapat membiarkan Flyway diaktifkan dan mengimplementasikan FlywayMigrationStrategy yang kosong :

@Configuration
public class EmptyMigrationStrategyConfig {
 
    @Bean
    public FlywayMigrationStrategy flywayMigrationStrategy() {
        return flyway -> {
            // do nothing 
        };
    }
}

Bahkan, ini akan menonaktifkan migrasi Flyway ketika aplikasi dimulai.

Tetapi kami masih dapat memulai migrasi secara manual:

@RunWith(SpringRunner.class)
@SpringBootTest
public class ManualFlywayMigrationIntegrationTest {
 
    @Autowired
    private Flyway flyway;
 
    @Test
    public void skipAutomaticAndTriggerManualFlywayMigration() {
        flyway.migrate();
    }
}

5. Bagaimana Flyway Bekerja


Untuk melacak kapan, oleh siapa dan migrasi apa yang diterapkan, tabel khusus dengan metadata ditambahkan ke skema database. Tabel ini juga menyimpan checksum migrasi dan informasi apakah migrasi berhasil atau tidak.

Kerangka kerjanya sebagai berikut:

  1. Periksa skema database untuk tabel metadata (default SCHEMA_VERSION). Jika tabel metadata tidak ada, itu akan membuatnya.
  2. Memindai classpath untuk migrasi yang tersedia.
  3. Membandingkan migrasi ke tabel metadata. Jika nomor versi kurang dari atau sama dengan versi yang ditandai sebagai saat ini, maka nomor tersebut diabaikan.
  4. Tandai semua migrasi yang tersisa sebagai yang tertunda. Kemudian mengurutkan mereka dalam nomor versi naik dan mengeksekusi mereka dalam urutan yang ditentukan.
  5. Saat migrasi berlaku, perbarui tabel metadata.

6. Tim


Flyway memiliki tim manajemen migrasi utama berikut:

  • Info . Menampilkan status / versi saat ini dari skema database. Informasi tentang migrasi mana yang diharapkan, yang diterapkan, status migrasi yang selesai dan tanggal penyelesaiannya.
  • Bermigrasi . Memperbarui skema database ke versi saat ini. Pindai classpath untuk menemukan migrasi yang tersedia dan menerapkan migrasi yang tertunda.
  • Baseline . Menginstal versi skema database, mengabaikan migrasi hingga dan termasuk baselineVersion. Baseline membantu Anda menggunakan Flyway pada basis data yang ada. Migrasi baru diterapkan seperti biasa.
  • Memvalidasi . Memeriksa skema database saat ini terhadap migrasi yang tersedia.
  • Memperbaiki . Pulihkan tabel metadata.
  • Clean. . , clean .
