UIテストを使用したAndroidでのテストによる開発の習得

みなさん、こんにちは。Android開発に関する基本コースと上級コースの新しいセットの立ち上げを見越して、興味深い資料の翻訳を準備しました。



バッファのAndroid開発チームの昨年の仕事を通じて、私たちはプロジェクトの清潔さとプロジェクトの安定性の向上について多くのことを話しました。要因の1つはテストの導入（）でした。これは、既にわかっているように、コードのリグレッションを回避し、提供する機能に対する信頼を高めるのに役立ちます。そして今、Bufferで新製品を発表するときは、以前と同じ状況にならないように、同じアプローチがそれらに適用されることを確認したいと思います。

バックグラウンド


バッファーアプリケーションの単体テストを作成するときは、テスト（テスト駆動開発-以降TDD）を通じて常に開発プラクティスを順守してきました。TDDとは何か、その利点、およびその出所については多くのリソースがありますが、インターネットには多くの情報があるため、このトピックについては詳しく説明しません。高いレベルで、私はそれらのいくつかを個人的に指摘します：

  • 開発時間の短縮
  • シンプルでクリーンでメンテナンスしやすいコード
  • 私たちの仕事にもっと自信を持って、より信頼できるコード。
  • 高いテストカバレッジ（これは一種の明白なようです）

しかし、最近まで、ユーザーインターフェイスに基づいていない実装の単体テストの形でのみTDDの原則に従いました...



わかっています、わかりました...実装されたものが完了した後は、常にUIテストを書く習慣がありました-そして、それは意味がありません。作成したコードがテストで定義された要件を満たすように、バックエンドのTDDを監視しましたが、ユーザーインターフェイステストに関しては、特定の機能の実装を満たすテストを作成しました。ご覧のとおり、これは少し物議を醸しています。いくつかの点で、そもそもTDDが使用されている理由です。

だから、ここではなぜそうなのか、そしてどのように変化を実験しているのかを考えたいと思います。しかし、なぜこれに最初から注意を払うのでしょうか。

アプリケーションで既存のアクティビティを処理することは、その記述方法が原因で常に困難でした。これは完全な言い訳ではありませんが、多くの依存関係、責任、密接な関係があるため、テストが非常に困難です。私たちが追加した新しいアクティビティについては、習慣から、私自身は常に実装後にUIテストを作成しました-習慣以外に、これ以外に理由はありませんでした。しかし、新しいプロジェクトの準備ができているテンプレートコードを作成するとき、変更について考えました。そして、あなたはこの習慣が破られたことを知って喜んでいるでしょう、そして今、私たちはUIテストのためにTDDを模索して自分たち自身で取り組んでいます

テストを追加する


したがって、実装が必要な状況が3つあるので、それらのテストをいくつか作成します。

  • ユーザーがメール入力フィールドに無効なメールアドレスを入力した場合、エラーメッセージを表示する必要があります。そこで、このエラーメッセージが表示されるかどうかを確認するテストを作成します。
  • ユーザーが電子メール入力フィールドにデータを再度入力し始めると、上記のエラーメッセージが消えるはずです。そのため、このテストを作成します。
  • 最後に、APIがエラーメッセージを返すと、警告ダイアログボックスに表示されるはずです。そのため、このテストも追加します。

続けて、もう一度テストを実行してみましょう



。先に進んでいinvalidEmailErrorHidesWhenUserTypes()ます。テストは1つだけ残っています。これはテストです。繰り返しますが、これは単純なケースですが、見てみましょう。

  • ユーザーがログインボタンをクリックしたときにメールアドレスがないか、メールアドレスが正しくない場合、ユーザーにエラーメッセージが表示されます。これはすでに実装済みですが、簡単にするために除外しました
  • ただし、ユーザーがフィールドにデータを再度入力し始めると、エラーメッセージは表示から削除されます。これを行うには、入力フィールドのテキストコンテンツが変更されたときにリッスンする必要があります。

private fun setupOnEmailTextChangedListener() {
    input_email.addTextChangedListener(object : TextWatcher {

        override fun afterTextChanged(s: Editable) {}

        override fun beforeTextChanged(s: CharSequence, start: Int,
                                       count: Int, after: Int) {
        }

        override fun onTextChanged(s: CharSequence, start: Int,
                                   before: Int, count: Int) {
            loginPresenter.handleEmailTextChanged(s)
        }
    })
}

これで、入力フィールドの内容を変更するときにエラーメッセージが非表示になることを保証するにはこれで十分です。ただし、変更を確認するためのテストがあります



。テストに合格すると、実装要件が満たされます。青信号が表示されるのは素晴らしいことです。
