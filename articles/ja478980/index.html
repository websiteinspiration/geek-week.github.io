Azure SDK for .NET：難しいバグ検索のストーリー

Azure SDK for .NETプロジェクトでエラーを探すことにしたとき、そのサイズに驚きました。「350万行のコード」と私たちは言って、プロジェクトの統計を調べました。これはあなたがそこに見つけることができる量です。しかし、ああ、ああ。プロジェクトは秘密であることが判明しました。このプロジェクトの特殊性と、それがどのようにテストされたか-この記事を読んでください。

プロジェクトについて


この記事は、前回の記事（Microsoft Azureに関連するプロジェクトに関するもの）のフォローアップとして作成しています。AzurePowerShell：「基本的に無害」。ですから、今回は、さまざまな興味深いエラーを確実に数えていました。実際、静的分析の場合、特にプロジェクト全体の1回限りのチェックでは、プロジェクトのサイズが通常重要になります。はい、実際にはこれは通常行われません。そしてもしそうなら、アナライザーの実装の段階でのみ。同時に、膨大な数の操作をすぐに理解する人はいません（アナライザーがこのモードで起動すると、かなりの数の警告が標準になります）が、メッセージ抑制メカニズムと特別なデータベース（質量抑制）に格納することで、それらを技術的な負債に入れます。研究目的で1回限りの査察を行っております。したがって、研究のための大きなプロジェクトは常に小さなプロジェクトよりも望ましいです。

ただし、Azure SDK for .NETプロジェクトはテストベンチとしてすぐに失敗を示しました。その印象的なサイズでさえ役に立たず、むしろ作業を複雑にさえしました。その理由は、次のプロジェクト統計に示されています。

  • ソースファイル.cs（テストを含まない）：16,500
  • Visual Studioソリューション（.sln）：163
  • 空でないコード行：3,462,000
  • 自動生成されたもの：約330万
  • プロジェクトリポジトリはGitHubで入手できます。

コードの約95％は自動的に生成され、このコードの大部分は何度も繰り返されます。静的アナライザーを使用してそのようなプロジェクトをチェックすることは、多くの作業があるため、通常は時間がかかり、役に立たないが、一見すると非論理的で冗長なコードです。これにより、多数の誤検知が発生します。

多数のVisual Studioソリューション（163）がケーキのチェリーとして機能したため、この大量のコードは「不鮮明」でした。残りのコード（自動生成されたものではない）を確認するために、いくつかの努力をしなければなりませんでした。これにより、自動生成されたすべてのコードが、相対パス「<Solution folder> \ src \ Generated」に沿ってソリューションサブフォルダーに配置されました。また、そのようなコードの各.csファイルには、<auto-generated>タグに特別なコメントが含まれています：

// <auto-generated>
// Copyright (c) Microsoft Corporation. All rights reserved.
// Licensed under the MIT License. See License.txt in the project root for
// license information.
//
// Code generated by Microsoft (R) AutoRest Code Generator.
// Changes may cause incorrect behavior and will be lost if the code is
// regenerated.
// </auto-generated>

実験の純度については、ランダムに選択された約10のソリューションを自動生成コードでランダムにチェックしました。結果は低くなります。

そのため、残っている「正直な」コードは少量ですが、それでも一定の数のエラーを見つけることができました。今回は、PVS-Studio診断の番号順にトリップしません。代わりに、検出されたソリューションごとに応答をグループ化します。

Azure SDK for .NETコードで見つけたものを見てみましょう。

結論


ご覧のとおり、プロジェクトのサイズが大きいからといって、必ずしも多数のエラーが保証されるわけではありません。しかし、リラックスする必要はありません。いつでも何かを見つけることができます。Azure SDK for .NETのような複雑なプロジェクトでもです。はい、これには追加の作業が必要ですが、結果はより快適になります。また、過度の労力をかける必要がないように、静的分析を使用することをお勧めします。また、新しいコードを作成するときは、開発者の職場でも使用することをお勧めします。これが最も効果的なアプローチです。PVS-Studioをダウンロードして実際に試してみてください。バグとの戦いで頑張ってください！
