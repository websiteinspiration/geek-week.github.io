私の人生のIoT。スマートオフィスを作成し、IoTを利用してアパートのジャガイモの成長を追跡する方法

ハブロフキアン、あなたに挨拶します！

2020年、私たちは皆、モノのインターネットとは何か、そしてなぜそれが必要なのかを知っています。しかし、IoTの最も重要なレイヤーの1つを表すクラウドプラットフォームに精通している人はどれくらいいますか？それを正しくしましょう。

プロトコルの異種性がスマートデバイスの接続プロセス、その構成、およびデータ処理を大幅に複雑にすることは秘密ではありません。同様の問題は、モノのインターネットのクラウドを通じて対処されています。今日、ロシアのモノのインターネットプラットフォームの1つを例として使用して、さまざまなプロトコルでデバイスを接続すること、および自動化プロセスを構築するために受け取った情報を使用することがいかに簡単かを示します。

私が自分のタスクに通常使用するプラットフォームでは、MQTT、Wialon Combine、Wialon IPS、Galileosky、Modbusなどのプロトコルを使用して動作するデバイスとの相互作用をすでに実装しています。

提示されたプロトコルを使用することに加えて、インターネットにアクセスできないデバイスの場合、ソフトウェアエージェントを作成することが可能です。これは、機器と、別のデバイス（Raspberry Piなど）にインストールされ、この機器に接続されているプラ​​ットフォームとの間の仲介者です。

提示されたプロトコルのいずれかで動作するデバイスとの対話を確実にしたいとします。この場合、3つのステップを完了するだけで十分です。

  1. 必要なパラメーターとコマンドを使用してモデルを構成します。
  2. プラットフォームで一意の識別子を持つオブジェクトを作成します。
  3. プラットフォームに接続するようにデバイスを構成します。

いくつかのケースを分析して、すべてがどのように関連しているかを見てみましょう。

ケースNo. 1アジャイルドラ


まず、ある日、私たちのチームはオフィスでの作業プロセスを自動化する方法について真剣に考えました。

そのため、アジャイルのコンセプトに従って、正午にはすべての従業員が毎日の会議に集まります。 Slackで開催される会議についての通知を見逃しがちで、時計に気を取られるのはあまり便利ではありません。そのため、自動音声通知システムであるAgile-gongを作成するというアイデアが生まれました。

どのように実装されていますか？ Ironは、NodeMCU（Wi-Fiモジュールが組み込まれた小型のArduinoアナログ）、サーボドライブ、およびコンデンサーです。平日の12時に、最後に衝撃装置を備えたサーボドライブの出力シャフトが、ゴングが鳴ってすべての上昇を通知するのに十分な角度で回転していることを確認する必要があります。

鉄の接続図は非常に簡単です：



NodeMCUに配線されたコードは以下を提供します：

  1. Wi-Fi接続のインストールとMQTTプロトコルを使用したプラットフォームへの接続
  2. サーボの初期位置を0度に設定します。
  3. 現在の状況に関するデータを含むメッセージの公開。
  4. コマンドをサブスクライブし、コマンドでサーボ角度を回転させます。

#include "Arduino.h"
#include "EspMQTTClient.h" /* https://github.com/plapointe6/EspMQTTClient */
// Servo library
#include <Servo.h>

// Object Servo with name myservo
Servo myservo;
int pos;

EspMQTTClient client(
  "<wifi-ssid>",
  "<wifi-password>",

  "<MQTT Broker server ip>",
  "<ric-mqtt-client-id>"
  );


void setup() {
  Serial.begin(9600);
  move(0);
}

void onConnectionEstablished() {
  Serial.println("connected");
  client.subscribe("move", [] (const String& payload)  {
	int angle = payload.toInt();
	if (angle != pos) {
  	move(angle);
	}
	client.publish("position", payload);
  });
}

void loop() {
  client.loop();
}

void move(const int angle)
{
	myservo.attach(5);
	myservo.write(angle);
	delay(800);
	myservo.detach();
	pos = angle;
}

プラットフォーム側では、デバイスモデルが開発されています。デバイスから受信できるパラメータと、デバイスに送信できるコマンドについて説明します。MQTTコマンドの解釈では、これらは特定のトピックとデータを持つクライアントへのメッセージです。この場合、データには必要な回転角度が含まれています。



次に、プラットフォーム上で認証が行われる識別子を使用してオブジェクトが作成されました。接続後の表示は次のようになります。



チームでは、0度と90度の角度で回転コマンドを送信するオプションがあります。



次に、自動化スクリプトを追加する必要があります。適切なタイミングになると90度回転状態になる自動機械を作成します。次に、構成可能な繰り返し回数のサイクルで、必要なストローク数を作成し、12時間の初期待機状態に戻ります。

各自動化シナリオは、オブジェクトの動作のロジックを定義する特定のブロック図です。このようなシナリオを作成することで、デバイスで発生したすべての変更を考慮に入れることができ、発生した変更に基づいて、デバイスはユーザーにコマンドを送信せずに適切なアクションを自動的に実行できます。
結果として得られるマシンは、特定のデバイスだけでなく使用することもできます。

たとえば、ゴングを使用してまったく同じシステムを作成し、オフィスの別のオフィスにインストールできます。次に、同じモデル、2つの異なるオブジェクト、および2つのオブジェクトで実行される1台のマシンを使用します。



ケースNo. 2二酸化炭素センサー


私たちにとって2番目に有用な解決策は、二酸化炭素センサーを接続することでした。 MQTT経由でも接続されます。繰り返しになりますが、鉄の組み立て方式は簡単です。



ええ、ところで、私たちは社内のハッカソンで1番目と2番目の両方のケースの実装に従事していました。そして私たちの誰もが鉄の仕事に没頭していなかったし、これの必要もありませんでした。



さらに、手順は同じです。モデルには、デバイスが送信するppmパラメーター（1000 ppm =2コンテンツの0.1％）が含まれていますが、それほど明確ではないため、別のパラメーター（CO2コンテンツのパーセンテージ）がモデルにすぐに表示されました。 ppmを10000で割って計算されます。



ここでは、電球をオンにする2つのコマンドにも注意してください。彼らはそれを表示に使用することにしました。もちろん、プラットフォームマシンから管理します。デバイスを接続すると、パラメータの表示は次のようになります。これらの値は受け入れられ、リアルタイムで表示されますが、履歴で過去に蓄積されたパケットを表示したり、特定の期間のパラメーターの変化のグラフを表示したりすることもできます。



このオブジェクトのオートマトンは次のように機能します。上の状態では、ライトは消えます。下部-タイマーを1分間開始し、電球をオンにします。最初の状態から2番目の状態への遷移は、ppm値が600ユニットを超える条件でデバイスからデータを受信するイベントによって発生します。タイマーがトリガーされると、戻り（2番目の状態から最初の状態への遷移）が発生します。



2つの質問があるかもしれません。

  1. なぜ自動ですか？そのような条件をハードウェア自体に登録する方が簡単ではないですか？結局のところ、ここではすべてがとても簡単です。
  2. なぜタイマーがあるのですか？

実際、このマシンの利点は、このような単純なケースにもあります。このセンサーを電球と一緒に机の上に置いてデバッグしましたが、マシンのしきい値が非常に低いため、仕事に来るたびにライトが点灯しました。しばらくの間、私はマシンで異なる値を試しました、そしてその結果、私は600ユニットの最適値に到達しました。目的の値を選択するには、マシンの値を変更して保存するだけです。デバイスの点滅なし。そして、最高の空気状態を維持する必要があり、頻繁な換気が必要なオフィスにこのデバイスを転送すると、値を再び簡単に変更できます。速くて便利。

こちらが1分のタイマーです。これは、しばらくの間高いCO2の状態にあり、しばらくの間高い値が続き続けるという事実に反応しないようにするために必要です。それ以外の場合は、常に電球を点滅させ、空気状態が正常化するまで移行します。別の方法で初期状態に遷移することが可能であることは、すでに推測できます。また、データを受信する場合でも、反対の条件が当てはまる場合-ppm <600。その後、通常の値になるまで、2番目の状態になります。

ケースNo. 3 ACS


最も複雑な例は、建物へのアクセスを制御し、通過時間とイベントを考慮に入れるように設計された電子モジュールである、アクセス制御と管理システムの操作の説明です。



コントローラーは、ウィーガンド出力インターフェースを備えたリーダーからの情報を処理し、組み込みのリレーを使用して、アクチュエーター（電磁ロック）を交換します。インターネット接続も、プラットフォームへの可視接続もありません。ただし、制御コンピューターとデータを交換するための独自のプロトコルを備えているため、コントローラーからの読み取り、コントローラーへの書き込み、ロックの開閉などのコマンドをコントローラーに送信できます。したがって、このケースでは、非標準的なアプローチが編成されました。これは、記事の冒頭で述べたエージェントの使用です。

コントローラプロトコルでの作業はC ++コードで実装され、Raspberry Piで実行するために起動されました。RaspberryPiは、インターフェースコンバータを介してRS-485を介してコントローラに接続されました。プログラムの主なタスクは、プラットフォームに接続し、コマンドをシリアル化し、コントローラーから受信したデータを非シリアル化することです。したがって、小さなソフトウェアレイヤーを使用してデバイスを「スマート」にすることができました。

  • イベント時間、
  • イベントコード
  • 従業員カード番号。

モデルは、さまざまなパラメーターのJSONフィールドを解析するためにも使用されます。

これは、コマンドを送信するためのインターフェースです。



イベントバッファを読み取るだけでなく、新しい境界を書き込むコマンドもあることに気付くでしょう。バッファ境界は、コントローラメモリに保存されます-最初と最後。読み取りコマンドがデバイスに到着すると、これらの境界が読み取られ、これらの制限内で、イベントバッファーからの読み取りが行われます。新しいイベントが受信されると、バッファー境界の終わりがコントローラー上で自動的にシフトされます。ただし、同じデータを再度読み取らないように、バッファの最初の境界を書き換える必要があります（最後の読み取り後の最後の境界を示します）。ただし、これは、イベントデータがプラットフォームに正常に送信された後でのみ行う必要があります。データの受信をロックしてから、マシンの境界を書き換えるコマンドを送信すると便利です。



このプロジェクトの継続は、社内CRMシステムとの統合にありました。このシステムでは、従業員に関する情報は、オフィスにいるかいないかに関する情報に常に表示されます。また、オフィスへの入退室時間を表示し、月間総時間を考慮しています。





データは、RESTful APIを使用してプラットフォームから取得されます。プラットフォームAPIは、Webポータル、モバイルアプリケーション、Webアプリケーションなどの外部システム、または私たちの場合のようにCRMシステムで、プラットフォームエンティティとそのデータを操作、対話、使用する機能を提供します。

来客・出前など、ドアを開ける必要のある方が来られた場合もございます。カードを使用しないようにして、ステータスに関する誤った測定値を送信しないようにするには、プラットフォームの「ロック解除」ボタンを使用できます。また、ドアで出会う必要がある場合は、モバイルアプリケーションから同じことを行うと便利です。
