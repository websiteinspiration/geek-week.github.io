プロセスを別のコンピューターにテレポートします！ 

かつて、同僚が分散コンピューティングクラスターのAPIについて考えを共有し、冗談めかして答えました。「明らかに、理想的なAPIは単純な呼び出しtelefork()であり、プロセスが各クラスターマシンで起動し、インスタンスIDの値を返します。」しかし、結局のところ、このアイデアは私の手に渡った。なぜそれがこんなに愚かで単純で、リモート作業用のどのAPIよりもはるかに単純であるのか、なぜコンピューターシステムがそのように機能していないように見えるのか理解できませんでした。私はこれがどのように実装されるかを理解しているようで、私はすでに良い名前を持っています。これはどのプロジェクトでも最も難しい部分です。だから私は働かなければなりませんでした。

最初の週末に彼は基本的なプロトタイプを作り、2番目の週末にはクラウド内の巨大な仮想マシンに処理し、複数のコアでのパストレースのレンダリングを無効にしてから、プロセスをテレフォークします。これらはすべて単純なAPIでラップされています。

ビデオは、クラウド内の64コアVMでのレンダリングが8秒で完了することを示しています（テレフォークごとに前後に6秒）。私のラップトップのコンテナーでローカルに同じレンダリングを行うと、40秒かかります。


どのようにしてプロセスをテレポートさせることができますか？この記事で説明する内容は次のとおりです。基本的な考え方は、低レベルでは、Linuxプロセスにはいくつかのコンポーネントしかないということです。ドナーからそれぞれを復元し、ネットワーク経由で転送して、クローンプロセスにコピーする方法が必要です。

「しかし、どのように[TCP接続などの難しいもの]を複製するのですか？」本当に。実際、コードをシンプルに保つためにこのような複雑なことを容認していません。つまり、本番環境で使用すべきではない楽しいテクニカルデモにすぎません。しかし、彼女は依然として、主に計算タスクの幅広いクラスをテレポートする方法を知っています！

それはどのように見えますか


コードはRustライブラリの形式で実装しましたが、理論的には、プログラムをC APIでラップし、FFIバインディングを実行して、Pythonプロセスでさえテレポートすることができます。実装は、わずか約500行のコード（および200行のコメント）です。

use telefork::{telefork, TeleforkLocation};

fn main() {
    let args: Vec<String> = std::env::args().collect();
    let destination = args.get(1).expect("expected arg: address of teleserver");

    let mut stream = std::net::TcpStream::connect(destination).unwrap();
    match telefork(&mut stream).unwrap() {
        TeleforkLocation::Child(val) => {
            println!("I teleported to another computer and was passed {}!", val);
        }
        TeleforkLocation::Parent => println!("Done sending!"),
    };
}

またyoyo、サーバーにテレフォークと呼ばれるヘルパーを作成し、送信されたクロージャを実行してから、テレフォークで戻します。これにより、たとえば処理能力が大幅に向上したリモートサーバーでコードを簡単に実行できるような錯覚が生じます。

// load the scene locally, this might require loading local scene files to memory
let scene = create_scene();
let mut backbuffer = vec![Vec3::new(0.0, 0.0, 0.0); width * height];
telefork::yoyo(destination, || {
  // do a big ray tracing job on the remote server with many cores!
  render_scene(&scene, width, height, &mut backbuffer);
});
// write out the result to the local file system
save_png_file(width, height, &backbuffer);

プロセスをテレフォークする方法


別のマシンでプロセスを再作成することを考えたのは私が最初ではありませんでした。したがって、rrデバッグおよび記録デバッガは非常によく似ています。このプログラムの作成者@rocallahanにいくつか質問を送ったところ、ホスト間でコンテナを「ホット」に移行するためのCRIUシステムについて教えてくれました。 CRIUはLinuxプロセスを別のシステムに転送でき、あらゆる種類のファイル記述子やその他の状態の回復をサポートしますが、コードは非常に複雑で、特別なカーネルアセンブリとルート権限を必要とする多くのシステムコールを使用します。 CRIU wikiページからのリンクを使用して、スーパーコンピューター上の分散タスクのスナップショット用に作成されたDMTCPを見つけ、後で再起動できるようにしました。このプログラムコードはよりシンプルであることがわかりました。

これらの例は、特別なランナーとインフラストラクチャを必要とする非常に複雑なプログラムであり、プロセスの可能な限り単純なテレポーテーションをライブラリ呼び出しとして実装したかったため、自分のシステムを実装する試みを放棄することを強制しませんでした。そこで、ソースコードの断片rr、CRIU、DMTCP、およびいくつかのptraceの例を調べ、独自の手順をまとめましたtelefork。私の方法は独自の方法で機能します。それはさまざまなテクニックの寄せ集めです。

これは私の大好きなテクニックの1つです。あまり知られていない抽象化レイヤーに飛び込みます。計算をテレポートすることは不可能または非常に困難に見えるかもしれません。状態全体をシリアル化する、バイナリ実行可能ファイルをリモートマシンにコピーする、状態をリロードするために特別なコマンドラインフラグを使用してそこで起動するなどの方法が必要になると思うかもしれません。しかし、いいえ、すべてがはるかに簡単です。好きなプログラミング言語の下に抽象化レイヤーがあり、かなり単純な関数のサブセットを選択できます。週末にかけて、500行のコードで任意のプログラミング言語で最も純粋な計算のテレポーテーションを実装します。おもうそのような別の抽象化レベルへの飛び込みは、多くの場合、より単純でより普遍的なソリューションにつながります。このような私のプロジェクトのもう一つはNumderline。

一見すると、そのようなプロジェクトは極端なハックのようであり、大部分はそうです。彼らは誰も期待していないようなことをし、壊れると抽象化のレベルでそれを行います。たとえば、同様のプログラムは機能しないはずです。たとえば、ファイル記述子が不思議な形で消えてしまいます。しかし、抽象化のレベルを正しく設定し、可能な状況をエンコードして、最終的にすべてがスムーズかつ魔法のように機能するようにすることもできます。ここでの良い例は、rr（テレフォークがそれを解約することはできましたが）とリアルタイムでの仮想マシンのクラウド移行（実際、ハイパーバイザーレベルでのテレフォーク）です。

また、これらのことを、コンピュータシステムを動作させる別の方法のアイデアとして提示したいです。クラスターコンピューティングAPIが、関数をクラスターに変換する単純なプログラムよりもはるかに複雑なのはなぜですか？ネットワークシステムのプログラミングがマルチスレッドよりもはるかに複雑なのはなぜですか？もちろん、あらゆる種類の正当な理由を示すことができますが、それらは通常、既存のシステムの例を作ることがどれほど難しいかに基づいています。それとも、適切な抽象化または十分な努力で、すべてが簡単かつシームレスに機能するでしょうか？基本的に、不可能はありません。
