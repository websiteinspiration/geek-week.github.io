NestJSでのRabbitMQの巧妙な作業

金融システムを開発する場合、AMQPプロトコルのほとんどの実装で完了したタスクを処理するための標準的なメカニズムが常に適切であるとは限りません。ある時点で、私たちはそのような問題に遭遇しましたが、最初に最初にしたことです。

NestJSのRabbitMQの標準実装により、装飾された関数でメッセージを簡単に受信できます。

@MessagePattern('notifications')
getNotifications(@Payload() data: number[], @Ctx() context: RmqContext) {
  console.log(context.getMessage());
}

これがどのように機能するかについての詳細はここで説明されています。
また、Nestでのキューの操作については、Habréに関するこの記事で詳しく説明されています。

他に何かが必要かもしれないようです。ただし、現在の実装にはいくつかの欠点があります。

問題1


ackの結果をチャネルに送信するには、チャネルをRmqContextコンテキストから手動で引き出し、信号をチャネルに送信する必要があります。

@MessagePattern('notifications')
getNotifications(@Payload() data: number[], @Ctx() context: RmqContext) {
  const channel = context.getChannelRef();
  const originalMsg = context.getMessage();
  channel.ack(originalMsg);
}

決定


関数から直接実行結果を通常のオブジェクトであるPromiseまたはObservableとして返すことにより、httpハンドラーでの作業に使用されたのと同じパターンを使用します。

問題2


結果を個別のキューで送信する必要がある場合は、Rabbitを使用する各コントローラーでその名前を知る必要があります。つまり、モジュールの初期化中にこれを1か所で構成し、暗黙的に使用する方法はありません。

決定


モジュール構成の段階で、操作の成功結果とエラーのキューを指定します。1つのキューで成功した操作の結果を送信し、別のキューで誤った操作の結果を送信することにしました。問題1の解決策を使用すると、戻り値PromiseまたはObservableが成功した場合、結果は成功した操作キューに送信され、失敗した場合、メッセージは拒否されてエラーキューに入れられます。RabbitMQは、x-deadオプションを使用してこれを簡単に実行できます。 -letter-exchangeおよびx-dead-letter-routing-keyは、キューの作成時に使用します。

問題3


ライブラリユーザーとして、開発者は次のメッセージのIDを取得したり、ACKとは何か、いつ呼び出すかなどを理解したりするために、AMQPプロトコルの詳細を知る必要があります。

決定


メッセージのIDを取得するデコレーターを追加します。ackの代わりに、ハンドラー関数から実行の結果を返します。

問題4


おそらく最も重要な問題は、ハンドラーにメッセージを複数回配信することです。金融取引に関しては、これは非常に重要なポイントです。これは、送金が既に行われていて、データベースに書き込んだり、メッセージブローカーに確認メッセージを送信したりするときに、最後のステップで操作が失敗する可能性があるためです。明白な解決策の1つは、コンシューマーがメッセージを受信したときに、プロデューサーが生成したメッセージIDをデータベースに書き込み、メッセージがまだない場合は処理する前に、メッセージがある場合はメッセージを拒否することです。ただし、AMQPプロトコルは、このメッセージが他のクライアントに配信されたかどうかを識別する再配信フラグを提供します。これを使用して、再配信されたメッセージを検出し、エラーのあるキューに送信できます。Nestの現在の実装では、そのようなメッセージを配信しない方法はありません。

決定


このメッセージをハンドラーに配信せず、ドライバーからメッセージを受信する段階でエラーをログに記録します。もちろん、この動作は、メソッドを装飾する段階で構成して、このタイプのアクションのメッセージを引き続き受信することを明示的に示すことができます。

上記の問題をすべて解決するために、独自のプロトコル実装が作成されました。初期化は次のようになります。

const amqp = await NestFactory.create(
 RabbitModule.forRoot({
   host: process.env.AMQP_QUEUE_HOST,
   port: parseInt(process.env.AMQP_QUEUE_PORT, 10),
   login: process.env.AMQP_QUEUE_LOGIN,
   password: process.env.AMQP_QUEUE_PASSWORD,
   tasksQueueNormal: process.env.AMQP_QUEUE_COMMAND_REQUEST,
   tasksQueueRedelivery: process.env.AMQP_QUEUE_REQUEST_ONCE_DELIVERY,
   deadLetterRoutingKey: process.env.AMQP_QUEUE_COMMAND_REQUEST_DEAD_LETTER,
   deadLetterRoutingKeyRedelivery: process.env.AMQP_QUEUE_COMMAND_REQUEST_ONCE_DELEVERY_DEAD_LETTER,
   exchange: process.env.AMQP_EXCHANGE_COMMAND,
   prefetch: parseInt(process.env.AMQP_QUEUE_PREFETCH, 10),
 }),
);
const transport = amqp.get<RabbitTransport>(RabbitTransport);
app.connectMicroservice({
 strategy: transport,
 options: {},
});

app.startAllMicroservices();

ここでは、メッセージ、結果、エラーを配信するためのキューの名前と、再配信の影響を受けない個々のキューを示します。

コントローラレベルでの作業は、httpでの作業と同様です。

@AMQP(‘say_hey’)
sayHay(@AMQPRequest requestId: string, @AMQPParam q: HeyMessage): Observable<Result> {
 return this.heyService.greet(requestId, q);
}

タスクの結果は、このObservableが実行されるとすぐに結果キューに入ります。@AMQPRequestで装飾されたパラメーターは、プロトコルのcorrelationIdフィールドに対応します。これは、配信されたメッセージの一意の識別子です。

@AMQPParamパラメータは、メッセージ本文自体と一致します。JSONの場合、メッセージはすでにオブジェクトに変換された関数に到着します。単純型の場合、メッセージはそのまま送信されます。

次のメッセージが配信不能レターに含まれます。

@AMQP(‘say_hey’)
sayHayErr(@AMQPRequest requestId: string, @AMQPParam q: HeyMessage): Observable<Result> {
 return throwError(‘Buy’);
}

調子はどう


タイプリフレクションをAMQPParamに追加して、メッセージ本文が渡されるクラスに変換されるようにします。今はタイプするだけのカーストです。
