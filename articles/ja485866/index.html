Zimbra Open-Source Editionとエンタープライズポータルの統合

企業ポータルの作成は、多数の情報システムを持つ企業の間で一般的な慣行です。作業に必要なすべてのリソースへの単一のアクセスポイントがあれば、何十ものブックマークで混乱する必要がなくなり、いくつかのパスワードを覚えておく必要がなくなります。つまり、会社の従業員はできるだけ作業に集中できます。もちろん、この場合の重要な情報システムの1つは、企業のコミュニケーションと電子メールシステムです。これは、企業の実質的にすべての従業員が何らかの方法で使用するためです。

Zimbra Collaboration Suite Open-Source Editionは、電子メールだけでなく、デジタルカレンダー、タスクリスト、アドレス帳へのアクセスを提供し、一連の拡張機能Zextras Suite Proの助けを借りて、ドキュメントの共同編集のために企業向けのデジタルワークステーションですテキストおよびビデオチャットと個人用ストレージドライブ。さらに、Zimbra OSEは企業ポータルに簡単に統合できます。例として、company.ruドメインを使用してこれを行う方法を見てみましょう。



Zimbra OSEをエンタープライズポータルと統合するための2つのインターフェイスがあります。 1つはURLインターフェースで、特別なリンクを作成することで構成されています。これに基づいてバナーを作成できます。クリックすると、企業ポータルで承認されたユーザーがZimbra OSEアカウントにリダイレクトされ、タスクを安全に実行できます。 2番目は、SOAPインターフェースを使用した事前許可に基づいています。最初のインターフェースは最も一般的であり、ほとんどのWebポータルとの統合が容易であるために使用されます。 2番目のインターフェースは、通常、認証トークンの作成とユーザーのリダイレクトの制御を必要とするポータルで使用されます。

どちらの場合も、最初に行う必要があるのは、メールドメインを準備することです。これを行うには、事前認証キーを作成します。これは、zmprov generateDomainPreAuthKey company.ruコマンドを使用して行われます。このコマンドの出力には64文字のキーが含まれます。これは、32文字をランダムに生成して暗号化することによって取得されます。たとえば、preAuthKey：9b34da63e5c1cba4cf7eb8262bacb18f712f6abafb02cf670234cb9bca63cb31のようになります。このキーは、その後zmprov gd company.ru zimbraPreAuthKeyコマンドを使用して再表示できます。その後、変更を有効にするためにメールサーバーを再起動する必要があります。

URLインターフェースを使用して事前認証を実装する方法を理解しましょう。次に、この事前認証キーに基づいて、企業ポータルですでに認証されているユーザーがパスワードを入力しなくてもZimbra OSEにログインできる特別なURLを作成します。URLは、mail.company.ru/service/preauth?{account-identifier{{by-value►{time{{expires►►&admin=1{{computed-preauth}のようになります。

このURLのパラメーターは次のとおりです。

  • {account-identifier}と{by-value}は、アカウントを識別するために必要です。最初のパラメーターはその識別子、およびこの識別子の2番目のタイプを示します。使用される{by-value}のタイプが指定されていない場合、名前による認証が使用されます。
  • {time}はUnixで変換されたタイムスタンプです
  • {expires}は、認証トークンが期限切れになるまでのミリ秒単位の時間です。このパラメーターの値として0が指定されている場合、デフォルトのアカウントのトークンの有効期限が使用されます
  • [＆admin = 1]このパラメーターは、事前認証を使用してWebクライアントだけでなく管理コンソールにもログインする場合にのみ必要です。このパラメーターが指定されている場合、HMACを計算するときに、{account-identifier}パラメーターと{by-value}パラメーターの間に入力します。
  • {computed-preauth}計算された事前認証値。

{computed-preauth}の計算は次のとおりです。

  1. 値{account-identifier}、{by-value}、{time}、{expires} をこの順序でまとめる必要があります
  2. それらを「|」で区切って並べます
  3. DomainPreAuthKeyを使用してHMACを計算する
  4. HMAC値を16進数に変換する

たとえば、次のデータがあるとします。

  • DomainPreAuthKey：9b34da63e5c1cba4cf7eb8262bacb18f712f6abafb02cf670234cb9bca63cb31
  • {アカウント識別子}：manager@company.ru
  • {値ごと}：名前
  • {有効期限}：0
  • {時間}：1579862639

次の一連の情報manager@company.ru|name|0|1579862639を作成できます。次に、この行からSHA-1 HMACを計算できます。preauth = hmac（ "manager@company.ru|0|1579862639"、 "9b34da63e5c1cba4cf7eb8262bacb18f712f6abafb02cf670234cb9bca63cb31"） ;その結果、値{computed-preauth} 93a43cd03a5b2fca40f46c3ce99b4a4dd6e5676aが取得されます。

これを使用して、URLをコンパイルします：mail.company.ru/service/preauth?account=manager@company.ru&expires=0×tamp=1579862639&preauth=93a43cd03a5b2fca40f46c3ce99b4a4dd6e5676a。このURLをクリックすると、Zimbra Webクライアントに入り、アカウント名やパスワードを入力しなくても認証トークンを受け取ることができます。

事前認証を実行する場合は、データ「admin@company.ru|1|name|0|1579862639」を使用してHMACを計算し、URL mail.company.ru/service/preauth?account=admin@companyを使用して次のように入力します.ru＆expires = 0＆timestamp = 1579862639＆preauth = 7b410ed71e4eb6c7724598587ae03078bca5e29f＆admin = 1

SOAPインターフェースの場合、次のパターンを使用して事前認証データが送信される標準のAuthRequestリクエストが使用されます。

{アカウント識別子}
{計算済み事前認証}

企業ポータルにZimbra事前認証を実装するシステム管理者のもう1つのタスクは、{computed-preauth}値の計算を自動化することです。これは、必要なすべてのデータを自動的に置き換え、出力で事前認証の準備ができたリンクを提供するスクリプトを使用して実現できます。そのようなスクリプトの例をいくつか示します

。Python/ Django

def webmail_redirect(request):
    from time import time
    import hmac, hashlib
    preauth_key = "9b34da63e5c1cba4cf7eb8262bacb18f712f6abafb02cf670234cb9bca63cb31"
    preauth_url = "http://mail.company.ru/service/preauth"
    timestamp = int(time()*1000)
    try:
        acct = request.user.email
        pak = hmac.new(preauth_key, '%s|name|0|%s'%(acct, timestamp), hashlib.sha1).hexdigest()
        return HttpResponseRedirect("%s?account=%s&expires=0&timestamp=%s&preauth=%s"%(preauth_url, acct, timestamp, pak))
    except:
        pass
    return HttpResponseRedirect("/not_logged_in/")

PHP

<?php
    /**
    *  .     config.inc.php     .
    */
    $PREAUTH_KEY="9b34da63e5c1cba4cf7eb8262bacb18f712f6abafb02cf670234cb9bca63cb31";
    $WEB_MAIL_PREAUTH_URL="http://mail.company.ru/service/preauth";
    /**
    *      .        GET.
    */
    $user = $_GET["user"];
    $domain=$_GET["domain"];
   
    $email = "{$user}@{$domain}";
    if(empty($PREAUTH_KEY)) {
        die("Need preauth key for domain ".$domain);
    }
   
    /**
    *    URL  
    */
    $timestamp=time()*1000;
    $preauthToken=hash_hmac("sha1",$email."|name|0|".$timestamp,$PREAUTH_KEY);
    $preauthURL = $WEB_MAIL_PREAUTH_URL."?account=".$email."&by=name&timestamp=".$timestamp."&expires=0&preauth=".$preauthToken;
   
    /**
     *   URL   Zimbra
     */
    header("Location: $preauthURL");
?>
