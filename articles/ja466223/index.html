クリーンな分解

この記事では、Clean Architectureを使用するときに、タスクをサブタスクに分割する方法を検討します。

分解の問題は、NullGravityモバイル開発チームが遭遇したものであり、その解決方法と最終的に何が起こったのかを以下に示します。




バックグラウンド


2018年の秋、通信事業者向けの次のアプリケーションを開発していました。しかし、今回は違いました。タイミングはかなりタイトで、クライアントのマーケティングキャンペーンに結びついていました。Androidチームは3人から6-7人の開発者に成長しました。スプリントではいくつかのタスクが行われ、問題はそれらを効果的に分解する方法でした。

効果的に話すとはどういう意味ですか。

  1. 並列タスクの最大数。
    これにより、使用可能なすべてのリソースを占有することができます。
  2. マージ要求のサイズを縮小します。
    それらはショーのために監視されません、そしてあなたはまだコードレビューの段階で潜在的な問題を見つけることができます。
  3. マージの競合の量を減らします。
    タスクの流れが速くなり、開発者を競合解決に切り替える必要がありません。
  4. 時間の支出の統計を収集する機会。
  5. Jiraでタスク作成を自動化します。


どのようにして問題を解決しましたか？


すべてのサブタスクを次のタイプに分類します。

  • データ
  • ドメイン
  • 空の
  • UI
  • 項目
  • カスタム
  • 統合


データとドメインは、クリーンアーキテクチャのレイヤーに対応しています。
空、UI、アイテム、カスタムはプレゼンテーションレイヤーを参照します。
統合は、ドメイン層とプレゼンテーション層の両方に適用されます。


図1. Clean Architectureレイヤーに関連するタスクの場所

各タイプを個別に見てみましょう。

データ


DTO、API、データベースの操作、データソースなどの説明

ドメイン


リポジトリインターフェース、ビジネスモデルの説明、インタラクター。

データレイヤーのリポジトリインターフェースも実装されています。
一見したところ、このようなやや非論理的な分離により、データやドメインなどのタスクを可能な限り分離することが可能になりました。

UI


基本的な画面レイアウトと、必要に応じて追加の状態を作成します。

項目


画面が要素のリストである場合、タイプごとにモデルを作成する必要があります-アイテム。アイテムをレイアウトにマッピングするには、AdapterDelegateが必要です。デリゲートアダプターの概念を使用しますが、いくつかの変更が加えられています。
次に、PresentationModelでリストアイテムを操作する例を作成します。

空の


UIやアイテムなどのタスクに必要な基本クラス：PresentationModel、Framgent、レイアウト、DIモジュール、AdapterDelagateファクトリ。バインディングインターフェイスと実装。画面エントリポイントを作成します。

タスクの結果はアプリケーション画面です。ツールバー、RecyclerView、ProgressViewなどが含まれます。つまり、共通のインターフェース要素であり、その追加は別の開発者によって複製される可能性があり、避けられないマージの競合につながります。

カスタム


非標準のUIコンポーネントの実装。

新しいコンポーネントの開発をUIなどのタスクから分離するには、追加のタイプが必要です。

統合


ドメインとプレゼンテーション層の統合。

原則として、これは最も時間がかかるタスクの1つです。2つのレイヤーを減らして、前のステージでは見逃していた可能性があるポイントを調整する必要があります。

タスクの順序


データ、空、カスタムなどのタスクは、スプリントの開始直後に開始できます。それらは他のタスクから独立しています。

ドメインタスクは、データタスクの後に実行されます。

空のタスクの後のUIタスクとアイテムタスク。

統合タスクは、前のすべてのタスクを完了する必要があるため、最後に完了する必要があります。


図2.タスクを完了するためのタイムライン

一部のタスクは他のタスクによってブロックされているという事実にもかかわらず、それらは同時にまたは少し遅れて開始できます。このようなタスクには、ドメイン、UI、およびアイテムが含まれます。したがって、開発プロセスが加速されます。


図3.ロックのあるタスクを実行するためのタイムライン。

特定の機能ごとに、一連のタスクが異なる場合があります。
空、UI、アイテム、統合のタスクの数は異なる場合があり、一部のタイプは単に存在しない場合があります。

プロセスの自動化と統計収集


タスクの作成時に統計を収集するために、ラベルが割り当てられています。将来このメカニズムにより、各タイプに費やされた時間を分析し、平均コストを形成できるようになります。収集した情報は、新しいプロジェクトを評価するときに適用できます。

自動化についても、解決策を見つけることができました。タスクは一般的なものであるため、Jiraでの説明が異なる理由。要約と説明のテンプレートを開発しました。最初は単なるjsonファイルであり、このファイルのPythonパーサーであり、タスクを生成するためにJira REST APIが接続されていました。

この形式では、スクリプトはほぼ1年続きました。今日では、PyQtとMVPアーキテクチャを使用してPythonで記述された本格的なデスクトップアプリケーションになっています。

多分MVPはオーバーヘッドでしたが、Tkinterの最初のバージョンがMacOSバージョン10.14.6をクラッシュさせ、すべてのチームがアプリケーションを使用できるとは限らなかった場合、PyQtのビューを半日で簡単に書き直して機能しました。ここでも、このような単純なタスクであっても、アーキテクチャアプローチを使用することには利点があると確信しました。JiraSubTaskCreatorのスクリーンショットを図4に示します。

調査結果


  1. 私たちは、タスクの相互依存を最小限に抑えたサブタスクに分解するアプローチを開発しました。
  2. タスクを説明するために生成されたテンプレート。
  3. 小さなマージ要求を受け取りました。これにより、コードを個別に注意深く確認および変更できます
  4. マージ要求との競合の数を減らしました。
  5. 各タイプのタスクに費やされた時間をより正確に評価および分析する機会を得ました。
  6. 日常業務の自動化された部分。
