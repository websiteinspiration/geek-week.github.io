Visual Studioの9つのクールな拡張ルール

Cool Visual Studio拡張機能には、他の拡張機能と区別するいくつかの主要な機能があります。彼らは見て、実際にはよく考えられており、機能的で信頼性があります。さらに、彼らは卓越したレベルまで必要なことを行い、Visual Studioの内部機能にネイティブに適合します。

優れた拡張機能を簡単に作成できるように、コミュニティと協力して簡単なチェックリストを作成しています。あなたが使用できるGitHub課題テンプレートさえあり  ます。この記事では、9つの優れた拡張ルールについて説明します。カットの下の詳細。



ルール


次のリストは順序が正しくありません。最良の結果を得るには、すべてのルールに従ってください。



ルール1：カバレッジのルールを順守する


Microsoft.VisualStudio.SDK.Analyzers NuGetパッケージをVSIXプロジェクトに追加します。これは、一般的なスレッドエラーの特定と解決に役立ちます。

ルール2：高品質のアイコンを追加する


すべての拡張機能には、関連付けられたアイコンが必要です。アイコンは、128×128ピクセルの解像度と96以上のDPIを持つ高品質の.pngファイルであることを確認してください。アイコンをVSIXプロジェクトに追加した後、アイコンとプレビューイメージとして.vsixmanifestファイルに登録します。Visual Studio Marketplaceはより大きなアイコンを使用しており、Visual Studioに表示されるときに動的に変化します。

ルール3：タイトルと説明


調査によると、ユーザーは短い説明的な名前とそれらに関する正確な情報を使用して、拡張機能をインストールすることが多いことがわかっています。名前が拡張機能の機能の本質を捉えていることを確認してください。.vsixmanifestファイルの説明は、拡張機能の動作に関する期待を作成する必要があります。全体として、拡張機能が解決する問題とその機能が重要であるものの簡単な説明。

ルール4：適切なマーケットプレイスの説明を書く


これは、拡張を成功させるために行う必要がある最も重要なことの1つです。適切な説明は次のとおりです。

  • 拡張機能によって追加されるもののスクリーンショット/ GIF
  • 詳細な機能の説明
  • 必要に応じて、詳細へのリンク

ルール5：ライセンスを示す


ライセンスは、Marketplace、VSIXインストーラー、およびExtensions Managerダイアログボックスに表示されます。ユーザーに期待を与えるために、常にライセンスを指定してください。適切なライセンスを見つけるには、choosealicense.comの使用を検討してください  。このルールの理由は、あいまいさを解決するためです。これは、多くのVisual Studioユーザーにとって重要です。

ルール6：プライバシーに関する通知を追加する


拡張機能がテレメトリなどのデータを収集する場合は、説明にこれに関するメモを追加します。

ルール7：可能な場合はKnownMonikerを使用する


Visual Studioには、KnownMonikersコレクションで使用できる何千ものアイコンが付属しています  。アイコンをボタンに追加するときは、チェックしてください。既存のKnownMonikersアイコンを使用できる場合があります。これは、Visual Studioユーザーが使い慣れたデザイン言語の一部であるためです。ここで完了ですKnownMonikersのリストは、あなたが使用することもできKnownMonikersエクスプローラの拡張機能をスクリプトに合ったものを見つけるために。

ルール8：ネイティブ拡張感を作り出す


Visual Studio自体が使用するのと同じパターンと設計原則に従います。これにより、拡張機能がユーザーにとって自然になります。また、ユーザーインターフェースの設計が不適切なために発生する注意散漫も軽減されます。すべてのボタン、メニュー、ツールバー、およびツールウィンドウは、ユーザーがそれらを使用するための適切なコンテキストにいる場合にのみ、デフォルトで表示されることを確認してください。いくつかのルールがあります：

  • 新しい最上位メニューを追加しないでください（ファイル、編集などの横）。
  • ボタン、メニュー、ツールバーは、それらが属していないコンテキストでは表示されません。
  • 自動ロードが必要な場合（ほとんどの場合不要  ）は、できるだけ早く実行してください。
  •  自動読み込みに頼るのではなく、VisibilityConstraintsを使用してコマンドの表示を切り替える

ルール9：正しいバージョン範囲を使用する


誰もが新しい拡張機能を使用できるように、Visual Studio 2010までのバージョンのVisual Studioをサポートしたくなるかもしれません。問題は、拡張機能がサポートする古いバージョンより後に導入されたAPIを使用できなくなることです。多くの場合、これらの新しいAPIは重要であり、拡張機能とVisual Studio自体の両方のパフォーマンスと信頼性の向上に役立ちます。

サポートするVisual Studioのバージョンを決定するためのガイドラインは次のとおりです。

  • Visual Studioの以前のバージョンと現在のバージョンのみをサポート-可能であれば、古いバージョンをサポートしない
  • 利用可能なバージョンの範囲を指定しないでください。例えば。[16.0、）。バージョンの詳細については、  こちらをご覧ください。
