.NET Core vs Node.js. Argumen dan Fakta

Artikel ini merupakan kelanjutan dari yang sebelumnya , yang menimbulkan banyak pertanyaan bagus dan menarik. Ketika saya mulai menulis jawaban atas komentar, saya menyadari bahwa saya belum banyak bercerita. Karena itu, saya memutuskan untuk menulis artikel baru yang lengkap, tetapi dari sudut yang berbeda: menambah pengalaman pribadi dan hubungan sebab akibat. Jika Anda belum membaca yang sebelumnya, maka tidak apa-apa. Anda mungkin tidak membacanya.

Ide utamanya adalah bahwa pada pertengahan 2017, saya memprakarsai perubahan dalam tumpukan pengembangan utama aplikasi backend yang dikembangkan oleh perusahaan kami dari .Net (C #) ke Node.js (naskah). Bagaimana ini bisa terjadi, apakah saya menyesali keputusan yang dibuat dan apakah saya tidak akan kembali? Tentang itu di bawah ini.

Latar Belakang


Pertama, cerita pendek tentang diri Anda. Saya terlahir sebagai programmer, bukan manajer. Dia membuat aplikasi pertamanya, yang tidak malu untuk dibicarakan, pada tahun 2003 di C ++ di IDE Visual Studio 6 sebagai seorang siswa. Itu adalah permainan Transformers menggunakan D3D, DirectSound, dan hampir semua modul lain yang merupakan bagian dari DirectX 9 pada waktu itu, tetapi mengapa, "Saya seorang programmer."



Teman sekelas saya menulis game ini selama sekitar satu tahun. Ini persis pemrograman yang saya impikan sejak kecil - sangat berteknologi tinggi, dengan penggunaan enkapsulasi, polimorfisme, dan daya tarik OOP lainnya, serta algoritma rumit dan struktur data yang tidak biasa, seperti tumpukan heaps.

Kemudian tampaknya bagi siswa tahun ke-3 tanpa pengalaman kerja dan portofolio yang jelas, ini adalah satu-satunya kesempatan untuk mendapatkan pekerjaan di perusahaan yang sangat populer pada saat GameDev. Tapi tidak ada yang datang darinya. Setelah wawancara 4 jam, di mana, antara lain, kami menulis di atas kertas algoritma untuk memotong dua baris pada assembler, kami ditolak, merujuk pada fakta bahwa kami tidak siap untuk menghubungi siswa sekarang.

C # .Net


Beberapa bulan setelah kegagalan ini, saya berakhir di sebuah perusahaan perangkat lunak besar (lebih dari 100 programmer), di mana, dengan munculnya C #, saya beralih ke sana.



Sebagai hasil komunikasi dengan kolega yang lebih berpengalaman dan berkualifikasi tinggi, dalam praktiknya saya menguasai hal-hal menarik seperti:

  • pemrograman berorientasi aspek (AOP) untuk menghilangkan kode utilitas dari logika bisnis;
  • tt-, ;
  • NServiceBus, MSMQ, WCF ;
  • Dependency Injection (DI);
  • Data Load Options (DLO) , . - GraphQL, ;

Kerugian utama adalah sejumlah kecil tugas menarik, di mana orang bisa menerapkan akumulasi pengetahuan tentang algoritma dan struktur data, serta semua jenis pola. Yah, saya seorang programmer, saya tidak bisa hidup tanpanya!



Divisi tempat saya bekerja mengembangkan antarmuka tempat data dari DBMS tersedia untuk sistem lain. Tahun demi tahun, idenya mulai tumbuh di benak saya bahwa dalam perusahaan tanpa jiwa, semuanya hanya itu dan itu OOP untuk tugas-tugas seperti "ambil ini, putar sedikit dan berikan kembali ke sana" adalah sebuah ide.

Di tingkat DBMS relasional dan kontrak data dalam REST API, OOP hanya didukung oleh antusiasme pengembang yang ingin menulis di OOP dan hanya di atasnya. Dan logika lapisan menengah (antara mengambil dari database dan memberikan kepada konsumen) dalam hal kompleksitas dan keterhubungan tidak pernah GameDev dan dapat dengan mudah ditulis dalam gaya fungsional.

Pada pertengahan 2015, dengan 8 tahun pengalaman dalam C #, saya pergi ke SoftMediaLab untuk tugas-tugas yang menarik .



Berkat pengalaman yang didapat baik di backend dan di depan, saya segera menjadi kepala pengembangan proyek baru yang cukup besar - platform pinjaman p2p dan b2p online. Backend adalah C # .Net Framework 4.6, arsitektur layanan mikro, komunikasi layanan mikro WCF, postgreSQL 9.5, ORMLite, pembuatan kode kelas ORM untuk tabel sejarah dan skrip untuk menggulung / memuat ulang objek DBMS, sekitar 150 tabel dalam DBMS, postsharp 2.1 untuk AOP, soket untuk AOP, soket untuk AOP SignalR, halaman arahan, dan API WEB berdasarkan ASP.Net MVC. Di depan, angular.js dan JS sejak itu naskah saat itu belum tersebar luas. CI / CD, streaming replikasi ke node cadangan.

Setelah proyek ini, ada yang berikut, yang kami juga tulis dalam C #. Tetapi untuk menutup masalah refactoring kode JavaScript di bagian depan, kami dengan suara bulat memutuskan untuk beralih dari angular.js ke angular.io dengan dukungan skrip di luar kotak.

Saya memiliki cinta untuk naskah pada pandangan pertama.



Tentu saja, ini bukan C # yang diketik dengan kuat, tetapi lebih merupakan petunjuk pada tipifikasi yang dengannya programmer menggambarkan harapannya dari struktur objek yang sedang dikerjakannya. Dan harapan kami, ini ... yah, Anda ingat ...



Tapi tetap saja, itu adalah terobosan nyata setelah JavaScript murni dan refactoring bagian ujung depan dari proyek ini sudah tidak lagi menjadi masalah nyata. Jadi tujuannya tercapai!

Node.js


Pada awal 2017, setelah naskah naskah menunjukkan yang terbaik di bagian depan, gagasan mencoba menulis backend naskah tampaknya logis bagi saya.

Pada saat itu, saya memiliki persyaratan berikut untuk pengembangan backend:

  • kode untuk menggulung / mengecat ulang DBMS, termasuk fungsi pemicu dan tabel histori, harus dibuat secara otomatis - oleh pembuatan kode atau ORM;
  • kode untuk membuat model dan pemetaan DTO ke / dari mereka harus dibuat secara otomatis menggunakan pembuatan kode;
  • Deskripsi angkuh API REST harus dihasilkan secara otomatis dari kode sumber;
  • Seharusnya ada sistem ORM lengkap yang memungkinkan bekerja dengan MS SQL, PostgreSQL, mongoDb, serta menulis kueri khusus untuk kasus khusus;
  • redis, rabbitMQ, socket, .. ;

Saya menghabiskan sebulan meneliti item yang tidak terkait dengan pembuatan kode dan menemukan solusi yang bermanfaat seperti typeorm, express.js, socket.io, redis, typescript-rest-swagger.

Pada pertengahan 2017, node.js sudah merupakan teknologi yang cukup stabil. Dalam beberapa jam di Google, saya menemukan sekitar selusin layanan yang bekerja di bawah kondisi pemuatan real-time yang tinggi yang diimplementasikan pada sebuah node. Saya juga belajar bahwa hambatan kinerja perlu dioptimalkan dalam C ++.

Untuk mempelajari kemungkinan pembuatan kode, saya menyewa lulusan yang masuk akal dari lembaga teknis dengan sedikit pengalaman dalam C # dan umumnya tidak memiliki pengalaman dalam JS / TypeScript. Untuk, antara lain, periksa ambang untuk memasukkan TypeScript.

Hasil bulanan karya lulusan cerdas adalah tiga perpustakaan -grunt-generate-view-model , grunt-generate-history-model , grunt-generate-database , yang masih digunakan perusahaan kami pada proyek. Hanya awalan "grunt-" yang tidak relevan lagi. Itu membentang dari versi asli ketika mereka bekerja juga sebagai plugin dari pembangun proyek kasar.

Pada akhir musim panas 2017, saya mengadakan pertemuan spesialis teknis di mana saya mempresentasikan kemampuan node.js, dan, dengan mayoritas suara, kami memutuskan untuk membuat proyek komersial berikutnya dalam kumpulan node.js (TS) + angular.io (TS).

Kebetulan saat ini di perusahaan kami dua proyek dimulai secara bersamaan:

  • platform komunikasi yang memungkinkan karyawan kantor untuk berkomunikasi dengan pelanggan melalui Telegram, WhatsApp, FB, VK, dan Viber dalam satu jendela;
  • dan sebuah proyek untuk mengotomatisasi proses bisnis internal salah satu pelanggan kami.

Kedua proyek berhasil diimplementasikan pada tumpukan baru. Sejak itu, kami telah melakukan semua proyek baru di TS. Backend pada node.js, bagian depan pada reaksi (mengapa beralih dari angular.io untuk bereaksi adalah topik untuk artikel terpisah), dan aplikasi seluler pada asli-reaksi.
