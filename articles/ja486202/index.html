AlpineはDockerビルドをPythonで50倍遅く収集し、イメージを2倍重い



Dockerのベースイメージとして、Alpine Linuxがよく推奨されます。Alpineを使用すると、ビルドが小さくなり、ビルドプロセスが速くなると言われています。

しかし、Alpine LinuxをPythonアプリケーションに使用すると、次のようになります。

  • ビルドをはるかに遅くします
  • 見た目を大きくします
  • あなたの時間を無駄に
  • そして最後に、それはランタイムエラーを引き起こす可能性があります

アルパインが推奨する理由と、Pythonのある場所で使用しない方がよい理由を見てみましょう。

なぜ人々はアルパインを推薦するのですか？


イメージの一部としてgccが必要であり、ビルド速度と最終的なイメージサイズの点でAlpine LinuxとUbuntu 18.04を比較するとします。

はじめに、2つの画像をダウンロードしてサイズを比較します。

$ docker pull --quiet ubuntu:18.04
docker.io/library/ubuntu:18.04
$ docker pull --quiet alpine
docker.io/library/alpine:latest
$ docker image ls ubuntu:18.04
REPOSITORY          TAG        IMAGE ID         SIZE
ubuntu              18.04      ccc6e87d482b     64.2MB
$ docker image ls alpine
REPOSITORY          TAG        IMAGE ID         SIZE
alpine              latest     e7d92cdc71fe     5.59MB

ご覧のとおり、アルパインのベースイメージははるかに小さくなっています。では、gccをインストールしてUbuntuから始めましょう。

FROM ubuntu:18.04
RUN apt-get update && \
    apt-get install --no-install-recommends -y gcc && \
    apt-get clean && rm -rf /var/lib/apt/lists/*

完璧なDockerfileを作成することは、この記事の範囲を超えています。

ビルド速度を測定してみましょう：

$ time docker build -t ubuntu-gcc -f Dockerfile.ubuntu --quiet .
sha256:b6a3ee33acb83148cd273b0098f4c7eed01a82f47eeb8f5bec775c26d4fe4aae

real    0m29.251s
user    0m0.032s
sys     0m0.026s
$ docker image ls ubuntu-gcc
REPOSITORY   TAG      IMAGE ID      CREATED         SIZE
ubuntu-gcc   latest   b6a3ee33acb8  9 seconds ago   150MB

同じことをAlpine（Dockerfile）でも繰り返します。

FROM alpine
RUN apk add --update gcc

組み立て、組み立ての時間とサイズを確認します。

$ time docker build -t alpine-gcc -f Dockerfile.alpine --quiet .
sha256:efd626923c1478ccde67db28911ef90799710e5b8125cf4ebb2b2ca200ae1ac3

real    0m15.461s
user    0m0.026s
sys     0m0.024s
$ docker image ls alpine-gcc
REPOSITORY   TAG      IMAGE ID       CREATED         SIZE
alpine-gcc   latest   efd626923c14   7 seconds ago   105MB

約束どおり、Alpineベースのイメージは、それ自体でより速くより少なく組み立てられます。30秒ではなく15秒で、イメージサイズは150MBに対して105MBです。これはかなり良いです！

しかし、Pythonアプリケーションの構築に切り替えた場合、すべてがそれほどバラバラではありません。

何が起こっていますか？

アルパインはホイールをサポートしていません


Debianベースのビルドを見ると、matplotlib-3.1.2-cp38-cp38-manylinux1_x86_64がダウンロードされていることがわかります。WHL。

これはホイールのバイナリです。 Alpineは `matplotlib-3.1.2.tarのソースをダウンロードします。gz `、標準のホイールをサポートしていないため。

どうして？ほとんどのLinuxディストリビューションは、C標準ライブラリのGNUバージョン（glibc）を使用します。これは、実際には、Pythonを含むすべてのCプログラムに必要です。しかし、Alpineは「musl」を使用します。これらのバイナリは「glibc」用であるため、それらは単にオプションではありません。

したがって、Alpineを使用する場合は、各PythonパッケージでCで記述されたすべてのコードをコンパイルする必要があります。

ええ、そうです、コンパイルする必要があるそのようなすべての依存関係のリストを自分で検索する必要があります。
この場合、次のようになります。

FROM python:3.8-alpine
RUN apk --update add gcc build-base freetype-dev libpng-dev openblas-dev
RUN pip install --no-cache-dir matplotlib pandas

そして、ビルド時間は...

... 25分57秒！そして、画像のサイズは851MBです。

アルパインベースのイメージははるかに長くかかり、それら自体が大きく、すべての依存関係を探す必要があります。もちろん、マルチステージビルドを使用してビルドサイズを小さくすることもできますが、それは、より多くの作業を行う必要があることを意味します。

それがすべてではありません！

アルパインは予期しないランタイムバグを引き起こす可能性があります


  • 理論的には、muslはglibcと互換性がありますが、実際には、違いが多くの問題を引き起こす可能性があります。そしてもしそうなら、それは確かに不愉快です。発生する可能性があるいくつかの問題を次に示します。
  • Alpineのデフォルトのストリームスタックは小さいため、Pythonでエラーが発生する可能性があります
  • 一部のユーザーは、muslがメモリを割り当てる方法（glibcとは異なります）が原因でPythonアプリケーションが遅くなることに気づきました。
  • 1人のユーザーが日付のフォーマット中にエラーを検出しました

確かにこれらのエラーはすでに修正されていますが、誰がさらに多くのことを知っていますか。

Pythonにアルパインイメージを使用しない


大規模で長いビルド、依存関係の検索、および潜在的なエラーに煩わされたくない場合は、ベースイメージとしてAlpine Linuxを使用しないでください。良いベースイメージを選択する。
