追加した2番目のステップは、Dockerビルドプロセスです。 DockerはDockerfileを読み取り、consignment-serviceという名前のイメージを作成します。ドットはディレクトリパスを示すため、ここではビルドプロセスで現在のディレクトリを確認するだけです。

Makefileに新しいエントリを追加します。 

run: 
    docker run -p 50051:50051 shippy-service-consignment

ここでは、ポート50051を開いてdockerイメージを起動します。Dockerは別のネットワークレイヤーで動作するため、ポートをリダイレクトする必要があります。たとえば、このサービスをポート8080で開始する場合は、-p引数を8080：50051に変更する必要があります。-dフラグを含めることで、コンテナーをバックグラウンドで実行することもできます。たとえば、docker run -d -p 50051：50051 consignment-service。

実行$メイクの実行を再び別の端末のパネルで、その後、実行main.goを行く$、それはまだ動作することを確認してください。

$ docker buildを実行すると、コードとランタイムがイメージに埋め込まれます。 Dockerイメージは、環境とその依存関係の移植可能なスナップショットです。 Docker Hubに投稿することで、Dockerイメージを共有できます。これは、npmまたはdockerイメージのyumリポジトリに似ています。 DockerfileでFROMを定義する場合、Dockerリポジトリからこのイメージをプルしてベースとして使用するようにDockerに指示します。次に、このベースファイルの一部を展開して再定義し、必要に応じて再定義できます。 Dockerイメージは公開しませんが、Dockerリポジトリを参照して、ほとんどすべてのソフトウェアがコンテナに既にパッケージ化されていることに注意してください。いくつかの本当に素晴らしいものがドッキングされました。

Dockerfileの各広告は、最初に作成されたときにキャッシュされます。これにより、変更を加えるたびにランタイム全体を再構築する必要がなくなります。 Dockerは、どの詳細が変更され、どの詳細を再構築する必要があるかを把握できるほどスマートです。これにより、ビルドプロセスが非常に高速になります。

コンテナについては十分です！コードに戻りましょう。

gRPCサービスを作成する場合、接続を作成するための多くの標準コードがあり、接続できるようにクライアントまたは別のサービスでサービスアドレスの場所をハードコーディングする必要があります。クラウドでサービスを開始すると、同じホストを使用しない場合や、サービスの再デプロイ後にアドレスまたはIPが変更される場合があるため、これは困難です。

これが、発見サービスの出番です。検出サービスは、すべてのサービスとその場所のディレクトリを更新します。各サービスは実行時に登録され、終了時に登録解除されます。次に、各サービスに名前または識別子が割り当てられます。したがって、サービス名が同じであれば、新しいIPアドレスまたはホストアドレスが存在する場合でも、他のサービスからのこのサービスへの呼び出しを更新する必要はありません。

原則として、この問題には多くのアプローチがありますが、プログラミングのほとんどの場合と同様に、誰かがこの問題にすでに対処している場合、ホイールを再発明することは意味がありません。 @Chuhnk（Asim Aslam）、Go-microの作成者素晴らしい明快さと使いやすさでこれらの問題を解決します。彼は独力で素晴らしいソフトウェアを生み出しています。あなたが見るものが好きなら彼を助けることを考慮してください！

マイクロ行く


Go-microはGoで記述された強力なマイクロサービスフレームワークであり、ほとんどの場合、Goで使用できます。ただし、サイドカーを使用して他の言語と対話することができます。

Go-microには、Goでマイクロサービスを作成するための便利な機能があります。しかし、おそらく彼が解決する最も一般的な問題から始めます。これがサービスの発見です。

go-microを使用するには、サービスにいくつかの更新を加える必要があります。Go-microはProtocプラグインとして統合されます。この場合、現在使用している標準のgRPCプラグインが置き換えられます。それでは、Makefileでこれを置き換えることから始めましょう。

go-micro依存関係を必ずインストールしてください：

go get -u github.com/micro/protobuf/{proto,protoc-gen-go}

Makefileを更新して、gRPCプラグインの代わりにgo-microプラグインを使用します。 

build: 
        protoc -I. --go_out=plugins=micro:. \ proto/consignment/consignment.proto
        GOOS=linux GOARCH=amd64 go build
        docker build -t consignment .
run: 
    docker run -p 50051:50051 shippy-service-consignment

go-microを使用するには、shippy-service-consignment / main.goを更新する必要があります。これにより、以前のgRPCコードのほとんどが抽象化されます。登録を簡単に処理し、サービスの作成を高速化します。

shippy-service-consignment / main.go
// shippy-service-consignment/main.go
package main

import (

	"fmt"
    
    //  protobuf 
	pb "github.com/EwanValentine/shippy/consignment-service/proto/consignment"
	"github.com/micro/go-micro"
	"context"
)
//repository -  
type repository interface {
	Create(*pb.Consignment) (*pb.Consignment, error)
	GetAll() []*pb.Consignment
}

// Repository -    ,
//      
type Repository struct {
	consignments []*pb.Consignment
}

func (repo *Repository) Create(consignment *pb.Consignment) (*pb.Consignment, error) {
	updated := append(repo.consignments, consignment)
	repo.consignments = updated
	return consignment, nil
}

func (repo *Repository) GetAll() []*pb.Consignment {
	return repo.consignments
}

//        
//       proto.
//           .
type service struct {
	repo repository
}

// CreateConsignment -        ,
//    create,     
//     gRPC.
func (s *service) CreateConsignment(ctx context.Context, req *pb.Consignment, res *pb.Response) error {

	// Save our consignment
	consignment, err := s.repo.Create(req)
	if err != nil {
		return err
	}

	// Return matching the `Response` message we created in our
	// protobuf definition.
	res.Created = true
	res.Consignment = consignment
	return nil
}

//GetConsignments -        
func (s *service) GetConsignments(ctx context.Context, req *pb.GetRequest, res *pb.Response) error {
	consignments := s.repo.GetAll()
	res.Consignments = consignments
	return nil
}

func main() {

	repo := &Repository{}

	//     Go-micro
	srv := micro.NewService(

		//           proto
		micro.Name("shippy.service.consignment"),
	)

	// Init will parse the command line flags.
	srv.Init()

	//  
	pb.RegisterShippingServiceHandler(srv.Server(), &service{repo})

	//  
        log.Println(" ")
	if err := srv.Run(); err != nil {
		fmt.Println(err)
	}
}


ここでの主な変更は、サービスの登録を処理するmico.NewService（）から適切に抽象化されたgRPCサーバーの作成方法です。最後に、接続自体を処理するservice.Run（）関数。以前と同様に、実装を登録しますが、今回は少し異なる方法を使用します。

2番目に大きな変更は、サービスメソッド自体に関するものです。引数と応答のタイプは、要求構造と応答構造の両方を引数として受け入れるようにわずかに変更され、現在はエラーのみを返します。私たちの方法では、go-microが処理する応答を設定します。

最後に、ポートをプログラミングしなくなりました。 Go-microは、環境変数またはコマンドライン引数を使用して構成する必要があります。アドレスを設定するには、MICRO_SERVER_ADDRESS =：50051を使用します。デフォルトでは、Microはローカルで使用するサービス検出ブローカーとしてmdns（マルチキャストDNS）を使用します。通常、mdnsを使用して本番環境のサービスを検出することはありませんが、Consulやetcdなどをローカルで実行してテストする必要はありません。これについては後で詳しく説明します。

これを反映するためにMakefileを更新しましょう。

build: 
        protoc -I. --go_out=plugins=micro:. \ proto/consignment/consignment.proto
        GOOS=linux GOARCH=amd64 go build
        docker build -t consignment .
run: 
     docker run -p 50051:50051 \ -e MICRO_SERVER_ADDRESS=:50051 \ shippy-service-consignment

-eは環境変数のフラグです。環境変数をDockerコンテナーに渡すことができます。たとえば、-e ENV = staging -e DB_HOST = localhostなどのように、各変数にフラグを設定する必要があります。

$ make runを実行すると、サービスディスカバリを含むDockerisedサービスが作成されます。それでは、これを使用するようにCliツールを更新しましょう。

package main

import (
	"encoding/json"
	"io/ioutil"
	"log"
	"os"

	"context"

	pb "github.com/EwanValentine/shippy-service-consignment/proto/consignment"

	micro "github.com/micro/go-micro"
)

const (
	address         = "localhost:50051"
	defaultFilename = "consignment.json"
)

func parseFile(file string) (*pb.Consignment, error) {
	var consignment *pb.Consignment
	data, err := ioutil.ReadFile(file)
	if err != nil {
		return nil, err
	}
	json.Unmarshal(data, &consignment)
	return consignment, err
}

func main() {
	service := micro.NewService(micro.Name("shippy.cli.consignment"))
	service.Init()

	client := pb.NewShippingServiceClient("shippy.service.consignment", service.Client())

	// Contact the server and print out its response.
	file := defaultFilename
	if len(os.Args) > 1 {
		file = os.Args[1]
	}

	consignment, err := parseFile(file)

	if err != nil {
		log.Fatalf("Could not parse file: %v", err)
	}

	r, err := client.CreateConsignment(context.Background(), consignment)
	if err != nil {
		log.Fatalf("Could not greet: %v", err)
	}
	log.Printf("Created: %t", r.Created)

	getAll, err := client.GetConsignments(context.Background(), &pb.GetRequest{})
	if err != nil {
		log.Fatalf("Could not list consignments: %v", err)
	}
	for _, v := range getAll.Consignments {
		log.Println(v)
	}
}

//Create -    
func (repo *Repository) Create(consignment *pb.Consignment) (*pb.Consignment, error) {
	repo.mu.Lock()

	repo.consignments = updated
	repo.mu.Unlock()
	return consignment, nil
}

       //

	if err := srv.Run(); err != nil {
		fmt.Println(err)
	}
}


ここでは、シップサービスのクライアントインスタンスを作成しました。これにより、サービス名を使用できます。 shipy.service.vesselは、クライアントとして船のサービスを呼び出し、そのメソッドと対話します。この場合、1つのメソッド（FindAvailable）のみです。船のサービスの仕様として、出荷するコンテナの数とともにバッチ重量を出荷します。これにより、この仕様に対応する船が返されます。

consignment-cli / consignment.jsonファイルを更新し、ハードコードされたship_idを削除します。これは、船の検索サービスが機能していることを確認するためです。また、コンテナをいくつか追加して、重量を増やします。例えば：
