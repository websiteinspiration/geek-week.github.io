Cómo usar ensamblados de terceros en el complemento

Problema



Yo, y creo que muchos desarrolladores de MS Dynamic CRM (en mi caso, este es el 2016 en las instalaciones) se encontraron con un problema, para usar ensambles de terceros, debe combinarlos con el ensamblaje principal y luego registrarlo en la Utilidad de registro de complementos.



Como ejemplo, muchos usan ILMerge o ILRepack.



Buscar



Busqué opciones de solución en varios sitios, pero en todas partes, excepto usando ILMerge o ILRepack, no encontré ninguna otra opción. Me gustaría implementar la tarea para que los ensamblajes puedan moverse junto con la base de datos a otro servidor, y todo funcione sin problemas.



Sin descubrimiento



Creo que la opción descrita por mí no traerá algo especialmente nuevo, pero por un lado no es una solución trivial que mi amigo y colega Yuriy Yurkovskiy me ayudó a encontrar .

Sugirió almacenar los ensamblajes necesarios en la base de datos y cargarlos a nivel de sitio utilizando el módulo.

  1. Crear repositorio para ensamblajes
  2. Implementación de carga de ensamblajes en la memoria, según sea necesario

Código completo
using System;

using System.Data.SqlClient;

using System.Reflection;

using System.Web;



namespace CrmExternalAssemblyLoader

{

public class AssemblyLoaderModule: IHttpModule

{

private static bool IsLoaded;

private readonly object _lock = new object();

public void Dispose()

{

throw new NotImplementedException();

}



public void Init(HttpApplication context)

{

lock (_lock)

{

if (!IsLoaded)

{

AppDomain.CurrentDomain.AssemblyResolve += CurrentDomain_AssemblyResolve;

IsLoaded = true;

}

}

}



private Assembly CurrentDomain_AssemblyResolve(object sender, ResolveEventArgs args)

{

var assemblyName = new AssemblyName(args.Name);

var assembly = GetAssembly(assemblyName.Name + ".dll");

if (assembly != null)

{

return Assembly.Load(assembly);

}

return null;

}



public byte[] GetAssembly(string assemblyName)

{

var sqlConnString = @«Data Source=[database]; Initial Catalog=[catalog]; Integrated Security=true;»;

var query = @«select Lib

from CrmExternalAssambly

where Name = name»;

using (var connection = new SqlConnection(sqlConnString))

{

using (var command = new SqlCommand(query, connection))

{

command.Parameters.Add(new SqlParameter("@name", assemblyName));

connection.Open();

var reader = command.ExecuteReader();

if (reader.HasRows)

{

while (reader.Read())

{

return (byte[])reader[0];

}

}

return null;

}

}

}

}

}

Hecho.



Cuando se inicia la ejecución del complemento, CRM intentará encontrar el ensamblado en la memoria, no lo encontrará y se arrastrará a AppDomain.CurrentDomain.AssemblyResolve, donde el ensamblado se resta de la base de datos y se carga en la memoria del dominio. La asamblea vivirá hasta que se vuelva a cargar el sitio.



Pros contras



+ No necesita fusionar ensamblajes en el complemento, por lo tanto, el tamaño del ensamblaje no aumentará, no habrá mucha carga en SQL cuando se inicie el complemento, especialmente si el complemento no está registrado en ninguno.

+ La velocidad de acceso al ensamblado es mayor ya que ya está cargada en el dominio.

+ Todos los ensamblados de terceros están en su base de datos y migran con la base.

+ Capacidad para almacenar ensamblajes de diferentes versiones (¡ este momento está mal probado! Por lo tanto, todavía hay algo que verificar)



- Quizás al actualizar CRM, deba agregar la línea del módulo a web.config CRM nuevamente.
