2020年にNode.jsから何が期待できますか？

Node.jsは2019年に10歳になりました。 npmレジストリで利用可能なパッケージの数は100万を超えました。毎年、Node.jsプラットフォーム自体のダウンロード量は40％増加します。 Node.jsのもう1つのマイルストーンは、このプロジェクトがOpenJS Foundationに参加したことです。これにより、プロジェクトの状態と安定性の改善に加えて、一般に、JavaScriptコミュニティのメンバー間の相互作用の分野における前向きな進展が期待できます。 Node.jsの世界では、1年以上の短期間に多くの興味深いことが起こったことがわかります。 Node.jsは毎年勢いを増しています。 2020年に他のものに依存する理由はありません。





Node.jsの次のリリースでは、多くの興味深い機能が見つかります。この資料は、2020年に登場する可能性がある最も重要なプラットフォームの革新に特化しています。

イノベーションNode.js 13


2019年12月の初め、この資料の執筆時点では、Node.js 13がこのプラットフォームの最新バージョンです。新しい年に向けて、実験を開始したり準備したりできる多くの新機能と改善点がすでにあります。それらの一部を次に示します。

  • ECMAScriptモジュール。
  • WebAssemblyのサポート。
  • 診断レポート。
  • 国際化の完全なサポート。特に、日付、時刻、数値、通貨の形式について話します。
  • QUICプロトコルのサポート。
  • V8 JavaScriptエンジンのパフォーマンスの向上。

これらの機能について詳しく説明する前に、Node.jsの新しいバージョンのリリースと、すでにリリースされているバージョンのプラットフォームのサポートの領域で何が期待できるかについて説明します。

2020年にNode.jsの新しいバージョンをリリースするプロセス


Node.jsの新しいメジャーバージョンが6か月ごとにリリースされます。1つは10月、もう1つは4月です。それらは「現在のバージョン」と呼ばれます。この記事の執筆時点では、Node.jsの現在のバージョンは、2019年10月にリリースされた13日です。

奇数のバージョン（v9、v11、v13）が毎年10月にリリースされます。それらは比較的短い期間に関連しており、本番環境で使用する準備ができているとは見なされていません。それらはプラットフォームのベータ版と考えることができます。これらは、偶数のNode.jsの次のバージョンに該当する新機能と変更をテストするように設計されています。

偶数のバージョン（v8、v10、v12）は毎年4月にリリースされます。そのようなバージョンのリリース後、以前の「偶数」バージョンのアップデートのリリースは停止します。これらのバージョンは「奇妙なバージョン」よりもはるかに安定していますが、それらに対するアクティブな作業はさらに6か月間継続します。現時点では、それらはリリース候補の状態にあると見なすことができます。

「偶数」バージョンが6か月以内に確定すると、ライフサイクルの新しい段階に入り、LTSバージョン（長期サポート、長期サポート）に変わります。 Node.jsのLTSバージョンは、本番環境で使用する準備ができていると見なされます。今後12か月間に、これらのバージョンのバグ修正、セキュリティアップデート、その他の改善がリリースされます。これはすべて、既存のアプリケーションの操作性の維持を考慮して行われます。

LTSステージの後、メンテナンスステージが始まります。現時点では、重大なバグ修正とセキュリティ更新のみがリリースされています。護衛段階は18ヶ月続きます。この時間が経過すると、対応するバージョンはライフサイクルの終わり（EOL、サポート終了）の段階に進み、サポートされなくなります。

ECMAScriptモジュール


バージョン13.2.0では、Node.jsはCommonJSモジュールと新しい標準ECMAScriptモジュール（ESモジュール）の両方をサポートし、サードパーティのツールは必要ありません。この手段は、開発者は最終的に命令を使用することができますimportし、exportJavaScriptでフロントエンドのプロジェクトを作成するとき、彼らはおそらく既に使用するのと同じものを。さらに、Node.jsのESモジュールはデフォルトでストリクトモードで動作することに注意することが重要です。その結果、このモードを有効にするために、各ファイルの先頭に追加する必要はありません"use strict";。

//  message
async function sendMessage { ... }
export { sendMessage };

//  index
import { sendMessage } from "./message";

ただし、開発者がESモジュールを使用していることをNode.jsに通知するには、何かを行う必要があります。ここでは、最も一般的に使用される2つの方法のいずれかを使用できます。これらの1つ目は、ファイルに拡張子が割り当てられること.mjsです。 2つ目は、package.json構造が最も近い親ファイルに配置されることです"type": "module"。

最初のアプローチではすべてが明確です.js--filesは-filesに名前が変更され.mjsます。 2番目の方法を使用する場合は、値を含むプロパティをルートファイルpackage.json、またはpackage.jsonESモジュールを含むフォルダーに配置されているプロパティに追加します。typemodule

{
   "type": "module"
}

ESモジュールでの作業に関する別の可能性は、package.jsonプロジェクトのルートファイルにそれらを含め、CommonJSモジュールを使用してファイル拡張子をに変更することです.cjs。

個人的に、拡張を使用していることに私には思われる.mjsかは.cjs特に良いアイデアではありません。したがって、ESモジュールで作業するには、ファイルを変更するだけで十分ですpackage.json。

WebAssemblyモジュールのインポート


Node.jsはESモジュールだけでなく、WebAssemblyモジュール（Wasmモジュール）のインポートもサポートするようになりました。wasmモジュールは、同様のJavaScriptコードよりも高速に解析でき、ネイティブと同様の速度で実行できるポータブルバイナリ形式のコードを含むファイルです。Wasmモジュールは、C / C ++、Go、C＃、Java、Python、Elixir、Rustなどの言語でソースコードを開発することで作成できます。

WebAssemblyモジュールのサポートはまだ実験段階です。この機能を有効にするには、アプリケーションの起動時にNode.jsに特別なコマンドラインフラグを渡す必要があります。

node --experimental-wasm-modules index.js

例を考えてみましょう。Wasmモジュールとして実装された画像処理ライブラリがあるとします。JSコードでそのようなライブラリを操作するには、次のようにします。

import * as imageUtils from "./imageUtils.wasm";
import * as fs from "fs";
( async () => {
   const image = await fs.promises.readFile( "./image.png" );
   const updatedImage = await imageUtils.rotate90degrees( image );
} )();

ここでは、モジュールをインポートし、そのモジュールで使用可能な関数を使用しました。

Node.jsのWasmモジュールは、新しい動的命令を使用してインポートできますimport()。

"use strict";
const fs = require("fs");
( async () => {
   const imageUtils = await import( "./imageUtils.wasm" );
   const image = await fs.promises.readFile( "./image.png" );
   const updatedImage = await imageUtils.rotate90degrees( image );
} )();

WebAssemblyシステムインターフェイス


JavaScriptのようなWebAssemblyテクノロジーは、セキュリティを考慮して設計されました。 Wasmコードは、サンドボックスと呼ばれることもある安全な環境で実行されます。これにより、そのようなコードから実行するオペレーティングシステムを保護できます。ただし、場合によっては、Node.js環境で実行されているWasmモジュールは、システムコールを実行する機能からのみ恩恵を受けることができます。

これは、WebAssemblyシステムインターフェイス（WebAssemblyシステムインターフェイス、WASI）がシーンに入る場所です。 WASIは、Wasmコードを実行するシステムにアクセスするための標準インターフェースとして設計されています。このようなシステムには、ホストアプリケーション、オペレーティングシステムなどが含まれます。

Node.jsリポジトリが最近のコミットを明らかにする初期WASIサポートの導入。実際のところ、WebAssemblyシステムインターフェイスはNode.jsの興味深い機能の1つであり、2020年にその外観が期待できます。

診断レポートのサポート


診断レポートは、ソフトウェアメカニズムの動作に関する情報を含む人々を対象としたJSONドキュメントです。そのような情報には、関数呼び出しスタック、オペレーティングシステムに関する情報、ロードされたモジュールに関するデータ、およびアプリケーションのサポートを支援することを目的としたその他の有用なインジケーターが含まれます。未処理の例外、重大なエラーが発生した場合も、同様のレポートを作成できます。これらは、プロセスからのシグナルによって、または新しいAPIを使用して形成できますprocess.report。Node.jsは、診断レポートが指定されたファイル名を使用して指定されたフォルダーに保存されるように構成できます。

現在、この機能は実験的です。これを有効にするには、アプリケーションの起動時にNode.jsに特別なフラグを渡す必要があります。

node --experimental-report --report-uncaught-exception --report-filename=./diagnostics.json index.js

国際化のサポートの拡張


Node.js 13.xには、ICU（International Components for Unicode）ライブラリのフルバージョンが含まれています。ICUは成熟した人気のあるプロジェクトです。このライブラリの多くの機能の中で、ローカライズされたフォーマットでの数値、日付、通貨、および時間出力のフォーマットのサポートに注目できます。彼女は時間間隔に関連する計算を実行でき、文字列を比較してテキストをUnicodeから他のエンコーディングに、またはその逆にトランスコードできます。

Node.jsの他のいくつかの新機能


Node.jsのさらに興味深いいくつかの機能を次に示します。その外観は2020年に期待できます。

  • QUICプロトコルのサポート。QUICは、アプリケーション間で信頼性の高い生産的な通信を確立できる最新のインターネットプロトコルです。
  • Python 3のサポートが改善されました。2020年には、Python 3を使用してNode.jsモジュールとネイティブモジュールをビルドできるようになります。
  • JSエンジンV8の更新バージョン。V8エンジンのバージョン7.8および7.9では、アプリケーションのパフォーマンスとWasmサポートが向上します。
  • 安定したAPIワーカースレッド。Worker Threads APIを使用すると、集中的なコンピューティングを並列化できます。
