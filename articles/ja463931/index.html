Pythonはどのようにデータ型をサポートしていますか？


Pythonは動的に型付けされた言語です。つまり、使用される変数のタイプは、プログラムの実行中にのみチェックされます。記事の前の部分で示した例では、Pythonで作成するプログラマーが変数のタイプを計画する必要がなく、データを格納するためにどれだけのメモリーが必要になるかを考える必要があることがわかります。

ここに実行のためにPythonコードを準備するときに何が起こるか：「Pythonでは、ソースコードはCPythonを使用して、バイトコードと呼ばれるはるかに単純な形式に変換されます。バイトコードは、本質的にプロセッサの命令に似ている命令で構成されています。ただし、これらはプロセッサではなく、仮想マシンと呼ばれるソフトウェアシステムによって実行されます。 （これは、オペレーティングシステム全体を実行できる機能を持つ仮想マシンに関するものではありません。私たちの場合、これは、プロセッサで実行されているプログラムが利用できる環境の簡易バージョンである環境です）。

CPythonは、プログラムを実行する準備をするときに、どのような変数の型がどうあるべきかをどのようにして知るのですか？結局、私たちはこれらのタイプを示しませんでした。 CPythonはこれについて知りません。彼は変数がオブジェクトであることを知っているだけです。 Pythonのすべてがオブジェクトです。少なくとも、何かがより具体的なタイプであることが判明するまでは。

たとえば、Pythonは一重引用符または二重引用符で囲まれたすべてを文字列と見なします。 Pythonは、数値を検出すると、対応する値を数値型と見なします。そのタイプのエンティティでは実行できないエンティティで何かを行おうとすると、Pythonによって後で通知されます。

システムは、文字列と浮動小数点数を追加できないことを通知します。また、事実name、これは文字列であり、そしてseconds試みが追加しましたまでの数であるが、システムは興味はなかったnameとseconds。

つまり、次のように記述することができます。 Pythonは、特定のオブジェクトがどの型を持っているかには関心がありません。システムが関心を持っているのは、additionメソッドへの意味のある呼び出しを返すかどうかだけです。そうでない場合、エラーが発行されます。

それはどういう意味ですか？これは、Pythonでプログラムを作成した場合、CPythonインタープリターがエラーのある同じ行の実行に従事するまでエラーメッセージを受信しないことを意味します。

このアプローチは、大規模なプロジェクトで作業しているチームに適用した場合、不便でした。実際のところ、このようなプロジェクトでは、個別の変数ではなく、複雑なデータ構造を扱います。そのようなプロジェクトでは、いくつかの関数が他の関数を呼び出し、それらが今度はいくつかの他の関数を呼び出します。チームメンバーは、プロジェクトのコードをすばやく確認できる必要があります。プロダクションに入る前にプロジェクトのエラーを検出する適切なテストを記述できない場合、そのようなプロジェクトでは大きな問題が発生する可能性があることを意味します。

厳密に言えば、ここでPythonの型注釈についての会話に行きます。

Pythonでの型注釈


これで、Pythonの型注釈について真剣に話す準備ができました。Python 2で記述されたプログラムを読み取ると、プログラマーがコードのリーダーに、関数によって返される変数または値のタイプをコードの読者に知らせるヒントを提供していることがわかります。

同様のコードはもともとこのように見えました：

users = [] # type: List[UserID]
examples = {} # type: Dict[str, Any]

型注釈は、以前は単純なコメントでした。しかし、偶然に、Pythonはアノテーションを処理するより均一な方法に徐々に移行し始めました。特に、関数の注釈に特化したドキュメントPEP 3107の登場について話します。 

次に、PEP 484の作業が開始されました。タイプアノテーションに特化したこのドキュメントは、スクリプトを実行する前にタイプを検証することを目的としたDropBoxプロジェクトであるmypyと密接に関連して開発されました。 mypyを使用すると、スクリプトの実行中に型チェックが実行されないことに注意してください。たとえば、このタイプがサポートしていないタイプを作成しようとすると、実行時にエラーメッセージが表示されることがあります。たとえば、辞書をスライスしたり.pop()、文字列のメソッドを呼び出そうとするとします。

アノテーションの実装の詳細について、PEP 484から学ぶことができるのは次のとおりです。「これらのアノテーションは、実行時に通常の属性を通じて利用できますがannotations、実行時に型チェックは実行されません。代わりに、この提案は、ユーザーが希望する場合、プログラムのソースコードをチェックできる独立したスタンドアロン型チェックツールの存在を提供します。一般に、類似の型チェックツールは非常に強力なリンターのように機能します。 （もちろん、個々のユーザーがタイプとランタイムソフトウェアをチェックするために同様のツールを使用することができ、けれども- 。設計によって契約の方法論を実現するための、またはJIT最適化のパフォーマンスのために、そのような楽器がまだ十分に成熟に達していないことに留意すべきである」。

たとえば、自分のアプリケーションがさまざまなのIDEでの作業を容易にする可能性を意味している。だから、PyCharmは、タイプ情報に基づいて、コード補完と検証を提案します。VS Codeでも同様の機能を利用できます。

型注釈はもう1つの理由で役立ちます。それは、開発者を愚かな間違いから保護するためです。ここで、このような保護の偉大な例。

人の名前を辞書に追加するとします。

names = {'Vicki': 'Boykis',

def append_name(dict, first_name, last_name):

これを修正しましょう：

from typing import Dict 

names_new: Dict[str, str] = {'Vicki': 'Boykis',
                             'Kim': 'Kardashian'}

def append_name(dic: Dict[str, str] , first_name: str, last_name: str):
    dic[first_name] = last_name

append_name(names_new,'Kanye',9.7)

mypyでは、文字列が期待される場所で数値を使用できないことがわかります。このようなチェックを定期的に使用する場合は、継続的インテグレーションシステムでコードテストが実行されるmypyを含めることをお勧めします。

型注釈の開始


コードベースの入力を開始するときに何を開始するかに関して、mypyのドキュメントで適切な推奨事項を見つけることができます。

  1. — , , , mypy.
  2. mypy. .
  3. mypy CI- , .
  4. , .
  5. , ; , .
  6. MonkeyType PyAnnotate .

独自のコードに注釈を付ける前に、何かを処理することは役に立ちます。

まず、文字列、整数、ブール値、および他の基本的なPython型の値以外のものを使用する場合は、タイピングモジュールをコードにインポートする必要があります。

第二に、このモジュールはいくつかの複合型を扱うことを可能にします。 -その中でもDict、Tuple、List、とSet。ビューの設計Dict[str, float]は、要素が文字列をキーとして使用し、浮動小数点数を値として使用するディクショナリを操作することを意味します。Optionalおよびと呼ばれるタイプもありUnionます。

3番目に、型注釈形式に慣れる必要があります。

import typing

def some_function(variable: type) -> return_type:
  do_something

プロジェクトにタイプアノテーションを適用する方法について詳しく知りたい場合は、多くの優れたチュートリアルがこれに捧げられていることに注意したいと思います。ここにその一つがあります。最高だと思います。習得したら、コードの注釈とその検証について学びます。

結果。Pythonで型注釈を使用する価値はありますか？


次に、Pythonで型注釈を使用する必要があるかどうかを自問してみましょう。実際、プロジェクトの機能によって異なります。以下は、mypyのドキュメントでGuido van Rossumがこれについて述べていることです。「mypyの目的は、静的に型付けされたPythonコードを書くように全員を説得することではありません。現在および将来、静的型付けは完全にオプションです。 Mypyの目標は、Pythonプログラマーにより多くのオプションを提供することです。これは、Pythonを、大規模なプロジェクトで使用される他の静的に型付けされた言語の競争力のある代替にするためのものです。プログラマーの生産性を高め、ソフトウェアの品質を向上させることです。」

mypyを構成し、特定のプログラムに必要なタイプを計画するために必要な時間は、小規模なプロジェクトや実験（Jupyterで実行されるプロジェクトなど）では正当化されません。どのプロジェクトを小規模と見なす必要がありますか？おそらく、注意深く見積もると、ボリュームが1000行を超えないものです。

型注釈は、より大きなプロジェクトで意味があります。そこで特に彼らは多くの時間を節約することができます。私たちは、バージョン管理システムとCIパイプラインの開発に使用される、プログラマーのグループによって開発されたプロジェクト、パッケージ、コードについて話しています。

型注釈は、今後数年間で現在よりもはるかに一般的になると思います。それらは通常の日常的なツールに変わる可能性があるという事実は言うまでもありません。そして、他の人より先に彼らと一緒に働き始める人は何も失うことはないと私は信じています。
