反応フォームの開発。KISS、YAGNI、DRYの原則

こんにちは。このチュートリアルでは、コードの品質に焦点を当てて、Reactで非常にシンプルでありながら制御されたフォームを開発する方法について説明します。

原因に！KISSとYAGNIを使用した単純なフォームの作成


それでは、承認フォームを実装するタスクがあるとします。

KISSとYAGNIの原則を分析することから開発を開始し、残りの原則について一時的に忘れます。

KISS-「コードはシンプルで愚かなままにしてください。」単純なコードの概念はおなじみだと思います。しかし、「ダム」コードはどういう意味ですか？私の理解では、これは最小限の抽象化を使用して問題を解決するコードですが、これらの抽象化の相互のネストも最小限です。

YAGNI-「あなたはそれを必要としないでしょう。」コードは、それが記述されている目的のみを実行できる必要があります。後で必要になる可能性がある機能や、アプリケーションを改善する機能は作成していません。タスクの実装に特に必要なことだけを行います。

これらの原則に厳密に従ってください。ただし、次の点も考慮してください。

  • initialDataそしてonSubmitためにLogInForm上部から来る（これはフォームが処理することができなければならない場合は特に、有用な技術であるcreateとupdate同時に）
  • 各フィールドに必要 label

さらに、これは別のチュートリアルのトピックであるため、私たちにとっては興味深くないし、有効ではないので、スタイル設定を見逃してみましょう。

上記の原則に従って、フォームを自分で実装してください。

私のフォームの実装
const LogInForm = ({ initialData, onSubmit }) => {
  const [logInData, setLogInData] = useState(initialData);

  const handleSubmit = e => {
    e.preventDefault();
    onSubmit(logInData);
  };

  return (
    <form onSubmit={handleSubmit}>
      <label>
        Enter your nickname
        <input
          value={logInData.nickname}
          onChange={e => setLogInData({ ...logInData, nickname: e.target.value })}
        />
      </label>
      <label>
        Enter your email
        <input
          type="email"
          value={logInData.email}
          onChange={e => setLogInData({ ...logInData, email: e.target.value })}
        />
      </label>
      <label>
        Enter your password
        <input
          type="password"
          value={logInData.password}
          onChange={e => setLogInData({ ...logInData, password: e.target.value })}
        />
      </label>
      <button>Submit</button>
    </form>
  );
};






各開発者の考え方が異なるため、おそらくあなたの決定は多少異なります。各フィールドに独自の状態を作成したり、ハンドラー関数を別の変数に配置したりできますが、これは重要ではありません。

しかし、すべてのフィールドに対して1つのハンドラー関数をすぐに作成した場合、これは問題に対する「最も愚かな」解決策ではなくなります。そして、この段階での目標は、後で全体像を見て、最適な抽象化を選択できるように、最も単純で最も「ダム」なコードを作成することです。

まったく同じコードを使用している場合、これはクールで、私たちの考え方が収束していることを意味します。

さらに、このコードで作業します。シンプルですが、理想からは程遠いです。



リファクタリングとDRY


それでは、DRYの原則について説明します。

DRY、簡単な表現-「コードを複製しないでください。」原理は単純に見えますが、欠点があります。コードの重複を取り除くには、抽象化を作成する必要があります。これらの抽象化が十分でない場合、KISSの原則に違反します。

コードをより速く書くためにDRYは必要ないことを理解することも重要です。そのタスクは、ソリューションの読み取りとサポートを簡素化することです。したがって、急いで抽象化をすぐに作成しないでください。コードの一部の単純な実装を作成し、コードの読み取りを簡素化し、必要に応じて変更箇所を減らすために、作成する必要がある抽象化を分析することをお勧めします。

正しい抽象化のチェックリスト：
  • 抽象化の名前はその目的と完全に一致しています
  • 抽象化は、特定の理解可能なタスクを実行します
  • 抽象化が抽出されたコードの読み取りが改善されました

それでは、リファクタリングに取り掛かりましょう。

最初の関数は、イベントから値を取得する詳細を説明しinputます。我々は、このロジックを保存することが可能な2つの抽象化の選択肢があります。InputFieldとLogInForm。

コードを参照する必要がある抽象化を正しく決定するには、DRY原則の完全な定式化に目を向ける必要があります。

具体例の知識の一部は、のイベントから値を取得する方法を知ることinputです。この知識を私たちの知識に保存するのであればLogInForm、InputField別の形では、知識を複製するか、それを別の抽象化して、そこから使用する必要があります。また、KISSの原則に基づいて、抽象化の数を最小限に抑える必要があります。実際、このロジックを私たちのロジックに入れるだけInputFieldで、外部コードが内部での入力の動作について何も知らないのに、なぜ別の抽象化を作成する必要があるのでしょうかInputField。それは、それが内向きに進むのと同じように、既成の意味を持つだけです。

将来イベントが必要になる可能性があると混乱している場合は、YAGNIの原則を思い出してください。onChangeEventコンポーネントにはいつでも追加の小道具を追加できますInputField。

それまでは次のようにInputFieldなります。
コードをコピー
const InputField = ({ label, type, value, onChange }) => (
  <label>
    {label}
    <input
      type={type}
      value={value}
      onChange={e => onChange(e.target.value)}
    />
  </label>
);



したがって、コンポーネントへの入力および出力中に型の均一性が観察され、外部コードで発生していることの本質が隠されます。今後、チェックボックスや選択などの別のUIコンポーネントが必要になった場合は、入出力の型の均一性も維持します。

このアプローチでは、フォームに入出力ソースを追加できるため、それぞれに固有のハンドラーを追加する必要がないため、柔軟性が高まります。

このヒューリスティックなトリックは、多くのフレームワークにデフォルトで組み込まれています。たとえば、これは主要なアイデアであるv-modelにVue多くの人々は、フォームでの作業の容易さのために愛するいます。

他に何ができますか？



プロジェクトに多くのフォームがある場合は、handleChange計算を別のフックに配置できますuseFieldChange。
コードをコピー
  // hooks/useFieldChange.js
  const useFieldChange = setState => fieldName => fieldValue => {
    setState(state => ({
      ...state,
      [fieldName]: fieldValue,
    }));
  };
  // LogInForm.js
  const handleChange = useFieldChange(setLogInData);



これは純粋な関数であるため（常に同じ関数を返すため、最初に呼び出されたとき）、フックである必要はありません。しかし、フックは概念的には、Reactのより適切で自然な解決策に見えます。インプレース

サポートを追加callbackしてfieldValue、通常の動作を以下setStateから完全に複製することもできますReact。
コードをコピー
  const isFunc = val => typeof val === "function";

  const useFieldChange = setState => fieldName => fieldValue => {
    setState(state => ({
      ...state,
      [fieldName]: isFunc(fieldValue) ? fieldValue(state[fieldName]) : fieldValue,
    }));
  };

その結果、17行のjsxコードで16行の構成を取得します。ブラボー！これは私がDRYを理解していることです。ここにこれらの入力が100個ある場合は、それぞれ605行と506行になります。

しかし、結果として、KISSの原則に違反する、より複雑なコードが得られました。結局のところ、これは2つの抽象化で構成されています。フィールドとアルゴリズム（はい、アルゴリズムも抽象化です）は、それをReact要素のツリーに変換します。このコードを読むとき、これらの抽象化の間を絶えずジャンプする必要があります。

しかし、このコードの最大の問題は、そのサポートと拡張です。
次の場合に彼に何が起こるか想像してみてください。
  • 可能なプロパティが異なるいくつかのフィールドタイプが追加されます
  • 他のフィールドの入力に基づいてフィールドがレンダリングされるか、レンダリングされない
  • 一部のフィールドでは、変更時に追加の処理が必要です
  • 選択の値は以前の選択に依存します

このリストは長い間続けることができます...

これを実装するには、常にいくつかの関数を設定に保存する必要があり、設定をレンダリングするコードは徐々にフランケンシュタインに変わり、さまざまな小道具の束を取り、さまざまなコンポーネントにそれらを投げます。

以前に作成されたコードは、コンポーネントの構成に基づいて静かに拡張され、必要に応じて変更されますが、中間抽象化（構成からReact要素のツリーを構築するアルゴリズム）がないため、それほど複雑にはなりません。このコードは重複していません。特定のデザインパターンに従うだけなので、「パターン化」されたように見えます。

役に立たない最適化


Reactにフックが現れた後、すべてのハンドラーとコンポーネントを無差別にuseCallbackとにラップする傾向がありましたmemo。お願い、それはやめて！これらのフックは、Reactが遅く、すべてを最適化する必要があるため、React開発者によって提供されませんでした。パフォーマンスの問題が発生した場合に備えて、アプリケーションを最適化する余地を提供します。そして、あなたは、このような問題が発生した場合でも、あなたはでプロジェクト全体をラップする必要はありませんmemoとuseCallback。使用プロファイラは、問題を特定し、だけにして適切な場所にmemoizeします。

メモ化すると、コードは常に複雑になりますが、生産性が向上するわけではありません。

実際のプロジェクトのコードを見てみましょう。関数がどのように見えるかを比較して比較しますuseCallback。
コードをコピー
  const applyFilters = useCallback(() => {
    const newSelectedMetrics = Object.keys(selectedMetricsStatus).filter(
      metric => selectedMetricsStatus[metric],
    );
    onApplyFilterClick(newSelectedMetrics);
  }, [selectedMetricsStatus, onApplyFilterClick]);

  const applyFilters = () => {
    const newSelectedMetrics = Object.keys(selectedMetricsStatus).filter(
      metric => selectedMetricsStatus[metric],
    );
    onApplyFilterClick(newSelectedMetrics);
  };



理想的なコードは欠けているため、ラッパーを削除すると、コードの可読性が明らかに向上しました。

この関数は次のように使用されるため、このコードのパフォーマンスは向上していません。
コードをコピー
  <RightFooterButton onClick={applyFilters}>APPLY</RightFooterButton>



場所RightFooterButton-それはstyled.buttonからのものでstyled-components、非常に速く更新されます。しかし、メモリの消費量は、常にメモリに保持されます反応するので、私たちのアプリケーションを増加させselectedMetricsStatus、onApplyFilterClickおよび機能のバージョンapplyFiltersこれらの依存性に関連します、。

これらの引数では不十分な場合は、このトピックについてより幅広く説明している記事を読んでください。
