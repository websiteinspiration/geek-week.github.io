Pythonバギーコード：開発者が犯す最も一般的な10の間違い

Pythonについて


Pythonは、動的なセマンティクスを持つ、解釈されたオブジェクト指向の高水準プログラミング言語です。組み込みの高レベルのデータ構造と動的型付けおよび動的リンクを組み合わせることで、BRPS（迅速なアプリケーション開発）や、既存のコンポーネントやサービスを接続するためのスクリプトや接続言語としての使用が非常に魅力的になります。Pythonはモジュールとパッケージをサポートしているため、プログラムのモジュール化とコードの再利用が促進されます。

この記事について


この言語を習得するシンプルさと使いやすさは、開発者（特にPythonの学習を始めたばかりの開発者）を混乱させる可能性があるため、重要な微妙な点を見失い、Pythonを使用したさまざまなソリューションの力を過小評価する可能性があります。

このことを念頭に置いて、この記事では、上級のPython開発者でも発生する可能性のある微妙で見つけにくいエラーの「トップ10」を紹介します。

間違い＃1：式を関数の引数のデフォルト値として誤用する


Pythonでは、デフォルト値を設定することにより、関数がオプションの引数を持つことができることを示すことができます。もちろん、これは言語の非常に便利な機能ですが、この値の型が変更可能である場合、不快な結果を招く可能性があります。たとえば、次の関数定義を考えてみます。

>>> def foo(bar=[]):        # bar -    
                            #      .
...    bar.append("baz")    #     ...
...    return bar

この場合のよくある間違いは、オプションの引数の値が、この引数の値なしで関数が呼び出されるたびにデフォルト値に設定されると考えることです。たとえば、上記のコードでは、foo（）関数が繰り返し呼び出されると（つまり、bar引数に値を指定せずに）、foo（）が呼び出されるたびに（なしで）「baz」が返されると想定されます。引数bar）を指定すると、barは[]（つまり、新しい空のリスト）に設定されます。

しかし実際に何が起こるか見てみましょう：

>>> foo()
["baz"]
>>> foo()
["baz", "baz"]
>>> foo()
["baz", "baz", "baz"]

そして？関数が毎回新しいリストを作成するのではなく、foo（）が呼び出されるたびに既存のリストにデフォルト値「baz」を追加し続けるのはなぜですか？

この質問に対する答えは、Pythonの「内部」で何が起こっているかをより深く理解することです。つまり、関数のデフォルト値は、関数の定義中に一度だけ初期化されます。したがって、bar引数は、foo（）が初めて定義された場合にのみデフォルトで初期化されます（つまり、空のリスト）。関数の最初の定義時に引数バー用に作成されます。

参考までに、このエラーの一般的な「回避策」は次の定義です。

>>> def foo(bar=None):
...    if bar is None:		# or if not bar:
...        bar = []
...    bar.append("baz")
...    return bar
...
>>> foo()
["baz"]
>>> foo()
["baz"]
>>> foo()
["baz"]

間違い＃2：クラス変数の誤用


次の例を考えてみます。

>>> class A(object):
...     x = 1
...
>>> class B(A):
...     pass
...
>>> class C(A):
...     pass
...
>>> print A.x, B.x, C.x
1 1 1

すべてが整然としているようです。

>>> B.x = 2
>>> print A.x, B.x, C.x
1 2 1

ええ、すべてが期待通りでした。

>>> A.x = 3
>>> print A.x, B.x, C.x
3 2 3

なんてこったい？！Axを変更しましたが、Cxも変更されたのはなぜですか？

Pythonでは、クラス変数は辞書のように扱われ、メソッド解決順序（MRO）と呼ばれるものに従います。したがって、上記のコードでは、属性xはクラスCにないため、その基本クラスにあります（上の例ではAのみですが、Pythonは多重継承をサポートしています）。つまり、CにはAから独立した独自のプロパティxはありません。したがって、Cxへの参照は実際にはAxへの参照です。これは、これらのケースが適切に処理されない場合に問題を引き起こします。したがって、Pythonを学習するときは、クラス属性とそれらの操作に特に注意してください。

間違い番号3：例外ブロックの不正なパラメーター


次のコードがあるとします。

>>> try:
...     l = ["a", "b"]
...     int(l[2])
... except ValueError, IndexError:  # To catch both exceptions, right?
...     pass
...
Traceback (most recent call last):
  File "<stdin>", line 3, in <module>
IndexError: list index out of range

ここでの問題は、例外式がこの方法で指定された例外のリストを受け入れないことです。むしろ、Python 2.xでは、「except Exception、e」という表現を使用して、オプションの2番目に指定された2番目のパラメーター（この場合はe）に例外をバインドし、さらに検査できるようにします。その結果、上記のコードでは、IndexError例外はexceptステートメントでキャッチされません。代わりに、例外は代わりにIndexErrorという名前のパラメーターへのバインドで終了します。

exceptステートメントを使用して複数の例外をキャッチする正しい方法は、最初のパラメーターを、キャッチしたいすべての例外を含むタプルとして指定することです。また、互換性を最大限に高めるには、asキーワードを使用します。この構文はPython 2とPython 3の両方でサポートされているためです。

>>> try:
...     l = ["a", "b"]
...     int(l[2])
... except (ValueError, IndexError) as e:  
...     pass
...
>>>

間違い＃4：Pythonのスコープルールの誤解


Pythonのスコープは、いわゆるLEGBルールに基づいています。これは、Local（関数（defまたはlambda）内で何らかの方法で割り当てられた名前で、この関数ではグローバルに宣言されていない）の略です。 defまたはlambda）、内部から外部へ）、グローバル（モジュールファイルのトップレベルで割り当てられた名前、またはファイル内のdefでグローバル命令を実行することによって）、組み込み（組み込みの名前モジュールで以前に割り当てられた名前：オープン、範囲、構文エラー、 ...）。簡単そうですね。まあ、実際には、これがPythonでどのように機能するかにはいくつかの微妙な点があります。次の例について考えてみます。

>>> x = 10
>>> def foo():
...     x += 1
...     print x
...
>>> foo()
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "<stdin>", line 2, in foo
UnboundLocalError: local variable 'x' referenced before assignment

何が問題ですか？

上記のエラーが発生するのは、スコープで変数を割り当てると、Pythonはそのスコープに対してローカルであると自動的に見なし、親スコープで同じ名前の変数を非表示にするためです。

したがって、以前に実行したコードでUnboundLocalErrorを受け取ったときに、関数本体のどこかに代入演算子を追加することによって変更されたときに、多くの人が驚いています。

リストを使用する場合、この機能は特に開発者を混乱させます。次の例を考えてみます。

>>> lst = [1, 2, 3]
>>> def foo1():
...     lst.append(5)   #   ...
...
>>> foo1()
>>> lst
[1, 2, 3, 5]

>>> lst = [1, 2, 3]
>>> def foo2():
...     lst += [5]      # ...    !
...
>>> foo2()
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "<stdin>", line 2, in foo
UnboundLocalError: local variable 'lst' referenced before assignment

そして？foo1が正常に動作しているときにfoo2が落ちるのはなぜですか？

答えは前の例と同じですが、一般的な考えでは、ここの状況はより微妙です。foo1は代入演算子をlstに適用しませんが、foo2は適用しません。lst + = [5]は実際にはlst = lst + [5]の省略形であることを念頭に置くと、値lstを割り当てようとしていることがわかります（したがって、Pythonはローカルスコープにあると想定します）。ただし、lstに割り当てる値は、まだ決定されていないlst自体（ここでもローカルスコープにあると見なされます）に基づいています。そして、エラーが発生します。

間違い＃5：反復中にリストを変更する


次のコードの問題はかなり明白です。

>>> odd = lambda x : bool(x % 2)
>>> numbers = [n for n in range(10)]
>>> for i in range(len(numbers)):
...     if odd(numbers[i]):
...         del numbers[i]  # BAD: Deleting item from a list while iterating over it
...
Traceback (most recent call last):
  	  File "<stdin>", line 2, in <module>
IndexError: list index out of range

反復中にリストまたは配列から項目を削除することは、経験豊富なソフトウェア開発者にはよく知られているPythonの問題です。しかし、上記の例はかなり明白かもしれませんが、経験豊富な開発者でも、はるかに複雑なコードでこのレーキに乗り出すことができます。

さいわい、Pythonには多くの洗練されたプログラミングパラダイムがあり、正しく使用すると、コードを大幅に簡略化および最適化できます。これのもう1つの楽しい結果は、より単純なコードでは、反復中にリスト項目を誤って削除するというエラーに陥る可能性がはるかに少ないということです。これらのパラダイムの1つはリストジェネレーターです。さらに、リストジェネレーターの操作を理解することは、この特定の問題を回避するのに特に役立ちます。これは、上記のコードの代替実装で示されているように、うまく機能します。

>>> odd = lambda x : bool(x % 2)
>>> numbers = [n for n in range(10)]
>>> numbers[:] = [n for n in numbers if not odd(n)]  # ahh, the beauty of it all
>>> numbers
[0, 2, 4, 6, 8]

間違い＃6：Pythonがクロージャー内の変数をバインドする方法を誤解している


次の例を考えてみます。

>>> def create_multipliers():
...     return [lambda x : i * x for i in range(5)]
>>> for multiplier in create_multipliers():
...     print multiplier(2)
...

次の出力が期待できます。

0
2
4
6
8

しかし、実際にはこれを取得します。

8
8
8
8
8

驚き！

これは、Pythonのレイトバインディングが原因です。つまり、クロージャーで使用される変数の値は、内部関数呼び出し中に検索されます。したがって、上記のコードでは、返された関数のいずれかが呼び出されるたびに、値iがその呼び出し中に周囲のスコープで検索されます（そのときまでにサイクルが既に完了しているため、最終結果が割り当てられていました-値4） 。

この一般的なPythonの問題の解決策は次のとおりです。

>>> def create_multipliers():
...     return [lambda x, i=i : i * x for i in range(5)]
...
>>> for multiplier in create_multipliers():
...     print multiplier(2)
...
0
2
4
6
8

出来上がり！ここではデフォルトの引数を使用して無名関数を生成し、目的の動作を実現します。このソリューションをエレガントと呼ぶ人もいます。いくつかは
微妙です。一部の人々はこれらのものを嫌います。しかし、Python開発者であれば、とにかく理解することが重要です。

間違い＃7：周期的なモジュールの依存関係を作成する


次のように、a.pyとb.pyの2つのファイルがあり、それぞれが他方をインポートするとします

。

import b

def f():
    return b.x
	
print f()

b.pyで：

import a

x = 1

def g():
    print a.f()

まず、a.pyをインポートしてみます。

>>> import a
1

うまくいきました。これはあなたを驚かせるかもしれません。結局、モジュールはお互いに周期的にインポートし、これはおそらく問題になるはずですよね？

その答えは、単純にモジュールを周期的にインポートすること自体は、Pythonの問題ではないということです。モジュールがすでにインポートされている場合、Pythonは十分に賢く、再インポートを試みません。ただし、各モジュールが別のモジュールで定義された関数または変数にアクセスしようとするポイントによっては、実際に問題が発生する場合があります。

したがって、例に戻ると、a.pyをインポートしたとき、b.pyはインポート中にa.pyを定義する必要がないため、b.pyのインポートに問題はありませんでした。b.pyでのaへの唯一の参照は、af（）の呼び出しです。しかし、これはg（）で呼び出され、a.pyまたはb.pyでg（）を呼び出さないものはありません。したがって、すべて正常に動作します。

しかし、（最初にa.pyをインポートせずに）b.pyをインポートしようとするとどうなりますか？

>>> import b
Traceback (most recent call last):
  	  File "<stdin>", line 1, in <module>
  	  File "b.py", line 1, in <module>
    import a
  	  File "a.py", line 6, in <module>
	print f()
  	  File "a.py", line 4, in f
	return b.x
AttributeError: 'module' object has no attribute 'x'

おおおお。これは良くない！ここでの問題は、b.pyのインポートプロセス中に、a.pyをインポートしようとし、次にf（）を呼び出して、bxへのアクセスを試みますが、bxはまだ定義されていません。したがって、AttributeError例外です。

この問題の少なくとも1つの解決策はかなり簡単です。b.pyを変更して、a.pyをg（）にインポートします。

x = 1

def g():
    import a	# This will be evaluated only when g() is called
    print a.f()

これをインポートすると、すべて正常です。

>>> import b
>>> b.g()
1	# Printed a first time since module 'a' calls 'print f()' at the end
1	# Printed a second time, this one is our call to 'g'

間違い＃8：Python標準ライブラリのモジュール名と交差する名前


Pythonの魅力の1つは、箱から出して使える多くのモジュールです。しかし、その結果、意識的にこれに従わない場合、モジュールの名前がPythonに付属する標準ライブラリのモジュールと同じ名前である場合があります（たとえば、コード内に名前のモジュールがある場合があります） email.pyは、同じ名前の標準ライブラリモジュールと競合します）。

これは深刻な問題を引き起こす可能性があります。たとえば、モジュールのいずれかがPython標準ライブラリからモジュールのバージョンをインポートしようとし、プロジェクトに同じ名前のモジュールがある場合、標準ライブラリからのモジュールの代わりに誤ってインポートされます。

したがって、Python標準ライブラリのモジュールと同じ名前を使用しないように注意する必要があります。プロジェクト内のモジュールの名前を変更する方が、標準ライブラリ内のモジュールの名前を変更するリクエストを送信して承認を受けるよりもはるかに簡単です。

間違い＃9：Python 2とPython 3の違いを考慮に入れていない


次のfoo.pyファイルを考えてみます。

import sys

def bar(i):
    if i == 1:
        raise KeyError(1)
    if i == 2:
        raise ValueError(2)

def bad():
    e = None
    try:
        bar(int(sys.argv[1]))
    except KeyError as e:
        print('key error')
    except ValueError as e:
        print('value error')
    print(e)

bad()

Python 2では正常に動作します。

$ python foo.py 1
key error
1
$ python foo.py 2
value error
2

しかし、それがPython 3でどのように機能するかを見てみましょう。

$ python3 foo.py 1
key error
Traceback (most recent call last):
  File "foo.py", line 19, in <module>
    bad()
  File "foo.py", line 17, in bad
    print(e)
UnboundLocalError: local variable 'e' referenced before assignment

ここで何が起こったのですか？「問題」は、Python 3では、例外ブロック内のオブジェクトがその外部では使用できないことです。（この理由は、それ以外の場合、ガベージコレクターが起動してそこへの参照を削除するまで、このブロックのオブジェクトはメモリに格納されるためです）。

この問題を回避する1つの方法は、例外ブロックオブジェクトへの参照をこのブロックの外部に保持して、使用可能なままにすることです。これは、この手法を使用して、Python 2とPython 3の両方に適したコードを取得する前の例のバージョンです。

import sys

def bar(i):
    if i == 1:
        raise KeyError(1)
    if i == 2:
        raise ValueError(2)

def good():
    exception = None
    try:
        bar(int(sys.argv[1]))
    except KeyError as e:
        exception = e
        print('key error')
    except ValueError as e:
        exception = e
        print('value error')
    print(exception)

good()

Python 3で実行します。

$ python3 foo.py 1
key error
1
$ python3 foo.py 2
value error
2

やったー！

間違い＃10：__del__メソッドの不適切な使用


次のようなmod.pyファイルがあるとします。

import foo

class Bar(object):
   	    ...
    def __del__(self):
        foo.cleanup(self.myhandle)

そして、あなたは別のanother_mod.pyからこれをやろうとしている：

import mod
mybar = mod.Bar()

そして、ひどいAttributeErrorを取得します。

どうして？ここで報告されているように、インタープリターがシャットダウンすると、モジュールのグローバル変数の値はすべてNoneになります。その結果、上記の例では、__ del__が呼び出されたときに、名前fooはすでにNoneに設定されていました。

この「アスタリスク付きのタスク」の解決策は、atexit.register（）を使用することです。したがって、プログラムが実行を終了すると（つまり、プログラムが正常に終了すると）、インタプリタが作業を完了する前にハンドルが削除されます。

これを念頭に置いて、上記のmod.pyコードの修正は次のようになります。

import foo
import atexit

def cleanup(handle):
    foo.cleanup(handle)


class Bar(object):
    def __init__(self):
        ...
        atexit.register(cleanup, self.myhandle)

このような実装は、通常のプログラム終了後に必要なクリーンアップを呼び出す簡単で信頼性の高い方法を提供します。明らかに、self.myhandle名に関連付けられているオブジェクトの処理方法の決定はfoo.cleanupに任されていますが、あなたはその考えを理解したと思います。
