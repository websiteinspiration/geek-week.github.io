高性能でネイティブなパーティション分割：TimescaleDBをサポートするZabbix

Zabbixは監視システムです。他のシステムと同様に、すべての監視システムの3つの主要な問題に直面しています。データの収集と処理、履歴の保存、およびそのクリーニングです。

データの取得、処理、および記録のステップには時間がかかります。それほど多くはありませんが、大規模なシステムの場合、これにより大きな遅延が発生する可能性があります。ストレージの問題はデータアクセスの問題です。これらは、レポート、チェック、およびトリガーに使用されます。データへのアクセスの遅延もパフォーマンスに影響します。データベースが大きくなると、無関係なデータを削除する必要があります。除去は、一部のリソースを消費する難しい操作です。



Zabbixでの収集と保存中の遅延の問題は、キャッシュによって解決されます：いくつかのタイプのキャッシュ、データベースでのキャッシュ。3番目の問題を解決するには、キャッシングは適切ではないため、ZabbixはTimescaleDBを使用しました。Zabbix SIAのテクニカルサポートエンジニアであるAndrey Gushchinがこれについて話します。Andreyは6年以上Zabbixをサポートしており、パフォーマンスに直接直面しています。 TimescaleDBはどのように機能し、通常のPostgreSQLと比較してどのようなパフォーマンスを発揮できますか？TimescaleDBでZabbixはどのような役割を果たしますか？ゼロから開始する方法とPostgreSQLで移行する方法、およびどの構成パフォーマンスの方が優れていますか？カットの下のこのすべてについて。




パフォーマンスの課題


各監視システムは、特定のパフォーマンスの課題に直面しています。データの収集と処理、保存、履歴のクリーニングの3つについてお話します。

迅速なデータ収集と処理。優れた監視システムは、すべてのデータを迅速に受信し、独自の基準に従ってトリガー式に従って処理する必要があります。処理後、システムはこのデータをデータベースにすばやく保存して、後で使用することもできます。

ストーリーを保つ。優れた監視システムは、データベースに履歴を保存し、メトリックへの便利なアクセスを提供する必要があります。ストーリーは、レポート、グラフ、トリガー、しきい値、およびアラートの計算されたデータ項目で使用するために必要です。

履歴をクリア。場合によっては、指標を保存する必要がない日が来ることがあります。5年前、1〜2か月前に収集されたデータが必要なのはなぜですか。一部のノードが削除され、一部のホストまたはメトリックは古くなって収集が停止したため、不要になりました。優れた監視システムは、データベースが大きくならないように、履歴データを保存し、時々削除する必要があります。

古いデータを消去することは、データベースのパフォーマンスに大きな影響を与えるホットな問題です。

Zabbixキャッシング


Zabbixでは、最初と2番目の呼び出しはキャッシュを使用して解決されます。RAMはデータの収集と処理に使用されます。ストレージ-トリガー、グラフ、および計算されたデータアイテムのストーリー。DB側では、チャートなどの主要なサンプルの特定のキャッシュがあります。

Zabbixサーバー自体のキャッシングは：

  • ConfigurationCache;
  • ValueCache;
  • HistoryCache;
  • TrendsCache。

より詳細に検討してみましょう。

ConfigurationCache


これは、メトリクス、ホスト、データ要素、トリガーを格納するメインキャッシュです。前処理とデータ収集に必要なすべてのものです。



これはすべて、データベースに不要なクエリを作成しないように、ConfigurationCacheに格納されます。サーバーの起動後、このキャッシュを更新し、構成を作成して定期的に更新します。

データ収集


スキームはかなり大きいですが、その主なものはアセンブラーです。これらはさまざまな「ポーラー」、つまり組み立てプロセスです。彼らはさまざまなタイプのアセンブリを担当しています。SNMP、IPMIを介してデータを収集し、すべてをPreProcessingに転送します。

コレクターはオレンジ色で囲まれています。

Zabbixは、検証の集計に必要な集計データ要素を計算しました。それらがある場合は、ValueCacheから直接それらのデータを取得します。

HistoryCacheの前処理


すべてのコレクターは、ConfigurationCacheを使用してジョブを受信します。次に、それらをPreProcessingに渡します。



PreProcessingは、ConfigurationCacheを使用してPreProcessingステップを受信します。このデータはさまざまな方法で処理されます。

PreProcessingを使用してデータを処理した後、それをHistoryCacheに保存して処理します。これでデータ収集が終了し、Zabbix- history syncerのメインプロセスに進みます。これは、モノリシックアーキテクチャであるためです。

注：前処理は非常に難しい操作です。v 4.2以降、プロキシに送信されています。多くのデータ要素と収集の頻度を持つ非常に大きなZabbixがある場合、これは作業を非常に容易にします。

ValueCache、履歴、トレンドキャッシュ


History Syncerは、各データ要素、つまり各値をアトミックに処理するメインプロセスです。

History Syncerは、HistoryCacheから値を取得し、構成のトリガーを計算してチェックします。もしそうなら、それは計算します。

History Syncerは、イベント、エスカレーションを作成してアラートを作成します（構成で必要な場合）。後続の処理のトリガーがある場合、履歴テーブルにアクセスしないように、この値をValueCacheに記憶します。したがって、ValueCacheには、トリガー、計算された要素の計算に必要なデータが入力されます。

History Syncerはすべてのデータをデータベースに書き込み、それをディスクに書き込みます。以上で処理は終了です。



DBキャッシング


グラフやイベントレポートを見たいときに、データベース側にはさまざまなキャッシュがあります。

  • Innodb_buffer_pool MySQL側;
  • shared_buffers PostgreSQL側;
  • effective_cache_size オラクルの側で;
  • shared_pool DB2側で。

他にも多くのキャッシュがありますが、これらはすべてのデータベースの主要なキャッシュです。これにより、クエリで必要になることが多いデータをメモリに保持できます。彼らはこれのために独自の技術を持っています。

データベースのパフォーマンスは重要です


Zabbixサーバーは常にデータを収集して書き込みます。再起動すると、履歴から読み取られてValueCacheに入力されます。スクリプトとレポートは、Webインターフェースに基づいて構築されたZabbix APIを使用します。Zabbix APIはデータベースにアクセスし、グラフ、レポート、イベントリスト、最近の問題に必要なデータを受け取ります。



視覚化のため-Grafana。ユーザーの間で、これは人気のあるソリューションです。Zabbix APIを介して直接データベースにリクエストを送信でき、データを受信するための一定の競争力を生み出します。したがって、結果とテストの迅速な出力に対応するために、データベースをより詳細に調整する必要があります。

お手伝いさん


Zabbixの3番目のパフォーマンス課題は、ハウスキーパーとの歴史をクリアすることです。これはすべての設定に従います-データ要素は、変更（トレンド）のダイナミクスを何日維持するかを示します。

TrendsCacheはオンザフライで計算されます。データが到着したら、1時間でデータを集計し、傾向の変化のダイナミクスを示すためにテーブルに書き込みます。

ハウスキーパーが起動し、通常の「選択」でデータベースから情報を削除します。これは必ずしも効果的ではありません。これは、内部プロセスのパフォーマンスグラフから理解できます。



赤いグラフは、History Syncerが常にビジーであることを示しています。上記のオレンジ色のチャートは、常に実行されているハウスキーパーです。彼は、データベースが指定したすべての行を削除することを期待しています。

いつ家政婦を消すべきですか？たとえば、「アイテムID」があり、特定の時間に最後の5000行を削除する必要があるとします。もちろん、これはインデックスによって行われます。しかし、通常、データセットは非常に大きく、データベースは引き続きディスクから読み取り、それをキャッシュに格納します。これは常にデータベースにとって非常に負荷の高い操作であり、データベースのサイズによっては、パフォーマンスの問題が発生する可能性があります。



家政婦はただの切断です。Webインターフェースには、ハウスキーパーの「管理全般」の設定があります。内部トレンド履歴の内部ハウスキーピングを無効にすると、これは管理されなくなります。

ハウスキーパーがオフになり、グラフが横ばいになりました。この場合の問題は何であり、3番目のパフォーマンスの課題を解決するのに何が役立つでしょうか。

パーティショニング-パーティショニングまたはパーティショニング


パーティショニングは通常、リストしたリレーショナルデータベースごとに異なる方法で構成されます。それぞれに独自のテクノロジーがありますが、一般的には似ています。新しいパーティションを作成すると、特定の問題が発生することがよくあります。

パーティションは通常、「セットアップ」（1日に作成されるデータの量）に応じて構成されます。原則として、パーティショニングは1日で公開されますが、これは最低限です。新しいパーティションの傾向について-1か月間。

非常に大きな「設定」の場合、値が変わる可能性があります。小さな「セットアップ」が最大5,000 nvps（1秒あたりの新しい値）、平均が5,000から25,000の場合、大きな「セットアップ」は25,000 nvpsを超えます。これらは、データベースの慎重な構成を必要とする、大規模で非常に大規模なインストールです。

非常に大規模なインストールでは、1日の実行が最適でない場合があります。1日あたり40 GB以上のMySQLパーティションを確認しました。これは問題を引き起こす可能性のある非常に大量のデータであり、削減する必要があります。

パーティショニングを与えるものは何ですか？


テーブルの分割。多くの場合、これらはディスク上の個別のファイルです。クエリプランは、1つのパーティションをより最適に選択します。パーティショニングは通常、ある範囲で使用されます-これはZabbixにも当てはまります。私たちはそこで「タイムスタンプ」を使います-時代の初めからの時間です。普通の数字があります。あなたは一日の始まりと終わりを設定します-これはパーティションです。

迅速な除去 - DELETE。削除する行の選択ではなく、単一のファイル/サブテーブルが選択されます。

データサンプリングを大幅に高速化 SELECT -テーブル全体ではなく、1つ以上のパーティションを使用します。2日前にデータを申請すると、キャッシュにロードして大きなテーブルではなく1つのファイルのみを発行する必要があるため、データベースからより速く選択されます。

多くの場合、多くのデータベースもスピードアップしますINSERT-子テーブルへの挿入。

Timescaledb


v 4.2では、TimescaleDBに注目しました。これは、ネイティブインターフェイスを持つPostgreSQLの拡張機能です。拡張機能は、リレーショナルデータベースの利点を失うことなく、時系列データを効果的に処理します。TimescaleDBも自動的にパーティション分割します。

TimescaleDBには、作成するハイパーテーブルの概念があります。チャンク -パーティションが含まれています。チャンクは、他のフラグメントに影響を与えないハイパーテーブルの自動的に制御されるフラグメントです。各チャンクには独自の時間範囲があります。



TimescaleDBとPostgreSQL


TimescaleDBは非常に効率的に機能します。拡張機能の製造元は、より正確な要求処理アルゴリズム、特に<code>挿入</ code>を使用していると主張しています。データセット挿入の次元が大きくなると、アルゴリズムは一定のパフォーマンスを維持します。



2億行を超えると、通常、PostgreSQLのたるみが大きくなり、パフォーマンスが最大で0まで低下します。TimescaleDBを使用すると、あらゆる量のデータの「挿入」を効果的に挿入できます。

取り付け


TimescaleDBのインストールは、どのパッケージでも簡単です。ドキュメントは詳細にすべてを説明する-それは公式PostgreSQLのパッケージに依存します。TimescaleDBは手動でコンパイルおよびコンパイルすることもできます。

Zabbixデータベースの場合は、拡張機能をアクティブにするだけです。

echo "CREATE EXTENSION IF NOT EXISTS timescaledb CASCADE;" | sudo -u postgres psql zabbix

extensionZabbixデータベース用に アクティベートして作成します。最後のステップは、ハイパーテーブルを作成することです。

TimescaleDBへの履歴テーブルの移行


これには特別な機能がありますcreate_hypertable：

SELECT create_hypertable(‘history’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
SELECT create_hypertable(‘history_unit’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
SELECT create_hypertable(‘history_log’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
SELECT create_hypertable(‘history_text’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
SELECT create_hypertable(‘history_str’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
SELECT create_hypertable(‘trends’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
SELECT create_hypertable(‘trends_unit’, ‘clock’, chunk_time_interval => 86400, migrate_data => true);
UPDATE config SET db_extension=’timescaledb’, hk_history_global=1, hk_trends_global=1

この関数には3つのパラメーターがあります。最初は、ハイパーテーブルを作成する必要があるデータベースのテーブルです。 2つ目は、作成するフィールド、chunk_time_intervalつまり使用するパーティションチャンクの間隔です。私の場合、間隔は1日-86,400です

。3番目のパラメーターはmigrate_dataです。 setの場合true、現在のすべてのデータが以前に作成されたチャンクに転送されます。自分で使ったmigrate_data。約1 TBあり、1時間以上かかりました。場合によっては、テストの際に、オプションの文字タイプの履歴データを削除して、転送しないようにしました。

最後のステップ-  UPDATE：db_extension設定timescaledbこの拡張があることをデータベースが理解できるようにします。Zabbixはそれをアクティブにし、データベースへの構文とクエリを正しく使用します-これらはTimescaleDBに必要な機能です。

PostgreSQLを使用していて、それを変更する予定がない場合は、Zabbixと組み合わせてTimescaleDB拡張機能を備えたPostgreSQLを使用することをお勧めします。このソリューションは、中程度の「セットアップ」で効果的に機能します。
