私はTelegramのJavaScript開発者のための競争で1位にならなかったので

Telegramのアクティブユーザー、特にPavel Durovに加入しているユーザーは、TelegramがiOS、Android、JavaScriptの開発者、およびこれらのインターネットサイトのデザイナーのためにコンテストを開催したという事実について聞いたことがあるでしょう。それは確かな賞金の分配を伴うかなり壮大なイベントであったという事実にもかかわらず（参加者の1人が最初に5万ドルを受け取り、最速かつ最も簡単なAndroidアプリケーションを書いた）、彼らはどういうわけか、少なくともRunetについてはほとんど何も書きませんでした。私のデビュー投稿は状況を修正しようとします。


私はフルスタックのJavaScript開発者（正確には、TypeScript開発者）なので、自分でテストすることにしました。マニラは賞金だけでなく、形式そのものでもあります。これは、抽象性と思考の速さが重要なプログラミング競争ではありません。複雑さのすべてがここで重要でした：経験、中期的な開発速度、UIの問題の味、コンピュータサイエンス全体の知識、および自己批判。コンテストの条件によると、iOS、Android、Webのいずれかのプラットフォームでグラフを表示するためのライブラリを開発する必要がありました。


異なるプラットフォームの開発者は互いに競争せず、各プラットフォームには独自の勝者がいました。主な基準は、作業の速度（古いデバイスを含む）、デザインとの一貫性、スムーズなアニメーション、最小のアプリケーションサイズでした。すでに存在するソリューションとライブラリは使用できず、すべてをゼロから作成する必要がありました。

それ以前は、開発者向けのコンテストに参加していましたが、すべてのタスクに割り当てられるのは5時間以下であり、これらの時間は多大なストレスに費やされました。テレグラムはタスクを完了するためにそのような緊張を必要としなかったという事実にもかかわらず、それは私が参加しなければならなかった最も困難なコンテストの1つです。見かけ上、単純なタスクは非常に大容量であることがわかり、私がそれを支払われた場合、コードのパフォーマンスとそのアーキテクチャの調和との間の妥協点を見つけようとして、これらの「グラフ」を数か月間カットすることができました。3つが解決策に際立っていたことを助けました（upd： 2つ、ありがとうvlad2711修正あたり）週。ライバルの一部は、特に時間をかけて競争に専念するために休暇を取っており、私は夜と週末の競争の発展をいつものように「オンタンタ」での仕事と組み合わせることにしました。

CANVASとSVG


私たち全員が直面した最も重要なアーキテクチャの問題は、グラフィックスレンダリングツールの選択でした。現在、Web標準は、オンザフライのsvgグラフィックスの生成と古き良きキャンバスの2つのアプローチを提供しています。ここにそれぞれの長所と短所があります。

キャンバス


+絶対的な汎用性-キャンバス上の任意のピクセルの色を変更する機能を備えており、好きなものを描画できます。
+ [潜在的]高いパフォーマンス-キャンバスを準備できれば、優れたパフォーマンスを発揮できます。 webglを使用することは素晴らしいことですが、スマートフォンでのサポートは貧弱です。

-すべての計算とすべての手によるレンダリング—ポリラインの中間点を一度設定できるSVGとは異なり、ビューボックスを操作して「カメラ」をポリラインのセクションに沿って移動できます。キャンバスではすべてがより複雑です。ここには「カメラ」はありません。左上隅からの座標。チャートの現在の表示領域を「移動」する必要がある場合は、表示領域の新しい位置を基準にして、すべてのポイントのすべての座標を再計算する必要があります。言い換えれば、svgでそのまま使用できるビューボックスは、キャンバスに手動で実装する必要があります。
- アニメーション全体は手動です-前の段落に基づいて、可能なすべてのアニメーションは、座標、色、透明度の値を再計算し、シーン全体を1秒あたりN回再描画することによって実装されます。シーンを再カウントして再描画することができれば、アニメーションはよりスムーズになります。

Svg


+簡単な描画-ビューポート、色、透明度のパラメーターを操作して、必要な線、形状などをSVGに一度追加するだけで、グラフナビゲーションが提供されます。
+アニメーションの単純な実装-再び、前の段落に基づいて、ビューボックス、色、および透明度の1秒あたりの新しい値を指定するだけで十分であり、画像自体が再描画され、ブラウザーがこれを処理します。さらに、SVGの形状とプリミティブはCSSでスタイル設定できるので、CSS3アニメーションを使用してアニメーション化できるので、最小限の労力でクールなアニメーションを取得できる可能性が広がります。
+デフォルトではパフォーマンスは悪くありません-遅く、何百ものリソースをキャンバスで簡単に埋めることができる場合、SVGに基づく結果は常に非常に軽量で、まともで滑らかなように見えます。 

しかし、コインには裏返しがあります。

-最適化の適度な機会-svgではなくブラウザを描画しているため、このプロセスを制御することは不可能です。たとえば、すでに描画された個々の要素をキャッシュするなどしてパフォーマンスを向上させたい場合、それを行うことはできません。ほとんどの場合、これはブラウザによって既に行われていますが、最後まで確信が持てません。
-限られたツール-SVGでは、キャンバスのすべてのピクセルを制御するのではなく、ベクタープリミティブのフレームワーク内で考えてコーディングします。ただし、このタスクの場合、これは取るに足らないマイナスであり、競争タスクのコンテキストでは多少の制限が課されますが、これも重要ではありません。

私は嫌なキャラクターの特性を持っているので、ツールを選択することを心配する必要はありませんでした。私はマキシマリストであり、私の仕事では私のお気に入りのツールのみを使用していました。学生時代から、DirectDrawを楽しんでいたとき、私のお気に入りのツールは、常に「やりたいことを実行する」キャンバスでした。そして、競争問題を解決するためのキャンバスは本当に優れていることがわかりましたが、実際に私の手に渡ったのはその1つだけです。最適化の最も広い可能性です。主な基準はアプリケーションのパフォーマンスでした。


タスクは明確です。キャンバス上の適切な場所に適切なタイミングでポイントを描画する必要があります。コードを書くことは残ります。繰り返しますが、今回は、1つの「フットクロス」を使用して生産的なコンパクトコードを手続き型で作成するか、あまり生産的ではなく、お気に入りのオブジェクト指向でさらにコンパクトでないかを選択する必要がありました。私が2番目のオプションを選択し、私のお気に入りのもう1つであるTypeScriptで味付けしたことは、おそらくご想像のとおりでしょう。

そして、この選択はあまり正しくありませんでした。抽象化とカプセル化を使用しているため、中間の計算結果を保存、送信、再利用できるとは限りません。これはパフォーマンスに悪影響を及ぼします。そして、これが広く使用されているため、JSでのOOPは不可能であり、サイズも重要ですが、コードは十分に縮小化されていません。

私の場合、新しい機能を追加する前に、古い機能のパフォーマンスを理解する必要がありました。私が解決した最初の問題は、

アニメーションのけいれんでした。

毎秒60フレームを生成するのに十分なパワーがあっても、要素の位置または透明度がアニメーションの開始からの経過時間によって決定されない場合、アニメーションはスムーズではありません。これは、ティック間の時間間隔が等しくないためです。たとえば、1ティックは10ミリ秒後に、2ティックは40後に機能し、1ティックと2ティックはオブジェクトが左に1ピクセル移動しました。視覚的には「単収縮」のように見えます。つまり、何か間違ったことをする必要があります。

let left = 10, interval = setInterval(() => {
  left += 1	
  if (left >= 90) {	
    clearInterval(interval)
  }
}, 10)

など：

let left = 10, startLeft = 10, targetLeft = 90, startTime = Date.now(), duration = 1000, interval = setInterval(() => {
  left = startLeft + (targetLeft - startLeft) * (Date.now() - startTime) / duration	
  if (left >= targetLeft) {	
    left = targetLeft
    clearInterval(interval)
  }
})
