ここで主な質問が発生します。なぜ新しいSDKがリリースされたのですか、それは現在のSDKとどう違うのですか？もしそうなら、すべては現在のソリューションで問題ありません。

現在のnRF5 SDKは単純なキューに基づいて機能し、ほとんどの場合これでほとんどすべてのタスクを実装できます（一部の企業はまだ独自のSDKを使用していますが、これらはルールの例外です）。新しいnRF Connect SDKは、Zephyr RTOSに基づく根本的に異なるアプローチを使用しています。より詳細に違いを検討してください。
RTOS（RTOS）には、特定の利点と既知の欠点の両方があります。後者には次のものが含まれます。

  • OSを維持するための追加のオーバーヘッド
  • 単純なプロジェクトでの開発は非常に複雑
  • ビルドの複雑さの増加

RTOSの残りの部分は次のとおりです。

  • 個々のフローの制御による信頼性の向上

Nordic内では、LTE Cat-M / NB-IoT / GPS-nRF91シリーズを備えた新しいシステムインパッケージ（SIP）の導入以来、これは興味深く関連性のあるものになっています。このSIPは、BLEからセルラーネットワークへの移行により出現した、新しいCortex-M33コアおよびネットワークコンポーネントの他の要件により、より高いパフォーマンスを発揮します。ここでのもう1つの革新は、SDRモデムでした。これは、別のコアで動作し、核間相互作用を編成する必要がありました。初めてRTOSベースのSDKがここに登場し、新しいアプローチに初めて遭遇した人のために、準備段階からいくつかの質問を投げかけました。開発環境を正しく組み立てるための特別なアシスタントもリリースされました-Getting Started Assistant。彼は、インストールする必要があるコンポーネント（多数あります。以下を参照）を通知し、すべてが正しくインストールされていることを確認します。



この観点から、ゼファーへの移行を最初の大量のARM Cortex-Mの外観および32ビットへの移行と比較できます。現在、大多数は32ビットMKを主なものとして使用しており、Habré に関する記事があります。また、最初は不必要に複雑に見えた移行についても説明します。しかし、時間の経過とともに、ほとんどすべての人がこれが標準になったという結論に達しました。

Nordicチップ上で実行されるRTOSはZephyr OSだけではないことは注目に値します。FreeRTOSを使用したプロジェクトの例2016年からSDK v.11で利用可能になり、SDK v.9の初期には、nRF51ファミリ（2015）のKeil RTXサポートがありました。ただし、以前はこれらはより実験的な機能であり、サポートはRTOSメーカーによって大幅に提供されていました。これは、原則として、現在真実です。

nRF5xファミリに対する非公式のZephyrサポートは、2016年に登場しました。

初めて同様のユーティリティセットに直面した人にとっては、すべてが不必要に複雑で、古いパラダイムを使用できるように思えるかもしれません。既存の開発アプローチは非常に効果的です。しかし、もっと深く見ると、すべてがはるかに面白くなります。

たとえば、Chocolateyとpipでは、OSとPythonのコンソールからそれぞれ必要なすべてのパッケージをインストールできます。問題のほとんどのソフトウェアと同様に、Python自体も1つのコマンドに入れられます。

hoco install python xxx

また、1つのコマンドで更新されます。

choco upgrade all

このアプローチは、Linuxのコンソールパッケージマネージャー（apt、zypperなど）に精通しているユーザーにとって、Windowsユーザーにとっては少し珍しいものです。 PCにOSを再インストールする場合にのみ、MKのソフトウェア開発者がソフトウェアを更新する状況によく気づきました。なぜそれが悪いのかについて話をしませんが、ここではこの問題は自動的に解決されます。

RTOSに切り替えると、新しいハードウェアにコードを適合させる問題を解決できます。 1つのファミリのチップ間でも、別のソフトデバイス（プリコンパイルされたBLEライブラリ）への移行を伴う場合、移行には開発側から特定のリソースが必要でした。たとえば、51または91シリーズから52シリーズへの移行は言うまでもなく、ハードウェアプラットフォーム自体が大幅に変更されます。これで、このタスクははるかに簡単かつ迅速に解決されます。

2つのSDKで作成されたプログラムの結果を直接比較することはできません。ライブラリとアプローチは大きく異なり、そのようなテストはまだありません。このソリューションは、ラインのミディアムチップやトップエンドチップ（nRF52832以降）で良好に動作し、リソースの大きな余力が残っていると言えます。ただし、新しいSDKがnRF52810のような新しいチップに適用できないとは言えません。問題をより詳細に検討する必要があります。

記事のタイトルの質問に戻ると、このパラダイムは間違いなく新しい現実であると言えます。一見すると、大幅に改善されます。世界最大のBLEメーカーの少なくとも2つの新しいチップファミリが正確に機能し、このテクノロジーのみで動作し、収益は期待できません。私の意見では、これは事前に準備された革命です。

更新：5月14日、Nordic は、5.0より古いBLEのすべてのバージョンがnRF Connect SDKでのみ利用可能になることを発表した新しいSDKに関するウェビナーを開催しました。したがって、Directino Finding aka AoA / AoD（BLE 5.1）とLE Audio（BLE 5.2）は、多くの人が待ち望んでいたもので、今年は新しいツールをもたらし、開発の変更は予想よりも早くなります。

調査結果：

  • 変更は根本的なものであり、現在のnRF5 SDKで動作するコードは新しい（nRF Connect SDK）と互換性がありません。
  • DevicetreeとKConfigを使用してRTOSに切り替えると、ハードウェアの抽象化のレベルを上げることができるため、プロジェクトの開発と新しいプラットフォームへの転送が大幅にスピードアップします。
  • Zephyrへの移行により、多数のプロトコルとライブラリのサポートがすぐに利用可能になります。IoTデバイスの場合、最も興味深いのは、Linuxが伝統的に強力であるネットワークとセキュリティ機能です。
  • Zephyr OSは、アセンブリ中に多くのツールを使用します。これにより、組み込みアプリケーションで特に重要な、アセンブリ時間とメモリ要件を大幅に削減できます。
  • 新しいSDKでは、高レベルの開発者を使用できます。高レベルの開発者は、低レベルの開発者よりもはるかに市場に出ています。これにより、人員と市場シェアの増加の問題が解決されます。
