これは、動的言語ランタイムシリーズの最後の部分です。前の記事：

この短い記事では、実際に動的を使用する主なケースを最後に分析します。動的なしでは実現できない場合と、それによって生活がはるかに容易になる場合です。



ダイナミックが不可欠な場合


そのようなケースはありません。静的なスタイルで、機能的に類似したコードをいつでも記述できます。唯一の違いは、読みやすさとコードの量です。たとえば、COMオブジェクトを使用する場合、動的ではなくリフレクションを使用できます。

ダイナミックが便利なとき


COMオブジェクトを操作する


まず第一に、これはもちろん、これすべてが開始されたために、COMオブジェクトでの作業です。取得したコードを動的およびリフレクションで比較します。

dynamic instance = Activator.CreateInstance(type);
instance.Run("Notepad.exe");

var instance = Activator.CreateInstance(type);
type.InvokeMember("Run", BindingFlags.InvokeMethod, null, instance, new[] { "Notepad.exe" });

原則として、リフレクションを介してCOMオブジェクトを操作するには、各メソッド/プロパティのラッパーを使用してブランチクラスを作成する必要があります。dynamicを介してメソッドを呼び出すときに、不要なパラメーターを入力しない機能（COMオブジェクトの観点からは必須）など、あまり明白でない利点もあります。

構成を操作する


もう1つの教科書の例は、XMLなどの構成での作業です。ダイナミックなし：

XElement person = XElement.Parse(xml);
	
Console.WriteLine(
	$"{person.Descendants("FirstName").FirstOrDefault().Value} {person.Descendants("LastName").FirstOrDefault().Value}"
	);

ダイナミック：

dynamic person = DynamicXml.Parse(xml);

Console.WriteLine(
	$"{person.FirstName} {person.LastName}"
);

もちろん、これには独自の動的クラスを実装する必要があります。最初のリストの代わりに、次のように機能するクラスを作成できます。

var person = StaticXml.Parse(xml);

Console.WriteLine(
	$"{person.GetElement("FirstName")} {person.GetElement("LastName")}"
);

しかし、ご覧のとおり、これはdynamicよりもはるかにエレガントに見えません。

外部リソースを操作する


前の段落は、外部リソースを使用するあらゆるアクションに一般化できます。常に2つの代替手段があります。動的を使用してネイティブCスタイルのコードを取得するか、「マジックライン」を使用して静的タイピングを行います。REST APIリクエストの例を見てみましょう。動的で、これを書くことができます：

dynamic dynamicRestApiClient = new DynamicRestApiClient("http://localhost:18457/api");
dynamic catsList = dynamicRestApiClient.CatsList;

プロパティのリクエストに応じて、ダイナミッククラスがフォームのリクエストを送信する場所

[GET] http://localhost:18457/api/catslist

それから彼はそれを逆シリアル化し、意図された使用の準備ができている猫の配列を私たちに返します。動的なしでは、次のようになります。

var restApiClient = new RestApiClient("http://localhost:18457/api");
var catsListJson = restApiClient.Get("catsList");
var deserializedCatsList = JsonConvert.DeserializeObject<Cat[]>(catsListJson);

リフレクションの交換


前の例では、質問があるかもしれません。なぜ、ある場合に戻り値を特定の型に逆シリアル化し、もう一方の場合にはそうでないのですか？事実、静的型付けでは、オブジェクトを操作するためにオブジェクトをCat型に明示的にキャストする必要があります。以下の場合に動的、逆シリアル化するのに十分であるJSONを私達の動的クラスとリターン内のオブジェクトの配列にオブジェクト[]それからいるので、動的反射の世話をします。これが機能する方法の2つの例を示します。

dynamic deserialized = JsonConvert.DeserializeObject<object>(serialized);
var name = deserialized.Name;
var lastName = deserialized.LastName;

Attribute[] attributes = type.GetCustomAttributes(false).OfType<Attribute>();
dynamic attribute = attributes.Single(x => x.GetType().Name == "DescriptionAttribute");
var description = attribute.Description;

COMオブジェクトを操作する場合と同じ原理。

ビジター


ダイナミックを 使用すると、このパターンを非常にエレガントに実装できます。千の言葉の代わりに：

public static void DoSomeWork(Item item)
{
    InternalDoSomeWork((dynamic) item);
}

private static void InternalDoSomeWork(Item item)
{
    throw new Exception("Couldn't find handler for " + item.GetType());
}

private static void InternalDoSomeWork(Sword item)
{
   //do some work with sword
}

private static void InternalDoSomeWork(Shield item)
{
    //do some work with shield
}

public class Item { }
public class Sword : Item {}
public class Shield : Item {}

これで、SwordタイプのオブジェクトをDoSomeWorkメソッドに渡すと、InternalDoSomeWork（Swordアイテム）メソッドが呼び出されます。

調査結果


ダイナミック を使用する長所：

  • ラピッドプロトタイピングに使用できます。ほとんどの場合、ボイラープレートコードの数が削減されます。
  • 原則として、コードの可読性と美学が改善されます（「マジックライン」から言語のネイティブスタイルへの移行により）。
  • 幅広い信念にもかかわらず、キャッシュメカニズムのおかげで、一般的なケースでは大きなパフォーマンスオーバーヘッドは発生しません。

動的使用の短所：

  • メモリとパフォーマンスに関連する明らかなニュアンスがあります。
  • このような動的クラスのサポートと読み取りにより、何が起こっているのかをよく理解する必要があります
  • プログラマーは型チェックとコンパイラーによって提供されるすべてのヘルス保証を奪われます

結論


私の意見では、開発者は以下の状況でダイナミックを使用することから最大の利益を得るでしょう：

  • プロトタイピングするとき
  • エラーのコストが低い小規模/ホームプロジェクトで
  • 長いランタイムを意味しない小さなコードベースのユーティリティ。ユーティリティが最悪のケースで数秒間実行される場合、通常、メモリリークやパフォーマンスの低下について考える必要はありません。

少なくとも物議を醸すのは、大規模なコードベースを使用する複雑なプロジェクトでの動的な使用です。静的なラッパーの作成に時間を費やすことで、目に見えない瞬間の数を最小限に抑えることができます。

COMオブジェクトまたはドメインをサービス/製品で使用する場合、長時間の連続稼働時間を意味するので、そのような場合のために作成されたにもかかわらず、dynamicを使用しないことをお勧めします。何をすべきか、どのようにすべきかを完全に知っていて、決してミスをしない場合でも、遅かれ早かれ、これを知らない新しい開発者が来るかもしれません。その結果、計算が困難なメモリリークが発生する可能性があります。
