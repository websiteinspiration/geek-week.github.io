リベースの本質


リベースはどのように正確に行われますか：


リベースとは、移動するブランチの固定を解除して、別のブランチに接続することです。この定義は真実ですが、もう少し深く見てみてください。ドキュメントを見ると、リベースについて次のように説明されています。「コミットを別のブランチに適用する（コミットを別の基本ヒントの上に再適用する）」。

リベースはブランチを別のブランチにコピーアンドペーストするだけではないため、ここでの主な言葉は適用することです。リベースは、選択されたブランチからすべてのコミットを順番に取得し、それらを新しいブランチに再適用します。

この動作は2つのポイントにつながります。

  1. コミットを再適用することにより、Gitは新しいコミットを作成します。同じ変更が含まれている場合でも、Gitは新しい独立したコミットと見なされます。
  2. Gitリベースはコミットをオーバーライドし、古いコミットを削除しません。つまり、リベースが完了した後も、古いコミットは.gitフォルダーの/ gjectsサブフォルダーに引き続き保存されます。Gitがどのようにコミットを保存し、考慮するかを完全に理解していない場合は、この記事の最初の部分を読んでください。

以下は、リベース中に発生することのより正確な解釈です。


ご覧のとおり、機能ブランチには完全に新しいコミットが含まれています。前に述べたように、Gitの観点からは、同じ一連の変更ですが、まったく新しいオブジェクトです。 

また、古いコミットは破棄されません。直接アクセスできなくなります。覚えていれば、ブランチはコミットへのリンクにすぎません。したがって、ブランチもタグもコミットを参照していない場合、ディスクには存在し続けますが、Gitを使用してアクセスすることはできません。

次に、ゴールデンルールについて説明します。

ゴールデンリベースルール


黄金のリベースルールは次のとおりです。 「共有ブランチをリベース決して！」共有ブランチとは、ネットワークリポジトリに存在し、あなた以外の人が作業できるブランチを指します。

多くの場合、このルールは適切な理解なしに適用されます。したがって、特にGitの作業をよりよく理解するのに役立つため、なぜこのルールが出現したかを分析します。

このソリューションは機能し、正常に機能しますが、問題に対して他のソリューションがあることを知っていると役立つ場合があります。それらの1つはプルリベースです。

プルリベースを行うと、Gitはブランチにのみ存在するコミットとネットワークリポジトリに存在するコミットを特定しようとします。次にGitは、ネットワークリポジトリからのコミットを、ローカルリポジトリとネットワークリポジトリの両方に存在する最新のコミットと結合します。次に、ローカルコミットをブランチの最後にリベースします。 

複雑に聞こえるので、次のように説明します。

  1. Gitは、ユーザーのリポジトリとネットワークリポジトリの両方にあるコミットにのみ注意を払います。

    GitHubリポジトリのローカルクローンのように見えます。

覚えているように、リベースGitがコミットを1つずつ適用する場合、つまりこの場合、マスターコミットE、次にF.をブランチの最後に適用します。結果は自身にリベースされます。見た目は良いですが、問題が発生します-なぜこれを行うのですか？

私の意見では、ブランチのマージに関する最大の問題は、コミットの履歴が汚染されていることです。したがって、pull-rebaseはよりエレガントなソリューションです。さらに進んで、ブランチに最新の変更をダウンロードする必要がある場合は、常にプルリベースを使用する必要があると言います。ただし、覚えておく必要があります。リベースはすべてのコミットを順番に適用するため、20のコミットをリベースすると、20の競合を次々に解決しなければならない場合があります。

このようなシナリオを実装するには、git rebase onがまさに意図されたものです。

まず、ドキュメントを読みます：

SYNOPSIS
       git rebase [-i | --interactive] [<options>] [--exec <cmd>]
               [--onto <newbase> | --keep-base] [<upstream> [<branch>]]
       git rebase [-i | --interactive] [<options>] [--exec <cmd>] 
[--onto <newbase>]
               --root [<branch>]
       git rebase (--continue | --skip | --abort | --quit | --edit-todo 
| --show-current-patch)


私たちはこれに興味があります：

OPTIONS
       --onto <newbase>
          Starting point at which to create the new commits. If the 
--onto option is not specified, the starting point is <upstream>. May be 
any valid commit, and not just an existing branch name.


このオプションを使用して、新しいコミットを作成するポイントを指定します。

このオプションを指定しない場合、上流が開始点になります。

理解のために、もう1つ画像を示します。

A--B--C        master
    \
     D--E      feature1
         \
          F--G feature2

Here we want to rebase feature2 to master beginning from feature1
                           |                                |
                        newbase                         upstream


つまり、masterブランチはnewbaseであり、feature 1ブランチは上流です。

したがって、前の図のように結果を取得したい場合は、feature2ブランチのgit rebase --onto master feature1を実行する必要があります。
