bin、sbin、usr / bin、usr / sbinの違いに関する別の意見

私は最近この記事を見つけました：bin、sbin、usr / bin、usr / sbinの違い。基準について私の意見を共有したいと思います。

/ 置き場


システム管理者とユーザーの両方が使用できるが、他のファイルシステムがマウントされていない場合（たとえば、シングルユーザーモード）に必要なコマンドが含まれています。スクリプトによって間接的に使用されるコマンドが含まれる場合もあります。

そこでは次のコマンドが想定されています：

cat、chgrp、chmod、chown、cp、date、dd、df、dmesg、echo、false、hostname、kill、ln、login、ls、mkdir、mknod、more、mount、mv、ps、pwd、rm、rmdir、sed、sh、stty、su、sync、true、umount、uname。

/ usrにシンボリックリンクを作成することは可能ですが、systemd / usrの時代には別のデバイスで見つかりませんでしたが、組み込みシステム、信号機、コーヒーグラインダー、およびPDP-11は、科学アカデミーの研究所の1つで重要なデバイスとして機能しています。

/ sbin


システム管理（およびその他のroot専用コマンド）に使用されるユーティリティ、/ sbinには、/ binのバイナリに加えて、システムの起動、復元、復元、および/または復元に必要なバイナリが含まれています。 / usrがマウントされた後に実行されたプログラム（問題がない場合）は、通常/ usr / sbinに配置されます。ローカルにインストールされたシステム管理プログラムは、/ usr / local / sbinに配置する必要があります。

予想：

fastboot、fasthalt、fdisk、fsck、getty、halt、ifconfig、init、mkfs、mkswap、reboot、route、swapon、swapoff、update。

ユーザーの遊び心からシステムを保護する1つの方法は、x属性を設定して、これらのユーティリティの実行を禁止することです。
さらに、/ binと/ sbinをアーカイブからのコピー（同じタイプのすべてのシステムで同じ）で置き換えると、パッケージマネージャーのないシステムをすばやく修正できます。

/ usr / bin


ここではすべてが簡単です。同じタイプのチームで、すべてのサーバー/コーヒーグラインダー会社で同じです。また、/ usr自体を別のOSに同じように展開できます（/ binと/ sbinの場合、これは通常は機能しません）。これらはアーキテクチャに依存しないプログラムです。/ optまたはネットワークの他の場所にあるperlまたはpythonインタープリターへのリンクが含まれている場合があります。

/ usr / sbin


/ usr / binと同じですが、管理者のみが使用します。

/ usr / local / binおよび/ usr / local / sbin


最も重要な場所の1つ。残りの部分とは異なり、/ usrは組織全体で同じにすることはできません。OSに依存し、ハードウェアに依存し、すべてのデバイスで必要とされない単なるプログラムがあります。マシンで/ usrを同期する場合、/ usr / localを除外する必要があります。

/ホーム/ $ USER / bin


ここでのケースは/ usr / localに似ており、特定のユーザーに固有のプログラムのみが存在します。ユーザーが移動したときに、別のマシンに転送（または同期）できます。転送できないものは/home/$USER/.local/binに追加されます。ドットなしのローカルを使用できます。/ home / $ USER /明らかな理由によるsbinがありません。

修正や追加を喜んでさせていただきます。
