Trusted Types-WebアプリケーションコードをXSS攻撃から保護する新しい方法

Googleは、最新のWebアプリケーションがXSS攻撃から、より具体的にはDOMへのJavaScriptインジェクション（DOMベースのクロスサイトスクリプティング）からフロントエンドを保護できるようにする APIを開発しました。

クロスサイトスクリプティング（XSS）は、最新のWebアプリケーションの脆弱性に関連する最も一般的なタイプの攻撃です。これはGoogleだけでなく、業界全体でも認められています。経験から、XSS攻撃に耐性のあるWebアプリケーションの開発は、特に複雑なプロジェクトに関しては、まだ簡単なことではないことがわかっています。開発者がバックエンドでこの問題を正常に解決した場合、フロントエンドではすべてがはるかに複雑になります。Googleの脆弱性報奨プログラムの一部として DOM XSS攻撃から保護するためのソリューションを提案した開発者は、ますます多くの賞を受賞しています。

XSSはコードに簡単に埋め込むことができます


DOM XSSは、ユーザーまたは他のブラウザーAPIの側面からのフィルターされていないデータが、埋め込まれたjsインジェクションとともにDOMに分類される攻撃の一種です。たとえば、アプリケーションが使用する特定のユーザーインターフェイステンプレートのスタイルシートをロードするように設計されたスニペットを考えてみます。

const templateId = location.hash.match(/tplid=([^;&]*)/)[1];
// ...
document.head.innerHTML += `<link rel="stylesheet" href="./templates/${templateId}/style.css">`

この場合、location.hashで攻撃者によって導入されたスクリプト（この場合、データソース、つまりソースの役割を果たします）はinnerHTMLに分類されます（データレシーバー、つまりシンクの役割を果たします）。この脆弱性を利用して、攻撃者は被害者がアクセスするURLを偽装する可能性があります。

https://example.com#tplid="><img src=x onerror=alert(1)>

特にコードが頻繁に変更される場合、この挿入は見落とされがちです。たとえば、昔々、templateIdがサーバー上で生成および検証され、その後、それを再確認することは誰にも起こりません。innerHTMLで何かを書くとき、その値が文字列であることを確認するだけです。しかし、この行の内容を信頼すべきですか？実際にどこから来たのですか？

さらに、問題はinnerHTMLに限定されません。典型的なDOMには、同じ問題を持つ60を超えるメソッドとプロパティがあります。そのため、DOM APIはデフォルトでは安全ではなく、XSSを防ぐための特別なアプローチが必要です。

また、コード全体を注意深く読んでも、コードベースに脆弱性があるかどうかを判断できない場合があります。この式を見てみましょう：

objが場所オブジェクト（location.hashについては前述）を指し、prop値が 'href'の場合、これはおそらくXSS DOMです。ただし、おそらくコード実行のプロセスでのみ見つけることができます。アプリケーションのどの部分でも仮想的に何かをDOMに書き込むことができるため、すべてのコードは手動のセキュリティチェックに合格する必要があり、レビュー担当者はエラーを検出するように細心の注意を払う必要があります。しかし、人的要因が取り消されていないため、このような検証方法に完全に依存することはできません。

信頼できるタイプ


Trusted Typesは、DSS XSSからのWebアプリケーションの保護など、上記の問題を根本的に排除するのに役立つ新しいブラウザーAPIです。

Trusted Types APIは、悪意のある挿入をブロックできる一連のクラス（ポリシー）に基づいて構築されています。これにより、DOMは脆弱ではなくなります。その要素は、文字列ではなく特別なオブジェクトを通じて値を受け取ります。信頼できるタイプを使用するには、コンテンツセキュリティポリシー（CSP）HTTP応答のヘッダーの特別なフィールドを初期化する必要があります。

Content-Security-Policy: trusted-types *

CSPは、クロスサイトスクリプティング（XSS）やデータインジェクション攻撃など、特定の種類の攻撃を認識して排除できる追加のセキュリティレイヤーです。CSPを有効にするには、応答でContent-Security-Policy HTTPヘッダーを使用するようにサーバーを構成する必要があります。

そして、誰もあなたのDOMの要素に任意の文字列を書き込むことができなくなります：

const templateId = location.hash.match(/tplid=([^;&]*)/)[1];
// typeof templateId == "string"
document.head.innerHTML += templateId //     (TypeError).

これを行うには、特別な型付きオブジェクトを作成して使用する必要があります。これらのオブジェクトは、Trusted Type Policiesと呼ばれる特別に設計された関数を使用してのみ、アプリケーションで作成できます。

例えば：

const templatePolicy = TrustedTypes.createPolicy('template', {
  createHTML: (templateId) => {
    const tpl = templateId;
    if (/^[0-9a-z-]$/.test(tpl)) {
      return `<link rel="stylesheet" href="./templates/${tpl}/style.css">`;
    }
    throw new TypeError();
  }
});

const html = templatePolicy.createHTML(location.hash.match(/tplid=([^;&]*)/)[1]);
// html —    TrustedHTML
document.head.innerHTML += html;

ここでは、渡されたIDパラメーターをチェックして結果のHTMLを作成するテンプレートテンプレートのポリシーを作成します。 create *メソッドのセットは、対応するユーザー定義関数を呼び出し、結果をTrusted Typeオブジェクトにラップします。たとえば、templatePolicy.createHTMLはtemplateIdの検証関数を呼び出し、TrustedHTMLを含むタイプのオブジェクトを返します（最初の例を参照）。同時に、ブラウザーを使用すると、innerHTMLのTrustedHTMLタイプのオブジェクトにデータを書き込むことができます（理論的には通常のHTMLを想定しています）。

TrustedHTMLクラスに加えて、TrustedScript、TrustedURL、およびTrustedScriptURLクラスを使用できます。

新しいAPIの唯一の利点は、このチェックを追加できることだと思われるかもしれません。

if (/^[0-9a-z-]$/.test(tpl)) { /* allow the tplId */ }

実際、この行はXSSから保護するために必要です。しかし、実際には、すべてがはるかに興味深いです。信頼できるタイプを使用して、DOMの脆弱性を解決すると、ポリシーを適切に適用することができます。他のコードは、ソースからシンクへのデータをチェックしません。したがって、政治家との不適切な作業のみがセキュリティ上の問題を引き起こす可能性があります。この例では、ポリシーは常に最初に検証に合格したかどうかをチェックするため、templateIdの値がどこから来たかは問題ではありません。したがって、特定のポリシーが担当する領域では、XSS攻撃はすべて撃退されます。

Uncaught TypeError: Failed to set the ’innerHTML’ property on ’Element’: This document requires `TrustedHTML` assignment.

ポリシーの制限


Content-Security-Policyヘッダーで使用した*値に気づきましたか？これは、アプリケーションが任意の数のポリシーを作成できることを示します。ただし、それぞれに固有の名前が付いている必要があります。アプリケーションが自由に作成できるポリシーが多すぎると、混乱を招く可能性があり、DOM XSSに対する保護を実装することがより困難になります。

したがって、使用するポリシー名のリストを明示的に指定することで、この数を制限できます。例えば：

Content-Security-Policy: trusted-types template

これにより、名前テンプレートを使用して1つのポリシーのみが作成されます。その後、ソースコードで簡単に識別して、その動作をテストできます。これにより、アプリケーションがDOM XSSから確実に保護されるようになります。そして、あなたは幸せになります！

実際には、最新のWebアプリケーションには少数のポリシーが必要です。一般的な推奨事項は、スクリプトコード、HTMLテンプレートライブラリ、またはHTMLサニタイザで、クライアントコードがHTMLまたはURLを生成するポリシーを作成することです。DOMに関連しない多くの依存関係はすべて、ポリシーを使用する必要はありません。そして、Trusted Typeの適切な使用により、XSSから保護されます。
