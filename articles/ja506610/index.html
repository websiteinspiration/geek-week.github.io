WAL-G：PostgreSQLデータベースのバックアップとリカバリ

SQLダンプでバックアップを作成する（pg_dumpまたはpg_dumpallを使用する）ことは良い考えではないことは長い間知られています。 PostgreSQLをバックアップするには、WALログのバイナリコピーを作成するpg_basebackupコマンドを使用することをお勧めします。しかし、バックアップと復元を作成するプロセス全体を検討し始めると、これらがすべて機能し、上からも下からも痛みを引き起こさないように、少なくとも2輪車を書く必要があることがわかります。苦痛を和らげるために、WAL-Gが開発されました。

WAL-Gは、PostgreSQLデータベース（最近ではMySQL / MariaDB、MongoDB、FoundationDB）をバックアップおよび復元するためにGoで記述されたツールです）Amazon S3ストレージ（およびYandex Object Storageなどの類似物）だけでなく、Google Cloud Storage、Azure Storage、Swift Object Storage、およびファイルシステムとの連携をサポートしています。全体の設定は簡単な手順に削減されますが、それに関する記事がインターネット上に散らばっているという事実のため、開始と終了のすべての手順を含む完全なハウツーマニュアルはありません（Habréにはいくつかの投稿がありますが、多くの点がそこにありません）。



この記事は、主に知識を体系化するために書かれています。私はDBAではなく、どこかでphilistine-development言語で表現できるので、修正は大歓迎です！

それとは別に、Ubuntu 18.04上のPostgreSQL 12.3には以下のすべてが関連し、検証されていることに注意してください。すべてのコマンドは特権ユーザーとして実行する必要があります。

取り付け


この記事の執筆時点では、WAL-Gの安定版はv0.2.15（2020年3月）です。これを使用します（ただし、masterブランチから自分でアセンブルする場合は、githubリポジトリにこれに関するすべての手順が含まれています）。ダウンロードしてインストールするには、次のことを行う必要があります。

#!/bin/bash

curl -L "https://github.com/wal-g/wal-g/releases/download/v0.2.15/wal-g.linux-amd64.tar.gz" -o "wal-g.linux-amd64.tar.gz"
tar -xzf wal-g.linux-amd64.tar.gz
mv wal-g /usr/local/bin/

その後、まずWAL-Gを構成し、次にPostgreSQL自体を構成する必要があります。

WAL-Gの設定


バックアップストレージの例として、Amazon S3を使用します（サーバーに近く、使用が非常に安いため）。これを使用するには、「s3バケット」とアクセスキーが必要です。

WAL-Gに関するこれまでのすべての記事では環境変数を使用した構成を使用していましたが、このリリースから、設定はpostgresユーザーのホームディレクトリにある.walg.jsonファイルに配置できます。これを作成するには、次のbashスクリプトを実行します。

#!/bin/bash

cat > /var/lib/postgresql/.walg.json << EOF
{
    "WALG_S3_PREFIX": "s3://your_bucket/path",
    "AWS_ACCESS_KEY_ID": "key_id",
    "AWS_SECRET_ACCESS_KEY": "secret_key",
    "WALG_COMPRESSION_METHOD": "brotli",
    "WALG_DELTA_MAX_STEPS": "5",
    "PGDATA": "/var/lib/postgresql/12/main",
    "PGHOST": "/var/run/postgresql/.s.PGSQL.5432"
}
EOF
#    :
chown postgres: /var/lib/postgresql/.walg.json

すべての点で少し説明します。

  • WALG_S3_PREFIX-バックアップが注がれるS3バケットへのパス（ルートとフォルダーの両方で可能）。
  • AWS_ACCESS_KEY_ID – S3 ( – ReadOnly Policy! );
  • AWS_SECRET_ACCESS_KEY – S3;
  • WALG_COMPRESSION_METHOD – , Brotli ( /);
  • WALG_DELTA_MAX_STEPS – «» ( , , );
  • PGD​​ATA -データベースのデータのディレクトリへのパス（あなたが実行して見つけることができpg_lsclustersがコマンド）。
  • PGHOST-データベースへの接続。ローカルバックアップでは、この例のように、unix-socketを介して行う方が適切です。

その他のパラメーターは、ドキュメント（https://github.com/wal-g/wal-g/blob/v0.2.15/PostgreSQL.md#configuration）にあります。

PostgreSQLのセットアップ


データベース内のアーカイバがWALログをクラウドにアップロードしてそれらから回復するために（必要な場合）、構成ファイル/etc/postgresql/12/main/postgresql.confにいくつかのパラメータを設定する必要があります。開始するには、以下の設定が他の値に設定されていないことを確認する必要があります。これにより、構成を再起動したときにDBMSが落ちないようにします。これらのパラメータは、次を使用して追加できます。

#!/bin/bash

echo "wal_level=replica" >> /etc/postgresql/12/main/postgresql.conf
echo "archive_mode=on" >> /etc/postgresql/12/main/postgresql.conf
echo "archive_command='/usr/local/bin/wal-g wal-push \"%p\" >> /var/log/postgresql/archive_command.log 2>&1' " >> /etc/postgresql/12/main/postgresql.conf
echo “archive_timeout=60” >> /etc/postgresql/12/main/postgresql.conf
echo "restore_command='/usr/local/bin/wal-g wal-fetch \"%f\" \"%p\" >> /var/log/postgresql/restore_command.log 2>&1' " >> /etc/postgresql/12/main/postgresql.conf

#     SIGHUP    
killall -s HUP postgres

設定するパラメーターの説明：

  • wal_level -WALマガジンに書き込む情報の量、「レプリカ」-すべてを書き込むため。
  • archive_mode - archive_commandパラメータのコマンドを使用して、WALログのダウンロードを有効にします。
  • archive_command-完了したWALログをアーカイブするコマンド。
  • archive_timeout-ログのアーカイブはそれが完了したときにのみ行われますが、サーバーがデータベースにデータを少し変更/追加する場合、ここで秒単位で制限を設定することは理にかなっています。このパラメーターを本番環境で設定することを拒否しました）;
  • restore_command-バックアップからWALログを復元するコマンド。最後のデータベース変更が「完全バックアップ」（ベースバックアップ）にない場合に使用されます。

これらすべてのパラメーターの詳細については、公式ドキュメントの翻訳https://postgrespro.ru/docs/postgresql/12/runtime-config-walをご覧ください。

バックアップスケジュールの設定


好むと好まざるとにかかわらず、開始するにはcronが最も便利です。これは、バックアップを作成するために構成するものです。フルバックアップを作成するコマンドから始めましょう。wal-gでは、これはbackup-pushを実行するための引数です。ただし、最初に、すべてが正常であること（およびアクセスエラーがないこと）を確認するために、postgresユーザーからこのコマンドを手動で実行することをお勧めします。

#!/bin/bash

su - postgres -c '/usr/local/bin/wal-g backup-push /var/lib/postgresql/12/main'

開始引数は、データディレクトリへのパスを示します-pg_lsclustersを実行することで認識できることを思い出します。

すべてが順調に進み、データがS3リポジトリーにアップロードされた場合は、crontabで定期的な起動を構成できます。
#!/bin/bash

echo "15 4 * * *    /usr/local/bin/wal-g backup-push /var/lib/postgresql/12/main >> /var/log/postgresql/walg_backup.log 2>&1" >> /var/spool/cron/crontabs/postgres
#       
chown postgres: /var/spool/cron/crontabs/postgres
chmod 600 /var/spool/cron/crontabs/postgres

この例では、バックアッププロセスは毎日午前4時15分に開始されます。

古いバックアップを削除する


ほとんどの場合、中生代のすべてのバックアップを完全に保存する必要はないため、定期的にストレージを「クリーンアップ」すると便利です（「完全バックアップ」とWALログの両方）。すべてをcronタスクでも行います。

#!/bin/bash

echo "30 6 * * *    /usr/local/bin/wal-g delete before FIND_FULL $(date -d '-10 days' '+%FT%TZ') --confirm >> /var/log/postgresql/walg_delete.log 2>&1" >> /var/spool/cron/crontabs/postgres
#          (        )
chown postgres: /var/spool/cron/crontabs/postgres
chmod 600 /var/spool/cron/crontabs/postgres

Cronはこのタスクを毎日午前6時30分に実行し、過去10日間のコピーを除くすべて（完全バックアップ、デルタ、WAL）を削除しますが、指定した日付の前に少なくとも1つのバックアップを残し、日付以降の任意のポイントがPITRに分類されるようにします。

バックアップから復元


健全な基盤の保証が内部のデータの完全性の定期的な回復と検証であることは秘密ではありません。 WAL-Gを使用して回復する方法-このセクションで説明します。チェックについては後で説明します。

それとは別に、テスト環境でのリカバリ（すべて本番環境ではない）には注意が必要です。誤ってバックアップを上書きしないように、S3で読み取り専用アカウントを使用する必要があります。 WAL-Gの場合、S3ユーザーにグループポリシーで次の権限を設定する必要があります（効果：許可）：s3：GetObject、s3：ListBucket、s3：GetBucketLocation。そしてもちろん、事前にpostgresql.conf設定ファイルでarchive_mode = offを設定することを忘れないでくださいテストベースが静かにバックアップしたくない場合。PostgreSQLの全データ（ユーザーを含む）を削除

し、わずかな手の動きでリカバリーを行うため、以下のコマンドを実行する際は十分ご注意ください。

#!/bin/bash

#     (, pgbouncer),    ,       
service pgbouncer stop
#   ,     (, monit),        (   pgsql12)
monit stop pgsql12
#    
service monit stop
#    
service postgresql stop
#       (!!!);     ,      
rm -rf /var/lib/postgresql/12/main
#      
su - postgres -c '/usr/local/bin/wal-g backup-fetch /var/lib/postgresql/12/main LATEST'
#      -   (. https://postgrespro.ru/docs/postgresql/12/runtime-config-wal#RUNTIME-CONFIG-WAL-ARCHIVE-RECOVERY ),        postgres
su - postgres -c 'touch /var/lib/postgresql/12/main/recovery.signal'
#   ,     
service postgresql start

回復プロセスを確認したい人のために、bashマジックの小さな断片が以下に用意されているため、回復で問題が発生した場合、スクリプトはゼロ以外の終了コードでクラッシュします。この例では、信号ファイルが削除されたかどうかを確認するために、120秒のチェックが5秒のタイムアウト（回復にはわずか10分）で行われます（これは、回復が成功したことを意味します）。

#!/bin/bash

CHECK_RECOVERY_SIGNAL_ITER=0
while [ ${CHECK_RECOVERY_SIGNAL_ITER} -le 120 ]
do
    if [ ! -f "/var/lib/postgresql/12/main/recovery.signal" ]
    then
        echo "recovery.signal removed"
        break
    fi
    sleep 5
    ((CHECK_RECOVERY_SIGNAL_ITER+1))
done

#        ,    
if [ -f "/var/lib/postgresql/12/main/recovery.signal" ]
then
    echo "recovery.signal still exists!"
    exit 17
fi

回復が成功したら、すべてのプロセス（pgbouncer / monitなど）を再開することを忘れないでください。

復元後のデータ検証


リカバリ後にデータベースの整合性を確認してください。これにより、バックアップが破損したり、乱れたりすることがなくなります。そして、アーカイブを作成するたびにこれを行うのが良いですが、場所と方法は想像に依存します（個々のサーバーを1時間ごとに上げるか、CIでテストを実行できます）。しかし、少なくとも-データベースのデータとインデックスを確認する必要があります。

データをチェックするには、それらをダンプで実行するだけで十分ですが、データベースを作成するときにチェックサム（データチェックサム）を用意することをお勧めします。

#!/bin/bash

if ! su - postgres -c 'pg_dumpall > /dev/null'
then
    echo 'pg_dumpall failed'
    exit 125
fi

インデックスをチェックするために-amcheckモジュールがあり、WAL-Gテストからそれに対するSQLクエリを取得し、周りに小さなロジックを構築します。

#!/bin/bash

#  sql-       
cat > /tmp/amcheck.sql << EOF
CREATE EXTENSION IF NOT EXISTS amcheck;
SELECT bt_index_check(c.oid), c.relname, c.relpages
FROM pg_index i
JOIN pg_opclass op ON i.indclass[0] = op.oid
JOIN pg_am am ON op.opcmethod = am.oid
JOIN pg_class c ON i.indexrelid = c.oid
JOIN pg_namespace n ON c.relnamespace = n.oid
WHERE am.amname = 'btree'
AND c.relpersistence != 't'
AND i.indisready AND i.indisvalid;
EOF
chown postgres: /tmp/amcheck.sql

#          
# (       – )
cat > /tmp/run_amcheck.sh << EOF
for DBNAME in \$(su - postgres -c 'psql -q -A -t -c "SELECT datname FROM pg_database WHERE datistemplate = false;" ')
do
    echo "Database: \${DBNAME}"
    su - postgres -c "psql -f /tmp/amcheck.sql -v 'ON_ERROR_STOP=1' \${DBNAME}" && EXIT_STATUS=\$? || EXIT_STATUS=\$?
    if [ "\${EXIT_STATUS}" -ne 0 ]
    then
        echo "amcheck failed on DB: \${DBNAME}"
        exit 125
    fi
done
EOF
chmod +x /tmp/run_amcheck.sh

#  
/tmp/run_amcheck.sh > /tmp/amcheck.log

#         exit code  grep’ 
if grep 'amcheck failed' "/tmp/amcheck.log"
then
    echo 'amcheck failed: '
    cat /tmp/amcheck.log
    exit 125
fi
