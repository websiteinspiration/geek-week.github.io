コメント


コードにはコメントが必要です（コードを読む場合）。次のルールは、何にどのようにコメントすべきかを説明しています。ただし、コメントは非常に重要ですが、完璧なコードは自己文書化されています。型と変数に「話す」名前を使用することは、コメントで記述する必要があるあいまいな名前よりもはるかに優れています。

次の読者に基づいてコードにコメントしてください：コードを理解する必要があるプログラマ。次の読者になれることを忘れないでください！

コメントのスタイル


//または/ * * /の いずれかを使用して、均一性に違反します。//または/ ** /の

いずれかを使用できますが、//の方がはるかに望ましいです。ただし、コメントスタイルは常に既存のコードに合わせてください。

ファイルヘッダー内のコメント


各ファイルの上部にライセンスヘッダーを挿入します。

ファイル内のコメントは、その内容を説明する必要があります。ファイルが1つの抽象化（すでにコメントがある）を宣言、説明、またはテストする場合、ファイルのヘッダーに追加の説明は必要ありません。それ以外の場合は、ファイルの先頭にコンテンツの説明を挿入します。

法的情報と著者リスト


各ファイルには、ライセンス情報が含まれている必要があります。説明の形式は、プロジェクトで使用されるライセンスによって異なります。各ライセンス（Apache 2.0、BSD、LGPL、GPLなど）には独自の設計要件がある場合があります。

ファイルに大幅な変更を加える場合は、以前の著者リストを削除することを検討してください。更新されたファイルには、著作権に関する言及と著者のリストが含まれなくなっている可能性があります。

ファイルの内容


.hがいくつかの抽象化を宣言している 場合、ファイルのヘッダーのコメントは、一般にファイルの内容と抽象化が相互にどのように関連しているかを説明する必要があります。通常、コメントの1、2文で十分です。より詳細な情報は、他の場所で署名されます（ファイルのヘッダーではありません）。.hおよび.ccファイル

内のコメントを複製しないでください。コメントは時間とともに変化します。

クラスのコメント


各クラス宣言（非常に明白なものを除く）には、クラスの目的と使用方法に関するコメントを付ける必要があります。

//   GargantuanTable.
// :
//    GargantuanTableIterator* iter = table->NewIterator();
//    for (iter->Seek("foo"); !iter->done(); iter->Next()) {
//      process(iter->key(), iter->value());
//    }
//    delete iter;
class GargantuanTableIterator {
  ...
};

クラスの解説は、クラスを使用する方法と時期、クラスを正しく使用するための追加要件を理解するのに十分なはずです。必要に応じて、クラスでの同期に関する制限（前提）を説明します。クラスのインスタンスを異なるスレッドから使​​用できる場合は、マルチスレッドで使用するためのルールを必ず書き留めてください。

また、クラスのコメントに短いコード例を提供して、クラスの使用がいかに簡単かを示すこともできます。

通常、クラスは異なるファイル（.hおよび.cc）で宣言/定義されます。クラスの使用を説明するコメントは、インターフェース定義の隣にある必要があります。実装の複雑さに関するコメントは、メソッド自体のコードに近いはずです。

関数コメント


関数宣言に関するコメントは、関数の使用法を説明する必要があります（最も明白な場合を除く）。関数定義に関するコメントは、実装について説明しています。

関数宣言


各関数の宣言には、コメント（宣言の直前）、関数の機能、およびその使用方法が必要です。コメントが省略できるのは、関数が単純でその使用法が明らかな場合のみです（たとえば、変数値を減算する関数など）。気分を盛り上げてコメントを開始してみてください（「ファイルを開く」）。命令ムード（「ファイルを開く」）の使用はお勧めしません。コメントは、関数の本質ではなく、関数の本質を説明しています。

関数宣言のコメントでは、次の点に注意してください。

  • 結果として返される関数の入力に供給されるもの。
  • クラスのメンバー関数の場合：インスタンスが引数への参照を保持するかどうか、
    メモリを解放する必要があるかどうか。
  • 関数は、呼び出しコードが削除する必要があるメモリを割り当てますか？
  • nullptr引数はありますか？
  • （アルゴリズム）関数の複雑さ。
  • 異なるスレッドからの呼び出しは同時に許可されますか？同期についてはどうですか？

例：

//    .    
//   .   
//    GargantuanTable  .
//
//      .
//
//    :
//    Iterator* iter = table->NewIterator();
//    iter->Seek("");
//    return iter;
//         ,
//    NewIterator()     .
Iterator* GetIterator() const;

ただし、当たり前のものを噛んではいけません。

オーバーロードされた関数を文書化する場合、元の関数と比較して、頑固な変更を加えます。変更がない場合（頻繁に発生します）、追加のコメントはまったく必要ありません。

コンストラクタとデストラクタについてコメントするときは、コードリーダーがその目的を知っていることに注意してください。したがって、「このオブジェクトを破壊する」タイプの解説はばかげています。コンストラクターが引数を使用して何を行うか（たとえば、ポインターの所有権を変更する）、またはデストラクターがどのようなクリーニング操作を行うかを説明できます。すべてが明確な場合は、何もコメントしないでください。通常、デストラクタには通常、コメントはありません（宣言されている場合）。

関数定義


関数の実装にトリックがある場合は、定義に説明コメントを追加できます。その中で、コードでトリックを記述し、計算のすべての段階の概要を示し、これまたはその実装の選択を説明することができます（特に、より良い代替案がある場合）。コードビットの同期の原理を説明できます（ここではブロックしますが、ここでは魚をラップします）。（.hファイルなどからの）関数の宣言に関するコメントは繰り返さない

でください。関数の機能を簡単に説明することはできますが、主な機能は、その方法を説明することです。

変数に関するコメント


良い意味で、変数名はその内容と理由をすぐに伝える必要がありますが、場合によっては追加のコメントが必要になります。

クラスデータメンバー


クラスの各メンバーの目的は明白である必要があります。明らかでない微妙な点（特別な意味、他のメンバーとのつながり、存続期間の制限）がある場合、これらすべてにコメントする必要があります。ただし、タイプと名前で十分な場合は、コメントを追加する必要はありません。

一方、特別な（そして明白ではない）値（nullptrまたは-1）の説明は役に立ちます。例えば：

private:
 //      
 // -1 - ,        
 int num_total_entries_;

グローバル変数


すべてのグローバル変数は、その目的、および（明確でない場合は）なぜグローバルでなければならないかについてコメントする必要があります。次に例を示します。

//   ,    
const int kNumTestCases = 6;

実装に関するコメント


非自明で興味深く、重要なコードの場合は、関数またはアルゴリズムの実装についてコメントしてください。

説明コメント


複雑または非標準のコードブロックの前には、コメントを付ける必要があります。例えば：

//    2.  x   
for (int i = 0; i < result->size(); ++i) {
  x = (x << 8) + (*result)[i];
  (*result)[i] = x >> 1;
  x &= 1;
}

行コメント


コメント（通常は行の最後にあります）を使用して、コードの行に明白でない意味を追加することをお勧めします。このコメントは、コードから2つのスペースで区切る必要があります。例えば：

//   ,   
mmap_budget = max<int64>(0, mmap_budget - index_->length());
if (mmap_budget >= data_size_ && !MmapData(mmap_chunk_bytes, mlock))
  return;  //

コードブロックには2つのコメントがあることに注意してください。1つはコードの機能を説明し、もう1つは関数からの戻りがある場合、エラーがすでにログにあることを思い出させます。

してはいけないこと


明白な説明はしないでください。特に、C ++を知っている人には明らかなことを説明する必要はありません。代わりに、このコードがそうする理由を説明できます（または、コードを自己記述的にすることもできます）。

比較：

//    .  <-- :  !
auto iter = std::find(v.begin(), v.end(), element);
if (iter != v.end()) {
  Process(element);
}

これとともに：

//  (Process) "element"    
auto iter = std::find(v.begin(), v.end(), element);
if (iter != v.end()) {
  Process(element);
}

自己記述コードにはコメントはまったく必要ありません。
上記のコードに関するコメントは、一般的に明白である可能性があります（必要ではありません）。

if (!IsAlreadyProcessed(element)) {
  Process(element);
}

句読点、スペル、文法


句読点、スペル、文法に注意してください。正しく書かれたコメントを読む方がはるかに簡単です。

コメントはストーリーとして書く必要があります。大文字と句読点を正しく配置してください。ほとんどの場合、完全な文はフレーズの断片よりも理解しやすいです。行ごとなどの短いコメントは、あまり正式ではないかもしれませんが、それでも一般的なスタイルに従う必要があります。

コードレビュー担当者がセミコロンの代わりにカンマを使用することに過度に重点を置くことは、少し煩わしく感じる可能性がありますが、高いレベルの読みやすさと理解しやすさを維持することが重要です。適切な句読点、スペル、文法がこれに大きく貢献しています。

コメントTODO


一時的なコードまたは十分な（中間ではなく完全な）ソリューションにはTODO コメントを使用します。

コメントにはTODO行（すべて大文字）を含め、その後に名前、電子メールアドレス、不具合ID、またはTODOが記述されている問題の性質と開発者を識別するためのその他の情報を含めます。この説明の目的は、後で詳細を見つけることができるようにすることです。TODOと説明指定されたプログラマが問題を修正するという意味ではありません。したがって、TODOを作成するときの通常の名前は自分の名前です。

// TODO(kl@gmail.com):  "*"  .
// TODO(Zeke)   .
// TODO(bug 12345):   " ".

TODOが「将来は別の方法で実行しよう」のように見える 場合は、特定の日付（「2005年11月に修正」）またはイベント（「すべてのクライアントがXML要求を処理するときにそのコードを削除する」）を指定します。
