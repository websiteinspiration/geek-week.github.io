Hardware


So, to create a minimally working prototype, we need:

  1. A piece of board or anything you can attach all the components to
  2. Almost any microcontroller. I took arduino uno as the easiest option
  3. GPS module (we take the date, time and coordinates from it). Theoretically, you can take the real-time clock module instead, but then you will have to enter your coordinates manually and the device will not work out of the box, but the time of the cold start will be greatly reduced
  4. Two servos, or better yet two stepper motors
  5. Laser pointer
  6. A different trifle: a soldering iron, hot-melt adhesive, a breadboard, wires, a button, a capacitor, and direct hands.

There are no tricks to assemble: just fix all the components on the base with hot melt adhesive.
Just try to ensure that the axes of the servos and the laser are as perpendicular to each other as possible, this will reduce the pointing error.

Connection


Nothing complicated here either:

  • GPS is connected via uart (only Rx is needed, since we do not need to send anything to the module
  • servos - in pins 10 (azimuth axis) and 11 (height axis)
  • button - 2 pin
  • power to all modules
  • optional - power capacitor

void GpsGetData()
{
  while (Serial.available() > 0)
    gps.encode(Serial.read());
  if (gps.location.isUpdated())
  {
    phi = gps.location.lat(); //   (double)
    lambda = gps.location.lng(); //    (double)
    yr = gps.date.year(); //  (2000+) (u16)
    mo = gps.date.month(); //  (1-12) (u8)
    d = gps.date.day(); //  (1-31) (u8)
    h = gps.time.hour(); //  (0-23) (u8)
    m = gps.time.minute(); //  (0-59) (u8)
    s = gps.time.second(); //  (0-59) (u8)
  }
}
void CalculateParams()
{
  float _time = (h + (float)m / 60 + (float)s / 3600) * 360.0 / 24.0;//   
  float sunlambda = (float)Days(d, mo, yr) * 360 / 365; //      
  sundelta = asin(0.398749 * sin(sunlambda * 2 * PI / 360));//    
  sunalpha = asin(tan(sundelta) / 0.434812); //      
  float tS = _time + lambda - 180 + sunalpha * 360 / (2 * PI) - alpha; //     
  height = 360 / 2 / PI * asin(sin(phi * 2 * PI / 360) * sin(delta * 2 * PI / 360) + cos(phi * 2 * PI / 360) * cos(delta * 2 * PI / 360) * cos(tS * 2 * PI / 360));// 
  az = 360 / 2 / PI * asin(sin(tS * 2 * PI / 360) * cos(delta * 2 * PI / 360) / cos(height * 2 * PI / 360)); //   (   )
  if (tS > 90 || tS < -90)
  {
    az = 180 - az;
  }
 // if (az > 180 && az < 270)

}

void setup() {
  Serial.begin(9600);
  yaw.attach(10, 90);
  pitch.attach(11, 180);
  yaw.setSpeed(20);
  yaw.setAccel(0.1);
  pitch.setSpeed(20);
  pitch.setAccel(0.1);
  yaw.setAutoDetach(false);
  pitch.setAutoDetach(false);
  pinMode(LED_BUILTIN, OUTPUT);
}

void loop() {
  but.tick();
  yaw.tick();
  pitch.tick();
  if (but.isSingle())
  {
    counter += 1;
    if (counter == 9)
      counter = 0;
  }
  alpha = Coordinates[counter][0];
  delta = Coordinates[counter][1];
  GpsGetData();
  CalculateParams();
  digitalWrite(LED_BUILTIN, gps.location.isValid());
  if (az > 90 && az < 180)
  {
    yaw.setTargetDeg(270 - az);
    pitch.setTargetDeg(height);
  }
  if (az < -90 && az > -180)
  {
    yaw.setTargetDeg(-90 - az);
    pitch.setTargetDeg(height);
  }
  if (az < 90 && az > -90)
  {
    yaw.setTargetDeg(90.0 - az);
    pitch.setTargetDeg(180 - height);
  }
}

Astronomy lesson


About astrotermins and something else

What this function should do: accept the coordinates of the star in the equatorial system (right ascension and declination), the time and coordinates of the observer, and give the height and azimuth of the object, i.e. essentially translate the coordinates of the star from the equatorial system (in which the stars are motionless) to horizontal (in which the stars move during the day).

This is realized using the formulas of spherical trigonometry , as well as spherical astronomy in a vacuum .

The algorithm is as follows:

  1. calculate the ecliptic longitude of the sun
  2. calculate sun declination
  3. calculate direct sun rise
  4. calculate the hour angle of a star
  5. calculate altitude and azimuth

Here's how it is implemented:

Error estimation


Let us evaluate the contribution of each factor to the error (sorted by contribution to inaccuracy)

  1. Hand curvature - this is the most important factor for me
  2. Inaccurate horizon position
  3. inaccurate position of zero azimuth (I recall that astronomers count the azimuth from the south clockwise)
  4. The fact that servos cannot rotate at a fractional angle
  5. inaccuracies in the translation algorithm (the position of the sun is not determined very accurately: for example, the ecliptic longitude of the sun is considered much more complicated (a simplified version in the code); the day of the vernal equinox does not always occur at the same time;
    also, I do not take into account the time equation) but the error from - for this, a small leak.
