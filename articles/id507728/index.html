Baiklah,



mari kita mulai: Kami membuat generator sumbu tanggal dalam rentang tertentu (biarkan itu menjadi fungsi bernilai tabel inline, untuk kemungkinan digunakan kembali dalam aplikasi yang berbeda): 



CREATE FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq
	FROM DR
)


Kami memeriksa: 



SELECT * FROM dbo.TF_Sys_DateGen('2020-06-20', '2020-07-01')

Namun, ada halangan kecil: jumlah hari yang dikembalikan oleh fungsi DATEPART bawaan tergantung pada nilai parameter DATEFIRST, dan nilai defaultnya, pada gilirannya, tergantung pada pengaturan regional server. Oleh karena itu, berharap bahwa kita memiliki sistem Amerika, dan minggu dimulai pada hari Minggu, umumnya tidak patriotik .



Apa yang bisa dilakukan untuk menyiasati ini? Misalnya, baca nilai DATEFIRST saat ini melalui variabel @@ DATEFIRST (yang mengembalikan jumlah hari pertama relatif terhadap presentasi default untuk Amerika Serikat, di mana hari nomor 1 adalah Minggu (!)) Dan kemudian memperoleh angka hari dalam notasi saat ini berdasarkan nilai saat ini @@ DATEFIRST sesuai dengan hari Sabtu dan Minggu. Luangkan waktu sejenak dan cobalah untuk melakukannya di pikiran Anda.

Saya memutuskan untuk tidak mengambil menit lagi dari bantuan klub, dan melakukannya secara berbeda - saya menerima hari-hari dalam minggu ini dalam notasi saat ini dari tanggal di mana saya tahu apriori bahwa itu adalah hari Sabtu dan Minggu - misalnya, 1 dan 2 Januari 2000. Dan kemudian dia hanya membandingkan jumlah hari yang dikeluarkan dengan ini apriori yang dikenal dalam konteks saat ini: 



ALTER FUNCTION [dbo].[TF_Sys_DateGen](@MinDate DATE, @MaxDate DATE)
RETURNS TABLE AS
RETURN (

	WITH DR AS (
		
		SELECT @MinDate AS DateSeq
		
		UNION ALL
		
		SELECT DATEADD(DY, 1, R.DateSeq)
		FROM DR R
		WHERE DATEADD(DY, 1, R.DateSeq) <= @MaxDate
	
	)

	SELECT 
		DR.DateSeq,
		CONVERT(BIT, CASE WHEN DATEPART(WEEKDAY, DR.DateSeq) IN (Q.DOW_Sat, Q.DOW_Sun) THEN 1 ELSE 0 END) AS IsHoliday
	FROM DR
	CROSS JOIN (
		SELECT TOP (1)
			DATEPART(WEEKDAY, '1/1/2000')	AS DOW_Sat,	--  
			DATEPART(WEEKDAY, '1/2/2000')	AS DOW_Sun	--  
	) Q

)


Orang mungkin bertanya, mengapa semua CROSS JOIN / SELECT TOP (1) ini ... dicampur ke dalam permintaan? Tapi untuk apa: karena TVF saya inline, dan fungsi tubuh pada dasarnya adalah satu SELECT, Anda tidak dapat memasukkan variabel apa pun - semua yang kita bisa andalkan terutama pada SELECT dapat berupa hardcode (dan inilah yang ingin kita hindari ), atau output ke subquery.



Karenanya, kami menggunakan CROSS JOIN (Anda bisa INNER ... ON 1 = 1, sesuka Anda).



TOP (1) di sini kita membutuhkan lebih banyak mesin SQL daripada kita sendiri (setelah semua, kita mengerti bahwa dalam subquery selalu ada satu baris!), Tetapi mereka mengatakanbahwa untuk konstruksi yang sama TOP (1) memberi tahu kompiler kueri bahwa subquery perlu dieksekusi satu kali sebelum yang utama, dan tidak seperti CROSS BERLAKU untuk setiap baris SELECT utama. Saya tidak 100% yakin akan hal ini, terutama jika tubuhnya subquery non-deterministik (saya akan memeriksanya entah bagaimana), tetapi biarlah demikian untuk saat ini.
