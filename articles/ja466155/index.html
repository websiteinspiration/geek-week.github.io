Tarantool Data Gridのアーキテクチャと機能



2017年、アルファバンクの投資ビジネスのトランザクションコアの開発競争を勝ち取り、働き始めました（HighLoad ++ 2018で、アルファバンクの投資ビジネスのトランザクションコアの責任者であるウラジミールドリンキンが投資ビジネスのコアについてプレゼンテーションを行いました）。このシステムは、さまざまなソースからのトランザクションデータをさまざまな形式で集約し、データを統一された形式にして保存し、それらへのアクセスを提供するためのものでした。

開発の過程でシステムが進化して機能するようになり、ある時点で、厳密に定義された範囲のタスクを解決するために設計されたアプリケーションソフトウェアだけでなく、それ以上のものを結晶化することに気づきました。永続的ストレージを備えた分散アプリケーションを構築するためのシステム。私たちの経験が新製品の基礎を形成しました-Tarantool Data Grid（TDG）。

TDGアーキテクチャと開発プロセスで得られたソリューションについて話し、基本的な機能を紹介し、製品が完全なソリューションを構築するための基盤になる方法を示したいと思います。

アーキテクチャ的には、システムを個別の役割に分割しました。各役割は、特定の範囲のタスクを解決する役割を果たします。アプリケーションの1つの実行中のインスタンスは、1つ以上のタイプのロールを実装します。クラスタは同じタイプのいくつかの役割を持つことができます：



コネクタ


コネクタは、外界との通信を担当します。そのタスクは、要求を受け入れ、それを解析し、それが成功した場合は、処理のためにデータを入力プロセッサーに送信することです。 HTTP、SOAP、Kafka、FIX形式をサポートしています。アーキテクチャーにより、新しいフォーマットのサポートを簡単に追加できます。IBMMQのサポートは近日提供予定です。リクエストの解析が失敗した場合、コネクタはエラーを返します。それ以外の場合は、その後の処理中にエラーが発生した場合でも、要求は正常に処理されたと返信します。これは、リクエストを繰り返す方法がわからないシステムで作業するために、またはその逆の場合、意図的に行われます。データを失わないようにするために、修復キューが使用されます。オブジェクトは最初にデータを入力し、処理が成功した後でのみ削除されます。管理者はオブジェクトに関するアラートを受け取ることができます。修復キューに残っており、ソフトウェアエラーまたはハードウェア障害を解消した後、再試行してください。

入力プロセッサ


入力プロセッサは、受信したデータを特性別に分類し、適切なハンドラを呼び出します。ハンドラーはサンドボックスで実行されるLuaコードであるため、システムの機能に影響を与えることはできません。この段階で、データを目的の形式にして、必要に応じて、必要なロジックを実装できる任意の数のタスクを実行できます。たとえば、Tarantool Data Grid上に構築されたMDM（マスターデータ管理）製品では、新しいユーザーを追加するときに、リクエストの処理が遅くならないようにするために、個別のタスクとしてゴールドレコードの作成を開始します。サンドボックスは、データの読み取り、変更、追加のリクエストをサポートします。これにより、ストレージなどのすべてのロールでいくつかの機能を実行し、結果を集計できます（マップ/縮小）。

ハンドラーはファイルで記述できます：

sum.lua

local x, y = unpack(...)
return x + y

そして、設定で宣言します：

functions:
  sum: { __file: sum.lua }

なぜルア？Luaは非常に単純な言語です。私たちの経験に基づいて、彼に会ってから数時間後、人々は問題を解決するコードを書き始めます。そして、これらはプロの開発者だけでなく、たとえばアナリストでもあります。さらに、jitコンパイラーのおかげで、Luaは非常に高速です。

ストレージ


ストレージは永続的なデータを保存します。保存する前に、データはデータスキーマに準拠しているかどうか検証されます。スキームを説明するために、拡張Apache Avro形式を使用します。例：

{
    "name": "User",
    "type": "record",
    "logicalType": "Aggregate",
    "fields": [ 
        { "name": "id", "type": "string"}, 
        {"name": "first_name", "type": "string"}, 
        {"name": "last_name", "type": "string"} 
    ], 
    "indexes": ["id"] 
}

この記述に基づいて、Tarantula DBMS用のDDL（データ定義言語）とデータアクセス用のGraphQLスキーマが自動的に生成されます。

非同期データ複製がサポートされています（同期を追加する計画）。

出力プロセッサー


新しいデータの到着を外部の消費者に通知する必要がある場合があります。これには、出力プロセッサの役割があります。データを保存した後、それらを適切なハンドラーに転送して（例えば、コンシューマーが必要とするフォームにデータを取り込むために）、コネクターに転送して送信できます。修復キューもここで使用されます。オブジェクトを誰も受け入れなかった場合、管理者は後で再試行できます。

スケーリング


コネクタ、入力プロセッサ、出力プロセッサの役割はステートレスであるため、システムを水平方向にスケーリングでき、必要なタイプの役割が含まれるアプリケーションの新しいインスタンスを追加するだけです。水平ストレージスケーリングでは、仮想バケットを使用してクラスターを編成するアプローチが採用されています。新しいサーバーを追加すると、バックグラウンドで古いサーバーのバケットの一部が新しいサーバーに移動します。これはユーザーに対して透過的に行われ、システム全体の操作には影響しません。

データのプロパティ


オブジェクトは非常に大きく、他のオブジェクトを含むことができます。データの追加と更新の原子性を確保し、1つの仮想バケットにすべての依存関係を持つオブジェクトを保存します。これにより、複数の物理サーバーにまたがるオブジェクトの「スミアリング」が解消されます。

バージョン管理がサポートされています。オブジェクトを更新するたびに新しいバージョンが作成され、いつでもタイムスライスを作成して、世界がどのように見えるかを確認できます。長い履歴を必要としないデータの場合、バージョンの数を制限したり、1つだけを保存したりすることもできます。つまり、特定のタイプのバージョン管理を実際に無効にします。また、履歴を時間で制限することもできます。たとえば、1年より古い特定のタイプのオブジェクトをすべて削除します。アーカイブもサポートされています。指定した時間より古いオブジェクトをアンロードして、クラスター内のスペースを解放できます。

タスク


興味深い機能のうち、スケジュールに従って、ユーザーの要求に応じて、またはサンドボックスからプログラムでタスクを実行できることは注目に値します。



ここで、別の役割-ランナーを確認します。この役割には状態がなく、必要に応じて、この役割を持つ追加のアプリケーションインスタンスをクラスターに追加できます。ランナーの責任は、タスクを完了することです。前述のように、サンドボックスから新しいタスクを作成することは可能です。それらはストレージでキューに入れられ、ランナーで実行されます。このタイプのタスクはジョブと呼ばれます。タスクと呼ばれるタイプのタスクもあります。これらは、（cron構文を使用して）実行するか、またはオンデマンドで実行するようにスケジュールされたユーザー定義のタスクです。このようなタスクを実行および追跡するために、便利なタスクマネージャがあります。この機能を使用できるようにするには、スケジューラの役割を有効にする必要があります。この役割には状態があるため、スケーリングされませんが、これは必須ではありません。ただし、他のすべてのロールと同様に、彼女は機能し始めるレプリカを持つことができます。マスターが突然拒否した場合。

ロガー


別の役割はロガーと呼ばれます。すべてのクラスターメンバーからログを収集し、Webインターフェイスを介してログをアップロードおよび表示するためのインターフェイスを提供します。

サービス


このシステムによりサービスの作成が容易になることは言及する価値があります。設定ファイルでは、サンドボックスで実行されているユーザー作成ハンドラーに送信する必要があるリクエストを指定できます。このハンドラでは、たとえば、ある種の分析クエリを実行して結果を返すことができます。

サービスは設定ファイルに記述されています：

services:
   sum:
      doc: "adds two numbers"
      function: sum
      return_type: int
      args:
         x: int
         y: int

GraphQL APIが自動的に生成され、サービスが呼び出し可能になります。

query {
   sum(x: 1, y: 2) 
}

これによりsum、結果を返すハンドラーコールが発生します。

クエリプロファイリングとメトリック


システムとクエリプロファイリングを理解するために、OpenTracingプロトコルのサポートを実装しました。システムは、オンデマンドで、このプロトコルをサポートするツール（Zipkinなど）に情報を送信できます。これにより、リクエストがどのように実行されたかを理解できます。



当然、システムは、Prometheusを使用して収集し、Grafanaを使用して視覚化できる内部メトリックを提供します。

配置する


Tarantool Data GridはRPMパッケージまたはアーカイブからデプロイでき、配信またはAnsibleのユーティリティを使用して、Kubernetes（Tarantool Kubernetes Operator）もサポートされています。

ビジネスロジック（構成、プロセッサ）を実装するアプリケーションは、UIを介して、または提供されているAPIを介してスクリプトを使用して、Tarantool Data Gridクラスタにアーカイブとしてロードされます。
