方法1.明白で、明らかに最良ではない


C ++でも使用される最も一般的な方法は、製造元のヘッダーファイルのレジスタ構造の説明を使用することです。デモでは、STM32F411マイクロコントローラの2つのポートAレジスタ（ODR-出力データレジスタとIDR-入力データレジスタ）を使用して、「刺しゅう」「Hello world」を実行してLEDを点滅させます。

int main() {
  GPIOA->ODR ^= (1 << 5) ;
  GPIOA->IDR ^= (1 << 5) ; //,     
}

ここで何が起こり、この設計がどのように機能するかを見てみましょう。マイクロプロセッサのヘッダーには、GPIO_TypeDefこの構造へのポインタの構造と定義がありますGPIOA。次のようになります。

typedef struct
{
  __IO uint32_t MODER;   //port mode register,  Address offset: 0x00      
  __IO uint32_t OTYPER;  //port output type register,  Address offset: 0x04
  __IO uint32_t OSPEEDR; //port output speed register,  Address offset: 0x08
  __IO uint32_t PUPDR;   //port pull-up/pull-down register, Address offset: 0x0C
  __IO uint32_t IDR;     //port input data register,  Address offset: 0x10 
  __IO uint32_t ODR;     //port output data register, Address offset: 0x14
  __IO uint32_t BSRR;    //port bit set/reset register, Address offset: 0x18
  __IO uint32_t LCKR;    //port configuration lock register, Address offset: 0x1C
  __IO uint32_t AFR[2];  //alternate function registers, Address offset: 0x20-0x24
} GPIO_TypeDef;

#define PERIPH_BASE     0x40000000U //Peripheral base address in the alias region  
#define AHB1PERIPH_BASE (PERIPH_BASE + 0x00020000U)
#define GPIOA_BASE          (AHB1PERIPH_BASE + 0x0000U)

#define GPIOA             ((GPIO_TypeDef *) GPIOA_BASE)

単純な人間の言葉で言えば、GPIO_TypeDefアドレスの「型」の構造全体が構造のGPIOA_BASE特定のフィールドを参照するとき、基本的にこの構造のアドレス+この構造の要素へのオフセットに目を向けます。削除する#define GPIOAと、コードは次のようになります。

((GPIO_TypeDef *) GPIOA_BASE)->ODR ^= (1 << 5) ;
((GPIO_TypeDef *) GPIOA_BASE)->IDR ^= (1 << 5) ; //

C ++プログラミング言語に関連して、整数アドレスは構造体型へのポインターに変換されますGPIO_TypeDef。ただし、C ++では、C変換を使用すると、コンパイラーは次の順序で変換を実行しようとします。

  • const_cast
  • static_cast
  • const_castの横にあるstatic_cast、
  • reinterpret_cast
  • const_castの隣のreinterpret_cast

それら。コンパイラーがconst_castを使用してタイプを変換できなかった場合、static_castなどを適用しようとします。その結果、呼び出しは：

((GPIO_TypeDef *) GPIOA_BASE)->ODR ^= (1 << 5) ;

他には何もありません： 

reinterpret_cast<GPIO_TypeDef *> (GPIOA_BASE)->ODR ^= (1 << 5) ;

実際、C ++アプリケーションの場合、次のように構造体をアドレスに「プル」するのが適切です。

GPIO_TypeDef * GPIOA{reinterpret_cast<GPIO_TypeDef *>(GPIOA_BASE)} ;

いずれにせよ、型変換のために、C ++のこのアプローチには大きなマイナスがあります。これはreinterpret_cast、constexprデザイナーや関数、テンプレートパラメーターのいずれにも使用できないため、マイクロコントローラーのC ++機能の使用が大幅に削減されます。
例を挙げて説明します。そうすることが可能です：

  const int a;
  const int b;
} ;

template<Test* mystruct>
constexpr const int Geta() {
  return mystruct->a;
}

Test test{1,2};
int main() {
  Geta<&test>() ;
}

しかし、これはすでに不可能です。

 
template<GPIO_TypeDef * mystruct>
constexpr volatile uint32_t GetIdr() {

int main() {
//GPIOA   reinterpret_cast<GPIO_TypeDef *> (GPIOA_BASE)

したがって、このアプローチを直接使用すると、C ++の使用に大きな制限が課されます。GPIOA言語の手段を使用して、ROM内のポインターを使用するオブジェクトを見つけることができず、そのようなオブジェクトのメタプログラミングを利用できません。
さらに、一般的に、この方法は安全ではありません（西側のパートナーによると）。結局のところ、いくつかのNON-FUNNESSを作ることは可能である。

方法3.明白で、明らかにより正しい


レジスタへのアクセスが構造体フィールドを介して行われる場合、構造体オブジェクトへのポインタの代わりに、構造体の整数アドレスを使用できます。構造体のアドレスは、製造元のヘッダーファイル（たとえば、GPIOAのGPIOA_BASE）にあるため、覚えておく必要はありませんが、テンプレートやconstexpr式で使用して、構造体をこのアドレスに「オーバーレイ」できます。

        //     addr

        GpioPort->ODR ^= (1 << pinNum) ;
      }
  };
int main() {
  using Led1 =  Pin<GPIOA_BASE, 5> ;

方法4.外旋ラップ


理解しやすいコードの愛好家のために、レジスターのラッパーを作成して、それらにアクセスすることを便利で便利にし、コンストラクターを作成し、演算子を再定義することができます。

class Register  {
    public:
      explicit Register(uint32_t addr) : ptr{ reinterpret_cast<volatile uint32_t *>(addr) } {
      }

      __forceinline inline Register& operator^=(const uint32_t right)  {
        *ptr ^= right;
        return *this;
      }

    private:
      volatile uint32_t *ptr; //   
  };

int  main() {
    Register Odr{GpioaOdrAddr};
    Odr ^= (1 << 5);
    Register Idr{GpioaIdrAddr};
    Idr ^= (1 << 5); //
}

方法7.愚かさを取り除く

  struct WriteReg {};
  struct ReadReg {};
  struct ReadWriteReg: public WriteReg, public ReadReg {};

これで、レジスタを書き込み用に設定でき、レジスタは読み取り専用です。

template<uint32_t addr, typename RegisterType>
  class Register

     //       WriteReg,

      Register &operator^=(const uint32_t right)

構造体の先頭に対するレジスターのシフトを見つけるには、構造体のメンバーへのポインターを使用できます。volatile uint32_t T::*memberこれは、構造体のメンバーの先頭からの相対オフセットをバイト単位で返します。たとえば、構造体GPIO_TypeDefがある場合、アドレス&GPIO_TypeDef::ODRは0x14になります。
この機会を打って、コンパイラーを使用してメソッド9からレジスターのアドレスを計算します。

struct WriteReg {};
  struct ReadReg {};
  struct ReadWriteReg: public WriteReg, public ReadReg {};

  template<uint32_t addr, typename T, volatile uint32_t T::*member, typename RegType>
  class Register {
    public:
      __forceinline template <typename T1 = RegType,
        class = typename std::enable_if_t<std::is_base_of<WriteReg, T1>::value>>
      inline static void Xor(const uint32_t mask)
      {
        reinterpret_cast<T*>(addr)->*member ^= mask ;
      }
  };

  template<uint32_t addr, typename T>
  struct Gpio
  {
    using Moder = Register<addr, GPIO_TypeDef, &GPIO_TypeDef::ODR, ReadWriteReg>;
    using Otyper = Register<addr, GPIO_TypeDef, &GPIO_TypeDef::OTYPER, ReadWriteReg>;
    using Ospeedr = Register<addr, GPIO_TypeDef, &GPIO_TypeDef::OSPEEDR, ReadWriteReg>;
    using Pupdr = Register<addr, GPIO_TypeDef, &GPIO_TypeDef::PUPDR, ReadWriteReg>;
    using Idr = Register<addr, GPIO_TypeDef, &GPIO_TypeDef::IDR, ReadReg>;
    using Odr = Register<addr, GPIO_TypeDef, &GPIO_TypeDef::ODR, WriteReg>;
  } ;

ボーナス Phytonを使用した言語拡張とparsimコードを紹介します


C ++でレジスタを操作する問題は、かなり以前からありました。人々はさまざまな方法でそれを解決します。もちろん、言語がコンパイル時にクラスの名前を変更するようなものをサポートしていたら素晴らしいでしょう。ええと、仮に次のような場合はどうでしょう。

   __forceinline  inline static void Xor(const uint32_t mask)  {

%% set port = gpio["port"] | upper
