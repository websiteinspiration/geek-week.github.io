Zabbix 5.0、またはIPMIによるテンプレートサーバーの新機能



機器を監視下に置く必要があり、お気に入りのZabbixシステムには、このタイプの機器用の既製のテンプレートはありません。身近な状況？誰もが自分のやり方でそれから抜け出します。1人の管理者がインターネットで解決策を探しています。2つ目は、独自の開発です。そして、いくつかはこの仕事をあきらめるでしょう。これで、Zabbixチームは新しいリリースごとに、システム内の事前定義されたテンプレートのセットを拡張します。たとえば、次のバージョン5.0では、IPMIを介してサーバーを監視するための新しいユニバーサルテンプレート-IPMIによるテンプレートサーバーが表示されます。同僚は、さまざまなメーカーの機器での彼の作業のデバッグについて支援を求めました。私たちにとって、これはまだ新しい機能のテストドライブを手配するユニークな機会です。結果を共有します。

新しいテンプレートはどのように見えますか？


このテンプレートを使用してサーバーを監視するには、IPMI経由の監視が構成されたシステムに「ホスト」を作成し、IPMIテンプレートによるテンプレートサーバーをそれに接続する必要があります（図1）。この操作の詳細な説明はありません。詳細な手順は、Zabbixの公式ドキュメントに記載されています。

図。 1. IPMIによるテンプレートサーバーテンプレート


このテンプレートの原理とその構造を検討してください。

テンプレートの基礎はipmitoolユーティリティです。 IPMIを介して機器から必要な統計を取得できます。このユーティリティの機能を使用して必要なすべてのデータを取得すると、IPMIエージェントのデータ要素タイプと特別なipmi.getキーを使用して、Webインターフェースからユーザーが利用できるようになります。これは、新しいバージョンでのipmi.getキーの出現のおかげでのみ可能でした。

Template Server by IPMIテンプレートでは、Item Get IPMIセンサーのデータ要素が、この新しい機能を使用して情報の収集を整理します（図2）。

図。 2. IPMIセンサーアイテムを取得するアイテム


Item Get IPMIセンサーデータエレメントの操作の結果として、構造化されたJSON形式の機器の状態に関する情報がZabbixシステムに表示されます（図3）。

図。3.操作項目Get IPMIセンサーの結果の例

テンプレートが1つではなく2つの検出ルールを使用するのはなぜですか？

離散センサーの検出により、値が文字列型のデータ要素を自動的に作成できます。また、検出ルールのしきい値センサーの検出により、値に「数値」タイプのデータ要素を自動的に作成できます。さらに、このルールでは、データ要素ごとに最大6つのトリガーを形成できます（図6）。

トリガー条件の値は、JSON、つまりデバイス自体から取得されます。トリガーは6つのしきい値に対して作成されます：危険度が低い、危険度が低い、危険度が低い、危険度が高い、危険度が高い、危険度が高い。 JSONでしきい値の値が欠落している場合、トリガーは作成されません。

教育を受けたトリガーでは、しきい値をZabbixレベルで再定義できます。ただし、私たちの意見では、トリガーを変更する最も論理的な方法は、トリガーを機器レベルで変換することです。これを行う方法は通常、デバイスの説明に示されています。
