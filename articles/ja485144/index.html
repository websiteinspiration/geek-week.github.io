壊れたベースをどのように修正したか

あるデータベースを「修正」するのを手伝ってほしいと頼まれたら。チェック中にCHECKDBがエラーのリストを提供し、その一部は「不正」としてマークされました。アプリケーションは機能しましたが、それでもなんとなく落ち着きがありませんでした。

はい、そのような状況での正しい解決策は、エラーがまだ表示されない瞬間からバックアップを取り、破損したデータをローカライズして、クリーンなコピーから上書きすることです。しかし...よくあることですが、エラーをキャッチするのが遅すぎたため、回復するものは何もありませんでした。一方、バックアップはありますが、この話はありません。

既往


最初に、悲劇の規模を理解するためにDBCC CHECKDBを起動します。チームはすべてのテーブルを正直に実行しましたが、ほとんどの場合、問題は見つかりませんでした。同じ結論で、「修正可能な」エラーの報告が100件ありました。そのように：

Index row (1:386974:44) with values (C_FK_6bb5032ec2f94557a7d4a9d39a356168 = '04DA7FC4-B8F2-4D97-B8D2-B207A918D3DF' and C_PK_dd87d9cad5504a1199d4ddaf511ea6a9 = 'F44E8C34-862D-4952-8821-D0E87143A74E') pointing to the data row identified by (C_PK_dd87d9cad5504a1199d4ddaf511ea6a9 = 'F44E8C34-862D-4952-8821-D0E87143A74E').

そして、いくつかのより深刻なエラー：

Table error: Object ID 1333579789, index ID 1, partition ID 72057609974841344, alloc unit ID 72057610075701248 (type In-row data). Page (1:20426) is missing a reference from previous page (1:267203). Possible chain linkage problem.

上手。作業の規模の概要がわかりました。始めましょう！

修正可能なエラー


一部のエラーを簡単に自動的に修正できる理由を理解するために、MS SQLでのインデックスの配置を思い出してみましょう。それらを2つのタイプに分類できます。クラスターと（驚くべきことに）非クラスターです。（列ストアインデックスなどの特別な問題については詳しく説明しません。これはここでは当てはまりません）。どちらもデータの検索に非常に便利なバランスの取れたツリーを表します。

「リーフ」レベルのクラスターインデックスは、テーブル行の内容を直接格納することが重要です。ただし、非クラスター化インデックスには、キーデータ（および、利用可能な場合は「インクルード」フィールド）とクラスターインデックス行へのリンクのみが格納されます。つまり、非クラスター化インデックスに問題がある場合は、クラスター化インデックスから破損したデータを取得して上書きするだけです。まあ、または単に打たれたインデックスを再構築します-良いです、フィールドの構成は既知であり、ソースデータは近くにあり、そのままです。

タスクは完全に創造的ではないので、安全に魂のない車に任せることができます。コマンドを実行する

DBCC CHECKDB (< >, REPAIR_REBUILD)

進捗レポートを調べます。

ソースログと「修正」後のログには、破損したページのアドレスが含まれています。これらのアドレスを比較し、「修正可能」とマークされたすべてのエラーが実際に正常に処理されたことを確認します。

データ接続の中断


今より深刻なもの。非クラスター化インデックスを修正し、情報メッセージからログを消去すると、レポートには「致命的な」エラーの3つのレコードが含まれます。

もちろん、手を振ってDBCC CHECKDBコマンド（<DB名>、REPAIR_ALLOW_DATA_LOSS）をチョップすることもできます。しかし...データを失いたくないだけです。可能な限りすべてを元に戻したい。したがって、エラーレポートの一般的な内容を詳しく見てみましょう。

Table error: Object ID 1333579789, index ID 1, partition ID 72057609974841344, alloc unit ID 72057610075701248 (type In-row data). Page (1:20426) is missing a reference from previous page (1:267203). Possible chain linkage problem.

Table error: Object ID 1333579789, index ID 1, partition ID 72057609974841344, alloc unit ID 72057610075701248 (type In-row data). The previous link (1:267203) on page (1:267204) does not match the previous page (1:20426) that the parent (1:218898), slot 213 expects for this page.

Table error: Object ID 1333579789, index ID 1, partition ID 72057609974841344, alloc unit ID 72057610075701248 (type In-row data). B-tree chain linkage mismatch. (1:20426)->next = (1:267204), but (1:267204)->Prev = (1:267203).

レポートには、1つのページが隣人に会うことを期待しているあいまいな話が含まれていますが、隣人はそれについて何も知りません。一部のペレビンスキー船員ジェレスニャック：甲板に行ったが甲板はない。

完全に理解するには、さらに詳細が必要であり、ページのコンテンツに直接移動する必要があります。しかしその前に、SQL Serverのインデックスページがどのように関連しているかを理解します。

明らかに、ページにはBツリーを形成する階層的な「垂直」リンクがあります。トップページには、下位レベルのページへのリンクなどが含まれています。すでに述べたように、これは値を検索するのに非常に便利です。「Vasya Pupkin」を見つけ、数ページ（「BからG」→「BaからBb」→「Vasya Pupkin」）の後に探しているものを見つけます。

ただし、クエリで文字列の全範囲をすぐに選択する必要がある場合があります（「VasyaからGrishaへ」）。この場合、ツリーを上から下に移動するたびに、ツリーにぶつかります。そのようなリクエストを実行するために、ページは「水平」リンクを保存します。各ページは「前」と「後」の隣接番号を知っています。これらの関係により、インデックススキャンの実行がはるかに簡単になります。

エラーログから判断すると、水平接続と垂直接続の不一致がありました。しかし、最終的にこれを確信するために、ページ自体を見てみましょう。

もっと深く行く必要があります！


ページを表示するために、古くて当然の文書化されていないDBCC PAGEチームを使用します。それは4つのパラメータを取ります：

  • イードベース
  • 基本ファイルID
  • ページID
  • 詳細レベル（0から3）

最後のパラメーターに応じて、サービスヘッダーのみ（0）、またはページのコンテンツ全体（3）、またはヘッダーとコンテンツの一部（1および2）のいずれかを表示できます。

ちなみに、SQL Server 2019では、最終的にドキュメント化されたビューが表示されましたsys.db_db_page_infoは、同様のタスクを実行します。残念ながら、ヘッダーデータ（ドリルダウン0のアナログ）のみが表示されるため、タスクに完全に答えることはできません。

したがって、まず、次のコマンドを実行します

DBCC TRACEON (3604, 1)

残りのDBCCコマンドの出力が、ErrorLogではなくコンソールに

送られるようにします。その後、ページヘッダー20426を確認します。

DBCC PAGE (11, 1, 20426, 0)



晴れ。ページは267203ページと267204ページの中間にあることを望んでいます。しかし、これらの同じページはどうでしょうか。





眠れず、精神もない！彼らは招かれざる客がいない。
さて、次の目次を見てみましょう：



パズルは徐々に発展します：

  • 「垂直」リンク（インデックスツリー構造）の観点から、ページ20426は267203と267204の間にある必要があります
  • 水平接続はこれと矛盾し、267203と267204の間に人はいないと言います。

それでは、どのようなデータが障害の影響を受けたかを理解してみましょう。

ページ内の行は、インデックスキーでソートされます。したがって、ページの最初と最後のキー値を知ることで、「影響を受ける」レコードの範囲を見つけることができます。

キーは、インデックスの説明で簡単に表示できます。これにはsp_helpindexコマンドを使用します。この場合、キーは1列のみです。彼女にすべての注意を払います。



ページで取得したキーを見つけるために、最大3レベルの詳細でDBCC PAGEを使用します。出力ログをスクロールすると、各行の「生の」内容と各フィールドの詳細が表示されます。



ちなみに、キー境界値（ページの最初の値）は、より高いインデックスページのDBCC PAGE出力にも表示されます（218898、上のスクリーンショットを参照）。それらは、ページ番号の直後の列に表示されます。

ページを調べると、20426ページに267203ページの行のちょうど半分が含まれていることがわかります。失敗の性質が明らかになります。データベースページがいっぱいになり、新しいデータを挿入するスペースがなくなると、2つのページに半分に分割されます。どうやら、ページ267203がいっぱいになると、問題ページ20426が作成され、DBMSはリンクの再構築を開始しました。インデックス構造に新しいページを書き込むことができました。しかし、何らかの理由で、水平レコードを更新できず、新しいページが未定義の状態で「ハング」しました。

これは私たちにとって何でしょうか？


確かにあまり良いことはありません。データにアクセスするには、ページリンクが明らかに必要です。クエリを実行するとき、DBMSはこのデータにアクセスする方法を個別に決定します。しかし、（例外が発生する可能性がある）一般的なルールは次のとおりです。

  • 「ワイド」選択（レポートなど）は、水平レコードを使用して実行されます。複数のページを順番にスクロールして、広い範囲を選択する方が簡単です
  • 「ポイント」クエリ（特定のレコードの更新）は、「目次」検索によって実行されます。

特定のレコードが変更されると、DBMSは「問題」ページ20426に到達します。レポートを実行すると、データが「水平に」読み取られ、変更が反映されません。繰り返しますが、実際には、アルゴリズムはより複雑になる可能性がありますが、起こり得る問題のクラスはまだ明確です。

ロシアの知識人の永遠の質問


正直なところ、これがどのように発生するかはまだはっきりしていません。最近のDBMSは実際にはかなり信頼できるものです。データベースファイルのすべての変更（水平および垂直関係の調整を含む）は、トランザクションで実行されます。これらの操作はトランザクションログに記録されます。このログに操作の成功が確認されない場合、すべての操作がロールバックされます。ここでは、トランザクションは正常に完了しましたが、データファイルへのパスに沿った一部の変更が「失われた」ことがわかります。

私に起こったもっともらしい説明は、ディスクサブシステムキャッシュの障害でした。すべてのデータがキャッシュに入れられ、データファイルとログのレコードの一部がディスクに書き込まれ、その後誰かがスイッチを引きました。その結果、変更された「水平」レコードはなんとか記録されませんでしたが、データベースはすでにこれについて何も知りませんでした。 （そして、ここでは、多くの読者が彼らの産業用ディスクコントローラーのバッテリーを急いでチェックしなければなりません）

何をすべきかははるかに明確です。テーブルの接続を復元するには、そのデータを別のテーブルにコピーして、コピー時に水平方向の読み取りが使用されるようにします。正確を期すために、FORCESCANヒントを使用して、DBMSに適切なアクセス方法を明示的に指定できます。

select * 
into T_bca79e9e77c24cdc8bbb7cfd0ddc16fd_BKP
from T_bca79e9e77c24cdc8bbb7cfd0ddc16fd with (FORCESCAN)

その後、ソーステーブルをクリアして、そこにコピーしたデータを返すことができます。

ただし、ページ20426で行った変更は失われるため、テーブルのコピーを作成する前に、ページ20426の識別子を含む行をコピーする必要があります。テーブルを復元した後、必要なエントリを修正します。ページ20426のレコードへのアクセスは、識別子によって明示的に必要です。

select * from T_bca79e9e77c24cdc8bbb7cfd0ddc16fd
where C_PK_bca79e9e77c24cdc8bbb7cfd0ddc16fd = ''

識別子は、同じDBCC PAGEでページのコンテンツを読み取ることによって取得できます。私がすでに書いたように、これらのページのちょうど半分、267203、つまり15のリンクがありました。

破損したページをバックアップし、テーブルをリロードして一致するレコードを照合します-数十分後、テーブルが復元されました。

やった、勝った！しかし、それは本当ですか？


真実。データが復元され、CHECKDBは太陽が窓の外を向いても、エラーの発生を停止しました。安全に自分を賞賛し、酒にふさわしいグラスを1杯上げることができます。データベースのデータはページにリンクされているだけではないことに注意してください。そこで、DBCC CHECKCONSTRAINTSを取得して、壊れた外部キーのリストに真っ向から突入します。しかし、これはまったく別の話です...
