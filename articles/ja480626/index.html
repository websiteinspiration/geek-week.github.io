Get best of the week

レガシーシステムとプロセスの継承またはCTOの役割における最初の90日間

CTOコンピテンシーは、この役割を果たす2回目のテストのみであることが知られています。会社で数年間働くことは1つのことであるため、それとともに進化し、同じ文化的背景の中で、徐々に責任を増やしていきます。そしてそれはまったく別のことです-じゅうたんの下にきちんと気づかれた古い荷物とたくさんの問題を抱えている会社のテクニカルディレクターの地位をすぐに引き受けること。

この意味で、彼がDevOpsConfで共有したLeon Fayerの経験は、直接的にユニークであるだけでなく、20年以上にわたって彼が自分で試してみた経験とさまざまな役割の数を掛け合わせることは非常に有用です。カットの下では、90日間にわたる出来事の年表と、他の人に起こったときに笑うのは楽しいが、直接会うのはそれほど楽しくない多くの物語。

レオンはロシア語でとてもカラフルなので、35〜40分ある場合は、ビデオを視聴することをお勧めします。以下の時間を節約するためのテキスト版。


レポートの最初のバージョンは、有用な推奨事項を含む、人とプロセスとの連携に関する適切に構造化された説明でした。しかし、彼女は途中で出会ったすべての驚きを伝えませんでした。そこで、フォーマットを変えて、新会社が目の前に浮かび上がってスナッフボックスから出てきたような問題と、それらを時系列で解決する方法を概説しました。

一ヶ月前


多くの良い物語のように、これはアルコールから始まりました。私たちは友達と一緒に居酒屋で座っていました。IT関係者の間では当然のことでしたが、誰もが問題を叫びました。彼らの1人は転職したばかりで、彼のテクノロジー、人、チームとの問題について話しました。私がこれ以上15年間解決してきた問題はまさにそのようなものだったので、聴く時間が長くなるほど、彼が私を雇う必要があることだけに気づきました。私は彼にそう言った、そして次の日我々はすでに労働環境で会った。同社は教育戦略と呼ばれていました。

Teaching Strategiesは、非常に幼い子供向けの教育プログラムの市場をリードしています-誕生から3年間。従来の「紙」企業はすでに40年の歴史があり、プラットフォームのデジタルSaaSバージョン10です。比較的最近、デジタル技術を企業の標準に適合させるプロセスが始まりました。 「新しい」バージョンは2017年に発売され、ほとんど古いバージョンのようでしたが、機能が低下しました。

最も興味深いのは、この会社のトラフィックが非常に予測可能であることです。毎日、毎年、何人がいつ来るかを非常に明確に予測できます。たとえば、午後1時から午後3時の間に、幼稚園に通うすべての子供たちが就寝し、教師が情報を入力し始めます。そしてこれは週末を除いて毎日起こります、なぜなら週末にはほとんど誰も働かないからです。



少し前を向いて、私は年間最大のトラフィックの期間中に仕事を始めました。これはさまざまな理由で興味深いです。

たった2年前のように見えるこのプラットフォームには、ColdFusionとSQL Server 2008という奇妙なスタックがありました。ColdFusionは、あなたが知らない場合、おそらく知らない場合でも、90年代半ばに登場したエンタープライズ向けのPHPであり、それ以来、私はそれについて聞いたこともありません。Ruby、MySQL、PostgreSQL、Java、Go、Pythonもあります。しかし、主要なモノリスは、ColdFusionとSQL Serverで動作しました。

問題


仕事やどんな問題が発生したかについて会社の従業員と話すほど、問題は本質的に技術的なものだけではないことに気づきました。さて、テクノロジーは古く、彼らはそれで機能しませんでしたが、チームとプロセスに問題があり、会社はこれを理解し始めました。

伝統的に、技術者たちは隅に座って、彼らの仕事のいくつかをしていました。しかし、ますます多くのビジネスがデジタル版を通過し始めました。したがって、私の仕事が始まる前の昨年の会社では、新しいもの、取締役会、CTO、CPO、QAディレクターが現れました。つまり、同社は技術分野への投資を開始しました。

重い遺産の痕跡はシステムだけではありませんでした。同社には、従来のプロセス、従来の人々、従来の文化がありました。これはすべて変更する必要がありました。絶対つまらないと思ってやってみることにしました。

二日前


新しい仕事を始める2日前に、私はオフィスに到着し、最後の書類に記入し、チームと知り合いになり、チームがその時点で問題に苦しんでいることに気付きました。これは、ページの平均読み込み時間が4秒、つまり2倍に急増したことにあります。



スケジュールから判断すると、明らかに何かが起こっており、何が起こっているのかは不明です。問題は、データセンターのネットワーク遅延にあることがわかりました。データセンターの5 msの遅延は、ユーザーの2秒に変換されました。なぜこれが起こったのかはわかりませんでしたが、いずれにしても問題はデータセンターにあることがわかりました。

最初の日


2日経ちましたが、最初の就業日に問題が解消されないことに気付きました。



2日間、ページのユーザーは平均4秒間読み込みました。彼らが問題が何であるかを見つけたかどうか尋ねます。

-はい、チケットをオープンしました。
-そして？
「まあ、彼らはまだ私たちに答えていません。」

その後、私は以前に言われたことはすべて、私が戦わなければならなかった氷山の一角にすぎないことに気づきました。

この場合に非常に適した良い引用があります：
「テクノロジーを変えるために組織を変える必要がある場合があります。」
しかし、私は1年で最も忙しい時期に仕事を始めたので、この問題の両方の解決策（迅速と長期の両方）を検討する必要がありました。そして、今重要なことから始めましょう。

三日目


したがって、読み込みには4秒かかり、13から15までが最大のピークです。



3日目のこの時間間隔では、ダウンロード速度は次のようになりました。



私の観点からは、何も機能しませんでした。他のすべての人の観点からは、通常よりも少し遅く動作しました。しかし、それはそのようには起こりません-これは深刻な問題です。

私はチームを説得しようとしましたが、チームはもっとサーバーが必要だと答えました。もちろん、これが問題の解決策ですが、必ずしも唯一かつ最も効果的なものではありません。なぜサーバーが足りないのか、どのくらいのトラフィックがあるのか​​と尋ねました。データを推定すると、1秒あたり約150のリクエストがあり、基本的には合理的な制限に適合しています。

しかし、正しい答えを得る前に、正しい質問をする必要があることを忘れてはなりません。次の質問は次のとおりです。フロントエンドサーバーはいくつありますか。「困惑しました」という答え-17台のフロントエンドサーバーがありました。

-質問して恥ずかしいですが、150を17で割ると、約8になりますか？各サーバーが1秒あたり8リクエストをスキップするとします。明日1秒あたり160リクエストがある場合、さらに2台のサーバーが必要ですか。

もちろん、サーバーを追加する必要はありませんでした。解決策はコード自体と表面上にありました：

var currentClass = classes.getCurrentClass();
return currentClass;

getCurrentClass()サイト上のすべてがクラスのコンテキストで正しく 機能するため、機能がありました。そして、各ページのこの1つの関数では、200以上のリクエストがありました。

この方法での解決策は非常に単純で、何も書き換える必要はありませんでした。同じ情報を再度要求しないでください。

if ( !isDefined("REQUEST.currentClass") ) {
    var classes = new api.private.classes.base();
   REQUEST.currentClass = classes.getCurrentClass();
}
return REQUEST.currentClass;

3日目だけが主な問題を見つけることにしたので、とても嬉しかったです。私は世間知らずだったので、これは非常に多くの問題の1つにすぎませんでした。



しかし、この最初の問題の解決策は、スケジュールを大幅に下げました。

同時に、他の最適化にも取り組みました。目に見えるのは修理できるすべてのものでした。たとえば、同じ3日目に、システムにまだキャッシュがあることがわかりました（最初は、すべての要求がデータベースから直接送信されると思いました）。キャッシュについて考えるときは、標準のRedisまたはMemcachedを紹介します。しかし、私がそう思ったのは、そのシステムでのキャッシュにはMongoDBとSQL Serverが使用されていたためです-データが読み取られたばかりの同じものです。

10日目


最初の週は、今すぐに解決する必要のある問題を扱っていました。 2週目のどこかで、私は最初にスタンドアップに来てチームと話し、何が起こっているのか、そしてプロセス全体がどのように進んでいるのかを見ました。

再び、興味深いことが発見されました。チームは、18人の開発者で構成されていました。 8人のテスター; 3人のマネージャー; 2人の建築家。そして、彼らは皆、共通の儀式に参加しました。つまり、毎朝30人以上の人々が立ち上がり、何をしているのかを話しました。会議に5〜15分を要しなかったことは明らかです。誰もが異なるシステムで作業しているため、誰も誰も聞いていません。この形式では、グルーミングセッションで1時間あたり2〜3枚のチケットがすでに良好な結果でした。

私たちが最初に行ったのは、製品ラインに沿ってチームをいくつかに分割することでした。さまざまなセクションとシステムについて、開発者、テスター、製品マネージャー、ビジネスアナリストを含む別々のチームを特定しました。

その結果、以下を受け取りました。

  • スタンドアップとラリーの削減。
  • 製品知識。
  • 所有感。以前の人々は常にシステムについて話していたとき、彼らはおそらく誰かがバグを処理しなければならないが、自分自身では処理できないことを知っていました。
  • グループ間のコラボレーション。QAは以前はプログラマーとあまりコミュニケーションをとっていなかった、製品は独自のことをしたなどとは言えません。今、彼らは責任の共通点を持っています。

私たちは主に効率、生産性、品質に焦点を当てました-これらはチーム変革によって解決しようとした問題です。

11日目


チームの構造を変更する過程で、ストーリーポイントのカウント方法を発見しました。1 SPは1日に相当し、それらに含まれていた各チケットには、開発とQAの両方のSP、つまり少なくとも2 SPが含まれていました。

どうやってこれを見つけたのですか？



バグが見つかりました。レポートが必要な期間の開始日と終了日が入力されているレポートの1つで、最終日は考慮されません。つまり、リクエストのどこかに<=ではなく、単に<でした。これらは3つのストーリーポイント、つまり3日間であると言われました。

その後、私たちは：

  • ストーリーポイントの評価システムを改訂しました。今すぐシステムを介して渡されることができるマイナーなバグを修正することがユーザーにすぐ届きます。
  • 開発とテストのために関連するチケットを組み合わせるようになりました。以前は、すべてのチケット、すべてのバグはクローズドエコシステムであり、他のものに関連付けられていませんでした。1つのページで3つのボタンを変更すると、1つのページで1つの自動テストを実行するのではなく、3つの異なるQAプロセスを持つ3つの異なるチケットになる可能性があります。
  • 彼らは、人件費を評価するためのアプローチについて開発者と協力し始めました。1つのボタンを変更する3日間はおかしくありません。

二十日


最初の月の半ばのどこかで、状況は少し安定しました、私は何が起こっているのかを理解しました、そして私はすでに未来を調べ、長期的な解決策について考え始めました。

長期目標：

  • マネージドプラットフォーム 各ページで何百ものリクエスト-これは深刻ではありません。
  • 予測可能な傾向。一見、他のメトリックと相関しない定期的なトラフィックのピークがありました。これが発生する理由を理解し、予測を学ぶ必要がありました。
  • プラットフォーム拡張。ビジネスは絶えず成長しており、ますます多くのユーザーが訪れ、トラフィックは増加しています。

それは多くの場合、過去に言われた：「[言語/フレームワーク]でレッツ・リライトすべてが、すべてが良い仕事します！」
ほとんどの場合、書き直されたものがまったく機能する場合、これはうまく機能しません。したがって、ロードマップを作成する必要がありました。これは、ビジネス目標の達成方法（実行する内容と理由）を段階的に示す具体的な戦略であり、次のことを行います。

  • プロジェクトの使命と目標を反映しています。
  • 主要な目標に優先順位を付けます。
  • 彼らの達成のスケジュールが含まれています。

それ以前は、変更が加えられた目的についてチームと話し合うことはありませんでした。これには適切な成功率が必要です。会社の歴史の中で初めて、技術グループのKPIを設定しました。これらの指標は組織の指標に関連付けられていました。



つまり、組織のKPIはチームによってサポートされており、チームのKPIはすでに個々のKPIによってサポートされています。そうでなければ、技術的KPIが組織的KPIと一致しない場合は、誰もが自分の上に毛布を引き寄せます。

たとえば、組織のKPIの1つは、新製品を通じて市場シェアを拡大​​することです。

より多くの新製品を持つという目標をどのようにサポートできますか？

  • まず、バグを修正するのではなく、より多くの時間を新製品の開発に費やしたいと考えています。これは、簡単に測定できる論理的なソリューションです。
  • 第2に、市場占有率が高ければ高いほど、ユーザー数が増加し、それに応じてトラフィックも増加するため、トランザクション量の増加をサポートしたいと考えています。



次に、グループ内で実行できる個々のKPIは、たとえば、主要な欠陥が発生した場所に配置されます。この特定のセクションに焦点を合わせると、欠陥の数を大幅に減らし、新製品の開発にかかる時間を増やし、組織のKPIをサポートすることができます。

したがって、コードの書き換えを含む各決定は、会社が私たちに設定した特定の目標（組織の成長、新機能、採用）をサポートする必要があります。

このプロセスの間に、技術者だけでなく会社全体にとってニュースになる興味深いことが浮上しました。すべてのチケットは少なくとも1つのKPIに焦点を当てるべきです。つまり、製品が新しい機能を作成したいと言っている場合、最初の質問をする必要があります：「この機能はどのKPIをサポートしていますか？」そうでない場合は、申し訳ありません-これは不要な機能のようです。

30日目


その月の終わりに、Opsチームの誰も私たちがクライアントと締結した契約を見たことがないというもう1つのニュアンスを発見しました。連絡先を表示する理由を尋ねる場合があります。

  • まず、SLAが契約に登録されているためです。
  • 次に、SLAはすべて異なります。各クライアントには要件があり、営業部門は見ずに署名しました。

別の興味深いニュアンス-最大の顧客の1人との契約では、プラットフォームがサポートするソフトウェアのすべてのバージョンはn-1、つまり最新バージョンではなく最後から2番目でなければならない、と書かれています。

プラットフォームがColdFusionとSQL Server 2008上にあり、7月にサポートが終了した場合、n-1からどれだけ離れていたかは明らかです。

四十五日目


2か月半ばのどこかで、座って、プロセス全体のバリューストリームマッピングを完全に行うための十分な時間があります。これらは、製品の作成から消費者への配送までの必要なステップであり、できるだけ詳細にペイントする必要があります。

プロセスを細かく分割し、時間がかかりすぎたり、最適化、改善できるものなどを確認したりします。たとえば、製品からのリクエストに必要な時間、グルーミングを通過する時間、開発者が取ることができるチケットに到達するタイミング、QAなど。個々のステップを詳細に見て、最適化できると思います。

私がこれを行ったとき、2つのことが私の目に留まりました。

  • QAから開発者への返品チケットの割合が高い。
  • プルリクエストのレビューに時間がかかりすぎました。

問題は、これらが次のような結論であるということでした：時間がかかるようですが、正確にはどれくらいかわかりません。
「測定できないものを改善することは不可能です。」
問題がどれほど深刻であるかを具体化する方法は？それは数日または数時間を費やしますか？

これを測定するために、Jiraプロセスに「開発の準備完了」と「QAの準備完了」という2つのステップを追加して、各チケットが待機する時間と特定のステップに戻る回数を測定しました。



また、「審査中」を追加して、審査中のチケットの平均数を確認しました。そのため、すでにチケットが踊っています。システムメトリックがありましたが、新しいメトリックを追加し、測定を開始しました。

  • プロセス効率：パフォーマンスと計画/提供。
  • プロセス品質：欠陥の数、QAからの欠陥。

何がうまくいって、何が悪いのかを理解することは本当に役立ちます。

50日目


もちろん、これはすべて良い面白さですが、2か月目の終わり頃に、原則として予測可能な何かが起こりましたが、そのような規模は期待していませんでした。トップが変わったので、人々は去り始めました。新しい人々がすべてを変え始めたリーダーシップに来て、古いものはやめました。そして、通常、数年前の会社では、すべての友達とすべてがお互いを知っています。

これは予想どおりでしたが、レイオフの規模は予想外​​でした。たとえば、1週間で2人のチームリーダーが同時に解雇を申請しました。他の問題を忘れるだけでなく、チームの作成に集中する必要がありました。これは長く困難な問題ですが、残った人々（またはそのほとんど）を救いたいと思ったため、彼女はそれに対処する必要がありました。チームの道徳を維持するために、人々が去ったという事実に何らかの形で反応することが必要でした。

理論的には、これは良いことです。チームのスキルを評価して担当者を置き換えることができる完全なカルテブランチを持つ新しい人物が到着します。実際には、非常に多くの理由で、単に新しい人を連れてくることはできません。常にバランスが必要です。

  • 古いものと新しいもの。私たちは、使命を変え、支援できる高齢者を維持する必要があります。しかし同時に、私たちは新しい血を持ち込む必要があります。これについては少し後で話します。
  • 経験。私は、やけどして私たちのために働きたいと思っている後輩たちとたくさん話しました。しかし、後輩を支え、メンターになる先輩が足りなかったので、私はそれらを取ることができませんでした。最初にトップを獲得する必要があり、次に初めて若者を獲得する必要がありました。
  • ニンジンとスティック。

どのバランスが正しいのか、それをどのように維持するのか、何人の人が去るのか、どれだけプッシュするのかという質問には、良い答えはありません。これは純粋に個別のプロセスです。

五十一日


私は自分が誰であるかを理解するためにチームを注意深く見始め、もう一度思い出しました：
「ほとんどの問題は人々の問題です。」
チームには、開発者とOpsの両方が3つの大きな問題を抱えていることがわかりました。

  • 現状に満足。
