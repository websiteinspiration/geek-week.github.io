従来のアンチウイルスがパブリッククラウドに適さない理由。そして何をすべきか？

ITインフラストラクチャ全体をパブリッククラウドに展開するユーザーが増えています。ただし、アンチウイルスコントロールが不十分な場合、深刻なサイバーリスクがお客様のインフラストラクチャで発生します。実務では、既存のウイルスの最大80％が仮想環境で完全に生存していることが示されています。この投稿では、パブリッククラウドのITリソースを保護する方法と、従来のアンチウイルスがこれらの目的に適さない理由について説明します。



最初に、通常のウイルス対策保護ツールはパブリッククラウドには適しておらず、リソースを保護するための他のアプローチが必要であるという結論に至った経緯について説明します。

まず、原則として、プロバイダーはクラウドプラットフォームの保護を高レベルで保証するために必要な手段を提供します。たとえば、＃CloudMTSでは、すべてのネットワークトラフィックを分析し、クラウドのセキュリティログを監視して、定期的にペンテストを実行しています。個々の顧客に提供されるクラウドセグメントも確実に保護する必要があります。

次に、サイバーリスクとの戦いのクラシックバージョンでは、各仮想マシンにウイルス対策とそのコントロールをインストールします。ただし、仮想マシンの数が多いと、この方法は非効率になり、大量のコンピューティングリソースを必要とするため、お客様のインフラストラクチャに負荷がかかり、クラウドの全体的なパフォーマンスが低下します。これは、顧客の仮想マシンに効果的なウイルス対策保護を構築するための新しいアプローチを見つけるための重要な前提条件となっています。

さらに、市場で入手可能なほとんどのウイルス対策ソリューションは、パブリッククラウド環境でITリソースを保護する問題を解決するように適合されていません。原則として、これらはヘビーウェイトEPPソリューション（エンドポイント保護プラットフォーム）であり、さらに、クラウドプロバイダーのクライアント側で必要なカスタマイズ機能を提供しません。

従来のウイルス対策ソリューションは、更新とスキャン中に仮想インフラストラクチャを真剣に読み込み、必要なレベルの役割管理と設定がないため、クラウドでの作業にはあまり適していません。次に、クラウドがアンチウイルス保護に新しいアプローチを必要とする理由を詳細に分析します。

パブリッククラウドでアンチウイルスが実行できること


それでは、仮想環境での作業の詳細に注意しましょう：

更新の効率と定期的なチェック。従来のアンチウイルスを使用するかなりの数の仮想マシンが一度に更新を開始すると、いわゆる「ストーム」の更新がクラウドで発生します。複数の仮想マシンをホストするESXiホストの能力では、デフォルトで起動される同様のタスクの急流を処理するには不十分な場合があります。クラウドプロバイダーの観点から見ると、このような問題は多数のESXiホストに追加の負荷をかける可能性があり、最終的にはクラウド仮想インフラストラクチャのパフォーマンスの低下につながります。これは、他のクラウドクライアントの仮想マシンのパフォーマンスに特に影響を与える可能性があります。一括スキャンを開始すると、同様の状況が発生する可能性があります。異なるユーザーからの同じタイプの多数のリクエストのディスクシステムによる同時処理は、クラウド全体のパフォーマンスに悪影響を及ぼします。高い確率で、ストレージシステムの稼働容量が減少すると、すべてのお客様に影響があります。このような突発的な負荷は、プロバイダーやその顧客を満足させません。クラウド内の「近隣」に影響を与えるからです。この観点から、従来のウイルス対策は大きな問題になる可能性があります。

.ウイルスに感染している可能性のあるファイルまたはドキュメントがシステムで検出された場合、検疫に送信されます。もちろん、感染したファイルはすぐに削除できますが、ほとんどの企業ではこれを受け入れられないことがよくあります。プロバイダーのクラウドで動作するように適合されていない企業のアンチウイルスには、通常、共通の隔離ゾーンがあります。感染したすべてのオブジェクトがそれに分類されます。たとえば、会社のユーザーのコンピュータで見つかります。クラウドプロバイダーのクライアントは、独自のセグメント（またはテナント）で「ライブ」です。これらのセグメントは不透明で隔離されています。顧客はお互いを認識しておらず、もちろん他のユーザーがクラウドに何を配置しているのかわかりません。クラウド内のすべてのウイルス対策ユーザーがアクセスする一般的な検疫では、機密情報または企業秘密が含まれている可能性のあるドキュメントが侵入する可能性があることは明らかです。これは、プロバイダーとその顧客には受け入れられません。したがって、解決策は1つしかありません。これは、そのセグメント内の各クライアントの個人検疫であり、プロバイダーも他のクライアントもアクセスできません。

個別のセキュリティポリシー。クラウド内の各クライアントは、IT部門が独自のセキュリティポリシーを設定する個別の会社です。たとえば、管理者はスキャンルールとウイルス対策スキャンスケジュールを定義します。したがって、各組織には、ウイルス対策ポリシーを構成するための独自のコントロールセンターが必要です。同時に、設定はクラウドの他のクライアントに影響を与えるべきではなく、プロバイダーは、たとえば、すべてのクライアント仮想マシンに対してウイルス対策更新が正常に実行されることを確認できる必要があります。

請求とライセンスの組織。クラウドモデルは柔軟性が高く、顧客が使用したITリソースの量に対してのみ支払う必要があります。たとえば、季節要因を考慮して必要がある場合、リソースの量は、すべて現在の計算能力のニーズに基づいて、すばやく増減できます。従来のアンチウイルスはそれほど柔軟ではありません。通常、クライアントは、所定の数のサーバーまたはワークステーションのライセンスを1年間購入します。クラウドユーザーは、現在のニーズに応じて追加の仮想マシンを定期的に切断および接続します。したがって、ウイルス対策ライセンスは同じモデルをサポートする必要があります。

2番目の質問は、ライセンスが正確に何に適用されるかです。従来のアンチウイルスは、サーバーまたはワークステーションの数によってライセンスされています。保護された数の仮想マシンのライセンスは、クラウドモデルに完全には適合しません。クライアントは、使用可能なリソース（たとえば、5台または10台）から任意の数の仮想マシンを作成できます。ほとんどのお客様にはこの番号がありません。プロバイダーとしての変化を追跡することはできません。 CPUによるライセンス交付は技術的に不可能です。クライアントは、ライセンスが必要な仮想プロセッサー（vCPU）を受け取ります。したがって、新しいウイルス対策保護モデルには、顧客がウイルス対策ライセンスを受け取るために必要なvCPUの数を決定する可能性を含める必要があります。

法律の遵守。適用されるソリューションはレギュレーターの要件に確実に準拠する必要があるため、重要なポイントです。たとえば、多くの場合、クラウドの「住民」は個人データを操作します。この場合、プロバイダーには個別の認定クラウドセグメントが必要です。これは、個人データに関する法律の要件に完全に準拠しています。そうすれば、企業は個人データを扱うためにシステム全体を「構築」する必要がなくなります。つまり、認定済みの機器を購入し、接続して構成し、認定に合格する必要があります。そのようなクライアントのISPDのサイバー保護のために、アンチウイルスはロシアの法律の要件にも準拠し、FSTEC証明書を持っている必要があります。

アンチウイルス保護がパブリッククラウドで満たさなければならない必須の基準を調査しました。次に、プロバイダーのクラウドで動作するようにアンチウイルスソリューションを適応させる際の私たち自身の経験を共有します。

どうすれば友達のウイルス対策とクラウドを作ることができますか


私たちの経験が示したように、説明とドキュメントのソリューションを選択することは1つのことであり、すでに実行中のクラウド環境でそれを実践することは、複雑さの点で完全に異なるタスクです。実際に何をしたか、プロバイダーのパブリッククラウドで動作するようにウイルス対策をどのように適応させたかをお伝えします。ウイルス対策ソリューションベンダーはKasperskyで、同社のポートフォリオにはクラウド環境向けのウイルス対策保護ソリューションが含まれています。 Kaspersky Security for Virtualization（Light Agent）を採用しました。

Kaspersky Security Centerの単一のコンソールが含まれています。 Light AgentとSecurity Virtual Machine（SVM）およびKSC Integration Server。

カスペルスキーのソリューションのアーキテクチャを調査し、ベンダーのエンジニアと一緒に最初のテストを実施した後、サービスをクラウドに統合することに関する疑問が生じました。最初の実装は、モスクワのクラウドサイトで共同で実施されました。そして、それは私たちが理解したことです。

ネットワークトラフィックを最小限に抑えるために、各ESXiホストにSVMを配置し、SVMをESXiホストに「バインド」することが決定されました。この場合、保護された仮想マシンのライトエージェントは、それらが実行されている特定のESXiホストのSVMにアクセスします。メインKSCには別の管理テナントが選択されました。その結果、KSCの部下は各クライアントのテナントに配置され、管理セグメントに配置された上位のKSCに向かいます。このようなスキームにより、クライアントのテナントで発生する問題をすばやく解決できます。

アンチウイルスソリューション自体のコンポーネントの引き上げに関する問題に加えて、追加のVxLANの作成を通じてネットワークの相互作用を整理するという課題に直面しました。また、このソリューションはもともとプライベートクラウドを備えたエンタープライズクライアント向けでしたが、NSX Edgeのエンジニアリングの工夫と技術の柔軟性により、テナントとライセンスの分離に関連するすべての問題を解決することができました。

カスペルスキーのエンジニアと緊密に連携しました。したがって、システムコンポーネント間のネットワーク相互作用の観点からソリューションアーキテクチャを分析するプロセスでは、ライトエージェントからSVMへのアクセスに加えて、SVMからライトエージェントへのフィードバックも必要であることがわかりました。クラウドの異なるテナントに仮想マシンの同一のネットワーク設定が存在する可能性があるため、このネットワーク接続はマルチテナント環境では不可能です。そのため、ベンダーの同僚は、SVMからLight Agentへのネットワーク接続の必要性をなくすという観点から、要請に応じて、Light AgentとSVM間のネットワーク相互作用のメカニズムを再設計しました。

ソリューションをモスクワのクラウドサイトに展開してテストした後、認定されたクラウドセグメントを含む他のサイトにソリューションを複製しました。現在、このサービスは国のすべての地域で利用できます。
