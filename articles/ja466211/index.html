Android 10のプロジェクトメインライン

昨日、Googleのモバイルデバイス向けの次のOSアップデートがリリースされました。Android 10の主な革新の1つは「プロジェクトメインライン」でした。私はそれが何であるか、なぜ、そしてそれがどのように機能するかを理解しようとします。

Googleは何年もの間、Androidのセキュリティアップデートの問題を解決しようとしてきました。Androidは現在数十億台のデバイスにインストールされていますが、ほとんどのメーカーはリリース後にデバイスをサポートすることに非常に消極的です。これにより、既知の脆弱性を持つバージョンがインストールされている多くのデバイスが動作しているという事実につながります。

Androidの更新を簡素化する最初の大きなステップは、Androidをハードウェア依存と独立の2つの部分に「分割」した「プロジェクトトレブル」でした。これにより、Androidの新しいバージョンへのアップデートのリリースが容易になりました。次の大きなステップはProject Mainlineでした。アプリケーションの更新と同様に、システム全体を更新することなく、オペレーティングシステムの個々のコンポーネントを更新できます。

なぜこれが必要なのですか？


Androidアーキテクチャでは、オペレーティングシステム（ファームウェアと呼ばれることが多い）は、ユーザーデータやアプリケーションから厳密に分離されています。フラッシュメモリの別のセクションにあり、読み取り専用であり、最新バージョンのAndroidでは整合性制御のためにデジタル署名されています。オペレーティングシステムの一部であるAndroidアプリケーションの場合、ユーザーのメモリセクションにアプリケーションの新しいバージョンを配置することで更新が可能です。 Linuxディストリビューションのように、パッケージマネージャーを使用するなど、オペレーティングシステムの他の部分（サービス、ライブラリなど）を更新することは不可能でした。そのような修正のために、メーカーはシステム全体のアップデートをリリースすることを余儀なくされました。同時に、Googleモバイルサービス（大多数のデバイスで使用）を使用する場合、更新ごとにGoogleによる認定が必要です。そして「Project Mainline」がこの問題を解決し、Androidアプリケーションと同様に、システムコンポーネントを個別に更新できるようになりました。セキュリティ更新プログラムに加えて、システム設定とタイムゾーン（tzdata）などのデータの更新も簡素化されます。

使い方


Project Mainlineの中核は、APEXシステムアプリケーション（Android Pony EXpressの略）の新しいコンテナーです。

Android 10では、Googleはシステム内で13個のAPEXパッケージを特定しました。これらは個別に更新できます。

  • セキュリティ：メディアコーデック、メディアフレームワークコンポーネント、DNSリゾルバー、Conscrypt
  • プライバシー：ドキュメントUI、アクセス許可コントローラー、ExtServices
  • 一貫性：タイムゾーンデータ、ANGLE（開発者のオプトイン）、モジュールメタデータ、ネットワークコンポーネント、キャプティブポータルログイン、ネットワークアクセス許可の構成

APEXパッケージは、「パッケージインストーラー」、adb、またはGoogle Playを使用してAPKと同様にインストールできます。

頂点


APEXファイルは、Andoird APKアプリケーションで使用されるものと同様です。

APEXは4つのメインファイルを含むzipアーカイブです。

  • apex_manifest.json-パッケージの名前とそのバージョンが含まれます。
  • AndroidManifest.xml-パッケージメタデータ（APKと同様）;
  • apex_payload.img-ext4ファイルシステムのイメージ。
  • apex_pubkey-画像の署名を検証するための公開鍵。

APEXマネージャーシステムサービス（apexd）はAPEXパッケージで動作します。

Apexマネージャー


  • パッケージインストーラーは、APEXパッケージであると判断すると、それをAPEXマネージャーに渡します。
  • APEXマネージャは、パッケージとそのバージョンをチェックします。
  • チェックに合格した場合は、メモリのユーザーセクションに展開し、データベースのレコードを更新して、デバイスを再起動します。
  • APEXが起動すると、マネージャーはデータベースからすべてのパッケージをチェックし、ext4イメージのループデバイスを作成して、パス/ apex / name @ verに沿ってマウントします。

APEXパッケージの画像の内部には、次のようなものがあります。

  • 実行可能ファイル
  • 共有（.so）ライブラリ、
  • JARライブラリ、
  • データファイル
  • 構成ファイル。

署名APEXパッケージ


APEXパケットは2つの異なるキーで2回署名されます。ext4イメージは1つのキーで署名され（Android検証済みブートのようにdm-verityが使用されます）、APEXパッケージ（zipファイル）はAPKパッケージと同様に2番目のキーで署名されます。

Linuxカーネル


APEXは、ループデバイス、DM-verityなど、いくつかのLinuxカーネルメカニズムを使用します。
APEXは、カーネルバージョン4.4以降でサポートされています。古いバージョンのカーネルのデバイスでは、更新をサポートしない「フラット」モードでのみ動作します。
