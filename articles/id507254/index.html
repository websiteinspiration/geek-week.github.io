Zabbix - memperluas batas makro

Membuat solusi untuk klien, ada 2 tugas yang ingin saya pecahkan dengan indah dan dengan fungsionalitas Zabbix biasa.

Tugas 1. Melacak versi firmware saat ini di router Mikrotik.

Tugas ini diselesaikan dengan mudah - dengan menambahkan agen ke template HTTP. Agen menerima versi saat ini dari situs web Mikrotik, dan pelatuk membandingkan versi saat ini dengan versi saat ini dan, dalam kasus perbedaan, mengeluarkan peringatan.

Ketika Anda memiliki 10 router, algoritma ini tidak kritis, tetapi apa yang harus dilakukan dengan 3000 router? Kirim 3000 permintaan ke server? Tentu saja, skema seperti itu akan berhasil, tetapi gagasan 3000 pertanyaan tidak cocok untuk saya, saya ingin mencari solusi lain. Selain itu, ada kelemahan dalam algoritme seperti itu: pihak lain dapat menghitung sejumlah permintaan dari satu IP untuk serangan DoS, mereka hanya dapat melarangnya.

Tugas 2Menggunakan sesi otorisasi di berbagai agen HTTP.

Ketika melalui agen HTTP Anda perlu menerima informasi dari halaman "tertutup", Anda memerlukan cookie otorisasi. Untuk melakukan ini, biasanya ada formulir otorisasi standar dengan pasangan login / kata sandi dan pengaturan ID sesi dalam cookie.

Tetapi ada masalah, Anda tidak dapat mengakses data dari item lain dari satu item agen HTTP untuk menggantikan nilai ini di Header.

Ada juga "skenario web", ia memiliki batasan lain, tidak memungkinkan untuk menerima konten untuk analisis dan penyimpanan lebih lanjut. Anda hanya dapat memeriksa keberadaan variabel yang diperlukan pada halaman atau meneruskan variabel yang diperoleh sebelumnya antara langkah-langkah skrip web.

Setelah sedikit merenungkan tugas-tugas ini, saya memutuskan untuk menggunakan makro yang terlihat sempurna di bagian mana pun dari sistem pemantauan: di templat, host, pemicu atau item. Dan makro dapat diperbarui melalui antarmuka web API.

Zabbix memiliki dokumentasi API yang baik dan terperinci. Untuk bertukar data melalui api, format data Json digunakan. Detailnya dapat ditemukan di dokumentasi resmi .

Urutan tindakan untuk mendapatkan data yang kita butuhkan dan merekamnya dalam makro disajikan pada diagram di bawah ini.



Langkah 1


Langkah pertama dapat terdiri dari satu tindakan atau banyak tindakan. Pada langkah pertama, semua logika dasar diletakkan, dan yang utama adalah 3 langkah terakhir.

Dalam contoh saya, langkah pertama adalah mendapatkan cookie otorisasi di PBX untuk tugas pertama. Untuk tugas kedua, saya mendapat nomor versi firmware Mikrotik saat ini.

  • upgrade.mikrotik.com/routeros/LATEST.6 - URL dari versi Stabil saat ini
  • upgrade.mikrotik.com/routeros/LATEST.6fix — URL LTS

Langkah pertama adalah sepenuhnya individual untuk setiap kasus dan logika kerjanya mungkin berbeda. Itu semua tergantung pada tugas Anda.
Saat bekerja dengan skrip web, pantau metode mana yang Anda perlukan untuk mendapatkan respons. Header respons HTTP atau badan respons itu sendiri tanpa header?
Jika cookie otorisasi diperlukan, maka setel metode respons Header seperti dalam kasus Asterisk.

Jika Anda membutuhkan data, seperti halnya respons server mikrotik, letakkan tubuh Respons tanpa header.

Langkah 2


Kami lolos ke langkah kedua. Mendapatkan sesi otorisasi:

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc": "2.0",
    "method": "user.login",
    "params": {
        "user": "Admin"
        "password": "zabbix"
    },
    "id": 1,
    "auth": null
}

jsonrpc adalah versi protokol JSON-RPC yang digunakan;
Zabbix mengimplementasikan JSON-RPC versi 2.0;

  • metode - metode yang disebut;
  • params - parameter yang dilewati oleh metode;
  • id - pengidentifikasi permintaan sewenang-wenang;
  • auth - kunci otentikasi pengguna; seperti kita belum memilikinya, kita setel ke nol.

Untuk bekerja dengan API, saya membuat akun terpisah dengan hak terbatas. Pertama, Anda tidak perlu memberikan akses ke tempat yang tidak Anda butuhkan. Dan kedua, sebelum versi 5.0, kata sandi yang ditentukan melalui makro dapat dibaca. Karenanya, jika Anda menggunakan kata sandi admin Zabbix, akun admin mudah dicuri.

Ini akan benar terutama ketika bekerja dengan API melalui skrip pihak ketiga dan menyimpan kredensial di samping.

Sejak versi 5.0, opsi untuk menyembunyikan kata sandi yang disimpan di makro telah muncul.



Saat membuat akun terpisah untuk memperbarui data melalui API, pastikan untuk memeriksa apakah data yang Anda butuhkan tersedia melalui antarmuka web dan apakah mungkin untuk memperbaruinya. Saya tidak memeriksa, dan kemudian untuk waktu yang lama saya tidak bisa mengerti mengapa makro yang saya butuhkan tidak terlihat oleh API.



Setelah Anda menerima otorisasi dalam API, kami melanjutkan untuk mendapatkan daftar makro.

Langkah 3


API tidak mengizinkan memperbarui makro host dengan nama, untuk ini Anda harus terlebih dahulu mendapatkan ID makro. Selain itu, untuk mendapatkan daftar makro untuk host tertentu, Anda perlu mengetahui ID host ini, dan ini adalah permintaan tambahan. Anda tidak dapat menggunakan makro reguler {HOST.ID} dalam permintaan. Keterbatasan memutuskan untuk mengatasi ini:



Saya membuat makro lokal dengan ID host ini. Sangat mudah untuk menemukan ID host dari antarmuka web.

Respons dengan daftar semua makro host yang diberikan dapat disaring oleh templat:

regex:{"hostmacroid":"([0-9]+)"[A-z0-9,":]+"{\$MIKROTIK_VERSION}"



Dengan demikian, kami mendapatkan ID makro yang kami butuhkan, di mana MIKROTIK_VERSION adalah nama makro yang kami cari. Dalam kasus saya, ini mencari makro MIKROTIK_VERSION yang telah ditugaskan ke host.

Permintaan itu sendiri terlihat seperti ini:

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.get",
    "params":{
        "output":"extend",
        "hostids":"{$HOST_ID}"
    },
    "auth":"{sid}",
    "id":1
}

Variabel {sid} diperoleh pada langkah kedua dan akan digunakan terus-menerus, di mana perlu untuk bekerja dengan antarmuka API.

Final 4 LANGKAH - pembaruan makro


Sekarang kita tahu ID makro yang perlu diperbarui, cookie otorisasi, atau versi firmware router. Anda dapat memperbarui makro itu sendiri.

POST http://company.com/zabbix/api_jsonrpc.php HTTP/1.1
Content-Type: application/json-rpc

{
    "jsonrpc":"2.0",
    "method":"usermacro.update",
    "params":{
        "hostmacroid":"{hostmacroid}",
        "value":"{mikrotik_version}"
    },
    "auth":"{sid}",
    "id":1
}

{mikrotik_version} - nilai yang diperoleh di langkah pertama. Dalam contoh saya - versi firmware mikrotik
{hostmacroid} saat ini - nilai diperoleh pada langkah ketiga - id makro, yang kami perbarui.

kesimpulan


Pendekatan untuk memecahkan masalah dengan fungsionalitas reguler jauh lebih rumit dan lebih lama. Terutama jika Anda tahu pemrograman dan dapat dengan cepat membuang logika yang diperlukan dalam skrip.

Keuntungan nyata dari pendekatan ini adalah "portabilitas" dari solusi antara server yang berbeda.
