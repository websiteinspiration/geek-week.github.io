今私はほとんど誰も彼らのメールを暗号化しない理由を理解しています



電子メールの暗号化は困難で苦痛な手順です。最近、私自身がどれだけ気づいたのですか。情報セキュリティの分野で非常に進んだ私の友人が私に彼女のPGP公開鍵を送り、暗号化に切り替えるように頼みました。いいえ、彼女はNSAやCIAの出身ではなく、彼女のプライバシーを気にする普通の人です。これまで暗号化された手紙を送ったことはありませんでしたが、「なぜそうでないのですか？」長年私はこれを学びたかったが、暗号化された手紙を交換する人がいなかった。

LinuxマシンにGnuPGをインストールすることから始めました。

GnuPGは、暗号化と復号化に公開鍵と秘密鍵を使用するという点でPGPに似ていますが、オープンソースコードと多くのLinuxディストリビューションが付属しています。 PGPの別のオープンソースバージョンはOpenPGPです。

Gpg4winというGnuPG for Windowsは公式サイトからダウンロードできます。メッセージを暗号化および復号化する一般的な方法を選択しました。これは、1つのプロバイダーのみを暗号化する機能に制限したくなかったため、どの電子メールプロバイダーにも関連付けられていません。さらに、メールプロバイダーもあなたの手紙にアクセスできないことを確認する唯一の方法は、自分でそれらを暗号化することです。How-To Geekで最も包括的なGnuPGガイドを見つけました。

次のコマンドを使用して友達の公開鍵をインポートしました：

gpg --import her_public_key_file.key

このコマンドher_public_key_file.keyは、実際の公開鍵ファイルに置き換えられます。次に、彼女のキーが正常にインポートされたことを確認しました。

gpg --list-keys

このようなメッセージが表示されました：

pub 2048R/FFE7947D 2019-10-11 [expires: 2021-10-10]
uid her_email@her_email_provider.com
sub 2048R/AB48FEC2 2019-10-11

これは、GnuPGがその公開鍵を有効であると認識し、将来の使用のために保存することを意味します。

次に、彼女の公開鍵に署名しました：

gpg --sign-key her_email_address@email_provider.com

公開鍵に署名すると、GnuPGにあなたがこの鍵を信頼している、つまり、本当にこの人からのものであることが通知されます。この公開鍵の所有者からのメッセージを復号化する前に、各公開鍵に署名する必要があります。私の意見では、このステップは意味がありません。適切な人物からのキーではないと思われるのに、なぜキーをインポートするのですか？

次に、キーを生成しようとしました：

gpg --output ~/temp.key --armor --export My Name

まあ、そのようなコマンドの後、新しい行の終わりが見つかったというエラーメッセージがポップアップしました。これは私の名前のスペースと、名前とメールアドレスの間にあるスペースが原因であると考えたので、さまざまな形式を試しました。約1時間と多くの失敗した試行の結果、そのうちのいくつかは成功したように見えましたが、成功しなかったため、ようやく正しいコマンドを見つけました。

gpg --gen-key

上記のコマンドでは、一連の質問が表示されます。最初はオプション「1」を選択し、次に2048ビットのキー長のオプションを選択しました。無制限のキーの有効期間を設定します。名前フィールドに姓を指定しませんでしたが、そこでスペースもサポートされています。名前と姓の両方を指定できます。次に、彼のメールアドレスを入力しました。コメントフィールドは空白のままです。そして最後に、彼は長いパスフレーズを導入しました。パスフレーズは、単にパスワードの同義語です。失われないように書き留めました。実際、Truecryptを使用して暗号化したファイルに入れました。ハードドライブにGnuPGを再インストールしてキーを再インポートする場合にメッセージを暗号化するために、後でパスフレーズが必要になります。そして、キー自体をバックアップすることを忘れないでください！

キーを作成したら、結果を確認します。

gpg --list-keys

このコマンドは次のようなものを生成します：

pub 2048R/FFE7947D 2019-10-11 [expires: 2021-10-10]
uid her_email@her_email_provider.com
sub 2048R/AB48FEC2 2019-10-11


pub 2048R/3A785D3F 2020-02-22
uid My Name
sub 2048R/A7B384FE 2020-02-22

この結果が公開/秘密鍵のペアがあるのか​​、それとも1つの鍵しかないのかわからなかったので、鍵生成を2回実行しました。どういうわけか、最終的に2つのキーペアを作成したことに気付いたので、2つ目のペアを削除しました。

次に、公開鍵をファイルにエクスポートしようとしました。公開鍵は、暗号化されたレターを送信するすべての人に送信して、解読できるようにする必要があります。使用したコマンドはわかりませんが、で始まるテキストファイルが生成されました-----BEGIN PGP PUBLIC KEY BLOCK----- Version: GnuPG v1。私はこの手紙を私の友人に送りました、そして、彼女は暗号化されたメッセージで私に答えました。

私はmessage.asc彼女の手紙でファイルを解読しようとしました：

gpg --decrypt message.asc > plain.txt

しかし、GnuPGは秘密鍵を見つけられなかったというエラーメッセージを出しました。

gpg: encrypted with RSA key, ID XXXXXXXX
gpg: decryption failed: secret key not available

しかし、私は秘密鍵を見ました。それをエクスポートすることもできました。私は彼がそうだと知っていました！私が言えることは、プログラマーが本当にひどいエラーメッセージを書くことです。この時点で、私はすでにいくつかの苛立ちを経験し始めていました。

何を間違えたかわかりませんでした。私が推測できる唯一のことは、どういうわけか私はキーを正しく作成できなかったということでした。それか、間違った公開鍵をエクスポートしました。だから私は最初からやり直すことにしました。次のコマンドを使用して、キーペアを削除し、新しいキーペアを生成して、公開キーをファイルにエクスポートしました。

gpg --output ~/my_public_gpg_key.key --armor --export My Name my-email@my-email-povider.com

それから彼は友人に新しい公開鍵を送りました。パラメータ--armorは、テキスト形式で公開鍵ファイルを作成するようにGnuPGに指示します。ファイル名- my_public_gpg_key.key。

彼女は再びメッセージを暗号化して私に送りました。それを解読しようとしたとき、私は再び同じエラーメッセージを見ました：

gpg: encrypted with RSA key, ID XXXXXXXX
gpg: decryption failed: secret key not available

今回、キー識別子が新しい公開キーではなく、古い公開キーと一致することに気付きました。私は彼女が間違っていると判断し、新しい鍵でもう一度試すように頼みました。次に、自分が間違っていないことを確認するために、使用したキーが本当に新しいキーであることを確認することにしました。送信した新しいキーは、公開キーの1つでもありませんでした。で始まったのは彼女の公開鍵でした-----BEGIN PGP PUBLIC KEY BLOCK----- Version: GnuPG v1！どういうわけか私は彼女の鍵を追加してエクスポートしましたVersion: GnuPG v1！私がこれを知っているのは、彼女が送信した彼女のキーがまだファイルにあり、それがで始まるため-----BEGIN PGP PUBLIC KEY BLOCK-----です。ありませんVersion: GnuPG v1！そして彼女の公開鍵は私が送ったものと一致します。

上記と同じコマンドを使用して、公開鍵を再エクスポートしました。今回は、自分の鍵であることをメールで送信する前に確認しました。インターネットGnuPGの参考資料から、人々が異なるバージョンのGnuPGを使用している場合、秘密鍵がないというメッセージが表示される可能性があることを知りました。メールで新しいキーを送信したときに、このことを彼女に伝えました。

数週間後、私は返答の手紙を受け取りました。この時までに、私は彼女が降伏したとすでに決めていました。次の数週間、コンピュータに問題が発生し、ハードドライブを再フォーマットする必要がありました。そのため、GnuPGを再インストールし、キーを再インポートして、その公開キーを再インポートする必要がありました。幸いにも、バックアップを作成しました。それから私は彼女の最後の手紙にこのコマンドで暗号化した別の手紙で答えました：

gpg --encrypt --sign --armor -r her-email@her-email-provider.com --passphrase my-pass-phrase my-msg.txt

もちろん、これは私の本当のパスワードではありません。私はそれをに置き換えmy-pass-phrase、暗号化されていないメッセージはに置き換えましたmy-msg.txt。暗号文が生成され、そのファイルに名前が付けられましたmy-msg.tx.gpg。自分の暗号文を自分で復号化する方法に関する情報が見つからなかったため、正しく暗号化したかどうかわからずに、彼女に送信する必要がありました。私は彼が適切なサイズのように見えることを知っていました。数週間後、私は彼女から別のトピックに関する新しい暗号化された手紙を受け取りました。それで初めて、暗号化された電子メールを介して正常に通信できるようになりました。最初から最後までのプロセス全体が1か月以上かかりました！私はまだGnuPGコマンドを自動化しようとしているので、自分が何をしているかについてあまり考えずにそれらを使用できます。

この経験から、暗号化された電子メールは興味深い演習ではありますが、あまり実用的ではないことに気付きました。あなたがNSAによって監視されているEdward Snowdenでない限り、暗号化メールを作成して使用するための努力を正当化することはできそうにありません。そして、あなたが重要な秘密を持つエドワード・スノーデンでないなら、誰もあなたとの通信を暗号化することにエネルギーを費やすことを望んでいないでしょう。

電子メールの暗号化と復号化は、私がGnuPGで行っているように、非常に面倒です。私はすでにこれに多くの労力を費やしているので、オンラインの友人と通信するためにこれを使い続けます。しかし、これを回避できるのであれば、普通の人がGnuPGを使用して電子メールを暗号化することはお勧めしません。Protonmailなどの特定のメールプロバイダーからのより簡単なソリューションが市場にあることを理解していますが、ほとんどの人はそれらを使用するためにプロバイダーを変更したくないでしょう。メールプロバイダーを変更せずに暗号化メールを送受信する簡単な方法がすべて必要です。GnuPGはソリューションではありません。メールを暗号化するより簡単な方法を知っている場合は、コメントを残すことを検討してください。
