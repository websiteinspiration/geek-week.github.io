Golang配列とスライスの完全ガイド

記事の翻訳は、今日始まるクラスであるGolang Developerコースの学生のために特別に準備されました。



最初は、配列とスライスを同じものとして簡単に認識できますが、名前は異なります。どちらもコレクションを表すためのデータ構造です。しかし、実際にはそれらは非常に異なっています。

この記事では、Goでの違いと実装について説明します。

例を参照して、それらをどこに適用するかについてより情報に基づいた決定を行えるようにします。

配列


配列は、固定サイズのコレクションです。ここで重要なのは固定サイズです。これは、配列の長さを設定するとすぐに変更できないためです。

例を見てみましょう。4つの整数値の配列を作成します。

arr := [4]int{3, 2, 5, 4}


長さとタイプ


上記の例では、変数はarrタイプの配列として定義されてい[4]intます。つまり、配列は4つの要素で構成されています。サイズは4型定義に含まれていることに注意することが重要です。

このことから、実際には、長さが異なる配列は、タイプが異なる配列であることがわかります。この場合、異なる長さの配列を互いに同一視することはできず、ある配列の値を別の配列の値に割り当てることはできません。

longerArr := [5]int{5, 7, 1, 2, 0}

longerArr = arr
// This gives a compilation error

longerArr == arr
// This gives a compilation error


配列は構造の観点から話しやすいことがわかりました。配列に似た構造を作成しようとすると、おそらく次のようになります。

// Struct equivalent for an array of length 4
type int4 struct {
  e0 int
  e1 int
  e2 int
  e3 int
}

// Struct equivalent for an array of length 5
type int5 struct {
  e0 int
  e1 int
  e2 int
  e3 int
  e5 int
}

arr := int4{3, 2, 5, 4}
longerArr := int5{5, 7, 1, 2, 0}

実際には、これは推奨されませんが、これは、長さが異なる配列が異なる型の配列である理由を理解するための良い方法です。


メモリ表現


配列はn、特定のタイプのブロックのシーケンスとして格納され



ます。このメモリは、配列変数を初期化するときに割り当てられます。

リンクで渡す


Goには参照渡しなどの機能はありませんが、すべて値で渡されます。配列の値を別の変数に割り当てると、割り当てられた値がコピーされます。



配列への「リンク」のみを渡したい場合は、ポインタを使用します。



メモリを割り当てるときや関数内では、配列は実際には単純なデータ型であり、構造体とほとんど同じように機能します。

スライス


スライスは、配列の拡張実装と見なすことができます。
スライスはGoに実装され、コレクションの動的なサイズ変更など、開発者がコレクションを操作するときに遭遇する非常に一般的な使用例のいくつかをカバーしています。

スライス宣言は、長さ指定子が省略されていることを除いて、配列宣言と非常に似ています。

slice := []int{4, 5, 3}


コードだけを見ると、スライスと配列は非常に似ているようですが、主な違いは実装と使用条件にあります。

メモリ表現


スライスは配列とは異なる方法で割り当てられ、基本的には変更されたポインタです。各スライスには、3つの情報ブロックが含まれています。

  1. データのシーケンスへのポインター。
  2. 長さ（長さ）。スライスに現在含まれている要素の数を決定します。
  3. 容量（容量）、提供されるメモリセルの総数を決定します。




したがって、異なる長さのスライスを互いに割り当てることができます。それらは同じタイプであり、ポインター、長さ、およびボリュームは異なる場合があります。

slice1 := []int{6, 1, 2}
slice2 := []int{9, 3}

// slices of any length can be assigned to other slice types
slice1 = slice2


スライスは、配列とは異なり、初期化中にメモリを割り当てません。実際、スライスはnull（nil）値で初期化されます。

リンクで渡す


スライスを別の変数に割り当てるときも、値を渡します。ここでは、値はポインタ、長さ、およびボリュームのみを参照し、要素自体が占有するメモリは参照しません。



新しいアイテムを追加する


新しい要素をスライスに追加するには、関数を使用する必要がありますappend。

nums := []int{8, 0}
nums = append(nums, 8)


内部的には、新しい要素に指定された値を割り当て、その後新しいスライスを返すように見えます。新しいスライスの長さがもう1つになります。



要素を追加するときに、長さが1つ増えて宣言されたボリュームを超える場合は、新しいボリュームを提供する必要があります（この場合、現在のボリュームは通常2倍になります）。

そのため、事前に指定された長さとボリュームでスライスを作成することが最も推奨されます（特に、必要なスライスのサイズが明確にわかっている場合）。

arr := make([]int, 0, 5)
// This creates a slice with length 0 and capacity 5


使用するもの：配列またはスライス？


配列とスライスはまったく異なるものであるため、それらの使用例も異なります。

何をいつ使用するかを理解するために、いくつかのオープンソースの例とGo標準ライブラリを見てみましょう。

ケース1：UUID


UUIDは、オブジェクトまたはエンティティをマークするためによく使用される128ビットのデータです。通常、それらはダッシュで区切られた16進値として表されます。

e39bdaf4-710d-42ea-a29b-58c368b0c53c


Google UUID ライブラリでは、UUIDは16バイトの配列として表されます。

type UUID [16]byte

UUIDが128ビット（16バイト）で構成されていることがわかっているため、これは理にかなっています。UUIDにバイトを追加したり削除したりすることはないので、それを表すために配列を使用することになります。

ケース2：整数値の並べ替え


この例ではsort.Ints、ソート標準ライブラリの関数を使用します。

s := []int{5, 2, 6, 3, 1, 4} // unsorted
sort.Ints(s)
fmt.Println(s)
// [1 2 3 4 5 6]


この関数sort.Intsは、整数のスライスを受け取り、値の昇順で並べ替えます。2つの理由から、ここでスライスを使用することをお勧めします。

  1. 整数の数は指定されていません（ソートする整数の数は任意です）。
  2. 番号は昇順で並べ替える必要があります。配列を使用すると、整数のコレクション全体が値として確実に渡されるため、関数は渡されたコレクションではなく、独自のコピーをソートします。


結論


配列とスライスの主な違い、およびそれらの使用例を見てきたので、使用する設計を簡単に決定できるように、いくつかのヒントを紹介します。

  1. エンティティが、固定長の空でない要素のセットで記述されている場合は、配列を使用します。
  2. 追加またはコレクションからアイテムを削除するコレクションを説明するときは、スライスを使用します。
  3. コレクションに任意の数の要素を含めることができる場合は、スライスを使用します。
  4. コレクションを何らかの方法で変更しますか？その場合は、スライスを使用する必要があります。


ご覧のとおり、スライスはGoアプリを構築するためのほとんどのシナリオをカバーしています。ただし、配列には存在する権利があり、さらに、特に適切なユースケースが出現した場合には、それらは非常に有用です。
