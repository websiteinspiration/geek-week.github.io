一般情報



オイルパンは、マークアンドスイープアルゴリズムを使用するガベージコレクションシステムを使用しています。このアルゴリズムは、ガベージコレクションプロセスを2つのフェーズに分割します。最初のフェーズは、ヒープの調査と、メモリから削除できない「ライブ」オブジェクトのマーク付けです。 2番目のフェーズは、不要な（「デッド」）オブジェクトによって占有されているヒープメモリのスイープです。



アルゴリズムの実装のマークフェーズについてはすでに書きました。そのマテリアルの主なアイデアを思い出すと、「生きている」オブジェクトの検索は、オブジェクトがノードであり、オブジェクトが相互に参照しているポインタがグラフの端であるグラフのバイパスと見なすことができることがわかります。ウォーキングはルートエンティティから始まりますこれらは、「生きている」ことが事前にわかっているオブジェクトです。たとえば、これはグローバルオブジェクトであり、アクティブな機能です。



この点で、JavaScriptのC ++も同じです。確かに、JavaScriptオブジェクトとは異なり、C ++オブジェクトは静的に型付けされます。その結果、プログラムの実行中に自分のビューを変更することはできません。 Oilpanを使用してC ++オブジェクトを操作する場合、この事実が考慮され、他のオブジェクト（グラフエッジ）へのポインターの説明がVisitorパターンを使用して提供されます。 Oilpanオブジェクトの記述に使用される基本的なパターンは次のとおりです。 



class LinkedNode final : public GarbageCollected<LinkedNode> {
 public:
  LinkedNode(LinkedNode* next, int value) : next_(next), value_(value) {}
  void Trace(Visitor* visitor) const {
    visitor->Trace(next_);
  }
 private:
  Member<LinkedNode> next_;
  int value_;
};

LinkedNode* CreateNodes() {
  LinkedNode* first_node = MakeGarbageCollected<LinkedNode>(nullptr, 1);
  LinkedNode* second_node = MakeGarbageCollected<LinkedNode>(first_node, 2);
  return second_node;
}

C ++のメモリをクリアする



オイルパンの作業中にオブジェクトがどのようにマークされるかについては、別の機会に説明します。ここで、オブジェクトのマーキングがすでに完了しており、Oilpanがオブジェクトメソッドを使用して到達可能なすべてのオブジェクトをすでに検出しているという前提から進みますTrace



。タグ付けフェーズが完了すると、対応するビットが到達可能なオブジェクトに設定されます。



クリーニングは、死んだオブジェクトが破壊されるときのオイルパンの作業のフェーズです（オブジェクトをマークするフェーズでは到達できなかったオブジェクト）。それらが占有するメモリは、オペレーティングシステムに返されるか、他のオブジェクトへの割り当てに使用できるようになります。以下では、オイルパンがメモリクリーニングを実行する方法について説明します。このプロセスの機能とその制限について検討します。さらに、オイルパンのメモリクリーニングメカニズムが、死んだオブジェクトを破壊するときに高速を達成する方法について説明します。



メモリクリーニングシステムは、ヒープメモリをソートしてオブジェクトの対応するビットをチェックすることにより、死んだオブジェクトを見つけます。 C ++のセマンティクスを保持するために、メモリクリーニングシステムは、占有しているメモリを解放する前に、「死んだ」オブジェクトのそれぞれのデストラクタを呼び出す必要があります。重要なデストラクタはファイナライザとして実装されています。



ここでは、プログラマーの観点から、メモリクリーニングシステムで使用される検索メカニズムは破棄されたオブジェクトの作成順序を考慮していないため、デストラクタが呼び出される所定の順序はありません。これにより、ファイナライザに制限が課せられ、ヒープ上の他のオブジェクトにアクセスできなくなります。私たちは非常に頻繁に遭遇する課題に直面しています。それは次のように構成されます。オブジェクト（Javaなど）のファイナライズの順序の指定をサポートしないプラットフォームがありますが、同時に、このプラットフォームでは、ファイナライザを呼び出す特定の順序を必要とするコードを記述する必要があります。オイルパンはClangプラグインを使用します。静的モードでは、オブジェクトの破棄中にヒープオブジェクトへのアクセスを禁止します（これはClangの多くの機能の1つにすぎません）。 



class GCed : public GarbageCollected<GCed> {
 public:
  void DoSomething();
  void Trace(Visitor* visitor) {
    visitor->Trace(other_);
  }
  ~GCed() {
    other_->DoSomething();  // error: Finalizer '~GCed' accesses
                            // potentially finalized field 'other_'.
  }
 private:
  Member<GCed> other_;
};


別のオブジェクトの破棄中にヒープからオブジェクトにアクセスしようとすると、このコードでエラーが発生します。



オブジェクトの破棄の問題に関心がある場合は、Oilpanがプログラマにファイナライザの前に呼び出されるコールバックを提供することを知ってください。これにより、オブジェクトが破棄される前に他のオブジェクトにアクセスする必要がある場合に、オブジェクトを破棄するための複雑なスキームを実装できます。このようなコールバックにより、ガベージコレクション中にシステムにかなりの負荷が追加されます。この負荷は、デストラクタが作成できるものよりも大きくなります。したがって、このようなコールバックがBlinkで使用されることは比較的まれです。

インクリメンタルメモリのクリア



インクリメンタルアプローチを使用する場合、オブジェクトのマーキングと破棄のフェーズは互いに分離されます。同時に、メモリクリーニング手順はいくつかの部分に分かれており、メインスレッドで実行される個別のタスクで表されます。最良の場合、このようなタスクはダウンタイム中にのみ実行されます。通常のアプリケーションメカニズムを表すタスクとの競合を回避するシステム。ガベージコレクターの内部メカニズムは、メモリを小さなタスクにクリーンアップするという1つの大きなタスクを共有します。この分離は、「ページ」の概念に基づいています。ページには2つの状態があります。一部のページは、クリアされる（スイープされる）の待機状態にありますが、他のページは既にクリアされています（すでにスイープされています）。メモリ割り当てメカニズムは、既にクリアされているページのみを考慮し、使用可能なメモリフラグメントのリストを格納する空きメモリリストからローカルメモリ割り当てバッファ（ローカル割り当てバッファ、LAB）を補充します。メモリ（空きメモリのリストに含まれている情報）を取得するために、アプリケーションはまず、すでにクリアされているページを参照しているメモリを見つけようとします。次に、アプリケーションは、システムがメモリの割り当てを試みる場所であるメモリクリーニングシステムを使用して、クリーニングを待機しているページの処理を支援しようとします。上記のアクションの結果、アプリケーションが必要なメモリを受信できなかった場合にのみ、新しいメモリがオペレーティングシステムから要求されます。



オイルパンでは、インクリメンタルメモリークリーニングメカニズムが数年間使用されてきました。しかし、アプリケーション、つまりオブジェクトのグラフがますます増えるにつれて、メモリのクリアがアプリケーションのパフォーマンスに影響を及ぼし始めました。ガベージコレクションのメカニズムを改善するために、バックグラウンドタスクを使用して、競合するメモリクリーニングを実装し始めました。メモリクリーニングを実行するバックグラウンドタスクと新しいオブジェクトにメモリを割り当てるアプリケーションとの間の「データの競合」を解消するために、2つのルールがあります。



  • メモリクリーニングシステムは、「デッド」オブジェクトのメモリのみを処理します。これは、定義上、アプリケーションからは達成できません。
  • アプリケーションは、クリアされたページに属するメモリのみを使用してメモリを割り当てます。そのようなページは、もはやメモリクリーニングシステムに関心がありません。


どちらのルールでも、同じオブジェクトとそのメモリについて、複数のエンティティが互いに競合しないようにすることができます。残念ながら、C ++はファイナライザとして実装されているデストラクタに大きく依存しています。オイルパンはメインスレッドでファイナライザを実行して、開発者を支援し、アプリケーションコード自体のデータ競合を排除します。この問題を解決するために、Oilpanはメインスレッドでオブジェクトのファイナライザを延期することを計画しています。より具体的には、競合するメモリクリーニングシステムは、ファイナライザ（デストラクタ）を持つオブジェクトに遭遇すると、それをファイナライズキューに入れ、ファイナライズの特別な段階で処理されます。この段階のコードは、常にメインスレッドで実行されます。メインスレッドでは、アプリケーションコードも実行されます。ここでは、一般的な用語で、競争力のあるメモリ消去スキームのように見えます。
