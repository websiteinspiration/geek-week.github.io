デバッグ領域は、XcodeでのiOS開発者の作業に役立つ機能です。 iOSの開発を習得し始め、使い慣れた印刷方法から離れて、特定の期間におけるシステムの状態を理解するためのより高速でより便利な方法を見つけたら、デバッグ領域の調査を開始します。

ウォッチポイント


デバッガーのもう1つの便利な機能は、プログラムのウォッチポイントの値を追跡することです。ウォッチポイントはKVOに似ています。ブレークポイントを設定してオブジェクトの状態を変更します。オブジェクトの状態が変更されるたびに、プログラムの実行プロセスが停止し、値が変更された場所と場所を確認できます。たとえば、セルにウォッチポイントを設定して、テーブルがページングされ、新しいセルが初期化されるとどうなるかを調べます。コマンドのリストが非常に大きいことが判明したので、セル内のレイアウトビューの実行と制約の設定、アニメーション、セルの状態の設定など、いくつかだけは触れません。

ウォッチポイントを値に設定するには、監視するプロパティのスコープでブレークポイントプログラムを停止し、[デバッグ変数]パネルでプロパティを選択して、[<パラメータ>]を監視する必要があります。



変数を持つウォッチポイントを削除するには、ブレークポイントナビゲーターを調べる必要があります。そこに、残りのブレークポイントと共にウォッチポイントがあります。

ブレークポイントUIの変更


反発しようとしているオブジェクトの詳細を調べる必要がある場合があります。最も簡単なオプションは、「po」を使用してオブジェクトに関する情報を表示し、メモリ内のオブジェクトの場所を同じ場所で確認することです。しかし、オブジェクトへの直接リンクがない場合があり、それはAPIビューに表示されず、その上にあるか、ライブラリによって隠されている可能性があります。オプションの1つはビュー階層を使用することですが、これは常に便利であるとは限らず、目的のビューが見つかったことを理解することは必ずしも難しいことではありません。次のコマンドを使用してみてください。

expression self.view.recursiveDescription()

これはObj-Cにありますが、言語の特殊性のためにSwiftで削除されたため、実行できませんが、DebuggerはObj-Cで動作するため、理論的にはこのコマンドをフィードでき、彼はあなたが何を望んでいるかを理解できます。コンソールでObj-Cコードを実行するには、次のコマンドを入力する必要があります。

expression -l objc -O - - [`self.view` recursiveDescription]

ここには何が見えますか？時間が経つにつれて慣れる可能性があるかなり不便な構造が表示されますが、実行しない方がいいですが、コマンドを単純化するためにタイプエイリアスを使用します。

command alias poc expression -l objc -O —

現在、私たちのチームは縮小および簡素化していますが、作業を続けています。

poc [`self.view` recursiveDescription]
