ACIDでのマーケティングはどのくらいですか？

みなさん、こんにちは。ウラジスラフ・ロダンと接触。現在、私はOTUSのHigh Load Architectコースの責任者を務めており、ソフトウェアアーキテクチャのコースも教えています。

教えることに加えて、お気づきかもしれませんが、私はハーバートのOTUSブログの著作権資料も書いています。今日募集中の「データベース」コースを開始するための本日の記事に合わせたいと思います。



序文


トランザクションは70年代に登場し、競争環境におけるフォールトトレランスとデータアクセスの問題を解決するためのデータベースツールとして提供されました。次に、割り当てられたタスクを実行するためにトランザクションが所有しなければならないいくつかのプロパティが定式化され、これらのプロパティの大文字が正しい順序で設定されて、美しい頭字語ACIDを構成しました。

これらのイベントが発生した期間は、高負荷、インターネット、およびパフォーマンスの問題がないことを特徴としており、垂直スケーリング方法によってのみ解決できました。その後、2000年代の初めに、NoSQLデータベースにトレンドが現れ、BASEという略語が登場しました。現在、ACIDには逆の傾向があります。 NoSQLのMongoDBもACIDをサポートしています。

この略語が何を意味するか、そしてどれほどのマーケティングがあるか見てみましょう。

ACIDは4つのプロパティを表します

。A=原子性
C =一貫性
I =分離
D =耐久性

各プロパティについて個別に説明します。

A =原子性


原子性は過負荷の用語であり、データベース内のトランザクションのコンテキストでは、「オールオアナッシング」の原則として定式化できます。トランザクションに10個の挿入操作が含まれている場合、10個すべてが実行される（コミットトランザクションが実行されます）か、どれも実行されません（ロールバックトランザクションが実行されます）。

このプロパティはどのように提供されますか？実際には、同じ10個の挿入を含むトランザクションがデータベースに来ても、データは変化し始めません。トランザクションの前に、トランザクションはログ内のデータに直接書き込まれ、そこで行われた変更が記録されます。このジャーナルは、データを複製するために使用することもできますが、たとえば、ここで説明したように、それと完全に無関係にすることもできます。。このジャーナルのおかげで、トランザクションはデータに直接コミットできます。その場合は、トランザクションをロールバックできます。トランザクションをできるだけ早く閉じるというルールは、このプロパティを実装するための原則の理解に直接従います。データベースは、トランザクションが開いている間はこのログをクリーンアップすることが禁じられているため、ログが詰まり、非常に不愉快な結果につながる可能性があります。

C =整合性（整合性または整合性）


ACIDに関しては、一貫性はCAPの定理と同じ意味ではありません（分散システムの理論では、この一貫性には多くの程度があります）。一貫性とは、次のことを意味します。システムのいくつかの事前定義された不変条件は、コミットトランザクションの前後の両方で実行する必要があります。システムの不変条件の例を表すことができます。借方はローンと収束し、従業員の総給与は予算を超えません。会社の従業員数は、以前にオープンした必要要員の数と同じです。データベース言語では、これはすべての制約の履行のみを意味します。

制約の存在がビジネスロジックの一部がアプリケーションからデータベースレベルに移行したことを示している可能性があるため、データベースレベルでの一貫性の必要性についての質問はかなり物議を醸しています。最終的に、アプリケーション自体が無効なデータをコミットしないことを決定する場合があります。一貫性は略語を美しくするためだけに追加されたという意見があります（マーケティング）。

I =分離


分離は、並列トランザクションを順次として実行できるデータベースプロパティです。結局のところ、データベースが同時に複数のトランザクションを実行することを禁止する人はいません。それらが互いに影響を及ぼさないことを確認して、競合状態タイプの異常が発生しないようにする必要があります。実際、競争の激しい環境でのデータアクセスの問題を解決するのは分離です。

そのボリュームのために分離の概念を開示することは、別の記事に値する。なぜなら、ACIDの心臓部と言えるのは、まさに分離だからです。トランザクション性の価格は、多くの場合、分離を保証する価格まで低下します。そのため、分離にはさまざまなレベルがあり、それぞれに競合状態に対する独自のレベルの保護を提供し、1つまたは別のオーバーヘッドをもたらします。分離は完全にシリアライズ可能なレベルでのみ保証されます。これは実装が非常に難しく、めったに必要とされません。

D =耐久性


耐久性は、トランザクションが適用された場合、決して消えてはならないことを示しています。実際、これは次のことを意味します。データベースがトランザクションがコミットされたと応答した場合、トランザクションは不揮発性メモリにコミットされました。これは、fsyncシステムコールが発生したことを意味します。つまり、バッファがハードドライブにフラッシュされ、正常に応答しました。

耐久性は完全には提供できないため、マーケティング用語でもあります。 「エイリアンによって地球を燃やす」という人為的なシナリオを破棄しても、その後データベースやトランザクションはなくなり、トランザクションが記録された特定のハードドライブが物理的に破壊される可能性は高くなりますが、極端なシナリオになると、fsyncシステムコールが保証することを思い出すことができます。コントローラのハードドライブをたたきます。これには、まだ揮発性バッファが含まれています。それに費やされる時間は短いですが、0とは異なります。その結果、「適切な」瞬間に正確に電源をオフにすると、トランザクションが依然として失われる可能性があります。

fsyncの問題は高価なOracleデータベースで解決されていますが、マシンの物理的な破壊に関連する問題から私たちを保護できるものはありません。保証を増やすことができるのは、バックアップとレプリケーションのみです。

調査結果


ACIDは原子性や分離などの非常に興味深いプロパティを提供するという事実にもかかわらず、これらのプロパティの一部は単なるマーケティングであり、BASEと比較してその厳密さで知られていますが、ACIDはトランザクション損失の機会がないことと結果の影響を完全に保証していませんお互いに同時にトランザクションを実行します（分離はまだ構成する必要があります！）。
