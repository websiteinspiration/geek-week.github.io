RutokenでGOST-2012キーを使用してLinuxでローカル認証にPAMモジュールを使用する方法



単純なパスワードでは保護されず、複雑なパスワードは記憶できません。そのため、キーボードの下のステッカーやモニターによく見られます。パスワードを「忘れやすい」ユーザーの頭に留めておき、保護の信頼性が失われないようにするために、2要素認証（2FA）があります。

デバイスの所有権の要素とそのPINコードの知識の組み合わせにより、PINコード自体はよりシンプルで覚えやすくなります。PINコードの長さまたはランダム性の欠如は、PINコードの検索に対する物理的な所有権および制限の要件によって補われます。

さらに、州の機関では、GOSTに従ってすべてが機能することを望んでいます。Linuxに移行するためのこのオプション2FAについては、後で説明します。遠くから始めます。

PAMモジュール


プラグ可能認証モジュール（PAM）は、標準APIと、アプリケーションでのさまざまな認証メカニズムの実装を備えたモジュールです。
PAMと連携できるすべてのユーティリティとアプリケーションは、それらを取得し、それらを使用してユーザーを認証できます。
実際には、これは次のように機能します。loginコマンドはPAMになり、構成ファイルで指定されたモジュールを使用して必要なすべてのチェックを実行し、結果をloginコマンドに返します。

librtpam


Active社によって開発されたモジュールは、最新の国内暗号化標準に従って、非対称キーを使用したスマートカードまたはUSBトークンによるユーザーの2要素認証を追加します。

その仕事の原則を考えてみましょう：
  • トークンはユーザーの証明書とその秘密鍵を格納します。
  • 証明書は、信頼できるものとしてユーザーのホームディレクトリに保存されます。


認証プロセスは次のとおりです。
  1. Rootokenはユーザーの個人証明書を検索します。
  2. トークンPINが要求されます。
  3. ランダムデータは、Rutokenチップで直接秘密鍵に署名されます。
  4. 受信した署名は、ユーザー証明書の公開鍵を使用して検証されます。
  5. モジュールは、署名検証の結果を呼び出し側アプリケーションに返します。


キーGOST R 34.10-2012（長さ256または512ビット）または古いGOST R 34.10-2001を使用して認証できます。

キーのセキュリティについて心配する必要はありません。キーはRutokenで直接生成され、暗号化操作中にメモリを決して離れません。



Rutoken EDS 2.0は、NDV 4に従ってFSBおよびFSTECによって認定されているため、機密情報を処理する情報システムで使用できます。

7）認証を構成する


PAMモジュールの構成は完全に標準であり、他のモジュールの設定と同じように行われます。/usr/share/pam-configs/rutoken-gost-pamモジュールの完全な名前（デフォルトで含まれるかどうか、モジュールの優先順位、および認証パラメーター）を含むファイルを作成します。
認証パラメーターには、操作を成功させるための要件が​​あります。
  • 必須：このようなモジュールは肯定的な応答を返す必要があります。モジュール呼び出しの結果に否定の回答が含まれている場合、認証エラーが発生します。リクエストはリセットされますが、残りのモジュールが呼び出されます。
  • requisite（必須）：requiredに似ていますが、すぐに認証が失敗し、残りのモジュールは無視されます。
  • 十分：そのようなモジュールの前に、必要なモジュールまたは十分なモジュールのいずれも否定的な結果を返さなかった場合、モジュールは肯定的な答えを返します。残りのモジュールは無視されます。
  • オプション（オプション）：スタックに必要なモジュールがなく、十分なモジュールがどれも肯定的な結果を返さなかった場合、オプションのモジュールの少なくとも1つが肯定的な答えを返す必要があります。

9）トークンを抽出するときにコンピューターのロックを構成する


パッケージにlibpam-pkcs11は、pkcs11_eventmgr,PKCS＃11イベントが発生したときにさまざまなアクションを実行できるユーティリティが含まれています。
構成するにpkcs11_eventmgrは、構成ファイルを使用します。/etc/pam_pkcs11/pkcs11_eventmgr.conf
さまざまなLinuxディストリビューションでは、スマートカードまたはトークンを削除するときにアカウントをロックするコマンドが異なります。を参照してくださいevent card_remove。
構成ファイルの例を以下に示します。

pkcs11_eventmgr
{
    #   
    daemon = true;
     
    #   
    debug = false;
 
    #    
    polling_time = 1;
 
    #  -   
    # - 0
    expire_time = 0;
 
    #  pkcs11     
    pkcs11_module = usr/lib/librtpkcs11ecp.so;
 
    #   
    #  :
    event card_insert {
        #     (  )
        on_error = ignore ;
 
        action = "/bin/false";
    }
 
    #  
    event card_remove {
        on_error = ignore;
         
        #    
        
        #  GNOME 
        action = "dbus-send --type=method_call --dest=org.gnome.ScreenSaver /org/gnome/ScreenSaver org.gnome.ScreenSaver.Lock";
        
        #  XFCE
        # action = "xflock4";
        
        #  Astra Linux (FLY)
        # action = "fly-wmfunc FLYWM_LOCK";
    }
 
    #    
    event expire_time {
        #     (  )
        on_error = ignore;
 
        action = "/bin/false";
    }
}


その後、アプリケーションpkcs11_eventmgrをスタートアップに追加します。これを行うには、.bash_profileファイルを編集します。
$ nano /home/<_>/.bash_profile
行のpkcs11_eventmgrをファイルの最後に追加して、再起動します。

説明されているオペレーティングシステムの設定手順は、国内向けのものを含む最新のLinuxディストリビューションでの手順として使用できます。
