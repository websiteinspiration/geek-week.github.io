ウェブサイトのアーキテクチャに関する微妙な考え

もちろん、私たちはWIT-eにいます。もちろん、アンブラーです。独自のERPシステム（ここで説明-1Cなしでどうすればよいですか？）、独自のCRMシステム、ディストリビューターとの通信用の独自のM2M（「他にどんな巧妙な単語や略語を知っていますか？」）。そしてもちろん、この3文字のパラダイムの枠内に留まるためのWWWへのアプローチ。

すべてはMicrosoftへの愛情から始まり、90年代後半のサイトの初期バージョンの一部はASPテクノロジを使用して作成され、その下のデータベースとして通常のMS Accessファイルが置かれていました。ちなみに、プロバイダーはまだ上ホスティング提供良い古いあなたはそのすべての栄光にレガシーシステムを持ってここに- ASP.NETへのアップグレード後18年、ASP。

一般的に、これは非常に便利です。内部データベースもMS Accessで記述されているため、サイトのデータを準備する手順は非常に簡単で、あるデータ形式から別のデータ形式（MySQLなど）への再変換はありません。 Accessは、「IN <外部データベース名>」という形式のSQL言語の拡張をサポートしています。これは、INSERT、UPDATE、DELETEなどのDML命令の後に追加できます（別の3文字の省略形）。

もちろん、このリンクが大きくなると、恥ずかしくないほどの速度で減速し始めました（さらに、データベース全体がmdbファイルでロックされ、サイト全体がしっかりとロックされた場合に発生するものは不明確です）。サイトをASP.NETに変換しても問題は根本的に解決されず、MS SQL Serverをベースとして切り替えることも必要でしたが、その後、プロセスは別の方向に進みました。 Webサイトのパフォーマンス向上の問題を少し異なる視点から見てみましょう。

（ちなみに、私の1Gb.ruプロバイダーは、ASP.NETが標準のLAMPバンドル（Linux / Apache / MySQL / PHP）よりも平均的に高速であると書いています。 ）

免責事項 -後続のアイデアは、一方では絶対的なレベルまで引き上げられた理論的構造を表します。これは、しばしば不合理に削減されることを意味します。他方では、具体的な実装であるため、作者が空想的で完全に切り離されているとは言えません。現実。

質問2. Webサーバーでスクリプトを使用する必要があるのはなぜですか？

私はコードの断片を数行に分けて、故意に簡素化し、クレイジーなプログラミング言語にしましたVBA（1Cを除く）

        Set IE = CreateObject("InternetExplorer.Application")

        IE.Navigate "wit.ru"
        While IE.ReadyState < READYSTATE_COMPLETE
        Wend

        Set str = IE.Document.DocumentElement
        HTML = str.innerhtml

このコードは次のことを実行します。ある有名な会社で非常に人気のあったブラウザを介してページを実行し、サーバースクリプトを実行した結果を純粋なHTMLとして保存します。おそらく、次に何が提供されるのでしょうか。まったく正しい-90年代のサイトを純粋にHTMLのように作成することはかなり可能です。

（1GB.ruから：「IISは静的ファイルの要求を非常に迅速かつ効率的に処理します」）

同時に、
