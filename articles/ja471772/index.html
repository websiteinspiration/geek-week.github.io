1. JDK、JRE、JVM


Java開発キットは、Javaアプリケーション開発キットです。これは、Javaの開発ツールとJavaランタイム環境（JREを）。

Java開発ツールには、約40の異なるツールが含まれます。javac（コンパイラ）、java（アプリケーションランチャー）、javap（javaクラスファイル逆アセンブラ）、jdb（javaデバッガ）など

。JREランタイムは、コンパイルされたツールを実行するために必要なすべてのパッケージです。 Javaプログラム。JVM仮想マシンとJavaクラスライブラリが含まれています。

JVMは、バイトコードを実行するように設計されたプログラムです。 JVMの最初の利点は原則です「一度書けばどこでも実行できる。」つまり、Javaで作成されたアプリケーションは、すべてのプラットフォームで同じように動作します。これは、JVMとJava自体の大きな利点です。

Javaが登場する前は、多くのコンピュータプログラムが特定のコンピュータシステム用に作成されていたため、より効率的で予測可能な手動メモリ管理が優先されていました。 1990年代の後半以降、Javaの出現後、自動メモリ管理が一般的な方法になりました。

商用およびオープンソースの両方で、多くのJVM実装があります。新しいJVMを作成する目的の1つは、特定のプラットフォームのパフォーマンスを向上させることです。各JVMはプラットフォームごとに個別に作成されますが、特定のプラットフォームでより高速に動作するように作成することもできます。最も一般的なJVM実装はOpenJDK JVM Hotspot です。IBM J9、Excelsior JETの実装もあります。

3.クラスローダーとその階層


クラスローダーに戻ると、これらはJVMの一部である特別なクラスです。クラスをメモリにロードし、実行できるようにします。ローダーはすべてのクラスで動作します。私たちのクラスとJavaに直接必要なクラスの両方です。

状況を想像してみてください。私たちはアプリケーションを作成し、標準クラスに加えて、クラスがあり、それらはたくさんあります。JVMはこれでどのように機能しますか？Javaは遅延クラスローディング、つまり遅延ローディングを実装しています。つまり、クラスがアプリケーションで呼び出されるまで、クラスのロードは実行されません。

クラスローダーの階層





最初のクラスローダーはBootstrapクラスローダーです。 C ++で書かれています。これは、rt.jarアーカイブからすべてのシステムクラスをロードするベースローダーです。同時に、rt.jarからのクラスのロードとクラスのロードには若干の違いがあります。JVMがrt.jarからクラスをロードすると、他のクラスファイルをロードするときに実行されるすべての検証手順が実行されません。 JVMは最初、これらすべてのクラスがすでに検証済みであることを認識しています。したがって、このアーカイブにはファイルを含めないでください。

次のブートローダーはExtensionクラスローダーです。jre / lib / extフォルダーから拡張クラスをロードします。Javaマシンが起動するたびにクラスをロードするとします。これを行うには、ソースクラスファイルをこのフォルダーにコピーすると、自動的に読み込まれます。

別のブートローダーはSystemクラスローダーです。アプリケーションの起動時に指定したクラスパスからクラスをロードします。

クラスをロードするプロセスは階層構造で行われます：

  • まず、システムクラスローダーキャッシュでの検索を要求します（システムローダーキャッシュには、既にロードされているクラスが含まれています）。
  • クラスがシステムローダーのキャッシュで見つからなかった場合は、キャッシュ拡張機能クラスローダーを調べます。
  • クラスが拡張ローダーキャッシュで見つからない場合、クラスはブートストラップローダーから要求されます。

クラスがBootstrapキャッシュで見つからない場合は、このクラスをロードしようとします。Bootstrapがクラスをロードできなかった場合、クラスのロードを拡張ローダーに委任します。この時点でクラスがロードされると、クラスは拡張機能クラスローダーのキャッシュに残り、クラスのロードが完了します。

4.クラスファイルの構造と起動プロセス


Classファイルの構造に直接進みます。

Javaで記述された1つのクラスは、拡張子が.classの単一のファイルにコンパイルされます。Javaファイルに複数のクラスがある場合、1つのJavaファイルを、拡張子が.classの複数のファイル（これらのクラスのバイトコードファイル）にコンパイルできます。

すべての数値、文字列、クラス、フィールド、メソッドへのポインタは、定数プール - メタ空間メモリ領域に格納されます。クラスの説明は同じ場所に保存され、名前、修飾子、スーパークラス、スーパーインターフェース、フィールド、メソッド、属性が含まれています。次に、属性には追加情報が含まれます。

したがって、クラスをロードするとき：

  • クラスファイルの読み取り、つまり形式の検証
  • クラス表現は定数プール（メタスペース）に作成されます
  • スーパークラスとスーパーインターフェイスが読み込まれます。それらがロードされない場合、クラス自体はロードされません

5. JVMでのバイトコードの実行


まず、バイトコードを実行するために、JVMはそれを解釈できます。解釈はかなり遅いプロセスです。解釈の過程で、インタプリタはクラスファイルを1行ずつ「実行」して、JVMが理解できるコマンドに変換します。

また、JVMはそれをブロードキャストできます。CPUで直接実行されるマシンコードにコンパイルします。

頻繁に実行されるコマンドは解釈されませんが、すぐにブロードキャストされます。

6.コンパイル


コンパイラーは、高水準プログラミング言語で記述されたプログラムのソース部分を、コンピューターが「理解できる」機械語プログラムに変換するプログラムです。

コンパイラは次のように分かれています。

  • 最適化していません
  • シンプルな最適化（Hotspot Client）：すばやく動作しますが、最適ではないコードを生成します
  • 複雑な最適化（Hotspot Server）：バイトコードを生成する前に複雑な最適化変換を実行します

7. Java


スタックは、 LIFOスキームに従って動作するJavaでのメモリの領域である- 「最後の-・ファーストアウト」または「最後には、まずアウト」。



メソッドを格納するために必要です。スタック上の変数は、それらが作成されたメソッドが実行される限り存在します。

Javaでメソッドが呼び出されると、フレームまたはメモリ領域がスタック上に作成され、メソッドがその最上位に配置されます。メソッドの実行が完了すると、そのメソッドはメモリから削除され、次のメソッドのためにメモリが解放されます。スタックメモリがいっぱいの場合、Javaはjava.lang.StackOverFlowError例外をスローします。たとえば、それ自体を呼び出す再帰関数があり、スタックに十分なメモリがない場合に、これが発生する可能性があります。

スタックの主な機能：

  • 新しいメソッドが呼び出されて完了すると、スタックにデータが追加されて解放されます。
  • このメモリ領域へのアクセスは、ヒープよりも高速です。
  • スタックサイズは、オペレーティングシステムによって決定されます。
  • スレッドごとに個別のスタックが作成されるため、スレッドセーフです。

Javaのメモリのもう1つの領域は、ヒープまたはヒープです。オブジェクトとクラスを格納するために使用されます。新しいオブジェクトは常にヒープ上に作成され、それらへの参照はスタックに格納されます。ヒープ上のすべてのオブジェクトはグローバルアクセスを持ちます。つまり、アプリケーションのどこからでもアクセスできます。

ヒープは、世代と呼ばれるいくつかの小さな部分に分かれています。

  • 若い世代 -最近作成されたオブジェクトが配置されている領域
  • 古い（保有）世代 -「長期間」オブジェクトが保存される領域
  • Java 8より前は、クラス、メソッド、および静的変数に関するメタ情報を含む別の領域- 永続生成 -がありました。Java 8の登場後、この情報をヒープの外側、つまりメタ空間に個別に格納することが決定されました




なぜ永久的な世代を放棄したのですか？まず、これは領域のオーバーフローに関連するエラーが原因です。Permは一定のサイズで動的に拡張できなかったため、遅かれ早かれメモリが不足し、エラーがスローされ、アプリケーションがクラッシュしました。

メタスペースには動的なサイズがあり、実行時にJVMメモリーサイズに拡張できます。

主なヒープ機能：

  • このメモリ領域がいっぱいになると、Javaはjava.lang.OutOfMemoryErrorをスローします
  • ヒープアクセスがスタックアクセスより遅い
  • ガベージコレクターは未使用のオブジェクトを収集するように機能します
  • ヒープは、スタックとは異なり、どのスレッドもアクセスできるため、スレッドセーフではありません。

8.ガベージコレクター


ガベージコレクターは、 ヒープ（アクセスできないオブジェクトを取り除くJava仮想マシンで実行されるプログラム）で動作しています。

JVMが異なればガベージコレクションアルゴリズムも異なり、ガベージコレクターも異なります。

最も単純なコレクターのシリアルGCについて説明します。System.gc（）を使用してガベージコレクションを要求します。



すでに上で述べたように、ヒープは2つの領域に分けられます。新世代と旧世代です。

新世代（若い世代）には、エデン、サバイバー0、サバイバー1の 3つのリージョンが含まれます。

古い世代にはTenuredリージョンが含まれます。

Javaでオブジェクトを作成するとどうなりますか？

まず、オブジェクトはEdenに分類されます。すでに多くのオブジェクトを作成していて、Edenにスペースがなくなった場合、ガベージコレクターが起動してメモリを解放します。これは、いわゆる小さなガベージコレクションです。最初のパスで、エデン領域をクリーンアップし、「存続」オブジェクトをSurvivor 0領域に配置します。したがって、エデン領域は完全に解放されます。Eden

領域が再びいっぱいになった場合、ガベージコレクターはEden領域とSurvivor領域0での作業を開始します。現在忙しいです。クレンジング後、生き残ったオブジェクトは別のリージョン（Survivor 1）に分類され、他の2つはクリーンなままになります。その後のガベージコレクションで、Survivor 0が宛先領域として再び選択されます。そのため、Survivorリージョンの1つが常に空であることが重要です。

JVMは、ある領域から別の領域に常にコピーおよび移動されるオブジェクトを監視します。そして、このメカニズムを最適化するために、特定のしきい値の後、ガベージコレクターはそのようなオブジェクトをTenured領域に移動します。Tenuredの新しいオブジェクト

に十分なスペースがない場合、完全なガベージコレクション-Mark -Sweep-Compactがあります。



このメカニズムの実行中に、使用されなくなったオブジェクトが特定され、領域からこれらのオブジェクトが削除され、Tenuredメモリー領域が最適化されます。必要なオブジェクトで順次埋められます。
