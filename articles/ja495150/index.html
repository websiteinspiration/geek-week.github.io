同じWindowsマシン上のDockerとVMWare Workstation

作業は簡単で、Windowsを搭載した作業用ラップトップにDockerを置きました。提供されたDocker Desktopで作成されたコンテナーはすべて正常ですが、VMWare Workstationが仮想マシンの実行をエラーで停止したことがすぐにわかりました。

VMware Workstation and Device/Credential Guard are not compatible. VMware Workstation can be run after disabling Device/Credential Guard.


仕事が立ち上がった、緊急に修正する必要がある




Googleを使用して、このエラーは同じマシン上のVMWare WorkstationとHyper-Vの非互換性が原因で発生することが判明しました。この問題は既知であり、Microsoft Manage Windows Defender Credential Guardナレッジベースを参照して、修正方法に関する公式のVMWareソリューションがあります。解決策は、Defender Credential Guardを無効にすることです（Windows Defender Credential Guardの無効化セクションの段落4が役立ちました）。

mountvol X: /s
copy %WINDIR%\System32\SecConfig.efi X:\EFI\Microsoft\Boot\SecConfig.efi /Y
bcdedit /create {0cb3b571-2f2e-4343-a879-d86a476d7215} /d "DebugTool" /application osloader
bcdedit /set {0cb3b571-2f2e-4343-a879-d86a476d7215} path "\EFI\Microsoft\Boot\SecConfig.efi"
bcdedit /set {bootmgr} bootsequence {0cb3b571-2f2e-4343-a879-d86a476d7215}
bcdedit /set {0cb3b571-2f2e-4343-a879-d86a476d7215} loadoptions DISABLE-LSA-ISO
bcdedit /set {0cb3b571-2f2e-4343-a879-d86a476d7215} device partition=X:
mountvol X: /d

再起動後、WindowsはDefender Credential Guardが本当に無効になっているかどうかを尋ねてきます。はい！これにより、VMWare Workstationは通常の動作に戻り、Dockerをインストールする前と同じ場所になります。

Hyper-VとVMWare Workstationを調整するためのソリューションを見つけたことはありません。新しいバージョンで友達になれることを願っています。

私は長い間、さまざまな目的でVMWare Workstationに夢中になっていて、Hyper-VとVirtualBoxに取り掛かろうとしましたが、機能が私のタスクを満たさなかったため、まだ座っています。VMWare、Docker、VSCodeを1つの作業環境にまとめる方法に対する解決策があることがわかりました。

Docker Machine-仮想ホスト上でDocker Engineを実行し、リモートとローカルの両方でそれに接続できます。そして、彼にはVMWare Workstationとの互換性ドライバーがあり、GitHub

はい、残念ながらDocker Desktopも必要です。解体した場合は、再度インストールしますが、今回は、VMWare Workstationが再度停止しないように、OSに変更を加えることに関するチェックボックスを削除します。

簡単なユーザーからはすべて正常に機能し、インストールプログラムは必要に応じて権限の昇格を要求しますが、コマンドラインのすべてのコマンドとスクリプトは現在のユーザーから実行されます。

その結果、チームは：
$ docker-machine create --driver=vmwareworkstation dev

Boot2Dockerから仮想devが作成され、その中にDockerがあります。

対応するvmxファイルを開くことにより、この仮想マシンをVMWare Workstationグラフィカルインターフェイスに接続できます。ただし、VSCodeはPowerShellでスクリプトを使用して実行する必要があるため、これは必要ありません（何らかの理由で、docker-machineおよびdocker-machine-driver-vmwareworkstationがbinフォルダーにあることが判明しました）。

cd ~/bin
./docker-machine env dev | Invoke-Expression

VSCodeが開き、ローカルマシンのコードと仮想マシンのDockerを操作します。Docker for Visual Studio Codeプラグインを使用すると、コンソールにアクセスしなくても、仮想マシン内のコンテナーを簡単に管理できます。

難しさ：

docker-machineを作成するプロセスで、私のプロセスがフリーズします：

Waiting for SSH to be available...



そしてしばらくして、仮想マシンとの接続を確立しようとする試みを超えて終了しました。

それはすべて証明書の政治に関するものです。仮想マシンを作成すると、ディレクトリ〜\ .docker \ machine \ machine \ devがこのディレクトリに表示されます。SSH経由で接続するための証明書ファイルがあります：id_rsa、id_rsa.pub。 OpenSSHはアクセス権の問題があると考えているため、それらの使用を拒否する場合があります。ここでのみdocker-machineはこれについて何も通知しませんが、退屈するまで再接続します。

解決策：新しい仮想マシンの作成が開始したらすぐに、〜\ .docker \ machine \ machine \ devディレクトリに移動し、指定したファイルへの権限を一度に1つずつ変更します。

ファイルの所有者は現在のユーザーである必要があり、現在のユーザーとSYSTEMへのフルアクセスのみが可能です。管理者のグループや管理者自身を含む他のすべてのユーザーは削除する必要があります。

絶対パスをWindows形式からPosixに変換したり、シンボリックリンクを含むボリュームをバインドしたりするときにも問題が発生する場合があります。しかし、それは別の話です。
