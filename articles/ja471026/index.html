TypeScript 決してパワー

初めてという言葉を初めて見たとき、TypeScriptで型がいかに役に立たないかと思いました。時間の経過とともに、TSに深く沈み込んで、私はこの単語がいかに強力であるかを理解し始めました。そして、この力は、私が読者と共有するつもりの実際のユースケースから生まれます。誰も気にしない、猫へようこそ。

決してないもの


歴史に突入すると、TypeScriptバージョン2.0の黎明期には型が表示されなかったことがわかります。その目的はやや控えめに説明されています。ts開発者のバージョンを簡単かつ自由に再販する場合、型は決して発生しない値の記号を表すプリミティブ型になることはありません。または、無限ループなどのループのため、または割り込みのために、値を返さない関数の記号。そして、言われたことの本質を明確に示すために、以下の例を見ることを提案します。

/**    */
function error(message: string): never {
    throw new Error(message);
}

/**   */
function infiniteLoop(): never {
    while (true) {
    }
}

/**   */
function infiniteRec(): never {
    return infiniteRec();
}

おそらくそのような例のために、タイプは明確にするために必要であるという第一印象を得ました。

型システム

/** Exclude from T those types that are assignable to U */
type Exclude<T, U> = T extends U ? never : T;

/** Extract from T those types that are assignable to U */
type Extract<T, U> = T extends U ? T : never;

/** Construct a type with the properties of T except for those in type K. */
type Omit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;

/** Exclude null and undefined from T */
type NonNullable<T> = T extends null | undefined ? never : T;

/** Obtain the parameters of a function type in a tuple */
type Parameters<T extends (...args: any) => any> = T extends (...args: infer P) => any ? P : never;

将来の変更の監視


プロジェクトの人生がつかの間ではないとき、開発は特別な性格を帯びます。コードの重要なポイントを管理し、あなたやあなたのチームメンバーが時間がかかるときに重要なコードの修正を忘れないようにしたいと思います。このようなコードのセクションは、何かの状態または列挙がある場合に特に効果的です。たとえば、エンティティに対する一連のアクションを一覧表示します。例を見て、何が起こっているのかを理解することをお勧めします。

//    -   ,     ,   ActionEngine
type AdminAction = "CREATE" | "ACTIVATE";

// ,     ,   AdminAction   . 
class ActionEngine {
    doAction(action: AdminAction) {
        switch (action) {
            case "CREATE":
                //  
                return "CREATED";
            case "ACTIVATE":
                //  
                return "ACTIVATED";
            default:
                throw new Error("   ");
        }
    }
}

上記のコードは、重要なポイントに焦点を合わせるためにできるだけ簡略化されています。AdminActionタイプは別のプロジェクトで定義されており、チームが同行しない可能性さえあります。プロジェクトは長期間存続するため、知らないうちにAdminActionタイプの変更からActionEngineを保護する必要があります。 TypeScriptは、この問題を解決するためのいくつかのレシピを提供しています。そのうちの1つは、neverタイプを使用することです。これを行うには、NeverErrorを定義し、それをdoActionメソッドで使用する必要があります。

class NeverError extends Error {
    //         - ts  
    constructor(value: never) {
        super(`Unreachable statement: ${value}`);
    }
}

class ActionEngine {
    doAction(action: AdminAction) {
        switch (action) {
            case "CREATE":
                //  
                return "CREATED";
            case "ACTIVATE":
                //  
                return "ACTIVATED";
            default:
                throw new NeverError(action);
                //                   ^       switch  .
        }
    }
}

次に、新しい「BLOCK」値をAdminActionに追加し、コンパイル時にエラーを取得します。タイプ「“ BLOCK”」の引数は、タイプ「never」.ts（2345）のパラメーターに割り当てることができません。

原則として、これを達成しました。スイッチコンストラクトがAdminAction要素の変更やセットからの削除から保護するという興味深い点に言及する価値があります。実際には、期待どおりに機能していると言えます。

NeverErrorクラスを導入したくない場合は、never型の変数を宣言することでコードを制御できます。このような：

type AdminAction = "CREATE" | "ACTIVATE" | "BLOCK";
class ActionEngine {
    doAction(action: AdminAction) {
        switch (action) {
            case "CREATE":
                //  
                return "CREATED";
            case "ACTIVATE":
                //  
                return "ACTIVATED";
            default:
                const unknownAction: never = action; // Type '"BLOCK"' is not assignable to type 'never'.ts(2322)
                throw new Error(`   ${unknownAction}`);
        }
    }
}

Context Limit：this + never


次のトリックは、疲労や不注意の中のばかげた間違いから私をしばしば救います。以下の例では、選択したアプローチの質の評価は行いません。私たち、事実上、これが起こります。クラスのフィールドにアクセスできないクラスでメソッドを使用するとします。はい、それは恐ろしく聞こえます-これはすべて政府のコードです...

@SomeDecorator({...})
class SomeUiPanel {

    @Inject
    private someService: SomeService;

    public beforeAccessHook() {
        //    ,    ,     SomeUiPanel
        this.someService.doInit("Bla bla");
        //  ^       :  beforeAccessHook   ,     
    }
}

より広いケースでは、独自の実行コンテキストを持つコールバック関数または矢印関数にすることができます。そして、タスクは次のとおりです。ランタイムエラーから身を守る方法は？TypeScriptには、このコンテキストを指定する機能があります。

@SomeDecorator({...})
class SomeUiPanel {

    @Inject
    private someService: SomeService;

    public beforeAccessHook(this: never) {
        //    ,    ,     SomeUiPanel
        this.someService.doInit("Bla bla");
        //  ^ Property 'someService' does not exist on type 'never'
    }
}

公平に言えば、それは決してメリットとは言えません。代わりに、voidと{}を使用できます。しかし、コードを読んだときに決して注目されるのは、このタイプではありません。

不変


絶対的ではないという明確な考えがあるので、次のコードが機能するはずだと思いました：

type Maybe<T> = T | void;

function invariant<Cond extends boolean>(condition: Cond, message: string): Cond extends true ? void : never {
    if (condition) {
        return;
    }
    throw new Error(message);
}

function f(x: Maybe<number>, c: number) {
    if (c > 0) {
        invariant(typeof x === "number", "When c is positive, x should be number");

        (x + 1); // works because x has been refined to "number"
    }
}

しかし悲しいかな。式（x + 1）はエラーをスローします：演算子 '+'はタイプ 'Maybe'および '1'には適用できません。この例自体は、記事「FlowからTypeScriptへの30,000行のコードの転送」で取り上げました。

柔軟なバインディング

function variants<Type extends number | string>(x: Type, c: Type extends number ? number : never): number {
    if (typeof x === "number") {
        return x + c;
    }
    return +x;
}

const three = variants(1, 2); // ok
// 2  - never,     string. ,
