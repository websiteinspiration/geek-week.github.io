BeerPHPドリンクキャップにスムーズに変化するミタップの発表（モスクワおよびオンライン）

こんにちは、8月にモスクワで他の都市のスピーカーと一緒にmitapを開催し、BeerPHPに会い、参加できないすべての人の公式部の翻訳を行います。

今日はスピーカーの紹介を始めています。Sergey Zhukは、Bryanskから会議に出席します-彼の街にはパーティーはありませんが、非同期PHPについて何か説明することがあります。これについての本や一連の記事などを書いています。以下は、これに関する最近のポッドキャストの書き起こし、リリースの試聴と表示へのリンク、およびmitap自体の詳細です。



Pyotr Myazin akaPQR：今日、私はReactPHPの主要な専門家の1人と連絡を取り合っています。セルゲイ、私は最近reactphp.orgに行って、メインページであなたを見つけました。あなたはトピックについてたくさん書いています、あなたはビデオの説明を含むあなた自身のチャンネルさえ持っています。どのようにしてこれに至ったのか、ReactPHPに惹かれた理由を教えてください。

実際、長年書いている言語を再発見します


セルゲイ別名セレガジュク：2年前、以前の仕事では、常に顧客を呼び、非常に迅速に作業するスクリプトを作成する必要がありました。タスクは「昨日」完了する必要があり、PHP開発者だけがチームに参加していました。新しいスタックを作成する時間はありませんでした。しかし、誰もがReactPHPについて何かを聞いたことがわかりました。彼はプロダクションの準備ができていると位置付け、すべてが非常に簡単でした。Composerを介して会社をインストールしただけで、非同期コードを記述できました。その結果、彼らはそれにソリューションを書いて、それをローンチしました、誰もが満足しました。

ツールがあることに気づきましたが、ロシア語でも英語でも、「このような問題があり、このように解決しました」などの実用的な例はほとんどありませんでした。その結果、私は自分のブログにいくつかの記事を書き、肯定的なフィードバックを得ました、そしてそれだけです-それはどういうわけか私を引きずりました。 Request-Responseモデルに慣れていますが、非同期コードにロードする価値があるので、すぐに次のように質問します。

ピーター：もう一度話しましょう。非同期コードで解決できる問題は何ですか。なぜReactPHPが特に必要なのですか。
セルゲイ：ほとんどすべてのアプリケーションには、API呼び出し、ファイルシステムとのやり取り、またはデータベースクエリのいずれかがあります。彼らからの回答を待つとき、私たちのプロセッサーはアイドル状態です-何か有用なことをする代わりに。答えを待つことはできませんが、非ブロッキング操作を開始して次に進みます。「I / Oのあらゆる場所でReactPHPを使用すると、アプリケーションは何倍も速くなりますか？」

残念ながら、アプリケーションを取得して非同期アプリケーションに書き直すことはできません。



セルゲイジュークです
SeryozhaはReactPHP For Beginners、Learning Event-Driven PHP With ReactPHPとPHP OOP Way、およびその他の有用な書籍の著者です。

Sergey：パフォーマンスが重要で、I / Oがすべてをブロックするボットロックである場合に、ReactPHPの使用を検討します。さらに、アプリケーション全体を非同期レールに転送しないこと、つまりこの特定のボットロックを書き直すことをお勧めします。原則として、非同期スライスは、アプリケーション全体を書き直すよりも、従来のブロッキングコードに挿入する方がはるかに簡単です。

ピーター：わかりました、でもそれなら、なぜいくつかのPHPプロセスを開始しないのですか？これはアイドルですが、次はビジーです。

セルゲイ：そして、どうすれば結果を調整できますか？ 3つのソースからデータを取得する必要があると想定し、3つのAPIを呼び出してから、結果を作成してユーザーに提供します。 3つの別々のストリームを作成すると、結果を調整するという問題が発生します。

非同期アプローチでは、単純にそれらを実行して、いわば並行して、結果を取得して処理します。スレッドでは、どういうわけかこのボールを調整する必要があります。つまり、状態に問題があります。これは非同期のアプローチよりも複雑なようです。

ピーター：わかりました。ユーザーの数を増やしたいのなら、このことを頭の中で思いました。 1秒あたりに処理された場合、Symfonyのアプリケーションスタック全体が非同期バージョンでどのように実行され、より多くの接続を処理しますか？

セルゲイ：そんなことはしません。多くの場合、アプリケーションは直接完全にブレーキをかけません。パフォーマンスの問題が発生した場合は、内部にいくつかのボトルネックが存在している可能性が高く、その結果、I / Oが遅くなる可能性があります。そして今、私はこれらのボトルネックをReactPHPに変換します。

これは、通常のブロッキングコード内の小さな非同期プログラムのように見えます。たとえば、3つの要求を並行して呼び出して処理し、ユーザーに応答を送信しました。合計実行時間は、最も遅い要求の時間と等しくなります。従来のアプローチの場合、最初のリクエスト、2番目と3番目のリクエストを待ちます。しかし、アプリケーションはこれから非同期になりませんでした。

ピーター：それをすべてNodeJS、たとえばGoと比較すると、実際にはアプリケーション全体が非同期です。データベース、ファイルシステムへのすべてのリクエスト-それらはすべて内部で非同期です。これは、原則として、すべてが非同期形式で記述されている場合、利益があることを意味しますか？

セルゲイ：最初の問題がI / Oであれば利益が出るでしょう。その後、はい、それを解決します。私たちのコードは可能な限り効率的であり、アイドル状態でこの出力が処理されるのを待つのではなく、最大数のタスクを処理できます。しかし、最初に問題がI / Oにない場合、すべてを非同期で書き込むと、何も勝てない可能性が高く、読み取りや保守が難しい複雑なコードを取得できます。

ピーター：PHPに戻ります。つまり、ReactPHPがあり、有名なアナログのSwoole、Amp、Ext-asyncがあります。比較して教えてください、これらのプロジェクトの違いは何ですか？ReactPHPはどの程度優れていますか？それは高速で、同じSwooleと比べて欠点はありますか？

Sergey： 3つのうち、正直に言うと、私はAmpのみをテストしました。これは、ReactPHPのように、箱から出してすぐにサポートされているためです。つまり、必要なコンポーネントをcomposerに配置すれば完了です。そしてここでは、拡張機能付きのストーリーはあまり好きではありません。私たちがPHPと非同期に一人でいる場合、私には、言語とネイティブのものを使用する方が合理的であると考えられます。追加の拡張機能を使用する場合、特に非同期PHPではなく、PHPの非同期拡張機能を使用します。

ReactPHPは実際にどのように本番環境に対応していますか？


ピーター：彼らは彼らのウェブサイトに「長期的なサポート」を書いています。つまり、APIに重大な変更を加えない、ある種の長期サポートです。そのようなものです？すべてが十分高いですか？

セルゲイ：はい。 ReactPHPプロジェクト自体はすでにかなり古いものです。彼は12年目から働いています、彼は7歳であることが判明しました。彼はすでに十分な生産準備の時間があります。そして、はい、さらに2年間の主要コンポーネントの長期サポートがここにあります。つまり、これらのコンポーネントに安全に結び付けられ、修正、更新を受け取り、下位互換性について心配する必要はありません。彼らは2年間、バージョンを維持することを約束し、何も壊れません。これは本当に素晴らしいと思います。これは大きな前進です。私が覚えている限りでは、最近まで、Ampはそのコンポーネントのいくつかについて説明していましたが、インターフェースとそのパブリックAPIを完全には決定していませんでした。

ピーター：ReactPHPの学習をどこから始めることをお勧めしますか？もちろん、公式ドキュメントとビデオチュートリアルに加えて。すべてのコミュニティはどこにありますか、電報チャットや不和なチャット、フォーラムはありますか？

セルゲイ：残念ながら、私はそのような直接的なコミュニティを知りません。そこアクティブになっているTwitterアカウントは：あなたは、尋ねる質問をし、そこに興味をすることができます。そして、鉱山労働者は非常に迅速に応答します-あなたは個人的にTwitterで彼らに手紙を書くことができます。私はそうしました、彼らにいくつかのことを尋ねました。
