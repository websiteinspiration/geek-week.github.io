カスタマージャーニーマップによる客観的な品質スコアテスト

こんにちは、ハブロフスク市民！

私は職業別のアナリストで、システムの設計と要件の作成を10年間行っています。 UX / UIをテストするためのアプローチについて議論することを提案します。

検疫前に、飛行機のチケットを購入しなければならない状況に遭遇しました。約30分、エアレススペースに入りました。すべてのフィールドに入力している間（サービスにはプロファイルがあります）、次に一度ではなくすべてを再確認し、すべての有料オプションを慎重に削除しました。
私見、これは非人道的です。

自分のデータを覚えて置き換えてみませんか？なぜ追加のオプションを課すのですか-忠誠心を減らしますか？

私は、不快でわかりにくいUX / UIに悩まされているのは私だけではないと思います。質問：「それはどのように扱われますか？」



これまでのところ、解決策はシステム品質指標の体系的で独立したテストである可能性があるという結論に達しました。研究所や教科書で話されている人：人間工学、便利さ、理解しやすさ、タスクの実行速度。

通常、品質テストは次のようになります。

  • チームは時間通りに機能を作成するために全力を尽くして実行し、UX / UIについては誰も考えていません
  • UX / UIは、レイアウトの遵守についてのみチェックされます
  • UX / UIテストは、テスターの好みと個人的な評価に基づいて行われます。はい、これは一般的な規則によって正当化できます（たとえば、Cooperの本の「On Interfaces」から）。しかし、カスタムおよびB2Bシステムでは、多くの場合、重要な解決策があります。チームの美しさについては、好みや考え方が異なる場合があります。結果として、これは意見の違いによる非建設的な紛争につながります。それらの中で最強の勝利。

その後、製品の要件を収集するときにも同様の話が起こると思いました。クリエーターが苦痛の決定を下し、エンドユーザーのニーズを閉じてしまうと思うことが起こります。現実をマッピングするには、カスタマージャーニーマップの方法があります。

この方法の本質は、特定の問題を解決するときに、統計サンプルで複数のユーザーの経験を利用することです。したがって、他の人々が問題に直面しているかどうか、この状況が少数の人々のグループに固有のものであるか、それとも全体的な問題であるかを知ることができます。

この原則とUX / UIを確認してみませんか？

テストアルゴリズム：

1）テストポリシーを決定します。

  • テスト用のユーザースクリプトのセット
  • テストされた品質指標：利便性、人間工学、理解力など

2）テストする回答者を見つける

3〜5人の回答者をテストします。エンドユーザーまたは別のチームのメンバーにすることができます。主なことは、開発チームからの独立です。

3）調査を実施します。各回答者で

サイクルを繰り返す
>ユーザーカードテンプレートが取得されます。
>回答者は、タスクへの導入及び実行システム与えられる/プロトタイプ/レイアウト
>回答者のステップ、ステップのための彼のフィードバック、時間-このすべては、テンプレートに記録されている
端



集計データ）4。

1つのマップテンプレートは1つの意見であるため、回答者の共通部分を見つける必要があります。シナリオごとに、ギャップマップマトリックスがコンパイルされます（CJMの最良の方法で）。



  • マトリックス列-スクリプトステップ。
  • 線は不満の指標です。
  • 交差点-回答者のマップからの結果。
  • 最も時間を費やしたステップまたは否定的な感情、誤解-これらはシステムの品質の問題です。

5）見つかった問題のリストを作成します。

GrowthPointsListという名前を付けました。



データは、問題のリストとシステムがどのように動作するかについての回答者の期待にまとめられています。スクリプトの通過に対する影響のレベルを指定できます。

  1. スクリプトを完了できません
  2. 否定的だがスクリプトは合格
  3. 一部の回答者は否定的でしたが、なんとか台本を通り抜けました。

これにより、レイアウト段階での開発前であっても、より低コストでレイアウトをテストし、問題を解消することができます。テスト結果は妥当であり、参加者と顧客の間で論争を引き起こしません。

このようにして、アプリケーションの世界は、エンドユーザーにとって便利でシンプルなものに変化する可能性があると思います。
