分子動力学のCUDAへの転送。パートI：基本

この記事の目的は、CUDAテクノロジーを使用した分子動力学（MD）法を使用した数値モデリング用のプログラムのコードの並列化に関する問題を提起することです。 CUDAで動作するものを含むMDソフトウェアパッケージがすでに存在するので、なぜこれが必要なのですか？実際のところ、私は「既存の力場」という独自の概念を開発しているが、それは既存のMDプログラムでは実施されていない。



これらのニーズに対応するために他の誰かのコードを作り直すことは、決してありがたいことではないので、すでに書いたシーケンシャルコードを転送すると同時に、いくつかの考えを共有することにしました。また、これは多くの場合のようにCUDAの記事にここにちらつくコメントへの応答で、この。



では、分子動力学とは何でしょうか？Habréには、このトピックに関するいくつかの投稿があります。たとえば、ここやここです。簡単に言うと、MDは、多くの粒子（原子、イオン、分子を含む）の動きをシミュレートし、この動きに応じてシステムの集合的な特性を計算できる方法です。使い方？N個の粒子のセットについて、いくつかの初期座標、速度、質量、および（最も重要な！）それらの間の相互作用の法則が与えられていると仮定します。速度に応じて座標を変更します。相互作用の法則に基づいて、粒子間に作用する力を計算します。私たちは強度と質量を知っているので、加速度を知っています。加速を考慮して速度を調整します。そして再び、座標の変更に移ります。そして、あなたが疲れるまで、私たちは千回を繰り返します 十分な統計が得られません。







実際のアプリケーションでは、一連の粒子座標関数の展開に基づいて、「座標-速度-力」の計算シーケンスがいくぶん複雑になります。最も一般的なのはVelocity Verletアルゴリズムです。 Verletインテグレーターに基づくMDプログラムの疑似コードは次のとおりです。 



init_md;
int iStep = 0;
while (iStep < nStep)
{

    verlet_1stage;
    forces_calculation;

final_md;


メインプログラムループは、ユーザーが指定した回数実行します（nStep）。このループの反復は、タイムステップ、または単にタイムステップと呼ばれます。関数INIT_MD、reset_quantites、some_output、final_mdは考慮されません。これらは、入力情報の読み取り、ゼロ化された値、データの出力、メモリの割り当てと解放などを行います。情報出力の観点から見ると、分子動力学はビデオカードでの計算に最適です。ほとんどの計算をアンロードする必要はありません。



関数verlet_1stage、verlet_2stage、temp_scaleおよびtermostat運動方程式と温度の積分器に関連しています。force_calculation関数は粒子間に作用する力を計算し、clear_cell_list関数は計算を加速する手法を参照します（これについては後で詳しく説明します）。

__device__ void put_periodic(float3& xyz, float3 vel, float mass, int type, cudaMD *md)
{
   if (xyz.x < 0)
   {
      xyz.x += md->leng.x;

      atomicAdd(&md->specAcBoxNeg[type].x, 1); 
      atomicAdd(&md->negMom.x, mass * (-vel.x)); 
   }
   else
     if (xyz.x >= md->leng.x)
     {
        xyz.x -= md->leng.x;

        atomicAdd(&md->specAcBoxPos[type].x, 1);
        atomicAdd(&md->posMom.x, mass * vel.x);
     }
}


この関数は、長方形のボックスのジオメトリを持つボックスに対して機能します。コードはxを測定するためのもので、残りの測定は完全に類似しています。LENGとrevLengフィールドは、対応する次元におけるボックスサイズとその反転値です。一般に、パーティクルをボクシングに戻すには、条件演算子なしで実行できます。 



xyz.x -= floor(xyz.x * md->revLeng.x) * md->leng.x;

「温度」関数temp_scaleおよびtermostat。分子システムは、温度の概念を持っているという点で機械的システムとは異なります。 MDは、分子運動理論から温度の定義を継承しました。温度は、粒子の平均運動エネルギーに比例する値です（これは誤りであると信じる理由があります）。そのような用語で目的の温度を維持するには、temp_scale関数が行うように、粒子速度を単純にスケーリングできます。 



__global__ void temp_scale(int atPerBlock, int atPerThread, cudaMD *md)
{
   float k;
   int i;
   int id0 = blockIdx.x * atPerBlock + threadIdx.x * atPerThread;
   int N = min(id0 + atPerThread, md->nAt);

   k = sqrt(md->teKin / md->engKin);
   for (i = 0; i < N; i++)
   {
      md->vls[i].x *= k;
      md->vls[i].y *= k;
      md->vls[i].z *= k;
   }

   if (!blockIdx.x)
    if (!threadIdx.x)
     {
        md->engKin = md->teKin;
     }
}


teKin フィールドには、ターゲットの運動エネルギーが含まれ、ユーザーが設定した温度に比例します。この関数は、速度ベクトルのすべての成分にkを乗算するだけで、1つのストリームは、運動エネルギーを指定された値に等しく設定します。ここでは、変数kは各スレッドで定義されていますが、特に平方根が含まれているため、不要な処理を行わないようにするために、変数kを1つに定義できます（同じであるため）。このような基本的な温度スケーリングは、通常、シミュレーションの準備段階（平衡期間）で数ステップの頻度で使用されます。より洗練された温度維持アルゴリズム（サーモスタット）は、特定の温度の特定のリザーバーとの粒子衝突をシミュレートします。最も一般的なのは、Nose-Hooverサーモスタットです。

2力の計算とセルのリスト



分子システムでは、粒子に作用する力の主な原因（すでに述べた外部フィールドを除く）は他の粒子です。これらの力の計算は、MDの中で最も複雑で時間がかかり、あいまいな部分です。最も単純な近似の1つは、システム内のすべての相互作用が、粒子の各ペア間の相互作用の合計であることです（ペアワイズ近似）。これらの目的のために、ペアポテンシャルが導入されます-2つの粒子の相互作用のポテンシャルエネルギーの、それらの間の距離への関数依存性。最も有名なペアポテンシャル：レナードジョーンズ（または "6-12"）、バッキンガム、ボーンメイヤーハギンズのポテンシャル。たとえば、最初の2つの分析式を示します。





U=4ϵ(σ6r6−σ12r12)





そして





U=Aexp⁡(−rρ)−Cr6,







ここで、Uは粒子のペアのポテンシャルエネルギー、rは粒子間の距離、その他すべての文字は、相互作用する粒子のタイプのみに依存するパラメーターであり、初期化段階で設定されます。ただし、エネルギー以上に、これらの粒子間の相互作用の力に関心があります。力はペアポテンシャルの導関数を介して取るに足らないものです。そのため、分析式を取得してプログラムに縫い込むことも簡単です。そして、作用の力は反応の強さに等しいので、両方の粒子に作用する力を互いに即座に取得します、擬似コード： 



U = 0;
for (i = 0; i < N; i++)
	for (j = i + 1; j < N-1; j++)
	{
		dx = x[i] – x[j];
		dy = y[i] – y[j];
		dz = z[i] – z[j];
		r = sqrt(dx*dx + dy*dy + dz*dz);
		U += pair_pot(r, types[i], types[j]);
		F = force(r, types[i], types[j])
		fx[i] = F * dx;
		fx[j] = -F * dx;
		fy[i] = F * dy;
		fy[j] = -F * dy;
		fz[i] = F * dz;
		fz[j] = -F * dz;

	}


コードは単純です。粒子間の距離を見つけ、それをペアポテンシャルの関数と力の関数に置き換え（どちらも相互作用する粒子のタイプに依存します）、力の予測を計算します。サイクルはN（N-1）/ 2回実行されます。ここで、Nは粒子の数です。少なくとも何らかの方法で計算数を減らすために、これらのポテンシャルの値は距離とともに急速に減少するという事実を利用できます。 6〜8オングストロームを超える距離では、ポテンシャルは計算できず、ほとんどゼロになります。距離（またはその二乗）をチェックするための条件を追加すると、計算の数は減りますが、サイクルの反復数は減りません。大規模なシステムの場合、ほとんどのチェックはfalseになります。次の手法では、既知の大きな原子間距離を持つペアの一部を削除することにより、チェックの数を減らすことができます。

__device__ void pair_between_cell1(float3 xyzA, float3* fxyzA, int typeA, float3 xyzB, float3* fxyzB, int typeB, float3 shift, cudaMD* md, float& engVdw, float& engCoul)
{
    float r, r2, f;
    float dx, dy, dz;
    cudaVdW* vdw;

    dx = xyzA.x - xyzB.x - shift.x;
    dy = xyzA.y - xyzB.y - shift.y;
    dz = xyzA.z - xyzB.z - shift.z;
    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)
    {
        r = 0.0;
        f = 0.0;

        f += real_ewald(r2, r, md->chProd[typeA][typeB], md->alpha, engCoul); 

        vdw = md->vdws[typeA][typeB];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw);


        fxyzA->x += f * dx;
        atomicAdd(&(fxyzB->x), -f * dx);
        fxyzA->y += f * dy;
        atomicAdd(&(fxyzB->y), -f * dy);
        fxyzA->z += f * dz;
        atomicAdd(&(fxyzB->z), -f * dz);
    }
}

__device__ float cu_fe_lj(float r2, cudaVdW* vdw, float& eng)
// U = 4e[(s/r)^12 - (s/r)^6]
{
    float r2i = 1.0 / r2;
    float sr2 = vdw->p1 * r2i;
    float sr6 = sr2 * sr2 * sr2;

    eng += vdw->p0 * sr6 * (sr6 - 1.0);
    return vdw->p2 * r2i * sr6 * (2.0 * sr6 - 1.0);
}

__device__ float cu_fer_buck(float r2, float& r, cudaVdW* vdw, float& eng)
// U = A exp(-r/ro) - C/r^6
{
    float r2i = 1.0 / r2;
    float r4i = r2i * r2i;
    if (r == 0.0)    //  r , 
        r = sqrt(r2);

    eng += vdw->p0 * exp(-r / vdw->p1) - vdw->p2 * r4i * r2i;
    return vdw->p0 * exp(-r / vdw->p1) / r / vdw->p1 - 6.0 * vdw->p2 * r4i * r4i;
}


関数は力（距離ベクトルの投影で乗算する必要があります）を返し、変数engにエネルギーを追加します。これらの関数の1つへのポインターはcudaVdW構造に格納されます。一般的に言えば、運動の軌跡を取得するには、エネルギーを計算することはできません。さまざまな統計に必要です。これらの関数からエネルギーの式を削除すると、計算が加速する可能性があります。そしてこれを行うことができます：完全な関数を呼び出すためのいくつかのステップと、残りの時間-縮小されたアナログのみ。クーロン相互作用の一部は、real_ewald関数によって計算されます。 



__device__ float real_ewald(float r2, float& r, float chprd, float alpha, float& eng)
{
    float ar; 
    float erfcar;
    float kqq = chprd * d_Fcoul_scale; // q[i]*q[j]*1/4pie0;

    r = sqrt(r2);
    ar = alpha * r;
    erfcar = erfc(ar);

    eng += kqq * erfcar / r;
    return kqq / r / r2 * (erfcar + 2 * ar / d_sqrtpi * exp(-ar * ar));
}


これはおなじみのクーロンの法則とは異なりますが、それについては別の記事で詳しく説明します。



実際の例でのテストでは、ブロックに割り当てられたすべてのフローをロードするために、セルのペアにパーティクルがほとんどないことが示されました。関数cell_list2aおよびcell_list2bをさらに最適化するには、1つのブロックで一度に複数のペアを処理します。これまでのところ、フローがセルのいくつかのペアに単純に分割される最も単純なバージョンのみを記述しました。 1つのブロックは1つのペアではなく、4：0-1、1-2、3-4、5-6と処理します。この場合、共有メモリにデータをロードすると同時に、ペア0-2、0-3、0-4などを処理できることに気付くでしょう。ただし、これには、データの準備段階で追加のトリックが必要です。関数cell_list2bの場合も共通のセル（たとえば、同じ0-2、0-3、0-4など）でペアをロードする場合、セル0からデータを1回だけコピーすることにより、グローバルメモリへの呼び出し回数を減らすことができます。

__global__ void all_pair(cudaMD* md)
{
    int i, j;
    int step = blockDim.x * gridDim.x;
    int ex = 0;  // exit flag
    int nAt = md->nAt;

    i = 0;
    j = blockIdx.x * blockDim.x + threadIdx.x + 1;
    while (1)
    {
        while (j >= nAt)
        {
            i++;
            if (i >= nAt - 1)
            {
                ex = 1;
                break;
            }
            j = i + 1 + j - nAt;
        }
        if (ex) break;
        global_pair(i, j, iStep, md);
        j = j + step;
    }
} 

__device__ void global_pair(int i, int j, cudaMD* md)
{
    float r, r2, f;
    float engCoul = 0;
    float engVdw = 0;
    cudaVdW* vdw;
    int ti, tj;

    float dx = md->xyz[i].x - md->xyz[j].x;
    float dy = md->xyz[i].y - md->xyz[j].y;
    float dz = md->xyz[i].z - md->xyz[j].z;

    //   
    if (dx > md->halfLeng.x)
        dx -= md->leng.x;
    else
        if (dx < -md->halfLeng.x)
            dx += md->leng.x;

    if (dy > md->halfLeng.y)
        dy -= md->leng.y;
    else
        if (dy < -md->halfLeng.y)
            dy += md->leng.y;

    if (dz > md->halfLeng.z)
        dz -= md->leng.z;
    else
        if (dz < -md->halfLeng.z)
            dz += md->leng.z;

    r2 = dx * dx + dy * dy + dz * dz;

    if (r2 <= md->r2Real)    
    {
        r = 0.0;
        f = 0.0;

        ti = md->types[i];
        tj = md->types[j];

        f += real_ewald(r2, r, md->chProd[ti][tj], md->alpha, engCoul);

        vdw = md->vdws[ti][tj];
        if (vdw != NULL)
            if (r2 <= vdw->r2cut)   
                f += vdw->feng_r(r2, r, vdw, engVdw); 

        dx *= f;
        dy *= f;
        dz *= f;

        atomicAdd(&(md->frs[i].x), dx);
        atomicAdd(&(md->frs[j].x), -dx);
        atomicAdd(&(md->frs[i].y), dy);
        atomicAdd(&(md->frs[j].y), -dy);
        atomicAdd(&(md->frs[i].z), dz);
        atomicAdd(&(md->frs[j].z), -dz);

        atomicAdd(&(md->engCoul1), engCoul);
        atomicAdd(&(md->engVdW), engVdw);

    }
}


計算結果は一致した。確かに、明らかに、丸め順序が異なるため、小数点以下3桁以内の粒子の座標は、最初から最後まで異なる可能性があります。約1,400粒子で6オングストロームのカットオフ半径のシステムでは、関数cell_list2aとcell_list2bは、単純な列挙の実行時間の約2倍を失うことに注意してください。複数のブロック（8および16）を1つのブロックで同時に処理する場合、速度が比較され、all_pair関数よりもわずかに早くなる場合があります。。ただし、これはシステムのサイズに大きく依存します。原子数の増加に伴い、セルリストアルゴリズムの速度は直線的に増加し、単純な列挙は二次的に増加します。さらに、セルリストは、ガスなどの放電システム（粒子間の距離が長い）のモデリングに非常に効果的です。
