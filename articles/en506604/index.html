I admit, but I do not know Python. Just because I don‚Äôt use it. Nevertheless, taking up its development, as well as in trying to decipher the mysterious abbreviation GIL, I went to an article describing the ‚Äúinexplicable magic phenomena‚Äù of a parallel version of a CPU-dependent function in Python. There was a desire to recheck this test and compare it with the equivalent implementation in the form of a state machine model (Finite-state machine or FSM for short) in the environment of Visual Component Programming (automatic) - VKP (a).

Obviously anythe program is to some extent CPU dependent. On the other hand, if this is not only assembler, then by testing in one or another high-level language, we will more likely investigate the software layer hidden by it. Therefore, considering Python, it would be more correct to talk about the CPU dependence of its interpreter. It can even be argued that a Python program will have a speed depending on the version of the interpreter, and will have a characteristic ‚Äúmysticism‚Äù for it.

At the same time, there are situations when there may be almost no dependence on the CPU (we will see this). We are talking about languages ‚Äã‚Äãwhose computational model is different from the typical processor architecture. The computational model of Python corresponds to it, and the automaton model of computations, which will be discussed later, has a different architecture and this will determine the specifics of its testing. What will be the speed and whether the mystic will take place will be revealed during the testing of the ‚Äúautomaton code‚Äù .

The Python code borrowed from the article, in serial and parallel versions, is shown in Listings 1 and 2, respectively. In our case, it is supplemented with lines fixing the time of work, and the sequential test is enclosed in a ‚Äúwrapper‚Äù that allows you to set the number of repetitions (a consequence of my acquaintance with decorators in Python).

Listing 2. Parallel CPU test 

import time
from threading import Thread
def count(n):
  start_time = time.time()
  while n > 0:
    n -= 1
  print("speed CPU-%s seconds ---" % (time.time() - start_time))

k = 2

  print("k=%d " % k)

  t2.join();

Listing 3. Implementing a C ++ test 

class FsmCount
{
public:
    FsmCount(int n) { nCount = n; nState = 0; };
    void ExecOneStep();
    int nState;
protected:
    int nCount;
    int x1();
    void y1();
};

#include "FsmCount.h"

int FsmCount::x1() { return nCount > 0; }
void FsmCount::y1() { nCount--; }
void FsmCount::ExecOneStep()
{
    switch (nState) {
    case 0:
        if (x1())
            y1();
        else nState = 1;
        break;
    }
}

#include <QCoreApplication>
#include <QTime>
#include "FsmCount.h"

int main(int argc, char *argv[])
{
    QCoreApplication a(argc, argv);

    QTime time;
    FsmCount *pFsmCount{nullptr};

    pFsmCount = new FsmCount(100000000);
    time.start();
    while (!pFsmCount->nState)
        pFsmCount->ExecOneStep();
    double dTime = QString::number(time.elapsed()).toDouble();
    printf("%f", dTime);
    if (pFsmCount) delete pFsmCount;

    return a.exec();
}


The test run time on the mentioned hardware base is within 250 ms. This is over 55 times faster than Python code. A comparison was made with a parallel test case for two threads (see Fig. 2).

As a result, the test speed decreased to less than 1 ms (it‚Äôs difficult to say more precisely, because the test gives a value equal to zero). If even this value is taken as a basis, then the speed is more than 14000 (!) Times higher than in Python. It becomes clear why C ++ is preferred when designing the same real-time systems.

Using C ++ as a design language of the VKP (a) makes it quite simple to solve the speed problem that has arisen. Listing 6 shows the code for an automaton using standard system threads based on the QThread class of the Qt library. As a result, the time dropped to 70-80 ms even when the cycles increased to 100000000. Thus, the test using the stream is already 200 times faster than in Python.

Note that the result obtained is a slower version of the test, synchronized with the environment of the CPSU (a). If you comment out a line (Listing 6):
while (bIfExecuteStep) { bIfExecuteStep = false; },

then the time will ‚Äúcollapse‚Äù to 1 ms or less. As a result, the speed with respect to Python will already be more than 14,000 times higher.

Note 1. The bIfExecuteStep flag is managed by the VKP (a) environment. It is set by it at the beginning of the current discrete measure. Introduced to synchronize with the discrete time of the automaton space of the process.

Automata, in comparison with other models, have discrete time. The value of the discrete clock can be fixed for synchronous machines or floating - for asynchronous (classification according to Glushkov V.M.). The next test will be to check the environment of the CPSU (a) for synchronism - the ability to maintain a constant measure of discrete time. Recall that the synchronous operation of automatic machines is especially in demand in the design of automatic control systems (see details [1]). At the same time, this mode serves as an example of the actual independence of the computational model from CPU speed .

The test run time with a discrete clock cycle (in the theory of automatic control (TAU) called the quantization interval) with a duration of 1 ms and the number of cycles of 100,000 should be 100 seconds. The results obtained during testing are shown in Fig. 7.
