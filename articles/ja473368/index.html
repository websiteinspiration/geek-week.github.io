MIROはオープンな屋内ロボットプラットフォームです。パート3-ソフトウェアコンポーネント：ESP8266



MIROプラットフォームのソフトウェアコンポーネントの分解を開始します。ファームウェアとワイヤレス通信にESP8266を使用してARDUINO UNOを「調理」する方法を見てみましょう。

目次：パート1、パート2、パート3、パート4、パート5。

ESP8266のために何かを開発してほしくありません。長い間、私が必要とする機能を備えたターンキーソリューションがあると確信していました。そしてMIROにとって、この機能は非常にシンプルでした。

  1. ソフトウェアには、ネットワーク設定用のWebインターフェイスが必要です。
  2. ソフトウェアはATMEGA328ワイヤレスファームウェアを実装する必要があります。
  3. ソフトウェアには、ワイヤレスデバッグと管理のためのWiFi-UARTブリッジ機能が必要です。

おそらく、これらの機能のほとんどには、よく知られているESP-LINKがあります。。そして彼は本当に強力です。いくつかのプロジェクトで、異なるインターフェース間のブリッジとして使用しました-非常に便利です。しかし、ここ数か月前、私たちがこの問題に密接に対処するために座ったとき、彼はARDUINO UNOを点滅させていなかったことが判明しました。彼らはどこでもそれが点滅していると書いていますが、彼は点滅しませんでした。その時点では、なぜ標準的な手順であるのか、そしてそれが何であるのかさえ深く理解できませんでした。私の友人と私は、ホストの仮想COMポートを特定のESP IPアドレスに設定するだけです。ESP-LINK自体で、リセット行を指定してフラッシュしようとしました。 ATMEGA328のリセットは成功しましたが、ファームウェアはフラッディングされませんでした。さらに、ネットワークには、ESP-LINKを使用してそのようなファームウェアを編成することに関する多くのレッスンがあります。しかし、あなたが彼らの問題を読めば、彼らはいつも誰かが点滅していない何かをそこに持っています。

失望し、彼は代替案を探し始めました。そして彼女を形で見つけたjandrassyというニックネームの開発者によるGitHub上の元のArduino WiFiのフォーク。
その年の2月に、READMEにファームウェアから開発環境まですべてを構成するという膨大な指示がありましたが、今では大幅に簡素化されています。それでも、環境をセットアップするには少しの努力が必要です。ただし、ARDUINOファームウェアは完璧です。これは、WindowsとLinuxの両方からテストされました（MacOSでは、何もテストしたことがないようです-プロジェクトの直接の環境で誰もテストしていません）。

環境を準備する順序は次のとおりです。

  1. ここから dfuライブラリをダウンロードしてインストールします。
  2. ( esp8266-serial-arduinouno-hacked.cpp) , Reset ATMEGA328 GPIO5 ESP8266. MIRO – GPIO2.

:

esp8266_serial_arduinouno_hacked_target_reset(struct dfu_interface *iface)
{
	pinMode(5, OUTPUT);
	digitalWrite(5, 0);
	delay(1);
	digitalWrite(5, 1);
	delay(200);
	return 0;
}

:

esp8266_serial_arduinouno_hacked_target_reset(struct dfu_interface *iface)
{
	pinMode(2, OUTPUT);
	digitalWrite(2, 0);
	delay(1);
	digitalWrite(2, 1);
	delay(200);
	return 0;
}


jandrassyの作者は、その実装がターゲットチップとブートローダー（Optibootを備えたATMEGA328P）のこの構成でのみ機能することを示しています。

私はまだ解決策があり、おそらくもっと良い解決策があると確信しています。誰かが確実に機能するオプションを知っている場合は、私に知らせてください。多数の既製のコードにもかかわらず、ESP-LINKに加えて別の2つまたは3つのプロジェクトをチェックしたため、それらは機能しませんでした。場合によっては、ファームウェアが1回通過し、場合によってはまったく機能しませんでした。

おそらく、見つかったソリューションの主な利点の1つは、プロジェクトのコードベースが比較的小さいことと、Arduino Coreのファームウェアが開発されていることです。

jandrassyプロジェクトになかったのは、WiFi-UARTワイヤレスブリッジでした。元のArduino.org WiFi Linkファームウェアではすでに、ターミナルウィンドウのあるページが切り取られており、ブラウザーを介してUARTチップATMEGA328を使用できます。これと同じように、ESP-LINKで実行できます。そして、Arduino UNO WiFiに関する最初の出版物（たとえば、ここでは）に目を向けると、ファームウェアの最初のバージョンでは、ウィンドウと対応するメニュー項目（「WiFiコンソール」）の両方がそうであることが明らかです。また、リリース1.0.0のアーカイブでも、console.jsファイル（このページを実装しただけ）は存在しますが、実際にはサーバーに関与していません（確認しました）。彼らがそれを飲んだ理由と理由は明らかではありません。

私は何かを仕上げなければなりませんでした。現在、ファームウェアには別のサーバーがあります-明らかなポート23のTELNETで、多くの端末プログラムが問題なく動作します。

しかし、現時点では、すべてが機能するために、ESP自体のワイヤレスファームウェアを担当する部分をjandrassyコードから削除する必要がありました。最初の要件では、この機能はありませんでした。ただし、コメントの読者が1つのファームウェアでESP8266自体と「外部」MCU（この場合はATMEGA328）の両方のワイヤレスファームウェアの可能性を実装できるかどうかを肯定的に教えてくれれば、この機能をESPファームウェアに戻すことを検討します。記事の作成中にすでに間接的に、このカナダ人の草案でこれを確認しました。しかし、チェックしませんでした。

その間、ESPは有線でのみフラッシュされますが、ARDUINOはWiFi-UARTブリッジを介してワイヤレスファームウェアとTELNET通信の両方を実行できます。そしてとても便利です！私の目標は達成されました。

まあ、プロジェクトには、元のように、設定用のWEBインターフェイスがあります（写真はクリック可能です）。





プロジェクトのWEBインターフェイススタイルのCSSファイルを変更し、小さな「松葉杖」を導入しました。現在、ESPは常にAP + STAモードです。この松葉杖は、ESPを「STAのみ」モードに切り替える可能性を排除するために作成されました。ロボットを別のサブネットに転送すると、ユーザーは新しいネットワークでロボットを構成できなくなります。ロボットは新しいネットワークに接続できませんが、外部アクセスもありません。 -ESPを再フラッシュする必要があります。AP + STAモードでは、ユーザーはいつでもロボットのアクセスポイントに接続し、新しいWiFiネットワークへの接続を構成できます。
