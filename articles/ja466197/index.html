PVS-Studio 7.04


夏は休暇シーズンだけでなく、実りある仕事の時間でもあります。晴れた日はとても活気があり、遅い散歩や膨大なコードのコミットに十分なエネルギーがあります。PVS-Studio 7.04の2番目の夏のリリースは非常に大きいことが判明したため、すべてについて説明するプレスリリースに注目します。

PVS-Studioは、C、C ++、C＃、およびJavaで記述されたプログラムのソースコードのエラーおよび潜在的な脆弱性を検出するためのツールです。Windows、Linux、およびmacOSの環境で動作します。

アナライザーの機能は、さまざまなオープンソースプロジェクトの検証中に見つかったコード内の広範なエラーのコレクションによって十分に実証されています。 PVS-Studio 7.04リリースに含まれる新機能の概要を紹介します。



コピーレフトライセンスを持つファイルを検索する


コードを取得しない開発者がタスクに取り組んでいます。コピーと貼り付けのコードの一般的なソースは、Stackoverflowサイトなどです。ただし、プログラマーがオープンソースプロジェクトからコードを取得し、ライセンス要件をチェックしない場合があります。したがって、Copyleft-licenseを使用したオープンソースプロジェクトのいくつかのファイルが、誤ってクローズドソースプロジェクトに入る可能性があります。すべてのプロジェクトコードを公開する義務があります。従業員が多い企業では、これを追跡することは困難であり、そのような行動のためにリスクや問題が深刻になる可能性があります。そのため、サポートされているすべての言語（C、C ++、C＃、Java）のPVS-Studioでは、このようなファイルを見つけるのに役立つ診断が表示されます。

異なる言語の診断番号：

  • C、C ++：V1042
  • C＃：V3144
  • Java：V6071

これらの診断について詳しく説明し、診断の目的を分析しましょう。アナライザーが警告を発行するコメントの例：

クローズドプロジェクトの場合


このようなライセンス（この場合はGPL3）を含むファイルを閉じたプロジェクトに追加すると、このライセンスの性質上、残りのソースコードを開く必要があります。

このタイプのコピーレフトライセンスは、他のプロジェクトファイルに拡散できるため、「バイラル」ライセンスと呼ばれます。問題は、閉じたプロジェクトで同様のライセンスを持つ少なくとも1つのファイルを使用すると、すべてのソースコードが自動的に開かれるため、バイナリファイルと共に配布する必要があるということです。

Diagnosticsは、次の「バイラル」ライセンスを検索します。

  • AGPL-3.0
  • GPL-2.0
  • GPL-3.0
  • LGPL-3.0

次のオプションがあり、クローズドプロジェクトでコピーレフトライセンスを持つファイルの使用を発見した場合に実行できる操作は次のとおりです。

  1. プロジェクトでこのコード（ライブラリ）を使用することを拒否します。
  2. 使用済みライブラリを交換してください。
  3. プロジェクトを開きます。

オープンソースプロジェクトの場合


この診断はオープンソースプロジェクトには適切ではないことを理解しています。PVS-Studioチームは、オープンプロジェクトの開発に貢献し、それらのエラーの修正を支援し、無料のライセンスオプションを提供します。ただし、この製品はB2Bソリューションであるため、この診断はデフォルトで有効になっています。

コードが上記のコピーレフトライセンスのいずれかで配布されている場合、次の方法でこの診断を無効にできます（C \ C ++の場合はV1042です）。

  • PVS-Studio Visual Studio, , Options > PVS-Studio > Detectable Errors > 1.General Analysis > V1042 . , ( ). , , .
  • プラグインを使用しない場合は、コマンド全体のルールをブロックするか、アナライザーレポートからメッセージを削除する必要があります。構成ファイル（.pvsconfig）またはグローバルヘッダーファイルの1つにコメント「//-V :: 1042」を追加できます。Visual C ++を使用する開発者にとって、このオプションをstdafx.hファイルに追加することをお勧めします。このコメントは、アナライザーにV1042診断を無効にするように指示します。コメントを使用した診断の無効化の詳細については、ドキュメントを参照してください。
  • Plog Converterユーティリティを使用してレポートを変換する場合は、「-d」スイッチを使用して診断をオフにできます。

C＃の場合はそれぞれV3144を意味し、Javaの場合-V6071を意味します。

危険なライセンスのリストに追加する


ツールで現在検出されていない「バイラル」ライセンスの種類もわかっている場合は、フィードバックフォームからそれらをお知らせください。そして、次のリリースでそれらの識別を追加します。

新しい診断


C、C ++（一般）


  • V1040。事前定義されたマクロ名のスペルのタイプミスの可能性。
  • V1041。クラスメンバーはぶら下がり参照で初期化されます。
  • V1042。このファイルはコピーレフトライセンスでマークされているため、派生ソースコードを開く必要があります。
  • V1043。ヘッダーでグローバルオブジェクト変数が宣言されています。このヘッダーファイルを含むすべての翻訳単位で、そのコピーが複数作成されます。

それとは別に、番号V1040の診断に注目します。ベータステータスであっても、彼女はすでに有名なCMakeプロジェクトのライブラリで興味深い間違いを発見しました：

V1040事前定義されたマクロ名のスペルのタイプミスの可能性。「__MINGW32_」マクロは「__MINGW32__」に似ています。winapi.h 4112

/* from winternl.h */
#if !defined(__UNICODE_STRING_DEFINED) && defined(__MINGW32_)
#define __UNICODE_STRING_DEFINED
#endif

ここで、彼らは__MINGW32_という名前のタイプミスをしました。最後に、1つのアンダースコアが欠落しています。この名前のコードで検索すると、プロジェクトで実際に両側に2つの下線が付いたバージョンが使用されていることを確認できます。


CMakeプロジェクトで見つかったすべてのエラーは、記事「CMake：コードの品質がプロジェクトに許されない場合」で確認できます。

SonarQube 7.9 LTS


待望のリリース


SonarQube 6.7のLTSバージョンの最後のリリースからほぼ2年が経過しました。SQ 7.x以降、特別な熱意を持った新しいバージョンが期待されていました。LTSリリースが行われると、ユーザーは積極的にそれに切り替え始め、さまざまな問題が発生しました。SQ 7.9.1 LTSはすぐにマイナーな修正とともにリリースされ、サードパーティのプラグイン開発者もパッチを準備しました。

幸い、PVS-Studioプラグインでは、Java 11への移行に関連する小さな問題が1つしかありませんでしたが、それをすぐに修正し、お客様はすぐに作業バージョンに切り替えました。

以前のバージョンのSonarQubeとの互換性も維持しており、サポートされているバージョンのリストは次のようになっています：SonarQube 6.7 LTS以降。

サポートされるより多くの言語プラグイン


PVS-Studioプラグインは、分析結果をSonarQubeデータベースの形式に変換するだけです。つまり、PVS-Studuio分析結果をSQにロードするだけです。ただし、SonarScannerユーティリティを完全に操作するには、プログラミング言語用のプラグインをインストールする必要があります。私たちは既存のものを開発し始めなかったので、既存の人気のあるプラグインのサポートを追加するだけです。このバージョンでは、Sonar CコミュニティおよびSonarCFamily言語プラグインとの互換性を追加しました。

サポートされる言語プラグインのリスト全体は次のようになります。

  • Sonar C ++コミュニティ
  • ソナーCコミュニティ
  • SonarC＃
  • ソナージャワ
  • SonarCFamily（SQ Developer Edition）

PVS-Studio分析結果をダウンロードするには、このリストから少なくとも1つのプラグインをインストールするだけで十分です。ほとんどのユーザーは、Sonar C ++コミュニティ、SonarC＃またはSonarJavaをインストールするだけで済みます。より具体的なプロジェクトでは、他のプラグインが必要になる場合があります。

プラグインを選択するときは、コミュニティプラグインがSonarCFamilyと互換性がないという事実を考慮する必要があります。ただし、SonarQube Community Editionのみを使用している場合は、このような問題は発生しません。

Jenkinsのプラグイン


PVS-Studioプラグイン

PVS-Studioプラグインは、Jenkins継続的インテグレーションシステムのPVS-Studioアナライザーの結果をHTML形式で公開するように設計されています。以前は、このプラグインはWindowsユーザーのみが利用できました。レポートコンバーターを自動的に呼び出し、Windowsでのみ実行しました。 PVS-Studio 7.04では、プラグインはHTMLレポートのみをサポートします。これは別のステップで生成する必要がありますが、これによりプラグインはクロスプラットフォームになりました。

警告NGプラグイン

Jenkinsには、さまざまなツールからの分析結果を表示するための便利な警告次世代プラグインプラグインがあります。最近、PVS-Studioのサポートを追加しました。このプラグインを使用してPVS-Studio分析結果をロードする機能は、バージョン6.0.0で利用可能になりましたPVS-Studio 7.04のリリースと同時にリリースされたリリース：


その他の改善


PVS-Studio_Cmd

A 特殊な動作モード-資格情報が追加されましたPVS-Studio_Cmd.exe。このモードでは、GUIインターフェイス（たとえば、Visual StudioのプラグインまたはユーティリティCおよびC ++コンパイラモニタリングUI）を使用せずに、設定ファイルを作成してライセンス情報を入力できます。このモードは、ビルドサーバー（GUIユーティリティが説明されていない場合があります）、コンテナー内、およびクラウドソリューションと統合されている場合に特に関連します。

アンリアルエンジンプロジェクトの分析

Visual Studio用のPVS-Studioプラグインで、AutoloadUnrealEngineLogオプションが追加されましたを含めると、分析に合格した後、アナライザーレポートをPVS-Studio出力ウィンドウに自動的にアップロードできます。このオプションがない場合、ログのロードはプラグインメニューから手動で行う必要があります。

また、ドキュメントセクション「Unreal Engineプロジェクトのテスト」では、1つのアクションでアセンブリと分析を可能にする標準のアセンブリスクリプトの変更について説明しました。スクリプトを変更せずに（-StaticAnalyzer = PVSStudioフラグを起動引数に追加する場合）、プロジェクトのみを分析し、ビルドする必要はありません。

CLMonitor

CLMonitorユーティリティでは、特定のプロセスのコンパイラの起動を追跡する機能が追加されました。これにより、複数のプロジェクトを並行してビルドする場合でも、特定のプロジェクトにのみ関連するコンパイラーの開始を追跡できます。このモードで作業するには、フラグ--parentProcessID％PID％（CLMonitorが指定された子に対して相対的な子プロセスを監視する）と--attach（CLMonitorが現在のコンソールに対して相対的な子プロセスを監視する）を使用します。

pvs-studio-analyzer LinuxおよびMacOSでプロジェクトをテストするように設計されている

ユーティリティpvs-studio-analyzerに、フラグ--ignore-ccacheが追加され、以下を提供します。

pvs-studio-analyzer analyze ... --ignore-ccache ...

プロジェクトアセンブリでccacheユーティリティが使用されている場合、PVS-Studioはインクリメンタル分析モードで動作します。ccacheユーティリティのキャッシュを削除せずにプロジェクトを完全にチェックするには、このフラグを使用します。
