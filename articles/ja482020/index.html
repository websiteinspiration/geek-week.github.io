R、モンテカルロおよびエンタープライズタスク

うらやましいほどの規則性で実際的な問題を解決するとき、会議で提起されるか、空中にある2つの典型的なアプローチに対処する必要があります。


  • オプション1-一般的な方法で特定の問題を解決しましょう。この方法でそれを解決すると、私たちは途中でもっと多くを解決することができます。
  • オプション2-非線形システムの動作を予測するために、予測、ML、および流行の多くを予測しましょう。やっぱり涼しい。

ただし、そのようなアプローチの適用は、問題の最初の定式化に常に適切であるとは限りません。
これまでの刊行物の続きです。


なぜそのようなアプローチは冗長なのでしょうか？はい、多くの理由で：問題の解決策を見つけるのにかかる時間、計算にかかる時間、コンピューティングリソースの量の要件、高精度での計算の実行、間違ったモデルの構築、直接的な問題に対する分析ソリューションの高い複雑さ、逆問題の解決の高い複雑さなど。


しかし、そのような問題を解決する古典的な方法があります。これは、強力なコンピュータを手元に持っている場合に特に便利です。モンテカルロ法。統計 直接的な問題に対する複数のソリューションの結果の分析。これは、原則として、アルゴリズム化が可能です。
必要な答えは、最初の概算として、問題の理解、コーディング、および計算計算の実行を含め、数時間で取得できます。


以下は2つの例です。


例1.ネストされたリストの組み合わせ


最初のビジネスタスクは、少数の入力フレーズに基づいてチャットボットのトレーニングベースを拡張することです。ソースフレーズの大規模な選択は、単に物理的にではありません。しかし、それは文の構造の予備的な分析と言語の規則によって許可された順列の生成によって大幅に拡張できます。たとえば、「部屋6のパイプフロー」と「部屋6のパイプフロー」は本質的に同じです。

ff <- function(x){
  #    ,          
  res <- if(is.list(x)) {
    sample(x, length(x), replace = FALSE) %>%
      purrr::map(ff)
  } else {
    x
  }

  res
}

procLine <- function(row){
  purrr::map(row, ff) %>%
    #   
    rlang::squash_chr() %>%
    stri_c(collapse = "")
}

wks <- 1
future::plan(multiprocess, workers = wks)
tic(glue("Generating permutations @ {wks} thread(s)"))
