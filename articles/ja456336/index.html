Reactアプリケーションの開発時にReduxを使用するための11のヒント

Reactアプリケーションの開発に関しては、コードアーキテクチャの観点から、小規模なプロジェクトの方が大規模なプロジェクトよりも柔軟であることがよくあります。より大きなアプリケーションを対象とした実用的な推奨事項を使用して、そのようなプロジェクトを作成することに問題はありません。しかし、これはすべて、小規模なプロジェクトの場合、単に不要な場合があります。アプリケーションが小さいほど、「最適」ではないが、その実装に多くの時間を必要としない、単純なソリューションの使用を「優先」します。 これにもかかわらず、この資料で提供される推奨事項のいくつかは、あらゆる規模のReactアプリケーションを対象としています。





本番アプリケーションを作成したことがない場合、この記事は大規模ソリューションの開発の準備に役立ちます。このようなものは、次のプロジェクトの1つになる可能性があります。プログラマーに起こり得る最悪の事態は、彼がプロジェクトに取り組み、アプリケーションのスケーラビリティーと保守性を改善するために大量のコードをリファクタリングする必要があることに気付いたときです。リファクタリングの前にプロジェクトに単体テストがないと、すべてがさらに悪くなります。

この資料の作者は、読者にそれについて彼の言葉を取るように求めます。彼は同じような状況にありました。したがって、彼は特定の時間内に解決する必要があるいくつかのタスクを取得しました。最初、彼は自分がしたことはすべて素晴らしいと思った。そのような考えのもとは、彼のWebアプリケーションは、変更を加えた後も機能し続け、同時に迅速に機能し続けるというものでした。彼はReduxの使い方、ユーザーインターフェイスコンポーネント間の通常の相互作用を確立する方法を知っていました。彼は還元剤と作用の概念を深く理解しているようでした。彼は不死身だと感じました。

しかしその後、未来は忍び寄った。

アプリケーションでの作業の数ヶ月後、15以上の新しい機能が追加されました。その後、プロジェクトは制御不能になりました。Reduxライブラリを使用したコードは、保守が非常に困難になりました。なぜそうなったのですか？最初は、プロジェクトが長く雲のない生活を期待していたように見えませんでしたか？

記事の著者は、そのような質問をして、彼が自分の手でプロジェクトに時限爆弾を置いたことに気付いたと言っています。

大規模なプロジェクトでReduxライブラリを正しく使用すると、これらのプロジェクトが成長しても、コードをサポートされた状態に保つことができます。

Reduxを使用してスケーラブルなReactアプリケーションを開発したい人のための11のヒントを紹介します。

1.アクションコードと定数を1か所に配置しないでください


定数とすべてのアクションが同じ場所に配置されているいくつかのReduxチュートリアルに出くわすかもしれません。ただし、このアプローチは、アプリケーションが成長するにつれて、すぐに問題を引き起こす可能性があります。定数は、たとえばに個別に保存する必要があります./src/constants。その結果、定数を検索するには、1つだけのフォルダーを調べ、複数のフォルダーを調べる必要はありません。

さらに、アクションを格納する個別のファイルの作成は完全に正常に見えます。このようなファイルは、互いに直接関連するアクションをカプセル化します。たとえば、1つのファイル内のアクションは、何をどのように使用するかという点で類似点がある場合があります。

export const upgradeStaff = (slot, enhancements) => {
  return (dispatch, getState, { api }) => {
    //              
    const state = getState()
    const currentEquipment = state.classes.sorceress.equipment.current
    const staff = currentEquipment[slot]
    const isMax = staff.level >= 9
    if (isMax) {
      return
    }
    dispatch({ type: consts.UPGRADING_STAFF, slot })

    api.upgradeEquipment({
      type: 'staff',
      id: currentEquipment.id,
      enhancements,
    })
    .then((newStaff) => {
      dispatch({ type: consts.UPGRADED_STAFF, slot, staff: newStaff })
    })
    .catch((error) => {

    })
  }

3.有益な変数名を使用する


変数に名前を付けることは、一見すると、基本的な作業のように思えるかもしれません。しかし、実際には、この作業は最も難しい作業の1つであることが判明する場合があります。

変数名の選択は、一般に、クリーンなコードを書くための実用的なガイドラインに関連しています。一般に「変数名」というものが存在するのは、コード開発のこの側面が実際に非常に重要な役割を果たすためです。変数名の選択に失敗すると、将来自分やチームメンバーに危害を加えることになります。

他の人のコードを編集しようとしたときに、このコードが正確に何をしているかを理解するのに困難に直面したことがありますか？他の人のプログラムを実行して、期待どおりに機能しないことに気づいたことがありますか？

そのような場合に、いわゆる「ダーティコード」に遭遇したことを証明したいと思います。

大規模なアプリケーションで同様のコードを処理する必要がある場合、これは悪夢です。残念ながら、これはかなり頻繁に発生します。

ここに人生からの1つのケースがあります。私は1つのアプリケーションからReactフックコードを編集しましたが、その時点で彼らは私にタスクを送信しました。それは、医師に関する追加情報を表示する機能をアプリケーションで実現することでした。この情報は、医師のプロフィール写真をクリックした患者に表示されているはずです。テーブルからそれを取得する必要があり、サーバーへの次の要求を処理した後にクライアントに到達する必要がありました。

このタスクは難しくありませんでした。私が遭遇した主な問題は、必要なものがプロジェクトコードのどこにあるのかを見つけるのに多くの時間を費やす必要があったことです。

私はに応じてコードに見えたinfo、dataToSend、dataObject、私の見解では、サーバから受信したデータに接続され、他の人。 5〜10分後、必要なデータを処理するコードを見つけました。彼らが発見されたオブジェクトは、paymentObject。私の意見では、支払いに関連するオブジェクトには、CVVコード、クレジットカード番号、支払者の郵便番号、およびその他の同様の情報が含まれている可能性があります。私が発見したオブジェクトには11のプロパティがありました。そのうち3つだけが支払いに関連していました。支払い方法、支払いプロファイルID、およびクーポンコードのリストです。

私の前のタスクを解決するために必要なこのオブジェクトに変更を加える必要があったため、状況も改善しませんでした。

4.構成済みのアプリケーションデータストリームのデータ構造またはタイプを変更しないでください。


私がこれまでに犯した最大の間違いの1つは、すでに構成されているアプリケーションデータストリームのデータ構造を変更することでした。新しいデータ構造は、配列を反復する代わりに、メモリに格納されたオブジェクト内のデータを検索するための高速な方法を使用したため、パフォーマンスが大幅に向上します。しかし、それは遅すぎました。

これをしないようにお願いします。おそらく、このようなものは、これがアプリケーションのどの部分に影響を与えるかを正確に知っている人にしか提供できません。

そのようなステップの結果は何ですか？たとえば、何かが最初に配列になり、次にオブジェクトになった場合、アプリケーションの多くの部分の操作が中断される可能性があります。構造化データの表示の変更によって影響を受ける可能性のあるコード内のすべての場所を思い出すことができたと信じて、私は大きな間違いを犯しました。ただし、このような場合、変更の影響を受けるコードが常に存在し、誰も覚えていません。

6.ユニット、エンドツーエンド、統合テストを作成する


アプリケーションのサイズが大きくなると、プログラマーがテストでカバーされていないコードを編集することはさらに恐ろしくなります。たとえば、誰かがに保存されているコードを編集し、src/x/y/z/それを本番環境に送信することを決定した可能性があります。同時に、加えられた変更がプロジェクトのプログラマが考えていなかった部分に影響を与える場合、実際のユーザーが遭遇するエラーですべてが終了する可能性があります。プロジェクトにテストがある場合、プログラマーはコードが本番稼働するずっと前にエラーについて調べます。

7.ブレインストーム


プログラマは、プロジェクトに新しい機能を導入する過程で、ブレインストーミングを拒否することがよくあります。これは、そのようなアクティビティがコードの記述に関連していないために発生します。これは、タスクを完了する時間がほとんどない場合に特によく発生します。

ところで、なぜアプリケーションの開発中にブレーンストーミングをしなければならないのですか？

実際には、アプリケーションが複雑になるほど、プログラマーは個々の部分に注意を払わなければなりません。ブレーンストーミングは、コードのリファクタリングにかかる​​時間を短縮するのに役立ちます。それらが開催された後、プログラマーはプロジェクトの完了中に何が問題になるかについての知識で武装します。多くの場合、プログラマーは、アプリケーションの開発中に、すべてを最良の方法で行う方法について少なくとも少しでも考えなくてもかまいません。

そのため、ブレーンストーミングは非常に重要です。そのようなイベントの間、プログラマーはコードのアーキテクチャーを検討し、プログラムに必要な変更を加える方法を考え、これらの変更のライフサイクルを追跡し、それらを操作するための戦略を作成できます。すべての計画を自分の頭で独占的に保持する習慣をつけることは価値がありません。これは、過度に自信のあるプログラマが行うことです。しかし、すべてを絶対に思い出すことは単に不可能です。そして、何かがおかしいとすぐに、問題が次々と現れます。これは、ドミノの動作原理です。

ブレーンストーミングはチームでも役立ちます。たとえば、仕事中に誰かが問題に遭遇した場合、彼のために生じた問題はすでに十分に検討されている可能性があるため、ブレーンストーミングセッションの資料に目を向けることができます。ブレーンストーミングセッション中に作成されたメモは、問題を解決するための計画の役割を果たす可能性があります。この計画により、実行された作業量を明確に評価できます。

8.アプリケーションのモックアップを作成する


アプリケーションの開発を開始する場合は、アプリケーションの外観とユーザーがアプリケーションを操作する方法を決定する必要があります。つまり、モックアプリケーションを作成する必要があります。これにはさまざまなツールを使用できます。

Moqupsは、よく耳にするアプリモックアップツールの1つです。これは、HTML5とJavaScriptを使用して作成された高速なツールであり、システムに特別な要件を課しません。

モックアプリケーションを作成すると、開発プロセスが大幅に簡素化および高速化されます。レイアウトは、アプリケーションの個々の部分の関係、およびそのページに表示されるデータの種類に関する情報を開発者に提供します。

9.アプリケーションのデータフローを計画する


アプリケーションのほぼすべてのコンポーネントは、いくつかのデータに関連付けられます。一部のコンポーネントは独自のデータソースを使用しますが、ほとんどのコンポーネントはコンポーネント階層でそれらの上位エンティティからデータを受け取ります。同じデータが複数のコンポーネントによって共有されるアプリケーションのこれらの部分については、階層の最上位にある集中化された情報ストレージを提供すると便利です。Reduxライブラリが開発者に非常に貴重な支援を提供できるのは、このような状況です。

アプリケーションの作業中に、このアプリケーションでデータが移動する方法を示す図を作成することをお勧めします。これは、明確なアプリケーションモデルの作成に役立ちます。さらに、プログラマーによるコードとアプリケーションの認識について話します。このようなモデルは、さらに、レデューサーの作成にも役立ちます。

10.データアクセス機能を使用する


アプリケーションのサイズが大きくなると、そのコンポーネントの数も大きくなります。また、コンポーネントの数が増えると、セレクター（react-redux ^ v7.1）またはの使用頻度でも同じことが起こりmapStateToPropsます。コンポーネントやフックが、のような構造を使用して、アプリケーションのさまざまな部分の状態フラグメントにアクセスすることが多いと思いますuseSelector((state) => state.app.user.profile.demographics.languages.main)。もしそうなら、それはあなたがデータアクセス関数の作成について考える必要があることを意味します。そのような機能を持つファイルは、コンポーネントとフックがそれらをインポートできるパブリックな場所に保存する必要があります。同様の関数は、フィルター、パーサー、またはデータ変換用のその他の関数です。

ここにいくつかの例を示します。

11.構造化解除と構文の拡散を使用して、プロパティ内のデータのフローを制御する


建設props.somethingより建設を使用する利点は何somethingですか？

分解を使用しない場合の外観は次のとおりです。

const Display = (props) => <div>{props.something}</div>

以下は同じですが、すでに分解を使用しています。

const Display = ({ something }) => <div>{something}</div>

解体を使用すると、コードが読みやすくなります。しかし、これは彼のプロジェクトへのプラスの影響に限定されません。構造化解除を使用すると、プログラマーは、コンポーネントが正確に何を受け取り、コンポーネントが正確に何を出力するかについて決定を迫られます。これにより、他の誰かのコードを編集しrenderて、コンポーネントが使用するすべてのプロパティを検索するためにメソッドの各行を調べる必要がなくなります。

さらに、このアプローチは、デフォルトのプロパティ値を設定するための便利な機会を提供します。これはコンポーネントコードの最初に行われ、コンポーネント本体に追加のコードを記述する必要がなくなります。

const Display = ({ something = 'apple' }) => <div>{something}</div>

以前に次の例のようなものを見たことがあるかもしれません。

const Display = (props) => (
  <Agenda {...props}>
    {' '}
    //     Agenda
    <h2><font color="#3AC1EF">Today is {props.date}</font></h2>
    <hr />
    <div>
      <h3><font color="#3AC1EF">▍Here your list of todos:</font></h3>
      {props.children}
    </div>
  </Agenda>
)

このような構造は読みにくいですが、これが唯一の問題ではありません。そのため、エラーがあります。アプリケーションが子コンポーネントも表示する場合はprops.children、画面に2回表示されます。プロジェクトの作業がチームで行われ、チームメンバーが十分に注意していない場合、そのようなエラーの可能性は非常に高くなります。

代わりにプロパティを破棄すると、コンポーネントコードがより明確になり、エラーの可能性が減少します。

const Display = ({ children, date, ...props }) => (
  <Agenda {...props}>
    {' '}
    //     Agenda
    <h2><font color="#3AC1EF">Today is {date}</font></h2>
    <hr />
    <div>
      <h3><font color="#3AC1EF">▍Here your list of todos:</font></h3>
      {children}
    </div>
  </Agenda>
)
