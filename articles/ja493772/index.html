Kubernetesを使用する最も簡単なチーム


まず、Kubernetesを操作する上でおそらく最も単純で最も便利なアクションです。次のコマンドkubectlは、bashシェルでコマンドのオートコンプリートを有効にします。

echo "source <(kubectl completion bash)" >> ~/.bashrc

オートコンプリートkubectlは.bashrcファイルに書き込まれ、シェルが起動するたびに自動的にアクティブになります。これにより、などの長いコマンドとパラメータのセットが高速化されall-namespacesます。詳細については、Kubernetes bashのヘルプをご覧ください。

名前空間のメモリとCPUに対するデフォルトの制限


たとえば、アプリケーションが正しく記述されていない場合、たとえば、毎秒新しいデータベース接続が開かれますが、決して閉じられない場合、クラスターでメモリリークが発生します。また、デプロイメント中にアプリケーションのメモリ制限がない場合、ノード障害が発生する可能性があります。

これを防ぐために、Kubernetesでは各名前空間にデフォルトの制限を設定できます。それらは特定の名前空間のyamlファイルに書き込まれます。このようなファイルの例を次に示します。

apiVersion: v1
kind: LimitRange
metadata:
  name: mem-limit-range
spec:
  limits:
  - default:
      memory: 512Mi
    defaultRequest:
      memory: 256Mi
    type: Container

そのようなyamlを作成し、任意の名前空間に適用します。たとえば、名前空間にlimit-example。これで、この名前空間にデプロイされたコンテナには、このコンテナに別の個別の制限が追加で設定されていない限り、512Mi制限が適用されます。

古いバージョンのKubernetesのガベージコレクション


Kubeletはデフォルトで、var / lib / dockerが使用可能なディスク容量の90％を占めると、ガベージコレクションを開始します。これは問題ありませんが、Kubernetes 1.7より前では、使用されるiノード（inodes）の数にデフォルトの制限はありませんでした。これは、ファイルシステム内のファイルの数に対応しています。

潜在的に、var / lib / dockerコンテナはディスク領域の50％しか使用できませんが、iノードが不足し、ワーカーに問題が発生する可能性があります。

1.4から1.6までの古いバージョンのkubeletでは、次のフラグを追加する必要があります。

--eviction-hard
=memory.available<100Mi,nodefs.available<10%,nodefs.inodesFree<5%

1.7以降のバージョンでは、このフラグはデフォルトで設定されています。ただし、以前のバージョンはiノードの制限に従っていません。

Minikube ...小さくても強力なローカルKubernetes


Minikubeは、ローカルのKubernetesクラスタを起動する最も簡単な方法です。それは簡単なコマンドで始まります：

minikube start

このコマンドの結果、実際のKubernetesクラスターがコンピューターで実行されています。

囲炉裏の予算を管理する


Kubernetesクラスター内のアプリケーションのダウンタイムがないことを保証する方法は？ PodDisruptionBudgetおよびPodDisruptionBudget。

クラスターは定期的に更新され、ノードは空になります。何も立ち止まらず、それが現実です。複数のインスタンスがあるすべてのデプロイメントでは、必ずPDB（PodDisruptionBudget）を含める必要があります。これは、クラスターに適用される単純なyamlファイルで作成されます。特定のPDBのカバレッジエリアは、ラベルセレクターによって決定されます。

注： PDB予算は、自発的な混乱の場合にのみ考慮されます。ハードウェア障害のような状況では、PDBは機能しません。

PDBの例：

apiVersion: policy/v1beta1
kind: PodDisruptionBudget
metadata:
  name: app-a-pdb
spec:
  minAvailable: 2
  selector:
      matchLabels:
        app: app-a

2つの主なパラメーターはmatchLabelsおよびminAvailableです。最初のパラメータは、どのアプリケーションに予算があるかを示します。たとえば、app: app-aとタグを使用したデプロイがある場合app: app-b、このPDBは最初のものにのみ適用されます。ユニットを空にする（クリーニングする）ときに、

パラメーターがminAvailable考慮されます。たとえば、この例では、荒廃中にapp: app-a2つのインスタンスを除くすべてのインスタンスが置き換えられます。

これにより、アプリケーションの特定の時点で実行するインスタンスの数を制御できます。

アプリケーションヘルスモニタリング


このような監視は、2つの方法で可能です。準備サンプルまたは活性サンプルを使用します。

最初の準備テストでは、コンテナがトラフィックを受信する準備ができているかどうかを判断します。

2番目（活性度）は、コンテナーが機能しているか、再起動する必要があるかを示します。

適切な構成は、デプロイのためにyamlに追加されるだけです。そこで、タイムアウト、遅延時間、再試行回数を指定できます。詳細については、Kubernetesのドキュメントをご覧ください。

どこでもタグ付け


タグは、Kubernetesの基本的な概念の1つです。それらは、オブジェクトが互いに自由に通信することを可能にし、ラベルに基づいてクエリを作成します。 Kubernetesでは、クライアントに移動して、特定のタグでイベントを監視することもできます。

ラベルを使用すると、ほとんどすべてを行うことができますが、1つのクラスターでプログラムを実行するための複数の環境を作成するのが良い例です。

あなたが同じクラスタを使用していると仮定devしてqa。つまり、アプリケーションapp-aが両方の環境qaとで同時に実行されている可能性がありますdev。この場合、適切なパラメーターを指定することで、特定の環境のアプリケーションインスタンスに個別にアクセスできますenvironment。例えば、app: app-aおよびenvironment: dev単一の環境のために、とapp: app-aとenvironment: qa二番目。

これにより、アプリケーションの両方のインスタンスにアクセスして、同時にテストすることができます。
