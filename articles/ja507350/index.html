PVS-Studioアナライザー：潜在的なJava SE API互換性問題の特定











2019年は会議に関して非常に忙しい年でした。私たちのチームは何週間も出張に行くことができました。そしてご存じのように、会議は知識を共有する時です。プレゼンテーションを行い、スタンドで多くの興味深いことを伝えただけでなく、会議の参加者とのコミュニケーションやスピーカーからも多くのことを学びました。したがって、秋の会議Joker 2019で、Dalia Abo Sheashaのレポート「Migrating beyond Java 8」は、Javaの異なるバージョン間のJava SE APIの非互換性を特定できる新しい診断ルールを実装するように私たちを鼓舞しました。これについては後で説明します。



Java SE互換性問題の関連性



Java SE 14はすでにリリースされていますが、それでも多くの企業が以前のバージョンのJava（Java SE 6、7、8、...）を使用し続けています。時間が実行されており、Javaは常に更新されているため、Java SE APIの異なるバージョン間の互換性の問題は毎年ますます重要になっています。



Java SEの新しいバージョンがリリースされると、通常、以前のバージョンとの下位互換性があります。たとえば、Java SE 8に基づいて開発されたアプリケーションは、Javaの11バージョンで問題なく実行できます。ただし、実際には、いくつかのクラスとメソッドで一部の非互換性が発生する場合があります。この非互換性は、一部のAPIが変更されるという事実にあります。APIは削除され、動作が変更され、廃止されたものとしてマークされます。



この問題は、プロジェクトを最新のJava SEに移行することを考え始めたときにのみ悪化します。または、アプリケーションのテクニカルサポートが、アプリケーションが正しく動作しない、またはまったく起動できないという手紙を受け取ることが増えたとき。



これは注目を集めるのに十分だと思います！



既存のツールキット



Javaのあるバージョンから別のバージョンに切り替えるための万能のソリューションはありません。また、アプリケーションがさまざまな重要なソリューションを使用した長年にわたる継続的な開発の結果である場合、Javaの最新バージョンへの移行に戸惑うと、かなり面倒なプロセスに遭遇する可能性があります。



このプロセスは、問題のある、または問題のある可能性のあるAPIを特定することで構成されています。これらのAPIを確認し、代替ソリューションで置き換える必要があります。これはビジネスロジックに深刻な影響を与える可能性があり、修正とテストに長い時間がかかる可能性があります。新しいバージョンのJava SEに切り替えるだけでなく、多くのバージョンとの互換性を確保する必要がある場合、このタスクはさらに複雑になります。



さまざまなJava SE間のAPIの非互換性を特定するというトピックについてインターネットをじっくり見てきたので、私はJDKに付属しているツール、javac、jdeps、jdeprscanに出くわしました。



このビジネスにはサードパーティのツールはありませんでしたが、Joker 2019レポート-Migration Toolkit for Application Binariesで私が会えて光栄でした。



javac



アプリケーションを開発するときは、コンパイラの警告を忘れないでください。警告に対する考え方は人それぞれです。



  • 誰かが厳密にそれらを監視し、少し何かがすぐに警報を鳴らします。おそらく少し偏執狂かもしれませんが、将来的には技術的なリスクを減らす可能性があります。
  • 誰かがそれらを別のファイルに集め、彼は必ずそれらをレビューし、彼がどのように解放されるかを考えます。
  • 通常、警告は無視されます。


たとえば、非推奨としてマークされたメソッドまたはクラスを使用しないだけで、問題の一部を遅滞なく解決できるため、最初のオプションが適切です。このようなAPIの使用はブロッキングの問題ではありませんが、新しくリリースされたバージョンのJavaを使用すると、アプリケーションの動作が異なったり、場合によってはクラッシュする可能性があるため、注意が必要です。



jdeps



アプリケーションをJava SEのより新しいバージョンに移行する問題に既に困惑している場合は、jdepsが急いでお手伝いします。



jdepsは、アプリケーションとライブラリの依存関係の静的分析を実行するコマンドラインツールで、* .classファイルまたは* .jarを入力として受け入れます。 Java SE 8以降、JDKにバンドルされています。



そして、ここで--jdk-internalsオプションを指定してこのツールを実行すると、各クラスがどの内部JDK APIに依存しているかがわかります。これは、内部APIがJavaの将来のバージョンで変更されないことを保証しないため、非常に重要なポイントです。



例を見てみましょう。 Java 8アプリケーションを長い間開発していて、新しいバージョンで使用されるJava SE APIの互換性に戸惑ったことがないとしましょう。アプリケーションをJava 11などに移行することについて質問がありました。この場合、抵抗を最小限に抑え、すぐにJava 11を使用してアプリケーションを起動できます。しかし、何か問題が発生し、アプリケーションがクラッシュし始めたとします。この場合、アプリケーションの* .jarファイルを設定することにより、Java 11からjdepsを実行できます。



結果は次のようになります。













出力は、Java 11の依存関係sun.misc.BASE64Encoderが削除されることを示しています。また、Java 11で最初に起動したときのアプリケーションのクラッシュは、java.lang.NoClassDefFoundErrorエラーが原因である可能性が高いです。 jdepsは、どちらでもないことを通知するだけでなく、現在の依存関係の代わりに使用できる代替依存関係を提供できます。この場合、リモートの依存関係をjava.util.Base64に置き換えることを提案しました。



ツールの2番目の警告は、コード内でまだ別の内部依存関係を使用していることを示していますが、今のところそれは正しいです。 Javaの将来のバージョンでは、このAPIに変更がありますが、これを考慮する必要があります。



もちろん、jdepsは内部JDKコンポーネントの使用をテストするだけではありません。これはこの記事の範囲には含まれていませんが、公式ページでその機能に慣れることができます。



jdeprscan



jdeprscanの目標はjdepsの目標とまったく同じです。つまり、不要で問題のあるAPIを見つけるのに役立ちます。



jdeprescanは* .jarファイル（または* .classファイルのその他のコレクション）をスキャンして、古いAPI要素を使用するための静的分析ツールです。古いAPIの使用はブロッキングの問題ではありませんが、注意が必要です。Java SE 9以降、JDKにバンドルされています。



また、アプリケーションをJava 11に移行する必要があると仮定します。この場合、コマンドを実行して 



jdeprscan --release 8 app.jar


Java 8で非推奨となったAPI、つまり、Javaの将来のバージョンで削除される可能性があるAPIのリストを取得します。すべての警告を修正した後、実行できます 

jdeprscan --release 11 app.jar


これにより、Java 11ですでに非推奨となっているAPIのリストが生成されます。このようにして、推奨されていないすべてのAPIを見つけて（必要に応じて）修正できます。



アプリケーションバイナリ用の移行ツールキット



このツールは、さまざまなサーバー（JBOSS、WebShere、Tomcat、WebLogicなど）にデプロイする前に、潜在的な問題についてユーザーアプリケーションをすばやく評価できるように設計されています。このツールでは、すべての機能に加えて、異なるバージョンのJava SE APIの違いを識別することもできます。



このツールの概要を見てみましょう。



ツールのクイック起動は次のようになります。 



java -jar binaryAppScanner.jar yourApp.jar --analyzeJavaSE 
--sourceJava=oracle8 --targetJava=java11 ....


analyzeJavaSE オプションは、ヘルプを呼び出すことで学習できるさまざまなパラメーターの使用を意味します。



分析を開始すると、すぐにWebブラウザーにレポートが表示されます。















すべてが画面に収まるわけではありません=（（そして、このツールをまだ実行していないので、言葉で説明します。



レポートでは、3つのレベルの重大度を持つルールのトリガーを確認できます。



  • 重大度の高いルール（リモートAPI、アプリケーションを動作不能にして修正する必要がある動作の変更）。
  • 警告ルール（問題を引き起こす可能性のあるAPIの動作を変更するため、調査する価値があります）;
  • 情報規則（古いAPI、動作をわずかに変更する可能性があるがプログラムには影響しないAPI動作を使用）。


各操作を展開して、何をどのように行うかの説明を表示できます。情報メッセージには推奨事項があり、さらにjdepsを実行するように言われています。私たちはアプリケーションの移行に焦点を合わせていると彼らは言っており、jdepsは（発見したものに加えて）内部JDKパッケージの問題を検出するのにさらに役立ちます。



また、レポートの下部には、分析が実行されたルールのリストが表示されます。



Eclipse IDEを使用している場合は、プラグインを使用できます。より詳細な調査については、彼らのページをご覧ください。



PVS-Studioでの実装



検討したツールを調査した後、Java SE APIの異なるバージョンの潜在的な互換性の問題の検索は、静的分析にふさわしい作業であるという結論に達しました。



説明したツールを使用すると、アプリケーションの移行や、Java SEの最新バージョンでのアプリケーションのパフォーマンスを低下させる可能性のあるAPIの検索が簡単になります（アプリケーションを移行しない場合）。しかし、問題を特定するために開発プロセスとは別にコマンドラインで実行する必要があるときは常に、これらのツールを考えると、これはあまり便利ではないと判断しました。また、開発の非常に早い段階で問題のあるコードまたは潜在的に問題のあるコードを検出するには静的分析が必要であるという事実から始めて、診断ルールV6078を実装しました「問題」APIについて通知します。



Rule V6078は、コードが特定のJava SE API関数およびクラスに依存していることを事前に警告します。これにより、Javaの将来のバージョンで問題が発生する可能性があります。また、特定の機能の実装の最初の段階では、このAPIに拘束されないため、将来の技術的なリスクが軽減されます。



診断ルールは、次の場合に警告を発行します。



  • メソッド/クラス/パッケージがターゲットバージョンのJavaで削除された場合。
  • メソッド/クラス/パッケージがターゲットバージョンのJavaで非推奨とマークされている場合。
  • メソッドが署名を変更した場合。


このルールでは現在、バージョン8から14までのOracle Java SEの互換性を分析できます。ルールをアクティブにするには、構成する必要があります。

  • 除外パッケージ-互換性分析から除外するパッケージ（パッケージはコンマでリストされます）。












Gradleのプラグイン



Gradleプラグインを使用して、build.gardleでアナライザー設定を構成する必要があります。 



apply plugin: com.pvsstudio.PvsStudioGradlePlugin
pvsstudio {
    ....
    compatibility = true
    sourceJava = /*version*/  
    targetJava = /*version*/
    excludePackages = [/*pack1, pack2, ...*/]
}


Maven用プラグイン



mavenプラグインを使用して、pom.xmlでアナライザー設定を構成する必要があります。 



<build>
  <plugins>
    <plugin>
      <groupId>com.pvsstudio</groupId>
      <artifactId>pvsstudio-maven-plugin</artifactId>
      ....
      <configuration>
        <analyzer>
          ....
          <compatibility>true</compatibility>
          <sourceJava>/*version*/</sourceJava>
          <targetJava>/*version*/</targetJava>
          <excludePackages>/*pack1, pack2, ...*/</excludePackages>        
        </analyzer>
      </configuration>
    </plugin>
  </plugins>
</build>


カーネルを直接使用する



コマンドラインから直接アナライザーを使用する場合、選択したJava SE APIの互換性分析をアクティブにするには、次のパラメーターを使用する必要があります。 



java -jar pvs-studio.jar /*other options*/ --compatibility 
--source-java /*version*/ --target-java /*version*/ 
--exclude-packages /*pack1 pack2 ... */


アナライザーの操作



Java SE 8に基づくアプリケーションを開発していて、次の内容のクラスがあるとします。 



/* imports */
import java.util.jar.Pack200;
public class SomeClass
{
  /* code */
  public static void someFunction(Pack200.Packer packer, ...)
  {
    /* code */
    packer.addPropertyChangeListener(evt -> {/* code */});
    /* code */
  }
}


診断ルールのさまざまな設定で静的分析を実行すると、次の図が観察されます。



  • ソースJava SE-8、ターゲットJava SE-9
    • 「addPropertyChangeListener」メソッドは削除されます。


  • ソースJava SE-8、ターゲットJava SE-11
    • 「addPropertyChangeListener」メソッドは削除されます。
    • 'Pack200'クラスは非推奨としてマークされます。


  • ソースJava SE-8、ターゲットJava SE-14
    • 'Pack200'クラスは削除されます。




まず、Java SE 9では、「Pack200.Packer」クラスの「addPropertyChangeListener」メソッドが削除されました。バージョン11では、クラス 'Pack200'が非推奨としてマークされたという事実がこれに追加されました。また、バージョン14では、このクラスは完全に削除されました。



したがって、Java 11でアプリケーションを起動すると、「java.lang.NoSuchMethodError」が表示されます。Java14で実行すると、「java.lang.NoClassDefFoundError」が表示されます。



この情報を知っていると、アプリケーションを開発するときに、タスクの代替ソリューションを検討することになります。



さらなる発展のためのアイデア



診断ルールの実装中に、拡張についてのアイデアが浮上しました。



  • jdeprscanとjdepsは、リフレクションを使用してカプセル化されたAPIにアクセスすることについて警告できないため、ルールを変更して、使用しようとしているAPIの種類を把握しようとします。結果は完璧ではないかもしれませんが、なぜでしょうか？！
  • さまざまなJDK実装があります（Oracle、IBM、Red Hatなど）。原則として、これらは互いに互換性があります。しかし、内部JDK APIはどの程度異なりますか？結局のところ、開発者をそれに関連付けることができるため、アプリケーションをあるJDKから別のJDKに移行するときに潜在的な問題が発生する可能性があります。


これらはすべて調査の質問であり、最終的に何が判明するか、時間がわかります=）この診断の興味深いシナリオを知っていて、PVS-Studioでそれらを見たい場合は、私たちに書いてください。



結論



潜在的な非互換性エラーの検索は、イデオロギーと完全に一致しています。PVS-Studioを使用して、コード作成の初期段階でエラーを見つけて修正します。タイプミスのように、特定の関数への呼び出しはいつでもコードに入力できるため、プロジェクトでPVS-Studioを定期的に実行すると、2倍の有用性が得られます。V6078



診断は、バージョン7.08以降のアナライザーで使用できます。ダウンロードページで、プロジェクトのアナライザーをダウンロードして試すことができます。
