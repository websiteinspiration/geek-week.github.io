数ステップでMatTableレンダリングを高速化

ほとんどのアプリケーションがさまざまなリストとテーブルで構成されていることがよくあります。毎回ホイールを再発明しないために、私は多くの人と同じように、Angular Materialテーブルをよく使用しました。

その後、これらのテーブルが大きくなり、テキストデータとネストされた要素の両方がセルに配置されました。これらすべてが大きくなり、ユーザーのマシンに適した負荷になります。そしてもちろん、誰もこれを好きではありません。

私の最後のホームプロジェクトでは、セルのほとんどがさまざまなフィールドで埋められたテーブルがありました（1つの大きなフォームであるとも言えます）。

レンダリングには約8秒かかりました（40 x 40テーブル）。

では、どのようにして大規模なリストに対してMatTableを最適化できますか？



テストケース


他の人がこの問題に対処するのを助けるために、私は小さなテストアプリケーションを書きました。 5つの列を持つすべて同じMatTableテーブル（最初は要素のID、残りは通常のMatFormFieldテキストフィールドです）。



これは単純なテーブルのように思われますが、100をそのようなテーブルのドレインにレンダリングしても、2秒以上かかることはありません（ああ、このマテリアルとその "最高の"パフォーマンス）。



この場合のレンダリング自体は（860ミリ秒）しかかかりませんでしたが、ページが2.2秒間フリーズし、ユーザーは動揺しました。

さて、300行のテーブルをレンダリングしてみましょう。ドラムロールは少し待っています。スクリプトとレンダリングに合計で約10秒かかったことがわかります。つまり、ユーザーがそのようなテーブルを300行で表示したい場合、ページは10秒間フリーズします。これは非常に恐ろしくて驚くべきことです（実際にはありません）。



実際、1000個の要素を描画するのにかかる時間を測定しようとしていましたが、私の悲惨なi7はそれに耐えられず、ページは常に落ちていました。
すでに適用されているソリューションを使用して、これを後で試行します。

問題の解決策


したがって、誰もがブラウザベースのユーティリティを使用して測定を行うことができますが、この問題の解決策はありません。

この決定は、問題の本質そのものを論じることによって私に導かれました。
問題を正しく理解することは、すでにその解決策の少なくとも半分です。
これは、テーブルデータが最初にスクリプトによって処理され、次に1つの部分として吐き出されるため、これが起こっていることを誰もが理解していると思います。次に、これがまさに特徴的なサスペンションを見る理由です。

問題が見つかりました。今では、それを解決する方法を理解する必要があります。最初の解決策が思い浮かびます。問題がすべてのデータを一度に処理している場合は、テーブルでデータを部分的に処理する必要があります。

しかし、それを行う方法は？

箱から出してすぐにこれについて何があるか考えてみましょう。最初に頭に浮かぶのは、Track By機能です。データソースを変更すると、テーブル全体は再レンダリングされず、変更のみが再レンダリングされます。

この関数をテーブルに追加しましょう：

<mat-table [trackBy]="trackByFn" [dataSource]="commonDataSource">

でも、テーブルデータがなんらかの形で変化するという条件はありません。これは、今の会話ではありません。しかし、パイプを作成すると、データソースを初期化するときに、データが分割され、テーブルにバッチで渡されます。次に、trackBy関数は、完全なレンダラーを回避するのに役立ちます。

@Pipe({
  name: 'splitSchedule'
})
export class SplitPipe implements PipeTransform {
  public transform(value: any, takeBy: number = 4, throttleTime: number = 40): Observable<Array<any>> {
    return Array.isArray(value)
      ? this.getSplittedThread(value, takeBy, throttleTime)
      : of(value);
  }

  private getSplittedThread(data: Array<any>, takeBy: number, throttleTime: number): Observable<Array<any>> {
    const repeatNumber = Math.ceil(data.length / takeBy);
    return timer(0, throttleTime).pipe(
      map((current) => data.slice(0, takeBy * ++current)),
      take(repeatNumber)
    );
  }
}

これは、ハードウェアがこのような大きなテーブルをレンダリングするのに役立つような小さなコードです。

このパイプをデータソースに適用します。


<mat-table [trackBy]="trackByFn"
           [dataSource]="commonDataSource | splitSchedule | async">

それでは、測定してみましょう。100、300、1000要素をレンダリングします。







そして、私たちは何を見ますか？実際の成功は、私たちが期待したものではありません。

  • 300要素が1秒早くレンダリングされました
  • 1000が11秒でレンダリングされ、タブは消えませんでした
  • 100要素は通常150ミリ秒長くレンダリングされます

しかし、急いで結論を出すのではなく、最初に両方の場合のページの動作を見てみましょう。





ご覧のとおり、通常の場合、ページは数秒間フリーズし、ユーザーは現時点では何もできませんが、trackByへのリンクを持つパイプを使用すると、ユーザーはテーブルをほぼ瞬時に初期化でき、アプリケーションの使用中に不快感を与えることはありません。
