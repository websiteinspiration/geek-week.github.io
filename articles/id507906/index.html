Kami menggunakan DI di Angular secara maksimal - konsep penyedia swasta

Angular memiliki mekanisme Injeksi Ketergantungan yang sangat kuat. Ini memungkinkan Anda untuk mentransfer data apa pun melalui aplikasi Anda, mengonversi dan mendefinisikan kembali data di bagian yang tepat.



Oleh karena itu, kita dapat membuat arsitektur aplikasi lebih sederhana dan lebih fleksibel: aliran data yang dapat dimengerti, koherensi kode minimal, kemudahan pengujian atau mengganti dependensi.



Namun demikian, aplikasi DI dalam digunakan cukup sederhana. Sebagai aturan, ini adalah pengenalan layanan atau transfer beberapa data global dari atas ke bawah melalui pohon injeksi ketergantungan.



Pada artikel ini, saya ingin menunjukkan cara alternatif untuk bekerja dengan data yang diperoleh dari DI. Tujuan: Untuk menyederhanakan komponen, arahan dan layanan yang menggunakan data ini.







Bagaimana DI biasanya digunakan dalam Angular



Saya melakukan review harian kode sudut di tempat kerja dan di open source. Biasanya, di sebagian besar aplikasi DI, ia turun ke fungsi berikut:



  1. Dapatkan entitas Angular dari pohon dependensi: ChangeDetectorRef, ElementRef, dan sebagainya.
  2. Dapatkan layanan untuk menggunakannya dalam komponen.
  3. Dapatkan beberapa konfigurasi global untuk token, yang dinyatakan di suatu tempat di atas. Misalnya, setel token API_URL di modul root dan dapatkan dari DI di mana saja dalam aplikasi, jika perlu.


Yang kurang umum adalah kasus di mana pengembang melangkah lebih jauh dan mengubah token global yang ada menjadi bentuk yang lebih nyaman. Contoh konversi yang baik adalah token WINDOW dari paket @ ng-web-apis / common .



Angular menyediakan token DOCUMENT sehingga Anda bisa mendapatkan objek halaman dari mana saja dalam aplikasi: komponen Anda tidak tergantung pada objek global, mudah diuji, tidak ada yang akan putus dengan SSR.



Jika Anda secara teratur membutuhkan akses ke objek WINDOW, Anda dapat menulis token ini: 



import {DOCUMENT} from '@angular/common';
import {inject, InjectionToken} from '@angular/core';

export const WINDOW = new InjectionToken<Window>(
    'An abstraction over global window object',
    {
        factory: () => {
            const {defaultView} = inject(DOCUMENT);

            if (!defaultView) {
                throw new Error('Window is not available');
            }

            return defaultView;
        },
    },
);


Ketika seseorang meminta token WINDOW untuk pertama kalinya dari pohon DI, pabrik token akan dieksekusi - ia akan menerima objek DOCUMENT dari Angular dan mendapatkan tautan ke objek jendela dari itu.



Selanjutnya, saya mengusulkan untuk mempertimbangkan pendekatan yang berbeda untuk transformasi tersebut - ketika mereka dilakukan secara langsung di penyedia komponen atau arahan, yang menyuntikkan hasilnya.



Penyedia swasta



Di tim kami, kami secara aktif menggunakan DI ketika bekerja dengan Angular dan mulai memperhatikan bahwa seringkali kami perlu melakukan beberapa transformasi data yang diperoleh dari DI sebelum menggunakannya. Faktanya, komponen kami membutuhkan beberapa data, dan kami menanamkan yang lain di dalamnya dan menjalankan logika transformasi di dalamnya.







Mari kita lihat contoh yang solid segera. Erin Coglar, dalam laporannya The Architecture of Components di konferensi besar internasional Angular Connect, menunjukkan contoh:





Jika Anda merasa tidak nyaman untuk membuka dan menonton video, maka mari masuki kasing di sini.



Kita punya:



  • Komponen yang bertanggung jawab untuk menampilkan informasi pada entitas tertentu - organisasi.
  • Parameter-kueri dari rute, yang menunjukkan id organisasi yang saat ini kami gunakan.
  • Layanan yang oleh id mengembalikan Observable dengan informasi tentang organisasi.


Apa yang ingin kita lakukan?



Ambil id organisasi dari parameter kueri, berikan ke metode layanan, dan dapatkan aliran dengan informasi tentang organisasi sebagai tanggapan. Tampilkan informasi ini di komponen.



Pertimbangkan tiga cara untuk mencapai yang diinginkan dan menganalisanya.



Cara melakukannya tidak perlu.



Terkadang saya menemukan gaya bekerja dengan data dalam komponen. Tolong jangan lakukan ini: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent implements OnInit {
   organization: Organization;

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}

   ngOnInit() {
       this.activatedRoute.params
           .pipe(
               switchMap(params => {
                   const id = params.get('orgId');

                   return this.organizationService.getOrganizationById$(id);
               }),
           )
           .subscribe(organization => {
               this.organization = organization;
           });
   }
}


Untuk menggunakan data yang diterima dalam templat: 



<p *ngIf="organization">
   {{organization.name}} from {{organization.city}}
</p>


Kode ini akan berfungsi, tetapi memiliki sejumlah masalah:



  • organization: . undefined. ( strict TypeScript), (organization?: Organization) .
  • . , ngOnInit, .

Mari kita lakukan dengan baik.



Laporan Erin dari video yang saya terapkan di atas dilakukan dengan baik. Dengan variannya ternyata seperti ini: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OrganizationComponent {
   readonly organization$: Observable<Organization> = this.activatedRoute.params.pipe(
       switchMap(params => {
           const id = params.get('orgId');
           return this.organizationService.getOrganizationById$(id);
       }),
   );

   constructor(
       private readonly activatedRoute: ActivatedRoute,
       private readonly organizationService: OrganizationService,
   ) {}
}


Untuk menggunakan data yang diterima dalam templat: 

<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>


Kode ini berfungsi dengan baik dan bebas dari kelemahan pendekatan sebelumnya: tampilannya cukup rapi, tidak memiliki bidang tambahan. Jika kita ingin memperluas komponen dengan aliran yang serupa, kita hanya menambahkan satu lagi - untuk membuatnya lebih mudah, karena untuk menambahkan aliran baru kita tidak perlu menyentuh kode yang sebelumnya.



Selain itu, aliran data menjadi lebih transparan: kami hanya memiliki aliran yang dibuat ketika kelas komponen dibuat. Ketika mengeluarkan data, informasi dalam templat kami akan ditampilkan.



Mari membuatnya lebih keren: penyedia swasta



Mari kita lihat lebih dekat solusi sebelumnya.



Bahkan, komponennya tidak tergantung pada router dan bahkan dari OrganizationService. Itu tergantung pada organisasi $. Tetapi tidak ada entitas seperti itu di pohon injeksi ketergantungan kami, jadi kami terpaksa melakukan transformasi dalam komponen.



Tetapi bagaimana jika Anda melakukan konversi ini sebelum data masuk ke komponen? Mari kita menulis Penyedia khusus untuk komponen di mana transformasi yang diperlukan akan terjadi.



Untuk kenyamanan, kami menempatkan penyedia dalam file terpisah di sebelah komponen, menerima struktur file berikut:







Penyedia akan berlokasi di file organization.providers.ts untuk konversi data dan tanda untuk menerimanya dalam komponen: 



export const ORGANIZATION_INFO = new InjectionToken<Observable<Organization>>(
   'A stream with current organization information',
);
         :

export const ORGANIZATION_PROVIDERS: Provider[] = [
   {
       provide: ORGANIZATION_INFO,
       deps: [ActivatedRoute, OrganizationService],
       useFactory: organizationFactory,
   },
];

export function organizationFactory(
   {params}: ActivatedRoute,
   organizationService: OrganizationService,
): Observable<Organization> {
   return params.pipe(
       switchMap(params => {
           const id = params.get('orgId');

           return organizationService.getOrganizationById$(id);
       }),
   );
}


Tentukan array penyedia untuk komponen. Nilai untuk token ORGANIZATION_INFO akan diperoleh dari pabrik tempat kami akan melakukan konversi data yang diperlukan.



Catatan tentang pengoperasian DI - deps memungkinkan Anda untuk mengambil entitas yang diperlukan dari pohon DI dan meneruskannya sebagai argumen ke pabrik nilai token. Di dalamnya, Anda bisa mendapatkan entitas apa pun dari DI, termasuk menggunakan dekorator DI, misalnya: 



{
       provide: ACTIVE_TAB,
       deps: [
           [new Optional(), new Self(), RouterLinkActive],
       ],
       useFactory: activeTabFactory,
}


Menyatakan penyedia dalam komponen: 



@Component({
   ..
   providers: [ORGANIZATION_PROVIDERS],
})


Dan kami siap menggunakan data dalam komponen: 



@Component({
   selector: 'organization',
   templateUrl: 'organization.template.html',
   styleUrls: ['organization.style.less'],
   changeDetection: ChangeDetectionStrategy.OnPush,
   providers: [ORGANIZATION_PROVIDERS],
})
export class OrganizationComponent {
   constructor(
       @Inject(ORGANIZATION_INFO) readonly organization$: Observable<Organization>,
   ) {}
}


Kelas komponen dikurangi menjadi satu baris dengan penerimaan data.



Templatnya tetap sama: 



<p *ngIf="organization$ | async as organization">
   {{organization.name}} from {{organization.city}}
</p>
