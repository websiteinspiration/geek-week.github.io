Datenbankmigrationen mit Flyway

Vor Beginn des Java Developer- Kurses wurde eine Übersetzung des Artikels erstellt .


1. Einleitung


Dieser Artikel beschreibt die Schlüsselkonzepte von Flyway und ein Beispiel für die Verwendung dieses Frameworks zum kontinuierlichen Ändern des Datenbankschemas unter Verwendung der speicherinternen H2-Datenbank als Beispiel unter Verwendung des Flyway-Maven-Plugins.

Flyway aktualisiert Datenbankversionen mithilfe von Migrationen. Migrationen können in SQL (mit einer für ein bestimmtes DBMS spezifischen Syntax) oder in Java geschrieben werden.

Migrationen können versioniert oder wiederkehrend sein. Die ersteren haben eine einzigartige Version und werden genau einmal angewendet. Die zweiten haben keine Versionsnummer und gelten, wenn sich ihre Prüfsumme ändert.

Wiederholte Migrationen innerhalb eines einzelnen Laufs werden immer angewendet, nachdem versionierte Migrationen abgeschlossen sind. Wiederholte Migrationen werden in der beschriebenen Reihenfolge angewendet. Bei einer Migration werden alle Vorgänge innerhalb einer einzelnen Datenbanktransaktion ausgeführt.

In diesem Artikel konzentrieren wir uns auf die Verwendung des Maven-Plugins für Datenbankmigrationen.

2. Flyway Maven Plugin


Fügen Sie das Flyway Maven Plugin hinzu zu pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
</plugin>

Die aktuelle Version des Plugins kann in Maven Central angezeigt werden .
Eine Liste der Plugin-Parameter finden Sie in der Dokumentation . Plugin-Parameter können auf vier verschiedene Arten konfiguriert werden.

2.1. <configuration>Plugin- Bereich
Parameter können direkt im Tag im Plugin-Bereich angegeben werden in pom.xml:

<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>4.0.3</version>
    <configuration>
        <user>databaseUser</user>
        <password>databasePassword</password>
        <schemas>
            <schema>schemaName</schema>
        </schemas>
        ...
    </configuration>
</plugin>

2.2. Maven Eigenschaften


Sie können das Plugin auch konfigurieren, indem Sie die folgenden Parameter angeben <properties>:

<project>
    ...
    <properties>
        <flyway.user>databaseUser</flyway.user>
        <flyway.password>databasePassword</flyway.password>
        <flyway.schemas>schemaName</flyway.schemas>
        ...
    </properties>
    ...
</project>

2.3. Externe Konfigurationsdatei


Oder beschreiben Sie die Konfiguration in einer separaten .propertiesDatei:

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=schemaName
...

Der Standardname ist die Konfigurationsdatei flyway.properties. Diese Datei muss sich im selben Verzeichnis wie die Datei befinden pom.xml. Die Codierung wird im Parameter festgelegt flyway.encoding(Standard ist UTF-8).

Wenn Sie einen anderen Namen für die Datei verwenden (z. B. customConfig.properties), müssen Sie diesen beim Aufruf von maven explizit angeben:

$ mvn -Dflyway.configFile=customConfig.properties

2.4. Systemeigenschaften


Schließlich können Parameter als Systemeigenschaften angegeben werden, wenn maven über die Befehlszeile aufgerufen wird:

$ mvn -Dflyway.user=databaseUser -Dflyway.password=databasePassword
  -Dflyway.schemas=schemaName

Wenn die Konfiguration auf verschiedene Arten angegeben wird, lautet die Priorität wie folgt:

  1. Systemeigenschaften
  2. Externe Konfigurationsdatei
  3. Abschnitt <Eigenschaften>
  4. Plugin - <Konfiguration Abschnitt>

3. Migrationsbeispiel


In diesem Abschnitt werden die erforderlichen Schritte zum Migrieren eines Datenbankschemas anhand der speicherinternen H2-Datenbank als Beispiel mit dem Maven-Plugin betrachtet. Um Flyway zu konfigurieren, verwenden wir eine externe Datei.

3.1. POM ändert sich


Fügen Sie zunächst eine Abhängigkeit von H2 hinzu:

<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>1.4.196</version>
</dependency>

Hier können wir auch die neueste verfügbare Treiberversion in Maven Central überprüfen . Fügen Sie ein Plugin für Flyway hinzu, wie zuvor beschrieben.

3.2. Konfigurieren Sie Flyway in einer externen Datei


Erstellen Sie eine $PROJECT_ROOTDatei myFlywayConfig.propertiesmit folgendem Inhalt:

flyway.user=databaseUser
flyway.password=databasePassword
flyway.schemas=app-db
flyway.url=jdbc:h2:mem:DATABASE
flyway.locations=filesystem:db/migration

Die obige Konfiguration zeigt an, dass sich die Migrationsskripte im Verzeichnis befinden db/migrationund über H2 databaseUserund eine Verbindung zur Datenbank hergestellt werden sollen databasePassword.

Datenbankschema für die Anwendung app-db.

Natürlich in den Parametern flyway.user, flyway.passwordund flyway.urlSie müssen den Namen, Passwort und URL Ihrer Datenbank angeben.

3.3. Erste Migration


Unter Flyway-Konventionen müssen die Namen der Migrationsskripte das folgende Format haben:

<Prefix<code>><</code>Version>__<code><</code>Description>.sql

Wo:

  • <Präfix> - Präfix. Bei versionierten Migrationen ist der Standardwert "V". Das Präfix wird über die Eigenschaft flyway.sqlMigrationPrefix konfiguriert.
  • <Version> - Versionsnummer der Migration. Die Haupt- und Nebenversion können durch Unterstreichen getrennt werden. Die Version sollte immer mit 1 beginnen.
  • <Beschreibung> - Eine Textbeschreibung der Migration. Die Beschreibung muss durch zwei Unterstriche von der Versionsnummer getrennt sein.

Beispiel: V1_1_0__my_first_migration.sql

Erstellen wir also ein Verzeichnis db/migrationin $PROJECT_ROOTder Skriptmigration V1_0__create_employee_schema.sqlund SQL, um die Tabelle zu erstellen employee:

CREATE TABLE IF NOT EXISTS `employee` (
 
    `id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name` varchar(20),
    `email` varchar(50),
    `date_of_birth` timestamp
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;

3.4. Wir führen Migrationen durch


$PROJECT_ROOTFühren Sie als Nächstes den folgenden Befehl maven aus, um die Datenbankmigrationen anzuwenden:

$ mvn clean flyway:migrate -Dflyway.configFile=myFlywayConfig.properties

Unsere erste Migration muss abgeschlossen sein.

Das Datenbankschema sieht nun folgendermaßen aus:
Mitarbeiter:

+----+------+-------+---------------+
| id | name | email | date_of_birth |
+----+------+-------+---------------+

Wir können die vorherigen Schritte wiederholen, um andere Migrationen durchzuführen.

3.5. Zweite Migration


Erstellen Sie für die zweite Migration eine Datei mit dem Namen V2_0_create_department_schema.sql, der die folgenden zwei Anforderungen enthält:

CREATE TABLE IF NOT EXISTS `department` (
 
`id` int NOT NULL AUTO_INCREMENT PRIMARY KEY,
`name` varchar(20)
 
)ENGINE=InnoDB DEFAULT CHARSET=UTF8;
 
ALTER TABLE `employee` ADD `dept_id` int AFTER `email`;

Lassen Sie uns die Migration genauso durchführen wie bei der ersten Migration.

Jetzt hat sich das Schema unserer Datenbank geändert: employeeEine neue Spalte wurde hinzugefügt und eine neue Tabelle wurde erstellt department:

employee:
+----+------+-------+---------+---------------+
| id | name | email | dept_id | date_of_birth |
+----+------+-------+---------+---------------+

department:

+----+------+
| id | name |
+----+------+

Führen Sie den folgenden Befehl maven aus, um zu überprüfen, ob beide Migrationen erfolgreich waren:

$ mvn flyway:info -Dflyway.configFile=myFlywayConfig.properties

4. Deaktivieren von Flyway im Spring Boot


Manchmal müssen Sie möglicherweise Flyway-Migrationen deaktivieren.

Dies kann während Tests erforderlich sein, wenn ein Datenbankschema basierend auf Entitäten generiert wird. In diesem Fall können Sie Flyway für das Testprofil deaktivieren.

In Spring Boot ist dies sehr einfach.

4.1. Federstiefel 1.x.


Alles was wir tun müssen, ist die Eigenschaft flyway.enabledin der Datei festzulegen application-test.properties:

flyway.enabled=false


4.2. Federstiefel 2.x.


In späteren Versionen von Spring Boot wurde diese Eigenschaft in geändert 

spring.flyway.enabled:
spring.flyway.enabled=false

4.3 FlywayMigrationStrategy leeren


Wenn Sie beim Start nur die automatische Flyway-Migration deaktivieren möchten, die Migrationen jedoch manuell starten möchten, funktioniert die Verwendung der oben genannten Eigenschaften für uns nicht.

Dies liegt an der Tatsache, dass Spring Boot Flyway-Bins nicht automatisch konfiguriert und wir sie daher selbst konfigurieren müssen, was nicht sehr praktisch ist.

In diesem Fall können wir Flyway aktiviert lassen und eine leere FlywayMigrationStrategy implementieren :

@Configuration
public class EmptyMigrationStrategyConfig {
 
    @Bean
    public FlywayMigrationStrategy flywayMigrationStrategy() {
        return flyway -> {
            // do nothing 
        };
    }
}

Tatsächlich werden dadurch Flyway-Migrationen deaktiviert, wenn die Anwendung gestartet wird.

Wir können die Migration jedoch weiterhin manuell starten:

@RunWith(SpringRunner.class)
@SpringBootTest
public class ManualFlywayMigrationIntegrationTest {
 
    @Autowired
    private Flyway flyway;
 
    @Test
    public void skipAutomaticAndTriggerManualFlywayMigration() {
        flyway.migrate();
    }
}

5. Wie Flyway funktioniert


Um zu verfolgen, wann, von wem und welche Migrationen angewendet wurden, wird dem Datenbankschema eine spezielle Tabelle mit Metadaten hinzugefügt. In dieser Tabelle werden auch die Prüfsummen der Migrationen und Informationen darüber gespeichert, ob die Migration erfolgreich war oder nicht.

Das Framework funktioniert wie folgt:

  1. Überprüft das Datenbankschema auf eine Metadatentabelle (Standard SCHEMA_VERSION). Wenn die Metadatentabelle nicht vorhanden ist, wird sie erstellt.
  2. Durchsucht den Klassenpfad nach verfügbaren Migrationen.
  3. Vergleicht Migrationen mit einer Metadatentabelle. Wenn die Versionsnummer kleiner oder gleich der als aktuell gekennzeichneten Version ist, wird sie ignoriert.
  4. Markiert alle verbleibenden Migrationen als ausstehend. Anschließend werden sie in aufsteigender Versionsnummer sortiert und in der angegebenen Reihenfolge ausgeführt.
  5. Aktualisiert die Metadatentabelle, wenn Migrationen angewendet werden.

6. Teams


Flyway verfügt über die folgenden Hauptteams für das Migrationsmanagement:

  • Info . Zeigt den aktuellen Status / die aktuelle Version des Datenbankschemas an. Informationen darüber, welche Migrationen erwartet werden, welche angewendet wurden, den Status abgeschlossener Migrationen und das Datum ihres Abschlusses.
  • Migrieren . Aktualisieren des Datenbankschemas auf die aktuelle Version. Scannen Sie einen Klassenpfad, um verfügbare Migrationen zu finden und ausstehende Migrationen anzuwenden.
  • Basislinie . Installieren der Datenbankschemaversion, Ignorieren von Migrationen bis einschließlich baseVersion. Mit Baseline können Sie Flyway in einer vorhandenen Datenbank verwenden. Neue Migrationen werden wie gewohnt angewendet.
  • Die validieren . Überprüfen des aktuellen Datenbankschemas auf verfügbare Migrationen.
  • Reparieren . Stellen Sie die Metadatentabelle wieder her.
